## Label Justifications

- `P1`: The PR targets a high-priority channel workflow where an existing WeChat account can stop receiving messages after another account logs in.
- `merge-risk: 🚨 compatibility`: The changed account restart semantics can alter existing multi-account channel reload and plugin disable/re-enable behavior.
- `merge-risk: 🚨 message-delivery`: Starting, dropping, or reviving the wrong channel account can cause missed or misrouted WeChat messages.

## Summary

Keep this PR open, but it is not merge-ready: the patch still misses the actual WeChat login config-reload path and can also revive stale accounts after a disabled-plugin reload.

## What This Changes

The PR adds a gateway known-account restart path for plugin reloads, documents WeChat multi-account hot reload behavior, and adds gateway plus agent regression coverage.

## Best Possible Solution

Keep the fix in the gateway lifecycle layer, but rework the handoff so WeChat login config-prefix reloads preserve currently running accounts while ordinary config removals and disabled-plugin re-enables do not restore stale accounts.

## Reproduction Assessment

Yes. Source inspection shows WeChat login writes `channels.openclaw-weixin.channelConfigUpdatedAt`, which takes the config-only restart branch that this PR does not preserve with known accounts.

## Solution Assessment

No. The proposed plugin-reload-only safety net is only a plausible partial fix; the best fix must cover the actual login reload path while keeping the stale-account removal guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve accounts on the login reload path:** `src/gateway/server-reload-handlers.ts:486-490`
  - body: The WeChat login flow writes `channels.openclaw-weixin.channelConfigUpdatedAt`, which maps to a channel restart with `plan.reloadPlugins === false`; this branch therefore calls `startChannel(name)` without `includeKnownAccounts`. Because this PR also marks non-manual stops as caller-deferred, the previously running account reaches the new early return in `server-channels.ts` instead of current-main recovery, so account A can still stay offline when the plugin list temporarily contains only account B.
  - confidence: 0.88
- **[P2] Avoid reviving accounts after disabled-plugin reloads:** `src/gateway/server-reload-handlers.ts:486-488`
  - body: When a plugin reload disables or removes a channel, the pre-replace non-manual stop can leave an old account as `restartPending`. A later plugin reload that re-enables the channel takes this `includeKnownAccounts` branch and unions that stale account into the fresh start set even though there was no same-reload handoff; gate the safety net to channels stopped before this same replacement and still active, or clear preserved state for inactive channels.
  - confidence: 0.8
