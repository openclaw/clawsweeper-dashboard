## Summary

Keep this PR open: it is still a useful Gateway bug-fix candidate, but the current patch can carry stale known-account state across a disabled/removed plugin interval and later restart an account the fresh plugin listing no longer includes.

## What This Changes

The PR adds a Gateway channel hot-reload option to include already-known accounts during plugin reloads, updates WeChat docs, and adjusts gateway/agent regression tests.

## Best Possible Solution

Keep the generic Gateway-owned hot-reload repair, but make preserved account handoff state single-reload scoped so disabled or removed plugin accounts cannot be revived later.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Clear preserved accounts when plugin reload disables a channel:** `src/gateway/server-reload-handlers.ts:487`
  - body: When a plugin reload disables or removes a channel, `reloadAttachedGatewayPlugins` skips the final restart, but the pre-replace `stopChannel(..., { manual: false })` now leaves the old account in runtime state as `restartPending`. On the next plugin reload that re-enables the channel, this unconditional known-account union can restart that stale account even if the fresh plugin listing no longer includes it. Clear the preserved handoff state for intentionally inactive channels, or restrict the safety net to the same reload handoff that actually restarts the channel.
  - confidence: 0.83
