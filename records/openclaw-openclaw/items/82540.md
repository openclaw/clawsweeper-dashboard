## Summary

Keep this PR open: the gateway patch looks focused and I did not find a new line-level code defect, but the user-visible WeChat A+B QR-login behavior still lacks live provider proof or explicit maintainer acceptance of gateway-level proof.

## What This Changes

This PR adds a gateway `includeKnownAccounts` channel restart option, gates it to plugin/account-index hot reloads, updates WeChat docs, and adds gateway/agent regression tests.

## Best Possible Solution

Land only after redacted live WeChat A+B proof is supplied, or after maintainers explicitly accept the targeted gateway regression proof as sufficient for this provider-owned setup gap.

## Reproduction Assessment

Yes from source and targeted regression shape, but not from a live provider session. Current main restarts channel hot reloads from the plugin-listed account ids only, while the PR body and tests model the temporary account-index case where only account B is listed after account B login.

## Solution Assessment

Yes for the core boundary, with a proof caveat. The PR keeps the fix in the generic gateway hot-reload handoff instead of adding WeChat-specific core policy, and it avoids ordinary config reloads, but live provider validation or maintainer acceptance is still required.
