## Summary

Keep open: the PR targets a plausible gateway hot-reload bug, but the proposed lifecycle safety net can preserve stale `connected: true` state and restart accounts that were stopped or removed. It also still needs after-fix real behavior proof; the body only lists tests/build/checks.

## What This Changes

This PR changes gateway channel hot reload to stop/restart as a manager-owned handoff, include previously live accounts during restart, update WeChat docs, and add gateway regression tests.

## Best Possible Solution

Keep the hot-reload handoff, but preserve only accounts that are active in that handoff or explicitly pending restart, clear or ignore stale liveness on stop, add a regression with `connected: true`, and provide real WeChat proof.

## Reproduction Assessment

No for the reported live WeChat behavior: the PR body does not provide a real setup reproduction or after-fix proof. Source inspection does give a clear reproduction path for the patch regression when a stopped account still has `connected: true`.

## Solution Assessment

No. The direction is reasonable, but using `snapshot.connected` as a known-live source without clearing it on stop is not the narrowest maintainable fix because it preserves stale liveness across manual stops and account removal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Do not treat stale connected flags as live accounts:** `src/gateway/server-channels.ts:366`
  - body: After a clean stop, `setRuntime` updates `running` and `restartPending` but leaves an earlier `connected: true` field intact. Because the new safety net includes any snapshot with `connected`, a manually stopped or removed account that is absent from `listAccountIds()` can survive eviction and restart during `includeKnownAccounts`; clear or ignore stale liveness and cover the `connected: true` case.
