## Label Justifications

- `P1`: The underlying WeChat multi-account hot-reload failure can leave an account offline and silently miss channel messages until recovery or restart.
- `merge-risk: 🚨 compatibility`: The PR changes channel restart semantics and can preserve stale runtime-known accounts across plugin disable/re-enable upgrade paths.
- `merge-risk: 🚨 message-delivery`: A stale or missing account restart decision in this path can leave a channel account offline or revive the wrong account for message delivery.

## Summary

Current main and the shipped v2026.5.18 release already fix the reported WeChat multi-account hot-reload failure through non-manual reload stops and per-account recovery; the PR’s remaining handoff rewrite is redundant and carries stale-account restart risk.

## What This Changes

The PR adds a Gateway `includeKnownAccounts` hot-reload start option, manager-owned restart handoff state, WeChat docs wording, and gateway/agent regression test updates.

## Best Possible Solution

Keep the shipped gateway lifecycle fix; if maintainers still want zero-downtime account handoff, open a fresh narrow PR that covers the actual config-prefix reload path and proves account additions cannot resurrect removed accounts.

## Reproduction Assessment

Yes for source-level behavior: the historical WeChat failure maps to `channels.openclaw-weixin` reloads and current main now stops non-manually and recovers by account id. I did not run a live WeChat A+B session in this read-only review.

## Solution Assessment

No; this branch is not the best way to solve the issue because the shipped main-line fix already covers the reported failure and the new known-account union adds stale-account risk. A new follow-up would need to distinguish account additions from removals on the actual config-prefix reload path.
