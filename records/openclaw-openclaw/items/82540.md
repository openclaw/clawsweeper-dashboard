## Summary

Keep open: the fix direction is valid for the WeChat multi-account hot-reload bug, but the current head still applies the known-account safety net to ordinary channel config reloads, which can restart removed or renamed accounts. The PR also still has only targeted Vitest proof, not real WeChat A+B hot-reload proof.

## What This Changes

The PR adds a ChannelManager `includeKnownAccounts` start option, uses it during gateway hot reload, updates WeChat docs, and expands gateway plus agent tests.

## Best Possible Solution

Keep the hot-reload account preservation, but pass `includeKnownAccounts` only for plugin-reload/login handoffs and prove both WeChat A+B login preservation and ordinary account removal behavior.

## Reproduction Assessment

Yes for the PR-introduced defect from source inspection: an ordinary channel config reload with `plan.reloadPlugins === false` now stops the channel with `manual: false` and restarts with `includeKnownAccounts: true`. I did not run tests because this review is read-only.

## Solution Assessment

No. The maintainable fix is to scope known-account preservation to the plugin-reload/login handoff that needs it, while preserving current list-only behavior for ordinary config reloads so removed or renamed accounts are honored.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restrict known-account restarts to plugin reload handoffs:** `src/gateway/server-reload-handlers.ts:482`
  - body: This call also runs for ordinary channel config reloads where `plan.reloadPlugins` is false. The preceding `stopChannel(..., { manual: false })` marks old running accounts as restart-pending, and `includeKnownAccounts` unions those accounts back into the next start set, so removing or renaming `channels.<id>.accounts` can be ignored until a full gateway restart. Keep the safety net on the plugin-reload/login handoff and add a regression that ordinary reloads do not restart removed accounts.
  - confidence: 0.9
