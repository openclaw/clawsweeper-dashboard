## Summary

Keep open: the patch appears to address the account-index hot-reload gap without a new line-level code blocker, but the user-visible WeChat A+B behavior is still proven only by gateway regression tests and build/typecheck output, not by a live QR-login provider session.

## What This Changes

The PR adds a gateway `includeKnownAccounts` restart option for plugin/account-index hot reloads, narrows it away from ordinary config edits, updates WeChat docs, and adds gateway/agent regression coverage.

## Best Possible Solution

Land the channel-agnostic gateway lifecycle fix only after redacted live WeChat A+B proof is supplied, or after maintainers explicitly accept the gateway regression proof for this provider-owned live setup gap.

## Reproduction Assessment

No live current-main reproduction was run in this review. The linked issue logs, current main source, and the published WeChat plugin contract make the failure source-reproducible through `channels.openclaw-weixin.channelConfigUpdatedAt` hot reload, but provider-level proof is still missing.

## Solution Assessment

Yes, this appears to be the right layer: the patch keeps OpenClaw core channel-agnostic and limits known-account preservation to plugin/account-index handoffs rather than ordinary config edits. The remaining blocker is proof/maintainer acceptance, not a narrower code repair I can identify from the diff.
