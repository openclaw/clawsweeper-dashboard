## Summary

Keep open: this is a plausible channel-agnostic fix for the linked WeChat multi-account bug, but the latest head still misses the plugin-reload pre-replace stop path and only has tests/CI instead of real WeChat hot-reload proof. ## What This Changes

The PR changes gateway channel hot reload to non-manual stop/restart with known live accounts, updates WeChat docs, and adds gateway lifecycle regression tests.

## Best Possible Solution

Complete the generic gateway hot-reload handoff across both pre-replace and restart stops, keep WeChat docs aligned, and land only after focused tests plus redacted real WeChat multi-account proof.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. The linked issue has concrete v2026.5.12 logs, and the Weixin package plus gateway source make the stop/restart failure path source-reproducible enough to review the PR.

## Solution Assessment

No, not yet. The generic gateway handoff is the right direction, but the proposed patch must also use a non-manual handoff for the pre-replace plugin stop before it covers the reported plugin hot-reload path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Propagate the non-manual handoff before plugin replacement:** `src/gateway/server-reload-handlers.ts:401`
  - body: When plugin reload stops a channel in `beforeReplace`, it still uses the default manual stop and records the channel in `channelsStoppedBeforePluginReload`. The restart path added here then skips the new non-manual stop, so the old account is no longer `restartPending` or connected when `includeKnownAccounts` runs; if the reloaded WeChat plugin temporarily lists only account B, account A can still be dropped. Pass `{ manual: false }` through the pre-replace stop as well.
  - confidence: 0.9
