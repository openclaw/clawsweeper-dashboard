## Summary

The PR addresses a real gateway hot-reload account-loss path, but its exact-head preservation rule can retain `restartPending` state across an intentional plugin-disabled interval and later union a removed account into a re-enabled channel. Current main does not contain the proposed fix, and the PR also lacks after-fix live WeChat A+B proof, so it should remain open for a narrow lifecycle repair and provider validation.

## What This Changes

The PR adds gateway channel-manager options and reload handling to preserve running accounts across plugin replacement, plus WeChat documentation and gateway regression tests.

## Best Possible Solution

Capture the accounts active immediately before plugin replacement and consume that set only for channels that remain active after the same replacement; discard it on disable/removal, keep config-only reloads unchanged, and validate with a redacted live WeChat A+B reload.

## Reproduction Assessment

Yes for the gateway state machine: the source and focused tests establish the stop/restart path and the stale `restartPending` sequence. No high-confidence live WeChat A+B reproduction or after-fix provider run is available.

## Solution Assessment

No. Preserving accounts across a plugin replacement is the right direction, but deriving them from durable manager runtime state is broader than the single handoff and permits stale account resurrection.
