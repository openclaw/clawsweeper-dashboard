## Summary

Keep this PR open: it targets a real WeChat multi-account hot-reload bug, but the current patch can resurrect stopped accounts through stale `connected: true` state and still lacks required real WeChat proof.

## What This Changes

The PR changes gateway channel hot reload to stop/restart as a manager-owned handoff, include known live accounts during restart, update WeChat docs, and add gateway regression tests.

## Best Possible Solution

Preserve hot-reload handoff only for accounts known active in that reload/restart lifecycle, clear or ignore stale `connected` liveness on stop, and land with focused regression coverage plus redacted live WeChat proof.

## Reproduction Assessment

No for live WeChat: the PR lacks real setup proof. Yes for the patch regression: PR head source inspection shows `connected: true` can remain after clean stop and be re-added by `includeKnownAccounts`.

## Solution Assessment

No. The direction is reasonable, but using `snapshot.connected` as a known-live source without clearing it on stop is not the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Do not treat stale connected flags as live accounts:** `src/gateway/server-channels.ts:366`
  - body: After any channel reports `connected: true`, the clean stop path only updates `running` and `restartPending`, leaving `connected` intact. Since the new safety net includes any snapshot with `connected`, a manually stopped or removed account that is absent from `listAccountIds()` can survive eviction and restart during `includeKnownAccounts`; clear or ignore stale liveness and add a `connected: true` regression.
  - confidence: 0.91
