## Label Justifications

- `P1`: The PR targets a P1 message-loss hot-reload bug but currently introduces a high-impact account lifecycle regression that can affect real channel workflows.
- `merge-risk: 🚨 compatibility`: Merging this changes upgrade-time channel config reload semantics by preserving accounts that may have been removed or renamed.
- `merge-risk: 🚨 message-delivery`: A stale account kept alive by the broad safety net can continue receiving or sending channel messages after the operator expected it to stop.
- `merge-risk: 🚨 security-boundary`: Account removal is an operator control boundary, and this diff can keep a removed account running until a full gateway restart.

## Summary

Keep this PR open, but it is not merge-ready: the latest diff still applies the known-account safety net to ordinary channel config restarts, which can keep removed or renamed accounts alive, and the supplied proof is mocked gateway coverage rather than live WeChat account A+B behavior.

## What This Changes

The PR adds a gateway `startChannel` option to include already-known channel accounts during hot reload, calls it from reload restart paths, updates WeChat docs, and adds gateway regression tests.

## Best Possible Solution

Keep the channel-agnostic fix, but constrain the known-account safety net to the WeChat login/plugin-reload handoff that needs it while preserving normal account removal and rename semantics.

## Reproduction Assessment

Yes for the patch risk by source inspection: a non-plugin channel config reload stops with `manual: false`, then the PR restarts with `includeKnownAccounts`, and the PR's stop path marks those accounts `restartPending`. The original live WeChat A+B flow was not reproduced here; the linked issue supplies logs, while the PR proof is mocked tests only.

## Solution Assessment

No. The broad `includeKnownAccounts` restart is not the safest fix because it conflates login handoff preservation with normal account removal or rename; the narrow maintainable path is to gate this safety net to the handoff that needs it and test both sides.
