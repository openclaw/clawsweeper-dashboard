## Summary

Keep open: current main still has the reported diagnostic/recovery gap for native tool-call stalls, and the existing tests explicitly assert that recovery is not invoked while a bash/native tool call is active.

## What This Changes

The issue reports that `openclaw sessions --agent dex --limit 10 --json` can wedge a live gateway turn when run from inside an active bash tool call, leaving diagnostics at `recovery=none`.

## Best Possible Solution

Add a narrow fail-fast or safe-read path for `openclaw sessions` from live tool-call ancestry, and make `blocked_tool_call` diagnostics lead to bounded cancellation or a clear actionable recovery outcome with regression coverage.

## Reproduction Assessment

Yes for the diagnostic/recovery failure from source: the existing diagnostic test constructs the active bash tool-call state and verifies no recovery is invoked. A live nested Telegram/gateway run was not performed in this read-only review.

## Solution Assessment

Unclear as an implementation choice. The narrowest maintainable direction is likely a targeted nested `openclaw sessions` guard or safe non-blocking session read first, with native tool-call killing reserved for a clearly bounded recovery policy.
