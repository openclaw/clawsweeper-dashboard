## Summary

Keep open: the report matches current `main` behavior, because blocked native tool calls are classified but deliberately not recovery-eligible, and the normal restart drain can wait far longer than the generated systemd stop budget.

## What This Changes

The issue reports a gateway deadlock when an active tool call invokes `openclaw sessions`, with diagnostics classifying the stall as `blocked_tool_call` but reporting `recovery=none`.

## Best Possible Solution

Define a core recovery policy for blocked native tool calls under gateway supervision, likely combining a fail-fast guard for unsafe nested CLI calls with a restart drain path that reaps stale work before the service manager kills the process.

## Reproduction Assessment

Yes, at source level. The issue has live logs, and current tests/source show the same `blocked_tool_call` classification with no recovery call; I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear. The desired outcome is correct, but the safest fix needs a maintainer decision about whether to block nested `openclaw sessions`, abort stale native tool calls, change restart drain policy, or do a narrower combination.
