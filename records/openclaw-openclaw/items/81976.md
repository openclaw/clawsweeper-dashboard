## Summary

Keep open: current main still has the reported unrecovered `blocked_tool_call` path, and the closely related open PRs have unresolved proof/design blockers rather than a landed fix.

## What This Changes

The issue asks OpenClaw to prevent or recover a gateway stall when a live bash tool call runs `openclaw sessions`, and to avoid `blocked_tool_call` diagnostics ending at `recovery=none`.

## Best Possible Solution

Keep this issue open as concrete live evidence, then reconcile it with the existing recovery and restart-drain PRs by extending the existing stuck-session abort contract to stale tool-call runs and preserving the public restart-drain config contract.

## Reproduction Assessment

Yes, source-level: current main deterministically classifies stale bash/tool activity as `blocked_tool_call` with `recoveryEligible: false`, which produces the reported `recovery=none` log. I did not establish a live Telegram repro in this read-only pass.

## Solution Assessment

Unclear for any one proposed patch: the local wrapper is a useful mitigation but not the product fix, and the open watchdog/restart-drain PRs still need proof and policy decisions. The narrowest maintainable direction is to reuse the existing diagnostic abort threshold and restart-drain contracts rather than add an unrelated one-off guard.
