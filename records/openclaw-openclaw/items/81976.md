## Summary

Current main still has the reported recovery gap: `blocked_tool_call` diagnostics are explicitly non-recovery-eligible, restart drain can wait the normal five-minute window, and I did not find a nested-tool-call guard around `openclaw sessions`. The issue should stay open as fresh production evidence, with the broader active tool-call recovery policy coordinated with the already-open canonical recovery tracker.

## What This Changes

Request to prevent or recover gateway deadlocks when `openclaw sessions` is invoked from inside an active agent tool call and diagnostics report `recovery=none`.

## Best Possible Solution

Keep this as concrete production evidence and either fold it into the open recovery tracker or implement an owner-approved narrow guard/recovery path for nested `openclaw sessions` and blocked native tool calls.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the diagnostic symptom: source and tests show `blocked_tool_call` currently produces `recovery=none` and is not recovered. The exact nested `openclaw sessions` deadlock has strong live report evidence but was not reproduced in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No final fix is present yet; the maintainable direction is either a narrow fail-fast guard for unsafe nested `openclaw sessions` or a deliberate recovery policy for blocked native tool calls, coordinated with the open recovery tracker.
