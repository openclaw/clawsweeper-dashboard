## Summary

Keep open: current main and v2026.6.9 still stat the literal local `exec.workdir` and fall back before command execution, while open linked PRs now own competing fix semantics.

## What This Changes

The issue requests that explicit `exec.workdir` values with leading `~` either expand consistently or fail without executing from a fallback directory.

## Best Possible Solution

Land one canonical exec workdir fix with focused regression coverage, keeping sandbox and node routing separate and making the chosen local fallback or expansion semantics explicit.

## Reproduction Assessment

Yes. Source inspection on current main shows local gateway exec passes explicit `workdir` into `resolveWorkdir()`, which stats the literal value and returns a fallback with a warning before `runExecProcess()`; I did not run a live command because this review is read-only.

Unclear as the final product choice. Keeping this issue open is the right triage state until maintainers choose fail-fast, OS-home expansion, or a combined behavior and land one linked PR.
