## Summary

Keep open: current main and the latest release still resolve local exec workdir values literally, warn on stat failure, and execute from a fallback directory. Five open PRs already reference this issue with closing syntax, so maintainer cleanup should pick or consolidate a canonical fix instead of closing the report or spawning another repair job.

## What This Changes

The issue reports that an explicit local `exec.workdir` beginning with `~` can be treated as unavailable, fall back to another cwd, and still run the command from the wrong directory.

## Best Possible Solution

Land a single canonical fix that expands leading home paths where appropriate and fails explicit unavailable local workdirs before command execution; keep implicit/default cwd fallback only if maintainers intentionally preserve it.

## Reproduction Assessment

Yes from source inspection: a gateway exec call with explicit `workdir` reaches `resolveWorkdir()`, which stats the literal string and returns a fallback path after failure. I did not run a live exec command in this read-only review, but the current-main code path and Node stat behavior match the report.

## Solution Assessment

No, closing this issue is not the best path because current main and the latest release still have the behavior. The best fix should be selected from the open linked PRs, with preference for eliminating execution from a fallback directory for explicit invalid workdirs.
