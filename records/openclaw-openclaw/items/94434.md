## Summary

Keep open. Current main and v2026.6.8 still resolve an unavailable explicit gateway workdir to a fallback directory, and the linked open fix PR now owns the focused fail-fast repair.

## What This Changes

The issue asks OpenClaw exec to stop running commands from a fallback directory when an explicit workdir such as a literal leading `~` path is unavailable.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/94441 or an equivalent focused fix that fails explicit invalid local workdirs before command execution while keeping implicit/default fallback behavior intentional.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.8 still call `statSync()` on the raw explicit workdir, warn, return a fallback directory, and continue toward command execution; I did not run a live tool call in this read-only review.

## Solution Assessment

Yes, the linked fail-fast direction is the narrowest maintainable fix for the central safety bug. Leading-home expansion can be considered separately, but preventing command execution from a fallback directory solves the reported wrong-location behavior without new config.
