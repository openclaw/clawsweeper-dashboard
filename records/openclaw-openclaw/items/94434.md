## Label Justifications

- `impact:other`: The issue affects local exec working-directory selection, a meaningful runtime behavior outside the more specific owned impact labels.

## Summary

Keep open: current main and latest release v2026.6.9 still validate local `exec.workdir` as a literal string and fall back before command execution, while open closing PRs already target the bug with different semantics.

## What This Changes

The issue asks OpenClaw to stop running local `exec` commands from a fallback cwd when an explicit leading-tilde `workdir` is unavailable, either by expanding `~` or failing before execution.

## Best Possible Solution

Land one canonical exec workdir fix after maintainers choose the local semantics, with regression coverage for explicit leading-tilde/unavailable workdirs and no change to sandbox or node routing beyond the chosen scope.

## Reproduction Assessment

Yes. Source inspection shows local gateway exec reaches `resolveWorkdir()`, which stats a literal `~/...` path and falls back before command execution; I did not run a live repro in this read-only review.

## Solution Assessment

Unclear. The report is valid, but maintainers still need to choose between fail-fast explicit invalid workdirs and OS-home tilde expansion; either solution should add focused regression coverage and keep sandbox/node behavior separate.
