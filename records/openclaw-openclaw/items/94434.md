## Summary

Keep open: current `main` and the latest release still resolve local `exec.workdir` by statting the raw string and falling back before command execution, while open closing PRs already target the bug with different semantics.

## What This Changes

The issue requests that explicit local `exec.workdir` values beginning with `~` stop executing from an unintended fallback directory, either by leading-home expansion or by failing before execution.

## Best Possible Solution

Land one canonical exec workdir fix with regression coverage, keeping sandbox and node routing separate and making the chosen local fallback or home-expansion semantics explicit.

## Reproduction Assessment

Yes at source level. Local non-sandbox, non-node `exec` with `workdir: "~/..."` reaches `resolveWorkdir()`, which stats the literal string and returns a fallback before execution.

## Solution Assessment

Unclear as a product choice, but keeping the issue open is the right triage state. Maintainers should choose between fail-fast explicit workdirs, OS-home expansion, or a combined expand-then-fail approach before closing the report.
