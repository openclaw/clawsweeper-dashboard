## Label Justifications

- `P2`: This is a normal-priority exec runtime bug with real but bounded impact on agent commands that use explicit path-like workdirs.
- `impact:other`: The issue affects the exec tool's command working directory, which is maintainer-visible runtime behavior outside the more specific owned impact labels.

## Summary

Keep open: current main and v2026.6.8 still pass the literal local `workdir` into `statSync()`, warn, and continue with a fallback directory before execution; several open linked PRs are candidate fixes but none has merged.

## What This Changes

The issue asks OpenClaw exec to avoid running commands from a fallback directory when an explicit `workdir` such as `~/...` is unavailable or unexpanded.

## Best Possible Solution

Keep this issue as the canonical report while maintainers choose and land one narrow exec fix, with regression coverage proving explicit invalid local workdirs do not execute from an unintended fallback directory.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a local `exec` call with `workdir: "~/..."` reaches `resolveWorkdir()`, stats the literal value, returns a fallback, and the process runner uses that fallback cwd.

## Solution Assessment

Unclear which exact fix shape is best because the open PRs differ on fail-closed versus tilde expansion semantics. The best triage action is to keep this canonical issue open until maintainers pick, land, or reject one candidate.
