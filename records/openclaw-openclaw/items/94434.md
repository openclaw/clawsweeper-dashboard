## Summary

Keep open: current main and v2026.6.9 still resolve local `exec.workdir` by statting the literal path and falling back before execution, while two open mergeable PRs with closing syntax propose different semantics for the same bug.

## What This Changes

The issue requests that local `exec` stop running commands from a fallback directory when an explicit `workdir` such as `~/...` is unavailable or unexpanded.

## Best Possible Solution

Land one canonical exec workdir fix with regression coverage, keeping sandbox and node routing separate and making the maintainer-chosen local semantics explicit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: local `exec` with `workdir: "~/..."` reaches `resolveWorkdir()`, stats the literal value, and returns a fallback before command execution.

## Solution Assessment

Unclear which final semantics are best without maintainer choice. The best implementation should either fail explicit invalid workdirs before execution or expand leading `~` with the chosen host-home contract, then cover that behavior with focused tests.
