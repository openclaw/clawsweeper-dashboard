## Summary

Keep open: current main still has the reported local exec behavior, and two open linked PRs already claim to close this issue with different semantics. The maintainer cleanup path is to choose and land or reject the canonical fix before closing the user report.

## What This Changes

The issue reports that explicit local `exec.workdir` values beginning with `~` are treated literally, fall back to another cwd, and still allow the command to run from that fallback directory.

## Best Possible Solution

Land one canonical local `exec.workdir` contract with focused regression coverage that proves commands do not run from the wrong fallback directory, then close this issue through the merged PR.

## Reproduction Assessment

Yes. Source inspection shows a literal `~/...` local workdir misses `statSync`, `resolveWorkdir()` returns a fallback cwd, and the exec path later passes that cwd to the command runner; I did not run the mutating full exec path in this read-only review.

## Solution Assessment

Unclear. Failing invalid explicit workdirs is the strongest guard against wrong-directory execution, while leading-home expansion directly addresses the tilde input; maintainers need to choose the intended local path contract before merge.
