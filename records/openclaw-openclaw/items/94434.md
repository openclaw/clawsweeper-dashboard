## Summary

Keep open: current main and v2026.6.8 still stat the literal local `exec.workdir` and fall back before command execution, while several open PRs now reference this issue with closing syntax. The right cleanup is to land or reject one canonical fix before closing the user report.

## What This Changes

The issue reports that explicit local `exec.workdir` values beginning with `~` are treated literally, trigger a fallback cwd, and still run the command.

## Best Possible Solution

Land one canonical exec workdir fix with regression coverage, keeping sandbox and node routing separate and making maintainer-chosen local semantics explicit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: local `exec` with `workdir: "~/..."` reaches `resolveWorkdir()`, stats the literal value, and returns the fallback before execution.

## Solution Assessment

Unclear which final semantics are best without maintainer choice, but keeping the issue open is correct. The best implementation should either fail explicit invalid workdirs before execution or expand leading `~` with the chosen host-home contract, then cover that behavior with focused tests.
