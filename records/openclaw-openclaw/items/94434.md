## Summary

Keep open: current main and the latest release still stat the literal local `exec.workdir` and fall back before command execution, while two open closing PRs already target the bug with different semantics. The issue remains necessary until maintainers choose and land or reject one canonical fix.

## What This Changes

The issue reports that an explicit `exec.workdir` beginning with `~` is treated literally, falls back to another directory, and still executes the command from that fallback.

## Best Possible Solution

Land one canonical exec workdir fix with focused regression coverage, keeping sandbox and node routing separate and making the maintainer-chosen local semantics explicit.

## Reproduction Assessment

Yes. Source inspection shows local gateway exec reaches `resolveWorkdir()` with the explicit workdir, and the resolver falls back after `statSync(workdir)` fails on a literal `~/...` path.

## Solution Assessment

Unclear. The bug is real, but maintainers still need to choose fail-fast explicit workdir semantics, leading OS-home expansion, or a combined expansion-plus-fail-fast behavior before one linked PR lands.
