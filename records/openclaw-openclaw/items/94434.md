## Summary

Keep open: current main and the latest release still use the literal `workdir` value for local exec resolution and can fall back before command execution. Four open PRs now reference this issue with closing syntax, so the right cleanup is to review and land or reject one canonical fix before closing the report.

## What This Changes

The issue reports that `exec.workdir` values beginning with `~` are treated literally, causing commands to run from a fallback directory instead of the intended path.

## Best Possible Solution

Land one narrow exec fix that either expands leading home paths or fails explicit invalid workdirs before command execution, with regression coverage for gateway/local exec.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a local `exec` call with `workdir: "~/..."` reaches `resolveWorkdir()`, stats the literal value, and returns the fallback. I did not run the live command because this review is read-only.
