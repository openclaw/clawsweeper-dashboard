## Summary

Keep open: current main and v2026.5.18 still implement a fixed renewable inbound debounce window, while the requested adaptive increment/max-cap timing is absent and needs maintainer product/config approval. The closed renewable debounce issue covers same-sender batching, but it does not supersede this narrower adaptive timing request.

## What This Changes

The issue requests adaptive inbound debounce timing that starts shorter, grows when additional messages arrive in the debounce window, and caps at a configurable maximum.

## Best Possible Solution

Add an explicit opt-in adaptive inbound debounce mode to the shared messages.inbound contract, preserving fixed-window behavior by default and updating schema, docs, and focused debouncer/channel tests.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing contract. Source inspection gives high confidence that current behavior is fixed renewable debounce with no accepted adaptive increment or cap fields.

## Solution Assessment

Unclear until maintainers approve the public timing policy. The maintainable path is an explicit opt-in shared config/API contract, not replacing the existing fixed debounce semantics implicitly.
