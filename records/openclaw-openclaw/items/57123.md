## Summary

Keep open: current main and the latest release still expose fixed renewable inbound debounce timing, while the requested adaptive increment/max-cap behavior is absent and needs maintainer config/product approval. Related batching issues cover renewable concatenation, media aggregation, semantic aggregation, or Feishu-specific behavior, but none supersede this narrower adaptive timing request.

## What This Changes

The issue requests adaptive inbound debounce timing with an initial delay, per-message increment, and max cap under `messages.inbound`.

## Best Possible Solution

Add an explicit adaptive inbound debounce mode to the shared `messages.inbound` contract, preserving fixed-window behavior by default and updating schema, metadata/docs, and focused debounce tests.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken existing behavior. Source inspection shows fixed renewable debouncing and no accepted adaptive increment or cap fields on current main.

## Solution Assessment

Unclear until maintainers approve the public timing policy. The maintainable path is a documented shared config/API contract rather than treating the existing fixed debouncer as sufficient.
