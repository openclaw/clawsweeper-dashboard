## Summary

Keep open: current main and v2026.6.8 still expose fixed-window inbound debounce only, while this asks for adaptive increment/max and typing-aware timing, which is a feature/config product decision.

## What This Changes

Add an opt-in adaptive inbound debounce policy where same-sender burst windows grow with additional messages and cap at a maximum, possibly informed by typing-state signals.

## Best Possible Solution

Preserve fixed debounce as the default and design a minimal opt-in adaptive inbound debounce API, with docs/tests and a clear channel typing-signal boundary, before implementation.
