## Summary

Keep open: current main still exposes fixed renewable inbound debounce timing, and the requested adaptive/incremental policy with increment and cap is still absent. The related renewable-concat issue was closed as implemented, but the discussion explicitly distinguishes this as a complementary latency policy rather than a duplicate.

## What This Changes

The issue requests an adaptive inbound message debounce policy where same-sender bursts increase the wait time up to a configurable cap.

## Best Possible Solution

Add a documented adaptive inbound debounce mode to the shared inbound debounce contract while preserving fixed-window behavior by default, with schema/generated metadata/docs and channel-agnostic regression tests.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken existing behavior. Source inspection on current main shows fixed renewable debouncing and no accepted adaptive increment/cap fields.

## Solution Assessment

Unclear until maintainers approve the public timing policy. A maintainable solution would add adaptive behavior behind an explicit shared config/API contract rather than treating the existing fixed debouncer as sufficient.
