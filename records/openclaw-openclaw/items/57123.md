Keep open: current main and v2026.6.10 only support renewable fixed-window same-sender inbound debounce; adaptive/incremental timing remains unimplemented and needs maintainer config/API direction.

## What This Changes

Request to add adaptive inbound debounce timing with an initial delay, per-message increment, max cap, and possible typing-aware behavior.

## Best Possible Solution

Keep the fixed same-sender debounce baseline and design one generic opt-in adaptive timing policy before implementation, with related semantic aggregation and Telegram bundle-metadata work kept in their narrower trackers.
