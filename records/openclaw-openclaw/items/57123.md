## Summary

Keep open: current main and v2026.5.19 still implement a fixed renewable inbound debounce window, while adaptive increment/max-cap timing is absent and requires maintainer product/config approval. The related semantic aggregation and renewable debounce threads are useful context, but they do not fully supersede this narrower deterministic adaptive timing request.

## What This Changes

Requests an adaptive `messages.inbound` debounce that starts with a short wait, grows for rapid same-sender bursts, and caps at a configurable maximum.

## Best Possible Solution

Add an explicit opt-in adaptive inbound debounce mode to the shared `messages.inbound` contract, preserving fixed-window behavior by default and updating schema, docs, and focused debouncer/channel tests.

## Reproduction Assessment

Not applicable; this is a feature/config request rather than a broken existing contract. Source inspection gives high confidence that current main and v2026.5.19 still use fixed renewable debounce with no accepted adaptive increment or cap fields.

## Solution Assessment

Unclear until maintainers approve the public timing policy. The maintainable path is an explicit opt-in shared config/API contract, not silently replacing existing fixed debounce semantics.
