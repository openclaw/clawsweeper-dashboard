## Summary

Keep open: current main and the latest release still expose fixed renewable inbound debounce timing, while the requested adaptive increment and max-cap policy is absent and needs maintainer config/product approval.

## What This Changes

Request adaptive inbound debounce where the delay starts short, grows for rapid message bursts, and caps at a configurable maximum.

## Best Possible Solution

Add an explicit adaptive inbound debounce mode to the shared inbound debounce contract, preserving fixed-window behavior by default and updating schema, generated metadata, docs, and channel-agnostic regression tests.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken existing behavior. Source inspection shows fixed renewable debouncing and no accepted adaptive increment or cap fields on current main.

## Solution Assessment

Unclear until maintainers approve the public timing policy. The maintainable path is a documented shared config/API contract rather than treating the existing fixed debouncer as sufficient.
