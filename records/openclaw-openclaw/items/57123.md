## Summary

Keep open: current main and v2026.6.10 still implement fixed same-sender inbound debounce only, while adaptive increments, max-window caps, and typing-aware flush behavior remain unimplemented product/config work.

## What This Changes

Requests an opt-in adaptive inbound debounce policy that starts with a shorter wait, extends on subsequent messages, and caps the total delay.

## Best Possible Solution

Keep the fixed debounce baseline and design one generic, opt-in adaptive inbound policy with cap/typing hooks, schema/help/docs, and focused ordering tests before implementation.
