## Summary

Keep open. Current main still provides fixed, renewable inbound debounce windows, while this issue asks for adaptive timing with a first-message delay, per-message growth, and a cap. The related #55835 work covers renewable concatenation and does not supersede the separate adaptive latency policy the reporter described.

## What This Changes

The issue requests adaptive inbound debounce timing with an initial delay, per-message increment, and maximum cap instead of one fixed renewable debounce window.

## Best Possible Solution

Add an opt-in adaptive inbound debounce policy to `messages.inbound`, keep the existing fixed-window behavior backward compatible, update config types/schema/generated metadata/docs, compute the delay from same-key burst state with a hard cap, and cover fixed plus adaptive behavior with focused debouncer and channel-wrapper tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The absence is verifiable on current main by inspecting the shared debouncer, strict config schema, generated schema, and Messages docs, all of which expose fixed windows only.

## Solution Assessment

Unclear until maintainers choose the public config contract and defaults. The narrowest maintainable path appears to be an additive, opt-in adaptive policy in the generic inbound debounce layer rather than channel-specific one-offs.
