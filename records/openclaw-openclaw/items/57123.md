## Summary

Keep open: current main and v2026.5.22 still expose fixed renewable inbound debounce only; adaptive increment/max timing is not implemented and would change public config plus cross-channel turn grouping. The broader semantic aggregation issue references this focused request rather than superseding it.

## What This Changes

The issue requests adaptive inbound debounce timing that starts with a short wait, grows for successive messages in the same burst, and caps at a maximum delay.

## Best Possible Solution

Add an explicit opt-in adaptive inbound debounce contract in messages.inbound, preserve fixed-window behavior by default, and update schema, docs, and focused debouncer/channel tests after maintainer approval.

## Reproduction Assessment

Not applicable; this is a feature/config request rather than a broken existing contract. Source, schema, docs, and v2026.5.22 inspection give high confidence that adaptive increment/max timing is absent.

## Solution Assessment

Unclear until maintainers approve the public timing policy. The maintainable direction is an explicit opt-in config/API contract, not silently replacing fixed debounce semantics.
