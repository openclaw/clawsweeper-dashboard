## Summary

Keep open: current main and v2026.6.6 still implement fixed receive-time same-sender inbound batching, while this request asks for new adaptive/max-window timing and optional typing-state semantics that need maintainer product/API direction.

## What This Changes

Request adaptive inbound debounce timing that starts short, grows during same-sender message bursts, caps the wait, and may later use typing-state signals.

## Best Possible Solution

Keep fixed debounce as the default, then add a minimal opt-in adaptive/max-window policy in the shared inbound debounce surface only after maintainers choose the config shape and per-channel typing-signal boundary.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing contract. Source inspection confirms the current behavior is fixed-window same-sender batching, not adaptive debounce.

## Solution Assessment

Unclear; adaptive debounce is plausible, but the best solution depends on maintainer choices for defaults, config keys, cap semantics, and whether typing-state signals belong in the same policy.
