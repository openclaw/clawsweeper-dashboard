## Summary

Keep open: current main still provides a fixed, renewable inbound debounce window, while the requested adaptive timing policy with increment and cap remains absent. Related #55835 addressed renewable debounce with concatenation, but it does not supersede this separate UX request.

## What This Changes

The issue requests an adaptive inbound message debounce policy where same-sender bursts increase the wait time up to a configurable cap.

## Best Possible Solution

Add a documented adaptive inbound debounce mode to the shared inbound debounce contract while preserving fixed-window behavior by default, with schema/generated metadata/docs and channel-agnostic regression tests.

## Reproduction Assessment

Not applicable; this is a feature request rather than a failing current behavior. Source inspection on current main shows only fixed renewable debounce behavior, and targeted search found no adaptive fields or implementation.

## Solution Assessment

No; treating the existing fixed debouncer as sufficient would miss the requested latency policy. The maintainable path is a maintainer-approved config/API design followed by focused core debounce, schema, docs, and test changes.
