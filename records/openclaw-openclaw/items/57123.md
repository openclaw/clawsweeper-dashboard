## Summary

Keep open: current main and the latest release still implement fixed-window same-key inbound debounce, but not adaptive increment/max timing or typing-aware flush signals. The related broader semantic-aggregation issue explicitly points adaptive timing back to https://github.com/openclaw/openclaw/issues/57123, so this remains the focused product/API decision rather than a cleanup close.

## What This Changes

Request adaptive inbound debounce that starts with a shorter wait, grows for subsequent messages in the same burst, and caps the maximum wait, with a related typing-state extension idea.

## Best Possible Solution

Keep the shipped fixed debounce default, then design a minimal opt-in adaptive policy at the shared inbound debounce/SDK boundary with schema, docs, and channel tests once maintainers choose the config shape and typing-signal boundary.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing contract. Source inspection gives high confidence that current behavior is fixed-window same-key batching, not adaptive debounce.

## Solution Assessment

Unclear: adaptive debounce is plausible, but adding the requested fields exactly is not proven as the best shape until maintainers choose defaults, max-window semantics, and whether typing-state events belong in the same policy.
