## Summary

Keep open: current main and v2026.5.20 still expose fixed renewable inbound debounce only; adaptive increment/max timing would change public config and cross-channel turn grouping, so it needs maintainer product/API approval rather than cleanup closure.

## What This Changes

The issue requests an adaptive inbound debounce window that starts short, grows when more same-burst messages arrive, and caps at a configured maximum delay.

## Best Possible Solution

Add an explicit opt-in adaptive inbound debounce mode in the shared messages.inbound contract, preserving fixed-window behavior by default and updating schema, docs, and focused debouncer/channel tests.

## Reproduction Assessment

Not applicable; this is a feature/config request rather than a broken existing contract. Source, schema, docs, and v2026.5.20 inspection give high confidence that adaptive increment/max timing is absent.

## Solution Assessment

Unclear until maintainers approve the public timing policy. The maintainable direction is an explicit opt-in config/API contract, not silently replacing fixed debounce semantics.
