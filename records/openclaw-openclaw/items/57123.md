## Summary

Keep open: current main and the latest release already batch fixed-window same-sender inbound messages, but they do not implement adaptive increment/max timing or typing-aware flush signals, so the remaining request is a real feature/config product decision.

## What This Changes

Requests adaptive inbound debounce that starts with a short wait, extends for repeated same-sender messages, caps the maximum wait, and may later use channel typing-state signals.

## Best Possible Solution

Preserve the shipped fixed debounce default and design a minimal, opt-in adaptive policy only after maintainers choose the config/API shape and channel typing-signal boundary.
