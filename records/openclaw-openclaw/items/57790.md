## Summary

Keep open. Current main still lacks the requested gateway shutdown/restart path that walks active sessions and fires per-session `session_end`; the existing shutdown implementation fires gateway-level lifecycle hooks and `gateway_stop`, while `session_end` remains wired to reset/delete/replacement/compaction paths. I also found no open implementation PR referencing this issue.

## What This Changes

The issue requests that gateway shutdown or restart fire `session_end` hooks for all active sessions before exit so downstream lifecycle consumers do not retain ghost active sessions.

## Best Possible Solution

Add a gateway-owned, bounded shutdown/restart session finalization step that discovers active session entries at close time, emits typed `session_end` with documented shutdown/restart reasons, and updates hook types, docs, and focused tests while keeping `gateway_stop` as the separate plugin-service lifecycle hook.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to register a plugin `session_end` hook, create an active session, then send SIGTERM/SIGINT/SIGUSR1; current main proceeds through `server.close` and gateway lifecycle hooks without any active-session walk or `runSessionEnd` call. I did not run a live gateway process because this review was read-only.

## Solution Assessment

Yes, the requested direction is the right fix boundary: the Gateway owns process shutdown/restart and is the only layer that can reliably emit per-session finalization before teardown. Using `gateway_stop` as a substitute would not satisfy downstream consumers that depend on typed session lifecycle events.
