## Summary

Keep this PR open: the OOM hardening target is valid, but the patch is not merge-ready because it duplicates the existing shared provider JSON reader, currently fails lint, and lacks after-fix real behavior proof.

## What This Changes

The PR caps fal image, music, and video JSON success-response reads at 1 MiB by replacing `response.json()` with `readResponseWithLimit` plus `JSON.parse`.

## Best Possible Solution

Use the shared `readProviderJsonResponse` helper with an explicit fal JSON byte cap, preserve fal-specific malformed-response behavior where needed, and add focused oversized-stream proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main calls `await response.json()` on fal image, music, and video success responses, so an oversized streamed 200 OK body would be drained before parsing. I did not run a live fal or local harness reproduction in this read-only review.

## Solution Assessment

No, not as written: bounding the reads is the right fix direction, but the best OpenClaw shape is to route through the existing provider JSON helper instead of creating a second inline JSON-reader policy in fal.
