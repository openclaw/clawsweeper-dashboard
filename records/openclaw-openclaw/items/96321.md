## Summary

Keep open: the hardening target is valid, but the current branch is not merge-ready because it duplicates the shared provider JSON helper, fails lint, and lacks real behavior proof for the new cap.

## What This Changes

The PR changes fal image, music, and video success-response JSON reads to use bounded 1 MiB reads before JSON parsing and updates fal response mocks in tests.

## Best Possible Solution

Route fal success JSON through `readProviderJsonResponse(..., { maxBytes })`, preserve fal-specific malformed-response behavior where needed, and add redacted real behavior proof for oversized and under-cap fal JSON paths before merge.

## Reproduction Assessment

Yes, source-reproducible: current main calls `await response.json()` on fal image, music, and video success responses, so oversized streamed 200 OK bodies would be drained before parsing. I did not run a live fal request or local harness in this read-only review.

## Solution Assessment

No, not as written: bounding the reads is the right direction, but the best OpenClaw shape is to route through the existing provider JSON helper instead of creating a second inline JSON-reader policy in fal.
