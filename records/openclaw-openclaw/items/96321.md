## Summary

Keep open: the hardening need is real, but this PR is not merge-ready because it duplicates the shared provider JSON reader, fails lint in the new video catch path, and lacks contributor real-behavior proof for the fal cap.

## What This Changes

The branch changes fal image, music, and video generation success-response JSON reads from `response.json()` to bounded `readResponseWithLimit` reads with a 1 MiB cap.

## Best Possible Solution

Use `readProviderJsonResponse(..., { maxBytes })` for the fal success JSON paths, keep fal-specific malformed-response behavior where required, and add focused oversized-stream and under-cap proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main calls `await response.json()` on fal image, music, and video success responses, so an oversized streamed 200 OK body would be drained before parsing. I did not run a live fal request or local harness in this read-only review.

## Solution Assessment

No, not as written: bounding the reads is the right direction, but the best OpenClaw shape is to route through the existing provider JSON helper instead of creating a second inline JSON-reader policy in fal.
