## Label Justifications

- `P1`: A Codex startup hang can hold an agent lane for the full turn timeout and break active agent workflows for operators.
- `merge-risk: 🚨 compatibility`: The new 120s default can fail an existing slow startup that previously had the full turn budget to initialize.
- `merge-risk: 🚨 availability`: The patch changes startup watchdog timing for the Codex app-server path, which directly affects whether agent lanes hang, fail fast, or restart.

## Summary

Keep open: current main still binds Codex app-server startup to the per-turn budget, and the PR is a focused fix, but merge should wait for after-fix real runtime proof and maintainer acceptance or tuning of the new 120s default.

## What This Changes

The branch adds Codex `appServer.startupTimeoutMs` config/manifest/docs/runtime wiring and regression tests so startup watchdog timing is independent of the per-turn timeout.

## Best Possible Solution

Land the focused timeout split after real after-fix runtime proof is supplied and maintainers accept or tune the 120s default, while keeping the separate runtime-deps and shared-client recovery work in their own PRs.

## Reproduction Assessment

Yes at source level: current main feeds `params.timeoutMs` into the Codex startup watchdog, and the watchdog rejects only after that computed timeout. The PR body also supplies before-fix operator logs showing approximately 30-minute lane holds.

## Solution Assessment

Yes on code shape: a Codex app-server startup ceiling belongs in the app-server runtime config and call site. Merge readiness is still blocked by after-fix runtime proof and maintainer acceptance or tuning of the new default.
