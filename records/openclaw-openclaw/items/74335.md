## Summary

Keep open. Current main still has the reported Codex app-server startup timeout coupling, while the PR is a narrow, CI-green fix; it needs conflict/rebase handling before it can merge.

## What This Changes

The PR adds `appServer.startupTimeoutMs` to Codex config, manifest, docs, runtime options, and tests, then uses it instead of the per-turn timeout for Codex app-server startup.

## Best Possible Solution

Update the PR branch onto current main, preserve the separate startup, turn, and request timeout semantics, then land the narrow Codex extension change after targeted Codex app-server validation and CI are green on the rebased head.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: a never-resolving app-server client factory with `params.timeoutMs = 30 * 60_000` would leave startup bounded by the 30-minute turn timeout because that value is passed directly to `withCodexStartupTimeout`.

## Solution Assessment

Yes. The PR uses the narrow maintainable fix: add a resolver-backed optional `startupTimeoutMs`, expose it in docs/manifest, and change only the startup wrapper call site while leaving turn and control-plane request timeouts alone.
