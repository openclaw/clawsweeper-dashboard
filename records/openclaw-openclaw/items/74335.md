## Summary

Keep open. Current main still couples Codex app-server startup to the per-turn timeout, and this PR is a narrow additive fix with matching docs, manifest metadata, config coverage, and regression tests; however GitHub currently reports the branch as dirty, so it needs an update before it can land.

## What This Changes

The PR adds `appServer.startupTimeoutMs` to the Codex plugin config/runtime options/docs/manifest/tests and uses it instead of the per-turn timeout for Codex app-server startup.

## Best Possible Solution

Update the contributor branch onto current main, preserve the separate startup, turn, and control-plane request timeout semantics, then land the focused Codex plugin change once targeted app-server validation and CI are green on the rebased head.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: a never-resolving app-server client factory with `params.timeoutMs = 30 * 60_000` leaves startup bounded by that 30-minute value because it is passed directly to `withCodexStartupTimeout`.

## Solution Assessment

Yes. The PR uses the narrow maintainable solution: add a resolver-backed optional startup timeout and change only the startup wrapper call site, without changing per-turn or control-plane request timeout semantics.
