Keep open. Current main still couples Codex app-server startup to the per-turn timeout, and this PR is a narrow, additive fix; the remaining blocker is branch freshness/mergeability and validation on a rebased head.

## What This Changes

The branch adds optional Codex `appServer.startupTimeoutMs` config/defaults/docs/manifest coverage, uses it for app-server startup, and adds regression tests plus a changelog entry.

## Best Possible Solution

Update the PR onto current main, preserve separate startup/turn/request timeout semantics, then land after the focused Codex app-server tests, extension typecheck, and changed gate pass.

## Reproduction Assessment

Yes, at source level. On current main, a never-resolving app-server startup with `params.timeoutMs = 30 * 60_000` is bounded by that same per-turn value because it is passed directly into `withCodexStartupTimeout`.

## Solution Assessment

Yes. The proposed direction is the narrow maintainable fix: resolve a startup-specific timeout, use it only around app-server startup, and leave per-turn and control-plane request timeouts unchanged.
