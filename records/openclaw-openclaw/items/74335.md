## Summary

Keep open. Current main still couples Codex app-server startup to the per-turn timeout, and this PR is the right narrow fix, but the branch is currently dirty/unmergeable against main and needs an update plus validation before merge.

## What This Changes

The PR adds optional Codex `appServer.startupTimeoutMs` config/schema/manifest/docs coverage, uses it for app-server startup, updates fixtures and regression tests, and adds a changelog entry.

## Best Possible Solution

Update the branch onto current main, keep startup, turn, and request timeout semantics separate, then land after focused Codex app-server tests, extension typecheck, and the changed gate pass.

## Reproduction Assessment

Yes, at source level. On current main, a never-resolving app-server startup with `params.timeoutMs = 30 * 60_000` is bounded by that same per-turn value because it is passed directly into `withCodexStartupTimeout`.

## Solution Assessment

Yes. The proposed direction is the narrow maintainable fix: resolve a startup-specific timeout, use it only around app-server startup, and leave per-turn and control-plane request timeouts unchanged.
