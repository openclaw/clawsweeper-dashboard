## Summary

Keep open. Current main still has the long-startup-timeout behavior, and the PR is a narrow, appropriate fix, but the branch needs normal maintainer/author update against current main before landing.

## What This Changes

The PR adds optional Codex `appServer.startupTimeoutMs` config, manifest/docs coverage, and tests, then uses that value instead of the per-turn timeout for Codex app-server startup.

## Best Possible Solution

Rebase or update the PR onto current main, preserve separate startup, turn, and request timeout semantics, then land the focused Codex plugin change once targeted validation and CI pass.

## Reproduction Assessment

Yes. A source-level reproduction is clear: on current main, a never-resolving Codex app-server client factory with `params.timeoutMs = 30 * 60_000` leaves startup bounded by the 30-minute turn budget because that exact value is passed into `withCodexStartupTimeout`.

## Solution Assessment

Yes. The proposed direction is the narrow maintainable fix: add a resolver-backed optional startup timeout and use it only around startup, without changing per-turn or control-plane request timeout behavior.
