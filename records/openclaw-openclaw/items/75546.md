## Summary

Keep this open. Current main still has a source-level cold-path failure: the CLI turn post-processing path resolves the default context engine during CLI transcript compaction without first registering built-in context engines, so the reported `Context engine "legacy" is not registered` error remains plausible and narrowly actionable.

## What This Changes

The issue reports that CLI-backed transcript persistence/compaction logs `Context engine "legacy" is not registered` after a CLI agent turn.

## Best Possible Solution

Initialize built-in context engines before the CLI compaction lifecycle resolves the active context engine, preferably at the CLI compaction boundary, and add regression coverage that exercises the real default resolver in a cold registry path without changing plugin slot semantics.

## Reproduction Assessment

Yes. A cold CLI backend turn with a persisted `sessionFile` and positive `contextTokens` reaches `runCliTurnCompactionLifecycle()`, which resolves the default `legacy` context engine before any local initialization and can hit the registry error shown in the title.

## Solution Assessment

No current-main behavior is not the best state. The narrow maintainable fix is to mirror the existing embedded compaction ordering by calling the context-engine initializer before CLI compaction resolution, plus a focused regression test.
