Keep open. Current main and the latest release still only expose SSRF/private-network browser navigation policy, not an operator URL/domain/path denylist or warnlist, and the requested surface needs maintainer product/security design before implementation.

## What This Changes

The issue requests a browser URL/domain/path denylist, and optionally a warnlist, so agents get a clear pre-navigation response for known blocked sites.

## Best Possible Solution

Define a central browser URL policy primitive with typed config, strict schema/help/docs, pre-navigation plus final-URL/redirect enforcement, warnlist response semantics, default-list ownership rules, and focused regression coverage.

Not applicable as a runtime bug reproduction; this is a feature/config request. Source inspection against current main and v2026.5.12 shows only SSRF/private-network browser policy, not denylist/warnlist fields.

## Solution Assessment
