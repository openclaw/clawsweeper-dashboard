## Summary

Keep open. Current main still only has browser SSRF/private-network navigation policy; it does not provide an operator-configurable URL/domain/path denylist or warnlist, and adding one would create a new public, security-sensitive browser config surface.

## What This Changes

Request adding `browser.blockedUrls` or `browser.denylist`, plus an optional warnlist, so browser navigation refuses or warns on operator-specified URL patterns.

## Best Possible Solution

Define a central browser URL policy primitive with typed config, strict schema/help/docs, pre-navigation and final-URL/redirect enforcement, warnlist response semantics, default-list ownership rules, and focused regression coverage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature/config request rather than broken existing behavior. Source inspection on current main shows the requested browser denylist/warnlist config and navigation policy fields are absent.

## Solution Assessment

Unclear. A centralized browser navigation policy is likely the right layer, but maintainers still need to decide key names, matching semantics, redirect handling, warnlist response shape, and default-list ownership.
