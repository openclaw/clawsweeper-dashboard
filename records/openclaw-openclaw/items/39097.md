## Summary

Keep open. Current main still lacks an operator-configurable browser URL/domain/path denylist or warnlist; the browser policy surface remains SSRF/private-network focused, and the existing April 26 ClawSweeper comment reached the same conclusion with no newer implementation visible in the checked-out main tree.

## What This Changes

The issue requests a `browser.denylist` or `browser.blockedUrls` config surface, plus optional warnlist support, so browser navigation can reject known bot-blocked URL/domain/path patterns with a clear error before wasting agent turns.

## Best Possible Solution

Add a maintainer-approved browser URL policy primitive that flows through config types, strict zod/generated schema/help, browser config resolution, all managed/raw-CDP/existing-session navigation guards, route error responses, docs, and focused tests; treat warnlist behavior and bundled default domains as explicit product decisions rather than accidental scope creep.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature request. The missing capability is high-confidence from static current-main inspection of config types, strict schema, resolver, navigation guard, routes, tests, and docs.

## Solution Assessment

Unclear. A centralized browser config plus navigation-guard implementation is the right layer, but maintainers still need to choose the public key name, pattern semantics, redirect handling, warnlist behavior, and whether default blocked domains belong in the same change.
