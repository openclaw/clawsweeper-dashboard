## Summary

Keep open. Current main still has only the browser SSRF/private-network navigation policy, with no operator-configurable browser URL/domain/path denylist or warnlist, and the requested public policy needs maintainer product/security decisions before implementation.

## What This Changes

The issue requests a browser configuration option for blocked URL/domain/path patterns, plus optional warnlist behavior, so browser navigation can fail fast with a clear message on known bot-blocked sites.

## Best Possible Solution

Define a maintainer-approved browser URL policy primitive that covers typed config, strict schema/help, docs, navigation and open-tab guards, redirect behavior, warnlist response shape, and focused regression tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature request. Source inspection on current main confirms the requested config fields and navigation policy behavior are absent.

## Solution Assessment

Unclear. A centralized browser navigation policy is likely the right layer, but maintainers still need to choose public key names, matching semantics, redirect handling, warnlist behavior, and default-list policy.
