## Summary

Keep open. Current main still has browser SSRF/private-network hostname policy, but no operator-configurable URL/domain/path denylist or warnlist; implementing this would add public browser config and security-sensitive navigation semantics.

## What This Changes

The issue requests a browser URL/domain/path denylist and optional warnlist so known-blocked sites fail before navigation with a clear browser-tool message.

## Best Possible Solution

Define a maintainer-approved browser URL policy primitive covering typed config, strict schema/help/docs, open/navigate/final-URL/redirect guards, warnlist response shape, default-list ownership, and focused regression coverage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature/config request. Source inspection on current main confirms the requested denylist/warnlist fields and navigation policy behavior are absent.

## Solution Assessment

Unclear. A centralized browser navigation policy is likely the right layer, but maintainers still need to decide key names, matching semantics, redirect handling, warnlist response shape, and default-list ownership.
