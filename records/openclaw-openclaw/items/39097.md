## Summary

Keep open. Current main and the latest release still expose browser SSRF/private-network navigation policy only; the requested denylist/warnlist is a new public, security-sensitive browser config surface that needs maintainer design before implementation.

## What This Changes

The issue requests a browser URL/domain/path denylist and optional warnlist so automated browser navigation can fail early with a clear operator-curated message.

## Best Possible Solution

Define a central browser URL policy primitive with typed config, strict schema/help/docs, pre-navigation plus final-URL/redirect enforcement, warnlist response semantics, default-list ownership rules, and focused regression coverage.

## Reproduction Assessment

Not applicable; this is a feature/config request rather than a broken existing behavior report. Source inspection of current main and v2026.5.12 shows browser navigation has SSRF/private-network policy but no denylist or warnlist fields.

## Solution Assessment

Unclear. A centralized browser navigation policy is the likely layer, but maintainers still need to decide config names, matching semantics, redirect/subframe handling, warnlist response shape, and default-list ownership.
