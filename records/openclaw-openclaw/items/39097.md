## Summary

Keep open. Current main still has browser SSRF/private-network navigation policy only, with no operator-configurable URL/domain/path denylist or warnlist, and the request adds a new public browser config and security policy surface.

## What This Changes

The issue requests a browser URL/domain/path denylist config and optional warnlist so known bot-blocking sites fail early with a clear browser tool response.

## Best Possible Solution

Define a central browser URL policy primitive with typed config, strict schema/help/docs, pre-navigation plus final-URL/redirect enforcement, warnlist response semantics, default-list ownership rules, and focused regression coverage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature/config request rather than broken existing behavior. Source inspection on current main shows the requested browser denylist/warnlist config and navigation policy fields are absent.

## Solution Assessment

Unclear. A centralized browser navigation policy is the likely layer, but maintainers still need to decide config key names, matching semantics, redirect/subframe handling, warnlist response shape, and default-list ownership.
