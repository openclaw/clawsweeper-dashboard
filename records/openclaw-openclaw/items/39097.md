## Summary

Keep open. Current main and v2026.6.10 still provide browser SSRF/private-network policy plus hostname allow exceptions, but not the requested browser blocked-URL/path denylist or warnlist; the related web-tool egress issue overlaps but does not cover the browser-specific navigation and advisory semantics.

## What This Changes

The issue requests a browser configuration surface for blocked URL or domain patterns that makes navigation fail early with a clear denial message, plus an optional warning-only list.

## Best Possible Solution

Define one maintainer-approved browser navigation policy contract in the browser plugin, with schema, docs, and tests covering host/path matching, redirects, subframes/subresources, warn versus hard-block semantics, and its relationship to broader web-tool egress policy.

## Solution Assessment

Unclear as proposed. The browser navigation guard is the likely enforcement boundary, but the best fix needs maintainer agreement on advisory hints versus hard blocking and the exact request classes covered.
