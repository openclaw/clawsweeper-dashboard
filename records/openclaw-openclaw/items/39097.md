Keep open. Current main and v2026.6.10 still provide browser SSRF/private-network policy plus hostname allow exceptions, but not the requested browser blocked-URL/path denylist or warnlist; the related web-tool egress issue overlaps but does not cover the browser-specific navigation and advisory semantics.

## What This Changes

The issue requests a browser-level blocked URL/path or domain denylist, optionally with warnings, so agents get clear feedback before navigating to known-blocked sites.

## Best Possible Solution

Define one maintainer-approved browser navigation policy contract in the browser plugin, with schema, docs, and tests covering host/path matching, redirects, subframes/subresources, warn versus hard-block semantics, and its relationship to the broader web-tool egress policy.
