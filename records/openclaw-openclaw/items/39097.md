## Summary

Keep open. Current main and v2026.6.5 have centralized browser SSRF/navigation checks and allowlist-style exceptions, but they do not implement operator deny/warn lists or URL path patterns; the related web-tool egress tracker does not fully cover browser-specific navigation semantics, so this still needs product/security design.

## What This Changes

The issue requests a browser URL/domain denylist or warnlist so navigation to known bot-blocked sites can fail early with a clear message.

## Best Possible Solution

Define an operator-owned browser navigation policy in the browser plugin guard, with explicit host/path matching, redirect and frame scope, advisory-versus-hard-block semantics, docs/schema/help updates, and no bundled third-party default list unless maintainers approve that policy.

## Reproduction Assessment

Not applicable; this requests a new browser policy/config capability rather than reporting behavior that violates an existing documented contract. Source and release checks show the requested surface is absent, not broken.

## Solution Assessment

No, not as currently specified. The browser navigation guard is the likely implementation boundary, but maintainers first need to define whether this is advisory retry avoidance, hard egress control, or both, and exactly which browser requests are covered.
