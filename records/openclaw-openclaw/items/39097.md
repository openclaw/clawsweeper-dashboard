## Summary

Keep open. Current main still exposes only the browser SSRF/private-network allowlist policy, with no operator-configurable browser URL/domain/path denylist or warnlist, and the requested public policy needs maintainer security/product decisions before implementation.

## What This Changes

The issue requests operator-configurable browser URL/domain/path deny and warning policy so known bot-blocked sites fail fast with a clear tool error.

## Best Possible Solution

Define a maintainer-approved browser URL policy primitive that covers config types, strict schema/help, docs, navigation/open-tab guard call sites, redirects, and focused regression tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature request. Static inspection of current main confirms the requested config fields and navigation policy behavior are absent.

## Solution Assessment

Unclear. A centralized browser policy implemented through config resolution and navigation guards is the likely right layer, but maintainers still need to choose public key names, matching semantics, redirect handling, warnlist behavior, and default-list policy.
