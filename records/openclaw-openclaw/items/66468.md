## Summary

Keep this issue open. Current main still has the reported stale-status path: model fallback knows each attempted provider/model, but that in-flight identity is not emitted to WebSocket clients and the CLI TUI footer still renders only `sessionInfo` model fields populated from the session row after completion.

## What This Changes

The issue asks for the TUI footer to update in real time with the provider/model currently being attempted during gateway model fallback.

## Best Possible Solution

Add a typed live fallback-attempt status path that is emitted immediately before each actually attempted candidate, carries runId/sessionKey/provider/model/attempt/total, is broadcast through the existing agent event path, and is consumed by the CLI TUI with session/run filtering to update the footer before `sessions.list` catches up.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a fallback chain, force the primary candidate to fail, send from the TUI, and the current code has no live provider/model event while the footer reads only `sessionInfo` populated from `sessions.list` and post-run session-store updates.

## Solution Assessment

Unclear as written. Emitting a live model-switch/fallback-attempt event and consuming it in the TUI is the right direction, but the safest fix should include sessionKey/run filtering and should align the event shape with the existing agent lifecycle fallback data where possible.
