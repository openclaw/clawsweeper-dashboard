## Summary

Keep open. Current main still has the source-level gap described in the report: fallback candidate identity is available during the loop and recorded diagnostically, but no live model-attempt event updates the TUI footer before the run completes.

## What This Changes

The issue asks the TUI footer to show the provider/model currently being attempted while model fallback cycles through candidates.

## Best Possible Solution

Add a live session/run-scoped active fallback-attempt model signal and have the TUI footer consume it only for the visible session, without changing fallback ordering, cooldown, or provider-selection semantics.

## Reproduction Assessment

Yes, at source level. A fallback chain with a forced primary failure reaches runWithModelFallback with candidate provider/model data available, while the TUI footer still reads sessionInfo fields and has no live fallback-model handler.

## Solution Assessment

Yes. A narrow live fallback-attempt event plus a session-scoped TUI footer handler is the maintainable fix; changing fallback policy or adding broader per-reply attribution would be unnecessary scope.
