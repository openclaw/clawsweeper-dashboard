## Summary

Keep open. Current main still has a source-level gap: fallback candidate identity is available and recorded for diagnostics, but no live session-scoped model-attempt signal reaches the TUI footer during the fallback loop.

## What This Changes

The issue asks for the TUI footer to show the provider/model currently being attempted during model fallback instead of stale session model fields.

## Best Possible Solution

Add a live session/run-scoped active fallback-attempt model signal and have the TUI footer consume it only for the visible session, without changing fallback ordering, cooldown, or provider-selection semantics.

## Reproduction Assessment

Yes, at source level. Configure a fallback chain and force the first candidate to fail: the fallback loop has candidate provider/model data, while the TUI footer still renders sessionInfo fields and has no live fallback-model event handler.

## Solution Assessment

Yes. A narrow live fallback-attempt event plus a session-scoped TUI footer handler is the maintainable fix; changing fallback policy or adding a broader per-reply attribution feature would be unnecessary scope.
