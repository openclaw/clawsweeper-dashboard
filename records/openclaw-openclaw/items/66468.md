## Summary

Keep open. Current main still has the stale footer path: the fallback loop knows each candidate, but the agent bridge only records fallback steps to trajectory, and the TUI footer still renders the last `sessionInfo` provider/model rather than an in-flight fallback attempt.

## What This Changes

Request to update the TUI footer in real time as provider fallback attempts each provider/model candidate.

## Best Possible Solution

Add a first-class live fallback-attempt status event from the runtime path, carry `runId`, `sessionKey`, provider, model, attempt, and total for actually attempted candidates, and have the CLI TUI update the footer only when the event belongs to the active session/run.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a fallback chain, force an early candidate to fail, and the code has no live provider/model event while the footer reads only `sessionInfo` fields refreshed from post-run session state.

## Solution Assessment

Yes, with one refinement. The requested live model-switch event is the narrow maintainable fix, but it should include session/run filtering data and avoid cross-session updates rather than blindly updating the footer before the normal run filter.
