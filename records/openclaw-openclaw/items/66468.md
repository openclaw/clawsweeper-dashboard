## Summary

Keep open. Current main still has the stale footer path: fallback candidate identity is available inside the runtime loop, but it is only recorded internally and the TUI footer still reads session-list model fields that update after a run completes.

## What This Changes

Request live TUI footer updates for the provider/model currently being attempted during model fallback.

## Best Possible Solution

Add a live, session-scoped fallback-attempt model event from the agent fallback path and update the TUI footer only for the active session/run.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a fallback chain, force the first candidate to fail, and current main has no live provider/model event while the footer reads sessionInfo fields updated from post-run session state.

## Solution Assessment

Yes, with scoping. The requested live model-switch/fallback event is the narrow maintainable fix, but it should carry sessionKey/runId and be filtered before mutating the TUI footer.
