## Summary

Keep open. Current main still records fallback attempts diagnostically and the latest release has the same behavior, but the TUI footer still renders from sessionInfo and has no live fallback-attempt model handler.

## What This Changes

The issue asks for the TUI status footer to show the provider/model currently being attempted during model-provider fallback instead of a stale configured or final session model.

## Best Possible Solution

Add a live session/run-scoped active fallback-attempt model signal and have the TUI footer consume it for the visible session without changing fallback ordering, cooldown, or provider-selection semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: fallback candidate provider/model data is available before each attempt, while the TUI footer still reads sessionInfo and lacks a live fallback-attempt model handler.

## Solution Assessment

Is this the best way to solve the issue? Yes: a narrow live fallback-attempt event plus a session-scoped TUI footer handler is the maintainable fix; broader per-message model attribution or fallback-policy changes are unnecessary for this bug.
