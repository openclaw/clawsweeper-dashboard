## Summary

Keep open. Current main still has the source-level gap described in the report: fallback candidate provider/model identity is available in the runner, but no live session-scoped fallback-attempt event reaches the TUI footer before the run finishes.

## What This Changes

The issue requests real-time TUI footer updates for the provider/model currently being attempted during model fallback.

## Best Possible Solution

Add a live session-scoped fallback-attempt model signal and have the TUI footer consume it only for the visible session, without changing fallback ordering, cooldown, or provider-selection semantics.

## Reproduction Assessment

Yes, at source level. A fallback chain exposes each candidate in runWithModelFallback, while the TUI footer still renders sessionInfo fields and has no live fallback-model event path before terminal refresh.

## Solution Assessment

Yes. A narrow live fallback-attempt event plus a session-scoped TUI handler is the maintainable fix; broader fallback policy or persistent model-selection changes would be unnecessary scope.
