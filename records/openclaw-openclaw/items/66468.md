## Summary

Keep open. Current main still has the stale TUI footer path: fallback candidate identity is available in the runtime loop and support/trajectory surfaces, but the TUI footer still renders sessionInfo model fields and has no live fallback-model handler.

## What This Changes

The issue asks for the TUI footer to update live to the provider/model currently being attempted during model fallback.

## Best Possible Solution

Add a live, session-scoped fallback-attempt model signal from the agent fallback loop and update the TUI footer only when that event belongs to the visible session/run.

## Reproduction Assessment

Yes, at source level. Configure a fallback chain and force the first candidate to fail: current main records fallback identity internally, while the TUI footer still reads sessionInfo.modelProvider/model and has no live fallback-model event handler.

## Solution Assessment

Yes, with scoping. A live session/run-scoped fallback model event plus a TUI footer handler is the narrow maintainable fix; avoid changing fallback ordering, cooldown policy, or provider selection semantics.
