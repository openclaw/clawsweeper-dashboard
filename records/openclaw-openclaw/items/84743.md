## Summary

Keep open: current `main` and the latest release still have the installer shell logic reported here, while the TypeScript npm helper already uses the safer raw-npmrc detection that the shell installers are missing.

## What This Changes

The issue reports that hosted installer npm calls can still emit `--before` when a raw npmrc `min-release-age` policy is present, causing npm to reject the install.

## Best Possible Solution

Port the raw npmrc freshness-mode detection from the TypeScript helper into the hosted installer scripts, then add script-level regression coverage for raw `min-release-age` npmrc files so the installer never emits `--before` for that policy shape.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: current installer code can select `--before` from npm's derived `before` value while the raw npmrc still contains `min-release-age`, and npm 11.12.1 marks those configs mutually exclusive. I did not run a live global install because that would mutate the machine.

## Solution Assessment

Yes, the best fix is to reuse or mirror the existing raw npmrc detection semantics in the shell installers instead of adding another npm fallback stack. The TypeScript helper already defines the maintainable behavior boundary.

## Repair Work Prompt

Fix the installer npm freshness bypass for https://github.com/openclaw/openclaw/issues/84743. Current `scripts/install.sh` and `scripts/install-cli.sh` can choose `--before=<now>` when npm derives `before` from a raw npmrc `min-release-age`, which npm rejects because `before` and `min-release-age` are mutually exclusive. Mirror the raw npmrc detection semantics from `src/infra/npm-install-env.ts`: raw `min-release-age` policies should use `--min-release-age=0`, raw `before` policies should use current `--before`, and unset policies should keep the safe default. Check whether `scripts/install.ps1` needs the same fix before deciding scope. Add focused script tests covering raw npmrc/userconfig min-release-age behavior; update docs/changelog only if the user-visible installer behavior changes beyond the documented promise. Do not change package manager defaults or broad installer flow.
