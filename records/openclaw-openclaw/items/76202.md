## Summary

Keep open. Current main appears to gate the reconcile GET when native commands are disabled, but it still awaits a Discord application-command clear request and the docs intentionally describe clear-on-disable behavior, so this is not safe to close as fixed.

## What This Changes

Request that `channels.discord.commands.native=false` skip the Discord application-command lifecycle entirely at startup.

## Best Possible Solution

Define the disabled-native semantics first, then align code, docs, and tests around either a strict no-REST opt-out or an explicit/nonblocking cleanup path.

## Reproduction Assessment

Partial yes. Source inspection gives a high-confidence path for an awaited clear REST call when native commands are explicitly disabled, but the specific `/applications/.../commands` GET is not high-confidence on current main because reconcile deploy is gated by `nativeEnabled`.

## Solution Assessment

Unclear. Skipping all Discord command REST calls would satisfy this report, but it conflicts with the documented clear-on-disable contract; a safer design may split disabling from cleanup or move cleanup off the blocking startup path.
