## Summary

Keep open. Current main appears to gate off the reported reconcile GET when native commands resolve false, but explicit `native=false` still awaits a Discord application-command clear REST call, and docs plus the prior stale-command issue intentionally promise clear-on-disable behavior.

## What This Changes

The issue asks for `channels.discord.commands.native=false` to skip the entire Discord native slash-command REST lifecycle on startup.

## Best Possible Solution

Define disabled-native semantics first, then either split cleanup into an explicit or background path, or change `native=false` to a pure no-REST opt-out with matching docs and regression tests.

## Reproduction Assessment

Partial yes. Source inspection gives high confidence that explicit `native=false` still awaits a Discord application-command clear REST call, but the exact reported reconcile GET is not apparent on current main because deploy/reconcile is gated off.

## Solution Assessment

Unclear. Directly skipping all REST calls would satisfy this issue, but it conflicts with the documented clear-on-disable contract, so maintainers should choose the intended semantics before implementation.
