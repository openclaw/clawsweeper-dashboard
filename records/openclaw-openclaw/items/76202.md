## Summary

Keep open. Current main appears to skip the reported reconcile GET when native commands are disabled, but it still awaits a Discord application-command clear call, and the requested zero-REST behavior conflicts with the documented clear-on-disable contract and prior stale-command cleanup context.

## What This Changes

The issue asks for `channels.discord.commands.native=false` to skip the entire Discord native slash-command REST lifecycle on startup.

## Best Possible Solution

Define disabled-native semantics first, then either make clearing explicit/nonblocking or change `native=false` to a pure no-REST opt-out with matching docs and tests.

## Reproduction Assessment

Unclear for the exact reported GET, because current main gates deploy/reconcile behind `nativeEnabled=false`. Yes for the remaining startup REST behavior: explicit disable awaits `clearDiscordNativeCommands` before the logged-in path.

## Solution Assessment

Unclear. Directly skipping all REST would satisfy this report, but the safer maintainable fix depends on whether maintainers still want automatic stale-command cleanup when native commands are disabled.
