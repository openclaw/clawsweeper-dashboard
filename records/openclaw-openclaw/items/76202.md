## Summary

Keep open. Current main gates off the reported reconcile GET path when native commands resolve false, but explicit `native=false` still awaits a Discord application-command clear REST call and the current docs intentionally promise that clear-on-disable behavior, so this needs a maintainer semantics decision rather than a cleanup close.

## What This Changes

The issue requests that `channels.discord.commands.native: false` skip the Discord slash-command lifecycle entirely on startup, avoiding Discord application-command REST calls.

## Best Possible Solution

Define the disabled-native semantics, then align code, docs, and regression tests around either a true no-REST opt-out or a separate explicit/background cleanup path.

## Reproduction Assessment

Partial yes. Source inspection gives high confidence that explicit `native=false` still awaits the native-command clear REST path, but the exact reported reconcile GET is not apparent on current main because deploy/reconcile is gated off.

## Solution Assessment

Unclear. Directly skipping all command REST calls would satisfy this report, but it conflicts with the documented clear-on-disable contract, so maintainers should choose the intended semantics before implementation.
