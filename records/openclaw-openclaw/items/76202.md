## Summary

Keep open. Current main does not obviously run the reconcile GET when native commands resolve false, but it still performs an awaited Discord command-clear REST call and docs intentionally promise that behavior, so maintainers should decide whether disabled means clear or no REST before changing code.

## What This Changes

The issue requests that Discord `channels.discord.commands.native: false` skip the slash-command REST lifecycle entirely during gateway startup.

## Best Possible Solution

Define the intended disabled-native semantics first, then either make command clearing an explicit/background cleanup path or update docs and tests so `native=false` is a pure no-REST opt-out.

## Reproduction Assessment

Partial yes. Source inspection gives a high-confidence reproduction for the remaining awaited clear REST call when native commands are explicitly disabled, but the exact reported GET path is unclear on current main because deploy/reconcile is gated by `nativeEnabled`.

## Solution Assessment

Unclear. A direct skip of all REST calls would satisfy this issue, but it conflicts with the existing documented clear-on-disable contract; a safer design may split disable from cleanup or make cleanup nonblocking.
