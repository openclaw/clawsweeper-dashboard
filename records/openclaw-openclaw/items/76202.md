## Summary

Keep open. Current main skips the reconcile GET when native commands resolve false, but it still awaits a Discord application-command clear REST call while the docs and #63075 intentionally promise cleanup-on-disable, so this needs a maintainer semantics decision rather than closure.

## What This Changes

The issue requests that `channels.discord.commands.native: false` skip the entire Discord slash-command lifecycle on startup, including `GET /applications/.../commands` and cleanup REST calls.

## Best Possible Solution

Define disabled-native semantics, then implement either a pure no-REST opt-out or a separate/background cleanup mode with docs and tests aligned.

## Reproduction Assessment

Partial yes. Source inspection gives a high-confidence path for the remaining awaited cleanup REST call when native commands are explicitly disabled, but the specific reconcile GET is gated off on current main when `nativeEnabled` is false.

## Solution Assessment

Unclear. Skipping all Discord command REST would satisfy this report, but it conflicts with the current documented clear-on-disable contract; a safer fix may split disable from cleanup or make cleanup explicit/nonblocking.
