## Summary

Keep open: the bug being fixed is real and this PR moves the Codex projection boundary in the right direction, but the reserve config mapping has a correctness issue and the head is currently dirty against main.

## What This Changes

The PR scales Codex app-server context-engine projection caps from runtime context budget and compaction reserve settings, with focused tests and changelog text.

## Best Possible Solution

Land a narrowed, rebased Codex projection fix that scales from contextTokenBudget while preserving reserveTokensFloor as a floor-only guard and leaving exact accounting telemetry to https://github.com/openclaw/openclaw/issues/80765.

## Reproduction Assessment

Yes, at source level. Current main passes the runtime token budget into context-engine assembly but then renders the assembled Codex prompt through a fixed 24k-character projection cap.

## Solution Assessment

No, not as written. Budget-aware projection is the right boundary, but the PR should preserve the existing reserveTokens/reserveTokensFloor contract so floor-only configs do not remove generation/tool headroom.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve reserve headroom when only the floor is configured:** `extensions/codex/src/app-server/context-engine-projection.ts:97-99`
  - body: Returning `reserveTokensFloor` as the actual reserve changes the documented meaning of a floor-only config. If a user sets `reserveTokensFloor: 0` to disable the floor guard without also setting `reserveTokens`, this passes `0` into projection sizing and lets the expanded Codex prompt consume the full context window instead of keeping the default reply/tool headroom. Keep the default reserve unless `reserveTokens` is explicitly set, and cover the floor-only case.
  - confidence: 0.9
