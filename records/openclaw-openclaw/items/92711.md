## Summary

Keep this PR open. The latest head has repaired the earlier setup-lock problem by restoring the initial retained lock, but the new no-held-lock `releaseForPrompt()` path can now refresh the session fence without owning the write lock, and the supplied proof still does not show the reported queued follow-up delivery path working after the patch.

## What This Changes

The PR changes `src/agents/embedded-agent-runner/run/attempt.session-lock.ts` so post-prompt cleanup no longer reacquires a retained session write lock, and updates two session-lock test expectations.

## Best Possible Solution

Keep setup serialization and per-write transcript locking, but make no-held-lock prompt release verify rather than trust the fence, add a regression for the second-release window, and provide redacted real queued-follow-up proof.

## Reproduction Assessment

Yes at source level, but not by a live run here. Current main reacquires a retained session write lock after prompt completion, and the linked issue provides concrete Discord follow-up steps plus production `SessionWriteLockTimeoutError` logs.

## Solution Assessment

No. The latest PR shape is closer than the earlier revision, but the no-held-lock prompt-release branch can trust session-file state changed outside this controller; the safer fix preserves per-write locking without refreshing the fence from an unowned state.
