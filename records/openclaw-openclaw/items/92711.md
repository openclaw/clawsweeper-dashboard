## Summary

Keep open: the bug is real, but this patch is not merge-ready because it removes the prep-phase session write lock that currently serializes session repair/open/setup writes, and the PR only provides a typecheck instead of real behavior proof.

## What This Changes

The PR removes the embedded attempt controller's eager session write-lock acquisition and changes prompt reacquire/fence handling so transcript writes rely on per-write locking.

## Best Possible Solution

Keep session repair/open/prep under the session write lock, narrow only the long non-write phases, and add focused regression coverage plus redacted real gateway/channel proof for queued follow-up delivery.

## Reproduction Assessment

Yes. The linked issue gives a Discord follow-up scenario and production `SessionWriteLockTimeoutError` logs, and source inspection shows current main can hold or reacquire the session lock across long post-prompt phases; I did not run the full live scenario.

## Solution Assessment

No. Removing the initial held lock is too broad because session repair/open/setup still performs direct `SessionManager` writes that are not independently lock-wrapped.
