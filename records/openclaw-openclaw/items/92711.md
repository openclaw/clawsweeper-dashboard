## Summary

Keep open: this PR targets a real P1 session/message-loss bug, but the current patch can refresh the session-file fence without holding or owning the lock, and the supplied proof does not exercise the queued follow-up delivery path.

## What This Changes

The PR removes the post-prompt retained session write-lock reacquisition in `attempt.session-lock.ts`, factors fence capture into a helper, and updates related session-lock lifecycle test expectations.

## Best Possible Solution

Keep the per-write lock direction, but make the no-held-lock prompt release verify or no-op instead of refreshing the fence, add a regression for that second-release window, and attach redacted queued-follow-up proof.

## Reproduction Assessment

Yes at source level, but not by a live run here. Current main reacquires a retained session write lock after prompt completion, and the linked issue provides concrete Discord follow-up steps plus production `SessionWriteLockTimeoutError` logs.

## Solution Assessment

No. The PR targets the right owner path, but the no-held-lock release branch should not refresh trust from unlocked file state; the safer shape preserves per-write locking while keeping fence trust tied to a lock or owned-write proof.
