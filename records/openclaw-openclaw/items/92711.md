## Summary

Keep open: the reported bug is credible, but this PR is not merge-ready because it removes the setup/prep lock while session creation still writes through SessionManager outside per-write locking, and the supplied proof does not show queued follow-up delivery in the real gateway/channel path.

## What This Changes

The PR changes the embedded attempt session-lock controller so it no longer acquires or reacquires a retained session write lock around the full turn, relying on per-write locking and fence checks instead.

## Best Possible Solution

Preserve a real write lock through session repair/open/setup, release it before long prompt/model phases, keep each later transcript mutation under `withSessionWriteLock`, and add focused regression coverage plus redacted real transport proof.

## Reproduction Assessment

Yes at source level, but not by a live run here: current main acquires/reacquires the session write lock across the prompt lifecycle, and the linked issue provides a concrete Discord follow-up scenario plus production `SessionWriteLockTimeoutError` logs.

## Solution Assessment

No. Removing the retained lock entirely is too broad; the safer fix is to keep setup writes locked while releasing only the long non-write prompt/model phases and proving queued delivery.
