## Summary

Keep this PR open, but it is not merge-ready: the bug is real and high-impact, while the current patch can trust an unowned session-file state and still lacks real queued-follow-up delivery proof.

## What This Changes

The PR changes the embedded attempt session-lock controller so post-prompt reacquire no longer retains a session write lock and a no-held-lock prompt release captures a session-file fence, with two test expectations updated.

## Best Possible Solution

Keep per-write transcript locking while preserving takeover detection: do not capture a new fence from unowned state, add regression coverage for the second-release window, and require real queued-follow-up proof.

## Reproduction Assessment

Yes at source level, but not by a live run here. Current main reacquires a retained session write lock after prompt completion, and the linked issue provides concrete Discord follow-up steps plus production SessionWriteLockTimeoutError logs.

## Solution Assessment

No. The direction is plausible, but the current patch can refresh the fence from an unowned state in a reachable second release path; the safer fix verifies or preserves the existing fence and proves queued delivery in a real gateway/channel setup.
