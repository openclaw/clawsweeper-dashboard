## Summary

Keep this PR open, but do not merge the current patch: the reported lock-lifetime bug is plausible, yet this implementation removes the eager write lock before session preparation paths that still write directly to the transcript.

## What This Changes

The PR changes `src/agents/embedded-agent-runner/run/attempt.session-lock.ts` so the embedded attempt controller no longer acquires a session write lock at creation time and instead relies on per-write lock acquisition.

## Best Possible Solution

Keep the bug fix focused on shortening the long post-provider lock hold while preserving locked session preparation and adding regression plus real gateway/channel proof for queued follow-up delivery.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main releases the lock for provider streaming but reacquires and retains it for later phases, so long tool/cleanup work can block queued follow-up runs until `session.writeLock.acquireTimeoutMs`. The linked issue also includes production timeout logs for that path.

## Solution Assessment

No. The PR is a plausible direction, but removing the initial held lock is too broad because session preparation still has direct `SessionManager` writes that are not independently lock-wrapped.
