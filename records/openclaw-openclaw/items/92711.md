## Summary

Keep open: the linked bug is valid, but this PR is not merge-ready because the current patch removes the setup-phase write lock before session repair/open/setup and still lacks proof that queued follow-up messages deliver through a real gateway/channel path.

## What This Changes

The PR changes the embedded attempt session-lock controller to stop retaining a session write lock across the full turn and rely on per-write locking plus fence checks instead.

## Best Possible Solution

Keep session repair/open/setup under a real session write lock, release only the long prompt/model phases, keep later transcript mutations under `withSessionWriteLock`, and add focused regression plus redacted real transport proof.

## Reproduction Assessment

Yes at source level, but not by a live run here: current main holds the session write lock across setup/prompt lifecycle, and the linked issue provides concrete Discord follow-up steps plus production `SessionWriteLockTimeoutError` logs.

## Solution Assessment

No. Removing the retained lock entirely is too broad; the safer fix is to preserve setup serialization and release only the long non-write phase with proof that queued delivery works.
