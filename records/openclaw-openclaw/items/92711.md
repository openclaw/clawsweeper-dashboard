## Summary

Keep this PR open: the final diff still removes setup-phase session serialization, which can let transcript repair/open/session setup writes interleave before a fence is captured, and the supplied proof does not show queued gateway/channel delivery after the change.

## What This Changes

The branch changes `src/agents/embedded-agent-runner/run/attempt.session-lock.ts` to stop acquiring the session write lock at controller creation and rely on per-write locking plus fence checks.

## Best Possible Solution

Keep session repair/open/setup under a real session write lock, release only the long prompt/model phase, keep transcript mutations under `withSessionWriteLock`, and add focused regression plus redacted real transport proof.

## Reproduction Assessment

Yes at source level, but not by a live run here: current main holds the session write lock across setup and prompt lifecycle, and the linked issue provides concrete Discord follow-up steps plus production `SessionWriteLockTimeoutError` logs.

## Solution Assessment

No. Removing the retained lock entirely is too broad; the safer fix is to preserve setup serialization and release only the long non-write phase with proof that queued delivery works.
