## Summary

Keep this PR open, but it is not merge-ready: it targets a real session write-lock bug, while the patch removes the eager controller lock broadly enough to leave session repair/open/setup writes outside the cross-process transcript lock and still lacks after-fix real behavior proof.

## What This Changes

The PR changes `src/agents/embedded-agent-runner/run/attempt.session-lock.ts` so the embedded attempt controller no longer acquires or reacquires a held session write lock around the turn lifecycle and relies on per-write locking instead.

## Best Possible Solution

Keep session repair/open/setup writes serialized, narrow the long post-provider lock hold, and prove queued follow-up delivery with focused regression coverage plus redacted real gateway/channel evidence.

## Reproduction Assessment

Yes. Source inspection shows current main releases the lock for provider streaming and then reacquires a retained lock for later phases, while the linked issue provides production `SessionWriteLockTimeoutError` logs for queued follow-up turns.

## Solution Assessment

No. Per-write locking is the right direction for transcript events, but this patch is too broad because session repair/open/setup still rely on the current eager prep lock.
