## Summary

Keep open: this is a focused fix for a real queued-message/session-lock bug, but the latest head still advances trusted fence state from a no-held-lock path and the supplied terminal proof does not show the reported queued follow-up being delivered without `SessionWriteLockTimeoutError`.

## What This Changes

The PR removes the embedded attempt runner’s post-prompt retained session write-lock reacquire, factors session-file fence capture, and updates lock-lifecycle test expectations.

## Best Possible Solution

Keep the short-lived write-lock direction, but only advance trusted fence state from held-lock or explicitly published owned-write paths, add a regression for the real runner sequence, and require redacted queued follow-up proof before merge.

## Reproduction Assessment

Source-reproducible, not live reproduced here: current main reacquires a retained session write lock after prompt streaming, and the linked issue provides concrete queued Discord follow-up steps plus production `SessionWriteLockTimeoutError` logs. I did not run the live Discord/Gateway follow-up scenario in this read-only review.

## Solution Assessment

No. The per-write locking direction is maintainable, but this PR also refreshes trusted fence state without owning the retained lock; the safer fix keeps fence advancement tied to lock ownership or explicit owned-write publication.
