## Summary

Keep open: the helper-level stale-thread protection is useful, but the PR’s raw `to` comparison can drop a valid fallback `threadId` for equivalent channel target aliases, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The branch adds `mergeDeliveryContext` guards and unit tests to stop fallback `threadId` inheritance on same-channel target mismatches and orphan fallback thread IDs.

## Best Possible Solution

Keep helper-level protection only after target mismatch uses channel-aware canonical target equivalence, or move the guard to a layer that has loaded channel target parsing available, with a regression for equivalent encodings.

## Reproduction Assessment

Yes from source inspection: current main inherits fallback `threadId` for same-channel merges, and the PR diff plus Slack target parsing show an equivalent-target regression path. I did not run a live Slack route in this read-only review.

## Solution Assessment

No, not as written. The helper is a reasonable place to block true cross-conversation thread leakage, but dropping fallback thread state should be based on channel-canonical target comparison rather than raw `to` string inequality.
