## Summary

Keep open: the PR targets a real delivery-context thread leak, but the current diff can drop valid thread context because it compares raw same-channel targets instead of the existing route-equivalence contract; it also still lacks external real behavior proof.

## What This Changes

The PR updates `mergeDeliveryContext` to suppress fallback `threadId` inheritance for same-channel target mismatches and orphan fallback thread IDs, plus focused unit tests.

## Best Possible Solution

Keep helper-level stale-thread protection only after same-channel target mismatch is based on canonical route equivalence, or move the guard to a layer that has loaded channel target parsing, with a regression for equivalent target encodings and real routing proof.

## Reproduction Assessment

Yes from source inspection: current main inherits fallback `threadId` for same-channel delivery-context merges, and the PR tests encode the mismatch and orphan cases. The PR regression is also source-reproducible because channel target aliases can represent the same conversation while differing as raw `to` strings.

## Solution Assessment

No, not as written. The helper is a reasonable place to block true cross-conversation leakage, but target mismatch should use channel-aware canonical target equivalence or run where loaded plugin target parsing is available.
