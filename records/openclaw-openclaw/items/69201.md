## Summary

This PR should stay open: it targets a real delivery-context bug, but the current patch can suppress valid threaded delivery when two same-channel `to` strings are equivalent under a channel-owned target grammar, and the external PR still lacks real behavior proof.

## What This Changes

The branch updates `mergeDeliveryContext` to drop fallback `threadId` inheritance for same-channel target mismatches and orphan fallback thread IDs, with focused unit tests.

## Best Possible Solution

Keep helper-level stale-thread protection only after target mismatch is based on channel-aware canonical target equivalence, with a regression for equivalent encodings and redacted real routing proof.

## Reproduction Assessment

Yes from source inspection: current main inherits fallback `threadId` for same-channel delivery-context merges, and the PR tests encode the mismatch and orphan cases. The PR regression is also source-reproducible because Slack target aliases can represent the same conversation while differing as raw `to` strings.

## Solution Assessment

No, not as written. The helper is a reasonable place to block true cross-conversation thread leakage, but it should compare channel-canonical targets or run at a layer that has loaded plugin target parsing before dropping fallback thread state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve threads for equivalent channel targets:** `src/utils/delivery-context.shared.ts:128`

Compare parsed or canonical same-channel targets before dropping the thread.
