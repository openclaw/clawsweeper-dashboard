## Summary

Keep open: the PR targets a real stale-thread leak, but the current patch can drop valid thread context for same-conversation target aliases and still lacks real after-fix behavior proof from the external contributor.

## What This Changes

The branch updates `mergeDeliveryContext` to suppress fallback `threadId` inheritance for same-channel target mismatches and orphan fallback thread IDs, with focused unit tests.

## Best Possible Solution

Keep the helper-level stale-thread protection only after target mismatch uses channel-aware canonical comparison where available, preserving orphan-thread suppression and adding a regression for equivalent target aliases.

## Reproduction Assessment

Yes from source inspection: current main inherits fallback `threadId` on same-channel different-`to` merges, and the PR tests encode that helper path. I did not execute tests because this review is read-only.

## Solution Assessment

No, not as written. The helper is a reasonable place for true cross-conversation suppression, but raw `to` string comparison is unsafe before channel-owned target canonicalization is considered.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve threads for equivalent same-channel targets:** `src/utils/delivery-context.shared.ts:128`

Compare parsed/canonical same-channel targets before dropping the thread.
