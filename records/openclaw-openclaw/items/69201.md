## Summary

Keep this PR open: current main still has the stale-thread merge behavior, and the PR’s helper-level fix looks functionally sound, but it still needs the required user-facing changelog entry before merge.

## What This Changes

The PR updates `mergeDeliveryContext` to suppress fallback `threadId` values when same-channel targets differ or the fallback thread ID has no `to`, and adds focused unit coverage.

## Best Possible Solution

Land the helper-level guard and unit tests after adding a single Unreleased Fixes changelog entry, while keeping the broader spawn-site backfill out of this PR because gateway-side seeding already owns that behavior on main.

## Reproduction Assessment

Yes. Current main can be checked at unit level by merging same-channel contexts with different `to` values and a fallback `threadId`; the existing fallback expression inherits the stale thread ID, and the PR adds that regression case plus the orphan fallback case.

## Solution Assessment

Yes for the functional fix. A shared helper guard is the narrowest maintainable place to prevent same-channel cross-conversation thread leakage, and the remaining merge-readiness issue is only the missing changelog line.
