## Summary

Keep open: the stale fallback-thread problem is source-reproducible, but this PR’s helper-level raw `to` comparison can drop a valid fallback thread when two channel target strings parse to the same conversation. The external PR also still needs redacted after-fix real behavior proof before merge.

## What This Changes

The PR adds `mergeDeliveryContext` guards and unit tests to suppress fallback `threadId` inheritance on same-channel target mismatch and orphan fallback-thread merges.

## Best Possible Solution

Preserve the stale-thread protection, but compare parsed/canonical channel routes before dropping a fallback thread, or keep the guard in callers that already have channel target parsing available.

## Reproduction Assessment

Yes. Current `main` is source-reproducible because `mergeDeliveryContext` inherits a fallback `threadId` whenever channels match, and the PR regression is also source-reproducible because channel grammars can map different raw target strings to the same conversation.

## Solution Assessment

No, not as written. The helper is a reasonable place to block true cross-conversation leakage, but raw `to` inequality is not the route contract; use parsed/canonical route equivalence or a caller layer with loaded channel parsing.
