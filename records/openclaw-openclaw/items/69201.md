## Summary

Keep open for normal PR handling: current main still has the older channel-only merge behavior, and the functional patch is narrow and well-scoped. The only concrete gap I found is the required changelog entry for a user-facing delivery-routing fix.

## What This Changes

The PR updates `mergeDeliveryContext` to suppress fallback `threadId` values when same-channel targets differ or when the fallback thread is orphaned, and adds unit coverage for mismatch, matching-target inheritance, and orphan-thread cases.

## Best Possible Solution

Land the helper and unit-test change after adding the missing release-note entry, while keeping the earlier spawn-site backfill out of this PR because the gateway-side seeding path already owns that behavior on main.

## Reproduction Assessment

Yes. A unit-level call to `mergeDeliveryContext({ channel: "demo-channel", to: "channel:CURRENT" }, { channel: "demo-channel", to: "channel:STALE", threadId: "99" })` reproduces the current-main stale-thread inheritance, and the PR adds that case plus the orphan fallback case as regression tests.

## Solution Assessment

Yes for the functional fix. A helper-level guard is the narrowest maintainable place for same-channel cross-conversation thread leakage, and the tests preserve the intended inheritance path when `to` matches; the safer remaining step is only to add the changelog entry.
