## Summary

Keep open. The PR targets a real delivery-context leak, but the current patch can also drop valid fallback thread IDs because it compares raw `to` strings instead of channel-canonical route targets, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds two `mergeDeliveryContext` guards and unit tests to suppress fallback `threadId` inheritance when same-channel targets differ or fallback has a thread ID without a target.

## Best Possible Solution

Use channel-aware canonical route equivalence before dropping fallback `threadId`, or keep the guard at a caller layer that has loaded-channel parsing; add a regression for equivalent target encodings and post real routing proof.

## Reproduction Assessment

Yes, from source inspection. Current main inherits fallback `threadId` for same-channel delivery-context merges, and the PR regression is also source-reproducible because plugin grammars can map different raw targets to the same conversation.

## Solution Assessment

No, not as written. The helper is a reasonable place to block true cross-conversation leakage, but target mismatch needs parsed/canonical same-channel route equivalence or a caller layer with loaded plugin target parsing.

## Review Findings
