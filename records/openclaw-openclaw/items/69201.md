## Summary

Keep open: the stale-thread problem is real, but the helper-level guard still compares raw `to` strings, so it can drop valid thread context for channels whose target parser maps different raw forms to the same conversation; after-fix real behavior proof is also missing.

## What This Changes

The PR changes shared delivery-context merging to suppress fallback `threadId` inheritance on same-channel target mismatches or orphan fallback thread IDs, with focused unit tests.

## Best Possible Solution

Keep the stale-thread protection, but compare parsed or canonical route refs before suppressing fallback `threadId`, or keep the guard at callers that already have route facts, with tests for equivalent raw forms and true mismatches.

## Reproduction Assessment

Yes, at source level. Current main inherits fallback `threadId` whenever channels match, and existing target-parsing tests show raw target forms can differ while still representing the same conversation.

## Solution Assessment

No, not as written. The helper is a reasonable place to block true stale-thread leakage, but same-channel target mismatch needs canonical route comparison or a caller-layer guard with parsed route facts.
