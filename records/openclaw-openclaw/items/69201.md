## Summary

Keep open: this is a focused delivery-context bug fix, but the current patch still risks dropping valid thread context because it compares `to` strings inside a shared helper without using the channel route parsing contract, and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds `mergeDeliveryContext` guards and unit tests to suppress fallback `threadId` inheritance when same-channel targets differ or when the fallback has an orphan thread ID.

## Best Possible Solution

Keep the stale-thread protection, but compare parsed/canonical route refs before suppressing fallback `threadId`, or keep the guard in callers that already carry route facts, with tests for equivalent raw forms and true mismatches.

## Reproduction Assessment

Yes, at source level. Current main inherits fallback `threadId` whenever channels match, and existing route parsing tests show raw target forms can differ while representing the same conversation.

## Solution Assessment

No, not as written. The helper is a reasonable hardening point for true stale-thread leakage, but it needs parsed/canonical route comparison or narrower caller-level guards before merge.
