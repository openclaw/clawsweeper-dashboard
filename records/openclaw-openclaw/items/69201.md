## Summary

Keep open: the stale-thread bug is plausible, but the helper-level raw `to` comparison can break existing threaded delivery for channel grammars where different raw targets resolve to the same conversation, and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds `mergeDeliveryContext` guards and unit tests to drop fallback `threadId` inheritance on same-channel target mismatches or orphan fallback threads.

## Best Possible Solution

Keep the stale-thread protection, but base same-channel decisions on parsed/canonical channel routes or confine the guard to callers with route facts, with tests for equivalent raw targets and true mismatches.

## Reproduction Assessment

Yes, at source level. Current `main` inherits fallback `threadId` whenever channels match, and existing target-parsing tests show raw target forms can differ while still representing the same conversation.

## Solution Assessment

No, not as written. The helper is a reasonable place to block true stale-thread leakage, but same-channel target mismatch needs canonical route comparison or a caller-layer guard with parsed route facts.
