## Summary

Keep open. Current main and the latest release still expose only payload-level light context for cron jobs; there is no per-job cache/context-pruning override in the stored cron contract, gateway protocol, docs, CLI/UI surfaces, or isolated runner config composition.

## What This Changes

The issue requests per-cron-job cache or context-pruning control so isolated cron jobs that do not need LLM continuity can avoid prompt-cache/context-pruning cost.

## Best Possible Solution

Add an additive top-level cron-job contextPruning override using the existing AgentContextPruningConfig contract, then thread it consistently through storage, protocol, CLI, Control UI, docs, generated clients, and isolated-run config composition.

## Reproduction Assessment

Yes, at source level. Current main has no disableCache or contextPruning field in the cron job contract, closed gateway schemas, public cron docs, CLI/UI cron payload surfaces, or isolated-run config composition.

## Solution Assessment

Yes directionally, but the final API needs maintainer choice. Reusing the existing contextPruning contract looks more maintainable than a runner-only boolean, provided every public cron surface is updated together.
