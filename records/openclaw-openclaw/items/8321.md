## Summary

Keep open. Current main and the latest shipped tag still lack any per-cron-job cache/context-pruning override; the only prior implementation attempt is closed unmerged, and the direct script/no-LLM cron request is complementary rather than a replacement for agent-run cron jobs that still need per-job pruning policy.

## What This Changes

The issue requests adding a per-cron-job cache or context-pruning override so simple isolated cron jobs can avoid prompt-cache context costs.

## Best Possible Solution

Add an additive top-level cron job contextPruning override using the existing AgentContextPruningConfig contract, round-trip it through storage/protocol/CLI/UI/tool/docs, and merge it into effective agent defaults for that scheduled run.

## Reproduction Assessment

Yes at source level: current main has no disableCache or contextPruning path in cron job types, persisted shape, gateway schemas, CLI/UI surfaces, or isolated execution. I did not run tests because this review is explicitly read-only.

## Solution Assessment

Yes directionally, but the maintainable API should reuse AgentContextPruningConfig with mode off/cache-ttl unless maintainers explicitly choose a simpler boolean surface. The closed prior PR is useful background, but it is not merged and did not cover all public surfaces.
