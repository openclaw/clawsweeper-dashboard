## Summary

Keep open: current main still lacks a per-cron-job cache/context-pruning override, the matching implementation PR was closed unmerged, and direct-exec cron work is complementary rather than a replacement for agent-run cron jobs.

## What This Changes

The issue requests per-cron-job cache/context-pruning controls so isolated cron jobs can disable or tune prompt caching independently of global agent defaults.

## Best Possible Solution

Add an additive top-level cron contextPruning?: AgentContextPruningConfig override using the existing off/cache-ttl contract, round-trip it through all cron API/client surfaces, and apply it to the effective isolated-run agent config for that job only.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source inspection on current main gives high confidence that the requested job-level cache/pruning field is absent and pruning remains driven by agent defaults.

## Solution Assessment

Yes. Reusing the existing contextPruning object is the narrowest maintainable direction because it avoids a one-off disableCache boolean and preserves the current off/cache-ttl pruning contract.
