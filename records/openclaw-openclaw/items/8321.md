## Summary

Keep open. Current main and v2026.6.6 now have zero-LLM command cron jobs for deterministic scripts, but they still do not expose per-job `contextPruning` or `disableCache` for model-backed cron jobs.

## What This Changes

The issue requests per-cron-job cache/context-pruning control so isolated cron jobs can disable prompt caching independently of global agent defaults.

## Best Possible Solution

Keep the issue open and let maintainers decide whether to add an additive `contextPruning?: AgentContextPruningConfig` cron-job override for model-backed jobs, while treating command cron as the shipped answer for deterministic scripts.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 expose command payloads and `payload.lightContext`, but no cron job-level `contextPruning` or `disableCache` field across the cron type, protocol schema, persistence envelope, or runner config merge.

## Solution Assessment

Partly. Direct command jobs are now the best solution for deterministic scripts, but the proposed cache-control API remains a new config/product surface for model-backed cron jobs and should be maintainer-designed rather than auto-implemented.
