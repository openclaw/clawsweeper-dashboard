## Summary

Keep open. Current main still lacks job-level cache or context-pruning control for cron jobs, and adding it would change public cron config, protocol, UI, and tool behavior that needs maintainer API/product judgment.

## What This Changes

The issue requests a per-cron-job option to disable or override prompt-cache/context-pruning behavior for isolated scheduled jobs.

## Best Possible Solution

Add an additive job-level `contextPruning?: AgentContextPruningConfig` override, round-trip it through cron storage, gateway, CLI, UI, tool, and docs surfaces, and merge it into the effective agent defaults for that scheduled run.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction, but yes at source level: current main has no `disableCache` or `contextPruning` path in cron add/update/list schemas or isolated execution.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally, but the maintainable path should reuse the existing `AgentContextPruningConfig` contract unless maintainers explicitly choose a simpler boolean API.
