## Summary

Keep open. Current main and the latest release still lack per-cron-job cache/context-pruning control; the prior implementation PR is closed unmerged, and the direct exec/cache-affinity related work is adjacent rather than a replacement for per-job pruning on agent-run cron jobs.

## What This Changes

Request adds per-cron-job cache/context-pruning controls so isolated scheduled jobs can disable or tune prompt caching independently of global agent defaults.

## Best Possible Solution

Add an additive top-level cron job contextPruning override using AgentContextPruningConfig, round-trip it through storage, protocol, CLI, Control UI, agent tool, docs, and clients, then merge it into effective agent defaults before isolated cron execution.

## Reproduction Assessment

Yes, at source level. Current main has no disableCache or contextPruning path in the cron job type, persisted shape, gateway schema, CLI/UI/tool surfaces, docs, or isolated execution config merge.

## Solution Assessment

Yes directionally, but the exact API needs maintainer choice. Reusing AgentContextPruningConfig as a top-level cron override is more maintainable than a runner-only flag or storage-only migration.
