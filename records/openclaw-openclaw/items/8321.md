## Summary

Keep open. Current main still lacks per-job cache or context-pruning control for cron jobs, and the remaining work is a public cron API/config decision rather than conservative cleanup.

## What This Changes

Request adds a per-cron-job cache or context-pruning override so isolated cron jobs can opt out of the global `agents.defaults.contextPruning` policy.

## Best Possible Solution

Add an additive job-level `contextPruning?: AgentContextPruningConfig` override, round-trip it through cron storage/API/CLI/UI/tool/docs surfaces, and apply it when building the effective agent defaults for that scheduled run.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction, but yes at source level: current main lacks job-level `disableCache` or `contextPruning` fields while isolated cron pruning still comes from effective agent defaults.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally, but the maintainable solution should reuse the existing `AgentContextPruningConfig` contract unless maintainers explicitly choose a simpler boolean API.
