## Summary

Keep open: current main still has only payload-level `lightContext` for cron jobs, while cache/context pruning remains driven by agent defaults. The earlier implementation PR is closed unmerged, and the related direct-script cron work is complementary rather than a replacement for per-job pruning on agent-turn cron jobs.

## What This Changes

The issue requests per-cron-job cache or context-pruning control so isolated cron jobs can opt out of prompt caching independently of global agent defaults.

## Best Possible Solution

Add a top-level cron `contextPruning?: AgentContextPruningConfig` override using the existing `mode: "off" | "cache-ttl"` contract, round-trip it through cron APIs and clients, and apply it to the effective isolated-run agent config for that job only.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection on current main gives high-confidence evidence that the requested job-level cache/pruning field is absent and pruning remains driven by agent defaults.

## Solution Assessment

Yes. Reusing the existing `contextPruning` object is the narrowest maintainable direction because it avoids a one-off `disableCache` boolean and preserves the current off/cache-ttl pruning contract.
