## Summary

Keep open. Current main still has no per-job context-pruning override for agent-turn cron jobs; the newer command payload solves deterministic no-LLM scripts but does not provide the requested cache policy for jobs that still need an agent. The previous implementation at https://github.com/openclaw/openclaw/pull/8341 closed without merging, so no viable related PR currently owns the remaining work. Likely related people: steipete is the strongest routing candidate from recent cron and protocol history; vincentkoc and jose-velez have adjacent runner and lightweight-context history.

## What This Changes

Add an optional per-job context-pruning override so isolated agent cron jobs can use a different cache policy from the global or per-agent defaults.

## Best Possible Solution

Keep direct command payloads as the zero-LLM path for deterministic scripts; for jobs that still require an agent, add one additive contextPruning override using the existing canonical pruning type and apply it after global/per-agent defaults while keeping persistence, schemas, clients, UI, docs, and regression coverage aligned.

## Reproduction Assessment

Yes, at source level. Current main constructs isolated cron agent configuration from global and per-agent defaults, while the persisted job and protocol surfaces expose no per-job pruning override; no live token-cost run was performed during this read-only review.

## Solution Assessment

Yes, conditionally. Reusing the existing AgentContextPruningConfig contract is cleaner than adding a separate disableCache boolean, but maintainers must approve the scope and placement of the new public configuration surface.
