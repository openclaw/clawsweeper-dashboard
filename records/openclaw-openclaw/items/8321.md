## Summary

Keep open. Current main and the latest release still lack a per-cron-job cache or context-pruning override: cron jobs only expose payload-level `lightContext`, while prompt-cache pruning is still driven through effective agent defaults; the prior implementation PR is closed unmerged and direct script cron work is complementary, not a replacement.

## What This Changes

Request adds per-cron-job cache/context-pruning control so isolated cron jobs can disable or tune prompt caching independently of global defaults.

## Best Possible Solution

Add a top-level `contextPruning?: AgentContextPruningConfig` cron job override using `mode: "off" | "cache-ttl"`, then round-trip it through cron types, normalization, persistence, Gateway schema, CLI, Control UI, docs, and the agent cron tool before merging it into effective agent defaults for isolated runs.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high-confidence proof that current main and v2026.5.12 only support payload-level `lightContext` plus defaults-level `contextPruning`, not per-job cache control.

## Solution Assessment

Yes directionally, but the maintainable form should reuse the existing `AgentContextPruningConfig` and `mode: "off" | "cache-ttl"` contract unless maintainers explicitly prefer a separate boolean API.
