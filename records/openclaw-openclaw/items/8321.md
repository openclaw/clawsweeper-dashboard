## Summary

Keep this issue open. Current main still has no per-cron-job cache or context-pruning override: cron jobs expose payload-level light context only, while pruning setup is still derived from `agents.defaults.contextPruning`, and the prior implementation PR is closed unmerged.

## What This Changes

The issue requests per-job cache/context-pruning control for isolated cron jobs so simple scheduled agent runs can opt out of global pruning/cache behavior.

## Best Possible Solution

Implement an additive top-level `contextPruning?: AgentContextPruningConfig` cron job override using the existing `mode: "off" | "cache-ttl"` contract, thread it through persisted cron types, normalization, Gateway schemas, CLI, Control UI, docs, and the agent cron tool, then merge it into the effective `cfgWithAgentDefaults` before isolated cron invokes the embedded runner. Keep direct script/no-LLM execution and per-job elevated exec scoping as separate work.

## Reproduction Assessment

Yes. A code-level reproduction is clear: creating or updating a cron job with `contextPruning` or `disableCache` is rejected by the closed Gateway schema, and even a hand-injected stored field would not be read by the isolated runner or pruning extension.

## Solution Assessment

Yes. The narrowest maintainable solution is an additive job-level `contextPruning` override rather than a boolean `disableCache`; if maintainers want provider cache-write suppression too, that should be an explicit cache-retention design rather than hidden behind pruning mode.
