## Summary

Keep open. Current main still lacks per-cron-job cache/context-pruning control, the earlier implementation PR is closed unmerged, and related direct-script cron work is complementary rather than a replacement for agent-run cron jobs that need per-job pruning policy.

## What This Changes

Add optional per-job cache or context-pruning control for isolated cron jobs so simple scheduled agent runs can avoid prompt-cache pruning/caching costs.

## Best Possible Solution

Add an additive top-level cron job contextPruning override using the existing AgentContextPruningConfig contract, round-trip it through storage, protocol, CLI, UI, tool, generated clients, and docs, then merge it into effective agent defaults for that scheduled run.

## Reproduction Assessment

Yes, at source level. Current main has no disableCache or contextPruning path in cron job types, persisted shape, gateway schema, CLI/tool/UI surfaces, docs, or isolated execution.

## Solution Assessment

Yes directionally, but the final API needs maintainer choice. The maintainable path is likely a top-level contextPruning override reusing AgentContextPruningConfig rather than a runner-only boolean or storage-only migration.
