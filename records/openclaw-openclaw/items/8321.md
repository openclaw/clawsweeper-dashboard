## Summary

Keep open. Current main still lacks a per-cron-job cache/context-pruning override, the earlier implementation PR is closed unmerged, and the separate direct script/no-LLM cron work does not replace per-job pruning for agent-run cron jobs.

## What This Changes

The issue requests per-cron-job cache or context-pruning control so isolated cron jobs that do not need LLM continuity can avoid prompt-cache costs.

## Best Possible Solution

Add an additive top-level cron job contextPruning override using the existing AgentContextPruningConfig contract, round-trip it through storage, protocol, CLI, UI, tool clients, generated clients, and docs, then merge it into effective agent defaults for that scheduled run.

## Reproduction Assessment

Yes, at source level. Current main has no disableCache or contextPruning path in cron job types, persisted shape, gateway schemas, public cron docs, Control UI controller, cron tool schema, CLI add/edit surfaces, or isolated execution config composition.

## Solution Assessment

Yes directionally, but the final API needs maintainer choice. A top-level contextPruning override reusing AgentContextPruningConfig looks more maintainable than a runner-only boolean, but it must be threaded through every public cron surface.
