## Summary

Keep open. Current main and the latest release still lack a per-cron-job `contextPruning` or `disableCache` override; the only existing cron-side cost control is payload-level `lightContext`, and the earlier implementation pull request was closed unmerged.

## What This Changes

The issue requests a per-cron-job override to disable or customize context pruning/prompt caching for isolated cron jobs.

## Best Possible Solution

Add a maintainer-approved top-level cron `contextPruning?: AgentContextPruningConfig` override, thread it through storage/protocol/CLI/UI/docs/tool surfaces, and merge it into the effective agent defaults before isolated cron starts the embedded runner.

## Reproduction Assessment

Yes, from source inspection: current main exposes cron `payload.lightContext`, but the cron type, persisted base, gateway schema, CLI, UI, and runner have no per-job pruning/cache override. I did not run a live cron job because this is a missing public configuration feature rather than a runtime-only bug.

## Solution Assessment

Yes, the requested direction is reasonable, but the full `contextPruning.mode: "off" | "cache-ttl"` override is the cleaner maintainable shape than a one-off `disableCache` boolean. Maintainers should approve the public API before implementation because it changes the cron wire/storage contract.
