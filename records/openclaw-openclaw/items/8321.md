## Summary

Keep this open. Current main still has no per-cron-job cache/context-pruning override, the earlier implementation PR is closed and unmerged, and the direct-script/no-LLM discussion is complementary rather than a replacement for agent-run cron jobs.

## What This Changes

The issue requests per-cron-job control over prompt cache/context pruning so isolated script-like cron jobs can avoid unnecessary caching.

## Best Possible Solution

Implement an additive top-level `contextPruning?: AgentContextPruningConfig` cron job override, reuse the existing `mode: "off" | "cache-ttl"` contract, and merge it into the effective agent defaults before isolated cron invokes the embedded runner.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source review gives high confidence that current main lacks the requested job-level field and that isolated cron still reads pruning from agent defaults.

## Solution Assessment

Yes. A top-level `contextPruning` override is the maintainable direction because it reuses the existing pruning config type instead of adding a one-off `disableCache` boolean.

## Repair Work Prompt

Implement per-cron-job prompt cache/context-pruning control for https://github.com/openclaw/openclaw/issues/8321. Add an additive top-level `contextPruning?: AgentContextPruningConfig` field to cron job create/update/list/storage/UI types and gateway protocol schemas, using the existing `mode: "off" | "cache-ttl"` contract. Thread the field through normalization, persistence, CLI cron add/edit if applicable, Control UI form/list/edit, agent-facing cron tool schema/recovery, docs, and isolated cron execution. In `prepareCronRunContext`, merge the job-level override into the effective `cfgWithAgentDefaults.agents.defaults.contextPruning` before the embedded runner/context-pruning extension sees the config. Add regression coverage proving a job-level `mode: "off"` overrides global or per-agent `mode: "cache-ttl"`, while a job without the override preserves current defaults. Do not implement or conflate a direct script/no-LLM execution path; keep that separate from this agent-run cron feature. Add a user-facing changelog entry because this changes public cron configuration behavior.
