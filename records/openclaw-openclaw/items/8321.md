## Summary

Keep open. Current main still has no job-level `contextPruning` or `disableCache` cron field across the type, protocol, CLI, UI, and runner surfaces; the prior implementation PR is closed unmerged, and the direct-script cron request does not replace agent-run pruning control.

## What This Changes

The issue requests optional per-cron-job cache/context-pruning control so simple isolated cron jobs can avoid unnecessary prompt-cache costs.

## Best Possible Solution

Add an additive top-level `contextPruning?: AgentContextPruningConfig` cron job override using the existing `mode: "off" | "cache-ttl"` shape, and apply it only to that job's isolated agent run.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current main lacks the requested job-level field and still reads pruning from effective agent defaults.

## Solution Assessment

Yes, the narrowed `contextPruning` override is the best current direction. Reusing the existing pruning config avoids a one-off `disableCache` boolean while preserving future-compatible `off` and `cache-ttl` behavior.
