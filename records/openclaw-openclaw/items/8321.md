## Summary

Keep open. Current main still lacks a per-cron-job cache/context-pruning override, the prior implementation pull request was closed unmerged, and the direct-exec cron discussion is complementary rather than a replacement for agent-run cron jobs that need per-job pruning policy.

## What This Changes

The issue requests a per-cron-job cache/context-pruning override so isolated cron jobs can avoid prompt-cache pruning when scheduled work does not need LLM continuity.

## Best Possible Solution

Add an additive job-level contextPruning override using the existing AgentContextPruningConfig contract, round-trip it through cron storage/protocol/CLI/UI/tool/docs, and merge it into effective agent defaults for that scheduled run.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main has no disableCache or contextPruning path in cron job types, persistence, gateway schemas, CLI/UI surfaces, or isolated execution.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally, but the maintainable API should reuse AgentContextPruningConfig mode off/cache-ttl unless maintainers explicitly choose a simpler boolean surface.
