## Summary

Keep open. Current main and the latest release still lack per-cron-job cache or context-pruning control; only payload-level light context exists, while cache-ttl pruning is still driven by effective agent defaults.

## What This Changes

Request per-cron-job cache/context-pruning control so simple isolated cron jobs can opt out without changing global agent defaults.

## Best Possible Solution

Add an additive top-level cron contextPruning override, thread it through persistence, gateway protocol, CLI, Control UI, cron tool/docs, and isolated-run config composition, then cover job-level mode off overriding defaults-level cache-ttl.

## Reproduction Assessment

Yes, at source level. Current main lacks job-level disableCache or contextPruning in cron types, persistence, gateway schema, CLI/UI-facing surfaces, and isolated-run config composition.

## Solution Assessment

Unclear. A top-level contextPruning object looks more maintainable than a one-off boolean, but maintainers still need to decide the public cron API and merge semantics.
