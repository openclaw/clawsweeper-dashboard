## Summary

Keep open. Current main still lacks per-cron-job cache/context-pruning control, the prior implementation PR is closed unmerged, and the direct script/no-LLM cron work is complementary rather than a replacement for agent-run cron jobs.

## What This Changes

Requests per-cron-job cache or context-pruning control so simple isolated cron jobs can skip prompt caching or override global pruning defaults.

## Best Possible Solution

Add an additive top-level cron job contextPruning override using the existing AgentContextPruningConfig contract, round-trip it through storage, protocol, CLI, UI, tool/generated clients, and docs, then merge it into effective agent defaults for that scheduled run.

## Reproduction Assessment

Yes at source level. Current main has no disableCache or contextPruning path in cron job types, persisted shape, gateway schemas, CLI/UI/tool surfaces, docs, or isolated execution.

## Solution Assessment

Yes directionally, but the final API needs maintainer choice. A top-level contextPruning override reusing AgentContextPruningConfig looks more maintainable than a runner-only boolean, but it must be threaded through every public cron surface.
