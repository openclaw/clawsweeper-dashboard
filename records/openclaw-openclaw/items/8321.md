## Summary

Keep open: current main still lacks per-cron-job `contextPruning` or `disableCache`, the prior implementation PR was closed unmerged, and isolated cron still derives pruning from effective agent defaults.

## What This Changes

Request an optional per-cron-job cache/context-pruning override for isolated cron jobs so simple scheduled agent turns can avoid unnecessary prompt-cache costs.

## Best Possible Solution

Add a top-level `contextPruning?: AgentContextPruningConfig` cron field using the existing `off`/`cache-ttl` contract, round-trip it through cron APIs and clients, and apply it only to that job's effective isolated-run agent config.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection on current main verifies no job-level `contextPruning`/`disableCache` field and shows pruning still reads `agents.defaults.contextPruning`.

## Solution Assessment

Yes. Reusing the existing `contextPruning` object is the narrowest maintainable direction because it avoids a one-off `disableCache` boolean and preserves the current pruning contract.

## Repair Work Prompt

Implement per-job cache/context-pruning control for cron jobs from https://github.com/openclaw/openclaw/issues/8321. Use an additive top-level `contextPruning?: AgentContextPruningConfig` cron field with the existing `mode: "off" | "cache-ttl"` contract; do not make a boolean-only `disableCache` the primary public API unless a maintainer explicitly redirects. Thread the field through cron persisted/shared types, normalization, create/update/list contracts, gateway TypeBox schemas, agent cron tool schemas, CLI add/edit/show/list surfaces, Control UI types/form/controller/view as needed, docs, and generated/protocol artifacts required by repo checks. In isolated cron execution, merge the job-level `contextPruning` into the effective per-run `cfgWithAgentDefaults` before the embedded runner builds context-pruning extensions, so a job-level `mode: "off"` overrides global or per-agent `cache-ttl`. Add focused regression coverage proving the override disables cache-ttl pruning for that job while leaving jobs without the field on current defaults. Leave direct exec/script cron behavior to its separate tracking path and do not change unrelated cron scheduling, delivery, or elevation policy. Add a user-facing changelog entry.
