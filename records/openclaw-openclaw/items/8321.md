## Summary

Keep open: current main and the latest release still lack per-cron-job cache/context-pruning control, the prior implementation PR is closed unmerged, and the direct-exec cron discussion is complementary rather than a replacement for agent-run cron jobs.

## What This Changes

The issue requests optional per-cron-job cache or context-pruning control so simple isolated cron jobs can avoid prompt caching/token cost.

## Best Possible Solution

Add a top-level cron contextPruning?: AgentContextPruningConfig override using the existing off/cache-ttl contract, round-trip it through cron APIs and clients, and apply it to the effective isolated-run agent config for that job only.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source inspection on current main and v2026.5.12 gives high-confidence evidence that the requested job-level cache/pruning field is absent and pruning remains driven by agent defaults.

## Solution Assessment

Yes. Reusing the existing contextPruning object is the narrowest maintainable direction because it avoids a one-off disableCache boolean and preserves the existing off/cache-ttl pruning contract.
