## Summary

Keep open: current main still has no per-cron-job cache or `contextPruning` override, the earlier implementation attempt is closed unmerged, and the remaining work is a focused new cron configuration capability rather than stale cleanup.

## What This Changes

Request per-job cache/context-pruning control so isolated cron jobs can disable or tune prompt-cache pruning independently of global agent defaults.

## Best Possible Solution

Add a top-level `contextPruning?: AgentContextPruningConfig` cron field using the existing `mode: "off" | "cache-ttl"` contract, round-trip it through cron APIs and clients, and apply it only to that job's effective isolated-run agent config.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source inspection on current main confirms no job-level `contextPruning` or `disableCache` field and shows pruning still comes from effective agent defaults.

## Solution Assessment

Yes. Reusing the existing `contextPruning` object is the narrowest maintainable direction because it avoids a one-off `disableCache` boolean and preserves the current pruning mode contract.
