## Summary

Keep open. Current main still lacks any job-level `contextPruning` or `disableCache` cron field, and the runner still derives prompt-cache pruning from effective agent defaults rather than the individual cron job.

## What This Changes

Request to add per-cron-job prompt-cache/context-pruning control so simple isolated cron jobs can skip cache-ttl pruning/caching costs.

## Best Possible Solution

Add an additive top-level `contextPruning?: AgentContextPruningConfig` cron job override using the existing `mode: "off" | "cache-ttl"` contract, then apply it only to that job's effective isolated-run agent config.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection on current main gives high confidence that the requested job-level field is absent and pruning still reads from effective agent defaults.

## Solution Assessment

Yes. Reusing the existing `contextPruning` object is the narrowest maintainable direction because it avoids a one-off `disableCache` boolean and preserves the current `off`/`cache-ttl` contract.

## Repair Work Prompt

Implement per-job context pruning control for cron jobs. Add an additive top-level `contextPruning?: AgentContextPruningConfig` field to cron job create/update/list/storage types using the existing `mode: "off" | "cache-ttl"` contract, not a new `disableCache` boolean. Thread it through normalization, persistence, Gateway TypeBox schemas/protocol exports, the cron CLI add/edit surfaces, Control UI cron form/types, and the agent cron tool schema. In isolated cron execution, merge a job-level `contextPruning` override into the effective `cfgWithAgentDefaults.agents.defaults.contextPruning` before the embedded/CLI runner receives config, so a job-level `mode: "off"` overrides global or per-agent `cache-ttl`. Add focused regression coverage for create/update round-trip and isolated execution override behavior. Update user-facing cron/prompt-cache docs and add a CHANGELOG entry. Do not implement direct script/exec cron payloads from #18160/#24597 and do not change unrelated context-pruning semantics.
