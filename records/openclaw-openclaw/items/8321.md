## Summary

Keep open. Current main still lacks any per-cron-job cache/context-pruning control, the prior implementation PR is closed unmerged, and the related direct script/no-LLM cron work is complementary rather than a replacement for agent-run cron jobs that need per-job pruning policy.

## What This Changes

The issue requests an additive per-cron-job cache/context-pruning override so isolated cron jobs can avoid prompt caching when they do not need LLM continuity.

## Best Possible Solution

Add an additive top-level cron job contextPruning override using the existing AgentContextPruningConfig contract, round-trip it through storage, protocol, CLI, UI, tool, generated clients, and docs, then merge it into effective agent defaults for that scheduled run.

## Reproduction Assessment

Yes at source level: current main has no disableCache or contextPruning path in cron job types, persisted shape, gateway schemas, CLI/UI/tool surfaces, docs, or isolated execution. I did not run tests because this review is read-only and the missing public contract is visible from source.

## Solution Assessment

Yes directionally, but the final API needs maintainer choice. The maintainable path is likely a top-level contextPruning override reusing AgentContextPruningConfig rather than a runner-only field or storage-only migration.
