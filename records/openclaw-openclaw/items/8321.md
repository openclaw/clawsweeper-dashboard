## Summary

Keep open. Current main still lacks job-level cache or context-pruning control for cron jobs, and the remaining work is a public cron API/config decision rather than conservative cleanup.

## What This Changes

Request to add per-cron-job cache/context-pruning overrides so simple scheduled agent jobs can avoid unnecessary cache-ttl prompt pruning behavior.

## Best Possible Solution

Add an additive job-level `contextPruning?: AgentContextPruningConfig` override, round-trip it through cron storage/API/CLI/UI/tool surfaces, and merge it into the effective agent defaults only for that scheduled run.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction; source inspection gives high-confidence proof that current main lacks job-level `disableCache` or `contextPruning` fields while runtime pruning still reads effective agent defaults.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally, but the maintainable solution should reuse the existing `AgentContextPruningConfig` contract unless maintainers explicitly prefer a separate boolean API.
