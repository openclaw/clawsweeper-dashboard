## Summary

Keep open: current main still lacks a per-cron-job cache/context-pruning override. The cron contract, protocol schema, CLI/UI surfaces, and isolated runner only expose payload-level `lightContext` while pruning remains driven by agent defaults; the earlier fixing PR is closed unmerged, and direct script execution is complementary rather than a replacement.

## What This Changes

Add per-cron-job cache/context-pruning control so isolated cron jobs can disable or tune prompt caching independently of global agent defaults.

## Best Possible Solution

Add a top-level cron `contextPruning?: AgentContextPruningConfig` override using the existing `mode: "off" | "cache-ttl"` contract, round-trip it through cron APIs and clients, and merge it into the effective isolated-run agent config for that job only.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection on current main gives high-confidence evidence that the requested job-level cache/pruning field is absent and pruning remains driven by agent defaults.

## Solution Assessment

Yes. Reusing the existing `contextPruning` object is the narrowest maintainable direction because it avoids a one-off `disableCache` boolean and preserves the current off/cache-ttl pruning contract.
