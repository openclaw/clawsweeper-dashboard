## Summary

Keep open: current main still lacks per-cron-job `contextPruning` or `disableCache`, the closed implementation attempt at https://github.com/openclaw/openclaw/pull/8341 was not merged, and isolated cron still derives pruning from effective agent defaults.

## What This Changes

The issue requests a per-cron-job cache or context-pruning override so simple isolated scheduled jobs can avoid prompt-cache costs.

## Best Possible Solution

Add a top-level `contextPruning?: AgentContextPruningConfig` cron field using the existing `off`/`cache-ttl` contract, round-trip it through cron APIs and clients, and apply it only to that job's effective isolated-run agent config.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection on current main verifies no job-level `contextPruning` or `disableCache` field and shows pruning still reads effective agent defaults.

## Solution Assessment

Yes. Reusing the existing `contextPruning` object is the narrowest maintainable direction because it avoids a one-off `disableCache` boolean and preserves the current pruning contract.
