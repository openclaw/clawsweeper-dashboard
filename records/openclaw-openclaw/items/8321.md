## Summary

Keep open. Current main and the latest release still lack per-cron-job cache/context-pruning control; the prior implementation PR is closed unmerged, and direct exec cron work is complementary rather than a replacement for agent-run cron pruning policy.

## What This Changes

The issue requests an optional per-cron-job cache/context-pruning override so simple isolated cron jobs can avoid prompt-cache retention.

## Best Possible Solution

Add a maintainer-approved, additive top-level cron contextPruning override that round-trips through persistence, gateway protocol, CLI, Control UI, docs, and isolated-run config composition, with regression coverage for job-level mode off overriding defaults-level cache-ttl.

## Reproduction Assessment

Yes, at source level. Current main lacks job-level disableCache or contextPruning in cron types, persistence/schema, gateway protocol, CLI/UI-facing surfaces, and isolated-run config composition.

## Solution Assessment

Unclear. A top-level contextPruning object appears more maintainable than a one-off boolean because it reuses the existing mode contract, but maintainers still need to decide the public cron API and merge semantics.
