## Summary

Keep open: current main still lacks a per-cron-job `contextPruning` or `disableCache` contract, the prior implementation attempt is closed and unmerged, and the isolated cron runner still derives pruning from effective agent defaults.

## What This Changes

Add per-cron-job cache/context-pruning control so isolated cron jobs can disable or tune prompt caching independently of global agent defaults.

## Best Possible Solution

Add a top-level `contextPruning?: AgentContextPruningConfig` cron field using the existing pruning contract, round-trip it through cron APIs and clients, and apply it only to that job's effective isolated-run agent config.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection on current main gives high-confidence evidence that the requested job-level cache/pruning field is absent and pruning is still default-driven.

## Solution Assessment

Yes. Reusing the existing `contextPruning` object is the narrowest maintainable direction because it avoids a one-off `disableCache` boolean and preserves the current `off`/`cache-ttl` pruning contract.
