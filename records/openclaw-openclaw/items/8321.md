## Summary

Keep open. Current main and the latest release still expose only payload-level `lightContext` for cron jobs; there is no per-job `disableCache` or `contextPruning` contract, and the prior implementation PR was closed unmerged.

## What This Changes

Add per-cron-job cache/context-pruning control so isolated cron jobs can disable or tune prompt caching independently of global `agents.defaults.contextPruning`.

## Best Possible Solution

Define an additive top-level cron-job `contextPruning` override, document its precedence, and thread it through storage, protocol, CLI, Control UI, docs, agent tools, and isolated-run config composition with regression coverage.

## Reproduction Assessment

Yes, at source level. Current main has no `disableCache` or `contextPruning` field in the cron job type, persisted base, closed gateway schemas, public cron docs, or isolated-run config composition.

## Solution Assessment

Unclear until maintainers choose the public API shape. Directionally, reusing the existing `AgentContextPruningConfig` contract looks more maintainable than a runner-only boolean if every cron surface is updated together.
