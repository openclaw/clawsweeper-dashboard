## Summary

Current main now addresses the reported deterministic shell-script examples through model-free command cron payloads and reduces bootstrap overhead through `lightContext`, but model-backed cron jobs still cannot override inherited context-pruning policy per job. The linked implementation PR closed unmerged and no viable canonical item owns this remaining behavior, so the issue remains necessary but requires a product decision on another persisted cron setting.

## What This Changes

Requests per-cron-job control over inherited context pruning or prompt caching for isolated agent-turn jobs.

## Best Possible Solution

Keep command payloads as the zero-LLM path for deterministic scripts; if maintainers approve the residual model-backed use case, add one canonical optional `contextPruning` job override using the existing `off` and `cache-ttl` modes, with explicit precedence and complete persistence, protocol, CLI, UI, docs, and regression coverage.

## Reproduction Assessment

Yes at source level: current main clearly lacks a per-job context-pruning field for model-backed cron payloads, while command payloads and `lightContext` are narrower supported alternatives.

## Solution Assessment

Partly. Command payloads are the best solution for deterministic scripts, but genuine model-backed jobs would need an additive override of the existing pruning contract rather than a separate `disableCache` boolean.
