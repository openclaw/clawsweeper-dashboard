## Summary

Keep this issue open. Current main still lacks per-cron-job cache/context-pruning control: the public cron job shape only carries payload-level `lightContext`, Gateway cron schemas reject top-level `contextPruning`/`disableCache`, and isolated cron still derives pruning from agent defaults rather than the job.

## What This Changes

Add a per-job cron option to disable or override context pruning/cache behavior for isolated cron jobs.

## Best Possible Solution

Implement an additive top-level `contextPruning?: AgentContextPruningConfig` cron job override using the existing `mode: "off" | "cache-ttl"` contract, thread it through cron persistence/API/CLI/UI/docs/tool surfaces, and apply it before isolated cron invokes the embedded runner.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: cron create/update schemas reject the proposed top-level fields, persisted cron types do not store them, and the isolated runner reads pruning only from `agents.defaults.contextPruning`.

## Solution Assessment

Yes. Based on the existing maintainer discussion, the narrowest maintainable path is a job-level `contextPruning` override rather than a boolean `disableCache`; provider cache-retention semantics should be documented separately if they are added later.
