## Summary

Keep open. Current main and the latest release still expose only payload-level `lightContext` for cron jobs, while prompt-cache pruning remains driven from `agents.defaults.contextPruning`; the closed implementation PR was not merged and the direct-script cron work is complementary rather than a replacement.

## What This Changes

The issue requests an additive per-cron-job cache/context-pruning override so isolated cron jobs that do not need LLM continuity can avoid prompt-cache costs.

## Best Possible Solution

Add an additive top-level cron `contextPruning?: AgentContextPruningConfig` override, thread it through persistence, gateway protocol, CLI, Control UI, docs, and cron tool surfaces, and merge it into the effective isolated-run config before the embedded runner evaluates pruning.

## Reproduction Assessment

Yes, at source level. Current main still has no job-level `disableCache` or `contextPruning` field in cron types, persistence, gateway schema, or isolated-run config composition.

## Solution Assessment

Unclear. The additive `contextPruning` direction looks more maintainable than a one-off boolean because it reuses the existing pruning contract, but maintainers still need to choose the public cron API and merge semantics.
