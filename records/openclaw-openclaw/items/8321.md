## Summary

Keep open. Current main and the latest release still lack a per-cron-job cache or context-pruning override, the earlier implementation PR is closed unmerged, and the direct script/no-LLM work is complementary rather than a replacement for agent-run cron jobs.

## What This Changes

Requests a per-cron-job cache/context-pruning control so isolated agent-turn cron jobs can disable or override prompt-cache pruning instead of inheriting global agent defaults.

## Best Possible Solution

Add an explicit, additive cron job contextPruning override using the existing mode contract, thread it through persistence, gateway schema, CLI, Control UI, docs, agent tool surfaces, and merge it into the effective isolated-run config before the embedded runner starts.

## Reproduction Assessment

Yes. Source inspection reproduces the gap: cron types, persistence, gateway schemas, CLI/UI-visible payloads, and isolated-run config composition expose lightContext but no disableCache or contextPruning override.

## Solution Assessment

Unclear. A full contextPruning override is likely the more maintainable direction than a one-off disableCache boolean, but maintainers still need to choose the public cron config contract before implementation.
