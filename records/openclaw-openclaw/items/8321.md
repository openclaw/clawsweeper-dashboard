## Summary

Keep open. Current main still has no per-cron-job `contextPruning` or `disableCache` field, the isolated cron runner still derives pruning from effective agent defaults, and the earlier implementation attempt at https://github.com/openclaw/openclaw/pull/8341 was closed unmerged.

## What This Changes

Request per-cron-job cache/context-pruning control so isolated cron jobs can disable or tune prompt-cache pruning independently of global agent defaults.

## Best Possible Solution

Add an additive top-level `contextPruning?: AgentContextPruningConfig` cron job override, round-trip it through cron APIs and clients, and apply it only to that job's effective isolated-run agent config.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection on current main gives high confidence that the requested job-level field is absent and pruning still reads from effective agent defaults.

## Solution Assessment

Yes. Reusing the existing `contextPruning` object is the narrowest maintainable direction because it avoids a one-off `disableCache` boolean and preserves the current `off`/`cache-ttl` contract.
