## Summary

Keep open. Current main still lacks per-job cache or context-pruning control for cron jobs, the prior implementation PR is closed unmerged, and the remaining work changes public cron config, protocol, UI, and tool surfaces that need maintainer API/product approval.

## What This Changes

The issue requests a per-cron-job cache or context-pruning override so simple isolated cron jobs can avoid unnecessary prompt caching costs.

## Best Possible Solution

Add an additive job-level contextPruning?: AgentContextPruningConfig override, round-trip it through cron storage, gateway, CLI, UI, tool, and docs surfaces, and merge it into the effective agent defaults for that scheduled run.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a strict bug repro, but yes at source level: current main has no disableCache or contextPruning path in cron job types, protocol add/update/list schemas, persistence, or isolated execution.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally, but the maintainable API should reuse AgentContextPruningConfig mode off/cache-ttl unless maintainers explicitly choose a simpler boolean surface.
