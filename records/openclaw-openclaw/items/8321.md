- `impact:session-state`: The request changes per-run context pruning and prompt-cache behavior for isolated cron sessions.

## Summary

Keep open. Current main still has no per-cron-job cache/context-pruning control; the closed implementation PR never merged, and adjacent direct-exec or per-agent context-pruning work does not replace per-job pruning for agent-run cron jobs.

## What This Changes

Add per-cron-job cache/context-pruning control so isolated jobs can disable prompt caching without changing global agent defaults.

## Best Possible Solution

Add an additive top-level cron-job contextPruning override using the existing AgentContextPruningConfig contract, then thread it through storage, protocol, CLI, UI, tool clients, docs, and isolated-run config composition.

## Reproduction Assessment

Yes, at source level. Current main has no disableCache or contextPruning path in cron job types, persisted shape, gateway schemas, public cron docs, Control UI/CLI surfaces, cron tool schema, or isolated execution config composition.

## Solution Assessment

Yes directionally, but the final API needs maintainer choice. A top-level contextPruning override reusing AgentContextPruningConfig looks more maintainable than a runner-only boolean, but it must be threaded through every public cron surface.
