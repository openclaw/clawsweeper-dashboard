## Summary

Keep open. Current main still lacks a per-cron-job cache/context-pruning control, the only closing implementation PR is closed and unmerged, and the remaining choice is the public cron API shape rather than whether the source path exists.

## What This Changes

Add per-cron-job control to disable or override context pruning/prompt-cache behavior for isolated cron jobs.

## Best Possible Solution

Add an additive top-level cron-job contextPruning override using the existing AgentContextPruningConfig contract, then thread it consistently through cron storage, protocol, CLI, UI, docs, generated clients, and isolated-run config composition.

## Reproduction Assessment

Yes, at source level. Current main has no disableCache or contextPruning field in the cron job contract, closed gateway schemas, public cron docs, or isolated-run config composition.

## Solution Assessment

Yes directionally, but the final API needs maintainer choice. Reusing the existing contextPruning contract looks more maintainable than a runner-only boolean, provided every public cron surface is updated together.
