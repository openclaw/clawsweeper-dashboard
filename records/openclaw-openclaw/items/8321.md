## Summary

Keep this open: current main and v2026.6.6 support command cron jobs for deterministic no-LLM scripts, but isolated agent-turn cron jobs still cannot set per-job cache/context-pruning policy.

## What This Changes

Request per-cron-job cache or context-pruning control for isolated scheduled jobs so simple agent-run cron work can avoid unnecessary prompt-cache token cost.

## Best Possible Solution

Decide whether OpenClaw wants a per-job agent-turn pruning override; if yes, add an additive `contextPruning?: AgentContextPruningConfig` cron job field across types, protocol, storage, CLI, UI, docs, and runner config composition with regression coverage for `mode: "off"` overriding global cache-ttl.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction: current cron types and closed protocol schemas do not accept `contextPruning` or `disableCache`, and the embedded runner reads pruning only from `agents.defaults.contextPruning`. I did not run a live CLI/API repro because this review is read-only.

## Solution Assessment

No, current main is not the complete solution: command cron jobs are the right path for deterministic scripts, but they do not give agent-turn cron jobs per-job pruning control. If maintainers accept the feature, an additive `contextPruning` override using the existing `off | cache-ttl` contract is cleaner than a one-off `disableCache` flag.
