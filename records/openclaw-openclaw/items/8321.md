## Summary

Keep open: current main still lacks per-cron-job `contextPruning` or `disableCache`, the closed implementation attempt was not merged, and isolated cron still derives pruning from the effective agent defaults.

## What This Changes

The issue requests an additive per-cron-job cache/context-pruning override so isolated agent cron jobs can disable prompt-cache pruning independently of global defaults.

## Best Possible Solution

Add a top-level `contextPruning?: AgentContextPruningConfig` cron field using the existing `off`/`cache-ttl` contract, round-trip it through cron APIs and clients, and apply it only to that job's effective isolated-run agent config.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection on current main verifies no job-level `contextPruning`/`disableCache` field and pruning still reads `agents.defaults.contextPruning`.

## Solution Assessment

Yes. Reusing the existing `contextPruning` object is the narrowest maintainable direction because it avoids a one-off `disableCache` boolean and preserves the current pruning contract.
