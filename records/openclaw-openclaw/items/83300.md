## Summary

Keep open: the ACPX trace attribution is useful and not obsolete, but the patch currently exposes the raw gateway startup trace through the public plugin service context and still lacks real split-branch Gateway trace proof.

## What This Changes

The PR threads startup tracing into plugin service contexts, adds ACPX-owned startup/probe subspans, records restart resource-count metrics, and adds focused tests for those paths.

## Best Possible Solution

Expose a scoped per-service trace wrapper, or an ACPX-internal seam, that prefixes and encodes subspans under the existing service trace name, then add redacted Gateway trace output from this branch as real behavior proof.

## Reproduction Assessment

Yes for the review finding: source inspection of the diff shows the raw gateway `startupTrace` is forwarded into every public plugin service context. The user-facing feature itself is observability work, so there is no bug reproduction path to run against current main.

## Solution Assessment

No. The ACPX subspan attribution is the right direction, but the trace object should be scoped before crossing the public plugin-service boundary so owner attribution cannot collide with core or other plugin trace names.
