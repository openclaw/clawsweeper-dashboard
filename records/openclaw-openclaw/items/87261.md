## Summary

Keep open: the durability direction is useful, but this branch still uses required durable Telegram action sends without the replay-safety contract that current recovery requires, and the external PR proof remains synthetic-only.

## What This Changes

The PR routes Telegram `sendMessage` actions through the durable outbound batch path, carries pin scope through the outbound adapter, and adds focused queue/pin regression tests.

## Best Possible Solution

Keep the durable action goal, but either add and prove a real Telegram unknown-send reconciliation path or keep this action path best-effort/direct until Telegram can satisfy required durability safely.

## Reproduction Assessment

Yes, from source inspection: the PR sets Telegram action sends to required durability, while current Telegram capabilities omit `reconcileUnknownSend` and recovery refuses blind replay for post-send-start unknown states. I did not run tests because this review is read-only.

## Solution Assessment

No. Routing through the durable queue is the right direction, but required durability is not the safest implementation until Telegram can reconcile unknown sends or the action path remains best-effort-compatible.
