## Summary

Keep open: the issue is recent, externally reported with detailed Windows/Codex timings, and current `main` still has the same 60s `models.authStatus` cache plus cold auth/usage work that can plausibly match the reported cold-session latency. Related performance issues and the Microsoft tracker are useful context, but none is a merged fix or exact canonical replacement for this Windows CLI/Codex OAuth path.

## What This Changes

The issue reports Windows npm-global Codex-backed CLI sessions on OpenClaw 2026.5.18 spending 10-24s in cold `models.authStatus`, making trivial `openclaw agent` runs take 27-45s.

## Best Possible Solution

Keep this open and make normal `models.authStatus` reads cheap by coalescing or background-refreshing heavy auth/usage work while preserving explicit refresh and auth invalidation, then prove the Windows Codex OAuth CLI path before closing.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction from this review. The issue gives concrete Windows steps and timings, and current source confirms the same cache/cold-work shape, but the Windows Codex OAuth path still needs live timing proof.

## Solution Assessment

Yes for keeping it open; the narrow maintainable direction is to remove blocking cold `models.authStatus` work from normal reads, not only raise a timeout or TTL. The exact cache, coalescing, and freshness policy needs maintainer judgment because auth/quota status must remain refreshable.
