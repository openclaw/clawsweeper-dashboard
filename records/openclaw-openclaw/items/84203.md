## Label Justifications

- `P1`: The report describes a recent regression that can make a trivial Windows Codex-backed CLI run take tens of seconds, with a remaining Telegram-related multiplier after the workaround.
- `impact:auth-provider`: The affected path is model auth status, Codex OAuth profile discovery, provider usage enrichment, and provider/runtime routing.

## Summary

Keep open: the reporter's latest update shows the Codex 0.132 path substantially improves the original regression, but it does not prove the Windows CLI path is back under the expected latency target, and Telegram enabled still appears to multiply latency. Current main still has the same process-local `models.authStatus` cache and cold auth/usage work that match the reported pattern.

## What This Changes

The issue asks OpenClaw to remove Windows cold-session `models.authStatus` latency from Codex-backed CLI runs so trivial responses are fast again.

## Best Possible Solution

Profile the current Windows Codex OAuth CLI path with and without Telegram, then make normal `models.authStatus` reads cheap through coalescing or background refresh while preserving explicit refresh and invalidation; split a Telegram-specific follow-up only if profiling proves it is a separate bottleneck.

## Reproduction Assessment

No, not at high confidence. The reporter provided concrete Windows steps and timing logs, and current source confirms the matching cold auth-status path, but no live Windows 11 npm-global plus Codex OAuth current-main run was executed.

## Solution Assessment

Yes for keeping it open and requiring profiling; there is no proposed patch to accept. The maintainable fix is not only raising a TTL, but bounding or decoupling cold auth/usage work while preserving freshness controls.
