## Summary

Keep open: the report is recent, externally authored, detailed, and current `main` still has the same `models.authStatus` cold path and 60s handler cache that match the reported latency pattern. The source supports a plausible auth/provider-status regression, but it still needs live Windows + Codex OAuth profiling before choosing the cache/freshness fix.

## What This Changes

The issue reports 10-24s cold `models.authStatus` latency on Windows 11 with OpenClaw 2026.5.18 and the Codex plugin, making trivial `openclaw agent` CLI runs take 27-45s.

## Best Possible Solution

Keep the issue open and make `models.authStatus` cheap for normal gateway/CLI reads by coalescing or background-refreshing heavy auth/usage work while preserving explicit refresh and invalidation, then prove the Windows Codex OAuth CLI path.

## Reproduction Assessment

No, we do not have a high-confidence local reproduction. The reporter provided concrete Windows steps and timings, and current source confirms the 60s cache plus cold auth/usage path, but no live Windows/Codex OAuth run was performed here.

## Solution Assessment

Yes for keeping it open; the narrow maintainable fix is to remove blocking cold `models.authStatus` work from normal reads rather than only raising a constant. The exact cache/coalescing shape should be validated against auth freshness and explicit refresh behavior.
