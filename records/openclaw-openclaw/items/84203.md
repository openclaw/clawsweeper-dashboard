## Summary

Keep open: current main still has the same 60-second `models.authStatus` cache and blocking cold auth/usage path that match the reported Windows Codex latency pattern, and the related open report at https://github.com/openclaw/openclaw/issues/83963 is broader Control UI startup latency rather than a duplicate of this Windows CLI cold-session case.

## What This Changes

The issue reports a Windows 11 regression where each cold Codex-backed `openclaw agent` CLI session spends 10-24s in `models.authStatus` and 27-45s total for a trivial response.

## Best Possible Solution

Keep this issue open, profile the current Windows Codex OAuth cold CLI path, and make normal `models.authStatus` reads cheap through coalescing or background refresh while preserving explicit refresh and invalidation.

## Reproduction Assessment

No, we do not have a high-confidence current-main reproduction. The reporter provided concrete Windows steps and timings, and source inspection confirms the same 60s cache plus cold auth/usage path, but no live Windows Codex OAuth run was performed here.

## Solution Assessment

Yes for keeping it open; the best current path is trace-guided performance repair rather than closing or guessing. The exact implementation should be chosen after measuring the Windows cold path and preserving explicit auth-status freshness controls.
