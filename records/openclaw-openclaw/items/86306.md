Keep this PR open, but do not merge it as-is: the change is in the right area, yet it only scopes the picker search/load-more RPC and leaves the initial no-search picker rows backed by the unscoped session list; it also appears to leave existing request-parameter tests stale and has no real behavior proof.

## What This Changes

The PR adds the active agent id to the Control UI chat session picker `sessions.list` request params in `ui/src/ui/chat/session-controls.ts`.

## Best Possible Solution

Scope both the default picker rows and search/load-more RPC to the active agent, update focused chat picker tests, and include redacted real Control UI proof with multiple configured agents.

## Reproduction Assessment

Yes for source reproduction: current main renders the no-search picker from `state.sessionsResult` when `chatSessionPickerResult` is null, and the shipped request builder lacks `agentId`. I did not run a live browser flow because this review is read-only.

## Solution Assessment

No: adding `agentId` to the picker RPC is necessary but not sufficient because the default popover rows remain unfiltered and focused tests are not updated. The narrow maintainable fix should reuse the backend `sessions.list` agent filter and cover the no-search picker case.
