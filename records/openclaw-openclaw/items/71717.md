Keep open. Current main still has a plausible Windows host-exec spawn-time failure path that could surface `spawn EPERM`, and the related Microsoft tracker is an open draft tracker rather than a merged fix.

## What This Changes

The issue reports that every exec command sent through Feishu or the web UI on Windows 11 fails immediately with `spawn EPERM` while the CLI itself runs.

## Best Possible Solution

Capture native Windows exec diagnostics first, then add either a constrained validated fallback or clearer diagnostic on the host spawn path with focused regression coverage.

No. The issue gives plausible Windows 11/npm-global steps and current source shows where an `EPERM` from `child_process.spawn` would propagate, but there is no native Windows run or log identifying the selected shell, cwd, service context, or original error object.

## Solution Assessment

Unclear. A targeted diagnostic or constrained fallback may be appropriate, but a generic `EPERM` retry or bypass in host exec would be risky without native Windows evidence.
