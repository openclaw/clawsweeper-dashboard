## Summary

Keep open. Current main still has a plausible Windows host-exec failure path: Windows exec selects PowerShell, non-PTY exec passes that argv through the supervisor child adapter, and the shared spawn fallback still retries only EBADF by default, not EPERM. The related Microsoft tracker is open inventory, not a merged fix.

## What This Changes

Report that on Windows 11 with OpenClaw 2026.4.23 installed via npm, every agent exec command from Feishu or the web UI immediately fails with `spawn EPERM` while the CLI itself works.

## Best Possible Solution

Add a focused native Windows exec investigation path that records selected shell path, cwd, command text, service/user context, and original spawn error fields, then implement either a constrained validated fallback or a clearer diagnostic with mocked coverage and a native Windows smoke check.

## Reproduction Assessment

No. The issue gives plausible Windows 11 steps and current main still contains a code path where spawn-time EPERM would fail every command, but there is no native Windows reproduction log or selected-shell evidence in the report or checkout.

## Solution Assessment

Unclear. There is no proposed patch, and a blanket EPERM retry is not the best first move for host exec; the safer path is targeted Windows reproduction followed by a constrained fallback or diagnostic.
