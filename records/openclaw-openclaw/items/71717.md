## Summary

Keep open. Current main includes the related exec-approvals EPERM fallback from https://github.com/openclaw/openclaw/pull/77907, but this report specifically says `spawn EPERM`; the Windows host-exec spawn path can still propagate spawn-time EPERM, and there is no native Windows log tying this issue to the already-fixed approvals writer.

## What This Changes

The issue reports Windows 11 host exec commands from Feishu or the web UI failing immediately with `spawn EPERM` on OpenClaw 2026.4.23.

## Best Possible Solution

Capture native Windows exec diagnostics first, then add a constrained host-spawn diagnostic or fallback only if the failing shell/path and original spawn error prove the root cause.

## Reproduction Assessment

No. Source inspection shows a spawn-time EPERM would propagate through the current Windows host-exec path, but there is no native Windows run or log proving the all-commands failure on current main.

## Solution Assessment

Unclear. The merged approvals fallback is the right fix for the separate `exec-approvals.json` rename EPERM, but this issue needs targeted Windows evidence before changing shell selection or spawn retry behavior.
