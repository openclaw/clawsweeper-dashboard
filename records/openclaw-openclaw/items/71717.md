## Summary

Keep open. Current main still has a plausible Windows host-exec spawn failure path matching the report: Windows exec selects PowerShell, passes it through the child-process supervisor adapter, and the shared spawn fallback still only retries EBADF by default. The related Microsoft tracker is an open inventory item, not a merged or shipped fix for this failure.

## What This Changes

The issue reports that on Windows 11 all agent exec-tool commands fail immediately with `spawn EPERM` while the OpenClaw CLI itself works.

## Best Possible Solution

Add a focused native Windows exec investigation path that records the selected shell path, cwd, command text, service/user context, and original spawn error fields, then implement either a constrained validated fallback or a clearer diagnostic with mocked Windows coverage and a native Windows smoke check.

## Reproduction Assessment

No. The report gives plausible Windows 11 steps and current main still has a matching spawn-time failure path, but there is no native Windows log or selected-shell evidence to reproduce the exact EPERM source with high confidence.

## Solution Assessment

Unclear. There is no proposed patch, and a blanket EPERM retry is not the safest first fix for host exec; targeted Windows reproduction should decide between a constrained fallback and a diagnostic improvement.
