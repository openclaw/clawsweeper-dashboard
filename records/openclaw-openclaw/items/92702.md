## Summary

Keep this PR open for normal maintainer review: current main still has the sessionKey rebind gap, the patch targets the owning MCP runtime manager, and I found no blocking correctness or security finding in the diff.

## What This Changes

The PR extracts shared MCP session disposal logic, disposes the prior registered unleased runtime when a sessionKey is rebound to a new sessionId, and adds regression tests for rebind disposal and leased-runtime preservation.

## Best Possible Solution

Land the manager-level sessionKey rebind cleanup after maintainer lifecycle review, while keeping the reset-timeout cleanup in https://github.com/openclaw/openclaw/pull/92696 as adjacent work for a different transition path.

## Reproduction Assessment

Yes, by source inspection. Current main can persist a new sessionId for the same sessionKey after compaction, and getOrCreate then remaps the key while leaving the prior registered runtime alive until idle sweep.

## Solution Assessment

Yes. The session MCP runtime manager owns the sessionKey-to-sessionId invariant, and disposing only the prior registered unleased runtime is narrower than per-server restart policy, MCP-server self-healing, or changing the session-reset path alone.
