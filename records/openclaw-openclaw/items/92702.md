## What This Changes

The PR refactors session MCP runtime disposal into a shared helper, disposes the prior registered unleased runtime when a sessionKey is rebound to a new sessionId, and adds regression coverage for rebind disposal and leased-runtime preservation.

## Best Possible Solution

Land the manager-level sessionKey rebind cleanup after normal maintainer review, while treating the separate gateway reset-timeout PR as adjacent work for a different cleanup path.

## Reproduction Assessment

Yes. Source inspection shows current main can persist a new sessionId for the same sessionKey after compaction, then getOrCreate remaps that sessionKey while leaving the prior registered runtime alive until idle sweep.

## Solution Assessment

Yes. The manager owns the sessionKey to sessionId invariant, and disposing only the prior registered unleased runtime is narrower and safer than per-server restart policy or MCP-server self-healing.
