## Summary

Keep open. Current main still sends `openclaw logs --follow` fetch failures through the fatal Gateway error path, and the discussion already links open PR #75059 as an unmerged implementation candidate, so this issue should wait for that PR or an equivalent fix to land.

## What This Changes

The issue requests that `openclaw logs --follow` auto-reconnect with warning/backoff on transient Gateway disconnects instead of ending the log-tail session.

## Best Possible Solution

Land or adapt one focused CLI change that retries only transient follow-mode Gateway transport failures with bounded backoff and clear warning output, while preserving fatal handling for non-recoverable errors and the existing implicit-loopback local log fallback.

## Reproduction Assessment

Yes. The manual path is to run `openclaw logs --follow`, restart the Gateway, and observe the CLI exit; the code-level path is to make `callGatewayFromCli` reject with a closed or timed-out `GatewayTransportError` inside the follow loop.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs around `src/cli/logs-cli.ts` follow polling with targeted retry classification and tests that preserve explicit URL behavior and the implicit-loopback fallback boundary.
