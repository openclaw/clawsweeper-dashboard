## Summary

Keep open. The checked-out `main` still exits from the `openclaw logs --follow` polling loop on non-fallback Gateway transport errors, and the provided GitHub timeline shows open PR #75059 uses closing syntax for this issue, so the issue should wait for that PR or an equivalent fix to land.

## What This Changes

Request retry/reconnect behavior for `openclaw logs --follow` when transient Gateway disconnects occur during follow-mode polling.

## Best Possible Solution

Land or adapt one focused CLI fix that retries only transient follow-mode Gateway transport failures with bounded backoff and clear warning output, while preserving existing local fallback and fatal handling for non-recoverable errors.

## Reproduction Assessment

Yes. The manual path is to run `openclaw logs --follow`, restart or interrupt the Gateway, and observe the CLI exit; the code-level path is to mock a closed or timed-out `GatewayTransportError` in the follow loop with an explicit `--url`.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs in `src/cli/logs-cli.ts` with targeted tests in `src/cli/logs-cli.test.ts` and docs/changelog updates, rather than changing every one-shot Gateway RPC call.
