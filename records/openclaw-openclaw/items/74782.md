## Summary

Keep open. Current main still has the reported fatal path for non-fallback Gateway transport failures in `openclaw logs --follow`, and PR #75059 is an open unmerged implementation candidate with closing syntax for this issue, so the issue should wait for that PR or an equivalent fix to land.

## What This Changes

The issue requests that `openclaw logs --follow` auto-reconnect with warning/backoff on transient Gateway disconnects instead of exiting immediately.

## Best Possible Solution

Ship a single focused CLI implementation that retries only transient follow-mode Gateway transport failures with bounded backoff and clear warning output, while preserving the existing implicit-loopback local log fallback and fatal handling for non-recoverable errors.

## Reproduction Assessment

Yes. The manual path is to run `openclaw logs --follow`, restart or interrupt the Gateway, and observe the CLI exit; the code-level path is to mock a closed or timed-out `GatewayTransportError` in the follow loop with an explicit `--url`.

## Solution Assessment

Yes in direction. The narrow maintainable solution belongs around `src/cli/logs-cli.ts` follow polling, with tests in `src/cli/logs-cli.test.ts` and docs updates for the retry behavior rather than a broad change to all one-shot Gateway RPC calls.
