## Summary

Keep open. Current `main` still exits from the `openclaw logs --follow` loop on non-fallback Gateway transport errors, and the provided timeline shows open PR #75059 uses closing syntax for this issue, so the issue should wait for that PR or an equivalent fix to land.

## What This Changes

Request retry/reconnect behavior for `openclaw logs --follow` when transient Gateway disconnects occur during follow-mode polling.

## Best Possible Solution

Land or adapt one focused CLI fix that retries only transient follow-mode Gateway transport failures with bounded backoff and clear warning output, while preserving local fallback and fatal error handling.

## Reproduction Assessment

Yes. The manual path is to run `openclaw logs --follow`, restart or interrupt the Gateway, and observe the CLI exit; the code-level path is the existing explicit-URL closed `GatewayTransportError` test expectation.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs in `src/cli/logs-cli.ts` with targeted tests and docs/changelog updates, rather than changing every one-shot Gateway RPC call.
