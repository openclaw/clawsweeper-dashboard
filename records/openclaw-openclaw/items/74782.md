## Summary

Keep open. Current main still exits from the `openclaw logs --follow` polling loop on non-fallback Gateway transport failures, and open PR #75059 is linked with closing syntax as an implementation candidate, so this issue should remain open until that PR, #75372, or an equivalent fix lands.

## What This Changes

Request that `openclaw logs --follow` retry transient Gateway WebSocket/RPC disconnects with warning/backoff instead of exiting immediately.

## Best Possible Solution

Land or adapt a focused CLI change that retries only transient `logs --follow` Gateway transport failures with bounded backoff and warning output, while preserving fatal handling for non-recoverable errors and the existing implicit-loopback local-file fallback boundary.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a `callGatewayFromCli`/`fetchLogs` transport rejection inside the follow loop; current tests already assert that an explicit Gateway URL close error exits with status 1.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs around `src/cli/logs-cli.ts` follow polling, with targeted tests for explicit-URL transient retry behavior and preservation of implicit-loopback fallback semantics.
