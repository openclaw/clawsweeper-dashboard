## Summary

Keep open. Current main still has the fatal follow-loop error path, and #75059 is an open closing PR for the same CLI surface, so closing this issue before merge would lose the canonical tracker.

## What This Changes

Request `openclaw logs --follow` to reconnect with warning/backoff after transient Gateway disconnects instead of terminating.

## Best Possible Solution

Land or adapt one focused CLI fix that retries only transient follow-mode Gateway transport failures with bounded backoff and clear warning output while preserving fatal handling and the existing local file fallback.

## Reproduction Assessment

Yes. The manual path is to run `openclaw logs --follow`, restart or interrupt the Gateway, and observe the CLI exit; the code path is the `fetchLogs` rejection inside the follow loop reaching `process.exit(1)`.

## Solution Assessment

Yes in direction. The narrow maintainable solution belongs in `src/cli/logs-cli.ts` with targeted tests, docs, and changelog coverage rather than adding blanket retries to every one-shot Gateway RPC call.
