## Summary

Keep open. Current `main` still exits from the `openclaw logs --follow` polling loop on non-fallback Gateway transport errors, and open PR #75059 uses closing syntax for this issue, so the issue should wait for that PR or an equivalent fix to land.

## What This Changes

Request that `openclaw logs --follow` retry transient Gateway disconnects with warning/backoff instead of exiting immediately.

## Best Possible Solution

Land or adapt one focused CLI fix that retries only transient follow-mode Gateway transport failures with bounded backoff and clear notice output, while preserving existing local fallback and fatal error handling.

## Reproduction Assessment

Yes. The manual path is to run `openclaw logs --follow`, restart or interrupt the Gateway, and observe the CLI exit; the code-level path is the existing explicit-URL test that mocks a closed `GatewayTransportError` and expects `process.exit(1)`.

## Solution Assessment

Yes in direction. The narrow maintainable solution belongs in `src/cli/logs-cli.ts` with targeted tests and docs/changelog updates, rather than changing all one-shot Gateway RPC behavior.
