## Summary

Keep open. Current main still exits on non-fallback Gateway transport failures in the `openclaw logs --follow` polling loop, and open PR #75059 uses closing syntax for this issue, so cleanup should wait for that PR or an equivalent canonical fix to land.

## What This Changes

The issue requests that `openclaw logs --follow` auto-reconnect with warning/backoff on transient Gateway disconnects instead of exiting immediately.

## Best Possible Solution

Land or adapt one focused CLI implementation that retries only transient follow-mode Gateway transport failures with bounded backoff and clear warning output, while preserving the existing implicit-loopback local log fallback and fatal handling for non-recoverable errors.

## Reproduction Assessment

Yes. The manual path is to run `openclaw logs --follow`, restart or interrupt the Gateway, and observe the CLI exit; the code-level path is to mock a closed or timed-out `GatewayTransportError` in the follow loop with an explicit `--url`.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs around `src/cli/logs-cli.ts` follow polling with targeted retry classification, focused tests in `src/cli/logs-cli.test.ts`, and docs/changelog updates rather than broad auto-retry for every one-shot Gateway RPC.
