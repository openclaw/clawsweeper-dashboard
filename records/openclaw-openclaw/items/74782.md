## Summary

Keep open. Current main still exits on non-fallback Gateway transport failures in the `openclaw logs` follow loop, and PR #75059 is an open, unmerged implementation candidate linked to this issue, so cleanup should wait for that PR or an equivalent fix to land.

## What This Changes

The issue requests retry/backoff behavior for `openclaw logs --follow` after transient Gateway disconnects instead of terminating the tail session.

## Best Possible Solution

Land or adapt a focused CLI change that retries only transient follow-mode Gateway transport failures with bounded backoff and clear warning output, while preserving fatal handling for non-recoverable errors and the existing implicit-loopback local log fallback.

## Reproduction Assessment

Yes. The manual path is to run `openclaw logs --follow`, restart the Gateway, and observe the CLI exit; the code-level path is to mock `callGatewayFromCli` rejecting with a closed or timed-out `GatewayTransportError` in the follow loop.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs around `src/cli/logs-cli.ts` follow polling with targeted retry classification, tests for explicit URL retry behavior, and preservation of the implicit-loopback fallback boundary.
