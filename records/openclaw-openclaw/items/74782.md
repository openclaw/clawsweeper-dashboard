## Summary

Keep open. Current main still treats `openclaw logs --follow` Gateway fetch failures as fatal, and open PR #75059 is already linked as the closing implementation candidate with PR #75372 overlapping the same retry surface.

## What This Changes

Request `openclaw logs --follow` to retry transient Gateway disconnects with warning/backoff behavior instead of exiting immediately.

## Best Possible Solution

Land or adapt one focused CLI follow-mode retry implementation that retries only transient transport failures while preserving local fallback and fatal-error handling.

## Reproduction Assessment

Yes. Source inspection shows the follow loop exits after a `fetchLogs` transport failure, and the manual path is to run `openclaw logs --follow`, restart or interrupt the Gateway, and observe the CLI terminate.

## Solution Assessment

Yes in direction. The narrow maintainable solution belongs in `src/cli/logs-cli.ts` with targeted tests and docs, rather than adding blanket retries to every one-shot Gateway RPC call.
