## Summary

Keep open. Current main still treats non-fallback `openclaw logs --follow` Gateway transport failures as fatal, and open PR #75059 is an unmerged implementation candidate with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests auto-reconnect, warning output, and backoff behavior for `openclaw logs --follow` after transient Gateway disconnects.

## Best Possible Solution

Land or adapt a focused CLI change that retries only transient `logs --follow` transport failures with bounded backoff and clear warning output, while preserving fatal handling for non-recoverable Gateway errors and the existing local fallback boundary.

## Reproduction Assessment

Yes. A high-confidence code reproduction is to mock `callGatewayFromCli` rejecting with a closed or timed-out `GatewayTransportError` inside the follow loop; current `src/cli/logs-cli.test.ts` already asserts the explicit-URL close path exits with status 1.

## Solution Assessment

Yes in direction. The narrow maintainable solution belongs around `src/cli/logs-cli.ts` follow polling with targeted retry classification and tests for explicit-URL retry behavior plus implicit-loopback fallback preservation.
