## Summary

Keep open. Current main still treats non-fallback Gateway transport failures in the `openclaw logs --follow` polling path as fatal, and PR #75059 is an open, unmerged implementation candidate linked with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Request that `openclaw logs --follow` keep tailing across transient Gateway WebSocket disconnects by warning and retrying with backoff instead of exiting immediately.

## Best Possible Solution

Land or adapt a focused CLI fix that retries only transient follow-mode Gateway transport failures with bounded backoff and warning output, preserves fatal handling for non-recoverable errors, keeps the existing local fallback boundary intact, and updates targeted tests, docs, and the changelog if the user-facing behavior changes.

## Reproduction Assessment

Yes. For the remaining gap, a `fetchLogs` or `callGatewayFromCli` transport rejection in the follow loop still reaches the catch block that exits, and the explicit-URL close test already asserts fatal behavior.

## Solution Assessment

Yes in direction. The narrowest maintainable fix belongs around `src/cli/logs-cli.ts` follow polling rather than changing the shared one-shot Gateway RPC helper, with tests covering explicit-URL retry and implicit-loopback fallback preservation.
