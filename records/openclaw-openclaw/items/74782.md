## Summary

Keep open. Current main still exits on non-fallback `openclaw logs --follow` Gateway failures, and open PR #75059 uses closing syntax for the same behavior, so this issue should stay open until that PR or an equivalent canonical fix lands.

## What This Changes

The issue requests warning output and bounded auto-reconnect behavior for `openclaw logs --follow` after transient Gateway disconnects.

## Best Possible Solution

Land one focused CLI follow-loop fix that retries only transient Gateway transport close/timeout failures with bounded backoff and appropriate text/JSON notices, preserves existing fallback boundaries, updates docs/changelog as needed, and then let the merged PR close this issue.

## Reproduction Assessment

Yes. A high-confidence code reproduction is to make `callGatewayFromCli` reject with a closed or timed-out `GatewayTransportError` during `openclaw logs --follow --url ...`; current main reaches the fatal `process.exit(1)` path, and the existing explicit-URL test already asserts that behavior.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs in `src/cli/logs-cli.ts` around follow-mode polling, with tests proving transient retry, fatal non-transient errors, JSON-mode notices, and preservation of the implicit-loopback fallback.
