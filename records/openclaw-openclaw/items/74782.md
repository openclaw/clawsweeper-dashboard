## Summary

Keep open. Current main still exits from the `logs --follow` polling loop after a Gateway fetch failure that is not eligible for implicit local-file fallback, and open PR #75059 already uses closing syntax with PR #75372 overlapping the same retry surface.

## What This Changes

The issue asks for `openclaw logs --follow` to retry or reconnect with backoff on transient Gateway disconnects instead of terminating the tail session.

## Best Possible Solution

Review and land one focused follow-mode retry implementation that classifies transient transport failures, preserves implicit local fallback, and leaves fatal auth, pairing, and configuration errors terminal.

## Reproduction Assessment

Yes, for the remaining explicit or non-local Gateway tail path. Source inspection shows `openclaw logs --follow --url ws://127.0.0.1:18789` reaches the catch block and exits after a transport failure; the original implicit loopback case is now partially mitigated by local-file fallback, so I did not mark this as live-reproduced.

## Solution Assessment

Yes in direction. The narrow maintainable solution belongs in `src/cli/logs-cli.ts` with targeted tests and docs, and should be reconciled between #75059 and #75372 rather than adding blanket retries to every one-shot Gateway RPC call.
