## Summary

Keep open. Current main still drops keyRef/tokenRef-only auth profiles when building PI credentials, so auth-backed model availability and runtime discovery can treat configured OpenRouter auth as unavailable even though the canonical auth-profile resolver supports those refs.

## What This Changes

Make PI model discovery resolve keyRef/tokenRef-backed auth profiles so OpenRouter models are treated as authenticated in model picker and runtime model-selection paths.

## Best Possible Solution

Fix the generic auth-profile-to-PI bridge in the agents auth discovery layer: resolve api_key and token profiles through the shared auth-profile resolver, keep OpenRouter out of core conditionals, keep resolved secrets memory-only, and add focused regression tests for keyRef-only and tokenRef-only PI discovery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an auth profile for openrouter with type api_key, keyRef, and no inline key: current resolvePiCredentialMapFromStore() returns no openrouter credential, while the persisted store intentionally strips the inline key when keyRef exists.

## Solution Assessment

Yes. Reusing resolveApiKeyForProfile() or the same shared resolver is the narrowest maintainable fix because it aligns PI discovery with runtime auth resolution without adding an OpenRouter-specific or duplicate SecretRef resolver.
