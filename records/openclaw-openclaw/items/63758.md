## Summary

Keep open. Current main still has the reported cron producer gap: the task ledger and status surfaces support active `progressSummary`, but scheduled, startup catch-up, and manual cron task rows are created without one and only receive terminal summaries at finish time.

## What This Changes

Managed cron task runs should publish coarse live progress into the background-task ledger so `/tasks` and `session_status` are informative while cron work is active.

## Best Possible Solution

Keep this issue open and fix the cron producer path: seed useful `progressSummary` text when scheduled, startup catch-up, and manual cron task rows are created; update it through `recordTaskRunProgressByRunId` at meaningful execution phases; keep cron's `silent` notification default; and add focused regression coverage proving active cron work has visible detail before `terminalSummary` is written.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: follow a scheduled, startup catch-up, or manual cron run through `tryCreateCronTaskRun`/`tryCreateManualTaskRun` and observe that no progress field is written until finalization, while active status rendering only uses `progressSummary` for detail.

## Solution Assessment

Yes, the best fix is in the cron producer lifecycle rather than the shared task registry or status renderers. The registry already supports progress, and other producers use it, so cron should call the existing seam and preserve current notification policy.
