## Summary

Keep open: current main and v2026.6.8 still route ordinary heartbeat runs through the normal reply/tool path unless the due-commitment branch disables tools, and the related heartbeat allowlist PR is open, conflicting, and only a partial skill-filter candidate.

## What This Changes

The issue requests safety guardrails so unattended heartbeat polls cannot use mutating tools or auto-merge fabricated work without an explicit user/workstream signal.

## Best Possible Solution

Adopt a maintainer-approved heartbeat safety policy that defaults unattended heartbeat turns to non-mutating or narrow tools, requires explicit audited opt-in for write/exec paths, and covers routine heartbeat, HEARTBEAT.md tasks, due commitments, cron wakes, and exec-event wakes with docs and tests.

## Reproduction Assessment

Yes at source level: ordinary heartbeat runs can still enter getReplyFromConfig without disableTools or toolsAllow, and undefined toolsAllow keeps the full runtime tool surface. The exact stochastic model hallucination was not replayed.

## Solution Assessment

Unclear until maintainers choose the product/security policy. The maintainable direction is trigger-scoped tool narrowing with explicit audited opt-in for mutating heartbeat work, not prompt-only NO_REPLY compliance.
