## Summary

Keep open: current main and v2026.6.9 still route ordinary heartbeat runs into the normal reply path without a heartbeat-scoped `disableTools` or `toolsAllow` policy, so the destructive unattended-tool authority problem remains a real security/product item.

## What This Changes

The issue reports that unattended heartbeat polls can hallucinate work and use mutating tools such as file writes and shell commands without a user request.

## Best Possible Solution

Adopt a maintainer-approved heartbeat safety policy that defaults unattended heartbeat turns to non-mutating or narrow tools, requires explicit audited opt-in for write/exec paths, and covers routine heartbeat, HEARTBEAT.md tasks, due commitments, cron wakes, and exec-event wakes with docs and tests.

## Reproduction Assessment

Yes at source level: routine heartbeat runs still enter `getReplyFromConfig` without `disableTools` or `toolsAllow`, and undefined `toolsAllow` keeps the normal runtime tool surface. The stochastic hallucinated wiki edit itself was not replayed.

## Solution Assessment

Unclear until maintainers choose the security policy. A prompt-only `HEARTBEAT_OK` fix or skill-filter-only API is not sufficient for the callable write/exec boundary; the durable path is trigger-scoped tool narrowing with explicit audited opt-in for mutating heartbeat work.
