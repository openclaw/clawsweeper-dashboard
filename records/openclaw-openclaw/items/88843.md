## Summary

Keep open: current main still leaves ordinary heartbeat runs on the normal tool path unless a separate caller disables tools, so the reported safety boundary is not already fixed. The exact hallucination is stochastic, but the architecture that can give heartbeat turns mutating tools is source-verifiable.

## What This Changes

The issue asks OpenClaw to prevent autonomous heartbeat polling sessions from using destructive tools or auto-merging fabricated work when no real user request exists.

## Best Possible Solution

Add a core heartbeat safety policy that defaults autonomous heartbeat turns to a narrow, non-mutating tool set, with explicit audited opt-in for write/exec paths and regression coverage for heartbeat, due-task, and cron-wake behavior.

## Reproduction Assessment

No high-confidence replay exists for the exact stochastic hallucination. Source inspection does give high-confidence proof of the architectural precondition: a heartbeat run can proceed without disableTools and, with no toolsAllow, construct the normal coding tool surface.

## Solution Assessment

Unclear until maintainers choose the product policy. The best direction appears to be trigger-scoped tool narrowing with explicit opt-in for mutating heartbeat work, rather than relying on prompt compliance alone.
