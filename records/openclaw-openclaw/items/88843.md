## Summary

Keep open: current main and v2026.6.8 still let ordinary heartbeat runs enter the normal reply path without `disableTools` or `toolsAllow`; the open allowlist PR is only a partial skill-filter candidate and does not settle mutating tool authority.

## What This Changes

The issue reports that autonomous heartbeat polling can retain destructive tool access, allowing a hallucinated workstream to create and merge fabricated wiki edits without a user request.

## Best Possible Solution

Adopt a maintainer-approved heartbeat safety policy that defaults unattended heartbeat turns to non-mutating or narrow tools, requires explicit audited opt-in for write/exec paths, and covers routine heartbeat, HEARTBEAT.md tasks, due commitments, cron wakes, and exec-event wakes with docs and tests.

## Reproduction Assessment

Yes at source level: ordinary heartbeat runs enter `getReplyFromConfig` with no `disableTools` or `toolsAllow`, while undefined `toolsAllow` keeps the normal runtime tool surface. The stochastic hallucinated wiki edit itself was not replayed.

## Solution Assessment

Unclear until maintainers choose the security policy. A prompt-only NO_REPLY fix or skill-filter-only API is not sufficient for the callable write/exec boundary; the durable path is trigger-scoped tool narrowing with explicit audited opt-in for mutating heartbeat work.
