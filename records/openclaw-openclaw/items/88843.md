## Summary

Keep open: current main and v2026.6.8 still leave routine heartbeat runs on the normal reply/tool path unless a due-commitment branch disables tools, and the open heartbeat allowlist PR is only a partial, unmerged candidate rather than a settled safety boundary.

## What This Changes

The issue asks OpenClaw to prevent autonomous heartbeat polls from using mutating tools or auto-landing fabricated work when no user request exists.

## Best Possible Solution

Adopt a maintainer-approved heartbeat safety policy that defaults unattended heartbeat turns to non-mutating or narrow tools, adds explicit audited opt-in for write/exec paths, and covers routine heartbeat, HEARTBEAT.md tasks, due commitments, cron wakes, and exec-event wakes with tests and docs.

## Reproduction Assessment

Yes at source level: ordinary heartbeat runs can still enter getReplyFromConfig without disableTools or toolsAllow, while undefined toolsAllow builds the full coding tool surface. No high-confidence replay exists for the exact stochastic hallucination event from the external wiki session.

## Solution Assessment

Unclear until maintainers choose the product/security policy. The best direction is trigger-scoped tool narrowing with explicit audited opt-in for mutating heartbeat work, not relying on HEARTBEAT.md or prompt-only NO_REPLY compliance.
