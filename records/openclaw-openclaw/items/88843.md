## Summary

Keep open: current main still lets ordinary heartbeat turns reach the normal agent reply path without a heartbeat-scoped no-tools or read-only allowlist, so the central safety boundary is not implemented. The exact wiki incident is stochastic and private, but the architecture that permits mutating heartbeat tool access is source-verifiable.

## What This Changes

The issue asks OpenClaw to prevent autonomous heartbeat polling from receiving or using destructive tools when no user-initiated work exists.

## Best Possible Solution

Add a core heartbeat safety policy that defaults autonomous heartbeat turns to a non-mutating tool set, with explicit audited opt-in for write/exec paths and regression coverage for idle, due-task, and cron-wake heartbeat behavior.

## Reproduction Assessment

Yes for the architectural issue: current source shows routine heartbeat turns do not force disableTools or a no-tools allowlist before entering embedded agent tool construction. No for the exact hallucination replay, which depends on a private stochastic model trajectory.

## Solution Assessment

Unclear until maintainers choose the policy: prompt compliance alone is not enough, but the exact default/opt-in shape for mutating heartbeat work needs product and security judgment.
