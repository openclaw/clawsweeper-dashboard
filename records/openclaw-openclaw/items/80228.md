## Summary

Keep this PR open: it is an active feature PR with missing real behavior proof and review-blocking safety/default-activation concerns, not a conservative cleanup-close candidate.

## What This Changes

The PR adds a Codex continuity bridge to the bundled Codex plugin with new gateway routes, Telegram-facing `/codex` commands, watch/handoff state, write gates, docs, tests, startup activation, and a `fast-uri` override.

## Best Possible Solution

Keep the bridge work in review, make write dry-run a true no-op, make background Codex startup opt-in or lazy, wire the docs into the existing Codex docs surface, and require redacted real runtime proof before merge.

## Reproduction Assessment

Not applicable as a current-main bug report, but the review findings are source-reproducible from the PR diff: the write path checks dry-run after app-server requests, and startup activation leads to an immediate bridge poll.

## Solution Assessment

No; the feature direction may be useful, but this patch is not yet the best implementation because it makes dry-run mutate, starts Codex by default, and lacks real behavior proof for the new runtime surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep dry-run from submitting Codex turns:** `extensions/codex/src/continuity/bridge.ts:303-315`
  - body: `telegramDryRun` is checked only after the app-server request has already been sent. With writes enabled and methods confirmed, an operator testing dry-run mode can still submit `turn/start` or `turn/steer`, so move the dry-run/no-op branch before creating the shared client or calling `client.request`.
  - confidence: 0.92
- **[P2] Avoid starting Codex from every gateway startup:** `extensions/codex/openclaw.plugin.json:22`
  - body: Changing the manifest to `onStartup: true` loads this plugin as a gateway sidecar, and the registered service immediately calls `checkWatches`, whose snapshot reads the Codex app-server before checking whether any watches exist. That makes ordinary gateway startup start/probe Codex by default; keep activation opt-in or make the bridge lazy until an explicit watch/route needs it.
  - confidence: 0.86
- **[P3] Publish the bridge docs through the Codex docs surface:** `docs/codex-continuity-bridge.md:1`
  - body: The new docs page is added at the docs root but is not added to `docs/docs.json` navigation or linked from the existing Codex harness docs, so users browsing the published docs will not discover it. Move/link it under the Codex plugin docs and add the navigation entry with the rest of the Codex harness pages.
  - confidence: 0.77
