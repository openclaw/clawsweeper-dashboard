  - body: `telegramDryRun` is checked only after the bridge has submitted `turn/steer` or `turn/start` to Codex, so a dry-run configuration can still mutate the target thread. Return before acquiring the client or sending the write, while recording any audit state without dispatching the request.
  - confidence: 0.97
- **[P1] Normalize app-server thread status objects:** `extensions/codex/src/continuity/app-server-reader.ts:81-85`
  - body: Current Codex `thread/list` returns `Thread.status` as a tagged object such as `{ type: "active" }`, not a string. The bridge's string-only read path maps real active, idle, and system-error states to `unknown`, making active-thread display and watch decisions unreliable.
  - confidence: 0.94
- **[P2] Read goals from the Codex goals database:** `extensions/codex/src/continuity/config.ts:34-39`
  - body: The SQLite fallback defaults to `state_5.sqlite` and joins `thread_goals` there, but current Codex stores goals in `goals_1.sqlite` through a separate GoalStore. On current installs this fallback fails instead of providing the stale-labeled goal/thread data promised by the bridge.
  - confidence: 0.92
- **[P1] Keep the bridge opt-in on startup:** `extensions/codex/openclaw.plugin.json:22`
  - body: Changing the bundled Codex plugin to `onStartup: true` while `codexBridge.enabled` defaults to true makes existing installs start the bridge and poll Codex without an explicit user choice. Leave startup activation/default bridge enablement opt-in, or include a maintainer-approved upgrade path and proof for the background behavior.
  - confidence: 0.89
