## Summary

Keep open: this is a same-day external feature PR with active follow-up, no implementation on current main, missing real behavior proof, and blocking source-review findings in the new Codex bridge write/startup/command surfaces.

## What This Changes

The PR adds a bundled Codex continuity bridge with gateway routes, Telegram-facing `/codex` bridge commands, watch and handoff state, write policy gates, docs, tests, startup activation, and a `fast-uri` override.

## Best Possible Solution

Keep the bridge work in review, make dry-run a true no-op before any app-server request, preserve existing `/codex status` and `/codex threads` behavior, make bridge startup opt-in or lazy, wire docs into the Codex docs surface, and require redacted real runtime proof before merge.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The review findings are high-confidence from source inspection of the PR diff against current main command, app-server, startup, and docs contracts.

## Solution Assessment

No; the feature direction may be useful, but this patch is not yet the best implementation because dry-run can still submit Codex turns, existing `/codex` commands are intercepted, default startup probes Codex, and real behavior proof is missing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep dry-run from submitting Codex turns:** `extensions/codex/src/continuity/bridge.ts:303-315`
  - body: `telegramDryRun` is checked only after the bridge has already sent `turn/steer` or `turn/start` to the Codex app-server. With writes enabled and methods confirmed, a dry-run operator can still mutate Codex state; return the dry-run result before creating the shared client or calling `client.request`.
  - confidence: 0.94
- **[P2] Preserve existing Codex status and thread commands:** `extensions/codex/src/command-handlers.ts:191-193`
  - body: The new continuity dispatcher runs before the existing `/codex status` and `/codex threads [filter]` handlers, so the bridge consumes both commands whenever it is registered. That removes the documented app-server account/rate-limit/MCP/skills status output and ignores the existing thread filter; expose bridge status under a distinct subcommand or let the existing handlers fall through.
  - confidence: 0.88
- **[P2] Avoid starting Codex from every gateway startup:** `extensions/codex/openclaw.plugin.json:22`
  - body: Setting the Codex plugin to `onStartup: true` makes gateway startup load the plugin service, and `start()` immediately calls `checkWatches`, whose snapshot reads the Codex app-server before checking whether any watch exists. That makes ordinary gateway startup start or probe Codex by default; keep activation opt-in or make the bridge lazy until an explicit watch or route needs it.
  - confidence: 0.87
- **[P3] Publish the bridge docs through the Codex docs surface:** `docs/codex-continuity-bridge.md:1`
  - body: The new docs page is added at the docs root but is not added to `docs/docs.json` navigation or linked from the existing Codex harness docs. Users browsing the published docs will not discover it; move or link it under the Codex plugin docs and add the navigation entry with the other Codex pages.
  - confidence: 0.78
