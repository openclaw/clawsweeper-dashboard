Full review comments:

- **[P1] Keep dry-run from submitting Codex turns:** `extensions/codex/src/continuity/bridge.ts:328-329`
  - body: `telegramDryRun` is checked only after the shared app-server client has sent `turn/start` or `turn/steer`. With writes enabled and methods confirmed, an operator testing dry-run mode can still submit a real Codex turn, so move the dry-run/no-op branch before creating the client or calling `client.request`.
  - confidence: 0.93
- **[P2] Avoid starting Codex from every gateway startup:** `extensions/codex/openclaw.plugin.json:22`
  - body: Changing the manifest to `onStartup: true` makes the Codex plugin load as a gateway startup plugin, and the registered service immediately starts the bridge and backfill check. That path snapshots Codex state before any explicit watch or route is used, so ordinary gateway startup can start/probe Codex by default; keep activation opt-in or make bridge reads lazy.
  - confidence: 0.87
- **[P2] Preserve the existing Codex status and threads commands:** `extensions/codex/src/command-handlers.ts:191-194`
  - body: The new continuity handler runs before the existing `status` and `threads` branches, and it consumes both subcommands. In a registered runtime, `/codex status` no longer returns the app-server model/account/rate-limit/MCP/skills probes and `/codex threads <filter>` no longer passes the filter to `thread/list`; put bridge status/threads behind `bridge`/`continuity` or otherwise preserve the established commands.
  - confidence: 0.82
- **[P3] Publish the bridge docs through the Codex docs surface:** `docs/codex-continuity-bridge.md:1`
  - body: The new page is added at the docs root but is not wired into `docs/docs.json` or linked from the existing Codex harness docs, so users browsing the published docs will not discover it. Add it under the Codex plugin docs/navigation or link it from the existing Codex docs surface.
  - confidence: 0.78
