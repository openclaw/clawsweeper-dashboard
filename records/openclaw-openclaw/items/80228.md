## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Honor dry run before sending Codex writes:** `extensions/codex/src/continuity/bridge.ts:343-344`
  - body: `telegramDryRun` is evaluated only after the bridge has already submitted `turn/steer` or `turn/start` to Codex. A dry-run configuration therefore still mutates the target Codex thread before returning `{ dryRun: true }`; return before acquiring the client or sending the write, while still recording audit if needed.
  - confidence: 0.97
- **[P1] Preserve existing `/codex status` and `/codex threads`:** `extensions/codex/src/command-handlers.ts:191-193`
  - body: Calling the continuity handler before the existing command handlers means the new bridge takes over established `/codex status` and `/codex threads` output whenever the bridge singleton exists. Keep the current commands intact and put bridge-specific status/threads behind a new subcommand or explicit opt-in path.
  - confidence: 0.94
- **[P1] Keep the bridge opt-in on startup:** `extensions/codex/openclaw.plugin.json:22`
  - body: Changing the bundled Codex plugin to `onStartup: true` while `codexBridge.enabled` defaults to true makes existing installs start the bridge and poll Codex without an explicit user choice. Leave startup activation/default bridge enablement opt-in, or include a maintainer-approved upgrade path and proof for the background behavior.
  - confidence: 0.9
- **[P1] Normalize the real app-server thread status:** `extensions/codex/src/continuity/app-server-reader.ts:81-85`
  - body: Current Codex `thread/list` returns `Thread.status` as a tagged object such as `{ type: "active" }`, not a string, and the list payload does not include the goal status this code falls back to. Live app-server snapshots can normalize active/terminal state as `unknown`, so `/codex watch` cannot reliably detect active or completed work.
  - confidence: 0.93
- **[P2] Read goals from the goals DB or remove the fallback:** `extensions/codex/src/continuity/sqlite-reader.ts:81-83`
  - body: The SQLite fallback joins `thread_goals` inside `state_5.sqlite`, but current Codex stores goals in `goals_1.sqlite` and drops `thread_goals` from the state DB. On a real current Codex install this fallback can fail with a schema error instead of providing the stale-labeled thread data promised by the bridge.
  - confidence: 0.92
