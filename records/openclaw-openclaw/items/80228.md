## Summary

Keep open: source review still shows blocking command-regression, dry-run write-path, default startup activation, docs-discovery, and missing real behavior proof issues before this can merge.

## What This Changes

The PR adds a bundled Codex continuity bridge with Telegram/gateway `/codex` routes and commands, watch and handoff state, gated Codex write support, docs, tests, and a `fast-uri` override.

## Best Possible Solution

Keep the bridge plugin-local, preserve existing `/codex` behavior, make dry-run a true no-op before any app-server call, keep startup lazy or explicitly opted in, publish docs under the Codex plugin docs surface, and require redacted runtime proof before merge.

## Reproduction Assessment

Not applicable as a current-main bug report. Yes for the PR review findings: the command shadowing, dry-run ordering, and startup activation paths are directly visible from PR-head source plus current-main startup contracts.

## Solution Assessment

No. The feature direction may be useful, but this implementation is not the narrowest maintainable solution until it avoids existing command regressions, makes write safety controls pre-call, keeps startup lazy or explicitly opted in, and includes real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep dry-run from submitting Codex turns:** `extensions/codex/src/continuity/bridge.ts:303-315`
  - body: `telegramDryRun` is checked only after the app-server request has already been sent. With writes enabled and methods confirmed, an operator testing dry-run mode can still submit `turn/start` or `turn/steer`, so move the dry-run/no-op branch before creating the shared client or calling `client.request`.
  - confidence: 0.93
- **[P2] Keep bridge dispatch from shadowing existing commands:** `extensions/codex/src/command-handlers.ts:191-193`
  - body: Because the bridge singleton is registered during plugin setup, this early dispatch returns for `status` and `threads` before the existing handlers run. That replaces the current `/codex status` probes and ignores `/codex threads <filter>`, so only intercept new bridge-specific verbs or put bridge status under a non-conflicting subcommand.
  - confidence: 0.9
- **[P2] Avoid starting Codex from every gateway startup:** `extensions/codex/openclaw.plugin.json:22`
  - body: Changing the manifest to `onStartup: true` loads the Codex plugin as a gateway sidecar, and the registered service immediately starts watch checking. That can start or probe Codex on ordinary gateway startup; keep activation opt-in or make the bridge lazy until an explicit route, watch, or command needs it.
  - confidence: 0.87
- **[P3] Publish bridge docs through Codex docs navigation:** `docs/codex-continuity-bridge.md:1`
  - body: The new docs page is added at the docs root but is not added to `docs/docs.json` navigation or linked from the existing Codex plugin docs, so users browsing the published docs will not discover it.
  - confidence: 0.78
