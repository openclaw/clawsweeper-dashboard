## Summary

Keep open: current main does not implement the bridge, and the PR still has merge-blocking issues in write safety, existing `/codex` command compatibility, gateway startup behavior, HTTP side-effect guards, docs discoverability, and missing real behavior proof.

## What This Changes

The PR adds a Codex continuity bridge to the bundled Codex plugin with Telegram/gateway `/codex` routes, watch and handoff state, gated write controls, docs, tests, and a `fast-uri` override.

## Best Possible Solution

End state should be a plugin-local bridge with explicit opt-in writes, no existing `/codex` regressions, lazy or explicitly opted-in activation, discoverable Codex plugin docs, and redacted live Telegram/gateway proof.

## Reproduction Assessment

Not applicable as a current-main bug; this is a feature PR. Yes for the review findings: PR-head source shows dry-run after `client.request`, bridge dispatch before existing status/threads handlers, `onStartup: true` plus service startup, and a watch-check route without a POST guard.

## Solution Assessment

No. The bridge direction may be useful, but this implementation should first preserve existing command behavior, make safety controls pre-call, keep startup lazy or explicitly opted in, add the missing method guard, and publish proof.
