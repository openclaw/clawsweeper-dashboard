## Summary

Keep open: this is a plausible but security-sensitive feature PR, current main does not implement the bridge, and PR head still has blocking review findings plus missing real behavior proof.

## What This Changes

The branch adds a Codex continuity bridge to the bundled Codex plugin, including `/codex` Telegram/gateway routes, watch and handoff state, gated writes, docs, tests, and a `fast-uri` override.

## Best Possible Solution

Keep the bridge plugin-local, preserve existing `/codex` behavior, make write safety controls pre-call, keep startup opt-in or lazy, publish docs through the Codex plugin docs surface, and require redacted runtime proof before merge.

## Reproduction Assessment

Not applicable as a current-main bug report. Yes for the PR review findings: PR-head source shows dry-run after app-server writes, continuity dispatch before existing `status`/`threads`, and startup polling through current-main service contracts.

## Solution Assessment

No. The feature direction may be useful, but this implementation should first avoid existing command regressions, make write safety controls true no-ops before app-server calls, keep startup lazy or explicitly opted in, and add real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep dry-run from submitting Codex turns:** `extensions/codex/src/continuity/bridge.ts:303-315`
  - body: `telegramDryRun` is checked only after the app-server request has already been sent. With writes enabled and methods confirmed, an operator testing dry-run mode can still submit `turn/start` or `turn/steer`, so move the dry-run/no-op branch before creating the shared client or calling `client.request`.
  - confidence: 0.93
- **[P2] Keep bridge dispatch from shadowing existing commands:** `extensions/codex/src/command-handlers.ts:191-193`
  - body: Because the bridge singleton is registered during plugin setup, this early dispatch returns for `status` and `threads` before the existing handlers run. That replaces current `/codex status` probes and ignores `/codex threads <filter>`, so only intercept new bridge-specific verbs or put bridge status under a distinct subcommand.
  - confidence: 0.9
- **[P2] Avoid starting Codex from every gateway startup:** `extensions/codex/openclaw.plugin.json:22`
  - body: Changing the manifest to `onStartup: true` loads the Codex plugin as a gateway sidecar, and the registered service immediately schedules `checkWatches`, whose snapshot reads app-server threads before loading watches. That can start or probe Codex during ordinary gateway startup; keep activation opt-in or lazy until a route/watch needs it.
  - confidence: 0.87
- **[P2] Require POST before checking watches:** `extensions/codex/src/continuity/http.ts:86-89`
  - body: `/codex/watch/check` is documented as POST-only, but this route calls `checkWatches` for any method. Since `checkWatches` can send Telegram notifications, add the same method guard used by the other side-effecting routes before invoking the bridge.
  - confidence: 0.82
- **[P3] Publish bridge docs through Codex docs navigation:** `docs/codex-continuity-bridge.md:1`
  - body: The new docs page is added at the docs root but is not added to `docs/docs.json` navigation or linked from the existing Codex plugin docs, so users browsing the published docs will not discover it. Move or link it under the Codex plugin docs surface and add the navigation entry with the rest of the Codex pages.
  - confidence: 0.78
