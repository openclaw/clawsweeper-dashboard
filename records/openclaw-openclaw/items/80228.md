## Summary

Keep this PR open. The bridge is a substantial new feature, but the patch currently regresses existing `/codex` command behavior, has write-safety issues, lacks after-fix real behavior proof, fails a prompt snapshot CI gate, and includes a root dependency override that needs explicit maintainer security review.

## What This Changes

This PR adds a Codex continuity bridge to the bundled Codex plugin with new commands, gateway HTTP routes, state stores, config schema, SQLite fallback, docs, tests, and a `fast-uri` override.

## Best Possible Solution

Keep the PR open until it preserves existing `/codex` commands, fails closed on write provenance and dry-run paths, refreshes required prompt snapshots, supplies redacted real behavior proof, and either gets explicit approval for the dependency override or splits it out.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the PR registers the bridge globally at plugin startup and dispatches continuity commands before the existing `status` and `threads` handlers, so those established commands are shadowed in real runtime. The write-safety issues are also visible directly in the new policy and bridge source.

## Solution Assessment

No. The maintainable path is to add the bridge without stealing existing command semantics, make all write gates fail closed before mutation, keep generated prompt artifacts aligned, and handle the dependency override through explicit maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require a sender id for trusted Telegram writes:** `extensions/codex/src/continuity/policy.ts:123-127`
  - body: When `trustedTelegramSenders` is configured, omitting `requestedBySenderId` skips the allowlist entirely because the check only runs when that field is present. A write request with valid provenance, repo allowlist, and confirmed methods can therefore bypass the sender gate advertised in the PR safety model; reject missing sender ids whenever the allowlist is non-empty.
  - confidence: 0.92
- **[P2] Honor write dry-run before mutating Codex:** `extensions/codex/src/continuity/bridge.ts:343-345`
  - body: `telegramDryRun` is only checked after `turn/steer` or `turn/start` has already been sent to the Codex app-server, so a response marked `dryRun: true` can still mutate a real Codex thread. Move the dry-run return before acquiring the client and sending the write, or remove the dry-run marker if this flag is not meant to cover writes.
  - confidence: 0.86
- **[P2] Preserve the existing Codex status and thread commands:** `extensions/codex/src/command-handlers.ts:191-193`
  - body: Because the plugin now installs a global bridge at startup, this early dispatch intercepts `status` and `threads` before the established handlers run. That drops the documented `/codex status` account/rate/MCP/skills readout and makes `/codex threads <filter>` ignore the existing filter/resume behavior; keep bridge status under a bridge-specific subcommand or only intercept the new subcommands.
  - confidence: 0.9
- **[P2] Escape thread ids in watch replies:** `extensions/codex/src/continuity/commands.ts:38-48`
  - body: The new watch command echoes Codex thread ids directly in chat responses. Existing Codex command tests already treat thread ids as untrusted display text because they can contain markdown, mentions, or control characters, so these replies should use the same formatter or otherwise avoid pasteable unsafe content.
  - confidence: 0.87
- **[P2] Refresh prompt snapshots for the startup prompt change:** `extensions/codex/openclaw.plugin.json:22`
  - body: CI shows `prompt:snapshots:check` failing because `test/fixtures/agents/prompt-snapshots/codex-runtime-happy-path/discord-group-codex-message-tool.md` differs from generated output. Since this PR changes Codex startup/plugin prompt surface, the matching snapshot needs to be regenerated or the prompt change narrowed so the check stays green.
  - confidence: 0.82
