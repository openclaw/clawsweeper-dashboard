## Summary

Keep open. This PR adds a useful but new Codex/Telegram bridge, and it is not merge-ready because the diff still has source-proven safety/startup issues and no required real behavior proof.

## What This Changes

The branch adds a Codex continuity bridge with gateway routes, Telegram-facing `/codex` commands, watch and handoff state, gated Codex write support, docs, tests, startup activation, and a `fast-uri` override.

## Best Possible Solution

Keep the bridge in review, make dry-run a true no-op, make bridge activation lazy or explicitly opt-in, publish the docs through the Codex docs surface, refresh prompt snapshots, and require redacted real Telegram/Codex runtime proof before merge.

## Reproduction Assessment

Not applicable as a current-main bug report. The PR defects are source-reproducible from the diff: the dry-run guard is after the app-server write request, and startup activation leads to an immediate bridge poll and app-server snapshot read.

## Solution Assessment

No. The feature direction may be useful, but this patch is not yet the best solution because it mutates during dry-run, changes default startup behavior, leaves docs/prompt snapshots incomplete, and lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep dry-run from submitting Codex turns:** `extensions/codex/src/continuity/bridge.ts:303-315`
  - body: `telegramDryRun` is checked only after the bridge has already sent `turn/steer` or `turn/start`. With writes enabled and methods confirmed, a dry-run request can still mutate Codex before returning a dry-run decision, so move the no-op branch before acquiring the shared client or calling `client.request`.
  - confidence: 0.93
- **[P2] Avoid starting Codex from every gateway startup:** `extensions/codex/openclaw.plugin.json:22`
  - body: Setting the manifest to `onStartup: true` loads the plugin as a gateway sidecar, and the registered service immediately calls `checkWatches`; `snapshot()` then reads app-server threads before checking watches. That makes ordinary gateway startup probe or start Codex by default, so keep the bridge lazy or explicitly opt-in.
  - confidence: 0.88
- **[P3] Publish the bridge docs in the Codex docs nav:** `docs/codex-continuity-bridge.md:1`
  - body: The new page is added at the docs root, but it is not in `docs/docs.json` navigation or linked from the existing Codex harness docs. Published docs users will not discover it; move or link it under the Codex plugin docs and add the nav entry.
  - confidence: 0.78
- **[P3] Refresh prompt snapshots for the new help entries:** `extensions/codex/src/command-formatters.ts:237-239`
  - body: CI reports `prompt:snapshots:check` drift after these new `/codex` help entries. Update the generated prompt snapshot for the intended command surface, or avoid adding prompt-visible help until the commands are ready to ship.
  - confidence: 0.82
