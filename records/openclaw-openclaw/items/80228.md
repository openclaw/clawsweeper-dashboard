## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep dry-run from submitting Codex turns:** `extensions/codex/src/continuity/bridge.ts:303-315`
  - body: `telegramDryRun` is checked only after this method has already sent `turn/steer` or `turn/start` to the Codex app-server. A dry-run write request can therefore mutate a real Codex thread before the bridge reports `dryRun: true`, which breaks the advertised safety control.
  - confidence: 0.96
- **[P2] Normalize Codex thread status objects:** `extensions/codex/src/continuity/app-server-reader.ts:81-85`
  - body: Current Codex app-server `Thread.status` is a tagged object, not a plain string or boolean. This code reads only string/boolean fields before calling `normalizeStatus`, so active app-server threads become `unknown` and watch/status selection can fail against the preferred live source.
  - confidence: 0.9
- **[P2] Preserve existing Codex status and threads commands:** `extensions/codex/src/command-handlers.ts:191-193`
  - body: The continuity dispatcher runs before the existing command handlers, and `commands.ts` consumes both `status` and `threads`. That replaces the current app-server status probe output and drops the existing `/codex threads [filter]` behavior instead of extending it.
  - confidence: 0.94
- **[P2] Keep the bridge off ordinary gateway startup:** `extensions/codex/openclaw.plugin.json:22`
  - body: Flipping `onStartup` to true makes the plugin service start on gateway startup; the service immediately schedules and backfills `checkWatches`, which calls `snapshot` and probes Codex app-server/SQLite before knowing whether any watch exists. Existing users who only rely on Codex harness activation should not pay that startup behavior without an explicit opt-in or upgrade proof.
  - confidence: 0.9
- **[P2] Require POST before checking watches:** `extensions/codex/src/continuity/http.ts:86-89`
  - body: `/codex/watch/check` runs `checkWatches` without checking `req.method`, so a GET to this trusted route can perform side effects and send Telegram notifications. Match the neighboring side-effect routes and reject non-POST requests before invoking the bridge.
  - confidence: 0.92
- **[P3] Publish bridge docs through the docs nav:** `docs/codex-continuity-bridge.md:1`
  - body: This adds a public docs page, but it is not wired into `docs/docs.json` near the existing Codex plugin docs. The page will be hard to discover in the published docs unless the navigation is updated with the new bridge page.
  - confidence: 0.78
