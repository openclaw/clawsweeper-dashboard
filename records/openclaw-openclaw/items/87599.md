## Summary

Keep open: current main and the latest release still gate ACP `resumeSessionId` on requester-owned local `acp.identity`, so the requested explicit backend-session restore path remains missing and needs a maintainer-owned authorization decision.

## What This Changes

The issue requests an explicit ACP backend-session restore path that can create fresh OpenClaw ACP metadata and bindings for a known valid Claude/backend session after local ACP metadata was cleaned up.

## Best Possible Solution

Define a separate explicit backend-session restore contract, or deliberately change the existing `resumeSessionId` contract, so authorized recovery creates fresh ACP metadata and bindings while stale wrappers remain non-respawnable.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: remove or lack a local `acp.identity` match for the backend ID, then call `sessions_spawn` with that `resumeSessionId`; current main returns `resume_forbidden` before ACP runtime initialization.

## Solution Assessment

Unclear as a direct code fix. The requested direction is plausible, but maintainers need to choose whether to add a distinct backend-restore operation, broaden `resumeSessionId`, or keep the local-recorded-only contract and fix the docs/tool wording.
