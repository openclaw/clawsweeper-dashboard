## Label Justifications

- `P2`: This is a bounded but real ACP session-continuity failure with a fresh-session workaround and a security-sensitive design decision.
- `impact:session-state`: The failure is caused by ACP identity metadata changing from resolved to pending state, which prevents restoring prior backend session context.
- `impact:auth-provider`: The requested recovery path depends on proving authorization for a Claude/ACP backend session id without weakening requester ownership checks.

## Summary

Keep open: current main and latest release still require a requester-owned resolved ACP metadata record for `resumeSessionId`, while reset cleanup deliberately downgrades ACP identity to pending metadata without the backend session id. The user-visible idle-reset workaround can delay cleanup, but it does not provide the requested recovery path once metadata has already been stripped.

## What This Changes

The issue requests an authorized way to create a fresh ACP wrapper around a known valid Claude/backend session id after local ACP metadata cleanup removes the existing binding.

## Best Possible Solution

Maintainers should define an explicit backend-restore authorization contract that creates a fresh ACP wrapper and binding from a known backend id while preserving the stale-respawn safeguards from the closed stale-session issue.

## Reproduction Assessment

Yes, at source level: reset cleanup writes pending ACP metadata without `agentSessionId` or `acpxSessionId`, and `sessions_spawn` with that backend id cannot satisfy the current ownership validator. I did not run a live Discord/Claude ACP repro in this read-only review.

## Solution Assessment

Unclear as a direct code change: the requested split between ACP resume and backend restore is plausible, but maintainers need to choose the durable ownership/admin proof model before implementation.
