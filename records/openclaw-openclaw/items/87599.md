## Summary

Keep open: current main and v2026.6.8 still reject ACP `resumeSessionId` unless local ACP metadata contains a resolved backend id owned by the requester; reset cleanup intentionally strips those backend ids into pending metadata, so the explicit backend-session restore path remains missing.

## What This Changes

The issue requests an authorized way to restore a valid Claude/ACPX backend session after ACP metadata cleanup removed the local identity mapping.

## Best Possible Solution

Define a maintainer-approved explicit backend-restore contract, or deliberately broaden `resumeSessionId`, so authorized recovery creates a fresh ACP wrapper and binding while preserving stale-respawn safeguards.

## Reproduction Assessment

Yes at source level: after reset cleanup writes pending ACP metadata without `agentSessionId` or `acpxSessionId`, calling `sessions_spawn` with that backend id cannot match the current ownership validator and returns `resume_forbidden` before runtime initialization.

## Solution Assessment

Unclear as a direct code fix. The requested restore direction is plausible, but maintainers need to choose the authorization model and whether this is a new admin-only operation or a changed `resumeSessionId` contract.
