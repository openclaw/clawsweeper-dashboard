## Summary

Keep open: current main and the latest release still gate ACP `resumeSessionId` on requester-owned local `acp.identity`, so the requested explicit backend-session restore path remains missing; the newer idle-reset workaround can preserve metadata longer but does not create an authorized restore path once metadata is gone.

## What This Changes

The issue requests an explicit, authorized way to create a fresh ACP wrapper around a known valid Claude/backend session ID after local ACP metadata cleanup.

## Best Possible Solution

Maintainers should either define a separate authorized backend-session restore contract that creates fresh ACP metadata and bindings, or explicitly keep the local-recorded-only contract and align the docs/tool wording with that decision.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `sessions_spawn` with `runtime: "acp"` and a backend `resumeSessionId` that has no matching local `acp.identity` for the requester; current main returns `resume_forbidden` before runtime initialization.

## Solution Assessment

Unclear as a direct fix. The requested split between ACP resume and backend restore is plausible, but maintainers need to choose the authorization model and decide whether the existing `resumeSessionId` contract should broaden or stay local-recorded-only.
