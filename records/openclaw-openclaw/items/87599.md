## Label Justifications

- `P2`: This is a bounded but real ACP session-continuity failure with a fresh-session workaround and a security-sensitive design decision.
- `impact:session-state`: The failure occurs when ACP lifecycle cleanup removes resolved identity metadata needed to continue prior backend session context.
- `impact:auth-provider`: The requested restore path depends on proving authorization for a Claude/ACPX backend session id without weakening provider/session ownership checks.

## Summary

Keep open: current main and v2026.6.9 still authorize ACP `resumeSessionId` only when a resolved ACP identity is already recorded for the requester, while reset cleanup intentionally rewrites ACP identity to pending metadata without backend ids. The merged SQLite metadata work and the idle-reset workaround reduce how often metadata is lost, but they do not provide the requested explicit backend-session restore path after cleanup.

## What This Changes

Requests an explicit authorized ACP backend-session restore path after ACP metadata cleanup removes the local resolved identity mapping.

## Best Possible Solution

Define a maintainer-approved backend-restore contract that creates a fresh ACP wrapper and binding for authorized recovery while preserving the stale-respawn safeguards fixed by related ACP lifecycle work.

## Reproduction Assessment

Yes, at source level. After reset cleanup writes pending ACP metadata without `agentSessionId` or `acpxSessionId`, `sessions_spawn` with that backend id cannot match the current ownership validator and returns `resume_forbidden` before runtime initialization.

## Solution Assessment

Unclear as a direct code fix. The requested recovery direction is plausible, but maintainers need to choose the authorization model and whether this becomes a new admin-only backend-restore operation or a changed `resumeSessionId` contract.
