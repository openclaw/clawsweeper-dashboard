## Label Justifications

- `P2`: This is a real but edge-case ACP restore/session-continuity bug with a fresh-session workaround and limited blast radius.
- `impact:session-state`: The failure is caused by missing ACP identity metadata after cleanup, which blocks restoration of prior backend session context.
- `impact:auth-provider`: The requested recovery path depends on proving ownership or admin authorization for a Claude/ACPX backend session id.

## Summary

Keep open: current main and the latest release still reject `sessions_spawn.resumeSessionId` unless the id matches requester-owned local ACP identity metadata, and there is still no explicit backend-session restore path after that metadata is cleared. The remaining work is real, but it needs a maintainer-owned authorization/session-state contract before implementation.

## What This Changes

The issue asks for an authorized explicit path to restore a valid Claude ACP backend session after local ACP identity metadata has been cleared.

## Best Possible Solution

Define a separate explicit backend restore contract, or deliberately broaden `resumeSessionId`, with durable ownership/admin proof, fresh ACP wrapper and binding creation, docs/schema alignment, and regression tests that preserve stale-session respawn protection.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: remove or lack a requester-owned local `acp.identity` match for a valid backend id, then call `sessions_spawn` with `resumeSessionId`; current main returns `resume_forbidden` before runtime initialization. I did not run a live Claude ACP restore because this review is read-only and the scenario depends on real backend session state.

## Solution Assessment

Unclear as a direct code fix: the requested direction is plausible, but the best implementation needs maintainer choice about the authorization contract and whether this belongs in `resumeSessionId`, a new `restoreBackendSessionId`, or durable tombstone ownership state.
