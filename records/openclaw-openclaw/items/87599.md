## Summary

Keep open: current main and the latest release still only allow `sessions_spawn.resumeSessionId` when a matching local ACP identity is recorded for the requester, so the requested explicit backend-session restore path remains missing and needs a maintainer-owned auth/session-state decision.

## What This Changes

Request an explicit ACP backend-session restore path for a known valid Claude/backend session after local ACP identity metadata has been cleaned up.

## Best Possible Solution

Define a separate explicit backend-session restore contract, or deliberately update the existing `resumeSessionId` contract, so authorized recovery creates fresh ACP metadata and bindings while stale wrappers remain non-respawnable.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: remove or lack a local `acp.identity` match for the backend ID, then call `sessions_spawn` with that `resumeSessionId`; current main returns `resume_forbidden` before ACP runtime initialization.

## Solution Assessment

Unclear as a direct code fix. The requested direction is plausible, but maintainers need to choose whether to add a distinct backend-restore operation, broaden `resumeSessionId`, or keep the local-recorded-only contract and fix the docs/tool wording.
