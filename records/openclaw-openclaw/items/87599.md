## Label Justifications

- `P2`: This is a bounded but real ACP session-continuity failure with a fresh-session workaround and a security-sensitive design choice.
- `impact:session-state`: The failure is caused by ACP identity metadata being downgraded or missing after lifecycle cleanup, blocking restoration of prior backend context.
- `impact:auth-provider`: The requested recovery path depends on proving authority to resume a Claude/ACPX backend session id without weakening requester ownership checks.

## Summary

Keep open: current main and the latest release still authorize ACP `resumeSessionId` only when local ACP metadata contains a matching resolved backend id owned by the requester, while reset cleanup intentionally downgrades that metadata to pending without the backend ids. The merged SQLite metadata PR helps durability but does not add the explicit authorized backend-session restore path requested here.

## What This Changes

The issue asks for a supported ACP recovery path that can create a fresh OpenClaw ACP wrapper around a known valid Claude/backend session id after local ACP metadata cleanup.

## Best Possible Solution

Define a maintainer-approved backend-restore contract that creates a fresh ACP wrapper and binding from an authorized backend session id while preserving the stale-respawn protections from the older ACP cleanup work.

## Reproduction Assessment

Yes, at source level: reset cleanup writes pending ACP metadata without `agentSessionId` or `acpxSessionId`, and a later `sessions_spawn` with that backend id cannot match the current ownership validator, returning `resume_forbidden` before runtime initialization. I did not run a live Claude ACP restore in this read-only review.

## Solution Assessment

Unclear as a direct code fix. The requested restore direction is plausible, but maintainers need to choose whether this is an admin-only backend-restore operation, a broader `resumeSessionId` contract, or a durable ownership tombstone before implementation.
