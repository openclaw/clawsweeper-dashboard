## Summary

Keep open: current main and v2026.6.9 still require `resumeSessionId` to match a requester-owned recorded ACP identity, while reset cleanup intentionally removes backend session IDs from live ACP metadata. The idle-reset workaround can delay the failure, but it does not provide the requested explicit backend-session restore path after cleanup.

## What This Changes

Request an authorized ACP backend-session restore path that can wrap a known valid Claude/backend session ID after local ACP metadata cleanup.

## Best Possible Solution

Define a maintainer-approved backend-restore contract that creates a fresh ACP wrapper and binding for authorized recovery while preserving stale-respawn safeguards and requester ownership checks.

## Reproduction Assessment

Yes, at source level. After reset cleanup rewrites ACP identity to pending metadata without `agentSessionId` or `acpxSessionId`, `sessions_spawn` with that backend ID cannot match the current ownership validator and returns `resume_forbidden` before runtime initialization.

## Solution Assessment

Unclear as a direct implementation path. The requested recovery direction is plausible, but maintainers need to choose the authorization model and whether this is a new admin-only backend-restore operation or a changed `resumeSessionId` contract.
