## Summary

Keep open: the reported channel-leak bug is plausible on current main, but this branch still has blocking correctness issues and no contributor-provided real Anthropic/Bedrock channel proof.

## What This Changes

The PR adds a `hadThinkingInMessage` subscribe state flag and derives a synthetic `final_answer` delivery phase for unphased non-OpenAI Responses text after thinking events.

## Best Possible Solution

Attach reliable phase metadata at the Anthropic/Bedrock transport boundary or structurally suppress thinking-adjacent/tool-use prelude before channel delivery, with focused regression coverage and redacted real channel proof.

## Reproduction Assessment

Yes for a source-level path: Anthropic emits separate thinking events and unphased text blocks, current delivery filters only explicit phase metadata, and the visible-text extractor falls back to unphased text. I did not run a live Bedrock, Slack, or Telegram replay in this read-only review.

## Solution Assessment

No. The PR labels the delivery path as `final_answer` but leaves the assistant text unphased, so the extractor can still return the same narration; a safer fix would classify text at the transport boundary or suppress the prelude structurally before delivery.
