## Summary

Keep this draft open for redesign, but it is not merge-ready: the proposed opt-in is outside OpenClaw’s canonical strict config contract, and the evaluator derives security capabilities from non-authoritative tool-name substrings. The current evidence is isolated tests only, with no runtime producer or emitted audit event.

## What This Changes

The PR adds an observe-only Security Matrix package with default policy decisions, tool-name capability inference, an audit-event adapter, a proposed opt-in config predicate, focused tests, and documentation.

## Best Possible Solution

Project one audit event from the existing authoritative tool-policy and approval boundary, require explicit capability metadata with an `unknown` fallback, and add a schema-backed opt-in only alongside a real event consumer.

## Reproduction Assessment

Yes for the patch defects: current source proves the strict schema rejects `security.matrix`, and the resolver deterministically elevates unrelated tool identifiers containing reserved substrings.

## Solution Assessment

No. A parallel policy model with identifier-based inference is not the safest canonical solution; audit data should come from existing runtime owners and explicit metadata.

Full review comments:

- **[P1] Register the advertised audit config before exposing it:** `src/security/security-matrix/before-tool-call.ts:8-14`
  - body: `security.matrix.audit.enabled` is accepted only by this local structural type. OpenClaw parses `security` with a strict canonical schema that currently permits only `audit` and `installPolicy`, so an operator cannot enable this path with valid configuration. Add the complete config type/schema/help/docs contract together with a real consumer, or remove the predicate and opt-in claims until that work exists.
  - confidence: 0.99
- **[P1] Require authoritative capability metadata instead of name substrings:** `src/security/security-matrix/tool-capability.ts:48-76`
  - body: The fallback treats arbitrary identifiers containing `token`, `config`, `git`, `write`, `file`, and similar text as privileged capabilities. Plugin and MCP tool identifiers are open-ended, so benign tools can receive false security decisions while dangerous tools without those words remain `unknown`. Consume explicit capability metadata from the tool owner and preserve `unknown` when it is absent.
  - confidence: 0.97
