## Summary

Keep open: the PR is in the right Telegram command-menu helper, but current head still verifies only the base default and group command scopes before trusting the cached hash while the same sync path writes per-language command variants. The available Telegram Desktop proof is useful, but it was captured for an earlier head and does not prove the latest scope/localization recovery path.

## What This Changes

The PR changes Telegram command-menu sync to call `getMyCommands()` for each configured base command scope before skipping registration when the local command hash matches.

## Best Possible Solution

Keep the command-hash optimization, but skip sync only after verifying every command state the helper writes: base default/group scopes plus each supported `language_code` variant, with focused regression tests and current-head Telegram proof.

## Reproduction Assessment

Yes from source inspection: populate the cached hash, keep default and group command scopes non-empty, clear one `language_code` command variant, and the PR returns after base-scope `getMyCommands()` checks without re-registering the localized menu. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The PR is in the right helper and covers the default/group-scope gap, but the best fix must verify the same scope-plus-language command states that `setMyCommands` writes before honoring the cache.
