## Summary

Current main still blocks private or CGNAT model provider URLs unless the existing per-provider opt-in is set, and the reporter’s diagnostic complaint remains source-reproducible. This should stay open for a narrow diagnostics/config-guidance fix, not cleanup closure.

## What This Changes

The issue asks OpenClaw to avoid or clearly diagnose 2026.5.12 SSRF blocks for private or CGNAT self-hosted model providers and to document the per-provider opt-in path.

## Best Possible Solution

Keep the secure SSRF default, but make the failure path self-diagnosing and point operators to `models.providers.<id>.request.allowPrivateNetwork` for trusted self-hosted endpoints.

## Reproduction Assessment

Yes, by source inspection: a model provider using a private or CGNAT base URL without `request.allowPrivateNetwork: true` reaches `fetchWithSsrFGuard` with no private-network opt-in and will throw `SsrFBlockedError`. I did not execute a live private-provider run in this read-only review.

## Solution Assessment

Yes for a narrow fix: preserve the secure default, but classify/format `SsrFBlockedError` as an SSRF/private-network provider failure and point to the documented per-provider command. Changing the default to allow private networks would be a security/product decision and should not be the automated fix.
