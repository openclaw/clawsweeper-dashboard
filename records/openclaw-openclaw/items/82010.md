## Summary

Keep open. Current main still has a source-reproducible path where non-loopback private or CGNAT model-provider base URLs fail unless `models.providers.<id>.request.allowPrivateNetwork` is set, and the report’s SSRF-specific diagnostics/config-guidance complaint is not fully covered by the active exact-origin PR.

## What This Changes

The issue reports a v2026.5.12 regression where default SSRF protection blocks private/CGNAT self-hosted model-provider endpoints and surfaces poor guidance for the existing per-provider opt-in.

## Best Possible Solution

Decide and land the exact-origin SSRF policy in the active PR if approved, while adding SSRF-specific provider failure guidance that points operators to the existing per-provider opt-in when a block still occurs.

## Reproduction Assessment

Yes. Source inspection shows a non-loopback private or CGNAT provider base URL without `request.allowPrivateNetwork: true` reaches the guarded fetch without a private-network opt-in and will be rejected by the SSRF guard; I did not run a live private-provider setup.

## Solution Assessment

Unclear for the behavior change until maintainer/security review decides the exact-origin trust boundary in the active PR. The safe narrow direction is to preserve the secure default and make SSRF provider failures self-diagnosing with the exact provider-scoped config path.
