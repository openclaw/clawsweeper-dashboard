## Summary

Keep this PR open: the runtime-parity work is useful and the PR body now includes real CLI proof, but the new classifier can miss a core drift case because it sorts tool calls before comparing them.

## What This Changes

The branch adds Codex-vs-Pi QA runtime parity tooling, reports, fixtures, workflow lanes, and a Codex dynamic-tool status fix.

## Best Possible Solution

Keep the new harness direction, but preserve execution-order comparison in the drift classifier and cover swapped tool-call order with a focused regression test before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence unit reproduction: build two runtime cells with the same tool calls in different orders and the classifier sorts them before comparison, so the ordering drift can be missed.

## Solution Assessment

No. The PR is close, but sorting the captured tool-call sequence is not the best solution for a runtime parity gate whose contract is to detect trajectory drift; compare in execution order or report order drift explicitly.
