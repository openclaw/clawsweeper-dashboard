The PR adds a qa-lab Codex-vs-Pi runtime parity harness with runtime-pair execution, reports, tool fixtures, JSONL replay scaffolding, release workflow wiring, and a Codex accepted-status dynamic-tool fix.

## Best Possible Solution

Land the harness only after runtime parity reporting preserves `skip` as non-blocking report-only drift while still showing the drift details in the Markdown and JSON output.

## Reproduction Assessment

Yes, source-reproducible. The PR writes known report-only drift as `skip`, but the runtime report maps non-pass statuses to `fail`, so a summary with report-only rows can fail `qa parity-report --runtime-axis`.

## Solution Assessment

No. The harness direction is reasonable, but the report builder needs to preserve the suite's report-only semantics instead of converting skip rows into hard failures.
