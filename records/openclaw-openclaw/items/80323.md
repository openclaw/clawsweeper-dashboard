## Summary

Keep this PR open: the runtime parity harness is useful and current main does not already contain it, but the current head still has two merge-blocking correctness issues in the report-only skip path and the scheduled live token lane.

## What This Changes

The PR adds a qa-lab Codex-vs-Pi runtime parity harness across CLI flags, runtime reports, scenario fixtures, workflows, token-efficiency reporting, JSONL replay scaffolding, and a Codex dynamic-tool accepted-status fix.

## Best Possible Solution

Land the harness after report-only skip semantics stay non-blocking in runtime reports and the live token-efficiency lane is either gated or backed by proven Codex live auth staging.

## Reproduction Assessment

Yes, source-reproducible. The suite emits report-only rows as skip while the runtime report converts every non-pass status to fail, and the workflow/source path shows the scheduled live token lane runs without the live Codex auth staging the PR says is still unproven.

## Solution Assessment

No. The harness direction is reasonable, but the current patch should preserve report-only semantics and avoid enabling the live token lane until its auth/proof boundary is real.

## Review Findings

- **[P2] Preserve report-only skips in runtime reports:** `extensions/qa-lab/src/agentic-parity-report.ts:645`

The workflow only stages `OPENAI_API_KEY`, and the runtime env patch adds Codex app-server auth overrides only for mock-openai, so the nightly live Codex cell can fail before producing a useful token-efficiency report. Gate this lane or implement and prove live Codex auth staging first.
  - confidence: 0.82
