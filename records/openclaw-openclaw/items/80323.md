## Summary

Keep open: the harness is useful and current main does not already contain it, but the PR still has two source-reproducible P2 blockers in runtime report skip handling and the scheduled live token lane.

## What This Changes

The PR adds a qa-lab Codex-vs-Pi runtime parity harness with CLI flags, runtime reports, scenario fixtures, JSONL replay scaffolding, token-efficiency reporting, workflow wiring, and a Codex dynamic-tool accepted-status fix.

## Best Possible Solution

Land the harness after report-only skip semantics stay non-blocking in runtime reports and the live token-efficiency lane is gated or backed by proven Codex live auth staging.

## Reproduction Assessment

Yes, source-reproducible. The suite emits report-only rows as skip, but the runtime report converts skip to fail and the scheduled live job runs a forced Codex cell without the auth staging the PR says remains untested.

## Solution Assessment

No. The harness direction is maintainable, but this patch needs to preserve report-only semantics and avoid enabling the live token lane until the auth/proof boundary is real.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve report-only skips in runtime reports:** `extensions/qa-lab/src/agentic-parity-report.ts:645`

  - confidence: 0.93

The workflow only stages `OPENAI_API_KEY`, and the forced-Codex env patch only adds Codex app-server auth overrides for `mock-openai`, so the nightly live Codex cell can fail before producing a useful token-efficiency report.
  - confidence: 0.82
