## Summary

Keep open: this is a plausible cron-session bug fix, but the current PR is not safe to merge because it misses supported retention-disabled configs, only updates hidden run aliases, and lacks contributor real-behavior proof.

## What This Changes

The branch adds a two-hour stale-running recovery pass to the cron session reaper, returns an `orphansRecovered` count, logs recoveries, and adds one unit test.

## Best Possible Solution

Keep the PR open for a rebased recovery patch that runs independently of pruning, repairs stable base rows plus legacy run aliases, uses approved interrupted-run terminal semantics, and adds focused regression coverage.

## Reproduction Assessment

Yes, by source inspection: current main has no orphan recovery, and concrete stale session-store rows can exercise the retention-disabled, base-key, and long-running-run gaps. I did not run tests because this review is read-only.

## Solution Assessment

No. A reaper-based repair pass is plausible, but this implementation should not depend on pruning, should not rely on `updatedAt` alone as abandonment proof, and must repair the visible base cron session state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Prove a cron run is abandoned before finalizing it:** `src/cron/session-reaper.ts:106-109`
  - body: This branch marks any `status: "running"` run row older than two hours as `done`, but cron agent-turn jobs can run longer or disable timeouts and `updatedAt` is not a heartbeat. That can silently finalize active work and let a later sweep prune it.
  - confidence: 0.93
- **[P1] Recover the stable cron session row:** `src/cron/session-reaper.ts:106`
  - body: The recovery loop only updates `:run:` alias keys, while current isolated cron persistence writes visible session state to the stable `agent:...:cron:<job>` key and session listing hides run aliases. The UI can still show the cron session as `running`.
  - confidence: 0.92
- **[P2] Run recovery when retention pruning is disabled:** `src/cron/session-reaper.ts:83-85`
  - body: The `retentionMs === null` return still exits before orphan scanning. With `cron.sessionRetention: false`, stale running sessions remain stuck even though disabling pruning should not disable crash recovery.
  - confidence: 0.94
- **[P3] Avoid writing epoch timestamps for missing updates:** `src/cron/session-reaper.ts:106`
  - body: When `updatedAt` is missing, this branch uses the existing `0` fallback as `endedAt`, which can surface as the Unix epoch. Use the sweep time or another valid terminal timestamp instead.
  - confidence: 0.82
