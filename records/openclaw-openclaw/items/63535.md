## Summary

Keep open: current main still lacks stale-running cron recovery, but this PR is not safe to merge as-is because recovery is retention-gated, run-alias-only, age-based, and missing real behavior proof.

## What This Changes

The PR adds a cron reaper pass that marks stale running cron run session rows as done, returns an `orphansRecovered` count, logs recoveries, and adds a unit test.

## Best Possible Solution

Rework the cron recovery patch to repair crash-abandoned cron session state without pruning coupling, cover stable base keys, and align terminal status with existing interrupted-run semantics.

## Reproduction Assessment

Yes, by source inspection: a stale run row with `cron.sessionRetention: false`, a stale stable `agent:main:cron:<job>` row, or a long/no-timeout active cron run exercises the PR gaps on current main and the PR diff.

## Solution Assessment

No. A reaper-based repair pass is plausible, but this implementation should run independently of pruning, repair stable base rows plus legacy aliases, and use lifecycle proof plus a maintainer-approved terminal status.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Prove a cron run is abandoned before finalizing it:** `src/cron/session-reaper.ts:106-109`
  - body: The new branch marks any `status: "running"` run row older than two hours as `done`, but cron agent-turn jobs can disable or extend timeout and `updatedAt` is not a heartbeat. An active long/no-timeout run can be silently finalized and later pruned.
  - confidence: 0.93
- **[P1] Recover the stable cron session row:** `src/cron/session-reaper.ts:106`
  - body: Current isolated cron state is written under `agent:...:cron:<job>` and session listing hides `:run:` aliases, so updating only matching `:run:` keys can report recovery while the visible row remains `running`.
  - confidence: 0.92
- **[P2] Run recovery when retention pruning is disabled:** `src/cron/session-reaper.ts:83-85`
  - body: The `retentionMs === null` return runs before orphan scanning. With `cron.sessionRetention: false`, stale running sessions still remain stuck even though disabling pruning should not disable crash-recovery.
  - confidence: 0.94
- **[P3] Avoid `endedAt: 0` for missing timestamps:** `src/cron/session-reaper.ts:106`
  - body: When `updatedAt` is missing, this branch uses the existing `0` fallback as `endedAt`, which can surface as the Unix epoch. Use the sweep time or another valid terminal timestamp.
  - confidence: 0.82
