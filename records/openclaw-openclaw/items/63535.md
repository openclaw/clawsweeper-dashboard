## Summary

Keep this PR open, but do not merge it as-is: the current branch is a useful focused cron recovery attempt, yet source review shows it is still retention-gated, updates only hidden run aliases, can finalize legitimately active long/no-timeout cron runs, and lacks real behavior proof.

## What This Changes

The PR adds cron session reaper logic to mark stale `status: "running"` cron run rows as `done`, returns `orphansRecovered`, logs recoveries, and adds one unit test.

## Best Possible Solution

Keep a focused cron recovery patch that runs independently of pruning, updates the visible base cron row plus any legacy run aliases only after lifecycle ownership proves the run is abandoned, uses the approved interrupted-run terminal status, and covers retention-disabled/base-key cases.

## Reproduction Assessment

Yes, by source inspection: a stale stable `agent:main:cron:<job>` row, a stale run row with `cronConfig: { sessionRetention: false }`, or a no-timeout/long-timeout active cron run exercises the PR gaps. I did not run tests because this review is read-only.

## Solution Assessment

No. A reaper-based repair pass is a reasonable place to fix crash leftovers, but this implementation is retention-coupled, run-alias-only, and relies on age rather than ownership/lifecycle proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Prove the run is abandoned before marking it done:** `src/cron/session-reaper.ts:106-109`
  - body: The new recovery branch finalizes any running cron run row older than two hours, but cron agent-turn jobs can set `timeoutSeconds: 0` for no timeout or a long explicit timeout. Since `updatedAt` is not a heartbeat, an active run can be marked done and later pruned.
  - confidence: 0.93
- **[P1] Recover the visible base cron row:** `src/cron/session-reaper.ts:106`
  - body: The loop only writes the matched `:run:` alias, while current isolated cron state is persisted under the stable `agent:...:cron:<job>` key and session listing hides run aliases. The sweep can report a recovery while the visible cron session remains `running`.
  - confidence: 0.92
- **[P2] Run recovery when retention pruning is disabled:** `src/cron/session-reaper.ts:83-85`
  - body: This early return still exits before the new orphan scan when `cron.sessionRetention` is `false`. Disabling pruning to preserve history should not also disable status repair for crashed cron runs.
  - confidence: 0.94
- **[P3] Use the sweep time when `updatedAt` is missing:** `src/cron/session-reaper.ts:106`
  - body: When a running row lacks `updatedAt`, the fallback value is `0`, so the recovery persists `endedAt: 0` and can display or compare as the Unix epoch. Use the recovery timestamp as the fallback terminal time.
  - confidence: 0.82
