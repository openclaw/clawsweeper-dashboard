## Summary

Keep open: the crash-recovery bug is plausible, but this PR is not safe to merge as written because it uses age-only recovery, updates hidden run aliases instead of the visible base cron row, skips recovery when retention pruning is disabled, and still lacks real behavior proof.

## What This Changes

The PR adds a cron reaper pass that marks stale `status: "running"` cron run rows as `done`, returns `orphansRecovered`, logs recoveries, and adds a unit test.

## Best Possible Solution

Keep the PR open for a reworked cron recovery patch that proves abandonment, repairs the visible base cron row plus legacy run aliases, runs independently of pruning, and uses the approved interrupted-run status.

## Reproduction Assessment

Yes, source-reproducible: current main has no orphan-recovery pass, and synthetic stale session-store rows can exercise the retention-disabled, base-key, and age-only recovery gaps. I did not run tests because this was a read-only review.

## Solution Assessment

No. A reaper-based repair is plausible, but this implementation depends on pruning, updates hidden run aliases, relies on `updatedAt` alone as abandonment proof, and chooses `done` without an approved terminal-status contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Prove a run is abandoned before finalizing it:** `src/cron/session-reaper.ts:106-109`
  - body: This marks any `status: "running"` cron run row older than two hours as `done`, but cron jobs can run longer or disable timeouts and `updatedAt` is not a heartbeat. An active long run can be finalized and later pruned while still executing.
  - confidence: 0.93
- **[P1] Recover the visible base cron row:** `src/cron/session-reaper.ts:106-109`
  - body: The recovery loop only updates `:run:` alias rows, while current isolated cron state is persisted and listed through the stable base cron key. The sweep can report recovery while the UI-visible cron session remains `running`.
  - confidence: 0.92
- **[P2] Run recovery when pruning is disabled:** `src/cron/session-reaper.ts:83-85`
  - body: The `retentionMs === null` early return exits before orphan scanning. With `cron.sessionRetention: false`, stale running sessions still never recover even though disabling pruning should not disable crash-status correction.
  - confidence: 0.94
- **[P3] Use a valid fallback endedAt:** `src/cron/session-reaper.ts:106`
  - body: When `updatedAt` is missing, this branch persists `endedAt: 0`, which can display or compare as the Unix epoch. Use the sweep time or another valid terminal timestamp instead.
  - confidence: 0.82
