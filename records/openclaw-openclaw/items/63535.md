## Summary

Keep open. Current main still lacks stale-running cron session recovery, and this PR is relevant, but the patch is not merge-ready because the recovery path is disabled when retention pruning is off and it only updates hidden `:run:` alias rows instead of the visible stable cron session row.

## What This Changes

The PR adds cron reaper logic to mark two-hour-old cron `:run:` rows with `status: "running"` as `done`, returns an `orphansRecovered` count, logs recoveries, and adds a focused unit test.

## Best Possible Solution

Land a revised cron session recovery implementation that runs independently of retention pruning, updates the visible stable cron session row as well as any legacy run alias, follows the existing lifecycle terminal-status semantics, and covers retention-disabled, base-key, stale-versus-active, and short-retention cases.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a session store containing either a stale `agent:main:cron:<job>` base row or a stale row with `cron.sessionRetention: false`; inspection shows the PR either ignores the stable base row or returns before scanning.

## Solution Assessment

No. Reaper-based recovery is a reasonable place to solve the problem, but this diff is not the best version because recovery is coupled to pruning, targets only hidden aliases, and does not settle the terminal-status mapping against existing lifecycle behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Recover stable cron session rows:** `src/cron/session-reaper.ts:108`
  - body: The recovery branch only writes back to `store[key]`, but current isolated cron lifecycle state is persisted under the stable `agent:...:cron:<job>` key and session listing hides `:run:` aliases. After a crash, the visible base row can remain `running` even though the hidden alias was marked done.
  - confidence: 0.92
- **[P2] Run recovery when retention is disabled:** `src/cron/session-reaper.ts:83-85`
  - body: This early return skips the orphan scan when `cron.sessionRetention` is `false`. That setting disables pruning to preserve history, but stale crashed sessions still need status recovery, so supported configs can keep rows stuck as `running` indefinitely.
  - confidence: 0.94
