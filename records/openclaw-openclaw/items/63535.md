## Summary

Keep this PR open: the reported cron recovery gap is real, but the patch misses the visible stable cron session row and skips recovery when retention pruning is disabled.

## What This Changes

The PR adds a stale-running cron run recovery pass to the cron session reaper, returns an `orphansRecovered` count, logs recoveries, and adds a unit test.

## Best Possible Solution

Land a revised cron session recovery implementation that runs independently of retention pruning, repairs the visible stable cron session row plus legacy run aliases, follows maintainer-approved terminal status semantics, and includes focused regression tests.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a `sessions.json` containing either a stale `agent:main:cron:<job>` base row or a stale row with `cron.sessionRetention: false`; source inspection shows the PR ignores the first and returns before scanning the second.

## Solution Assessment

No. Reaper-based recovery is a reasonable location, but this patch is not the best implementation because it is retention-coupled, run-alias-only, and unsettled on terminal status semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

  - body: Current isolated cron state is persisted under the stable `agent:...:cron:<job>` key, and session listing hides `:run:` aliases. Updating only the run alias can leave the row users see stuck as `running` after recovery.
