## Summary

Keep open: the PR targets a plausible cron session-state bug, but the current patch can mis-finalize active runs, misses the visible base cron row, still skips retention-disabled configurations, and lacks real behavior proof.

## What This Changes

The PR adds a two-hour stale-running cron run recovery pass to the session reaper, returns an `orphansRecovered` count, and adds a unit test for marking old cron run rows done.

## Best Possible Solution

Keep the PR open for a rebased recovery design that runs independently of pruning, proves abandonment, updates stable base cron rows plus any aliases, follows approved interrupted-run status semantics, and includes focused regression coverage plus real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path with synthetic session-store rows: current main lacks stale-running recovery, while the PR path still exposes the retention-disabled bypass and base-key visibility gap; I did not run tests because this review is read-only.

## Solution Assessment

No. A reaper recovery pass is plausible, but this implementation relies on age-only `updatedAt`, only updates run aliases, depends on pruning being enabled, and chooses terminal status without an approved contract.

Full review comments:

- **[P1] Prove abandonment before finalizing runs:** `src/cron/session-reaper.ts:106-109`
  - body: This marks any `status: "running"` cron run row older than two hours as `done`, but cron agent-turn jobs can run longer or disable timeouts and `updatedAt` is not a heartbeat. An active run can be finalized and later pruned while it is still executing.
  - confidence: 0.93
- **[P1] Recover the stable cron session row:** `src/cron/session-reaper.ts:106`
  - body: The recovery branch only writes the `:run:` alias row. Current isolated cron persistence stores visible state under the stable `agent:...:cron:<job>` key and session listing hides run aliases, so the user-visible row can remain stuck as `running`.
  - confidence: 0.92
- **[P2] Run recovery when pruning is disabled:** `src/cron/session-reaper.ts:83-85`
  - body: The `retentionMs === null` return exits before the orphan scan, so users who set `cron.sessionRetention: false` still get stuck `running` rows after a crash even though they only disabled pruning.
  - confidence: 0.94
- **[P3] Avoid epoch endedAt values:** `src/cron/session-reaper.ts:106`
  - body: When `updatedAt` is missing, this branch persists `endedAt: 0`, which can display as the Unix epoch. Use the sweep time or another valid terminal timestamp fallback instead.
  - confidence: 0.82
