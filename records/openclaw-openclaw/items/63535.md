Full review comments:

- **[P1] Prove abandonment before finalizing runs:** `src/cron/session-reaper.ts:106-109`
  - body: This marks any `status: "running"` cron run row older than two hours as `done`, but `timeoutSeconds=0` explicitly disables the cron timeout and `updatedAt` is not a heartbeat. An active long-running job can be finalized and later pruned while it is still executing.
  - confidence: 0.93
- **[P1] Recover the stable cron session row:** `src/cron/session-reaper.ts:106`
  - body: The recovery branch only writes the `:run:` alias row. Current isolated cron persistence stores visible state under the stable `agent:...:cron:<job>` key and session listing hides run aliases, so the user-visible row can stay stuck as `running`.
  - confidence: 0.92
- **[P2] Run recovery when pruning is disabled:** `src/cron/session-reaper.ts:83-85`
  - body: The `retentionMs === null` return exits before the orphan scan, so users who set `cron.sessionRetention: false` still get stuck `running` rows after a crash even though they only disabled pruning.
  - confidence: 0.94
- **[P3] Exercise the defer-before-prune path:** `src/cron/session-reaper.test.ts:289`
  - body: The new test expects `pruned` to stay zero, but all test rows are only three hours old under the default 24-hour retention. Add a short-retention case to prove the recovery branch actually defers deletion for one sweep.
  - confidence: 0.82
