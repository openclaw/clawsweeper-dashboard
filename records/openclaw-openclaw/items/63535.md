## Summary

Keep open: this is a useful focused bug-fix PR, but it is not mergeable as-is because the patch misses current visible cron session rows, skips recovery when pruning is disabled, lacks contributor real-behavior proof, and still needs terminal-status semantics confirmed.

## What This Changes

The PR adds a two-hour stale-running cron run recovery pass to the cron session reaper, returns an `orphansRecovered` count, logs recoveries, and adds one unit test.

## Best Possible Solution

Keep the PR open for a revised recovery implementation that runs independently of pruning, repairs the stable base cron row plus legacy run aliases, uses the approved terminal status, and adds focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a `sessions.json` with either a stale stable `agent:main:cron:<job>` row or a stale run row plus `cronConfig: { sessionRetention: false }`; the PR's code path skips both cases.

## Solution Assessment

No. A reaper-based recovery pass is a reasonable location, but this implementation is not the narrowest maintainable fix because it is retention-coupled, run-alias-only, and not aligned with existing interrupted-run status semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Recover the stable cron session row:** `src/cron/session-reaper.ts:108`
  - body: Current isolated cron runs persist visible lifecycle state under the stable `agent:...:cron:<job>` key, and session listing filters `:run:` aliases. Because this recovery loop only visits run-alias keys, a crashed cron session can still appear stuck as `running` after the sweep reports recovery.
  - confidence: 0.92
- **[P2] Run recovery when pruning is disabled:** `src/cron/session-reaper.ts:83-85`
  - body: When `cron.sessionRetention` is `false`, `resolveRetentionMs` returns `null` and this early return exits before the new orphan scan. Disabling pruning should not disable status repair, so stale crashed sessions can remain `running` indefinitely in that supported config mode.
  - confidence: 0.94
