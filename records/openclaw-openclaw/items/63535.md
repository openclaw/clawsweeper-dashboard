## Summary

Keep open: current main still lacks stale-running cron session recovery, but this PR has blocking correctness issues around active-run detection, stable cron rows, and retention-disabled configs, plus missing real behavior proof.

## What This Changes

The PR adds a stale-running cron run recovery pass to the cron session reaper, reports `orphansRecovered`, logs recoveries, and adds unit coverage.

## Best Possible Solution

Rework the cron recovery path to prove the run is truly abandoned, run independently of retention pruning, repair the stable base row plus any legacy aliases, use the approved terminal status, and cover those cases with focused tests.

## Reproduction Assessment

Yes, by source inspection rather than a local run. A stale stable `agent:main:cron:<job>` row, a stale run row with `cronConfig: { sessionRetention: false }`, or an active run older than two hours against the 48-hour default timeout exercises the current PR gaps.

## Solution Assessment

No. A reaper-based recovery pass is a reasonable location, but this implementation is retention-coupled, run-alias-only, and uses a fixed age heuristic that can finalize active jobs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Check ownership before recovering long-running cron rows:** `src/cron/session-reaper.ts:108`
  - body: The new scan marks any running cron row older than two hours as done, but cron agent-turn runs can legitimately run far longer because the agent timeout defaults to 48h and can be overridden. Since lifecycle `updatedAt` is not a heartbeat, this can finalize and later prune an active run.
  - confidence: 0.93
- **[P1] Recover the stable cron session row:** `src/cron/session-reaper.ts:108`
  - body: Current isolated cron lifecycle state is persisted under the stable `agent:...:cron:<job>` key, and session listing filters out `:run:` aliases. Updating only the run alias can leave the visible cron session stuck as `running` after the sweep reports recovery.
  - confidence: 0.92
- **[P2] Run recovery when pruning is disabled:** `src/cron/session-reaper.ts:83-85`
  - body: When `cron.sessionRetention` is `false`, this early return exits before the orphan scan. Disabling pruning should not also disable status repair, so stale crashed rows remain `running` indefinitely in that supported config mode.
  - confidence: 0.94
- **[P3] Use recovery time when updatedAt is missing:** `src/cron/session-reaper.ts:108`
  - body: If a running row lacks `updatedAt`, the branch uses the `0` fallback as `endedAt`, which can surface as a Unix-epoch end time. Use the recovery timestamp as the fallback while preserving the cutoff comparison.
  - confidence: 0.82
