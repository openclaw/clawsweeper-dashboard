## Summary

Keep this PR open. Current main still lacks a `sessions_send` plugin hook, but this head should not merge as-is because the delegated path trusts caller-supplied announce metadata, runs core reply-history reads before hook-owned delivery can claim the send, and leaves public SDK/docs/changelog gaps; the open replacement #75369 addresses the main security/ordering blockers but still needs maintainer integration review.

## What This Changes

The PR adds a typed `sessions_send` plugin hook, Plugin SDK exports/API baseline updates, hook-runner wiring, sessions tool interception, and tests for declined and delegated hook paths.

## Best Possible Solution

Land a rebased implementation, likely by integrating #75369 plus the missing subtype exports, plugin hook docs, changelog entry, and current-main behavior preservation, so the new hook is generic and plugin-owned without weakening A2A trust boundaries.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: a delegated hook with caller args containing `task.runtime.roundOneReply` reaches `startA2AFlow`, and the A2A helper treats that value as the primary/latest reply; the ordering bug is reproduced by making the pre-hook `chat.history` baseline read fail before a direct/delegated plugin handler can run.

## Solution Assessment

No. The hook direction is useful, but this branch is not the best merge shape until the trust boundary, hook ordering, public SDK exports, docs/changelog, and current-main rebase preservation are handled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Stop trusting caller-supplied announce metadata:** `src/agents/tools/sessions-send-tool.ts:445`
  - body: In delegated mode this passes `hookTask?.runtime?.roundOneReply`, parsed from raw tool args, into `startA2AFlow`. Since tool args can include extra fields, a caller can seed A2A ping-pong or announce content without it coming from the delegated run or plugin-owned result; derive the seed from hook output or verified reply history instead.
  - confidence: 0.86
- **[P2] Invoke the hook before reading reply history:** `src/agents/tools/sessions-send-tool.ts:414`
  - body: For waited sends, the code still takes `baselineReply` before `runSessionsSend`; if `chat.history` fails, a plugin that would handle direct or delegated delivery never runs. Move the core baseline read into the fallback or delegated wait path so hook-owned delivery is not blocked by core history access.
  - confidence: 0.78
- **[P3] Export the hook task and cancel-target types:** `src/plugin-sdk/core.ts:102-106`
  - body: The new public event/result types refer to `PluginHookSessionsSendTask` and `PluginHookSessionsSendCancelTarget`, but `plugin-sdk/core` only re-exports context/event/result. Plugin authors cannot name the task or cancel-target contract from the same SDK surface without type gymnastics.
  - confidence: 0.74
- **[P3] Document and changelog the new hook:** `src/plugins/hook-types.ts:67-99`
  - body: This adds a public Plugin SDK hook/API surface, but the PR only updates the generated baseline and does not add the hook to the plugin hook docs or `CHANGELOG.md`. Public plugin contracts need both before merge so plugin authors can discover the new seam and release notes can track it.
  - confidence: 0.7

## Security Review

Status: needs_attention

Summary: The diff introduces a trust-boundary issue by allowing caller-controlled tool metadata to seed delegated A2A announce and ping-pong content.

Concerns:

- **[medium] Caller-controlled announce seed:** `src/agents/tools/sessions-send-tool.ts:445`
  - body: The delegated branch forwards `hookTask?.runtime?.roundOneReply` from raw tool args into `startA2AFlow`; that value is then treated as the primary/latest reply for follow-up messaging. A caller should not be able to spoof delegated run output through untrusted task metadata.
