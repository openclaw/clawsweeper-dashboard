## Summary

Keep this PR open. Current main still lacks the requested `sessions_send` plugin hook, but the reviewed head should not merge as-is because it has a delegated announce trust-boundary bug, invokes the hook after core reply-history reads, omits public subtype exports/docs/changelog, and is now paired with the open hardening follow-up #75369.

## What This Changes

The branch adds a typed `sessions_send` plugin hook, hook-runner and Plugin SDK exports, direct/delegated interception in the sessions tool, generated SDK baseline updates, and focused sessions tool tests.

## Best Possible Solution

Land one clean rebased implementation, either by updating this branch or consolidating through #75369, that adds the generic hook seam while preserving current main A2A behavior and only seeding delegated announce from plugin-owned output or core-verified reply history.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: a delegated hook with `timeoutSeconds: 0` and raw args containing `task.runtime.roundOneReply` reaches A2A seeding, and a waited send can fail `chat.history` before the plugin hook gets a chance to handle it.

## Solution Assessment

No. The hook direction is useful, but this PR is not the best merge shape until the trust boundary, hook ordering, SDK subtype exports, docs, changelog, and stale current-main behavior are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Stop trusting caller-supplied announce metadata:** `src/agents/tools/sessions-send-tool.ts:445`
  - body: In delegated mode this passes `hookTask?.runtime?.roundOneReply`, parsed from raw tool args, into `startA2AFlow`. Since tool args can include extra fields, a caller can seed A2A ping-pong or announce content without it coming from the delegated run or plugin-owned result; derive the seed from hook output or verified reply history instead.
  - confidence: 0.86
- **[P2] Invoke the hook before reading core reply history:** `src/agents/tools/sessions-send-tool.ts:414`
  - body: For waited sends, the code still takes `baselineReply` before `runSessionsSend`; if `chat.history` fails, a plugin that would handle direct or delegated delivery never runs. Move the core baseline read into the fallback or delegated wait path so hook-owned delivery is not blocked by core history access.
  - confidence: 0.78
- **[P3] Export the hook task and cancel-target types:** `src/plugin-sdk/core.ts:102-106`
  - body: The new public event/result types refer to `PluginHookSessionsSendTask` and `PluginHookSessionsSendCancelTarget`, but `plugin-sdk/core` only re-exports context/event/result. Plugin authors cannot name the task or cancel-target contract from the same SDK surface without type gymnastics.
  - confidence: 0.74
- **[P3] Document and changelog the new public hook:** `src/plugins/hook-types.ts:99`
  - body: This adds a public Plugin SDK hook/API surface, but the PR only updates the generated baseline and does not add the hook to the plugin hook docs or `CHANGELOG.md`. Public plugin contracts need both before merge so plugin authors can discover the new seam and release notes can track it.
  - confidence: 0.7

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete trust-boundary issue by letting caller-controlled tool metadata seed delegated A2A announce and ping-pong content.

Concerns:

- **[medium] Caller-controlled announce seed:** `src/agents/tools/sessions-send-tool.ts:445`
  - body: The delegated branch forwards `hookTask?.runtime?.roundOneReply` from raw tool args into `startA2AFlow`; that value is then treated as the primary/latest reply for follow-up messaging. A caller should not be able to spoof delegated run output through untrusted task metadata.
