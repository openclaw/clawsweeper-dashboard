## Summary

Keep this issue open: current `main` and the latest release still force a rebuild for dirty build-relevant paths before the mtime freshness check can run, and an open linked fork PR is still the active implementation candidate but is conflicted.

## What This Changes

The issue asks for `gateway:watch`/`run-node` startup to skip rebuilding when the working tree is dirty but no build-relevant source file is newer than the build stamp.

## Best Possible Solution

Maintainers should decide the dirty-tree freshness contract; if accepted, land an updated branch that changes `resolveBuildRequirement()`, the `dirty_watched_tree` tests, and the gateway-watch regression helper together.

## Reproduction Assessment

Yes, by source inspection. On current `main`, a dirty build-relevant path causes `resolveBuildRequirement()` to return `dirty_watched_tree` before `hasSourceMtimeChanged()` can compare source mtimes against the build stamp.

## Solution Assessment

Unclear. The proposed mtime fallthrough is small and plausible, but because current tests and the gateway-watch helper treat dirty watched trees as intentionally rebuild-triggering, the best fix needs an explicit maintainer decision on that contract.
