## Summary

Keep this issue open. Current main still returns `dirty_watched_tree` before consulting source mtimes, the existing run-node coverage asserts that behavior, and the target-repository fix PR was closed unmerged; the remaining work is a narrow dev-loop performance fix rather than a closeable cleanup item.

## What This Changes

The issue requests that `gateway:watch` skip a rebuild after a fresh build when the working tree is dirty but no watched source file is newer than the build stamp.

## Best Possible Solution

Change `scripts/run-node.mjs` so dirty watched source trees fall through to the existing source mtime freshness check, while preserving force-build, missing stamp/dist, config-newer, HEAD-changed, and clean-tree fast paths. Update `src/infra/run-node.test.ts`, the gateway-watch regression helper, and the changelog so the documented/tested contract matches the new behavior.

## Reproduction Assessment

Yes. The issue gives concrete steps, and current source plus tests reproduce the decision path: a dirty build-relevant source file returns `dirty_watched_tree` before mtime freshness is checked; I did not run live build/watch commands because this review was read-only.

## Solution Assessment

Yes, the proposed dirty-tree fallthrough to mtime is the narrowest maintainable fix, provided the existing `dirty_watched_tree` tests and gateway-watch regression helper are updated with the same contract. The safer alternative would be a more expensive content hash, but that would undermine the startup-overhead optimization this path was built for.
