## Summary

Keep open: current main and the latest release still force a build for dirty watched source trees before the mtime fallback runs. The same-repository fix PR is closed unmerged, and the open cross-repository closing PR is not a shipped or merged OpenClaw fix.

## What This Changes

The issue asks for `pnpm gateway:watch`/`run-node` to skip rebuilds when the watched source tree is dirty but no source file is newer than the build stamp.

## Best Possible Solution

Define the intended dirty-tree freshness contract; if maintainers accept mtime fallthrough, land a narrow upstream patch that updates `resolveBuildRequirement()`, run-node tests, and the gateway-watch regression helper together.

## Reproduction Assessment

Yes by source inspection: current `resolveBuildRequirement()` still short-circuits dirty watched source trees to `dirty_watched_tree` before `hasSourceMtimeChanged()` can run. I did not run the slow live `gateway:watch` scenario during this read-only review.

## Solution Assessment

Unclear. The proposed mtime fallthrough is a narrow plausible fix and matches the pre-`f86840f4` mtime-based behavior, but current tests and the regression helper now encode dirty-tree rebuilds as intentional enough that maintainers should confirm the contract first.
