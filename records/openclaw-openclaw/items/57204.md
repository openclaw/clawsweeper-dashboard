## Summary

Keep this open: current main still forces the dirty watched-source rebuild before the mtime freshness check can run, and a live linked PR remains the focused implementation candidate.

## What This Changes

The issue asks OpenClaw to avoid full TypeScript rebuilds in `gateway:watch` when the working tree is dirty but no build-relevant file is newer than the build stamp.

## Best Possible Solution

Review and land the linked fix or an equivalent narrow patch so dirty watched source trees fall through to source mtime freshness, with regression coverage for dirty-but-not-newer and dirty-and-newer cases.

## Reproduction Assessment

Yes. Source inspection on current main shows the dirty watched-source branch returns before the mtime check, and the current unit test still asserts that rebuild reason; I did not run tests because this review is read-only.

## Solution Assessment

Yes. Letting dirty watched trees fall through to the existing mtime freshness check is the narrow maintainable direction, provided the linked PR is updated or replaced against current main and preserves the other rebuild triggers.
