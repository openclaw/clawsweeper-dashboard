## Summary

Keep this issue open: current main and the latest release still force a rebuild on dirty watched source before checking source mtimes, and an open linked fork PR already carries the focused implementation candidate.

## What This Changes

The issue asks for `pnpm gateway:watch` and `pnpm dev` to skip rebuilding when the working tree is dirty but no watched source file is newer than the build stamp.

## Best Possible Solution

Land or replace the linked focused fix so dirty build-relevant source trees fall through to mtime freshness while preserving the existing force, HEAD, config, and missing-output rebuild triggers.

## Reproduction Assessment

Yes. Source inspection on current main shows the dirty watched-source branch returns before the mtime check, and the current unit test still asserts that rebuild reason; I did not run artifact-creating commands during this read-only review.

## Solution Assessment

Yes. Letting dirty watched trees fall through to the existing mtime freshness check is the narrow maintainable fix, as long as the linked PR or replacement preserves the other rebuild triggers.
