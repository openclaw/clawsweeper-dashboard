## Summary

Keep open. Current main still has the reported TUI lifecycle gap: the launcher forwards hangup signals, but the TUI runtime only handles SIGINT/SIGTERM and its exit path still waits on normal, unbounded terminal drain without first stopping status render activity or installing a hard fallback.

## What This Changes

The issue requests that `openclaw tui` exit cleanly on terminal close, SIGTERM, and gateway restart instead of leaving high-CPU, high-RSS orphaned TUI processes that require SIGKILL.

## Best Possible Solution

Land a focused TUI lifecycle fix that handles POSIX hangup, makes signal shutdown bounded and idempotent, stops status loader/timer activity before terminal teardown, adds focused regression tests, and leaves unrelated CPU normalization or session-bloat work to their canonical trackers.

## Reproduction Assessment

Yes. The issue gives concrete macOS LaunchAgent and terminal-close/gateway-restart steps, and current main has matching source evidence for missing SIGHUP handling plus unbounded signal cleanup; this read-only review did not run the live macOS reproduction.

## Solution Assessment

Yes for the repair direction. The narrow maintainable fix belongs in `src/tui/tui.ts` and focused tests, using the existing child-process bridge and terminal-drain seams rather than changing unrelated model-discovery or session-history code.

## Repair Work Prompt

Fix the TUI lifecycle bug reported in https://github.com/openclaw/openclaw/issues/75289. Current main still registers SIGINT/SIGTERM only in `src/tui/tui.ts`, uses unbounded `drainAndStopTuiSafely(tui)` during `requestExit()`, and leaves status loader/timers active until normal render transitions. Treat closed PR #75294 as useful prior context but create a fresh narrow fix. Expected boundary: update `src/tui/tui.ts` and focused tests in `src/tui/tui.test.ts` only unless a tiny helper split is clearly necessary; add a user-facing `CHANGELOG.md` fix entry. Implement POSIX SIGHUP handling, remove the listener on finish, make signal shutdown idempotent and bounded, stop status loader/statusTimer/waitingTimer before terminal teardown, and add a hard fallback if terminal drain/stop hangs. Verify the installed/source contract for `@mariozechner/pi-tui` before relying on `drainInput(maxMs, idleMs)`. Do not attempt to solve broader TUI/model-discovery CPU work from #75137/#75503 or session-bloat reports unless profiling in the PR proves a shared root cause.
