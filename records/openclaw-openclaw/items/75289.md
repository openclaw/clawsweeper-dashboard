## Summary

Keep open. Current main still lacks explicit TUI SIGHUP handling and the signal shutdown path still depends on unbounded terminal drain before finishing; open PR #75294 directly targets this terminal-hangup lifecycle bug and is still open, so this issue should wait for that PR or an equivalent replacement. The later CPU profiling context is useful, but it points to an adjacent model-discovery startup CPU path tracked by #75137/#75503 rather than fully superseding this report.

## What This Changes

The issue reports that `openclaw tui` processes can survive terminal close or gateway restart, continue spinning at high CPU/RSS, and require SIGKILL instead of exiting on SIGHUP/SIGTERM.

## Best Possible Solution

Land PR #75294 or an equivalent narrow TUI lifecycle fix that treats POSIX SIGHUP as shutdown, makes signal exit idempotent and bounded, stops status UI timers/loaders before terminal cleanup, and adds focused regression tests; keep the model-normalization CPU work in #75137/#75503 unless profiling proves a shared root cause.

## Reproduction Assessment

Yes. The issue gives concrete macOS LaunchAgent and terminal-close steps, and current main has matching source evidence for missing SIGHUP handling, unbounded drain on exit, and active status timers that are not stopped by `requestExit()`.

## Solution Assessment

Yes at the direction level. The narrowest maintainable fix belongs in `src/tui/tui.ts` with focused TUI shutdown tests; broader startup/model-discovery CPU work should stay separate unless a live profile ties it to this terminal-hangup path.
