## Summary

Keep open: current `main` still has a source-level path where spawned subagents can lose MCP tools because the inherited child allowlist is captured before bundle-MCP tools are appended, and the later bundled-tool policy pass applies that inherited allowlist to MCP tools. The reporter’s exported 2026.5.18 evidence and a second user report are consistent with this code path, and there is no current-main or shipped fix provenance that closes it.

## What This Changes

The issue reports that `sessions_spawn` subagent sessions omit allowed `mcp.servers` tools from their tool schema and prompt despite `bundle-mcp`, per-tool subagent allowlist entries, and target-agent allowlist entries.

## Best Possible Solution

Add a focused `sessions_spawn` regression with a stub MCP runtime and restrictive parent/target/subagent policies, then ensure the child inherited tool surface includes the already-filtered bundle-MCP tool names without bypassing any existing policy layer.

## Reproduction Assessment

Yes, from source inspection plus the reporter’s exported 2026.5.18 artifacts: current `main` builds the inherited child allowlist before materializing bundle-MCP tools, then applies that inherited policy to bundled tools in the child. I did not execute the full live current-main reproducer.

## Solution Assessment

Yes in direction, but it needs maintainer review for policy safety: the narrow fix should move or supplement inherited allowlist population so allowed filtered bundle-MCP tools survive into spawned children, not add a new config surface or bypass final policy.
