## Summary

Keep open: the report and latest root-cause comment match current main. The current tool-inheritance paths still replace restrictive parent/subagent policy with concrete tool names before bundle MCP tools are materialized, while the embedded child run only creates the MCP runtime if its allowlist still contains `bundle-mcp`, `group:plugins`, a concrete MCP tool name, or `*`.

## What This Changes

The issue requests that MCP tools configured through `mcp.servers` and allowed by subagent plus per-agent tool policy be injected into sessions spawned by `sessions_spawn`.

## Best Possible Solution

Fix the inheritance boundary so spawned subagents retain enough explicit MCP/plugin policy intent to create the bundle MCP runtime, then apply the final effective tool policy to the materialized MCP tools before prompt/schema injection.

## Reproduction Assessment

Yes at the source level, but not with a live current-main run in this review. The current code still has the described inheritance/runtime-ordering path, and the issue includes exported runtime evidence from 2026.5.18 plus an independent root-cause trace in the comments.

## Solution Assessment

Unclear until maintainers choose the exact policy-preservation shape. The likely maintainable direction is to carry explicit MCP/plugin allow tokens or defer inheritance until materialized MCP tools exist, while keeping the final effective policy pass as the security boundary.
