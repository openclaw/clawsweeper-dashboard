## Summary

Keep open: current main still has a plausible source-level gap where spawned subagents inherit only the parent’s already-materialized core/plugin tool names, while bundle MCP tools are materialized later only if the child allowlist still contains `bundle-mcp`, `group:plugins`, `*`, or a concrete MCP tool name.

## What This Changes

The issue reports that `mcp.servers` tools do not appear in `sessions_spawn` subagent tool schemas or prompts despite `bundle-mcp`, per-tool subagent allowlists, and target-agent allowlists.

## Best Possible Solution

Preserve or reconstruct the intended bundle MCP policy at the subagent inheritance boundary so configured MCP tools can materialize in child runs, then add regression coverage for `mcp.servers` plus `tools.subagents.tools.allow` and per-agent allowlists.

## Reproduction Assessment

Yes, source-reproducible: the issue includes concrete exported trajectory evidence, and current main still shows a code path where bundle MCP entries can be absent from the child inherited allowlist before MCP runtime creation. I did not run a live current-main gateway reproduction in this read-only pass.

## Solution Assessment

Unclear on final shape, but current main is not sufficient. The likely fix belongs at the subagent tool-policy inheritance/materialization boundary, not in provider-specific model behavior or a reporter-side config workaround.
