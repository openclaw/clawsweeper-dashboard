## Summary

Keep open: current main and the latest release still rebuild spawned-child inherited tool allowlists from concrete materialized tools, while bundled MCP runtime creation later depends on preserving `bundle-mcp`, `group:plugins`, or `server__tool` selector tokens. ## What This Changes

The issue reports that `sessions_spawn` child sessions lose configured MCP tools even when `bundle-mcp`, per-tool subagent allowlist entries, and the target agent allowlist should expose them.

## Best Possible Solution

Land or rework `https://github.com/openclaw/openclaw/pull/92521` or an equivalent narrow fix that preserves MCP selector tokens through subagent inheritance while applying the final effective policy intersection, with focused tests and MCP child-session proof.

## Reproduction Assessment

Yes from source inspection: `resolveGatewayScopedTools` captures only concrete tools into `inheritedToolAllowlist`, and `shouldCreateBundleMcpRuntimeForAttempt` later requires a bundle/plugin/MCP selector token to create the child MCP runtime. I did not run a live MCP server reproduction in this read-only review.
