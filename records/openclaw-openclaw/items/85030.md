## Summary

Keep open: the report has strong released-version evidence and a second user confirmation, while current main still lacks an exact `sessions_spawn` MCP regression proving the intended behavior. The current code also has an intentional delegated-session inheritance policy that can filter child tools by the caller's effective tool surface, so the remaining question is a maintainer tool-policy decision, not a safe cleanup close.

## What This Changes

The issue asks for MCP tools configured through `mcp.servers` and allowed by subagent and target-agent tool policy to be injected into `sessions_spawn` child sessions.

## Best Possible Solution

Define and test the delegated MCP contract: either document that the caller must allow MCP tools too, or add a maintainer-approved explicit delegation policy that lets selected target agents receive their own MCP tools without silently widening restricted callers.

## Reproduction Assessment

Yes for a source-level current-main path: spawned child sessions persist inherited caller tool policy and the final bundled-tool pass applies that policy to MCP tools. I did not run the reporter's exact all-combinations live reproducer against current main.

## Solution Assessment

Unclear as an implementation fix: the narrow code path is identifiable, but changing it is a security-sensitive policy choice about whether target agents may receive tools the spawning caller did not effectively have.
