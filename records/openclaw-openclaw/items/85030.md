## Summary

Keep open: the latest issue discussion and current source still show a real subagent bundle-MCP policy gap. The inherited subagent allowlist is rebuilt from gateway-visible concrete tool names, while configured MCP tools are materialized later in the embedded runner, so a spawned child can still filter out MCP tools even when `bundle-mcp` is explicitly allowed.

## What This Changes

MCP tools configured through `mcp.servers` are missing from native `sessions_spawn` child sessions even when `bundle-mcp`, per-tool subagent allowlists, and target-agent allowlists are configured.

## Best Possible Solution

Have maintainers choose a policy-safe fix that preserves explicit bundle-MCP inheritance or equivalent gateway-side MCP visibility, then add regression coverage for a spawned child with a stub MCP server and intersecting subagent/agent allowlists.

## Reproduction Assessment

Yes, source-level on current main: gateway inherited allowlists are built from concrete gateway-visible tools before bundle-MCP materialization, and the child final policy later filters bundled MCP tools through that inherited allowlist. I did not run the live `sessions_spawn` trajectory repro in this read-only review.

## Solution Assessment

Unclear which implementation shape maintainers prefer, but the right boundary is the existing inherited-tool-policy and bundle-MCP materialization path rather than a new user config surface. A narrow fix must preserve explicit bundle-MCP policy intent without broadening subagent access.
