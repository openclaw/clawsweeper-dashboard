## Summary

Keep open: checked-out main, the fetched origin/main tip, and latest release v2026.6.11 still rebuild spawned-child inherited tool allowlists from concrete materialized tools, while child bundled MCP startup under a restrictive allowlist still needs selector tokens such as `bundle-mcp`, `group:plugins`, or `server__tool`. The strongest direct fix candidate is https://github.com/openclaw/openclaw/pull/92521, but it is still open and conflicting.

## What This Changes

The issue requests that MCP tools configured through `mcp.servers` and permitted by subagent and per-agent tool policy appear in `sessions_spawn` child sessions' tool schema and prompt.

## Best Possible Solution

Rebase, review, and land https://github.com/openclaw/openclaw/pull/92521 or an equivalent narrow fix that preserves MCP selector tokens through inherited subagent allowlists while enforcing the final effective policy, with focused tests and real or stubbed MCP child-session proof.

## Reproduction Assessment

Yes. Source inspection shows the current inherited child allowlist is rebuilt from concrete tool names only, while child MCP materialization under a restrictive allowlist requires `bundle-mcp`, `group:plugins`, or an MCP-style selector; I did not run a live MCP server reproduction in this read-only review.

## Solution Assessment

No current-main solution is in place. The narrow selector-preservation approach in https://github.com/openclaw/openclaw/pull/92521 appears to be the best candidate, but it needs conflict resolution and maintainer security-policy review before landing.
