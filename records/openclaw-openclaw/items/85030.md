## Summary

Keep open: current main and v2026.5.22 still have a source-level path where a restricted delegator's inherited tool allowlist can clamp the spawned worker before MCP tools survive into the child tool surface, and there is still no exact current-main `sessions_spawn` MCP schema/prompt proof that the reported workflow is fixed.

## What This Changes

The issue asks for MCP tools from `mcp.servers` to appear in `sessions_spawn` child sessions when allowed by subagent policy and the target agent's tool policy.

## Best Possible Solution

Maintainers should decide the delegated-session MCP contract, then add a regression with a stub MCP server and a restricted delegator that proves the chosen behavior in the child schema and prompt, with matching docs or diagnostics.

## Reproduction Assessment

Yes at source level for the restricted-delegator shape, but not by an executed current-main run. Current code persists the caller's effective allowlist into the child and then applies that inherited allowlist to bundled MCP tools, which can remove worker MCP tools even when subagent and target-agent policy allow them.

## Solution Assessment

Unclear until maintainers choose the intended security contract. A narrow code fix is not safe if the current inherited-tool clamp is intentional; the maintainable path may be docs/diagnostics, or a carefully tested policy exception if maintainers want target-agent MCP tools to exceed the delegator surface.
