## Summary

Keep open: current `main` still has a credible source-level path where delegated-session inherited allowlists are captured before bundled MCP tools are appended, so a configured `sessions_spawn` child can lose MCP schemas even when direct sessions see them.

## What This Changes

The issue reports that MCP tools configured through `mcp.servers` are not injected into `sessions_spawn` subagent tool schemas or system prompts despite `bundle-mcp`, per-tool subagent allow entries, and target-agent allow entries.

## Best Possible Solution

Add a focused current-main regression that spawns a child against a stub MCP server and asserts the child provider schema and prompt include only MCP tools allowed by the parent effective surface, target agent, and `tools.subagents.tools`, then repair the inherited allowlist timing without weakening delegation policy.

## Reproduction Assessment

Yes at source level: the reporter supplied exported 2026.5.18 subagent evidence, and current `main` still captures delegated inherited allowlists before bundled MCP tools are appended. A live current-main repro remains the missing runtime proof.

## Solution Assessment

No patch is proposed here; the best fix is a narrow regression around the existing `sessions_spawn` and bundled-MCP policy path, not a new config surface or a bypass of inherited delegation restrictions.
