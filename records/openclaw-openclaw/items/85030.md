## Summary

Keep open: current main still has a credible source-level gap where restrictive subagent tool inheritance is computed before bundle MCP tools are materialized, so a spawned child can lose the `bundle-mcp` or `<server>__<tool>` token needed to initialize MCP tools. The later commenter’s root-cause trace matches the current code path and makes this stronger than the earlier needs-info state.

## What This Changes

The issue asks for MCP tools configured under `mcp.servers` to appear in `sessions_spawn` child sessions when allowed by subagent and target-agent tool policy.

## Best Possible Solution

Preserve the explicit MCP/plugin policy signal, or materialize and intersect MCP tool names before storing the inherited child allowlist, while keeping the final policy pass authoritative for subagent and target-agent restrictions.

## Reproduction Assessment

Yes at source level, but not via a live run in this read-only review. The current code stores the inherited child allowlist before bundle MCP tools are materialized, and the child runtime only initializes MCP when its allowlist still contains `bundle-mcp`, `group:plugins`, or a concrete MCP tool token.

## Solution Assessment

No, current main is not the best solution yet because it can collapse the parent policy into built-in concrete tool names before the child can initialize MCP. The maintainable fix should carry the explicit MCP/plugin policy signal or concrete MCP names through the spawn handoff without bypassing final policy filtering.
