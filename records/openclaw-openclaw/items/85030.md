## Label Justifications

- `P1`: A current and shipped regression breaks a documented multi-agent workflow for multiple users and has no safe merged fix.
- `impact:session-state`: The spawned child session persists an incomplete inherited tool capability set that suppresses configured MCP tools.
- `impact:security`: The defect and proposed correction directly affect parent-to-child tool authorization and have driven users toward unmanaged credential-bearing HTTP workarounds.

## Summary

Current main and the latest release still contain the reported MCP/subagent policy deadlock. A focused open PR targets it, but its current approach removes a parent-to-child authorization boundary, so the issue remains necessary until a safe implementation lands.

## What This Changes

Ensure MCP tools authorized by subagent and per-agent policy are materialized and exposed in native sessions created through `sessions_spawn`.

## Best Possible Solution

Retain inherited parent restrictions, carry an explicit authorized MCP-materialization fact or equivalent prepared policy into the child attempt, then apply the existing runtime allowlist and final effective policy to every discovered MCP tool; cover allowed and denied cases across persisted child turns.

## Reproduction Assessment

Yes, at high-confidence source level. Current main deterministically converts restrictive policy to a built-in-only inherited allowlist, persists it to the child, and then declines MCP runtime creation; the reporter's exported tools and system prompt show that exact outcome.

## Solution Assessment

No, the existing one-line deletion is not the best fix because it removes inherited allow enforcement. The safer fix preserves that authorization boundary while carrying authorized MCP materialization intent forward and filtering the resulting concrete tools.
