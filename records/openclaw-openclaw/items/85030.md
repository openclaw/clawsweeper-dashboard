## Label Justifications

- `P1`: The bug blocks a documented multi-agent workflow where spawned subagents should receive allowed MCP tools for delegated work, and multiple users confirmed the failure.
- `impact:session-state`: The failure is caused by child-session inherited tool state omitting the MCP selector tokens needed by the spawned runtime.
- `impact:security`: The affected path controls whether MCP-backed external capabilities and credentials can cross the subagent authorization boundary.

## Summary

Keep open: current main and the latest release still rebuild spawned-child inherited tool allowlists from concrete tools before bundled MCP tools are materialized, so configured `bundle-mcp` and MCP selector entries can still be lost for `sessions_spawn` child sessions.

## What This Changes

The issue reports that MCP tools configured through `mcp.servers`, subagent tool policy, and per-agent allowlists are missing from `sessions_spawn` child sessions, leaving subagents with only built-in tools.

## Best Possible Solution

Review, refresh, and land PR https://github.com/openclaw/openclaw/pull/92521 or an equivalent policy-filtered selector-preservation fix that keeps child MCP inheritance aligned with final materialized-tool filtering.

## Reproduction Assessment

Yes, source-reproducible. Current main stores a concrete-only inherited allowlist on child sessions, while child bundle-MCP runtime creation requires selector tokens that the inherited list can lose; I did not run a fresh live MCP server repro in this read-only pass.

## Solution Assessment

Yes, provisionally. Preserving policy-approved deferred selectors is narrower than removing inherited allow enforcement, but the authorization-boundary design needs maintainer approval before landing.
