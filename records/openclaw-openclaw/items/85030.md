## Summary

Keep open: the newer report is not safely covered by the earlier MCP fixes because current main still computes the delegated-session inherited allow surface before bundled MCP tools are appended, so subagent final policy can filter MCP tools back out. The fix needs maintainer review because it touches the security invariant from delegated tool-policy inheritance: child sessions should not exceed the caller's effective tool surface.

## What This Changes

The issue requests that MCP tools allowed by `mcp.servers`, `tools.subagents.tools`, and the target agent policy appear in `sessions_spawn` subagent tool schemas and prompts.

## Best Possible Solution

Keep the parent-surface inheritance invariant, but include post-materialized bundled MCP tools that survived the caller policy in the inherited allow surface, then add a regression test where caller, subagent policy, and target agent all allow the same MCP tool.

## Reproduction Assessment

Yes, by source inspection: inherited allow state is prepared before bundle-MCP tools are appended, then applied later to filter bundled MCP tools in the child run. I did not run a live gateway repro, but the reported trajectory matches that code path.

## Solution Assessment

Unclear for the reporter's broad expectation, because allowing a child to exceed a restricted caller is a policy decision. The maintainable fix is narrower: preserve caller inheritance and make inherited allow include bundled MCP tools only when they survived the caller's own effective policy.
