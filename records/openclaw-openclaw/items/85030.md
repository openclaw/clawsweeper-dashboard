## Summary

## What This Changes

The issue reports that MCP tools configured through `mcp.servers` are absent from `sessions_spawn` child subagent tool schemas and prompts even when subagent and target-agent allowlists include the MCP bundle and specific MCP tool names.

## Best Possible Solution

Add a focused current-main regression that spawns a child session against a stub MCP server and asserts the exported child tool schema/prompt include only tools allowed by sandbox, subagent, target-agent, and inherited parent policy; then fix the first policy/materialization layer that fails that contract.

## Reproduction Assessment

Yes for the reported 2026.5.18 symptom from the issue evidence, but only source-reproducible for current main: the code clearly has multiple gates that can strip bundle-MCP from subagents, and no exact current-main `sessions_spawn` export was run here.

## Solution Assessment

Unclear as an implementation choice until maintainers confirm the intended inherited-tool contract. The narrowest next step is a regression around the current policy intersection, not a new config surface or a broad MCP rewrite.
