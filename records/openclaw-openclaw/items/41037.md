## What This Changes

The PR backfills missing auth profile provider values from `<provider>:<name>` profile ids before config schema validation and adds regression coverage.

## Best Possible Solution

Land a rebased pre-schema normalization that infers provider only when it is omitted or intentionally blank, leaves malformed explicit values schema-fatal, preserves the current validation pipeline, and includes regression coverage plus contributor real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main requires `auth.profiles[*].provider` in the schema and validates without deriving it from a `qwen-portal:default` profile id first.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but it must preserve malformed explicit provider values as validation errors and be rebased onto the current validation pipeline.
