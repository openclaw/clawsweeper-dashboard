## Summary

This PR should stay open: it targets a real current-main config-validation bug, but the proposed backfill masks malformed explicit `provider` values and the external-contributor real behavior proof gate is still unmet.

## What This Changes

The PR adds pre-schema auth-profile provider inference from `<provider>:<name>` profile ids in config validation and regression tests for omitted-provider and explicit-provider precedence cases.

## Best Possible Solution

Land a narrow rebased fix that infers omitted or blank-string auth-profile providers before schema validation while preserving malformed explicit providers for schema rejection.

## Reproduction Assessment

Yes, by source inspection. Current main passes raw auth profiles to `OpenClawSchema.safeParse(...)`, and the schema requires `auth.profiles[*].provider` to be a string, so `{ mode: "oauth" }` is rejected before runtime handling.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch should infer only omitted or blank-string providers and leave non-string explicit providers for schema validation to reject.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve invalid explicit provider values:** `src/config/validation.ts:249-250`
  - body: The backfill treats every non-string `profileValue.provider` as absent, then replaces it with the provider inferred from the profile id. A config such as `{ provider: 123, mode: "oauth" }` would become valid instead of letting the schema reject the malformed explicit provider; infer only when the field is omitted or a blank string.
  - confidence: 0.88
