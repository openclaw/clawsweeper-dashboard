## Summary

Keep open. Current main still lacks auth-profile provider inference, and this PR targets a real config-validation bug, but the submitted patch can mask malformed explicit provider values and still needs the required changelog entry plus current-main rebase proof.

## What This Changes

The PR adds pre-schema config normalization to infer missing `auth.profiles[*].provider` values from `<provider>:<name>` profile ids, plus regression tests for inferred and explicit providers.

## Best Possible Solution

Land a narrow current-main fix that infers only omitted or blank auth-profile providers before schema validation, preserves malformed explicit values for schema rejection, covers edge cases, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a deterministic path: `auth.profiles["qwen-portal:default"] = { mode: "oauth" }` reaches schema validation with `provider` absent while the schema requires `provider: z.string()`.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch should infer only omitted or blank-string providers and preserve invalid explicit values for schema rejection.
