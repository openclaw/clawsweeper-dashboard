## What This Changes

The PR adds pre-schema config normalization to infer missing `auth.profiles[*].provider` values from `<provider>:<name>` profile ids and adds regression tests.

## Best Possible Solution

Land a rebased pre-schema normalization that infers provider only when the field is omitted or blank, leaves malformed explicit values schema-fatal, preserves the current validation pipeline, and includes regression coverage plus contributor real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main validates `auth.profiles` against a schema requiring `provider`, with no pre-schema inference for a profile keyed like `qwen-portal:default` and valued as `{ mode: "oauth" }`.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but it should only backfill omitted or blank-string providers and must preserve current validation behavior when rebased.
