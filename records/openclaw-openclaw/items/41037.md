## Summary

Keep this PR open: it targets a real current-main config-validation bug and is the canonical fix path for the closed linked report, but the proposed patch still has a correctness bug and the external contributor has not supplied after-fix real behavior proof.

## What This Changes

The PR adds pre-schema config normalization to infer missing `auth.profiles[*].provider` values from `<provider>:<name>` profile ids and adds regression coverage.

## Best Possible Solution

Land a rebased narrow fix that infers omitted or blank-string auth-profile providers before schema validation, preserves non-string explicit provider values for schema rejection, and keeps focused regression coverage for both paths.

## Reproduction Assessment

Yes. Source inspection on current main shows raw auth profiles reach `OpenClawSchema.safeParse(...)` before any provider inference, while the schema requires `auth.profiles[*].provider`.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch should only infer omitted or blank-string providers and preserve non-string explicit providers so malformed config is still rejected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve malformed explicit providers:** `src/config/validation.ts:252-253`
  - body: The backfill treats every non-string `profileValue.provider` as absent, then writes the provider inferred from the profile id. A config such as `{ provider: 123, mode: "oauth" }` would become valid instead of letting the schema reject the malformed explicit provider; infer only when the field is omitted or a blank string.
