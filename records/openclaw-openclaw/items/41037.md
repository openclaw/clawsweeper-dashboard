## Summary

Keep open: current main still lacks auth-profile provider inference and the PR addresses a real config-loading bug, but the proposed patch can mask malformed explicit provider values and still lacks required after-fix real behavior proof.

## What This Changes

The PR backfills missing `auth.profiles[*].provider` values from `<provider>:<name>` profile ids before config schema validation and adds focused regression tests.

## Best Possible Solution

Land a rebased narrow config normalization that backfills only omitted or blank-string providers, preserves malformed explicit providers for schema rejection, and includes focused regression coverage plus real behavior proof.

## Reproduction Assessment

Yes. Source inspection plus the linked report show current main sends auth profiles directly to a schema where `auth.profiles[*].provider` is required, so `{ mode: "oauth" }` fails before inference; I did not run a local command because this review was read-only.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch should distinguish omitted or blank-string providers from malformed explicit provider values and preserve the latter for schema rejection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve malformed explicit providers:** `src/config/validation.ts:252-253`

A config such as `{ provider: 123, mode: "oauth" }` would become valid instead of letting the schema reject the malformed explicit provider; only infer when the field is omitted or a blank string.
