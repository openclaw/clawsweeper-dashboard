## Summary

Keep this PR open because it targets a real current-main config validation bug, but it is not merge-ready: the patch can hide malformed explicit provider values, still lacks contributor real behavior proof, and is conflicting with newer validation code.

## What This Changes

The PR adds auth-profile provider backfill before config schema validation and focused regression tests for omitted and explicit string providers.

## Best Possible Solution

Land a rebased narrow fix that infers providers only when the field is omitted or blank, preserves malformed explicit provider values for schema rejection, and keeps focused regression coverage plus real behavior proof.

## Reproduction Assessment

Yes, from source and the linked report: current main still sends auth profiles directly to a schema where auth.profiles[*].provider is required, so { mode: "oauth" } fails before inference. I did not run a local test because this was a read-only review.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch must preserve non-string explicit provider values so malformed config remains invalid after the rebase.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve malformed explicit providers:** `src/config/validation.ts:252-253`
