## Summary

Keep open: current main still lacks the auth-profile provider backfill, and the linked bug is source-reproducible, but this PR still needs real behavior proof and a small correctness fix before merge.

## What This Changes

Adds a pre-schema config normalization step that infers missing auth profile providers from `<provider>:<name>` profile ids, plus regression tests.

## Best Possible Solution

Land a rebased narrow fix that infers omitted or blank-string auth-profile providers before schema validation, preserves malformed explicit provider values for schema rejection, and includes regression coverage for both paths.

## Reproduction Assessment

Yes. Source inspection on current main shows `validateConfigObjectRaw()` sends the raw profile value into `OpenClawSchema.safeParse(...)`, while the schema requires `auth.profiles[*].provider`, so `{ mode: "oauth" }` is rejected before runtime handling.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch should infer only omitted or blank-string providers and preserve non-string explicit providers so the schema can reject malformed config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve malformed explicit providers:** `src/config/validation.ts:252-253`
  - body: The backfill treats every non-string `profileValue.provider` as absent, then replaces it with the provider inferred from the profile id. A config such as `{ provider: 123, mode: "oauth" }` would become valid instead of letting the schema reject the malformed explicit provider; infer only when the field is omitted or a blank string.
  - confidence: 0.88
