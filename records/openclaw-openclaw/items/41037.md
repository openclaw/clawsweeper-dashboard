## Summary

Keep open: current main still rejects auth profile entries that omit `provider`, and this PR remains the canonical fix path for the linked reproduced bug, but the proposed patch has a blocking correctness issue, lacks contributor real behavior proof, and is conflicting with current main.

## What This Changes

The PR adds pre-schema config normalization to infer missing auth profile providers from `<provider>:<name>` profile ids and adds regression tests for inferred and explicit providers.

## Best Possible Solution

Land a rebased narrow fix that infers omitted or blank-string auth-profile providers before schema validation, preserves malformed explicit provider values for schema rejection, and keeps focused regression coverage.

## Reproduction Assessment

Yes. The linked report gives a concrete config shape, and current-main source still sends that shape to a schema where `auth.profiles[*].provider` is required before any provider inference.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch should preserve non-string explicit providers so malformed config remains invalid after the rebase.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve malformed explicit providers:** `src/config/validation.ts:252-253`
  - body: The backfill treats every non-string `profileValue.provider` as absent, then writes the provider inferred from the profile id.
