## Summary

Keep open. This PR targets a source-reproducible config/auth bug, but it still needs contributor real behavior proof and a small correctness fix so malformed explicit providers are not overwritten before schema validation.

## What This Changes

The PR adds pre-schema normalization that fills missing auth profile provider values from `<provider>:<name>` profile ids and adds focused config regression tests.

## Best Possible Solution

Land a rebased normalization that infers provider only when omitted or blank-string, preserves malformed explicit values for schema rejection, and includes regression coverage plus real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows current main still validates auth profiles against a schema requiring `provider`, with no pre-schema inference for a `{ mode: "oauth" }` profile keyed by `<provider>:<name>`.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch should only backfill omitted or blank-string providers and leave malformed explicit provider values for schema rejection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve malformed explicit providers:** `src/config/validation.ts:252-253`
  - body: The backfill treats every non-string `profileValue.provider` as absent, then writes the provider inferred from the profile id. A config such as `{ provider: 123, mode: "oauth" }` would become valid instead of letting the schema reject the malformed explicit provider; only infer when the field is omitted or a blank string.
  - confidence: 0.88
