## What This Changes

The PR adds pre-schema config normalization to infer missing auth profile `provider` values from `<provider>:<name>` profile ids and adds regression tests.

## Best Possible Solution

Land a rebased narrow fix that infers omitted or blank-string auth-profile providers before schema validation, preserves malformed explicit provider values for schema rejection, and includes regression coverage for both paths.

## Reproduction Assessment

Yes. Source inspection on current main shows `validateConfigObjectRaw()` sends the raw profile value into `OpenClawSchema.safeParse(...)`, while the schema requires `auth.profiles[*].provider`, so `{ mode: "oauth" }` is rejected before runtime handling.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch should infer only omitted or blank-string providers and preserve non-string explicit providers so the schema can reject malformed config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

A config such as `{ provider: 123, mode: "oauth" }` would become valid instead of letting the schema reject the malformed explicit provider; infer only when the field is omitted or a blank string.
