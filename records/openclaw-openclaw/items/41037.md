## Summary

Keep open: the linked config bug still exists on current main, but this PR still has a correctness blocker that can accept malformed explicit provider values, and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR adds pre-schema normalization to infer missing `auth.profiles` provider values from `<provider>:<name>` keys and adds config regression tests.

## Best Possible Solution

Land a rebased narrow normalization that infers only absent or blank-string auth-profile providers before schema validation, leaves malformed explicit providers for schema rejection, and includes regression coverage plus real CLI/output proof.

## Reproduction Assessment

Yes. Source inspection on current main shows a config with `auth.profiles["qwen-portal:default"] = { mode: "oauth" }` reaches `OpenClawSchema.safeParse(...)`, while the schema requires `provider: z.string()`.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch should infer only omitted or blank-string providers and preserve non-string explicit provider values for schema rejection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve invalid explicit provider values:** `src/config/validation.ts:249-250`
  - body: The backfill treats every non-string `profileValue.provider` as absent, then replaces it with the provider inferred from the profile id. A config such as `{ provider: 123, mode: "oauth" }` would become valid instead of letting the schema reject the malformed explicit provider; infer only when the field is omitted or a blank string.
  - confidence: 0.88
