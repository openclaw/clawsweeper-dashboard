## Summary

Keep open: the underlying config-load bug is still source-reproducible on current main, but this PR is not merge-ready because it is merge-conflicting, lacks contributor real behavior proof, and can turn malformed explicit provider values into valid config before schema validation.

## What This Changes

The PR backfills missing `auth.profiles[*].provider` values from `<provider>:<name>` profile ids before config schema validation and adds focused regression tests.

## Best Possible Solution

Land a rebased pre-schema normalization that infers provider only when omitted or blank, keeps malformed explicit provider values schema-fatal, and includes focused regression coverage plus redacted real config-load proof.

## Reproduction Assessment

Yes, by source inspection. Current main still requires `auth.profiles[*].provider` in the schema and does not infer it from a key like `qwen-portal:default` before parsing.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but it must not overwrite malformed explicit provider values before schema validation can reject them.
