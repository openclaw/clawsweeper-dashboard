## Summary

Keep this PR open: the linked config-validation bug still exists on current main and the proposed seam is useful, but the branch should not merge while it can overwrite malformed explicit provider values and still lacks contributor real behavior proof.

## What This Changes

The branch adds pre-schema auth profile provider inference in src/config/validation.ts and regression coverage in src/config/config.schema-regressions.test.ts.

## Best Possible Solution

Land a rebased normalization that infers provider only when omitted or blank-string, preserves malformed explicit values for schema rejection, and includes focused regression coverage plus contributor real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main still validates auth profiles against a schema requiring provider, with no pre-schema inference for a profile keyed like qwen-portal:default and valued as { mode: "oauth" }.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch should only backfill omitted or blank-string providers and leave malformed explicit provider values for schema rejection.

## Review Findings
