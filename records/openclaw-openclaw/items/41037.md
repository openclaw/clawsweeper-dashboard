## Summary

Keep open. The underlying config bug is still present on current main, but this PR is not merge-ready because the proposed backfill can mask malformed explicit provider values and the user-facing fix needs a changelog entry.

## What This Changes

The PR adds pre-schema config normalization that infers missing `auth.profiles[*].provider` values from `<provider>:<name>` profile ids and adds regression tests for inferred and explicit providers.

## Best Possible Solution

Land a rebased fix that performs auth-profile provider inference in `validateConfigObjectRaw` before legacy and schema validation, preserves malformed explicit provider values for Zod rejection, covers omitted, blank, explicit, non-string, and malformed-id cases, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a direct reproduction: `auth.profiles["qwen-portal:default"] = { mode: "oauth" }` reaches schema validation with `provider` absent, while the schema requires `provider: z.string()`.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the PR should infer only omitted or blank-string providers, preserve invalid explicit values, and be rebased onto the current validation pipeline.
