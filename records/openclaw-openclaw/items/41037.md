## Summary

Keep open: current main still has the reported config-validation gap, and this PR is a plausible fix direction, but it is not merge-ready because it conflicts with current config validation code, lacks contributor real behavior proof, and can mask malformed explicit provider values.

## What This Changes

The PR adds a pre-validation auth profile provider backfill and regression tests for omitted providers in `auth.profiles`.

## Best Possible Solution

Land a rebased pre-schema normalization that infers provider only when the field is omitted or blank, keeps malformed explicit provider values schema-fatal, and includes focused regression coverage plus redacted real config-load proof.

## Reproduction Assessment

Yes, by source inspection. Current main validates `auth.profiles` against a schema requiring `provider`, with no pre-schema inference for a profile keyed like `qwen-portal:default` and valued as `{ mode: "oauth" }`.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch should only backfill omitted or blank-string providers and leave malformed explicit provider values for schema rejection.
