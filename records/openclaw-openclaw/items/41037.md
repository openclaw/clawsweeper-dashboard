## Summary

Keep open. The underlying bug is still present on current main, but this PR should not merge as-is because its backfill masks malformed explicit `provider` values before schema validation and the patch is based on an older validation flow.

## What This Changes

The PR adds pre-schema normalization for auth profiles so missing `provider` fields are inferred from `<provider>:<name>` profile ids, plus regression tests for inferred and explicit providers.

## Best Possible Solution

Keep the pre-schema normalization approach, but implement it on current main after deprecated-key stripping and before legacy/schema validation; infer only omitted or blank-string providers from well-formed profile ids, preserve malformed explicit providers for Zod rejection, and cover omitted, blank, explicit, non-string explicit, and malformed-id cases with focused regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: current main sends a profile like `{ auth: { profiles: { "qwen-portal:default": { mode: "oauth" } } } }` into a schema that requires `provider`, with no prior provider-id inference.

## Solution Assessment

No, not as currently proposed. The normalization seam is the narrow maintainable fix, but the PR needs a current-main rebase and must not overwrite non-string explicit `provider` values before schema validation.

## Review Findings
