## Summary

Keep open: current main still has the missing-provider validation failure, and this PR targets the right compatibility bug, but the branch is conflicting, lacks contributor real behavior proof, and has a normalization bug that can mask malformed explicit provider values.

## What This Changes

The PR adds an auth-profile provider backfill in `src/config/validation.ts` and regression tests in `src/config/config.schema-regressions.test.ts`.

## Best Possible Solution

Land a rebased pre-schema normalization that infers provider only when provider is omitted or a blank string, keeps malformed explicit values schema-fatal, and includes focused regression coverage plus contributor real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main still validates `auth.profiles` against a schema requiring `provider`, with no pre-schema inference for a profile keyed like `qwen-portal:default` and valued as `{ mode: "oauth" }`.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch should only backfill omitted or blank-string providers and leave malformed explicit provider values for schema rejection.

## Review Findings

Overall confidence: 0.88
