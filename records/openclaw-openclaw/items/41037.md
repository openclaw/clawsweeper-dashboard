## Summary

Keep this PR open: it targets a real current-main config validation bug, but the proposed normalization can mask malformed explicit provider values and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds pre-schema auth-profile provider backfill from `<provider>:<name>` profile ids and focused config regression tests.

## Best Possible Solution

Land a rebased normalization that infers provider only when omitted or blank-string, preserves malformed explicit values for schema rejection, and includes regression coverage plus real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows current main sends auth profiles directly into a schema where `auth.profiles[*].provider` is required, so `{ mode: "oauth" }` remains a schema failure before inference.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but it should only backfill omitted or blank-string providers and leave malformed explicit provider values for schema rejection.

## Review Findings

Overall confidence: 0.88
