## Summary

Keep open: this is a real remaining config-validation bug on current main, but this PR is not merge-ready because it can mask malformed explicit provider values, lacks after-fix real behavior proof, and is currently conflicting with main.

## What This Changes

The PR adds a pre-schema auth profile provider backfill in `src/config/validation.ts` and regression tests for omitted and explicit string providers.

## Best Possible Solution

Land a rebased narrow config normalization that backfills only omitted or blank-string auth profile providers, preserves malformed explicit providers for schema rejection, and includes focused regression coverage plus real behavior proof.

## Reproduction Assessment

Yes. Source inspection plus the linked report show current main still sends auth profiles directly to a schema where `auth.profiles[*].provider` is required, so `{ mode: "oauth" }` fails before inference; I did not execute the repro because this review was read-only.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch should distinguish omitted or blank-string providers from malformed explicit provider values and preserve the latter for schema rejection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve malformed explicit providers:** `src/config/validation.ts:252-253`
  - body: The backfill treats every non-string `profileValue.provider` as absent, then writes the provider inferred from the profile id. A config such as `{ provider: 123, mode: "oauth" }` would become valid instead of letting the schema reject the malformed explicit provider; only infer when the field is omitted or a blank string.
