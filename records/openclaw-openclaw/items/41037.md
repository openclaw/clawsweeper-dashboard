## Summary

This PR should stay open because it targets a real config-validation bug that is still present on current main, but the patch still masks malformed explicit provider values and lacks contributor-provided real behavior proof.

## What This Changes

Adds config-validation normalization that infers missing auth profile providers from `<provider>:<name>` profile ids before schema validation, plus regression tests.

## Best Possible Solution

Land a rebased narrow fix that infers omitted or blank auth-profile providers before schema validation, preserves malformed explicit provider values for schema rejection, and covers both paths with regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `validateConfigObjectRaw()` sends the raw config into `OpenClawSchema.safeParse(...)`, while `auth.profiles` values require `provider: z.string()`, so `{ mode: "oauth" }` is rejected before runtime handling.

## Solution Assessment

No, not as currently proposed. Pre-schema normalization is the right seam, but the patch should infer only omitted or blank-string providers and preserve non-string explicit provider values for schema rejection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve invalid explicit provider values:** `src/config/validation.ts:249-250`
  - body: The backfill treats every non-string `profileValue.provider` as absent, then replaces it with the provider inferred from the profile id. A config such as `{ provider: 123, mode: "oauth" }` would become valid instead of letting the schema reject the malformed explicit provider; infer only when the field is omitted or a blank string.
