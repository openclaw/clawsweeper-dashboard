## Summary

Keep open. Current main still has a hard-coded 2.5s Telegram startup `getMe` probe that matches the reported `fetch-timeout` symptom, while the broader gateway and pairing symptoms need separate evidence.

## What This Changes

The issue requests longer/configurable Telegram request timeouts, retry handling, clearer timeout/auth errors, and a possible scope-upgrade recovery path for Telegram startup failures.

## Best Possible Solution

Fix the Telegram plugin startup path to reuse the existing `getMe`/configured timeout policy for the probe, with focused tests, and keep pairing bypass or gateway-port diagnosis separate unless independently reproduced.

## Reproduction Assessment

Partial yes: source inspection gives a high-confidence unit reproduction by stubbing the startup probe/fetch to exceed the hard-coded 2500ms budget. The gateway-not-listening and scope-upgrade sequence is not reproducible from the report alone.

## Solution Assessment

Yes for a narrow timeout repair: wiring startup `getMe` to the existing Telegram timeout policy is the maintainable fix. A blanket 30s+ timeout increase or bypassing pairing for `dmPolicy="allowlist"` would be broader than the confirmed bug.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75783 narrowly in the Telegram plugin. Current main still calls `resolveTelegramProbe()(token, 2500, ...)` during `gateway.startAccount`, so slow Telegram `getMe` startup probes can abort after 2.5s even though the runtime request policy has `getme: 15000` and `channels.telegram.timeoutSeconds`. Replace the hard-coded startup probe budget with the existing Telegram `getMe`/configured timeout policy, preserving unauthorized-token fast failure and existing fallback behavior for non-auth probe failures. Add focused coverage in `extensions/telegram/src/channel.gateway.test.ts` proving startup uses the intended timeout for default and configured/per-account timeout cases. Do not change pairing/scope-upgrade authorization semantics, do not add a blanket 30s timeout for every Telegram method, and do not move Telegram-owned behavior into core. Add a single-line `CHANGELOG.md` Fixes entry for this user-facing bug, crediting the reporter if policy allows. Validate with the focused Telegram tests and formatter first, then the changed gate in Testbox if available.
