## Summary

Keep open. Current main still has a hard-coded 2.5s Telegram gateway startup `getMe` probe that matches the reported `timeoutMs:2500` symptom, even though other Telegram API paths now have stronger timeout handling and public timeout configuration.

## What This Changes

The issue reports Telegram Bot API fetch timeouts, a gateway connectivity probe failure, and native approval pairing failure on OpenClaw v2026.4.29 under Linux/Node 22.

## Best Possible Solution

Keep the fix in the Telegram plugin/runtime surface: align the gateway startup `getMe` probe budget with the existing Telegram timeout/status configuration, add focused tests, and preserve device pairing approval semantics. If the gateway-port symptom remains afterward, split it into a separate gateway health report with logs and config.

## Reproduction Assessment

Partial: source inspection gives a high-confidence unit reproduction for the `getMe` timeout by stubbing the startup probe/fetch path past the fixed 2.5s budget. The broader gateway-not-listening and scope-upgrade sequence is not reproducible from the report alone.

## Solution Assessment

Unclear for the whole issue: increasing every Telegram timeout to 30s+ or bypassing pairing is too broad and could weaken security expectations. A targeted startup probe timeout fix is the narrowest maintainable first step.
