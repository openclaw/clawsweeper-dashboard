## Summary

Keep open. Current main already has Telegram client timeout configuration and stronger runtime Bot API request guards, but the reported `fetch-timeout` `getMe` symptom still maps to a current hard-coded 2.5s Telegram startup probe budget, so this is not safe to close as implemented.

## What This Changes

The issue asks OpenClaw to handle Telegram Bot API fetch timeouts on Linux/Node 22 by using longer configurable timeouts, retries, clearer errors, and avoiding a reported approval/pairing deadlock.

## Best Possible Solution

Keep this in the Telegram plugin/runtime area and make the startup `getMe` probe budget consistent with the existing Telegram timeout/status configuration while preserving device pairing approval semantics. If the gateway-port symptom remains after that, it should be split into a focused gateway health report with logs and config.

## Reproduction Assessment

Partial: source inspection gives a high-confidence unit reproduction for the `fetch-timeout` `getMe` symptom by stubbing Telegram probing to exceed the hard-coded 2.5s startup budget. The wider gateway-not-listening and scope-upgrade sequence is not reproducible from the report alone.

## Solution Assessment

Unclear for the whole issue: increasing every Telegram timeout to 30s+ or bypassing pairing is too broad and weakens security expectations. A targeted fix to the startup probe budget, plus tests and clearer diagnostics, is the narrowest maintainable first step.
