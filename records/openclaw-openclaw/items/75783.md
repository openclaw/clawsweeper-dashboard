## Summary

Keep open. Current main still has a hard-coded 2.5s Telegram startup `getMe` probe that matches the reported `fetch-timeout` symptom, even though the broader Telegram runtime already has configurable and method-specific request guards.

## What This Changes

The issue reports Telegram fetch timeouts on v2026.4.29 and asks for longer configurable timeouts, retries, clearer diagnostics, and a pairing/scope-upgrade fix.

## Best Possible Solution

Keep the fix in the Telegram plugin startup path: replace the hard-coded startup probe budget with the existing `getMe`/configured timeout policy, add focused tests, and leave pairing approval bypass or gateway-port diagnosis for separate evidence if symptoms remain.

## Reproduction Assessment

Partial yes: source inspection gives a high-confidence reproduction for the `getMe` `fetch-timeout` by stubbing the startup probe/fetch to exceed 2.5s. The gateway-not-listening and scope-upgrade sequence is not reproducible from the report alone.

## Solution Assessment

Partially: a targeted startup-probe timeout fix is the narrow maintainable path. Increasing every Telegram timeout to 30s+ or bypassing pairing for `dmPolicy="allowlist"` is too broad and would cross auth semantics without enough evidence.
