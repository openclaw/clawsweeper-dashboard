## Summary

Keep open. Current main still has the hard-coded 2.5s Telegram startup `getMe` probe that matches the reported `fetch-timeout`, while the normal Telegram runtime already has longer method-specific and configurable request timeouts.

## What This Changes

Fix Telegram startup/API timeout handling so slow Telegram Bot API calls do not fail behind the current 2.5s startup probe budget.

## Best Possible Solution

Keep the fix inside the Telegram plugin startup path: use the existing `getMe`/configured timeout policy for the startup probe, add focused tests, and leave pairing approval or gateway-port behavior unchanged unless separately reproduced.

## Reproduction Assessment

Partial yes: source inspection gives a high-confidence reproduction for the startup `getMe` timeout by stubbing the Telegram probe/fetch to exceed 2.5s. The gateway-not-listening and scope-upgrade sequence is not reproducible from the report alone.

## Solution Assessment

Yes for the narrow timeout gap: replacing the hard-coded startup probe budget with the existing Telegram timeout policy is the maintainable fix. Raising every Telegram timeout to 30s+ or bypassing pairing for `dmPolicy="allowlist"` is broader than the evidence supports.
