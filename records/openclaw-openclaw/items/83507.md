## Summary

This should stay open: current main and the latest release still have the reported asymmetry. The `agent.send` rotation path persists a new session and emits `sessions.changed`, while the existing reset/create paths use the session lifecycle hook helpers; the related fix PR was closed unmerged, so there is no implementation on main to close against.

## What This Changes

The issue requests that `agent.send` session ID auto-rotation emit plugin session lifecycle hooks consistently with reset/create paths.

## Best Possible Solution

Add a focused gateway fix that emits the paired session lifecycle boundary from `agent.send` auto-rotation using the existing helpers, derives the correct end reason and `resumedFrom` metadata, and covers caller-supplied, daily/idle, and failed-transcript rotation cases while preserving `sessions.changed`.

## Reproduction Assessment

Yes, by source inspection rather than a live gateway run: `agent.send` sets `isNewSession`, persists the rotated session ID, and emits only `sessions.changed`, while reset/create paths call the lifecycle helpers. I did not run a live plugin reproduction in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to reuse the established gateway lifecycle helpers at the existing `isNewSession` boundary, preserving `sessions.changed` and adding regression coverage for ordering, reason, and `resumedFrom` metadata.
