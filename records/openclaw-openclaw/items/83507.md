## Summary

This should stay open: current main and the latest release still have the reported gateway session-lifecycle asymmetry, and the closed related PR was not merged. Source inspection shows `agent.send` rotates/persists a new session ID and emits `sessions.changed`, but the lifecycle hook helper used by reset/create paths is not called there.

## What This Changes

The issue asks for `agent.send` session ID auto-rotation to emit the existing `session_start` plugin hook consistently with other session-creation paths.

## Best Possible Solution

Reuse the existing gateway lifecycle helpers in the `agent.send` auto-rotation path, preserving `sessions.changed` while adding focused coverage for stale daily or idle rotation and caller-supplied replacement session IDs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run `agent.send` against a stale or explicitly replaced `sessionKey` and observe that the rotation branch emits `sessions.changed` without calling the lifecycle hook helper used by reset/create paths.

## Solution Assessment

Yes. The narrow maintainable fix is to route the existing `isNewSession` transition through the established session lifecycle helper surface, with tests proving the lifecycle hook(s) and existing `sessions.changed` event fire exactly once and in the expected order.
