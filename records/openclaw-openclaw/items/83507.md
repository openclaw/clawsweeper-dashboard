## Summary

This should stay open: current main still has the reported asymmetry, and it is a narrow gateway session lifecycle bug rather than stale cleanup. The `agent.send` path rotates and persists a new session ID, then emits `sessions.changed`, while the documented session lifecycle hook helper is only used by reset/create paths.

## What This Changes

The issue requests that `agent.send` emit the `session_start` plugin hook when it auto-rotates a `sessionKey` to a new `sessionId`.

## Best Possible Solution

Reuse the existing gateway session lifecycle helper path for `agent.send` auto-rotation, preserving `sessions.changed` while adding regression coverage for stale daily rotation and explicit replacement session IDs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: run `agent.send` against a stale or explicitly replaced `sessionKey` and observe that only `sessions.changed` is emitted from the rotation path, while reset/create paths call the lifecycle hook helper.

## Solution Assessment

Yes. The narrow maintainable fix is to route the existing `isNewSession` transition through the established session lifecycle hook helper, with tests proving the hook and the existing `sessions.changed` event both fire exactly once.
