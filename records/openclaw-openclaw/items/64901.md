## Summary

Keep open, but do not merge as-is: the branch adds a useful generic warning path, but it still does not warn for the stated Bedrock fallback because current discovery materializes fallback values as model metadata before the guard sees them.

## What This Changes

The branch adds a `shouldWarnDefault` guard flag, logs when the resolved context-window source is `default`, and extends context-window guard tests.

## Best Possible Solution

Fold this warning into one provenance-aware Bedrock fallback path so known models use table values and unknown literal defaults carry explicit provenance and emit a clear runtime warning.

## Reproduction Assessment

Yes, by source inspection. Current Bedrock discovery can still produce a positive fallback `contextWindow`, and current guard resolution treats any positive model value as `source: "model"`, so this PR's `source === "default"` warning does not fire for that stated path.

## Solution Assessment

No, not by itself. A source-default warning is useful, but the Bedrock case needs fallback provenance or non-materialized defaults before the warning accurately covers the user-visible problem.
