## Summary

Keep open: this PR targets a real source-reproducible async gateway exec output bug, but it should not merge as written because the new truncation notice uses a flag that also fires when only the pending poll buffer was capped, so intact approval output can be falsely marked truncated.

## What This Changes

The PR preserves multiline output for approved async gateway exec follow-ups, carries exec truncation state into the follow-up outcome, and adds regression coverage for Discord-routed async approval output over 400 characters.

## Best Possible Solution

Keep the narrow gateway approval formatter fix, but split aggregate-output truncation from pending-buffer truncation before merge, then land after focused regression coverage and normal maintainer review.

## Reproduction Assessment

Yes for source-level reproduction: current main still builds approved gateway exec follow-ups with a 400-character tail and whitespace normalizer. I did not run a live Discord approval round trip in this read-only review.

## Solution Assessment

Not yet as written. The formatter boundary is the right fix shape, but the PR needs a separate aggregate truncation signal so it does not add false capture-limit notices when only the pending poll buffer was capped.
