## Summary

Keep this PR open: it fixes a real `session_status` reasoning-default display bug, but the new helper does its per-agent lookup with raw `entry.id === agentId`, which can miss agent configs that the rest of the agent stack resolves through normalized ids.

## What This Changes

The branch changes `session_status` to resolve reasoning visibility from the active session override, then per-agent `reasoningDefault`, global default, and `off`, with focused tests.

## Best Possible Solution

Keep the precedence fix, but read the per-agent reasoning default through the same normalized agent-config lookup used elsewhere and add a regression for a non-canonical configured agent id.

## Reproduction Assessment

Yes. On current main, a session without `reasoningLevel` passes `off` into `buildStatusText`, so configured reasoning defaults cannot be displayed; the remaining PR edge case is also source-reproducible with a configured agent id that normalizes differently from the session key.

## Solution Assessment

No, not as written. The precedence approach is the right narrow fix, but the new helper should reuse or mirror the normalized agent-config lookup instead of exact raw id matching.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Normalize the per-agent reasoning lookup:** `src/agents/tools/session-status-tool.ts:84`
  - body: `agentId` has already been normalized from the session key, but this new lookup compares it to raw `agents.list[].id`. Other agent config lookups normalize entry ids, so a config like `{ id: "Docs", reasoningDefault: "on" }` still resolves the agent elsewhere while `session_status` falls back to the global/default reasoning value here. Use the same normalized agent-entry lookup before reading `reasoningDefault`.
  - confidence: 0.88
