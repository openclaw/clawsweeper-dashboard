## Summary

Keep open. The PR targets a valid MCP stdio environment inheritance bug, and the reporter added useful production context, but the branch is stale against current main's stdio env safety hardening, lacks after-fix real behavior proof, and needs an explicit NODE_OPTIONS startup-policy decision.

## What This Changes

The branch adds MCP stdio launch-time propagation of gateway proxy/NODE_OPTIONS env defaults and tests for precedence, empty proxy values, case deduplication, and inspect-flag filtering.

## Best Possible Solution

Land a refreshed fix that propagates safe ambient proxy defaults at the stdio launch boundary with user-config precedence, preserves toMcpEnvRecord/onDroppedEnv filtering, and documents/tests a deliberate NODE_OPTIONS host-inheritance policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, from source inspection: start the gateway with HTTPS_PROXY or NODE_OPTIONS, configure a stdio MCP server without explicit env, and current main resolves no ambient proxy/default startup env before the transport adds only the MCP SDK allowlist.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently proposed. The resolver boundary is right, but the branch must be rebased around current toMcpEnvRecord/onDroppedEnv filtering and the NODE_OPTIONS inheritance policy.
