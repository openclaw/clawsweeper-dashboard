## Summary

Keep this PR open. The linked production report and prior Slack proxy fix make the MCP stdio proxy inheritance gap credible, but the branch is stale against current main's stdio env safety filter and would re-open configured interpreter-startup env propagation if rebased as-is.

## What This Changes

The PR adds MCP stdio launch-time propagation of ambient proxy-related environment variables and sanitized NODE_OPTIONS from the gateway process, plus tests for precedence, empty values, and inspect-flag filtering.

## Best Possible Solution

Land a refreshed MCP stdio launch-path change that propagates safe ambient proxy defaults with explicit user-config precedence, preserves the current configured-env denylist and warnings, and documents the accepted host NODE_OPTIONS behavior before closing the linked bug.

## Reproduction Assessment

Yes. Set HTTPS_PROXY or NODE_OPTIONS on the gateway process, resolve a stdio MCP server config with no explicit env on current main, and the resolver returns no proxy defaults while the transport later adds only the MCP SDK's narrow default environment.

## Solution Assessment

No, not as currently proposed. The resolver boundary is the right place for the fix, but the branch must be rebased around toMcpEnvRecord/onDroppedEnv and needs a maintainer-approved NODE_OPTIONS policy to be the narrowest maintainable solution.
