## Summary

Keep open. The requested behavior is still valid and current main does not propagate ambient proxy/NODE_OPTIONS into MCP stdio children, but this PR cannot merge as-is because it bypasses current stdio env-safety filtering, lacks after-fix real behavior proof, and needs an explicit NODE_OPTIONS inheritance decision.

## What This Changes

The PR adds MCP stdio launch-time propagation of ambient proxy and NODE_OPTIONS env defaults, plus tests for precedence, case handling, empty values, and inspector-flag stripping.

## Best Possible Solution

Land a refreshed fix that propagates safe ambient proxy defaults at the stdio launch boundary with user-config precedence, preserves toMcpEnvRecord/onDroppedEnv filtering, and documents/tests a deliberate NODE_OPTIONS host-inheritance policy.

## Reproduction Assessment

Yes, from source inspection: start the gateway with HTTPS_PROXY or NODE_OPTIONS, configure a stdio MCP server without explicit env, and current main resolves no ambient proxy/default startup env before the transport adds only the MCP SDK allowlist.

## Solution Assessment

No, not as currently proposed. The resolver boundary is right, but the branch must be rebased around current toMcpEnvRecord/onDroppedEnv filtering and an explicit NODE_OPTIONS inheritance policy.
