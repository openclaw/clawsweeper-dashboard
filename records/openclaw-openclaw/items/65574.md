## Summary

Keep this PR open, but it should not merge as-is. Current main still does not propagate ambient gateway proxy env or NODE_OPTIONS into MCP stdio child processes, and the linked reporter provided a credible production proxy-only failure; however the PR branch is stale against the merged stdio env safety filter and would need a careful rebase/rework before review can proceed.

## What This Changes

The PR adds MCP stdio launch-time propagation of ambient proxy-related environment variables and sanitized NODE_OPTIONS from the gateway process, plus tests for proxy precedence, empty values, and inspect-flag filtering.

## Best Possible Solution

Land a refreshed MCP stdio launch-path change that propagates safe ambient proxy defaults with explicit user-config precedence, preserves current configured-env safety filtering and warnings, and documents the accepted host NODE_OPTIONS behavior before closing the linked bug.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to set HTTPS_PROXY or NODE_OPTIONS on the gateway process, resolve a stdio MCP server config with no explicit env on current main, and observe that the resolver returns no proxy defaults while the transport later only merges the MCP SDK's narrow default environment.

## Solution Assessment

No, not as currently proposed. The resolver boundary is the right place for the fix, but this branch must be rebased around toMcpEnvRecord/onDroppedEnv and needs maintainer-approved NODE_OPTIONS policy before it is the narrowest maintainable solution.
