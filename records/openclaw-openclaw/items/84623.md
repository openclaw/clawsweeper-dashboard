## Summary

Keep open: the reporter provided concrete WeChat duplicate-send logs, and current main still has a plausible shared reply-dispatch gap where duplicate final payload entries are delivered one by one while existing direct-block-key dedupe only covers part of the path.

## What This Changes

The issue asks OpenClaw to stop non-streaming `dispatchReplyFromConfig` from delivering the same final text or `MEDIA:` payload twice to channel plugins.

## Best Possible Solution

Add a focused regression for non-streaming media/text dispatch and fix the shared reply payload filtering or dispatcher path so an already-delivered logical reply is not delivered again while preserving intentional multi-message replies.

## Reproduction Assessment

No high-confidence live reproduction yet; source inspection and reporter logs make the duplicate-delivery path reproducible at medium confidence. A focused regression should exercise `dispatchReplyFromConfig` with `disableBlockStreaming: true`, a direct media block, and a final payload that would otherwise be delivered again.

## Solution Assessment

No, the plugin-side dedupe workaround is not the best fix. The maintainable solution belongs in the shared reply dispatch/payload pipeline so every channel plugin gets exactly-once delivery for one logical reply.
