## What This Changes

The issue reports duplicate text or media delivery from non-streaming `dispatchReplyFromConfig` when a channel plugin disables block streaming.

## Best Possible Solution

Add focused regression coverage for non-streaming text and media delivery through the shared reply-dispatch path, then fix core payload filtering or delivery state so one logical reply is delivered once while preserving intentional multi-message replies and fallback media.

## Reproduction Assessment

## Solution Assessment

No, the plugin-side time-window dedupe workaround is not the best solution. The durable fix should live in the shared reply dispatch or payload filtering path so channel plugins do not have to suppress duplicated core deliveries themselves.
