## Summary

Keep open: the report describes a real duplicate outbound delivery class, and current main still lacks a focused proof that identical non-streaming final payloads from `dispatchReplyFromConfig` are collapsed before delivery.

## What This Changes

The issue asks OpenClaw to prevent non-streaming `dispatchReplyFromConfig` from invoking a channel plugin's delivery callback twice for the same final text/media payload.

## Best Possible Solution

Add focused core regression coverage for non-streaming text and media delivery through the shared reply dispatcher, then fix the shared payload filtering or final dispatch loop so one logical reply is delivered once while intentional multi-message replies still pass through.

## Reproduction Assessment

Yes, at source level with medium confidence: duplicate `replyResult` array entries would still be sent once per entry by the final dispatch loop, and the reporter supplied concrete duplicate media logs. I did not run a live WeChat/plugin reproduction in this read-only review.

## Solution Assessment

Yes, the best fix belongs in the shared reply dispatch or payload filtering path, not in each plugin's delivery callback. The repair should preserve direct-block/media dedupe and intentional multi-payload replies while eliminating accidental duplicate final delivery.
