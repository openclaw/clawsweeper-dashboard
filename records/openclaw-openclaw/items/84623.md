## Summary

Keep open: current `main` still sends each non-reasoning final payload entry in the `replyResult` array, and the exact shared-dispatcher fix is represented by an open, unmerged PR that uses closing syntax for this issue.

## What This Changes

The issue reports duplicate final reply delivery from non-streaming `dispatchReplyFromConfig` when `disableBlockStreaming: true` is used, causing identical text or media payloads to be delivered twice.

## Best Possible Solution

Keep this issue open while maintainers review https://github.com/openclaw/openclaw/pull/94294, then close it once that dispatcher-level dedupe lands or keep it open if maintainers reject that fix path.

## Reproduction Assessment

Yes at source level: current `main` sends every final payload entry returned in the `replyResult` array, and the issue provides logs showing duplicate identical deliveries. I did not run a live WeChat/openclaw-weixin reproduction in this read-only review.

## Solution Assessment

Yes, likely: the shared dispatcher loop is the narrow owner because plugin-specific dedupe would duplicate policy and `buildReplyPayloads` only handles message-tool/direct-block dedupe. The linked PR should remain the active fix path until maintainers resolve its proof bar.
