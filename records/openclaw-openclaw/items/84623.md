## What This Changes

The issue reports duplicate outbound deliveries from `dispatchReplyFromConfig` when a channel plugin uses non-streaming replies with `disableBlockStreaming: true`.

## Best Possible Solution

Add a focused regression around non-streaming media/text dispatch and fix the core reply payload filtering or dispatcher path so an already-delivered payload is not delivered again, while preserving intentional multi-message replies, unsent media, errors, and fallback notices.

## Reproduction Assessment

Yes from source inspection at medium confidence, not from a live run: duplicate resolver payloads would be sent once per array entry, and non-streaming media blocks can be sent before final payload filtering. The existing direct-block-key tests mean the exact WeChat path still needs a focused regression case.

## Solution Assessment

No, the plugin-side dedupe workaround is not the best fix; the durable fix should live in the shared reply dispatch/payload pipeline so all channel plugins get exactly-once delivery for a single logical reply.
