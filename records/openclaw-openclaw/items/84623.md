## What This Changes

The issue reports duplicated final reply delivery when a plugin uses non-streaming `dispatchReplyFromConfig` with `disableBlockStreaming: true`.

## Best Possible Solution

Add a focused regression around non-streaming media and text dispatch, then fix the shared reply payload or dispatcher path so one logical reply is delivered once while intentional multi-message replies still work.

## Reproduction Assessment

Yes from source inspection at medium confidence: duplicate resolver payload entries would each pass through the final delivery loop. I did not run a live WeChat reproduction, and existing tests only prove the direct-block-key suppression path.

## Solution Assessment

No, the plugin-side dedupe workaround is not the best fix. The durable fix should live in the shared reply dispatch or payload filtering path so all channel plugins get exactly-once delivery for a single logical reply.
