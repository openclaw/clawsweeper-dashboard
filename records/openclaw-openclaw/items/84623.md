## What This Changes

The issue reports that non-streaming channel replies delivered through `dispatchReplyFromConfig` call the channel `deliver` callback twice for the same final text or media payload.

## Best Possible Solution

Add a focused regression for duplicate non-streaming final media/text dispatch, then fix the shared reply payload or dispatcher path so one logical final reply is delivered exactly once while preserving intentional multi-message replies and unsent media.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection gives a medium-confidence path: duplicate final resolver payload entries are delivered once per array entry, and the reporter supplied real WeChat logs matching that symptom.

## Solution Assessment

No, the plugin-side dedupe workaround is not the best fix. The durable solution should live in shared reply dispatch or payload filtering so every channel using the SDK path gets exactly-once delivery for one logical reply.
