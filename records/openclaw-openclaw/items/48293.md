## Summary

Keep open. Current main still does not contain the transcript-tail detector or compaction guard config surface, so the PR is not obsolete, but the draft stacked branch should not merge as-is because the detector has a concrete false negative around `Current time:` lines and still depends on open config scaffolding work.

## What This Changes

The PR branch stacks reserved compaction guard config/schema/docs scaffolding from its dependency with a new pure transcript-tail detector and focused tests for repeated tool failures, duplicate assistant replies, stale system recurrence, and trailing ungrounded user turns.

## Best Possible Solution

Keep this as the PR2 candidate for #48238, but rebase it after or together with #48278, narrow the timestamp exemption to standalone timestamp refreshes or known post-compaction prefixes, update the current config docs location if guard scaffolding remains in the branch, and then rerun the targeted detector/config tests plus the changed gate.

## Reproduction Assessment

Not applicable as a standalone bug report, but yes for the blocking review finding: pass repeated directive/reminder entries that also contain a `Current time:` line, and the PR logic skips them before stale recurrence counting.

## Solution Assessment

No, not as-is. A pure detector is the right boundary for this stage, but the broad `Current time:` exemption is not the narrowest maintainable implementation and should be tightened before runtime consumers depend on the signal.
