- `impact:other`: The issue affected visible macOS Canvas rendering behavior, which does not fit the more specific owned impact classes.

## Summary

Current main and the shipped v2026.5.12+ macOS node runtime cover the reported A2UI push failure by resolving A2UI through the node-scoped Canvas plugin surface URL, refreshing that capability on readiness miss, then applying the pushed messages in the WKWebView. The reporter's v2026.5.7 path used the older `canvasHostUrl` field; the related open hosted-Canvas URL work is separate and tracked at https://github.com/openclaw/openclaw/issues/91083.

## What This Changes

The issue reports that macOS `canvas.a2ui.push` and `canvas.a2ui.pushJSONL` return success while the payload does not render in the WKWebView A2UI host.

## Best Possible Solution

Keep the shipped node-scoped plugin-surface A2UI delivery and capability-refresh path; leave the separate hosted Canvas URL issue to https://github.com/openclaw/openclaw/issues/91083.

## Reproduction Assessment

Yes from source and release comparison, though I did not run a live macOS WKWebView smoke in this read-only review. The reported v2026.5.7 runtime used the legacy `canvasHostUrl` path, while v2026.5.12+ uses the node-scoped plugin surface URL and refresh path before applying A2UI messages.

## Solution Assessment

Yes. The shipped implementation is the right owner-boundary fix because the macOS node session owns and refreshes the scoped Canvas capability URL before the runtime evaluates `openclawA2UI.applyMessages()` in the canvas WebView.
