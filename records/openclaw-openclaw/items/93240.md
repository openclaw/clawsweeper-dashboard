## Summary

Keep this PR open, but the patch is not merge-ready: it addresses a real external-plugin startup regression, yet the new fallback is broader than the reported channel-plugin case and bypasses existing manifest and feature-selection startup predicates.

## What This Changes

The PR adds a gateway startup fallback that loads non-bundled plugins when their effective activation state is explicitly enabled, and updates startup tests to expect those plugins to load.

## Best Possible Solution

Narrow the fix to the reported external channel activation gap, or make install/login write canonical channel activation config, while preserving existing `activation.onStartup=false`, disabled feature, unselected slot, allow, and deny behavior.

## Reproduction Assessment

Yes at source level: current main derives gateway runtime loading from `startup.pluginIds`, and a plugin-entry-only external channel does not match the configured-channel predicate. This review did not run the reported macOS/Homebrew live setup.

## Solution Assessment

No. The patch is a plausible bug-fix direction, but it is too broad because it makes `plugins.entries.<id>.enabled=true` a startup trigger for every non-bundled plugin instead of only fixing the reported external channel activation gap.
