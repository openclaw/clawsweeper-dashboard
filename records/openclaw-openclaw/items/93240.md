## Summary

Keep this PR open, but not merge-ready: it targets a real external-plugin startup regression, yet the new fallback is broader than the channel-plugin case and bypasses existing manifest/feature startup gates; real gateway proof is also missing.

## What This Changes

The PR adds a gateway startup predicate for explicitly enabled non-bundled plugins and updates startup planner tests to expect those plugins to load.

## Best Possible Solution

Keep the fix in gateway startup planning, but narrow it to explicitly enabled installed channel owners needed by the documented Weixin plugin-entry flow while preserving startup-lazy and feature-slot gates for unrelated plugin classes.

## Reproduction Assessment

Yes at source level: the linked issue and Weixin docs show an external channel plugin enabled only through plugins.entries.openclaw-weixin.enabled=true, while current startup planning only loads ids that match narrower startup predicates. I did not run a live packaged gateway reproduction.
