Keep this PR open for review, but it is not merge-ready because the current patch suppresses documented opt-in Telegram compaction notices and uses a broad raw system-prefix filter without real Telegram proof.

## What This Changes

The PR adds Telegram outbound payload normalization that drops `isCompactionNotice` payloads and text starting with the system-message glyph, plus unit coverage for those cases.

## Best Possible Solution

Narrow Telegram suppression to the unwanted internal notice class while preserving documented `notifyUser` compaction notices and intentional user-visible system-marked messages, then add focused tests plus redacted Telegram proof.

## Reproduction Assessment

Yes, by source inspection: current main creates `isCompactionNotice` payloads for the documented `notifyUser` path, and this PR returns `null` for those payloads before Telegram delivery. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Suppressing all compaction notices and all raw system-prefixed text at the Telegram adapter is broader than the documented contract; the safer fix needs a typed or narrower suppression boundary.
