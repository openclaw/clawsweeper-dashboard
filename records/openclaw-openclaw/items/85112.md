## Summary

Keep open: the docs gap is addressed, but the bypass currently treats any runtime binding as a per-thread binding, so a room-level Matrix binding can wrongly bypass `requireMention` for unmentioned thread replies. The PR also still lacks after-fix proof from a patched Matrix runtime.

## What This Changes

This PR adds a default-off Matrix `threadBindings.bypassMentionInBoundThreads` config option, wires it into the Matrix monitor handler, documents it, and adds handler tests for opt-in and default-off behavior.

## Best Possible Solution

Keep the default-off feature, but require the bypass to prove an actual thread-scoped runtime binding, cover room-binding/non-thread cases, and land only after maintainer acceptance plus patched Matrix runtime proof.

## Reproduction Assessment

Yes. Source inspection shows current main drops unmentioned Matrix room messages unless they are authorized control commands, and the PR body includes pre-fix live logs; I did not run a live Matrix server in this read-only review.

## Solution Assessment

No, not as currently written. The default-off config and docs are the right shape, but the implementation must require a thread-scoped binding and still needs after-fix live Matrix proof.

- **[P2] Require a thread-scoped binding before bypassing mentions:** `extensions/matrix/src/matrix/monitor/handler.ts:1041-1045`
