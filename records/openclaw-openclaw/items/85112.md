## Summary

Keep this PR open: it is a useful Matrix opt-in feature with meaningful live proof, but the branch still has a message-delivery bug because a room-level runtime binding can satisfy the bypass when Matrix thread routing is disabled.

## What This Changes

Adds a default-off Matrix `channels.matrix.threadBindings.bypassMentionInBoundThreads` option, wires it into mention gating, documents it, and adds handler tests for opt-in and default-off behavior.

## Best Possible Solution

Keep the default-off option, but require the bypass to prove the route or binding is for the same Matrix thread root before it can skip `requireMention`.

## Reproduction Assessment

Yes, source inspection and the supplied live Matrix logs give a high-confidence path: current main drops unmentioned room messages before routing unless the control-command bypass fires. I did not run a live Matrix server in this read-only review.

## Solution Assessment

No, not yet: the opt-in shape is maintainable, but the implementation needs a stricter thread-scoped discriminator for runtime bindings. The safer fix is to require `thread.threadId` or an equivalent route flag proving the binding belongs to the same thread root.
