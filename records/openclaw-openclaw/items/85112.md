## Summary

Keep this PR open: the opt-in Matrix thread continuation feature is useful and well evidenced, but the current discriminator can bypass `requireMention` for room-bound or flat sessions instead of proving the binding belongs to the same Matrix thread.

## What This Changes

Adds a default-off Matrix `channels.matrix.threadBindings.bypassMentionInBoundThreads` option, wires it into the inbound mention gate, documents it, and adds handler coverage for bypass and default-off behavior.

## Best Possible Solution

Keep the default-off option, but require the bypass to prove `thread.threadId` or an equivalent same-root thread binding/session before it can skip `requireMention`.

## Reproduction Assessment

Yes. Source inspection shows current main drops unmentioned Matrix room replies before routing unless the control-command bypass fires, and the PR supplies live Matrix logs for the requested opt-in behavior.

## Solution Assessment

No, not yet. The default-off config shape is reasonable, but the implementation needs a stricter thread-scoped discriminator before it is the narrow maintainable fix.

Full review comments:

- **[P2] Require a thread route before bypassing mentions:** `extensions/matrix/src/matrix/monitor/handler.ts:1032-1033`
  - body: The new bypass only checks that the event has an `m.thread` relation, then accepts `threadSessionExists || _runtimeBindingId`. When `threadReplies` is `off`, `resolveMatrixThreadRouting` produces no `threadId`, so `_route` and `_runtimeBindingId` can come from the room-level route; this lets a room binding or flat session deliver an unmentioned threaded reply. Gate the bypass on a real thread route or a binding/session proven to belong to the same thread root, and cover the room-level case.
  - confidence: 0.88

  - body: The docs say the bot must have a runtime session binding created automatically by `threadBindings.enabled`, but the current implementation also accepts a session-store timestamp and the PR body notes natural continuation does not create a runtime binding. Update this bullet to describe the actual accepted signals so operators do not configure against the wrong condition.
  - confidence: 0.79
