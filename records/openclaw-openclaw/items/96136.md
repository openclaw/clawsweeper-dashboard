- `merge-risk: 🚨 compatibility`: Merging as-is can make previously valid OpenAI-compatible image generations fail when base64 JSON responses exceed the generic 16 MiB provider JSON cap.

## Summary

Keep open: the unbounded read is real, but the proposed 16 MiB default cap is not compatibility-safe for supported base64 image JSON responses, and the PR only supplies mock/unit evidence rather than real after-fix behavior proof.

## What This Changes

The PR routes OpenAI-compatible image generation/edit success JSON through the shared bounded provider JSON reader and updates the focused tests.

## Best Possible Solution

Use a bounded reader with an image-response-specific byte budget that covers documented count and size behavior, add overflow plus valid-large-payload tests, and provide terminal proof from the real read path.

## Reproduction Assessment

Yes, at source level: current main calls `response.json()` on a provider-controlled success body after the HTTP check, so an oversized or never-ending JSON body is unbounded. I did not run a streaming harness because this review kept the checkout read-only.

## Solution Assessment

No: routing through a bounded helper is the right shape, but the generic 16 MiB JSON cap is too low for supported base64 image responses. The safer fix is an image-specific cap with compatibility coverage for valid large payloads.
