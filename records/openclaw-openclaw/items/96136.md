## Summary

Keep this PR open because the source-level unbounded image success-body read is real, but the proposed generic 16 MiB JSON cap can reject valid OpenAI-compatible base64 image responses and the contributor has only mock/static proof so far.

## What This Changes

The PR routes OpenAI-compatible image generation/edit success JSON through the shared bounded provider JSON reader and updates focused tests.

## Best Possible Solution

Use a bounded reader with an image-response-specific byte budget that covers the supported count and media-size envelope, add valid-large-payload plus overflow coverage, and include real terminal/log proof from the actual provider read path.

## Reproduction Assessment

Yes, at source level: current main calls response.json() on a provider-controlled image success body after the HTTP check. I did not execute a streaming harness because this cleanup review kept the checkout read-only.

## Solution Assessment

No: routing through a bounded reader is the right shape, but using the generic 16 MiB provider JSON cap is too narrow for supported base64 image responses. The safer fix is an image-specific cap with compatibility coverage for valid large payloads.
