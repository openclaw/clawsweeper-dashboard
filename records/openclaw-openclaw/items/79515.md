## Summary

Keep open: this is a fresh v2026.5.7 report with concrete live systemd output, and current main does not show a proven fix for the active user-unit versus `stopped (state failed)` status mismatch.

## What This Changes

The issue asks for `openclaw status` to report a Linux systemd user Gateway service as running when `systemctl --user show` reports the unit active/running and the Gateway probe is reachable.

## Best Possible Solution

Align `openclaw status` service-state resolution with the actual managed systemd user unit and add regression coverage for a reachable active Gateway whose prior failed state should not render as stopped.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The issue includes detailed live v2026.5.7 evidence and current source identifies the status/systemd runtime path, but this review did not establish a live Ubuntu systemd user-unit reproduction.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The narrow maintainable direction is to fix the systemd service-state probe so it queries the same canonical user unit it inspected, with regression coverage, rather than letting gateway reachability alone override service status.
