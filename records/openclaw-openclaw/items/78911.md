## Summary

Keep open: current main still blocks the compaction fields through the agent-facing gateway tool, while `agents.defaults.model.fallbacks` appears already covered by the existing model allowlist. The remaining compaction request changes the model-to-operator config-write boundary and needs maintainer product/security judgment.

## What This Changes

Allow operator-directed agent gateway `config.patch` writes for selected `agents.defaults.compaction` tuning fields and model fallback ordering without direct config-file edits.

## Best Possible Solution

Make an explicit maintainer policy decision: either allowlist only the low-risk compaction leaves with focused guard tests and docs/security notes, or route them through the broader protected-config approval design; handle the `sessions_history` archive fallback as a separate issue.

## Reproduction Assessment

Yes, at source level for the compaction fields: the current agent gateway mutation guard omits `agents.defaults.compaction`, so those changed leaves are rejected before the Gateway RPC write. I did not run a live Gateway in this read-only review.

## Solution Assessment

Unclear until maintainer review: a narrow allowlist addition is mechanically possible, but broadening agent-mediated persistent config writes should be decided against the documented trust-boundary policy. A safer alternative may be the owner-approved protected-config flow tracked at https://github.com/openclaw/openclaw/issues/77886.
