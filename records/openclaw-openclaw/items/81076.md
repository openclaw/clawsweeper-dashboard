Keep open: current main and v2026.6.6 still drop top-level browser `act` fields when a nested `request` is present, and this PR is the active narrow fix path. The patch direction looks correct, but the external PR proof gate is still not satisfied because the PR body gives prose/table claims rather than concrete after-fix terminal output, copied live output, linked artifact, screenshot/recording, or redacted log evidence.

## What This Changes

The PR backfills allowlisted top-level browser `act` fields into nested request payloads when missing, keeps nested fields authoritative, and adds focused regression tests.

## Best Possible Solution

Land this narrow parser-layer compatibility fix, or an equivalent one, after the PR gets concrete after-fix real behavior proof or a maintainer records a proof override.

## Reproduction Assessment

Yes. A source-level reproduction is clear: `action: "act"` with top-level `ref` and `request: { kind: "click" }` returns the nested request before copying `ref`, then `/act` rejects the stripped body.

## Solution Assessment

Yes for the code direction. Backfilling only allowlisted top-level act fields in the browser tool parser is narrower than changing the `/act` service contract, and it preserves nested-field precedence.
