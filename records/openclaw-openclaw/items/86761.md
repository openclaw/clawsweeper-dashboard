## Summary

Keep open: current main and the latest release include adjacent stale CLI-session cleanup and billing cooldown probing, but a Claude live `is_error` result for `out of extra usage` still fails the active turn, only schedules the live process for idle close, and the fresh CLI-session retry gate explicitly excludes billing-shaped failures. The open auth-cooldown work is related, not a merged fix for this live-session retry boundary.

## What This Changes

The issue reports that long-running Claude CLI live sessions can get stuck returning false `out of extra usage` billing errors despite available subscription capacity until the gateway restarts.

## Best Possible Solution

Add a maintainer-approved Claude CLI live-session recovery rule that closes a suspected poisoned live subprocess and proves fresh-process retry or next-turn recovery without weakening true billing backoff.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live high-confidence reproduction was run, but source inspection reproduces the OpenClaw side: a Claude live `is_error` extra-usage result becomes a billing FailoverError, fails the turn, and is excluded from fresh-session retry.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The likely fix belongs at the Claude CLI live-session recovery boundary, but maintainers need to decide how billing-shaped errors should interact with true billing cooldowns and same-auth fallback behavior.
