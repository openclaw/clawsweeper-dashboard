## Summary

Keep open for normal maintainer review. Current main and v2026.6.8 still allow the linked wrong-target Skill Workshop create path, while this PR adds a focused agent-tool guard with targeted tests and terminal proof; I found no blocking code-review findings. The remaining maintainer decision is accepting the intentional fail-closed behavior for agent `action=create` while CLI/Gateway service callers stay unchanged.

## What This Changes

The PR adds a Skill Workshop agent-tool guard plus tests and docs so create calls and create-proposal revisions that reference existing workspace skill paths are rejected before creating a sibling proposal.

## Best Possible Solution

Land the scoped agent-tool guard after maintainer acceptance of the compatibility change and green exact-head checks, or move the policy into the shared service if maintainers want CLI and Gateway proposal surfaces to match.

## Reproduction Assessment

Yes. Source inspection of current main and v2026.6.8 shows `action=create` forwards proposal content to `proposeCreateSkill` without inspecting existing skill path references; I did not run a live MCP reproduction in this read-only review.

## Solution Assessment

Yes, if maintainers want the conservative agent-tool mitigation: it fixes the reported entry point without changing CLI/Gateway service behavior. If maintainers want a uniform public proposal API contract, the better pre-merge shape is to move the guard into the shared service.
