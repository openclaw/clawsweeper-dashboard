## Summary

Keep open: current main still has the Active Memory recall path calling the embedded runner without an explicit lane, while omitted lanes still resolve to the global main lane. The closed focused fix PR is not merged, and the broader open reliability PR does not cover this lane isolation.

## What This Changes

The issue reports that Active Memory's embedded recall subagent can deadlock during before_prompt_build because it runs without an explicit non-main lane.

## Best Possible Solution

Add a stable dedicated Active Memory lane at the plugin-owned embedded recall call site, keep the embedded runner default unchanged, and cover it with a focused regression test.

## Reproduction Assessment

Yes, by source inspection: before_prompt_build awaits Active Memory recall, the recall call omits lane, the embedded runner defaults omitted lanes to main, and command lanes are serial by default. I did not run the live Telegram/macOS canary in this read-only review.

## Solution Assessment

Yes. Passing a dedicated non-main lane at the Active Memory recall call site is the narrowest maintainable fix; changing runEmbeddedPiAgent defaults globally would alter other callers, and the broader reliability PR only bounds latency rather than removing this self-deadlock.
