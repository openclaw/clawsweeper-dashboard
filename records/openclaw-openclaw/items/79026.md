## Summary

Keep open: current main and the latest release still let Active Memory recall call `runEmbeddedPiAgent` without an explicit lane while the embedded runner maps missing lanes to `main`, so the reported self-deadlock remains source-reproducible. The attempted direct fix in https://github.com/openclaw/openclaw/pull/79098 was closed unmerged, and the broader draft reliability PR does not add the lane isolation this issue asks for.

## What This Changes

The issue requests preventing Active Memory recall from enqueueing its embedded subagent on the global main lane during `before_prompt_build`.

## Best Possible Solution

Add a stable plugin-owned `active-memory` lane to the embedded recall call with a focused regression assertion, while leaving broader nonblocking/QMD startup policy to the existing reliability track.

## Reproduction Assessment

Yes, by source inspection: an eligible `before_prompt_build` hook awaits Active Memory recall, that recall still omits `lane`, and the embedded runner maps omitted lanes to `main`. I did not run the live Telegram/macOS canary in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to set a stable non-main lane at the Active Memory recall call site. Changing `runEmbeddedPiAgent` defaults globally would alter other callers and plugin contracts, while latency-budget changes alone would only reduce timeout impact.
