## Summary

Keep open: current main still lets Active Memory recall enqueue its embedded run without an explicit lane, while the embedded runner maps omitted lanes to the global main lane, so the reported self-block remains source-reproducible.

## What This Changes

The issue asks for Active Memory recall runs spawned from `before_prompt_build` to avoid reusing the occupied main command lane.

## Best Possible Solution

Add a stable dedicated Active Memory lane at the embedded recall call site, cover it with a regression test, and leave broader nonblocking/default-latency policy to the existing reliability track.

## Reproduction Assessment

Yes, by source inspection: an eligible `before_prompt_build` hook awaits `maybeResolveActiveRecall`, that recall calls `runEmbeddedPiAgent` without `lane`, and `resolveGlobalLane(undefined)` maps the nested run to `main`. I did not run the reporter's macOS/Telegram canary in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to set a stable non-main lane at the Active Memory recall call site and test it. Changing the embedded-runner default globally would affect other callers, and the open latency PR does not remove this lane self-block.
