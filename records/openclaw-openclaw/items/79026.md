## Summary

Keep open: current main still awaits Active Memory recall from `before_prompt_build` and calls `runEmbeddedPiAgent` without a lane, while omitted lanes resolve to `main`, matching the reported self-block path.

## What This Changes

Active Memory recall should run its embedded subagent on a non-main lane when invoked during prompt building so it cannot queue behind the user-visible turn waiting for it.

## Best Possible Solution

Add a stable dedicated Active Memory lane at the embedded recall call site, assert it in Active Memory hook coverage, and keep broader nonblocking/QMD reliability policy in the existing reliability track.

## Reproduction Assessment

Yes, by source inspection: an eligible `before_prompt_build` hook awaits Active Memory recall, that recall calls `runEmbeddedPiAgent` without `lane`, and the embedded runner maps omitted lanes to `main`. I did not run the live Telegram/macOS canary in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable solution is to set a stable non-main lane at the Active Memory recall call site. Changing the global embedded-runner default would affect other callers, and https://github.com/openclaw/openclaw/pull/73667 reduces timeout exposure without removing this lane self-block.
