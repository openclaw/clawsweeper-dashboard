## Summary

Keep open: current main still lets Active Memory recall enter `runEmbeddedPiAgent` without an explicit lane from an awaited `before_prompt_build` hook, so the reported nested-main-lane stall remains source-reproducible.

## What This Changes

Active Memory recall should avoid queuing its embedded memory subagent on the same global main lane as the user-visible turn that is waiting for it.

## Best Possible Solution

Set a stable non-main lane at the Active Memory recall call site, add regression coverage that asserts the lane on the embedded run, and leave broader fail-open/QMD policy to the existing reliability track.

## Reproduction Assessment

Yes, by source inspection: an eligible `before_prompt_build` hook awaits Active Memory recall, that recall calls `runEmbeddedPiAgent` without `lane`, and the embedded runner maps omitted lanes to `main`. I did not run the live Telegram/macOS canary in this read-only review.

## Solution Assessment

Yes, a call-site lane assignment is the narrowest maintainable fix for this specific self-block. Changing `runEmbeddedPiAgent` defaults globally would affect other callers and the open latency PR only shortens the timeout path rather than isolating the nested work.

In `extensions/active-memory/index.ts`, make the embedded recall call use a stable non-main lane, such as `lane: "active-memory"`, so a recall spawned from `before_prompt_build` cannot queue behind the user-visible turn waiting for it.
