## Summary

Keep open: current main still routes Active Memory recall through `runEmbeddedPiAgent` without an explicit lane, while omitted lanes still resolve to `main`. The issue is already paired with the open closing PR at https://github.com/openclaw/openclaw/pull/80255, so this issue should stay open until that PR is reviewed and landed or rejected.

## What This Changes

Active Memory recall should use a non-main embedded-run lane during `before_prompt_build` so it cannot self-block behind the user-visible main lane.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80255 if it remains correct, adding the dedicated Active Memory recall lane and regression test while leaving broader nonblocking/QMD reliability work to the separate reliability track.

## Reproduction Assessment

Yes. By source inspection, an eligible `before_prompt_build` hook awaits Active Memory recall, the recall call omits `lane`, and the embedded runner maps an omitted lane to `main`; I did not run the live Telegram/macOS canary in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to set a stable non-main lane at the Active Memory recall call site and cover it with a focused test; changing the embedded-runner default globally would affect other callers and plugin contracts.
