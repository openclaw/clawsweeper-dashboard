## Summary

Keep open: current main still routes active-memory recall through runEmbeddedPiAgent without a dedicated lane while the prompt-build hook awaits that recall, and an open closing PR is already paired with this issue. The source path still matches the reporter's deadlock analysis and the newer independent repro comment strengthens the report.

## What This Changes

The issue asks for active-memory recall to avoid enqueueing its embedded subagent on the global main lane from inside before_prompt_build.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80255 or a narrow equivalent that gives active-memory recall a stable non-main lane and keeps the regression assertion in the plugin test harness.

## Reproduction Assessment

Yes, by source inspection: an eligible before_prompt_build hook awaits active-memory recall, that recall still calls runEmbeddedPiAgent without lane, and the embedded runner maps omitted lanes to main. I did not run the live Telegram/macOS canary in this read-only review.

## Solution Assessment

Yes, the dedicated lane at the active-memory call site is the narrowest maintainable solution. Changing the embedded-runner default globally would affect other callers, while the broader latency PR only reduces timeout cost and does not remove this self-block.
