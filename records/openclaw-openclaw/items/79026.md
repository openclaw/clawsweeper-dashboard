## Summary

Keep open: current main still lets active-memory recall call the embedded runner without a lane, while the embedded runner still maps an omitted lane to `main`. The closed unmerged fix PR showed the narrow lane assignment and real after-fix logs, but that change is not on main or in the latest release.

## What This Changes

Request that active-memory embedded recall use a non-main lane so `before_prompt_build` cannot wait on a subagent queued behind the parent turn.

## Best Possible Solution

Add a stable dedicated active-memory lane at the plugin-owned embedded recall call site, cover it with a regression assertion, and keep broader active-memory timeout policy in the separate reliability track.

## Reproduction Assessment

Yes, by source inspection: an eligible `before_prompt_build` hook awaits active-memory recall, the recall call omits `lane`, and the embedded runner maps an omitted lane to `main`. I did not run the live Telegram canary in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to pass a dedicated non-main lane from active-memory recall. Changing the global embedded-runner default would affect unrelated callers, and the broader reliability PR only reduces latency cost rather than removing this lane self-block.
