## Summary

Keep open: current main still has the source-level active-memory self-deadlock path, and an open closing PR already targets the narrow dedicated-lane fix. The issue should remain open until that PR lands or is replaced.

## What This Changes

The issue asks for active-memory embedded recall to use a non-main lane so a before_prompt_build hook cannot wait on work queued behind its own main-lane turn.

## Best Possible Solution

Merge or replace the dedicated-lane active-memory recall fix with regression coverage, and keep broader fail-open or latency policy work in the active-memory reliability follow-ups such as https://github.com/openclaw/openclaw/issues/72015.

## Reproduction Assessment

Yes, at source level: an eligible before_prompt_build hook awaits active-memory recall, the recall call omits `lane`, omitted embedded lanes resolve to `main`, and command lanes run one active task by default. I did not live-run the Telegram canary in this read-only review.

## Solution Assessment

Yes. Passing a plugin-owned `active-memory` lane at the recall call site is the narrowest maintainable fix; changing embedded-runner defaults globally would affect unrelated callers and plugin contracts.
