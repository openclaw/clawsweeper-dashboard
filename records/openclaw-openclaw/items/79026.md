## Summary

Keep open: current main still has the active-memory recall path calling the embedded runner without a lane, and an open implementation PR already references this issue with closing syntax, so the issue should stay open until that PR is reviewed and merged or closed.

## What This Changes

The issue reports that active-memory recall can self-deadlock because its before_prompt_build hook awaits an embedded subagent run that defaults to the main lane.

## Best Possible Solution

Land the dedicated active-memory recall lane assignment with regression coverage in the linked PR, while leaving broader active-memory fail-open and QMD startup reliability work to the separate reliability track.

## Reproduction Assessment

Yes, by source inspection: before_prompt_build awaits active-memory recall, runRecallSubagent omits lane, the embedded runner maps omitted lanes to main, and command lanes are single-concurrency. I did not run the live Telegram/macOS canary in this read-only review.

## Solution Assessment

Yes. Passing a stable non-main lane at the active-memory recall call site is the narrowest maintainable fix; changing runEmbeddedPiAgent defaults globally would alter unrelated embedded-runner callers and plugin contracts.
