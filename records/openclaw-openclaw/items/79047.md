## Summary

Keep open. Current main has related transcript mirroring, CLI resume validation, and fresh-session reseed machinery, but it still does not define the requested opt-in handoff for user-initiated runtime switches, especially when an existing CLI session binding is reused instead of starting a fresh CLI session.

## What This Changes

Request an opt-in continuity bridge so the same agent can carry bounded conversation context when switching between runtime backends such as claude-cli and the embedded Pi runtime.

## Best Possible Solution

Define a maintained one-shot runtime handoff contract that stages a bounded summary or recent-turn excerpt on explicit model/runtime switch, consumes it once in the target runtime, and documents token/privacy behavior.

## Reproduction Assessment

Yes. The current-main source path shows session model patches do not stage handoff context, CLI bindings are provider-keyed, and existing CLI resumes ignore the OpenClaw history prompt; I did not run a live claude-cli/Pi setup in this read-only review.

## Solution Assessment

Unclear because there is no patch. The requested opt-in direction is plausible, but the maintainable solution should be a bounded one-shot handoff surface rather than mutating provider-owned session stores directly.
