## Summary

Keep open. Current main still lacks a WebSocket protocol-level ping path for gateway clients, while this PR is a narrow implementation of that behavior; it should not be closed as already implemented. The patch looks functionally sound, but it still needs the required user-facing changelog entry and maintainer choice against the overlapping open #74783 ping/telemetry design.

## What This Changes

This PR adds a 25-second per-connection `socket.ping()` timer in the gateway WebSocket connection handler and clears that timer during connection teardown.

## Best Possible Solution

Keep exactly one gateway-owned WebSocket protocol ping path. Either merge this focused keepalive after the changelog repair, or land #74783 first and treat its shared maintenance-loop ping/pong implementation as canonical while adapting or closing this PR to avoid duplicate timers.

## Reproduction Assessment

Yes. The linked report gives concrete WebChat long-tool-call steps and code 1006 disconnect logs, and current-main source verification shows only application-level `tick` events with no WebSocket protocol ping owner. The actual intermediary timeout still needs deployment-level validation.

## Solution Assessment

Unclear. This PR is the narrowest maintainable fix for idle drops and matches the `ws` dependency contract, but #74783 is likely better if maintainers want RTT telemetry and a single shared maintenance-loop ping owner.
