## Summary

Keep open: the PR addresses a real CLI mismatch, but the branch currently misclassifies explicit empty aliases as built-in aliases because it uses a falsy alias check instead of the defaults contract.

## What This Changes

The PR exports the default model alias table, adds a clearer `models aliases remove` error for materialized built-in aliases, and adds focused alias command tests.

## Best Possible Solution

Keep the focused error-message fix, but make the built-in check match `applyModelDefaults` exactly by treating only `alias === undefined` as materialized and add empty-alias regression coverage.

## Reproduction Assessment

Yes. Source inspection shows `list` reads resolved config while `remove` mutates source config, so a configured built-in target with no source alias can be listed and then rejected as not found on current main.

## Solution Assessment

No, not yet. The fix direction is narrow and maintainable, but the PR should use the same `alias !== undefined` contract as `applyModelDefaults` instead of a falsy alias check.
