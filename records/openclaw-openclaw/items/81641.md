## Summary

Keep open: the PR addresses a real CLI mismatch and includes sufficient real terminal proof, but the proposed guard still misclassifies explicit empty aliases as built-in aliases because it checks falsiness instead of the existing `alias === undefined` materialization contract.

## What This Changes

The PR exports the default model alias table, adds built-in-alias-specific error handling to `models aliases remove`, and adds focused alias command tests.

## Best Possible Solution

Keep the focused error-message fix, but make built-in detection match `applyModelDefaults` exactly by requiring `alias === undefined` and add empty-alias regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main lists resolved, default-materialized aliases while removal mutates only source config, so a configured built-in target without a source alias is listed but rejected as not found.

## Solution Assessment

No, not yet. The fix direction is narrow and maintainable, but the PR should use the same `alias !== undefined` contract as `applyModelDefaults` instead of a falsy alias check.
