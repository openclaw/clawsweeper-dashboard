## Summary

Keep open: the reported CLI bug is real and the proof is sufficient, but the proposed built-in detection still diverges from the default alias contract, so it can return the wrong error for explicit empty aliases and normalized Gemini model keys.

## What This Changes

The branch exports `DEFAULT_MODEL_ALIASES`, uses it in `models aliases remove` to return a built-in-alias error for materialized aliases, and adds focused alias command tests.

## Best Possible Solution

Land a focused CLI error fix that reuses the same default materialization predicate and model-key normalization as `applyModelDefaults`, with regression coverage for empty aliases and normalized Gemini keys.

## Reproduction Assessment

Yes. Source inspection shows a configured built-in target with no source alias is materialized in the resolved config used by `list`, while `remove` searches only source aliases and falls through to the generic not-found path.

## Solution Assessment

No, not yet. The fix direction is narrow and maintainable, but the PR should match `applyModelDefaults` exactly instead of approximating it with a raw key lookup and falsy alias check.
