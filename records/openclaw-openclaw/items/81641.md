## Summary

Keep open: the PR addresses the reported misleading built-in-alias error path, but it introduces a regression for the existing `alias: ""` disable contract and needs a small condition/test adjustment before merge.

## What This Changes

The PR exports the default model alias table, adds built-in-alias handling to `models aliases remove`, and adds focused alias command tests.

## Best Possible Solution

Keep the focused error-message fix, but make the built-in check match `applyModelDefaults` exactly by treating only `alias === undefined` as materialized and add empty-alias regression coverage.

## Reproduction Assessment

Yes. Source inspection shows `list` reads resolved config while `remove` mutates source config, and a configured built-in target with no source alias is therefore listed but rejected as not found on current main.

## Solution Assessment

No, not yet. The fix direction is narrow and maintainable, but the PR should use the same `alias !== undefined` contract as `applyModelDefaults` instead of a falsy alias check.
