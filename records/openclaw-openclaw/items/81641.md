## Summary

Keep open: the PR fixes the direct built-in-alias case and has solid terminal proof, but it still checks raw source model keys while current main normalizes retired Gemini keys before materializing default aliases, leaving one listed-alias path with the old misleading `Alias not found` error.

## What This Changes

The PR exports `DEFAULT_MODEL_ALIASES`, adds a built-in-alias-specific error path to `models aliases remove`, and adds focused alias command tests.

## Best Possible Solution

Normalize the source model map, or reuse the same model-map normalization as defaults, before the built-in-alias check and cover the retired Gemini key case in the alias command tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a source config with `agents.defaults.models["google/gemini-3-pro-preview"]` normalizes to the default alias target before `list` output, while the PR checks raw `nextModels` in `remove`.

## Solution Assessment

No, not yet. The chosen error-path approach is appropriate, but it must share the default model-key normalization before it fully aligns `list` and `remove` behavior.
