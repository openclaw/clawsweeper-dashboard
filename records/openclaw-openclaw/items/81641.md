## Summary

Keep open: the PR fixes a real CLI error-message mismatch, but the proposed guard is too broad and misclassifies explicitly disabled built-in aliases as visible built-ins.

## What This Changes

The branch exports the default model alias table, adds a built-in-alias error path to `models aliases remove`, and adds aliases command tests.

## Best Possible Solution

Keep the clearer built-in error, but make the condition match `applyModelDefaults` exactly and add coverage for explicit empty-alias disables.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a source config with a configured built-in target and no explicit alias is shown by `list` after defaults but rejected by `remove` from source config. I did not run the CLI in this read-only review.

## Solution Assessment

No, the direction is right but the implementation is not yet the best fix because it uses alias truthiness instead of the established `alias === undefined` contract. The narrow fix is to align the guard with `applyModelDefaults` and test `alias: ""`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Match the built-in check to undefined aliases:** `src/commands/models/aliases.ts:105`

Check for `alias === undefined` and add that case to the new tests.
