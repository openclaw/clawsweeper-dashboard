## Summary

Keep open: current main still has a source-visible path where doctor emits the generic no-active-memory-plugin warning before considering a healthy gateway memory probe, matching the reported false-positive shape.

## What This Changes

The issue asks doctor to recognize the configured loaded memory runtime, or provide slot/plugin-specific remediation instead of warning that no active memory plugin exists.

## Best Possible Solution

Make doctor treat a healthy gateway memory probe or `memory status`-equivalent runtime evidence as authoritative enough to avoid the generic warning, and make the fallback remediation name the selected memory slot/plugin state.

Do we have a high-confidence way to reproduce the issue? No live reproduction; source inspection gives a focused unit path by making `resolveActiveMemoryBackendConfig` return null while `gatewayMemoryProbe` is checked and ready, which currently still emits the reported warning.
