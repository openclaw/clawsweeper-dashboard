## Summary

Close as implemented on current main: doctor now resolves the configured memory slot runtime before deciding no active memory plugin exists, so a valid `memory-core` slot should no longer produce the reported warning. The fix is on `main` only and is not contained in `v2026.5.2`.

## What This Changes

The issue reports that `openclaw doctor --non-interactive` warns about no active memory plugin even though bundled `memory-core` and `active-memory` are enabled and memory status is healthy.

## Best Possible Solution

Keep the current slot-based runtime resolution on `main` and let the next release carry it; users on `v2026.5.2` need a build containing `1bc97aee4ecbef0c6560099ff9afa93afde6cdf0`.

## Reproduction Assessment

No current-main failing reproduction is needed: source inspection shows `memory-core` slot configs now load the memory runtime before the doctor warning path. The reported `v2026.5.2` symptom matches the fixed doctor/runtime boundary.

## Solution Assessment

Yes. The implemented direction is narrow and maintainable because it reuses the configured memory-slot runtime loader instead of adding a hard-coded memory plugin special case or broad plugin loading.
