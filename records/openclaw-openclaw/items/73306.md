## Summary

Keep open. Current main still shows the Active Memory timeout-budget mismatch for the original report, and the issue has an open closing PR plus newer v2026.4.29 reports that should be validated instead of swept away.

## What This Changes

The issue reports provider-agnostic Active Memory runs timing out with zero summary output and blocking Telegram/chat replies after upgrading to v2026.4.25 through v2026.4.29.

## Best Possible Solution

Land #74480 or an equivalent narrow fix that computes one effective Active Memory recall timeout and passes it consistently through the hook, outer watchdog, and embedded runner. Then re-test the v2026.4.29 reports and split any remaining hook-layer/setup-overhead behavior into narrower follow-ups such as #75412, #65309, or #66804.

## Reproduction Assessment

Yes. The issue has multiple independent log-based reproductions, and current-main source gives a focused reproduction path: run Active Memory where setup fits inside the setup grace but exceeds the raw configured timeout, then observe that the embedded runner still receives only the raw timeout.

## Solution Assessment

Yes for the original embedded-recall timeout mismatch: fixing the timeout handoff is the narrowest maintainable solution. It is unclear whether that also solves the newer 32-55s hook/prep reports, so those need post-merge validation or separate tracking.
