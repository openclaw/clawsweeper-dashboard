## Summary

Keep open. Current main still shows the original Active Memory timeout handoff mismatch: the hook and outer watchdog use the configured timeout plus setup grace, but the embedded recall runner still receives only the raw configured timeout. The issue also has an open closing PR, so the correct maintainer path is to review/land that PR and then recheck the newer hook-layer/prep-overhead reports.

## What This Changes

The issue reports provider-agnostic Active Memory timeout and empty-summary failures after upgrading to v2026.4.25/v2026.4.26, with later comments adding hook-layer timeout, memory-runtime, and setup-overhead evidence on v2026.4.29.

Land #74480 or an equivalent narrow fix that computes one effective Active Memory recall timeout and passes it consistently through the hook, outer watchdog, and embedded runner.
