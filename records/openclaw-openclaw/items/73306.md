## Summary

Keep open. Current main still passes Active Memory's raw configured timeout into the embedded recall runner while the outer watchdog and hook use a setup-grace-extended budget, and an open closing PR already targets that mismatch. Newer v2026.4.29 comments also add hook-layer and prep-overhead symptoms that should be rechecked after the narrow timeout-handoff fix lands.

## What This Changes

The issue asks OpenClaw to fix Active Memory runs that time out with empty output and block chat replies after recent releases, especially around the 15s recall timeout path.

## Best Possible Solution

Land #74480 or an equivalent narrow fix that computes one effective Active Memory recall timeout and passes it consistently through the hook, outer watchdog, and embedded runner; then re-test the OpenRouter, MiniMax, Xiaomi, and Ollama reports and split any remaining hook-layer or prep-overhead failure into a narrower memory/runtime issue.

## Reproduction Assessment

Yes. The discussion has multiple independent log-based reproductions, and current-main source gives a focused reproduction path: trigger Active Memory with setup work that exceeds the raw timeout but fits inside setup grace, then observe that the embedded runner still receives only the raw timeout.

## Solution Assessment

Yes for the original embedded-recall timeout mismatch: a single effective timeout handoff is the narrowest maintainable fix. It is unclear whether that fully solves the newer hook-layer/prep-overhead reports, so those should be validated after the handoff fix lands.
