## Summary

Keep open. Current main still shows the Active Memory timeout-budget mismatch from the original report: the hook and outer watchdog use the configured timeout plus setup grace, while the embedded recall runner still receives only the raw configured timeout. The issue also has an open closing PR, so it should remain open until that PR lands and the newer v2026.4.29 hook/prep-overhead reports are rechecked.

## What This Changes

The issue reports Active Memory timing out on eligible Telegram/direct-chat runs with empty summaries and blocking replies after upgrading to v2026.4.26 and later v2026.4.29.

## Best Possible Solution

Land #74480 or an equivalent narrow change that computes one effective Active Memory recall timeout and passes it consistently through the hook, outer watchdog, and embedded runner. After merge, retest the OpenRouter, MiniMax, Xiaomi, Ollama, and Telegram reports and keep any remaining fixed setup-overhead work in a narrower follow-up such as #75412.

## Reproduction Assessment

Yes. The issue has multiple independent log-based reproductions, and current-main source gives a focused seam reproduction: simulate setup work that fits inside setup grace but exceeds the raw configured timeout, then observe that the embedded runner still receives only the raw timeout.

## Solution Assessment

Yes for the original embedded-recall timeout mismatch: fixing the timeout handoff is the narrowest maintainable solution. It is unclear whether that fully solves the newer hook-layer/prep-overhead reports, so those should be validated after #74480 or an equivalent fix lands.
