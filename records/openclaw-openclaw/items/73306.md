## Summary

Keep open. Current main still has the Active Memory timeout-budget mismatch: the hook and outer watchdog use the configured timeout plus setup grace, while the embedded recall runner still receives only the raw configured timeout. The issue also has an open closing PR for that narrow fix, and newer comments add v2026.4.29 hook/setup-overhead symptoms that should be validated after that PR lands rather than swept closed now.

## What This Changes

The issue reports Active Memory timing out with empty recall output across Telegram and multiple providers, initially around the embedded recall timeout and later at the plugin hook/lane layer.

## Best Possible Solution

Keep this issue open until #74480 or an equivalent fix lands, then validate Active Memory against the OpenRouter, MiniMax, Xiaomi, Ollama, and Telegram reports. The desired end state is one consistent effective timeout budget through the hook, watchdog, embedded runner, and lane timeout, with any remaining fixed setup-overhead or memory-query slowness split into a narrower follow-up.

## Reproduction Assessment

Yes. The issue has multiple independent log-based reproductions, and current main gives a focused source-level reproduction: configure setup work that fits inside setup grace but exceeds raw `timeoutMs`, then observe the embedded runner still receives only the raw timeout while the hook waits longer.

Unclear for the whole issue. #74480 is the narrowest maintainable fix for the original embedded timeout handoff, but it does not by itself prove the newer hook-layer/prep-overhead reports are fixed, so the issue should stay open through PR review and post-merge validation.
