## Summary

Keep open. Current main still lacks the configurable Control UI title path, and the closed feature issue points to this PR as the canonical implementation, but the branch is stale/conflicting and needs focused fixes before it is mergeable.

## What This Changes

The PR adds a `gateway.controlUi.title` config option, carries it through the Control UI bootstrap response, applies it to `document.title`, and adds bootstrap-loader tests.

## Best Possible Solution

Expose a documented config key through the authenticated bootstrap payload and update the browser title whenever bootstrap runtime settings refresh, while preserving the default title when the key is unset.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives a high-confidence current-main check that the config type/schema, bootstrap payload, docs, and UI loader still have no `gateway.controlUi.title` path.

## Solution Assessment

No as submitted. The implementation direction is narrow, but it needs current-main rebase/adaptation, runtime-setting placement, updated fixtures, and public config metadata/docs before merge.

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply the title during runtime bootstrap refreshes:** `ui/src/ui/controllers/control-ui-bootstrap.ts:43-44`
  - body: Current main refreshes bootstrap runtime settings after auth with `{ applyIdentity: false }`. Placing `document.title` inside the identity update path means that refresh can receive the configured title without applying it; move the title assignment with the other runtime settings.
  - confidence: 0.88

  - confidence: 0.9

  - body: Adding a public `gateway.controlUi.title` key only to the raw Zod shape leaves config help, labels, and the configuration reference without the setting, so operators cannot discover it through the normal config/schema surfaces.
  - confidence: 0.84
