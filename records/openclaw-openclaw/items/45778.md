- `impact:security`: Both proposed paths expand background authority: either unattended command execution or persisted chat route metadata for later sends.

## Summary

Keep open: current main still lacks the requested platform seam for skills to do zero-LLM background chat delivery. The direct-exec cron half is already covered by the open canonical tracker, but the foreground reply-route env option remains a distinct product and security decision for this SignalRadar-style workflow.

## What This Changes

Request adding either zero-LLM direct cron command execution with channel delivery or foreground reply-route environment variables for skills so system crontab jobs can send chat alerts.

## Best Possible Solution

Keep the direct-exec work aligned with https://github.com/openclaw/openclaw/issues/18160, and decide whether OpenClaw also wants a scoped, authorized reply-route env contract for skills that need system-crontab delivery.

## Reproduction Assessment

Yes for source-level capability absence, not as a strict bug reproduction. Current main still has no direct cron exec payload and no OPENCLAW_REPLY_CHANNEL/OPENCLAW_REPLY_TARGET env contract; I did not run a live SignalRadar or system crontab scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product and security contract. Native direct-exec cron belongs with the open canonical tracker, while reply-route env injection may be smaller but needs explicit authorization, route scoping, and stale-target behavior.
