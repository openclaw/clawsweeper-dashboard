## Summary

Keep open: current main still lacks both requested platform seams, and the SignalRadar/ClawHub context shows this is an active feature/API gap that needs product and security design rather than cleanup closure.

## What This Changes

The issue requests a zero-LLM way for skills to send background notifications, either through a direct cron execution payload with delivery or scoped reply-route environment injection.

## Best Possible Solution

Define one narrow authorized background-delivery contract, then implement either scoped reply-route injection or an opt-in direct command cron payload with documented security controls and regression coverage.

## Reproduction Assessment

No, not as a strict bug reproduction; this is a feature/API gap. Source inspection gives high confidence that current main still only supports systemEvent or LLM-backed agentTurn cron jobs and does not expose the requested reply-route env vars.

## Solution Assessment

Unclear until maintainers choose the product and security contract. Reply-route env injection is the smaller delivery seam, while directExec needs explicit operator opt-in and execution policy before it is safe to add.

## Security Review

Status: needs_attention

Summary: The requested capability is security-sensitive because it would expand background messaging authority or add direct command execution outside an active user turn.

Concerns:

- **[medium] Define controls before direct background execution:**
  - body: A direct cron execution payload would run skill-provided commands as the gateway process user, so implementation needs opt-in, command/source restrictions, timeouts, output limits, and audit logging.
  - confidence: 0.9
- **[medium] Scope persistent reply-route metadata:**
  - body: Foreground reply-route env injection would let skills persist chat delivery coordinates for later system crontab sends, so the contract needs user authorization, account/thread scoping, stale-target handling, and limits on route reuse.
  - confidence: 0.86
