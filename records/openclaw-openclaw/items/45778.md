## Summary

Keep open: current main and v2026.5.22 still lack the requested zero-LLM skill notification path. The direct-exec cron half is already covered by the open canonical tracker https://github.com/openclaw/openclaw/issues/18160, but the foreground reply-route env injection option is distinct and needs product/security review.

## What This Changes

The issue asks for skills to send background notifications without LLM turns, either through a direct cron command payload with delivery or foreground reply-route env vars that system crontab jobs can reuse.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/18160 as the canonical direct-exec cron tracker, and separately decide whether OpenClaw wants a scoped, user-authorized reply-route env contract for foreground skill runs.

## Reproduction Assessment

Yes for source-level capability absence, not as a strict runtime bug reproduction. Current main has no direct cron exec payload and no OPENCLAW_REPLY_CHANNEL/OPENCLAW_REPLY_TARGET env contract; I did not run a live SignalRadar or system crontab scenario.

## Solution Assessment

Unclear until maintainers choose the product and security contract. A native direct-exec cron payload belongs with the open canonical tracker, while reply-route env injection is smaller but needs explicit authorization, scoping, route lifetime, and stale-target behavior.

## Security Review

Status: needs_attention

Summary: The requested fixes are security-sensitive because they either schedule unattended host commands or expose persistent chat delivery coordinates to skill code.

Concerns:

- **[medium] Define unattended cron exec controls:** `src/security/dangerous-tools.ts:9`
  - body: A direct cron execution payload would run skill- or user-provided commands outside an active agent turn, so it needs explicit opt-in, sandbox/approval behavior, cwd/env binding, timeout/cancellation, output limits, and audit logging.
  - confidence: 0.92
- **[medium] Scope persistent reply-route metadata:** `src/channels/route-projection.ts:122`
  - body: Foreground reply-route env injection would let skills persist channel/account/thread targets for later system-cron sends, so the contract needs authorization, route lifetime, stale-target handling, and limits on reuse.
  - confidence: 0.88
- **[medium] Preserve exec approval trust boundaries:** `docs/tools/exec-approvals.md:55`
  - body: Exec approval docs state approvals reduce accidental execution risk but are not a per-user auth boundary, so cron exec must not bypass that host trust model.
  - confidence: 0.86
- **[medium] Keep cron validation under secops review:** `.github/CODEOWNERS:44`
  - body: CODEOWNERS already routes cron job validation to the security owner group, and both proposed solutions change that boundary.
