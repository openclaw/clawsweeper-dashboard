## Summary

Keep open. Current main still lacks both requested platform paths: cron has no zero-LLM direct command payload, and skill environment injection does not expose a foreground reply route that a later system crontab job can reuse. The reporter reconfirmed on 2026-04-29 that the SignalRadar ClawHub skill still relies on webhook delivery as the workaround, so this needs maintainer product/security scoping rather than stale cleanup.

## What This Changes

Request to let skills perform zero-LLM background monitoring while still sending notifications, either through a directExec cron payload or scoped reply-route environment variables.

## Best Possible Solution

Expose one explicit zero-LLM notification contract that preserves channel permissions: either scoped foreground reply-route injection for user-authorized skill runs, or an opt-in directExec cron payload restricted to approved installed-skill commands and wired through existing cron delivery, logging, timeout, and failure-notification semantics.

## Reproduction Assessment

Yes. Static inspection of current main is enough: cron types/schema/validation/execution accept only systemEvent or agentTurn, and the skill env path injects configured skill env/apiKey values rather than OPENCLAW_REPLY_* route variables.

## Solution Assessment

Unclear until maintainers choose the contract. Option A is the broader scheduler fix but needs host-exec controls; Option B is smaller but still needs permission, account/thread scoping, persistence, and revocation semantics.
