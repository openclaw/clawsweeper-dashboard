## Summary

Keep open: current main still lacks a zero-LLM chat-delivery seam for system crontab skills. The direct-exec cron half is covered by https://github.com/openclaw/openclaw/issues/18160, but the scoped foreground reply-route env option remains a distinct product and security decision for this SignalRadar-style workflow.

## What This Changes

Request a platform-supported way for skills like SignalRadar to deliver background messages without an LLM turn, via direct cron execution or scoped reply-route env injection.

## Best Possible Solution

Keep native cron direct-exec design aligned with https://github.com/openclaw/openclaw/issues/18160, and separately decide whether OpenClaw wants a scoped, authorized reply-route env contract for skills that use system crontab delivery.

## Reproduction Assessment

Yes for source-level capability absence, not as a strict bug reproduction. Current main has no direct cron exec payload and no OPENCLAW_REPLY_CHANNEL/OPENCLAW_REPLY_TARGET env contract; I did not run a live SignalRadar or system crontab scenario.

## Solution Assessment

Unclear until maintainers choose the product and security contract. A native direct-exec cron payload belongs with the open canonical tracker, while reply-route env injection is smaller but needs explicit authorization, scoping, and stale-target behavior.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because it would either schedule unattended host command execution or expose reusable chat delivery coordinates to skill code.

Concerns:

- **[medium] Define unattended cron exec controls:** `src/security/dangerous-tools.ts:9`
  - body: A direct cron execution payload would run skill- or user-provided commands outside an active agent turn, so it needs explicit opt-in, sandbox/approval behavior, cwd/env binding, timeout/cancellation, output limits, and audit logging.
  - confidence: 0.92
- **[medium] Scope persistent reply-route metadata:**
  - body: Foreground reply-route env injection would let skills persist channel/account/thread targets for later system crontab sends, so the contract needs authorization, route lifetime, stale-target handling, and limits on reuse.
  - confidence: 0.88
- **[medium] Preserve host env restrictions:** `docs/tools/exec.md:82`
  - body: Any route or command env feature must preserve existing host execution env filtering so route metadata does not become a vehicle for unsafe PATH, loader, proxy, or TLS override behavior.
  - confidence: 0.82
- **[medium] Keep cron validation under security review:** `.github/CODEOWNERS:38`
  - body: CODEOWNERS already marks cron job validation as security-owned, and both proposed solutions would change that boundary.
  - confidence: 0.86
