## Summary

Keep open: SignalRadar is already handling the plugin side through ClawHub/webhook fallback, but current main still lacks both requested OpenClaw seams. The direct-exec cron half is covered by the open canonical tracker https://github.com/openclaw/openclaw/issues/18160, while the reply-route env-injection option remains distinct and security-sensitive.

## What This Changes

The issue requests zero-LLM background alert delivery for skills, either via a direct cron execution payload with delivery or foreground reply-route environment variables that a skill can persist.

## Best Possible Solution

Keep direct cron execution design aligned with https://github.com/openclaw/openclaw/issues/18160, and separately decide whether OpenClaw wants a scoped, authorized reply-route env contract for skills that persist a user-approved chat route.

## Reproduction Assessment

Yes for source-level capability absence, not as a strict bug reproduction. Current main has no direct cron exec payload and no OPENCLAW_REPLY_CHANNEL/OPENCLAW_REPLY_TARGET env contract; I did not run a live SignalRadar or system crontab scenario.

## Solution Assessment

Unclear until maintainers choose the product and security contract. A native direct-exec cron payload belongs with the open canonical tracker, while reply-route env injection is smaller but needs explicit authorization, scoping, route lifetime, and stale-target behavior.
