## Summary

Keep open: current main still lacks a zero-LLM chat-delivery path for system-crontab skills. The native cron direct-exec half is already tracked by https://github.com/openclaw/openclaw/issues/18160, but this SignalRadar workflow also asks for a distinct foreground reply-route env contract that is not implemented or fully owned by another canonical issue.

## What This Changes

Request a zero-LLM way for skills scheduled by system crontab to deliver chat messages, either through a direct cron exec payload or injected reply-route environment variables.

## Best Possible Solution

Keep the direct cron exec design consolidated in https://github.com/openclaw/openclaw/issues/18160, and separately decide whether OpenClaw should expose a scoped, authorized reply-route env contract for skills that intentionally hand off later delivery to system crontab.

## Reproduction Assessment

Yes, for source-level capability absence rather than a strict bug reproduction. Current main has no direct cron exec payload and no OPENCLAW_REPLY_CHANNEL/OPENCLAW_REPLY_TARGET-style env contract; I did not run a live SignalRadar/system-crontab scenario.

## Solution Assessment

Unclear until maintainers choose the product and security contract. A native direct-exec cron payload belongs with the canonical direct-exec tracker, while reply-route env injection is smaller but needs explicit authorization, route lifetime, stale-target, and account/thread semantics.
