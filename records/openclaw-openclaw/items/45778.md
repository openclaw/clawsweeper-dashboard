## Summary

Keep open: current main still lacks the requested zero-LLM cron delivery path, and the active discussion points to a real product/security decision rather than stale support cleanup.

## What This Changes

Request a platform seam that lets skills send background chat messages without an LLM turn, either through a direct cron exec payload or injected reply-route environment variables.

## Best Possible Solution

Define a narrow, documented core/plugin contract for authorized background delivery, then implement it with regression coverage and docs while preserving the existing cron delivery security boundaries.

## Reproduction Assessment

Not applicable as a strict bug reproduction. Source inspection confirms the current cron and CLI surfaces only support systemEvent or LLM-backed agentTurn jobs, and the requested reply-route env vars are absent.

## Solution Assessment

Unclear until maintainers choose the contract. Env injection is likely the smaller reply-route seam, while directExec needs a stronger opt-in security model before it is safe to implement.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because both proposed solutions expand background messaging authority or command execution outside an active agent turn.

Concerns:

- **[medium] Define controls before direct background execution:**

  - body: OPENCLAW_REPLY_CHANNEL and OPENCLAW_REPLY_TARGET would let a foreground skill persist a route for later sends, so the contract needs user authorization, account/thread scoping, and stale target handling.
