## Summary

Keep open: current main and v2026.6.10 still expose `subagents` as a direct-controller list with descendant summaries, not a recursive or depth-limited listing. The request is valid, but it needs maintainer API/product scoping because read-only ancestor visibility should align with broader session-chain work without expanding control permissions.

## What This Changes

Add recursive or depth-limited `subagents list` visibility so a parent agent can inspect nested subagent labels, status, runtime, model, and hierarchy metadata.

## Best Possible Solution

Define a maintainer-approved read-only descendant listing that keeps direct-child listing as the default, adds explicit depth/recursive plus parent/depth metadata, and reuses canonical lineage/traversal without expanding control permissions.

## Reproduction Assessment

Yes, at source level. Current main has no `depth` or `recursive` parameter on the `subagents` tool or slash-command path and filters list results to direct controller-owned runs, while nested workers are only summarized as descendants.

## Solution Assessment

Unclear until maintainers choose the API semantics. A depth flag is plausible, but the best fix should align with broader session-chain work and preserve existing control boundaries.
