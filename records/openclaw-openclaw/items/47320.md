## Summary

Keep open: current main still exposes only direct controller-owned subagent runs, while nested workers are summarized as descendant counts or child session keys; the request is valid but needs a maintainer product/API decision because recursive visibility changes the current direct-child scoping model and overlaps the broader session-chain work.

## What This Changes

Add a `depth` or recursive option to `subagents list` so a top-level orchestrator can inspect nested worker labels, status, runtime, model, task, and parent linkage.

## Best Possible Solution

Define a maintainer-approved read-only descendant listing that keeps the direct-child default, exposes explicit depth/parent metadata, reuses canonical lineage or registry traversal, and leaves control permissions scoped to the owning parent.

## Reproduction Assessment

Yes, at source level. Current main has no `depth` or recursive parameter on the `subagents` tool and filters visible runs to the exact controller session, while nested children only appear as descendant counts or child session keys.

## Solution Assessment

Unclear. A recursive read-only listing is plausible, but the best shape depends on a maintainer decision about cross-depth visibility and whether to build on the broader session-chain contract in https://github.com/openclaw/openclaw/issues/11040.
