## Summary

Keep open: current main and v2026.6.10 still expose `subagents` as a direct-controller list, with depth-2 workers summarized as descendant counts or child session keys rather than returned by a depth/recursive API. The request is valid, but it needs maintainer API/product scoping because read-only ancestor visibility should align with session-chain work without expanding control permissions.

## What This Changes

Request to add depth-limited or recursive descendant visibility to `/subagents list` and the model-callable `subagents` list action.

## Best Possible Solution

Define a maintainer-approved read-only descendant listing that keeps direct-child listing as the default, adds explicit depth/recursive plus parent/depth metadata, and reuses canonical lineage/traversal without expanding control permissions.

## Reproduction Assessment

Yes, at source level. Current main has no `depth` or `recursive` parameter on the `subagents` tool or slash-command path, and the list implementation filters to direct controller-owned runs while only summarizing descendants.

## Solution Assessment

Unclear until maintainers choose the API semantics. A depth flag is plausible, but the best fix should align with broader session-chain work and preserve existing control boundaries.
