## Summary

Keep open. Current main still exposes `subagents list` as a direct-child view only; depth-2 workers are summarized as descendants but not returned as rows with their own labels, status, runtime, model, or parent metadata, and the corroborating user comment confirms this is a real nested-orchestrator workflow gap.

## What This Changes

The issue requests a recursive or depth-limited `subagents list` mode that exposes nested subagents with parent/session hierarchy metadata.

## Best Possible Solution

Add bounded read-only descendant traversal for list output, preserving the direct-only default and returning flat rows with `depth` plus parent session metadata for tool and slash-command consumers.

## Reproduction Assessment

Yes, by source inspection for the feature gap. A main -> orchestrator -> worker setup currently lists runs by exact controller session key, so the worker owned by the orchestrator is omitted from the root caller's rows.

## Solution Assessment

Yes. A backward-compatible `depth` or `recursive` list option is the narrow maintainable path if it reuses existing descendant traversal/liveness logic and keeps control operations direct-scoped.
