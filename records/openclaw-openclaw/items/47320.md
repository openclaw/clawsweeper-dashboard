## Summary

Keep open. Current main still keeps `subagents` listing scoped to direct children, while descendant data is only available as counts or helper internals; the corroborating issue comment also confirms this is real orchestration friction rather than a one-off support request.

## What This Changes

The issue requests `subagents list` support for depth-bounded or recursive descendant visibility from an ancestor agent.

## Best Possible Solution

Add bounded read-only descendant traversal for list output, preserving direct-only default behavior and returning flat rows with `depth` plus parent session metadata for both tool and slash-command consumers.

## Reproduction Assessment

Yes, by source inspection. In a main -> orchestrator -> worker setup, current list requests call `listControlledSubagentRuns` with the main session key, so the worker owned by the orchestrator is omitted as its own row.

## Solution Assessment

Yes. A backward-compatible `depth` or `recursive` list option is the narrowest maintainable path if it reuses existing descendant traversal and keeps control operations direct-scoped.
