## Best Possible Solution

Keep the registry-boundary hardening, but guard plugin-owned reads that happen before malformed validation and add an execute-getter sibling regression before maintainer review/merge.

## Reproduction Assessment

Yes from source inspection. Current main directly reads plugin-owned tool fields during validation and descriptor capture, and the PR head still has a pre-validation `execute` read that can reproduce the remaining sibling-drop path.

## Solution Assessment

No. The registry boundary is the right layer, but the patch is incomplete until the pre-validation `isAgentTool`/wrapping path also treats throwing `execute` getters as one malformed tool instead of a whole-entry factory failure.

Full review comments:

- **[P2] Guard execute getters before wrapping plugin tools:** `src/plugins/tools.ts:466-469`
  - body: The new malformed-tool branch catches `tool.execute` here, but plugin factory results are first passed through `wrapPluginToolFactoryResult`, where `isAgentTool` reads `execute` directly before validation. Please guard that pre-validation read or defer wrapping until after validation, and add an execute-getter sibling regression.
