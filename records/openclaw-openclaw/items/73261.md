## Summary

Keep open. Current main does not implement the proposed target-provider fast path, and the PR head still has a blocking security/correctness gap: it accepts the on-disk provider after provider-level checks even though runtime request construction can consume model-level transport fields from `models.json`.

## What This Changes

This PR adds a `targetProvider`/`targetModel` options type for `ensureOpenClawModelsJson`, a cold-cache disk-vs-config short-circuit that compares provider-level `apiKey`, `baseUrl`, `headers`, and `auth`, and a new test file for match, mismatch, and cache behavior.

## Best Possible Solution

Keep the optimization only if the fast path proves the full runtime request-shaping surface is consistent with the planned provider state, or make it fail closed whenever unvalidated disk model/request fields could affect transport. Add focused regression coverage for tampered `models[].baseUrl`, `models[].api`, and model headers/request fields before merge.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to generate `models.json`, edit the target provider's model entry to add a malicious `baseUrl` or `api` while leaving provider-level `apiKey`/`baseUrl`/`headers`/`auth` matching config, clear the in-memory cache, and call `ensureOpenClawModelsJson` with `targetProvider`; the PR fast path can return before the full plan rewrites that model entry.

## Solution Assessment

No. The proposed provider-level comparison is narrower than the runtime request surface; the safer maintainable solution is to validate or reject model-level and request-shaping drift before skipping the full planning path.
