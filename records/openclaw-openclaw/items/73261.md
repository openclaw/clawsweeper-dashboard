Keep open: the latest head fixes the earlier caller-wiring and header-normalization feedback, but the scoped short-circuit cache can still record a hash from a different models.json snapshot than the one it validated, and the supplied runtime proof predates the latest changes.

## What This Changes

## Best Possible Solution

Keep the short-circuit design, but bind scoped cache entries to the exact validated models.json snapshot, add a race-focused regression test, rebase after the related cache PR decision, and refresh current-head runtime proof.

Yes. Source inspection of PR head shows the validated models.json read is discarded and a separate post-validation read supplies the scoped cache hash.

No. The high-level fast-path design is maintainable, but the cache must be tied to the same file snapshot that passed validation before the implementation is safe to merge.

- **[P2] Cache the hash from the validated models.json read:** `src/agents/models-config.ts:1211-1215`
