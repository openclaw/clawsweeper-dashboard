## Summary

Keep this PR open. The latest head addresses several earlier review comments around env-ref comparison, asymmetric provider `baseUrl`, prototype-key lookup, and ready-cache ordering, but the new fast path still accepts on-disk `models.json` after validating only provider-level fields while the runtime can use model-level transport fields from that same file.

## What This Changes

This PR adds `targetProvider`/`targetModel` options for `ensureOpenClawModelsJson`, a provider-level disk-vs-config short-circuit for `models.json`, a focused target-provider test suite, and a few unrelated formatting-only changes.

## Best Possible Solution

Keep the optimization only if the cold-cache fast path proves the full runtime request-shaping surface is consistent with regenerated provider state, or fail closed whenever unvalidated disk model/request fields could affect transport. Add regression coverage for tampered `models[].baseUrl`, `models[].api`, and model headers/request fields, plus the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to generate `models.json`, edit the target provider's model entry to add a malicious `baseUrl`, `api`, or headers while leaving provider-level fields matching config, clear the in-memory cache, and call `ensureOpenClawModelsJson` with `targetProvider`; the PR fast path can return before full planning rewrites the model entry.

## Solution Assessment

No. The proposed provider-level comparison is narrower than the runtime request surface; the safer maintainable solution is to validate or reject model-level and request-shaping drift before skipping the full planning path.
