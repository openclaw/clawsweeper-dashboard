## Summary

Keep open. Current main and the latest release still load explicit hosted Canvas targets as plain WKWebView requests, while the linked closing PR that attempts the capability-URL fix remains open and unmerged.

## What This Changes

The issue reports that macOS remote-node Canvas paths served through the gateway show a 401 in WKWebView because the load is not authenticated or capability-scoped.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/91443 with a node-session Canvas/A2UI capability URL resolver that uses current and refreshed `pluginSurfaceUrls.canvas`, preserves local and external loads, and redacts capability-bearing diagnostics.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection gives a high-confidence path: node Canvas targets are passed unchanged to a plain WKWebView HTTP request, while the gateway protocol expects scoped plugin surface URLs for protected Canvas routes.

## Solution Assessment

Yes for the capability URL direction, but not for raw bearer-header injection, query tokens, or unauthenticated Canvas routes. The narrow maintainable fix should use the existing node-session current/refresh Canvas surface URL path.
