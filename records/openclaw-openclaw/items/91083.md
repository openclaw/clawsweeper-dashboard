## Summary

Keep open. Current main and v2026.6.9 still do not resolve explicit macOS Canvas hosted paths through the scoped Canvas capability URL, and the open linked fix PR is still unmerged with review/proof blockers.

## What This Changes

The issue requests that macOS Canvas remote-node hosted paths load with valid gateway authorization instead of showing a 401 in WKWebView.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/91443 with a node-session Canvas/A2UI capability URL resolver that uses current and refreshed `pluginSurfaceUrls.canvas`, preserves local and external loads, and redacts capability-bearing diagnostics.

## Reproduction Assessment

Yes, at source level rather than from a live macOS run. Current main passes node Canvas targets into WKWebView without capability rewriting, while the gateway protocol requires scoped and refreshable Canvas plugin surface URLs for protected hosted routes.

## Solution Assessment

Yes for the capability URL direction, but not for raw bearer-header injection, query-token shortcuts, or unauthenticated Canvas routes. The maintainable fix should use the existing node-session current/refresh Canvas surface URL path.
