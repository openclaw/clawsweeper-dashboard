## Summary

Keep open. Current main and v2026.6.6 still route macOS node Canvas targets into the existing WKWebView load path without a generic hosted Canvas capability rewrite, and the linked open PR is the active but unmerged fix path.

## What This Changes

The issue reports that macOS Canvas remote-node `present` loads hosted Canvas paths without gateway auth or capability routing, causing 401 responses behind Tailscale.

## Best Possible Solution

Land a repaired version of https://github.com/openclaw/openclaw/pull/91443 or an equivalent focused fix that resolves hosted Canvas/A2UI paths through the node-session scoped Canvas capability URL, refreshes expiring capabilities, and keeps credentials out of logs.

## Reproduction Assessment

Yes from source, but not from live execution here. The reported node `canvas.present` target is forwarded into the existing Canvas load path while the Gateway docs require node-scoped capability URLs for protected Canvas WebView access.

## Solution Assessment

No final fix is on main yet. The safest direction is capability URL rewriting through the node session, not shared bearer-header exposure, query-token auth, or unauthenticated Canvas routes.
