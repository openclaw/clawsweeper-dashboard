## Summary

Keep open. Current main still has the macOS Canvas load path using plain WKWebView URL requests for explicit HTTP(S) targets, while the repository already has a scoped Canvas capability-url pattern that should be applied to hosted Canvas paths instead of weakening gateway auth.

## What This Changes

The issue asks for remote macOS Canvas presentation to authenticate hosted Gateway Canvas URLs when WKWebView loads them through Tailscale.

## Best Possible Solution

Teach the macOS Canvas path resolver to map safe Gateway Canvas/A2UI paths onto the current node-scoped canvas plugin surface URL, with capability refresh on failure, while leaving arbitrary external URLs and local canvas paths unchanged.

## Reproduction Assessment

No, not with high confidence from live execution. Source inspection shows a plausible current-main failure path for explicit hosted Canvas URLs, but the macOS/Tailscale WKWebView flow was not exercised here.

## Solution Assessment

No, the issue's preferred bearer-header fix is not the best shape because it is auth-sensitive and may not cover subresources cleanly. The safer existing pattern is node-scoped capability URL rewriting, already used by the Control UI and documented for node WebViews.
