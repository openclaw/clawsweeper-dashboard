## Summary

Keep open. Current main and v2026.5.19 still lack the requested durable Gateway/OpenResponses Excel and generic-binary `input_file` contract; adjacent WebChat attachment offload exists, but `/v1/responses` file inputs still reject unsupported Office/binary MIME types or decode accepted non-PDF files into prompt text rather than storing receipt-backed artifacts.

## What This Changes

The issue requests first-class Excel and generic binary Gateway `input_file` uploads with MIME policy, durable workspace storage, receipt metadata, stable tool-readable paths, and documentation.

## Best Possible Solution

Define a documented OpenResponses binary upload contract that fail-closes on MIME, size, and URL policy, stores accepted binaries through managed workspace media with receipt metadata, exposes stable tool-readable paths, and then add Excel coverage and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `/v1/responses` base64 `input_file` with an Excel MIME is outside the default allowlist and rejects before durable storage or receipt creation.

## Solution Assessment

Unclear as a concrete implementation. The direction is valid, but the maintainable fix is a managed binary storage and security contract, not only adding Excel MIME strings.
