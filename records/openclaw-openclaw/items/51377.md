- `impact:data-loss`: Unsupported or non-materialized binary uploads can leave user-provided operational files absent from the agent workflow and backupable workspace state.

## Summary

Keep open: current main partially solves chat.send attachments, including xlsx/generic binary offload, but OpenResponses input_file still rejects Excel/generic binary by default and only turns allowed files into prompt context rather than a durable workspace receipt.

## What This Changes

Request support for Excel and generic binary uploads in Gateway/OpenResponses input_file with MIME allowlists, size limits, workspace receipts, and documentation.

## Best Possible Solution

Define a maintainer-approved Gateway/OpenResponses binary upload contract that persists explicitly allowed files to managed media or workspace storage with metadata, limits, redaction/security docs, and tool/skill access rules; keep Office parsing or tabular previews as a separate extractor/plugin decision.

## Reproduction Assessment

Yes. Source inspection shows OpenResponses input_file defaults still exclude Excel/generic binary MIME types and the handler only injects allowed file content into prompt context, while chat.send has only a partial offload path.

## Solution Assessment

No current implementation is complete. The chat.send offload path is useful precedent, but the best fix is an explicit OpenResponses/Gateway binary persistence contract with security rules rather than only expanding MIME decoding.
