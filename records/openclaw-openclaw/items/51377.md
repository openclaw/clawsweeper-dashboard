## Summary

Keep open: current main and v2026.6.10 still only extract text/PDF content for OpenResponses `input_file`, while merged chat attachment work covers `chat.send` staging but not durable Excel/generic binary `input_file` receipts for tools or skills.

## What This Changes

The issue requests first-class Excel and generic binary upload support in Gateway/OpenResponses `input_file`, with allowlists, limits, workspace persistence, metadata receipts, and docs.

## Best Possible Solution

Define a Gateway/OpenResponses binary upload contract that stores explicitly allowed files as managed workspace/media artifacts with metadata, limits, redaction rules, and documented tool/skill access, while keeping provider-native forwarding and read-tool extraction as separate tracked concerns.

## Reproduction Assessment

Yes, at source level. Current main rejects Excel/generic binary MIME types on the OpenResponses `input_file` path and only emits extracted prompt context for allowed files, while `chat.send` staging covers a different entrypoint.

## Solution Assessment

No, current main is only a partial solution. Extending MIME defaults alone is not the best fix because binary persistence, receipts, redaction, and tool/workspace access need an explicit Gateway/OpenResponses contract.
