## Summary

Keep open. Current main and the latest release still do not implement the durable Excel/generic-binary OpenResponses upload contract: Excel MIME types remain outside the default input_file allowlist, unsupported types fail before storage, and accepted files are only decoded into ephemeral prompt context. Related document-read and webchat attachment work covers adjacent surfaces, but it does not supersede this Gateway/OpenResponses request.

## What This Changes

The issue requests first-class Excel and generic binary uploads through Gateway/OpenResponses input_file, including MIME policy, size limits, durable workspace storage with metadata receipts, and documentation.

## Best Possible Solution

Define a documented OpenResponses upload contract that fail-closes on MIME, size, and URL policy, stores accepted binaries through managed workspace media with receipt metadata, exposes stable tool-readable paths, and then adds Excel coverage and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a /v1/responses base64 input_file with application/vnd.openxmlformats-officedocument.spreadsheetml.sheet is outside the default allowlist and throws before any durable persistence path.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best fix is not just adding Excel MIME strings; it should connect OpenResponses to managed storage, receipts, redaction, and sandbox staging semantics.
