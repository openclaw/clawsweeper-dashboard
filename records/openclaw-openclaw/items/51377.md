## Summary

Keep open: current main and v2026.6.10 have partial attachment improvements, but OpenResponses `input_file` still defaults to text/JSON/PDF extraction and does not provide durable Excel or generic binary workspace receipts.

## What This Changes

Request support for Excel and generic binary uploads in Gateway/OpenResponses `input_file`, with MIME and size policy, durable workspace metadata, clear unsupported errors, and docs.

## Best Possible Solution

A maintainer-approved Gateway media contract should stage allowed binary `input_file` payloads as named workspace artifacts with metadata and explicit unsupported errors, while leaving richer parsing and provider-native forwarding to plugin/provider follow-ups.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: an `.xlsx` `input_file` is outside the default allowlist, and allowed non-PDF files are decoded into prompt text rather than persisted as binary workspace artifacts.

## Solution Assessment

No, an allowlist-only change is not the best fix because OOXML and generic binaries need a receipt/staging contract, not raw text decoding; the maintainable path is a Gateway-level binary attachment contract with security policy.
