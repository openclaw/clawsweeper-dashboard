## Summary

Keep open. Current main still rejects Excel MIME types on the OpenResponses `input_file` path and turns supported files into ephemeral prompt context rather than durable workspace/media receipts; the newer `chat.send` non-image attachment path is adjacent progress but does not satisfy this Gateway API request.

## What This Changes

Request support for Excel and generic binary `input_file` uploads in Gateway/OpenResponses with explicit MIME allowlists, size limits, durable workspace materialization, metadata, and documentation.

## Best Possible Solution

Keep this as the canonical Gateway/OpenResponses file-ingest request. The desired end state is a documented `input_file` contract that fail-closes on MIME and size policy, accepts explicitly configured Excel/binary types, stores accepted binaries through managed workspace/media storage, records filename/MIME/size/SHA metadata, and exposes stable receipts or paths to tools without weakening existing URL-fetch protections.

## Reproduction Assessment

Yes. A `/v1/responses` request with a base64 `input_file` using `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` follows `extractFileContentFromSource` and fails the default allowlist before persistence; the source path is direct enough to reproduce without broad test execution.

## Solution Assessment

Unclear. The requested direction is the right product surface, but the best maintainable fix is not a simple allowlist addition; it should deliberately unify or adapt the existing chat media offload path for OpenResponses with clear security and retention semantics.
