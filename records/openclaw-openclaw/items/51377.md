## Summary

Keep open. Current main still implements OpenResponses `input_file` as text/PDF prompt extraction, not a durable Excel or generic binary upload path, and the broader request needs an API/security design before a narrow MIME-list patch would be maintainable.

## What This Changes

Add first-class Excel and generic binary uploads for Gateway/OpenResponses `input_file` with allowlists, size limits, durable workspace receipts, and documentation.

## Best Possible Solution

Define a documented Gateway/OpenResponses `input_file` contract that fail-closes on MIME and size policy, persists accepted binaries as managed workspace media with receipt metadata and tool-readable paths, then add Excel coverage and docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: a `/v1/responses` base64 `input_file` with the standard `.xlsx` MIME misses the default allowlist and throws before storage; even if allowlisted, the current non-PDF path decodes bytes into prompt text instead of producing a durable receipt.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers decide the upload contract; the request is directionally sound, but a MIME-list-only patch is not the best fix because storage, receipts, redaction, retention, and tool staging need to be designed together.
