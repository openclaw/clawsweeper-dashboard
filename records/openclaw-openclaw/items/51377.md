## Summary

Keep open. Current main still rejects default Excel MIME types on `/v1/responses` `input_file` and only turns allowed file content into ephemeral prompt context; the requested generic binary path needs Gateway/API storage and security design before implementation.

## What This Changes

Requests first-class Excel and generic binary support for Gateway/OpenResponses `input_file` uploads, including allowlists, size limits, workspace persistence, metadata receipts, and docs.

## Best Possible Solution

Define a documented OpenResponses upload contract that fail-closes on MIME, size, and URL policy, stores accepted binaries through managed workspace media with receipt metadata, exposes stable tool-readable paths, and then adds Excel coverage and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `/v1/responses` base64 `input_file` with `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` misses the default allowlist and throws before persistence; no live request was run because this review is read-only.

## Solution Assessment

Unclear. The requested Gateway surface is reasonable, but the best fix is not only adding Excel MIME strings; it should connect OpenResponses to managed storage, receipts, redaction, and sandbox staging semantics.
