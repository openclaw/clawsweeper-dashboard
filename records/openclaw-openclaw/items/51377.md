## Summary

Keep open. Current main still rejects default Excel MIME types on `/v1/responses` `input_file` and only turns allowed files into ephemeral prompt context; adjacent `chat.send` non-image offload support does not satisfy the requested durable Gateway/API upload contract.

## What This Changes

The issue requests first-class Gateway/OpenResponses `input_file` support for Excel and generic binary uploads, including allowlists, limits, durable workspace receipts, tool access, and documentation.

## Best Possible Solution

Define a documented OpenResponses `input_file` contract that fail-closes on MIME and size policy, stores accepted binaries through managed media/workspace storage, records receipt metadata, and exposes stable tool-readable paths before adding Excel coverage and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `/v1/responses` base64 `input_file` with `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` misses the default allowlist and throws before media persistence.

## Solution Assessment

Unclear. The requested Gateway surface is reasonable, but the best fix is not a narrow MIME-list addition; it should deliberately connect OpenResponses to managed storage, receipts, redaction, and staging semantics.
