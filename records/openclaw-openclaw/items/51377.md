## Summary

Keep open: current main still rejects Excel/generic binary `input_file` payloads by default and only extracts allowlisted text/PDF content into prompt context; the newer chat.send attachment offload work partially covers a different Gateway path but does not provide the durable OpenResponses receipt/path contract requested here.

## What This Changes

The issue requests first-class Excel and generic binary upload support through Gateway/OpenResponses `input_file`, with allowlists, size limits, durable workspace metadata, security behavior, and docs.

## Best Possible Solution

Define one bounded Gateway/OpenResponses file-ingest contract: default-deny allowlists and size limits, explicit errors for disallowed types, durable media-store receipt/path metadata for allowed binary files, transcript/redaction rules, and focused tests/docs for Excel, zip/octet-stream, and rejection cases.

## Reproduction Assessment

Yes, source-level: current main's OpenResponses `input_file` defaults omit Excel/binary MIME types and throw on unsupported MIME after sniffing. I did not run a live request because this review is read-only, but the code path is direct.

## Solution Assessment

Unclear as an implementation plan: the requested direction is plausible, but the safest solution needs a maintainer-owned upload/receipt/security contract rather than a simple allowlist expansion.
