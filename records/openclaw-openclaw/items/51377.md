## Label Justifications

- `P2`: The issue blocks real spreadsheet and binary-upload workflows, but it is a feature/design gap with bounded blast radius rather than an outage.
- `impact:data-loss`: Unsupported or silently absent operational file uploads can prevent expected spreadsheet/binary artifacts from persisting into agent workflows.
- `impact:security`: The requested upload path affects MIME allowlists, URL-fetch policy, transcript redaction, retention, and workspace/tool exposure.

## Summary

Keep open: current main and latest release still treat OpenResponses `input_file` as text/PDF extraction only, while merged `chat.send` and file-only `/v1/responses` work covers adjacent pieces without delivering durable Excel/generic binary upload receipts for tools or skills.

## What This Changes

The issue requests first-class Excel and generic binary upload support for Gateway/OpenResponses `input_file`, including MIME policy, size limits, persistence metadata, tool access, and documentation.

## Best Possible Solution

Define a Gateway/OpenResponses binary upload contract that stores explicitly allowed files as managed workspace/media artifacts with metadata, limits, redaction rules, and documented tool/skill access, while keeping provider-native forwarding and read-tool binary safeguards tracked by their narrower related issues.

## Reproduction Assessment

Yes, at source level. Current main's default `input_file` allowlist excludes Excel/generic binary MIME types, and the OpenResponses handler only produces extracted prompt context rather than a durable file receipt.

## Solution Assessment

No, current main is only a partial solution. Adding MIME defaults alone is not the best fix because binary persistence, redaction, workspace access, and security policy need an explicit Gateway/OpenResponses contract.
