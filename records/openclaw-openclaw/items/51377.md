## Summary

Keep open: current main and v2026.6.10 still implement OpenResponses input_file as text/PDF extraction into prompt context, while merged chat.send staging and file-only acceptance are useful partial overlaps that do not provide Excel/generic binary receipts or durable tool/skill access.

## What This Changes

The issue requests first-class Excel and generic binary upload support through Gateway/OpenResponses input_file, including persistence, allowlists, limits, security behavior, and documentation.

## Best Possible Solution

Define a Gateway/OpenResponses binary upload contract that stores explicitly allowed files as managed workspace/media artifacts with metadata, limits, redaction rules, and documented tool/skill access, while keeping provider-native forwarding and read-tool parsing as separate tracked concerns.

## Reproduction Assessment

Yes, at source level. Current main rejects Excel/generic binary MIME types on the OpenResponses input_file path and only emits extracted prompt context for allowed files, while chat.send staging covers a different entrypoint.

## Solution Assessment

No, not as a simple allowlist extension. The maintainable fix needs an explicit binary receipt, persistence, redaction, and tool/workspace access contract before Excel or generic binary defaults are broadened.
