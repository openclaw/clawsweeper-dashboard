- `impact:data-loss`: The issue is about unsupported or silently absent operational binary uploads that users expect to persist for agent workflows.

## Summary

Keep open: current main and v2026.6.6 only partially address the chat.send attachment path; OpenResponses input_file still lacks the requested Excel/generic binary durable workspace receipt path, and the remaining scope is product/security-sensitive.

## What This Changes

Request support for Excel spreadsheets and generic binary uploads as first-class Gateway/OpenResponses input_file inputs with limits, metadata, workspace exposure, and documentation.

## Best Possible Solution

Define a core Gateway/OpenResponses binary upload contract that persists explicitly allowed files to managed media/workspace storage with metadata, limits, redaction docs, and tool/skill access rules, while keeping Office parsing or tabular preview as separate extractor/plugin work unless maintainers choose to bundle it.

## Reproduction Assessment

Yes. Source inspection shows OpenResponses input_file defaults still exclude Excel/generic binary MIME types and the handler only turns allowed file content into prompt context, not a durable workspace receipt; I did not run a live Gateway request in this read-only review.

## Solution Assessment

No, current main is only a partial solution. The chat.send offload work is useful precedent, but the best fix is an explicit OpenResponses binary persistence contract with security rules rather than only expanding MIME decoding.
