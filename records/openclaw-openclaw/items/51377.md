## Summary

Keep open: current main and the latest release still reject Excel and generic binary OpenResponses `input_file` payloads by default and do not provide durable receipt-backed workspace storage; WebChat offload and related read/document issues are adjacent but do not supersede this Gateway contract.

## What This Changes

Requests first-class Gateway/OpenResponses support for Excel and generic binary `input_file` uploads with fail-closed policy, durable workspace metadata, tool-readable paths, and documentation.

## Best Possible Solution

Define a documented OpenResponses binary upload contract that fail-closes on MIME, size, and URL policy, stores accepted binaries through managed workspace media with receipt metadata, exposes stable tool-readable paths, and routes format-specific previews/extraction through document-extractor or plugin seams.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `/v1/responses` base64 `input_file` with an Excel MIME is outside the default allowlist and rejects before durable storage or receipt creation.

## Solution Assessment

Unclear as a concrete implementation. The direction is valid, but the maintainable fix is a managed binary storage and security contract rather than only adding Excel MIME strings.
