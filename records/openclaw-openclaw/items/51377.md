## Summary

Keep open: current main partially covers non-image `chat.send` staging and file-only Responses acceptance, but OpenResponses `input_file` still has no first-class Excel/generic binary persistence or receipt contract for tools and skills.

## What This Changes

Request first-class Excel and generic binary uploads through Gateway/OpenResponses `input_file`, with MIME policy, size limits, persistence metadata, tool/skill access, and docs.

## Best Possible Solution

Define a Gateway/OpenResponses binary upload contract that stores explicitly allowed files as managed workspace/media artifacts with metadata, limits, redaction rules, and documented tool/skill access, while keeping provider-native forwarding and read-tool extraction as separate concerns.

## Reproduction Assessment

Yes, at source level. Current main excludes Excel/generic binary MIME types from the OpenResponses `input_file` defaults and throws before producing any tool-readable receipt, while the implemented `chat.send` staging path is a different entrypoint.

## Solution Assessment

No, current main is only a partial solution. The best fix is not merely adding MIME defaults; it needs a durable persistence, receipt, redaction, and workspace/tool access contract for OpenResponses inputs.
