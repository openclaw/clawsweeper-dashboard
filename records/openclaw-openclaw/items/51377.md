## Summary

Keep open: current main and the latest release still reject Excel and generic binary `input_file` payloads by default and do not provide durable receipt-backed OpenResponses storage; adjacent WebChat offload and read/document work do not supersede this Gateway contract.

## What This Changes

The issue requests first-class Excel and generic binary upload support for Gateway/OpenResponses `input_file`, with allowlists, size limits, durable workspace storage, receipt metadata, tool access, and docs.

## Best Possible Solution

Define a documented OpenResponses binary upload contract that fail-closes on MIME, size, and URL policy, stores accepted binaries through managed workspace media with receipt metadata, exposes stable tool-readable paths, and routes format-specific previews or extraction through document-extractor/plugin seams.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `/v1/responses` base64 `input_file` with an Excel MIME is outside the default allowlist and rejects before durable storage or receipt creation.

## Solution Assessment

Unclear as a concrete implementation. The direction is valid, but the maintainable fix is a managed binary storage and security contract, not only adding Excel MIME strings.
