## Summary

Keep open: current main and v2026.6.6 still limit OpenResponses `input_file` to text/PDF extraction, while the merged `chat.send` attachment work only partially covers the requested durable Excel/generic binary upload path.

## What This Changes

Add first-class Excel and generic binary upload support in Gateway/OpenResponses with MIME policy, size limits, workspace persistence or receipts, and docs for tools/skills.

## Best Possible Solution

Define a Gateway/OpenResponses binary upload contract that stores explicitly allowed files as managed workspace/media artifacts with metadata, limits, redaction rules, and documented tool/skill access; keep provider-native forwarding and spreadsheet preview as separate design follow-ups unless maintainers choose to combine them.

## Reproduction Assessment

Yes, at source level: current `input_file` defaults reject Excel/generic binary MIME types and the handler only injects extracted file context. I did not run a live Gateway request in this read-only review.

## Solution Assessment

No, current main is only a partial solution. The merged `chat.send` offload path is useful precedent, but the requested `input_file` binary persistence contract still needs explicit product and security direction.
