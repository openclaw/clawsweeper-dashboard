## Summary

Keep open: the branch is a useful status-display fix, but it still advertises configured fallbacks for user-pinned `/model` sessions even though the runtime treats those selections as strict, and the visible Telegram proof does not show the corrected post-selection `/status` result.

## What This Changes

The PR filters the `/status` configured fallback display against the selected model and adds status-message unit tests for fallback-list rendering.

## Best Possible Solution

Make `/status` match the effective fallback policy: show configured fallbacks for default or auto-fallback selections, hide or explicitly mark no configured fallbacks for user-pinned selections, add the strict-session regression test, and provide visible `/model` then `/status` proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: build a status message with configured fallbacks plus a session `modelOverride` sourced from `user`; the PR still leaves a fallback line even though runtime fallback resolution returns no configured fallbacks for that session.

## Solution Assessment

No. Filtering only the selected model is too narrow; the status message should follow the same strict user-selection policy used by runtime fallback resolution.
