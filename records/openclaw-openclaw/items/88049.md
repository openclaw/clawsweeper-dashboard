## Summary

Keep open: the PR targets the right `/status` surface, but the fix is too narrow because user-pinned model selections are strict and should not continue to advertise other configured fallbacks; the supplied proof is also build/unit-test output rather than real `/status` behavior.

## What This Changes

The PR filters `/status` configured fallback display entries that exactly match the selected model and adds unit tests for that filter.

## Best Possible Solution

Make `/status` derive the fallback line from the effective fallback policy, hiding it for user or legacy pinned selections and showing it only when configured fallbacks can actually run, then prove the `/model` to `/status` flow in a real surface.

## Reproduction Assessment

Yes from source inspection: `buildStatusMessage()` displays configured fallbacks independently of user override strictness, while runtime policy returns no fallbacks for user-pinned model selections. I did not run a live `/status` flow in this read-only review.

## Solution Assessment

No: filtering only the selected model is not the best fix because user session overrides should not advertise any configured fallback candidates. The narrower maintainable fix is to align the displayed fallback line with the same effective fallback policy used by the runner.
