## Summary

Keep open: the security-hardening direction is still useful, but current main has not implemented explicit NUL-byte or path-length validation, and this PR needs byte-length enforcement, clearer boundary wording/tests, real behavior proof, and a careful rebase over current include-root behavior before merge.

## What This Changes

This PR adds explicit `$include` path validation for NUL bytes and a 4096-length cap in `src/config/includes.ts`, with config tests and gateway docs.

## Best Possible Solution

Land a revised narrow hardening patch that rejects NULs before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves allowed-root semantics, and adds boundary plus multibyte coverage.

## Reproduction Assessment

Yes for the PR defect: the diff uses `.length`, and a Node probe shows 2048 CJK characters are 2048 code units but 6144 UTF-8 bytes. This is review proof for the proposed patch rather than a current-main bug reproduction.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary in errors and docs, add boundary and multibyte tests, and rebase over current allowed-root include behavior.

## Review Findings

  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. Because the docs and rationale tie the cap to PATH_MAX-style path length, multibyte UTF-8 paths can exceed 4096 bytes while passing validation; use byte length for both checks and add multibyte coverage.

After switching to byte length, phrase the message and docs as an exclusive fewer-than-4096-byte limit and add acceptance-side boundary coverage.
  - confidence: 0.82
