## Summary

Keep open. Current main still lacks the proposed deterministic `$include` null-byte and path-length validation, but this PR should not merge as-is because its PATH_MAX-style cap is implemented with JavaScript string length instead of UTF-8 byte length.

## What This Changes

The PR adds early `$include` validation for null bytes and a 4096-character path cap, expands include tests, and updates gateway configuration docs.

## Best Possible Solution

Land a revised narrow hardening patch that rejects null bytes before path resolution, enforces a clearly documented fewer-than-4096 UTF-8-byte limit on both raw and normalized/resolved include paths, preserves the current allowed-roots semantics, and adds raw-limit, resolved-limit, near-boundary, and multibyte boundary tests.

## Reproduction Assessment

Yes for the review finding: a source/diff check plus a byte-length probe reproduces why the proposed `.length` guard does not enforce the documented filesystem-style boundary. This is a security-hardening PR rather than a reported runtime bug.

## Solution Assessment

No. The direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary in errors/docs, and rebase onto the current include resolver before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-207`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. A multibyte UTF-8 path can stay below 4096 code units while exceeding the PATH_MAX-style byte boundary this PR documents, so validation still falls through to filesystem behavior instead of the intended deterministic `ConfigIncludeError`. Use UTF-8 byte length for both checks and add a multibyte boundary test.
  - confidence: 0.93
- **[P3] Clarify the exclusive path length boundary:** `src/config/includes.ts:199-209`
  - body: The guards reject values at `>= MAX_INCLUDE_PATH_LENGTH`, but the errors say `maximum length (4096 characters)`, which reads as if 4096 is allowed. After switching to byte length, phrase the message and docs as an exclusive fewer-than-4096-byte bound.
  - confidence: 0.82
