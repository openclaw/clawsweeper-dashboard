## Summary

Keep open. The hardening is still absent on current main, and this PR is useful but not merge-ready because the proposed length guard measures JavaScript UTF-16 code units while the PR/docs justify a PATH_MAX-style filesystem boundary; existing review comments also identify ambiguous exclusive-boundary wording and missing boundary coverage.

## What This Changes

The PR adds config `$include` validation for null bytes and path length limits, updates include tests, and documents the new gateway include path constraints.

## Best Possible Solution

Land a revised narrow hardening patch that rejects null bytes before path resolution, enforces a clearly documented fewer-than-4096 UTF-8-byte limit on both raw and normalized/resolved include paths, aligns errors and docs with that exclusive byte boundary, and adds raw-limit, resolved-limit, near-boundary acceptance, and multibyte boundary tests.

## Reproduction Assessment

Not applicable as a security-hardening PR rather than a reported runtime bug. The review has a high-confidence source/diff check plus a byte-length probe showing why the proposed guard does not enforce the documented filesystem-style byte limit.

## Solution Assessment

No. The direction is appropriate, but the implementation should measure UTF-8 bytes for both raw and resolved include paths and make the exclusive boundary clear in errors, docs, and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-207`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. A multibyte UTF-8 path can stay below 4096 code units while exceeding the PATH_MAX-style byte boundary the PR documents, so validation still falls through to filesystem behavior instead of the intended deterministic `ConfigIncludeError`. Use UTF-8 byte length for both checks and add a multibyte boundary test.
  - confidence: 0.93
- **[P3] Clarify the exclusive path length boundary:** `src/config/includes.ts:199-209`
  - body: The guards reject values at `>= MAX_INCLUDE_PATH_LENGTH`, but the errors say `maximum length (4096 characters)`, which reads as if 4096 is allowed. After switching to byte length, phrase the message and docs as an exclusive fewer-than-4096-byte bound.
  - confidence: 0.82
