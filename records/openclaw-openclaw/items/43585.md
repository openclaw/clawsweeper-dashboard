## Summary

Keep open: the hardening is still relevant on current main, but this PR is not ready to merge because its length cap uses JavaScript code-unit length instead of filesystem byte length, the branch conflicts with newer include-root behavior, and the external contributor has not provided real after-fix behavior proof.

## What This Changes

The PR adds `$include` NUL-byte rejection, raw/resolved include-path length checks, focused include tests, and gateway configuration docs updates.

## Best Possible Solution

Land a rebased narrow hardening patch that rejects NULs before path resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized paths, preserves allowed-root guarded reads, and adds boundary plus multibyte coverage.

## Reproduction Assessment

Yes for the proposed-patch defect: source inspection shows the guards use `.length`, and a Node byte-length probe demonstrates a multibyte path can exceed 4096 UTF-8 bytes while staying below the guard. Current-main source also confirms the requested explicit NUL/length validation is not yet implemented.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary, add boundary and multibyte tests, and rebase over current allowed-root behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-211`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. Because the docs tie this limit to PATH_MAX-style filesystem length, multibyte UTF-8 paths can exceed 4096 bytes while passing validation; use byte length for both raw and normalized checks and add multibyte coverage.
  - confidence: 0.93

  - confidence: 0.82
