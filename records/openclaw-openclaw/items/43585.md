## Summary

Keep open: current main does not already include the requested NUL-byte and include-path length hardening, but the PR is not merge-ready because the proposed length cap uses JavaScript string length instead of UTF-8 byte length and the external contributor has not posted real after-fix behavior proof.

## What This Changes

The PR adds `$include` NUL-byte rejection, raw and resolved include path length guards, related include tests, and gateway configuration docs updates.

## Best Possible Solution

Land a revised narrow hardening patch that rejects NUL bytes before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves allowed-root semantics, and adds boundary plus multibyte tests.

## Reproduction Assessment

Yes for the PR review finding: the PR head uses `.length` for both guards, and a Node byte-length probe shows multibyte paths can stay under 4096 code units while exceeding 4096 UTF-8 bytes. This is not a current-main user bug reproduction because the item is a proposed security-hardening patch.

## Solution Assessment

No. The direction is appropriate, but the implementation should measure UTF-8 bytes, make the exclusive fewer-than-4096-byte boundary clear in docs and errors, and rebase without losing current allowed-root behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-211`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. Because the docs and rationale tie the cap to PATH_MAX, a multibyte UTF-8 path can exceed 4096 bytes while passing validation, leaving the hardening dependent on filesystem errors instead of the intended deterministic `ConfigIncludeError`; use UTF-8 byte length for both checks and add a multibyte boundary test.
  - confidence: 0.93
- **[P3] Clarify the exclusive path length boundary:** `src/config/includes.ts:199-213`
  - body: The guards reject values at `>= MAX_INCLUDE_PATH_LENGTH`, but the errors say `maximum length (4096 characters)`, which reads as if 4096 characters are allowed. After switching to byte length, phrase the errors and docs as an exclusive fewer-than-4096-byte limit so users do not retry with a rejected boundary value.
  - confidence: 0.82
