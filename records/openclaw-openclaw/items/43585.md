## Summary

Keep open. Current main still lacks explicit `$include` NUL/path-length validation, but this external security PR is not merge-ready because the length cap uses UTF-16 code units, the branch conflicts with current include-root handling, and no after-fix real behavior proof is posted.

## What This Changes

The PR adds `$include` NUL-byte rejection, raw/resolved include-path length validation, focused include tests, and gateway configuration docs updates.

## Best Possible Solution

Land a rebased hardening patch that rejects NULs before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves approved include roots, and covers boundary plus multibyte cases.

## Reproduction Assessment

Yes for the proposed-patch defect: the PR diff uses `.length` on raw and normalized paths, and a Node probe shows multibyte paths can exceed 4096 UTF-8 bytes while staying below the guard. Current main source also confirms the requested explicit validation is still absent.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary, rebase over current approved-root behavior, and include after-fix proof before merge.
