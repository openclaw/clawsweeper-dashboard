## Summary

Keep open: current main does not already implement deterministic include path validation, but the PR still has a blocking correctness gap because its PATH_MAX-style guard counts UTF-16 code units instead of filesystem bytes, and external real-behavior proof is absent.

## What This Changes

The PR adds config `$include` path validation for NUL bytes and long paths, with matching tests and gateway configuration docs.

## Best Possible Solution

Apply NUL rejection and byte-based raw/resolved include path caps to current `src/config/includes.ts`, with docs/tests that match byte semantics and real config-load proof.

## Reproduction Assessment

Yes for the review finding: the PR's guard uses JavaScript string length, and a local Node check shows 2048 multibyte characters can occupy 6144 UTF-8 bytes. The original hardening is also source-reproducible because current main has no deterministic NUL or length validator before path resolution.

## Solution Assessment

No. Early validation is the right direction, but the path cap should be byte-based, and the tests/docs/error wording should align with that contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-216`
  - body: The new PATH_MAX-style guard uses `includePath.length` and `normalized.length`, which count UTF-16 code units instead of filesystem bytes. A multibyte path can exceed 4096 UTF-8 bytes while remaining below 4096 code units, so it bypasses the intended deterministic `ConfigIncludeError` and falls through to platform filesystem behavior. Use byte length for both raw and resolved path checks and add multibyte boundary coverage.
  - confidence: 0.86
