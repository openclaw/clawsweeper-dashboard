## Summary

Keep open: the hardening is useful, but this PR should not merge as-is because the proposed length cap is code-unit based instead of byte based, the external contributor has not supplied real after-fix behavior proof, and the branch is now conflicting with current include-root semantics.

## What This Changes

The PR adds `$include` NUL-byte rejection, a raw/resolved include-path length cap, focused include tests, and gateway configuration docs updates.

## Best Possible Solution

Land a rebased narrow hardening patch that rejects NULs before path resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized paths, preserves allowed-root guarded reads, and adds boundary plus multibyte coverage.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the proposed guards use `.length`, and a Node byte-length probe confirms multibyte paths can exceed 4096 UTF-8 bytes while remaining below 4096 code units. This is review proof for the proposed patch rather than a current-main bug reproduction.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary, add boundary and multibyte tests, and rebase over current allowed-root behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-211`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. Because the docs tie the cap to PATH_MAX-style path length, multibyte UTF-8 paths can exceed 4096 bytes while passing validation; use byte length for both checks and add multibyte coverage.
  - confidence: 0.93

  - confidence: 0.82
