## Summary

Keep open: the hardening is useful and current main does not yet document or explicitly enforce the proposed NUL/path-length validation, but this PR should not merge as-is because the PATH_MAX-style cap is measured with JavaScript string length instead of UTF-8 bytes and contributor real-behavior proof is still missing.

## What This Changes

The PR adds config `$include` NUL-byte rejection, raw and normalized path length guards, tests, and gateway docs for invalid include path format.

## Best Possible Solution

Land a revised narrow hardening patch that rejects NULs before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves allowed-root semantics, and adds boundary plus multibyte coverage.

## Reproduction Assessment

Yes for the PR defect: the diff uses JavaScript `.length` for both path guards, and a Node byte-length probe shows multibyte paths can stay under 4096 code units while exceeding 4096 UTF-8 bytes. This is not a current-main bug reproduction because the item is a proposed security-hardening patch.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary in errors/docs, add boundary and multibyte tests, and rebase over current allowed-root include behavior.
