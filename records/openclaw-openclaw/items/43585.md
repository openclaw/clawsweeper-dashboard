## Summary

Keep open: the hardening is still useful because current main does not explicitly reject NUL include paths or raw/resolved path lengths, but this PR cannot merge as-is due to the byte-length bug, missing real behavior proof, and conflicts with the current include-root resolver.

## What This Changes

The PR adds `$include` NUL rejection, raw/resolved include-path length checks, focused include tests, and gateway configuration docs updates.

## Best Possible Solution

Land a rebased narrow hardening patch that rejects NULs before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized paths, preserves allowed-root behavior, and adds boundary plus multibyte coverage.

## Reproduction Assessment

Yes for the proposed-patch defect: source inspection shows `.length` guards, and a Node byte probe shows multibyte paths can exceed 4096 UTF-8 bytes while staying below the guard. Current main source also shows the requested explicit validation is still absent.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary, rebase over current allowed-root behavior, and add boundary/multibyte tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-211`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. Because the docs tie the 4096 cap to Linux PATH_MAX, multibyte UTF-8 paths can pass validation while exceeding the intended filesystem byte boundary; use byte-length checks for both paths.
  - confidence: 0.93
- **[P3] Clarify the exclusive path length boundary:** `src/config/includes.ts:199-213`
  - body: The guards reject `>= MAX_INCLUDE_PATH_LENGTH`, but the errors say `maximum length (4096 characters)`, which reads as if 4096 characters are allowed. Reword this after switching to bytes and add acceptance-side boundary coverage.
  - confidence: 0.82
