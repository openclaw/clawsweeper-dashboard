## Summary

Keep open. Current main still lacks explicit `$include` NUL-byte and path-length validation, so the hardening is not obsolete, but this PR is not merge-ready because the proposed PATH_MAX-style cap is measured with UTF-16 string length, the contributor proof gate is still missing, and the branch must preserve the include-root behavior from https://github.com/openclaw/openclaw/pull/75746 when rebased.

## What This Changes

The PR adds config `$include` NUL-byte and length validation, related regression tests, and gateway configuration docs updates.

## Best Possible Solution

Land a rebased hardening patch that rejects NULs before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves approved include roots, and adds boundary plus multibyte coverage with redacted real behavior proof.

## Reproduction Assessment

Yes for the proposed-patch defect: the PR diff uses string `.length`, and a Node probe shows multibyte paths can exceed 4096 UTF-8 bytes while staying below that guard. Current main source also confirms the requested explicit validation is still absent.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary, rebase over current approved-root behavior, and include after-fix proof before merge.
