## Summary

Keep open: the hardening direction is useful and current `main` still lacks this exact NUL/length contract, but this PR should not merge as-is because the PATH_MAX-style limit is implemented with JavaScript string length and the branch predates current include-root semantics. Existing Greptile/Codex review comments correctly identify the boundary and byte-length issues.

## What This Changes

The PR adds NUL-byte and 4096-length validation for config `$include` paths, expands include tests, and documents the new path constraints.

## Best Possible Solution

Land a revised narrow hardening patch that rejects NUL bytes before resolution, enforces a clearly documented fewer-than-4096 UTF-8-byte limit on raw and normalized/resolved include paths, preserves allowed-root behavior, and adds near-boundary plus multibyte coverage.

## Reproduction Assessment

Yes for the review finding: the PR diff uses `.length`, and a byte-length probe shows a multibyte path can stay under 4096 JavaScript code units while exceeding 4096 UTF-8 bytes. No current-main user bug is being reproduced because this is a security-hardening PR review.

## Solution Assessment

No. The direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive fewer-than-4096-byte boundary, and rebase onto the current allowed-roots include resolver before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-207`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. Because the docs/rationale tie the cap to `PATH_MAX`, a multibyte UTF-8 path can exceed 4096 bytes while passing the guard, so validation still falls through to filesystem-dependent behavior instead of the deterministic `ConfigIncludeError` this hardening promises. Use UTF-8 byte length for both checks and add a multibyte boundary test.
  - confidence: 0.93
- **[P3] Clarify the exclusive path length boundary:** `src/config/includes.ts:199-209`
  - body: The guards reject values at `>= MAX_INCLUDE_PATH_LENGTH`, but the errors say `maximum length (4096 characters)`, which reads as if 4096 is allowed. After switching to byte length, phrase the message and docs as an exclusive fewer-than-4096-byte bound.
  - confidence: 0.82
