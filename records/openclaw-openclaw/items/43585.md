## Summary

Keep this PR open, but it is not ready to merge: the hardening direction is useful, yet the current patch enforces the PATH_MAX-style limit with JavaScript string length instead of UTF-8 byte length, still needs a rebase over the current include-roots resolver, and lacks after-fix real behavior proof.

## What This Changes

The PR adds NUL-byte rejection and raw/resolved `$include` path length guards, with include tests and gateway docs updates.

## Best Possible Solution

Land a revised narrow hardening patch that rejects NUL bytes before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves allowed-root semantics, and adds boundary plus multibyte tests.

## Reproduction Assessment

Yes for the PR review finding: the diff uses `.length`, and a byte-length probe shows a multibyte path can stay under 4096 JavaScript code units while exceeding 4096 UTF-8 bytes. There is no current-main user bug reproduction because this is a security-hardening PR review.

## Solution Assessment

No. The direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive fewer-than-4096-byte boundary, and rebase onto the current allowed-roots include resolver before merge.

  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. Because the docs and rationale tie the cap to PATH_MAX, a multibyte UTF-8 path can exceed 4096 bytes while passing the guard, so validation still falls through to filesystem-dependent behavior instead of the deterministic `ConfigIncludeError` this hardening promises. Use UTF-8 byte length for both checks and add a multibyte boundary test.
