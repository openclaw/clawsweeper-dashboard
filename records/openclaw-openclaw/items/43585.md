## Summary

Keep open: current main still lacks explicit `$include` NUL-byte and path-length validation, but this external security PR is not merge-ready because the proposed cap measures UTF-16 code units, lacks real behavior proof, and needs rebasing over current include-root handling.

## What This Changes

The PR adds `$include` NUL-byte rejection, a 4096-character raw/resolved path cap, regression tests, and gateway configuration docs updates.

## Best Possible Solution

Land a rebased narrow hardening patch that rejects NULs before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves allowed-root guarded reads, and adds boundary plus multibyte coverage.

## Reproduction Assessment

Yes for the PR defect: the diff uses `.length` guards, and a Node probe shows multibyte paths can be far over 4096 UTF-8 bytes while remaining under 4096 code units. This is review proof for the proposed patch rather than a current-main bug reproduction.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary, add boundary and multibyte tests, and rebase over current allowed-root behavior.

## Review Findings

  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. Because the docs tie the cap to PATH_MAX-style path length, multibyte UTF-8 paths can exceed 4096 bytes while passing validation; use byte length for both checks and add multibyte coverage.

  - confidence: 0.82
