## Summary

Keep open: this is a plausible security-hardening PR, but the patch still has a byte-length enforcement bug, needs rebase over newer include-root behavior, and lacks real after-fix behavior proof.

## What This Changes

The branch adds `$include` NUL-byte rejection, raw and resolved include path length checks, tests, and gateway configuration docs for config include path validation.

## Best Possible Solution

Land a revised narrow hardening patch that rejects NUL bytes before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves allowed-root semantics, and adds boundary plus multibyte tests.

## Reproduction Assessment

Yes for the PR review finding: the PR head uses `.length` for both path guards, and a Node byte-length probe shows multibyte paths can stay under 4096 code units while exceeding 4096 UTF-8 bytes. This is not a current-main bug reproduction because the item is a proposed security-hardening patch.

## Solution Assessment

No. The direction is appropriate, but the implementation should measure UTF-8 bytes, make the exclusive fewer-than-4096-byte boundary clear in docs and errors, and rebase over current allowed-root include behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-211`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. Because the docs and rationale tie the cap to PATH_MAX, a multibyte UTF-8 path can exceed 4096 bytes while passing validation; use UTF-8 byte length for both checks and add a multibyte boundary test.

  - confidence: 0.82
