## Summary

Keep open: the hardening remains useful, but the current patch is not safe to merge because the documented PATH_MAX-style limit is enforced with UTF-16 code units and the external PR has no real behavior proof.

## What This Changes

The branch adds `$include` NUL-byte rejection, raw/resolved include path length checks, tests, and gateway docs for config include path validation.

## Best Possible Solution

Land a revised narrow hardening patch that rejects NULs before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves allowed-root semantics, and adds boundary plus multibyte coverage.

## Reproduction Assessment

Yes for the PR defect: the diff uses `.length` for both guards, and a Node byte-length probe shows multibyte paths can stay under 4096 code units while exceeding 4096 UTF-8 bytes. This is not a current-main bug reproduction because the item is a proposed security-hardening patch.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary in errors/docs, add boundary and multibyte tests, and rebase over current allowed-root include behavior.
