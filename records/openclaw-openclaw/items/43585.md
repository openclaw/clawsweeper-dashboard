## Summary

Keep open: the hardening is still useful and is not on current main, but this PR cannot merge as-is because the path cap is character-based, the branch conflicts with current include-root behavior, and the contributor has not posted real behavior proof.

## What This Changes

The PR adds `$include` path validation for NUL bytes and raw/resolved 4096-character length limits, plus include tests and gateway configuration docs.

## Best Possible Solution

Land a rebased hardening patch that rejects NULs before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves approved include roots, and covers boundary plus multibyte cases.

## Reproduction Assessment

Yes for the proposed-patch defect: the PR diff uses `.length` on raw and normalized paths, and a Node probe shows multibyte paths can exceed 4096 UTF-8 bytes while staying below that guard. Current main source also confirms the requested explicit validation is still absent.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary, rebase over current approved-root behavior, and include after-fix proof before merge.
