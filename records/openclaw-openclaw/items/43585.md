## Summary

Keep this PR open: the hardening is still useful and current main does not implement the proposed explicit NUL/length validation, but the patch is not merge-ready because it measures a filesystem-style limit with JavaScript string length, lacks real behavior proof, and predates current include-root semantics.

## What This Changes

The PR adds `$include` NUL-byte rejection, a 4096 raw/resolved include-path cap, focused tests, and gateway configuration docs updates.

## Best Possible Solution

Land a rebased narrow hardening patch that rejects NULs before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves allowed-root guarded reads, and adds boundary plus multibyte coverage.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the proposed guards use `.length`, and a Node probe confirms multibyte paths can be far over 4096 UTF-8 bytes while staying under 4096 code units. This is review proof for the proposed patch rather than a current-main bug reproduction.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary, add boundary and multibyte tests, and rebase over current allowed-root behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-211`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. Because the docs tie the cap to PATH_MAX-style path length, multibyte UTF-8 paths can exceed 4096 bytes while passing validation; use byte length for both checks and add multibyte coverage.
  - confidence: 0.93
- **[P3] Clarify the exclusive path length boundary:** `src/config/includes.ts:199-213`
  - body: The guards reject values at `>= MAX_INCLUDE_PATH_LENGTH`, but the errors say `maximum length (4096 characters)`, which reads as if 4096 characters are allowed. After switching to byte length, phrase the limit as fewer than 4096 bytes and add acceptance-side boundary coverage.
  - confidence: 0.82
