The PR adds NUL-byte rejection, a 4096-length raw/resolved `$include` path cap, config include tests, and gateway config docs updates.

## Best Possible Solution

Land a rebased hardening patch that rejects NULs before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves approved include roots, and includes boundary/multibyte tests plus redacted real behavior proof.

## Reproduction Assessment

Yes for the proposed-patch defect: source inspection shows `string.length` on raw and resolved paths, and a Node probe showed multibyte paths can be far over 4096 UTF-8 bytes while staying below that guard. Current main also confirms the requested explicit NUL/length validation is not already implemented.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes or deliberately document character semantics, rebase over approved include-root behavior, and provide after-fix proof before merge.
