## Summary

Keep this PR open: current main still lacks explicit NUL-byte and path-length validation for config includes, but the proposed patch is not merge-ready because the security boundary is enforced with JavaScript string length, the branch conflicts with newer include-root behavior, and real behavior proof is still missing.

## What This Changes

The PR adds NUL-byte rejection, a 4096-character raw/resolved `$include` path limit, tests, and gateway configuration docs for config include path validation.

## Best Possible Solution

Land a rebased hardening patch that rejects NULs before path resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves approved include roots, and includes boundary/multibyte tests plus redacted real behavior proof.

## Reproduction Assessment

Yes for the proposed-patch defect: the diff uses `string.length`, and a Node probe shows multibyte paths can exceed 4096 UTF-8 bytes while staying below that guard. Current main also confirms the requested explicit NUL/length validation is not already implemented.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary, rebase over approved include-root behavior, and include after-fix proof before merge.
