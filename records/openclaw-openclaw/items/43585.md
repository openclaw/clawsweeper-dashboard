## Summary

Keep open: this is a plausible narrow security hardening PR, but it is not merge-ready because the proposed length cap uses JavaScript string length rather than UTF-8 byte length, the exclusive boundary messaging/tests are incomplete, the branch conflicts with current allowed-root include behavior, and contributor real behavior proof is still missing.

## What This Changes

The branch adds `$include` NUL-byte validation, raw/resolved path length checks, tests, and gateway docs for config include path validation.

## Best Possible Solution

Land a revised narrow hardening patch that rejects NUL bytes before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves allowed-root semantics, and adds boundary plus multibyte tests.

## Reproduction Assessment

Yes for the PR review finding: the PR diff uses `.length` for both guards, and a Node byte-length probe shows multibyte paths can stay under 4096 code units while exceeding 4096 UTF-8 bytes. This is not a current-main bug reproduction because the item is a proposed security-hardening patch.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary in errors/docs, add boundary and multibyte tests, and rebase over current allowed-root include behavior.
