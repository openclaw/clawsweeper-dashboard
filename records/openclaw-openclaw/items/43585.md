## Summary

Keep open. Current main still lacks the requested explicit NUL-byte and path-length validation, but this PR should not merge as-is because the PATH_MAX-style limit is implemented with JavaScript character counts and the external real behavior proof gate is still missing.

## What This Changes

The PR adds config `$include` NUL-byte and 4096-length validation, related regression tests, and gateway configuration docs updates.

## Best Possible Solution

Land a rebased hardening patch that rejects NULs before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves approved include roots, and adds boundary plus multibyte coverage with redacted real behavior proof.

## Reproduction Assessment

Yes for the proposed-patch defect: the PR diff uses `.length` on raw and normalized paths, and a Node probe shows multibyte paths can exceed 4096 UTF-8 bytes while staying below that guard. Current main also confirms the requested explicit validation is still absent.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary, rebase over current approved-root behavior, and include after-fix proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-211`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. Because the docs tie the cap to Linux `PATH_MAX`, multibyte UTF-8 paths can pass validation while exceeding the intended filesystem byte boundary; use byte-length checks for both raw and resolved paths.

  - confidence: 0.82
