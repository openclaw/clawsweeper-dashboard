## Summary

Keep open: the hardening is still useful because current main lacks explicit `$include` NUL/length validation, but this branch is conflicting and the proposed PATH_MAX-style limit is enforced with UTF-16 string length instead of filesystem bytes.

## What This Changes

The PR adds ConfigIncludeError validation for NUL bytes and a 4096-character raw/resolved `$include` path cap, plus config include tests and gateway docs updates.

## Best Possible Solution

Land a rebased hardening patch that rejects NULs before resolution, enforces a documented fewer-than-4096 UTF-8-byte cap on raw and normalized include paths, preserves approved include roots, and includes boundary/multibyte coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes for the proposed-patch defect: the diff uses `string.length`, and a Node probe shows multibyte paths can exceed 4096 UTF-8 bytes while staying below that guard. Current main also confirms the requested explicit NUL/length validation is not already implemented.

## Solution Assessment

No. The hardening direction is appropriate, but the implementation should measure UTF-8 bytes, clarify the exclusive boundary, rebase over approved include-root behavior, and include after-fix proof before merge.
