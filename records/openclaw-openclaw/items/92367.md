## Summary

The requested gateway-native service credential, distinct scope sets, and agent-target restrictions are already covered by two older open canonical design threads: the service-principal/scoped-identity RFC and the multi-token per-agent authorization issue. This issue adds a useful concrete webhook-router threat model, but no separate implementation boundary that should be tracked independently.

## What This Changes

Requests multiple gateway bearer tokens with per-token operation scopes and optional per-agent dispatch restrictions.

## Best Possible Solution

Keep one coordinated gateway identity/RBAC design across the two canonical issues, and preserve this webhook-router threat model as acceptance criteria for scoped service principals and agent-target authorization.

## Reproduction Assessment

Yes at source level: current gateway shared-secret auth resolves one token, while no configurable bearer-principal mapping supplies distinct scopes or allowed agents. This verifies the gap, although it is a missing feature rather than a failing documented contract.

## Solution Assessment

No as a separate tracking and configuration design; the gateway is the correct enforcement owner, but the durable solution should extend the existing principal and operator-scope model through the canonical RFCs and cover every agent-dispatch entry point.
