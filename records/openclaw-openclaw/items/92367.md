## Summary

The issue is not safely closeable as a duplicate: related reports cover multi-user tokens and a future service-principal phase, but this report adds a concrete plugin-facing least-privilege use case and asks maintainers to decide sequencing. Current main has a closed operator-scope framework, yet shared bearer authentication still lacks independently configured principals and principal-bound agent targeting, so meaningful product and security work remains.

## What This Changes

Add multiple gateway bearer principals with bounded operation scopes and optional restrictions on which agents each principal may dispatch.

## Best Possible Solution

Create one canonical gateway-principal design that reuses existing roles and operator scopes, assigns stable principal identity for audit and rotation, and carries a typed allowed-agent constraint enforced centrally across all dispatch paths; consolidate the three related issues into that design after maintainers choose its scope and sequencing.

## Reproduction Assessment

Yes at the source level: current main has shared bearer authentication and separate device role/scope authorization, but no configured bearer principal can carry its own limited scopes and allowed-agent set. This establishes the capability gap without treating the documented single-token behavior as a bug.

## Solution Assessment

No, the exact proposed `mode: "tokens"` and new operation-name vocabulary are not yet the best proven shape. The safer maintainable direction is to extend the existing principal, role, and operator-scope authorization context and enforce agent targets centrally.
