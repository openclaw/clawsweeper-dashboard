## Summary

Keep open: current main and v2026.6.8 still use a singular shared Gateway token/password that maps to full operator access, while this asks for a new scoped Gateway principal model and token-bound agent dispatch authorization that needs maintainer security/product design.

## What This Changes

Requests native scope-bound Gateway token entries plus an optional token-bound allowed-agent dispatch restriction for plugin or external service callers.

## Best Possible Solution

Keep this as the focused scoped-service-principal design thread until maintainers define the Gateway principal schema, scope vocabulary, dispatch constraints, docs, migration/doctor behavior, and validation plan.

## Reproduction Assessment

No runtime bug reproduction is applicable, but there is a high-confidence source capability check. Current main and v2026.6.8 only support singular shared token/password auth plus trusted-proxy/device identity paths, with no scoped token list or token-bound dispatch lane.

## Solution Assessment

Unclear as proposed. Gateway core is the plausible owner for bearer-token authority, but the exact schema, scope names, wildcard semantics, dispatch-lane behavior, and upgrade path need maintainer security/product direction first.
