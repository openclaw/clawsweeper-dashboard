## Summary

Keep open: current main still has a single shared Gateway token/password model with no `gateway.auth.tokens[]`, per-token scopes, or token-bound dispatch lane. The request is security-relevant and belongs as a maintainer Gateway auth/product design item rather than an autonomous cleanup close.

## What This Changes

The issue requests scope-bound Gateway bearer tokens plus optional per-agent dispatch restrictions for session spawning.

## Best Possible Solution

Maintainers should define one Gateway principal model that composes scoped bearer credentials, dispatch target restrictions, trusted-proxy/device auth, HTTP/plugin routes, and existing tool policy without parallel auth semantics.

## Reproduction Assessment

Yes, as a source-reproducible capability gap: current config, schema, authorization code, and docs all show one shared Gateway bearer path rather than `gateway.auth.tokens[]`.

## Solution Assessment

Unclear as an implementation shape: scoped Gateway tokens are plausible and aligned with the threat model, but the config shape, scope vocabulary, dispatch binding, and compatibility story need maintainer design first.
