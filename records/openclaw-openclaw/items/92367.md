## Summary

Keep open: current main and v2026.6.8 still model Gateway token/password auth as one shared operator secret, with no token-bound scope set or token-bound agent dispatch lane. The related issues overlap, but none replaces this focused scoped service-credential design request.

## What This Changes

The issue requests an opt-in `gateway.auth.tokens[]` model with per-token scopes plus an optional per-token allowed-agent dispatch constraint.

## Best Possible Solution

Have maintainers sponsor a Gateway principal design that adds opt-in scoped tokens or principals plus token-bound dispatch target limits, coordinated with the per-user token and service-identity threads.

## Reproduction Assessment

Yes, as a source-level capability check rather than a bug repro. Current main and v2026.6.8 expose singular shared token/password auth and no `gateway.auth.tokens[]` or token-bound allowed-agent dispatch primitive.

## Solution Assessment

Unclear as proposed. Gateway core is the plausible owner boundary, but the exact token/principal shape, scope vocabulary, SecretRef behavior, revocation model, and dispatch-lane semantics need maintainer security/product direction first.
