## Summary

Keep open: current main and v2026.6.8 still have singular shared-secret Gateway token/password auth, not a scoped token-principal list or token-bound allowed-agent dispatch primitive. The related issues overlap, but none replaces this focused security/product design request.

## What This Changes

The issue requests native scoped Gateway bearer tokens plus an optional per-token allowed-agent dispatch constraint.

## Best Possible Solution

Maintainers should choose one canonical Gateway principal model that supports scoped credentials and agent target limits without duplicating existing trusted-proxy, device-token, or agent allowlist policy.

## Reproduction Assessment

Yes for the capability gap: current source and docs show singular shared-secret auth and full-operator shared-secret semantics, not gateway.auth.tokens[] or token-bound allowed agents. This is not a bug reproduction because the issue requests a new auth primitive.

## Solution Assessment

Unclear as proposed: the Gateway is the right layer for credential authority, but the exact config mode, scope vocabulary, wildcard behavior, SecretRef model, and dispatch-lane relationship need maintainer design before implementation.
