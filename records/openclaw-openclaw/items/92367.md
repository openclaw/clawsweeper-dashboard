## Summary

Keep open: current main and the latest release still have a single shared Gateway token/password path, while related issue history now treats this as the canonical scoped-token and dispatch-lane design request.

## What This Changes

Request native scope-bound Gateway auth tokens plus an optional per-token agent dispatch restriction so leaked plugin/webhook credentials cannot act as full Gateway operator credentials.

## Best Possible Solution

Design one canonical Gateway principal model that covers per-token scopes, identity, and optional agent target restrictions, then implement it with security review, migration behavior, docs, and focused tests.

## Reproduction Assessment

Not a bug reproduction; source inspection gives a high-confidence capability check. Current main and v2026.6.6 support singular gateway.auth.token/password plus trusted-proxy identity, with no gateway.auth.tokens[] or token-bound allowedAgents path.

## Solution Assessment

Unclear as a concrete implementation: the direction is plausible, but the best fix is a maintainer-approved Gateway principal model rather than adding this exact config shape automatically.
