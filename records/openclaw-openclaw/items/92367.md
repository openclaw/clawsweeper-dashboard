## Summary

Keep open: current main and the latest release still only model Gateway shared-secret auth as a singular token/password, with no per-token scope bindings or per-agent dispatch-lane restriction. This is a security-sensitive new Gateway auth/config primitive that needs maintainer product and security design rather than conservative cleanup closure.

## What This Changes

Request native Gateway support for multiple scope-bound bearer tokens plus optional per-token allowed-agent dispatch restrictions.

## Best Possible Solution

Design one canonical Gateway principal model that composes per-token scopes, trusted-proxy/device-token identity, shared-secret HTTP/WS semantics, and agent dispatch restrictions, then sequence it with the existing tokenScopes and per-user-token work.

## Reproduction Assessment

No bug reproduction is applicable, but source inspection gives a high-confidence capability check. Current main and v2026.6.6 support singular gateway.auth.token/password plus trusted-proxy/device-token identity, with no supported gateway.auth.tokens[] or token-bound allowedAgents primitive.

## Solution Assessment

Unclear as proposed: a native scoped-token primitive is plausible and likely belongs in core, but the exact tokens[] shape and dispatch-lane semantics need maintainer security/product design before implementation.
