## Summary

Keep open: current main and v2026.6.6 still model Gateway shared-secret auth as a single full-operator secret, while this issue asks for a new scoped Gateway principal model plus dispatch target limits. The related discussions are complementary: https://github.com/openclaw/openclaw/issues/89314 was closed toward this narrower issue and https://github.com/openclaw/openclaw/issues/43903 tracks the per-user identity variant rather than replacing this route-scoped/service-credential threat model.

## What This Changes

The request is for scope-bound `gateway.auth.tokens[]` entries plus an optional per-token `allowedAgents` dispatch restriction for Gateway-authenticated agent dispatch.

## Best Possible Solution

Design one canonical Gateway principal model for scoped tokens/service principals, method scopes, dispatch-target restrictions, compatibility, and docs before accepting an implementation PR.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug reproduction, but yes as a source-level capability check: current main and v2026.6.6 accept only a singular shared Gateway token/password and have no `gateway.auth.tokens[]` or token-bound `allowedAgents` path.

## Solution Assessment

Is this the best way to solve the issue? Unclear as specified: native scoped principals likely belong in the Gateway, but maintainers still need to choose the principal model, scope vocabulary, and whether dispatch target limits live on the same token entry or a separate primitive.
