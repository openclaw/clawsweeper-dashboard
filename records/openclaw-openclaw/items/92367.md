## Summary

Keep open: current main and v2026.6.6 still model Gateway shared-secret auth as one token/password with full operator semantics, while this request asks for a new scoped Gateway principal model and per-agent dispatch authorization that needs maintainer security/product design. The related open issues are complementary, but none fully owns the per-route bearer plus dispatch-lane primitive requested here.

## What This Changes

The issue requests native Gateway support for multiple bearer tokens with per-token scopes plus an optional per-agent dispatch-lane restriction.

## Best Possible Solution

Maintainers should design one canonical Gateway principal model for scoped bearer credentials, service/user identity, and dispatch target limits, then implement it incrementally with config/schema/docs/tests and explicit compatibility guidance.

## Reproduction Assessment

Yes for the capability gap, by source inspection rather than a failing runtime bug: current main and v2026.6.6 expose only singular gateway.auth.token/password and trusted-proxy/device paths, with no gateway.auth.tokens or allowedAgents dispatch binding.

## Solution Assessment

Unclear as proposed: gateway-owned scoped principals look like the right layer, but the exact tokens[] and dispatch.allowedAgents shape needs maintainer security/product design across the existing operator-scope, trusted-proxy, device-token, and related multi-token issues.
