## Summary

Keep open: current main and the latest release still model gateway shared-secret auth as one token/password with full shared-operator semantics, and the closest related issues remain open or were closed stale rather than merged as a fix. This is aligned security work, but it needs maintainer product/security design before implementation.

## What This Changes

The issue requests native scope-bound gateway bearer tokens plus an optional per-token dispatch restriction to limit which agents a token can spawn.

## Best Possible Solution

Maintain a gateway-owned principal/token design discussion that reuses the existing operator-scope machinery, preserves the single-token default, adds docs/doctor/tests for any new config, and decides per-agent dispatch constraints after the scoped-token primitive is accepted.

## Reproduction Assessment

Yes for the source-level gap: current main and the latest release expose one shared token/password path, and docs say shared-secret bearer auth is full operator access. This is not a broken existing contract; it is a requested new least-privilege auth primitive.

## Solution Assessment

Unclear as a final design. Native gateway-owned scoping appears to be the right boundary compared with per-plugin reinvention, but the exact `tokens[]`, scope vocabulary, migration, and `allowedAgents` model need maintainer product/security approval.
