## Summary

Keep open: current main and v2026.6.8 still expose one shared Gateway token/password operator credential, while related threads overlap without replacing this route-bound scoped-token and per-agent dispatch-lane request.

## What This Changes

The issue requests scope-bound gateway bearer tokens plus an optional per-agent dispatch-lane restriction for gateway session spawning.

## Best Possible Solution

Keep this as the focused design item for native scoped Gateway principals and token-bound dispatch restrictions, coordinated with the per-user token and tokenScopes threads without merging their distinct requirements.

## Reproduction Assessment

Yes, at source level. Current main has only a single-token shared-secret auth mode and documents that shared-secret bearer auth receives full operator scope semantics.

## Solution Assessment

Unclear. Native scoped tokens fit the reported security gap, but the exact config shape, scope vocabulary, SecretRef handling, and dispatch-lane semantics require maintainer security/product direction before code.
