## Summary

Keep open: current main and v2026.6.6 still model Gateway shared-secret auth as one full-operator credential, while this asks for a new scoped token/principal model plus token-bound agent dispatch restrictions. The request overlaps related Gateway token work but remains a distinct security/product design item, not a duplicate or automated fix candidate.

## What This Changes

Request native `gateway.auth.tokens[]` entries with per-token scopes and an optional per-token agent dispatch allowlist for Gateway session spawning.

## Best Possible Solution

Design one canonical Gateway principal model covering token lists, per-token operator scopes, SecretRef handling, agent dispatch restrictions, and composition with trusted-proxy/device identities before implementation.

## Reproduction Assessment

No runtime bug repro applies; source inspection gives a high-confidence capability check because current config, schema, resolver, token auth, and shared-secret HTTP scope code all model one full-operator shared secret.

## Solution Assessment

Unclear as a final shape: the request is well-motivated, but maintainers should align it with the existing `operator.*` scope model, timing-safe SecretRef-backed token matching, and current subagent allowlist semantics before accepting the proposed config surface.
