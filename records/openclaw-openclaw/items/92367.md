## Summary

Keep open: current main and v2026.6.6 still model Gateway token/password auth as one shared operator secret, while this issue asks for a new scoped Gateway principal model plus token-bound agent target limits. The related broad team-collaboration issue was closed in favor of this narrower thread, and the open per-user token issue is complementary rather than a replacement.

## What This Changes

The issue requests scoped gateway bearer tokens plus a per-agent dispatch restriction primitive for safer inbound plugin and multi-agent gateway access.

## Best Possible Solution

Maintain one canonical Gateway principal design that supports scoped bearer credentials and optional agent target constraints, then implement it through the core auth/config/schema/docs/doctor path with clear interaction rules for trusted-proxy, device tokens, and existing subagent allowlists.

## Reproduction Assessment

No bug reproduction is applicable, but there is a high-confidence source capability check: current main and v2026.6.6 only support singular gateway.auth.token/password plus trusted-proxy identity, with no gateway.auth.tokens[] or token-bound allowedAgents path.

## Solution Assessment

Unclear as proposed: the gateway is the right layer for auth authority, but the exact tokens[] mode, scope names, wildcard semantics, and dispatch-lane shape need maintainer security/product design before a PR.
