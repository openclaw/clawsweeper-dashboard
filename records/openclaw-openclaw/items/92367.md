## Summary

The least-privilege gap exists on current main and is not fully owned by another open item, but the permanent principal model, authorization vocabulary, and phasing require explicit security and product decisions.

## What This Changes

Add scope-bound gateway service credentials with optional restrictions on which agents each credential may dispatch.

## Best Possible Solution

Introduce one canonical restricted service-principal model that reuses `operator.*` scopes, carries optional allowed-agent constraints in the authenticated principal, and enforces both through shared method and dispatch authorization boundaries.

## Reproduction Assessment

Yes, at the source-contract level. Current main exposes one shared gateway credential with full operator authority, while existing allowed-agent policy is not bound to that credential.

## Solution Assessment

The security objective is sound, but the proposed standalone `tokens[]` mode and new scope vocabulary are not yet the best proven shape; the implementation should extend the existing principal, `operator.*` scope, and centralized dispatch-policy contracts.
