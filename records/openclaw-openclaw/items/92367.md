## Summary

Keep open: current main and the latest release still model shared gateway token/password auth as a single full-operator credential, and the existing subagent `allowAgents` policy is requester-agent config rather than credential-bound dispatch authorization. The related multi-token issues overlap, but none is an exact canonical replacement for this per-route bearer plus per-agent dispatch-lane security primitive.

## What This Changes

Request to add `gateway.auth.tokens[]` with per-token scopes and an optional per-token allowed-agent dispatch restriction for gateway-spawned sessions.

## Best Possible Solution

Consolidate this with the active gateway auth-scope issues into a maintainer-approved design for scoped service/token identities, then implement an opt-in config/schema/auth/docs/test path and token-bound agent target checks if approved.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction of the gap: current config and resolver code only support a single shared token/password path, and docs state shared-secret bearer auth restores full operator scopes. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear as written: native scoped gateway tokens are a plausible security-boundary fix, but the exact config shape, scope vocabulary, and per-agent dispatch binding should be decided alongside the existing multi-token, tokenScopes, and scoped device-token issues.
