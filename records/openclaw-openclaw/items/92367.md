## Summary

Keep open: current main and v2026.6.8 still expose a single shared Gateway bearer/password secret, while this asks for a new scoped Gateway principal model plus token-bound agent target restrictions. The related per-user and broad team-collaboration threads overlap, but they do not replace this narrower service/route scoped-token design item.

## What This Changes

Request scoped `gateway.auth.tokens[]` credentials with per-token scopes and an optional per-token allowed-agents dispatch lane.

## Best Possible Solution

Keep this open as the canonical scoped service/route Gateway-token design thread until maintainers choose one principal model that composes scoped credentials, trusted-proxy, device tokens, and agent-target limits.

## Reproduction Assessment

Yes, as a source-level capability check rather than a runtime bug repro. Current main and v2026.6.8 support singular shared token/password auth plus trusted-proxy/device scopes, with no `gateway.auth.tokens[]` or token-bound `allowedAgents` path.

## Solution Assessment

Unclear as proposed. Gateway auth is the right layer for the authority boundary, but the exact token registry shape, scope vocabulary, wildcard semantics, dispatch-lane model, and upgrade story need maintainer security/product direction first.
