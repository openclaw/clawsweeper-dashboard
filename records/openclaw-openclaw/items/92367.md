## Summary

Keep open: current main still has one shared gateway token/password path and no native per-token scope binding or token-bound dispatch allowlist. The related per-user token issue remains complementary, and the broad team-collaboration issue was closed in favor of this narrower scoped-token thread.

## What This Changes

The issue requests scope-bound gateway bearer tokens plus an optional per-agent dispatch constraint for gateway-triggered session dispatch.

## Best Possible Solution

Keep this as the canonical scoped-token gateway design thread: decide the gateway principal model, scope vocabulary, timing-safe token matching, and whether per-agent dispatch belongs on token entries or a separate dispatch policy before implementation.

## Reproduction Assessment

No bug reproduction is applicable; this is a feature gap. Source inspection gives a high-confidence capability check: current main supports singular gateway.auth.token/password and method scopes, but no gateway.auth.tokens[] or token-bound dispatch policy.

## Solution Assessment

Unclear as proposed. Native scoped tokens are plausible and aligned with the security goal, but the best fix needs maintainer design across existing operator scopes, trusted-proxy/device auth, shared-secret HTTP behavior, and subagent allowlists.
