## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep one limiter chain per normalized provider:** `src/agents/api-request-delay.ts:40`
  - body: Keying limiter state by `${providerKey}:${intervalMs}` creates a new cold limiter whenever config changes the delay for the same provider, so the next request can start immediately instead of honoring the previous last-start time. Keep one limiter per normalized provider and update the active interval without resetting the chain.
  - confidence: 0.88
- **[P2] Use normalized provider lookup for overrides:** `src/agents/api-request-delay.ts:58`
  - body: The per-provider delay lookup indexes `models.providers[trimmedProvider]`, but provider config lookup elsewhere normalizes aliases with `findNormalizedProviderValue`. Alias-keyed configs such as `z.ai`, `kimi-coding`, or `aws-bedrock` can miss the stricter provider delay and fall back to the global value.
  - confidence: 0.86
- **[P2] Refresh the config docs baseline:** `src/config/zod-schema.core.ts:255`
  - body: Adding public model config keys changes the generated config baseline, but the PR does not update `docs/.generated/config-baseline.sha256`. Regenerate the config baseline so changed-gate checks validate the new schema surface.
  - confidence: 0.8
