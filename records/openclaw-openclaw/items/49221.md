## What This Changes

This PR adds global and per-provider model API request-delay config, an agent stream wrapper and tests, docs/help/labels/schema updates, and hooks the delay into embedded runs and compaction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep limiter history per provider:** `src/agents/api-request-delay.ts:37-41`
  - body: Keying the limiter map by `${providerKey}:${intervalMs}` creates a fresh limiter whenever config changes the interval, so the next call for the same provider starts from a cold `lastStart` and can burst immediately after reload. Keep one limiter per normalized provider and update the interval without resetting the chain.
  - confidence: 0.86
- **[P2] Resolve provider delays with normalized IDs:** `src/agents/api-request-delay.ts:56`
  - body: The per-provider override lookup uses an exact `models.providers[trimmedProvider]` key, while provider selection elsewhere normalizes aliases such as `z.ai`/`zai` and `aws-bedrock`/`amazon-bedrock`. With an alias-shaped config key, the stricter provider delay is skipped and the run falls back to the global delay.
  - confidence: 0.84
- **[P2] Regenerate the public config schema:** `src/config/zod-schema.core.ts:238`
  - body: Adding requestDelayMs to the Zod schema without updating `src/config/schema.base.generated.ts` leaves the public/runtime config schema stale, so config UI/schema consumers will not see or validate the new setting consistently. Run the schema generator and include the generated file.
  - confidence: 0.81
- **[P2] Add the required changelog attribution:** `CHANGELOG.md:41`
  - body: The new changelog entry does not include the required `Thanks @...` attribution, so the repo's changelog attribution guard should reject it. Credit the reporter and/or PR author on the same single-line entry.
  - confidence: 0.9
