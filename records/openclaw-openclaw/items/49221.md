Full review comments:

- **[P2] Keep one limiter chain per provider:** `src/agents/api-request-delay.ts:38-41`
  - body: Keying the limiter by `${providerKey}:${intervalMs}` resets the provider's last-start history whenever config changes, so the first request after a delay change can start immediately and bypass the intended minimum interval. Keep one limiter per normalized provider and update the active interval without resetting the chain.
  - confidence: 0.86
- **[P2] Resolve overrides with normalized provider IDs:** `src/agents/api-request-delay.ts:56`
  - body: The override lookup uses the raw trimmed provider key, while current provider config lookup normalizes aliases such as `z.ai`, `kimi-coding`, and `aws-bedrock`. Configs keyed by an alias can skip the stricter per-provider delay and fall back to the global value.
  - confidence: 0.84

  - body: The PR adds Zod/type/help entries but does not update `src/config/schema.base.generated.ts`, so generated-schema consumers and schema checks will not see `models.requestDelayMs` or the per-provider key. Regenerate and include the generated schema artifact.
  - confidence: 0.82
