## Review Findings

Full review comments:

- **[P2] Keep limiter history per provider:** `src/agents/api-request-delay.ts:38-41`
  - body: Keying the limiter map by `${providerKey}:${intervalMs}` creates a fresh limiter whenever the configured interval changes, so the next call for the same provider starts from a cold timestamp and can burst immediately after reload. Keep one limiter per normalized provider and update the interval without resetting the chain.
  - confidence: 0.86
- **[P2] Resolve provider delays with normalized IDs:** `src/agents/api-request-delay.ts:56`
  - body: The per-provider override lookup uses an exact `models.providers[trimmedProvider]` key, while current model config lookup falls back through normalized provider IDs. Alias-shaped keys such as `z.ai` or `aws-bedrock` can skip the stricter per-provider delay and fall back to the global value.
  - confidence: 0.84

  - body: Adding `requestDelayMs` to the Zod schema without updating `src/config/schema.base.generated.ts` leaves public/runtime schema consumers unaware of the new setting. Run the schema generator and include the generated file.
  - confidence: 0.81
