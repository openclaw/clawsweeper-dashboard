## Label Justifications

- `impact:auth-provider`: The issue is about OAuth bearer/token handling being lost between OpenClaw MCP login state and Codex native MCP tool discovery.

## Summary

Keep open: current main and v2026.6.11 still project MCP-native `auth: "oauth"` user servers into native Codex config without resolving OpenClaw's stored OAuth token, while https://github.com/openclaw/openclaw/pull/96120 explicitly leaves the no-`oauth.authProfileId` path unchanged.

## What This Changes

The issue requests extending Codex MCP bearer injection so `auth: "oauth"` servers authenticated with `openclaw mcp login` and no `oauth.authProfileId` can expose their tools in native Codex app-server sessions.

## Best Possible Solution

Keep this as the narrow follow-up for MCP-native OAuth Codex projection, then add either a native-token bearer source with local-only/redacted thread rebinding or a clear unsupported-projection diagnostic after the overlapping OAuth projection decisions settle.

## Reproduction Assessment

Yes from source, though not from a live OAuth server in this read-only review: current main projects an `auth: "oauth"` user MCP server with no headers into Codex as a bare URL while the embedded MCP runtime uses OpenClaw's OAuth provider.

## Solution Assessment

Unclear until maintainer review: reusing the bearer injection pipeline looks like the narrow implementation shape, but the native OAuth token handoff and overlap with active OAuth PRs need an owner decision before automation should code it.
