## Summary

Keep open: current main and the latest release still project MCP-native `auth: "oauth"` user MCP servers into native Codex config without resolving OpenClaw's stored MCP OAuth token, while #96120 explicitly covers only the `oauth.authProfileId` auth-profile path.

## What This Changes

Extend Codex app-server MCP projection so OpenClaw MCP-native `auth: "oauth"` servers authenticated by `openclaw mcp login` either receive refreshed bearer auth or fail with a clear diagnostic.

## Best Possible Solution

Keep this as the scoped native-MCP-OAuth follow-up and implement either local-only refreshed bearer projection with redacted thread rebinding or a deliberate unsupported-projection diagnostic after the overlapping OAuth PR decisions settle.

## Reproduction Assessment

Yes from source: an `auth: "oauth"` user MCP server without headers projects to Codex as a bare URL while embedded MCP transport uses `createMcpOAuthClientProvider`; I did not run a live OAuth server in this read-only review.

## Solution Assessment

Unclear until maintainer/security review: reusing #96120's bearer injection pipeline is the narrow likely fix, but native OAuth token materialization into Codex config needs explicit local-only and redaction guardrails.
