## Summary

Keep open: the PR targets a real systemd owner-selection bug and now has sufficient terminal proof, but the install-side retirement path is not merge-ready because it can stop or block system-owned gateway setups before a safe user-service replacement is established.

## What This Changes

The PR changes systemd gateway scope detection to handle same-name user/system unit conflicts, retires opposite-scope units during install/control paths, and adds systemd regression tests.

## Best Possible Solution

Keep the system-owned restart fix, but make user-install retirement conditional on a live active/enabled conflict and order the handoff so the replacement user unit is staged and preflighted before the system unit is stopped.

## Reproduction Assessment

Yes from source and the linked report: current main selects the user unit before inspecting a same-name system unit, which can restart the stale user service beside the system service. I did not run a live current-main systemd reproduction in this read-only review.

## Solution Assessment

No: the owner-selection direction is good, but the install-side retirement is too broad and ordered too early. The safer fix is to retire only live conflicts and only after the replacement path is staged/preflighted.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Gate system-unit retirement on an active conflict:** `src/daemon/systemd.ts:861`
  - body: This branch treats any same-name system unit file as a conflict and immediately runs `systemctl disable --now`, even when that system unit is inactive or disabled. A stale disabled `/etc/systemd/system/openclaw-gateway.service` would now block a normal user-service install behind sudo despite there being no dueling service to stop, so check active/enabled first or leave disabled files alone.
  - confidence: 0.86
- **[P1] Stage the replacement before stopping the owner:** `src/daemon/systemd.ts:1165`
  - body: `installSystemdService` now disables the existing system unit before `writeSystemdUnit` runs the user-systemd preflight or writes the replacement unit. If the user bus is unavailable, the unit write fails, or activation rejects the generated environment, the previously working system service has already been stopped/disabled and the install exits with no replacement.
  - confidence: 0.84
