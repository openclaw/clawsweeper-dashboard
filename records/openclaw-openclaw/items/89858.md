## Summary

This PR is still necessary because the linked gateway/systemd issue remains open and current main still resolves an existing user unit before considering an active/enabled system unit, so the conflict fix is not already implemented. The patch is directionally correct from the source and tests I inspected, but it should remain open until live systemd upgrade proof is added because the PR body explicitly says real environment proof is pending.

## What This Changes

The PR changes systemd gateway scope selection and activation paths to detect same-name user/system unit conflicts, prefer the active/enabled system unit, retire the opposite-scope unit, and adds regression tests in the systemd daemon test file.

## Best Possible Solution

Land the conflict-resolution fix after live systemd upgrade proof shows a stale user/system unit pair is retired without breaking the managed gateway, then let the linked issue close through the PR.

## Solution Assessment

Yes, this looks like the right owner boundary for the bug: the systemd service resolver and activation/control paths are where the stale opposite-scope unit is selected and should be retired. The remaining question is upgrade proof, not a different code ownership path.
