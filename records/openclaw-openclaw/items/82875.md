## Summary

Close as duplicate/superseded by the still-open canonical tracker at https://github.com/openclaw/openclaw/issues/44925. That issue already tracks the same separation between subagent completion, parent/main transcript state, generated visible reply, and confirmed external delivery, including the exact “internal completion exists but no user-visible message” variant reported here.

## What This Changes

The issue reports that a completed subagent can leave parent completion delivery pending after a gateway request timeout even though the requester/main session generated the user-facing reply.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/44925 as the canonical tracker for durable async subagent completion delivery, and treat this Discord timeout trace as supporting evidence for that state-machine work.

## Reproduction Assessment

Yes, source-level evidence plus the issue logs give a high-confidence path for the delivery-state problem: a completion direct `agent` handoff can time out, return `delivered: false`, and leave lifecycle retry/pending bookkeeping rather than confirming external delivery from the generated requester reply. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

Yes, closing this duplicate is the best triage outcome because the remaining fix should be handled once in the open canonical durable-delivery tracker. A separate narrow patch here risks splitting the state model from the broader single-delivery, dedupe, timeout, and user-visible failure semantics already tracked there.
