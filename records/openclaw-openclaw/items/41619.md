## Summary

Keep open: current main still exposes the Gemini CLI OAuth path and still has unresolved failure surfaces matching the report. Token exchange and project lookup use guarded fetch without trusted env-proxy mode, the browser handoff still happens before the localhost callback listener is started, and failures collapse to the same generic OAuth help. An open related PR covers the proxy-related AbortError, but it is not merged and does not address the listener-readiness concern.

## What This Changes

The issue asks OpenClaw to make Google Gemini CLI OAuth complete successfully from onboarding/auth on macOS/Homebrew instead of aborting after the browser OAuth flow.

## Best Possible Solution

Keep the issue open until current main has a narrow Google OAuth fix that starts the callback listener before browser handoff, respects trusted env proxy settings for Google OAuth endpoint fetches, preserves manual fallback, and adds focused regression tests plus clearer diagnostics.

## Reproduction Assessment

No for a high-confidence live reproduction: I did not run real Google OAuth. Source inspection gives a medium-confidence reproduction path for the token-exchange AbortError behind proxy-required networks and the localhost listener-ordering race.

## Solution Assessment

Yes, a narrow Google plugin/auth-runtime repair is the best direction. Current main is not enough, and the related open proxy PR should either be split/landed narrowly or replaced with a focused fix that also covers listener readiness.
