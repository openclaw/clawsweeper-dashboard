## Summary

Keep open: current main and the latest shipped tag still expose the Gemini CLI OAuth setup path, still open the browser before the localhost callback listener is started, and still present token/OAuth failures as the generic Gemini CLI access help from the report.

## What This Changes

The issue reports that Google Gemini CLI OAuth fails during onboarding on macOS/Homebrew with `AbortError: This operation was aborted` even for an account believed to have Gemini CLI access.

## Best Possible Solution

Keep this open for the Google plugin/auth owner to live-reproduce on current main or `v2026.5.2`, then make the localhost listener ready before browser handoff and separate callback, token-timeout, project, and account-access diagnostics in the provider-owned OAuth flow.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a current-main path to investigate: browser handoff happens before the callback listener is started, and token exchange still goes through a timeout-capable fetch path surfaced as generic OAuth help.

## Solution Assessment

Unclear for the exact code change until a live OAuth run confirms the failing branch. The maintainable boundary is still narrow: `extensions/google` plus the shared callback helper, not a broad core config migration.
