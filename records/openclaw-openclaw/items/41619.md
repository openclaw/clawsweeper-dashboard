## Summary

Keep open: current main appears to have lost the issue-specific macOS manual-paste OAuth fallback, so the reported Gemini CLI OAuth callback failure is still actionable.

## What This Changes

The issue asks OpenClaw to complete `google-gemini-cli` OAuth on macOS instead of aborting after the browser authorization flow.

## Best Possible Solution

Restore a provider-owned macOS-safe Gemini CLI OAuth path, preferably by reinstating the darwin manual-paste fallback or making the localhost listener reliably ready before opening the browser, with regression coverage in the Google plugin.

## Reproduction Assessment

Yes, at source level: on current main, a local macOS run does not enter manual mode and the browser is opened before the callback wait path is started. I did not live-run the external Google OAuth flow on macOS.

## Solution Assessment

No, current main is not the best solution because it dropped a prior fix made specifically for this macOS callback failure. The narrow maintainable repair is to restore that provider-local behavior or make callback startup ordering robust, then test the flow selection.
