## Summary

Keep open. Current main still exposes the bundled Google Gemini CLI OAuth setup path, still opens the browser before starting the localhost callback listener, still sends token exchange through the guarded 10s fetch path that can surface `AbortError`, and still collapses OAuth failures into the same generic Gemini CLI access help shown in the report. The latest shipped tag has the same OAuth-flow ordering, so this is not fixed on main or in release.

## What This Changes

The issue reports that Google Gemini CLI OAuth fails during onboarding/model auth on macOS/Homebrew with `AbortError` even when the Google account should have Gemini access.

## Best Possible Solution

Keep this open for the Google plugin/auth owner. Reproduce Gemini CLI OAuth on current main or `v2026.4.27`, then make the localhost callback listener ready before browser handoff and improve token-exchange timeout/network/account diagnostics, with focused regression coverage for listener readiness and `AbortError` handling.

## Reproduction Assessment

Unclear for the exact live account-side failure: this read-only review did not exercise Google OAuth. There is a high-confidence static reproduction target for the listener-order race because `openUrl` is called before `server.listen`, and the 10s token exchange path can still surface `AbortError`.

## Solution Assessment

Yes for the likely fix direction: the narrowest maintainable path is in the bundled Google OAuth flow and, if needed, the shared OAuth callback helper, with diagnostics kept provider-owned. A broad core migration or closing as stale would miss the active shipped auth path.
