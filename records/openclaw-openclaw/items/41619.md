## Summary

Keep open: current main and the latest release still ship source-visible Gemini CLI OAuth failure paths matching this issue and follow-up comments. The proxy-specific part is better documented in https://github.com/openclaw/openclaw/issues/46184 and attempted by https://github.com/openclaw/openclaw/pull/47518, but that PR remains open/broad and the localhost listener-readiness concern is not resolved on main.

## What This Changes

The issue reports that Google Gemini CLI OAuth aborts during OpenClaw onboarding/model auth even when the Google account has Gemini CLI access.

## Best Possible Solution

Land a focused current-main Google auth/runtime fix that starts the callback listener before browser handoff, routes only known Google OAuth calls through target-aware trusted env-proxy handling, preserves manual fallback, and adds regression coverage plus clearer diagnostics.

## Reproduction Assessment

No high-confidence live reproduction was established because I did not run real Google OAuth. Source inspection gives a medium-confidence path for both reported failure classes: the local listener starts after browser launch, and OAuth fetches remain in strict SSRF mode without target-aware env-proxy opt-in.

## Solution Assessment

Yes, a narrow Google plugin/auth-runtime repair is the best direction. Merging the broad related PR is not the best path because it touches unrelated Discord, LINE, Nostr, and SDK surfaces and already has maintainer/security review concerns.
