## Summary

Current main and shipped release notes cover the reported Gemini CLI OAuth abort class: the provider now handles personal OAuth without forcing Code Assist project discovery, and related Gemini CLI OAuth failures were closed after the same provider-local fixes landed.

## What This Changes

The issue reports Gemini CLI OAuth failing during OpenClaw onboarding/auth on macOS v2026.3.8 despite the Google account having Gemini CLI access.

## Best Possible Solution

Keep the shipped provider-local Gemini CLI OAuth split and ask any future failures to include current version, Gemini CLI auth mode, install layout, and redacted auth logs.

## Reproduction Assessment

No live current-main reproduction was established because the flow requires an external Google account and OAuth callback. The old failure path is source-reproducible from related reports: token exchange succeeded, then personal OAuth accounts were forced through Code Assist project discovery.

## Solution Assessment

Yes. The current solution is provider-local and narrow: detect Gemini CLI personal OAuth, skip loadCodeAssist/onboardUser only for that mode, and preserve the workspace Code Assist path plus credential-discovery fixes.
