## Summary

Keep open: current main still exposes the bundled Gemini CLI OAuth login path, and source inspection still shows the browser handoff happening before the localhost callback listener is started while failures collapse into the generic OAuth help shown in the report.

## What This Changes

The issue asks for Google Gemini CLI OAuth login to complete for accounts with Gemini access instead of aborting during the auth/token-exchange flow.

## Best Possible Solution

Make the Gemini CLI localhost callback listener ready before opening the browser, keep the manual fallback, and add focused regression coverage plus clearer token/project failure diagnostics without changing OAuth scopes or provider policy.

## Reproduction Assessment

No for the exact live account-side failure because this review did not run Google OAuth. Source inspection on current main gives a concrete reproduction path for the listener-ordering concern and the unchanged generic failure surface.

## Solution Assessment

Yes for the likely fix direction: the narrowest maintainable repair belongs in the Google plugin OAuth flow and, if needed, an additive shared callback helper. A broader provider redesign is not needed for this report.

## Repair Work Prompt

Fix the Google Gemini CLI OAuth login regression tracked by https://github.com/openclaw/openclaw/issues/41619. On current main, `extensions/google/oauth.ts` opens the browser before the localhost:8085 callback listener is started by `waitForLocalCallback`; refactor the Google flow, and add an additive shared helper only if needed, so the listener is ready before browser handoff. Preserve remote/manual mode, EADDRINUSE/manual fallback, OAuth scopes, redirect URI, client credential discovery, and the unofficial-account-risk warning. Improve or preserve specific diagnostics for token exchange and Code Assist project discovery failures instead of hiding all failures behind generic access advice. Add focused regression tests in the Google OAuth tests and shared provider-auth runtime tests. Add a user-facing CHANGELOG.md fix entry. Do not change unrelated providers or model defaults unless a test proves the shared callback helper requires an additive update.
