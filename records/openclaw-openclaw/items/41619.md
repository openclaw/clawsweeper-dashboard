## Summary

Keep open: current main still has a plausible Gemini CLI OAuth callback-listener failure path, and the related merged fixes cover different `loadCodeAssist`/metadata failures rather than this localhost callback report.

## What This Changes

Google Gemini CLI OAuth fails during onboarding on macOS with `Gemini CLI OAuth failed` / `AbortError` despite the account having Gemini CLI access.

## Best Possible Solution

Make Gemini CLI OAuth bind and confirm the localhost callback listener before opening the browser, while preserving manual paste fallback for remote/WSL2 or listener failure.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the local macOS path opens the browser before starting the callback listener, matching the user comment about no process listening on port 8085.

## Solution Assessment

No, closing this as already fixed would conflate it with separate closed `loadCodeAssist` 400 fixes. The narrow maintainable fix is to start the callback server before browser launch and cover that ordering in tests.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/41619. Current `extensions/google/oauth.ts` calls `ctx.openUrl(authUrl)` before starting `waitForLocalCallback`, so on local macOS the Google redirect can hit `localhost:8085` before OpenClaw is listening. In the Google plugin and/or shared provider-auth runtime helper, make the localhost callback server bind and report readiness before opening the browser; preserve remote/WSL2 manual paste behavior and fall back to manual paste when listener setup fails. Do not change Cloud Code Assist metadata, model catalog behavior, or unrelated provider auth. Add regression coverage that proves non-remote login starts the listener before invoking `openUrl`, plus bind-failure fallback coverage if the helper changes. Add a single `CHANGELOG.md` fix entry if this user-facing fix needs one.
