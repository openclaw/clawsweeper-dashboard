## Summary

Keep open: this is a plausible gateway auth hardening PR, but it has the protected `maintainer` label, lacks required external-contributor real behavior proof, and needs maintainer acceptance for a backward-incompatible trusted-proxy boundary change.

## What This Changes

The branch adds gateway auth/network logic and regression tests to reject trusted-proxy requests whose immediate peer is a non-loopback local interface address.

## Best Possible Solution

Have the gateway/security owner decide whether non-loopback host-interface peers should be rejected, then update trusted-proxy docs/troubleshooting and require redacted after-fix runtime proof before merge.

## Reproduction Assessment

Yes by source inspection: current `main` accepts a non-loopback peer once its `remoteAddress` matches `gateway.trustedProxies`, so a host interface address configured there reaches the trusted-proxy header checks. I did not run a live gateway replay in this read-only review.

## Solution Assessment

Unclear as merge-ready: the code direction is narrow and plausibly improves the trust boundary, but the backward-incompatible behavior needs maintainer sign-off, updated docs, and real behavior proof before it is the best complete solution.
