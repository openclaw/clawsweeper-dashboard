## Summary

Keep open: the PR addresses a real Feishu card-action data-loss problem, but the patch is incomplete against current main because live callbacks are parsed before the handler and the parser/decoder path still drops these fields. The contributor addressed the earlier handler-test/name-only feedback, but the remaining parser, fallback, lifecycle-test, and changelog gaps are concrete merge blockers.

## What This Changes

The PR adds optional Feishu card action fields to `FeishuCardActionEvent`, serializes form/input/select fields in `handleFeishuCardAction`, and adds direct handler tests for input/form/select/name-only cases.

## Best Possible Solution

Implement one Feishu-owned callback-field pipeline end to end: preserve the fields at raw parsing, expose them through typed event and decoder/fallback surfaces, keep legacy button command extraction compatible, add regression coverage, and add a single-line changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to route a raw `card.action.trigger` payload containing `action.form_value` or `action.option` through `parseFeishuCardActionEventPayload`; current main returns only `action.value` and `action.tag`, so the handler cannot receive the submitted field.

## Solution Assessment

No. The handler-only approach is not the best complete fix because live callbacks are normalized before the handler runs; the narrow maintainable fix should preserve and test these fields from parser through decoder/fallback and dispatch.
