The PR extends Feishu card-action typing, handler payload construction, and handler tests to forward `form_value`, `input_value`, `name`, `option`, and `options` as structured JSON when those fields are present.

## Best Possible Solution

Land one canonical Feishu plugin patch that preserves supported card-action sibling fields at the parser boundary, carries them through fallback/handler dispatch, keeps legacy command extraction compatible, covers the chosen payload contract with tests, and includes redacted live Feishu proof.

Yes, source-level: a raw Feishu `card.action.trigger` payload with sibling form/input/select fields is normalized to only `action.value` and `action.tag`, then fallback dispatch serializes only `action.value`. I did not run a live Feishu callback.

## Solution Assessment

No. The handler-only branch is incomplete; the maintainable fix must start at `parseFeishuCardActionEventPayload`, preserve legacy `value.text`/`value.command` behavior, and settle whether this broader contract or the narrower overlapping PR is canonical.
