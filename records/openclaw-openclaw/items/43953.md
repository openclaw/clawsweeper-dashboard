## Summary

Keep open. The PR targets a real Feishu card-action data-loss problem, but the patch is incomplete for the production callback path because live `card.action.trigger` payloads are normalized before the handler and current main still drops the sibling form/select/input fields there.

## What This Changes

The PR extends Feishu card-action typing and handler tests so `form_value`, `input_value`, `name`, `option`, and `options` can be serialized into a synthetic message while preserving simple button extraction.

## Best Possible Solution

Implement one Feishu-owned callback-field pipeline end to end: preserve sibling fields at the raw parser boundary, expose them through the typed event and decoder/fallback surfaces, keep existing button command extraction compatible, add raw callback regression tests, and include the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to route a raw `card.action.trigger` payload containing `action.form_value` through `parseFeishuCardActionEventPayload`; current main returns only `action.value` and `action.tag`, so the handler cannot receive the submitted field.

## Solution Assessment

No. The handler-only approach is not the best complete fix because live callbacks are normalized before the handler runs; the safer narrow fix preserves and tests these fields from parser through decoder/fallback and dispatch.
