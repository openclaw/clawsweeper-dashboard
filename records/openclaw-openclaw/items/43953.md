## Summary

Keep open: the PR addresses a real Feishu callback-field gap, but the patch still misses the production parser boundary where live card-action callbacks are normalized, and it lacks real Feishu after-fix proof.

## What This Changes

The PR extends Feishu card-action handling and tests to forward input_value, form_value, option/options, and name data for form-like card actions while preserving plain button command extraction.

## Best Possible Solution

Select or update one canonical Feishu plugin patch that preserves form/input/select fields at the parser boundary, carries them through fallback and handler dispatch, intentionally covers input_value/name, adds parser-to-handler tests, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes, source-level: a raw card.action.trigger payload with sibling form/input/select fields is normalized to action.value and action.tag only, and fallback dispatch then serializes only action.value. I did not run a live Feishu replay in this read-only review.

## Solution Assessment

No. The handler-level branch is useful, but the durable fix has to preserve the fields in parseFeishuCardActionEventPayload and align parser, fallback, decoder, and handler behavior with the selected Feishu callback payload contract.
