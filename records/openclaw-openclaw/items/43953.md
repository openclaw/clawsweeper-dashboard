## Summary

Keep open: the PR targets a real Feishu card-action data-loss gap, but the patch is incomplete because live callbacks are normalized before this handler and the parser still drops the new fields.

## What This Changes

The PR extends Feishu card-action typing and handler tests so form/input/select action fields are serialized into a synthetic JSON message while preserving plain button command extraction.

## Best Possible Solution

Implement one Feishu-owned callback-field pipeline end to end: preserve the fields at raw parsing, expose them through typed event and decoder/fallback surfaces, keep legacy button command extraction compatible, add lifecycle/unit regression tests, and add a single-line changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to route a raw `card.action.trigger` payload containing `action.form_value` or `action.option` through `parseFeishuCardActionEventPayload`; current main returns only `action.value` and `action.tag`, so the handler cannot receive the submitted field.

## Solution Assessment

No. The handler-only approach is not the best complete fix because live callbacks are normalized before the handler runs; the narrow maintainable fix should preserve and test these fields from parser through decoder/fallback and dispatch.
