## Summary

Keep open: current main still lacks this Feishu callback-field support and this PR carries useful input_value/name scope, but the patch is not mergeable as-is because production callbacks drop those fields before the changed handler branch can see them and no live Feishu proof is attached.

## What This Changes

The PR extends Feishu card-action handling to add action.input_value/name/form_value/option/options, emit structured JSON for form-like callbacks, and add handler unit tests.

## Best Possible Solution

Choose one Feishu callback contract that preserves raw fields at the parser boundary, carries form_value/option/options plus input_value/name through fallback and handler behavior, adds parser-to-handler regression coverage, and includes redacted live Feishu proof before merge.

## Reproduction Assessment

Yes, source-level. A raw card.action.trigger payload with sibling form/select/input fields is normalized to action.value and action.tag only before the handler runs, so this PR's new branch cannot receive those fields in production.

## Solution Assessment

No. The handler-level direction is useful, but the narrow maintainable fix must preserve the fields in parseFeishuCardActionEventPayload and carry them through fallback/handler behavior consistently with the selected payload contract.
