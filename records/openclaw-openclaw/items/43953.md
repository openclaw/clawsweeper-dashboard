## Summary

Keep open. The PR addresses a valid Feishu card-action data-loss problem, but the patch is incomplete against the production callback path: it adds handler-side fields and direct handler tests, while current main still normalizes raw `card.action.trigger` payloads through a parser that keeps only `action.value` and `action.tag`. The branch is also stale/dirty against current main and overlaps open issue #43202 and open PR #42908, with the newer related attempts closed unmerged.

## What This Changes

The PR extends Feishu card-action event typing and handler tests so `form_value`, `input_value`, `name`, `option`, and `options` can be serialized into a structured synthetic message while preserving plain button command extraction.

## Best Possible Solution

Implement one Feishu-owned callback-field pipeline end to end: preserve form/input/select sibling fields at the raw parser boundary, expose them through the typed event and decoder/fallback surfaces, keep existing plain button command extraction compatible, add raw `card.action.trigger` regression tests, and include the required user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by routing a raw `card.action.trigger` payload containing `action.form_value` through `parseFeishuCardActionEventPayload`; current main returns only `action.value` and `action.tag`, so the handler cannot receive the submitted field.

## Solution Assessment

No. The handler-only approach is not the best complete fix because live callbacks are normalized before the handler runs; the safer solution is to preserve and test these fields from parser through decoder/fallback and dispatch.
