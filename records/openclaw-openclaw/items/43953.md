## Summary

Keep open: this PR targets a real Feishu card-action data-loss path, but the current branch only updates the handler and direct handler tests while production callbacks still pass through a parser that strips the sibling fields before the handler can see them.

## What This Changes

The PR adds optional Feishu card-action fields to the handler type, emits structured JSON when those fields are present, and adds direct handler unit tests for input, form, and select cases.

## Best Possible Solution

Preserve the fields in the raw Feishu card-action parser, carry them through decoder/fallback/handler using a maintainer-approved payload shape, and add parser-to-handler regression tests before merging one canonical PR.

## Reproduction Assessment

Yes, source-level. A raw `card.action.trigger` payload with sibling `action.form_value`, `action.input_value`, `action.option`, or `action.options` is normalized to an event containing only `action.value` and `action.tag`, so the PR's new handler branch is bypassed in production.

## Solution Assessment

No. The patch is pointed at the right user problem, but the current implementation is not the best fix because it starts at the handler after production parsing has already discarded the fields.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve card fields before the handler:** `extensions/feishu/src/card-action.ts:41-44`
  - body: This new branch depends on `event.action.form_value`/`input_value`/`option(s)` reaching `handleFeishuCardAction`, but real `card.action.trigger` callbacks are first normalized by `parseFeishuCardActionEventPayload`, which rebuilds `action` with only `value` and `tag`. The direct unit tests construct `FeishuCardActionEvent` by hand, so the PR appears to work while live Feishu form/select/input fields are still dropped before this code can run. Preserve these fields in the parser and cover the parser-to-handler path.
