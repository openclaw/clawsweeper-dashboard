## Summary

Keep open. Current main still drops sibling Feishu card-action fields at the raw parser boundary, and this PR’s handler-only branch would still miss live callbacks because `card.action.trigger` normalizes callbacks to `value`/`tag` before dispatch; the repair also needs the required changelog entry.

## What This Changes

This PR extends Feishu card-action typing and handler tests to serialize `form_value`, `input_value`, `name`, `option`, and `options` into synthetic messages while preserving simple button command extraction.

## Best Possible Solution

Implement one Feishu-owned callback-field pipeline end to end: preserve the fields at raw callback parsing, expose them through the typed event and decoder/fallback surfaces, keep legacy button command extraction compatible, add raw lifecycle regression tests, and add a single-line changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to route a raw `card.action.trigger` payload containing `action.form_value` or `action.option` through `parseFeishuCardActionEventPayload`; current main returns only `action.value` and `action.tag`, so the handler cannot receive the submitted field.

## Solution Assessment

No. The handler-only approach is not the best complete fix because live callbacks are normalized before the handler runs; the narrow maintainable fix should preserve and test these fields from parser through decoder/fallback and dispatch.
