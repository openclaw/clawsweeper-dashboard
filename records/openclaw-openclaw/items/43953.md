## Summary

Keep open: the Feishu data-loss problem is real on current main, but this PR still patches the handler after the live callback parser has already dropped the requested sibling fields, and it lacks real Feishu after-fix proof.

## What This Changes

The PR adds optional Feishu card-action fields to the handler event type, emits a structured JSON synthetic-message payload for form/input/select callbacks, and adds handler-level unit tests.

## Best Possible Solution

Choose one canonical Feishu card-action callback-field fix that preserves raw fields at the parser boundary, carries them through decoder/fallback/handler behavior with regression coverage, includes input_value/name scope deliberately, and has redacted live Feishu proof before merge.

## Reproduction Assessment

Yes. Source inspection shows a raw card.action.trigger payload with sibling form/input/select fields is normalized to action.value and action.tag only before the handler runs, so this PR's new branch cannot receive those fields in production.

## Solution Assessment

No. The direction is useful, but a handler-centered patch is not the best fix because the live parser must preserve the fields before the handler and fallback path can serialize them consistently.
