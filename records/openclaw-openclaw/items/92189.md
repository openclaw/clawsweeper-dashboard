## Summary

Current main and v2026.6.6 still propagate the requester’s effective tool restrictions into the spawned child and apply them after the target agent’s policy. The report identifies a real regression path, but changing this security boundary requires an explicit decision about which upstream restrictions are delegable versus mandatory.

## What This Changes

Correct sessions_spawn policy composition so a delegated target agent can use its configured tools without escaping genuine upstream authorization constraints.

## Best Possible Solution

Define a closed policy model that carries only non-delegable authorization ceilings across the spawn boundary while allowing target-agent policy to control intentionally delegated capabilities, with regression tests for both specialist delegation and nested anti-escalation.

## Reproduction Assessment

Yes, at source level with high confidence: current main exports the requester’s effective allow/deny policy into sessions_spawn, persists it on the child, and reapplies it after target policy. The reported requester-deny and target-allow configuration directly exercises that path.

## Solution Assessment

The reported diagnosis is directionally correct, but deleting all inherited policy would not be the safest fix. Maintainers should first classify which policy layers are delegable capability choices and which are mandatory authorization ceilings.
