The issue reports that one isolated-session `agentTurn` cron job can saturate the Gateway event loop, making Telegram/Discord and readiness probes unresponsive for minutes.

Keep this as the cron-specific Gateway responsiveness tracker and design a bounded workload-isolation or scheduling fix that preserves `/readyz` and chat delivery during one isolated cron agent turn, while landing adjacent setup-timeout fixes only as partial mitigations.
