## Label Justifications

- `P1`: A single scheduled cron agent turn reportedly makes core readiness and chat transports unresponsive for minutes, which is an urgent Gateway availability failure.
- `impact:crash-loop`: The issue is about process-level availability and event-loop saturation, which fits the crash/hang/restart-loop impact class.

## Summary

Keep open: the report describes a credible Gateway availability failure, and current main still routes isolated cron `agentTurn` work through the Gateway cron service and embedded agent runner rather than a separately isolated runtime that would clearly prevent event-loop saturation.

## What This Changes

The issue reports that one isolated `agentTurn` cron run can saturate the Gateway event loop, making `/readyz` and chat transports unresponsive for minutes.

## Best Possible Solution

Keep this issue open and make isolated cron agent turns unable to monopolize the Gateway event loop, with proof that `/readyz` and at least one chat transport stay responsive during a real or high-fidelity cron agent run.

## Reproduction Assessment

Yes for the source path, but not for the full live symptom: current main still executes isolated `agentTurn` cron jobs through the Gateway-owned cron and embedded runner path. I did not run a live Gateway plus Telegram/Discord repro to confirm the reported multi-minute transport outage on current main.

## Solution Assessment

No current solution fully solves this issue. Current main has enqueue-and-poll manual runs, cron concurrency limits, and cron-nested lanes, but the best fix needs to bound or isolate the actual cron agent-turn runtime and prove Gateway responsiveness under load.
