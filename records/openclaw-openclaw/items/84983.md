## What This Changes

The issue reports gateway event-loop saturation during isolated cron agent-turn execution, making chat transports and readiness checks unresponsive.

## Best Possible Solution

Keep this open and design a cron/gateway runtime-boundary fix that prevents one isolated cron agent turn from monopolizing the Gateway event loop, with live proof for `/readyz` and at least one chat transport during the run.

## Reproduction Assessment

Yes for source-level reproduction of the risky path, but not for a live current-main saturation run. Current main still routes isolated cron agent turns through Gateway cron, while the issue supplies concrete production/runtime metrics that need live validation.

## Solution Assessment

No current implementation is the full solution. Existing enqueue, concurrency, startup-deferral, and watchdog work are useful mitigations, but the best fix needs a maintainer-owned runtime isolation or budgeting decision with transport/readiness proof.
