## Best Possible Solution

Keep this open and design/prove cron isolated agent-turn execution so one scheduled run cannot monopolize the Gateway event loop; proof should cover `/readyz` and at least one chat transport during a real or high-fidelity cron run.

## Solution Assessment

No current solution fully solves the issue. Current main enqueues manual cron runs and limits cron/nested concurrency, but the maintainable fix needs to bound or isolate the actual cron agent-turn work and prove Gateway responsiveness during the run.
