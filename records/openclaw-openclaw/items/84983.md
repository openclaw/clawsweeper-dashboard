## Label Justifications

- `P1`: A single scheduled cron agent turn reportedly makes the core Gateway, readiness checks, and chat transports unresponsive for minutes.
- `impact:crash-loop`: The reported failure is process-level availability loss from event-loop saturation even though the Gateway process may remain alive.
- `impact:message-loss`: Telegram and Discord stop responding during the saturated cron run, so user-visible channel delivery can be delayed or suppressed.

## Summary

Keep open: current main still runs isolated cron agent turns through the Gateway-owned cron service and embedded/CLI agent runner, while the reported failure is gateway-wide event-loop saturation during that work. Existing cron queue, watchdog, and `cron-nested` lane mitigations do not prove `/readyz` or chat transports stay responsive during a single heavy scheduled agent turn.

## What This Changes

The issue reports that an isolated-session native cron `agentTurn` can saturate the Gateway event loop and make Telegram/Discord transports unresponsive for minutes.

## Best Possible Solution

Fix the native isolated cron agent-turn execution path so one scheduled run cannot monopolize the Gateway event loop, then prove `/readyz` and at least one real chat transport remain responsive during the run.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. The report has concrete steps and metrics, and source inspection confirms the isolated cron agent-turn path still runs inside the Gateway process through the cron runner, but live `/readyz` and transport proof is still missing.

## Solution Assessment

No current solution fully addresses the issue. Current main has enqueueing, watchdog, and `cron-nested` lane mitigations, but the maintainable fix must bound or isolate the actual agent-turn work and prove Gateway responsiveness during execution.
