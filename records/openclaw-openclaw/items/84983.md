## What This Changes

The issue asks OpenClaw to prevent isolated `agentTurn` cron jobs from saturating the Gateway event loop and making `/readyz` plus chat transports unresponsive.

## Best Possible Solution

Keep this as the cron-specific tracker and fix the isolated cron agent-turn execution path so one scheduled run cannot monopolize the Gateway event loop; the fix should include live or high-fidelity proof that `/readyz` and at least one chat transport stay responsive during the run.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. The issue gives concrete steps and field metrics, and current source still runs isolated cron agent turns through the Gateway cron path, so the source path remains reproducible but the exact runtime saturation needs live proof.

## Solution Assessment

No current implementation fully solves this issue. Current main has enqueueing, watchdogs, setup-timeout restart recovery, and diagnostics, but the best fix needs workload isolation, backpressure, or scheduling policy that protects Gateway readiness and transports while cron agent work runs.
