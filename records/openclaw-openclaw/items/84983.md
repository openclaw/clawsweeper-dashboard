## Label Justifications

- `P1`: A single scheduled cron agent turn reportedly makes core chat transports and readiness checks unresponsive for minutes, affecting real gateway operation.
- `impact:crash-loop`: The issue is about process-level availability and event-loop saturation rather than a localized cron result error.

## Summary

Keep open: the report describes a real availability failure mode that current main does not clearly eliminate. Current cron.run is already enqueue-and-poll, but isolated cron agent turns still execute through the Gateway-owned cron path and embedded/CLI runner surfaces, so the remaining problem is single-job runtime saturation rather than the manual RPC waiting behavior.

## What This Changes

The issue asks for isolated native cron agent turns to stop monopolizing the Gateway event loop and making chat transports unresponsive during a scheduled run.

## Best Possible Solution

Keep the issue open and fix the cron agent-turn execution path so one isolated scheduled run cannot monopolize the Gateway event loop; the fix should include /readyz and chat-transport responsiveness proof during a real or high-fidelity cron agent run.

## Reproduction Assessment

Unclear for a fully live current-main reproduction: the report gives concrete steps and metrics, and current source still runs isolated cron agent turns through the Gateway process. I did not run a live Gateway plus chat-transport repro in this read-only review.

## Solution Assessment

No current solution fully solves the issue. Current main enqueues cron.run and limits cron concurrency, but the best fix needs to bound or move the actual isolated agent-turn work off the Gateway hot path and prove /readyz plus chat transports stay responsive.
