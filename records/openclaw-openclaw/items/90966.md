## Summary

Keep open; the Feishu health gap is real, but this patch should not merge as-is because it publishes WebSocket connectivity before SDK readiness, treats app/webhook events as transport liveness, and only provides unit-test proof after the patch.

## What This Changes

The PR wires a Feishu account status sink through the channel monitor, publishes connected/lastEventAt/lastTransportActivityAt from WebSocket, webhook, and message paths, and adds transport-status unit tests.

## Best Possible Solution

Publish Feishu connected/disconnected state from the SDK lifecycle callbacks, keep lastTransportActivityAt limited to real recurring transport proof, and require redacted patched-runtime Feishu proof before merge.

## Reproduction Assessment

Yes, source inspection reproduces the current gap: Feishu startup only sets port status while the gateway health policy depends on connected and transport-activity fields for stale/disconnected decisions. I did not run a live Feishu account in this read-only review.

## Solution Assessment

No; the status-sink direction is plausible, but this implementation is not the best fix because it uses wsClient.start as a readiness signal and feeds app-level events into transport liveness. The safer fix is to use the SDK lifecycle callbacks and reserve transport timestamps for real transport probes or heartbeats.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Publish WS status from SDK readiness callbacks:** `extensions/feishu/src/monitor.transport.ts:264-274`
  - body: The Feishu SDK does not make `start()` a handshake barrier: in 1.66.0 it sets the dispatcher, calls `reConnect(true)`, and resolves; `onReady` is the first-handshake callback. Publishing `connected: true` here can mark slow or failed connections healthy, so wire `createFeishuWSClient` with `onReady`/`onReconnected`/`onReconnecting`/`onError` and publish status from those callbacks instead.
  - confidence: 0.93
- **[P2] Keep app events out of transport liveness:** `extensions/feishu/src/monitor.message-handler.ts:330-333`
  - body: `lastTransportActivityAt` drives stale-socket restarts after 30 minutes, and the shared policy treats it as transport/heartbeat/poll proof rather than ordinary app traffic. Updating it for Feishu messages means a healthy quiet account can be restarted once the last app event ages out; keep app events as `lastEventAt` or inbound activity unless there is recurring transport proof.
  - confidence: 0.86
