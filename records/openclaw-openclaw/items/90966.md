## Summary

The PR threads a Feishu account status sink through channel startup and monitor paths, publishes connected/event status from Feishu transport callbacks and inbound events, and adds focused transport-status tests.

## Best Possible Solution

Keep the status-sink plumbing, but fix the Feishu liveness contract before merge: use an SDK-supported WebSocket liveness signal, avoid stale-socket timers for event-driven webhooks without a real cadence, and add redacted patched-runtime proof.

## Solution Assessment

No. The status-sink boundary is the right owner layer, but this version should not rely on callbacks whose watchdog is not armed or treat event-driven webhook quiet time as socket staleness without a maintainer-approved liveness contract.

- **[P1] Arm SDK liveness before relying on reconnect callbacks:** `extensions/feishu/src/monitor.transport.ts:270-274`
  - body: This new status path relies on `onReconnecting`/`onError` to mark a dead WebSocket disconnected, but the pinned Feishu SDK only arms its pong watchdog from `wsConfig.pingTimeout`.   - confidence: 0.88

  - body: After one successful Feishu webhook delivery, this records `lastTransportActivityAt`; the gateway then treats no further traffic for the stale threshold as `stale-socket`. Quiet webhook accounts are valid without a recurring transport cadence, so this can restart healthy webhook servers after ordinary idle periods.
  - confidence: 0.9
