## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Publish WS status from readiness callbacks:** `extensions/feishu/src/monitor.transport.ts:269-276`
  - body: The pinned Feishu SDK does not make `start()` a handshake barrier: it assigns the dispatcher and calls `reConnect(true)` without awaiting it, while `onReady`/`onReconnected` are the connection callbacks. Publishing `connected: true` here can mark a slow or failed first handshake healthy, so wire status to SDK lifecycle callbacks instead.
  - confidence: 0.93
- **[P1] Avoid stale-socket timestamps without recurring proof:** `extensions/feishu/src/monitor.transport.ts:274`
  - body: `lastTransportActivityAt` starts the gateway stale-socket timer. Seeding it from one-time WebSocket start or webhook listen events means quiet but healthy Feishu accounts can be restarted after the threshold unless a recurring transport/heartbeat/poll signal refreshes it.
  - confidence: 0.9
- **[P2] Keep app messages out of transport liveness:** `extensions/feishu/src/monitor.message-handler.ts:329-332`
  - body: The shared health policy says app-level events are not socket liveness. This handler should update `lastEventAt` only unless Feishu exposes a real transport frame or heartbeat signal for `lastTransportActivityAt`.
  - confidence: 0.86
