## Summary

Keep open: this PR targets a real Feishu health-monitor gap, but the current patch still publishes health status from the wrong runtime signals and has only test-output proof after the patch, so it is not merge-ready.

## What This Changes

The PR wires a Feishu account status sink through channel startup, publishes connected and activity fields from WebSocket, webhook, and message-handler paths, and adds Feishu transport-status tests.

## Best Possible Solution

Publish Feishu connected/disconnected state from SDK lifecycle callbacks, reserve lastTransportActivityAt for real transport or heartbeat proof, and land only with redacted patched-runtime status and health-monitor evidence.

## Reproduction Assessment

Yes for the current status gap by source inspection: Feishu only publishes port status while the gateway health policy needs connected and transport-activity fields for stale-socket handling. I did not live-reproduce the Feishu WebSocket kill scenario in this read-only review.

## Solution Assessment

No; the status-sink direction is right, but this patch uses WSClient.start as a readiness signal and feeds app-level events into transport liveness. The safer fix is to use the SDK lifecycle callbacks and only publish transport timestamps from real transport proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Publish WS status from SDK readiness callbacks:** `extensions/feishu/src/monitor.transport.ts:267-277`
  - body: The pinned Feishu SDK does not make `start()` a handshake barrier: it assigns the dispatcher, calls `reConnect(true)`, and returns before `onReady`. Publishing `connected: true` here can mark slow or failed first handshakes healthy, so wire status to `onReady`/`onReconnected`/`onReconnecting`/`onError` instead.
  - confidence: 0.93
- **[P2] Stop treating app messages as transport activity:** `extensions/feishu/src/monitor.message-handler.ts:331-335`
  - body: `lastTransportActivityAt` drives stale-socket restarts and the shared policy says ordinary app events are not socket liveness. Updating it for Feishu messages means a healthy quiet workspace can be restarted once the last app event ages out; keep app traffic as `lastEventAt` unless there is recurring transport proof.
  - confidence: 0.87
