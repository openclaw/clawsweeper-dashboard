## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Publish WS status from readiness callbacks:** `extensions/feishu/src/monitor.transport.ts:264-274`
  - body: `WSClient.start()` in `@larksuiteoapi/node-sdk@1.66.0` only assigns the dispatcher and calls `reConnect(true)`; it does not wait for the first handshake. Publishing `connected: true` here can make the gateway health monitor treat a slow or failed Feishu socket as healthy, so status should come from `onReady`/`onReconnected`/`onReconnecting`/`onError` instead.
  - confidence: 0.93
- **[P2] Keep app messages out of transport liveness:** `extensions/feishu/src/monitor.message-handler.ts:323-333`
  - body: The gateway health policy treats `lastTransportActivityAt` as transport/heartbeat/poll proof and restarts connected channels after the stale threshold. Updating it from ordinary Feishu message events means a healthy quiet account can look stale once the last app message ages out, so message handling should update only app-event fields unless there is a recurring transport signal.
  - confidence: 0.88
