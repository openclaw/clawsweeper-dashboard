## Summary

Keep open. Current main has Discord READY/reconnect status handling and inbound dedupe once a `MESSAGE_CREATE` event is delivered, but it still has no REST history catch-up for messages sent before the WebSocket listener receives events.

## What This Changes

Request to queue or catch up inbound Discord messages sent while the gateway is running but the Discord WebSocket is not yet ready after startup or reconnect.

## Best Possible Solution

Implement a Discord-plugin-owned READY/reconnect catch-up path that fetches recent eligible messages, dedupes them with the existing inbound replay key shape, and feeds them through the same preflight and message-run queue as live events.

## Reproduction Assessment

Yes. The issue includes a concrete restart timeline, and current source inspection shows messages sent before `MESSAGE_CREATE` delivery have no monitor-side REST replay path.

## Solution Assessment

Yes. Since OpenClaw cannot queue Discord events that never reached its WebSocket listener, the narrow maintainable fix is Discord-owned recent-history catch-up after READY/reconnect rather than a core queue change.
