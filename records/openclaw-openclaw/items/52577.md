## Summary

Keep this open: current main still handles Discord messages only after gateway delivery, while this issue tracks the remaining READY/reconnect-window data-loss case where messages can be missed before the listener path receives them.

## What This Changes

The issue requests buffering or replaying Discord inbound messages sent during gateway reconnect or READY gaps after restart.

## Best Possible Solution

Implement Discord-plugin-owned readiness-gap recovery that records reconnect/startup windows, fetches bounded recent channel history after READY or RESUMED, re-drives eligible messages through existing preflight/dedupe/run-queue paths, and preserves exactly-once processing.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection on current main gives a clear path: delivered events are queued and deduped, but messages missed before READY have no catch-up hook.

## Solution Assessment

Yes. A narrow Discord plugin fix that reuses the existing REST message listing, inbound replay guard, preflight filters, and per-session run queue is the best direction; it should avoid new global core policy or config unless maintainers request it.
