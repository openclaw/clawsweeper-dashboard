## Summary

Keep open: current main still handles Discord inbound messages only after live gateway delivery, with no post-READY or RESUMED catch-up for messages missed during the reconnect/startup window. The open candidate fix at https://github.com/openclaw/openclaw/pull/81226 has not merged, so this issue should remain the Discord-specific tracker.

## What This Changes

The issue requests Discord to queue or backfill inbound messages sent during the gateway WebSocket startup/reconnect READY window instead of silently losing them.

## Best Possible Solution

Land a Discord-plugin-owned bounded backfill/replay after READY or RESUMED using existing REST message listing, preflight, replay dedupe, and the run queue, then close this issue after live or maintainer-accepted proof.

## Reproduction Assessment

Yes, at source level: current main registers live Discord MESSAGE_CREATE handling and waits for READY, but has no history replay hook for messages missed before gateway delivery. I did not run a live Discord restart in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable direction is a Discord plugin fix that reuses REST channel history, existing preflight filters, replay dedupe, and run-queue behavior rather than adding global core policy.
