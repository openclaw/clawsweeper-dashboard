## Summary

Keep open. Current main and v2026.6.6 still process Discord inbound messages only after Discord emits a live MESSAGE_CREATE event; there is no startup/READY/RESUMED history replay or inbound catch-up path, and the related backfill PR is closed unmerged.

## What This Changes

The issue requests queuing or recovering inbound Discord messages sent during gateway startup or WebSocket reconnect readiness gaps.

## Best Possible Solution

Keep this as the focused Discord inbound READY/reconnect tracker while https://github.com/openclaw/openclaw/issues/55792 owns the broader channel catch-up design, then land a provider-owned bounded catch-up path with dedupe and live Discord proof once maintainers choose scope and defaults.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows the current Discord monitor only queues messages after a live MESSAGE_CREATE dispatch and has no post-READY history replay path.

## Solution Assessment

Unclear. Queueing or bounded REST replay is plausible, but the best fix needs maintainer choice on Discord-only recovery versus broader provider replay semantics, dedupe persistence, and default behavior.
