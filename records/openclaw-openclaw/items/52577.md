## Summary

Keep open. Current main and the latest v2026.6.6 release still handle Discord inbound messages only after a live `MESSAGE_CREATE` reaches the Discord listener; the requested startup/reconnect catch-up remains unimplemented, and the related backfill PR is closed unmerged.

## What This Changes

Request to avoid losing Discord inbound messages sent during the Discord gateway startup or reconnect window by queueing them or replaying recent channel history after connect.

## Best Possible Solution

Keep this as the focused Discord reconnect/READY-window tracker and land a provider-owned bounded catch-up path with dedupe and live Discord proof, aligned with the broader design discussion at https://github.com/openclaw/openclaw/issues/55792.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection does show that current Discord code only queues already-delivered `MESSAGE_CREATE` events and lacks a post-READY history replay path.

## Solution Assessment

Unclear. A bounded Discord REST catch-up path is plausible, but the best fix needs maintainer choice on scope, dedupe/persistence, default behavior, and whether to consolidate with the broader provider catch-up design.
