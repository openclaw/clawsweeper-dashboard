## Summary

Keep open. Current main has stronger Discord READY/reconnect handling and queues work after Discord delivers a `MESSAGE_CREATE`, but it still has no READY/reconnect catch-up that fetches recent Discord history and replays messages missed before the WebSocket listener received them.

## What This Changes

The issue requests buffering or replaying Discord inbound messages sent while the Discord WebSocket is not fully connected after startup or reconnect.

## Best Possible Solution

Implement a Discord-plugin-owned READY/reconnect catch-up path: keep per-account/channel last-seen cursors, fetch recent eligible messages after READY or runtime reconnect, dedupe with the existing inbound replay key shape, and feed replayed events through the same preflight and message-run queue as live `MESSAGE_CREATE` events.

## Reproduction Assessment

Yes. The issue includes a concrete restart timeline, and the current source path shows that a message sent before Discord delivers `MESSAGE_CREATE` has no monitor-side REST replay path to reach the handler.

## Solution Assessment

Yes. Since OpenClaw cannot queue Discord events that Discord never delivered to the WebSocket listener, the maintainable fix is a Discord-owned recent-history catch-up after READY/reconnect, not a core queue migration.
