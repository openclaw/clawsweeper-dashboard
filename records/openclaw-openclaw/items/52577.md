## Summary

Keep this open: current main fixes the older Discord READY/IDENTIFY startup races, but the canonical remaining problem is still Discord inbound data loss around reconnect/READY lifecycle transitions, and this issue is the open tracker for that work.

## What This Changes

Fix Discord inbound messages being dropped during the post-restart or reconnect WebSocket readiness window by queueing delivered work or replaying recent eligible channel history after READY.

## Best Possible Solution

Implement Discord-plugin-owned reconnect recovery that records readiness gaps, replays recent eligible messages through existing preflight/dedupe/run-queue paths after READY, and preserves exactly-once processing for messages already delivered live.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection gives a clear current-main path: delivered events can be queued, but there is no catch-up hook for messages missed before READY and queued work can be skipped after lifecycle deactivation.

## Solution Assessment

Yes, a narrow Discord plugin fix is the right direction if it reuses the existing REST message listing, inbound replay guard, preflight filters, and per-session run queue. It should stop short of adding new global core policy or config unless maintainers explicitly request that.
