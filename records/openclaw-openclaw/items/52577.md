## Summary

Keep open. Current main and the latest release still only process Discord inbound messages after a live Discord Gateway MESSAGE_CREATE dispatch; there is no READY/RESUMED or startup history replay path, and the related Discord backfill PR is closed unmerged.

## What This Changes

Request queueing or backfilling inbound Discord messages sent during the post-restart WebSocket reconnect window so they are not silently missed.

## Best Possible Solution

Keep this focused Discord tracker open while the broader provider catch-up design remains on https://github.com/openclaw/openclaw/issues/55792, then land a provider-owned bounded recovery path with dedupe and live Discord proof after maintainers choose scope and defaults.
