## Summary

Keep open: current main still only processes Discord inbound messages after live gateway delivery, with no READY/RESUMED history replay or reconnect backfill, and the focused fix candidate remains open at https://github.com/openclaw/openclaw/pull/81226.

## What This Changes

The issue asks OpenClaw to queue or replay Discord inbound messages sent during the post-boot WebSocket readiness gap so they are not silently lost.

## Best Possible Solution

Land or replace the Discord-plugin-owned reconnect backfill in https://github.com/openclaw/openclaw/pull/81226 using REST history after READY/RESUMED, existing preflight filters, the run queue, and replay dedupe, then close this issue after live or maintainer-accepted proof.

Yes at source level: current main registers live MESSAGE_CREATE handling and waits for READY without a history replay hook for messages missed before gateway delivery. I did not run a live Discord restart in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable direction is a Discord plugin fix that reuses REST channel history, existing preflight filters, replay dedupe, and run-queue behavior rather than adding global core policy.
