## Summary

Keep open. Current main has backend pending-approval RPCs and event-driven Control UI approval modals, but it still lacks the requested discoverable Web dashboard or CLI queue for finding and acting on missed live exec approvals. The issue discussion is materially useful: mayank6136 correctly separates the existing policy-oriented `openclaw approvals` CLI from the live pending-request queue requested here.

## What This Changes

Request a Web and/or CLI surface to list pending exec approvals and approve or deny them when popup delivery is missed.

## Best Possible Solution

Add a first-class Pending approvals surface in the Control UI that loads `exec.approval.list` and `plugin.approval.list` on connect/reconnect, updates from requested/resolved events, resolves through the existing scoped resolve RPCs, shows command/plugin/session/agent/expiry/risk metadata, and clears stale entries. If maintainers want CLI recovery too, add narrowly scoped `openclaw approvals pending|approve|deny` commands backed by the same RPCs and document both workflows.

## Reproduction Assessment

Yes for the central missing surface: current-main inspection shows pending-list backend RPCs exist while the Control UI only renders event-driven modals and the CLI is policy-only. The intermittent popup failure itself is unclear without channel logs or a live reproduction.

## Solution Assessment

Unclear until maintainer design is settled, but the best technical direction is to reuse the existing `exec.approval.*` and `plugin.approval.*` pending/resolve RPCs rather than adding a second pending store. The Web vs CLI scope, authorization UX, audit behavior, and timeout handling should be decided before code is queued.
