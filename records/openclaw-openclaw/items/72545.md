## Summary

Keep open. Current main has pending exec/plugin approval list and resolve RPCs, but the Web UI still only shows event-driven modals and the CLI still manages approval policy rather than a discoverable live pending-approval queue.

## What This Changes

The issue asks for a Web GUI, and possibly CLI, surface to list pending live exec approvals and approve or deny them when popups are missed.

## Best Possible Solution

Add a first-class Pending approvals surface in Control UI that loads `exec.approval.list` and `plugin.approval.list` on connect/reconnect, updates from approval events, resolves through the existing scoped RPCs, and documents the recovery workflow; add CLI pending/approve/deny commands only if maintainers want terminal recovery too.

## Reproduction Assessment

Yes for the central missing surface: source inspection shows pending-list backend RPCs exist while the Control UI only renders event-driven modals and the CLI only manages policy/allowlists. The intermittent popup non-delivery case remains unclear without channel logs or a live failing run.

## Solution Assessment

Not applicable because there is no patch. The maintainable direction is to reuse the existing `exec.approval.*` and `plugin.approval.*` RPCs under `operator.approvals`, rather than introducing a second approval store.
