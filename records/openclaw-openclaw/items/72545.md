## Summary

Keep open. Current main has pending-approval backend RPCs and event-driven Control UI approval modals, but it still does not provide the requested discoverable Web dashboard or CLI queue for finding and acting on missed exec approvals.

## What This Changes

Request a Web GUI, and possibly CLI, surface to list live exec approvals waiting on a human and approve or deny them without relying on popup delivery.

## Best Possible Solution

Add a first-class Pending approvals surface in the Control UI that loads `exec.approval.list` and `plugin.approval.list` on connect/reconnect, updates from approval events, resolves through the existing scoped resolve RPCs, shows command/plugin/session/agent/expiry/risk metadata, and clears stale entries. If maintainers want CLI recovery too, add narrowly scoped `openclaw approvals pending|approve|deny` commands using the same RPCs and document both workflows.

## Reproduction Assessment

Yes for the central missing surface: current-main inspection shows pending-list backend RPCs exist, while the CLI is policy-only and the Control UI only renders event-driven modal prompts. The intermittent popup non-delivery part is unclear without channel logs or a live run.

## Solution Assessment

Unclear pending maintainer design, but the best technical direction is to reuse the existing `exec.approval.*` and `plugin.approval.*` pending RPCs rather than adding another pending store. The Web/CLI scope, authorization UX, audit behavior, and timeout handling should be decided before implementation.
