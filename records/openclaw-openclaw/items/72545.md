## Summary

Keep open. Current main has backend pending-approval list/resolve RPCs and event-driven Control UI approval modals, but it still lacks the requested discoverable Web dashboard or CLI queue for finding missed live exec approvals and approving or denying them outside the popup flow.

## What This Changes

The issue requests a Web GUI and possibly CLI surface that lists live exec approvals waiting on a human and lets an operator approve or deny them when popup prompts are missed.

## Best Possible Solution

Add a first-class Pending approvals surface in the Control UI that calls `exec.approval.list` and `plugin.approval.list` on connect/reconnect, stays updated from approval events, resolves through the existing scoped resolve RPCs, displays command/plugin/session/agent/expiry/risk metadata, and clears stale entries. If maintainers want CLI recovery too, add narrow `openclaw approvals pending|approve|deny` commands using the same RPCs and document both workflows.

## Reproduction Assessment

Yes for the central missing surface: source inspection shows pending-list backend RPCs exist, while the CLI is policy-only and the Control UI only renders event-driven modal prompts. The intermittent popup non-delivery case is unclear without channel logs or a live failing run.

## Solution Assessment

Unclear because no patch is proposed. The maintainable direction is to reuse the existing `exec.approval.*` and `plugin.approval.*` pending RPCs rather than adding a second pending store, with explicit maintainer review of Web/CLI authorization, audit, and timeout behavior.
