## Summary

Keep open. Current main has pending approval RPCs, scoped authorization, channel replay, and event-driven Control UI modals, but it still lacks the requested discoverable Web dashboard or CLI queue for finding and resolving missed exec approvals. The remaining work is security-sensitive operator UX, not conservative cleanup closure.

## What This Changes

Request a Web GUI, and possibly CLI, surface that lists pending exec approvals and lets an operator approve or deny them when popup prompts are missed.

## Best Possible Solution

Add a first-class Pending approvals surface in the Control UI that loads existing `exec.approval.list` and `plugin.approval.list` records on connect/reconnect, updates from approval events, resolves through the existing scoped `*.approval.resolve` RPCs, shows command/plugin/session/agent/expiry/risk metadata, and clears stale entries. If maintainers want CLI recovery too, add narrow `openclaw approvals pending|approve|deny` commands that reuse the same RPCs and document both workflows.

## Reproduction Assessment

Yes for the central missing surface: current-main inspection shows pending-list backend RPCs exist, while the CLI is policy-only and the Control UI only renders event-driven modal prompts. The intermittent missed-popup delivery itself is unclear without channel logs or a live reproduction.

## Solution Assessment

Yes in direction, but the maintainable fix should reuse the existing `exec.approval.*` and `plugin.approval.*` pending RPCs rather than adding another pending store. The exact Web/CLI scope, authorization UX, audit behavior, and timeout handling need maintainer review first.
