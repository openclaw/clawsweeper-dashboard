## Summary

Keep open. The PR targets a real installer/docs problem, but the proposed `bash -s --` documentation change does not fix stdin inheritance and would add a misleading guarantee while the linked installer bug remains open.

## What This Changes

This PR changes the Getting Started macOS/Linux installer one-liner to `curl ... | bash -s --` and adds a note saying those flags prevent stdin consumption.

## Best Possible Solution

Validate the default WSL2 piped installer path, then either harden `scripts/install.sh` so interactive children use terminal-backed stdin or update all public installer one-liners to one tested stdin-safe pattern.

## Reproduction Assessment

Yes for the PR's claimed fix: a focused pipe smoke shows a child still reads from the script pipe under both `bash` and `bash -s --`. The full WSL2 hosted-installer hang remains only partially verified from the linked report and needs live installer validation.

## Solution Assessment

No. `bash -s --` is appropriate for passing script arguments, but it is not a stdin-isolation fix; installer-side `/dev/tty` handling or a consistent caller pattern that leaves stdin free is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the false stdin guarantee:** `docs/start/getting-started.md:48`
  - body: The new note says `-s --` prevents the installer from consuming stdin, but those flags do not make child prompts or subprocesses terminal-backed. Child processes still inherit the piped script stdin, so this documents the linked installer hang as fixed when the same failure class can still occur.
