## Summary

Keep open. The PR is a focused docs attempt for a real installer report, but its central claim is incorrect: `bash -s --` does not isolate child-process stdin, so the proposed note would document the linked installer hang as fixed without proving that it is.

## What This Changes

This PR changes the Getting Started macOS/Linux install snippet to pipe the installer into `bash -s --` and adds a note claiming those flags prevent stdin consumption.

## Best Possible Solution

Validate the WSL2 path, fix any remaining installer subprocess or onboarding handoffs to use terminal-backed stdin or noninteractive mode, then update all public installer one-liners consistently with regression coverage.

## Reproduction Assessment

Yes for the PR's claimed fix: a focused pipe smoke reproduced that `bash -s --` still gives child processes non-TTY stdin and lets a child consume the next script line. The full WSL2 hosted-installer hang from the linked issue was not executed in this read-only review.

## Solution Assessment

No. `bash -s --` is appropriate when passing arguments to a piped script, but it is not a stdin-isolation fix; installer-side `/dev/tty` handling or a consistent tested download/process-substitution pattern is the safer direction.
