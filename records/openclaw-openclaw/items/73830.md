## Summary

Keep this PR open for normal review rather than cleanup closure. It addresses a real open installer report, but the proposed docs change is not merge-ready because `bash -s --` does not prevent child prompts from inheriting piped stdin, and the change would update only one quickstart snippet while other public install docs still show the raw pipe command.

## What This Changes

This PR changes the Getting Started macOS/Linux install snippet to pipe `install.sh` into `bash -s --` and adds a note claiming those flags prevent stdin consumption.

## Best Possible Solution

Validate the WSL2 installer failure, then either harden `scripts/install.sh` so interactive subprocesses consistently use terminal-backed stdin or adopt one consistent public install command pattern across the install docs, with focused regression coverage for the default interactive path.

## Reproduction Assessment

Yes for the PR's claimed fix: a focused pipe smoke shows `bash -s --` still leaves child processes with non-TTY stdin. The complete WSL2 hosted-installer hang from #73814 remains not fully reproduced in this read-only review.

## Solution Assessment

No. This is not the best way to solve the issue because it adds a false stdin guarantee and leaves the install docs inconsistent; the safer path is validated installer stdin handling or a docs-wide caller pattern that actually frees stdin.
