## Summary

Keep open. The linked installer report is real enough to investigate, but this PR is not merge-ready because `bash -s --` does not prevent child processes from inheriting the piped script stdin, and the patch adds that false guarantee to user-facing docs.

## What This Changes

This PR changes the Getting Started macOS/Linux install snippet to pipe the hosted installer into `bash -s --` and adds a note saying those flags prevent stdin consumption.

## Best Possible Solution

Validate the WSL2 installer failure, then either harden `scripts/install.sh` so interactive children use terminal-backed stdin or update all public installer one-liners to one consistent, tested caller pattern.

## Reproduction Assessment

Yes for the PR's claimed fix: a focused pipe smoke shows `bash -s --` still leaves child processes with non-TTY stdin that can consume subsequent script input. The full WSL2 hosted-installer hang from #73814 remains unverified in this read-only pass.

## Solution Assessment

No. `bash -s --` is appropriate for passing script arguments, but it is not a stdin-isolation fix; installer-side `/dev/tty` handling or a consistent process-substitution/download pattern is the safer direction.
