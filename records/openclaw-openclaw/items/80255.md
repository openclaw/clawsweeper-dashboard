## Label Justifications

- `P1`: The PR addresses a reported Active Memory prompt-build deadlock that can delay or time out user-visible replies.
- `merge-risk: 🚨 availability`: Merging the fixed process-wide recall lane can serialize slow Active Memory recalls across sessions and stall waiting parent replies.

## Summary

Keep open: this targets a real source-reproducible Active Memory self-deadlock, but the new recall lane is process-wide and the latest-head real behavior proof is not sufficient, so it needs maintainer scheduling/proof review before merge.

## What This Changes

The branch adds a dedicated Active Memory recall lane, passes it to `runEmbeddedAgent`, and adds a regression assertion for that lane.

## Best Possible Solution

Use a non-main recall lane that cannot self-deadlock and whose concurrency scope is explicit, then land it with current-head gateway proof plus the focused regression test.

## Reproduction Assessment

Yes, by source inspection: current main calls Active Memory recall without a lane, and the embedded runner defaults missing lanes to `main`, which can self-block when called from a parent prompt-build hook already occupying that lane.

## Solution Assessment

Unclear as submitted: passing a non-main lane is the right fix shape for the immediate deadlock, but a single process-wide unconfigured lane leaves an implicit availability policy that maintainers should accept or scope before merge.
