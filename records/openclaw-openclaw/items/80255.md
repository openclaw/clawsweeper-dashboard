## Summary

Keep open: current main still omits an explicit Active Memory recall lane, while this PR adds the narrow lane fix and focused coverage. The previous changelog blocker is gone, real behavior proof is now sufficient, and I found no blocking correctness or security findings; maintainers still need to accept the process-wide recall-lane scheduling shape before merge.

## What This Changes

This PR adds a dedicated `active-memory` embedded-run lane for Active Memory recall and a regression test asserting the lane is passed from the `before_prompt_build` hook.

## Best Possible Solution

Land this focused lane fix with its regression coverage if maintainers accept the process-wide recall lane; use a scoped-lane follow-up only if concurrency proof shows cross-session recall blocking.

## Reproduction Assessment

Yes, source-reproducible: the linked report has live timeout logs, and current main calls the embedded recall without `lane` while omitted embedded lanes resolve to `main`. I did not rerun a live gateway reproduction in this read-only review.

## Solution Assessment

Yes, the call-site lane is the narrowest maintainable fix for this specific self-deadlock and avoids changing embedded-runner defaults for other callers. The only remaining judgment is whether the fixed `active-memory` lane should be process-wide or scoped more narrowly.
