## Summary

Keep open: current main still has active-run queueing and Telegram isolated-ingress spool primitives, but I did not find a native reload-window admission queue that preserves messages after they enter Telegram handling while the target agent/session path is unavailable.

## What This Changes

Request to buffer inbound Telegram/channel messages during gateway agent reload or session reset, then drain them FIFO instead of dropping them.

## Best Possible Solution

Choose the owner boundary first, then make reload-time Telegram dispatch failures retryable through the existing ingress spool or a generic channel turn admission queue with focused hot-reload regression coverage.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction. Source inspection gives a medium-confidence path: hot reload only waits for already-counted work, existing followup queues are reset-gated, and Telegram dispatch failures can be handled without requeueing the spooled update.

## Solution Assessment

Unclear; the requested FIFO queue is plausible, but the maintainable fix should decide whether to reuse Telegram isolated ingress or define a generic channel admission contract before adding TTL/depth config.
