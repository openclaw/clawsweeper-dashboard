## Summary

Keep open: current main has active-run queueing and Telegram ingress-spool retry primitives, but I did not find a settled reload-window admission path that guarantees Telegram messages are preserved after they enter handling while the target session/reload path is unavailable.

## What This Changes

The issue requests a transient inbound message queue during agent/session reload so Telegram messages are buffered and drained instead of silently dropped.

## Best Possible Solution

Choose the owner boundary first, then make reload-time Telegram dispatch failures retryable through the existing ingress spool or a generic channel admission contract with focused hot-reload regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a medium-confidence path through hot reload channel stop/start, reset-gated followup queue behavior, and Telegram dispatch failures that may not requeue through isolated ingress.

## Solution Assessment

Unclear; a FIFO queue is plausible, but the maintainable fix should decide whether this belongs in Telegram isolated ingress or a generic channel admission contract before adding TTL/depth config.
