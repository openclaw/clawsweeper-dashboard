## Summary

Keep open: current main has active-run queueing and Telegram ingress spool primitives, but I did not find a native reload-window admission queue that preserves messages after the Telegram handler accepts an update while the target session path is resetting or unavailable. The existing labels and prior ClawSweeper review correctly route this to maintainer/product review and live repro rather than a fresh automated fix PR.

## What This Changes

The issue requests a transient per-agent/channel inbound queue so Telegram messages arriving during agent or session reload are buffered and drained instead of silently lost.

## Best Possible Solution

Choose the owner boundary first, then make reload-time Telegram dispatch failures retryable through the existing ingress spool or a generic channel admission queue, with focused hot-reload regression coverage and docs only if new config ships.

## Reproduction Assessment

No; the issue gives production timing and a plausible live repro, but I did not run a Telegram hot-reload scenario. Source inspection gives a medium-confidence path because reload stop/start, reset queue cleanup, and Telegram dispatch error swallowing are visible on current main.

## Solution Assessment

Unclear; the proposed FIFO queue is plausible, but the narrow maintainable fix should first decide whether to extend Telegram ingress spool or introduce a generic channel admission contract instead of adding channel-local TTL/depth config directly.
