## Summary

Keep open: current main still has active-run queueing and Telegram ingress spool primitives, but I did not find a native reload-window inbound admission queue that preserves Telegram messages when the target agent/session path is unavailable during reload. This remains a plausible message-loss bug that needs maintainer direction on the owner boundary and configuration surface.

## What This Changes

The issue requests transient per-agent/channel inbound queueing during agent reloads so Telegram messages are buffered and drained instead of lost.

## Best Possible Solution

Decide whether the owner boundary is Telegram isolated ingress or a generic channel admission contract, then make reload-time dispatch unavailability retryable with focused hot-reload regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a medium-confidence path through channel stop/start reload, reset-gated followup queues, and Telegram dispatch errors that can be swallowed before the spool layer retries.

## Solution Assessment

Unclear; the requested FIFO queue with TTL/depth is plausible, but the maintainable fix should first decide whether to extend Telegram ingress spool or add a generic channel admission queue.
