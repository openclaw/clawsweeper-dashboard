## Summary

Keep open: current main has active-run queueing and Telegram isolated-ingress spool primitives, but I did not find a native/documented reload-window admission queue that preserves Telegram messages after they enter the handler while the target agent/session path is unavailable.

## What This Changes

The issue requests buffering inbound Telegram messages during agent reset/reload, then draining them FIFO with TTL and queue-depth controls.

## Best Possible Solution

Choose the owner boundary first, then make reload-time Telegram dispatch failures retryable through the existing ingress/spool or a generic channel admission queue with focused hot-reload regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a medium-confidence path: hot reload stops/starts channels after active-work deferral, existing followup queues are reset-gated, and Telegram dispatch errors can be swallowed before the spool layer retries.

## Solution Assessment

Unclear; the requested FIFO queue is plausible, but the maintainable fix should decide whether this belongs in Telegram isolated ingress or a generic channel turn admission contract before adding new TTL/depth config.
