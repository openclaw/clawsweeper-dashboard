## Summary

Current main already addresses the likely direct-session failure mode: Telegram DMs enter the shared inbound reply turn, and the CLI runner now drops stale Claude CLI resume bindings that could trap Telegram direct auto-reply turns after a broken update.

## What This Changes

The issue reports Telegram webhook DMs reaching OpenClaw but not producing an agent response, while outbound DMs and Telegram topic routing still work.

## Best Possible Solution

Keep the current-main stale CLI session invalidation and ship it in the next release so affected Telegram direct sessions recover without manual session-store surgery.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: a Telegram direct session key with a stale claude-cli sessionId and missing transcript is covered by the current regression test; I did not run a live Telegram webhook.

## Solution Assessment

Is this the best way to solve the issue? Yes; invalidating the stale CLI binding in the shared runner is narrower and more durable than adding Telegram-specific migration or routing special cases.
