## Label Justifications

- `P2`: This is a provider-specific cost/performance feature with concrete correctness blockers but limited blast radius.
- `merge-risk: 🚨 compatibility`: The patch changes existing `cacheRetention` semantics for Nova users and can override an explicit persisted opt-out.

## Summary

Keep open: the provider-owned direction is useful, but the PR currently bypasses an explicit Nova `cacheRetention: "none"` opt-out and only matches direct `amazon.nova-*` IDs, leaving OpenClaw-discovered regional/global Nova profiles on the old path.

## What This Changes

The PR changes the bundled Amazon Bedrock plugin so direct Amazon Nova model IDs avoid the no-cache wrapper and receive Bedrock `cachePoint` injection, with updated provider tests.

## Best Possible Solution

Keep the provider-owned Nova support, but make the Nova branch preserve explicit opt-outs, cover direct and discovered profile IDs, update public prompt-caching docs, and add redacted live Bedrock proof before merge.

## Reproduction Assessment

Yes, at source level. With the PR branch, a Nova request carrying `cacheRetention: "none"` reaches the new Nova `onPayload` hook and still receives a hardcoded short `cachePoint`; I did not run a live Bedrock request in this read-only review.

## Solution Assessment

No. The provider-owned fix boundary is right, but this implementation is too narrow and should preserve explicit cache settings, cover profile IDs, and refresh docs/proof before merge.

Full review comments:

- **[P1] Honor cacheRetention none for Nova:** `extensions/amazon-bedrock/register.sync.runtime.ts:710`
  - body: This new Nova path injects `cachePoint` with hardcoded short retention even when the caller passed `cacheRetention: "none"`. That silently overrides the documented opt-out and can cache prompts that users explicitly configured not to cache, so derive the retention from `merged.cacheRetention` and return early for `none` with regression coverage.
  - confidence: 0.92
- **[P2] Cover Nova inference profile IDs:** `extensions/amazon-bedrock/register.sync.runtime.ts:94`
  - body: `startsWith("amazon.nova-")` only handles direct foundation model IDs. OpenClaw discovers regional/global inference profiles and prior Nova reports used IDs such as `global.amazon.nova-2-lite-v1:0`, so those users would still be forced through the no-cache path after this PR.
  - confidence: 0.88
- **[P3] Update the prompt-caching docs:** `extensions/amazon-bedrock/register.sync.runtime.ts:441`
  - body: After this behavior change, the public prompt-caching docs would still say non-Anthropic Bedrock models are forced to `cacheRetention: "none"`. Update the Bedrock/Nova prompt-caching docs alongside the runtime change so users understand which Nova IDs are supported and how `none`/short retention behave.
  - confidence: 0.82
