## Label Justifications

- `P2`: The issue describes a real managed Codex plugin install/update reliability bug with platform-specific blast radius and a clear source-level failure path.
- `impact:crash-loop`: The reported bad install can leave the Codex app-server failing at runtime while macOS update automation repeatedly restarts the Gateway.

## Summary

Keep open: current main still lacks the central install-time guard for Codex's host-native optional dependency. The managed npm path has rollback/index/update scaffolding, but it still preserves platform/optional npm config pollution and only verifies the top-level plugin package before recording success.

## What This Changes

The issue requests hardening managed Codex npm installs and updates against missing or wrong-platform native optional dependencies, stale install indexes, and macOS update loops.

## Best Possible Solution

Land a maintainer-approved focused fix that scrubs managed npm platform/optional config pollution and validates the Codex host-native optional package and vendor binary before persisting install/update success, with launchd backoff and stale-index precedence handled as explicit subtasks.

## Reproduction Assessment

Yes, source-level. A managed npm install that preserves polluted platform/arch or optional-dependency npm config can leave the host Codex optional package absent, and upstream Codex maps darwin arm64 to @openai/codex-darwin-arm64 before throwing the reported missing optional dependency error; no live macOS arm64 run was executed here.

## Solution Assessment

No, current main is not a completed fix. The narrow maintainable direction is install/update-time validation plus managed npm env scrub at the managed npm or Codex plugin boundary, rather than relying on runtime binary lookup after success is persisted.
