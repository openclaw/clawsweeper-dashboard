## Label Justifications

- `P2`: The issue describes a real managed Codex install/update reliability bug with platform-specific but user-visible runtime impact.
- `impact:crash-loop`: A bad managed Codex install can leave startup failing while macOS update automation repeatedly restarts the Gateway.

## Summary

Keep open: current main and v2026.6.6 still do not validate the host-required Codex native optional dependency before a managed plugin install is recorded as successful, and upstream Codex still fails at runtime when that package or vendor binary is missing.

## What This Changes

The issue requests managed npm install hardening for Codex native optional dependencies, bad-project rebuild/quarantine, stale install-index handling, and macOS update launchd safeguards.

## Best Possible Solution

Land a maintainer-approved reliability fix that rejects or rebuilds host-incompatible managed Codex dependency trees before install/update success is persisted, with launchd updater and stale-index behavior handled as scoped subtasks or follow-ups.

## Reproduction Assessment

Yes at source level: a managed npm install that inherits platform/arch or optional-dependency npm config can produce the wrong or missing Codex native package, and upstream Codex then throws the reported missing optional dependency error. I did not execute the macOS arm64 upgrade scenario in this read-only pass.

## Solution Assessment

No, current main is not yet the best completed solution: it validates top-level plugin install metadata and command-path existence, but not the nested host-native Codex optional package and vendor binary. The better fix belongs at the managed npm/Codex install-update validation boundary with repair and upgrade tests.
