## Summary

Keep open: current main and v2026.6.8 now cover the generic install-time guard for missing current-platform packages, but the requested reliability work is only partially implemented. The managed npm environment still does not scrub platform/arch/optional-dependency npm config pollution, and the remaining Codex-specific validation, stale-index repair, and macOS updater backoff pieces still need maintainer direction.

## What This Changes

The issue requests hardening managed Codex plugin install/update so host-native Codex optional dependencies are validated before success and macOS update/plugin repair paths do not repeatedly restart into a bad install.

## Best Possible Solution

Keep the shipped required-platform-package guard, then land maintainer-approved follow-up work that scrubs managed npm platform/optional config and explicitly decides whether Codex-specific nested version/vendor validation, stale-index repair, and launchd updater backoff belong together or as separate tracked fixes.

## Reproduction Assessment

Yes at source and dependency-contract level: current code still preserves the relevant npm platform/optional config keys, and upstream Codex throws when the host-native optional package or vendor binary is absent. I did not establish a fresh live macOS arm64 upgrade repro during this read-only review.

## Solution Assessment

No, current main is only a partial solution. The generic required-platform-package guard is the right direction, but the best fix also needs maintainer-approved env hardening and a decision on the Codex-specific validation and macOS updater/index recovery pieces.
