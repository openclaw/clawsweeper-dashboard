## Label Justifications

- `P2`: This is a real managed Codex plugin install/update reliability bug with concrete source evidence and limited platform-specific blast radius.
- `impact:crash-loop`: The reported bad install can leave Codex startup failing while macOS update automation repeatedly restarts the Gateway.
- `impact:auth-provider`: A missing native @openai/codex package prevents the configured Codex app-server provider runtime from starting.

## Summary

Keep open: current main and the latest release still lack the central install-time guard. OpenClaw now has rollback/quarantine scaffolding and some managed-service update handoff hardening, but managed npm installs still inherit platform/optional-dependency npm config and only verify the top-level plugin package before recording success.

## What This Changes

The issue asks OpenClaw to harden managed Codex plugin installs and macOS update behavior so a missing or wrong-platform @openai/codex native optional dependency is caught and repaired before runtime startup.

## Best Possible Solution

Keep this open until a maintainer-scoped fix scrubs managed npm platform/optional-dependency pollution, validates the host-native Codex optional package before persistence, and explicitly scopes the launchd and stale-index repair behavior.

## Reproduction Assessment

Yes at source level: polluted npm platform/arch/optional env can survive into managed npm install, and upstream Codex throws when the host-native optional package is absent. I did not run the macOS arm64 install/update scenario in this read-only review.

## Solution Assessment

Unclear as one bundled issue: the env scrub plus Codex optional-dependency verifier is the right central fix, but the launchd and stale-index policy pieces need maintainer scoping before implementation.
