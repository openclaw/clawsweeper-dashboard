- `impact:crash-loop`: A bad managed Codex install can leave the app-server failing at runtime while update automation repeatedly restarts or attempts to recover the Gateway.

## Summary

Keep open: current main still has only partial managed npm/update hardening, and the central request to validate the host-required nested Codex native optional dependency before install/update success is recorded remains missing.

## What This Changes

The issue requests hardening managed Codex plugin install/update so host-native optional dependencies are validated and bad macOS update/install states are rebuilt instead of surfacing at runtime.

## Best Possible Solution

Keep this issue open and land a scoped reliability fix that scrubs host-platform and optional-dependency npm config for managed installs, validates the nested host-native Codex optional package and vendor binary before persisting success, and adds repair coverage for bad managed projects; treat launchd/backoff and stale-index precedence as explicit maintainer-scoped subtasks.

## Reproduction Assessment

Yes. Source inspection shows polluted npm platform/optional config can survive into managed installs while current verification only checks the top-level package resolution, and sibling Codex source throws the reported missing optional dependency error when the host package is absent.

## Solution Assessment

No. Current main is not the best completed solution because it lacks nested host-native Codex optional-dependency validation before recording managed install/update success; the safer direction is a narrow managed install/update verifier plus tests, with launchd policy split if needed.
