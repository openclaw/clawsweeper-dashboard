## Summary

Keep open: current main and the latest release still lack install-time validation for Codex's host-native optional dependency and do not scrub platform/optional npm config pollution from managed plugin installs. The report is valid reliability work, but it spans managed npm policy, Codex-specific validation, stale install-index handling, and macOS update behavior, so maintainer review remains the right lane.

## What This Changes

Validate the host-native `@openai/codex` optional dependency during managed `@openclaw/codex` install/update, scrub npm platform/optional env pollution, and harden update/stale-index recovery.

## Best Possible Solution

Land a maintainer-approved focused repair that scrubs managed npm platform/optional config pollution and validates the Codex host-native optional package and vendor binary before install/update success is persisted, with launchd and stale-index hardening handled as explicit subtasks.

## Reproduction Assessment

Yes, source-level. A managed npm install that preserves polluted platform/arch or optional-dependency npm config can leave the host Codex optional package absent, and upstream Codex maps darwin arm64 to `@openai/codex-darwin-arm64` before throwing the reported error.

## Solution Assessment

No, current main is not a completed fix. The maintainable direction is install/update-time env scrub plus Codex native optional package and vendor-binary validation, rather than discovering the missing binary only when the app-server starts.
