- `impact:crash-loop`: The reported missing native Codex package can leave the app-server failing at startup while the update job repeatedly restarts the Gateway.

## Summary

Keep open: current main still preserves npm platform/optional-dependency config during managed plugin installs and only verifies top-level npm metadata, so the reported Codex native optional dependency failure can still be recorded as a successful install.

## What This Changes

The issue asks OpenClaw to fail fast or repair managed Codex plugin installs when npm produces a host-incompatible or missing native optional dependency, and to harden related macOS update/index cleanup paths.

## Best Possible Solution

Keep this open as a maintainer-scoped reliability item and land focused installer/repair validation for managed Codex native optional dependencies, with launchd and stale-index policy either explicitly included or split into narrower follow-ups.

## Reproduction Assessment

Yes for source reproduction: current main can preserve polluted npm platform/optional config through managed plugin install and then validates only top-level package metadata, while upstream Codex fails when the host-native package or vendor binary is absent. I did not run a live macOS arm64 update repro.

## Solution Assessment

No, current main is not a complete fix; the best path is installer/repair-level npm env hardening plus Codex native optional package and binary validation before persisting install/update success, with separate maintainer scoping for launchd and stale-index policy.
