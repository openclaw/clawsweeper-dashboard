## Label Justifications

- `P2`: The report describes a real managed Codex plugin install/update reliability bug with limited platform-specific blast radius.
- `impact:crash-loop`: The reported bad install can leave the Codex app-server failing at runtime while macOS update automation repeatedly restarts the Gateway.

## Summary

Keep open: current main has managed npm rollback/quarantine scaffolding, static plugin payload checks, stale updater-job detection, and partial installed-index recovery, but it does not yet validate the host-required Codex native optional dependency or scrub platform/arch/optional npm config pollution before recording a managed Codex install as successful.

## What This Changes

The issue requests hardening managed Codex plugin npm installs, native optional dependency validation, bad install quarantine/rebuild, launchd update safeguards, and stale install-index cleanup.

## Best Possible Solution

Keep this issue open and land a focused reliability fix that hardens managed npm install env handling, validates the host-required Codex optional package and vendor binary before install/update success is persisted, and adds upgrade/repair regression coverage, with launchd/backoff and stale-index precedence handled as explicit subtasks if needed.

## Reproduction Assessment

Yes, we have a high-confidence source reproduction path: a managed npm install that preserves polluted platform/arch or optional-dependency npm config can install the wrong or missing Codex native child, and upstream Codex then throws the reported missing optional dependency error. I did not execute the macOS arm64 scenario in this read-only review.

## Solution Assessment

No, current main is not the best completed solution because it validates the top-level plugin payload but not the nested host-native Codex optional dependency. The better direction is a narrow install/update verifier at the managed npm or Codex plugin boundary, plus env scrub and repair tests.
