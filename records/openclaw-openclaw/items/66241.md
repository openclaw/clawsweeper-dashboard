## Summary

Keep open. Current main still has a memory-only Discord component/modal registry, so the user-visible restart bug remains, but this PR is not merge-ready because its new persistence path is anchored to the home-directory cache instead of the active OpenClaw state/plugin-state storage now available on main.

## What This Changes

The PR adds best-effort JSON persistence, validation, pruning, and tests for Discord component/modal registry entries across gateway restarts.

## Best Possible Solution

Rework Discord component and modal registry persistence onto the active runtime state system, preferably api.runtime.state.openKeyedStore namespaces such as discord.components and discord.modals, while preserving consume-vs-peek semantics, TTLs, route overrides, allowlists, validation, expiry pruning, and bounded growth.

## Reproduction Assessment

Yes. A source-level reproduction is clear: register a Discord component, restart the gateway so the process-global Maps are empty, then click the still-visible Discord button; current main resolves null and replies that the component has expired.

## Solution Assessment

No. The PR addresses the right bug, but the home-cache JSON store is not the narrowest maintainable solution now that current main has state-dir based plugin keyed storage for restart-safe registries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use active state storage for registry persistence:** `extensions/discord/src/components-registry.ts:30-34`
  - body: `resolveRegistryPath` anchors the new registry under `homedir()/.openclaw/cache`, so deployments or tests that set `OPENCLAW_STATE_DIR` still read and write Discord component state outside the active state root. A gateway restarted with relocated state can therefore miss the registry this feature is meant to preserve; derive this from `api.runtime.state.openKeyedStore` or the state-dir helper instead.
  - confidence: 0.89
