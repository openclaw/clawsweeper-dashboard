## Summary

Current main already implements the PR’s intended Discord restart-survival behavior via the merged plugin-state backed registry from #75584, and it does so through the active runtime state store rather than the PR branch’s raw home-cache JSON file.

## What This Changes

The PR branch adds best-effort JSON file persistence and tests for Discord component/modal registry entries so buttons and forms can resolve after gateway restarts.

## Best Possible Solution

Close this PR as implemented on current main and keep the merged keyed-store approach from #75584 as the canonical implementation; any remaining refinements should build on the runtime state store rather than reviving a separate home-cache JSON registry.

## Reproduction Assessment

Yes. The source-level reproduction is to register a Discord component, restart the gateway so process-local maps are empty, then click the still-visible Discord component; current main’s persistence-aware lookup now covers that path when the Discord runtime state store is available.

## Solution Assessment

No for this PR branch. It fixes the right user-visible problem, but current main’s `api.runtime.state.openKeyedStore` implementation is the narrower maintainable solution because it follows the active OpenClaw state directory, plugin scoping, TTL, and store limits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the runtime state store for registry persistence:** `extensions/discord/src/components-registry.ts:30-34`
  - body: `resolveRegistryPath()` anchors the new registry under `homedir()/.openclaw/cache`, so deployments or tests that relocate OpenClaw state with `OPENCLAW_STATE_DIR` still write Discord component state outside the active state root.
