## What This Changes

This PR routes ACP task cancellation through Gateway `tasks.cancel`, fails ACP CLI cancellation when Gateway is unavailable, dedupes late-derived ACP task rows, fans out same-run ACP cancellation, and adds command, registry, and Gateway handler tests.

## Best Possible Solution

Land only after redacted real ACP spawn/cancel/readback and Gateway-down proof is attached, with maintainer acceptance that ACP cancellation now requires the live Gateway path.

## Reproduction Assessment

Yes from source inspection: current main routes only cron cancellation through Gateway, while documented ACP task cancellation is supposed to kill the child session. I did not run a live ACP repro in this read-only review.

## Solution Assessment

Yes, the owner-boundary shape is the right fix: CLI delegates ACP cancellation to live Gateway `tasks.cancel`, and Gateway owns the ACP control runtime. It is not merge-ready until real behavior proof is inspectable and maintainers accept the fail-closed upgrade behavior.
