## Summary

Keep this PR open: the code change is focused and appears to address the ACP cancellation ownership bug, but it is not merge-ready until the ACP Gateway-required cancellation behavior has inspectable real behavior proof and maintainer-visible acceptance of the fail-closed CLI behavior.

## What This Changes

The PR routes ACP task cancellation through Gateway `tasks.cancel`, fails ACP CLI cancellation when Gateway cancellation is unavailable, dedupes ACP task rows across late-derived one-task flow ids, and updates cancellation across same-run ACP rows.

## Best Possible Solution

Keep the PR open and land it only after redacted real ACP Gateway cancellation proof is available and maintainers accept the Gateway-required fail-closed CLI behavior.

## Reproduction Assessment

Yes from source inspection: current main only routes cron task cancellation through Gateway before local fallback, so ACP task cancellation reaches the local process path instead. I did not run a live ACP repro in this read-only review.

## Solution Assessment

Yes, likely: Gateway `tasks.cancel` is already the SDK/control-plane cancellation owner, and extending the CLI Gateway-first path from cron to ACP is the clean owner-boundary fix. The remaining concern is not the patch shape, but proof and compatibility acceptance for the fail-closed behavior.
