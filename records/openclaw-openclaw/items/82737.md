## Summary

Keep open: the TUI footer fix is aimed at a real display bug, but the latest diff still treats every missing total with a known context window as zero, losing the existing `totalTokensFresh=false` unknown/stale distinction.

## What This Changes

The PR adds a TUI footer helper, tests, and a repro script so known-context sessions with missing totals render `tokens 0/ctx` instead of `tokens ?/ctx`.

## Best Possible Solution

Thread `totalTokensFresh` through the TUI session state and only show `0/ctx` for genuinely context-known, no-usage/default rows while preserving stale missing totals as unknown.

## Reproduction Assessment

Yes. Current main source shows the footer passes a null total with known context into `formatTokens`, and the PR-introduced stale-total regression is source-reachable through `sessions.list` rows with `totalTokensFresh=false` and no `totalTokens`.

## Solution Assessment

No. The helper is the right narrow surface, but it needs freshness metadata from the session API instead of zeroing every missing total with a known context window.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve stale unknown totals before zeroing the footer:** `src/tui/tui-formatters.ts:439-440`
  - body: This helper returns `0` whenever `contextTokens` is known and `totalTokens` is absent. `sessions.list` can emit rows with no `totalTokens` and `totalTokensFresh=false`; current session/status contracts treat those as unknown, but this patch would render them as `tokens 0/ctx`. Thread `totalTokensFresh` into TUI session state and only default to zero for non-stale context-only rows.
  - confidence: 0.88
