## What This Changes

The PR adds a `getRuntimeConfigSnapshot()` fallback in `src/agents/model-auth.ts` so custom provider `apiKey: "secretref-managed"` markers can resolve from the runtime config snapshot.

## Best Possible Solution

Resolve non-env SecretRef markers only through a source-matched runtime snapshot, then add regression coverage for both matching and mismatched snapshots before landing.

## Reproduction Assessment

Yes, from source: current main treats `secretref-managed` as a non-secret marker and returns `null` for the custom-provider auth path. I did not run a gateway repro because this review is read-only.

## Solution Assessment

No. The direction is right, but the implementation should use the existing source/runtime snapshot matching contract instead of a raw object-identity check.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate the snapshot fallback by source config:** `src/agents/model-auth.ts:341`
  - body: The new fallback accepts any active runtime snapshot as long as it is not the same object as `params.cfg`. This helper is called with caller-provided configs from auth availability and status paths, while the config subsystem already tracks the source snapshot to avoid stale runtime data. After a config change or a call with an unrelated config that still contains `apiKey: "secretref-managed"`, this can return a resolved key from the previous active snapshot instead of treating auth as unavailable. Please only use the runtime key when the active source snapshot matches the input config, and cover both match and mismatch cases.
  - confidence: 0.87
