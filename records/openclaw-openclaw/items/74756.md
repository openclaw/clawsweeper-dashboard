## Summary

Keep this PR open, but do not merge it as-is. The linked bug is real on current main, and the PR moves in the right direction, but its ACPX wrapper only forwards `pid` fields that the pinned `acpx@0.6.1` runtime does not expose through its public handle or status contract, so real ACPX-backed spawns would still omit the PID. The host-PID exposure semantics and changelog entry also need maintainer follow-through.

## What This Changes

The PR adds optional `pid` fields to ACP runtime/session metadata, forwards them through the ACPX wrapper and ACP control-plane manager into `sessions_spawn` results, and adds PID-present/PID-absent unit tests.

## Best Possible Solution

Define or consume a typed ACPX process-metadata contract first, including which process identifier is exposed and how stale/reused PIDs are handled, then propagate that contract through the plugin SDK runtime types, ACP control-plane metadata, `sessions_spawn`, `session_status`, docs, changelog, and integration-style coverage.

## Reproduction Assessment

Yes. Source and dependency inspection is enough: current main omits PID fields from ACP spawn/session metadata, and `acpx@0.6.1` does not expose the typed PID fields this PR expects.

## Solution Assessment

No. The direction is useful, but this patch is not the best fix because it propagates mocked/non-contract fields; the safer maintainable path is an upstream ACPX process-metadata contract plus an explicit OpenClaw exposure policy.
