## Summary

Keep open. The linked bug remains valid, but this PR is not mergeable as-is because it reads PID fields that the pinned ACPX runtime does not expose through its public handle/status contract, and it needs an explicit host-PID exposure policy plus the required changelog entry. Target: https://github.com/openclaw/openclaw/pull/74756

## What This Changes

The PR adds optional PID fields to ACP runtime/session metadata, propagates them through ACPX, ACP control-plane status, and `sessions_spawn`, and adds PID-present/PID-absent unit tests.

## Best Possible Solution

Add or consume a typed ACPX process-metadata contract first, define adapter-vs-child PID semantics and exposure policy, then propagate that contract through SDK/runtime types, ACP control-plane status, `sessions_spawn`, docs, changelog, and integration-style coverage.

## Reproduction Assessment

Yes. Source and dependency inspection is enough: current main omits PID fields from spawn/session metadata, and `acpx@0.6.1` does not expose the typed `pid` fields this PR expects.

## Solution Assessment

No. The direction addresses a real observability gap, but copying undeclared ACPX mock fields is not the narrowest maintainable fix; the safer path is a typed upstream/runtime PID contract plus documented OpenClaw exposure policy.
