## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested ACP PID observability, while this branch reads PID values from ACPX fields that the pinned `acpx@0.6.1` runtime does not emit and exposes host PIDs without a settled visibility contract or changelog entry.

## What This Changes

The PR adds optional `pid` fields to ACP runtime/session metadata, propagates the initialized ACP handle PID into `sessions_spawn` results, extracts PID-like values from the ACPX delegate, and adds PID-present/PID-absent unit tests.

## Best Possible Solution

Add a typed ACPX process-metadata contract first, then propagate that additive contract through the OpenClaw plugin SDK runtime types, ACP control-plane metadata, `sessions_spawn`, `session_status`, docs, changelog, and tests with explicit semantics for adapter PID versus child-agent PID and stale identifiers.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the linked bug: ACP spawn results, runtime handle/status types, and persisted ACP metadata omit PID/process metadata, while docs still require manual process-table correlation.

## Solution Assessment

No. The direction is useful, but this PR is not the best fix because it reads top-level or `details.pid` fields that `acpx@0.6.1` does not emit; the safer path is a real ACPX process-metadata contract with documented exposure rules.
