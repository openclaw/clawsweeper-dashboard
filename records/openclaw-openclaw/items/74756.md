## Summary

Keep this PR open, but do not merge it as-is. The linked bug is valid, but the patch reads PID fields that the pinned ACPX runtime does not expose through its public handle/status contract, so real ACPX spawns would still omit the new `sessions_spawn` PID field.

## What This Changes

The PR adds optional PID fields to ACP runtime/session metadata, propagates them through the ACPX wrapper, control-plane manager, and `sessions_spawn` result, and adds PID-present/PID-absent unit tests.

## Best Possible Solution

Add or consume a typed ACPX process-metadata contract first, with explicit adapter-vs-child PID semantics and stale/reused PID handling, then propagate that contract through the plugin SDK runtime types, ACP control-plane metadata, `sessions_spawn`, `session_status`, docs, changelog, and integration-style coverage.

## Reproduction Assessment

Yes. Source and dependency inspection is enough: current main omits PID fields from spawn/session metadata, and `acpx@0.6.1` does not expose the typed `pid` fields this PR expects.

## Solution Assessment

No. The direction is useful, but this is not the best fix because it propagates mocked or non-contract fields; the safer maintainable path is an upstream ACPX process-metadata contract plus documented OpenClaw exposure policy.
