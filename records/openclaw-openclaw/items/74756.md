## Summary

Keep this PR open, but do not merge it as-is. The underlying PID observability request is valid and still missing on current main, but this patch reads PID from ACPX shapes that the pinned public runtime does not expose, omits the required changelog entry, and needs maintainer/API/security review for host PID exposure.

## What This Changes

Adds optional PID fields to ACP runtime/session metadata and propagates a delegate-provided PID into ACP `sessions_spawn` results, with unit coverage for PID-present and PID-absent cases.

## Best Possible Solution

Define or consume a typed ACPX process-metadata contract first, document PID meaning and visibility/staleness rules, then propagate it through the SDK runtime types, ACP manager, `sessions_spawn`, `session_status`, docs, changelog, and integration-style coverage.

## Reproduction Assessment

Yes. Source and dependency inspection are enough: current main omits PID from ACP spawn/session metadata, and the pinned `acpx@0.6.1` public runtime handle/status contract does not expose the PID shape this PR reads.

## Solution Assessment

No. The direction is useful, but this patch is not the best fix because it propagates non-contract mock fields; the maintainable path is to expose a real ACPX process-metadata source and land the OpenClaw API/security policy with it.

Concerns:
