## Summary

Keep this PR open, but it should not merge as-is. The linked `sessions_spawn(runtime="acp")` PID observability bug is still real on current `main`, but the patch forwards PID fields that the pinned ACPX runtime contract does not actually expose, and it also needs explicit PID exposure semantics plus a changelog entry.

## What This Changes

The PR adds optional PID fields to ACP runtime/session metadata, tries to extract PID values from ACPX delegate handle/status results, propagates them through control-plane metadata and `sessions_spawn`, and adds PID-present/PID-absent unit tests.

## Best Possible Solution

Add a typed ACPX process-metadata contract first, then propagate that additive contract through the OpenClaw plugin SDK runtime types, ACP control-plane metadata, `sessions_spawn` results, `session_status`, docs, changelog, and tests with explicit semantics for which process ID is exposed and how stale or reused identifiers are handled.

## Reproduction Assessment

Yes. Source inspection on current `main` is enough to reproduce the linked bug: ACP spawn results, ACP runtime types, and stored ACP metadata all omit typed PID/process metadata while docs still direct operators to manual `ps` correlation.

## Solution Assessment

No. The PR's direction is useful, but it is not the best fix because it reads top-level or `details.pid` fields that `acpx@0.6.1` does not emit; the safer path is a real ACPX process-metadata contract with documented exposure rules.
