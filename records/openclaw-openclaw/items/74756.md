## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested ACP PID surface, while the PR depends on PID fields that the pinned `acpx@0.6.1` runtime handle/status contract does not expose and it needs an explicit host-PID exposure policy before persisting and returning that metadata.

## What This Changes

The PR adds optional PID fields to ACP runtime/session metadata, attempts to extract PID values from ACPX delegate handle/status results, returns them from `sessions_spawn`, and adds PID-present/PID-absent unit coverage.

## Best Possible Solution

Add a typed ACPX process-metadata contract first, then propagate that additive contract through the OpenClaw plugin SDK runtime types, ACP control-plane metadata, `sessions_spawn` results, `session_status`, docs, changelog, and tests with explicit semantics for adapter PID versus child-agent PID and stale/reused identifiers.

## Reproduction Assessment

Yes. Source inspection on current main is enough: ACP spawn results, ACP runtime handle/status types, and persisted ACP metadata all omit typed PID/process metadata, while the ACP docs still require manual `ps` correlation.

## Solution Assessment

No. The PR's direction is useful, but it reads top-level or `details.pid` fields that `acpx@0.6.1` does not emit; the safer path is a real ACPX process-metadata contract with documented exposure rules before OpenClaw propagation.
