## Summary

Keep open. The linked bug is real on current main, but this PR is not merge-ready because it forwards PID fields that the pinned ACPX runtime does not actually expose, so real ACPX-backed spawns would still omit `pid`. The host PID exposure also needs an explicit API/security contract before it becomes agent-visible metadata.

## What This Changes

The PR adds optional PID fields to ACP runtime/session metadata, attempts to extract PID values from ACPX delegate handle/status results, persists them, returns them from `sessions_spawn`, and adds PID-present/PID-absent unit coverage.

## Best Possible Solution

Add a typed ACPX process-metadata contract first, then propagate that additive contract through the OpenClaw plugin SDK runtime types, ACP control-plane metadata, `sessions_spawn` results, `session_status`, docs, changelog, and integration-style tests with explicit semantics for adapter PID versus child-agent PID and stale identifiers.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the linked bug: ACP spawn results, ACP runtime handle/status types, and persisted ACP metadata all omit typed PID/process metadata, while the docs still require manual `ps` correlation.

## Solution Assessment

No. The PR’s direction is useful, but it is not the best fix because it reads top-level or `details.pid` fields that `acpx@0.6.1` does not emit; the safer path is a real ACPX process-metadata contract with documented exposure rules.
