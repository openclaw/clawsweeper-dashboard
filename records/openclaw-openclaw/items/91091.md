## Label Justifications

- `P2`: The PR fixes a live but bounded memory/session-state hazard where one transient filesystem error can wipe an index and force an expensive rebuild.
- `merge-risk: 🚨 compatibility`: The branch adds two public plugin SDK exports and changes failure behavior in shared sync and reindex paths.

## Summary

Current main still treats failed filesystem enumeration as an empty source and can destructively reconcile memory/session state from it; this PR applies a coherent authoritative-scan guard across the affected prune, reindex, retry, and export paths, but its protected maintainer status and public SDK expansion require explicit maintainer handling before merge.

## What This Changes

The PR adds authoritative session and memory-file scan results, prevents failed scans from driving destructive pruning, export reconciliation, or empty reindex replacement, retains dirty state for retry, and adds focused regression coverage.

## Best Possible Solution

Land the authoritative-scan invariant across every destructive memory/session consumer while preserving legacy list behavior for non-destructive callers, with explicit maintainer acceptance and final API-baseline verification for the two SDK exports.

## Reproduction Assessment

Yes. Current main converts filesystem enumeration failures into empty listings, and the full reconciliation and export paths treat those listings as deletion truth; the added regression scenario directly targets that source path.

## Solution Assessment

Yes at the behavioral level: destructive reconciliation must require an authoritative enumeration, while genuinely empty directories must still prune. The remaining maintainer decision is whether the two scan-result helpers should be permanent public SDK exports or a narrower memory-core seam.
