## Summary

Keep open: current main still has the failed-scan-as-empty-listing hazard, and the PR appears to address it without a discrete line-level correctness defect. It is maintainer-authored and has the protected `maintainer` label, and the remaining decision is whether to accept the new public SDK scan-result surface and session-state retention semantics before merge.

## What This Changes

Adds authoritative scan-result APIs for session and memory file enumeration and uses them to skip destructive pruning, reindex completion, and QMD export reconciliation after failed scans, with regression coverage.

## Best Possible Solution

Land the authoritative-scan guard after memory and plugin-SDK owners accept the public API boundary, with subtree failure reporting tracked separately against fs-safe.

## Reproduction Assessment

Yes from source inspection: current main converts non-missing scan failures into empty listings and destructive memory/session consumers can prune against those listings. I did not run tests in this read-only review, but the PR adds focused failing-main regression coverage for the hazard.

## Solution Assessment

Yes for the runtime behavior: carrying scan authority into destructive consumers is the narrow maintainable fix. The open solution question is whether the new scan-result helpers should be public plugin SDK contracts or internalized before merge.
