## What This Changes

The PR adds `bot.init()` before Telegram isolated polling drains spooled updates and updates polling-session tests to assert initialization precedes `handleUpdate`.

## Best Possible Solution

Land the narrow Telegram initialization fix after normal CI and maintainer review, then close https://github.com/openclaw/openclaw/issues/81973 through the merged PR.

## Reproduction Assessment

Yes. Current main's isolated ingress path drains updates with `handleUpdate` without the runner's automatic init path, and grammY 1.42.0 throws from `handleUpdate` until `init()` or constructor `botInfo` supplies bot info.

## Solution Assessment

Yes. Calling grammY's idempotent `bot.init()` before starting the isolated worker/drain is the narrow maintainable fix and keeps recoverable setup failures on the existing retry path.
