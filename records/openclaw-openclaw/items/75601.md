## Summary

Keep open. This draft PR has the protected `maintainer` label and the current patch is not a high-confidence fix for the linked Gateway event-loop saturation: it lowers the subagent lane cap, but current main still performs Gateway/agent preparation before the embedded run reaches that bounded lane. If maintainers accept changing the default, the docs and changelog credit also need to move with it.

## What This Changes

The PR lowers the default subagent max concurrency from 8 to 3, adds config-default unit coverage for the bounded default and override path, adds a changelog entry, and leaves a whitespace-only change in `src/agents/subagent-spawn.ts`.

## Best Possible Solution

Bound or offload the subagent startup/preparation phase before it can monopolize the Gateway event loop, or move that preparation under a bounded subagent startup lane. Keep the user override path, update the public subagent docs and changelog credit if the default changes, and validate against the #75378 three-parallel-spawn workload with event-loop delay/utilization and liveness outcome captured.

## Reproduction Assessment

Unclear. The linked #75378 report gives concrete v2026.4.29 steps and liveness logs for three simultaneous heavy-model `sessions_spawn` calls, and code review confirms the relevant prep/queue boundary on current main, but this review did not live-run that workload.

## Solution Assessment

No. Lowering `DEFAULT_SUBAGENT_MAX_CONCURRENT` is not the narrowest proven fix because it caps the embedded subagent command lane after Gateway/agent preparation has already begun; the safer fix is to gate or offload that preparation path and prove it with the reported workload.
