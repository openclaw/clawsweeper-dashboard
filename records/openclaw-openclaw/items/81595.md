## Summary

Keep open: current main still exposes `bundle-tools` as aggregate prep timing, and neither the related latency/duplicate-child issues nor the aggregate prep-stage PR provide per-MCP-server attribution.

## What This Changes

Request per-MCP-server timing sub-spans inside `bundle-tools` diagnostics so cold-start cost can be attributed to individual configured MCP servers.

## Best Possible Solution

Add gated per-server `mcp.tools_list` timeline spans at the bundle MCP catalog/listing boundary, preserve the aggregate `bundle-tools` prep-stage value, and keep diagnostic attributes redaction-safe.

## Reproduction Assessment

Yes. Source inspection shows MCP server connect/list work happens inside the bundle MCP runtime while current main only marks aggregate `bundle-tools` afterward, and the issue comments include a local timeline run that found no bundle/MCP child spans.

## Solution Assessment

Yes for the timeline-only part. Reusing the existing nested diagnostics span API is the narrow maintainable solution; a new compact log flag or public hook field should remain optional maintainer judgment.
