## What This Changes

The PR adds a gateway auth reason, detail code, and message for device-token scope mismatches, propagates `verifyDeviceToken`'s `scope-mismatch` result, updates a focused auth test, and adds a changelog entry.

## Best Possible Solution

Land the diagnostic propagation only after the new detail code is handled as a terminal re-pairable auth failure across the affected clients and the contributor supplies redacted real gateway/client evidence.

## Reproduction Assessment

Yes for the source-level seam: current main's verifier can return `scope-mismatch`, while `resolveConnectAuthDecision` ignores that field and emits the generic device-token failure path. I did not establish a live macOS/iOS/Control UI reconnect reproduction in this read-only review.

## Solution Assessment

No as currently patched: carrying the verifier reason forward is the right narrow server fix, but introducing a new public detail code needs matching client reconnect/pairing handling or it regresses the recovery path.
