## Summary

Keep open. Current main has related Telegram group-ID docs but not this multi-agent recipe, and the PR still has one blocking docs/security defect: the sample uses open group policy without `groups`, which current docs/runtime treat as unscoped group access.

## What This Changes

Adds a `docs/concepts/multi-agent.md` section showing two Telegram bot accounts bound to two agents in one shared group.

## Best Possible Solution

Keep the Telegram multi-agent recipe, but scope each account to the intended group chat and keep sender IDs under the documented Telegram sender allowlist fields.

## Reproduction Assessment

Yes for the docs defect. The PR-head sample has `groupPolicy: "open"` and no `groups` in either account block, while current docs/runtime show that this allows any group to pass chat checks.

## Solution Assessment

No, not as proposed. The docs direction is useful, but the sample should follow the current Telegram groups contract with per-account group scoping or an equivalent documented allowlist pattern.
