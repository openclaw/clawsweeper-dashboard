## Summary

Keep open: the docs recipe is useful and current main still lacks a dedicated multiple-Telegram-agents-in-one-group example, but this branch still teaches unscoped Telegram group access and lacks contributor-supplied real Telegram proof. The merged related PR at https://github.com/openclaw/openclaw/pull/80563 clarifies Telegram group IDs, but it does not supersede this multi-agent recipe.

## What This Changes

The PR adds a `docs/concepts/multi-agent.md` section showing two Telegram bot accounts routed to separate agents in one shared group, with config and setup notes.

## Best Possible Solution

Keep the dedicated recipe, but revise it so each Telegram account scopes the intended group with account-level `groups` or an equivalent allowlist pattern, align it with current Telegram group-ID docs, and add redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection reproduces the review issue: the PR sample uses `groupPolicy: "open"` without `groups`, while current docs and runtime show that this allows any group the bot is added to.

## Solution Assessment

No. The docs direction is useful, but the current PR is not the best final form until the sample scopes the intended Telegram group and the contributor supplies redacted real behavior proof.
