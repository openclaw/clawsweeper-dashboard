## Summary

Keep this PR open. Current main still lacks the dedicated Telegram same-group multi-agent walkthrough, and the contribution is useful, but the proposed sample is not merge-ready because it teaches an unscoped `groupPolicy: "open"` setup for a single intended group.

## What This Changes

The branch adds a `docs/concepts/multi-agent.md` section showing multiple Telegram bot accounts bound to separate agents in one group, including BotFather privacy-mode, sender allowlist, and group-policy setup notes.

## Best Possible Solution

Keep the docs contribution, but revise the example so each Telegram account explicitly scopes the shared group under `channels.telegram.accounts.<id>.groups`, while preserving the Privacy Mode remove/re-add note, sender-ID wording for `groupAllowFrom`, and per-account `groupPolicy` precedence guidance.

## Reproduction Assessment

Yes. Reproduce by comparing the PR sample to current `resolveChannelGroupPolicy`: `groupPolicy: "open"` with no `groups` leaves group allowlisting disabled, and Telegram multi-account configs do not inherit channel-level groups.

## Solution Assessment

No. The docs direction is useful, but the current diff is not the narrowest safe solution until the example includes account-level `groups` entries or an equivalent allowlist-scoped pattern for the intended Telegram group.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Restrict the sample to the intended Telegram group:** `docs/concepts/multi-agent.md:529`
  - body: The example is framed as one shared Telegram group, but `groupPolicy: "open"` with no `groups` entries lets any group the bots join pass group-ID checks. Add the shared group under each participating `channels.telegram.accounts.<id>.groups` entry, or otherwise make the example allowlist-scoped.
  - confidence: 0.93
