## Summary

Keep open: the docs addition fills a real Telegram multi-agent gap, but current main does not already contain it and the PR sample still teaches unscoped open group access; the external PR also has only CI/prose, not real behavior proof.

## What This Changes

The PR adds a `docs/concepts/multi-agent.md` section showing two Telegram bot accounts bound to separate agents in one group, with BotFather privacy-mode and `groupPolicy`/`groupAllowFrom` setup notes.

## Best Possible Solution

Keep the docs contribution, but make the sample explicitly scope each Telegram account to the intended group with `channels.telegram.accounts.<id>.groups` or an equivalent allowlist-scoped pattern before merge.

## Reproduction Assessment

Yes. Source inspection reproduces the issue by comparing the PR sample to the current Telegram docs, `resolveChannelGroupPolicy`, Telegram multi-account config merging, and existing tests for group inheritance.

## Solution Assessment

No. The docs direction is useful, but the current patch is not the best final form until the example scopes each account to the intended Telegram group instead of relying on unscoped open group policy.
