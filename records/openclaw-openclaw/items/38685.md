Keep open: the docs contribution fills a real Telegram multi-agent gap, but the current sample still teaches an unscoped open group policy and lacks contributor-provided real behavior proof.

## What This Changes

This PR adds a `docs/concepts/multi-agent.md` section showing two Telegram bot accounts bound to two agents in one group, with BotFather privacy-mode and group allowlist guidance.

## Best Possible Solution

Keep the contribution, but revise the sample so each Telegram account scopes the intended group under `channels.telegram.accounts.<id>.groups` or an equivalent scoped allowlist pattern, then add redacted live setup proof.

## Reproduction Assessment

Yes. Source inspection reproduces the review issue by comparing the PR sample to Telegram docs, `resolveChannelGroupPolicy`, account config merging, and tests for multi-account group inheritance.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Restrict the Telegram sample to the intended group:** `docs/concepts/multi-agent.md:529`
  - body: The section is framed around one shared Telegram group, but the example sets `groupPolicy: "open"` without `accounts.<id>.groups`. Current Telegram docs and runtime treat open-without-groups as allowing any group the bot is added to through chat checks, and multi-account Telegram accounts do not inherit channel-level groups, so users copying this can expose both agents outside the intended chat. Add account-level `groups` entries or an equivalent scoped allowlist pattern.
