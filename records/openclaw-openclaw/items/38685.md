## Summary

The PR should stay open because current main still does not include a Telegram multi-agent group recipe, but the proposed docs should not merge as-is: the sample opens group access without scoping either account to the intended Telegram group. The related merged Telegram group-ID docs clarify the needed contract, but they do not supersede this multi-agent example.

## What This Changes

The PR adds a new `docs/concepts/multi-agent.md` section showing how to configure multiple Telegram bot accounts for separate agents in one shared Telegram group.

## Best Possible Solution

Keep the dedicated Telegram multi-agent group recipe, but revise it to use the current Telegram contract: account-level `groups` for the intended chat, sender filtering via `allowFrom` / `groupAllowFrom`, and setup wording aligned with the merged group-ID clarification.

## Reproduction Assessment

Yes for the docs defect. Source inspection of the PR diff shows `groupPolicy: "open"` without account-level `groups`, and current docs/runtime/tests show `groups` is the Telegram chat allowlist while open policy permits group access after chat checks.

## Solution Assessment

No. The docs direction is useful, but the current PR sample is not the best mergeable solution because it should scope each account to the intended group using account-level `groups` or an equivalent documented allowlist pattern.
