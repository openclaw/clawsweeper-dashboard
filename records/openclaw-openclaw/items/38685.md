## Summary

Keep this PR open: current main still lacks the dedicated Telegram same-group walkthrough, and the docs contribution is useful, but the current diff remains incorrect because the sample uses `groupPolicy: "open"` without account-level `groups`, which conflicts with the current Telegram group-scoping contract.

## What This Changes

The PR adds a Telegram multi-account same-group setup example to `docs/concepts/multi-agent.md`, including BotFather privacy-mode notes, sender allowlist guidance, account bindings, and setup pitfalls.

## Best Possible Solution

Keep the docs contribution, but revise the example so each Telegram account explicitly scopes the shared group under `channels.telegram.accounts.<id>.groups` or uses an equivalent allowlist-scoped pattern.

## Reproduction Assessment

Yes. Source inspection reproduces the defect by comparing the PR sample to `resolveChannelGroupPolicy` and Telegram account merging: `groupPolicy: "open"` with no `groups` leaves chat allowlisting disabled, and multi-account Telegram configs do not inherit channel-level groups.

## Solution Assessment

No. The docs direction is useful, but the current patch is not the narrowest safe solution until the sample includes account-level `groups` entries for the intended Telegram group or an equivalent scoped allowlist pattern.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Restrict the sample to the intended Telegram group:** `docs/concepts/multi-agent.md:529`
  - body: The example is framed as one shared Telegram group, but `groupPolicy: "open"` with no `groups` entries lets any group the bots join pass group-ID checks. In multi-account Telegram setups, account configs do not inherit channel-level groups, so users copying this sample can expose both agents in unrelated groups. Add the shared group under each participating `channels.telegram.accounts.<id>.groups` entry, or otherwise make the example allowlist-scoped.
  - confidence: 0.93
