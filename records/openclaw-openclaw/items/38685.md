## Summary

Keep this PR open, but it still needs changes before merge. The Telegram multi-agent recipe is useful and is not already present on current main, but the sample still recommends unscoped group access for a one-group setup, which conflicts with the current Telegram group allowlist contract and can broaden real deployments.

## What This Changes

The PR adds a docs/concepts/multi-agent.md section showing two Telegram bot accounts bound to two agents in one shared group.

## Best Possible Solution

Keep the Telegram multi-agent recipe, but make the copy-paste sample scope each account to the intended group chat with account-level groups or an equivalent documented allowlist pattern.

## Reproduction Assessment

Yes for the docs defect by source inspection: the PR-head sample uses groupPolicy "open" without groups entries, while current docs and runtime show that admits any group where the bot is added before sender filtering.

## Solution Assessment

No, not as currently proposed. The docs direction is useful, but the sample should follow the existing Telegram group contract with per-account group scoping before merge.
