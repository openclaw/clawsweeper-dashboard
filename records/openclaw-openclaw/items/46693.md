## Summary

Keep this PR open, but do not merge it as-is. The OpenAI passthrough bug still exists on current main, and the linked analysis in #27515 is valid, but this branch is stale against the current model-aware resolver and would regress Bedrock/custom/Google cache eligibility if rebased directly.

## What This Changes

The PR extracts cache-retention resolution into a new helper, imports it from extra-params, re-exports it from the old Anthropic wrapper, and adds tests for explicit OpenAI/OpenAI-Codex cacheRetention passthrough.

## Best Possible Solution

Extend the existing src/agents/pi-embedded-runner/prompt-cache-retention.ts resolver to pass explicitly configured cacheRetention through for direct openai and openai-codex providers without adding an implicit default, while preserving current Anthropic, Anthropic Vertex, Bedrock, custom anthropic-messages, and Google eligibility rules and adding end-to-end OpenAI Responses payload coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: resolveCacheRetention({ cacheRetention: "long" }, "openai", "openai-responses", "gpt-5") returns undefined on current main, and applyExtraParamsToAgent only forwards cacheRetention when that resolver returns a value.

## Solution Assessment

No. The requested behavior is correct, but this PR is not the best current-main implementation because the provider-only helper loses modelApi/modelId eligibility gates; the safer fix is to extend the current resolver in place.
