## Summary

Keep open: the OpenAI cache-retention passthrough gap still exists on current main, but this PR is stale against the current model/API-aware resolver and would drop important eligibility gates if rebased as-is.

## What This Changes

The PR extracts cache-retention resolution into a provider-neutral helper, re-exports the old Anthropic symbol, imports the helper from extra-params, and adds OpenAI/OpenAI-Codex explicit passthrough tests.

## Best Possible Solution

Extend the current prompt-cache-retention resolver in place to pass explicit cacheRetention through for direct openai and eligible openai-codex routes without adding defaults, while preserving Anthropic-family, Bedrock, custom anthropic-messages, Google, and native Codex unsupported-param rules.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: resolveCacheRetention({ cacheRetention: 'long' }, 'openai', 'openai-responses', 'gpt-5') returns undefined on current main, and applyExtraParamsToAgent only forwards cacheRetention when that resolver returns a value.

## Solution Assessment

No. The requested behavior is correct, but this PR is not the best current-main solution because the provider-only helper would discard current model/API eligibility gates; extending the existing resolver is narrower and safer.
