## Summary

Keep open. Current main still has the reported contract mismatch: the docs say `tools.codeMode` works in agent or runtime config, but the per-agent schema and type omit `codeMode`, and the current runtime helper reads only top-level `config.tools.codeMode`. The report is tied to the generic code-mode work from https://github.com/openclaw/openclaw/pull/80600 and looks like a narrow config-contract bug, not stale cleanup.

## What This Changes

The issue asks for documented agent-level `tools.codeMode` configuration to validate and enable code mode, or for the docs to be corrected to top-level-only.

## Best Possible Solution

Honor the documented per-agent contract by adding `codeMode` to the per-agent config schema/types and making code-mode activation resolve the active agent's tools override, with focused schema and runtime regression coverage.

## Reproduction Assessment

Yes from source inspection, but not by executing the gateway in this read-only pass. `agents.list[].tools` is validated by a strict `AgentToolsSchema` that lacks `codeMode`, so the reported config shape will reject before startup.

## Solution Assessment

Yes, per-agent support is the best fit because the public docs already promise agent-level enablement. The fix should cover schema, TypeScript config types, and runtime resolution; only changing the docs would remove a useful documented behavior rather than repairing it.
