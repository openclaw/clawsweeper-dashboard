## Label Justifications

- `P1`: A stalled optional memory lookup can block a live Telegram turn and make the bot appear dead to users.
- `impact:crash-loop`: The issue is about a hang/liveness failure in the live turn pipeline rather than a harmless empty memory result.
- `impact:message-loss`: The reported failure mode is that the user receives no assistant reply after `memory_search` is invoked.

## Summary

Keep open: current main and the latest release still have unbounded active memory-core `memory_search` setup/search awaits, and the open closing PR is conflicting and targets deleted legacy tool files instead of the active plugin path.

## What This Changes

The issue asks for `memory_search` to fail open with a hard timeout so memory manager setup or search stalls cannot wedge a live turn.

## Best Possible Solution

Port or replace the linked PR with an `extensions/memory-core` fix that wraps lazy load, manager setup, and search in a hard fail-open deadline and adds regression coverage for never-settling setup and search promises.

## Reproduction Assessment

Yes, for a source-level reproduction: make `getMemorySearchManager` or `memory.manager.search` return a never-settling promise and the active `extensions/memory-core` tool path continues awaiting it. I did not run the live Telegram/iCloud setup in this read-only review.

## Solution Assessment

Yes, the issue direction is the right fix: the active memory-core tool boundary needs a hard fail-open deadline. The current linked PR should be ported or replaced because it targets deleted legacy files rather than the active implementation.
