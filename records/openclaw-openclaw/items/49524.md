## Summary

Keep open: current main still has an unbounded active memory-core memory_search path, and an open linked implementation PR exists but has not landed and appears to target the retired legacy tool path.

## What This Changes

The issue asks for memory_search to fail open with an unavailable tool result instead of stalling a live session when memory setup or search never completes.

## Best Possible Solution

Port or replace the linked PR with a memory-core fix that wraps runtime loading, manager acquisition, and active search in a hard fail-open deadline, returns timeout-specific unavailable metadata, and adds regression coverage for never-settling setup and search promises.

## Reproduction Assessment

Yes, for a source-level reproduction: a stubbed getMemorySearchManager or memory.manager.search that returns a never-settling promise will keep the current memory_search execute path awaiting forever. I did not run a live Telegram/iCloud setup in this read-only review.

## Solution Assessment

Yes for the issue direction: a hard fail-open deadline at the active memory-core tool boundary is the narrowest maintainable fix. The linked PR should not land as-is because its current diff targets retired legacy files rather than the active memory-core implementation.
