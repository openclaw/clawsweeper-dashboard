## Summary

Keep open: current main still awaits memory manager setup and search without a hard memory-tool deadline, and open PR #49794 is the active unmerged implementation candidate that links this issue as its close target.

## What This Changes

Require memory_search to fail open with a bounded unavailable tool result when memory manager initialization, QMD setup, or search execution stalls.

## Best Possible Solution

Review and land PR #49794, or replace it with an equivalent memory-core fix that wraps the full memory_search tool boundary in a hard deadline and always returns the existing unavailable payload with regression coverage for never-settling setup and search promises.

## Reproduction Assessment

Yes for a source-level reproduction: a stubbed getMemorySearchManager or memory.manager.search that returns a never-settling promise will keep the current memory_search execute path awaiting forever. I did not run a live Telegram/iCloud reproduction because this review is read-only.

## Solution Assessment

Yes, a hard fail-open deadline around the memory_search tool boundary is the narrowest maintainable fix. A generic transcript/tool-result watchdog would not cover the current direct awaits inside the memory tool itself.
