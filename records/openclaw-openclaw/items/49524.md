## Summary

Keep open: current main and the latest release still await the active `memory_search` setup/search path without an outer fail-open deadline, and the linked closing PR is still open/conflicting while targeting retired legacy files.

## What This Changes

The issue requests a fail-open `memory_search` boundary so stalled memory manager setup or search cannot wedge a live agent turn.

## Best Possible Solution

Port or replace the linked PR with a memory-core fix that wraps the full active tool boundary in a hard fail-open deadline and adds stalled-promise regression coverage.

## Reproduction Assessment

Yes, for a source-level reproduction: a stubbed runtime load, `getMemorySearchManager`, or `memory.manager.search` that never settles will keep the active memory-core execute path pending. I did not reproduce the live Telegram/iCloud setup.

## Solution Assessment

Yes for the issue direction: a hard fail-open deadline at the active memory-core tool boundary is the narrowest maintainable fix. The current linked PR should be ported or replaced because it changes retired legacy files rather than the active implementation.
