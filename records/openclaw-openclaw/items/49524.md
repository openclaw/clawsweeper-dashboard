## Summary

Keep open: current main and the latest release still await memory manager setup and `memory.manager.search()` without a tool-boundary deadline, so a never-settling memory path can still withhold the tool result and stall the live turn. The target-repo timeout PRs are closed and unmerged; the only open closing reference is a fork PR, not an active OpenClaw landing path.

## What This Changes

The issue requests a bounded fail-open `memory_search` path that always returns an unavailable tool result instead of stalling a live session when memory setup or search hangs.

## Best Possible Solution

Add a bounded fail-open wrapper at the active memory-core tool boundary so lazy runtime load, manager setup, memory search, and optional supplement search always resolve to a structured unavailable tool result, with regression coverage for never-settling setup and search promises.

## Reproduction Assessment

Yes, at source level: make `getMemorySearchManager`, lazy memory runtime loading, or `memory.manager.search` return a never-settling promise and the active memory-core tool awaits it without a deadline. I did not run the live Telegram/iCloud setup in this read-only review.

## Solution Assessment
