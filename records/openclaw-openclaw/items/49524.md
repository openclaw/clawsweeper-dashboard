## Label Justifications

- `impact:message-loss`: The reported failure mode is that the user receives no assistant reply after memory_search is invoked.
- `impact:crash-loop`: The issue is about a hang/liveness failure in the live turn pipeline rather than a harmless empty memory result.

## Summary

Keep open: current main still awaits the active memory-core manager setup and search path without a tool-boundary deadline, so a never-settling memory_search can still withhold the tool result and stall a live turn. The prior target-repo PRs are closed and the remaining cross-repo fork reference is not a mergeable OpenClaw PR, so the issue remains actionable in this repository.

## What This Changes

The issue asks for memory_search to fail open with unavailable/timeout metadata instead of wedging a live session when memory manager setup or search stalls.

## Best Possible Solution

Add an active memory-core tool-boundary timeout/fail-open wrapper around lazy runtime load, manager setup, and search, with timeout-specific unavailable metadata and regression coverage, without reviving the deleted legacy tool path.

## Reproduction Assessment

Yes, at source level: make getMemorySearchManager or memory.manager.search return a never-settling promise and the active memory-core tool awaits it without a deadline. I did not run the live Telegram/iCloud setup in this read-only review.

## Solution Assessment

Yes, the fail-open timeout belongs at the active memory-core tool boundary. The old PR direction was conceptually useful, but it targeted deleted legacy files and should be ported or replaced in extensions/memory-core.
