- `impact:message-loss`: The reported failure mode is that the user receives no assistant reply after memory_search is invoked.

## Summary

Keep open: current main still has a source-reproducible memory-core path where manager setup and search are awaited without a tool-boundary deadline, so a never-settling memory_search can withhold the tool result and stall a live turn. The target-repo timeout PRs are closed and unmerged; the remaining open closing reference is a fork PR that edits the deleted legacy src/agents/tools path.

## What This Changes

The issue asks for memory_search to fail open with an unavailable tool result instead of wedging a live session when memory setup or search stalls.

## Best Possible Solution

Add a bounded memory-core tool-boundary timeout/fail-open wrapper around lazy runtime load, manager setup, and search, with timeout-specific unavailable metadata and regression coverage in the active extensions/memory-core path.

## Reproduction Assessment

Yes, at source level: make getMemorySearchManager or memory.manager.search return a never-settling promise and the active memory-core tool awaits it without a deadline. I did not run the live Telegram/iCloud setup in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs at the active memory-core tool boundary so setup and search always produce a fail-open tool result; the old PR direction was conceptually useful but targeted deleted legacy files.
