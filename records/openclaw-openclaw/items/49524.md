## Label Justifications

- `P1`: A stalled optional memory lookup can block a live Telegram turn and make the bot appear dead to users.
- `impact:message-loss`: The reported failure mode is that no assistant reply is delivered after `memory_search` is invoked.
- `impact:crash-loop`: The issue is about a hang/liveness failure in the live turn pipeline rather than a harmless empty memory result.

## Summary

Keep open: current main and the latest release still have an active `memory_search` path that awaits memory manager setup and `manager.search()` without a tool-boundary deadline, so a never-settling memory path can still withhold the tool result. The earlier target-repo timeout PRs are closed and unmerged, and the only open closing reference is a fork PR rather than a target-repo landing path.

## What This Changes

The issue requests that `memory_search` fail open with a bounded unavailable result instead of stalling an assistant turn when memory setup or search never settles.

## Best Possible Solution

Add a bounded fail-open wrapper at the active memory-core tool boundary so lazy runtime load, manager setup, memory search, and optional supplement search always resolve to a tool result without touching deleted legacy tool files.

## Reproduction Assessment

Yes, at source level: make `getMemorySearchManager` or `memory.manager.search` return a never-settling promise and the active memory-core tool awaits it without a deadline. I did not run the live Telegram/iCloud setup in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is a fail-open timeout at the active memory-core tool boundary, with regression coverage for manager setup and search promises that never settle.
