## Label Justifications

- `P1`: A stalled optional memory lookup can block a live Telegram turn and make the bot appear dead to users.
- `impact:message-loss`: The reported failure mode is that the user receives no assistant reply after memory_search is invoked.
- `impact:crash-loop`: The issue is about a hang/liveness failure in the live turn pipeline rather than a harmless empty memory result.

## Summary

Keep open: current main still has an active memory-core path where manager setup and search are awaited without a tool-boundary deadline, so a never-settling memory_search can still withhold the tool result and stall the live turn. The target-repo timeout PRs are closed, and the only open closing reference is a fork PR against a different repository path that edits files no longer present on main.

## What This Changes

The issue asks for memory_search to fail open with unavailable metadata instead of stalling a live session when memory manager setup or search does not complete.

## Best Possible Solution

Add a memory-core tool-boundary timeout/fail-open wrapper around lazy runtime load, manager setup, and search, with timeout-specific unavailable metadata and regression coverage, without reviving the deleted legacy src/agents/tools implementation.

## Reproduction Assessment

Yes, at source level: make getMemorySearchManager or memory.manager.search return a never-settling promise and the active memory-core tool awaits it without a deadline. I did not run the live Telegram/iCloud setup in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is at the active memory-core tool boundary so setup and search always produce a fail-open tool result. The old PR direction was conceptually useful, but it targeted deleted legacy files and should be ported or replaced in extensions/memory-core.
