## Summary

Keep open. Current main still has an unbounded memory tool path for manager setup and search execution, and open PR #49794 is the active unmerged implementation candidate that links this issue as its close target.

## What This Changes

Request a hard fail-open timeout for `memory_search` so a stalled memory manager or search backend cannot wedge a live agent turn.

## Best Possible Solution

Land PR #49794 or an equivalent narrow memory-core fix that wraps the full `memory_search` boundary, including runtime load, manager context creation, QMD pending creation, builtin manager creation, and `manager.search`, in a hard deadline that always returns the existing unavailable payload. Add fake-timer regression coverage for never-settling manager-init and search promises.

## Reproduction Assessment

Yes for a code-level reproduction: stub `getMemoryManagerContext`, pending QMD creation, `MemoryIndexManager.get`, or `memory.manager.search` to return a never-settling promise and call `memory_search`; current main awaits those promises without a hard deadline. A live Telegram/iCloud reproduction was not run in this read-only review.

## Solution Assessment

Yes, the hard fail-open memory tool boundary is the narrowest maintainable direction. Generic transcript or tool-result repair is not enough because current tool execution still awaits `tool.execute(...)`, and the memory tool does not consume the passed abort signal while awaiting manager setup and search.
