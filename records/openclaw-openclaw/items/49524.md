## Label Justifications

- `P1`: A stalled optional memory lookup can block a live channel turn and make the assistant appear dead to users.
- `impact:message-loss`: The reported failure mode is that the assistant invokes `memory_search` and no final reply is delivered.
- `impact:crash-loop`: The issue is about a hang/liveness failure in the live turn pipeline rather than a harmless empty memory result.

## Summary

Keep open: current main and `v2026.5.22` still await memory manager setup and `memory.manager.search()` without a tool-boundary deadline, so a never-settling memory path can still withhold the tool result and stall a live turn.

## What This Changes

The issue reports that a stalled `memory_search` call can block a live session instead of returning a fail-open unavailable tool result.

## Best Possible Solution

Add a narrow memory-core tool-boundary deadline around manager acquisition and search, return the existing unavailable payload on timeout, and cover a never-resolving manager/search promise with deterministic regression tests.

## Reproduction Assessment

Yes, at source level: make `getMemoryManagerContext()` or `memory.manager.search()` return a never-settling promise and the active memory-core tool awaits it without a deadline. I did not run the original live Telegram/iCloud setup in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is a fail-open timeout at the active memory-core tool boundary, with regression coverage for manager setup and search promises that never settle.
