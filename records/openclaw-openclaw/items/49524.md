## Label Justifications

- `P1`: A stalled optional memory lookup can block a live Telegram turn and make the bot appear dead to users.
- `impact:message-loss`: The reported failure mode is that the user receives no assistant reply after memory_search is invoked.
- `impact:crash-loop`: The issue is about a hang/liveness failure in the live turn pipeline rather than a harmless empty memory result.

## Summary

Keep open: current main and the latest shipped tag still await the active memory-core manager setup/search path without a tool-boundary deadline, so a never-settling memory_search can still withhold the tool result and stall a live turn.

## What This Changes

The issue asks for memory_search to fail open with timeout metadata so optional memory failures cannot stall a live session.

## Best Possible Solution

Port or replace the linked PR at https://github.com/openclaw/openclaw/pull/49794 with an extensions/memory-core fix that wraps lazy runtime load, manager setup, and search in a fail-open deadline with timeout-specific unavailable metadata and regression coverage.

## Reproduction Assessment

Yes, at source level: make getMemorySearchManager or memory.manager.search return a never-settling promise and the active memory-core tool awaits it without a deadline. I did not run the live Telegram/iCloud setup in this read-only review.

## Solution Assessment

Yes, the requested fail-open timeout is the right boundary for this bug. The current linked PR should be ported or replaced because it targets deleted legacy files rather than the active memory-core implementation.
