## Summary

Keep open: current `main` still has an unbounded active `memory_search` path in the `memory-core` plugin, and the linked implementation PR is open, conflicting, and targets the retired legacy tool files rather than the active implementation.

## What This Changes

The issue reports that a stalled `memory_search` call can wedge a live session instead of failing open with an unavailable tool result.

## Best Possible Solution

Port or replace the linked PR with a memory-core fix that wraps the full active tool boundary in a hard fail-open deadline and adds regression coverage for stalled setup and search promises.

## Reproduction Assessment

Yes, for a source-level reproduction: a stubbed `getMemorySearchManager` or `memory.manager.search` that returns a never-settling promise will keep the active `extensions/memory-core/src/tools.ts` execute path awaiting forever. I did not run a live Telegram/iCloud setup in this read-only review.

## Solution Assessment

Yes for the issue direction: a hard fail-open deadline at the active memory-core tool boundary is the narrowest maintainable fix. The linked PR should not land as-is because its current diff targets retired legacy files rather than the active memory-core implementation.
