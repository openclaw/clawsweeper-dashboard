## Label Justifications

- `P1`: A stalled memory_search can block a live agent/channel turn and make the bot appear dead to users.
- `impact:crash-loop`: The report is about a liveness hang in the turn pipeline rather than a simple empty-result memory failure.
- `impact:message-loss`: The user-visible symptom is that the live Telegram turn produced no reply even though the gateway and token were still healthy.
- `impact:session-state`: The report says the active session appeared wedged until manual intervention.

## Summary

Keep open: current main still has unbounded active memory_search setup and search awaits, and the linked closing PR at https://github.com/openclaw/openclaw/pull/49794 is open and unmerged. A never-settling memory manager or search promise can still withhold the tool result and leave the turn waiting.

## What This Changes

Issue requests fail-open, timeout-protected memory_search behavior so memory retrieval stalls cannot wedge a live agent turn.

## Best Possible Solution

Add a hard fail-open deadline around the active memory-core tool boundary, including setup and search, with regression coverage for never-settling manager setup and search promises.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: stub getMemorySearchManager or memory.manager.search to return a never-settling promise and the active memory-core tool path will await forever. I did not run the live Telegram/iCloud setup in this read-only review.

## Solution Assessment

Yes, the requested hard fail-open deadline at the active memory-core tool boundary is the narrow maintainable fix. The linked PR should be reviewed, ported, replaced, or closed based on whether it now targets extensions/memory-core rather than retired legacy files.
