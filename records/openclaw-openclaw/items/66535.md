## Summary

Keep open. Current main and the latest release still start manual `/compact` without an abort signal or active embedded-run registration, so the existing `/stop` and embedded-run abort paths cannot reliably discover or cancel that compaction; the linked fix PRs are all closed and unmerged.

## What This Changes

The issue asks for manual `/compact` runs to be cancelable through the existing chat abort, `/stop`, and `abortEmbeddedPiRun(sessionId)` paths.

## Best Possible Solution

Register manual `/compact` as a compacting active embedded run with a local `AbortController`, pass that signal into `compactEmbeddedPiSession`, clear the handle in `finally`, and add focused cancellation regression coverage.

## Reproduction Assessment

Yes, at source level. Current main shows manual `/compact` starts compaction without an `abortSignal` or active-run registration, while `/stop` and direct embedded-run abort only reach registered handles or active reply runs.

## Solution Assessment

Yes. Reusing the existing `abortSignal` parameter plus the active embedded-run registry is the narrow maintainable fix; broader scheduling and shared-signal behavior should remain separate.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/66535. On current main, manual `/compact` in `src/auto-reply/reply/commands-compact.ts` invokes `runtime.compactEmbeddedPiSession` without an `abortSignal` and without registering the compaction in the active embedded-run registry, so `/stop`, `chat.abort`, and `abortEmbeddedPiRun(sessionId)` cannot cancel it. Keep the scope to manual `/compact` cancellation: expose or reuse the existing `setActiveEmbeddedRun` and `clearActiveEmbeddedRun` lifecycle helpers through the compact runtime boundary, create a local `AbortController`, register a compacting non-streaming handle after any prior run has drained, pass its signal to `compactEmbeddedPiSession`, and clear the exact handle in `finally`. Do not change broader compaction lane scheduling, do not add a config option, and do not share the main reply operation abort signal. Add focused tests in `commands-compact.test.ts` and, if needed, `runs.test.ts`; add a user-facing changelog entry if the repair lands.
