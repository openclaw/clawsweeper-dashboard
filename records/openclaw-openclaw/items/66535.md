## Summary

Keep open. Current main still starts manual `/compact` without passing a cancellation signal or registering the compaction as an active embedded run, so `/stop`, `chat.abort`, and direct embedded-run abort cannot reliably discover or interrupt it; the linked fix PRs are closed and unmerged.

## What This Changes

The issue asks for manual `/compact` compactions to be cancelable through the existing embedded-run abort paths.

## Best Possible Solution

Wire manual `/compact` into the existing active-run lifecycle: register an abortable compacting handle while work is in flight, pass a local abort signal into compaction, clean up in `finally`, and keep broader scheduling or preflight-abort work separate.

## Reproduction Assessment

Yes, at source level. Current main shows `/compact` starts compaction without an abort signal or active-run registration, while `/stop` and `abortEmbeddedPiRun(sessionId)` only reach registered handles or active reply runs; no live long-running compaction was executed.

## Solution Assessment

Yes. Reusing the existing `abortSignal` parameter plus `setActiveEmbeddedRun`/`clearActiveEmbeddedRun` is the narrow maintainable path; the repair should avoid unrelated launchd/profile changes and leave broader compaction scheduling issues to their own threads.
