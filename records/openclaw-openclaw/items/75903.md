## Summary

Keep open: current main still concurrently fans out Telegram account startup into the lazy polling runtime path, and I found no targeted fix, changelog entry, or canonical duplicate for the reported cold-start `register/babel.cjs` errors.

## What This Changes

Fix transient Telegram startup `register/babel.cjs` module-not-found warnings when multiple configured accounts cold-start concurrently.

## Best Possible Solution

Add a narrow first-load readiness barrier around the proven cold path, with regression coverage for multi-account Telegram startup and without reintroducing runtime dependency repair during gateway startup.

## Reproduction Assessment

Yes. The issue gives concrete fresh packaged-startup steps with at least three Telegram accounts, and current main still has the concurrent account fanout needed to exercise that path; I did not run the live packaged repro.

## Solution Assessment

Unclear. A Telegram polling/runtime warmup guard is likely safer than serializing all channel startup, but the patch should first identify whether the failure originates in Telegram lazy imports or the shared Jiti/runtime-deps loader.
