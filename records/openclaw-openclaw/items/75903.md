## Summary

Keep open: current main still starts configured channel accounts concurrently through the Telegram monitor/polling path, and I did not find a merged fix, changelog entry, or canonical duplicate for the reported `register/babel.cjs` startup race.

## What This Changes

The issue asks OpenClaw to eliminate transient Telegram startup errors when multiple accounts concurrently hit a cold Babel/Jiti register dependency path.

## Best Possible Solution

Add a narrow first-load readiness barrier for the proven cold path, preferably Telegram polling/bot warmup unless investigation shows the shared plugin runtime-dependency loader needs the guard.

## Reproduction Assessment

Yes. The issue gives concrete steps with at least three Telegram accounts and a fresh packaged gateway, and current main still contains the concurrent account fanout needed to exercise that path; I did not execute the live packaged repro.

## Solution Assessment

Unclear. Serializing only Telegram's first cold polling/bot dependency path is likely safer than global channel serialization, but the repair should first prove whether the failure originates in Telegram lazy imports or packaged runtime-deps staging.

Investigate and fix https://github.com/openclaw/openclaw/issues/75903. Current main still starts all accounts for a channel via `Promise.all` in `src/gateway/server-channels.ts`, and Telegram `startAccount` enters `resolveTelegramMonitor()` per account. Reproduce or model the first cold packaged startup path that can emit transient `Cannot find module .../register/babel.cjs` failures when several Telegram accounts start in parallel. Prefer a narrow fix: serialize or pre-warm only the first Telegram polling/bot dependency path, or add a runtime-dependency readiness barrier if that is the proven root cause.
