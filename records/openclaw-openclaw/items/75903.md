## Summary

Keep open: current main still starts configured Telegram accounts concurrently into the per-account monitor/bot path, and I did not find a merged fix or canonical duplicate that specifically covers the `register/babel.cjs` multi-account cold-start race.

## What This Changes

Investigate and prevent transient Telegram startup `register/babel.cjs` module failures when several configured accounts initialize concurrently.

## Best Possible Solution

Add a narrow, tested cold-start barrier for the Telegram monitor/bot loader or the relevant packaged dependency readiness path, while preserving normal parallel account startup after the first load is warm.

## Reproduction Assessment

Yes. The issue gives a concrete path: configure at least three Telegram accounts, restart a fresh packaged gateway after dependency preparation, and watch for duplicate `register/babel.cjs` failures; current main still has the concurrent fanout that makes this reproducible path plausible.

## Solution Assessment

Unclear. A scoped Telegram first-load gate is likely safer than globally serializing all channel account startup, but the fix should first prove whether the race is in Telegram bot import warmup or packaged runtime-deps visibility.
