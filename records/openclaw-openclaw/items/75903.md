## Summary

Keep open: current main still starts all configured channel accounts concurrently and each Telegram account reaches the monitor/polling runtime path without a first-load gate, so the reported cold-start race remains plausible and unproven as fixed.

## What This Changes

The issue reports transient Telegram startup `register/babel.cjs` module-load errors when multiple configured Telegram accounts cold-start concurrently, and asks for first-load serialization or pre-warm.

## Best Possible Solution

Add a narrow Telegram cold-start gate or runtime-dependency readiness barrier that makes the first polling/bot loader path deterministic while preserving normal parallel startup once the dependency path is warm.

## Reproduction Assessment

Yes. The issue gives concrete steps with three or more Telegram accounts and a fresh packaged startup, and current main still contains the concurrent account fanout that would exercise that path.

## Solution Assessment

Unclear. Serializing or pre-warming only Telegram's first cold polling/runtime load is likely the narrowest maintainable fix, but the repair should first prove whether the root cause is the Telegram runtime import or packaged dependency staging.
