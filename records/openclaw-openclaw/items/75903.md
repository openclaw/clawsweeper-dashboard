## Summary

Keep open: current main still has the Telegram multi-account startup path fan out through concurrent account tasks, and I did not find a merged fix or canonical duplicate that specifically prevents the reported transient `register/babel.cjs` startup race.

## What This Changes

The issue asks OpenClaw to avoid transient Telegram startup `register/babel.cjs` failures when multiple configured Telegram accounts bootstrap together.

## Best Possible Solution

Add a narrow Telegram cold-start gate or runtime-dependency readiness barrier that makes the first polling/bot loader path deterministic, while preserving normal parallel startup after the dependency path is warm.

## Reproduction Assessment

Yes, the issue gives concrete steps: configure at least three Telegram accounts, restart a fresh packaged gateway after dependency preparation, and watch for duplicate `register/babel.cjs` failures. I did not execute that live path, but current main still contains the concurrent account fanout that makes the report plausible.

## Solution Assessment

Unclear: serializing only Telegram's first cold-start dependency path is likely safer than globally serializing all channel accounts, but the repair should first prove whether the failing edge is Telegram bot import warmup or packaged runtime-deps staging.
