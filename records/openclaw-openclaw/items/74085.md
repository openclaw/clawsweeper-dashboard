## Summary

Keep open. Current main still awaits provider usage auth resolution before JSON status can emit output, and that auth path is not bounded by the existing timeout; open PR #74185 already targets this exact gap with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for `openclaw status --usage --json` to return machine-readable JSON from non-TTY subprocesses again instead of hanging or exiting with empty output after the 2026.4.26 regression.

## Best Possible Solution

Land PR #74185 or an equivalent narrow change that bounds provider usage auth resolution before JSON status output waits on it, returns parseable degraded JSON on auth timeout, and adds regression coverage for a never-settling auth resolver.

## Reproduction Assessment

Yes. The issue includes a concrete non-TTY `spawnSync('openclaw', ['status','--usage','--json'])` reproduction, and current main still has a source path where JSON output waits on unbounded provider usage auth before writing stdout.

## Solution Assessment
