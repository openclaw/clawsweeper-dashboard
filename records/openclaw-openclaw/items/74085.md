## Summary

Keep open. Current main still routes `openclaw status --usage --json` through a JSON path that waits for provider usage auth before writing stdout, and that auth resolution is not bounded by the existing provider usage timeout. Open PR #74185 already targets this exact gap with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for `openclaw status --usage --json` to return machine-readable JSON reliably from non-TTY subprocess automation instead of hanging or exiting without output.

## Best Possible Solution

Review and land PR #74185, or an equivalent narrow change, so provider usage auth resolution is bounded by the existing timeout budget and JSON status returns parseable degraded output instead of hanging when auth resolution never settles.

## Reproduction Assessment

Yes. The issue includes a concrete non-TTY `spawnSync('openclaw', ['status','--usage','--json'])` reproduction, and current main still has a source path where JSON output waits on unbounded provider usage auth before stdout is written.

## Solution Assessment

Yes. Keeping the issue open while PR #74185 or an equivalent focused auth-timeout fix is reviewed is the narrow maintainable path; broader status rewrites or TTY-specific branching are unnecessary.
