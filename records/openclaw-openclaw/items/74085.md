## Summary

Keep open. Current main still waits for provider usage auth resolution before `status --usage --json` can emit JSON, and open PR #74185 is already the paired implementation candidate with closing syntax for the same auth-timeout gap.

## What This Changes

Make `openclaw status --usage --json` reliably return machine-readable JSON from non-TTY subprocesses by bounding provider usage/auth work.

## Best Possible Solution

Review and land PR #74185 or an equivalent narrow change that bounds provider usage auth resolution before JSON output waits on it, returns parseable degraded JSON on auth timeout, and adds a regression test where auth resolution never settles.

## Reproduction Assessment

Yes. The issue provides a concrete non-TTY `spawnSync('openclaw', ['status','--usage','--json'])` reproduction, and current main still has a source path where JSON output waits for unbounded usage auth resolution before writing output.

## Solution Assessment

Yes. The narrow maintainable fix is to bound `resolveProviderAuths` in the usage-summary path and test the never-settling auth case; current main has not done that yet, and PR #74185 is the active candidate.
