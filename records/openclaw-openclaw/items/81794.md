## Summary

Keep open: this PR targets a plausible Control UI gateway feature, but the proposed Accept-Encoding negotiation violates q-value rules and the external PR does not include after-fix real behavior proof.

## What This Changes

The PR adds gzip/Brotli negotiation for Control UI static file responses and adds focused HTTP handler tests for gzip, HEAD, and image behavior.

## Best Possible Solution

Keep the feature in the gateway static-file path, but implement q-aware Accept-Encoding selection, add edge-case tests for wildcard and q ordering, and require real curl/browser proof before merge.

## Reproduction Assessment

Yes for the PR defects by source inspection: selectStaticAssetEncoding uses a Set and hard-codes Brotli before gzip, so headers like `gzip;q=0, *;q=1` and `br;q=0.1, gzip;q=1` select the wrong coding. The linked feature gap has a curl repro in the issue, but I did not run the gateway locally in this read-only review.

## Solution Assessment

No. The gateway Control UI file-serving path is the right place for this feature, but this implementation needs q-value-aware negotiation before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Honor explicit q=0 before wildcard fallback:** `src/gateway/control-ui.ts:768-769`
  - body: `Accept-Encoding: gzip;q=0, *;q=1` explicitly rejects gzip while allowing other unspecified codings, but this fallback still selects gzip because `*` is present. Track explicit q-values so wildcard matching cannot re-enable a coding the client marked unacceptable.
  - confidence: 0.93
- **[P2] Choose encoding by q value:** `src/gateway/control-ui.ts:765-767`
  - body: This always returns Brotli whenever `br` appears, even when gzip has a higher q-value such as `br;q=0.1, gzip;q=1`. Parse and compare q-values so the selected compression matches the client's stated preference.
  - confidence: 0.9
