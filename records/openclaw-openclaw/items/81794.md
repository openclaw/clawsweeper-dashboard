## Summary

Keep open: the compression feature is still missing from current main and the PR has strong live HTTP proof, but the patch should not merge until the new compression work is moved off the synchronous Gateway request path.

## What This Changes

The PR adds gzip/Brotli Accept-Encoding negotiation for Control UI static files and regression tests for gzip, Brotli, q-value, HEAD, wildcard, and binary passthrough behavior.

## Best Possible Solution

Serve negotiated gzip/Brotli from the Control UI handler using cached or async/streamed compression while preserving the current routing, CSP, and file-safety boundaries.

## Reproduction Assessment

Yes. Current main's static serving helpers send raw file bodies without negotiated Content-Encoding, and the linked report provides a curl path that observes the missing compression.

## Solution Assessment

No. The Control UI handler is the right boundary, but the current implementation uses sync zlib on the request path; the safer implementation is cached or async/streamed compression with the same negotiation behavior.

- **[P2] Avoid synchronous compression on the request path:** `src/gateway/control-ui.ts:803-811`
  - body: `compressStaticAssetBody` is called from each negotiated Control UI static response, but it uses `brotliCompressSync` and `gzipSync` on the shared Gateway HTTP path. Use cached variants or async/streamed zlib before sending the encoded response. ([nodejs.org](https://nodejs.org/docs/latest-v22.x/api/zlib.html))
