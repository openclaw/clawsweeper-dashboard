## Summary

Keep open: the feature is useful and current main still serves Control UI static assets without compression, but this PR's Accept-Encoding negotiation violates q-value/wildcard semantics and the proof remains mock-only.

## What This Changes

The branch adds gzip/Brotli negotiation for Control UI static files in `src/gateway/control-ui.ts` and focused HTTP handler tests.

## Best Possible Solution

Implement q-aware content-coding negotiation in the Control UI static-file path, add wildcard/q=0/preference regression tests, then merge after real curl or browser proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: `selectStaticAssetEncoding` chooses `br` before `gzip` and lets `*` re-enable gzip, so headers like `br;q=0.1, gzip;q=1` and `gzip;q=0, *;q=1` are handled incorrectly. I did not run the gateway or tests in this read-only review because they would create artifacts.

## Solution Assessment

No. The gateway Control UI static-file path is the right surface, but this implementation needs q-value-aware negotiation and real gateway proof before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Honor explicit q=0 before wildcard fallback:** `src/gateway/control-ui.ts:768-769`
  - body: `Accept-Encoding: gzip;q=0, *;q=1` explicitly forbids gzip while allowing other unspecified codings, but this fallback still selects gzip because `*` is present. Track explicit q-values so wildcard matching cannot re-enable a coding the client marked unacceptable.
  - confidence: 0.93
- **[P2] Choose encoding by highest non-zero q value:** `src/gateway/control-ui.ts:765-767`
  - body: This always returns Brotli whenever `br` appears, even when gzip has a higher q-value such as `br;q=0.1, gzip;q=1`. Parse and compare q-values so the selected compression matches the client's stated preference.
  - confidence: 0.9
