## Summary

Current main and the shipped Telegram channel implementation already switch to webhook-only monitoring when `channels.telegram.webhookUrl` is configured; the polling runner and its `deleteWebhook`/`getUpdates` cleanup path are only reached when webhook mode is not selected.

## What This Changes

The issue requests a Telegram webhook-only transport path so a configured webhook is not wiped by the long-polling `getUpdates` lifecycle.

## Best Possible Solution

Keep the existing automatic webhook transport selection; any new report should include evidence that the installed package’s compiled Telegram code differs from the tagged source or that another process is clearing the webhook.

## Reproduction Assessment

No for current main: the reported steps cannot reach the polling session when `webhookUrl` is truthy because channel startup selects `useWebhook` and the monitor returns from the webhook branch. I did not run a live Telegram bot, so the conclusion is source and shipped-test based.

## Solution Assessment

Yes: automatic webhook mode is the best existing solution because it avoids adding another config flag and prevents dual polling/webhook transports for one bot token.
