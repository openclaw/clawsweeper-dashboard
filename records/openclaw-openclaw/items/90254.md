## Summary

## What This Changes

The issue asks for Telegram webhook mode to stop clearing its registered webhook during gateway operation so inbound messages continue without an external guard.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94506 or an equivalent focused lifecycle fix that preserves webhook registration across transient restarts while keeping polling-mode cleanup intact.

## Reproduction Assessment

Yes, partially from source and reporter evidence: current main still calls `deleteWebhook` when the webhook abort signal fires, and the reporter provided repeated `getWebhookInfo` clearing observations. I did not run a live Telegram current-main reproduction, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Yes for the linked lifecycle direction, not for the original new-config-flag proposal. Current main already selects webhook mode from `webhookUrl`; the narrow maintainable fix is to stop transient webhook shutdown from deregistering Telegram while leaving polling-mode cleanup separate.
