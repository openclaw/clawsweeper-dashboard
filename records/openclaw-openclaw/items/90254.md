## Summary

Keep open: current main and the latest stable release already route `webhookUrl` accounts away from the polling session, so the original disable-polling fix is not the right shape, but the later reported webhook-lifecycle problem remains plausible because webhook shutdown still calls `deleteWebhook` on any abort.

## What This Changes

The issue reports Telegram webhook mode losing its registered webhook on a roughly 32-second cadence, causing inbound Telegram messages to be missed.

## Best Possible Solution

Keep this issue open and fix the Telegram plugin lifecycle so webhook mode stays webhook-only, transient restart/reload paths do not leave Telegram with an empty webhook, and failed re-registration is logged and retried with live Telegram proof.

## Reproduction Assessment

Yes, partially from source and reporter evidence: the issue includes repeated `getWebhookInfo` observations, and current source still calls `deleteWebhook` on webhook abort. I did not run a live Telegram current-main reproduction, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

No: adding a new disable-polling flag is not the best fix because current code already selects webhook mode when `webhookUrl` is configured. The better fix is to repair webhook lifecycle and re-registration semantics around abort/restart paths.
