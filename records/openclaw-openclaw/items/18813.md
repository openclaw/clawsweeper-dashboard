## Summary

Keep open. Current `main` has general security CI, but it does not enforce the TLS-specific checks requested on PR/push paths, and the Android gateway code still contains the exact patterns the issue called out.

## Risks / Open Questions

- Some of the Android TLS code appears intentional for fingerprint probing/TOFU flows, so a future fix likely needs a narrowly-scoped rule rather than a blanket ban.
- Scheduled CodeQL may still catch some unsafe TLS patterns, but that is weaker than the requested PR/push CI enforcement and does not address the explicit `SSLContext.getInstance("TLS")` usage.
