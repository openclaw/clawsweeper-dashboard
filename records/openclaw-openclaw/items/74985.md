## Summary

Keep open. Current main still selects isolated cron subagent primaries from `subagents.model`, but fallback resolution and embedded-run `fallbackConfigured` still read only payload fallbacks or the regular `agents.*.model` fallback policy, so `subagents.model.fallbacks` remain ignored.

## What This Changes

The issue reports that isolated cron embedded-agent runs time out on `kimi/kimi-code` without using fallbacks configured under `subagents.model`.

## Best Possible Solution

Add generic subagent-aware fallback-policy plumbing so the same `subagents.model` object that supplies an isolated cron primary also supplies its fallback chain, while preserving payload fallback precedence and explicit empty fallback arrays.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or `agents.list[].subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current source selects the subagent primary while fallback resolution and `fallbackConfigured` ignore those subagent fallbacks.

## Solution Assessment

Yes. The narrow maintainable fix is to align cron/subagent fallback-source resolution with the selected subagent model config, not to add Kimi-specific timeout handling or provider-specific failover behavior.
