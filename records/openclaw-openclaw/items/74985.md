## Summary

Keep open. Current main still has the reported subagent fallback mismatch: isolated cron model selection can pick a `subagents.model` primary, but the fallback override and embedded-runner `fallbackConfigured` paths only read payload fallbacks or normal `agents.*.model` fallback policy. Related PR #72984 is adjacent subagent primary-resolution work, not a complete fix for this cron fallback-policy path.

## What This Changes

The issue asks for isolated embedded-agent cron runs that use `subagents.model` to fail over through the configured `fallbacks` chain when the primary Kimi model times out.

## Best Possible Solution

Add shared subagent-aware fallback-policy plumbing so the same `subagents.model` object that supplies an isolated cron subagent primary also supplies its fallback chain, while preserving payload fallback precedence and explicit empty arrays.

## Reproduction Assessment

Yes, source-reproducible. Configure `agents.defaults.subagents.model` or `agents.list[].subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current source selects that subagent primary while fallback resolution and `fallbackConfigured` ignore the subagent fallback list.

## Solution Assessment

Yes. The narrow maintainable fix is generic subagent-aware fallback-policy resolution, not Kimi-specific timeout handling or provider-specific failover behavior.
