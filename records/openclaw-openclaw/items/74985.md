## Summary

Keep open: current main still has the reported subagent fallback mismatch, and the linked PR #72984 is adjacent primary-model resolution work rather than a merged fix for the cron fallback-policy path.

## What This Changes

The issue requests that isolated embedded-agent cron runs use the fallback chain configured on the selected `subagents.model` primary when that primary times out.

## Best Possible Solution

Add shared subagent-aware fallback-policy plumbing so the same `subagents.model` object that supplies the isolated cron subagent primary also supplies its fallback chain, while preserving payload fallback precedence and explicit empty arrays.

## Reproduction Assessment

Yes. ## Solution Assessment

## Repair Work Prompt

Fix the isolated cron embedded-agent fallback-policy bug from https://github.com/openclaw/openclaw/issues/74985. When an isolated cron `agentTurn` selects its primary from `agents.defaults.subagents.model` or `agents.list[].subagents.model` and that value is an object with `fallbacks`, the same fallback list must be used by `runWithModelFallback` and reflected by the embedded-runner `fallbackConfigured` decision. Preserve existing precedence: payload `fallbacks` wins, including explicit `[]`; payload `model` remains a cron primary and may use configured fallbacks unless payload fallbacks are provided; normal `agents.defaults.model` and `agents.list[].model` behavior must not regress. Do not add Kimi-specific logic, new config keys, startup migrations, or provider-specific timeout policy. Add focused regression coverage for default and per-agent `subagents.model` object fallbacks, explicit empty fallback disabling, and existing payload fallback precedence. Check related PR #72984 for adjacent model-primary work but do not depend on it unless it has landed.
