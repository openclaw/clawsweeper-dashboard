## Summary

Keep open. Current main still matches the prior ClawSweeper review: isolated cron can select its primary from `subagents.model`, but the cron fallback path and embedded runner fallback flag still ignore `subagents.model.fallbacks`, so the reported timeout can fail without trying configured fallbacks.

## What This Changes

The issue asks isolated embedded-agent cron runs to fail over from a timed-out Kimi primary to the fallback models configured for subagents.

## Best Possible Solution

Add generic subagent-aware fallback-policy plumbing so the same `subagents.model` object that supplies the isolated cron primary also supplies its fallback chain, while preserving payload fallback precedence and explicit empty fallback arrays.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or `agents.list[].subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current source selects the subagent primary while fallback resolution and `fallbackConfigured` ignore those subagent fallbacks.

## Solution Assessment

Yes. The narrow maintainable fix is to align cron/subagent fallback-source resolution with the selected subagent model config, not to add Kimi-specific timeout handling or provider-specific failover behavior.
