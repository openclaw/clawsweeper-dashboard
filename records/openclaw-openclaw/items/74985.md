## Summary

Keep open. Current main still has the reported mismatch: isolated cron can select its primary from `subagents.model`, but cron fallback resolution and embedded `fallbackConfigured` only inspect payload fallbacks or normal `agents.*.model` fallback policy. The related open PR #72984 is adjacent primary-resolution work, not a complete fallback-policy fix.

## What This Changes

Make isolated cron embedded-agent runs fail over to configured subagent model fallbacks when the primary embedded model times out.

## Best Possible Solution

Add generic subagent-aware fallback-policy plumbing so the same `subagents.model` object that supplies an isolated cron primary also supplies its fallback chain, while preserving payload fallback precedence, explicit empty fallback arrays, and strict behavior for unrelated user-selected model overrides.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or `agents.list[].subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current source selects the subagent primary while fallback resolution and `fallbackConfigured` ignore those subagent fallbacks.

## Solution Assessment

Yes. The narrow maintainable fix is to align cron/subagent fallback-source resolution with the selected subagent model config, not to add Kimi-specific timeout handling or provider-specific failover behavior.
