## Summary

Keep open. Current main still has a narrow source-level mismatch: isolated cron can choose its primary model from `subagents.model`, but the cron fallback override and embedded-runner `fallbackConfigured` observation still resolve only payload fallbacks or normal agent/default `model` fallbacks.

## What This Changes

The issue requests that isolated cron embedded-agent runs fail over to fallbacks configured under `subagents.model` when the primary Kimi model times out.

## Best Possible Solution

Add generic sub-agent-aware fallback-policy plumbing so the same `subagents.model` object that supplies an isolated cron primary also supplies its fallback chain, while preserving explicit payload fallback overrides and explicit empty fallback arrays.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or a per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current source selects the sub-agent primary while fallback resolution and `fallbackConfigured` do not read those sub-agent fallbacks.

## Solution Assessment

Yes. The maintainable fix is to make fallback resolution follow the same sub-agent config source as model selection, rather than adding Kimi-specific timeout behavior or changing provider failover policy.
