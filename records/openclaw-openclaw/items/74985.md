## Summary

Keep open. Current `origin/main` still has the reported mismatch: isolated cron model selection can take its primary from `agents.defaults.subagents.model` or per-agent `subagents.model`, but cron fallback resolution and the embedded runner's `fallbackConfigured` check still consult only payload fallbacks or normal `agents.*.model` fallback policy. The related open PR #72984 is adjacent subagent primary-resolution work, but it does not supersede this fallback-policy bug.

## What This Changes

The issue reports that isolated embedded-agent cron runs can select `kimi/kimi-code` from `subagents.model` but fail to use the fallback chain configured in that same subagent model stack after a timeout.

## Best Possible Solution

Add generic subagent-aware fallback-policy plumbing so the same `subagents.model` object that supplies an isolated cron primary also supplies its fallback chain, while preserving payload fallback precedence, explicit empty fallback arrays, and strict behavior for unrelated user-selected model overrides.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or `agents.list[].subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current source selects the subagent primary while fallback resolution and `fallbackConfigured` do not read those subagent fallbacks.

## Solution Assessment

Yes. The best fix is a narrow generic fallback-source alignment tied to cron/subagent model selection, not Kimi-specific timeout handling or a provider-specific failover special case.
