## Summary

Keep open. Current main still has the reported mismatch: isolated cron subagent model selection can use a `subagents.model` primary, but fallback resolution and embedded-runner `fallbackConfigured` still read payload fallbacks or normal `agents.*.model` fallback policy, not the subagent fallback list.

## What This Changes

The issue requests that isolated embedded subagent runs fail over to configured `subagents.model.fallbacks` when the primary model times out.

## Best Possible Solution

The end state should make subagent primary selection and fallback policy share the same resolved model-policy source, while preserving payload fallback precedence and explicit empty-array fallback disablement.

## Reproduction Assessment

Yes, source-reproducible on current main. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and the source selects the subagent primary while fallback resolution and `fallbackConfigured` ignore the subagent fallback list.

## Solution Assessment

Yes. The narrow maintainable fix is generic subagent-aware fallback-policy resolution in the shared cron/agent fallback helpers, not Kimi-specific timeout handling or a new config surface.
