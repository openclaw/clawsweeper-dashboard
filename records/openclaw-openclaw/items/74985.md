## Summary

Keep open. Current main still selects isolated cron embedded-agent primaries from `subagents.model`, but the cron fallback resolver and embedded-runner `fallbackConfigured` path only consult payload fallbacks or normal `agents.*.model` fallback policy, so the reported timeout-without-fallback behavior remains valid.

## What This Changes

Isolated cron embedded-agent runs should use fallback chains configured on `agents.defaults.subagents.model` or per-agent `subagents.model` when the selected primary times out.

## Best Possible Solution

Add generic subagent-aware fallback-policy plumbing so the same `subagents.model` object that supplies the isolated cron primary also supplies its fallback chain, while preserving payload fallback precedence, explicit empty fallback arrays, and strict behavior for unrelated user-selected model overrides.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or `agents.list[].subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current source selects the subagent primary while fallback resolution and `fallbackConfigured` ignore those subagent fallbacks.

## Solution Assessment

Yes. The narrow maintainable fix is to align cron/subagent fallback-source resolution with the selected subagent model config, not to add Kimi-specific timeout handling or provider-specific failover behavior.
