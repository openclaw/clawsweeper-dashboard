## Summary

Keep open. Current main still has the reported subagent fallback mismatch: isolated cron can run on a subagent-selected model, but the fallback override and embedded-runner `fallbackConfigured` checks only read payload fallbacks or normal `agents.*.model` fallback policy, so `subagents.model.fallbacks` are not applied. Related PR #72984 is adjacent subagent primary-resolution work, not a complete fix for this fallback-policy path.

## What This Changes

The issue asks for isolated cron embedded sub-agent runs to fail over from `kimi/kimi-code` to the configured `subagents.model` fallback chain when the primary times out.

## Best Possible Solution

Add shared subagent-aware fallback resolution so the same `subagents.model` object that supplies an isolated cron subagent primary also supplies its fallback chain, while preserving payload fallback precedence, explicit empty arrays, and strict behavior for unrelated user-selected model overrides.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or `agents.list[].subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current source selects a subagent primary while fallback resolution and `fallbackConfigured` ignore those subagent fallbacks.

## Solution Assessment

Yes. The narrow maintainable solution is generic subagent-aware fallback-policy plumbing, not Kimi-specific timeout handling or provider-specific failover behavior.
