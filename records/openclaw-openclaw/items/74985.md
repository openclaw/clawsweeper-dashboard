## Summary

Keep open. Current main still has the reported mismatch: isolated cron model selection can use `subagents.model` as the embedded-agent primary, but cron fallback resolution and the embedded runner's `fallbackConfigured` observation still look only at payload fallbacks or normal `agents.*.model` fallback policy. The related open PR #72984 is adjacent subagent primary-resolution work, but it does not supersede this fallback-policy bug.

## What This Changes

This issue requests that isolated embedded-agent cron runs fail over through fallbacks configured on the same `subagents.model` object that supplies the primary model.

## Best Possible Solution

Add subagent-aware fallback-policy plumbing so the same `subagents.model` object that supplies an isolated cron primary also supplies its fallback chain, while preserving explicit payload fallback overrides, explicit empty fallback arrays, and the existing strict behavior for unrelated user-selected models.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or `agents.list[].subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current source selects the subagent primary while fallback resolution and `fallbackConfigured` do not read those subagent fallbacks.

## Solution Assessment

Yes. The narrow maintainable solution is generic subagent-aware fallback resolution tied to the same model-selection source, not Kimi-specific timeout handling or a provider-specific failover special case.
