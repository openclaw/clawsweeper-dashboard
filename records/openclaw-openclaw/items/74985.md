## Summary

Keep open. Current main still shows the reported mismatch: isolated cron model selection can read `subagents.model` for the primary, but the cron fallback override and embedded failover observation still resolve only payload fallbacks or normal `agents.*.model` fallback policy, so `subagents.model.fallbacks` can be skipped.

## What This Changes

The issue asks OpenClaw to use configured `subagents.model.fallbacks` for isolated embedded-agent cron runs when the selected sub-agent primary model times out.

## Best Possible Solution

Add generic sub-agent-aware fallback-policy plumbing so the same `subagents.model` object that supplies an isolated cron primary also supplies its fallback chain, while preserving payload fallback overrides and explicit empty fallback arrays.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current source selects the sub-agent primary while fallback resolution and `fallbackConfigured` do not read those sub-agent fallbacks.

## Solution Assessment

Yes. The narrow maintainable fix is to make fallback resolution follow the same sub-agent config source as model selection, instead of adding Kimi-specific timeout handling or changing the generic provider failover loop.
