## Summary

Keep open. Current main still has the reported mismatch: isolated cron can select an embedded-agent primary from sub-agent model config, but the cron fallback override and embedded failover observation still resolve fallbacks from payload-level overrides or normal `agents.*.model` policy, so `subagents.model.fallbacks` can be skipped.

## What This Changes

Embedded isolated cron agent runs should fail over through configured sub-agent model fallbacks when the primary Kimi model times out.

## Best Possible Solution

Add generic sub-agent-aware fallback-policy plumbing so the same `subagents.model` object that supplies an isolated cron primary also supplies its fallback chain, while preserving explicit payload fallback overrides and explicit empty fallback arrays.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current source can select that sub-agent primary while fallback resolution and `fallbackConfigured` do not read those sub-agent fallbacks.

## Solution Assessment

Yes. The narrow maintainable solution is to make fallback resolution follow the same sub-agent config source as model selection, rather than adding Kimi-specific timeout handling or changing provider failover policy.
