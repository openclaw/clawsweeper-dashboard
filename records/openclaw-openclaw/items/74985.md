## Summary

Keep open. Current main still has the reported mismatch: isolated cron model selection can use a `subagents.model` primary, but cron fallback resolution and the embedded runner's `fallbackConfigured` flag only inspect payload fallbacks or normal `agents.*.model` fallback policy. Related #72984 is adjacent primary-model resolution work, and #44353 is a broader fallback report, but neither proves this path is fixed on main.

## What This Changes

The issue asks for embedded isolated cron sub-sessions to fail over from `kimi/kimi-code` to the fallback models configured in the subagent `model_stack` after a timeout.

## Best Possible Solution

Carry the selected subagent model policy into cron execution and embedded failover detection so the primary and fallback chain come from the same `subagents.model` source while preserving explicit payload fallback precedence.

## Reproduction Assessment

Yes, source-reproducible. Configure `agents.defaults.subagents.model` or `agents.list[].subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current source selects the subagent primary while fallback resolution and `fallbackConfigured` ignore the subagent fallback list.

## Solution Assessment

Yes. The narrow maintainable fix is generic subagent-aware fallback-policy plumbing, not Kimi-specific timeout handling or provider-specific failover changes.
