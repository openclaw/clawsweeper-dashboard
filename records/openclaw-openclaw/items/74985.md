## Summary

Keep open. Current main still resolves an isolated cron subagent primary from subagent model config while the cron fallback resolver and embedded runner fallback observation only consult payload fallbacks or normal agent/default model fallback policy. The related open PR #72984 is adjacent subagent primary-resolution work, but it does not supersede this fallback-policy bug.

## What This Changes

The issue asks for isolated embedded-agent cron runs to fail over from Kimi to fallbacks configured in the subagent model stack when the primary times out.

## Best Possible Solution

Add generic subagent-aware fallback-policy plumbing so the same subagents.model object that supplies the isolated cron primary also supplies its fallback chain, while preserving payload fallback precedence, explicit empty fallback arrays, and strict behavior for unrelated user-selected model overrides.

## Reproduction Assessment

Yes. Configure agents.defaults.subagents.model or an agent subagents.model as { primary: "kimi/kimi-code", fallbacks: [...] }, run an isolated cron agentTurn without payload fallbacks, and current source selects the subagent primary while fallback resolution and fallbackConfigured do not read those subagent fallbacks.

## Solution Assessment

Yes. The narrow maintainable fix is to align cron/subagent model fallback resolution with the already-selected subagent model config, not add Kimi-specific timeout handling or provider-specific failover behavior.
