## Summary

Keep open. Current main still selects the isolated cron embedded-agent primary from sub-agent model config, but the fallback resolver and embedded failover observation consult only payload fallbacks, normal agent model fallbacks, or normal default model fallbacks, so `subagents.model.fallbacks` can still be skipped exactly as reported.

## What This Changes

The issue requests that isolated embedded-agent cron/sub-agent runs fail over to the fallback chain configured under `subagents.model` when the selected primary model times out.

## Best Possible Solution

Add generic sub-agent-aware fallback-policy plumbing so the same `subagents.model` object that supplies an isolated cron/sub-agent primary also supplies its fallback chain, while preserving explicit payload fallback overrides and explicit empty fallback arrays.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current main selects the sub-agent primary while fallback resolution and `fallbackConfigured` do not read those sub-agent fallbacks.

## Solution Assessment

Yes. The narrow maintainable solution is to make fallback resolution follow the same sub-agent config source as primary model selection, rather than adding Kimi-specific timeout handling or changing provider-level retry policy.
