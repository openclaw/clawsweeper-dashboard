## Summary

Keep open. Current main still has the reported source-level mismatch: isolated cron model selection can choose a `subagents.model` primary, while both cron fallback override resolution and embedded-runner `fallbackConfigured` still consult only payload fallbacks or normal `agents.*.model` fallback policy.

## What This Changes

The issue requests that embedded isolated cron agent runs fail over from a timed-out subagent primary model to the configured `subagents.model.fallbacks` chain.

## Best Possible Solution

Centralize subagent-aware fallback-policy resolution so the selected `subagents.model` object supplies both the primary and fallback chain, while preserving payload fallback precedence and explicit empty arrays.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `subagents.model` as `{primary,fallbacks}`, run an isolated cron `agentTurn` without payload fallbacks, and current main selects the subagent primary while fallback policy and the embedded `fallbackConfigured` flag ignore that fallback list.

## Solution Assessment

Yes. The narrow maintainable fix is shared subagent-aware fallback-policy plumbing, not Kimi-specific timeout handling and not a new configuration surface.
