## Summary

Keep open. Current main still selects a subagent-configured primary for isolated cron, but the fallback override and embedded-runner fallback eligibility paths only read payload fallbacks or normal `agents.*.model` fallback policy, so `subagents.model.fallbacks` can still be ignored.

## What This Changes

The issue reports that isolated embedded-agent cron sub-sessions use a configured subagent primary model but do not apply `subagents.model.fallbacks` when the primary times out.

## Best Possible Solution

Add shared subagent-aware fallback resolution so the same `subagents.model` object that supplies an isolated cron subagent primary also supplies its fallback chain, while preserving payload fallback precedence and normal agent-model fallback behavior.

## Reproduction Assessment

Yes by source inspection: configure `agents.defaults.subagents.model` or `agents.list[].subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current main selects the subagent primary while fallback resolution ignores those subagent fallbacks. I did not execute the run because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is generic subagent-aware fallback-policy plumbing, not Kimi-specific timeout handling or provider-specific failover behavior.
