## Summary

Keep open. Current main still intentionally treats Feishu broadcast-only `@all`/`@_all` as not mentioning the bot, and that behavior is backed by source, regression tests, docs, and changelog; the remaining question is a Feishu channel policy/config decision rather than an obsolete report.

## What This Changes

Request to stop Feishu from dropping broadcast-only `@all`/`@_all` at the mention-gate layer and instead handle suppression through prompt instructions or an explicit config option.

## Best Possible Solution

Keep the issue open for a Feishu channel policy decision. The maintainable end state should separate broadcast mention context from the boolean bot-mention gate, preserve the #37706 false-positive protection, and then document/test whether the default is ignore, prompt-visible context, or an explicit config option.

## Reproduction Assessment

Yes. Current source and regression tests provide a high-confidence static reproduction path: broadcast-only `@_all`, `@all` metadata, and post `user_id: all` produce `mentionedBot=false`, and mention-gated groups return before dispatch.

## Solution Assessment

No for the requested direct revert as written. A safer solution would preserve broadcast-only false-positive protection while adding an explicit context or config path if maintainers want agents to reason about `@all`.
