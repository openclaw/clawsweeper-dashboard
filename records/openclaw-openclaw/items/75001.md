## Summary

Keep open: current main and v2026.6.10 still intentionally treat Feishu broadcast-only @all/@_all as not mentioning the bot, so the requested prompt/config-aware behavior is not implemented. The remaining decision is product policy for Feishu safe defaults and any opt-in awareness path, not conservative cleanup.

## What This Changes

The issue requests changing Feishu @all handling so broadcast mentions reach agent context or are controlled by prompt/config instead of being suppressed at the Feishu mention gate.

## Best Possible Solution

Define the supported Feishu broadcast-mention contract, preserve the safe default unless maintainers choose otherwise, and add a documented opt-in path with focused tests and live Feishu proof if @all agent awareness becomes supported.

## Reproduction Assessment

Yes, at source level: broadcast-only Feishu @all sets mentionedBot=false, and a requireMention group returns before dispatch. I did not run a live Feishu workspace, but current tests, docs, and v2026.6.10 source assert the same behavior.

## Solution Assessment

Unclear. Reverting the filter is not clearly the best fix because it conflicts with the shipped safe-default behavior; maintainers should first choose whether the supported path is prompt context, an explicit opt-in config, or keeping the current contract.
