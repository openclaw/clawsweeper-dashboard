## Summary

Keep open. Current main still intentionally treats Feishu broadcast-only `@all`/`@_all` as not mentioning the bot, and the source, tests, docs, and changelog all encode that policy; the remaining question is whether Feishu should preserve broadcast mention context for prompt/config handling without reintroducing the old false-positive reply behavior.

## What This Changes

The issue requests changing Feishu `@all` handling so broadcast mentions are preserved for agent or prompt-level handling instead of being filtered by channel-level mention gating.

## Best Possible Solution

Keep this open for Feishu channel policy. The maintainable end state should separate broadcast mention context from the boolean bot-mention gate, preserve the #37706 false-positive protection, and then document and test the chosen behavior as default ignore, prompt-visible context, or an explicit config option.

## Reproduction Assessment

Yes. Current source and regression tests provide a high-confidence static reproduction path: broadcast-only `@_all`, `@all` metadata, and post `user_id: all` produce `mentionedBot=false`, and mention-gated groups return before dispatch. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No for the requested direct revert as written. A safer solution would preserve broadcast-only false-positive protection while adding an explicit context or config path if maintainers want agents to reason about `@all`.
