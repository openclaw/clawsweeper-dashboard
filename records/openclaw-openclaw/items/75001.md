## Summary

Keep open: current main and the latest release still treat Feishu broadcast-only `@all` / `@_all` as not mentioning the bot, so the reported product question remains unresolved rather than implemented or stale.

## What This Changes

The issue requests changing Feishu broadcast `@all` handling so agents can receive or opt into those events instead of having them filtered by the channel mention gate.

## Best Possible Solution

Decide the Feishu `@all` policy explicitly, preserving the safe default unless maintainers approve an opt-in path that gives agents context without causing unwanted all-bot replies.

## Reproduction Assessment

Yes from source, not from a live Feishu run: with an allowlisted group and `requireMention: true`, broadcast-only `@all` makes `ctx.mentionedBot=false`, the group ingress does not dispatch, and tests assert that behavior.

## Solution Assessment

Unclear: reverting the API-level filter would conflict with current shipped docs/tests and may revive unwanted broadcast activation, so the best fix needs an explicit maintainer decision on prompt context versus an opt-in config surface.
