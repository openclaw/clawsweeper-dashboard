## Summary

Keep open: current main and the latest release still intentionally treat Feishu broadcast-only `@all` / `@_all` as not addressing the bot, while this issue asks maintainers to choose a different Feishu policy or opt-in surface.

## What This Changes

The issue asks OpenClaw to stop hard-gating Feishu broadcast-only `@all` / `@_all` before the agent can see it, or to add an operator-controlled suppression option.

## Best Possible Solution

Decide the Feishu broadcast-mention contract explicitly, then implement one documented and tested path that preserves safe defaults while allowing agent awareness only through a maintainer-approved prompt, context, or opt-in policy surface.

## Reproduction Assessment

Yes, source-reproducible: current main filters broadcast-only Feishu `@all` from bot mention detection and the `requireMention` group gate returns before dispatch. I did not run a live Feishu tenant replay in this read-only review.

## Solution Assessment

Unclear. Reverting the filter would satisfy the reporter's use case but may undo the safe-default behavior from the merged Feishu `@all` fix, so maintainers should choose prompt context, opt-in config, or the current channel gate deliberately.
