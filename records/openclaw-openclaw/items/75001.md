## Summary

Keep open. Current main and the latest shipped release intentionally preserve the Feishu broadcast-mention behavior the reporter disputes, with source, tests, docs, and changelog all aligned, so this is a live channel policy question rather than an obsolete cleanup item. A direct revert is not safe without maintainer direction because the current behavior was introduced to fix the opposite noisy-reply case where Feishu/Lark `@all` alone counted as a bot mention.

## What This Changes

The issue requests that Feishu broadcast `@all`/`@_all` mentions be preserved for agent context and handled by prompt or config policy instead of being filtered by the Feishu channel mention gate.

## Best Possible Solution

Keep this open until Feishu channel maintainers choose the intended product behavior. If the policy changes, model broadcast mentions as a distinct Feishu channel fact that can be passed into agent context, while separately testing whether broadcast-only messages satisfy mention gating, remain context-only, or require explicit config/prompt opt-in.

## Reproduction Assessment

Yes. Current fixtures for broadcast-only `@_all` text, `@all` mention metadata, and post-content `user_id: all` all produce `mentionedBot=false`; with group `requireMention=true`, the handler returns before agent dispatch.

## Solution Assessment

No. The requested direct revert is not clearly the safest fix because it can reintroduce #37706; a narrower maintainable solution would separate broadcast mention context from the boolean bot-mention gate and add explicit policy/config coverage if maintainers want broadcast-only messages to reach agents.
