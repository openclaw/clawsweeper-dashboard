## Summary

Keep open. Current main still intentionally treats Feishu broadcast-only `@all`/`@_all` as not mentioning the bot and returns before agent dispatch when `requireMention` is true; the report is fresh, reproducible from source/tests, and represents a real Feishu channel policy question rather than an obsolete or duplicate cleanup item.

## What This Changes

Request to stop Feishu broadcast `@all`/`@_all` mentions from being silently filtered by mention gating and instead expose them to the agent prompt or a config-controlled policy.

## Best Possible Solution

Keep the issue open for a Feishu maintainer/product decision. The maintainable end state should separate broadcast mention visibility from the boolean bot-mention gate, preserve the #37706 false-positive protection, and then document/test whether the default is ignore, prompt-visible context, or an explicit config option.

## Reproduction Assessment

Yes. Current source and regression tests provide a high-confidence static reproduction path: broadcast-only `@_all`, `@all` metadata, and post `user_id: all` produce `mentionedBot=false`, and mention-gated groups return before dispatch.

## Solution Assessment

No for the requested direct revert as written. A safer solution would preserve broadcast-only false-positive protection while adding an explicit context or config path if maintainers want agents to reason about `@all`.
