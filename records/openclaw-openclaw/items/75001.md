## Summary

Keep open. Current main still intentionally treats Feishu broadcast-only `@all`/`@_all` as not mentioning the bot, and the source, tests, docs, and changelog all encode that policy; the reporter is challenging a live product/channel decision rather than reporting obsolete behavior.

## What This Changes

The issue asks OpenClaw to stop dropping Feishu broadcast `@all`/`@_all` mentions at the channel gate and instead preserve them for prompt or config-level handling.

## Best Possible Solution

Keep this open for Feishu channel maintainer policy. The maintainable end state would separate broadcast mention context from the boolean bot-mention gate, then document and test whether broadcast-only messages are ignored by default, passed to the agent as context, or controlled by an explicit config/prompt opt-in.

## Reproduction Assessment

Yes. Current fixture-style tests show broadcast-only `@_all`, `@all` metadata, and post `user_id: all` produce `mentionedBot=false`; the group handler then returns before dispatch when `requireMention=true`. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No for the requested direct revert as written. A safer solution would preserve the #37706 false-positive protection while adding an explicit broadcast-mention context/config path if maintainers want agents to reason about `@all`.
