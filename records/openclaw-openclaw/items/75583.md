## Summary

Keep open. Current main still keeps Slack thread participation in process-local memory only, so the restart-survival problem this PR targets is real; the proposed Slack-only SDK state-store implementation looks coherent and I found no discrete blocking correctness or security finding, but the PR is draft and has the protected `maintainer` label, so it should stay in maintainer review rather than be auto-closed or auto-repaired.

## What This Changes

This PR adds best-effort SDK-backed persistence for Slack bot-participated thread markers, wires inbound thread mention checks to consult that persistence on memory misses, records the route agent id on delivered replies, and updates Slack docs plus the changelog.

## Best Possible Solution

Land a single canonical Slack persistence implementation that keeps the in-memory cache as the hot path, uses the bundled-plugin SDK state store for restart durability, documents the restart-safe behavior, and closes or supersedes the older overlapping Slack persistence PRs once merged.

## Reproduction Assessment

Yes. On current main, the reproduction path is to record Slack thread participation, restart the Gateway so the process-local dedupe cache is lost, then observe that inbound thread replies no longer satisfy `bot_thread_participant`; the current code has no persistence path for that marker.

## Solution Assessment

Yes, this appears to be the narrowest maintainable direction: it stays inside the Slack plugin, uses the existing SDK state-store seam, keeps SQLite failures best-effort, and avoids a core startup migration. The remaining decision is maintainer review and canonicalization with the overlapping drafts, not a different code shape I can prove is safer.
