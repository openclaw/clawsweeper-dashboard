## Summary

Keep open: current main still forwards an unbounded `sessions_history.limit` into the `chat.history` API that validates `limit <= 1000`, so the central bug remains. This PR is the narrower fix path compared with the closed gateway-limit alternatives, but the branch is now conflicting with current main and still needs real behavior proof.

## What This Changes

The PR constrains `sessions_history.limit` to the gateway `chat.history` cap, clamps outgoing requests to 1000, and adds a schema assertion.

## Best Possible Solution

Port the caller-side max and clamp onto current main, add execute-level regression coverage for `limit: 2000`, and keep the gateway `chat.history` cap at 1000 unless maintainers make a separate protocol decision.

## Reproduction Assessment

Yes, at source level. Current main accepts a positive integer `sessions_history.limit` and forwards it unchanged to `chat.history`, while the gateway schema rejects values over 1000; the linked report includes matching `limit: 2000` logs.

## Solution Assessment

Yes, the caller-side schema bound plus runtime clamp is the best fix direction; raising the gateway cap or catch-all retry paths were weaker alternatives. The submitted branch needs a current-main rebase and stronger execute-level proof before merge.
