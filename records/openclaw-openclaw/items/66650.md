## Summary

Keep open. Current main still lets `sessions_history` forward `limit: 2000` into `chat.history`, which the Gateway schema rejects before the handler clamp can run; the PR’s caller-side clamp is the right direction but still needs release-note coverage and a direct execute-level regression test before merge.

## What This Changes

The PR adds a 1000 maximum to `sessions_history.limit`, clamps outgoing `chat.history` limits to 1000, and adds a schema assertion.

## Best Possible Solution

Land the caller-side `sessions_history` schema and runtime clamp while keeping the Gateway `chat.history` cap at 1000, with direct execute-level coverage and an Unreleased changelog entry.

## Reproduction Assessment

Yes. Current main shows `sessions_history` forwards oversized numeric limits, and `chat.history` rejects values above 1000 before its handler clamp; the linked issue logs match that path.

## Solution Assessment

Yes, with small completion items. Clamping at the tool boundary is narrower and safer than raising the Gateway protocol cap to 2000, but the branch should add direct runtime coverage and release-note coverage before landing.
