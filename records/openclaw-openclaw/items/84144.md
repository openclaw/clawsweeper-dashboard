## Summary

Keep open: current main and v2026.5.18 still have the reported authorization shape, and the issue is now paired with an open closing PR that needs maintainer/security-boundary review before the issue can be resolved.

## What This Changes

The issue reports that automation can approve node device pairing after v2026.5.18 but cannot approve exec-capable node command surfaces because `openclaw nodes approve` ultimately requires `operator.admin` scope.

## Best Possible Solution

Review the open linked PR or replace it with a narrower maintainer-approved path that keeps the intended node command-surface trust boundary clear, then close this issue only after the chosen fix lands.

## Reproduction Assessment

Yes. Source inspection shows the CLI reaches `node.pair.approve` through a pairing-scoped method while the approval handler requires derived admin scope for `system.run`-class pending nodes; the supplied Multipass script is a plausible live reproduction but was not run here.

## Solution Assessment

Unclear. The issue is valid, but lowering all node approvals to pairing-only would relax a documented security gate, so maintainers should decide whether to preserve that gate with an admin-capable automation path or intentionally reclassify node approval policy.
