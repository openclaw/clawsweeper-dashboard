## Summary

Keep open: current main and v2026.5.18 still have the reported authorization shape, and the linked open PR is an implementation candidate that needs maintainer/security-boundary review rather than an issue-close reason.

## What This Changes

The issue reports that `openclaw nodes approve` cannot approve exec-capable node surfaces from automation because the post-2026.5.18 approval path requires `operator.admin`.

## Best Possible Solution

Keep this issue open until maintainers choose and land an automation-safe node approval path that preserves or deliberately redefines the exec-capable node authorization boundary with docs, tests, and real workflow proof.

## Reproduction Assessment

Yes. Source inspection shows `openclaw nodes approve` requests only method-level pairing scope while current approval logic requires `operator.admin` for `system.run`-class pending nodes; the supplied Multipass scripts are a plausible live reproduction path but were not run here.

## Solution Assessment

Unclear. The issue is real, but the safest fix requires a maintainer policy decision between preserving command-sensitive approval scopes with an admin-capable automation path or explicitly redefining node approval as pairing-only.
