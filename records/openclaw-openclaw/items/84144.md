## Label Justifications

- `P1`: The report is a recent released regression that blocks automated provisioning of exec-capable nodes and affects an authorization-sensitive workflow.
- `impact:security`: The affected behavior is an authorization boundary around approving high-risk node command surfaces.
- `impact:auth-provider`: The failure is caused by operator scope selection and session-token authorization semantics.

## Summary

Keep open: current main still has the reported authorization shape, and the open linked PR needs maintainer/security judgment because its proposed fix relaxes a documented node-approval boundary.

## What This Changes

The issue reports that `openclaw nodes approve` requests only pairing scope, blocking automation from approving exec-capable node surfaces after node commands became hidden until node approval.

## Best Possible Solution

Preserve the command-sensitive approval gate by default, then repair `nodes approve` automation through a documented admin-capable approval path with regression coverage for commandless, non-exec, and exec-capable nodes.

## Reproduction Assessment

Yes. Source inspection shows the CLI asks for the method-level pairing scope while approval of `system.run`-class nodes requires admin-scoped session authorization; the supplied Multipass script is a plausible live repro but was not run here.

## Solution Assessment

Unclear. Lowering every node approval to pairing scope is not the best default because it weakens the existing boundary; the safer direction is to preserve command-sensitive checks and fix how trusted automation obtains or requests the needed scope.
