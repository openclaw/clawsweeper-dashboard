## Label Justifications

- `P1`: The report is source-reproducible as a recent regression that blocks automated provisioning of exec-capable nodes.
- `impact:security`: The affected behavior is an authorization boundary around approving high-risk node command surfaces.
- `impact:auth-provider`: The failure is caused by operator scope selection and token/session authorization semantics.

## Summary

Keep open: current `main` and the v2026.5.18 tag still have the reported authorization shape. The CLI sends `node.pair.approve` with only the method-level pairing scope, while the handler separately requires admin scope when the pending node advertises `system.run`-class commands.

## What This Changes

The issue reports that `openclaw nodes approve` cannot approve exec-capable node surfaces because the CLI session lacks `operator.admin` after the node command-surface gate shipped.

## Best Possible Solution

Keep the node command-surface approval fail-closed, but provide a documented automation-safe approval path that can use already-trusted admin credentials for exec-capable node requests, with regression coverage for pairing-only, write, and admin cases.

## Reproduction Assessment

Yes. Source inspection shows `openclaw nodes approve` requests only `operator.pairing`, and the approval handler requires `operator.admin` for pending `system.run`-class node commands; the supplied Multipass script is a plausible live reproduction path but was not run here.

## Solution Assessment

Unclear. Mutating paired-device scope records is not the right fix; maintainers should decide the intended admin credential path before changing CLI scope selection or adding an explicit approval option.
