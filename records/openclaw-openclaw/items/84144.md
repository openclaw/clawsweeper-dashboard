## Label Justifications

- `P1`: This is a recent source-reproducible regression blocking automated provisioning of exec-capable nodes after the shipped node surface gate.
- `impact:security`: The affected behavior is the authorization boundary for approving high-risk node command surfaces such as `system.run`.
- `impact:auth-provider`: The failure is caused by gateway operator scope selection and session authorization semantics.

## Summary

Keep open: current main and v2026.5.18 still have the reported authorization mismatch. The CLI approval command requests only the method-level pairing scope, while exec-capable node approval is still enforced as pairing plus admin at approval time.

## What This Changes

The issue reports that `openclaw nodes approve` blocks automated exec-capable node provisioning after node command surfaces became hidden until node-pair approval.

## Best Possible Solution

Keep exec-capable node approval fail-closed, but make the CLI approval path derive and request the pending node's required scopes through an explicit trusted admin credential path, with regression coverage for pairing-only, write, and admin approvals.

## Reproduction Assessment

Yes. Source inspection shows the CLI requests only `operator.pairing` for `node.pair.approve`, while current main and v2026.5.18 require `operator.admin` at approval time for pending `system.run`-class node requests.

## Solution Assessment

Unclear as proposed. Lowering the admin requirement is not the safest fix; the maintainable path is to preserve the fail-closed gate and give the CLI an explicit derived-scope approval path like the existing agent tool path.
