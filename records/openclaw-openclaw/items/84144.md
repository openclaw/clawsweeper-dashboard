## Label Justifications

- `P1`: The issue is source-reproducible as a recent regression that blocks automated provisioning of exec-capable nodes.
- `impact:security`: The affected behavior is an authorization boundary around approving high-risk node command surfaces.
- `impact:auth-provider`: The failure is caused by operator scope selection and session-token authorization semantics.

## Summary

Keep open: current main and the v2026.5.18 release still have the reported authorization shape, where the CLI requests only the pairing method scope while exec-capable node approval requires an admin-scoped session.

## What This Changes

The issue reports that `openclaw nodes approve` cannot automate approval of exec-capable node surfaces after the node surface gate because the CLI session lacks `operator.admin`.

## Best Possible Solution

Keep exec-capable node surface approval fail-closed, but make the documented headless CLI approval path request or obtain the derived admin-capable approval scopes safely, with coverage for commandless, non-exec, and exec-capable node requests.

## Reproduction Assessment

Yes. Source inspection shows `openclaw nodes approve` requests only `operator.pairing`, while the handler requires `operator.admin` for pending `system.run`-class node commands; the supplied Multipass script is plausible but was not run here.

## Solution Assessment

Unclear. The proposed pairing-only downgrade is not the safest fix because current docs and code intentionally require admin for exec-capable surfaces; maintainers should choose the admin credential/scope path for CLI automation.
