## Label Justifications

- `P1`: The report is source-reproducible as a recent released regression that blocks automated provisioning of command-bearing nodes.
- `impact:security`: The affected behavior is an authorization boundary for approving high-risk node command surfaces, including system.run.
- `impact:auth-provider`: The failure turns on operator scope selection and active token/session authorization semantics.

## Summary

Keep open: current main and v2026.5.18 still match the reported authorization deadlock, and the linked open fix PR is now the active place to review the policy tradeoff before this issue can close.

## What This Changes

The issue asks for an automation-safe way to approve exec-capable node surfaces after the 2026.5.18 gate without requiring an unobtainable operator.admin session.

## Best Possible Solution

Preserve command-sensitive node approval as the default unless maintainers explicitly reclassify the trust boundary, and land a reviewed automation path with docs, regression tests, and live proof through the linked PR or a replacement.

## Reproduction Assessment

Yes. Source inspection shows the CLI reaches node.pair.approve with pairing scope while the approval handler requires admin for system.run-class pending nodes; the supplied Multipass script is a plausible live reproduction path but was not run here.

## Solution Assessment

Unclear. Lowering all node approvals to operator.pairing is not the safest default because it weakens the documented authorization boundary; maintainers should decide whether to add an admin-capable automation path or intentionally reclassify pairing as the approval boundary.
