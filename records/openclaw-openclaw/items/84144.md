## Summary

Keep open: current main and v2026.5.18 still have the source-level node approval scope mismatch, and the linked closing PR is the active fix path but needs maintainer/security policy review before this issue can close.

## What This Changes

The issue reports that `openclaw nodes approve` cannot approve exec-capable node surfaces for automation because the CLI reaches the pairing method with a pairing-scoped session while approval requires `operator.admin`.

## Best Possible Solution

Keep this issue open and resolve it through https://github.com/openclaw/openclaw/pull/84251 or a replacement PR that preserves the security boundary by default while providing a documented automation-safe approval path with regression coverage.

## Reproduction Assessment

Yes. Current main source and tests show the CLI reaches `node.pair.approve` with the pairing method scope while exec-capable pending nodes require `operator.admin`; the issue also provides a plausible Multipass reproduction, but I did not run it here.

## Solution Assessment

Unclear. The report identifies a real deadlock, but simply lowering every node approval to `operator.pairing` is not the best fix unless maintainers intentionally reclassify that security boundary.
