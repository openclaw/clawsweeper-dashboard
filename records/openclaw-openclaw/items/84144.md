## Summary

Keep open: current main still has the reported authorization shape, and the issue is already paired with an open unmerged fix PR. The remaining decision is how to restore automation without unintentionally weakening the documented node command approval boundary.

## What This Changes

The issue requests an automation-safe way to approve exec-capable node command surfaces after the 2026.5.18 node-surface gate made `openclaw nodes approve` require `operator.admin`.

## Best Possible Solution

Keep the command-sensitive approval gate fail-closed by default, then land a maintainer-approved automation path or policy update that documents how trusted automation obtains the required approval authority.

## Reproduction Assessment

Yes. Source inspection shows `openclaw nodes approve` reaches `node.pair.approve` with pairing scope while the approval handler derives `operator.admin` for `system.run`-class pending nodes; the reporter's Multipass scripts are a plausible live path but were not run here.

## Solution Assessment

Unclear. The linked pairing-only PR is not the best proven solution without maintainer approval because it relaxes a documented security boundary; a safer solution is an explicit admin-capable automation path or a deliberate docs-and-tests policy reclassification.
