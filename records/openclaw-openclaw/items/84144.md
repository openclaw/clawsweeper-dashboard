## Summary

Keep open: current main and v2026.5.18 still have the reported authorization shape, and an open linked PR is already the implementation candidate but needs maintainer/security-boundary review before the issue can close.

## What This Changes

The issue reports that `openclaw nodes approve` requires `operator.admin` for exec-capable nodes after the node surface gate, blocking automation that only has pairing-scoped credentials.

## Best Possible Solution

Review or replace the linked implementation with a maintainer-approved fix that keeps node command-surface approval fail-closed while providing a documented automation-safe approval path, plus docs and regression coverage for pairing-only, write, and admin cases.

## Reproduction Assessment

Yes. Source inspection on current main shows the CLI requests only the method-level pairing scope while the handler derives `operator.admin` for `system.run` nodes; the supplied Multipass scripts are a credible live reproduction path but were not run here.

## Solution Assessment

Unclear. The straightforward lower-scope fix conflicts with current docs, tests, and merged authz history, so maintainers should choose whether to preserve the layered gate with an admin-capable automation path or explicitly reclassify node approval policy.
