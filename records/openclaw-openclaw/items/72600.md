## Summary

Keep open. Current main still lacks the requested Discord auto-thread gate and delivery text cleanup, so this is not obsolete, but the PR is not mergeable as-is because it targets pre-split Discord threading code and introduces bounded-message and reply-threading regressions.

## What This Changes

The PR adds Discord auto-thread gating, outbound Discord text sanitization/chunking changes, cron failure message humanization, generated config metadata, and related tests.

## Best Possible Solution

Rebase the branch onto current main, move the auto-thread gate into the current `threading.auto-thread.ts` implementation, preserve existing explicit reply-reference semantics while letting shared outbound policy handle single-use implicit replies, cap shared failure-detail text, add the required changelog entry, and then have a maintainer explicitly accept or reject the new default Discord threading heuristic.

## Reproduction Assessment

Yes for the review findings: current-main source and tests show where the PR conflicts with the split threading helper, bounded cron failure text, and reply-reference behavior. For the broader feature, reproduction is an absence check on current main: `rg` finds no `autoThreadGate`, and `maybeCreateDiscordAutoThread` has no content gate.

## Solution Assessment

No, not as-is. The direction is plausible, but the patch must be rebased into the current helper layout, keep bounded failure details and explicit reply references, document the user-facing config/default change, and receive maintainer approval for the Discord auto-threading behavior.
