## Summary

Keep open: current main and v2026.6.10 still provide non-interactive OpenAI setup through the API-key path only, while OpenAI/Codex OAuth and device-code setup remain interactive and need a maintainer-approved auth/security contract before implementation.

## What This Changes

Request to add a non-interactive way to configure OpenAI/Codex OAuth for model-provider setup, such as supplying an OAuth callback URL through environment/config.

## Best Possible Solution

Design an OpenAI-plugin-owned non-interactive OAuth import contract that preserves PKCE/state validation, avoids secret-bearing env/log exposure, documents the CLI/env behavior, and includes focused tests plus redacted real-flow proof.

## Reproduction Assessment

Yes, by source inspection rather than a live bug repro: OpenAI OAuth/device-code methods have no runNonInteractive hook, and the non-interactive setup path rejects those methods. This is a feature/config request, not a broken existing behavior report.

## Solution Assessment

Unclear: a callback URL environment variable is one possible interface, but it is not clearly the safest one. Maintainers should choose a leak-resistant provider-owned code, token, or auth-session contract before implementation.
