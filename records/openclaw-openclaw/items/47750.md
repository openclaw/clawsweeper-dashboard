## Summary

Keep open: current main and the latest release still do not provide a non-interactive Codex OAuth setup path. The request is a valid provider-auth feature, but accepting OAuth callback material through env or automation is a new auth/config surface that needs maintainer product and security design.

## What This Changes

Requests an env-driven or otherwise non-interactive way to provision OpenAI Codex OAuth for model provider setup using a ChatGPT/Codex subscription.

## Best Possible Solution

Design an OpenAI-plugin-owned non-interactive Codex auth setup contract that validates state/PKCE, avoids persisting callback material in env/logs, documents CLI/env behavior, and covers browser, device-code, and automation flows with focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection on current main gives high-confidence current behavior: non-interactive provider setup requires runNonInteractive, while OpenAI Codex OAuth/device-code only expose interactive run handlers.

## Solution Assessment

Unclear: an env var accepting a full callback URL is one possible interface, but OAuth callback material needs a provider-owned security contract before that can be called the best solution.
