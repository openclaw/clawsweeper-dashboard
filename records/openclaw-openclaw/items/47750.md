## Summary

Keep open: current main and v2026.6.8 still support non-interactive OpenAI setup through the API-key path only, while ChatGPT/Codex OAuth and device-code setup remain interactive and need a maintainer-approved auth/security contract before implementation.

## What This Changes

The issue requests a non-interactive way to configure the OpenAI provider with Codex/ChatGPT OAuth, with an environment-supplied callback URL proposed as one possible interface.

## Best Possible Solution

Design an OpenAI-plugin-owned non-interactive Codex auth contract that preserves PKCE/state validation, avoids leaking OAuth material, documents the CLI/env behavior, and covers credential storage for fresh setup and upgrades.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Current behavior is source-confirmed because non-interactive provider setup requires runNonInteractive, OpenAI OAuth/device-code methods lack it, and the only OpenAI env setup metadata is OPENAI_API_KEY.

## Solution Assessment

Unclear: a callback URL env var is one possible interface, but it is not clearly the safest one. Maintainers should first choose a leak-resistant provider-owned OAuth code/state, token, or auth-session contract.
