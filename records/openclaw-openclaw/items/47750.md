## Summary

Keep open: current main still has non-interactive OpenAI setup only for API-key auth, while OpenAI Codex browser OAuth and device-code auth remain interactive provider methods. The requested callback/env import is a new security-sensitive auth-provider contract, so it needs maintainer product and security design rather than cleanup closure.

## What This Changes

Add a non-interactive way, such as an environment-supplied OAuth callback URL, to configure OpenAI Codex OAuth for model-provider setup.

## Best Possible Solution

Design an OpenAI-plugin-owned non-interactive Codex auth contract that preserves PKCE/state validation, avoids leaking callback material, documents the CLI/env behavior, and covers credential storage for fresh setup and upgrades.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives a high-confidence current-behavior check that non-interactive provider setup requires runNonInteractive, and OpenAI Codex OAuth/device-code methods do not implement it.

## Solution Assessment

Unclear: accepting a full callback URL from env is one possible interface, but maintainers should first choose a leak-resistant provider-owned OAuth code/state handling contract.
