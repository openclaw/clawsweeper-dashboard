## Summary

Keep open: current main and v2026.6.10 still provide non-interactive OpenAI setup through the API-key path, while OpenAI/Codex OAuth remains browser/device-code/manual-paste based and needs a maintainer-approved auth/security contract before implementation.

## What This Changes

The issue requests a non-interactive setup path for the OpenAI/Codex OAuth provider, such as supplying a callback URL through environment configuration.

## Best Possible Solution

Design an OpenAI-plugin-owned non-interactive OAuth import contract that preserves PKCE/state validation, avoids secret-bearing env/log exposure, documents the CLI/env behavior, and includes focused tests plus redacted real-flow proof.

## Reproduction Assessment

Yes. By source inspection, `openclaw models auth login` requires an interactive TTY and OpenAI OAuth/device-code methods lack `runNonInteractive`, while the API-key method is the env-backed non-interactive path.

## Solution Assessment

Unclear. A callback URL environment variable is one possible interface, but maintainers should choose the leak-resistant provider-owned code, token, callback, or auth-session contract before implementation.
