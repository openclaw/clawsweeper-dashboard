## Summary

Keep open. Current main supports interactive Codex browser OAuth plus a device-code/manual-paste fallback, but it does not implement a non-interactive callback/env setup path: direct auth login rejects non-TTY execution, and non-interactive onboarding requires provider methods to implement `runNonInteractive`, which the OpenAI Codex auth methods do not.

## What This Changes

The issue requests a non-interactive way to configure the OpenAI Codex OAuth provider, such as supplying the OAuth callback URL through an environment variable.

## Best Possible Solution

Add a provider-owned non-interactive Codex OAuth completion path that preserves OAuth state/PKCE validation and safe credential storage, likely as an explicit CLI/onboarding contract rather than an implicit env-only shortcut. Keep the current device-code flow as the headless interactive fallback and document the chosen automation path.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: `openclaw models auth login --provider openai-codex` rejects non-TTY input, and non-interactive onboarding would reject the Codex auth choice because its methods lack `runNonInteractive`.

## Solution Assessment

Unclear. The requested capability is valid, but the proposed env callback URL is not yet proven to be the safest or most maintainable API; a maintainer should choose the OAuth automation shape before implementation.
