## Label Justifications

- `P1`: This is a current-release regression that prevents an installed external messaging channel from starting, breaking a real channel workflow.
- `impact:message-loss`: The affected plugin is a chat channel, and the gateway not loading it prevents inbound and outbound Weixin messages from being delivered.

## Summary

This should stay open: current main still routes gateway runtime loading through a startup plugin-id plan, and that plan does not include an externally installed channel plugin merely because `plugins.entries.<id>.enabled=true`, even though the reported plugin’s published install flow uses exactly that configuration.

## What This Changes

The issue asks OpenClaw Gateway to load enabled externally installed npm plugins from the installed plugin index so the Weixin channel can start after install/login.

## Best Possible Solution

Make the gateway startup planner include explicitly enabled installed channel plugins from the persisted index/manifest contract, or have the install/login flow write the canonical channel activation state, with regression coverage for the plugin-entry-only Weixin flow.

## Reproduction Assessment

Yes at source level, but not by a live packaged run: use an installed channel plugin whose manifest declares a channel but no `activation.onStartup`, set only `plugins.entries.<id>.enabled=true`, and the current startup plan will not pass that id to the gateway runtime loader.

## Solution Assessment

Yes, the right fix is in the generic startup planning or install/login activation handoff, not a Weixin-specific loader exception; it must keep existing allow/deny and explicit-trust rules intact.
