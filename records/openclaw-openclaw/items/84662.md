## Summary

Keep open: current main and the latest release still serialize the OpenClaw runtime-context wrapper into Codex `turn/start.input`, and the linked open draft PR is only a fix candidate until it lands or is replaced.

## What This Changes

The issue reports that Codex app-server per-turn OpenClaw runtime context is persisted as normal native Codex user history, causing repeated context growth across later turns.

## Best Possible Solution

Land the linked Codex app-server fix, or an equivalent narrow replacement, so the real user request stays in Codex turn input while OpenClaw runtime context is delivered through a turn-scoped non-authoritative context channel.

## Reproduction Assessment

Yes for source-level reproduction: current main still builds the `OpenClaw runtime context for this turn:` wrapper and routes the decorated text into Codex `turn/start.input` as a normal text user input item. The reporter's structured logs add live evidence that historical `role=user` entries with that marker were replayed into later `response.create` and compaction inputs.

## Solution Assessment

No for current main: preserving current-turn context is necessary, but persisting the decorated wrapper as user input is the wrong lifetime boundary. The best path is the linked additional-context split or an equivalent narrow fix that avoids elevating user-editable workspace context to developer/system authority.
