## Summary

Keep open: current main and v2026.6.8 still build the reported runtime-context wrapper and submit it as Codex `turn/start.input` text, while the open linked fix PR is draft/conflicting and has not landed. The issue should stay open until that PR, or an equivalent maintainer-approved fix, lands or is closed.

## What This Changes

The issue asks OpenClaw to stop persisting per-turn Codex app-server runtime context as native Codex user history so later model calls do not replay repeated context blocks.

## Best Possible Solution

Land or replace the linked PR with a narrow Codex app-server fix that sends only the real user request as `turn/start.input` and keeps OpenClaw runtime context in an untrusted current-turn context lane with regression coverage.

## Reproduction Assessment

Yes: current main clearly builds the runtime-context wrapper and passes the decorated text into Codex `turn/start.input` as a text user item. I did not run a live multi-turn Codex session in this read-only pass, but the reporter supplied structured live logs showing the same marker accumulating as model-visible `role=user` history.

## Solution Assessment

Yes: the best maintainable fix is to keep OpenClaw runtime context available to the current Codex turn without making it the real user input text. The linked PR’s `additionalContext` direction matches the inspected Codex contract, but it still needs maintainer review and merge-readiness work.
