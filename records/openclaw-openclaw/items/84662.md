## Summary

Keep open: current main still constructs `OpenClaw runtime context for this turn:`, prepends it into Codex `turn/start.input`, and serializes that decorated prompt as a normal text user input item. The only directly linked fix PR is closed unmerged, and its review history shows the remaining fix needs a Codex protocol and prompt-boundary decision rather than cleanup closure.

## What This Changes

The issue reports that Codex app-server stores per-turn OpenClaw runtime context in native Codex user history, causing repeated context to accumulate in later model inputs.

## Best Possible Solution

Choose a Codex-supported non-persisted reference-context path, keep `turn/start.input` as the real user request, sanitize stale decorated history before replay/compaction, and add multi-turn plus stale-bound-thread regression coverage with redacted real app-server proof.

## Reproduction Assessment

Yes, source-level with high confidence: current main still builds the reported marker, prepends it into the Codex turn prompt, and serializes it as `turn/start.input`. The reporter also supplied structured live logs showing that marker accumulating in model-visible `role=user` history, but I did not run a fresh live native Codex session during this read-only review.

## Solution Assessment

Not yet: the issue direction is correct, but the previous PR was closed unmerged after protocol and prompt-authority concerns remained. The safest fix should be selected by maintainers before another implementation tries to move runtime/workspace context into a different app-server channel.
