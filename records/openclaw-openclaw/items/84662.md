## Summary

Keep open: current `main` still builds a Codex app-server turn prompt by prepending OpenClaw runtime context and then serializes that decorated text as normal `turn/start` user input, so the reported native-history growth remains a plausible current bug rather than a cleanup close.

## What This Changes

The issue asks OpenClaw to keep per-turn Codex app-server runtime context available for the active model call without saving that decorated prompt as ordinary native Codex user history.

## Best Possible Solution

Keep current-turn OpenClaw reference context model-visible for Codex while ensuring only the real user request is persisted or replayed, with regression coverage for multi-turn native threads and compaction/replay.

## Reproduction Assessment

Partially: source inspection gives a high-confidence path where decorated runtime context is serialized as normal Codex user input, but I did not establish a live multi-turn `response.create` trace against current `main`. A focused mocked native-thread test or live app-server trace should reproduce whether historical copies accumulate.

## Solution Assessment

Yes: the maintainable fix is to make this OpenClaw runtime context current-turn-only or sanitize historical decorated user messages while preserving the real request. Promoting all workspace and memory context to developer instructions would be a riskier solution because some of that content is user-editable reference data.
