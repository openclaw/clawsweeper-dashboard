## Summary

Keep open: current main still builds the `OpenClaw runtime context for this turn:` block and sends the decorated text as normal Codex `turn/start` user input, while the reporter's structured live logs show those decorated user items accumulating in `response.create` and compact inputs. The linked fix PR was closed unmerged, so there is no implemented cleanup to close against.

## What This Changes

The issue reports that Codex app-server persists per-turn OpenClaw runtime context as native user history, causing repeated runtime-context blocks to grow across later Codex turns.

## Best Possible Solution

Choose a supported non-persisted, lower-authority current-turn context path for Codex app-server, or a compatibility-safe fresh-thread/sanitized-history design, then add multi-turn and compaction regression coverage plus real after-fix Codex proof.

## Reproduction Assessment

Yes, source-plus-log reproducible: current main still decorates Codex `turn/start` text input with the runtime-context marker, and the reporter supplied structured live counts showing that marker accumulating in model-visible `role=user` history. I did not run a live native Codex session in this read-only review.

## Solution Assessment

Unclear until maintainers choose the prompt-boundary and protocol path: the desired end state is sound, but the closed fix PR shows that moving context channels or resetting native bindings needs explicit approval and real upgrade proof.
