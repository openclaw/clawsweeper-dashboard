## Label Justifications

- `P1`: The report describes runaway native Codex history growth that can make an active agent workflow slow, expensive, and hard to compact.
- `impact:session-state`: The affected state is native Codex conversation history and replay context accumulating per-turn OpenClaw runtime scaffolding.

## Summary

Keep this open: current main still builds the `OpenClaw runtime context for this turn:` block, prepends it to the Codex app-server turn text, and submits that decorated text as the normal `turn/start.input` text item. The report is narrow and source-supported, but the safest fix needs maintainer review because the available local protocol surfaces are either persistent user input or higher-priority turn developer instructions.

## What This Changes

The issue asks OpenClaw to keep per-turn Codex app-server runtime context available for the active model call without saving it as ordinary native Codex user history and replaying it on later turns.

## Best Possible Solution

Add or use a non-persistent, lower-priority current-turn context path for Codex app-server turns; if that is not available, narrowly rewrite historical decorated native user items so only the real user request remains before replay or compaction.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only sweep. Source inspection confirms current main sends the decorated runtime-context block as normal `turn/start` user text, while the issue's log evidence covers the native `response.create` replay accumulation.

## Solution Assessment

Unclear as an automation fix: the report's direction is right, but the safest implementation depends on the Codex app-server history/transient-context contract. Avoid promoting user-editable context to developer instructions without an explicit maintainer decision.
