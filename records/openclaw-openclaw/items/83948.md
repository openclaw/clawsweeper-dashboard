## Summary

Keep open: the reported warning path still appears valid when a configured Telegram `defaultAccount` is also the first sorted account id, because current warning logic does not distinguish an explicit default from fallback selection.

## What This Changes

The issue requests that Telegram default-account warning logic honor `channels.telegram.defaultAccount` so `openclaw status` does not warn when an explicit default is already configured.

## Best Possible Solution

Fix the Telegram account-selection warning to suppress `accounts.default is missing` whenever `channels.telegram.defaultAccount` resolves to a configured account, and add a multi-account regression where that explicit default sorts first.

## Reproduction Assessment

Yes by source inspection: a config with multiple Telegram accounts, no `accounts.default`, and `defaultAccount` set to the first sorted account still satisfies `shouldWarnMissingDefault`. I did not run `openclaw status` in this read-only review.

## Solution Assessment

Yes, the suggested direction is the narrow maintainable fix: carry whether a valid explicit `defaultAccount` was used into the warning decision instead of inferring fallback solely from the resolved account id.
