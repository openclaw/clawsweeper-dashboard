## Summary

Keep open: current main still treats lightweight context and tool allowlists as optional per-job controls, Control UI defaults isolated agent-turn jobs to full context unless the checkbox is set, and there is no workspace-wide cron defaults config; resolving this needs maintainer product/security judgment rather than cleanup closure.

## What This Changes

The issue asks for isolated `agentTurn` cron jobs to default or warn toward lightweight context and restricted tools, with workspace/UI/API controls to prevent runaway token use.

## Best Possible Solution

Maintainers should choose a canonical safe-default policy for isolated cron agent turns, then align cron tool/API/CLI/UI/docs and diagnostics so low-cost scripted jobs are easy while full-context/full-tool jobs remain explicit.

## Reproduction Assessment

Yes, at source level: omitted `toolsAllow` still means the full embedded tool list, `lightContext` is opt-in or heuristic, and Control UI defaults `payloadLightContext` to false. I did not run a live cron job to verify the exact 60K-100K token counts.

## Solution Assessment

No, not as an automatic one-line default flip. The best fix needs a maintainer decision across safe defaults, explicit full-context/full-tool opt-in, workspace config, Control UI parity, and warnings so existing cron jobs do not silently change semantics.
