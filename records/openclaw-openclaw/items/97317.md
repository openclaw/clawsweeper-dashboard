## Summary

Keep open: current main still leaves isolated cron `agentTurn` jobs with opt-in `lightContext`, optional/creator-inherited `toolsAllow`, Control UI full-context defaults, and no workspace-level cron defaults, so the reported safe-default/cost-control problem is not implemented on main and needs product/security review.

## What This Changes

The issue requests safer isolated cron `agentTurn` defaults, workspace/UI controls, warnings, and docs so omitted `toolsAllow` and `lightContext` do not silently load broad tools and full workspace context.

## Best Possible Solution

Maintainers should choose a canonical safe-default policy for isolated cron agent turns, then align cron tool/API/CLI/UI/config/docs and diagnostics so lightweight scripted jobs are cheap by default while full-context or full-tool jobs remain explicit.

## Reproduction Assessment

Yes, at source level: omitted `toolsAllow` still means broad embedded tool availability or creator-inherited tools, `lightContext` remains opt-in or heuristic, and Control UI defaults `payloadLightContext` to false. I did not run a live cron job to verify the exact 60K-100K token totals.

## Solution Assessment

No, not as an automatic one-line default flip. The maintainable fix needs a maintainer decision across safe defaults, explicit full-context/full-tool opt-in, workspace config, Control UI parity, warnings, and compatibility for existing cron jobs.
