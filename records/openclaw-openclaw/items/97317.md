## Summary

Keep open: current main still allows isolated agent-turn cron jobs to run with full context/full tool surface when the payload does not explicitly narrow it, and fixing that default crosses security, compatibility, UI, config, and docs policy.

## What This Changes

The issue requests safer isolated agent-turn cron defaults, workspace-level cron default controls, UI/tooling guardrails, and docs for the cost of omitted `toolsAllow` and `lightContext`.

## Best Possible Solution

Maintain the current per-job controls until maintainers choose a compatibility-safe cron default policy, then align the cron tool/API/CLI/UI/docs and add regression coverage for both new and existing jobs.

## Reproduction Assessment

Yes at source level: current main treats omitted `lightContext` as full context except for a narrow command-style heuristic, and omitted `toolsAllow` can inherit the creator tool surface or remain unrestricted. I did not run a live model-backed cron job to verify the reported token counts.

## Solution Assessment

No for an automatic default flip; the maintainable fix needs a maintainer-selected policy for safe defaults, explicit full-context/full-tool opt-in, workspace config, UI parity, and warnings without silently changing existing cron semantics.
