## Summary

Keep open: current main has per-job `toolsAllow` and `lightContext`, but it still does not provide the reported safe isolated-cron default, workspace-wide cron defaults, or a Control UI `toolsAllow` field; changing these defaults needs maintainer product/security review because it can affect existing scheduled jobs.

## What This Changes

Request to make isolated `agentTurn` cron jobs cheaper and safer by default by tightening or configuring `lightContext` and `toolsAllow`, adding UI/API guardrails, and documenting token-cost implications.

## Best Possible Solution

Maintainers should choose an explicit cron default policy that preserves compatibility where needed while making safe low-context/tool-restricted scheduled runs obvious in the CLI, API, Control UI, and docs.

## Reproduction Assessment

Yes, at source level: create or inspect an isolated `agentTurn` cron payload without `toolsAllow` and `lightContext`; current main keeps those fields optional, the UI defaults light context off, and the runtime only uses lightweight context for explicit or command-like cases. I did not run a live token-count reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation choice: defaulting every isolated cron to lightweight/minimal authority addresses the waste, but can break legitimate jobs that need full context or broader tools. The best fix needs maintainer policy on default compatibility, explicit opt-in/opt-out, UI/API shape, and warnings.
