## Summary

Keep open: the report is credible and current main still has an inconsistent systemd token-handling boundary where diagnostics reject inline `OPENCLAW_GATEWAY_TOKEN` but service install/repair can still render durable token-shaped environment into supervisor metadata. The related #76346 work addressed the nvm/system Node path portion and explicitly left token/env-file behavior out of scope.

## What This Changes

The issue asks for Linux systemd gateway install and doctor repair to stop regenerating units that inline `OPENCLAW_GATEWAY_TOKEN` after `gateway install --force`.

## Best Possible Solution

Make systemd install and doctor render managed gateway auth material through owner-only environment files or config, pass value-source metadata through the normal install path, and keep the status warning for genuinely stale inline tokens.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current audit code flags inline `OPENCLAW_GATEWAY_TOKEN`, and the install/render path still has sources that can reach systemd `Environment=` instead of a file-backed source. The issue body also provides concrete v2026.5.3 Ubuntu commands and redacted unit output.

## Solution Assessment

No, the current behavior is not the best solution because `gateway install --force` should not recreate the state that status/doctor immediately warns about. The narrow maintainable fix belongs in service env planning/rendering and install metadata propagation, not by weakening the audit warning.
