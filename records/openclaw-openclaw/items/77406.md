## Summary

Current main already resolves the Linux systemd token audit loop: install/doctor planning no longer needs to serialize durable gateway token values inline, and systemd staging moves state-dir `.env` values into an owner-only EnvironmentFile while filtering them from `Environment=` lines.

## What This Changes

The issue asks for Linux systemd gateway install and doctor repair to stop regenerating units with inline `OPENCLAW_GATEWAY_TOKEN` after v2026.5.3.

## Best Possible Solution

Keep the current main behavior: durable gateway token values should load from config/SecretRef or the systemd EnvironmentFile, while the generated unit keeps only non-secret service metadata inline.

## Reproduction Assessment

Yes. The report gives a high-confidence v2026.5.3 Linux systemd reproduction with redacted `systemctl --user cat` output after both doctor repair and hard reinstall; I did not reproduce a failing path on current main because source and regression tests now cover the fixed path.

## Solution Assessment

Yes. The current main solution is the narrow maintainable fix: install/doctor preserve source metadata and systemd emits dotenv-backed secrets through an EnvironmentFile instead of changing the audit warning or adding a policy knob.
