## Summary

Keep open: current main still drops gateway install environment source metadata before calling the platform service installer, and the linked open PR #77427 explicitly uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that Linux systemd gateway install and doctor repair paths still regenerate service units with inline `OPENCLAW_GATEWAY_TOKEN` despite diagnostics recommending `gateway install --force` to remove it.

## Best Possible Solution

Review and land #77427, or an equivalent narrow patch that threads `environmentValueSources` and existing source metadata through the gateway install handoff with regression coverage, then let the issue close from the merged PR.

## Reproduction Assessment

Yes by source inspection: the primary `gateway install` path computes value-source metadata but does not pass it to `service.install`, while the systemd writer and audit require that metadata to avoid inline token warnings. I did not run a live Ubuntu systemd reproduction in this read-only pass.

## Solution Assessment

Yes, the linked PR direction is the narrow maintainable fix for the primary install loop: preserve source metadata through the install handoff and cover it with a regression test. Maintainers should also confirm adjacent repair/start paths do not drop the same metadata.
