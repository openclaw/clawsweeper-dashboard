## Repair Work Prompt

Fix the Linux systemd gateway service inline-token remediation loop reported in https://github.com/openclaw/openclaw/issues/77406. The observable bug is that `openclaw doctor` repair and `openclaw gateway install --force` can regenerate a systemd user unit containing inline `Environment=OPENCLAW_GATEWAY_TOKEN=...`, while `gateway status --deep --require-rpc` immediately says reinstall should remove that embedded token. Keep the change scoped to gateway service install/doctor repair/audit behavior; do not change gateway auth policy, token generation semantics, the Node path fix from #76346, or unrelated macOS LaunchAgent behavior unless a shared helper is clearly necessary. Start by adding a regression test for the systemd forced reinstall and doctor repair paths with an existing embedded token and/or generated/configured gateway token, then make the generated unit either use an owner-only EnvironmentFile source or no service token metadata at all. Ensure status/audit no longer reports `gatewayTokenEmbedded` after the repaired install. Add a user-facing changelog fix entry if runtime behavior changes.
