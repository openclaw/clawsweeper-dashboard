## Label Justifications

- `P2`: This is a normal-priority Control UI behavior bug with wrong-session selection risk for multi-agent users but no crash or core runtime outage.
- `impact:session-state`: The picker can display and switch to sessions outside the active agent, mis-associating the user's active chat session state.
- `impact:message-loss`: A user can accidentally send a message to a different agent/session after selecting a row that should not appear in the scoped picker.

## Summary

Keep this open: current main and v2026.5.22 still have a source-proven unscoped Control UI picker path, and open fix PRs already own the implementation path, so this issue should wait for a fix to land rather than close now.

## What This Changes

The issue reports that Control UI Chat's session picker calls and renders `sessions.list` without the selected agent scope, letting multi-agent users see and select other agents' sessions.

## Best Possible Solution

Review and land the more complete scoped-picker fix in https://github.com/openclaw/openclaw/pull/85965, or reconcile https://github.com/openclaw/openclaw/pull/86187 into that path, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, by source inspection: current main builds chat picker `sessions.list` params without `agentId` and renders raw result rows, while the gateway already supports agent-scoped listing. I did not run a live UI repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is to scope Control UI picker requests to the active agent and render through the existing filtered session option projection; https://github.com/openclaw/openclaw/pull/85965 appears to cover that fuller path better than the narrower linked PR.
