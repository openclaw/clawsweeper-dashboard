## Label Justifications

- `P2`: This was a normal-priority Control UI behavior bug for multi-agent users with wrong-session selection risk but no crash or security bypass.
- `impact:session-state`: The report concerns the session picker showing and allowing selection of sessions outside the active agent scope.

## Summary

Close: current main and the shipped v2026.6.6 tag already scope the Control UI Chat session picker by active agent, with regression coverage and merged provenance from https://github.com/openclaw/openclaw/pull/85965.

## What This Changes

The issue asks for the Control UI Chat session picker to pass the active agentId to sessions.list so it only shows sessions for the selected agent.

## Best Possible Solution

Keep the shipped scoped-picker implementation from https://github.com/openclaw/openclaw/pull/85965 and handle distinct remaining picker pagination or subagent-volume problems under their own reports.

## Reproduction Assessment

No current-main failure repro remains: current main sends agentId in picker requests and tests cover switching from main to ops. The original v2026.5.22 report was source-proven against older code and is now fixed.

## Solution Assessment

Yes. The landed fix is the narrow maintainable solution because it reuses the existing sessions.list agentId contract and Control UI filtering without changing the Gateway contract.
