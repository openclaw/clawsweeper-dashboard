## Label Justifications

- `P2`: This is a normal-priority Control UI behavior bug with wrong-session selection risk but no crash, security bypass, or data loss.
- `impact:session-state`: The picker can present and switch to sessions from the wrong agent, which mis-associates the user's active chat session state.
- `impact:message-loss`: A user can accidentally send a message to another agent/session after selecting a row that should not appear in the scoped picker.

## Summary

Keep this issue open: current main still lets the Chat session picker render rows from the unscoped session result, and the picker search request does not include the active `agentId` even though the Gateway contract already supports agent-scoped listing.

## What This Changes

The issue asks for the Control UI Chat session picker to list only sessions belonging to the currently selected agent.

## Best Possible Solution

Thread the selected agent into the Chat session picker request and render path, then add a focused multi-agent regression test so the popover cannot list another agent's sessions.

## Reproduction Assessment

Yes for source reproduction: current main renders picker rows from the unfiltered session result and sends picker search requests without `agentId`, while the Gateway already supports agent filtering. I did not perform a live UI run in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow Control UI change that reuses the existing `sessions.list` `agentId` contract and adds regression coverage for a selected agent with other agents present.
