## Summary

Keep this issue open: current main still lets the chat session picker render the unfiltered `sessionsResult` rows and its picker search request does not pass the active `agentId`, while the Gateway contract already supports agent filtering.

## What This Changes

The issue asks for the Control UI Chat session picker to list only sessions for the currently selected agent instead of all configured agents.

## Best Possible Solution

Keep the picker visually and remotely scoped to the active chat agent by filtering the displayed rows and passing the active `agentId` on picker search/load-more requests, with regression coverage for multi-agent results.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: with multi-agent `sessionsResult` loaded and `chatSessionPickerResult` null, the popover repeats raw `state.sessionsResult.sessions`, and picker search calls `sessions.list` without `agentId`. I did not run the browser flow live in this read-only review.

## Solution Assessment

Yes, the narrowest fix is to keep the Control UI picker scoped at both display and RPC-request boundaries while relying on the existing Gateway `sessions.list({ agentId })` contract; no backend or product change is needed.
