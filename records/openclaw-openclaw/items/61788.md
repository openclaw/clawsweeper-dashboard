## Summary

Keep open. Current main still lets a WhatsApp opening-phase 408/ETIMEDOUT from `waitForWaConnection` escape before the listener reconnect path is active; the merged timing and idle-keepalive work covers adjacent socket behavior but does not close this initial connection retry gap.

## What This Changes

The issue reports that WhatsApp pairing succeeds but the WebSocket fails during initial gateway startup with a Boom-style 408/ETIMEDOUT, causing the channel to exit instead of staying connected.

## Best Possible Solution

Add a WhatsApp-owned opening-phase 408/ETIMEDOUT retry path that reuses the existing bounded reconnect policy and keeps terminal logged-out/conflict handling intact.

## Reproduction Assessment

Yes for a source-level path: emit or mock a Boom-like pre-open `connection.update` close with status 408/ETIMEDOUT while `monitorWebChannel` opens the WhatsApp connection. Current main closes and rethrows before listener-based reconnect handling is active; no live WhatsApp server reproduction was run.

## Solution Assessment

No. Current main exposes useful Baileys timing controls and handles post-open 408s, but the narrow maintainable fix is to route retryable opening-phase 408/ETIMEDOUT through the same bounded reconnect policy instead of adding a new config knob or unbounded loop.
