## Summary

Keep open. Current main still lets a Boom-like WhatsApp opening-phase 408/ETIMEDOUT escape before the listener reconnect path is active, while the related TCP keepalive and socket-timing work covers already-established idle connections rather than this initial connection timeout.

## What This Changes

The issue reports that WhatsApp Web credentials are created after QR pairing, but the gateway immediately exits with a WebSocket 408/ETIMEDOUT during startup.

## Best Possible Solution

Add a WhatsApp-owned opening-phase 408/ETIMEDOUT retry path that reuses the existing bounded reconnect policy while preserving terminal status handling and existing socket timing controls.

## Reproduction Assessment

Yes, for a source-level path: mock waitForWaConnection to reject with a Boom-like 408/ETIMEDOUT while monitorWebChannel opens the WhatsApp connection. Current main closes and rethrows before listener-based reconnect handling is active; no live WhatsApp server reproduction was run.

## Solution Assessment

No, current main is not the complete solution because it retries opening-phase 428 but not retryable opening-phase 408. The narrow maintainable repair is to route 408/ETIMEDOUT through the same bounded reconnect policy instead of adding a new config knob or unbounded loop.
