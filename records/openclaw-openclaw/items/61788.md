## Summary

Keep open. Current main includes a shipped WhatsApp/Baileys socket timing improvement that may mitigate some 408 timeout loops, but the reported startup failure class is still present: a close during the initial `waitForWaConnection` path is rethrown before the established listener reconnect policy can run, and the monitor only retries the special auth-unstable startup error.

## What This Changes

The issue requests that WhatsApp Web stay connected or recover after successful QR pairing instead of exiting on startup with an ETIMEDOUT/WebSocket 408 close.

## Best Possible Solution

Keep the shipped Baileys timing support, then add a narrow WhatsApp startup retry path for retryable non-auth initial closes such as 408/ETIMEDOUT. The retry should reuse the existing reconnect policy and non-retryable status classification, and add focused tests for a pre-listener startup close plus logged-out/conflict stop behavior.

## Reproduction Assessment

Yes for a source-level reproduction: mock `waitForWaConnection` to reject with a Boom-like 408/ETIMEDOUT during `controller.openConnection`, and current `monitorWebChannel` will rethrow instead of scheduling a reconnect. No live WhatsApp network reproduction was attempted.

## Solution Assessment

No, current main is not yet the best complete solution. The socket timing work is a useful partial fix, but the narrow maintainable fix for this issue is to make retryable startup closes follow the same bounded reconnect policy as post-listener closes.

## Repair Work Prompt

Fix the WhatsApp startup ETIMEDOUT gap tracked by https://github.com/openclaw/openclaw/issues/61788. Current main already has the Baileys timing support from #73580; do not rework TCP keepalive, proxy handling, or the `web.whatsapp.*` config shape. Implement a narrow retry path for retryable non-auth failures thrown while `monitorWebChannel` is awaiting `controller.openConnection`, especially Boom/status 408 or ETIMEDOUT closes from `waitForWaConnection`.
