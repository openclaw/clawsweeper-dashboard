## Summary

Keep open. Current main still has the WhatsApp pre-listener startup gap described in the prior ClawSweeper review: retryable 408/ETIMEDOUT closes from `waitForWaConnection` are rethrown before the managed listener and normal reconnect policy are active.

## What This Changes

The issue reports WhatsApp Web pairing succeeding but the channel exiting shortly after startup with a WebSocket 408/ETIMEDOUT.

## Best Possible Solution

Keep the shipped Baileys timing controls, then add a narrow startup retry path that sends retryable non-auth initial closes through the existing reconnect policy and status classification.

## Reproduction Assessment

Yes for a source-level reproduction: mock `waitForWaConnection` to reject with a Boom-like 408/ETIMEDOUT during `WhatsAppConnectionController.openConnection`, then run `monitorWebChannel`; current main rethrows before scheduling reconnect. No live WhatsApp server reproduction was attempted.

## Solution Assessment

No, current main is not yet the best complete solution. The narrow maintainable fix is to reuse the existing bounded reconnect policy for retryable startup closes instead of adding a separate ad hoc loop.
