## Summary

Keep open. Current main includes the shipped WhatsApp/Baileys socket timing work that may mitigate some 408 timeout loops, but the startup failure class described here still exists: a close during `waitForWaConnection` is rethrown before the managed listener and normal reconnect decision are active, and the monitor startup catch only retries the special auth-unstable path.

## What This Changes

The issue reports that WhatsApp Web pairing succeeds, but gateway startup exits within seconds on a WebSocket 408/ETIMEDOUT close instead of connecting or retrying.

## Best Possible Solution

Keep the shipped Baileys timing controls, then add a narrow WhatsApp startup retry path for retryable non-auth initial closes. It should reuse the existing reconnect policy and non-retryable status classification, preserve 401 logout and 440 conflict stop behavior, and add focused tests for pre-listener 408 retry plus terminal startup closes.

## Reproduction Assessment

Yes for a source-level reproduction: mock `waitForWaConnection` to reject with a Boom-like 408/ETIMEDOUT during `WhatsAppConnectionController.openConnection`, then run `monitorWebChannel`; current main rethrows before scheduling reconnect. No live WhatsApp server reproduction was attempted.

## Solution Assessment

No, current main is not yet the best complete solution. The timing knobs are useful, but the narrow maintainable fix is to send retryable startup closes through the same bounded reconnect policy used after listener setup instead of adding a separate ad hoc loop.
