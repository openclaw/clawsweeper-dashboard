## Summary

Keep open. Current main has adjacent WhatsApp mitigations, including socket timing controls and opening-phase 428 retry, but the reported post-login opening 408/ETIMEDOUT still goes through `waitForWaConnection` and is rethrown before the normal reconnect listener path is active.

## What This Changes

The issue reports WhatsApp Web pairing succeeding, then the gateway exiting on an opening WebSocket ETIMEDOUT/408 instead of staying connected.

## Best Possible Solution

Add a WhatsApp-owned startup retry path that classifies retryable opening-phase 408/ETIMEDOUT through the existing reconnect policy while preserving terminal 401/440 behavior and the shipped socket timing controls.

## Reproduction Assessment

Yes, for a source-level path: mock `waitForWaConnection` to reject with a Boom-like 408/ETIMEDOUT while `monitorWebChannel` opens the WhatsApp connection. Current main rethrows that startup error before listener-based reconnect handling is active; no live WhatsApp server reproduction was run.

## Solution Assessment

No, current main is not yet the best complete solution. The narrow maintainable fix is to reuse the existing bounded reconnect policy for retryable opening-phase 408/ETIMEDOUT rather than adding a separate unbounded loop or changing timing defaults.
