## Summary

Keep open. Current main still treats a pre-open WhatsApp WebSocket 408/ETIMEDOUT close as a startup failure before the normal listener/reconnect path owns the socket, and the adjacent socket-timing, keepalive, QR-login, and post-open reconnect work does not fully cover this gateway-after-login path.

## What This Changes

The issue asks WhatsApp Web to connect and remain running after QR pairing instead of exiting when the initial post-login WebSocket open fails with ETIMEDOUT/408.

## Best Possible Solution

Add a WhatsApp-owned opening-phase 408/ETIMEDOUT retry path that reuses the existing bounded reconnect policy while preserving terminal logged-out, conflict, QR-login, auth-unstable, socket-timing, and relink behavior.

## Reproduction Assessment

Yes, source-level: mock a Baileys pre-open `connection.update` close with status 408/ETIMEDOUT while `monitorWebChannel` opens the connection. Current main rejects before the listener reconnect path is active; no live WhatsApp server run was performed.

## Solution Assessment

No, current main is not the best end state yet. The maintainable fix is to route retryable opening-phase 408/ETIMEDOUT through the existing reconnect policy instead of adding another config knob or relying only on troubleshooting docs.
