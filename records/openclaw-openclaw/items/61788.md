## Summary

Keep open. Current main still handles post-open WhatsApp 408 reconnects and opening-phase 428 retries, but a pre-open 408/ETIMEDOUT from Baileys still escapes the startup wait before the listener reconnect path is active.

## What This Changes

The issue reports WhatsApp Web QR pairing succeeding, then the gateway channel exiting within seconds because the initial WebSocket connection fails with ETIMEDOUT/408.

## Best Possible Solution

Add a WhatsApp-owned opening-phase 408/ETIMEDOUT retry path that reuses the existing bounded reconnect policy while preserving terminal logged-out and conflict handling.

## Reproduction Assessment

Yes, for a source-level path: mock a Baileys pre-open connection.update close with status 408/ETIMEDOUT while monitorWebChannel opens the connection. Current main rejects before the listener reconnect path is active; no live WhatsApp server run was performed.

## Solution Assessment

No, current main is not the best end state yet. The maintainable fix is to route retryable opening-phase 408/ETIMEDOUT through the existing reconnect policy instead of adding another config knob or relying only on timing documentation.
