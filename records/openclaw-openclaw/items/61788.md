## Summary

Keep open. Current main still has the startup failure class: a WhatsApp 408/ETIMEDOUT close during the initial Baileys wait is rethrown before the managed listener and normal reconnect decision are active, so the shipped socket timing work is only a partial mitigation.

## What This Changes

The issue reports that WhatsApp Web pairs successfully but the gateway then exits on an ETIMEDOUT/408 WebSocket startup close instead of staying connected.

## Best Possible Solution

Keep the shipped Baileys timing controls, then route retryable non-auth startup closes through the existing bounded WhatsApp reconnect policy before the monitor exits.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock waitForWaConnection to reject with a Boom-like 408/ETIMEDOUT during WhatsAppConnectionController.openConnection under monitorWebChannel; current main rethrows before scheduling reconnect.

## Solution Assessment

Yes for a narrow repair path. The maintainable fix is not another timing knob, but reusing the existing reconnect classification/backoff for retryable pre-listener startup closes while preserving terminal auth and conflict statuses.

## Repair Work Prompt

Fix the current issue: WhatsApp Web can pair successfully, then the gateway exits on a retryable 408/ETIMEDOUT startup close before the managed listener is installed. In current main, `WhatsAppConnectionController.openConnection` awaits `waitForWaConnection` before listener setup, and `monitorWebChannel` only retries `WhatsAppAuthUnstableError` in that startup catch. Add a narrow retry path for retryable non-auth initial closes that reuses the existing reconnect policy, status normalization, and max-attempt behavior used after listener setup. Preserve 401 logged-out cleanup/relink guidance, 440 conflict stop behavior, existing auth-unstable retry behavior, Baileys timing config, and the shipped docs/schema surface. Add focused tests for pre-listener 408/ETIMEDOUT retry and terminal startup closes, and update `CHANGELOG.md` because this is user-facing WhatsApp reliability behavior. Do not revive the closed TCP keepalive-agent PR unless a maintainer explicitly asks for that separate WSL2 path.
