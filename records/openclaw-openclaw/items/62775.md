## Summary

This PR is superseded: the default-on loop-detection policy was explicitly rejected in the narrower follow-up discussion, current main already has the safer no-progress-gated `generic_repeat` critical block and a merged post-compaction guard, and the remaining exec/sandbox changes are security-sensitive scope that should not ride on this stale mixed branch.

## What This Changes

The PR enables loop detection by default, lowers loop thresholds, adds aggregate `generic_repeat` critical blocking, changes exec approval ask merging, strips trailing error turns before runner sanitization, and allows `SANDBOX_POLICY.json` through `agents.files.set`.

## Best Possible Solution

Keep the current default-off loop-detection policy, use the safer no-progress-gated generic block and merged post-compaction guard on main, and require separate focused review for any exec approval or sandbox file-write policy changes.

Yes for the source-level behavior: current main and tests show default-off loop detection, no-progress-gated generic critical blocking, and stricter exec ask merging. The PR's remaining differences can be compared directly against those contracts without a live setup.

## Solution Assessment

No. The branch mixes a rejected default-policy change with safer behavior already present on main and unrelated security-sensitive exec/sandbox changes, so the maintainable path is to close this branch and use focused follow-ups only where needed.

  - body: The new `params.ask === "off" || approvals.agent.ask === "off"` branch lets an approvals-file `ask: "off"` suppress a caller that requested `ask: "always"`. Current main has a test explicitly protecting that stricter requested mode, so this weakens the approval boundary.

  - body: The PR blocks `generic_repeat` using total matching calls in the history window, so a healthy session that revisits the same command intermittently can be hard-blocked even when results are changing. Current main uses the safer no-progress streak before returning a critical result.
  - confidence: 0.9

  - confidence: 0.86

  - body: Changing `DEFAULT_LOOP_DETECTION_CONFIG.enabled` to `true` now contradicts the explicit default-off maintainer decision and the current docs/tests. Any future policy change would need a new focused owner-approved proposal.
  - confidence: 0.88
