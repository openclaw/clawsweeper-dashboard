## Summary

Keep open. Current main still ships tool-loop detection disabled by default and generic_repeat remains warning-only, so the core problem is real and not implemented on main. This PR is not merge-ready because it changes host exec approval precedence against the current documented/tested contract, hard-blocks generic repeats using aggregate history-window counts, and adds separate transcript/sandbox-policy changes that need their own focused review.

## What This Changes

This PR changes loop-detection defaults and generic_repeat escalation, changes exec approval ask precedence, strips trailing infrastructure-error turns before runner sanitization, and allows SANDBOX_POLICY.json through agents.files.set.

## Best Possible Solution

Land a narrower loop-detection fix that updates source, docs, tests, and changelog together: enable the guard only if maintainers accept that default change, and make generic_repeat hard-block only for consecutive or no-progress repeats. Keep exec approval precedence aligned with the current stricter-of-two-layers contract unless product/security owners explicitly approve a policy change, and split QUEUEFLOOD and SANDBOX_POLICY work into separate focused PRs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve host approval prompts when config asks off:** `src/agents/bash-tools.exec-host-shared.ts:195-198`
  - body: This branch makes `params.ask === "off"` force `hostAsk` to `"off"` even when the host-local approvals file says `ask: "always"`. Current docs and tests define exec approvals as the stricter of the requested config and host approvals, so this would let a gateway/session config silently suppress a host operator's explicit prompt requirement for real host exec.
  - confidence: 0.89
- **[P2] Count only consecutive generic repeats before blocking:** `src/agents/tool-loop-detection.ts:485-500`
  - body: The new critical generic_repeat branch blocks on `recentCount`, which is the total number of matching tool+args calls anywhere in the history window. With the lower default threshold and detection enabled by default, a healthy run that revisits the same read/status command between other successful work can be hard-blocked even though it is not stuck; use consecutive/no-progress evidence for the blocking path.
  - confidence: 0.86
- **[P1] Preserve all tool-call block variants when stripping errors:** `src/agents/pi-embedded-runner/run/attempt.strip-error-turns.ts:76`
  - body: `isPureInfrastructureError` only checks for `type === "toolCall"`, but current transcript repair treats `toolCall`, `toolUse`, and `functionCall` as tool-call state. An errored assistant turn containing `toolUse` or `functionCall` would be classified as disposable infrastructure noise and stripped, losing partial tool state that repair/replay paths need to preserve.
  - confidence: 0.88
