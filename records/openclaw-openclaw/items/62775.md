## Summary

Keep open. The underlying loop/approval problems are still present on current main, but this PR is not safe to merge as-is because it weakens the current exec approval contract and adds hard-blocking/error-stripping behavior that conflicts with existing tests and tool-state handling.

## What This Changes

The PR enables loop detection by default, lowers loop thresholds, adds critical blocking for `generic_repeat`, changes exec approval ask merging, and adds queueflood error-turn stripping plus a `SANDBOX_POLICY.json` workspace-file allowlist.

## Best Possible Solution

Split this into focused owner-reviewed changes: land loop detection with updated tests/docs/changelog and consecutive/no-progress generic blocking, handle exec approval policy separately, and keep queueflood plus sandbox-policy file writes in their own reviewed PRs.

## Reproduction Assessment

Yes. Source and tests provide a high-confidence static reproduction: the PR's default flip contradicts `is disabled by default`, and its approval short-circuit contradicts the current stricter-ask test.

## Solution Assessment

No. The proposed branch addresses real symptoms, but the safest maintainable path is narrower changes that preserve the current approval boundary and avoid aggregate-window hard blocks for healthy repeated commands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not let host ask=off suppress stricter asks:** `src/agents/bash-tools.exec-host-shared.ts:196-199`
  - body: The new `params.ask === "off" || approvals.agent.ask === "off"` branch lets an `exec-approvals.json` value of `ask: "off"` override a caller that requested `ask: "always"`. Current main has a test explicitly preventing that downgrade, so this would weaken the approval boundary and break the existing contract.
  - confidence: 0.92
- **[P2] Count consecutive no-progress repeats before blocking:** `src/agents/tool-loop-detection.ts:484-486`
  - body: The critical `generic_repeat` path uses total matches in the history window, not a consecutive or no-progress streak. A healthy session that revisits the same read/exec command intermittently can be hard-blocked after enough total occurrences, even though it is not stuck in a tight retry loop.
  - confidence: 0.9
- **[P1] Preserve all tool-call block variants when stripping errors:** `src/agents/pi-embedded-runner/run/attempt.strip-error-turns.ts:55`
  - body: The new infrastructure-error predicate preserves only `toolCall` content, but current repair/normalization code treats `toolUse` and `functionCall` as tool-call state too. Stripping an errored assistant turn with either variant can drop partial tool state and break transcript repair.
  - confidence: 0.86
- **[P2] Update the default loop-detection contract everywhere:** `src/agents/tool-loop-detection.ts:34`
  - body: Changing the default to enabled without updating the colocated tests, user docs, schema help/generated schema, and config examples leaves the public contract inconsistent and should fail the existing disabled-by-default test.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `src/agents/tool-loop-detection.ts:34`
