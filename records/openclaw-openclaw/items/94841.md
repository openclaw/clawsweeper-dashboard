## Summary

Keep this PR open, but not ready as-is: the parser-boundary fix and Windows proof are strong, yet the new guard is broader than the reported backslash bug and can break existing forward-slash Windows command strings with trailing tokens.

## What This Changes

The PR adds a QMD command-token helper that bypasses `splitShellArgs` for drive-letter and UNC Windows absolute paths and adds regression tests for backslash, UNC, spaced, and quoted Windows command values.

## Best Possible Solution

Land one canonical QMD parser-boundary fix that preserves backslash drive-letter and UNC commands before POSIX tokenization while preserving existing token extraction for forward-slash drive-letter command strings.

## Reproduction Assessment

Yes. Current main sends `memory.qmd.command` through `splitShellArgs`, whose unquoted backslash branch strips Windows separators; the PR body also supplies Windows before/after terminal output through the real config resolver and availability probe.

## Solution Assessment

No as proposed. The QMD config parser is the right fix boundary, but the implementation should not turn existing forward-slash drive-letter command strings with trailing tokens into raw Windows spawn commands.
