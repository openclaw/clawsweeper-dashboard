## Summary

Keep open: this is a focused fix for the canonical Windows QMD command-path bug with strong Windows terminal proof, but the patch still has one compatibility defect around forward-slash Windows command strings that current main parses to the executable token.

## What This Changes

The PR changes QMD memory backend config parsing to preserve bare Windows drive-letter and UNC absolute command paths and adds regression tests for those command forms.

## Best Possible Solution

Land one canonical QMD parser fix after preserving the forward-slash command-string compatibility boundary or documenting and testing the intended path-only contract, then close the linked issue and duplicate PR attempts.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main runs `memory.qmd.command` through a POSIX-style tokenizer that consumes unquoted Windows backslashes before QMD spawn resolution. The PR body also supplies before/after Windows terminal output for the real config resolver and availability probe.

## Solution Assessment

No, not as-is: the fix is in the right owner boundary, but the absolute-path bypass is too broad for forward-slash Windows command strings with trailing tokens. The safer solution is a narrower parser rule plus a regression test for that compatibility case.
