## Summary

Keep open: the PR targets a real current-main bug, but it is not merge-ready because the new Windows-path branch truncates paths at spaces and the supplied proof is Linux string parsing rather than Windows OpenClaw/QMD behavior.

## What This Changes

The PR adds QMD command-token helpers that bypass POSIX shell parsing for Windows drive-letter and UNC absolute paths and adds backend-config regression tests for those two path forms.

## Best Possible Solution

Land one canonical fix that preserves complete Windows drive-letter and UNC executable paths, including spaces, before POSIX shell parsing, with regression coverage and redacted Windows OpenClaw QMD proof.

## Reproduction Assessment

Yes, source-level: current main routes memory.qmd.command through splitShellArgs, whose unquoted backslash handling strips Windows separators before QMD spawn resolution. I did not run a live Windows QMD repro in this read-only review.

## Solution Assessment

No as submitted: preserving Windows absolute command tokens before POSIX parsing is the right boundary, but the helper should not cut the path at the first space and the PR still needs Windows/OpenClaw proof.
