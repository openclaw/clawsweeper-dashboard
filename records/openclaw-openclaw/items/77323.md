## Summary

Keep open: current main still has Feishu REST/probe calls delegated through the Lark default HTTP instance without REST proxy normalization while managed proxy mode forces process proxy env and global-agent routing, so this remains a narrow plugin-owned bug.

## What This Changes

The issue reports that Feishu bot identity and REST traffic fail under `proxy.enabled=true` because Lark SDK HTTP calls mishandle the managed proxy environment.

## Best Possible Solution

Fix Feishu’s plugin-local HTTP wrapper so Lark REST/token/probe calls avoid Axios/env-proxy double handling while preserving managed proxy routing, then add regression coverage and a changelog entry.

## Reproduction Assessment

Yes, source-reproducible: current main sets managed proxy env and forced global-agent, while Feishu bot identity and REST sends route through the Lark default HTTP instance without REST-specific proxy normalization. I did not perform a live credential run against a local proxy.

## Solution Assessment

Yes for an owner-local Feishu HTTP fix that preserves the managed proxy contract and prevents Axios/env-proxy double handling. No for a broad direct-bypass workaround unless maintainers explicitly decide Feishu REST egress should bypass the operator proxy.
