## Summary

Keep open: the current Feishu REST/token/probe path still delegates to the Lark SDK Axios instance without proxy normalization while managed proxy mode forces process-wide proxy env, so the reported startup failure remains a valid plugin-owned bug.

## What This Changes

The issue asks OpenClaw to keep Feishu REST bot identity and token calls working when `proxy.enabled=true` is configured.

## Best Possible Solution

Add a Feishu-owned HTTP-instance normalization that disables Axios env proxy parsing for Lark REST/token calls while preserving OpenClaw managed proxy egress, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes, source-reproducible: current main sets the managed proxy env and forced global-agent, while Feishu bot identity/token calls use the Lark SDK REST HTTP instance without REST-specific proxy normalization. I did not perform a live Feishu credential run against a local proxy.

## Solution Assessment

Yes for an owner-local Feishu HTTP wrapper fix that prevents Axios env-proxy double handling and preserves managed proxy routing. No for the temporary direct-agent plus `GLOBAL_AGENT_FORCE_GLOBAL_AGENT=false` workaround, because that weakens the documented managed proxy guardrail.
