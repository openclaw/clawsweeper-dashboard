## Summary

Keep open: current main still routes Feishu bot identity, tenant-token, and REST probe calls through the Lark Axios-backed HTTP instance without explicit managed-proxy handling, while the overlapping Feishu proxy PR remains open with proof, policy, and review blockers. The duplicate reports at https://github.com/openclaw/openclaw/issues/79701 and https://github.com/openclaw/openclaw/issues/79744 were already closed back into this canonical proxy failure cluster.

## What This Changes

Fix Feishu startup bot identity failures when `proxy.enabled=true` routes Lark SDK REST/token calls through the managed proxy environment.

## Best Possible Solution

Keep this issue as the canonical Feishu managed-proxy REST bug and fix the Feishu-owned Lark HTTP adapter so token/probe/API calls handle Axios/global-agent proxy behavior explicitly while preserving the documented managed-proxy contract.

## Reproduction Assessment

Yes, at source level: enable managed proxy mode with Feishu configured, and current main combines forced process-wide proxy state with Feishu REST/token/probe calls through the Lark Axios-backed HTTP instance. I did not run live Feishu credentials through a local proxy in this read-only review.

## Solution Assessment

No for the temporary direct-bypass workaround because it can route Feishu outside the operator-managed proxy guardrail. The maintainable solution is a Feishu-local HTTP adapter or wrapper that explicitly handles the Lark/Axios proxy contract, preserves timeout/User-Agent behavior, and has focused proxy regression coverage; open PR https://github.com/openclaw/openclaw/pull/41943 targets the right area but still needs proof and rework.
