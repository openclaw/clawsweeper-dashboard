## Summary

Keep open: source, docs, and dependency review support a current Feishu plugin bug. Current main still leaves Lark SDK REST/Axios requests without managed-proxy normalization while the Feishu WebSocket path is already proxy-aware.

## What This Changes

The issue requests fixing Feishu bot identity REST probes and other Feishu REST calls when `proxy.enabled=true` routes OpenClaw through a managed HTTP proxy.

## Best Possible Solution

Fix the Feishu plugin's Lark SDK HTTP wrapper so REST, token, and bot-probe calls avoid Axios env-proxy double handling while preserving managed proxy routing, then add focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes, source-reproducible: with managed proxy enabled, OpenClaw forces proxy env/global-agent and clears NO_PROXY, while Feishu bot identity goes through the Lark SDK Axios HTTP instance without REST-specific proxy normalization. I did not live-test Feishu credentials against a local Clash/Surge proxy in this read-only review.

## Solution Assessment

No for the temporary direct-agent bypass as a repository fix; the narrow maintainable solution is Feishu-owned HTTP request normalization that lets OpenClaw's managed proxy layer remain the routing authority.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77323 in the Feishu plugin. Current behavior: with `proxy.enabled=true`, OpenClaw sets proxy env and forced global-agent, while Feishu REST/token/probe calls use the Lark SDK Axios default HTTP instance without normalization, so Axios env-proxy handling can conflict with managed proxy routing and bot identity fails. Keep the fix owner-local to `extensions/feishu`; do not change core proxy lifecycle, weaken NO_PROXY clearing, add a config/env knob, or route Feishu REST direct around the managed proxy. Prefer preventing Axios/proxy-from-env from applying a second env proxy layer for Lark SDK REST calls when OpenClaw managed proxy is active, while leaving global-agent/managed proxy as the egress path. Add regression coverage for REST request option normalization plus the existing probe path as useful, keep existing timeout and User-Agent behavior intact, and add a single-line user-facing `CHANGELOG.md` fix entry without bot/maintainer attribution guessing.
