## Summary

Keep open: current main still matches the report, with Feishu bot identity REST calls using the Lark Axios HTTP instance without Feishu-owned proxy normalization while managed proxy mode forces process-wide proxy env and global-agent routing.

## What This Changes

The issue asks Feishu REST and bot identity probes to work when `proxy.enabled=true` instead of failing with `ECONNRESET`.

## Best Possible Solution

Fix Feishu’s owner-local HTTP instance so Lark REST/token/probe calls avoid Axios env-proxy double handling while preserving OpenClaw managed proxy routing, then add regression coverage and a changelog entry.

## Reproduction Assessment

Yes, source reproduction is high confidence: current main combines forced managed proxy env/global-agent routing with a Feishu REST wrapper that delegates to the Lark Axios instance without REST proxy normalization. I did not perform a live Feishu credential run against a local proxy.

## Solution Assessment

Yes for a narrow Feishu-owned HTTP wrapper fix that prevents Axios env-proxy double handling and keeps the managed proxy contract intact. The reporter’s direct-agent workaround is useful evidence but is not the best repository fix if it broadly bypasses operator proxy routing.
