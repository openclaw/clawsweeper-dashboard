## Summary

Keep open: current main still routes Feishu bot identity/token REST traffic through the Lark Axios-backed HTTP instance without an explicit managed-proxy transport decision, and the duplicate Feishu proxy reports have already been folded into this canonical thread.

## What This Changes

The issue asks for Feishu bot identity and REST calls to work when OpenClaw managed proxy mode is enabled.

## Best Possible Solution

Keep this as the canonical Feishu managed-proxy REST bug and land a Feishu-owned HTTP adapter that explicitly handles Lark/Axios proxy behavior while preserving the documented operator-managed proxy contract.

## Reproduction Assessment

Yes, at source level: enable managed proxy mode with Feishu configured and startup bot identity resolution reaches Lark REST/token calls without explicit managed-proxy handling. I did not run live Feishu credentials through a local proxy in this read-only review.

## Solution Assessment

No for the proposed direct-bypass workaround, because it can route Feishu outside the managed proxy guardrail. The maintainable solution is a Feishu-local, target-aware transport adapter or wrapper that makes the Lark/Axios proxy contract explicit and regression-tested.
