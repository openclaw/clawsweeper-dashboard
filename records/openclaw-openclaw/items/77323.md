## Summary

Keep open: current main still routes the Feishu startup identity probe through the Lark SDK Axios-backed HTTP instance with only timeout/User-Agent wrapping, while managed proxy mode still injects HTTP_PROXY/HTTPS_PROXY and clears NO_PROXY. The Proxyline migration improves the process-wide proxy runtime but does not remove Axios's own proxy-from-env pre-routing in this SDK path, and the overlapping fix PR is still open, conflicting, and previously reviewed as needing policy changes and real behavior proof.

## What This Changes

The issue reports that Feishu bot identity probing and REST/token calls fail when `proxy.enabled=true` because the Lark SDK HTTP path collides with OpenClaw managed proxy routing.

## Best Possible Solution

Keep this as the managed-proxy Feishu identity bug and implement a Feishu-owned Lark HTTP adapter that prevents Axios double proxy handling while preserving managed Proxyline egress, timeout/User-Agent behavior, target eligibility, focused tests, and redacted real startup proof.

## Reproduction Assessment

Yes, source-level high confidence: enable `proxy.enabled=true` with Feishu configured, then startup probing reaches `probeFeishu` -> `createFeishuClient` -> the Lark Axios-backed HTTP instance while managed proxy env is active. I did not run live Feishu credentials through the reported local proxy.

## Solution Assessment

No for the suggested direct-bypass workaround because it can route Feishu outside the operator-managed proxy guardrail. The safer solution is a target-aware Feishu HTTP adapter at the Lark `httpInstance` seam that prevents Axios/global proxy conflicts without silently changing egress policy.
