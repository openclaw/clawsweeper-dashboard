## Summary

Keep open: current main still matches the reported Feishu managed-proxy REST/token failure path, and the same symptom was already preserved by closing related reports as duplicates of this canonical issue. No merged fix covers the Feishu Lark Axios HTTP instance path yet.

## What This Changes

Fix Feishu REST/token/bot identity requests so they work when OpenClaw managed proxy mode is enabled.

## Best Possible Solution

Keep this as the canonical Feishu managed-proxy bug and fix the Feishu-owned Lark HTTP wrapper so REST/token/probe calls avoid Axios env-proxy double handling while preserving the documented managed-proxy contract.

## Reproduction Assessment

Yes, at source level: configure Feishu, enable `proxy.enabled=true` with a local HTTP proxy, and startup identity probing runs through the Lark Axios HTTP instance while managed proxy mode injects process-wide proxy env and clears NO_PROXY. I did not perform a live Feishu credential/proxy run in this read-only review.

## Solution Assessment

No for the temporary direct-bypass workaround because it can weaken the managed-proxy guardrail. The maintainable direction is an owner-local Feishu HTTP adapter that disables Axios env-proxy double handling while still routing according to OpenClaw's managed proxy contract.
