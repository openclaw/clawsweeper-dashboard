## Summary

Keep open: current main still delegates Feishu Lark REST/token/probe calls through the Axios-backed SDK default HTTP instance without explicit managed-proxy handling, while managed proxy mode still forces HTTP(S) proxy env/global-agent and clears NO_PROXY. The overlapping proxy PR at https://github.com/openclaw/openclaw/pull/41943 remains open/conflicting with review blockers, and the duplicate reports at https://github.com/openclaw/openclaw/issues/79701 and https://github.com/openclaw/openclaw/issues/79744 were already folded back into this cluster.

## What This Changes

The issue asks OpenClaw to fix Feishu bot identity and tenant-token REST probing when `proxy.enabled=true` is active.

## Best Possible Solution

Keep this as the canonical managed-proxy Feishu REST bug and implement a Feishu-owned Lark HTTP adapter that avoids Axios proxy double-handling while preserving the documented managed-proxy contract, timeout/User-Agent behavior, and focused proxy regression coverage.

## Reproduction Assessment

Yes, source-level high confidence: enable managed proxy mode with Feishu configured, then startup bot identity probing reaches `probeFeishu` -> `createFeishuClient` -> the Lark Axios-backed HTTP wrapper while `startProxy` has forced proxy env/global-agent state active. I did not run live Feishu credentials through the reported local proxy.

## Solution Assessment

No for the temporary direct-bypass workaround because it can route Feishu outside the operator-managed proxy guardrail. The best path is a target-aware Feishu HTTP adapter at the Lark `httpInstance` seam that prevents Axios/global-agent proxy conflicts without making an accidental egress-policy change.
