## Summary

Keep open. Current main still leaves Feishu probe/token/API REST calls on the Lark SDK Axios-backed HTTP instance with timeout/User-Agent wrapping only, while managed proxy mode injects process-wide proxy env and clears NO_PROXY; the linked duplicate reports were closed toward this issue rather than proving a shipped fix.

## What This Changes

The issue reports Feishu bot identity/probe REST calls failing when `proxy.enabled=true` because the Lark SDK HTTP path is not proxy-aware under OpenClaw managed proxy mode.

## Best Possible Solution

Add a Feishu-owned HTTP proxy policy at the Lark SDK `httpInstance` seam that preserves timeout/User-Agent behavior, uses the current shared proxy/dependency surface, respects the chosen managed-proxy or direct/NO_PROXY semantics, and adds focused client/probe tests plus docs.

## Reproduction Assessment

Yes, source-reproducible with high confidence. Current main combines managed proxy env/global-agent activation with Feishu probe/token REST calls through the Lark SDK Axios-backed HTTP instance without REST-specific proxy normalization; I did not run a live Feishu credential/proxy session.

## Solution Assessment

No for the local dist patch or unconditional direct-bypass workaround as the final solution. The maintainable fix should be target-aware and Feishu-owned, coordinate with existing proxy docs and helper contracts, and avoid silently changing operator egress policy.
