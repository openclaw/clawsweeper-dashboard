## Summary

Keep open: current main still matches the reported Feishu managed-proxy failure path, and related reports were already closed as duplicates of this canonical managed-proxy issue. The open proxy PR targets the same area but remains unmerged and previously reviewed as needing policy, dependency, and real-behavior-proof work.

## What This Changes

The issue reports Feishu bot identity and REST token/probe calls failing when `proxy.enabled=true` because the Lark SDK HTTP path conflicts with OpenClaw’s managed proxy environment.

## Best Possible Solution

Keep this as the canonical managed-proxy Feishu bug and fix the Feishu-owned Lark HTTP wrapper so REST/token/probe calls avoid Axios env-proxy double handling while preserving OpenClaw’s managed proxy contract, then add focused regression tests.

## Reproduction Assessment

Yes, at source level: enable managed proxy mode with Feishu configured, and current main combines process-wide proxy/global-agent state with Feishu REST/token/probe calls through the Lark Axios-backed default HTTP instance without REST proxy normalization. I did not run live Feishu credentials through a local proxy.

## Solution Assessment

No for the temporary direct-bypass workaround because it can weaken the documented managed proxy guardrail. The maintainable direction is an owner-local Feishu HTTP adapter or wrapper that preserves timeout/User-Agent behavior, handles the Lark/Axios proxy contract explicitly, and is covered by proxy regression tests.
