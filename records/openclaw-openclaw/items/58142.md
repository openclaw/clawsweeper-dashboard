## Summary

Keep open: this is a useful implementation candidate for per-skill model routing, but it still has merge-blocking correctness issues, an unsettled security/product contract, and insufficient after-fix runtime proof.

## What This Changes

The PR adds SKILL.md `model`/`modelProfile` metadata parsing, builds a skill-to-model map, wraps the read tool and stream function to switch models during a run, and adds router unit tests.

## Best Possible Solution

Settle the skill-routing contract first, then land or replace this branch with an implementation that preserves allowlists, uses current provider/model resolution, defines consent and precedence, updates docs, and includes focused tests plus redacted runtime proof.

## Reproduction Assessment

Yes for the review findings via source inspection. Current main uses the `@earendil-works` PI packages, threads `agentId` through skill loading, prepares provider transport from `params.model`, and compacts skill prompt paths to `~/...`, while the PR diverges from those contracts.

## Solution Assessment

No, not as currently proposed. The direction is plausible, but the patch needs a rebase plus maintainer-approved consent, precedence, fallback, and provider-routing semantics before it is the maintainable fix.

Full review comments:

- **[P1] Use the current PI package scope:** `src/agents/skills/model-router.ts:2-4`
  - body: Current main declares and imports `@earendil-works/pi-*`, but the new router and tests import `@mariozechner/pi-*`. Those packages are not the current dependency surface, so a rebased branch will fail to build until the imports use the current scope.
  - confidence: 0.95
- **[P1] Preserve agent-scoped skill loading:** `src/agents/pi-embedded-runner/skills-runtime.ts:59`
  - body: Current main passes `agentId` into `resolveEmbeddedRunSkillEntries()` and then into `loadWorkspaceSkillEntries()` so per-agent skill filters shape runtime entries. This branch's load path calls `loadWorkspaceSkillEntries(params.workspaceDir, { config })`, so a rebase would expose skill metadata outside the active agent's allowlist.
  - confidence: 0.9
- **[P2] Route through the selected provider transport:** `src/agents/skills/model-router.ts:206`
  - body: The router substitutes only the `Model` argument after run setup has already prepared the stream function, auth provider id, extra params, and provider transforms for `params.model`. Routing a skill to another provider can therefore use the wrong transport/auth path or fail despite logging an override.
  - confidence: 0.88
- **[P2] Expand compacted skill paths before lookup:** `src/agents/skills/model-router.ts:177`
  - body: Current main compacts home-directory skill locations to `~/...` in the skills prompt and the read tool expands those paths. This lookup uses `path.resolve(filePath)`, so a `~/.../SKILL.md` read resolves under the cwd instead of matching the absolute path stored in `skillModelMap`, causing home-directory skills to miss routing.
  - confidence: 0.86

## Security Review
