Overall confidence: 0.9

Full review comments:

- **[P1] Use the current PI package scope:** `src/agents/skills/model-router.ts:2-4`
  - body: Current main depends on `@earendil-works/pi-*`, but this new module imports the old `@mariozechner/pi-*` packages. Rebasing this branch as-is will leave unresolved imports in the router and tests, so update the new files to the current package scope.
  - confidence: 0.95
- **[P1] Preserve agent-scoped skill loading:** `src/agents/pi-embedded-runner/skills-runtime.ts:59`
  - body: Current main passes `agentId` through skill loading so `agents.defaults.skills` and `agents.list[].skills` are honored. This helper reloads workspace skills without `agentId`, which can expose or hide the wrong skills for an agent and also builds the routing map from that unscoped set.
  - confidence: 0.91
- **[P1] Route the override before provider setup:** `src/agents/pi-embedded-runner/run/attempt.ts:930-934`
  - body: The stream function and auth provider are registered for `params.model` before this wrapper swaps in `ctx.activeModel`. A skill that routes from one provider to another can therefore run through the original provider stream/auth path instead of the selected model's provider, which breaks the multi-provider use case this PR is meant to support.
  - confidence: 0.9
- **[P2] Expand home-relative skill paths before lookup:** `src/agents/skills/model-router.ts:177-178`
  - body: Skill prompt locations can be compacted to `~/.../SKILL.md`, but `path.resolve('~/...')` treats `~` as a normal relative segment. Managed or personal home-dir skills with model metadata can miss the map and silently fail to route.
  - confidence: 0.85
- **[P2] Bound the override to a real skill scope:** `src/agents/skills/model-router.ts:184-189`
  - body: The router deliberately keeps `activeModel` across non-SKILL reads, so after a model-routed skill is read every later LLM call in the attempt remains on that model until another SKILL.md is read. That makes unrelated follow-up work inherit a skill's model choice instead of returning to the session default.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces a security-sensitive provider/model routing surface from skill metadata without a settled allowlist, consent, auth-profile, or data-egress boundary.

Concerns:

- **[medium] Skill metadata can silently change provider boundaries:** `src/agents/skills/model-router.ts:103`
  - body: A workspace or installed skill can request any model present in the registry, and the code silently applies that model after the skill is read. That needs an explicit maintainer policy for trusted skills, allowed providers, user overrides, and how provider auth/data egress is constrained.
  - confidence: 0.86
- **[medium] Routing happens after auth provider binding:** `src/agents/pi-embedded-runner/run/attempt.ts:930`
  - body: The runtime auth wrapper resolves credentials for the original session provider, while the PR later substitutes a different model object. This can fail closed, misroute, or blur provider/auth assumptions for cross-provider routing.
  - confidence: 0.83
