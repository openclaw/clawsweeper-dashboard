## Summary

Keep open. The feature is useful and still tracked by open related issues, but this PR is not mergeable as-is because it imports the obsolete PI package scope, diverges from current agent-scoped skill loading, changes model selection without provider/auth-safe routing or consent policy, and still lacks sufficient after-fix runtime proof.

## What This Changes

The PR adds SKILL.md `model`/`modelProfile` metadata parsing, builds a skill model map, wraps the read tool and stream function to route later LLM calls, reconstructs cached skill entries, and adds router unit tests.

## Best Possible Solution

Settle the per-skill routing contract first, then rebase or replace this PR with a provider-aware, allowlist-safe implementation that includes consent/precedence/fallback docs, focused tests, and redacted runtime proof.

## Reproduction Assessment

Yes, for the review findings via source inspection. Current main contracts and the PR diff show the build-scope, allowlist, provider-routing, and path-lookup problems; this is otherwise a feature request rather than a failing current-main bug.

## Solution Assessment

No, not as currently proposed. The direction is plausible, but the patch swaps a late `Model` argument instead of using early provider-aware model selection, and it lacks maintainer-approved consent, precedence, fallback, and allowlist semantics.

  - body: The read interceptor runs before the underlying read tool normalizes aliases and home paths, but it ignores `file_path` and resolves `~/...` relative to cwd. SKILL.md reads using current prompt/read conventions can miss the map and never activate the override.
  - confidence: 0.86
