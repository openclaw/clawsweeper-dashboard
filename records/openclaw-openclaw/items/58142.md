- **[P1] Use the current PI package scope:** `src/agents/skills/model-router.ts:2-4`
  - body: Current main no longer depends on `@mariozechner/pi-*`; package.json and surrounding agent code use `@earendil-works/pi-*`. This added module imports the old package scope, so after rebasing the branch it will fail to typecheck unless these imports are updated with the rest of the current PI surface.
  - confidence: 0.97
- **[P1] Resolve provider routing before wrapping the stream:** `src/agents/skills/model-router.ts:210`
  - body: This wrapper only swaps the `model` argument after `attempt.ts` has already registered the provider stream, auth profile, API key lookup, transport, and extra params for the original session model. If a skill selects a different provider, the call can still use the original provider stream/credentials instead of the skill model's provider boundary; route through the runtime model selection path before building the stream instead.
  - confidence: 0.94
- **[P1] Preserve agent-scoped skill loading:** `src/agents/pi-embedded-runner/skills-runtime.ts:57-60`
  - body: Current main passes `agentId` into skill entry resolution so per-agent skill filters and limits are honored. This rewritten resolver has no `agentId` parameter and calls `loadWorkspaceSkillEntries` with only `{ config }`, so non-default agents can build routing metadata from the wrong skill set.
  - confidence: 0.9
- **[P2] Expand compact home paths before lookup:** `src/agents/skills/model-router.ts:177-178`
  - body: Available-skill prompts compact home paths to `~/.../SKILL.md`, and the read tool expands `~` before reading. The interceptor checks `path.resolve(filePath)` directly, so reads of those prompted home-relative skill paths miss `skillModelMap` and never activate the override for common installed skills.
  - confidence: 0.86
- **[P2] Bound the override to an explicit skill scope:** `src/agents/skills/model-router.ts:184-189`
  - body: Clearing only when another unmatched `SKILL.md` is read means the override can remain active for later unrelated LLM calls in the same attempt if no later skill file is opened. The feature needs an explicit turn/tool/scope boundary that restores the session model after the intended skill work, with tests for the final answer and follow-up tool calls.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: Skill frontmatter can silently reroute LLM traffic across configured providers without an explicit trust, allowlist, or consent contract.

Concerns:

- **[medium] Untrusted skill metadata can change provider boundary:** `src/agents/skills/model-router.ts:103`
  - body: The PR allows `SKILL.md` metadata to select any available registry model and silently falls back otherwise; without a user/agent policy, precedence rule, or audit surface, third-party skills can move run context to another configured provider than the user selected.
  - confidence: 0.88
- **[medium] Cross-provider overrides reuse original stream auth:** `src/agents/skills/model-router.ts:210`
  - body: Because routing happens by substituting the model argument after provider/auth stream setup, a skill-selected provider can be called through credentials and transport prepared for the original provider, which is an auth-provider boundary bug.
  - confidence: 0.86
