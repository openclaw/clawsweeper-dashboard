## Summary

Keep open. The feature is not on current main and the PR is still a plausible implementation candidate, but it is not mergeable as-is because it regresses agent skill allowlist filtering, misses home-relative skill paths, and needs maintainer agreement on the model-routing security contract.

## What This Changes

The PR adds SKILL.md `model`/`modelProfile` metadata, a read-tool and StreamFn based skill model router, cached-snapshot SkillEntry reconstruction, and router tests.

## Best Possible Solution

Define the public skill-routing contract first, then land a rebased implementation that preserves agent skill allowlists, mirrors existing path/model-selection helpers, documents consent and precedence, and covers parser, cached-snapshot, routing, fallback, and security cases.

## Reproduction Assessment

Yes, for the review findings via source inspection. Current main threads `sessionAgentId` through skill loading and compacts home skill paths, while PR head drops the agent ID and compares raw `~/...` reads with `path.resolve()`.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the PR needs allowlist preservation, home-path normalization, and a maintainer-approved security/precedence contract before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve agent skill allowlists when loading entries:** `src/agents/pi-embedded-runner/skills-runtime.ts:59`
  - body: The PR removes `agentId` from `resolveEmbeddedRunSkillEntries()` and then calls `loadWorkspaceSkillEntries` with only `{ config }`. Current main passes `sessionAgentId` so per-agent skill allowlists filter prompt-visible and runtime skills; merging this would let an agent see and route against skills its config did not allow.
  - confidence: 0.91
- **[P2] Expand home-relative skill paths before lookup:** `src/agents/skills/model-router.ts:177-178`
  - body: Available-skills prompts compact home paths to `~/...`, and the read tool expands that form before reading. This lookup uses `path.resolve(filePath)` directly, so a personal skill read as `~/.agents/skills/.../SKILL.md` will not match the absolute map key and its model override will never activate.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces metadata-driven provider/model switching for untrusted skill files and currently bypasses agent skill allowlist isolation.

Concerns:

- **[medium] Define consent for skill-declared model routing:** `src/agents/skills/model-router.ts:210`
  - body: `wrapStreamFnSkillModelRouter` substitutes a model selected from SKILL.md metadata for later LLM calls. Because third-party skills are documented as untrusted, this needs explicit precedence, consent, and audit behavior so a skill cannot silently move prompt content to a different configured provider or a more expensive model.
  - confidence: 0.82

  - body: Dropping the `agentId` pass-through removes the filter that keeps per-agent skill allowlists scoped. That can expose disallowed skills to prompts and to the new model-routing map.
