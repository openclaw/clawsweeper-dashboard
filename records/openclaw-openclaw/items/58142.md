## Summary

Keep open: this PR targets a real feature gap and includes Docker log proof for the explicit model path, but the current implementation lets SKILL.md metadata silently change the provider/model without an operator consent, allowlist, or precedence contract.

## What This Changes

This PR adds optional `model` and `modelProfile` SKILL.md metadata, builds a skill-to-model map, wraps the read tool and stream function to route later LLM calls, and adds model-router unit tests.

## Best Possible Solution

Settle the public skill-routing contract, then land or replace this PR with an implementation that requires explicit trust/allowlisting, documents precedence and fallback behavior, preserves current skill allowlists, and covers parser, cached-snapshot, routing, and security tests.

## Reproduction Assessment

Not applicable as a bug reproduction. The feature gap is source-auditable on current main, and the PR body provides Docker log proof for the new explicit-model behavior.

## Solution Assessment

No, not yet. The implementation direction is plausible, but silent SKILL.md-driven model switching is not the best merge shape until maintainers define consent, allowlist, fallback, and precedence semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Require consent before applying skill model overrides:** `src/agents/skills/model-router.ts:179-180`
  - body: The wrapper trusts any resolved `metadata.openclaw.model` from SKILL.md and stores it as `ctx.activeModel`, so installing or reading an untrusted skill can silently route subsequent model calls to another configured provider/model. Gate this behind an operator-controlled allowlist or explicit opt-in, and make the precedence/fallback behavior visible before applying the override.
