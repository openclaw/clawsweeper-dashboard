## Summary

Keep open: the report has credible screenshot evidence and current source shows built-in memory search can service channel sessions without the QMD-style chat-scope guard, while the Discord and memory docs are internally inconsistent about whether guild memory access is intended.

## What This Changes

The issue asks OpenClaw to prevent MEMORY.md-backed recall from being available in Discord guild channels by default, matching the documented DM-only memory expectation.

## Best Possible Solution

Set one explicit product/security contract for MEMORY.md in shared channels, then make built-in memory search/get, QMD scope, Discord docs, and regression coverage enforce that contract consistently.

## Reproduction Assessment

Yes, source inspection gives a high-confidence targeted path: in a Discord channel session with the built-in memory backend, memory_search is available and MemoryIndexManager.search has no chat-type scope guard before querying indexed memory. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the intended security contract. A safe fix should align built-in and QMD memory behavior with Discord docs, while preserving an explicit opt-in path for users who want guild memory recall.
