## Summary

Keep open: current main still injects `MEMORY.md` into ordinary Discord channel sessions and exposes built-in memory tools there, while user-facing docs say long-term memory should only auto-load in private sessions; the fix needs an explicit security/product contract before changing default shared-channel behavior.

## What This Changes

The issue reports that `MEMORY.md` private long-term memory is available in Discord guild channels despite docs saying guild channels do not auto-load it.

## Best Possible Solution

Define one shared-channel memory contract, then enforce it consistently across bootstrap injection, built-in memory search/get, QMD scope, Discord docs, and regression coverage with an explicit opt-in path if guild recall remains supported.

## Reproduction Assessment

Yes, at source level. A Discord channel session key is neither subagent nor cron, so `filterBootstrapFilesForSession` keeps `MEMORY.md`, and the built-in memory tools do not apply a channel/chat-type deny before returning memory-file content.

## Solution Assessment

Unclear until maintainers choose the intended security contract. The safest direction is to make shared-channel memory private by default and provide an explicit, documented opt-in if channel memory recall remains a supported workflow.
