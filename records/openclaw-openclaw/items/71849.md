## Summary

Keep open. Current main still lacks the requested Voice Call realtime fast-context path: remembered-context turns still go through the full `openclaw_agent_consult` handler, non-final transcript chunks are ignored before call-state processing, and the strict config, manifest, and public docs expose no `realtime.fastContext` surface. The maintainer discussion explicitly says this remains a real gap rather than a duplicate of the existing realtime consult work.

## What This Changes

Add an opt-in fast memory/session-context lookup path for Voice Call realtime calls, with bounded latency, partial transcript handling, graceful misses, and full consult fallback.

## Best Possible Solution

Implement a default-off, plugin-owned `realtime.fastContext` feature in Voice Call that accumulates and normalizes partial user transcript chunks, detects context-seeking turns, performs a hard-deadline memory/session search through public memory-host seams, injects snippets or an explicit miss into the realtime session, resets for follow-up questions, and preserves full `openclaw_agent_consult` as fallback.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main has no fast-context config/docs/manifest surface, registers only the full consult handler for realtime remembered-context turns, and discards non-final transcript chunks before call-state processing.

## Solution Assessment

Yes. The narrow maintainable direction is an opt-in Voice Call plugin path using existing memory-host search surfaces; moving this into core or replacing the full consult tool would be broader than the issue requires.

## Repair Work Prompt

Implement the Voice Call realtime fast-context path requested by openclaw/openclaw#71849. Keep the feature plugin-owned and default-off. Add a `realtime.fastContext` config block to the Voice Call plugin config and manifest, with fields like `enabled`, `timeoutMs`, `sources`, `maxResults`, and `fallbackToConsult`. In the realtime call handler/runtime, accumulate and normalize partial user transcript chunks enough to detect common remembered-context questions, perform a bounded memory/session search through public memory-host search seams, inject relevant snippets or an explicit empty/miss result into the realtime provider, reset the gate for multiple questions in one call, and keep `openclaw_agent_consult` as fallback for deeper work. Do not move owner-specific behavior into core and do not replace the existing consult tool. Coordinate with the adjacent open PR #72237 if it has landed or conflicts, but do not treat consult fast-mode overrides as this feature. Add docs for https://docs.openclaw.ai/plugins/voice-call explaining fast context vs full consult latency, and add a user-facing CHANGELOG.md entry crediting Thanks @amzzzzzzz.
