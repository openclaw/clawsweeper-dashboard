## What This Changes

The branch preserves Codex tool payload evidence in context-engine projections and records successful Telegram `sendMessageTelegram` sends into the Telegram prompt-context cache, with regression tests.

## Best Possible Solution

Move Telegram context recording to the unified visible-final-delivery boundary, cover preview-finalized and durable-send paths, and split or explicitly approve the Codex tool-result prompt-boundary expansion.

## Reproduction Assessment

Yes at source level: current main has separate `sendMessageTelegram`, durable-send, and preview-finalized streaming paths, and the PR only records the first helper path. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No: recording inside `sendMessageTelegram` is too narrow for the reported assistant-reply context gap because streamed finals can be finalized by editing/retaining the preview message. The maintainable fix is to record visible final Telegram deliveries at the delivery boundary and separately decide the Codex prompt-boundary expansion.
