## What This Changes

This PR makes the Telegram prompt-context cache async and plugin-state backed, records outbound `sendMessageTelegram` sends and preview-finalized streamed finals, and adds Telegram cache/send/dispatch tests plus a changelog entry.

## Best Possible Solution

Land the Telegram-owned cache recording after live Telegram proof demonstrates a later follow-up using a prior assistant reply and maintainers accept the plugin-state migration and retention behavior.

## Reproduction Assessment

Yes at source level: current main builds Telegram conversation context from the observed message cache, while direct outbound sends and preview-finalized streamed finals do not populate that cache. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Mostly yes: recording at the Telegram send and draft-stream finalization boundaries is the right ownership boundary. Merge should still wait for live Telegram proof and maintainer acceptance of the plugin-state cache migration semantics.
