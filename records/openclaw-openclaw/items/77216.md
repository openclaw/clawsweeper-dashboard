## Summary

Keep open: current main does not implement AI-generated session titles, and this PR has blocking model-boundary, live-update, and generated-schema defects. The default-on background model call also needs maintainer product/privacy review before landing.

## What This Changes

The PR adds default-on AI session-title generation, a new `sessionTitle` config block, `autoTitle` session persistence, derived-title UI display, tests, and a changelog entry.

## Best Possible Solution

If maintainers accept the feature, implement it through the active session model/runtime, emit and merge live title updates, regenerate public config artifacts, and cover the gateway/UI paths with focused tests.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the new title job resolves the agent default model, persists without a live change event, and leaves `derivedTitle` out of live session projection/merge. A runtime reproduction is not needed to identify these blockers.

## Solution Assessment

No, this is not the best implementation as-is. The safer path is to preserve the active session model boundary, make title updates observable to connected clients, and refresh generated config artifacts before maintainer product review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the active session model for title generation:** `src/auto-reply/reply/session-title-generator.ts:91-98`
  - body: This background job calls `generateConversationLabel`, whose current implementation resolves the agent default model. Since the call does not pass the active `provider`/`model` from the reply run, a session using a local/private or alternate model can have its first user messages summarized by a different configured provider.
  - confidence: 0.9
- **[P2] Broadcast the generated title after persisting it:** `src/auto-reply/reply/session-title-generator.ts:104-109`
  - body: The title is written after reply transcript events have already been emitted, and `updateSessionStoreEntry` only persists the store. Connected Control UI clients will keep showing the old title until a full sessions reload unless this path emits a session change or otherwise refreshes subscribers.
  - confidence: 0.86
- **[P2] Propagate derived titles through live session events:** `ui/src/ui/chat/session-controls.ts:444-452`
  - body: Using `row.derivedTitle` in the display name only helps rows loaded through `sessions.list`. The live `session.message`/`sessions.changed` snapshot and Control UI event merge omit `derivedTitle`, so sidebar rows can remain stale after a title is generated.
  - confidence: 0.84
- **[P2] Regenerate the config schema artifacts:** `src/config/zod-schema.ts:617-624`
  - body: Adding the top-level `sessionTitle` schema changes the public config surface, but the PR does not update `src/config/schema.base.generated.ts` or `docs/.generated/config-baseline.sha256`. The schema/docs checks will report drift and runtime schema responses will not include the new block.
  - confidence: 0.92
