## Summary

Keep open. This PR is a plausible implementation direction and improves on the related raw-provider-fetch approach by reusing the existing conversation-label path, but it still lacks contributor real behavior proof and currently persists the generated title without emitting an immediate session update, so the sidebar can stay stale until another refresh or message.

## What This Changes

The PR adds async AI-generated session-title generation after successful replies, persists an `autoTitle` session field/config surface, and wires gateway/UI session rows to display the derived title.

## Best Possible Solution

Consolidate on one implementation that uses OpenClaw's existing simple-completion/runtime-auth path, persists generated titles with clear precedence, emits an immediate session update after persistence, documents the config, and proves generation/persistence/display in a real session.

## Reproduction Assessment

Not applicable as a current-main bug; this is a feature PR for https://github.com/openclaw/openclaw/issues/77165. The PR blocker is source-reproducible from the async `autoTitle` persistence path and the existing event/update contract.

## Solution Assessment

No. Reusing the existing conversation-label path is the safer direction, but the PR should emit or trigger a session update after `autoTitle` persistence and provide real runtime proof before merge.
