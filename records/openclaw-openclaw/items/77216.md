## Summary

Keep open: this is a feature PR, not a current-main bug fix, and the proposed implementation still has a source-reproducible label/derived-title correctness bug plus missing real behavior proof.

## What This Changes

The PR adds asynchronous AI-generated session titles, `sessionTitle` config, `autoTitle` session metadata, Gateway/UI derived-title propagation, generated config hash updates, session title derivation tests, and a changelog entry.

## Best Possible Solution

Land a product-approved implementation that stores generated titles as derived `autoTitle` metadata, preserves manual `label`/`displayName`/`subject` values, documents config/privacy semantics, and includes redacted runtime proof.

## Reproduction Assessment

Not applicable as a current-main bug because this is a feature PR. The blocking PR defect is source-reproducible by following the generated-title lifecycle event from `session-title-generator.ts` into `server-session-events.ts` and then the Control UI label-priority path.

## Solution Assessment

No. The direction is plausible, but the patch should publish generated values through the derived-title path, avoid generation for manually named sessions, and get product/privacy approval plus real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Publish generated titles as derived titles:** `src/auto-reply/reply/session-title-generator.ts:162`
  - body: `maybeGenerateSessionTitle` persists `autoTitle`, but the live update emits the generated text as `label`. The lifecycle snapshot then forwards that label, and the Control UI resolves labels before `displayName` or `derivedTitle`, so a generated title can replace a user-named session in the live sidebar. Emit a derived-title refresh without `label` and skip sessions with manual `label`, `displayName`, or `subject`.
  - confidence: 0.88
