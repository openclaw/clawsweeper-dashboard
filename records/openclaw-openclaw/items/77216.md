## What This Changes

The PR adds asynchronous AI session-title generation after replies, `sessionTitle` config, `autoTitle` session metadata, Gateway/UI derived-title plumbing, session-utils coverage, generated config hashes, and a changelog entry.

## Best Possible Solution

Choose one product-approved implementation that uses session-scoped model/auth runtime, stores generated titles only as derived metadata, preserves manual labels/display names/subjects, documents the config/privacy semantics, and includes redacted runtime proof.

## Solution Assessment

No. The direction is plausible, but this patch should not merge until it threads session auth profile through title completion, emits derived-title updates without overriding manual labels, and gets product/privacy approval plus runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve session auth profiles for title generation:** `src/auto-reply/reply/get-reply-run.ts:1118-1127`
  - body: The reply path resolves a session `authProfileId`, but the new background title generator only receives `provider` and `model`. For sessions using an auth-profile override, transcript excerpts can be summarized with the default provider credentials instead of the session's selected account, or fail even though the reply worked; thread the session auth profile through the title completion path.
  - confidence: 0.9
- **[P2] Publish generated titles as derived titles:** `src/auto-reply/reply/session-title-generator.ts:162`
  - body: `maybeGenerateSessionTitle` persists `autoTitle` but broadcasts the generated text as `label`. Session lifecycle snapshots and the Control UI prefer `label` before `displayName` or `derivedTitle`, so a generated title can replace a manually named session in the live sidebar; refresh the derived title instead and avoid sessions with manual names.
  - confidence: 0.88
