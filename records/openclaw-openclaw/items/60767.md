## Summary

Keep open. Current main has useful partial loop and compaction safeguards, but it still does not implement the proposal’s always-on deterministic validation-error breaker, repair planner, verifier-gated retry, durable per-turn repair state, or full provider/gateway/session self-heal flow.

## What This Changes

The issue proposes a self-healing execution loop that classifies tool failures, repairs deterministic errors, preserves turn failure state, verifies fixes, and prevents duplicate retries or messages.

## Best Possible Solution

Keep this as a core runtime design item and split it into scoped follow-ups for deterministic validation-error breaking, durable per-turn failure state, verifier-gated repair hooks, provider/gateway/session repair actions, and channel duplicate-send suppression.

## Reproduction Assessment

Yes for the central symptom, but not for the whole proposal as one runnable repro. The related open Feishu/Kimi report gives a concrete missing-command exec loop, and current-main source shows the default rolling detector is still off while enabled generic repeats remain warn-first.

## Solution Assessment

Unclear as one broad subsystem. The maintainable path is staged through existing loop detection, tool validation, compaction, model fallback/doctor, and channel delivery seams rather than one large self-healing change.
