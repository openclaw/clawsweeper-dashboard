## Summary

Keep open. Current main has partial safeguards for repeated tool calls, unknown-tool loops, malformed streamed arguments, compaction pressure, and narrow duplicate reply suppression, but it still does not implement the proposed deterministic validation-error circuit breaker, repair planner, verifier-gated retry, durable per-turn repair state, or general outbound message dedupe; the related concrete symptom in #55694 also remains open.

## What This Changes

Request a core self-healing execution mechanism that classifies tool/runtime errors, plans and verifies repairs, preserves per-turn failure state, limits deterministic retries, and deduplicates repeated messages.

## Best Possible Solution

Keep this as a core runtime design item and split it into scoped follow-ups for deterministic validation-error breaking, durable per-turn failure state, verifier-gated repair hooks, provider/gateway/session repair actions, and channel duplicate-send suppression.

## Reproduction Assessment

Yes for the central symptom, but not for the entire proposal as one repro. Related #55694 gives a concrete Feishu/Kimi missing-`command` `exec` loop, and current-main inspection shows the deterministic always-on breaker is still absent.

## Solution Assessment

Unclear as a single broad subsystem. The maintainable path is staged through existing loop detection, before-tool hooks, tool validation, compaction, provider fallback, and delivery seams rather than one large self-healing change.
