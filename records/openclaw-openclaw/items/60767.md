## Summary

Keep open. Current main has useful partial safeguards for repetitive tool calls, malformed tool-call argument streams, idempotent chat sends, and mid-turn compaction pressure, but it still does not implement the proposal's always-on deterministic validation-error breaker, repair planner, verifier-gated retry, durable per-turn repair state, or provider/gateway/session self-heal flow.

## What This Changes

The issue proposes a broad self-healing execution system that classifies tool failures, plans repairs, preserves per-turn failure state, verifies repairs, and prevents deterministic retry loops.

## Best Possible Solution

Keep this as a core runtime design item and split it into scoped follow-ups for deterministic validation-error breaking, durable per-turn failure state, verifier-gated repair hooks, provider/gateway/session repair actions, and channel duplicate-send suppression.

## Reproduction Assessment

Yes for the central symptom, but not for the entire proposal as one repro. Related #55694 gives a concrete Feishu/Kimi missing-`command` exec loop, and current-main inspection shows the always-on deterministic breaker and repair planner are still absent.

## Solution Assessment

Unclear as one broad subsystem. The maintainable path is staged through existing loop detection, tool validation, compaction, model fallback/doctor, and channel delivery seams rather than one large self-healing change.
