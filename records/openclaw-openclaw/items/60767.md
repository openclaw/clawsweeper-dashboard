## Summary

Keep open. Current main has useful partial safeguards for malformed tool-call arguments, unknown-tool loops, opt-in tool-loop detection, and narrow compaction/inbound dedupe cases, but it does not yet implement the proposal’s deterministic validation-error breaker, durable per-turn repair state, repair planner, verifier-gated retry, provider/gateway/session repair flow, or general duplicate outbound message suppression.

## What This Changes

The issue proposes a core self-healing execution loop that classifies tool failures, stores per-turn failure state, runs bounded repairs, verifies them, and suppresses duplicate retries/messages.

## Best Possible Solution

Keep this as a core runtime design item and split it into scoped follow-ups for deterministic validation-error breaking, durable per-turn failure state, verifier-gated repair hooks, provider/gateway/session repair actions, and outbound duplicate-send suppression.

## Reproduction Assessment

Yes for the central symptom, but not for the entire proposal as one repro. Related #55694 gives a concrete Feishu/Kimi missing-command exec loop, and current-main source inspection shows the deterministic always-on validation-error breaker is still absent.

## Solution Assessment

Unclear as one broad subsystem. The maintainable path is staged work through the existing loop-detection, before-tool, validation, compaction, provider fallback, gateway recovery, and delivery seams rather than one large self-healing rewrite.
