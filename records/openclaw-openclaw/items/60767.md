## Summary

Keep open. Current main has partial loop-safety, post-compaction, malformed-argument repair, and retry-budget pieces, but it still does not implement the proposed self-healing execution architecture or a single canonical tracker that fully supersedes it.

## What This Changes

The issue proposes a broad self-healing execution architecture for classifying tool failures, planning repairs, preserving run-state outside compaction, verifying repair actions, and stopping deterministic retry and duplicate-message loops.

## Best Possible Solution

Keep this as a maintainer-scoped runtime design tracker and land changes through focused follow-ups for validation-error terminality, duplicate visible-message suppression, replay-loop ownership, consecutive-error detection, compaction guards, and gateway recovery.

Unclear as one broad subsystem. The maintainable path is staged maintainer design through existing loop detection, validation-error handling, retry ownership, compaction guards, gateway recovery, provider fallback, and channel duplicate-send seams.
