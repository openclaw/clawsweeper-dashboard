## Summary

Keep open. Current main has partial safeguards for tool-call loops, broad retry exhaustion, malformed streamed arguments, and narrow duplicate delivery suppression, but it still does not implement the requested deterministic validation-error circuit breaker, repair planner, verifier-gated retry, durable per-turn repair state, or general duplicate outbound-message guard.

## What This Changes

Request a self-healing execution loop that classifies tool failures, repairs and verifies before retrying, caps repeated deterministic errors, preserves runtime state outside compaction, and deduplicates repeated tool/message attempts.

## Best Possible Solution

Keep this as a core runtime design item and split it into scoped follow-ups for deterministic schema/validation failure breaking, durable per-turn failure state, verifier-gated repair hooks, provider/gateway/session repair actions, and channel-level duplicate-message suppression.

## Reproduction Assessment

Yes for the central symptom, but not for the entire proposal as one repro. Related #55694 gives a concrete Feishu/Kimi path with repeated `exec` validation failures from a missing `command`, and current-main inspection shows the always-on deterministic breaker is still absent.

## Solution Assessment

Unclear as a single proposal. The maintainable path is staged through existing loop-detection, before-tool hooks, retry/failover, compaction, and delivery seams rather than one large self-healing subsystem.
