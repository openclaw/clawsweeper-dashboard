## Summary

Keep open. Current main has useful partial safeguards for tool-call loops, unknown tools, incomplete-turn retries, compaction summaries, and idempotent chat sends, but it still does not implement the proposal's always-on deterministic validation-error breaker, repair planner, verifier-gated retry, durable per-turn repair state, or full provider/gateway/session self-heal flow.

## What This Changes

Request a self-healing execution loop that classifies tool failures, repairs deterministic errors, preserves per-turn failure state, verifies repairs, and suppresses duplicate retries/messages.

## Best Possible Solution

Keep this as a core runtime design item and split it into scoped follow-ups for deterministic validation-error breaking, durable per-turn failure state, verifier-gated repair hooks, provider/gateway/session repair actions, and channel duplicate-send suppression.

## Reproduction Assessment

Yes for the central symptom, but not for the entire proposal as one repro. Related #55694 gives a concrete Feishu/Kimi missing-command exec loop, and current-main source inspection shows the deterministic always-on breaker is still absent.

## Solution Assessment

Unclear as one broad subsystem. The maintainable path is staged through existing loop detection, tool validation, compaction, model fallback/doctor, and channel delivery seams rather than one large self-healing change.
