## Summary

Keep open. Current main has partial safeguards for repeated tool calls, unknown-tool loops, malformed streamed tool arguments, context-pressure recovery, and narrow duplicate reply suppression, but it still does not implement the proposed deterministic validation-error circuit breaker, repair planner, verifier-gated retry, durable per-turn repair state, or general outbound message dedupe; the concrete related symptom in #55694 remains open.

## What This Changes

The issue proposes a self-healing execution loop that classifies tool errors, plans repairs, preserves per-turn failure state, verifies repairs, and blocks repeated deterministic failures or duplicate messages.

## Best Possible Solution

Keep this as a core runtime design item and split it into scoped follow-ups for deterministic validation-error breaking, durable per-turn failure state, verifier-gated repair hooks, provider/gateway/session repair actions, and channel duplicate-send suppression.

## Reproduction Assessment

Yes for the central symptom, but not for the entire proposal as one repro. Related #55694 provides a concrete Feishu/Kimi missing-`command` `exec` loop with repeated visible messages, and current-main inspection shows the deterministic always-on breaker is still absent.

## Solution Assessment

Unclear as one broad subsystem. The maintainable path is staged through existing loop detection, before-tool hooks, tool validation, compaction, provider fallback, and delivery seams rather than one large self-healing change.
