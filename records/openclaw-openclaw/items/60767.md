## Summary

Keep open. Current main has useful loop-safety pieces and several narrower related trackers, but it still does not implement the broader self-healing execution architecture or default deterministic validation-error and duplicate-send policy requested here.

## What This Changes

The issue proposes a self-healing execution architecture that classifies tool failures, avoids blind retries, preserves per-turn failure state, runs repair actions, verifies repairs, and suppresses duplicate sends.

## Best Possible Solution

Keep this as a broad runtime design tracker, but route implementation through scoped follow-ups such as https://github.com/openclaw/openclaw/issues/73781, https://github.com/openclaw/openclaw/issues/75923, https://github.com/openclaw/openclaw/issues/75468, and the concrete user-visible loop report at https://github.com/openclaw/openclaw/issues/55694.

## Reproduction Assessment

Yes for the central repeated tool-failure symptom at source/report level: related open reports describe real Feishu/Telegram loop behavior and current source shows only partial loop guards. No for the entire self-healing architecture as one runnable current-main reproduction path.

## Solution Assessment

Unclear as one broad subsystem. The safer path is staged design through existing loop detection, validation-error handling, retry ownership, compaction guards, gateway recovery, provider fallback, and channel duplicate-send seams rather than one large automatic repair change.
