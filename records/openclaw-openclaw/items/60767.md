## Summary

Keep open. Current main has useful partial safeguards for repeated tool calls, unknown-tool loops, no-progress outcomes, and narrow compaction duplicate cleanup, but it still does not implement the proposal's deterministic validation-error breaker, repair planner, verifier-gated retry, durable per-turn repair state, provider/gateway/session self-heal flow, or general outbound message dedupe.

## What This Changes

The issue proposes a self-healing execution loop that classifies tool failures, repairs deterministic errors, verifies repairs, preserves per-turn failure state, and blocks repeated failed calls or messages.

## Best Possible Solution

Keep this as a core runtime design item and split it into scoped follow-ups for deterministic validation-error breaking, durable per-turn failure state, verifier-gated repair hooks, provider/gateway/session repair actions, and channel duplicate-send suppression.

## Reproduction Assessment

Yes for the central symptom, but not for the entire proposal as one repro. Related #55694 gives a concrete Feishu/Kimi missing-command exec loop, and current-main source inspection shows the deterministic always-on breaker is still absent.

## Solution Assessment

Unclear as a single broad subsystem. The maintainable path is staged through existing loop detection, before-tool hooks, validation, compaction, provider fallback, gateway recovery, and delivery seams rather than one large self-healing change.
