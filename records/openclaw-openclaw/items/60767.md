## Summary

Keep open. Current main has meaningful partial safeguards for loop detection and post-compaction repeats, but it still does not implement the proposal’s full self-healing execution design: always-on deterministic validation-error breaking, repair planning, verifier-gated retry, durable per-turn repair state, provider/gateway/session repair actions, or duplicate visible-send suppression.

## What This Changes

The issue proposes a core self-healing execution system that classifies tool/gateway/session failures, repairs them, verifies recovery, and prevents repeated failed tool calls or duplicate messages.

## Best Possible Solution

Keep this as a core runtime design item and split it into scoped follow-ups for deterministic validation-failure breaking, durable per-turn failure state, verifier-gated repair hooks, provider/gateway/session repair actions, and channel duplicate-send suppression.

## Reproduction Assessment

Yes for the central repeated tool-failure symptom, but not for the whole self-healing architecture as one runnable repro. The open related report gives concrete Feishu/Telegram failure loops, and current source shows the remaining default/policy gaps.

## Solution Assessment

Unclear as one broad subsystem. The maintainable solution is staged through existing loop detection, tool validation, compaction, model fallback/doctor, gateway recovery, and channel delivery seams rather than one large self-healing change.
