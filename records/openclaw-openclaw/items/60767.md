## Summary

Keep open. Current main has several partial safeguards for repetitive tool calls and noisy delivery, but it still does not implement the requested self-healing execution loop: deterministic validation-error circuit breaking, repair planning, verifier-gated retry, persisted per-turn repair state, or general duplicate outbound-message suppression.

## What This Changes

The issue proposes a self-healing execution loop that classifies tool errors, repairs deterministic failures, persists per-turn failure state outside compaction, verifies repairs, limits retries, and suppresses duplicate messages.

## Best Possible Solution

Keep this open as a core runtime design item and split it into scoped follow-ups: a deterministic validation/schema failure circuit breaker, durable per-run repair state, verifier-gated repair hooks, provider/gateway/session repair actions that reuse existing doctor and failover paths, and channel-level duplicate outbound-message suppression aligned with current idempotency behavior.

## Reproduction Assessment

Partially. The proposal itself is architectural, but related #55694 gives a concrete symptom path involving repeated Feishu/Kimi `exec` validation failures and repeated messages; current-main inspection shows the default deterministic breaker and durable repair state are still missing.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable path is likely staged through existing runtime seams rather than one broad self-healing subsystem that bypasses current loop detection, failover, transcript repair, compaction, and channel delivery logic.
