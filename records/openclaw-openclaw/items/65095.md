## Summary

Keep this open. Current main still enforces strict Gateway protocol matching, the reference client still advertises only the current protocol version by default, the macOS app still rejects older gateway versions, and the public docs/tests describe rejection rather than a grace period, compatibility flag, or offline-safe version-skew policy. The related stable-release cadence issue #65081 was closed as broader release-channel work; it does not supersede this narrower compatibility request.

## What This Changes

The issue asks OpenClaw to relax strict app/gateway and client/server version compatibility so remote or offline users are not forced to update immediately after releases.

## Best Possible Solution

Define a maintainer-approved Gateway compatibility policy: supported app/gateway and client/server skew ranges, explicit protocol negotiation semantics, docs for remote/offline deployment, and regression tests for old/new handshakes. Any bypass should still reject genuinely incompatible wire contracts.

## Reproduction Assessment

Yes. The source and existing test path provide a high-confidence reproduction: send a connect request whose minProtocol/maxProtocol range excludes PROTOCOL_VERSION and the Gateway rejects it as a protocol mismatch; the macOS version check likewise marks older installed gateways incompatible.

## Solution Assessment

No, the suggested broad skipVersionCheck flag is not the best fix by itself. The maintainable path is explicit compatibility negotiation plus a documented skew policy, with only narrow break-glass behavior if maintainers approve the security tradeoff.
