## Summary

Keep this open. Current main and the latest release still enforce strict Gateway protocol matching and macOS app-to-gateway version compatibility, so the reported remote/offline version-skew problem remains unresolved and needs maintainer policy rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to avoid mandatory update lockouts by supporting a Gateway/client grace period, compatibility mode, stable channel, or offline deployment guidance for remote users.

## Best Possible Solution

Define a maintainer-approved Gateway compatibility policy covering app/gateway and client/server skew, implement explicit protocol negotiation for safe compatible versions, document remote/offline deployment expectations, and add regression tests that still reject genuinely incompatible wire contracts.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: send a Gateway connect request whose minProtocol/maxProtocol range excludes PROTOCOL_VERSION and the current handler rejects it as a protocol mismatch; the macOS check similarly rejects installed gateways older than the app-required version.

## Solution Assessment

No, the broad suggested skipVersionCheck flag is not the best standalone fix. The maintainable direction is explicit compatibility negotiation and a documented skew policy, with any bypass treated as a narrow maintainer-approved break-glass mode.
