## Summary

Keep this open. Current main and the latest stable release still require protocol v4 for ordinary clients, keep strict protocol-range rejection, and the macOS app still marks older gateway packages incompatible, so the remote/offline version-skew request remains an unresolved product and protocol policy item.

## What This Changes

The issue requests a grace period, compatibility mode, stable/offline guidance, or similar support so older local clients and gateways remain usable after new releases.

## Best Possible Solution

Define a supported app/client/gateway skew policy, implement negotiated compatibility only for safe protocol ranges, and document remote/offline expectations with any bypass treated as an explicit break-glass mode.

## Reproduction Assessment

Yes. A high-confidence source-level path is to connect to a current v4 Gateway as an ordinary non-probe client advertising only protocol v3; the handler rejects the range with protocol mismatch, and the macOS app also marks older gateway packages incompatible.

## Solution Assessment

No. A standalone skipVersionCheck flag is not the best default fix; the maintainable path is a maintainer-approved compatibility policy with protocol negotiation and carefully scoped offline guidance.
