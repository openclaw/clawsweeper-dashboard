## Summary

Keep open: current main and the latest release still log Gateway WebSocket timeout and pre-connect close outcomes without a last-completed handshake phase, and both linked implementation PRs are closed unmerged.

## What This Changes

The issue asks for Gateway WebSocket failure logs to include the last completed handshake phase so timeout and pre-connect close logs can distinguish network, auth, session attach, subscription, and ready-state stalls.

## Best Possible Solution

Add a small internal handshake phase tracker in the Gateway WebSocket connection/message-handler path and include it in timeout and pre-connect close logs, with focused tests for timeout, closed-before-connect, and at least one auth or registration failure phase.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: trigger a pre-connect timeout or close and the current `handshake timeout` / `closed before connect` logs contain only coarse cause/state fields, not a last-completed phase.

## Solution Assessment

Yes, the best direction is a narrow Gateway-internal diagnostic tracker with tests; it should not introduce config, protocol fields, or broader session/model changes.
