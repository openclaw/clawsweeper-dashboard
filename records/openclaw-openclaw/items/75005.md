## Summary

Keep open. Current main still rejects plugin-only local/no-auth media providers before audio or video handlers can run, and this PR is a focused implementation that preserves normal auth precedence; the remaining blocker is the required changelog entry for a user-facing media fix.

## What This Changes

This PR adds a media-provider `resolveSyntheticAuth` hook, uses it as a missing-key fallback for audio/video media execution, documents local no-auth media providers, and adds regression coverage.

## Best Possible Solution

Add the missing Unreleased/Fixes changelog line, keep the resolver-first provider-owned synthetic auth design, then land this PR or an equivalent patch with the existing focused regression coverage.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction path: configure a plugin-only media audio provider with no credentials and trigger transcription; the runner resolves provider auth and requires an API key before `transcribeAudio` can run.

## Solution Assessment

Yes. The PR's missing-key-only fallback is the narrowest maintainable fix because literal, env, stored-profile, and explicit-profile auth still take precedence, while unmarked providers remain fail-closed.
