## Summary

## What This Changes

The PR adds a `MediaUnderstandingProvider.resolveSyntheticAuth` hook, uses it as a missing-media-provider-key fallback for audio/video execution, documents local no-auth media providers, and adds regression tests.

## Best Possible Solution

Land the resolver-first synthetic auth hook with its regression coverage and docs, add the required Unreleased/Fixes changelog entry, and keep broader no-auth provider discovery or manifest semantics as separate follow-up work if maintainers want them.

## Reproduction Assessment

Yes. Current main has a high-confidence source-path reproduction: a plugin-only media audio provider without credentials reaches `resolveProviderExecutionAuth`, `resolveApiKeyForProvider`, and `requireApiKey` before `transcribeAudio` can run.

## Solution Assessment

Yes for the runtime approach. The PR is narrower than weakening media auth globally because normal literal, resolver, env/profile, and explicit profile failures remain ahead of the provider-owned synthetic fallback; the remaining gap is changelog coverage, not core behavior.
