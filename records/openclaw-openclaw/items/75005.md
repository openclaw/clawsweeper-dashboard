## Summary

Keep open. Current main still hard-requires model-provider auth before audio/video media provider execution, and this active PR addresses the linked no-auth local media provider bug with a narrow provider-owned fallback; I did not find a blocking correctness or security issue in the proposed patch.

## What This Changes

The PR adds an explicit `resolveSyntheticAuth` hook to media understanding providers, wires it into audio/video auth resolution after normal auth lookup fails, and adds docs plus local no-auth regression coverage.

## Best Possible Solution

Review and land the resolver-first, provider-owned synthetic-auth hook with its docs and regression tests; keep the linked issue open until this PR merges, and track any broader auto-selection semantics separately.

## Reproduction Assessment

Yes. Current main has a high-confidence code-path reproduction: a configured plugin-only audio provider with no credentials reaches `resolveProviderExecutionAuth`, then `resolveApiKeyForProvider` and `requireApiKey`, before `transcribeAudio` can run.

## Solution Assessment

Yes. The PR is the narrow maintainable fix for the reported execution path because it preserves normal auth precedence and explicit profile failures while requiring a media provider to opt into synthetic no-auth execution.
