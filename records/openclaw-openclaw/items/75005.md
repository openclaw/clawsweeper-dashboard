The PR adds a `MediaUnderstandingProvider.resolveSyntheticAuth` hook, uses it as a missing-provider-key fallback for audio/video media execution, documents the local no-auth provider pattern, and adds regression tests.

## Best Possible Solution

Yes. Current main has a high-confidence code-path reproduction: a configured plugin-only audio provider without auth reaches `resolveProviderExecutionAuth`, then `resolveApiKeyForProvider` and `requireApiKey`, before `transcribeAudio` can run.

## Solution Assessment

Yes for the configured execution path. The proposed hook is narrower than weakening core auth globally because normal literal, env, profile, and resolver auth still take precedence and only explicit provider-owned fallback can synthesize a non-secret key.
