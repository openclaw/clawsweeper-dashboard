## Summary

Keep open: current main still only partially exposes the existing auth cooldown behavior through config schema/docs and CLI output; it does not provide the requested dashboard/configure workflow for per-profile cooldown visibility, profile-count-aware rotation settings, or clearing cooldowns. The existing ClawSweeper comment reached the same conclusion, and a fresh check against current main did not find an implementation that would make this closeable.

## What This Changes

Request to surface `auth.cooldowns.rateLimitedProfileRotations` / `overloadedProfileRotations`, per-profile cooldown state, and a clear-cooldown action in `openclaw configure` and/or the dashboard settings UI.

## Best Possible Solution

Design a dedicated auth-profile status/control surface: extend `models.authStatus` with cooldown metadata, show rotation limits alongside actual profile counts, and add a narrowly authorized clear-cooldown RPC only if maintainers approve that control.

## Reproduction Assessment

Not applicable: this is a feature request for new UI exposure of existing runtime/config state, not a broken documented behavior. Source inspection confirms the knobs and cooldown state exist while the requested dashboard/configure controls remain absent.

## Solution Assessment

Unclear pending maintainer UX/API direction: extending the existing auth-status dashboard path looks like the right implementation seam, but the clear-cooldown action should be deliberately designed as a gateway mutation rather than hidden in generic config editing.
