## Summary

Keep open: current main has the underlying config, runtime behavior, docs, CLI cooldown visibility, generic schema-backed config editing, and a provider-level Model Auth card, but it still lacks the requested per-profile dashboard cooldown data, profile-count-aware rotation UI, and authorized clear-cooldown action. The related reports at https://github.com/openclaw/openclaw/issues/80040 and https://github.com/openclaw/openclaw/issues/80514 are adjacent auth/fallback bugs, not a canonical replacement for this UI/API request.

## What This Changes

Expose existing auth-profile rotation knobs and per-profile cooldown state/actions in the configure wizard or dashboard.

## Best Possible Solution

Add a deliberate auth-profile management surface that shows rotation limits with provider profile counts, includes per-profile cooldownUntil/cooldownReason in the dashboard status path, and provides an explicit authorized clear-cooldown mutation if approved.

## Reproduction Assessment

Not applicable: this is a feature request for new UI/API exposure of existing config and runtime state, not a broken documented behavior. Source inspection confirms the underlying knobs and cooldown state exist while the requested dashboard/configure surfaces remain incomplete.

## Solution Assessment

Unclear pending maintainer UX/API direction: generic schema editing partly exposes the numeric knobs, but the full request needs per-profile status data and a safe state-mutation path. The narrowest maintainable read path is likely extending models.authStatus; clear-cooldown should be an explicit gateway method rather than raw state-file editing.
