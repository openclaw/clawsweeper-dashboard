## Summary

Keep open: current main and v2026.6.6 still drop Slack room/group `/elevated` text commands before `tools.elevated.allowFrom.slack` can be evaluated, and the remaining fix touches a security-sensitive command authorization boundary that needs maintainer product/security judgment.

## What This Changes

The issue requests that Slack room/group elevated directives honor `tools.elevated.allowFrom.slack` or document and implement an equivalent fallback instead of being silently blocked by command authorization.

## Best Possible Solution

Adopt a maintainer-approved Slack elevated authorization model that makes `/elevated` directive access explicit before the early room/group drop, preserves the elevated security boundary, and covers Slack plus the related Discord fallback surface with focused tests and docs.

## Reproduction Assessment

Yes, at source level: a Slack room/group sender listed only in `tools.elevated.allowFrom.slack` can hit `commandAccess.shouldBlockControlCommand`, causing `prepare.ts` to return `null` before elevated authorization runs. I did not run a live Slack workspace in this read-only review.

## Solution Assessment

Unclear until maintainers choose the auth model. The best fix should be narrower than a blanket command-authorized bypass and should either run an elevated-directive-specific authorization before the early drop or add an extension-owned fallback that can affect the inbound gate safely.
