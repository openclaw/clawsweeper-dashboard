## Summary

Keep open: current main still drops Slack room/group text control commands before `tools.elevated.allowFrom.slack` can be evaluated, and no exact open or merged PR appears to cover this Slack-specific ingress gap.

## What This Changes

The issue reports that Slack room/group `/elevated` activation is silently blocked when only `tools.elevated.allowFrom.slack` is configured.

## Best Possible Solution

Implement an elevated-specific Slack authorization path that lets room/group `/elevated` reach `resolveElevatedPermissions` without granting other control commands, with regression coverage and docs clarifying Slack fallback or prerequisite behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Slack room/group message containing `/elevated` from a sender listed only in `tools.elevated.allowFrom.slack` reaches the Slack command gate, has no matching command authorizer, and returns `null` before elevated permissions run.

## Solution Assessment

Unclear until maintainers choose the authorization seam. The safer direction is an elevated-specific ingress or fallback path, not broadening the normal Slack command gate for every control command.
