## Summary

Keep open: current main and v2026.6.8 still drop Slack room/group `/elevated` text commands before `tools.elevated.allowFrom.slack` can be evaluated, and the remaining fix is security-sensitive because it changes elevated command authorization.

## What This Changes

The issue reports that Slack room/group `/elevated` messages from users listed only in `tools.elevated.allowFrom.slack` are silently blocked by the Slack text-command gate.

## Best Possible Solution

Adopt a maintainer-approved Slack elevated authorization model that evaluates `tools.elevated.allowFrom.slack` for `/elevated` without widening unrelated Slack control commands, with focused Slack and sibling fallback tests/docs.

## Reproduction Assessment

Yes, at source level. A Slack room/group `/elevated` message from a sender listed only in `tools.elevated.allowFrom.slack` reaches the Slack command gate and returns `null` before elevated authorization runs; I did not run a live Slack workspace.

## Solution Assessment

No landed fix is present. The best repair still needs maintainer/security choice between directive-scoped command-gate authorization, earlier elevated admission handling, or another explicit Slack security model.
