## Summary

Keep open: current main still drops Slack room/group text control commands before `tools.elevated.allowFrom.slack` can be checked, and the related Discord/tiered-auth items do not supersede this Slack-specific authorization path.

## What This Changes

Slack room/group `/elevated` messages from users listed only in `tools.elevated.allowFrom.slack` should reach elevated permission resolution instead of being dropped by Slack command authorization.

## Best Possible Solution

Implement an elevated-only Slack authorization path that lets `/elevated` reach `resolveElevatedPermissions`, with focused room/group regression tests and docs clarifying Slack fallback or prerequisites.

## Reproduction Assessment

Yes. A high-confidence source path is a Slack room/group message containing `/elevated on` from a sender listed only under `tools.elevated.allowFrom.slack`; current `prepareSlackMessage` returns `null` before `resolveElevatedPermissions` runs.

## Solution Assessment

Unclear until a maintainer chooses the authorization seam. The safer direction is elevated-specific handling, not treating the elevated allowlist as general Slack command authorization.
