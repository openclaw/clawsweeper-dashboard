## Summary

Keep open: current main and v2026.6.8 still block Slack room/group `/elevated` text commands before `tools.elevated.allowFrom.slack` can be evaluated, and no merged Slack-specific fix or canonical replacement owns the remaining security-sensitive authorization decision.

## What This Changes

The issue asks for Slack elevated allowlists to work for `/elevated` activation, or for an explicit documented Slack fallback instead of silent command-gate blocking.

## Best Possible Solution

Adopt a maintainer-approved Slack elevated authorization model that evaluates `tools.elevated.allowFrom.slack` for `/elevated` without widening unrelated Slack control commands, with focused Slack and sibling fallback tests/docs.

## Reproduction Assessment

Yes, at source level. A Slack room/group `/elevated` message from a sender listed only in `tools.elevated.allowFrom.slack` reaches `shouldBlockControlCommand` and returns `null` before elevated authorization runs; I did not run a live Slack workspace.

## Solution Assessment

No current fix is present. The best repair still needs maintainer/security choice between directive-scoped command-gate authorization, earlier elevated admission handling, or another explicit Slack security model.
