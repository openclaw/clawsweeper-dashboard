## Summary

Keep open: current main still drops Slack room/group `/elevated` text commands before the elevated allowlist can be checked, and the related Discord fallback work does not cover this Slack-specific ingress gate.

## What This Changes

Allow Slack group senders listed only in `tools.elevated.allowFrom.slack` to activate elevated mode, or document and implement the required Slack fallback/prerequisite.

## Best Possible Solution

Implement an elevated-only Slack authorization path that lets `/elevated` reach `resolveElevatedPermissions`, with regression coverage and docs clarifying the Slack room/group prerequisite or fallback behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Slack room/group message containing `/elevated on` from a sender listed only under `tools.elevated.allowFrom.slack` returns `null` in `prepareSlackMessage` before elevated permissions run.

## Solution Assessment

Unclear until maintainers choose the authorization seam. The safer approach is an elevated-specific ingress/fallback path, not broadening the normal Slack command gate to all elevated-allowlisted senders.
