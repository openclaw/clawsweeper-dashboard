## Summary

Keep open. Current main still drops Slack room/group `/elevated` text commands before `tools.elevated.allowFrom.slack` can be evaluated, and the related Discord/tiered-auth items do not supersede this Slack-specific authorization bug.

## What This Changes

The issue asks Slack room/group `/elevated` messages from senders listed only in `tools.elevated.allowFrom.slack` to reach elevated authorization or have a documented Slack fallback.

## Best Possible Solution

Fix Slack narrowly so `tools.elevated.allowFrom.slack` lets only `/elevated` reach elevated permission resolution, then cover the room/group case with a focused Slack regression test and public docs.

## Reproduction Assessment

Yes. A source-level reproduction is a Slack room/group message containing `/elevated on`, with the sender listed only under `tools.elevated.allowFrom.slack`; current `prepareSlackMessage` returns `null` before `resolveElevatedPermissions` runs.

## Solution Assessment

Unclear until a maintainer chooses the authorization seam. The safer direction is elevated-specific handling, not treating the elevated allowlist as general Slack command authorization.
