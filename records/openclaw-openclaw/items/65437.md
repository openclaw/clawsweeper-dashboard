## Summary

Keep open. Current main still has the Slack room/group early-drop path described in the issue, and martingarramon's trace in the discussion matches the current source. The related Discord fallback and tiered-command-auth items are useful context, but they do not supersede this Slack-specific elevated authorization bug.

## What This Changes

The issue asks for Slack `/elevated` activation to work, or for an implemented and documented fallback, when the sender is listed only in `tools.elevated.allowFrom.slack`.

## Best Possible Solution

Fix the Slack room/group `/elevated` path narrowly so senders listed in `tools.elevated.allowFrom.slack` can reach elevated permission resolution without gaining unrelated Slack command authority. Add a focused Slack regression test and update the public elevated or Slack docs to describe the resulting Slack behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Slack room/group text message containing `/elevated on`, with access groups enabled, no Slack account/channel command allowlist, and the sender listed only under `tools.elevated.allowFrom.slack`; current `prepareSlackMessage` returns `null` before elevated permissions are checked.

## Solution Assessment

Unclear until a maintainer chooses the authorization seam. The safest implementation direction is elevated-specific handling, either via the existing channel `elevated.allowFromFallback` seam or a pre-gate authorizer that applies only to `/elevated`, not a general command-gate widening.
