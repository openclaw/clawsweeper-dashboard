## Summary

Keep open. Current main has improved adjacent status behavior, but `memory status --deep` still runs the embedding readiness probe and renders a failed probe as generic `Embeddings: unavailable` beside populated index counts, so the reported diagnostic ambiguity remains.

## What This Changes

The issue asks for `memory status --deep` to distinguish persisted memory index state from live embedding-provider probe failures or flapping.

## Best Possible Solution

Make the memory diagnostics surface explicitly label persisted index counts, vector-store/semantic-vector state, and live or cached embedding-provider probe state as separate concepts, with focused CLI tests, docs, and changelog coverage.

## Reproduction Assessment

Yes for the diagnostic ambiguity: current source shows `memory status --deep` calls the embedding probe and renders it as generic `Embeddings: ready|unavailable` beside index counts. The exact network flapping remains reporter-observed rather than locally reproduced.

## Solution Assessment

Yes, the narrow maintainable repair belongs in memory-core CLI rendering/probe metadata and tests. PR #75106 may provide a useful base for timeout semantics, but the remaining provider-probe wording/state split should be handled there or in a focused follow-up.
