## Summary

Keep open. Current main still runs the `memory status --deep` embedding readiness probe and renders the live/cached provider result as generic `Embeddings: ready|unavailable` beside persisted index counts, so the diagnostic ambiguity remains; the related #75106 branch is an active timeout/hang candidate, not an implemented fix for this issue.

## What This Changes

The issue asks for `openclaw memory status --deep` to distinguish persisted memory index health from transient live embedding-provider probe readiness.

## Best Possible Solution

Update memory-core diagnostics so persisted index/source/vector/FTS/cache state and live or cached embedding-provider readiness are explicitly separate, with failed probes reported as provider/probe availability and covered by focused CLI tests plus docs/changelog updates.

## Reproduction Assessment

Yes for the diagnostic ambiguity: current source shows `memory status --deep` runs `probeEmbeddingAvailability()` and renders the result as generic embeddings availability beside index counts. The exact remote-network flapping remains reporter-observed rather than locally reproduced.

## Solution Assessment

Yes for the repair direction: the narrow maintainable solution is in memory-core CLI rendering and shared probe metadata only if needed. Provider fetch behavior, proxy routing, and the doctor default cached path should stay out of scope unless a small shared type is required.
