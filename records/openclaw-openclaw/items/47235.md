## Summary

Keep open. Current main still makes `openclaw memory status --deep` execute the embedding readiness probe and render that live or cached provider result as generic `Embeddings: ready|unavailable` beside persisted index counts, so the diagnostic ambiguity reported here remains. The adjacent doctor/status cache work for #71568 improves the gateway default path, but it does not change this CLI deep-status output.

## What This Changes

The issue asks for `openclaw memory status --deep` to distinguish persisted memory index health from transient live embedding-provider probe readiness.

## Best Possible Solution

Update the memory diagnostics surface so persisted index presence/counts, source counts, vector/FTS state, embedding-cache state, and live or cached embedding-provider readiness are explicitly separate. A failed probe should read as provider/probe availability rather than index corruption, with focused CLI regression coverage plus docs and changelog updates.

## Reproduction Assessment

Yes for the diagnostic ambiguity: current source shows `memory status --deep` runs `probeEmbeddingAvailability()` and renders the result as generic embeddings availability beside index counts. The exact remote-network flapping is unclear from public data alone, but that is not the blocker for the CLI diagnostics bug.

## Solution Assessment

Yes. The narrow maintainable fix is in the memory-core CLI rendering and, only if needed, shared probe metadata; provider fetch behavior and the doctor default cached path should stay out of scope unless a shared type is required.
