## Summary

Keep open. Current main and the latest release still run a live or cached embedding probe for `memory status --deep` and render its result as generic `Embeddings: ready|unavailable` next to persisted index counts, so the diagnostic ambiguity remains.

## What This Changes

The issue asks for `memory status --deep` to distinguish persisted memory index state from the live embedding readiness probe when the probe flaps or fails.

## Best Possible Solution

Update memory-core diagnostics so persisted index/source/vector/FTS/cache state and live or cached embedding-provider probe state are explicitly separate, with focused CLI tests, docs, and changelog coverage.

## Reproduction Assessment

Yes for the diagnostic ambiguity: current source shows `memory status --deep` runs `probeEmbeddingAvailability()` and renders the result as a generic embeddings state beside index counts. The exact remote-network flapping remains reporter-observed rather than locally reproduced.

## Solution Assessment

Yes for the repair direction: the narrow maintainable fix belongs in memory-core CLI rendering and probe metadata, with shared SDK type changes only if JSON needs a clearer state. Provider fetch routing and doctor default cached behavior should stay out of scope unless the existing open PR changes that surface first.
