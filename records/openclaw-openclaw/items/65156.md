## Summary

Keep open. Current main and v2026.5.12 still treat a successful sqlite-vec load call as availability without probing `vec_version()` or a temporary `vec0` table, so the reported silent load-success/no-functions failure remains plausible and source-reproducible.

## What This Changes

The issue asks OpenClaw to restore reliable memory vector search when sqlite-vec loads but registers no usable functions after a SQLite ABI mismatch.

## Best Possible Solution

Add a loader/readiness health probe before marking sqlite-vec available, preserve a clear load error when the probe fails, and keep the degraded FTS/in-process fallback path working.

## Reproduction Assessment

Yes, by source inspection: make `db.loadExtension(...)` or `sqliteVec.load(db)` succeed while `vec_version()` or a temporary `vec0` table probe fails, and current main still records the store as available before any usability check. The issue discussion adds multi-platform real-world confirmations, but I did not run a live affected-host repro.

## Solution Assessment

No, the suggested dependency pin or source rebuild is not the best first repository fix because it is broader dependency policy. The narrow maintainable fix is a post-load sqlite-vec health probe with focused regression coverage and clean degraded-state reporting.
