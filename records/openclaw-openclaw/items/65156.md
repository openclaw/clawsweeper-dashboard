## Summary

Keep open. Current main still treats a successful sqlite-vec load call as availability without proving `vec_version` or the `vec0` module were registered, so the reported load-success/no-functions degradation remains a valid Memory Core bug.

## What This Changes

The issue asks OpenClaw to restore sqlite-vec-backed memory vector indexing when the native extension loads but registers no SQL functions because of a SQLite ABI mismatch.

## Best Possible Solution

Memory Core should only mark sqlite-vec available after a loader-owned health probe verifies `vec_version` and a temporary `vec0` table, then preserve a clear load error and continue through the existing degraded fallback path when the probe fails.

## Reproduction Assessment

Yes. The issue has concrete Linux steps plus independent confirmations, and current main is source-reproducible by making the sqlite-vec load call succeed while `vec_version` or `CREATE VIRTUAL TABLE ... USING vec0` fails.

## Solution Assessment

No for the suggested dependency pin or rebuild as the first repository fix. The narrower maintainable fix is a loader-level sqlite-vec health probe with regression coverage and clear degraded-state reporting.

Keep the change narrow: update `packages/memory-host-sdk/src/host/sqlite-vec.ts` so `loadSqliteVecExtension` only returns `ok: true` after a post-load health probe proves sqlite-vec registered its SQL surface, preferably `SELECT vec_version()` and a temporary `CREATE VIRTUAL TABLE ... USING vec0`/drop probe using a small dimension. Add a user-facing changelog fix entry.
