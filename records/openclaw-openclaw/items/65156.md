## Summary

Keep open. Current main still lacks a sqlite-vec usability probe after native load, so the confirmed load-success/no-functions path can still be cached as vector-available before vec0 is exercised.

## What This Changes

The issue asks OpenClaw to restore sqlite-vec-backed memory vector indexing when sqlite-vec loads but does not register usable vector functions under SQLite 3.51.x.

## Best Possible Solution

Add a sqlite-vec post-load health probe that proves the registered functions and vec0 module work before marking the vector store available, then keep the existing degraded fallback with a clear loadError when the probe fails.

## Reproduction Assessment

Yes, at source level. Make loadSqliteVecExtension's native load call succeed while a vec_version or temporary vec0 table probe fails; current main has no such probe and marks the vector store available before vec0 is exercised.

## Solution Assessment

No for the suggested dependency pin or rebuild as the first repository fix. The narrowest maintainable repair is a loader-level sqlite-vec health check with regression coverage and degraded-state reporting.

## Repair Work Prompt

Fix the memory sqlite-vec silent-load health handling for this issue. Current main returns ok=true from packages/memory-host-sdk/src/host/sqlite-vec.ts after loadExtension/sqliteVec.load without proving sqlite-vec registered usable functions/modules, and extensions/memory-core/src/memory/manager-sync-ops.ts caches that as vector.available=true. Verify the sqlite-vec 0.1.9 exported health surface first, then add a post-load probe that proves the vector functions and a vec0 virtual table are usable before returning ok=true. If the probe fails, return ok=false with a clear error and make Memory Core keep the documented degraded fallback instead of marking the vector store available or throwing from later table creation. Add regression coverage for load success followed by failed vec function or vec0 probing, plus manager warning/status behavior as needed. Do not pin better-sqlite3, rebuild sqlite-vec, add a new config option, or change public memory configuration unless a maintainer explicitly asks. Add a user-facing CHANGELOG.md fix entry.
