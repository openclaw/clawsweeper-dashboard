## Summary

Keep open. Current main still treats a successful sqlite-vec load call as availability without probing registered SQL functions or the vec0 module, and the issue now has independent Linux, Windows, and macOS confirmations.

## What This Changes

The issue requests restoring memory vector search when sqlite-vec appears to load but registers no usable SQL functions after the SQLite/runtime mismatch.

## Best Possible Solution

Memory Core should only mark sqlite-vec store-ready after a post-load probe proves vec_version and a temporary vec0 table work, then preserve a clear loadError and continue through the existing degraded fallback path when that probe fails.

## Reproduction Assessment

Yes. The issue has concrete reproduction steps and multiple platform confirmations, and current main is source-reproducible by making loadExtension/sqliteVec.load succeed while vec_version or vec0 creation fails.

## Solution Assessment

No for the suggested dependency pin or rebuild as the first repository fix, because that is broader dependency management. The narrow maintainable fix is a loader-level sqlite-vec health probe with regression coverage and clearer degraded-state reporting.
