## Summary

Keep open. Current main still treats a successful sqlite-vec load call as vector readiness before proving that sqlite-vec registered `vec_version` or the `vec0` module, which is the same load-success/unusable-extension class reported here.

## What This Changes

The issue reports that Memory Core vector indexing degrades to FTS-only because sqlite-vec can load without registering usable vector functions/modules after a SQLite ABI mismatch.

## Best Possible Solution

Add a Memory Core/sqlite-vec loader health check that only marks vector search available after `vec_version()` and `vec0` table creation work, then keep the existing degraded fallback path with a clear load error when the probe fails.

## Reproduction Assessment

Yes. The live ABI mismatch was not executed in this read-only review, but the issue has concrete Linux confirmations and current source can be unit-reproduced by making the native load call succeed while a `vec_version()` or temporary `vec0` probe fails.

## Solution Assessment

No for the suggested dependency pin or sqlite-vec rebuild as the first repository fix. The narrower maintainable solution is a post-load sqlite-vec health probe with regression coverage and unchanged dependency policy.
