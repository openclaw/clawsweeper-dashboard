## Summary

Keep open. Current main still treats a successful sqlite-vec load call as sufficient, while the reported failure mode is that loading can succeed without usable vec functions or a vec0 module.

## What This Changes

The issue asks OpenClaw to restore memory vector indexing when sqlite-vec loads but registers no usable vector functions after a SQLite ABI mismatch.

## Best Possible Solution

Add a sqlite-vec health probe after loading, report load failure when vec functions or vec0 table creation are unusable, preserve degraded fallback behavior, and cover the silent-success case with regression tests plus a changelog entry.

## Reproduction Assessment

Yes. The issue has concrete multi-platform reports, and current main has a clear source reproduction: make db.loadExtension() or sqliteVec.load() succeed while vec_version or CREATE VIRTUAL TABLE USING vec0 fails, then the loader still reports ok=true.

## Solution Assessment

No for the suggested dependency pin or rebuild as the first repository fix. The narrower maintainable fix is a loader/store availability health probe that only marks sqlite-vec usable after the registered functions and vec0 module work.

## Repair Work Prompt

Fix openclaw/openclaw#65156 by making sqlite-vec availability truthful when the native load call succeeds but no usable vector functions/module are registered. In `packages/memory-host-sdk/src/host/sqlite-vec.ts`, add a post-load health probe before returning ok=true, such as checking `vec_version()` and/or creating/dropping a tiny temporary vec0 table; return ok=false with a clear error when the probe fails. In Memory Core, keep the existing degraded fallback behavior and make sure `vector.available`, `storeAvailable`, `loadError`, and warnings reflect the probe failure. Add regression coverage for a mocked load success followed by probe failure, plus any Memory Core status/warning coverage needed. Do not pin or rebuild sqlite-vec/better-sqlite3 as part of this fix unless a maintainer explicitly approves a dependency strategy. Add a user-facing CHANGELOG.md fix entry.
