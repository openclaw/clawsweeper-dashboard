## Summary

Keep open. Current main still treats a successful sqlite-vec load call as vector availability without proving vec functions or the vec0 module registered, and the related packaging/warning fixes do not supersede this load-success/no-functions failure mode.

## What This Changes

The issue asks OpenClaw to restore sqlite-vec backed memory vector indexing when the native extension loads but fails to register vec functions because of a SQLite ABI mismatch.

## Best Possible Solution

Add a post-load sqlite-vec usability probe before caching vector availability, preserve a clear loadError when the probe fails, and keep memory indexing/search in the documented degraded fallback path.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: make db.loadExtension or sqliteVec.load succeed, then make vec_version() or CREATE VIRTUAL TABLE ... USING vec0 fail. I did not run a packaged ABI-mismatch host reproduction in this read-only pass.

## Solution Assessment

No for the suggested dependency pin or rebuild as the first repository fix, because that is broader dependency policy. The narrow maintainable fix is a loader/readiness health probe with regression coverage and degraded fallback behavior.
