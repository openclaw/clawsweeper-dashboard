## Summary

Keep open. Current main still treats a successful sqlite-vec native load as sufficient proof that vec0 APIs are usable, while the reported failure mode is exactly a load-success/no-functions state. Related runtime-dependency, warning, fallback-search, and Windows DLL-suffix work is useful context but does not close this Linux ABI/no-functions bug.

## What This Changes

Fix memory vector search degradation when sqlite-vec loads successfully but does not register the vec0 functions or module needed by the builtin memory backend.

## Best Possible Solution

Memory Core should only mark sqlite-vec available after a narrow post-load health probe proves the required vec0 module/functions work, then record an actionable loadError and use the existing JSON embedding plus bounded cosine fallback when the probe fails.

## Reproduction Assessment

Yes. The issue has concrete steps plus two independent Linux confirmations, and current main can be unit-reproduced by making the loader return ok while a vec0 version/function check or temporary vec0 table creation fails.

## Solution Assessment

No for the suggested dependency pin or rebuild as the first repository fix; that is broader dependency management. The narrow maintainable fix is a post-load sqlite-vec usability probe wired into the existing memory vector availability path.
