## Summary

Keep open. Current main still treats a successful sqlite-vec load call as availability, while this report is specifically the silent load-success/no-functions failure mode, and the related items do not supersede this cross-platform vector-readiness bug.

## What This Changes

The issue asks to restore memory vector indexing when sqlite-vec loads but does not register usable vec functions, leaving `chunks_vec` absent and memory search degraded to FTS-only.

## Best Possible Solution

Add a post-load sqlite-vec usability probe before marking the vector store available, preserve a clear `loadError` when the probe fails, and keep the existing degraded fallback path working.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: make `loadExtension` or `sqliteVec.load` succeed, then make `vec_version()` or temporary `vec0` table creation fail. I did not run a current-main live host reproduction in this read-only review, but the issue has multiple concrete runtime confirmations and current source still lacks the probe.

## Solution Assessment

No for the suggested dependency pin or rebuild as the first repository fix, because that is broader dependency policy. The narrower maintainable solution is a loader-level sqlite-vec health probe with regression coverage and clear degraded-state reporting.
