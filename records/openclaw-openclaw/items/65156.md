## Summary

Keep open. Current main still treats a successful sqlite-vec native load as proof that vec0 is usable, without probing the registered SQL functions or module before marking vector search available; the later runtime-deps and fallback-search work addresses adjacent failures but not this load-success/no-functions mode.

## What This Changes

The issue requests fixing Linux memory vector search degradation where sqlite-vec appears to load but registers no vec0 functions or virtual-table module after an SQLite ABI mismatch.

## Best Possible Solution

Add a narrow sqlite-vec post-load health probe in the memory host SDK loader, make memory-core vector availability depend on that probe, surface the probe failure through vector.loadError, and preserve the existing JSON embedding writes plus bounded fallback search when vec0 is unusable.

## Reproduction Assessment

Yes. The issue has concrete steps plus two independent Linux confirmations, and current main can be unit-reproduced by making the sqlite-vec loader return ok while vec0_version or CREATE VIRTUAL TABLE ... USING vec0 fails.

## Solution Assessment

No for the proposed dependency pin or rebuild as the first repair. The narrower maintainable fix is to validate sqlite-vec usability after load and degrade with a clear error when vec0 APIs are absent, leaving dependency rebuilds as a packaging follow-up if still needed.
