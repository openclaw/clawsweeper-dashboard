## Summary

Keep open. Current main still has the load-success/no-functions blind spot: the sqlite-vec loader reports success immediately after the native load call, Memory Core caches that as available, and no merged related item has fixed the post-load usability check.

## What This Changes

The issue asks OpenClaw to restore Memory Core sqlite-vec vector indexing when the native extension appears to load but registers no SQLite vector functions or vec0 module.

## Best Possible Solution

Add a loader-level sqlite-vec health validation step before marking the vector store available, using vec_version() plus a transient vec0 table probe, then preserve degraded fallback with a clear loadError when the probe fails.

## Reproduction Assessment

Yes, source-reproducible. The issue has Linux, Windows, and macOS confirmations, and current main can be unit-reproduced by making the load call succeed while a vec_version()/vec0 probe fails; I did not run a live ABI-mismatch host in this read-only checkout.

## Solution Assessment

Yes for the health-probe direction, not for dependency pinning or rebuilding as the first repository fix. The narrow maintainable repair is to prove sqlite-vec usability after load and fall back cleanly when the proof fails.

## Repair Work Prompt

Fix the sqlite-vec load-success/no-functions bug tracked by https://github.com/openclaw/openclaw/issues/65156. Keep the change focused on Memory Core and memory-host SDK: after loading sqlite-vec, verify usability before reporting ok/available, preferably by checking vec_version() and creating/dropping a temporary vec0 virtual table. If the probe fails, return ok=false with a useful error, keep the existing degraded fallback behavior, and avoid broad dependency pin/rebuild changes unless the health probe proves they are still necessary. Add regression coverage where loadExtension/sqliteVec.load succeeds but vec_version or vec0 creation fails, and make sure existing degraded-warning behavior remains accurate. Add a CHANGELOG.md fix entry because this is user-facing memory behavior. Do not conflate this with the separate Windows suffix fallback PR or embedding-provider warning wording unless the same focused probe naturally improves those paths.
