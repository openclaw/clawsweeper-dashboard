## Summary

Keep open: current main only partially separates volatile inbound metadata; `groupIntro` can still flow into the CLI reuse hash, and empty Claude CLI output can still be treated as a successful run with no payload.

## What This Changes

The issue asks OpenClaw to keep per-turn chat metadata out of Claude CLI session identity, reset stale tmux transcript state after resume healing, and fail closed with diagnostics when CLI output is empty.

## Best Possible Solution

Land one focused CLI-session continuity fix that hashes only stable extra-system prompt inputs, preserves auth and MCP invalidation, and adds explicit retry-or-error handling for empty CLI output.

## Reproduction Assessment

Yes for the session-reset path from source inspection: consecutive group turns can differ only by `groupIntro` while that text is still part of `extraSystemPromptStatic`, which feeds the reuse hash. I did not run a live Feishu/Claude CLI session, so the tmux empty-output symptom is source-supported but not live-reproduced here.

## Solution Assessment

Yes, the requested direction is broadly the right narrow fix: split stable hash identity from full delivered prompt text, then fail closed on empty CLI output. The implementation should avoid weakening auth/MCP invalidation unless maintainers explicitly choose that broader policy.
