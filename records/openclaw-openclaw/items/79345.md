## Summary

Keep open: current main resolves literal Telegram named-account tokens, but Telegram still does not mirror Discord's active runtime snapshot selector for source-config SecretRef cases, and the original report needs credential-exposure follow-up before this is a clean automation target.

## What This Changes

The issue reports that Telegram named-account bot tokens appear configured in status/config output but resolve to undefined at runtime, breaking non-default accounts.

## Best Possible Solution

Keep this issue as the canonical report, rotate/redact the exposed tokens, close the earlier placeholder duplicate, then add a Telegram-owned runtime-snapshot/token-status fix that mirrors Discord while preserving literal botToken behavior.

## Reproduction Assessment

Partially: source inspection gives a reproducible path for source-config SecretRef-backed Telegram named-account tokens lacking Discord-style active runtime snapshot selection. I do not have a high-confidence current-main reproduction for literal plain botToken values becoming undefined.

## Solution Assessment

Yes for the source-proven part: the narrow maintainable fix belongs in the Telegram plugin, reusing the Discord runtime snapshot selector/status pattern and adding Telegram regression tests. A broad core migration or new config option is not the best first step.
