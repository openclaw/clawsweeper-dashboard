## Summary

Keep open: current main still reads Telegram account bot tokens from the config object passed into `resolveTelegramToken`, so a source snapshot with SecretRef-backed named-account tokens can look configured in status/read paths while runtime token resolution fails or returns no token. The discussion also confirms a redacted Telegram bot token leak, so credential rotation should be handled before repair work proceeds.

## What This Changes

Telegram should resolve SecretRef-backed multi-account bot tokens from the active runtime snapshot so status/read paths and runtime token use agree.

## Best Possible Solution

Rotate or revoke the exposed Telegram credentials, then add a Telegram-owned runtime-config selector with focused regression coverage for SecretRef-backed named-account bot tokens while preserving strict unresolved SecretRef behavior.

## Reproduction Assessment

Yes, at source level: pass a source config with `channels.telegram.accounts.<id>.botToken` as a SecretRef while the active runtime snapshot contains the resolved token, then call `resolveTelegramToken` or `resolveTelegramAccount` with the source config. Current main reads the source config directly instead of selecting the runtime snapshot.

## Solution Assessment

Yes for the code direction: the narrow maintainable fix belongs in the Telegram plugin by mirroring the Discord runtime snapshot selection pattern before credential reads. A new config option, core migration, or broad resolver rewrite is not needed.
