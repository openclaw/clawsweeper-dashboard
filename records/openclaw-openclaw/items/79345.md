## Summary

Keep open: current main still lacks a Telegram runtime-config selector before token reads, so SecretRef-backed named-account bot tokens can appear configured in read/status paths while runtime token resolution still reads the source snapshot.

## What This Changes

The issue reports that Telegram multi-account status recognizes configured accounts, but SecretRef-backed bot tokens resolve to empty or undefined values at runtime.

## Best Possible Solution

Rotate or revoke any exposed Telegram credentials, then add a Telegram-owned runtime-config selector and focused regression tests for SecretRef-backed named-account bot tokens while preserving strict unresolved SecretRef behavior.

## Reproduction Assessment

Yes, source-level: set a source config with channels.telegram.accounts.<id>.botToken as a SecretRef and a matching active runtime snapshot with the resolved token, then call resolveTelegramToken with the source config. Current main reads the source config directly instead of selecting the runtime snapshot.

## Solution Assessment

Yes for the code bug: the narrow fix belongs in the Telegram plugin by reusing the runtime snapshot selection pattern already used by Discord. A core migration, new config option, or broad resolver rewrite is not needed.
