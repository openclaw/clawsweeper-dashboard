## Summary

Keep open: current main and the latest shipped tag still leave Telegram token/account resolution without the runtime-config selector required for SecretRef-backed named-account bot tokens, and the prior discussion adds a credential-rotation follow-up.

## What This Changes

The issue reports Telegram multi-account SecretRef tokens appearing configured in status/read paths while runtime account token resolution returns no usable token.

## Best Possible Solution

Rotate or revoke any exposed Telegram credentials, then add a Telegram-owned runtime-config selector mirroring Discord before token/account credential reads with focused regression coverage for SecretRef-backed named accounts.

## Reproduction Assessment

Yes, source-level: set a source config with `channels.telegram.accounts.<id>.botToken` as a SecretRef and a matching active runtime snapshot with the resolved token, then call `resolveTelegramToken` with the source config. Current main reads the source config directly instead of selecting the runtime snapshot.

## Solution Assessment

Yes for the code bug: the narrow maintainable fix is a Telegram plugin runtime-config selector reused by token/account helpers, not a core migration, new config option, or broad SecretRef rewrite.
