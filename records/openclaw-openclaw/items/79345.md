## Summary

Keep open: current main still lacks Telegram's equivalent of Discord's active runtime snapshot selector for SecretRef-backed named account bot tokens, while literal per-account botToken strings appear to be handled. The security side also needs human follow-up because prior discussion says Telegram bot tokens were exposed, even though the current issue body is now empty.

## What This Changes

The issue reports that Telegram multi-account status can show configured accounts while runtime token resolution produces no usable token.

## Best Possible Solution

Rotate/redact any exposed Telegram bot credentials, then add a Telegram-owned runtime config selector matching Discord's pattern and regression tests for SecretRef-backed named account tokens while preserving literal botToken behavior.

## Reproduction Assessment

Yes for the source-level SecretRef path: a matching source config with channels.telegram.accounts.<id>.botToken as a SecretRef can reach Telegram token/account resolution without first selecting the active resolved runtime snapshot. No for the exact live plain-token wording, because the current issue body is empty and current source covers literal account botToken strings.

## Solution Assessment

Yes for the source-proven part: the narrow maintainable fix belongs in the Telegram plugin, reusing the Discord selectApplicableRuntimeConfig pattern and adding Telegram token/account/status regression coverage. A core migration, new config option, or broad resolver rewrite is not required for the first fix.
