## Summary

Keep open: current main still lacks a Telegram-owned active runtime snapshot selector in the token/account resolver, so SecretRef-backed named-account bot tokens can still diverge between status/read paths and runtime token reads. Adjacent fixes for disabled accounts, inbound reply execution, and Discord do not fully cover this Telegram resolver gap, and the redacted token exposure needs human follow-up.

## What This Changes

The issue reports Telegram multi-account status showing accounts while runtime token resolution returns undefined.

## Best Possible Solution

Rotate or revoke any exposed Telegram bot credentials, then add a Telegram runtime-config selector matching Discord's pattern and regression tests for SecretRef-backed named-account bot tokens while preserving strict unresolved SecretRef behavior.

## Reproduction Assessment

Yes, source-level: set a source config with channels.telegram.accounts.<id>.botToken as a SecretRef and a matching active runtime snapshot with the resolved token, then call resolveTelegramToken with the source config. Current main's resolver reads the source config directly instead of selecting the runtime snapshot; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes for the source-proven part: the narrow maintainable fix belongs in the Telegram plugin, reusing the runtime snapshot selection pattern already used by Discord. A core migration, new config option, or broad resolver rewrite is not needed for this bug boundary.
