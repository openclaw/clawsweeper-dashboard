## Summary

Keep open: current main still lacks a Telegram runtime-config selector before `resolveTelegramToken` reads account bot tokens, while the earlier duplicate was closed in favor of this canonical issue. The code bug is narrow, but prior discussion also calls out exposed Telegram credentials that need human rotation/redaction follow-up before repair work proceeds.

## What This Changes

The issue reports that Telegram multi-account status can show configured accounts while runtime token resolution returns no usable token for SecretRef-backed account tokens.

## Best Possible Solution

Rotate or revoke any exposed Telegram credentials, then add a Telegram-owned runtime-config selector and focused regression tests for SecretRef-backed named-account bot tokens while preserving strict unresolved SecretRef behavior.

## Reproduction Assessment

Yes, source-level: use a source config with `channels.telegram.accounts.<id>.botToken` as a SecretRef plus a matching active runtime snapshot containing the resolved token, then call `resolveTelegramToken` with the source config. Current main reads the source config directly instead of selecting the runtime snapshot.

## Solution Assessment

Yes for the code bug: the narrow maintainable fix belongs in the Telegram plugin by reusing the runtime snapshot selection pattern already used by Discord. A core migration, new config option, or broad resolver rewrite is not needed.
