## Summary

Keep open: current main still reads the Telegram source config directly during named-account token resolution, so SecretRef-backed account tokens can be present in runtime snapshots but unavailable to runtime callers.

## What This Changes

The issue reports Telegram multi-account status/read paths showing configured accounts while runtime token resolution returns no token for SecretRef-backed named-account bot tokens.

## Best Possible Solution

Rotate or revoke any exposed Telegram credentials, then add a Telegram-owned runtime-config selector and focused regression tests for SecretRef-backed named-account bot tokens while preserving strict unresolved SecretRef behavior.

## Reproduction Assessment

Yes, source-level: set a source config with channels.telegram.accounts.<id>.botToken as a SecretRef and a matching active runtime snapshot containing the resolved token, then call resolveTelegramToken with the source config. Current main reads the source cfg directly instead of selecting the active runtime snapshot.

## Solution Assessment

Yes for the code bug: the narrow maintainable fix belongs in the Telegram plugin by mirroring the Discord runtime-config selector before token/account credential reads. A new config option, core migration, or broad resolver rewrite is not needed.
