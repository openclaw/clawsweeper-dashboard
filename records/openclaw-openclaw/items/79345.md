## Summary

Keep open: current main still lacks a Telegram runtime-config selector before token reads, so SecretRef-backed named-account bot tokens can be present in the active runtime snapshot while resolver code still reads the source config.

## What This Changes

The issue reports that Telegram multi-account status can show configured accounts while runtime bot-token resolution returns undefined.

## Best Possible Solution

Rotate or revoke any exposed Telegram credentials, then add a Telegram-owned runtime-config selector before token/account credential reads with focused regression coverage for SecretRef-backed named-account bot tokens.

## Reproduction Assessment

Yes, source-level: use a source config with channels.telegram.accounts.<id>.botToken as a SecretRef and a matching active runtime snapshot containing the resolved token, then call resolveTelegramToken with the source config. Current main still reads the source config directly instead of selecting the runtime snapshot.

## Solution Assessment

Yes for the code path: the narrow maintainable fix is to mirror the Discord runtime-config selector pattern inside the Telegram plugin. A new config option, broad core migration, or resolver rewrite is not needed.
