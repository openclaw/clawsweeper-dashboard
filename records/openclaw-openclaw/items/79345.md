## Summary

Keep open: current main still has the Telegram resolver gap. `resolveTelegramToken` and `resolveTelegramAccount` read the passed Telegram config directly, while the runtime contract requires selecting the active runtime snapshot before credential reads, so SecretRef-backed named-account bot tokens can remain unavailable at runtime even when status/read paths see the account.

## What This Changes

Report that Telegram multi-account status can show configured accounts while runtime token resolution returns an unavailable token for SecretRef-backed named-account bot tokens.

## Best Possible Solution

Rotate or revoke any exposed Telegram bot credentials, then add a Telegram-owned runtime-config selector matching Discord's pattern and regression coverage for SecretRef-backed named-account bot tokens while preserving strict unresolved SecretRef behavior.

## Reproduction Assessment

Yes, source-level: configure `channels.telegram.accounts.work.botToken` as a SecretRef in the source config with a matching active runtime snapshot containing the resolved token, then call `resolveTelegramToken(sourceCfg, { accountId: "work" })`. Current main reads the source config directly instead of selecting the runtime snapshot; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes for the code boundary: the narrow maintainable fix is a Telegram plugin runtime-config selector reused by token/account resolution, plus focused regression tests. A core migration, new config option, or broad resolver rewrite is not needed.
