## Summary

Keep open: current main still has the Telegram SecretRef runtime-snapshot gap, and the later secret-scanning notice confirms credential-rotation follow-up is part of the remaining work.

## What This Changes

The issue reports that Telegram multi-account status can show configured accounts while runtime token resolution returns no token for SecretRef-backed account bot tokens.

## Best Possible Solution

Rotate or revoke the exposed Telegram credentials, then add a Telegram-owned runtime-config selector before token/account credential reads with focused regression coverage for SecretRef-backed named accounts.

## Reproduction Assessment

Yes, source-level: use a source config with channels.telegram.accounts.<id>.botToken as a SecretRef plus a matching active runtime snapshot containing the resolved token, then call resolveTelegramToken with the source config. Current main reads the source config directly instead of selecting the runtime snapshot.

## Solution Assessment

Yes for the code direction: the narrow maintainable fix is to mirror Discord’s runtime-config selection pattern inside the Telegram plugin while preserving strict unresolved SecretRef behavior. A new config option or broad core migration is not needed.
