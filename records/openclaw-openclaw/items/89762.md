## Summary

Keep open; the feature is still missing from current main, but this PR is not merge-ready because the new config fallback blurs explicit `off` with inherited default, lacks public config help/docs, and is currently conflicting against main.

## What This Changes

Adds `messages.responseUsage` as a global or per-channel config default for the per-reply usage footer and wires reply rendering to fall back to it when a session has no stored mode.

## Best Possible Solution

Land a focused usage-footer default only after maintainers choose the public config owner, preserve explicit off versus clear-to-inherit across chat/TUI/gateway session updates, align help/docs, and refresh the branch on current main.

## Reproduction Assessment

Yes, for the PR defect at source level: configure a non-off usage default, select or patch `responseUsage` to `off`, and current code deletes the session field so the new fallback would re-enable the configured footer. I did not run a live chat repro in this read-only review.

## Solution Assessment

No. The config fallback is a plausible narrow implementation, but it is not the best merge-ready solution until explicit-off/inherit semantics, config metadata/docs, and the conflicting branch are resolved.
