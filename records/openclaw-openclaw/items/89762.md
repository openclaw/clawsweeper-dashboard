## Best Possible Solution

## Reproduction Assessment

Yes, at source level for the review findings: with `messages.responseUsage` configured and stored session `responseUsage` unset, PR head renders the inherited footer while ACP/TUI presentation and no-argument `/usage` toggles still read the stored row as off. ## Solution Assessment

No. The render fallback and explicit-off persistence are useful pieces, but the best merge-ready solution needs one maintainer-approved config contract and one effective responseUsage state model shared by rendering, presentation, toggles, reset/clear semantics, docs, tests, and proof.
