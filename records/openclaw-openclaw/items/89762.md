## What This Changes

## Best Possible Solution

Merge a focused implementation only after maintainers choose one public config contract and the effective responseUsage state is consistent across reply rendering, presentation/toggles, reset/clear semantics, docs, tests, and current-head real proof.

## Reproduction Assessment

Yes, at source level: with `messages.responseUsage` set and stored session `responseUsage` unset, the PR renders the inherited footer while ACP/TUI presentation and toggles still read the unset session as off. The reset mismatch is also source-reproducible because `session-reset-service.ts` preserves `responseUsage` while the new docs say reset clears it.

## Solution Assessment

No. The render fallback and explicit-off persistence are plausible pieces, but the best merge-ready solution needs one approved config contract and one effective responseUsage model shared by rendering, presentation, toggles, reset/clear semantics, docs, and proof.
