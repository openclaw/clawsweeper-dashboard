- **[P1] Wire defaults into queued channel delivery:** `src/auto-reply/reply/agent-runner.ts:2230`
  - body: This only seeds `messages.responseUsage` in `agent-runner`. Current `followup-runner` persists usage but never resolves `responseUsage` or appends a usage line, so a config such as `{ "telegram": "full" }` can be accepted without producing a Telegram/channel footer. Please wire the same effective default through the queued delivery path, or narrow the config/docs to the surfaces actually supported.
  - confidence: 0.88
- **[P2] Align reset inheritance docs with reset behavior:** `docs/concepts/usage-tracking.md:51-52`
  - body: The new docs say session reset clears the explicit choice and re-inherits the default, but current reset code copies `currentEntry.responseUsage` into the new session entry. After a user runs `/usage off`, reset will keep `off` instead of re-inheriting a non-off default, so either change reset behavior with coverage or remove that promise.
  - confidence: 0.9
- **[P2] Apply inherited state to presentation and toggles:** `src/tui/tui-command-handlers.ts:600`
  - body: The TUI write now persists `next`, but `next` is still computed from stored `state.sessionInfo.responseUsage` only. With an inherited `full` default and no stored value, no-argument `/usage` sees `off` and writes `tokens` instead of disabling the inherited footer; ACP/TUI presentation similarly shows `off`. Please expose an effective response-usage value or avoid no-argument toggles for inherited state.
  - confidence: 0.82
