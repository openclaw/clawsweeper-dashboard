## Summary

Keep this PR open, but it is not merge-ready: the implementation adds a useful focused config default, yet PR head still splits effective `responseUsage` between reply rendering and UI/toggle read paths, documents reset inheritance that reset does not perform, and lacks current-head real behavior proof for the later explicit-off/clear semantics.

## Best Possible Solution

Merge only after maintainers choose the public config contract and one effective responseUsage model is consistently applied across rendering, presentation, toggles, reset/clear semantics, docs, tests, and current-head real proof.

## Reproduction Assessment

Yes, at source level for the review findings: with `messages.responseUsage` configured and stored session `responseUsage` unset, PR head renders the inherited footer while ACP/TUI presentation and no-argument toggles still read the stored row as off. ## Solution Assessment

No. The render fallback and explicit-off persistence are useful pieces, but the best merge-ready solution needs one maintainer-approved config contract and one effective responseUsage state model shared by rendering, presentation, toggles, reset/clear semantics, docs, tests, and proof.
