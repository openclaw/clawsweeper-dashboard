## Summary

Keep open: current main and v2026.6.9 still install a process-wide EnvHttpProxyAgent at Gateway startup when proxy env is set, and the report includes concrete QQ Bot COS media-upload failure logs. The exact NO_PROXY diagnosis needs maintainer triage because undici 8.5.0 source supports leading-dot subdomain matches, so this is not safe to close as stale, duplicate, or external-only.

## What This Changes

The issue reports that OpenClaw v2026.6.9 routes QQ Bot COS chunked media uploads through the gateway-global Undici env proxy dispatcher and fails image delivery when HTTPS_PROXY is present.

## Best Possible Solution

Keep the issue open, capture the real Gateway proxy env and COS request path, then fix the generic core/plugin boundary: either repair EnvHttpProxyAgent/NO_PROXY propagation if it is actually broken, or expose and use a plugin-owned direct dispatcher for presigned uploads without hard-coding QQ Bot or COS policy in core.

## Reproduction Assessment

Partially yes from source: current main and v2026.6.9 install the Gateway global EnvHttpProxyAgent when HTTPS_PROXY is present, and the issue includes concrete COS PUT failure logs. I did not reproduce the Tencent COS/mihomo failure or prove why NO_PROXY was not honored; upstream undici 8.5.0 source says leading-dot subdomain entries should match.

## Solution Assessment

Unclear until maintainers verify the real env and owner boundary. A hard-coded core bypass for QQ Bot/COS would be the wrong layer; the maintainable fix is a generic proxy/NO_PROXY repair or a plugin-facing direct-dispatcher seam used by the QQ Bot upload path.
