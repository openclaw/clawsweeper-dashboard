## Summary

Keep open: current main and v2026.6.9 still install a process-wide EnvHttpProxyAgent when proxy env is present, the reporter provided credible qqbot COS upload failure logs, and multiple open PRs are already competing to fix the bug; the exact low-level cause should be corrected during PR review because locked undici source does support leading-dot no_proxy matching.

## What This Changes

Fix a regression where OpenClaw’s global undici env-proxy dispatcher can route qqbot COS chunked uploads through a proxy despite intended NO_PROXY bypasses.

## Best Possible Solution

Land one core fix that applies generic enhanced proxy-bypass handling to the global dispatcher without hard-coding Tencent domains, preserves existing proxy semantics, and then close or supersede the duplicate PRs.

## Reproduction Assessment

Unclear for a high-confidence live reproduction: I did not run a real qqbot/COS upload, and undici v8.5.0 source supports leading-dot no_proxy matching. Source still supports a credible path because gateway startup installs a global EnvHttpProxyAgent under proxy env, the reporter provided logs, and open fix PRs target the same failure.

## Solution Assessment

Unclear until PR review chooses the exact mechanism. The best fix should live in core global-dispatcher/proxy helpers, preserve ALL_PROXY and timeout rebuild behavior, and avoid plugin-specific domain hard-coding unless maintainers explicitly choose that policy.
