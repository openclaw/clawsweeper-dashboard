## Summary

Keep open: current main and the latest release still install a process-wide Undici EnvHttpProxyAgent when proxy env is configured, the qqbot chunked upload path still performs COS PUT requests, and https://github.com/openclaw/openclaw/pull/96060 is an open linked fix candidate for this issue. The reporter's exact leading-dot root-cause claim needs care because undici 8.5.0 supports leading-dot NO_PROXY matching, but the reported message-loss workflow remains unresolved until one canonical fix lands.

## What This Changes

The issue reports qqbot COS chunked media uploads failing when gateway proxy environment variables cause the global Undici dispatcher to route presigned COS PUT requests through a proxy instead of bypassing them with NO_PROXY.

## Best Possible Solution

Land one generic core fix for global dispatcher proxy-bypass behavior, validate it against qqbot COS uploads or an equivalent real proxy/COS path, then close this issue and supersede the duplicate fix PRs.

## Reproduction Assessment

No for high-confidence live reproduction: I did not run a real qqbot/COS upload with a proxy. Source and logs show the active gateway global-dispatcher and qqbot COS PUT path, but undici 8.5.0 supports leading-dot NO_PROXY matching, so the precise failing condition still needs live confirmation.

## Solution Assessment

Unclear until PR review selects the mechanism. The best fix should live in the core global-dispatcher/proxy helpers, preserve existing proxy and security semantics, and avoid Tencent-domain hard-coding unless maintainers explicitly choose that policy.
