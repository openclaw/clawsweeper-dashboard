## Summary

Keep open: current main has a prior startup-fanout mitigation, but this newer report is a specific Windows 16-Telegram-account starvation case and source inspection still shows per-account Telegram startup probes under the reported phase.

## What This Changes

The issue asks OpenClaw to stop severe event-loop starvation and control-plane RPC stalls when starting or running 16 Telegram bot accounts on Windows.

## Best Possible Solution

Keep this as a focused Telegram multi-account performance investigation: reproduce on Windows with 16 accounts, then either make Telegram account startup nonblocking/backpressured or link it to the canonical gateway-saturation fix if the root cause is shared.

## Reproduction Assessment

No, there is not yet a high-confidence current-main live reproduction. Source inspection confirms the reported `channels.telegram.start-account` path still wraps per-account Telegram `getMe` startup probes, but a Windows 16-account run is needed to prove the 65s starvation on current main.

## Solution Assessment

Unclear. The suggested directions are plausible, but current main already caps startup fanout at four, so the next fix should be based on proof of whether the remaining blocker is Telegram probe timeout waves, polling startup, or shared gateway CPU work.
