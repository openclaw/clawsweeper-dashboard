## Summary

Keep open. The current-release clarification and source review point to a plausible OpenAI-compatible completions transport gap: current code covers reasoning fields inside streaming deltas, but the report describes current v2026.5.4 hangs before any assistant result or terminal stream event.

## What This Changes

The issue reports that OpenAI-compatible completions providers hang the primary reply lane on current-release reasoning models across OpenRouter and direct Moonshot/Kimi-style paths.

## Best Possible Solution

Add a generic OpenAI-compatible completions compatibility path that either parses provider JSON/padded responses or terminates with a clear timeout/failover error while preserving reasoning fields, without baking provider IDs into core.

## Reproduction Assessment

No live provider reproduction was available in this worker, but the issue has current v2026.5.4 reporter confirmation and a clear source path through the OpenAI-completions streaming adapter. The strongest next reproduction is a mock OpenAI-compatible endpoint that returns padded or full JSON reasoning responses on the streamed request path.

## Solution Assessment

Yes, a narrow transport-level fix is the best direction. The repair should stay generic to OpenAI-compatible completions behavior, add regression coverage for `reasoning_content` and `reasoning_details`, and avoid moving this into a provider-specific policy change unless live evidence proves one provider needs a documented wrapper.
