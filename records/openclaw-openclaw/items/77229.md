## Summary

Keep open: the ACP repeated approval-loop fix is still useful and not on current main or the latest release, but this PR is not merge-ready because it makes ACP rely on the shared-token backend self-pairing boundary now tracked as security-sensitive work and it still lacks after-fix real behavior proof.

## What This Changes

The PR changes local loopback shared-auth ACP GatewayClient startup to use a backend gateway-client identity with explicit operator scopes and no device identity, extracts a loopback shared-auth helper, reuses it in operator approvals, adds regression tests, and edits CHANGELOG.md.

## Best Possible Solution

Rebase and redesign the ACP loopback fix around a server-verifiable ACP/local runtime authority or wait for the backend self-pairing decision, while preserving the current approvalRuntimeToken approvals path and adding real after-fix ACP initialization proof.

## Reproduction Assessment

Yes, at source level: current main still constructs ACP as a cli/cli GatewayClient with the default device identity path, while GatewayClient defaults to operator.admin scopes and the server requests scope-upgrade pairing when a paired baseline lacks those scopes. I did not run a live ACP process in this read-only review.

## Solution Assessment

No. The patch is a plausible mitigation for the stale ACP baseline loop, but the best fix should not expand shared-token backend self-pairing without explicit gateway/security approval; a server-verifiable ACP runtime authority is the safer direction.
