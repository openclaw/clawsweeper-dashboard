## Summary

Keep open: the ACP repeated approval-loop problem still has a real source-level bug signal, but this PR is not merge-ready because it makes ACP depend on the shared-token backend self-pairing exception that related gateway security work is trying to remove.

## What This Changes

The PR changes ACP loopback shared-auth startup to omit device identity, use the backend gateway-client shape with explicit operator scopes, extracts a shared loopback-auth helper, adds gateway/ACP tests, and edits the changelog.

## Best Possible Solution

Rebase and redesign the ACP loopback fix around the current gateway auth model, ideally using a server-verifiable ACP/local runtime credential or waiting for the backend self-pairing decision in https://github.com/openclaw/openclaw/pull/90154, then add visible real behavior proof for ACP initialization.

## Solution Assessment

No. The patch is a plausible mitigation for the stale baseline loop, but the better fix must avoid expanding the shared-token backend self-pairing exception or get explicit gateway/security acceptance first.
