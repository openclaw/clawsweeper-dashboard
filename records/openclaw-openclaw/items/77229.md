## Summary

Keep open: the PR targets a valid ACP stale-pairing bug, but the proposed device-identity omission conflicts with the Gateway's current missing-device scope policy and would leave local shared-auth ACP connections without the scopes their normal methods require.

## What This Changes

The branch extracts loopback shared-auth device-identity omission into a gateway helper, applies it to ACP startup, updates operator-approval usage, adds tests, and records a changelog fix.

## Best Possible Solution

Keep the loopback shared-auth fix, but make ACP use a scope-preserving trusted local path and cover a real Gateway ACP request so #74650 is fixed without dropping permissions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR makes ACP device-less while it still identifies as a CLI client, and current Gateway connect policy clears scopes for that shape before ACP calls scoped methods.

## Solution Assessment

No. Extracting the helper is reasonable, but applying it to ACP without preserving scopes is not the narrowest maintainable fix; the ACP path needs the same scope-preserving trust boundary as the backend approval client or equivalent Gateway coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve ACP scopes when omitting the device identity:** `src/acp/server.ts:65-76`
  - body: When this path sets `deviceIdentity: null`, ACP still connects as `cli`/`cli`. The Gateway clears requested scopes for device-less shared-token operator clients unless they are the local `gateway-client`/`backend` exemption, so normal ACP calls such as `sessions.list`, `chat.send`, and `sessions.patch` will fail with missing-scope errors after connect. Please preserve the authorized scopes for this trusted ACP path and add coverage through an actual Gateway request, not only the constructor option.
  - confidence: 0.88
