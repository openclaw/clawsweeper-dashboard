## Summary

Keep open: the PR targets a real ACP repeated approval-loop bug, but the current patch is not merge-ready because it moves ACP onto the shared-token backend self-pairing path while that Gateway security boundary is still under active maintainer review.

## What This Changes

The PR changes loopback shared-auth ACP GatewayClient startup to omit the persisted device identity, identify as backend gateway-client with explicit operator read/write/admin scopes, extracts a loopback shared-auth helper, and adds regression tests plus changelog text.

## Best Possible Solution

Rebase and redesign the ACP loopback fix around the current gateway auth model, ideally using a server-verifiable ACP/local runtime credential or waiting for the backend self-pairing decision in https://github.com/openclaw/openclaw/pull/90154, then add visible real behavior proof for ACP initialization.

## Reproduction Assessment

Yes, at source level. Current main still constructs ACP as cli/cli with default device identity, while GatewayClient defaults to operator.admin scopes and the server requests a scope-upgrade pairing when a paired device baseline lacks those scopes.

## Solution Assessment

No. The patch is a plausible mitigation for the stale baseline loop, but the better fix must avoid expanding the shared-token backend self-pairing exception or get explicit gateway/security acceptance first.
