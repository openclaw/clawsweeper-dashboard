## Summary

This PR should stay open because the stale-pairing ACP failure is a real bug, but the patch would make ACP connect as a device-less CLI client whose requested scopes are cleared by the current Gateway policy before normal ACP requests run.

## What This Changes

The branch extracts loopback shared-auth device-identity omission into a shared Gateway helper, applies it to ACP startup, reuses it from operator approvals, adds focused tests, and adds a changelog fix for #74650.

## Best Possible Solution

Keep the loopback shared-auth stale-identity fix, but route ACP through a scope-preserving trusted local path or adjust the Gateway policy with end-to-end ACP coverage so normal scoped ACP calls still work.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: apply the PR, start ACP against a loopback shared-auth Gateway, and the device-less CLI-shaped connection has its requested scopes cleared before ACP calls scoped methods.

## Solution Assessment

No. The helper extraction is reasonable, but applying deviceIdentity:null to ACP without preserving scopes is not the narrowest maintainable fix; the ACP path needs Gateway-level coverage for real scoped ACP requests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve ACP scopes when omitting the device identity:** `src/acp/server.ts:65-76`
  - body: When this path sets `deviceIdentity: null`, ACP still identifies as the CLI client. The Gateway clears requested scopes for device-less shared-auth operator clients unless they are the direct-local backend `gateway-client` exemption, so normal ACP calls such as `chat.send`, `sessions.patch`, and `sessions.reset` can fail with missing-scope errors after connect. Please keep the stale-identity fix while preserving ACP's authorized scopes, and cover an actual Gateway ACP request rather than only the constructor option.
  - confidence: 0.88
