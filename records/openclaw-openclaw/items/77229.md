## Summary

Keep open: the production direction now matches the intended trusted-local Gateway path, but the added e2e regression still calls a read-scoped method while claiming to prove admin-scope preservation.

## What This Changes

The PR extracts loopback shared-auth identity omission into a shared Gateway helper, changes trusted local ACP launches to gateway-client/backend with explicit operator scopes and no device identity, adds tests, and records a changelog fix for #74650.

## Best Possible Solution

Keep the ACP trust-shape change, but make the Gateway regression exercise an admin- or write-gated request so it proves the exact scopes ACP needs are preserved.

## Reproduction Assessment

Yes. Source inspection shows current main starts ACP as cli/cli with the persisted device identity, and the linked report plus PR body describe the repeated scope-approval loop against a stale low-scope baseline.

## Solution Assessment

No. The implementation direction is reasonable, but the proposed validation is not the best proof because `health` only requires read scope; the e2e test should use `sessions.patch`, `sessions.reset`, `chat.send`, or another admin/write-gated assertion.
