## Summary

Keep open: the ACP stale-baseline bug is still meaningful, but this PR is not merge-ready because its fix moves ACP onto the shared-token backend self-pairing path that is under active security review and still lacks real after-fix behavior proof.

## What This Changes

The PR changes trusted-loopback ACP startup to use backend `gateway-client` identity with explicit operator scopes and no device identity, extracts a loopback shared-auth helper, updates operator approvals to import it, adds ACP/gateway tests, and edits `CHANGELOG.md`.

## Best Possible Solution

Redesign the ACP loopback fix around current gateway auth, preferably with a server-verifiable ACP/local runtime authority or after the backend self-pairing decision in https://github.com/openclaw/openclaw/pull/90154, then add redacted live ACP initialize proof.

## Reproduction Assessment

Yes, at source level: current main still constructs ACP as cli/cli with default device identity behavior, while GatewayClient loads the persisted device identity and the server can request scope upgrades against paired baselines. I did not run a live ACP repro in this read-only review.

## Solution Assessment

No. The PR is a plausible mitigation for the stale paired baseline loop, but the better fix should avoid expanding the shared-token backend self-pairing exception or get explicit gateway/security approval first.
