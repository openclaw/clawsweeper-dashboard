## Summary

Keep this PR open: the production direction appears source-correct, but the latest head still contains the same regression-test blocker that ClawSweeper already called out. The added e2e test claims to prove an admin-gated ACP path, yet it calls `health`, which bypasses normal scope authorization.

## What This Changes

The PR extracts loopback shared-auth device-identity omission into a shared gateway helper, changes trusted local ACP launches to gateway-client/backend with explicit operator scopes and no device identity, and adds tests plus a changelog entry for #74650.

## Best Possible Solution

Keep the ACP trust-shape change, but make the live Gateway regression exercise an admin- or write-gated request so it proves the exact scopes ACP needs are preserved before merge.

## Reproduction Assessment

Yes. Source inspection shows current main starts ACP as cli/cli with the persisted device identity, and the linked issue plus PR body describe the repeated approval loop against a stale low-scope paired baseline; I did not run a live failing reproduction in this read-only pass.

## Solution Assessment

No. The implementation direction is narrow and likely maintainable, but the proposed e2e validation is not the best proof because `health` bypasses scope enforcement; use `sessions.patch`, `sessions.reset`, `chat.send`, or another admin/write-gated method instead.
