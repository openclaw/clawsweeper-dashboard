## Summary

Keep open: current main still does not expose an enforced per-spawn tool restriction for `sessions_spawn`, and the active implementation candidate is open but unmerged, so this remains the canonical API/security feature request.

## What This Changes

Request adds a per-call tool allow/deny restriction to `sessions_spawn` so spawned subagents can run with narrower capabilities.

## Best Possible Solution

Keep this issue as the canonical tracker and land one additive `sessions_spawn` tool-policy API with enforced final narrowing, documented precedence, focused regression coverage, and fail-closed behavior for unsupported runtimes.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main gives high-confidence proof that the public schema, native params, spawn handoff, resolver, docs, and closed Gateway RPC do not expose a per-spawn tool-policy path.

## Solution Assessment

Unclear until maintainers settle the final API shape. Reusing the existing embedded `toolsAllow` primitive looks maintainable for native runs, but the requested security boundary also needs explicit decisions for deny semantics, no-tools mode, and unsupported runtimes.
