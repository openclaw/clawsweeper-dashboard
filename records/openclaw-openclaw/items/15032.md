## Summary

Keep open: current `main` still does not expose a per-spawn tool restriction on `sessions_spawn`; the related implementation PR is open but unmerged, so this remains the canonical API/security feature request.

## What This Changes

The issue requests an optional per-spawn allow/deny tool policy for subagents so callers can enforce least-privilege child sessions for untrusted-content workflows.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers accept and land an additive `sessions_spawn` tool-policy API with docs, protocol/schema updates, regression coverage, and clear precedence against existing global/subagent policy.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current `main` high-confidently shows the missing public `sessions_spawn` policy path in schema, spawn params, child Gateway RPC, session metadata, resolver, and docs.

## Solution Assessment

Unclear until maintainers settle the API shape. The safest direction is to reuse the existing embedded `toolsAllow` machinery for native child runs while explicitly deciding deny semantics, no-tools behavior, and fail-closed handling for runtimes that cannot enforce the policy.
