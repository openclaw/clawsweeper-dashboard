## Summary

Keep open: current main and v2026.6.8 still do not expose per-spawn tool restrictions on `sessions_spawn`, and the open implementation candidate is unmerged and currently conflicting, so the canonical security/API tracker is still needed.

## What This Changes

Request adding per-spawn tool allow/deny restrictions to `sessions_spawn` so child agents can run with narrower capabilities for untrusted-content workflows.

## Best Possible Solution

Keep this issue as the canonical per-spawn `sessions_spawn` tool-policy tracker, then land or replace the open implementation after maintainers accept the API name, precedence, persistence, unsupported-runtime behavior, and security boundary.

## Reproduction Assessment

Yes, as a source-level reproduction: current main's `sessions_spawn` schema, native spawn params, and child-session policy path have no per-call tool restriction field. I did not run a live agent flow because the missing API surface is directly visible in source.

## Solution Assessment

Unclear for the final API shape. The open `toolsAllow` PR is a plausible implementation path, but maintainers still need to choose the shipped contract across allow-only, deny semantics, tool-less mode, plugin/MCP names, persistence, and unsupported runtimes.
