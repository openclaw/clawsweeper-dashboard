## Summary

Keep open: current main still exposes only global subagent tool policy and lacks a per-spawn policy carrier, persisted child-session policy, or resolver merge path for `sessions_spawn`. The closed unmerged PR #23506 and related open items #23906/#58434 are useful context, but they do not ship this sessions_spawn API.

## What This Changes

Add a per-spawn `sessions_spawn` tool policy parameter so child agents can be restricted to allowed or denied tools without changing global subagent policy.

## Best Possible Solution

Keep this issue open as the canonical sessions_spawn capability-isolation request, then implement an additive, documented per-spawn policy contract with focused schema, session metadata, resolver, unit/e2e, and docs coverage.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main confirms the sessions_spawn schema, spawn params, child metadata, and subagent policy resolver have no per-spawn tool policy path.

## Solution Assessment

Unclear until maintainers settle the API shape. The narrow maintainable direction is an additive sessions_spawn policy contract that reconciles `allow`/`deny` with `tools: "none"` and preserves current behavior when omitted.
