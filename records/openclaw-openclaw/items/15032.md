## Summary

Keep open: current main and v2026.6.6 still lack a public per-spawn tool policy for `sessions_spawn`; the open implementation candidate is useful context but is not merged, is currently conflicting, and still needs maintainer/security agreement on the final API semantics.

## What This Changes

Request to let `sessions_spawn` restrict a child sub-agent's tools per spawn for least-privilege isolation in untrusted-content workflows.

## Best Possible Solution

Keep this issue as the canonical tracker, then land or revise https://github.com/openclaw/openclaw/pull/78441 or an equivalent additive `sessions_spawn` policy API with documented precedence, persisted child-session metadata, focused tests, and docs at https://docs.openclaw.ai/tools/subagents.

## Reproduction Assessment

Yes, as a source-level reproduction: current main's `sessions_spawn` schema, spawn params, and policy resolver do not accept or apply a per-spawn child tool policy. I did not run a live agent flow because the missing API is visible in source.

## Solution Assessment

Yes on direction, but not yet on final shape: a core `sessions_spawn` capability boundary is the right layer for this security use case, while the exact allow/deny/tool-less API and ACP/plugin behavior need maintainer review.
