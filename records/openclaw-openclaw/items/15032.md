## Summary

Keep open: current main still lacks an enforced per-spawn tool restriction for `sessions_spawn`, and the active implementation candidate is open, unmerged, and narrower than the original allow/deny security-boundary request.

## What This Changes

Request an optional per-spawn `sessions_spawn` tool policy so child sub-agents can be restricted to an allowlist, denylist, or tool-less mode.

## Best Possible Solution

Land one reviewed additive `sessions_spawn` tool-policy API with enforced final narrowing, documented precedence, focused coverage, fail-closed unsupported runtime behavior, and clear handling for no-tools and deny semantics.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main gives high-confidence proof that the schema, native params, spawn handoff, resolver, docs, and Gateway request schema do not expose a per-spawn tool-policy path.

## Solution Assessment

Unclear until maintainers approve the final API shape. Reusing the existing embedded `toolsAllow` primitive may be a narrow first step, but the full request still needs decisions on deny semantics, no-tools mode, unsupported runtimes, and persistence.
