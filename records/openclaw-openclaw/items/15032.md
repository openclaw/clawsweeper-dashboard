## Summary

Keep open. Current main and the latest release still do not expose a public per-spawn `sessions_spawn` tool-policy parameter, and the open related implementation candidate is not merged and is currently conflicting, so this remains the canonical product/security/API decision.

## What This Changes

The issue requests a per-spawn tool policy for `sessions_spawn` so child sub-agents can be restricted to specific allowed or denied tools instead of inheriting the parent tool surface.

## Best Possible Solution

Keep this as the canonical core tracker and land either https://github.com/openclaw/openclaw/pull/78441 or a replacement only after maintainers approve the API name, allow/deny or no-tools semantics, policy precedence, session persistence, unsupported-runtime behavior, focused tests, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes for the reported missing capability: current source and v2026.6.8 both show `sessions_spawn` has no per-spawn tool-policy parameter or downstream carrier.

## Solution Assessment

Yes on direction, unclear on final shape. A per-spawn core API is the right boundary for hard least-privilege subagents, but maintainers still need to choose the public contract and whether the current `toolsAllow` candidate is sufficient versus an allow/deny or no-tools model.
