## Summary

Keep open: current main and v2026.6.6 still do not expose a per-call tool policy on `sessions_spawn`. Cron/runtime allowlist plumbing now exists and https://github.com/openclaw/openclaw/pull/78441 is useful implementation context, but that PR is still open, currently conflicting, and needs maintainer API/security acceptance before this issue can close.

## What This Changes

The issue requests an additive `sessions_spawn` per-call tool allow/deny policy so spawned child agents can run with narrower enforced tool access.

## Best Possible Solution

Land a maintainer-approved additive `sessions_spawn` per-spawn tool policy that reuses the existing runtime allowlist substrate, persists the child policy deliberately, rejects unsupported runtimes clearly, and documents the shipped contract.

## Reproduction Assessment

Yes, as a source-level feature-gap reproduction: current main and v2026.6.6 have no model-facing `sessions_spawn` field or native spawn parameter that carries a per-spawn tool restriction. I did not run a live prompt-injection exploit because the missing API and handoff path are visible in source.

## Solution Assessment

Unclear until maintainers choose the public contract. A per-spawn capability boundary is the right core layer for this request, but the remaining decision is whether the shipped API should be allow-only, allow/deny, tool profile based, native-only, or broader.
