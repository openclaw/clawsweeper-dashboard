## Summary

Keep open. Current main still lacks an enforced per-spawn tool restriction API for `sessions_spawn`; the existing implementation only supports global sub-agent tool policy plus depth/role restrictions. The Claw Drive comment and closed unmerged PR #23506 remain useful implementation context, while #23906 is related but not a shipped replacement.

## What This Changes

The issue requests an optional per-call tool allow/deny policy for spawned sub-agents so callers can enforce least-privilege isolation for untrusted-content workflows.

## Best Possible Solution

Keep this as the canonical `sessions_spawn` core API/security gap and implement an additive per-spawn policy that is parsed by the tool schema, carried through native spawn params, persisted on child session metadata, enforced by the sub-agent policy resolver, covered by focused unit/e2e tests, and documented on the public sub-agents page.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: current main has no schema field, spawn parameter, session metadata field, or policy resolver input for per-spawn tool restrictions, so passing `tools` or `denyTools` to `sessions_spawn` cannot enforce the requested isolation.

## Solution Assessment

Unclear on the exact API spelling, but the best direction is a narrow additive core API rather than prompt-only instructions or a plugin workaround. The implementation should reconcile this issue's allow/deny object with #23906's `tools: "none"` use case before landing.
