## Summary

Keep open. Current `main` and the latest release still lack a public per-spawn tool restriction on `sessions_spawn`; the earlier allow/deny implementation was closed unmerged, and the newer `toolsAllow` implementation remains open and conflicting rather than landed.

## What This Changes

The issue requests an enforced per-spawn tool policy for `sessions_spawn` so child sub-agents can run with a narrower tool set than the parent.

## Best Possible Solution

Keep this as the canonical core API/security gap and resolve it by reviewing and landing https://github.com/openclaw/openclaw/pull/78441 or a replacement that threads an additive per-spawn narrowing policy through the tool schema, spawn params, session metadata/protocol, runtime filtering, tests, and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction of the API gap: current `sessions_spawn` schema, native spawn params, child-session metadata, and gateway protocol do not carry a per-spawn tool restriction.

## Solution Assessment

Unclear as a final implementation decision. Reusing lower-level `toolsAllow` is plausible, but maintainers still need to choose the public API and security precedence before this should be treated as an autonomous fix.
