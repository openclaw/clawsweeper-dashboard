## Summary

Keep open: current main still lacks a model-facing `sessions_spawn` allow/deny/none parameter and durable per-spawn policy carrier. The related `toolsAllow` PR is open and useful, but it is not merged and leaves maintainer API/security-scope choices unresolved.

## What This Changes

The issue requests a per-spawn tool policy on `sessions_spawn` so spawned sub-agents can be restricted to an allowlist, denylist, or no tools for least-privilege workflows.

## Best Possible Solution

Keep this as the canonical tracker and land one maintainer-approved additive `sessions_spawn` policy contract that only narrows child capabilities, composes with existing policy layers, and ships with focused tests plus docs.

## Reproduction Assessment

Not applicable: this is a feature and security-boundary request, not a broken existing behavior. Source inspection gives high confidence that current main lacks the schema field, spawn-param carrier, durable policy metadata, and resolver input needed for the requested contract.

## Solution Assessment

Unclear until maintainers approve the public API and policy precedence. The maintainable direction is to reuse the existing embedded-runner/tool-policy pipeline with additive narrowing semantics that cannot expand child capabilities.
