## Summary

Keep open: current main and the latest release still do not expose a caller-controlled per-spawn tool restriction on `sessions_spawn`. The open implementation at https://github.com/openclaw/openclaw/pull/78441 is the active candidate, but it is not merged and still needs maintainer product/security review for the public API shape and enforcement semantics.

## What This Changes

The issue requests a per-spawn `sessions_spawn` tool restriction API so sub-agents can be hard-limited to specific tools for security-sensitive orchestration.

## Best Possible Solution

Keep this issue open as the canonical core API/security-boundary request and resolve it by reviewing and either landing or replacing the open `toolsAllow` implementation with a maintainer-approved per-spawn policy shape.

## Reproduction Assessment

Yes. Source inspection of current main shows the public `sessions_spawn` schema and native spawn params do not accept a caller-supplied per-spawn allow/deny policy, while the runtime only carries inherited/global policy.

## Solution Assessment

Yes, a core `sessions_spawn` API extension is the right layer because prompt instructions are not an access-control boundary and plugins cannot retrofit a model-visible core tool parameter. The unresolved part is maintainer selection of the exact API shape and runtime support matrix.
