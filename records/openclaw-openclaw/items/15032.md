## Summary

Keep open: current main still lacks an enforced per-spawn tool restriction on `sessions_spawn`, and the active implementation candidate is open, unmerged, and narrower than the original allow/deny request.

## What This Changes

The issue requests an optional per-call tool policy for `sessions_spawn` so spawned sub-agents can run with a restricted tool set for least-privilege workflows.

## Best Possible Solution

Keep this issue as the canonical tracker and land one reviewed additive `sessions_spawn` tool-policy API with enforced final narrowing, documented precedence, focused regression coverage, and fail-closed unsupported-runtime behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main gives high-confidence proof that the public schema, native params, spawn handoff, resolver, docs, and Gateway agent schema do not expose a per-spawn tool-policy path.

## Solution Assessment

Unclear until maintainers approve the final API shape. Reusing the existing embedded `toolsAllow` primitive may be a narrow first step, but the full request also needs decisions on deny semantics, no-tools mode, unsupported runtimes, and persistence.
