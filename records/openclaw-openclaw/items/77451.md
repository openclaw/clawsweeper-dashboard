## Summary

Keep open: this draft PR is not implemented on current main, and the proposed global memoization conflicts with the documented plugin control-plane cache boundary. The existing ClawSweeper review comment correctly identified the same architecture blocker, so the next step is owner review or redesign rather than cleanup closure.

## What This Changes

The PR adds a bounded LRU cache around `loadPluginManifestRegistryForInstalledIndex`, clears it on installed-index writes, and adds focused memoization tests.

## Best Possible Solution

Preserve the performance goal by reusing an explicit current-flow `PluginMetadataSnapshot`, `PluginLookUpTable`, or caller-owned manifest registry, or update the cache boundary with plugin-owner approval before adding persistent memoization.

## Reproduction Assessment

Not applicable as a draft performance PR rather than a user-facing bug report. The blocker is source-reviewable from the PR diff, `src/plugins/AGENTS.md`, and the plugin architecture docs.

## Solution Assessment

No. The PR targets a plausible hot path, but a module-level installed manifest-registry cache is not the narrowest maintainable solution under the current plugin metadata freshness contract.
