## Summary

Keep open: this draft PR targets a plausible plugin hot path, but the proposed process-wide installed-manifest-registry cache conflicts with the documented plugin cache boundary, and the user-facing perf change is still missing a changelog entry.

## What This Changes

The PR adds a bounded LRU cache around installed-index manifest registry reconstruction, clears it from installed-index writers, and adds focused memoization tests.

## Best Possible Solution

Route the perf fix through current-flow `PluginMetadataSnapshot`/`PluginLookUpTable`/explicit registry reuse, or deliberately update the plugin cache contract before adding any process-wide installed-registry cache.

## Reproduction Assessment

Not applicable: this is a draft performance PR rather than a bug report. The blocking behavior is source-reviewable from the PR diff, current main, and the plugin cache-boundary docs.

## Solution Assessment

No. The PR optimizes a plausible hot path, but a hidden module-level metadata cache is not the narrowest maintainable solution under the current plugin architecture; caller-owned snapshot or registry reuse is the safer path.
