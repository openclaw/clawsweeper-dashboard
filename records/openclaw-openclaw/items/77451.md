## Summary

Keep this draft PR open for maintainer review. The performance goal is understandable, but the proposed module-level installed manifest registry cache conflicts with the plugin boundary rule that control-plane manifest registry answers stay fresh unless owned by an explicit current-flow handle.

## What This Changes

The PR adds a bounded 16-entry LRU cache around `loadPluginManifestRegistryForInstalledIndex`, clears it from installed-index writes, and adds focused memoization tests.

## Best Possible Solution

Preserve the control-plane freshness contract and move the performance fix to an allowed caller-owned snapshot/registry path, or update the plugin cache boundary with owner approval before adding a persistent manifest-registry cache.

## Reproduction Assessment

Not applicable as a draft performance PR rather than a user-facing bug report. The blocker is source-reviewable from the PR diff and `src/plugins/AGENTS.md`.

## Solution Assessment

No. The PR improves a plausible hot path, but a module-level manifest registry cache is not the narrowest maintainable solution under the current plugin boundary rules.
