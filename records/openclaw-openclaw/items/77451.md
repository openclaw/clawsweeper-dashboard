## Summary

Keep open: the draft PR still adds a persistent module-level manifest-registry cache that conflicts with the plugin cache-boundary contract, and the user-facing perf change also lacks the required changelog entry.

## What This Changes

The PR adds a 16-entry LRU memoization cache around `loadPluginManifestRegistryForInstalledIndex`, clears it from installed-index write paths, and adds focused memoization tests.

## Best Possible Solution

Route the performance fix through existing current-flow `PluginMetadataSnapshot`/lookup-table/explicit registry reuse, or have a plugin owner deliberately update the cache contract before adding a bounded installed-registry cache.

## Reproduction Assessment

Not applicable: this is a draft performance PR rather than a bug report. The blocker is source-reviewable from the PR diff plus `src/plugins/AGENTS.md` and the plugin architecture docs.

## Solution Assessment

No. The PR targets a plausible hot path, but the current design adds a hidden persistent control-plane cache where the documented path is caller-owned snapshot, lookup-table, or explicit registry reuse.
