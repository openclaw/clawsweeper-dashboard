## Summary

Keep this draft PR open for maintainer review: the performance target is plausible, but the patch conflicts with the current plugin cache-boundary rules and has a config-sensitive cache-key correctness problem.

## What This Changes

The PR adds a module-level 16-entry LRU cache around `loadPluginManifestRegistryForInstalledIndex`, clears it from installed-index writers, and adds focused memoization tests.

## Best Possible Solution

Move the optimization into a caller-owned `PluginMetadataSnapshot` or `PluginLookUpTable` flow that carries the existing control-plane fingerprint, or first update the plugin cache-boundary contract with owner approval.

## Reproduction Assessment

Not applicable as a draft performance PR rather than a user-facing bug report. The blocking issues are source-reviewable from the PR diff, the scoped plugin guide, and config-sensitive manifest-registry behavior on current main.

## Solution Assessment

No. The proposed module-level cache is not the narrowest maintainable fix under the current plugin boundary rules, and its key can reuse a registry across different plugin configs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid caching installed manifest registries globally:** `src/plugins/manifest-registry-installed.ts:38`
  - body: `src/plugins/AGENTS.md` says manifest registry and installed-index metadata must stay fresh unless a caller owns an explicit current-flow snapshot or registry. This module-level Map persists across callers and returns the same registry on cache hits, so write-path and mtime invalidation still change the current control-plane freshness contract. Please move the perf work to a caller-owned snapshot/lookup-table path or get the cache boundary changed with owner review.
  - confidence: 0.9
- **[P2] Include plugin config in the cache key:** `src/plugins/manifest-registry-installed.ts:53-59`
  - body: `loadPluginManifestRegistry` derives registry diagnostics from `config.plugins`, but this key only includes index, env, workspace, pluginIds, and includeDisabled. Two calls with the same index but different plugin config can therefore share a registry built under the wrong config, for example preserving a channel-config warning after the plugin is disabled or denied.
  - confidence: 0.86
