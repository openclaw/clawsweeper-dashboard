## Summary

Keep open. Current main still has retention-based cron cleanup and one-shot/direct `deleteAfterRun` cleanup, but no public completion-based `ephemeral` mode for recurring isolated cron jobs or hook mappings.

## What This Changes

The issue requests an opt-in ephemeral session mode that deletes or archives one-shot cron and hook-created session state immediately after the agent run completes.

## Best Possible Solution

Add a public opt-in `ephemeral` cleanup policy for isolated cron jobs and hook agent mappings, delete only the exact run-owned session entry after terminal completion, preserve cron run logs, and document transcript archive/delete behavior while leaving existing retention settings as the fallback.

## Reproduction Assessment

Yes. A source-level reproduction is enough: `ephemeral` is absent from cron CLI/protocol/config surfaces and rejected or ignored by strict hook mapping config, while completed isolated work still flows through retention or narrow `deleteAfterRun` paths.

## Solution Assessment

Yes, the requested completion-cleanup primitive is the right fit if it is opt-in and limited to owned isolated sessions. The safer alternative is not to broaden existing `deleteAfterRun` cleanup to arbitrary live session targets.
