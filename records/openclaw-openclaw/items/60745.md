## Summary

Keep open. Current main still offers retention-based cron session pruning and adjacent `deleteAfterRun` cleanup, but not the requested public completion-based `ephemeral` mode for recurring isolated cron jobs or hook mappings.

## What This Changes

Add an opt-in ephemeral session mode for cron jobs and hook agent mappings that disposes completed one-shot workload sessions immediately instead of waiting for retention pruning.

## Best Possible Solution

Add a public opt-in `ephemeral` cleanup policy for isolated cron jobs and hook agent mappings, delete only the exact completed run-owned session entry after terminal completion, preserve cron run logs, and document transcript archive/delete behavior.

## Reproduction Assessment

Yes. A source-level reproduction is enough: current strict cron and hook config surfaces omit `ephemeral`, hook mappings reject unknown fields, and completed isolated cron cleanup still flows through retention or narrow `deleteAfterRun` paths.

## Solution Assessment

Yes. The requested completion-cleanup primitive is the narrow maintainable direction if it is opt-in and scoped to owned isolated sessions; the safer alternative is not to broaden existing `deleteAfterRun` cleanup to arbitrary live session targets.

## Repair Work Prompt

Implement opt-in ephemeral session cleanup for this issue. Add a public `ephemeral?: boolean` field for cron jobs and hook agent mappings so isolated one-shot workloads can delete their completed run-owned session entry immediately after terminal completion instead of waiting for `cron.sessionRetention` or session maintenance. Scope the fix narrowly: support cron JSON/protocol/tool/CLI/config surfaces, hook mapping config/types/resolution, and runtime cleanup after success or error. Preserve cron run logs. Reuse existing session-store/transcript archive/delete helpers where possible. Do not broaden existing `deleteAfterRun` behavior for `current` or `session:<id>` live bindings, and add a guard/regression aligned with related #69467 so ephemeral cleanup only deletes owned isolated run sessions. Update docs and `CHANGELOG.md` because this is a user-facing feature; use plugin terminology and keep generated config docs/checks aligned. Do not change unrelated session retention defaults or global maintenance policy.
