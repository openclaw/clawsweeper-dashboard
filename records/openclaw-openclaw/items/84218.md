## Label Justifications

- `P1`: The report describes a repeated overflow, compaction, and heartbeat restart loop that can stop scheduled agent work on real deployments.
- `impact:crash-loop`: The issue is about deterministic heartbeat overflow prechecks followed by repeated restart cycles.
- `impact:session-state`: The affected behavior is stale heartbeat context being associated with the stable heartbeat session key despite fresh session ids.

## Summary

Keep open: current `main` still has a source-backed mismatch between the documented fresh isolated heartbeat contract and the stable heartbeat session key passed into context-engine assembly, which can plausibly replay accumulated heartbeat context into later ticks.

## What This Changes

The issue reports that heartbeat runs with `isolatedSession=true` and `lightContext=true` can replay prior heartbeat context and enter repeated context-overflow/restart loops.

## Best Possible Solution

Make `isolatedSession=true` honor the documented fresh-context contract by preventing context-engine/session replay of prior heartbeat ticks while preserving stable delivery routing and existing isolated-key convergence behavior.

## Reproduction Assessment

No executed current-main reproduction was established in this read-only review. Source inspection gives a clear path: isolated heartbeat ticks get fresh session ids while context-engine lifecycle calls still receive the stable `<base>:heartbeat` session key.

## Solution Assessment

Yes: the narrow maintainable direction is to honor the existing docs by making isolated heartbeat context truly fresh/bounded by default, not to add a new config knob as the primary repair.
