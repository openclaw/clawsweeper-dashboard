## Label Justifications

- `P1`: The report describes a repeated context-overflow, compaction, and restart loop that can stop scheduled heartbeat work on real deployments.
- `impact:crash-loop`: The issue is about deterministic overflow prechecks followed by repeated heartbeat restart cycles.
- `impact:session-state`: The affected behavior is stale heartbeat context being associated with a stable heartbeat session key despite fresh session ids.

## Summary

Keep open: current main still threads the stable isolated heartbeat `SessionKey` into context-engine assembly and lifecycle calls, so the documented fresh-context contract is not yet implemented. The open fix candidate is https://github.com/openclaw/openclaw/pull/84248, which specifically closes this issue.

## What This Changes

Make `isolatedSession=true` heartbeat runs truly fresh or bounded by preventing prior heartbeat context-engine/session replay from entering later ticks.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/84248 with a narrow key split: keep the stable heartbeat delivery `SessionKey`, but pass a fresh per-tick context-engine identity through bootstrap, assemble, after-turn/ingest, and compaction paths before closing this issue.

## Reproduction Assessment

No executed high-confidence current-main reproduction was run in this read-only review. The source path is clear: isolated heartbeat creates a fresh session id but still passes the stable `:heartbeat` key into context-engine lifecycle calls, matching the supplied production overflow trace.

## Solution Assessment

Yes: the narrow maintainable fix is to honor the existing documented contract by separating delivery routing from context-engine state identity. Adding a new history knob can be a follow-up, but it should not be required to make `isolatedSession=true` fresh by default.
