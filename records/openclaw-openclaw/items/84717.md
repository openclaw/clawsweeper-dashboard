Keep open because current main still has the reported limiter behavior, and the paired open fix PR has not merged. The issue should close only after https://github.com/openclaw/openclaw/pull/84719 or an equivalent fix lands with regression coverage.

## What This Changes

The issue requests that `maxTrackedKeys` be treated as a hard active-key cap so the webhook in-flight limiter cannot evict counters for still-running requests.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84719 or an equivalent narrow patch, then close this issue after the fix is on main with regression coverage for the full active-key table case.

## Reproduction Assessment

Yes. Source inspection directly matches the reported sequence: acquire `a`, acquire `b` with `maxTrackedKeys: 1`, prune the oldest active key, then `release('a')` becomes a no-op because the counter is gone.

## Solution Assessment

Yes. The linked fix direction is the narrow maintainable solution: reject brand-new keys when the active key table is full while preserving existing-key acquisition up to `maxInFlightPerKey`.
