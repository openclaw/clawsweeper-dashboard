## Summary

Keep open: current main and the latest release still use the insert-then-prune active-counter behavior, and the linked candidate fix is open, unmerged, and conflicting rather than landed.

## What This Changes

The issue asks OpenClaw to stop the shared webhook in-flight limiter from evicting active per-key request counters when `maxTrackedKeys` is full.

## Best Possible Solution

Land or replace the linked candidate with a refreshed plugin-SDK fix that preserves active counters, documents the hard-cap behavior, and includes focused regression plus runtime proof.

## Reproduction Assessment

Yes. Source inspection on current main shows that with `maxInFlightPerKey: 1` and `maxTrackedKeys: 1`, acquiring `a` then `b` can prune `a`, making `release("a")` a no-op.

## Solution Assessment

Yes for the likely code location, but maintainer review is still needed because making the tracked-key cap reject brand-new active keys changes public plugin-SDK behavior under key-spray pressure.
