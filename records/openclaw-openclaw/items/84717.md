## What This Changes

The issue asks for `createWebhookInFlightLimiter` to reject brand-new keys when the active key table is full instead of pruning active in-flight counters.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84719, or an equivalent narrow fix, so new keys are rejected at active-key capacity while existing tracked keys keep normal acquire and release semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with `maxInFlightPerKey: 1` and `maxTrackedKeys: 1`, acquiring `a` then `b` admits `b` and prunes oldest key `a`.

## Solution Assessment

Yes. Rejecting only brand-new keys when the active table is full is the narrow maintainable fix because it preserves same-key accounting and avoids deleting live counters.
