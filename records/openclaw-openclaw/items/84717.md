## What This Changes

The issue asks for `createWebhookInFlightLimiter` to preserve active request counters by rejecting brand-new keys when `maxTrackedKeys` is already full.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84719 or an equivalent patch with regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: acquire `a`, acquire `b` with `maxTrackedKeys: 1`, then `a` can be pruned while active and `release('a')` becomes a no-op.

## Solution Assessment

Yes, the narrow maintainable fix is to check active-key capacity before inserting a brand-new key while preserving existing-key acquisition up to `maxInFlightPerKey`.
