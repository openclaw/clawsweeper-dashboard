## What This Changes

The issue asks for `createWebhookInFlightLimiter` to reject brand-new keys when active key tracking is full instead of pruning counters for active requests.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/84719, then close this issue only after the fix is on main with regression coverage.

## Reproduction Assessment

Yes. Source inspection directly reproduces the sequence: acquire `a`, acquire `b` with `maxTrackedKeys: 1`, prune `a`, then `release('a')` becomes a no-op because the counter is gone.

## Solution Assessment

Yes. The narrow maintainable fix is to check active-key capacity before inserting a brand-new key while preserving existing-key acquisition up to `maxInFlightPerKey`.
