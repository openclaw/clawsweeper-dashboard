## What This Changes

The issue asks for `createWebhookInFlightLimiter` to preserve active request counters by rejecting brand-new keys when the active-key table is already at `maxTrackedKeys`.

## Best Possible Solution

Land a refreshed version of the linked fix that preserves active in-flight counters, keeps the public SDK behavior intentional, and adds focused regression coverage.

## Reproduction Assessment

Yes, source inspection is enough to reproduce the failure path. With `maxInFlightPerKey: 1` and `maxTrackedKeys: 1`, current main can acquire `a`, acquire `b`, prune `a`, and then make `release("a")` a no-op.

## Solution Assessment

Yes for the code shape: checking capacity before inserting a brand-new key is the narrowest way to preserve active counters. The remaining decision is whether maintainers accept the public SDK behavior change from pruning to rejecting new keys at capacity.
