## Summary

Keep this open: current main and latest release v2026.6.8 still use the set-then-prune limiter behavior, and https://github.com/openclaw/openclaw/pull/84719 is an open, unmerged candidate fix.

## What This Changes

The issue requests preserving active webhook in-flight counters by rejecting brand-new keys when the active key table is full instead of pruning active entries.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84719 or an equivalent narrow patch that preserves active counters, keeps regression coverage, and records the intended 429-at-capacity behavior.

## Reproduction Assessment

Yes. Source inspection on current main matches the reported sequence: acquire `a`, acquire `b` with `maxTrackedKeys: 1`, prune `a`, then `release('a')` becomes a no-op.

## Solution Assessment

Yes for the code shape: checking capacity before inserting a brand-new key is the narrow fix that preserves active counters. It still needs maintainer acceptance because it changes overload behavior for new webhook keys at capacity.
