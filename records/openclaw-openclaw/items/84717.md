Keep open: current main and the latest release still have the reported source-level limiter bug, and an open linked fix PR is already paired with this issue. The issue should close only after that PR or an equivalent fix lands on main.

## What This Changes

The issue asks for `maxTrackedKeys` in `createWebhookInFlightLimiter` to act as a hard active-key capacity cap instead of pruning still-active counters.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84719 or an equivalent patch with regression coverage, then close this issue once main preserves active counters at capacity.

## Reproduction Assessment

Yes. Source inspection matches the reported sequence: acquire `a`, acquire `b` with `maxTrackedKeys: 1`, prune the oldest active key, then `release('a')` becomes a no-op because the counter is gone.

## Solution Assessment

Yes. Rejecting brand-new keys when the active table is full while preserving existing-key acquisition up to `maxInFlightPerKey` is the narrow maintainable fix; the linked PR follows that direction but still needs normal review/proof before merge.
