## Summary

This should stay open because current main still has the reported limiter behavior, and an open linked fix PR is already paired with the issue. The source path is clear: a new key can be accepted and then prune the oldest active key counter, so the best cleanup path is to review and land or reject the linked PR rather than close this issue early.

## What This Changes

The issue reports that `createWebhookInFlightLimiter` treats `maxTrackedKeys` as a prune limit instead of a hard cap for active in-flight key counters.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/84719 or an equivalent patch, then close this issue only after the fix lands on main with regression coverage.

## Reproduction Assessment

Yes. Source inspection directly matches the reported sequence: acquire `a`, acquire `b` with `maxTrackedKeys: 1`, prune the oldest active key, then `release('a')` becomes a no-op because the counter is gone.

## Solution Assessment

Yes, the linked fix direction is the narrow maintainable solution: reject brand-new keys when the active key table is full while preserving existing-key acquisition up to `maxInFlightPerKey`.
