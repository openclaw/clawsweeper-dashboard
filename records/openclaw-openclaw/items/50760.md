## Summary

Keep this PR open, but it is not merge-ready: the Control UI feature is useful, yet the branch is conflicting, lacks real behavior proof, and builds the board from sessions/cron instead of the current task ledger.

## What This Changes

The PR adds a read-only Control UI `/tasks` route with navigation/app-state wiring, task-board components, sessions and cron adapters, a loader service, and focused tests.

## Best Possible Solution

Rebase the feature around `tasks.list`/`TaskSummary` as the primary board source, keep sessions and cron as supplemental detail, route visible copy through i18n, and add redacted browser proof before merge.

## Reproduction Assessment

Yes for the review findings by source inspection: the loader omits `tasks.list`, error sessions receive `waitingForSec`, never-run cron health falls back to update time, and session-targeted cron can double-count. Full runtime reproduction of `/tasks` is not established because no real browser proof was provided.

## Solution Assessment

No. A Control UI Task Board is plausible, but the proposed sessions/cron-only reconstruction should be replaced with a ledger-first implementation that follows current task, refresh, and i18n contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the task ledger as the board source:** `ui/src/ui/task-board/service.ts:23-50`
  - body: `loadTaskBoard` builds the page only from `sessions.list`, `cron.list`, and `cron.runs`, but current main exposes durable task summaries through `tasks.list`. ACP, subagent, CLI, media, and other ledger-backed work can therefore be absent from a page presented as a Task Board, creating a competing source of truth for operators.
  - confidence: 0.91
- **[P2] Restrict waiting duration to waiting sessions:** `ui/src/ui/task-board/adapters/session-adapter.ts:138-140`
  - body: Failed, killed, timed-out, or aborted rows map to `status: "error"`, but this condition still sets `waitingForSec` for every non-running status. Those cards can render an error blocker plus a misleading waiting duration.
  - confidence: 0.94
- **[P2] Use next run time for never-run cron health:** `ui/src/ui/task-board/adapters/cron-adapter.ts:91-102`
  - body: For an enabled job without a recorded run, health falls back to `job.updatedAtMs`, so a daily or weekly job created more than an hour ago can show warning or stale even when its first `nextRunAtMs` is still in the future.
  - confidence: 0.88
- **[P2] Move Task Board strings into i18n:** `ui/src/ui/components/task-board/page.ts:43-67`
  - body: The new page hard-codes user-visible Chinese and English copy in the component instead of the Control UI locale source. That leaves other locales with mixed-language UI and bypasses the existing i18n contract.
  - confidence: 0.9
- **[P2] Exclude session-targeted cron runs from active cards:** `ui/src/ui/task-board/adapters/session-adapter.ts:129`
  - body: This filter only removes cron-namespace session keys. Cron jobs can target `current` or `session:<key>` and reuse an ordinary session key, so the same scheduled work can appear in both active and scheduled lanes and be mislabeled as independent active work.
  - confidence: 0.8
