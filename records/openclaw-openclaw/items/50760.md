## Summary

Keep open. Current main still does not have a Control UI Task Board route, but the branch is not ready to merge because it predates the task-ledger RPC contract, has source-provable UI mapping bugs, and lacks real behavior proof for the new route.

## What This Changes

The PR adds a read-only `/tasks` Control UI tab with navigation/state wiring, session and cron adapters, task-board components, and focused UI tests.

## Best Possible Solution

Refresh the board around `tasks.list`/`TaskSummary` as the primary source, use sessions and cron only as secondary facets where they add signal, move copy through Control UI i18n, fix the adapter bugs, and add redacted browser proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR diff against current-main contracts shows the `tasks.list` omission, incorrect waiting duration, cron health fallback, hard-coded copy, and cron/session double-counting path. I did not run the browser route, and the PR still lacks after-fix real behavior proof.

## Solution Assessment

No. A read-only Control UI Task Board is plausible, but this sessions/cron-only slice is no longer the narrowest maintainable design now current main exposes a durable task-ledger RPC contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the task ledger as the board source:** `ui/src/ui/task-board/service.ts:23-45`
  - body: Current main exposes `tasks.list`/`TaskSummary` as the durable background-task contract, but this loader only requests `sessions.list`, `cron.list`, and `cron.runs`. A general Task Board would omit non-cron ledger work such as ACP, subagent, CLI, and media tasks, so refresh the board around `tasks.list` and treat sessions/cron as secondary facets.
  - confidence: 0.9
- **[P2] Restrict waiting duration to waiting sessions:** `ui/src/ui/task-board/adapters/session-adapter.ts:138-141`
  - body: Failed, killed, timed-out, and aborted rows map to `status === "error"`, but this condition sets `waitingForSec` for every non-running status. Those cards would render an error blocker plus a misleading waiting duration; only populate this value for actual waiting rows.
  - confidence: 0.94
- **[P2] Base never-run cron health on its schedule:** `ui/src/ui/task-board/adapters/cron-adapter.ts:91-98`
  - body: For an enabled job with no recorded run, this falls back to `job.updatedAtMs`, so a daily or weekly job created more than an hour ago can show warning or stale even if its first `nextRunAtMs` is still in the future. Use schedule state for never-run jobs instead of treating definition age as degraded health.
  - confidence: 0.88
- **[P2] Move Task Board copy through i18n:** `ui/src/ui/components/task-board/page.ts:46-47`
  - body: The new page, toolbar, and cards hard-code visible Chinese and English strings directly in components, bypassing the Control UI locale source of truth. In non-Chinese locales this renders mixed-language UI, and generated locale bundles cannot pick up the new surface.
  - confidence: 0.9
- **[P2] Filter cron-targeted sessions from active cards:** `ui/src/ui/task-board/adapters/session-adapter.ts:129`
  - body: This filter only removes cron-namespace session keys. Cron jobs can target `current` or `session:<id>` sessions that reuse ordinary session keys, so the same scheduled work can appear in both lanes and be mislabeled as an independent active session.
  - confidence: 0.8
