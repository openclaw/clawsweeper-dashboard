## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard target-user workspace clears:** `src/agents/sandbox/user-backend.ts:227`
  - body: With `workspaceAccess: "rw"`, `targetDir` can be the configured target-user `workspaceDir`, and this command clears every child before upload. Because the path resolver accepts broad paths such as `~` or `/`, a misconfigured backend can delete the target user's home or root-level contents; require a dedicated contained workspace/root guard like the SSH backend before deleting.
  - confidence: 0.93
- **[P2] Use the validated exec command builder:** `src/agents/sandbox/user-backend.ts:141-145`
  - body: SSH validates model-generated shell with `buildValidatedExecRemoteCommand` before remote launch, but the new user backend calls the quote-only builder directly. That lets malformed placeholders or unbalanced shell that sibling backends reject run through `su`, so this backend should use the same validated construction and parity tests.
  - confidence: 0.88
- **[P2] Register a user backend workdir resolver:** `src/agents/sandbox/backend.ts:87-90`
  - body: Current main uses `resolveWorkdir` registrations so `ensureSandboxWorkspaceForSession` can report the sandbox `containerWorkdir` without starting a backend. The new `user` registration only installs a factory and manager, so after rebasing this backend needs a resolver or explicit coverage showing resolverless user sandboxes do not break prompt, media, and workspace-prep callers.
  - confidence: 0.78
