Full review comments:

- **[P1] Guard target-user workspace clears:** `src/agents/sandbox/user-backend.ts:227`
  - body: With `workspaceAccess: "rw"`, `targetDir` can be the configured target-user `workspaceDir`, and this command clears every child before upload. Because the resolver accepts broad paths such as `~` and `/`, a misconfigured backend can delete the target user's home or root-level contents; require a contained workspace/root guard before deleting.
  - confidence: 0.93
- **[P2] Validate exec commands before launching su:** `src/agents/sandbox/user-backend.ts:141-145`
  - body: Current SSH and OpenShell execution validates model-generated shell with `buildValidatedExecRemoteCommand` before remote launch, but the new user backend calls the quote-only builder directly. That lets malformed shell that sibling backends reject run through `su`, so use the same validated construction and parity tests.
  - confidence: 0.88
- **[P2] Register current-main user workdir and skills handling:** `src/agents/sandbox/backend.ts:87-90`
  - body: Current main uses backend workdir resolvers and materialized sandbox skills so prompt, media, and workspace-prep callers see the sandbox path without starting the backend. The new `user` registration only installs a factory and manager, and the backend never copies the materialized skills workspace, so rebase and add the same contracts before merge.
  - confidence: 0.82
- **[P1] Keep user helpers out of the public SDK until approved:** `src/plugin-sdk/sandbox.ts:31-48`
  - body: These user-backend-specific helpers would become public plugin SDK contract for privilege-switching execution. Remove the new SDK exports until maintainers approve this as a stable API, or include explicit API direction, docs, and compatibility tests for the new contract.
