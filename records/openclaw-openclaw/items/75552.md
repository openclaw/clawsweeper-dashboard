## Summary

Keep this PR open, but block merge/automerge on the current head. The PR is protected by MEMBER author association and the `maintainer` label, and the diff still has blocking functional issues: it introduces raw Pi `type: "message"` transcript appends without `parentId`, and parent-linked transcript hot paths still materialize the full file through `SessionManager.open()`.

## What This Changes

The PR replaces several gateway transcript reads with streaming/count/tail helpers, adjusts oversized `chat.inject` append behavior, updates session history/artifact/compaction paths and tests, and adds a changelog entry.

## Best Possible Solution

Keep the bounded transcript-read direction, but preserve the parentId write invariant for `chat.inject` and split read helpers so count, tail, first-page history, artifact scans, send sequence, subscription broadcasts, and manual compaction line tails never need full `SessionManager` materialization. Full active-branch traversal should remain reserved for callers that truly need complete branch context.

## Reproduction Assessment

Yes. The raw-append regression is reproduced by the existing server-methods guardrail test and public CI annotation, and the remaining OOM path is reproducible with a large current-format v3 JSONL transcript containing `id` and `parentId` entries that enters `visitSessionMessages()` and then `SessionManager.open()`.

## Solution Assessment

No. The PR is aimed at the right hot paths, but this implementation both violates the transcript write contract and leaves current-format parent-linked transcripts unbounded on the new shared helper path; the safer solution is a bounded JSONL traversal/append design that keeps parentId semantics intact.
