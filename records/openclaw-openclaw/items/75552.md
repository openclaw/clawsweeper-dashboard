## Summary

Keep this PR open, but not merge-ready. The maintainer/automerge PR is still useful because current main has the transcript hot paths it targets, but the PR head keeps parent-linked transcripts on an unbounded `SessionManager.open()` path inside the new shared helper, so large current-format transcripts can still hit the memory failure the PR is meant to prevent.

## What This Changes

The PR streams or tails gateway session transcript access for session detail, send, compact, artifacts, HTTP/SSE history, message broadcasts, chat.inject append behavior, regression tests, and the changelog.

## Best Possible Solution

Keep the bounded transcript direction, but split the helper contracts so count, tail, recent-message, artifact, subscription, send-sequence, and first-page history paths stream parent-linked JSONL safely while reserving full `SessionManager` traversal for complete branch-context callers; then re-review the exact automerge head with parent-linked stress coverage.

## Reproduction Assessment

Yes. A large v3 JSONL transcript containing non-session entries with `id` and `parentId` will take the PR's tree branch, and the pinned dependency source shows `SessionManager.open()` reads the entire transcript before returning the active branch.

## Solution Assessment

No. The PR is moving in the right direction, but this head is not the best complete fix because the shared bounded helper still has an unbounded parent-linked path; the safer alternative is separate streaming count/tail traversal with focused parent-linked regressions.
