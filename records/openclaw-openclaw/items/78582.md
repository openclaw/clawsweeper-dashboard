## Summary

Keep open. Current main still lacks this watchdog, but the PR is not mergeable as-is because it suppresses later stalls on reused session keys, misses the generated config baseline, lacks real behavior proof, and needs maintainer agreement on default-on force-finalization policy.

## What This Changes

The PR adds a configurable gateway subagent watchdog that force-idles diagnostic sessions classified as `blocked_tool_call` after a timeout and emits a new diagnostic event with docs, schema, OTel, changelog, and tests.

## Best Possible Solution

Revise the watchdog so suppression is scoped to a concrete run/tool/generation lifecycle, regenerate the config baseline, add redacted real gateway proof, and get maintainer approval for the default policy before merging.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection on current main shows stale tool-call sessions can be classified as `blocked_tool_call` with `recovery=none`, but the PR lacks after-fix runtime proof from a real stalled-subagent path.

## Solution Assessment

No, not as currently proposed. The heartbeat is a plausible implementation point, but this version suppresses future stalls on reused session keys, misses the config baseline, and needs maintainer approval for the default-on auto-abort policy.

## Review Findings

Full review comments:

- **[P2] Reset watchdog suppression for later stalls:** `src/logging/diagnostic.ts:1059-1067`
  - body: `watchdogAbortedSessions` is keyed by session key/id and only cleared by the test reset. Since lanes such as `main` are reused, the first watchdog abort permanently suppresses future watchdog aborts for that lane; a later `blocked_tool_call` can return to repeat `session.stalled` warnings. Scope suppression to a concrete run/tool/generation or clear it when the session makes new progress.
  - confidence: 0.86
