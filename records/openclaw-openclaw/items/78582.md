## Summary

Keep open: current main still has the noisy `blocked_tool_call` path, but this PR lacks real behavior proof and has merge-blocking lifecycle/config-baseline defects; the default-on auto-abort policy also needs maintainer approval.

## What This Changes

The PR adds a default-on `gateway.subagentWatchdog` config, diagnostic event/docs/schema coverage, and heartbeat logic/tests that force-idle a `blocked_tool_call` diagnostic session after a threshold.

## Best Possible Solution

Revise the watchdog so suppression is scoped to a concrete generation/run/tool lifecycle, regenerate the config baseline, add redacted real gateway proof, and get maintainer agreement on the default policy before merging.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does show current main can classify a stale active tool call as `blocked_tool_call` with `recovery=none`, but the PR lacks real after-fix runtime logs or artifacts.

## Solution Assessment

No. The heartbeat is a plausible implementation point, but this version suppresses future stalls on reused session keys, misses the required config baseline update, and needs proof plus maintainer approval for the default-on auto-abort behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reset watchdog suppression for later stalls:** `src/logging/diagnostic.ts:1065-1067`
  - body: `watchdogAbortedSessions` is keyed by the session key/id and is only cleared by the test reset. Since lanes such as `main` are reused, the first watchdog abort permanently suppresses future watchdog aborts for that lane; a later `blocked_tool_call` can return to repeat `session.stalled` warnings. Scope this to a concrete run/tool/generation or clear it when the session makes new progress.
  - confidence: 0.86
- **[P2] Regenerate the config docs baseline:** `src/config/zod-schema.ts:999-1004`
  - body: This adds a new `gateway.subagentWatchdog` config surface, but the PR does not update `docs/.generated/config-baseline.sha256`. The repo's `config:docs:check` gate treats intentional schema/help changes as baseline drift until `pnpm config:docs:gen` is run and the updated hash is committed.
  - confidence: 0.9
