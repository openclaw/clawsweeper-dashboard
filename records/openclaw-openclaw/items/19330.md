## Label Justifications

- `P2`: The issue has source-level evidence for a real privacy and correctness gap, but the fix needs design work rather than an urgent narrow regression patch.
- `impact:security`: The reported behavior lets media intended for one agent or trust boundary remain readable through the shared inbound media pool.

## Summary

Keep open: current main and v2026.6.10 still leave inbound media in shared flat paths for representative channel downloads, reject scoped managed inbound references, and flatten sandbox staging, so the core privacy/correctness problem remains. The related Feishu and channel-namespace issues are narrower slices, and the broad implementation PR was closed unmerged.

## What This Changes

The issue asks for scoped inbound media storage so media received for one agent or account is not shared with other agents through the global inbound media pool.

## Best Possible Solution

Keep this as the canonical cross-channel media-isolation tracker and design a maintainer-approved scoped inbound-media contract covering save paths, managed references, sandbox staging, cleanup, docs, compatibility, and regression tests.

## Reproduction Assessment

Yes at source level: current main still saves representative inbound media into a shared inbound bucket, rejects scoped managed refs, and stages sandbox media into flat media/inbound/<file> paths. I did not run a live multi-agent channel deployment, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear as proposed: account-id subdirectories are a plausible input, but a save-time-only change would leave references, staging, cleanup, compatibility, and explicit sharing semantics incomplete. The best fix is a maintainer-approved scoped media contract across the pipeline.
