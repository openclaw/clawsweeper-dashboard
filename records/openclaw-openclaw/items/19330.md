## Summary

Keep open: current main and the latest release still converge representative inbound media into a shared flat `media/inbound` store, while read-side references and sandbox staging still assume flat paths. The closed implementation PR was not merged, and the open Feishu item is only a narrower account-specific slice of this broader security/design problem.

## What This Changes

Request scoped inbound media storage and staging so media received by one agent or account is not exposed to other agents in multi-agent deployments.

## Best Possible Solution

Define a backward-compatible scoped inbound-media contract across storage, channel/plugin saves, managed URI and absolute-path resolution, local media roots, sandbox staging, cleanup, docs, regression tests, and explicit sharing/access rules.

## Reproduction Assessment

Yes, at source level. Current main still saves representative channel downloads into the shared inbound bucket while rejecting scoped managed refs and flattening staged sandbox paths; I did not run a live multi-agent transport repro.

## Solution Assessment

No, the proposed save-time account subdirectory is not sufficient by itself. It may be one input, but the maintainable fix needs a scoped managed-media contract that covers read paths, staging, cleanup, compatibility, local-root policy, and explicit sharing.
