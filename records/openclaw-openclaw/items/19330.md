## Summary

Keep open. Current main still stores representative inbound media in a shared managed `inbound` bucket, still rejects scoped managed inbound references, still stages sandbox media into flat `media/inbound/<file>` paths, and the direct implementation attempt was closed unmerged while the Feishu item remains only a narrower slice.

## What This Changes

Request scoped inbound media storage so media received by one agent cannot be read or staged by other agents in multi-agent deployments.

## Best Possible Solution

Define a backward-compatible scoped inbound-media contract across storage, SDK/runtime callers, managed references, local access roots, sandbox staging, cleanup, docs, and regression tests, with explicit sharing/access policy.

## Reproduction Assessment

Yes, at source level. Configure multiple agents or channel accounts that receive media, and current main still routes representative saves into shared `media/inbound` while read and staging paths assume first-level flat files.

## Solution Assessment

No. The proposed account-subdirectory approach is a useful starting point, but the maintainable fix needs a scoped managed-media contract across storage, references, local access policy, cleanup, staging, and channel/plugin callers.
