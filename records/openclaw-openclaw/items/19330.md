- `impact:security`: The issue concerns sensitive inbound media being readable across agent or account trust boundaries.

## Summary

Keep open: current main and v2026.5.20 still preserve a flat shared inbound-media store, flat managed references, and flat sandbox staging; the open Feishu item is only a narrower slice and the prior implementation PR closed unmerged.

## What This Changes

The issue requests scoped inbound channel media storage so one agent/account cannot read another agent's received files from the shared media pool.

## Best Possible Solution

Define a backward-compatible scoped inbound-media contract across storage, bundled/plugin save paths, managed URI and path resolution, local access roots, sandbox staging, cleanup, docs, regression tests, and explicit sharing/access policy.

## Reproduction Assessment

Yes, at source level: current main still writes representative channel media to shared inbound locations, rejects scoped managed refs, exposes global media roots, and flattens staged sandbox paths. I did not run live multi-agent channel traffic in this read-only review.

## Solution Assessment

No, the proposed save-time account subdirectory is not sufficient by itself. It can be part of the fix, but the maintainable solution needs a scoped managed-media contract that also covers read paths, local roots, staging, cleanup, compatibility, docs, and explicit sharing.
