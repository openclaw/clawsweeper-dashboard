## Summary

Keep open. Current main still stores representative inbound media in a shared managed `inbound` bucket, read/staging paths still assume flat first-level inbound files, and the direct implementation PR was closed unmerged while the Feishu follow-up is only a narrower slice.

## What This Changes

Request per-agent or per-account isolation for inbound media so files received by one agent or channel account are not stored in a shared pool visible to other agents.

## Best Possible Solution

Define a backward-compatible scoped inbound-media contract across storage, SDK/runtime callers, managed references, access checks, sandbox staging, cleanup, docs, and regression tests, with maintainer direction on scope and explicit sharing.

## Reproduction Assessment

Yes, at source level. Configure multiple agents or channel accounts that receive media, and current main still routes representative saves into shared `media/inbound` while read and staging paths assume first-level flat files.

## Solution Assessment

No. The proposed account-subdirectory approach is a useful starting point, but the maintainable fix needs a scoped managed-media contract across storage, references, local access policy, cleanup, staging, and channel/plugin callers.
