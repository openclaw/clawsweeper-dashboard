## Label Justifications

- `P2`: This is a credible multi-agent privacy and correctness gap with a bounded but cross-cutting media pipeline surface, not an all-deployment emergency.
- `impact:security`: The issue is about sensitive inbound media being readable across agent, account, or tenant trust boundaries through shared managed media storage.
- `impact:session-state`: An agent can discover or process media associated with another agent, account, or conversation context.

## Summary

Keep open: current main and the latest release still use a shared flat inbound media namespace for representative save paths, reject scoped managed inbound refs, and flatten sandbox staging, so the broad cross-agent media isolation problem remains unsolved.

## What This Changes

The issue requests per-agent or per-account inbound media isolation so media received by one agent cannot be read or staged by another agent through the shared inbound media store.

## Best Possible Solution

Define one scoped managed inbound-media contract and apply it consistently across save paths, managed refs, local media authorization, sandbox staging, cleanup, docs, and tests.

## Reproduction Assessment

Yes, at source level: current main still saves representative inbound media to the shared `inbound` bucket, rejects scoped managed refs, and stages media into flat `media/inbound/<basename>` paths. I did not run a live multi-agent channel deployment in this read-only review.

## Solution Assessment

Unclear as proposed: account-id subdirectories are plausible, but the best fix needs a maintainer-approved scoped-media contract that covers read paths, staging, cleanup, local-root authorization, compatibility, and docs together.
