- `impact:security`: The issue concerns inbound media being readable across agent/account trust boundaries.
- `impact:session-state`: A shared inbound pool can cause media from one agent/account context to be associated with another agent's turn or session.

## Summary

Keep open: current main and v2026.5.22 still use a shared flat inbound media store, scoped managed references and sandbox staging are missing, the Feishu follow-up is narrower, and the earlier implementation PR closed unmerged.

## What This Changes

Request per-agent or per-account isolation for inbound media so files sent to one agent are not visible or processed through another agent's shared inbound store.

## Best Possible Solution

Define a backward-compatible scoped inbound-media contract across storage, bundled and plugin save paths, managed references, local-root policy, sandbox staging, cleanup, docs, regression tests, and explicit sharing/access policy.

## Reproduction Assessment

Yes, at source level: current main still writes representative inbound media into a shared bucket, rejects scoped managed inbound references, includes global media roots, and flattens sandbox staging. I did not run live multi-agent channel traffic in this read-only review.

## Solution Assessment

No, the proposed save-time account subdirectory is not sufficient by itself. It can be part of the fix, but the maintainable solution needs a scoped managed-media contract that also covers read paths, access checks, staging, cleanup, compatibility, docs, and explicit sharing.
