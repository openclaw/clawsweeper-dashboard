## Summary

Keep open: current main and v2026.6.6 still leave representative inbound media in the shared managed inbound bucket, and the open Feishu item is a narrower slice rather than a replacement for this cross-channel security tracker.

## What This Changes

The issue requests per-agent or per-account isolation for inbound media so one agent cannot read media intended for another agent.

## Best Possible Solution

Define a managed scoped inbound-media contract that preserves scope through channel saves, plugin SDK media helpers, managed refs, local-root authorization, sandbox staging, cleanup, docs, and tests, with maintainer-selected identity semantics.

## Reproduction Assessment

Yes, at source level: current main still saves representative inbound media to the shared `inbound` bucket and rejects scoped managed refs. I did not run a live multi-agent channel deployment.

## Solution Assessment

Unclear as proposed: account-id subdirectories are plausible, but the best fix needs a maintainer-approved scoped-media contract that covers read paths, staging, cleanup, local-root authorization, compatibility, and docs together.
