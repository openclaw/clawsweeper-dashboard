## Label Justifications

- `P2`: The issue describes a credible multi-agent privacy and correctness gap with real but non-emergency blast radius.
- `impact:security`: Inbound media intended for one agent can be reachable by another agent that has filesystem access to the shared media root.
- `impact:session-state`: The report is about media context crossing agent/session boundaries in multi-agent deployments.

## Summary

Keep open: current main and v2026.6.5 still save representative inbound media into a shared flat inbound store, reject scoped inbound references, and stage sandbox media as flat `media/inbound/<basename>`; the prior implementation PR is closed unmerged and the open Feishu item is only a narrower slice.

## What This Changes

The issue requests per-agent or per-account isolation for inbound media so agents cannot read media intended for another agent.

## Best Possible Solution

Keep this as the canonical cross-channel media-isolation tracker and design a managed scoped inbound-media contract that covers saves, references, local access, sandbox staging, cleanup, docs, and compatibility for existing flat files.

## Reproduction Assessment

Yes, at source level: current main still saves representative inbound media into the shared `inbound` bucket, rejects scoped inbound references, and stages files into a flat sandbox path. I did not run live multi-agent Discord/Telegram/Signal traffic.

## Solution Assessment

No, the proposed save-time account subdirectory is only a partial fix. The maintainable solution needs a scoped managed-media contract that also covers read-side URI/path resolution, local access checks, sandbox staging, cleanup, compatibility, and explicit sharing semantics.
