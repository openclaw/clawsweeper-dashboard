## Summary

Keep open: current main and latest release still use shared flat inbound media storage and flat sandbox staging, while the related Feishu item is only a narrower account-specific slice of this broader media-isolation security tracker.

## What This Changes

The issue requests scoped inbound media storage so files received for one agent cannot be read, resolved, or staged by another agent.

## Best Possible Solution

Define a managed scoped inbound-media contract that preserves and authorizes scope through channel saves, plugin SDK media helpers, managed refs, local-root authorization, sandbox staging, cleanup, docs, and tests, with maintainer-selected identity semantics.

## Reproduction Assessment

Yes, at source level: current main still saves representative inbound media to the shared `inbound` bucket, rejects scoped managed refs, and stages media into flat `media/inbound/<basename>` paths. I did not run a live multi-agent channel deployment in this read-only review.

## Solution Assessment

Unclear as proposed: account-id subdirectories are plausible, but the best fix needs a maintainer-approved scoped-media contract that covers read paths, staging, cleanup, local-root authorization, compatibility, and docs together.
