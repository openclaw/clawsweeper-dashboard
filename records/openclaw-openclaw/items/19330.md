## Summary

Keep open: current main still leaves the central media-isolation problem unsolved, with representative inbound saves using the shared `inbound` bucket, scoped managed references rejected, and sandbox staging flattened to `media/inbound/<file>`. The closed implementation PR did not merge, and the open Feishu item is a narrower slice rather than a replacement for this cross-channel tracker.

## What This Changes

Request per-agent or account-scoped isolation for inbound media so files received by one agent are not readable or staged for another.

## Best Possible Solution

Keep this as the canonical core media-isolation tracker and define a managed scoped inbound-media contract that carries scope through writes, references, local-root checks, sandbox staging, cleanup, docs, and regression or live proof; keep the Feishu issue as the narrower account-specific slice unless the generic fix fully covers it.

## Reproduction Assessment

Yes, at source level: current main still routes representative inbound saves into the shared `inbound` bucket and flat-stages media into `media/inbound/<file>`. I did not run a live multi-agent channel setup.

## Solution Assessment

Unclear as proposed: account-ID-only save scoping is plausible but not sufficient by itself. The best fix needs maintainers to choose the isolation identity and carry that contract through references, staging, local-root policy, cleanup, docs, and tests.
