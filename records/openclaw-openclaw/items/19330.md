## Label Justifications

- `P2`: This is a real security and correctness improvement with source-level evidence, but it needs maintainer design rather than an urgent narrow regression fix.
- `impact:security`: The issue concerns cross-agent access to sensitive inbound files through a shared media directory.
- `impact:session-state`: Shared inbound media can be staged into the wrong agent/session context and affect which files an agent sees.

## Summary

Keep open: current main and the latest release still use a shared flat inbound media store, scoped `media://inbound/...` refs remain unsupported, sandbox staging still flattens media into `media/inbound/<file>`, and the only broad implementation attempt was closed unmerged while the open Feishu item is only a narrower slice.

## What This Changes

Request to isolate inbound media per agent or account so media received by one agent cannot be read or staged by another agent in multi-agent deployments.

## Best Possible Solution

Keep this as the canonical cross-channel media-isolation tracker and design a scoped inbound-media contract that covers save paths, managed references, sandbox staging, cleanup, docs, and regression tests.

## Reproduction Assessment

Yes at source level: current main saves representative inbound channel media into the shared `inbound` bucket, rejects scoped managed refs, and stages sandbox media into flat `media/inbound/<file>` paths. I did not run a live multi-agent channel deployment, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear as proposed: account-id subdirectories are a plausible start, but a save-time-only change would leave managed refs, staging, cleanup, compatibility, and explicit sharing semantics incomplete. The best fix is a maintainer-approved scoped media contract across the media pipeline.
