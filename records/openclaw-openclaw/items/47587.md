## Summary

Keep open. Current main preserves Telegram album media paths and supports opt-in multi-image understanding, but it still mirrors local paths into URL metadata and renders duplicate `path | path` media-header entries.

## What This Changes

The issue requests Telegram album image attachments to appear once in the media header and all configured image attachments to be described.

## Best Possible Solution

Keep the existing configurable first/all media-understanding policy, de-duplicate identical local path/url media-header rendering, and keep users pointed to `tools.media.image.attachments` with `mode: "all"` when they want every album image described.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by constructing a Telegram album or context with identical local `MediaPaths` and `MediaUrls`; current formatter output includes duplicated `path | path` header entries.

## Solution Assessment

Yes for a narrow repair. De-duplicating identical local path/url rendering is the maintainable fix, while changing the shared first-image default is unnecessary because multi-image processing already has documented config.
