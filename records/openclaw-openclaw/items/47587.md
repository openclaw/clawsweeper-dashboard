## Label Justifications

- `P2`: The issue affects Telegram album prompt context for media users, but it is bounded to media prompt assembly rather than a core outage.
- `impact:message-loss`: Some user-sent image context can be absent or misleading in the model-visible prompt when only one image is described and the rest are listed with duplicated local references.
- `impact:session-state`: Conversation history can preserve incomplete visual context, making later session state diverge from what the user actually sent.

## Summary

Keep open. Current main has already fixed the part where described images were removed from the media header, but it still source-reproduces the duplicate `path | path` formatter bug, and the open closing PR is the active narrow fix path.

## What This Changes

The issue asks OpenClaw to handle Telegram multi-image messages consistently by listing every image once in the media header and ensuring image descriptions cover the intended attachments.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/77279 for the duplicate identical path/url header, while keeping all-image transcription as the existing documented attachment-policy configuration unless maintainers choose a separate default change.

## Reproduction Assessment

Yes, at source level. Passing equal `MediaPaths` and `MediaUrls` into `buildInboundMediaNote` follows the same shape Telegram currently creates and renders duplicated `path | path` header lines on current main.

## Solution Assessment

Yes for the narrow duplicate-header bug: the shared formatter should dedupe identical sanitized path/url values. The request to transcribe every image by default is a separate product choice because `tools.media.image.attachments` already supports `mode: "all"`.
