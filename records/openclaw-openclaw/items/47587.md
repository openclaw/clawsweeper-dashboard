- `P2`: The issue affects Telegram album prompt context for media users, but it is bounded to media prompt assembly rather than a core outage.
- `impact:message-loss`: Some image context is effectively absent from the model-visible prompt because only one image is described while others are only listed as attachments.

## Summary

Keep open. Current main still source-reproduces the duplicated Telegram media header, and the linked open PR remains the active formatter fix path; the broader request to describe every album image by default is a separate media-understanding policy choice because all-image processing is already configurable.

## What This Changes

The issue reports that Telegram multi-image messages describe only the first image while the remaining images appear as duplicated `path | path` media-header entries.

## Best Possible Solution

Refresh and land or replace https://github.com/openclaw/openclaw/pull/77279 for the duplicate path/url formatter bug, while keeping all-image transcription as the documented attachment-policy configuration unless maintainers choose a separate default change.

## Reproduction Assessment

Yes, at source level. Passing equal `MediaPaths` and `MediaUrls` into `buildInboundMediaNote` reproduces the duplicated `path | path` header, and Telegram currently maps each media item to equal path/url values before that formatter runs.

## Solution Assessment

Yes for the narrow duplicate-header bug: the shared formatter should dedupe identical sanitized path/url values. Changing the first-image default to all images is a separate product decision because `tools.media.image.attachments` already supports `mode: "all"`.
