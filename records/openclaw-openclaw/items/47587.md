## Summary

Keep open. Current main preserves all Telegram album media paths, but the reported behavior is not fully resolved: Telegram still mirrors local paths into URL fields, the prompt media note still renders duplicated `path | path` text, and shared image understanding still defaults to selecting only the first image unless configured otherwise. The related closed Telegram media-group reports cover path-loss/fetch failure pieces, not this remaining header and all-image understanding behavior.

## What This Changes

The issue requests that Telegram album messages expose every downloaded image once in the media header and run image understanding for every image.

## Best Possible Solution

Make Telegram album behavior explicit and covered: preserve every album media path once, suppress URL rendering when the URL value is the same local path, and either intentionally opt Telegram image albums into all-image understanding with regression tests or document/configure `tools.media.image.attachments.mode = "all"` as the supported behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by constructing a Telegram album with multiple image paths: current context assembly creates parallel identical `MediaPaths`/`MediaUrls`, the prompt media note renders both values, and the shared selector chooses only the first image under default policy.

## Solution Assessment

Unclear. Header de-duplication is a narrow maintainable fix, but all-image transcription by default needs maintainer policy judgment because it would diverge from the documented shared `first`/`maxAttachments: 1` media-understanding default.
