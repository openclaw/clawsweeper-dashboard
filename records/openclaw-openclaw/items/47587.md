## Label Justifications

- `P2`: The issue affects Telegram album media prompts for users sending multiple images, but the failure is bounded to media context assembly rather than a core outage.
- `impact:message-loss`: Some image context is effectively lost from the model-visible prompt because only one image is described while the remaining images are only listed as attachments.
- `impact:session-state`: Conversation history can preserve incomplete visual context, causing later session state to drift from what the user actually sent.

## Summary

Keep open. Current main still has the source-reproducible duplicate `path | path` media header when Telegram mirrors local media paths into URLs, and the open closing PR https://github.com/openclaw/openclaw/pull/77279 remains the active fix path; changing the default from first-image understanding to all-image understanding is a separate product decision.

## What This Changes

The issue requests that Telegram album images all be included once in the media header and all be processed by image understanding instead of only the first image being described.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/77279 for the duplicate path/url formatter bug, then treat all-image transcription defaults as a separate media-understanding policy decision.

## Reproduction Assessment

Yes, at source level. Passing equal `MediaPaths` and `MediaUrls` into `buildInboundMediaNote` reproduces the duplicate `path | path` header shape that Telegram creates by setting both fields to `media.path`.

## Solution Assessment

Yes for the narrow duplicate-header bug: the shared formatter should dedupe identical sanitized path/url values. Changing the first-image default to all images is not the same fix because `mode: "all"` already exists and changing the default needs product review.
