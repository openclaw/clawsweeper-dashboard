## Summary

Keep open. Current main preserves all Telegram album media paths and the shared image-understanding policy already supports opt-in multi-image processing, but Telegram still mirrors local paths into URL metadata and the media-note formatter still renders duplicated `path | path` header entries.

## What This Changes

The issue requests Telegram album messages to include every image once in the media header and process every image through configured image understanding.

## Best Possible Solution

Keep the existing configurable first/all media-understanding policy, fix duplicate local path/url media-note rendering, and point users who need all album images described to `tools.media.image.attachments` with `mode: "all"`.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by constructing a Telegram album with multiple local image paths: current context assembly mirrors paths into `MediaUrls`, and the media-note formatter renders both path and URL.

## Solution Assessment

Yes for a narrow repair. De-duplicating identical local path/url rendering is maintainable, while changing the shared first-image default is unnecessary because multi-image processing already has documented config.
