## Label Justifications

- `P2`: The issue affects Telegram/media prompt context for users sending albums, but it is bounded to media prompt assembly rather than a core outage.
- `impact:message-loss`: Some user-sent image context can be absent or misleading in the model-visible prompt when only one image is described and remaining media lines duplicate local references.
- `impact:session-state`: Conversation history can preserve incomplete visual context, causing later session state to diverge from what the user actually sent.

## Summary

Keep open. Current main still source-reproduces the duplicated `path | path` media header, while the broader request to describe every image by default is a documented attachment-policy choice and an open closing PR already carries the narrow formatter fix.

## What This Changes

Telegram album messages can produce incomplete or misleading media prompt context: one image may be described while other image paths are listed with duplicated local path/url text.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/77279 for the duplicate identical path/url header, while keeping all-image descriptions on the existing `tools.media.image.attachments` configuration path unless maintainers choose a separate default change.

## Reproduction Assessment

Yes, at source level. Passing equal `MediaPaths` and `MediaUrls` into `buildInboundMediaNote` follows the Telegram/shared payload shape and renders duplicated `path | path` header lines on current main.

## Solution Assessment

Yes for the narrow duplicate-header bug: the shared formatter should dedupe identical sanitized path/url values. Changing all-image transcription defaults is not the same fix because current docs already expose `mode: "all"` and the default policy needs maintainer approval.
