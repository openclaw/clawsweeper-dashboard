## Summary

Keep open: current main and the latest release still hash volatile exec output into no-progress detection, and the linked open pull request is the active fix path rather than a reason to close this issue early.

## What This Changes

The issue reports that repeated exec calls with slightly varying output do not accumulate no-progress streaks, so generic repeat critical detection and the global circuit breaker fail to stop runaway tool loops.

## Best Possible Solution

Land a narrow, tested fix for exec no-progress fingerprinting, then track broader volatility or similarity-based loop-detection design separately if maintainers want that scope.

## Reproduction Assessment

Yes, source-reproducible: current main hashes completed/failed exec output and resets `noProgressStreak` on any different result hash. I did not run tests because this review was read-only, but the source path directly matches the reported failure.

## Solution Assessment

Yes for the central bug: a narrow exec progress-fingerprint fix with focused regression tests is the best immediate repair. The broader similarity/volatility registry ideas are separable follow-up design work, not a prerequisite to keep this issue actionable.
