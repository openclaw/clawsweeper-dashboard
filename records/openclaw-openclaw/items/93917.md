## Summary

Keep open: current main and the latest release still hash failed `exec` output into the no-progress result hash, so slightly varying failed output can still reset the streak before critical or global blocking. The linked closing PR is open and unmerged, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that repeated failed `exec` calls with identical arguments can avoid `genericRepeat` critical and global circuit-breaker escalation when the result text varies slightly.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94050, or replace it with an equivalent narrow fix that strips volatile failed-exec output while preserving completed exec output as progress.

## Reproduction Assessment

Yes, source-level. Current main hashes failed exec output into `resultHash`, and `getNoProgressStreak` breaks when that hash changes; I did not run a live Gateway replay in this read-only review.

## Solution Assessment

Yes, the linked PR's completed/failed exec split is the narrowest maintainable fix for this issue. Broader fuzzy matching or volatility registries may be useful follow-up work, but they are not required to fix the reported failed-exec volatility path.
