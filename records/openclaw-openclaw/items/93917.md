## Summary

Keep open: current main and v2026.6.8 still include exec output in completed/failed no-progress hashes, so slightly varying failed exec output can prevent critical/global escalation. The linked PR is the active candidate fix, but it is open and has a current review blocker around completed exec output semantics.

## What This Changes

The issue reports that repeated exec loops with slightly varying result output do not escalate from warning to critical or global circuit-breaker blocking because no-progress hashing requires exact result hashes.

## Best Possible Solution

Land or replace the linked candidate with a narrow failed-exec no-progress fingerprint fix, keep completed exec output as progress, and track broader similarity/volatility-registry design separately if maintainers want it.

## Reproduction Assessment

Yes. Source inspection shows current main hashes completed/failed exec output into `resultHash`, while `getNoProgressStreak` breaks when hashes differ; I did not run tests because this review is read-only.

## Solution Assessment

Yes for the central bug direction. A narrow failed-exec progress-fingerprint fix is the maintainable immediate repair; broader fuzzy matching should be a separate design discussion.
