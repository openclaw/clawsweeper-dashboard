## Summary

Keep open: current main and v2026.6.9 still hash completed/failed exec output into the no-progress result hash, so slightly varying failed exec output can still reset the streak before critical/global blocking. The linked candidate fix is open and unmerged, so the issue should stay open until that path lands or is replaced.

## What This Changes

Make loop detection escalate repeated exec calls whose status and exit state are unchanged even when output text varies slightly.

## Best Possible Solution

Land a narrow fix that strips volatile failed exec output from no-progress fingerprints while preserving completed exec output as progress, with focused regression tests and direct after-fix proof.

## Reproduction Assessment

Yes. Source inspection shows current main hashes failed exec output into resultHash, while getNoProgressStreak breaks when hashes differ; no live replay was run because this review is read-only.

## Solution Assessment

No fix has landed yet. The best path is to review and land a narrow linked fix, or replace it with an equivalent patch that fixes failed exec volatility without treating changing completed output as no progress.
