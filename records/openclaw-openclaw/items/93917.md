## Summary

Keep open: current main and v2026.6.9 still hash failed exec output into the no-progress result hash, so slightly varying failed exec output can keep critical/global loop detection at warning-only behavior. The linked PR is the active fix candidate, but it is open and currently has review concerns around completed exec semantics and proof.

## What This Changes

The issue reports that repeated exec calls with slightly varying outputs do not reach critical or global loop-detection thresholds.

## Best Possible Solution

Land or replace the linked candidate with a narrow failed-exec no-progress fingerprint fix, keep completed exec output as progress, and track broader similarity or volatility-registry design separately.

## Reproduction Assessment

Yes. Source and current tests show failed exec output drift changes result hashes, and the critical/global paths require exact no-progress streaks; I did not run tests because this review is read-only and tests can create artifacts.

## Solution Assessment

Yes for the immediate bug direction. A narrow failed-exec progress-fingerprint fix is the maintainable repair; the broader fuzzy matching and volatility registry ideas need separate product/design review.
