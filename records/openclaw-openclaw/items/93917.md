## Summary

Keep open: current main and v2026.6.9 still hash completed/failed exec output into the no-progress fingerprint, so slightly varying failed exec output can reset the streak before the critical or global breaker fires. The linked closing PR is open and unmerged, so this issue should stay open until that path lands or is replaced.

## What This Changes

The issue asks OpenClaw to make genericRepeat critical/global circuit breaking fire for repeated exec commands whose failed outputs vary slightly.

## Best Possible Solution

Land a narrow loop-detection fix that treats volatile failed exec output as no-progress noise while preserving completed exec output as progress, with focused regression coverage and after-fix proof.

## Reproduction Assessment

Yes. Source inspection shows current main hashes failed exec output into `resultHash`, while `getNoProgressStreak` breaks when hashes differ; no live replay was run because this review is read-only.

## Solution Assessment

Yes for the narrow failed-exec hash split; a broader similarity or volatility registry may be follow-up work, but the best immediate fix is to repair failed exec no-progress fingerprints without changing completed-output progress semantics.
