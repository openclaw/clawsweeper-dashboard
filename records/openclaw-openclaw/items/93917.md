## Summary

Keep open: current main and the latest release still hash completed/failed exec output into the no-progress result hash, so slightly varying failed exec output can keep critical/global loop detection at warning-only behavior. The linked fix PR is open and unmerged, so this issue should remain open until that PR or a narrower replacement lands.

## What This Changes

Requests fixing tool-loop detection so repeated exec calls with slightly varying results still trip genericRepeat critical and global circuit-breaker thresholds.

## Best Possible Solution

Land or replace the linked candidate with a narrow failed-exec progress fingerprint fix that strips volatile failure output while preserving completed output as progress; track broader similarity or volatility-registry design separately.

## Reproduction Assessment

Yes, source-level. Current main hashes completed/failed exec output into `resultHash`, while `getNoProgressStreak` breaks on changed hashes; I did not run tests because this review is read-only.

## Solution Assessment

Yes for the central bug direction. A narrow failed-exec progress-fingerprint fix is the maintainable immediate repair; broader fuzzy matching should be a separate design discussion.
