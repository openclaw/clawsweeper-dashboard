## Summary

Keep open: current main and v2026.6.8 still include exec output in completed/failed outcome hashes, while critical and global loop blocking still require exact result-hash equality. The open linked fix PR is the active candidate, not a reason to close this issue before merge.

## What This Changes

The issue reports that repeated exec calls with slightly varying output do not accumulate no-progress streaks, so critical loop detection and the global circuit breaker do not fire.

## Best Possible Solution

Land or replace the linked PR with a narrow, tested exec no-progress fingerprint fix that strips volatile failed-output noise while preserving completed-output progress, then close this issue with the merged commit and release provenance.

## Reproduction Assessment

Yes, source-reproducible: current main hashes completed/failed exec output, then breaks the no-progress streak when result hashes differ, so varied exec output can keep the critical/global paths from reaching their thresholds. I did not run tests because this review is read-only and must not create checkout artifacts.

## Solution Assessment

Yes for keeping this issue open: the narrow maintainable path is to repair the exec outcome fingerprint in the linked PR or a replacement, not to close before merge. The broader similarity-registry proposal is a separate design follow-up, not required to resolve the central failed-exec bug.
