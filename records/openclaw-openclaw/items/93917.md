## Summary

Keep open: current main and the latest release still hash failed `exec` output into the no-progress result, so the reported critical/global breaker miss remains real. The issue should stay open while the linked implementation PRs are reviewed or replaced, especially https://github.com/openclaw/openclaw/pull/94068 and https://github.com/openclaw/openclaw/pull/94050.

## What This Changes

The issue reports that `genericRepeat` critical blocking and the global circuit breaker never fire for repeated failed `exec` calls when each result has slightly different output text.

## Best Possible Solution

Land a narrow loop-detection fix that excludes volatile failed-exec output while preserving stable failure metadata, adds focused regression coverage, and then close this issue; treat the broader volatility-registry or similarity proposal as separate follow-up design work if maintainers want it.

## Reproduction Assessment

Yes by source inspection: a repeated failed `exec` call with the same args and varying `aggregated` or text output will produce different result hashes, so `noProgressStreak` will not reach the critical/global thresholds. I did not run a local reproducer because this review was read-only.

## Solution Assessment

No fix is on main yet. The best fix is a narrow failed-exec progress fingerprint with regression tests, not closing the issue or broadening it into a fuzzy-match architecture change before the immediate blocker lands.
