## Summary

Keep open: current main and v2026.6.9 still include exec output in the completed/failed result hash, and the no-progress streak still stops when that hash varies. The open linked PR is the right candidate path to review or replace, but it has not merged.

## What This Changes

The issue asks OpenClaw to make generic repeat critical blocking and the global loop breaker fire for repeated exec failures whose output text varies slightly.

## Best Possible Solution

Land a narrow fix that strips volatile failed exec output from no-progress fingerprints while preserving completed exec output as progress, with focused regression coverage and direct after-fix proof.

## Reproduction Assessment

Yes, source-reproducible: repeated failed exec calls with identical args but varying output still produce different result hashes, and `getNoProgressStreak` stops when hashes differ. I did not run a live replay or tests because this review is read-only.

## Solution Assessment

Yes, the best current path is a narrow failed-exec volatility fix through the linked PR or an equivalent replacement. A fuzzy comparison or volatility registry would be broader follow-up work, not required to solve this central report.
