## Summary

Keep open: current main and the latest release still hash exec completed/failed output into the no-progress fingerprint, and the open linked fix PR has not merged.

## What This Changes

The issue reports that repeated `exec` calls with identical args but slightly varying results reach warning-only detection and never reach the critical or global circuit-breaker thresholds.

## Best Possible Solution

Land or replace the focused linked PR with a stable exec no-progress fingerprint and regression tests, then close this issue with the merged commit and release provenance.

## Reproduction Assessment

Yes, source-level: current main hashes completed/failed exec output and `getNoProgressStreak` breaks on any changed result hash, so a same-command failed exec loop with varying output stays below critical/global thresholds. I did not run tests because this review was read-only.

## Solution Assessment

Yes, a focused detector-layer exec outcome fingerprint is the maintainable fix because the before-tool-call wrapper already blocks critical detector results. Broader fuzzy similarity or default-on behavior should be tracked separately unless maintainers explicitly choose that direction.
