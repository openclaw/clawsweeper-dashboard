## Summary

Keep open. Current main has partial mitigations for common `chat.history` and bulk `sessions.list` paths, but the reported blocking class remains in synchronous full-transcript reads and full synchronous scans used by agent prep, cursor history, session get/send, session events, artifacts, status, and TUI history paths. The Linux and macOS follow-up comments add credible independent reproductions, including the second usage-fallback path.

## What This Changes

The issue requests making Gateway/session transcript reads non-blocking or bounded so large session stores do not stall WebSocket handshakes, Telegram, TUI, and session/history RPCs.

## Best Possible Solution

Land a focused Gateway/session performance fix that removes synchronous full-transcript materialization from latency-sensitive request, event, and agent-prep paths. Use cached sequence/usage metadata or bounded tail readers for hot paths, reserve full transcript traversal for explicit historical/export operations, and keep the physical archive cleanup tracked separately by #75658/#75672.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: create a large session JSONL with missing stored usage/sequence metadata, then call `sessions.get`, cursor-based HTTP history, `sessions.send`, a session message broadcast with subscribers, TUI `loadHistory`, or agent prompt-token estimation; current main still reaches synchronous full reads or scans on those paths.

## Solution Assessment

Yes for the issue's direction, but a wholesale async conversion of one helper is not enough. The best repair should combine bounded readers, cached message/usage metadata, and targeted async/offloaded full traversal while preserving transcript branch and pagination semantics.

## Repair Work Prompt

Fix the remaining Gateway/session transcript event-loop blocking reported in https://github.com/openclaw/openclaw/issues/75656. Current main already has partial mitigations for first-page `chat.history` and bulk `sessions.list`, so do not implement the physical archive cleanup from #75658/#75672 and do not broaden this into a session storage redesign. Remove or avoid synchronous full-transcript materialization/full scans from latency-sensitive paths: `readSessionMessages`, `readSessionMessageCount`/`readRecentSessionMessagesWithStats`, `readLatestSessionUsageFromTranscript`, `sessions.send`/`sessions.create` messageSeq calculation, `sessions.get`, cursor HTTP history, `SessionHistoryState`, session message broadcasts, artifact lookup, TUI history, status usage fallback, and agent prompt-token estimation. Prefer cached sequence/usage snapshots and bounded tail reads for hot paths; reserve full traversal for explicit historical/export operations and make it yielding/async/offloaded where needed. Preserve compaction/tree transcript behavior, real sequence metadata, cursor pagination, SSE delivery, artifact lookup, reset/delete semantics, and existing public protocol shapes. Add focused regression tests that prove large transcripts do not require full synchronous reads/scans on the hot paths. Add a user-facing `CHANGELOG.md` fix entry crediting Thanks @DerFlash, and include commenters if the final patch uses their reproduced surfaces.
