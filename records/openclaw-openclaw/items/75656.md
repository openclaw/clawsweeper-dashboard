## Summary

Keep open. Current main has partial mitigations for first-page history and session-list paths, but synchronous full transcript reads and scans still remain on Gateway send, cursor-history, session-event, and usage-fallback paths; the open #75875 fix PR should be reviewed or replaced before this issue closes.

## What This Changes

The issue requests removing synchronous session transcript I/O from latency-sensitive Gateway paths that can block WebSocket handshakes and channel responsiveness.

## Best Possible Solution

Land a focused Gateway/session fix that removes synchronous full transcript materialization and unbounded scans from request, cursor-history, event, and session-row paths, using bounded readers or cached sequence/usage metadata where possible.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create large or usage-metadata-missing session transcripts, then call `sessions.send`, cursor-based history, session event subscription updates, or row loading paths that currently invoke the synchronous transcript helpers.

## Solution Assessment

Yes. The issue’s async/bounded I/O direction is the right class of fix, with the best implementation keeping full transcript traversal out of hot Gateway paths and preserving separate cleanup work under #75658/#75672.
