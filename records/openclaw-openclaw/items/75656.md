## Summary

Keep open. Current main still has synchronous full-transcript reads on Gateway request, session-history, event, and session-row paths; the related model-catalog fix and archived-file cleanup work do not remove this remaining event-loop blocker.

## What This Changes

The report asks to remove synchronous full transcript reads from Gateway/session hot paths because large session transcripts can block the event loop and cause WebSocket and Telegram timeouts.

## Best Possible Solution

Land a focused Gateway/session performance fix that removes synchronous full-transcript materialization from latency-sensitive request, SSE, broadcast, and session-row paths; use bounded tail reads, cached message/usage counters, or async/offloaded full scans where full history is genuinely required, and leave physical archive cleanup to #75658/#75672.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to create a large session JSONL and hit `sessions.get`, `/sessions/:key/history`, `sessions.send`, or transcript-update broadcast; current main synchronously reads and parses the full transcript on those paths, matching the reporter's liveness and timeout logs.

## Solution Assessment

Yes, the requested direction is the right class of fix, but the best implementation is narrower than converting every helper to async: preserve branch-aware history semantics while replacing hot full scans with bounded readers, cached counters, or offloaded scans.
