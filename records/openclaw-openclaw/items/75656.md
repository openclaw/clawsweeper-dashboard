## Summary

Keep open. Current main has a partial mitigation for common session-history and session-list paths, but it still leaves synchronous full-transcript reads or full synchronous transcript scans on Gateway/session event paths, so the reported event-loop blocking class is not fully fixed or superseded.

## What This Changes

The issue requests removing synchronous full-transcript/session reads from Gateway session preparation, RPC, history, and event paths that can block WebSocket handshakes and channel responsiveness.

## Best Possible Solution

Land a focused Gateway/session performance fix that removes synchronous full-transcript materialization from latency-sensitive RPC, history, event, and session-row paths; use cached sequence/usage metadata or bounded tail reads for hot paths, and reserve full transcript traversal for explicit historical/export operations with yielding or offloaded work.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: create a large session JSONL, then call `sessions.get`, a cursor-based history request, `sessions.send`, or emit a transcript update while stored usage/sequence metadata is missing; current main still performs synchronous full scans or full reads on those paths.

## Solution Assessment

Yes, the issue’s requested direction is the right class of fix, but the best implementation should be narrower than converting every helper to async: add cached counters/usage snapshots and bounded readers for hot paths, while preserving full branch traversal only where the product genuinely needs complete history.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75656. Current main partially bounded some session transcript paths, but Gateway hot paths still synchronously scan or fully read transcripts. Remove synchronous full-transcript work from latency-sensitive paths without changing unrelated session cleanup behavior.

Expected boundary:
- Keep full-history behavior for explicit cursor/history/export paths only when necessary, but do not block WebSocket/session/event hot paths on unbounded synchronous reads.
- Replace `readSessionMessageCount` use in `sessions.send`, session create, and transcript event broadcasts with cached or stored sequence metadata where possible, or a bounded/offloaded/yielding fallback.
- Avoid `loadGatewaySessionRow()` event payloads triggering uncapped `readLatestSessionUsageFromTranscript()`; use stored fields, bounded tail usage, or omit stale fallback data on push events.
- For `sessions.get` and first-page HTTP/SSE history, avoid counting the whole transcript just to return a limited tail window; preserve correct `hasMore`/cursor semantics with bounded metadata or a cheap approximate/known count.
- Preserve Pi session DAG/`parentId` semantics; do not raw-append tree transcript messages.
- Do not fold in #75658 physical archive cleanup or #75672 CLI cleanup unless that PR has already landed and a tiny call-site adjustment is strictly needed.
