## Summary

Keep open. Current main still has synchronous full-transcript reads on Gateway/session paths, including `readSessionMessages()` and transcript-usage fallback reads; the report is concrete, current, and not covered by the earlier model-catalog timeout fix or the separate physical cleanup issue.

## What This Changes

The issue requests moving session transcript reads off the Gateway event-loop hot path by making full transcript reads async or bounded, caching usage/token counts where appropriate, and leaving physical archive cleanup to the related cleanup issue.

## Best Possible Solution

Keep the issue open and land a focused Gateway/session performance fix: remove synchronous full-transcript work from request, SSE, event, and session-row paths; use bounded tail reads or cached counters where full history is not required; make any unavoidable full scan asynchronous or offloaded; and keep physical archive cleanup tracked separately in #75658.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: create a large session JSONL and hit `sessions.get`, `/sessions/:key/history`, session event emission, or a session-row path missing stored usage fields, then observe the synchronous full-file scan blocking the Node event loop; the issue body adds matching liveness and timeout logs from a real install.

## Solution Assessment

Yes, the requested direction is the right class of fix, but it should be narrower than simply making every helper async: prefer bounded reads, cached message counts/usage snapshots, and async/offloaded full scans only for paths that genuinely need full branch materialization.
