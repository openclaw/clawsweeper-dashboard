## Summary

Current main has moved the reported Gateway transcript hot paths onto async, chunked, or bounded readers, with tests and changelog coverage; the fix is present on main but I could not prove it is in a tagged release yet.

## What This Changes

Replace synchronous full transcript reads in Gateway/session hot paths with async or bounded I/O so large session stores do not block WebSocket, Telegram, and RPC responsiveness.

## Best Possible Solution

Keep the current main implementation and ship it in the next release; continue handling physical archived-transcript cleanup under the separate #75658/#75672 track.

## Reproduction Assessment

Yes. The source-level reproduction is a large session JSONL plus Gateway calls such as history, list, send, or transcript-event paths; current-main inspection shows those paths no longer use the synchronous full-transcript reader that caused the reported stalls.

## Solution Assessment

Yes. Async chunked transcript indexing plus bounded tail/usage reads is the narrow maintainable fix for event-loop responsiveness, while disk garbage collection remains a separate cleanup concern.
