## Summary

Keep open. Current main has partial mitigations for common session-list and first-page history paths, but the reported event-loop blocking class is still present: several Gateway/session paths still perform synchronous full transcript reads or full synchronous transcript scans for history, message sequence, usage fallback, and artifact/attachment indexing. The Linux and macOS follow-up comments add credible cross-platform confirmation, and the related cleanup work only addresses archived file buildup, not the remaining hot-path transcript reads.

## What This Changes

The issue requests removing synchronous session transcript reads and scans that block the Gateway event loop, causing WebSocket handshake timeouts and channel/UI unresponsiveness on large session stores.

## Best Possible Solution

Land a focused Gateway/session performance fix that removes synchronous full transcript materialization and unyielded full JSONL scans from latency-sensitive RPC, history, send/create, event broadcast, artifact, and attachment paths. Persist or cache sequence and usage metadata where possible, use bounded tail/cursor readers for UI/history windows, and reserve full transcript traversal for explicit export/maintenance operations with yielding or off-thread work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a large session JSONL with missing cached sequence/usage metadata, then call `sessions.get`, cursor-based `chat.history`, `sessions.send`, or a transcript update broadcast; current main reaches readFileSync or an unyielded openSync/readSync JSONL scan on those paths.

## Solution Assessment

Yes, the issue's requested direction is the right class of repair, but the best implementation should be narrower than mechanically making every helper async. The maintainable fix is to avoid repeated full scans on hot paths through cached metadata and bounded readers, with async/yielding/offload only where full traversal is unavoidable.
