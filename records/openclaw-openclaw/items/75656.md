## Summary

Keep open. Current main has partial mitigations for common session-list and first-page history reads, but the reported class is still present: synchronous full transcript materialization or full synchronous transcript scans remain on Gateway history, send/create, session-event sequencing, and usage-fallback paths. The cross-platform issue comments add useful confirmation that cleanup of archived files helps disk pressure but does not remove the active synchronous read/parse bottleneck.

## What This Changes

The issue requests removing synchronous session transcript reads and scans from Gateway hot paths because they block the event loop and cause WebSocket, TUI, and Telegram timeouts.

## Best Possible Solution

Land a focused Gateway/session performance repair that removes synchronous full-transcript materialization and full synchronous transcript scans from latency-sensitive RPC, HTTP history, send/create, and broadcast paths. Use cached sequence/usage metadata or bounded/yielding readers for hot paths, keep full traversal only for explicit historical/export operations, and leave physical archived-file cleanup to #75658/#75672.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: create a large session transcript with missing cached usage/sequence metadata, then call cursor history, `sessions.send`, create-with-initial-message, or emit a transcript update; current main still performs synchronous full reads or scans on those paths. The issue comments also provide Pi, Linux x86-64, and macOS arm64 runtime confirmations.

## Solution Assessment

Yes, the issue's requested direction is the right class of fix, but the maintainable solution should be narrower than a blanket async conversion. The best path is to remove hot-path full scans with cached counters/usage snapshots and bounded readers while preserving full transcript traversal for operations that explicitly need complete history.
