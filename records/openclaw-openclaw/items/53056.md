## Summary

Keep open. Current main still routes plain Slack replies through Slack mrkdwn formatting that encodes raw ampersands to `&amp;`, and the latest release snapshot has the same formatter behavior. The linked fix attempts were closed unmerged, and the discussion still needs a live Slack rendering decision that preserves exactly one safe entity-escaping layer.

## What This Changes

The issue reports that Slack-bound messages containing raw ampersands render as visible `&#38;` instead of `&`, likely from double entity encoding.

## Best Possible Solution

Keep the issue open until Slack rendering is verified directly, then update the Slack-owned formatter and preview paths so outbound text receives exactly one safe entity-escaping layer while preserving intentional Slack angle tokens and literal-entity safety, with regression coverage for raw ampersands, already escaped entities, mentions, special mentions, and links.

## Reproduction Assessment

Partial. Source-level reproduction is high confidence because `R&D` goes through `markdownToSlackMrkdwnChunks` and becomes `R&amp;D` before Slack posting, but the visible `&#38;` rendering still needs live Slack proof.

## Solution Assessment

Unclear. A broad removal of `&` escaping is not yet proven to be the best fix because the safer contract must avoid both double display encoding and accidental decoding of literal entities into active Slack tokens.
