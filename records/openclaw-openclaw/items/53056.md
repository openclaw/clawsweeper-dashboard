## Summary

Keep open. Current main still routes Slack outbound text through formatters that encode raw ampersands, and the latest release snapshot preserves the same behavior; the linked fix PRs were closed unmerged. Slack's own docs say `&`, `<`, and `>` are control characters that must be entity-escaped when not used for parsing, so the remaining work needs live rendering proof and an exactly-one-escape contract rather than a blind removal of ampersand escaping. ([docs.slack.dev](https://docs.slack.dev/messaging/formatting-message-text/))

## What This Changes

Fix Slack outbound rendering so literal ampersands display as `&` instead of `&#38;` without breaking Slack mrkdwn token escaping.

## Best Possible Solution

Keep this open until Slack rendering is verified directly, then update the Slack-owned formatter, send paths, preview/slash helpers, and focused tests so outbound text gets exactly one safe entity-escaping layer while preserving intentional Slack angle tokens and literal-entity safety.

## Reproduction Assessment

Unclear. Source inspection gives a high-confidence path showing OpenClaw currently emits `&amp;` for raw ampersands, but the reported visible `&#38;` result still needs a live Slack smoke using raw `R&D`, already-escaped `R&amp;D`, literal `&lt;@U123&gt;`, real `<@U123>`, `<!here>`, and link tokens.

## Solution Assessment

Unclear. Removing ampersand escaping is not currently proven to be the best fix because Slack documents `&` as a control character that should be escaped when literal; the maintainable solution is a verified exactly-one-escaping policy with regression coverage for both display text and active Slack tokens.
