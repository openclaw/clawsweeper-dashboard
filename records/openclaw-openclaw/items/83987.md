## Summary

Keep this PR open for maintainer review: the patch is focused and I found no blocking correctness defect, but it deliberately extends a Codex native-hook relay security boundary after successful turns. The related user report at https://github.com/openclaw/openclaw/issues/73723 and follow-up comments support that this is a real Codex gateway-path failure, while the merge decision is whether the bounded relay grace is acceptable.

## What This Changes

The PR defers Codex native-hook relay unregister after successful app-server turns, makes relay cleanup and renewal generation-aware, and adds regression coverage for late callbacks and failure cleanup paths.

## Best Possible Solution

Land after maintainers explicitly accept the bounded successful-turn relay grace and keep immediate cleanup for aborted, timed-out, interrupted, and error-terminal paths.

## Reproduction Assessment

Yes at source level: current main synchronously unregisters the relay during final cleanup, and the PR body plus linked discussion include before/after late-callback evidence. I did not run a live gateway Codex agent in this read-only review.

## Solution Assessment

Yes, with maintainer risk acceptance: deferring only successful cleanup and making handles generation-aware is a focused fix for late callbacks without keeping aborted or error-terminal turns open. The cleaner alternative would be an upstream hook-subprocess drain signal, but that is not available in the current OpenClaw/Codex contract.
