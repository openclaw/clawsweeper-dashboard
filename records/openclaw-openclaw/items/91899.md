## Summary

Keep open: the older one-shot Feishu reconnect failure is already addressed on current main and in v2026.6.6, but the reported silent-health problem remains because Feishu still does not publish connected or transport-liveness fields into gateway status. The related open PR https://github.com/openclaw/openclaw/pull/90966 targets that remaining gap, but current review context shows it still needs maintainer review and stronger runtime proof.

## What This Changes

The issue reports Feishu WebSocket disconnects that can leave inbound messages silently lost while `openclaw status --deep` still reports the channel as healthy.

## Best Possible Solution

Land a Feishu plugin-owned transport-status fix that publishes connected/disconnected state from SDK lifecycle or heartbeat-backed proof, then validate it with focused tests plus redacted live Feishu status/health-monitor evidence.

## Reproduction Assessment

Yes for the remaining status/health-monitor gap by source inspection: Feishu still publishes only port status while the gateway health policy needs connected and transport-activity fields. I did not live-reproduce the reported Feishu WebSocket outage.

## Solution Assessment

Yes, keeping this issue open is the right outcome until the related Feishu transport-status PR or an equivalent fix lands. The best implementation should use SDK lifecycle or true transport activity rather than app-event timestamps as socket liveness.
