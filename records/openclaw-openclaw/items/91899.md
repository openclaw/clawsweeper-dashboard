## Summary

Keep open: current main and v2026.6.5 already include the earlier Feishu WebSocket supervisor/backoff fixes, but source inspection still shows Feishu does not publish transport liveness into the gateway health monitor, so a silent dead WebSocket can still look healthy. The narrower active path is the Feishu transport-status work in https://github.com/openclaw/openclaw/pull/90966, plus live proof for the reported v2026.6.5 failure mode.

## What This Changes

The issue reports Feishu WebSocket long-connection drops on v2026.6.5, insufficient reconnect/liveness handling, and status reporting OK despite silent message loss.

## Best Possible Solution

Land a narrow Feishu transport-liveness fix, likely via https://github.com/openclaw/openclaw/pull/90966 or equivalent, so Feishu publishes connected and transport-activity status that the existing gateway health monitor can restart on.

## Reproduction Assessment

Yes for the status/health-monitor gap by source inspection: Feishu does not publish transport liveness fields, and the gateway health policy treats missing transport activity as healthy. I did not live-reproduce the reported v2026.6.5 WebSocket drop.

## Solution Assessment

Unclear as a completed fix: the old reconnect-loop part is already shipped, but the best remaining fix is to wire Feishu transport status into the existing health monitor and verify it against a real or faithfully simulated dead WebSocket.
