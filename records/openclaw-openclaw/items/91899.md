## Summary

Keep open: current main and v2026.6.8 have the persistent Feishu WebSocket reconnect loop, but Feishu still publishes only account/port status, so the gateway health monitor still lacks connected or transport-activity fields for silent-socket detection. The remaining gap is already targeted by open PR https://github.com/openclaw/openclaw/pull/90966, which still needs maintainer review and stronger runtime proof before this issue can close.

## What This Changes

The issue reports Feishu WebSocket disconnects causing silent message loss, with reconnect recovery and status checks that still show the channel as healthy.

## Best Possible Solution

Land a Feishu plugin-owned transport-liveness fix that publishes real connected/disconnected and transport activity signals, then close this issue once the linked PR or an equivalent implementation reaches main.

## Reproduction Assessment

Yes for the remaining status/health-monitor gap by source inspection: Feishu still publishes only account and port status, while the gateway policy needs connected and transport-activity fields to detect stale sockets. I did not live-reproduce the reporter's Feishu WebSocket outage.

## Solution Assessment

Yes, keeping this issue open behind https://github.com/openclaw/openclaw/pull/90966 is the best current path because the reconnect subset is already fixed but the status-liveness subset is not on main. The implementation should use SDK lifecycle or heartbeat-backed transport activity, not ordinary app-event timestamps alone.
