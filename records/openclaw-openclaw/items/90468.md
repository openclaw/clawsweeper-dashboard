## Summary

Keep this PR open: it targets a real shipped prompt-template parsing bug and the fix is in the right shared parser, but the proposed quote-opening check still corrupts ordinary prose when a command contains more than one apostrophe.

## What This Changes

The PR changes prompt-template argument parsing so unmatched quote characters, including English apostrophes, are treated as literal argument text while balanced quotes still group arguments.

## Best Possible Solution

Keep the shared parser fix, but refine the quote-opening rule so apostrophes inside word tokens remain literal while balanced delimiter quotes still group arguments, with regression coverage for multiple contractions.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main enters quote mode on apostrophes in `parseCommandArgs`, and the PR predicate still fails for multiple contractions. I did not run repository tests because this was a read-only review, but the parser behavior is deterministic and was checked with a non-mutating node probe of the PR algorithm.

## Solution Assessment

No. The fix is in the best location, but `chars.includes(char, i + 1)` is too broad because a later apostrophe in another word is treated as a closing quote rather than ordinary prose.

- **[P2] Preserve apostrophes across multiple contractions:** `packages/agent-core/src/harness/prompt-template-arguments.ts:16`
  - body: The new `chars.includes` guard only treats an apostrophe as literal when there is no later apostrophe anywhere in the remaining input.
