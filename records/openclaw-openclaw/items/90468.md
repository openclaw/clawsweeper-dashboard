## Summary

Keep open: the branch fixes the multiple-contraction apostrophe case and has sufficient real behavior proof, but the new token-boundary quote rule regresses existing shell-style mid-token quoted spans such as `--title="two words"`, so it needs a narrow parser/test repair before merge.

## What This Changes

The branch changes the shared prompt-template argument parser and tests so apostrophes in slash-command template arguments remain literal while balanced quoted spans still group.

## Best Possible Solution

Keep the shared parser as the fix point, but classify apostrophes inside word tokens as literals while preserving existing shell-style quoted spans in mid-token positions, with regression coverage for both patterns.

## Reproduction Assessment

Yes. Source inspection plus a non-mutating parser probe show the PR fixes multiple contractions but regresses current `foo="bar baz"`/`--title="bar baz"` parsing into split tokens.

## Solution Assessment

No. The shared parser is the right layer, but requiring quote delimiters to start at whitespace is too narrow; the safer repair is to treat apostrophes inside words as literal without rejecting shell-style mid-token quoted spans.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve mid-token quoted spans:** `packages/agent-core/src/harness/prompt-template-arguments.ts:33`
  - body: The new `opensQuotedSpan` requires the opening quote to be at start/after whitespace, but current `parseCommandArgs` treats quotes after other characters as shell-style delimiters. With this patch, `/cmd --title="two words" next` parses as `["--title=\"two","words\"","next"]` instead of the current `["--title=two words","next"]`, so existing templates using `$1` or `$ARGUMENTS` can send corrupted instructions. Please make apostrophes inside words literal without rejecting quoted spans embedded in a token, and add a regression for that case.
  - confidence: 0.88
