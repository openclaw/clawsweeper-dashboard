## Summary

Keep open. Current main still has source evidence for LINE webhook processing that can block or delay the HTTP 200 path, and the open ACK-first fix in #78186 likely overlaps the silent-bot symptom, but this report's `lastInboundAt` claim is distinct enough that it should not be closed as a duplicate until the linked fix lands and the LINE inbound path is retested.

## What This Changes

The issue asks OpenClaw to fix LINE webhook startup and inbound handling so a configured `@openclaw/line` bot processes POSTed messages and replies on 2026.5.4.

## Best Possible Solution

Review and land an ACK-first LINE webhook fix such as #78186 if it remains correct, then retest this report specifically for `lastInboundAt`; if it still stays unset, fix the LINE monitor/webhook dispatch path or add diagnostics that expose why the POST did not become an inbound message.

## Reproduction Assessment

Partly. Current main has a high-confidence source path for one reported symptom: a valid signed LINE POST with a slow or pending `handleWebhook` delays HTTP 200 until event processing finishes, but I do not have a high-confidence current-main reproduction of `lastInboundAt` never being set on a real LINE message.

## Solution Assessment

Unclear as a complete fix. ACKing after signature and payload validation is the narrow fix for the known timeout/silent-reply path, but this issue should remain open until that fix lands and the `lastInboundAt` symptom is verified or separately repaired.
