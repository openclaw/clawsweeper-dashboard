## Summary

Keep this open: this issue is still the strongest canonical tracker for the duplicate streamed-reply regression. A merged Telegram-specific fix covers one replay path, but current main still has source-visible shared auto-reply behavior where partial previews and final payload delivery are not collapsed together.

## What This Changes

The issue reports duplicate or partial duplicate assistant replies after the 2026.5.20 update, especially for long streamed responses across Telegram, WhatsApp, and Web UI.

## Best Possible Solution

Restore a shared auto-reply preview/final collapse path that suppresses final text already delivered through onPartialReply while preserving unsent media or genuinely new final suffixes, then validate the shared path plus Telegram-visible behavior.

## Reproduction Assessment

Yes, at source level. The issue comments describe a minimal runReplyAgent path where onPartialReply streams text and the same text returns as final payloads, and current tests/source still show final payloads being kept after partial preview streaming.

## Solution Assessment

No, the merged Telegram-specific replay fix is not the complete solution. The best fix should live in shared auto-reply final-payload construction or adjacent shared delivery state, not only in one channel lane.
