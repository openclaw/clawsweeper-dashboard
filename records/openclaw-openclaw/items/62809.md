## Summary

Keep open. Current main still uses the isolated cron run session id as the OpenAI prompt-cache affinity key, while isolated cron deliberately creates a fresh session id per run; the newer related report was closed as a duplicate of this issue, and the adjacent open prompt-prefix issues do not cover this cross-run key-affinity gap.

## What This Changes

The issue requests stable cross-run OpenAI prompt-cache affinity for recurring isolated cron jobs without sacrificing isolated run sessions.

## Best Possible Solution

Add an internal cron-owned cache-affinity identity for OpenAI-family transports, derived from the job and effective provider/model/stable-prefix context, while keeping isolated session ids, session keys, transcripts, and cleanup semantics per run.

## Reproduction Assessment

Yes, at source level. Two isolated runs of the same cron job still get different random session ids, and current OpenAI-family request builders still use that per-run session id as prompt_cache_key.

## Solution Assessment

Yes. A separate cron cache-affinity key is the narrow maintainable fix; reusing the isolated runtime session id would weaken isolation, and adding user config is optional rather than required for the first repair.
