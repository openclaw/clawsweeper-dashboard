## Summary

Keep open. Current main still derives OpenAI-family prompt-cache affinity from the per-run session id, and isolated cron still creates a fresh session id for each run; the adjacent open issues cover prompt-prefix drift and volatile time injection, not this cross-run cache-affinity gap.

## What This Changes

The issue requests stable OpenAI-family prompt-cache affinity across recurring isolated cron runs without weakening isolated session semantics.

## Best Possible Solution

Add an internal stable cron cache-affinity identity derived from job id plus effective OpenAI-family provider/model context, and pass it separately to provider transports while keeping isolated session ids, session keys, and transcripts run-scoped.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: two isolated runs of the same cron job get different random `sessionId` values, and OpenAI-family request builders currently use that value as `prompt_cache_key`.

## Solution Assessment

Yes. A separate cron-owned cache-affinity key is the narrow maintainable fix; reusing the stable cron session key as the runtime session id would weaken isolation, and a new user config knob is not needed for the first repair.
