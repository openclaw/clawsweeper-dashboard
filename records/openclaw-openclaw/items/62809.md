## Summary

Keep open. Current main still derives OpenAI prompt-cache affinity from the per-run session id, while isolated cron intentionally creates a fresh session id and run-scoped session key for each run, so the reported first-turn cross-run cache instability remains a real runtime gap rather than a duplicate of the adjacent prompt-prefix issues.

## What This Changes

The issue requests stable cross-run OpenAI prompt-cache affinity for recurring isolated cron jobs while preserving fresh isolated cron transcripts.

## Best Possible Solution

Add a cron-owned prompt-cache affinity identity that is stable for a recurring job and model, but keep isolated cron transcripts, session ids, run session keys, ownership, and cleanup run-scoped.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction: two isolated cron runs receive different random `sessionId` values, and the OpenAI request builders currently use that value as `prompt_cache_key`.

## Solution Assessment

Yes. A separate stable cache-affinity key is the narrow maintainable fix; reusing the cron session id or run session key would weaken the intentional isolation and cleanup semantics.
