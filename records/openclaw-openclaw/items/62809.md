## Summary

Keep open. Current main and the latest shipped tag still route OpenAI-family prompt-cache affinity through the per-run isolated cron session id, while isolated cron deliberately creates a fresh id for each run; the closed related issue points back here as the canonical tracker for the remaining cross-run cache-affinity repair.

## What This Changes

Issue requests stable OpenAI prompt-cache affinity for recurring isolated cron jobs so first turns can reuse cache across runs.

## Best Possible Solution

Add an internal cron-owned cache-affinity identity for OpenAI-family transports, derived from job plus effective provider/model/stable-prefix context, while keeping isolated transcripts and cleanup per run.

## Reproduction Assessment

Yes, at source level. Two isolated runs of the same cron job still get different random session ids, and current OpenAI-family request builders still use that per-run session id as prompt_cache_key.

## Solution Assessment

Yes. A separate internal cron cache-affinity key is the narrow maintainable fix; reusing the isolated session id would weaken isolation, while a new user-facing config knob can remain optional follow-up design.

## Repair Work Prompt

Fix recurring isolated cron OpenAI prompt-cache affinity without weakening session isolation. Current isolated cron runs force a fresh session id per run and OpenAI-family transports use options.sessionId as prompt_cache_key, so the first turn cannot reliably reuse cache across repeated runs of the same job. Add an internal cron-owned cache-affinity identity that is stable across runs for the same effective job/provider/model/stable prompt context, thread it to OpenAI Responses and opted-in OpenAI-compatible Completions prompt_cache_key construction, and keep transcript session ids, session keys, session files, cleanup, auth overrides, and isolated state per run. Do not add a public config option unless the existing contracts make that unavoidable, and do not claim this fixes unrelated prompt-prefix drift from time injection or tool/instruction changes. Add focused regression coverage proving two isolated runs keep distinct session ids/session keys but use the same provider cache-affinity key when the stable context matches, and prove cacheRetention none still suppresses prompt_cache_key.
