## Summary

Keep open. Current `main` still routes OpenAI prompt-cache affinity through the per-run `sessionId`, while isolated cron deliberately creates a fresh random session id and run-scoped session key for each run, so the reported first-turn cross-run cache instability remains a real runtime gap.

## What This Changes

The issue requests stable cross-run OpenAI prompt-cache affinity for recurring isolated cron jobs so unchanged job prompts can reuse first-turn cache across fresh runs.

## Best Possible Solution

Add a cron-owned prompt-cache affinity identity separate from runtime session identity: isolated cron should keep fresh transcripts, run-scoped session keys, active-run ownership, and cleanup behavior, while OpenAI-family payloads receive a stable job-scoped cache key when caching is enabled and the endpoint supports it.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence architectural reproduction: two isolated cron runs will get different random `sessionId` values, and the OpenAI request builders currently use that `sessionId` as `prompt_cache_key`.

## Solution Assessment

Yes, a separate stable `prompt_cache_key` for recurring isolated cron is the narrow maintainable direction. Reusing the cron session id or run session key would be the wrong fix because those identities also enforce transcript isolation and runtime ownership.

## Repair Work Prompt

Implement stable OpenAI prompt-cache affinity for recurring isolated cron jobs without weakening isolated cron session/runtime isolation. Add a separate prompt-cache key or affinity field, not a replacement for `sessionId` or `sessionKey`, and have OpenAI-family request builders use it for `prompt_cache_key` when cache retention is enabled and the provider/API supports that field. For default isolated `cron:<jobId>` jobs, derive a stable bounded key from cron job identity plus agent identity and preferably provider/model or a short hash so repeated runs share routing while distinct jobs do not collide. Preserve fresh transcript ids, run-scoped session keys, active-run ownership, MCP/browser cleanup, and provider compatibility stripping for non-OpenAI/proxy endpoints. Do not add broad cron cache-affinity config unless needed for this narrow fix. Add focused tests proving two isolated runs keep different runtime session ids/keys but produce the same OpenAI `prompt_cache_key`, plus transport tests for Responses and opted-in Completions behavior. Add a user-facing changelog fix entry if the active changelog requires it, crediting @lubao515, and avoid trying to solve the adjacent prompt-drift issues in #64217 and #71973 in the same PR.
