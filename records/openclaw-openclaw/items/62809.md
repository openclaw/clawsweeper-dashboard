## Summary

Keep open. Current main still derives OpenAI-family prompt-cache affinity from the per-run session id, while isolated cron deliberately creates a fresh session id and run-scoped session key for each run, so the reported first-turn cross-run cache instability remains unresolved rather than covered by the adjacent prompt-prefix issues.

## What This Changes

The issue asks OpenClaw to give recurring isolated cron jobs a stable cross-run OpenAI prompt-cache affinity identity without reusing isolated transcripts.

## Best Possible Solution

Add an internal cron-owned prompt-cache affinity key derived from stable job identity and effective OpenAI-family model, then pass that to provider transports while keeping isolated runtime sessions run-scoped.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: two isolated runs of the same cron job get different random `sessionId` values, and OpenAI request builders currently use that value as `prompt_cache_key`.

## Solution Assessment

Yes. A separate stable cache-affinity key is the narrow maintainable fix; reusing the cron session id or run session key would weaken the intended isolation semantics.
