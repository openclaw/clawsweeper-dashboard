## Summary

Keep open. Current main still derives OpenAI-family prompt-cache affinity from the isolated cron run session id, which is intentionally fresh per run, so the reported first-turn cross-run cache instability remains unresolved and is not covered by the adjacent prompt-prefix issues.

## What This Changes

The issue requests stable cross-run OpenAI prompt-cache affinity for recurring isolated cron jobs so first turns can reuse warm prompt caches.

## Best Possible Solution

Add an internal stable cron prompt-cache affinity key derived from stable job identity and effective OpenAI-family model, then pass it to provider transports separately from run/session ids.

## Reproduction Assessment

Yes, by source inspection. Two isolated runs of the same cron job deterministically get different random `sessionId` values, and current OpenAI request builders use that value as `prompt_cache_key`; I did not run a live OpenAI cron repro in this read-only review.

## Solution Assessment

Yes. A separate stable cache-affinity key is the narrow maintainable fix because reusing the cron runtime session id or run session key would weaken the intended isolated-run semantics.
