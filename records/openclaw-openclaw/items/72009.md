## Summary

Keep open: current main still lacks configurable typing TTL, and the PR is useful, but the latest diff still documents the new TTL as a hard per-run keepalive limit instead of the stale-refresh cleanup timeout implemented by the controller.

## What This Changes

The PR adds `typingTtlSeconds` to session and agent defaults, wires it into the auto-reply typing controller, raises the default TTL to 300 seconds, updates docs/schema/help/tests/changelog, and carries small merge-conflict formatting/test-fixture changes.

## Best Possible Solution

Keep the runtime/config direction if maintainers accept the new knob and default, but repair docs and schema help to describe cleanup after no successful typing refresh for the configured duration.

## Reproduction Assessment

Yes for the missing capability: source inspection shows current main has a hardcoded 2-minute TTL and no `typingTtlSeconds` schema/config path. The remaining PR defect is also source-reproducible from the PR diff and the current `refreshTypingTtl()` behavior.

## Solution Assessment

No, not quite as written. The runtime/config threading is narrow, but user-facing docs and schema help should describe TTL as the timeout since the last successful typing refresh, not as a maximum duration for one run.
