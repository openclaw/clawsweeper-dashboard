## Summary

Keep open: the PR is useful, but the latest diff still documents `typingTtlSeconds` as a hard per-run maximum even though the controller refreshes the timeout after successful typing ticks/text signals, and the PR head is currently dirty against the base branch.

## What This Changes

The PR adds `typingTtlSeconds` to session and agent-default config, raises the typing controller default TTL to 300 seconds, updates schema/help/docs/changelog, and carries a small typing persistence test plus merge-conflict formatting/test-fixture changes.

## Best Possible Solution

Correct the docs and generated schema-help wording to describe TTL as stale-refresh cleanup, keep the focused config/runtime behavior if maintainers approve the new knob/default, then rebase and land after exact-head checks.

## Reproduction Assessment

Yes for the missing capability: source inspection shows current main has a hardcoded 2 minute `typingTtlMs`, no `typingTtlSeconds` schema key, and no config-to-controller TTL wiring. The remaining PR defect is also source-reproducible from the diff and `refreshTypingTtl()` implementation.

## Solution Assessment

No, not quite as written. The runtime/config threading is narrow, but the user-facing docs and schema help should describe TTL as the timeout since the last successful typing refresh, not the maximum duration of one run.
