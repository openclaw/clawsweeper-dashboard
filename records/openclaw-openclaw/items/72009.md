## Summary

Keep this PR open: it implements the requested typing TTL config surface, but the latest diff still describes TTL as a maximum run duration even though the controller refreshes the timeout after successful typing activity.

## What This Changes

The branch adds `typingTtlSeconds` to session and agent-default config, raises the fallback typing TTL to 300 seconds, updates schema/help/docs/changelog, and adjusts related tests/fixtures.

## Best Possible Solution

Keep the narrow runtime/config threading if maintainers accept the new knob and default, but correct docs and generated schema help to describe TTL as stale-refresh cleanup after no successful typing refresh for the configured duration.

## Reproduction Assessment

Yes. Source inspection shows current main has a hardcoded 2 minute TTL with no `typingTtlSeconds` schema/config wiring, and the remaining PR defect is visible from the diff plus `refreshTypingTtl()` clearing and restarting the timer.

## Solution Assessment

No, not as written. The runtime/config path is narrow, but the user-facing docs and schema help should describe a refreshed stale-timeout window rather than a hard maximum duration for a single run.
