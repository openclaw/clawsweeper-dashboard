## Summary

Keep open: current main still lacks the configurable TTL, and the PR is useful but not merge-ready because its docs/schema help describe TTL as a hard per-run cap while the controller uses a refreshable stale-timeout.

## What This Changes

The PR adds `typingTtlSeconds` to session and agent-default config, raises the typing controller default TTL to 300 seconds, and updates generated schema/help/docs/tests plus conflict-resolution formatting changes.

## Best Possible Solution

Keep the runtime/config change if maintainers accept the new knob, but correct the docs and generated schema help to describe TTL as stale-refresh cleanup rather than a hard per-run cap.

## Reproduction Assessment

Yes for the missing capability: current source has a hardcoded two-minute `typingTtlMs` default and no `typingTtlSeconds` schema or config-to-controller wiring. The remaining PR defect is also source-reproducible from `refreshTypingTtl()` resetting the timeout on each successful typing tick.

## Solution Assessment

No, not quite as written. The config/runtime threading is narrow, but the user-facing docs and schema help should describe TTL as cleanup after no successful typing refresh for that duration, not as the maximum duration of a single run.
