## Summary

Keep this PR open: the runtime/config change is narrow, but the latest diff still documents `typingTtlSeconds` as a hard maximum duration for a run even though the controller refreshes the TTL after each successful typing tick.

## What This Changes

The PR adds `typingTtlSeconds` to session and agent-default config, threads it into the typing controller, raises the default TTL to five minutes, and updates docs, schema, tests, and changelog.

## Best Possible Solution

Keep the narrow implementation if maintainers accept the new knob/default, but revise docs and schema help/generated wording so `typingTtlSeconds` is described as the stale-refresh cleanup window refreshed by successful keepalive ticks.

## Reproduction Assessment

Yes. Source inspection shows current main has a hardcoded two-minute TTL and no `typingTtlSeconds` config wiring, and the remaining PR defect is reproducible from the diff against `refreshTypingTtl()` semantics.

## Solution Assessment

No, not as written. The runtime/config threading is narrow, but the user-facing docs and schema help should describe TTL as a stale-refresh cleanup timeout rather than a hard maximum duration for a run.
