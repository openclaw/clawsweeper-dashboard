## Summary

Keep this PR open: the runtime/config wiring is narrow and addresses the requested TTL knob, but the latest provided diff still documents `typingTtlSeconds` as a maximum per-run keepalive duration even though the current controller refreshes the TTL after each successful typing tick.

## What This Changes

The PR adds `typingTtlSeconds` to session and agent-default config, passes it to the typing controller, raises the default TTL to 300 seconds, and updates schema, docs, tests, changelog, and merge-refresh test fixtures.

## Best Possible Solution

Keep the PRâ€™s narrow config/runtime approach if maintainers approve the new knob and default, but update docs and schema help/generated text to describe TTL as stale-refresh cleanup after no successful typing tick for that duration.

## Reproduction Assessment

Yes. Source inspection shows current main hardcodes the TTL and lacks `typingTtlSeconds`, and the remaining PR defect is source-reproducible because `refreshTypingTtl()` resets the timer while the provided PR docs describe a hard per-run maximum.

## Solution Assessment

No, not as written. The runtime/config threading is the right narrow shape, but the user-facing docs and schema help should describe TTL as an idle/stale refresh cleanup window rather than a maximum duration for one run.
