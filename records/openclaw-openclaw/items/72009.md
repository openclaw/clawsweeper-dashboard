## Summary

Keep open: the PR addresses a real missing config surface, but it still documents `typingTtlSeconds` as a hard maximum for one run even though the runtime refreshes the TTL after successful typing ticks. The latest head also has non-green CI, so it should not merge until the wording and exact-head checks are clean.

## What This Changes

The PR adds `typingTtlSeconds` to session and agent-default config, wires it into the typing controller, raises the default TTL to 300 seconds, and updates docs, schema, tests, and changelog alongside merge-refresh formatting/test-fixture changes.

## Best Possible Solution

Keep the narrow config/runtime implementation if maintainers approve the new knob/default, but update docs and schema help/generated text to describe TTL as cleanup after no successful typing refresh for that duration, then rerun targeted schema/docs/typing checks and exact-head CI.

## Reproduction Assessment

Yes. Source inspection shows current main has a hardcoded 2-minute TTL and no `typingTtlSeconds` config wiring; the remaining PR defect is source-reproducible from the diff and `refreshTypingTtl()` implementation.

## Solution Assessment

No, not as written. The runtime/config threading is narrow, but the user-facing docs and schema help should describe TTL as a stale-refresh cleanup window rather than a hard maximum duration for one run.
