## Summary

Keep open: current main has partial mitigations for boot-md session isolation, shared-workspace duplication, and startup readiness, but the source still runs BOOT.md on every gateway startup whenever the file exists. Changing that to run once per workspace or config generation is a product/state decision, not implemented current behavior.

## What This Changes

The issue requests that the internal boot-md hook stop rerunning BOOT.md on every gateway restart and become idempotent or first-run-only.

## Best Possible Solution

Maintainers should decide whether BOOT.md is a per-startup checklist or a run-once initialization task; if run-once wins, store completion in a canonical state owner with reset/doctor/docs behavior while preserving the current isolated boot session and detached startup path.

## Reproduction Assessment

Yes, at source level: a configured gateway:startup event still queues boot-md, and runBootOnce skips only missing or empty BOOT.md before executing the boot run. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No as proposed: a simple flag file would add sidecar runtime state and would not settle reset or upgrade semantics. The safer solution needs a maintainer-approved boot-md contract and canonical state or documentation path.
