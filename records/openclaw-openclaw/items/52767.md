## Summary

Keep open. Current main still executes a non-empty BOOT.md on each gateway startup, while current docs describe BOOT.md as repeatable startup automation and reserve BOOTSTRAP.md for first-run setup, so this needs a maintainer product decision rather than cleanup closure.

## What This Changes

The issue requests durable idempotency for boot-md so BOOT.md does not execute on every gateway restart.

## Best Possible Solution

Decide and document the BOOT.md contract, then either add durable per-agent/workspace completion and reset semantics or keep BOOT.md repeatable and steer one-time initialization to BOOTSTRAP.md while preserving startup safety.

## Reproduction Assessment

Yes for the current behavior. Source inspection shows that enabling internal hooks with a non-empty BOOT.md and restarting the gateway twice dispatches gateway:startup twice and reaches runBootOnce both times without persistent completion state.

## Solution Assessment

Unclear. A sentinel guard is plausible, but current docs define BOOT.md as startup automation, so the maintainable fix needs a product decision about one-time versus repeatable semantics.
