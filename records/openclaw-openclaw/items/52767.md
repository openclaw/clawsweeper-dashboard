## Summary

Keep open: current main and v2026.6.10 still run BOOT.md whenever the gateway startup hook fires and the file exists. Related fixes mitigated shared-workspace duplication, boot-session pollution, and startup readiness blocking, but they did not decide or implement persistent run-once semantics.

## What This Changes

The issue requests making the bundled boot-md startup hook run BOOT.md only once per workspace or otherwise stop treating it as an every-restart hook.

## Best Possible Solution

Maintainers should decide the BOOT.md contract first; if run-once behavior is desired, implement completion/reset state in the canonical state owner with docs and regression coverage while preserving the isolated boot session.

## Reproduction Assessment

Yes, at source level: a configured gateway:startup event still queues boot-md, and runBootOnce skips only missing or empty BOOT.md before executing the boot run. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No, the suggested flag-file guard is not the best repo-shaped solution because it adds sidecar runtime state and leaves reset/upgrade semantics unsettled. A maintainer-approved boot-md contract plus canonical state ownership is the safer path if run-once behavior wins.
