## Summary

Keep open. Current main and v2026.5.12 still run a non-empty BOOT.md on each configured gateway startup; the production stall comment is useful evidence, but docs currently define BOOT.md as repeatable startup automation and reserve one-time setup for BOOTSTRAP.md, so durable run-once behavior needs a maintainer contract decision rather than cleanup closure.

## What This Changes

Request durable idempotency so the bundled boot-md hook does not execute BOOT.md again on every gateway restart.

## Best Possible Solution

Define the BOOT.md contract explicitly; if it should be once-per-workspace or once-per-config-generation, add durable completion/reset semantics with tests and docs, otherwise keep BOOT.md repeatable and steer one-time setup to BOOTSTRAP.md or explicit idempotent checks.

## Reproduction Assessment

Yes, source-level. Enable boot-md, keep a non-empty BOOT.md in the resolved workspace, and restart the gateway twice; each startup dispatch creates a fresh handler-local seenWorkspaces set and runBootOnce has no durable completion check before agentCommand.

## Solution Assessment

Unclear. A sentinel guard would address the reporter's expected once-only behavior, but it would change the currently documented BOOT.md startup contract, so maintainers should choose repeatable versus durable-once semantics and reset rules first.
