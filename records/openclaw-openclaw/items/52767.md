## Summary

Keep open. Current main and the latest release still run a non-empty BOOT.md on every gateway startup; that behavior is documented as startup automation, so changing it to durable run-once behavior needs a maintainer contract decision rather than cleanup closure.

## What This Changes

The issue requests that the bundled boot-md gateway startup hook stop re-running BOOT.md on every restart, likely via durable idempotency, reset semantics, or a redesign away from startup hooks.

## Best Possible Solution

Define BOOT.md's contract explicitly, then either implement documented once-per-workspace or once-per-config-generation behavior with reset/session-isolation semantics, or keep repeatable startup behavior and steer one-time setup to BOOTSTRAP.md or idempotent BOOT.md checks.

## Reproduction Assessment

Yes, source-level. Enable boot-md, keep a non-empty BOOT.md in the resolved workspace, and restart the gateway twice; each startup schedules gateway:startup, creates a fresh handler-local seenWorkspaces Set, and runBootOnce has no durable completion check before agentCommand.

## Solution Assessment

Unclear. A sentinel guard would satisfy the requested once-only behavior, but it would change the documented BOOT.md startup checklist contract, so maintainers should choose repeatable versus durable-once semantics and reset rules first.
