## Summary

Keep open. Current main still runs non-empty `BOOT.md` on every configured gateway startup, but current docs describe `BOOT.md` as repeatable startup automation and reserve one-time first-run setup for `BOOTSTRAP.md`, so this needs maintainer product judgment rather than conservative cleanup closure.

## What This Changes

The issue asks for the bundled `boot-md` startup hook to stop re-running non-empty `BOOT.md` on every gateway restart and instead become idempotent or explicitly one-time.

## Best Possible Solution

Define the `BOOT.md` contract explicitly, then either implement durable per-agent/workspace completion with reset semantics or keep `BOOT.md` repeatable and steer one-time setup to `BOOTSTRAP.md`, while preserving boot-session isolation and channel startup safety.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable internal hooks, place a non-empty `BOOT.md` in the resolved workspace, and restart the gateway twice; each startup dispatch reaches `runBootOnce` with no persistent completion check.

## Solution Assessment

Unclear. A sentinel guard is plausible, but current docs define `BOOT.md` as startup automation, so the maintainable fix needs a product decision about reset and re-run semantics before changing behavior.
