## Summary

Keep open: current main and v2026.6.10 still run BOOT.md whenever the boot-md gateway startup hook fires and the file exists. Related merged work reduced duplicate workspace runs, session pollution, and startup blocking, but persistent run-once semantics remains an explicit product/state-ownership decision.

## What This Changes

Request to make bundled boot-md stop re-executing BOOT.md on every gateway restart and add persistent/idempotent first-run behavior.

## Best Possible Solution

Maintainers should decide whether BOOT.md is per-restart or run-once; if run-once wins, store completion/reset state in the canonical state owner, update docs, and keep the isolated boot session behavior.

## Reproduction Assessment

Yes, at source level: the boot-md handler queues runBootOnce on gateway startup, and runBootOnce skips only missing or empty BOOT.md before agentCommand. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No, the suggested flag-file guard is not the best repo-shaped solution because it adds ad hoc state and leaves reset/upgrade semantics unsettled. A maintainer-approved BOOT.md contract plus canonical state ownership is the safer path if run-once behavior wins.
