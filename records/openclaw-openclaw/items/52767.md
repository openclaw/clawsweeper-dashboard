## Summary

Keep open: current main still executes BOOT.md on every gateway startup when the file is present. Recent merged fixes mitigate shared-workspace duplication and session pollution, but durable run-once behavior remains a maintainer product/state decision.

## What This Changes

The issue asks for boot-md to stop rerunning BOOT.md on every gateway restart and to avoid restart-triggered session/channel stalls from repeated boot work.

## Best Possible Solution

Maintainers should decide whether BOOT.md is a per-startup checklist or a run-once initialization task; if run-once wins, store completion in canonical state with reset/doctor/docs behavior while preserving the isolated boot session and detached startup path.

## Reproduction Assessment

Yes, at source level: a configured gateway:startup event still queues boot-md, and runBootOnce skips only missing or empty BOOT.md before executing the boot run. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No as proposed: a raw flag file would add sidecar runtime state and would not settle reset or upgrade semantics. The safer solution needs a maintainer-approved boot-md contract and canonical state or documentation path.
