## Summary

Keep open. Current main still reaches the boot-md execution path on every configured `gateway:startup` event and `runBootOnce` still only skips missing or empty `BOOT.md`; recent changes provide fresh boot-session isolation and workspace-level deduplication, but not durable completion or reset semantics.

## What This Changes

The issue asks for boot-md/BOOT.md startup handling to become idempotent or otherwise stop re-executing one-time initialization on every gateway restart.

## Best Possible Solution

Decide the `BOOT.md` contract explicitly, then either implement a durable per-agent workspace completion guard with documented reset/re-run semantics or update docs/UX to state that `BOOT.md` is repeatable startup automation and steer one-time initialization to `BOOTSTRAP.md`; preserve the current boot-session isolation and ensure boot work cannot stall channel readiness.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to configure internal hooks, place a non-empty `BOOT.md` in the resolved workspace, and restart the gateway twice; current source shows each startup dispatch reaches `runBootOnce`, and `runBootOnce` has no persistent completion check.

## Solution Assessment

Unclear. The requested idempotency is plausible, but the suggested flag-file guard is not proven best because current docs define `BOOT.md` as startup behavior; a maintainable fix needs an explicit reset/re-run contract before changing semantics.
