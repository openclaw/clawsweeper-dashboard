## Summary

Keep open. Current main and the latest release still execute a non-empty BOOT.md on each configured gateway startup, but the docs describe BOOT.md as repeatable startup automation and reserve one-time first-run setup for BOOTSTRAP.md, so this needs maintainer product judgment rather than conservative cleanup closure.

## What This Changes

Request to stop the bundled boot-md hook from executing BOOT.md on every gateway restart, likely by adding durable idempotency or redesigning startup-hook behavior.

## Best Possible Solution

Define the BOOT.md contract explicitly, then either implement durable per-workspace or per-generation completion with reset semantics and tests, or keep BOOT.md repeatable and steer one-time setup to BOOTSTRAP.md while preserving detached startup and boot-session isolation.

## Reproduction Assessment

Yes, source-level. Enable boot-md, keep a non-empty BOOT.md in the resolved workspace, and restart the gateway twice; each gateway:startup event creates a fresh seenWorkspaces set and reaches runBootOnce with no durable completion check.

## Solution Assessment

Unclear. A sentinel guard is plausible, but it would change currently documented BOOT.md startup semantics, so maintainers should first choose repeatable versus once-per-workspace behavior and define reset/session-isolation rules.
