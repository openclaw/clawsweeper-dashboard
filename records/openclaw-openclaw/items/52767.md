## Summary

Keep open: current main and v2026.6.10 still run BOOT.md whenever the gateway startup hook fires and BOOT.md exists. Related work fixed same-workspace duplication, isolated boot sessions, and startup readiness blocking, but persistent run-once semantics remain a maintainer product decision.

## What This Changes

The issue requests making boot-md idempotent so BOOT.md does not re-run on every gateway restart.

## Best Possible Solution

Maintainers should decide the BOOT.md contract first; if run-once behavior is desired, store completion/reset state in the canonical state owner and update docs/tests while preserving boot-session isolation and non-blocking startup.

## Reproduction Assessment

Yes, at source level: a configured gateway:startup event still queues boot-md, and runBootOnce skips only missing or empty BOOT.md before executing the boot run. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No, the suggested flag-file guard is not the best repo-shaped solution because it adds sidecar runtime state and leaves reset/upgrade semantics unsettled. A maintainer-approved boot-md contract plus canonical state ownership is the safer path if run-once behavior wins.
