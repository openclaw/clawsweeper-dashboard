## Summary

Keep open. Current main still dispatches `boot-md` on each configured gateway startup and `runBootOnce` has no durable completion guard; current code has improved per-startup workspace dedupe and boot-session isolation, but the once-only lifecycle requested here remains unresolved and conflicts with the current docs wording.

## What This Changes

Request `BOOT.md` startup execution to become once-only or idempotent instead of re-running on every gateway restart.

## Best Possible Solution

Decide the `BOOT.md` lifecycle contract, then either document it as repeatable startup automation and steer first-run work to `BOOTSTRAP.md`, or add a durable per-agent/workspace completion guard with explicit reset semantics while preserving boot-session isolation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable internal hooks, put a non-empty `BOOT.md` in the resolved workspace, and restart the gateway twice; each startup can reach `runBootOnce` because there is no persisted completion check.

## Solution Assessment

Unclear. A sentinel guard is plausible, but the current docs define `BOOT.md` as startup automation, so the maintainable fix needs a product decision about repeatability and reset semantics before changing behavior.
