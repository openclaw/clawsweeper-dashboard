## Summary

Keep open. Current main still runs a non-empty BOOT.md on each gateway startup, while the public docs describe BOOT.md as startup automation and BOOTSTRAP.md as the one-time first-run ritual, so a durable run-once change needs an explicit maintainer contract decision rather than cleanup closure.

## What This Changes

The issue requests changing the internal boot-md startup hook so BOOT.md is not re-executed on every gateway restart.

## Best Possible Solution

Define the BOOT.md contract explicitly, then either add durable once-per-workspace/config-generation behavior with clear reset semantics and docs, or preserve repeatable startup behavior and steer one-time initialization to BOOTSTRAP.md or idempotent BOOT.md checks.

## Reproduction Assessment

Yes, source-level. Enable boot-md, keep a non-empty BOOT.md in the resolved workspace, and restart the gateway twice; each startup creates a fresh handler-local seenWorkspaces Set and runBootOnce has no durable completion check before agentCommand.

## Solution Assessment

Unclear. A sentinel guard would satisfy the requested once-only behavior, but it would change the documented BOOT.md startup contract, so maintainers should choose repeatable versus durable-once semantics and reset rules first.
