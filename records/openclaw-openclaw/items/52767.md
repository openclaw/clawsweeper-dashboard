## Summary

Keep open: current main and v2026.6.8 still run BOOT.md on every gateway startup when the file exists. Merged work mitigated shared-workspace duplication and boot-session isolation, but the requested run-once behavior remains a product/state decision.

## What This Changes

The issue requests changing the bundled boot-md startup hook so BOOT.md does not execute on every gateway restart while the file remains present.

## Best Possible Solution

Maintainers should decide whether BOOT.md is a per-startup checklist or a run-once initialization task; if run-once wins, store completion in a canonical state owner with reset, docs, and doctor behavior while preserving the current isolated boot session.

## Reproduction Assessment

Yes, at source level: a configured gateway:startup event still queues boot-md, and runBootOnce skips only missing or empty BOOT.md before executing the boot run. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No, the suggested flag-file guard is not the best repo-shaped solution because it would add sidecar runtime state and still leave reset/upgrade semantics unsettled. The maintainable path is a maintainer-approved boot-md contract plus canonical state ownership if run-once behavior is desired.
