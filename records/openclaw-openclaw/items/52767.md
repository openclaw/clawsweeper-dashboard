## Summary

Keep open for maintainer judgment: current main and the latest release still run `BOOT.md` whenever the gateway startup hook fires, while recent merged fixes only isolate the boot session and dedupe shared-workspace agents. Changing boot-md to run only once would alter documented startup behavior and needs a product/compatibility decision rather than an automatic cleanup close.

## What This Changes

The issue asks for boot-md to stop re-running `BOOT.md` on every gateway restart, likely by adding idempotent run state or redesigning the startup hook.

## Best Possible Solution

Decide the intended boot-md contract, then either preserve the documented startup-run behavior or add a tested idempotent run-state/reset path that avoids session pollution and does not block channel readiness.

## Reproduction Assessment

Yes, by source inspection: a configured `gateway:startup` event still queues `runBootOnce`, and `runBootOnce` skips only missing or empty `BOOT.md` before running the agent command. I did not run a live gateway restart scenario in this read-only review.

## Solution Assessment

Unclear as proposed. The report's flag-file approach would add runtime state outside the repository's preferred storage model, and the documented startup-hook behavior means maintainers should first decide whether boot-md is recurring startup work or a one-shot initialization mechanism.
