- `P2`: The source-visible behavior and production impact are credible, but the remaining work is a bounded BOOT.md contract decision rather than an emergency runtime outage.
- `impact:session-state`: The report and production follow-up involve boot work interacting with long-lived session state, old attachment history, and possible session contamination during startup.
- `impact:message-loss`: The production follow-up reports Slack disconnects and Telegram/Slack polling stalls while repeated boot work runs.
- `impact:crash-loop`: The issue reports loop-like gateway freeze behavior when heavy BOOT.md work repeats on restart.

## Summary

Keep open. Current main and the latest release still execute a non-empty BOOT.md whenever the gateway startup hook fires, while the docs define BOOT.md as startup automation and BOOTSTRAP.md as the first-run ritual, so changing BOOT.md to durable run-once behavior needs maintainer contract approval rather than cleanup closure.

## What This Changes

Request that boot-md stop rerunning BOOT.md on every gateway restart by adding durable idempotency or redesigning the startup hook.

## Best Possible Solution

Define the BOOT.md contract explicitly, then either add durable once-per-workspace or once-per-config-generation behavior with reset semantics, docs, and regression tests, or preserve repeatable startup behavior and steer one-time initialization to BOOTSTRAP.md or an opt-in boot-md setting.

## Reproduction Assessment

Yes, source-level. Enable boot-md, keep a non-empty BOOT.md in the resolved workspace, and restart the gateway twice; each startup dispatches gateway:startup, constructs a fresh handler-local seenWorkspaces set, and runBootOnce has no durable completion check before agentCommand.

## Solution Assessment

Unclear. A sentinel guard would satisfy the requested once-only behavior, but it would change the documented BOOT.md startup contract, so maintainers should decide repeatable versus durable-once semantics and reset rules first.
