## Summary

Keep open. Current main still executes a non-empty `BOOT.md` during each configured gateway startup; the newer fixes cover per-workspace deduplication and boot-session isolation, but not durable completion or reset semantics.

## What This Changes

The issue asks for `boot-md` to stop re-running `BOOT.md` on every gateway restart and avoid repeated one-time initialization work.

## Best Possible Solution

Decide the `BOOT.md` contract explicitly, then either add a durable per-agent/workspace completion guard with documented reset semantics or keep `BOOT.md` repeatable and steer one-time initialization to `BOOTSTRAP.md`, while preserving boot-session isolation and startup readiness safety.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable internal hooks, place a non-empty `BOOT.md` in the resolved workspace, and restart the gateway twice; each startup dispatch reaches `runBootOnce`, which has no persistent completion check.

## Solution Assessment

Unclear. A sentinel guard is plausible, but current docs define `BOOT.md` as startup automation, so the maintainable fix needs a product decision about reset/re-run semantics before changing behavior.
