## Summary

Keep open. Current main still lacks the Feishu read-only group reply guard, and the PR is useful scoped work, but the proposed patch still has two merge-blocking correctness gaps: the new no-op dispatchers do not satisfy the shared reply dispatcher contract, and bare Feishu group chat IDs bypass the outbound block.

## What This Changes

The PR adds a Feishu `disableGroupReplies` config/docs switch, routes group replies through no-op dispatchers when enabled, blocks some outbound group targets, adds focused tests, and refreshes the secrets baseline.

## Best Possible Solution

Keep the feature in the Feishu plugin and finish it as a narrow channel guardrail: add the config/schema/docs key, use one shared type-complete no-op reply dispatcher for all accepted group inbound paths when the flag is enabled, and reject every outbound target whose resolved Feishu receive type is `chat_id` while preserving DM/user sends.

## Reproduction Assessment

Yes. Current-main source inspection proves the feature is absent, and the PR bypass is reproducible with `disableGroupReplies: true` plus a bare `oc_*` target through `resolveFeishuSendTarget`; the no-op dispatcher issue is reproducible through the current `ReplyDispatcher` type contract.

## Solution Assessment

No. The feature direction is a good plugin-local solution, but this patch is not yet the best implementation because it uses prefix-based outbound blocking and introduces no-op dispatchers that are not type-complete.
