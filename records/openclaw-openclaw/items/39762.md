## Summary

Keep this PR open. Current main still lacks the Feishu read-only group mode, and the PR is useful but not yet merge-ready because the proposed outbound guard still allows bare Feishu chat IDs and the new no-op dispatcher literals do not satisfy the current ReplyDispatcher contract.

## What This Changes

The PR adds a Feishu `disableGroupReplies` config/schema/docs switch, swaps group replies to no-op dispatchers in inbound paths, blocks prefixed outbound group targets, adds targeted tests, and refreshes `.secrets.baseline`.

## Best Possible Solution

Keep the feature in the Feishu plugin, but finish it as a narrow channel-level guardrail: add the config/schema/docs key, use a shared type-complete no-op ReplyDispatcher in both accepted group inbound paths, reject every outbound target classified as a Feishu group chat when the flag is enabled, cover raw and prefixed group IDs in tests, and rebase away unrelated baseline churn where possible.

## Reproduction Assessment

Yes. The current-main absence is proven by source/docs inspection, and the remaining PR bypass can be reproduced with a focused `resolveFeishuSendTarget` case using `disableGroupReplies: true` and a bare `oc_*` target.

## Solution Assessment

No. The requested feature is a good fit for the Feishu plugin, but this patch is not yet the best implementation because the guard is prefix-based instead of receive-type-based and the no-op dispatcher should satisfy the shared ReplyDispatcher contract.

Reject `chat_id` targets and cover bare IDs in tests.
