## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate channel file URLs before forwarding:** `extensions/synology-chat/src/client.ts:177`
  - body: `sendFileUrlToChannel` posts the raw `fileUrl`, while the existing DM media path first calls `assertSafeWebhookFileUrl`. Group media sends can make the NAS fetch private or otherwise blocked URLs, so run the same safety check before building the channel payload.
  - confidence: 0.94
- **[P2] Resolve channel override URLs for group sends:** `extensions/synology-chat/src/channel.ts:354`
  - body: `sendText` and `sendMedia` route every `channel:` or `group:` target through `account.incomingUrl`, ignoring `channels.<id>.incomingUrl`. Shared-account multi-channel sends can post to the wrong channel; lookup the target key before falling back to the account URL.
  - confidence: 0.86
- **[P2] Honor channel allowlists before open policy:** `extensions/synology-chat/src/group-access.ts:57-58`
  - body: This early return allows every sender whenever the account uses `groupPolicy: "open"`, so `channels.<id>.allowFrom` cannot restrict a sensitive channel under an open account default. Resolve the channel override before accepting the open policy.
  - confidence: 0.9
- **[P2] Coerce allowlist entries before lowercasing:** `extensions/synology-chat/src/group-access.ts:86-87`
  - body: The config schema is passthrough, so raw JSON can contain numeric Synology IDs in `groupAllowFrom` or `channels.*.allowFrom`. Those entries reach `.toLowerCase()` and throw a `TypeError`, turning authorization into a 500 instead of allow/deny.
  - confidence: 0.87
- **[P2] Normalize target prefixes before group checks:** `extensions/synology-chat/src/channel.ts:353`
  - body: The target resolver accepts `channel:` and `group:` prefixes case-insensitively, but these send branches only check lowercase `startsWith`. Inputs like `CHANNEL:general` take the DM path with `user_ids`; normalize once before the text and media checks.
  - confidence: 0.85
- **[P2] Mock the channel file helper export:** `extensions/synology-chat/src/channel.test-mocks.ts:76-80`
  - body: `channel.ts` now imports `sendFileUrlToChannel`, but this shared `./client.js` Vitest mock omits that export. Tests that load the real plugin module with this mock can fail during module initialization before reaching assertions.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive group authorization and channel media paths that still need repair.

- **[high] Channel media bypasses file URL SSRF policy:** `extensions/synology-chat/src/client.ts:177`
  - body: The new channel media helper forwards `file_url` without the existing `assertSafeWebhookFileUrl` check, so group media sends can make the NAS fetch private or blocked network URLs that current DM media rejects.
  - confidence: 0.94

  - confidence: 0.9

  - body: The new group allowlist path lowercases raw array entries without runtime coercion; numeric Synology IDs from JSON config can throw and turn group traffic into server errors instead of controlled access decisions.
