## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate channel file URLs before forwarding:** `extensions/synology-chat/src/client.ts:177`
  - body: `sendFileUrlToChannel` builds the channel payload with the raw `fileUrl`, while the existing DM media helper first calls `assertSafeWebhookFileUrl`. Group media sends can therefore make the NAS fetch private or otherwise blocked URLs again; run the same validation before posting the channel webhook payload.
  - confidence: 0.92
- **[P2] Honor channel allowlists before open group policy:** `extensions/synology-chat/src/group-access.ts:57-58`
  - body: `channels.<id>.allowFrom` is documented as a per-channel restriction, but this early return allows every sender whenever the account `groupPolicy` is `open`. Resolve any channel override first so a sensitive channel can still be restricted under an open default.
  - confidence: 0.88
- **[P2] Coerce allowlist entries before lowercasing:** `extensions/synology-chat/src/group-access.ts:86-87`
  - body: Raw config can contain numeric Synology IDs in `groupAllowFrom` or `channels.*.allowFrom`. Those entries flow into `effectiveAllowFrom` and then call `id.toLowerCase()`, turning webhook authorization into a `TypeError` instead of an allow/deny decision.
  - confidence: 0.86
- **[P2] Match group target prefixes case-insensitively:** `extensions/synology-chat/src/channel.ts:353`
  - body: The target resolver accepts `channel:` and `group:` prefixes with `/i`, but the send branches only check lowercase `startsWith`. Inputs like `CHANNEL:general` are accepted as ID-like and then use the DM payload path with `user_ids`; normalize case before both text and media branch checks.
  - confidence: 0.86
- **[P2] Export sendFileUrlToChannel from the client mock:** `extensions/synology-chat/src/channel.test-mocks.ts:76-80`
  - body: `channel.ts` imports `sendFileUrlToChannel`, but this shared `./client.js` mock omits that export. Tests that load the real plugin module with this mock can fail during module initialization before exercising the new channel media path.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `docs/channels/synology-chat.md:9`
  - body: This is a user-facing `feat` change to the bundled Synology Chat plugin, but the PR does not update `CHANGELOG.md`. Add one single-line entry under the active version with appropriate contributor credit before merge.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR changes group webhook authorization and outbound channel media delivery and still has concrete SSRF and access-control concerns before merge.

Concerns:

- **[high] Channel media bypasses file URL SSRF policy:** `extensions/synology-chat/src/client.ts:177`
  - body: The new channel media helper forwards `file_url` without the existing `assertSafeWebhookFileUrl` check, so group media sends can make the NAS fetch private or blocked network URLs that the current DM media path rejects.
  - confidence: 0.92
- **[medium] Per-channel allowlist bypass:** `extensions/synology-chat/src/group-access.ts:57`
  - body: `evaluateSynologyChatGroupAccess` returns allowed for `groupPolicy: "open"` before reading `channels.<id>.allowFrom`, so a stricter per-channel override under an open account would still accept every sender.
  - confidence: 0.86
- **[low] Numeric allowlists can crash authorization:** `extensions/synology-chat/src/group-access.ts:87`
  - body: The new group allowlist path lowercases raw array entries without runtime coercion; numeric Synology IDs from JSON config can throw and turn group traffic into server errors instead of controlled access decisions.
  - confidence: 0.78
