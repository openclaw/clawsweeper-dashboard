## Review Findings

Overall confidence: 0.87

Full review comments:

- **[P2] Honor channel allowlists before open group policy:** `extensions/synology-chat/src/group-access.ts:57-58`
  - body: channels.<id>.allowFrom is documented as a per-channel restriction, but this early return allows every sender whenever the account groupPolicy is open. Resolve the channel override before honoring the open default so a sensitive channel can still be restricted.
  - confidence: 0.88
- **[P2] Coerce allowlist entries before lowercasing:** `extensions/synology-chat/src/group-access.ts:86-87`
  - body: Raw config can contain numeric Synology IDs in groupAllowFrom or channels.*.allowFrom arrays. Those entries flow into effectiveAllowFrom and then call id.toLowerCase(), turning webhook authorization into a TypeError instead of an allow/deny decision.
  - confidence: 0.84
- **[P2] Match group target prefixes case-insensitively:** `extensions/synology-chat/src/channel.ts:353`
  - body: The target resolver accepts channel:/group: prefixes with /i, but the send branches only check lowercase startsWith. Inputs like CHANNEL:general are accepted as ID-like and then fall through to the DM payload path with user_ids; normalize case before both text and media checks.
  - confidence: 0.86
- **[P2] Export sendFileUrlToChannel from the client mock:** `extensions/synology-chat/src/channel.test-mocks.ts:76-80`
  - body: channel.ts imports sendFileUrlToChannel, but this shared ./client.js mock omits that export. Tests that load channel.ts with the mock can fail during module initialization before exercising the new channel media path.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `docs/channels/synology-chat.md:9`

## Security Review

Status: needs_attention

Summary: The PR changes group webhook authorization and still has concrete access-control and denial-of-service concerns before merge.

Concerns:

- **[medium] Per-channel allowlist bypass:** `extensions/synology-chat/src/group-access.ts:57`
  - body: evaluateSynologyChatGroupAccess returns allowed for groupPolicy open before reading channels.<id>.allowFrom, so an operator who configures a stricter per-channel override under an open account would still allow every sender in that channel.
  - confidence: 0.86
- **[low] Numeric allowlist entries can crash authorization:** `extensions/synology-chat/src/group-access.ts:87`
  - body: The new group allowlist path lowercases raw array entries without runtime coercion. Numeric Synology IDs from JSON config can throw during webhook authorization, turning group traffic into server errors rather than controlled access decisions.
  - confidence: 0.78
