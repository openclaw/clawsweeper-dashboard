## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate channel file URLs before forwarding:** `extensions/synology-chat/src/client.ts:177`
  - body: `sendFileUrlToChannel` posts the raw `fileUrl`, while the existing DM media path first calls `assertSafeWebhookFileUrl`. Group media sends can therefore make the NAS fetch URLs that current Synology Chat media policy rejects; run the same validation before building the channel payload.
  - confidence: 0.92
- **[P2] Honor channel allowlists before open group policy:** `extensions/synology-chat/src/group-access.ts:57-58`
  - body: The documented resolution order says per-channel `allowFrom` wins, but this branch returns allowed for every sender whenever the account has `groupPolicy: "open"`. Resolve the channel override first so sensitive channels can remain restricted under an open default.
  - confidence: 0.88
- **[P2] Coerce allowlist entries before lowercasing:** `extensions/synology-chat/src/group-access.ts:86-87`
  - body: Raw config can provide numeric Synology IDs in `groupAllowFrom` or `channels.*.allowFrom`; those entries reach `id.toLowerCase()` and turn authorization into a `TypeError`. Convert entries to strings or type-guard them before wildcard and equality checks.
  - confidence: 0.86
- **[P2] Normalize group target prefixes before branching:** `extensions/synology-chat/src/channel.ts:353`
  - body: The target resolver accepts `channel:` and `group:` prefixes case-insensitively, but the send branches only check lowercase `startsWith`. Inputs like `CHANNEL:general` are accepted as ID-like and then use the DM payload path with `user_ids`; normalize case before both text and media branch checks.
  - confidence: 0.86
- **[P2] Export the channel media helper from the client mock:** `extensions/synology-chat/src/channel.test-mocks.ts:76-80`
  - body: `channel.ts` imports `sendFileUrlToChannel`, but this shared `./client.js` Vitest mock omits that export. Tests that load the real plugin module with this mock can fail during module initialization before exercising the new channel media path.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds group webhook authorization and channel media delivery with concrete SSRF and access-control regressions still present.

Concerns:

- **[high] Channel media bypasses file URL SSRF policy:** `extensions/synology-chat/src/client.ts:177`
  - body: The new channel media helper forwards `file_url` without the existing `assertSafeWebhookFileUrl` check, so group media sends can make the NAS fetch private or otherwise blocked network URLs that the current DM media path rejects.
  - confidence: 0.92
- **[medium] Per-channel allowlist bypass:** `extensions/synology-chat/src/group-access.ts:57`
  - body: `evaluateSynologyChatGroupAccess` returns allowed for `groupPolicy: "open"` before reading `channels.<id>.allowFrom`, so a stricter per-channel override under an open account would still accept every sender.
  - confidence: 0.86
- **[low] Numeric allowlists can crash authorization:** `extensions/synology-chat/src/group-access.ts:87`
  - body: The new group allowlist path lowercases raw array entries without runtime coercion; numeric Synology IDs from JSON config can throw and turn group traffic into server errors instead of controlled access decisions.
  - confidence: 0.78
