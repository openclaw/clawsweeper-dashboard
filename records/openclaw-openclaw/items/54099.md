## Summary

Keep this PR open. Current main is still Synology Chat DM-only, so the branch is not obsolete, and the later PR head appears to address earlier review feedback around `requireMention` and group reply delivery. The patch still has review-blocking access-control, outbound routing, and test-mock defects that should be fixed before merge.

## What This Changes

The PR expands the bundled Synology Chat plugin, docs, and tests to support group/channel webhook accounts, group policies and allowlists, mention gating, channel replies/media, and group session routing.

## Best Possible Solution

Keep the PR open and fix the branch in place: make per-channel group allowlists take precedence as documented, normalize allowlist entries defensively, make outbound group target detection match the resolver's case-insensitive behavior, sync the shared client mock, then run the targeted Synology Chat extension tests and the appropriate changed gate.

## Reproduction Assessment

Yes. The branch-level problems are reproducible from code inspection with focused unit cases: an open account plus channel `allowFrom` bypasses the override, uppercase `CHANNEL:` targets pass resolver checks but send through the DM path, numeric allowlist entries can throw on `.toLowerCase()`, and the shared mock lacks a newly imported client export.

## Solution Assessment

No. The overall feature direction is plausible and current main does not already solve it, but this patch is not yet the best mergeable implementation because it leaves access-control semantics, outbound routing, and test infrastructure inconsistent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor channel allowlists before open group policy:** `extensions/synology-chat/src/group-access.ts:57-58`
  - body: `channels.<id>.allowFrom` is documented as a per-channel allowlist, but this returns `allowed` for every sender as soon as the account policy is `open`. A mixed setup with an open default and a restricted channel will accept unauthorized users, so resolve channel overrides before the open-policy return and let configured `allowFrom` restrict the channel.
  - confidence: 0.88
- **[P2] Normalize allowlist entries before lowercasing:** `extensions/synology-chat/src/group-access.ts:86-87`
  - body: Raw plugin config can pass numeric Synology IDs in `groupAllowFrom` or `channels.*.allowFrom`, and `parseAllowedUserIds`/the passthrough channel override path do not coerce array entries. Calling `id.toLowerCase()` on a number turns authorized group webhooks into 500s instead of an allow/deny decision; coerce entries to strings before wildcard and equality checks.
  - confidence: 0.82
- **[P2] Match group target prefixes case-insensitively:** `extensions/synology-chat/src/channel.ts:353`
  - body: The resolver now treats `channel:` and `group:` prefixes case-insensitively, but the send branches only check lowercase `startsWith`. Inputs like `CHANNEL:general` are accepted as ID-like and then fall through to the DM `sendMessage`/`sendFileUrl` path with `user_ids`, so normalize case or use the same case-insensitive predicate in both text and media sends.
  - confidence: 0.86
- **[P2] Add sendFileUrlToChannel to the shared mock:** `extensions/synology-chat/src/channel.test-mocks.ts:76-80`
  - body: `channel.ts` imports `sendFileUrlToChannel`, but the shared `./client.js` mock used by the Synology integration tests does not export it. Tests that import `channel.test-mocks.ts` before loading `channel.ts` can fail at module load with a missing mock export; add the new mocked function alongside `sendFileUrl`.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR changes group webhook authorization and currently leaves a concrete per-channel allowlist bypass under `groupPolicy: "open"`.

Concerns:

- **[medium] Per-channel allowlist bypass:** `extensions/synology-chat/src/group-access.ts:57`
  - body: `evaluateSynologyChatGroupAccess` returns allowed for `groupPolicy: "open"` before reading `channels.<id>.allowFrom`, so an operator who configures a stricter channel override under an open account will still allow every sender in that channel.
  - confidence: 0.86
- **[low] Numeric allowlist entries can crash authorization:** `extensions/synology-chat/src/group-access.ts:87`
  - body: The new group allowlist path lowercases entries without runtime coercion. Numeric IDs from JSON config can throw during webhook authorization, turning group traffic into server errors rather than controlled access decisions.
  - confidence: 0.78
