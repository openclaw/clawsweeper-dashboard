## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Restore the sleep import before calling retry delays:** `extensions/synology-chat/src/client.ts:143-144`
  - body: `sendToChannel` calls `sleep`, but this PR head no longer imports it in `client.ts`. TypeScript will fail with `Cannot find name 'sleep'` before the Synology Chat plugin can build or run.
  - confidence: 0.94
- **[P1] Validate channel file URLs before forwarding:** `extensions/synology-chat/src/client.ts:177`
  - body: `sendFileUrlToChannel` posts the raw `fileUrl`, while the existing DM media path first runs `assertSafeWebhookFileUrl`. Group media sends need the same validation so the NAS cannot be asked to fetch private or blocked URLs that current OpenClaw rejects.
  - confidence: 0.93
- **[P2] Honor channel allowlists before open policy:** `extensions/synology-chat/src/group-access.ts:57-58`
  - body: This early return accepts every group sender whenever the account default is `groupPolicy: "open"`, so a stricter `channels.<id>.allowFrom` override never takes effect. Resolve the channel override before accepting the open default.
  - confidence: 0.9
- **[P2] Coerce allowlist entries before lowercasing:** `extensions/synology-chat/src/group-access.ts:86-87`
  - body: Raw JSON config can contain numeric Synology IDs in `groupAllowFrom` or `channels.*.allowFrom`. Those entries reach `.toLowerCase()` here and throw a `TypeError`, turning authorization into a 500 instead of an allow/deny decision.
  - confidence: 0.88
- **[P2] Match group target prefixes case-insensitively:** `extensions/synology-chat/src/channel.ts:353`
  - body: The target resolver accepts `channel:` and `group:` prefixes case-insensitively, but the send branches only check lowercase `startsWith`. Inputs like `CHANNEL:general` can fall through to the DM path with `user_ids` instead of using the channel webhook.
  - confidence: 0.86
- **[P2] Export the channel media helper from the client mock:** `extensions/synology-chat/src/channel.test-mocks.ts:76-80`
  - body: `channel.ts` now imports `sendFileUrlToChannel`, but this shared `./client.js` mock does not provide it. Tests that load the plugin with this mock can fail during module initialization before reaching the new media path.
  - confidence: 0.9
- **[P2] Use the runtime config snapshot instead of loadConfig:** `extensions/synology-chat/src/inbound-turn.ts:79`
  - body: This branch calls `rt.config.loadConfig()` from bundled plugin runtime code, while current main uses `rt.config.current()` and the runtime type marks `loadConfig` deprecated and architecture-guarded. Keep the prepared runtime config snapshot instead of reloading config on the message path.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR introduces security-sensitive channel media forwarding and group authorization behavior that needs repair before merge.

Concerns:

- **[high] Channel media bypasses file URL SSRF policy:** `extensions/synology-chat/src/client.ts:177`
  - body: The new channel media helper forwards `file_url` without the existing `assertSafeWebhookFileUrl` check, so group media sends can make the NAS fetch URLs that current DM media delivery rejects as private or blocked.
  - confidence: 0.93
- **[medium] Per-channel allowlist can be bypassed:** `extensions/synology-chat/src/group-access.ts:57`
  - body: `evaluateSynologyChatGroupAccess` returns allowed for `groupPolicy: "open"` before reading `channels.<id>.allowFrom`, so a stricter per-channel override under an open account would still accept every sender.
  - confidence: 0.9
- **[low] Numeric allowlists can crash authorization:** `extensions/synology-chat/src/group-access.ts:87`
  - body: The new group allowlist path lowercases raw array entries without runtime coercion; numeric Synology IDs from JSON config can throw and turn group traffic into server errors instead of controlled access decisions.
  - confidence: 0.84
