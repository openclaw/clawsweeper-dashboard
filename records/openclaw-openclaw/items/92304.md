## Summary

The PR targets a real timezone-loss bug, but it moves preservation into a client-side read/merge/write path that can reject valid edits and race with concurrent updates; the canonical cron service already owns the existing schedule and is the safer fix boundary. The supplied proof is useful source-level evidence, but it does not execute the changed handler through its new lookup and update calls.

## What This Changes

The PR makes `cron edit --cron` load the existing job and preserve omitted `tz` and `staggerMs` fields, while adapting one CLI test to mock the added schedule lookup.

## Best Possible Solution

Preserve omitted cron metadata atomically where the stored job is updated in `src/cron/service/jobs.ts`, retaining explicit overrides and conversion behavior, with focused service and CLI regressions for `tz`, `staggerMs`, and `--exact`.

## Reproduction Assessment

Yes, at source level: `--cron` constructs a replacement schedule without an omitted `tz`, and current service logic does not restore that field. Current main separately preserves omitted `staggerMs`, so the central remaining failure is timezone loss.

## Solution Assessment

No. Preserving cron metadata is correct, but the stored-job service update is the narrowest atomic owner boundary; a CLI `cron.list` read followed by `cron.update` adds avoidable lookup failures and stale-state races.
