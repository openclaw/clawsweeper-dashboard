## What This Changes

Routes OpenRouter video submit/poll JSON parsing through the shared bounded provider JSON reader and adds regression coverage for oversized streamed success bodies plus the private-network denial assertion.

## Best Possible Solution

Land the narrow OpenRouter video change if maintainers accept applying the shared 16 MiB provider JSON cap to submit and poll control responses while keeping private-network denial unchanged.

## Reproduction Assessment

Yes from source and PR proof: current main calls `response.json()` for OpenRouter video submit/poll success bodies, while the PR body shows a loopback streamed response now stops at the 16 MiB cap. I did not rerun the harness in this read-only review.

## Solution Assessment

Yes. Reusing the shared `readProviderJsonResponse` in the local OpenRouter parse helper is the narrowest maintainable fix because both submit and poll already share that helper; a custom OpenRouter-only cap would duplicate existing SDK behavior.
