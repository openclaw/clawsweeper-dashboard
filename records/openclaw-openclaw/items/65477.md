Keep open. Current main and the latest release still omit active provider/model attribution from the per-response usage footer, while several open PRs cover overlapping footer/model work and need maintainer consolidation rather than another automated fix branch.

## What This Changes

The issue requests adding the active model name to the per-response usage footer shown in messaging channels.

## Best Possible Solution

Consolidate the open implementations into one narrow footer change that renders the actual resolved provider/model while preserving token, cache, estimated-cost, and session output semantics, then decide separately whether broader response-footer or inherited-default config belongs in the same change.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but source inspection is high confidence: current main and v2026.5.18 do not render provider/model in the usage footer while the runner already has resolved provider/model values.

## Solution Assessment

Is this the best way to solve the issue? Yes in direction: threading the resolved provider/model into the existing usage-footer path is the narrow maintainable fix, but maintainers still need to choose the final wording and scope across the overlapping PRs.
