## Summary

Keep open. Current main still has the same product gap: the per-response usage footer is built from token/cache/cost/session data only, while the reply runner already knows the actual provider and model that answered.

## What This Changes

Request to show the active model identifier in the messaging-channel per-response `/usage` footer.

## Best Possible Solution

Add actual provider/model attribution to the existing response-usage footer path by passing `providerUsed` and `modelUsed` from `runReplyAgent` into the formatter, using the existing provider/model label helper, and preserving current token/cache/cost/session behavior with focused formatter and runner tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but the source-level check is high confidence: enabling `responseUsage` reaches a formatter that has no provider/model input and therefore cannot emit the requested label.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is to render the actual resolved provider/model in the existing usage-footer path, while reviewing or consolidating the already-open implementation PRs instead of spawning another competing repair job.
