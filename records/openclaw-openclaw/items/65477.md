## Summary

Keep open. Current main still builds the per-response usage footer from token/cache/cost/session data only; the runner has the resolved provider/model, but that data is not passed into the footer formatter, and the related implementation PRs remain open and unmerged.

## What This Changes

Add the active provider/model name to the per-response `/usage` footer shown in messaging channel replies.

## Best Possible Solution

Thread the resolved `providerUsed` and `modelUsed` from `runReplyAgent` into the usage-footer formatter, format the label with the existing provider/model helper, and keep the current token/cache/cost/session behavior covered by focused formatter and runner tests.

## Reproduction Assessment

Not applicable as a feature request, but the source-level check is high confidence: enabling `responseUsage` reaches a formatter that has no provider/model input, so current main cannot emit the requested label in the usage footer.

## Solution Assessment

Yes. The narrow maintainable solution is to reuse the already-resolved runtime provider/model at the existing footer assembly point; the open PRs should be reviewed or consolidated rather than starting another independent fix.
