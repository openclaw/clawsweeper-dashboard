## Summary

Keep open. Current `main` and latest release still format the response usage footer from token/cache/cost/session data only, while `runReplyAgent` already has the resolved provider/model available; this remains a valid, narrow feature request pending consolidation of overlapping open implementation PRs.

## What This Changes

Request to show the active provider/model label in the per-response `/usage` footer for messaging channels.

## Best Possible Solution

Land or consolidate a narrow implementation that renders the actual resolved provider/model in the existing usage-footer path while preserving token/cache/cost/session output.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but source inspection is high confidence: the `/usage` formatter has no provider/model input and the runner does not pass its resolved provider/model into the footer call.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrow maintainable direction is to render the actual resolved provider/model in the existing usage-footer path; maintainers should choose among or consolidate the open implementation PRs before changing the footer contract.
