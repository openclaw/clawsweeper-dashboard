## What This Changes

Adds a module-level 5ms cache to `reconcileInspectableTasks()` and clears it from the task-maintenance test reset helper.

## Best Possible Solution

Share one reconciled task snapshot inside `getStatusSummary()`, or make any broader cache lifecycle-owned with mutation/config invalidation and stale-read tests.

## Reproduction Assessment

Yes. Source inspection shows `getStatusSummary()` calls two task helper paths that each call `reconcileInspectableTasks()` before `listTaskRecords()` clones and sorts task records.

## Solution Assessment

No. The PR’s module-level 5ms TTL cache is less safe and likely ineffective on the reported slow path; a per-status shared snapshot is the narrower maintainable fix.

Full review comments:

- **[P1] Timestamp the cache after reconciliation completes:** `src/tasks/task-registry.maintenance.ts:901`
  - body: The cache entry stores `now` from before `ensureTaskRegistryReady()` and the full `listTaskRecords().map(...)` reconciliation. On the reported 10K-task workload, the first reconciliation takes hundreds or thousands of milliseconds, so the immediate second call can see an entry older than the 5ms TTL and miss the cache.
  - confidence: 0.98
- **[P2] Invalidate cached task snapshots on lifecycle changes:** `src/tasks/task-registry.maintenance.ts:886-887`
  - body: Returning this module-level cached array makes task inspection fresh only by wall-clock time. The helper also feeds audit and restart-blocker reads, while the registry emits lifecycle events and `configureTaskRegistryMaintenance()` can change reconciliation inputs; scope the optimization to `getStatusSummary()` or invalidate on those lifecycle/config changes with tests.
  - confidence: 0.91
