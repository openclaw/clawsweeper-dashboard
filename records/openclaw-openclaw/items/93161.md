## Summary

This PR should stay open: the performance bug is real, but the submitted cache timestamps entries before the expensive reconciliation work, so the reported large-registry path would miss the 5ms cache on the immediately following call; the module-level cache also lacks lifecycle invalidation for nearby task/config changes.

## What This Changes

The PR adds a module-level 5ms memoization cache around `reconcileInspectableTasks()` in `src/tasks/task-registry.maintenance.ts` and clears it from the task-maintenance test reset helper.

## Best Possible Solution

Share one reconciled task snapshot inside the status-summary path, or make any broader cache lifecycle-owned with mutation/config invalidation and focused stale-read tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `getStatusSummary()` calls two helpers that each call `reconcileInspectableTasks()`, and the PR's pre-computation cache timestamp expires during slow reconciliation before the second helper can use it.

## Solution Assessment

No, this is not the best way to solve the issue because a 5ms cache measured from call start fails the reported slow path and a module-level cache needs lifecycle invalidation; a per-status shared snapshot is narrower.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.97

Full review comments:

- **[P1] Timestamp the cache after reconciliation completes:** `src/tasks/task-registry.maintenance.ts:901`
  - body: The cache entry stores the `now` captured before `ensureTaskRegistryReady()` and the full `listTaskRecords().map(...)` reconciliation run. On the reported 10K-task workload the first reconciliation takes hundreds or thousands of milliseconds, so by the time the immediate second call starts, `Date.now() - reconcileCache.time` is already greater than the 5ms TTL and the cache is missed. Stamp the entry after computing `result`, or use an explicit dirty cache instead.
  - confidence: 0.98
- **[P2] Invalidate cached task snapshots on lifecycle changes:** `src/tasks/task-registry.maintenance.ts:886-887`
  - body: Returning a module-level cached array here makes task inspection fresh only by wall-clock time. This helper also feeds audit and restart-blocker reads, while the task registry emits `restored`, `upserted`, and `deleted` events and `configureTaskRegistryMaintenance()` can change reconciliation inputs. A task mutation or config change followed by another inspection inside the TTL can report stale state; scope the optimization to `getStatusSummary()` or add lifecycle invalidation with tests.
  - confidence: 0.91
