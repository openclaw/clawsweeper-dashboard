## Summary

Keep open: the PR targets a real duplicated `openclaw status` task-reconciliation path, but the submitted module-level 5ms cache is not merge-ready because its timestamp can expire before the second call on the reported workload, it lacks lifecycle/config invalidation, and the contributor proof is only tests/build/grep output.

## What This Changes

The PR adds a module-level 5ms memoization cache around `reconcileInspectableTasks()` and clears it from the task-maintenance test reset helper.

## Best Possible Solution

Compute one reconciled task snapshot inside `getStatusSummary()` and share it with the registry summary and audit helpers, or make any broader cache lifecycle-owned with mutation/config invalidation and stale-read regression tests.

## Reproduction Assessment

Yes. Source inspection shows `getStatusSummary()` calls two task helpers that each call `reconcileInspectableTasks()` before `listTaskRecords()` clones and sorts task records; I did not run a live 10K-task timing repro in this read-only review.

## Solution Assessment

No. The module-level TTL cache is less safe and likely ineffective on the reported slow path; a per-status shared snapshot is the narrower maintainable fix unless maintainers deliberately choose a lifecycle-owned cache.

- **[P1] Timestamp the cache after reconciliation completes:** `src/tasks/task-registry.maintenance.ts:901`
  - body: The cache stores `now` from before `ensureTaskRegistryReady()` and the full `listTaskRecords().map(...)` reconciliation.   - confidence: 0.98

Task registry writes already emit lifecycle events and `configureTaskRegistryMaintenance()` can change reconciliation inputs, so audit/restart-blocker reads can use stale task state unless the optimization is scoped to `getStatusSummary()` or invalidated with tests.
  - confidence: 0.92
