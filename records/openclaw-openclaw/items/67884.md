## Summary

Keep this PR open, but do not merge the current patch. Current main still has a generic hook-context normalization mismatch, so the PR is addressing a real area; however the proposed blanket stripping would break resolver-owned typed conversation ids such as Discord `user:` and `channel:` identities, matching the existing bot review concern.

## What This Changes

The PR adds a shared `normalizeMessagingConversationId` helper, applies it to plugin message hook contexts, and updates mapper tests to expect stripped channel/generic prefixes for received and sent hook contexts.

## Best Possible Solution

Revise the mapper so generic channel-prefixed contexts can correlate with inbound claims without stripping semantically meaningful resolver-owned ids. Add regression coverage for both sides: a no-resolver generic prefix case and a resolver-owned typed id case such as the existing claim-chat fixture or Discord-style `user:`/`channel:` identities.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to apply the PR change to a canonical sent or received context whose `conversationId` is `user:<id>` or `channel:<id>`; the new helper would return the bare id, while current resolver and binding tests expect the typed id to remain intact.

## Solution Assessment

No. The PR solves the generic fallback mismatch too broadly; the safer fix is to centralize message-hook conversation resolution around the same resolver/fallback contract used by inbound claims, or otherwise only strip provider/channel wrappers without removing typed `user:`/`channel:` identities.
