## Summary

Keep open and do not merge as-is: the PR strips typed `user:` and `channel:` conversation ids from message hook contexts even though current Discord resolver, inbound claim, and binding paths treat those prefixes as canonical, and the PR lacks real behavior proof.

## What This Changes

The PR adds a shared conversation-id normalizer for message hook contexts and updates mapper tests to expect Telegram, Slack, and Discord-style prefixes to be stripped.

## Best Possible Solution

Keep hook conversation ids aligned with inbound claims by normalizing only non-semantic wrapper prefixes, while preserving resolver-owned typed identities and adding regression coverage for both resolver and fallback paths.

## Reproduction Assessment

Yes. From source inspection, applying the PR helper to `user:<id>` or `channel:<id>` makes message hooks see bare ids while inbound claim and Discord binding paths preserve typed ids; no local test run was performed because this review is read-only.

## Solution Assessment

No. The PR strips too broadly; the safer fix is resolver-aware normalization or a narrower wrapper-prefix cleanup that does not remove provider-owned typed grammar.
