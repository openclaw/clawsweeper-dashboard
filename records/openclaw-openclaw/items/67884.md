## Summary

Keep open: the patch is a focused attempt at hook correlation, but the current implementation is unsafe because it strips provider-owned typed conversation IDs that Discord and resolver-backed hooks intentionally preserve. It also lacks real behavior proof for the reported Telegram/Slack pending-lookup problem.

## What This Changes

The PR adds a normalizer for plugin hook conversation IDs, applies it in `toPluginMessageContext`, and updates mapper tests to expect Telegram, Slack, and Discord-style prefixes to be stripped.

## Best Possible Solution

Preserve provider-owned typed IDs in plugin hook contexts, normalize only the specific fallback or wrapper-prefix cases that caused the mismatch, and verify the final contract with focused Discord binding tests plus real Telegram/Slack hook proof.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows the diff strips `user:` and `channel:` from message hook contexts while inbound claim resolver paths and Discord binding tests preserve those IDs. The original Telegram/Slack pending-lookup bug is not proven with real setup evidence in the PR.

## Solution Assessment

No. Applying generic prefix stripping to every plugin message context is too broad; the maintainable fix should be resolver-aware or fallback-only and preserve channel-owned identity grammar.
