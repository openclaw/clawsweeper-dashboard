## Summary

Keep open. The PR targets a real hook-correlation problem, but the current diff strips semantically meaningful `user:` and `channel:` conversation identities, which current resolver and Discord binding paths intentionally preserve.

## What This Changes

The PR adds a shared conversation-id normalizer for plugin message hook contexts and updates mapper tests to expect stripped Telegram, Slack, and Discord-style prefixes.

## Best Possible Solution

Revise the mapper so generic provider/channel wrapper prefixes can correlate with inbound claims while preserving resolver-owned typed identities, with regression coverage for both paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to apply the PR helper to a sent or received context whose `conversationId` is `user:<id>` or `channel:<id>`; the message hook sees the bare id while inbound claim and binding state keep the typed id.

## Solution Assessment

No. The PR strips too broadly; the safer fix is to share the resolver/fallback contract used by inbound claims or only remove provider wrapper prefixes without removing typed conversation grammar.
