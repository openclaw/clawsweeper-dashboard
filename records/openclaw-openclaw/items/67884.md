## Summary

Keep open: the patch tries to fix hook correlation, but it strips semantic Discord `user:` and `channel:` conversation identities from plugin message contexts while current inbound claim and binding paths preserve those identities. The earlier Greptile and ClawSweeper review comments match current-main source evidence, and the PR still lacks real after-fix behavior proof.

## What This Changes

The PR adds a shared message-hook conversation-id normalizer, applies it to plugin message contexts, and updates mapper tests for prefixed Telegram, Slack, and Discord-style ids.

## Best Possible Solution

Keep hook conversation ids aligned by normalizing only non-semantic wrapper prefixes or fallback paths while preserving resolver-owned typed identities, with regression coverage for both generic fallback and Discord resolver/binding behavior.

## Reproduction Assessment

Yes. Source inspection shows the PR would strip `user:` and `channel:` in plugin message contexts while current inbound claim and Discord binding paths preserve those typed identities.

## Solution Assessment

No. Applying `stripChannelPrefix` to every plugin message context is too broad; the safer fix is resolver-aware or fallback-only normalization that does not erase provider-owned conversation grammar.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve typed conversation ids in hook contexts:** `src/hooks/message-hook-mappers.ts:178`
  - body: The new normalizer strips `user:` and `channel:` from every `message_received` and `message_sent` hook context, but Discord uses those prefixes as canonical conversation identities and binding keys. After this change, hooks can see a bare id while inbound claims and session bindings still use `user:<id>` or `channel:<id>`, breaking correlation and DM-vs-channel semantics.
  - confidence: 0.93
