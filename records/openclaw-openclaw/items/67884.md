## Summary

Keep open: this PR is not merge-ready because it broadly strips provider-owned conversation ID prefixes that current Discord identity, inbound-claim, and binding paths preserve, and it still lacks real setup proof for the reported pending-lookup mismatch.

## What This Changes

This PR adds a shared hook conversation-id normalizer, applies it to `toPluginMessageContext`, and updates mapper tests to expect common Telegram, Slack, and Discord-style prefixes to be stripped.

## Best Possible Solution

Preserve provider-owned canonical IDs in plugin-facing hook contexts, and if pending lookup normalization is needed, apply it only in the resolver-aware lookup/fallback path with coverage for Discord bindings plus Telegram/Slack hook lookup behavior.

## Reproduction Assessment

Yes for the PR-introduced regression from source inspection: the branch strips `user:` and `channel:` in message hook contexts while current inbound claim and Discord binding paths preserve those IDs. The original Telegram/Slack pending-lookup bug is not reproduced with real setup evidence.

## Solution Assessment

No. Broadly stripping common prefixes in every plugin message context is not the narrowest maintainable fix; the safer direction is resolver-aware or fallback-only normalization that preserves provider-owned identity grammar.
