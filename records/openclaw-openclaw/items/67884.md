## Summary

Keep open: the PR targets a plausible hook-correlation bug, but the current diff still strips provider-owned typed IDs in message hook contexts while inbound claim and binding paths preserve them, and the contributor has not provided real after-fix behavior proof.

## What This Changes

The PR exports a shared hook conversation-id normalizer, applies it to plugin message contexts, and updates mapper tests to expect channel/common prefixes to be stripped.

## Best Possible Solution

Keep hook conversation IDs aligned through resolver-aware or fallback-only normalization that preserves provider-owned typed identities, with coverage for Discord binding behavior and the Telegram/Slack mismatch this PR targets.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows the diff strips `user:` and `channel:` from message hook contexts while current inbound claim and Discord binding paths preserve those IDs. The original pending-lookup bug still lacks real setup proof.

## Solution Assessment

No. Applying prefix stripping to every plugin message context is too broad; the safer solution should preserve resolver-owned/provider-owned conversation grammar and normalize only the fallback or wrapper-prefix cases that actually mismatch.
