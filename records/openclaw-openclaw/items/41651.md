## Summary

Keep open. Current main still performs the skill realpath containment check per configured root and emits the escaped-symlink warning before records are merged by skill name; related implementation PRs are open candidates, not shipped fixes.

## What This Changes

The issue requests suppressing noisy escaped-root skill warnings when symlinked personal or managed skill entries point at trusted skill copies that also load from bundled, managed, or other known skill roots.

## Best Possible Solution

Implement or land a focused skill-loader fix in src/agents/skills/workspace.ts that suppresses repeated trusted duplicate symlink warnings while still rejecting unknown root escapes and preserving bundled-source allowlist semantics; update focused regression tests and docs if the containment contract changes.

## Reproduction Assessment

Yes. The current code path is enough to reproduce with high confidence: a symlinked skill directory whose realpath is outside its configured root is discovered, warned, skipped, and only later can another source provide the same skill name.

## Solution Assessment

Unclear. The issue’s preferred duplicate-warning suppression is plausibly the narrowest direction, but the final implementation should be reviewed against the strict containment boundary, unknown external symlinks, and skills.allowBundled behavior before merge.
