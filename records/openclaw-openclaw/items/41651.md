- `impact:security`: The issue concerns filesystem containment and trust boundaries for skill loading through symlinks.

## Summary

Close as implemented: current main now accepts symlinked skill directories from managed and personal skill-manager roots, including the reported `~/.agents/skills` layout, while preserving fail-closed containment for `SKILL.md` and stricter roots.

## What This Changes

The issue asks OpenClaw to avoid repeated false-positive skill containment warnings when personal skill directories are symlinks to trusted bundled or manager-owned skill roots.

## Best Possible Solution

Keep the shipped source-scoped symlink policy: personal and managed skill-manager roots may follow skill-directory symlinks, while stricter roots and each `SKILL.md` remain contained.

## Reproduction Assessment

No on current main. The historical failure path is clear from the old source behavior, but current main routes personal `~/.agents/skills` skill-directory symlinks around configured-root containment while still checking each `SKILL.md`.

## Solution Assessment

Yes. The shipped source-scoped policy is more maintainable than suppressing all duplicate warnings because it fixes the trusted local-manager workflow without weakening workspace, project-agent, extra-dir, bundled, or per-file containment.
