## Summary

Close as implemented: current main now handles the reported managed/personal skill-directory symlink case without dropping the skill or emitting the escaped-root warning, while preserving stricter containment for workspace, extra, project-agent, bundled, and per-skill SKILL.md paths.

## What This Changes

The issue asks OpenClaw to stop noisy false-positive skill containment warnings when user skill entries are symlinks to already trusted bundled or configured skill roots.

## Best Possible Solution

Keep the landed source-scoped policy: managed and personal skill-manager roots may follow directory symlinks, workspace/project/extra roots use explicit allowSymlinkTargets, and bundled plus per-skill SKILL.md containment stay fail-closed.

## Reproduction Assessment

No, not as a current-main failure. The original 2026.3.x behavior was source-reproducible, but current main has source and regression-test coverage showing managed/personal directory symlinks now load while escaped SKILL.md paths still warn and fail closed.

## Solution Assessment

Yes. The landed source-scoped fix is narrower than globally weakening realpath checks because it only relaxes user-owned managed/personal skill-directory roots and keeps explicit allowlists or containment for the other roots.
