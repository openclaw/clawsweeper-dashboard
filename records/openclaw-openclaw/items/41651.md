## Summary

Close as implemented. Current main and shipped releases now allow managed and personal skill-directory symlinks without the repeated escaped-root warning, while preserving strict containment for workspace, project-agent, extra-dir, bundled, and per-skill SKILL.md paths.

## What This Changes

The issue reports noisy false-positive skill-loader containment warnings when symlinked personal skills resolve to bundled or other trusted skill roots.

## Best Possible Solution

Keep the shipped source-scoped policy: managed and personal skill-manager roots may follow skill-directory symlinks, while workspace/project/extra roots use explicit allowSymlinkTargets and bundled/SKILL.md containment stays fail-closed.

## Reproduction Assessment

No, not against current main: source inspection shows the reported managed/personal symlink warning path has been replaced by source-scoped handling, with regression coverage for the fixed behavior. The old failure path was high-confidence from the earlier code and related reports.

## Solution Assessment

Yes. The landed fix is the narrow maintainable solution because it quiets the trusted managed/personal skill-manager case without disabling containment for untrusted roots or bundled skills.
