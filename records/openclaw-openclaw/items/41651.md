## Summary

Current main now addresses the reported noisy symlink path: user-owned managed and personal skill roots can load directory symlinks without the escaped-root warning, while workspace/project/extra roots use the explicit trusted-target contract and strict file containment remains in place. The fix is current-main-only after v2026.5.12, so this can close as implemented on main rather than as shipped in the latest release.

## What This Changes

The issue requests reducing false-positive skill-loader warnings when symlinked skill folders resolve to trusted bundled or configured skill roots.

## Best Possible Solution

Keep the current split policy: personal and managed skill-manager roots accept directory symlinks, workspace/project/extra roots require explicit trusted targets, and bundled-root plus per-skill file containment stay strict.

## Reproduction Assessment

No. Source inspection and current regression coverage show the reported personal/managed symlink warning path no longer reproduces on current main; I did not run tests in this read-only review.

## Solution Assessment

Yes. The current-main implementation is narrower than globally downgrading or removing warnings because it accepts only user-managed/personal directory symlinks by default and uses explicit trusted targets for stricter roots.
