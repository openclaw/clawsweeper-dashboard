## Summary

Keep open. Current main still rejects and warns on symlinked skill paths whose realpath escapes the active scan root before same-named skills are merged from trusted sources, and the overlapping same-repo implementation PRs are still open rather than shipped.

## What This Changes

The issue asks for the skill loader to stop repeatedly warning when a personal or managed skill entry is a symlink to an already trusted bundled or configured skill root.

## Best Possible Solution

Keep this issue as the canonical tracking item until maintainers land or reject a focused skill-loader fix that suppresses trusted duplicate symlink warnings while preserving containment and bundled-skill allowlist semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a symlinked skill under a scanned root whose realpath is outside that root hits resolveContainedSkillPath, logs warnEscapedSkillPath, returns null, and only later can another source provide the same skill name.

## Solution Assessment

Unclear. Duplicate-warning suppression is plausibly the narrowest fix, but maintainers still need to choose it against the known-root symlink proposals because the containment boundary is security-sensitive.
