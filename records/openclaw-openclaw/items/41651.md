## Summary

Keep open. Current main still rejects and warns on symlinked skill paths whose realpath escapes the active scan root before same-named skills are merged from trusted sources, and the related implementation PRs are still open rather than shipped.

## What This Changes

The issue asks OpenClaw to stop producing repeated skill-loader containment warnings for symlinked skill entries that point to bundled or otherwise trusted skill roots.

## Best Possible Solution

Keep this issue as the canonical tracking item until a reviewed skill-loader fix lands that reduces duplicate trusted symlink warning noise while preserving unknown-root rejection and bundled allowlist semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a symlinked skill under a scanned root whose realpath is outside that root hits resolveContainedSkillPath, logs a warning, returns null, and only later can another source provide the same skill name.

## Solution Assessment

Unclear. The preferred duplicate-warning suppression is plausibly narrow, but the final fix should be selected against the strict containment boundary, known-root symlink proposals, and skills.allowBundled behavior.
