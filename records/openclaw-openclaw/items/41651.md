- `P2`: The reported warning flood affected a bounded skill-loader workflow with noisy startup/reload logs but not core runtime availability.
- `impact:security`: The item concerns symlink handling around the skill-loader containment boundary.

## Summary

Current main and the shipped v2026.5.20 release now cover the reported false-positive warning path: managed and personal skill roots accept skill-directory symlinks without warning while workspace, extra, bundled, and per-skill SKILL.md containment still fail closed.

## What This Changes

The issue asks OpenClaw to stop repeatedly warning when intentional symlinked skills resolve outside their configured scan root but are already trusted or duplicated elsewhere.

## Best Possible Solution

Keep the shipped source-scoped symlink policy: managed and personal skill-manager roots may follow skill-directory symlinks, while untrusted roots and SKILL.md file containment remain fail-closed.

## Reproduction Assessment

No, not on current main: the reported personal/managed skill-directory symlink path no longer reaches the escaped-root warning branch. The old behavior is source-proven by the related history, and the fixed behavior is covered by current source, tests, docs, and release evidence.

## Solution Assessment

Yes. The shipped fix is narrower than downgrading all warnings because it quiets user-owned managed/personal skill-manager symlinks while preserving explicit allowlists and fail-closed containment for untrusted roots.
