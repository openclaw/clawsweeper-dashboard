## Summary

Keep open: current main still supports and recommends `[[reply_to_current]]`, and the existing visible-reply suppression is global/group/harness-scoped rather than the requested per-agent directive guard plus per-agent automatic-send policy.

## What This Changes

Request per-agent runtime config to disable `[[reply_to_current]]` processing and optionally suppress automatic assistant-text replies so only explicit message-tool sends are visible.

## Best Possible Solution

Define one agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and explicit message-tool sends, and layer predictably over existing visible-reply modes; update config types, strict schema/help/generated metadata, prompt guidance, reply parsing/plumbing, docs, changelog, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` and resolved to the current message id, while automatic direct-chat delivery remains enabled unless global/harness visible-reply policy suppresses it.

## Solution Assessment

Unclear. The requested runtime guard is plausible, but the exact public API needs maintainer design because it overlaps global/group visible-reply controls, native command behavior, message-tool fallback, and warning feedback to the agent.
