## Summary

Keep open. Current main still implements the behaviors this report wants guarded: `[[reply_to_current]]` is parsed, prompted, and honored, and automatic visible delivery is controlled only by existing global/group `messages.visibleReplies` policy rather than a per-agent outbound policy. The related #73272 tracks only the narrower directive guard, while this issue also requests per-agent automatic-send suppression, so it is not fully duplicate or superseded.

## What This Changes

The issue requests per-agent config to disable `[[reply_to_current]]` processing and optionally disable automatic assistant text sends so only explicit message-tool calls produce visible channel output.

## Best Possible Solution

Define one coherent agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and message-tool sends, and control automatic visible final replies with clear precedence over `messages.visibleReplies`, `messages.groupChat.visibleReplies`, native-command bypasses, and message-tool unavailable fallback. The implementation should update config types/schema/help/docs, prompt guidance, parser/reply plumbing, and focused regression tests together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` and resolved to the current message id, while direct chats still resolve to automatic visible delivery unless the global `messages.visibleReplies: "message_tool"` mode is set.

## Solution Assessment

Unclear. A runtime guard is plausible, but the exact public API should be designed by maintainers because it overlaps existing visible-reply policy, native command exceptions, message-tool availability fallback, and next-turn warning behavior.
