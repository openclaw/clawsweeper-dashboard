## Summary

Keep open: current main still processes and recommends `[[reply_to_current]]`, and visible-reply suppression is global/group/harness scoped rather than the requested per-agent directive guard plus per-agent auto-send policy. The same-author #73272 remains open as a narrower directive-only repair target, but it does not supersede this broader issue.

## What This Changes

Requested per-agent runtime controls to disable `[[reply_to_current]]` handling and suppress automatic assistant text delivery so agents must use explicit message-tool sends.

## Best Possible Solution

Define one agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and message-tool sends, and layer predictably over existing visible-reply modes with schema, docs, prompt, and regression tests updated together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` and resolved to the current message id, while direct-chat automatic delivery remains available unless global or harness visible-reply policy suppresses it.

## Solution Assessment

Unclear. A per-agent directive guard is plausible and #73272 is narrow enough to repair, but the broader auto-send policy needs maintainer design because it overlaps `messages.visibleReplies`, native command behavior, and message-tool availability fallback.
