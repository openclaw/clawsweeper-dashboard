## Summary

Keep open: current main still accepts and resolves `[[reply_to_current]]`, still teaches agents to prefer it, and has only global/group visible-reply suppression rather than the requested per-agent directive guard plus per-agent automatic-send suppression. The related open #73272 overlaps the directive half only, so it is not a complete superseding tracker for this broader request.

## What This Changes

Request a per-agent outbound policy that can disable the `[[reply_to_current]]` directive and optionally suppress automatic visible replies so only explicit `message` tool sends reach the channel.

## Best Possible Solution

Define one coherent agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and message-tool sends, layer predictably over existing visible-reply modes, and update config types, strict schema/help/docs, prompt guidance, parser/reply plumbing, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` and resolved to the current message id, while direct-chat automatic delivery remains enabled unless the global `messages.visibleReplies: "message_tool"` mode applies.

## Solution Assessment

Unclear. The requested runtime guard is plausible, but the exact public API needs maintainer design because it overlaps global/group visible-reply controls, native-command exceptions, message-tool availability fallback, and warning feedback to the agent.
