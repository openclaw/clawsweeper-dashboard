## Summary

Keep open: current `main` still accepts and recommends `[[reply_to_current]]`, and the existing visible-reply suppression is global/group/harness scoped rather than the requested per-agent directive guard plus per-agent automatic-send policy. The related #73272 is open but narrower, so it does not supersede this broader report.

## What This Changes

Add agent-scoped runtime controls to disable `[[reply_to_current]]` handling and optionally suppress automatic assistant-text delivery so only explicit message-tool sends are visible.

## Best Possible Solution

Define one agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and explicit message-tool sends, and layer predictably over existing visible-reply modes, with config types/schema/help/docs/changelog and regression tests updated together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` and resolved to the current message id, while direct-chat automatic delivery remains the default unless a global or harness visible-reply policy suppresses it.

## Solution Assessment

Unclear. The requested runtime guard is directionally plausible, but the final API should be maintainer-designed because it overlaps `messages.visibleReplies`, group defaults, native command behavior, message-tool fallback, and warning feedback to the agent.
