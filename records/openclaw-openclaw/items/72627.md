## Summary

Keep open: current `main` still parses and recommends `[[reply_to_current]]`, and the available visible-reply suppression is global/group/harness scoped rather than the requested per-agent directive guard plus per-agent auto-send policy. The related #73272 is open but narrower, so it does not supersede this broader report.

## What This Changes

The issue requests per-agent runtime controls to disable `[[reply_to_current]]` and suppress automatic assistant text sends.

## Best Possible Solution

Define one agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and message-tool sends, and layer predictably over existing visible-reply modes with schema, docs, prompt, and regression tests updated together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` and resolved to the current message id, while direct-chat automatic delivery remains available unless global or harness visible-reply policy suppresses it.

## Solution Assessment

Unclear. The requested runtime guard is plausible, but the best fix needs maintainer design because it overlaps global visible-reply policy, group overrides, harness defaults, native command behavior, message-tool availability fallback, and agent feedback semantics.
