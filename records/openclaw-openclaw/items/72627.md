## Summary

Keep open. Current main still parses and honors `[[reply_to_current]]`, still prompts agents to prefer it, and only offers global/group visible-reply suppression rather than the requested per-agent directive guard plus per-agent automatic-send suppression. The related open #73272 overlaps the directive half but does not cover the broader auto-send policy requested here.

## What This Changes

The issue requests per-agent runtime config to disable `[[reply_to_current]]` processing and to suppress automatic source-channel replies so visible output must use explicit `message` tool sends.

## Best Possible Solution

Define one coherent agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and message-tool sends, and layer clearly over existing visible-reply modes; then update config types, strict schema/help/docs, prompt guidance, parser/reply plumbing, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` and resolved to the current message id, while automatic direct-chat delivery remains enabled unless the global `messages.visibleReplies: "message_tool"` mode applies.

## Solution Assessment

Unclear. The requested runtime guard is plausible, but the exact public API needs maintainer design because it overlaps global/group visible-reply controls, native-command exceptions, message-tool availability fallback, and warning feedback to the agent.
