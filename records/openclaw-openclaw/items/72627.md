## Summary

Keep open: current main still has no per-agent `directives.replyToCurrent` or `autoReply` policy, and the shipped visible-reply controls only provide global/group/harness suppression with native-command and message-tool-unavailable fallbacks. The narrower directive-only work is already represented by #73272, but this issue's broader per-agent automatic-send policy still needs maintainer design.

## What This Changes

The issue requests per-agent runtime policy to disable `[[reply_to_current]]` processing and suppress automatic assistant text sends so only explicit message-tool sends are visible.

## Best Possible Solution

Define one agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, suppress automatic final delivery when configured, preserve explicit `reply_to:<id>` and message-tool sends, and layer predictably over existing visible-reply modes with schema, docs, prompt, changelog, and regression tests updated together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent`, while automatic final delivery is controlled by global/group/harness visible-reply policy rather than per-agent config.

## Solution Assessment

Unclear. The directive-only guard is plausible and #73272 is the narrower repair path, but the broader per-agent automatic-send policy should be designed against existing visible-reply and fallback behavior before implementation.
