## Summary

Keep open: current main still has no per-agent `directives.replyToCurrent` or `autoReply` config surface. `[[reply_to_current]]` remains parsed and recommended, while existing automatic-delivery controls are global, group, session, or harness scoped rather than the requested per-agent outbound policy.

## What This Changes

Request adds per-agent config to disable `[[reply_to_current]]` handling and suppress automatic assistant-text channel sends so only explicit `message` tool sends are visible.

## Best Possible Solution

Define one coherent agent-scoped outbound policy that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and message-tool sends, and document how it composes with existing visible-reply and send-policy controls.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: assistant text containing `[[reply_to_current]]` is parsed into reply metadata, while automatic source delivery is controlled only by existing global/group/session/harness policy and not by a per-agent `autoReply` key.

## Solution Assessment

Unclear. The directive-only guard is plausible and already has the narrower open #73272, but this issue’s per-agent automatic-send policy needs maintainer design because it overlaps existing visible-reply and send-policy behavior.
