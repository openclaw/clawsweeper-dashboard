## Summary

Keep open: current `main` still parses, resolves, tests, and prompts for `[[reply_to_current]]`, while automatic visible-reply suppression is only global/group/harness scoped rather than the requested per-agent directive guard and send policy. The related #73272 remains open but is narrower, so it does not supersede this broader report.

## What This Changes

Requests agent-scoped outbound controls that can disable `[[reply_to_current]]` parsing and automatic assistant-text channel sends while leaving explicit message-tool sends available.

## Best Possible Solution

Define one agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, suppress automatic visible replies when requested, preserve explicit `reply_to:<id>` and message-tool sends, and update schema, docs, prompts, doctor warnings, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` and resolved to the current message id, and direct-chat automatic delivery remains enabled unless global/harness visible-reply policy suppresses it.

## Solution Assessment

Unclear. The requested runtime guard is plausible, but the best fix should first settle the agent-scoped policy shape because it overlaps existing visible-reply delivery policy, native command behavior, message-tool fallback, prompt guidance, and next-turn warning semantics.
