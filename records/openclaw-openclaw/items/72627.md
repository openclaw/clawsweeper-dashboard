## Summary

Keep open: current main still parses and recommends `[[reply_to_current]]`, the public per-agent config contract has no `directives.replyToCurrent` or `autoReply` setting, and the existing visible-reply suppression is global/group/harness scoped rather than the requested per-agent outbound policy. The related #73272 is open but narrower, so it does not supersede the broader auto-send part of this issue.

## What This Changes

Request per-agent runtime config to disable `[[reply_to_current]]` handling and suppress automatic assistant-text delivery so visible output comes only from explicit message-tool sends.

## Best Possible Solution

Define one agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and message-tool sends, and layer predictably over existing visible-reply modes with config, schema/help metadata, prompt guidance, docs, changelog, and regression tests kept aligned.

## Reproduction Assessment

Yes. A code-level reproduction is clear: assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` and resolved to the current message id, while automatic source delivery remains enabled for direct chats unless a global/harness visible-reply policy suppresses it.

## Solution Assessment

Unclear. A runtime guard is plausible, but the best API needs maintainer design because this spans per-agent config, reply parsing, next-turn warning feedback, global visible-reply controls, and message-tool fallback semantics.
