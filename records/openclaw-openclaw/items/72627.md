## Summary

Keep open. Current main still parses and honors `[[reply_to_current]]`, still prompts agents to prefer it, and does not expose the requested per-agent directive guard or per-agent automatic-send suppression; the related open #73272 only covers the narrower directive guard, not the broader auto-send request.

## What This Changes

The issue requests per-agent runtime config to disable `[[reply_to_current]]` and optionally suppress automatic visible assistant-text sends so agents must use explicit message-tool delivery.

## Best Possible Solution

Define one coherent agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and message-tool sends, and layer cleanly over existing visible-reply controls; then update config types, strict schema/help/docs, prompt guidance, parser/reply plumbing, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed as `replyToCurrent` and mapped to the current message id, while direct chats still deliver automatic final text unless the global `messages.visibleReplies: "message_tool"` mode is configured.

## Solution Assessment

Unclear. The runtime guard is plausible, but the exact public API needs maintainer design because per-agent directive policy and per-agent auto-send suppression overlap global/group visible-reply controls, native-command exceptions, prompt guidance, and warning feedback.
