## Summary

Keep open: current main still parses and recommends `[[reply_to_current]]`, and there is no per-agent `directives.replyToCurrent` or `autoReply` config surface. The related #73272 covers only the narrower directive opt-out, while this issue also asks for broader per-agent automatic-send policy.

## What This Changes

Request adds per-agent runtime controls to disable `[[reply_to_current]]` handling and suppress automatic assistant text delivery so selected agents must use explicit message-tool sends.

## Best Possible Solution

Define one agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and message-tool sends, and layer predictably over existing visible-reply modes with schema, docs, prompt, and regression tests updated together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent`, while automatic final delivery is controlled only by global/group/harness visible-reply policy.

## Solution Assessment

Unclear. The directive-only guard is plausible and #73272 is the narrower repair path, but this issue's per-agent auto-send policy needs maintainer design because it overlaps existing visible-reply modes and fallback behavior.
