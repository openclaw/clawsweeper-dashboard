## Summary

Keep open: current main still accepts and recommends `[[reply_to_current]]`, and the existing visible-reply suppression is `messages`/harness scoped rather than the requested per-agent directive guard plus per-agent automatic-send policy. The related #73272 is open but narrower, so it does not supersede this broader issue.

## What This Changes

Request per-agent config to disable `[[reply_to_current]]` and opt an agent out of automatic assistant-text delivery so only explicit `message` tool sends reach the channel.

## Best Possible Solution

Define one agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and explicit message-tool sends, and layer predictably over existing visible-reply modes. Update config types, strict schema, generated config metadata/help, prompt guidance, reply parsing/plumbing, docs, changelog, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` and resolved to the current message id, while direct-chat automatic delivery remains the default unless a global/harness visible-reply policy suppresses it.

## Solution Assessment

Unclear. The requested runtime guard is plausible, but the exact API should be maintainer-designed because it overlaps global/group visible-reply controls, native command behavior, message-tool fallback, and warning feedback to the agent.
