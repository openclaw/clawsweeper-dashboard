## Summary

Keep open: current `main` still recognizes and recommends `[[reply_to_current]]`, and the visible-reply controls are global/group/harness-scoped rather than the requested per-agent directive guard plus per-agent automatic-send policy. The reporter's Feishu DM follow-up gives concrete real-world context, and related #73272 is open but narrower, so it does not supersede this broader issue. https://github.com/openclaw/openclaw/issues/72627

## What This Changes

Add per-agent controls to disable `[[reply_to_current]]` processing and suppress automatic visible replies so agents must use explicit message-tool sends.

## Best Possible Solution

Define one agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, keep explicit `reply_to:<id>` and message-tool sends working, and layer predictably over existing visible-reply modes with schema, docs, prompt, changelog, and regression tests updated together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` and resolved to the current message id, while automatic direct/source delivery remains controlled only by existing global/group/harness policy.

## Solution Assessment

Unclear. The requested runtime guard is directionally plausible, but the best fix needs maintainer design because it overlaps `messages.visibleReplies`, group overrides, native-command bypass, message-tool-unavailable fallback, and model warning semantics.
