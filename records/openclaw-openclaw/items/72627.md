## Summary

Keep open: current main still treats `[[reply_to_current]]` as a supported outbound reply directive, still recommends it in the agent prompt, and has only global/group visible-reply controls rather than the requested per-agent directive and auto-send policy.

## What This Changes

Add optional per-agent runtime config to disable `[[reply_to_current]]` and suppress automatic assistant-text delivery so explicit message-tool sends are required.

## Best Possible Solution

Define one agent-scoped outbound policy contract that can block `reply_to_current` and suppress automatic source delivery while preserving explicit `message` tool sends, with schema, docs, prompt text, and regression tests updated together.

## Reproduction Assessment

Yes. Source inspection on current main shows assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent`, while automatic final delivery is controlled only by requested/global/group/harness visible-reply policy rather than per-agent config.

## Solution Assessment

Unclear. A directive-only guard is plausible and #73272 is the narrower path for that slice, but this issue's broader per-agent auto-send suppression should be designed as a single outbound policy rather than patched separately into directive parsing.
