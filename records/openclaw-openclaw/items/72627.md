## Summary

Keep open: current main still parses and prompts `[[reply_to_current]]`, the strict per-agent config does not expose the requested directive or automatic-send controls, and existing visible-reply suppression is global/group/harness-scoped rather than per-agent. The related #73272 is open but narrower, so it does not supersede this broader issue.

## What This Changes

The issue requests per-agent runtime config to disable `[[reply_to_current]]` and suppress automatic assistant text delivery so only explicit message-tool sends are visible.

## Best Possible Solution

Define a documented agent-scoped outbound policy that can reject or rewrite `reply_to_current`, preserve explicit reply ids and message-tool sends, and layer predictably over existing visible-reply modes with config schema, prompt, docs, changelog, and regression tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` and resolved to the current message id, while direct automatic delivery remains the default unless a global or harness visible-reply policy suppresses it.

## Solution Assessment

Unclear. A runtime guard is plausible, but the maintainable fix should be designed as one outbound policy rather than adding isolated flags that conflict with current visible-reply and message-tool fallback behavior.
