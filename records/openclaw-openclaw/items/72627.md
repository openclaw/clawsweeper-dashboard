## Summary

Keep open. Current main still accepts and honors `[[reply_to_current]]`, still prompts agents to prefer it, and only has global/group visible-reply suppression rather than the requested per-agent directive guard plus per-agent automatic-send suppression.

## What This Changes

The issue requests agent-scoped runtime controls to disable `[[reply_to_current]]` processing and suppress automatic assistant text delivery so visible output must use explicit message-tool sends.

## Best Possible Solution

Define one coherent agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and message-tool sends, and layer clearly over existing `messages.visibleReplies`; then update config types, strict schema/help/docs, prompt guidance, parser/reply plumbing, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed as `replyToCurrent` and mapped to the current message id, while direct chats still use automatic final delivery unless the global `messages.visibleReplies: "message_tool"` mode is set.

## Solution Assessment

Unclear. The requested runtime guard is plausible, but the exact public API needs maintainer design because it overlaps existing global/group visible-reply controls, native-command exceptions, message-tool availability fallback, and next-turn warning semantics.
