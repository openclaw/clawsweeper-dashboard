## What This Changes

The issue requests per-agent config to disable `[[reply_to_current]]` processing and automatic channel sends.

## Best Possible Solution

Define one documented agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and message-tool sends, and compose predictably with existing visible-reply modes.

## Reproduction Assessment

Yes. Source inspection shows assistant text containing `[[reply_to_current]]` is still parsed into reply metadata, and automatic final delivery is controlled by global/group/requested/harness visible-reply policy rather than per-agent config.

## Solution Assessment

Unclear. A per-agent directive guard is plausible and bounded, but combining it with per-agent auto-send suppression requires maintainer design because it overlaps existing visible-reply modes, native-command exceptions, and fallback behavior.
