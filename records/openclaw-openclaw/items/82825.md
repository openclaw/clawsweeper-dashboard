## What This Changes

The PR adds realpath-based exec approval trust helpers and applies them to allowlist matching, safe-bin trust, skill auto-allow, Allow Always persistence, approval audit metadata, tests, and changelog.

## Best Possible Solution

Land a corrected version that uses one trust-path contract across runtime checks, persistence/audit metadata, doctor diagnostics, and focused regression coverage, with explicit maintainer/security signoff.

## Reproduction Assessment

Yes. Current main source shows path-shaped allowlist checks use resolvedPath while approved execution pins resolvedRealPath, so a symlinked executable with a distinct realpath reproduces the mismatch without live credentials.

## Solution Assessment

No, not as-is. Realpath canonicalization is the right direction, but this branch should fix the portable test failure, align doctor diagnostics with the new runtime trust path, and provide real behavior proof before merge.

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Use an exact trusted safe-bin realpath in the test:** `src/infra/exec-approvals-safe-bins.test.ts:358`
  - body: This injected trustedDirs set is not normalized. On a case-sensitive platform, the resolved directory `/opt/homebrew/Cellar/...` will not equal the lowercase `/opt/homebrew/cellar/...`, so the new expectation fails; use the exact-cased realpath dir or construct the set through the normal trusted-dir normalization path.
  - confidence: 0.9

  - body: The new runtime safe-bin check now evaluates the executable trust path and a candidate fallback, but `scanExecSafeBinTrustedDirHints()` still checks only the PATH-resolved candidate path. A config that trusts only the canonical target directory can be accepted at runtime while doctor warns incorrectly, so the diagnostic should use the same trust-path logic.
  - confidence: 0.84
