## Summary

Keep open: this is a plausible broader fix for the real exec approval identity bug, but the PR has a protected maintainer label, changes security-sensitive trust semantics, lacks real behavior proof beyond tests, and has source-visible issues to address before merge.

## What This Changes

The PR adds realpath-based exec approval trust helpers and applies them to path-shaped allowlists, safe-bin trust, skill auto-allow, Allow Always persistence, approval audit metadata, tests, and changelog.

## Best Possible Solution

Land a corrected version that uses one realpath trust contract across runtime checks, persistence/audit, skill trust, safe-bin diagnostics, and tests, with explicit maintainer signoff on the symlink-approval policy.

## Reproduction Assessment

Yes. Source inspection shows current main matches trust decisions against `resolvedPath` while approved execution pins `resolvedRealPath`, so a symlinked executable can reproduce the reported miss/drift without needing live credentials.

## Solution Assessment

No, not as-is. Realpath canonicalization is the right direction, but this branch should fix the portable safe-bin test failure and align doctor diagnostics with the new runtime trust path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Use the exact normalized safe-bin trust dir in the test:** `src/infra/exec-approvals-safe-bins.test.ts:358`
  - body: This expectation is not portable: `isTrustedSafeBinPath()` normalizes the resolved executable directory but does not normalize an injected `trustedSafeBinDirs` set. On case-sensitive Linux, `/opt/homebrew/Cellar/...` will not equal the lowercase `/opt/homebrew/cellar/...`, so the new test will fail; use the exact-cased realpath dir or build the set through the normal trusted-dir normalization helper.
  - confidence: 0.9
- **[P2] Align doctor safe-bin hints with the new trust path:** `src/infra/exec-approvals-allowlist.ts:102-110`
  - body: After this runtime check moves safe-bin trust to the executable realpath, `scanExecSafeBinTrustedDirHints()` still validates `resolution.execution.resolvedPath`. A Homebrew config that trusts `/opt/homebrew/bin` can pass doctor while runtime denies the safe-bin, and a config that trusts the Cellar realpath can be warned incorrectly; update the doctor scan to use the same trust-path helper before claiming the linked issue is fixed.
  - confidence: 0.86
