## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Send supported session verbose levels:** `apps/linux/src/section_sessions.c:176-178`
  - body: The dropdown sends `none`, `low`, and `high` as `verboseLevel`, but the current `sessions.patch` contract accepts `off`, `on`, and `full`. Users changing verbose behavior from the Linux app will hit schema validation instead of updating the session.
  - confidence: 0.93
- **[P1] Preserve cron schedule and session targets:** `apps/linux/src/section_cron.c:274-276`
  - body: Saving an edited cron job always emits a `cron` schedule and a collapsed target value, but current jobs may use `at`, `every`, or `session:*`. A no-op edit from this UI can fail validation or silently retarget/reschedule a job created elsewhere.
  - confidence: 0.91
- **[P2] Allow empty skill values to clear stored settings:** `apps/linux/src/section_skills.c:133-134`
  - body: The dialog returns before sending `skills.update` when the value is empty, but the gateway uses empty API key/env values as delete operations. This makes stale skill secrets or env overrides impossible to remove from the Linux UI.
  - confidence: 0.88
- **[P2] Resolve local session logs from session ids:** `apps/linux/src/section_sessions.c:264-266`
  - body: The local log opener builds `<store_dir>/<sessionKey>/transcript.jsonl`, while current session transcripts are resolved as session-id JSONL files. The Log action will often miss existing local transcripts and fall back to the dashboard unnecessarily.
  - confidence: 0.86

## Security Review

Summary: The PR adds new credential, device identity, SSH tunnel, and exec-approval surfaces that need explicit maintainer security and storage review.

  - body: `device_identity.c` writes the device private seed into `identity/device.json`; even with file permissions, this is a new persistent secret store outside the canonical SQLite/runtime-state guidance and needs an approved ownership and migration model.
