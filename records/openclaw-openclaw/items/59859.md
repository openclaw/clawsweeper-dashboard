## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Send supported session verbose values:** `apps/linux/src/section_sessions.c:176-184`
  - body: The Linux Sessions UI sends `none`, `low`, or `high` as `verboseLevel`, but current `sessions.patch` validation only accepts `off`, `on`, or `full` (or null/undefined). This makes the visible verbose dropdown fail at runtime instead of updating the session.
  - confidence: 0.95
- **[P1] Preserve cron schedule variants on edit:** `apps/linux/src/section_cron.c:270-279`
  - body: The edit-save path always serializes the patch as `schedule.kind = "cron"` with an `expr`, so opening and saving an existing `at` or `every` job rewrites or invalidates its schedule even when the user only changes metadata. The patch needs to carry the original schedule variant, or expose an explicit schedule-kind editor.
  - confidence: 0.92
- **[P2] Resolve local logs from session transcript metadata:** `apps/linux/src/section_sessions.c:264-267`
  - body: This path probes `<dirname(storePath)>/<sessionKey>/transcript.jsonl`, but current gateway transcript resolution is based on `sessionId`, optional `sessionFile`, and the sessions directory. Normal local transcripts will be missed and the app will fall back to the dashboard even when the file is present.
  - confidence: 0.88
- **[P1] Move the Linux client ID to the current protocol package:** `src/gateway/protocol/client-info.ts:13`
  - body: The branch adds the Linux client ID in the old `src/gateway/protocol` location, while current main owns the handshake client-id registry in `packages/gateway-protocol/src/client-info.ts`. Rebase this change into the shared package or the new client will still fail current gateway validation after conflict resolution.
  - confidence: 0.9
