- **[P1] Preserve existing cron schedule variants on edit:** `apps/linux/src/section_cron.c:274-275`
  - body: Saving any edited cron job always sends `schedule: { kind: "cron", expr: ... }`. Current gateway jobs may be `at` or `every`, so opening one of those jobs and saving an unrelated field can fail validation or rewrite the schedule semantics.
  - confidence: 0.91
- **[P1] Preserve custom cron session targets on edit:** `apps/linux/src/test_seams.c:47`
  - body: `session:<id>` is a valid cron target, but this maps every unknown value to the New Session row and the save path emits `isolated`. Editing an existing session-scoped job can silently retarget it even when the user changes another field.
  - confidence: 0.89
- **[P2] Resolve local transcripts with the sessionId layout:** `apps/linux/src/section_sessions.c:266`
  - body: This probes `<store_dir>/<session-key>/transcript.jsonl`, but current transcript resolution uses the sessionId/sessionFile candidates such as `<sessionsDir>/<sessionId>.jsonl`. The Log action will miss local transcripts and fall back to dashboard even when the file exists.
  - confidence: 0.87
