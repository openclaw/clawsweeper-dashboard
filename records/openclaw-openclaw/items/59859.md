## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve cron schedule variants when editing jobs:** `apps/linux/src/section_cron.c:274-275`
  - body: The edit save path always sends `patch.schedule` as `{ kind: "cron", expr: ... }`. Existing jobs can use `kind: "every"` or `kind: "at"`, so saving an unrelated edit rewrites the schedule shape and can fail validation or change when the job runs. Carry the original schedule variant through the dialog or block editing unsupported variants.
  - confidence: 0.9
- **[P2] Resolve local session transcripts with session IDs:** `apps/linux/src/section_sessions.c:266`
  - body: This builds the local transcript path as `<storeDir>/<session-key>/transcript.jsonl`, but the gateway resolves transcripts as direct `sessionId` or `sessionFile` paths under the sessions directory. The Log action will usually miss existing local transcripts and fall back to the dashboard, which breaks offline or no-dashboard access.
  - confidence: 0.87
- **[P2] Bracket IPv6 literals before formatting gateway URLs:** `apps/linux/src/gateway_config.c:1034-1040`
  - body: The HTTP and WebSocket URL builders interpolate `config->host` directly into `scheme://host:port`. For an IPv6 literal such as `::1`, this produces malformed URLs like `http://::1:18789`, breaking health checks, WebSocket connection, and dashboard handoff on valid IPv6/custom-bind setups.
  - confidence: 0.84
- **[P2] Keep session-scoped cron targets when editing jobs:** `apps/linux/src/test_seams.c:46-47`
  - body: `session_target_to_index()` maps any unrecognized target to the “New Session” row, and saving that row emits `isolated`. Valid `session:<id>` cron targets therefore get silently retargeted when a user edits a job, even if they only changed the name or prompt.
  - confidence: 0.82
