## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve non-cron schedules when editing jobs:** `apps/linux/src/section_cron.c:274-275`
  - body: The edit save path always writes `patch.schedule` as `{ kind: "cron", expr: ... }`. Current cron jobs may have `kind: "every"` or `kind: "at"`, so saving an existing non-cron job after changing unrelated fields rewrites its schedule shape and can fail validation or change when the job runs. Carry the original schedule kind/fields through the dialog or prevent editing unsupported variants.
  - confidence: 0.91
- **[P2] Keep session-scoped cron targets on edit:** `apps/linux/src/test_seams.c:42-47`
  - body: Valid cron targets include `session:<id>`, but this mapper treats anything other than the three built-in values as index 0, which the edit flow saves back as `isolated`. Editing a session-scoped job can silently retarget it away from the intended session even when the user changes an unrelated field.
  - confidence: 0.9
- **[P2] Send supported verbose levels:** `apps/linux/src/section_sessions.c:176-184`
  - body: The sessions UI sends `none`, `low`, and `high`, but `sessions.patch` normalizes verbose overrides to `off`, `on`, and `full`; verbose changes from this app will be rejected. Map the combo values to the server contract before dispatching `sessions.patch`.
  - confidence: 0.92
- **[P2] Resolve local session logs from session IDs:** `apps/linux/src/section_sessions.c:264-266`
  - body: This builds the local transcript path as `<storeDir>/<session-key>/transcript.jsonl`, but the gateway stores and resolves transcripts by `sessionId` or `sessionFile`. The Log action will usually miss existing local transcripts and fall back to the dashboard, which breaks offline or no-dashboard access.
  - confidence: 0.87
- **[P2] Bracket IPv6 hosts before composing URLs:** `apps/linux/src/gateway_config.c:1153-1155`
  - body: The dashboard URL builder interpolates `config->host` directly into `scheme://host:port`. For an IPv6 literal such as `::1`, that produces malformed URLs like `http://::1:18789/`, breaking dashboard and log handoff on valid IPv6/custom-bind setups. Normalize IPv6 literals to `[host]` before formatting these URLs.
  - confidence: 0.84
- **[P2] Read channel account labels from name:** `apps/linux/src/gateway_data.c:262`
  - body: `channels.status` account snapshots expose the human-readable account label as `name`, but this parser reads `displayName`. Normal connected account rows therefore fall back to account IDs instead of showing configured friendly labels.
  - confidence: 0.9
