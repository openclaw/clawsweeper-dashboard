## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve non-cron schedules when editing jobs:** `apps/linux/src/section_cron.c:274-275`
  - body: The edit save path always sends `patch.schedule` as `{ kind: "cron", expr: ... }`. Current cron jobs may use `kind: "every"` or `kind: "at"`, so saving an existing non-cron job after changing unrelated fields rewrites its schedule shape and can fail validation or change when the job runs.
  - confidence: 0.91
- **[P2] Preserve session-scoped cron targets:** `apps/linux/src/test_seams.c:42-47`
  - body: Valid cron targets include `session:<id>`, but the edit UI maps every unknown target to the "New Session" index and then saves it back as `isolated`. Editing a session-scoped job can silently retarget it away from the intended session even when the user changes only metadata.
  - confidence: 0.88
- **[P2] Resolve session logs from session IDs:** `apps/linux/src/section_sessions.c:265-266`
  - body: This builds the local transcript path as `<storeDir>/<session-key>/transcript.jsonl`, but the gateway resolves transcripts by `sessionId`/`sessionFile` under the sessions directory, and the PR parser already exposes `session_id`. As written, the Log action usually misses existing local transcripts and falls back to the dashboard.
  - confidence: 0.87
- **[P2] Bracket IPv6 hosts before composing gateway URLs:** `apps/linux/src/gateway_config.c:1034-1040`
  - body: The URL builders interpolate `config->host` directly into `scheme://host:port`. For an IPv6 literal such as `::1`, that produces malformed URLs like `http://::1:18789`, breaking health checks, WebSocket connection, and dashboard handoff on valid IPv6/custom-bind setups.
  - confidence: 0.84
