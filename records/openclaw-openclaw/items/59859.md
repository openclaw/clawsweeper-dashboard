## Review Findings

Full review comments:

- **[P1] Preserve existing cron schedule variants:** `apps/linux/src/section_cron.c:274-275`
  - body: The edit handler always sends `schedule.kind = "cron"` with an `expr` value. Current `CronScheduleSchema` also supports `at` and `every`, so saving an existing one-time or interval job from the Linux app can fail validation or rewrite its schedule even when the user only edits metadata.
  - confidence: 0.91
- **[P1] Keep session-scoped cron targets when editing:** `apps/linux/src/section_cron.c:351-352`
  - body: The edit dialog converts any unrecognized persisted target, including valid `session:<id>` targets, to combo index 0; saving then serializes that index as `isolated`. That silently retargets jobs that were explicitly bound to a session, which the gateway protocol accepts and current cron tests cover.
  - confidence: 0.88
- **[P2] Resolve Linux product-state storage ownership:** `apps/linux/src/product_state.c:33-35`
  - body: This adds a persistent `product-state.ini` sidecar for app preferences under the user state dir. OpenClaw policy says app-owned runtime state belongs in an approved SQLite/state owner unless it is a named product artifact, so this needs to move to an approved store or get explicit maintainer ownership before merge.
  - confidence: 0.8
