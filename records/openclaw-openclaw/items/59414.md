## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Reuse shared home-path redaction:** `src/commands/doctor-node-runtime.ts:242-243`
  - body: `buildNodeRuntimeSummary` only treats `/` as the home boundary, so on Windows `USERPROFILE=C:\Users\Alice` and `execPath=C:\Users\Alice\.volta\bin\node.exe` falls through and prints the full username-bearing path in support-oriented Doctor output. Use `shortenHomePath` or equivalent slash/backslash handling.
  - confidence: 0.93
- **[P2] Cover all accepted Node release lines:** `src/commands/doctor-node-runtime.ts:121-136`
  - body: `NODE_RELEASE_SCHEDULE` only contains Node 22, but OpenClaw's runtime floor accepts newer lines and upstream currently lists Node 24 LTS, Node 25 maintenance ending 2026-06-01, and Node 26 current. Doctor would skip lifecycle warnings for those runnable versions, so cover the accepted release lines or derive the table from an authoritative source.
  - confidence: 0.9
- **[P3] Keep the minimum Node warning in sync:** `src/commands/doctor-node-runtime.ts:167`
  - body: `runtimeSatisfies` is driven by current main's Node 22.19.0 floor, but the new warning still says `>=22.14.0`. If Doctor reaches this branch for Node 22.18.x, it gives users the wrong minimum; derive the label from the shared runtime contract or update it.
  - confidence: 0.91
- **[P3] Align manager names with runtime markers:** `src/commands/doctor-node-runtime.ts:59-68`
  - body: The new display map duplicates `runtime-paths.ts` and already misses markers that `isVersionManagedNodePath` recognizes, including `.local/share/fnm`, macOS fnm Application Support, and mise. Those installs will be detected as version-managed but lose the requested manager name.
  - confidence: 0.84
