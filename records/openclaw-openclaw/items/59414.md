## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Reuse shared home-path redaction:** `src/commands/doctor-node-runtime.ts:242-243`
  - body: `buildNodeRuntimeSummary` only treats `/` as the home boundary, so on Windows `USERPROFILE=C:\Users\Alice` and `execPath=C:\Users\Alice\.volta\bin\node.exe` falls through and prints the full username-bearing path in `openclaw doctor` output. Use the existing `shortenHomePath` helper, which already handles `/` and `\`.
  - confidence: 0.93
- **[P2] Cover all accepted Node release lines:** `src/commands/doctor-node-runtime.ts:121-136`
  - body: `NODE_RELEASE_SCHEDULE` only includes Node 22, but accepted runnable lines include Node 24 and Node 25; Node 25 reaches EOL on 2026-06-01 while still satisfying the runtime floor, so Doctor would emit no lifecycle warning. Cover all relevant accepted lines or derive this from an authoritative schedule.
  - confidence: 0.9
- **[P3] Keep the minimum Node warning in sync:** `src/commands/doctor-node-runtime.ts:167`
  - body: `runtimeSatisfies` is driven by current main's Node 22.19.0 floor, but the new warning still says `>=22.14.0`. If Doctor reaches this branch for Node 22.18.x, it gives users the wrong minimum; derive the label from the shared runtime contract or update it.
  - confidence: 0.91
- **[P3] Align manager names with runtime-path markers:** `src/commands/doctor-node-runtime.ts:59-68`
  - body: This new display map duplicates `runtime-paths.ts` and is already missing markers that `isVersionManagedNodePath` recognizes, such as `.local/share/fnm`, macOS fnm Application Support, and mise. Those installs will be detected as version-managed but lose the requested manager name, so the mapping should share one contract or cover the same markers.
  - confidence: 0.84
