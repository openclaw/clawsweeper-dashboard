## Summary

Keep open. Current main still models memory search around one resolved embedding provider/model/store, rejects the proposed multi-index config shape, and handles provider/model changes through reindexing; the request is a broad feature that needs maintainer API and migration design.

## What This Changes

Add first-class per-model memory embedding indexes with active/fallback routing, independent ingestion state, status/metrics, and backward-compatible config.

## Best Possible Solution

Design an additive memory contract that preserves current single-index installs while adding canonical chunks, per-model indexes, per-index sync state, selected-index query routing, controlled failover, validation, observability, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs give high-confidence evidence that current main still resolves one memory provider/model/store and treats provider/model changes as reindex events.

## Solution Assessment

Unclear for the exact proposed field names, but the direction is sound. The maintainable path is a designed additive config/storage/query contract rather than a small fallback patch.
