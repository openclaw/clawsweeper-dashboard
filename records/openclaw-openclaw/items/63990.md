## Summary

Keep open. Current main still models memory search around one active provider/model/store and full reindex on provider/model/dimension changes, while the requested multi-index failover would add public config, storage schema, ingestion, query routing, status/metrics, migration, and docs work that needs maintainer API design.

## What This Changes

The issue requests first-class multi-index embedding memory with per-model indexes, active/fallback routing, ingestion status, observability, and backward-compatible single-index behavior.

## Best Possible Solution

Design an additive memory contract that keeps current single-index installs as the default while adding explicit per-model index definitions, canonical chunk storage, per-index backfill/status tracking, single-index query routing with controlled failover, dimension/model validation, and CLI/status/metrics coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current limitation is high-confidence from source, docs, and tests: main resolves one memory provider/model/store and treats provider/model changes as full-reindex events.

## Solution Assessment

Unclear for the exact proposed field names, but the product direction is sound. The best path is a maintainer-designed additive contract across config, storage, ingestion, query routing, observability, docs, and migration rather than a narrow fallback patch.
