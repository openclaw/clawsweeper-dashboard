## Summary

Keep open. Current main and v2026.6.6 still expose one active memorySearch provider/model/store and protect vector correctness by pausing mismatched indexes, not by maintaining model-aware per-index failover.

## What This Changes

Request to add first-class multi-index embedding memory so each embedding model has its own index with model-aware query failover and observability.

## Best Possible Solution

Keep this as the canonical memory-core design tracker for a maintainer-approved per-index schema, ingestion state machine, query router, status surface, and doctor/migration plan inside the memory plugin/API boundary.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature proposal. Source and release inspection confirm the current behavior remains a single active memorySearch provider/model/index, with provider/model changes treated as index identity mismatches requiring repair or rebuild.

## Solution Assessment

No, not as an autonomous fix yet. The right solution needs maintainer-approved memory-core/plugin API design for per-index storage, ingestion status, routing, observability, and upgrade behavior rather than a narrow patch.
