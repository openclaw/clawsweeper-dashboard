## Summary

Keep open. Current main still has one resolved memory embedding provider/model/store and treats provider or model changes as reindex events, while the request is a broader multi-index feature spanning config, storage, ingestion, routing, observability, and migration.

## What This Changes

Request first-class memory support for multiple per-model embedding indexes with active/fallback routing, independent ingestion status, observability, and backward-compatible single-index behavior.

## Best Possible Solution

Design an additive memory contract that keeps current single-index installs working while adding canonical chunks, per-model indexes, per-index backfill/status, single-index query routing with controlled failover, dimension validation, and status/metrics/docs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source, docs, and tests give high-confidence evidence that current main resolves one memory provider/model/store and treats provider/model changes as full-reindex events.

## Solution Assessment

Unclear for the exact proposed field names, but the multi-index direction is sound. The maintainable path is an approved additive contract across config, storage, ingestion, routing, observability, migration, tests, and docs rather than a narrow fallback-only patch.
