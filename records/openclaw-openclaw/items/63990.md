## Summary

Keep open. Current main still lacks first-class multi-index embedding memory: configuration, schema, resolver, storage, ingestion, fallback, docs, and tests all model a single active memory embedding provider/model/store, with provider or model changes handled by full reindex rather than routing between prebuilt per-model indexes.

## What This Changes

The issue requests multi-index embedding memory so each embedding model has its own index, independent ingestion state, model-aware query routing, controlled failover, observability, and backward-compatible single-index defaults.

## Best Possible Solution

Design an additive memory contract that keeps the current single-index setup as the default while adding explicit per-model index definitions, canonical chunk storage, per-index backfill/status tracking, single-index query routing with controlled failover, dimension/model validation, and CLI/status/metrics coverage for readiness, lag, errors, and failover events.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current limitation is high-confidence from source, docs, and tests: current main resolves one active memory provider/model/store and reindexes on model/provider changes.

## Solution Assessment

Unclear for the exact proposed field names, but the requested product direction is sound. The best path is a maintainer-designed additive contract across config, storage, ingestion, query routing, docs, and migration rather than a one-file fallback change.
