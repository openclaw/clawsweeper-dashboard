## Summary

Keep open. Current main still exposes a single active memory embedding provider/model/store and rebuilds the index when the provider/model identity changes, while this request is the canonical open tracker for a broader multi-index memory design; the narrower related fallback requests are closed in favor of it.

## What This Changes

The issue requests multi-index embedding memory with one vector index per embedding model, coordinated ingestion, active/fallback index routing, observability, and backward-compatible single-index behavior.

## Best Possible Solution

Design an additive memory contract that preserves current single-index installs while adding canonical chunks, per-model indexes, per-index backfill/status, single-index query routing with controlled failover, dimension validation, and status/metrics/docs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source, docs, and tests give high-confidence evidence that current main has one active memory provider/model/store and treats provider/model changes as full-reindex events.

## Solution Assessment

Unclear for the exact proposed field names, but the multi-index direction is sound. The maintainable path is an approved additive contract across config, storage, ingestion, routing, observability, migration, tests, and docs rather than a narrow fallback-only patch.
