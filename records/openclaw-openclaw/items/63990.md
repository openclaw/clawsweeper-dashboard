## Summary

Keep open. Current main still has a single resolved memory embedding provider/model/store, and fallback still means switching one provider followed by a full reindex, while the requested multi-index failover is a broad public config, storage, ingestion, query routing, observability, migration, and docs feature.

## What This Changes

Request first-class multi-index memory embeddings so each provider/model has its own index, active/fallback index routing, and no mixed vector spaces.

## Best Possible Solution

Design an additive memory contract that preserves current single-index installs while adding per-model indexes, canonical chunks, per-index backfill/status, single-index query routing with controlled failover, validation, and observability.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs give high-confidence evidence that current main resolves one memory provider/model/store and treats provider/model changes as full-reindex events.

## Solution Assessment

Unclear for the exact proposed field names, but the multi-index direction is sound. The maintainable path is a designed additive contract across config, storage, ingestion, routing, observability, migration, and docs rather than a narrow fallback patch.
