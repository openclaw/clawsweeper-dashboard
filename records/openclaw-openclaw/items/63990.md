## Summary

Keep open. Current main still exposes memory search as one provider/model/fallback/store per agent, with provider/model changes triggering a full reindex rather than routing between already-built per-model indexes. The related closed broader request #55590 does not cover this issue's stricter per-index ingestion, no mixed vector spaces, active/fallback index routing, and observability requirements; #43760 is adjacent but only covers separate fallback remote configuration.

## What This Changes

Add first-class per-model memory embedding indexes with model-aware active/fallback routing, independent ingestion status, and observability while preserving existing single-index configs.

## Best Possible Solution

Design an additive memory contract that keeps current single-index behavior as the default while adding explicit per-model index definitions, a canonical chunk source, per-index ingestion/backfill state, read routing to exactly one selected ready index with controlled failover, dimension/model validation, and CLI/status/metrics for coverage, lag, errors, and failover events.

## Reproduction Assessment

Not applicable as a bug reproduction, but the current limitation is high-confidence from source and docs inspection: the schema, resolver, storage schema, ingestion path, and status docs still model one active index per agent.

## Solution Assessment

Unclear for the exact proposed config names, but the product direction is sound. The best solution should be a maintainer-designed additive contract spanning config, storage migration, memory-core ingestion/query routing, and plugin/backward-compatibility boundaries rather than a one-file fallback change.
