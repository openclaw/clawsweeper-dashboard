## Label Justifications

- `P1`: The report describes a live multi-node agent shell workflow where an explicit node execution target still runs on the gateway/container.
- `impact:security`: The issue concerns a shell execution boundary: commands can run on the gateway despite an explicit trusted-node target.
- `impact:session-state`: The session accepts and carries exec host/node defaults, but the Codex native shell path appears not to honor that session state.

## Summary

Keep open: the documented `/exec host=node` contract is still valid, and current main still enables the Codex native shell surface based on tool allowlist/sandbox checks without honoring the session node exec override. The related open PR is a partial mitigation and still needs maintainer/security review plus full behavior proof before this issue can close.

## What This Changes

The issue reports that `/exec host=node node=<remote-macos-node>` is accepted for a session, but Codex-backed agent shell commands still run on the gateway/container instead of the selected node.

## Best Possible Solution

Make the session exec target the single source of truth for shell execution: when a session resolves to `host=node`, Codex native shell must either be disabled or routed through OpenClaw's node-backed exec path, with regression coverage and a real multi-node proof.

## Reproduction Assessment

Yes, at source level: in a Codex app-server-backed session, `/exec host=node node=<mac-node>` is resolved into OpenClaw exec overrides, but the Codex native shell surface is still enabled without checking those overrides. I did not live-run the required Linux gateway plus macOS node setup.

## Solution Assessment

No, current main is not the complete solution because it only routes the OpenClaw dynamic `exec` tool through node execution; it does not make Codex native shell execution honor or yield to `host=node`. The safer fix needs an explicit maintainer decision on native-shell routing or fail-closed behavior.
