## Label Justifications

- `P1`: The report describes a broken multi-node agent shell workflow where explicit node execution still runs on the gateway/container.
- `impact:security`: The issue concerns a shell execution boundary: commands can run on the gateway despite an explicit trusted-node target.
- `impact:session-state`: The session accepts and persists exec host/node defaults, but the Codex native shell path appears not to honor that session state.

## Summary

Keep open: current main still appears to let Codex native shell execution ignore the session's node exec target, and the reporter added fresh node-side verification that the macOS node itself returns Darwin.

## What This Changes

The issue reports that a TUI session configured with `/exec host=node ...` still has agent shell commands run in the Linux gateway/container instead of the selected macOS node.

## Best Possible Solution

Make the session exec target the single source of truth for Codex-backed shell surfaces, with Codex native shell disabled/rerouted or explicitly rejected for `host=node`, plus regression coverage and real multi-node proof.

## Reproduction Assessment

Yes, at source level: `/exec host=node` is documented and persisted, normal OpenClaw exec routes node hosts through `executeNodeHostCommand`, but the Codex native tool-surface gate ignores `execOverrides.host`. I did not live-run the required gateway/container plus macOS node setup.

## Solution Assessment

No, current main is not the complete solution because it threads exec overrides into OpenClaw dynamic tools but leaves Codex native shell enabled independently. The maintainable fix should gate or reroute that native shell path, or clearly fail closed when node routing is unsupported.
