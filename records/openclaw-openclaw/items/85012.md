## Label Justifications

- `P1`: The report describes a broken multi-node shell workflow where explicit node execution still runs in the gateway/container.
- `impact:security`: Shell commands can execute on a different trusted host boundary than the session selected.
- `impact:session-state`: The session accepts and persists exec host/node defaults, but a Codex native shell path can ignore them.

## Summary

Keep open: this is a valid existing-contract bug, not stale or implemented. The docs and current source support `/exec host=node`, but the Codex app-server native shell surface can still be enabled without considering the session's node exec override.

## What This Changes

The issue reports that `/exec host=node node=<remote-node>` is accepted for a session but Codex-backed agent shell execution still runs in the gateway/container.

## Best Possible Solution

Make the session exec target the single source of truth for all shell surfaces: when a session selects `host=node`, Codex native shell should either be disabled or routed through OpenClaw's node-backed exec path, with regression coverage and real multi-node proof.

## Reproduction Assessment

Yes, at source level: set `/exec host=node node=<mac-node>` in a Codex app-server-backed session, then ask for `uname -s`; current source still leaves native Codex shell available without checking the node override. I did not live-run the required multi-host setup.

## Solution Assessment

No, current main is not the complete solution because it threads exec overrides into OpenClaw dynamic tools but does not make Codex native shell execution honor or yield to `host=node`. The maintainable fix should gate or reroute the native shell path rather than relying on model tool choice.
