## Label Justifications

- `P1`: A fresh repro indicates explicit multi-node exec routing is ignored for agent shell commands, blocking a real multi-node workflow.
- `impact:security`: The item concerns shell command execution on the wrong host, which is part of OpenClaw's command-execution security boundary.
- `impact:session-state`: The reported failure is that session-level `/exec` state is accepted and persisted but not honored by the later agent run.

## Summary

Keep open. The fresh report matches current source: `/exec` settings are persisted and chat reply runs pass them as `execOverrides`, but the gateway/TUI agent-command path only forwards session id/key and elevated defaults into embedded/Codex tool construction, so default gateway routing can remain active.

## What This Changes

Session-level `/exec host=node ...` is reported as accepted but not honored by TUI/Codex agent shell tool calls, which continue to execute on the gateway/container.

## Best Possible Solution

Thread session-level exec defaults into gateway/TUI agent-command runs so OpenClaw dynamic exec and Codex app-server tools honor explicit `host=node`/`node`/security/ask overrides while preserving gateway defaults when no override is set.

## Reproduction Assessment

Yes at source level, but not via a live multi-node run in this read-only review. The stored `/exec` session fields and reporter output line up with the gateway/agent-command path not passing `execOverrides` into embedded/Codex tool construction.

## Solution Assessment

Yes. The narrow maintainable fix is to derive and pass existing session exec defaults through the TUI/gateway agent-command path, not to change node exec policy or add a new config surface.
