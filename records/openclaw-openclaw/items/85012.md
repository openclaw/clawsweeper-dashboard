## Label Justifications

- `P1`: The report describes a broken multi-node agent shell workflow where an explicit node execution target still runs commands on the gateway/container.
- `impact:security`: The issue concerns the shell execution host boundary: commands execute on a different trusted host than the session selected.
- `impact:session-state`: The session accepts and persists exec host/node defaults, but the Codex native shell path appears not to honor that session state.

## Summary

This should stay open: the documented `/exec host=node` contract is still source-supported, and current `main` still lets the Codex app-server native shell surface remain enabled without considering the session’s node exec override.

## What This Changes

Issue reports that `/exec host=node node=<remote-node>` is accepted but subsequent agent shell commands still execute on the Linux gateway/container instead of the selected macOS node.

## Best Possible Solution

Make the effective session exec target the single source of truth for all shell surfaces: implicit Codex sessions with `host=node` should use OpenClaw-managed node exec or fail closed with a clear unsupported-runtime message, with real multi-node proof before release.

## Reproduction Assessment

Yes, at source level: set `/exec host=node node=<mac-node>` in an implicit OpenAI/Codex-backed session and ask for a shell host check; current source keeps Codex native code mode enabled without checking the node override. I did not live-run the required Linux gateway plus macOS node setup.

## Solution Assessment

Unclear as a final product decision: current `main` is not the solution, and https://github.com/openclaw/openclaw/pull/84377 is a plausible narrow mitigation for implicit Codex routing but needs maintainer/security acceptance and full managed-exec proof.
