## Label Justifications

- `P1`: The report describes a broken multi-node agent shell workflow where explicit node execution still runs on the gateway/container.
- `impact:security`: The issue concerns a shell execution boundary because commands can run on the gateway despite an explicit node target.
- `impact:session-state`: The session accepts and persists exec host/node defaults, but the Codex native shell path can ignore that session state.

## Summary

This should stay open: current main still has an existing `/exec host=node` contract, but the OpenAI/Codex harness path can select the Codex native shell surface without checking the session's node exec override. There is also an open related PR that appears to target the implicit Codex-runtime placement part of this problem, so the best next step is maintainer/security review rather than closing or queueing another autonomous fix.

## What This Changes

The issue reports that `/exec host=node node=<remote-macos-node>` is accepted but Codex-backed agent shell commands still run on the gateway/container instead of the selected node.

## Best Possible Solution

Make session exec placement the single authority for shell execution: review or adapt https://github.com/openclaw/openclaw/pull/84377 for implicit Codex routing, and define a security-reviewed policy for explicit Codex native shell when `host=node` is active.

## Reproduction Assessment

Yes, at source level: `/exec host=node node=<mac-node>` persists and reaches OpenClaw dynamic tools, while Codex native code mode is enabled without checking that exec override. I did not live-run the required Linux gateway plus macOS node setup.

## Solution Assessment

No, current main is not the complete solution because the native Codex shell path is policy-relayed but not routed through node exec. The maintainable fix should either prevent implicit Codex runtime/native shell from taking over managed node placement or explicitly route native shell through the node-backed exec path.
