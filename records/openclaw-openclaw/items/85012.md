## Label Justifications

- `P1`: The report describes a broken multi-node agent shell workflow where explicit node execution still runs on the gateway/container.
- `impact:security`: The issue concerns a shell execution boundary: commands can run on the gateway despite an explicit node target.
- `impact:session-state`: The session accepts and persists exec host/node defaults, but the Codex native shell path appears not to honor that session state.

## Summary

Keep open: the documented `/exec host=node` contract is still supported by OpenClaw's managed exec tool, but current `main` still lets the Codex app-server native shell surface start without considering the session's node exec override.

## What This Changes

Make agent shell/exec calls honor `/exec host=node node=<remote-node>` so the selected node runs the command instead of the gateway/container.

## Best Possible Solution

Make the session exec target the single routing contract for all shell surfaces: when a session selects `host=node`, Codex native shell must be disabled or routed through OpenClaw's node-backed exec path, with regression coverage and live multi-node proof.

## Reproduction Assessment

Yes, at source level: set `/exec host=node node=<mac-node>` in a Codex app-server-backed session and ask for a shell host probe; current source still enables Codex native code mode without checking the node override. I did not live-run the required Linux gateway plus macOS node setup.

## Solution Assessment

No, current `main` is not complete because it threads exec overrides into OpenClaw dynamic tools but does not make Codex native shell execution honor or yield to `host=node`. The maintainable fix should gate or reroute the native shell path rather than relying on model/tool selection luck.
