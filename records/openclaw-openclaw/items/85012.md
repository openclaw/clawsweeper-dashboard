## Label Justifications

- `P1`: The report describes a broken multi-node agent shell workflow where an explicit node execution target still runs commands on the gateway/container.
- `impact:security`: The issue concerns a shell execution boundary because commands can run on a different trusted host than the selected session target.
- `impact:session-state`: The session accepts and persists exec host/node defaults, but the Codex native shell path does not appear to honor that session state.

## Summary

This should stay open: current main still lets the Codex app-server native tool surface be enabled without considering the effective `/exec host=node` target, and an open linked fix PR is already the right review path.

## What This Changes

The issue reports that Codex app-server agent shell commands accept `/exec host=node node=<id>` but still execute in the gateway/container instead of the selected node.

## Best Possible Solution

Review the linked PR after security review and real multi-node proof; the end state should make the effective exec target the single source of truth for all Codex shell surfaces.

## Reproduction Assessment

Yes, at source level: set `/exec host=node node=<mac-node>` in a Codex app-server-backed session and then ask for `uname -s`; current main can still enable Codex native shell without checking the node exec override. I did not live-run the Linux-gateway plus macOS-node setup.

## Solution Assessment

No, current main is not the complete solution because the OpenClaw dynamic exec path honors node routing but the Codex native shell surface can still be exposed. The best fix should gate or reroute native shell use when the effective exec host is node, with regression coverage and live proof.
