## Label Justifications

- `P1`: The report describes a broken multi-node shell workflow where an explicit node execution target still runs commands on the gateway/container.
- `impact:security`: The issue concerns a shell execution boundary: commands can execute on a different trusted host than the session selected.
- `impact:session-state`: The session accepts and carries exec host/node defaults, but the Codex native shell path can ignore that session state.

## Summary

This should stay open: current main still supports the reported split where `/exec` session defaults feed OpenClaw exec dynamic tools, but the Codex app-server native shell surface is enabled without checking `host=node`. The open mitigation at https://github.com/openclaw/openclaw/pull/84377 is relevant but unmerged and still needs real managed-exec proof, so this is not cleanup-closeable.

## What This Changes

The issue asks OpenClaw to make `/exec host=node node=...` affect subsequent agent shell commands in Codex-backed sessions instead of running them in the gateway/container.

## Best Possible Solution

Make session exec placement authoritative for every shell surface: when a session selects `host=node`, Codex-native shell should be routed through OpenClaw's node exec path or suppressed in favor of the managed `exec` tool, with real multi-node proof before closing.

## Reproduction Assessment

Yes, at source level: set `/exec host=node node=<mac-node>` in a Codex app-server-backed session, then ask for a host-identifying shell command; current source leaves Codex native shell enabled without checking the node override. I did not live-run the required Linux gateway plus macOS node setup.

## Solution Assessment

Unclear as an implementation choice: the current behavior is not acceptable for the documented contract, but maintainers must decide whether the right fix is PR https://github.com/openclaw/openclaw/pull/84377's implicit-runtime fallback, native-shell rerouting, or fail-closed native shell gating for node sessions.
