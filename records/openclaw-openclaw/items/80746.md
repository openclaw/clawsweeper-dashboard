## Summary

Keep open, but this PR should not merge as written: the new `!aborted` gate also catches normal internal timeouts, so it disables the existing high-context timeout-compaction recovery path, and the PR still lacks real Control UI stop proof.

## What This Changes

The PR adds an `!aborted` guard to timeout-triggered compaction and rewrites related tests to expect skipped compaction for aborted timeout results.

## Best Possible Solution

Preserve timeout compaction for internal high-context timeouts, add a targeted user-abort guard using `externalAbort` or abort reason, and keep regression coverage that real timeout attempts may be both `timedOut` and `aborted`.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: ordinary internal timeouts call `abortRun(true)`, which sets both `aborted` and `timedOut`, so the new guard skips timeout compaction. No live proof was provided for the contributor's claimed Control UI stop scenario.

## Solution Assessment

No. The fix should distinguish user-initiated external aborts from internal timeout aborts instead of blocking all aborted timeout attempts from timeout recovery.
