## Summary

Keep this PR open with changes requested: the proposed `!aborted` guard conflates user aborts with internal LLM timeouts, so it would skip timeout-recovery compaction for normal timed-out attempts, and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds an `!aborted` guard to timeout-triggered compaction and updates timeout-compaction tests to expect no compaction for aborted timed-out attempts.

## Best Possible Solution

Distinguish user/caller aborts from internal timeout aborts, likely with `externalAbort` or abort reason, while preserving timeout compaction for internal timed-out attempts and adding focused regression coverage for both paths.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: internal timeouts call `abortRun(true)`, which sets both `aborted` and `timedOut`, so the new guard skips timeout recovery. No for the claimed Control UI stop scenario because the PR has no real after-fix proof.

## Solution Assessment

No. The patch should target user-initiated or external aborts instead of blocking every aborted timed-out attempt from timeout-compaction recovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94
