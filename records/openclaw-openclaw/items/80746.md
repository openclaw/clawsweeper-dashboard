## Summary

Keep open: the patch currently breaks normal timeout recovery because runner-initiated timeouts set both `aborted` and `timedOut`, and the PR still lacks real Control UI stop proof.

## What This Changes

The PR adds an `!aborted` guard to timeout-triggered compaction in `runEmbeddedPiAgent` and rewrites timeout-compaction tests to expect skipped compaction for aborted runs.

## Best Possible Solution

Distinguish user-initiated external aborts from internal timeout aborts so user stops skip timeout compaction without disabling timeout recovery.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: normal timeouts call `abortRun(true)`, which sets `aborted` and `timedOut`, so the new guard prevents timeout compaction. The claimed Control UI stop bug itself is not live-proven in the PR.

## Solution Assessment

No. The proposed `!aborted` guard conflates internal timeout aborts with external/user aborts; the safer fix is to key off `externalAbort` or abort reason while preserving existing timeout-compaction coverage.
