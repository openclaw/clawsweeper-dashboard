## Summary

This draft PR should stay open because it targets a plausible user-stop problem, but the proposed guard regresses ordinary internal timeout recovery and still lacks real after-fix behavior proof for the Control UI stop path.

## What This Changes

The PR adds an `!aborted` guard to timeout-triggered compaction in `runEmbeddedPiAgent` and rewrites timeout-compaction tests to expect skipped compaction for aborted timeout results.

## Best Possible Solution

Preserve normal timeout-triggered compaction for internal timeouts while adding a targeted user-abort distinction, likely by carrying an explicit abort source or reason through the runner result and covering both paths with regression tests.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: the runner's internal timeout path sets both `timedOut` and `aborted`, so the added `!aborted` guard suppresses ordinary timeout recovery. The original Control UI stop behavior is not proven by a live artifact in the PR.

## Solution Assessment

No. The narrow maintainable fix should distinguish user-requested aborts from internal timeout aborts instead of treating every aborted timeout as ineligible for compaction.
