## Summary

Keep open with changes requested: the PR targets a plausible user-abort timeout-compaction problem, but the patch conflates user aborts with internal timeouts and would block normal timeout recovery; the external PR also has no real after-fix proof.

## What This Changes

The PR adds an `!aborted` guard to timeout-triggered compaction in `runEmbeddedPiAgent` and rewrites related tests to expect skipped compaction for aborted timeouts.

## Best Possible Solution

Distinguish external/user aborts from internal LLM timeouts, likely with `externalAbort` or an explicit abort reason, while keeping tests that prove real timed-out aborted attempts still compact and user stops do not.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: internal timeouts call `abortRun(true)`, which sets both `aborted` and `timedOut`, so the new guard skips timeout compaction. No high-confidence real reproduction was provided for the claimed Control UI stop scenario.

## Solution Assessment

No. The patch should use the existing external-abort signal or a specific abort reason instead of blocking all aborted timeout attempts from timeout recovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94
