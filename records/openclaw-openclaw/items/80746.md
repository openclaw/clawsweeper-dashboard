## Summary

Keep open: the PR targets a plausible user-abort cleanup, but the patch blocks normal timeout-compaction recovery because internal timeouts also set `aborted`, and the external contributor has not provided real Control UI stop-flow proof.

## What This Changes

The PR adds an `aborted` guard to timeout-triggered compaction in `runEmbeddedPiAgent` and changes related tests to expect skipped compaction for aborted timeout attempts.

## Best Possible Solution

Preserve normal internal timeout-triggered compaction, add a targeted user-abort guard based on `externalAbort` or abort reason, and cover both real timeout and user stop cases with focused tests plus real behavior proof.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: ordinary timeout paths call `abortRun(true)`, which sets both `aborted` and `timedOut`, so the new guard skips compaction for real internal timeouts. No high-confidence real reproduction was provided for the claimed Control UI stop scenario.

## Solution Assessment

No. The patch should distinguish user/caller aborts from internal timeout aborts instead of using the generic `aborted` flag and deleting regression coverage for timed-out attempts that set `aborted`.
