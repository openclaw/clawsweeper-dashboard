## Summary

Keep open. The hook direction fits OpenClaw's plugin-first architecture, but the proposed merge point would let plugin output bypass the current host exec environment sanitizer, including keys that docs and tests treat as part of the host execution security boundary.

## What This Changes

The PR adds a typed `resolve_exec_env` plugin hook, exposes a hook-runner method, calls it from exec environment construction, and adds focused hook tests.

## Best Possible Solution

Revise this into a documented, additive exec-environment seam that preserves the existing host env reserved-key policy or adds an explicit trusted-plugin gate, carries the needed channel/account/thread/sender context, and has focused coverage for gateway, node, and sandbox behavior. If general env injection is too broad, narrow the design to typed execution metadata for the sender/channel identity use case.

## Reproduction Assessment

Yes. Register a `resolve_exec_env` handler returning a blocked key such as `PATH` for `host=gateway`; in the PR diff, `Object.assign(env, pluginEnv)` runs after `sanitizeHostExecEnvWithDiagnostics`, so the blocked value would be present in the final execution env.

## Solution Assessment

No, not as-is. The hook direction is plausible, but the proposed merge point needs reserved-key filtering or explicit trusted-plugin semantics before it is the narrowest maintainable solution for channel-specific exec context.
