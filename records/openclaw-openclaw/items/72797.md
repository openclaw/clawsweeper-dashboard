## Summary

Keep open. The requested hook addresses a real plugin-owned channel/context need, and related #73343/#72796 show the broader identity use case, but this PR is not merge-ready because the new exec-env hook is wired after host env sanitization and is not forwarded to node execution.

## What This Changes

The PR adds a typed `resolve_exec_env` plugin hook, exposes a hook-runner method, invokes it during exec environment construction, and adds focused hook tests.

## Best Possible Solution

Rework this into a documented additive exec-environment seam that preserves the existing host env security policy or adds an explicit trusted-plugin path, forwards the audited override set consistently to gateway, sandbox, and node hosts, and carries the channel/sender context needed for the identity use case without hardcoding channel plugins into core.

## Reproduction Assessment

Yes. A handler returning `PATH` or `LD_PRELOAD` would be merged after the existing host sanitizer in the PR diff, and a handler returning `FOO=bar` for `host=node` would only mutate local `env` while node `system.run` receives `requestedEnv: params.env`.

## Solution Assessment

No. A dedicated hook can be the right shape, but the current merge point bypasses the exec security contract and misses the node forwarding path, so maintainers should settle the reserved-key/trusted-plugin semantics before merge.

Please route the sanitized hook env through the node request path and add node coverage.
