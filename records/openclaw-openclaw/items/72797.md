## Summary

Keep open. The PR is a plausible additive SDK feature, but the current branch should not merge yet because hook output bypasses the existing exec env security boundary, does not reach node `system.run`, lacks the sender/chat context needed for the motivating per-user env vars, and still has no real behavior proof.

## What This Changes

The PR adds a `resolve_exec_env` plugin hook, hook runner/types/tests, and merges plugin-provided environment variables into exec tool execution.

## Best Possible Solution

Ship a documented additive exec-env SDK contract that reuses host/node/sandbox env validation, forwards audited values consistently, threads the needed sender/chat context, and aligns with https://github.com/openclaw/openclaw/issues/73343.

## Reproduction Assessment

Yes for the PR defects by source inspection: hook output is merged after host env sanitization, node execution builds `system.run` env from `requestedEnv: params.env`, and exec defaults do not carry sender/chat context. The requested hook itself is a new capability, so bug-style reproduction is otherwise not applicable.

## Solution Assessment

No, not as currently patched. A dedicated hook is plausible, but the implementation needs sanitizer/audit reuse, node forwarding, sender/chat context, docs, and real runtime proof before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve host env filtering for hook output:** `src/agents/bash-tools.exec.ts:1540`
  - body: The hook output is assigned after `sanitizeHostExecEnvWithDiagnostics` rejects blocked request env keys, so a plugin can reintroduce `PATH`, loader variables, proxy/TLS overrides, or runtime-control keys for host execution. Run hook output through the same reserved-key policy, or an explicit trusted-plugin policy, before merging it into the process env.
  - confidence: 0.91
- **[P2] Forward hook env to node execution:** `src/agents/bash-tools.exec.ts:1540`
  - body: For `host=node`, this merge only mutates the local `env` object. The node path still builds the `system.run` payload env from `requestedEnv: params.env`, so variables returned by `resolve_exec_env` will not be sent to node execution.
  - confidence: 0.88
- **[P2] Pass sender and chat context into the hook:** `src/agents/bash-tools.exec.ts:1534-1538`
  - body: The PR's examples include per-user vars such as `FEISHU_USER_ID` and `TELEGRAM_USER_ID`, but the exec hook context passed here only includes agent/session/provider/channel fields. `createOpenClawCodingTools` already receives `senderId`, yet `ExecToolDefaults` does not carry it into exec, so plugins cannot compute the motivating per-sender env values from this hook.
  - confidence: 0.84
