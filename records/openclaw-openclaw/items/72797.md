## Summary

Keep open: the feature direction is plausible and tied to https://github.com/openclaw/openclaw/issues/73343, but this branch still bypasses exec env filtering, does not forward hook env to node execution, and lacks external real behavior proof.

## What This Changes

The PR adds a public `resolve_exec_env` plugin hook, hook runner plumbing/tests, and an exec-time call that merges plugin-returned environment variables into the exec environment.

## Best Possible Solution

Ship an additive exec-env SDK contract that preserves reserved-key filtering, has an explicit trusted or declared env policy, forwards the audited env consistently across gateway/sandbox/node, and aligns with the sender/channel identity contract.

## Reproduction Assessment

Yes for the PR defects: source inspection shows hook output would be merged after the sanitizer, and node execution sends `params.env` rather than the mutated local `env`. The feature itself is a new capability, so bug-style reproduction is not applicable.

## Solution Assessment

No, not as patched. A dedicated hook is a plausible direction, but the merge point must preserve the existing exec env security contract and route the same audited env through node execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve host env filtering for hook output:** `src/agents/bash-tools.exec.ts:1627`
  - body: The hook output is assigned after `sanitizeHostExecEnvWithDiagnostics` has rejected blocked request env keys, so a plugin can reintroduce `PATH`, loader, proxy/TLS, or runtime-control keys for gateway execution. Please run hook output through the same reserved-key policy, or an explicit trusted-plugin policy, before merging it into the process env.
  - confidence: 0.91
- **[P2] Forward hook env to node execution:** `src/agents/bash-tools.exec.ts:1627`
  - body: For `host=node`, this merge only mutates the local `env` object. The node path still builds `system.run` env from `requestedEnv: params.env`, so variables returned by `resolve_exec_env` will not be sent to node execution.
  - confidence: 0.87
