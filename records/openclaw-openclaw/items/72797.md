## Summary

Keep open: current main does not implement the proposed hook, and the PR addresses a real plugin-boundary need, but the branch is not mergeable as-is because it weakens the host exec env security contract and misses node-host forwarding.

## What This Changes

The PR adds a typed `resolve_exec_env` plugin hook, exposes a hook-runner method, calls it during exec environment construction, and adds focused hook tests.

## Best Possible Solution

Revise this into a documented additive exec-environment seam that preserves host reserved-key filtering or adds an explicit trusted-plugin gate, forwards the final audited env consistently for gateway, sandbox, and node hosts, and carries the needed channel/sender context without hardcoding channel plugins into core.

## Reproduction Assessment

Yes. A handler returning `PATH` or `LD_PRELOAD` would be merged after host env sanitization, and a handler returning `FOO=bar` under `host=node` would mutate local `env` while node `system.run` still receives `requestedEnv: params.env`.

## Solution Assessment

No, not as-is. A dedicated hook can be the right shape, but the merge point must preserve the exec security contract, the node forwarding path, and the documented SDK/changelog surface before this is the narrowest maintainable solution.
