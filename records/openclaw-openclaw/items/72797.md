## Summary

Keep open: the PR proposes a useful SDK seam, but the current branch still has blocking env-security and node-forwarding issues, and it lacks after-fix real behavior proof.

## What This Changes

The branch adds a public `resolve_exec_env` plugin hook and calls it during exec environment construction so plugins can contribute channel-specific environment variables.

## Best Possible Solution

Ship an additive exec-env SDK contract that preserves host/node env filtering, routes the final audited env consistently across gateway/sandbox/node, and aligns with the sender/channel identity design.

## Reproduction Assessment

Yes for the PR defects: source inspection shows hook output would be merged after host env sanitization, and node execution builds its `system.run` env from `params.env` rather than the mutated local `env`. For the feature itself, reproduction is not applicable beyond confirming current main lacks the new hook.

## Solution Assessment

No, not as-is. A dedicated hook is a plausible direction, but the merge point must preserve the exec security contract and route the final audited env through node execution before this is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve host env filtering for hook output:** `src/agents/bash-tools.exec.ts:1627`
  - body: The hook output is assigned to `env` after `sanitizeHostExecEnvWithDiagnostics` has rejected reserved host keys, so a plugin can reintroduce `PATH`, loader variables, or other blocked env pivots for gateway execution.   - confidence: 0.91

  - confidence: 0.87
