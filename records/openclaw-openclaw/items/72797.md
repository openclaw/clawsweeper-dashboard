## Summary

Keep open. The hook direction is plausible, but the current PR is not mergeable as-is because its env merge can bypass the existing host exec reserved-key policy, and the same hook output would not reach node-host execution.

## What This Changes

The PR adds a typed `resolve_exec_env` plugin hook, exposes a hook-runner method, calls it during exec environment construction, and adds focused hook tests.

## Best Possible Solution

Revise this into a documented additive exec-environment seam that preserves host reserved-key filtering or adds an explicit trusted-plugin gate, forwards the final audited env consistently for gateway, sandbox, and node hosts, and carries enough channel/sender context for the identity use case without hardcoding channel plugins into core.

## Reproduction Assessment

Yes. For the security finding, a `resolve_exec_env` handler returning `PATH` or `LD_PRELOAD` would be merged after the host sanitizer in the PR diff; for the node finding, a handler returning `FOO=bar` for `host=node` would only mutate local `env`, while node `system.run` uses `requestedEnv`.

## Solution Assessment

No, not as-is. A dedicated hook can be the right shape, but the merge point must preserve the exec security contract and the node forwarding path before this is the narrowest maintainable solution.

The node path builds the `system.run` target env from `requestedEnv: params.env`, so variables returned by `resolve_exec_env` never reach the node host. Please route the sanitized hook env through the node request path and add node coverage.
