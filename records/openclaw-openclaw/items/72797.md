## Summary

Keep open. Current main does not already implement this hook, but the PR is not merge-ready because the latest diff still bypasses part of the existing exec env reserved-key policy, does not carry the sender/chat context needed for the motivating per-user env values, and has no real behavior proof.

## What This Changes

The PR adds a `resolve_exec_env` plugin SDK hook, hook runner/types/tests, and merges plugin-provided environment variables into exec tool execution for gateway/node paths.

## Best Possible Solution

Ship a documented additive exec-env SDK contract that reuses the full exec env validation/audit policy for hook output, threads the sender/chat context needed by https://github.com/openclaw/openclaw/issues/73343, and includes real plugin/runtime proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: hook output is merged and forwarded after the current request-env sanitizer, and the hook context lacks sender/chat identity. The requested hook is a new capability, so bug-style reproduction of an existing behavior is otherwise not applicable.

## Solution Assessment

No, not as currently patched. A dedicated hook may be the right direction, but it should reuse the established exec env policy or define an explicit trusted-plugin policy, carry the needed identity context, document the public contract, and show real runtime behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Run hook env through the full exec sanitizer:** `src/agents/bash-tools.exec.ts:1547-1551`
  - body: The latest filter only calls `isDangerousHostEnvOverrideVarName` before merging and forwarding hook output, but current host exec validation also blocks `PATH` and blocked-everywhere keys/prefixes such as `LD_*`, `BASH_ENV`, and `NODE_OPTIONS`. A plugin can still reintroduce values the request-env path rejects for gateway/node execution; route hook output through the same sanitizer, or a documented trusted-plugin policy, before use.
  - confidence: 0.94
- **[P2] Thread sender context into the exec-env hook:** `src/agents/bash-tools.exec.ts:1538-1543`
  - body: The motivating examples include per-user values like `FEISHU_USER_ID` and `TELEGRAM_USER_ID`, but the context passed here only contains agent/session/provider/channel fields. Since `ExecToolDefaults` does not carry the run's sender/chat identity, channel plugins cannot compute the per-user env values this hook is meant to provide unless the companion sender-context work lands with it.
  - confidence: 0.86
- **[P3] Document the new public hook contract:** `src/plugins/hook-types.ts:1007-1010`
  - body: This adds a public plugin SDK hook without docs for when it fires, host coverage, merge order, override behavior, and reserved-key filtering. Plugin authors need that contract before depending on a security-sensitive exec environment seam.
  - confidence: 0.78
