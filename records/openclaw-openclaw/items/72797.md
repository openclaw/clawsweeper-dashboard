## Summary

Keep open. Current main still lacks this SDK hook, but the PR is a recent feature proposal with blocking exec-env correctness and security issues plus no real behavior proof, so it needs maintainer decision before merge rather than cleanup closure.

## What This Changes

The PR adds a typed `resolve_exec_env` plugin hook, exposes a hook-runner method, calls it during exec environment construction, and adds focused hook tests.

## Best Possible Solution

Revise this into a documented additive exec-env SDK seam that preserves host env filtering, forwards audited env consistently for gateway/sandbox/node, and coordinates sender/channel context with the related identity work.

## Reproduction Assessment

Yes. Source inspection shows a handler returning `PATH` or `LD_PRELOAD` would be merged after host env sanitization, and a handler returning a normal key under `host=node` would mutate local `env` while node `system.run` still receives `requestedEnv: params.env`.

## Solution Assessment

No, not as-is. A dedicated hook can be the right shape, but the merge point must preserve the exec security contract and route the final audited env through node execution before this is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve host env filtering for hook output:** `src/agents/bash-tools.exec.ts:1627`
  - body: The hook output is assigned to `env` after `sanitizeHostExecEnvWithDiagnostics` has rejected reserved host keys, so a plugin can reintroduce `PATH`, loader variables, proxy/TLS overrides, or other blocked env pivots for gateway execution. Please apply the same reserved-key policy, or add an explicit trusted-plugin gate, before merging hook output into the execution environment.
  - confidence: 0.91
- **[P2] Forward hook env to node execution:** `src/agents/bash-tools.exec.ts:1627`
  - body: For `host=node`, this merge only changes the local `env` object. The node path builds the `system.run` target env from `requestedEnv: params.env`, so variables returned by `resolve_exec_env` never reach the node host. Please route the sanitized hook env through the node request path and add node coverage.
  - confidence: 0.87
