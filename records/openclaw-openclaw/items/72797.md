## Summary

Keep open: the requested SDK hook is not on current main and remains relevant to the open identity-in-exec feature request, but this PR still has blocking correctness/security defects and lacks real behavior proof.

## What This Changes

The PR adds a new `resolve_exec_env` plugin hook, a hook runner method, unit tests, and exec-time merging of plugin-provided environment variables.

## Best Possible Solution

Ship an additive exec-env SDK seam that preserves the audited exec env boundary, forwards values consistently through gateway/sandbox/node execution, and composes with the channel identity context tracked by the related feature request.

## Reproduction Assessment

Yes for the PR defects: source inspection shows hook output is merged after host env filtering and node execution still forwards only `params.env`. The requested hook itself is a new capability, so bug-style reproduction is not applicable.

## Solution Assessment

No, not as currently patched. A dedicated hook is a plausible direction, but it needs the same reserved-key policy or an explicit trusted-plugin policy and must route the final audited env through node execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve host env filtering for hook output:** `src/agents/bash-tools.exec.ts:1543`
  - body: The hook output is merged after `sanitizeHostExecEnvWithDiagnostics` rejects blocked request env keys. For host execution this lets a plugin reintroduce reserved values such as `PATH`, `HOME`, loader/runtime options, proxy/TLS overrides, or credential/config pivots that the current code and docs reject. Run hook output through the same policy, or add an explicit trusted-plugin policy, before merging it.
  - confidence: 0.91
- **[P2] Forward hook env to node execution:** `src/agents/bash-tools.exec.ts:1543`
  - body: For `host=node`, this mutates the local `env` object, but the remote `system.run` target is built from `params.requestedEnv`, and the call site still passes `requestedEnv: params.env`. Hook-provided variables will be absent from node execution, so the new hook silently fails for node hosts.
  - confidence: 0.88
