## Summary

Keep open: the PR adds a plausible core SDK seam, but the current patch still bypasses part of the established exec environment sanitizer, lacks the sender/chat context needed for its per-user examples, and has no real behavior proof from a plugin/channel exec run.

## What This Changes

The PR adds a `resolve_exec_env` plugin hook, runner/types/tests, and exec-tool wiring so plugins can contribute environment variables to exec runs.

## Best Possible Solution

Ship a documented additive exec-env SDK contract that reuses the full exec env validation/audit policy, coordinates with the sender/chat identity contract if per-user env is in scope, and includes focused tests plus real runtime proof.

## Reproduction Assessment

Yes for the PR blockers by source inspection: PR head merges hook env after the current sanitizer and passes context without sender/chat identity. Bug-style reproduction of an existing current-main failure is not applicable because this is a new SDK capability.

## Solution Assessment

No, not as currently patched. A dedicated hook may be the right direction, but it should preserve the established exec env policy or document a trusted-plugin exception, carry the needed identity context, document the public contract, and show real runtime behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Run hook env through the exec sanitizer:** `src/agents/bash-tools.exec.ts:1550-1554`
  - body: The hook filters only `isDangerousHostEnvOverrideVarName` before merging and forwarding plugin output, but host exec validation also blocks `PATH`, invalid env keys, and blocked-everywhere keys/prefixes such as `BASH_ENV`, `NODE_OPTIONS`, and `LD_*`. Route hook output through the same sanitizer, or define a documented trusted-plugin policy, before it reaches gateway/node exec.
  - confidence: 0.94
- **[P2] Thread sender context into the exec-env hook:** `src/agents/bash-tools.exec.ts:1541-1546`
  - body: The motivating examples include per-user values like `FEISHU_USER_ID` and `TELEGRAM_USER_ID`, but this call only passes agent/session/provider/channel fields and `ExecToolDefaults` has no sender/chat identity. Channel plugins cannot compute the per-user env this hook is meant to provide unless the companion identity contract lands with it.
  - confidence: 0.86
- **[P3] Document the new hook contract:** `src/plugins/hook-types.ts:1010-1013`
  - body: This adds a public plugin SDK hook without docs for when it fires, which hosts it covers, merge order, override behavior, reserved-key filtering, or privacy/security expectations. Plugin authors need that contract before depending on a security-sensitive exec environment seam.
  - confidence: 0.78
