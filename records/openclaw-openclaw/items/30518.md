## Summary

Keep open: current main has Telegram delivery-pin support and low-level Bot API helpers, but it still does not expose standalone Telegram `pin`/`unpin` message actions through discovery, runtime dispatch, config gating, or Telegram docs. The related implementation PRs I found are closed unmerged, and the open pinned-message read request is related but distinct.

## What This Changes

The issue requests Telegram `pin` and `unpin` message actions so agents can programmatically manage pinned status messages.

## Best Possible Solution

Add Telegram-plugin-owned `pin`/`unpin` action discovery and runtime dispatch behind a consistent `pins` action gate, reuse the existing send helpers, document it, add focused regression coverage, and keep pinned-message reads in the separate related issue.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection on current main gives high confidence that standalone Telegram `pin`/`unpin` action routing is still missing while the low-level helpers exist.

## Solution Assessment

Yes, if maintainers approve the new action surface. The narrow maintainable solution is Telegram plugin-local action discovery and runtime dispatch that reuses existing Bot API helpers and avoids core action-contract changes.
