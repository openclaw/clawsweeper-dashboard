## Summary

Keep this issue open: current main has generic and lower-level Telegram pin/unpin support, but the agent-facing Telegram descriptor and runtime still do not expose standalone actions. Maintainers must approve the chat-administration action gate, default availability, and pin-notification semantics before implementation.

## What This Changes

Expose standalone Telegram message pin and unpin operations through the agent-facing message tool.

## Best Possible Solution

Add canonical `pin` and `unpin` through Telegram's existing action map, discovery, schema contribution, account gate, runtime dispatch, and focused tests, reusing the current helpers after maintainers approve safe defaults and notification semantics.

## Reproduction Assessment

Yes, at source level. Current main's Telegram tool discovery omits pin/unpin and its runtime alias normalizer rejects both names, although lower-level Bot API helpers exist.

## Solution Assessment

The typed Telegram message-action pipeline is the best implementation boundary; a raw Telegram-only escape hatch would duplicate policy. Maintainers still need to decide the action-config default, notification behavior, and unpin-without-message semantics.

## Risks / Open Questions

- A new Telegram chat-administration action needs an explicit account action gate and default; exposing it without one would bypass the existing operator-controlled action model.
- The pin action needs a documented `notify` default, while unpin must define whether an omitted message ID unpins the current message or all messages.
- Adding a new Telegram action-config key affects exported types, schema/help metadata, account overrides, and fresh-install behavior, so these surfaces must remain aligned.
