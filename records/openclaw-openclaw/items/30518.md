## Summary

Keep open: current main still has Telegram send-layer pin/unpin helpers and delivery-pin support, but it does not expose agent-facing Telegram `pin` or `unpin` message actions through discovery or dispatch. The remaining work is real, but it needs maintainer product/security direction before automation should queue a fix.

## What This Changes

Request to expose Telegram `pin` and `unpin` as message tool actions for agents maintaining pinned Telegram status messages.

## Best Possible Solution

Add Telegram-plugin-owned `pin` and `unpin` message actions with explicit gating, conservative unpin semantics, docs, focused action/runtime tests, and live Telegram proof after maintainer product/security approval.

## Reproduction Assessment

Yes, at source level: current Telegram discovery and dispatch omit `pin` and `unpin`, and the runtime alias path has no supported action for those names. I did not run a live Telegram probe because there is no current action surface to exercise.

## Solution Assessment

Unclear until maintainers choose the privileged action contract. The likely best fix is a Telegram-plugin-owned action surface with a dedicated gate and conservative unpin semantics, not a core workaround or delivery-pin-only behavior.
