## Summary

Keep open: current main and the latest release still have Telegram send-layer pin/unpin helpers, but the Telegram message-tool discovery and runtime dispatch surfaces do not expose agent-facing `pin` or `unpin` actions. The remaining work is real, bounded, and security/product-sensitive because it would let agents mutate persistent Telegram pinned state.

## What This Changes

Requests Telegram `pin` and `unpin` actions in the agent-facing message tool so agents can manage pinned status/dashboard messages programmatically.

## Best Possible Solution

Add Telegram-plugin-owned `pin` and `unpin` message actions with explicit privileged gating, message-id-safe unpin behavior, docs, focused action/runtime tests, and live Telegram proof after maintainer product/security approval.

## Reproduction Assessment

Yes, at source level: current Telegram discovery and dispatch omit `pin` and `unpin`, and unmapped actions are rejected before reaching the lower-level helpers. I did not run a live Telegram probe because there is no current action surface to exercise.

## Solution Assessment

Unclear until maintainers choose the privileged action contract. The likely best implementation is a Telegram-plugin-owned action surface with a dedicated gate and conservative unpin semantics, not a core workaround or reuse of delivery-pin alone.
