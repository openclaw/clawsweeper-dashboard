## Summary

Keep open: current main and v2026.6.8 still have Telegram delivery-pin/send helpers, but the Telegram message action discovery and runtime dispatch surfaces do not expose agent-facing `pin` or `unpin` actions. The related pinned-message read issue is adjacent context, not a replacement for this mutation request.

## What This Changes

The issue requests Telegram `pin` and `unpin` actions through the shared agent-facing message tool.

## Best Possible Solution

Add Telegram-plugin-owned `pin` and `unpin` message actions with explicit action gating, message-id-safe unpin semantics, docs, focused action/runtime tests, and live Telegram proof after maintainer product/security approval.

## Reproduction Assessment

Yes, at source level: current Telegram discovery and dispatch omit `pin` and `unpin`, and the alias path rejects unmapped actions. I did not run a live Telegram probe because there is no current action surface to exercise.

## Solution Assessment

Unclear until maintainers choose the privileged action contract. The likely best implementation is a Telegram-plugin-owned action surface with a dedicated gate and conservative unpin semantics, not a core workaround or reuse of delivery-pin alone.
