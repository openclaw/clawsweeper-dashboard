## Summary

Keep open: this PR targets a real, source-proven cron run-history deep-link bug, but it is not ready because after-fix real behavior proof is missing and the new regression test appears to fail against existing pre-transcript cron persistence behavior.

## What This Changes

The PR passes `runSessionKey` into cron session persistence, writes the persisted cron session entry under both the stable cron key and per-run key, and adds helper-level coverage.

## Best Possible Solution

Keep the narrow alias-write approach, fix the added test to respect non-resumable pre-transcript cron rows, then merge after contributor-provided real behavior proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-reproducible: current main records a per-run cron `runSessionKey` for run history while session persistence only stores the stable `agentSessionKey`, so `sessions.get` can return empty history for the linked key.

## Solution Assessment

Is this the best way to solve the issue? The implementation direction is the narrowest maintainable fix, but this PR must first repair the failing test expectation and provide real behavior proof before merge.
