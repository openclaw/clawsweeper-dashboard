## Summary

Keep open for normal maintainer review: the prior proof gap is addressed by a successful real Linux NFS proof run, and I found no blocking line-level defect, but maintainers still need to accept the NFS-only journal-mode compatibility tradeoff before merge.

## What This Changes

The branch adds NFS-backed filesystem detection to the shared SQLite WAL helper, switches detected paths to DELETE journaling, passes memory-core database paths into the helper, and adds focused regression tests.

## Best Possible Solution

Land the centralized NFS journal-mode mitigation once maintainers accept the NFS-specific rollback-journaling tradeoff, keeping local filesystems on WAL and avoiding a new config surface.

## Reproduction Assessment

No, I did not establish a live current-main Azure/NFS corruption reproduction in this read-only review. The source path is strong: current main forces WAL, SQLite documents WAL as unsupported over network filesystems, and the reporter tied the failure to ACA NFS storage.

## Solution Assessment

Yes, this looks like the narrow maintainable fix: centralize the journal decision in the existing SQLite helper and pass memory-core the missing path context. The broader configurable mmap PR at https://github.com/openclaw/openclaw/pull/60349 is not a cleaner canonical path for this specific WAL-on-NFS failure.
