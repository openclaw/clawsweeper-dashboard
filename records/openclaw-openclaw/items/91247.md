## Summary

Keep open for maintainer review: the patch looks correct and has sufficient real NFS proof, but the automatic WAL-to-rollback switch for existing NFS-backed state volumes is a compatibility and persisted-state tradeoff that needs explicit maintainer acceptance.

## What This Changes

The branch adds NFS-backed filesystem detection to SQLite WAL maintenance, switches detected database paths to DELETE journaling, passes memory-core database paths into the helper, and adds regression tests.

## Best Possible Solution

Land the centralized NFS journal-mode mitigation once maintainers accept the NFS-specific rollback-journaling tradeoff, keeping local filesystems on WAL and avoiding a new config surface.

## Reproduction Assessment

No high-confidence live current-main Azure/NFS corruption reproduction was established in this read-only review. Source inspection shows current main forces WAL and SQLite documents WAL as unsupported over network filesystems, matching the linked Azure Container Apps restart failure.

## Solution Assessment

Yes, this appears to be the narrow maintainable fix: centralize the journal decision in the existing SQLite helper and pass memory-core the missing path context. The broader configurable mmap PR at https://github.com/openclaw/openclaw/pull/60349 is not the cleaner canonical path for this WAL-on-NFS failure.
