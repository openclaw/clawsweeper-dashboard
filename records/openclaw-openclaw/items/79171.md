## Summary

Keep open. This protected MEMBER/maintainer PR is still meaningful, but the latest head does not look clean yet because the normal Proxyline-managed undici path appears to miss the IP-literal SNI handling added to the validation/helper paths.

## What This Changes

The PR expands managed proxy routing to accept HTTPS proxy endpoints, adds scoped proxy CA config and CLI validation, threads CA options through runtime transports, and updates docs/tests/baselines/changelog.

## Best Possible Solution

Keep the HTTPS proxy/CA design, but make the Proxyline-managed runtime path use the same HTTPS proxy TLS/SNI behavior as validation and plugin helper dispatchers before landing.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible, not live-reproduced: current main shows HTTP-only behavior, and the remaining PR blocker follows from PR head plus upstream Proxyline/Undici source.

## Solution Assessment

Is this the best way to solve the issue? No, not yet; the direction is maintainable, but the Proxyline-managed runtime path needs the same HTTPS proxy TLS/SNI handling as the validation and helper dispatchers.

- **[P2] Carry the IP-SNI fix into Proxyline routing:** `src/infra/net/proxy/proxy-lifecycle.ts:242-245`

Update the Proxyline-managed dispatcher path or dependency before enabling HTTPS proxy URLs there.
