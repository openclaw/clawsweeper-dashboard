## Summary

Keep open. Current main still lacks an opt-in native audio path: inbound audio is transcribed into text and the runner only submits image media to the model, while #48900 remains an unmerged implementation candidate.

## What This Changes

Request an opt-in config path that forwards inbound voice-note audio directly to omni-modal models, with transcription fallback and continuity metadata.

## Best Possible Solution

Evolve or replace #48900 with a shared opt-in native-audio pipeline that has typed raw media prompt parts, provider capability checks, STT fallback, size limits, and minimal hash/duration/interpretation metadata.

## Reproduction Assessment

Yes. Source inspection of current main shows Telegram voice notes enter preflight transcription, media understanding emits only audio.transcription, and the embedded runner only forwards images to session.prompt.

## Solution Assessment

Unclear. Native forwarding is the right end state, but the maintainable solution needs a shared typed media contract and audit/fallback policy rather than a channel-specific bypass.
