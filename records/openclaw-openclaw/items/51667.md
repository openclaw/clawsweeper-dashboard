## Summary

Keep open: current main and the latest release still have text/image-only model content and transcript-based audio handling, while the closest implementation PR is open but blocked on provider API, security, and product-direction decisions.

## What This Changes

Request an opt-in native audio path so voice notes can be sent to audio-capable models instead of only injecting STT transcript text.

## Best Possible Solution

Define a shared typed native-audio content/provider contract with explicit opt-in egress, capability checks, STT fallback, transcript or hash audit semantics, docs, tests, and live channel proof before landing this feature.

## Reproduction Assessment

Yes. Source inspection on current main shows model turns support text/image content only, current-turn native attachments are image-only, and audio media understanding writes audio.transcription outputs rather than a native model audio part.

## Solution Assessment

No, the proposed config flag alone is not the best complete solution. The maintainable path needs a shared typed audio content contract, provider serializers/capability gates, fallback semantics, payload limits, and transcript/audit handling before exposing a public opt-in.
