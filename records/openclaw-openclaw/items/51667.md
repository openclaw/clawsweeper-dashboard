## Summary

Keep open: current main and v2026.5.19 still process inbound audio as transcription-oriented media understanding, while the remaining native-audio work is already represented by open implementation work that needs product, provider-contract, session, and raw-media security decisions.

## What This Changes

The issue requests an opt-in way to send voice-note audio directly to omni-modal models instead of always converting audio to transcript text first.

## Best Possible Solution

Keep this issue as the canonical tracker and converge implementation through a typed native audio/video media contract with provider capability gates, fallback behavior, schema/help/docs, raw-media egress policy, session/audit handling, and WhatsApp/Telegram proof.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an opt-in feature request. Source inspection gives high confidence that current main still processes inbound audio as `audio.transcription` text and has no native forwarding config or prompt-media path.

## Solution Assessment

Unclear for the final implementation: native audio is a plausible direction, but the maintainable path needs a typed media/provider contract and explicit raw-media policy rather than an untyped or channel-specific bypass.
