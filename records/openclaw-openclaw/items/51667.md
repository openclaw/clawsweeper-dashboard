## Label Justifications

- `P2`: Native audio forwarding is a clear provider/media feature with real user value, but it is not an urgent regression, crash, or data-loss incident.
- `impact:session-state`: Skipping STT changes what enters the durable transcript and can affect replay, search, and session continuity.
- `impact:security`: Native forwarding would send original voice/audio bytes to model providers and needs explicit egress, retention, and audit review.
- `impact:auth-provider`: The feature depends on provider capability metadata, model routing, and preservation of provider auth and transport controls.

## Summary

Keep open: current main and v2026.6.8 still have transcript-oriented audio handling, while native raw-audio model input remains an unresolved provider/content/session/security design. The closest implementation candidate is open but conflicting and still blocked on typed media contracts, transport/auth preservation, output consumers, and proof.

## What This Changes

Request native handling for inbound voice/audio so audio-capable models can receive raw media instead of only STT transcript text, with opt-in fallback configuration.

## Best Possible Solution

Define a shared typed native-audio content/provider contract with explicit opt-in, capability gates, payload limits, STT fallback, transcript/hash audit semantics, docs, tests, and live Telegram or WhatsApp proof before landing this feature.

## Reproduction Assessment

Yes at source level: current main types normal model turns as text/image content, agent-turn attachments are image-only, and audio media understanding returns audio.transcription. I did not run a live WhatsApp/OpenRouter repro in this read-only review.

## Solution Assessment

No, the proposed config flag alone is not the best complete solution. The maintainable path needs shared typed audio content, provider serializers/capability gates, fallback semantics, payload limits, and transcript/audit handling before exposing public opt-in.
