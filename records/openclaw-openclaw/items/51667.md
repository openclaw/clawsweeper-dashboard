## Summary

Keep open: current main and v2026.5.18 still handle inbound audio through transcription-oriented media understanding, while the remaining native-audio path is being explored in open PR work with unresolved typed media, transport, fallback, and raw-media policy blockers.

## What This Changes

Request an opt-in configuration path that sends inbound voice-note audio directly to audio-capable multimodal models instead of always converting it to STT transcript text.

## Best Possible Solution

Keep this issue as the canonical tracker and converge implementation through a typed native audio/video media contract with provider capability gates, fallback behavior, schema/help/docs, raw-media egress policy, session/audit handling, and WhatsApp/Telegram proof.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an opt-in feature request. Source inspection gives high confidence that current main still processes inbound audio as audio.transcription text and has no native forwarding config path.

## Solution Assessment

Unclear for the final implementation: native audio is a plausible direction, but the maintainable path needs a typed media/provider contract and explicit raw-media policy rather than an untyped or channel-specific bypass.
