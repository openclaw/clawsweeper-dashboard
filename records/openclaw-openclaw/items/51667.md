## Summary

Keep open. Current main still uses transcription-oriented audio media understanding, while the related native-audio implementation candidates are open and blocked on prompt-media contracts, config/schema, and raw-media egress policy.

## What This Changes

Request an opt-in native-audio mode that forwards inbound voice-note bytes to audio-capable multimodal models instead of only injecting STT text.

## Best Possible Solution

Keep this as the canonical issue and land or replace one implementation with typed native audio/video media support, provider capability/fallback rules, payload limits, raw-media egress documentation, audit metadata, schema/help/docs updates, and focused Telegram/WhatsApp coverage.

## Reproduction Assessment

Not applicable as a bug reproduction, but source inspection gives high confidence in the current behavior. Current main exposes transcription-only audio media output, Telegram preflight transcribes voice notes, and the embedded runner only forwards image prompt media.

## Solution Assessment

Unclear for implementation, but keeping this issue open is the right triage result. The requested direction is plausible, yet the maintainable fix needs a shared typed media contract and explicit egress/audit policy rather than a channel-specific bypass.
