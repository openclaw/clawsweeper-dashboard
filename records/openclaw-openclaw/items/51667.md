## Summary

Keep open. Current main still routes inbound audio through the transcription/media-understanding path and only submits prompt-local images as native media, while the directly related implementation PR #48900 remains open and unmerged.

## What This Changes

The issue requests an opt-in native audio-input mode that forwards voice-note bytes to omni-modal models instead of always using STT transcription first.

## Best Possible Solution

Keep this issue open and review or evolve PR #48900 into a shared opt-in native-audio pipeline with provider capability checks, typed raw media prompt parts, STT fallback, and minimal hash/duration/interpretation metadata for continuity.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: Telegram or WhatsApp voice notes enter preflight transcription, media-understanding emits audio.transcription, and the embedded runner has only image prompt submission support.

## Solution Assessment

Unclear. The requested direction is plausible, but the maintainable solution should be the shared opt-in implementation path already under review in PR #48900, with typed provider support and raw-media safety controls.
