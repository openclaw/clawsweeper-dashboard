## Summary

Keep open. Current main still routes inbound voice/audio through transcription-oriented media understanding, and the same native-audio work is being explored in an open implementation PR that still needs prompt-media, config/schema, fallback, and raw-media egress decisions.

## What This Changes

The issue requests an opt-in native-audio path that sends voice-note media directly to omni-modal models instead of forcing STT transcription first.

## Best Possible Solution

Keep this as the canonical issue and land or replace the open native-forwarding PR only after typed audio/video prompt media, capability/fallback rules, payload limits, raw-media egress docs, audit metadata, schema/help updates, and Telegram/WhatsApp coverage are settled.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence in the current behavior: audio attachments are transcribed into text context and the embedded prompt boundary forwards images only.

## Solution Assessment

No for an automated direct repair. The requested direction is plausible, but the maintainable solution needs a shared typed prompt-media contract and explicit raw-media egress/audit policy rather than a channel-specific bypass.
