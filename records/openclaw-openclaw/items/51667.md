## Summary

Keep open: current main and v2026.6.8 still route inbound audio through transcription/text media-understanding, not native audio model-input blocks, and the open implementation candidate is conflicting with unresolved provider, security, and session/audit decisions.

## What This Changes

Request an opt-in path to send inbound voice/audio attachments directly to omni-modal models instead of always converting them through STT first.

## Best Possible Solution

Design a capability-gated native audio media contract with explicit provider serialization, fallback behavior, and session/audit metadata before choosing the user-facing config surface.

## Reproduction Assessment

Yes, source-level. Current main routes audio through `transcribeAudio` and injects transcript text into the agent context, while normalized LLM message content still supports text/images only.

## Solution Assessment

Unclear. The requested opt-in flag names the desired user control, but the best implementation needs maintainer decisions on content contracts, provider routing, fallback behavior, raw-media egress, and transcript/audit semantics before code should be queued.
