## Summary

Keep open: current main still routes inbound audio through transcription text and only forwards image attachments as native turn media. The old native-forwarding PR was closed in favor of this issue, and the newer open implementation at https://github.com/openclaw/openclaw/pull/78797 still needs maintainer product/API guidance before it can be treated as the fix.

## What This Changes

The issue requests opt-in native audio forwarding to omni-modal models so voice notes can reach the selected model without first being reduced to STT text.

## Best Possible Solution

Keep this issue as the canonical tracker and converge any implementation on a typed native-audio media contract with provider capability checks, bounded raw-media egress, transcript/audit policy, and fallback semantics.

## Reproduction Assessment

Yes, by source inspection rather than a live WhatsApp/Telegram run: current main transcribes audio into audio.transcription text and forwards only image/* attachments into ACP turns. That reproduces the requested gap as a missing feature, not a broken existing contract.

## Solution Assessment

No, the proposed config flag is not by itself the best solution. The maintainable fix needs a shared native-audio content contract, provider serialization/capability gates, fallback behavior, payload limits, and transcript/audit handling.
