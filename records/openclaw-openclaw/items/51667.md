## Summary

Keep open: current main still uses transcription-oriented audio handling, while the implementation paths found are open PRs with unresolved contract, schema, fallback, and raw-media policy blockers.

## What This Changes

Request opt-in native audio forwarding for omni-modal models so voice notes can reach the model as raw media instead of transcript-only text.

## Best Possible Solution

Keep this issue as the canonical tracker and converge the open PRs only after typed native audio/video media contracts, provider capability/fallback gates, schema/help/docs, raw-media egress/audit policy, and WhatsApp/Telegram proof are settled.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an opt-in feature request. Source inspection on current main gives high confidence that inbound audio still goes through transcription-oriented media understanding and lacks the requested native forwarding option.

## Solution Assessment

Unclear for the implementation, but keeping this issue open is the right triage result. The maintainable solution needs a shared typed prompt-media/provider contract and explicit raw-media egress policy rather than an untyped or channel-specific bypass.
