## Summary

Keep open. Current main still routes inbound audio through transcription/preflight paths and only forwards native image blocks into the embedded prompt path, while the directly related implementation PR #48900 remains open and unmerged.

## What This Changes

Request an opt-in native-audio path that sends WhatsApp/Telegram voice-note bytes directly to audio-capable multimodal models instead of forcing STT transcription first.

## Best Possible Solution

Keep this issue open and use PR #48900 as the implementation candidate or source material for a shared opt-in pipeline with typed audio content parts, model capability checks, STT fallback, bounded raw-media handling, and minimal hash/duration/interpretation metadata for continuity.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: send an uncapped WhatsApp or Telegram voice note with no text, and current main reaches `transcribeFirstAudio`/`audio.transcription` before the embedded runner, whose prompt submission only forwards image blocks.

## Solution Assessment

Unclear. The requested opt-in direction is plausible, but the maintainable solution should align the media-understanding contract, embedded prompt runner, provider content types, and audit metadata rather than adding an audio flag only in one channel path.
