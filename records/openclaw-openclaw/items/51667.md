## Summary

Keep open: current main and v2026.6.8 still have a transcript/text-image media path, not a native audio model-input path, and the remaining work needs maintainer direction on content contracts, provider serialization, raw-media egress, fallback behavior, and session/audit semantics.

## What This Changes

Requested support for routing inbound voice/audio media directly to native audio-capable models instead of always using STT transcription.

## Best Possible Solution

Keep this as the broad native-audio product/API tracker until maintainers choose the typed content contract, provider serialization path, raw-media egress policy, fallback/session semantics, and validation strategy.

## Reproduction Assessment

Yes. Source inspection is enough for the feature gap: main has text/image LLM content, image-only serializers, and audio.transcription media outputs rather than a native audio model-input path; this is not a broken current contract, so no failing runtime repro was required.

## Solution Assessment

No. The issue's config-flag sketch is not sufficient by itself; the maintainable solution needs a shared audio content/provider contract, capability checks, provider serialization, raw-media egress policy, and session/audit behavior before config can safely expose the mode.
