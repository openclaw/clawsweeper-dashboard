## Summary

Keep open: current main and the latest release still use the transcript-oriented audio path, while the remaining native-audio work is now a product/security/provider-contract decision with an open implementation candidate that is not merged.

## What This Changes

The issue requests an opt-in path to send voice-note audio directly to omni-modal models, with fallback to transcription and enough session metadata for auditability.

## Best Possible Solution

Keep this issue as the canonical tracker and converge implementation through a typed native audio/video media contract with provider capability gates, fallback behavior, schema/help/docs, raw-media egress policy, session/audit handling, and WhatsApp/Telegram proof.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an opt-in feature request. Source inspection gives high confidence that current main still processes inbound audio as audio.transcription text and has no accepted native forwarding config or prompt-media path.

## Solution Assessment

Unclear for the final implementation: native audio is a plausible direction, but the maintainable path needs a typed media/provider contract and explicit raw-media policy rather than an untyped or channel-specific bypass.
