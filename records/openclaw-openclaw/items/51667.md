## Summary

Keep open. Current main still routes inbound audio through transcription-oriented media understanding, and the only native raw-audio implementation candidate is an open PR with unresolved prompt-media, config/schema, fallback, and raw-media egress policy blockers.

## What This Changes

The issue requests an opt-in native-audio path that sends voice-note audio directly to omni-modal models instead of requiring STT transcription first.

## Best Possible Solution

Keep this as the canonical issue and land or replace the open implementation with typed native audio/video prompt media, provider capability and fallback semantics, schema/help/docs updates, raw-media egress/audit policy, and focused Telegram/WhatsApp coverage.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main still uses transcription-oriented audio media understanding and does not expose the requested native-audio forwarding option.

## Solution Assessment

Unclear for the implementation, but keeping this issue open is the right triage result. The maintainable solution needs a shared typed prompt-media contract and explicit egress/audit policy rather than a channel-specific bypass.
