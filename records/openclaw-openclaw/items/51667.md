## Summary

Keep open. Current main still implements audio media understanding as transcription text, and the native raw-audio implementation candidate is open rather than merged, with unresolved config/schema, prompt-media contract, and raw-media egress decisions.

## What This Changes

Request an opt-in native audio path that sends voice-note audio directly to omni-modal models with transcription fallback and minimal audit metadata.

## Best Possible Solution

Keep this as the canonical issue and land or replace the open implementation with typed native audio/video prompt-media support, schema/help/docs, provider capability and fallback rules, payload limits, audit metadata, raw-media egress documentation, and focused Telegram/WhatsApp coverage.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main still follows the transcription-only audio path and has no native audio forwarding config or prompt-media contract.

## Solution Assessment

Unclear for implementation, but keeping this issue open is the right triage result. The requested direction needs a shared typed media contract plus explicit raw-media egress and audit policy rather than a channel-specific bypass.
