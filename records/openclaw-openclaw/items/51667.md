## Summary

Keep open: current main and v2026.6.10 still route inbound audio through transcription/text media-understanding, not native audio model-input blocks. The open implementation candidate is conflicting and still needs product, provider, session/audit, and raw-media security decisions before it can safely own the fix.

## What This Changes

Issue requests an opt-in native audio path that sends inbound voice/audio attachments directly to audio-capable multimodal models instead of always using STT transcription.

## Best Possible Solution

Define a capability-gated native audio media contract with typed content blocks, provider serialization, explicit raw-media egress policy, fallback behavior, and session/audit metadata before repairing or replacing the open implementation PR.

## Reproduction Assessment

Yes, source-level. Current main routes audio through `transcribeAudio` and injects transcript text into the agent context, while normalized LLM message content supports text/images only.
