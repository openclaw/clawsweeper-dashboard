## Label Justifications

- `P2`: Native audio forwarding is a clear provider/media feature with real user value, but it is not an urgent regression, crash, or data-loss incident.
- `impact:session-state`: Skipping STT changes what enters the durable transcript and can affect replay, search, and session continuity.
- `impact:security`: Native forwarding would send original voice/audio bytes to model providers and needs explicit egress, retention, and audit review.
- `impact:auth-provider`: The implementation depends on provider capability metadata, model routing, and preserving provider auth and transport controls.

## Summary

Keep open: current main and the latest release still have transcript-oriented audio handling, and the remaining native-audio work needs maintainer API, provider, session, and security direction before implementation.

## What This Changes

The issue requests opt-in native forwarding of inbound audio to audio-capable omni-modal models instead of always forcing STT transcription first.

## Best Possible Solution

Define a shared typed native-audio content/provider contract with explicit opt-in egress, capability checks, STT fallback, transcript or hash audit semantics, docs, tests, and live channel proof before landing this feature.

## Reproduction Assessment

Yes at source level: current main types normal model turns as text/image content and media understanding turns audio into audio.transcription. No live WhatsApp/OpenRouter repro was run in this read-only review.

## Solution Assessment

No, the proposed config flag alone is not the best complete solution. The maintainable path needs shared typed audio content, provider serializers/capability gates, fallback semantics, payload limits, and transcript/audit handling before exposing public opt-in.
