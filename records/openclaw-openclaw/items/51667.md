## Summary

Keep open: current main and v2026.6.8 still route inbound audio through transcription/text context, while native raw-audio model input needs a shared content, provider, security, and session design; the related implementation PR remains open but blocked.

## What This Changes

The issue requests an opt-in native audio mode that sends voice/audio attachments directly to omni-modal models instead of using STT transcription first.

## Best Possible Solution

Define a shared typed native-audio content/provider contract with explicit opt-in egress, capability gates, STT fallback semantics, transcript or hash audit handling, docs, focused tests, and live channel proof before landing core support.

## Reproduction Assessment

Yes. Source inspection on current main shows normal model turns support text/image content only, current-turn attachment forwarding is image-only, and audio media understanding writes audio.transcription outputs into transcript text.

## Solution Assessment

No, the proposed config flag alone is not the best complete solution. The maintainable path needs a typed audio content contract, provider serializers and capability gates, fallback semantics, payload limits, and transcript/audit handling before exposing a public opt-in.
