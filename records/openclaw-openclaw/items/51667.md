## Summary

Keep open: current main and v2026.6.10 still route batch voice-note audio through transcription-only media understanding, and the closest native audio/video PR is open but conflicting and blocked on provider, security, and content-contract decisions.

## What This Changes

Request to add an opt-in native audio path so inbound voice notes can be sent to audio-capable models as raw media instead of only STT transcript text.

## Best Possible Solution

Define a maintainer-approved typed native audio content and provider serialization contract, with explicit raw-media egress policy, fallback behavior, session/audit semantics, schema/docs/tests, and then review a rebased implementation.

## Reproduction Assessment

Yes for the current behavior by source inspection: inbound batch audio still runs through audio.transcription and transcript injection, with no native audio content block or accepted native config key. This is a feature request rather than a broken existing contract.

## Solution Assessment

No for a simple config-flag patch. The maintainable solution needs a shared typed native media contract, provider serialization, explicit raw-media egress policy, and session/audit semantics before implementation.
