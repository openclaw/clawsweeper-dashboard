## What This Changes

The PR extracts QA-lab stable hashing into `extensions/qa-lab/src/stable-hash.ts`, uses it from harness/runtime parity, and adds harness regressions for unreadable and cyclic schema values.

## Best Possible Solution

Keep this PR open, refresh it over current main, and land the shared guarded QA-lab hash helper if the rebased diff preserves current parity changes and the normal gate passes.

## Reproduction Assessment

Yes from source inspection: current main reads schema fields inside duplicated stable hash helpers without guarding getters or cycles, so hostile captured schema values can abort parity classification. I did not execute the failing probe in this read-only review.

## Solution Assessment

Yes; centralizing the guarded stable hash in QA-lab is the narrowest maintainable fix because both harness parity and runtime parity share the same invariant. Per-call-site guards would preserve duplicate hashing logic and drift risk.
