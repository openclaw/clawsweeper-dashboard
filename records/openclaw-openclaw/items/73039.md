## Summary

Keep open: the performance direction is still useful, but the current PR head is not safe to merge because it predates current optional media factory planning, auth/deferred-resolution behavior, and late-bound web runtime policy. Current main does not already contain this exact built-in factory memoization, and related default/cold tool-prep latency reports remain open.

## What This Changes

The PR adds a memoizeToolFactory helper, wraps image/media/PDF/web tool factory creation in createOpenClawTools, and adds unit coverage for cache-key behavior.

## Best Possible Solution

Port the memoization onto current main by caching only planned, exposable factory outputs and preserving execute-time web runtime config binding, then add regression tests and refreshed timing proof.

## Reproduction Assessment

Yes for the PR blockers: source comparison of PR head 6312a19ca7e5 against current main ea16a5e9e10c shows the media factories bypass the current plan and the web factories omit lateBindRuntimeConfig. I did not rerun the original Telegram latency benchmark in this read-only review.

## Solution Assessment

No, not in the current branch shape. Factory memoization can be the right implementation direction, but it needs to wrap the current media planning/auth/deferred-resolution path and keep web runtime policy late-bound at execution time.
