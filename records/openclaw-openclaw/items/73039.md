## Summary

Keep this PR open, but do not merge it as-is. The performance direction is useful, and the author addressed the first cache-key review, but the current PR head is stale against main's optional-media planning and late-bound web runtime contracts.

## What This Changes

The PR adds a memoizeToolFactory helper and wraps image, media-generation, PDF, web search, and web fetch factory construction in createOpenClawTools, with unit tests for cache behavior.

## Best Possible Solution

Rebase the memoization onto current main so it caches only stable expensive factory work while preserving optional-media planning, auth-store reuse, deferred model resolution, and execute-time web runtime binding.

## Reproduction Assessment

Yes. The merge blockers are source-reproducible by comparing PR head call sites with current main's optional-media planner and web runtime context; I did not run a live latency benchmark.

## Solution Assessment

No. Process-lifetime memoization is a reasonable performance fix, but this branch is not the best merge shape until it is rebased around current main's auth-aware factory planning and late-bound web execution semantics.

Overall correctness: patch is incorrect

- **[P2] Preserve optional media factory planning:** `src/agents/openclaw-tools.ts:167-184`
  - body: Current main only builds optional image-generation, video-generation, music, and PDF tools after resolveOptionalMediaToolFactoryPlan applies auth availability plus allow/deny policy. The memoized call sites close over the current runtimeWebTools clone and omit lateBindRuntimeConfig, so a cached tool can miss later runtime provider/config changes; when runtime metadata is present, the cloned object identity also defeats warm cache hits.
  - confidence: 0.88
