## Summary

Keep open. Current main still builds these tools directly, so the performance work is not obsolete, but the PR head is not safe to merge as-is: the web-tool cache key still uses a cloned runtime metadata object that defeats warm hits, the media/PDF memoization ignores mutable auth-profile availability, and the required changelog entry is absent.

## What This Changes

The PR adds a `memoizeToolFactory` helper, unit tests, and wraps image, media generation, PDF, and web tool factory construction inside `createOpenClawTools`.

## Best Possible Solution

Revise the cache boundary so web tools key on stable runtime-provider fingerprints or a stable readonly snapshot, and keep auth/profile-dependent media and PDF tool availability invalidatable by auth-store/config snapshot changes. Then add focused production-path tests and a single-line Unreleased changelog entry for the performance improvement.

## Reproduction Assessment

Yes. For the web cache bug, set active runtime web metadata and call `createOpenClawTools` twice; the production getter returns a fresh clone each time, so `refs: [config, runtimeWebTools]` misses. For auth staleness, cache a media-generation factory result before an auth-profile mtime change and the memoized result bypasses the factory path that current main reruns each turn.

## Solution Assessment

No. Process-lifetime memoization is a plausible optimization, but this implementation keys too coarsely for mutable auth/profile state and too unstably for cloned runtime web metadata; deeper registry caching or explicit versioned invalidation is safer.
