## Summary

Keep open: the performance direction is useful, but the current PR head is behind main and would bypass newer media factory planning plus execute-time web runtime binding, so it is not mergeable as-is.

## What This Changes

The PR adds a memoization helper and wraps selected media, PDF, web search, and web fetch tool factories inside `createOpenClawTools`.

## Best Possible Solution

Repair or replace the branch so memoization sits behind current media policy/auth planning and keeps web search/fetch runtime config late-bound at execute time.

## Reproduction Assessment

Yes for the merge blockers: source comparison shows the PR head bypasses current main's media planning and omits web late binding. I did not rerun the original latency benchmark in this read-only pass.

## Solution Assessment

No, not in the current branch shape. Factory memoization can be the right optimization, but it must be rebuilt on current main's policy/auth planning and runtime web binding contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve optional media factory planning:** `src/agents/openclaw-tools.ts:145-254`
  - body: This cached construction path builds image/media/PDF tools directly and never applies current main's `resolveOptionalMediaToolFactoryPlan`, `resolveImageToolFactoryAvailable`, run `authProfileStore`, or `deferAutoModelResolution`. After rebase, tools denied by policy or unavailable by auth can still be constructed on the hot path.
  - confidence: 0.91

  - confidence: 0.9
