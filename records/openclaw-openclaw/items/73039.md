## Summary

Keep open. The performance direction is plausible, but the current PR head still keys web-tool memoization on the active runtime metadata object, while the production getter returns a fresh structured clone on every read. That preserves correctness but defeats the warm-cache path for active runtime web providers, and the PR also lacks the required changelog entry for a user-facing performance change.

## What This Changes

The PR adds a WeakMap-chain `memoizeToolFactory` helper with unit tests and wraps image, media-generation, PDF, `web_search`, and `web_fetch` factory construction inside `createOpenClawTools`.

## Best Possible Solution

Revise the web-tool memoization to key on stable runtime-provider fingerprints, or expose a stable readonly runtime metadata snapshot, then add production-path coverage through the real runtime metadata getter and include an Unreleased changelog entry for the performance improvement.

## Reproduction Assessment

Yes. Set active runtime web metadata, call `createOpenClawTools` twice, and the production getter returns a fresh clone each time; with the PR head key `refs: [options?.config, runtimeWebTools]`, active web tools miss the warm cache.

## Solution Assessment

No. Process-lifetime memoization is a reasonable narrow approach, but keying web tools on the cloned runtime metadata object defeats the intended warm-cache behavior under active runtime providers; a stable provider fingerprint or stable snapshot is safer.
