## Summary

Keep open: the latency problem is credible, but this PR head is not merge-safe against current main because it memoizes older tool-construction paths that bypass optional-media planning and late-bound web runtime policy.

## What This Changes

The branch adds a memoizeToolFactory helper with tests and wraps image, media generation, PDF, web search, and web fetch factory construction in createOpenClawTools.

## Best Possible Solution

Keep the memoization direction, but rebase it onto current main so cached factory output preserves optional-media planning, auth/deferred-resolution behavior, and execute-time web runtime binding.

## Reproduction Assessment

Yes for the merge blockers: source comparison shows PR head wraps older direct factory calls while current main requires optional-media planning and late-bound web runtime config. I did not rerun the original latency benchmark in this read-only review.

## Solution Assessment

No, not in the current branch shape. Factory memoization may be the right fix, but it needs to wrap the current contracts rather than replacing them with stale direct construction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve optional-media factory planning:** `src/agents/openclaw-tools.ts:145-254`
  - body: This wraps the older direct image/media/PDF factories. Current main first computes resolveOptionalMediaToolFactoryPlan with auth state and allow/deny lists, then passes authProfileStore and deferAutoModelResolution into the factories; bypassing that can build denied or unavailable tools and re-enter the PDF/image model-resolution hot path.
  - confidence: 0.91

  - confidence: 0.9
