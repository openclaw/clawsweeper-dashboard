## Summary

Keep open. The performance direction is still relevant, but this PR head is not safe to merge because it memoizes an older `createOpenClawTools` shape and would need to preserve current main's auth-aware media planning plus late-bound web runtime behavior before it can land.

## What This Changes

The PR adds a `memoizeToolFactory` helper and unit tests, then wraps image, media generation, PDF, web search, and web fetch factory construction inside `createOpenClawTools`.

## Best Possible Solution

Rebase the optimization onto current main, keep the optional-media planner and web late-binding contracts intact, and cache only the stable expensive work with focused regression and performance proof.

## Reproduction Assessment

Yes. The patch-level problems are source-reproducible by comparing PR head against current main's auth-aware media planner and late-bound web runtime path; I did not run a live latency benchmark in this read-only pass.

## Solution Assessment

No. Process-lifetime memoization is plausible, but this branch needs to be reshaped around current main's availability planning and stable runtime invalidation before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve the auth-aware media factory plan:** `src/agents/openclaw-tools.ts:167-184`
  - body: Current main first calls `resolveOptionalMediaToolFactoryPlan` with `authProfileStore` plus allow/deny lists and passes the auth store into image/video/music/PDF factories. This branch wraps those factories directly and never forwards `authProfileStore`, so after rebase auth-profile-only providers can disappear and policy-denied optional tools can still be built; keep the planner in front of memoization and include the auth/profile inputs in the cache boundary.
  - confidence: 0.9
- **[P2] Keep web tools late-bound with a stable cache key:** `src/agents/openclaw-tools.ts:255-276`
  - body: Current main passes `lateBindRuntimeConfig: true` so `web_search` and `web_fetch` read active runtime secrets/config at execution time, and the state getter returns a fresh structured clone on each read. Keying the cache on that clone while capturing it at construction either bypasses warm caching on active-runtime turns or freezes stale runtime metadata; memoize on a stable provider/version fingerprint or leave the late-bound tool un-memoized.
  - confidence: 0.88
