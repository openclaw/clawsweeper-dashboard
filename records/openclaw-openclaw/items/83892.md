## Summary

Keep open. Current main still spawns the detached restart script without either a synchronous `spawn()` guard or a `ChildProcess` `error` listener; the ENOENT case is an unhandled child-process error event rather than the exact synchronous rejection described, but it can still terminate the CLI during the update restart handoff.

## What This Changes

The issue asks for `runRestartScript` to handle restart script spawn failures so update restart handoff cannot surface an unhandled process error.

## Best Possible Solution

Make `runRestartScript` best-effort and non-fatal by handling child `error` events and synchronous `spawn` validation failures while preserving detached restart behavior.

## Reproduction Assessment

Yes, at source level. Current main lacks a child-process error listener, and a direct Node probe shows failed spawn emits an unhandled `error` event unless the child handles it; I did not run a full OpenClaw update end-to-end.

## Solution Assessment

Yes, the recommended direction is the narrow maintainable fix. Handle `ChildProcess` `error` events and synchronous `spawn` throws inside `runRestartScript`, with a focused regression test, rather than relying on the caller to catch a post-return child event.
