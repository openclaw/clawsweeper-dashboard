Keep open. Current main still has source-level paths for the reported compaction reliability cluster: cache-inclusive prompt counts can feed context-engine compaction, large empty preparations still commit a structured fallback summary, and the reply path still runs preflight compaction before the documented memory flush step.

## What This Changes

The issue reports premature/empty compactions caused by cache-inclusive token accounting, strict empty-preparation fallback summaries, and memory flush ordering that can trail compaction.

## Best Possible Solution

Land a coordinated compaction reliability fix that uses trusted active-context snapshots for pressure, prevents large empty fallback summaries from replacing real sessions, and guarantees memory flush cannot trail a destructive compaction for the same cycle.

Yes, at source level. The issue includes concrete production traces, and current main still has the cache-inclusive currentTokenCount path, empty fallback boundary path, and preflight-before-flush ordering; I did not run a live Opus/proxy replay.

Fix the compaction reliability cluster reported in openclaw/openclaw#72964 on current main. Keep the boundary narrow: do not remove cacheRead from every usage surface, and do not rework reserve-token/soft-threshold policy beyond what is needed. Separate trusted active prompt/context estimates from provider billing or aggregate cache totals so anomalous cacheRead values do not prematurely trigger destructive compaction, while preserving legitimate cached prompt snapshots needed by #66520/#66716. Prevent `compactionSafeguardExtension` from committing an empty structured fallback for large sessions that have real bulk but no entries passing the strict real-conversation filter; fall through to safe summarization or cancel/preserve history instead. Coordinate with #74010 reserve-token changes and stop for maintainer review if the fix would require changing public compaction policy. Add focused regression coverage and a user-facing changelog fix entry.
