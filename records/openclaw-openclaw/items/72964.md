## Summary

Keep open. Current main still has the reported context-retention risk: cache-inclusive prompt token derivation remains in the compaction/session accounting paths, the safeguard still commits an empty structured fallback whenever strict real-message filtering finds no entries, and reply handling still checks preflight compaction before the memory-flush gate. The related #66716 PR is open and targets the opposite cached-token under-trigger report, so it is not a merged fix for this issue.

## What This Changes

The issue reports that compaction can trigger from cache-read-inflated token usage, then commit an empty fallback summary before the configured memory flush has preserved context.

## Best Possible Solution

Implement a focused agents/compaction reliability fix that uses a trustworthy prompt/envelope or transcript snapshot for compaction pressure, treats provider billing totals as telemetry unless normalized, refuses to commit a large empty fallback summary without real summarization or a safe defer path, and gives configured memory flush a chance to run before destructive compaction commits.

## Reproduction Assessment

Yes. The issue includes concrete production traces, and current main has a high-confidence source-level reproduction path: feed a cacheRead-inflated last-call usage into prompt/context derivation, prepare a high tokensBefore compaction whose entries fail isRealConversationMessage, and observe the empty fallback boundary plus preflight-before-flush ordering.

## Solution Assessment

No. Current main is not the best solution because it still uses cache-inclusive prompt derivation in compaction-facing paths, keeps the large empty fallback path unconditional, and runs preflight compaction before memory flush. A safer fix should coordinate token pressure, fallback summarization, and flush ordering together.
