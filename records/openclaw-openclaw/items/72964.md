## Summary

Keep open. Current main still has a source-level path for the reported reliability cluster: cache-inclusive prompt accounting can feed compaction pressure, empty preparations still commit fallback summaries with the reported token count, and reply handling still runs preflight compaction before memory flush; the related open PR #66716 should be coordinated with this issue rather than treated as a complete fix.

## What This Changes

The issue asks OpenClaw to make compaction safer by avoiding anomalous cache-read token pressure, blocking large empty fallback summaries, and ordering memory flush before destructive compaction.

## Best Possible Solution

Land a focused compaction reliability fix that uses trusted prompt/envelope snapshots or bounded cache accounting, prevents large empty-preparation fallback summaries from replacing real context, and ensures memory flush cannot arrive after a destructive compaction for the same cycle.

## Reproduction Assessment

Yes. The issue includes concrete production traces, and current main has a source-level reproduction path using anomalously high `lastCallUsage.cacheRead`, an empty preparation that fails `isRealConversationMessage`, and the preflight-before-flush ordering in `runReplyAgent`.

## Solution Assessment

Partly. The requested guardrails are the right direction, but the best fix should coordinate with #66716 and preserve valid cached-context detection instead of unconditionally dropping `cacheRead` everywhere.
