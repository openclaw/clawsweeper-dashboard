## Summary

Keep open. Current main still leaves the reported context-retention failure plausible across the same surfaces: cache-inclusive usage can feed compaction pressure, the empty-preparation safeguard still commits a fallback boundary regardless of large `tokensBefore`, and preflight compaction is evaluated before the memory-flush gate. The related #66716 PR overlaps the token-pressure area but does not cover the empty-fallback or flush-ordering hazards.

## What This Changes

The issue reports production context loss when anomalous Anthropic cache-read usage inflates compaction pressure, the compaction safeguard emits an empty fallback summary, and memory flush runs after destructive compaction.

## Best Possible Solution

Land a focused reliability fix that prefers explicit prompt snapshots or envelope estimates over unvalidated provider cache totals, avoids committing empty fallback summaries for large preparations, and ensures memory flush completes before destructive compaction for the same cycle.

## Reproduction Assessment

Yes. The issue provides concrete production traces, and current main has a source-level reproduction path: feed inflated cache-derived usage into prompt pressure, pass a high-`tokensBefore` preparation with no entries accepted by `isRealConversationMessage`, and observe the fallback boundary plus preflight-before-flush ordering.

## Solution Assessment

Partly. The requested direction is right, but excluding `cacheRead` unconditionally is not the safest fix; the maintainable path is to prefer explicit prompt snapshots/envelope estimates and add guardrails for anomalous provider cache usage.
