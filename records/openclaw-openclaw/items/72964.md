## Summary

Keep open. Current main still has the reported reliability risk: cache-inclusive usage can feed context-engine prompt pressure, the empty-preparation safeguard still commits a fallback boundary without considering large `tokensBefore`, and chat replies still run preflight compaction before memory flush.

## What This Changes

The issue reports premature destructive compaction when provider cache-read usage inflates prompt pressure, strict real-message filtering emits an empty fallback summary, and memory flush runs after compaction.

## Best Possible Solution

Land a focused reliability fix that uses trustworthy prompt/envelope snapshots for compaction pressure, makes large empty preparations non-destructive, and guarantees memory flush before destructive compaction for the same cycle.

## Reproduction Assessment

Yes. The issue includes concrete production traces, and current main has a source-level path: pass a high cacheRead last-call usage into after-turn context-engine maintenance and pass a high-tokensBefore empty preparation into the safeguard to observe the empty fallback boundary.

## Solution Assessment

Partly. The requested direction is right, but unconditionally excluding cacheRead is not the safest rule because cached prompt content can still be real current context; the maintainable fix should prefer explicit prompt snapshots/envelope estimates and guard anomalous provider usage.
