## Summary

Keep open. Current main still has the relevant context-retention hazards: prompt/context token derivation can still use cache-inclusive provider usage as a fresh context snapshot, the compaction safeguard still commits an empty structured fallback whenever strict real-message filtering finds no entries regardless of a large `tokensBefore`, and the reply path still runs preflight compaction before the memory-flush gate.

## What This Changes

The issue reports premature compaction from cache-inclusive token accounting, empty fallback summaries for large real sessions, and memory-flush ordering after destructive compaction.

## Best Possible Solution

Land a focused reliability fix that separates billing usage from trustworthy prompt/context pressure, prevents large empty preparations from committing an empty boundary without a real summarization or safe deferral path, and guarantees configured memory flush runs before any destructive compaction commit for the same cycle.

## Repair Work Prompt

Fix the current compaction reliability issue in OpenClaw main. The observable bug is that cache-inclusive or billing-derived token snapshots can drive destructive compaction, large preparations with no messages passing `isRealConversationMessage` can commit an empty structured fallback summary, and `runReplyAgent` currently runs preflight compaction before the configured memory-flush gate. Keep the fix scoped to agents/session compaction and memory-flush behavior. Preserve the valid #66520 requirement that real cached prompt content can still count toward context pressure, and preserve the #41981 loop breaker so empty heartbeat/cron sessions do not spin on cancelled compactions. Do not change provider dependency pins, release metadata, or unrelated model catalogs.

Expected implementation direction: introduce a trustworthy prompt/context pressure distinction rather than blindly treating provider billing totals as destructive compaction pressure; gate or defer the empty fallback path when `tokensBefore` is large enough to indicate real bulk; and ensure memory flush gets a chance to run before any destructive compaction commit for the current compaction cycle, possibly by moving ordering or adding a compaction-safeguard/preflight refusal when `memoryFlush.enabled` is pending. Add focused tests in usage, compaction-safeguard, agent-runner-memory, and runReplyAgent ordering surfaces. Add a user-facing `CHANGELOG.md` fix entry with reporter credit. Coordinate with #66716/#66520 rather than regressing cached-token under-trigger behavior.
