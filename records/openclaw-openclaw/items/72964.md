## Summary

Keep open. Current main still has source-level paths for the reported compaction reliability cluster: cache-inclusive prompt pressure can feed compaction context, the empty fallback boundary has no large-token guard after branch fallback misses, and reply maintenance still runs preflight compaction before memory flush.

## What This Changes

The issue requests safer compaction accounting, non-destructive fallback summaries for large real sessions, and memory-flush ordering that actually precedes compaction.

## Best Possible Solution

Land one coordinated reliability fix that separates trusted active-context estimates from provider billing/cache totals, blocks large destructive empty fallbacks, and guarantees enabled memory flush cannot trail the compaction cycle it is meant to protect.

## Reproduction Assessment

Yes, at source level. The production trace is concrete, and current main still exposes the cache-inclusive currentTokenCount path, high-token empty fallback branch, and preflight-before-memoryFlush ordering; I did not run a live Opus/proxy replay.

## Solution Assessment

Partly. The requested direction is correct, but a blanket cacheRead removal would conflict with related cache-hit compaction bugs, so the maintainable fix should prefer trusted prompt/context snapshots and guard fallback/flush ordering explicitly.
