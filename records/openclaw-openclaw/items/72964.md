## Summary

Keep open. Current main still has source-level paths for the reported compaction reliability cluster, and the related merged/open PRs only partially address adjacent symptoms rather than the full cache pressure, fallback-summary, and memory-flush ordering issue.

## What This Changes

The issue asks OpenClaw to prevent cacheRead-inflated compaction pressure, avoid empty fallback summaries for substantive sessions, and coordinate memoryFlush before destructive compaction.

## Best Possible Solution

Land one coordinated reliability fix that separates trusted active-context estimates from provider billing/cache totals, blocks destructive large empty fallbacks, and ensures enabled memoryFlush cannot trail the compaction cycle it is meant to protect.

## Reproduction Assessment

Yes, at source level. The production trace is concrete, and current main still exposes the cache-inclusive currentTokenCount path, high-token empty fallback path after branch fallback misses, and preflight-before-memoryFlush ordering; I did not run a live Opus/proxy replay.

## Solution Assessment

Partly. The reported direction is right, but a blanket cacheRead removal would conflict with https://github.com/openclaw/openclaw/issues/66520 and https://github.com/openclaw/openclaw/issues/63892, so the safer fix is to prefer trusted prompt/context snapshots and guard fallback/flush ordering explicitly.
