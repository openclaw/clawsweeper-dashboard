## Summary

Keep open. Current main still leaves the reported context-retention failure plausible across the same surfaces: cache-derived prompt pressure can drive context-engine compaction, the empty-preparation safeguard still commits a structured fallback regardless of large `tokensBefore`, and preflight compaction is still evaluated before the memory-flush gate.

## What This Changes

The issue asks OpenClaw to avoid cache-inflated compaction triggers, prevent large empty fallback summaries, and ensure memory flush runs before destructive compaction.

## Best Possible Solution

Land a focused reliability fix that uses a trustworthy prompt/envelope snapshot for compaction pressure, treats provider cache counters as fallback or validated inputs, defers or summarizes large empty preparations instead of committing an empty boundary, and enforces memory flush before destructive compaction for the same cycle.

## Reproduction Assessment

Yes. The issue includes concrete production traces, and current main has a source-level reproduction path: feed inflated cache-derived usage into prompt pressure, pass a high-`tokensBefore` preparation with no entries accepted by `isRealConversationMessage`, and observe the fallback boundary plus preflight-before-flush ordering.

## Solution Assessment

Partly. The requested direction is right, but excluding `cacheRead` unconditionally is not the safest fix because cached prompt content can be real current context; the maintainable solution is to prefer explicit prompt snapshots/envelope estimates and add guardrails for untrusted or anomalous provider cache usage.
