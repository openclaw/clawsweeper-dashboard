## Summary

Keep open. Current main still has the reported context-retention risk: cache-inclusive provider usage can still become the prompt/context snapshot used by compaction paths, the safeguard still writes an empty structured fallback whenever strict real-message filtering finds no entries, and reply handling still runs preflight compaction before the memory-flush gate. The related #66716 PR is open and targets the opposite cached-token under-trigger report in #66520, not a merged fix for this issue.

## What This Changes

Prevent compaction pressure, empty fallback summaries, and memory-flush ordering from discarding real long-context conversations when provider cache usage spikes.

## Best Possible Solution

Implement a focused compaction reliability change that treats provider billing totals as telemetry unless normalized into a trustworthy prompt snapshot, derives compaction pressure from an explicit envelope/transcript footprint where possible, defers or escalates large empty preparations instead of committing an empty boundary, and guarantees the configured memory flush has a chance to run before destructive compaction commits.

## Reproduction Assessment

Yes. The issue includes concrete production traces, and current main provides a high-confidence source-level reproduction path: feed inflated cache-derived usage into prompt-token derivation, prepare a high tokensBefore compaction where all entries fail isRealConversationMessage, and observe the empty fallback boundary and preflight-before-flush ordering.

## Solution Assessment

No. Current main is not the best solution because it still allows cache-derived prompt snapshots without a stronger envelope source, keeps the large empty fallback path unconditional, and runs preflight compaction before memory flush. The safer solution is a coordinated gate across token pressure, fallback summary, and memory-flush ordering.
