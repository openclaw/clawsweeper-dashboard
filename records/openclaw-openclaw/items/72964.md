## Summary

Keep open. Current main still has the reported reliability gaps: prompt/context snapshots can still be derived from cacheRead/cacheWrite usage, the compaction safeguard still writes an empty structured fallback whenever strict real-message filtering finds no messages, and the main reply path runs preflight compaction before memory flush. The related open PR #66716 is adjacent cached-token work for #66520, not a merged fix for this issue.

## What This Changes

The issue requests fixing compaction token accounting, empty fallback summaries, and memory-flush ordering so prompt-cache spikes do not compact away real conversations.

## Best Possible Solution

Implement a focused agents/compaction reliability fix that uses a provider-normalized prompt footprint or explicit prompt snapshot for compaction pressure, escalates or refuses large empty-fallback compactions, and ensures memory-flush metadata is recorded before any destructive preflight or context-engine compaction can commit.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: feed inflated cacheRead into prompt-token derivation, build a high-token compaction preparation where all entries fail isRealConversationMessage, and observe runReplyAgent reaching preflight compaction before memory flush.

## Solution Assessment

Unclear as written. Excluding all cacheRead would risk regressing legitimate prompt-cache context accounting, but an explicit prompt snapshot/envelope plus a large-empty-fallback guard and flush-before-commit ordering is the narrowest maintainable direction.
