## Summary

Keep open. Current main still exposes the reported reliability cluster: cache-inclusive prompt usage can feed compaction pressure, empty preparations still commit structured fallback summaries with the reported `tokensBefore`, and reply handling still runs preflight compaction before memory flush.

## What This Changes

The issue requests safer compaction accounting, non-empty fallback behavior for large real sessions, and memory-flush ordering before destructive compaction.

## Best Possible Solution

Land a focused reliability fix that prefers trusted prompt/envelope snapshots over raw provider totals, blocks large empty-preparation fallback summaries, and ensures memory flush cannot arrive after a destructive compaction for the same cycle.

## Reproduction Assessment

Yes. The issue includes concrete production traces, and current main has a source-level reproduction path through inflated `lastCallUsage.cacheRead`, strict real-message filtering that emits an empty fallback, and preflight-before-flush ordering.

## Solution Assessment

Partly. The requested direction is correct, but unconditionally dropping `cacheRead` would conflict with #66520; the safer solution is a prompt-snapshot/envelope-based accounting path plus guardrails for anomalous provider cache usage.
