## Summary

Keep open. Current main still has the reported reliability gaps: prompt/context snapshots can fall back to cache-derived usage, the compaction safeguard still writes an empty structured fallback whenever strict real-message filtering finds nothing, and the reply path still runs preflight compaction before memory flush. The related PR #66716 is open and aimed at the opposite cached-token under-trigger bug in #66520, not a merged fix for this issue.

## What This Changes

The issue requests safer compaction behavior for cached-token accounting, large empty-fallback summaries, and memory-flush ordering before destructive compaction.

## Best Possible Solution

Implement a focused compaction reliability fix that derives compaction pressure from an explicit prompt/envelope snapshot or transcript-derived prompt footprint, treats provider billing totals as telemetry unless normalized, escalates large empty preparations to real summarization or safe deferral instead of writing an empty boundary, and ensures memory flush is recorded before any destructive compaction cycle commits.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: feed inflated cacheRead/cacheWrite-derived usage into prompt-token derivation, prepare a high tokensBefore compaction where all entries fail isRealConversationMessage, and observe runReplyAgent invoking preflight compaction before memory flush.

## Solution Assessment

No. Current main is not the best solution because it still allows cache-derived prompt snapshots without a safer envelope source, unconditional empty fallback boundaries, and preflight compaction before memory flush; the safer fix should coordinate those three gates together.
