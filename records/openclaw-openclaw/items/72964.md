## Summary

Keep open. Current main still has the reported reliability cluster: cache-inclusive usage can feed context pressure, large empty preparations still commit a structured fallback summary, and the reply path still runs preflight compaction before the memory flush that docs describe as pre-compaction.

## What This Changes

The issue asks OpenClaw to fix compaction token accounting, empty fallback summarization, and pre-compaction memory flush ordering for Anthropic cached-token sessions.

## Best Possible Solution

Land a focused reliability fix that uses trusted prompt/envelope snapshots for compaction pressure, blocks large empty-preparation fallback summaries, and guarantees memory flush cannot trail a destructive compaction for the same cycle.

## Reproduction Assessment

Yes. The issue includes concrete production traces, and current main has a source-level reproduction path through inflated cacheRead-derived context counts, strict real-message filtering, and preflight-before-flush ordering.

## Solution Assessment

Partly. The requested direction is right, but the safest fix should preserve #66520 semantics by separating provider billing totals from trusted active-context snapshots instead of unconditionally dropping cacheRead.

## Repair Work Prompt

Fix issue #72964 as a focused compaction reliability repair. The observable bug is that Anthropic cached-token usage can inflate compaction pressure, an empty preparation can still commit a structured fallback summary at very large tokensBefore, and the reply path can compact before the memory flush that should preserve durable context. Keep the fix scoped to token/context accounting, compaction safeguard fallback gating, and memory-flush/compaction ordering. Coordinate with #66716 and #66520: do not unconditionally drop cacheRead in ways that make cached prompt content invisible when it is a trusted active-context snapshot. Prefer prompt/envelope snapshots or explicit context-engine currentTokenCount inputs over raw provider billing totals, add a high-token guard that falls through to real summarization or cancels safely instead of writing an empty fallback, and ensure memoryFlushCompactionCount/compactionCount gating prevents a flush from arriving after a destructive compaction. Add/update tests near the touched files and include a user-facing CHANGELOG.md fix entry unless maintainers decide this is internal only.
