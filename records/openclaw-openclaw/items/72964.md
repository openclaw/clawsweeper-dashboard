## Summary

Keep open. Current `main` still has the reported reliability cluster: cache-inclusive prompt token derivation can feed context-engine pressure, the safeguard still commits an empty structured fallback whenever the filtered preparation is empty, and the reply path still runs preflight compaction before the memory flush that docs describe as pre-compaction.

## What This Changes

The issue reports premature compaction from cache-inclusive token counts, empty fallback summaries for large real sessions, and memory flush running after destructive compaction.

## Best Possible Solution

Land a focused reliability fix that uses trusted prompt/envelope snapshots for compaction pressure, blocks large empty-preparation fallback summaries, and guarantees memory flush runs or explicitly gates destructive compaction for the same cycle.

## Reproduction Assessment

Yes. The issue includes concrete production traces, and current `main` has a source-level reproduction path through cacheRead-derived prompt counts, strict real-message filtering, and preflight-before-flush ordering.

## Solution Assessment

Partly. The requested direction is right, but the safest implementation should separate provider billing totals from trusted active-context snapshots instead of unconditionally dropping `cacheRead` everywhere.
