## Summary

Keep open. Current main still leaves the reported compaction reliability problem unresolved: context/prompt snapshots can still inherit cache-derived usage, the compaction safeguard still commits an empty structured fallback whenever strict real-message filtering finds nothing, and the main reply path still runs preflight compaction before memory flush. The related open PR #66716 addresses an adjacent cached-token auto-compaction gap for #66520, but it is not merged and does not cover the large empty-fallback or flush-ordering parts of this issue.

## What This Changes

The issue requests safer compaction token accounting, large-session fallback-summary gating, and memory-flush ordering so cached-token spikes cannot destroy active conversation context.

## Best Possible Solution

Implement a focused agents/compaction reliability fix that derives compaction pressure from a trustworthy prompt footprint, escalates or refuses large empty-fallback boundaries, and ensures the configured memory-flush cycle is recorded before any destructive compaction boundary can commit.

Yes. ## Solution Assessment

No. Current main is not the best solution because it still allows cache-derived prompt snapshots, unconditional empty fallback compaction after strict filtering, and preflight compaction before the memory-flush path.
