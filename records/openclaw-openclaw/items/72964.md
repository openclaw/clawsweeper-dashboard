## Summary

Keep open. Current main still has the reported compaction reliability gaps: cache-derived prompt snapshots can become context pressure, empty compaction preparations still write a structured fallback boundary regardless of size, and the reply path still performs preflight compaction before the memory-flush gate. The related PR #66716 is open and aimed at the opposite cached-token under-trigger in #66520, so it is not a merged fix for this issue.

## What This Changes

The issue requests a compaction reliability fix so prompt-cache billing counters do not trigger destructive compaction, large empty preparations do not emit an empty fallback summary, and memory flush runs before compaction.

## Best Possible Solution

Implement a focused agents/compaction fix that derives destructive compaction pressure from an explicit prompt/envelope or transcript-derived snapshot instead of raw provider billing totals, treats large empty preparations as a summarization/deferral path rather than an empty boundary, and records memory flush before committing a destructive compaction cycle.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: model inflated cacheRead/cacheWrite-derived usage into prompt-token derivation, prepare a high tokensBefore compaction whose entries fail isRealConversationMessage, and observe runReplyAgent evaluating preflight compaction before memory flush.

## Solution Assessment

No. Current main is not the best solution because it still allows cache-derived prompt snapshots, unconditional empty fallback boundaries, and preflight compaction before memory flush; the safer fix should coordinate those gates together.
