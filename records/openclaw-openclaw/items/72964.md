## Summary

Keep open. Current main still contains the reported reliability cluster: cache-inclusive usage can feed compaction pressure, high-token empty preparations still write a minimal fallback summary, and reply startup still runs preflight compaction before the memory flush lane.

## What This Changes

Fix premature and empty compaction behavior in cached long-context sessions by correcting compaction token accounting, empty fallback safeguards, and memory-flush ordering.

## Best Possible Solution

Land a focused core reliability fix that uses trusted prompt/envelope snapshots for compaction pressure, blocks large empty-preparation fallback summaries, and ensures memory flush runs before destructive compaction for the same cycle.

## Reproduction Assessment

Yes. The issue includes concrete production traces, and current main has a source-level reproduction path through inflated lastCallUsage.cacheRead, strict real-message filtering that emits an empty fallback, and preflight-before-flush ordering.

## Solution Assessment

Yes in direction, but the token-accounting fix should prefer prompt snapshots or anomaly guards rather than universally excluding cacheRead, so it does not reintroduce #66520's under-compaction behavior.
