## Summary

Keep open. Current main still has the reported reliability cluster: cache-inclusive prompt counts can feed context-engine compaction, empty preparations still commit a structured fallback regardless of token size, and the reply path still runs preflight compaction before the memory-flush gate.

## What This Changes

The issue asks OpenClaw to prevent cached-token inflation, large empty fallback summaries, and memory-flush-after-compaction ordering from destroying long-session context.

## Best Possible Solution

Land a focused reliability fix that uses trusted prompt/envelope snapshots for compaction pressure, prevents large empty-preparation fallback summaries from committing destructive boundaries, and guarantees the memory flush cannot trail a destructive compaction for the same cycle.

## Reproduction Assessment

Yes. The issue includes concrete production traces, and current main has a source-level reproduction path: inflated lastCallUsage.cacheRead can become context-engine currentTokenCount, then an empty preparation can still commit the structured fallback while memory flush runs later in the reply path.

## Solution Assessment

Yes for keeping this open and repairing it, but the implementation should not blindly remove cacheRead because #66520 needs cached prompt tokens to count when they represent the current prompt; the safer path is to prefer explicit prompt/envelope snapshots and only fall back to provider usage when trusted.
