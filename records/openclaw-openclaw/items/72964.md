## Summary

Keep open. Current main still has the reported failure surfaces: cache-inclusive last-call usage can feed compaction pressure, the empty-preparation safeguard still commits a structured fallback with large `tokensBefore`, and the reply runner evaluates preflight compaction before memory flush; the related #66716 work overlaps only part of this cluster.

## What This Changes

Request is to fix compaction token-pressure accounting, empty fallback summaries, and memory-flush ordering for cached Anthropic Opus 1M sessions.

## Best Possible Solution

Land a focused reliability fix that prefers explicit prompt/envelope snapshots over untrusted provider totals, prevents large empty preparations from committing an empty boundary, and ensures memory flush runs before destructive compaction for the same cycle.

## Reproduction Assessment

Yes. The issue includes concrete production traces, and current main has a source-level reproduction path through inflated `lastCallUsage.cacheRead`, empty strict real-message filtering with high `tokensBefore`, and preflight-before-flush ordering.

## Solution Assessment

Partly. The requested direction is right, but unconditionally dropping `cacheRead` would regress cached-context detection; the safer fix is to use trustworthy prompt snapshots/envelope estimates plus guardrails for anomalous provider cache usage.
