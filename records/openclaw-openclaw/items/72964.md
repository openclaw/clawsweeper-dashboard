## Summary

Keep open. Current main still has the core reliability hazards described here: cache-inclusive provider usage is still used as a context-pressure snapshot in several paths, the compaction safeguard still commits an empty structured fallback whenever strict real-message filtering finds no entries regardless of large `tokensBefore`, and the reply runner still calls preflight compaction before the pre-compaction memory flush gate.

## What This Changes

The issue requests a reliability fix for premature compaction, empty fallback summaries, and memory-flush ordering when Anthropic prompt-cache usage reports inflated cached token counts.

## Best Possible Solution

The end state should derive compaction pressure from a trustworthy prompt/context snapshot or envelope, preserve real cached-context detection for #66520, prevent large empty preparations from committing an empty boundary without a real summary or safe deferral path, and guarantee configured memory flush completes before any destructive compaction commit for the same cycle.

## Reproduction Assessment

Yes. The issue includes concrete production traces, and current main has a source-level reproduction path: feed inflated cache-derived usage into the context snapshot path, provide a high-`tokensBefore` preparation with no entries passing strict real-message filtering, and observe an empty fallback boundary while reply pre-maintenance runs compaction before memory flush.

## Solution Assessment

Partly yes. The requested direction is right, but the safest implementation should not blindly drop all `cacheRead`; it should separate billing/telemetry from a trustworthy prompt snapshot, preserve legitimate cached prompt pressure, and add ordering/fallback guards around compaction commits.
