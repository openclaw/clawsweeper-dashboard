## Summary

Keep open. Current main still leaves the reported reliability failure plausible: cache-derived prompt pressure can still feed compaction, the empty-preparation safeguard still commits a structured fallback even for large `tokensBefore`, and the reply runner still evaluates preflight compaction before the memory-flush gate.

## What This Changes

The issue reports that compaction can use cache-inflated token counts, emit an empty fallback summary for a non-empty large session, and run memory flush after destructive compaction.

## Best Possible Solution

Land a focused reliability fix that uses trustworthy prompt/envelope snapshots for compaction pressure, treats anomalous provider cache counters as untrusted inputs, refuses large empty fallback summaries, and ensures memory flush precedes destructive compaction for the same cycle.

## Reproduction Assessment

Yes. The issue includes concrete production traces, and current main has a source-level reproduction path: feed inflated `cacheRead` usage into prompt pressure, pass a high-`tokensBefore` preparation with no messages accepted by `isRealConversationMessage`, and observe fallback compaction before memory flush.

## Solution Assessment

Partly. The requested guardrails are directionally right, but dropping `cacheRead` unconditionally is not the safest fix because cached prompt content can be real current context; the maintainable fix should prefer explicit prompt snapshots or envelope estimates and add anomaly guards around provider cache usage.
