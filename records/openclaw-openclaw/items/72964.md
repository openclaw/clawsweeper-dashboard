## Summary

Keep open. Current main still contains the reported source-level failure paths: cache-inclusive token pressure can reach compaction, empty preparations can commit the structured fallback summary at high token counts, and reply handling still runs preflight compaction before the memory-flush gate that should protect the cycle.

## What This Changes

The issue reports a compaction reliability bug where Anthropic cache-read token accounting, empty fallback-summary handling, and memory-flush ordering can combine to discard real conversation context.

## Best Possible Solution

Land a coordinated reliability fix that separates trusted active-context estimates from provider billing/cache totals, blocks large empty-preparation fallback summaries, and guarantees memory flush cannot trail a destructive compaction for the same cycle.

## Reproduction Assessment

Yes, at source level. The issue provides concrete production traces, and current main still has the cache-inclusive currentTokenCount path, strict empty fallback path, and preflight-before-flush ordering; I did not run a live Opus/proxy replay.

## Solution Assessment

Partly. The requested direction is valid, but the safest fix should coordinate with #66716 and #74010 so OpenClaw still counts legitimate cached prompt context while excluding untrusted billing-style cacheRead inflation from destructive compaction decisions.
