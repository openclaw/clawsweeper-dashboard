## Summary

Keep open. Current main still has source-level paths for the reported compaction reliability cluster: cache-inclusive token pressure can reach context-engine compaction, large empty preparations still commit a structured fallback summary, and reply handling still runs preflight compaction before the memory flush that docs describe as pre-compaction.

## What This Changes

The issue asks OpenClaw to stop premature cache-driven compaction, avoid empty fallback summaries for large real sessions, and ensure memory flush runs before destructive compaction.

## Best Possible Solution

Land a coordinated reliability fix that uses trusted prompt/context snapshots for compaction pressure, blocks large empty-preparation fallback summaries, and guarantees memory flush cannot trail a destructive compaction for the same cycle.

## Reproduction Assessment

Yes, at source level. The issue includes concrete production traces, and current main still has the cache-inclusive currentTokenCount path, strict real-message empty fallback path, and preflight-before-flush ordering; I did not run a live Opus/proxy replay.

## Solution Assessment

Partly. The requested direction is right, but the safest fix should preserve legitimate cached prompt snapshots for #66520 while separating trusted active-context estimates from provider billing/cache totals and coordinating with #74010 reserve-token work.
