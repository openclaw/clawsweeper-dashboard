## Summary

Keep open. Current main still exposes the reported reliability cluster: cache-inclusive usage can feed context-engine compaction pressure, large empty preparations can still commit an empty fallback boundary, and the reply runner still evaluates preflight compaction before memory flush; the related #66716 work overlaps only the cached-token trigger area and is not a complete fix for this issue.

## What This Changes

The issue asks OpenClaw to prevent prompt-cache usage spikes from triggering destructive compaction, avoid empty fallback summaries for large real sessions, and coordinate memory flush before compaction.

## Best Possible Solution

Land a focused reliability fix that uses trustworthy prompt/envelope snapshots for compaction pressure, blocks destructive empty fallback boundaries for large sessions, and makes memory flush happen before any compaction that can discard the current cycle.

## Reproduction Assessment

Yes. The issue includes concrete production traces, and current main provides a source-level reproduction path through inflated lastCallUsage.cacheRead/cacheWrite, empty strict real-message filtering with high tokensBefore, and preflight-before-flush ordering.

## Solution Assessment

Partly. The requested direction is right, but globally dropping cacheRead would conflict with cached-context accounting; the safer fix is to prefer explicit prompt snapshots/envelope estimates and add guardrails for anomalous provider cache usage.
