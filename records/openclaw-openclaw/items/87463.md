## Summary

Keep open: the target bug is real, but this patch normalizes the shared runtime provider too early, so fallback, context, diagnostics, and auth-sensitive paths can see `openai` instead of the actual `openai-codex` provider before persistence.

## What This Changes

The branch maps `openai-codex` to `openai` in provider selection used by session-store, reply-runner, and followup-runner session usage persistence.

## Best Possible Solution

Keep raw runtime provider identity through execution/fallback/context/diagnostics, derive a separate persisted session-route provider only when the configured route is canonical `openai` with Codex runtime, and cover the session-store/reply/doctor paths.

## Reproduction Assessment

Yes. The linked report and current source show runtime `agentMeta.provider` feeds session route persistence, so `openai-codex` metadata can be re-written into session state; I did not run a live repro because this review is read-only.

## Solution Assessment

No. Normalizing the shared `providerUsed` variable is broader than the session persistence bug; the safer fix is a separate persisted-route provider resolved from the configured route/runtime.
