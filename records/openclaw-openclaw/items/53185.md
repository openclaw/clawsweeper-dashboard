## Summary

Keep open: current main still has external-content wrappers, partial web_fetch hidden-content cleanup, and a generic tool-result hook, but not the requested web-result sanitizer/classifier contract with structured threat_level and HIGH/CRITICAL blocking across both web_fetch and web_search. The closed generic hook issue at https://github.com/openclaw/openclaw/issues/46913 and the open umbrella at https://github.com/openclaw/openclaw/issues/54508 are useful context, but they do not fully replace this specific web pipeline hardening request.

## What This Changes

Request to route web_fetch and web_search results through a content sanitizer or preprocessing hook that strips hidden/HTML content, emits threat metadata, and blocks high-risk output.

## Best Possible Solution

Define a trusted web-result sanitizer/classifier contract for OpenClaw-owned web_fetch and web_search first, including SDK/plugin boundaries, threat metadata, fail-closed behavior, docs, and focused regression tests before accepting arbitrary script execution.

## Reproduction Assessment

Not applicable as a feature/security-hardening request. Source inspection on current main gives a high-confidence verification path: the web tool config and execution paths lack the requested sanitizer/classifier, threat_level, and high-risk blocking contract.

## Solution Assessment

No, directly wiring content-sanitizer.py or arbitrary configured scripts is not yet the best demonstrated solution. The safer direction is a reviewed typed sanitizer/classifier or trusted plugin contract with explicit trust, timeout, logging, and fail-closed semantics.
