## Summary

Keep open. Current main still has only partial `web_fetch` cleanup plus external-content wrapping; it does not provide the requested sanitizer/script preprocessing contract, structured `threat_level` output, or HIGH/CRITICAL block behavior across `web_fetch` and managed `web_search`. The reporter’s follow-up correctly reframes this as a trust-boundary/API question, and related #54508 remains the broader open security umbrella rather than a merged fix.

## What This Changes

The issue requests routing `web_fetch` and `web_search` results through a sanitizer or preprocessing path that strips or classifies malicious content, emits threat metadata, and blocks high-risk results.

## Best Possible Solution

Define a trusted web result sanitizer/classifier contract instead of wiring a private local script path directly. The end state should include explicit threat metadata, fail-closed HIGH/CRITICAL behavior, audit/logging semantics, config/schema and docs coverage, and regression tests for both `web_fetch` and managed `web_search`, with any extension point clearly limited to trusted code.
