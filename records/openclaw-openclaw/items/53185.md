## Summary

Keep open. Current main has useful external-content wrapping, special-token sanitization, and partial web_fetch hidden-content cleanup, but it still does not provide the requested content-sanitizer/script preprocessing path, structured threat_level output, or HIGH/CRITICAL block behavior across both web_fetch and web_search. The proposed hook would change a security trust boundary, so it needs maintainer security/product review rather than conservative cleanup closure.

## What This Changes

Request to route web_fetch and web_search results through a content-sanitizer or configurable preprocessing hook that can strip unsafe content, detect injection patterns, emit threat_level metadata, and block high-risk results.

## Best Possible Solution

Define a trusted web-result sanitizer/classifier contract before implementation. The end state should cover both managed web_fetch and managed web_search, expose explicit threat metadata, define fail-closed behavior for high-risk content, document plugin/script trust boundaries, and include focused regression tests; avoid wiring a private local script path directly into core without that policy.

## Reproduction Assessment

Not applicable as a feature/security hardening request. The current-main verification path is source inspection: web_fetch has partial cleanup and wrapping, while web_search lacks a shared sanitizer/classifier path and the config schema lacks preprocessing or threat-level fields.

## Solution Assessment

No, the direct content-sanitizer.py script-hook proposal is not yet demonstrably the best solution because it adds local code execution and output rewriting at a security boundary. A narrower, typed sanitizer/classifier or reviewed plugin contract with explicit failure semantics would be safer.
