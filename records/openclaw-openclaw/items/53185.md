## Summary

Keep open: current main has external-content wrapping, special-token sanitization, and partial `web_fetch` hidden-content cleanup, but it still lacks the requested sanitizer/preprocessing contract, structured `threat_level`, and HIGH/CRITICAL blocking behavior across both `web_fetch` and `web_search`.

## What This Changes

Route `web_fetch` and `web_search` results through a sanitizer or preprocessing path that strips hidden/encoded HTML attacks and emits threat metadata or blocking decisions.

## Best Possible Solution

Define a trusted web-result sanitizer/classifier contract covering both managed `web_fetch` and `web_search`, with explicit threat metadata, fail-closed behavior for high-risk content, documentation, and focused regression tests before wiring any script execution path.

## Reproduction Assessment

Not applicable as a feature/security-hardening request. The current-main verification path is source inspection: web_fetch has partial cleanup and wrapping, while web_search and config lack a shared sanitizer/classifier or threat-level contract.

## Solution Assessment

No. Directly wiring `content-sanitizer.py` or arbitrary configured scripts into core is not yet the best demonstrated solution because it changes a security trust boundary; a typed sanitizer/classifier or reviewed plugin contract should come first.
