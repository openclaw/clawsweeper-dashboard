## Best Possible Solution

Use https://github.com/openclaw/openclaw/issues/79168 as the canonical scanner/blocking contract, with https://github.com/openclaw/openclaw/issues/62123 and https://github.com/openclaw/openclaw/issues/48486 carrying the hook API and web_fetch sanitizer slices.

## Reproduction Assessment

Not applicable as a security-hardening feature request. Source inspection gives a high-confidence gap check: current main lacks a web-result threat_level/classifier/blocking contract even though it has partial wrapping and hidden-content cleanup.

## Solution Assessment

No, directly wiring content-sanitizer.py or arbitrary configured scripts is not the best standalone fix. The safer path is the canonical tool-output scanner and hook-contract design now tracked in the linked open issues, with explicit trust, timeout, failure, and blocking semantics.
