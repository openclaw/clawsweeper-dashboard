## Summary

Keep open: the reporter narrowed this from runtime auth failure to a Telegram `/status` display bug, and current source still has a plausible mismatch where the native `/status` fast path can render auth for the default selected provider instead of the session's custom provider.

## What This Changes

The issue reports that Telegram `/status` can display the wrong auth source for a selected custom provider even though actual routing/key use is correct.

## Best Possible Solution

Make Telegram `/status` derive the displayed provider/auth source from the same selected session/runtime provider used by `session_status`, while preserving runtime credential precedence.

## Reproduction Assessment

No live reproduction was run, but there is a source-reproducible path: Telegram/native `/status` passes default provider/model into `buildStatusReply` while the target session entry may contain the custom provider selection.

## Solution Assessment

Yes: the narrow fix is to align the status display provider selection with the session's effective model provider, not to change model routing or provider credential resolution.
