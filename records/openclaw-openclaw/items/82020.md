## Summary

Keep open: the reporter narrowed this from a runtime auth regression to a Telegram `/status` display bug, and current source still has a distinct Telegram status path that can diverge from the `session_status` tool's custom-provider auth label handling.

## What This Changes

Telegram `/status` should show a custom provider's configured `models.json` auth source instead of an unrelated built-in or env auth label.

## Best Possible Solution

Unify Telegram `/status` auth-source display with the `session_status` tool by carrying the selected custom provider identity or an explicit `modelAuthOverride` into `buildStatusText`, with focused regression coverage for a custom provider backed by `models.json`.

## Reproduction Assessment

No high-confidence live reproduction was run, but source inspection gives a credible current-main reproduction path: native Telegram `/status` builds its own status card while `session_status` has separate custom-provider auth-label handling. A focused test can reproduce the mismatch without real Z.AI credentials.

## Solution Assessment

Yes, a narrow status-display fix is the best path now that the reporter confirmed runtime routing works. The safer alternative to changing auth resolution is to make the Telegram status card use the same selected-provider/auth-label logic already proven by `session_status`.
