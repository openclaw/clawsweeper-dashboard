## Summary

Keep open: the latest reporter clarification narrows this to a valid Telegram status display bug, and current main can still resolve the auth label from the default provider while rendering the model from the session override. The runtime credential path should not be broadly changed for this report.

## What This Changes

The issue asks for Telegram `/status` to display the actual custom-provider auth source from `models.json` instead of a built-in provider env key after selecting a custom provider.

## Best Possible Solution

Make `/status` resolve its auth label from the same selected provider/model that it renders, add a regression for session `providerOverride`, and leave runtime credential precedence unchanged.

## Reproduction Assessment

Yes, at source level: a session with `providerOverride: "zai-light"` can render the `zai-light` model while `buildStatusText` resolves the selected auth label from the default provider passed by the `/status` command path. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to align status auth-label provider selection with the selected session model, not to change API routing or credential precedence.
