## Summary

Keep open: the report is fresh, concrete, and not safe to close as implemented or as a clean duplicate. Current main still has auth paths where profile/env resolution can precede the selected provider entry's configured `apiKey`, and the related open split-provider auth work does not clearly cover the reporter's broader “all custom providers” clarification.

## What This Changes

The issue reports that custom providers, including a `zai-light` provider sharing ZAI's base URL, resolve to the wrong API key instead of their configured provider key.

## Best Possible Solution

Add a focused regression for custom-provider auth selection, including `zai` plus `zai-light` with the same base URL and an unrelated env key present, then patch the selected provider/auth path so the configured provider id and key survive Telegram selection, model resolution, status labeling, and runtime auth.
