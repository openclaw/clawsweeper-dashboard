## What This Changes

The issue requests that custom provider status/auth labels report the provider entry's own `models.json` API key instead of a built-in or env key when providers share the same upstream base URL.

## Best Possible Solution

Land the linked PR or an equivalent focused fix that makes `resolveModelAuthLabel` honor literal provider-entry API keys before env fallback and keeps regression coverage for the shared-base-url custom provider case.

## Reproduction Assessment

Yes. Source inspection shows current main's label helper checks env before literal per-entry custom provider keys, while runtime auth can resolve the provider-entry key first; I did not run the Telegram flow in this read-only review.
