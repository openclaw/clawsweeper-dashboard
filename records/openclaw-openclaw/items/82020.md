Keep open: the reporter narrowed this to a real Telegram `/status` display bug, and current `main` still has a plausible source path where the native slash-command fast path can build status from the default provider/model before stored model overrides are applied.

Telegram `/status` should show the selected custom provider's configured `models.json` API-key source instead of an unrelated env key while actual API routing remains correct.

## Best Possible Solution

Make native Telegram `/status` resolve the same selected provider/model as the normal reply path, then add a focused regression proving a custom provider override displays `api-key (models.json)` without changing runtime credential routing.

The reporter's latest comment confirms the actual API call is correct and only the status display is wrong.

## Solution Assessment

Yes, the narrowest maintainable fix is to align native status provider/model resolution with the normal reply path or shared stored-override helper. It should not change provider alias policy, config format, or actual API-key resolution.
