## Summary

Keep open: the schema-compatibility direction is useful, but the PR still weakens JSON Schema validation by dropping sibling constraints when it resolves a local `$ref`. The protected `maintainer` label also means this should remain in explicit maintainer handling rather than cleanup close.

## What This Changes

The PR inlines local `$ref` parameter schemas for bundled MCP tools, adds provider-neutral normalizer coverage, adds bundled MCP materialization coverage, and records a changelog fix for the linked Notion-style `oneOf` case.

## Best Possible Solution

Keep the shared bundled MCP schema-normalization approach, but resolve local refs in a JSON Schema-preserving way that retains adjacent validation siblings while keeping the existing Notion-style coverage.

## Reproduction Assessment

No for the original current-main Notion pre-dispatch rejection: maintainer live validation reached Notion on both main and PR head. Yes for the PR regression: source inspection of the PR resolver shows adjacent validation siblings are dropped after a successful local `$ref` resolution.

## Solution Assessment

No. Normalizing local refs at the shared tool-schema boundary is the right shape, but the resolver must preserve or compose adjacent sibling validation keywords to keep JSON Schema semantics intact.
