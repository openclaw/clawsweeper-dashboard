## Summary

Keep open: the compatibility direction and proof are useful, but the PR still weakens valid JSON Schema 2020-12 tool schemas by dropping validation keywords adjacent to resolved local `$ref` entries and by turning cyclic local refs into permissive `{}` schemas.

## What This Changes

The PR inlines local JSON Pointer `$ref` schemas during shared tool-parameter normalization, applies that normalization to bundled MCP tool materialization, adds regression coverage, and updates the changelog.

## Best Possible Solution

Keep the shared normalizer and bundled MCP materialization boundary, but merge only after the resolver preserves valid `$ref` sibling constraints and handles recursive local refs without weakening validation.

## Reproduction Assessment

No for the original Notion pre-dispatch failure: maintainer live validation reached Notion on current main and PR head. Yes by source inspection for the PR regression: resolved `$ref` schemas with validation siblings and recursive local refs are weakened by the proposed resolver.

## Solution Assessment

No. Normalizing local refs at the shared tool-schema boundary is the right shape, but the implementation must preserve JSON Schema 2020-12 sibling constraints and leave recursive refs safe instead of replacing them with permissive schemas.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve $ref sibling constraints:** `src/agents/pi-tools-parameter-schema.ts:446-448`
  - body: When a local `$ref` resolves, this branch returns the referenced schema and only copies `title`, `description`, and `default`. JSON Schema 2020-12 allows validation keywords beside `$ref`, so `{ $ref: "#/$defs/Id", minLength: 8 }` becomes a plain string schema and accepts too-short values; preserve those siblings, for example with `allOf`, and add a regression.
  - confidence: 0.91
- **[P2] Keep cyclic local refs from becoming permissive schemas:** `src/agents/pi-tools-parameter-schema.ts:430-431`
  - body: A recursive local `$ref` that is already in `refStack` returns `{}` and the parent walk then drops the definition block, so that recursive position validates as anything. Keep the ref plus definitions unresolved, or otherwise represent the cycle without weakening the schema.
  - confidence: 0.82
