## Summary

Keep open with one blocking correctness finding. The PR is aimed at the right bundled MCP schema-normalization boundary and has convincing real behavior proof for the Notion-shaped case, but the new resolver drops non-annotation sibling keywords next to resolved `$ref`s, which can change JSON Schema validation for MCP tools.

## What This Changes

The branch inlines local `$defs`/`definitions` `$ref`s during tool-parameter normalization, applies that to bundled MCP tool materialization, adds Notion-style regression coverage, and updates the changelog.

## Best Possible Solution

Keep the PR open, preserve JSON Schema `$ref` sibling semantics in the resolver, then rerun focused schema/materialization tests and let maintainers decide the final issue framing.

## Reproduction Assessment

No for the original Notion failure on current main: maintainer live validation reached Notion instead of reproducing pre-dispatch rejection. The PR’s schema-sibling regression is source-reproducible from the diff and the JSON Schema/MCP contracts.

## Solution Assessment

No. Normalizing local refs at the shared tool-schema boundary is the right shape, but the resolver must preserve or compose non-`$ref` sibling keywords instead of copying only annotations.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve constraints beside resolved $ref:** `src/agents/pi-tools-parameter-schema.ts:361-363`
  - body: When a local `$ref` resolves, this branch replaces the referring schema with the referenced schema and only copies `title`, `description`, and `default`. MCP tool schemas are JSON Schema 2020-12, where validation keywords beside `$ref` still apply; dropping siblings such as `minLength`, `pattern`, `required`, or `additionalProperties` silently loosens or changes validation before the MCP call. Merge those siblings into the resolved schema, or compose with `allOf`, and add a regression for a `$ref` with a validation sibling.
  - confidence: 0.87
