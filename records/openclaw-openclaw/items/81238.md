## Summary

Keep open: the PR is a plausible bundled MCP schema-compatibility fix and has sufficient real behavior proof, but the current head still changes JSON Schema semantics by dropping validation keywords adjacent to an inlined local `$ref`.

## What This Changes

The PR inlines local `$ref` schemas during tool parameter normalization, applies that normalization to bundled MCP tool materialization, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the shared normalizer approach, but preserve or compose `$ref` sibling validation keywords while retaining the bundled MCP materialization regression coverage.

## Reproduction Assessment

No for the original current-main Notion rejection: maintainer live validation reached Notion on both main and PR head. Yes for the PR regression: source inspection shows resolved `$ref` schemas copy only the referenced schema plus annotations, dropping adjacent validation siblings.

## Solution Assessment

No. Normalizing local refs at the shared tool-schema boundary is the right shape, but the resolver must preserve or compose adjacent sibling keywords to keep JSON Schema semantics intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve validation siblings when resolving $ref:** `src/agents/pi-tools-parameter-schema.ts:361-363`
  - body: When a local `$ref` resolves, this branch replaces the referring schema with the referenced schema and only copies `title`, `description`, and `default`. JSON Schema allows validation keywords beside `$ref`, so a schema like `{ $ref: "#/$defs/Id", minLength: 8 }` loses the `minLength` constraint before tool validation; merge those siblings or compose with `allOf`, and add a regression.
  - confidence: 0.88
