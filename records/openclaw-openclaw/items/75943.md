## Summary

Keep open. The PR is protected by the `maintainer` label and the latest head still routes Discord subagent spawn policy through the shared resolver without first resolving Discord's configured default account, which can bypass account-scoped thread-spawn controls.

## What This Changes

The PR consolidates split thread-bound subagent and ACP spawn flags into `threadBindings.spawnSessions`, changes default thread-bound spawn/context behavior, shares parent fork policy, and updates schemas, docs, generated metadata, and tests.

## Best Possible Solution

Keep the consolidation only after Discord resolves the effective account before shared policy lookup, with regression coverage for omitted requester account ids plus `channels.discord.defaultAccount`.

## Reproduction Assessment

Yes. A focused reproduction is `handleDiscordSubagentSpawning` with `requester.channel: "discord"`, no requester account id, `channels.discord.defaultAccount: "work"`, and `accounts.work.threadBindings.spawnSessions: false`; current main checks `work`, while the PR checks `default`.

## Solution Assessment

No. The consolidation direction is plausible, but this implementation is not the best fix until the shared policy call receives the effective Discord account or the shared resolver learns channel `defaultAccount` semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve default Discord account policy:** `extensions/discord/src/subagent-hooks.ts:97`
  - body: The shared policy is now called with `event.requester?.accountId`; when Discord omits that id, the resolver checks literal `default` instead of `channels.discord.defaultAccount`. In a multi-account config with `defaultAccount: "work"` and `accounts.work.threadBindings.spawnSessions: false`, this can still allow thread-bound subagent spawning, so resolve the effective Discord account before checking policy.
  - confidence: 0.86
