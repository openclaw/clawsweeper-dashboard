## Summary

Keep open: current main still has the interactive confirmation bypass, and the linked closing PR is open rather than merged. The issue remains a narrow, source-proven safety bug on the secrets migration path.

## What This Changes

The issue asks for `openclaw secrets configure` to show the irreversible migration warning whenever apply proceeds interactively, not only when `--apply` was passed.

## Best Possible Solution

Land or replace the linked focused PR so the irreversible prompt is based on actual write intent after the interactive apply question, with regression coverage for the interactive yes path and unchanged `--yes`/`--json` behavior.

## Reproduction Assessment

Yes, at source level: mock the first interactive `confirm` in `secrets configure` to return true with no `--apply`, and current main reaches `runSecretsApply` while `needsIrreversiblePrompt` remains false. I did not run the interactive CLI because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to derive the irreversible prompt from `shouldApply` after interactive confirmation while preserving the existing `--yes` and `--json` bypasses.
