## Summary

Keep open: current main still ties the irreversible confirmation to `opts.apply`, so an interactive user who accepts `Apply this plan now?` reaches write mode without the one-way migration warning. The docs describe the apply path as one-way, and the existing CLI test covers interactive apply without asserting the second confirmation.

## What This Changes

The issue requests that `openclaw secrets configure` show the one-way migration confirmation when a user accepts the interactive apply prompt, not only when `--apply` was passed.

## Best Possible Solution

Derive the irreversible confirmation from the actual write intent after the interactive prompt, still respecting `--yes` and `--json`, and add a focused regression test that requires the second confirmation before apply.

## Reproduction Assessment

Yes, at source level: mock the first `confirm` in `secrets configure` to return true with no `--apply`, and current code will call `runSecretsApply` without a second irreversible confirmation. I did not execute the interactive CLI in this read-only review.

## Solution Assessment

Yes, the narrow fix is to base the irreversible prompt on `shouldApply` after all confirmations instead of on how apply was requested. That preserves the existing CLI contract while closing the bypass.
