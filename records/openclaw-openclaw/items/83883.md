## Label Justifications

- `P2`: This is a normal-priority CLI safety bug with a narrow secrets-migration surface and clear source-level evidence.
- `impact:data-loss`: The skipped warning precedes a one-way scrub of plaintext credential/config values without rollback in the configure apply path.
- `impact:security`: The affected command migrates credential material and SecretRef configuration, so the missing warning sits on a sensitive-data path.

## Summary

Keep open: current main and v2026.5.18 still derive the irreversible confirmation from `opts.apply`, so an interactive user who answers yes to `Apply this plan now?` can reach apply without the one-way warning. The open fixing PR https://github.com/openclaw/openclaw/pull/84225 references this issue with closing syntax, so the issue should wait for that PR to land or be replaced.

## What This Changes

Ensure `secrets configure` shows the irreversible migration warning when the user chooses apply from the interactive prompt.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/84225 or an equivalent fix that derives the irreversible prompt from final write intent, with focused regression coverage and real CLI proof before closing this issue.

## Reproduction Assessment

Yes, at source level: mock the first `confirm` in `secrets configure` to return true with no `--apply`, and current code reaches `runSecretsApply` without a second irreversible confirmation. I did not execute the interactive CLI in this read-only review.

## Solution Assessment

Yes: deriving the warning from final `shouldApply` state after all confirmations is the narrow maintainable fix. The open PR follows that shape and should be reviewed for proof, tests, and docs alignment before landing.
