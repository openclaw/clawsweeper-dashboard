## Label Justifications

- `P1`: The issue reports recurring Telegram inbound stalls where users receive no agent response until gateway restart or manual database repair.
- `impact:session-state`: The failure is a durable mismatch between session diagnostics, command lane state, and SQLite ingress claim ownership.
- `impact:message-loss`: Inbound Telegram messages can be accepted into OpenClaw state but remain pending or claimed without reaching the agent.
- `impact:crash-loop`: The practical recovery path is process restart or manual SQLite cleanup, so the gateway can appear live while core channel availability is broken.

## Summary

Keep open: current main still preserves fresh Telegram ingress claims owned by another live process, while `release_lane` only resets the in-process command lane and can report `released=0` without touching `channel_ingress_events`. This issue remains necessary until a linked fix PR lands or maintainers choose the safe live-owner recovery policy.

## What This Changes

Recover Telegram ingress claims stuck in `claimed` when a live worker stops progressing, so later inbound messages drain without a gateway restart.

## Best Possible Solution

Land one focused Telegram ingress recovery fix that token-guards non-progressing live-owner claims, preserves active handler work, defines fail/retry semantics, and includes queue regression tests plus real Telegram proof.

## Reproduction Assessment

Yes at source level: current tests show a fresh other-live-process Telegram claim remains claimed and blocks the later pending update, and the issue supplies matching logs and SQLite rows. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear. The requested end state is right, but maintainers still need to choose whether timed-out live-owned claims should be failed, released for retry, or recovered through diagnostics because each option has delivery and duplicate-processing tradeoffs.
