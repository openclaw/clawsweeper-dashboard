## Label Justifications

- `P1`: The issue reports recurring Telegram inbound stalls where users receive no agent response until restart or manual SQLite repair.
- `impact:session-state`: The failure is a durable mismatch between diagnostic session state, command-lane state, and SQLite ingress claim ownership.
- `impact:message-loss`: Inbound Telegram messages can be accepted into OpenClaw state but remain pending or claimed without reaching the agent.
- `impact:crash-loop`: The practical recovery is gateway restart or manual database cleanup, so this affects process-level availability even when the process appears healthy.

## Summary

Keep open: current main and v2026.6.9 still have the reported failure shape, because diagnostic `release_lane` resets command lanes rather than durable Telegram ingress rows, while Telegram recovery still skips fresh claims owned by another live process. The direct closing PRs https://github.com/openclaw/openclaw/pull/95282 and https://github.com/openclaw/openclaw/pull/95299 are open and unmerged, so this issue should remain the canonical tracker.

## What This Changes

The issue asks OpenClaw to recover live-owned, non-progressing Telegram ingress claims so stuck `channel_ingress_events` rows do not block later agent responses until gateway restart.

## Best Possible Solution

Review and land one safe linked PR, or an equivalent fix, that token-guards live-owned Telegram claim recovery, records an explicit failure/retry reason, and proves both durable queue behavior and Telegram-visible recovery.

## Reproduction Assessment

Yes at source level. Current main can report `release_lane released=0` without touching `channel_ingress_events`, while Telegram recovery skips fresh claims owned by another live process; I did not run the live Telegram reproduction.

## Solution Assessment

Unclear. The requested durable-claim recovery target is correct, but the best implementation must settle live-owner timeout, fail-versus-release, and duplicate-processing policy rather than blindly clearing any live PID claim.
