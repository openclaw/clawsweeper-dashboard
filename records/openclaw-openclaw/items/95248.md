## Summary

The issue should stay open: current main still has the reported stuck-lane failure mode because diagnostic recovery calls the current-worker claim release path, which intentionally skips claims owned by a live worker, so `released=0` is expected for the user’s observed live-but-stalled Telegram ingress claims. The report is concrete, recent, and tied to a high-impact message-delivery stall that current source and tests do not show as already fixed.

## What This Changes

The issue requests that stuck Telegram channel ingress lane claims held by non-progressing live workers be force-released or failed so inbound messages do not block until gateway restart.

## Best Possible Solution

Add a bounded diagnostic recovery path that distinguishes actively progressing workers from stalled live claims, safely fails or releases only expired stuck ingress events, and covers Telegram ingress with regression tests for active-worker preservation and stuck-lane recovery.

## Reproduction Assessment

Yes from source, but not from a live run in this read-only review: the diagnostic path calls current-worker lane release, and that release path returns zero for claims owned by a live worker, matching the reported `released=0` loop. The user also provided repeated production steps and database state, but a Telegram live reproduction still belongs in the fix proof.

## Solution Assessment

The reporter’s proposed force-release direction is plausible, but the best fix should be narrower than unconditionally clearing live-worker claims: preserve active work, define a stalled-claim threshold, mark only expired stuck events failed with a clear reason, and add coverage around duplicate/lost delivery behavior.
