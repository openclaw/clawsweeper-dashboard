## Label Justifications

- `P1`: The issue reports a recurring Telegram inbound workflow stall where users receive no agent responses until operator restart or manual database repair.
- `impact:session-state`: The failure is a durable mismatch between diagnostic session state, command lane state, and SQLite ingress claim ownership.
- `impact:message-loss`: Inbound Telegram messages can be accepted into OpenClaw state but remain pending or claimed without reaching the agent.
- `impact:crash-loop`: The practical recovery is gateway restart or manual SQLite cleanup, so this affects process-level availability even when the process appears healthy.

## Summary

Keep open: current main still has the reported failure shape for live-worker Telegram ingress claims. `release_lane` only resets the in-process command lane and can report `released=0`, while Telegram ingress recovery deliberately skips claims owned by another live process, so the requested live-worker/non-progressing claim recovery is not implemented by main or by the latest release.

## What This Changes

Requests that diagnostic `release_lane` actually unblock Telegram SQLite ingress claims held by non-responsive live workers instead of logging `released=0` and requiring a gateway restart.

## Best Possible Solution

Add a core-owned recovery path that ties diagnostic stuck-session state to durable ingress claims and safely reclaims non-progressing live-worker claims with token/race safeguards, explicit policy, focused tests, and Telegram proof.

## Reproduction Assessment

Yes. Source inspection shows `release_lane` can report `released=0` without touching `channel_ingress_events`, while Telegram recovery skips fresh live-owner claims; the issue adds concrete logs and database rows, though I did not run a live Telegram repro.

## Solution Assessment

Unclear. The requested end state is correct, but the exact live-worker force-release rule needs maintainer policy around duplicate processing, failure marking, periodic recovery, and coordination with existing stale-ingress work.
