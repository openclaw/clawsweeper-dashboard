## Summary

Keep open: current main still has a source-visible path where stuck-session recovery can report `release_lane` with `released=0`, and the live issue discussion adds fresh production evidence for Telegram ingress claims held by a live but non-progressing worker.

## What This Changes

The issue reports that Telegram inbound events can remain claimed or pending behind a live-but-stalled worker while diagnostic recovery logs `release_lane released=0` until the gateway is restarted.

## Best Possible Solution

Add or extend a bounded ingress recovery path that distinguishes active progress from stalled live claims, clears only expired stuck events with race-safe claim-token checks, and preserves active-worker delivery semantics.

## Reproduction Assessment

Yes, source-reproducible at medium confidence: current main can produce `release_lane released=0`, and Telegram recovery preserves live-owner claims. I did not run the live Telegram/database reproduction from the report.

## Solution Assessment

No, not as an unconditional force-release: the best fix needs a stalled-owner threshold and race-safe failure/release behavior so genuinely active handlers are not duplicated or dropped.
