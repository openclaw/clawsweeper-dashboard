## Label Justifications

- `P1`: The issue reports a recurring Telegram inbound workflow stall where users receive no agent responses until restart or manual SQLite repair.
- `impact:session-state`: The failure is a durable mismatch between diagnostic session state, command lane state, and SQLite ingress claim ownership.
- `impact:message-loss`: Inbound Telegram messages can be accepted into OpenClaw state but remain pending or claimed without reaching the agent.
- `impact:crash-loop`: Gateway restart or manual database cleanup is the reported practical recovery, so the issue affects process-level availability even when the process appears healthy.

## Summary

Keep open: current main and v2026.6.9 have useful Telegram spooled-handler timeout recovery, but they still do not implement the requested live-owned claim recovery when a claim is held by another live worker without active handler state; two open closing PRs now own that remaining policy choice.

## What This Changes

The issue requests recovery for Telegram `channel_ingress_events` claims that stay live-owned and block later inbound messages while diagnostic `release_lane` reports `released=0`.

## Best Possible Solution

Land or revise one narrow Telegram ingress recovery PR that token-guard fails timed-out live-owned claims without active handler state, while preserving deferred and truly active claim guards and adding Telegram-visible proof.

## Reproduction Assessment

Yes at source level: current main still skips other-live Telegram spooled claims and diagnostic `release_lane` does not touch `channel_ingress_events`, matching the reported `released=0` durable-claim gap. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear as a final product decision: the requested recovery is the right bug surface, but maintainers still need to choose the live-owner timeout/failure policy and consolidate the open PRs. The current shipped handler-timeout recovery is only a partial mitigation for active in-process handlers.
