## Summary

Current main has moved the expensive runtime plugin load out of the pre-bind `plugins.bootstrap` path, so the reported 2026.4.26 signature where the Gateway never opens port 18789 before `ready` is addressed on main. The fix is not proven shipped in the latest local release tag, so this should close as implemented on main rather than shipped in a release.

## What This Changes

The issue reports a deterministic 2026.4.26 Docker Gateway startup hang after `control-ui.seed`, before `plugins.bootstrap` completes and before the HTTP listener opens.

## Best Possible Solution

Keep the current-main startup sequence that binds HTTP before plugin runtime loading, and let any broader Windows Docker dispatch symptoms continue under their separate tracker.

## Reproduction Assessment

Yes for the historical bug: the issue gives a deterministic Docker recreate on 2026.4.26 with startup trace stopping after `control-ui.seed` and no port 18789 listener. I did not rerun Docker in this read-only review, but the current-main source path and reporter's rebuilt-main result match the fix.

## Solution Assessment

Yes. Moving runtime plugin loading and dependency staging out of the pre-bind bootstrap path is the narrow maintainable fix for this hang signature because it preserves plugin startup behavior while preventing plugin runtime work from blocking the HTTP listener.
