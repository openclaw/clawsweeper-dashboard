## Summary

Current main resolves the reported 2026.4.26 `plugins.bootstrap` startup hang signature: plugin runtime loading is no longer done inside the pre-listen bootstrap span, startup metadata is reused instead of rescanning manifests, and the reporter’s own current-main rebuild reached `ready`.

## What This Changes

The issue asks OpenClaw to account for and fix a v2026.4.26 Docker gateway hang during `plugins.bootstrap` before `ready` or `/healthz` are reachable.

## Best Possible Solution

Keep the current staged startup design and ask for a new current-main trace only if a pre-listen `plugins.bootstrap` hang still reproduces after the next release/image update.

## Reproduction Assessment

Yes for the affected release: the issue includes deterministic Docker recreate steps, startup trace ending after `control-ui.seed`, absent port 18789, and current-main rebuild verification. I did not rerun Docker locally because this review was read-only.

## Solution Assessment

Yes; moving runtime plugin loading to the post-listen phase and reusing metadata snapshots is narrower and safer than adding broad startup timeouts to every maintenance hook for this specific release-band hang.
