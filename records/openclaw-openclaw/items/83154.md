## Summary

Keep open: this is a focused Doctor cleanup PR with sufficient supplied real-output proof, but the proposed scan still misses stale CLI runtime pins when the user stays on the same provider plugin and moves back to PI or another non-CLI runtime. No security or supply-chain concern was found in the diff.

## What This Changes

The PR updates Doctor session-route cleanup so stale plugin-owned runtime override pins for Anthropic and Google CLI runtimes can be detected and cleared, with focused contract and repair tests.

## Best Possible Solution

Keep the plugin-owned Doctor metadata, but make pinned-runtime cleanup compare stored runtime ids against the current route runtime separately from provider ownership, while preserving explicit model choices and configured CLI runtime routes.

## Reproduction Assessment

Yes. Current main source shows agentRuntimeOverride-only rows are skipped before plugin route-state scanning, and the existing pinned-runtime repair only clears agentHarnessId; I did not run a local repro because this review is read-only.

## Solution Assessment

No. The proposed direction is right, but the implementation should evaluate stale pinned runtimes by runtime id even when the same provider plugin remains configured through a canonical provider/model route.

Please compare the stored pinned runtime to the current route runtime separately and add coverage for the same-provider canonical-route case.
