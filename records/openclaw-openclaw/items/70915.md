## Summary

Keep open. The PR targets the right ZAI vision-model 404, but the request-scoped config rewrite can be defeated by the existing `models.json` merge path, so the patch is not safe to merge as-is; it also still lacks the required changelog entry.

## What This Changes

The PR wraps the ZAI media-understanding provider to rewrite known ZAI vision model coding base URLs to standard base URLs and adds provider-local tests.

## Best Possible Solution

Implement ZAI vision endpoint selection at the transport/model-resolution layer so existing `models.json` coding endpoints cannot win, add a regression for that case, and include the Unreleased changelog entry.

## Reproduction Assessment

Yes. Configure `agents.defaults.imageModel` as `zai/glm-4.6v` with ZAI on a coding base URL and an existing `models.json` containing that coding URL; the current code path preserves that base URL before model discovery, matching the reported 404 route.

## Solution Assessment

No. The PR's cloned-config approach is not the best fix because the generic runtime immediately re-enters the shared models.json merge path; a durable fix should bypass or explicitly override that persisted provider base URL for ZAI vision requests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Route vision requests outside the models.json merge path:** `extensions/zai/media-understanding-provider.ts:77`
  - body: The patched config is still passed into `describeImageWithModel`, which calls `ensureOpenClawModelsJson` before discovery. In default merge mode, an existing `models.json` ZAI provider entry preserves its non-empty coding `baseUrl`, so a normal gateway that already wrote the coding endpoint can still send this vision request to `/api/coding/paas/v4` and reproduce the 404. The fix needs to apply the standard endpoint after model resolution or use a runtime seam that bypasses persisted baseUrl preservation.
  - confidence: 0.88
