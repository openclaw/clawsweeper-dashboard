## Summary

Keep open: current main still lacks a commitments-specific model override, and adding one would be a new public config/model-routing feature. The request is narrow and supported by adjacent precedent, but it needs maintainer approval for the config contract and fallback semantics.

## What This Changes

The issue requests optional `commitments.model` and possibly `modelFallback` config so hidden commitment extraction can use a different model than the main agent.

## Best Possible Solution

Approve a clear commitments model-routing contract, then implement it across type/schema/help/docs and the commitments resolver with focused tests for override precedence, default inheritance, and any chosen fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source inspection gives high confidence that current main has no `commitments.model` key and that hidden extraction currently inherits the agent/default model through `resolveCommitmentDefaultModelRef`.

## Solution Assessment

Unclear until maintainers choose the public contract. A per-feature override matches active-memory and compaction precedent, but `modelFallback`, fast-mode interaction, and fallback-chain semantics need product approval before implementation.
