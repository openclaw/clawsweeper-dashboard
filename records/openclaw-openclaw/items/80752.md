## Summary

Keep open: current main and the latest release still expose only `commitments.enabled` and `commitments.maxPerDay`, and commitment extraction still resolves through the agent default model rather than a commitments-specific override. The request is useful and concrete, but it adds a public config/model-routing surface, so maintainers should settle the override and fallback contract before implementation.

## What This Changes

Request an optional commitments-specific model override so hidden commitment extraction can use a different model than the main agent.

## Best Possible Solution

Add a narrow documented commitments extraction model override only after maintainers confirm the public config name, model-ref normalization, and fallback semantics.

## Reproduction Assessment

Yes for the missing capability: current source, schema, and docs show no `commitments.model` surface, and the resolver always inherits the agent default. I did not run a live saturation reproduction.

## Solution Assessment

Unclear until maintainers approve the public contract. The requested direction is narrow and matches adjacent precedent, but the implementation should reuse existing model-ref parsing/normalization rather than the issue’s raw string split and should define fallback behavior explicitly.
