## Summary

Keep open: current main does not expose a commitments-specific model override, and adding one would be a new public config/model-routing feature rather than cleanup of an already-fixed bug. The request is narrow and well-grounded by adjacent precedent, but it needs maintainer approval for the config contract and fallback semantics.

## What This Changes

The issue requests optional `commitments.model` and possibly `commitments.modelFallback` settings so hidden commitment extraction can run on a different model than the agent default.

## Best Possible Solution

Decide the public commitments model-routing contract, then implement it in the commitments resolver with schema/help/docs updates and focused tests for override precedence plus default inheritance.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source inspection gives high confidence that current main has no `commitments.model` key and that hidden extraction inherits the agent default through `resolveCommitmentDefaultModelRef`.

## Solution Assessment

Unclear until maintainers choose the config contract. A narrow resolver/schema/docs/test change looks maintainable, but `modelFallback` semantics and interaction with existing fast-mode/fallback policy need explicit product approval first.
