## Summary

Keep open: the PR addresses a real Volcengine TTS fake-IP/TUN use case that current main and v2026.6.6 still do not support, but the security-sensitive config surface is still named/commented as a private-network/RFC1918 bypass while the implementation only opens fake-IP ranges, and the posted proof does not show a successful after-fix TTS call.

## What This Changes

The PR adds a Volcengine TTS provider config flag that conditionally passes RFC2544 and IPv6 ULA SSRF exemptions to both Seed Speech and legacy TTS paths, with unit tests for policy forwarding.

## Best Possible Solution

Expose a fake-IP/TUN-specific Volcengine TTS opt-in with accurate docs and tests, keep the default SSRF policy closed, and require redacted after-fix proof showing a successful TTS call.

## Reproduction Assessment

Yes from source, but not from a live TUN setup in this review. Current main and v2026.6.6 use only a hostname allowlist for Volcengine TTS, while the shared SSRF/net-policy code blocks RFC2544 and IPv6 ULA ranges unless the narrow opt-in bits are present.

## Solution Assessment

No, not as submitted. The narrow SSRF bits are the right implementation direction, but the public config contract and docs should say fake-IP/TUN rather than private network/RFC1918, and the PR still needs after-fix real behavior proof.
