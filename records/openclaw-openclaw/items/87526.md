## Summary

Keep open. The PR addresses a real Volcengine TTS fake-IP/TUN gap that is still present on current main and v2026.6.6, but it should not merge until the security-sensitive config contract matches the narrow fake-IP behavior and after-fix real TTS proof is supplied.

## What This Changes

The PR adds a Volcengine TTS `allowPrivateNetwork` config flag, forwards it through the speech provider, and uses it to add RFC2544 and IPv6 ULA SSRF policy exemptions in both TTS request paths with unit coverage.

## Best Possible Solution

Expose a fake-IP/TUN-specific Volcengine TTS opt-in with accurate docs and tests, keep default SSRF behavior closed, and require redacted after-fix proof showing a successful TTS call.

## Reproduction Assessment

Yes from source, but not from a live TUN setup in this review. Current main and v2026.6.6 use only a hostname allowlist for Volcengine TTS, while the shared SSRF/net-policy code blocks RFC2544 and IPv6 ULA ranges unless the narrow opt-in bits are present.

## Solution Assessment

No, not as submitted. The narrow SSRF bits are the right implementation direction, but the public config contract and docs should say fake-IP/TUN rather than private network/RFC1918, and the PR still needs after-fix real behavior proof.
