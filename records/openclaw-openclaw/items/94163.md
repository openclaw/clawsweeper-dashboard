## Summary

Keep open: this PR is the stronger linked candidate for the canonical skill snapshot bug, but it still needs contributor proof from a real OpenClaw path and a focused upgrade-path test for old persisted snapshots missing `skills[].skillKey`.

## What This Changes

The branch adds optional `skillKey` metadata to skill/session snapshots, writes it during workspace snapshot building, uses it for snapshot env override lookup, adds regression tests, and forces reusable snapshots without `skillKey` to refresh.

## Best Possible Solution

Land one narrow linked fix after it proves old persisted snapshots missing `skillKey` refresh safely, preserves new-snapshot lookup by resolved key, and includes redacted real runtime proof.

## Reproduction Assessment

Yes from source inspection: current main builds snapshots without the resolved `skillKey` and later resolves snapshot env config by `skill.name`, so config under `skills.entries.<skillKey>` is missed when the display name differs. I did not run tests because this review is read-only.

## Solution Assessment

Mostly yes: persisting the resolved key and using it in snapshot env lookup is the right owner-boundary fix, and refreshing old snapshots missing the field is the right upgrade shape. The remaining gap is proof, especially a focused old persisted-snapshot test and real runtime evidence.
