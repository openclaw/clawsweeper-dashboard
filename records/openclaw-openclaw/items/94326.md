## Summary

Keep open for normal maintainer review; the patch is a focused fix for the current memory-wiki `Index` page loss and I found no blocking correctness or security issue.

## What This Changes

The PR adds `slugifyWikiPageStem` and routes memory-wiki synthesis and ingest direct page writes through it so title `Index` uses a scannable `index-<hash>` path.

## Best Possible Solution

Land this direct-page-stem guard after normal required checks complete, and keep https://github.com/openclaw/openclaw/pull/93843 for the broader distinct-title collision work.

## Reproduction Assessment

Yes. Current main maps direct `Index` synthesis/source writes to `index.md`, and compile/query/status treat that filename as generated index metadata rather than queryable user content.

## Solution Assessment

Yes. The narrowed helper fixes the two direct writers that can collide with generated `index.md`; changing `slugifyWikiSegment` would churn sibling persisted identities, and making scanners treat `index.md` as user content would fight the compiler-owned index contract.
