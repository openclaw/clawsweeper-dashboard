## What This Changes

The PR adds Codex migration planning/apply logic, tests, docs, and changelog updates for activating source-installed openai-curated Codex plugins through the Codex app-server instead of OpenClaw dynamic tools.

## Best Possible Solution

Wire `--plugin` through the migrate CLI/types/context, keep discovery against the source Codex home, run apply-time `plugin/install` against the resolved managed target agent app-server home, add regression coverage for both paths, and supply live migration-apply proof after the base PR lands.

## Reproduction Assessment

Yes for the PR review findings: the PR head documents `--plugin` while the CLI/types/context still omit it, and source inspection shows apply sets `CODEX_HOME` to the migration source while current app-server launch code preserves explicit `CODEX_HOME`. I did not run tests because this review is read-only and the source evidence is sufficient.

## Solution Assessment

No, not as currently proposed. The maintainable shape is to add the missing CLI/context plumbing and separate source-home discovery from target managed-home plugin installation.

The app-server bridge preserves explicit `CODEX_HOME`, while later OpenClaw Codex turns use the managed per-agent home, so migration can report success without making the plugin available in the target session.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff touches a plugin-install control-plane path, and the current apply code can install or enable plugins in the source Codex home instead of OpenClaw's managed target home.

Concerns:

- **[medium] Plugin install targets the source Codex home:** `extensions/codex/src/migration/apply.ts:108`
  - body: Because apply sets `CODEX_HOME` from `ctx.source`, app-server `plugin/install` can mutate the user's personal Codex runtime directory rather than the managed OpenClaw agent directory. Plugin activation is a capability surface, so this should be fixed before merge.
