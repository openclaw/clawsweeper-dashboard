## Summary

Keep open. Current main already supports `agents.defaults.verboseDefault` as persisted config and runtime input, but the model-facing `gateway config.patch` path still intentionally blocks it through a fail-closed allowlist, so the remaining request is a feature and trust-boundary policy decision rather than an implemented bug fix.

## What This Changes

Allow verbose mode defaults to be set persistently through the agent-facing gateway `config.patch` flow, or add another persistent verbose configuration key.

## Best Possible Solution

Make an explicit maintainer policy call at the gateway-tool trust boundary; if approved, allow the existing verbose default paths with guard coverage and docs rather than adding a second persistent key.

## Reproduction Assessment

Yes. Source inspection shows an agent-facing `gateway config.patch` for `agents.defaults.verboseDefault` reaches the allowlist guard and is rejected as a protected config path.

## Solution Assessment

Unclear until the policy decision is made. If this should be agent-tunable, reusing the existing `verboseDefault` config is narrower than adding a new `verbosePersist` key.
