## Summary

Keep open. Current main already defines, documents, and honors `agents.defaults.verboseDefault`, but the reported gap is that the agent-facing Gateway config tool refuses to persist that field; changing that allowlist is a security/product decision rather than conservative cleanup.

## What This Changes

The issue requests persisting verbose mode across sessions by allowing `agents.defaults.verboseDefault` to be set through `gateway config.patch` or by adding an equivalent persistent setting.

## Best Possible Solution

Decide the Gateway config mutation policy first; if verbose defaults are approved as agent-tunable, allowlist the exact global and per-agent verbose paths with guard coverage and docs instead of adding a duplicate verbose setting.

## Reproduction Assessment

Yes. Source inspection on current main shows `agents.defaults.verboseDefault` is accepted and used, but an agent-facing `gateway config.patch` changing that path is outside `ALLOWED_GATEWAY_CONFIG_PATHS` and will be rejected as protected before the Gateway RPC runs.

## Solution Assessment

Unclear until maintainers decide the policy boundary. If this should be agent-tunable, reusing the existing `verboseDefault` field is narrower than adding a second persistent verbose key.
