## Summary

Keep open: current main supports `agents.defaults.verboseDefault` as a runtime config value, but the agent-facing `gateway config.patch` write guard intentionally fail-closes to a narrow allowlist that does not include this path. Deciding whether verbose disclosure defaults are safe for persistent agent-driven config writes needs maintainer product/security judgment.

## What This Changes

Request to let users persist verbose mode across sessions by making `agents.defaults.verboseDefault` writable through `gateway config.patch` or by adding an equivalent persistent setting.

## Best Possible Solution

If maintainers want this behavior, add `agents.defaults.verboseDefault` and likely `agents.list[].verboseDefault` to the gateway tool allowlist with guard tests, then update docs to explain the security and session-default precedence.

## Reproduction Assessment

Yes from source inspection: `agents.defaults.verboseDefault` is schema/runtime-supported, but `gateway config.patch` rejects changed paths outside `ALLOWED_GATEWAY_CONFIG_PATHS`, and that list does not include verbose defaults. I did not run a live mutation because this review is read-only.

## Solution Assessment

Unclear as proposed; removing protection may be the narrow code change, but the maintainable fix requires a maintainer decision that verbose defaults are safe as agent-tunable persistent config. If approved, the established pattern is an explicit allowlist addition plus guard coverage, not a broad protection removal.
