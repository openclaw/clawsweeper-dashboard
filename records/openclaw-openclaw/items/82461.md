## Summary

Keep open. Current main already has and honors `agents.defaults.verboseDefault`, but the remaining request is to expand the agent-facing `gateway config.patch` write boundary, and that still needs maintainer product/security judgment rather than conservative cleanup.

## What This Changes

Allow persistent verbose defaults to be set through the agent-facing Gateway config patch path so verbose mode survives session resets and restarts.

## Best Possible Solution

Make an explicit maintainer decision at the Gateway tool policy boundary; if approved, allowlist the exact global and per-agent verbose default paths with guard coverage and docs that distinguish direct operator config edits from agent-driven config writes.

## Reproduction Assessment

Yes. Source inspection on current main shows an agent-facing patch to `agents.defaults.verboseDefault` is outside `ALLOWED_GATEWAY_CONFIG_PATHS` and will throw the reported protected-path error before the Gateway RPC runs.

## Solution Assessment

Unclear until maintainers decide the policy boundary. If the path should be agent-tunable, reusing the existing `verboseDefault` config is narrower than adding a second persistent verbose key.
