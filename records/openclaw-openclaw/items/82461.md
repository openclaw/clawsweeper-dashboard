## Summary

Keep open: current main already supports `agents.defaults.verboseDefault` as a schema/runtime config value, but the agent-facing `gateway config.patch` guard intentionally rejects non-allowlisted paths and does not include verbose defaults. Persistently enabling verbose output from an agent-driven config write has disclosure implications, so this needs maintainer product/security judgment before implementation.

## What This Changes

The issue requests making `agents.defaults.verboseDefault` persistently settable through `gateway config.patch`, or adding an equivalent persistent verbose default option.

## Best Possible Solution

If approved, add explicit allowlist entries for `agents.defaults.verboseDefault` and likely `agents.list[].verboseDefault`, keep the existing `off|on|full` contract, add guard tests, and document the disclosure/precedence behavior.

## Reproduction Assessment

Yes. Source inspection shows `agents.defaults.verboseDefault` is valid and read at runtime, but `gateway config.patch` rejects any changed path outside `ALLOWED_GATEWAY_CONFIG_PATHS`, and verbose defaults are not in that allowlist.

## Solution Assessment

Unclear as proposed. Removing protection broadly is not the best path; the maintainable fix, if approved, is a narrow allowlist expansion with regression coverage and docs/security wording.
