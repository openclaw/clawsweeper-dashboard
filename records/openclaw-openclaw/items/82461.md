## Summary

Keep open. Current main already supports `agents.defaults.verboseDefault` as config and consumes it at runtime, but the agent-facing `gateway` tool still deliberately blocks that path through its narrow `config.patch` allowlist, so the remaining request is a trust-boundary feature decision rather than an implemented bug fix.

## What This Changes

Allow persistent verbose mode to be set through `gateway config.patch` by making `agents.defaults.verboseDefault` agent-tunable, or by adding an equivalent persistent config key.

## Best Possible Solution

Decide at the gateway-tool policy boundary whether `verboseDefault` belongs in the narrow agent-tunable allowlist; if approved, add global and likely per-agent verbose allowlist entries with guard coverage and docs that distinguish direct operator config from agent-driven config edits.

## Reproduction Assessment

Yes. Source inspection shows an agent-facing `gateway config.patch` that changes `agents.defaults.verboseDefault` resolves to a path outside `ALLOWED_GATEWAY_CONFIG_PATHS` and throws the reported protected-path error.

## Solution Assessment

Unclear until the policy decision is made. If the path should be agent-tunable, reusing the existing `verboseDefault` config is narrower than adding a second `verbosePersist` key.
