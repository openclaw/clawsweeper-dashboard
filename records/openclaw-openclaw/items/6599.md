## What This Changes

Requested behavior: add a `/models test-fallback` command that intentionally skips the primary model, probes configured fallback candidates, and reports which fallback responds.

## Best Possible Solution

Define one canonical fallback-chain probe that reuses the real model/auth fallback path, intentionally starts at configured fallback candidates, reports the responding model, supports first-fallback and all-fallback modes, and documents how it differs from broader model health checks.

## Reproduction Assessment

Yes, at source level: current main has no `models test-fallback` CLI command or `/models test-fallback` action, and the existing probe path is provider/auth oriented. This is missing requested capability rather than a failing existing behavior contract.

## Solution Assessment

Unclear as an exact implementation until maintainers choose the canonical probe UX. A targeted fallback-chain probe is plausible, but it should be reconciled with broader configured-model health checks before code is queued.
