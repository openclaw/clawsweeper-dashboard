## Summary

Current main still lacks a proactive fallback-chain test: existing status probes validate provider auth, while automatic primary probing serves runtime recovery rather than operator-requested fallback verification. The request remains useful, but maintainers should consolidate its public interface with the broader configured-model health-check proposal before implementation.

## What This Changes

Add a command that intentionally skips the primary model, sends real requests to the first or all configured fallbacks, and reports each result.

## Best Possible Solution

Define one canonical configured-model health surface with fallback-order targeting, then reuse production fallback candidate, authentication, and request-execution paths while reporting each tested model distinctly.

## Reproduction Assessment

Yes at source level: current command registrations, parser behavior, documentation, and fallback runtime conclusively show that the requested operator-facing diagnostic is absent. This is a feature gap rather than a failing existing contract.

## Solution Assessment

The proactive-verification goal is sound, but the dedicated slash-command proposal is not yet proven to be the best interface. Maintainers should first decide whether it becomes fallback-scoped behavior within the broader per-model execution probe or remains a distinct command.
