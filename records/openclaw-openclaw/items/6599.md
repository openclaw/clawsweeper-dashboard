## Summary

Keep open: current main and v2026.6.6 still lack a first-class `/models test-fallback` fallback-chain probe, and the exact implementation attempts are closed unmerged while the open related requests are broader product work.

## What This Changes

Request to add a `/models test-fallback` command that intentionally skips the primary model and probes configured fallback models before a real outage.

## Best Possible Solution

Ship one canonical fallback-chain probe that reuses the real model/auth fallback path, intentionally starts at configured fallback candidates, supports first-fallback and all-fallback modes, reports the responding model, and documents how it differs from auth-profile probes and broader model health checks.

## Reproduction Assessment

Yes, at source level: current main has no `models test-fallback` CLI command or `/models test-fallback` action, and the existing probe path is provider/auth oriented. This is missing requested capability rather than a failing existing behavior contract.

## Solution Assessment

Unclear as an implementation until maintainers choose the canonical probe UX. A targeted fallback-chain probe is plausible, but it should be reconciled with broader `models status --probe --all` health checking before code is queued.
