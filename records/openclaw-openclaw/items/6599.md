## Summary

Keep open: current main and the latest release still lack a targeted fallback-chain test command, the direct implementation PRs are closed unmerged, and the broader model-health request does not replace this specific primary-skipping fallback probe.

## What This Changes

The issue requests a `/models test-fallback` command that intentionally skips the primary model and verifies configured fallback candidates before a real outage.

## Best Possible Solution

Define one canonical fallback-chain probe that reuses the real model/auth fallback path, starts at configured fallback candidates, reports which model answered, supports first-fallback and all-fallback modes, and documents how it differs from broad model-health probes.

## Reproduction Assessment

Yes. At source level, current main has no `models test-fallback` CLI command or `/models test-fallback` action, and the existing `models status --probe` path is auth/profile oriented rather than a primary-skipping fallback-chain test.

## Solution Assessment

Unclear as an exact implementation until maintainers choose the canonical probe UX. A targeted fallback-chain probe is plausible, but it should be reconciled with broader configured-model health checks before implementation is queued.
