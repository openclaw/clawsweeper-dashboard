## Summary

Current main still lacks a proactive fallback-chain test, the existing live status probe validates auth profiles rather than fallback execution, and no viable open or merged item supersedes this request; keep it open for a maintainer product decision on the canonical command surface.

## What This Changes

Add a command that skips the primary model, live-tests the first or all configured fallback candidates, and reports each result.

## Best Possible Solution

Build one shared core diagnostic that resolves the real configured fallback candidates and auth profiles, probes each candidate in isolation, and exposes consistent first-fallback and all-fallback reporting through the chosen CLI and slash-command presentations.

## Reproduction Assessment

Yes at source level: current main has no registered fallback-test action, and the existing live status probe follows provider-auth semantics rather than the requested fallback execution path. This is a capability-gap verification, not a broken-behavior reproduction.

## Solution Assessment

Yes in intent, but the best implementation is not a standalone slash-command probe; it should reuse a shared canonical fallback diagnostic that can also support the CLI and avoid competing probe semantics.
