## Summary

Keep open. Current main still lacks the requested fallback-chain test command, the direct implementation attempts are closed unmerged, and the open related probe work is broader per-model health checking rather than the primary-skipping fallback-chain probe requested here.

## What This Changes

The issue requests a first-class `/models test-fallback` style command to proactively verify configured model fallbacks without waiting for a real provider failure.

## Best Possible Solution

Ship a focused fallback-chain probe that reuses OpenClaw’s real model/auth execution path, starts at configured fallback candidates, reports which model answered, supports first-fallback and all-fallback modes, and documents how it differs from auth-profile probes and broader per-model health checks.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs inspection on current main shows no `/models test-fallback` or `openclaw models test-fallback` surface exists.

## Solution Assessment

Yes. A narrow fallback-chain probe is the best fit; `models status --probe --all` style health checks are useful but do not by themselves prove the requested primary-skipping fallback behavior.

## Repair Work Prompt

Implement the fallback-chain test command requested by https://github.com/openclaw/openclaw/issues/6599. Add a first-class `/models test-fallback` path and, if consistent with existing CLI conventions, `openclaw models test-fallback`, reusing the real model/auth execution path while intentionally starting at configured fallback candidates instead of the primary. Default to probing the first fallback and support an all-fallback mode. Report success/failure and the model that responded. Respect agent-scoped fallback overrides, isolate each probe so later fallbacks do not mask the candidate being tested, and avoid changing normal runtime fallback ordering semantics. Coordinate with https://github.com/openclaw/openclaw/pull/63259 if it has landed or changed the probe helpers; this work should remain the targeted fallback-chain test, not a broad per-model health-check rewrite. Add focused unit coverage for slash parsing/handling, CLI registration, fallback candidate selection, no-fallback/error cases, and docs/changelog for the user-facing command.
