## Summary

Keep open. The previous ClawSweeper reviews still hold on current main: OpenClaw has fallback runtime and auth probes, but no proactive fallback-chain test command on the CLI or `/models` chat surface, and the direct implementation PR is closed unmerged.

## What This Changes

The issue requests a `/models test-fallback` style command that intentionally skips the primary model and verifies configured fallback candidates before a real provider failure occurs.

## Best Possible Solution

Add a first-class fallback-chain probe that reuses OpenClaw’s real fallback/auth resolution, starts at configured fallback candidates, reports per-candidate outcomes and the responding model, and documents how it differs from auth-profile probes and broader model health checks.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The high-confidence verification path is static inspection of current main’s CLI registration, slash parser/handler, docs, tests, and fallback runtime, which all show the requested command is absent.

## Solution Assessment

Yes. A focused fallback-chain probe is the narrow maintainable direction; #63145 may share lower-level probe code, but it is broader per-model health checking and does not replace the requested `/models test-fallback` workflow.
