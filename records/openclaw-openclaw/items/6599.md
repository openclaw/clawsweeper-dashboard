## Summary

Current main still lacks the requested fallback-chain execution probe: `/models` remains a browse/select surface, `models status --probe` tests auth profiles rather than deliberately starting at configured fallbacks, and the only direct implementation PR closed unmerged. The broader configured-model health request overlaps but does not fully replace this fallback-order-specific workflow, so this issue remains useful pending a maintainer product decision on the canonical probe surface.

## What This Changes

Add a `/models test-fallback` command that skips the primary model, tests the first or all configured fallbacks, and reports which model responds.

## Best Possible Solution

Create one shared real-request model diagnostic layer, then expose fallback-specific first/all behavior through the maintainer-chosen CLI or slash surface while keeping auth-profile probes distinct and documenting token, timeout, and rate-limit effects.

## Reproduction Assessment

Yes. Current main’s CLI registration, slash parser/handler, docs, and probe implementation directly show that no command intentionally executes the configured fallback chain starting after the primary.

## Solution Assessment

Unclear as proposed. The capability is valid, but maintainers should choose whether it belongs under `/models test-fallback`, the CLI `models status` probe family, or a shared surface with https://github.com/openclaw/openclaw/issues/63145 before code is written.
