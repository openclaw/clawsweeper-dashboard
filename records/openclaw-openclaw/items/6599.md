## Summary

Current main still has no operator-facing command that deliberately starts with configured fallback candidates and reports which model succeeds. The existing `models status --probe` path checks provider/auth-profile reachability, while the newer automatic fallback probe periodically tests whether a session can return from its active fallback to the primary; neither verifies fallback-chain readiness before an outage. The broader configured-model health request at https://github.com/openclaw/openclaw/issues/63145 overlaps but does not explicitly preserve fallback order or exercise the runtime chain, so this issue remains useful pending a maintainer decision on one canonical probe UX.

## What This Changes

Add a model command that proactively exercises the configured fallback chain, optionally testing only the first fallback or every fallback, and reports success or failure.

## Best Possible Solution

Provide one canonical core model-execution probe with explicit scopes for fallback-order validation and all-configured-model health, reuse normal provider/auth/model resolution without mutating session fallback state, and keep its output clearly distinct from auth-profile probes.

## Reproduction Assessment

Yes at source level: current main clearly lacks a manual fallback-chain execution action, and the available auth and automatic recovery probes have different contracts. This is a feature gap rather than a failing existing behavior that can be reproduced as a runtime bug.

## Solution Assessment

Unclear. A proactive real-request probe is the right capability, but a standalone slash action is not yet proven better than extending the canonical `models status --probe` surface shared with https://github.com/openclaw/openclaw/issues/63145.
