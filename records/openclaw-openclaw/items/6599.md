## Summary

Current main and the latest release still lack a proactive fallback-chain execution probe; the existing status probe checks provider authentication, and no viable open or merged PR supersedes the request. Keep this issue open for a maintainer decision on whether the capability should be a dedicated fallback command or part of the broader configured-model health surface.

## What This Changes

Requests a proactive command that exercises configured model fallbacks without waiting for a provider failure.

## Best Possible Solution

Provide one canonical, side-effect-free core probe that reports each attempted fallback and the responding model, supports first-fallback and full-chain scopes, and clearly distinguishes fallback execution from provider-auth and general per-model health checks.

## Reproduction Assessment

Yes at source level: current command registration, slash parsing, auth probing, and fallback execution can be inspected deterministically, and none exposes the requested action. This is a feature gap rather than a failing existing contract.

## Solution Assessment

Unclear: the requested behavior is useful and bounded, but maintainers should first choose whether it belongs under `/models`, `openclaw models status --probe`, or a unified configured-model health command to avoid parallel probe interfaces.
