## Summary

Keep open: current main and the latest release still lack a first-class fallback-chain test command, and the open related issues are broader product requests rather than exact replacements. The request is useful but remains a feature/product decision because it would add a live model-probe surface that needs a canonical UX and safety contract.

## What This Changes

Add a `/models test-fallback` command that intentionally skips the primary model and probes one or all configured fallback models.

## Best Possible Solution

Define one canonical core model-probe surface that can validate fallback order through the real model/auth resolution path while clearly distinguishing it from auth-profile probes and all-configured-model health checks.

## Reproduction Assessment

Yes, at source level: current main has no `models test-fallback` CLI command or `/models test-fallback` action, and the current probe path is provider/auth oriented. This is a feature request rather than a broken existing behavior repro.

## Solution Assessment

Unclear as a final product shape: a fallback-chain probe is useful, but maintainers should choose whether it is slash-only, CLI-backed, or part of a broader shared `models status --probe --all` design before implementation.
