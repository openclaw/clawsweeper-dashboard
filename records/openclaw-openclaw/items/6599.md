## Summary

Keep open: current main and v2026.6.6 still lack a `/models test-fallback` or equivalent fallback-chain probe, and the direct implementation attempts are closed unmerged. The related `models status --probe --all` issue is broader per-model health checking, so this remains a distinct product decision for a primary-skipping fallback test surface.

## What This Changes

Request to add a `/models test-fallback` command that proactively probes configured model fallback candidates without waiting for a real primary failure.

## Best Possible Solution

Provide one maintainer-approved fallback-chain probe that reuses OpenClaw's real model/auth fallback resolution, intentionally starts at configured fallback candidates, reports which model answered, supports first-fallback and all-fallback modes, and documents how it differs from auth-profile probes and broader per-model health checks.

## Reproduction Assessment

Yes, at source level: current main has no `models test-fallback` CLI command or `/models test-fallback` slash action, and the existing probe path is provider/auth-profile oriented. This is a feature request rather than a broken existing behavior contract.

## Solution Assessment

Unclear as a final product shape: a targeted fallback-chain probe is plausible, but maintainers still need to choose the canonical relationship with broader per-model health probes, live-call cost, timeout behavior, and output contract.
