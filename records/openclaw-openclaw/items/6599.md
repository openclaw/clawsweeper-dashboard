## Summary

Keep open: current main and the latest release still lack the requested fallback-chain test command, and the direct implementation attempts at https://github.com/openclaw/openclaw/pull/43342 and https://github.com/openclaw/openclaw/pull/43380 are closed unmerged. The broader health-check request at https://github.com/openclaw/openclaw/issues/63145 overlaps, but it is not an exact replacement for this issue.

## What This Changes

This issue requests a `/models test-fallback` command that intentionally probes configured fallback models before a real provider failure.

## Best Possible Solution

Ship one canonical fallback-chain probe that reuses OpenClaw’s real model/auth resolution path, intentionally starts at configured fallback candidates, reports which model answered, supports first-fallback and all-fallback modes, and documents how it differs from auth-profile probes and broader per-model health checks.

## Reproduction Assessment

Yes at source level: current main has no `/models test-fallback` action, and the existing status probe validates auth/profile reachability instead of intentionally walking the configured fallback chain. I did not run live provider probes because the missing behavior is a command-surface feature gap.

## Solution Assessment

Unclear until maintainers choose the canonical UX. A shared probe backend that can serve slash-command fallback testing and any future broader status health probe is safer than adding competing partial probe surfaces.
