## Summary

Current main and v2026.6.5 still lack a proactive fallback-chain execution probe; the existing status probe checks provider authentication, and the directly related implementation PR closed unmerged. The remaining question is which command surface should own this capability, because the broader configured-model health proposal overlaps substantially.

## What This Changes

Add a proactive command that executes the first configured fallback or tests every model in the fallback chain.

## Best Possible Solution

Provide one canonical real-completion model-health surface with explicit fallback-only and all-configured-model scopes, per-model results, and a clear distinction from the existing auth-profile probe.

## Reproduction Assessment

Yes at source level: current main and v2026.6.5 expose only provider-auth probing and no fallback-chain execution command, directly confirming the capability gap without inducing an actual provider outage.

## Solution Assessment

No, the exact standalone `/models test-fallback` proposal is not yet proven to be the best permanent shape; consolidating fallback scopes into the broader configured-model health surface would avoid duplicate diagnostics.
