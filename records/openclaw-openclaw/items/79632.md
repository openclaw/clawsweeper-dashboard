## Summary

Keep open: current main still has bundled-only provider-owner/runtime filters that can exclude global npm plugins, and the linked repair PRs for Brave/runtime and doctor repair are open rather than merged. The guardrail-bridge evidence makes this broader than the already-open Brave-only report, so duplicate cleanup would be premature.

## What This Changes

The issue asks for external npm plugins installed/configured through OpenClaw to be consistently visible to plugin listing, inspection, diagnostics, and runtime provider resolution.

## Best Possible Solution

Consolidate or land a scoped fix that lets explicitly configured global plugin providers participate in owner/runtime resolution, repairs unhealthy external install records through existing doctor/update paths, and preserves managed-root plus sandbox trust boundaries.

## Reproduction Assessment

Yes for the Brave/global-provider portion from source inspection: current main still filters explicit provider owner lookup and capability runtime loading to bundled plugin manifests. The broader guardrail-bridge registry inconsistency is plausible from the same install/diagnostic surface but was not live-reproduced here.

## Solution Assessment

Yes in direction, but not by broadly scanning workspace node_modules. The maintainable fix should use manifest/install-record ownership, remove bundled-origin filtering only where global providers are valid, and reuse doctor repair for broken external installs.
