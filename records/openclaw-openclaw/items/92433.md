## Summary

Keep open: current main still treats the queued user `message_end` transcript boundary as successful delivery, so the reported requester-run teardown window can still mark a completion delivered before any requester assistant processing or fallback turn occurs.

## What This Changes

The issue reports a race where a steered subagent completion is marked delivered after transcript enqueue even if the requester run ends before consuming it.

## Best Possible Solution

Define one consumed-or-recoverable completion delivery contract: active-run steering should either get a durable requester-adoption acknowledgement or route an unconsumed internal completion into exactly one follow-up direct delivery path at teardown.

## Reproduction Assessment

Yes, at source level: emit the matching queued user `message_end`, then end the requester run before any assistant response; current code has already resolved the wake and can mark delivery successful. The reporter's v2026.6.6 transcript and SQLite timing provide corresponding real-run evidence, but this read-only review did not execute the race.

## Solution Assessment

Unclear: the required invariant is correct, but the best implementation boundary still needs maintainer choice between a real consumed/adopted acknowledgement and a narrower teardown recovery path. A fix that only waits for transcript commit is not sufficient for this report.
