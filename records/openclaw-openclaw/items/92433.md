## Label Justifications

- `P1`: The issue can silently drop completed subagent results in active multi-agent workflows, leaving the requester waiting without visible failure.
- `impact:message-loss`: The central failure is a completion event being written but never processed, retried, or visibly delivered.
- `impact:session-state`: The registry can stamp the subagent run as delivered while the requester transcript contains an unprocessed trailing completion event.

## Summary

This issue should stay open: current main and the latest stable release still treat a queued transcript commit as successful subagent completion delivery, while the linked attempted fix was closed unmerged after maintainer review rejected its acknowledgement model.

## What This Changes

The issue asks OpenClaw to stop silently losing subagent completion handoffs when an active requester run accepts the queued internal completion event but ends before processing it.

## Best Possible Solution

Add a lifecycle-level delivery outcome contract that distinguishes not committed, adopted, terminal unconsumed, and committed pending, with exactly-once fallback only for definitively unconsumed terminal outcomes.

## Reproduction Assessment

Yes, at source level: emit the queued internal completion user message_end, then let requester agent_end occur before any requester assistant response. Current main resolves the wait at message_end and maps the steered outcome to delivered.

## Solution Assessment

No final implementation path is settled. The current bug is real, but the closed assistant-ack PR was rejected, so the best fix needs the maintainer-approved closed delivery outcome model rather than a narrow replay-prone acknowledgement patch.
