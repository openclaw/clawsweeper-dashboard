## Label Justifications

- `P1`: A completed subagent result can be recorded as delivered without being processed by the requester, breaking active multi-agent workflows for real users.
- `impact:message-loss`: The failure can leave the completion event in the transcript with no assistant processing, no retry, and no visible delivery.
- `impact:session-state`: The transcript, subagent run delivery state, and pending-final-delivery bookkeeping can disagree about whether the requester actually consumed the completion.

## Summary

Source and shipped-release checks both show the central race is still present: current main and v2026.6.10 still treat a queued-and-committed active-run steer as delivered before requester consumption. The prior direct fix PR was closed unmerged after maintainer feedback requesting a closed lifecycle delivery outcome contract, so this issue remains the canonical thread rather than a close candidate.

## What This Changes

The issue reports that subagent completion announcements can be silently credited as delivered when active-run steering commits the internal event to the requester transcript but the requester run ends before processing it.

## Best Possible Solution

Define and implement a closed delivery outcome contract that distinguishes not-committed, adopted/consumed, terminal-unconsumed, and committed-pending states, with exactly-once fallback only for the terminal-unconsumed case.

## Reproduction Assessment

Yes, at source level: current main resolves active-run steering at the queued user message_end transcript boundary, maps the steer to delivered, and lifecycle cleanup stamps delivered state from that result. I did not run a live timing reproduction because this read-only review should not create runtime artifacts and the observed race is intermittent.

## Solution Assessment

No final implementation path is settled. The requested behavior is a real bug fix, but the prior assistant-ack solution was rejected, so the safest solution is the maintainer-requested closed lifecycle outcome contract rather than another one-sided acknowledgement patch.
