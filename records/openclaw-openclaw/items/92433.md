## Label Justifications

- `P1`: A completed subagent result can be marked delivered without requester processing, breaking active multi-agent workflows for real users.
- `impact:message-loss`: The failure commits an internal completion event but never processes, retries, or visibly delivers the subagent answer.

## Summary

Keep open: current main and v2026.6.10 still credit active-run subagent completion steering at the transcript-commit boundary, and the same-root assistant-ack fix was closed unmerged after maintainer rejection of that design.

## What This Changes

The issue asks OpenClaw to avoid silently losing subagent completion announces when an active requester run ends after queue commit but before processing the completion event.

## Best Possible Solution

Define a closed completion-delivery outcome contract, then implement exactly-once fallback only for terminal-unconsumed completion events while preserving committed-pending reconciliation.

## Reproduction Assessment

Yes at source level: current main resolves active completion steering at queued transcript commit and then records the steer as delivered. I did not run a live timing repro because the teardown window is intermittent and this review is read-only.

## Solution Assessment

No final implementation path is settled. The current code is not the best solution, and the rejected assistant-ack PR shows maintainers need a closed lifecycle delivery outcome contract before another fix lands.
