## Summary

Current main still has the reported queue-accepted-to-delivered gap, and no merged fix or canonical replacement owns this exact requester-teardown race. The issue should remain open for an owner-led exactly-once delivery design, with the closely related open `sessions_yield` work considered before implementation.

## What This Changes

Prevent a queued subagent completion from being marked delivered when requester-run teardown occurs before the queued event is processed.

## Best Possible Solution

Keep one canonical persistent completion state until either requester consumption is acknowledged or teardown atomically transfers the event to the existing direct-delivery path, with idempotency covering both outcomes and no global change to ordinary transcript-commit queue semantics.

## Reproduction Assessment

Yes, at high source confidence. Current main acknowledges transcript persistence as successful steering and has no later consumption acknowledgement, so ending the requester run between those phases establishes the reported failure path even though this read-only review did not execute the timing race.

## Solution Assessment

No, neither proposed mechanism can be selected safely without defining the exactly-once owner. The narrowest maintainable solution is a subagent-specific durable consumption or teardown-handoff contract coordinated with existing retry and idempotency state, rather than changing shared queue semantics.

- The acknowledgement is ambiguous after enqueue: retrying directly can duplicate a completion if the requester consumed it just before teardown, while treating enqueue as final can silently lose it.
- Changing the shared embedded-message queue contract could affect `sessions_send` and other callers that intentionally use transcript commit rather than agent-consumption acknowledgement.
- Several open completion-delivery PRs touch adjacent lifecycle branches, so implementing this independently could create overlapping fallback paths or inconsistent delivery semantics.
