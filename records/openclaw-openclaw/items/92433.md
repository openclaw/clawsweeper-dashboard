## Summary

Keep open: current main and the latest stable release still credit active-run subagent completion steering at the transcript-commit boundary, and the direct assistant-ack fix candidate was closed unmerged after maintainer review requested a safer lifecycle delivery contract.

## What This Changes

The issue reports that subagent completion delivery can be marked successful when an active requester run commits the internal completion message to the transcript but ends before consuming it, causing silent message loss.

## Best Possible Solution

Define the closed completion-delivery outcome contract, then implement exactly-once recovery only for terminal-unconsumed completions while keeping committed-pending events owned by reconciliation.

## Reproduction Assessment

Yes, at source level: the active-run queue wait resolves at queued user `message_end`, while dispatch maps that steered outcome to delivered. I did not run a live reproduction because this read-only review cannot safely force the intermittent requester-teardown window.

## Solution Assessment

No final implementation path is settled. The bug is real, but the durable fix should follow the maintainer-requested lifecycle delivery outcome contract rather than another one-sided acknowledgement patch.
