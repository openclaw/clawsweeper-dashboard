## Summary

Keep this open. Current main and the latest release still treat the active-run transcript commit as a delivered subagent completion, and the only PR that directly linked this issue was closed unmerged after maintainer review said the bug is valid but the acknowledgement design is unsafe.

## What This Changes

The issue reports a race where a subagent completion announcement can be steered into an active requester run, transcript-committed, marked delivered, and then left unprocessed when that requester run exits.

## Best Possible Solution

Keep this precise race report open and fix it with a lifecycle-level delivery outcome that distinguishes uncommitted, adopted, terminal-unconsumed, and committed-pending states before fallback or retry.

## Reproduction Assessment

Yes at source level: queue a completion into an active requester, observe the queued user `message_end`, then let requester `agent_end` occur before any assistant response. Current main has already resolved the wake and mapped it to delivered before that post-commit terminal outcome can be represented.

## Solution Assessment

No final implementation is settled. The closed PR shows assistant-response acknowledgement is only a plausible fix; maintainer review asked for a closed lifecycle delivery outcome instead.
