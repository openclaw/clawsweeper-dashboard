## Label Justifications

- `P1`: The latest stable release can silently lose a core subagent completion during a realistic requester teardown race, leaving no retry or operator-visible error.
- `impact:message-loss`: A completion can be committed and recorded as delivered while never being processed or relayed to the requester.

## Summary

Current main still finalizes a steered subagent completion at transcript-commit time, before requester processing is proven, so the reported teardown race remains possible and is not owned by another open fix. The May timing fix narrowed the earlier enqueue race but did not cover the later commit-to-consumption window demonstrated by the reporter’s v2026.6.6 transcript and state evidence.

## What This Changes

Prevent subagent completion announcements from being marked delivered when they are queued into a requester run that exits before processing them.

## Best Possible Solution

Carry a completion identity through requester adoption and atomically recover through the existing direct-delivery path when requester teardown occurs without that acknowledgement, with one lifecycle component owning the transition.

## Reproduction Assessment

Yes at source level: commit the queued internal completion, then terminate the active requester before any assistant response adopts it; current code has already classified delivery as successful. The reporter’s v2026.6.6 transcript and SQLite timestamps provide matching real-run evidence, although this read-only review did not execute an instrumented reproduction.

## Solution Assessment

The consumed-or-recoverable invariant is the correct solution, but the issue’s two suggested implementation locations are alternatives rather than changes that should both be added. An agent-lifecycle owner should select one canonical acknowledgement and atomic fallback boundary.
