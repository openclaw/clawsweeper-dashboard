## Label Justifications

- `P1`: A successful child completion can be silently marked delivered while the requester never processes it, breaking active multi-agent workflows.
- `impact:message-loss`: The reported failure durably writes an internal completion event but can leave no assistant turn, retry, or visible follow-up for the user.

## Summary

Keep open: current main still treats transcript commit as successful subagent completion delivery, while the reported race is commit-before-requester-consumption. The reporter supplied stable-release forensics, a later commenter independently confirmed the current-main contract gap, and no merged or viable open PR owns this exact post-commit/pre-consumption failure.

## What This Changes

The issue asks OpenClaw to stop silently losing subagent completion events that are steered into an active requester run but never processed before that run ends.

## Best Possible Solution

Delivery bookkeeping should be tied to a durable requester-adoption or teardown-recovery signal so a post-commit unconsumed completion either becomes visible exactly once or remains pending for retry.

## Reproduction Assessment

Yes at source level, though I did not run a live current-main orchestration. Emit the matching queued user message_end, then end the requester run before an assistant response; current code has already resolved the announce as delivered.

## Solution Assessment

Unclear as an implementation choice. The invariant is right, but maintainers still need to choose between a consumed/adopted acknowledgement and a narrower teardown recovery that reroutes only unconsumed internal completions.
