## Label Justifications

- `P1`: An intermittent race can silently suppress a subagent completion in an active multi-agent workflow without recording a delivery failure.
- `impact:message-loss`: The requester permanently misses an expected completion while persisted bookkeeping incorrectly records successful delivery.

## Summary

The issue remains necessary: current main treats transcript commitment as successful completion delivery, but the supplied stable-release forensics show a distinct post-commit teardown window where the requester never processes the event. The earlier merged fix addressed pre-commit loss only, and no canonical open PR owns this remaining race.

## What This Changes

Ensure a subagent completion is not finalized as delivered when the requester transcript records the steered event but the active run ends before processing it.

## Best Possible Solution

Introduce one lifecycle-owned, exactly-once acknowledgement that distinguishes transcript persistence from requester consumption, and route an unconsumed internal completion through the existing direct completion turn when requester teardown wins the race.

## Reproduction Assessment

Yes at source level. Current main has a concrete interleaving where transcript commitment completes the delivery promise before requester processing, and the reporter supplied matching v2026.6.6 transcript and SQLite evidence; this review did not execute a live timing reproduction.

## Solution Assessment

The reported fallback-or-flush direction is valid, but the best implementation requires an agent-lifecycle decision about the canonical consumption acknowledgement so the repair cannot produce duplicate completions.
