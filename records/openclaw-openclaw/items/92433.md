## Summary

The issue identifies a source-valid race in the current stable and main delivery path: transcript persistence is treated as completed delivery even though requester-run consumption is not acknowledged, so teardown can silently strand the completion.

## What This Changes

Fix a race where a subagent completion is recorded as delivered after being queued into a requester run that terminates before consuming it.

## Best Possible Solution

Keep steered completion delivery pending through transcript commitment, then atomically mark it delivered on requester consumption or claim it for the existing direct-delivery path when requester teardown wins the race.

## Reproduction Assessment

Yes at source level with high confidence: transcript `message_end` is the delivery boundary, and the supplied timestamps describe a valid requester teardown after that event but before processing. A deterministic lifecycle test still needs to synthesize the ordering.

## Solution Assessment

Yes in direction: recovery belongs at the run and delivery-state lifecycle boundary, not in a periodic transcript sweep. The implementation must provide exactly-once ownership between consumption and direct fallback.
