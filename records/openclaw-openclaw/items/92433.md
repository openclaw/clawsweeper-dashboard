## Summary

This issue is still necessary: current main and the latest stable path still resolve completion steering at the queued user-message transcript boundary, not at requester consumption. ## What This Changes

The issue reports a subagent completion-delivery race where a queued internal completion can be committed to an active requester transcript but never consumed before the requester run ends.

## Best Possible Solution

Implement a maintainer-approved closed delivery outcome contract for completion handoffs, with pending-delivery reconciliation owning committed-pending states and exactly-once fallback only for definitively terminal-unconsumed completions.

## Reproduction Assessment

Yes. The source-level path is clear: emit the matching queued user message_end, then end the requester run before any assistant response; current code has already resolved the wait and maps the steer to delivered.

Unclear as a final design. Keeping the issue open is correct, but the closed assistant-ack PR was not accepted; the best fix now needs the lifecycle-contract shape described in the related PR review.
