## Summary

The issue remains necessary: current delivery success is proven only at the queued user-message transcript boundary, not at requester processing, so the reported post-commit/pre-assistant teardown window can still strand a completion without retry. The report supplies precise stable-release forensics, current source supports the interleaving, and no linked or canonical open PR currently owns this exact remaining race.

## What This Changes

Recover subagent completion delivery when a requester run ends after the steered completion reaches its transcript but before an assistant turn processes it.

## Best Possible Solution

Extend the existing announce state machine with a completion-identity acknowledgement that proves requester adoption, and atomically fall back to direct delivery when requester termination occurs without that acknowledgement.

## Reproduction Assessment

Yes at source level: emit the matching queued user `message_end`, then end the requester run before any assistant response; current code has already resolved and classified the announce as delivered. The reporter's OpenClaw v2026.6.6 transcript and SQLite timing provide corresponding real-run evidence, though this read-only review did not execute an instrumented reproduction.

## Solution Assessment

Yes on the required consumed-or-recoverable invariant, but the issue's two suggested implementation locations are alternatives rather than a settled design. The narrowest maintainable solution is one exactly-once adoption signal shared by active-run delivery and persistent announce bookkeeping.
