## Summary

Keep open. Current main still has no legacy `channels.imessage` auto-acknowledgment or native typing-feedback path before agent dispatch, and the latest maintainer comment explicitly kept this legacy channel request separate from BlueBubbles. BlueBubbles already covers automatic typing on the recommended iMessage route, but that does not implement the requested behavior for the legacy `imsg` channel.

## What This Changes

Add immediate pre-dispatch acknowledgment or typing feedback for inbound messages on the legacy iMessage channel.

## Best Possible Solution

Keep this issue open until maintainers either implement or explicitly decline legacy `channels.imessage` feedback. If implemented, add a default-off typed config/schema/docs surface, send feedback only after access and mention gating but before `dispatchInboundMessage`, prefer native typing only after validating current `imsg` support, and cover text fallback with echo-cache regression tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature request. The current-main verification path is high-confidence: the strict iMessage config schema has no `autoAck` field, and the legacy monitor has no pre-dispatch feedback call.

## Solution Assessment

Unclear. The requested text auto-ack is a narrow implementable fallback, but the best final shape needs maintainer choice between a text acknowledgment, verified native `imsg` typing, or declining additional legacy-channel behavior now that BlueBubbles is the recommended iMessage route.
