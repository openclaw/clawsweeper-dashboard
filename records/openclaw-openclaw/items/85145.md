## Label Justifications

- `P1`: The report describes a broken headless model-auth workflow for OpenAI Codex API-key users that can block real agent use on hosts without OAuth browser access.
- `impact:auth-provider`: The affected surface is provider auth profile creation, provider matching, and credential resolution for OpenAI Codex models.

## Summary

Current main still lacks a non-interactive API-key auth writer for OpenAI Codex, while an open focused PR already references this issue with closing syntax. This issue should stay open until https://github.com/openclaw/openclaw/pull/85162 lands or is replaced.

## What This Changes

The issue asks for headless OpenAI Codex API-key auth to write an `api_key` profile shape instead of forcing users through `paste-token` token credentials or hand-edited auth JSON.

## Best Possible Solution

Land or replace the focused `paste-api-key` implementation, preserving token paste semantics while adding the correct `api_key` profile/config path, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Source inspection shows current main's `paste-token` command writes token-mode credentials, while the existing API-key resolver consumes `api_key` profiles through `key` or `keyRef`; I did not run a live credentialed Codex request in this read-only review.

## Solution Assessment

Yes. A separate `paste-api-key` command is the narrow maintainable solution because it preserves token/OAuth semantics and uses the existing `api_key` credential and config contracts.
