## Summary

Keep this issue open: current main still lacks a non-interactive API-key paste path for OpenAI Codex auth, and the linked open fix PR should be reviewed or landed before this issue is closed.

## What This Changes

The issue reports that `paste-token` stores OpenAI Codex API keys as token credentials instead of API-key credentials and asks for a non-interactive API-key CLI path plus clearer diagnostics.

## Best Possible Solution

Review and land the linked focused PR if it continues to match the existing auth-profile contract, then close this issue as implemented after merge.

## Reproduction Assessment

Yes. Source inspection shows current `paste-token` writes token profiles, API-key resolution consumes `key` on `api_key` profiles, and the Codex app-server bridge sends those two credential modes through different login params.

## Solution Assessment

Yes. The narrow maintainable solution is a first-class API-key paste path that writes the existing `api_key` profile schema while preserving `paste-token` for token/OAuth-style credentials.
