## What This Changes

Headless OpenAI Codex users need a CLI path that writes `api_key` auth profiles with the existing `key` field instead of token-style credentials.

## Best Possible Solution

## Reproduction Assessment

Yes at source level: current main's `paste-token` writes token profiles, the API-key resolver consumes `key` on `api_key` profiles, and the Codex bridge treats those credential types differently. I did not run a live credentialed Codex request in this read-only review.

## Solution Assessment

Yes, the linked PR's direction is the narrow maintainable path: add an explicit API-key paste command while preserving existing token/OAuth semantics. The issue should remain open until that implementation or an equivalent fix lands.
