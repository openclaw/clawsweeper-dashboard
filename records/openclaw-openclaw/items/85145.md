## Summary

Keep open: current main still allows the reported `paste-token` path to create a token-mode `openai-codex` profile, while the working API-key path lives in provider-owned API-key auth rather than this `models auth` command surface.

## What This Changes

The issue reports that `models auth paste-token --provider openai-codex` writes OAuth-token shaped credentials for an OpenAI API key flow and asks for a safe API-key CLI path or clearer diagnostics.

## Best Possible Solution

Keep API-key storage on the provider-owned `api_key` contract and make `models auth` expose or route to that flow while preventing `paste-token` from creating unusable Codex token profiles.

## Reproduction Assessment

Yes from source inspection: `paste-token` writes `type: "token"` and config `mode: "token"` for `openai-codex`, while the Codex app-server bridge treats token profiles as ChatGPT/OAuth tokens. I did not run the CLI because this review is read-only and that command mutates auth stores.

## Solution Assessment

No, using `paste-token` for OpenAI API keys is not the best solution. The maintainable fix is to reuse the existing provider API-key auth method for a non-interactive `models auth` path and add a guard or diagnostic so token paste cannot create this bad Codex profile.
