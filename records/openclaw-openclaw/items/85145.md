## Label Justifications

- `P1`: The report describes a broken headless auth workflow for OpenAI Codex models that blocks real agent use with API keys.
- `impact:auth-provider`: The affected behavior is provider auth profile creation, provider matching, and model auth credential resolution.

## Summary

Current main still has a real gap: the CLI only offers token-paste semantics, while the Codex auth runtime treats token profiles as OAuth ID tokens and API-key profiles require the `key` field plus matching provider config. The report is actionable, recent, and source-reproducible from the current auth/profile code, so it should stay open as a narrow fix candidate.

## What This Changes

The issue requests a non-interactive OpenAI Codex API-key auth path, or corrected `paste-token` behavior and diagnostics, for headless deployments.

## Best Possible Solution

Add a first-class non-interactive API-key auth path for provider profiles, preserve token paste for OAuth-style tokens, and improve the wrong-field diagnostic so users do not need to hand-edit auth JSON.

## Reproduction Assessment

Yes from source inspection: current `paste-token` writes a token profile, Codex token profiles are forwarded as external auth, and the API-key resolver only accepts `key` on an `api_key` profile. I did not run a live Codex request because this read-only review cannot exercise credentials.

## Solution Assessment

Yes, a focused CLI/auth-profile fix is the right shape: expose or route non-interactive API-key profile creation without changing OAuth token semantics, and add a targeted diagnostic for `api_key` versus `key`.
