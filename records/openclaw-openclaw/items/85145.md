- `impact:auth-provider`: The affected surface is provider auth profile creation, provider matching, and API-key credential resolution.

## Summary

Keep open: current main still has the reported auth CLI gap, and the linked focused PR is the right implementation candidate. This issue should stay open until that PR lands or maintainers choose a replacement fix.

## What This Changes

The issue asks for a non-interactive OpenAI Codex API-key auth path because `paste-token` writes token credentials that do not match the existing `api_key` profile contract.

## Best Possible Solution

Land a narrow API-key paste path that writes the existing `type: "api_key"` plus `key` profile shape and `mode: "api_key"`, while keeping `paste-token` token-only and covering the CLI/auth-store behavior with focused tests.

## Reproduction Assessment

Yes, from source inspection: current main has only `paste-token`, that command writes token-mode profiles, and API-key resolution requires an `api_key` profile with `key`. I did not run a live credential flow, but the failing shape is clear from the CLI and resolver code.

## Solution Assessment

Yes, the linked separate `paste-api-key` direction is the cleanest fix because it adds the missing non-interactive API-key path without changing token/OAuth semantics. A broader auth migration is not needed for this issue.
