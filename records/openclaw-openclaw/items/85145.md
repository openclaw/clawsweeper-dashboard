## Label Justifications

- `P1`: The issue blocks headless OpenAI Codex API-key authentication for real model use on hosts where OAuth login is not available.
- `impact:auth-provider`: The affected behavior is provider auth profile creation, provider matching, and model credential resolution.

## Summary

Current main and the latest release still lack a non-interactive API-key paste path, and the linked fix PR remains open and unmerged, so this issue should stay open as the canonical tracker.

## What This Changes

Request a supported non-interactive API-key auth path for openai-codex so headless users can store api_key profiles with the key field and api_key config mode instead of token profiles.

## Best Possible Solution

Land the linked focused paste-api-key PR, or an equivalent core auth CLI change, with docs/tests while preserving paste-token semantics for token/OAuth-style credentials.

## Reproduction Assessment

Yes. Source inspection shows paste-token writes token-mode credentials and current main lacks paste-api-key, while the runtime API-key contract requires an api_key profile with key; I did not run a live credential flow.

## Solution Assessment

Yes. A separate paste-api-key command is the narrow maintainable solution because it uses the existing api_key credential/config contract without changing token paste semantics.
