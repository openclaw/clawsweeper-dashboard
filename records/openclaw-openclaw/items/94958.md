## Summary

Keep open: this PR targets the real Windows QMD command-path bug in https://github.com/openclaw/openclaw/issues/92302, but the broad backslash gate can break existing POSIX-style escaped command values and the PR still lacks real Windows OpenClaw/QMD runtime proof. https://github.com/openclaw/openclaw/pull/94841 is a stronger same-root-cause candidate, but it is not merged or proof-cleared enough to safely close this PR as superseded.

## What This Changes

This PR changes QMD backend config parsing so any `memory.qmd.command` containing a backslash bypasses `splitShellArgs`, and adds backend-config tests for Windows-style command paths.

## Best Possible Solution

Land one canonical config-parser fix that preserves Windows drive-letter and UNC QMD executable paths before POSIX parsing, keeps POSIX escapes intact, and includes Windows OpenClaw/QMD proof.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main routes `memory.qmd.command` through `splitShellArgs`, whose unquoted backslash branch consumes Windows path separators before QMD spawn resolution.

## Solution Assessment

No. The fix is in the right owner boundary, but using any backslash as the Windows signal is too broad; the safer fix is drive-letter/UNC detection while preserving POSIX shell parsing.
