## Summary

Keep open: the linked Windows QMD command-path bug is real and this PR targets the right area, but the broad backslash gate would regress existing POSIX shell-style command values and the supplied proof is unit-test only. Same-author sibling fix PRs are open, but they are not proof-positive enough to safely supersede this PR.

## What This Changes

The PR changes QMD memory backend config parsing so `memory.qmd.command` paths containing backslashes bypass `splitShellArgs`, and adds backend-config tests for Windows-style command paths.

## Best Possible Solution

Narrow the parser so only Windows drive-letter or UNC executable paths bypass POSIX tokenization, while existing POSIX shell-style quoting and escaping continue to use `splitShellArgs`.

## Reproduction Assessment

Yes, source-level: current main parses `memory.qmd.command` with `splitShellArgs`, whose unquoted backslash branch consumes Windows separators. I did not run a live Windows QMD setup.

## Solution Assessment

No. The fix is in the right owner boundary, but using any backslash as the Windows signal is too broad because POSIX shell escaping is an existing command-string behavior.
