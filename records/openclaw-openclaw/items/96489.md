## Summary

Keep open: current main and the latest release still allow literal token-shaped values in top-level `env.vars`, while the secrets audit scans registered credential targets and `.env` but does not yet flag this config-env surface. Existing config permission checks cover part of the report, but the requested schema/audit warning remains useful and security-sensitive.

## What This Changes

The issue requests warnings and audit coverage for plaintext API keys or channel tokens stored in the `openclaw.json` `env.vars` config block.

## Best Possible Solution

Add focused `env.vars` security guidance and high-confidence audit detection for plaintext secret-shaped values while keeping broader credential encryption and SecretRef support in their existing security/product tracks.

## Reproduction Assessment

Yes, by source inspection. Current main accepts string values in `env.vars`, applies them to runtime env handling, and the current secrets-audit target registry does not include `env.vars`.

## Solution Assessment

Unclear as proposed. Schema/docs warning and audit coverage are a good narrow direction, but startup permission enforcement and token-shape heuristics need maintainer/security direction to avoid duplicate warnings and false positives.
