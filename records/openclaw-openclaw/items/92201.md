## Summary

Keep this issue open. Current main includes the earlier multi-chunk signature fix and a newer durable transcript repair after successful recovery, but neither explains the reported generation of newly invalid signatures nor makes the recovery predicate inspect structured raw provider details when the visible error text is generic.

## What This Changes

Investigate intermittent Anthropic streamed-thinking signature corruption and ensure replay recovery can recognize the underlying provider rejection despite genericized user-facing error text.

## Best Possible Solution

Carry sanitized structured provider rejection details through the assistant stream event, make Anthropic recovery match those details narrowly, and capture or instrument the failing SSE assembly sequence before changing signature construction again.

## Reproduction Assessment

Partly yes. The missed-recovery mechanism is source-reproducible because recovery ignores structured error details and matches only errorMessage text; the intermittent generation of invalid signatures has strong production isolation evidence but no deterministic current-main event sequence yet.

## Solution Assessment

The structured-error recovery change is a narrow and appropriate fix, but the stream-assembly half should not be changed speculatively until raw events or focused instrumentation identify how a currently concatenated signature becomes invalid.
