## Summary

Current main contains the earlier signature-delta fix and a newly merged durable transcript repair, but neither establishes that this report’s intermittent fresh-signature corruption is fixed, and the stream recovery predicate still reads only the assistant event’s displayed `errorMessage`; the issue remains necessary.

## What This Changes

Fix intermittent Anthropic thinking-signature corruption in the embedded runner and make thinking-block recovery recognize the underlying provider rejection when the user-facing error is genericized.

## Best Possible Solution

Keep signed thinking intact by default, extend the existing recovery boundary to match sanitized structured provider details as well as displayed text, and use a redacted failing SSE/session capture to add a focused regression test and correct the exact assembly defect.

## Reproduction Assessment

Partly yes: source inspection gives a high-confidence path for the recovery miss when a terminal event exposes only generic `errorMessage` text. The intermittent creation of a bad fresh signature is not yet reproducible on current main without the offered redacted session/SSE artifacts.

## Solution Assessment

Yes for the two-part direction, but not for the build-time workaround: structured raw-error matching is the narrow recovery fix, while producer-side assembly should be changed only after a failing event sequence identifies the exact invariant being violated.
