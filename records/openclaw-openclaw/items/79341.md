## Summary

This issue remains actionable on current main: the image tool exposes an explicit model override, but the execution path still throws when no image model config can be resolved before that override reaches the fallback runner. The sessions_spawn attachment failure is also real, though its default-off behavior is documented and should be handled as diagnostics rather than silently changing the opt-in boundary.

## What This Changes

The issue asks OpenClaw to restore upgraded image-tool behavior by honoring explicit model overrides and adding doctor or migration guidance for missing imageModel and sessions_spawn attachment configuration.

## Best Possible Solution

Honor explicit image-tool model overrides without requiring global imageModel, then add doctor diagnostics for missing imageModel and attachment opt-in while preserving secure defaults.

## Reproduction Assessment

Yes, by source inspection: with `agents.defaults.imageModel` unset and no auto-resolved image provider candidate, the image tool throws before a qualified per-call `model` override can reach the fallback runner. I did not run a live gateway scenario because this review was constrained to read-only inspection.

## Solution Assessment

Yes for the narrow fix: the maintainable path is to let qualified explicit overrides construct the effective image model config before the missing-default error. Doctor should warn about missing defaults and attachment opt-in, but should not silently enable sessions_spawn attachments without maintainer approval.
