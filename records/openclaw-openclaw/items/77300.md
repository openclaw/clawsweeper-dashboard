## Summary

Keep open: current main still deletes the matching auth-profile `keyRef` for `models.providers.<provider>.apiKey` apply targets, and the linked closing PR remains open rather than merged.

## What This Changes

The issue asks `openclaw secrets apply` to preserve existing auth-profile SecretRef credentials when applying a `models.providers.<provider>.apiKey` plan with auth-profile scrub enabled.

## Best Possible Solution

Land #77329 or an equivalent narrow fix that preserves SecretRef-backed auth-profile refs while still scrubbing plaintext residues.

## Reproduction Assessment

Yes. Source inspection on current main shows the reported plan target enters provider auth-profile scrubbing and deletes `keyRef`; the current test suite also asserts that destructive post-state.

## Solution Assessment

Yes. Preserving `keyRef` or `tokenRef` when the auth-profile ref is already a SecretRef object is the narrow maintainable fix and aligns apply with audit's auth-profile precedence contract.
