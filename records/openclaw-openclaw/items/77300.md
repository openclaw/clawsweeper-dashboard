## Summary

Keep open: current main still sends `models.providers.<id>.apiKey` targets through auth-profile scrubbing and deletes the matching profile's `keyRef`; the linked fix PR #77329 is open but not merged.

## What This Changes

The report asks OpenClaw to preserve `auth-profiles.<provider>:default.keyRef` when applying `models.providers.<provider>.apiKey` SecretRef plans with auth-profile scrubbing enabled.

## Best Possible Solution

Preserve existing auth-profile SecretRef fields during provider API-key scrubbing, update the destructive test expectation with regression coverage, and land or replace PR #77329 after targeted secrets validation.

## Reproduction Assessment

Yes. Source inspection shows the provider API-key target is tracked for auth-profile scrubbing and the scrub loop deletes the matching profile ref field; the current test suite also asserts `keyRef` becomes undefined.

## Solution Assessment

Yes. The narrow maintainable fix is to make the scrub preserve canonical SecretRef objects, or refuse the no-op shadowed apply, rather than adding a new option that leaves the destructive default in place.
