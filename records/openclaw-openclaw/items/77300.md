## Summary

Keep open: current main still routes `models.providers.<id>.apiKey` targets into auth-profile scrubbing and deletes the matching profile ref field. The linked fix PR #77329 uses closing syntax and is still open, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Preserve `auth-profiles.<provider>:default.keyRef` when applying secrets plans that target `models.providers.<provider>.apiKey` with auth-profile scrubbing enabled.

## Best Possible Solution

Land PR #77329 or an equivalent narrow fix that preserves SecretRef-backed auth-profile refs, removes only plaintext credentials, and updates the regression expectation.

## Reproduction Assessment

Yes. From source inspection, a `models.providers.<id>.apiKey` target enters the provider scrub set, and the scrub loop deletes the matching auth profile's `keyRef`; the current test suite also asserts that destructive post-state.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve SecretRef objects in auth profiles during provider-target scrubbing, with regression coverage for the reported plan shape, rather than adding another option or changing the precedence model.
