## Summary

Keep open: current main still routes `models.providers.<id>.apiKey` targets into provider auth-profile scrubbing, and that scrub deletes the matching `keyRef`/`tokenRef` even though audit/docs say auth-profile credentials take precedence.

## What This Changes

Issue reports that `openclaw secrets apply` removes an existing `auth-profiles.<provider>:default.keyRef` when applying a `models.providers.<provider>.apiKey` SecretRef plan with auth-profile scrubbing enabled.

## Best Possible Solution

Preserve existing auth-profile SecretRef fields when applying provider API-key targets, scrub only actual plaintext residues, and add regression coverage that matches the reported `models.providers.<id>.apiKey` plan shape.

## Reproduction Assessment

Yes. Source inspection shows a `models.providers.<id>.apiKey` target is added to `providerTargets`, then `scrubAuthStoresForProviderTargets` deletes the matching profile's `keyRef`; I did not run the CLI because the review had to remain read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to align apply with the existing precedence contract by preserving authoritative auth-profile refs, or refusing the no-op/shadowed apply with a clear message instead of deleting the ref.
