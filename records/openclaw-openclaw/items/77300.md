## Summary

Keep open. Current `main` still has the destructive scrub path: a `models.providers.*.apiKey` plan target records the provider for auth-profile scrubbing, and the scrub then deletes both the plaintext field and the `keyRef`/`tokenRef` field from matching static auth profiles, contrary to the documented auth-profile precedence model.

## What This Changes

The issue asks for `secrets apply` to stop removing `auth-profiles.<provider>:default.keyRef` when applying a `models.providers.<provider>.apiKey` SecretRef plan with auth-profile scrubbing enabled.

## Best Possible Solution

Preserve existing auth-profile `keyRef`/`tokenRef` values for provider-target migrations, scrub only obsolete plaintext static fields, and add regression coverage plus a changelog entry for the credential-loss fix.

## Reproduction Assessment

Yes, from source inspection: a `models.providers.*.apiKey` target with the default scrub option reaches `scrubAuthStoresForProviderTargets`, which deletes the matching profile's ref field. I did not execute the CLI because this was a read-only review, but the current test expectation also confirms the path.

## Solution Assessment

No, current `main` is not the best way to solve it because the apply scrub contradicts the auth-profile precedence and ref-only support documented elsewhere. The narrower maintainable fix is to preserve canonical auth-profile refs while still normalizing the provider config target and removing migrated plaintext only.
