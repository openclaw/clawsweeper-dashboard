## Summary

Keep open: current main still deletes the matching auth profile ref field for `models.providers.<id>.apiKey` targets, and an open fix PR already references this issue with closing syntax, so the issue should wait for that PR to merge rather than be closed early.

## What This Changes

`secrets apply` should preserve `auth-profiles.<provider>:default.keyRef` when applying a `models.providers.<provider>.apiKey` SecretRef plan because auth profiles are the authoritative credential source.

## Best Possible Solution

Land or revise PR #77329 so provider API-key applies preserve existing auth-profile SecretRef objects while continuing to scrub plaintext credentials and covering the reported plan shape with regression tests.

## Reproduction Assessment

Yes. Source inspection shows `models.providers.*.apiKey` targets add the provider to the scrub set and the scrub loop deletes the matching auth profile `refField`; the current test suite also asserts that destructive post-state.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve authoritative auth-profile SecretRefs, or refuse a no-op shadowed apply, instead of adding another option that leaves the default destructive path active.
