## What This Changes

The issue requests that `secrets apply` preserve `auth-profiles.<provider>:default.keyRef` when applying a `models.providers.<provider>.apiKey` target.

## Best Possible Solution

Preserve existing auth-profile SecretRef fields when provider API-key targets are applied, with regression coverage for the reported plan shape and apply/audit/docs precedence kept aligned.

## Reproduction Assessment

Yes. Source inspection on current main shows a provider API-key target enters the provider scrub set and the scrub loop deletes the matching auth-profile `keyRef`; I did not run the CLI because this was a read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve existing auth-profile SecretRef objects during provider-target scrub or refuse the no-op, not to add another option that leaves the destructive default in place.
