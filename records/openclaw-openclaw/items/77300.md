## Summary

Keep open: current main still routes `models.providers.<id>.apiKey` apply targets into auth-profile scrubbing and deletes the matching auth profile `keyRef`, while audit/docs say auth-profile credentials take precedence.

## What This Changes

Prevent `openclaw secrets apply` from removing `auth-profiles.<provider>:default.keyRef` when applying `models.providers.<provider>.apiKey` targets.

## Best Possible Solution

Preserve existing auth-profile SecretRef fields when applying provider API-key targets, update the destructive test into a regression test, and keep apply/audit/docs aligned on auth-profile precedence.

## Reproduction Assessment

Yes. Source inspection shows a current-main reproduction path: a `models.providers.<id>.apiKey` target marks the provider for scrubbing, and the scrub loop deletes the matching auth profile `keyRef`.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve authoritative auth-profile refs, or refuse a no-op/shadowed apply with a clear message; adding another option would leave the default destructive path intact.
