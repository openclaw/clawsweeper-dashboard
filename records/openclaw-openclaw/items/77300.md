## Summary

Keep open: current main still routes `models.providers.<id>.apiKey` targets into auth-profile scrubbing and deletes the matching profile's `keyRef`, while audit/docs say auth-profile credentials take precedence.

## What This Changes

The issue reports that `secrets apply` can remove an existing provider auth-profile `keyRef` when applying a `models.providers.<id>.apiKey` SecretRef plan with auth-profile scrubbing enabled.

## Best Possible Solution

Preserve existing auth-profile SecretRef fields when applying provider API-key targets, add regression coverage for the reported plan shape, and keep auth-profile precedence aligned across apply, audit, and docs.

## Reproduction Assessment

Yes, from source inspection: a `models.providers.<id>.apiKey` target marks the provider for scrubbing, and the scrub loop deletes the matching auth profile's `keyRef`. I did not run the CLI because the review is read-only, but the current test suite also asserts the destructive post-state.

## Solution Assessment

Yes, the narrow maintainable fix is to change provider-target scrubbing so authoritative auth-profile refs are preserved or the no-op/ref-shadowed apply is refused with a clear message. Adding another scrub option would leave the default destructive path in place.
