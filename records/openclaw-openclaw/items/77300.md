Keep open: current main still deletes the matching auth-profile `keyRef` for `models.providers.<id>.apiKey` apply targets, and two open PRs already reference this issue with closing syntax, so the issue should stay open until one lands.

## What This Changes

The issue requests that `openclaw secrets apply` preserve an existing `auth-profiles.<provider>:default.keyRef` when applying a `models.providers.<provider>.apiKey` target.

## Best Possible Solution

Land one of the open fix PRs, or an equivalent narrow patch, that preserves authoritative auth-profile SecretRefs while keeping plaintext scrub behavior and adding regression coverage for the reported plan shape.

## Reproduction Assessment

Yes. Source inspection on current main shows a `models.providers.<id>.apiKey` target enters the provider scrub set and the scrub loop deletes the matching auth profile's `keyRef`; the current test suite also asserts that destructive post-state.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve SecretRef-backed `keyRef`/`tokenRef` fields, or refuse the no-op when an auth profile is already authoritative, while keeping plaintext credential scrubbing intact.
