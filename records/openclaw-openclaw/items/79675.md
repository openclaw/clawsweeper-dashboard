## Summary

Keep open: this is a fresh, detailed performance bug against the latest release, and current main still routes Control UI session switching through `sessions.list` and `chat.history` paths without a proven fix for the post-first-message Windows latency pattern.

## What This Changes

The issue reports persistent Control UI session-switching and `sessions.list` latency after the first chat message completes on Windows.

## Best Possible Solution

Profile current main or v2026.5.7 on Windows around first `chat.send` completion and subsequent `sessions.list`/`chat.history`, then land a targeted core fix in the gateway session-store or run-cleanup path with regression coverage or live smoke proof.

## Reproduction Assessment

No high-confidence live current-main reproduction is established. The reporter gives detailed v2026.5.7 Windows steps and source inspection traces the affected Control UI switch path to `sessions.list` and `chat.history`, but the post-first-message slowdown was not reproduced locally.

## Solution Assessment

Unclear because there is no proposed patch and the root cause is not isolated. The maintainable direction is profiling-backed core triage rather than closing as a duplicate of the broader `sessions.list` issue or applying another generic cache tweak.
