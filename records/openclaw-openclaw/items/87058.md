## Summary

Keep open: current main and the latest release still register empty effective node commands for unapproved or approved-empty node pairing surfaces, while Android still declares a non-empty node surface. The remaining fix is security-sensitive because it must choose or repair the Android device-role versus explicit node-surface approval boundary.

## What This Changes

The issue reports that an Android node paired through gateway onboarding connects with empty caps/commands and that onboarding recovery can race the connect challenge handshake.

## Best Possible Solution

Use a maintainer-approved Android setup or migration path that restores low-risk Android node commands without weakening explicit node-surface approval for non-Android and high-risk commands, then prove it with focused gateway tests and a real Android pairing run.

## Reproduction Assessment

Yes for the primary gateway behavior by source inspection: current main still converts unapproved or approved-empty node surfaces into empty effective live commands. The exact Android retry race was not live-reproduced, and the old PairingAutoRetryEffect file is no longer present on current main.

## Solution Assessment

Unclear: disabling a stale retry loop would not solve the primary empty-command surface, and auto-adopting Android declared commands needs a security/product decision. The safer direction is to repair the Android node-surface approval or migration path with explicit boundary tests.
