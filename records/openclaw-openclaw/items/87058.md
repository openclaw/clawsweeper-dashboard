## Summary

Keep open: current main still has the reported Android node command-surface collapse path, and the onboarding retry race is source-reproducible from the current timing constants. The fix is worth tracking, but the gateway part changes a node command approval boundary and needs maintainer direction rather than automatic cleanup.

## What This Changes

Fix Android node pairing so device-approved Android nodes retain an approved command surface, and prevent onboarding auto-retry from closing an in-flight connect-nonce handshake.

## Best Possible Solution

Preserve explicit node-surface approval for non-Android and high-risk commands, add a maintainer-approved Android setup/device migration path for the low-risk declared surface, and disable Finishing-state auto-retry while a connect challenge can still complete.

## Reproduction Assessment

Yes, by source inspection: current main still turns unapproved or approved-empty node surfaces into empty effective commands and stores those as the live node session command set; the Android retry timing also matches the reported nonce race. I did not run a live Android/Tailscale Serve scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the approval boundary: disabling Finishing auto-retry is a narrow fix, but auto-adopting Android declared commands needs explicit security/product approval or a safer node-surface approval/migration path.
