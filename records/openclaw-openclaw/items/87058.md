## Summary

Keep open: current main still keeps a first-time Android node's declared command surface non-effective until separate node-pairing approval, while the remaining Android approval-boundary choice is security-sensitive and needs maintainer direction.

## What This Changes

The issue requests a fix for Android nodes connecting with empty effective caps/commands after device pairing, plus avoiding an onboarding retry path that can close the connect-nonce handshake before it completes.

## Best Possible Solution

Have maintainers choose and document the Android approval model: either keep explicit node-surface approval with clear Android recovery UX, or narrowly authorize a baseline Android command surface with security tests and high-risk command gates preserved.

## Reproduction Assessment

Yes for the primary command-surface behavior by source inspection: Android declares commands, gateway reconciliation records them as pending, and live registration uses an empty effective surface until node approval. The full Android/Tailscale retry loop was not freshly live-reproduced against current main.

## Solution Assessment

Unclear: the proposed Android auto-adoption would address the symptom, but it is not clearly the best fix without maintainer approval-boundary direction. A safer path is either explicit approval/recovery UX or a narrowly documented Android baseline surface with high-risk command gates intact.
