## Label Justifications

- `P1`: A real Android companion-node workflow remains blocked because paired Android nodes can connect but expose no usable commands or Talk capability.
- `impact:session-state`: The issue is about live node session capability and command state collapsing to an empty effective surface while declarations exist separately.
- `impact:auth-provider`: The failure crosses device pairing, node pairing, role tokens, approval scopes, and connect handshake state.

## Summary

Keep open: current main and v2026.6.10 still keep Android node declarations separate from the effective command surface until node-surface approval, and fresh user evidence reports the same empty caps/commands behavior on gateway v2026.6.10 with the Play Store Android build. The remaining repair is real but security/product-sensitive because it must define whether Android device-role approval can promote a declared node surface or whether the explicit node approval/recovery path must be fixed.

## What This Changes

The issue asks OpenClaw to fix Android nodes that pair and connect but end up with empty effective commands/capabilities, plus an onboarding reconnect race around the connect-nonce handshake.

## Best Possible Solution

Define one Android approval model and implement it end to end: either make device-backed Android node approval intentionally authorize a narrow platform-allowlisted baseline while preserving high-risk command gates, or repair the explicit node-surface approval/recovery path so Android users can approve and promote the declared surface reliably.

## Reproduction Assessment

Yes at source level: Android declares caps/commands, gateway reconciliation can produce empty effective caps/commands, and node.invoke rejects an empty command session. I did not run a fresh live Android reproduction, but user comments report the same path on v2026.6.9 and v2026.6.10.

## Solution Assessment

Unclear: the reporter's auto-adopt sketch is plausible but not safely best without maintainer approval-boundary direction. The safer target is a documented Android trust model with regression tests proving high-risk commands and explicit node-surface approval still behave as intended.
