## Summary

Keep open: current main and the latest release still register an empty effective command surface until explicit node-pairing approval exists, and the remaining Android behavior is an approval/security boundary that needs maintainer direction rather than cleanup closure.

## What This Changes

The issue reports Android node connections losing their advertised commands and an onboarding reconnect race during the connect-challenge handshake.

## Best Possible Solution

Maintainers should choose and document one Android path: explicit recoverable node-surface approval after device pairing, or a narrowly tested Android baseline command adoption that preserves high-risk command gates.

## Reproduction Assessment

Yes for the primary gateway behavior by source inspection: a node connect with declared commands and no paired node record returns empty effective commands and registers that empty surface. I did not establish a fresh live Android reproduction for the retry-race half in this read-only review.

## Solution Assessment

Unclear: the proposed Android auto-adoption would fix the symptom, but it changes the documented approval boundary. A maintainable solution should either make node-surface approval explicit and recoverable in Android onboarding, or adopt only a maintainer-approved Android baseline with security tests.
