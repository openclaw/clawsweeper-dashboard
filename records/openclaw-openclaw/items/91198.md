## Summary

Keep open: the latest patch moves in the right direction, but the real Windows process-snapshot evidence path can still treat the pre-existing foreground gateway as proof that the Scheduled Task launched, and the PR still has only mocked Linux test proof for a Windows Task Scheduler handoff.

## What This Changes

The PR snapshots pre-existing Windows gateway listener PIDs before `schtasks /Run`, ignores those PIDs in listener-based launch evidence, terminates them before fallback launch, and adds startup-fallback regression tests.

## Best Possible Solution

Filter every Scheduled Task launch-evidence path, including Win32 process snapshots, through the pre-existing PID set, add a regression for that branch, then require redacted native Windows foreground-to-managed handoff proof before merge.

## Reproduction Assessment

Yes as a source path: current main can accept a pre-existing gateway listener as running evidence while schtasks remains not-yet-run, matching the linked report's foreground-window failure. I did not run a native Windows reproduction in this read-only review.

## Solution Assessment

No; the proposed direction is the right layer, but the PR is still incomplete because one Windows process-snapshot launch-evidence branch does not apply the same pre-existing PID filter.
