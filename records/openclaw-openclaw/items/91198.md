## Summary

Keep open: the current head addresses the previous false-positive launch-evidence path, but it also terminates every pre-existing listener PID returned by the broad port resolver before fallback, which can kill a non-OpenClaw process already bound to the configured gateway port.

## What This Changes

The PR changes Windows Scheduled Task launch verification to snapshot pre-existing listener PIDs, ignore them as launch evidence, terminate them before Startup fallback, and add startup-fallback regression tests.

## Best Possible Solution

Separate launch-evidence exclusion from killable gateway ownership: filter all pre-existing PIDs from evidence, but terminate only verified OpenClaw gateway PIDs or fail with port diagnostics, then capture native Windows handoff proof.

## Reproduction Assessment

Yes, at source level: current main can treat a foreground listener as Scheduled Task launch evidence and skip fallback. I did not run a native Windows Scheduled Task reproduction in this read-only review.

## Solution Assessment

No; the daemon-layer launch-evidence fix is the right shape, but the pre-fallback cleanup must be narrowed to verified gateway ownership before this is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Only terminate verified pre-existing gateway PIDs:** `src/daemon/schtasks.ts:1181-1182`
  - body: `preExistingListenerPids` is populated by `resolveScheduledTaskGatewayListenerPids`, and that helper falls back to returning every PID on a busy port when it cannot verify a gateway command line. On the fallback path this loop will terminate those PIDs before launching the Startup entry, so `openclaw gateway install` can kill an unrelated process occupying the configured port. Keep the broad set for launch-evidence filtering if needed, but only pass verified gateway PIDs to `terminateGatewayProcessTree` or fail with port diagnostics before fallback.
  - confidence: 0.88
