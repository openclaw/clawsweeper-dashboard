## Summary

Keep open: current main still has the source-level false launch-evidence path, but this PR is not merge-ready because it only proves that a fallback process is spawned while the foreground gateway still owns the port, not that a managed gateway survives after that foreground process exits.

## What This Changes

The PR snapshots gateway listener PIDs before `schtasks /Run`, ignores those pre-existing PIDs as launch evidence, bypasses listener-backed status during launch polling, and adds one startup-fallback regression test.

## Best Possible Solution

Make the Windows managed-start path either wait/defer until the foreground listener releases the port or return a clear actionable state, then prove on native Windows that the managed gateway remains reachable after the foreground window closes.

## Reproduction Assessment

Yes at source level: current main can count an already-running gateway listener as Scheduled Task launch evidence, matching the linked issue's status text and failed persistence symptom. I did not run a live native Windows Task Scheduler reproduction in this read-only review.

## Solution Assessment

No, not yet: filtering pre-existing listener PIDs is the right area, but spawning the fallback while that listener still owns the port does not prove or create a persistent managed gateway. A safer fix must handle the still-busy port or report an explicit retry/close-foreground action.
