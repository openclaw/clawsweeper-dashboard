## Summary

Keep open. Current main still has the reported provenance gap: outbound `message_sending` hooks can rewrite or cancel delivery text, but neither the event nor typed context includes the resolved runtime provider/model, even though the runner and final-reply layers know that identity. The issue remains security-sensitive and overlaps with open hook-contract PRs, so this needs maintainer architecture/security review rather than cleanup closure.

## What This Changes

The issue requests gateway-level runtime model/provider identity to be passed to or enforced by outbound delivery hooks so visible model tags cannot silently diverge from the model that generated a reply.

## Best Possible Solution

Define one backwards-compatible provenance contract for final outbound replies, then thread resolved provider/model identity from the agent runner through final reply assembly into the chosen hook surface. The fix should cover normal final replies, followups, CLI-delivered agent replies, and channel direct-delivery paths, with tests proving hooks can see truthful runtime identity after fallback without trusting model-authored text.

## Reproduction Assessment

Partially yes. Source inspection gives a high-confidence reproduction for the missing hook metadata: register a `message_sending` hook and the event cannot contain runtime provider/model because the type and call sites do not pass it; the exact live fallback sequence from the report is less certain on current main because session model override semantics are now documented as strict.

## Solution Assessment

No, the issue's suggested small metadata patch is probably not the best complete fix. The current code shows model identity lives in the runner/final-reply layer while `message_sending` is a lower delivery layer, so a safer solution is a coordinated provenance hook or additive message context contract that behaves consistently across reply paths.
