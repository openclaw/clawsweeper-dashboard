## Summary

Keep this open: current main and v2026.6.8 still do not add authoritative provider/model identity to `message_sending` or enforce truthful model self-tags by default; `reply_payload_sending.usageState` is a useful partial mitigation on a separate hook contract.

## What This Changes

The issue asks for gateway-authoritative provider/model identity to reach outbound delivery hooks so forged model self-tags can be detected, corrected, blocked, or alerted on.

## Best Possible Solution

Use the existing reply-payload provenance path or an approved successor as the canonical model identity contract, then add maintainer-approved default enforcement or opt-in policy instead of a small ad hoc message_sending metadata patch.

## Reproduction Assessment

Yes by source inspection for the remaining gap: message_sending still lacks provider/model identity and delivery does not enforce self-tag truthfulness. I did not run the original live Telegram fallback flow, and current model override behavior may change that exact trigger.

## Solution Assessment

No for the tiny metadata patch alone; current main has a richer reply-payload provenance hook, so the maintainable fix is a maintainer-approved provenance and enforcement contract rather than ad hoc expansion of one hook.
