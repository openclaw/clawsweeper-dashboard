## Summary

Keep open: current main and v2026.5.7 still leave outbound hooks without resolved runtime provider/model provenance, and the remaining work is a security-sensitive public hook/enforcement contract rather than cleanup.

## What This Changes

Request gateway-level runtime model provenance for outbound delivery so self-tagged model claims can be verified, corrected, blocked, or alerted on mismatch.

## Best Possible Solution

Define one backward-compatible provenance contract for final outbound replies, then thread resolved provider/model identity from the runner into the chosen hook or enforcement surface with fallback and direct-delivery coverage.

## Reproduction Assessment

Yes for the core metadata gap: current source shows message_sending hooks cannot receive resolved provider/model identity while runReplyAgent computes it upstream. The exact live deprecated-model fallback sequence was not rerun on current main.

## Solution Assessment

No for the issue's delivery-only metadata patch as a complete fix. The maintainable solution needs a coordinated public provenance or final-reply contract because truthful identity is computed upstream and multiple delivery paths invoke hooks.
