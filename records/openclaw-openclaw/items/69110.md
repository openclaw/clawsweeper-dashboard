## Summary

Keep open: current main still does not pass resolved runtime provider/model provenance to outbound `message_sending` hooks or an equivalent final-reply hook, and the remaining fix is a security-sensitive public hook contract decision already overlapping with open hook PRs.

## What This Changes

Expose truthful runtime provider/model identity to outbound or final-reply delivery hooks so self-tagged model labels can be verified, corrected, blocked, or audited.

## Best Possible Solution

Define one backward-compatible provenance contract for finalized outbound replies, thread resolved provider/model identity from the runner through every final reply and direct-delivery hook path, then document and test the contract across fallback and non-fallback cases.

## Reproduction Assessment

Yes for the missing metadata path: source inspection and current tests show `message_sending` receives content, target, and correlation fields but no provider/model. The exact fallback-from-legacy-override sequence is unclear on current main because those overrides are now strict, visible failures.

## Solution Assessment

No for the issue’s suggested small delivery-only metadata patch as a complete fix. The truthful identity is computed upstream of delivery and multiple hook paths can send final replies, so the maintainable fix is a coordinated additive provenance contract or final-reply hook.
