## Summary

Keep open: current main and the latest release still leave outbound `message_sending` hooks without resolved runtime provider/model provenance, and the remaining work is a security-sensitive public hook/enforcement contract rather than conservative cleanup.

## What This Changes

The issue asks OpenClaw to pass actual runtime provider/model identity into outbound delivery hooks so plugins can detect or prevent forged model self-tags.

## Best Possible Solution

Define one backward-compatible provenance contract for final outbound replies, then thread resolved provider/model identity from the runner through final reply assembly into the chosen public hook or enforcement surface with tests for fallback and direct delivery paths.

## Reproduction Assessment

Yes for the core metadata gap: current source shows `message_sending` hooks cannot receive runtime provider/model because the event type and call sites never pass it. The exact live deprecated-model fallback sequence is not high-confidence on current main because current model-failover docs treat user and legacy session overrides as strict.

## Solution Assessment

No for the issue's suggested delivery-only metadata patch as a complete fix. The maintainable solution needs a coordinated public provenance/final-reply contract because truthful identity is computed upstream and multiple delivery paths invoke hooks.
