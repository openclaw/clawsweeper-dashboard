- `P2`: The issue is a normal-priority security-sensitive hook/provenance gap with real impact but it needs API/product coordination rather than an emergency patch.
- `impact:security`: Outbound policy hooks cannot currently validate model identity claims against the actual runtime model used for a response.
- `impact:auth-provider`: The missing data is the resolved provider/model routing identity that produced the outbound message.

## Summary

Keep open: current main still does not expose resolved runtime provider/model provenance to outbound `message_sending` hooks, and the related final-reply/provenance work is still open. The exact deprecated `modelOverride` fallback path is now better guarded for user/legacy selections, but the hook/enforcement gap remains a real security-sensitive contract decision.

## What This Changes

The issue asks for resolved runtime provider/model identity to reach outbound delivery hooks so model self-tags can be verified or corrected.

## Best Possible Solution

Define one backward-compatible outbound provenance contract, then thread resolved provider/model identity from the runner into the chosen hook or enforcement surface with docs, security review, and focused coverage.

## Reproduction Assessment

Yes for the source-visible metadata gap: current main calls `runMessageSending` without provider/model fields while `runReplyAgent` computes `providerUsed` and `modelUsed` upstream. The exact live deprecated-model Telegram fallback sequence was not rerun and appears partially mitigated by stricter user/legacy override fallback policy.

## Solution Assessment

No for the issue's delivery-only metadata patch as a complete fix. The maintainable solution needs a coordinated public provenance or final-reply contract because truthful identity is computed upstream and multiple delivery paths invoke hooks.
