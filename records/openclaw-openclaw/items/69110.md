## Summary

Keep open: current main and latest release still do not give `message_sending` authoritative provider/model identity or enforce truthful model self-tags; the newer `reply_payload_sending.usageState` path is a useful partial mitigation, not a default trust boundary.

## What This Changes

Request to expose authoritative runtime provider/model identity to outbound delivery hooks and define enforcement for mismatched model self-tags.

## Best Possible Solution

Adopt one maintainer-approved provenance and enforcement contract, likely through `reply_payload_sending.usageState` or its successor, then document default behavior and hook guarantees instead of adding ad hoc metadata to one hook.

## Reproduction Assessment

Yes for the source-level gap: current main shows `message_sending` receives no provider/model identity and delivery does not verify self-tags. I did not live-rerun the original Telegram fallback flow.

## Solution Assessment

No for a tiny metadata-only patch. The best fix is a maintainer-approved provenance and enforcement contract that avoids creating duplicate hook semantics beside `reply_payload_sending.usageState`.
