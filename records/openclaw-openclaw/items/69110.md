## Label Justifications

- `P2`: The issue is a bounded but security-sensitive provenance gap in outbound model attribution rather than an emergency runtime outage.
- `impact:security`: The report concerns whether outbound messages can claim a model identity that the runtime did not verify at send time.
- `impact:auth-provider`: The mismatch depends on provider/model routing and fallback identity not being available to delivery hooks.

## Summary

Keep open: current main still does not pass authoritative runtime provider/model identity into the outbound `message_sending` enforcement point. The exact legacy `modelOverride` fallback path is now stricter on main, but the remaining security-sensitive provenance/API decision is real and overlaps the broader message/reply hook work.

## What This Changes

The issue asks for outbound delivery hooks to receive or enforce the actual runtime provider/model identity so agent-visible model tags cannot silently differ from the model that produced the reply.

## Best Possible Solution

Define one authoritative reply provenance contract that carries run id plus resolved provider/model from the agent attempt into final reply and `message_sending`/`message_sent` hooks, then decide any enforcement or alert behavior as a maintainer-owned security policy.

## Reproduction Assessment

Yes for the remaining problem: current source shows `message_sending` lacks runtime provider/model fields while separate LLM/model hooks carry that identity. I did not run the live Telegram scenario in this read-only review, and the exact legacy override fallback path appears narrower on current main.

## Solution Assessment

Unclear as an implementation: adding a few metadata fields is plausible, but the best fix needs maintainer agreement on the durable provenance contract and whether send-time enforcement should block, rewrite, or only alert.
