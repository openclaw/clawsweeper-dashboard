## Summary

Keep open: current main still exposes `message_sending` delivery hooks without resolved runtime provider/model provenance, while the reply runner already computes that identity upstream. The remaining work is a security-sensitive public hook/enforcement contract with open implementation candidates, not a cleanup-close case.

## What This Changes

The issue asks OpenClaw to expose or enforce actual runtime provider/model provenance during outbound delivery so agent-authored model tags cannot silently disagree with the model that produced the reply.

## Best Possible Solution

Define one backward-compatible runtime provenance contract for final outbound replies, then thread resolved provider/model identity from the runner into the chosen hook or enforcement surface with docs and focused coverage.

## Reproduction Assessment

Yes for the source-visible metadata gap: current main calls `runMessageSending` without provider/model fields while `runReplyAgent` computes `providerUsed` and `modelUsed` upstream. The exact live deprecated-model Telegram fallback sequence was not rerun.

## Solution Assessment

No for the issue's delivery-only metadata patch as a complete fix. The maintainable solution needs a coordinated public provenance or final-reply contract because truthful identity is computed upstream and multiple delivery paths invoke hooks.
