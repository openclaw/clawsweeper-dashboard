## Summary

Keep this open: current main and v2026.6.8 still do not pass authoritative provider/model identity to `message_sending` or enforce truthful model self-tags by default; `reply_payload_sending.usageState` is a useful but separate partial mitigation that needs security/product direction before becoming the enforcement surface.

## What This Changes

The issue asks OpenClaw to make actual runtime provider/model identity available at outbound delivery time so forged or stale model self-tags can be detected, corrected, blocked, or alerted.

## Best Possible Solution

Use an approved single provenance contract, likely the existing reply-payload provenance path or a successor, then add maintainer-approved default enforcement or explicit policy for model self-tags rather than an ad hoc metadata-only expansion of one hook.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear reproduction for the remaining gap: `message_sending` lacks provider/model identity and delivery does not enforce self-tag truthfulness; I did not run the original live Telegram fallback flow.

## Solution Assessment

Is this the best way to solve the issue? No for a tiny `message_sending` metadata patch alone; the maintainable fix is a maintainer-approved provenance and enforcement contract that avoids competing hook semantics.
