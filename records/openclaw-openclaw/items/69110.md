## Summary

Keep open: current main still leaves outbound `message_sending` hooks without resolved runtime provider/model provenance, and the remaining fix is a public hook/security contract decision rather than cleanup closure.

## What This Changes

Add truthful gateway-level runtime model/provider provenance to outbound delivery hook metadata so self-authored model tags can be verified or corrected.

## Best Possible Solution

Define one backwards-compatible provenance contract for final outbound replies, then thread resolved provider/model identity from the runner through final reply assembly into the chosen hook surface with coverage for final replies, followups, CLI/direct delivery paths, and fallback cases.

## Reproduction Assessment

Yes for the missing metadata path: source inspection shows a registered `message_sending` hook cannot receive runtime provider/model because the event type and current call sites never pass it. The exact live fallback sequence from the report is less certain on current main because user and legacy session overrides are documented and implemented as strict selections.

## Solution Assessment

No for the issue's suggested small delivery-only metadata patch as a complete solution. The truthful identity is computed upstream of delivery and multiple reply paths invoke hooks, so maintainers should choose a coordinated additive provenance hook or message context contract.
