## Summary

Keep open: current main still does not expose resolved runtime provider/model provenance to the outbound `message_sending` hook, and the related final-reply/provenance work is still open. The exact stale `modelOverride` fallback path appears better guarded now, but the reported hook/enforcement gap remains a real security-sensitive contract decision.

## What This Changes

The issue requests gateway-level runtime provider/model provenance or enforcement for outbound replies so model self-tags cannot silently diverge from the actual model used.

## Best Possible Solution

Define one backward-compatible final outbound provenance contract, then thread resolved provider/model identity from the runner into the chosen hook or enforcement surface with docs and focused coverage.

## Reproduction Assessment

Yes for the source-visible metadata gap: current main calls `runMessageSending` without provider/model fields while `runReplyAgent` computes `providerUsed` and `modelUsed` upstream. The exact live deprecated-model Telegram fallback sequence was not rerun.

## Solution Assessment

No for the issue's delivery-only metadata patch as a complete fix. The maintainable solution needs a coordinated public provenance or final-reply contract because truthful identity is computed upstream and multiple delivery paths invoke hooks.
