## Summary

Keep open: current main still does not pass authoritative provider/model identity to `message_sending` or enforce truthful model self-tags; the newer `reply_payload_sending.usageState` path is useful context, but it is a separate hook and not a default trust boundary.

## What This Changes

Requests authoritative gateway model/provider identity in outbound delivery hooks and enforcement against mismatched self-tagged model labels.

## Best Possible Solution

Define one maintainer-approved provenance and pre-send enforcement boundary that carries requested and resolved model identity through outbound delivery, then document and test the plugin semantics across live reply, durable/replay, and fallback paths.

## Reproduction Assessment

Yes for the core source-level gap: current main’s `message_sending` event and context do not carry provider/model identity. No live reproduction of the reported Telegram fallback/self-tag forgery was run in this read-only review.

## Solution Assessment

Unclear: adding a few metadata fields is plausible but likely not the best complete fix. The safer solution is a maintainer-chosen provenance/enforcement API that does not split policy between `message_sending`, `reply_payload_sending`, and broader outbound delivery.
