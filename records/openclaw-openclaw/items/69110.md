## Summary

Keep open: current main still leaves outbound `message_sending` hooks without resolved runtime provider/model provenance, and the remaining fix is a security-sensitive public hook/enforcement contract rather than cleanup closure.

## What This Changes

The issue asks for resolved runtime provider/model identity to reach outbound delivery hooks so model self-tags can be verified against the actual model used.

## Best Possible Solution

Define one backward-compatible provenance contract for final outbound replies, then thread resolved provider/model identity from the runner through final reply assembly into the chosen hook/enforcement surface with coverage for fallback and all reply paths.

## Reproduction Assessment

Yes for the missing metadata path: current source shows registered `message_sending` hooks cannot receive runtime provider/model because the event type and call sites never pass it. The exact live fallback sequence from the report is less certain on current main because user and legacy session overrides are now documented as strict selections.

## Solution Assessment

No for the issue's suggested small delivery-only metadata patch as a complete solution. The truthful identity is computed upstream and multiple reply paths invoke hooks, so maintainers should choose a coordinated additive provenance hook or message context contract.
