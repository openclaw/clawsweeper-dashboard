## Summary

Keep open. Current `main` still launches bundle LSP stdio servers through a direct `spawn(config.command, ...)`, and the linked closing PR #75343 is the active unmerged implementation path rather than a reason to close this issue early.

## What This Changes

The issue requests fixing Windows LSP plugin startup by resolving npm `.cmd` shims before spawning configured LSP server commands.

## Best Possible Solution

Review and land PR #75343, or a narrow replacement, so bundle LSP spawning uses the shared Windows resolver with the same final child environment used for launch, adds focused LSP runtime coverage for `.cmd` shim resolution, and includes the required user-facing changelog entry.

## Reproduction Assessment

Yes. The issue gives a concrete Windows 11 `typescript-language-server` setup, and current main statically matches the failure path because configured bundle LSP stdio servers still launch through direct `spawn(config.command, ...)` without Windows `.cmd` resolution.

## Solution Assessment

Yes. Reusing `resolveWindowsSpawnProgram()` is the narrow maintainable direction because it avoids broad `shell: true` behavior and matches adjacent process-launch seams already present in core.
