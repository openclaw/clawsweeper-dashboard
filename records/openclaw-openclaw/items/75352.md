## Summary

Keep open. Current main still launches bundled LSP stdio servers with a direct `spawn(config.command, ...)`, while the explicitly linked fix PR #75343 is open and unmerged, so this issue should remain open until that PR or a narrow replacement lands.

## What This Changes

The issue requests that bundled LSP server startup on Windows resolve npm `.cmd` shims before spawning commands such as `typescript-language-server`.

## Best Possible Solution

Review and land PR #75343, or a narrow replacement, so bundled LSP spawning uses the shared Windows resolver with the final child environment, includes focused LSP runtime coverage, and adds the required user-facing changelog entry.

## Reproduction Assessment

Yes. The issue gives a concrete Windows 11 `typescript-language-server` setup, and current main statically matches the failure path because `.lsp.json` stdio servers still launch through direct `spawn(config.command, ...)` without Windows `.cmd` resolution.

## Solution Assessment

Yes, the shared `resolveWindowsSpawnProgram()` seam is the narrow maintainable direction instead of `shell: true`; the implementation should pass the merged launch env into resolution and spawn, then lock the behavior with targeted tests.
