## Summary

Keep open. Current main still launches bundled LSP stdio servers with a direct bare-command spawn, and the linked closing PR #75343 is still documented as open and unmerged in the provided GitHub context, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Request that bundled LSP stdio server launches on Windows resolve npm `.cmd` shims through the shared Windows spawn helper instead of spawning the bare command directly.

## Best Possible Solution

Review and land PR #75343, or a narrow replacement, wiring `resolveWindowsSpawnProgram()` plus `materializeWindowsSpawnProgram()` into the bundled LSP spawn path with focused Windows shim coverage and the required user-facing changelog entry.

## Reproduction Assessment

Yes. The report gives a concrete Windows 11 `typescript-language-server` setup, and current main statically matches the failure path because LSP stdio servers still launch through direct `spawn(config.command, ...)` without Windows `.cmd` resolution.

## Solution Assessment

Yes. Reusing the shared Windows spawn resolver is the narrow maintainable fix and avoids broad `shell: true`; the implementation should pass the final child environment into resolution and spawn, then lock it with targeted tests.
