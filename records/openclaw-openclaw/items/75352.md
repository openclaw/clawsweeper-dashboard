## Summary

Keep open. Current main still launches bundled LSP stdio servers with a direct `spawn(config.command, ...)`, and the explicitly linked fix PR #75343 is open and unmerged, so this issue should remain open until that PR or a narrow equivalent lands.

## What This Changes

The issue requests Windows-safe LSP server spawning so npm-installed `.cmd` shims such as `typescript-language-server.cmd` resolve correctly for bundled LSP plugins.

## Best Possible Solution

Review and land PR #75343, or a narrow replacement, so bundled LSP spawning uses the shared Windows resolver, preserves the final child environment during resolution and spawn, adds focused LSP runtime coverage, and keeps the user-facing changelog entry.

## Reproduction Assessment

Yes. The issue gives a concrete Windows 11 setup with `typescript-language-server`, current main statically reaches direct `spawn(config.command, ...)`, and the Node 22.14.0 docs confirm `.cmd` files are not executable on their own on Windows.

## Solution Assessment

Yes as a direction: reusing the existing shared Windows spawn resolver is the narrow maintainable fix instead of switching the whole LSP path to unconditional shell execution. The open PR still needs normal review around env propagation, tests, and fallback policy before this issue can close.
