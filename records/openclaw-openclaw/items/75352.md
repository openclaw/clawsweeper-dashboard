Keep open. Current main still launches bundled LSP stdio servers by directly spawning the configured command, and the linked closing PR is open and unmerged, so this issue should stay open until that fix path lands. https://github.com/openclaw/openclaw/issues/75352

## What This Changes

The issue requests Windows-safe resolution of npm `.cmd` shims before bundled LSP stdio servers spawn commands such as `typescript-language-server`.

## Best Possible Solution

Land PR #75343 or a narrow equivalent that reuses `resolveWindowsSpawnProgram` and `materializeWindowsSpawnProgram` in the bundled LSP spawn path, resolves against the final child environment, adds focused Windows-wrapper coverage for the LSP runtime, and includes the required user-facing changelog entry.

## Reproduction Assessment

Yes. The report gives a concrete Windows 11, Node 22.14.0, and `typescript-language-server` setup, and current main still statically reaches a direct spawn of the configured LSP command with no Windows wrapper resolution.

## Solution Assessment

Yes as a direction. Reusing the existing shared Windows spawn resolver is the narrow maintainable fix; broad shell execution would be riskier and less consistent with adjacent hardened spawn paths.
