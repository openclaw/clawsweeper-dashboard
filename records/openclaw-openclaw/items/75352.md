## Summary

Keep open. Current main still launches bundle LSP servers by directly spawning the configured command, while the linked fix PR is open, unmerged, and explicitly paired with this issue, so the issue should remain open until that PR or a narrow replacement lands.

## What This Changes

The issue requests that bundled LSP server startup on Windows resolve npm `.cmd` shims instead of directly spawning the configured LSP command.

## Best Possible Solution

Update, review, and land PR #75343 or a narrow replacement so LSP spawning uses the shared Windows resolver with the same intentionally supported child environment, focused Windows `.cmd` runtime coverage, and a valid changelog entry; then let the issue close through the merge.

## Reproduction Assessment

Yes. The report gives a concrete Windows 11 setup with `typescript-language-server`, and current main statically follows the failure path by spawning the bare LSP command without Windows PATH/PATHEXT wrapper resolution.

## Solution Assessment

Yes in direction. Reusing the shared Windows resolver at the LSP spawn boundary is the narrow maintainable fix, but the active PR still needs mergeability and spawn-environment review before it is the best landed solution.
