## Summary

Keep open. Current main still sends configured bundled LSP commands directly to `child_process.spawn`, and the linked closing PR #75343 is open and unmerged, so this issue remains valid until that fix or a narrow equivalent lands.

## What This Changes

Route bundled LSP server startup through Windows command-shim resolution so npm-installed `.cmd` LSP servers work on Windows.

## Best Possible Solution

Review and land PR #75343 or an equivalent focused change that reuses the shared Windows spawn resolver in the bundled LSP spawn path with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. A Windows 11 setup with Node 22 and globally installed `typescript-language-server` reaches the current raw `spawn("typescript-language-server", ...)` path from a supported LSP bundle config; this review verified the source path rather than running live Windows.

## Solution Assessment

Yes as a direction. Reusing `resolveWindowsSpawnProgram` and `materializeWindowsSpawnProgram` is the narrow maintainable fix because the resolver already handles Windows npm `.cmd` wrappers and is used by an adjacent spawn surface.
