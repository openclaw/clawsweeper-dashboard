## Summary

Keep open. Current main still launches bundled LSP stdio servers with a direct child_process spawn of the configured command, while the linked closing PR #75343 is open and unmerged, so the reported Windows .cmd shim failure still needs the active fix path to land.

## What This Changes

The issue requests wiring bundled LSP server startup through the shared Windows spawn resolver so npm .cmd shims such as typescript-language-server.cmd resolve correctly on Windows.

## Best Possible Solution

Land PR #75343 or a narrow equivalent that reuses resolveWindowsSpawnProgram/materializeWindowsSpawnProgram in the bundled LSP spawn path, resolves against the final child environment, adds focused Windows-wrapper coverage for the LSP runtime, and includes the required user-facing changelog entry.

## Reproduction Assessment

Yes. The issue gives a concrete Windows 11, Node 22.14.0, and typescript-language-server setup, and current main still statically reaches a direct spawn of the configured LSP command with no Windows wrapper resolution.

## Solution Assessment

Yes as a direction. Reusing the existing shared Windows spawn resolver is the narrow maintainable fix; switching the LSP path to broad shell execution would be riskier and less consistent with the hardened resolver policy already used elsewhere.
