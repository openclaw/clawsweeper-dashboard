## Summary

Keep open. Current main still starts bundled LSP stdio servers through raw `child_process.spawn`, and PR #75343 is an open, unmerged closing PR for the same Windows `.cmd` shim failure.

## What This Changes

Fix Windows LSP server startup so npm-installed `.cmd` shims such as `typescript-language-server.cmd` are resolved before spawning stdio LSP processes.

## Best Possible Solution

Review and land PR #75343 or a narrow equivalent that reuses the shared Windows spawn resolver in the LSP spawn path, with focused coverage and the required changelog entry.

## Reproduction Assessment

Yes. The issue gives a concrete Windows 11, Node 22.14.0, and `typescript-language-server` setup, and current main statically reaches the raw spawn path that matches the reported `.cmd` shim failure.

## Solution Assessment

Yes as a direction. Reusing the existing Windows resolver is the narrow maintainable fix; this issue should remain open until the linked implementation merges or is replaced by an equivalent fix.
