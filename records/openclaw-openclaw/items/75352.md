## Summary

Keep open. Current main still runs bundled LSP server commands through a direct `child_process.spawn(config.command, ...)` path, while the explicitly linked fix PR #75343 remains open and unmerged, so the issue should stay open until that PR or a narrow replacement lands.

## What This Changes

The issue requests wiring Windows `.cmd` shim resolution into the bundled LSP server spawn path so npm-installed language servers can start on native Windows.

## Best Possible Solution

Review and land PR #75343, or a narrow replacement, so bundled LSP spawning uses the shared Windows resolver with the final child environment, includes targeted LSP runtime coverage, and adds any required user-facing changelog entry.

## Reproduction Assessment

Yes. The issue provides a concrete Windows 11 `typescript-language-server` failure, and current main statically matches the failure path because bundle LSP stdio servers still spawn the bare command without Windows `.cmd` resolution.

## Solution Assessment

Yes. Reusing the shared `resolveWindowsSpawnProgram()`/materialization path is the narrow maintainable fix; using broad `shell: true` would be a weaker default for command execution.
