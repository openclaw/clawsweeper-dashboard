## Summary

Keep open. Current main still starts bundled LSP stdio servers by passing the configured command directly to child_process.spawn, while the linked closing PR #75343 is still the active unmerged fix path, so this issue should remain open until that PR or an equivalent patch lands.

## What This Changes

The issue requests that bundled LSP server startup on Windows resolve npm .cmd shims before spawning language server commands such as typescript-language-server.

## Best Possible Solution

Land PR #75343 or a narrow equivalent that reuses the shared Windows spawn resolver in the bundled LSP spawn path, resolves against the final child environment, adds focused Windows-wrapper coverage for LSP startup, and includes the required user-facing changelog entry.

## Reproduction Assessment

Yes. The issue gives a concrete Windows 11 and Node 22.14.0 failure path, Node documents the .cmd/.bat spawning limitation on Windows, and current main statically reaches direct spawn of the configured LSP command.

## Solution Assessment

Yes as a direction. Reusing resolveWindowsSpawnProgram/materializeWindowsSpawnProgram is the narrow maintainable fix because it matches existing hardened spawn policy, while broad shell execution would add avoidable command-execution risk.
