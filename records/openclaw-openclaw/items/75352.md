## Summary

Keep open. Current main still directly spawns the configured bundled LSP command on Windows, and the linked closing PR #75343 is open and unmerged, so this issue should remain the canonical user report until that fix or a narrow equivalent lands.

## What This Changes

Fix Windows LSP server startup by resolving npm `.cmd` shims before spawning configured stdio LSP commands.

## Best Possible Solution

Land PR #75343 or a narrow equivalent that uses the shared Windows spawn resolver in the bundled LSP spawn path, adds focused Windows shim coverage, and includes the required user-facing changelog entry.

## Reproduction Assessment

Yes. The report gives a concrete Windows 11, Node 22.14.0, and `typescript-language-server` setup, and current main still statically reaches a direct spawn of the configured LSP command without wrapper resolution.

## Solution Assessment

Yes as a direction. Reusing the existing Windows spawn resolver is the narrow maintainable fix; switching the LSP path broadly to shell execution would be less consistent with adjacent hardened spawn paths.
