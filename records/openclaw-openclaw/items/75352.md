## Summary

Keep open. Current main still direct-spawns bundled LSP server commands, and the open closing PR #75343 has not merged, so this issue remains valid and should keep tracking that fix path.

## What This Changes

Request Windows LSP server startup to resolve npm `.cmd` shims via the shared Windows spawn resolver instead of directly spawning bare commands.

## Best Possible Solution

Land PR #75343 or a narrow equivalent that resolves LSP spawn invocations against the final child env, uses the shared Windows resolver, adds Windows shim coverage, and includes a user-facing changelog entry.

## Reproduction Assessment

Yes. The report gives a concrete Windows 11, Node 22.14.0, and `typescript-language-server` setup, and current main still routes that command directly to `child_process.spawn` without Windows wrapper resolution.

## Solution Assessment

Yes as a direction, but not yet landed. Reusing `resolveWindowsSpawnProgram` and `materializeWindowsSpawnProgram` is the narrow maintainable fix; broad shell execution should be avoided unless an explicit fallback policy is reviewed.
