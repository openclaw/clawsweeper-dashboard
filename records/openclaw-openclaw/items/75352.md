## Summary

Keep open. Current main still launches bundled LSP stdio servers through a raw child-process spawn, and the linked closing PR is still open and unmerged, so this issue should remain open until that fix or an equivalent lands.

## What This Changes

The issue reports Windows LSP server startup failures because the bundled LSP runtime spawns npm CLI shim names directly instead of resolving `.cmd` wrappers.

## Best Possible Solution

Review and land PR #75343, or a narrow equivalent, that reuses the shared Windows spawn resolver in the LSP spawn path with focused coverage and any required changelog entry.

## Reproduction Assessment

Yes. The issue gives a concrete Windows 11, Node 22.14.0, and `typescript-language-server` setup, and current main statically reaches the raw spawn path that matches the reported `.cmd` shim failure.

## Solution Assessment

Yes as a direction. Reusing the existing Windows resolver is the narrow maintainable fix; the linked PR review should scrutinize the exact shell fallback policy and test coverage before merge.
