## Summary

Keep open. Current main still launches bundled LSP stdio servers with a direct `spawn(config.command, ...)`, while the linked closing PR remains the active implementation path and should land before this issue closes.

## What This Changes

The issue asks OpenClaw to resolve Windows npm `.cmd` shims for bundled LSP server commands by wiring the existing Windows spawn helper into the LSP runtime.

## Best Possible Solution

The desired end state is a narrow LSP-runtime fix that resolves the final child environment through the shared Windows spawn helper, preserves non-Windows behavior, adds focused Windows shim coverage for bundled LSP startup, and keeps the user-facing changelog entry.

## Reproduction Assessment

Yes. The report gives a concrete Windows 11 `typescript-language-server` setup, and current main statically routes LSP startup through direct `spawn(config.command, ...)` while the test locks that direct command shape.

## Solution Assessment

Yes as a direction. Reusing the existing Windows spawn resolver is narrower and safer than switching the LSP path to unconditional shell execution, though the open PR still needs normal review before merge.
