## Summary

Keep open. Current main still launches bundled LSP stdio servers by directly spawning the configured command, and the issue is already paired with open, unmerged PR #75343, so the terminal path is to review that PR rather than close this issue or queue a duplicate repair.

## What This Changes

The issue requests resolving Windows npm `.cmd` shims for bundled LSP server startup by reusing the shared Windows spawn resolver before spawning stdio LSP servers.

## Best Possible Solution

Land PR #75343 or a narrow equivalent that resolves the LSP command against the final child environment, materializes the Windows spawn invocation before `spawn`, preserves the fail-closed wrapper policy unless maintainers explicitly accept shell fallback, adds focused LSP runtime coverage, and includes the required user-facing changelog entry.

## Reproduction Assessment

Yes. The report gives a concrete Windows 11 and Node 22.14.0 failure with `typescript-language-server`, and current main statically reaches a direct `spawn` of that configured command without Windows wrapper resolution.

## Solution Assessment

Yes as a direction. Reusing the existing Windows spawn resolver is the narrow maintainable fix; broad shell execution would be a less consistent command-execution policy for this surface.
