## Summary

Keep open: current main still does not propagate configured provider `request.allowPrivateNetwork` into the OpenAI-compatible memory embedding SSRF policy, and two open PRs are already linked as candidate fixes. This is a security-boundary/product decision, not a conservative cleanup close.

## What This Changes

The issue requests an explicit, SSRF-safe trust path for memory-core OpenAI-compatible embedding endpoints on private/internal networks.

## Best Possible Solution

Keep this issue open until maintainers review and land one canonical fix, preferably provider-policy inheritance if they accept that memory embeddings should reuse the existing provider transport trust contract.

## Reproduction Assessment

Yes for the source-level policy gap: current main routes configured OpenAI-compatible memory providers into the generic adapter, and that adapter does not consume `request.allowPrivateNetwork`. I did not run a live private endpoint repro in this read-only review, and ordinary exact-host private endpoints already appear narrower-trusted on current main.

## Solution Assessment

Unclear until maintainers make the security/product call. Reusing existing provider `request.allowPrivateNetwork` appears narrower than adding a second memory-only knob, but it needs maintainer acceptance because it extends that trust policy to memory embeddings.
