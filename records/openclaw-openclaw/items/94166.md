## Summary

Keep open: current main and v2026.6.9 still build the OpenAI-compatible memory embedding SSRF policy only from the configured embedding base URL, while the open candidate fix at https://github.com/openclaw/openclaw/pull/94344 is the active path for the remaining provider-policy/security decision.

## What This Changes

Request a supported explicit trust path so memory-core OpenAI-compatible embedding requests can use operator-configured private/internal endpoints without relaxing SSRF protection by default.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/94344; if accepted, keep provider-level request.allowPrivateNetwork as the canonical memory embedding trust surface with strict defaults and omitted/false/true tests.

## Reproduction Assessment

Yes, for the source-level policy gap: current main resolves configured OpenAI-compatible provider ids into the generic embedding adapter, but that adapter does not consume provider request.allowPrivateNetwork. I did not run a live private endpoint repro in this read-only pass.

## Solution Assessment

Unclear until maintainer/security acceptance, but https://github.com/openclaw/openclaw/pull/94344 is the best visible code direction because it reuses the existing provider request-policy surface instead of adding a memory-only trust knob.
