## Summary

Keep open: current main and v2026.6.8 still build OpenAI-compatible memory embedding SSRF policy from the configured base URL hostname only, and the linked candidate fix at https://github.com/openclaw/openclaw/pull/94344 is open rather than merged.

## What This Changes

Request a supported explicit opt-in so memory-core OpenAI-compatible embedding endpoints can use trusted private/internal endpoints without weakening default SSRF blocking.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/94344 as the canonical provider-policy fix, keeping this issue open until that decision is complete.

## Reproduction Assessment

Yes, at source level: current main routes memory-core generic providers through the OpenAI-compatible embedding adapter, and that adapter does not read configured provider `request.allowPrivateNetwork`. I did not run a live private endpoint repro in this read-only review.

## Solution Assessment

Unclear until maintainer/security review finishes. Reusing provider-level `request.allowPrivateNetwork` in the open candidate PR looks narrower than adding a memory-specific knob, but it is still a trust-boundary product decision.
