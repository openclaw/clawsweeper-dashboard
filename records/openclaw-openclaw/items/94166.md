## Summary

Keep open: current main and the latest release still build the OpenAI-compatible memory embedding SSRF policy from the endpoint hostname only, without consuming the configured provider `request.allowPrivateNetwork` policy. Two open pull requests now reference this issue as their closing target, so this should stay open until maintainers land one of those fixes or reject the security/config direction.

## What This Changes

The issue requests an explicit, safe opt-in path so memory-core OpenAI-compatible embedding requests can use configured private/internal endpoints without weakening SSRF protection by default.

## Best Possible Solution

Keep the issue open while maintainers review the linked fixes, preferably landing the narrow existing-provider-policy inheritance path if they confirm memory embeddings should inherit that transport trust contract.

## Reproduction Assessment

Yes for the source-level policy gap: current main routes memory OpenAI-compatible embeddings through the generic adapter, but that adapter does not read `models.providers.<id>.request.allowPrivateNetwork`. I did not run a live private endpoint reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security/config contract. Reusing provider-level `request.allowPrivateNetwork` appears narrower than adding a new memory-specific knob because it avoids duplicating trust policy.
