## Summary

Keep open: current main supports generic OpenAI-compatible memory embeddings and already trusts exact configured hostnames for ordinary private addresses, but the memory embedding adapter still does not propagate a configured provider's explicit `request.allowPrivateNetwork` policy. That leaves a real security-boundary/product decision rather than a conservative cleanup close.

## What This Changes

The issue requests an explicit operator-approved path for memory-core OpenAI-compatible embedding requests to reach configured private/internal endpoints without weakening SSRF protection by default.

## Best Possible Solution

Keep the issue open and define a narrow memory-embedding trust contract, preferably by honoring provider-level `request.allowPrivateNetwork` when a configured provider id backs the OpenAI-compatible embedding adapter, with tests that preserve default blocking and metadata/link-local protections.

## Reproduction Assessment

Yes, for the source-level policy gap: configured provider `request.allowPrivateNetwork` is not propagated into OpenAI-compatible memory embeddings. Ordinary exact configured private hosts appear allowed already, so the remaining repro is specifically the missing explicit request-policy inheritance path.

## Solution Assessment

Unclear. Reusing provider-level `request.allowPrivateNetwork` looks like the narrowest existing-surface fix, but maintainers should confirm that memory embeddings are intended to inherit that provider transport trust policy.
