## Summary

Keep open: current main still builds the OpenAI-compatible memory embedding SSRF policy from the base URL hostname only, while two open candidate PRs propose different trust-contract fixes that need maintainer review.

## What This Changes

The issue asks OpenClaw to support explicitly trusted private/internal OpenAI-compatible embedding endpoints for memory-core without relaxing SSRF protection by default.

## Best Possible Solution

Keep this issue as the canonical thread until maintainers choose one explicit trust contract, then land or close the open candidate PRs accordingly; the narrowest-looking direction is provider-policy inheritance with tests that preserve default SSRF blocking.

## Reproduction Assessment

Yes for the source-level policy gap: current main routes configured provider ids into the OpenAI-compatible embedding adapter, but that adapter does not read or propagate request.allowPrivateNetwork. I did not run a live private endpoint in this read-only review.

## Solution Assessment

Unclear until maintainers choose the trust contract. Reusing provider-level request.allowPrivateNetwork appears narrower than adding a second memory-specific config surface, but that security/product choice should be explicit before closing this issue.
