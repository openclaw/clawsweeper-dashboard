## Summary

Keep open: current main and v2026.6.9 still build the OpenAI-compatible memory embedding SSRF policy from the configured embedding base URL only, while the open closing PR owns the provider-policy/security decision.

## What This Changes

The issue asks memory-core OpenAI-compatible embeddings to keep SSRF blocking by default while honoring an explicit private-endpoint trust policy for configured providers.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/94344; if accepted, keep provider-level request.allowPrivateNetwork as the canonical memory embedding trust surface with strict defaults and omitted/false/true coverage.

## Reproduction Assessment

Yes, for the source-level policy gap: current main resolves configured OpenAI-compatible provider ids into the generic embedding adapter, but that adapter does not consume provider request.allowPrivateNetwork. I did not run a live private endpoint repro in this read-only pass.

## Solution Assessment

Unclear until maintainer/security acceptance, but the open provider-policy PR is the best visible code direction because it reuses the existing provider request-policy surface instead of adding a memory-only trust knob.
