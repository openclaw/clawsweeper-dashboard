## Summary

Keep open: current main still does not consume a configured provider's `request.allowPrivateNetwork` policy in the OpenAI-compatible memory embedding client, and https://github.com/openclaw/openclaw/pull/94344 is an open closing PR with real-behavior proof for the remaining contract.

## What This Changes

Request a secure opt-in path for memory-core OpenAI-compatible embedding requests to use explicitly configured private/internal endpoints.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/94344; the desired end state is a single explicit provider-trust contract for memory embeddings that keeps default SSRF blocking and metadata/link-local protections intact.

## Reproduction Assessment

Yes, at source level: current main routes custom memory provider IDs into the generic OpenAI-compatible embedding adapter, but that adapter does not read `models.providers.<id>.request.allowPrivateNetwork`. I did not run a live private endpoint repro in this read-only review.

## Solution Assessment

Unclear until maintainer/security acceptance: reusing provider-level `request.allowPrivateNetwork` looks narrower than adding a memory-only knob, but it is still an explicit security-boundary product decision.
