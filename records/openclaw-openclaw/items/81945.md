## Summary

This active draft PR is a valid fix path for the managed-service self-update bug, but it should stay open because the pending-sentinel retry path can still consume the final update sentinel without refreshing the cached `update.status` value.

## What This Changes

The PR reroutes package-managed Gateway `update.run` through a detached managed-service CLI handoff, delays restart continuations until health passes, adds regression coverage, and updates updater docs.

## Best Possible Solution

Keep the detached CLI handoff design, but refresh or record the latest update sentinel when the pending retry observes the final result before removing the sentinel file.

## Reproduction Assessment

Yes. Source inspection shows current main routes supervised global `update.run` into the in-process updater, and the linked field report describes real LaunchAgent-backed offline/unloaded outcomes; I did not run a destructive self-update locally.

## Solution Assessment

No. The handoff approach is the right fix direction, but the PR needs one more cache update in the restart-sentinel retry path so `update.status` reflects the final ok/error sentinel reliably.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Refresh cached update status before consuming final sentinels:** `src/gateway/server-restart-sentinel.ts:517-529`
  - body: When the restarted gateway first sees a pending handoff sentinel, it caches that pending payload and schedules this retry. If the CLI later writes the final `ok` or `error` sentinel, the retry re-reads and consumes it, but nothing updates `latestUpdateRestartSentinel`; `update.status` only returns that cached value, so callers can keep seeing `restart-health-pending` after the continuation has run. Record or refresh the final update sentinel before removing the file on this retry path.
  - confidence: 0.88
