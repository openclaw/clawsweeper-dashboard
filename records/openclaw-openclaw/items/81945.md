## Summary

Keep open. The reported bug is valid and the handoff direction is appropriate, but this patch has blocking correctness issues in the managed-service handoff and only mock-based proof for the destructive managed-update path.

## What This Changes

This PR reroutes Gateway `update.run` for supervised package installs through a detached CLI update handoff, adds control-plane restart sentinel metadata, expands updater tests, and documents the new response shape.

## Best Possible Solution

Keep the handoff design, but preserve service identity while removing only self-supervision markers, make continuation delivery contingent on verified restart health, and prove the flow on a disposable managed install.

## Reproduction Assessment

Yes, source inspection plus the linked field report give a high-confidence reproduction path: supervised global `update.run` on current `main` reaches `runGatewayUpdate` inside the live Gateway process. I did not run the destructive managed macOS self-update scenario locally.

## Solution Assessment

No. The proposed direction is the right narrow fix, but this implementation strips service identity needed by daemon adapters and can let startup consume a success continuation before the CLI marks restart health failed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve service identity for the handoff CLI:** `src/gateway/server-methods/update-managed-service-handoff.ts:163-168`
  - body: `stripSupervisorHintEnv` deletes `OPENCLAW_LAUNCHD_LABEL`, `OPENCLAW_SYSTEMD_UNIT`, and `OPENCLAW_WINDOWS_TASK_NAME` along with the process-supervision markers. Those variables are how the daemon adapters resolve custom service labels/units/tasks, so the helper can target the default service, leave the actual managed gateway running, and perform the package swap while that service is live.
  - confidence: 0.89
- **[P2] Delay success continuations until restart health passes:** `src/cli/update-cli/update-command.ts:2685-2689`
  - body: This writes an `ok` restart sentinel with a continuation before `maybeRestartService` verifies the restarted gateway. Startup consumes and removes any sentinel it sees shortly after boot, so a bad restart or version mismatch can deliver the success continuation before `markControlPlaneUpdateRestartSentinelFailure` can rewrite the file.
  - confidence: 0.82
