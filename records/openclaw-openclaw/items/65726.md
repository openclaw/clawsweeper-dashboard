## Summary

Keep open. Current main still lacks the requested ACP binding/runtime notify-policy config, and the existing duplicate guard only collapses duplicate task rows for the same ACP run rather than defining active Slack bound-thread ownership across separate sessions.

## What This Changes

Request to add ACP binding-level `notifyPolicy` configuration and a guard that prevents duplicate replies when an ACP-bound Slack thread is already being handled.

## Best Possible Solution

Define one backward-compatible notify-policy config contract and a separate active-bound-thread ownership rule, then plumb both through schemas, docs, ACP binding materialization, task creation, and focused regression coverage.

## Reproduction Assessment

Partial. The `notifyPolicy` config gap is source-reproducible from strict schemas and ACP task creation defaulting to `done_only`, while the Slack cross-session duplicate still needs live or integration proof.

## Solution Assessment

Unclear as one bundled change. Config plumbing is narrow, but duplicate suppression needs a maintainer-approved ownership rule so explicit operator or human-targeted sends are not hidden unexpectedly.
