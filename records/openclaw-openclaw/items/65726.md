Keep open. Current main still lacks the requested ACP binding/runtime `notifyPolicy` config, and its duplicate handling is limited to narrow task-run collapse rather than active Slack bound-thread ownership; related work is adjacent but does not fully supersede this issue.

## What This Changes

Request adds ACP binding/runtime `notifyPolicy` configuration and an active bound-thread duplicate-response guard for Slack ACP bindings.

## Best Possible Solution

Define one backward-compatible notify-policy config contract and a separate active-bound-thread ownership rule, then plumb both through schema, types, docs, ACP binding materialization, task creation, and focused regression coverage.

## Reproduction Assessment

Partial. The `notifyPolicy` config gap is source-reproducible from strict schemas and ACP task creation defaulting to `done_only`; the Slack cross-session duplicate needs live or integration proof.

## Solution Assessment

Unclear as one bundled change. Config plumbing is narrow, but duplicate suppression needs a maintainer-approved ownership rule so explicit operator or human-targeted sends are not hidden unexpectedly.
