## Summary

Keep open. Current main and v2026.5.18 still lack the requested ACP binding/runtime notify-policy config, and existing duplicate guards cover same-run task rows or parent-owned sessions rather than active Slack bound-thread ownership; https://github.com/openclaw/openclaw/issues/64081 and https://github.com/openclaw/openclaw/pull/65242 are adjacent but not a merged complete fix.

## What This Changes

Add an ACP binding/runtime notify-policy config and a thread-level guard to prevent duplicate responses when a Slack thread is already handled by an ACP binding.

## Best Possible Solution

Choose one backward-compatible notify-policy contract and a separate active ACP bound-thread ownership rule, then update schemas, types, docs, binding materialization, task creation, and Slack-focused proof together.

## Reproduction Assessment

Partial. The notify-policy config gap is source-reproducible from current strict schemas, types, docs, and task defaults; the Slack cross-session duplicate scenario still needs live or integration proof.

## Solution Assessment

Unclear as one automated fix. Config plumbing is plausible, but binding-level versus global policy and active-thread suppression semantics need maintainer direction before implementation.
