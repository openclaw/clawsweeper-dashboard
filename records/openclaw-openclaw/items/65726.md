## Summary

Keep open. Current main still lacks ACP binding/runtime `notifyPolicy` config propagation, and the duplicate-reply portion is only partially covered by bound-delivery routing and same-run terminal task dedupe rather than a general active bound-thread ownership rule.

## What This Changes

Request adds configurable ACP task notification policy for ACP bindings/runtime config plus a guard against duplicate Slack replies when a thread is already handled by an ACP session.

## Best Possible Solution

Add typed ACP runtime and binding-level notifyPolicy validation/docs and pass the effective value into ACP task creation; split or explicitly define the active bound-thread ownership rule before suppressing other session replies.

## Reproduction Assessment

Partial. Source inspection gives a high-confidence path for the notify-policy problem because the strict config schema omits the key and ACP task creation falls back to done_only; the cross-session Slack duplicate still needs a live or targeted integration reproduction.

## Solution Assessment

Unclear as one bundled change. notifyPolicy propagation is a narrow maintainable config addition, but duplicate suppression needs an approved ownership rule so explicit operator or human-targeted sends are not hidden unexpectedly.
