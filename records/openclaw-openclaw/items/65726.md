## Summary

Keep open. Current main still has no ACP runtime or binding-level `notifyPolicy` config path, and the duplicate-reply half remains only partially addressed by task-completion routing and same-run dedupe rather than a general active bound-thread ownership rule.

## What This Changes

Requests ACP binding/runtime notification-policy configuration plus a guard against duplicate replies when a Slack thread is already handled by a persistent ACP session.

## Best Possible Solution

Add typed ACP runtime and binding-level `notifyPolicy` validation/docs and pass the effective value into ACP task creation; separately define or split the active bound-thread ownership rule before suppressing other sessions' replies.

## Reproduction Assessment

Partial. The notifyPolicy config gap is source-reproducible from strict schemas, binding materialization, and ACP task creation defaulting to `done_only`; the cross-session Slack duplicate still needs live or targeted integration reproduction.

## Solution Assessment

Unclear as one bundled change. The notifyPolicy propagation is a narrow maintainable config addition, but duplicate suppression needs an approved ownership rule so explicit operator or human-targeted sends are not hidden unexpectedly.
