## Summary

Keep open. Current main still lacks config-level ACP task notify policy propagation, and the duplicate-reply portion remains only partially addressed by bound-session retargeting and same-message dedupe rather than an active bound-thread ownership policy.

## What This Changes

The issue requests ACP binding/runtime config for task notify policy plus a thread-level guard to prevent duplicate replies when a Slack thread is already handled by a bound ACP session.

## Best Possible Solution

Add notifyPolicy as a typed ACP runtime and binding-level config field, validate and document it, carry it through configured binding normalization/materialization, and pass the effective value into ACP task creation. Separately define a narrow active-binding ownership rule for Slack/thread dispatch that suppresses only unintended automated non-owner sends while preserving explicit human/operator delivery.

## Reproduction Assessment

Partial. The notify-policy issue is reproducible with high confidence from source inspection: the config key is absent from strict schemas and ACP task creation falls back to done_only; the cross-session Slack duplicate requires a live multi-session scenario, but current code lacks a general active-owner suppression guard.

## Solution Assessment

Unclear as a single bundled fix. The notifyPolicy portion is a narrow maintainable config propagation change, but the dedup guard should be split or explicitly approved because reply suppression is product-sensitive.
