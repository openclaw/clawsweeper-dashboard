## Summary

Keep open. Current main still lacks an ACP binding/runtime `notifyPolicy` config path, and the duplicate-reply request remains a broader active-bound-thread ownership decision rather than something already implemented by the narrower completion-routing fixes.

## What This Changes

Requests ACP binding/runtime `notifyPolicy` configuration plus an active bound-thread guard to prevent Slack ACP completion banners and duplicate replies.

## Best Possible Solution

Add typed/schema/docs support for an effective ACP `notifyPolicy` before ACP task records are created, and separately define active bound-thread ownership semantics before suppressing other sessions.

## Reproduction Assessment

Partial. The `notifyPolicy` config gap is source-reproducible from strict schemas, binding materialization, and ACP task creation defaulting to `done_only`; the cross-session Slack duplicate still needs targeted live or integration proof.

## Solution Assessment

Unclear as one bundled change. The config plumbing is a narrow API addition, but duplicate suppression needs a maintainer-approved ownership rule so explicit operator or human-targeted sends are not hidden unexpectedly.
