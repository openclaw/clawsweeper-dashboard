## Summary

Keep open. Current main still lacks the requested config-level ACP task notify policy, and the bound-thread duplicate handling is only partially covered by existing bound-session retargeting rather than a general policy guard for suppressing separate-session replies.

## What This Changes

The issue requests an optional ACP binding/runtime `notifyPolicy` setting and a bound-thread ownership guard to prevent duplicate Slack replies from concurrent ACP/Claude Code sessions.

## Best Possible Solution

Add `notifyPolicy` to the typed ACP binding/runtime config contract, schema/help/docs, binding normalization/materialization, binding metadata, and ACP task creation. For duplicate replies, define a narrow active-binding ownership claim that suppresses only unintended automated non-owner dispatches while preserving explicit human/operator sends.

## Reproduction Assessment

Yes for the notify-policy bug: current source inspection shows the proposed config key is not accepted or propagated, and ACP tasks without an explicit policy default to `done_only`. The full duplicate-session Slack scenario is only partially reproducible from static code because current main has bound ACP retargeting, but no general suppress-other-session guard.

## Solution Assessment

Unclear as a single automated fix: the notify-policy portion is narrow and maintainable, but the duplicate guard needs product policy before suppressing replies from other sessions. A safer path is to split the config/default work from an explicitly reviewed bound-thread ownership policy.
