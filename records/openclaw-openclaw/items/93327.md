## Summary

Keep open: current main and the latest stable release still reject the explicit zero-duration CLI form, and the linked open fix PR is the right item to review before this issue closes.

## What This Changes

The issue asks OpenClaw to allow `openclaw message timeout --duration-min 0` through the Discord timeout CLI so it can clear an existing timeout.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/93329, or an equivalent narrow change that aligns the CLI validator with the existing Discord non-negative runtime contract and regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main validates `durationMin` as strictly positive before dispatch, while the Discord runtime accepts non-negative duration minutes. I did not run the CLI in this read-only review, but the call path and existing tests make the failure boundary clear.

## Solution Assessment

Yes, the best fix is to move only `durationMin` to the CLI non-negative integer bucket while keeping malformed and fractional values rejected before dispatch. The linked PR follows that narrow shape.
