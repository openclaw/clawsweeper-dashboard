## Label Justifications

- `P1`: The report describes a broken restart-recovery agent/channel workflow where users can miss the final reply after a gateway restart.
- `impact:message-loss`: Recovered agent replies can be produced but not delivered through channel plugins that depend on the documented pre-delivery hook.
- `impact:session-state`: The failure occurs while resuming persisted main-session state after an interrupted run.

## Summary

Keep open: current main and v2026.6.8 still omit `replyPayloadSendingHook` from restart-recovered agent-command final delivery, and the open closing PR at https://github.com/openclaw/openclaw/pull/90300 is not merged yet.

## What This Changes

The issue asks core to run the existing `reply_payload_sending` pre-delivery hook for restart-recovered agent replies so channel plugins can deliver them.

## Best Possible Solution

Resolve and land or replace https://github.com/openclaw/openclaw/pull/90300 so `deliverAgentCommandResult` supplies hook metadata to `sendDurableMessageBatch` while preserving current restart-abort handling.

## Reproduction Assessment

Yes, source-reproducible: restart recovery calls Gateway `agent` with `deliver: true`, while current agent-command delivery sends through durable delivery without `replyPayloadSendingHook`. I did not run a live restart in this read-only review.

## Solution Assessment

Yes. Reusing the existing durable `reply_payload_sending` hook metadata is the narrow maintainable fix; a new recovery event or last-reply API would duplicate the documented pre-delivery hook contract.
