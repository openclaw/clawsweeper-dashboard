## Label Justifications

- `P1`: The report describes a broken restart-recovered agent/channel workflow where users can miss the final reply after a gateway restart.
- `impact:message-loss`: Recovered agent replies can be produced but not delivered through channel plugins that depend on the documented pre-delivery hook.
- `impact:session-state`: The failure occurs while resuming persisted main-session state after an interrupted run.

## Summary

Keep open: current main and latest release still send restart-recovered agent-command final replies without `replyPayloadSendingHook`, and the linked closing PR is open but currently conflicting rather than merged.

## What This Changes

Ensure restart-recovered agent replies run `reply_payload_sending` before channel delivery so channel plugins do not lose replies after a gateway restart.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/90300 so agent-command durable delivery supplies the existing reply hook metadata while preserving current restart-abort handling.

## Reproduction Assessment

Yes. Source inspection shows restart recovery calls Gateway `agent` with delivery enabled, while current main sends the recovered final payload without `replyPayloadSendingHook`; I did not run a live external-channel restart.

## Solution Assessment

Yes. Reusing the existing durable `reply_payload_sending` hook metadata is the narrow maintainable fix; adding a new recovery event or last-reply API would duplicate the established pre-delivery hook path.
