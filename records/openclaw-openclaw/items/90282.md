## Label Justifications

- `P1`: The report describes a broken restart-recovered agent/channel workflow where users can miss final replies after a gateway restart.
- `impact:message-loss`: Recovered agent replies can be produced but not delivered through channel plugins that depend on the pre-delivery hook.
- `impact:session-state`: The failure occurs while resuming persisted main-session state after an interrupted run.

## Summary

Keep open: current main and `v2026.6.9` still route restart-recovered agent-command final replies through durable delivery without `replyPayloadSendingHook`, and the open closing PR is the active fix path rather than a merged implementation.

## What This Changes

The issue reports restart-recovered agent replies being appended without the `reply_payload_sending` hook, causing channel plugins to miss delivery.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/90300 with the narrow hook metadata pass-through while preserving current restart-abort delivery handling.

## Reproduction Assessment

Yes. Source inspection shows restart recovery calls Gateway `agent` with delivery enabled, while current main and `v2026.6.9` send the recovered final payload without `replyPayloadSendingHook`; I did not run a live restart.

## Solution Assessment

Yes. Reusing the existing durable `reply_payload_sending` hook metadata is the narrow maintainable fix; adding a new recovery event or last-reply API would duplicate the established pre-delivery hook path.
