## Label Justifications

- `P1`: The report describes a broken restart-recovery agent/channel workflow where users can miss the final reply after a gateway restart.
- `impact:message-loss`: Recovered agent replies can be produced but not delivered through channel plugins that depend on the documented pre-delivery hook.
- `impact:session-state`: The failure occurs while resuming persisted main-session state after an interrupted run.

## Summary

Keep open: current main and the latest release still omit `reply_payload_sending` metadata from restart-recovered agent-command final delivery, and the active linked fix remains open at https://github.com/openclaw/openclaw/pull/90300.

## What This Changes

The issue reports restart-recovered agent-command final replies being delivered without `reply_payload_sending` hook metadata, causing channel plugins to miss delivery or fallback hooks.

## Best Possible Solution

Review, update, and land or replace https://github.com/openclaw/openclaw/pull/90300 so `deliverAgentCommandResult` supplies the existing hook metadata to durable delivery while preserving restart-abort handling.

## Reproduction Assessment

Yes, at source level. Restart recovery calls Gateway `agent` with `deliver: true`, while current agent-command delivery sends through durable delivery without `replyPayloadSendingHook`; no live restart was run in this read-only review.

## Solution Assessment

Yes. Reusing the existing durable `reply_payload_sending` hook metadata is the narrow maintainable fix; a new recovery event or last-reply API would duplicate the documented pre-delivery hook contract.
