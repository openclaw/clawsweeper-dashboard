## Label Justifications

- `P1`: The report describes a broken restart-recovery agent/channel workflow where users can miss the final reply after a gateway restart.
- `impact:message-loss`: Recovered agent replies can be produced but not delivered through channel plugins that depend on the documented pre-delivery hook.
- `impact:session-state`: The failure occurs while resuming persisted main-session state after an interrupted run.

## Summary

Keep open: current main and the latest release still omit `replyPayloadSendingHook` from agent-command durable final delivery, and the linked closing PR remains open rather than merged.

## What This Changes

The issue reports that restart-recovered agent replies do not fire `reply_payload_sending`, causing channel plugins to miss final reply delivery after gateway restart.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/90300 so agent-command durable delivery supplies the existing reply hook metadata while preserving current restart-abort handling.

## Reproduction Assessment

Yes. Source inspection shows restart recovery calls Gateway `agent` with delivery enabled, while current agent-command delivery sends the recovered final payload without `replyPayloadSendingHook`; I did not run a live restart in this read-only review.

## Solution Assessment

Yes. Reusing the existing durable `reply_payload_sending` hook metadata is the narrow maintainable fix; adding a new recovery event or last-reply API would duplicate the documented pre-delivery hook contract.
