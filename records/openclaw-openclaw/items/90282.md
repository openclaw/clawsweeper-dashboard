## Label Justifications

- `P1`: The issue describes a broken restart-recovery agent/channel workflow where users can miss the final reply after a gateway restart.
- `impact:message-loss`: Recovered agent replies can be produced but not delivered through channel plugins that depend on the documented pre-delivery hook.
- `impact:session-state`: The failure occurs while resuming persisted main-session state after an interrupted run.

## Summary

Keep open: current main and the latest release still route restart-recovered agent-command replies through durable delivery without `replyPayloadSendingHook`, and the linked closing PR remains open and conflicting rather than merged.

## What This Changes

The issue reports that restart-recovered main-session agent replies skip the documented `reply_payload_sending` hook, causing channel plugins that depend on that hook to miss final reply delivery.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/90300 so the current agent-command durable send call supplies replayable `replyPayloadSendingHook` metadata while preserving restart-abort delivery handling.

## Reproduction Assessment

Yes. Source inspection shows restart recovery calls Gateway `agent` with delivery enabled, while current agent-command delivery sends the recovered final payload without `replyPayloadSendingHook`; I did not run a live restart in this read-only review.

## Solution Assessment

Yes. Reusing the existing durable `reply_payload_sending` hook metadata is the narrow maintainable fix; adding a new recovery event or last-reply API would duplicate the documented pre-delivery hook contract.
