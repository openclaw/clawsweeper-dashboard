## Label Justifications

- `impact:message-loss`: The reported failure drops the recovered reply before channel plugins can send it.
- `impact:session-state`: The affected path depends on persisted session restart recovery and pending delivery state.

## Summary

Keep open: current main and the latest release still route restart-recovered agent replies through agent-command durable delivery without passing the existing `reply_payload_sending` hook, even though normal auto-reply delivery and the plugin docs make that hook part of the pre-delivery pipeline.

## What This Changes

Request to ensure restart-recovered main-session replies trigger `reply_payload_sending` before channel delivery so channel plugins can deliver recovered replies.

## Best Possible Solution

Thread the existing `reply_payload_sending` hook metadata through agent-command durable delivery for restart recovery and cover the recovery/cancel cases with tests, rather than adding a parallel recovery event or last-reply API.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: mark a main session aborted with a recovery delivery context, let restart recovery call Gateway `agent` with `deliver: true`, and observe that `deliverAgentCommandResult` sends without `replyPayloadSendingHook`. I did not run a live restart in this read-only review.

## Solution Assessment

Yes, the best fix is to reuse the existing hook contract in the agent-command durable delivery path. A new `session.recovered` event or `session.getLastReply()` API would duplicate an already documented pre-delivery hook.
