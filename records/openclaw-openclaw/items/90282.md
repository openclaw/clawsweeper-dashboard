## Label Justifications

- `P1`: The report describes a broken restart-recovery agent/channel workflow where users can miss final replies after a gateway restart.
- `impact:message-loss`: Recovered agent replies can be produced but not delivered through channel plugins that depend on the documented pre-delivery hook.
- `impact:session-state`: The failure occurs while resuming persisted main-session state after an interrupted run.

## Summary

Keep open: current main and the latest release still omit `replyPayloadSendingHook` from restart-recovered agent-command final delivery, and the open linked fix at https://github.com/openclaw/openclaw/pull/90300 has closing syntax but is currently conflicting.

## What This Changes

The issue requests that restart-recovered agent replies trigger `reply_payload_sending` so channel plugins can deliver replies after gateway restart.

## Best Possible Solution

Update and land or replace https://github.com/openclaw/openclaw/pull/90300 so agent-command durable final delivery supplies existing hook metadata while preserving current restart-abort handling.

## Reproduction Assessment

Yes, at source level: restart recovery calls Gateway `agent` with `deliver: true`, while current `deliverAgentCommandResult` calls `sendDurableMessageBatch` without `replyPayloadSendingHook`. I did not run a live gateway restart in this read-only review.

## Solution Assessment

Yes: reusing the existing durable `replyPayloadSendingHook` path is the narrow maintainable fix. A new recovery event or last-reply API would duplicate the documented pre-delivery hook contract.
