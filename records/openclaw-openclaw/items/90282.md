## Label Justifications

- `P1`: The report describes a broken restart-recovery agent/channel workflow where real users can never receive the final reply after gateway restart.
- `impact:message-loss`: Recovered agent replies can be produced but not delivered to channel plugins that rely on the documented pre-delivery hook.
- `impact:session-state`: The failure occurs while resuming persisted main-session state after an interrupted run.

## Summary

Keep open: current `main` and latest stable release still omit `reply_payload_sending` metadata from restart-recovered agent-command delivery, and the linked fix at https://github.com/openclaw/openclaw/pull/90300 is still open and conflicting.

## What This Changes

The issue requests that restart-recovered agent replies run the existing `reply_payload_sending` delivery hook so channel plugins can deliver recovered replies.

## Best Possible Solution

Resolve or replace https://github.com/openclaw/openclaw/pull/90300 so current `main` threads the existing hook metadata through agent-command durable delivery while preserving restart-abort handling.

## Reproduction Assessment

Yes. Source inspection shows restart recovery calls Gateway `agent` with `deliver: true`, while current `deliverAgentCommandResult` sends the recovered final payload without `replyPayloadSendingHook`; I did not run a live restart in this read-only review.

## Solution Assessment

Yes. Reusing the existing `reply_payload_sending` durable hook metadata is the narrow maintainable fix; adding a new recovery event or last-reply API would duplicate an already documented pre-delivery hook contract.
