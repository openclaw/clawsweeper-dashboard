## Summary

Keep this PR open: the linked user confusion is real and the earlier config-path typo appears fixed, but the current wording is now too absolute because current main supports `openai-codex` OAuth as a fallback for native OpenAI Realtime voice flows when no direct API key is configured. The PR also still lacks acceptable after-fix real behavior proof for the contributor gate.

## What This Changes

The PR updates `docs/providers/openai.md` and `CHANGELOG.md` to document that OpenAI Realtime voice/Talk use OpenAI Platform billing rather than Codex/ChatGPT subscription quota.

## Best Possible Solution

Revise the docs to explain the `insufficient_quota` Platform API-key failure mode while preserving the current `openai-codex` OAuth fallback for native OpenAI Realtime voice, then attach acceptable rendered-docs or terminal proof.

## Reproduction Assessment

Yes for the blocking review finding: source and tests on current main show that OpenAI Realtime voice can use `openai-codex` OAuth fallback when no direct API key is configured. I did not run a live Realtime session in this read-only review.

## Solution Assessment

No in the current form: adding a billing/auth warning is the right docs direction, but the absolute `OPENAI_API_KEY`/Platform-credits wording conflicts with current source-backed behavior. The safer fix is a narrower troubleshooting note that distinguishes direct API-key billing failures from the registered realtime provider id and OAuth fallback.
