## Summary

Keep open: the docs direction is useful and the linked report describes a real OpenAI Platform billing confusion, but the current diff still conflicts with OpenClaw's current Realtime auth and Talk config contracts.

## What This Changes

The PR updates the OpenAI provider docs and changelog to warn that OpenAI Realtime voice/Talk uses OpenAI Platform billing rather than Codex/ChatGPT subscription quota.

## Best Possible Solution

Revise the note to document the `insufficient_quota` Platform billing failure while preserving the current provider id/auth contract: Realtime provider `openai`, Control UI keys under `talk.realtime.*`, Voice Call keys under the plugin realtime path, and `openai-codex` OAuth as a supported fallback where current source allows it.

## Reproduction Assessment

Yes for the review findings: current source and docs show Talk realtime config is read from `talk.realtime.*`, and tests/source show the `openai-codex` OAuth fallback. I did not run a live Realtime billing session in this read-only review.

## Solution Assessment

No in the current form. Adding a Platform billing troubleshooting note is the right direction, but the PR should use the realtime config keys and avoid saying API-key auth is the only supported Realtime route.
