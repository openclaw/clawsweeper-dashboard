## Summary

Keep open. Current main still lacks WhatsApp delete/revoke support, and open PR #70813 is already paired with this issue via closing syntax but has not merged; the latest discussion adds safety requirements rather than making the request obsolete.

## What This Changes

Add a WhatsApp message delete/revoke action so agents can retract their own previously sent WhatsApp messages through the message tool.

## Best Possible Solution

Keep this as the canonical bundled WhatsApp plugin request until #70813 or an equivalent focused PR lands with maintainer-approved action naming, WhatsApp-owned runtime/listener support, safety checks, docs, tests, and changelog coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: WhatsApp discovery omits delete/revoke, the adapter only supports react, the runtime rejects non-react actions, the send API has no delete payload, and the listener API has no delete helper.

## Solution Assessment

Yes. The best current path is to keep this issue open while maintainers review and land or revise the paired WhatsApp plugin PR, rather than closing the issue before the implementation merges.

## Security Review

Status: needs_attention

Summary: The issue requests a destructive WhatsApp message-mutation capability, so any implementation needs explicit safety constraints even though no patch is attached here.

Concerns:

- **[medium] Constrain WhatsApp message mutation:**
  - body: A delete/revoke action can remove visible chat content or mislead agents if server-side revoke silently falls back to local deletion, so implementation should require own-message/fromMe behavior, reuse outbound authorization and send gates, handle group participant keys correctly, and surface hard errors for age-window or Baileys restrictions.
  - confidence: 0.9
