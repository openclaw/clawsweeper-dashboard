## Summary

Keep open. Current main and the latest release still do not expose WhatsApp delete/revoke through the message tool, while https://github.com/openclaw/openclaw/pull/70813 is still open and unmerged with closing metadata for this issue. This should remain the canonical tracker; https://github.com/openclaw/openclaw/issues/38723 was already closed as the duplicate, and the remaining work needs maintainer review because it adds security-sensitive WhatsApp message mutation.

## What This Changes

Add a WhatsApp message delete/revoke action so agents can retract their own previously sent WhatsApp messages by target and message id.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/70813, or an equivalent WhatsApp plugin-owned implementation, with agreed action naming, own-message authorization, LID-aware routing, group key handling, stale revoke errors, focused tests, docs, changelog, and redacted live WhatsApp proof.

## Reproduction Assessment

Yes. Source inspection on current main shows WhatsApp delete is not advertised, is declined by the plugin action adapter, and would hit the shared unsupported-action path; I did not run a live WhatsApp account replay.

## Solution Assessment

Yes for the owner boundary, unclear on final semantics. A WhatsApp plugin-owned implementation through the existing message action, gateway, and listener seams is the narrow maintainable path, but naming and mutation-safety behavior need maintainer approval.
