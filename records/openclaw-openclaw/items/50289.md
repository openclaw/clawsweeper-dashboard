## Summary

Keep open: current main still has the reported command-authorization hot path, and the live closing implementation path is the open PR at https://github.com/openclaw/openclaw/pull/73440.

## What This Changes

The issue asks OpenClaw to avoid severe gateway slowdowns when `commands.ownerAllowFrom` contains thousands of entries.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73440 after any needed repair, or land an equivalent narrow command-auth change that caches compiled owner authorization state while preserving provider-prefix filtering and owner-only fail-closed behavior.

## Reproduction Assessment

Yes, from source inspection. Configure a large `commands.ownerAllowFrom` list and send an inbound message; current main builds command context, resolves owner allowlists, rebuilds owner arrays, and performs linear membership checks on that path.

## Solution Assessment

Yes, directionally: a narrow command-auth cache or compiled lookup is the maintainable fix boundary. Wildcard owner access or a broad access-control redesign would change security semantics and should not be the default fix.
