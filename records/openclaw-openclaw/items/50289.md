## Summary

Keep open: current main still performs repeated linear owner allowlist normalization and membership checks on the inbound command authorization path, and the linked closing PR is still open in the provided GitHub context.

## What This Changes

The issue asks to remove severe gateway latency when large `commands.ownerAllowFrom` lists make inbound authorization repeatedly walk and search thousands of owner IDs.

## Best Possible Solution

Review and land PR #73440, or an equivalent narrow `src/auto-reply/command-auth.ts` change, that caches or compiles normalized owner allowlists per raw array/provider/account/plugin and uses efficient membership checks while preserving fail-closed owner-only authorization semantics.

## Reproduction Assessment

Yes, from source inspection. Configure a large `commands.ownerAllowFrom` list and send a normal inbound message; current main builds command context, resolves owner allowlists twice, and performs linear membership checks on that path.

## Solution Assessment

Yes, directionally. A narrow cache or compiled lookup inside `src/auto-reply/command-auth.ts` is the maintainable fix boundary; wildcard owner access or a broad access-control redesign would be riskier than necessary for this bug.
