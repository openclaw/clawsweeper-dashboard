## Summary

Keep open: current main still has the reported command-authorization hot path, and the issue is explicitly paired with an open unmerged PR that uses closing syntax.

## What This Changes

The issue reports severe gateway slowdown when `commands.ownerAllowFrom` contains thousands of entries because command authorization repeatedly normalizes and searches the allowlist.

## Best Possible Solution

Land the linked PR after repair, or an equivalent narrow command-auth change, that caches or compiles owner allowlist authorization state per raw array/provider/account/plugin and uses efficient membership checks while preserving provider-prefix filtering and fail-closed owner-only behavior.

## Reproduction Assessment

Yes, from source inspection. Configure a large `commands.ownerAllowFrom` list and send an inbound message; current main builds command context, resolves owner allowlists, derives owner state, and checks owner membership with linear array scans.

## Solution Assessment

Yes, directionally. A narrow cache or compiled lookup inside `src/auto-reply/command-auth.ts` is the maintainable fix boundary; wildcard owner access or broader access-control redesign would be a riskier product change.
