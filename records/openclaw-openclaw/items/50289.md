## Summary

Keep open. Current `main` still performs per-message owner allowlist normalization and linear sender membership checks, and the discussion already has open PR #73440 linked as the active closing fix candidate. The config reparse part of the original diagnosis is less certain on current `main` because runtime config is pinned after first load, but the command-auth hot-path cost remains real.

## What This Changes

The issue asks OpenClaw to avoid severe gateway latency when `commands.ownerAllowFrom` contains thousands of entries by removing repeated O(n) owner authorization work from normal message handling.

## Best Possible Solution

Review and land PR #73440, or an equivalent narrow command-auth change, that caches or compiles normalized owner allowlists per stable raw list/provider/account/plugin while preserving config reload behavior, channel prefix semantics, and fail-closed owner-only authorization.

## Reproduction Assessment

Yes. Configure a large `commands.ownerAllowFrom` list and send a normal inbound message; current `main` builds command context, resolves owner allowlists twice, and performs linear membership checks on that path.

## Solution Assessment

Yes, directionally. A narrow cache or compiled lookup inside `src/auto-reply/command-auth.ts` is the maintainable fix; wildcard owner access or a broad access-control redesign would be a workaround or product change, not the right primary bug fix.
