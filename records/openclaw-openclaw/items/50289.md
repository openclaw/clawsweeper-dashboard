## Summary

Keep open. Current main still performs repeated linear normalization and membership checks for large `commands.ownerAllowFrom` lists on the inbound command authorization path, and open PR #73440 is already linked with closing semantics but is not merged in the provided GitHub context.

## What This Changes

The issue requests reducing severe Gateway latency when `commands.ownerAllowFrom` contains thousands of IDs by avoiding repeated linear owner allowlist processing during message authorization.

## Best Possible Solution

Review and land PR #73440, or an equivalent narrow command-auth change, that caches or compiles normalized owner allowlists per raw array/provider/account/plugin and uses efficient membership checks while preserving config reload behavior and fail-closed owner-only authorization semantics.

## Reproduction Assessment

Yes. Configure a large `commands.ownerAllowFrom` list and send a normal inbound message; current main builds command context, resolves owner allowlists twice, and performs linear list membership checks on that path.

## Solution Assessment

Yes, directionally. A narrow cache or compiled lookup inside `src/auto-reply/command-auth.ts` is the maintainable fix boundary; wildcard owner access or a broad access-control redesign would be riskier than necessary for this bug.
