## Summary

Keep open. Current main contains important mitigations for this report, including WhatsApp socket timing config, shared outbound reconnect draining, legacy runtime-deps cleanup, and bounded/yielding session-list transcript reads, but it still does not satisfy the central asks: liveness warnings do not identify the blocking async resource or call path, and direct WhatsApp auto-replies still bypass the durable shared delivery queue.

## What This Changes

The issue asks OpenClaw to investigate WhatsApp 408 disconnects attributed to long Node event-loop stalls, add blocker-attributed diagnostics, replay outbound replies after reconnect, expose Baileys timing options, and garbage-collect stale plugin runtime-deps trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The desired end state is privacy-safe blocker attribution in liveness diagnostics, removal or offloading of any confirmed synchronous hot path, reconnect-safe WhatsApp auto-reply delivery with duplicate-send protection, and an upgrade/doctor cleanup model that matches packaged installs.

## Reproduction Assessment

Unclear. The reporter provided credible production liveness samples and deterministic 408 timing, and current source inspection confirms missing blocker attribution, but there is no self-contained current-main reproduction for the 100s event-loop blocks or the exact direct auto-reply drop.

## Solution Assessment

No. The current implementation is a mitigation set rather than the complete solution; the remaining work needs maintainer scoping around diagnostic attribution and durable WhatsApp reply replay semantics before a narrow fix is safe.
