## Summary

Keep open. Current main has useful mitigations for WhatsApp 408 reliability, runtime-deps cleanup, and large transcript responsiveness, but it still does not identify the blocking async resource/call path and direct WhatsApp auto-replies still use the immediate delivery callback rather than a durable reconnect replay path.

## What This Changes

Investigate WhatsApp 408 disconnects attributed to long Gateway event-loop stalls and add blocker attribution, reconnect-safe reply replay, Baileys timing controls, and runtime-deps garbage collection.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation: add privacy-safe blocker attribution to liveness diagnostics, remove or offload any confirmed synchronous blocker, and make direct WhatsApp auto-reply delivery reconnect-safe with duplicate-send protection.
