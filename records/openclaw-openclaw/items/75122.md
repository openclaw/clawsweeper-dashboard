## Summary

Keep open. Current main has several partial mitigations for the report, including WhatsApp Baileys timing knobs, reconnect-aware direct send retries, generic queued outbound recovery, and some new runtime-deps startup hardening, but the core report is not fully resolved: liveness warnings still only expose aggregate event-loop metrics, direct WhatsApp auto-replies are not durably queued for post-disconnect replay, and versioned runtime-deps roots are still preserved rather than garbage-collected.

## What This Changes

Investigate WhatsApp 408 disconnects attributed to long Node event-loop stalls and add blocker attribution, reconnect-safe reply replay, Baileys timing configuration, and stale plugin-runtime-deps cleanup.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The maintainable end state is low-noise blocker attribution in liveness diagnostics, removal or offloading of the confirmed blocking path, reconnect-safe WhatsApp auto-reply delivery with explicit idempotency, and a documented retention policy for versioned runtime-deps roots.

## Reproduction Assessment

Unclear. The reporter provided credible production liveness logs and timing that match the 408 symptom, and source inspection confirms remaining gaps, but there is no deterministic local or live current-main reproduction for the long event-loop blocks or dropped auto-reply-after-disconnect path.

## Solution Assessment

No. Current main is a partial mitigation set rather than the complete solution; a safe fix needs maintainer scoping around blocker attribution, durable reply replay semantics, and runtime-deps retention before choosing a narrow patch.
