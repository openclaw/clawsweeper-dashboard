## Summary

Keep open. Current main contains useful mitigations for WhatsApp 408s, runtime-deps cleanup, and large transcript responsiveness, but it still lacks the requested liveness blocker attribution and direct WhatsApp auto-replies are bounded in-process reconnect retries rather than a durable replay path.

## What This Changes

The issue asks OpenClaw to diagnose and fix WhatsApp 408 disconnects attributed to long event-loop blocking, add reconnect-safe outbound replay, expose Baileys timing knobs, and clean stale plugin runtime dependency trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation: add privacy-safe blocker attribution to liveness diagnostics, remove or offload any confirmed synchronous blocker, and make direct WhatsApp auto-reply delivery reconnect-safe with duplicate-send protection.

## Reproduction Assessment

Unclear. The reporter supplied credible production timing and current source confirms the requested blocker attribution is absent, but there is no self-contained current-main reproduction for the long event-loop stalls or direct auto-reply drop.

## Solution Assessment

No. Current main is a mitigation set, not the complete fix; the narrowest maintainable solution needs diagnostic attribution first, then a separately scoped delivery replay design if direct replies still drop after reconnect retries.
