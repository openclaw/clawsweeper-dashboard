## Summary

Keep open. Current main already adds useful mitigations for Baileys timing, transcript/event-loop pressure, reconnect queue draining, and stale runtime-deps cleanup, but it still does not provide liveness blocker attribution and direct WhatsApp auto-replies still use live socket delivery instead of durable write-ahead replay.

## What This Changes

Investigate WhatsApp 408 disconnects tied to event-loop blocking, add blocker attribution, replay dropped replies after reconnect, expose Baileys timing knobs, and clean stale plugin runtime dependency trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation: add privacy-safe blocker attribution, remove any confirmed synchronous blockers, and route direct WhatsApp replies through durable reconnect-safe delivery with dedupe protection.

## Reproduction Assessment

Unclear for the full 408 loop: the production timing evidence is credible, and source inspection confirms the missing attribution and non-durable direct-reply path, but I did not live-reproduce the current-main disconnect. The remaining gaps are source-reproducible from the liveness event schema and WhatsApp direct delivery path.

## Solution Assessment

No. Current main has useful mitigations, but the best maintainable path is diagnostic attribution first, followed by targeted blocker removal and a deliberate durable replay design for direct WhatsApp replies.
