## Summary

Keep open. Current main has several relevant mitigations for transcript/event-loop stalls, Baileys timing, reconnect draining, and stale runtime-deps cleanup, but the requested blocker attribution and durable direct WhatsApp auto-reply replay are still not fully implemented.

## What This Changes

Investigate WhatsApp 408 disconnects attributed to Node event-loop stalls, add blocker attribution/replay behavior, expose Baileys timing controls, and clean stale runtime-deps caches.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability item: ship privacy-safe blocker attribution, confirm the known synchronous blockers are gone, and make direct WhatsApp replies durable through reconnect with dedupe protection.

## Reproduction Assessment

No high-confidence live reproduction was established for the current-main 408 loop. Source inspection does confirm two remaining reproducible gaps: liveness warnings cannot attribute the blocker, and direct WhatsApp auto-replies still use live delivery outside the shared write-ahead queue.

## Solution Assessment

No. Current main is a useful mitigation set, but the narrow maintainable solution is blocker attribution first, then targeted blocker removal and durable direct-reply replay semantics after maintainer review.
