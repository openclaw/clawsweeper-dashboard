## Summary

Keep open. Current main has partial mitigations for this report, including Baileys timing config, replacement-socket retry for direct replies, generic outbound reconnect-drain plumbing, and legacy runtime-deps pruning, but the central reliability asks are not fully done: liveness warnings still do not attribute the blocking resource and direct WhatsApp auto-replies are not durably queued for replay after a disconnect.

## What This Changes

Investigate WhatsApp 408 disconnects attributed to event-loop stalls, add blocker attribution, reconnect-safe reply replay, Baileys timing controls, and runtime-deps cleanup.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The maintainable end state is blocker attribution in liveness diagnostics, removal or offloading of the confirmed synchronous path, durable reconnect-safe WhatsApp auto-reply delivery, and an explicit runtime-deps retention policy.

Unclear. The reporter supplied credible production timing and liveness logs, and current source inspection confirms missing attribution/replay gaps, but there is no self-contained current-main reproduction for the long event-loop stalls.

## Solution Assessment

No. Current main is a partial mitigation set; the best fix needs maintainer scoping around diagnostics attribution, durable direct-reply replay semantics, and runtime-deps retention before choosing a narrow patch.
