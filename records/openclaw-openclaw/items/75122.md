## Summary

Keep open. Current main has useful mitigations for WhatsApp 408 reliability, Baileys timing knobs, transcript responsiveness, and runtime-deps cleanup, but the central asks remain unresolved because liveness warnings still lack async-resource/blocking-call attribution and direct WhatsApp auto-replies still use immediate socket delivery rather than durable reconnect-safe replay.

## What This Changes

The issue asks OpenClaw to investigate event-loop-blocking WhatsApp 408 disconnects, add blocker attribution, replay dropped auto-replies after reconnect, expose Baileys timing options, and garbage-collect stale plugin runtime-deps trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation: add privacy-safe blocker attribution, remove or offload any confirmed synchronous blocker, and make direct WhatsApp auto-reply delivery reconnect-safe with dedupe protection.
