## Summary

Keep open. Current main has several relevant mitigations for this report, including WhatsApp Baileys timing knobs, reconnect draining for the shared outbound queue, runtime-deps pruning, and bounded/yielding session-list transcript reads, but the central asks are not fully satisfied: liveness warnings still do not identify the blocking async resource/call path, and direct WhatsApp auto-replies still go through the auto-reply delivery callback rather than a durable reconnect replay queue.

## What This Changes

The issue asks OpenClaw to investigate WhatsApp 408 disconnects attributed to long Node event-loop stalls, add blocker-attributed diagnostics, replay queued outbound replies after reconnect, expose Baileys timing options, and garbage-collect stale plugin-runtime-deps trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The desired end state is privacy-safe blocker attribution in liveness diagnostics, removal or offloading of the confirmed synchronous path, reconnect-safe WhatsApp auto-reply delivery with duplicate-send protection, and a documented runtime-deps retention model that matches packaged upgrade behavior.

## Reproduction Assessment

Unclear. The reporter provided credible production liveness samples and deterministic 408 timing, and current source inspection confirms the requested blocker attribution is absent, but there is no self-contained current-main reproduction for the long event-loop blocks or the exact direct auto-reply drop.

## Solution Assessment

No. Current main is a useful mitigation set, not the complete solution; the remaining work needs maintainer scoping around diagnostic attribution and durable WhatsApp reply replay semantics before a narrow fix is safe.
