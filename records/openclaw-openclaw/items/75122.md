## Summary

Keep open. Current main has partial mitigations for the report, including WhatsApp Baileys timing knobs, generic queued outbound recovery, reconnect drain plumbing, and unknown runtime-deps pruning, but the central asks are still not fully satisfied: liveness warnings still do not attribute the blocking async resource, the direct WhatsApp auto-reply path is not durably queued for replay after disconnect, and versioned runtime-deps roots are still intentionally preserved.

## What This Changes

The issue requests fixes for WhatsApp 408 disconnects attributed to long Node event-loop blocks, including blocker attribution, reconnect-safe outbound replay, Baileys timing controls, and stale runtime-deps garbage collection.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The maintainable end state is low-noise blocker attribution in liveness diagnostics, removal or offloading of the confirmed synchronous path, reconnect-safe WhatsApp auto-reply delivery with explicit idempotency, and a documented retention policy for versioned runtime-deps roots.

Unclear. The reporter provided credible production liveness logs and 408 timing, and source inspection confirms several remaining gaps, but there is no self-contained current-main reproduction for the 33-100s event-loop blocks or the exact dropped auto-reply-after-disconnect path.

## Solution Assessment

No. Current main is a partial mitigation set, not the complete solution; a safe fix needs maintainer scoping around blocker attribution, durable reply replay semantics, and runtime-deps retention before a narrow patch is chosen.
