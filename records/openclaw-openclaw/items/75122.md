## Summary

Keep open. Current main has meaningful partial mitigations for this WhatsApp 408 report, including Baileys timing knobs, reconnect-drain plumbing, stale runtime-deps pruning, and session/transcript responsiveness work, but it still does not satisfy the central reliability asks for blocker-attributed liveness diagnostics or durable direct WhatsApp auto-reply replay after socket loss.

## What This Changes

The issue requests fixes for WhatsApp 408 disconnects attributed to long Gateway event-loop blocks, including async-resource blocker diagnostics, reconnect reply replay, Baileys timeout/keepalive configuration, and stale runtime-deps garbage collection.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The desired end state is blocker-attributed liveness diagnostics, removal or offloading of the confirmed synchronous path, reconnect-safe WhatsApp reply delivery with duplicate-send protection, and a documented runtime-deps retention model that matches packaged upgrade behavior.
