## Summary

Keep open. Current main has partial mitigations for this WhatsApp 408 report, especially Baileys socket timing config, generic reconnect-drain plumbing, and runtime-deps cleanup, but the central reliability asks are still not fully satisfied: liveness warnings still do not attribute the blocking async resource/call path, and the direct WhatsApp auto-reply path is not durably queued before delivery for replay after a socket close.

## What This Changes

The issue requests fixes for WhatsApp 408 disconnects attributed to long OpenClaw gateway event-loop blocks, including async-resource blocker diagnostics, reconnect reply replay, Baileys timeout/keepalive knobs, and stale runtime-deps garbage collection.

## Best Possible Solution

Keep this as an active WhatsApp/gateway reliability investigation. The desired end state is blocker-attributed liveness diagnostics, removal or offloading of the confirmed synchronous path, reconnect-safe WhatsApp reply delivery with explicit duplicate-send protection, and a documented runtime-deps retention model that matches packaged upgrade behavior.

## Reproduction Assessment

Unclear. The reporter provided credible production liveness samples and deterministic 408 timing, and current source confirms the requested blocker attribution is absent, but there is no self-contained current-main reproduction for the long event-loop blocks or the exact in-flight auto-reply drop.

## Solution Assessment

No. Current main is a useful mitigation set, not the complete solution; the remaining fix needs maintainer scoping around diagnostic attribution, durable WhatsApp reply replay semantics, and runtime-deps retention before a narrow implementation can be chosen.
