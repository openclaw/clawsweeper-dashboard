## Summary

Keep open. Current main includes several mitigations for the reported WhatsApp reliability cluster, but it still does not surface the blocking async resource/call path in liveness diagnostics and direct WhatsApp auto-replies still use the immediate provider send path rather than a durable reconnect replay path.

## What This Changes

Investigate WhatsApp 408 disconnects by attributing event-loop blockers, making reply delivery reconnect-safe, exposing Baileys timing knobs, and cleaning stale plugin runtime dependency trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation: add privacy-safe blocker attribution to liveness diagnostics, remove or offload any confirmed synchronous blocker, and make direct WhatsApp auto-reply delivery reconnect-safe with duplicate-send protection.

## Reproduction Assessment

Unclear. The production logs and timing table are credible, and current source inspection confirms the missing attribution and direct-send gap, but there is no self-contained current-main reproduction for the long event-loop blocks or dropped direct auto-reply.

## Solution Assessment

No. Current main is a mitigation set, not the complete solution; the remaining work needs maintainer scoping around diagnostic attribution and durable WhatsApp reply replay semantics before a narrow fix is safe.
