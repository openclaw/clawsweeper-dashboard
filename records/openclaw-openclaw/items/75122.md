## Summary

Keep open. Current main has useful mitigations for WhatsApp socket timing, queued outbound reconnect draining, and stale runtime-deps cleanup, but liveness warnings still expose only aggregate metrics and direct WhatsApp auto-replies still use live socket delivery instead of durable reconnect-safe replay.

## What This Changes

The issue requests investigation and fixes for WhatsApp 408 disconnects attributed to event-loop blocking, including blocker attribution, reconnect-safe reply replay, Baileys timing config, and runtime-deps cleanup.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation: add privacy-safe blocker attribution, remove confirmed synchronous blockers, and route direct auto-replies through durable reconnect-safe delivery with dedupe protection.

## Reproduction Assessment

Unclear for the full 408 failure: the production logs are credible, and source inspection confirms the remaining missing attribution and non-durable direct-reply path, but I did not establish a live current-main WhatsApp 408 reproduction.

## Solution Assessment

No. Current main is a partial mitigation set; the maintainable path is privacy-safe diagnostic attribution first, then targeted blocker removal and durable direct-reply replay semantics after duplicate-send policy is decided.
