## Summary

Keep open. Current main has mitigations for WhatsApp 408s, Baileys timing knobs, transcript responsiveness, and runtime-deps cleanup, but the central reliability asks are still unresolved: liveness warnings do not identify the blocking async resource/call path, and direct WhatsApp auto-replies still use immediate socket delivery rather than durable reconnect-safe replay.

## What This Changes

The issue asks OpenClaw to investigate event-loop-blocking WhatsApp 408 disconnects, add blocker attribution, replay dropped auto-replies after reconnect, expose Baileys timing options, and garbage-collect stale plugin runtime-deps trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation: add privacy-safe blocker attribution, remove or offload any confirmed synchronous blocker, and make direct WhatsApp auto-reply delivery reconnect-safe with dedupe protection while keeping the shipped Baileys timing and doctor-cleanup mitigations.

## Reproduction Assessment

Unclear. The reporter provided credible production liveness samples and deterministic 408 timing, and source inspection confirms missing blocker attribution plus a non-durable direct reply path, but there is no self-contained current-main reproduction for the long event-loop blocks or dropped auto-reply.

## Solution Assessment

No. Current main is a mitigation set rather than the complete fix; the narrowest maintainable path is to add diagnostic attribution first, then fix confirmed blockers and replay semantics with duplicate-send safeguards.
