## Summary

Keep open. Current main has useful mitigations for WhatsApp 408 reliability, stale runtime-deps cleanup, and large transcript responsiveness, but liveness diagnostics still do not identify the blocking async resource or call path, and direct WhatsApp auto-replies still use immediate socket delivery rather than a durable reconnect replay path.

## What This Changes

The issue reports WhatsApp 408 disconnects tied to long Gateway event-loop stalls and asks for blocker attribution, reconnect-safe reply replay, Baileys timing controls, and runtime-deps garbage collection.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation: add privacy-safe blocker attribution, offload any proven synchronous blocker, and make direct WhatsApp auto-reply delivery reconnect-safe with duplicate-send protection.

## Reproduction Assessment

Unclear. The reporter provided credible production liveness samples and deterministic 408 timing, and current source inspection verifies missing blocker attribution, but there is no self-contained current-main reproduction or live account proof for the long stalls and dropped direct auto-replies.

## Solution Assessment

No. Current main is a mitigation set rather than a complete fix; the narrow maintainable path is profile-guided blocker removal plus an owner-reviewed durable delivery design for direct WhatsApp replies.
