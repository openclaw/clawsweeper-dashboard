## Summary

Keep open. Current main has useful mitigations for Baileys timing, transcript scan responsiveness, reconnect queue draining, and legacy runtime-deps cleanup, but the central reliability asks remain unresolved: liveness warnings still lack async-resource/blocking-call attribution, and direct WhatsApp auto-replies still bypass durable reconnect-safe replay.

## What This Changes

The issue asks OpenClaw to diagnose and fix event-loop blocking that causes WhatsApp 408 disconnects, add blocker attribution, replay queued replies after reconnect, expose Baileys timing options, and garbage-collect stale plugin runtime-dependency trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation: add privacy-safe blocker attribution, remove or offload confirmed synchronous blockers, and route direct WhatsApp auto-reply delivery through durable reconnect replay with dedupe protection.
