## Summary

Keep open. Current main has useful partial mitigation, especially WhatsApp socket timing controls and a generic reconnect drain, but it still does not satisfy the central diagnostic and durability asks: liveness warnings remain aggregate-only, the direct WhatsApp auto-reply path still sends through the live inbound message callbacks, and runtime-deps startup pruning only removes `openclaw-unknown-*` roots while intentionally preserving versioned roots.

## What This Changes

The issue asks OpenClaw to diagnose and fix WhatsApp 408 disconnects linked to long Node event-loop stalls, add reconnect-safe outbound replay, expose WhatsApp timing knobs, and clean stale runtime-deps trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The maintainable end state is payload-safe blocker attribution in liveness warnings, removal or offloading of the confirmed synchronous blocker, reconnect-safe WhatsApp reply delivery with clear idempotency, and a documented retention policy for stale versioned runtime-deps roots.

## Reproduction Assessment

Unclear. The reporter provided credible production liveness logs and deterministic 408 timing, and source inspection verifies missing requested behaviors, but current main still lacks a self-contained reproduction for the 33-100s event-loop stalls or the exact dropped auto-reply-after-disconnect case.

## Solution Assessment

No. Closing or sending this straight to a narrow repair PR would be premature because current main only partially mitigates the report and the remaining work spans performance attribution, delivery semantics, and runtime-deps retention policy.
