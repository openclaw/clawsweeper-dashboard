## Summary

Keep open. Current main has useful partial mitigations for the report, including WhatsApp Baileys timing knobs, reconnect drain plumbing for generic queued outbound sends, bounded transcript/history work, and runtime-deps pruning. The central reliability asks are still not complete: liveness warnings still provide aggregate event-loop metrics without blocker attribution, and the direct WhatsApp auto-reply path still sends through the live socket/retry closure rather than a durable replay queue.

## What This Changes

The issue asks OpenClaw to diagnose and fix long event-loop stalls causing WhatsApp 408 disconnects, durably replay dropped replies after reconnect, expose Baileys timing controls, and garbage-collect stale plugin runtime-deps roots.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The maintainable end state is low-noise blocker attribution in liveness diagnostics, removal or offloading of the confirmed synchronous path, reconnect-safe WhatsApp auto-reply delivery with explicit idempotency, and docs/tests that distinguish already-shipped timing/runtime-deps mitigations from the remaining delivery and attribution work.

## Reproduction Assessment

Unclear. The issue has credible production liveness logs and 408 timing, and source inspection confirms remaining gaps, but there is no self-contained current-main reproduction for the 33-100s event-loop stalls or the exact dropped auto-reply-after-disconnect case.

## Solution Assessment

No. Current main is a partial mitigation set, not the complete solution; the narrowest maintainable fix still needs blocker attribution plus a deliberate durable replay design for direct WhatsApp auto-replies.
