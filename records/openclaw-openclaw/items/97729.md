## Summary

Keep open for maintainer review: the patch appears correct and has sufficient live-output proof, but merging intentionally expands the approved desktop node command surface and changes pairing authorization for exec-approval management commands.

## What This Changes

Adds a Gateway exec-approvals node command preflight, adds system.execApprovals.get/set to desktop node command allowlists, makes those commands admin-scoped for pairing, and adds focused unit/integration coverage.

## Best Possible Solution

Land this PR or an equivalent narrow Gateway fix after maintainers explicitly accept the admin-scoped exec-approval command surface, then retire weaker duplicate candidate PRs.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.11 relay exec.approvals.node.get/set without a command-surface preflight, and the linked issue gives concrete Docker/Windows-node steps. I did not rerun the full Windows node setup in this read-only review.

## Solution Assessment

Yes, this is the best code layer if maintainers accept the command-surface policy: the Gateway relay should validate against the shared node command policy before invoking. A CLI-only guard would leave other Gateway callers exposed, while a generic NodeRegistry guard would conflict with the dedicated exec.approvals.node.* boundary.
