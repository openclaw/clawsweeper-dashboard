## Summary

Keep open: the PR targets the right gateway bug and includes useful live RPC proof, but the new relay guard checks only the stored approved command list and can bypass the current `gateway.nodes.denyCommands` policy that generic `node.invoke` still enforces.

## What This Changes

The branch adds a gateway preflight for `exec.approvals.node.get/set`, extends desktop node command allowlists and pairing admin scopes for exec-approval commands, and adds gateway/infra regression tests.

## Best Possible Solution

Keep the gateway-side exec approvals preflight, but implement it through the shared node command policy and add `denyCommands` regression coverage before merge.

## Reproduction Assessment

Yes, source-reproducible rather than live-run here: connect a node whose approved commands include `system.execApprovals.get/set`, then deny one of those command IDs in `gateway.nodes.denyCommands` and call `exec.approvals.node.*`. Generic `node.invoke` would recheck policy, while this PR's relay guard would proceed because `nodeSession.commands` still includes the command.

## Solution Assessment

No, not yet. The gateway relay is the right layer for the bug fix, but the preflight needs to reuse the shared node command policy instead of a direct `commands.includes` check.
