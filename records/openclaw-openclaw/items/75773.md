## Summary

Keep open. The PR is aimed at a real WhatsApp reconnect bug, but it should not merge as-is because the pinned Baileys version uses status 428 for generic `connectionClosed` paths and the linked live logs show intermittent recovery after 428, so blanket terminal handling can stop recoverable sessions with relink guidance.

## What This Changes

The PR changes the WhatsApp Web monitor to treat any opening or post-open HTTP/Boom 428 as a terminal non-retryable relink condition and adds two e2e regression tests.

## Best Possible Solution

Keep the fix in the WhatsApp plugin, but only stop with relink guidance when the 428 payload or connection context proves an invalid-session condition; leave generic Baileys `connectionClosed` 428 on the reconnect path, add focused regression coverage, and include a changelog entry.

## Reproduction Assessment

Yes, for a source-level path: inject a Boom-like status 428 from `waitForWaConnection` or emit a listener close with status 428, and current main either rethrows it or treats it as retryable. The scheduled live recurrence was not reproduced in this read-only review.

## Solution Assessment

No. The WhatsApp monitor/controller boundary is the right place, but blanket 428 terminal handling is too broad for Baileys 7.0.0-rc.9; the safer fix is to require invalid-session or relink-required proof before stopping.
