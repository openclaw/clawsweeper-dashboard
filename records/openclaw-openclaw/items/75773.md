## Summary

Keep open: the WhatsApp 428 retry path is a plausible narrow fix, but the new agents resource-loader mitigation does not actually bypass Pi's synchronous resource discovery, and the current head lacks after-fix proof for the Windows/resource-loader behavior.

## What This Changes

The PR routes WhatsApp opening-phase 428 errors through reconnect handling, adds 428 regression coverage and changelog entries, and changes embedded Pi runs to construct a resource loader with Pi discovery flags disabled.

## Best Possible Solution

Keep the WhatsApp 428 retry fix, but replace the agents resource-loader mitigation with a dependency-aware path that actually bypasses Pi package/resource discovery, then prove the Windows stall path with live logs or a focused dependency-level regression test.

## Reproduction Assessment

Yes for source-level reproduction: current main rethrows opening-phase 428 setup errors, and the upstream Pi loader source shows the new no* flags do not bypass synchronous auto-discovery. I did not reproduce the full Windows 11 live stall.

## Solution Assessment

No as submitted. The WhatsApp retry change is the right ownership boundary, but the agents fix filters loaded resources after upstream discovery has already happened, so it is not the narrow reliable fix for the Windows resource-loader stall.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Bypass Pi discovery before reloading resources:** `src/agents/pi-embedded-runner/resource-loader.ts:19-21`
  - body: This helper only passes `noExtensions`/`noSkills`/related flags into `DefaultResourceLoader`, but upstream `reload()` still calls `packageManager.resolve()` first, and that path auto-discovers `.pi` and `.agents` resources with synchronous filesystem walks before those flags are applied. That leaves the Windows session-resource-loader stall described in https://github.com/openclaw/openclaw/issues/77443 in place for users with slow resource trees; use a loader path or upstream option that skips `packageManager.resolve()`/auto-discovery entirely for embedded runs.
  - confidence: 0.86
