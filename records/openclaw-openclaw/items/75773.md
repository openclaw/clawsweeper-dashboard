## Summary

Keep open. The PR targets a real WhatsApp 428 failure path, but the current patch blanket-classifies Baileys 428 as terminal relink-required even though the dependency treats 428 as generic `connectionClosed` and reconnect-safe.

## What This Changes

The PR changes WhatsApp Web monitoring to stop with relink guidance on HTTP/Boom 428 during opening or after a close, and adds two regression tests for those paths.

## Best Possible Solution

Keep the fix in the WhatsApp plugin, but make terminal 428 handling depend on a proven invalid-session or relink-required signal while preserving reconnects for generic Baileys `connectionClosed` 428, with regression tests and a changelog entry.

## Reproduction Assessment

Yes for source-level reproduction: make `openConnection` reject with a Boom-like 428 and current main rethrows it from the monitor catch, or emit a post-open 428 and current main retries it. The exact live scheduled WhatsApp backend recurrence was not reproduced.

## Solution Assessment

No. The WhatsApp monitor/controller boundary is the right place, but treating every 428 as terminal is not the narrowest maintainable solution because Baileys uses 428 for generic reconnect-safe connection closure.
