## Summary

Keep open. The linked WhatsApp 428 regression is valid, but this PR’s blanket terminal handling for every 428 is unsafe because the pinned Baileys dependency uses 428 for generic `connectionClosed`, and related live logs show 428/408 cycles can reconnect.

## What This Changes

The PR makes WhatsApp Web opening and post-open HTTP 428 closes stop with relink guidance and adds two e2e regression tests.

## Best Possible Solution

Keep the fix in the WhatsApp plugin, but classify 428 as terminal only when the error proves an invalid session or relink-required condition; keep generic Baileys `connectionClosed` 428 on the reconnect path and add a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by making `openConnection` reject with a Boom-like 428 or by emitting a listener close with status 428; the exact scheduled live recurrence was not reproduced locally.

## Solution Assessment

No. The monitor/controller boundary is the right place, but stopping all 428s is not the narrowest maintainable fix because Baileys also uses 428 for generic connection closure.
