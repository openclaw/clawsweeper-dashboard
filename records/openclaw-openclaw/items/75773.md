## Summary

Keep open. The linked bug is real, but this patch should not merge as-is because Baileys 7.0.0-rc.9 uses status 428 as the generic `connectionClosed` reason and the related live report shows 428/408 flapping that later reconnects, so blanket relink-required handling risks turning transient disconnects into unnecessary channel stops.

## What This Changes

The PR changes the WhatsApp web monitor to stop with relink guidance on HTTP/Boom 428 during opening and post-open close handling, and adds two regression tests for those paths.

## Best Possible Solution

Keep the fix scoped to the WhatsApp plugin, but classify 428 as terminal only when the monitor can identify a true invalid-session or relink-required condition; otherwise keep generic Baileys `connectionClosed` 428 on the reconnect path with clearer operator diagnostics, focused tests, and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by making `openConnection` reject with a Boom-like 428 or by emitting a post-open close with status 428; current main rethrows or retries, while this PR makes both paths terminal. The exact scheduled live recurrence is supported by logs but was not reproduced against WhatsApp’s backend here.

## Solution Assessment

No. The WhatsApp monitor is the right boundary, but treating every Baileys 428 as non-retryable is not the safest implementation because upstream uses 428 for generic connection-closed events that can be transient.
