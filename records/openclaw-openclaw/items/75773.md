## Summary

Keep open. The PR targets a real WhatsApp Web regression and the main status-classification direction is narrow, but the current diff tells operators to relink after an opening-phase 428 without clearing the cached Web auth state first, so the suggested CLI relink can reuse the same invalid session; it also still lacks the required changelog entry.

## What This Changes

The PR changes WhatsApp Web monitoring so HTTP/Boom status 428 stops with relink guidance instead of retrying or surfacing as a generic channel exit, and adds regression tests for opening and post-open 428 paths.

## Best Possible Solution

Keep the fix plugin-local in WhatsApp Web monitoring, but clear cached Web auth for opening-phase 428 before giving relink guidance, preserve the post-open 428 terminal handling, add focused auth-clear coverage, and add a single-line Unreleased Fixes changelog entry crediting @dataCenter430. Keep #75736 open until the corrected PR merges.

## Reproduction Assessment

Yes. Make monitorWebChannel encounter a Boom-like object with output.statusCode 428 during openConnection, or a post-open close reason with status 428; current main rethrows or retries, and the PR tests exercise those two paths.

## Solution Assessment

No, not as currently proposed. The WhatsApp monitor is the right boundary, but the opening-phase 428 path should clear the stale auth state, or explicitly direct operators to logout first, before the relink command is advertised as the recovery path.
