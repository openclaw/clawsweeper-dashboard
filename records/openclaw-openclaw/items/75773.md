## Summary

Keep open. The PR is scoped to the right WhatsApp plugin area, but it is not merge-ready because it treats every Baileys 428 as terminal even though the pinned dependency uses 428 for generic `connectionClosed` events and linked live logs show intermittent 428 reconnect recovery.

## What This Changes

The PR changes WhatsApp Web monitoring to stop with relink guidance on HTTP/Boom 428 during both opening and post-open close handling, and adds two regression tests for those cases.

## Best Possible Solution

Keep the fix in the WhatsApp plugin, but classify 428 as terminal only when the code can distinguish a real invalid-session/precondition failure from Baileys' generic `connectionClosed`; otherwise route opening-phase 428 through WhatsApp-specific reconnect/status handling with clearer diagnostics, focused tests, and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction exists by making `waitForWaConnection`/`openConnection` reject with a Boom-like 428, and #75736 now also has live logs showing intermittent 408/428 reconnect churn.

## Solution Assessment

No. Handling the failure in the WhatsApp monitor/controller boundary is right, but this PR's all-428 terminal classification is not the safest solution because the upstream dependency uses 428 for generic connection closures.
