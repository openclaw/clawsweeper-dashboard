## Summary

Keep open. The WhatsApp bug is real and current main still mishandles the reported opening-phase 428 path, but this PR over-classifies Baileys 428 as terminal even though upstream uses it for generic WebSocket connection-closed events; it also still needs the required changelog entry.

## What This Changes

The PR changes the WhatsApp monitor to stop with relink guidance on HTTP/Boom status 428 during opening and post-open closes, and adds two regression tests for those paths.

## Best Possible Solution

Keep the fix scoped to the WhatsApp plugin, but classify 428 only when the code can distinguish a terminal invalid-session condition from Baileys' generic connectionClosed; otherwise route opening-phase 428 into the normal reconnect loop with clearer operator diagnostics and add focused tests plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by making waitForWaConnection/openConnection reject with a Boom-like 428 or by emitting a post-open close reason with status 428; current main rethrows or retries, and the PR tests cover those seams.

## Solution Assessment

No. Handling the bug in the WhatsApp monitor is the right boundary, but treating every Baileys 428 as non-retryable is not the safest implementation because upstream uses 428 for generic connectionClosed events.
