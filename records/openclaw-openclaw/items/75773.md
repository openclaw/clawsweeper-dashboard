## Summary

Keep open. The linked WhatsApp regression is plausible, but the PR's blanket HTTP/Boom 428 terminal handling conflicts with the pinned Baileys contract where 428 is the generic `connectionClosed` status, so this needs WhatsApp owner review and a narrower invalid-session signal before merge.

## What This Changes

The PR changes the WhatsApp Web monitor to stop with relink guidance on HTTP/Boom 428 during opening and post-open close handling, and adds two regression tests for those paths.

## Best Possible Solution

Keep the fix inside the WhatsApp plugin, but classify 428 as terminal only when the monitor can identify a true invalid-session or relink-required condition; otherwise keep generic Baileys `connectionClosed` 428 on the reconnect path with clearer diagnostics, focused retryable and terminal tests, and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level path exists by making `openConnection` reject with a Boom-like 428 or by emitting a post-open close with status 428; current main rethrows or retries, while this PR makes both terminal. The scheduled live recurrence was not verified against WhatsApp's backend here.

## Solution Assessment

No. The WhatsApp monitor is the right boundary, but blanket non-retryable handling for 428 is not the safest solution because the pinned Baileys dependency uses 428 for generic connection-closed events as well as the reported termination message.
