## Summary

Keep open. The PR targets a real WhatsApp reconnect gap, but the current patch treats every Baileys 428 as terminal even though Baileys documents 428 as generic `connectionClosed` and safe to reconnect, so this needs WhatsApp owner judgment before merge.

## What This Changes

The PR changes the WhatsApp monitor to stop with relink guidance on HTTP/Boom status 428 during opening and post-open closes, and adds two regression tests for those paths.

## Best Possible Solution

Keep the fix scoped to the WhatsApp plugin, but classify 428 as terminal only when the code can distinguish an invalid or expired session from Baileys' generic connectionClosed condition; otherwise keep generic 428s on the reconnect path with clearer operator diagnostics, focused regression tests, and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by making waitForWaConnection/openConnection reject with a Boom-like 428 or by emitting a post-open close reason with status 428; current main rethrows or retries, and the PR tests exercise those seams.

## Solution Assessment

No. Handling this in the WhatsApp monitor is the right boundary, but treating every Baileys 428 as non-retryable is not the safest solution because upstream uses 428 for generic connectionClosed events.
