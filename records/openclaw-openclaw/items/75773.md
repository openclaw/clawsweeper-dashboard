## Summary

Keep open. The PR addresses a plausible WhatsApp regression, but the current patch makes every Baileys 428 terminal even though the pinned Baileys version uses 428 for generic `connectionClosed` socket/query paths; that policy needs WhatsApp owner review before merge.

## What This Changes

The PR changes the WhatsApp Web monitor to stop with relink guidance on HTTP/Boom 428 during opening and post-open close handling, and adds two regression tests for those paths.

## Best Possible Solution

Keep the fix inside the WhatsApp plugin, but classify 428 as terminal only when the monitor can identify a true invalid-session, precondition, or relink-required payload; otherwise keep generic Baileys `connectionClosed` 428 on the reconnect path with clearer diagnostics, focused terminal/retry tests, and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by making `openConnection` reject with a Boom-like 428 or by emitting a post-open close with status 428; current main rethrows or retries, while this PR makes both terminal.

## Solution Assessment

No. The WhatsApp monitor is the right boundary, but blanket status-code classification is not the safest fix because the pinned Baileys dependency uses 428 for ordinary `connectionClosed` paths as well as the reported termination message.
