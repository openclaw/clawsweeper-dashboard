## Summary

Keep open and do not merge as-is. The PR targets the right WhatsApp plugin boundary and adds useful regression coverage, but the patch classifies every Baileys 428 as terminal even though the dependency treats 428 as a generic connection-closed condition and the linked live logs show intermittent recovery after 428/408 flapping.

## What This Changes

The PR changes the WhatsApp Web monitor to stop with relink guidance on HTTP/Boom 428 during opening and post-open close handling, and adds two regression tests for those paths.

## Best Possible Solution

Keep the fix in the WhatsApp plugin, but classify 428 as terminal only when the monitor can identify an invalid-session or relink-required condition; otherwise keep generic Baileys `connectionClosed` 428 on the reconnect path with clearer diagnostics, focused tests, and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level path exists by making `openConnection` reject with a Boom-like 428 or by emitting a post-open close with status 428; current main rethrows or retries, while this PR makes both terminal. The scheduled live recurrence is supported by logs but was not reproduced against WhatsApp’s backend here.

## Solution Assessment

No. The WhatsApp monitor is the correct boundary, but blanket non-retryable handling for 428 is not the safest solution because the pinned Baileys dependency uses 428 for generic connection-closed events, not only invalid sessions.
