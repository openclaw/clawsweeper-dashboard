## Summary

Keep open. The PR targets a real WhatsApp 428 handling gap, but the patch makes every Baileys 428 terminal even though Baileys uses 428 for generic connection-closed paths and the linked live logs show recovery after 428.

## What This Changes

The PR changes WhatsApp Web monitoring so opening and post-open HTTP/Boom 428 closes stop with relink guidance, and adds two regression tests for those paths.

## Best Possible Solution

Keep the fix inside the WhatsApp plugin, but classify 428 as terminal only when the error proves an invalid session or relink-required condition; keep generic Baileys `connectionClosed` 428 on the reconnect path with focused tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by making `openConnection` reject with a Boom-like 428 or by emitting a post-open close with status 428; the exact live scheduled recurrence was not reproduced here.

## Solution Assessment

No. The WhatsApp monitor/controller boundary is right, but blanket non-retryable 428 handling is not the safest fix because the pinned Baileys dependency also uses 428 for transient generic connection closures.
