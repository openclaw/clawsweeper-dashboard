## Summary

Keep this PR open for review rather than cleanup-closing it: the linked bug is still open, but the latest patch still has source-backed blockers around effective default resolution and cross-channel no-arg `/verbose` behavior, plus missing real Telegram proof.

## What This Changes

The PR adds `argsMenu: "auto"` to `/verbose`, updates the registry test to expect a no-arg menu, and prepends current verbose/reasoning levels to Telegram native command menu titles.

## Best Possible Solution

Preserve the no-arg current-level contract while adding buttons: resolve menu titles through the same effective-level chain used by directive status and either update every affected native menu surface or scope `/verbose` menu behavior to surfaces that can show the current level.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main lacks `/verbose` argsMenu, while the PR head intercepts no-arg `/verbose` as a menu and still misses the current effective-level contract on some paths.

## Solution Assessment

No. Adding buttons is the right direction, but this implementation should reuse the existing effective-level resolution contract and preserve current-level status on every surface affected by the shared registry change.

## Review Findings

Overall confidence: 0.87

Full review comments:

- **[P2] Use the full effective-level default chain:** `extensions/telegram/src/bot-native-commands.ts:1099-1102`
  - body: The menu title now falls back through `resolveAgentConfig`, but that helper does not include `agents.defaults.reasoningDefault` and can be absent when only global defaults are configured.   - confidence: 0.89

That breaks the documented no-arg status behavior outside Telegram unless those menu surfaces also get current-level titles or the change is scoped narrower.
  - confidence: 0.86
