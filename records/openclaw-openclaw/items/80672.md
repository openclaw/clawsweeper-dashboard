## Summary

Current `main` and the latest release already implement the cron announce delivery repair: isolated cron delivery now resolves the explicit outbound target into a destination session route and only reports delivery after a durable channel send succeeds.

## What This Changes

The issue reports isolated cron jobs with `delivery.mode: "announce"` completing successfully while the configured Feishu delivery is marked `not-delivered`.

## Best Possible Solution

Keep the shipped route-based cron delivery implementation and ask for a new issue only if Feishu isolated announce delivery still reproduces on `v2026.6.6` or newer with fresh logs.

## Reproduction Assessment

Yes at source level for the old failure class: the report gives a concrete isolated cron announce configuration, and current source/tests show that path now resolves and sends through the outbound target route. I did not run a live Feishu delivery because this was a read-only review without credentials.

## Solution Assessment

Yes. The shipped route-based direct delivery path is a better fit than the closed PR's generic stable failure-session key because it binds delivery and transcript metadata to the actual destination channel session.
