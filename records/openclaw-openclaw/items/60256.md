## Summary

Keep open: current `main` and `v2026.6.8` still preserve cron `delivery.to: "env:..."` as raw Discord target text, while the docs describe concrete Discord targets such as `channel:<id>` or `user:<id>`. This needs a maintainer decision on whether cron should support env/SecretRef-backed target indirection or reject it earlier with a clearer diagnostic.

## What This Changes

The issue requests cron Discord announce delivery to treat `delivery.to: "env:DISCORD_BACKUPS_CHANNEL"` as an environment-backed channel target and record successful delivery as `ok` / `delivered`.

## Best Possible Solution

Choose the cron delivery-target contract, then either add documented env/SecretRef-backed target resolution with focused regression and live Discord proof, or reject unsupported `env:` targets during cron create/update with a clear diagnostic and docs pointing to concrete `channel:<id>` / `user:<id>` targets.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current `main` and `v2026.6.8` preserve `delivery.to: "env:..."` as raw target text and can persist `unknown` delivery state when requested delivery returns no delivered ack.

## Solution Assessment

Unclear. Supporting env-backed `delivery.to` may be useful, but the documented cron contract uses concrete targets and SecretRef shorthand is limited to supported SecretInput fields, so the maintainable fix depends on maintainer direction before implementation.
