## Summary

Keep open. Current main still treats `delivery.to: "env:DISCORD_BACKUPS_CHANNEL"` as a literal Discord target, while the documented cron target contract only describes explicit targets such as `channel:<id>` / `user:<id>` and config env substitution uses `${VAR}` rather than `env:` delivery syntax.

## What This Changes

The issue asks for Discord cron delivery targets written as environment-variable references to resolve correctly and persist an `ok` / `delivered` result after successful delivery.

## Best Possible Solution

Define one central cron delivery target contract: either resolve env-backed delivery targets through the existing `${VAR}` or SecretRef model and document it, or reject `env:` at cron add/update/preview time with a clear validation error.

## Reproduction Assessment

Yes, by source inspection: an isolated announce cron job with `channel: "discord"` and `to: "env:DISCORD_BACKUPS_CHANNEL"` is preserved as a literal target through planning and target resolution. A non-best-effort Discord send failure can then return an error without `delivered`, which the timer persists as `unknown`.

## Solution Assessment

Unclear. The narrow maintainable fix belongs in central cron delivery target validation/resolution, but maintainers need to choose whether `env:` is a supported public target form or an invalid configuration that should be rejected earlier.
