## Summary

Keep open. Current main still accepts `delivery.to: "env:DISCORD_BACKUPS_CHANNEL"` as a literal Discord target instead of resolving it or rejecting it up front, so the reported `error` / `unknown` delivery state remains reachable.

## What This Changes

The issue asks cron announce delivery to resolve an env-var Discord target so successful sends persist `ok`, `delivered`, and `lastDeliveryStatus: "delivered"`.

## Best Possible Solution

Define one central cron delivery target contract: either resolve env-backed targets through the existing `${VAR}`/SecretRef activation model and document it, or reject `env:` at create/update/preview time with an actionable validation error.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: create an isolated announce cron job with `channel: "discord"` and `to: "env:DISCORD_BACKUPS_CHANNEL"`; current main carries that exact string into Discord delivery and can persist `unknown` when required delivery fails or omits `delivered`.

## Solution Assessment

Unclear. The narrow maintainable fix should be central cron target resolution or validation before channel-specific outbound normalization, but maintainers need to choose whether `env:` is public syntax or an invalid target form.
