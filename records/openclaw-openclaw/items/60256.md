## Summary

Keep open: current main and the latest release still treat `delivery.to` as a plain delivery target, while the documented cron target syntax uses concrete channel/user prefixes rather than `env:` indirection. The remaining decision is whether to add env/SecretRef target resolution or reject this syntax earlier with a clearer diagnostic.

## What This Changes

The issue asks for cron Discord delivery configured with `delivery.to: "env:..."` to finish as delivered/ok instead of error/unknown when the agent run succeeds.

## Best Possible Solution

Decide the delivery-target contract first: either add a safe env/SecretRef-backed resolver for cron targets with docs and regression coverage, or reject `env:` targets at cron create/update time with guidance to use concrete `channel:<id>`/`user:<id>` targets or documented config substitution where applicable.

## Reproduction Assessment

No high-confidence live reproduction of the exact Discord delivered-but-error claim was established. Source inspection does show with medium confidence that current main does not dereference `delivery.to: "env:..."` and can persist `unknown` delivery state when requested delivery returns no delivered ack.

## Solution Assessment

Unclear. Supporting `env:` targets may be useful, but the best fix depends on maintainer product direction; the safer default is explicit validation and diagnostics unless env/SecretRef target indirection is accepted as a cron contract.
