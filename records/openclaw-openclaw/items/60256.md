## Summary

Keep open. Current main still treats `delivery.to: "env:DISCORD_BACKUPS_CHANNEL"` as a literal Discord target rather than resolving it or rejecting it before execution, so a run that already used the message tool can still be marked failed by the fallback delivery path.

## What This Changes

Cron delivery should not report a failed Discord delivery when the target is supplied through an environment-variable reference and the message was otherwise delivered.

## Best Possible Solution

Define and enforce one central cron delivery target contract: either resolve env-backed targets through the existing documented secret/config model, or fail create/update/preview early with an actionable validation error for `env:` targets.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: create an isolated announce cron job with Discord `delivery.to` set to `env:DISCORD_BACKUPS_CHANNEL`; current main carries that literal into Discord delivery and can still fail the cron run after the agent sends a message elsewhere.

## Solution Assessment

Unclear. The narrow maintainable implementation belongs in central cron delivery target resolution or validation, but maintainers need to decide whether `env:` is supported public syntax or an invalid target form.
