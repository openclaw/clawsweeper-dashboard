## Summary

Keep open. Current main still treats `delivery.to: "env:DISCORD_BACKUPS_CHANNEL"` as an ordinary literal target: cron planning preserves it, delivery target resolution forwards it, and Discord outbound normalization returns nonnumeric targets unchanged. That leaves the reported error/unknown delivery state path reachable when the agent succeeds but delivery target resolution or send fails.

## What This Changes

The issue requests that cron announce delivery resolve an env-var Discord target so successful sends persist `ok` and `delivered` instead of `error` and `unknown`.

## Best Possible Solution

Define a single cron delivery target resolution contract before outbound normalization. If env-backed targets are supported, resolve them centrally with the existing env-template or SecretRef machinery and document the syntax; otherwise reject `env:` during job create/edit with a clear error so it cannot fail later as a literal Discord target.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: create an isolated cron announce job with `channel: "discord"` and `to: "env:DISCORD_BACKUPS_CHANNEL"`; current main carries that exact string into Discord delivery and can persist `unknown` when delivery does not report success.

## Solution Assessment

Unclear until maintainers choose the public syntax. The narrowest maintainable fix is centralized cron delivery target resolution or validation before channel-specific outbound normalization, not a Discord-only special case.
