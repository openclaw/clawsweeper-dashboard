## Summary

Keep open. Current main still does not resolve `env:` cron delivery targets or reject them early, so the literal value can flow into Discord delivery and leave a successful message-tool send untrusted by the fallback delivery path.

## What This Changes

The issue requests that Discord cron announce delivery handle `delivery.to: "env:DISCORD_BACKUPS_CHANNEL"` without marking a successful delivered run as failed.

## Best Possible Solution

Define one central cron delivery-target contract: either support env/SecretRef-backed targets before target matching and outbound send, or reject `env:` at create/update/preview with an actionable validation error.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an isolated announce cron job with Discord `delivery.to` set to `env:DISCORD_BACKUPS_CHANNEL` preserves that literal target through planning, target resolution, Discord normalization, and message-tool match checks.

## Solution Assessment

Unclear. The narrow code owner is central cron delivery target resolution or validation, but maintainers need to decide whether `env:` is supported public syntax or an invalid target form before implementation.
