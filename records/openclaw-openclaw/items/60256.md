## What This Changes

The issue asks for cron Discord announce delivery using an `env:` channel target to end with successful delivery status instead of an error/unknown run state.

## Best Possible Solution

Define one central cron delivery-target contract, then either resolve env-backed targets before planning/status evaluation or reject them during add/edit/preview with an actionable validation error and docs coverage.

## Reproduction Assessment

Yes, by source inspection rather than a live run. Current main and `v2026.5.19` preserve `env:DISCORD_BACKUPS_CHANNEL` through cron planning, delivery target resolution, and Discord outbound normalization as a literal target.

## Solution Assessment

Unclear. A Discord-only special case would be the wrong boundary; the maintainable solution is a central cron delivery-target contract with either env resolution or focused validation/rejection tests.
