## Summary

Keep open: current main and the latest release still treat cron `delivery.to` as literal target text, so `env:...` delivery-target indirection remains an undecided cron/Discord contract rather than an implemented fix.

## What This Changes

The issue asks for Discord cron announce delivery to handle an environment-variable-backed target without marking an otherwise successful run as failed.

## Best Possible Solution

Choose the cron delivery-target contract, then either implement documented env/SecretRef-backed target resolution with focused cron and live Discord proof, or reject unsupported `env:` targets during cron create/edit with a clear diagnostic.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main and v2026.6.10 preserve `delivery.to: "env:..."` as raw target text, and existing cron state handling can report `unknown` when delivery is requested but not verified.

## Solution Assessment

Unclear. Supporting env-backed `delivery.to` may be useful, but the documented cron target examples use concrete targets and SecretRef shorthand is limited to supported credential fields, so maintainers need to choose support versus fail-fast validation first.
