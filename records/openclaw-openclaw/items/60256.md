## Summary

Keep open. Current main and the latest release still route `delivery.to` through normal channel target parsing, so `env:DISCORD_BACKUPS_CHANNEL` is treated as a literal Discord channel target; the remaining decision is whether env-backed targets should become supported cron config or fail validation with clearer guidance.

## What This Changes

The issue requests correct cron run and delivery status when a Discord announce target is configured as an environment-variable reference.

## Best Possible Solution

Make one explicit contract decision for cron delivery targets, then update runtime resolution, validation/status behavior, docs, and focused regression coverage so env-backed targets either work intentionally or fail before a misleading run result is recorded.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection does show the current-main path that treats `env:DISCORD_BACKUPS_CHANNEL` as a literal Discord target and can leave delivery status `unknown` on error.

## Solution Assessment

Unclear until maintainers choose the contract. Supporting env expansion in cron delivery targets is plausible, but the narrower alternative is to reject `env:` targets during cron config validation and keep explicit `channel:<id>` or `user:<id>` as the documented path.
