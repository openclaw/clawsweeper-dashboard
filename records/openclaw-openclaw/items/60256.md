## Summary

Keep open: current main does not show a supported path that dereferences `delivery.to: "env:..."` for cron Discord delivery, and docs still describe concrete channel/user targets rather than env-backed delivery targets. The remaining question is a product/config contract decision, not a stale fixed bug.

## What This Changes

The issue asks cron announce delivery to resolve a Discord `delivery.to` value from an environment variable and record successful runs as `ok`/`delivered`.

## Best Possible Solution

Decide the contract first: either add a safe SecretRef/env-backed cron delivery-target resolver with tests and docs, or reject `env:` targets early with clear validation telling users to store concrete `channel:<id>`/`user:<id>` targets.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the env-style target is passed as a raw delivery target and the status mapping can produce `unknown` when delivery fails without a delivered ack.

## Solution Assessment

Unclear. Supporting `env:` in `delivery.to` may be useful, but the best fix depends on whether maintainers want env/SecretRef indirection for delivery targets or stricter validation and docs for concrete targets only.
