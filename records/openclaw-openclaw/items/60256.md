## Summary

Keep open: current main and the latest stable release still treat cron `delivery.to` as literal target text, while docs only define concrete Discord targets, so `env:...` recipient indirection remains an undecided cron/Discord contract rather than implemented behavior.

## What This Changes

Request that cron Discord announce delivery resolve `delivery.to: "env:DISCORD_BACKUPS_CHANNEL"` to the environment variable's channel target and report successful delivered runs correctly.

## Best Possible Solution

Maintainers should choose the contract, then either document and implement env/SecretRef-backed cron delivery targets with validation and live Discord proof, or reject `env:` targets at cron create/edit/runtime with a clear diagnostic.

## Reproduction Assessment

No live current-main repro was run. Source inspection shows `delivery.to` reaches cron and Discord target parsing as literal `env:DISCORD_BACKUPS_CHANNEL`, and cron state can report `unknown` when requested delivery does not carry verified delivered metadata.

## Solution Assessment

Unclear. Supporting env-backed `delivery.to` may be useful, but current docs require explicit target syntax and env/SecretRef target semantics would expand cron config behavior; fail-fast validation may be safer if maintainers do not want that surface.
