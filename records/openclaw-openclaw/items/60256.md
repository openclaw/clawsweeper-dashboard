## Summary

Keep open: current main and v2026.6.8 still treat cron `delivery.to: "env:..."` as raw target text rather than an environment-backed Discord target, and no related merged or open PR owns that exact contract. The remaining choice is whether cron should support env-backed target indirection or reject it earlier with a clearer validation error.

## What This Changes

The issue asks for cron Discord delivery targets using `env:` values to report successful delivery instead of leaving the run in an error or unknown delivery state.

## Best Possible Solution

Decide the cron delivery-target contract first, then either add documented env/SecretRef-backed target resolution with regression coverage or reject unsupported `env:` targets during cron create/update with guidance to use concrete `channel:<id>` or `user:<id>` targets.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main and v2026.6.8 preserve `delivery.to: "env:..."` as raw target text and can persist `unknown` delivery state when requested delivery returns no delivered ack.

## Solution Assessment

Unclear. Supporting `env:` target indirection may be useful, but the documented cron contract uses concrete targets, so the maintainable fix depends on maintainer direction before implementation.
