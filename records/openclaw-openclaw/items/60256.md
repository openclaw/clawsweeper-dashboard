## Summary

Keep open: current main and the latest release still do not dereference `env:` in cron `delivery.to`, while the cron docs describe concrete Discord targets such as `channel:<id>` and `user:<id>`. No merged related item owns this exact contract choice, so maintainers still need to decide whether to support env-backed target indirection or reject it earlier with a clearer diagnostic.

## What This Changes

The issue requests that cron Discord announce delivery using `delivery.to: "env:DISCORD_BACKUPS_CHANNEL"` finish with delivered/ok status when the underlying delivery succeeds.

## Best Possible Solution

Decide the cron delivery-target contract first, then either add a documented env/SecretRef-backed target resolver with focused regression coverage or reject unsupported `env:` targets during cron create/update with guidance to use concrete `channel:<id>` or `user:<id>` targets.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show current main preserves `delivery.to: "env:..."` as raw text and can persist `unknown` delivery state when requested delivery has no delivered ack.

## Solution Assessment

Unclear. Supporting `env:` target indirection may be useful, but the current documented contract uses concrete Discord targets, so the maintainable fix depends on maintainer direction before implementation.
