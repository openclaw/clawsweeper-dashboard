## Summary

Keep open. Current main still carries `env:DISCORD_BACKUPS_CHANNEL` as a literal Discord delivery target, the user-facing cron docs only describe explicit `channel:`/`user:` Discord targets, and no linked or searched PR currently owns the env-backed target contract.

## What This Changes

The issue requests cron announce jobs with Discord `delivery.to` set to an `env:` value to report `ok`/`delivered` instead of `error`/`unknown` when the message is delivered.

## Best Possible Solution

Define one central cron delivery-target contract, then either resolve env-backed targets before planning/matching/send or reject them during add/edit/preview with an actionable validation error and docs coverage.

## Reproduction Assessment

Yes by source inspection, not by a live run. Current main preserves `env:DISCORD_BACKUPS_CHANNEL` through cron planning, delivery target resolution, and Discord outbound normalization as a literal target.

## Solution Assessment

Unclear. A Discord-only special case would be the wrong boundary; the maintainable solution is a central cron delivery-target contract with focused validation or resolution tests.
