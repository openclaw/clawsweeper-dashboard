## Summary

Keep open. Current main and the latest shipped release still treat `env:DISCORD_BACKUPS_CHANNEL` as a raw Discord delivery target, while the docs only define explicit Discord target prefixes, so the env-backed cron target contract remains unresolved.

## What This Changes

The issue asks cron delivery to treat Discord targets supplied as `env:...` the same as explicit channel targets so successful delivery reports `ok` and `delivered`.

## Best Possible Solution

Define one central cron delivery-target contract, then either resolve env-backed targets before planning/status evaluation or reject them during add/edit/preview/run with an actionable validation error and docs coverage.

## Reproduction Assessment

Yes, by source inspection rather than a live run. Current main preserves `env:DISCORD_BACKUPS_CHANNEL` through cron planning and Discord outbound normalization as a literal target, then can record `unknown` delivery status when the non-best-effort delivery target fails unresolved.

## Solution Assessment

Unclear. A Discord-only special case would be the wrong boundary; the maintainable solution is a central cron delivery-target contract with either env resolution or focused validation/rejection tests.
