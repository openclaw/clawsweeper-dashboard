## Summary

Keep open. Current main and the latest shipped release still preserve `env:` as a literal Discord delivery target, while the docs only describe explicit target prefixes, so this needs a maintainer contract decision rather than cleanup closure.

## What This Changes

The issue requests cron announce delivery to support env-backed Discord channel targets without marking successful agent runs as delivery errors.

## Best Possible Solution

Define one central cron delivery-target contract, then either resolve env-backed targets before planning/matching/send or reject them at add/edit/preview with an actionable validation error and docs coverage.

## Reproduction Assessment

Yes by source inspection, not by a live run. An isolated announce job with Discord `delivery.to` set to `env:DISCORD_BACKUPS_CHANNEL` preserves that literal through planning, target resolution, Discord normalization, and message-tool delivery matching.

## Solution Assessment

Unclear. A Discord-specific special case would be the wrong boundary; the maintainable fix is a central cron delivery-target contract plus focused validation or resolution tests.
