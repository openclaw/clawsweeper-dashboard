Keep open. Current main still contains the reported HTTP SSE streaming hazards around bare tag-fragment buffering, delta-only SSE emission, and synchronous `[DONE]` finalization, so this should not be cleaned up as stale or already fixed.

The issue reports malformed or truncated OpenAI-compatible HTTP SSE output for assistant responses containing custom XML-style tags.

## Best Possible Solution

Fix the SSE stream pipeline so custom XML text is preserved and `[DONE]` is emitted only after all assistant content is flushed, while preserving existing `<final>/<think>` stripping behavior.

The reported bug is that `/v1/chat/completions` with `stream:true` can emit malformed custom XML-style assistant content by dropping a leading `<`, and can send `data: [DONE]`/close before the final assistant content chunk. First add focused failing regression coverage for the current behavior: split assistant text deltas such as `<` then `xiaohai-banli>...` must preserve the literal custom tag in the OpenAI-compatible SSE output, and final assistant content must be observed before `[DONE]` even when lifecycle/result completion happens around queued stream flushes. Keep the fix narrow: likely areas are the tag-fragment scanner/assistant stream data in `src/agents/pi-embedded-subscribe.ts` and `src/agents/pi-embedded-subscribe.handlers.messages.ts`, the SSE delta resolver in `src/gateway/agent-event-assistant-text.ts`, and finalization ordering in `src/gateway/openai-http.ts`; audit `src/gateway/openresponses-http.ts` for the same ordering hazard. Preserve existing stripped `<final>`, `<think>`, thinking, tool-call, usage, and disconnect behavior. Do not add a config option or change plugin/channel policy. Add a single user-facing Fixes changelog entry if the final patch changes shipped behavior.
