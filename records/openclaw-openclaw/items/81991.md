## What This Changes

The issue reports that OpenAI-compatible HTTP SSE streaming can drop a leading `<` and close before the final content chunk when assistant output contains custom XML-style tags.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82355, or an equivalent narrow patch with regression coverage and real SSE proof, then close this issue from the merge.

Yes, at source level: current main buffers a bare `<`, prefers assistant `delta` over cumulative `text`, and can synchronously close SSE while a `text_end` flush remains queued as a microtask. I did not run a live HTTP repro in this read-only review.
