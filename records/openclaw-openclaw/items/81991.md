Keep open: current main still has the implicated SSE, assistant-delta, and text-end flush paths, and I found no canonical duplicate or merged fix. ## What This Changes

The issue reports that `/v1/chat/completions` SSE can drop a leading `<` from streamed custom XML-style content and can send `[DONE]` before the final content chunk.

## Best Possible Solution

Add focused SSE regression coverage for custom XML-style tags and late final chunks, then fix the gateway/streaming delta and finalization path while preserving split `<final>`/thinking tag stripping.

## Reproduction Assessment

No high-confidence runtime reproduction was run. Source inspection matches the reported SSE and text-end timing surfaces closely enough to keep the issue open, but the final verdict needs a failing focused test or live SSE proof.

## Solution Assessment

No, the suggested bare-`<` predicate change is not the best fix because existing tests depend on buffering bare `<` to strip split `<final>` tags. A safer fix should derive SSE deltas from cumulative text when needed and/or defer finalization until pending text-end flushes are observable, with regressions for both cases.
