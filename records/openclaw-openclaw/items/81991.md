## Summary

Keep open. Current main still has both reported failure shapes in the HTTP SSE path: bare `<` is buffered as a possible stripped block tag, while `/v1/chat/completions` can synchronously write `[DONE]` and close before the text-end block-reply microtask emits its final assistant event.

## What This Changes

Investigate an HTTP SSE streaming bug where `/v1/chat/completions` can omit a leading `<` from custom XML-style content and close before the final content chunk is delivered.

## Best Possible Solution

Fix the stream pipeline so literal custom XML text is preserved in SSE deltas and final assistant content is emitted before `[DONE]`, while retaining the existing stripped `<final>/<think>` behavior.

## Reproduction Assessment

Yes, at source level: emit assistant deltas split as `<` then `xiaohai-banli>...` through the current stream handler, then finalize after `text_end`. I did not execute the HTTP repro because this review was read-only, but the relevant current-main branches are concrete.

## Solution Assessment

No patch is proposed yet; the suggested two-part direction is plausible and narrow. The repair should add focused agent-stream and HTTP SSE regression tests, and should explicitly check whether the `/v1/responses` finalizer shares the same timing hazard.
