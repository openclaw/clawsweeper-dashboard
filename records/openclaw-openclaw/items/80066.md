## Summary

Keep open. Current main still has a source-backed path that can surface the repeated Feishu warning, and the related open PR has not merged and appears to target only the tool-kind path while the warning is dispatched as a final reply.

## What This Changes

The issue reports repeated Feishu direct-message warnings that `AGENTS.md` edit failed even when the workspace file update succeeds.

## Best Possible Solution

Fix the stale warning either in the shared embedded-run warning policy or at the Feishu final-delivery boundary, with regression coverage that suppresses recovered edit/write warnings while preserving genuine final failures.

## Reproduction Assessment

Yes, source-level: current main builds mutating edit/write failures as `isError` warning payloads and then sends reply payloads through final dispatch to the Feishu dispatcher. I did not establish a live Feishu workspace reproduction.

## Solution Assessment

No, the currently linked PR is not clearly the best fix as written because it suppresses only `kind: "tool"` payloads while the source-backed warning path is a final reply. The safer fix should target the actual final warning path or the shared warning policy and preserve real final error delivery.
