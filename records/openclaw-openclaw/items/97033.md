## Summary

Keep open: the provider-local fix is the right boundary and the PR now adds regression tests, but the submitted precedence still lets model `contextWindow` outrank provider `contextTokens` caps, and the contributor proof is mock/local-only rather than a real LM Studio load log.

## What This Changes

The PR resolves LM Studio embedding preload context length from existing model/provider `contextTokens` and `contextWindow` config, passes it to `ensureLmstudioModelLoaded`, and adds embedding-provider unit tests.

## Best Possible Solution

Keep the provider-local fix, but resolve effective preload context as model `contextTokens`, provider `contextTokens`, model `contextWindow`, then provider `contextWindow`, with regression coverage and redacted real LM Studio load diagnostics before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main calls LM Studio embedding preload without `requestedContextLength`, while the load helper falls back to the large default context before capping by model limit. I did not run a live CUDA reproduction in this read-only review.

## Solution Assessment

No as submitted. The provider-local pass-through is the right layer, but the precedence should honor provider `contextTokens` caps before model `contextWindow`, and real LM Studio proof is still missing.
