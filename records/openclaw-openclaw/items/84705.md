## Summary

Keep this PR open. The hardening direction is useful, but the current patch is incomplete for the documented config snapshot surfaces, uses strict JSON parsing for JSON5 config backups, and only provides targeted test output rather than after-fix real behavior proof.

## What This Changes

The PR extends `openclaw secrets audit` to enumerate adjacent config backup filenames, scan them for plaintext config secret targets, and add regression coverage for a stale backup with a provider `apiKey`.

## Best Possible Solution

Merge this as a broader secrets-audit hardening only after it scans all managed adjacent config snapshots with active-config parsing semantics, reports unreadable snapshots clearly, and includes redacted real CLI proof.

## Reproduction Assessment

Yes. From source, create an adjacent `openclaw.json.pre-update` or JSON5-formatted `openclaw.json.bak` containing a plaintext secret; the proposed matcher/parser path will not report it.

## Solution Assessment

No. The PR is in the right audit surface, but the best fix should reuse config-compatible parsing and include all managed adjacent config snapshots rather than only a subset of `.bak` names.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Include managed config snapshots in backup audit:** `src/secrets/storage-scan.ts:34-38`
  - body: `listConfigBackupPaths()` only matches `.bak`-style names, but OpenClaw also writes adjacent `openclaw.json.pre-update`, `openclaw.json.rejected.*`, and `openclaw.json.clobbered.*` snapshots that can contain the same historical plaintext values. Leaving those out means the new audit can report clean while documented adjacent config copies still hold secrets.
  - confidence: 0.86
- **[P2] Parse backup configs with the config parser:** `src/secrets/audit.ts:269-272`
  - body: The backup scan goes through `readJsonObjectIfExists()`, which uses strict `JSON.parse`, while `openclaw.json` is documented and loaded as JSON5. A valid backup with comments or trailing commas will be skipped without a finding, so the scan should parse backups with config-compatible JSON5 handling and surface parse/oversize errors instead of treating them as clean.
  - confidence: 0.83
