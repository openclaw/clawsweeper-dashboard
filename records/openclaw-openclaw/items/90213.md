## Summary

Current main and v2026.6.5 still preserve conflicting legacy migration sources, so the warning loop remains real. Two open closing PRs already own the plugin-index repair, while the latest issue evidence also identifies a related Memory Core legacy-source case that needs coverage or a focused follow-up.

## What This Changes

Make `openclaw doctor --fix` reach a clean terminal state so safely superseded legacy plugin and memory sources stop producing recurring migration warnings.

## Best Possible Solution

Adopt source-specific migration rules that archive legacy inputs only when canonical SQLite completeness and identity are proven, retain actionable diagnostics for unresolved conflicts, and add second-run regression coverage for every confirmed source class.

## Reproduction Assessment

Yes. Current source deterministically leaves a conflicting legacy plugin index at its original path, the current test asserts that warning path, and reporters supplied repeated output on v2026.6.1 and v2026.6.5.

## Solution Assessment

No to unconditional conflict archival; the safer path is the narrow canonical-supersession approach represented by https://github.com/openclaw/openclaw/pull/90267, extended or paired with focused handling for the confirmed Memory Core variant.
