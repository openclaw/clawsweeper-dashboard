## Summary

Current main and the latest v2026.6.5 release still retain conflicting legacy migration inputs, so later commands can repeat the warning after doctor runs; this issue should remain open while the linked fix PRs are reviewed or consolidated.

## What This Changes

Stop legacy-state migration warnings from recurring after doctor repair when canonical SQLite state already conflicts with retained legacy plugin or memory sources.

## Best Possible Solution

Adopt one doctor-owned terminal-state policy that archives safely superseded legacy sources only when canonical SQLite data is demonstrably authoritative, while preserving true conflicts with specific remediation guidance and two-run regression coverage.

## Reproduction Assessment

Yes. Current source leaves a conflicting legacy plugin index in place, current tests assert that warning path, and the discussion contains repeated real Linux observations including v2026.6.5.

## Solution Assessment

No to blanket conflict archival. The narrower maintainable solution must prove the SQLite record supersedes the legacy source, archive it into a non-repeating end state, and leave unresolved drift visible and actionable.
