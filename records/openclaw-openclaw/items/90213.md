## What This Changes

The issue reports repeated legacy-state migration warnings after upgrade and `openclaw doctor --fix` when legacy files remain beside canonical SQLite state.

## Best Possible Solution

Land a maintainer-approved migration policy that retires legacy sources only when canonical SQLite state is proven authoritative across plugin index, Memory Core, plugin-state, task/flow, and channel sidecars, while preserving true conflicts with actionable warnings.

## Reproduction Assessment

Yes, at source level: seed a conflicting legacy `plugins/installs.json` or Memory Core legacy source beside already-populated SQLite state, then run the legacy-state migration path again. Current main leaves those sources in place for unresolved or row-exists cases, so later doctor/status/startup flows can warn again.

## Solution Assessment

Unclear until maintainers choose the compatibility rule. The right layer is core/plugin doctor state migration, but automatic archival needs a shared authoritative-SQLite predicate rather than one-off warning suppression.
