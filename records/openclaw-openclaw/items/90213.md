## Summary

Keep open: current main and the latest release still preserve legacy sources on conflict or already-migrated-row branches, so repeated warnings can recur after doctor; the open linked PRs cover only slices of the needed migration policy.

## What This Changes

Request is to stop recurring legacy state migration warnings after `openclaw doctor --fix` when SQLite already holds authoritative migrated state.

## Best Possible Solution

Land one maintainer-approved migration policy that retires legacy sources only when canonical SQLite state is proven authoritative, covers the plugin install-index and sibling row-exists cases, and keeps unsafe conflicts visible with actionable guidance.

## Reproduction Assessment

Yes at source level: leave a conflicting legacy `plugins/installs.json` or a Memory Core legacy source beside already-populated SQLite state, then run the legacy-state migration path again. Current main leaves those legacy sources in place, so later doctor/status/startup paths can warn again; I did not run a live packaged upgrade in this read-only review.

## Solution Assessment

No final implementation is selected yet. The right layer is core/plugin doctor state migration, but maintainers still need to choose the compatibility rule for when SQLite is authoritative enough to archive legacy sources across the sibling paths.
