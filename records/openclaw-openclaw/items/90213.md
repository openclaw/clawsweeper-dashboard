## Summary

Keep open: current main and the latest v2026.6.5 release still have source paths that leave legacy state sources in place after conflicts or already-migrated SQLite rows, so the warning can recur after doctor. Open PRs already track the plugin install-index part, but they are not merged and a later comment shows the same class also affects Memory Core legacy sources.

## What This Changes

The issue requests that post-upgrade legacy state migration warnings stop recurring after `openclaw doctor --fix` when canonical SQLite state already exists.

## Best Possible Solution

Land one maintainer-approved migration policy that retires legacy sources when canonical SQLite state is proven authoritative, covers both plugin install-index and Memory Core row-exists cases, and keeps unsafe conflicts visible with actionable guidance.

## Reproduction Assessment

Yes at source level: leave a conflicting legacy `plugins/installs.json` beside a shared SQLite installed-plugin index, then run legacy state migration twice. The current branch returns before archiving, so the legacy file remains and the warning repeats; I did not execute the flow in this read-only review.

## Solution Assessment

Yes, the right layer is core doctor/state migration rather than plugin runtime. The best fix still needs maintainer policy on when SQLite is authoritative enough to archive or remove legacy sources instead of preserving conflict warnings forever.
