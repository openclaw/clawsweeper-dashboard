## What This Changes

The PR changes `extensions/qqbot/src/engine/gateway/outbound-dispatch.ts` so lane-dispatch failures send Chinese timeout or generic error text when no response has been recorded.

## Best Possible Solution

Keep the focused QQ Bot catch fix, but make timeout notices terminal or cancel late delivery, add focused lane rejection/timeout regression coverage, then require redacted real QQ Bot proof before merge.

## Reproduction Assessment

Yes, from source: on current main, make `runtime.channel.inbound.run` reject or let `responseTimeoutMs` fire before any deliver, and the catch only clears `timeoutId` without sending user feedback. I did not execute a live QQ Bot repro in this read-only review.

## Solution Assessment

No. The catch block is the right surface for lane-rejection feedback, but timeout handling must be terminal or cancelable before merge so users do not receive both retry text and a late answer.
