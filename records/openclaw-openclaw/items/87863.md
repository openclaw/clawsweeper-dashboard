## Summary

Keep open. This is a plausible focused fix for QQ Bot lane failures, but it is not merge-ready because real QQ Bot proof is still absent and the timeout branch can emit retry text while the in-flight dispatch later delivers the original answer.

## What This Changes

The PR changes QQ Bot outbound dispatch so a rejected lane dispatch or response timeout sends a localized user-facing error message when no response has been recorded.

## Best Possible Solution

Repair the QQ Bot dispatcher so lane rejection sends one sanitized failure reply, timeout feedback is terminal or late replies are suppressed, and focused regression coverage plus redacted real QQ Bot proof exist before merge.

## Reproduction Assessment

Yes. Source inspection shows that a rejecting QQ Bot runtime channel run or responseTimeoutMs expiry before any deliver callback reaches the catch path; current main only clears timeoutId and sends no user-facing text.

## Solution Assessment

No. The catch path is the right QQ Bot owner-boundary surface for lane-rejection feedback, but the PR is not the best mergeable fix until timeout feedback is terminal or late replies are suppressed and real behavior proof is supplied.
