## What This Changes

The PR changes QQ Bot outbound dispatch to send a localized timeout or generic failure reply when the dispatch race rejects before any response is recorded.

## Best Possible Solution

Make QQ Bot dispatch send one sanitized failure reply for lane rejection, make timeout feedback terminal or suppress late replies, and add focused regression coverage plus redacted real QQ Bot proof before merge.

## Reproduction Assessment

Yes. Source inspection shows QQ Bot dispatch waits on Promise.race and current main only clears timeoutId when lane dispatch rejects or responseTimeoutMs fires before any delivery.

## Solution Assessment

No. The catch path is the right QQ Bot owner-boundary surface for lane-rejection feedback, but this is not the best mergeable fix until timeout feedback cannot race late replies and real behavior proof exists.
