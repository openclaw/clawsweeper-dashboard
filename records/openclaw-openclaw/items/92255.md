## Summary

The central request is already shipped: the generic gateway CLI can call `sessions.send`, and the gateway implements it through the canonical session pipeline with authenticated `operator.write` authorization. The narrower least-privilege token and authenticated-sender work remains tracked separately.

## What This Changes

Expose an authenticated `sessions.send` gateway RPC for cross-machine agent-to-agent messaging with delivery acknowledgment.

## Best Possible Solution

Keep the shipped gateway session-delivery implementation, correct the CLI help text separately if discoverability remains confusing, and resolve least-privilege credentials and sender provenance through the linked security-focused issues.

## Reproduction Assessment

No. Current main and shipped releases register `sessions.send`, authorize it with `operator.write`, and accept it through the generic gateway-call CLI; the reported unsupported-method conclusion appears to come from interpreting stale help text as an allowlist.

## Solution Assessment

Yes for the central request. Reusing the existing gateway session handler and authorization framework is narrower and more maintainable than introducing a parallel cross-machine messaging path; method-specific tokens remain separate security work.
