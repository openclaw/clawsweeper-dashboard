## Summary

Keep open: current main still has the source-level race, and the linked open fix PR is the right implementation path; this issue should close only after that PR or an equivalent fix merges.

## What This Changes

The issue reports that `chat.abort` can return `aborted:false` when called immediately after an attachment-heavy `chat.send` because the run is not registered yet.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84306, or an equivalent early-registration fix, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level: current main awaits attachment preprocessing before inserting the run into `chatAbortControllers`, while `chat.abort` returns `aborted:false` when that entry is missing. I did not run a live gateway repro in this read-only review.

## Solution Assessment

Yes: the maintainable fix is to register the chat-send abort controller after validation/dedupe checks but before awaited attachment/model preprocessing, with cleanup for aborted or failed pre-dispatch paths.
