## Summary

Keep open: current main still has a plausible source-level race where `chat.send` awaits attachment preparation before registering the run, while `chat.abort` returns `aborted:false` when the runId is not yet in the active-run map.

## What This Changes

The issue asks for `chat.abort` to reliably abort a `chat.send` run even when the abort request is sent immediately with the same runId.

## Best Possible Solution

Move chat-send active-run registration to the earliest post-validation/post-policy point before awaited attachment work, then clean it up on preparation or startup failure and cover abort-before-ack with a focused regression test.

## Reproduction Assessment

Yes at source level: current main awaits attachment preparation before inserting the run into `chatAbortControllers`, and `chat.abort` returns `aborted:false` when that entry is missing. I did not run a live gateway repro in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to register the chat-send abort controller before any awaited attachment/model preparation once validation and send policy have passed, mirroring the agent RPC early-registration pattern and adding cleanup paths.
