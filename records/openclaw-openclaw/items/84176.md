## Summary

Keep open: current main still has the source-level abort-registration race, and https://github.com/openclaw/openclaw/pull/84306 is an open closing PR for this issue. The issue should remain open until that PR, or an equivalent fix, lands.

## What This Changes

The issue reports that `chat.abort` can return `aborted:false` when called immediately after an attachment-heavy `chat.send` because the run is not yet registered in `chatAbortControllers`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84306, or an equivalent early-registration fix, then close this issue as implemented after merge.

## Reproduction Assessment

Yes, source-reproducible: current main awaits attachment preprocessing before inserting the run into `chatAbortControllers`, and `chat.abort` returns `aborted:false` when that entry is missing. I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Yes: the best fix is to register the chat-send abort controller after validation/dedupe checks but before awaited preprocessing, with cleanup for preparation failures and pre-dispatch aborts. The open closing PR is the current canonical implementation path.
