## What This Changes

The issue asks for `chat.abort` to reliably cancel a `chat.send` run even when abort arrives immediately during attachment preprocessing.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84306, or an equivalent early-registration fix with the same cleanup coverage, then close this issue after merge.

## Reproduction Assessment

Yes at source level: current main awaits attachment preprocessing before inserting the run into `chatAbortControllers`, while `chat.abort` returns `aborted:false` when that entry is missing. I did not run a live Gateway repro in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to register the abort controller after validation/policy/idempotency checks but before awaited attachment work, with cleanup for failed preprocessing and abort-before-dispatch paths.
