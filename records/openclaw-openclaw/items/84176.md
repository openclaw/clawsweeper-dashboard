## Summary

Keep open: current `main` still has a source-level race where attachment-bearing `chat.send` can await preparation before the run is registered, while `chat.abort` reports `aborted:false` for a missing run id.

## What This Changes

The issue requests that `chat.abort` reliably abort a just-started `chat.send` run even when abort is called immediately with the same run id.

## Best Possible Solution

Register a validated `chat.send` run before awaited attachment/model preparation, clean that early registration on preparation or startup failure, and add a focused abort-before-ack regression test.

## Reproduction Assessment

Yes at source level: current `chat.send` awaits attachment preparation before inserting the run into `chatAbortControllers`, and `chat.abort` returns `aborted:false` when that entry is missing. I did not run a live gateway repro in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to move chat-send active-run registration earlier after validation and policy checks, matching the adjacent agent RPC pattern, with explicit cleanup on failure paths.
