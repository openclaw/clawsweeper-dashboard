## Label Justifications

- `P2`: This is a bounded session provenance bug with limited blast radius and no evidence of outage or crash-loop impact.
- `impact:session-state`: The issue concerns preserving spawned child task lineage so session input is not misclassified as first-party user input.

## Summary

Keep open: current main and the latest release still omit explicit `inputProvenance` on `sessions_spawn` child `agent` requests, while existing inter-session paths and the gateway contract already support the requested metadata. The linked open PR at https://github.com/openclaw/openclaw/pull/47087 remains the active implementation path, so this issue should not be closed before that work lands or is replaced.

## What This Changes

Request to attach explicit inter-session `inputProvenance` with `sourceTool: "sessions_spawn"` to child tasks created through `sessions_spawn`.

## Best Possible Solution

Refresh and land, or replace, https://github.com/openclaw/openclaw/pull/47087 with a narrow fix covering both native and ACP `sessions_spawn` child dispatches plus focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main shows both native and ACP child `agent` dispatches omit `inputProvenance`, while the gateway accepts it and existing inter-session paths already pass it.

## Solution Assessment

Yes for the requested direction. The maintainable fix is to attach the existing inter-session provenance payload at the child dispatch boundary, preserving requester session/channel and setting `sourceTool: "sessions_spawn"`.
