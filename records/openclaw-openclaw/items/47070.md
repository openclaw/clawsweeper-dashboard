## Label Justifications

- `P2`: This is a bounded session provenance bug in multi-agent spawn workflows with limited blast radius and no evidence of outage or data loss.
- `impact:session-state`: The missing metadata can make a spawned child task lose inter-session lineage and treat agent-originated input like clean first-party session input.

## Summary

Keep open: current main and latest release still omit `inputProvenance` on `sessions_spawn` child-agent dispatch, and this issue is already paired with an open draft PR that explicitly closes it. The IdentyClaw comment is useful complementary plugin-level lineage context, but it does not replace OpenClaw's core session provenance metadata.

## What This Changes

The issue requests explicit `inputProvenance` on child tasks spawned via `sessions_spawn`, including `sourceTool: "sessions_spawn"` and requester session/channel lineage.

## Best Possible Solution

Land a focused fix that attaches inter-session `inputProvenance` at every in-scope `sessions_spawn` child-agent dispatch, preserves requester session/channel metadata, and proves the native and any chosen ACP behavior with targeted tests before closing this issue.

## Reproduction Assessment

Yes, from source inspection: current `spawnSubagentDirect()` and `spawnAcpDirect()` call the `agent` gateway method without `inputProvenance`, while existing agent ingress and `sessions_send` paths already support and consume that metadata. I did not run a live spawn scenario in this read-only review.

## Solution Assessment

Yes for keeping this issue open: the narrow maintainable fix is to reuse the existing `inputProvenance` contract at the spawn dispatch boundary and review the linked PR or an equivalent replacement. The only scope question is whether ACP `sessions_spawn` should be covered alongside native spawn before closure.
