## Label Justifications

- `P2`: This is a bounded session provenance bug with limited blast radius and no evidence of outage or crash-loop impact.
- `impact:session-state`: The issue concerns preserving spawned child task lineage so session input is not misclassified as first-party user input.

## Summary

Keep open: current main and v2026.5.22 still omit `inputProvenance` from native and ACP `sessions_spawn` child `agent` dispatches. The existing inter-session provenance contract already supports the requested metadata, and the linked open PR at https://github.com/openclaw/openclaw/pull/47087 remains the active but incomplete implementation path.

## What This Changes

The issue requests explicit inter-session `inputProvenance` on child tasks started by `sessions_spawn`, including `sourceTool: "sessions_spawn"` and requester lineage.

## Best Possible Solution

Refresh and land, or replace, https://github.com/openclaw/openclaw/pull/47087 with a narrow fix covering both native and ACP `sessions_spawn` child dispatches plus focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main shows both native and ACP child `agent` dispatches omit `inputProvenance`, while the gateway accepts it and existing inter-session paths already pass it. I did not run a live spawned task because this review is read-only.

## Solution Assessment

Yes for the requested direction. The maintainable fix is to attach the existing inter-session provenance payload at the child dispatch boundary, preserving requester session/channel and setting `sourceTool: "sessions_spawn"` across both native and ACP spawn paths.
