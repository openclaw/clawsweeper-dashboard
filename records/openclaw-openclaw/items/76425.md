## Summary

Keep open: current main has strong mitigations for supported `openclaw update` and `update.run` paths, but this remains the canonical stale live-Gateway/mixed-`dist` tracker with recent 2026.5.7 and Telegram evidence not proven fixed by current code.

## What This Changes

Ensure upgrades do not leave a running Gateway importing stale hashed `dist` chunks after package replacement, including Feishu/DingTalk and later Slack/Telegram failures.

## Best Possible Solution

Define and enforce one upgrade recovery contract: either make raw package-manager upgrades refresh the managed Gateway as reliably as `openclaw update`, or route them through clear diagnostics to a force restart/reinstall path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not against current main in this read-only pass; the issue has concrete 2026.5.2 logs plus later 2026.5.7 and Telegram evidence, and source/docs confirm the stale live-Gateway import path.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Current main is the right direction for supported updater paths, but it does not fully settle raw package-manager upgrade support or safe-vs-force restart recovery.
