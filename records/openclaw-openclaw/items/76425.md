## Summary

Keep open: the original stale hashed-module failure is still supported by a newer 2026.5.7 field update, and current main only partially mitigates the package-swap/live-Gateway problem through supported updater paths and selected legacy aliases.

## What This Changes

The issue asks OpenClaw upgrades to avoid leaving a running Gateway with lazy imports that point at removed hashed `dist` files.

## Best Possible Solution

Define a single supported recovery contract for live package swaps: supported updater paths should restart outside the stale Gateway before old code can lazy-load removed files, and manual package-manager upgrades should get clear diagnostics that point to a force service restart or `openclaw update`.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. The issue has concrete 2026.5.2 logs plus a 2026.5.7 field update, and current source shows the package-swap, lazy-chunk alias, and restart-coalescing paths that plausibly produce the reported failure.

## Solution Assessment

Unclear. Current main has useful mitigations for supported update paths and selected stale chunks, but the final fix needs a maintainer choice between fail-closed/manual-upgrade guidance, force-restart diagnostics, or an updater handoff that avoids running package replacement inside a stale Gateway process.
