## Summary

Keep open: current main has strong mitigations for supported `openclaw update` and some stale-chunk aliases, but this issue now has fresh 2026.5.7 field evidence that a replaced package tree can still leave an old Gateway process serving stale hashed imports until a force restart. The related open issues cover narrower agent-self-update, `update.run`, and respawn-path slices, but none fully owns the manual package-manager/Homebrew stale-Gateway recovery path reported here.

## What This Changes

The issue reports stale hashed runtime imports after upgrading OpenClaw globally to 2026.5.2 while an old Gateway process may still be running.

## Best Possible Solution

Define the supported boundary for replaced-package updates, then either harden stale Gateway detection/recovery for manual package-manager and Homebrew upgrades or fail/document that path clearly in favor of `openclaw update`.

## Reproduction Assessment

Unclear for a high-confidence live current-main reproduction. The issue and 2026.5.7 field update provide real stale-import evidence, and current source/docs show the replaced-package lazy-import hazard, but this read-only review did not mutate a live global install to reproduce it.

## Solution Assessment

Unclear. Current main is a good mitigation for supported update paths, but closing this as implemented or fully duplicate would skip the still-distinct manual package-manager/Homebrew stale-Gateway recovery question.
