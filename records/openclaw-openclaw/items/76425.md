## Summary

Current main addresses the stale hashed chunk failure for the supported package-update path by stopping or blocking managed Gateway package updates before the package tree is replaced, with tests and docs covering the exact lazy-load removed-chunks failure mode. The fix is main-only; it is not contained in v2026.5.2.

## What This Changes

The issue asks OpenClaw to avoid ERR_MODULE_NOT_FOUND crashes from stale hashed dist chunks after upgrading a global npm install while a Gateway is running.

## Best Possible Solution

Keep the current main behavior: package updates should go through `openclaw update` or stop the Gateway first, so the package tree is not replaced while the old process can still import removed chunks.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes: the report gives concrete npm/global-upgrade steps, and current tests model the unsafe active-Gateway package update path; I did not run a live macOS/fnm upgrade.

## Solution Assessment

Is this the best way to solve the issue? Yes: stopping or blocking the managed Gateway before replacing package files is narrower and safer than trying to regenerate hashed runtime assets inside a process that can still lazy-load removed chunks.
