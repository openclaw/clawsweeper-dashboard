## Summary

Keep open: current main and v2026.6.8 still lack a configurable umask or file-mode policy for user-facing agent outputs, and implementing one would add security-sensitive config surface that needs maintainer product/security review.

## What This Changes

Request to add a configurable umask or file-mode policy for agent-created user-facing files across exec, write/edit, and other workspace-writing tools without changing sensitive internal file permissions.

## Best Possible Solution

Adopt a maintainer-approved policy that clearly separates user-facing output from private OpenClaw state and applies consistently to file tools and exec/sandbox child processes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main exposes tools.fs.workspaceOnly and tools.exec options, but no umask/fileMode/dirMode policy, and write/edit/exec paths do not consume such a policy.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers define the security boundary; a configurable policy may be right, but it must avoid loosening private state/config writes while covering real user-facing output paths.

## Risks / Open Questions

- A permission knob for user-facing files can become a security footgun unless maintainers define the boundary between operator-requested output and private OpenClaw config, state, credentials, transcripts, and caches.
- The requested scope spans write/edit/apply_patch, exec child processes, Docker/sandbox paths, and possibly code-mode or plugin tools, so a one-path fix could leave inconsistent behavior.
- Adding tools-level file mode config changes public config schema and upgrade expectations; maintainers need to decide defaults, validation, docs, and whether doctor should warn or migrate anything.
