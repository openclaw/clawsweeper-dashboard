## Summary

Keep open. Current main still has no supported tools-level umask, file mode, dir mode, or filePermissions policy for agent-created user-facing files; the strict config schema only covers exec routing/security and fs workspace containment, while current write paths either inherit process defaults or use fixed restrictive modes. The request remains valid core work because it spans built-in exec, write/edit, apply_patch, and sandbox filesystem surfaces rather than an optional plugin boundary.

## What This Changes

Request a configurable umask or explicit file/directory mode policy for user-facing files created by agent tools without loosening sensitive OpenClaw-managed files.

## Best Possible Solution

Keep this issue open and design a scoped, documented file-creation policy for user-facing workspace output, preferably per-agent aware and validated as octal input. It should apply consistently to host write/edit, apply_patch, sandbox bridge writes, and exec-created output where feasible, while preserving restrictive modes for OpenClaw internal and secret-bearing state.

## Reproduction Assessment

Not applicable as a feature request, but the missing behavior is high-confidence from static verification: the current schema rejects the proposed fields and the relevant write paths have no configurable mode/umask input.

## Solution Assessment

Yes directionally, but not as a broad process.umask toggle. The safer solution is a narrow user-output file creation policy with explicit validation and call-site integration, plus secops review for the sensitive-file boundary.
