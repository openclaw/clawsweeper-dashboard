## Summary

Keep open. Current main does not expose a configurable umask or file-mode contract for agent-created user-facing files, and the adjacent open permissions issue covers sensitive OpenClaw-managed state rather than this workspace/output-file surface.

## What This Changes

Request a configurable umask or explicit file/directory modes for agent-created user-facing files from exec, write/edit, and similar output paths.

## Best Possible Solution

Define a category-scoped permission policy for user-facing workspace/output files, wire it through config schema/docs and write/edit/apply_patch/sandbox/exec paths, and keep internal state writers and security audit/fix behavior private by default.

## Reproduction Assessment

Not applicable: this is a feature/config-policy request, not a bug report. Source inspection gives a high-confidence current-main gap because the tools schema and write/exec paths do not expose a permission or umask setting.

## Solution Assessment

Unclear. A configurable output-file policy is plausible, but a simple global tools.umask is not clearly the safest maintainable shape because user-facing files must stay separated from sensitive OpenClaw state and exec spans multiple runtimes.
