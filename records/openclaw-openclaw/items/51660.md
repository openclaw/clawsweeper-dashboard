## Summary

Keep open. Current main still has no supported tools-level user-output umask or file-mode policy; the open Write/Edit PR only addresses a hard-coded subset, and the related sensitive-state permissions issue covers a different trust boundary.

## What This Changes

The issue requests configurable umask/file and directory modes for user-facing files created by agent tools such as exec, Write/Edit, and tool output.

## Best Possible Solution

Define a maintainer-approved, category-scoped user-output permission policy with validated octal inputs, wire it through config/schema/docs and approved write surfaces, and preserve restrictive internal-state permissions.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main has no accepted config fields or cross-surface plumbing for the requested user-output permission policy.

## Solution Assessment

Yes directionally, but not as a broad process-wide umask or a hard-coded 0o644 default. The safer maintainable solution is a scoped, reviewed user-output policy that avoids sensitive internal state.
