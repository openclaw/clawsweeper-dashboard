## Summary

Keep open. Current main still lacks a supported tools-level umask, file-mode, or directory-mode policy for agent-created user-facing output; the prior Write/Edit PR was closed unmerged as superseded by this issue, and adjacent open issues cover narrower or more sensitive permission surfaces rather than replacing this request.

## What This Changes

The issue requests an opt-in tools-level umask or explicit file/directory mode policy for user-facing files created by agent write, edit, exec, apply_patch, and sandbox write paths.

## Best Possible Solution

Define a maintainer-approved, opt-in user-output permission policy with validated octal modes, apply it consistently to host and sandbox write surfaces plus exec-created output, and keep internal OpenClaw state on private defaults.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that current main has no accepted config fields or cross-surface runtime plumbing for the requested user-output permission policy.

## Solution Assessment

Yes directionally, but not as a broad process-wide umask. The maintainable path is a scoped, opt-in user-output policy that cannot affect OpenClaw-managed secrets or internal state and is tested across host, sandbox, apply_patch, and exec paths.
