## Summary

Keep open. Current main and the latest release still restrict memory-flush writes to one exact planned file, while this issue asks for validated writes beneath the `memory/` tree; the related reports show this is part of a broader write-target contract, not stale support noise.

## What This Changes

Allow memory-flush writes to target validated subpaths under `memory/`, such as `memory/daily/YYYY-MM-DD.md`, instead of only the generated root daily file.

## Best Possible Solution

Define a narrow memory-flush write-target contract that supports safe additional targets, preserves append-only behavior, rejects traversal and sibling-prefix escapes, and updates the plan/config/docs/tests together.

## Reproduction Assessment

Yes. A high-confidence unit path is to instantiate the wrapper with `memory/YYYY-MM-DD.md` and execute the write tool against `memory/daily/YYYY-MM-DD.md`; current source reaches the exact-path comparison and throws the reported restriction error.

## Solution Assessment

No. The issue's raw `startsWith(memoryBase + "/")` sketch is not enough because the append path remains the planned file and prefix checks must be segment-safe; a maintainable fix should validate and append to the explicitly permitted target.
