## Summary

Keep this PR open: the forwarded-env hardening is the right direction, but the latest branch still has a command-approval blocker because a node command marker can satisfy durable approval when no concrete allow-always coverage is currently provable.

## What This Changes

The PR changes host=node exec approval prechecks to analyze with the forwarded node environment, clears PATH fallback, adds node command allow-always marker handling, and expands regression coverage.

## Best Possible Solution

Keep the forwarded-env hardening, but require complete non-empty concrete allow-always coverage before a node command marker can suppress approval, then rebase and rerun focused node exec approval checks.

## Reproduction Assessment

Yes, from source inspection. Current main uses the gateway/request env for node approval analysis while the run payload forwards node-targeted env, and the PR-head marker helper still returns durable approval for a matching marker when coverage has no expected patterns.

## Solution Assessment

No, not yet. The forwarded-env precheck is the right fix shape, but the PR must fail closed on marker plus empty or incomplete coverage before it is safe to merge.
