## Summary

Keep open: the forwarded-env hardening is the right direction, but the latest PR head still has a blocking command-approval bug where a matching node command marker can satisfy durable approval even when no concrete allow-always coverage was derived.

## What This Changes

This PR changes host=node approval prechecks to analyze against the node-forwarded env, adds node allow-always command markers, updates node approval tests, and adds a changelog entry.

## Best Possible Solution

Keep the forwarded-env hardening, but require node marker reuse to prove complete non-empty allow-always coverage before durable approval can suppress a prompt.

## Reproduction Assessment

Yes, from source inspection. A node allow-always marker plus unresolved coverage reaches the expectedPatterns.length === 0 branch and returns true, which feeds durableApprovalSatisfied and suppresses approval.

## Solution Assessment

No, not as currently patched. The core env fix is narrow, but marker reuse must require complete non-empty coverage and a regression test before this is the maintainable security fix.
