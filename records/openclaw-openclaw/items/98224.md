## Summary

Keep open: the PR targets a real silent-reply leak, but the current patch strips punctuation before exact matching and can break existing custom tokens that intentionally include punctuation at the edges.

## What This Changes

The PR strips leading/trailing Unicode punctuation before exact silent-reply token detection and adds tests for punctuation-wrapped `NO_REPLY` cases.

## Best Possible Solution

Preserve exact custom-token matching before edge-punctuation normalization, add regression coverage, and align the final punctuation policy with https://github.com/openclaw/openclaw/pull/93510.

## Reproduction Assessment

Yes, at source level. Current main's exact matcher rejects punctuation-wrapped token-only payloads, and the PR regression is visible from the diff because it strips punctuation before exact custom-token matching.

## Solution Assessment

No. The shared matcher is the right layer, but this implementation should preserve exact custom-token matches before trying normalized edge-punctuation matching and should coordinate with https://github.com/openclaw/openclaw/pull/93510.
