## Summary

Keep open: current main and the latest release still use the strict exact matcher, so the reported punctuation-wrapped silent-token leak is not obsolete. The PR is a plausible shared-helper fix, but it still misses whitespace-wrapped punctuation and needs maintainer acceptance of the exported SDK/message-delivery contract and coordination with the overlapping open matcher PR.

## What This Changes

The PR changes silent-reply token detection to retry after stripping leading and trailing Unicode punctuation, with tests for punctuation-wrapped `NO_REPLY` and punctuation-edged custom tokens.

## Best Possible Solution

Land one shared classifier policy that covers optional whitespace-wrapped edge punctuation, preserves exact custom-token matches and substantive-reply guards, and either folds or clearly supersedes the overlapping matcher PR.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.11` use the strict exact matcher, so punctuation-wrapped token-only replies miss the shared suppression classifier. I did not run a live Slack/Telegram channel path in this read-only review.

## Solution Assessment

No, not quite as submitted: the shared matcher is the right boundary, but the fallback only strips punctuation at absolute string edges and misses whitespace-wrapped punctuation that the exact matcher otherwise supports. The safer fix is to preserve exact custom-token matching first, then normalize optional whitespace plus edge punctuation and coordinate the final policy with the overlapping open PR.
