## Summary

Keep open: the bugfix targets a real silent-token leak, but the latest head still normalizes before preserving exact custom-token matches, which can break exported helper behavior for punctuation-edged custom tokens.

## What This Changes

The PR changes `isSilentReplyText` to strip leading/trailing Unicode punctuation before matching the silent token and adds unit cases for punctuated `NO_REPLY` variants.

## Best Possible Solution

Preserve the current exact-token match first, then apply edge-punctuation normalization for token-only leaks while coordinating the final punctuation contract with the overlapping matcher PR.

## Reproduction Assessment

Yes. Current main's exact matcher rejects punctuation-wrapped token-only payloads, and the PR regression is source-visible because the latest head strips punctuation before exact custom-token matching.

## Solution Assessment

No. The shared matcher is the right layer, but this implementation should preserve exact custom-token matches before trying normalized edge-punctuation matching.
