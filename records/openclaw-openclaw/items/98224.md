## Summary

Keep open: this is a useful fix candidate for the linked silent-token leak, but the patch should not merge as-is because it strips edge punctuation before preserving the current exact custom-token match, changing exported helper behavior.

## What This Changes

The PR adds edge-punctuation normalization to `isSilentReplyText()` and adds unit cases for `.NO_REPLY`, `*NO_REPLY*`, quoted tokens, and mixed-content preservation.

## Best Possible Solution

Preserve the current exact-token match first, then apply edge-punctuation normalization for default token-only leaks while keeping substantive-reply guards and coordinating with https://github.com/openclaw/openclaw/pull/93510.

## Reproduction Assessment

Yes. Current main and the latest release use the exact matcher that rejects `.NO_REPLY` and `*NO_REPLY*`, and the patch-level custom-token regression is source-reproducible from the new normalize-before-match call.

## Solution Assessment

No. The central matcher is the right layer, but this implementation should preserve exact custom-token matches before trying normalized edge-punctuation matching.
