## Summary

Keep this PR open: the performance target is valid, but the current diff makes `fs.existsSync` results process-global across plugin discovery, which conflicts with the plugin freshness boundary and can stale marker-file detection before merge.

## What This Changes

The PR adds a shared memoized `fs.existsSync` helper, routes plugin bundle discovery, bundled-directory, and SDK alias probes through it, and adds cache invalidation unit tests.

## Best Possible Solution

Keep the startup optimization, but make the existence cache owned by a single discovery/startup pass or explicit loader cache, cover stale marker changes with regression tests, and add the changelog entry.

## Reproduction Assessment

Yes, from source inspection: the PR’s module-level `Map` stores misses indefinitely, and current install/discovery paths can re-run marker detection in the same process without an owning invalidation boundary.

## Solution Assessment

No. Memoizing repeated probes is reasonable, but the safer maintainable shape is a scan-owned or loader-owned cache instead of a shared process-global helper.
