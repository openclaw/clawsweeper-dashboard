## Summary

Keep this PR open. Current main still has the repeated synchronous plugin-scan existence probes the PR targets, but the patch needs changes because the new cache is process-wide without production invalidation and the user-facing perf change lacks the required changelog entry.

## What This Changes

The PR adds a shared cached `fs.existsSync` helper, routes plugin bundle/bundled-dir/SDK-alias probes through it, and adds unit tests for memoization and invalidation.

## Best Possible Solution

Keep the existence-probe optimization, but scope memoization to a single plugin scan or alias-resolution flow, or invalidate it from all plugin filesystem mutation paths, and add the required changelog entry before merge.

## Reproduction Assessment

Yes for the structural issue: current main still has repeated synchronous `fs.existsSync` probes in `bundle-manifest.ts`, `bundled-dir.ts`, and `sdk-alias.ts`. I did not rerun the Windows CPU profile, so the exact 25-second self-time remains reporter evidence.

## Solution Assessment

No as written. Caching repeated probes is the right direction, but a process-level cache with no automatic production invalidation is not the narrowest maintainable solution for plugin discovery freshness.
