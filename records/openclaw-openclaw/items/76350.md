## Summary

Keep this PR open. The performance target is valid and the diff hits the cited hot paths, but the process-wide cache can make plugin discovery observe stale filesystem state across install/repair/reload flows, and the user-facing perf change still needs a changelog entry.

## What This Changes

The PR adds a shared `existsSyncCached` helper, rewires plugin manifest/bundled-dir/SDK-alias existence probes through it, and adds unit coverage for cache hits and invalidation.

## Best Possible Solution

Keep the existence-probe optimization, but make the cache owned by a single plugin scan flow or wire explicit invalidation into every plugin filesystem mutation path, then add the required changelog entry.

## Reproduction Assessment

Yes for the structural issue: current main still has repeated synchronous `fs.existsSync` probes in the plugin scan and SDK alias paths. I did not rerun the Windows CPU profile, so the exact 25-second self-time remains reporter evidence rather than live proof.

## Solution Assessment

No as written. Caching repeated probes is the right direction, but a process-level cache without production invalidation is not the narrowest maintainable solution for plugin discovery freshness.
