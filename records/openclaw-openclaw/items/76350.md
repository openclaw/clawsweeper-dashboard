## Summary

Keep this PR open but do not merge it as-is. The performance goal is still relevant on current main, but the branch introduces a process-global existence cache that can keep stale negative plugin-discovery answers alive across later in-process install/discovery work, and it is missing the required changelog entry for a user-facing perf change.

## What This Changes

The PR adds a shared memoized `fs.existsSync` helper, routes plugin bundle discovery and SDK alias existence probes through it, and adds unit tests for cache invalidation.

## Best Possible Solution

Keep the startup optimization, but make existence memoization scan-scoped or owned by the plugin loader/discovery flow, add stale false-to-true regression coverage, and add the changelog entry.

## Reproduction Assessment

Yes. From source inspection, the PR stores false `existsSyncCached` results indefinitely and routes plugin marker detection through that cache while install/discovery callers do not invalidate it before later probes in the same process.

## Solution Assessment

No. The optimization direction is useful, but a shared process-lifetime cache is broader than the plugin discovery boundary allows; a scan-scoped or explicitly owned cache is the narrower maintainable fix.
