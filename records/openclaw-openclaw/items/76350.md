## Summary

Keep open: current main still has direct `fs.existsSync` probes in the plugin scan paths and no shared cache, while the latest PR head appears to address the linked cold-start profile with scan-scoped caching and no blocking review finding.

## What This Changes

The PR adds a shared `createExistsSyncCache` helper, threads per-scan existence caching through plugin bundle discovery, bundled-plugin directory resolution, and SDK alias probing, and adds cache tests plus a changelog entry.

## Best Possible Solution

Land the scan-scoped cache approach after final maintainer review, keeping cache lifetime owned by plugin scan or loader operations and leaving the pnpm-install diagnostic uncached.

## Reproduction Assessment

Yes, source-reproducible: current main still has repeated direct `fs.existsSync` probes in the plugin scan paths, and the linked report provides a Windows cold-start CPU profile showing `existsSync` as the largest self-time. I did not run a live profile in this read-only pass.

## Solution Assessment

Yes: the PR’s scan-scoped cache is narrower and safer than process-global memoization and aligns with the plugin boundary’s current-flow cache rule. The remaining landing detail is current-main review/rebase policy, not a code repair.
