## Summary

Current main and the latest release still lack a Zen model catalog for the bundled `opencode` provider, and the active fix is already tracked by https://github.com/openclaw/openclaw/pull/92495. Keep this issue open until that PR lands or provider owners explicitly choose a different seed/discovery policy.

## What This Changes

The issue reports that the bundled `opencode` Zen provider lacks a provider-owned model catalog and dynamic resolver, forcing users to hand-register Zen models and routing metadata.

## Best Possible Solution

Review and land or explicitly reject the linked Zen catalog repair, then close this issue after the provider-owned catalog behavior is on main.

## Reproduction Assessment

Yes, source-level: current `extensions/opencode/index.ts` lacks catalog, augmentation, and dynamic model resolver hooks while the sibling `opencode-go` provider has them. I did not run a live Zen request in this read-only cleanup review.

## Solution Assessment

Yes on shape, pending policy: restoring a provider-owned Zen catalog and resolver in the `opencode` plugin is the maintainable owner-boundary fix. The unresolved part is whether maintainers accept the proposed 48-row static seed and shared credential fallback in the open repair PR.
