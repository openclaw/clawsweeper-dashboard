## Summary

Keep open: current `main` still lacks provider-owned OpenCode Zen catalog metadata and runtime catalog hooks, while the open linked repair PR is the candidate fix and uses closing syntax for this issue.

## What This Changes

The issue requests restoring a provider-owned catalog and dynamic discovery for the bundled `opencode` Zen provider so Zen models do not require manual `models.providers.opencode.models[]` registration.

## Best Possible Solution

Review and land or explicitly replace the linked provider-boundary fix that restores the `opencode` Zen catalog, resolver, discovery, manifest metadata, and regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: choose an `opencode/<model>` ref beyond the default and current `main` has neither Zen catalog rows nor a dynamic resolver to satisfy it. I did not run the live onboarding path because this review is read-only, but the missing hooks and manifest metadata directly match the reported `model_not_found` behavior.

## Solution Assessment

Yes on owner-boundary shape, but the seed scope and credential fallback policy still need maintainer acceptance. Restoring Zen catalog/discovery inside `extensions/opencode` is more maintainable than asking users to hand-register every Zen model or adding generic core fallbacks.
