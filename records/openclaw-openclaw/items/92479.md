## Summary

Keep this open: current main and the latest release still lack the bundled `opencode` Zen catalog/resolver, and the open linked repair PR is the right implementation path until it lands or maintainers choose a different seed/discovery policy.

## What This Changes

The issue asks to restore a provider-owned OpenCode Zen model catalog and live discovery so `opencode/<model>` refs work without hand-registering every model.

## Best Possible Solution

Restore the Zen catalog inside the bundled `opencode` provider via the linked PR or an owner-approved variant, then close this issue after that behavior is on main.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.6` document `opencode/...` Zen refs but provide no `opencode` catalog runner, manifest catalog, dynamic resolver, or catalog augmentation hook; a live `/zen/v1/models` request also confirms the provider has rows to discover.

## Solution Assessment

Yes on owner boundary, pending policy. Restoring a provider-owned Zen catalog/resolver in `extensions/opencode` matches the sibling `opencode-go` seam; maintainers still need to accept or adjust the static seed and credential fallback policy before landing the linked PR.
