## Summary

Current main still lacks a Zen model catalog, while the sibling OpenCode Go plugin establishes the intended plugin-owned discovery seam. The earlier Zen discovery PR was closed during a repository-wide feature freeze rather than rejected technically, so this remains useful work, but maintainers should first choose the static metadata seed and live-discovery policy.

## What This Changes

Add a bundled and dynamically discovered OpenCode Zen model catalog so users do not need to hand-register every Zen model and its routing metadata.

## Best Possible Solution

Add plugin-owned Zen catalog hooks with a small reviewed static metadata seed and authenticated live discovery for additional IDs, using conservative defaults and explicit fallback behavior consistent with the maintained OpenCode Go catalog.

## Reproduction Assessment

Yes at source level: current main has no Zen catalog or dynamic-resolution hooks, while the live Zen endpoint exposes 48 models. This read-only review did not perform a credentialed inference request.

## Solution Assessment

Yes for the plugin ownership and catalog-hook pattern, but not for statically seeding the entire live catalog. A curated metadata seed plus conservative live discovery is the narrower maintainable direction pending maintainer approval.

## Risks / Open Questions

- The live endpoint does not provide rich model metadata, so statically seeding every returned ID would create ongoing pricing, context-window, capability, and retirement maintenance.
- Default authenticated discovery introduces network-availability and remote-catalog-change behavior; the fallback and caching lifecycle should be explicitly defined before implementation.
- Unknown discovered models need conservative metadata defaults so discovery does not advertise unsupported inputs, token limits, reasoning behavior, or transport capabilities.
