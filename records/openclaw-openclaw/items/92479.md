## Label Justifications

- `P1`: A previously shipped provider workflow regressed, leaving normal Zen model selection broken unless users manually reconstruct catalog and routing metadata.
- `impact:auth-provider`: The missing catalog affects provider model resolution and can cause incomplete manual entries to use the wrong transport endpoint for Zen credentials.

## Summary

Current main still omits OpenCode Zen catalog and dynamic-resolution hooks after a previously shipped Zen catalog was removed, so the provider now requires manual model and transport registration; the regression is valid, but the maintained offline seed and default live-discovery policy need provider-owner judgment before implementation.

## What This Changes

Restore catalog-backed OpenCode Zen model resolution so users do not need to hand-register every Zen model and its routing metadata.

## Best Possible Solution

Restore a provider-owned Zen catalog through the existing plugin SDK, keep an approved curated offline seed with correct API/base-URL metadata, enrich available IDs through bounded authenticated discovery, and cover both offline fallback and live discovery in resolver and onboarding tests.

## Reproduction Assessment

Yes, from source with high confidence: current main has no Zen catalog hooks, the prior shipped implementation supplied catalog metadata, and the live endpoint exposes the reported model IDs. This read-only review did not execute a full authenticated agent request.

## Solution Assessment

The proposed plugin-local restoration is the correct ownership boundary and reuses established SDK seams, but mirroring the sibling wholesale is not yet fully specified because maintainers must choose Zen’s static seed scope and default discovery policy.
