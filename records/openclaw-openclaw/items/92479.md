## Label Justifications

- `P1`: A released bundled provider lost catalog-backed model resolution, breaking its documented model workflow unless users manually reconstruct provider and transport metadata.
- `impact:auth-provider`: The missing catalog affects provider model availability, endpoint routing, model choice, and correct use of OpenCode Zen credentials.

## Summary

Current main and v2026.6.6 still ship the OpenCode Zen plugin without any model catalog or dynamic-resolution hooks, after the previously shipped Zen catalog was deleted during the provider-plugin migration. The issue remains necessary, but the offline seed scope and default live-discovery policy need provider-owner judgment before implementation.

## What This Changes

Restore bundled OpenCode Zen model catalog and discovery so valid Zen models resolve without manual provider and transport registration.

## Best Possible Solution

Restore a plugin-owned Zen catalog with correct endpoint and per-family API metadata, retain an owner-approved curated offline seed, explicitly define the live-discovery policy, and add focused catalog-listing and dynamic-resolution regression coverage.

## Reproduction Assessment

Yes. Source gives a high-confidence path: select a documented Zen model without a manual provider row; current main and v2026.6.6 provide no Zen catalog or dynamic resolver capable of constructing that model entry, so resolution falls through to the unknown-model behavior.

## Solution Assessment

Partly. Restoring the catalog in the OpenCode plugin is the narrowest maintainable fix, but mirroring every Go default is not automatically best because seed maintenance and default network discovery require an explicit provider-owner decision.
