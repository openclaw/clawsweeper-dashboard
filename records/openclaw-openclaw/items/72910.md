## Summary

Keep open: current main still hardcodes the 30-second completion-cache refresh timeout in both update and doctor paths, and the PR is a focused fix with no definite code-level correctness finding. Merge should still wait for after-fix real behavior proof and maintainer choice against the overlapping configurable-timeout PR.

## What This Changes

The PR raises update and doctor completion-cache spawn timeouts from 30 seconds to 120 seconds, adds non-JSON progress and duration logging, updates tests, and adds a changelog entry.

## Best Possible Solution

Land one coherent completion-cache timeout policy: either this bounded 120-second default plus progress logging or the newer configurable-timeout approach, with update and doctor coverage and after-fix proof from a real install.

## Reproduction Assessment

Yes. The PR body gives a concrete slow-cache reproduction where direct completion generation succeeds after about 36 seconds but the shipped 30-second spawnSync timeout returns ETIMEDOUT, and source inspection confirms current main still uses that timeout in the update and doctor paths.

## Solution Assessment

Mostly yes. Extending the bounded timeout and adding progress output is a narrow maintainable fix for slow-but-successful cache generation, though maintainers should compare it with the newer configurable-timeout PR before choosing the final policy.
