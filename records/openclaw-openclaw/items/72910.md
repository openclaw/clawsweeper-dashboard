## Summary

Keep open. Current main still has the reported 30-second completion-cache refresh timeout in both the package-update follow-up and doctor repair path, and the PR is a focused implementation of the remaining behavior left after the message-only #72850 fix. I did not find a discrete correctness or security finding in the patch; the remaining merge work is maintainer review plus resolving or rerunning the failing exact-head CI.

## What This Changes

The PR raises the update and doctor completion-cache write timeout from 30 seconds to 120 seconds, adds non-JSON update progress and success-duration logging, adds focused regression tests, and adds a changelog entry.

## Best Possible Solution

Land this PR or an equivalent maintainer patch after updating against current main and resolving CI: keep the bounded 120-second timeout in both update and doctor cache generation, preserve JSON-mode silence and the manual-refresh hint, keep visible progress for the slower non-JSON update step, and retain focused tests plus the changelog entry.

## Reproduction Assessment

Yes. The PR body gives a concrete reproduction where direct cache generation succeeds in about 36 seconds but the same command under the shipped 30-second spawnSync timeout returns ETIMEDOUT, and current main still contains that 30-second timeout in both affected paths.

## Solution Assessment

Yes. The proposed change is the narrowest maintainable fix for slow-but-successful completion-cache rebuilds; making the refresh fully asynchronous could reduce waiting further but would be a larger behavior change than this follow-up needs.
