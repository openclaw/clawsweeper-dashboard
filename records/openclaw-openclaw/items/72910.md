## Summary

Keep open for maintainer handling: current main still uses the 30-second completion-cache timeout in both update and doctor repair paths, and this PR is a focused follow-up to the already-merged message-only fix. I did not find a discrete code-review bug in the diff, but the exact PR head currently has failing/unstable CI checks, so merge should wait for a fresh green run or an explicit unrelated-failure triage.

## What This Changes

This PR raises the completion-cache refresh timeout from 30 seconds to 120 seconds in update and doctor repair, adds non-JSON update progress/duration logging, adds focused regression tests, and records the user-facing fix in the changelog.

## Best Possible Solution

Land this PR or an equivalent maintainer patch once CI is current: keep the bounded 120-second timeout in both update and doctor completion-cache generation, preserve JSON-mode silence and the manual-refresh hint, keep progress output if maintainers accept the UX, and retain regression coverage for both paths.

## Reproduction Assessment

Yes. The PR body gives a concrete slow-install reproduction where direct cache refresh completes in about 36 seconds but the same command under the shipped 30-second `spawnSync` timeout returns ETIMEDOUT, and current main still contains that 30-second cap in both affected paths.

## Solution Assessment

Yes. This is a narrow maintainable fix for slow-but-successful cache rebuilds; a background/asynchronous refresh could reduce waiting further, but that would be a larger behavior change than needed for this follow-up.
