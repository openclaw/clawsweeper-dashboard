The PR raises completion-cache refresh timeouts from 30 seconds to 120 seconds in update and doctor paths, adds non-JSON update progress logging, adds regression coverage, and updates the changelog.

## Best Possible Solution

Yes. The PR body gives a concrete slow-cache reproduction where direct completion generation succeeds in about 36 seconds but the shipped 30-second spawnSync timeout returns ETIMEDOUT, and current main still has the 30_000 ms timeout in both affected paths.

## Solution Assessment

Yes, with one branch-maintenance fix. Extending the existing bounded post-update/doctor timeout and adding visible progress is the narrowest fix for slow-but-successful cache generation; making the refresh async would be a larger behavior change.
