## Summary

Keep open. Current main still uses the 30-second completion-cache timeout in both affected paths, while the PR is a focused implementation for the slow-but-successful cache refresh case; no blocking code or security finding was identified, but exact-head CI is red and needs maintainer rerun/inspection before merge.

## What This Changes

The PR raises completion-cache refresh timeouts from 30 seconds to 120 seconds in update and doctor paths, adds non-JSON update progress/duration logging, updates regression tests, and adds a changelog entry.

## Best Possible Solution

Land this PR or an equivalent maintainer patch after refreshing against current main and getting exact-head validation green; keep the 120-second cap bounded, preserve JSON-mode silence and the manual-refresh hint, and leave the broader onboarding slowdown tracked separately in #74879.

## Reproduction Assessment

Yes. The PR body gives a concrete path where direct completion-cache generation succeeds in about 36 seconds but the same command under the shipped 30-second spawnSync timeout returns ETIMEDOUT, and current main still has that 30-second cap in both update and doctor paths.

## Solution Assessment

Yes. Raising the existing bounded post-update/doctor timeout and adding non-JSON progress output is the narrowest maintainable fix for slow-but-successful cache refreshes; making the refresh fully asynchronous would be a broader behavior change than this follow-up needs.
