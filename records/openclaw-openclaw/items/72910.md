## Summary

Keep open: current main still hardcodes the 30-second completion-cache timeout in both update and doctor paths, and this PR is a focused plausible bug fix. It should not be cleanup-closed because it still offers useful work, but it needs after-fix real behavior proof and a maintainer choice against the overlapping configurable-timeout PR before merge.

## What This Changes

The PR raises update and doctor completion-cache refresh timeouts from 30 seconds to 120 seconds, adds non-JSON progress and duration logging, and updates regression coverage plus changelog text.

## Best Possible Solution

Land one coherent completion-cache timeout policy, either this bounded 120-second cap with progress logging or the configurable-timeout approach, with update and doctor coverage plus after-fix proof from a real install.

## Reproduction Assessment

Yes from source and PR evidence, though not from a local live run in this read-only review. Current main still uses a 30_000 ms spawnSync timeout, and the PR body reports direct completion generation succeeding after 36.28s while the 30s spawn wrapper returns ETIMEDOUT.

## Solution Assessment

Mostly yes. A bounded 120-second cap plus progress output is a narrow maintainable fix for slow-but-successful cache generation, but maintainers should choose between that default change and the newer configurable-timeout PR before merge.
