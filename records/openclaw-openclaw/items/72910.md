## Summary

Keep open. Current main still kills update and doctor completion-cache refreshes after 30 seconds, while the PR's bounded 120-second timeout, visible non-JSON progress, and regression coverage match the reported slow-but-successful cache generation path; I found no blocking correctness or security issue in the proposed patch.

## What This Changes

The PR increases completion-cache refresh timeouts from 30 seconds to 120 seconds in update and doctor paths, adds non-JSON update progress and duration logging, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Land this PR or an equivalent maintainer patch after exact-head validation: keep the bounded 120-second timeout in both update and doctor cache generation, preserve JSON-mode silence and the manual-refresh hint, keep visible progress for slow non-JSON update follow-up work, and leave broader onboarding performance to a separate focused investigation.

## Reproduction Assessment

Yes. The PR body gives a concrete reproduction where direct completion-cache generation succeeds in about 36 seconds but the same command under the shipped 30-second spawnSync timeout returns ETIMEDOUT, and current main still uses 30_000 ms in both affected paths.

## Solution Assessment

Yes. Raising the existing bounded post-update/doctor timeout and adding progress logging is the narrowest maintainable fix for slow-but-successful completion-cache rebuilds; moving the refresh fully async would be a larger behavior change than this follow-up requires.
