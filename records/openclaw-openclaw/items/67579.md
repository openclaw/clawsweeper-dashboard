## Summary

This should stay open but is not mergeable as-is: the latest head still has source-proven correctness blockers, missing contributor real-behavior proof, and a package-publishing workflow that needs explicit maintainer security/product review.

## What This Changes

The PR adds Control UI provider API-key onboarding with curated catalogs, a FireClaw WhatsApp topbar/web-login fallback, a FireClaw Docker publishing workflow, and many unrelated docs/runtime/test updates.

## Best Possible Solution

Split this into a narrow OpenClaw provider-onboarding change that keeps Gateway as the model-visibility source, handles config/session write failures explicitly, and leaves FireClaw WhatsApp/Docker publishing out unless maintainers approve that scope.

## Reproduction Assessment

Yes. The blockers are source-reproducible from PR head: the browser filters Gateway-visible models, sessions.patch errors are suppressed, stale config hashes can be reused, and the new workflow publishes a mutable GHCR tag.

## Solution Assessment

No. The current PR is not the best way to solve provider onboarding because it combines fork-specific UI/publishing changes with a browser-side duplicate of Gateway-owned model visibility.
