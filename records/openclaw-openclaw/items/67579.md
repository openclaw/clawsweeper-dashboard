## Summary

Keep open. Current main does not contain the proposed API-key panel, and the current PR head still has blocking functional and supply-chain issues, so this needs maintainer review and changes rather than cleanup closure.

## What This Changes

The PR adds a Control UI multi-provider API-key panel with curated provider catalogs, model filtering/session-default updates, WhatsApp topbar/web-login fallback code, assorted docs/scripts/runtime/test changes, and a FireClaw GHCR Docker workflow.

## Best Possible Solution

Split the Control UI provider/API-key setup into a narrow OpenClaw change that relies on Gateway `models.list` configured visibility, preserves the config hash/write contract, surfaces post-save session/model refresh failures, and adds focused UI/Gateway tests. Keep fork-specific FireClaw/WhatsApp/Docker publishing work out of the core PR unless a maintainer explicitly accepts that product scope.

## Reproduction Assessment

Yes. The patch regressions are reproducible from source: an env/auth-profile-backed provider appears in current `models.list` configured view, then disappears under the PR's config-only filter once any keyed provider exists; the session-switch failure path follows from `sessions.patch` catalog validation plus the PR's swallowed error; the tag-publishing issue is visible in the workflow YAML.

## Solution Assessment

No. This PR is not the best way to solve provider onboarding because it duplicates Gateway model visibility logic in the browser, suppresses important write/runtime failures, and mixes unrelated fork-specific publishing and WhatsApp work into the provider-key UI change.
