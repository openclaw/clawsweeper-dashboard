## Summary

Keep open. Current main does not already implement the proposed Control UI provider-key panel, and this recent PR still has source-reproducible functional blockers, a GHCR publishing concern, broad FireClaw-specific scope, and no contributor real-behavior proof.

## What This Changes

The branch adds a Control UI multi-provider API-key panel with curated catalogs and bundles FireClaw WhatsApp topbar/web-login fallback code, a Docker publishing workflow, and unrelated docs/runtime/test changes.

## Best Possible Solution

Split provider/API-key onboarding into a narrow OpenClaw change that uses Gateway `models.list` visibility, preserves config write/retry semantics, surfaces post-save runtime failures, and leaves FireClaw/WhatsApp/Docker publishing work out unless explicitly accepted as product scope.

## Reproduction Assessment

Yes. The merge blockers are source-reproducible from the PR diff and current-main contracts: the browser model filter bypasses Gateway visibility, the session switch can fail through validated `sessions.patch`, and the workflow publishes an unguarded mutable tag.

## Solution Assessment

No. The PR is not the best current solution because it duplicates Gateway model visibility in the browser, suppresses runtime failures, and bundles fork-specific product and publishing changes with provider onboarding.
