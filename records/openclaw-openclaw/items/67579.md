## Summary

Keep open. Current main does not contain the proposed API-key panel or FireClaw/WhatsApp UI additions, and the PR head still introduces blocking functional and supply-chain regressions that need maintainer review before merge.

## What This Changes

The PR adds a Control UI multi-provider API-key panel with curated model catalogs and also bundles FireClaw-specific WhatsApp topbar, web-login fallback, Docker publishing workflow, docs, scripts, and runtime/test changes.

## Best Possible Solution

Keep this PR open for maintainer review, then split the provider/API-key Control UI work from FireClaw fork, WhatsApp, Docker publishing, and unrelated maintenance changes. The provider panel should use the existing Gateway configured-model visibility contract, refresh config hashes on write conflicts, preserve current catalog state across transient reload failures, and surface session-switch errors before merge consideration.

## Reproduction Assessment

Yes for the patch regressions. The model-filter bug follows from an env/auth-profile-backed provider in `models.list` plus any saved config-backed provider; the Docker tag issue is directly visible in the workflow, and the session-switch failure path follows from `sessions.patch` catalog validation during config reload debounce.

## Solution Assessment

No. The current PR is not the best way to solve API-key setup because it bypasses existing Gateway model visibility semantics, suppresses important write failures, and bundles fork-specific publishing/UI changes into a broad feature PR.
