## Summary

Keep open. Current main does not implement the proposed provider-onboarding UI, and the PR head still has source-reproducible correctness and security blockers plus missing contributor real behavior proof.

## What This Changes

The branch adds a Control UI provider API-key panel with curated model catalogs and also bundles FireClaw WhatsApp UI, web-login fallback, Docker publishing, docs, runtime, and test changes.

## Best Possible Solution

Split this into a narrow OpenClaw provider-onboarding change that keeps Gateway authoritative for model visibility, surfaces config/session write failures, and leaves FireClaw WhatsApp/Docker publishing out unless maintainers explicitly approve that scope.

## Reproduction Assessment

Yes. The main blockers are source-reproducible from PR head: browser-side model filtering duplicates and narrows Gateway visibility, sessions.patch failures are swallowed, stale config hashes are reused, transient model reload failures can clear the picker, and the workflow publishes a mutable GHCR tag.

## Solution Assessment

No. The current PR is not the best way to solve provider onboarding because it mixes unrelated fork-specific UI/workflow scope into the feature and moves model visibility policy into the browser instead of relying on Gateway contracts.
