## Summary

Keep open: the PR still has source-reproducible correctness and security blockers, lacks contributor-run real behavior proof, and mixes provider onboarding with fork-specific FireClaw WhatsApp/Docker publishing scope that needs maintainer judgment.

## What This Changes

The branch adds Control UI provider API-key onboarding with curated catalogs, a FireClaw WhatsApp topbar/web-login fallback, a FireClaw Docker publishing workflow, and many unrelated docs/runtime/test updates.

## Best Possible Solution

Split this into a narrow OpenClaw provider-onboarding change that keeps Gateway as the source of model visibility, reports config/session write failures, and leaves FireClaw WhatsApp/Docker publishing out unless maintainers explicitly approve that scope.

## Reproduction Assessment

Yes. The blocking issues are source-reproducible from PR head: browser-side filtering hides Gateway-visible models, post-save sessions.patch errors are suppressed, stale config hashes are reused, and the new workflow publishes a mutable GHCR tag.

## Solution Assessment

No. The PR is not the best way to solve provider onboarding because it duplicates Gateway model-visibility policy in the browser and bundles fork-specific WhatsApp/Docker publishing changes with the UI feature.
