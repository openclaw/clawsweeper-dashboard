The PR adds a Control UI multi-provider API-key panel with curated provider catalogs, FireClaw WhatsApp topbar/web-login fallback code, a FireClaw Docker publishing workflow, and assorted docs/runtime/test changes.

## Best Possible Solution

Split provider/API-key onboarding into a narrow OpenClaw change that keeps Gateway as the model-visibility source, handles config/session write failures explicitly, and leaves FireClaw/WhatsApp/Docker publishing out unless maintainers accept that scope.

## Reproduction Assessment

Yes. The blockers are source-reproducible from the PR diff and current-main contracts: browser-side model filtering bypasses Gateway visibility, sessions.patch failures are suppressed, stale config hashes are reused, and a new package-writing workflow emits a mutable tag.

## Solution Assessment

  - body: This new workflow grants `packages: write` and always emits the mutable `fireclaw-latest` tag, so merging it would publish a fork-specific image from the upstream repository outside the existing Docker release gate.   - confidence: 0.9

  - body: After writing config, this sends `sessions.patch` for the newly saved default model and ignores rejection. Because `sessions.patch` validates against the Gateway catalog, the UI can report success while the active session remains on the old model during catalog reload or validation failure.
  - confidence: 0.88

  - body: `ensureConfigLoaded` returns as soon as any snapshot exists, so after another client edits config and this panel gets a hash-mismatch error, retries keep using the same stale `baseHash`. Reload `config.get` after write conflicts before allowing another save attempt.
  - confidence: 0.84

  - body: `loadModels` returns an empty array on RPC failure, and this assignment replaces the existing model picker during the post-save reload window. Preserve the previous catalog or only overwrite after a successful fetch so a transient Gateway reconnect does not empty the dropdown.
  - confidence: 0.82
