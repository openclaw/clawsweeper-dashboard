## Summary

Keep open: current main still allows fresh setup to persist an env-derived Gateway token as a literal while local CLI/probe credentials can prefer `OPENCLAW_GATEWAY_TOKEN` and Gateway auth remains config-first; the existing warning detects drift but does not align fresh install token sources.

## What This Changes

The issue asks installer and Gateway token setup to avoid writing a literal `gateway.auth.token` that can diverge from `OPENCLAW_GATEWAY_TOKEN` and break CLI calls.

## Best Possible Solution

Choose one install-time source of truth for Gateway token auth, likely preserving an env SecretRef when `OPENCLAW_GATEWAY_TOKEN` is selected or present, and add focused coverage for quickstart, non-interactive setup, and drift diagnostics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist a literal `gateway.auth.token`, set a different `OPENCLAW_GATEWAY_TOKEN`, then local CLI/probe credentials can present env while the Gateway expects config-first auth; I did not run the live one-liner in this read-only review.

## Solution Assessment

Unclear. The report identifies a real drift path, but the best fix requires maintainer choice around Gateway token source precedence and fresh-install defaults rather than a purely mechanical code change.
