## Summary

Keep open: current main still has a source-reproducible path where setup can persist a literal Gateway auth token while local CLI/probe credentials prefer `OPENCLAW_GATEWAY_TOKEN` and the running Gateway resolves configured auth first. Existing diagnostics warn about the drift, but they do not make fresh install token sources stay aligned.

## What This Changes

The issue reports that fresh installs can leave `gateway.auth.token` and `gateway.remote.token` using different token sources, causing CLI calls to fail with a Gateway token mismatch.

## Best Possible Solution

Choose one install-time source of truth for Gateway token auth, likely preserving an env SecretRef when `OPENCLAW_GATEWAY_TOKEN` is selected or present, and add focused coverage for quickstart, non-interactive setup, and drift diagnostics.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create literal `gateway.auth.token`, set a different `OPENCLAW_GATEWAY_TOKEN`, then local CLI/probe auth can present env while the Gateway expects config. I did not run the live one-liner because this review is read-only.

## Solution Assessment

Unclear. The report identifies a real drift path, but the best fix requires maintainer choice around Gateway token source precedence and fresh-install defaults rather than a purely mechanical code change.
