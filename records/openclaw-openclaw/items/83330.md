## Summary

Keep open: current main and v2026.6.8 still allow the source-backed Gateway token drift class. Diagnostic warnings now flag env/config conflicts, but onboarding can still persist `OPENCLAW_GATEWAY_TOKEN` as a literal `gateway.auth.token`, while generic local call/probe paths can select a different env token than the managed Gateway service.

## What This Changes

The issue asks for fresh install/onboarding to keep Gateway server and client token sources aligned when `OPENCLAW_GATEWAY_TOKEN` is used instead of writing a conflicting literal `gateway.auth.token`.

## Best Possible Solution

Choose one canonical Gateway token source for fresh setup and re-onboard, preserve an env SecretRef when env auth is intended, and add doctor/status coverage for existing drift.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: persist a literal `gateway.auth.token`, set a different `OPENCLAW_GATEWAY_TOKEN`, and local clients/probes can send a token that the managed Gateway service does not accept.

## Solution Assessment

Unclear. The current warning path is useful but incomplete; the best fix needs a maintainer decision on the canonical setup token source and migration behavior.
