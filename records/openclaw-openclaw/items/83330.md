## Summary

Keep open: current main and v2026.6.8 still preserve the source-backed mismatch class. The merged diagnostic work warns about env/config token conflicts, but quickstart and non-interactive onboarding can still persist an ambient `OPENCLAW_GATEWAY_TOKEN` as a literal `gateway.auth.token`, while normal local clients and the Gateway service/runtime can select different token sources.

## What This Changes

The issue requests that fresh install/onboarding keep Gateway server and client token sources aligned when `OPENCLAW_GATEWAY_TOKEN` is present instead of writing a divergent literal token.

## Best Possible Solution

Choose and implement one Gateway token source of truth for fresh install/onboarding, likely preserving an env SecretRef when env auth is selected or present, with doctor migration/diagnostic coverage for existing mismatches.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist a literal `gateway.auth.token`, set a different `OPENCLAW_GATEWAY_TOKEN`, then normal local clients can use env-first credentials while the Gateway runtime/service uses config-first auth.

## Solution Assessment

Unclear. The warning implementation is useful but incomplete; the durable fix needs a maintainer decision on whether fresh setup should canonicalize env SecretRefs, plaintext config, or credential precedence plus migration behavior.
