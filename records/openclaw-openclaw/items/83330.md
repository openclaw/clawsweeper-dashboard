- `impact:auth-provider`: The issue is about Gateway auth token selection, SecretRef handling, and client/server credential precedence.

## Summary

Keep open: current main and v2026.5.18 still allow setup to persist a literal Gateway token while non-service CLI credentials can prefer `OPENCLAW_GATEWAY_TOKEN`, and the existing diagnostic only warns after drift already exists.

## What This Changes

The issue asks fresh installs to keep `gateway.auth.token` and env-configured CLI token sources aligned instead of producing a literal token that can diverge from `OPENCLAW_GATEWAY_TOKEN`.

## Best Possible Solution

Choose one install-time Gateway token source of truth, likely preserving an env SecretRef when `OPENCLAW_GATEWAY_TOKEN` is selected or present, and add focused coverage for quickstart, non-interactive setup, and drift diagnostics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist literal `gateway.auth.token`, set a different `OPENCLAW_GATEWAY_TOKEN`, then non-service CLI paths can present env credentials while Gateway auth expects config.

## Solution Assessment

Unclear. The report identifies a real drift path, but the best fix requires a maintainer decision about the canonical Gateway token source for fresh installs and env-configured clients.
