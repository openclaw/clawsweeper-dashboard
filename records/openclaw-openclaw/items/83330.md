## Summary

Keep open: current main still has a source-backed Gateway auth drift path where setup/onboarding can persist `OPENCLAW_GATEWAY_TOKEN` as a literal `gateway.auth.token`, while local CLI/probe callers can prefer the env token and gateway runtime auth remains config-first. The already-merged warning path helps diagnose the mismatch but does not make fresh install token sources stay aligned.

## What This Changes

The issue requests that fresh native installs keep Gateway server and CLI token configuration aligned when `OPENCLAW_GATEWAY_TOKEN` is used.

## Best Possible Solution

Choose one install-time Gateway token source of truth, preferably preserving an env SecretRef when the env var is selected or already present, and add focused quickstart/non-interactive setup plus drift-diagnostic coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist a literal `gateway.auth.token`, set a different `OPENCLAW_GATEWAY_TOKEN`, then local CLI/probe credentials can present env while gateway runtime auth expects config.

## Solution Assessment

Unclear. The report identifies a real drift path, but the best fix requires maintainer choice around Gateway token source precedence and fresh-install defaults rather than a purely mechanical code change.
