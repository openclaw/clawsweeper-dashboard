## Summary

Keep open: current main still has a source-reproducible token drift path where setup can persist a literal Gateway auth token while local CLI/probe credentials prefer `OPENCLAW_GATEWAY_TOKEN` and Gateway auth resolves configured auth first. Existing diagnostics and an explicit SecretRef flag help operators, but they do not make the default fresh-install token sources stay aligned.

## What This Changes

The issue requests that installer/setup keep Gateway server and CLI token sources aligned when `OPENCLAW_GATEWAY_TOKEN` is used, instead of persisting a divergent literal `gateway.auth.token`.

## Best Possible Solution

Choose one install-time Gateway token source of truth, likely preserving an env SecretRef when `OPENCLAW_GATEWAY_TOKEN` is selected or present, then cover quickstart, non-interactive onboarding, and drift diagnostics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: literal `gateway.auth.token` plus a different `OPENCLAW_GATEWAY_TOKEN` makes local CLI/probe credentials env-first while Gateway auth stays config-first; I did not run the live one-liner in this read-only review.

## Solution Assessment

Unclear. The requested direction is plausible, but choosing env-ref preservation, plaintext defaults, or precedence changes is an auth contract decision; the existing `--gateway-token-ref-env` path is only an explicit workaround, not a default fresh-install fix.
