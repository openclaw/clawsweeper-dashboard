- `P2`: The issue is a normal-priority install/auth mismatch with limited blast radius but real operator impact.
- `impact:auth-provider`: The failure is caused by Gateway token source precedence and SecretRef/env credential resolution drift.

## Summary

Keep open: current main and the latest shipped tag still preserve a source-reproducible Gateway token drift path. The existing diagnostics now warn about the conflict, but they do not make fresh install/setup token sources stay aligned when `OPENCLAW_GATEWAY_TOKEN` and `gateway.auth.token` diverge.

## What This Changes

Fresh installer/setup flows should keep Gateway server and local CLI token sources aligned when `OPENCLAW_GATEWAY_TOKEN` is used, instead of persisting a divergent literal `gateway.auth.token`.

## Best Possible Solution

Choose one install-time Gateway token source of truth, likely preserving an env SecretRef when `OPENCLAW_GATEWAY_TOKEN` is selected or present, and cover quickstart, non-interactive setup, shipped upgrade behavior, and conflict diagnostics together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: setup can persist `OPENCLAW_GATEWAY_TOKEN` as a literal `gateway.auth.token`, local CLI/probe paths can prefer the env token, and Gateway auth remains config-first; I did not run the live installer one-liner.

## Solution Assessment

Unclear. The report identifies a real bug, but the best fix requires maintainers to choose the Gateway token source-of-truth contract before changing setup defaults or credential precedence.
