## Summary

Keep open: current main still has a source-reproducible installer/onboarding path where an env-provided Gateway token can be persisted as a literal config value, while command/probe credentials can prefer `OPENCLAW_GATEWAY_TOKEN` and the Gateway auth runtime remains config-first. The merged diagnostic warns about this split, but it does not make fresh install token sources stay aligned.

## What This Changes

The issue asks OpenClaw to keep installer-written `gateway.auth.token` and env-configured Gateway CLI credentials aligned so fresh installs do not fail with token mismatch.

## Best Possible Solution

Choose one install-time source of truth for Gateway token auth, likely preserving an env SecretRef when `OPENCLAW_GATEWAY_TOKEN` is selected or present, and add focused coverage for quickstart, non-interactive setup, and drift diagnostics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create a literal `gateway.auth.token`, set a different `OPENCLAW_GATEWAY_TOKEN`, then local command/probe auth can present env while Gateway auth expects the configured token.

## Solution Assessment

Unclear. The report identifies a real drift path, but the best fix requires maintainer choice around Gateway token source precedence and fresh-install defaults rather than a purely mechanical code change.
