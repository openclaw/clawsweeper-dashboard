## Summary

Keep open: current main and the latest release still show a source-reproducible Gateway token-source mismatch path. The installer delegates fresh setup to onboarding, quickstart/non-interactive setup can persist an env-derived token as a literal, and current credential precedence can leave local clients and the managed Gateway using different token sources; existing diagnostics warn but do not align fresh installs.

## What This Changes

The issue asks installer/onboarding to keep Gateway auth token config and `OPENCLAW_GATEWAY_TOKEN` aligned so fresh installs do not produce token mismatches.

## Best Possible Solution

Choose one Gateway token source of truth for fresh install/onboarding, likely preserving an env SecretRef when `OPENCLAW_GATEWAY_TOKEN` is selected or present, and cover quickstart, non-interactive setup, service install, and drift diagnostics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: persist a literal `gateway.auth.token`, set a different `OPENCLAW_GATEWAY_TOKEN`, then local call/probe credentials can use env while the Gateway runtime uses config-first auth.

## Solution Assessment

Unclear. The reported direction is valid, but the best fix requires maintainer choice around the Gateway token source-of-truth contract for fresh installs and upgrades.
