## Summary

Current `main` has useful guardrails and an explicit `--gateway-token-ref-env` path, but the native promptable setup path can still persist an env-derived Gateway token as a plaintext `gateway.auth.token`, and local client/service precedence can still diverge when `OPENCLAW_GATEWAY_TOKEN` later differs. This should stay open for the existing maintainer/product decision rather than be swept closed.

## What This Changes

The issue asks OpenClaw setup to avoid Gateway auth token drift when `OPENCLAW_GATEWAY_TOKEN` is used alongside `gateway.auth.token` or `gateway.remote.token`.

## Best Possible Solution

Maintainers should choose and document one canonical Gateway token source for setup, then align interactive/non-interactive onboarding, daemon install, CLI credential precedence, and doctor/status repair around that source.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction shape: create or keep a literal `gateway.auth.token`, set a different `OPENCLAW_GATEWAY_TOKEN`, then normal clients can choose the env token while the managed Gateway process chooses the config token. I did not run the official one-liner because this review is read-only and should not mutate the checkout or machine state.

## Solution Assessment

Unclear. The requested env-ref-by-default direction is plausible, but the best fix needs maintainer confirmation because it changes setup source-of-truth and likely requires doctor/status migration behavior for existing plaintext configs.
