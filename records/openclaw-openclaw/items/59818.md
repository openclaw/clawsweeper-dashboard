## Summary

The PR should stay open for review, not cleanup-close: current main still has the ASCII-only Discord mention matcher, but this branch introduces wrong-user mention and identity-safety risks that block merge as written.

## What This Changes

The PR expands Discord mention rewriting for CJK/Unicode handles, adds cross-account directory-cache resolution and preflight cache seeding, and wires rewriting into Discord reply paths with tests.

## Best Possible Solution

Land a Discord-only fix that supports Unicode/CJK mention candidates while preserving tight mention boundaries, account/guild isolation, policy-gated cache writes, and fail-closed bot identity handling.

## Reproduction Assessment

Yes. A focused source reproduction is to cache a CJK handle and call `rewriteDiscordKnownMentions("@架构师", { accountId: "default" })`; current main's regex only accepts ASCII handles, so the CJK candidate is never resolved.

## Solution Assessment

No. Unicode-aware parsing is the right direction, but this PR's account-global fallback, pre-policy cache writes, guild nickname caching, and unverified identity fallback are not the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope cross-account fallback before rewriting mentions:** `extensions/discord/src/directory-cache.ts:123-130`
  - body: The fallback searches every other account cache and returns the first matching handle. In a process serving multiple accounts or guilds, duplicate display names resolve by insertion order, so outbound text can ping the wrong Discord user; scope this by guild/context or require an unambiguous authorized match.
  - confidence: 0.91
- **[P1] Move directory cache writes behind policy gates:** `extensions/discord/src/monitor/message-handler.preflight.ts:378-437`
  - body: Preflight caches author and mentioned-user handles before later own-message, mention-required, allowBots, and ignoreOtherMentions drops. Rejected senders or channels can seed the shared cache that later outbound rewrites consume; populate after acceptance or scope entries to the validated context.
  - confidence: 0.89
- **[P1] Keep guild nicknames out of the account-global cache:** `extensions/discord/src/monitor/message-handler.preflight.ts:393`
  - body: `member.nick` is guild-scoped, but this cache is keyed only by account and outbound rewriting has no guild context. The same nickname in two guilds can overwrite the mapping and later rewrite to the wrong user.
  - confidence: 0.9
- **[P2] Keep mention matching out of URLs and canonical mentions:** `extensions/discord/src/mentions.ts:11`
  - body: The new `[^\p{L}\p{N}]` prefix treats `/` and `<` as valid mention separators. Cached handles can therefore rewrite URL paths like `https://x.com/@alice` or already-canonical mention text like `<@123>`; keep a tighter Unicode-aware boundary.
  - confidence: 0.84
- **[P1] Preserve verified bot identity resolution:** `extensions/discord/src/monitor/provider.startup.ts:226-236`
  - body: When `fetchUser("@me")` fails, the PR returns `applicationId` as `botUserId`. Current main accepts a token-derived id or a verified fetch result; using a configured or probed application id as bot identity can make self-message and mention-required checks unreliable.
  - confidence: 0.82
