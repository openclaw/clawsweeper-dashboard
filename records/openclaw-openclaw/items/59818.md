## Summary

Keep open: current main still has an ASCII-only Discord mention candidate matcher, so the CJK rewrite problem is real, but this branch is conflicting, lacks after-fix real behavior proof, and introduces context/identity risks that should block merge.

## What This Changes

The PR expands Discord mention rewriting to Unicode/CJK handles, adds cross-account directory-cache seeding and resolution in Discord startup/preflight/reply paths, and adds tests.

## Best Possible Solution

Land a rebased Discord-only Unicode mention fix that preserves account/guild isolation, caches only accepted context-scoped identities, keeps tight mention boundaries, and retains fail-closed bot identity handling.

## Reproduction Assessment

Yes. Source inspection gives a focused reproduction: cache `架构师` with `rememberDiscordDirectoryUser` and call `rewriteDiscordKnownMentions("@架构师", { accountId: "default" })`; current main's ASCII regex cannot match that handle.

## Solution Assessment

No. Unicode mention parsing is the right direction, but the branch also broadens directory resolution, writes shared cache entries before policy acceptance, and weakens bot identity failure handling instead of making the narrow parser fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope cross-account fallback before rewriting mentions:** `extensions/discord/src/directory-cache.ts:123-130`
  - body: The fallback searches every account sub-map and returns the first matching handle. In a process serving multiple accounts or guilds, duplicate display names resolve by insertion order, so outbound text can ping the wrong Discord user; scope this by guild/context or require an unambiguous authorized match.
  - confidence: 0.91
- **[P1] Move cache writes behind accepted-message gates:** `extensions/discord/src/monitor/message-handler.preflight.ts:386-437`
  - body: Preflight caches author and mentioned-user handles before own-message, guild/channel, member access, bot, mention-required, and ignoreOtherMentions drops. Rejected traffic can seed the shared cache that outbound rewrites later consume, so populate after acceptance or scope entries to the validated context.
  - confidence: 0.89
- **[P1] Keep guild nicknames out of the account-global cache:** `extensions/discord/src/monitor/message-handler.preflight.ts:393`
  - body: `member.nick` is guild-scoped, but this cache is keyed only by account and outbound rewriting has no guild context. The same nickname in two guilds can overwrite the mapping and later rewrite to the wrong user.
  - confidence: 0.9
- **[P1] Preserve fail-closed bot identity startup:** `extensions/discord/src/monitor/provider.startup.ts:231-236`
  - body: When `fetchUser("@me")` fails, the PR returns `applicationId` as `botUserId`. Current main intentionally throws before lifecycle on identity failure; using an unverified fallback can make self-message and mention-required checks unreliable.
  - confidence: 0.82
- **[P2] Keep Unicode mentions out of URLs and canonical mentions:** `extensions/discord/src/mentions.ts:11`
  - body: The new `[^\p{L}\p{N}]` prefix treats `/` and `<` as valid mention separators. Cached handles can therefore rewrite URL paths like `https://x.com/@alice` or already-canonical mention text like `<@123>`; keep a tighter Unicode-aware boundary.
  - confidence: 0.84
