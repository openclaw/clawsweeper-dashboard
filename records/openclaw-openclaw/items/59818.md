## Summary

Keep this PR open: the underlying Discord CJK mention bug is valid, but the current patch still needs real behavior proof and has blocking correctness/security issues around unscoped directory resolution, pre-policy cache writes, and bot identity handling.

## What This Changes

The PR updates Discord mention rewriting for Unicode/CJK handles, adds cross-account directory-cache seeding/resolution in startup and preflight paths, applies rewriting in reply paths, and adds tests.

## Best Possible Solution

Land a Discord-only fix that supports Unicode mention candidates while preserving tight mention boundaries, verified bot identity handling, and context-scoped or explicit directory resolution instead of account-global fallback.

## Reproduction Assessment

Yes. A focused source reproduction is to cache `架构师` with `rememberDiscordDirectoryUser` and call `rewriteDiscordKnownMentions("@架构师", { accountId: "default" })`; current main's ASCII matcher cannot match that handle.

## Solution Assessment

No. The Unicode parser direction is appropriate, but the proposed account-global fallback, pre-policy cache writes, guild nickname caching, and configured-application-id identity fallback are not the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Scope cross-account fallback before rewriting mentions:** `extensions/discord/src/directory-cache.ts:123-130`
  - body: The fallback searches every account sub-map and returns the first matching handle. In a process serving multiple accounts or guilds, duplicate display names resolve by insertion order, so outbound text can ping the wrong Discord user; scope this by guild/context or require an unambiguous authorized match.
  - confidence: 0.91
- **[P1] Move directory cache writes behind policy gates:** `extensions/discord/src/monitor/message-handler.preflight.ts:378-437`
  - body: Preflight caches author and mentioned-user handles before own-message, bot, and mention-required drops. Rejected traffic can seed the shared cache that later outbound rewrites consume; populate after acceptance or scope entries to the validated context.
  - confidence: 0.89
- **[P1] Keep guild nicknames out of the account-global cache:** `extensions/discord/src/monitor/message-handler.preflight.ts:388-393`
  - body: `member.nick` is guild-scoped, but this cache is keyed only by account and outbound rewriting has no guild context. The same nickname in two guilds can overwrite the mapping and later rewrite to the wrong user.
  - confidence: 0.9
- **[P2] Keep mention matching out of URLs and canonical mentions:** `extensions/discord/src/mentions.ts:11`
  - body: The new `[^\p{L}\p{N}]` prefix treats `/` and `<` as valid mention separators. Cached handles can therefore rewrite URL paths like `https://x.com/@alice` or already-canonical mention text like `<@123>`; keep a tighter Unicode-aware boundary.
  - confidence: 0.84
- **[P1] Preserve fail-closed bot identity resolution:** `extensions/discord/src/monitor/provider.startup.ts:226-236`
  - body: When `fetchUser("@me")` fails, the PR returns a configured application id as `botUserId`. Current main fails closed unless identity is token-derived or fetched, so this fallback can make self-message and mention-required checks unreliable.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the diff introduces Discord data-isolation and identity-gating concerns that need owner review.

Concerns:

- **[medium] Account-global fallback can mention the wrong user:** `extensions/discord/src/directory-cache.ts:123`
  - body: The cross-account fallback resolves handles from another account or guild without context, so display-name collisions can notify the wrong Discord user.
  - confidence: 0.9
- **[medium] Rejected messages can poison the handle cache:** `extensions/discord/src/monitor/message-handler.preflight.ts:378`
  - body: Preflight cache writes happen before several policy drops, allowing ignored senders or channels to influence later outbound mention rewriting in allowed contexts.
  - confidence: 0.87
- **[medium] Configured application id can weaken identity checks:** `extensions/discord/src/monitor/provider.startup.ts:226`
  - body: Falling back from failed `fetchUser("@me")` to a configured/probed application id can keep the monitor running with an unverified bot user id, affecting self-message and mention-required gates.
  - confidence: 0.82
