## Summary

Keep open: current main still appears to have the CJK mention-rewrite gap, but this PR is not merge-ready because it broadens Discord identity/cache resolution in ways that can ping the wrong user and still lacks real Discord behavior proof.

## What This Changes

The branch expands Discord mention rewriting to Unicode/CJK handles, adds cross-account directory-cache seeding and lookup, rewrites additional outbound paths, and adds focused Discord mention/cache tests.

## Best Possible Solution

Land a Discord-only fix that supports Unicode/CJK mention candidates while keeping handle resolution scoped and deterministic, writing cache entries only after accepted policy gates, and proving the result in a real Discord setup.

## Reproduction Assessment

Yes, at source level: current main's regex only matches ASCII handles, so cached CJK handles such as @架构师 are not selected for rewriting. I did not run a live Discord chain in this read-only review.

## Solution Assessment

No. Unicode-aware parsing is the right direction, but this branch broadens directory resolution, cache writes, and startup identity fallback beyond a narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope cross-account fallback before rewriting mentions:** `extensions/discord/src/directory-cache.ts:123-130`
  - body: The fallback searches every other account cache and returns the first matching handle. In a process serving multiple accounts or guilds, duplicate display names resolve by insertion order, so outbound text can ping the wrong Discord user; scope this by guild/context or require an unambiguous authorized match.
  - confidence: 0.91
- **[P1] Move directory cache writes behind policy gates:** `extensions/discord/src/monitor/message-handler.preflight.ts:386-437`
  - body: Preflight caches author and mentioned-user handles before later own-message, allowlist, allowBots, mention-required, and command authorization drops. Rejected traffic can seed the shared cache that later outbound rewrites consume; populate after acceptance or scope entries to the validated context.
  - confidence: 0.89
- **[P1] Keep guild nicknames out of the account-global cache:** `extensions/discord/src/monitor/message-handler.preflight.ts:393`
  - body: `member.nick` is guild-scoped, but this cache is keyed only by account and outbound rewriting has no guild context. The same nickname in two guilds can overwrite the mapping and later rewrite to the wrong user.
  - confidence: 0.9
- **[P2] Keep mention matching out of URLs and canonical mentions:** `extensions/discord/src/mentions.ts:11`
  - body: The new non-letter/non-number prefix treats `/` and `<` as valid mention separators. Cached handles can therefore rewrite URL paths or already-canonical mention text like `<@123>`; keep a tighter Unicode-aware boundary.
  - confidence: 0.84
- **[P1] Preserve verified bot identity resolution:** `extensions/discord/src/monitor/provider.startup.ts:229-236`
  - body: When `fetchUser("@me")` fails, the PR returns `applicationId` as `botUserId`. Current main accepts a token-derived id or a verified fetch result; using a configured or probed application id as bot identity can make self-message and mention-required checks unreliable.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: No workflow or dependency supply-chain issue was found, but the diff introduces Discord identity and mention-isolation risks that need owner review.

Concerns:

- **[medium] Account-global fallback can ping the wrong Discord user:** `extensions/discord/src/directory-cache.ts:123`
  - body: Cross-account fallback resolves handles from another account or guild without context, so duplicate display names can notify the wrong user.
  - confidence: 0.9
- **[medium] Rejected messages can poison handle resolution:** `extensions/discord/src/monitor/message-handler.preflight.ts:386`
  - body: Preflight cache writes happen before later policy drops, allowing blocked channels or senders to influence outbound mention rewriting in allowed contexts.
  - confidence: 0.87
- **[medium] Guild nicknames are stored without guild scope:** `extensions/discord/src/monitor/message-handler.preflight.ts:393`
  - body: Guild-specific nicknames enter an account-global cache, so the same nickname in another guild can later rewrite to the wrong user ID.
  - confidence: 0.88
- **[medium] Application-id fallback can weaken identity gates:** `extensions/discord/src/monitor/provider.startup.ts:229`
  - body: Falling back from failed `fetchUser("@me")` to `applicationId` can keep the monitor running with an unverified bot user id, affecting self-message and mention-required gates.
  - confidence: 0.82
