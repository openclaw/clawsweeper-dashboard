## Summary

Keep open. Current main still has the ASCII-only Discord mention candidate regex, so the reported CJK mention gap is real, but this PR is not mergeable as-is because its account-global cross-account cache fallback and guild nickname caching can rewrite mentions to the wrong Discord user across accounts or guilds.

## What This Changes

The PR expands Discord mention rewriting to Unicode handles, adds cross-account directory-cache lookup plus startup/preflight cache seeding, wires outbound reply/process rewriting, and adds CJK/cache regression tests.

## Best Possible Solution

Land a current-main Discord plugin fix that broadens mention candidates to Unicode while preserving safe mention boundaries, populates directory data only from accepted/authorized context, and scopes any cross-account resolution by guild/context or limits it to unambiguous bot identity entries; consolidate the remaining Unicode parser work with #43357.

## Reproduction Assessment

Yes. A focused unit reproduction is to cache a CJK handle with `rememberDiscordDirectoryUser` and call `rewriteDiscordKnownMentions("@架构师", { accountId: "default" })`; current main's ASCII regex never matches it, although I did not run tests because this review was read-only.

## Solution Assessment

No. The Unicode regex/test direction is appropriate, but the account-global cache fallback, pre-policy cache writes, and guild nickname handling are not the narrowest maintainable solution because they can affect unrelated accounts or guilds.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Scope cross-account fallback before rewriting mentions:** `extensions/discord/src/directory-cache.ts:122-129`
  - body: The new fallback returns the first matching handle from any other account cache. In a process serving multiple Discord accounts or guilds, duplicate display names resolve by map insertion order, so `@alice` can be rewritten to the wrong `<@id>`. Make this lookup guild-scoped or only use it when the match is unambiguous.
  - confidence: 0.91
- **[P1] Move directory cache writes behind policy checks:** `extensions/discord/src/monitor/message-handler.preflight.ts:386-409`
  - body: The preflight cache population runs immediately after `allowBotsMode`, before channel allowlist, member access, mention-required, and bot-message drops. A blocked sender or channel can seed account-global handles that later affect allowed outbound messages, causing wrong-user pings.
  - confidence: 0.88
- **[P1] Do not cache guild nicknames in an account-global map:** `extensions/discord/src/monitor/message-handler.preflight.ts:396`
  - body: `member.nick` is guild-scoped, but this cache is keyed only by account. If the same bot account is in multiple guilds, the same nickname can map to different users and the most recent cache write can make later outbound rewrites mention the wrong user.
  - confidence: 0.9
- **[P2] Keep mention parsing out of URLs and canonical mentions:** `extensions/discord/src/mentions.ts:9`
  - body: Changing the prefix to any non-letter/digit also matches URL paths like `/@alice` and already-formatted mentions like `<@123>`. Cached handles can corrupt links or double-wrap mention markup; keep a tighter boundary that adds Unicode/CJK punctuation without matching these contexts.
  - confidence: 0.84
- **[P2] Seed startup cache with display names too:** `extensions/discord/src/monitor/provider.ts:1014`
  - body: Startup seeding stores only `botUserName`, which is computed as `username ?? globalName`. Since Discord bot usernames are normally present, a CJK display/global name is not cached at startup, so first-contact `@displayName` rewrites can still fail until that bot appears in inbound preflight traffic.
  - confidence: 0.8
