## Summary

Keep open. The CJK mention bug is real on current main, but this PR is not safe to merge as-is because it makes Discord mention resolution account-global, writes cache entries before inbound policy gates, and conflicts with current-main bot-identity safety behavior.

## What This Changes

The PR expands Discord mention rewriting to Unicode handles, adds cross-account directory-cache lookup plus startup/preflight cache seeding, wires outbound reply/process rewriting, and adds CJK/cache regression tests.

## Best Possible Solution

Land a current-main Discord plugin fix that broadens outbound mention parsing to Unicode while keeping tight mention boundaries, preserving fail-closed identity handling, and scoping any directory-cache sharing by authorized guild/context or explicit aliases.

## Reproduction Assessment

Yes. A focused reproduction is to cache a CJK handle with `rememberDiscordDirectoryUser` and call `rewriteDiscordKnownMentions("@架构师", { accountId: "default" })`; current main's ASCII regex never matches it. I did not run tests because this review was read-only.

## Solution Assessment

No. The Unicode parser direction is appropriate, but the proposed account-global fallback and pre-policy cache writes are not the narrowest maintainable fix for Discord mention rewriting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope cross-account fallback before rewriting mentions:** `extensions/discord/src/directory-cache.ts:123-130`
  - body: The fallback returns the first matching handle from any other account cache. In a process serving multiple accounts or guilds, duplicate display names resolve by insertion order, so `@alice` can be rewritten to the wrong Discord user ID. Scope this lookup by guild/context or only use it when the match is unambiguous.
  - confidence: 0.91
- **[P1] Move directory cache writes behind policy checks:** `extensions/discord/src/monitor/message-handler.preflight.ts:386-433`
  - body: Preflight cache population runs before own-message, bot, guild/channel, member access, and mention-required drops. A blocked sender or channel can seed account-global handles that later affect allowed outbound messages, causing wrong-user pings. Populate the cache only after the message is accepted or scope entries to the validated context.
  - confidence: 0.89
- **[P1] Do not cache guild nicknames in an account-global map:** `extensions/discord/src/monitor/message-handler.preflight.ts:393`
  - body: `member.nick` is guild-scoped, but this cache is keyed only by account and outbound rewriting has no guild context. If the same bot account is in multiple guilds, the same nickname can map to different users and later rewrites can mention the wrong user.
  - confidence: 0.9
- **[P1] Preserve fail-closed bot identity startup:** `extensions/discord/src/monitor/provider.startup.ts:231-236`
  - body: This fallback lets the monitor continue with `applicationId` when `fetchUser("@me")` fails. Current main intentionally fails closed when bot identity cannot be verified, because own-message and mention-required checks depend on the real bot user ID. Keep that behavior or use only a verified token-derived bot ID path.
  - confidence: 0.84
- **[P2] Keep mention matching out of URLs and canonical mentions:** `extensions/discord/src/mentions.ts:9`
  - body: Changing the prefix to any non-letter/digit also matches URL paths like `/@alice` and already-formatted mentions like `<@123>`. Cached handles can corrupt links or double-wrap mention markup; keep a tighter boundary that adds Unicode/CJK punctuation without matching those contexts.
  - confidence: 0.84
- **[P2] Rewrite full reply text before chunking:** `extensions/discord/src/monitor/reply-delivery.ts:302`
  - body: `sendDiscordPayloadText` chunks the reply before this new rewrite runs on each chunk. If an `@handle` lands across a chunk boundary, neither chunk contains a complete mention candidate and the rewrite is skipped. Apply mention rewriting to the full outbound text before chunking.
  - confidence: 0.78
- **[P2] Seed startup cache with display names too:** `extensions/discord/src/monitor/provider.ts:1017`
  - body: Startup seeding stores only `botUserName`, which is computed as `username ?? globalName`. Since Discord bot usernames are normally present, a CJK display/global name is not cached at startup, so first-contact `@displayName` rewrites can still fail until that bot appears in inbound preflight traffic.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the diff introduces Discord data-isolation and identity-gating concerns that need owner review.

Concerns:

- **[medium] Account-global fallback can mention the wrong user:** `extensions/discord/src/directory-cache.ts:123`
  - body: The cross-account fallback resolves handles from another account or guild without context, so display-name collisions can notify the wrong Discord user.
  - confidence: 0.9
- **[medium] Rejected messages can poison the handle cache:** `extensions/discord/src/monitor/message-handler.preflight.ts:386`
  - body: Preflight cache writes happen before policy drops, allowing blocked channels or senders to influence later outbound mention rewriting in allowed contexts.
  - confidence: 0.87
- **[medium] Bot identity fallback weakens fail-closed startup:** `extensions/discord/src/monitor/provider.startup.ts:231`
  - body: Returning an application-id fallback after REST identity failure can keep the monitor running with an unverified bot user ID, making self-message and mention-required checks unreliable.
  - confidence: 0.82
