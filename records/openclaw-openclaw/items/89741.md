## Summary

Keep open: current main still has retry-only memory-index rename behavior, and this PR is a focused fix with adequate Windows CLI proof. I found no blocking line-level patch findings, but the fallback changes SQLite memory-index swap behavior, so maintainers should consciously accept the bounded session-state merge risk.

## What This Changes

The PR adds a copyFile/unlink fallback after exhausted EPERM rename retries in memory-core atomic reindexing, plus focused regression coverage and a Windows CI memory-index proof script.

## Best Possible Solution

Land the narrow memory-core EPERM fallback after maintainer review of the session-state risk, and keep broader EBUSY or orphan-cleanup handling as follow-up unless maintainers want that scope in this PR.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main exhausts transient EPERM retries in `renameWithRetry` and then throws on the `openclaw memory index --force` safe-reindex path. I did not reproduce the original Windows locked-handle environment locally.

## Solution Assessment

Yes, this is an acceptable narrow fix because it changes the existing memory index swap helper after the existing retry budget and adds both deterministic EPERM coverage and Windows CLI proof. The safer broader alternative is to extend cleanup/retry handling for unlink or persistent EBUSY before merge, but that is not required for the EPERM-only bug scope.
