## Summary

Keep open: this PR addresses a real current-main and shipped gateway reload-planning failure, but the proposed guard is still too narrow because a throwing high-priority prefix getter can be treated as an empty list while lower-priority hot/no-op prefixes remain active.

## What This Changes

The PR wraps plugin reload registration reads in gateway config reload planning and adds a regression test for a throwing plugin reload registration getter.

## Best Possible Solution

Land one canonical gateway reload metadata hardening path that treats unreadable prefix groups atomically as malformed or fail-closed, with regression coverage for plugin and channel descriptor cases.

## Reproduction Assessment

Yes, from source inspection and the PR's recorded red proof. Current main and v2026.6.9 directly dereference plugin reload registration fields during rule construction, and a throwing getter would abort planning before this guard.

## Solution Assessment

No. Guarding the registration getter is useful, but per-prefix catch-and-empty semantics can still let lower-priority hot or no-op rules survive after a higher-priority prefix getter throws.
