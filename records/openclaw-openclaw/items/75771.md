## Summary

Close as not actionable in the OpenClaw source repository. The reported WeChat target failure is a plugin-owned target semantics gap in the external `@tencent-weixin/openclaw-weixin` package, while current OpenClaw core already exposes and documents the `messaging.targetResolver` and `inferTargetChatType` hooks that the plugin needs.

## What This Changes

The issue requests that `openclaw message send --channel openclaw-weixin` stop rejecting WeChat targets such as `filehelper`, `wxid_*`, and other opaque `to_user_id` strings before dispatch reaches the external plugin.

## Best Possible Solution

Keep OpenClaw core channel-agnostic and have the external Weixin plugin publish target-resolution support: at minimum `messaging.targetResolver.resolveTarget` plus suitable `looksLikeId` and `inferTargetChatType` behavior for `filehelper`, `wxid_*`, and other valid WeChat `to_user_id` forms, or a directory surface that resolves those contacts.

## Reproduction Assessment

Yes, partially. Current main gives a high-confidence static reproduction for bare opaque targets such as `filehelper` when the loaded plugin has an empty/missing directory and no `resolveTarget` hook; the stronger claim that every prefixed form must fail is not fully supported by current source because id-like prefixes can take the normalized fallback path.

## Solution Assessment

No, the requested generic core pass-through is not the best fix. The narrower maintainable solution is plugin-owned target resolution, because accepting arbitrary opaque strings in core would weaken validation for every plugin channel and would encode WeChat-specific assumptions outside the WeChat plugin.
