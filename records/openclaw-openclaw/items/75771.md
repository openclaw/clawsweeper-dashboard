## Summary

Keep open. Current `main` still has the resolver behavior that rejects a bare opaque WeChat target when the loaded channel plugin has no directory match and no `messaging.targetResolver.resolveTarget` hook; the remaining question is whether OpenClaw should add a generic compatibility/opt-in path or require the external WeChat plugin to implement the existing resolver seam.

## What This Changes

The issue reports that `openclaw message send` rejects `openclaw-weixin` outbound targets before the plugin can validate or send opaque WeChat IDs such as `filehelper`.

## Best Possible Solution

Keep core channel-agnostic and resolve the product boundary explicitly: either update the external WeChat plugin to declare `looksLikeId`/`resolveTarget` for `filehelper`, `wxid_*`, and other valid direct IDs, or add a documented generic opt-in for opaque targets that channel plugins must enable intentionally. Avoid hardcoding WeChat-specific target names in core.

## Reproduction Assessment

Yes. Static source inspection gives a high-confidence reproduction path: a loaded channel plugin with no directory entries and no `messaging.targetResolver.resolveTarget` receives a bare non-id-like target such as `filehelper`, then `resolveMessagingTarget` falls through to `unknownTargetError`.

## Solution Assessment

Unclear. A broad host pass-through for all unresolved plugin targets is not clearly the best fix because the existing SDK already gives channel plugins a target resolver; the safer direction is a plugin-owned resolver or an explicit core opt-in rather than implicit global fallback.
