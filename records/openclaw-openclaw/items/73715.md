## Summary

Keep open. Current main still forwards the user-supplied model casing on both local and gateway `infer model run` paths, and open closing PRs already target this bug, so the issue should remain open until maintainers land or consolidate a fix.

## What This Changes

The issue asks the CLI to canonicalize or reject case-mismatched `--model <provider>/<model>` values before dispatch so mixed-case Anthropic model ids do not surface as downstream no-output errors.

## Best Possible Solution

Land one consolidated fix that canonicalizes unique known catalog matches, or rejects case-only mismatches before provider calls, across both local and gateway dispatch with focused regression coverage.

## Reproduction Assessment

Yes. The issue provides a deterministic CLI matrix, and current-main source inspection confirms mixed-case model values still flow through local and gateway dispatch without a catalog-aware canonicalization or upfront rejection step.

## Solution Assessment

Yes for the requested direction. Catalog-aware unique-match canonicalization is the safest maintainable fix because it handles known model refs while preserving unknown custom ids; fail-fast `Unknown model` is also acceptable if maintainers prefer literal model semantics.
