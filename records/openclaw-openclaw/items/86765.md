## What This Changes

The PR changes memory-core sync/close ordering to register provider-initialization-blocked syncs before teardown, track providers observed during shutdown, and add regression tests for pending-sync provider close.

## Best Possible Solution

Land the focused memory-core lifecycle fix after exact-head CI/maintainer review, keeping the provider and SQLite teardown ordering covered by the new regression tests.

## Reproduction Assessment

Yes. Source inspection of current main gives a high-confidence interleaving: sync does not register this.syncing until after provider initialization, while close snapshots pending sync before that await; I did not execute tests in this read-only review.

## Solution Assessment

Yes. Registering sync before provider initialization and waiting for the current sync before provider/SQLite teardown directly addresses the race without adding config, API, or product surface.
