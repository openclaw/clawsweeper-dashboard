- **[P1] Keep runtime.channel.turn compatibility:** `src/plugins/runtime/runtime-channel.ts:182-186`
  - body: This replaces the shipped `runtime.channel.turn` object with `runtime.channel.inbound` without a forwarding alias. Existing channel plugins that still call `runtime.channel.turn.run(...)`, `runPrepared(...)`, or `buildContext(...)` will either fail type-checking or hit `undefined` at runtime after upgrade, so keep a deprecated `turn` shim until an approved breaking release.
  - confidence: 0.93
- **[P1] Restore channel-inbound Turn aliases:** `src/plugin-sdk/channel-inbound.ts:79-87`
  - body: The `channel-inbound` entrypoint no longer exports shipped Turn compatibility names such as `buildChannelTurnContext`, `filterChannelTurnSupplementalContext`, `buildChannelTurnMediaPayload`, and the related Turn type aliases. Those names are present in the latest release, so external plugins importing them from this public subpath will break unless they remain as deprecated aliases.
  - confidence: 0.92
- **[P1] Restore inbound-reply-dispatch legacy names:** `src/plugin-sdk/inbound-reply-dispatch.ts:80-99`
  - body: This renames `runPreparedInboundReplyTurn`/`runInboundReplyTurn` and the dropped-history exports on the shipped `inbound-reply-dispatch` subpath without keeping the old names. Please re-export the previous function and type names as deprecated forwarding aliases so existing plugins do not fail import resolution on upgrade.
  - confidence: 0.91
- **[P1] Re-export legacy channel-message predicates:** `src/plugin-sdk/channel-message.ts:11-29`
  - body: `channel-message` now forwards mostly to `channel-outbound` and no longer exposes the shipped `hasFinalChannelTurnDispatch`, `hasVisibleChannelTurnDispatch`, and `resolveChannelTurnDispatchCounts` names. Existing plugin code importing those predicates from `openclaw/plugin-sdk/channel-message` will break, so keep deprecated aliases on the old entrypoint.
  - confidence: 0.88
- **[P2] Preserve Signal quote sender without quote text:** `extensions/signal/src/monitor/event-handler.ts:208-214`
  - body: `resolveSignalQuoteContext` can produce a visible quote sender even when the quoted message has empty text, but this block only passes supplemental quote context when `entry.replyToBody` is truthy. Replies to media-only or empty-text Signal messages therefore still lose quote sender metadata in the prompt context; pass the quote object when sender or quote metadata is visible, not only when body text exists.
  - confidence: 0.86
