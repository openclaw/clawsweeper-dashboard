## Summary

This PR is a focused security/auth-provider hardening change and should stay open for maintainer review rather than cleanup closure. The patch appears correct, with real CLI proof and focused tests, but the fail-closed behavior is compatibility-sensitive because existing configs that explicitly map `models.providers.google.api` to an OpenAI-compatible transport will stop before dispatch.

## What This Changes

Adds a Google provider/API mismatch guard in `registerProviderStreamForModel` plus focused tests for mismatch rejection, env-only Gemini auth, secret redaction, explicit Google API config, and Google Vertex.

## Best Possible Solution

Land this only after provider/auth owners agree that explicit `models.providers.google.api` mismatches are unsupported and that preserving env-only Gemini auth is the desired compatibility behavior.

## Reproduction Assessment

Yes, source-level: current main can construct Google-provider models with an OpenAI-compatible API and `registerProviderStreamForModel` has no mismatch guard before stream dispatch. I did not execute a current-main CLI repro in this read-only review.

## Solution Assessment

Yes, if maintainers accept the narrowed policy: the guard is placed at the provider stream boundary and the tests preserve documented env-only Gemini auth while rejecting explicit mismatches. The safer alternative, if compatibility wins, is to keep legacy behavior by default and make strict fail-closed behavior explicit.
