## Summary

Keep open. Current main still documents the Notion skill with `NOTION_KEY=$(cat ...)` plus multi-line `curl` examples, while the exec approval docs and tests confirm that command substitution, shell expansion/control syntax, and line continuations are allowlist-hostile in strict modes. The two linked fix PRs were closed without merging, so no current-main implementation supersedes the report.

## What This Changes

The issue asks to revise the bundled Notion skill instructions so strict exec allowlist users are not led to shell-wrapped `curl` commands that miss the allowlist.

## Best Possible Solution

Update `skills/notion/SKILL.md` in place: align examples with `NOTION_API_KEY`, remove the shell-local `NOTION_KEY=$(cat ...)` pattern, add a short strict-allowlist note, and document a file-backed helper or direct command shape that does not depend on shell substitution or chaining. Core exec approval semantics should remain unchanged.

## Reproduction Assessment

Yes. The current docs still contain the problematic command shape, and current exec-approval docs/tests show why command substitution, line continuations, and macOS shell control/expansion syntax can fail before any Notion API request is made.

## Solution Assessment

Yes, a focused bundled-skill docs fix is the narrowest maintainable solution. Changing core allowlist behavior would be broader and security-sensitive, while a helper-script example or small skill-owned helper keeps the workaround local to Notion usage.
