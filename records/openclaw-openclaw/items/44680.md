## Summary

Keep open: the checksum direction is useful, but this PR is not mergeable against current main because it bypasses current backup hardening and leaves schema-v2 integrity edge cases source-reproducible.

## What This Changes

The PR adds schema v2 backup manifests with per-asset SHA-256 hashes, checksum verification output/JSON fields, docs, and backup verify coverage.

## Best Possible Solution

Rebase or rebuild the checksum implementation on current main, hash archived bytes while preserving temp-root/filter/retry behavior, fail closed for schema-v2 digests, define empty-directory/link handling, and update focused tests/docs/changelog together.

## Reproduction Assessment

Yes. The blocking PR regressions are source-reproducible by comparing the PR head with current main's backup-create contracts and tests, especially the direct tar.c calls and schema-v2 hasChecksums gate.

## Solution Assessment

No. Per-asset archive checksums are a useful direction, but this branch is not the narrowest maintainable solution until it is rebased onto current backup hardening and the schema-v2 integrity contract fails closed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve backup filters on the first pack:** `src/infra/backup-create.ts:413-423`
  - body: The first `tar.c` pass no longer applies current backup filters, so `state/extensions/*/node_modules` and known volatile state files enter the initial archive and are preserved by the final repack. Current main intentionally skips those paths, and tests lock that behavior.
  - confidence: 0.95
- **[P2] Keep tar writes on the retry wrapper:** `src/infra/backup-create.ts:413-423`
  - body: The new pack path calls `tar.c` directly instead of `writeTarArchiveWithRetry`, dropping the current retry/backoff behavior for live-file EOF races. Backups on running installations can fail again on files current main retries or filters out.
  - confidence: 0.9
- **[P1] Fail schema v2 when all digests are missing:** `src/commands/backup-verify.ts:468-470`
  - body: When every schema-v2 asset has `sha256` stripped, `hasChecksums` is false and the verifier skips checksum validation entirely, returning success without content integrity checks. Schema v2 should reject missing digests even when all of them are missing.
  - confidence: 0.94
- **[P2] Accept empty directory assets during checksum verify:** `src/commands/backup-verify.ts:340-343`
  - body: Create can compute a valid digest for an empty directory asset, but verify drops directory entries and throws when there are no non-directory entries. A fresh backup with an empty workspace or state subdirectory can fail its own checksum verification.
  - confidence: 0.88
- **[P2] Hash link entries consistently:** `src/infra/backup-create.ts:287-290`
  - body: The create-side tree walk only includes `item.isFile()` entries, while verify hashes every non-directory tar entry. Assets containing symlinks or other link entries can mismatch immediately after archive creation.
  - confidence: 0.82
