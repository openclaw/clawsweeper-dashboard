Full review comments:

- **[P1] Preserve the current backup writer filters:** `src/infra/backup-create.ts:413-424`
  - body: The initial archive is now written by a direct `tar.c` call, so it no longer uses current main's volatile-file filter, installed-plugin `node_modules` exclusion, skip counter, or retry wrapper. That can put files current backups intentionally omit into checksum archives and can reintroduce live-write backup failures.
  - confidence: 0.96
- **[P1] Reject v2 manifests when all digests are missing:** `src/commands/backup-verify.ts:468-470`
  - body: `hasChecksums` is false when every schema-v2 asset has its `sha256` field stripped, so verification skips checksum validation and returns success. A schema-v2 archive without required digests should fail closed, not downgrade itself to structural verification.
  - confidence: 0.95
- **[P1] Handle symlink entries consistently:** `src/commands/backup-verify.ts:306-313`
  - body: Create-side directory hashing skips symlinks because it only hashes `Dirent.isFile()` entries, but verify-side hashing includes every non-directory tar entry. A directory containing symlinks can therefore fail verification immediately after `backup create`.
  - confidence: 0.9
- **[P1] Accept checksummed empty directory assets:** `src/commands/backup-verify.ts:340-343`
  - body: A directory asset with no file entries receives the empty directory hash during create, but verify throws `no entries found` after excluding directory entries. An empty external workspace can make a freshly created checksum archive fail verification.
  - confidence: 0.88
- **[P2] Keep malformed manifest errors sanitized:** `src/commands/backup-verify.ts:106`
  - body: Current main intentionally reports invalid manifest JSON without parser internals, but this branch adds `String(err)` to the user-facing error. That regresses the existing hardening and its test coverage.
  - confidence: 0.91
