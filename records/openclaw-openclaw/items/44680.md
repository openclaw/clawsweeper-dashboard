## Summary

Keep open: current main has not implemented backup checksum verification, and this branch is still unsafe to merge because it conflicts with current backup hardening and can skip schema-v2 integrity validation.

## What This Changes

The PR adds schema v2 backup manifests with per-asset SHA-256 hashes, checksum verification output, docs, and backup verify tests.

## Best Possible Solution

Rebase or replace the checksum implementation on current main, hash archived bytes, preserve temp-root/filter/retry behavior, require schema-v2 digests for every asset, define empty-directory/link handling, and update focused tests, docs, and changelog together.

## Reproduction Assessment

Yes. The blocking regressions are source-reproducible by comparing current main's backup-create tests and implementation contracts with the PR head's direct os.tmpdir and tar.c calls, plus the schema-v2 hasChecksums gate.

## Solution Assessment

No. Per-asset archive checksums are a useful direction, but this branch is not the best mergeable implementation until it is rebased onto current backup hardening and fails closed for schema-v2 integrity metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep the temp manifest outside backed-up assets:** `src/infra/backup-create.ts:383`
  - body: The branch creates tempDir and stagingDir directly under os.tmpdir() again. Current main moved this through chooseBackupTempRoot because TMPDIR can be inside the state asset; in that case the recursive asset walk includes the staged manifest and the explicit manifest, producing duplicate root manifest entries in the archive.
  - confidence: 0.96
- **[P1] Preserve backup filters when creating the initial archive:** `src/infra/backup-create.ts:413-423`
  - body: The first tar.c pass no longer applies the current backup filters before extracting and repacking. That re-includes state/extensions/*/node_modules and known volatile state files that current main deliberately skips, and the final archive preserves those entries.
  - confidence: 0.95
- **[P2] Keep tar writes on the retry wrapper:** `src/infra/backup-create.ts:413-423`
  - body: The new pack path calls tar.c directly instead of writeTarArchiveWithRetry. This drops the current retry/backoff behavior for live-file EOF races, so backups on running installations can fail again on files that current main retries or filters out.
  - confidence: 0.9
- **[P1] Fail closed when schema v2 has no digests:** `src/commands/backup-verify.ts:468-470`
  - body: When every schema-v2 asset has sha256 stripped, hasChecksums is false and the verifier skips verifyAssetChecksums entirely, returning success without content validation. Schema v2 should reject missing digests even when all of them are missing.
  - confidence: 0.94
- **[P2] Allow empty directory assets to verify:** `src/commands/backup-verify.ts:340-343`
  - body: Create can compute a valid digest for an empty directory asset, but verify builds assetEntries only from non-directory entries and throws when the list is empty. A fresh backup containing an empty workspace or state subdirectory can fail its own checksum verification.
  - confidence: 0.89
- **[P2] Hash link entries consistently:** `src/infra/backup-create.ts:287-290`
  - body: Checksum creation walks the extracted tree and only includes item.isFile() entries, while verification hashes every non-directory archive entry. Assets containing symlinks or other link entries can therefore omit them during create but include them during verify, causing a fresh archive to mismatch.
  - confidence: 0.82
