## Summary

Keep open; the checksum-integrity direction is useful, but this PR is not merge-ready because it bypasses current backup writer safeguards, can accept schema-v2 archives with stripped checksum metadata, fails fresh archives for some valid directory assets, and still lacks real behavior proof.

## What This Changes

The PR adds schema-v2 backup manifests with per-asset SHA-256 checksums, checksum-aware `backup verify` output/result fields, backup verification tests, and backup CLI docs.

## Best Possible Solution

Rebuild checksum support on current main's backup writer, hash archived payload bytes while preserving filters, retry, temp-root handling, and sanitized errors, require every schema-v2 asset digest, cover symlinks and empty directories, then add redacted real-run proof.

## Reproduction Assessment

Yes for the PR blockers via source inspection: the PR head directly shows the direct `tar.c` writer path, schema-v2 `hasChecksums` skip, symlink/empty-directory mismatch, and malformed JSON error regression. This is not a current-main bug reproduction because checksum integrity is a new feature absent from main.

## Solution Assessment

No. Per-asset checksums are a reasonable direction, but this branch needs to be rebuilt on current main's backup writer and fail closed for schema-v2 checksum metadata before it is the maintainable solution.

Full review comments:

- **[P1] Preserve the current backup writer safeguards:** `src/infra/backup-create.ts:413-424`
  - body: The initial archive is now written by a direct `tar.c` call, so it bypasses current main's volatile-file filter, installed-plugin `node_modules` exclusion, temp-root selection, skip counter, and retry wrapper. That changes archive contents and can reintroduce live-write backup failures.
  - confidence: 0.96
- **[P1] Reject schema-v2 manifests when all digests are missing:** `src/commands/backup-verify.ts:468`
  - body: `hasChecksums` is false when every schema-v2 asset has its `sha256` field stripped, so verification skips checksum validation and returns success. Schema-v2 archives should fail closed when required digest metadata is absent.
  - confidence: 0.95
- **[P1] Hash symlink entries consistently:** `src/infra/backup-create.ts:289-292`
  - body: Create-side directory hashing only includes `Dirent.isFile()` entries, while verify-side hashing includes every non-directory tar entry. A directory containing symlinks can therefore fail verification immediately after `backup create`.
  - confidence: 0.9
- **[P1] Accept checksummed empty directory assets:** `src/commands/backup-verify.ts:340-344`
  - body: A directory asset with no file entries receives the empty directory hash during create, but verify throws `no entries found` after excluding directory entries. An empty workspace can make a freshly created checksum archive fail verification.
  - confidence: 0.88
- **[P2] Keep malformed manifest errors sanitized:** `src/commands/backup-verify.ts:106`
  - body: Current main intentionally reports invalid manifest JSON without parser internals, but this branch adds `String(err)` to the user-facing error. That regresses the existing hardening and its test coverage.
  - confidence: 0.91
