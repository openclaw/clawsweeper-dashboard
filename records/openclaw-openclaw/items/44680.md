## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve current backup writer safeguards:** `src/infra/backup-create.ts:413-423`
  - body: The new two-pass path packs the first archive with a direct `tar.c` call from an older writer path. Current main wraps the tar call in retry handling, filters volatile state, excludes installed plugin dependency trees, chooses a temp root outside every source asset, and records skipped volatile files; dropping those safeguards can archive unwanted live state and make valid backups fail on live-write races.
  - confidence: 0.94
- **[P1] Fail closed when schema v2 has no checksums:** `src/commands/backup-verify.ts:468-475`
  - body: `hasChecksums` is based on `some()`, so a schema-v2 manifest with every `sha256` removed skips `verifyAssetChecksums()` and still returns `ok`. Schema v2 should require a digest for every asset, otherwise a stripped or corrupted manifest can pass as structurally valid without integrity verification.
  - confidence: 0.93
- **[P1] Hash symlink entries consistently:** `src/infra/backup-create.ts:289-292`
  - body: The create-side directory hash only includes `Dirent.isFile()` entries, but the verify path hashes every non-directory tar entry. A directory asset containing a symlink is therefore omitted from the manifest hash during creation and included during verification, causing a freshly created archive to fail checksum verification.
  - confidence: 0.89
- **[P1] Accept empty directory asset checksums:** `src/commands/backup-verify.ts:340-343`
  - body: Create computes a valid hash for an empty directory asset, but verify rejects any checksummed asset with zero non-directory entries. An empty workspace or other empty top-level asset can make `backup create --verify` fail against its own archive.
  - confidence: 0.87
