## Summary

Keep open: current main still performs structural-only backup verification, and this PR is useful but not mergeable because the latest diff has P1 integrity regressions in the checksum path.

## What This Changes

The PR adds schema v2 backup manifests with per-asset SHA-256 digests, checksum verification in `openclaw backup verify`, docs updates, and backup verify tests.

## Best Possible Solution

Land a rebased checksum implementation that hashes archived bytes, preserves existing backup filters, requires schema v2 digests for every asset, defines empty-directory and link-entry behavior, and updates focused tests, docs, and changelog together.

## Reproduction Assessment

Yes. The failures are source-reproducible from the PR head: the first pack pass lacks the current backup filter, `hasChecksums` skips all-missing v2 manifests, directory verification excludes directory entries, and create/verify treat symlink entries differently.

## Solution Assessment

No. Per-asset archived-byte checksums are a reasonable direction, but this branch is not the best implementation until it preserves current backup behavior and closes the schema v2, empty-directory, and link-entry integrity gaps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve the backup dependency filter:** `src/infra/backup-create.ts:413-423`
  - body: The first packing pass no longer applies the current `state/extensions/*/node_modules` filter. Because the final archive is repacked from that staged output, merging this would re-include rebuildable plugin dependency trees that current main deliberately omits and documents as skipped.
  - confidence: 0.95
- **[P1] Reject schema v2 manifests with no checksums:** `src/commands/backup-verify.ts:468-470`
  - body: When every schema v2 asset has its `sha256` stripped, `hasChecksums` is false and the verifier skips checksum validation entirely. A v2 integrity manifest should fail closed when required digests are missing, including the all-missing case.
  - confidence: 0.94
- **[P2] Accept empty checksummed directories:** `src/commands/backup-verify.ts:340-343`
  - body: Create can emit a valid digest for an empty directory asset, but verify filters directory entries out and throws when no non-directory entries remain. A fresh backup containing an empty state or workspace directory can therefore fail its own checksum verification.
  - confidence: 0.9
- **[P2] Hash link entries consistently:** `src/infra/backup-create.ts:287-290`
  - body: Checksum creation walks the extracted tree and only includes `Dirent.isFile()` entries, while verification hashes every non-directory tar entry. Assets containing symlinks or other link entries can mismatch immediately after creation because create omits them but verify includes them.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Needs attention: no new dependencies, CI, scripts, network calls, or secret handling are added, but schema v2 checksum enforcement can be bypassed by stripping all per-asset digests.

Concerns:

- **[medium] Stripped v2 checksum manifests pass verification:** `src/commands/backup-verify.ts:468`
  - body: The verifier only performs checksum validation when at least one asset has `sha256`; a v2 manifest with all digests removed can return success without content hashing, undermining the integrity guarantee.
  - confidence: 0.94
