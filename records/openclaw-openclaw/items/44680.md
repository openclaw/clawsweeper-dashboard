## Summary

Keep open: current main does not implement content checksum verification, and this PR is a useful direction, but the latest branch still has blocking correctness and integrity regressions in the backup create/verify path.

## What This Changes

The PR changes backup create, backup verify, backup docs, and backup verify tests to write schema v2 manifests with per-asset SHA-256 checksums and validate archived content while preserving schema v1 compatibility.

## Best Possible Solution

Land a polished checksum implementation that preserves the existing archive content filters, makes schema v2 checksums mandatory for every asset, and defines consistent checksum semantics for empty directories and link-like archive entries with targeted tests and docs.

## Reproduction Assessment

Yes. The filter regression is visible by comparing current main's filter use and archive test with the PR head tar.c call; the checksum bypass follows directly from a schema v2 manifest whose assets all omit sha256, making hasChecksums false.

## Solution Assessment

No. Per-asset archived-byte checksums are a maintainable direction, but this branch is not the best implementation until it preserves current backup contracts and enforces v2 integrity before returning success.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve the plugin dependency filter:** `src/infra/backup-create.ts:413-423`
  - body: The new two-pass pack path no longer passes buildExtensionsNodeModulesFilter to tar.c, so installed plugin node_modules under state/extensions are archived again. That regresses the documented archive contents and the existing real-archive test that expects those dependency trees to be omitted.
  - confidence: 0.95
- **[P1] Reject schema v2 archives before skipping checksums:** `src/commands/backup-verify.ts:468-470`
  - body: hasChecksums is false when every schema v2 asset digest is stripped, so the verifier never calls verifyAssetChecksums and can return ok with no content validation. Schema v2 should fail when any required asset digest is missing, including the all-missing case.
  - confidence: 0.94
- **[P2] Hash link entries consistently:** `src/infra/backup-create.ts:289-293`
  - body: Create-side directory hashing records only regular files from the extracted archive, while verify-side hashing includes every non-directory tar entry. Archives containing symlinks or tar-emitted hardlinks can therefore fail verification immediately after creation.
  - confidence: 0.88
- **[P2] Accept checksummed empty directory assets:** `src/commands/backup-verify.ts:340-343`
  - body: Create computes a valid empty Merkle hash for an empty directory asset, but verify throws before comparing it because directory entries were filtered out. Empty state or workspace directories can make create --verify fail on a fresh archive.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies, CI, scripts, network calls, or secret handling, but schema v2 checksum enforcement can be bypassed by stripping all per-asset digests.

Concerns:

- **[medium] Stripped v2 checksum manifests pass verification:** `src/commands/backup-verify.ts:468`
  - body: The verifier only calls checksum validation after assets.some(asset.sha256). If every schema v2 sha256 field is removed, content hashing is skipped and the archive can return success, undermining the PR's integrity guarantee.
  - confidence: 0.94
