## Summary

Keep open: current main still only performs structural backup verification, so the PR is not obsolete, but the latest branch still has blocking backup-integrity regressions. Later commits address some earlier review feedback around TOCTOU and repeated archive scans, but the branch regresses the current plugin dependency filter and can still report checksum success for malformed schema v2 archives.

## What This Changes

The PR changes backup create, backup verify, backup docs, and backup verify tests to write schema v2 manifests with per-asset SHA-256 checksums and verify archived content while preserving schema v1 compatibility.

## Best Possible Solution

Land an archived-byte checksum implementation that preserves existing backup archive contents, makes schema v2 checksums mandatory for every asset, defines consistent behavior for empty directories and link-like entries, and updates focused tests, docs, and changelog together.

## Reproduction Assessment

Yes. The filter regression is visible by comparing current main's filter use and archive test contract with the PR head tar.c call; the checksum bypass is reproducible with a schema v2 manifest whose asset sha256 fields are all removed. Empty directory and link-entry failures also follow directly from the create/verify hashing branches.

## Solution Assessment

No. Per-asset archived-byte checksums are a good direction, but this branch is not the best mergeable implementation until it preserves current backup filters and closes the schema v2, empty-directory, and link-entry integrity gaps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve the plugin dependency filter:** `src/infra/backup-create.ts:413-423`
  - body: The new two-pass packing path calls `tar.c` for the initial archive without `buildExtensionsNodeModulesFilter`. Current main filters `state/extensions/*/node_modules` and has a real-archive/docs/changelog contract for that, so merging this branch would re-include rebuildable plugin dependency trees.
  - confidence: 0.95
- **[P1] Reject v2 archives before skipping checksum validation:** `src/commands/backup-verify.ts:468-470`
  - body: `hasChecksums` is false when every schema v2 asset has its `sha256` stripped, so `backupVerifyCommand` skips `verifyAssetChecksums` and returns success without content validation. Schema v2 should fail if any required digest is missing, including all-missing manifests.
  - confidence: 0.94
- **[P2] Hash link entries consistently:** `src/infra/backup-create.ts:287-290`
  - body: Create hashes extracted directory assets by walking only `Dirent.isFile()` entries, while verify hashes every non-directory tar entry. Archives with symlinks or tar-emitted hardlinks can therefore fail verification immediately after creation because links are omitted on create but included on verify.
  - confidence: 0.88
- **[P2] Accept checksummed empty directory assets:** `src/commands/backup-verify.ts:340-343`
  - body: Create can produce a valid empty-directory asset digest, but verify filters directory entries out and throws when no non-directory payload entries remain. An empty state or workspace directory can make a fresh `backup create --verify` fail.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies, CI, scripts, network calls, or secret handling, but schema v2 checksum enforcement can be bypassed by stripping all per-asset digests.

Concerns:

- **[medium] Stripped v2 checksum manifests pass verification:** `src/commands/backup-verify.ts:468`
  - body: The verifier only calls checksum validation after `assets.some(asset.sha256)`. If every schema v2 `sha256` field is removed, content hashing is skipped and the archive can return success, undermining the PR's integrity guarantee.
