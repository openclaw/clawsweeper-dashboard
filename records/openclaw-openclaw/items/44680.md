## Best Possible Solution

Rebuild checksum integrity on top of current main's backup writer contracts, require schema-v2 digest completeness, handle empty directories and symlinks consistently, and land only with redacted real create/verify/tamper proof.

## Reproduction Assessment

Yes for the PR blockers via source inspection: the latest head directly shows the unfiltered tar writer, schema-v2 checksum skip, empty-directory rejection, and symlink hash mismatch. This is not a current-main bug reproduction because checksum integrity is a new feature absent from main.

## Solution Assessment

No. The feature should be rebuilt on current main's backup writer so checksum integrity does not remove existing backup exclusions, retry behavior, temp-root handling, reporting contracts, or verifier invariants.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the current backup writer safeguards:** `src/infra/backup-create.ts:413-424`
  - body: This direct `tar.c` path no longer uses current main's temp-root selection, volatile/dependency filter, retry wrapper, log hook, or `skippedVolatileCount`. A merge would reintroduce duplicate-manifest failures when tmp is inside state and would include files main intentionally skips, so the checksum implementation needs to wrap the current writer path instead of replacing it.
  - confidence: 0.94
- **[P1] Reject checksumless schema-v2 manifests:** `src/commands/backup-verify.ts:468`
  - body: For a schema-v2 manifest where every `sha256` field has been stripped, `hasChecksums` is false and verification never calls the code that rejects missing v2 digests, so a tampered v2 manifest can pass structural checks while reporting `checksumsVerified: false`. Schema v2 should fail if any asset lacks its required digest.
  - confidence: 0.92
- **[P1] Handle empty directory assets during checksum verify:** `src/commands/backup-verify.ts:340-344`
  - body: Create can emit a checksum for an empty directory asset, but verify drops directory entries from `normalizedToRaw` and then throws when no file entries are found. An empty workspace or other empty top-level asset can make a freshly-created backup fail its own verification.
  - confidence: 0.88
- **[P1] Hash symlink entries consistently:** `src/infra/backup-create.ts:289-292`
  - body: The create-side hash walk only includes `Dirent.isFile()` entries, while verify hashes every non-directory tar entry, including symlinks. Directory assets containing symlinks can therefore fail verification immediately after creation; either skip the same entry types on both sides or encode link metadata consistently.
  - confidence: 0.87
