## Summary

Keep open. The checksum feature is not on current main, but this PR is not merge-ready because the branch is conflicting, lacks real behavior proof, and source inspection shows checksum paths that bypass current backup safeguards or can produce false integrity results.

## What This Changes

The PR changes backup creation and verification to emit schema-v2 manifests with SHA-256 asset checksums, verify those checksums, add backup tests, and update backup CLI docs.

## Best Possible Solution

Rebase the checksum implementation onto the current backup writer, hash the final archived payload while preserving filters/temp-root/retry behavior, handle empty directories and symlinks consistently, fail closed for schema-v2 digest metadata, and then add redacted real-run proof.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the PR head's initial tar pass bypasses current backup safeguards, and the verifier can skip all schema-v2 checksum validation when digest fields are stripped. This is not a current-main bug reproduction because checksum integrity is a new feature absent from main.

## Solution Assessment

No. Per-asset backup checksums are a useful direction, but this branch should preserve current backup invariants and fail-closed verifier behavior before it is considered the best maintainable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Preserve backup filters and retry in checksum pack:** `src/infra/backup-create.ts:413-424`
  - body: The first pass writes the payload that is later hashed and repacked, but it calls `tar.c` directly without current main's backup filter or `writeTarArchiveWithRetry`. That lets volatile state files and installed plugin `node_modules` enter checksum archives and loses the existing retry behavior for live-write EOF races.
  - confidence: 0.96
- **[P1] Accept empty directory assets during checksum verify:** `src/commands/backup-verify.ts:340-344`
  - body: A workspace directory can exist with no non-directory entries; create computes the empty directory hash, but verify filters directories out and throws when the asset entry list is empty. `backup create --verify` can fail against its own fresh archive for an empty workspace asset.
  - confidence: 0.9
- **[P1] Hash symlink entries consistently:** `src/infra/backup-create.ts:289-293`
  - body: The create-side directory hash only includes `Dirent.isFile()` entries, so symlinks inside an asset are omitted from the manifest digest. The verifier later hashes every non-directory tar entry, so backups containing common workspace symlinks can fail checksum verification immediately after creation.
  - confidence: 0.9
- **[P1] Fail schema-v2 archives with no digests:** `src/commands/backup-verify.ts:468-476`
  - body: `hasChecksums` is false when a schema-v2 manifest has all `sha256` fields stripped, so checksum validation is skipped and the result is still `ok`. A v2 checksum manifest missing every digest should fail closed rather than downgrade to structural-only verification.
  - confidence: 0.95
- **[P3] Keep malformed manifest errors sanitized:** `src/commands/backup-verify.ts:106`
  - body: This reintroduces the underlying JSON parser message into the public backup error. Current main deliberately reports `Backup manifest is not valid JSON.` and has coverage that avoids leaking parser internals such as positions or raw SyntaxError text.
  - confidence: 0.82
