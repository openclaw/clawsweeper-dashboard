## Summary

Keep this PR open, but it is not mergeable as-is: the checksum feature is useful, yet the branch regresses current backup filtering/retry/temp-root behavior, still has a schema-v2 integrity bypass, and lacks required real behavior proof from the contributor.

## What This Changes

The PR adds schema v2 backup manifests with per-asset SHA-256 checksums, checksum-aware `backup verify` output/JSON fields, tests, and backup documentation updates.

## Best Possible Solution

Rebuild or rebase the checksum implementation on current main, hash archived bytes while preserving the existing temp-root/filter/retry path, fail closed for schema-v2 digests, and define empty-directory/link handling with focused tests and docs.

## Reproduction Assessment

Yes. The blocking regressions are source-reproducible by comparing the PR-head direct `tar.c` and `hasChecksums` paths with current main's filtered retry-backed backup creation contract and tests.

## Solution Assessment

No. Per-asset archive checksums are a reasonable direction, but this branch is not the narrowest maintainable solution until it preserves current backup behavior and fails closed for schema-v2 integrity metadata.

  - body: The initial archive is now the source for the final archive, but this `tar.c` call bypasses current main's temp-root selection and filters.   - confidence: 0.95

  - body: The first pack calls `tar.c` directly instead of `writeTarArchiveWithRetry`, so live-file EOF races that current main retries can fail `backup create` again before checksums are even computed.
  - confidence: 0.9
- **[P1] Fail schema v2 when every digest is missing:** `src/commands/backup-verify.ts:468-470`
  - body: For schema-v2 archives with all `sha256` fields removed, `hasChecksums` is false and the verifier skips checksum validation entirely, returning success without checking content integrity. Schema v2 should reject missing digests even when none remain.
  - confidence: 0.94
- **[P2] Accept empty directory assets during checksum verify:** `src/commands/backup-verify.ts:336-343`
  - body: Create can produce a digest for an empty directory asset, but verify excludes directory entries before collecting asset entries and then throws `no entries found`. An empty workspace asset can fail verification immediately after creation.
  - confidence: 0.86
- **[P2] Hash link entries consistently:** `src/infra/backup-create.ts:287-290`
  - body: The create-side directory hash only includes entries where `Dirent.isFile()` is true after extraction, while verify hashes every non-directory tar entry. Assets containing symlinks or other link entries can mismatch on a fresh archive.
  - confidence: 0.8
