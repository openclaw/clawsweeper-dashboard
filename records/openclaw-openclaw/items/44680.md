## Summary

Keep open: current main still only performs structural backup verification, while this PR is an unmerged feature/security-hardening branch with blocking integrity and current-behavior regression findings.

## What This Changes

The PR changes backup creation, backup verification, docs, and backup verify tests to emit schema v2 manifests with per-asset SHA-256 checksums and verify archived content while preserving schema v1 compatibility.

## Best Possible Solution

Land a rebased checksum implementation that hashes archived bytes, preserves existing backup filters, makes schema v2 digests mandatory for every asset, defines empty-directory/link behavior, and updates focused tests, docs, and changelog together.

## Reproduction Assessment

Yes. The filter regression is source-reproducible by comparing current main's filtered tar creation/test contract with the PR head's unfiltered first pack pass, and the v2 bypass, empty-directory, and link-entry failures follow directly from the changed checksum branches.

## Solution Assessment

No. Per-asset archived-byte checksums are a good direction, but this branch is not the best mergeable implementation until it preserves current backup filters and closes the schema v2, empty-directory, and link-entry integrity gaps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve the backup dependency filter:** `src/infra/backup-create.ts:413-423`
  - body: The first packing pass no longer applies the current state/extensions/*/node_modules filter. Since the final archive is repacked from that staged output, merging this would re-include rebuildable plugin dependency trees that current main deliberately omits and tests/docs promise to skip.
  - confidence: 0.95
- **[P1] Reject schema v2 manifests with no checksums:** `src/commands/backup-verify.ts:468-470`
  - body: When every schema v2 asset has its sha256 stripped, hasChecksums is false and the verifier skips verifyAssetChecksums, returning success without content validation. Schema v2 should fail closed when any required digest is missing, including the all-missing case.
  - confidence: 0.94
- **[P2] Accept empty checksummed directories:** `src/commands/backup-verify.ts:340-343`
  - body: Create can produce a valid digest for an empty directory asset, but verify filters directory entries out and throws when no non-directory entries remain. A fresh backup containing an empty workspace or state subdirectory can fail its own checksum verification.
  - confidence: 0.9
- **[P2] Hash link entries consistently:** `src/infra/backup-create.ts:287-290`
  - body: Checksum creation walks the extracted tree and only includes Dirent.isFile() entries, while verification hashes every non-directory tar entry. Assets containing symlinks can therefore mismatch immediately after creation because links are omitted on create but included on verify.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Security review needs attention: no dependencies, CI, scripts, network calls, or secret handling are added, but schema v2 checksum enforcement can be bypassed by stripping all per-asset digests.

Concerns:

  - body: The verifier only performs checksum validation when at least one asset has sha256; a v2 manifest with all digests removed can return ok without content hashing, undermining the integrity guarantee.
