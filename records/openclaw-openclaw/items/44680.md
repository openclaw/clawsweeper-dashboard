## Summary

Keep open: the checksum feature is useful and not implemented on current main, but this branch should not merge as-is because it is conflicting, missing real behavior proof, and regresses current backup packing and verification contracts.

## What This Changes

This PR adds schema-v2 SHA-256 checksums to backup manifests, checksum verification in `openclaw backup verify`, regression tests, and backup CLI docs.

## Best Possible Solution

Rebuild the checksum implementation on current main so hashes are derived from archived bytes while preserving filtered retry-backed temp-root packing, then fail closed for schema-v2 digest metadata and cover empty-directory/link-entry cases with tests and real CLI proof.

## Reproduction Assessment

Yes for the PR review findings: the blocking regressions are source-reproducible by comparing PR-head direct `tar.c` and `hasChecksums` paths with current main's filtered retry-backed backup creation contract. This is not a current-main bug reproduction because the checksum feature is new and not present on main.

## Solution Assessment

No. Per-asset archive checksums are a reasonable direction, but this branch is not the best implementation until it is rebased around current backup packing safeguards and fails closed for schema-v2 integrity metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve current backup filters during the first pack:** `src/infra/backup-create.ts:413-424`
  - body: The initial archive becomes the source for the final checksummed archive, but this direct `tar.c` call does not pass current main's filter for volatile state files or installed plugin dependency trees. Those entries can be reintroduced into backups even though current main intentionally excludes them.
  - confidence: 0.95
- **[P2] Keep source packing on the retry wrapper:** `src/infra/backup-create.ts:413-424`
  - body: The first pack calls `tar.c` directly instead of current main's `writeTarArchiveWithRetry`, so live-file EOF races that current main retries can fail `backup create` before checksums are computed.
  - confidence: 0.9
- **[P2] Keep the temp manifest outside backup sources:** `src/infra/backup-create.ts:383-384`
  - body: This branch still stages the manifest under `os.tmpdir()` instead of current main's `chooseBackupTempRoot` path. When the system temp directory is inside the state tree, the manifest can be traversed as payload and explicitly included, recreating the duplicate root-manifest failure current main fixed.
  - confidence: 0.88
- **[P1] Fail schema v2 when every digest is missing:** `src/commands/backup-verify.ts:468-470`
  - body: For schema-v2 archives with all `sha256` fields removed, `hasChecksums` is false and the verifier skips checksum validation entirely, returning success without checking content integrity.
  - confidence: 0.94
- **[P2] Accept empty directory assets during checksum verify:** `src/commands/backup-verify.ts:340-343`
  - body: Create can produce a digest for an empty directory asset, but verify excludes directory entries before collecting asset entries and then throws `no entries found`. An empty workspace asset can fail verification immediately after creation.
  - confidence: 0.86
- **[P2] Hash link entries consistently:** `src/infra/backup-create.ts:287-290`
  - body: The create-side directory hash only includes extracted regular files, while verify hashes every non-directory tar entry. Assets containing symlinks or other link entries can mismatch on a fresh archive.
  - confidence: 0.8
