## Summary

## What This Changes

The branch adds a bundled `extensions/oci-genai` provider plugin with OCI profile loading/signing, native client and embeddings support, provider docs, tests, changelog, and lockfile workspace metadata.

## Best Possible Solution

Move this OCI provider to an external ClawHub-published plugin and revisit OpenClaw core only if a missing, generic provider SDK seam blocks that path.

## Reproduction Assessment

Not applicable; this is a feature PR, not a bug report. The scope decision is backed by current docs/source policy, and the observed patch blockers are source-reproducible from the diff plus current provider/plugin loader code.

## Solution Assessment

No; bundling a new optional OCI provider in core is not the best route under the current VISION and plugin docs. The narrower maintainable solution is an external ClawHub provider plugin using the existing provider and memory embedding APIs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire signed fetch into OCI chat:** `extensions/oci-genai/provider-catalog.ts:29-31`
  - body: This catalog goes through `defineSingleProviderPluginEntry` and the shared API-key catalog path, so the `--oci-profile` value becomes `provider.apiKey` for the OpenAI SDK. The proposed `createOciSignedFetch` helper is never connected to chat requests, so OCI chat will not receive the RSA `Signature` Authorization header the PR says it requires.
  - confidence: 0.9
- **[P2] Declare the memory embedding contract:** `extensions/oci-genai/openclaw.plugin.json:6`
  - body: The entry registers `ociMemoryEmbeddingProviderAdapter`, but the manifest has no `contracts.memoryEmbeddingProviders`. Current loading treats that as an error for non-memory plugins and skips registration, so the OCI embedding adapter will not be available.
  - confidence: 0.88
