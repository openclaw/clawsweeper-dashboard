## Summary

Keep open. Current main still has the central Telegram partial-streaming delivery-state shape described here: ambiguous or timeout final preview edit failures can be retained and marked delivered, and the later Telegram fallback path is gated by that delivered snapshot. The linked open reports cover narrower pieces, but this issue adds useful combined macOS evidence across transport stalls, Web UI-only replies, DM silent-reply filler, and session model state.

## What This Changes

The issue reports a combined Telegram macOS regression involving polling stalls, partial-streaming replies visible only in the Web UI, visible silent-reply filler in DMs, and persistent session modelOverride pollution after fallback or Web UI session patches.

## Best Possible Solution

Keep this as the combined incident tracker until the concrete fixes are split or prioritized: Telegram partial preview finalization should distinguish confirmed final delivery from uncertain preview retention, undelivered finals should fall back or surface a delivery failure, and the NO_REPLY plus modelOverride symptoms should be routed to focused owner fixes without losing this incident context.

## Reproduction Assessment

Yes for the central delivery-state class: a focused unit reproduction can make a final Telegram editPreview throw a timeout or ambiguous network error after a preview exists and observe preview-retained plus delivered state suppressing fallback. The full macOS long-polling incident remains unclear without live environment proof.

## Solution Assessment

No, the current behavior is not the best way to solve the issue. Retaining an ambiguous preview may avoid duplicate Telegram messages, but marking it delivered can hide genuine non-delivery; a safer implementation needs an explicit uncertain/retained state that drives fallback, retry, or visible failure policy.
