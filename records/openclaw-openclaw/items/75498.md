## Summary

Keep open. Current main has partial fixes for polling liveness and Telegram DM filler, but the partial-stream final-delivery path can still treat an ambiguous retained preview as delivered and suppress a fresh fallback send, which matches the central Web UI-visible but Telegram-missing symptom.

## What This Changes

The issue reports a macOS Telegram regression where partial streaming, polling stalls, silent-reply fallback, and session model fallback state combined to leave final replies visible in Web UI but missing from Telegram.

## Best Possible Solution

Define a Telegram delivery-state contract where confirmed final preview edits, retained ambiguous edits, fallback sends, and failed sends have distinct outcomes, then route safe retries or operator-visible failures through the Telegram delivery path without regressing duplicate prevention.

## Reproduction Assessment

Yes for the central source-level path: a final Telegram partial-stream update with sendMayHaveLanded but no message id is retained, marked delivered, and prevents the normal fallback send. The full macOS polling incident remains log-backed rather than live-reproduced here.

## Solution Assessment

No, current main is not yet the best solution for the reported combined delivery problem. The polling and silent-DM parts are partially addressed, but ambiguous final preview retention still conflates possible delivery with confirmed Telegram final delivery.
