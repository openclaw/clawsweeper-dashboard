## Summary

Keep open. Current main still has a source-reproducible Telegram partial-stream path where an ambiguous final preview with no message id is marked delivered, which can suppress the normal fallback send; polling liveness and silent-DM filler have partial fixes, but the central Web UI-visible/Telegram-missing symptom is not fully resolved.

## What This Changes

The issue asks for reliable Telegram final delivery during partial streaming and polling stalls, without silent DM filler or unintended long-lived model override state.

## Best Possible Solution

Define confirmed, ambiguous, and failed Telegram final-preview outcomes, then route safe retries or operator-visible undelivered states through the Telegram delivery path while leaving auto model-fallback expiry/reset policy to the model-failover track.

## Reproduction Assessment

Yes for the central source-level path: a final Telegram partial stream with sendMayHaveLanded but no message id is retained and marked delivered, which can suppress fallback delivery. The full macOS polling incident remains log-backed rather than live-reproduced here.

## Solution Assessment

No, current main is not yet the best solution because it still conflates possible final preview delivery with confirmed Telegram delivery. A safer fix needs a maintainer-approved policy for ambiguous final edits and duplicate avoidance.
