## Summary

Keep open. Current main still has a source-reproducible Telegram partial-preview path where ambiguous final edit/network failures are retained and marked delivered, which can suppress the standard fallback send; related silent-DM and model-fallback symptoms are still separate open or policy work.

## What This Changes

The report asks OpenClaw to fix a macOS Telegram incident where partial preview streaming plus polling stalls left final replies visible only in Web UI, avoid silent-DM filler, and prevent fallback model state from becoming persistent session overrides.

## Best Possible Solution

Define Telegram delivery state so only confirmed final edits or readback-confirmed matches count as delivered, land or replace the silent-DM fix, and keep or revise auto model-override persistence as an explicit maintainer policy.

## Reproduction Assessment

Yes for the central delivery-state class: current source and tests show a final edit timeout/recoverable network error is retained and marked delivered, which suppresses fallback. The full macOS transport-stall incident remains log-backed rather than live-run reproduced here.

## Solution Assessment

No, current main is not yet the best solution for the reported user problem. Retaining ambiguous edits may reduce duplicate messages, but treating uncertain delivery as confirmed delivery conflates possible Telegram delivery with confirmed final delivery.
