## Summary

Keep open. Current main still treats ambiguous Telegram final preview edit failures as delivered/retained, while the silent-DM and modelOverride symptoms are either separately open or documented as current model-failover policy.

## What This Changes

The issue asks to fix Telegram partial-streaming delivery loss on macOS where Web UI transcript writes succeed without Telegram delivery, plus avoid silent-reply filler and unintended session modelOverride persistence.

## Best Possible Solution

Split the work into focused fixes: make Telegram preview delivery distinguish confirmed delivery from retained/unknown state, land or close the silent-DM follow-up, and decide whether auto fallback model persistence should remain the intended UX.

## Reproduction Assessment

Yes for the central delivery-state class: the current unit path can make a final preview edit time out after an existing preview and observe retained plus delivered state suppressing fallback. The complete macOS transport-stall incident remains only partially reproducible from the supplied logs.

## Solution Assessment

No, current main is not the best solution for the reported user problem. Retaining uncertain preview edits may avoid duplicates, but marking them delivered conflates uncertainty with confirmed Telegram delivery.
