## Summary

Keep open. Current main still has the central Telegram partial-preview delivery-state problem: ambiguous final preview edit failures can be retained and marked delivered, which suppresses the normal fallback send; the polling, silent-DM, and model-fallback parts are only partially fixed, separately tracked, or policy-sensitive.

## What This Changes

The issue asks for Telegram partial streaming to avoid losing final replies during polling stalls, keep no-visible Telegram DM turns silent, and prevent temporary model fallback from persisting as session overrides.

## Best Possible Solution

Define Telegram delivery state so only confirmed final preview edits count as delivered, keep the silent-DM and polling recovery work in their focused related items, and make auto-fallback model persistence/reset semantics explicit.

## Reproduction Assessment

Yes for the central source-level path: a final preview edit timeout or recoverable network error after an existing preview returns retained, calls markDelivered(), and prevents fallback. The full macOS multi-account polling incident is not live-reproduced here.

## Solution Assessment

No, current main is not the best solution for the reported delivery loss. Retaining ambiguous edits may avoid duplicates, but treating that state as confirmed delivery conflates uncertain Telegram delivery with a landed final reply.
