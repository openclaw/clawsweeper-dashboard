## Summary

Keep open. Current main still matches the central Telegram partial-streaming delivery concern: ambiguous final preview edit failures can be retained and marked delivered, which suppresses the normal fallback send path. The issue also carries useful combined evidence across polling recovery, DM silent-reply chatter, and model override state that is not fully covered by a single merged fix.

## What This Changes

The issue reports a macOS OpenClaw 2026.4.29 incident where Telegram replies appeared only in Web UI during partial streaming and polling stalls, with related DM silent-reply filler and persistent session modelOverride state.

## Best Possible Solution

Keep this open until maintainers split or prioritize the concrete fixes: distinguish confirmed final Telegram delivery from uncertain retained preview state, land or replace the focused DM silent-turn fix, isolate the sessions.patch/fallback modelOverride behavior, and coordinate polling transport work with the existing Telegram stall reports.

## Reproduction Assessment

Yes for the central delivery-state class: a focused unit path can make a final Telegram preview edit throw a recoverable timeout after a preview exists and observe retained-plus-delivered state with no fallback send. The complete macOS polling-stall incident is not fully reproducible from source alone.

## Solution Assessment

No, the current delivery semantics are not the best way to solve the reported user problem. Retaining ambiguous preview edits may prevent duplicates, but treating uncertainty as delivered hides failed Telegram delivery; a safer design would track uncertain retention separately and either confirm, fall back, or surface an undelivered state.
