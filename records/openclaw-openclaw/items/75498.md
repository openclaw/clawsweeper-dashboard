## Summary

Keep open. Current main still has a source-reproducible Telegram partial-stream finalization path that can mark an ambiguous preview as delivered without a durable Telegram send, and recent issue comments report the same user-visible Web UI-only delivery shape on newer builds.

## What This Changes

The issue reports Telegram partial-streaming replies being written to the Web UI transcript without Telegram delivery, alongside polling stalls, silent DM fallback chatter, and persistent session modelOverride pollution.

## Best Possible Solution

Define confirmed, ambiguous, and failed Telegram partial-stream final outcomes, then fall back or surface undelivered state only when safe while keeping polling, silent-reply, and modelOverride policy work linked to their narrower trackers.

## Reproduction Assessment

Yes for the central source-level path: current main can mark a final Telegram partial-stream preview delivered when sendMayHaveLanded is true but no message id exists, and tests assert no durable send in that case. The full macOS polling incident remains log-backed rather than live-reproduced here.

## Solution Assessment

No, current main is not sufficient because it still treats possible final preview delivery as confirmed delivery. The maintainable fix needs an explicit Telegram duplicate-risk policy for ambiguous final previews plus focused regression coverage.
