## Summary

Keep open. Current main still has a source-reproducible Telegram partial-stream path where an ambiguous final preview with no message id is marked delivered, which can suppress normal fallback delivery; polling, silent-DM filler, and model-failover state have partial or separate fixes but do not close the central delivery-state bug.

## What This Changes

The issue reports a macOS Telegram partial-streaming delivery regression where completed replies appear in Web UI/transcript but not Telegram, alongside polling stalls, visible silent-reply filler, and persistent model override state.

## Best Possible Solution

Define confirmed, ambiguous, and failed Telegram final-preview outcomes, then use safe fallback or operator-visible undelivered states for ambiguous finals while keeping polling, silent-reply, and model-failover follow-ups on their narrower tracks.

## Reproduction Assessment

Yes for the central source-level path: a final Telegram partial stream with sendMayHaveLanded but no message id is retained and marked delivered, which can suppress fallback delivery. The full macOS polling incident remains log-backed rather than live-reproduced here.

## Solution Assessment

No, current main is not the best solution because it still conflates possible final preview delivery with confirmed Telegram delivery. A safer fix needs a maintainer-approved policy for ambiguous final edits and duplicate avoidance.
