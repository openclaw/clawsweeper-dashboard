## Summary

Keep open. Current main still matches the reported failure class: Telegram final preview edits with ambiguous network errors can be retained and marked delivered, which suppresses the normal fallback send path, while the related DM silent-reply and model override behaviors are still represented in current code/tests and need maintainer scoping with the related open items.

## What This Changes

The issue reports a combined Telegram/macOS regression where partial streaming can leave Web UI-only final replies during polling stalls, DM silent fallback emits visible filler, and temporary fallback models persist as session model overrides.

## Best Possible Solution

Keep this open until maintainers split or prioritize the concrete fixes: distinguish confirmed Telegram final delivery from uncertain retained previews, make fallback/surfacing behavior explicit when no final Telegram message is confirmed, land or replace the focused DM silent-turn fix, and decide whether auto fallback model selections should remain persistent session state.

## Reproduction Assessment

Yes for the central delivery-state class: a focused unit path can make a final Telegram preview edit time out after a preview exists and observe retained plus delivered state suppressing fallback. The complete macOS polling-stall sequence remains unclear without live transport reproduction.

## Solution Assessment

No, current behavior is not the best way to solve the reported user problem. Retaining ambiguous preview edits may avoid duplicates, but marking that state as delivered conflates uncertainty with confirmed Telegram delivery; the safer design is a separate uncertain/retained state that falls back or reports undelivered when no final message is confirmed.
