## Summary

Keep open. Current main still has the central Telegram delivery-state problem: ambiguous final preview edit network errors are retained and marked delivered, which can suppress the normal fallback send path; related DM silent-reply and auto modelOverride persistence are still present enough to need maintainer scoping rather than cleanup closure.

## What This Changes

The issue requests fixes for Telegram partial-streaming delivery ambiguity during polling stalls, Telegram DM silent no-reply chatter, and fallback/session modelOverride pollution.

## Best Possible Solution

Keep this open until maintainers split or prioritize the remaining fixes: distinguish confirmed Telegram final delivery from uncertain retained previews, make fallback or undelivered reporting explicit when no final Telegram message is confirmed, land or replace the focused DM silent-turn fix, and decide whether auto fallback model choices should remain visible persistent session overrides.

## Reproduction Assessment

Yes for the central delivery-state class: a focused unit path can make a final Telegram preview edit time out after a preview exists and observe retained plus delivered state suppressing fallback. The complete macOS polling-stall sequence is still unclear without a live Telegram transport reproduction.

## Solution Assessment

No, current main is not the best way to solve the central issue. Retaining an ambiguous preview may avoid duplicates, but marking that state as delivered conflates uncertainty with confirmed Telegram delivery; a safer design is a separate uncertain/retained state that either falls back, confirms delivery, or reports undelivered when no final Telegram message is proven.
