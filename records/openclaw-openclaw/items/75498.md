## Summary

Keep open. Current main still encodes the central Telegram partial-preview delivery concern: recoverable or ambiguous final preview edit failures can be retained and marked delivered, which can suppress the normal final-send fallback even though the transcript has final text.

## What This Changes

The report asks OpenClaw to fix a macOS Telegram incident involving polling stalls, partial-streaming replies visible only in Web UI, DM silent-reply chatter, and persistent session model overrides after fallback.

## Best Possible Solution

Keep this open until maintainers split or scope the fixes: separate uncertain Telegram preview retention from confirmed delivery, route no-confirmation finals to fallback or visible undelivered state, land or replace the focused DM silent-turn fix, and clarify auto-fallback model override UX.

## Reproduction Assessment

Yes for the central delivery-state class: the current unit path can mock an ambiguous final preview edit failure or timeout and observe retained plus delivered state without sendPayload. The full macOS polling-stall incident remains only partially reproducible from the supplied logs without a live Telegram transport run.

## Solution Assessment

No, current main is not yet the best solution for the reported user problem. Retaining ambiguous previews avoids duplicate messages, but marking that uncertainty as delivered conflates possible delivery with confirmed final Telegram delivery; a safer design would track uncertain retention separately and fall back or report undelivered when no final message is confirmed.
