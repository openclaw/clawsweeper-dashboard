## Summary

Keep open. Current main still matches the core delivery-state concern: Telegram final preview edit errors that may have landed are retained and marked delivered, and the fallback send path is gated by that delivered snapshot. The related DM silent-reply behavior is still encoded in tests while #70630 remains open, and current model-fallback tests explicitly preserve auto failover overrides across turns until reset, so this needs maintainer scoping rather than cleanup closure.

## What This Changes

The issue requests fixing a combined Telegram regression where partial streaming and polling stalls can leave final replies visible in Web UI but undelivered to Telegram, while fallback model selection persists as session model overrides and DM silent turns produce visible filler text.

## Best Possible Solution

Keep this open until maintainers split or prioritize the concrete fixes: represent uncertain retained Telegram previews separately from confirmed delivery, fall back or surface undelivered state when no final Telegram message is confirmed, land or replace the focused DM silent-turn fix, and clarify whether auto fallback model selections should remain persistent session state or be visible as transient runtime state only.

## Reproduction Assessment

Yes for the central delivery-state class: a focused unit path can make a final Telegram preview edit time out after a preview exists and observe retained plus delivered state suppressing fallback. The complete macOS polling-stall incident remains only partially reproducible from the provided logs without live transport reproduction.

## Solution Assessment

No, current behavior is not the best way to solve the reported user problem. Retaining ambiguous preview edits may avoid duplicates, but marking them as delivered conflates uncertainty with confirmed Telegram delivery; a safer design would track uncertain/retained delivery separately and fallback or report undelivered when no final message is confirmed.
