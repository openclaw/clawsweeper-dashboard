## Summary

Keep open. Current main still has a source-reproducible Telegram partial-preview delivery-state path where ambiguous final edit failures are retained and marked delivered; the silent-DM symptom is separately tracked by an open PR, and model-fallback persistence is documented behavior.

## What This Changes

The issue asks OpenClaw to prevent Telegram Web UI-only replies during polling/partial-streaming stalls, avoid visible silent-DM filler, and avoid persistent session model overrides from temporary fallback state.

## Best Possible Solution

Define Telegram delivery state so only confirmed final preview edits count as delivered, land or replace the silent-DM fix tracked by #70630, and either keep the documented auto-fallback model policy or make an explicit maintainer policy change.

## Reproduction Assessment

Yes for the central delivery-state class: current source and tests show a final Telegram preview edit timeout can be retained and marked delivered, suppressing fallback. The full macOS transport-stall sequence remains log-backed rather than live-reproduced here.

## Solution Assessment

No, current main is not yet the best solution for the reported user-visible delivery problem. Retaining ambiguous edits may avoid duplicates, but marking uncertain delivery as confirmed delivered conflates possible Telegram delivery with verified final delivery.
