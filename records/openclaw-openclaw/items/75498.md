## Summary

Keep open. Current main has several Telegram transport and model-reset improvements after the reported v2026.4.29 release, but the central partial-streaming delivery-state concern still exists: ambiguous final preview edit failures can be retained and marked delivered, which prevents the normal Telegram send fallback even when the transcript has final text.

## What This Changes

The issue reports a combined Telegram regression where macOS v2026.4.29 partial streaming can leave final replies visible only in Web UI, Telegram polling stalls, DM silent turns emit filler text, and temporary fallback models persist as session overrides.

## Best Possible Solution

Keep the issue open and split or prioritize the concrete fixes: represent retained/uncertain Telegram preview edits separately from confirmed delivery, fall back or surface undelivered state when no final Telegram message is confirmed, land or replace the focused DM silent-turn fix, and decide whether auto-fallback model selections should remain visible as persistent session state or move to clearer transient/runtime state.

## Reproduction Assessment

Yes for the central delivery-state class: a focused unit path can make a final Telegram preview edit time out after an existing preview and observe retained plus delivered state suppressing fallback. The complete macOS polling-stall incident remains only partially reproducible from the supplied logs without live Telegram transport reproduction.

## Solution Assessment

No, current main is not yet the best solution for the reported user problem. Retaining ambiguous preview edits may avoid duplicate Telegram messages, but marking that state as delivered conflates uncertainty with confirmed delivery; a safer design would track uncertain retention separately and fall back or report undelivered when no final message is confirmed.
