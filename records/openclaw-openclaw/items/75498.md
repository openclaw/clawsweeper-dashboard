## Summary

Keep open. Current main still has a source-reproducible Telegram partial-preview path where ambiguous final edit failures are retained and marked delivered, while related silent-DM and model-fallback symptoms are either open follow-up work or documented policy.

## What This Changes

Ensure Telegram final replies are reliably delivered under partial streaming and stall recovery, avoid silent-DM filler chatter, and prevent unintended session model override persistence.

## Best Possible Solution

Define Telegram delivery state so only confirmed final preview edits count as delivered, land or replace the silent-DM fix tracked by #70630, and keep model-fallback persistence aligned with the documented reset policy unless maintainers choose a new model policy.

## Reproduction Assessment

Yes for the central delivery-state class: current source and tests show a final Telegram preview edit timeout can be retained and marked delivered, suppressing fallback. The full macOS transport-stall incident remains log-backed rather than live-reproduced here.

## Solution Assessment

No, current main is not yet the best solution for the reported user problem. Retaining ambiguous edits may avoid duplicates, but marking uncertain delivery as delivered conflates possible Telegram delivery with confirmed final delivery.
