## Summary

Keep open. Current main still treats ambiguous Telegram final preview failures as delivered, while the DM silent-turn and auto fallback model override behaviors remain encoded or policy-sensitive rather than fully resolved.

## What This Changes

The issue asks to fix a combined Telegram regression where macOS partial streaming can leave replies visible only in Web UI, DM silent turns emit filler text, and fallback/session patch flows persist unwanted model overrides.

## Best Possible Solution

Split the issue into focused maintainer-approved fixes: distinguish confirmed from uncertain Telegram preview delivery before suppressing fallback, review or land the DM silent-turn fix, and decide the intended UX for auto fallback model persistence.

## Reproduction Assessment

Yes for the central delivery-state class: a unit path can make the final Telegram preview edit time out after a visible preview and observe retained plus delivered state suppressing fallback. The complete macOS polling-stall incident is only partially reproducible from the supplied logs.

## Solution Assessment

No, current main is not the best solution for the reported behavior. Retaining ambiguous preview edits may avoid duplicates, but marking that state as delivered conflates uncertainty with confirmed Telegram delivery.
