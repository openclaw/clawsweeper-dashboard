## Summary

Keep open. The current main code still has the central Telegram partial-streaming delivery-state shape described here: final preview edit failures that are network/ambiguous can be marked delivered and retained, which can suppress the normal fallback send path. The linked reports cover pieces of the incident, but this issue adds useful combined evidence for Web UI-only replies, Telegram stall recovery, silent DM fallback, and session model state.

## What This Changes

The issue requests fixing Telegram partial-streaming and polling-stall behavior that can leave replies visible only in Web UI, while also preventing temporary model fallback/session patch state and silent-reply rewrites from polluting user-visible session behavior.

## Best Possible Solution

Keep this open until maintainers split or prioritize the concrete fixes: make Telegram partial preview finalization distinguish confirmed delivery from ambiguous retention, ensure an undelivered final reply either falls back or is surfaced as failed, and route the modelOverride and NO_REPLY parts to their narrower owner paths without losing this combined incident evidence.

## Reproduction Assessment

Yes for the central delivery-state class: a focused unit reproduction can make Telegram final editPreview throw a timeout or ambiguous error after a preview exists and observe preview-retained plus delivered state. The full macOS polling-stall incident remains live-environment dependent and is not fully reproduced from source alone.

## Solution Assessment

No, the current behavior is not the best solution for the reported user problem. Retaining ambiguous preview edits avoids duplicates, but marking them delivered can hide genuine non-delivery; a safer fix needs a delivery state that separates confirmed final delivery from uncertain retention and triggers fallback or visible failure when no final Telegram message is confirmed.
