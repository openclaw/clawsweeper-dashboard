## Summary

Keep open. Current main still has a source-reproducible Telegram partial-preview path where uncertain final edit failures are marked delivered and retained, which can suppress normal final fallback delivery; the bundled silent-DM and model-fallback symptoms are either separately tracked or documented policy rather than proof that this issue is resolved.

## What This Changes

The issue reports a macOS Telegram regression where partial streaming can leave final replies visible only in the Web UI, alongside polling-stall, silent-DM fallback, and session model-override symptoms.

## Best Possible Solution

Define Telegram delivery state so only confirmed final preview edits count as delivered, preserve an explicit duplicate-risk policy for uncertain edits, and route the silent-DM, polling, and model-fallback symptoms through their canonical follow-ups.

## Reproduction Assessment

Yes for the central delivery-state class: current source and tests show a final Telegram preview edit timeout can be retained and marked delivered, suppressing fallback. The complete macOS incident remains log-backed rather than live-reproduced in this read-only review.

## Solution Assessment

No, current main is not the best solution for the reported user-visible failure. Retaining uncertain edits may avoid duplicates, but marking them delivered conflates possible Telegram delivery with confirmed final delivery.
