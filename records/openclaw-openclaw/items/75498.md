## Summary

Keep open. Current main still has a source-reproducible Telegram partial-preview path where ambiguous final edit failures are retained and counted as delivered, while the issue also bundles separate silent-DM, polling, and model-fallback behaviors that are not fully resolved by one existing fix.

## What This Changes

The issue reports macOS Telegram turns whose final replies appear in the Web UI transcript but do not reach Telegram during partial streaming and polling stalls, plus related silent-DM filler and session model override persistence.

## Best Possible Solution

Define Telegram delivery state so only confirmed final preview edits count as delivered, fall back or log undelivered ambiguous finals explicitly, and keep the silent-DM/model-fallback/polling pieces aligned with their canonical follow-ups.

## Reproduction Assessment

Yes for the central delivery-state class: current source and tests show a final Telegram preview edit timeout can be retained and marked delivered, suppressing fallback. The full macOS transport-stall incident remains log-backed rather than live-reproduced here.

## Solution Assessment

No, current main is not yet the best way to solve the reported user problem. Retaining ambiguous edits may avoid duplicates, but marking uncertain final delivery as delivered conflates possible Telegram delivery with confirmed channel delivery.
