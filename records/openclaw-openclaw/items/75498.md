## Summary

Keep open. Current main still has a source-backed Telegram partial-preview finalization path that marks recoverable or ambiguous final edit failures as delivered, which can suppress the fallback `sendMessage`; related silent-DM, polling, and model-fallback symptoms are only partly handled or tracked separately.

## What This Changes

Fix a Telegram partial-streaming incident where completed agent replies can appear only in Web UI while Telegram misses final delivery, with related polling, silent-reply, and model override state symptoms.

## Best Possible Solution

Define Telegram delivery state so only confirmed final preview edits count as delivered, then handle silent-DM fallback and model-failover policy through their existing narrower tracks.

## Reproduction Assessment

Yes for the central delivery-state class: current source and tests show a final Telegram preview edit timeout can be retained and marked delivered, suppressing fallback. The full macOS transport-stall sequence is not live-reproduced here.

## Solution Assessment

No, current main is not the best solution for the reported user problem. Retaining ambiguous final edits may avoid duplicate Telegram messages, but marking uncertain delivery as delivered conflates possible delivery with confirmed final delivery.
