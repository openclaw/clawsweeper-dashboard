## Summary

Keep open. Current main still has a source-reproducible Telegram partial-stream path where an ambiguous final preview with no message id is marked delivered, and the issue also has fresh 2026-05-18 field evidence; polling and model-fallback pieces are partially fixed or separately tracked, but they do not close the delivery-state bug.

## What This Changes

The issue asks for Telegram partial-streamed replies to either land visibly, fall back to normal delivery, or surface failure without polluting persistent session model overrides or emitting DM silent-reply filler.

## Best Possible Solution

Define confirmed, ambiguous, and failed Telegram final-preview outcomes, mark delivery only on confirmed platform-visible success or a documented retained-preview policy, and keep fallback persistence changes on the separate model policy track.

## Reproduction Assessment

Yes for the central source-level path: a final Telegram stream with no message id and sendMayHaveLanded true is retained and marked delivered, which suppresses normal fallback. The full macOS polling incident was not live-reproduced here.

## Solution Assessment

No, current main is not yet the best solution because it still conflates possible final preview delivery with confirmed Telegram delivery. A safer fix needs a maintainer-approved ambiguous-send policy plus observable delivery outcomes.
