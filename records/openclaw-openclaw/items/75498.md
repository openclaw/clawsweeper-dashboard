## Summary

Keep open. Current main still has a source-reproducible Telegram partial-stream delivery-state path where an ambiguous final preview can be treated as delivered without a confirmed Telegram message, and the fresh comments plus related open issues show the user-visible no-sendMessage problem is still active rather than cleaned up.

## What This Changes

The issue asks OpenClaw to stop Telegram partial-stream turns from completing only in the transcript/Web UI without Telegram delivery, and to avoid fallback-driven session modelOverride pollution.

## Best Possible Solution

Define confirmed, ambiguous, and failed Telegram final-preview outcomes, then make ambiguous/no-id finals fall back visibly or surface as undelivered while keeping polling and model-fallback policy work on their narrower trackers.

## Reproduction Assessment

Yes for the central source-level path: current main still has final partial-stream branches that can treat a no-message-id preview as delivered or leave a DM without fallback. The full macOS polling incident was not live-reproduced here.

## Solution Assessment

No, current main is not the best solution because it still conflates possible preview delivery with confirmed Telegram delivery. A safer fix needs an explicit maintainer policy for ambiguous Telegram sends and duplicate avoidance.
