## Summary

Keep open. Current main still has the central Telegram partial-streaming delivery-state shape reported here: ambiguous final preview edit failures can be retained and counted as delivered, so the normal fallback send path can be suppressed. The related open issues cover pieces of the incident, but this report adds useful combined evidence tying delivery, polling recovery, silent DM fallback, and model state together.

## What This Changes

The issue reports a macOS Telegram regression where partial streaming can leave final replies visible only in the Web UI, while related polling stalls, silent DM fallback text, and persistent session modelOverride pollution occur in the same incident window.

## Best Possible Solution

Keep this open until maintainers split or prioritize the concrete fixes: make Telegram partial preview finalization distinguish confirmed final delivery from uncertain retained previews, surface or retry undelivered finals without treating Web UI transcript writes as Telegram delivery, and route the silent DM and modelOverride symptoms to their narrower owner paths without losing this combined incident evidence.

## Reproduction Assessment

Yes for the central delivery-state class: a focused unit reproduction can make a Telegram final preview edit throw a timeout or ambiguous network error after a preview exists and observe retained plus delivered state. The full macOS polling-stall incident remains environment- and timing-dependent, so source inspection does not fully reproduce that live sequence.

## Solution Assessment

No, the current retained-and-delivered behavior is not the best way to solve the reported user problem. Retaining uncertain previews reduces duplicate risk, but confirmed delivery and uncertain retention need separate state so fallback or visible failure can still occur when no final Telegram message is proven delivered.
