## Summary

Keep open. Current `main` still preserves the central Telegram delivery-state concern: recoverable final preview edit failures after an existing preview are retained and marked delivered, which can suppress a normal Telegram `sendMessage` fallback even when the Web UI transcript has the final assistant text. The report also bundles still-open related work for Telegram polling stalls, DM silent-reply chatter, and model override policy, so there is no strong cleanup close reason.

## What This Changes

The issue reports a macOS v2026.4.29 Telegram regression combining polling stalls, partial-streaming final reply non-delivery despite Web UI transcript success, visible DM silent-reply filler, and persistent session-level model override pollution after model fallback.

## Best Possible Solution

Keep the issue open until maintainers split or prioritize the concrete fixes: represent uncertain Telegram preview retention separately from confirmed delivery, fall back or surface undelivered state when no final Telegram message is confirmed, land or replace the focused DM silent-turn fix, and clarify whether auto fallback model selections should remain persistent session state or be visible as transient runtime state only.

## Reproduction Assessment

Yes for the central delivery-state class: a focused unit path can make a final Telegram preview edit time out after a preview exists and observe retained plus delivered state suppressing fallback. The complete macOS polling-stall incident remains only partially reproducible from the provided logs without live transport reproduction.

## Solution Assessment

No, current behavior is not the best way to solve the reported user problem. Retaining ambiguous preview edits may avoid duplicates, but marking them as delivered conflates uncertainty with confirmed Telegram delivery; a safer design would track uncertain/retained delivery separately and fallback or report undelivered when no final message is confirmed.
