## Summary

Keep open. Current main still has a Telegram partial-preview path where ambiguous final edit failures are retained and marked delivered, while the polling recovery, DM silent-reply, and auto model-override symptoms are only partially fixed, separately tracked, or current policy.

## What This Changes

The issue requests reliable Telegram final delivery during partial streaming and polling stalls, no Web UI delivery conflation, no DM silent-reply chatter, and no unintended persistent session model overrides.

## Best Possible Solution

Define Telegram delivery state so only confirmed final preview edits count as delivered, then split the remaining silent-turn, polling recovery, and failover persistence decisions into focused follow-ups.

## Reproduction Assessment

Yes for the central delivery-state class: a focused unit path can make a final Telegram preview edit time out after an existing preview and observe retained plus delivered state suppressing fallback. The full macOS polling-stall incident remains partially reproducible from supplied logs only.

## Solution Assessment

No, current main is not yet the best solution for the reported user problem. Retaining ambiguous edits may prevent duplicates, but marking that state as delivered conflates uncertain Telegram delivery with confirmed final delivery.
