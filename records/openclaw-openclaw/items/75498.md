## Summary

Keep open. Current main still has the central Telegram partial-streaming delivery-state shape: a final preview edit timeout or ambiguous network error can be retained and marked delivered, and that delivered snapshot can suppress the normal fallback send path. Some adjacent transport/model pieces have mitigation code, but the combined user-visible failure is not fully resolved or superseded.

## What This Changes

The issue asks OpenClaw to fix a combined Telegram regression where partial streaming can lose final Telegram delivery during polling stalls, no-visible DM turns can emit filler text, and temporary fallback model state can persist as session overrides.

## Best Possible Solution

Keep this open until maintainers split or prioritize the concrete fixes: make Telegram preview finalization distinguish confirmed final delivery from uncertain preview retention, route no-visible DM chatter through #70630 or an equivalent fix, validate the sessions.patch/fallback modelOverride path separately, and keep polling-stall transport work coordinated with #68494 and #75377.

## Reproduction Assessment

Yes for the central delivery-state class: a focused unit reproduction can make a Telegram final preview edit throw a timeout or ambiguous network error after a preview exists and observe preview-retained plus delivered state. The complete macOS polling-stall incident remains live-environment dependent and is not fully reproducible from source alone.

## Solution Assessment

No, the current behavior is not the best way to solve the reported user problem. Retaining ambiguous preview edits may avoid duplicate messages, but marking them delivered conflates uncertainty with confirmed Telegram delivery; the safer design is to record uncertain retention separately and then fall back or surface failure when no final Telegram message is confirmed.
