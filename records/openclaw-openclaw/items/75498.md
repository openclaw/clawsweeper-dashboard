## Summary

Keep open. Current main still preserves the central Telegram delivery-state concern: ambiguous final preview edit failures can be retained and marked delivered, which suppresses the normal Telegram send fallback even when the transcript has final assistant text.

## What This Changes

The issue reports a combined Telegram regression where macOS v2026.4.29 partial streaming can leave final replies visible only in Web UI, polling stalls, DM silent turns emit filler text, and fallback models persist as session overrides.

## Best Possible Solution

Keep this open until maintainers split or prioritize the concrete fixes: distinguish uncertain retained Telegram previews from confirmed delivery, fall back or surface undelivered state when no final Telegram message is confirmed, land or replace the focused DM silent-turn fix, and clarify the intended UX for auto fallback model state.

## Reproduction Assessment

Yes for the central delivery-state class: a focused unit path can make a final Telegram preview edit time out after an existing preview and observe retained plus delivered state suppressing fallback. The full macOS polling-stall incident remains only partially reproducible from the supplied logs without live Telegram transport reproduction.

## Solution Assessment

No, current main is not yet the best solution for the reported user problem. Retaining ambiguous preview edits may avoid duplicate Telegram messages, but marking that state as delivered conflates uncertainty with confirmed delivery; a safer design would track uncertain retention separately and fall back or report undelivered when no final message is confirmed.
