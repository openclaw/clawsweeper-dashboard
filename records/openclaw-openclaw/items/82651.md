## Summary

This is a valid current-main boundary bug in the session usage time-series helper. `maxPoints: 0` reaches the downsampling branch and produces one aggregate bucket; related invalid values are also not handled before the loop.

## What This Changes

The issue requests that `loadSessionUsageTimeSeries` return an empty `points` array for zero, negative, and non-finite `maxPoints` values while preserving positive downsampling.

## Best Possible Solution

Add a small guard before downsampling so non-positive or non-finite caps return no points, then cover those boundary values in the existing session-cost usage tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: `maxPoints: 0` computes an infinite step and the loop emits one bucket. This was not promoted to a full test reproduction because the review was constrained to read-only inspection.

## Solution Assessment

Yes. The narrow fix is to validate `maxPoints` before the existing downsampling branch and keep the current positive-cap aggregation behavior intact.
