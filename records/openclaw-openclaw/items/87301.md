## Summary

Keep open: the contributor added useful after-fix gateway log proof, but the branch still has a correctness blocker because it can freeze `unknown` when the early hostname lookup is empty.

## What This Changes

The PR adds a logger hostname initializer, imports it before tslog entry points, and replaces the file-log `HOSTNAME` constant with a cached lazy helper.

## Best Possible Solution

Use one recoverable hostname path for both `_meta.hostname` and the top-level file-log field, cache only non-empty hostnames, and add regression coverage for an empty-first then valid lookup.

## Reproduction Assessment

Yes, source-level: current main caches the top-level hostname at module load, and pinned tslog builds `_meta.hostname` from early environment metadata. I did not live-reproduce the intermittent macOS timing, but the provided after-fix log output supports the real user path.

## Solution Assessment

No. The direction is right, but this implementation still relies on one early hostname sample for tslog metadata and still caches `unknown` in the lazy helper.

- **[P2] Do not seed tslog from a single transient lookup:** `src/logging/logger-hostname-init.ts:8`
  - body: The new initializer still calls `os.hostname()` during module evaluation, which is the timing window described by the report.   - confidence: 0.86
- **[P2] Do not cache the fallback hostname:** `src/logging/logger.ts:105`
  - body: `cachedHostname = h || "unknown"` still makes the first empty hostname result permanent for the top-level file-log field. Cache only non-empty hostnames and return `unknown` transiently until a valid hostname is available.
  - confidence: 0.9
