## Summary

Keep open: the patch is aimed at the right logger bug, but it still permanently caches `unknown` if the first lazy lookup sees an empty hostname, and the external PR has no after-fix real macOS log proof.

## What This Changes

The PR replaces the module-level `HOSTNAME` constant in `src/logging/logger.ts` with a lazily initialized `getHostname()` helper for structured file-log records.

## Best Possible Solution

Cache only non-empty hostnames or retry resolution until a valid hostname appears, add focused regression coverage for the transient-empty case, and attach redacted after-fix macOS log proof.

## Reproduction Assessment

Yes, at source level: if `os.hostname()` is empty when current main initializes the module or when this PR first calls the lazy helper, `unknown` is cached for later file-log records. I did not live-reproduce the intermittent macOS timing in this Linux read-only review.

## Solution Assessment

No. Lazy resolution is the right direction, but caching `h || "unknown"` preserves the same poisoned-cache failure if the first lazy lookup is empty; cache only non-empty values or retry until a valid value appears.
