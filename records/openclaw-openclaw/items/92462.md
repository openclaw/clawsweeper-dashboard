## Summary

Current main already covers much of the stated baseline—Memory Wiki has a Control UI view, CLI/RPC ingestion and compilation, provenance-aware browsing, and unified corpus search—but the requested richer uploads and external-provider UI contract remain meaningful product/API work. The issue should stay open for scoping rather than close as implemented or ClawHub-only, because external plugins can contribute retrieval today but do not appear to have a documented generic Control UI/document-lifecycle contract.

## What This Changes

Requests richer uploads and visual management for the bundled Memory Wiki plus a standardized way for external knowledge-base plugins to participate in retrieval and Control UI.

## Best Possible Solution

Split the roadmap into bounded Memory Wiki UI improvements and a separate design for the smallest generic plugin metadata/document-lifecycle seam that is not already covered by corpus supplements, plugin tools, and gateway methods; keep external service implementations plugin-owned.

## Reproduction Assessment

Not applicable: this is a feature request, and source inspection shows several claimed baseline gaps are already implemented while richer upload and external UI capabilities are genuinely new.

## Solution Assessment

No, not as the single broad `knowledgeBaseProvider` interface proposed; the maintainable path is to reuse additive corpus supplements and existing wiki RPCs, then add only narrowly proven document-management or UI extension contracts.
