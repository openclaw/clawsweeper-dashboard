## Summary

The central fix remains necessary: current main still treats the curated marketplace appearing as sufficient inventory readiness, while activation immediately reports the selected plugin missing when its entry is absent. The PR has convincing before/after proof and no correctness or security defect, but its May branch should be refreshed against current main before landing so the focused condition and regression test are reviewed atop the newer Codex migration lifecycle.

## What This Changes

The PR makes Codex migration poll target `plugin/list` until the selected curated plugin entry appears, reuses the inventory matcher, adds delayed-entry regression coverage, and adds a changelog entry.

## Best Possible Solution

Apply the selected-plugin readiness condition and its deterministic regression test to current main while preserving the newer auth, shared-client, timeout, and migration lifecycle code, and leave changelog generation to the release process.

## Reproduction Assessment

Yes. A maintainer reproduced the old early-return behavior on current main with the new regression test, and source inspection confirms the same marketplace-only stop condition remains.

## Solution Assessment

Yes for the core fix: the selected plugin entry is the activation prerequisite, and the existing canonical matcher provides the narrowest consistent readiness check. The safest integration is a current-main refresh rather than landing the stale branch context unchanged.
