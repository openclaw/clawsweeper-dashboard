## Summary

Keep open: the PR targets a real Codex migration race, but the patch adds a blocking matcher regression for namespaced curated plugin IDs and the external real-behavior proof does not exercise the changed apply-time path.

## What This Changes

The PR changes Codex migration apply-time plugin/list polling to wait for the selected curated plugin entry before installing it, and adds a provider test for a delayed target plugin entry.

## Best Possible Solution

Keep the PR open, reuse the existing curated plugin identity matching behavior for the wait predicate, add a delayed namespaced-id regression test, then provide redacted real apply-time proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main stops polling once `openai-curated` exists, then reports `plugin_missing` when the selected plugin entry is absent from that transient response.

## Solution Assessment

No. Waiting in the target plugin-list wrapper is the right area, but the proposed predicate must match the same plugin identities as the existing inventory resolver rather than only `plugin.name`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Match delayed plugins by id as well as name:** `extensions/codex/src/migration/apply.ts:340`
  - body: The new readiness check only compares `plugin.name` with the selected `pluginName`. Reuse the existing identity matching behavior and cover the namespaced-id case in the delayed-entry test.
