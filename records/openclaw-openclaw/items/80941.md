## Summary

Keep open: the linked bug is real, but this PR changes the wrong contract by making auto-managed config metadata user-writable, and the external contribution still lacks real after-fix behavior proof.

## What This Changes

The PR makes `stampConfigVersion` skip stamping `meta.lastTouchedVersion` and `meta.lastTouchedAt` when those paths were explicitly set, and adds write-config regression tests for that behavior.

## Best Possible Solution

Keep the metadata stamp system-managed and fix the false-success path at the `config set` boundary by rejecting or warning for `meta.lastTouchedVersion` and `meta.lastTouchedAt`, with CLI-level regression coverage.

## Reproduction Assessment

Yes. The current-main source path shows `config set` can report success while `writeConfigFile` stamps over the requested `meta.*` value; I did not run the CLI because this review is read-only.

## Solution Assessment

No. The PR fixes the symptom by persisting user metadata, but the narrower maintainable fix is to preserve the stamp contract and stop the CLI from reporting a successful user write to auto-managed fields.

## Review Findings
