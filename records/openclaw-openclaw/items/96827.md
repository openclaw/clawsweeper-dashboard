## Summary

Keep open: current main and v2026.6.10 still record delivered message_tool_only source replies without returning a terminate hint, and the open related PR is only a candidate fix with maintainer/proof blockers.

## What This Changes

The issue reports a regression where message_tool_only embedded runs can continue after a successful source reply and produce repeated self-replies within one run.

## Best Possible Solution

Land a targeted source-reply lifecycle guard that prevents repeated implicit current-source message(send) deliveries in one message_tool_only run while preserving follow-up non-message tool execution.

## Reproduction Assessment

Yes at source level: current main and v2026.6.10 record delivered message_tool_only source replies without returning a terminate hint, and the agent loop only stops a batch when every finalized result has terminate: true. I did not run live Telegram in this read-only pass.

## Solution Assessment

No for the reporter's one-line workaround as a final fix: restoring unconditional terminate would conflict with the Slack continuation behavior fixed by https://github.com/openclaw/openclaw/pull/92343. A narrower repeated-source-reply guard is the maintainable direction.
