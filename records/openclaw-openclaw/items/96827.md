## Summary

Keep open: current main and v2026.6.10 still record a delivered `message_tool_only` source reply without adding a termination hint, and the linked candidate fix remains open pending proof and maintainer acceptance of the delivery invariant.

## What This Changes

The issue asks OpenClaw to stop a `message_tool_only` embedded run from repeatedly sending visible source replies after a delivered `message(send)`.

## Best Possible Solution

Land a targeted guard that prevents repeated implicit current-source `message(send)` deliveries in one `message_tool_only` run while preserving first-send follow-up tool execution, with Telegram proof and Slack regression coverage.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 return no termination hint from the message-tool-only hook, and the core loop continues unless every finalized tool result terminates. I did not rerun live Telegram during this read-only review.

## Solution Assessment

Unclear as proposed: restoring unconditional `terminate: true` would match the workaround but likely undo the Slack continuation fix. The best path is a narrower repeated-delivery guard or lifecycle decision that preserves follow-up non-message tools.
