## Summary

Keep open: current main and v2026.6.10 still record delivered `message_tool_only` source replies without returning a termination hint, so the reported repeated-send failure remains plausible from source. The issue is now the canonical tracker for the behavior, with an open candidate PR that still needs maintainer review and transport proof before this can close.

## What This Changes

The issue asks for `message_tool_only` source replies to avoid repeated self-reply cascades after a successful `message(send)` delivery.

## Best Possible Solution

Review or replace the open candidate fix with a lifecycle guard that prevents repeated implicit current-source sends while preserving follow-up non-message tools, then prove it on Telegram and the Slack continuation path.

## Reproduction Assessment

Yes, at source level: current main records delivered `message_tool_only` source replies without returning `terminate`, and the loop continues unless every finalized result terminates. I did not rerun the live Telegram cascade in this read-only review.

## Solution Assessment

No, the issue's one-line restore is not the best final fix because it conflicts with the Slack continuation fix in https://github.com/openclaw/openclaw/pull/92343. The best path is a narrower repeated-source-send invariant with live Telegram and Slack continuation proof.
