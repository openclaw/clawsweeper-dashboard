## Summary

Keep open: current main and the latest release still record a delivered `message_tool_only` source reply without terminating the loop, and the open candidate PR has not landed or settled the one-reply versus two-reply invariant.

## What This Changes

The issue reports a `message_tool_only` regression where a delivered source reply can lead to repeated self-replies within one embedded agent run.

## Best Possible Solution

Define and prove the intended `message_tool_only` invariant: prevent repeated implicit current-source sends while preserving first-send follow-up tool execution, then land or replace the open candidate PR with live Telegram and Slack continuation proof.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.10` still return no termination hint from the message-tool-only hook, and agent-core continues when the finalized tool batch is not terminal. I did not rerun the live Telegram cascade in this read-only review.

## Solution Assessment

No, not as the one-line restore in the issue body: that would conflict with the Slack continuation fix from https://github.com/openclaw/openclaw/pull/92343. The best fix is a targeted lifecycle guard, or explicit maintainer acceptance of the two-message cap proposed in https://github.com/openclaw/openclaw/pull/96887.
