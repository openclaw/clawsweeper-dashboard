## Label Justifications

- `P1`: The issue describes a recent user-visible channel regression that can send repeated Telegram group messages and waste model turns in one agent run.
- `impact:message-loss`: The affected behavior duplicates visible channel messages from repeated `message(send)` deliveries in `message_tool_only` mode.

## Summary

Keep open: current main and the latest release still return no termination hint after a delivered `message_tool_only` source reply, while the open candidate fix remains unmerged and under maintainer/proof review.

## What This Changes

The issue asks OpenClaw to stop cascading self-replies after a delivered `message_tool_only` source reply without regressing first-send follow-up tool execution.

## Best Possible Solution

Land a targeted lifecycle guard that prevents repeated implicit current-source `message(send)` deliveries in one `message_tool_only` run while preserving first-send follow-up tools and duplicate-presentation suppression.

## Reproduction Assessment

Yes, at source level: current main records delivered `message_tool_only` source replies without returning a termination hint, and the core loop continues when finalized tool results do not terminate. I did not rerun the live Telegram group reproduction during this read-only review.

## Solution Assessment

No, the one-line unconditional terminate restore is too broad because it would undo the Slack continuation behavior from https://github.com/openclaw/openclaw/pull/92343. The best fix needs a narrower repeated-source-send guard or explicit maintainer acceptance of a bounded duplicate cap.
