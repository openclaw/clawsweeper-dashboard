## Label Justifications

- `P1`: The issue describes a recent user-visible channel regression that can send repeated Telegram group messages within one embedded run.

## Summary

Keep open: current `main` and the latest `v2026.6.10` release still record delivered `message_tool_only` source replies without adding a terminate hint, and the open candidate fix at https://github.com/openclaw/openclaw/pull/96887 has not landed yet.

## What This Changes

The issue requests a bounded termination rule for delivered `message_tool_only` source replies so an embedded run does not repeatedly answer its own visible messages.

## Best Possible Solution

Land or replace the open candidate fix with a targeted lifecycle guard that prevents repeated implicit current-source `message(send)` deliveries while preserving legitimate follow-up non-message tools.

## Reproduction Assessment

Yes, at source level: current `main` still records delivered source replies without returning `terminate`, and agent-core continues when finalized tool results do not terminate. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No for the issue's one-line restore by itself: it would stop the cascade but risks undoing the Slack continuation fix from https://github.com/openclaw/openclaw/pull/92343. The better solution is a narrower guard that handles repeated source sends without blocking valid follow-up tools.
