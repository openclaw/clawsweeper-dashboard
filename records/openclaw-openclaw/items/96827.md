## Summary

Keep open: current `main` and the latest `v2026.6.10` release still record a delivered `message_tool_only` source reply without terminating the tool batch, and the core loop is explicitly tested to continue unless `terminate: true` is returned. The report is credible, but the proposed one-line restore would conflict with the merged Slack continuation fix, so maintainers should choose the targeted invariant before repair.

## What This Changes

The issue reports that `message_tool_only` runs can keep sending repeated source replies in one run instead of stopping after the first delivered visible reply.

## Best Possible Solution

Add a targeted source-reply lifecycle guard that prevents repeated implicit current-source `message(send)` deliveries in one `message_tool_only` run while preserving follow-up non-message tool execution and duplicate-presentation suppression.

## Reproduction Assessment

Yes, at source level: current `main` still returns no termination hint from the message-tool-only hook, and the core loop continues when finalized tool results do not terminate. A live Telegram reproduction was not rerun during this read-only review.

## Solution Assessment

No. Restoring unconditional `terminate: true` matches the reporter's workaround but would undo the Slack follow-up-tool behavior fixed by https://github.com/openclaw/openclaw/pull/92343; the better fix needs a narrower repeated-source-send guard.
