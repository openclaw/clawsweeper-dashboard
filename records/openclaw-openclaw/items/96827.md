## Summary

Keep open: current `main` and the latest `v2026.6.10` release still record delivered `message_tool_only` source replies without adding a termination hint, so the reported repeated visible-send cascade remains source-reproducible. The open candidate fix at https://github.com/openclaw/openclaw/pull/96887 should stay paired with this issue, but it still needs maintainer agreement on the delivery invariant and transport proof.

## What This Changes

The issue reports that `message_tool_only` source replies can keep the embedded agent loop running and generate repeated self-replies within one run.

## Best Possible Solution

Implement or land a targeted `message_tool_only` lifecycle guard that prevents repeated implicit current-source sends before another visible delivery while preserving follow-up non-message tool execution, with Telegram and Slack proof.

## Reproduction Assessment

Yes, at source level: current `main` still records delivered `message_tool_only` source replies without returning `terminate`, and the agent loop continues unless finalized results terminate. I did not rerun the live Telegram cascade in this read-only review.

## Solution Assessment

No for the one-line restore described as a workaround, because it would likely undo the Slack follow-up-tool continuation fixed by https://github.com/openclaw/openclaw/pull/92343. The better solution is a narrower repeated-source-send guard with explicit transport proof.
