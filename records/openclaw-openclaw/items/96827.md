## Summary

Keep open: current main and the latest release still record a delivered `message_tool_only` source reply without returning a termination hint, while the linked open fix candidate still needs maintainer invariant review and stronger live proof before it can close this report.

## What This Changes

The issue reports that `message_tool_only` embedded agent runs can keep generating and delivering repeated source replies after the first successful `message(action=send)`.

## Best Possible Solution

Preserve same-turn follow-up tool execution while adding a targeted guard that prevents repeated implicit current-source `message(send)` deliveries, then prove the result in Telegram and the Slack continuation path.

## Reproduction Assessment

Yes, at source level: current main still records delivered `message_tool_only` source replies without returning `terminate`, and the loop continues unless finalized tool results terminate. I did not rerun a live Telegram cascade in this read-only review.

## Solution Assessment

No, the reporter's one-line unconditional restore is not the best fix because it conflicts with the Slack continuation repair in https://github.com/openclaw/openclaw/pull/92343. The best path is a narrower lifecycle invariant that blocks repeated source sends while preserving follow-up tools.
