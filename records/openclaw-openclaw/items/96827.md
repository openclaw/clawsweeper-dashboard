## Summary

Keep open: current main and v2026.6.10 still record a delivered `message_tool_only` source reply without adding the termination hint needed to stop the loop, and the only targeted fix path is still an open PR with unresolved proof/design concerns.

## What This Changes

The issue reports a regression where `message_tool_only` source replies can cascade into repeated visible self-replies within one embedded agent run.

## Best Possible Solution

Land a targeted lifecycle guard that prevents repeated implicit current-source sends in `message_tool_only` runs while preserving first-reply follow-up tools and duplicate-presentation suppression.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 return no termination hint after delivered source replies, while the core loop requires `terminate: true` to stop. I did not run a live Telegram reproduction during this read-only review.

## Solution Assessment

No for the reporter's one-line restore because it conflicts with the Slack continuation fix; the better solution is a narrower repeated-source-send invariant with Telegram and Slack continuation proof.
