## Label Justifications

- `merge-risk: 🚨 auth-provider`: The PR changes how cron-originated provider failures affect shared auth-profile health and routing state.

## Summary

Keep open. The PR is a narrow, correct-looking bug fix: current main lets embedded cron runs inherit the shared auth-profile failure policy, while the PR sets cron failures to local and adds focused call-site coverage; the remaining decision is the intended auth-profile cooldown policy for scheduled runs.

## What This Changes

The PR adds `authProfileFailurePolicy: "local"` to embedded cron agent runs and extends the cron auth-profile propagation test to assert that field is forwarded.

## Best Possible Solution

Land the cron-local failure policy if maintainers agree scheduled automation should report failures at the cron job/run level instead of mutating shared interactive-session auth-profile cooldowns.

## Reproduction Assessment

Yes from source inspection: current main omits `authProfileFailurePolicy` in the embedded cron call, the resolver treats missing policy as shared, and shared `overloaded` failures can become profile-wide cooldowns. I did not run a live overloaded provider scenario in this read-only review.

## Solution Assessment

Yes, this is the narrowest maintainable fix: setting the policy at the embedded cron call site preserves shared auth-profile health for interactive runs, while changing the resolver or cooldown writer globally would affect unrelated callers.
