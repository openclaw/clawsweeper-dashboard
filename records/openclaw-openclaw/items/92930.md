## Summary

Keep open: the PR targets a real status context-window bug, but the patch still warms the cache asynchronously and can leave the triggering cold status render on the 200K fallback; the new screenshots show status output after 16-17 minutes of gateway uptime, not the immediate cold first-call path.

## What This Changes

The PR removes status async-load guards, delegates status-summary context resolution to the shared resolver, starts gateway startup context-window cache warming, and adds cache-warming regression tests.

## Best Possible Solution

Make status compute the corrected context window for the response being rendered from safe read-only metadata, or await a bounded read-only warmup before formatting while explicitly preserving or approving status read-only semantics.

## Reproduction Assessment

Yes by source inspection. The PR-head test asserts the first status-style call still returns 200_000 before async warmup is flushed, so the cold first render remains a concrete source-reproducible path.

## Solution Assessment

No. Delegating to the shared resolver is useful, but relying on background warmup is not the best fix for a status response that needs the corrected value in the same render.
