## Summary

Keep open: current main still has the reported status/context-window gap, but this PR does not deterministically fix the first cold status read because it starts cache warming in the background and returns immediately.

## What This Changes

The PR removes status `allowAsyncLoad: false` guards, delegates CLI status context resolution to `src/agents/context.ts`, starts context-window cache warming during Gateway startup/status lookups, converts two catalog imports to static imports, and updates tests.

## Best Possible Solution

Make the first status render use bounded read-only provider/context metadata before formatting, preserve lazy startup/status boundaries where possible, and consolidate to one branch with redacted cold-start proof.

## Reproduction Assessment

Yes, source-reproducible for the PR blocker: PR-head code starts warmup with `void`, and the added test expects the first status-style lookup to remain `200_000` until async warmup is flushed. No live cold-start command was run in this read-only review.

## Solution Assessment

No. Delegating status resolution to the shared resolver is useful, but fire-and-forget warmup is only a partial mitigation; the first status render should deterministically use bounded metadata before formatting.
