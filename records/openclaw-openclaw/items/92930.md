## Summary

Keep open: the new recording is useful real behavior proof, but the patch still starts context warming in the background, so a first cold status render can still use stale fallback metadata, and it also widens the context import/startup boundary.

## What This Changes

The PR changes status summary/text and Gateway startup paths to trigger background context-window cache warming, delegates status summary runtime to the shared resolver, converts catalog loaders to static imports, and updates related tests.

## Best Possible Solution

Preserve lazy catalog boundaries and make one canonical status path either obtain bounded read-only context metadata before formatting the first cold output or explicitly accept background-only correction with maintainer approval.

## Reproduction Assessment

Yes. The PR blocker is source-reproducible: PR-head code starts warming with `void`, and the added test expects the first status-style lookup to remain `200_000` until async warmup is flushed.

## Solution Assessment

No. Delegating status to the shared resolver is useful, but fire-and-forget warming is not the best fix for a claimed first cold status result, and the static imports move the catalog loader boundary in the wrong direction.
