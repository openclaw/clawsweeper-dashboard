## Summary

Keep open: this targets a real status/context-window bug, but the patch still relies on fire-and-forget warmup before the displayed value is built and changes read-only status paths to start background catalog work. External real behavior proof is also missing.

## What This Changes

The PR changes context-window lookup to start background catalog warmup from status/summary paths, adds Gateway startup warming, imports catalog loaders statically, and updates related tests.

## Best Possible Solution

Keep the displayed status value sourced from already-prepared static/read-only metadata or an awaited safe lifecycle warmup, preserving read-only status semantics and adding real CLI proof for the linked bug.

## Reproduction Assessment

Yes, source-reproducible. The PR code schedules cache loading without awaiting it, and the added test itself shows the first status-style call still returns 200_000 until the async warmup is flushed.

## Solution Assessment

No. Delegating to a shared resolver is a reasonable direction, but fire-and-forget warmup is not the best fix for a one-shot status command; the safer path is a deterministic read-only/static metadata lookup or an awaited safe warmup before formatting.
