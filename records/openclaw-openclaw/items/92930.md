## Summary

Keep open: the PR targets a real status context-window bug, but the patch still warms the cache fire-and-forget, so a cold first `/status` can render before provider context metadata is available. The screenshot proof shows corrected output, but not fresh startup plus the first status call.

## What This Changes

The PR changes context-window resolution so status and gateway startup kick off model catalog/cache warming, delegates status summary to the shared context resolver, converts context catalog imports to static imports, and updates related tests.

## Best Possible Solution

Revise the PR so the first status render deterministically uses provider-owned context metadata, then add redacted cold-start proof showing startup and the first `/status` result.

## Reproduction Assessment

Yes for the PR defect from source inspection: the added tests themselves show the first status-style call returning 200_000 before warmup is flushed. No live cold-start command was run in this read-only review.

## Solution Assessment

No. Background warming is only a plausible partial mitigation; the best fix should make the first status render deterministic without opening unbounded startup/status side effects.
