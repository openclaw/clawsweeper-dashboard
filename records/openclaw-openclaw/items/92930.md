## Summary

Keep open: the focused bug fix is useful and the contributor's media shows corrected status output, but this PR is still blocked by a source-proven first-cold-status race and by moving catalog loaders into eager context imports, which creates status/startup compatibility and availability risk.

## What This Changes

The PR removes read-only status context lookup guards, delegates CLI status summary to the shared context resolver, starts context-window cache warming during Gateway startup, converts catalog loaders to static imports, and updates related tests.

## Best Possible Solution

Keep one canonical status context-resolution path that produces deterministic first-cold output without widening context.ts import/startup cost, or explicitly approve background-only correction with latency proof before merge.

## Reproduction Assessment

Yes. Source inspection and the PR-head tests reproduce the review blocker: the warmup is fired with void and the added test expects the first status-style lookup to remain 200_000 until flushAsyncWarmup runs.

## Solution Assessment

No. Delegating status to the shared resolver is the right direction, but fire-and-forget warmup plus static catalog imports is not the best fix for deterministic first-cold status output or startup safety.
