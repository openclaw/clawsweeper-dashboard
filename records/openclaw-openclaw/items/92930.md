## Summary

Keep open. The narrower direction is useful and the earlier committed screenshot-artifact blocker is gone, but current head still starts context-window warming in the background and can format a cold status response before discovered context rows are available; the screenshots show warmed output after a gateway had been up for 16+ minutes, not the first cold-start status path.

## What This Changes

The PR changes status/context-window lookup to start background catalog warmup from Gateway startup and status paths, delegates status-summary resolution to the shared context resolver, and updates related tests.

## Best Possible Solution

Make status compute or await bounded read-only context metadata before formatting the first cold response, while preserving explicit configured caps and limiting any startup background warmup policy.

## Reproduction Assessment

Yes by source inspection. PR-head code starts warmup with `void` and the added test still expects the first startup/status-style lookup to return `200_000` before async warmup is flushed.

## Solution Assessment

No. Delegating to the shared resolver is a useful direction, but fire-and-forget warmup is only a plausible mitigation; the best fix should make the first status response deterministic or preserve the read-only contract.
