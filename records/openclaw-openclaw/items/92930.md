## Summary

Keep open: the branch has a useful direction, but it still warms the cache asynchronously; its own added test expects a status-style first call to return 200_000 before the warmup finishes.

## What This Changes

The PR removes read-only async-load guards from status paths, delegates status-summary context resolution to the shared resolver, and starts context-window cache warming from Gateway startup.

## Best Possible Solution

Make the displayed status value deterministic from safe read-only metadata, or await a bounded read-only warmup before formatting, while preserving read-only status semantics unless maintainers explicitly choose the new side effect.

## Reproduction Assessment

Yes by source inspection. The PR-head test exercises a status-style call and expects 200_000 before flushAsyncWarmup(), so the cold first render can still read the fallback before the async cache is populated.

## Solution Assessment

No. Delegating to the shared resolver is useful, but relying on background warmup is not the best fix for a status command that needs the corrected value in the response being rendered.
