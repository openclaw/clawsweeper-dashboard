## What This Changes

The PR changes status and Gateway startup paths to use the shared context-window resolver and fire background model-catalog warmup, with focused context/status test updates.

## Best Possible Solution

Make the displayed status value deterministic by resolving from prepared static/read-only metadata or awaiting a bounded read-only warmup before formatting, while preserving read-only status semantics unless maintainers deliberately accept the new side effect.

## Reproduction Assessment

Yes. Source inspection and the PR's own added test show first status-style reads return before async cache warmup completes, so a cold one-shot status render can still see 200_000.

## Solution Assessment

No. Delegating to the shared resolver is a reasonable direction, but background-only warmup is not the best fix for a status command that needs the correct value before render.
