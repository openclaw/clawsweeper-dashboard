## Summary

Keep open: current main still applies symmetric jitter after a finite `retryAfterMs`, so the reported Retry-After undercut remains. This PR is a narrow active implementation path with no blocking review findings; the linked issue remains open and the competing PR is closed unmerged.

## What This Changes

This PR changes `retryAsync` to use a private positive-only jitter mode for satisfiable `retryAfterMs` delays and adds regression coverage for Retry-After lower-bound cases.

## Best Possible Solution

Review and land this narrow retry-helper fix after rebase and CI, preserving symmetric jitter for ordinary exponential backoff while enforcing server-supplied Retry-After as a lower bound whenever the configured max delay can honor it.

## Reproduction Assessment

Yes: there is a deterministic source-level reproduction by mocking `generateSecureFraction` to `0` and calling `retryAsync` with `retryAfterMs: () => 1000` and `jitter: 0.5`, which current main computes as a `500` ms delay. The PR’s new regression test encodes that path directly.

## Solution Assessment

Yes: the proposed private positive-only jitter mode is the narrowest maintainable fix because it changes only retry delay math, keeps the public API unchanged, and preserves symmetric jitter for non-Retry-After callers and unsatisfiable over-cap hints.
