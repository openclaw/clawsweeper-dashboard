- `impact:session-state`: The failure is caused by replayed conversation reasoning state on continued turns.
- `impact:auth-provider`: The affected behavior is provider-specific Responses request construction for microsoft-foundry.

## Summary

Keep open: current main and the latest release still contain the same Responses replay shape that drops reasoning item ids while preserving `encrypted_content`, and the related Azure Foundry issue is broader rather than a merged or safer canonical fix for this specific replay failure.

## What This Changes

The issue reports that microsoft-foundry reasoning models can fail continued Responses threads with `400 invalid_encrypted_content` because replayed encrypted reasoning items lose their `id`.

## Best Possible Solution

Add a narrow microsoft-foundry Responses replay policy or transport option that keeps `id` with same-session encrypted reasoning items, with focused regression coverage and live Foundry proof.

## Reproduction Assessment

We have a high-confidence source-level reproduction path: current main converts stored thinking signatures back into reasoning items, then the default shared Responses path drops ids while preserving `encrypted_content`. I did not run a live Foundry request, so this is not marked fully reproduced.

## Solution Assessment

Yes, the best direction is a provider-scoped fix that preserves ids for microsoft-foundry encrypted reasoning replay. A blanket shared default change would be riskier for other OpenAI-compatible store-disabled providers.
