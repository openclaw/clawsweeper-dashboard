## Label Justifications

- `P1`: The report describes a deterministic continued-thread failure for Foundry reasoning models that can make an active conversation unrecoverable except by starting over.
- `impact:session-state`: The failure occurs when replaying prior reasoning history into a continued thread and can strand that session.
- `impact:auth-provider`: The affected behavior is provider-specific Responses payload shaping for the microsoft-foundry model provider.

## Summary

Keep open: the report is credible and current main still contains a tested Responses replay shape that drops reasoning item ids while preserving encrypted_content, which matches the reported Foundry rejection. There is not enough source/history proof that the newer boundary-aware transport fully eliminates this path for microsoft-foundry, so this should stay actionable.

## What This Changes

The issue reports that microsoft-foundry reasoning models can fail on continued turns because OpenClaw replays encrypted reasoning items without their item ids.

## Best Possible Solution

Add a provider-scoped replay fix for microsoft-foundry encrypted reasoning items, with regression coverage proving encrypted_content and the required id stay paired without changing unrelated OpenAI store-disabled behavior.

## Reproduction Assessment

Yes, with source-level confidence: the reporter supplied a deterministic Foundry probe, and current source still has a path that serializes encrypted_content without id. I did not run a live Foundry request in this review.

## Solution Assessment

No patch is proposed here; the best fix is provider-scoped rather than a global replay-id reversal, because the existing OpenAI store-disabled behavior was intentionally changed and tested.
