## Label Justifications

- `impact:auth-provider`: The observed failure dispatches a GPT model identifier under the unrelated ZAI provider.

## Summary

The report remains necessary: the stale Codex sidecar theory is not sufficient, but current main still constructs and persists provider/model values through separable fields, while its preservation rule covers only named inter-session tools rather than generic runtime continuations. The supplied Discord evidence shows a repeatable invalid `zai/gpt-5.5` route on both sidecar and no-sidecar sessions, so there is not enough proof to close as fixed or duplicate.

## What This Changes

Prevent embedded continuations from combining a GPT model from fallback or persisted runtime state with a different active provider.

## Best Possible Solution

Carry one validated resolved model reference through continuation selection, Codex binding use, execution metadata, transcript snapshots, and session persistence, and reject or recover before dispatch when the full provider/model pair is not a valid configured route.

## Reproduction Assessment

Yes at source level: current main has separable provider/model selection and persistence surfaces, and the supplied logs show the invalid pair repeatedly in sidecar and no-sidecar sessions. A clean live reproduction against current main was not performed, so this is not marked fully reproduced.

## Solution Assessment

No, clearing Codex sidecars or replacing the product default alone is not the best solution. The maintainable fix is to preserve and validate one atomic resolved route across all internal-continuation and persistence boundaries.
