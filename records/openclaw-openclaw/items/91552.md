## Summary

This issue remains necessary: current main still permits provider and model identity to come from separable persisted fields, and Codex thread resume also accepts independently supplied model and provider overrides. The reporter’s successive theories are not individually proven, but the supplied logs and state samples identify a credible unresolved class of cross-provider session corruption; adjacent open issues and PRs do not safely supersede this exact embedded-continuation failure.

## What This Changes

Investigate and fix cross-provider model/session state corruption that can dispatch invalid routes such as `zai/gpt-5.5` during embedded continuations.

## Best Possible Solution

Represent provider, opaque provider-relative model ID, runtime, and auth identity as one validated model selection; migrate legacy split session state through a named doctor/migration owner, and make snapshots, embedded continuations, fallback handling, and Codex thread reuse consume that same selection without recombining independent fields.

## Reproduction Assessment

No, not at high confidence on current main. The reporter supplied detailed production logs and persisted-state examples, and current source confirms the implicated split-state paths, but a controlled run has not isolated which writer first creates the invalid route.

## Solution Assessment

No. Sidecar deletion is too narrow, changing the hardcoded default addresses a downstream fallback, and slash-based override detection can corrupt legitimate provider-relative model IDs; the repair should instead establish one atomic route-selection boundary.
