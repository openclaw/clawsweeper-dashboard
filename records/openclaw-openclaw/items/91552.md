## Summary

Keeping this open. Current main has partial cleanup for stored session overrides, but I could not prove the reported embedded Codex continuation route is fixed: Codex bindings still persist model/modelProvider separately, model snapshots record the runner-supplied route, and upstream Codex cannot correct provider at turn/start once a thread-level provider/model binding is stale.

## What This Changes

The issue reports stale Codex app-server sidecar, model-snapshot, or session override state mixing a GPT model with a non-OpenAI provider during embedded continuations.

## Best Possible Solution

Add a maintainer-approved canonical route validation boundary for Codex bindings, model snapshots, and embedded continuations so provider/model/auth come from one resolved selection object, with doctor cleanup for shipped stale persisted state.

## Reproduction Assessment

No live current-main reproduction was run, but source inspection plus detailed reporter logs show a plausible path where persisted route fragments can reach embedded continuations without full provider/model validation.

## Solution Assessment

Unclear; the best fix should be at the route-resolution and Codex binding lifecycle boundary, not only hardcoded defaults or slash stripping, because wrapper providers intentionally use slashy model IDs.
