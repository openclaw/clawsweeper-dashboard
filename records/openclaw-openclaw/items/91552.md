## Summary

Keep this issue open: the reported invalid provider/model route is credible and user-impacting, but the discussion now contains several competing root-cause theories, and no current-main reproduction proves that the shipped override normalization or newer Codex request handling resolves the broader ZAI-only continuation case.

## What This Changes

Prevent embedded continuations and persisted session state from combining a provider with a model selected from a different session, runtime, or stale binding.

## Best Possible Solution

Resolve provider and model once into a validated route object, carry that object through override persistence, model snapshots, continuation and memory-flush runs, and Codex request fields, then add a two-session regression proving that a ZAI-only session cannot inherit the OpenAI model used concurrently by another session.

## Reproduction Assessment

No. The installed-version logs and persisted-state examples are strong, but there is not yet a controlled current-main reproduction that isolates which of the changing root-cause hypotheses remains valid.

## Solution Assessment

No single workaround proposed in the discussion is the best complete fix. Canonical full-route propagation and validation is safer than deleting Codex sidecars, changing the hardcoded default, or normalizing one override shape in isolation.
