## Summary

Keep open: current main and the latest release still lack the requested message-tool `channel` schema metadata, and the open draft PR at https://github.com/openclaw/openclaw/pull/82868 is the active closing implementation path.

## What This Changes

Request to add provider-id guidance, and possibly a dynamic enum, to the message tool `channel` parameter so agents do not pass Slack or Discord destination ids as provider ids.

## Best Possible Solution

Land a narrow schema/test patch that describes `channel` as a configured deliverable provider id and derives any enum from registered deliverable channel ids, avoiding internal-only examples.

## Reproduction Assessment

Yes for the source-level issue: current main and `v2026.5.12` expose `channel` as a bare optional string and spread it into the message tool schema. I did not run a live Slack or Discord workspace repro.

## Solution Assessment

Yes for the requested direction: schema metadata plus focused regression coverage is the narrow maintainable fix. Any enum should be derived from registered deliverable channel ids rather than hard-coded.
