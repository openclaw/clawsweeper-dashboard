## Summary

Keep open: current main and the latest release still expose the shared message tool `channel` parameter as an undescribed optional string, and live GitHub metadata shows two open closing pull requests that should be reviewed before this issue is closed.

## What This Changes

Request to add provider-oriented schema metadata to the message tool `channel` parameter so agents pass provider ids like `slack` or `discord`, not destination channel or conversation ids.

## Best Possible Solution

Review and land one maintainer-preferred linked pull request for the narrow schema/test change, then close the duplicate linked pull request if superseded; any enum should be registry-derived rather than hard-coded.

## Reproduction Assessment

Yes for a source-level reproduction: current main exposes `channel` as a bare optional string, and invalid explicit channel values still have an `Unknown channel` path. I did not run a live Slack or Discord workspace reproduction.

## Solution Assessment

Yes: adding provider-oriented schema metadata plus focused regression coverage is the narrow maintainable fix. A static enum would be weaker unless it is derived from registered or configured channel/plugin ids.
