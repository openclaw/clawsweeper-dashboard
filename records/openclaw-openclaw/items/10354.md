## Summary

Keep open: current main and the latest release still expose the shared message tool `channel` parameter as an undescribed optional string, and two open pull requests already target this issue as implementation paths.

## What This Changes

Request to clarify the message tool `channel` schema so agents pass messaging provider ids such as `slack` or `discord` rather than destination channel or conversation ids, optionally with an enum.

## Best Possible Solution

Review and land one maintainer-preferred linked PR that adds provider-id schema guidance and focused coverage, while deciding whether enum support should be registry-derived or deferred.

## Reproduction Assessment

Yes for a source-level reproduction: current main exposes `channel` as a bare optional string, and the outbound path still has an `Unknown channel` branch for unrecognized explicit channel values. I did not run a live Slack or Discord setup.

## Solution Assessment

Yes for the description portion: adding TypeBox metadata plus focused schema coverage is the narrow maintainable fix. The enum portion is unclear and should be derived from registered channel/plugin ids if included.
