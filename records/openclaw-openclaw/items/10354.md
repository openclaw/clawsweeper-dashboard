## Summary

Keep open. Current main still exposes the shared message tool `channel` parameter as a bare optional string with no schema description or enum, and the two linked implementation PRs are closed unmerged, so the requested schema guidance remains a valid narrow repair.

## What This Changes

Add schema guidance to the message tool `channel` parameter so agents pass a messaging provider or channel plugin id, not a destination channel/conversation id.

## Best Possible Solution

Land a narrow core update that keeps `channel` optional, adds a TypeBox description explaining that it expects a messaging provider or channel plugin id such as `slack` or `discord`, and adds a focused schema assertion. Treat enum support as a follow-up unless it can be derived from the plugin-aware channel values used at runtime.

## Reproduction Assessment

Yes. Source/schema inspection against current main reproduces the issue: `createMessageTool()` reaches `buildRoutingSchema()`, and its `channel` property has no `description` or enum metadata.

## Solution Assessment

Yes for the description portion. The narrowest maintainable fix is schema metadata plus a focused test; a hard-coded static enum is not the best path unless it is derived from registered channel/plugin values.
