## Summary

Keep open. Current main and the latest release still expose the message tool `channel` parameter as an undescribed optional string, and the open linked PR is the active implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Request to add tool-schema guidance, and possibly enum support, so the message tool `channel` parameter is used for provider or channel plugin ids rather than destination conversation ids.

## Best Possible Solution

Keep this issue as the canonical tracker until the linked PR lands or is closed; land the narrow schema/test repair, then split dynamic enum support into a separate follow-up only if maintainers want that behavior.

## Reproduction Assessment

Yes, source-level: current main exposes `channel` as an undescribed arbitrary string while outbound routing interprets explicit channel values as provider/channel ids and can throw `Unknown channel`. I did not run a live Slack or Discord workspace reproduction in this read-only review.

## Solution Assessment

Yes for the description portion: TypeBox schema metadata plus a focused schema assertion is the narrow maintainable fix. Enum support is still a product/API choice and should be derived from registered or configured channels if pursued.
