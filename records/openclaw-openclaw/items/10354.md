## Summary

Keep open. Current main and the latest release still expose the message tool `channel` schema as an undescribed optional string, and the active linked implementation at https://github.com/openclaw/openclaw/pull/84589 is still open rather than merged.

## What This Changes

The issue requests clearer message-tool schema metadata so `channel` means the messaging provider or channel plugin id, not the destination conversation or channel id.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84589 if description-only is accepted, or adjust that PR and track registry-derived enum support separately if maintainers still want it.

## Reproduction Assessment

Yes, source-level: current main exposes `channel` as an undescribed optional string while outbound routing interprets it as a provider/channel id and can still throw `Unknown channel` for invalid explicit values without fallback.

## Solution Assessment

Yes for the description portion: TypeBox metadata plus a focused schema assertion is the narrow maintainable repair. The enum portion still needs maintainer direction and should be derived from registered channel/plugin ids if pursued.
