- `P2`: This is a bounded agent message-tool schema issue tied to real outbound messaging failures, with limited blast radius.

## Summary

Keep open. Current main and the latest release still expose the shared `message` tool `channel` parameter as an undescribed optional string, and https://github.com/openclaw/openclaw/pull/84589 is the open closing PR that now owns the narrow schema/test repair.

## What This Changes

Add clearer schema metadata, and possibly an enum, for the shared `message` tool `channel` parameter so agents pass provider/channel ids such as `slack` instead of destination conversation ids.

## Best Possible Solution

Review and land or deliberately close https://github.com/openclaw/openclaw/pull/84589, adjusting the closing reference or opening a follow-up if maintainers want enum support tracked separately.

## Reproduction Assessment

Yes, source-level: current main exposes `channel` as a bare optional string while prompt/docs and outbound routing treat it as a provider/channel id. I did not rerun the reporter's Slack workspace scenario live.

## Solution Assessment

Yes for the description portion: TypeBox metadata plus a focused schema assertion is the narrow maintainable fix. Enum support needs maintainer direction and should be derived from registered/configured channels if pursued.
