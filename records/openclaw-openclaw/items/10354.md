## Summary

Current main and v2026.6.5 still expose the shared message tool `channel` parameter as an undescribed optional string, while the open, mergeable closing PR provides the narrow description-and-test implementation with positive generated-schema proof. The issue should remain open until that PR is merged or closed; a static enum should not be folded into this fix without a plugin-aware product decision.

## What This Changes

Add provider-oriented schema guidance, and optionally a plugin-aware enum, for the message tool’s optional `channel` parameter.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/84589 with the provider-name description and focused schema test, and defer enum enforcement unless maintainers choose a deterministic enum derived from the registered plugin/channel snapshot.

## Reproduction Assessment

Yes at source level. Current main accepts an arbitrary undescribed `channel` string, and the outbound selector interprets it as a provider/channel ID, so destination-style values can reach the unknown-channel error path; no live Slack workspace reproduction was run in this read-only review.

## Solution Assessment

Yes for the description-only linked PR: it is the narrowest maintainable fix and preserves runtime behavior. The optional enum is not yet the best solution because valid channel IDs include dynamically registered plugins and aliases.
