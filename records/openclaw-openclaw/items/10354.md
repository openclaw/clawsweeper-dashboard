## Summary

Keep open. Current main still exposes the shared message tool `channel` parameter as a bare optional string with no description or enum metadata, and the linked implementation PRs are closed unmerged.

## What This Changes

Add clear schema metadata for the shared message tool `channel` parameter so agents know to pass a provider or channel plugin id rather than a destination channel id.

## Best Possible Solution

The shared message tool schema should keep `channel` optional, add provider/plugin-id wording, and only add enum metadata if it is derived from the same plugin-aware channel registry used by routing.

## Reproduction Assessment

Yes. Source inspection against current main shows the `message` tool schema still comes from `buildRoutingSchema()`, where `channel` has no description or enum metadata.

## Solution Assessment

Yes for the description portion. The narrowest maintainable fix is TypeBox schema metadata plus a focused assertion; a hard-coded enum should be skipped unless it uses plugin-aware channel ids.
