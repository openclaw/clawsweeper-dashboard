## Summary

Keep open: current main still exposes the shared message tool `channel` parameter as an undescribed optional string, and https://github.com/openclaw/openclaw/pull/82868 is an open draft implementation that explicitly fixes this issue. The issue should remain open until that PR is either landed or closed/replaced.

## What This Changes

Request provider-oriented schema guidance, and possibly plugin-aware enum metadata, for the `message` tool `channel` parameter so agents pass provider ids instead of destination channel ids.

## Best Possible Solution

Review and land or replace the linked PR with a narrow schema/test update that adds provider-id guidance and only uses enum metadata derived from registered deliverable channel ids.

## Reproduction Assessment

Yes for the source-level schema gap: current main shows `channel` is a bare optional string and is spread into the message tool schema. I did not run a live Slack or Discord workspace repro in this read-only review.

## Solution Assessment

Yes for the description portion: schema metadata plus focused regression coverage is the narrowest maintainable repair. Enum support should be derived from registered channel/plugin ids rather than hard-coded.
