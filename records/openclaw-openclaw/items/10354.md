## Summary

Keep open: current main and the latest release still expose the shared message tool `channel` parameter as an undescribed optional string, while open linked PRs already carry the active fix path.

## What This Changes

The issue requests provider-oriented schema metadata for the `message` tool `channel` parameter so agents pass provider ids such as `slack` or `discord` instead of conversation ids.

## Best Possible Solution

Review and land one maintainer-preferred linked PR for the narrow schema/test change, then close the duplicate linked PR if it is superseded; any enum should be derived from registered deliverable channel ids rather than hard-coded.

## Reproduction Assessment

Yes for the source-level issue: current main and `v2026.5.19` still expose `channel` as a bare optional string, and current routing still has an `Unknown channel` path for unrecognized explicit channel values. I did not run a live Slack or Discord workspace reproduction.

## Solution Assessment

Yes: adding provider-oriented schema metadata plus focused regression coverage is the narrow maintainable fix. A static enum would be weaker than a registry-derived list because deliverable channels include registered plugin ids.
