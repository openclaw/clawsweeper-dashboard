## Summary

Keep open: current main still exposes the shared message tool `channel` parameter as an undescribed optional string, while the prompt/runtime fallback paths do not add the requested tool-schema guidance and the two direct implementation PRs are closed unmerged.

## What This Changes

Request adding tool-schema description, and possibly enum metadata, for the shared message tool `channel` parameter so agents pass provider or plugin ids instead of destination conversation ids.

## Best Possible Solution

Add a focused TypeBox description for `channel`, keep it optional, add a schema assertion, and only add enum metadata if it is dynamically derived from registered deliverable channel ids.

## Reproduction Assessment

Yes for the schema gap: current-main source inspection shows the shared message tool exposes `channel` as a bare optional string that is spread into the public tool parameters. A live Slack or Discord workspace is not needed to verify the missing metadata.

## Solution Assessment

Yes for the description portion: schema metadata plus a focused assertion is the narrowest maintainable repair. A hard-coded static enum is not the best default unless it is generated from plugin-aware channel ids.
