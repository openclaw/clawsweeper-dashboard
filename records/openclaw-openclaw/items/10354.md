## Summary

Keep open. Current main and the latest release still expose the message tool channel parameter as an undescribed optional string, and the open linked implementation PR should be reviewed or adjusted before this issue closes.

## What This Changes

The issue requests clearer message-tool schema metadata so the optional channel parameter is understood as a messaging provider id rather than a destination conversation or channel id.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84589 if description-only is accepted, or adjust that PR and track dynamic enum support separately if maintainers still want it.

## Reproduction Assessment

Yes, source-level: current main exposes channel as an undescribed optional string while outbound routing interprets channel as a provider/channel id and still throws Unknown channel for invalid explicit values without fallback.

## Solution Assessment

Yes for the description portion: TypeBox metadata plus a focused schema assertion is the narrow maintainable fix. The enum portion still needs maintainer direction and should be dynamically derived if pursued.
