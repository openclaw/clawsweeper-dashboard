## Summary

Keep open: current main and the latest release still expose the message tool channel parameter without the requested schema guidance, and the open linked pull request is the active implementation path rather than a reason to close this issue early.

## What This Changes

The issue requests clearer schema metadata for the message tool channel parameter so agents pass provider ids like slack or discord instead of destination channel ids.

## Best Possible Solution

Land or adjust the narrow schema/test repair in the linked PR, and track any registry-derived enum support separately if maintainers want that stricter tool contract.

## Reproduction Assessment

Yes, source-level: current main exposes channel as an undescribed arbitrary string while outbound routing interprets it as a provider/channel id and can throw Unknown channel. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes for the description portion: TypeBox metadata plus a focused schema assertion is the narrow maintainable repair. The enum portion still needs maintainer direction because channel/plugin ids are dynamic and a static enum could drift.
