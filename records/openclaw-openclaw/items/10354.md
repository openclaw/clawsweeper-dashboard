## Summary

Keep open. Current main and the latest release still expose the shared message tool `channel` parameter as a bare optional string, while the related implementation PRs are closed unmerged and no open closing PR was found.

## What This Changes

Request to add schema guidance, and possibly enum metadata, for the shared message tool's optional `channel` parameter so agents pass a messaging provider or channel plugin id rather than a destination conversation id.

## Best Possible Solution

Add a focused TypeBox description for `channel`, keep it optional, add a schema assertion, and only add enum metadata if it is derived from plugin-aware channel ids.

## Reproduction Assessment

Yes for the schema gap: current-main source inspection shows the shared message tool exposes `channel` as a bare optional string. A live Slack or Discord workspace is not needed to verify the missing tool metadata.

## Solution Assessment

Yes for the description portion: schema metadata plus a focused assertion is the narrowest maintainable repair. A hard-coded static enum is not the best default unless it is generated from plugin-aware channel ids.

Fix https://github.com/openclaw/openclaw/issues/10354 with a narrow message-tool schema update. In `src/agents/tools/message-tool.ts`, keep `channel` optional but add a TypeBox description explaining that it expects a messaging provider or channel plugin id such as `slack` or `discord`, not a destination conversation/channel/user id. Add a focused assertion in `src/agents/tools/message-tool.test.ts` that the message tool schema exposes that description.
