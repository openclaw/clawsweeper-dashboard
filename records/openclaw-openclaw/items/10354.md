## Summary

Keep open: current main and the latest release still expose the message tool `channel` parameter without schema guidance, and the issue is already paired with an open unmerged closing PR for the narrow schema-description/test fix.

## What This Changes

The issue requests adding tool-schema guidance, and possibly enum constraints, so agents pass a messaging provider id in the shared message tool's `channel` parameter instead of a destination channel or conversation id.

## Best Possible Solution

Land or revise the open schema-description PR after maintainers decide the enum scope; if enum support remains desired, track it as a registry-derived follow-up rather than blocking the narrow description repair indefinitely.

## Reproduction Assessment

Yes. Source inspection shows current main exposes `channel` as an undescribed optional string while outbound routing interprets it as provider/channel selection and still throws `Unknown channel` for invalid explicit values without a valid fallback.

## Solution Assessment

Yes for the description portion: TypeBox metadata plus a focused schema assertion is the narrow maintainable repair. The enum portion is unclear and should be a maintainer decision, preferably derived from registered channel/plugin ids if pursued.
