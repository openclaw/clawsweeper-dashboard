## Summary

Keep open: current main and the latest release still expose the shared message tool `channel` parameter as an undescribed optional string, and open linked implementation PRs already own the remaining schema guidance work.

## What This Changes

The issue requests clearer schema guidance for the shared message tool's optional `channel` parameter so agents pass provider ids such as `slack` instead of destination channel ids.

## Best Possible Solution

Review and land one narrow linked PR that adds provider-id guidance to `channel`; include enum support only if it is dynamically derived from registered channel/plugin ids and covered by focused schema tests.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.5.22` show the exposed routing schema still uses a bare optional string for `channel`. I did not rerun the live Slack failure, but the schema gap itself is directly visible in source.

## Solution Assessment

Yes for keeping this open behind the existing linked PRs. The narrow maintainable fix is schema description plus focused coverage; enum support is plausible but needs maintainer choice around dynamic channel/plugin availability.
