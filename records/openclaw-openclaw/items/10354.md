## Summary

Keep open. Current main and the latest release still expose the message tool `channel` parameter as an undescribed optional string, while https://github.com/openclaw/openclaw/pull/84589 is an open fix candidate rather than merged behavior.

## What This Changes

Request to add schema guidance, and possibly enum support, for the `message` tool `channel` parameter so agents pass provider/channel plugin ids instead of destination conversation ids.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84589 for the schema-description/test repair if accepted, then handle any dynamic enum work as a separate maintainer-approved follow-up.

## Reproduction Assessment

Yes, source-level. Current main exposes `channel` as an undescribed optional string while outbound routing interprets it as provider/channel selection and still throws `Unknown channel` for invalid explicit values without a valid fallback.

## Solution Assessment

Yes for the description portion: TypeBox metadata plus a focused schema assertion is the narrow maintainable repair. The enum portion is unclear and should be decided separately, preferably using registered channel/plugin ids if pursued.
