## Summary

Keep open: current main and the latest release still expose the shared `message` tool `channel` parameter as an undescribed string, while https://github.com/openclaw/openclaw/pull/84589 is the active open closing PR for the narrow schema fix.

## What This Changes

The issue requests clearer provider-name guidance, and possibly a safe enum, for the shared `message` tool `channel` parameter so agents do not pass Slack or Discord conversation IDs as provider ids.

## Best Possible Solution

Land or adjust https://github.com/openclaw/openclaw/pull/84589 as the narrow schema/test repair, then track any registry-derived enum work separately if maintainers want that behavior.

## Reproduction Assessment

Yes, source-level. Current main exposes `channel` as an undescribed arbitrary string while outbound routing interprets it as a provider/channel id; I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes for the description portion: TypeBox metadata plus a focused schema assertion is the narrow maintainable repair. The enum portion is still a product/API decision because channel ids include registered plugin ids and should not be hard-coded casually.
