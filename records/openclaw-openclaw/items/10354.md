## Summary

Keep open: current main and v2026.5.12 still expose the shared message tool `channel` parameter as an undescribed optional string, and an open draft PR already targets this issue with closing syntax.

## What This Changes

The issue requests provider-id guidance, and optionally a safe enum, for the shared `message` tool `channel` schema.

## Best Possible Solution

Review, land, or replace https://github.com/openclaw/openclaw/pull/82868 with a narrow schema/test update that adds provider-id guidance and derives any enum from registered deliverable channel ids.

## Reproduction Assessment

Yes for the source-level schema gap: current main and v2026.5.12 define `channel` as a bare optional string and spread it into the shared message tool schema. I did not establish a live Slack or Discord workspace repro in this read-only review.

## Solution Assessment

Yes for the description portion: schema metadata plus focused regression coverage is the narrowest maintainable repair. Any enum support should be plugin-aware and derived from registered deliverable channel ids, not hard-coded.
