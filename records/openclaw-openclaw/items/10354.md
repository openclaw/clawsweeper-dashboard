## Summary

Keep open: current main and the latest release still expose the message tool `channel` parameter as an undescribed optional string, and https://github.com/openclaw/openclaw/pull/84589 is an open unmerged fix candidate for the narrow schema-description/test path.

## What This Changes

The issue requests clearer message tool schema guidance, and possibly enum metadata, so agents pass provider names such as `slack` or `discord` instead of destination channel IDs.

## Best Possible Solution

Land or revise the open schema-description PR after maintainers decide the closing scope; if enum support is still desired, derive it from registered channel/plugin ids or track it as a focused follow-up.

## Reproduction Assessment

Yes at source level. Current main exposes `channel` as an undescribed optional string while outbound routing treats explicit values as provider/channel ids and can still throw `Unknown channel` for invalid destination-style input.

## Solution Assessment

Yes for the description portion: TypeBox metadata plus a focused schema assertion is the narrow maintainable repair. The enum portion needs maintainer direction and should be registry-derived if pursued.
