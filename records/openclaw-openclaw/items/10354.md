## Summary

Keep open: current main and the latest release still expose the shared message tool `channel` parameter as an undescribed optional string, and open linked implementation PRs already target the remaining schema guidance work.

## What This Changes

Request to add provider-name guidance, and possibly enum constraints, to the message tool `channel` parameter so agents do not pass destination conversation or channel IDs there.

## Best Possible Solution

Review and land one maintainer-preferred linked PR that adds provider-id schema guidance and focused coverage; derive any enum from registered channel/plugin ids or explicitly defer it.

## Reproduction Assessment

Yes, source-level: current main exposes `channel` as a bare optional string, and the outbound path still rejects unrecognized explicit values such as Slack-shaped channel ids. I did not run a live Slack or Discord setup.

## Solution Assessment

Yes for the description portion: TypeBox metadata plus a focused schema assertion is the narrow maintainable fix. The enum portion is unclear and should be registry-derived or deferred by maintainer decision.
