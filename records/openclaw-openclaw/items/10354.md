## Summary

## What This Changes

Add message-tool schema guidance, and preferably a registry-derived enum, so `channel` means provider id rather than destination channel/user id.

## Best Possible Solution

Review, repair, and land or close https://github.com/openclaw/openclaw/pull/82868; if it closes unmerged, revive the same narrow schema/test fix with a registry-derived deliverable-provider enum.

## Reproduction Assessment

Yes for the source-level issue: current `main` and `v2026.5.12` expose `channel` as a bare optional string, and the outbound selection path still has an `Unknown channel` failure for invalid explicit channels. I did not run a live Slack or Discord workspace repro.

## Solution Assessment

Yes for the requested direction: schema metadata plus focused regression coverage is the narrow maintainable fix. Any enum should be derived from registered deliverable channel ids rather than hard-coded.
