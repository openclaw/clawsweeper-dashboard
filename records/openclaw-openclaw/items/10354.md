## Summary

Keep open. Current main still exposes the shared message tool `channel` parameter as an optional bare string with no schema description or enum metadata, and the linked implementation PRs are closed unmerged.

## What This Changes

The issue requests adding tool-schema guidance, and possibly enum metadata, so the message tool `channel` parameter is understood as a provider/plugin id rather than a destination channel id.

## Best Possible Solution

The shared message tool schema should expose clear provider/plugin-id wording for `channel`, while leaving enum support dynamic and plugin-aware or omitted.

## Reproduction Assessment

Yes. Source inspection against current main shows `createMessageTool()` receives schema composed from `buildRoutingSchema()`, where `channel` has no description or enum metadata.

## Solution Assessment

Yes for the description portion. The narrowest maintainable fix is TypeBox schema metadata plus a focused assertion; a static enum should be skipped unless it uses the plugin-aware channel ids.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/10354 with a narrow message-tool schema update. In `src/agents/tools/message-tool.ts`, keep `channel` optional but add a TypeBox description explaining that it expects a messaging provider or channel plugin id such as `slack` or `discord`, not a destination conversation/channel/user id. Add or update a focused assertion in `src/agents/tools/message-tool.test.ts` proving `createMessageTool()` exposes that description on `parameters.properties.channel`. Do not change runtime fallback/routing behavior, and do not hard-code a bundled-only enum; add enum metadata only if it is derived from the plugin-aware channel list. Add a single-line user-facing `CHANGELOG.md` entry unless maintainers decide this is internal-only.
