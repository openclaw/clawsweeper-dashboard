## Summary

Keep open. Current main still exposes the shared message tool `channel` parameter as a bare optional string, so the requested schema guidance remains unimplemented while the linked implementation PRs are closed unmerged.

## What This Changes

Request to add schema metadata clarifying that the message tool `channel` parameter expects a messaging provider or channel plugin id, not a destination channel or conversation id.

## Best Possible Solution

Land a narrow schema/test update that keeps `channel` optional, adds provider/plugin-id wording, and only adds enum metadata if it is derived from the plugin-aware channel list.

## Reproduction Assessment

Yes. Source inspection against current main reproduces the schema gap: `createMessageTool()` uses `buildRoutingSchema()`, and that `channel` property has no `description` or enum metadata.

## Solution Assessment

Yes for the description portion. The narrowest maintainable fix is schema metadata plus a focused assertion; static enum support should be skipped unless it uses the same plugin-aware channel ids as runtime routing.
