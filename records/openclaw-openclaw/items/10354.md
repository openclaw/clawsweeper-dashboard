## Summary

Keep open: current main and the latest release still expose the shared message tool `channel` parameter as an undescribed string, and an open linked PR is the active fix candidate using closing syntax for this issue.

## What This Changes

The issue asks for the shared `message` tool `channel` parameter to describe that it expects a messaging provider name, with possible enum guidance to prevent agents from passing destination IDs.

## Best Possible Solution

Review or land https://github.com/openclaw/openclaw/pull/84589 for the narrow schema-description/test repair, and split or explicitly defer dynamic enum support if maintainers do not want that scope to block the description fix.

## Reproduction Assessment

Yes, source-level: current main exposes `channel` as an undescribed arbitrary string while outbound routing treats explicit `channel` values as provider/channel selection. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes for the description portion: TypeBox metadata plus focused schema coverage is the narrow maintainable repair. The enum portion is unclear because channel ids come from bundled and registered plugin channels, so that part needs maintainer direction rather than a hard-coded static enum.
