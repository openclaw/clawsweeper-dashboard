Keep open. Current main and the latest release still expose the shared message tool `channel` parameter as an undescribed optional string, while the active fix path is the open linked PR; this issue should stay open until that PR is landed or maintainers decide the remaining enum scope.

## What This Changes

The issue requests clearer schema guidance for the message tool `channel` parameter so agents pass provider ids such as `slack` or `discord` rather than destination channel IDs.

## Best Possible Solution

Review and either land or adjust https://github.com/openclaw/openclaw/pull/84589; if maintainers want enum support too, track that as a dynamic plugin-aware follow-up rather than hard-coding a static enum.

## Reproduction Assessment

Yes, source-level: current main and the latest release still expose `channel` as an undescribed optional string in the shared message tool schema. I did not run a live Slack or Discord workspace reproduction in this read-only review.

## Solution Assessment

The best path is a narrow schema-description update with lightweight regression coverage. Enum support should remain a maintainer decision unless it is dynamically derived from registered channel/plugin ids.
