## Summary

Keep open: current main and v2026.5.22 still expose the shared message tool `channel` parameter as an undescribed optional string, and two open linked PRs already target the remaining schema guidance work.

## What This Changes

The issue requests provider-oriented description metadata, and possibly enum guidance, for the message tool `channel` parameter so agents do not pass destination channel IDs there.

## Best Possible Solution

Review and land one maintainer-preferred linked PR that adds provider-id schema guidance and focused coverage; derive any enum from registered channel/plugin ids or defer it.

## Reproduction Assessment

Yes for a source-level reproduction: current main exposes `channel` as a bare optional string, and the outbound path still has an `Unknown channel` branch for unrecognized explicit channel values. I did not run a live Slack or Discord setup.

## Solution Assessment

Yes for the description portion: adding TypeBox metadata plus focused schema coverage is the narrow maintainable fix. The enum portion is unclear and should be registry-derived or explicitly deferred.
