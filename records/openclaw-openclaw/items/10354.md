## Summary

Keep open. Current main still exposes the shared message tool `channel` parameter as a bare optional string, while prompt and fallback paths do not add the requested tool-schema guidance and the prior implementation PRs are closed unmerged.

## What This Changes

Add schema guidance, and possibly plugin-aware enum metadata, for the message tool `channel` parameter so agents pass provider/plugin ids instead of destination ids.

## Best Possible Solution

Add a focused TypeBox description for `channel`, keep it optional, and only add enum metadata if it can be derived from plugin-aware deliverable channel ids without broad runtime loading.

## Reproduction Assessment

Yes for the requested schema gap: source inspection on current main shows the `channel` property is still a bare optional string in the message tool schema. I did not need a live Slack setup to verify the schema omission.

## Solution Assessment

Yes for the description portion: schema metadata plus a focused assertion is the narrowest maintainable repair. A static enum is not the best default unless it is generated from the same plugin-aware channel id source used by routing.

Do not hard-code a static enum unless you can derive it from plugin-aware deliverable channel ids without broad channel/plugin runtime loading; description-only is acceptable. Add or update focused coverage in `src/agents/tools/message-tool.test.ts` asserting `createMessageTool().parameters` exposes the `channel` description. Add a concise `CHANGELOG.md` entry if treating the schema guidance as user-facing; do not add bot-only attribution. Do not change runtime routing, Slack target normalization, or channel fallback policy as part of this repair.
