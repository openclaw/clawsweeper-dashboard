## Summary

Keep open: current main and the latest release still expose the shared message tool `channel` parameter as an undescribed optional string, and https://github.com/openclaw/openclaw/pull/82868 is an open draft implementation that targets this issue.

## What This Changes

Add provider-id guidance, and possibly registry-derived enum metadata, to the message tool `channel` schema so agents do not pass destination channel or user IDs there.

## Best Possible Solution

Review, land, or replace https://github.com/openclaw/openclaw/pull/82868 with a narrow schema/test update that describes configured deliverable provider ids and derives any enum from registered channel plugins.

## Reproduction Assessment

Yes for the source-level issue: current main and v2026.5.12 define `channel` as a bare optional string and spread it into the shared message tool schema. I did not run a live Slack or Discord workspace repro.

## Solution Assessment

Yes for the requested direction: schema metadata plus focused regression coverage is the narrow maintainable fix. Any enum should be derived from registered deliverable channel ids rather than hard-coded.
