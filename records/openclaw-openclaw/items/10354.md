## Summary

Keep open: current main and the latest release still expose the message tool `channel` parameter without schema guidance, and the issue is already paired with an open unmerged fix PR for the narrow description/test path.

## What This Changes

Add provider-name guidance, and possibly enum support, to the message tool `channel` parameter so agents do not pass destination channel or conversation ids there.

## Best Possible Solution

Keep this issue open until the active linked PR is landed or closed; if maintainers accept description-only closure, track registry-derived enum support separately only if it remains needed.

## Reproduction Assessment

Yes. Source inspection of current main and the latest release shows `channel` is still an undescribed optional string while outbound routing rejects unknown explicit channel values without a valid fallback.

## Solution Assessment

Yes for the description portion: TypeBox metadata plus a focused schema assertion is the narrow maintainable repair. Enum support should be a maintainer decision and, if pursued, should be registry-derived rather than hard-coded.
