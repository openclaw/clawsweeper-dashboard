## Summary

Keep open: current main and the latest release still expose the shared message tool `channel` parameter without provider-name schema guidance, and the remaining work is already owned by an open linked fix PR.

## What This Changes

The issue requests description metadata and possible enum guidance for the message tool `channel` parameter so agents pass a messaging provider or channel plugin id rather than a destination conversation id.

## Best Possible Solution

Review and land the linked schema/test PR after deciding the enum scope; keep `channel` optional with a clear provider-name description, and only add enum support if it is registry-derived rather than hard-coded.

## Reproduction Assessment

Yes. Source inspection on current main shows the tool schema accepts `channel` as an undescribed optional string while outbound routing treats explicit values as provider/channel ids and can throw `Unknown channel` without a valid fallback.

## Solution Assessment

Yes for the description part: TypeBox metadata plus a focused schema assertion is the narrowest maintainable fix. Enum support remains a product/API choice because channel ids can include registered plugin ids and should not drift from runtime registration.
