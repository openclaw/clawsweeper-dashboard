## Summary

Keep open: current main and the latest release still expose the shared message tool `channel` parameter as an undescribed optional string, and an open draft implementation already targets this issue with closing syntax. The issue should stay open until that linked PR is landed, closed, or replaced.

## What This Changes

The request is to add provider-id guidance and possibly dynamic enum metadata to the shared message tool `channel` parameter so agents choose providers instead of destination IDs.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/82868 with a narrow schema/test update that adds provider-id guidance and derives any enum from registered deliverable channel ids.

## Reproduction Assessment

Yes for the source-level schema gap: current main and the latest release define `channel` as a bare optional string and spread it into the message tool schema. I did not run a live Slack or Discord workspace repro in this read-only review.

## Solution Assessment

Yes for the description portion: schema metadata plus focused regression coverage is the narrowest maintainable repair. Any enum support should be derived from registered channel/plugin ids rather than hard-coded.
