## Summary

Keep open. Current main still exposes the shared message tool `channel` parameter as a bare optional string with no schema description or enum, while the linked implementation PRs are closed unmerged, so the requested schema guidance remains valid and narrow.

## What This Changes

Request adding schema guidance, and possibly a safe enum, for the shared message tool `channel` parameter so agents pass provider or plugin ids instead of destination channel IDs.

## Best Possible Solution

Land a narrow core schema/test change that keeps `channel` optional, adds a TypeBox description saying it expects a provider or channel plugin id such as `slack` or `discord`, and only adds enum metadata if it is derived from the plugin-aware runtime channel list.

## Reproduction Assessment

Yes. Source inspection against current main reproduces the issue: `createMessageTool()` reaches `buildRoutingSchema()`, and its `channel` property has no `description` or enum metadata.

## Solution Assessment

Yes for the description portion. The narrowest maintainable fix is schema metadata plus a focused assertion; a static enum is only appropriate if it is generated from the same plugin-aware channel ids used at runtime.

Add a TypeBox description in `buildRoutingSchema()` explaining that `channel` is the messaging provider or channel plugin id, for example `slack` or `discord`, not the destination conversation/channel/user id.
