## Summary

Keep open: current main and the latest release still expose the shared message tool `channel` parameter as an undescribed optional string, and two open linked pull requests are already competing implementation paths that should be reviewed before closure.

## What This Changes

The issue requests provider-oriented schema guidance, and possibly an enum, for the message tool `channel` parameter so agents pass provider ids rather than destination channel ids.

## Best Possible Solution

Review and land one maintainer-preferred linked pull request, decide the enum scope explicitly, and close or retarget the duplicate implementation path afterward.

## Reproduction Assessment

Yes for a source-level reproduction: current main exposes `channel` as a bare optional string, and the outbound path still has an `Unknown channel` branch for unrecognized explicit channel values. I did not run a live Slack or Discord workspace reproduction.

## Solution Assessment

Yes for the description portion: adding TypeBox metadata plus focused schema coverage is the narrow maintainable fix. The enum portion needs maintainer direction and should be derived from registered or configured channel/plugin ids if included.
