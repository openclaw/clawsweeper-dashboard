- `impact:message-loss`: Agents can pass a destination id as the channel provider and hit Unknown channel, preventing the intended message operation from being delivered.

## Summary

Keep open: current main and the latest release still expose the message tool channel parameter without schema guidance, and the issue is already paired with an open unmerged closing PR for the narrow schema-description/test fix.

## What This Changes

Request to add provider-name guidance, and possibly a dynamic enum, to the message tool channel parameter so agents do not pass Slack or Discord conversation IDs as provider ids.

## Best Possible Solution

Land or revise the open schema-description PR, keeping channel optional and provider-oriented; if enum support is still desired, derive it from registered channel ids or track it as a separate maintainer-approved follow-up.

## Reproduction Assessment

Yes. Source inspection shows current main exposes channel as an undescribed optional string while outbound routing treats it as provider/channel selection and still throws Unknown channel for invalid explicit values without a valid fallback.

## Solution Assessment

Yes for the description portion: TypeBox metadata plus a focused schema assertion is the narrow maintainable repair. The enum portion is unclear and should be a maintainer decision, preferably derived from registered channel/plugin ids if pursued.
