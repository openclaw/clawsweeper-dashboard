## Summary

Keep this PR open because current main still does not implement non-persistent `/think <level> <body>` chat behavior, and the PR remains the visible implementation candidate. The latest PR diff still has a correctness issue: valid one-shot think messages are intentionally kept classified as control commands, so channel ingress paths that intercept `hasControlCommand` can block them before the new `oneShotThinkLevel` directive logic runs.

## What This Changes

The PR adds one-shot `/think <level> <body>` parsing, a non-persistent `oneShotThinkLevel` directive field, Telegram multiline command-body normalization, xhigh validation coverage, focused tests, and generated Plugin SDK API baseline updates.

## Best Possible Solution

Keep the PR open, but require a revision that separates command authorization/debounce from command interception or otherwise lets validated one-shot `/think <level> <body>` messages reach the normal reply path. The final patch should preserve directive-only tails as commands, keep the new non-persistent directive slot, retain Telegram multiline normalization, and refresh or remove generated baseline churn before landing.

## Reproduction Assessment

Yes. Current main can be reproduced by static path inspection: `/think high explain this` matches `hasControlCommand`, mixed directive text clears `thinkLevel`, and `resolvedThinkLevel` has no one-shot fallback; the PR body also provides a clear session non-pollution scenario.

## Solution Assessment

No. The proposed direction is the right narrow product shape, but the latest diff is not the best implementation because keeping one-shot messages classified as control commands prevents some channels from ever reaching the new reply-path logic; the safer fix is a separate gating predicate or a validated one-shot exception while keeping directive-only tails protected.
