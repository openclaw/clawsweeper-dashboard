## Summary

Keep open: current main still leaves successful paired-device token authentication and paired WebSocket reconnects without durable device-level last-seen updates, and the open closing PR at https://github.com/openclaw/openclaw/pull/81189 should be reviewed before this issue is closed.

## What This Changes

The issue asks for paired-device records to persist creation and last-seen timestamps so operators can identify stale paired clients.

## Best Possible Solution

Land a focused metadata refresh for successful paired auth and reconnects, preserving the current `createdAtMs`/`lastSeenAtMs` contract and existing auth, pairing, and scope gates.

## Reproduction Assessment

Yes. Current main source inspection shows successful `verifyDeviceToken` updates only token `lastUsedAtMs`, and paired reconnects persist display/IP metadata only, so accepted auth or reconnect can leave device-level `lastSeenAtMs` unchanged.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing paired-device metadata seam to set `lastSeenAtMs`/`lastSeenReason` after accepted auth or reconnect, without adding config, schema migration, or new legacy field names.
