## Summary

Keep open: current main has the paired-device timestamp fields and writes creation time on approval, but accepted device-token auth and paired reconnects still do not refresh durable paired-device `lastSeenAtMs`. The open linked fix PR uses closing syntax for this issue, so the issue should remain open until that PR lands or is closed.

## What This Changes

Request that paired-device records populate creation timestamps and refresh last-seen metadata so stale paired clients can be identified.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81189, or an equivalent focused fix, so successful paired-client auth and reconnects update the existing durable last-seen metadata without changing pairing policy or adding a config knob.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: pair a device, authenticate later with an accepted device token or reconnect as an approved paired client, and current main persists token `lastUsedAtMs` or access metadata without updating paired-device `lastSeenAtMs`.

## Solution Assessment

Yes. The narrow maintainable fix is to write the existing last-seen metadata after successful paired-device auth/reconnect, with focused regression coverage and no schema or auth-policy expansion.
