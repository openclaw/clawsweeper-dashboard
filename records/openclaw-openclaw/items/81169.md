## Summary

Keep open: current main has the paired-device timestamp fields and writes creation time on approval, but accepted device-token auth and paired WebSocket reconnects still do not refresh durable device-level last-seen metadata. The linked open PR is the active fix path, so this issue should remain open until that PR or an equivalent patch lands.

## What This Changes

The issue requests durable `createdAt` and `lastSeenAt` metadata for paired devices so operators can identify stale paired clients.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81189 or an equivalent focused patch, then close this issue after merge while preserving the current `*Ms` schema and existing auth/scope gates.

## Reproduction Assessment

Yes. Source inspection of current main shows approval writes creation metadata, but accepted token auth only updates token `lastUsedAtMs` and paired reconnects persist access metadata without `lastSeenAtMs`, so the stale-client last-seen gap is reproducible from the code path.

## Solution Assessment

Yes, but not by closing the issue now. The narrow maintainable solution is to wire successful paired-client auth/reconnect into the existing `updatePairedDeviceMetadata` last-seen fields, which is what the linked open PR attempts without changing auth policy or config.
