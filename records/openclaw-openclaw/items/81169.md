## Summary

Keep open: current main has the paired-device timestamp fields and writes creation time on approval, but the accepted-auth/reconnect path still does not update device-level lastSeen metadata, so the stale-client problem is only partially addressed.

## What This Changes

The issue asks for paired device records to populate creation time on first pairing and refresh last-seen time when paired clients authenticate.

## Best Possible Solution

Keep the current `createdAtMs`/`lastSeenAtMs` store contract, refresh durable last-seen metadata on successful paired-client authentication with regression coverage, and avoid a startup migration or new config knob unless a separate maintainer decision asks for it.

## Reproduction Assessment

Yes, source-reproducible: pair an operator or node device, reconnect with accepted paired-device auth, and current main leaves the paired-device `lastSeenAtMs` unset while only token `lastUsedAtMs` or explicit node-presence events update timestamps.

## Solution Assessment

Yes, the narrow fix is to wire successful paired-client auth/reconnect into the existing `updatePairedDeviceMetadata` last-seen fields, preferably at connection granularity or with bounded throttling rather than changing auth policy or file schema.
