## Summary

## What This Changes

The issue asks for paired-device creation and last-seen timestamps so operators can identify stale paired clients.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/81189; the desired end state is to refresh `lastSeenAtMs`/`lastSeenReason` after successful paired auth or reconnect without changing pairing policy or reviving legacy field names.

Yes. Source inspection gives a high-confidence path: pair a device, authenticate or reconnect with an accepted device token, and current main updates token/access metadata without refreshing durable device-level `lastSeenAtMs`.

## Solution Assessment

Yes. The narrowest maintainable fix is to use the existing paired-device metadata seam after successful auth or reconnect, with focused regression coverage and no schema/config change.
