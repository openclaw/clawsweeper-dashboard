## What This Changes

The issue requests non-null paired-device creation metadata and durable last-seen updates so stale paired clients can be identified.

## Best Possible Solution

Land the focused linked fix after maintainer review and required checks, preserving existing pairing/auth/scope gates while updating durable last-seen metadata on successful accepted auth and reconnect.

Yes, source-reproducible: pair a device, reconnect with accepted device-token auth, and current main updates only token lastUsedAtMs while durable device lastSeenAtMs remains unset. The WebSocket reconnect metadata path likewise omits last-seen fields.

## Solution Assessment

Yes: the narrow maintainable fix is to use the existing lastSeenAtMs/lastSeenReason metadata seam after successful auth/reconnect, without changing pairing approval, token verification, config, or schema.
