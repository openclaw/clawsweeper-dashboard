## Label Justifications

- `P1`: A bundled Mattermost channel callback can accept forged interaction tokens when the attacker knows the bot token and target interaction context.
- `impact:security`: The issue concerns authentication/integrity of Mattermost button callback requests and potential forged system-event dispatch.

## Summary

Keep open. Current main and latest release v2026.6.1 still derive the Mattermost interaction signing secret from the bot token with a public HMAC constant, and the linked fix PR is open/unmerged and needs maintainer security/product review before this issue can close.

## What This Changes

The issue reports a Mattermost interaction-forgery risk caused by deriving `_token` signing secrets from the bot token with a hardcoded public HMAC key.

## Best Possible Solution

Replace the bot-token-derived signing root with a separate per-account Mattermost interaction secret, update docs/tests and any needed doctor or credential migration path, then close this issue after the linked PR or a maintainer-approved replacement merges.

## Reproduction Assessment

Yes, source-reproducible: `setInteractionSecret(accountId, botToken)` stores `HMAC-SHA256(public constant, botToken)`, and the handler accepts a matching `_token` before dispatching the interaction. I did not run a live Mattermost callback, but current tests and docs confirm the deterministic contract.

## Solution Assessment

Yes on direction: a separate per-account interaction secret is the right security shape. The exact persistence/config and upgrade path is still a maintainer decision because current docs expose the bot-token derivation to external scripts.
