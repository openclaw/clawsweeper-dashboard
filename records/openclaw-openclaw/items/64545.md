## Label Justifications

- `P1`: A forgeable Mattermost callback token can trigger approval-style channel actions when an attacker has bot-token access, so this is an urgent channel security workflow issue.
- `impact:security`: The report concerns the authorization boundary for Mattermost interaction callbacks and credential-derived signing secrets.

## Summary

Current main and the latest release still derive Mattermost interaction signing from the bot token with a public HMAC key, and the linked closing PR is open rather than merged, so this security issue remains necessary.

## What This Changes

The issue requests replacing Mattermost's bot-token-derived interaction signing secret with a separate durable per-account secret so button interaction tokens cannot be forged from the bot token alone.

## Best Possible Solution

Land a maintainer-approved version of https://github.com/openclaw/openclaw/pull/64546, or a replacement, that uses a separate durable per-account Mattermost interaction secret with upgrade-safe behavior and regression coverage.

## Reproduction Assessment

Yes, source reproduction is high-confidence: current main still derives the Mattermost interaction signing root from the bot token with a public HMAC key, and the verification path accepts tokens made from that derived secret. I did not run a live Mattermost server reproduction in this read-only review.

## Solution Assessment

Yes on direction, but not yet on merge-ready implementation: a separate per-account secret is the right security boundary, while the linked PR still needs a durable storage and upgrade contract before this issue can close.
