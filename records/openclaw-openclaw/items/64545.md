## Label Justifications

- `P1`: A forgeable Mattermost callback token can trigger approval-style channel actions when an attacker has bot-token access, so this is an urgent channel security workflow issue.
- `impact:security`: The issue concerns the authorization boundary for Mattermost interaction callbacks and credential-derived signing secrets.

## Summary

Current main and the latest release still derive the Mattermost interaction signing secret from the bot token using a public HMAC key, while the linked fix PR is open and not merge-ready, so this security issue remains necessary.

## What This Changes

Fix Mattermost interaction callback signing so `_token` values cannot be forged from the bot token and the public HMAC derivation key.

## Best Possible Solution

Land a maintainer-approved version of https://github.com/openclaw/openclaw/pull/64546, or a replacement, that uses a durable per-account Mattermost interaction secret with setup/doctor/docs coverage, upgrade behavior, restart or cross-process proof, and old-forgery rejection tests.

## Reproduction Assessment

Yes. Source inspection shows current main still derives Mattermost interaction signing from the bot token with a public HMAC key, and the verification path accepts tokens produced from that derived secret; I did not run a live Mattermost server reproduction in this read-only review.

## Solution Assessment

Yes on direction, but not yet on an accepted implementation. A separate durable per-account interaction secret is the right security boundary, while the active PR still needs maintainer security/product approval for persistence and upgrade behavior.
