## Summary

Keep open: current main does not yet provide the audio-scoped config/local-auth behavior, and the PR is useful, but it expands a security-sensitive private-network request surface that needs maintainer/security sequencing before merge.

## What This Changes

The PR adds `tools.media.audio.request.allowPrivateNetwork`, per-model audio request config, OpenAI-compatible local audio no-auth handling, docs, generated config metadata, changelog, and regression tests.

## Best Possible Solution

Land this PR or an equivalent narrow patch after SSRF hardening or explicit security approval, preserving audio-only opt-in, image/video rejection, explicit request-auth precedence, and no upstream OpenAI key reuse for local OpenAI-compatible audio.

## Reproduction Assessment

Yes. Source inspection shows current main rejects or strips audio-level `allowPrivateNetwork` and always sends OpenAI-compatible audio Authorization; I did not run a live repro in this read-only review.

## Solution Assessment

Yes, with security sequencing. The PR uses the existing provider request contract and scopes the new config to audio, but the private-network opt-in should merge only with or after the related SSRF hostname-policy hardening or explicit maintainer/security approval.
