## Summary

Keep this PR open: the audio config/auth direction is useful and source-reproducible, but it intentionally widens an audio private-network request surface while the related SSRF hostname/metadata hardening remains unresolved.

## What This Changes

The PR adds audio-scoped `allowPrivateNetwork` request config, per-model audio request forwarding, OpenAI-compatible local-audio no-auth handling, docs/config metadata updates, changelog text, and regression tests.

## Best Possible Solution

Land the audio-specific config/auth support only after or together with the SSRF metadata/hostname hardening, with regression coverage proving audio private endpoints work while metadata/internal destinations remain blocked.

## Reproduction Assessment

Yes. Current main source shows audio `allowPrivateNetwork` is rejected or stripped and OpenAI-compatible audio always sends provider auth; the PR body also includes after-fix terminal output from a local WhisperLive transcription run.

## Solution Assessment

No, not as-is. The implementation direction is reasonable, but the new private-network config surface should land with equivalent SSRF hostname/metadata hardening and a rebase that preserves current exact-origin protections.
