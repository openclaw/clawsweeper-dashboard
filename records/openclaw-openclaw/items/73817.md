## Summary

Keep open. Current main still lacks the audio-scoped config/auth behavior, and the PR addresses the gap with a narrow schema/runtime/docs/test change. I did not find a blocking correctness issue in the patch, but the touched surface is SSRF/auth-sensitive media networking, so it should get maintainer review and exact-head validation before merge.

## What This Changes

The PR adds audio-only `tools.media.audio.request.allowPrivateNetwork` and per-model audio request config, threads those overrides into media provider execution, suppresses synthetic local OpenAI-compatible audio auth, and updates docs, metadata, changelog, and tests.

## Best Possible Solution

Review and land this PR, or an equivalent narrow patch, preserving the audio-only private-network opt-in, image/video rejection, explicit `request.auth` precedence, and the default SSRF block for private-network media requests.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: the schema rejects audio `request.allowPrivateNetwork`, audio-level request sanitization strips that field, and OpenAI-compatible audio always builds provider-default Authorization.

## Solution Assessment

Yes. The proposed direction uses the existing model-provider request contract only for audio media config and keeps the auth change scoped to the existing synthetic local marker; I did not find a narrower maintainable fix.
