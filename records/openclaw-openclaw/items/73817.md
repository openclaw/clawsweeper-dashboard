The PR adds audio-scoped `allowPrivateNetwork` request config, forwards audio/per-model request overrides, changes OpenAI-compatible local audio auth handling, and updates schema, docs, tests, and changelog.

## Best Possible Solution

Land the audio-specific config/auth support after or together with the SSRF metadata hardening, with regression coverage for private audio endpoints and blocked metadata/internal destinations.

Yes. Source inspection on current main shows audio `allowPrivateNetwork` is rejected or stripped and OpenAI-compatible audio always sends provider auth; the PR body also provides after-fix live CLI output from a local WhisperLive transcription run.

## Solution Assessment

No, not as-is. The implementation direction is reasonable, but exposing the new audio private-network config should be sequenced with equivalent SSRF hostname/metadata hardening and a rebase that preserves the current trust boundary.
