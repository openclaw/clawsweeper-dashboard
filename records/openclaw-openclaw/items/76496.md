## Summary

Keep open. Current main still slices the discovered sessions before loading usage and accumulating totals, so the reported under-count remains present; there is an open broader PR that includes the central fix, but it is not merged and has review/proof blockers.

## What This Changes

The issue reports that sessions.usage aggregate totals and breakdowns should cover all matching sessions while limit only caps the returned sessions array.

## Best Possible Solution

Land a narrow gateway fix, or split it from the broader open PR, so all totals and aggregate breakdowns iterate the full matched session set while only the returned sessions list is limited.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create more matching sessions than the requested limit and current main can only aggregate the sliced limitedEntries loop.

## Solution Assessment

Yes. The narrow maintainable solution is to decouple aggregate computation from response paging in sessions.usage and add a regression test where changing limit changes returned rows but not totals.
