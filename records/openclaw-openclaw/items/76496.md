## Summary

Keep open. Current main still computes `sessions.usage` totals and aggregate breakdowns only after applying the per-session `limit`, so the reported under-count is present in source and is narrow enough for a focused repair.

## What This Changes

The issue asks for `sessions.usage` aggregate totals and breakdowns to cover all matching sessions while `limit` only caps the returned `sessions` detail array.

## Best Possible Solution

Compute totals and every aggregate over all discovered entries in the date range, build the returned `sessions` array only for the limited subset, and add a regression test where changing `limit` changes detail count but not totals.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create more matching sessions than `limit` and the current handler can only aggregate the sliced `limitedEntries`. I did not run a live gateway call because this was a read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is in `sessions.usage`: decouple aggregate computation from response-size limiting without changing the API shape or adding configuration.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76496. In `sessions.usage`, make `limit` cap only the returned `sessions` detail array while `totals` and `aggregates` are computed from every discovered session matching the date range. Keep the API shape unchanged, do not add a config option, and avoid reading the same visible session twice. Add a regression test in `src/gateway/server-methods/usage.sessions-usage.test.ts` with more discovered sessions than the limit, asserting that `sessions.length` follows the limit while `totals.totalCost`, tokens/messages, and at least one aggregate include all sessions. Add a user-facing `CHANGELOG.md` fix entry unless maintainers decide this is still unreleased-only.
