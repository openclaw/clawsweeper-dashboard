## Summary

Keep open. Current main still slices the matched session list before loading usage and accumulating totals, so the reported under-count remains present; the related open PR includes the central fix but is not merged and is currently conflicting with review/proof blockers.

## What This Changes

The issue asks for `sessions.usage` totals and aggregate breakdowns to include all matching sessions while `limit` only caps returned session rows.

## Best Possible Solution

Land or split a narrow gateway fix so `sessions.usage` computes totals and aggregate breakdowns over all matched sessions while applying `limit` only to the returned `sessions` array, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: create more matching sessions than `limit`; the handler slices `mergedEntries` before the only aggregation loop, so totals can only include the returned subset.

## Solution Assessment

Yes. The narrow maintainable fix is to decouple aggregate computation from response paging and add a regression where changing `limit` changes returned rows but not totals.
