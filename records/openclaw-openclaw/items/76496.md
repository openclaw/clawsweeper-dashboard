## Summary

Keep open. Current main still aggregates `sessions.usage` totals from the limited session slice, and the issue is already paired with open PR #76524, so it should stay open until that fix lands or is replaced.

## What This Changes

The report asks for `sessions.usage` to use `limit` only for returned session details while totals and aggregate breakdowns include all matching sessions.

## Best Possible Solution

Land or replace the linked narrow fix so all aggregate totals are computed over every matching session while `limit` only bounds the returned `sessions` entries, with a regression test proving totals stay stable as `limit` changes.

## Reproduction Assessment

Yes. Source inspection on current main shows `limit` defaults to 50, `mergedEntries` is sliced before the only aggregation loop, and a dataset with more matching sessions than `limit` will undercount without needing a live gateway run.

## Solution Assessment

Yes. The narrow maintainable fix is to decouple aggregate computation from response-size limiting inside `sessions.usage`, preserving the API shape and adding focused regression coverage.
