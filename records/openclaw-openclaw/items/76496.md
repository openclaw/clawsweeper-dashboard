## Summary

Keep this open: current `main` still applies `limit` before loading and aggregating `sessions.usage`, so the reported under-count remains valid. A prior exact fix PR was closed unmerged, and the related stale PR points back to this issue as the canonical remaining work.

## What This Changes

The issue reports that `sessions.usage` aggregate totals are silently capped by the `limit` parameter instead of only limiting the returned session rows.

## Best Possible Solution

Implement the narrow gateway fix: aggregate totals and aggregate buckets from all matched entries, while using `limit` only for the returned `sessions` rows and adding a focused regression test.

## Reproduction Assessment

Yes from source inspection: current `sessions.usage` slices `mergedEntries` into `limitedEntries` before loading and aggregating usage. I did not run a live gateway repro in this read-only review.

## Solution Assessment

Yes. The best fix is to separate aggregate scope from response pagination in `src/gateway/server-methods/usage.ts`, not to add a new flag or change the API contract.
