## Summary

Keep open: current main and the latest release still build `sessions.usage` totals and aggregate buckets only from `limitedEntries`, so increasing `limit` can still change aggregate cost/token/message totals. The closed related usage-dashboard PR was not merged and explicitly left this issue as the canonical remaining work.

## What This Changes

The issue requests that `sessions.usage` use `limit` only for returned session rows, not for aggregate totals and breakdowns.

## Best Possible Solution

Compute `sessions.usage` totals and aggregate buckets from the full matched session set while using `limit` only for the returned `sessions` array, with focused regression coverage for `limit` smaller than the match count.

## Reproduction Assessment

Yes from source. A focused repro can mock more discovered sessions than `limit`, return distinct usage totals, and assert that `sessions` is capped while `totals` and `aggregates` include every matched session.

## Solution Assessment

Yes, the requested direction is the best fix if implemented through the existing cache/background usage-loading path. It should not add a config knob or change dashboard semantics beyond making aggregate scope independent from returned row count.
