## Summary

Keep open: current main still has the explicit loopback preflight fallback that can resolve `gateway.bind="loopback"` to `0.0.0.0`, and the runtime guard still throws the reported startup error when that happens. The linked target-repo fix PR was closed unmerged, and no current target-repo PR or shipped release appears to own this exact fix.

## What This Changes

The report asks OpenClaw to keep macOS local Gateway `gateway.bind="loopback"` on `127.0.0.1` instead of resolving to `0.0.0.0` and failing startup after a loopback preflight miss.

## Best Possible Solution

Keep explicit `gateway.bind="loopback"` pinned to `127.0.0.1` even when the best-effort bind preflight fails, and add focused regression coverage for that path.

## Reproduction Assessment

Yes, source-level: if the `127.0.0.1` preflight returns false, current main resolves explicit `gateway.bind="loopback"` to `0.0.0.0`, and `resolveGatewayRuntimeConfig` rejects it with the reported error. I did not run a live macOS foreground Gateway repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make explicit loopback return `127.0.0.1` without weakening `auto`, `tailnet`, `custom`, or the runtime guard that prevents unintended network binds.
