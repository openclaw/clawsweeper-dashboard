## Summary

Keep open: current main and the latest release still allow explicit `gateway.bind: "loopback"` to fall back to `0.0.0.0` after a failed preflight, and the open closing pull request at https://github.com/openclaw/openclaw/pull/81306 already owns the exact resolver/test fix.

## What This Changes

The issue reports that macOS Gateway startup can fail when explicit `gateway.bind: "loopback"` resolves to `0.0.0.0` and trips the non-loopback guard.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81306, or an equivalent replacement, so explicit loopback always resolves to `127.0.0.1` while `auto`, `tailnet`, `custom`, and the fail-closed guard keep their current semantics.

## Reproduction Assessment

Yes, source-reproducible. If `canBindToHost("127.0.0.1")` returns false, current main returns `0.0.0.0` for explicit loopback and the runtime guard throws the reported error.

## Solution Assessment

Yes. The narrow maintainable fix is to pin explicit loopback to `127.0.0.1` and add failed-preflight regression coverage without weakening the fallback behavior for other bind modes.
