## Summary

Keep open: current main and the latest release still contain the explicit-loopback preflight fallback, and the issue is now paired with an open closing pull request that targets this exact bug. Repository policy says this issue should stay open until that pull request lands or is replaced.

## What This Changes

The issue reports that explicit macOS `gateway.bind: "loopback"` can resolve through the Gateway bind resolver to `0.0.0.0`, causing the runtime guard to reject startup.

## Best Possible Solution

Land the focused resolver/test fix from the open closing pull request, or an equivalent replacement, so explicit `loopback` always resolves to `127.0.0.1` while `auto`, `tailnet`, `custom`, and the fail-closed guard keep their current semantics.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: if the explicit loopback preflight returns false, current main resolves to `0.0.0.0` and `resolveGatewayRuntimeConfig` throws the reported non-loopback error.

## Solution Assessment

Yes. The narrow maintainable fix is to pin explicit loopback to `127.0.0.1` and add regression coverage, without weakening the safety guard or changing non-loopback bind modes.
