Keep open: current `main` and the latest release still contain the explicit-loopback fallback to `0.0.0.0`, and live GitHub shows an open closing pull request for this exact bug. Repository policy should keep this issue open until that PR lands or is replaced.

## What This Changes

The issue reports that explicit `gateway.bind: "loopback"` on macOS can resolve to `0.0.0.0`, causing Gateway startup to fail under the loopback safety guard.

## Best Possible Solution

Land the focused resolver/test fix from https://github.com/openclaw/openclaw/pull/81306, or an equivalent replacement, so explicit `loopback` always resolves to `127.0.0.1` while preserving `auto`, `tailnet`, `custom`, and fail-closed guard semantics.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: if the explicit loopback preflight returns false, current `main` resolves to `0.0.0.0` and `resolveGatewayRuntimeConfig` throws the reported non-loopback error.

## Solution Assessment

Yes. The narrow maintainable fix is to pin explicit loopback to `127.0.0.1` and add regression coverage, without weakening the safety guard or changing non-loopback bind modes.
