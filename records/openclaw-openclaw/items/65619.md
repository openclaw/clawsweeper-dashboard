## Summary

Keep open: current main still contains the source-level path where explicit loopback bind can fall back to `0.0.0.0`, and the fail-closed runtime guard then throws the reported startup error. The linked target-repo fix PR was closed unmerged, and the latest shipped release still has the affected resolver behavior.

## What This Changes

The issue reports a macOS regression where explicit `gateway.bind="loopback"` can resolve to `0.0.0.0` after a failed preflight probe, causing Gateway startup to fail.

## Best Possible Solution

Pin explicit loopback binds to `127.0.0.1` regardless of preflight result, while preserving `auto`, container, tailnet, and custom-bind behavior with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes, at source level: make the loopback preflight return false, then `resolveGatewayBindHost("loopback")` returns `0.0.0.0`, and `resolveGatewayRuntimeConfig` rejects that non-loopback result. I did not run a live macOS Gateway foreground reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to remove the preflight fallback for explicit loopback and keep the security guard and other bind modes intact.
