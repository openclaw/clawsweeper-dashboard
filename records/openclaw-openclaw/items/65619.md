## Summary

Keep this open: current main still lets explicit `gateway.bind="loopback"` fall back to `0.0.0.0` when the loopback preflight fails, and the runtime guard then throws the same non-loopback bind error reported here. The linked target-repo fix PR is closed unmerged, while the later open copy is only in a fork.

## What This Changes

The issue reports macOS local Gateway startup failing because explicit loopback bind can resolve to `0.0.0.0` after a failed preflight probe and then abort.

## Best Possible Solution

Keep explicit `gateway.bind="loopback"` pinned to `127.0.0.1` when the loopback preflight probe fails, while leaving `auto`, `tailnet`, `custom`, and the runtime fail-closed guard intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make the loopback preflight return false, then resolve explicit `gateway.bind="loopback"`; current main returns `0.0.0.0`, and runtime config rejects that with the reported error.

## Solution Assessment

Yes. The narrow resolver fix is the best maintainable direction: keep the safety guard fail-closed, but stop explicit loopback from being downgraded to a network bind before the guard runs.
