## Summary

Keep this open: current main still downgrades explicit `gateway.bind="loopback"` to `0.0.0.0` when the loopback preflight bind probe fails, and the runtime guard then throws the reported startup error. The linked target-repo fix PR was closed without merging, so there is still a narrow gateway resolver bug to fix.

## What This Changes

Fix macOS gateway loopback mode so an explicit `gateway.bind="loopback"` does not resolve to `0.0.0.0` and fail startup after a failed loopback preflight probe.

## Best Possible Solution

Keep explicit `gateway.bind="loopback"` pinned to `127.0.0.1` when the preflight probe fails, while preserving the fail-closed runtime guard and existing `auto`, `tailnet`, `lan`, and `custom` semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make `canBindToHost("127.0.0.1")` return false for explicit loopback mode; current main then returns `0.0.0.0`, and runtime config rejects it with the reported error.

## Solution Assessment

Yes. The narrow resolver fix is the best maintainable direction: prevent explicit loopback from being downgraded before the guard runs, rather than relaxing the guard or changing service/probe behavior broadly.
