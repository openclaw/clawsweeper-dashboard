## What This Changes

The issue reports that `gateway.bind: "loopback"` on macOS can resolve through `0.0.0.0` and make Gateway startup/health fail with a 1006 probe error.

## Best Possible Solution

Review and land the narrow resolver/test fix in https://github.com/openclaw/openclaw/pull/81306, then close this issue from the merge once explicit loopback is pinned to `127.0.0.1`.

## Reproduction Assessment

Yes at source level: current `main` returns `0.0.0.0` for explicit `loopback` when the `127.0.0.1` preflight fails, and the runtime config guard rejects that host with the reported error family. I did not run a live macOS foreground Gateway reproduction in this read-only review.

## Solution Assessment

Yes, the linked PR's approach is the narrow maintainable direction: pin explicit `loopback` to `127.0.0.1`, leave `auto`, `tailnet`, `custom`, and `lan` fallback behavior alone, and keep the safety guard in place.
