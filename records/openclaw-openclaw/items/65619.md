## Summary

Keep open: current main still contains the explicit-loopback preflight fallback that can resolve to `0.0.0.0`, and an open closing pull request already targets this exact bug. Repository policy should keep this issue open until that pull request lands or is replaced.

## What This Changes

Report that macOS Gateway with `gateway.bind: "loopback"` can resolve to `0.0.0.0` and fail startup/health through the non-loopback guard.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81306, or an equivalent replacement, so explicit `loopback` always resolves to `127.0.0.1` while other bind modes and the fail-closed guard keep their current semantics.

## Reproduction Assessment

Yes. Source inspection shows that if the current-main `127.0.0.1` preflight returns false, explicit loopback resolves to `0.0.0.0` and the runtime guard throws the reported error.

## Solution Assessment

Yes. The narrow maintainable fix is to pin explicit loopback to `127.0.0.1` and add regression coverage, without weakening `auto`, `tailnet`, `custom`, or the non-loopback safety guard.
