## Summary

Keep open: current main and the latest release still contain the explicit-loopback preflight fallback that can resolve to `0.0.0.0`, and an open closing pull request already targets this exact gateway bug.

## What This Changes

The issue reports that macOS Gateway startup can fail when explicit `gateway.bind: "loopback"` falls back to `0.0.0.0` after a failed `127.0.0.1` bind probe.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81306, or an equivalent replacement, so explicit `loopback` always resolves to `127.0.0.1` while other bind modes and the fail-closed guard keep their current semantics.

## Reproduction Assessment

Yes. Source inspection shows that when the `127.0.0.1` preflight returns false, explicit loopback resolves to `0.0.0.0` and the runtime guard throws the reporter's exact error.

## Solution Assessment

Yes. The narrow maintainable fix is to pin explicit `loopback` to `127.0.0.1` and add regression coverage for the failed-preflight branch, without weakening `auto`, `tailnet`, `custom`, or the safety guard.
