## Summary

Keep this open. Current `main` and the latest release still contain the reported explicit-loopback failed-preflight path: `resolveGatewayBindHost("loopback")` can still return `0.0.0.0`, and runtime config still rejects that as the same non-loopback bind failure reported here. The target-repo fix PR linked in the discussion is closed unmerged; the later open copy is in a fork, not a mergeable target-repo PR.

## What This Changes

The issue asks for macOS local Gateway startup to keep `gateway.bind="loopback"` on a loopback address instead of resolving to `0.0.0.0` and aborting with a non-loopback bind error after a failed preflight probe.

## Best Possible Solution

Keep explicit `gateway.bind="loopback"` pinned to `127.0.0.1` when the loopback preflight probe fails, leave `auto`, `tailnet`, and `custom` fallback policy intact, and add focused regression coverage plus a user-facing changelog credit.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make the loopback preflight return false, then resolve explicit `gateway.bind="loopback"`; current `main` returns `0.0.0.0`, and `resolveGatewayRuntimeConfig` rejects that with the reported error.

## Solution Assessment

Yes. The narrow resolver fix is the best maintainable direction: keep the runtime safety guard fail-closed, but stop explicit loopback from being downgraded to a network bind before the guard runs.
