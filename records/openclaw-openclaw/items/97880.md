## Summary

Keep open for maintainer judgment: the reported behavior is real on current main, but it is also the currently shipped explicit-selection policy for key-free managed web_search providers rather than a straightforward bug.

## What This Changes

The issue requests that enabled keyless DuckDuckGo become an implicit web_search fallback, or that OpenClaw warn/configure users when DuckDuckGo is enabled but not selected.

## Best Possible Solution

Keep keyless providers explicit unless maintainers deliberately reverse that policy; if the UX gap remains, add diagnostics/setup guidance that points users to select DuckDuckGo or fold the broader no-selected-provider warning into the related diagnostics issue.

## Reproduction Assessment

Yes, at source level: current main explicitly rejects implicit keyless provider selection, and tests assert no keyless web_search provider is selected when no provider is configured. I did not run a live cron job in this read-only review.

## Solution Assessment

No for the preferred auto-select fix without maintainer approval: it reverses the shipped explicit-selection policy from https://github.com/openclaw/openclaw/pull/93616. A safer path is explicit opt-in plus diagnostics or setup guidance for the no-selected-provider case.
