## Summary

Keep open: the regression is source-reproducible on current main because Ollama is startup-lazy and its lightweight provider-policy artifact does not expose the thinking profile that includes `max`, so validation can still fall back to base levels.

## What This Changes

The issue requests that `/think max` work for `ollama/deepseek-v4-pro:cloud` without forcing the Ollama plugin to activate at startup.

## Best Possible Solution

Keep Ollama startup-lazy and add a startup-safe Ollama `resolveThinkingProfile` policy artifact, with regression coverage proving `max` is accepted when the full plugin registry is not active.

## Reproduction Assessment

Yes, source-reproducible: current main can fall through to the base thinking levels when Ollama is not active because the lightweight Ollama policy artifact lacks `resolveThinkingProfile`. I did not run a live Ollama/Fedora session.

## Solution Assessment

Yes, the narrow maintainable fix is to expose the existing Ollama thinking profile through the lightweight provider-policy artifact; changing `activation.onStartup` to true would mask the issue and weaken the startup-lazy boundary.
