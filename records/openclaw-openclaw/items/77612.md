## Summary

Keep open: current main still has a source-visible path where Ollama's runtime thinking profile is unavailable unless the plugin has been activated, so `/think max` can fall back to the base levels despite the Ollama docs promising max for thinking-capable native models.

## What This Changes

The issue reports that `ollama/deepseek-v4-pro:cloud` rejects `/think max` in v2026.5.3-1 unless the Ollama plugin is forced to start at gateway startup.

## Best Possible Solution

Expose Ollama's thinking profile through a lightweight provider-policy surface or another lazy, provider-owned hook so `max` is available without forcing full Ollama startup activation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection and the reporter's logs: with no active Ollama provider, the support check falls back to base thinking levels, while the max-capable profile exists only on the registered runtime provider.

## Solution Assessment

Is this the best way to solve the issue? No for the workaround: setting Ollama to startup activation would work but broadens runtime activation, while a lightweight provider-owned thinking policy fits the existing lazy plugin boundary better.

## Repair Work Prompt

Fix the Ollama `/think max` regression reported at https://github.com/openclaw/openclaw/issues/77612. Current main lets `ollama/deepseek-v4-pro:cloud` fall back to base thinking levels when the Ollama plugin is enabled but not active in the runtime registry; the runtime provider profile supports `max`, but `extensions/ollama/provider-policy-api.ts` does not expose that profile on the lightweight fallback path. Implement a provider-owned, lazy solution so reasoning-capable Ollama models expose `off, low, medium, high, max` without setting `activation.onStartup` to true. Prefer a lightweight `resolveThinkingProfile` public policy artifact or a small shared helper used by both `extensions/ollama/index.ts` and `extensions/ollama/provider-policy-api.ts`; preserve plugin laziness and do not add a config option. Add regression coverage for `resolveThinkingProfile`/`isThinkingLevelSupported` with an inactive runtime path and a catalog entry for `ollama/deepseek-v4-pro:cloud` or equivalent reasoning-capable Ollama model. Update `CHANGELOG.md` because this is user-facing. Related context: #71584 previously fixed max support once the provider profile is registered.
