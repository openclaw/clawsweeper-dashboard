## Summary

Keep open. The branch is a plausible Telegram liveness fix, but it is not mergeable yet because it uses undeclared session fields, regresses suppressed error delivery, and still lacks proof of the visible Telegram behavior it changes.

## What This Changes

This PR adds persisted reply-turn state, a Telegram slow-turn progress acknowledgement with an environment rollback switch, status reporting for stale or failed turns, diagnostic stale-turn recovery, and focused tests.

## Best Possible Solution

Finish the typed session contract, preserve visible error and progress semantics, then prove the direct/forum Telegram acknowledgement and stale-turn recovery in a real Telegram run before aligning or landing it with the broader liveness design.

## Reproduction Assessment

No high-confidence live reproduction of the original Telegram stall or after-fix chat behavior is present. Source inspection does provide a high-confidence reproduction path for the patch defects: typechecking the PR would hit undeclared replyTurn* SessionEntry fields, and the changed suppression guard drops visible text-only errors.

## Solution Assessment

No, not yet. The direction may be useful, but the patch needs the session contract fix, preserved error and progress delivery semantics, and real Telegram proof before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Declare the reply-turn session fields:** `src/status/status-text.ts:179-183`
  - body: The new status, dispatch, and recovery code reads and writes replyTurnState, replyTurnStartedAt, replyTurnUpdatedAt, replyTurnRunId, and replyTurnLastError as SessionEntry fields, but the PR never adds them to the SessionEntry contract. updateSessionStoreEntry takes Partial<SessionEntry>, so this fails typecheck before the runtime path can run.
  - confidence: 0.93
- **[P2] Keep error payloads visible when suppressing progress:** `src/auto-reply/reply/dispatch-from-config.ts:1501-1502`
  - body: This drops the existing isError exception, so when suppressDefaultToolProgressMessages is active a text-only tool error is swallowed the same as routine progress. The previous condition deliberately let errors through while suppressing normal tool summaries.
  - confidence: 0.78
- **[P2] Only count actually delivered progress as visible:** `src/auto-reply/reply/dispatch-from-config.ts:1490`
  - body: The PR marks every normalized tool payload as visible before the suppression guard runs. With suppressDefaultToolProgressMessages enabled, a text-only progress update can prevent the long-turn acknowledgement even though no Telegram message was sent, which undercuts the silent-turn case this PR is trying to fix.
  - confidence: 0.72
