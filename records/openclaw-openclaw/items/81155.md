## Summary

This recent draft PR should stay open for author/maintainer follow-up, but it is not merge-ready: the diff uses new reply-turn session fields without extending the typed session contract, changes suppressed tool-error delivery semantics, can mark suppressed progress as visible before anything is actually sent, and still lacks real Telegram proof of the user-visible behavior.

## What This Changes

The branch adds persisted reply-turn state, a Telegram long-turn progress acknowledgement with an environment rollback switch, stale/failed turn status and recovery handling, and targeted tests.

## Best Possible Solution

Finish the session contract, mark visible outbound only after real delivery, preserve error payload visibility, and provide redacted real Telegram proof before deciding whether this Telegram-specific recovery path should land.

## Reproduction Assessment

No high-confidence live reproduction of the original Telegram stall or after-fix chat behavior is present. Source inspection does provide a high-confidence reproduction path for the PR defects through the diff and current SessionEntry contract.

## Solution Assessment

No, not yet. The direction may be useful, but the current patch is not the best maintainable solution until it fixes the typed session contract, preserves error delivery, avoids suppressing its own acknowledgement, and includes real Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Declare the reply-turn session fields:** `src/status/status-text.ts:178-180`
  - body: The new status, dispatch, and recovery code reads and writes `replyTurnState`, `replyTurnStartedAt`, `replyTurnUpdatedAt`, `replyTurnRunId`, and `replyTurnLastError`, but the branch never adds them to `SessionEntry`. Since `updateSessionStoreEntry` is typed around `Partial<SessionEntry>`, this should fail typecheck before the runtime path can run.
  - confidence: 0.94
- **[P2] Mark progress visible only after it is sent:** `src/auto-reply/reply/dispatch-from-config.ts:1487`
  - body: This marks a text-only tool result as visible before the suppression branch decides whether to deliver it. With `suppressDefaultToolProgressMessages` enabled, a suppressed routine tool result can set `visibleOutboundAttempted`, causing the long-turn timer to skip the Telegram acknowledgement even though the user saw nothing.
  - confidence: 0.84
- **[P2] Keep error payloads visible when suppressing progress:** `src/auto-reply/reply/dispatch-from-config.ts:1498`
  - body: This removes the existing `isError` exception, so a text-only tool error is swallowed the same as routine progress when default tool-progress messages are suppressed. Current behavior deliberately lets those errors through for direct and preview-suppressed flows.
  - confidence: 0.8
