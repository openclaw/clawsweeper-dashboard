## Summary

Keep open: the timeout/fence approach matches the linked session-stall bug, but the hard 120000 ms default for all existing background context-engine turn maintenance is compatibility-sensitive and needs maintainer approval before merge.

## What This Changes

The PR adds a configurable per-task timeout and late-side-effect fence for deferred context-engine turn maintenance, throttles repeated session.stalled warnings, updates docs/config metadata, and adds regression tests.

## Best Possible Solution

Land the timeout/fence only after maintainers approve upgrade-safe default semantics and the operator docs/tests cover how long-running engines should opt out or raise the budget.

## Reproduction Assessment

Yes at source level: current main enqueues deferred turn maintenance without taskTimeoutMs, and the linked issue supplies production stall logs for queued user work behind a wedged maintenance run. I did not run the live lock-contention scenario in this read-only review.

## Solution Assessment

Mostly yes: using the existing command-lane timeout seam plus a late-side-effect fence is the narrow core repair. The unresolved part is whether the hard default is the best upgrade-safe policy for existing background engines.
