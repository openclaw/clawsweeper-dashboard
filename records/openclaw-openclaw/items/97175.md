## Label Justifications

- `P1`: The PR targets an interactive session stall where queued user messages can wait minutes behind stuck maintenance, and the current patch has a blocking session-state risk.
- `merge-risk: 🚨 compatibility`: The new default can time out existing long-running background maintenance after 120s unless maintainers accept and document the upgrade behavior.
- `merge-risk: 🚨 session-state`: The timeout path clears the active maintenance run while the original worker can still perform late transcript or task-state side effects.

## Summary

Keep open: this targets a real linked session-stall bug, but the patch is not merge-ready because the timeout releases the maintenance lane while the original worker can keep running and later mutate transcript state; the new config surface also needs public docs and real runtime proof.

## What This Changes

The branch adds a configurable 120s `taskTimeoutMs` to deferred context-engine turn-maintenance enqueues, cancels the task record on timeout, throttles repeated `session.stalled` diagnostics, and adds unit coverage plus config schema metadata.

## Best Possible Solution

Keep the command-lane timeout only with an abort or generation fence that prevents late maintenance side effects after timeout, document the new timeout knob, and require redacted runtime proof for the linked stall before merge.

## Reproduction Assessment

Yes at source level: current main enqueues deferred turn maintenance without task-timeout options while the command queue supports them. I did not run a live lossless-claw lock-contention reproduction, so the exact production stall remains issue-log-backed rather than locally reproduced here.

## Solution Assessment

No as submitted: using the existing lane timeout is a plausible mechanism, but releasing the lane without aborting or fencing the worker is not the safest fix. The maintainable path is timeout plus late-side-effect fencing, public docs for the config surface, and real runtime proof.
