## Label Justifications

- `P1`: The linked report describes an interactive session where queued user messages can wait minutes behind wedged background maintenance.
- `merge-risk: 🚨 compatibility`: The new default timeout can stop existing background context-engine maintenance that previously ran longer than 120 seconds.
- `merge-risk: 🚨 session-state`: The PR intentionally fences late transcript rewrite and task terminal writes after timeout, changing how session maintenance state is allowed to complete.

## Summary

Keep open: the PR is a plausible and well-proven fix for the source-visible maintenance-lane stall, but it is still an active merge candidate rather than cleanup-close material; maintainers should explicitly accept the new default timeout upgrade behavior and the small config-doc gap should be fixed before landing.

## What This Changes

The PR adds a configurable per-task timeout for deferred context-engine turn maintenance, fences late maintenance side effects, throttles repeated stalled-session diagnostics, updates config schema/help, and adds focused tests/docs.

## Best Possible Solution

Adopt the timeout-and-fence approach after maintainers choose the upgrade semantics for the default timeout and the new config key is documented in the canonical agent config reference.

## Reproduction Assessment

Yes at source level: current main queues deferred turn maintenance without taskTimeoutMs, so a never-resolving maintenance worker can hold the maintenance lane until broader session recovery. I did not run the runtime reproduction in this read-only review, but the PR body includes redacted real-gateway logs for the fixed path.

## Solution Assessment

Mostly yes: using the existing command-lane task timeout plus a synchronous late-side-effect fence is the narrowest owner-boundary fix I found. The unresolved question is not the mechanism, but whether maintainers accept the new 120s default and config surface as the best upgrade behavior.
