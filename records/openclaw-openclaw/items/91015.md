## Summary

Keep open: the Docker sandbox hang is still real on current main/latest release, but this PR is not merge-ready because it adds a compatibility-sensitive timeout config/default without complete docs/help metadata, lacks redacted real wedged-Docker proof, and overlaps an open broader timeout design.

## What This Changes

The PR adds an optional Docker exec deadline for sandbox initialization, threads `agents.defaults.sandbox.docker.initTimeoutMs` through Docker sandbox control-plane calls, updates doctor messaging, and adds mock regression coverage.

## Best Possible Solution

Land one maintainer-approved sandbox timeout design that bounds wedged Docker initialization, documents any new config/help surface, preserves long-running user command execution, and includes redacted real wedged-engine proof plus upgrade behavior evidence.

## Reproduction Assessment

Yes for source reproduction: current main and `v2026.6.6` have no timeout on Docker sandbox init calls, and the linked issue describes a real OrbStack/Docker wedge where the child never exits. I did not run a live wedged Docker engine in this read-only review.

## Solution Assessment

Unclear as the best final shape: bounding Docker init is the right problem to solve, but this PR adds a new per-call config/default surface while a broader cooperative sandbox-context timeout remains open, so maintainers should choose the canonical timeout boundary.
