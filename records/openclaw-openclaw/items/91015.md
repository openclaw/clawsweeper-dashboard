## Summary

Keep open: this PR targets a real Docker sandbox availability bug, but it still needs real wedged-Docker proof and the timeout coverage is incomplete because pre-init prune can reach unbounded Docker manager calls before the new deadline path.

## What This Changes

Adds an optional Docker exec timeout and `agents.defaults.sandbox.docker.initTimeoutMs`, threads it through Docker sandbox initialization, updates doctor/config-hash behavior, and adds mock regression tests.

## Best Possible Solution

Land one canonical sandbox-init timeout that also bounds pre-init prune/manager Docker calls, documents the new config/default, and includes redacted real wedged-engine proof before closing the linked availability bug.

## Reproduction Assessment

Yes by source inspection: current main awaits sandbox context creation through Docker helpers with no deadline, so a Docker child that never exits can keep the run path pending. I did not run a live wedged OrbStack/Docker reproduction in this read-only review.

## Solution Assessment

No as-is: the per-call Docker deadline is useful, but the fix is incomplete until pre-init prune is bounded and the new config/default is documented and proven. The broader context-level timeout PR remains a competing design that maintainers should compare.

Full review comments:

- **[P1] Bound prune before claiming Docker init cannot hang:** `src/agents/sandbox/docker.ts:720`
  - body: `resolveSandboxContext` runs `maybePruneSandboxes` before the backend factory, and prune can call `dockerSandboxBackendManager.removeRuntime`, whose `docker rm -f` still has no timeout. With an expired sandbox registry entry and a wedged Docker engine, a run can hang before it reaches this newly bounded `dockerContainerState` call, so the reported availability failure is still possible for upgraded users. Thread the same deadline through prune/removeRuntime or make prune fail fast before sandbox init.
  - confidence: 0.83

Please add those entries with this schema change so operators can discover and tune the timeout consistently.
  - confidence: 0.88
