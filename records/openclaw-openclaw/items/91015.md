Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve sandbox skills workspace propagation:** `src/agents/sandbox/docker-backend.ts:44-49`
  - body: Current main passes `skillsWorkspaceDir` into `ensureSandboxContainer`, and Docker mount computation uses it for materialized read-only skills in writable sandboxes. The PR head's changed call omits that field, so a refresh must carry it through or skill mounts can be computed from the wrong workspace.
  - confidence: 0.88
- **[P1] Bound prune before the Docker init deadline:** `src/agents/sandbox/docker.ts:723`
  - body: `resolveSandboxContext` runs `maybePruneSandboxes` before backend creation, and prune can call Docker `rm -f` through `dockerSandboxBackendManager.removeRuntime` with no timeout. With an expired registry entry and a wedged Docker engine, a run can still hang before it reaches this newly bounded container-state call.
  - confidence: 0.84
- **[P2] Pass the deadline through config hash inspection:** `src/agents/sandbox/docker.ts:723`
  - body: On the existing-container path, `readContainerConfigHash` still calls `readDockerContainerLabel` without the new timeout after the bounded state inspect. If Docker wedges or blocks on that label inspect, sandbox init can still pend during the code path this PR is trying to bound.
  - confidence: 0.78
- **[P2] Document the new sandbox timeout config:** `src/config/zod-schema.agent-runtime.ts:208`
  - body: Adding `agents.defaults.sandbox.docker.initTimeoutMs` creates a public config surface, but the config labels/help text and sandbox docs do not describe the field, default, or upgrade behavior. Add those entries with the schema change so operators can discover and tune the timeout consistently.
  - confidence: 0.9
