  - body: Current main passes `params.skillsWorkspaceDir` into `ensureSandboxContainer`, and Docker mount computation uses it for materialized read-only skills in writable sandboxes.   - confidence: 0.9

  - body: `resolveSandboxContext` runs `maybePruneSandboxes` before backend creation, and prune can call Docker removal with no timeout. Because this PR starts the timeout inside `ensureSandboxContainer`, an expired registry entry plus a wedged engine can still hang before the new deadline exists.
  - confidence: 0.84
- **[P2] Pass the deadline into config-hash inspection:** `src/agents/sandbox/docker.ts:736`
  - body: On the existing-container path, `readContainerConfigHash` still calls Docker label inspection without the new timeout after the bounded state inspect. If Docker wedges on that label read, sandbox init can still pend in the path this PR is meant to bound.
  - confidence: 0.79
- **[P2] Document the sandbox init timeout config:** `src/config/zod-schema.agent-runtime.ts:208`
  - body: Adding `agents.defaults.sandbox.docker.initTimeoutMs` creates a public config surface, but the sandbox docs and schema help do not describe the field, default, or upgrade behavior. Add those entries with the schema change so operators can discover and tune the timeout consistently.
