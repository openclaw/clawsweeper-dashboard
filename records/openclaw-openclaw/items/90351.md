## Summary

## What This Changes

The PR changes autogenerated message-tool send idempotency keys to hash stable delivery parameters instead of the transient tool-call id, with regression coverage.

## Best Possible Solution

Land the stable autogenerated key approach if maintainers accept same-run exact-send dedupe semantics, while keeping explicit idempotencyKey as the escape hatch for distinct logical sends.

## Reproduction Assessment

## Solution Assessment

Yes, this is the best bounded fix I found: it changes the message-tool key source before the existing gateway dedupe boundary and preserves explicit caller keys. Gateway-side content dedupe or Telegram-specific TTL filtering would be broader and less precise.
