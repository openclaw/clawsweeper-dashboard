## Risks / Open Questions

- Generated message-tool keys now have a new shape, so any internal logs, mirrors, or tests that assumed `runId:message-tool:<toolCallId>` will see different autogenerated values.
- Exact same autogenerated sends after a failed or ambiguous gateway call now favor dedupe over reattempting with a fresh key; that is the desired duplicate-send fix, but it is still a message-delivery semantic change maintainers should explicitly accept.
- The implementation applies the new autogenerated key strategy to message actions generally, while the motivating visible proof is Telegram `send`. The action name is included in the fingerprint, but the broader action scope should be intentional.
