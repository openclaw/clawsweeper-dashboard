## Summary

Keep open. Current main still has a source-level split where CLI web search uses runtime provider discovery, but agent `web_search` can execute through bundled-only provider resolution and miss or null out the configured Brave runtime provider.

## What This Changes

The issue reports that agent `web_search` cannot use a configured Brave provider even though CLI web-search provider discovery and search work.

## Best Possible Solution

Align agent `web_search` execution with CLI runtime provider discovery for configured Brave and installed/external web-search providers, with focused regression coverage for the contract-artifact versus runtime-artifact path.

## Reproduction Assessment

Yes at source level: current main can route agent `web_search` through bundled-only provider resolution while CLI search defaults to runtime provider resolution. The reporter also provided a live Gateway reproduction, but this read-only review did not repeat the live API call.

## Solution Assessment

No current-main solution is present. The narrowest fix is to make agent execution use runtime providers whenever it needs an executable web-search provider, without broadening setup/secret-resolution contract loading.

## Repair Work Prompt

Fix the current-main bug where agent `web_search` can fail to use a configured Brave provider even though CLI web search works. Keep the boundary narrow: align agent `web_search` execution with runtime executable provider discovery for configured Brave and installed/external web-search providers, while preserving non-executable contract/public-artifact paths for setup and secret preflight. Add regression coverage for the case where Brave is configured under `plugins.entries.brave.config.webSearch.apiKey` and `tools.web.search.provider = "brave"`, and ensure the agent path does not load the contract artifact with `createTool: () => null` for execution. Do not add a new config flag or change provider credentials semantics. Add a `CHANGELOG.md` fix entry if behavior code changes.
