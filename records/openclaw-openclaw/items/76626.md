## Summary

Keep open. The report matches a current source-level split: CLI web search uses runtime provider discovery, but agent `web_search` can force bundled-only provider resolution when runtime web-search metadata is absent, which can hide an installed external Brave provider even though CLI discovery reports it as configured.

## What This Changes

Configured Brave web search should be available to agent `web_search` and cron `agentTurn` jobs whenever the CLI web capability reports Brave available and selected.

## Best Possible Solution

Align agent `web_search` provider resolution with the CLI for configured installed/external web-search providers, while preserving the bundled fast path when a bundled provider is definitively selected.

## Reproduction Assessment

Yes at source level: with an installed external Brave provider and missing or none runtime web-search metadata, `createWebSearchTool` can pass `preferRuntimeProviders: false`, causing `runWebSearch` to search bundled candidates only. The issue body provides the live Gateway/CLI reproduction details; this review did not repeat the live API call.

## Solution Assessment

No current-main solution is present. The narrow fix is to make the agent path use runtime/installed provider discovery whenever the configured provider is external or plugin-scoped web-search config requires it, instead of deriving that solely from runtime metadata.

The CLI `openclaw capability web search` path calls `runWebSearch` with runtime provider discovery, but agent `web_search` currently passes `preferRuntimeProviders: false` unless runtime metadata already names a non-bundled provider. Preserve the bundled-only fast path for definitely bundled providers and do not change Brave API transport semantics, plugin install records, or provider credentials.
