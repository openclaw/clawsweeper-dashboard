## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the serializer before queue work starts:** `src/infra/outbound/deliver.ts:1278`
  - body: The lock is taken only after the delivery has already been enqueued and actively claimed. If an earlier same-target send is slower in that queue/claim path, a later send can reach `serialize()` first and still be delivered first, so the patch does not guarantee FIFO by send initiation.
  - confidence: 0.9
- **[P1] Derive the key from the routed destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The key uses raw `accountId` and `to`, but account IDs and recipients are normalized or rerouted later by channel code, including thread-aware Discord targets and prefix-stripping Signal targets. Equivalent chats can bypass the lock, while distinct threads can block each other unnecessarily.
  - confidence: 0.86
- **[P2] Bound stalled sends before they hold the lane:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: This serializer waits for `fn()` to settle before advancing the same-recipient lane. If a platform send hangs, every later delivery for that key remains blocked until process restart; the serialized path needs timeout or abort handling before one stuck HTTP call becomes a recipient-wide outage.
  - confidence: 0.82
