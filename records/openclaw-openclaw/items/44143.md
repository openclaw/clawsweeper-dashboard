## Summary

Keep open: current main has no generic DeliverySerializer, and the PR is still a plausible bug fix, but the patch is not mergeable because it locks after queue persistence, derives the mutex from raw destination inputs, lacks real-channel proof, and adds an unrelated runtime dependency.

## What This Changes

The PR adds an in-memory DeliverySerializer around outbound payload delivery, tests it, and adds a Mattermost TypeBox dependency plus lockfile entry.

## Best Possible Solution

Implement serialization at the durable outbound intent boundary using a canonical adapter-resolved key that includes account, recipient, and thread identity, while preserving abort/timeout behavior and removing unrelated dependency changes.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection and the author's mock signal-cli run support a concurrent send race, but the PR still lacks real Telegram, Discord, Signal, or Mattermost after-fix proof.

## Solution Assessment

No. The proposed lock is too late and uses raw destination inputs; the safer fix is to serialize before durable queue ordering can invert, using canonical destination identity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Acquire the ordering lock before queue writes:** `src/infra/outbound/deliver.ts:526-528`
  - body: The serializer wraps only deliverOutboundPayloadsCore, after enqueueDelivery has already completed. If an earlier same-target send has slower queue I/O, a later send can enter the serializer first and deliver first, so FIFO by send initiation is still not guaranteed.
  - confidence: 0.9
- **[P2] Build the mutex key from the canonical destination:** `src/infra/outbound/deliver.ts:523`
  - body: The key uses raw accountId and to and omits thread identity. Account IDs and adapter targets are normalized later, so equivalent conversations can bypass the lock while distinct threaded conversations can be serialized together.
  - confidence: 0.88
