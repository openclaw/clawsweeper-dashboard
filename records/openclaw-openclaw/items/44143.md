## Summary

Keep open: this fixes a real outbound ordering bug, but the branch still has merge-blocking delivery and availability issues because the lane is acquired too late, keyed from raw rather than canonical destination identity, and held until platform sends settle.

## What This Changes

The PR adds an in-memory DeliverySerializer, wraps outbound payload delivery in a per channel/account/recipient FIFO lane, and adds serializer unit tests.

## Best Possible Solution

Move lane acquisition to the earliest outbound send boundary, key it with canonical channel/account/recipient/thread route identity, and add abort/timeout protection with focused regression tests before merge.

## Reproduction Assessment

Yes for the behavior class: current main sends concurrent outbound deliveries independently after queue persistence, and Mantis Telegram Desktop evidence shows baseline inversion versus candidate ordering. I did not run a fresh current-main repro because this is a read-only review.

## Solution Assessment

No: the serializer is a plausible mitigation, but it is not the best fix until it runs before queue work, uses canonical routed identity, and bounds stalled sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the lane before queue persistence can reorder sends:** `src/infra/outbound/deliver.ts:1278`
  - body: The serializer is entered only inside `deliverOutboundPayloadsWithQueueCleanup`, after `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` have already completed. If an earlier same-target send has slower queue I/O or claim work, a later send can reach this line first and still deliver first, so FIFO by send initiation is not guaranteed.
  - confidence: 0.88
- **[P1] Derive the lane key from canonical routed destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The lane key uses raw `channel`, `accountId`, and `to`, and omits `threadId`. Existing route helpers and adapters normalize account IDs and target/thread identity later, so equivalent destinations can bypass serialization while distinct threaded destinations can share the wrong lane.
  - confidence: 0.9
- **[P1] Bound platform sends while holding the lane:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: The serializer advances only after `fn` settles. Since at least the Discord webhook path performs an unbounded `fetch()` without an abort signal or timeout, one stalled platform request can hold the recipient lane and block every later same-recipient reply until restart.
  - confidence: 0.82
