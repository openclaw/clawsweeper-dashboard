## Summary

Keep open: the PR addresses a real user-visible outbound ordering bug and has strong Telegram proof, but the current patch is not merge-ready because the serializer lane can miss equivalent destinations, wedge later sends, and still allow queue I/O to reorder same-target deliveries.

## What This Changes

The PR adds an in-memory DeliverySerializer, wraps outbound delivery core calls with a per channel/account/recipient lane, and adds serializer unit tests.

## Best Possible Solution

Repair the branch so canonical destination identity defines the lane, the lane is acquired before reorderable queue work, and every lane-held platform send has a bounded abort or timeout release path.

## Reproduction Assessment

Yes. Concurrent same-recipient outbound sends reproduce the class of failure, the inspected Mantis Telegram Desktop baseline shows visible inversion, and current main still lacks an equivalent generic same-destination delivery lane.

## Solution Assessment

No. A serializer is a plausible fix, but this implementation is not the best merge shape until it uses canonical lane identity, locks before reorderable queue work, and bounds lane-held sends.

  - body: The serializer key uses raw channel, accountId, and to values, but account IDs and channel targets are canonicalized later and threaded channels include thread identity in the real destination.   - body: The new lane is held across platform delivery, but paths such as Discord webhooks await fetch without a timeout or abort signal. One stalled request can wedge every later same-lane reply until restart, so lane-held sends need a bounded release path.
  - confidence: 0.86
- **[P2] Acquire the lane before queue work can reorder sends:** `src/infra/outbound/deliver.ts:1278`
  - body: The serializer is entered only after enqueueDelivery and the active-delivery claim, so two same-target sends can still invert if the earlier request has slower queue I/O and reaches the lane second. Acquire the lane before reorderable queue work if FIFO is meant to follow send initiation order.
  - confidence: 0.82
