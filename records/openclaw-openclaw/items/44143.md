## Summary

Keep this PR open: the reported outbound ordering problem is real and the branch has useful proof, but the current patch is not merge-ready because it serializes on raw send parameters and can turn one stalled platform send into a per-recipient delivery stall.

## What This Changes

The PR adds a process-local DeliverySerializer around outbound payload delivery plus unit tests for FIFO ordering, concurrency, error handling, sync throws, and cleanup.

## Best Possible Solution

Serialize on a canonical routed channel/account/target/thread lane and make work while holding that lane abortable or bounded, while preserving the existing write-ahead delivery queue semantics.

## Reproduction Assessment

Yes for the reported symptom on the PR base: Mantis captured baseline Telegram delivery inversion and candidate ordered delivery. Current main still lacks an equivalent serializer by source inspection, but I did not rerun the live Telegram lane against current main in this read-only review.

## Solution Assessment

No. The PR is a plausible direction, but it is not the best fix yet because the lane is derived before canonical routing and held without a timeout or abort bound.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Derive the lane from canonical routed destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The key is built from raw `params.channel`, `params.accountId`, and `params.to` before channel routing/canonicalization and without `threadId`. Discord sends to different threads can share one raw `to` while routing to different platform destinations, and Signal or Telegram aliases can use different raw values for the same chat, so this does not reliably enforce the delivery invariant.
  - confidence: 0.9
- **[P1] Bound work before it can hold a recipient lane:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: `deliverySerializer.serialize` now waits for the entire platform send before releasing later sends for the same lane, but the lane itself has no timeout or abort guard. A hanging send path, such as the Discord webhook fetch, can now stall every later message for that recipient until restart.
  - confidence: 0.86
