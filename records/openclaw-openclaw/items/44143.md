## Summary

Keep this PR open. Current main still lacks per-destination outbound delivery serialization, and the PR discussion includes a useful mock-delay reproduction plus a resolved synchronous-throw follow-up, but the proposed patch should not merge as-is because it locks after queue persistence, keys on raw destination fields, can introduce a new head-of-line liveness failure, and carries an unrelated Mattermost dependency change.

## What This Changes

This PR adds an in-memory outbound DeliverySerializer around delivery, serializer unit tests, and a Mattermost package/lockfile @sinclair/typebox dependency.

## Best Possible Solution

Implement same-destination ordering at the outbound boundary before queue persistence or recovery can invert call order, derive the ordering key from canonical normalized channel/account/recipient/thread identity, preserve cancellation and bounded transport behavior, and keep dependency/package changes out of the fix unless directly required.

## Reproduction Assessment

Yes. The PR discussion gives a concrete mock HTTP delay reproduction, and current-main source inspection confirms same-recipient deliveries are independent queue entries with no destination-level lock.

## Solution Assessment

No. The serializer concept targets the right user-visible ordering problem, but this patch is not the narrowest maintainable fix because it locks too late, uses raw non-canonical key fields, lacks liveness safeguards, and includes unrelated package metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Serialize before queue persistence:** `src/infra/outbound/deliver.ts:526-528`
  - body: The lock is acquired only around `deliverOutboundPayloadsCore`, after `enqueueDelivery(...)` has already completed. Two same-target sends can still invert if the earlier call has slower queue I/O and the later call reaches the serializer first, so the patch does not guarantee FIFO by send initiation order.
  - confidence: 0.93
- **[P2] Build the key from canonical destination identity:** `src/infra/outbound/deliver.ts:522`
  - body: The mutex key uses raw `channel`, `accountId`, and `to` values. Equivalent destinations can bypass serialization after account/recipient normalization, while threaded channels can unnecessarily serialize distinct conversations; derive the key from canonical channel/account/recipient/thread identity instead.
  - confidence: 0.9
- **[P2] Avoid wedging later sends behind a stalled delivery:** `src/infra/outbound/delivery-serializer.ts:42-44`
  - body: The serializer waits on the previous same-key promise with no abort or timeout path. A hung transport call, such as an unbounded Discord webhook fetch, can permanently block every later send to that recipient until process restart.
  - confidence: 0.84
