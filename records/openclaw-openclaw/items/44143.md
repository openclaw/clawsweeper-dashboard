## Summary

Keep open: current main still has the outbound delivery race surface, but this PR’s serializer is placed too late in the delivery path and keys on raw, incomplete destination fields, so it does not reliably fix same-recipient ordering. It also needs real channel proof before merge.

## What This Changes

The PR adds an in-memory outbound delivery serializer around same-channel/account/recipient sends, tests for the serializer, and an unrelated Mattermost dependency/lockfile update.

## Best Possible Solution

Add ordering at the durable outbound intent boundary using canonical account and adapter-resolved destination identity, preferably reusing the existing keyed queue helper, and verify it with focused regression tests plus one real chat-channel proof run.

## Reproduction Assessment

Partly. Source inspection shows current main has no same-target outbound delivery serializer, and the author’s mock signal-cli run supports the race, but this PR still lacks a real channel reproduction/proof path.

## Solution Assessment

No. A serializer after queue persistence and keyed by raw `accountId`/`to` is not the narrowest maintainable fix because it can still reorder by queue timing and can miss or over-share real adapter destinations.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the ordering lock before queue writes:** `src/infra/outbound/deliver.ts:526-528`
  - body: The new lock is entered only around `deliverOutboundPayloadsCore`, but `deliverOutboundPayloadsInternal` has already awaited `enqueueDelivery` before this point. Two same-target sends can still invert if the earlier call has a slower queue write and the later call reaches this serializer first, so the PR does not guarantee FIFO by send initiation.
  - confidence: 0.9
- **[P2] Key the serializer with the canonical destination:** `src/infra/outbound/deliver.ts:523`
  - body: The mutex key uses raw `accountId` and `to` and omits thread identity. Current adapters normalize these fields later, for example account IDs canonicalize to `default`, Signal strips `signal:`, and Discord can route by `threadId`, so equivalent sends can bypass the lock while unrelated threaded sends can block each other.
  - confidence: 0.88
