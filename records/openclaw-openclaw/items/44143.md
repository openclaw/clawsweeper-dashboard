## Summary

Keep open: the delivery-ordering bug is valid, but this patch still does not guarantee same-target FIFO because the serializer is acquired after queue persistence/claim, uses raw noncanonical destination identity, and can wedge later recipient sends behind one stalled platform call; the visible Telegram candidate proof is also negative.

## What This Changes

The PR adds an in-memory DeliverySerializer around outbound delivery, keyed by channel/account/recipient, plus unit tests for FIFO ordering, error handling, sync throws, concurrency, and cleanup.

## Best Possible Solution

Rework ordering at the durable delivery boundary before enqueue/claim, derive the key from canonical channel/account/thread identity, bound or abort stalled sends, and then add corrected real transport proof.

## Reproduction Assessment

Yes for the bug class: current main has no same-target serialization before platform send, and the Mantis baseline recording shows the visible ordering inversion. I did not run a fresh live repro on the exact current-main SHA, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

No. A serializer is a plausible mitigation, but this placement is too late and too raw; the maintainable fix should live at the durable delivery-intent boundary with canonical destination identity and bounded waits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Acquire serialization before queue persistence:** `src/infra/outbound/deliver.ts:1278`
  - body: The lock is taken only inside `deliverOutboundPayloadsWithQueueCleanup`, but each call has already awaited `enqueueDelivery` and the active-claim path before reaching this point. If the earlier send has slower queue I/O, the later send can enter the serializer first, so this still fails the FIFO guarantee the PR is meant to provide.
  - confidence: 0.92
- **[P2] Use a canonical destination key:** `src/infra/outbound/deliver.ts:1274`
  - body: The serializer key uses raw `accountId` and `to` and omits `threadId`, while adapters canonicalize later. Discord routes threaded sends through `channel:${threadId}`, and Signal strips a `signal:` prefix, so equivalent chats can bypass the lock and unrelated threaded chats can block each other.
  - confidence: 0.87
- **[P2] Bound stalled sends before blocking a recipient:** `src/infra/outbound/deliver.ts:1278`
  - body: Serializing the full adapter promise turns any hung platform call into a head-of-line block for every later send to that key. The Discord webhook path currently awaits `fetch` without an abort signal or timeout, so one stalled webhook can wedge the recipient queue until restart.
  - confidence: 0.84
