## Label Justifications

- `P2`: This is a real outbound message-ordering bug with limited but user-visible channel impact and a non-trivial repair path.
- `merge-risk: 🚨 message-delivery`: The current patch can still misorder same-target sends and can use a route key that misses or over-groups real destinations.
- `merge-risk: 🚨 security-boundary`: Persisting an aborted send before a serialized wait can cause canceled, potentially sensitive outbound content to replay after restart.
- `merge-risk: 🚨 availability`: A stalled platform send can now create head-of-line blocking for all later same-key deliveries.

## Summary

Keep open: current main still has no per-target outbound serializer and the reported ordering bug is plausible, but this draft PR is not merge-ready because the lock is placed after durable enqueue, the route key is not canonical, cancellation/availability risks remain, and after-fix real channel proof is still missing.

## What This Changes

The PR adds an in-memory outbound DeliverySerializer and tests, then wraps outbound payload delivery in `deliver.ts` by a raw `channel:accountId:to` key.

## Best Possible Solution

Implement ordering at the durable route boundary before enqueue/platform send, key it from canonical channel/account/conversation/thread identity, make waits abort/timeout-aware, and prove the behavior in a real Telegram or Discord channel.

## Reproduction Assessment

No high-confidence executed reproduction was run in this review. Source inspection shows the race remains possible on current main, and the PR body adds useful before-fix live Telegram observation, but there is no after-fix real-channel proof.

## Solution Assessment

No. The current PR is a useful direction, but locking after queue I/O with a raw route key is not the narrowest maintainable fix for durable, threaded, abortable delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Move the serializer before enqueueing:** `src/infra/outbound/deliver.ts:1276`
  - body: `enqueueDelivery(...)` still runs before this lock is acquired, so two same-target sends can enter the serializer in queue-write completion order rather than initiation order. That leaves the central out-of-order delivery race intact when the earlier send has slower queue I/O.
  - confidence: 0.91
- **[P2] Build the key from the canonical route:** `src/infra/outbound/deliver.ts:1272`
  - body: The key uses raw `accountId` and `to` and omits thread/topic identity. Equivalent destinations can bypass serialization after adapter normalization, while distinct threads/topics can be unnecessarily serialized together.
  - confidence: 0.84
- **[P2] Make serialized waits abort and timeout aware:** `src/infra/outbound/deliver.ts:1276`
  - body: A send can now wait behind the per-recipient queue after it was persisted, and a hung platform call can keep the queue from advancing. The lock needs abort handling and bounded platform waits, or canceled/stalled sends can block or replay later delivery.
  - confidence: 0.82

## Risks / Open Questions

- No redacted after-fix proof from a real affected Telegram or Discord channel is posted; current after-fix validation is mock/unit-level only.
- Because the serializer is acquired after `enqueueDelivery`, same-target calls can still enter the lock in queue-write completion order rather than user-visible send initiation order.
- The raw `channel:accountId:to` key can miss equivalent routes after account/recipient normalization and can block distinct thread/topic conversations that share the same raw `to`.

- One stalled platform send, including a Discord webhook fetch without a local timeout/abort at this call site, can block every later same-key delivery until restart.
