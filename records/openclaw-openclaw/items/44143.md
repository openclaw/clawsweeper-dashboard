## Summary

Keep this PR open: the ordering bug is real and current main/latest release do not implement a same-destination delivery serializer, but this draft patch still has blocking delivery-boundary, route-key, and head-of-line stall issues before it can merge.

## What This Changes

The PR adds an in-memory outbound DeliverySerializer and wraps outbound core delivery so sends for the same raw channel/account/recipient key run FIFO while other keys stay concurrent.

## Best Possible Solution

Implement an abort-aware per-canonical-route delivery boundary before reorderable queue/claim work, with route-key and stalled-send regression coverage in the real outbound delivery wrapper.

Yes, source/proof gives a high-confidence reproduction path even though I did not run a fresh live repro. Current main has no same-recipient outbound serializer, and the attached Mantis Telegram Desktop baseline shows the visible ordering inversion that the candidate improves.

## Solution Assessment

No. The proposed serializer is a plausible mitigation, but it is not the best fix because it locks after queue work, keys from raw pre-route fields, and adds an unbounded head-of-line blocking path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Acquire the destination lock before queue work:** `src/infra/outbound/deliver.ts:1278`
  - body: This call enters the serializer only after `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` have completed. If an earlier same-target send has slower queue I/O, a later send can reach this line first and still deliver first, so the FIFO guarantee does not hold.
  - confidence: 0.92
- **[P2] Key the mutex from the canonical route:** `src/infra/outbound/deliver.ts:1274`
  - body: The key is built from raw `channel`, `accountId`, and `to`, but adapters can normalize recipients and route by `threadId` later. That can both miss equivalent same-destination sends and serialize unrelated threaded conversations together.
  - confidence: 0.88
- **[P1] Bound stalled sends before chaining later deliveries:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: The serializer advances only when `fn` settles. Since core delivery includes direct HTTP paths such as Discord webhooks, one hung platform call can now keep every later same-destination reply waiting until process restart.
  - confidence: 0.86
