## Review Findings

Full review comments:

- **[P1] Derive the lock key from the canonical destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The key is built from raw `accountId` and `to` and drops `threadId`, but the send path normalizes account IDs and resolves channel-specific targets later. Equivalent same-target sends can still bypass the mutex, while different threads in the same chat can block each other; derive the key from the canonical channel/account/recipient/thread identity before serializing.
  - confidence: 0.9
- **[P2] Bound stalled sends before blocking a recipient:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: The serializer waits on the previous same-key delivery without any timeout or abort path. Since existing adapters include platform calls such as Discord webhook `fetch()` without an abort signal, one stalled send can now hold every later delivery to that key until restart; make the wait/send path abortable or otherwise bounded.
  - confidence: 0.85
