## Summary

Keep open: current main has not implemented this ordering fix, but this branch is not ready because the lock is acquired after async durable-queue work can still reorder same-target sends, the key uses non-canonical target fields, and the after-fix proof is mock-only.

## What This Changes

The PR adds an in-memory outbound delivery serializer with tests and also adds a Mattermost Typebox dependency plus lockfile entry.

## Best Possible Solution

Implement FIFO at the durable send or queue boundary using canonical channel/account/recipient/thread identity, remove unrelated dependency changes, and prove the result with a real channel run.

## Reproduction Assessment

No. The race is plausible from source because current main has asynchronous queue work before platform delivery and no per-target mutex, but the PR only supplies mock-server proof rather than a high-confidence real OpenClaw channel reproduction.

## Solution Assessment

No. A per-target FIFO is a reasonable direction, but this patch locks too late and keys on raw, non-canonical fields; the safer fix belongs at the durable delivery boundary with canonical target identity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Acquire the serializer before queue I/O:** `src/infra/outbound/deliver.ts:523-525`
  - body: The serializer is entered only around deliverOutboundPayloadsCore, after this function has already awaited enqueueDelivery. If an earlier same-target send is delayed by that queue write while a later send reaches this line first, the later payload is still delivered first, so the main ordering race remains. The per-target lock needs to cover the queue/write-ahead boundary that can reorder concurrent sends.
  - confidence: 0.88
- **[P2] Use canonical target identity for the lock key:** `src/infra/outbound/deliver.ts:520-522`
  - body: The key is built from raw channel/account/to strings, but adapters canonicalize the real destination later. For example Signal treats signal:+1555 and +1555 as the same target, and Discord thread sends can route by threadId, so equivalent chats can bypass serialization or unrelated threads can block each other. Build the key from the canonical delivery target including thread identity where applicable.
  - confidence: 0.82
