## Summary

Keep open: the reported delivery race is plausible and current main has no per-target outbound serializer, but this draft patch still serializes too late in the durable path, uses an incomplete route key, and lacks after-fix real channel proof.

## What This Changes

The PR adds an in-memory DeliverySerializer around outbound payload delivery by channel/account/recipient plus unit coverage for ordering, concurrency, failures, sync throws, and cleanup.

## Best Possible Solution

Implement ordering at the prepared durable route boundary using canonical channel/account/conversation/thread identity, abort/timeout-aware waits, focused regression tests, and real proof on an affected channel.

## Reproduction Assessment

No high-confidence current-main repro was run here. Source inspection shows current main has no per-target outbound serializer and the PR body reports a live Telegram inversion, so the race is plausible but not fully reproduced in this review.

## Solution Assessment

No. The current patch is not the best fix because it locks after durable enqueue, keys on raw route fields, and does not make serialized waits abort/timeout-aware.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the serializer before enqueueing:** `src/infra/outbound/deliver.ts:1277`
  - body: The lock is taken only inside `deliverOutboundPayloadsWithQueueCleanup`, but every normal send has already awaited `enqueueDelivery` and acquired the queue claim by then. If an earlier same-target call has a slower queue write than a later one, the later call reaches this serializer first and is still delivered first, so the patch does not guarantee FIFO by send initiation.
  - confidence: 0.91
- **[P2] Key the serializer by the canonical route:** `src/infra/outbound/deliver.ts:1273`
  - body: The key uses raw `channel`, `accountId`, and `to` and omits the effective thread/topic route. Adapters normalize accounts/recipients later and Discord rewrites threaded sends to `channel:<threadId>`, so equivalent destinations can bypass the mutex while distinct threads can block each other.
  - confidence: 0.88
- **[P2] Abort queued waits before replay can send canceled work:** `src/infra/outbound/deliver.ts:1277`
  - body: Because the entry is persisted before this wait and `abortSignal` is only checked inside the serialized core delivery, an aborted send can sit behind another same-key send after being enqueued. If the process restarts before the wait reaches the abort check, recovery no longer has that signal and can send work the user canceled.
  - confidence: 0.82
- **[P2] Bound serialized sends so one platform stall cannot wedge a chat:** `src/infra/outbound/deliver.ts:1277`
  - body: The new per-recipient chain waits for the previous platform send to settle before any later same-key send can start. Some channel paths still have unbounded platform calls, so one stalled request can now block every later delivery to that chat until process restart.
  - confidence: 0.78
