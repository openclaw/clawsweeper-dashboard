## Summary

## What This Changes

The PR adds a process-local outbound delivery serializer and wraps queued delivery cleanup so same channel/account/recipient sends run sequentially.

## Best Possible Solution

Keep the PR open and repair the serializer so it uses canonical lane identity, acquires the lane before reorderable queue work, and guarantees lane release for aborted or stalled sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Key the lane from the canonical destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The serializer key uses raw `channel`, `accountId`, and `to`, but outbound adapters normalize accounts and targets later, and threaded channels include `threadId` in the real destination. Equivalent conversations can bypass the lane while distinct threads can unnecessarily block each other; derive the lane from the canonical channel/account/target/thread identity instead.
  - confidence: 0.9
- **[P1] Bound platform sends before they hold a lane:** `src/infra/outbound/deliver.ts:1278`
  - body: The new lane is held across the platform send, but paths such as Discord webhooks can await an unbounded fetch without an abort signal. One stalled request can wedge every later same-lane reply until restart, so the send held by the serializer needs a timeout or abort path that reliably releases the lane.
  - confidence: 0.86
- **[P2] Acquire the lane before queue I/O can reorder sends:** `src/infra/outbound/deliver.ts:1278`
  - body: The serializer is entered after `enqueueDelivery` and active-claim work, so two same-target sends can still invert when the earlier request has slower queue I/O and the later request reaches the lane first. Acquire the lane before the reorderable queue work if FIFO is meant to follow send initiation order.
  - confidence: 0.82
