## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Acquire serialization before queue persistence:** `src/infra/outbound/deliver.ts:1278-1295`
  - body: Both `enqueueDelivery(...)` and active-claim acquisition happen before this serializer call, so a later same-target send whose queue write finishes first can still enter the mutex first and be delivered first. Move the ordering boundary ahead of the asynchronous queue work so FIFO reflects send initiation order.
  - confidence: 0.98
- **[P2] Key the serializer by the canonical delivery destination:** `src/infra/outbound/deliver.ts:1274`
  - body: This key uses raw channel/account/recipient strings, but the send path later normalizes accounts, recipient aliases, and thread/topic destinations. Equivalent chats can bypass the mutex, while unrelated threaded destinations or delimiter-colliding identifiers can be serialized together; derive a structured key from canonical delivery identity instead.
  - confidence: 0.96
- **[P2] Make queued waits and formatted sends abort-aware:** `src/infra/outbound/deliver.ts:1278-1295`
  - body: A send can be persisted and then wait behind an earlier same-key send even after its abort signal fires, and formatted adapter contexts support abortSignal but this path does not pass it through. Canceled work can therefore delay later replies or be replayed after restart; cancellation needs to remove or fail the queued intent while waiting and reach the adapter send.
  - confidence: 0.9
- **[P2] Bound platform calls before chaining later deliveries:** `src/infra/outbound/deliver.ts:1278-1295`
  - body: The chain advances only after the complete adapter promise settles. A stalled platform send, such as an unbounded Discord webhook request, can become a permanent head-of-line block for later same-key messages; add a bounded send contract before serializing behind it.
  - confidence: 0.92
