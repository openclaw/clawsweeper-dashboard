## Summary

Keep this PR open because it targets a real visible message-ordering problem, but the current patch is not merge-ready: the lock is acquired too late, the key is not the channel-canonical destination, and the prepared Telegram candidate proof does not show the fixed order.

## What This Changes

The PR adds an in-memory outbound DeliverySerializer around message delivery plus unit tests for same-key FIFO ordering, error handling, sync throws, concurrency, and cleanup.

## Best Possible Solution

Keep the PR open, but rework the fix so durable outbound delivery owns same-target ordering before enqueue/claim, derives the key from canonical channel destination identity, and bounds or aborts stalled sends before new live proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the bug class: the Mantis baseline recording shows the ordering inversion and current source still has no same-target serialization before platform send, though I did not run a fresh live repro on the exact current main SHA.

## Solution Assessment

Is this the best way to solve the issue? No. A serializer is plausible, but this placement is too late and too raw; the safer fix belongs at the durable delivery-intent boundary with canonical destination identity and bounded waits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Acquire serialization before queue persistence:** `src/infra/outbound/deliver.ts:1278`
  - body: The lock is taken only inside `deliverOutboundPayloadsWithQueueCleanup`, but each call has already awaited `enqueueDelivery` and the active-claim path before reaching this point. If the earlier send has slower queue I/O, the later send can enter the serializer first, so this still fails the FIFO guarantee the PR is meant to provide.
  - confidence: 0.92
- **[P2] Use a canonical destination key:** `src/infra/outbound/deliver.ts:1274`
  - body: The serializer key uses raw `accountId` and `to` and omits `threadId`, while adapters canonicalize later. For example Discord routes threaded sends through `channel:${threadId}`, and Signal strips a `signal:` prefix, so equivalent chats can bypass the lock and unrelated threaded chats can block each other.
  - confidence: 0.87
- **[P2] Bound stalled sends before blocking a recipient:** `src/infra/outbound/deliver.ts:1278`
  - body: Serializing the full adapter promise turns any hung platform call into a head-of-line block for every later send to that key. The Discord webhook path currently awaits a bare `fetch` without an abort signal or timeout, so one stalled webhook can wedge the recipient queue until restart.
  - confidence: 0.84

## Risks / Open Questions
