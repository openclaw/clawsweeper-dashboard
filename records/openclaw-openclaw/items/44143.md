## Summary

Keep open: the ordering bug is real and proof-backed, but the current patch locks too late, keys raw destination data, and can wedge later replies behind stalled sends, so it needs repair or a narrow replacement before merge.

## Best Possible Solution

Adopt per-destination serialization around queue persistence, active claims, adapter send, and cleanup using normalized account/canonical target/thread identity, with abort/stall release tests and refreshed Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Acquire the delivery lane before queue work can reorder sends:** `src/infra/outbound/deliver.ts:1278-1279`
  - body: The serializer is entered only inside `deliverOutboundPayloadsWithQueueCleanup`, after `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` have awaited. Two same-target calls can therefore reach this line in queue-write or claim-completion order instead of caller order, so the later reply can still deliver first.
  - confidence: 0.93
- **[P1] Key delivery lanes from canonical channel targets:** `src/infra/outbound/deliver.ts:1274`
  - body: `serializerKey` uses raw `params.accountId` and `params.to` and omits thread/topic identity, while adapters normalize accounts and destinations later. Equivalent destinations can bypass the serializer, and distinct threads/topics can block each other behind one lane.
  - confidence: 0.9
- **[P2] Make queued lanes abort- and stall-aware:** `src/infra/outbound/deliver.ts:1278-1279`
  - body: Every later same-key delivery now waits for the adapter promise, but the lane has no abort, timeout, or release path. A canceled or never-settling send can wedge later replies to that conversation until restart.
  - confidence: 0.82
