## Summary

Keep open: the ordering bug is real and the PR contains useful work, but the patch is not merge-ready because the lock is acquired too late, the key is not canonical across channel adapters, a stalled send can wedge later same-target sends, and the current Telegram candidate proof still shows the wrong order.

## What This Changes

The PR adds an in-memory outbound delivery serializer keyed by channel, account, and recipient, and adds unit tests for same-key FIFO ordering, error handling, synchronous throws, concurrency, and cleanup.

## Best Possible Solution

Rework ordering at the durable delivery-intent boundary with canonical channel/account/thread identity, bound or abort stalled sends, and attach corrected redacted Telegram or Discord proof before merge.

## Solution Assessment

No. A serializer is plausible, but this implementation is too late in the durable queue path, uses raw channel parameters instead of canonical destination identity, and waits on unbounded adapter promises.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Acquire serialization before queue persistence:** `src/infra/outbound/deliver.ts:1278`
  - body: The lock is taken only inside `deliverOutboundPayloadsWithQueueCleanup`, after each call has already awaited `enqueueDelivery` and the active-claim path. If an earlier send has slower queue I/O, a later same-target send can enter the serializer first, so this does not guarantee FIFO by send initiation.
  - confidence: 0.92
- **[P2] Use a canonical destination key:** `src/infra/outbound/deliver.ts:1274`
  - body: The serializer key uses raw `accountId` and `to` and omits `threadId`, while adapters canonicalize later. Discord routes threaded sends to `channel:<threadId>` and Signal strips a `signal:` prefix, so equivalent chats can bypass the lock and unrelated threaded chats can block each other.
  - confidence: 0.88
- **[P2] Bound stalled sends before blocking a recipient:** `src/infra/outbound/deliver.ts:1278`
  - body: Serializing the full adapter promise turns any hung platform call into a head-of-line block for every later send to that key. The Discord webhook path currently awaits `fetch` without an abort signal or timeout, so one stalled webhook can wedge that recipient queue until restart.
  - confidence: 0.84
