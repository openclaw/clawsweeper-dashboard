## Summary

Keep open: the race is plausible and worth fixing, but this draft still has blocking delivery-boundary issues, uses an incomplete route key, can introduce head-of-line stalls, and does not yet show after-fix behavior in a real affected channel.

## What This Changes

The PR adds an in-memory `DeliverySerializer` and wraps outbound delivery so sends with the same channel/account/recipient key run FIFO while different keys can run concurrently.

## Best Possible Solution

Move ordering to the durable outbound route boundary using canonical channel/account/conversation/thread identity, make waits abort/timeout aware, and prove the fixed path with focused tests plus redacted live Telegram or Discord evidence.

## Reproduction Assessment

Yes, at source level: current main has no cross-session same-recipient delivery lock, and the PR body reports a live Telegram inversion plus mock reproduction. I did not run a live current-main Telegram/Discord repro in this read-only review.

## Solution Assessment

No. The intended fix direction is reasonable, but this implementation locks too late, keys on raw route fields, and does not account for cancellation or stalled platform sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Serialize before writing the durable delivery entry:** `src/infra/outbound/deliver.ts:1278`
  - body: The lock is acquired inside `deliverOutboundPayloadsWithQueueCleanup`, but each caller has already awaited `enqueueDelivery(...)` before getting here. If an earlier same-target send has slower queue I/O than a later one, the later call enters the serializer first and can still be delivered first, so the patch does not guarantee FIFO by send intent.
  - confidence: 0.9
- **[P2] Use canonical route identity for the serializer key:** `src/infra/outbound/deliver.ts:1274`
  - body: The key is built from raw `channel`, `accountId`, and `to` and omits `threadId`. That lets equivalent destinations bypass serialization after adapter normalization, while different threads/topics under the same `to` can block each other even though they are separate conversations.
  - confidence: 0.84
- **[P2] Add abort or timeout handling to serialized waits:** `src/infra/outbound/deliver.ts:1278`
  - body: A same-key delivery now waits behind the previous platform send before it can reach the core abort checks. Since at least the Discord webhook sender performs a fetch without a signal, one stalled send can wedge every later same-key delivery until the process restarts or the request settles.
  - confidence: 0.78
