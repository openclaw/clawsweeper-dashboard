## Summary

Keep open: the same-destination ordering problem is valid and current main still lacks a destination-level FIFO gate, but this PR should not merge as-is because it locks after queue persistence, keys on raw destination fields, can wedge behind stalled transports, and includes unrelated package metadata.

## What This Changes

This PR adds an in-memory outbound DeliverySerializer, serializer unit tests, and a Mattermost package/lockfile dependency entry.

## Best Possible Solution

Implement ordering at the outbound boundary before persistence or recovery can invert send initiation order, derive the lane from canonical channel route identity including threadId, preserve cancellation and bounded transport behavior, and split package metadata out.

## Reproduction Assessment

Yes. The PR discussion gives a concrete mock HTTP delay reproduction, and current-main source inspection confirms live sends are independent queue entries claimed only by queue id rather than destination.

## Solution Assessment

No. The serializer concept addresses the right race, but this patch is not the narrowest maintainable fix because it locks too late, uses non-canonical keys, lacks liveness safeguards, and mixes an unrelated dependency change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Serialize before queue persistence:** `src/infra/outbound/deliver.ts:526-528`
  - body: The lock is acquired only around `deliverOutboundPayloadsCore`, after `enqueueDelivery(...)` has already completed. Two same-target sends can still invert if the earlier call has slower queue I/O and the later call reaches the serializer first, so this does not guarantee FIFO by send initiation order.
  - confidence: 0.93
- **[P2] Build the key from canonical destination identity:** `src/infra/outbound/deliver.ts:522`
  - body: The mutex key uses raw `channel`, `accountId`, and `to` values. Equivalent destinations can bypass serialization after account/recipient normalization, while threaded channels can unnecessarily serialize distinct conversations; derive the key from canonical channel/account/recipient/thread identity instead.
  - confidence: 0.9
- **[P2] Avoid wedging later sends behind a stalled delivery:** `src/infra/outbound/delivery-serializer.ts:42-44`
  - body: The serializer waits on the previous same-key promise with no abort or timeout path. A hung transport call can permanently block every later send to that recipient until process restart.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/infra/outbound/deliver.ts:526-528`
  - body: This is a user-facing delivery behavior fix, but the PR does not update `CHANGELOG.md`. Repo policy requires a changelog entry for user-visible fix changes before merge.
  - confidence: 0.78
