## Summary

Keep open: this PR addresses a real outbound message-ordering bug and has strong Telegram-visible proof, but the current patch is still not merge-ready because the serializer is acquired too late, keys the wrong destination identity, and can wedge a recipient lane behind a stalled platform send.

The PR adds an in-memory outbound delivery serializer and tests to run same channel/account/recipient sends FIFO while preserving concurrency across different targets.

## Best Possible Solution

Move serialization to the earliest outbound lane boundary, key it from canonical channel/account/recipient/thread identity, and make lane-held sends timeout- and abort-aware before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the lane before queue persistence can reorder sends:** `src/infra/outbound/deliver.ts:1278`
  - body: The new lock is entered inside `deliverOutboundPayloadsWithQueueCleanup`, after `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` have already completed in the caller. If an earlier same-target send has slower queue I/O or claim work, a later send can reach this serializer first and still deliver first, so FIFO by send initiation is not guaranteed.
  - confidence: 0.88
- **[P1] Derive the lane key from canonical routed destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The lane key uses raw `params.channel`, `params.accountId`, and `params.to`, and omits `threadId`. Current adapters normalize account ids and targets later, including Discord threaded sends routing to `channel:${threadId}` and Signal stripping `signal:`, so equivalent destinations can bypass serialization while distinct threads can share the wrong lane.
  - confidence: 0.9
- **[P1] Bound platform sends while holding the lane:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: The serializer advances only after `fn` settles. Discord webhook delivery currently calls `fetch` without an abort or timeout signal on this path, so one stalled platform call can now hold the recipient lane and block every later same-recipient reply until restart.
  - confidence: 0.82
