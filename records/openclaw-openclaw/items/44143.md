## Summary

Keep open: current main still lacks same-destination outbound serialization, and the Telegram proof shows the bug, but this PR is not merge-ready because the lock is acquired too late, uses raw destination keying, and can introduce per-destination head-of-line stalls.

## What This Changes

The PR adds an in-memory `DeliverySerializer`, wraps the outbound delivery core send path with it, and adds focused serializer unit tests.

## Best Possible Solution

Move same-destination serialization to the earliest durable send-intent boundary before queue persistence and active claims, derive the key from canonical route/account/thread data, and make queued waits or platform sends abort-aware or bounded.

## Reproduction Assessment

Yes for source/proof purposes: Mantis captured a baseline Telegram ordering inversion, and current main still has independent outbound sends with no same-destination serializer. I did not run a fresh live current-main repro in this read-only review.

## Solution Assessment

No. The in-memory FIFO idea is a plausible fix direction, but the current placement is too late, the key is not canonical, and the chain needs bounded or abort-aware behavior before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Acquire the destination lock before queue work:** `src/infra/outbound/deliver.ts:1277`
  - body: The serializer is entered only inside `deliverOutboundPayloadsWithQueueCleanup`, but each call has already awaited `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)`. If an earlier same-target call has slower queue or claim work, a later call can reach this lock first and still deliver first, so the PR does not guarantee FIFO by send initiation.
  - confidence: 0.88
- **[P2] Key the mutex from the canonical route:** `src/infra/outbound/deliver.ts:1273`
  - body: This raw `channel:accountId:to` string bypasses existing route/account normalization and omits thread identity. Equivalent destinations can use different locks, while threaded destinations can share one lock unnecessarily; build the key from canonical route data including normalized account and thread values.
  - confidence: 0.84
- **[P2] Bound stalled sends before chaining later deliveries:** `src/infra/outbound/deliver.ts:1277`
  - body: Once a send enters this chain, later same-target deliveries wait for it to settle. Sibling paths include Discord webhook sends that await `fetch()` without a local timeout or passed abort signal, so a hung request can wedge that destination until restart.
  - confidence: 0.8
