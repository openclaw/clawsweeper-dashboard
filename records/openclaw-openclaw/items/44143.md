## Summary

Keep this PR open because it targets a real outbound ordering bug with useful proof, but it is not merge-ready: the serializer is acquired after queue work can reorder sends, uses a raw destination key instead of the canonical channel target, and can wedge later replies behind one stalled platform send.

## What This Changes

The PR adds a process-local outbound delivery serializer and wraps core delivery so sends to the same channel/account/recipient run FIFO while different keys remain concurrent.

## Best Possible Solution

Land a bounded outbound serializer only after lane identity comes from canonical channel/account/target/thread data and lane-held platform sends can abort or time out without wedging later deliveries.

## Reproduction Assessment

Yes, for the reported behavior: Mantis captured a baseline/candidate Telegram ordering proof, and current main still has the same no-serializer outbound source path. I did not run a fresh current-main live Telegram reproduction in this read-only review.

## Solution Assessment

No. A serializer is a plausible fix, but this implementation is not the best merge shape until it uses canonical lane identity, acquires the lane before queue I/O, and bounds lane-held sends.

- **[P1] Key the lane from the canonical destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The mutex key uses raw `params.channel`, `params.accountId`, and `params.to`, but delivery later routes through channel-specific normalization and thread handling.   - confidence: 0.9
- **[P1] Bound platform sends before they hold a lane:** `src/infra/outbound/deliver.ts:1278`
  - body: Wrapping `deliverOutboundPayloadsCore` means one never-settling platform send owns the per-recipient lane. The Discord webhook path uses `fetch` without a timeout or signal, so later replies to that lane can remain stuck until restart; add abort/timeout handling before serializing production sends.
  - confidence: 0.88
- **[P2] Acquire the lane before queue I/O can reorder sends:** `src/infra/outbound/deliver.ts:1278`
  - body: The serializer is entered only after `enqueueDelivery` and the active claim. If an earlier same-target send takes longer to persist than a later one, the later call can acquire the lane first and deliver first, so this does not guarantee FIFO by send initiation.
  - confidence: 0.8
