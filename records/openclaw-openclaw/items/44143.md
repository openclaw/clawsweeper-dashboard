## Summary

Keep this PR open: the outbound ordering bug is valid and the Telegram proof is strong, but the current patch is still only a partial fix because the lock is acquired after durable queue work, uses a raw destination key, and can introduce unbounded head-of-line blocking.

## What This Changes

Adds an in-memory `DeliverySerializer` and wraps outbound payload delivery so deliveries sharing a channel/account/recipient key run FIFO, with focused serializer unit tests.

## Best Possible Solution

Keep the per-destination ordering fix, but move it to the durable delivery boundary with a canonical route key and a bounded stall policy, then prove it with deliver-level regression coverage and the existing Telegram proof lane.

## Reproduction Assessment

Yes. Current main has no cross-session destination lock before queue persistence, and the inspected Mantis Telegram Desktop proof shows the reported baseline inversion with ordered candidate behavior; I did not run a fresh local E2E in this read-only review.

## Solution Assessment

No. Per-destination serialization is the right direction, but this implementation is not yet the best fix because it locks below the durable queue boundary, keys raw inputs instead of the canonical route, and lacks a stalled-send bound.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the destination lock before queue work:** `src/infra/outbound/deliver.ts:1278`
  - body: Here the serializer wraps only `deliverOutboundPayloadsCore`, but the caller has already awaited `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` before reaching this line. If two same-target sends start in order but the later queue write or claim completes first, it enters this serializer first and can still send before the earlier message, so FIFO by send initiation is not guaranteed.
  - confidence: 0.9
- **[P1] Key the mutex with the canonical delivery route:** `src/infra/outbound/deliver.ts:1274`
  - body: This key uses raw `channel/accountId/to` and omits `threadId`, while adapters canonicalize those fields later. Equivalent destinations such as normalized Signal targets can bypass serialization, and distinct Discord threads can share a lock, leaving ordering unfixed and adding cross-thread latency.
  - confidence: 0.86
- **[P1] Bound stalled sends before chaining later deliveries:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: `serialize()` waits for each platform send before allowing the next same-key send, but the chain has no timeout or abort handoff. A hung adapter call, such as the Discord webhook path's bare `fetch`, leaves this tail unresolved and blocks every later delivery to that destination until restart.
  - confidence: 0.84
