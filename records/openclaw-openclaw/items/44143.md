## Label Justifications

- `P2`: This is a real outbound message-ordering bug, but the impact is bounded to concurrent same-recipient delivery rather than core runtime availability for all users.
- `merge-risk: 🚨 message-delivery`: The diff changes outbound delivery ordering and can still allow out-of-order sends when queue persistence races before the serializer is entered.
- `merge-risk: 🚨 security-boundary`: The new wait can leave a persisted canceled message unreconciled until restart, risking delivery of content after the caller's abort boundary.
- `merge-risk: 🚨 availability`: Serializing platform sends means one stalled request can block later sends to the same key, and at least the Discord webhook path lacks the normal request timeout.

## Summary

Keep open: the outbound ordering bug is real, but this draft is not merge-ready because the serializer is entered after queue persistence/claiming, waits are not abort-aware, and the key does not match canonical thread/target routing. The after-fix proof is still mock-only rather than a real Telegram/Discord run.

## What This Changes

The PR adds an in-memory DeliverySerializer and wraps outbound payload delivery so concurrent sends to the same channel, account, and recipient are intended to run FIFO while different recipients stay concurrent.

## Best Possible Solution

Keep the ordering fix, but move it to an abort-aware queue/send boundary using a canonical destination key from the channel routing contract and bound platform waits before considering merge.

## Reproduction Assessment

Yes from source and discussion: current main has no shared per-recipient send lock, and the PR body reports a live Telegram inversion. I did not run a live repro in this read-only review.

## Solution Assessment

No: the proposed approach is a plausible mitigation, but not the best current fix because it locks after queue persistence and keys on raw, pre-adapter routing fields. The safer fix is canonical, abort-aware serialization at the durable send boundary with timeout coverage for locked platform calls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Acquire the lock before queue writes can reorder sends:** `src/infra/outbound/deliver.ts:1278`
  - body: This only enters the serializer after `enqueueDelivery(...)` and the active-claim step have completed. If the earlier call is slower to persist or claim its queue entry, a later same-target call reaches this line first and sends first, so the PR still fails to guarantee FIFO by send initiation; put the same-target ordering gate around the full durable enqueue/send path or persist and drain by queue order.
  - confidence: 0.88
- **[P2] Make queued serializer waits abort-aware:** `src/infra/outbound/deliver.ts:1278`
  - body: A send can now be persisted, then wait behind another same-target delivery before `deliverOutboundPayloadsCore` gets a chance to check `abortSignal`. If the user cancels and the process restarts while it is waiting, recovery can replay that queued entry because the abort cleanup never ran; the wait needs to observe cancellation and ack/delete the queued entry before replay is possible.
  - confidence: 0.84
- **[P2] Key the mutex by the canonical delivery target:** `src/infra/outbound/deliver.ts:1274`
  - body: The new key uses raw `channel/accountId/to`, but adapters later normalize or rewrite the actual destination: Discord threads become `channel:${threadId}`, Telegram uses `messageThreadId`, and Signal strips `signal:` prefixes. That can both miss equivalent recipients and serialize unrelated thread/topic conversations, so derive the key from the same canonical target contract the adapter will send to.
  - confidence: 0.8
- **[P2] Bound platform sends before serializing behind them:** `src/infra/outbound/deliver.ts:1278`
  - body: Once this lock wraps the platform call, one hung send blocks every later same-key reply. The Discord webhook path currently uses bare `fetch`/`proxyFetch` without the RequestClient timeout, so a stalled webhook can wedge a recipient queue until restart; either make locked sends timeout/abort consistently or keep unbounded paths outside the serializer.
  - confidence: 0.76
