## Summary

Keep this PR open: it targets a real outbound message-ordering bug and now has usable Telegram Desktop proof, but the current patch is not merge-ready because the serializer is acquired too late and keyed too narrowly for current-main routing.

## What This Changes

The PR adds an in-memory `DeliverySerializer` and wraps outbound platform sends so deliveries to the same raw `channel:accountId:to` key run FIFO, with focused serializer tests.

## Best Possible Solution

Move serialization to the earliest same-destination send-intent boundary, derive the key from canonical channel route/account/thread data, and bound or abort queued platform work so one stalled send cannot wedge a destination.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for source/proof purposes: Mantis captured a baseline Telegram inversion, and current main still has independent outbound sends with no same-destination serializer, though I did not run a fresh live repro during this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No: the in-memory FIFO idea is plausible, but the lock is acquired after queue work that can reorder sends and the key ignores existing canonical route/account/thread helpers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Acquire the destination lock before queue work:** `src/infra/outbound/deliver.ts:1277`
  - body: The serializer is entered only inside `deliverOutboundPayloadsWithQueueCleanup`, but current `deliverOutboundPayloadsInternal` awaits `enqueueDelivery(...)` and the active claim first. If an earlier same-target call has slower queue or claim work, a later call can reach this line first and still deliver first, so the PR does not guarantee FIFO by send initiation.
  - confidence: 0.86
- **[P2] Key the mutex with the canonical route:** `src/infra/outbound/deliver.ts:1273`
  - body: This raw `channel:accountId:to` string bypasses the existing route/account normalization and omits thread identity. Equivalent destinations such as Signal-prefixed targets can use different locks, while threaded destinations can share one lock unnecessarily; use the canonical route key including normalized account and thread data instead.
  - confidence: 0.82
- **[P2] Bound stalled sends before chaining later deliveries:** `src/infra/outbound/deliver.ts:1277`
  - body: Once a send enters this chain, later same-target deliveries wait for it to settle. Sibling paths include platform calls such as Discord webhook `fetch()` without a local timeout, so a hung request can wedge that destination until restart; make the queued wait/platform send abort-aware or bounded before merging.
  - confidence: 0.78
