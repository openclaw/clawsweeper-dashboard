## Summary

Keep open: this is a real outbound message-ordering bug and the PR is focused, but the current patch is not merge-ready because the serializer is acquired after queue persistence/claiming, uses a raw non-canonical destination key, and can turn one stalled adapter send into a same-recipient outage.

## What This Changes

The PR adds an in-memory DeliverySerializer around outbound payload delivery plus unit tests for same-key FIFO ordering, error handling, synchronous throws, concurrency, and cleanup.

## Best Possible Solution

Move the ordering boundary to a canonical per-destination delivery intent before write-ahead queue/platform send, preserve crash-recovery semantics, make waits abortable or bounded, and prove the fixed path with corrected real Telegram or Discord ordering evidence.

## Reproduction Assessment

Yes, source inspection and the baseline Telegram proof give a high-confidence reproduction shape: concurrent same-recipient deliveries race independently through the outbound path. I did not run a fresh current-main live repro in this read-only review.

## Solution Assessment

No. The PR is a plausible mitigation, but the best fix needs to lock at the canonical destination boundary before queue persistence/claiming and make the serialized wait bounded or abort-aware.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire serialization before queue persistence:** `src/infra/outbound/deliver.ts:1278`
  - body: The new lock is acquired only inside `deliverOutboundPayloadsWithQueueCleanup`, but each live send already awaited queue persistence and `withActiveDeliveryClaim` before reaching this line. If an earlier same-target send has slower queue I/O and a later send reaches this call first, the later message still delivers first, so the PR does not guarantee FIFO by send initiation.
  - confidence: 0.9
- **[P2] Derive keys from canonical destinations:** `src/infra/outbound/deliver.ts:1274`
  - body: The serializer key is built from raw `channel:accountId:to` and omits thread identity, while adapters normalize the real destination later: Signal strips `signal:` prefixes and Discord threaded sends route to `channel:${threadId}`. Equivalent recipients can bypass the lock, and distinct threaded conversations can be serialized together.
  - confidence: 0.86
- **[P2] Bound stalled sends before blocking later replies:** `src/infra/outbound/deliver.ts:1278`
  - body: `serialize()` waits for the full platform adapter promise before letting later same-key sends run. Existing paths include Discord webhook `fetch()` without a signal/timeout and formatted sends that do not receive the delivery abort signal, so one stalled or canceled send can wedge every later reply to that recipient until restart.
  - confidence: 0.82
