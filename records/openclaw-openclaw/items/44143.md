## Summary

Keep this PR open because current main does not implement the requested per-recipient outbound ordering, but the proposed patch still has merge-blocking correctness issues and lacks real affected-channel proof.

## What This Changes

The PR adds an in-memory DeliverySerializer around outbound delivery calls, focused serializer tests, and an unrelated Mattermost dependency/lockfile change.

## Best Possible Solution

Serialize at the durable outbound intent boundary using canonical channel/account/recipient/thread identity, preserve queue claim/abort behavior, remove unrelated dependency churn, and require redacted real-channel proof.

## Reproduction Assessment

No high-confidence live reproduction is established. Current-main source inspection and the author's mock signal-cli run support the concurrent-send ordering race, but the PR still lacks real affected-channel proof.

## Solution Assessment

No. The current patch is not the best fix because it serializes after queue persistence and derives the mutex key from raw route inputs instead of canonical route identity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move the ordering lock before queue persistence:** `src/infra/outbound/deliver.ts:526-528`
  - body: The serializer is entered only after `enqueueDelivery` completes. If an earlier same-target send is slower writing its queue entry, a later send can reach the mutex first and deliver first, so FIFO by send initiation is still not guaranteed.
  - confidence: 0.9
- **[P2] Use canonical route identity for the serializer key:** `src/infra/outbound/deliver.ts:523`
  - body: The key is built from raw channel/account/to values and omits thread identity. Account and target identity are normalized later, so equivalent chats can bypass the lock while distinct threads can block each other.
  - confidence: 0.88
- **[P2] Bound serialized sends with abort or timeout handling:** `src/infra/outbound/deliver.ts:526-528`
  - body: A serialized delivery waits for `fn` without a timeout or abort-aware release path. A stalled platform send, such as Discord's webhook fetch path, can leave every later same-key delivery stuck behind it until process restart.
  - confidence: 0.78
