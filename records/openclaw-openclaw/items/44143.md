## Summary

Keep open: the Telegram proof is now readable and useful, but the patch is not merge-ready because the serializer keys raw destinations and can also wedge later same-target deliveries behind one stalled send.

## What This Changes

The PR adds an in-memory outbound DeliverySerializer around core delivery plus unit tests for FIFO ordering, errors, sync throws, concurrency, and cleanup.

## Best Possible Solution

Move the ordering boundary to a canonical delivery-destination identity and make the serialized wait bounded or abort-aware before merge.

## Reproduction Assessment

Yes from source and proof, though I did not run a fresh current-main live repro in this read-only review. Current main sends same-recipient outbound calls independently, and the Mantis baseline recording shows ORDER B arriving above ORDER A.

## Solution Assessment

No. The PR is a plausible mitigation, but the best fix should key by canonical destination identity and prevent one stuck platform call from wedging later same-target sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Key serialized delivery by canonical destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The new mutex key is built from raw `channel/accountId/to` and omits thread identity, but current send paths canonicalize those fields later: account ids normalize through `normalizeAccountId`, Signal strips a leading `signal:`, and Discord threaded sends route to `channel:${threadId}`. That lets equivalent same chats bypass serialization while unrelated threads can block each other; derive the key from the same canonical destination identity the adapter will actually send to.
  - confidence: 0.88
- **[P2] Bound serialized sends so one stall cannot wedge a chat:** `src/infra/outbound/delivery-serializer.ts:39-44`
  - body: `serialize()` keeps the key locked until the full platform promise settles. Discord threaded webhooks can await a bare `fetch()` without timeout or abort signal, so one stalled HTTP call can leave every later message for that key pending until process restart; before this PR those later sends were not behind that stalled promise. Add a bounded or abort-aware serialized section before merging.
  - confidence: 0.82
