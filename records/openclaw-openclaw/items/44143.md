## Summary

Keep open: current main does not implement the requested outbound ordering guard, but this PR still has correctness blockers around lock placement and destination identity, lacks real-channel proof, and includes an unrelated Mattermost dependency.

## What This Changes

The branch adds an in-memory DeliverySerializer around outbound payload delivery, tests for it, and a Mattermost TypeBox dependency plus lockfile update.

## Best Possible Solution

Implement serialization at the durable outbound intent boundary with a canonical adapter-resolved channel/account/recipient/thread key, preserve abort and timeout behavior, remove the unrelated dependency, add focused regression coverage, and collect real-channel proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection and the author’s isolated mock signal-cli run support the concurrent-send ordering race, but the PR still lacks real-channel before/after proof.

## Solution Assessment

No. The current PR is not the best fix because it locks after durable queue persistence and derives the mutex key from raw inputs rather than canonical account, recipient, and thread identity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Acquire the ordering lock before queue writes:** `src/infra/outbound/deliver.ts:526-528`
  - body: The serializer is entered only after enqueueDelivery has finished. Two same-target sends can still invert if the earlier call has slower queue I/O, because the later call can reach the mutex and deliver first, so FIFO by send initiation remains unfixed.
  - confidence: 0.9
- **[P2] Build the mutex key from canonical destination identity:** `src/infra/outbound/deliver.ts:523`
  - body: The key uses raw accountId and to and omits thread identity. Account IDs and adapter targets normalize later, while Discord threadId changes the routed destination, so equivalent chats can bypass the lock and distinct threaded chats can be serialized together.
  - confidence: 0.88
