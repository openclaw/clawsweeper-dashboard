## Summary

Keep this PR open, but it still needs changes before merge: the Telegram proof shows the race is real and the idea is useful, but the current lock is acquired too late, uses a raw destination key, and can turn one stalled send into a same-recipient head-of-line block.

## What This Changes

The PR adds an in-memory `DeliverySerializer` and wraps outbound platform delivery so concurrent sends to the same raw channel/account/recipient key run FIFO while different keys stay concurrent.

## Best Possible Solution

Land a repaired implementation that serializes at the canonical outbound destination boundary, preserves per-topic concurrency, and prevents a stuck send from wedging the lane.

## Reproduction Assessment

Yes. The PR body and Mantis proof show the visible Telegram ordering bug, and source inspection shows current main has no central outbound serializer while concurrent calls can race before platform send.

## Solution Assessment

No. The proposed serializer is a plausible mitigation, but the lock is too late, the key is not the canonical destination identity, and stalled sends need bounded cancellation/timeout behavior before this is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the serializer before queue work starts:** `src/infra/outbound/deliver.ts:1276`
  - body: The lock is taken inside `deliverOutboundPayloadsWithQueueCleanup`, but `deliverOutboundPayloadsInternal` awaits `enqueueDelivery(...)` and the active-claim path before it gets there. If two same-target sends race and the later queue write finishes first, the later send acquires this serializer first, so the PR still does not guarantee FIFO by send initiation.
  - confidence: 0.9
- **[P1] Derive the key from the canonical routed destination:** `src/infra/outbound/deliver.ts:1272`
  - body: The key uses raw `params.accountId` and `params.to`, but the actual send path normalizes accounts and rewrites targets later, such as Signal stripping a `signal:` prefix and Discord routing threaded sends to `channel:${threadId}`. Equivalent recipients can bypass serialization, while distinct threads can unnecessarily share one lane.
  - confidence: 0.86
- **[P1] Bound stalled sends before they hold the lane:** `src/infra/outbound/deliver.ts:1276`
  - body: This wraps the whole platform delivery in a per-recipient promise chain, but some platform sends can stall without a timeout, including Discord webhook `fetch`. One hung request can now block every later reply for that serialized destination until restart, which is an availability regression from independent sends.
  - confidence: 0.82
