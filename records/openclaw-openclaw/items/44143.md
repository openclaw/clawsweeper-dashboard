## Summary

Keep open: the ordering bug is plausible and current main has no serializer, but this PR still locks after queue persistence, keys by raw pre-normalized destination data, lacks real chat-channel proof, and includes an unrelated Mattermost dependency change.

## What This Changes

The PR adds an in-memory DeliverySerializer around outbound core sends, serializer unit tests, and a Mattermost TypeBox dependency plus lockfile entry.

## Best Possible Solution

Move ordering to the durable outbound intent boundary with a canonical adapter-resolved destination key, preserve abort semantics, remove the unrelated dependency change, and verify with focused regression tests plus one real chat-channel proof run.

## Reproduction Assessment

No, not at high confidence. Source inspection and the author's mock signal-cli run support the race, but there is still no live current-main Telegram, Discord, Signal, or Mattermost reproduction path.

## Solution Assessment

No. The proposed patch locks too late and uses raw destination inputs; the narrower maintainable solution is to serialize at the durable intent boundary using canonical account, recipient, and thread identity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the ordering lock before queue writes:** `src/infra/outbound/deliver.ts:526-528`
  - body: The new lock is entered only around `deliverOutboundPayloadsCore`, but each call has already awaited `enqueueDelivery`. If an earlier same-target send has slower queue I/O, a later send can reach the serializer first and deliver first, so this still does not guarantee FIFO by send initiation.
  - confidence: 0.9
- **[P2] Key serialization by the canonical destination:** `src/infra/outbound/deliver.ts:523`
  - body: The mutex key uses raw `accountId` and `to` and omits thread identity. Current account normalization and channel adapters can resolve equivalent inputs later, so same real conversations can bypass the lock while distinct threads can be unnecessarily serialized.
  - confidence: 0.88
