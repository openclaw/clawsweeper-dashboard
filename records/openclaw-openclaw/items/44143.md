## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the lane before queue persistence can reorder sends:** `src/infra/outbound/deliver.ts:1273-1277`
  - body: The serializer is entered only after `enqueueDelivery(...)` and the active delivery claim have completed. Two same-recipient sends can still invert when the earlier call has slower queue I/O and the later call reaches this lock first, so the patch does not guarantee FIFO by send initiation.
  - confidence: 0.88
- **[P1] Derive the lane key from canonical routed destination:** `src/infra/outbound/deliver.ts:1273`
  - body: The key uses raw `${channel}:${accountId}:${to}`, but channels normalize account IDs, recipient syntax, and thread targets before sending. Equivalent recipients can bypass serialization while distinct threaded sends can share the wrong lane; derive the key from the canonical route identity instead.
  - confidence: 0.9
- **[P1] Bound stalled sends before they hold the recipient lane:** `src/infra/outbound/delivery-serializer.ts:36`
  - body: The new queue advances only when the platform send promise settles. Paths such as Discord webhook fetches and formatted adapter sends are not consistently timeout or abort bounded here, so one stalled delivery can now block every later same-recipient reply until the process restarts.
  - confidence: 0.82
