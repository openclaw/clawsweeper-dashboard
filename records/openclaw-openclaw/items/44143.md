## Summary

This PR addresses a reproduced outbound ordering bug and includes useful baseline Telegram evidence, but the patch serializes too late to guarantee initiation order and uses a non-canonical key that does not match the actual destination boundary. Current main has also substantially rewritten the same delivery transaction since the branch base, so the useful fix should remain open but be rebased and repaired before merge.

## What This Changes

The PR adds an in-memory FIFO serializer around outbound platform sends keyed by channel, account ID, and recipient, plus unit tests for serialization behavior.

## Best Possible Solution

Rebase onto current main and make one canonical, abort-aware delivery transaction own ordering before queue persistence, keyed by normalized account, recipient, and thread/topic identity, with bounded platform sends and focused regression coverage.

## Reproduction Assessment

Yes. The discussion documents a live Telegram inversion in normal use, and the Mantis baseline recording visibly exercises the same ordering scenario.

## Solution Assessment

No. A platform-send-only mutex is plausible but not the best fix because ordering-sensitive queue work occurs before it and the raw key does not represent the canonical channel destination.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.97

Full review comments:

- **[P1] Acquire serialization before queue persistence:** `src/infra/outbound/deliver.ts:1278`
  - body: Both `enqueueDelivery(...)` and active-claim acquisition happen before this call, so a later same-target invocation whose queue write finishes first enters the serializer first and is still delivered first. Move the ordering boundary ahead of the asynchronous queue work so FIFO reflects send initiation rather than queue-I/O completion.
  - confidence: 0.98
- **[P2] Key the serializer by the canonical delivery destination:** `src/infra/outbound/deliver.ts:1274`
  - body: This raw concatenation differs from the normalized account, recipient, and thread/topic identity used by channel adapters. Equivalent targets can bypass serialization, distinct threads can block each other, and delimiter-containing components can collide; derive a structured key after canonical target resolution instead.
  - confidence: 0.96
- **[P2] Make queued waits and formatted sends abort-aware:** `src/infra/outbound/deliver.ts:1278-1295`
  - body: A call already persisted to the recovery queue can wait behind an earlier send even after its abort signal fires, and formatted adapter wrappers do not consistently forward that signal. The canceled work can therefore delay later replies and may be replayed after restart; cancellation must remove or fail the queued intent while waiting and reach the actual adapter send.
  - confidence: 0.9
- **[P2] Bound platform calls before chaining later deliveries:** `src/infra/outbound/deliver.ts:1278-1295`
  - body: The chain advances only when the complete adapter promise settles. A stalled Discord webhook fetch has no local timeout, so this change turns one hung request into a permanent block for every later message sharing the key; add a bounded send contract before serializing behind it.
  - confidence: 0.93
