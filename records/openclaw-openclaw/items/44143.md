## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Move the lock ahead of queue persistence:** `src/infra/outbound/deliver.ts:526-528`
  - body: `enqueueDelivery(...)` still runs before the serializer, so two same-target sends can invert if the later queue write completes first and reaches this lock first. That means the patch does not guarantee FIFO ordering by send initiation, which is the behavior it claims to fix.
  - confidence: 0.88
- **[P2] Derive the key from canonical route identity:** `src/infra/outbound/deliver.ts:523`
  - body: The key is built from raw `channel`, `accountId`, and `to`, but current routing has normalized account IDs, target parsers, and `threadId` as part of route identity. Use the canonical route helper or equivalent resolved target facts so equivalent routes do not bypass the lock and different threads do not unnecessarily block each other.
  - confidence: 0.86
- **[P2] Preserve abort and timeout behavior while queued:** `src/infra/outbound/deliver.ts:526-528`
  - body: A send waiting behind a long or hung same-key delivery does not observe abort until after the previous task finishes, and some platform sends such as Discord webhooks have no timeout. This turns one stalled send into a head-of-line block for later replies that previously could proceed independently.
  - confidence: 0.8
