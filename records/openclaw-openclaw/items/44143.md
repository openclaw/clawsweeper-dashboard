- `P2`: This is a normal-priority delivery-ordering bugfix with real user impact but limited blast radius and no emergency runtime outage.
- `merge-risk: 🚨 message-delivery`: The patch changes outbound delivery sequencing and can still misorder same-target sends because the lock is acquired after queue I/O.
- `merge-risk: 🚨 security-boundary`: A canceled persisted send can be replayed after restart without the original abort signal, potentially delivering content the user tried to stop.
- `merge-risk: 🚨 availability`: The new per-key mutex can turn one stalled platform request into a same-recipient delivery stall until restart.

## Summary

Keep open, but this PR is not merge-ready: the ordering bug is plausible and current main still lacks a generic per-route outbound serializer, but this patch serializes too late, uses a raw destination key, creates cancellation/head-of-line risks, and still lacks after-fix real channel proof.

## What This Changes

The PR adds an in-memory DeliverySerializer around outbound payload delivery keyed by channel/account/recipient, plus unit coverage for ordering, concurrency, failures, synchronous throws, and cleanup.

## Best Possible Solution

Implement ordering at the canonical durable route boundary before enqueue/send, using normalized account, target, and thread/topic identity with abort/timeout-aware waits, then prove same-chat ordering and cross-chat/thread concurrency in a real Telegram or Discord setup.

## Reproduction Assessment

Unclear for a full live current-main user path: source inspection shows direct outbound delivery still lacks per-target serialization, while current main already serializes same-turn auto-reply delivery through the reply dispatcher. The PR body provides useful before-fix Telegram evidence, but this review did not run a real current-main reproduction.

## Solution Assessment

No. The right direction is a durable-route ordering fix before enqueue/platform send with canonical route keys and abort/timeout behavior, not a late raw-key mutex around only deliverOutboundPayloadsCore.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Serialize before durable queue I/O:** `src/infra/outbound/deliver.ts:1273`
  - body: The lock is acquired only after `enqueueDelivery(...)` and the active-claim setup have completed. Two same-target sends can still invert if the earlier call has a slower queue write and the later call reaches `deliverySerializer.serialize(...)` first, so the patch does not guarantee FIFO by send initiation.
  - confidence: 0.9
- **[P2] Use a canonical route key for the mutex:** `src/infra/outbound/deliver.ts:1269`
  - body: The mutex key uses raw `channel`, `accountId`, and `to` and omits `threadId`. Equivalent account/target aliases can bypass serialization, while distinct thread/topic destinations can be serialized together and regress latency for unrelated conversations.
  - confidence: 0.86
- **[P2] Make serialized waits abort and timeout aware:** `src/infra/outbound/deliver.ts:1273`
  - body: Because the send is persisted before it waits on this mutex, an aborted send can sit behind a long or stalled same-key delivery and later replay after restart without the original abort signal. The queue needs cancellation handling before persistence or an abort-aware serialized boundary.
  - confidence: 0.82
