## Summary

Keep open: this targets a real outbound ordering bug with sufficient Telegram proof, but the patch is not merge-ready because the new lock is acquired after queue work, uses raw destination identity, and can wedge later deliveries behind an unsettled platform send.

## What This Changes

Adds an in-memory DeliverySerializer around outbound payload delivery, plus unit tests for same-key ordering, cross-key concurrency, error propagation, synchronous throws, stress ordering, and cleanup.

## Best Possible Solution

Move serialization to the earliest durable outbound lane boundary, derive the lane from canonical channel/account/recipient/thread identity, and make lane-held sends abort- and timeout-bounded before merge.

## Solution Assessment

No. The serializer is a plausible direction, but it is not the best fix until it runs before queue work, keys canonical routed identity, and bounds stalled sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the lane before queue persistence can reorder sends:** `src/infra/outbound/deliver.ts:1278`
  - body: The serializer is entered only after enqueueDelivery and the active delivery claim have completed. Two same-recipient sends can still invert if the earlier call has slower queue I/O and the later call reaches this lock first, so this does not guarantee FIFO by send initiation.
  - confidence: 0.88
- **[P1] Derive the lane key from canonical routed identity:** `src/infra/outbound/deliver.ts:1274`
  - body: The key uses raw channel/accountId/to values, but channels normalize account IDs, recipient syntax, and thread targets before sending. Equivalent recipients can bypass serialization while unrelated threaded destinations can share a lane; use canonical route identity instead.
  - confidence: 0.9
- **[P1] Bound stalled sends before they hold the recipient lane:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: The new queue advances only when the platform send promise settles. Paths such as Discord webhook sends are not bounded at this serializer boundary, so one stalled delivery can block every later same-recipient reply until process restart.
  - confidence: 0.82
