## Summary

Keep open: this PR targets a plausible outbound ordering bug, but it is not mergeable as-is because the lock is after the durable queue boundary, the key uses raw route inputs, real affected-channel proof is still missing, and the branch includes an unrelated Mattermost dependency change.

## What This Changes

Adds an in-memory outbound DeliverySerializer around core delivery calls, serializer tests, and a Mattermost typebox dependency plus lockfile entry.

## Best Possible Solution

Serialize at the durable outbound intent boundary with canonical channel/account/recipient/thread route identity, preserve abort and timeout behavior, remove unrelated dependency churn, and prove the fix with a real affected-channel run.

## Reproduction Assessment

No high-confidence live reproduction is established. Source inspection and the author's mock signal-cli run support the concurrent-send ordering race, but the PR still lacks real affected-channel proof on current main.

## Solution Assessment

No. The proposed patch is not the best fix because it serializes after queue persistence and derives the mutex key from raw inputs instead of canonical route identity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move the ordering lock before queue persistence:** `src/infra/outbound/deliver.ts:526-528`
  - body: The serializer is entered only after enqueueDelivery completes. If an earlier same-target send is slower writing its queue entry, a later send can reach the mutex first and deliver first, so FIFO by send initiation is still not guaranteed.
  - confidence: 0.9
- **[P2] Use canonical route identity for the serializer key:** `src/infra/outbound/deliver.ts:523`
  - body: The key is built from raw channel/account/to values and omits thread identity. Account and target identity are normalized later, so equivalent chats can bypass the lock while distinct threads can block each other.
  - confidence: 0.88
