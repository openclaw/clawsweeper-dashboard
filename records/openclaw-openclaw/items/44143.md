## Summary

Keep this PR open because it targets a real outbound ordering bug and has useful Telegram proof, but the patch is not merge-ready: the serializer is acquired too late to guarantee FIFO, keys are built from raw rather than canonical routed destinations, and stalled platform sends can now head-of-line block a recipient lane.

## What This Changes

Adds an in-memory outbound `DeliverySerializer` around `deliverOutboundPayloadsCore` with tests for same-key FIFO, cross-key concurrency, failures, synchronous throws, and cleanup.

## Best Possible Solution

Move serialization to the canonical outbound lane boundary before queue persistence/claim ordering can invert sends, derive the lane from normalized channel/account/recipient/thread identity, and ensure held platform sends are abort or timeout bounded before refreshing Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the lane before queue persistence can reorder sends:** `src/infra/outbound/deliver.ts:1278-1279`
  - body: The serializer is entered only around `deliverOutboundPayloadsCore`, after the durable queue/enqueue path has already run. Two concurrent same-target calls can complete queue persistence or active-claim setup in the opposite order and then enter this lane reversed, so the later logical send can still deliver first; the lane needs to cover the queue/order boundary or use queue order as the canonical FIFO source.
  - confidence: 0.9
- **[P1] Derive the lane key from the canonical routed destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The key uses raw `params.accountId` and `params.to`, but existing outbound routing normalizes accounts, recipients, and thread destinations later. Equivalent recipients can bypass serialization while distinct threaded destinations can share one lane, so build the key from the canonical routed channel/account/recipient/thread identity instead of raw input strings.
  - confidence: 0.87
- **[P1] Bound platform sends while holding the recipient lane:** `src/infra/outbound/deliver.ts:1278-1279`
  - body: The lane advances only after the platform send promise settles. Adapter paths such as Discord webhook sends can wait on an unbounded `fetch()`, so one stalled same-recipient send can now block every later reply to that chat until restart; add timeout/abort coverage or otherwise prove held sends are bounded before merging this shared serializer.
  - confidence: 0.84
