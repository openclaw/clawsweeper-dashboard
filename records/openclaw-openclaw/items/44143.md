## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Derive the lane from the routed destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The key is built from raw `params.channel`, `params.accountId`, and `params.to` before channel routing/canonicalization and without `threadId`. Discord sends to different threads can share one raw `to` while routing to different platform destinations, and Signal/Telegram aliases can use different raw `to` values for the same chat, so this still misses the invariant the PR is trying to enforce.
  - confidence: 0.9
- **[P1] Bound sends before they can hold a recipient lane:** `src/infra/outbound/deliver.ts:1278`
  - body: `deliverySerializer.serialize` now waits for the entire platform send before releasing later sends for the same lane, but the lane itself has no timeout or abort guard. A hanging send path, such as Discord webhook fetch, can now stall every later message for that recipient until restart; previously those later sends were not held behind that platform call.
  - confidence: 0.86
