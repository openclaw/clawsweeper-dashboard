## Summary

## Best Possible Solution

Implement serialization as an abort-aware per-canonical-route delivery boundary before queue persistence and active claims, with keys derived from the final channel/account/thread destination and regression coverage in the real outbound delivery wrapper.

No. This is a plausible mitigation, but the best fix needs the lock before queue work, canonical destination keying, and an abort/timeout bound so it preserves FIFO without creating a stuck recipient queue.

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Acquire the destination lock before queue work:** `src/infra/outbound/deliver.ts:1278`
  - body: This locks only around `deliverOutboundPayloadsCore`, after `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` have already run. If the earlier send has slower queue I/O and the later send reaches this line first, the later payload still gets delivered first, so the FIFO guarantee does not hold.
  - confidence: 0.91
- **[P2] Key the mutex from the canonical route:** `src/infra/outbound/deliver.ts:1274`
  - body: The key uses raw channel/account/to and omits `threadId`, but threaded adapters route by the resolved thread destination later. That can serialize unrelated threads together while equivalent same-recipient spellings still bypass the lock, leaving both ordering and concurrency wrong.
  - confidence: 0.87
- **[P1] Bound stalled sends before chaining later deliveries:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: The serializer advances only when `fn` settles. Core sends include direct HTTP paths such as Discord webhooks, so one hung platform call now keeps every later reply for that destination waiting until process restart; add an abort/timeout bound before holding the chain.
  - confidence: 0.86
