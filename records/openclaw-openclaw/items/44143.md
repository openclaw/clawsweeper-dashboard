## Summary

Keep open: the PR targets a real user-visible ordering bug and has useful Telegram proof, but the current patch is not merge-ready because it serializes too late, keys the lane from raw destination fields, and can turn a stalled adapter send into a same-recipient availability block.

## What This Changes

The branch adds an in-memory `DeliverySerializer` around outbound delivery keyed by channel/account/recipient, plus unit tests for ordering, concurrency, error handling, synchronous throws, and cleanup.

## Best Possible Solution

Adopt the ordering fix only after the per-destination lane wraps queue persistence, active claiming, send, and cleanup; derives its key from canonical target/account/thread data; and has bounded abort/stall behavior with refreshed Telegram proof.

## Reproduction Assessment

Yes at source/proof level, but not by a fresh current-main local run. Mantis Telegram Desktop proof shows the visible long-send/short-follow-up inversion on the baseline and the source path on current main still lacks a generic outbound serializer.

## Solution Assessment

No. The PR is a plausible mitigation, but the best fix needs to serialize before queue persistence/claims, key from canonical channel targets, and make the lane abort- and stall-aware.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Acquire the delivery lane before queue work can reorder sends:** `src/infra/outbound/deliver.ts:1278-1279`
  - body: The new lock is taken only around `deliverOutboundPayloadsCore`, after `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` have both awaited. Two same-recipient sends can therefore enter the serializer in queue-write or claim completion order instead of caller order, so the later reply can still deliver first. Acquire the per-destination lane before persisting/claiming the queue entry and keep queue cleanup inside that lane.
  - confidence: 0.93
- **[P1] Key delivery lanes from canonical channel targets:** `src/infra/outbound/deliver.ts:1274`
  - body: `serializerKey` uses raw `params.accountId` and `params.to` and omits thread/topic identity, while current channel paths normalize account IDs and destinations later. Equivalent destinations can bypass the serializer, and distinct threads can block one another. Build the key from the canonical target ref, including normalized account and thread/topic identity.
  - confidence: 0.9
- **[P2] Make queued lanes abort- and stall-aware:** `src/infra/outbound/deliver.ts:1278-1279`
  - body: Every later same-key delivery now waits for this adapter promise, but the serializer has no abort or timeout/release path. A cancelled or never-settling send can wedge later replies to that conversation until restart, especially because formatted adapter abort handling is not uniformly propagated from core. Add bounded release/abort behavior and tests before holding this lane around channel adapter work.
  - confidence: 0.82
