## Summary

Keep open: the reported outbound ordering bug is real and the Telegram proof is strong, but this patch is not merge-ready because the new lock is acquired after durable queue work, keys lanes from raw/non-threaded delivery inputs, and can turn a stalled platform send into a per-recipient head-of-line block.

## What This Changes

The PR adds an in-memory DeliverySerializer around outbound payload delivery and unit tests to serialize concurrent sends for the same raw channel/account/recipient key.

## Best Possible Solution

Serialize on a canonical delivery-lane identity before queue persistence and active-claim work, include normalized account/target/thread semantics, and bound or abort platform sends so a stuck call cannot hold the lane indefinitely.

## Reproduction Assessment

Yes for the reported symptom by source/proof, but not as a fresh current-main local run. The PR body gives a live Telegram inversion and Mantis run 27906272198 captured baseline/candidate Telegram Desktop ordering evidence for this branch.

## Solution Assessment

No. The in-memory serializer is a plausible mitigation, but the best fix must lock the canonical destination before durable queue work and prevent stalled platform sends from holding the lane forever.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Acquire the lane before queue work can reorder sends:** `src/infra/outbound/deliver.ts:1278-1279`
  - body: The serializer is entered only after `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` have both completed, so two same-target sends can still swap order while doing queue I/O and the later send can acquire the lock first. That leaves the central FIFO guarantee broken; the lane needs to be acquired before any per-send durable queue work that can reorder callers, or the queue itself needs to supply the ordering key.
  - confidence: 0.88
- **[P1] Key delivery lanes from canonical targets:** `src/infra/outbound/deliver.ts:1274`
  - body: The key uses raw `params.channel`, `params.accountId`, and `params.to`, but current delivery code normalizes account ids, strips Signal prefixes, and rewrites Discord thread sends to the thread target later. Equivalent sends can bypass the mutex while distinct threads can block each other, so derive the lane from the canonical delivery identity, including thread/topic identity where it is part of the destination.
  - confidence: 0.87
- **[P1] Bound stalled sends before holding a delivery lane:** `src/infra/outbound/deliver.ts:1278-1279`
  - body: Once a call enters `deliverySerializer.serialize`, any platform send that never settles prevents all later same-lane sends from running. That is a new availability regression for paths like Discord webhook delivery, which awaits `fetch()` without a timeout; add a bounded timeout/abort/release strategy before making this lock the shared outbound path.
  - confidence: 0.79
