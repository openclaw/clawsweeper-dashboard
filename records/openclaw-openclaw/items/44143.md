## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the lane before queue work can reorder sends:** `src/infra/outbound/deliver.ts:1275-1276`
  - body: The serializer is entered only after enqueueDelivery and withActiveDeliveryClaim. If an earlier same-target call is slower writing or claiming its queue entry, a later call can reach the serializer first and deliver first, so this does not guarantee FIFO by send initiation.
  - confidence: 0.91
- **[P1] Key delivery lanes from canonical targets:** `src/infra/outbound/deliver.ts:1271`
  - body: The lane key uses raw channel/accountId/to, but current adapters normalize account ids and targets later, and threadId can be the real destination. Equivalent sends can bypass the lane while distinct threads/topics can block each other; derive the key from normalized account plus canonical target/thread identity.
  - confidence: 0.88
- **[P1] Bound stalled sends before holding the lane:** `src/infra/outbound/delivery-serializer.ts:37`
  - body: A lane advances only when fn settles. Because platform paths such as Discord webhook fetch have no visible timeout or abort signal, one hung request can hold the same-recipient lane forever and stall every later reply to that conversation until restart.
  - confidence: 0.86
