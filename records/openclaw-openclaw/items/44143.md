## Best Possible Solution

Implement an abort-aware per-canonical-route delivery boundary before reorderable queue and claim work, with regression coverage for queue-order inversion, normalized/threaded destinations, and stalled sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Acquire the destination lock before queue work:** `src/infra/outbound/deliver.ts:1278-1283`
  - body: The serializer is entered only after `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` complete. If an earlier same-target send has slower queue I/O, a later send can reach this call first and deliver first, so the FIFO guarantee still does not hold.
  - confidence: 0.92
- **[P2] Key the mutex from the canonical route:** `src/infra/outbound/deliver.ts:1274`
  - body: The key uses raw `channel`, `accountId`, and `to`, but adapters normalize account/recipient values and route by thread later. That can both miss equivalent same-destination sends and serialize unrelated threaded conversations together.
  - confidence: 0.88
- **[P1] Bound stalled sends before chaining later deliveries:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: The serializer advances only when `fn` settles. Since core delivery includes HTTP paths such as Discord webhooks, one hung platform call can now keep every later same-destination reply waiting until process restart.
  - confidence: 0.86
