## Summary

This PR is still necessary because current main and the latest release do not serialize same-recipient outbound delivery, and the Telegram proof now shows the intended real transport behavior. It is not merge-ready because the lock is taken after durable queue work, uses raw destination fields instead of canonical channel identity, and can hold later same-recipient sends behind a stalled platform call.

## What This Changes

The branch adds an in-memory DeliverySerializer around outbound payload delivery and tests same-key FIFO ordering, cross-key concurrency, error propagation, synchronous throws, and queue cleanup.

## Best Possible Solution

Make the durable outbound delivery boundary own same-target ordering before enqueue/claim, derive the key from channel-owned canonical destination identity including thread/topic where applicable, and add bounded or abort-aware waits before merging.

## Reproduction Assessment

Yes. The source path is clear and the Mantis baseline Telegram proof shows the visible inversion, although I did not run a fresh live repro against the exact current main SHA during this read-only review.

## Solution Assessment

No. The serializer is a plausible direction, but this implementation is too late in the queue flow and uses the wrong identity boundary; the narrow maintainable fix belongs at the durable delivery/canonical destination seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Acquire serialization before queue persistence:** `src/infra/outbound/deliver.ts:1274`
  - body: This lock is entered only inside `deliverOutboundPayloadsWithQueueCleanup`, after the caller has already awaited `enqueueDelivery` and the active-claim path. If the earlier same-target send is slower in queue I/O or claim setup, a later send can reach this serializer first, so the patch still does not guarantee FIFO by send initiation.
  - confidence: 0.92
- **[P2] Use canonical channel destination identity:** `src/infra/outbound/deliver.ts:1272`
  - body: The key is built from raw `channel`, `accountId`, and `to`, but adapters normalize or reroute destination identity later; Discord threaded sends become `channel:${threadId}` and Signal strips a `signal:` prefix. Equivalent chats can bypass the lock, and unrelated threaded chats can block each other.
  - confidence: 0.88
- **[P2] Bound stalled sends before holding the recipient queue:** `src/infra/outbound/deliver.ts:1274`
  - body: The serializer waits for the whole platform adapter promise before releasing later sends to that recipient. Discord webhook delivery awaits a fetch without a local timeout or abort signal, so one stalled request can wedge every later same-key reply until process restart.
  - confidence: 0.84
