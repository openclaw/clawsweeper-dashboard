## Summary

Keep this PR open: current main still lacks a same-destination outbound serializer and the Telegram proof shows a real ordering bug, but the submitted patch remains unsafe to merge because it locks too late, keys before canonical routing, and can head-of-line block later replies behind a stalled send.

## What This Changes

The PR adds an in-memory outbound delivery serializer around core payload delivery plus serializer tests for FIFO ordering, error handling, sync throws, concurrency across keys, and cleanup.

## Best Possible Solution

Keep the useful serializer direction, but move the ordering boundary before queue/claim races, key it from canonical normalized account plus routed recipient/thread identity, and bound stalled sends with abort or timeout behavior before merge.

Full review comments:

- **[P1] Acquire the serializer before queue work starts:** `src/infra/outbound/deliver.ts:1278`
  - body: The lock is taken only inside `deliverOutboundPayloadsWithQueueCleanup`, after `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` have already run. If an earlier same-target call is slower in queue/claim work, a later call can reach `serialize()` first and still deliver first, so the patch does not guarantee FIFO by send initiation.
  - confidence: 0.9
- **[P1] Derive the key from the routed destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The serializer key uses raw `accountId` and `to`, but channel code later normalizes accounts and recipients and can reroute by thread/topic. Equivalent chats can bypass the lock, while distinct Discord threads can block each other unnecessarily.
  - confidence: 0.86
- **[P2] Bound stalled sends before they hold the lane:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: The serializer waits for `fn()` to settle before advancing the same-recipient chain. A platform request with no timeout or abort support, such as the Discord webhook fetch path, can hold every later send for that key until process restart.
  - confidence: 0.82
