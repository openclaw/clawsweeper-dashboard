## Summary

Keep open: the PR addresses a real outbound message-ordering bug and has useful Telegram proof, but the proposed lane currently uses raw request identity, is acquired after reorderable queue work, and can hold later same-recipient sends behind an unbounded platform send.

## What This Changes

This PR adds an in-memory outbound delivery serializer, wraps core outbound delivery for same channel/account/recipient sends, and adds serializer unit tests for ordering, errors, sync throws, concurrency, and cleanup.

## Best Possible Solution

Repair the branch so canonical destination identity defines the lane, lane acquisition covers reorderable queue work, and each lane-held send has a bounded abort or timeout release path.

## Reproduction Assessment

Yes. Concurrent same-recipient outbound sends reproduce the class of failure; Mantis Telegram Desktop proof shows the baseline inversion, and current main lacks a generic same-destination delivery lane.

## Solution Assessment

No. A serializer is a plausible fix, but this implementation is not the best merge shape until identity, lock placement, and bounded release semantics match real channel delivery.

Full review comments:

- **[P1] Key the lane from canonical delivery identity:** `src/infra/outbound/deliver.ts:1274`
  - body: The serializer key uses raw channel, accountId, and to values, but account IDs and channel targets are canonicalized later and threaded channels include thread identity in the real destination. Equivalent conversations can bypass the lane while distinct threads can block each other, so derive the lane from canonical channel/account/target/thread identity.
  - confidence: 0.9
- **[P1] Release the lane when platform sends stall:** `src/infra/outbound/deliver.ts:1278`
  - body: The new lane is held across platform delivery, but paths such as Discord webhooks await fetch without a timeout or abort signal. One stalled request can wedge every later same-lane reply until restart, so lane-held sends need a bounded release path.
  - confidence: 0.86
- **[P2] Acquire the lane before queue work can reorder sends:** `src/infra/outbound/deliver.ts:1278`
  - body: The serializer is entered only after enqueueDelivery and the active-delivery claim, so two same-target sends can still invert if the earlier request has slower queue I/O and reaches the lane second. Acquire the lane before reorderable queue work if FIFO is meant to follow send initiation order.
  - confidence: 0.82
