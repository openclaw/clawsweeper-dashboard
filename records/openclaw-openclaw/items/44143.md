## Summary

Keep open: the outbound ordering race is plausible and current main does not already provide a generic DeliverySerializer, but this PR is not ready because the lock is too late, the key is not canonical, real-channel proof is missing, and the branch includes an unrelated Mattermost dependency change.

## What This Changes

The branch adds an in-memory DeliverySerializer around outbound delivery, unit coverage for that serializer, and a Mattermost TypeBox dependency plus lockfile entry.

## Best Possible Solution

Serialize at the durable outbound intent boundary before queue writes, use a canonical adapter-resolved key that includes account, recipient, and thread identity, preserve abort/timeout behavior, remove the unrelated dependency, and require real-channel proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection and the author’s mock signal-cli run support a concurrent-send ordering race, but the PR still lacks real-channel before/after proof.

## Solution Assessment

No. The current PR is not the best fix because it locks after queue persistence and keys on raw inputs; the safer solution is to lock before durable queue ordering can invert and derive the key from canonical destination identity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Acquire the ordering lock before queue writes:** `src/infra/outbound/deliver.ts:526-528`
  - body: The serializer wraps only `deliverOutboundPayloadsCore`, after `enqueueDelivery` has already completed. If an earlier same-target send has slower queue I/O, a later send can enter the serializer first and deliver first, so FIFO by send initiation is still not guaranteed.
  - confidence: 0.9
- **[P2] Build the mutex key from canonical destination identity:** `src/infra/outbound/deliver.ts:523`
  - body: The key uses raw `accountId` and `to` and omits thread identity. Account IDs and adapter targets normalize later, so equivalent conversations can bypass the lock while distinct threaded conversations can be serialized together.
  - confidence: 0.88
