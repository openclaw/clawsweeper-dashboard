## Summary

Keep open. The underlying outbound ordering bug is real and not implemented on current main, but this branch is not merge-ready: it locks after durable queue persistence/claiming, keys on raw destination fields, can wedge later same-recipient sends behind a stalled adapter promise, and the candidate Telegram recording still shows ORDER B above ORDER A.

## What This Changes

The PR adds an in-memory per-channel/account/recipient DeliverySerializer around outbound payload delivery plus unit tests for FIFO order, errors, sync throws, concurrency, and cleanup.

## Best Possible Solution

Move ordering ownership to the outbound intent boundary: derive a canonical destination key before queue persistence/claiming, keep durable recovery semantics, make waits abortable or bounded, and prove the fixed path with a real Telegram or Discord ordering run.

## Reproduction Assessment

Yes. The Mantis baseline Telegram recording and the PR body's production observation show same-recipient message inversion, and current main has no per-destination serialization around outbound delivery.

## Solution Assessment

No. The proposed serializer is only a plausible mitigation because it locks after queue persistence and before channel-specific destination canonicalization; the best fix should serialize the canonical durable send intent and preserve cancellation/recovery behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire serialization before queue persistence:** `src/infra/outbound/deliver.ts:1274-1289`
  - body: The lock is taken only inside `deliverOutboundPayloadsWithQueueCleanup`, after each call has already awaited `enqueueDelivery(...)` and entered its active claim. If two same-target sends race and the earlier send has slower queue persistence, the later send can reach this serializer first and still deliver first, leaving the Telegram/Discord ordering bug this PR is meant to fix.
  - confidence: 0.91
- **[P2] Key serialized sends by canonical destination:** `src/infra/outbound/deliver.ts:1272`
  - body: The serializer key uses raw `channel/accountId/to`, but transports canonicalize destinations later and threaded sends can route by `threadId`. Signal strips a `signal:` prefix during send, while Discord can route a threaded send to `channel:${threadId}`, so equivalent chats can bypass the lock and unrelated threaded chats can block each other.
  - confidence: 0.86
- **[P2] Bound stalled sends before blocking a recipient:** `src/infra/outbound/delivery-serializer.ts:29-31`
  - body: The mutex waits for the whole adapter promise before advancing the key. Since the Discord webhook path still awaits `fetch` without an abort signal or timeout, one hung platform request can keep this tail unresolved and block every later send to that recipient until process restart, which is a new availability regression for that chat.
  - confidence: 0.82
