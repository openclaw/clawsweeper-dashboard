## Summary

Keep this PR open: it targets a real outbound message-ordering bug and has useful Telegram proof, but the current patch is still not merge-ready because the lane is acquired too late, keyed from raw destination fields, and can create a new per-recipient availability stall.

## What This Changes

Adds an in-memory outbound DeliverySerializer around platform sends, plus tests for same-target FIFO ordering, cross-target concurrency, failures, sync throws, and cleanup.

## Best Possible Solution

Move serialization to the earliest outbound lane boundary, key it from canonical normalized channel/account/recipient/thread identity, and bound serialized sends with abort or timeout handling before merge.

## Reproduction Assessment

Yes for the behavior class: current main still queues and sends lower-level outbound deliveries independently, and inspected Mantis Telegram screenshots show baseline inversion versus candidate ordering. I did not run a fresh current-main live reproduction because this is a read-only review.

## Solution Assessment

No: the serializer is a plausible mitigation, but it is not the best fix until it runs before queue work, uses canonical routed identity, and bounds stalled sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Acquire the lane before queue persistence can reorder sends:** `src/infra/outbound/deliver.ts:1278`
  - body: The new lock is entered only inside `deliverOutboundPayloadsWithQueueCleanup`, after each call has already awaited `enqueueDelivery(...)` and then `withActiveDeliveryClaim(...)`. If an earlier same-target send has slower queue I/O or claim work, a later send can reach this serializer first and still deliver first, so FIFO by send initiation is not guaranteed.
  - confidence: 0.9
- **[P1] Derive the lane key from canonical routed destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The lane key uses raw `channel`, `accountId`, and `to`, and it omits `threadId`. Current adapters normalize account ids and targets later, including Discord threaded sends routing to `channel:<threadId>` and Signal stripping `signal:`, so equivalent destinations can bypass serialization while distinct threads can share the wrong lane.
  - confidence: 0.9
- **[P1] Bound platform sends while holding the lane:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: The serializer advances only after `fn` settles. A stalled platform delivery now holds the recipient lane, so every later same-recipient reply can remain stuck until restart; the lane needs timeout or abort behavior while the send is serialized.
  - confidence: 0.84
