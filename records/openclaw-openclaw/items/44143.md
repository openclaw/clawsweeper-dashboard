## Summary

Keep open: current main does not implement outbound per-recipient serialization, but this PR should not merge as-is because its proof is mock-only, the lock is placed after the durable queue write, the key uses raw route inputs, and the branch includes unrelated dependency churn.

## What This Changes

The PR adds an in-memory outbound DeliverySerializer around core delivery calls, serializer tests, and a Mattermost typebox dependency plus lockfile entry.

## Best Possible Solution

Serialize at the durable outbound intent boundary with canonical channel/account/recipient/thread identity, preserve queue/claim/abort behavior, remove unrelated dependency churn, and require real affected-channel proof.

## Reproduction Assessment

No high-confidence live reproduction is established. Source inspection and the author's mock signal-cli run support the concurrent-send ordering race, but the PR still lacks real affected-channel proof on current main.

## Solution Assessment

No. The proposed patch is not the best fix because it serializes after queue persistence and derives the mutex key from raw route inputs instead of canonical route identity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move the ordering lock before queue persistence:** `src/infra/outbound/deliver.ts:526-528`
  - body: The serializer is entered only after enqueueDelivery completes. If an earlier same-target send is slower writing its queue entry, a later send can reach the mutex first and deliver first, so FIFO by send initiation is still not guaranteed.
  - confidence: 0.9
- **[P2] Use canonical route identity for the serializer key:** `src/infra/outbound/deliver.ts:523`
  - body: The key is built from raw channel/account/to values and omits thread identity. Account and target identity are normalized later, so equivalent chats can bypass the lock while distinct threads can block each other.
  - confidence: 0.88
