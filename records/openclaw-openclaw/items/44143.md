## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Serialize before queue work can reorder sends:** `src/infra/outbound/deliver.ts:1269-1272`
  - body: The lock is taken only after the delivery has already been enqueued and claimed. If an earlier same-target send is slower in that queue/claim path, a later send can reach `serialize()` first and still be delivered first, so the patch does not guarantee FIFO by send initiation.
  - confidence: 0.9
- **[P1] Key the lock from canonical destination identity:** `src/infra/outbound/deliver.ts:1269`
  - body: The key uses raw `accountId` and `to`, but account IDs and recipients are normalized or rerouted later by channel code, including thread-aware Discord targets and prefix-stripping Signal/Telegram targets. Equivalent chats can bypass the lock, while distinct threads can block each other unnecessarily.
  - confidence: 0.86
- **[P2] Bound stalled sends before queuing later replies behind them:** `src/infra/outbound/deliver.ts:1272-1288`
  - body: Wrapping the platform send in a per-recipient chain means one hung adapter call now wedges every later send for that target. The Discord webhook path can wait on a bare fetch without an explicit timeout, so the serializer needs timeout/abort handling before it becomes the delivery gate.
  - confidence: 0.82
