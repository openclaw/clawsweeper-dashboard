## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Key the lane from the canonical destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The new mutex key uses raw `channel`, `accountId`, and `to`, but delivery later routes through channel-specific normalization. Signal strips a leading `signal:` prefix and Discord thread sends can resolve to `channel:${threadId}`, so equivalent same conversations can still use different lanes and race while distinct threads under the same `to` can block each other. Derive the lane from the normalized target/account/thread identity before relying on FIFO.
  - confidence: 0.9
- **[P1] Bound platform sends before they hold a recipient lane:** `src/infra/outbound/deliver.ts:1278`
  - body: After this call, one never-settling platform send owns the per-recipient queue. The Discord webhook path calls `fetch` without a timeout or signal, and the formatted adapter wrappers do not forward `abortSignal`, so later replies to that lane can remain stuck until restart. Add abort/timeout handling before serializing production sends.
  - confidence: 0.88
- **[P2] Acquire the ordering lane before queue work can reorder sends:** `src/infra/outbound/deliver.ts:1278`
  - body: The serializer is entered only after `enqueueDelivery` and the active claim. If an earlier same-target send takes longer to persist than a later one, the later call can acquire the lane first and still deliver first, so this does not guarantee FIFO by send initiation. Capture the lane/order token before awaited queue I/O or define and test queue-completion order as the intended contract.
  - confidence: 0.78
