## Summary

Keep open: the bug and proposed direction are useful, but the current diff is not merge-ready because the mutex starts after ordering-sensitive queue work, keys raw destinations, and can wedge later sends behind a stalled adapter call; the candidate Telegram proof also does not visibly show ordered after-fix delivery.

The branch adds an in-memory per-key DeliverySerializer around outbound delivery plus unit tests for FIFO ordering, errors, sync throws, concurrency, and cleanup.

## Best Possible Solution

Move ordering to the canonical durable delivery boundary, key by normalized account/recipient/thread identity, bound or abort waiting sends, and attach corrected redacted Telegram proof.

## Solution Assessment

No. The serializer is a plausible mitigation, but it is not the best fix because it starts after queue work and uses raw target fields instead of the canonical delivery destination.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Acquire serialization before queue persistence:** `src/infra/outbound/deliver.ts:1278`
  - body: The mutex starts only inside `deliverOutboundPayloadsWithQueueCleanup`, after `enqueueDelivery` and `withActiveDeliveryClaim` have already run. A later same-target call whose queue write finishes first can still enter the serializer first and deliver before the earlier call, so this does not guarantee FIFO by send initiation.
  - confidence: 0.97
- **[P2] Key by canonical destination identity:** `src/infra/outbound/deliver.ts:1274`
  - body: The key uses raw channel/account/recipient strings, but adapters normalize account IDs, recipient aliases, and thread or topic destinations later. Equivalent chats can bypass serialization while distinct threads can block each other; derive the key from the canonical account, recipient, and thread identity instead.
  - confidence: 0.95
- **[P2] Bound serialized sends so one stall cannot wedge a chat:** `src/infra/outbound/deliver.ts:1278-1295`
  - body: The queue advances only after the adapter promise settles. Paths such as Discord webhook sends have no local timeout or forwarded abort signal, so one stalled request can hold every later same-key delivery until process restart.
  - confidence: 0.91
