## Summary

Keep open: the ordering bug is credible and current main does not already provide a per-target outbound serializer, but this branch still serializes too late in the durable delivery path and has route-key, abort, and head-of-line blocking issues that can affect real message delivery.

## What This Changes

The PR adds an in-memory `DeliverySerializer` around outbound payload delivery and unit tests for ordering, concurrency, failures, synchronous throws, and cleanup.

## Best Possible Solution

Rework the fix at the durable route boundary so the ordering key is canonical, thread-aware, abort/timeout-aware, and covered by focused queue-order and live-channel proof.

## Reproduction Assessment

Yes at source level: current main has no per-target outbound serializer, and the delivery path writes each queue entry independently before calling the platform adapter. I did not run a live current-main repro in this read-only review.

## Solution Assessment

No: the proposed serializer is directionally useful, but placing it after durable enqueue and keying it from raw fields is not the narrowest maintainable fix for ordered delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Serialize before durable enqueue:** `src/infra/outbound/deliver.ts:1278`
  - body: Both callers still await `enqueueDelivery(...)` before reaching this new serializer. If the earlier send has a slower queue write, the later send can acquire the mutex first and hit the platform first, so this does not guarantee FIFO ordering by send initiation.
  - confidence: 0.9
- **[P2] Use a canonical route key:** `src/infra/outbound/deliver.ts:1274`
  - body: The key is built from raw `accountId` and `to` and omits `threadId`, but the outbound path later normalizes account/target values and Discord rewrites threaded sends to the thread channel. Equivalent chats can bypass the serializer, while distinct threads can block each other.
  - confidence: 0.88
- **[P2] Make queued waits abort-aware:** `src/infra/outbound/deliver.ts:1278`
  - body: A send can now be persisted and then sit behind the serializer before `deliverOutboundPayloadsCore` checks `abortSignal`. If the user cancels and the process restarts during that wait, recovery can replay the queued entry without the original abort signal.
  - confidence: 0.83
- **[P2] Bound stalled platform sends:** `src/infra/outbound/deliver.ts:1278`
  - body: The new per-recipient mutex makes one stalled platform call block every later same-key reply. At least the Discord webhook path currently calls `fetch` without a passed abort signal or timeout, so this patch can turn a single hung send into an availability issue for that chat.
  - confidence: 0.78
