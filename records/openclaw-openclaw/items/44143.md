## Summary

Keep open: this is a useful, well-proven bug fix direction, but the current patch still has blocking delivery-order and availability defects because the lane is acquired after queue/claim work, keyed before canonical routing, and held across unbounded platform sends.

## What This Changes

The branch adds an in-memory DeliverySerializer around outbound payload delivery and unit tests for FIFO same-key sends, cross-key concurrency, error propagation, synchronous throws, and cleanup.

## Best Possible Solution

Move serialization to the earliest outbound lane boundary, key it from canonical channel/account/recipient/thread identity, and make lane-held sends timeout- and abort-aware before merge.

Yes for the behavior class: current main queues and sends concurrent outbound deliveries independently, and Mantis Telegram Desktop proof shows baseline inversion versus candidate ordering. I did not execute a fresh local reproduction because this is a read-only review.

## Solution Assessment

No: the serializer is a plausible mitigation, but it is not the best fix until it runs before queue work, uses canonical routed identity, and bounds stalled sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the lane before queue persistence:** `src/infra/outbound/deliver.ts:1278`
  - body: The new lock is entered inside `deliverOutboundPayloadsWithQueueCleanup`, after `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` have already completed in the caller. If an earlier same-target send has slower queue I/O or claim work, a later send can reach this serializer first and still deliver first, so FIFO by send initiation is not guaranteed.
  - confidence: 0.89
- **[P1] Derive the key from canonical routed destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The lane key uses raw `params.channel`, `params.accountId`, and `params.to`, and omits thread identity. Current adapters normalize account IDs and targets later, including Discord threaded sends routing to `channel:${threadId}` and Signal stripping `signal:`, so equivalent destinations can bypass serialization while distinct threads can share the wrong lane.
  - confidence: 0.9
- **[P1] Bound sends while holding the recipient lane:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: The serializer advances only after `fn` settles. Discord webhook delivery can call `fetch` without an abort or timeout signal on this path, so one stalled platform call can now hold the recipient lane and block every later same-recipient reply until restart.
  - confidence: 0.84
