## Summary

This PR targets a real outbound ordering bug, but the proposed patch is not merge-ready because it serializes too late, keys the lock from non-canonical destination fields, and adds an unrelated Mattermost dependency change.

## What This Changes

This PR adds an in-memory outbound DeliverySerializer around delivery, serializer unit tests, and a Mattermost package/lockfile dependency entry.

## Best Possible Solution

Implement same-destination ordering at the outbound boundary before queue persistence or recovery can invert call order, derive the ordering key from canonical channel/account/recipient/thread identity, preserve cancellation and bounded transport behavior, and split unrelated package metadata out of this fix.

## Reproduction Assessment

Yes. The PR discussion gives a concrete mock HTTP delay reproduction, and current-main source inspection confirms same-recipient deliveries are independent queue entries with no destination-level lock.

## Solution Assessment

No. The serializer concept addresses the right user-visible race, but this patch is not the narrowest maintainable fix because it locks after queue I/O, uses raw destination fields, lacks liveness safeguards, and mixes in unrelated package metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Serialize before queue persistence:** `src/infra/outbound/deliver.ts:526-528`
  - body: The lock is acquired only around `deliverOutboundPayloadsCore`, after `enqueueDelivery(...)` has already completed. Two same-target sends can still invert if the earlier call has slower queue I/O and the later call reaches the serializer first, so this does not guarantee FIFO by send initiation order.
  - confidence: 0.93
- **[P2] Build the key from canonical destination identity:** `src/infra/outbound/deliver.ts:522`
  - body: The mutex key uses raw `channel`, `accountId`, and `to` values. Equivalent destinations can bypass serialization after account/recipient normalization, while threaded channels can unnecessarily serialize distinct conversations; derive the key from canonical channel/account/recipient/thread identity instead.
  - confidence: 0.9
- **[P2] Avoid wedging later sends behind a stalled delivery:** `src/infra/outbound/delivery-serializer.ts:42-44`
  - body: The serializer waits on the previous same-key promise with no abort or timeout path. A hung transport call can permanently block every later send to that recipient until process restart.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/infra/outbound/deliver.ts:526-528`
  - body: This is a user-facing delivery behavior fix, but the PR does not update `CHANGELOG.md`. Repo policy requires a changelog entry for user-visible fix changes before merge.
  - confidence: 0.78
