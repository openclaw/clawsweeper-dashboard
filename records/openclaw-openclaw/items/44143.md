## Summary

Keep open: the ordering bug is real and the Telegram proof is strong, but the patch still serializes after queue/claim work, keys from raw destination fields, and can create head-of-line stalls for later replies.

## What This Changes

The PR adds an in-memory DeliverySerializer and wraps outbound delivery core calls per raw channel/account/recipient key, with unit tests for ordering, error handling, synchronous throws, and cleanup.

## Best Possible Solution

Adopt per-destination serialization only after the lane wraps queue persistence, active claims, adapter send, and cleanup with canonical account/target/thread identity plus bounded abort or stall behavior.

## Reproduction Assessment

Yes at source/proof level, but not by a fresh local current-main run. Mantis Telegram Desktop proof shows the ordering problem on the baseline, and current main still has no generic outbound per-destination serializer.

## Solution Assessment

No. The PR is a plausible mitigation, but the best fix needs to serialize before queue persistence and active claims, key from canonical channel targets, and make the lane abort- and stall-aware.

- **[P1] Acquire the delivery lane before queue work can reorder sends:** `src/infra/outbound/deliver.ts:1278-1279`
  - body: The serializer is entered only after `enqueueDelivery` and `withActiveDeliveryClaim` have awaited. Two same-target sends can therefore reach the lock in queue-write or claim-completion order instead of caller order, so the later reply can still deliver first.
  - confidence: 0.93

  - confidence: 0.9

  - body: Every later same-key delivery now waits for the adapter promise, but the lane has no abort or timeout/release path. A canceled or never-settling send can wedge later replies to that conversation until restart.
