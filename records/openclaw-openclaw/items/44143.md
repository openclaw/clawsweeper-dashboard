## Summary

Keep this PR open, but not merge-ready: the reported ordering race is plausible and useful to fix, while the current patch locks too late, keys on non-canonical route data, and adds cancellation and head-of-line blocking risks without real after-fix channel proof.

## What This Changes

The PR adds an in-memory outbound delivery serializer keyed by channel/account/recipient plus unit tests for ordering, concurrency, failures, sync throws, and cleanup.

## Best Possible Solution

Implement ordering at the prepared durable route boundary using canonical channel/account/conversation/thread identity, abort/timeout-aware waits, focused regression tests, and real proof on an affected channel.

## Reproduction Assessment

No, I did not run a high-confidence current-main live reproduction. Source inspection shows there is no per-target outbound serializer on main and the PR body reports a live before-fix Telegram inversion, so the race is plausible but not fully reproduced here.

## Solution Assessment

No. The proposed fix locks after durable enqueue, keys on raw route fields, and does not make serialized waits abort- or timeout-aware; the safer fix belongs at the canonical durable route boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire ordering before durable enqueue:** `src/infra/outbound/deliver.ts:1277`
  - body: The serializer is taken only inside queue cleanup, but normal sends have already awaited enqueueDelivery by then. If an earlier same-target call has a slower queue write than a later one, the later call reaches the serializer first and still sends first, so FIFO by send initiation is not guaranteed.
  - confidence: 0.91
- **[P2] Key serialization by the canonical route:** `src/infra/outbound/deliver.ts:1273`
  - body: The key uses raw channel/accountId/to and omits the effective thread or topic route. Adapters normalize accounts and recipients later, and Discord rewrites threaded sends to channel:<threadId>, so equivalent destinations can bypass the mutex while distinct threads can block each other.
  - confidence: 0.88
- **[P2] Reject aborted sends before persisted waits replay:** `src/infra/outbound/deliver.ts:1277`
  - body: Because the entry is persisted before this wait and abortSignal is only checked inside the serialized core delivery, an aborted send can sit behind another same-key send after being enqueued. If the process restarts before the wait reaches the abort check, recovery no longer has that signal and can send work the user canceled.
  - confidence: 0.82
- **[P2] Bound serialized platform sends:** `src/infra/outbound/deliver.ts:1277`
  - body: The new per-recipient chain waits for the previous platform send to settle before later same-key sends can start. Some channel paths still have unbounded platform calls, so one stalled request can now block every later delivery to that chat until process restart.
  - confidence: 0.78
