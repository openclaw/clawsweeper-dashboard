## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Acquire the destination lock before queue work:** `src/infra/outbound/deliver.ts:1277`
  - body: The serializer is entered only inside deliverOutboundPayloadsWithQueueCleanup, but each call has already awaited enqueueDelivery and withActiveDeliveryClaim. If an earlier same-target call has slower queue or claim work, a later call can reach this lock first and still deliver first.
  - confidence: 0.88
- **[P2] Key the mutex from the canonical route:** `src/infra/outbound/deliver.ts:1273`
  - body: This raw channel/account/to string bypasses route and account normalization and omits thread identity. Equivalent destinations can use different locks, while distinct threaded conversations can share one lock unnecessarily.
  - confidence: 0.84
- **[P2] Bound stalled sends before chaining later deliveries:** `src/infra/outbound/deliver.ts:1277`
  - body: Once a send enters this chain, later same-target deliveries wait for it to settle. Sibling paths include Discord webhook sends that await fetch without a local timeout or passed abort signal, so a hung request can wedge that destination until restart.
  - confidence: 0.8
