## Summary

Keep this PR open, but it should not merge as-is: the reported ordering problem is plausible, yet the patch still does not guarantee FIFO delivery, lacks real affected-channel proof, and includes an unrelated dependency change.

## What This Changes

The PR adds an in-memory outbound delivery serializer keyed by channel/account/recipient, serializer tests, and a Mattermost dependency plus lockfile entry.

## Best Possible Solution

Implement serialization at the durable outbound intent boundary using canonical channel/account/recipient/thread route identity, preserve abort and timeout behavior, remove the unrelated dependency, add focused regression coverage, and provide real-channel proof.

## Reproduction Assessment

No high-confidence live reproduction is established. Source inspection and the author's isolated mock signal-cli run support the concurrent-send ordering race, but the PR still lacks real affected-channel before/after proof.

## Solution Assessment

No. The current patch is not the best fix because it serializes after queue persistence and derives the mutex key from raw inputs instead of canonical route identity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move the ordering lock before queue persistence:** `src/infra/outbound/deliver.ts:526-528`
  - body: The serializer is entered only after enqueueDelivery completes. If an earlier same-target send is slower writing its queue entry, a later send can reach the mutex first and deliver first, so FIFO by send initiation is still not guaranteed.
  - confidence: 0.9
- **[P2] Use canonical route identity for the serializer key:** `src/infra/outbound/deliver.ts:523`
  - body: The key is built from raw channel/account/to values and omits threadId. Account and target identity are normalized later, and threaded channels route by thread metadata, so equivalent chats can bypass the lock while distinct threads can block each other.
  - confidence: 0.88
