Full review comments:

- **[P1] Key the lane from canonical delivery identity:** `src/infra/outbound/deliver.ts:1274`
  - body: The new key is built from raw `params.channel`, `params.accountId`, and `params.to` before account IDs, targets, and thread/topic routes are canonicalized. Same conversation aliases can bypass the lane while distinct threads can block each other, so derive the lane from canonical channel/account/target/thread identity.
  - confidence: 0.9
- **[P1] Release the lane when platform sends stall:** `src/infra/outbound/deliver.ts:1278`
  - body: The serializer holds the lane across platform delivery, but Discord webhook delivery awaits `fetch` without a signal/timeout and formatted send contexts do not currently receive `abortSignal`. One stalled or canceled send can wedge later same-lane replies until restart, so lane-held sends need a bounded abort or timeout release path.
  - confidence: 0.87
- **[P2] Acquire the lane before queue work can reorder sends:** `src/infra/outbound/deliver.ts:1278`
  - body: The PR enters the serializer only after `enqueueDelivery` and the active-delivery claim, so an earlier send with slower queue I/O can reach the lane after a later send and still deliver second. Acquire the lane before reorderable queue work if FIFO means send initiation order.
  - confidence: 0.84
