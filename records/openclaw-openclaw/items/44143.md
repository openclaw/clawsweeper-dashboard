## Summary

Keep open: the bug and Telegram proof are strong, but the patch is not merge-ready because the serializer is acquired after durable queue work, uses a raw non-canonical destination key, and can turn a stalled platform send into a same-recipient delivery wedge.

## Best Possible Solution

Serialize a canonical destination lane before durable queue and claim work, using normalized channel/account/recipient/thread identity and bounded abort-aware sends with focused regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the lane before queue persistence can reorder sends:** `src/infra/outbound/deliver.ts:1273`
  - body: The serializer is acquired only inside `deliverOutboundPayloadsWithQueueCleanup`, but each call has already awaited `enqueueDelivery(...)` and the live queue claim before reaching this point. If the earlier same-recipient send has slower queue I/O, the later send can enter this serializer first and still deliver before it, so the patch does not guarantee FIFO by send initiation.
  - confidence: 0.9
- **[P1] Derive the lane key from the canonical routed destination:** `src/infra/outbound/deliver.ts:1273`
  - body: This raw `${channel}:${accountId}:${to}` key does not match the destination adapters actually send to. Discord rewrites threaded sends to `channel:<threadId>`, Signal strips prefixes like `signal:`, and account IDs are normalized elsewhere, so equivalent chats can bypass the lock while distinct threads can block each other.
  - confidence: 0.88
- **[P1] Bound stalled sends before they hold the recipient lane:** `src/infra/outbound/delivery-serializer.ts:36`
  - body: A lane advances only when the previous `fn()` settles. Some platform paths, including the Discord webhook fetch path, do not pass a timeout or abort signal, so one stalled HTTP send can now block every later same-recipient reply until process restart.
  - confidence: 0.84
