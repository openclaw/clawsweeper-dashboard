## Summary

Keep open: the ordering bug is real and current main does not provide a same-destination outbound serializer, but this PR is not merge-ready because the mutex is acquired too late, uses raw destination keys, and can create per-recipient stalls.

## What This Changes

The PR adds an in-memory outbound delivery serializer, wraps the core outbound send path, and adds focused serializer tests.

## Best Possible Solution

Move same-destination serialization to the earliest send-intent boundary before durable queue and claim work, key it from canonical route/account/thread data, and make queued waits or platform sends abort-aware or bounded.

No. The in-memory FIFO idea is a plausible fix direction, but the current placement is too late, the key is not canonical, and the chain needs bounded or abort-aware behavior before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Acquire the destination lock before queue work:** `src/infra/outbound/deliver.ts:1277`
  - body: The serializer is entered only inside `deliverOutboundPayloadsWithQueueCleanup`, but `deliverOutboundPayloadsInternal` has already awaited `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)`. If an earlier same-target call has slower queue or claim work, a later call can reach this line first and still deliver first, so the PR does not guarantee FIFO by send initiation.
  - confidence: 0.86
- **[P2] Key the mutex from the canonical route:** `src/infra/outbound/deliver.ts:1273`
  - body: This raw `channel:accountId:to` string bypasses existing route/account normalization and omits thread identity. Equivalent destinations can use different locks, while threaded destinations can share one lock unnecessarily; build the key from the canonical route, including normalized account and thread data.
  - confidence: 0.82
- **[P2] Bound stalled sends before chaining later deliveries:** `src/infra/outbound/deliver.ts:1277`
  - body: Once a send enters this chain, later same-target deliveries wait for it to settle. Sibling paths include Discord webhook sends that await `fetch()` without a local timeout or passed abort signal, so a hung request can wedge that destination until restart.
  - confidence: 0.78
