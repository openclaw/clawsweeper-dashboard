## Summary

Keep this PR open. Current main still lacks destination-level outbound serialization, so the requested behavior is not obsolete, but the proposed patch is not merge-ready because it can still reorder sends before the serializer, uses an ambiguous raw destination key, and adds an unrelated Mattermost runtime dependency.

## What This Changes

This PR adds an in-memory outbound DeliverySerializer around delivery, serializer unit tests, and a Mattermost @sinclair/typebox dependency plus lockfile entry.

## Best Possible Solution

Implement same-destination ordering at the outbound boundary before queue persistence or recovery can invert request order, derive the key from canonical normalized channel/account/recipient/thread identity, preserve cancellation and bounded transport behavior, and keep unrelated plugin dependency changes out of the fix.

## Reproduction Assessment

Yes. The PR discussion gives a concrete mock HTTP delay reproduction, and current main source inspection confirms same-recipient deliveries are independent queue entries with no per-destination lock.

## Solution Assessment

No. The serializer idea addresses the right problem, but this diff is not the narrowest maintainable solution because it locks too late, keys on raw unnormalized fields, and lacks safeguards for aborted or hung sends waiting behind the new queue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Serialize before queue writes can reorder sends:** `src/infra/outbound/deliver.ts:525-527`
  - body: The lock is acquired only after `enqueueDelivery(...)` completes, so two same-target calls can still invert if the earlier call has slower queue I/O and the later call reaches `deliverySerializer.serialize(...)` first. That preserves the current user-visible race for send initiation order; take the destination lock before the queue write or otherwise order queued entries by the original call order.
  - confidence: 0.93
- **[P2] Build the key from canonical destination identity:** `src/infra/outbound/deliver.ts:522`
  - body: The mutex key uses raw `channel`, `accountId`, and `to`, but adapters normalize account ids, recipient prefixes, and thread targets later. Equivalent destinations such as omitted/default account ids or normalized Signal targets can bypass serialization, while distinct threads can unnecessarily block each other; derive the key from normalized channel/account/recipient/thread identity.
  - confidence: 0.9
- **[P2] Avoid wedging a recipient behind a stalled send:** `src/infra/outbound/delivery-serializer.ts:42-44`
  - body: The serializer waits on the previous same-key promise without any abort or timeout path, so one hung transport call becomes a permanent head-of-line block for later messages to that recipient. Existing paths such as Discord webhook `fetch()` can wait indefinitely, making this a new liveness regression unless waiting sends can abort and transports are bounded.
  - confidence: 0.84
