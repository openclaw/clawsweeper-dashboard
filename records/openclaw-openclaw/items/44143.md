## Summary

Keep open. The reported message-ordering bug is real and has Telegram proof, but the current patch is not merge-ready because the serializer is acquired after queue work can already reorder same-target sends, it keys lanes from raw noncanonical destination fields, and it can turn a stalled send into a per-recipient head-of-line block.

## What This Changes

The PR adds a process-local DeliverySerializer around outbound delivery sends plus unit tests for per-key FIFO ordering.

## Best Possible Solution

Implement the ordering boundary at the durable outbound lane with a canonical channel target key that includes normalized account and thread identity, releases on abort/stall, and then refreshes queue integration tests plus Telegram Desktop proof.

## Solution Assessment

No. The current PR is a useful mitigation, but the best fix needs to acquire ordering before queue work, key from canonical destination identity, and avoid indefinite lane blocking on aborts or stalled adapter calls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the delivery lane before queue work can reorder sends:** `src/infra/outbound/deliver.ts:1278`
  - body: The serializer is entered only around `deliverOutboundPayloadsCore`, after `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` have already run. If two same-recipient calls race and the later call finishes that pre-lock queue work first, it acquires the lane first and the messages can still reach the adapter out of order, so this does not guarantee FIFO by send initiation.
  - confidence: 0.91
- **[P1] Key delivery lanes from the canonical outbound target:** `src/infra/outbound/deliver.ts:1274`
  - body: The lane key is built from raw `params.channel`, `params.accountId`, and `params.to`, but adapters can normalize the account/recipient and route by `threadId` later. Equivalent destinations can bypass the lock, while distinct threads under the same raw `to` can be unnecessarily serialized; derive the key from canonical channel target identity instead.
  - confidence: 0.88
- **[P1] Release queued lanes on aborts and stalled sends:** `src/infra/outbound/deliver.ts:1278`
  - body: Once this wrapper is added, a hung adapter request holds the same-recipient lane indefinitely, and formatted sends do not currently forward `abortSignal` to adapters despite the outbound context supporting it. A stalled Discord webhook or canceled formatted send can therefore block later replies to that destination until restart.
  - confidence: 0.82
