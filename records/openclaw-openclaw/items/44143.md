## Summary

Keep this PR open: the outbound ordering bug is real and the Telegram proof is strong, but the current patch should not merge because the serializer keys lanes from pre-adapter raw destination data and can wedge later replies behind one non-settling send.

## What This Changes

The PR adds an in-memory `DeliverySerializer` singleton around outbound payload delivery, keyed by channel/account/recipient, with unit tests for FIFO ordering, concurrency, failures, synchronous throws, and cleanup.

## Best Possible Solution

Land a repaired serializer that uses the canonical routed channel/account/thread destination as the lane key and bounds or cancels platform sends while preserving durable queue recovery and cross-target concurrency.

## Reproduction Assessment

Yes for the bug class: Mantis baseline proof on the PR base shows a short second Telegram delivery arriving before the long first delivery, and current `main` still lacks a serializer. I did not run a fresh live current-main repro, so the structured status is source-reproducible rather than reproduced.

## Solution Assessment

No, this is not the best fix as-is. The correct shape is a canonical routed-destination lane with cancellation/timeout protection, not a raw `channel:accountId:to` key wrapped around potentially unbounded platform sends.

Add timeout/abort handling and regression coverage before holding a lane globally.
