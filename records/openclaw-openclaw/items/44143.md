## Summary

This PR should stay open because the outbound ordering bug is real and still unsolved on current main, but the proposed serializer is not merge-ready: it locks after queue persistence/claim, uses raw destination identity, can wedge later sends, and the candidate Telegram proof still shows ORDER B above ORDER A.

## What This Changes

The PR adds an in-memory per-channel/account/recipient DeliverySerializer around outbound payload delivery and unit tests for same-key FIFO ordering, errors, synchronous throws, concurrency, and cleanup.

## Best Possible Solution

Rework same-target ordering at the durable delivery intent or queue boundary, key it from canonical channel/account/thread identity, bound or abort stalled sends, and rerun real Telegram or Discord proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the bug class: the Mantis baseline recording shows the Telegram ordering inversion and current main still has no same-target serialization before platform send, though I did not run a fresh live repro on the exact current main SHA.

## Solution Assessment

Is this the best way to solve the issue? No. A per-target serializer is plausible, but this placement and raw keying are too late; the safer fix belongs at the durable delivery-intent boundary with canonical destination identity and bounded waits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Acquire serialization before queue persistence:** `src/infra/outbound/deliver.ts:1278`
  - body: The serializer is taken only inside `deliverOutboundPayloadsWithQueueCleanup`, after each call has already awaited `enqueueDelivery` and the active-claim path. If the earlier send has slower queue I/O, a later same-target send can enter the serializer first, so the PR still fails the FIFO guarantee it is meant to provide.
  - confidence: 0.92
- **[P2] Use a canonical destination key:** `src/infra/outbound/deliver.ts:1274`
  - body: The serializer key uses raw `accountId` and `to` and omits `threadId`, while adapters canonicalize later. Discord routes threaded sends through `channel:${threadId}` and Signal strips a `signal:` prefix, so equivalent chats can bypass the lock and unrelated threaded chats can block each other.
  - confidence: 0.87
- **[P2] Bound stalled sends before blocking a recipient:** `src/infra/outbound/deliver.ts:1278`
  - body: Serializing the full adapter promise turns any hung platform call into a head-of-line block for every later send to that key. The Discord webhook path currently awaits a bare `fetch` without an abort signal or timeout, so one stalled webhook can wedge the recipient queue until restart.
  - confidence: 0.84
