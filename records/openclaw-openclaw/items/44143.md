## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Acquire serialization before queue persistence:** `src/infra/outbound/deliver.ts:1278`
  - body: Both `enqueueDelivery(...)` and the active delivery claim happen before this serializer call, so a later same-target invocation whose queue write finishes first can still enter the mutex first and deliver first. The ordering boundary needs to move ahead of the asynchronous queue work so FIFO reflects send initiation order.
  - confidence: 0.97
- **[P2] Key the serializer by canonical destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The key uses raw channel/account/recipient strings, but account IDs, recipient aliases, and thread/topic destinations are normalized later by routing and channel adapters. Equivalent destinations can bypass serialization, while distinct threads can block each other; derive a structured key from the canonical delivery target instead.
  - confidence: 0.95
- **[P2] Bound serialized platform sends:** `src/infra/outbound/deliver.ts:1278-1295`
  - body: The chain advances only when the full adapter promise settles. Since paths like Discord webhook sends have no local timeout or abort signal, one stalled request can wedge every later same-key delivery until restart; add a bounded or abort-aware send contract before serializing behind it.
  - confidence: 0.91
