Full review comments:

- **[P1] Key the lane from canonical delivery identity:** `src/infra/outbound/deliver.ts:1274`
  - body: The serializer key uses raw channel, accountId, and to values, but account ids and channel targets are canonicalized later and threaded channels include thread identity in the real destination. Equivalent conversations can bypass the lane while distinct threads can block each other, so derive the lane from canonical channel/account/target/thread identity.
  - confidence: 0.9
- **[P1] Release the lane when platform sends stall:** `src/infra/outbound/deliver.ts:1278`
  - body: The new lane is held across platform delivery, but paths such as Discord webhooks await fetch without a timeout or abort signal. One stalled request can wedge every later same-lane reply until restart, so lane-held sends need a bounded release path.
  - confidence: 0.86
- **[P2] Acquire the lane before queue work can reorder sends:** `src/infra/outbound/deliver.ts:1278`
  - body: The serializer is entered only after enqueueDelivery and active-claim work, so two same-target sends can still invert if the earlier request has slower queue I/O and reaches the lane second. Acquire the lane before reorderable queue work if FIFO is meant to follow send initiation order.
  - confidence: 0.82
