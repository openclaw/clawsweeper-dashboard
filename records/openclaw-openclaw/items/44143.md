## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Key the lane from the canonical destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The serializer key is built from raw `params.channel`, `params.accountId`, and `params.to`, and it omits `threadId`. Existing routing normalizes account IDs, route keys include thread identity, Signal strips `signal:` before send, and Discord threaded sends route to `channel:${threadId}`. Equivalent same-recipient sends can still bypass the lane, while distinct threads can now block each other.
  - confidence: 0.9
- **[P1] Bound sends that hold a recipient lane:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: `serialize()` advances only after the wrapped platform send settles, so one unsettled send holds every later send for the same key. Discord webhook delivery currently calls `fetch` without an abort signal or timeout, which means this PR can wedge later replies to that lane until restart. Bound or abort lane-held sends before queueing the next delivery behind them.
  - confidence: 0.86
