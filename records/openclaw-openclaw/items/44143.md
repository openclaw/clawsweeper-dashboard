## Summary

Keep open, but this PR is not merge-ready. The ordering problem is plausible, but the patch serializes too late in the outbound path, uses a non-canonical key, lacks abort/timeout-aware queue behavior, includes unrelated dependency churn, and still has only mock-based behavior proof.

## What This Changes

The branch adds an in-memory outbound DeliverySerializer, wires it around outbound delivery calls, adds focused serializer tests, and adds a Mattermost dependency/lockfile entry.

## Best Possible Solution

Serialize at the durable outbound intent boundary using canonical channel/account/target/thread identity, with abort/timeout-aware release behavior and no unrelated dependency churn.

## Reproduction Assessment

No high-confidence live reproduction is established. Source inspection and the author's mock signal-cli run support the race, but no redacted real affected-channel before/after proof is posted.

## Solution Assessment

No. The PR serializes after durable queue persistence with a raw route key, so a durable-intent serializer keyed by canonical route identity is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move serialization before queue persistence:** `src/infra/outbound/deliver.ts:526-528`
  - body: The serializer is entered only after `enqueueDelivery`. If the earlier same-target send is slower persisting its queue entry, a later send can acquire the mutex first and still deliver first, so FIFO by send initiation is not guaranteed.
  - confidence: 0.9
- **[P2] Build the key from canonical route identity:** `src/infra/outbound/deliver.ts:523`
  - body: The key uses raw channel/account/to and omits thread identity. Accounts and targets are normalized later, and threaded channels route by thread, so equivalent chats can bypass the lock while unrelated threads block each other.
  - confidence: 0.88
- **[P2] Keep queued waits abortable and bounded:** `src/infra/outbound/deliver.ts:526-528`
  - body: A send waiting behind the serializer is not abort-aware, and the queued operation has no bounded release. A stalled platform send can pin every later same-key delivery until restart.
  - confidence: 0.86
