## Summary

Keep open: the outbound ordering bug is real and current main does not already solve it, but this PR is not merge-ready because the new serializer is entered too late, uses raw destination identity, and can hold a recipient lane indefinitely behind a stalled platform send.

## What This Changes

The branch adds an in-memory DeliverySerializer and tests, then wraps outbound payload delivery in a per channel/account/recipient FIFO lane.

## Best Possible Solution

Move serialization to the earliest outbound lane boundary, key it from canonical channel/account/recipient/thread identity, and make lane-held sends timeout- and abort-aware before merge.

## Reproduction Assessment

Yes for the behavior class: current main queues and sends concurrent outbound deliveries independently, and Mantis Telegram Desktop evidence shows baseline inversion versus candidate ordering. I did not execute a fresh current-main test because this is a read-only review.

## Solution Assessment

No: the serializer is a plausible mitigation, but it is not the best fix until it runs before queue work, uses canonical routed identity, and bounds stalled sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Acquire the lane before queue persistence can reorder sends:** `src/infra/outbound/deliver.ts:1278`
  - body: The lock is entered only inside `deliverOutboundPayloadsWithQueueCleanup`, after `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` have already completed. If an earlier same-target send has slower queue I/O or claim work, a later send can reach this serializer first and still deliver first, so FIFO by send initiation is not guaranteed.
  - confidence: 0.88
- **[P1] Derive the lane key from canonical routed destination:** `src/infra/outbound/deliver.ts:1274`
  - body: The lane key uses raw `params.channel`, `params.accountId`, and `params.to`, and omits `threadId`. Current adapters normalize account IDs and targets later, including Discord threaded sends routing to `channel:${threadId}` and Signal stripping `signal:`, so equivalent destinations can bypass serialization while distinct threads can share the wrong lane.
  - confidence: 0.9
- **[P1] Bound platform sends while holding the lane:** `src/infra/outbound/delivery-serializer.ts:39`
  - body: The serializer advances only after `fn` settles. Discord webhook delivery currently performs a bare fetch without an abort or timeout signal on this path, so one stalled platform call can hold the recipient lane and block every later same-recipient reply until restart.
  - confidence: 0.82
