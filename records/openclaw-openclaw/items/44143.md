## Summary

Keep this PR open: it targets a real outbound message-ordering bug and has useful Telegram proof, but the current patch is not merge-ready because the serializer is acquired too late, uses raw destination keying, and can turn a stalled platform send into a same-recipient delivery stall.

## What This Changes

The PR adds an in-memory outbound delivery serializer and tests for same-key FIFO delivery, cross-key concurrency, failures, synchronous throws, and queue cleanup.

## Best Possible Solution

Serialize on a canonical delivery-lane identity before queue persistence and active-claim work, and ensure platform sends release or abort the lane instead of wedging later same-lane replies.

## Reproduction Assessment

Yes. Source inspection shows current outbound deliveries persist and claim queue entries before platform sends without a shared per-recipient lane, and the Mantis Telegram Desktop screenshots provide real before/after behavior evidence; I did not run a fresh local repro in this read-only review.

## Solution Assessment

No. The in-memory serializer is a plausible mitigation, but the best fix must lock the canonical destination before durable queue work and prevent stalled platform sends from holding the lane forever.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Acquire the lane before queue work can reorder sends:** `src/infra/outbound/deliver.ts:1278-1279`
  - body: The serializer is entered only after `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` have completed, so two same-target sends can still swap order while doing queue I/O and the later send can acquire the lock first. Acquire the lane before per-send durable queue work that can reorder callers, or make the queue supply the ordering key.
  - confidence: 0.88
- **[P1] Key delivery lanes from canonical targets:** `src/infra/outbound/deliver.ts:1274`
  - body: The key uses raw `params.channel`, `params.accountId`, and `params.to`, but current delivery code normalizes account IDs, strips provider target prefixes, and rewrites threaded sends later. Equivalent sends can bypass the mutex while distinct threads can block each other, so derive the lane from canonical delivery identity including thread/topic identity.
  - confidence: 0.87
- **[P1] Bound stalled sends before holding a delivery lane:** `src/infra/outbound/deliver.ts:1278-1279`
  - body: Once a call enters `deliverySerializer.serialize`, any platform send that never settles prevents all later same-lane sends from running. That is a new availability regression for paths like Discord webhook delivery, which awaits `fetch()` without a timeout; add a bounded timeout, abort, or release strategy before making this lock the shared outbound path.
  - confidence: 0.79
