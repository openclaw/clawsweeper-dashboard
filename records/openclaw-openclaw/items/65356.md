## Summary

Keep open. Current main still uses a module-local sandbox backend registry while plugin-loaded backends register through the SDK, and this PR is a narrow, plausible fix; it still needs external real behavior proof and ideally split-module regression coverage before merge.

## What This Changes

The PR changes `src/agents/sandbox/backend.ts` to store sandbox backend factories in a `Symbol.for` process-global `resolveGlobalSingleton` registry instead of a module-local `Map`.

## Best Possible Solution

Land this or an equivalent singleton-backed registry fix with split-module/jiti regression coverage and redacted real third-party sandbox backend verification.

## Reproduction Assessment

Yes, source-reproducible with high confidence. A jiti-loaded sandbox backend plugin can register through `openclaw/plugin-sdk/sandbox`, then core sandbox context resolves `agents.defaults.sandbox.backend` through `requireSandboxBackendFactory`; current main still uses a module-local map for that lookup.

## Solution Assessment

Yes. A `Symbol.for` process-global registry through the existing singleton helper is the narrow maintainable fix for duplicate module instances; the safer alternative is not a loader rewrite but adding targeted regression coverage and real setup proof.
