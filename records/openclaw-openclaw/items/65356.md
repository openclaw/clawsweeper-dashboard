## Summary

Keep open. Current main still uses a module-local sandbox backend registry while the plugin SDK exposes that registry to jiti-loaded plugins, so this PR is not obsolete and should not be closed as implemented. The code change appears narrow and correct; the remaining concrete cleanup is the required changelog entry before landing.

## What This Changes

The PR changes `src/agents/sandbox/backend.ts` so sandbox backend factories are stored in a `resolveGlobalSingleton(Symbol.for("openclaw.sandboxBackendFactories"), ...)` process-global registry instead of a module-local `Map`.

## Best Possible Solution

Land the singleton-backed sandbox backend registry change with a single-line changelog entry and, ideally, a targeted regression test proving a backend registered through a jiti-loaded plugin SDK copy is visible to runtime sandbox resolution.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to load a sandbox backend plugin through the jiti plugin loader so it registers via `openclaw/plugin-sdk/sandbox`, then resolve that backend from the native runtime copy; current main can miss because the registry is module-local and `requireSandboxBackendFactory` throws on a split registry.

## Solution Assessment

Yes, with a process caveat. A `Symbol.for` global singleton is the narrowest maintainable fix for this process-local registry and preserves the public SDK shape; the PR should add the required changelog entry before landing.
