## What This Changes

The PR changes `src/agents/sandbox/backend.ts` so the sandbox backend factory registry is stored in a `Symbol.for("openclaw.sandboxBackendFactories")` process-global singleton instead of a module-local `Map`.

## Best Possible Solution

Keep the sandbox backend registry shared across native ESM and jiti-loaded plugin SDK module graphs, preferably with a focused regression test proving a backend registered from a plugin-loaded copy is visible to runtime sandbox resolution.

## Reproduction Assessment

Yes. Current `main` uses a module-local map while the documented plugin runtime loads native plugins via jiti and exposes sandbox backend registration through `openclaw/plugin-sdk/sandbox`, so a plugin-loaded copy can register into a different module instance than runtime lookup uses.

## Solution Assessment

Yes, with a validation caveat. Moving this process-local registry onto a `Symbol.for` global singleton is the narrowest maintainable fix and matches the local helper contract for process-local registries; a targeted split-module regression test would make the solution easier to protect.
