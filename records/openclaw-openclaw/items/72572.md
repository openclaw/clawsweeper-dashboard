## Summary

Keep open: current main still rejects the reported symlinked `~/.openclaw` approvals directory, and two open PRs already use closing syntax for this issue but are not landed.

## What This Changes

Allow `openclaw exec-policy preset yolo` to succeed when the default `~/.openclaw` approvals directory is a trusted first-level symlink under the user's home.

## Best Possible Solution

Land one security-reviewed implementation that permits only a validated first-level `.openclaw` symlink under the trusted home boundary while keeping deeper symlinks and symlinked `exec-approvals.json` writes blocked.

## Reproduction Assessment

Yes, from source inspection. `openclaw exec-policy preset yolo` reaches `saveExecApprovals()`, and current `ensureDir()` still rejects a symlinked `.openclaw` directory before writing.

## Solution Assessment

Yes in direction, but it is not implemented on current main. A narrow first-level `.openclaw` symlink exception with ownership and permission validation is safer than broadly following symlinks or realpathing the whole approvals path.
