## Summary

Keep open: current `main` and the latest release still reject the reported symlinked `~/.openclaw` approvals directory, and two open closing PRs already target the issue but have not landed.

## What This Changes

The issue reports that `openclaw exec-policy preset yolo` rejects a trusted symlinked `~/.openclaw` directory under the normal home directory.

## Best Possible Solution

Review and land one security-reviewed implementation that permits only a validated first-level `.openclaw` symlink under the trusted home boundary while preserving the existing exec-approvals hardening.

## Reproduction Assessment

Yes, source-reproducible with high confidence. `openclaw exec-policy preset yolo` reaches `saveExecApprovals()`, and the current approvals-directory guard rejects a symlinked `.openclaw` component before writing.

## Solution Assessment

Yes in direction, but it is not implemented on current main. A narrow first-level `.openclaw` symlink exception with ownership and permission validation is safer than broadly following symlinks or adding a global bypass.
