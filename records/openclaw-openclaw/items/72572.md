## Summary

Keep open. Current `main` still rejects the reported symlinked `~/.openclaw` layout, the latest inspected release keeps the same guard, and open PRs #72650 and #72713 already reference this issue as their closing target, so this should wait for a reviewed implementation to land.

## What This Changes

Allow `openclaw exec-policy preset yolo` to accept a trusted first-level symlinked `~/.openclaw` while preserving deeper exec-approvals symlink rejection.

## Best Possible Solution

Land one reviewed implementation that permits exactly one trusted, current-user-owned, non-group/other-writable first-level `.openclaw` symlink and keeps rejecting chained or deeper symlinks plus symlinked `exec-approvals.json` destinations, with focused store coverage and docs if this becomes a supported layout.

## Reproduction Assessment

Yes. Create a home where `~/.openclaw` is a symlink and run `openclaw exec-policy preset yolo`; source inspection shows the command calls `saveExecApprovals()`, then `ensureDir()`, which rejects the symlinked `.openclaw` component before writing.

## Solution Assessment

Yes in direction, but it is not implemented on current `main`. The narrow first-level `.openclaw` exception with owner and mode validation is the maintainable shape; broader state-dir or macOS changes should land only after the same hardening review.
