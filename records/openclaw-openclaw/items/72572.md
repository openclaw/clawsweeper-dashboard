## Summary

Keep open: current main still rejects the reported symlinked `~/.openclaw` path, and open closing PRs #72650 and #72713 should be reviewed or consolidated before this issue is resolved.

## What This Changes

The issue asks OpenClaw to allow a trusted first-level `~/.openclaw` symlink for exec approvals while continuing to reject deeper symlink traversal.

## Best Possible Solution

Land one reviewed implementation that permits exactly one trusted, current-user-owned, non-group/other-writable first-level `.openclaw` symlink under the home boundary while continuing to reject deeper traversal and symlinked approval-file destinations.

## Reproduction Assessment

Yes. With `~/.openclaw` as a symlink under the resolved home directory, `openclaw exec-policy preset yolo` reaches `saveExecApprovals()`, and current `assertNoSymlinkPathComponents()` throws before the approvals file can be written.

## Solution Assessment

Yes in direction, but current main has not implemented it. A narrow first-level `.openclaw` exception with target validation is safer than broadly following symlinks or realpathing the whole approvals path.
