## Summary

Keep open. Current main still rejects the reported symlinked `~/.openclaw` layout, and two open PRs in the supplied GitHub context already reference this issue with closing syntax, so the issue should stay open until one reviewed implementation lands.

## What This Changes

Support a trusted first-level `~/.openclaw` symlink for exec approval policy storage while preserving deeper symlink traversal protections.

## Best Possible Solution

Land one reviewed implementation that permits exactly one trusted, user-owned, non-group/other-writable first-level `.openclaw` symlink and continues rejecting deeper or chained symlinks plus symlinked `exec-approvals.json` destinations, with focused store coverage and docs if the supported layout changes.

## Reproduction Assessment

Yes. The source path is high-confidence: create a home where `.openclaw` is a symlink, run `openclaw exec-policy preset yolo`, and the command reaches `saveExecApprovals()` -> `ensureDir()` -> `assertNoSymlinkPathComponents()`, which rejects that symlinked component.

## Solution Assessment

Yes in direction, but it is not implemented on main yet. The maintainable fix is a narrow, owner/mode-checked first-level `.openclaw` exception; broader macOS state-dir or socket-hardening changes should land only if review shows they are necessary for the same bug.
