## What This Changes

Allow `openclaw exec-policy preset yolo` to accept one trusted, user-owned first-level `~/.openclaw` symlink while preserving deeper exec-approvals symlink rejection.

## Best Possible Solution

Land one reviewed implementation that permits exactly one trusted, user-owned, non-group/other-writable first-level `.openclaw` symlink, keeps chained/deeper symlinks and symlinked `exec-approvals.json` destinations blocked, and adds focused store coverage plus docs if the supported layout changes.

## Reproduction Assessment

Yes. A source-level reproduction is clear: make the normal home’s `.openclaw` child a symlink, then run `openclaw exec-policy preset yolo`; the command calls `saveExecApprovals()`, whose directory guard rejects the symlinked component before writing.

## Solution Assessment

Yes in direction, but it is not implemented on current main. The narrow first-level `.openclaw` exception with ownership and mode validation is the maintainable shape; broader state-dir or macOS hardening should land only after review under the same exec-approval threat model.
