## Summary

Keep open: current main still rejects a symlinked `.openclaw` component below the resolved home directory, and open closing PRs #72650 and #72713 should be reviewed or consolidated before this issue is resolved.

## What This Changes

Restore support for a trusted first-level `~/.openclaw` symlink in the exec approvals store while preserving deeper symlink rejection.

## Best Possible Solution

Land one reviewed implementation that permits exactly one trusted, current-user-owned, non-group/other-writable first-level `.openclaw` symlink under the home boundary while continuing to reject deeper traversal and symlinked approval-file destinations.

## Reproduction Assessment

Yes. With `~/.openclaw` as a symlink under the resolved home directory, `openclaw exec-policy preset yolo` reaches `saveExecApprovals()`, and current `assertNoSymlinkPathComponents()` throws before the approvals file can be written.

## Solution Assessment

Yes in direction, but current main has not implemented it. The narrow first-level `.openclaw` exception with target validation is safer than broadly following symlinks or realpathing the whole approvals path.
