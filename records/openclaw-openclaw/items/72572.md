## Summary

Keep open: current main and the latest release still reject the reported symlinked `~/.openclaw` path, and open closing PRs #72650 and #72713 already target the remaining fix.

## What This Changes

Allow `openclaw exec-policy` to write the local approvals store when the default `~/.openclaw` directory is a trusted first-level symlink under the user's home directory.

## Best Possible Solution

Land or consolidate one reviewed implementation that permits exactly one trusted, current-user-owned, non-group/other-writable first-level `.openclaw` symlink while continuing to reject deeper traversal and symlinked approval-file destinations.

## Reproduction Assessment

Yes. A live symlink setup was not run because this review is read-only, but the current source path is deterministic: `exec-policy preset yolo` calls `saveExecApprovals()`, and `assertNoSymlinkPathComponents()` throws when `.openclaw` is the first symlink under the resolved home directory.

## Solution Assessment

Yes in direction, but not yet on main. A narrow first-level `.openclaw` exception with target ownership and mode validation is safer than broadly realpathing or following arbitrary approvals-path symlinks.
