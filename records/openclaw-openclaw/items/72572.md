## Summary

Keep open. Current main still rejects the reported symlinked `~/.openclaw` layout, the docs still identify `~/.openclaw/exec-approvals.json` as the local approvals store, and open PRs #72650 and #72713 already reference this issue with closing syntax, so this should remain open until one reviewed fix lands.

## What This Changes

The issue asks for `openclaw exec-policy preset yolo` to accept a trusted first-level symlinked `~/.openclaw` directory while preserving deeper symlink traversal and approvals-file destination hardening.

## Best Possible Solution

Land one reviewed implementation that permits exactly one first-level `.openclaw` symlink under the trusted home directory after owner and mode validation, continues rejecting chained/deeper symlinks and symlinked approvals-file destinations, and updates focused store tests plus docs if the supported layout changes.

## Reproduction Assessment

Yes. The reproduction path is deterministic from source: create a home where `.openclaw` is a symlink, then `exec-policy preset` calls `saveExecApprovals()`, which reaches `ensureDir()` and throws while walking that symlinked component.

## Solution Assessment

Yes in direction, but it is not implemented on current main. The narrow first-level `.openclaw` exception with ownership and writability validation is the maintainable shape; broader platform hardening should land only if reviewed under the same exec-approvals threat model.
