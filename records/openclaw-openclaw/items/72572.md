## Summary

Keep open. Current `main` still rejects the reported symlinked `~/.openclaw` layout, the latest inspected release has the same guard, and two open PRs already reference this issue as their closing target, so cleanup should wait for a reviewed implementation to land.

## What This Changes

The issue asks OpenClaw to allow a trusted first-level `~/.openclaw` symlink for exec approvals while continuing to reject deeper symlink traversal and unsafe approval-file destinations.

## Best Possible Solution

Land one reviewed implementation that permits exactly one trusted, current-user-owned, non-group/other-writable first-level `.openclaw` symlink and keeps rejecting chained or deeper symlinks plus symlinked `exec-approvals.json` destinations, with focused store tests and docs if this becomes a supported layout.

## Reproduction Assessment

Yes. Create a home where `~/.openclaw` is a symlink and run `openclaw exec-policy preset yolo`; the current source path calls `saveExecApprovals()`, then `ensureDir()`, which rejects the symlinked `.openclaw` component before writing.

## Solution Assessment

Yes in direction, but it is not implemented on current `main`. The narrow first-level `.openclaw` exception with owner and mode validation is the maintainable shape; broader state-dir or macOS changes should land only after the same hardening review.
