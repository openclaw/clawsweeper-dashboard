## Summary

Keep open: current `main` and the latest release still reject the reported symlinked `.openclaw` component, and open closing PRs #72650 and #72713 already target this issue for maintainer review or consolidation.

## What This Changes

Allow `openclaw exec-policy preset yolo` to work when `~/.openclaw` is a trusted first-level symlink while preserving deeper exec-approvals symlink hardening.

## Best Possible Solution

Land one security-reviewed implementation that permits exactly one current-user-owned, non-group/other-writable first-level `.openclaw` symlink under home while still rejecting deeper traversal and symlinked approval-file writes.

## Reproduction Assessment

Yes. With `~/.openclaw` as a symlink under the resolved home directory, `openclaw exec-policy preset yolo` calls `saveExecApprovals()`, and the current path walker throws on the `.openclaw` component before writing.

## Solution Assessment

Yes in direction, but current `main` has not implemented it. The narrow trusted first-level `.openclaw` exception with target validation is safer than broadly following symlinks or realpathing the whole approvals path.
