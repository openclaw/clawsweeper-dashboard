## Summary

Keep open: current main and latest release still reject the reported symlinked `.openclaw` component, and open PRs #72650 and #72713 already target this issue.

## What This Changes

Support `openclaw exec-policy preset yolo` when the default `~/.openclaw` directory is a trusted first-level symlink under the OS home directory while still rejecting unsafe deeper exec-approval symlinks.

## Best Possible Solution

Land one reviewed implementation that allows exactly one trusted, current-user-owned, non-group/other-writable first-level `.openclaw` symlink and keeps all deeper approval-path and file-destination protections intact.

## Reproduction Assessment

Yes. With `.openclaw` as an immediate symlink below the home root, `openclaw exec-policy preset yolo` reaches `saveExecApprovals()`, and the current `lstatSync()` path walk throws on that symlink component.

## Solution Assessment

Yes in direction, but not yet on main. The narrow trusted-hop exception is the maintainable fix; the immediate path is to review or consolidate #72650 and #72713 rather than close this issue early.
