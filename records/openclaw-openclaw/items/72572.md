Keep open: current `main` and `v2026.6.8` still reject a symlinked default `~/.openclaw` approvals directory, and two open linked PRs already use closing syntax for this issue but have not merged.

## What This Changes

The issue asks OpenClaw to allow a trusted first-level `~/.openclaw` symlink for exec approvals while continuing to reject unsafe deeper symlink traversal.

## Best Possible Solution

Land one security-reviewed fix that accepts only a validated first-level `~/.openclaw` symlink and preserves all deeper symlink, destination-file, ownership, permission, and hardlink protections, then close this issue after merge.

## Reproduction Assessment

Yes, from source inspection. `exec-policy preset` writes approvals through `saveExecApprovals()`, and current `main` still rejects a symlinked `.openclaw` child before or at the final approvals directory check; I did not run a filesystem repro because this review is read-only.

## Solution Assessment

Yes to keeping the issue open, but the implementation still needs maintainer security review. The narrowest maintainable fix is a single validated first-level symlink allowance, not a broad symlink bypass or parallel repair PR.
