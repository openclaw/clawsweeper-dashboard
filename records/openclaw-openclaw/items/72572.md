## Summary

Keep open. Current main and release v2026.4.29 still reject the reported symlinked `~/.openclaw` layout, and two open PRs already reference this issue as their closing target, so cleanup should wait for one reviewed implementation to land.

## What This Changes

The issue asks OpenClaw to allow a trusted first-level symlinked `~/.openclaw` approvals directory for `exec-policy` while continuing to reject unsafe deeper symlinks.

## Best Possible Solution

Land one reviewed implementation that permits exactly one trusted, current-user-owned, non-group/other-writable first-level `.openclaw` symlink and keeps rejecting chained symlinks, deeper symlinks, and symlinked `exec-approvals.json` destinations, with focused store coverage and docs if this layout becomes supported.

## Reproduction Assessment

Yes. Create a home where the immediate `.openclaw` child is a symlink and run `openclaw exec-policy preset yolo`; the current source path reaches `saveExecApprovals()` and rejects the symlinked component before writing.

## Solution Assessment

Yes in direction, but it is not implemented on current main. The narrow first-level `.openclaw` exception with ownership and mode validation is the maintainable fix; broader state-dir or macOS changes need the same hardening review if included.
