## Summary

Keep open. Current main still rejects the reported symlinked `~/.openclaw` layout, the latest release tag inspected has the same behavior, and open PRs #72650 and #72713 already reference this issue with closing syntax, so the issue should stay open until one reviewed implementation lands.

## What This Changes

The issue requests support for a trusted first-level `~/.openclaw` symlink in the exec approvals path while preserving deeper symlink and approval-file destination hardening.

## Best Possible Solution

Land one reviewed implementation that permits exactly one trusted, user-owned, non-group/other-writable first-level `.openclaw` symlink and continues rejecting chained or deeper symlinks plus symlinked `exec-approvals.json` destinations, with focused store tests and docs if the supported layout changes.

## Reproduction Assessment

Yes. Create a home where `~/.openclaw` is a symlink and run `openclaw exec-policy preset yolo`; the source path calls `saveExecApprovals()`, then `ensureDir()`, which rejects the symlinked `.openclaw` component before writing.

## Solution Assessment

Yes in direction, but it is not implemented on current main. The narrow first-level `.openclaw` exception with owner and mode validation is the maintainable shape; broader state-dir or macOS changes should land only if reviewed under the same hardening model.
