## Summary

Keep open. Current `main` and release `v2026.4.29` still implement the narrower behavior from #72377: a symlinked `OPENCLAW_HOME` root is accepted, but a symlinked `.openclaw` component below the home root is still rejected. Open PRs #72650 and #72713 already reference this issue as their closing target, so cleanup should wait for maintainer review or consolidation of those implementations.

## What This Changes

The issue requests support for `openclaw exec-policy preset yolo` when the normal home directory contains a trusted first-level `~/.openclaw` symlink, while preserving deeper exec-approvals symlink hardening.

## Best Possible Solution

Land one reviewed implementation that permits exactly one trusted, current-user-owned, non-group/other-writable first-level `.openclaw` symlink under the home boundary while continuing to reject deeper traversal and symlinked approval-file destinations, with focused store coverage and docs if this layout becomes supported.

## Reproduction Assessment

Yes. With a home directory whose immediate `.openclaw` child is a symlink, `openclaw exec-policy preset yolo` reaches `saveExecApprovals()`, and current `assertNoSymlinkPathComponents()` throws before writing.

## Solution Assessment

Yes in direction, but it is not solved on current `main`. The narrow first-level `.openclaw` exception with ownership and mode validation is the maintainable path; the immediate next step is to review or consolidate the existing closing PRs.
