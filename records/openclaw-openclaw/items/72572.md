## Summary

Keep open: current main and the latest release still reject the reported symlinked `~/.openclaw` approvals directory, and two open closing PRs target this issue but have not landed.

## What This Changes

Restore support for `openclaw exec-policy preset yolo` when the default `~/.openclaw` directory is a trusted first-level symlink while preserving deeper exec-approvals symlink rejection.

## Best Possible Solution

Land one security-reviewed implementation that permits only a validated first-level `.openclaw` symlink under the trusted home boundary while preserving existing exec-approvals hardening.

## Reproduction Assessment

Yes. Source inspection shows `openclaw exec-policy preset yolo` reaches `saveExecApprovals()`, and the current approvals-directory guard rejects a symlinked `.openclaw` component before writing.

## Solution Assessment

Yes in direction, but it is not implemented on current main. A narrow first-level `.openclaw` symlink exception with ownership and permission validation is safer than broadly following symlinks or adding a global bypass.
