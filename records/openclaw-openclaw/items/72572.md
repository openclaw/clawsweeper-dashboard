## Summary

Keep open: current main still rejects the reported symlinked `~/.openclaw` approvals directory, and live closing PRs remain open implementation candidates.

## What This Changes

The issue asks OpenClaw to allow a validated first-level `~/.openclaw` symlink for exec approval policy writes while preserving deeper symlink hardening.

## Best Possible Solution

Land one security-reviewed implementation that permits only a validated first-level `.openclaw` symlink under the trusted home boundary while keeping deeper symlinks and symlinked `exec-approvals.json` writes blocked.

## Reproduction Assessment

Yes, from source inspection. With `.openclaw` as a symlink under the resolved home directory, `openclaw exec-policy preset yolo` reaches `saveExecApprovals()`, and the current guard rejects the symlinked approvals directory before writing.

## Solution Assessment

Yes in direction, but it is not implemented on current main. A narrow first-level `.openclaw` symlink exception with ownership and permission validation is safer than broadly following symlinks or realpathing the full approvals path.
