## Summary

Keep open: current main still rejects the reported symlinked `~/.openclaw` approvals directory, the latest release has the same guard, and open closing PRs #72650 and #72713 remain the implementation candidates to review or consolidate.

## What This Changes

The issue requests allowing `openclaw exec-policy preset yolo` to accept a user-owned first-level `~/.openclaw` symlink while still rejecting deeper unsafe symlinks.

## Best Possible Solution

Land one security-reviewed implementation that permits only a validated first-level `.openclaw` symlink under the trusted home boundary while preserving deeper traversal and destination-file guards.

## Reproduction Assessment

Yes, from source inspection on current main. `exec-policy preset yolo` calls `saveExecApprovals()`, `ensureDir()` walks the approvals directory from the resolved home root, and an `lstatSync` symlink at `.openclaw` throws before writing.

## Solution Assessment

Yes in direction, but it is not implemented on current main. A narrow first-level `.openclaw` exception with target validation is safer than broadly following symlinks or realpathing the full approvals path.
