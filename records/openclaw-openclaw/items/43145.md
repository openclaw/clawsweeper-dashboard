## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Authorize timeline reads before returning spans:** `src/gateway/server-methods/agent.ts:710-724`
  - body: `agent.timeline` accepts any `runId` and returns `sessionKey` plus the full span list without checking that the caller owns that run. A paired read client that learns another run id can read tool names, model/provider metadata, and free-form error/note fields, so bind traces to the caller/session or require an elevated scope with redaction.
  - confidence: 0.9
- **[P1] Classify run errors before returning critic replans:** `src/cron/isolated-agent/run.ts:797-805`
  - body: The critic `needs_replan` branch returns `status: "ok"` before the function computes `runLevelError` and fatal error payloads. With the critic loop enabled, a run that only produced error payloads can skip normal cron failure handling; classify errors first or gate the critic return on no fatal error.
  - confidence: 0.88
- **[P1] Reject dynamic Python imports during sandbox validation:** `src/agents/sandbox/execution-policy.ts:170-171`
  - body: `validateSandboxPythonExecution` only checks modules returned by the static import parser. Code such as `__import__("subprocess")` produces no parsed import and bypasses the deny lists, so detect or prohibit dynamic import primitives before accepting generated code.
  - confidence: 0.86
- **[P2] Cap running trace timelines and span growth:** `src/agents/agent-run-trace.ts:104-106`
  - body: New running timelines are stored in a process-global map and spans are appended without a hard running-count or per-timeline cap. A caller can hold many incomplete runs and many spans until age pruning, so add bounded running/per-session eviction and a per-run span limit.
  - confidence: 0.82
- **[P2] Redact cron errors before persisting playbook memory:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: The playbook memory layer persists `signal.error` verbatim as `lastError`. These strings can include URLs, tokens, user content, or stack details and are unbounded, so redact and truncate before saving the JSON state file.
  - confidence: 0.82
- **[P2] Strip inline comments from parsed import segments:** `src/agents/sandbox/execution-policy.ts:54-59`
  - body: `parseImportSegment` strips aliases but not inline comments, so valid Python like `import os  # path ops` is parsed as a module named `os  # path ops` and misclassified. Strip comments before allow/deny classification.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces concrete gateway trace-disclosure, sandbox policy-bypass, memory-retention, and persisted-error risks.

Concerns:

- **[high] Timeline endpoint can disclose other run traces:** `src/gateway/server-methods/agent.ts:710`
  - body: `agent.timeline` returns trace data solely by supplied `runId` and includes session/span details without caller ownership checks or redaction.
  - confidence: 0.9
- **[high] Sandbox execution policy misses dynamic imports:** `src/agents/sandbox/execution-policy.ts:170`
  - body: The new validator checks only static import statements, so dynamic import primitives can bypass blocked module policy.
  - confidence: 0.86
- **[medium] Running trace timelines can grow without hard bounds:** `src/agents/agent-run-trace.ts:104`
  - body: Running timelines and per-run spans are not capped, allowing repeated incomplete runs to retain many objects until age pruning.
  - confidence: 0.82
- **[medium] Cron playbook memory persists raw error text:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: Cron error strings are stored to disk without redaction or truncation, which can retain sensitive data or large payloads in state.
  - confidence: 0.82
