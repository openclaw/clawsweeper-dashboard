## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Defer critic gating until after error classification:** `src/cron/isolated-agent/run.ts:797-805`
  - body: This early `needs_replan` return runs before `hasFatalErrorPayload` and `runLevelError` are computed, so an embedded run that only returned error payloads can be reported as `status: "ok"`. Move critic evaluation after the fatal-error classification, or gate this branch on there being no fatal run error.
  - confidence: 0.92
- **[P1] Authorize timeline reads before returning spans:** `src/gateway/server-methods/agent.ts:715-724`
  - body: `agent.timeline` accepts any `runId` and returns `sessionKey` plus raw spans without checking that the caller owns that run. A read-scoped client that learns a run id can retrieve another run's trace metadata, so the handler needs object-level authorization and a minimized/redacted response.
  - confidence: 0.88
- **[P1] Reject dynamic Python imports in sandbox validation:** `src/agents/sandbox/execution-policy.ts:170-171`
  - body: The sandbox policy only checks modules found by static `import`/`from` parsing. Code such as `__import__("subprocess")` produces no import usage and bypasses the deny list, so validation should reject dynamic import primitives or parse them before rendering executable code.
  - confidence: 0.9
- **[P2] Redact cron errors before persisting playbook memory:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: `signal.error` is stored verbatim in the playbook memory state. Cron/tool/provider errors can contain URLs, identifiers, or secret-bearing messages, so persist a redacted and length-bounded value instead of the raw string.
  - confidence: 0.84
- **[P2] Cap running trace storage:** `src/agents/agent-run-trace.ts:104-106`
  - body: Each trace can keep appending spans, and new running timelines are retained by time only with no running-count or per-timeline span cap. A client that starts many unique or long-running runs can grow memory until the retention window clears, so add hard caps or eviction for running traces and spans.
  - confidence: 0.78
- **[P2] Strip inline comments from parsed import segments:** `src/agents/sandbox/execution-policy.ts:54-59`
  - body: `parseImportSegment` strips aliases but not trailing Python comments, so `import math  # used below` is parsed as a module name containing the comment and can be rejected incorrectly. Strip comments before alias/root normalization.
  - confidence: 0.81

## Security Review

Status: needs_attention

Summary: The PR introduces concrete security-sensitive regressions in trace access control, sandbox policy enforcement, persisted error text, and trace memory bounds.

Concerns:

- **[high] Timeline endpoint lacks object authorization:** `src/gateway/server-methods/agent.ts:715`
  - body: `agent.timeline` returns `sessionKey` and raw trace spans for a caller-supplied `runId` without verifying the requesting client/session owns that run, while the method is added to read scope.
  - confidence: 0.88
- **[high] Sandbox deny lists miss dynamic imports:** `src/agents/sandbox/execution-policy.ts:170`
  - body: The new sandbox execution validator only inspects static import statements, so dynamic import primitives can access blocked modules while passing validation.
  - confidence: 0.9
- **[medium] Cron playbook memory stores raw errors:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: The playbook memory layer persists `signal.error` without redaction or size bounding, which can retain secret-bearing provider/tool/delivery errors on disk.
  - confidence: 0.84
- **[medium] Trace store has weak memory bounds:** `src/agents/agent-run-trace.ts:104`
  - body: Running timelines and per-timeline spans are not hard-capped, so repeated unique or long-running runs can grow process memory until time-based pruning occurs.
  - confidence: 0.78
