## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the central retained run-trace timeline surface, while the PR remains a useful implementation candidate with unresolved correctness, security, and scope issues already echoed by prior bot reviews and the related open #68204 design thread.

## What This Changes

The PR adds an in-memory agent run-trace timeline store and `agent.timeline` Gateway RPC with protocol/client/docs/tests, while also changing block reply and chunker behavior and adding unrelated cron critic-loop, plan-search, sandbox execution, simulation, procedural-memory, roadmap, and test updates.

## Best Possible Solution

Use #68204 to settle a canonical trace schema, then refresh this work as a narrow trace-only PR that adds the store, RPC, protocol schema, generated clients, docs, authorization, redaction, retention limits, and focused tests without the unrelated autonomy/sandbox/cron changes.

## Reproduction Assessment

Yes for the implementation gap: current main can be checked directly with `rg`/`nl`, and it has no `agent.timeline` method, schema, generated model, or docs. The PR review findings also have source-level reproduction paths from the PR head diff, especially the unauthorised runId lookup and the critic early return before fatal-error classification.

## Solution Assessment

No, this PR branch is not the best way to solve the issue as currently shaped. The maintainable path is a split, rebased trace-only implementation aligned with #68204 and hardened before exposing trace data to operator clients.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Authorize timeline reads before returning spans:** `src/gateway/server-methods/agent.ts:710-724`
  - body: `agent.timeline` looks up a caller-supplied `runId` and returns `sessionKey` plus spans without checking that the requester owns or started that run. A read-scoped operator client that learns another run id could retrieve another session's trace details, so the handler needs object-level authorization before returning the timeline.
  - confidence: 0.93
- **[P1] Cap running trace state before accepting new runs:** `src/agents/agent-run-trace.ts:104-106`
  - body: Every new run id is inserted into the process-global timeline map, but pruning only caps terminal timelines and evicts running timelines by age. Since agent run ids come from caller-provided idempotency keys, many incomplete runs can accumulate for the retention window; add global/per-session running caps and a span cap.
  - confidence: 0.87
- **[P1] Classify cron failures before critic returns:** `src/cron/isolated-agent/run.ts:797-805`
  - body: The `needs_replan` critic branch returns `status: "ok"` before the later fatal-error payload and run-level error classification runs. With critic mode enabled, a failed embedded run can be reported as a successful replan request and skip normal failure handling.
  - confidence: 0.9
- **[P1] Reject dynamic Python import escapes:** `src/agents/sandbox/execution-policy.ts:170-171`
  - body: The sandbox execution validator only checks modules returned by the static import parser. Code such as `__import__("subprocess")` produces no usage entry, so denied modules can bypass the new policy; detect or reject dynamic import primitives before rendering the template.
  - confidence: 0.88
- **[P2] Strip inline comments from import segments:** `src/agents/sandbox/execution-policy.ts:54-59`
  - body: `parseImportSegment` strips aliases but leaves trailing comments in the module name. Valid Python like `import math  # used below` is parsed with the comment included, which can misclassify or reject otherwise allowed imports.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR introduces concrete security-sensitive concerns in the new timeline RPC and sandbox execution policy.

Concerns:

- **[high] Timeline RPC lacks object-level authorization:** `src/gateway/server-methods/agent.ts:710`
  - body: The new handler fetches by caller-provided `runId` and returns `sessionKey` plus raw spans without verifying the requester owns the run, creating an information-disclosure path for trace metadata and error strings.
  - confidence: 0.93
- **[medium] Running timeline cache lacks hard caps:** `src/agents/agent-run-trace.ts:104`
  - body: Running timelines are stored globally and retained by age without a count or span cap, so repeated distinct idempotency keys can grow in-memory trace state during the retention window.
  - confidence: 0.87
- **[high] Sandbox policy misses dynamic imports:** `src/agents/sandbox/execution-policy.ts:170`
  - body: The new Python execution validator enforces allow/deny lists only for static `import` and `from` statements, leaving dynamic import paths able to load blocked modules unless separately detected or prohibited.
  - confidence: 0.88
