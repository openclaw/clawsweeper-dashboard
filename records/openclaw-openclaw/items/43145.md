## Summary

Keep open. Current main still lacks the central `agent.timeline`/agent run-trace feature, and this PR is not safe to merge as-is because the latest branch still has blocking security and correctness issues plus no real behavior proof.

## What This Changes

The PR adds an agent run timeline RPC and in-memory trace store, while also changing block reply delivery, fenced block chunking, planner search, sandbox execution policy/templates, cron critic/playbook features, protocol models, docs, and tests.

## Best Possible Solution

Define a scoped, redacted, capped, versioned trace contract tied to the broader trace-schema tracker, then split unrelated cron, sandbox, planner, docs, and Telegram-adjacent chunking changes into focused PRs with real behavior proof.

## Reproduction Assessment

No, not for the feature request as a runtime bug; it is a new capability. Source inspection does give a high-confidence review path for the PR blockers and confirms current main does not already implement the requested `agent.timeline` surface.

## Solution Assessment

No. The current PR is too broad and introduces security/correctness regressions; the safer path is a narrower trace design with authorization, redaction, bounded retention, protocol docs, and focused tests before any merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Authorize timeline reads before returning spans:** `src/gateway/server-methods/agent.ts:715-724`
  - body: This handler returns a trace by caller-supplied `runId` without checking that the requesting client/session owns that run. Since the response includes `sessionKey` and raw span details, any paired read client that learns a run id can read another run's trace data; bind traces to the caller or require a narrower elevated scope before returning them.
  - confidence: 0.92
- **[P1] Cap running timelines and spans:** `src/agents/agent-run-trace.ts:104-106`
  - body: Every distinct run id creates a running timeline and each trace span is appended indefinitely, but only stale running traces after 30 minutes and terminal timelines are pruned. A caller can create many incomplete runs or a single span-heavy run and grow gateway memory for the retention window; add hard global/per-session running caps and a per-timeline span cap.
  - confidence: 0.86
- **[P1] Classify run errors before returning needs_replan:** `src/cron/isolated-agent/run.ts:797-805`
  - body: The critic-loop `needs_replan` branch returns `status: "ok"` before the function computes fatal error payloads and `finalRunResult.meta.error`. With critic mode enabled, an actually failed agent run can be reported as a successful replan request, suppressing normal failure handling and recovery signals.
  - confidence: 0.9
- **[P1] Reject dynamic Python imports:** `src/agents/sandbox/execution-policy.ts:170-171`
  - body: The sandbox policy only validates modules found by the static import parser, so code such as `__import__("subprocess")` produces no usage entry and bypasses the configured import/dependency deny lists. Detect and reject dynamic import primitives, or otherwise execute under a policy that cannot be bypassed by runtime import calls.
  - confidence: 0.88
- **[P2] Redact persisted cron errors:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: The playbook memory persists `signal.error` verbatim. Those strings can contain provider URLs, tokens, user content, stack traces, or other private details, and they are not length-bounded; redact and truncate before saving or store only a classification/hash by default.
  - confidence: 0.86
- **[P2] Strip inline comments from import segments:** `src/agents/sandbox/execution-policy.ts:54-60`
  - body: `parseImportSegment` strips aliases but not trailing comments, so valid Python like `import os  # path ops` is parsed as a root named `os  # path ops` and then rejected as an unallowlisted dependency. Strip inline comments before classifying the module name.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces new trace-read, sandbox-execution, and cron-memory surfaces with concrete confidentiality, resource-exhaustion, and policy-bypass risks.

Concerns:

- **[high] Unscoped trace disclosure:** `src/gateway/server-methods/agent.ts:715`
  - body: `agent.timeline` returns trace spans and `sessionKey` for any supplied `runId` without object-level authorization, while the method is added to normal read scope.
  - confidence: 0.92
- **[high] Unbounded trace memory growth:** `src/agents/agent-run-trace.ts:104`
  - body: Running timelines and per-timeline spans lack hard caps; a caller controlling run ids can accumulate many active traces until retention cleanup.
  - confidence: 0.86
- **[high] Sandbox policy bypass via dynamic imports:** `src/agents/sandbox/execution-policy.ts:170`
  - body: The Python execution policy validates static import syntax only, so runtime import primitives can load denied modules without being reported by the parser.
  - confidence: 0.88
- **[medium] Sensitive cron errors persisted:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: Cron playbook memory stores raw error strings to disk without redaction or truncation, which can retain secrets, private endpoints, or user content.
  - confidence: 0.86
