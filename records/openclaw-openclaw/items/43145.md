## Summary

Keep open, but do not merge as-is: the trace capability is not on current main, the branch mixes several unrelated runtime/config surfaces, and the PR still has concrete P1 correctness/security blockers plus missing real behavior proof.

## What This Changes

The PR adds an in-memory agent run-trace timeline and `agent.timeline` gateway RPC, while also changing block reply chunking, cron critic/playbook memory, plan search, sandbox execution/simulation, protocol models, docs, and Telegram test timing.

## Best Possible Solution

Land focused owner-reviewed changes: first a versioned authorized trace contract aligned with https://github.com/openclaw/openclaw/issues/68204, then split any cron, sandbox, planner, or chunking work into separate PRs with targeted proof.

## Solution Assessment

No. The trace direction is useful, but this PR is not the narrowest maintainable solution because it exposes security-sensitive trace data and mixes unrelated cron, sandbox, planner, docs, generated protocol, and test timing changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Authorize timeline reads before returning spans:** `src/gateway/server-methods/agent.ts:710-724`
  - body: `agent.timeline` looks up any caller-supplied `runId` and returns `sessionKey` plus the full span list without checking that the requesting client owns that run. A paired read client that learns another run id can read tool names, provider/model metadata, and free-form error/note fields, so bind traces to the caller/session or require an elevated scope and redact the response.
  - confidence: 0.9
- **[P1] Classify run errors before returning critic replan:** `src/cron/isolated-agent/run.ts:797-805`
  - body: The new critic `needs_replan` branch returns `status: "ok"` before this function computes run-level fatal error payloads. With critic loop enabled, an embedded-agent run that only produced error payloads can be reported as ok and skip normal cron failure handling, alerts, and playbook signals.
  - confidence: 0.88
- **[P1] Reject dynamic Python imports in sandbox validation:** `src/agents/sandbox/execution-policy.ts:170-171`
  - body: `validateSandboxPythonExecution` only checks modules returned by the static import parser. Code such as `__import__("subprocess")` produces no parsed import and can bypass the deny lists, so the validator needs to detect or prohibit dynamic import primitives before accepting generated code.
  - confidence: 0.86
- **[P2] Cap running trace timelines and span growth:** `src/agents/agent-run-trace.ts:104-107`
  - body: The trace store caps terminal timelines, but new running timelines are added to a process-global map without a running-count cap and without per-timeline span limits. A caller that creates many incomplete runs can hold many timelines and spans until age-based cleanup, so add running/per-session and per-run bounds.
  - confidence: 0.82
- **[P2] Redact cron errors before persisting playbook memory:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: The playbook memory layer stores `signal.error` verbatim as `lastError`. Those strings can contain URLs, tokens, user content, or stack details and are unbounded in length, so redact and truncate before saving the JSON state file.
  - confidence: 0.82
- **[P2] Strip inline comments from import segments:** `src/agents/sandbox/execution-policy.ts:54-59`
  - body: `parseImportSegment` only removes aliases, so valid Python like `import os  # path ops` is parsed as a module name containing the comment and then rejected as an unallowlisted dependency instead of matching the import policy. Strip inline comments before classifying the segment.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces concrete gateway trace disclosure, sandbox policy bypass, memory-retention, and persisted-error risks.

Concerns:

- **[high] Timeline endpoint can disclose other run traces:** `src/gateway/server-methods/agent.ts:710`
  - body: `agent.timeline` returns a trace solely by supplied `runId` and includes session/span details without caller ownership checks or redaction.
  - confidence: 0.9
- **[high] Sandbox execution policy misses dynamic imports:** `src/agents/sandbox/execution-policy.ts:170`
  - body: The new validator checks static imports only, so dynamic Python import primitives can bypass allow/deny policy for blocked modules.
  - confidence: 0.86
- **[medium] Running traces can grow without a hard cap:** `src/agents/agent-run-trace.ts:104`
  - body: Running timelines and their spans are stored in process-global memory without a hard running-count or per-run span cap.
  - confidence: 0.82
- **[medium] Cron playbook memory persists raw errors:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: The new playbook memory state stores raw cron error strings on disk without redaction or length bounding.
  - confidence: 0.82
