## Summary

Keep open. Current main does not implement the proposed `agent.timeline` surface, and the branch still contains unique but broad feature work; however, it is not mergeable as-is because the diff introduces security/correctness blockers and lacks after-fix real behavior proof.

## What This Changes

The PR adds an in-memory agent run-trace timeline and `agent.timeline` gateway API, plus cron critic-loop config, plan search, sandbox execution/simulation helpers, protocol/docs/tests, and block-reply/chunker fixes.

## Best Possible Solution

Split this into focused maintainer-reviewed changes: first settle the versioned run-trace contract tracked by https://github.com/openclaw/openclaw/issues/68204, then land independently hardened timeline, block-reply/chunker, cron, and sandbox changes with targeted tests and real behavior proof.

## Reproduction Assessment

Not applicable as a feature PR, but the blocking findings are source-reproducible from the PR head. The `agent.timeline` handler, cron critic return path, sandbox import validator, and playbook persistence code each show the problematic behavior directly in source.

## Solution Assessment

No, this is not the best way to solve the issue as submitted. The run-trace idea may be useful, but the branch is too broad and needs hardened authorization/redaction, safer sandbox policy, corrected cron error ordering, and a split aligned with the canonical trace-schema tracker.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Authorize timeline reads before returning spans:** `src/gateway/server-methods/agent.ts:710-724`
  - body: `agent.timeline` accepts a caller-supplied `runId` and returns `sessionKey` plus raw trace spans without checking that the caller owns that run. Any paired read-scope client that learns or guesses a run id can read another run's provider/model/tool/error metadata, so this needs object-level authorization and redaction before merge.
  - confidence: 0.89
- **[P1] Classify run errors before critic early return:** `src/cron/isolated-agent/run.ts:797-805`
  - body: The `needs_replan` branch returns `status: "ok"` before the later fatal-error payload classification runs. With critic loop enabled, a run that only produced an embedded error payload can bypass normal cron failure handling and recovery signals, so classify run-level errors before applying the critic outcome.
  - confidence: 0.87
- **[P1] Reject dynamic imports in sandbox validation:** `src/agents/sandbox/execution-policy.ts:170-171`
  - body: The sandbox policy only validates modules returned by the static import parser. Generated Python can call dynamic import primitives such as `__import__("subprocess")` without producing an entry here, bypassing the configured import deny list before execution.
  - confidence: 0.86
- **[P2] Cap running timelines and per-run spans:** `src/agents/agent-run-trace.ts:104-106`
  - body: New running timelines are added to the module-level map and each span is appended without a hard count cap. Time-based pruning eventually removes stale running timelines, but many concurrent or long-lived runs can still accumulate unbounded timeline and span data during the retention window.
  - confidence: 0.8
- **[P2] Redact cron errors before persisting playbook memory:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: `lastError` stores the raw cron error string in the playbook memory state. Runtime error strings can contain tokens, URLs, paths, user content, or very large stack traces, so this should apply existing redaction and length bounds before writing to disk.
  - confidence: 0.82
- **[P2] Strip inline comments from import segments:** `src/agents/sandbox/execution-policy.ts:55-59`
  - body: `parseImportSegment` only strips aliases, so a valid line like `import math  # comment` treats the comment as part of the module name. That turns allowlisted imports into unknown dependencies and rejects legitimate generated code.
  - confidence: 0.77

## Security Review

Status: needs_attention

Summary: The diff introduces concrete authorization, data-retention, sandbox-policy, and sensitive-error persistence concerns.

Concerns:

- **[high] Timeline API lacks object-level authorization:** `src/gateway/server-methods/agent.ts:710`
  - body: `agent.timeline` returns `sessionKey` and raw spans for any valid `runId` without checking that the requesting client owns that run, creating an information-disclosure path for paired read-scope clients.
  - confidence: 0.89
- **[medium] Running trace storage lacks hard caps:** `src/agents/agent-run-trace.ts:104`
  - body: The timeline map and per-run span arrays are bounded only by time for running traces, so many concurrent/incomplete runs can accumulate memory during the running retention window.
  - confidence: 0.8
- **[high] Sandbox import policy can be bypassed:** `src/agents/sandbox/execution-policy.ts:170`
  - body: The Python validator checks only static imports and does not reject dynamic import primitives, so denied modules can still be imported at runtime by generated code.
  - confidence: 0.86
- **[medium] Cron playbook memory persists raw errors:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: Raw cron error strings are persisted as `lastError` without redaction or size limits, which can retain secrets, private URLs, paths, user content, or large stack traces on disk.
  - confidence: 0.82
- **[low] Simulation log hash chain is unauthenticated:** `src/agents/sandbox/simulation-lane.ts:380`
  - body: The simulation lane uses plain SHA-256 for a hash chain; if this is presented as immutable audit evidence, a writer with file access can modify entries and recompute hashes unless the wording or implementation limits that trust model.
  - confidence: 0.68
