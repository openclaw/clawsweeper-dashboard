## What This Changes

This PR adds an in-memory agent run trace timeline endpoint and related protocol/client/docs/tests while also adding plan search, sandbox execution/simulation helpers, cron critic/playbook memory, block-reply delivery, and chunking changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Classify run errors before returning critic replans:** `src/cron/isolated-agent/run.ts:797-801`
  - body: The `needs_replan` return runs before later fatal-error classification, so a run that only produced error payloads can be reported as `status: "ok"` whenever the critic loop is enabled. Move error classification before this branch or gate the branch on there being no fatal run error.
  - confidence: 0.88
- **[P1] Reject dynamic imports in sandbox validation:** `src/agents/sandbox/execution-policy.ts:170-171`
  - body: The sandbox validator only checks modules found by static `import`/`from` parsing. Generated code can still call `__import__("subprocess")` and bypass the denylist, so the policy needs to detect or reject dynamic import primitives too.
  - confidence: 0.9
- **[P1] Authorize trace reads before returning spans:** `src/gateway/server-methods/agent.ts:709-724`
  - body: `agent.timeline` accepts an arbitrary `runId` and returns the stored timeline, including `sessionKey` and free-form span details, without checking that the caller owns that run. Bind traces to the requesting principal or require an elevated/admin trace scope and return a redacted view by default.
  - confidence: 0.86
- **[P2] Redact persisted cron playbook errors:** `src/cron/procedural-playbook-memory-v0.ts:234-243`
  - body: `entry.lastError = signal.error?.trim()` persists arbitrary runtime error strings to disk without redaction or length bounds. Cron/tool/provider errors can include sensitive URLs, tokens, user text, or very large strings, so normalize through the existing redaction/truncation path before saving.
  - confidence: 0.82
- **[P2] Cap running trace storage:** `src/agents/agent-run-trace.ts:72-94`
  - body: The trace store caps only terminal timelines; running timelines and per-timeline spans can accumulate until retention pruning. Add a hard running-trace and span cap so repeated incomplete runs cannot grow gateway memory unbounded for the retention window.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive gateway trace access, sandbox execution validation, and cron error persistence paths that need fixes before merge.

Concerns:

- **[high] Trace endpoint lacks object-level authorization:** `src/gateway/server-methods/agent.ts:709`
  - body: `agent.timeline` reads a timeline by caller-supplied `runId` and returns session and span details without proving the caller owns that run, creating an information-disclosure path for paired operator clients.
  - confidence: 0.86
- **[high] Sandbox policy misses dynamic imports:** `src/agents/sandbox/execution-policy.ts:170`
  - body: The Python sandbox policy validates only static imports, so dynamic import primitives can bypass allow/deny lists for blocked modules such as `subprocess`.
  - confidence: 0.9
- **[medium] Cron playbook memory persists raw errors:** `src/cron/procedural-playbook-memory-v0.ts:239`
  - body: The new playbook memory stores raw error text on disk without redaction or bounds, which can retain secrets, PII, non-public endpoints, or oversized strings from upstream failures.
  - confidence: 0.82
- **[medium] Running trace cache lacks a hard cap:** `src/agents/agent-run-trace.ts:72`
  - body: Running timelines are retained separately from terminal cap pruning, so repeated incomplete runs can grow process memory until retention cleanup unless a running-trace/span cap is added.
  - confidence: 0.78
