## Summary

Keep open. Current main does not already contain the `agent.timeline` run-trace endpoint, and the proposed branch still has blocking correctness/security findings plus missing real behavior proof.

## What This Changes

The PR adds an in-memory agent run-trace timeline store exposed through `agent.timeline`, updates protocol models/docs/tests, and also bundles broader cron critic-loop, sandbox execution, plan-search, simulation-lane, and tool-adapter changes.

## Best Possible Solution

Split this into focused PRs, with the trace-timeline work first: define an authorized/redacted trace read contract, cap retained running trace state, align it with the unified trace-schema direction, and leave unrelated cron/sandbox/planner work for separate review.

## Reproduction Assessment

No high-confidence runtime reproduction was run because this is a read-only PR review. The blocking defects are source-reproducible from the PR head: the endpoint reads by runId without authorization, the critic return precedes error classification, and the sandbox validator only checks static imports.

## Solution Assessment

No. The proposed branch is not the best merge shape because it combines several product surfaces and leaves authorization, redaction, retention, and sandbox-policy holes unresolved; a narrower trace-contract PR would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Authorize timeline reads before returning spans:** `src/gateway/server-methods/agent.ts:709-724`
  - body: `agent.timeline` looks up a caller-supplied `runId` and returns the stored timeline, including `sessionKey` and raw span error/note data, without checking that the requesting client is allowed to inspect that run. Bind the trace to the requester/session and redact sensitive fields before exposing it over an operator-readable RPC.
  - confidence: 0.89
- **[P1] Classify run errors before the critic return:** `src/cron/isolated-agent/run.ts:797-805`
  - body: The critic `needs_replan` branch returns `status: "ok"` before `hasFatalErrorPayload` and `runLevelError` are computed below. A failed embedded run with error payloads can therefore skip normal cron failure handling and playbook/error signals when critic mode is enabled.
  - confidence: 0.86
- **[P1] Block dynamic imports in sandbox validation:** `src/agents/sandbox/execution-policy.ts:170-171`
  - body: `validateSandboxPythonExecution` only checks modules returned by the static import parser. Code such as `__import__("subprocess")` does not produce an import usage, so a blocked module can bypass the new allow/deny policy.
  - confidence: 0.85
- **[P2] Bound running trace storage:** `src/agents/agent-run-trace.ts:74-79`
  - body: Running timelines are retained for a time window but are not hard-capped by count, and spans are appended without a per-timeline limit. Since agent run IDs come from the client idempotency key, repeated incomplete runs can grow gateway memory until the running-retention window expires.
  - confidence: 0.8
- **[P2] Redact cron errors before storing playbook memory:** `src/cron/procedural-playbook-memory-v0.ts:234-237`
  - body: The new playbook memory writes `signal.error` directly into persisted state. Runtime error strings can include URLs, headers, user content, or stack traces, so persist a redacted and length-bounded value instead of the raw error text.
  - confidence: 0.76
- **[P2] Strip inline comments from parsed imports:** `src/agents/sandbox/execution-policy.ts:54-59`
  - body: `parseImportSegment` strips aliases but not trailing comments, so valid Python like `import os  # needed` is classified as an unknown dependency root instead of the denied stdlib module. Strip inline comments before root classification.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The diff introduces a trace-read RPC, sandbox execution policy, and persisted cron error memory with concrete security gaps.

Concerns:

- **[high] Timeline endpoint lacks object authorization:** `src/gateway/server-methods/agent.ts:709`
  - body: `agent.timeline` returns a timeline by caller-supplied `runId` without verifying ownership against request context, while the response can include `sessionKey`, tool/model metadata, and free-form error/note fields.
  - confidence: 0.89
- **[high] Sandbox import policy misses dynamic imports:** `src/agents/sandbox/execution-policy.ts:170`
  - body: The Python execution policy validates static imports only, allowing dynamic import primitives to bypass denied modules such as `subprocess`.
  - confidence: 0.85
- **[medium] Running trace state can grow in memory:** `src/agents/agent-run-trace.ts:74`
  - body: Running timelines and spans lack hard count limits, creating a memory growth path under repeated incomplete runs.
  - confidence: 0.8
- **[medium] Cron playbook memory stores raw errors:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: Persisting unredacted runtime error strings can retain secrets or private user data on disk.
  - confidence: 0.76
