## Summary

Keep open. Current main does not contain the proposed run-trace endpoint, but this PR is not mergeable as-is because the branch adds multiple new protocol/config/security-sensitive surfaces, has concrete correctness/security findings, and still lacks after-fix real behavior proof.

## What This Changes

The PR adds `agent.timeline` run trace timelines and also introduces cron critic-loop config, sandbox execution policy/templates/simulation helpers, plan-search metadata, generated protocol models, docs, and test changes.

## Best Possible Solution

Split this into focused maintainer-owned PRs: first a versioned run-trace/timeline API aligned with the open unified trace tracker and hardened with authorization, retention caps, and redaction; then separate cron critic-loop, sandbox execution, plan-search, and documentation changes with their own proof.

## Solution Assessment

No. The branch is too broad and the trace endpoint needs a safer contract with caller authorization, data minimization, bounded retention, and alignment with the open unified trace-schema tracker before this is the right solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Require run ownership before returning timelines:** `src/gateway/server-methods/agent.ts:710-724`
  - body: `agent.timeline` looks up any caller-supplied `runId` and returns `sessionKey` plus raw spans without checking that the requesting client owns that run. Since agent run IDs come from idempotency keys and this method is placed in read scope, a read-capable client can retrieve another run's trace if it learns or guesses the ID. Bind traces to the caller/session or require an elevated scope and redact sensitive fields.
  - confidence: 0.93
- **[P1] Reject dynamic Python imports in sandbox validation:** `src/agents/sandbox/execution-policy.ts:170-171`
  - body: The sandbox policy only checks modules returned by `extractPythonImportUsages`, which parses static `import` and `from` statements. Generated code can still call `__import__("subprocess")` and pass validation because no usage is produced, bypassing the deny list. Detect and reject dynamic import primitives before rendering/execution.
  - confidence: 0.91
- **[P1] Classify cron run errors before critic early return:** `src/cron/isolated-agent/run.ts:797-805`
  - body: When the critic returns `needs_replan`, this branch exits with `status: "ok"` before the later `hasFatalErrorPayload` and `runLevelError` checks run. A failed embedded agent result that also scores below the critic threshold will be recorded as a successful replan instead of an execution error, suppressing normal failure handling. Move error classification before the critic return or gate the return on no fatal run error.
  - confidence: 0.9
- **[P1] Cap running trace storage:** `src/agents/agent-run-trace.ts:104-106`
  - body: Each new run stores a timeline in the module-level map, but the cap only applies to terminal timelines while running timelines and their span arrays can grow until age-based pruning. Because run IDs are caller-provided idempotency keys, repeated long-running or abandoned runs can retain unbounded trace state. Add hard global/per-session running caps and a per-timeline span cap.
  - confidence: 0.86
- **[P2] Strip inline comments from import segments:** `src/agents/sandbox/execution-policy.ts:54-60`
  - body: `parseImportSegment` only strips aliases, so valid Python such as `import os  # path ops` is parsed with the comment as part of the module name. That changes the root classification and can reject otherwise valid allowlisted imports. Remove trailing comments before alias/module parsing.
  - confidence: 0.82
- **[P2] Redact persisted cron error text:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: The playbook memory stores `signal.error` verbatim in `lastError`. Those strings can include provider URLs, tokens, user text, or stack details and are also unbounded in length. Redact and truncate before saving, or store only a classification/hash by default.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces concrete security-sensitive concerns in the new trace endpoint, sandbox execution validator, trace retention, and persisted cron error memory.

Concerns:

- **[high] Trace endpoint lacks object-level authorization:** `src/gateway/server-methods/agent.ts:710`
  - body: `agent.timeline` returns timeline metadata and raw spans by runId without checking caller ownership, while the method is registered under read scope and run IDs are derived from client idempotency keys.
  - confidence: 0.93
- **[high] Sandbox validator misses dynamic import bypasses:** `src/agents/sandbox/execution-policy.ts:170`
  - body: The new Python execution policy validates only static import syntax, so dynamic import primitives can reach denied modules without producing a validation issue.
  - confidence: 0.91
- **[medium] Running trace storage lacks hard caps:** `src/agents/agent-run-trace.ts:104`
  - body: The in-memory timeline map caps terminal entries but not running entries or spans per timeline, allowing repeated abandoned runs to retain unbounded trace state for the running retention window.
  - confidence: 0.86
- **[medium] Cron playbook memory persists raw errors:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: `lastError` is saved from runtime error strings without redaction or length bounds, which can persist secrets, private endpoints, user content, or large stack strings to disk.
  - confidence: 0.82
