## Summary

Keep open: current main has some small supporting fixes, but not the run trace timeline endpoint, and this PR still carries unique feature work. The branch is not mergeable as-is because source inspection found correctness and security blockers, and the external contributor has not provided real behavior proof.

## What This Changes

The PR adds an in-memory agent run-trace timeline and `agent.timeline` RPC, plus block-reply/chunker fixes, plan search, cron critic/playbook memory, sandbox execution/simulation helpers, protocol/client updates, tests, and docs.

## Best Possible Solution

Split this into maintainer-reviewed pieces, starting with a secured, versioned trace contract aligned with the open unified run-trace tracker, then land unrelated cron/sandbox/planner work separately after focused security review and real behavior proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection of the PR head shows the critic early return before fatal-error classification, the unauthenticated `agent.timeline` read path, and the static-only sandbox import validation. I did not run the branch or establish live after-fix behavior proof.

## Solution Assessment

No. The trace idea is useful, but this XL branch combines multiple product/config/security surfaces and exposes trace data without a clear authorization or schema boundary; a narrower secured trace PR is the maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Classify run errors before returning critic replan:** `src/cron/isolated-agent/run.ts:797-805`
  - body: With critic loop enabled, this branch returns `status: "ok"` for `needs_replan` before the later fatal error payload classification runs. A run that only produced error payloads can therefore skip normal failure handling and downstream error signals.
  - confidence: 0.91
- **[P1] Reject dynamic Python import escapes:** `src/agents/sandbox/execution-policy.ts:170-171`
  - body: The validator only checks modules returned by `extractPythonImportUsages`, which covers static `import`/`from` syntax. Code such as `__import__("subprocess")` can bypass the allow/deny policy, so dynamic import primitives need to be detected or forbidden.
  - confidence: 0.9
- **[P1] Authorize timeline reads before returning spans:** `src/gateway/server-methods/agent.ts:709-725`
  - body: `agent.timeline` looks up traces by caller-supplied `runId` and returns `sessionKey` plus raw spans without checking that the caller owns that run. A paired read client that learns a run id could read another run's trace details.
  - confidence: 0.88
- **[P2] Cap running trace timelines and spans:** `src/agents/agent-run-trace.ts:104-107`
  - body: Each new running timeline is stored in a process-global map, but only terminal timelines get a count cap. Many incomplete runs with distinct ids can retain unbounded timeline/span data until age pruning catches up.
  - confidence: 0.86
- **[P2] Strip inline comments from import names:** `src/agents/sandbox/execution-policy.ts:54-59`
  - body: `parseImportSegment` removes aliases but not trailing comments, so valid Python like `import os  # path ops` is classified as a different dependency root and can be rejected incorrectly.
  - confidence: 0.84
- **[P2] Redact cron errors before persisting playbook memory:** `src/cron/procedural-playbook-memory-v0.ts:234-237`
  - body: The playbook memory stores `signal.error` verbatim in JSON state. Runtime error strings can include URLs, user content, stack traces, or tokens, so this should be redacted and length-bounded before persistence.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces new debug-data access, generated-code validation, and persisted-error surfaces with concrete security concerns.

Concerns:

- **[high] Timeline RPC lacks object-level authorization:** `src/gateway/server-methods/agent.ts:715`
  - body: The new `agent.timeline` handler returns trace spans and `sessionKey` by run id without verifying the requesting client owns that run, creating an information-disclosure risk for paired read clients.
  - confidence: 0.89
- **[medium] Running trace storage lacks hard caps:** `src/agents/agent-run-trace.ts:104`
  - body: Running timelines and their spans are stored process-wide with age pruning but no count/span cap, so repeated incomplete runs can grow memory until retention expires.
  - confidence: 0.86
- **[high] Sandbox validation misses dynamic imports:** `src/agents/sandbox/execution-policy.ts:170`
  - body: The sandbox policy only evaluates statically parsed imports, allowing dynamic import primitives to bypass denied module/dependency checks.
  - confidence: 0.9
- **[medium] Cron playbook memory persists raw errors:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: `signal.error` is written to persistent playbook memory without redaction or truncation, which can store secrets or private data from arbitrary runtime exceptions.
  - confidence: 0.8
