## Summary

Keep open. The PR targets a real Docker/Linux skill-install failure, but the proposed apt fallback widens the package-manager trust boundary and still needs maintainer strategy approval, branch refresh, and real Linux behavior proof before merge.

## What This Changes

The branch refactors apt installs into a shared helper, adds a Linux apt-get fallback for missing-Homebrew brew skill installers, and expands mocked fallback tests.

## Best Possible Solution

Choose one documented Docker/Linux skill-dependency strategy that preserves the untrusted-skill boundary; if apt fallback is selected, gate it to trusted or explicit opt-in installs, keep sudo non-interactive, add focused tests, and provide real Docker/Linux proof.

## Reproduction Assessment

Yes. Source inspection shows current main returns the brew-missing failure for brew installers when no brew executable resolves, and the linked Docker issue provides the user-facing openai-whisper path; I did not run a live Docker or apt install in this read-only review.

## Solution Assessment

No. The apt fallback addresses the visible error, but it is not the best mergeable solution as-is because it changes the privileged package-manager trust boundary and leaves sudo interactivity unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate apt fallback behind trusted or opt-in installs:** `src/agents/skills-install.ts:553-555`
  - body: The new missing-brew branch runs apt-get for any brew installer metadata after only warning on non-bundled skill sources. Workspace and personal skills can be attacker-controlled, so root or passwordless-sudo Linux hosts could now perform privileged package installs that current main rejects; gate this to trusted sources or an explicit opt-in before invoking apt.
  - confidence: 0.92
- **[P2] Pass non-interactive sudo to apt commands:** `src/agents/skills-install.ts:327-328`
  - body: The shared apt helper probes sudo with `sudo -n true` but then runs `sudo apt-get update/install` without `-n`. Since the command runner inherits stdin when no input is provided, sudo can still prompt or block until timeout when apt-get is denied or needs a password; keep the follow-up sudo commands non-interactive or close stdin.
  - confidence: 0.78
