## Summary

Keep open. The PR addresses a real Docker/Linux skill-install failure, but the proposed apt fallback changes the package-manager trust boundary and still lacks after-fix proof from a real Linux apt/root/sudo setup.

## What This Changes

The PR adds a Linux apt-get fallback for missing-Homebrew `brew` skill installers, refactors apt installs into a shared helper, and expands mocked fallback tests.

## Best Possible Solution

Choose one documented Linux/Docker dependency strategy that preserves the untrusted-skill boundary, then add focused regression coverage plus real Docker or Linux install proof.

## Reproduction Assessment

Yes. Source inspection shows current main returns the brew-missing failure for brew installers without Homebrew, and the PR diff shows the new apt-get execution path; I did not run a live Docker or apt install.

## Solution Assessment

No. The fallback addresses the visible failure, but it changes privilege and trust policy; a safer fix needs maintainer approval and either trusted/opt-in gating or one of the open Docker/onboarding alternatives.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate apt fallback to trusted skill sources:** `src/agents/skills-install.ts:553-555`
  - body: The new missing-brew branch invokes apt-get for any brew installer after only warning on non-bundled sources. Workspace and personal skill metadata can be attacker-controlled, so root or passwordless-sudo Linux hosts could run privileged package installs that current main rejects with a brew-missing error; gate this to trusted or allowlisted installers, or require explicit opt-in, before running apt.
  - confidence: 0.92
- **[P2] Keep sudo apt commands non-interactive:** `src/agents/skills-install.ts:327-328`
  - body: The shared apt helper probes sudo with `sudo -n true` but then runs `sudo apt-get update/install` without `-n`. If sudo allows the probe but prompts or denies apt-get, the installer can block at a password prompt until timeout; pass `-n` to the follow-up sudo commands or close stdin.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces privileged apt-get installs from skill metadata and leaves follow-up sudo apt commands interactive.

Concerns:

- **[high] Privileged apt install from skill metadata:** `src/agents/skills-install.ts:553`
  - body: The PR runs apt-get or sudo apt-get install for brew formulas when Homebrew is missing on Linux, but current main treats workspace/project/personal skill installer metadata as potentially attacker-controlled and only warns for non-bundled sources.
  - confidence: 0.92
- **[medium] Interactive sudo prompt can stall installs:** `src/agents/skills-install.ts:327`
  - body: After a non-interactive sudo probe succeeds, the helper runs sudo apt-get without `-n`; inherited stdin can allow sudo to prompt or block until timeout in sudoers configurations where apt-get still needs a password or is denied interactively.
  - confidence: 0.78
