## Summary

Keep open. Current main still returns the brew-missing failure for brew installers without Homebrew, but this PR should not merge as-is because it changes the skill installer trust and privilege model by adding automatic apt-get/sudo execution from skill metadata.

## What This Changes

The PR refactors skill apt installation and adds a Linux apt-get fallback plus tests for brew-based skill installers when Homebrew is unavailable.

## Best Possible Solution

Choose one documented Linux/Docker skill dependency strategy that preserves the untrusted-skill boundary, then implement it with focused installer/onboarding tests and a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main returns the brew-missing failure for brew installers without Homebrew, and the PR diff shows the new apt-get path that creates the review findings.

## Solution Assessment

No. The proposed fallback addresses the visible Docker install failure, but it changes privilege and trust boundaries; maintainers should first choose apt fallback, Linuxbrew Docker support, or container-specific onboarding behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate apt fallback before invoking apt:** `src/agents/skills-install.ts:553-555`
  - body: The new no-brew branch calls `installBrewFormulaViaApt` for any brew installer whenever Linux has `apt-get`, but current main only warns for workspace/project/personal skills and documents third-party skills as untrusted. On root or passwordless-sudo hosts this lets non-bundled skill metadata trigger privileged OS package installs where main currently returns a brew-missing error; gate this to trusted or allowlisted installers, or require explicit opt-in, before running apt.
  - confidence: 0.92
- **[P2] Keep sudo apt invocations non-interactive:** `src/agents/skills-install.ts:327-328`
  - body: After probing with `sudo -n true`, the shared apt helper runs `sudo apt-get update/install` without `-n`. The new brew fallback reuses this path, so sudoers configurations that allow the probe but prompt or deny apt-get can block until timeout; pass `-n` or close stdin on the follow-up sudo commands.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/agents/skills-install.ts:551-553`
  - body: This changes user-facing skill dependency installation behavior, but the PR only updates installer code and tests. Repo policy requires a `CHANGELOG.md` entry for user-facing fixes or features so release notes capture the Linux/Docker installer change.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces privileged apt-get/sudo execution from skill installer metadata and leaves follow-up sudo apt commands interactive.

Concerns:

- **[high] Privileged apt install from skill metadata:** `src/agents/skills-install.ts:553`
  - body: The PR adds `apt-get` or `sudo apt-get install` for brew formulas when Homebrew is missing on Linux, but current main treats workspace/project/personal skill installer metadata as potentially attacker-controlled and only warns for non-bundled sources.
  - confidence: 0.92
- **[medium] Interactive sudo apt path can stall installs:** `src/agents/skills-install.ts:327`
  - body: The helper probes with `sudo -n true` and then runs `sudo apt-get` without `-n`; because the process helper inherits stdin by default, sudo can prompt or block until timeout in some sudoers configurations.
  - confidence: 0.78
