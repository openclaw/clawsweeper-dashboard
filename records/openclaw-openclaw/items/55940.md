## Summary

Keep open for maintainer handling: the linked Docker skill-install failure still exists on current main, but this PR is blocked by privileged package-manager trust concerns, an interactive sudo path, missing real Linux proof, and a conflicting branch.

## What This Changes

The PR adds a Linux apt-get fallback for missing-Homebrew brew skill installers and expands mocked installer fallback tests.

## Best Possible Solution

Choose one documented Docker skill-dependency strategy; if apt fallback is selected, gate it to trusted or explicit opt-in installs, keep sudo non-interactive, update coverage/docs, and require real Linux proof.

## Reproduction Assessment

Yes from source inspection: current main still returns the brew-missing failure for brew installers when no brew executable resolves, and openai-whisper remains brew-only. I did not run a live Docker or apt install in this read-only review.

## Solution Assessment

No. The apt fallback addresses the visible symptom, but it is not the best mergeable solution until maintainers choose the package-manager trust posture and the patch gates privileged installs and keeps sudo non-interactive.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate apt fallback to trusted skill sources:** `src/agents/skills-install.ts:553-555`
  - body: This missing-brew branch runs apt-get or sudo apt-get for any brew installer metadata after current main only warns that workspace and personal skill metadata can be attacker-controlled. That changes the no-brew path from a safe failure into privileged package installation on root/passwordless-sudo Linux hosts, so gate the fallback to trusted or explicitly opted-in installers before invoking apt.
  - confidence: 0.92
- **[P2] Keep sudo apt commands non-interactive:** `src/agents/skills-install.ts:327-328`
  - body: The helper probes sudo with `sudo -n true` but then runs `sudo apt-get update/install` without `-n`. Because the command runner inherits stdin when no input is set, sudo can still prompt or block until timeout in sudoers configurations where apt-get needs a password or is denied; pass `-n` to the follow-up sudo commands or close stdin.
  - confidence: 0.78
