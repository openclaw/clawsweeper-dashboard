## Summary

Keep open. The PR targets a real Linux/Docker skill-install failure, but the current diff introduces a privileged apt-get path from skill installer metadata and needs maintainer security/product judgment before merge.

## What This Changes

The PR refactors the skill apt installer helper and adds a Linux apt-get fallback for `kind: "brew"` skill installs when Homebrew is unavailable, with fallback tests.

## Best Possible Solution

Choose one documented Linux/Docker dependency strategy that preserves the untrusted-skill boundary, such as optional Linuxbrew image support, container-specific hiding, or a tightly gated apt opt-in for trusted installers only.

## Reproduction Assessment

Yes. Source inspection shows current main returns the brew-missing failure for brew installers without Homebrew, and the PR diff shows the new apt-get execution path that creates the review findings.

## Solution Assessment

No. The proposed fallback fixes the visible install failure, but it broadens the installer trust and privilege model; a trusted-source gate, explicit opt-in, Linuxbrew Docker support, or container-specific onboarding behavior would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate apt fallback to trusted skill sources:** `src/agents/skills-install.ts:553-555`
  - body: The new no-brew branch invokes `apt-get` for any brew installer after only warning on non-bundled skill sources. Workspace and personal skill metadata can be attacker-controlled, so on root or passwordless-sudo Linux hosts this lets a third-party skill trigger privileged OS package installation where main currently returns a brew-missing error; gate this to trusted or allowlisted installers, or require explicit opt-in, before running apt.
  - confidence: 0.92
- **[P2] Keep sudo apt commands non-interactive:** `src/agents/skills-install.ts:327-329`
  - body: Now that the shared apt helper is used for brew formulas too, it probes with `sudo -n true` but then runs `sudo apt-get update/install` without `-n`. If sudo allows the probe but prompts or denies apt-get, the installer can block at a password prompt until timeout; use `sudo -n` for the follow-up apt commands or close stdin.
  - confidence: 0.78
