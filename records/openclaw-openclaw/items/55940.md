## Summary

Keep open. The PR addresses a real Docker skill-install pain point, but the current diff adds privileged apt installs from skill metadata, needs maintainer product/security approval, and lacks after-fix proof from a real Linux install path.

## What This Changes

The PR adds a shared apt installer helper, uses it to let missing-Homebrew brew skill installers fall back to apt-get on Linux, and adds mocked fallback tests.

## Best Possible Solution

Choose one documented Linux/Docker dependency strategy that preserves the untrusted-skill boundary, then add focused regression coverage and real Docker or Linux install proof.

## Reproduction Assessment

Yes. Source inspection shows current main returns the brew-missing failure for brew installers without Homebrew, and the PR diff shows the new apt-get execution path that creates the review findings.

## Solution Assessment

No. The proposed fallback addresses the visible install failure, but it changes the privilege and trust model; a safer path needs maintainer approval and either trusted/opt-in gating or one of the open Docker/onboarding alternatives.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate apt fallback to trusted skill sources:** `src/agents/skills-install.ts:553-555`
  - body: The new no-brew branch invokes apt-get for any brew installer after only warning on non-bundled skill sources. Workspace and personal skill metadata can be attacker-controlled, so root or passwordless-sudo Linux hosts could run privileged package installs that current main rejects with a brew-missing error; gate this to trusted or allowlisted installers, or require explicit opt-in, before running apt.
  - confidence: 0.92
- **[P2] Keep sudo apt commands non-interactive:** `src/agents/skills-install.ts:327-328`
  - body: The shared apt helper probes sudo with `sudo -n true` but then runs `sudo apt-get update/install` without `-n`. If sudo allows the probe but prompts or denies apt-get, the installer can block at a password prompt until timeout; pass `-n` to the follow-up sudo commands or close stdin.
  - confidence: 0.78
