## Summary

Keep open: current main and the latest release still route Codex app-server auth refresh callbacks through a forced OAuth refresh, while upstream Codex still cancels that callback after 10 seconds. The 120s OpenClaw OAuth refresh budget helps OpenClaw's own refresh manager, but it does not remove the upstream callback deadline that matches the reported failures.

## What This Changes

The issue reports valid OpenAI/Codex OAuth profiles failing cron, heartbeat, and embedded agent runs when token refresh exceeds a 10-second Codex app-server auth callback timeout.

## Best Possible Solution

Keep this issue as the canonical narrow bug until the Codex auth bridge can answer external auth callbacks from usable cached tokens quickly, with genuinely expired or invalid tokens taking the slower refresh path and broader alerting tracked separately.

## Reproduction Assessment

Yes from source inspection: Codex still imposes a 10s external-auth response deadline, while OpenClaw still answers that request through a forced OAuth refresh. I did not run a live OAuth current-main repro in this read-only review.

## Solution Assessment

No, closing this on the 120s OpenClaw timeout constant would be incomplete because the upstream Codex callback deadline remains 10s. The narrower maintainable fix is to return usable cached tokens quickly for that callback and reserve forced refresh for credentials that actually require it.
