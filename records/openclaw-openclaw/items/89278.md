## Summary

Keep open: current main and the latest v2026.6.8 release still route the Codex app-server auth-refresh callback through a forced OAuth refresh, while upstream Codex still cancels that server-to-client refresh RPC after 10 seconds. The recent maintainer-machine repro confirms the same timeout signature on v2026.6.8, so this is not implemented on main or superseded by the closed Google OAuth timeout PR.

## What This Changes

The issue reports that valid OpenAI/Codex OAuth sessions can pass status probes but fail cron, heartbeat, and agent runs when Codex app-server auth refresh takes longer than a 10-second callback timeout.

## Best Possible Solution

Have maintainers choose the owner-boundary fix, likely returning a still-usable non-expired Codex OAuth token from OpenClaw's callback and reserving forced refresh for expired or unauthorized cases while tracking upstream Codex configurability separately.

## Reproduction Assessment

Yes for source-level reproduction: current main still force-refreshes Codex app-server OAuth credentials while upstream Codex cancels the external auth refresh callback after 10 seconds. I did not run a live OAuth E2E reproduction in this read-only review.

## Solution Assessment

Unclear: increasing timeouts, adding a config knob, and cached-token reuse are all plausible, but cached-token reuse looks like the narrowest OpenClaw-side mitigation if maintainers accept that auth semantics change.
