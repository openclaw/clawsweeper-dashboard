## Summary

Keep open: current main and the latest v2026.6.8 release still force the Codex app-server token-refresh callback through a full OAuth refresh, while upstream Codex still cancels that external refresh RPC after 10 seconds. The closed Google OAuth timeout PR did not land and does not address this Codex app-server boundary, and the fresh v2026.6.8 discussion confirms the same user-visible failure signature.

## What This Changes

The issue asks OpenClaw to stop valid Codex OAuth sessions from failing cron, heartbeat, and embedded agent runs when the app-server auth-refresh callback exceeds a hard 10-second timeout.

## Best Possible Solution

Add a maintainer-approved Codex auth bridge fix that answers app-server refresh RPCs from still-valid cached OAuth credentials or otherwise avoids the upstream 10-second request budget, with any broader incident/status work tracked explicitly.

## Reproduction Assessment

Yes for source reproduction: current main forces the Codex app-server auth-refresh callback through OAuth refresh, and upstream Codex cancels that callback after 10 seconds. I did not run a live OAuth account repro in this read-only pass.

## Solution Assessment

No complete solution is on main. A one-off Google OAuth timeout bump is not the right fix; the maintainable path is an OpenClaw Codex bridge mitigation and/or upstream Codex timeout configurability, with maintainer direction on status reporting.
