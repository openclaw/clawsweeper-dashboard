## Summary

Keep open: current main still answers Codex app-server `account/chatgptAuthTokens/refresh` callbacks by forcing an OpenClaw OAuth refresh, while current upstream Codex still cancels that external-auth callback after 10 seconds. The 120s OpenClaw OAuth budget is real, but it does not remove the upstream callback deadline reported by the issue and reconfirmed on v2026.6.8.

## What This Changes

The issue reports valid OpenAI/Codex OAuth probes succeeding while cron, heartbeat, and embedded agent runs fail with a 10-second Codex auth refresh timeout.

## Best Possible Solution

Keep this issue open until maintainers choose and validate a Codex auth-bridge mitigation, such as avoiding unnecessary forced refresh when a profile is still usable, reducing refresh-lock contention in the callback path, improving degraded auth/status messaging, or coordinating an upstream configurable timeout.

## Reproduction Assessment

Yes, at source level: current OpenClaw still forces OAuth refresh for Codex app-server refresh callbacks, and current upstream Codex still imposes the exact 10-second timeout that matches the logs. I did not run a live OAuth repro in this read-only cleanup pass.

## Solution Assessment

No, merely pointing at OpenClaw's 120s OAuth refresh constant is not the best solution because upstream Codex still times out the external-auth callback after 10 seconds. The best fix needs an owner-approved bridge or upstream-contract change plus regression coverage for cron/heartbeat or app-server runs.
