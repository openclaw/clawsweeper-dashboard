## Summary

Keep open: current main has widened OpenClaw's internal OAuth refresh call budget to 120s, but the Codex app-server callback still forces a full OAuth refresh while current upstream Codex still cancels that external-auth callback after 10s. The latest release also still has the forced-refresh bridge, and the issue discussion includes fresh v2026.6.8 evidence of the same user-visible failure signature.

## What This Changes

The issue asks OpenClaw to prevent valid OpenAI/Codex OAuth sessions from failing cron, heartbeat, and agent runs when refresh takes longer than the Codex app-server's 10s auth callback budget.

## Best Possible Solution

Keep this issue as the narrow tracker for making the Codex app-server auth callback answer within the upstream 10s budget for valid usable credentials, or for wiring an upstream configurable timeout once available; broader alerting/quarantine work should stay in the related incident issue.

## Reproduction Assessment

Yes, source-reproducible: current OpenClaw still forces OAuth refresh in the Codex app-server callback, and current upstream Codex still cancels that callback after 10 seconds. I did not run a live OAuth account repro in this read-only pass.

## Solution Assessment

No, current main's 120s OpenClaw refresh budget is not the full fix because the Codex app-server still waits only 10s for the callback. The likely maintainable fix is a maintainer-approved cached-token or upstream-timeout path, not closing based on the internal timeout constant alone.
