## Summary

Keep open: current main and v2026.6.8 still force-refresh Codex app-server OAuth tokens, while the inspected upstream Codex app-server still cancels the external auth refresh RPC after 10 seconds. The fresh v2026.6.8 comment confirms the same user-visible failure remains, and the closed OpenClaw PR changed a different Google OAuth timeout path rather than this Codex app-server path.

## What This Changes

The issue reports valid OpenAI/Codex OAuth sessions failing cron, heartbeat, and agent turns when external auth refresh exceeds a hard 10-second timeout.

## Best Possible Solution

Choose the auth owner boundary, then implement either upstream timeout configurability or a narrow OpenClaw valid-token reuse mitigation with regression coverage and clearer degraded-auth surfacing.

## Reproduction Assessment

Yes from source and report evidence: current OpenClaw still force-refreshes Codex app-server OAuth tokens, and upstream Codex still times out the external refresh RPC after 10 seconds with the reported error. A live reproduction would require real OpenAI/Codex OAuth credentials and a slow or contended refresh path.

Unclear: the issue is valid, but the requested menu includes multiple possible fixes and the narrowest maintainable one depends on the Codex/OpenClaw owner boundary. A config knob should not be the default OpenClaw answer unless cached-token reuse or upstream configurability is insufficient.
