Keep open: current main still has a source-backed path from official OpenAI model selection to an unregistered Codex harness and then a generic OpenAI-compatible HTTP 500. The issue reports that `/v1/chat/completions` can pass readiness but fail with HTTP 500 after default OpenAI agent routing selects an unregistered `codex` harness.

## Best Possible Solution

Make the release/Docker Codex plugin contract, default OpenAI runtime selection, startup/readiness checks, provider/model PI override shape, and OpenAI-compatible error response agree in code and docs.
