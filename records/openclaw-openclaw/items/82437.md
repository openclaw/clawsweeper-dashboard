## Summary

Keep open: source inspection supports the reported failure path, and the remaining fix is not a cleanup close. Current main still combines default OpenAI-to-Codex runtime selection, release/Docker paths that can omit the Codex plugin, readiness that does not validate agent harness availability, and OpenAI-compatible HTTP fallback to a generic 500.

## What This Changes

The issue reports a v2026.5.12 regression where OpenAI-compatible chat completions pass readiness but fail with an unregistered Codex harness at request time.

## Best Possible Solution

Decide the intended release contract, then make default OpenAI runtime, Codex plugin availability, readiness, and OpenAI-compatible error handling agree in code and docs.

## Reproduction Assessment

Yes, source-reproducible: current main routes official openai models to Codex by default, Docker/release paths can omit the Codex plugin, and harness selection throws the exact unregistered Codex error that the HTTP handler can surface as a generic 500. I did not run the live image.

## Solution Assessment

Unclear: the report lists several valid product outcomes, and the narrowest maintainable fix depends on whether release images should include Codex or force operators to install/build it explicitly. At minimum, the endpoint should stop hiding this configuration/runtime error behind a generic 500.
