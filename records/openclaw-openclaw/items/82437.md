## Summary

Keep open: current main still has a source-backed path from official OpenAI model selection to an unregistered Codex harness and then a generic OpenAI-compatible HTTP 500, while the remaining fix crosses Docker/release packaging, startup/readiness semantics, and runtime policy.

## What This Changes

The issue reports a v2026.5.12 regression where `/v1/chat/completions` with `model: "openclaw:main"` fails after selecting an unavailable `codex` agent harness.

## Best Possible Solution

Make the release/Docker Codex plugin contract, default OpenAI runtime selection, startup/readiness checks, provider/model PI override shape, and OpenAI-compatible error response agree in code and docs.

## Reproduction Assessment

Yes, source-reproducible: official OpenAI model refs select Codex by default, Docker/package paths can omit the Codex plugin, harness selection throws the exact unregistered-Codex error, and the chat-completions handler maps that ordinary error to a generic 500. I did not run the live image.

Unclear: the report lists several acceptable product outcomes, and the narrowest maintainable fix depends on whether release images should include Codex, readiness should fail on missing runtime owners, or this deployment should be routed through provider/model-scoped PI. A small diagnostic-only PR would not fully solve this endpoint failure.
