## Summary

Keep this PR open, but it is not merge-ready: the submitted fix broadens the shared media SSRF boundary instead of proving a gateway-owned media path, and the PR still has no after-fix real Feishu/gateway proof.

## What This Changes

The PR changes `src/media/web-media.ts` to add `localhost` and `127.0.0.1` to the SSRF allowed hostnames before shared HTTP(S) media fetching.

## Best Possible Solution

Replace the broad localhost allowlist with a scoped OpenClaw-owned inbound media/media-store resolution path that keeps generic HTTP(S) loopback blocked, then add regression tests and real Feishu/gateway proof.

## Reproduction Assessment

No high-confidence live reproduction was run for the full Feishu flow. Source inspection does show literal HTTP localhost media URLs are blocked by the shared SSRF guard, while current Feishu code appears to hand off saved media as local/media-store paths.

## Solution Assessment

No. The proposed fix is too broad because it changes the shared remote media SSRF policy; the safer solution is scoped inbound media resolution that proves OpenClaw ownership before any local read.
