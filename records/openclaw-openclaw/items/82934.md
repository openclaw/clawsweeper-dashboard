## Summary

Keep open: the linked Feishu failure looks plausible, but this PR is not mergeable as-is because it broadly exempts loopback media URLs from the generic SSRF guard and lacks after-fix real behavior proof.

## What This Changes

This PR changes `src/media/web-media.ts` so HTTP(S) media URLs hosted on `localhost` or `127.0.0.1` get those hostnames added to the SSRF allowlist before `readRemoteMediaBuffer` runs.

## Best Possible Solution

Keep the linked bug open and replace this with a scoped gateway-media fix that resolves OpenClaw-managed inbound media without weakening generic remote media SSRF checks.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows any `http://localhost:18888/...` value passed to the image/media loader will hit the current SSRF block, and the linked issue provides a matching Feishu error log.

## Solution Assessment

No. The fix should be scoped to gateway-issued media references, exact trusted origin/path handling, or direct `media://inbound/...` resolution instead of allowlisting loopback hostnames in the shared remote media fetch path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92
