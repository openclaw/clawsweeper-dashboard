## Summary

Keep open: the linked Feishu media failure is plausible from source and issue evidence, but this PR should not merge because it weakens the generic media SSRF boundary and still lacks after-fix real behavior proof.

## What This Changes

The PR adds a localhost/127.0.0.1 SSRF allowlist override in `src/media/web-media.ts` before generic HTTP(S) media fetches.

## Best Possible Solution

Resolve OpenClaw-managed inbound media through `media://inbound` or an exact trusted gateway media origin/path contract without changing generic remote media SSRF checks.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows a current-main `http://localhost:18888/...` media URL reaches the SSRF-guarded HTTP fetch path, and the linked Feishu issue provides a matching failure log.

## Solution Assessment

No. The PR is not the best fix because it allows generic loopback media fetches; a scoped gateway-media or direct inbound-media resolution path is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep localhost out of the generic media SSRF allowlist:** `src/media/web-media.ts:536-540`
  - body: This helper runs for every HTTP(S) media URL, including prompt/tool-supplied URLs. Adding `localhost` and `127.0.0.1` to `allowedHostnames` lets arbitrary loopback media URLs skip the private-network checks, reopening the local-service SSRF class this guard is meant to block.
  - confidence: 0.94

## Security Review
