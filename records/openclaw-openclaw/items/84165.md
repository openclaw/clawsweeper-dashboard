## Label Justifications

- `P1`: The PR addresses silent premature termination of active WebChat agent runs that use media-returning tools on non-vision models.
- `merge-risk: 🚨 message-delivery`: The deferred-write implementation can drop already-delivered media from persisted chat history on dispatch failure.
- `merge-risk: 🚨 session-state`: The change alters when transcript rows enter the Pi session chain during agent-run lifecycle cleanup.

## Summary

Keep open: the fix targets a real current-main race and has useful proof, but the deferred media payload is only flushed on the success post-dispatch path, so media already delivered to WebChat can be dropped from history if dispatch or the pre-flush rewrite fails.

## What This Changes

The PR defers WebChat agent media transcript persistence from the live delivery callback to the post-dispatch phase in `src/gateway/server-methods/chat.ts`.

## Best Possible Solution

Keep the deferred transcript write, but flush it exactly once from shared post-settle cleanup on both success and error paths, with focused gateway coverage for the timing boundary.

## Reproduction Assessment

Yes, by source inspection and contributor proof: current main appends a `gateway-injected` assistant stop row during dispatcher delivery after `onAgentRunStart`, and the PR body includes v2026.5.18 WebChat recordings/logs showing cleanup before a follow-up tool call. I did not execute the live hy3/browser repro locally.

## Solution Assessment

No, not yet. Deferring the append until dispatch settles is the right boundary, but the proposed patch needs shared success/error cleanup so deferred media is flushed after failed runs or pre-flush rewrite errors too.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Flush deferred media on failed dispatches:** `src/gateway/server-methods/chat.ts:2708`
  - body: This flush only runs in the success post-dispatch branch. If the agent already delivered media and then `dispatchInboundMessage` rejects, or `rewriteUserTranscriptMedia()` throws before this line, execution jumps to the `.catch` path where the deferred payload is never persisted. Before this PR the media row was appended at delivery time, so this change can drop already-shown media from chat history for failed runs; the error cleanup should attempt the deferred flush after the run has settled too.
  - confidence: 0.86
