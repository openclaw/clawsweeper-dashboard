## Label Justifications

- `P1`: The PR targets a user-visible active agent run abort that can silently stop webchat tool workflows involving media-returning tools.
- `merge-risk: 🚨 message-delivery`: The patch changes when and which media payload is persisted to chat history, which can suppress later valid media messages if the first deferred payload cannot append.
- `merge-risk: 🚨 session-state`: The change intentionally moves gateway-injected transcript writes around an active Pi session lifecycle boundary, where ordering affects run completion state.

## Summary

Keep open: this PR addresses a real gateway/session-state race and the post-dispatch direction is right, but the final patch changes the existing success-based media persistence fallback by storing only the first media-bearing payload. That needs a small repair before merge.

## What This Changes

The PR defers webchat agent media transcript persistence during active runs and flushes one captured media payload in post-dispatch.

## Best Possible Solution

Defer active-run webchat media transcript writes until after dispatch, preserve the existing success-based single-append semantics across deferred payloads, and add a focused gateway seam test for that timing boundary.

## Reproduction Assessment

Yes. The PR body gives a concrete webchat/non-vision-model/media-tool path with before/after recordings and logs, and current main source shows `gateway-injected` media transcript appends can happen from the active-run delivery callback.

## Solution Assessment

No, not quite. Deferring the append until post-dispatch is the right direction, but the current first-payload capture should be replaced with ordered deferred processing that preserves the existing first-successful-append behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep trying later media until one append succeeds:** `src/gateway/server-methods/chat.ts:2523-2525`
  - body: This stores only the first media-bearing payload for the post-dispatch flush. On current main, `appendedWebchatAgentMedia` is set only after a successful append, so if an earlier media payload normalizes to no persisted media content or fails to append, a later valid media payload can still be recorded. After this change, the flush only replays the first captured payload and drops that fallback, leaving later valid media absent from chat history. Store an ordered queue or otherwise keep attempting deferred payloads until one append succeeds.
