## Label Justifications

- `P1`: The bug affects active webchat agent runs with non-vision models and media tools, causing premature cleanup and incomplete user responses.
- `merge-risk: 🚨 message-delivery`: The diff changes media transcript persistence and can drop later valid media from history if the first deferred payload cannot be persisted.
- `merge-risk: 🚨 session-state`: The touched path writes gateway-injected assistant entries into Pi session transcripts and changes when the active run observes those entries.

## Summary

Keep open: the defer-until-post-dispatch direction matches the active-run transcript race, but the branch currently changes media history persistence from first-successful append to first-captured payload, which can drop later valid media from chat history.

## What This Changes

The PR defers webchat agent media transcript appends during active gateway runs and flushes the captured media payload after dispatch completes.

## Best Possible Solution

Keep the active-run transcript deferral, but preserve first-successful append semantics by flushing deferred media payloads in delivery order until one append succeeds, with a focused gateway regression test.

## Reproduction Assessment

Yes. The PR body provides a concrete webchat, non-vision model, browser screenshot-to-snapshot path with before/after recordings and trace snippets, and current main source shows the gateway-injected append can happen from the active-run delivery callback.

## Solution Assessment

No, not yet. Deferring the append is the right boundary fix, but the implementation should preserve current first-successful media append behavior instead of capturing only the first media-bearing payload before persistability is known.
