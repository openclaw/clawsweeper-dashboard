## Label Justifications

- `P1`: The PR addresses a user-visible agent run abort where media-returning tools on non-vision models can silently stop follow-up tool execution.
- `merge-risk: 🚨 message-delivery`: The diff intentionally changes when assistant media is persisted, and the PR notes a post-dispatch failure could leave delivered media out of chat history.
- `merge-risk: 🚨 session-state`: The fix changes transcript JSONL timing to avoid confusing the Pi session lifecycle, so regression risk sits directly in session state and transcript ordering.

## Summary

Keep this PR open: the patch targets a real gateway/webchat session-state race and I found no blocking code or security finding, but maintainers should consciously accept the deferred transcript-persistence risk or ask for a focused timing regression test before merge.

## What This Changes

The PR defers webchat agent media transcript appends during active agent runs and flushes the captured media payload after dispatch completes.

## Best Possible Solution

Preserve immediate websocket media delivery, persist gateway-injected media only after the agent run is complete, and add a narrow seam test for the transcript timing when maintainers want stronger pre-merge guardrails.

## Reproduction Assessment

Yes. The PR body gives concrete webchat/hy3-preview steps plus before/after recordings and logs, and current main source shows the gateway-injected media transcript write happening during `agentRunStarted`; I did not rerun the live provider path in this read-only review.

## Solution Assessment

Yes, with one guardrail caveat. Deferring only the transcript append keeps live websocket delivery intact and targets the implicated lifecycle signal, but a focused seam test would make this safer to maintain.
