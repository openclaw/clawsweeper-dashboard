## Summary

Keep open. Current main still source-reproduces the retry-specific Telegram failure: durable recovery replays the stored `group:-...` target unchanged, while Telegram send normalization still rejects standalone `group:` recipients.

## What This Changes

The issue reports that Telegram retry delivery reuses a `group:-...` target instead of the numeric Telegram chat ID, causing retry attempts to fail and messages to end in the failed queue.

## Best Possible Solution

Preserve or normalize the canonical numeric Telegram chat ID before durable queue replay, preferably at the Telegram-owned outbound boundary, with coverage for legacy queued `group:-...` entries.

## Reproduction Assessment

Yes, at source level. A queued Telegram entry with `to: "group:-100..."` is replayed unchanged by recovery and then rejected by the Telegram send target resolver; I did not run a live Telegram retry.

## Solution Assessment

The suggested direction is right, but the safest implementation should avoid broad generic parser churn. Normalize the malformed legacy Telegram queue target at the Telegram outbound/retry boundary or carry the original numeric target through durable storage.
