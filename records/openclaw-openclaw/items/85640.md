## Summary

Keep open. Current main still replays malformed Telegram durable retry targets unchanged and still rejects a bare `group:-...` recipient, but https://github.com/openclaw/openclaw/pull/85656 is now the active closing PR with focused tests and live Telegram proof, so the issue should wait for that PR to land or be closed.

## What This Changes

Telegram durable retry can replay group recipients as `group:-...`, causing Telegram numeric chat ID validation to fail until the delivery is moved to the failed queue.

## Best Possible Solution

Review and land the Telegram-owned outbound-boundary normalization in https://github.com/openclaw/openclaw/pull/85656, or an equivalent scoped fix, then close this issue after merge.

## Reproduction Assessment

Yes, at source level. A queued Telegram entry with `to: "group:-100..."` is replayed unchanged by durable recovery and then rejected by Telegram send target resolution.

## Solution Assessment

Yes. The maintainable direction is to normalize the legacy bare `group:<numeric>` form at the Telegram outbound/retry boundary while keeping the generic delivery queue plugin-agnostic.
