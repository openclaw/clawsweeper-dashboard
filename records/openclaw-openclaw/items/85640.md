## Label Justifications

- `P1`: The issue reports a broken Telegram retry path that can permanently fail real group-chat reminders after transient send failure.
- `impact:message-loss`: Failed retries are moved to `delivery-queue/failed/`, leaving Telegram group messages undelivered until manually discovered.

## Summary

Keep open. Current main fixed the adjacent sessions_send announce target path in https://github.com/openclaw/openclaw/pull/81229, but durable delivery recovery still replays the stored Telegram `to` value unchanged, and current Telegram target normalization still rejects a standalone `group:-...` recipient.

## What This Changes

Fix Telegram durable-delivery retries so queued group sends replay with a numeric chat ID instead of standalone `group:<id>`.

## Best Possible Solution

Preserve or normalize the canonical numeric Telegram chat ID before durable queue replay, preferably at the Telegram-owned outbound boundary, with coverage for both new sends and legacy queued entries.

## Reproduction Assessment

Yes at source level, but not by running a live Telegram retry. A queued Telegram entry with `to: "group:-100..."` is replayed unchanged by recovery and then rejected by the Telegram send target resolver.

## Solution Assessment

The suggested direction is right, but the safest implementation should avoid broad generic parser churn. Normalize the malformed legacy Telegram queue target at the Telegram outbound/retry boundary or carry the original numeric target through durable storage.
