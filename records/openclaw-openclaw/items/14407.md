Keep open: current main and the latest release still only partially cover sensitive-field reveal controls, the related implementation PRs are closed unmerged, and the requested persisted visibility behavior needs product/security review.

## What This Changes

The issue requests a password or secret visibility toggle across web, desktop, iOS, and Android credential-entry surfaces.

## Best Possible Solution

Adopt a reusable, default-masked sensitive-input pattern per UI stack and keep reveal state ephemeral unless maintainers explicitly approve persisted visibility behavior.

## Reproduction Assessment

Yes for the source-level gap: current main has named credential-entry fields without reveal controls, and Android has sensitive fields rendered through a plain text component. This is not a failing-bug reproduction; it is a high-confidence feature gap from source inspection.

## Solution Assessment

No as proposed: reveal controls are a reasonable direction, but localStorage-persisted clear-text visibility is not the best default for sensitive fields. The safer path is a reusable, ephemeral reveal control with maintainers deciding any persistence policy explicitly.
