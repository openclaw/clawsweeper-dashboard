## Summary

Keep open. Current main still has only partial reveal support in the web Control UI, while the Skills API-key editor and native credential-entry surfaces still lack a consistent show/hide pattern; the linked implementation attempt #7124 is closed and unmerged in the provided context.

## What This Changes

Add show/hide controls for password, token, and API-key fields across web, desktop, iOS, and Android credential-entry surfaces.

## Best Possible Solution

Add reusable sensitive-input reveal components per UI stack, apply them to the remaining web/native credential fields, and keep visibility state local and non-persistent by default.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection on current main gives a high-confidence feature-gap check: Overview/Login and generic config have reveal support, while Skills and native credential fields still do not.

## Solution Assessment

Yes for the reveal-toggle direction, but no for the proposed persistent preference without maintainer approval. The narrow maintainable path is reusable per-stack sensitive-input UI with ephemeral visibility state and focused coverage for each touched surface.
