## Summary

Keep open. Current main and latest release still only have partial web reveal support; Skills API-key fields and native credential-entry surfaces remain without a consistent reveal control, so there is no implemented-on-main or duplicate close path.

## What This Changes

The issue requests masked-by-default password/token visibility toggles across web, desktop, iOS, and Android credential fields.

## Best Possible Solution

Define reusable sensitive-input reveal components per UI stack, apply them to the remaining credential fields, and keep reveal state session-local unless maintainers explicitly approve persistence.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The remaining gap is high-confidence by source inspection because current main still has concrete Skills, macOS, iOS, and Android credential fields without reveal controls.

## Solution Assessment

No, the current partial web implementation is not sufficient. The maintainable solution is a shared sensitive-input pattern per UI stack rather than another isolated single-field toggle.
