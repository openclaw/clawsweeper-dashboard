## Summary

Keep open. The PR fixes part of the configured redaction path, but it still leaves shared config-derived redaction using replacement semantics for log tailing and persisted transcript text, which is the security-sensitive part of the contract.

## What This Changes

The PR adds a default-plus-custom redaction helper, uses it for no-options text redaction and tool-detail display, and adds tests for combined defaults/custom patterns plus explicit caller-supplied replacement semantics.

## Best Possible Solution

Centralize config-derived default-plus-custom merging in the shared redaction resolver or a clearly named config resolver, route log tailing, file/console logs, persisted transcript text, tool-detail display, and safety-boundary payloads through that contract, and document explicit replacement semantics only for deliberate internal call sites with tests.

## Reproduction Assessment

Yes. Configure `logging.redactPatterns` with only an email regex and inspect the current-main log-tail or persisted-transcript paths: they compile only the configured pattern list, so default token/API-key/PEM patterns are absent.

## Solution Assessment

No. The PR is directionally useful, but the narrow maintainable fix is to move additive config behavior into the shared resolver used by all config-derived sinks, while preserving explicit replacement only where intentionally documented and covered.
