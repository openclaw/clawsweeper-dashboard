## Summary

This PR should stay open because current `main` does not contain its exact additive config-resolution changes, but the proposed patch is still incomplete: mode-only redaction callers can continue skipping configured custom patterns, and the external contributor has not provided real after-fix behavior proof.

## What This Changes

The PR changes redaction resolution so configured `logging.redactPatterns` can be merged with built-in defaults, updates tool detail/payload redaction paths, adds focused redaction tests, and hardens one doctor install fixture `git add` command.

## Best Possible Solution

Centralize config-derived redaction resolution so no-options, tool-detail, tool-payload, transcript, and mode-only safety callers all apply configured custom patterns plus defaults, while explicit `patterns` arrays continue to mean exact caller-supplied replacement.

## Reproduction Assessment

Yes, by source inspection rather than execution. A mode-only caller such as exec approval display passes `{ mode: "tools" }`, and the PR branch still routes that through explicit options without reading configured custom patterns.

## Solution Assessment

No. The merge helper is the right direction, but branching on the mere presence of an options object leaves mode-only safety callers incomplete; the safer fix is to branch only on explicit `patterns` or otherwise use one config-derived merge path for mode-only calls.
