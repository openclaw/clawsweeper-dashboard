## Summary

Keep open. The latest PR direction is useful, but it still leaves a config-derived transcript path where custom redaction patterns can disable the built-in token/API-key/PEM masking before session JSONL is persisted.

## What This Changes

The PR adds a default-plus-custom redaction helper, updates logging/tool redaction paths and tests, and narrows one doctor E2E fixture `git add` command.

## Best Possible Solution

Centralize config-derived redaction resolution so every sink that consumes `logging.redactPatterns` gets built-in defaults plus custom patterns, while keeping deliberate replacement semantics behind a clearly named internal path with focused transcript coverage.

## Reproduction Assessment

Yes. Configure `logging.redactSensitive: "tools"` with only a custom email regex, then persist transcript text containing that email and a default-matched token such as `OPENAI_API_KEY=sk-1234567890abcdef`; the PR still passes transcript config patterns as explicit options, so the token path skips defaults.

## Solution Assessment

No. The PR fixes some paths, but the maintainable solution is to route the transcript config path through the additive resolver/helper and add a custom-plus-default transcript regression test.

## Review Findings
