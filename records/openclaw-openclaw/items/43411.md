## Summary

Keep open. The Greptile test-quality concern was addressed, and the latest diff partially fixes additive redaction for no-options and tool-payload paths, but the patch still leaves config-derived transcript redaction able to replace built-in defaults with custom patterns. That is a security-sensitive shared logging contract, so this PR should not merge as-is.

## What This Changes

The PR adds a default-plus-custom redaction helper, routes no-options redaction and tool-detail/tool-payload display through it, and adds tests for combined default/custom patterns plus explicit replacement semantics.

## Best Possible Solution

Centralize config-derived redaction resolution so every sink that consumes `logging.redactPatterns` gets built-in defaults plus custom patterns, while keeping deliberate internal replacement semantics behind an explicit, clearly named API with focused tests.

## Reproduction Assessment

Yes. Configure `logging.redactPatterns` with only an email regex and persist transcript text containing both that email and a default-matched API key/token; the current transcript wrapper passes those config patterns as explicit options, so the PR still compiles only the custom pattern for that sink.

## Solution Assessment

No. The direction is useful, but the maintainable fix is to route config-derived sinks through one additive resolver/helper instead of making only no-options calls additive and leaving config passed as explicit options replacement-only.
