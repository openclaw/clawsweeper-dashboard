## Summary

Keep open. The PR targets a real redaction bug that current main still does not fully solve, but the patch remains incomplete because mode-only redaction callers still skip configured custom patterns, and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR changes logging redaction resolution so user redactPatterns are merged with built-in defaults, adds focused redaction tests, and tweaks one doctor install fixture command.

## Best Possible Solution

Centralize config-derived redaction pattern resolution so no-options, tool-detail, tool-payload, transcript, and mode-only safety callers get custom patterns plus defaults, while callers that pass an explicit patterns array keep replacement semantics.

## Reproduction Assessment

Yes, by source inspection. Configure logging.redactPatterns with an email regex, then send text containing that email plus an sk- token through a mode-only caller such as redactSensitiveText(value, { mode: "tools" }); the PR head still takes the explicit-options path and skips configured custom patterns.

## Solution Assessment

No. The shared additive resolver is the right direction, but this patch still keys off the presence of any options object instead of only explicit patterns, so documented safety and diagnostic surfaces remain incomplete.
