## Summary

Keep open: the PR targets a real redaction bug, but the latest head still skips configured `logging.redactPatterns` for mode-only callers and lacks required after-fix real behavior proof.

## What This Changes

The PR changes logging redaction option resolution to merge configured patterns with built-in defaults, updates tool-detail/payload redaction paths, adds focused redaction tests, and changes one doctor install fixture to `git add -- .`.

## Best Possible Solution

Centralize redaction option resolution so omitted `patterns` use configured custom patterns plus defaults across generic, tool-detail, and mode-only safety callers while explicitly supplied patterns remain exact.

## Reproduction Assessment

Yes. Source inspection shows the PR head treats `{ mode: "tools" }` as an explicit-options path, while current main has exec approval and diagnostic callers using that mode-only form; no live command was run because this review is read-only.

## Solution Assessment

No. The merge helper is the right direction, but the PR should branch on an explicit `patterns` field or use a shared config-derived resolver instead of branching on the presence of any options object.
