## Summary

Keep open: the redaction bug is real and the PR is pointed at the right area, but the current head still skips configured custom patterns for mode-only safety callers and has no contributor real-behavior proof.

## What This Changes

The PR adds a helper to merge configured redaction patterns with defaults, updates logging redaction paths and tests, and changes one doctor install fixture from `git add -A` to `git add -- .`.

## Best Possible Solution

Centralize redaction option resolution so omitted `patterns` use configured custom patterns plus defaults across generic, tool-detail, and mode-only safety callers, while explicitly supplied patterns remain exact.

## Reproduction Assessment

Yes by source inspection: the PR head treats `{ mode: "tools" }` as explicit options and never reads configured `logging.redactPatterns`, while current main has exec approval and diagnostics callers using that mode-only form. No live command was run because this is a read-only review.

## Solution Assessment

No. The merge helper is the right direction, but the PR should branch on whether `patterns` was explicitly supplied or use a shared config-derived resolver instead of branching on the presence of any options object.
