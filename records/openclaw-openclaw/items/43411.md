## Summary

Keep open. The branch fixes some logging redaction paths, but it still leaves a config-derived transcript path where custom patterns can replace built-in token/API-key/PEM masking, and the external PR still lacks real after-fix behavior proof.

## What This Changes

The PR adds default-plus-custom redaction merging for selected logging helpers, new redaction tests, and changes one doctor install fixture git command.

## Best Possible Solution

Land a rebased fix that centralizes additive default-plus-custom resolution for every config-derived logging, transcript, and tool-payload sink while preserving explicit low-level replacement semantics with regression coverage.

## Reproduction Assessment

Yes, by source inspection. Configure `logging.redactSensitive: "tools"` with only a custom email regex, then persist transcript text containing both that email and a default-matched API key; the transcript wrapper still passes config patterns as explicit replacements.

## Solution Assessment

No. The PR fixes some no-options and tool-payload paths, but the maintainable fix must also route persisted transcript config redaction through an additive resolver and add custom-plus-default transcript coverage.
