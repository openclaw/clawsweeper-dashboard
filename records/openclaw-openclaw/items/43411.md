Keep open: current main only covers the linked transcript/tool-payload parts of this redaction problem, while the PR still carries useful logging/tool-detail fixes but needs a rebase, real behavior proof, and one remaining config-resolution correction before merge.

## What This Changes

The branch adds default-plus-custom redaction resolution in `src/logging/redact.ts`, adds focused redaction tests, and changes one doctor install fixture from `git add -A` to `git add -- .`.

## Best Possible Solution

Centralize additive config-derived pattern resolution for logging, tool-detail, and documented mode-only safety sinks while preserving explicit replacement only when a caller supplies a `patterns` array.

## Reproduction Assessment

Yes, by source inspection. Configure `logging.redactSensitive: "tools"` with a custom email regex and route text containing that email plus an `sk-...` token through current-main no-options `redactSensitiveText()` or `redactToolDetail()`; current main resolves only the configured pattern in those paths.

## Solution Assessment

No, not yet. The shared additive resolver is the right shape, but the PR still treats mode-only explicit calls as fully explicit, so documented safety surfaces that call `redactSensitiveText(value, { mode: "tools" })` continue to skip custom config patterns.
