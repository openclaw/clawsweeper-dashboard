## Summary

Keep open. The PR is a plausible fix for part of the logging redaction bug, but it still leaves a config-derived transcript path where custom `logging.redactPatterns` can replace the built-in secret patterns, and it has no after-fix real behavior proof.

## What This Changes

The PR adds additive default-plus-custom redaction handling for some logging paths, adds focused redaction tests, and changes one doctor install fixture from `git add -A` to `git add -- .`.

## Best Possible Solution

Land a rebased fix that centralizes additive default-plus-custom resolution for all config-derived logging, transcript, and tool-payload sinks while preserving explicit low-level replacement semantics with regression coverage.

## Reproduction Assessment

Yes by source inspection. Configure `logging.redactSensitive: "tools"` with only a custom email regex, then persist transcript text containing both that email and a default-matched API key; the transcript path still passes config patterns as explicit replacements.

## Solution Assessment

No. The PR fixes some no-options/tool paths, but the maintainable fix must also route persisted transcript config redaction through the additive resolver and test custom-plus-default transcript persistence.
