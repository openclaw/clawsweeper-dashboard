## Summary

Keep open. The PR improves part of the redaction regression, but the proposed resolver split still leaves persisted transcript redaction using explicit config patterns, so custom redaction can disable built-in token/API-key/PEM masking before session JSONL writes.

## What This Changes

The PR adds a default-plus-custom redaction helper, routes no-options redaction and tool-detail/tool-payload display through it, adds tests for combined/default replacement semantics, and narrows a doctor E2E fixture `git add` command.

## Best Possible Solution

Centralize config-derived redaction resolution so every sink that consumes `logging.redactPatterns` gets built-in defaults plus custom patterns, while keeping deliberate replacement semantics behind a clearly named internal API with focused transcript coverage.

## Reproduction Assessment

Yes. Configure `logging.redactPatterns` with only an email regex and persist transcript text containing both that email and a default-matched API key/token; under the PR, the transcript wrapper still passes those config patterns as explicit options, so the built-in token pattern is skipped.

## Solution Assessment

No. The direction is useful, but the maintainable fix is to route config-derived transcript redaction through the additive resolver/helper instead of making only no-options calls and selected tool payloads additive.
