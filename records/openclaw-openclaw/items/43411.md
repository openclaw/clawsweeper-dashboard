## Summary

Keep open: this PR fixes some logging redaction paths, but it still leaves config-derived persisted transcript redaction on the explicit-patterns path, so a custom `logging.redactPatterns` entry can still disable built-in token/API-key/PEM masking for session JSONL.

## What This Changes

The branch adds a default-plus-custom redaction helper, updates logging/tool redaction paths and tests, and narrows one doctor E2E fixture `git add` command.

## Best Possible Solution

Centralize additive config-pattern resolution for every `logging.redactPatterns` sink, including persisted transcripts, while preserving deliberate explicit replacement semantics only behind a clearly tested internal call path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure `logging.redactSensitive: "tools"` with only a custom email regex, then persist transcript text containing that email and a default-matched token. The transcript path still passes config patterns as explicit options, and the PR's resolver keeps explicit patterns from merging defaults.

## Solution Assessment

No. The PR is directionally right for no-options and tool-payload calls, but the narrow maintainable fix must also route persisted transcript config redaction through the additive default-plus-custom resolver and cover that path with a regression test.
