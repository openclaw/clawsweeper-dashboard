## Summary

Keep open: the PR targets a real redaction bug, but current main still has the general logging/tool-detail additive-pattern gap and the PR head still misses the mode-only config-resolution path while lacking real behavior proof.

## What This Changes

The PR changes logging redaction resolution so configured patterns are additive with defaults, adds focused redaction tests, and tweaks one doctor install fixture command.

## Best Possible Solution

Centralize config-derived redaction pattern resolution so no-options, tool-detail, and mode-only safety callers get custom patterns plus defaults, while callers that pass a patterns array keep explicit replacement semantics.

## Reproduction Assessment

Yes, by source inspection. Configure logging.redactSensitive to "tools" with a custom email regex, then route text containing that email plus an sk- token through current-main no-options redactSensitiveText(), redactToolDetail(), or a mode-only safety caller; current main and the PR head each skip part of the intended custom-plus-default behavior.

## Solution Assessment

No. The shared additive resolver is the right direction, but the PR still treats any options object as fully explicit, so documented mode-only redaction surfaces continue to skip configured custom patterns.
