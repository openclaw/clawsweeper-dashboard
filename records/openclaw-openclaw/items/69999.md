## Summary

Keep open. Current main still lacks the wallboard surface, and this PR is the visible REL-433 implementation candidate, but the branch has a blocking packaged-app resource lookup issue and a smaller re-show reload bug before it should merge.

## What This Changes

The PR adds a macOS `--wallboard` launch path, a display resolver, a singleton wallboard manager, a borderless `WKWebView` window controller, a bundled placeholder resource, package resource wiring, and smoke tests.

## Best Possible Solution

Keep the wallboard shell work in the macOS app if maintainers want REL-433, but first use a main-bundle-first resource resolver for the placeholder, avoid reloading the placeholder when re-presenting an existing controller, have a macOS owner review the WebKit/private-KVC choices, and validate with Swift build/test plus a packaged-app launch on macOS.

## Reproduction Assessment

Not applicable for the feature request itself; source inspection verifies current main lacks the proposed `--wallboard` surface and the PR adds it. For the findings, yes: the PR diff directly shows the unsafe `Bundle.module` lookup and the repeated `loadBundledPlaceholder()` call, supported by the existing `DeviceModelCatalog` packaged-resource contract.

## Solution Assessment

No, not as-is. The feature split is plausible, but the placeholder lookup should follow the existing packaged-app bundle-resolution pattern and the manager should avoid reloading cached content before this is the narrowest maintainable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Resolve the placeholder from the app bundle first:** `apps/macos/Sources/OpenClaw/Wallboard/WallboardWindowController.swift:58-61`
  - body: `Bundle.module` can trap in packaged `.app` builds before this guard runs when SwiftPM cannot find its generated resource bundle. Current macOS resource code avoids that by checking `Bundle.main` first and only then falling back to `Bundle.module`; the wallboard placeholder should use the same pattern or `--wallboard` can crash in packaged builds.
  - confidence: 0.9
- **[P2] Skip placeholder reloads when re-showing the wallboard:** `apps/macos/Sources/OpenClaw/Wallboard/WallboardManager.swift:13-15`
  - body: When a controller already exists, `open()` presents it and then starts a fresh `WKWebView` navigation. Re-showing the cached wallboard should preserve the already loaded placeholder; otherwise users can see a blank/black flash on every reopen.
  - confidence: 0.82
