## Summary

Keep open: current main and v2026.6.8 still reject `models.providers.<provider>.modelIdNormalization`, and the open fix candidate at https://github.com/openclaw/openclaw/pull/94590 has not merged. The remaining decision is product/config ownership, not conservative cleanup.

## What This Changes

The issue requests a new provider-config `modelIdNormalization` option so native providers routed through Cloudflare AI Gateway REST or Unified endpoints can send provider-prefixed `author/model` wire IDs.

## Best Possible Solution

Keep this issue as the canonical narrow config-surface tracker and resolve it by reviewing or repairing https://github.com/openclaw/openclaw/pull/94590 after maintainers confirm the provider-config versus Cloudflare-plugin ownership boundary.

## Reproduction Assessment

Yes, at source level: current main has a strict provider config schema with no `modelIdNormalization` field, and normalization policies are collected from plugin manifests rather than `models.providers.<provider>` config. I did not run a live Cloudflare request in this read-only review.

## Solution Assessment

Unclear. The proposed config knob is plausible, but maintainers still need to decide whether endpoint-specific model ID normalization belongs in generic provider config or in a provider-owned Cloudflare REST bridge.
