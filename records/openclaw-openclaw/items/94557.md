## Summary

Keep open: current main and v2026.6.8 still only support model-id normalization from plugin manifests, while provider config is strict and has no `modelIdNormalization` field. The request is valid, but it adds config/provider-routing surface and overlaps the broader Cloudflare AI Gateway REST/bridge direction, so it needs maintainer product review before implementation.

## What This Changes

Allow provider-level `modelIdNormalization` in `models.providers.<provider>` so native providers pointed at Cloudflare AI Gateway REST or Unified endpoints can prefix bare model ids as `author/model`.

## Best Possible Solution

Maintainers should choose whether endpoint-specific model-id normalization is a supported provider-config contract or whether the Cloudflare plugin should own the REST/native-provider bridge, then implement the chosen path with schema/types, docs, focused normalization tests, and live Cloudflare proof.

## Reproduction Assessment

Yes, at source level: the current provider schema rejects unknown provider config keys and the normalization collector only reads plugin manifests. I did not run a live Cloudflare request because this was a read-only cleanup review.

## Solution Assessment

Unclear. The proposed config knob is plausible and narrow, but the best permanent fix may be provider-owned Cloudflare REST/bridge support rather than a new generic config surface.
