## Summary

Keep open: current main and v2026.6.9 still collect model-id normalization only from plugin manifests, while `models.providers.<provider>` remains a strict config surface with no `modelIdNormalization` field. The request is source-reproducible, but it adds provider config contract surface and overlaps broader Cloudflare AI Gateway REST/bridge decisions.

## What This Changes

The issue requests an optional `modelIdNormalization.prefixWhenBare` setting under provider config so native providers can send Cloudflare AI Gateway's required provider-prefixed wire model IDs.

## Best Possible Solution

Maintainers should first decide whether endpoint-specific model-id normalization is a supported generic provider-config contract or belongs in the Cloudflare plugin bridge, then implement schema, types, docs, catalog propagation, focused tests, and live Cloudflare proof for the chosen path.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.9 have a strict provider config schema with no `modelIdNormalization` key, while normalization policies are still collected from plugin manifests only. I did not run a credentialed Cloudflare request in this read-only review.

## Solution Assessment

Unclear. The proposed config knob is plausible, but the best permanent fix may be a Cloudflare provider-owned REST/bridge contract rather than a new generic provider config surface.
