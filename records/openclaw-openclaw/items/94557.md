## Summary

Keep open: current main and v2026.6.8 still only support model-id normalization from plugin manifests, while provider config remains strict and has no `modelIdNormalization` field. The request is source-reproducible, but adding a provider-config normalization contract overlaps broader Cloudflare AI Gateway REST/bridge work and needs maintainer product review first.

## What This Changes

The issue asks to allow `models.providers.<provider>.modelIdNormalization.prefixWhenBare` so native providers can send Cloudflare AI Gateway `author/model` IDs on REST or Unified endpoints.

## Best Possible Solution

Maintainers should choose whether endpoint-specific model-id normalization is a supported provider-config contract or whether the Cloudflare plugin should own the REST/native-provider bridge, then implement the chosen path with schema/types, docs, focused normalization tests, and live Cloudflare proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 have a strict provider config schema with no `modelIdNormalization` field, while normalization policies are collected only from plugin manifests. I did not run a live Cloudflare request in this read-only cleanup review.

## Solution Assessment

Unclear. The proposed config knob is plausible and narrow, but the best permanent fix may be provider-owned Cloudflare REST/bridge support rather than a new generic config surface.
