## Summary

Keep open: current main still rejects `modelIdNormalization` under `models.providers.<provider>`, while normalization policies are still sourced from manifests only. The issue is paired with an open implementation candidate, so the right cleanup action is to keep the issue open until that PR is either landed or closed by maintainers.

## What This Changes

Request to allow provider-configured `modelIdNormalization.prefixWhenBare` so native providers can be routed through Cloudflare AI Gateway REST or Unified API endpoints that require provider-prefixed wire model ids.

## Best Possible Solution

Review the open implementation candidate together with the broader Cloudflare AI Gateway REST/bridge issues, and land a compatibility-safe provider-config contract only if maintainers accept that product surface.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: provider config is strict and cannot accept `modelIdNormalization`, while normalization policies are only collected from manifests. I did not run a live Cloudflare request in this read-only review.

## Solution Assessment

Unclear as a final product direction: the proposed config knob is narrow and fits the existing normalization mechanism, but it adds provider config surface and overlaps open Cloudflare AI Gateway bridge/default-endpoint work.
