## Summary

Keep open: the PR targets a valid stale skills-snapshot bug, but the patch is not mergeable as-is because the new signature misses live eligibility inputs, only covers the reply-session cache, and hashes secret-bearing skills config into persisted metadata.

## What This Changes

The PR adds an `eligibilitySignature` field to session skill snapshots and refreshes reply-session snapshots when a hash of skills config, skill filter, PATH, and remote eligibility changes.

## Best Possible Solution

Implement a shared, secret-safe eligibility freshness mechanism used by reply sessions, CLI agent sessions, and isolated cron snapshots, keyed from sanitized prerequisite outcomes or non-secret boolean state rather than raw config.

## Reproduction Assessment

Yes, by source inspection. Cache a snapshot that excludes a skill gated by a binary, env var, or `requires.config`, then satisfy that prerequisite without changing SKILL.md or the PATH string; the PR's signature can remain unchanged while the existing snapshot is reused.

## Solution Assessment

No. The PR is directionally useful, and the earlier first-turn feedback appears addressed, but the maintainable fix needs one shared, secret-safe eligibility freshness contract across every persisted snapshot cache.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Include live eligibility outcomes in the signature:** `src/auto-reply/reply/session-updates.ts:48-52`
  - body: The signature only hashes `cfg.skills`, the normalized filter, PATH text, and remote eligibility, but skill inclusion is decided by live binary probes, process env/configured env/apiKey, and `requires.config` checks. Installing a binary in an existing PATH directory or toggling an env/config prerequisite can leave this signature unchanged and keep reusing a stale snapshot.
  - confidence: 0.94
- **[P2] Invalidate every persisted skill snapshot cache:** `src/auto-reply/reply/session-updates.ts:173-181`
  - body: This adds eligibility invalidation only to reply-session `ensureSkillSnapshot`. CLI agent sessions and isolated cron snapshots still reuse persisted snapshots from version/filter checks, so the same eligibility changes continue serving stale skill lists on those surfaces.
  - confidence: 0.91
- **[P2] Avoid deriving session metadata from skill secrets:** `src/auto-reply/reply/session-updates.ts:48-50`
  - body: `cfg.skills` can include `skills.entries.*.env` and `apiKey` values. Persisting a digest of that object as session metadata creates unnecessary secret-derived state; use sanitized eligibility outcomes or non-secret prerequisite booleans instead.
  - confidence: 0.86
