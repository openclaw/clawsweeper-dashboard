## Summary

Keep open: the underlying stale skills-snapshot bug is real, but this PR is not merge-ready. The current head addresses the earlier first-turn reuse feedback, yet the proposed signature still misses live eligibility outcomes, only covers the reply-session cache, and persists secret-derived metadata.

## What This Changes

The PR adds an eligibilitySignature field to persisted session skill snapshots and uses a hash of skills config, skill filter, PATH, and remote eligibility to refresh reply-session snapshots when that signature changes.

## Best Possible Solution

Implement a shared, secret-safe eligibility freshness mechanism used by reply sessions, CLI agent sessions, and isolated cron snapshots. The cache key should be based on sanitized eligibility outcomes or boolean prerequisite state for bins, env/API-key availability, config paths, remote eligibility, version, and filter, with focused regression tests for eligibility changes that do not touch SKILL.md.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to cache a snapshot excluding a skill gated by a binary, env var, or requires.config path, then satisfy that gate without changing SKILL.md or the PATH string; current main and this PR can both reuse the stale snapshot because the refresh predicate does not observe that actual eligibility outcome.

## Solution Assessment

No. The PR is a useful direction, and the earlier first-turn stale-snapshot feedback appears addressed, but this is not the best fix because it omits live eligibility outcomes, only updates one persisted cache, and derives persisted metadata from secret-bearing skills config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Include live eligibility outcomes in the signature:** `src/auto-reply/reply/session-updates.ts:48-52`
  - body: The new signature only hashes cfg.skills, the normalized filter, PATH text, and remote eligibility, but skill inclusion also depends on live binary probes, process env/API-key availability, and requires.config paths over the broader config. Installing a binary in an existing PATH directory or toggling an env/config prerequisite can leave the signature unchanged and keep reusing a stale snapshot.
  - confidence: 0.94
- **[P2] Invalidate every persisted skill snapshot cache:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: This patch adds the eligibility check only to ensureSkillSnapshot. The CLI agent and isolated cron snapshot paths still reuse persisted snapshots using missing/version/filter checks, so those surfaces can keep stale skill lists after the same eligibility changes this PR is meant to fix.
  - confidence: 0.91
- **[P2] Avoid deriving session metadata from skill secrets:** `src/auto-reply/reply/session-updates.ts:48-50`
  - body: cfg.skills can include skills.entries.*.env and apiKey values. Persisting a digest derived from that object as session metadata creates unnecessary secret-derived state; use sanitized eligibility outcomes or non-secret booleans instead.
  - confidence: 0.86
