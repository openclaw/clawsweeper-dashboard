## Summary

Keep open: this PR targets a real stale skillsSnapshot bug, but the current patch is still reply-session-only, misses live eligibility outcomes, persists secret-derived metadata, and lacks required real behavior proof.

## What This Changes

This PR adds an eligibilitySignature field to persisted session skill snapshots and uses a hash of skills config, skill filter, PATH, and remote eligibility to refresh reply-session snapshots.

## Best Possible Solution

Implement one shared, secret-safe snapshot freshness mechanism for reply sessions, CLI agent sessions, and isolated cron based on sanitized evaluated eligibility outcomes or explicit version bumps, while preserving current snapshot hydration behavior.

## Reproduction Assessment

Yes, by source inspection. Cache a snapshot that excludes a skill gated by a binary, env var, or requires.config, then satisfy that prerequisite without changing SKILL.md or the PR's hash inputs; the stale snapshot can still be reused on affected paths.

## Solution Assessment

No. The PR is a useful direction, but the maintainable fix needs a shared, secret-safe freshness contract across all persisted snapshot caches rather than a reply-session-only hash of raw skills config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Include live eligibility outcomes in the cache key:** `src/auto-reply/reply/session-updates.ts:48-52`
  - body: The signature hashes config, filter, PATH text, and remote eligibility, but inclusion also depends on evaluated binary probes, env/configured env/apiKey checks, and requires.config. Those prerequisites can change while this digest stays the same, so a stale snapshot can still be reused.
  - confidence: 0.94
- **[P2] Apply eligibility invalidation to every snapshot cache:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: This invalidation only runs in reply-session ensureSkillSnapshot. CLI agent sessions and isolated cron still reuse persisted snapshots from version/filter checks, so the same eligibility changes can keep serving stale skill lists on those surfaces.
  - confidence: 0.91
- **[P2] Avoid hashing skill secrets into session metadata:** `src/auto-reply/reply/session-updates.ts:48-50`
  - body: cfg.skills can include SkillConfig apiKey and env values. Persisting a digest derived from that object as session metadata creates unnecessary secret-derived state; use sanitized eligibility outcomes or non-secret prerequisite booleans instead.
  - confidence: 0.86
