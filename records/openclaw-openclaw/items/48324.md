## Summary

Keep this PR open for maintainer handling: it targets a real stale skill-snapshot bug, but the branch is stale/conflicting, lacks real behavior proof, and its freshness key is incomplete and security-sensitive.

## What This Changes

The PR adds an `eligibilitySignature` to persisted skill snapshots and refreshes auto-reply session snapshots when watcher version, skill filter, or that signature changes.

## Best Possible Solution

Rebase or replace this with a shared redacted/evaluated snapshot freshness contract that covers auto-reply, agent-command, and cron reuse paths while preserving the merged watcher-root ordering.

## Reproduction Assessment

Yes, source-level: current and PR code show snapshot reuse can depend on version/filter/signature while actual skill eligibility depends on env, binary filesystem state, remote, and full config inputs. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. The PR's narrow persisted eligibilitySignature is not the best fix because it misses eligibility inputs, is not shared across all snapshot reuse paths, and stores a raw-config-derived verifier; the safer path is a shared redacted/evaluated freshness contract or a narrower canonical PR.
