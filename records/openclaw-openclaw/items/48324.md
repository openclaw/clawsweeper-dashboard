## Summary

Keep open. The PR targets a real stale skillsSnapshot bug, but the branch is still incomplete: it keys only one snapshot path, misses several eligibility inputs, persists a digest derived from secret-adjacent skill config, and lacks after-fix real behavior proof.

## What This Changes

The PR adds an eligibilitySignature field to session skill snapshots and uses it in auto-reply session snapshot refresh decisions.

## Best Possible Solution

Centralize a secret-safe snapshot freshness contract in the skills/session layer and apply it consistently to reply sessions, CLI agent sessions, and isolated cron snapshot reuse.

## Reproduction Assessment

Yes, by source inspection. Current main can reuse persisted prompt/skills snapshots when version and skillFilter match even though skill eligibility also depends on env values, binary presence, and config gates.

## Solution Assessment

No. The PR points at the right bug class, but hashing partial raw inputs in only the auto-reply path is not the narrow maintainable solution; a shared sanitized freshness key or explicit invalidation contract is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Include actual runtime prerequisites in the freshness key:** `src/auto-reply/reply/session-updates.ts:48-52`
  - body: The new signature hashes cfg.skills, skillFilter, PATH text, and remote eligibility, but skill inclusion also depends on process env values, broader requires.config paths, and whether required binaries currently exist. Installing a binary into an existing PATH directory or toggling a non-skills config gate can keep this signature unchanged and continue reusing a stale snapshot.
  - confidence: 0.92
- **[P2] Apply freshness checks to every snapshot cache:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: This invalidation check only runs in ensureSkillSnapshot. The CLI agent and isolated cron paths still reuse cached snapshots when version and skillFilter match, so those sessions can keep stale skill lists even after this patch lands.
  - confidence: 0.9
- **[P2] Avoid persisting secret-derived snapshot metadata:** `src/auto-reply/reply/session-updates.ts:48-50`
  - body: The persisted eligibilitySignature is derived from raw cfg.skills, and SkillConfig can contain apiKey and env values. Even as a digest, session metadata should be based on sanitized eligibility state or prerequisite booleans rather than raw secret-adjacent config.
  - confidence: 0.86
