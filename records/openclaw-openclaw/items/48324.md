## Summary

Keep this PR open: current main still lacks eligibility-based invalidation, but the proposed patch is not ready because its signature misses several runtime eligibility inputs and only updates the reply-session cache path. The discussion shows the author addressed the earlier first-turn stale-snapshot review feedback, but later review feedback about actual eligibility coverage, shared cache coverage, and secret-safe metadata remains valid.

## What This Changes

The PR adds an eligibility signature to persisted session skill snapshots and uses it in the reply-session snapshot refresh decision.

## Best Possible Solution

Implement one shared, secret-safe eligibility freshness mechanism for persisted skill snapshots and use it consistently from reply sessions, openclaw agent sessions, and isolated cron snapshots. The cache key should reflect the actual eligibility outcomes or sanitized prerequisite state for bins, env/API-key availability, config paths, and remote eligibility, and regression tests should cover eligibility changes that happen without a SKILL.md watcher bump.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to cache a snapshot excluding a skill gated by a binary, env var, or requires.config path, then satisfy that gate without changing SKILL.md or the PATH string; current main and this PR can both keep reusing the stale snapshot because the refresh predicate does not observe that eligibility outcome.

## Solution Assessment

No. The PR is a useful direction, and the earlier first-turn stale-snapshot feedback appears addressed in the current head, but this is not the best complete fix because the signature omits live eligibility outcomes, persists a digest of secret-bearing config, and is not shared across all persisted snapshot caches.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Include actual runtime eligibility in the signature:** `src/auto-reply/reply/session-updates.ts:48-52`
  - body: The signature only hashes cfg.skills, the normalized filter, PATH text, and remote eligibility. Skill inclusion also depends on live binary probes, process env/API-key availability, and requires.config paths over the broader config, so installing a binary in an existing PATH directory, changing an env var, or toggling a config path can leave the signature unchanged and reuse a stale snapshot.
  - confidence: 0.94
- **[P2] Apply eligibility invalidation to every snapshot cache:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: This refresh check is only added to ensureSkillSnapshot. The CLI agent and isolated cron paths still reuse persisted snapshots using only missing/version/filter checks, so the same eligibility changes remain stale on those surfaces after this patch lands.
  - confidence: 0.91
- **[P2] Avoid hashing secret-bearing skill config into session state:** `src/auto-reply/reply/session-updates.ts:48-50`
  - body: cfg.skills can include skills.entries env values and apiKey settings. Persisting a digest derived from those values in the session snapshot creates unnecessary secret-derived metadata; use sanitized eligibility outcomes or boolean prerequisite state instead.
  - confidence: 0.86
