## Summary

Keep open, but not merge-ready: the stale skills snapshot problem is real, yet this branch lacks real behavior proof, conflicts with newer snapshot work, and its auto-reply-only raw eligibility signature misses inputs that actually decide skill eligibility.

## What This Changes

This PR adds a persisted `eligibilitySignature` to skill snapshots and uses it in auto-reply session snapshot refresh decisions.

## Best Possible Solution

Rebase or replace this with one shared redacted snapshot-freshness contract applied to all persisted snapshot reuse paths, with real runtime proof for config, env, binary/PATH, and remote eligibility changes.

## Reproduction Assessment

Yes, source-reproducible for the PR defects: `shouldIncludeSkill()` depends on env vars, binary filesystem checks, PATHEXT, and full-config `requires.config` values that the new signature does not cover. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. The auto-reply-only raw `cfg.skills`/`PATH` signature is narrower than the actual eligibility contract; the safer solution is a shared redacted freshness contract across persisted snapshot paths.
