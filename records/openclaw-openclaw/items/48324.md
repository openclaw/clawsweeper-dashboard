## Summary

Keep open: the PR targets a real stale `skillsSnapshot` bug, but the current patch still leaves important eligibility changes uncovered, only updates one cache path, and persists a secret-derived digest without contributor real behavior proof.

## What This Changes

The branch adds an `eligibilitySignature` field to session skill snapshots and refreshes reply-session snapshots when the watcher version, skill filter, or signature changes.

## Best Possible Solution

Implement one shared, secret-safe snapshot freshness mechanism for reply sessions, CLI agent sessions, and isolated cron based on sanitized evaluated eligibility outcomes or explicit version bumps.

## Reproduction Assessment

Yes, by source inspection: cache a snapshot that excludes a skill gated by a binary, env var, or `requires.config`, then satisfy that prerequisite without changing `SKILL.md` or the PR's hash inputs. The stale snapshot can still be reused on affected paths.

## Solution Assessment

No. The PR is a useful direction, but the maintainable fix needs a shared, secret-safe freshness contract across all persisted snapshot caches rather than a reply-session-only hash of raw skills config.

## Review Findings

  - body: The signature hashes config, filter, PATH text, and remote eligibility, but inclusion also depends on evaluated binary probes, process env, configured env/apiKey checks, and `requires.config`.   - confidence: 0.94

  - body: This invalidation only runs in reply-session `ensureSkillSnapshot`. CLI agent sessions and isolated cron still reuse persisted snapshots from version/filter checks, so the same eligibility changes can keep serving stale skill lists on those surfaces.
  - confidence: 0.91
- **[P2] Avoid hashing skill secrets into session metadata:** `src/auto-reply/reply/session-updates.ts:48-50`
  - body: `cfg.skills` can include `SkillConfig` `apiKey` and `env` values. Persisting a digest derived from that object as session metadata creates unnecessary secret-derived state; use sanitized eligibility outcomes or non-secret prerequisite booleans instead.
  - confidence: 0.86
