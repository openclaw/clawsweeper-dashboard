## Summary

Keep this PR open, but it should not merge as-is: the bug is real by source inspection, while the branch only adds a reply-session eligibility digest, misses several eligibility inputs/cache paths, persists secret-derived metadata, and still lacks real behavior proof.

## What This Changes

The PR adds an eligibility signature to persisted session skill snapshots and uses it to refresh reply-session snapshots when the watcher version, skill filter, or signature changes.

## Best Possible Solution

Centralize a sanitized snapshot freshness contract in the skills/session layer and apply it consistently to reply sessions, CLI agent sessions, and isolated cron snapshot reuse.

## Reproduction Assessment

Yes, by source inspection. Cache a snapshot that excludes a binary/env/config-gated skill, satisfy that prerequisite without a SKILL.md watcher bump, and the current version/filter-only cache paths can still reuse stale snapshot data.

## Solution Assessment

No. The PR points at the right bug, but hashing partial raw inputs only in the reply-session path is not the narrow maintainable solution; the freshness key should be shared and based on sanitized evaluated eligibility or explicit version bumps.
