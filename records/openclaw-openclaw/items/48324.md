## Summary

Keep this PR open, but it is not ready to merge: the patch is directionally aimed at a real stale skillsSnapshot bug, yet the proposed cache key is too narrow, only covers the reply-session cache, and persists secret-derived metadata.

## What This Changes

This PR adds a persisted `eligibilitySignature` to reply-session skills snapshots and refreshes `ensureSkillSnapshot` when that signature changes.

## Best Possible Solution

Implement one shared, secret-safe snapshot freshness mechanism for reply sessions, CLI agent sessions, and isolated cron based on sanitized eligibility outcomes or non-secret prerequisite state.

## Reproduction Assessment

Yes, by source inspection. Cache a snapshot that excludes a skill gated by a binary, env var, or `requires.config`, then satisfy that prerequisite without changing `SKILL.md` or the PATH string; the PR's signature can remain unchanged while the old snapshot is reused.

## Solution Assessment

No. The PR is a useful direction, but the maintainable fix needs a shared, secret-safe freshness contract across all persisted snapshot caches rather than a reply-session-only hash of raw skills config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Include live eligibility outcomes in the cache key:** `src/auto-reply/reply/session-updates.ts:48-52`
  - body: The signature hashes config, filter, PATH text, and remote eligibility, but inclusion also depends on live binary probes, env/configured env/apiKey checks, and `requires.config`. Installing a binary into an existing PATH directory or toggling broader config/env can leave this digest unchanged and keep reusing a stale snapshot.
  - confidence: 0.94
- **[P2] Apply eligibility invalidation to every snapshot cache:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: This invalidation runs only inside reply-session `ensureSkillSnapshot`. CLI agent sessions and isolated cron still reuse persisted snapshots from version/filter checks, so the same eligibility changes continue serving stale skill lists on those surfaces.
  - confidence: 0.91
- **[P2] Avoid hashing skill secrets into session metadata:** `src/auto-reply/reply/session-updates.ts:48-50`
  - body: `cfg.skills` can include `entries.*.env` and `apiKey` values. Persisting a digest derived from that object as session metadata creates unnecessary secret-derived state; use sanitized eligibility outcomes or non-secret prerequisite booleans instead.
  - confidence: 0.86
