## Summary

Keep open: the PR targets a real stale skill-snapshot bug, but the current branch is still not merge-ready because its freshness key is partial, reply-session-only, and derived from secret-adjacent config values, and it lacks after-fix real behavior proof.

## What This Changes

The PR adds an eligibility signature to persisted session skill snapshots and refreshes reply-session snapshots when the watcher version, skill filter, or signature changes.

## Best Possible Solution

Centralize a sanitized skill-snapshot freshness contract in the skills/session layer and apply it consistently to reply sessions, CLI agent sessions, and isolated cron snapshot reuse.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: persist a snapshot that excludes an env/bin/config-gated skill, change the prerequisite without a SKILL.md watcher bump, and the version/filter-only caches can keep serving stale skill data.

## Solution Assessment

No. The PR points at the right stale-cache bug, but a reply-only hash of partial raw inputs is not the narrow maintainable fix; the freshness key should be shared and sanitized.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Include actual runtime prerequisites in the freshness key:** `src/auto-reply/reply/session-updates.ts:48-52`
  - body: The signature hashes `cfg.skills`, `skillFilter`, `PATH`, and remote eligibility, but skill inclusion also depends on actual env values, `skills.entries.*.env/apiKey`, `requires.config`, and binary availability. Installing a binary into an existing PATH directory or toggling env/config can leave the same signature and keep reusing a stale snapshot.
  - confidence: 0.92
- **[P2] Apply freshness checks to every snapshot cache:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: This invalidation check only runs in `ensureSkillSnapshot`; the CLI agent and isolated cron paths still reuse snapshots when version/filter match. Those session surfaces can keep stale skill lists after the same eligibility changes this PR is meant to fix.
  - confidence: 0.9
- **[P2] Avoid persisting secret-derived snapshot metadata:** `src/auto-reply/reply/session-updates.ts:49`
  - body: `cfg.skills` can contain `SkillConfig.apiKey` and `env`, and the PR stores a digest derived from those raw values on `SessionSkillSnapshot`. Use sanitized evaluated eligibility state or prerequisite booleans instead of persistent metadata derived from secret-adjacent config.
  - confidence: 0.86
