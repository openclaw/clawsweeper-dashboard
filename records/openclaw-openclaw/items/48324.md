## Summary

Keep open: this PR targets a real stale skills snapshot bug, but the patch is not merge-ready because its freshness key misses actual runtime eligibility inputs, applies only to the reply-session cache path, derives persisted metadata from secret-adjacent config, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a persisted eligibility signature to session skill snapshots and rebuilds reply-session snapshots when the watcher version, filter, or signature changes.

## Best Possible Solution

Centralize a sanitized snapshot freshness contract in the skills/session layer and apply it consistently to reply sessions, CLI agent sessions, and isolated cron snapshot reuse.

## Reproduction Assessment

Yes, by source inspection. Cache a session snapshot that excludes a bin/env/config-gated skill, satisfy that prerequisite without a SKILL.md watcher or covered skills.* config bump, and the current version/filter-only reuse paths can keep stale prompt/skills data.

## Solution Assessment

No. The PR points at the right bug, but a partial raw-input hash in ensureSkillSnapshot is not the narrow maintainable fix; freshness should be shared and based on sanitized evaluated eligibility or explicit version bumps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Include actual runtime eligibility in the freshness key:** `src/auto-reply/reply/session-updates.ts:48-52`
  - body: The signature hashes `cfg.skills`, the normalized filter, `PATH`, and remote eligibility, but skill inclusion also depends on process env values, full-config `requires.config` paths, and actual binary availability. Installing a binary into an existing PATH directory or toggling a non-`skills` config prerequisite can leave the same signature and reuse a stale snapshot.
  - confidence: 0.92
- **[P2] Share freshness across every snapshot cache:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: This invalidation is added only inside `ensureSkillSnapshot`. The CLI agent and isolated cron paths still reuse existing snapshots when version and skillFilter match, so the same eligibility transition remains stale outside reply sessions.
  - confidence: 0.9
- **[P2] Keep secret values out of persisted snapshot metadata:** `src/auto-reply/reply/session-updates.ts:48-50`
  - body: `cfg.skills` can include raw `apiKey` and `env` values, and the PR stores a digest derived from that payload on `SessionSkillSnapshot`. Use sanitized eligibility state or prerequisite booleans instead of persistent metadata derived from secret-adjacent config.
  - confidence: 0.86
