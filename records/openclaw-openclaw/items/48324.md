## Summary

Keep this PR open but not merge-ready: the branch targets a real stale skill-snapshot bug, but the proposed signature is incomplete, persists a raw-config-derived verifier, only covers one session path, lacks real behavior proof, and now conflicts with newer merged/shared snapshot work.

## What This Changes

The PR adds an `eligibilitySignature` to persisted skill snapshots and refreshes auto-reply session snapshots when the watcher version, skill filter, or signature changes.

## Best Possible Solution

Rebase or replace this with one shared redacted snapshot-freshness contract applied to every persisted snapshot reuse path, preserving the current watch-root invalidation and proving config/env/bin refresh behavior in a real session.

## Reproduction Assessment

Yes, at source level for the PR defects: skill inclusion depends on runtime binary/env/config facts that the proposed signature does not cover. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. The narrow auto-reply-only raw signature is not the best fix; the safer direction is a shared redacted/evaluated freshness contract across all persisted snapshot reuse paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Include real eligibility inputs in the signature:** `src/auto-reply/reply/session-updates.ts:48-53`
  - body: The new signature hashes `cfg.skills`, `skillFilter`, `PATH`, and remote eligibility, but skill inclusion also depends on `hasBinary()`, env vars, PATHEXT on Windows, and `requires.config` checks against the full config. Installing a required binary in an existing PATH directory or toggling a non-`skills` config/env gate can leave the persisted snapshot stale even after this patch.
  - confidence: 0.9
- **[P2] Refresh every persisted snapshot cache:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: This invalidation is only wired into `ensureSkillSnapshot`, while agent-command and cron/isolated-agent paths still reuse persisted snapshots by version/filter. Users entering through those paths can continue seeing stale skills after eligibility changes, so the freshness contract needs to be shared across all snapshot reuse sites.
  - confidence: 0.86
- **[P2] Avoid persisting secret-derived snapshot metadata:** `src/auto-reply/reply/session-updates.ts:48-54`
  - body: `cfg.skills` can contain configured env values or API-key-shaped fields, and this patch persists a digest of that raw object in `sessions.json`. Use a redacted presence/type fingerprint instead so session metadata does not become a stable verifier for secret-shaped config values.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch persists a SHA-256 digest derived from raw `cfg.skills`, which can expose a stable verifier for secret-shaped skill config in the session store.
