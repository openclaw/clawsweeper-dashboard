## Summary

Keep open: this addresses a real stale skillsSnapshot problem, but the PR is not merge-ready because it lacks real behavior proof, is merge-conflicting, and the freshness key still misses evaluated eligibility while adding secret-derived persisted metadata.

## What This Changes

The PR adds a persisted eligibilitySignature to session skill snapshots and refreshes reply-session snapshots when the watcher version, skill filter, or signature changes.

## Best Possible Solution

Replace the narrow persisted raw signature with a shared redacted/evaluated freshness contract or version-bump path used consistently by reply, cron, CLI-agent, watcher, and config reload snapshots.

## Reproduction Assessment

Yes, at source level: current snapshot reuse gates compare version/filter while skill inclusion depends on env, binary, remote, and config inputs that can change independently. I did not run a live gateway session in this read-only review.

## Solution Assessment

No. The current PR is too narrow and security-sensitive; the maintainable fix is a shared redacted/evaluated freshness contract or equivalent version bump applied to every persisted skill snapshot cache.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Include evaluated eligibility in the freshness key:** `src/auto-reply/reply/session-updates.ts:48-52`
  - body: The signature hashes cfg.skills, skillFilter, PATH, and remote eligibility, but skill inclusion is decided by hasBinary, env values, and requires.config checks against the full config. Installing or removing a binary in the same PATH directory, exporting or unsetting env, or toggling non-skills config can leave this signature unchanged and the session snapshot stale.
  - confidence: 0.9
- **[P2] Apply freshness to every snapshot cache:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: This invalidation only runs in ensureSkillSnapshot. Cron isolated-agent and openclaw agent session paths still reuse cached snapshots based on version/filter only, so the same eligibility transition can remain stale outside the reply-session path.
  - confidence: 0.86
- **[P2] Avoid persisting secret-derived snapshot metadata:** `src/auto-reply/reply/session-updates.ts:48-54`
  - body: The persisted eligibilitySignature is a hash of cfg.skills, which can include skills.entries.* apiKey or env fields. Freshness metadata should be derived from redacted or evaluated eligibility state so sessions.json does not gain a stable verifier for secret-shaped config.
  - confidence: 0.82
