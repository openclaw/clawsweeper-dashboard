## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Include the actual eligibility inputs:** `src/auto-reply/reply/session-updates.ts:48-53`
  - body: This signature changes only when `cfg.skills`, the filter, `PATH`, or remote eligibility changes, but skill inclusion also depends on env/apiKey presence, binary lookup results, PATHEXT, and `requires.config`. Existing sessions can therefore remain stale after common eligibility changes, so this does not reliably fix the reported bug.
  - confidence: 0.9
- **[P1] Avoid storing a raw secret-derived verifier:** `src/auto-reply/reply/session-updates.ts:54`
  - body: The digest is computed from raw `cfg.skills` and then persisted on the session snapshot. Since skill config supports `apiKey` and env values, this creates a stable offline verifier for secret-shaped values; use a redacted presence/type fingerprint or keep the key process-local.
  - confidence: 0.86
- **[P2] Apply the refresh contract to every snapshot reuse path:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: The new invalidation check is limited to `ensureSkillSnapshot`, while other persisted snapshot consumers still use version/filter-only refresh decisions. That leaves CLI agent and isolated-session paths able to serve stale skill state after the same eligibility changes.
  - confidence: 0.82
