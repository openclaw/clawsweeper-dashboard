## Summary

## What This Changes

The PR adds an eligibility signature to persisted skill snapshots so existing sessions can refresh after runtime skill eligibility changes without a SKILL.md watcher bump.

## Best Possible Solution

Rebase or replace this with a shared redacted/evaluated snapshot freshness contract that covers auto-reply, agent-command, and cron reuse paths, while letting the open config-fingerprint PR own config-driven exposure refresh if maintainers choose that path.

## Reproduction Assessment

Yes, source-level. Current main and the PR diff show snapshot reuse can depend on version/filter/signature while actual skill eligibility depends on env, binary filesystem state, remote state, and full config inputs; I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. The PR's narrow persisted eligibilitySignature is not the best fix because it misses eligibility inputs, is not shared across all snapshot reuse paths, and stores a raw-config-derived verifier; the safer path is a shared redacted/evaluated freshness contract or a narrower canonical PR.
