## Summary

Keep open: the PR targets a real stale skillsSnapshot bug, but the current branch is not correct enough to merge because the freshness signature is partial, reply-session-only, and persists secret-derived metadata. Current main has adjacent resolvedSkills cache work, but it does not provide a shared full-snapshot freshness contract.

## What This Changes

The PR adds an eligibility signature to persisted session skill snapshots so reply sessions rebuild cached snapshots when runtime eligibility appears to change without a SKILL.md watcher bump.

## Best Possible Solution

Centralize a sanitized snapshot freshness contract in the skills/session layer and apply it consistently to reply sessions, CLI agent sessions, and isolated cron snapshot reuse.

## Reproduction Assessment

Yes, by source inspection. Cache a snapshot that excludes a binary/env/config-gated skill, satisfy that prerequisite without a SKILL.md watcher bump, and the version/filter-only cache paths can still reuse stale prompt/skills data.

## Solution Assessment

No. The PR points at the right bug, but hashing partial raw inputs only in the reply-session path is not the narrow maintainable fix; the freshness key should be shared and based on sanitized evaluated eligibility or explicit version bumps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Include actual eligibility inputs in the freshness key:** `src/auto-reply/reply/session-updates.ts:48-52`
  - body: The signature only hashes cfg.skills, the normalized filter, PATH, and remote eligibility. Skill inclusion also depends on process env, skillConfig.env/apiKey, requires.config, and binary availability results, so changing those can leave the same signature and reuse a stale snapshot.
  - confidence: 0.92
- **[P2] Apply freshness checks to every snapshot cache:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: This invalidation check is added only inside ensureSkillSnapshot. The CLI agent path and isolated cron helper on current main still reuse snapshots when version/filter match, so eligibility changes remain stale outside reply sessions.
  - confidence: 0.9
- **[P2] Avoid persisting secret-derived snapshot metadata:** `src/auto-reply/reply/session-updates.ts:48-50`
  - body: The payload hashes params.cfg.skills and stores the digest on SessionSkillSnapshot, while SkillConfig can include apiKey and env values. Use sanitized eligibility state or prerequisite booleans instead of deriving persistent metadata from raw secret-adjacent config.
  - confidence: 0.86
