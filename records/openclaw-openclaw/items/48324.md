## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Include real eligibility inputs in the signature:** `src/auto-reply/reply/session-updates.ts:48-53`
  - body: The new signature hashes `cfg.skills`, `skillFilter`, `PATH`, and remote eligibility, but skill inclusion also depends on `hasBinary()`, env vars, PATHEXT on Windows, and `requires.config` checks against the full config. Installing a required binary in an existing PATH directory or toggling a non-`skills` config/env gate can leave the persisted snapshot stale even after this patch.
  - confidence: 0.9
- **[P2] Refresh every persisted snapshot cache:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: This invalidation is only wired into `ensureSkillSnapshot`, while agent-command and cron/isolated-agent paths still reuse persisted snapshots by version/filter. Users entering through those paths can continue seeing stale skills after eligibility changes, so the freshness contract needs to be shared across all snapshot reuse sites.
  - confidence: 0.86
- **[P2] Avoid persisting secret-derived snapshot metadata:** `src/auto-reply/reply/session-updates.ts:48-54`
  - body: `cfg.skills` can contain configured env values or API-key-shaped fields, and this patch persists a digest of that raw object in `sessions.json`. Use a redacted presence/type fingerprint instead so session metadata does not become a stable verifier for secret-shaped config values.
  - confidence: 0.84
