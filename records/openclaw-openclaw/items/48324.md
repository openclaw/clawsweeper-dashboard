## Summary

Keep open: the PR addresses a real stale skills snapshot bug, but the current branch is not mergeable as-is because its signature omits evaluated eligibility outcomes, only covers the reply-session cache, lacks real behavior proof, and introduces secret-derived persisted metadata.

## What This Changes

The branch adds an `eligibilitySignature` field to session skill snapshots and uses it in reply-session snapshot invalidation.

## Best Possible Solution

Implement one shared, secret-safe snapshot freshness mechanism for reply sessions, CLI agent sessions, and isolated cron based on sanitized evaluated eligibility outcomes or explicit version bumps.

## Reproduction Assessment

Yes, by source inspection. Cache a snapshot excluding a skill gated by a binary, env var, or `requires.config`, then satisfy that prerequisite without changing `SKILL.md`, the watcher version, or the PR's hash inputs; stale snapshots can still be reused on affected paths.

## Solution Assessment

No. The direction is useful, but hashing raw partial inputs only in the reply-session path is not the narrow maintainable fix; the freshness contract should be shared across snapshot caches and avoid secret-derived persisted metadata.
