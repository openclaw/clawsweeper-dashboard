## Summary

Keep open for maintainer review: the PR targets a real stale skill-snapshot problem, but it is not merge-ready because proof is missing and the proposed freshness key is partial, auto-reply-only, and raw-config-derived.

## What This Changes

The PR adds an eligibility signature to persisted session skill snapshots and uses it to refresh auto-reply snapshots when watcher version, skill filter, or eligibility inputs change.

## Best Possible Solution

Rebase or replace this with a shared redacted/evaluated snapshot freshness contract that covers auto-reply, agent-command, and cron reuse paths, while letting https://github.com/openclaw/openclaw/pull/82365 own the config-driven exposure fix if maintainers choose that path.

## Reproduction Assessment

Yes, source-level: current and PR code show persisted snapshot reuse can depend on version/filter/signature while actual skill eligibility depends on env, binary filesystem state, remote, and full config inputs. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. The PR's narrow persisted eligibilitySignature is not the best fix because it misses eligibility inputs, is not shared across all snapshot reuse paths, and stores a raw-config-derived verifier; the safer path is a shared redacted/evaluated freshness contract or a narrower canonical PR.
