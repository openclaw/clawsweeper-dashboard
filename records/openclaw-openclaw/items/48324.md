## Summary

Keep open, but the patch is not merge-ready. The underlying stale skillsSnapshot bug is source-reproducible, while this branch adds a partial reply-session-only signature, misses several eligibility inputs, persists a secret-derived digest, and still lacks after-fix real behavior proof.

## What This Changes

The PR adds an eligibility signature to persisted session skill snapshots and refreshes the auto-reply snapshot when that signature changes alongside watcher version or skill-filter changes.

## Best Possible Solution

Centralize a sanitized skills snapshot freshness contract in the skills/session layer, attach it from buildWorkspaceSkillSnapshot, and compare it consistently in reply sessions, CLI agent sessions, and isolated cron reuse while coordinating with https://github.com/openclaw/openclaw/pull/82365 and https://github.com/openclaw/openclaw/pull/71497.

## Reproduction Assessment

Yes, by source inspection. Cache a snapshot that excludes a config/env/bin-gated skill, then change the prerequisite without a watcher/version bump, and the current version/filter-only predicates can still reuse stale prompt and skill metadata.

## Solution Assessment

No. The PR points at the right bug, but a partial raw signature in one reply-session path is not the maintainable fix; the freshness key should be shared, sanitized, and applied to every persisted snapshot reuse path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Include actual eligibility inputs in the freshness key:** `src/auto-reply/reply/session-updates.ts:48-52`
  - body: The signature hashes `cfg.skills`, the normalized filter, `PATH`, and remote eligibility, but skill inclusion also depends on process env, per-skill `env`/`apiKey`, full-config `requires.config` paths, and actual binary availability. Changing those can leave the same signature and keep serving a stale snapshot.
  - confidence: 0.92
- **[P2] Apply freshness checks to every snapshot cache:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: This invalidation check only runs in `ensureSkillSnapshot`. The CLI agent and isolated cron paths on current main still reuse snapshots when version/filter match, so eligibility changes remain stale outside reply sessions unless the freshness contract is shared.
  - confidence: 0.9
- **[P2] Avoid persisting secret-derived snapshot metadata:** `src/auto-reply/reply/session-updates.ts:48-50`
  - body: The payload hashes `params.cfg.skills`, while `SkillConfig` supports `apiKey` and `env`. Even as a digest, persisted session metadata should be based on redacted config state or evaluated prerequisite booleans rather than raw secret-adjacent config values.
  - confidence: 0.86
