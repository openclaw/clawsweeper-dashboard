## Summary

Keep open: this targets a real stale skill-snapshot bug, but the branch is not merge-ready because it lacks real behavior proof, is currently conflicting, and adds a partial, secret-derived, reply-only freshness signature.

## What This Changes

The PR adds a persisted eligibility signature to session skill snapshots and rebuilds reply-session snapshots when the watcher version, skill filter, or signature changes.

## Best Possible Solution

Centralize a sanitized skill-snapshot freshness contract that captures evaluated eligibility inputs and apply it consistently to reply, CLI-agent, and cron snapshot caches with runtime upgrade proof.

## Reproduction Assessment

Yes. Source inspection plus the linked issue gives a high-confidence path: persist a snapshot that excludes an env/bin/config-gated skill, change the prerequisite without a watcher bump, and the version/filter-only cache decisions can keep serving stale skill data.

## Solution Assessment

No. The PR addresses one reply-session path but misses evaluated runtime prerequisites, other snapshot caches, and secret-safe persistence, so a shared sanitized freshness contract is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Include actual runtime prerequisites in the freshness key:** `src/auto-reply/reply/session-updates.ts:48-52`
  - body: The signature hashes cfg.skills, skillFilter, PATH text, and remote eligibility, but skill inclusion also depends on evaluated binary presence, env/API-key availability, and requires.config checks. Installing a binary in an existing PATH directory, exporting/unsetting a required env var, or toggling a non-skills config gate can leave the signature unchanged and the snapshot stale.
  - confidence: 0.88
- **[P2] Apply freshness checks to every snapshot cache:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: This invalidation only runs in ensureSkillSnapshot, while the cron and CLI-agent snapshot paths still reuse cached snapshots based on version/filter only. The same eligibility changes can therefore leave stale skill lists outside reply sessions after this patch lands.
  - confidence: 0.84
- **[P2] Avoid persisting secret-derived snapshot metadata:** `src/auto-reply/reply/session-updates.ts:48-54`
  - body: cfg.skills can contain apiKey/env-backed eligibility data, and this change writes a stable SHA-256 digest of that raw payload into persisted session metadata. Use a sanitized presence/shape contract instead of persisting a verifier derived from secret values.
  - confidence: 0.78
