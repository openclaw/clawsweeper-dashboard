## Summary

Keep this PR open, but it should not merge as-is: the current branch still leaves source-provable stale skill snapshot paths, derives persisted metadata from secret-adjacent skill config, and lacks real after-fix behavior proof.

## What This Changes

The PR adds an `eligibilitySignature` field to session skill snapshots and refreshes reply-session snapshots when the watcher version, skill filter, or signature changes.

## Best Possible Solution

Implement one shared, secret-safe snapshot freshness mechanism for reply sessions, CLI agent sessions, and isolated cron, based on sanitized evaluated eligibility outcomes or explicit version bumps.

## Reproduction Assessment

Yes, by source inspection: cache a snapshot excluding a skill gated by a binary, env var, or `requires.config`, then satisfy that prerequisite without changing `SKILL.md`, the watcher version, or the PR's hash inputs. The stale snapshot can still be reused on affected paths.

## Solution Assessment

No. The direction is useful, but hashing raw partial inputs in only the reply-session path is not the narrow maintainable fix; the freshness contract should be shared across snapshot caches and avoid secret-derived persisted metadata.

  - confidence: 0.91

  - body: `cfg.skills` can include `SkillConfig` `apiKey` and `env` values. Persisting a digest derived from that object as session metadata creates unnecessary secret-derived state; use sanitized eligibility outcomes or non-secret prerequisite booleans instead.
  - confidence: 0.86
