## Summary

Keep this PR open, but the current patch should not merge as-is: it addresses a real stale skillsSnapshot bug, but the signature only covers partial inputs, only applies to the reply-session cache, stores secret-derived metadata, and still lacks after-fix real behavior proof.

## What This Changes

The PR adds an `eligibilitySignature` field to persisted skill snapshots and uses it to refresh reply-session snapshots when watcher version, skill filter, or eligibility signature changes.

## Best Possible Solution

Centralize a sanitized snapshot freshness contract in the skills/session layer and apply it uniformly to reply sessions, CLI agent sessions, and isolated cron snapshot reuse.

## Reproduction Assessment

Yes. By source inspection, cache a snapshot excluding a binary/env/config-gated skill, satisfy that prerequisite without a SKILL.md watcher bump, and the version/filter-only cache paths can continue reusing the stale snapshot.

## Solution Assessment

No. The PR direction is useful, but hashing partial raw inputs only in the reply-session path is not the narrow maintainable fix; the freshness key should be shared and based on sanitized evaluated eligibility or explicit version bumps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Include actual runtime prerequisites in the signature:** `src/auto-reply/reply/session-updates.ts:48-52`
  - body: The signature hashes `cfg.skills`, `skillFilter`, `PATH`, and remote eligibility, but snapshot inclusion is decided by live binary probes, process/configured env, apiKey-backed env, and `requires.config` checks. Those prerequisites can change while this digest stays unchanged, so a stale snapshot can still be reused.
  - confidence: 0.94
- **[P2] Apply freshness to all persisted snapshot caches:** `src/auto-reply/reply/session-updates.ts:177-181`
  - body: This invalidation runs only in `ensureSkillSnapshot`. The CLI agent and isolated cron paths still reuse persisted snapshots when version/filter match, so the same eligibility changes can continue serving stale skill lists outside the reply-session path.
  - confidence: 0.91
- **[P2] Avoid deriving snapshot metadata from skill secrets:** `src/auto-reply/reply/session-updates.ts:48-50`
  - body: `cfg.skills` can contain `SkillConfig` `apiKey` and `env` values. Persisting a digest of that raw object creates unnecessary secret-derived session metadata; use sanitized evaluated eligibility or non-secret prerequisite booleans instead.
  - confidence: 0.86
