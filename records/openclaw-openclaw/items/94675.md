## Summary

Keep open for maintainer architecture review: current main and v2026.6.8 are still intentionally SQLite-first, and this is a broad product direction change rather than a narrow bug, duplicate, or already-implemented request.

## What This Changes

The issue requests a backend-neutral database abstraction that keeps SQLite first-class while enabling an async Postgres-class backend across OpenClaw-owned relational state, plugin state, memory/search, transactions, migrations, and backup/restore.

## Best Possible Solution

Keep this as a maintainer architecture/design item and require an RFC that reconciles the SQLite-first policy, backend selection, per-dialect migrations, plugin SDK transition, memory/search provider seams, and backup/restore semantics before implementation.

## Reproduction Assessment

Not applicable; this is an architecture feature request rather than a bug. Source and release inspection show current main and v2026.6.8 still use SQLite-specific state paths, so there is no failing current-main repro path to run.

## Solution Assessment

No, not as an immediate implementation task. The maintainable path is a maintainer-approved design/RFC that decides whether OpenClaw should change its SQLite-only storage policy and how the transition works across state, SDK, search, migrations, and backups.
