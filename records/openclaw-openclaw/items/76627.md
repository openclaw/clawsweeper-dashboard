## Summary

Close as config/support: the reported no-visible-reply symptom matches the current documented group/channel visible-reply policy, and the follow-up workaround uses the existing supported setting rather than a Telegram topic code fix.

## What This Changes

The issue asks OpenClaw to visibly respond again to Telegram supergroup forum-topic messages that show typing but no chat reply.

## Best Possible Solution

Use the existing `messages.groupChat.visibleReplies: "automatic"` setting for legacy visible group/topic replies, and open a new bug only if topic replies still fail after that config is active.

## Reproduction Assessment

Yes, for the visible symptom: current source and docs show group/topic turns without `messages.groupChat.visibleReplies: "automatic"` default to message-tool-only delivery, which can show typing while suppressing normal final chat output. I did not establish a separate Telegram delivery failure.

## Solution Assessment

Yes. The narrowest maintainable solution is the existing `messages.groupChat.visibleReplies: "automatic"` configuration; a code change is not justified without logs showing failure after that setting is applied.
