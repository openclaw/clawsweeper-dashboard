## Summary

Keep open. Current main still has the explicit Crestodian no-TTY path log the TTY error and return normally, and the issue is paired with open closing PRs #73928 and #74059 rather than a merged fix.

## What This Changes

The issue reports that explicit `openclaw crestodian` in a non-TTY context prints the interactive TTY error but exits successfully instead of returning a non-zero code.

## Best Possible Solution

Land one focused fix from #73928, #74059, or an equivalent maintainer patch that makes the explicit `runCrestodian()` no-TTY guard exit non-zero while preserving `--message` and `--json`; keep #73562 as separate prompt EOF cleanup unless maintainers choose a broader shared helper.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY, and the function logs the error then returns normally; the reporter also provided deterministic shell reproductions.

## Solution Assessment

Yes. The narrowest maintainable fix is to fail the existing Crestodian no-TTY guard through the CLI's non-zero path and add targeted regression tests for stdin-not-TTY and stdout-not-TTY behavior.
