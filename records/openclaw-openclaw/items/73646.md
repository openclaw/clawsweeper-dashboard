## Summary

Keep open. Current main still handles the explicit `openclaw crestodian` no-TTY path by printing the Crestodian error and returning normally, so the command wrapper has no thrown error to convert into a non-zero exit; open closing PRs #73928 and #74059 already target this narrow defect.

## What This Changes

Make `openclaw crestodian` exit non-zero when invoked without an interactive TTY and without `--message`, while preserving the existing `--message` and `--json` headless paths.

## Best Possible Solution

Review and land one focused fix from #73928, #74059, or an equivalent maintainer-selected patch that makes the existing Crestodian no-TTY guard fail through a non-zero CLI path with targeted stdin/stdout regression coverage, while preserving `--message` and `--json` headless behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY; current code logs the error and returns through a wrapper that only exits non-zero on thrown errors.

## Solution Assessment

Yes. The narrow maintainable repair is to make the existing Crestodian no-TTY guard fail non-zero and test that path; broader prompt/TTY unification can remain separate with #73562 if maintainers want that cleanup.
