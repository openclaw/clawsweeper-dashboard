## Summary

Keep open. Current main still has the reported explicit `crestodian` non-TTY path logging an error and returning normally, and open closing PRs #73928 and #74059 already target the same narrow defect, so the issue should wait for maintainer review and one merged fix.

## What This Changes

The issue requests that `openclaw crestodian` return a non-zero exit code when invoked without an interactive TTY and no `--message` override.

## Best Possible Solution

Review and land one focused fix from #73928, #74059, or an equivalent maintainer-selected patch that makes the existing `runCrestodian()` no-TTY guard fail non-zero with targeted stdin/stdout regression coverage while preserving `--message` and `--json` headless behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: call `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY, and it logs the error then returns normally; the reporter also provided deterministic shell reproductions against v2026.4.27.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing Crestodian no-TTY guard fail through a non-zero CLI path and cover it with focused tests; a broader shared TTY helper can be handled separately with #73562 if maintainers want that cleanup.
