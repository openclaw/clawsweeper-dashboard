## Summary

Keep open. Current main still has the explicit `crestodian` non-TTY path logging the right error but returning normally, and two open closing PRs already target the same defect, so the issue should stay open until one fix lands.

## What This Changes

The issue requests that `openclaw crestodian` exit non-zero when invoked without `--message` in a non-interactive TTY context.

## Best Possible Solution

Land one focused fix from #73928, #74059, or an equivalent maintainer patch that makes the existing `runCrestodian()` no-TTY guard fail with a non-zero CLI exit and adds targeted stdin/stdout regression coverage while preserving `--message` and `--json` headless behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY, and it logs the error then returns through a wrapper that only fails on thrown errors.

## Solution Assessment

Yes. The narrow maintainable solution is to make the existing Crestodian no-TTY guard exit non-zero and cover it with focused tests; a broader shared TTY helper can be handled separately with #73562 if maintainers want that cleanup.
