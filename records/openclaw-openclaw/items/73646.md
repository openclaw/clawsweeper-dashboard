## Summary

Keep open. Current main still has the reported explicit `crestodian` no-TTY path logging an error and returning normally, while two open closing PRs already target the same narrow defect, so the issue should wait for maintainer review and one merged fix.

## What This Changes

The issue asks for `openclaw crestodian` to exit non-zero when invoked without an interactive TTY and without a `--message` one-shot request.

## Best Possible Solution

Land one focused fix that makes the existing `runCrestodian()` no-TTY guard fail non-zero with targeted stdin/stdout regression coverage, while preserving the current `--message` one-shot and `--json` headless behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: call `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY, and it logs the error then returns normally; the reporter also provided deterministic shell reproductions against v2026.4.27.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing Crestodian no-TTY guard route through a non-zero CLI outcome and cover it with focused tests; broader TTY prompt normalization can stay with related work such as #73562.
