## Summary

Keep open. Current main still returns normally after the explicit Crestodian no-TTY error, and open closing PRs #73928 and #74059 already target the narrow fix, so this issue should wait for maintainer review and merge of one candidate patch.

## What This Changes

The issue asks for explicit `openclaw crestodian` invocations without an interactive TTY to exit non-zero instead of printing the no-TTY error and exiting successfully.

## Best Possible Solution

Review and land one focused fix from #73928, #74059, or an equivalent maintainer-selected patch that makes the existing Crestodian no-TTY guard fail through a non-zero CLI path while preserving `--message` and `--json` headless behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY; current code logs the error and returns through a wrapper that only exits non-zero on thrown errors.

## Solution Assessment

Yes. The narrow maintainable repair is to make the existing Crestodian no-TTY guard fail non-zero and add targeted regression coverage; broader prompt-normalization work can remain with related issue #73562 if maintainers want it.
