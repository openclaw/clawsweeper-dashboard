## Summary

Keep open. Current main still logs the Crestodian no-TTY error and returns normally, so the explicit `openclaw crestodian` subcommand can complete as success; open closing PRs #73928 and #74059 already target this narrow defect.

## What This Changes

Make explicit `openclaw crestodian` fail non-zero when invoked without an interactive TTY and without a `--message`, matching nearby interactive CLI command behavior.

## Best Possible Solution

Review and land one focused fix from #73928, #74059, or an equivalent maintainer-selected patch that makes the existing `runCrestodian()` no-TTY guard fail through a non-zero CLI path with targeted stdin/stdout regression coverage while preserving `--message` and `--json` headless behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY; current code logs the error and returns through a wrapper that only exits non-zero on thrown errors.

## Solution Assessment

Yes. The narrow maintainable repair is to make the existing Crestodian no-TTY guard fail non-zero and test that path; a broader shared TTY helper can be handled separately with #73562 if maintainers want that cleanup.
