## Summary

Keep open. Current main still has the explicit `crestodian` non-TTY path logging the TTY error and returning normally, while the CLI wrapper only exits non-zero for thrown errors; open PRs #73928 and #74059 already target this defect, so the issue should wait for one canonical fix to merge.

## What This Changes

The issue requests that `openclaw crestodian` exit non-zero when invoked without `--message` in non-TTY contexts, matching other interactive CLI commands.

## Best Possible Solution

Land one focused patch that makes the existing `runCrestodian()` no-TTY guard fail through a non-zero CLI path, adds stdin/stdout non-TTY regression coverage, preserves `--message` and `--json` headless behavior, and includes the required user-facing changelog entry.

## Reproduction Assessment

Yes. The reporter provided deterministic shell reproductions, and current-main source inspection gives the same path: call `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY, and it logs the error then returns normally.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing Crestodian no-TTY guard fail non-zero and cover that behavior with focused tests; a broader shared TTY helper can remain separate or be coordinated with #73562.
