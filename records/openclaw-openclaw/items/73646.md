## Summary

Keep open. Current main still has the explicit `openclaw crestodian` no-TTY path logging an error and returning normally, so the CLI wrapper treats it as success; open closing PRs #73928 and #74059 already target the same narrow defect and should be reviewed before closing the issue.

## What This Changes

Request that `openclaw crestodian` exit non-zero when invoked without an interactive TTY and without `--message`, matching sibling interactive CLI commands.

## Best Possible Solution

Land one focused fix from #73928, #74059, or an equivalent maintainer-selected patch that makes the existing `runCrestodian()` no-TTY guard fail non-zero with targeted regression coverage while preserving `--message`, `--json`, and interactive TUI behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: call `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY, and it logs the error then returns normally through a wrapper that only exits non-zero on thrown errors.

## Solution Assessment

Yes. The narrow maintainable fix is to make the current Crestodian no-TTY guard use the existing non-zero CLI failure path and add focused tests; a broader shared TTY helper can be considered separately with related prompt/TTY cleanup such as #73562.
