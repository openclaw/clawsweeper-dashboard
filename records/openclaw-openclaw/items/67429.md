## Summary

Keep this PR open, but it still needs changes before merge. The intended cron.update null-as-clear behavior is valid, yet the current branch preserves null in a normalizer shared with cron.add, which can regress create requests that previously tolerated optional null placeholders.

## What This Changes

The PR adds null-as-clear support for cron.update agent-turn payload model, fallbacks, and thinking overrides across cron types, normalization, gateway schema, service merge logic, and tests.

## Best Possible Solution

Land a revised patch that makes null-clear a cron.update-only contract while preserving cron.add create-time null dropping, with regression coverage for update clearing and create compatibility.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: cron.update normalizes patches before validation, current main drops these nulls, and the PR-side shared normalizer change can be exercised through normalizeCronJobCreate followed by create-schema validation.

## Solution Assessment

No. The requested behavior is the right direction, but preserving null inside the shared coercePayload path is not the safest implementation; patch-only normalization should carry the nulls while create normalization keeps dropping them.
