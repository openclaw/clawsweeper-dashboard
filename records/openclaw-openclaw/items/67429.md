## Summary

Keep open: the underlying cron.update fix is useful, but the current branch still preserves null in a shared create/update normalizer and can regress cron.add; it also lacks after-fix real behavior proof beyond tests/CI.

## What This Changes

The PR makes cron.update accept null for agent-turn payload model, fallbacks, and thinking patches across cron types, gateway schema, normalization, merge logic, and tests.

## Best Possible Solution

Land a revised patch that scopes null preservation to cron.update patches, keeps cron.add create-time optional-null dropping, deletes the three stored overrides on null, and covers both update clearing and create compatibility in regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: cron.update normalizes patches before validation, current main drops these nulls, and the PR-side shared coercePayload change can be reasoned through normalizeCronJobCreate into non-nullable create-schema validation.

## Solution Assessment

No. The intended null-clear behavior is the right narrow fix, but implementing it in shared coercePayload is not the safest solution; patch-only normalization should carry nulls while create normalization keeps dropping them.
