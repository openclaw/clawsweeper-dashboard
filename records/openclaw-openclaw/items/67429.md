## Summary

Keep open. The underlying `cron.update` clear bug is real on current main, but this PR still preserves nulls in a shared create/patch normalizer, which would make `cron.add` reject previously tolerated optional-null create payloads. It also needs the required user-facing changelog entry before merge.

## What This Changes

The PR adds null-as-clear semantics for `cron.update` `payload.model`, `payload.fallbacks`, and `payload.thinking` across cron patch types, normalization, gateway schema, service merge logic, and regression tests.

## Best Possible Solution

Land a revised version that defines null-clear as a `cron.update` patch-only contract, keeps create normalization tolerant by dropping optional nulls, deletes the three stored overrides during patch application, updates the patch schema/types, adds normalize-to-apply and create-compatibility tests, and records the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current `cron.update` normalizes before validation, current normalization drops the requested nulls, and current merge logic ignores null for those fields; the PR-branch regression follows from the same shared normalizer being used by `cron.add`.

## Solution Assessment

No. The requested direction is the narrow maintainable fix, but the current implementation is not the best way because null preservation belongs in patch normalization only, not the shared create path.
