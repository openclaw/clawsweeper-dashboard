## Summary

Keep this PR open: the reported cron.update clearing bug is source-reproducible on current main, but the current branch still introduces a create-path regression by preserving nulls in shared normalization and it lacks after-fix real behavior proof.

## What This Changes

The PR adds null-as-clear handling for cron.update payload.model, payload.fallbacks, and payload.thinking across cron types, normalization, gateway schema, merge logic, and regression tests.

## Best Possible Solution

Land a revised patch-only null-clear implementation that preserves cron.add optional-null tolerance, deletes stored overrides on cron.update, and covers normalize-to-apply plus create-compatibility regression paths.

## Reproduction Assessment

Yes, by source inspection: cron.update normalizes patches before validation, current main drops model/fallbacks/thinking nulls, and mergeCronPayload only deletes toolsAllow. I did not run tests because this review is read-only.

## Solution Assessment

No. The intended null-clear behavior is the right narrow direction, but preserving nulls in shared coercePayload is not the safest implementation because create and update normalization share that path.
