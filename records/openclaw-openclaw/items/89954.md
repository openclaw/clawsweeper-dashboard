## Summary

Keep open: the report is a credible Telegram channel outage, and current main has partial lifecycle fixes but no proof that the broken-IPv6 to IPv4 fallback plus getUpdates 409 cascade is fixed. The linked open IPv4-pinning PR is only a mitigation and does not cover the broader cancellation/backoff behavior described here.

## What This Changes

The issue asks OpenClaw to stop Telegram getUpdates 409 rebuild cascades after IPv6-to-IPv4 fallback by canceling stale long-polls, preserving the working transport policy, backing off around Telegram stale-poll TTLs, or adding an explicit IPv4 opt-in.

## Best Possible Solution

Cluster this with the open Telegram polling tracker and the IPv4-force PR, then land a Telegram-owned transport fix that breaks the default fallback/rebuild cascade without adding a new env surface unless maintainers explicitly accept that mitigation.

## Reproduction Assessment

No high-confidence current-main live reproduction is available. Source inspection still shows the default dispatcher-to-IPv4-fallback path and non-merged IPv4-force mitigation, while the exact 409 cascade needs an affected-host retest.

## Solution Assessment

Unclear. A hard IPv4 env pin is an acceptable escape hatch, but the best durable fix should address Telegram transport cancellation, fallback lifetime, and restart timing in the plugin rather than only adding another operator knob.
