## Summary

Keep open: current main and v2026.6.8 still contain the sticky IPv4 fallback, getUpdates 409 restart/backoff, and asynchronous stale-transport rebuild paths, and the open linked IPv4 env-var PR is only a partial operator mitigation rather than a merged durable fix.

## What This Changes

The issue reports Telegram polling entering a persistent getUpdates 409 conflict loop after broken IPv6 triggers IPv4 fallback and transport rebuilds without fully canceling stale long-polls.

## Best Possible Solution

Land a Telegram-owned polling lifecycle fix that prevents same-token long-poll overlap across fallback and rebuild paths, while treating any IPv4 startup pin as a separately approved operator mitigation.

## Reproduction Assessment

No live affected-host current-main reproduction was run in this read-only review. Source inspection shows the reported fallback, dirty transport rebuild, and getUpdates 409 restart paths still exist on current main and in v2026.6.8.

## Solution Assessment

No current implementation is proven to be the best fix. Current main has useful mitigations, but the durable solution should address Telegram polling cancellation, retry timing, and transport lifecycle with live Bot API proof.
