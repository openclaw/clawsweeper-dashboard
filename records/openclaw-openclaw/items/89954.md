## Summary

Current main already addresses the reported Telegram 409 restart cascade: the polling loop now aborts active fetches before restart, destroys stale undici dispatchers, rebuilds dirty transports, and uses a 30s-to-10m restart backoff instead of the reported fast retry loop.

## What This Changes

The issue reports Telegram getUpdates 409 conflicts cascading after IPv6-to-IPv4 fallback because transport rebuilds can restart polling before the previous long-poll is fully canceled.

## Best Possible Solution

Keep the current main behavior: preserve sticky fallback across normal polling restarts, abort and destroy stale transports on recovery, and use the 30s-plus restart backoff for 409/network rebuilds; reopen only with a fresh current-main repro showing the cascade still survives those guards.

## Reproduction Assessment

No: I did not run a live broken-IPv6 Telegram setup, and current main source/tests show the reported fast restart cascade is now mitigated rather than still reproducible.

## Solution Assessment

Yes: current main fixes this at the Telegram polling/transport owner boundary with abort, dispatcher cleanup, fresh transport rebuilds, sticky fallback reuse for clean restarts, and TTL-scale backoff instead of adding another config knob.
