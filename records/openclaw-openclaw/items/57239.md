## Summary

Keep this PR open. Current main still does not implement automatic fast-lane handling for ordinary short follow-up messages, and the linked feature request remains open, but the proposed patch still needs maintainer review because it changes the core per-session concurrency and tool-safety contract.

## What This Changes

The PR adds an auto-reply fast-lane path for short messages during an active session, threads a force-run-now queue-policy override, introduces transient run/session-lane handling, and adds queue-policy unit coverage.

## Best Possible Solution

The best end state is a maintainer-approved implementation of #56880 that either reuses the proven tool-less /btw side-result model for quick side questions or adds a documented, tested concurrency seam with hard tool/session/accounting isolation rather than prompt-only safety.

## Reproduction Assessment

Yes. Source and docs on current main reproduce the current behavior at the policy level: active non-heartbeat followups are enqueued, and per-session lanes remain serialized; the PR regressions are review-reproducible from the supplied diff plus local runner/path/accounting contracts.

## Solution Assessment

No. The PR is a plausible implementation candidate, but it is not the best mergeable solution yet because fast-lane tool safety, storePath-aware transcript placement, and transient-run session accounting are not fully isolated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Disable tools for fast-lane runs:** `src/auto-reply/reply/get-reply-run.ts:65-66`
  - body: The fast-lane run is allowed to execute while the parent session is still active, but the patch only adds a system prompt asking the model to avoid tools. Since the current queue contract serializes per-session runs to avoid shared-resource collisions, this should enforce a tool-less policy or equivalent hard guard before running in parallel.
  - confidence: 0.81
- **[P2] Resolve transient transcripts under the configured store:** `src/auto-reply/reply/get-reply-run.ts:614-619`
  - body: The fast-lane transcript path uses `resolveSessionTranscriptPath`, which ignores `storePath` and always points at the default sessions root. Users with a custom session store can get transient fast-lane files in the wrong directory or an unwritable home path; use the same store-aware path options as normal runs.
  - confidence: 0.76
- **[P2] Keep transient runs out of parent session accounting:** `src/auto-reply/reply/get-reply-run.ts:607-634`
  - body: The fast-lane context is transient, but keeping the parent `sessionKey` means the normal runner accounting path will persist usage, context-token, model, and CLI binding metadata back to the parent session after this short run. That can skew status and compaction state for the real in-flight session; fast-lane runs need an explicit skip or separate accounting path.
  - confidence: 0.73

## Security Review

Status: needs_attention

Summary: The diff introduces a security-sensitive concurrency concern because fast-lane runs can execute with tools available while the parent session is active.

Concerns:

- **[medium] Prompt-only tool suppression is not a hard boundary:** `src/auto-reply/reply/get-reply-run.ts:65`
  - body: The fast-lane system prompt discourages tool calls but does not disable them, so a parallel quick-message run can still invoke file, shell, or provider tools concurrently with the main session run under the same user authority.
  - confidence: 0.79
