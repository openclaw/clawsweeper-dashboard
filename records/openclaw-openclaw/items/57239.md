## Summary

Keep open. Current main still does not implement the proposed automatic fast-lane for ordinary short replies, and the linked concurrent-message feature remains open, but the PR branch should not merge as-is because the new parallel path weakens the existing per-session execution guarantees without hard tool isolation or enough session-event persistence safeguards.

## What This Changes

The PR adds an automatic fast-lane path for short non-command messages during active sessions by forcing immediate reply execution through a transient session context/lane, cleaning transient files, and extending queue-policy tests.

## Best Possible Solution

A mergeable solution should preserve the shipped per-session safety contract unless it introduces explicit, tested isolation: hard tool-less or allowlisted fast-lane execution, deterministic parent-session event handling, queue-order guarantees, and integration coverage for the active-run plus quick-followup path. If maintainers prefer a narrower UX, building on the existing /btw side-result model would be safer than automatic concurrent normal turns.

## Reproduction Assessment

Yes. The intended behavior can be checked on current main by tracing active collect/followup handling through resolveActiveRunQueueAction, which still enqueues followups; the PR-specific risks are reproducible by static path inspection because the diff creates a transient session lane/file while leaving normal tool execution available and building the prompt after parent system-event drain.

## Solution Assessment

No. The PR is a plausible implementation candidate for #56880, but prompt-only tool avoidance and transient transcript handling are not the narrowest maintainable safety boundary; a tool-less side-result path or a fully designed concurrent-turn contract would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Hard-disable tools for fast-lane turns:** `src/auto-reply/reply/get-reply-run.ts:641-645`
  - body: The fast-lane path moves the run onto a separate session lane while only asking the model not to call tools via the added system prompt. Normal tools remain available, so a short message can execute tool calls concurrently with the parent session run and break the existing one-run-per-session safety guarantee. Make this path tool-less or enforce a narrow allowlist before bypassing the parent lane.
  - confidence: 0.82
- **[P2] Keep drained system events in parent history:** `src/auto-reply/reply/get-reply-run.ts:642`
  - body: Fast-lane prompts are built after parent-scoped system events have already been drained, but this branch writes the turn to a transient session file and deletes it afterward. That can remove system-event context from the parent queue without persisting it to the real session transcript. Avoid draining parent events for transient turns, or persist them through the parent session path.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies or supply-chain changes, but it broadens an agent code-execution path by allowing a parallel fast-lane run with normal tools still available.

Concerns:

- **[medium] Parallel fast-lane tools are not hard-isolated:** `src/auto-reply/reply/get-reply-run.ts:643`
  - body: The PR relies on a system prompt to discourage tools while bypassing the parent session lane. Because tool availability is not restricted, an inbound quick reply can trigger concurrent tool-capable execution against the same workspace/session, which is a security-sensitive regression for command/tool safety.
  - confidence: 0.78
