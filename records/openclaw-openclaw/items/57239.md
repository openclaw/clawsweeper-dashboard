## Summary

Keep open. This PR is a plausible implementation candidate for the open per-session concurrency problem, but the branch still has merge-blocking correctness and security issues and lacks real behavior proof for the active-run fast-lane flow.

## What This Changes

The PR adds a fast-lane auto-reply path for short active-session follow-ups, including a queue-policy override, transient run/lane metadata, transcript cleanup, and unit coverage.

## Best Possible Solution

Land or replace this with a maintainer-approved quick-reply or concurrency design that isolates reply operations and session state, preserves ordering, enforces hard tool controls, and proves the real active-run flow.

## Reproduction Assessment

Yes by source inspection. With a parent session run active, an eligible fast-lane message still reaches runReplyAgent with the parent sessionKey, and current main rejects a second active ReplyOperation for that key; I did not run a live channel reproduction in this read-only pass.

## Solution Assessment

No. A fast lane may be useful, but this branch is not the best mergeable implementation until reply-operation identity, ordering/state isolation, hard tool controls, and real behavior proof are addressed.
