## Summary

Keep open. The PR targets a real latency problem, but the current branch still has merge-blocking active-run identity and tool-safety defects, lacks real behavior proof, and references an open canonical concurrency issue with closing syntax.

## What This Changes

The PR adds an automatic fast-lane path that tries to run short follow-up auto-replies immediately during an active session by adding queue-policy override support, transient lane/session handling, and queue-policy tests.

## Best Possible Solution

Replace or revise this with a maintainer-approved concurrency design that isolates reply identity, preserves queued-turn ordering and session state, enforces hard tool controls for quick turns, and includes regression plus real channel proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: during an active parent run, an eligible fast-lane message still passes the parent sessionKey into runReplyAgent, while current main rejects a second ReplyOperation for that active key.

## Solution Assessment

No. A fast lane may be useful, but this branch is not the best mergeable solution until reply-operation identity, ordering, state isolation, and tool-safety contracts are enforced in code rather than prompt guidance.
