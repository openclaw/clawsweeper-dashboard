## Summary

Keep open. Current main still preserves serialized per-session auto-reply behavior, and the PR is an unmerged, currently dirty implementation candidate with blocking concurrency and tool-safety issues on the proposed fast-lane path.

## What This Changes

The PR adds an auto-reply fast-lane path for short messages during an active session, a force-run-now queue-policy override, transient run/session-lane handling, and queue-policy unit coverage.

## Best Possible Solution

Land a maintainer-approved design that either extends the proven tool-less /btw side-result model or adds a documented fast-lane concurrency seam with hard tool disabling, isolated reply-operation identity, and regression coverage for ordering and session state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the PR regression: when a fast-lane message arrives during an active run, the PR still passes the parent sessionKey into runReplyAgent, and current main's reply registry rejects that same active key.

## Solution Assessment

No. The PR is a plausible implementation direction, but it is not the best mergeable solution yet because it does not work with current reply-run identity checks and relies on prompt-only tool suppression for a parallel run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use an isolated reply key for fast-lane runs:** `src/auto-reply/reply/get-reply-run.ts:672`
  - body: The fast-lane path sets an isolated command lane, but it still passes the parent `sessionKey` into `runReplyAgent`. `runReplyAgent` creates a `ReplyOperation` from that key, and current main rejects any key that already has an active run, so the fast-lane case returns the existing “previous run is still shutting down” path instead of running immediately.
  - confidence: 0.88
- **[P2] Disable tools for fast-lane runs:** `src/auto-reply/reply/get-reply-run.ts:647-649`
  - body: The parallel fast-lane turn only adds a system prompt asking the model to avoid tools. Since the execution path disables tools only through `opts.disableTools`, this still allows file, shell, or messaging tools to run concurrently with the parent session under the same authority.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces a security-sensitive parallel execution path whose tool suppression is advisory rather than enforced.

Concerns:

- **[medium] Prompt-only tool suppression is not a hard boundary:** `src/auto-reply/reply/get-reply-run.ts:647`
  - body: The fast-lane system prompt discourages tool calls but does not set the execution path's hard tool-disable flag, so an immediate parallel response can still invoke privileged tools while the parent run is active.
  - confidence: 0.83
