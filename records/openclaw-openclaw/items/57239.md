## Summary

Keep open. This PR is a plausible implementation candidate for https://github.com/openclaw/openclaw/issues/56880, but the latest head is merge-conflicting, lacks real behavior proof, and still has source-level blockers around reply-operation identity and prompt-only tool suppression.

## What This Changes

Adds an automatic fast-lane auto-reply path for short active-session follow-ups, including a queue-policy override, transient run/lane wiring, cleanup, and unit coverage.

## Best Possible Solution

Land a maintainer-approved quick-reply or concurrency design that isolates reply operations and session state, preserves ordering, enforces hard tool controls, and proves the real active-run flow.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with a parent session run active, an eligible fast-lane message reaches runReplyAgent with the parent sessionKey and current main rejects a second ReplyOperation for that key.

## Solution Assessment

No. A fast lane may be useful, but this branch is not the best mergeable implementation until reply-operation identity, ordering/state isolation, and tool-safety contracts are enforced in code and proven in a real setup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use an isolated reply operation key:** `src/auto-reply/reply/get-reply-run.ts:672`
  - body: The fast-lane path isolates the queue key and command lane, but still passes the parent sessionKey into runReplyAgent. Current main creates ReplyOperation from that key and rejects it while the parent run is active, so the fast-lane message cannot reliably run immediately.
  - confidence: 0.88
- **[P2] Enforce tool suppression with the runtime flag:** `src/auto-reply/reply/get-reply-run.ts:647-649`

  - confidence: 0.86
