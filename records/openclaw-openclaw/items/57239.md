## Summary

Keep open; the proposed active-run fast lane is not safe to merge yet because source review shows it still uses the parent reply-operation key and relies on prompt-only tool suppression, and the external PR lacks real behavior proof.

## What This Changes

The branch adds a fast-lane auto-reply path for short messages during active runs, a force-run-now queue-policy flag, transient session/lane handling, cleanup, and queue-policy tests.

## Best Possible Solution

Land a maintainer-approved concurrency design that isolates reply identity, preserves queued-turn ordering and session state, enforces hard tool controls for quick turns, and includes regression plus real channel proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: during an active normal run, an eligible fast-lane message still carries the parent sessionKey into runReplyAgent, while current main rejects a second ReplyOperation for that active key.

## Solution Assessment

No. A fast lane may be a valid direction, but this patch is not the best mergeable solution until reply-operation identity, ordering, state isolation, and tool-safety contracts are enforced in code rather than prompt guidance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use an isolated reply operation key:** `src/auto-reply/reply/get-reply-run.ts:672`
  - body: The fast-lane path isolates the queue key and embedded session lane, but still passes the parent sessionKey into runReplyAgent. Current main uses that key to create the ReplyOperation and rejects it while the parent run is active, so the fast-lane message cannot reliably run immediately.
  - confidence: 0.88
- **[P2] Disable tools with the runtime flag:** `src/auto-reply/reply/get-reply-run.ts:647-649`
  - body: The fast-lane turn only appends a system prompt asking the model not to call tools. Because the embedded runner already exposes a hard disableTools flag and this path does not set it, an eligible quick message can still invoke privileged tools concurrently with the parent run.
  - confidence: 0.86
