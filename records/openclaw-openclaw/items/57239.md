## Summary

Keep open. The PR targets a real latency problem, but the current branch is not mergeable as-is because its fast-lane path still collides with current main's active reply-operation identity and relies on prompt-only tool suppression for a concurrent run.

## What This Changes

Adds a fast-lane path for short active-session follow-ups using transient run/lane state, a queue-policy override, sessionLaneKey plumbing, transcript cleanup, and queue-policy tests.

## Best Possible Solution

A mergeable design should add a maintainer-approved bounded fast lane or reuse the existing side-question model, with isolated reply identity, hard tool controls, ordering guarantees, and real channel proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence failure path: an eligible fast-lane message still passes the parent sessionKey to runReplyAgent, and current main rejects a second active ReplyOperation for that key.

## Solution Assessment

No. The fast-lane idea may be useful, but this implementation is not the best mergeable solution until reply-operation identity, hard tool controls, ordering/state isolation, and real behavior proof are addressed.
