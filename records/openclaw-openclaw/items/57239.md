## Summary

Keep open. The fast-lane feature is not implemented on current main, and this PR still has merge-blocking correctness and security issues around reply-operation identity and hard tool suppression, plus missing real behavior proof.

## What This Changes

The PR adds a fast-lane auto-reply path for short follow-up messages during active runs by adding queue-policy override support and transient lane/session handling.

## Best Possible Solution

Land a maintainer-approved concurrency design that isolates reply identity, preserves queued-turn ordering and session state, enforces hard tool controls for quick turns, and includes regression plus real channel proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: during an active parent run, an eligible fast-lane message still passes the parent sessionKey into runReplyAgent, while current main rejects a second ReplyOperation for that active key.

## Solution Assessment

No. A fast lane may be a useful direction, but this branch is not the best mergeable solution until reply-operation identity, ordering, state isolation, and tool-safety contracts are enforced in code rather than prompt guidance.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Use an isolated reply operation key:** `src/auto-reply/reply/get-reply-run.ts:672`
  - body: The fast-lane path isolates the queue key and command lane, but still passes the parent sessionKey into runReplyAgent. Current main creates ReplyOperation from that key and rejects it while the parent run is active, so the fast-lane message cannot reliably run immediately.
