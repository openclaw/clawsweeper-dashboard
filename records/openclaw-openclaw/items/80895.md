## Summary

Keep open. The PR is the right narrow direction for the linked billing-copy bug, but the current patch only derives the billing auth mode from an auth profile id, so profileless OAuth/token credentials can still fall back to API-key top-up copy.

## What This Changes

The PR makes Anthropic billing-shaped assistant error copy auth-mode aware and threads auth profile mode through formatter, failover, lifecycle, subscription, and payload paths.

## Best Possible Solution

Thread one unified resolved billing auth mode through all user-facing billing-error formatting surfaces, preserving API-key copy for API-key credentials and neutral recovery copy for OAuth/token credentials.

## Reproduction Assessment

Yes at source level. Current main formats billing-shaped errors through API-key-only copy, and the linked issue gives a concrete Anthropic Max OAuth path; I did not live-run Max OAuth or Telegram in this review.

## Solution Assessment

No as written. The central formatter/context-propagation approach is right, but it should carry resolved credential mode, not only auth-profile type, so profileless OAuth/token routes are covered too.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Carry non-profile auth modes into billing copy:** `src/agents/pi-embedded-runner/run.ts:1489-1491`
  - body: This derives the billing copy mode only from `attemptAuthProfileStore.profiles[lastProfileId]`. When the active credential came from `ResolvedProviderAuth.mode` without a profile id, such as `ANTHROPIC_OAUTH_TOKEN` or Claude CLI synthetic OAuth/token auth, `lastProfileId` is undefined and the formatter still receives `undefined`, preserving the API-key/top-up message this PR is meant to remove. Please fall back to the resolved auth mode and cover a profileless OAuth/token case.
  - confidence: 0.83
