## Summary

Keep open. Current main has several adjacent primitives for cron, task tracking, subagent yielding, plugin next-turn injections, and scheduled session turns, but it still lacks the requested general runtime-level post-response async action queue with persisted callback/result delivery to the originating conversation.

## What This Changes

Request a first-class OpenClaw runtime API that lets an agent acknowledge a turn, enqueue asynchronous work, and later send or resume with the confirmed result.

## Best Possible Solution

Design this as a product/API extension that reuses the existing task ledger, cron/scheduled-turn machinery, subagent/session wake semantics, and channel delivery contracts instead of adding a separate background-job stack.

## Reproduction Assessment

Not applicable as a feature request. Current source/docs show related primitives but no existing documented post-response enqueue contract that can be reproduced as broken behavior.

## Solution Assessment

Unclear until maintainers choose the permanent API boundary. The safest direction is to extend existing task and continuation primitives, not add a parallel queue with competing callback semantics.
