## Summary

Keep open. Current main has related primitives for subagents, cron, task status tracking, and specialized background work, but not the requested general post-response enqueue API that persists arbitrary work after a visible assistant response and later delivers or resumes the originating conversation.

## What This Changes

Request a runtime-level post-response async action queue that can acknowledge immediately, persist job state, execute later, and deliver or resume the result.

## Best Possible Solution

Design this as a first-class continuation API that reuses the existing task ledger, cron/subagent/session wake semantics, and channel delivery contracts instead of adding a separate queue with competing status and callback behavior.

## Reproduction Assessment

Not applicable as a feature request. Source and docs show existing cron, task, and subagent continuation primitives, but no current-main general post-response enqueue API to reproduce as broken behavior.

## Solution Assessment

Unclear until maintainers choose the product/API shape. The best direction appears to be extending existing task and continuation primitives rather than introducing a parallel background-job system.
