## Summary

Keep open. Current main has several specialized continuation primitives, but not the requested general post-response enqueue API that persists arbitrary work after a visible assistant response and later delivers or resumes the originating conversation.

## What This Changes

The issue requests a runtime-level post-response async action queue that can persist background work after an assistant reply and later send or resume with the result.

## Best Possible Solution

Design this as one first-class continuation contract that reuses existing task, flow, session wake, and channel delivery semantics instead of adding a parallel queue.

## Reproduction Assessment

Not applicable. This is a feature request, and current source/docs show specialized continuation primitives rather than a broken existing API to reproduce.

## Solution Assessment

Unclear until maintainers choose the product/API shape. Extending existing task/session continuation primitives appears safer than introducing a separate background-job stack.
