## Summary

Keep open. Current main has task-ledger, cron, subagent, and generated-media continuation primitives, but not the requested general post-response enqueue API with a persisted callback/result contract for arbitrary agent work.

## What This Changes

Request for a first-class runtime queue that lets an agent acknowledge a turn, persist post-response async work, and later deliver or resume with the confirmed result.

## Best Possible Solution

Design one continuation contract that reuses existing task records, session wake/steering, and channel delivery semantics, with explicit status, cancellation, idempotency, and callback rules.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing contract. Current source inspection shows related primitives but no general post-response enqueue path to reproduce as broken behavior.

## Solution Assessment

Unclear until maintainers choose the API shape. The maintainable direction is to extend existing task, wake, and delivery primitives rather than adding a parallel queue with competing status and callback behavior.
