## Summary

This remains necessary as the canonical tracker: current main and v2026.6.5 still provide only whole-tool allow/deny policy, while the related confirmation-gate issue addresses a broader but distinct approval workflow. The requested authorization boundary needs maintainer product and security design before implementation.

## What This Changes

Requests action-aware tool policy so operators can block mutating actions such as message sends while retaining read-only actions.

## Best Possible Solution

Define one typed, documented action-policy contract enforced at the shared pre-execution boundary for both ordinary agent calls and Gateway invocation, with explicit precedence and no implicit privileged-workflow bypass.

## Reproduction Assessment

Yes, at source level: built-in policy admits or removes the complete tool by name, while the action becomes available only at the later pre-execution boundary. This establishes the reported structural limitation without a runtime reproduction.

## Solution Assessment

No, not exactly as proposed: action-aware enforcement is the right direction, but a generic shared authorization rule is safer than allowing Lobster or clawd.invoke to bypass a deny implicitly.
