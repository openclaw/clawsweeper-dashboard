## Summary

Keep open. Current main still enforces tool policy at whole-tool granularity, while the requested `message:send` or `gateway:config.patch` granularity remains a valid core authorization design item; the closed gateway reports point back here rather than replacing it.

## What This Changes

The issue requests action-aware tool policy so operators can deny actions such as `message:send` while preserving other actions on the same multiplexed tool.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware policy contract enforced before multiplexed tool actions execute, with regression coverage for `message` and `gateway` read/write splits.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new policy grammar. Source inspection on current main gives high confidence that enforcement still filters whole tool names before tool-call actions are read.

## Solution Assessment

Unclear until maintainer design is done. Action-level enforcement is the right direction for the reported safety gap, but syntax, wildcard behavior, precedence, Gateway HTTP behavior, and privileged workflow semantics need product/security approval before implementation.
