## Summary

Keep open. Current main still lacks a generic action-aware `tools.allow`/`tools.deny` contract; the newer message action allowlist partially covers message-only allowlisting, but it does not close the broader canonical request for `message:send`, `gateway:config.patch`, and privileged invocation semantics.

## What This Changes

The issue requests action-level tool policy so operators can deny side-effecting actions such as `message:send` while preserving read-only actions on the same multiplexed tool.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract enforced before multiplexed side effects execute.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for new authorization grammar. Source inspection gives high confidence that current main still applies shared tool policy by whole tool name before multiplexed actions are read.

## Solution Assessment

Unclear until maintainer design is done. The action-level direction fits the reported safety gap, but the maintainable path is a central policy contract rather than a message-only or gateway-only check.
