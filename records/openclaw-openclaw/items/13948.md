## Summary

Keep open. Current main and the latest release have a message-specific `tools.message.actions.allow` allowlist, but the requested generic action-level `tools.allow`/`tools.deny` contract for multiplexed tools such as `message` and `gateway` is still missing and remains security/product design work.

## What This Changes

Request action-level tool policy so operators can deny side-effecting actions such as `message:send` while preserving read-only actions on the same tool.

## Best Possible Solution

Keep this as the canonical core tracker for a typed, documented action-aware tool-policy contract enforced before multiplexed tool actions execute and covered for message and gateway read/write splits.

## Reproduction Assessment

Yes, from source inspection. Generic tool policy still filters by tool name while multiplexed tools dispatch on an `action` argument later, so action-qualified `tools.deny` entries cannot express the requested policy today.

## Solution Assessment

Unclear on exact syntax. A generic pre-execution action-aware authorization contract is the maintainable layer, but the colon grammar, nested maps, wildcard behavior, precedence, and trusted bypass semantics need maintainer design.
