## Summary

Keep open. Current main and v2026.6.10 still lack the requested generic action-level `tools.allow`/`tools.deny` contract; the existing message action allowlist is message-specific and does not cover deny syntax, `gateway`, or privileged approval workflow semantics.

## What This Changes

Request action-level tool policy so operators can deny side-effecting actions like `message:send` while keeping read/search/react actions available.

## Best Possible Solution

Keep this as the canonical core tracker for a maintainer-approved, typed, documented action-aware tool-policy contract with clear grammar, wildcard behavior, precedence, and pre-execution enforcement for multiplexed tools.

## Reproduction Assessment

Yes, as a source-level gap. Current main filters generic tool policy by whole tool name before action arguments are read, so action-qualified entries such as `message:send` or `gateway:config.patch` cannot express the requested policy today.

## Solution Assessment

Unclear until maintainers choose the security model. The maintainable path is a generic action-aware policy grammar and enforcement boundary, not a one-off message or gateway shim.
