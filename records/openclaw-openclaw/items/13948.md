## Summary

Keep open. Current main still enforces tool policy at whole-tool granularity, and the related gateway reports are already closed back to this issue as the canonical security-sensitive authorization design tracker.

## What This Changes

Request action-level tool allow/deny policy so actions such as `message:send` or `gateway:config.patch` can be denied while other actions on the same multiplexed tool remain available.

## Best Possible Solution

Keep this as the canonical core tracker and define a typed, documented action-aware tool-policy contract with shared pre-dispatch enforcement for multiplexed tools such as `message` and `gateway`.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new policy grammar. Source inspection on current main gives high confidence that existing enforcement filters by whole tool name only.

## Solution Assessment

Unclear until maintainer design is done. Action-level enforcement is the right direction for the reported safety gap, but syntax, wildcard behavior, precedence, Gateway HTTP behavior, and privileged workflow semantics need product/security approval before implementation.
