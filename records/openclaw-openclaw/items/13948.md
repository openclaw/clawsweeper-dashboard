## Summary

Keep open. Current main and the latest release partially address the message side with `tools.message.actions.allow`, but the requested generic action-level `tools.allow`/`tools.deny` contract and trusted approval/bypass semantics for multiplexed tools such as `gateway` are still missing.

## What This Changes

Request action-level tool policy so actions such as `message:send` or `gateway:config.patch` can be blocked while read-only actions remain available.

## Best Possible Solution

Keep this as the canonical core tracker for a typed, documented action-aware tool-policy contract that composes with sender/group/provider policy and trusted approval paths before multiplexed tool actions execute.

## Reproduction Assessment

Yes. Source inspection shows generic tool policy still filters by tool name while multiplexed tools dispatch on an `action` argument later, so action-qualified `tools.deny` entries cannot express the requested policy today.

## Solution Assessment

Unclear on the exact syntax. A generic pre-execution action-aware authorization contract is the maintainable layer, but the colon grammar, nested maps, wildcard behavior, precedence, and trusted approval bypass semantics need maintainer design.
