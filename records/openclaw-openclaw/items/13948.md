- `impact:security`: The issue is about enforcing authorization boundaries so agents cannot perform outbound or mutating actions that policy intends to block.

## Summary

Keep open: current main and the latest release still only provide whole-tool `tools.allow`/`tools.deny` for generic policy; the newer `tools.message.actions.allow` surface is a useful message-specific partial workaround, but it does not implement action-aware deny for `gateway` or define privileged-send semantics.

## What This Changes

Request action-level tool policy so mutating actions such as `message:send` and `gateway:config.patch` can be denied while read/read-only actions remain available.

## Best Possible Solution

Keep this as the canonical core tracker and define a typed, documented action-aware tool-policy contract with explicit syntax, precedence, enforcement points, and privileged workflow semantics for `message`, `gateway`, and future multiplexed tools.

## Reproduction Assessment

Yes, at source level. Current main's generic policy schema and matcher operate on tool names, while `message` and `gateway` read action arguments inside their multiplexed execution paths.

## Solution Assessment

Unclear until maintainer design is done. A generic action-aware policy contract is the maintainable direction; the existing message-only allowlist is useful but does not settle gateway actions, precedence, or privileged workflow semantics.
