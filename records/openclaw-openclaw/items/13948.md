## Summary

Keep open. Current main has partial mitigations through `tools.toolsBySender` and `tools.message.actions.allow`, but the requested generic action-qualified `tools.deny` contract and Gateway action split are still missing and remain security-sensitive product design work.

## What This Changes

The issue requests action-level deny rules for multiplexed tools so write actions like message send or gateway config.patch can be blocked while read actions remain available.

## Best Possible Solution

Define a maintainer-approved typed action-aware tool-policy contract with clear grammar, wildcard behavior, precedence, docs, and tests for `message` and `gateway`, including explicit privileged workflow semantics.

## Reproduction Assessment

Yes, by source inspection. Current main can enforce whole-tool policy and a message-only action allowlist, but it still cannot express generic action-qualified deny rules such as `message:send` or `gateway:config.patch` through the shared `tools.deny` contract.

## Solution Assessment

Unclear until maintainers choose the policy grammar and bypass semantics. The maintainable direction is a central action-aware policy contract, not another one-off message or Gateway workaround.
