## Summary

Keep open. Current main still implements whole-tool policy only, and prior comments correctly identify this as the canonical tracker for action-aware policy across multiplexed tools such as `message` and `gateway`.

## What This Changes

The issue requests action-level tool policy so operators can deny specific multiplexed actions such as `message:send` while keeping read-only actions available.

## Best Possible Solution

Keep this as the canonical core tracker and define a typed, documented action-aware tool-policy contract that is enforced before multiplexed `message` and `gateway` actions execute.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new policy grammar. Source inspection gives high confidence that current main lacks action-aware tool policy and filters only by whole tool name.

## Solution Assessment

No automated patch is the best next step yet. The maintainable solution needs maintainer product/security design for grammar, wildcard behavior, precedence, Gateway HTTP behavior, and privileged workflow semantics.
