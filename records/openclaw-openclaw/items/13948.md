## Summary

Keep open. Current main partially covers message-only action allowlisting, but the requested generic action-level `tools.allow`/`tools.deny` contract and trusted approval/bypass semantics for multiplexed tools such as `gateway` are still missing.

## What This Changes

The issue requests action-level tool policy so operators can deny unsafe write actions such as `message:send` while preserving read/search/react actions.

## Best Possible Solution

Keep this as the canonical core tracker for a typed action-aware authorization contract with documented grammar, precedence, trusted approval context, and regression coverage for message and gateway read/write splits.

## Reproduction Assessment

Yes. Source inspection shows generic tool policy still filters by tool name while `message` and `gateway` dispatch on an `action` argument later, so `tools.deny: ["message:send"]` or `tools.deny: ["gateway:config.patch"]` cannot currently express the requested policy.

## Solution Assessment

No for the proposed implicit bypass or ad hoc colon syntax alone. The maintainable path is a typed pre-execution authorization contract with explicit trusted approval semantics, using the existing message action allowlist only as the message-specific piece.
