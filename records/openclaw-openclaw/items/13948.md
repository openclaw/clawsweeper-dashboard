## Summary

Keep open: current main and v2026.6.5 now include a message-specific `tools.message.actions.allow` allowlist, but the requested generic action-level deny contract remains missing for `tools.allow`/`tools.deny`, Gateway actions, and privileged workflow semantics.

## What This Changes

The issue requests config-enforced action-level tool policy so operators can block specific mutating actions like message sends while preserving read/search/react access.

## Best Possible Solution

Define a maintainer-approved, typed action-aware tool-policy contract that covers generic policy grammar, precedence, `message` and `gateway` enforcement, and explicit privileged workflow semantics while preserving the shipped message action allowlist if it remains compatible.

## Reproduction Assessment

Yes, source-level: current main still filters generic tool policy by tool name, while `message` and `gateway` read `action` only inside their multiplexed execution paths. The shipped message-specific action allowlist proves part of the requested behavior but not the generic deny contract.

## Solution Assessment

Unclear until maintainer design is done. A generic action-aware policy contract is the maintainable direction; extending only the current message action allowlist would not settle gateway actions, policy precedence, or approved-send bypass semantics.
