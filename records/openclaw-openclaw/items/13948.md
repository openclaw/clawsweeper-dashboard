## Summary

Keep open. Current main still applies tool policy at whole-tool granularity, while `message` and `gateway` are multiplexed action tools; the related gateway reports were closed back to this as the canonical tracker, and the remaining work is a security-relevant product/API design change.

## What This Changes

Request action-level tool policy so operators can block actions like `message:send` or `gateway:config.patch` while keeping read-only actions available.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with clear grammar, wildcard behavior, precedence, privileged workflow semantics, and regression coverage for `message` and `gateway` action splits.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new policy grammar. Source inspection gives high confidence that current main lacks action-aware policy and filters by whole tool name only.

## Solution Assessment

Unclear until maintainer design is done. Action-level enforcement is the right direction for the reported safety gap, but the exact syntax, precedence, and privileged workflow bypass semantics need product/security approval before implementation.
