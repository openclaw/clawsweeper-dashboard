## Summary

Current main still admits or removes tools only by tool name, while `message` and `gateway` select materially different actions after admission; this remains the canonical open security/product-design tracker with no implementation PR.

## What This Changes

Add action-level tool authorization so operators can retain read-only actions while mechanically denying outbound or privileged actions.

## Best Possible Solution

Create one typed generic action-policy contract, enforce it immediately before multiplexed actions execute, document deterministic precedence and wildcard behavior, and represent approved workflows through explicit capabilities rather than bypassing policy.

## Reproduction Assessment

Yes at source level. Current main filters the available catalog using only the tool name, then `message` and `gateway` evaluate their action arguments inside the already-admitted tool execution.

## Solution Assessment

No, not exactly as proposed. Action-aware authorization is the appropriate direction, but an approved workflow should receive an explicit capability under the same policy system rather than bypassing an action deny.
