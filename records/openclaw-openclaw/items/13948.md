## Summary

Keep open. Current main and the latest tagged release still use whole-tool/name-based policy for the generic tool contract; `tools.message.actions.allow` is a useful message-specific mitigation, but it does not provide the requested generic action-level deny grammar, gateway action split, or privileged workflow semantics.

## What This Changes

The issue requests action-level tool deny policy so operators can block mutating actions such as message sends or gateway config writes while preserving read-only actions on the same multiplexed tool.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with explicit syntax, wildcard behavior, precedence, enforcement point, and privileged workflow semantics before changing multiplexed tool authorization.

## Reproduction Assessment

Yes, at source level. Current policy types/schema and filtering operate on tool names, while `message` and `gateway` read action arguments only inside their multiplexed tool execution paths.

## Solution Assessment

Unclear until maintainer design is done. The likely right direction is a generic action-aware policy contract; the existing message-specific `actions.allow` setting is only a partial mitigation and does not settle gateway or privileged-send semantics.
