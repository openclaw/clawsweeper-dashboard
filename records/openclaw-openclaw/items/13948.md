## Summary

Keep open. Current main and the latest release only partially cover the message side with `tools.message.actions.allow`; the generic `tools.allow`/`tools.deny` contract is still whole-tool only, and `gateway` remains a single multiplexed tool with action dispatch after policy admission.

## What This Changes

Request generic action-level tool policy so operators can block side-effecting actions like message sends while keeping read-only actions on the same multiplexed tool available.

## Best Possible Solution

Keep this as the canonical core tracker for a typed, documented action-aware tool-policy contract that composes with sender/group/provider policy and trusted approval paths before multiplexed actions execute.

## Reproduction Assessment

Yes. Source inspection shows generic tool policy still filters by tool name while multiplexed tools dispatch on an `action` argument later, so action-qualified `tools.deny` entries cannot express the requested policy today.

## Solution Assessment

Unclear on the exact syntax. A generic pre-execution action-aware authorization contract is the maintainable layer, but grammar, wildcard matching, precedence, and trusted bypass semantics need maintainer security/product design.
