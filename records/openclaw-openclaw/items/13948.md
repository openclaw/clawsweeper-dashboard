## Summary

Keep open. Current main still enforces tool policy at whole-tool inventory admission while `message` and `gateway` remain multiplexed action tools, so this remains the canonical core authorization design tracker.

## What This Changes

Request action-aware tool policy so operators can deny outbound actions such as `message` sends or `gateway` config writes while keeping read-only actions available.

## Best Possible Solution

Define a typed, documented action-aware tool-policy contract with explicit grammar, wildcard and precedence rules, Gateway HTTP behavior, and privileged workflow semantics, then enforce it before multiplexed action execution.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new policy grammar. Source inspection gives high confidence that current main lacks action-aware tool policy and filters only by whole tool name.

## Solution Assessment

No automated patch is the best next step yet. The maintainable solution needs maintainer product/security design for grammar, precedence, Gateway HTTP behavior, and privileged workflow semantics before implementation.
