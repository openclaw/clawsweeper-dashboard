## Summary

Keep open. Current main still enforces tool policy by whole tool name while `message` and `gateway` remain multiplexed action tools, and the linked gateway-specific reports were closed back to this as the canonical action-level authorization tracker.

## What This Changes

The issue requests action-level allow/deny policy so operators can block actions such as `message:send` or `gateway:config.patch` without disabling the whole tool.

## Best Possible Solution

Keep this as the canonical core tracker and define a typed, documented action-aware tool-policy contract with clear grammar, wildcard and precedence rules, Gateway HTTP behavior, privileged workflow semantics, and regression coverage for `message` and `gateway` read/write splits.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new policy grammar. Source inspection gives high confidence that current main lacks action-aware tool policy and filters by whole tool name only.

## Solution Assessment

Unclear until maintainer design is done. Action-level enforcement is the right direction for the reported safety gap, but the exact syntax, precedence, and privileged workflow bypass semantics need product/security approval before implementation.
