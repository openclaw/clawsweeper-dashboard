## Summary

Keep open. Current main still enforces tool policy at whole-tool granularity, and the existing discussion already treats this as the canonical tracker for action-aware authorization on multiplexed tools.

## What This Changes

Request action-level tool policy so operators can deny mutating actions such as message sends or gateway config writes while keeping read-only actions available.

## Best Possible Solution

Define a maintainer-approved, typed, documented action-aware tool-policy contract with explicit grammar, wildcard behavior, precedence, and regression coverage for message and gateway read/write splits.

## Reproduction Assessment

Yes for the current limitation by source inspection: tools.deny is matched only against tool names, denyActions is not in the strict schema, and message/gateway actions are read after policy admission. This is a feature gap rather than a broken documented behavior.

## Solution Assessment

Unclear. The requested direction is valid, but the maintainable fix should be a generic action-policy contract with explicit precedence and privileged-workflow semantics, not a one-off exception for message or gateway.
