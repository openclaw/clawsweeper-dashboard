## Summary

Keep open. Current main has a message-specific action allowlist, but the requested generic action-level deny contract is still missing: core tool policy remains whole-tool/name based, Gateway HTTP policy still filters by tool name, and related gateway/message authorization reports are closed back to this as the canonical tracker.

## What This Changes

Request action-aware tool policy so operators can deny mutating actions such as message sends or gateway config changes while preserving read/lookup actions.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with explicit precedence, wildcard behavior, enforcement point, and privileged workflow semantics before changing multiplexed tool authorization.

## Reproduction Assessment

Yes, at source level. Current policy types/schema and filtering still operate on tool names, while message and gateway read action arguments only inside their multiplexed tool execution paths.

## Solution Assessment

Unclear until maintainer design is done. The likely right direction is a generic action-aware policy contract with documented syntax and precedence; the existing message-specific actions.allow setting is only a partial mitigation and does not settle gateway or privileged-send semantics.
