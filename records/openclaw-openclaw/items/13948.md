## Summary

Keep open. Current main has partial mitigations through `tools.toolsBySender` and `tools.message.actions.allow`, but the generic action-qualified deny contract requested here is still missing and related gateway/message reports have been closed back to this as the canonical tracker.

## What This Changes

Request action-level tool policy so operators can deny write/send actions such as `message:send` or `gateway:config.patch` while preserving read/search actions.

## Best Possible Solution

Define a maintainer-approved typed action-aware tool-policy contract with clear grammar, wildcard behavior, precedence, docs, and regression coverage for `message` and `gateway`, including explicit privileged workflow semantics.

## Reproduction Assessment

Yes, by source inspection. Current main can enforce whole-tool policy and a message-only action allowlist, but it still cannot express generic action-qualified deny rules such as `message:send` or `gateway:config.patch` through the shared `tools.deny` contract.

## Solution Assessment

Unclear until maintainers choose the policy grammar and bypass semantics. The maintainable direction is a central action-aware policy contract, not another one-off message or Gateway workaround.
