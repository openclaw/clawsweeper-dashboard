## Summary

Keep open: current main and v2026.6.5 still expose allow/deny as whole-tool policy only, so the requested action-level control remains a real core authorization gap and this issue is still the canonical tracker for the related gateway/message reports.

## What This Changes

Request action-aware tool policy so operators can deny actions such as message sends or gateway config writes while preserving read-only actions on the same multiplexed tool.

## Best Possible Solution

Define one typed action-aware tool-policy contract in core, with documented grammar, wildcard and precedence rules, and enforcement before multiplexed message and gateway actions execute.

## Reproduction Assessment

Yes for the source-level gap: current main's config schema and policy matcher only accept whole tool names, and message/gateway actions are read after inventory filtering. This is a feature request rather than a failing bug reproduction.

## Solution Assessment

Unclear on the exact proposed syntax and bypass path: action-level policy is the right core boundary, but grammar, precedence, and privileged workflow semantics need maintainer design before a fix PR.
