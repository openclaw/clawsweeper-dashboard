## Summary

Keep open. Current main still enforces tool policy at whole-tool granularity: the config contract and runtime schema expose only allow/alsoAllow/deny/profile, the matcher receives only a tool name, and the message and gateway tools dispatch mutating/read-only actions after policy admission. The related gateway-specific reports are already closed back to this as the canonical tracker, so this remains valid core authorization design work.

## What This Changes

The issue requests action-level tool allow/deny policy so operators can block mutating actions such as message send or gateway config writes while preserving read-only actions and approved workflow paths.

## Best Possible Solution

Keep this as the canonical core tracker and define a maintainer-approved, typed, documented action-aware tool-policy contract enforced before multiplexed tool actions execute, with regression coverage for message and gateway read/write splits.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: tools.deny with message removes the whole message tool, while tools.deny with message:send is compared only to the tool name and does not block the send action; denyActions is not accepted by the strict schema.

## Solution Assessment

Unclear. The requested capability is the right direction, but the maintainable fix should be a generic action-policy contract with explicit precedence and privileged-workflow semantics, not a one-off message or gateway special case.
