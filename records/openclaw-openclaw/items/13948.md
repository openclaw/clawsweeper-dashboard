## Summary

Keep open: current main and the latest release still expose generic tool policy at whole-tool granularity, while the requested action-aware authorization for multiplexed `message` and `gateway` actions remains missing and security-sensitive.

## What This Changes

The issue requests action-level allow/deny policy so mutating actions such as `message:send` or `gateway:config.patch` can be blocked while read/search actions remain available.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with explicit syntax, precedence, enforcement point, and privileged workflow semantics for `message`, `gateway`, and future multiplexed tools.

## Reproduction Assessment

Yes, at source level. Current main's policy types/schema and filtering operate on tool names, while `message` and `gateway` read action arguments only inside their multiplexed execution paths.

## Solution Assessment

Unclear until maintainer design is done. A generic action-aware policy contract looks like the maintainable direction; extending the existing message-only allowlist would not settle gateway actions, precedence, or privileged workflow bypass semantics.
