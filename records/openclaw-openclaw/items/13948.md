## Summary

Keep open. Current main still enforces tool policy by whole tool name before multiplexed tool actions are read, and the issue remains the canonical tracker for an action-aware authorization contract rather than something already implemented or superseded.

## What This Changes

The issue requests action-level tool policy so operators can deny actions like `message:send` or `gateway:config.patch` while preserving safer actions on the same tool.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract enforced before multiplexed tool side effects execute.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for new policy grammar. Source inspection gives high confidence that current main still filters whole tool names before tool-call actions are read.

## Solution Assessment

Unclear until maintainer design is done. Action-level enforcement addresses the reported safety gap, but syntax, precedence, wildcard behavior, Gateway HTTP behavior, and privileged workflow semantics need product/security approval first.
