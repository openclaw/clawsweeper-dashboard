## Summary

Keep open. Current main and the latest release have partial adjacent controls, including per-agent/global `message.actions.allow` and sender-scoped whole-tool policy, but the shared `tools.allow`/`tools.deny` contract is still name-only and cannot express generic action-level deny for multiplexed tools such as `message` or `gateway`.

## What This Changes

Request to add action-aware tool policy so operators can block specific actions like `message:send` while keeping read/search actions available.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with clear allow/deny grammar, wildcard behavior, precedence, trusted action metadata, and regression coverage for `message` and `gateway` read/write splits.

## Reproduction Assessment

Yes. Source inspection shows `tools.deny` entries are matched against normalized tool names only, so an entry like `message:send` does not deny the `message` tool's `send` action, while `tools.deny: ["message"]` removes the whole tool.

## Solution Assessment

No, the current partial message action allowlist is not the full requested solution. The maintainable fix is a central action-aware policy contract rather than one-off message or gateway checks.
