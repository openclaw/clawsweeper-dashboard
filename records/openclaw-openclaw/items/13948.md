## Summary

Keep open. Current main still enforces tool policy at whole-tool granularity, while `message` and `gateway` remain multiplexed tools that choose mutating/read-only actions only after policy admission; related gateway-specific reports are already closed back to this issue as the canonical tracker.

## What This Changes

Add action-level tool policy so operators can deny mutating actions such as `message:send` while preserving read-only message or gateway actions.

## Best Possible Solution

Keep this as the canonical core tracker and define a typed, documented action-aware tool-policy contract enforced before multiplexed tool actions execute, with regression coverage for `message` and `gateway` read/write splits.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `tools.deny: ["message"]` removes the whole tool, `tools.deny: ["message:send"]` is compared only to the registered tool name, and `denyActions` is rejected by the strict runtime schema.

## Solution Assessment

Unclear. Action-aware policy is the right end state, but the exact syntax, precedence, wildcard behavior, and privileged workflow semantics need maintainer product/security design before implementation.
