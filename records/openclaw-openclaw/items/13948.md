## Summary

Keep open. Current main still lacks the requested generic action-level tool deny contract; the existing `tools.message.actions.allow` is a message-specific allowlist, while shared policy matching and Gateway HTTP tool exposure still operate at whole-tool granularity.

## What This Changes

The issue requests config-level action-aware tool policy such as denying `message:send` or `gateway:config.patch` while keeping read-only actions available.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with explicit syntax, wildcard behavior, precedence, enforcement point, and privileged workflow semantics for `message`, `gateway`, and future multiplexed tools.

## Reproduction Assessment

Yes, at source level. Current main's generic policy types/schema and filtering operate on tool names, while `message` and `gateway` read action arguments only inside their multiplexed execution paths.

## Solution Assessment

Unclear until maintainer design is done. The likely right direction is a generic action-aware policy contract; extending the current message-specific allowlist alone would not settle gateway actions or privileged-send semantics.
