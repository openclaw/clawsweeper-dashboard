## Summary

Keep open. Current main still enforces generic tool policy at whole-tool-name granularity; adjacent `tools.message.actions.allow` and `tools.toolsBySender` do not implement the requested generic action-level deny contract, and related gateway/sender reports point back here as the canonical authorization-policy tracker.

## What This Changes

Request action-level tool allow/deny policy so operators can block actions such as `message:send` while preserving read/react/search and approved send workflows.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware policy contract that composes with message action allowlists, sender policy, and Gateway direct invocation.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config contract request. Source inspection gives high confidence that current main cannot express action-qualified entries in generic `tools.allow` or `tools.deny`.

## Solution Assessment

Unclear until maintainer product/security design is done. A central action-aware policy contract is the maintainable direction, but syntax, precedence, wildcard behavior, Gateway HTTP behavior, and privileged workflow semantics need approval first.
