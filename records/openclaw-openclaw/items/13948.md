## Summary

Keep open. Current main and v2026.6.8 still lack the requested generic action-level `tools.allow`/`tools.deny` contract; the existing message-specific action allowlist only covers part of the chat use case and does not solve `gateway` or future multiplexed tools.

## What This Changes

Request generic action-level tool policy so operators can block side-effecting actions like `message:send` or `gateway:config.patch` while keeping read-only actions available.

## Best Possible Solution

Keep this as the canonical core tracker for a typed, documented action-aware tool-policy contract enforced before multiplexed action dispatch, with explicit precedence, wildcard, and trusted-bypass semantics.

## Reproduction Assessment

Yes, from source inspection. Generic tool policy still filters by tool name while `message` and `gateway` dispatch on an `action` argument later, so action-qualified `tools.deny` entries cannot express the requested policy today.

## Solution Assessment

Unclear on exact syntax. A generic pre-execution action-aware authorization contract is the maintainable layer, but the colon grammar, nested maps, wildcard behavior, precedence, and trusted bypass semantics need maintainer design.
