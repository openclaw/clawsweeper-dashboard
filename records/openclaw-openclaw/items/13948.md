## Summary

Keep open. Current main still applies tool policy at whole-tool granularity, while `message` and `gateway` are multiplexed action tools; this remains the canonical core tracker for a security-relevant policy/API design change.

## What This Changes

Request action-aware tool policy so operators can deny actions such as `message:send` while preserving read/search/react access to the same multiplexed tool.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with clear grammar, wildcard behavior, precedence, privileged workflow semantics, and regression coverage for `message` and `gateway` action splits.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new policy grammar. Source inspection on current main gives high confidence that the existing implementation filters by whole tool name only.

## Solution Assessment

Unclear until maintainer design is done. Action-level enforcement is the right direction for the reported safety gap, but the exact syntax, precedence, and privileged workflow bypass semantics need product/security approval before implementation.
