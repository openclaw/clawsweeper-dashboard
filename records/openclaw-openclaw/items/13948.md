## Summary

Keep open: current main and v2026.6.5 only have a message-specific `tools.message.actions.allow` partial mitigation; the requested generic action-level allow/deny policy for `tools.deny`, gateway actions, and privileged workflow semantics is still missing.

## What This Changes

The issue requests config-level action granularity so operators can deny mutating actions such as `message:send` while preserving read-oriented actions.

## Best Possible Solution

Define a maintainer-approved, typed action-aware tool-policy contract covering generic grammar, precedence, `message` and `gateway` enforcement, docs, tests, and explicit privileged workflow semantics.

## Reproduction Assessment

Yes, source-level: current main still filters generic tool policy by tool name, while `message` and `gateway` read `action` only inside their multiplexed execution paths.

## Solution Assessment

Unclear until maintainer design is done. A generic action-aware policy contract is the maintainable direction; extending only the current message action allowlist would not settle gateway actions, policy precedence, or approved-send bypass semantics.
