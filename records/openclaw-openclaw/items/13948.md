## Summary

Keep open. Current main still applies tool policy at whole-tool granularity, while `message` and `gateway` remain multiplexed action tools; this is valid core authorization design work, not stale cleanup.

## What This Changes

The issue requests action-level allow/deny policy such as denying `message:send` while allowing read-only message actions.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with clear grammar, wildcard behavior, precedence, privileged workflow semantics, and regression coverage for `message` and `gateway` action splits.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new config and policy grammar. Source inspection on current main gives high confidence that existing policy admission still filters by whole tool name only.

## Solution Assessment

Unclear until maintainer design is done. Action-level enforcement is the right direction for the reported safety gap, but exact syntax, precedence, wildcard behavior, and privileged workflow bypass semantics need product/security approval before implementation.
