## Summary

Keep open. Current main still applies tool policy at whole-tool granularity while `message` and `gateway` are multiplexed action tools, and the related gateway-specific reports are already closed back to this issue as the canonical core authorization-design tracker.

## What This Changes

Request action-level tool policy so operators can deny actions such as `message:send` or `gateway:config.patch` while keeping safer actions available.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with clear grammar, wildcard behavior, precedence, privileged workflow semantics, and regression coverage for `message` and `gateway` action splits.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new policy grammar. Source inspection on current main gives high confidence that existing policy filtering is whole-tool only and happens before multiplexed actions dispatch.

## Solution Assessment

Unclear until maintainer design is done. Action-level enforcement is the right direction for the reported safety gap, but syntax, precedence, wildcard behavior, and privileged workflow bypass semantics need product/security approval before implementation.
