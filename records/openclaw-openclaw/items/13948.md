## Summary

Keep open. Current main and the latest release still lack a generic action-level `tools.allow`/`tools.deny` contract; the existing message-specific action allowlist does not cover gateway actions or the requested deny semantics, and the remaining work is core security/config product design.

## What This Changes

The issue requests action-level allow/deny policy for multiplexed tools such as `message:send` and `gateway:config.patch` while keeping read-style actions available.

## Best Possible Solution

Keep this as the canonical core tracker and define a maintainer-approved, typed action-aware tool-policy contract with documented grammar, wildcard behavior, precedence, and regression coverage for message and gateway read/write splits.

## Reproduction Assessment

Yes. Source inspection shows generic tool policy still filters whole tool names before action arguments are read, so action-qualified entries such as `message:send` cannot express the requested policy today.

## Solution Assessment

No current implementation is the full solution. `tools.message.actions.allow` can restrict message actions, but the best fix needs a generic maintainer-approved policy grammar and pre-execution enforcement boundary for multiplexed tools.
