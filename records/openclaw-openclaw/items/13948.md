## Summary

Keep open. Current main has adjacent sender-scoped policy and message-specific action allowlists, but it still lacks the requested generic action-level `tools.allow`/`tools.deny` contract, and the issue is already marked for product/security review.

## What This Changes

Request action-level tool policy so operators can deny actions such as `message:send` or `gateway:config.patch` while leaving read-only actions available.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware policy contract that composes with message action allowlists, sender policy, and Gateway direct invocation.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config contract request. Source inspection gives high confidence that current main cannot express action-qualified entries in generic `tools.allow` or `tools.deny`.

## Solution Assessment

Unclear until maintainer design is done. A central action-aware policy contract is the maintainable direction, but syntax, precedence, wildcard behavior, Gateway HTTP behavior, and privileged workflow semantics need product/security approval first.
