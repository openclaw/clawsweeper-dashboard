## Summary

Keep open. Current main partially addresses the message side with `tools.message.actions.allow` and sender-scoped whole-tool policy, but the requested general action-level `tools.allow`/`tools.deny` contract and trusted approval/bypass semantics for multiplexed tools such as `gateway` are still missing.

## What This Changes

Request action-aware tool policy so operators can block side-effecting actions such as message send while retaining read/search/react access and routing approved sends through a privileged workflow.

## Best Possible Solution

Keep this as the canonical core tracker for a typed action-aware authorization contract with documented grammar, precedence, trusted approval context, and regression coverage for message and gateway read/write splits.

## Reproduction Assessment

Yes. Source inspection shows message actions can now be allowlisted, but shared tool policy still filters by tool name and gateway remains a single action-dispatching tool, so generic action-level deny cannot currently be expressed.

## Solution Assessment

No for the proposed implicit bypass or ad hoc colon syntax alone. The maintainable path is a typed pre-execution authorization contract with explicit trusted approval semantics, using the existing message action allowlist only as the message-specific piece.
