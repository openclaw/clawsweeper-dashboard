## Summary

Keep open. Current main still enforces allow/deny at whole-tool granularity, and maintainer/context already treats this as the canonical tracker for action-aware tool policy across `message` and `gateway`.

## What This Changes

Request action-level tool policy so operators can deny mutating actions such as `message:send` or `gateway:config.patch` while preserving read-only actions.

## Best Possible Solution

Keep this as the canonical core tracker and define a typed, documented action-aware tool-policy contract enforced before multiplexed tool actions execute, with regression coverage for `message` and `gateway` read/write splits.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `tools.deny: ["message"]` removes the whole `message` tool, `tools.deny: ["message:send"]` is compared only to the tool name and does not block the `send` action, and `denyActions` is rejected by the strict schema.

## Solution Assessment

Unclear. The requested capability is the right direction, but the maintainable fix should be a generic action-policy contract with explicit precedence and privileged-workflow semantics, not a one-off `message`/`gateway` exception.
