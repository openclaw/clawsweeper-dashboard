## Label Justifications

- `P2`: This is a valid security hardening feature for controlled deployments, but it is not an emergency outage or confirmed universal bypass.
- `impact:security`: The issue concerns mechanically enforcing authorization for side-effecting communication and administrative actions.

## Summary

Current main still lacks declarative action-aware tool policy, and the live discussion confirms this issue remains the canonical tracker. The plugin hook offers a useful enforcement seam but not a complete operator-facing substitute, while trusted approval bypass semantics require product and security ownership.

## What This Changes

Add action-level allow/deny enforcement so selected actions of multiplexed tools can be blocked while other actions remain available.

## Best Possible Solution

Define a typed core authorization decision at the existing pre-execution boundary over tool identity, normalized action, sender/session context, and an explicit trusted approval capability, with one documented precedence model shared by message, gateway, HTTP invocation, and plugin tools.

## Reproduction Assessment

Yes. Current main’s strict schema and matcher operate on whole tool names, while message and gateway evaluate their action arguments only inside execution, so the requested declarative split cannot currently be expressed.

## Solution Assessment

No for the proposed implicit workflow bypass and untyped colon syntax. The capability belongs at the shared pre-execution authorization boundary with typed actions, explicit precedence, and an unforgeable approval context.
