## Summary

Keep open. Current main has adjacent sender-scoped whole-tool policy and a message-only `tools.message.actions.allow` allowlist, but the generic `tools.allow`/`tools.deny` contract is still whole-tool-name based and cannot express action-qualified entries such as `message:send` or `gateway:config.patch`.

## What This Changes

The issue requests action-level allow/deny policy so operators can block specific actions inside multiplexed tools while keeping other actions available.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with grammar, wildcard behavior, precedence, direct-invoke semantics, and coverage for `message` and `gateway` read/write splits.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config contract request. Source inspection against current main gives high confidence that generic `tools.allow` and `tools.deny` cannot express action-qualified entries today.

## Solution Assessment

Unclear until maintainer product/security design is done. A central action-aware policy contract is the maintainable direction, but syntax, precedence, wildcard behavior, Gateway HTTP behavior, and privileged workflow semantics need approval first.
