## Summary

Keep open. Current main still enforces tool policy at whole-tool admission while message and gateway remain multiplexed action tools, and prior discussion already treats this as the canonical action-level authorization design tracker.

## What This Changes

Request action-level tool policy so operators can deny actions such as message send or gateway config.patch while leaving safer actions available.

## Best Possible Solution

Define a typed, documented action-aware tool-policy contract with grammar, wildcard and precedence rules, Gateway HTTP behavior, and privileged workflow semantics, then enforce it before multiplexed action execution.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new policy grammar. Source inspection gives high confidence that current main lacks action-aware tool policy and filters only by whole tool name.

## Solution Assessment

Unclear until maintainer design is done. Action-level enforcement is the right product direction, but the exact syntax and any privileged workflow bypass need security/product decisions before implementation.
