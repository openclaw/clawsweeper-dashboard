## Summary

Keep open. Current main still lacks a generic action-level tool policy contract; `tools.message.actions.allow` is a useful message-specific mitigation, but it does not implement the requested `tools.deny` action grammar, gateway action split, or privileged workflow semantics.

## What This Changes

The issue requests config-level action deny/allow policy so operators can block mutating actions like `message:send` while preserving read-only actions like `message:read`.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with explicit syntax, wildcard behavior, precedence, enforcement point, and privileged workflow semantics for `message`, `gateway`, and future multiplexed tools.

## Reproduction Assessment

Yes, at source level. Current generic policy types/schema and filtering operate on tool names, while `message` and `gateway` read action arguments only inside their multiplexed execution paths.

## Solution Assessment

Unclear until maintainer design is done. The likely right direction is a generic action-aware policy contract; extending the current message-specific allowlist alone would not settle gateway actions or privileged-send semantics.
