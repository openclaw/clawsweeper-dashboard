## Summary

Keep open. Current main still supports only whole-tool allow/deny policy, while `message` and `gateway` remain multiplexed tools whose `action` arguments are read after name-level policy admission; related #46684 and #54260 are already closed back to this as the canonical tracker.

## What This Changes

Request action-level tool policy so operators can deny actions such as `message:send` while keeping read-only or lower-risk actions available.

## Best Possible Solution

Define and document a maintainer-approved, typed action-aware tool-policy contract, then enforce it before multiplexed `message` and `gateway` actions execute with regression coverage for read/write splits.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new policy grammar. Source inspection gives high confidence that current main lacks action-aware tool policy and filters only by tool name.

## Solution Assessment

No automated patch is the best next step yet. The maintainable solution needs maintainer product/security design for grammar, precedence, wildcard behavior, Gateway HTTP behavior, and privileged workflow semantics.
