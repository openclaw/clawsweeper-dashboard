## Summary

Source inspection confirms the issue still describes a real gap: the shared tool policy contract and matcher are whole-tool only, while `message` and `gateway` remain multiplexed tools whose `action` is evaluated after inventory admission. Current main has partial mitigations through `toolsBySender`, gateway write guards, and a message-only `actions.allow`, but it does not implement the requested generic action-qualified deny contract.

## What This Changes

The issue requests a config-level way to deny specific actions within multiplexed tools, such as `message:send` or `gateway:config.patch`, while keeping other actions available.

## Best Possible Solution

Define a maintainer-approved typed action-aware tool-policy contract with grammar, wildcard behavior, precedence, docs, and regression coverage for `message` and `gateway`, including explicit privileged workflow semantics.

## Reproduction Assessment

Yes, by source inspection. Current main can enforce whole-tool policy and a message-only action allowlist, but it still cannot express generic action-qualified deny rules such as `message:send` or `gateway:config.patch` through the shared `tools.deny` contract.

## Solution Assessment

Unclear until maintainers choose the policy grammar and bypass semantics. The maintainable direction is a central action-aware policy contract, not another one-off message or Gateway workaround.
