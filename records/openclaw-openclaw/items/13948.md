## Summary

Keep open. Current main has partial mitigations such as `toolsBySender` and message-only `actions.allow`, but the shared tool policy still cannot express generic action-qualified denies like `message:send` or `gateway:config.patch`; this remains core authorization design work.

## What This Changes

The issue requests action-level tool deny rules so operators can block specific multiplexed tool actions while keeping other actions on the same tool available.

## Best Possible Solution

Define a central typed action-aware tool-policy contract with grammar, wildcard behavior, precedence, docs, and regression coverage for multiplexed tools before changing enforcement.

## Reproduction Assessment

Yes, by source inspection. Current main can enforce whole-tool policy and a message-only action allowlist, but it cannot express shared action-qualified deny rules such as `message:send` or `gateway:config.patch` through `tools.deny`.

## Solution Assessment

Unclear until maintainers choose the policy grammar and bypass semantics. The maintainable direction is a central action-aware authorization contract, not another per-tool workaround.
