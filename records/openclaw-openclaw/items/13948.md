## Summary

Keep open. Current main still applies tool policy at whole-tool granularity, while `message` and `gateway` remain multiplexed action tools; the related gateway reports were already closed back to this issue as the canonical core authorization-design tracker.

## What This Changes

The issue requests action-level tool policy, such as denying `message:send` while still allowing read-only message actions.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with shared enforcement for multiplexed tools such as `message` and `gateway`.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new policy grammar. Source inspection on current main gives high confidence that existing enforcement filters by whole tool name only.

## Solution Assessment

Unclear until maintainer design is done. Action-level enforcement is the right direction for the reported safety gap, but syntax, precedence, wildcard behavior, Gateway HTTP behavior, and privileged workflow semantics need product/security approval before implementation.
