## Summary

Keep open. Current main and the latest release still enforce tool policy at whole-tool inventory level, while `message` and `gateway` remain single multiplexed tools whose `action` arguments are read only after the tool is admitted; related gateway/message authorization reports are closed back to this as the canonical tracker, and no open PR currently closes it.

## What This Changes

The issue requests action-aware tool policy so operators can deny mutating actions such as `message:send` or `gateway:config.patch` while preserving read-only actions.

## Best Possible Solution

Keep this as the canonical core tracker and define a typed, documented action-aware tool-policy contract with clear grammar, wildcard behavior, precedence, and enforcement before multiplexed `message` and `gateway` actions execute.

## Reproduction Assessment

Yes, at source level. Current config types/schema and docs expose whole-tool policy only, runtime filtering uses `tool.name`, and `message`/`gateway` dispatch on `action` after policy admission.

## Solution Assessment

Unclear until maintainer design is done. The likely right direction is an action-aware policy contract with documented syntax and precedence, not an ad hoc privileged bypass around the existing whole-tool matcher.
