## Summary

Keep open: current main and v2026.6.5 still enforce whole-tool allow/deny only, so action-level denial for multiplexed message and gateway actions remains missing and this issue is still the canonical tracker for the related reports.

## What This Changes

The issue requests action-level tool policy so operators can deny send/write actions while keeping read-style actions available for multiplexed tools like message and gateway.

## Best Possible Solution

Define one typed core action-aware tool-policy contract with documented grammar, wildcard and precedence rules, and enforcement before multiplexed message and gateway actions execute.

## Reproduction Assessment

Yes for the source-level gap: current main's config schema and policy matcher only accept whole tool names, and message/gateway actions are read after inventory filtering. This is a feature request rather than a failing bug reproduction.

## Solution Assessment

Unclear on the exact syntax and bypass path: action-level policy belongs in the core authorization boundary, but grammar, precedence, and privileged workflow semantics need maintainer design before a fix PR.
