## Summary

Keep open. Current main still enforces tool policy at whole-tool granularity, and this issue remains the canonical tracker for action-aware policy; the adjacent sender/DM authorization issues are related but do not supersede the requested `message:send` / `gateway:config.patch` style contract.

## What This Changes

The issue requests action-level tool allow/deny policy so multiplexed tools such as `message` can block `send` while preserving read/search/react actions.

## Best Possible Solution

Keep this issue open as the canonical core tracker and design a typed, documented action-aware tool-policy contract enforced before multiplexed tool side effects execute.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new policy grammar. Source inspection on current main gives high confidence that enforcement still filters whole tool names before tool-call actions are read.

## Solution Assessment

Unclear until maintainer design is done. Action-level enforcement is the right direction for the reported safety gap, but syntax, wildcard behavior, precedence, Gateway HTTP behavior, and privileged workflow semantics need product/security approval before implementation.
