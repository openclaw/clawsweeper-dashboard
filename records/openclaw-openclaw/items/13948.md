## Summary

Keep open. Current main still lacks a generic action-level tool policy: allow/deny policy is typed and enforced by tool name, while `message` and `gateway` remain multiplexed tools whose `action` arguments are handled after policy admission; the newer message action allowlist is only a partial, message-specific mitigation.

## What This Changes

The issue requests an action-aware tool policy so operators can deny mutating actions such as `message` sends or `gateway` config writes while preserving read and lookup actions.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with explicit grammar, wildcard behavior, precedence, enforcement point, and privileged workflow semantics before changing multiplexed tool authorization.

## Reproduction Assessment

Yes, at source level. Current config types/schema and policy filtering operate on tool names, while `message` and `gateway` action arguments are read only inside their multiplexed execution paths.

## Solution Assessment

Unclear until maintainer design is done. The likely right direction is a generic action-aware policy contract with documented syntax and precedence; the current `tools.message.actions.allow` setting is only a partial message-specific mitigation.
