## Summary

Keep open. Current main and v2026.6.10 still lack generic action-aware tool allow/deny; the newer message-only action allowlist is only a partial workaround and does not cover generic policy grammar, gateway actions, or trusted approval-bypass semantics.

## What This Changes

The issue requests action-level tool policy so operators can deny specific multiplexed tool actions such as `message:send` while keeping read/search actions available.

## Best Possible Solution

Keep this as the canonical core tracker for a maintainer-approved, typed action-aware tool-policy contract with documented grammar, wildcard behavior, precedence, and pre-execution enforcement for multiplexed tools.

## Reproduction Assessment

Yes, as a source-level gap: current main filters generic tool policy by whole tool name before action arguments are read, so entries such as `message:send` or `gateway:config.patch` cannot express the requested policy today. I did not run tests because this was a read-only review.

## Solution Assessment

Unclear until maintainers choose the security model. The maintainable path is a generic action-aware policy grammar and enforcement boundary, not a one-off message or gateway shim; the existing message action allowlist is useful but incomplete for this request.
