## Summary

Keep open. Current main still supports only whole-tool policy, while `message` and `gateway` remain multiplexed tools that read `action` only after name-level policy admission; related #46684 and #54260 are closed back to this as the canonical tracker, so this remains core authorization design work.

## What This Changes

Request action-level tool allow/deny rules such as `message:send` or `gateway:config.patch` so risky actions can be blocked while read-only actions remain available.

## Best Possible Solution

Define and document a maintainer-approved action-aware tool-policy contract enforced before multiplexed actions execute, with regression coverage for `message` and `gateway` read/write splits.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new policy grammar. Source inspection gives high confidence that current main lacks action-aware tool policy and only filters by tool name.

## Solution Assessment

No automated patch is the best next step yet. The maintainable solution needs maintainer product/security design for grammar, precedence, wildcard behavior, and privileged workflow semantics before code changes.
