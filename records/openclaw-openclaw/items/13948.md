## Summary

Keep open. Current main still enforces tool policy at whole-tool granularity, while this request asks for a new action-aware authorization/config contract for multiplexed tools like `message` and `gateway`; related gateway-specific reports are already closed back to this as the canonical tracker.

## What This Changes

The issue requests action-level allow/deny policy, such as denying `message:send` or `gateway:config.patch` while keeping other actions on the same tool available.

## Best Possible Solution

Keep this as the canonical core tracker and define a typed, documented action-aware tool-policy contract enforced before multiplexed tool actions execute, with regression coverage for `message` and `gateway` read/write splits.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new policy grammar. Source inspection on current main gives high confidence that enforcement still filters whole tool names before tool-call actions are read.

## Solution Assessment

Unclear until maintainer design is done. Action-level enforcement is the right direction for the reported safety gap, but syntax, wildcard behavior, precedence, Gateway HTTP behavior, and privileged workflow semantics need product/security approval before implementation.
