## Summary

Keep open. Current main and v2026.6.8 still lack a generic action-level `tools.allow`/`tools.deny` contract; the message-only action allowlist is partial and does not cover deny syntax, `gateway`, or privileged workflow semantics.

## What This Changes

The issue requests action-level tool policy so operators can block specific multiplexed actions such as `message:send` or `gateway:config.patch` while keeping read-only actions available.

## Best Possible Solution

Keep this as the canonical core tracker and define a typed, documented action-aware tool-policy contract with clear grammar, wildcard behavior, precedence, and pre-execution enforcement for multiplexed tools.

## Reproduction Assessment

Yes, as a source-level gap. Current main filters generic tool policy by whole tool name before action arguments are read, so action-qualified entries such as `message:send` or `gateway:config.patch` cannot express the requested policy today.

## Solution Assessment

Unclear until maintainers choose the security model. The current message action allowlist is a useful partial control, but the best fix is a generic policy grammar and pre-execution enforcement boundary rather than a one-off message or gateway shim.
