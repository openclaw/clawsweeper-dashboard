## Summary

Keep open. Current main and the latest release still enforce tool policy at whole-tool inventory level, while `message` and `gateway` read their `action` arguments only inside single multiplexed tools, so the requested action-level safety control remains unimplemented and needs maintainer product/security design.

## What This Changes

The issue requests config-level action-aware tool policy so operators can deny mutating actions such as `message` send while keeping read/search actions available.

## Best Possible Solution

Define a typed, documented action-aware tool-policy contract with clear grammar, wildcard behavior, precedence, and enforcement before multiplexed `message` and `gateway` actions execute.

## Reproduction Assessment

Yes, at source level. Current config types/schema and runtime policy filtering only operate on tool names, while `message` and `gateway` dispatch on `action` after policy admission.

## Solution Assessment

Unclear until maintainer design is done. The likely right direction is a typed action-aware policy contract, but exact syntax, precedence, wildcard semantics, and privileged workflow bypass behavior need product/security approval first.
