## Summary

Keep open. Current main has partial mitigations through sender-scoped whole-tool policy and `tools.message.actions.allow`, but the requested generic action-qualified `tools.deny` contract, Gateway action split, and privileged workflow semantics are still missing.

## What This Changes

Request action-level tool policy so operators can deny specific multiplexed tool actions such as `message:send` while keeping read/search actions available.

## Best Possible Solution

Define a maintainer-approved typed action-aware tool-policy contract with clear grammar, wildcard behavior, precedence, docs, and tests for `message` and `gateway`, including explicit privileged workflow semantics.

## Reproduction Assessment

Yes, by source inspection. Current main can enforce whole-tool policy and a message-only action allowlist, but it still cannot express generic action-qualified deny rules such as `gateway:config.patch` through the shared `tools.deny` contract.

## Solution Assessment

Unclear until maintainers choose the policy grammar and bypass semantics. The maintainable direction is a central action-aware policy contract, not another one-off message or Gateway workaround.
