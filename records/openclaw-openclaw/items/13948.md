## Summary

Keep open. Current main partially covers adjacent cases through message-specific action allowlists and sender-scoped whole-tool policy, but the requested generic action-level deny contract for `tools.allow`/`tools.deny` is still missing and remains security-sensitive product design work.

## What This Changes

Request action-level tool policy so operators can deny actions such as message sends or gateway config writes while preserving read-only actions on the same multiplexed tool.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract that composes with existing message action allowlists and sender-scoped whole-tool policy while covering multiplexed `message` and `gateway` actions.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config contract request. Source inspection gives high confidence that current main cannot express action-qualified entries in generic `tools.allow` or `tools.deny`.

## Solution Assessment

Unclear until maintainer design is done. A central action-aware policy contract is the right direction, but syntax, wildcard behavior, precedence, Gateway direct-invoke behavior, and privileged workflow bypass semantics need product/security approval before implementation.
