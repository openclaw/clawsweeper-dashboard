## Summary

This remains necessary as the canonical action-level authorization tracker: current main and v2026.6.5 still admit tools by name before multiplexed `message` or `gateway` actions are known, while nearby confirmation-gate and read-only-profile work solves different problems. The capability fits OpenClaw’s security direction but needs explicit maintainer product and security design.

## What This Changes

Add action-aware allow/deny controls for multiplexed tools so read actions can remain available while sensitive actions such as sending are mechanically blocked.

## Best Possible Solution

Implement a single typed action-aware core authorization contract with explicit wildcard and precedence rules plus a narrowly defined trusted-caller capability, enforced before every multiplexed action handler executes.

## Reproduction Assessment

Yes. Current main’s supported policy schema is whole-tool only, its matcher receives only tool names, and `message`/`gateway` action arguments are evaluated only after admission, so action-specific denial cannot be expressed or enforced.

## Solution Assessment

The capability is appropriate, but neither proposed syntax nor a blanket Lobster bypass is yet the best proven design; maintainers should first define one generic action-policy grammar, precedence model, and trusted-caller boundary.
