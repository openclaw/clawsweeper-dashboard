## Summary

Keep open. Current main partially addresses the message-specific part through `tools.message.actions.allow` and sender-scoped whole-tool policy, but the requested generic action-level tool policy contract is still missing from `tools.allow`/`tools.deny`, the shared matcher, Gateway invoke, and multiplexed `gateway` policy behavior.

## What This Changes

The issue requests action-level allow/deny policy for multiplexed tools so actions like `message:send` or `gateway:config.patch` can be blocked while safer actions remain available.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract that composes with existing message action allowlists and sender-scoped policy while covering multiplexed tools such as message and gateway.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config contract request. Source inspection gives high confidence that current main can enforce message-specific action allowlists, but generic tools.allow/tools.deny still filter whole tool names and cannot express action-qualified policy for gateway or privileged invoke semantics.

## Solution Assessment

Unclear until maintainer design is done. A central action-aware policy contract is the right direction, but syntax, precedence, wildcard behavior, direct Gateway HTTP behavior, and privileged workflow bypass semantics need product/security approval before implementation.
