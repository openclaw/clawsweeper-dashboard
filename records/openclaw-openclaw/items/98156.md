## Label Justifications

- `P2`: The report describes a real session-continuity failure with a narrow provider-error and abort timing trigger, not a universal runtime outage.
- `impact:session-state`: The failure mode loses active conversational context across an embedded session reset.

## Summary

Keep open: current main still has the abort-settle timeout and embedded session-takeover path from the report, while the existing reset-transcript recovery code is scoped to `/new` and `/reset` memory capture rather than embedded takeover replacement sessions.

## What This Changes

The issue asks OpenClaw to preserve conversational context when embedded abort/takeover resets a session after provider errors and an abort settle timeout.

## Best Possible Solution

Define a maintainer-approved session-recovery contract that connects takeover replacement sessions to the preserved transcript or retry context using the existing session/transcript store, without adding sidecar state or a broad new fallback stack.

## Reproduction Assessment

Yes at source level, not as a live repro: current main contains the same abort-settle timeout, takeover detection, and command-only reset recovery surfaces described in the report. A high-confidence live repro would still need the provider-limit error loop and webchat abort timing.

## Solution Assessment

Unclear: preserving session continuity is the right outcome, but copying last messages into a new prompt is only one possible repair. A cleaner fix may attach reset archive metadata, prevent unnecessary takeover, or route recovery through the context engine/session lifecycle boundary.
