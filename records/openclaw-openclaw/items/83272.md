## Summary

Keep open: current main has fixed the account-wide serial drain tick wedge and now surfaces stale backlog health, but a never-settling spooled `bot.handleUpdate` still leaves the active lane claimed and blocks later same-lane delivery after the Telegram offset has advanced.

## What This Changes

The issue asks for a durable Telegram isolated-ingress fix so spooled updates cannot remain wedged after the worker advances the Telegram update offset.

## Best Possible Solution

Add an owner-approved bounded recovery path for stuck spooled `handleUpdate` tasks that preserves Telegram lane ordering, avoids duplicate dispatch, keeps offset/spool durability intact, and retains clear stale-backlog diagnostics.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main still has an active same-lane spooled handler that can await `bot.handleUpdate` forever, and the tests already model a wedged backlog. I did not run live Telegram in this read-only review.

## Solution Assessment

No, current main is only a partial solution: lane concurrency and health diagnostics are the right foundation, but they do not recover a never-settling same-lane handler. The next step should be a maintainer-approved timeout/requeue/delete policy with focused regression and live Telegram proof when feasible.
