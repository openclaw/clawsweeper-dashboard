## Summary

Keep open: current main fixed the worst serial drain-tick wedge and added offset/health mitigations, but it still has no bounded recovery for a never-settling spooled Telegram `handleUpdate`, leaving the reported same-lane ingress hang unresolved.

## What This Changes

The issue reports that isolated Telegram ingress can advance past a spooled update, fail to dispatch it into the session, and then hang future delivery because spooled `handleUpdate` work has no liveness bound.

## Best Possible Solution

Add an owner-reviewed bounded recovery path for stuck claimed Telegram spool handlers that preserves same-lane ordering and durable spool semantics, with focused polling-session regression coverage and live Telegram proof when feasible.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: make the polling-session test harness return a never-settling `handleUpdate` promise for a spooled update, then observe that current main pins the lane and only marks polling unhealthy. I did not run live Telegram in this read-only review.

## Solution Assessment

No, current main is only a partial solution: lane concurrency, offset persistence, and stale-health diagnostics are useful, but they do not bound or recover the stuck handler. The safer next step is a deliberate timeout and claim-recovery policy for claimed spooled updates.
