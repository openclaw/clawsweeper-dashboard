## Summary

Keep open: current main has already fixed the worst serial drain-tick wedge and added stale-backlog health diagnostics, but it still does not bound or recover a never-settling spooled `bot.handleUpdate` task. That leaves a meaningful remaining Telegram ingress reliability bug rather than a clean implemented-on-main close.

## What This Changes

The issue requests source-level recovery for isolated Telegram ingress so spooled updates cannot wedge after the Bot API offset advances.

## Best Possible Solution

Add a bounded recovery path for stuck spooled `handleUpdate` tasks in the Telegram polling session that preserves same-lane ordering, avoids duplicate dispatch, keeps offset/spool semantics durable, and has focused regression coverage plus live Telegram proof when feasible.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: v2026.5.12 awaited `bot.handleUpdate` under the isolated drain guard, and current main still has an unbounded per-lane handler task that can be made never to settle in the existing polling-session test harness. I did not run a live Telegram setup in this read-only pass.

## Solution Assessment

No, current main is only a partial fix: the lane-concurrency and stale-health changes are the right foundation, but they do not implement the reported handler deadline/recovery semantics. The safer next step is an owner-reviewed timeout and claim-recovery design rather than closing this as already implemented.
