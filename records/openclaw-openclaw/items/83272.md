## Summary

Keep open: current main fixed the release's account-wide serial drain wedge and added stale-backlog health reporting, but a spooled Telegram handler can still remain unbounded and block later same-lane ingress after the worker offset has advanced. That is a real remaining delivery/recovery policy issue, not a clean implemented-on-main close.

## What This Changes

The issue reports isolated Telegram polling advancing the offset after spooling while an unbounded spooled `handleUpdate` can hang, leaving inbound updates stuck until recovery or restart.

## Best Possible Solution

Add a bounded recovery path for stuck spooled Telegram handlers that preserves same-lane ordering, avoids duplicate dispatch where possible, keeps offset/spool durability coherent, and includes focused regression coverage plus live Telegram proof when feasible.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: make the current polling-session test harness use a never-settling spooled `handleUpdate`, and later same-lane spooled updates remain blocked while polling can still look alive or unhealthy. I did not run a live Telegram setup in this read-only pass.

## Solution Assessment

No, current main is only a partial solution. The lane-concurrency and stale-health changes are the right foundation, but the durable fix still needs an owner-reviewed timeout/requeue/delete policy for stuck spooled handlers.
