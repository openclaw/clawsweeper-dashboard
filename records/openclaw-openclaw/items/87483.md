## Summary

Keep open: current `main` and v2026.6.8 still keep `hold-exceeded` locks report-only, and the linked fixing PR is open and unmerged. The issue should stay open until maintainers land or reject that PR's `maxHoldMs` reclaim semantics.

## What This Changes

The issue reports stale session JSONL write-lock files that remain held past `maxHoldMs` by a live gateway process, causing later requests to fail with `session file locked` until manual cleanup or restart.

## Best Possible Solution

Keep this issue open while maintainers review the linked fixing PR as the canonical path for enforcing the holder's recorded hard hold limit across processes.

## Reproduction Assessment

Yes from source, not from a live overnight run: current main detects `hold-exceeded` when a lock exceeds the holder-recorded `maxHoldMs`, but the removal callback refuses to remove stale locks whose reasons are entirely report-only.

## Solution Assessment

Unclear until maintainers choose the lock-reclaim contract. The linked PR is the narrow source-level fix if `maxHoldMs` is intended to be a hard cross-process reclaim deadline; otherwise the safer solution would need an explicit opt-in policy.
