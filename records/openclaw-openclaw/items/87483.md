## Summary

Keep open: current main and the latest release still keep `hold-exceeded` in the report-only stale-lock set, and https://github.com/openclaw/openclaw/pull/89673 is the open linked candidate fix with closing syntax for this issue.

## What This Changes

The issue reports that live gateway session write-lock files can remain held far past `session.writeLock.maxHoldMs`, causing later session writes to fail until manual cleanup or restart.

## Best Possible Solution

Keep this issue open while maintainers review and either land or reject https://github.com/openclaw/openclaw/pull/89673 as the canonical maxHoldMs reclaim path; track same-process self-contention separately if it remains after that decision.

## Reproduction Assessment

Yes from source, but not from a live overnight run: current main detects `hold-exceeded` for locks older than the holder-recorded `maxHoldMs`, then refuses to remove locks whose stale reasons are all report-only. The linked PR adds the focused live-owner regression test for that path.

## Solution Assessment

Unclear until maintainers choose the lock-reclaim contract. The linked PR is the narrowest fix if `maxHoldMs` should be a hard cross-process reclaim deadline; otherwise the safer direction is an explicit opt-in policy rather than changing the default meaning.
