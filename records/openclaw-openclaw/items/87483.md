## Summary

Keep open: current main still treats `hold-exceeded` session locks as report-only, and the open linked fix PR is the active path for the reported maxHoldMs reclaim behavior.

## What This Changes

The issue reports stale session transcript lock files held past `session.writeLock.maxHoldMs`, causing later requests to fail with `session file locked` until manual cleanup or restart.

## Best Possible Solution

Keep this issue open while maintainers review https://github.com/openclaw/openclaw/pull/89673 as the canonical maxHoldMs reclaim fix, then close it only after that PR or an equivalent policy-approved fix lands.

## Reproduction Assessment

Yes, source-reproducible: current main detects `hold-exceeded` when a lock exceeds its recorded `maxHoldMs`, then refuses to remove locks whose stale reasons are all report-only. I did not establish a live overnight gateway reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the lock-reclaim contract. The linked PR is the narrow fix if `maxHoldMs` is intended to be a hard cross-process reclaim deadline; otherwise the safer path is an explicit opt-in reclaim policy.
