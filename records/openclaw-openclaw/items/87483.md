## Summary

Keep open: current main and the latest release still treat maxHoldMs-only `hold-exceeded` locks as report-only, and the open linked fixing PR is the canonical path that should be reviewed before this issue closes.

## What This Changes

The issue reports session JSONL write-lock files remaining held past `maxHoldMs` by a live gateway process, causing later session writes to fail with lock timeouts.

## Best Possible Solution

Keep this issue open while maintainers review and either land or reject https://github.com/openclaw/openclaw/pull/89673 as the canonical maxHoldMs live-holder reclaim fix; track self-contention separately if it remains.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main detects `hold-exceeded` for locks older than the holder-recorded `maxHoldMs`, then refuses to remove locks whose stale reasons are all report-only. I did not run a live overnight gateway reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the lock-reclaim contract. The linked PR is the narrow owner-boundary fix if `maxHoldMs` should be a hard cross-process reclaim deadline; otherwise the safer path is an explicit opt-in policy instead of changing the default meaning.
