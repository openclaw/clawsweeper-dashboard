Keep open: current main and the latest release still treat `hold-exceeded` session locks as report-only, and the open linked fix PR is the active implementation path rather than merged provenance.

## What This Changes

The issue reports session JSONL write locks persisting past their recorded hold limit and blocking later requests until manual cleanup or gateway restart.

## Best Possible Solution

Review and either land or explicitly reject https://github.com/openclaw/openclaw/pull/89673 as the canonical fix for live-holder locks that exceed their recorded `maxHoldMs`, while keeping broader lock-scope reports separate.

## Reproduction Assessment

No high-confidence live current-main reproduction was run for the overnight gateway case. Source inspection does show the unresolved path: current main detects `hold-exceeded` but keeps that reason report-only, and the linked PR adds the deterministic regression path.

## Solution Assessment

Unclear until maintainers choose the lock-reclaim contract. The linked PR is the narrow source-level fix if `maxHoldMs` should be a hard cross-process reclaim deadline, but that is a session-state policy decision.
