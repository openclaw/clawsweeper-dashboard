## Summary

Keep open: current main and latest release still keep `hold-exceeded` as a report-only stale-lock reason, and the open linked PR is the active candidate fix for the reported maxHoldMs reclaim behavior. The new self-contention comment is useful adjacent evidence, but it does not make this issue obsolete.

## What This Changes

The issue reports that session JSONL write-lock files can remain held past `maxHoldMs` by a live gateway process, causing later session writes to fail with `session file locked`.

## Best Possible Solution

Keep this issue open while maintainers review https://github.com/openclaw/openclaw/pull/89673 as the canonical maxHoldMs reclaim fix; close it only after that PR or an equivalent accepted fix lands.

## Reproduction Assessment

Yes, source-level: current main detects `hold-exceeded` after the holder-recorded `maxHoldMs`, then refuses to remove the lock when that stale reason is report-only. I did not live-reproduce the overnight macOS gateway stall.

## Solution Assessment

Unclear until maintainers decide the lock contract. The linked PR is the narrow best fix if `maxHoldMs` is meant to be a hard cross-process reclaim deadline; otherwise the safer end state is an explicit opt-in reclaim policy or a different migration story.
