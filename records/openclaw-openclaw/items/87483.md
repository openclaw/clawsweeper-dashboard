## Summary

Keep open: current main and v2026.6.9 still treat maxHoldMs-only `hold-exceeded` locks as report-only, and the open linked fixing PR is the canonical path before this issue can close.

## What This Changes

The issue reports that session write-lock files can persist past `maxHoldMs`/`staleMs`, leaving later session writes blocked with `session file locked` errors.

## Best Possible Solution

Keep this issue open while maintainers review and either land or reject https://github.com/openclaw/openclaw/pull/89673 as the canonical maxHoldMs live-holder reclaim fix; track self-contention separately if it remains.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a lock older than the holder-recorded `maxHoldMs` gets `hold-exceeded`, then current main refuses removal because that reason is report-only; I did not run a live overnight gateway reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the lock-reclaim contract. The linked PR is the narrow owner-boundary fix if `maxHoldMs` should be a hard cross-process reclaim deadline; otherwise the safer path is an explicit opt-in policy instead of changing the default meaning.
