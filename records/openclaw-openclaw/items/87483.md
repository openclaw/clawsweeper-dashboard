## Summary

Keep open: current main and latest release still detect `hold-exceeded` locks but keep that stale reason report-only, and the linked open PR is the active fix path rather than merged provenance.

## What This Changes

The issue reports live gateway session `.jsonl.lock` files persisting past `maxHoldMs`/`staleMs`, causing later requests to fail with `session file locked` until manual cleanup or restart.

## Best Possible Solution

Review and either land or explicitly reject https://github.com/openclaw/openclaw/pull/89673 as the canonical fix for live-holder locks that exceed their recorded `maxHoldMs`, while tracking broader lock-scope/message-loss reports separately.

## Reproduction Assessment

No live high-confidence current-main reproduction is available. Source inspection does show the unresolved path: current main records `hold-exceeded` for overdue live-holder locks but refuses to remove locks whose stale reasons are only report-only.

## Solution Assessment

Unclear until maintainers accept the concurrency policy. The linked PR is a narrow fix for the source-level gap, but it deliberately broadens `maxHoldMs` from holder-watchdog-only behavior to cross-process contender reclaim of unchanged live-holder locks.
