## Summary

Keep open: current main and `v2026.6.8` still treat `hold-exceeded` as report-only, and the open linked PR is the active candidate fix for the reported `maxHoldMs` reclaim behavior.

## What This Changes

The issue requests automatic reclamation of session write-lock files when a live holder exceeds `session.writeLock.maxHoldMs` and later requests hit `session file locked`.

## Best Possible Solution

Keep this issue open while maintainers review https://github.com/openclaw/openclaw/pull/89673 as the canonical path, then close the issue only if that PR or an equivalent fix lands.

## Reproduction Assessment

Yes from source, but not from a live overnight run: current main detects `hold-exceeded` when a lock exceeds the holder-recorded `maxHoldMs`, then refuses to remove locks whose stale reasons are all report-only.

## Solution Assessment

Unclear until maintainers choose the lock-reclaim contract. The linked PR is the narrow fix if `maxHoldMs` is intended to be a hard cross-process reclaim deadline; otherwise the safer path is an explicit opt-in policy.
