## Summary

Keep open. Current main still has the reported Telegram regression: non-off `replyToMode` builds a native-quote candidate for ordinary inbound text, and the preview-stream guard still treats any candidate as quote-incompatible, so plain `reply_to_message_id` delivery disables answer draft streaming. The closed unmerged #73534 remains useful source work; open #39905 is broader per-conversation configuration work, not a landed fix for this specific bug.

## What This Changes

The issue requests that Telegram `streaming.mode: "partial"` continue live preview edits when `replyToMode` is non-`off` for plain replies, while still avoiding preview streaming for true selected native quote replies.

## Best Possible Solution

Land a narrow Telegram dispatcher fix that separates final-delivery quote candidate collection from preview-stream compatibility, so plain reply threading can stream while true selected native quotes still use the non-preview path.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `replyToMode: "first"`, `streamMode: "partial"`, and a plain nonblank Telegram message populate `replyQuoteByMessageId`, make `hasNativeQuoteReply` true, and prevent the answer draft stream from being created.

## Solution Assessment

Yes. The narrowest maintainable fix is to key the preview-stream skip on actual selected quote metadata, such as `replyQuoteText != null && replyQuoteMessageId != null`, while preserving `replyQuoteByMessageId` for final delivery fallback.
