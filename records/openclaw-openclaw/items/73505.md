## Summary

Keep open. Current main still disables Telegram answer preview streaming whenever `replyToMode` is non-`off` and a native quote candidate exists; the dispatcher unconditionally creates that candidate for ordinary inbound text, and the current test suite still expects no draft stream for that plain-reply path. The closed related PR #73534 carried the right narrow fix, but it was not merged, so the report remains actionable.

## What This Changes

Restore Telegram partial preview streaming for plain `replyToMode` replies while still disabling it for actual Telegram selected-quote replies.

## Best Possible Solution

Land a narrow Telegram plugin fix that bases the streaming-disable guard on a real selected quote (`replyQuoteText` plus resolvable quote message id), keeps native quote candidates available for final delivery fallback, adds a focused regression test, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with `replyToMode: "first"` and `streamMode: "partial"`, a plain inbound text message populates `replyQuoteByMessageId`, makes `hasNativeQuoteReply` true, and prevents `createTelegramDraftStream`; the current test suite asserts that behavior.

## Solution Assessment

Yes. The suggested direction is the narrowest maintainable fix: only genuine selected Telegram quotes need to avoid edit-based preview, while plain reply threading is already supported by the draft stream's initial `sendMessage` reply target.
