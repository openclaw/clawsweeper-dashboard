## Summary

Keep open. Current main still has the Telegram regression: with `replyToMode` set to a non-`off` value, ordinary inbound text populates `replyQuoteByMessageId`, the answer draft-stream guard treats that as a native quote reply, and `streaming.mode: "partial"` never creates the preview stream. The latest release tag in the provided context still contains the same guard, and the related source PR #73534 was closed unmerged while #39905 is broader per-conversation configuration work rather than a landed fix for this bug.

## What This Changes

Restore Telegram `streaming.mode: "partial"` preview streaming for plain replies when `replyToMode` is non-`off`, while still avoiding preview streaming for true selected native quote replies.

## Best Possible Solution

Land a narrow Telegram plugin fix that separates final-delivery native quote candidate collection from preview-stream compatibility: plain reply threading should continue to create a draft preview, while true selected Telegram quotes should keep the non-preview path because quote parameters cannot be preserved through preview edits.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `replyToMode: "first"`, `streamMode: "partial"`, and a plain nonblank Telegram message populate `replyQuoteByMessageId`, make `hasNativeQuoteReply` true, and prevent the answer draft stream from being created.

## Solution Assessment

Yes. The narrowest maintainable fix is to key the preview-stream skip on actual selected quote metadata, such as `replyQuoteText != null && replyQuoteMessageId != null`, while preserving `replyQuoteByMessageId` for final delivery fallback.
