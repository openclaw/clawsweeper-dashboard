## Summary

Keep open. Current main still disables Telegram answer draft streaming whenever a non-off reply mode creates any native quote candidate, including ordinary inbound text, so the reported `replyToMode` plus `streaming.mode: "partial"` regression remains valid. The useful source PR #73534 is closed unmerged, and broader adjacent work such as #39905 does not replace this narrow bug fix.

## What This Changes

Telegram partial streaming should keep creating and editing a preview message when `replyToMode` is `first` or `all`; only real selected Telegram quote replies should disable draft streaming.

## Best Possible Solution

Land a narrow Telegram dispatcher fix that separates final-delivery native quote candidate collection from draft-stream compatibility, so plain reply threading can stream while selected Telegram quote replies continue to use the non-preview path. Add or update focused dispatcher tests and a user-facing changelog entry crediting the reporter and useful source work.

## Reproduction Assessment

Yes. With `replyToMode: "first"`, `streamMode: "partial"`, and a normal nonblank Telegram message, current source builds `replyQuoteByMessageId`, makes `hasNativeQuoteReply` true, and prevents `createTelegramDraftStream` from being created.

## Solution Assessment

Yes. The narrowest maintainable solution is to key the draft-stream skip on actual selected quote metadata, such as `replyQuoteText != null && replyQuoteMessageId != null`, while preserving `replyQuoteByMessageId` for final delivery fallback. The broader per-conversation settings PR is adjacent but not a substitute for this regression fix.
