## Summary

Keep open: current `main` still normalizes numeric ACP/JSON-RPC `RequestError` values using only the generic message, while the ACP SDK contract carries the useful diagnostic in `data.details`. This is a narrow, source-backed diagnostics bug with no merged or open PR found that specifically preserves those details.

## What This Changes

The issue requests that ACP runtime error normalization preserve upstream `data.details` when foreign ACP/JSON-RPC errors use numeric codes.

## Best Possible Solution

Extract a redacted diagnostic from ACP/JSON-RPC `RequestError.data.details` during ACP error normalization or formatting, preserve the cause chain, and add focused regression coverage without changing ACP runtime option semantics.

## Reproduction Assessment

Yes, at source level: pass a `RequestError`-shaped `Error` with numeric `code: -32603`, message `Internal error`, and `data.details` into `toAcpRuntimeError`, then render it with `formatAcpRuntimeErrorText`; current code drops the detail from visible text. I did not run a live Claude ACP session in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve and redact upstream `data.details` in the shared ACP runtime error path, with tests for numeric JSON-RPC errors, rather than adding a Claude-specific workaround or changing ACP control behavior.
