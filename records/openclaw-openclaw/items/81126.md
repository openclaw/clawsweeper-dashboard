## Summary

Keep open: current main and the latest release still render numeric ACP/JSON-RPC `RequestError` failures from `error.message` while ignoring the useful diagnostic in `error.data.details`, so the reported operator-visible loss of detail remains a narrow valid bug.

## What This Changes

The issue asks OpenClaw to preserve and surface ACP SDK `data.details` diagnostics when `toAcpRuntimeError` wraps numeric foreign error codes such as `-32603`.

## Best Possible Solution

Extract a redacted diagnostic string from ACP SDK `RequestError.data.details` during runtime error normalization and include it in ACP error text/log chains without changing ACP runtime option semantics.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction: pass a `RequestError`-shaped `Error` with numeric `code: -32603`, message `Internal error`, and `data.details` into `toAcpRuntimeError`, then render it with `formatAcpRuntimeErrorText`; current main drops the details from visible text. I did not run a live Claude ACP spawn.

## Solution Assessment

Yes, the best fix is a narrow normalization and formatting change in the ACP runtime error path, with regression coverage for numeric JSON-RPC errors and redacted structured details rather than a Claude-specific workaround.
