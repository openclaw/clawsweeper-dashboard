## Summary

Keep open: current main still preserves the numeric JSON-RPC error only as a cause code/message path and does not surface `RequestError.data.details`, so the reported opaque `Internal error` path remains source-reproducible.

## What This Changes

The issue asks OpenClaw to preserve ACP SDK `RequestError` numeric-code detail payloads, especially `data.details`, when wrapping them with `toAcpRuntimeError`.

## Best Possible Solution

Add a central redacted JSON-RPC detail extractor for ACP runtime errors, use it in ACP error chain/text formatting, and cover numeric `RequestError` objects with regression tests without changing timeout policy or ACP control semantics.

## Reproduction Assessment

Yes, at source level: construct an ACP SDK `RequestError(-32603, "Internal error", { details: "Unknown config option: timeout" })`, pass it through `toAcpRuntimeError` and the current formatters, and the displayed text omits `data.details`. I did not run the live Claude ACP setup in this read-only review.

## Solution Assessment

Yes for the narrow direction, but numeric JSON-RPC codes should stay as backend cause codes rather than becoming `AcpRuntimeErrorCode` values. The maintainable fix is to preserve and redact `data.details` in the wrapping/formatting path, with focused tests for ACP SDK `RequestError` shapes.
