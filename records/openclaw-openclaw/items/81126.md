## Summary

Keep open: current `main` still wraps numeric ACP/JSON-RPC `RequestError` values using only `error.message`, while the ACP SDK contract carries the useful diagnostic in `data`, so the reported loss of `data.details` remains a narrow valid bug.

## What This Changes

Preserve ACP SDK JSON-RPC error `data.details` when wrapping numeric-coded foreign errors into OpenClaw ACP runtime errors.

## Best Possible Solution

Extract a redacted diagnostic string from ACP/JSON-RPC error `data.details` during ACP runtime error normalization and include it in user-visible ACP error text without changing runtime option semantics.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction: pass a `RequestError`-shaped `Error` with numeric `code: -32603`, message `Internal error`, and `data.details` into `toAcpRuntimeError`, then render it with `formatAcpRuntimeErrorText`; current code drops `data.details` from the visible text.

## Solution Assessment

Yes, the best fix is a narrow normalization change in the ACP runtime error path, with regression coverage for numeric JSON-RPC errors and redacted structured details rather than a broader ACPX or Claude-specific workaround.
