## Summary

Close: the central CLI timeout parsing bug is already fixed on current main and shipped in v2026.6.6 by strict `cron edit --timeout-seconds` validation, so this older helper-swap PR is no longer useful to merge.

## What This Changes

The PR imports `parsePositiveIntOrUndefined` into `src/cli/cron-cli/register.cron-edit.ts` and uses it to omit invalid `timeoutSeconds` values from cron edit payload patches.

## Best Possible Solution

Keep the shipped strict CLI validation from v2026.6.6 and handle the separate Control UI zero-timeout contract in https://github.com/openclaw/openclaw/pull/41275.

## Reproduction Assessment

No for current main: invalid `cron edit --timeout-seconds` values now throw before `cron.update` can receive a bad timeout. The old negative-value forwarding path was source-reproducible on the PR base, but it is fixed in v2026.6.6.

## Solution Assessment

Not as a merge path now: the shipped strict validation is the better end state because it rejects invalid CLI input consistently with current `cron add`. The PR's old-base omit-invalid helper swap and overbroad comment are superseded.
