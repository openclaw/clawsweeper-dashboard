## Summary

Keep open. Current main still does not route `plugins.entries.openai.config.stt` or `tools.media.audio` into Control UI Talk realtime provider resolution; source and docs show Talk currently uses `talk.providers` and Voice Call realtime provider config, while OpenAI STT config is a separate batch or Voice Call streaming surface.

## What This Changes

The issue asks Talk mode to use a local OpenAI-compatible Whisper STT endpoint configured under the OpenAI plugin config instead of failing with an unconfigured OpenAI realtime voice provider error.

## Best Possible Solution

Define the Talk contract for local STT explicitly: either add a supported Talk/realtime provider path for local Whisper-backed browser Talk, or document and diagnose that batch `tools.media.audio` and OpenAI plugin STT config do not configure Talk realtime sessions.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure only the reported OpenAI plugin STT block, request `talk.realtime.session`, and current main builds no matching realtime provider config before the resolver reaches the unconfigured provider error path.

## Solution Assessment

Unclear. Implicitly treating an OpenAI-compatible batch transcription endpoint as a duplex Talk realtime provider is probably not the safest fix; an explicit Talk/realtime provider contract or clearer diagnostics would preserve the existing surface split.
