## Summary

Keep open. Current main still does not route `plugins.entries.openai.config.stt` or `tools.media.audio` into Control UI Talk realtime provider resolution, so the reported unconfigured OpenAI realtime provider error remains a valid product/API boundary gap rather than an implemented, duplicate, or stale report.

## What This Changes

Issue reports that Control UI Talk does not use `plugins.entries.openai.config.stt` for a local Whisper/OpenAI-compatible STT endpoint and instead errors that the OpenAI realtime voice provider is unconfigured.

## Best Possible Solution

Keep this open until maintainers define the Talk realtime contract for local STT. The maintainable end state is either explicit diagnostics/docs that batch `plugins.entries.openai.config.stt` does not configure Talk, or a dedicated realtime voice provider/plugin path for local Whisper-backed Talk if that product behavior is desired.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure only `plugins.entries.openai.config.stt`, request `talk.realtime.session`, and current main builds no matching realtime provider config before the resolver reaches the unconfigured OpenAI realtime provider branch.

## Solution Assessment

Unclear. Treating an OpenAI-compatible batch transcription endpoint as an implicit Talk realtime provider is probably not the best long-term fix; explicit `talk.providers`/realtime-provider configuration plus better diagnostics, or a dedicated local Whisper Talk provider, would be safer.
