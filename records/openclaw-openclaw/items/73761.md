## Summary

Close as ClawHub/plugin scope. Current main intentionally separates browser Talk realtime voice from batch STT/media-understanding config, so routing `plugins.entries.openai.config.stt` or `tools.media.audio` into the bundled OpenAI Talk provider would cross the provider/plugin ownership boundary rather than fix a supported core config path.

## What This Changes

The issue asks for Control UI Talk to initialize the OpenAI realtime voice provider from a local Whisper/OpenAI-compatible STT configuration under `plugins.entries.openai.config.stt`.

## Best Possible Solution

Leave the bundled OpenAI Talk path focused on supported realtime providers, and move local Whisper Talk support into a ClawHub/code-plugin effort using `registerRealtimeVoiceProvider` and the Gateway relay. If that plugin exposes a concrete missing SDK capability, open a narrower plugin API issue for that seam.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: configure only `plugins.entries.openai.config.stt`, start `talk.realtime.session`, and current main builds no matching realtime provider config before the resolver reaches the unconfigured-provider branch.

## Solution Assessment

Yes. Closing the core issue in favor of a plugin track is the narrowest maintainable cleanup because batch OpenAI-compatible STT config cannot safely be overloaded into a duplex realtime Talk provider contract.
