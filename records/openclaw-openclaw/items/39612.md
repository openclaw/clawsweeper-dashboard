## Summary

Keep open: current main now covers the original default raw-error leak path with generic fallback text, Anthropic thinking-block recovery, and the related group silent-reply policy fix, but it still intentionally allows detail-bearing failure text in verbose direct chats and does not implement the requested repeated-failure escalation/session-migration policy.

## What This Changes

Request framework-level LLM failure handling that retries or sends/suppresses friendly copy, escalates repeated failures, and detects thinking-block session corruption instead of forwarding raw provider errors.

## Best Possible Solution

Define one explicit external-channel LLM failure contract: generic or silent by default, diagnostic details limited to approved operator surfaces, and any repeated-failure escalation coordinated with the broader failover-notification policy.

## Reproduction Assessment

No high-confidence live reproduction of the original 2026.3.7 WhatsApp/Telegram leak was established. Source, tests, and docs do show the remaining current-main path where verbose direct chats can receive sanitized runner/provider failure details.

## Solution Assessment

Unclear. The generic fallback copy, silent-reply policy fix, and Anthropic thinking recovery are the right immediate repairs, but the strict no-details rule and repeated-failure escalation require maintainer product/security direction.
