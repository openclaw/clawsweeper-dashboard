## Summary

Keep open: current main covers the original non-verbose raw-error path, Anthropic thinking-block recovery, and the related group/channel silentReply bug, but current docs and tests still intentionally allow sanitized detail-bearing failure text in verbose direct external chats, and the requested repeated-failure escalation/session policy remains a maintainer product/security decision.

## What This Changes

The issue requests framework-level handling for LLM failures so external channels never receive raw provider errors, with retry or silence, repeated-failure escalation, and session corruption detection.

## Best Possible Solution

Define one external-channel LLM failure contract: generic or silent by default, diagnostics only on approved operator surfaces, and explicit maintainer policy for repeated-failure notification and session recovery.

## Reproduction Assessment

Yes for the remaining current-main behavior: source, docs, and tests show verbose direct external chats can receive sanitized detail-bearing failure text. No live reproduction of the original 2026.3.7 WhatsApp/Telegram leak was run.

## Solution Assessment

Unclear. The generic fallback copy, Anthropic thinking recovery, and silentReply policy repair are good immediate fixes, but a strict no-details rule and repeated-failure escalation need maintainer product/security direction.
