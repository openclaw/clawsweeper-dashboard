## Summary

Keep open: current main mitigates the original default raw-error leak and has Anthropic thinking-block recovery, but the stricter external-channel disclosure policy is still unresolved because verbose direct chats intentionally receive sanitized detail-bearing failure text and repeated-failure escalation remains a product/security decision.

## What This Changes

The issue requests framework-level handling so LLM/API failures do not leak raw provider errors to messaging users, with retry, escalation, and session-corruption recovery after thinking-block incompatibilities.

## Best Possible Solution

Define one external-channel LLM failure contract: generic or silent by default, diagnostics only on approved operator surfaces, and repeated-failure/session-recovery behavior coordinated with existing fallback notices.

## Reproduction Assessment

Yes, for the remaining current-main behavior at source level: source, docs, and tests show verbose direct external chats can receive sanitized detail-bearing failure text. No live reproduction of the original 2026.3.7 WhatsApp/Telegram leak was established in this read-only review.

## Solution Assessment

Unclear. The generic fallback copy, `silentReply` policy repair, and Anthropic thinking retry are good partial fixes, but the strict no-details rule and repeated-failure escalation require maintainer product/security direction.
