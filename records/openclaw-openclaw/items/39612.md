## Summary

Keep open. Current main partially mitigates the thinking-block corruption path, but it still intentionally sends sanitized provider/runner failure details to external direct chats when `/verbose` is enabled, which does not satisfy the request to never expose LLM/API failure details to WhatsApp or Telegram users.

## What This Changes

The issue requests framework-level handling so LLM/provider failures do not leak raw API errors to WhatsApp, Telegram, or similar user-facing channels, with retry/fallback behavior and session-corruption mitigation.

## Best Possible Solution

Keep the issue open until maintainers decide the external-channel failure disclosure policy. A maintainable end state would keep detailed provider/runner errors in logs or operator-only surfaces, send generic fallback copy or silence to WhatsApp and Telegram direct chats, preserve non-direct chat silence, and split session migration plus multi-agent escalation into focused follow-ups if still desired.

## Reproduction Assessment

Yes. Current main has a high-confidence unit-level reproduction: a mocked runner rejection in a WhatsApp direct context with `resolvedVerboseLevel: "on"` is expected to return the underlying failure detail in the final reply.

## Solution Assessment

No. The current implementation is useful for diagnostics and partially fixes thinking replay corruption, but it is not the best solution for the issue's external-chat safety goal because verbose direct chats can still receive provider/runner failure details.
