## Summary

Keep open: current main has fixed the original default raw-error leak path, Anthropic thinking-block recovery, group silent-reply policy, and model-fallback notices, but the strict external error-disclosure rule is still unresolved because verbose direct chats intentionally receive detail-bearing failure text.

## What This Changes

The issue requests framework-level handling for LLM failures so WhatsApp, Telegram, and other external channels do not receive raw provider/API errors, with fallback or silence, repeated-failure escalation, and session-corruption handling.

## Best Possible Solution

Define one explicit external-channel LLM failure contract: generic or silent by default, diagnostic details only on approved operator surfaces, and any repeated-failure escalation coordinated with the current fallback-notice and session-recovery paths.

## Reproduction Assessment

No high-confidence live reproduction of the original 2026.3.7 WhatsApp/Telegram raw-error leak was established. Yes for the remaining current-main behavior: source, docs, and tests show verbose direct external chats can receive sanitized detail-bearing failure text.

## Solution Assessment

Unclear. The current generic fallback copy, Anthropic thinking recovery, silent-reply policy fix, and fallback notices are good immediate repairs, but the strict no-details rule and repeated-failure escalation require maintainer product/security direction.
