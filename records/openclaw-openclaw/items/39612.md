## Summary

Keep this open: current main has already fixed much of the reported leak/recovery path, but the issue also asks for a stricter “never raw” external-chat policy and multi-agent escalation that are not fully settled in code.

## What This Changes

Request graceful handling of LLM/provider failures in chat channels with friendly or silent replies, bounded retry/recovery for thinking-block session corruption, and optional escalation to a main agent.

## Best Possible Solution

Define the external-channel failure contract explicitly: normal user chats should receive friendly or silent failure handling, debug details should stay in operator surfaces/logs unless explicitly allowed, and any failure-streak escalation should be designed as a separate multi-agent policy.

## Reproduction Assessment

No high-confidence live reproduction of the original 2026.3.7 WhatsApp/Telegram leak was established. Source and tests do show current main’s default path uses friendly copy while verbose direct-chat paths can still include sanitized provider details.

## Solution Assessment

Unclear. The merged fallback and thinking-recovery work is the right direction for the immediate leak, but a strict no-details rule and automatic main-agent escalation need maintainer policy before implementation.
