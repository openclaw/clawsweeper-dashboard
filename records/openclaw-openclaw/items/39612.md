## Summary

Keep this open: current main has generic fallback copy and Anthropic thinking-block recovery for the original raw-error path, but current code, tests, and docs still intentionally allow detail-bearing verbose failures in direct external chats and there is no general failure-streak escalation contract.

## What This Changes

Request framework-level handling so failed LLM/provider calls do not leak raw errors to WhatsApp or Telegram users, with retry/fallback behavior, failure escalation, and session-corruption recovery.

## Best Possible Solution

Define the external-channel failure contract explicitly: normal chats receive generic or silent failures, diagnostic details stay in logs/control UI or an approved operator surface, and repeated-failure escalation is designed as a separate multi-agent policy if maintainers want it.

## Reproduction Assessment

No high-confidence live reproduction of the original 2026.3.7 WhatsApp/Telegram leak was established. Source and tests do show the remaining strict-policy gap: verbose direct-chat paths can still include sanitized provider/runtime details.

## Solution Assessment

Unclear. The shipped generic fallback and Anthropic recovery are the right immediate fixes, but changing verbose direct-chat disclosure or adding automatic main/default-agent escalation needs maintainer product/security direction.
