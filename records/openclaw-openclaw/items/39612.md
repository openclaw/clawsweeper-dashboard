## Summary

Keep this open: current main covers the original raw-error path with generic fallback copy and Anthropic thinking-block recovery, but it still deliberately allows sanitized failure details in verbose direct chats and has no general repeated-failure escalation contract.

## What This Changes

Request framework-level LLM failure handling that prevents raw provider/API errors from reaching WhatsApp/Telegram users, retries or goes silent, escalates repeated failures, and handles corrupted thinking-block sessions.

## Best Possible Solution

Define the external-channel failure contract explicitly: normal external chats receive generic or silent failures, diagnostic details stay in logs/control UI or another approved operator surface, and repeated-failure escalation is designed separately if maintainers want it.

## Reproduction Assessment

No high-confidence live reproduction of the original 2026.3.7 WhatsApp/Telegram leak was established. Source, tests, and docs do show the remaining policy gap: verbose direct chats can still include sanitized runner/provider details.

## Solution Assessment

Unclear. The shipped generic fallback and Anthropic thinking recovery are the right immediate fixes, but the remaining strict no-details rule and repeated-failure escalation need maintainer product/security direction.
