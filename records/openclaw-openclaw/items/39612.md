## Summary

Keep open: current main partially mitigates Anthropic/Bedrock thinking-block corruption, but it still intentionally sends sanitized provider/runner failure details to external direct chats when `/verbose` is `on` or `full`, so the central “never expose raw LLM/API errors to users” request is not implemented.

## What This Changes

The issue requests framework-level graceful LLM failure handling so raw provider errors are not sent through WhatsApp/Telegram replies, with retry, escalation, and session-corruption mitigation follow-ups.

## Best Possible Solution

Define external-channel failure policy so WhatsApp/Telegram direct replies get generic or silent fallback copy while detailed provider/runner errors stay in logs or operator-only surfaces, then split session migration and multi-agent escalation into focused follow-ups.

## Reproduction Assessment

Yes. Source and test inspection give a high-confidence path: a mocked embedded-runner rejection in a WhatsApp direct context with `resolvedVerboseLevel: "on"` is expected to return the underlying failure text in the final reply.

## Solution Assessment

No. Current main’s documented `/verbose` behavior is useful for diagnostics, but it is not the best fit for the external-chat safety goal; the safer path is to keep detailed failures on operator/log surfaces and use generic external fallback text.
