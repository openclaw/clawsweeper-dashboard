## Summary

Keep open: current main mitigates the original default raw-error leak and includes Anthropic thinking-block recovery, but the stricter external-channel disclosure contract is still unresolved because verbose direct chats can intentionally receive sanitized failure details and repeated-failure escalation remains a product/security decision.

## What This Changes

Request framework-level LLM failure handling that prevents raw provider/runtime errors from reaching messaging-channel users, retries or falls back gracefully, escalates repeated failures, and detects corrupted thinking-block session history.

## Best Possible Solution

Define one external-channel LLM failure contract: generic or silent by default, diagnostics only on approved operator surfaces, and repeated-failure/session-recovery behavior coordinated with existing fallback notices.

## Reproduction Assessment

Yes, for the remaining current-main behavior at source level: source, docs, and tests show verbose direct external chats can receive sanitized detail-bearing failure text. I did not live-reproduce the original 2026.3.7 WhatsApp or Telegram leak.

## Solution Assessment

Unclear. The generic fallback copy, silentReply policy repair, and Anthropic thinking retry are good partial fixes, but the strict no-details rule and automatic main-agent escalation need maintainer product/security direction.
