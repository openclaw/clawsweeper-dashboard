## Summary

Keep open: current main still has a source-visible path where terminal ACP background tasks with a deliverable requester origin send generic `Background task ...` text directly to the channel, and open PR #77365 is already linked as the implementation candidate.

## What This Changes

The issue requests routing background ACP/subagent completions in group channels through the parent agent so users receive a real completion summary instead of only generic lifecycle text.

## Best Possible Solution

Review and land PR #77365, or an equivalent narrow change, so parent-owned ACP completions wake the parent announce path with rich child-result context and keep generic lifecycle text only as fallback telemetry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: create a pending ACP task with a deliverable `requesterOrigin`, let it reach a terminal state, and `maybeDeliverTaskTerminalUpdate` sends the formatted `Background task ...` message directly while existing tests assert the parent queue remains empty.

## Solution Assessment

Is this the best way to solve the issue? No for current main's direct delivery path; the maintainable fix is to route parent-owned ACP completion through the existing parent announce machinery and preserve direct lifecycle delivery only as fallback/status telemetry.
