## Summary

Keep open: current main still has a source-reproducible ACP task path that can send generic `Background task ...` terminal text directly to the requester channel, while the documented parent-owned ACP/subagent contract expects completion to flow through the parent-agent announce turn.

## What This Changes

The issue reports that background ACP or subagent completions in group channels can surface only generic lifecycle text instead of a parent-agent reply.

## Best Possible Solution

Route parent-owned ACP completions through the existing parent-agent announce/internal-event machinery and reserve generic task lifecycle text for fallback or secondary telemetry when no parent reply can be produced.

## Reproduction Assessment

Yes, at source level: create a pending ACP task with a deliverable requester origin, let it reach terminal state, and `maybeDeliverTaskTerminalUpdate` will format and send `Background task ...` directly. Existing task-registry tests assert the same visible channel behavior.

## Solution Assessment

No, the current direct generic delivery path is not the best way to solve parent-owned background completion. The narrow maintainable fix is to reuse the existing completion announce path for ACP parent-owned runs and keep generic lifecycle sends as fallback/status telemetry.
