## Summary

Keep open: current main still has a source-backed ACP task path that can send generic `Background task ...` terminal text directly to the requester channel instead of forcing a parent-agent completion reply, despite the documented parent-owned ACP delivery model.

## What This Changes

The issue asks OpenClaw to route background ACP/subagent completion results through the parent agent so group-channel users receive a real conversational reply rather than only lifecycle telemetry.

## Best Possible Solution

Parent-owned ACP completions should use the existing task-completion announce machinery so the parent agent receives internal completion context and produces the visible reply; generic lifecycle text should remain only fallback/status telemetry when announce delivery cannot run.

## Reproduction Assessment

Yes, at source level: create a pending ACP task with a deliverable requester origin, let it reach a terminal state, and `maybeDeliverTaskTerminalUpdate` sends formatted `Background task ...` text directly. Existing tests assert the same behavior and that no parent session event is queued.

## Solution Assessment

No, the current direct generic delivery path is not the best way to solve parent-owned background ACP completion. The narrow maintainable fix is to route those completions through the existing parent announce flow and keep direct lifecycle delivery as fallback telemetry.
