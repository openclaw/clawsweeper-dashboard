## Summary

Keep open: current main still has a source-level path where parent-owned ACP terminal tasks can direct-post generic `Background task ...` text to the requester channel, while the ACP docs and native subagent announce coverage expect a parent-agent announce turn to produce the user-facing reply.

## What This Changes

The issue asks OpenClaw to route background ACP/subagent completions in group channels through the parent agent instead of stopping at generic task-status messages.

## Best Possible Solution

Route parent-owned ACP terminal completions through the existing completion announce machinery with internal completion context, keeping generic task lifecycle text only as fallback/status telemetry when a parent announce cannot run.

## Reproduction Assessment

Yes, at source level: create a pending parent-owned ACP task with a deliverable requester origin, let it reach terminal state, and the current registry formats and sends `Background task ...` directly while tests assert the parent event queue remains empty. I did not run a live group-channel harness in this read-only review.

## Solution Assessment

Yes, the maintainable fix direction is to reuse the existing parent-agent completion announce path for parent-owned ACP runs. Direct lifecycle sends should remain a fallback for cases where no parent announce can be produced, not the primary visible completion.
