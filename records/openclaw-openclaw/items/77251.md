## Summary

Keep open: current main still has the reported source-visible ACP terminal-delivery path, and open PR #77365 is already linked with closing syntax as the implementation candidate.

## What This Changes

The issue requests routing group/channel ACP background completion results through the parent agent so users receive a real parent-authored summary instead of only generic task-status text.

## Best Possible Solution

Review and land PR #77365, or an equivalent narrow fix, so group/channel ACP completions wake the parent announce path with useful child-result context while keeping generic lifecycle text only as fallback telemetry.

## Reproduction Assessment

Yes, at source level: create a pending ACP task with a deliverable requester origin, let it reach a terminal state, and current main sends `Background task ...` directly while the existing test asserts no parent session event is queued. I did not run the test because this review is read-only and `pnpm` is not available in the checkout environment.

## Solution Assessment

No, current main's direct channel delivery is not the best solution because it contradicts the documented parent-owned ACP completion contract. The narrow maintainable path is to route parent-owned ACP terminal completion through the parent announce machinery and reserve direct lifecycle delivery for fallback/status cases.
