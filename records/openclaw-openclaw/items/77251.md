## Summary

Keep open: current main still has a source-reproducible ACP task terminal path that can post generic `Background task ...` text directly to the requester channel, while docs and the existing announce machinery point to a parent-agent completion handoff as the intended user-facing flow.

## What This Changes

The issue requests that background ACP/subagent completion events in group channels wake the parent agent to produce the user-facing reply instead of stopping at generic lifecycle text.

## Best Possible Solution

Route parent-owned ACP terminal completions into the existing task-completion announce/internal-event flow with enough child result context for the parent reply, leaving generic lifecycle text as fallback telemetry only.

## Reproduction Assessment

Yes, at source level: create a pending ACP task with a deliverable requester origin, let it reach a terminal state, and `maybeDeliverTaskTerminalUpdate` sends `Background task ...` directly while the parent queue stays empty. I did not run a live group-channel harness in this read-only review, but current tests assert the same behavior.

## Solution Assessment

Yes, the best fix direction is to reuse or extend the existing parent-agent completion announce machinery rather than direct-posting terminal ACP lifecycle text. The repair should be narrow and preserve direct delivery only as a fallback/status path.
