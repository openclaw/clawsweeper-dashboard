## Summary

Keep open: current main still has an ACP task-registry path that directly posts generic `Background task ...` terminal text to the requester channel, while the documented parent-owned ACP/subagent completion model expects a parent-agent announce turn to produce the user-facing reply.

## What This Changes

The issue asks OpenClaw to route background ACP/subagent completion results in group/channel contexts through the parent agent instead of leaving only generic task lifecycle messages visible.

## Best Possible Solution

Parent-owned ACP and native subagent completions should wake or run the parent-agent announce path with internal completion context, using generic task lifecycle text only as secondary fallback telemetry when a parent reply cannot be produced.

## Reproduction Assessment

Yes, at source level: create a pending ACP task with a deliverable requester origin, let it reach terminal state, and `maybeDeliverTaskTerminalUpdate` formats and sends `Background task ...` directly. I did not run a live group-channel harness because this review is read-only, but existing tests assert the same path.

## Solution Assessment

No, the current direct generic delivery path is not the best fix for parent-owned background work. The narrow maintainable direction is to reuse the existing completion announce machinery for ACP parent-owned runs and keep direct lifecycle sends as fallback/status telemetry.
