## Summary

Keep this PR open, but it is not mergeable as-is. The config-validation centralization may still be useful, while the branch also carries unrelated Slack and TTS changes, and the TTS parser change would silently drop inline directives that current main still supports in code, tests, and docs.

## What This Changes

The PR centralizes config validation known-plugin-ID collection, adds plugin-validation tests, adds Slack Bolt socket-mode `ignoreSelf: false` coverage, and changes TTS directive parsing/tests to require standalone directive lines except compact text blocks.

## Best Possible Solution

A narrow rebased config-validation patch should use the current plugin-registry and bundled metadata contracts, keep focused validation tests, preserve the existing inline TTS directive behavior, and omit the Slack hunk already covered on main.

## Reproduction Assessment

Yes. A source-level reproduction is a reply like `Hello [[tts:provider=edge]] world` with provider overrides enabled: current main parses and strips the inline directive, while the PR's newline-bounded `directiveRegex` would leave it in text and ignore the override.

## Solution Assessment

No. Centralizing known-ID resolution may be reasonable, but this mixed stale branch is not the best implementation because it couples that change to an unrelated TTS syntax regression and a Slack change already handled elsewhere on current main.

## Review Findings

- **[P1] Preserve inline TTS directive parsing:** `src/tts/directives.ts:75`
  - body: The new `directiveRegex` only matches `[[tts:...]]` markers on standalone lines. Keep inline parsing or ship an explicit migration before tightening the syntax.
