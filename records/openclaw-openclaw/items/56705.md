## Summary

Keep open: the useful config-validation idea is not on current main, but this PR is not merge-ready because it mixes unrelated TTS parsing changes that regress existing inline directive behavior, carries stale Slack hunks now implemented elsewhere on main, conflicts with current metadata boundaries, and lacks external real behavior proof.

## What This Changes

The branch adds a known-plugin-ID validation helper and tests, plus Slack Bolt `ignoreSelf` handling tests/config and TTS directive parsing/test changes.

## Best Possible Solution

Ship a focused config-validation patch against the current plugin metadata boundary, preserve existing inline TTS directive support, and leave the Slack `ignoreSelf` behavior as implemented on main.

## Reproduction Assessment

Yes, by source inspection for the blocking regression: current main parses inline `[[tts:provider=edge]]` directives and its contract suite asserts that behavior, while the PR head newline-bounds the directive regex. The config-validation false-positive is also source-reproducible from `ensureKnownIds` relying only on manifest registry IDs when bundled manifests are unavailable.

## Solution Assessment

No. Centralizing known plugin ID resolution is a reasonable narrow fix, but this branch is not the best vehicle because it mixes unrelated TTS behavior changes, stale Slack changes, and an outdated metadata import boundary.
