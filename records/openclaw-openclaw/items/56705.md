## Summary

Keep this PR open for maintainer handling. Current main still lacks the proposed config known-ID helper, but the branch mixes that useful idea with already-shipped Slack behavior and a TTS parser regression that prior review comments also flagged as a silent behavior break.

## What This Changes

The PR centralizes config validation known-plugin-ID collection, adds plugin-validation tests, adds Slack Bolt socket-mode `ignoreSelf: false` coverage, and changes TTS directive parsing/tests to require standalone directive lines except compact text blocks.

## Best Possible Solution

Split the useful config-validation change into a narrow rebased PR that uses current manifest and bundled metadata contracts, keeps focused validation tests, preserves inline TTS directive parsing, and drops the already-shipped Slack hunk unless maintainers want separate coverage.

## Reproduction Assessment

Yes. A source-level reproduction is a reply like `Hello [[tts:provider=edge]] world` with provider overrides enabled: current main parses and strips the inline directive, while the PR's newline-bounded `directiveRegex` would leave it in text and ignore the override.

## Solution Assessment

No. Centralizing known-ID resolution may be a reasonable direction, but this mixed branch is not the best implementation because it combines config validation with unrelated Slack and TTS behavior changes; a config-only rebase is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve inline TTS directive parsing:** `src/tts/directives.ts:75`
  - body: The new `directiveRegex` only matches `[[tts:...]]` markers on standalone lines. Current main, tests, and docs allow inline directives such as `Hello [[tts:provider=edge]] world`; with this patch those overrides are left in the reply and ignored, so tagged TTS can silently use the wrong provider or voice. Keep inline parsing or ship an explicit migration before tightening the syntax.
