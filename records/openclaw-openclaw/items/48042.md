## Summary

Keep open: current main and v2026.5.3 still let the reported `defaultProfile: "user"` select the synthesized Chrome MCP existing-session profile, so top-level `browser.cdpUrl` does not control the selected runtime path.

## What This Changes

Honor top-level `browser.cdpUrl` when it is configured with `defaultProfile: "user"` and no explicit `profiles.user` entry.

## Best Possible Solution

Preserve explicit `profiles.user` behavior, but when `profiles.user` is absent and top-level `browser.cdpUrl` is set, prevent the implicit `user` Chrome MCP profile from overriding the CDP-backed default path and add a regression test for that config.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass the reported config through `resolveBrowserConfig`, then resolve the selected profile: current main selects `user`, resolves it as `existing-session`, and leaves `cdpUrl` empty.

## Solution Assessment

Yes. A narrow resolver fix in the style of #48068 is the best path: keep explicit `profiles.user` intact while making the implicit user profile unable to bypass a configured top-level CDP URL.
