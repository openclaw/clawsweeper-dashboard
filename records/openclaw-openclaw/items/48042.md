## Summary

Keep open. Current main still has the source-reproducible resolver gap, and https://github.com/openclaw/openclaw/pull/80143 is an open closing PR for the same browser.cdpUrl/defaultProfile user behavior, so closing this issue before that PR lands would be premature.

## What This Changes

The issue reports that top-level `browser.cdpUrl` is ignored when `browser.defaultProfile` is `user`, causing Chrome MCP to fall back to DevToolsActivePort lookup instead of using the configured CDP endpoint.

## Best Possible Solution

Keep this issue open and land https://github.com/openclaw/openclaw/pull/80143, or an equivalent focused resolver fix, after the endpoint-redaction concern is addressed.

## Reproduction Assessment

Yes. Source inspection shows current main lets `defaultProfile: "user"` select an existing-session profile whose `cdpUrl` comes only from the profile-local field, while Chrome MCP falls back to auto-connect when no endpoint is passed.

## Solution Assessment

Yes, with one merge blocker. The narrow maintainable fix belongs in browser profile resolution so the selected existing-session profile receives the normalized top-level endpoint, but the linked PR should also preserve endpoint redaction in Chrome MCP errors before landing.
