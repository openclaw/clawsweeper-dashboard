## Summary

This should stay open: current main still lets `defaultProfile: "user"` select an existing-session profile without the top-level `browser.cdpUrl`, and an open linked PR already owns the fix path.

## What This Changes

The issue reports that top-level `browser.cdpUrl` is ignored when `browser.defaultProfile` selects the built-in `user` profile, causing Chrome MCP to fall back to `DevToolsActivePort` lookup.

## Best Possible Solution

Update and land https://github.com/openclaw/openclaw/pull/80143 after the endpoint-redaction concern is fixed, then close this issue as implemented.

## Reproduction Assessment

Yes. Source inspection on current main shows `defaultProfile: "user"` resolves to an existing-session profile with empty `cdpUrl`, which makes Chrome MCP choose `--autoConnect` instead of the configured endpoint.

## Solution Assessment

Yes, with one blocker on the linked PR. Resolver-side inheritance of the top-level endpoint is the right boundary, but the implementation should also redact sensitive CDP URLs in Chrome MCP failure text before merge.
