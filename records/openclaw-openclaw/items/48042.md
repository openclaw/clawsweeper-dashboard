## Summary

Keep open because current main and the latest release still leave this source-reproducible browser config gap, and an open closing PR already owns the focused fix path.

## What This Changes

The issue reports that top-level `browser.cdpUrl` is ignored when `browser.defaultProfile` is set to `user`, causing Chrome MCP to fall back to DevToolsActivePort discovery.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/80143 or an equivalent resolver fix lands with Chrome MCP endpoint redaction preserved.

## Reproduction Assessment

Yes. Source inspection shows `defaultProfile: "user"` selects an existing-session profile whose `cdpUrl` remains empty, so Chrome MCP falls back to `--autoConnect` instead of the configured endpoint.

## Solution Assessment

Yes. The narrow maintainable fix belongs in browser profile resolution so the selected existing-session profile inherits the normalized top-level endpoint, while preserving endpoint redaction in Chrome MCP failure text.
