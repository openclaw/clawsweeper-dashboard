## Summary

Keep open: current `main` still resolves the reported config to the implicit `user` existing-session profile, so top-level `browser.cdpUrl` can be bypassed and the Chrome MCP `DevToolsActivePort` path remains reachable. The closed-unmerged #48068 is the closest prior fix attempt, and latest release `v2026.4.29` has the same resolver shape.

## What This Changes

The issue asks for top-level `browser.cdpUrl` to be honored directly instead of falling through to Chrome MCP `DevToolsActivePort` lookup when `defaultProfile` is `user`.

## Best Possible Solution

Update the browser config resolver so a top-level `browser.cdpUrl` cannot be silently overridden by the implicit `user` Chrome MCP profile. Preserve explicit `profiles.user` behavior, but when `profiles.user` is absent and top-level `browser.cdpUrl` is set, resolve the default to a CDP-backed profile and add a regression test for the reported config.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to resolve the reported config through `resolveBrowserConfig` and `resolveProfile`: current main selects `user`, resolves it as `existing-session`, leaves `cdpUrl` empty, and then browser action routing treats it as Chrome MCP.

## Solution Assessment

Yes. A #48068-style resolver fix is the narrowest maintainable path because the bug is in default profile resolution, not in Chrome MCP or CDP transport itself; it should not remove explicit existing-session profiles or per-profile `cdpUrl` support.
