## What This Changes

The issue asks OpenClaw to honor top-level browser.cdpUrl when browser.defaultProfile selects the built-in user existing-session profile, instead of falling back to DevToolsActivePort discovery.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/80143 or an equivalent resolver fix lands with regression coverage and the existing Chrome MCP endpoint redaction preserved.

## Reproduction Assessment

Yes. Source inspection shows defaultProfile: "user" selects an existing-session profile whose cdpUrl remains empty, so Chrome MCP receives --autoConnect instead of the configured endpoint.

## Solution Assessment

Yes. The maintainable fix belongs in browser profile resolution so the selected existing-session default profile inherits the normalized top-level endpoint while explicit profile-local cdpUrl still wins.
