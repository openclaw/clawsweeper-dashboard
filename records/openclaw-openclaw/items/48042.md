## Summary

Keep open: current main and the latest release still source-reproduce the configured `browser.cdpUrl` being dropped when `defaultProfile: "user"` selects the injected existing-session profile, and the focused closing fix remains open at https://github.com/openclaw/openclaw/pull/80143.

## What This Changes

The issue reports that top-level `browser.cdpUrl` is ignored for `browser.defaultProfile: "user"`, causing Chrome MCP to fall back to `DevToolsActivePort` lookup instead of using the configured CDP endpoint.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80143 or an equivalent browser profile resolver fix that applies the normalized top-level endpoint to the selected existing-session default profile while preserving profile-local precedence and endpoint redaction.

## Reproduction Assessment

Yes. Source inspection shows `defaultProfile: "user"` selects an existing-session profile whose own `cdpUrl` is empty, so Chrome MCP receives `--autoConnect` instead of the configured endpoint.

## Solution Assessment

Yes. The maintainable fix belongs in browser profile resolution; the linked PR follows that direction but should retain endpoint redaction before merge.
