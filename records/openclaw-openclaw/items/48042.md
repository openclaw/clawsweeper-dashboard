## Summary

Keep open: current main and the latest release still leave the reported `browser.cdpUrl` plus `defaultProfile: "user"` path source-reproducible, and the focused closing fix remains open at https://github.com/openclaw/openclaw/pull/80143.

## What This Changes

The issue reports that a configured top-level `browser.cdpUrl` is ignored when `browser.defaultProfile` selects the built-in `user` existing-session profile, causing Chrome MCP to fall back to DevToolsActivePort discovery.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80143 or an equivalent resolver fix that applies normalized top-level `browser.cdpUrl` to the selected existing-session default profile when no profile-local endpoint exists, while preserving profile-local precedence and current endpoint redaction.

## Reproduction Assessment

Yes. Source inspection on current main shows `defaultProfile: "user"` selects an existing-session profile whose own `cdpUrl` is empty, so Chrome MCP receives `--autoConnect` instead of the configured endpoint.

## Solution Assessment

Yes. The maintainable fix belongs in browser profile resolution so the legacy top-level endpoint is normalized once and profile-local `cdpUrl` still wins.
