## Summary

Keep open: the PR targets a valid source-proven heartbeat pending-delivery bug, but the patch has a blocking correctness gap around non-default heartbeat ack semantics and needs stronger proof of the actual pending-state path.

## What This Changes

The PR adds heartbeat-specific pending final-delivery suppression/clearing for `HEARTBEAT_OK` acknowledgement text, runner defer handling, regression tests, and a changelog entry.

## Best Possible Solution

Thread the resolved heartbeat ack options into the pending-delivery classification, add regression coverage for non-default `ackMaxChars` and prefixed ack state, then show redacted terminal/log proof that the session-store pending path is skipped or cleared correctly.

## Reproduction Assessment

Yes. The linked report gives deterministic v2026.5.7 steps, and current main source shows the pending replay and defer loop; I did not execute a live gateway reproduction in this read-only review.

## Solution Assessment

No, not yet. Reusing `stripHeartbeatToken` is the right direction, but the pending/replay classifier must use the same resolved heartbeat ack settings as `runHeartbeatOnce` before this is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Honor heartbeat ackMaxChars when skipping pending replay:** `src/auto-reply/reply/agent-runner.ts:844`
  - body: This helper calls `stripHeartbeatToken` with the default 300-character ack window, while `runHeartbeatOnce` resolves `heartbeat.ackMaxChars` and existing coverage expects `ackMaxChars: 0` to deliver `HEARTBEAT_OK 🦞`. With this patch, that same text can be treated as ack-only before delivery recovery and either never persisted or cleared from pending replay, so users who tighten `ackMaxChars` lose durable delivery for content the heartbeat runner would send. Pass the resolved heartbeat ack options into the pending/replay classifier and cover `ackMaxChars: 0`.
  - confidence: 0.88
