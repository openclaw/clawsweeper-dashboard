## Summary

Keep open. Current main still has the reported timeout-loss mode: restart recovery uses a bounded synthetic Gateway resume, records failure when that call times out, and does not attach or import the existing transcript for the next session.

## What This Changes

The issue reports that main-session restart recovery times out after a gateway restart, gives up without importing transcript history, and leaves the prior context orphaned.

## Best Possible Solution

Add readiness-aware restart recovery that preserves the interrupted-session binding and ends with either a safe bounded transcript-backed attach/import or a visible recovery-failure notice.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main-session store entry with `abortedLastRun: true`, a resumable transcript tail, and a Gateway `agent` resume call that times out; current main records failure without transcript import.

## Solution Assessment

No. Retrying the same 10s synthetic resume path is not sufficient; the safer direction is readiness-aware retry plus explicit transcript recovery or user-visible failure handling.
