## Summary

Keep open. Current main still has the reported recovery timeout gap: the main-session restart recovery path retries a synthetic Gateway resume call, but a 10s Gateway timeout is still only counted as failed and there is no bounded transcript-backed attach/import path for the replacement active session. The open graceful-restart umbrella is related, and icophy’s workspace-first workaround is useful context, but this issue preserves a concrete current failure mode that still needs maintainer design.

## What This Changes

The issue requests reliable main-session recovery after Gateway restart timeouts, including retrying or reconstructing context from persisted JSONL transcripts instead of leaving the old context orphaned.

## Best Possible Solution

Fold this concrete timeout case into the graceful Gateway restart recovery design: keep the interrupted main-session binding while Gateway readiness settles, retry with explicit state, and provide either a bounded transcript-backed attach/import path or a clear user-visible recovery failure when live resume cannot proceed.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main-session store entry marked `abortedLastRun: true` with a resumable transcript tail where the synthetic Gateway `agent` resume RPC times out; current main then records a failed recovery without transcript import.

## Solution Assessment

No. A timeout increase alone would not be the best fix; the maintainable path needs readiness-aware retry plus safe transcript attach/import semantics aligned with the broader graceful-restart work.
