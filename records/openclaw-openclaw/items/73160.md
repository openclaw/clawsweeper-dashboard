## Summary

Keep open. Current main still has the reported failure mode: main-session restart recovery retries a synthetic Gateway `agent` call with a 10s timeout and records failure without a transcript-backed attach/import path, while the regular session handler can later allocate a new session id when the stored session is no longer fresh. The open #57425 umbrella and merged #71465 restart-drain work are related, but this issue preserves a concrete current-main timeout case that still needs maintainer design.

## What This Changes

The issue asks for Gateway restart recovery to retry or fall back to transcript-backed context recovery when main-session resume times out instead of orphaning the old transcript.

## Best Possible Solution

Fold this concrete timeout path into the graceful Gateway restart recovery design: make recovery readiness-aware, keep the interrupted main-session binding while retrying, and add a bounded transcript-backed attach/import or a clear user-visible recovery failure when live resume cannot safely continue.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main-session store entry with `abortedLastRun: true`, a resumable transcript tail, and a synthetic Gateway `agent` resume call that times out; current main then reports failed recovery without transcript import.

## Solution Assessment

No. A timeout increase alone is not the best fix; the maintainable solution needs Gateway readiness-aware retry plus safe transcript attach/import semantics aligned with the broader restart recovery design.
