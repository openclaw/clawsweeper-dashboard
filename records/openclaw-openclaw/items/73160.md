## Summary

Keep open. Current main still has the reported failure mode: restart recovery retries a bounded 10s synthetic Gateway resume, but a timeout leaves no transcript-backed attach/import or clear recovery handoff, and stale session freshness can later rotate the active session id.

## What This Changes

The issue requests reliable main-session restart recovery, retry, or transcript-backed fallback when Gateway resume times out so prior conversation context is not orphaned.

## Best Possible Solution

Fold this timeout path into the broader graceful Gateway restart design with readiness-aware retry, preserved interrupted-session binding, and a bounded transcript-backed attach/import or clear recovery-failure notice.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main-session row with `abortedLastRun: true`, a resumable transcript tail, and a Gateway `agent` resume call that times out; current main records failed recovery without importing the prior transcript.

## Solution Assessment

No. The current bounded synthetic retry is useful but not sufficient, and simply increasing the timeout would still leave no safe transcript attach/import or explicit recovery-failure path.
