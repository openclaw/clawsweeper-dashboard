## Summary

Keep open. Current main and the latest shipped release still route restart-aborted main sessions through a bounded synthetic `agent` resume; a timeout only records failure for that attempt and there is no transcript attach/import path for the next active session, so the central context-loss bug remains.

## What This Changes

The issue asks for main-session restart recovery to avoid losing context when the synthetic resume call times out and to fall back to transcript-backed recovery.

## Best Possible Solution

Add readiness-aware main-session restart recovery that preserves the interrupted session binding until recovery is definitive and ends with either safe bounded transcript-backed attach/import or a visible recovery-failure notice.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main-session store entry with `abortedLastRun: true`, a resumable transcript tail, and a Gateway `agent` resume call that times out; current main records the failure without importing or attaching the transcript.

## Solution Assessment

No. Retrying the same 10s synthetic resume path narrows some timing cases but does not solve the orphaned-transcript failure mode; the safer fix needs readiness-aware recovery plus explicit transcript preservation or user-visible failure handling.
