## Summary

Keep open. Current main still matches the reported failure mode: main-session restart recovery can mark a session for recovery and retry a synthetic Gateway `agent` call, but timeout/failure is still counted as failed without attaching or importing the existing JSONL transcript into the active replacement session. The related #57425 umbrella is relevant, and the icophy workspace-first comment is useful operational context, but this issue has concrete current-path timeout evidence that should remain available for maintainer follow-up or an explicit manual split/merge into that umbrella.

## What This Changes

Recover main chat context after Gateway restarts when restart recovery times out, including retry behavior and a transcript JSONL-backed fallback or import path.

## Best Possible Solution

Fold this into the graceful Gateway restart recovery design: preserve the interrupted main-session binding while Gateway readiness settles, make retry/failure state explicit, and add a bounded transcript-backed attach/import path or a clear user-facing recovery failure when live resume cannot be accepted.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main session with `abortedLastRun: true` and a resumable transcript tail where the synthetic `agent` resume RPC times out; current main then reports failure and lacks a transcript-backed fallback.

## Solution Assessment

No. A simple timeout bump or blind retry is not the best fix; the maintainable solution needs readiness-aware retry state plus safe transcript attach/import semantics aligned with #57425.
