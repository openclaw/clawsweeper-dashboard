## Summary

Keep open. Current main still has the reported timeout-loss mode: restart recovery retries the same 10s synthetic Gateway resume path, and repeated failure leaves no transcript-backed attach/import or explicit recovery handoff for the orphaned context.

## What This Changes

The issue requests resilient main-session restart recovery after Gateway resume timeouts, including retry/fallback behavior and transcript-backed context recovery.

## Best Possible Solution

Add a restart recovery design that preserves the interrupted-session binding through readiness-aware retry and ends with either safe bounded transcript-backed attach/import or a visible recovery-failure notice.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main-session store entry with `abortedLastRun: true`, a resumable transcript tail, and a Gateway `agent` resume call that times out; current main records failure without transcript import.

## Solution Assessment

No. Retrying the same 10s synthetic resume path is not sufficient; the safer direction is readiness-aware retry plus explicit transcript recovery or user-visible failure handling.
