## Summary

Keep open. Current main still has a real restart-recovery timeout gap: it retries the same synthetic Gateway resume call with a 10s per-call timeout, but repeated failure still leaves no transcript-backed attach/import path and a later session freshness rollover can strand the prior transcript. The open #57425 umbrella and merged #71465 restart-drain work are related, but this issue preserves a narrower current-main failure case that should stay available for maintainer design.

## What This Changes

The issue requests reliable main-session restart recovery, retry, or transcript-backed fallback when Gateway resume times out after a restart so old conversation context is not orphaned.

## Best Possible Solution

Fold this concrete timeout path into the graceful Gateway restart recovery design: make recovery readiness-aware, preserve the interrupted main-session binding while retrying, and add a bounded transcript-backed attach/import path or a clear user-visible recovery failure when live resume cannot safely continue.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main-session row with `abortedLastRun: true`, a resumable transcript tail, and a Gateway `agent` resume call that times out; current main then records failed recovery without transcript import.

## Solution Assessment

No. The current bounded retry is useful but not sufficient; a timeout increase alone would still leave no safe transcript attach/import or explicit recovery-failure path.
