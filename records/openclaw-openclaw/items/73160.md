## Summary

Keep open. Current main still has a concrete main-session restart recovery gap: it marks interrupted main sessions and retries a synthetic Gateway `agent` call, but a 10s timeout is still treated as failed without attaching, importing, or otherwise surfacing the existing JSONL transcript to the replacement session. The icophy workspace-first workaround is useful operational context, and #57425 is the broader design umbrella, but this issue preserves a current, narrower timeout failure mode that should not be swept away before a maintainer design decision.

## What This Changes

The issue requests reliable main-session recovery after gateway restarts, including retry/fallback behavior and transcript-backed access when the live resume RPC times out.

## Best Possible Solution

Fold this into graceful Gateway restart recovery: preserve the interrupted main-session binding while Gateway readiness settles, make retry/failure state visible, and add a bounded transcript-backed attach/import path or clear user-facing recovery failure when live resume cannot be accepted.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main session with `abortedLastRun: true` and a resumable transcript tail where the synthetic Gateway `agent` resume RPC times out; current main then reports failure and lacks a transcript-backed fallback.

## Solution Assessment

No. The current RPC-only retry path is not enough, and a simple timeout bump would be brittle; the maintainable fix needs readiness-aware retry state plus safe transcript attach/import semantics aligned with #57425.
