## Summary

Keep open. Current main still has the reported timeout-loss mode: restart recovery retries a bounded synthetic Gateway resume and, after failure, leaves the existing transcript referenced only by the old session entry rather than importing or visibly handing off recovery.

## What This Changes

The issue requests stronger main-session restart recovery when Gateway resume times out, including retry or transcript-backed context recovery.

## Best Possible Solution

Add readiness-aware restart recovery that preserves the interrupted-session binding and ends with either safe bounded transcript-backed attach/import or a visible recovery-failure notice.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main-session store entry with `abortedLastRun: true`, a resumable transcript tail, and a Gateway `agent` resume call that times out; current main records failure without transcript import.

## Solution Assessment

No. Retrying the same 10s synthetic resume path is not sufficient; the safer direction is readiness-aware retry plus explicit transcript recovery or user-visible failure handling.
