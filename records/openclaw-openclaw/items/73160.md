## Summary

Keep open. Current main still has the reported timeout-loss mode: main-session restart recovery gives the synthetic Gateway resume call 10 seconds, counts a timeout as failure, and has no transcript-backed attach/import or user-visible recovery handoff afterward.

## What This Changes

The issue asks OpenClaw to preserve or reconstruct main-session context after restart recovery times out instead of orphaning the existing transcript and starting fresh.

## Best Possible Solution

Preserve the interrupted-session binding through readiness-aware retry, then either safely attach/import a bounded transcript tail or surface an explicit recovery-failure notice that points users to the preserved history.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main-session store entry with `abortedLastRun: true`, a resumable transcript tail, and a Gateway `agent` resume call that times out; current main records failure without transcript import.

## Solution Assessment

No. Retrying the same 10-second synthetic resume path is not enough; the safer direction is readiness-aware retry plus explicit transcript recovery or user-visible failure handling.
