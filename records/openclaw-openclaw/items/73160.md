## Summary

Keep open. Current main still has the reported failure mode: restart recovery retries the same 10s synthetic Gateway resume path, but repeated timeout leaves no transcript-backed attach/import or user-visible recovery handoff, and normal session freshness can later rotate the stored session id.

## What This Changes

The issue requests reliable main-session restart recovery, retry, or transcript-backed fallback when Gateway resume times out after a restart so old conversation context is not orphaned.

## Best Possible Solution

Fold this timeout path into the broader graceful Gateway restart design with readiness-aware retry, preserved interrupted-session binding, and a bounded transcript-backed attach/import or clear recovery-failure notice.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main-session row with `abortedLastRun: true`, a resumable transcript tail, and a Gateway `agent` resume call that times out; current main records failed recovery without transcript import.

## Solution Assessment

No. The current bounded retry is useful but not sufficient; a timeout increase alone would still leave no safe transcript attach/import or explicit recovery-failure path.
