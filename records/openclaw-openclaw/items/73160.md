## Summary

Keep open. Current main still has a source-backed timeout-loss mode: main-session restart recovery uses a bounded synthetic Gateway resume, counts a timeout as failure, and has no transcript-backed attach/import or visible recovery-failure handoff for the next active session.

## What This Changes

The issue reports that gateway restart recovery can timeout, orphan existing main-session transcript files, and leave the agent continuing from a fresh session without prior context.

## Best Possible Solution

Add readiness-aware main-session restart recovery that preserves the interrupted session binding and ends with either a safe bounded transcript-backed attach/import or a visible recovery-failure notice.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main-session store entry with `abortedLastRun: true`, a resumable transcript tail, and a Gateway `agent` resume call that times out; current main records failure without transcript import.

## Solution Assessment

No. Retrying the same 10s synthetic resume path alone is not enough; the safer direction is readiness-aware retry plus explicit transcript recovery or user-visible failure handling.
