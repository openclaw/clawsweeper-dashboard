## Summary

Keep open. Current main still matches the reported failure mode: restart-aborted main-session recovery retries a synthetic Gateway `agent` RPC with a 10 second timeout, but a timeout/failure only increments the failed count and does not attach, import, or otherwise expose the existing JSONL transcript to the replacement active session. The related #57425 umbrella and icophy's workspace-first workaround are useful context, but this issue has concrete current-path timeout evidence worth preserving.

## What This Changes

The issue requests reliable main-session recovery after Gateway restarts when the synthetic resume RPC times out, including retry or transcript-backed context reconstruction instead of orphaning existing JSONL history.

## Best Possible Solution

Fold this into the graceful Gateway restart recovery design: preserve the interrupted main-session binding while Gateway readiness settles, make retry/failure state explicit, and add a bounded transcript-backed attach/import path or clear user-facing recovery failure when live resume cannot be accepted.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main session marked `abortedLastRun: true` with a resumable transcript tail where the synthetic `agent` resume RPC times out; current main then counts the attempt as failed and only retries the same RPC path.

## Solution Assessment

No. A simple timeout bump or blind retry is not the best fix; the maintainable solution needs readiness-aware retry state plus safe transcript attach/import semantics aligned with #57425.
