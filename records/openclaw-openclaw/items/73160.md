## Summary

Keep open. Current main still only has partial main-session restart recovery: marked sessions are retried through the same 10 second Gateway `agent` RPC path, and timeout/failure does not attach or import the existing transcript into the replacement active session.

## What This Changes

The issue reports that main-session restart recovery can time out after a Gateway restart, detach the existing JSONL transcript from the active session, and leave the user in a fresh context.

## Best Possible Solution

Fold this into graceful Gateway restart recovery: keep the interrupted main-session binding recoverable while Gateway readiness settles, make retry/failure state explicit, and add a bounded transcript-backed attach/import or clear user-visible recovery path when live resume cannot be accepted.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main session with `abortedLastRun: true` and a resumable transcript tail where the synthetic `agent` resume RPC times out; current main then reports failure and lacks a transcript-backed fallback.

## Solution Assessment

No. The current RPC-only retry path is a useful partial recovery mechanism, but it is not the narrow complete solution because it still cannot restore durable context after repeated Gateway timeouts.
