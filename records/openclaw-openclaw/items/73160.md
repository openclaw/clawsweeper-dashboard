## Summary

Keep open. Current main still has the reported timeout failure mode: restart recovery retries a synthetic Gateway `agent` resume call, but a timeout is only counted as failed and there is no transcript-backed attach/import path for the replacement active session. The open #57425 umbrella is related, but this report has concrete current-path evidence worth preserving for maintainer design.

## What This Changes

The issue asks for main-session restart recovery to avoid orphaning existing JSONL transcript context when the Gateway resume attempt times out.

## Best Possible Solution

Fold this concrete timeout case into the graceful Gateway restart recovery design: keep the interrupted main-session binding while Gateway readiness settles, retry with explicit state, and provide a bounded transcript-backed attach/import or clear user-visible recovery failure when live resume cannot proceed.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main-session store entry with `abortedLastRun: true` and a resumable transcript tail where the synthetic Gateway `agent` resume RPC times out; current main then records a failed recovery without transcript import.

## Solution Assessment

No. A timeout bump alone is not the best fix; the maintainable path needs readiness-aware retry plus safe transcript attach/import semantics aligned with the open graceful-restart umbrella.
