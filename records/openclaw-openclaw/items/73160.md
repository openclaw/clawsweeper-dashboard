## Summary

Keep open. Current main still has the reported failure mode: restart recovery uses a bounded 10s synthetic Gateway resume path, and repeated timeout still leaves no transcript-backed attach/import or clear recovery handoff for the old main-session context.

## What This Changes

Recover or preserve main-session conversation context when Gateway restart recovery times out instead of orphaning the existing transcript.

## Best Possible Solution

Fold this timeout path into the broader graceful Gateway restart design with readiness-aware retry, preserved interrupted-session binding, and a bounded transcript-backed attach/import or explicit recovery-failure notice.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main-session row with `abortedLastRun: true`, a resumable transcript tail, and a Gateway `agent` resume call that times out; current main records failed recovery without transcript import.

## Solution Assessment

No. The current bounded retry is useful but not sufficient, and simply increasing the timeout would still leave no safe transcript attach/import or explicit recovery-failure path.
