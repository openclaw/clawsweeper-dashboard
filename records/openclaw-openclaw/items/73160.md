## Summary

Keep open. Current main still has the timeout-loss mode: restart recovery can exhaust its bounded Gateway `agent` resume attempts and then records failure without preserving the old session binding, importing the transcript, or surfacing a recovery handoff to the new session.

## What This Changes

The issue requests reliable main-session restart recovery after Gateway restarts, including retry or transcript-backed fallback instead of silently losing active conversation context on timeout.

## Best Possible Solution

Add readiness-aware restart recovery that preserves the interrupted-session binding and ends with either safe bounded transcript-backed attach/import or a visible recovery-failure notice.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a main-session store entry with `abortedLastRun: true`, a recent resumable transcript tail, and a Gateway `agent` resume call that times out; current main records failure without transcript import.

## Solution Assessment

No. Repeating the same bounded synthetic resume path is not sufficient; the safer direction is readiness-aware retry plus explicit transcript recovery or user-visible failure handling.
