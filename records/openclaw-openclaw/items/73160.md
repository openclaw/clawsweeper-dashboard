## Summary

Keep open. Current main still has the reported timeout-loss mode: main-session recovery sends a synthetic Gateway resume with a 10s timeout, counts failure, and has no transcript-backed attach/import path for the orphaned context; the workspace-first workaround in the discussion is useful operational context but not a core fix.

## What This Changes

The issue requests reliable main-session recovery after Gateway restart timeouts, including retry/fallback behavior and access to the previous transcript-backed context.

## Best Possible Solution

Add readiness-aware restart recovery that preserves the interrupted session binding and ends with either safe bounded transcript-backed attach/import or a clear user-visible recovery failure notice.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a running main-session store entry with `abortedLastRun: true`, a resumable transcript tail, and a Gateway `agent` resume call that times out; current main records failure without transcript import.

## Solution Assessment

No. The current 10s synthetic resume retry is not sufficient; the safer direction is readiness-aware retry plus explicit transcript recovery or visible failure handling.
