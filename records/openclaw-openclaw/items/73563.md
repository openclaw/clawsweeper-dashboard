## Summary

Keep open: current main still has the transcript-redaction gap, but this PR is not ready because it regresses the current gateway transcript append guardrail, can report blocked writes as successful, and exposes the transcript guard through a broad public SDK barrel.

## What This Changes

The PR wraps several session transcript persistence paths with `guardSessionManager`, threads redaction config and update-mode options, updates Plugin SDK metadata, and adds transcript redaction/update regression tests.

## Best Possible Solution

Land a maintainer-reviewed security fix that applies redaction and hook-blocking semantics to every transcript persistence route while preserving current async parent-linked append helpers and keeping any SDK seam narrow, documented, and versioned.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main writes secret-bearing assistant messages through raw transcript append helpers that persist `params.message` directly, while docs promise persisted transcript redaction.

## Solution Assessment

No. Reusing the existing guard is the right direction, but the patch should preserve the async append path, handle blocked writes explicitly, and avoid broad public SDK exposure unless maintainers intentionally version that contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the async gateway transcript append helper:** `src/gateway/server-methods/chat-transcript-inject.ts:106-108`
  - body: On current main, `server-methods.test.ts` asserts gateway-injected transcript writes keep using `appendSessionTranscriptMessage({ ... useRawWhenLinear: true })` and avoid `SessionManager.open(params.transcriptPath)`. This change removes that helper and opens the transcript synchronously, so the current guardrail test will fail and the large-transcript path regresses.
  - confidence: 0.9
- **[P2] Return failure when the guard blocks transcript appends:** `src/config/sessions/transcript.ts:292-293`
  - body: `guardSessionManager` can make `appendMessage` return `undefined` when `before_message_write` blocks a message or sanitization drops it. Returning `ok: true` with that undefined message id makes callers treat a policy-blocked transcript write as persisted.
  - confidence: 0.87
- **[P2] Keep transcript guards off the public runtime barrel:** `src/plugin-sdk/agent-harness-runtime.ts:127`
  - body: `agent-harness-runtime` is a public Plugin SDK subpath, and exporting `guardSessionManager` there promotes mutable transcript redaction, hook blocking, idempotency, and update-emission internals to a broad supported surface. Use a narrow internal/bundled seam or document and version this contract explicitly before syncing the baseline.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/config/sessions/transcript.ts:165`
  - body: This is a user-facing security fix to persisted transcript redaction, but the PR does not change `CHANGELOG.md`. Repo policy requires a changelog entry for fix/security behavior changes.
  - confidence: 0.8
