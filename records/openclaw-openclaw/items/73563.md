## Summary

Keep open: the underlying transcript-redaction gap is still present on current main and this PR is a useful fix path, but the current diff still has merge-blocking correctness/security-boundary issues before it should land.

## What This Changes

The PR wraps several transcript append paths with `guardSessionManager`, threads config/update-mode options, updates the Plugin SDK baseline/export surface, and adds transcript redaction/update-mode regression tests.

## Best Possible Solution

Land a maintainer-reviewed security fix that applies redaction and hook-blocking semantics to every transcript persistence route, keeps any SDK seam narrow and intentional, and records the user-facing security fix in the changelog.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: write a secret-bearing assistant message through the delivery mirror, gateway injection, CLI/ACP persistence, or Codex mirror path on current main and inspect the JSONL transcript, where the current helper persists the message payload without the shared redaction guard.

## Solution Assessment

No. Reusing `guardSessionManager` is the right direction, but this patch still needs blocked-write handling, a narrower or explicitly approved SDK boundary, and a current-main rebase before it is the safest maintainable fix.

  - body: `guardSessionManager` can return `undefined` when `before_message_write` blocks the message or sanitization drops it. This helper still returns `ok: true` with that value as the message id, so callers and update listeners can treat a blocked transcript write as persisted.

  - body: Exporting `guardSessionManager` from `agent-harness-runtime` promotes transcript redaction, hook blocking, idempotency, and event-emission internals to a broad public SDK surface. Keep it on a narrow internal/bundled seam or document and version the contract before syncing the API baseline.
  - confidence: 0.83
