## Summary

Keep open: this PR is still not mergeable because the latest head does not route the raw transcript append paths through the guard it claims to use, drops default transcript update notifications, exposes a security-sensitive guard on the public SDK barrel, and lacks real behavior proof.

## What This Changes

The PR modifies transcript guard/update-mode plumbing, exports `guardSessionManager` from the agent harness runtime barrel, and adds transcript redaction/update tests.

## Best Possible Solution

Land a maintainer-reviewed security fix that routes every transcript persistence sink through a guarded/redacting append path, preserves existing update-mode notifications, keeps any SDK seam narrow or explicitly documented, adds regression coverage and a changelog entry, and includes redacted real-run proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: append a secret-bearing assistant message through `appendExactAssistantMessageToSessionTranscript` or gateway injection and inspect the JSONL, because the shared raw helper serializes `params.message` directly while docs promise persisted transcript redaction.

## Solution Assessment

No. Reusing `guardSessionManager` may be the right direction, but the current diff does not use it in the exact/gateway/Codex/CLI raw append paths and removes the default inline emit, so the safer path is a central guarded append wrapper or explicit redaction before raw writes.
