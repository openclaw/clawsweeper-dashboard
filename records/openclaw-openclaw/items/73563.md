## Summary

Keep this PR open. It addresses a real transcript-redaction gap and has fixed some earlier idempotency/update-mode review comments, but the current branch still has blocking security/API issues: blocked guard writes can be reported as successful, outbound mirror writes still skip active redaction config, the guard is exposed through a broad public SDK barrel, and the branch must reconcile current main's newer injected-transcript hot path before it can be considered complete.

## What This Changes

The PR wraps several transcript append paths with `guardSessionManager`, threads config/update-mode through core, gateway, agent, and Codex mirror call sites, updates Plugin SDK API metadata, and adds transcript redaction/update regression tests.

## Best Possible Solution

Land a corrected, maintainer-reviewed security fix that applies the shared redaction/blocking semantics to every transcript persistence route on current main, passes active `OpenClawConfig` through production mirror callers, reports blocked writes as failures, preserves or intentionally replaces the current bounded injected-transcript path, and keeps any transcript-guard SDK surface narrow and documented.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: register a `before_message_write` hook that blocks an append and call the guarded transcript helper, or configure custom `logging.redactPatterns` and exercise outbound delivery mirroring; the inspected PR head shows the former returns success without a persisted id and the latter still omits config.

## Solution Assessment

No. Reusing `guardSessionManager` is the right direction, but this branch is not yet the best fix because it leaves production config threading incomplete, misreports blocked writes, broadens a public SDK barrel, and is stale against current main's injected-transcript hot path.
