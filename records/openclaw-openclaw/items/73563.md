## Summary

Keep this PR open. The PR addresses a real transcript-redaction gap, but the current diff still leaves security-sensitive transcript paths and SDK-boundary behavior unresolved: the gateway raw append fallback remains outside the guard, outbound mirror callers still omit active config, hook-blocked appends can be reported as successful, and the guard is promoted through a broad public Plugin SDK barrel.

## What This Changes

The PR wraps several session transcript persistence paths with `guardSessionManager`, adds config/update-mode plumbing for transcript helpers and Codex/gateway/agent call sites, updates Plugin SDK API metadata, and adds transcript redaction/update tests.

## Best Possible Solution

Land a corrected, maintainer-reviewed security fix that applies the shared transcript guard or equivalent redaction path to every transcript persistence route, including raw fallbacks; passes the active `OpenClawConfig` through production mirror callers; treats hook-blocked writes as non-success; and keeps transcript guard plumbing on a narrow documented seam instead of a broad public SDK barrel.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: create an oversized append-only transcript that triggers `shouldUseRawAppend()` and inject a secret-bearing assistant message, or register a `before_message_write` hook that blocks an append and observe the guarded helper path returning no message id while callers can still report success.

## Solution Assessment

No. Reusing the existing guard is the right direction, but the current PR is not yet the best fix because it leaves the raw injection fallback unguarded, misses active config on outbound mirror callers, does not consistently handle blocked writes, and broadens the Plugin SDK contract without a narrow documented seam.
