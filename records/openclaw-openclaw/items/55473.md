## Summary

Keep open: the startup-degradation problem is still plausible on current main, but this PR is not merge-ready because it lacks real after-fix behavior proof and still has recovery correctness/security blockers.

## What This Changes

The PR adds gateway startup channel degradation for missing env-backed channel secrets, a recoveryConfig path for secrets.reload, merge-patch support, and gateway reload regression tests.

## Best Possible Solution

Rework startup channel degradation on current main so only truly unavailable channel secrets degrade, operator config edits are preserved during recovery, secrets.reload recovery uses the existing restart/rollback seam, and the revised startup contract is documented.

## Reproduction Assessment

Yes, at source level: current main prepares the full secrets snapshot and startup wraps any activation failure as a fatal startup error, and the focused test asserts that behavior. I did not run tests in this read-only review.

## Solution Assessment

No. The direction is useful, but this PR is not the best implementation because recovery bypasses the current channel restart/rollback path and can replay stale recoveryConfig over explicit channel-disable writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route recovery through the channel restart plan:** `src/gateway/server.impl.ts:1351-1354`
  - body: After degraded startup, the affected channel was never started because the active snapshot has it disabled. This reload path only activates a recovered secrets snapshot and returns, so the channel can become enabled in config/secrets state while its runtime remains offline until a later monitor or restart. Route recovery through the existing reload-plan stop/start path.
  - confidence: 0.88
- **[P2] Preserve explicit disabled-channel writes:** `src/secrets/runtime.ts:310-311`
  - body: During degraded startup, an operator write of `channels.<id>.enabled=false` can produce no recovery delta because the degraded source config already has `enabled=false`. The old recoveryConfig can therefore keep the channel enabled, and a later `secrets.reload` may re-enable a channel the operator explicitly disabled.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: Needs attention because stale recoveryConfig can re-enable an operator-disabled channel after secrets reload.

Concerns:

- **[medium] Prevent recovery from re-enabling disabled channels:** `src/secrets/runtime.ts:310`
  - body: The recoveryConfig forwarding computes changes from the degraded source state, so an explicit enabled=false edit can be invisible while stale recoveryConfig still has the original enabled state; a later secrets.reload can unexpectedly widen the active channel surface.
  - confidence: 0.8
