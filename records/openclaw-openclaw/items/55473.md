## Summary

Keep this PR open: the underlying gateway startup bug still appears real on current main, but the branch is not mergeable as-is because it lacks real behavior proof and the proposed recovery path can leave recovered channels offline or re-enable an operator-disabled channel.

## What This Changes

The PR adds channel-local gateway startup secret degradation, a `recoveryConfig` snapshot path for `secrets.reload`, merge-patch forwarding, and Discord env-ref regression coverage.

## Best Possible Solution

Implement the degradation on current main’s startup secret activator and current `server-aux-handlers` reload seam, degrading only missing active-channel env refs while preserving operator writes and immediately restarting recovered channels through the existing rollback plan.

## Reproduction Assessment

Yes, source-reproducible: current main has a single startup snapshot preparation path and wraps any startup secret failure as fatal, and the focused startup secret test asserts that fatal wrapper behavior. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. The product direction is reasonable, but this branch is not the best implementation because recovery bypasses the current channel restart/rollback seam and can replay stale recovery config over explicit disable writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Restart recovered channels during secrets reload:** `src/gateway/server.impl.ts:1351-1354`
  - body: After degraded startup, the affected channel was never started because the active snapshot has it disabled. This reload path only activates a recovered secrets snapshot and returns, so config/secrets state can show the channel enabled while its runtime remains offline until another restart or monitor cycle. Route recovery through the current reload-plan stop/start path.
  - confidence: 0.88
- **[P2] Preserve explicit channel disables in recovery config:** `src/secrets/runtime.ts:310-311`
  - body: During degraded startup, an operator write of `channels.<id>.enabled=false` can produce no recovery delta because both degraded source configs already have `enabled=false`. The old recovery config can therefore keep the channel enabled, and a later `secrets.reload` may re-enable a channel the operator explicitly disabled.
  - confidence: 0.82
