## Summary

Keep open: current main still fail-fasts active-channel SecretRef startup failures, but this PR is not mergeable as-is because it lacks after-fix real behavior proof and still has recovery-path defects that can leave recovered channels offline or undo explicit channel disables.

## What This Changes

The PR adds startup degradation for missing env-backed channel SecretRefs, a recoveryConfig field for later secrets.reload recovery, merge-patch forwarding, and gateway reload regression tests.

## Best Possible Solution

Rework the fix on current main so only missing env-backed channel secrets degrade startup, permanent/non-channel secret failures remain fatal, recovered channels restart through the existing secrets.reload plan with rollback, operator config edits are preserved, and the gateway secrets docs describe the new startup contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure an active channel token as an env SecretRef, leave the env var unset, and current main calls the startup secrets resolver then throws the fatal startup error that the existing test asserts.

## Solution Assessment

No. The direction is maintainable, but this branch is not the best current implementation because recovery bypasses the current channel restart/rollback seam and can replay stale recovery state over explicit operator disables.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Restart recovered channels during secrets reload:** `src/gateway/server.impl.ts:1351-1354`
  - body: After degraded startup, the affected channel was never started because the active snapshot has it disabled. This reload path only activates a recovered secrets snapshot and returns, so the channel can become enabled in config/secrets state while its runtime remains offline until a later monitor or restart; route recovery through the existing reload-plan stop/start path.
  - confidence: 0.88
- **[P2] Preserve explicit channel disables in recovery config:** `src/secrets/runtime.ts:310-311`
  - body: When an operator writes `channels.<id>.enabled=false` during degraded startup, both the previous and incoming source configs can already contain `enabled: false`, making this delta empty. The old recoveryConfig can therefore keep `enabled: true`, and a later `secrets.reload` may re-enable a channel the operator explicitly disabled.
  - confidence: 0.82
