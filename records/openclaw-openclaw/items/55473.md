## Summary

Keep open. Current main still has the fail-fast startup behavior and maintainer discussion agrees the channel-local degradation problem is worth solving, but this PR should not merge until recovery uses the existing reload restart/rollback path, explicit disables are preserved, and real behavior proof is supplied.

## What This Changes

The PR adds startup-time channel SecretRef degradation with a recoveryConfig snapshot path, merge-patch forwarding for recovery edits, and Discord env-ref regression coverage.

## Best Possible Solution

Implement startup channel degradation on current main by degrading only missing channel env refs, recovering through the existing secrets.reload restart/rollback plan, preserving operator config edits, and documenting the new startup contract.

## Reproduction Assessment

Yes. Source inspection of current main shows startup activation calls `prepareRuntimeSecretsSnapshot` and wraps any startup error as a fatal startup failure, and the existing startup secret test asserts that behavior for a missing env SecretRef.

## Solution Assessment

No. The direction is maintainable, but this branch is not the best implementation because recovery bypasses the current channel restart/rollback seam and can replay stale recoveryConfig over explicit channel-disable writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Restart recovered channels during secrets reload:** `src/gateway/server.impl.ts:1351-1354`
  - body: After degraded startup the affected channel was never started because the active snapshot has it disabled. This reload path only activates a recovered secrets snapshot and returns, so the channel can become enabled in config/secrets state while its runtime remains offline until a later monitor or restart. Route recovery through the existing reload-plan stop/start path.
  - confidence: 0.88
- **[P2] Preserve explicit channel disables in recovery config:** `src/secrets/runtime.ts:310-311`
  - body: During degraded startup, an operator write of `channels.<id>.enabled=false` can produce no delta because the degraded source config already has `enabled=false`. The old recoveryConfig can therefore keep the channel enabled, and a later `secrets.reload` may re-enable a channel the operator explicitly disabled.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The patch can unexpectedly re-enable an operator-disabled channel during secret recovery, which is a concrete channel-surface control risk.

Concerns:

- **[medium] Prevent recovery from re-enabling disabled channels:** `src/secrets/runtime.ts:310`
  - body: The recoveryConfig forwarding computes the patch from the degraded sourceConfig, so an explicit `enabled=false` edit can produce no delta; a later `secrets.reload` may prefer stale recoveryConfig and re-enable a channel the operator disabled, unexpectedly widening the active channel surface.
  - confidence: 0.8
