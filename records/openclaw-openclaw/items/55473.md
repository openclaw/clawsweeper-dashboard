## Summary

Keep open. Current main still fails gateway startup for missing active-channel SecretRefs, but this PR is not mergeable as-is because it predates the current secrets.reload restart path and still has recovery-state correctness blockers.

## What This Changes

The PR adds channel-only startup SecretRef degradation, a recoveryConfig snapshot for secrets.reload recovery, merge-patch support, and gateway reload regression coverage.

## Best Possible Solution

Rework the degradation on current main so only missing active-channel env SecretRefs degrade startup, recovery uses the existing secrets.reload channel restart and rollback plan, operator config edits are preserved, and docs/changelog describe the behavior.

## Reproduction Assessment

Yes. Source inspection shows current main wraps any startup prepareSecretsRuntimeSnapshot failure as `Startup failed: required secrets are unavailable`, and current tests/docs still assert startup fail-fast for unresolved required secrets.

## Solution Assessment

No. The degradation direction is maintainable, but this branch is not the best current implementation because recovery needs to integrate with the current reload restart/rollback path and preserve operator edits in recoveryConfig.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Restart recovered channels during secrets reload:** `src/gateway/server.impl.ts:1350-1351`
  - body: In the degraded-startup path the channel was never started because the active snapshot has `enabled: false`. This reload path swaps to `recoveryConfig` and returns after secret activation, so once the env secret appears `secrets.reload` can report success while the channel runtime remains offline; route recovery through the current reload-plan stop/start path.
  - confidence: 0.88
- **[P2] Preserve explicit channel disables in recoveryConfig:** `src/secrets/runtime.ts:307-308`
  - body: When an operator sets `channels.<id>.enabled=false` during a degraded session, both the previous degraded sourceConfig and the new sourceConfig can already contain `enabled: false`, so this delta is empty and recoveryConfig keeps the old enabled state. A later `secrets.reload` can re-enable a channel the operator explicitly disabled.
  - confidence: 0.78
- **[P3] Document startup degradation for operators:** `src/gateway/server.impl.ts:608-621`
  - body: This changes the startup activation contract from fail-fast to channel-local degradation for one SecretRef failure class, but the PR does not update docs or CHANGELOG. Add operator-facing docs and a changelog entry so startup and recovery behavior are visible.
  - confidence: 0.72
