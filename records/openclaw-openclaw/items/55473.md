## Summary

This PR should stay open: the current-main startup behavior still has the reported all-or-nothing secret failure path, but this branch is not mergeable because it lacks real behavior proof and still has two concrete degraded-recovery defects.

## What This Changes

The PR adds gateway startup degradation for missing active channel env secrets, carries a recoveryConfig for secrets.reload recovery, adds merge-patch forwarding, and adds Discord env-ref regression coverage.

## Best Possible Solution

Implement startup channel degradation on current main by degrading only truly missing channel env refs, preserving operator edits during degraded recovery, routing successful secrets.reload recovery through the existing channel restart/rollback plan, and documenting the revised startup contract.

## Reproduction Assessment

Yes. Current-main source and tests show a missing env SecretRef during startup rejects with the normalized fatal startup error instead of degrading only the affected channel.

## Solution Assessment

No. The direction is maintainable, but this PR is not the best implementation because recovery bypasses the current channel restart/rollback seam and can replay stale recoveryConfig over explicit channel-disable writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Restart recovered channels during secrets reload:** `src/gateway/server.impl.ts:1351-1354`
  - body: After degraded startup, the affected channel was never started because the active snapshot has it disabled. This reload path only activates a recovered secrets snapshot and returns, so the channel can become enabled in config/secrets state while its runtime remains offline until a later monitor or restart. Route recovery through the existing reload-plan stop/start path.
  - confidence: 0.88
- **[P2] Preserve explicit channel disables in recovery config:** `src/secrets/runtime.ts:310-311`
  - body: During degraded startup, an operator write of `channels.<id>.enabled=false` can produce no recovery delta because the degraded source config already has `enabled=false`. The old recoveryConfig can therefore keep the channel enabled, and a later `secrets.reload` may re-enable a channel the operator explicitly disabled.
  - confidence: 0.8

## Security Review

Status: needs_attention
