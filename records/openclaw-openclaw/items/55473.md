## Summary

Keep open. The underlying gateway behavior still appears real on current main, but this PR is not mergeable as-is: the branch is dirty against current main and its recovery path conflicts with the newer secrets.reload channel restart and rollback flow, while recoveryConfig forwarding can still undo an operator's explicit channel disable during a degraded session.

## What This Changes

The PR adds a startup path that disables active channels with missing env-backed SecretRefs, stores a recoveryConfig snapshot for later secrets.reload recovery, adds a merge-patch helper, and extends gateway reload tests for the degraded startup path.

## Best Possible Solution

Rework the degradation behavior on top of current main by adding the startup channel-isolation seam where createRuntimeSecretsActivator currently normalizes startup failures, then recover through the current server-aux-handlers secrets.reload reload-plan path so channels restart immediately and rollback remains intact. Recovery state should preserve operator config writes made during the degraded session, with tests covering missing env refs, policy failures, explicit disables, and recovered channel start.

## Reproduction Assessment

Yes. Configure an active channel with an env-backed token SecretRef such as `channels.discord.token`, leave the env var unset, and start the gateway; current main resolves active channel SecretRefs and createRuntimeSecretsActivator turns the startup preparation failure into the fatal startup error.

## Solution Assessment

No. The PR's product direction is reasonable, but this branch is not the best current implementation because it predates the current secrets.reload restart/rollback seam and can replay stale recoveryConfig state over explicit operator disable writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Restart recovered channels during secrets reload:** `src/gateway/server.impl.ts:1348-1355`
  - body: In the degraded-startup path the channel was never started because the active snapshot has `enabled: false`. This reload branch swaps in the recovery config and can return success without diffing/restarting channel runtimes, so after the env secret appears the channel can remain offline even though `secrets.reload` reports ok. Route recovery through the current reload-plan stop/start path.
  - confidence: 0.88
- **[P2] Preserve explicit channel disables in recoveryConfig:** `src/secrets/runtime.ts:304-309`
  - body: When an operator explicitly sets `channels.<id>.enabled=false` during a degraded session, both the old degraded sourceConfig and the new sourceConfig already contain `enabled: false`, so this delta is empty and recoveryConfig keeps the original enabled state. A later `secrets.reload` can re-enable a channel the operator disabled; track that explicit disable or merge against the pre-degrade baseline.
  - confidence: 0.78
