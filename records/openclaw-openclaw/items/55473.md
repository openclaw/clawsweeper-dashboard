## Summary

Keep this PR open: the underlying gateway secret-startup problem is still real on current main, but the branch is not mergeable as-is because degraded recovery bypasses the current secrets.reload restart path and still has recovery-config durability defects.

## What This Changes

The PR adds startup-only channel SecretRef degradation, a recoveryConfig snapshot field with merge-patch forwarding, and regression coverage for Discord env-ref startup recovery.

## Best Possible Solution

Rework the startup degradation on current main so only missing channel env refs degrade, non-channel/permanent secret failures remain fatal, recovered channels restart through the existing secrets.reload plan, operator config edits are preserved, and the gateway secrets docs describe the new contract.

## Reproduction Assessment

Yes. Source inspection shows current main still wraps startup SecretRef preparation failures as fatal, and the existing startup secret test asserts that behavior; the repro path is an active channel token SecretRef whose backing env var is unset.

## Solution Assessment

No. The direction is maintainable, but this branch is not the best current implementation because recovery bypasses the current channel restart/rollback seam and can replay stale recovery state over operator disables.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Restart recovered channels during secrets reload:** `src/gateway/server.impl.ts:1351-1354`
  - body: After degraded startup the affected channel was never started because the active snapshot has it disabled. This reload path only activates a recovered secrets snapshot and returns, so the channel can become enabled in config/secrets state while its runtime remains offline until some later monitor or restart. Route recovery through the existing reload-plan stop/start path.
  - confidence: 0.88
- **[P2] Preserve explicit channel disables in recovery config:** `src/secrets/runtime.ts:310-311`
  - body: When an operator writes `channels.<id>.enabled=false` during degraded startup, both the previous and incoming source configs can already contain `enabled: false`, making this delta empty. The old recoveryConfig can therefore keep `enabled: true`, and a later `secrets.reload` may re-enable a channel the operator explicitly disabled.
  - confidence: 0.8
