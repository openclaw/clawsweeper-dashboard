## Summary

Keep open: the PR addresses a real source-reproducible ACP metadata race and now has adequate terminal proof, but the patch introduces a blocking direct `saveSessionStore` regression that can discard caller-supplied ACP metadata.

## What This Changes

The PR changes session-store saves to reconcile ACP metadata from fresh disk state for stale writers and adds stale-writer regression tests.

## Best Possible Solution

Land baseline-aware ACP reconciliation that fixes stale update-helper writes while keeping direct `saveSessionStore` faithful to the caller-provided store unless an explicit ACP removal path is used.

## Reproduction Assessment

Yes, source-level: current main snapshots ACP metadata before the mutator and then saves without final disk ACP reconciliation, and the PR body supplies a temp-store terminal proof showing baseline loss versus PR-head preservation. I did not execute tests because this review is read-only.

## Solution Assessment

No, not as patched: final write-boundary reconciliation is the right area, but direct `saveSessionStore` calls must not treat disk ACP as authoritative when no writer-load baseline exists.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve caller ACP metadata on direct saves:** `src/config/sessions/store.ts:453`
  - body: When `saveSessionStore` is called directly, `hasRefreshBaseline` is false so this defaults to reading disk and then treats that disk ACP map as authoritative. A compatibility/offline rewrite or plugin runtime save that passes a store with new or updated `entry.acp` will have that metadata deleted or replaced if the current disk snapshot lacks it or has an older value. Gate authoritative refresh to baseline-aware writes, or otherwise preserve caller-provided ACP for direct saves, and add a regression test for direct `saveSessionStore`.
  - confidence: 0.9
