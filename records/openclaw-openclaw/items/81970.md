## Summary

This PR should stay open: it fixes a real stale-writer ACP metadata race, but the current patch introduces a direct-save regression that can drop or replace caller-provided ACP metadata.

## What This Changes

The PR reconciles ACP metadata from fresh disk state during stale session-store saves and adds session-store regression tests for ACP metadata preservation, updates, removals, and explicit drops.

## Best Possible Solution

Keep baseline-aware stale-writer ACP reconciliation, but make direct `saveSessionStore` faithful to the caller-provided store unless an explicit ACP removal path opts out.

## Reproduction Assessment

Yes, source-level: current main can save a stale `updateSessionStore` snapshot without a final fresh-disk ACP read, and the PR body supplies terminal proof showing baseline loss versus PR-head preservation. I did not execute tests because this review is read-only.

## Solution Assessment

No, not as currently patched: final write-boundary reconciliation is the right fix area, but direct saves need to preserve caller-supplied ACP when no writer-load baseline exists.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve caller ACP metadata on direct saves:** `src/config/sessions/store.ts:453`
  - body: When `saveSessionStore` is called directly, there is no refresh baseline, so this branch reads disk and treats that ACP snapshot as authoritative. A compatibility/offline rewrite or runtime save that passes new or updated `entry.acp` can have that metadata removed or replaced if disk lacks it or has an older value; gate authoritative disk refresh to baseline-aware writes or otherwise preserve caller ACP for direct saves, with a focused regression test.
  - confidence: 0.9
