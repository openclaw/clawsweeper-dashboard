## Summary

Keep open: the stale-writer fix is directionally right, but the PR currently makes direct whole-store saves treat existing disk ACP metadata as authoritative and can silently strip caller-supplied ACP metadata. The external proof is also still focused test/typecheck output rather than a real ACP/Gateway run.

## What This Changes

The PR changes session-store saving to reconcile ACP metadata from a fresh disk snapshot at save time and adds stale-writer regression coverage.

## Best Possible Solution

Keep the write-boundary reconciliation, but scope authoritative fresh-disk ACP reconciliation to baseline-backed stale writer saves, preserve caller-supplied ACP on direct offline saves, add direct-save regression coverage, and then add real ACP/Gateway behavior proof.

## Reproduction Assessment

Yes for source-level reproduction: current main can load a stale store, another writer can add entry.acp on disk, and the stale writer then saves without re-reading disk ACP metadata. I did not run tests because this review was read-only.

## Solution Assessment

No, not as proposed: the updateSessionStore direction is narrow, but making direct saves authoritative to the old disk snapshot can drop ACP metadata supplied by compatibility/offline rewrites. The safer fix is baseline-only disk reconciliation plus direct-save preservation coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep direct saves from dropping supplied ACP metadata:** `src/config/sessions/store.ts:453`
  - body: With no refresh baseline, direct `saveSessionStore()` calls now always treat disk as authoritative. If a compatibility/offline rewrite supplies an `entry.acp` that is not already in the current file, `reconcileAcpMetadataAuthoritatively` deletes it before serialization, so the save silently loses ACP metadata; limit this path to baseline-backed stale writer saves or preserve the caller-provided store for direct saves.
