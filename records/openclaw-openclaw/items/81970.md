## Summary

Keep this PR open: it targets a real source-level ACP session-store race, but the external-PR proof is still test/typecheck-only and the current diff appears to break direct `saveSessionStore` compatibility by deleting caller-supplied ACP metadata when the existing file lacks it.

## What This Changes

The PR updates session-store saves to reconcile ACP metadata from fresh disk snapshots during stale writes and adds ACP stale-writer regression tests.

## Best Possible Solution

Keep the final write-boundary reconciliation for updateSessionStore, preserve direct saveSessionStore compatibility/offline rewrites, add regression coverage for direct-save ACP behavior, and require redacted live ACP/Gateway proof before merge.

## Reproduction Assessment

Yes, there is a high-confidence source-level path: current main can load a stale store, another writer can add entry.acp on disk, and the stale writer then saves without re-reading disk ACP metadata. I did not execute tests because this review was read-only.

## Solution Assessment

No as currently written: reconciling ACP metadata at the final updateSessionStore write boundary is the right narrow direction, but applying authoritative disk absence to direct saveSessionStore calls can silently drop caller-supplied ACP metadata. The safer fix is to scope disk-authoritative reconciliation to writer-loaded updates or otherwise preserve direct-save input unless the caller explicitly opts into disk authority.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve ACP when direct saves provide it:** `src/config/sessions/store.ts:453`
  - body: When `saveSessionStore()` calls this helper directly, it does not pass a refresh baseline or fallback snapshot, so `shouldRefreshAcpMetadataFromDisk` becomes true and `reconcileAcpMetadataAuthoritatively()` treats the current file as authoritative. If the file already exists without `entry.acp`, a compatibility/offline rewrite that supplies ACP metadata in the store argument has that metadata deleted before serialization. Direct saves should keep caller-provided ACP unless the caller opted into disk-authoritative reconciliation or explicit ACP removal.
  - confidence: 0.82
