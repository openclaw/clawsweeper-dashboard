## Summary

Keep open: the PR addresses a real ACP session-store race and has sufficient terminal proof, but the current patch still introduces a direct `saveSessionStore` regression that should block merge.

## What This Changes

The PR changes session-store saves to reconcile ACP metadata from fresh disk state and adds stale-writer regression tests for ACP session metadata.

## Best Possible Solution

Land baseline-aware ACP reconciliation that fixes stale update-helper writes while keeping direct `saveSessionStore` faithful to caller-provided store contents unless an explicit ACP removal path is used.

## Reproduction Assessment

Yes, source-level: current main snapshots ACP metadata before the mutator and then saves without final disk ACP reconciliation, and the PR body supplies temp-store terminal proof showing baseline loss versus PR-head preservation.

## Solution Assessment

No, not as patched: final write-boundary reconciliation is the right area, but direct `saveSessionStore` calls must not treat disk ACP as authoritative when no writer-load baseline exists.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve caller ACP metadata on direct saves:** `src/config/sessions/store.ts:453`
  - body: When `saveSessionStore` is called directly, the patch has no writer-load baseline, so it defaults to reading disk and treats that disk ACP map as authoritative. A compatibility/offline rewrite or plugin runtime save that passes new or updated `entry.acp` will have that metadata deleted or replaced when the current disk snapshot lacks it or has an older value. Gate authoritative refresh to baseline-aware writes, or preserve caller-provided ACP for direct saves, and add a regression test for direct `saveSessionStore`.
  - confidence: 0.9
