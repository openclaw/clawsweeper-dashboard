## Summary

Keep open, but this patch should not merge as-is: source inspection confirms the outbound queue recovery bug still exists on current main, while this PR head is malformed, drops current replay metadata, and lacks after-fix real behavior proof.

## What This Changes

The PR adds legacy outbound delivery-queue normalization/tests to preserve replay metadata and also adds gateway agents.create readiness/runtime config refresh code.

## Best Possible Solution

Land a current-main outbound-only queue normalizer that preserves every QueuedDelivery replay field, with focused storage/recovery regression tests, and split any gateway readiness or secrets refresh work into a separate review.

## Reproduction Assessment

Yes, by source inspection. Current main reads queue JSON as QueuedDelivery and only backfills lastAttemptAt, while the recovery path forwards replay fields that a normalizer must preserve.

## Solution Assessment

No. Queue-storage normalization is the right boundary, but this PR is not the best mergeable solution because it bundles malformed gateway runtime refresh code and still drops current queue replay fields.

  - confidence: 0.96

Because load and fail paths persist normalized entries, retry/recovery can silently lose threading, formatting, media/session policy, and caller scopes.
  - confidence: 0.95
