## Summary

Keep open. Current main still lacks the legacy delivery-queue normalization this PR is trying to add, but the PR branch is not mergeable because it mixes unrelated gateway readiness/runtime secret refresh work, structurally breaks gateway startup, calls readiness helpers that are not defined in the current API surface, and still drops queued replay metadata during normalization.

## What This Changes

The PR branch adds legacy outbound delivery-queue normalization and regression tests, changes queued mirror typing/test helpers, and adds gateway agent-create runtime readiness refresh code and tests.

## Best Possible Solution

Land the outbound queue work as a narrow storage/recovery change: normalize legacy queue records on load and before failure bookkeeping, preserve all replay fields, add focused legacy-file regression tests, and move the gateway readiness/runtime refresh work to a separate owner-reviewed PR if it is still needed.

## Reproduction Assessment

Yes. A legacy queue JSON containing payload/target/attempt/createdAt can be inspected through loadPendingDeliveries or failDelivery on current main; the source shows those paths do not canonicalize those fields before replay or retry bookkeeping.

## Solution Assessment

No. The storage normalizer is the right boundary, but this PR is not the best mergeable form because it mixes unrelated gateway runtime work and the proposed normalization still loses queued replay options.
