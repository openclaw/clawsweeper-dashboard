Keep this PR open for human handling: it targets a real outbound recovery bug, but the current branch is conflicting, structurally breaks gateway startup, mixes in unrelated runtime config/secrets refresh behavior, and still drops queued replay metadata.

## What This Changes

The PR adds legacy outbound delivery queue normalization and regression tests, plus gateway agent-create readiness/runtime config refresh changes.

## Best Possible Solution

Land a narrow outbound-only queue normalizer on current main that remaps legacy entries without dropping any QueuedDeliveryPayload or recovery-state fields, with focused regression tests; split gateway readiness/runtime refresh into a separate review.

## Reproduction Assessment

Yes, by source inspection. Current main still reads queued JSON as QueuedDelivery in fail/load paths while recovery forwards rich replay metadata, so legacy remapping must be done before retry or replay writes.

## Solution Assessment

No. The storage normalizer is the right boundary, but this branch is not the best mergeable solution because it bundles malformed gateway runtime refresh code and does not preserve the full current-main queue replay surface.
