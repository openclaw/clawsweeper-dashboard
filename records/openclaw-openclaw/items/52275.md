## Summary

Keep open. Current main still lacks the legacy delivery-queue normalization this PR is trying to add, but the branch is not merge-ready: it mixes unrelated gateway readiness/runtime secrets refresh work, structurally breaks gateway startup, calls readiness helpers that are not present in the current context API, and still drops queued replay metadata during normalization.

## What This Changes

The PR branch adds legacy outbound delivery-queue normalization and regression tests, changes queued mirror typing/test helpers, and adds gateway agent-create runtime readiness refresh code and tests.

## Best Possible Solution

Land the outbound queue work as a narrow storage/recovery change: normalize legacy queue records on load and before failure bookkeeping, preserve all replay fields, add focused legacy-file regression tests, and move the gateway readiness/runtime refresh work to a separate owner-reviewed PR if still needed.

## Reproduction Assessment

Yes. A legacy queue JSON containing payload/target/attempt/createdAt can be inspected through loadPendingDeliveries or failDelivery on current main; source shows those paths do not canonicalize those fields before replay or retry bookkeeping.

## Solution Assessment

No. The storage normalizer is the right boundary, but this PR is not the best mergeable form because it mixes unrelated gateway runtime work and the proposed normalization still loses queued replay options.
