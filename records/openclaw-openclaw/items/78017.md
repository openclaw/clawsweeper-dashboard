## Summary

Keep open: current main and the latest shipped release still pin `@earendil-works/pi-tui` 0.74.0, whose published render path clears terminal scrollback with `\x1b[3J` when `firstChanged < viewportTop`; OpenClaw's TUI streaming path still updates the same markdown component via `setText`.

## What This Changes

Fix `openclaw tui` so long streaming assistant messages do not disappear from terminal scrollback after content-driven full redraws.

## Best Possible Solution

Land or approve a pi-tui fix that preserves scrollback for content-driven full redraws, bump OpenClaw to that dependency, and verify with `PI_DEBUG_REDRAW=1 openclaw tui` plus focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: OpenClaw still streams assistant text through `ChatLog.updateAssistant` to pi-tui `Markdown.setText`, and the pinned pi-tui tarball still clears scrollback when `firstChanged < prevViewportTop`. I did not run a live terminal recording against current main.

## Solution Assessment

Yes, the narrow maintainable fix is in pi-tui's redraw behavior, then OpenClaw should consume it by approved version bump or dependency patch. An OpenClaw-only workaround around markdown streaming would be more fragile than fixing the render engine branch that emits `\x1b[3J`.
