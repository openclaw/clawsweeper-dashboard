## Summary

Keep open: current main has moved from `@mariozechner/pi-tui` 0.73.0 to `@earendil-works/pi-tui` 0.74.0, but the published 0.74.0 render path still calls `fullRender(true)` with `\x1b[3J` when `firstChanged < viewportTop`. OpenClaw streaming assistant updates still reuse `setText` on the same markdown component, so the reported scrollback-wipe path remains source-reproducible.

## What This Changes

The issue reports long TUI assistant messages disappearing from terminal scrollback because pi-tui clears scrollback on content-driven full redraws.

## Best Possible Solution

Land a pi-tui version or approved dependency patch that prevents content-driven full redraws from clearing scrollback, then verify with `PI_DEBUG_REDRAW=1 openclaw tui` and focused regression coverage.

## Reproduction Assessment

Yes, source-backed: current main still streams assistant text through `setText`, and the pinned pi-tui 0.74.0 tarball still clears scrollback for `firstChanged < viewportTop`. I did not perform a live terminal run, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

No current-main solution is in place; the dependency rename/upgrade does not remove the `\x1b[3J` content-redraw path. The maintainable fix should happen upstream or through an explicitly approved dependency patch/upgrade boundary.
