## Summary

Keep open: current main and the latest release still pin `@earendil-works/pi-tui` 0.78.0, whose renderer clears terminal scrollback from the content-driven full-redraw path described in the report. The related upstream fix is closed unmerged, and the newer published 0.79.6 package still contains the same `ESC[3J`/`firstChanged < prevViewportTop` pattern, so this is not implemented or safely superseded.

## What This Changes

Fix `openclaw tui` so long assistant messages remain recoverable in terminal scrollback after content-driven pi-tui redraws.

## Best Possible Solution

Choose a maintainer-approved repair path, then prove in a PTY or real terminal that content-driven redraws no longer emit `ESC[3J` or erase scrollback while preserving resize behavior.

## Reproduction Assessment

Yes, at source level: current OpenClaw still streams assistant text through pi-tui Markdown updates, and the pinned renderer still clears scrollback when `firstChanged` is above the previous viewport. I did not run a live current-main PTY reproduction in this read-only review.

Unclear. The reported root-cause direction is credible, but the best maintainable fix needs a maintainer decision between upstream dependency repair, dependency patching, or a local OpenClaw-side streaming markdown/render workaround.
