## Summary

Keep open. Current main and the latest release still allow the managed memory-core dreaming cron path to swallow workspace-level failures and return a handled processed result, so the reported “ok run but no artifacts/events” symptom is still a plausible unresolved product bug rather than an obsolete report.

## What This Changes

The issue reports that the memory-core managed dreaming cron on OpenClaw 2026.4.24 reports successful runs with eligible REM/deep harness data but writes no `DREAMS.md`, `memory/dreaming/*` phase reports, or `memory.dream.completed` events.

## Best Possible Solution

Make the managed memory-core dreaming sweep enforce an observable success invariant: when dreaming is enabled and eligible material exists, the cron-triggered run should either write the expected phase report, diary, and completion-event artifacts, or return a clear failure/diagnostic that cron history cannot display as a generic ok run.

## Reproduction Assessment

Yes. The issue supplies a high-confidence operational reproduction path with enabled cron history, non-empty rem-harness/status output, and artifact/event inspection; a focused regression test can seed eligible memory, run the cron-triggered sweep, and assert artifacts/events or a failure result.

## Solution Assessment

No current patch is attached, and current main is not sufficient. The narrowest maintainable solution belongs in `extensions/memory-core` around cron-triggered artifact writes and failure propagation, without adding a core plugin API unless multiple owners need the same seam.
