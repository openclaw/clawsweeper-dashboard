## Summary

Keep open. Current main still has a silent-success path where cron-triggered memory-core dreaming can log per-workspace artifact/event write failures and still return the generic processed result, so an `ok` cron run does not prove that `DREAMS.md`, phase reports, or completion events were written.

## What This Changes

The issue reports that managed memory-core dreaming cron runs can report success with eligible REM/deep material while producing no new dreaming artifacts or completion events.

## Best Possible Solution

Enforce a managed-dreaming success invariant: when an enabled cron sweep attempts artifact or event writes, it should either produce the expected outputs or return a clear diagnostic failure while preserving legitimate disabled, limit-zero, missing-workspace, and no-material skips.

## Reproduction Assessment

Yes. The issue gives an operational reproduction with `ok` cron runs, non-empty `rem-harness` and status output, and missing artifact/event checks; current source also supports a focused unit reproduction by forcing phase or report writes to fail.

## Solution Assessment

Not applicable because no patch is attached. The narrowest maintainable fix is in `extensions/memory-core` failure propagation and regression tests around cron-triggered phase/report writes, not a broader scoring, backfill, or core plugin API rewrite.
