## Summary

Keep open. Current main still lets the managed memory-core dreaming cron return a handled processed result after workspace-level failures, so an all-failed sweep can still look successful while writing no documented artifacts or completion events; the related open reports cover adjacent but not identical symptoms.

## What This Changes

The issue requests that memory-core dreaming cron runs with eligible REM/deep material either write DREAMS.md, phase report, and memory.dream.completed artifacts or report a clear failure instead of an ok cron result.

## Best Possible Solution

Make the managed memory-core dreaming sweep enforce an observable success invariant: when dreaming is enabled and eligible material exists, the cron-triggered run should write the expected phase report, diary, and completion-event artifacts or return a clear failure/diagnostic that cron history cannot display as a generic ok run.

## Reproduction Assessment

Yes. The issue supplies a high-confidence operational reproduction path with enabled cron history, non-empty rem-harness/status output, and artifact/event inspections; a focused regression test can seed eligible memory, run the cron-triggered sweep, and assert reports/events or a failure result.

## Solution Assessment

No current patch is attached, and current main is not sufficient. The narrowest maintainable solution belongs in extensions/memory-core around cron-triggered artifact writes and failure propagation, without adding a core plugin API unless multiple owners need the same seam.
