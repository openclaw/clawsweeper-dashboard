## Summary

Keep open. Current main still has a source-level path where memory-core dreaming artifact/event write failures are logged per workspace while the cron handler returns the generic processed result, so a successful cron history can still hide missing dreaming outputs.

## What This Changes

The issue reports that managed memory-core dreaming cron runs return ok on 2026.4.24 with eligible REM/deep material but produce no new `DREAMS.md`, `memory/dreaming/*`, or `memory.dream.completed` artifacts.

## Best Possible Solution

Make managed dreaming success reflect the artifact/event contract: successful runs should write the expected phase reports, diary entries, or completion events when material exists, and artifact write failures should surface a clear failed/diagnostic result while preserving legitimate disabled, limit-zero, no-workspace, and no-material skips.

Yes, source-level. Current main has a deterministic path where phase/report write failures are caught per workspace and the handler still returns a handled processed result; I did not run tests because this review is read-only.
