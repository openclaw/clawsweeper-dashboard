## Summary

Keep open. Current main still lets cron-triggered memory-core dreaming catch per-workspace artifact/write failures and return the generic processed result, so successful cron history is not reliable proof that `DREAMS.md`, `memory/dreaming` reports, or `memory.dream.completed` events were written.

## What This Changes

The issue reports that the managed memory-core dreaming cron can report successful runs with eligible REM/deep material while producing no new dreaming artifacts or completion events.

## Best Possible Solution

Managed dreaming should only report success when an enabled sweep with eligible material either writes the expected artifact/event surface or returns a clear diagnostic failure, while preserving legitimate disabled, limit-zero, missing-workspace, and no-material skips.

## Reproduction Assessment

Yes. The issue gives a high-confidence operational path with ok cron runs, non-empty `rem-harness`/status output, and missing artifact/event checks; current source also exposes a deterministic unit path by forcing phase/report writes to fail and observing that the outer handler still returns processed.

## Solution Assessment

No current patch is attached. The narrowest maintainable fix belongs in `extensions/memory-core` failure propagation and regression tests around cron-triggered phase/report writes, not in a broader scoring, backfill, or core plugin API rewrite.
