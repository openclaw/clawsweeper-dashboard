Keep open. Current main still has a silent-success path where cron-triggered memory-core dreaming can log workspace write failures and return the generic processed result, so successful cron history does not prove that phase reports, Dream Diary entries, or completion events were written.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/72546. Current main lets `runShortTermDreamingPromotionIfTriggered` catch per-workspace dreaming artifact/event write failures, log them, and still return `memory-core: short-term dreaming processed`, which can make managed cron history show ok even when no `DREAMS.md`, `memory/dreaming/*`, or `memory.dream.completed` outputs were written. Implement the fix in the memory-core plugin, not core, and preserve legitimate skipped states for disabled dreaming, `limit=0`, missing workspace, and no eligible material. Add focused regression tests that force phase/report/event write failure and assert the cron result surfaces a diagnostic failure instead of generic success. Be careful with detached Dream Diary narrative semantics: do not accidentally make optional background narrative generation fail every cron run when there is no subagent or no diary material; focus on attempted artifact/event writes and clear reporting. Include a single-line user-facing `CHANGELOG.md` fix entry. Treat #74639 as related context only unless the same invariant naturally covers it; do not attempt the broad scoring/session-scan work from #71656.
