## Summary

Keep open. Current main still has a plausible silent-success path for the reported symptom: the managed memory-core dreaming cron can catch workspace-level failures, log them, and still return a handled `short-term dreaming processed` result even though artifact and completion-event writes happen only inside that per-workspace path. The related open reports cover adjacent dreaming quality and diary symptoms but do not supersede this missing-artifact invariant.

## What This Changes

The issue asks memory-core managed dreaming cron runs to stop reporting success without writing `DREAMS.md`, phase reports, or `memory.dream.completed` events when eligible REM/deep material exists.

## Best Possible Solution

Make the managed memory-core dreaming sweep enforce an observable success invariant: when dreaming is enabled and eligible material exists, a cron-triggered run should either write the expected phase report, diary, and completion-event artifacts or return a clear failure/diagnostic that cron history cannot display as a generic successful processed run.

## Reproduction Assessment

Yes. The issue provides a high-confidence operational reproduction path with cron history, non-empty harness/status output, and artifact/event inspection; a focused test can seed eligible memory and force the cron-triggered sweep through an artifact/write failure path.

## Solution Assessment

No current patch is attached, and current main is not sufficient. The narrowest maintainable fix belongs in `extensions/memory-core` around cron-triggered artifact writes and failure propagation, without adding a core plugin API unless multiple plugin owners need the same contract.
