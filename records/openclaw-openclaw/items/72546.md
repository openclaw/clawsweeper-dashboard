## Summary

Keep open. Current main still allows managed memory-core dreaming to log workspace write failures while returning the generic processed result, so an `ok` cron history can still hide missing Dream Diary, phase-report, and `memory.dream.completed` artifacts.

## What This Changes

The issue reports that managed memory-core dreaming cron runs on 2026.4.24 show `ok` with eligible REM/deep data but produce no new `DREAMS.md`, `memory/dreaming/*`, or dream completion events.

## Best Possible Solution

Make managed dreaming success reflect artifact/event writes: a sweep that attempts light, REM, or deep output should either write the expected artifacts/events or surface a clear cron-visible diagnostic failure while preserving legitimate disabled, limit-zero, missing-workspace, and no-material skips.

## Reproduction Assessment

Yes. The issue provides a high-confidence operational reproduction, and current source exposes a deterministic unit path by forcing phase/report/event writes to fail and observing that the outer cron handler can still return the generic processed result.

## Solution Assessment

Not applicable because there is no patch attached. The narrowest maintainable fix is in `extensions/memory-core` failure propagation and regression tests around cron-triggered phase/report writes, not a broader scoring, backfill, or plugin API rewrite.
