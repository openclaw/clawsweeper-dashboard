## Summary

Keep open. Current main still has a plausible memory-core path where the managed dreaming cron can return a handled processed result even when workspace-level failures prevent the documented artifacts and completion events from being written, and the related open reports do not fully supersede this no-artifacts symptom.

## What This Changes

The issue requests that memory-core dreaming cron runs with eligible REM/deep material either write the expected DREAMS.md, memory/dreaming phase report, and memory.dream.completed artifacts or report a clear failure instead of an ok cron result.

## Best Possible Solution

Make the managed memory-core dreaming sweep enforce an observable success invariant: when dreaming is enabled and eligible material exists, the cron-triggered run should write the expected phase report, diary, and completion-event artifacts or return a clear failure/diagnostic that cron history cannot display as a generic ok run.

## Reproduction Assessment

Yes. The issue supplies a high-confidence operational reproduction path with enabled cron history, non-empty rem-harness/status output, and artifact/event inspections; a focused regression test can seed eligible memory, run the cron-triggered sweep, and assert reports/events or a failure result.

## Solution Assessment

No current patch exists, and current main is not sufficient. The narrowest maintainable solution belongs in extensions/memory-core around cron-triggered artifact writes and failure propagation, without adding a core plugin API unless multiple owners need the same seam.
