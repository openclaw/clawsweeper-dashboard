## Summary

Keep open. Current main still has a plausible silent-success path: cron-triggered memory-core dreaming can catch workspace-level failures and return a generic processed result even though phase report, diary, and completion-event writes happen inside that failing path.

## What This Changes

The report asks memory-core dreaming cron to stop reporting success when eligible harness data exists but DREAMS.md, phase reports, and completion events are not written.

## Best Possible Solution

Make the managed memory-core dreaming sweep enforce an observable success invariant: with dreaming enabled and eligible material present, cron should either write the expected artifacts/events or return a clear failure/diagnostic instead of a generic processed result.

## Reproduction Assessment

Yes. The issue provides a high-confidence operational reproduction path, and current main exposes a deterministic unit-test boundary by forcing the cron-triggered sweep through an artifact/write failure path.

## Solution Assessment

No patch is attached. The narrowest maintainable solution belongs in extensions/memory-core around cron-triggered artifact writes and failure propagation, without adding a core plugin API.
