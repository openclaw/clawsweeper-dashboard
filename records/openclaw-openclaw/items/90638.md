## Summary

Keep open: current main still leaves a Slack DM path where accepted chat senders can have text recovery commands marked unauthorized, and the silent unauthorized command handling still matches the reported `/compact`, `/new`, and `/stop` failure mode.

## What This Changes

The issue reports that Slack DM text commands such as `/compact`, `/new`, and `/stop` are accepted as normal chat instead of being handled as session recovery commands.

## Best Possible Solution

Keep this issue open until maintainers choose the Slack DM command authorization boundary and land focused Slack coverage proving `/compact`, `/new`, and `/stop` either execute for accepted recovery senders or visibly deny without becoming plain chat.

## Reproduction Assessment

Yes at source level: Slack DM chat admission can be open while text control-command authorization remains false, and unauthorized whole-message recovery commands can return without a reply. I did not establish a live Slack workspace reproduction against current main.

## Solution Assessment

Unclear until maintainers choose the exact Slack DM command authorization boundary. The best fix should execute or visibly reject accepted-user session recovery commands without exposing owner-only commands to unintended senders.
