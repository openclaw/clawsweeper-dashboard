## Label Justifications

- `P1`: Slack users can lose the in-channel recovery path for wedged sessions, which is a broken channel workflow with real user impact.
- `impact:message-loss`: The reported Slack DM slash-command messages produce no useful command acknowledgement or command effect.
- `impact:session-state`: The affected `/compact`, `/new`, and `/stop` commands mutate or recover session state.

## Summary

Keep open: current main and the latest release still leave Slack DM text recovery commands dependent on command-owner authorization rather than the accepted DM admission path, so an accepted DM user can get no visible recovery action from `/compact`, `/new`, or `/stop`. The CLI/RPC discoverability part is related but already tracked separately at https://github.com/openclaw/openclaw/issues/90640.

## What This Changes

This issue asks for Slack DM text `/compact`, `/new`, and `/stop` to execute or visibly fail instead of silently leaving wedged Slack sessions without an in-channel recovery path.

## Best Possible Solution

Define a Slack DM recovery-command policy that lets accepted DM users run session-scoped `/compact`, `/new`, and `/stop` or receive a visible denial, while preserving owner-only gates for privileged commands and leaving the CLI/RPC work to https://github.com/openclaw/openclaw/issues/90640.

## Reproduction Assessment

Yes, by source inspection rather than live Slack execution. Current Slack DM ingress can admit a sender for chat while command authorization stays tied to command-owner allowlists, and non-native unauthorized command text can terminate without a visible denial.

## Solution Assessment

Unclear until maintainers choose the Slack DM command authorization boundary. The maintainable fix should be narrower than a blanket owner bypass: session recovery commands should execute or visibly reject without exposing owner-only commands to unintended senders.
