## Label Justifications

- `P1`: Slack users can lose the in-channel recovery path for wedged sessions, which is a broken channel workflow with real user impact.
- `impact:message-loss`: The reported slash-command messages do not produce the expected command effect or useful command acknowledgement in Slack DMs.
- `impact:session-state`: The affected `/compact`, `/new`, and `/stop` commands are session recovery and mutation controls.

## Summary

Keep open: current main and v2026.6.5 still let Slack DMs be accepted for chat while text recovery commands can remain unauthorized, so `/compact`, `/new`, and `/stop` do not reliably execute from the affected Slack DM path.

## What This Changes

The issue asks for Slack DM text slash commands such as `/compact`, `/new`, and `/stop` to execute or visibly reject instead of falling through as inert chat text.

## Best Possible Solution

Make Slack DM recovery commands follow the accepted DM session path with a visible execution or denial, add focused Slack DM coverage for `/compact`, `/new`, and `/stop`, and keep CLI/RPC discoverability in https://github.com/openclaw/openclaw/issues/90640.

## Reproduction Assessment

Yes. Source inspection shows a Slack DM can be admitted for chat while text control commands remain unauthorized, and the command handlers then skip the recovery action.

## Solution Assessment

Unclear until maintainers choose the Slack DM command authorization boundary. The maintainable fix should restore session recovery commands or visible denial without granting owner-only Slack commands to unintended senders.
