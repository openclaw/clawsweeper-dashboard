## Label Justifications

- `P1`: Slack users can lose the in-channel recovery path for wedged sessions, which is a broken channel workflow with real user impact.
- `impact:message-loss`: The reported slash-command messages produce no useful command acknowledgement or effect on Slack DMs.
- `impact:session-state`: `/compact`, `/new`, and `/stop` are session recovery and mutation commands, so the failure directly affects session state.

## Summary

Keep open: current main and the latest release still show a Slack DM command path where accepted DM senders can fail command authorization, leaving recovery commands such as `/compact`, `/new`, and `/stop` ineffective despite the documented text-command behavior.

## What This Changes

The issue requests that Slack DM text slash commands for session recovery be intercepted and executed or visibly rejected instead of falling through or silently no-oping.

## Best Possible Solution

Align Slack DM text recovery commands with the accepted DM session path, return a visible acknowledgement or denial, add focused `/compact`, `/new`, and `/stop` Slack DM coverage, and leave the CLI/RPC discoverability work to https://github.com/openclaw/openclaw/issues/90640.

## Reproduction Assessment

Yes, by source inspection: a Slack DM can be admitted for chat while command authorization remains false for text control commands, and the docs expect those commands to work in normal chat sessions. I did not run a live Slack workspace reproduction.

## Solution Assessment

Unclear until maintainers choose the exact Slack DM command authorization boundary. The best fix should be narrower than a blanket owner bypass: session recovery commands should execute or visibly reject without exposing owner-only commands to unintended senders.
