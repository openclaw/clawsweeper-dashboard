## Summary

Keep open: current main still has the Slack DM command path split between open-DM admission and owner-gated command authorization, and unauthorized recovery commands can still end without a visible reply. The merged CLI compaction PR only fixes the companion CLI/RPC gap, not Slack DM ingress.

## What This Changes

Slack DM text slash commands such as `/compact`, `/new`, and `/stop` should be intercepted and executed instead of being handled as ordinary agent text.

## Best Possible Solution

Restore Slack DM text recovery commands for the intended authorized sender class with visible execution or denial, preserve owner-only command boundaries, and add focused Slack DM tests plus live proof.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. Source inspection shows a medium-confidence current-main path where Slack DM text commands can be detected but remain separated from command authorization and can return no visible reply when blocked.

## Solution Assessment

Unclear until maintainers choose the Slack DM command authorization boundary. The best fix should execute or visibly reject recovery commands without exposing owner-only controls to unintended open-DM senders.
