## Summary

Keep open: current main and v2026.6.5 still show a Slack DM path where accepted DM messages can carry text control commands without being command-authorized, then continue into normal agent dispatch instead of executing or visibly rejecting the command. The remaining work needs a maintainer decision on the Slack DM command authorization boundary.

## What This Changes

Slack DM text slash commands `/compact`, `/new`, and `/stop` should execute or visibly reject instead of falling through as plain agent text.

## Best Possible Solution

Align Slack DM text recovery commands with the accepted DM session path, return a visible acknowledgement or denial, add focused Slack DM coverage for `/compact`, `/new`, and `/stop`, and leave the CLI/RPC discoverability work to https://github.com/openclaw/openclaw/issues/90640.

## Reproduction Assessment

Yes, by source inspection: Slack DM chat can be accepted while text control-command authorization remains false, and the DM branch falls through to normal agent dispatch instead of command execution or visible denial. I did not run live Slack reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the exact Slack DM command authorization boundary. The best fix should restore session-scoped recovery commands or visible denial without exposing owner-only controls to unintended DM senders.
