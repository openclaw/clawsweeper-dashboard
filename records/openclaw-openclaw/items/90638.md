## Summary

Keep open: current main and the latest release still make Slack DM recovery commands depend on command authorization, and this issue is freshly assigned for maintainer/product review. The CLI/RPC discoverability part is already split into a separate open issue, but the Slack DM recovery-command policy and visible acknowledgement behavior still need a maintainer decision.

## What This Changes

The issue asks OpenClaw to make Slack DM text commands such as `/compact`, `/new`, and `/stop` execute or visibly fail instead of being inert recovery commands.

## Best Possible Solution

Define and implement the Slack DM session-recovery policy narrowly: accepted DM users should either execute scoped `/new`, `/compact`, and `/stop` with visible acknowledgements or receive a visible denial, while owner-only commands remain gated and the CLI/RPC work stays tracked separately.

## Reproduction Assessment

Yes, by source inspection for the policy mismatch: Slack DM text command handling is separate from command authorization, and unauthorized text-command handlers for `/compact`, `/new`, and `/stop` stop without a visible Slack denial. I did not run a live Slack workspace reproduction.

## Solution Assessment

Unclear as an implementation choice until maintainers choose the authorization boundary. The best fix is not a blanket owner bypass; it should narrowly cover session recovery commands or produce explicit denials while keeping owner-only command surfaces protected.
