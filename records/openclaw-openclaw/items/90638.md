## Summary

Keep open: current main and the latest release still show a Slack DM command-auth boundary where admitted DM messages can fail control-command authorization or be silently suppressed, so the Slack recovery-command problem is not safely implemented on main.

## What This Changes

The issue requests Slack DM text commands such as `/compact`, `/new`, and `/stop` to execute or provide a visible recovery/error path instead of being inert plain chat text.

## Best Possible Solution

Define the Slack DM command boundary so session-scoped recovery commands execute or visibly reject for admitted DM users, keep owner-only commands protected, add focused Slack DM `/compact`, `/new`, and `/stop` coverage, and leave the CLI/RPC discoverability work to https://github.com/openclaw/openclaw/issues/90640.

## Reproduction Assessment

Yes, by source inspection: current Slack DM preparation can admit a message while carrying `CommandAuthorized=false` into the command handlers, and the documented recovery commands still depend on that authorization result. I did not run a live Slack workspace reproduction.

## Solution Assessment

Unclear until maintainers choose the exact Slack DM command authorization boundary. The best fix should be narrower than a blanket owner bypass: session recovery commands should execute or visibly reject without exposing owner-only commands to unintended Slack DM senders.
