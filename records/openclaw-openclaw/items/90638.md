## Summary

Keep open: current main still has the Slack DM command-authorization split and no-reply path that can make accepted DM recovery commands fail silently, while the merged CLI compaction work only fixed the programmatic recovery surface.

## What This Changes

The issue asks for Slack DM text commands such as `/compact`, `/new`, and `/stop` to execute or acknowledge recovery actions instead of falling through as ordinary agent text.

## Best Possible Solution

Restore Slack DM text recovery commands for accepted senders with visible execution or denial, preserve owner-only command boundaries, and add focused Slack DM coverage plus live proof.

## Reproduction Assessment

No, not with live proof in this review. Source inspection gives a medium-confidence reproduction path: Slack DM text commands can be detected but remain command-unauthorized against the owner allowlist, and unauthorized whole-message controls can return no reply.

## Solution Assessment

Unclear until maintainers choose the Slack DM command authorization boundary. The best fix should execute or visibly reject recovery commands without exposing owner-only controls to unintended senders.
