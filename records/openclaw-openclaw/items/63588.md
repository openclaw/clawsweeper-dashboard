The PR changes Slack media download fetching and tests so bearer Authorization is kept on Slack-owned redirect targets and removed after non-Slack redirects.

## Best Possible Solution

Land a rebased or replacement Slack plugin fix that reattaches Authorization only for Slack-owned HTTPS redirects, preserves the current SSRF hostname allowlist and dispatcher path, updates focused tests, and includes redacted real Slack proof.

Yes, at source level. Current main and `v2026.5.12` only preserve Authorization on same-origin redirects, and current Slack media tests still expect Slack CDN redirects to be unauthenticated; no live Slack repro was run in this read-only sweep.

## Solution Assessment

No, not as-is. The fix direction is narrow and maintainable, but the PR needs a resolvable test import, restored Slack-only redirect hostname policy, a rebase over current media changes, and real Slack proof before merge.
