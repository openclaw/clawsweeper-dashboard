## Summary

Keep open. Current main still lacks the requested selected/default versus active-runtime split for `session_status`: the shared renderer can resolve selected and active model identities, but it renders only the primary `Model:` line and the persisted `Fallback:` line, with no ordinary `Active:` line for routed non-fallback runtime drift. The related #75042 is already closed as a duplicate into this issue, and #51706 remains useful source context but is not a terminal fix against the current status module split.

## What This Changes

The issue requests that `session_status` keep the selected/session-default model visible while adding a separate active runtime model line when routing changes the model used for the current run.

## Best Possible Solution

Implement the split in the shared status path: keep `🧠 Model:` tied to the selected source, render `↪️ Active: provider/model` when active runtime identity differs and fallback state is inactive, continue rendering `↪️ Fallback:` for real fallback notices, and add focused coverage for runtime-routed sessions plus model-snapshot recovery.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by calling the shared status renderer or `session_status` with a configured default model and a session entry whose runtime `modelProvider`/`model` differ without matching fallback notice state; current tests lock that this produces no `Fallback:` and no `Active:` replacement.

## Solution Assessment

Yes. The narrow maintainable fix is a display-layer split in `src/status` and the `session_status` bridge, not mutating session defaults, model routing, or fallback persistence semantics.
