## Summary

Current `main` still does not implement the requested selected-versus-active display split. The renderer can resolve selected and active identities, but it only emits `Model:` plus fallback-specific output, and the linked implementation PR is still open/conflicting with prior blockers.

## What This Changes

The issue asks `session_status` to keep the selected/default model visible while also showing a distinct active model line when runtime routing uses a different model.

## Best Possible Solution

Land a focused status-rendering fix that preserves `Model:` as selected/default identity, emits `Active:` for non-fallback runtime drift, keeps `Fallback:` for true fallback notices, and covers both the shared renderer and `session_status` bridge.

## Reproduction Assessment

Yes, source-level. Call `buildStatusMessage` with a selected/default model and a session entry whose runtime `modelProvider`/`model` differ without matching `fallbackNotice*`; current `main` resolves those identities but renders no non-fallback `Active:` line.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared `src/status` display layer plus the `session_status` bridge, without changing routing, fallback persistence, or session defaults.
