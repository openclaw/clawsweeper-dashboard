## Summary

Keep open: current main still lacks the requested selected-versus-active model split. The shared status renderer only emits the selected `Model:` line plus fallback-specific output, while the `session_status` bridge can collapse a live active identity into the primary model instead of rendering both identities; the related implementation PR remains open and unmerged.

## What This Changes

The issue asks `session_status` to show the selected/session-default model and the active routed model separately when runtime routing changes the active model without mutating the session default.

## Best Possible Solution

Land a focused current-main fix in `src/status` and the `session_status` bridge that keeps `Model:` for selected/default, emits `Active:` for non-fallback runtime drift, preserves `Fallback:` for real fallback notices, and covers both renderer and bridge tests.

## Reproduction Assessment

Yes, source-level. Call `buildStatusMessage` with a selected/default model and a session entry whose `modelProvider`/`model` differ without matching `fallbackNotice*`; current main returns `Model:` and no `Active:` line, while the current-session bridge can collapse live active identity into the primary card model.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix. It belongs in the shared status renderer plus `session_status` bridge inputs, not in routing changes, session default mutation, or fallback persistence.
