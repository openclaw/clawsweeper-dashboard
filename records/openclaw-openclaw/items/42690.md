## Summary

Keep open: current main still does not render a separate selected/default-versus-active model split for ordinary routed runtime drift, and the related implementation candidate is open rather than merged.

## What This Changes

The issue requests that `session_status` keep showing the selected/default model while adding an explicit active model line when runtime routing uses a different model.

## Best Possible Solution

Land a focused shared status-renderer and `session_status` bridge fix that keeps `Model:` for selected/default identity, emits `Active:` for non-fallback runtime drift, keeps `Fallback:` for true fallback notices, and adds renderer plus bridge regression coverage.

## Reproduction Assessment

Yes, source-level. A `buildStatusMessage` input with a selected/default model and a different runtime active model, without matching fallback notice fields, still renders `Model:` and no general `Active:` line on current main.

## Solution Assessment

Yes. The requested selected-versus-active split is the narrow maintainable fix, and it should live in the shared status renderer plus `session_status` bridge inputs rather than mutating the stored session default or treating ordinary routing as fallback state.
