## Summary

Keep open: current main still lacks the requested selected/default-versus-active model split for ordinary runtime routing, and the related implementation candidate is open rather than merged.

## What This Changes

The issue requests that `session_status` keep showing the selected/default model while adding an explicit active model line when runtime routing uses a different model.

## Best Possible Solution

Land a focused fix in `src/status` and the `session_status` bridge that preserves `Model:` for selected/default identity, emits `Active:` for non-fallback runtime drift, keeps `Fallback:` for true fallback notices, and adds renderer plus bridge regression coverage.

## Reproduction Assessment

Yes, source-level. A `buildStatusMessage` input with a selected/default model and a different runtime active model, without matching fallback notice fields, still renders `Model:` and no general `Active:` line on current main.

## Solution Assessment

Yes, the requested selected-versus-active split is the narrow maintainable direction. It should be implemented in the shared status renderer plus `session_status` bridge inputs, not by mutating the session default or treating ordinary routing as fallback state.
