## Summary

Keep open: current main still has a source-reproducible gap where ordinary runtime model drift is not rendered as a selected/default-versus-active model split unless it matches persisted fallback notice state.

## What This Changes

The issue asks for `session_status` to keep showing the selected/default model while adding an explicit active-model line when runtime routing uses a different model.

## Best Possible Solution

Land one focused shared-status fix that keeps fallback-specific wording for real fallback notices, adds a general `Active:` line for non-fallback runtime drift, and covers both the renderer and `session_status` bridge with regression tests.

## Reproduction Assessment

Yes, source-level. A status-renderer input with a selected model and a different runtime active model, without matching fallback notice fields, still renders the selected `Model:` line and no general `Active:` line on current main.

## Solution Assessment

Yes. The requested split is the narrow maintainable fix; mutating the stored session default or replacing the selected line with the active model would not preserve the two distinct facts users need for routing diagnostics.
