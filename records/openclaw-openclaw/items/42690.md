- `impact:auth-provider`: The affected output is provider/model routing identity for an active run.

## Summary

Keep open: current main still does not preserve a selected/default-versus-active model split for ordinary routed runtime drift; the shared renderer only emits a fallback row for matching fallback notice state, and the current-session bridge can collapse the card to the active model instead of showing both identities.

## What This Changes

The issue asks for `session_status` to show the session selected/default model and the routed active runtime model as distinct status-card lines.

## Best Possible Solution

Land one focused shared status-renderer and `session_status` bridge fix that keeps selected/default identity visible, emits `Active:` for non-fallback runtime drift, preserves `Fallback:` for true fallback notices, and adds renderer plus bridge regression coverage.

## Reproduction Assessment

Yes, source-level. A `buildStatusMessage` input with a selected/default model and a different runtime active model, without matching fallback notice fields, still renders the selected `Model:` line and no general `Active:` line on current main.

## Solution Assessment

Yes. The requested selected-versus-active split is the narrow maintainable fix, and it should live in the shared status renderer plus `session_status` bridge inputs rather than mutating the stored session default or treating ordinary routing as fallback state.
