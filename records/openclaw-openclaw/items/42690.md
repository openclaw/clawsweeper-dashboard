## Label Justifications

- `P2`: The bug makes status output misleading for routed runs but does not describe data loss, a crash loop, or an unusable runtime.
- `impact:session-state`: The issue is about the session status card drifting from the actual active run state.
- `impact:auth-provider`: The affected output is the provider/model identity used for active routing and model selection.

## Summary

Keep open: current main and the latest release still do not provide a general selected/default-versus-active model split for `session_status`; active-model output is either collapsed into the card identity or shown only through fallback-notice state.

## What This Changes

The issue asks `session_status` to show the selected/session-default model and the routed active model separately when runtime routing changes the active model without mutating the session default.

## Best Possible Solution

Land one focused shared-renderer and `session_status` bridge fix that preserves the selected/default model, adds a general active-model row when the current run differs, keeps fallback-specific wording for true fallback notices, and adds regression coverage for both paths.

## Reproduction Assessment

Yes, source-level. A status render with selected/default model identity plus a different active runtime model and no matching fallback notice state still has no general `Active:` row on current main.

## Solution Assessment

Yes. The narrow maintainable fix is to keep selected/default identity intact and add an explicit active-model display in the shared renderer/bridge, rather than mutating the stored session default or treating all routing drift as fallback state.
