## Summary

Keep open: current main still lacks a selected/default-versus-active model split for ordinary routed runtime drift; it only shows an active model as a fallback row when persisted fallback notice state matches.

## What This Changes

The issue requests that `session_status` show both the selected/session-default model and the current run's routed active model when they differ.

## Best Possible Solution

Land one focused fix in the shared status renderer and `session_status` bridge that preserves the selected/default model, emits a general `Active:` line when the current run model differs, keeps fallback-specific wording for true fallback notices, and covers both renderer and bridge behavior with regression tests.

## Reproduction Assessment

Yes, source-level. A `buildStatusMessage` input with a selected/default model and a different runtime active model, without matching fallback notice fields, still renders the selected `Model:` line and no general `Active:` line on current main.

## Solution Assessment

Yes. The requested selected-versus-active split is the narrow maintainable fix, and it should live in the shared status renderer plus `session_status` bridge inputs rather than mutating the stored session default or treating ordinary routing as fallback state.
