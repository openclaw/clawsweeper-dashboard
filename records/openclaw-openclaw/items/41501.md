## Summary

Keep open: current main does not implement the Langfuse surface, and the PR still has blocking privacy, trace-attribution, and delivery-semantics findings. The delivery-confirmation slice looks useful, but it should be split and rebased away from the core telemetry PoC.

## What This Changes

This PR adds optional Langfuse tracing/env wiring for inbound requests, model generations, tool calls, and subagent spawns, updates agent delivery confirmation metadata, and adds related tests.

## Best Possible Solution

Split the delivery-confirmation bug fix into a narrow rebased PR, and handle Langfuse through the existing diagnostics/OTLP or plugin direction unless maintainers explicitly approve a core Langfuse client.

## Reproduction Assessment

Yes for the delivery-confirmation regression and the PR-introduced review findings by source inspection. Current main can mark a direct announce delivered from visible gateway payloads without delivery metadata, and the PR diff exposes concrete telemetry redaction/scope defects; no live Telegram run was performed in this read-only review.

## Solution Assessment

No. The delivery-confirmation direction is reasonable, but combining it with a new core Langfuse telemetry surface is not the narrowest maintainable solution; a split delivery fix plus a reviewed diagnostics/plugin design is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Redact tool error payloads before exporting them:** `src/observability/langfuse-agent-hooks.ts:220-223`
  - body: The error branch stringifies structured tool results directly into the Langfuse status message, bypassing the redaction path used for successful outputs. A failing tool that returns fields such as token or authorization would send those values to Langfuse when tracing is enabled.
  - confidence: 0.94
- **[P1] Exit request scope before draining followup work:** `src/auto-reply/reply/get-reply.ts:118-122`
  - body: Wrapping the whole reply turn in withLangfuseRequestScope lets async followup drains spawned during finalization inherit the previous inbound request scope. Queued work from later inbound messages can then attach generations and tool spans to the wrong trace under load.
  - confidence: 0.86
- **[P2] Sanitize payloads that exceed the redaction depth:** `src/observability/langfuse-agent-hooks.ts:29-30`
  - body: When depth exceeds MAX_OBJ_DEPTH, redactValue returns the original nested value unchanged. Secrets below that depth skip both key redaction and string truncation, which breaks the privacy guarantee for deeply nested tool args/results.
  - confidence: 0.9
- **[P2] Do not retain the raw Langfuse secret in exported config:** `src/observability/langfuse.ts:27-33`
  - body: LangfuseConfig stores secretKey, and the config is retained on the exported instrumentation object. If that object is logged, inspected, or serialized for diagnostics, the Langfuse secret can leak; keep only a configured boolean outside the local client-construction scope.
  - confidence: 0.82
- **[P2] Capture the run id before exception cleanup:** `src/auto-reply/reply/agent-runner.ts:405-406`
  - body: activeRunId is assigned only after runAgentTurnWithFallback returns successfully. If the run throws after opening tool spans, the finally block cannot call clearRunToolSpans, leaving Langfuse span records in the module-level map.
  - confidence: 0.83
- **[P2] Accept silent duplicate announces as delivered:** `src/agents/subagent-announce.ts:840-844`
  - body: The new delivery check treats any externally delivered announce response without result.delivery as a failed direct delivery. Non-completion announce prompts still allow NO_REPLY for a duplicate that was already delivered, so a legitimate silent duplicate now drives retry/deferred-cleanup behavior.
  - confidence: 0.8
