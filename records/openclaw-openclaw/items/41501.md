## Summary

Keep this PR open: current main does not implement the Langfuse PoC, and the delivery-confirmation bug is still source-plausible, but the branch is dirty against main and has blocking telemetry privacy and trace-attribution defects.

## What This Changes

The PR adds core Langfuse tracing/env wiring for inbound requests, model generations, tool calls, and subagent spawns, plus a cron/subagent direct-announce delivery-confirmation change and tests.

## Best Possible Solution

Split and rebase the work: land a narrow delivery-confirmation fix with regression coverage separately, and handle Langfuse through the existing diagnostics/OTLP direction or a reviewed observability design with explicit privacy controls.

## Reproduction Assessment

Yes for the delivery-confirmation bug by source inspection: current main can return payloads/meta from the gateway agent call while direct announce treats visible payloads as delivered without external delivery evidence. I did not run a live Telegram reproduction in this read-only review, but the PR body reports the same live behavior.

## Solution Assessment

No. The delivery-confirmation direction is useful, but this PR combines it with a new core Langfuse telemetry path and introduces blocking privacy and trace-scope bugs; a split delivery fix plus a separate observability decision is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Redact tool error payloads before exporting them:** `src/observability/langfuse-agent-hooks.ts:220-223`
  - body: The error branch serializes structured tool results directly into the Langfuse status message, bypassing the redaction path used for successful outputs. A failing tool that returns fields such as `token` or `authorization` would send those values to Langfuse when tracing is enabled.
  - confidence: 0.94
- **[P1] Exit request scope before draining followup work:** `src/auto-reply/reply/get-reply.ts:118-122`
  - body: Wrapping the whole reply turn in `withLangfuseRequestScope` lets async followup drains spawned during finalization inherit the previous inbound request scope. Queued work from later inbound messages can then attach generations and tool spans to the wrong trace under load.
  - confidence: 0.86
- **[P2] Sanitize payloads that exceed the redaction depth:** `src/observability/langfuse-agent-hooks.ts:29-30`
  - body: When `depth > MAX_OBJ_DEPTH`, `redactValue` returns the original nested value unchanged. Secrets below that depth skip both key redaction and string truncation, which breaks the privacy guarantee for deeply nested tool args/results.
  - confidence: 0.9
- **[P2] Do not retain the raw Langfuse secret in exported config:** `src/observability/langfuse.ts:27-33`
  - body: `LangfuseConfig` stores `secretKey`, and the config is retained on the exported instrumentation object. If that object is logged, inspected, or serialized for diagnostics, the Langfuse secret can leak; keep only a configured boolean outside the local client-construction scope.
  - confidence: 0.82
- **[P2] Capture the run id before exception cleanup:** `src/auto-reply/reply/agent-runner.ts:409`
  - body: `activeRunId` is assigned only after `runAgentTurnWithFallback` returns successfully. If the run throws after opening tool spans, the `finally` block cannot call `clearRunToolSpans`, leaving Langfuse span records in the module-level map.
  - confidence: 0.83
- **[P2] Accept silent duplicate announces as delivered:** `src/agents/subagent-announce.ts:840-844`
  - body: The new delivery check treats any externally delivered announce response without `result.delivery` as a failed direct delivery. Non-completion announce prompts still allow `NO_REPLY` for a duplicate that was already delivered, so a legitimate silent duplicate now drives retry/deferred-cleanup behavior.
  - confidence: 0.8
