## Summary

The PR is not merge-ready: its new capture getter restores the original payload after a delivery hook removes all sendable content, so the transcript can retain an assistant reply that Slack never received. The original same-channel transcript-loss bug remains valid, and the hook-based repair is narrowly fixable.

## What This Changes

The PR mirrors successful same-channel Slack final replies into the owning session transcript and adds focused regression coverage.

## Best Possible Solution

Use an explicit capture state that distinguishes no matching final from a matching final with no sendable content, mirror only the exact post-hook payload after successful delivery, and prove the result with focused hook tests plus a redacted live Slack roundtrip.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence path: a matching final transformed to empty by `appendBeforeDeliver` leaves `deliveredMetadata` undefined, after which the getter restores the original metadata.

## Solution Assessment

No, not yet. Intercepting the dispatcher’s post-hook final is the right boundary, but nullable fallback semantics must be replaced with an explicit observed state so filtered content cannot be restored.
