## Summary

Keep open: the false-positive warning is real and the helper direction is useful, but the PR currently makes `openclaw plugins update <id>` stop respecting documented exact pins for official npm plugins and still lacks after-fix real behavior proof.

## What This Changes

The PR adds a helper for choosing exact resolved npm specs, applies it to official plugin update/doctor/onboarding install-record writes, and makes `openclaw plugins update` sync official plugin records through the official catalog.

## Best Possible Solution

Preserve explicit exact pins for `openclaw plugins update <id>` while pinning only OpenClaw-managed official moving records, then add redacted real update/status proof.

## Reproduction Assessment

Yes. Source inspection shows current main's trust audit warns from bare `record.spec` values and update paths can write those bare official specs; the supplied screenshot confirms the warning in a real status run, though I did not run a live repro.

## Solution Assessment

No. Persisting exact specs in managed official write paths is plausible, but enabling catalog sync in `openclaw plugins update` overrides documented exact-pin behavior; the best fix needs to preserve explicit pins while addressing auto-managed official records.
