## Summary

Keep open. Current main still defaults scheduled heartbeat runs to the main session, uses append-only transcripts, and only filters HEARTBEAT_OK no-op prompt/ack pairs before replay; meaningful sent/tool heartbeat turns are intentionally preserved and can still influence the next scheduled tick. The reporter’s alternating logs and local patch notes remain credible, but the right fix needs maintainer choice around heartbeat replay isolation/filtering rather than restoring unsafe transcript truncation.

## What This Changes

The issue asks for each configured heartbeat tick to evaluate independently instead of alternating between delivered work and silent HEARTBEAT_OK acknowledgments that make the effective exploration interval twice the configured cadence.

## Best Possible Solution

Keep this as a core heartbeat replay-contract bug. Scheduled heartbeats should not treat previous heartbeat-generated sent/tool work as fresh main-session context, while preserving append-only auditability, HEARTBEAT.md and workspace injection, delivery routing, dedupe state, system-event handling, and safe session writes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level path: seed or observe a main-session transcript containing a meaningful heartbeat/tool turn, then confirm current replay filtering preserves it for the next scheduled heartbeat prompt. The exact alternating cadence still needs either a live model reproduction or a focused mocked decision-layer regression.

## Solution Assessment

Is this the best way to solve the issue? No current-main solution is in place. The append-only/no-op-pair filter is safer than fs.truncate-based pruning, but it does not address meaningful sent/tool heartbeat context; a targeted replay contract or explicit default isolation is the safer direction.
