## Summary

Keep open. Current main still defaults scheduled heartbeats to the main session, keeps heartbeat transcripts append-only, and filters only no-op HEARTBEAT_OK prompt/ack pairs before replay; meaningful sent/tool heartbeat turns can still remain visible to the next scheduled tick, so the reported alternating behavior is not proven fixed.

## What This Changes

The issue asks for each configured heartbeat tick to evaluate independently instead of alternating between delivered work and silent HEARTBEAT_OK acknowledgments that make the effective exploration interval twice the configured cadence.

## Best Possible Solution

Keep this as a core heartbeat replay-contract bug: scheduled heartbeats should avoid treating previous heartbeat-generated sent/tool work as fresh user-session context while preserving append-only auditability, HEARTBEAT.md/workspace injection, delivery routing, dedupe state, and system-event handling.

## Reproduction Assessment

Yes for the source-level path: seed or observe a main-session transcript containing a meaningful heartbeat/tool turn, then confirm current replay filtering preserves it for the next scheduled heartbeat prompt. The exact alternating cadence still needs either a live model reproduction or a focused mocked decision-layer regression.

## Solution Assessment

No current-main solution is in place. The append-only/no-op-pair filter is safer than fs.truncate-based pruning, but it does not address meaningful sent/tool heartbeat context; a targeted replay contract or explicit default isolation is the safer direction.
