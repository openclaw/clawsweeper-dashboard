## Summary

Keep this PR open: the hardening is in the right owner boundary, but the branch still rereads `tool.name` after validation, so a getter that returns a valid name once and then throws can pass validation and still register an unstable or empty-name tool.

## What This Changes

The PR wraps plugin tool shape reads and descriptor-cache capture in guarded accessors and adds optional-tool regressions for throwing name and parameter getters.

## Best Possible Solution

Preserve the PR’s owner-boundary approach, but make validation return stable checked fields or a sanitized wrapper so registration, metadata, descriptor caching, and provider consumers never reread hostile shape getters from the original tool.

## Reproduction Assessment

Yes, from source inspection: a tool whose `name` getter returns a valid string during `describeMalformedPluginTool` and throws on the next read reaches `tools.push(tool)` after `readPluginToolName` returns an empty string. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not yet: guarding individual reads is a good direction, but the best fix must carry the validated name/schema/description forward or wrap the tool so later consumers do not reread hostile getters.
