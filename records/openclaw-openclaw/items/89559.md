## Summary

Keep open: this maintainer/draft PR is useful plugin-runtime hardening, but source inspection confirms the current branch still has a post-validation getter gap that can leave a hostile tool registered instead of quarantined.

## What This Changes

The branch guards plugin tool runtime and descriptor-cache shape reads against throwing getters and adds optional-tool regression coverage for malformed getter-backed tools.

## Best Possible Solution

Snapshot or sanitize validated plugin tool fields before registration and descriptor capture, with a non-idempotent getter regression, or consolidate this branch into the broader maintainer-chosen hardening PR.

## Reproduction Assessment

Yes, source-reproducible. A plugin tool whose name getter returns a valid value during validation and then throws or changes on the registration read reaches the PR-head lines that convert the name to an empty string and still push the original tool.

## Solution Assessment

No. Guarding hostile reads is the right boundary, but the best fix must carry validated fields forward or return a sanitized tool so later consumers never reread the plugin-owned getters.
