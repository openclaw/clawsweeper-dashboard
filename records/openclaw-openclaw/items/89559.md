## Summary

Keep open: this focused plugin hardening PR still rereads and returns the original tool object after validation, so a non-idempotent getter-backed tool can pass validation and later crash provider, descriptor, or catalog consumers.

## What This Changes

The PR guards plugin tool shape reads and descriptor-cache capture against throwing getters and adds optional-tool regressions for malformed plugin tool shapes.

## Best Possible Solution

Keep the plugin-runtime hardening, but make validation produce stable checked fields or a sanitized tool object before registration, metadata, descriptor caching, and return.

## Reproduction Assessment

Yes. Source inspection of the PR merge result shows a tool whose name getter returns a valid string during validation and throws on the later read can still reach tools.push(tool); I did not run tests because this review is read-only.

## Solution Assessment

No. Guarding individual reads is a good direction, but the best fix must carry stable validated fields forward or return a sanitized tool so later consumers do not reread hostile getters.
