## Summary

Keep open: this draft maintainer PR is still useful, but it is not merge-ready because the patch can validate a getter-backed tool once and then register the original object for later consumers to reread.

## What This Changes

The PR adds guarded plugin-tool field reads in the resolver and descriptor cache, plus optional-tool regressions for throwing tool-name and parameter getters.

## Best Possible Solution

Make validation return stable checked fields or a sanitized tool wrapper, then add a non-idempotent getter regression before merging.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a tool name getter can return a valid string during describeMalformedPluginTool and fail on the later read at registration. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: guarding individual reads is a good direction, but the best fix must carry stable validated fields forward or return a sanitized tool so later consumers do not reread hostile getters.
