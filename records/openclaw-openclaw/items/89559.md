## Summary

Keep open: this maintainer-labeled draft still has a blocking source-level defect where a non-idempotent tool-name getter can pass validation, be reread as empty, and leave the original hostile tool registered for later consumers.

## What This Changes

The PR wraps plugin tool field reads in guarded accessors and adds optional-tool tests for throwing name and parameter getters.

## Best Possible Solution

Make validation return stable checked fields or a sanitized wrapper, then register and cache only that stable shape so later consumers never reread hostile getters from the original tool.

## Reproduction Assessment

Yes, from source inspection: a name getter that returns a valid string during validation and throws on the next read can reach registration because the PR rereads the name at `src/plugins/tools.ts:1313` and still pushes the original tool. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: guarded reads are the right direction, but the best fix must carry the validated fields forward or sanitize the returned tool object so downstream consumers do not reread hostile getters.
