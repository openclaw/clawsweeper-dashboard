## Summary

Keep open: this is still the canonical fix path for the CJK compaction undercount, and the supplied terminal proof is convincing, but the current PR head still has a blocking preemptive-routing bug where CJK-heavy tool-result recovery is reduced with a fixed chars/4 conversion after prompt pressure became CJK-aware.

## What This Changes

The PR adds a CJK-aware token-estimator wrapper, routes compaction/preemptive estimates through it, adds focused estimator tests, and updates the changelog.

## Best Possible Solution

Keep the CJK-aware estimator, but make preemptive tool-result recovery use the same CJK-aware token units and cover that route with a CJK-heavy regression before merge.

## Reproduction Assessment

Yes. Source inspection of PR head shows overflowTokens is CJK-aware while toolResultReducibleTokens is still maxReducibleChars / 4; a CJK-heavy toolResult where weighted recovery exceeds the truncate-only threshold but raw /4 does not reproduces the route mismatch.

## Solution Assessment

No. The wrapper is the right narrow fix for the main compaction undercount, but preemptive truncation recovery must be weighted before routing and covered by a CJK route test.
