## Summary

Keep open: this PR is still the canonical fix path for the closed CJK compaction bug, but the current head still underestimates CJK-heavy tool-result recovery in preemptive routing while current main has no equivalent fix.

## What This Changes

The PR adds a CJK-aware token-estimation wrapper, routes agent compaction and preemptive estimation through it, and adds compaction tests plus a changelog entry.

## Best Possible Solution

Keep the CJK-aware estimator approach, but make preemptive tool-result recovery use the same CJK-aware units as overflow pressure and cover that route with a focused regression before merge.

## Reproduction Assessment

Yes. Source inspection shows PR head makes overflowTokens CJK-aware while still converting raw toolResultPotential.maxReducibleChars with fixed /4; a CJK-heavy tool result near the truncate-only threshold reproduces the route mismatch.

## Solution Assessment

No. The wrapper is the right narrow fix for the main undercount, but the PR should also return or compute weighted tool-result recovery tokens before deciding truncate-only versus compact-then-truncate.
