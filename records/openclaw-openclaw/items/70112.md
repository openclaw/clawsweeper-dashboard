## Summary

Keep open. The CJK-aware estimator direction is correct and the supplied terminal proof is sufficient, but the current PR head still underestimates recoverable CJK tool-result truncation by converting raw reducible characters with a fixed chars/4 heuristic, so preemptive routing can still choose compaction when truncation alone would fit.

## What This Changes

The PR adds a CJK-aware wrapper around the upstream token estimator, routes compaction and preemptive estimates through it, adds regression tests, and records a changelog fix for CJK context-window overflow.

## Best Possible Solution

Keep the estimator wrapper, but make the preemptive truncation route compare CJK-weighted recoverable tool-result text against CJK-aware overflow and cover that route with a focused regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: use a CJK-heavy toolResult where weighted recoverable text exceeds the overflow plus buffer but raw chars/4 recovery does not, then assert the route should be truncate_tool_results_only.

## Solution Assessment

No. The wrapper is the right narrow direction for the main undercount, but the current route repair is incomplete because the recovery side still converts raw tool-result characters with a fixed chars/4 heuristic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Count CJK tool-result recovery before routing:** `src/agents/pi-embedded-runner/run/preemptive-compaction.ts:96-98`
  - body: The PR now makes `overflowTokens` CJK-aware, but it still computes recoverable tool-result tokens as raw `maxReducibleChars / 4`. For a Japanese or Chinese tool result, truncation can recover roughly one token per removed character, so this underreports recovery and can route to `compact_then_truncate` even when `truncate_tool_results_only` would fit. Use weighted recoverable text, or return weighted reduction potential, before comparing routes.
  - confidence: 0.9
