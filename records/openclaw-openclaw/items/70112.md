## Summary

Keep open. The PR is useful and targets a real CJK token-underestimation path, but it is not merge-ready because the preemptive route would use the new script-aware overflow estimate while the truncate-only decision still compares against raw character recovery; it also lacks the required user-facing changelog entry.

## What This Changes

The PR adds a CJK-aware wrapper around the upstream token estimator, routes agent compaction estimates through it, and adds focused regression tests for ASCII, CJK text, and toolCall arguments.

## Best Possible Solution

Keep this PR as the active implementation path, but compare preemptive overflow and tool-result recovery in consistent token-aware or script-aware units, rebase onto current main without dropping transcript-sanitizing behavior, and land it with focused compaction/preemptive-compaction regression coverage plus an Unreleased changelog entry.

Yes. A deterministic helper-level reproduction is a CJK-heavy prompt/tool-result case where the PR makes overflowTokens script-aware, but shouldPreemptivelyCompactBeforePrompt still multiplies that token value by 4 before comparing against raw maxReducibleChars, routing to compaction when truncation alone can cover the overflow.

## Solution Assessment

No. The CJK-aware estimator direction is right, but the proposed patch is incomplete until the preemptive truncation route uses consistent units and the required changelog entry is added.
