## Summary

Keep open: this remains the active fix path and the supplied terminal proof is acceptable, but the latest head still undercounts CJK tool-result truncation recovery in preemptive routing, so it can choose compaction when truncate-only would fit.

## What This Changes

The PR adds a CJK-aware token estimator wrapper, routes compaction and preemptive estimates through it, adds regression tests, and updates the changelog.

## Best Possible Solution

Keep the CJK-aware estimator direction, but make the preemptive truncation route compare CJK-weighted recoverable tool-result text against CJK-aware overflow and cover that route with a focused regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: use a CJK-heavy toolResult where weighted token recovery exceeds the overflow plus buffer but raw chars/4 does not, then assert the route should be truncate_tool_results_only.

## Solution Assessment

No. The estimator wrapper is the right narrow fix for the main compaction undercount, but the preemptive route still needs script-aware tool-result recovery before this is merge-ready.
