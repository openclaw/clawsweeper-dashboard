## Summary

Keep open. Current main has fixed the old bootstrap ratio budget gap and added compaction/context plugin surfaces, but oversized bootstrap files are still selected by fixed head/tail slicing, and the issue still spans several architecture decisions.

## What This Changes

The issue requests relevance-aware workspace bootstrap selection, less bulk-oriented compaction, and plugin-owned observation extraction from existing model output.

## Best Possible Solution

Split this into narrower trackers: define a relevance- or structure-aware bootstrap selector around buildBootstrapContextFiles, keep compaction work aligned with context-engine and compaction-provider contracts, and leave observation extraction plugin-owned unless a concrete SDK gap is found.

## Reproduction Assessment

Yes for the central bootstrap behavior from source inspection: oversized bootstrap files still go through fixed prefix/suffix slicing, and tests assert that shape. The compaction and observation parts are architecture proposals rather than standalone defects.

## Solution Assessment

No, not as one bundled implementation item. The maintainable path is to split bootstrap relevance from compaction-provider/context-engine work and observation-hook/plugin questions.
