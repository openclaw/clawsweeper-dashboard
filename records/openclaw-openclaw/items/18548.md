## Summary

Keep open. Current main still trims oversized bootstrap files by fixed head/tail slicing, and the compaction plus observation-extraction portions are only partially covered by existing plugin/context-engine surfaces, so this needs maintainer scoping rather than cleanup closure.

## What This Changes

The issue requests relevance-aware bootstrap context selection, incremental compaction, and lower-overhead observation extraction.

## Best Possible Solution

Split this into narrower trackers: design relevance- or structure-aware bootstrap selection in the bootstrap injection path, keep compaction aligned with provider/context-engine contracts, and leave observation extraction plugin-owned unless a concrete SDK gap appears.

## Reproduction Assessment

Yes for the central source-observable behavior: current main still trims oversized bootstrap files by fixed head/tail slicing, and tests assert that shape. The compaction and observation portions are architecture proposals rather than one failing reproduction path.

## Solution Assessment

No, not as one bundled implementation item. The maintainable route is to split bootstrap relevance from compaction-provider/context-engine work and observation-hook or plugin policy before a focused fix PR.
