## Summary

Keep open. Current main has fixed the old ratio arithmetic and provides adjacent compaction/context-engine/observation plugin surfaces, but oversized bootstrap files are still selected by deterministic prefix/suffix slicing, so the central relevance-aware bootstrap request remains unresolved and the bundled architecture proposal needs maintainer scoping.

## What This Changes

Request proposes relevance-aware bootstrap file selection, more incremental compaction, and extracting observations from existing LLM output rather than separate analysis calls.

## Best Possible Solution

Split the work into narrower tracks: define a relevance- or structure-aware bootstrap selector around buildBootstrapContextFiles and embedded injection, route compaction through existing provider/context-engine contracts, and keep observation extraction plugin-owned unless a concrete SDK gap appears.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the central bootstrap behavior: current main still routes oversized bootstrap files through deterministic prefix/suffix slicing, and tests assert that shape. The compaction and observation portions are architecture proposals rather than one reproducible defect.

## Solution Assessment

Is this the best way to solve the issue? No, not as one bundled implementation item. The maintainable path is to split bootstrap relevance from compaction-provider/context-engine work and observation-hook/plugin questions.
