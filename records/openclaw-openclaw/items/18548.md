## Summary

Keep open. Current main still drops the middle of oversized bootstrap files via deterministic prefix/suffix slicing, and a recent maintainer review already kept this open because the issue also bundles separate compaction and observation-extraction architecture questions.

## What This Changes

The issue requests relevance-aware bootstrap file selection, more incremental compaction behavior, and plugin-friendly observation extraction from existing LLM output.

## Best Possible Solution

Split this into narrower tracks: define a relevance- or structure-aware bootstrap selector around buildBootstrapContextFiles, use existing compaction provider/context-engine contracts for compaction work, and keep observation extraction plugin-owned unless a concrete SDK gap is found.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the central bootstrap behavior: current main still routes oversized bootstrap files through deterministic prefix/suffix slicing, and tests assert that shape. The compaction and observation parts are architecture proposals rather than one reproducible defect.

## Solution Assessment

Is this the best way to solve the issue? No, not as one bundled implementation item. The maintainable path is to split bootstrap relevance from compaction-provider/context-engine work and observation-hook/plugin questions.
