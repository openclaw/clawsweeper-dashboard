## Summary

Keep open. Current main still omits the middle of oversized bootstrap files by fixed prefix/suffix slicing, while the compaction and observation-extraction parts are partly covered by plugin surfaces but still need maintainer scoping before implementation.

## What This Changes

The issue requests relevance-aware bootstrap selection, less brittle incremental compaction, and observation extraction from existing model responses.

## Best Possible Solution

Split this into narrower trackers: design a relevance- or structure-aware bootstrap selector, keep compaction changes aligned with provider/context-engine contracts, and leave observation extraction plugin-owned unless a concrete SDK gap appears.

## Reproduction Assessment

Yes for the central bootstrap behavior: current source still truncates oversized bootstrap files by prefix/suffix slicing, and tests assert that shape. The compaction and observation portions are architecture proposals rather than one failing reproduction path.

## Solution Assessment

No, not as one bundled implementation item. The maintainable route is to split bootstrap relevance from compaction-provider/context-engine work and observation-hook/plugin policy before a focused fix PR.
