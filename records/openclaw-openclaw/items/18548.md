## Summary

Keep open: current main and the latest release still drop the middle of generic oversized bootstrap files by fixed prefix/suffix selection, while the compaction and observation-extraction parts overlap with existing plugin/context-engine seams and need maintainer scoping rather than cleanup closure.

## What This Changes

The issue requests relevance-aware bootstrap context selection, less bulk compaction behavior, and plugin-driven observation extraction from existing model responses.

## Best Possible Solution

Split this into narrower work: a focused relevance- or structure-aware bootstrap selector, plus separate plugin/API discussions for compaction and observation extraction only if current extension seams are insufficient.

## Reproduction Assessment

Yes for the bootstrap concern: source inspection on current main and v2026.6.8 shows oversized generic bootstrap files are deterministically reduced to fixed head and tail slices. The compaction and observation parts are architecture proposals rather than one failing runtime reproduction.

## Solution Assessment

No as a combined core change: the issue is too broad to solve in one maintainable implementation. The better path is a scoped bootstrap design and plugin-owned compaction/observation work unless a specific missing SDK/API seam is demonstrated.
