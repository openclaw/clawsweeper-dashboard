## Summary

Keep open. Current main has improved the bootstrap ratio arithmetic and has plugin surfaces for compaction/observation-adjacent work, but oversized bootstrap files are still trimmed by fixed prefix/suffix slicing and this issue still needs maintainer scoping across multiple architecture surfaces.

## What This Changes

Request relevance-aware bootstrap file selection, more incremental compaction, and plugin-driven observation extraction to reduce context and analysis overhead.

## Best Possible Solution

Split this into narrower trackers: define a relevance- or structure-aware bootstrap selector, keep compaction work aligned with existing compaction-provider/context-engine contracts, and leave observation extraction plugin-owned unless a concrete SDK hook gap is identified.

## Reproduction Assessment

Yes for the central bootstrap behavior: current source still sends oversized bootstrap files through fixed prefix/suffix slicing, and tests assert that shape. The compaction and observation portions are architecture proposals rather than one standalone failing reproduction path.

## Solution Assessment

No, not as one bundled implementation item. The maintainable route is to split bootstrap relevance from compaction-provider/context-engine work and observation-hook/plugin policy before any fix PR.
