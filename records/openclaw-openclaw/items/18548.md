## Summary

Keep open. Current main has improved bootstrap budget math and added compaction/context plugin surfaces, but oversized bootstrap files are still trimmed by fixed prefix/suffix slicing, and the issue still bundles multiple architecture decisions that need maintainer scoping.

## What This Changes

The issue requests relevance-aware bootstrap file selection, more incremental or robust compaction, and plugin-owned observation extraction from existing LLM responses.

## Best Possible Solution

Split this into narrower work: define a relevance- or structure-aware selector for bootstrap context, keep compaction changes aligned with the existing compaction-provider and context-engine contracts, and leave observation extraction plugin-owned unless maintainers identify a specific missing SDK hook.

## Reproduction Assessment

Yes for the central bootstrap behavior: current source still sends oversized bootstrap files through fixed prefix/suffix slicing, and the tests assert that shape. The compaction and observation parts are architecture proposals rather than standalone defects with one reproduction path.

## Solution Assessment

No, not as one bundled implementation item. The maintainable solution is to split bootstrap relevance from compaction-provider/context-engine work and observation-hook/plugin policy.
