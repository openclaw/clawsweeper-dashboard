## Summary

Keep open. Current main still trims oversized bootstrap files with fixed prefix/suffix selection, while the compaction and observation-extraction parts are separate plugin/context architecture questions that need maintainer scoping.

## What This Changes

The issue requests three architecture improvements: relevance-aware bootstrap file selection, incremental/safer compaction, and observation extraction from existing model responses.

## Best Possible Solution

Split the work into narrower trackers: define a relevance- or structure-aware bootstrap selector around buildBootstrapContextFiles, keep compaction work aligned with context-engine and compaction-provider contracts, and leave observation extraction plugin-owned unless a concrete SDK gap is found.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the central bootstrap behavior from source inspection: oversized bootstrap files still go through fixed prefix/suffix slicing, and tests assert that shape. The compaction and observation parts are architecture proposals rather than standalone defects.

## Solution Assessment

Is this the best way to solve the issue? No, not as one bundled implementation item. The maintainable path is to split bootstrap relevance from compaction-provider/context-engine work and observation-hook/plugin questions.
