## Summary

Keep open: current main fixes the old bootstrap ratio gap but still trims oversized bootstrap files by fixed head/tail position, and the compaction plus observation-extraction parts are only partly covered by existing plugin/context-engine surfaces. The issue is valid but broad enough to need maintainer scoping before a repair PR.

## What This Changes

The issue requests content-aware bootstrap selection, more incremental compaction, and lower-overhead observation extraction from existing LLM responses.

## Best Possible Solution

Split this into narrower maintainable tracks: design relevance- or structure-aware bootstrap selection in the injected context path, keep compaction work aligned with provider/context-engine contracts, and leave observation extraction plugin-owned unless a concrete SDK gap is identified.

## Reproduction Assessment

Yes for the central source-observable bootstrap behavior: current main still trims oversized files by fixed head/tail slicing and tests assert that shape. The compaction and observation portions are architecture proposals rather than one failing runtime reproduction.

## Solution Assessment

No, not as one bundled implementation item. The maintainable route is to split bootstrap relevance from compaction-provider/context-engine work and observation hook or plugin policy before a focused fix.
