## Summary

Keep open. Current main still trims oversized bootstrap files by deterministic prefix/suffix slicing, while the compaction and observation-extraction parts are separate architecture/product questions that need maintainer scoping rather than cleanup closure.

## What This Changes

Request to make bootstrap file injection relevance-aware, improve incremental compaction, and support observation extraction from existing LLM output.

## Best Possible Solution

Split the work into narrower trackers: define a relevance- or structure-aware bootstrap selector around buildBootstrapContextFiles, keep compaction changes aligned with existing compaction provider/context-engine contracts, and leave observation extraction plugin-owned unless a concrete SDK gap is found.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the central bootstrap behavior from source inspection: oversized bootstrap files still go through fixed prefix/suffix slicing, and tests assert that shape. The compaction and observation parts are architecture proposals rather than one reproducible defect.

## Solution Assessment

Is this the best way to solve the issue? No, not as one bundled implementation item. The maintainable path is to split bootstrap relevance from compaction-provider/context-engine work and observation-hook/plugin questions.
