## Summary

Keep open. Current main still trims oversized bootstrap files by deterministic prefix/suffix slicing, so the central relevance-aware bootstrap request remains unresolved; the compaction and observation-extraction parts have adjacent plugin/context-engine surfaces but still need maintainer scoping rather than cleanup closure.

## What This Changes

Request proposes three architecture improvements: relevance-aware bootstrap file selection, more incremental/robust compaction, and extraction of observations from existing LLM output instead of separate analysis calls.

## Best Possible Solution

Split the request into narrower tracks: define and implement a relevance- or structure-aware bootstrap selection path around buildBootstrapContextFiles and embedded bootstrap injection; handle incremental compaction through the existing compaction-provider/context-engine contracts; and keep observation extraction plugin-owned unless maintainers identify a concrete missing SDK hook.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the central bootstrap behavior: current main still routes oversized bootstrap files through deterministic prefix/suffix slicing, and tests assert that shape. The compaction and observation portions are architecture proposals rather than one reproducible defect.

## Solution Assessment

Is this the best way to solve the issue? No, not as one bundled implementation item. The maintainable path is to split bootstrap relevance from compaction-provider/context-engine work and from observation-hook/plugin questions.
