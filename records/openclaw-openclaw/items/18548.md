## Summary

Keep open: current main still has source-reproducible fixed head/tail bootstrap truncation, while the compaction and observation-extraction parts are only partially covered by existing plugin/context surfaces and newer narrower items, so maintainer scoping is still needed.

## What This Changes

The issue requests relevance-aware bootstrap selection, less bulk compaction, and plugin-style observation extraction from existing LLM output.

## Best Possible Solution

Split this into narrower maintainer-owned tracks: review the focused bootstrap policy path in https://github.com/openclaw/openclaw/issues/82920 and https://github.com/openclaw/openclaw/pull/82921, reconcile broader selective/lazy injection with https://github.com/openclaw/openclaw/issues/61347, scope compaction through provider/context-engine contracts, and keep observation extraction plugin-owned unless a concrete SDK gap appears.

## Reproduction Assessment

Yes for the central bootstrap behavior: oversized bootstrap files still run through fixed head/tail slicing, and current tests assert prefix/suffix preservation. The compaction and observation-extraction portions are architecture proposals rather than one failing runtime reproduction.

## Solution Assessment

No, not as one bundled implementation item. The maintainable route is to split bootstrap relevance from compaction-provider/context-engine work and observation hook or plugin policy before a focused fix.
