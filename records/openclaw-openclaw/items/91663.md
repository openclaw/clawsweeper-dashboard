## Summary

Keep this PR open for maintainer review: current main and the latest release still leave only normal-unwind cleanup for backup temp archives, while this PR is a coherent next-run mitigation with sufficient terminal proof. The unresolved merge question is whether maintainers accept deleting stale-looking files in the backup destination based on a 30 second mtime heuristic.

## What This Changes

The PR adds a non-fatal pre-write cleanup pass for stale backup `.tmp` archives plus tests for matching, scoping, size logging, and recent-file preservation.

## Best Possible Solution

Land the stale pre-run cleanup if maintainers accept the mtime-based active-writer guard, or require a lock/owner-marker style guard before merge if backup destinations need stronger overlapping-writer protection.

## Reproduction Assessment

Yes. At source level, current main creates adjacent random temp archives and only removes the active temp path during normal cleanup, so interrupted-run leftovers can remain for later runs.

## Solution Assessment

Unclear as a complete fix, but acceptable as a narrow mitigation if maintainers accept next-run cleanup. A stronger owner/lock signal would be safer for overlapping backups, and graceful signal/abort cleanup remains separate lifecycle work.
