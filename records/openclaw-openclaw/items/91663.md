## Summary

Keep open: this PR is still the focused candidate for stale backup `.tmp` cleanup, but it is currently conflicting with `main` and the mtime-based active-writer guard remains a maintainer safety decision before merge.

## What This Changes

Adds `cleanupStaleTempArchives()` and regression tests to remove older output-adjacent backup temp archives before creating a new backup while preserving recent matching temp files.

## Best Possible Solution

Rebase the branch, then either land the next-run stale cleanup with explicit acceptance of the mtime guard or replace it with a stronger active-writer signal such as a lock or owner marker.

## Reproduction Assessment

Yes, at source level. Current main writes the active tarball to an output-adjacent `.tmp` path and removes it only during normal JavaScript unwinding, so timeout or hard process termination can strand the file.

## Solution Assessment

Unclear as the best complete fix. The PR is a plausible narrow next-run mitigation, but maintainer review still needs to decide whether the mtime guard is safe enough or whether a lock/owner-marker or lifecycle cleanup is required.
