## Best Possible Solution

Rebase the branch, then replace the mtime-only active-writer heuristic with a lock, owner marker, or explicit concurrency contract, and decide whether this PR should also cover stale `openclaw-backup-` staging directories or leave that to the canonical follow-up.

## Solution Assessment

No as-is. The next-run cleanup is a plausible mitigation, but mtime alone is not the best active-writer guard for destructive cleanup before a backup run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use a real active-writer guard before deleting temp archives:** `src/infra/backup-create.ts:301-306`
  - body: The cleanup treats any matching temp file with `mtime` older than 30 seconds as stale and then removes it. A concurrent `openclaw backup create` can still be writing or stalled on an older temp archive for the same output path, and that process later publishes by the same path, so deleting it here can make the active backup fail. Please use an owner signal such as a lock/marker, or otherwise preserve active writers instead of relying only on file age.
  - confidence: 0.88
