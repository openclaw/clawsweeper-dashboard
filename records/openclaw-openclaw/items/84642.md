Keep open: the report is fresh, concrete, and current source still routes the affected `openclaw nodes` commands through Gateway RPC, so there is not enough evidence for an implemented, duplicate, stale, or out-of-repo close.

## What This Changes

The issue reports that `openclaw nodes list`, `pending`, `status`, and even nodes help can hang while `~/.openclaw/nodes/pending.json` contains a pending node request.

## Best Possible Solution

Keep the Gateway-owned pairing flow, but make the nodes CLI reliably return the pending list or a bounded, actionable transport/auth error without hanging, with regression coverage for help and Gateway handshake failure paths.

## Reproduction Assessment

No high-confidence live reproduction yet. The issue gives concrete commands and logs, and source confirms the affected commands must traverse Gateway RPC to read the pending store, but I did not run the reported Ubuntu/systemd setup against current main.

## Solution Assessment

Unclear. A transport/help-registration fix that preserves Gateway-owned pairing looks like the safest direction; a local pending-file fallback may solve the symptom but needs auth-boundary review.
