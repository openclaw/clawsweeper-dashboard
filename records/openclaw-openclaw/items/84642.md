The issue asks for `openclaw nodes` CLI commands to return pending node requests or fail quickly instead of hanging when a node pending registry exists.

## Best Possible Solution

Keep the Gateway-owned pairing flow, but make the nodes CLI reliably return the pending list or a bounded, actionable transport/auth error without hanging, with regression coverage for help and Gateway handshake failure paths.

No high-confidence live reproduction yet. The reporter gives concrete commands, registry state, and logs, and source confirms those commands must traverse Gateway RPC to read the pending store, but the Ubuntu/systemd hang was not reproduced here.

## Solution Assessment

Unclear. A transport or help-registration fix that preserves Gateway-owned pairing looks safest; a local pending-file fallback may solve the symptom but needs auth-boundary review.
