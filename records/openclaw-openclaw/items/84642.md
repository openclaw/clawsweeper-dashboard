## Summary

Keep open: the report is fresh, independently confirmed on 2026.5.19, and current main still routes the affected `openclaw nodes` list/pending/status paths through Gateway RPC before the node pending store is returned. Current main and v2026.5.20 appear to have a new fast path for `openclaw nodes --help`, but that does not address the Gateway-backed commands timing out while a pending request exists.

## What This Changes

The issue reports `openclaw nodes list`, `openclaw nodes pending`, `openclaw nodes status`, and `openclaw nodes --help` hanging or timing out while `nodes/pending.json` contains a pending node request.

## Best Possible Solution

Keep the Gateway-owned pairing flow, but make the nodes CLI reliably return pending requests or a bounded actionable transport/auth error without hanging; avoid a direct pending-file fallback unless the auth boundary is explicitly reviewed.

## Reproduction Assessment

No high-confidence live reproduction yet. The issue and a second comment provide concrete commands and logs, and source confirms the affected commands must traverse Gateway RPC to read the pending store, but I did not run the reported Ubuntu/systemd setup against current main.

## Solution Assessment

Unclear. A transport or CLI startup fix that preserves Gateway-owned pairing looks safest; a local pending-file fallback may solve the symptom but needs explicit auth-boundary review.
