## Summary

This should stay open: the report is a concrete latest-release node pairing CLI hang, and current main still routes `nodes pending/list/status` through gateway RPC before it can show the durable node pending state.

## What This Changes

The issue reports that `openclaw nodes` commands time out instead of showing or failing fast when `~/.openclaw/nodes/pending.json` contains a pending node request.

## Best Possible Solution

Keep the issue open and fix the node pairing CLI so pending node requests can be inspected, and possibly approved, through a security-preserving local or gateway path even when the normal loopback RPC handshake is stuck.

## Reproduction Assessment

No live repro was run in this read-only sweep. The reporter’s commands and logs map to current source: `nodes pending/list/status` must complete gateway RPC before exposing the on-disk node pairing state.

## Solution Assessment

Unclear as a complete fix until the handshake and `--help` timeout are isolated. A maintainable fix should preserve existing node approval authorization while adding a bounded local diagnostic/fallback or correcting the gateway handshake path.
