## Summary

Keep open: this is a fresh, concrete latest-release nodes pairing CLI hang with two independent reports, and current main still routes the affected commands through Gateway RPC before the node pending store can be shown. The related node-stability issue and PR are useful context, but they are still open and do not prove this specific CLI pending-request path is fixed.

## What This Changes

The issue requests that `openclaw nodes` commands return pending node requests, or fail quickly with an actionable error, instead of timing out when `nodes/pending.json` contains a request.

## Best Possible Solution

Preserve Gateway-owned pairing and make the nodes CLI/Gateway path return pending nodes or a bounded actionable transport error, coordinating with the related node-stability PR only if live proof shows the same root cause.

## Reproduction Assessment

No high-confidence live current-main reproduction yet. The issue and follow-up comment provide concrete release/platform evidence, and source confirms the affected commands must traverse Gateway RPC to read the pending node store, but I did not run the Ubuntu/systemd scenario.

## Solution Assessment

Unclear until the live failure is isolated. A transport or command-registration fix that keeps Gateway-owned pairing intact looks safer than a local pending-file fallback that could bypass approval boundaries.
