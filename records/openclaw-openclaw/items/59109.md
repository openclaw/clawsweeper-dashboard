## Summary

Keep open: current main has useful session primitives, but it does not implement the requested public SDK/Gateway shape for `forkSession(id)`, `resume: sessionId`, or `continue: true`. This is a new API/product-design request, not a stale bug or already-shipped fix.

## What This Changes

Request an SDK/Gateway session API for forking an existing session, resuming by session ID, and continuing the most recent session without its ID.

## Best Possible Solution

Define an additive Gateway protocol contract first, then expose matching SDK helpers and docs for explicit fork/resume/continue behavior with focused protocol and SDK regression coverage.

## Reproduction Assessment

Not applicable. This is a feature request, and source inspection shows the requested SDK/Gateway APIs are absent rather than an existing documented behavior failing.

## Solution Assessment

Unclear until maintainer API design is settled. The narrowest maintainable path is likely an additive protocol method or options contract plus SDK wrappers, not a quick SDK-only alias over today’s partial primitives.
