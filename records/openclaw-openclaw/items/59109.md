## Summary

Keep open: current main and the latest release expose explicit session-id/session-key reuse, parent-linked session creation, internal fork helpers, and checkpoint branch primitives, but they still do not provide the requested public `forkSession(id)`, resume option, or continue-most-recent contract. Related open items cover snapshots, chat `/resume`, and broader session controls, but none cleanly owns the full API-shaped request.

## What This Changes

The issue requests public session fork, resume-by-id, and continue-most-recent capabilities comparable to open-agent-sdk.

## Best Possible Solution

Define one maintainer-approved public session lifecycle contract across Gateway, SDK, CLI/chat UX, and docs, composing the existing session resolution, parent linkage, checkpoint branch, and internal fork primitives with clear authorization and lineage rules.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection confirm partial existing primitives and the missing public fork/resume/continue-most-recent contract.

## Solution Assessment

Unclear until maintainers choose the public API and UX. The maintainable path is to compose existing Gateway, SDK, CLI/chat, checkpoint, and internal fork primitives rather than exposing one internal helper or landing an isolated chat-only command as the complete solution.
