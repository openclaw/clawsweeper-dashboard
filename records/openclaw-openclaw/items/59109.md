## Summary

Keep open: current main has explicit session-id reuse and internal or adjacent fork primitives, but it still lacks one public Gateway/App SDK/CLI contract for forkSession(id), resume options, and continue-most-recent semantics.

## What This Changes

Request public session lifecycle APIs and UX for forking a session, resuming by session id, and continuing the most recent session.

## Best Possible Solution

Define one maintainer-approved public session lifecycle contract across Gateway, SDK, CLI/chat UI, and docs, reusing existing session resolution and transcript-branch/checkpoint primitives with clear permissions and lineage semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs show explicit session-id reuse exists, while the requested fork and continue-most-recent public contract is still absent.

## Solution Assessment

Unclear until maintainers choose the public API and UX. The maintainable path is to compose existing session-id resolution, Gateway protocol, SDK, and transcript-branch helpers instead of exposing an internal helper directly or adding a one-off CLI flag.
