## Summary

Keep open: current main partially supports explicit session-id routing and internal/checkpoint fork primitives, but it still lacks a public SDK/Gateway/CLI contract for `forkSession(id)`, resume-by-ID, and continue-most-recent behavior.

## What This Changes

The issue requests public session fork, resume-by-ID, and continue-most-recent capabilities similar to open-agent-sdk.

## Best Possible Solution

Design one maintainer-approved public session lifecycle contract across Gateway, SDK, and CLI that reuses existing session-id resolution, transcript branching, and latest-session selection semantics.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs inspection confirm related primitives exist, but the full public fork/resume/continue surface requested here is not present on current main.

## Solution Assessment

Unclear until maintainers choose the public API and UX. The maintainable path is to compose existing session-id resolution, Gateway protocol, SDK, and transcript-branch helpers rather than expose an ad hoc internal helper.
