## Summary

Keep open. Current main has partial lower-level session primitives, including exact-session run input and parent-key based session creation/compaction branching, but it still does not expose the requested open-agent-style `forkSession(id)` or `continue: true` public SDK/API surface; the related closed reports cover restart continuity and CLI resume bugs, not this App SDK feature.

## What This Changes

The issue requests first-class session fork, resume-by-ID, and continue-most-recent capabilities in OpenClaw similar to open-agent-sdk.

## Best Possible Solution

Define a first-class App SDK/Gateway session contract that maps cleanly onto OpenClaw’s `sessionKey`/`sessionId` model: keep exact-session resume, add an explicit fork API that resolves IDs safely before creating a parent-linked session, and add a most-recent-session helper only after scoping, ordering, and authorization rules are settled and documented.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The source/docs/test inspection gives a high-confidence verification path: current main has `sessionId`, `sessions.resolve`, `parentSessionKey`, and checkpoint branching primitives, but no public `forkSession(id)` or `continue: true` SDK/API surface.

## Solution Assessment

No, the current workaround is not the best final solution for the request. A public API design pass is safer than asking SDK users to manually stitch together `sessions.resolve`, `sessions.create({ parentSessionKey })`, and ad hoc most-recent-session selection.

## Risks / Open Questions

- The exact semantics for `continue: true` remain unresolved, especially how to choose the most recent session across agents, keys, channels, deleted-agent rows, and token scopes.
- A thin SDK wrapper over `parentSessionKey` would not fully solve fork-by-session-id unless it first resolves and authorizes the source session and defines failure behavior for ambiguous or stale matches.
