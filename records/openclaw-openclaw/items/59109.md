## Summary

Keep open: current main has internal/TUI fork and resume primitives plus compaction-checkpoint branch/restore, but it still does not expose the requested public SDK/Gateway/CLI forkSession, resume-by-ID, and continue-latest lifecycle contract.

## What This Changes

Request public session fork, resume-by-ID, and continue-latest capabilities similar to open-agent-sdk and Codex.

## Best Possible Solution

Define one canonical session lifecycle contract across Gateway protocol, SDK, CLI/TUI, storage, permissions, lifecycle hooks, and docs, then implement fork/resume/continue without adding parallel transcript state.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source and docs inspection give high confidence that public fork/continue semantics are absent while partial explicit session-id and internal fork/resume primitives exist.

## Solution Assessment

Unclear: the requested direction is useful, but the exact open-agent-sdk-shaped API may not be the best OpenClaw contract until maintainers choose the canonical Gateway, SDK, CLI, and storage semantics.

## Risks / Open Questions

- A direct open-agent-sdk-shaped API could create a second source of truth unless it is reconciled with session keys, session ids, parentSessionKey, compaction checkpoints, and TUI session files.
- Continue-latest semantics need a product decision about scope: per cwd, per agent, per channel/thread, per workspace, or global latest session.
- Fork and restore behavior can affect transcript DAG integrity, model/provider selection, lifecycle hooks, token accounting, and prompt-cache behavior, so the implementation boundary should be chosen before automation starts.
