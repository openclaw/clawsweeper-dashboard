## Summary

Keep open: current main and v2026.6.8 still have only adjacent internal/TUI fork-resume helpers and checkpoint-scoped Gateway branch/restore, not the requested public SDK/Gateway/CLI forkSession, resume-by-ID, and continue-latest lifecycle contract. Related items cover narrower session snapshots or chat resume commands, but none fully owns this public API shape.

## What This Changes

The issue requests a public session lifecycle API for forking an existing session, resuming by session id, and continuing the most recent session.

## Best Possible Solution

Define one canonical session lifecycle contract across Gateway protocol, SDK, CLI/TUI, storage, permissions, lifecycle hooks, and docs, then implement fork/resume/continue on that contract.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source, docs, protocol, and release inspection give high confidence that public fork/continue semantics are absent while partial explicit targeting and internal fork/resume primitives exist.

## Solution Assessment

Unclear as a concrete implementation path: the direction is useful, but the exact open-agent-sdk-shaped API should wait for a maintainer-approved OpenClaw Gateway, SDK, CLI, and storage contract.
