## Summary

Keep open. Current main still has only partial session primitives: explicit `sessionId`/`sessionKey`, `sessions.resolve`, and compaction-checkpoint branching, but not the requested public SDK `forkSession`, `resume` option, or `continue: true` most-recent-session workflow; related open issues cover broader snapshots or CLI resume disclosure, not this exact SDK/API request.

## What This Changes

Request an OpenClaw SDK/session API for forking a session, resuming by session ID, and continuing the most recent session.

## Best Possible Solution

Design an additive OpenClaw-native SDK and Gateway session contract that reuses `sessionId`, `sessionKey`, `sessions.resolve`, and sorted session listing, then explicitly defines arbitrary fork and continue-most-recent semantics with docs and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction. This is a feature/API request, and source inspection gives high confidence that current main has partial session primitives but not the requested public fork/resume/continue workflow.

## Solution Assessment

Unclear pending maintainer API direction. The open-agent-sdk shape is directionally useful, but OpenClaw should first define native fork, resume-by-selector, and continue-most-recent semantics that align with existing Gateway session visibility and the broader snapshot tracker.
