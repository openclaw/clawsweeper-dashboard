## Summary

Keep open: current main and v2026.6.8 have related session primitives, but they still do not expose the requested public forkSession(id), resume-by-session-id query, or continue-most-recent API/UX. A narrower public-fork issue was already closed back to this issue as the canonical tracker.

## What This Changes

Request a public OpenClaw session lifecycle surface for forking a session, resuming by session ID, and continuing the most recent session.

## Best Possible Solution

Define one canonical public session lifecycle contract across Gateway, SDK, CLI, and relevant UI surfaces, reusing existing create/resolve/send/checkpoint/internal fork primitives with explicit lineage and authorization rules.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API request. Source and docs inspection confirm current main has partial primitives but not the requested public fork/resume/continue surface.

## Solution Assessment

Unclear as a concrete implementation: copying the open-agent-sdk-style API verbatim is not yet proven best. The safer path is to reconcile it with existing Gateway session keys, compaction checkpoints, SDK sessions, and permission rules first.
