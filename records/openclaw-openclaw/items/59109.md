## Summary

Keep open. Current main partially covers resume primitives through `sessionId`, `sessionKey`, and `sessions.resolve`, but the requested public `forkSession`, `resume` option, and `continue: true` style workflow are not implemented, and no duplicate/canonical issue or PR was found.

## What This Changes

The issue requests App SDK/session APIs to fork an existing session, resume a session by ID, and continue the most recent session without knowing its ID.

## Best Possible Solution

Design an additive App SDK and Gateway session contract that reuses existing `sessionId`, `sessions.resolve`, and sorted `sessions.list` primitives, then adds explicit arbitrary fork and continue-most-recent semantics with tests and docs.

## Reproduction Assessment

Not applicable. This is a feature/API request, and source inspection shows partial primitives rather than a broken existing contract.

## Solution Assessment

Unclear. The current request is directionally aligned with the SDK reference design, but the exact OpenClaw-native API shape for arbitrary fork and continue-most-recent needs maintainer product/API review before implementation.
