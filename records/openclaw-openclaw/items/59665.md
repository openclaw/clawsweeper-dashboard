## Summary

Keep open. Current main intentionally generates a curated macOS Gateway service PATH instead of mirroring the login shell, and the merged Homebrew PATH fix at https://github.com/openclaw/openclaw/pull/79331 explicitly left this broader PATH-contract question unresolved.

## What This Changes

Request Gateway launchd install to capture the current login-shell PATH instead of generating a curated service PATH.

## Best Possible Solution

Define the macOS Gateway service PATH contract, then either document and keep the curated PATH with supported explicit additions or implement a sanitized login-shell import with audit and reinstall coverage.

## Reproduction Assessment

Yes for the mismatch: current main deterministically generates a curated macOS service PATH and tests assert no user-home entries, so it will not equal a login shell PATH containing `~/.bin` or `~/.cargo/bin`. I did not run a live macOS LaunchAgent reinstall in this read-only Linux review.

## Solution Assessment

Unclear. Full login-shell PATH mirroring would satisfy the reporter's expectation, but it conflicts with the current minimal-PATH hardening; the safer maintainable direction needs an explicit maintainer policy choice first.
