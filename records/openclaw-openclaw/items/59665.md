## Summary

Keep open: current main and the latest release still intentionally use a canonical macOS Gateway service PATH rather than copying the login shell PATH. The merged npm-global fix in https://github.com/openclaw/openclaw/pull/84475 narrows the failure by adding the active `openclaw` bin directory, but it does not settle this issue's broader login-shell PATH policy or security boundary.

## What This Changes

The issue asks for macOS Gateway LaunchAgent PATH generation during install/reinstall to match the current login shell PATH instead of the curated service PATH.

## Best Possible Solution

Maintainers should define the macOS Gateway service PATH contract first, then align install, restart, doctor/audit diagnostics, docs, and tests around either the current canonical policy or a deliberately sanitized opt-in path expansion design.

## Reproduction Assessment

Yes for source-level behavior: current main deterministically builds macOS LaunchAgent PATH from explicit runtime dirs plus canonical system dirs, not from `/bin/zsh -l -c 'echo $PATH'`. I did not run a live LaunchAgent reinstall in this read-only review.

## Solution Assessment

Unclear. Mirroring the login-shell PATH would satisfy the report, but it conflicts with the current documented canonical service PATH and needs a maintainer product/security decision before implementation.
