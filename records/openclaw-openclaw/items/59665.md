## Summary

Keep open: current main and the latest release still intentionally generate a canonical macOS LaunchAgent PATH instead of copying the login shell, and the related Homebrew-path fix only resolved stable Homebrew prefixes. The remaining request needs a maintainer product/security decision about the Gateway service PATH boundary.

## What This Changes

The issue asks macOS gateway install to write the LaunchAgent PATH from the current login shell instead of a generated canonical service PATH.

## Best Possible Solution

Define the macOS Gateway service PATH contract first, then align install, restart, doctor/audit diagnostics, docs, and tests around either the current canonical PATH policy or a deliberately sanitized login-shell capture design.

## Reproduction Assessment

Yes for source-level behavior: current main and `v2026.5.19` deterministically build the macOS LaunchAgent PATH through `buildMinimalServicePath` with Darwin user dirs disabled, rather than through login-shell PATH capture. No live launchd reinstall was run in this read-only review.

## Solution Assessment

Unclear. Mirroring login-shell PATH would satisfy the request, but it conflicts with the current documented canonical service PATH and needs an explicit maintainer product/security decision before implementation.
