## Summary

Keep open: current main and v2026.5.22 still intentionally generate a managed macOS service PATH rather than copying the login shell PATH. The merged OpenClaw-bin fix addresses one concrete prefix-install failure, but it does not settle the broader product/security decision requested here.

## What This Changes

Request that macOS Gateway LaunchAgent install derive PATH from the current login shell instead of the generated canonical service PATH.

## Best Possible Solution

Define the macOS Gateway service PATH contract explicitly, then align install, restart, doctor/audit, docs, and tests around a secure default plus any deliberately sanitized opt-in behavior maintainers approve.

## Reproduction Assessment

Yes for source-level behavior: current main and v2026.5.22 deterministically build a managed macOS service PATH instead of copying the login shell PATH. I did not run a live launchd reinstall in this read-only review.

## Solution Assessment

Unclear: copying the full login-shell PATH would satisfy the request, but it is not the safest maintainable default without an explicit product/security decision. The narrower current-main fix for active OpenClaw CLI bin reachability is a safer precedent.
