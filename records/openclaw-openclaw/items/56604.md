## Summary

Keep open. Current main still lacks per-dialed-number Voice Call routing: the plugin records provider `to` numbers, but config, UI metadata, inbound greeting, response agent/model/system prompt, realtime consult, and TTS selection all use one global Voice Call config. The issue discussion also includes a maintainer comment explicitly keeping this open, and related #59245 is broader outbound task-call work rather than a replacement for inbound per-number persona routing.

## What This Changes

Add per-phone-number Voice Call routing so different dialed Twilio/Telnyx/Plivo numbers can override voice, persona, model, prompt, greeting, and related response settings within one OpenClaw instance.

## Best Possible Solution

Implement the feature inside `extensions/voice-call` as a typed, documented per-number route table resolved at call creation. The resolved route id or effective config should be stable for the lifetime of the call, with global fallback, and should drive inbound greeting, response agent/model/system prompt, realtime consult configuration where applicable, and TTS voice selection without adding bundled plugin knowledge to core.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature request. Source inspection gives a high-confidence missing-behavior check: current main records provider `to` values but has no accepted config field or runtime path that routes by that value.

## Solution Assessment

Yes, a plugin-owned per-call effective config is the best direction. It keeps the behavior in the bundled Voice Call plugin, avoids core routing policy, and prevents active calls from changing persona if global config is edited mid-call.
