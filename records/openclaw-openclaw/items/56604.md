## Summary

Keep open. Current main still lacks per-dialed-number routing in the bundled Voice Call plugin: the dialed `to` value is captured and stored, but config, UI metadata, inbound greeting, response generation, realtime consult setup, and TTS voice selection still use one global Voice Call config. The issue discussion also includes a maintainer comment confirming the gap, and related #59245 is broader outbound task-call work rather than a replacement for this inbound routing request.

## What This Changes

Add per-phone-number Voice Call routing so the dialed number can select voice, persona, system prompt, response model, agent, and inbound greeting overrides.

## Best Possible Solution

Implement this inside `extensions/voice-call` as a typed per-number route table, resolved from the normalized dialed number when the call record is created. The resolved route or effective config should remain stable for the call lifetime and drive inbound greeting, agent/model/system prompt, realtime consult where applicable, and TTS voice selection, with global fallback and docs/changelog coverage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature request. There is a high-confidence source-level missing-behavior check: current main records provider `to` values but has no accepted config field or runtime path that routes by that value.

## Solution Assessment

Yes, a plugin-owned per-call effective config is the best direction. It keeps routing policy in the bundled Voice Call plugin, avoids core-specific persona logic, and prevents active calls from changing persona if global config changes mid-call.

## Repair Work Prompt

Implement per-dialed-number routing for the bundled Voice Call plugin. Add a typed route table under `plugins.entries.voice-call.config` (the issue proposes `numbers`) keyed by normalized/E.164 dialed numbers, with partial overrides for the user-facing persona fields: `inboundGreeting`, `tts`, `agentId`, `responseModel`, `responseSystemPrompt`, and any realtime consult settings needed to keep persona behavior consistent. Resolve the matching route once when an inbound webhook creates the call record from `event.to`; store either the route id and/or stable effective config in call context/metadata so later greeting, response generation, realtime consult, and TTS selection use the same effective config for the call lifetime. Preserve global fallback, existing webhook signature verification, existing inbound allowlist semantics, and outbound behavior unless explicitly covered by tests. Keep the change inside `extensions/voice-call`; do not add bundled plugin-specific policy to core. Update UI/plugin metadata, docs, and `CHANGELOG.md` with appropriate user-facing credit. Include route-matching tests for Twilio, Telnyx, and Plivo `to` formatting or shared normalization.
