## Summary

Keep open. Current `main` still appears to reject SSH remote gateway configs before any tunnel is established. The runtime path that builds/boots remote gateway connections ignores `gateway.remote.transport`, validates `gateway.remote.url` directly as plaintext WebSocket, and the only non-test SSH tunnel startup path I found is the `gateway status` probe flow, not normal remote client startup.

## Evidence

- **Connection builder still validates remote URL directly:** `buildGatewayConnectionDetailsWithResolvers()` reads `config.gateway.remote.url` into `remoteUrl`, picks that as the connection URL, and immediately throws if `isSecureWebSocketUrl(url)` fails. There is no branch on `gateway.remote.transport` or `sshTarget` in this function.
  - file: [src/gateway/connection-details.ts:47](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/gateway/connection-details.ts#L47)
  - command: `nl -ba src/gateway/connection-details.ts | sed -n '1,120p'`
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Bootstrap passes the same validated URL through:** `resolveGatewayClientBootstrap()` just calls `buildGatewayConnectionDetailsWithResolvers()` and returns `connection.url`; it does not establish an SSH tunnel or propagate any SSH transport flag.
  - file: [src/gateway/client-bootstrap.ts:29](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/gateway/client-bootstrap.ts#L29)

  - file: [src/gateway/client.ts:234](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/gateway/client.ts#L234)
