## Summary

Keep open. Current main still lacks the proposed session-run cancel seam, but this stacked PR is not merge-ready because the declined-hook fallback regresses current sessions_send provenance and lane routing, and the added startup fixture is stale against the current getRuntimeConfig contract.

## What This Changes

The PR stacks a sessions_send plugin hook, a plugin SDK session-run cancel runtime subpath, gateway chat-abort cancel wiring, a minimal-gateway guard test, generated API baseline, and a changelog entry.

## Best Possible Solution

Sequence the sessions_send hook first, then land a narrowed cancel seam that preserves current sessions_send semantics, documents the public subpath, updates the API baseline and changelog, and passes the focused gateway/session/plugin-SDK checks.

## Reproduction Assessment

Yes. Static comparison is enough: current main annotates sendParams and uses resolveNestedAgentLaneForSession, while the PR's declined-hook fallback rebuilds raw sendParams and the new fixture passes loadConfig instead of the required getRuntimeConfig.

## Solution Assessment

No. The minimal-gateway guard is a plausible part of the larger seam, but the current branch should be rebased and repaired so declined-hook fallback behavior and SDK documentation stay aligned with current main.
