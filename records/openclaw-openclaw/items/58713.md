## Summary

Keep open. Current `main` still lacks the requested ephemeral A2A mode: `sessions_send` has no ephemeral parameter, `tools.agentToAgent` has no `ephemeral` config field, the first send targets the resolved existing session, and the follow-up loop runs on the requester and target sessions. This is valid core session-lifecycle work, but the API, cleanup, retention, and security semantics need maintainer design before an automated fix chooses a contract.

## What This Changes

The issue requests an opt-in ephemeral agent-to-agent mode for `sessions_send` so peer agents can exchange messages in a temporary clean session without writing coordination turns into either agent's main conversation.

## Best Possible Solution

Design an additive core ephemeral A2A contract, then implement it behind an explicit opt-in that preserves existing allowlist and visibility checks, routes the exchange through a transient isolated session, defines cleanup and transcript retention behavior, and updates docs plus focused session-send/config tests.

## Reproduction Assessment

Not applicable as a traditional bug reproduction because this is a feature request. The current gap is high-confidence from static inspection: the tool/config schemas have no ephemeral option, the send path targets `resolvedKey`, and the A2A loop runs on requester/target sessions.

## Solution Assessment

Unclear. The requested direction is plausible, but the best maintainable fix needs maintainer product/security design around API shape, retention, cleanup, and reply delivery before code is written.
