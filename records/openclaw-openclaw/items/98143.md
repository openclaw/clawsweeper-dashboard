## Summary

Keep open: this PR is a focused hardening patch with no line-level defect found, but it intentionally changes a compatibility-sensitive bodyless MCP OAuth error fallback and needs maintainer/security policy acceptance before merge.

## What This Changes

The PR changes MCP HTTP fetch response normalization to preserve bodyless foreign-response text only when a safe decimal Content-Length is present and within a 1 MiB cap, with regression tests for oversized, missing, malformed, and dishonest lengths.

## Best Possible Solution

Merge this hardening only after maintainers accept the fail-closed Content-Length policy, or keep the canonical issue open and choose a stricter or deferred bodyless-response policy there.

## Reproduction Assessment

Yes. Current main reaches await response.text() for a bodyless non-204/205/304 MCP response, and the existing compatibility test exercises that text-only foreign-response shape.

## Solution Assessment

Mostly yes. The latest head avoids the prior readResponseWithLimit arrayBuffer trap and rejects malformed lengths, but it is the best fix only if maintainers accept the compatibility/security policy change.
