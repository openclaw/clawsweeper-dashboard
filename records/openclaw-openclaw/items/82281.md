## Summary

Keep open: the current Lobster docs do not enumerate runtime-injected shell environment variables or warn against invented step-output env vars, while the OpenClaw wrapper and Lobster dependency source show step output is resolved through stdin/template references rather than synthesized env names.

## What This Changes

The issue asks to update the Lobster docs to list workflow-injected shell environment variables and clarify that step outputs are accessed through stdin/template refs, not `LOBSTER_STEP_*` env vars.

## Best Possible Solution

Add a docs-only reference section in `docs/tools/lobster.md` that mirrors the current Lobster contract: inherited env plus the actual injected args env vars, explicit workflow/step env overrides, supported `$step` stdin/template access, and no synthesized `LOBSTER_STEP_*` output vars.

## Reproduction Assessment

Not applicable as a docs issue; source inspection gives high confidence in the documentation gap. Current docs lack `LOBSTER_*` env documentation, and the runtime source resolves prior step output through stdin/template references rather than env-var injection.

## Solution Assessment

Yes. A narrow docs update is the best path; adding env-based step-output exposure would create a new runtime contract and is not required to address the reported authoring trap.

Add a concise section near the workflow-file notes or tool parameters that enumerates the current Lobster shell environment contract: inherited process env, workflow/step `env` blocks, and the actual `@clawdbot/lobster` injected args variables (`LOBSTER_ARGS_JSON` and `LOBSTER_ARG_<NAME>` per v2026.4.6 source). No changelog is expected for a docs-only clarification.
