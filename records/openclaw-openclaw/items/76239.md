## Summary

Keep open: current `main` and the reported beta tag already contain the expected `OPENCLAW_NO_RESPAWN` guards, but the production report says the same tagged build still exits PID 1, so the actual exit path is not identified well enough to close.

## What This Changes

The issue requests that Gateway restart paths stay in-process when `OPENCLAW_NO_RESPAWN=1` so PID 1 is not cleanly exited in non-supervised containers.

## Best Possible Solution

Keep the issue open until the actual signal, restart reason, and exit log are captured in Cloudflare; then patch the narrow exit path and add a regression test that proves PID 1 stays alive with `OPENCLAW_NO_RESPAWN=1`.

## Reproduction Assessment

Unclear: the report has a strong production reproduction, but this review does not have a local or Cloudflare reproduction and the suspected source path should not exit when `OPENCLAW_NO_RESPAWN` is truthy. The missing piece is a log trace showing which run-loop branch calls `exitProcess(0)`.

## Solution Assessment

Unclear: adding the requested env gate is not obviously the best fix because that gate already exists in the respawn helpers on current `main` and the reported beta tag. The safer next step is to identify the real exit path before changing lifecycle behavior again.
