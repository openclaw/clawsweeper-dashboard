## Summary

Keep open: this is a fresh, concrete auth-routing report, and current source still has paths where explicit `openai/...` image understanding resolves through the OpenAI API-key provider while separate Codex media providers exist. The exact user config is missing, and the docs split `openai/*`, `openai-codex/*`, and `codex/*` in ways that need a maintainer decision before changing fallback behavior.

## What This Changes

Request image/media-understanding analysis to use Codex OAuth or the Codex media-understanding provider instead of requiring `OPENAI_API_KEY` when only Codex auth is available.

## Best Possible Solution

Decide and codify the intended media route: either make `openai/gpt-*` image understanding fall back to Codex OAuth like image generation, or document and diagnose that Codex-only installs must use `codex/gpt-*` or `openai-codex/*` media settings.

## Reproduction Assessment

No high-confidence live reproduction yet; current source strongly supports the reported failure when explicit image/media understanding is configured as `openai/...` and only `openai-codex` OAuth is present. The exact config and live Codex auth state are still needed for a full reproduction.

## Solution Assessment

Unclear; routing through the Codex media provider may be the right fix, but current docs also describe `codex/gpt-*` as the explicit bounded Codex app-server path. A maintainer should choose the contract before automation changes auth fallback behavior.
