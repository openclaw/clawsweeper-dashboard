## Summary

Keep open: current source still supports the reported auth-routing gap for explicit `openai/...` image understanding with only Codex OAuth available, but the desired fallback crosses the documented `openai/*`, `openai-codex/*`, and `codex/*` route boundary and needs maintainer direction.

## What This Changes

The issue asks for image/media understanding to route through Codex OAuth-backed vision instead of requiring `OPENAI_API_KEY` when Codex OAuth is the only OpenAI auth available.

## Best Possible Solution

Define and codify the media route contract: either allow `openai/*` vision to fall back to Codex OAuth when no direct OpenAI key exists, or diagnose and document that Codex-only installs must use `codex/*` or `openai-codex/*` media settings.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review, but the source path supports the failure for an explicit `openai/...` image model with only Codex OAuth available. The exact user config and live Codex auth state are still needed for full end-to-end proof.

## Solution Assessment

Unclear. Routing through the Codex media provider may be the right fix, but current docs also distinguish the normal OpenAI route from the bounded `codex/*` app-server route, so maintainers should choose the contract before automation changes fallback behavior.
