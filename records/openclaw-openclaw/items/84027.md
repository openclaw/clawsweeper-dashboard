## Summary

Keep open: current main still has source-backed paths where explicit `openai/...` image understanding resolves through the OpenAI provider and direct API-key auth, while separate `openai-codex` and `codex` media routes exist. The report is concrete, but the right fix depends on a maintainer decision about whether `openai/*` media understanding should fall back to Codex OAuth or remain the normal OpenAI route.

## What This Changes

The issue requests that image understanding use Codex OAuth-backed media understanding when Codex OAuth is the only OpenAI auth available, instead of requiring `OPENAI_API_KEY`.

## Best Possible Solution

Define and test the intended media-understanding contract: either route eligible `openai/gpt-*` image analysis through Codex OAuth like image generation, or document and diagnose that Codex-only installs must use `codex/*` or `openai-codex/*` media settings.

## Reproduction Assessment

No high-confidence live reproduction yet; source inspection supports the failure path for explicit `openai/...` image understanding with only Codex OAuth, but the reporter's exact config and live auth state were not exercised in this read-only review.

## Solution Assessment

Unclear; routing through a Codex media provider may be the right fix, but current docs and provider ids split direct OpenAI, `openai-codex`, and bounded `codex/*` behavior, so maintainers should choose the contract before implementation.
