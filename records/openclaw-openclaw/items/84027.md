## Summary

Keep open: current source still supports the reported OpenAI API-key failure when image understanding is explicitly routed through `openai/...`, while separate `openai-codex` and `codex` media-understanding routes already exist. The remaining question is a maintainer product/compatibility decision: should `openai/gpt-*` vision silently fall back to Codex OAuth, or should docs/diagnostics steer Codex-only users to the Codex media route.

## What This Changes

The issue requests Codex-OAuth-backed image/media understanding without requiring a direct `OPENAI_API_KEY`.

## Best Possible Solution

Codify one supported route: either add tested Codex OAuth fallback for `openai/gpt-*` image understanding, or keep the current split and improve diagnostics/docs so Codex-only users select `codex/gpt-*` or `openai-codex/*` intentionally.

## Reproduction Assessment

No high-confidence live reproduction yet. Source inspection strongly supports the failure when image understanding is selected as `openai/...` with only `openai-codex` OAuth available, but the exact config and live auth state are still missing.

## Solution Assessment

Unclear. Routing `openai/gpt-*` vision through Codex OAuth may match user expectations from image generation, but current docs also define `codex/gpt-*` as the bounded Codex image-understanding route, so maintainers should choose the contract before automation changes fallback behavior.
