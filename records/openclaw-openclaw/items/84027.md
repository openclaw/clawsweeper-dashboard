## Summary

Keep open: current main has Codex and OpenAI-Codex media-understanding providers, but the explicit OpenAI image-understanding path still resolves `openai` auth directly and can surface the reported `OPENAI_API_KEY` failure instead of using the Codex OAuth image-understanding route.

## What This Changes

The issue requests that media/image understanding use Codex OAuth when that is the only OpenAI auth available, matching the previously fixed image-generation flow.

## Best Possible Solution

Implement or explicitly reject a provider-owned fallback for public OpenAI media understanding that uses Codex OAuth only when direct OpenAI auth is absent, while preserving documented `openai-codex/*` and `codex/*` routes.

## Reproduction Assessment

Yes for the likely source path, but not as a live current-main run: an explicit `openai` image-understanding entry reaches `describeImageWithModel` and requires `openai` auth, matching the reported error when only Codex OAuth is available. The exact reporter config is still needed to prove whether auto-detection also misroutes.

## Solution Assessment

Unclear at the policy boundary: the code should either mirror image generation's public-OpenAI Codex OAuth fallback or clearly require explicit `openai-codex/*`/`codex/*` media refs. A maintainable fix should live in the OpenAI/Codex provider boundary, not as a broad core special case.
