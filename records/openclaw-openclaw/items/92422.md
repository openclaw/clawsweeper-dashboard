## Summary

Keep open for explicit maintainer handling: the yield-aware wait guard addresses the reported AbortError, but the branch makes `deleteAfterRun` ineffective for yielded jobs and overlaps a separate callback-routing PR that has not been reconciled.

## What This Changes

The PR skips completion-required media waiting after `sessions_yield` and retains yielded isolated cron sessions instead of immediately applying delete-after-run cleanup.

## Best Possible Solution

Use one canonical patch that skips only the post-yield synchronous wait, resumes generated-media completion through the requester-agent path, and applies delete-after-run cleanup after the resumed workflow reaches its true terminal turn.

## Reproduction Assessment

Yes from source inspection: the intentional yield leaves an aborted signal that the completion-required wait converts into the reported `AbortError: aborted`, matching the linked issue evidence. I did not execute a live current-main gateway reproduction during this read-only review.

## Solution Assessment

No. Passing yield state into the wait decision is the narrow fix for the immediate abort, but permanently bypassing delete-after-run is incomplete and the requester-agent callback path should be reconciled with https://github.com/openclaw/openclaw/pull/92146.
