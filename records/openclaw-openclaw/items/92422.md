## Label Justifications

- `P1`: The underlying regression reliably stops isolated cron workflows before their remaining user actions execute.
- `merge-risk: 🚨 session-state`: The PR adds persisted deletion intent whose cleanup depends on an optional retention process.
- `merge-risk: 🚨 message-delivery`: The branch can classify the run as yielded without ensuring that the later callback resumes the requester session and delivers remaining output.

## Summary

The immediate `AbortError` guard is valid, but this PR is not merge-ready because deferred deletion stops working when retention is disabled, the reported requester-session continuation remains unchanged, and the branch lacks real lifecycle proof.

## What This Changes

The PR makes yielded isolated cron runs skip pending-media completion waits and defers delete-after-run cleanup through a persisted cron-session marker consumed by the retention reaper.

## Best Possible Solution

Use one canonical repair that skips the post-yield wait, routes inactive generated-media completion through the requester session, and completes delete-after-run cleanup from the resumed workflow’s terminal lifecycle rather than optional retention.

## Reproduction Assessment

Yes at source level: a yielded cron run can pass its already-aborted signal into the completion-required media wait and throw `AbortError: aborted`; the linked issue also supplies a concrete reproduction from release 2026.6.5.

## Solution Assessment

No. Skipping the aborted wait is necessary, but this is not yet the narrowest complete repair because callback continuation is unchanged and delete intent is coupled to optional retention.
