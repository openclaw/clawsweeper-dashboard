## Label Justifications

- `P1`: The underlying regression consistently aborts isolated cron media workflows and prevents their remaining actions from executing.
- `merge-risk: 🚨 session-state`: The patch bypasses delete-after-run cleanup for yielded sessions even though current retention logic cannot prune those isolated base sessions.
- `merge-risk: 🚨 message-delivery`: The claimed outcome depends on the later media callback resuming the requester agent and executing the deferred send, which this PR does not prove live.

## Summary

The PR correctly identifies and fixes the immediate post-yield abort, but it is not merge-ready because its companion cleanup guard permanently defeats `deleteAfterRun` for yielded jobs, and the supplied tests do not prove the real callback-resume workflow. The overlapping proof-positive PR should be reconciled rather than allowing two competing fixes for the same session lifecycle.

## What This Changes

The branch makes completion-required cron media waits yield-aware and retains yielded isolated cron sessions instead of applying delete-after-run cleanup.

## Best Possible Solution

Consolidate one canonical patch that skips only the post-yield synchronous wait, routes the generated-media completion through the requester-agent continuation, and performs delete-after-run cleanup after the resumed workflow reaches its true terminal state.

## Reproduction Assessment

Yes. Source inspection shows the intentional `sessions_yield` abort reaches the completion-required media wait and is rethrown as the reported `AbortError: aborted`; the linked report and related PR provide matching runtime evidence.

## Solution Assessment

No. Adding `yieldDetected` to the wait decision is the correct narrow fix for the abort, but permanently suppressing `deleteAfterRun` is not a complete lifecycle solution and the callback-resume path must be reconciled with the related PR.
