## Summary

PR #93218 is useful but not merge-ready: the patch adds the requested session-scoped `/stream` command, yet it currently reads Discord/Slack overrides from the slash-command session instead of the target session and still lacks current-head live channel proof for the visible behavior.

## What This Changes

The PR adds a `/stream`/`/streaming` command that stores a per-session preview streaming mode override and wires Telegram, Discord, and Slack to prefer that override over channel config.

## Best Possible Solution

Fix the target-session lookup, keep preview mode distinct from block-delivery config in status output, then merge only after maintainers accept the new session/channel contract and current-head live channel proof confirms the visible command flow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Read stream overrides from the target session:** `extensions/slack/src/monitor/message-handler/dispatch.ts:677-686`
  - body: Slash/native commands store session-scoped settings on `CommandTargetSessionKey`, but this dispatch path resolves the override from `SessionKey` first. In Slack and Discord native command flows, `/stream progress` can update the routed target session while the next response still reads the slash-command session and falls back to account config, so the command appears to succeed without changing delivery. Prefer `CommandTargetSessionKey ?? SessionKey ?? route.sessionKey` here and in the Discord dispatch path.
  - confidence: 0.88
- **[P3] Keep block delivery out of preview status:** `src/auto-reply/reply/commands-session.ts:121-124`
  - body: `blockStreaming` is the legacy block-delivery toggle, not preview-mode configuration. Including it in `hasConfiguredPreviewStreaming` makes `/stream status` report a configured preview source and an `off` effective mode for channels that only enabled block replies, hiding their preview default instead of showing the real inherited mode.
  - confidence: 0.78
