## What This Changes

The PR changes gateway post-attach startup so default provider-auth sidecar installation keeps the failure rewarm hook but skips the startup warm timer unless explicitly enabled.

## Best Possible Solution

Consolidate on one mergeable branch for the hook-only provider-auth startup default, keep explicit startup warming and rewarm paths intact, and land only after maintainer sign-off on the startup tradeoff.

## Reproduction Assessment

Yes. Current-main source shows the normal gateway caller provides providerAuthPrewarm without enabled and the sidecar schedules startup warm by default; linked operational reports provide matching prewarm stall and timeout symptoms.

## Solution Assessment

Yes, if maintainers accept the compatibility tradeoff. The PR targets the scheduler that creates the default warm, while sibling reload/logout paths and explicit startup warming remain available.
