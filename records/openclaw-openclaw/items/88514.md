## Label Justifications

- `P1`: The PR targets a startup/event-loop regression that can break gateway RPCs and channel responsiveness for real Windows/WSL2/Docker users.
- `merge-risk: 🚨 compatibility`: The startup default changes for existing users because default gateway startup no longer eagerly warms provider-auth state.
- `merge-risk: 🚨 auth-provider`: The changed behavior affects when provider authentication state is computed and cached, even though the auth decision path itself is preserved.

## Summary

Keep this PR open for maintainer review: it is maintainer-authored, draft, and carries a protected `maintainer` label, while the small gateway change appears directionally correct and no blocking code finding surfaced. The main merge concern is not a patch bug, but an intentional startup-contract change: provider auth state is no longer warmed by default and will be computed lazily or after the failure hook.

## What This Changes

The PR changes gateway post-attach startup so default startup registers the provider-auth failure rewarm hook without scheduling the provider-auth warm sweep unless `providerAuthPrewarm.enabled === true`, and adds focused regression coverage.

## Best Possible Solution

Land the narrow lazy-startup behavior only after maintainers accept the provider-auth timing change and record whether WSL2 proof is sufficient or native Windows proof is still required.

## Reproduction Assessment

Yes, source-level: current main production startup passes `providerAuthPrewarm` without `enabled`, and the current post-attach code schedules prewarm unless `enabled` is false. I did not rerun the live Docker/WSL2 failure locally; the PR body supplies Crabbox after-fix proof and the linked issue supplies before logs.

## Solution Assessment

Yes, likely: the PR keeps the failure rewarm hook and explicit opt-in startup warm path while removing the unsolicited default warm sweep. The remaining merge decision is upgrade/operational proof, especially whether native Windows proof is required before landing.
