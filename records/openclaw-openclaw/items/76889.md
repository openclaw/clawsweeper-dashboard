## Summary

Keep open: the report is recent, detailed, and matches a real Discord SecretRef failure boundary in current main. Current source still has Discord account/token resolution paths that throw when a source config with unresolved named-account token refs reaches them, while the runtime docs require using the active runtime snapshot for credential reads.

## What This Changes

Discord named-account startup and `channels.status` should resolve env-backed account token SecretRefs from the active runtime snapshot before account resolution reads them.

## Best Possible Solution

Centralize Discord runtime-config selection for account resolution so startup, status, target, allowlist, and plugin account paths read the resolved active snapshot when their caller config is the matching active source snapshot.

## Reproduction Assessment

No high-confidence end-to-end reproduction was established in this read-only pass. Source inspection confirms the unresolved SecretRef throw path and the relevant Discord account-resolution surfaces, but a failing current-main `channels.status` or startup regression still needs to be added.

## Solution Assessment

Unclear until the failing regression is pinned. The likely best fix is a shared Discord/runtime-config selection helper using the existing active snapshot contract, not per-call manual SecretRef resolution.
