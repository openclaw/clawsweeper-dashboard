## Summary

Keep open: current main still has a source-backed Discord account-resolution path that reads token SecretRefs before selecting the active runtime snapshot, and the only linked fix PR is closed unmerged.

## What This Changes

The issue asks for Discord named-account token SecretRefs to be resolved from the active runtime snapshot before startup/status/account-resolution code reads the token.

## Best Possible Solution

Add a shared Discord account-runtime config selection path so startup, status, target resolution, allowlist, and plugin account resolution read the resolved active snapshot when the caller passes the matching source config, with focused regression tests.

## Reproduction Assessment

Yes, there is a high-confidence source-backed reproduction path: set a runtime snapshot with a resolved Discord account token and matching source config with a SecretRef, then call Discord account resolution with the source config; current main reads the source token and throws before consulting the snapshot. I did not run the Vitest case in this read-only review.

## Solution Assessment

Yes, the issue's runtime-snapshot direction is the maintainable fix, but the closed related PR's env-only token-resolution approach is not enough evidence of the best complete solution. The safer implementation should route through the existing runtime config selector before Discord reads credentials.
