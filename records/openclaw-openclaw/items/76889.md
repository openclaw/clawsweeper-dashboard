## Summary

Keep open: current main still has Discord account resolution reading token fields strictly from the supplied config, while docs and secret-runtime tests support env-backed SecretRefs for named Discord accounts. The related fix PR was closed unmerged, so the shipped user problem remains unproven as fixed.

## What This Changes

The issue asks Discord named-account token SecretRefs to resolve from the active runtime snapshot before startup and channels.status read account tokens.

## Best Possible Solution

Add a shared Discord-side runtime-config selection path so account resolution uses the active runtime snapshot when the caller passes the matching source snapshot, then cover startup, status, and account-resolution regressions.

## Reproduction Assessment

Yes, source-level: a named Discord account with token as an env SecretRef will still throw if discordPlugin.config.resolveAccount receives the source config, because resolveDiscordToken reads the unresolved object strictly. I did not execute the failing Vitest case in this read-only pass.

## Solution Assessment

Yes, the reporter's runtime-snapshot direction is the narrowest maintainable fix. A direct Discord process.env fallback like the closed PR described is less aligned with the existing SecretRef provider/runtime snapshot contract.
