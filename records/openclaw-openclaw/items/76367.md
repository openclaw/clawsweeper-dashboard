## Summary

Keep open: current main still has a source-visible path where doctor/status can report no active memory runtime before proving the selected bundled `memory-core` runtime is unavailable.

## What This Changes

The issue reports that `openclaw doctor --non-interactive` warns that no active memory plugin is registered even when bundled `memory-core` is selected, loaded, and memory search works.

## Best Possible Solution

Make doctor and status resolve the selected memory runtime for `memory-core` in CLI health paths, or use gateway-backed active-memory evidence before warning, while keeping distinct diagnostics for no slot selected versus selected runtime unavailable.

## Reproduction Assessment

Yes. The issue is source-reproducible: current doctor emits the exact warning when local backend resolution returns null, and that lookup can remain cold even when `memory-core` is the selected active memory slot.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the memory runtime bootstrap and doctor/status health paths, not in a new config option or product-policy change.
