## Repair Work Prompt

Fix this issue: `openclaw doctor --non-interactive` and `openclaw status --json --all` can report no active memory plugin / `memory: null` even when bundled `memory-core` is selected and working. On current main, `noteMemorySearchHealth` calls `resolveActiveMemoryBackendConfig`, while `src/plugins/memory-runtime.ts` only consults `getLoadedRuntimePluginRegistry`; in CLI doctor/status paths there may be no active runtime registry even though the running gateway has memory-core loaded. Implement a narrow fix so selected memory-core can be resolved for doctor/status without broad plugin loading or changing plugin API behavior. Likely direction: use the appropriate scoped runtime loader for `plugins.slots.memory` (for example `resolveRuntimePluginRegistry`/load options with `onlyPluginIds: [slot]`) or otherwise trust gateway memory status before claiming no active plugin exists. Preserve `plugins.slots.memory = "none"`, avoid bootstrapping runtime in `closeActiveMemorySearchManagers`, and do not change external memory plugin policy. Add focused regression coverage for selected `memory-core` with no prior active registry, doctor not emitting the no-active warning, and status inspecting memory when memory search is explicitly configured. Add a user-facing `CHANGELOG.md` fix entry.
