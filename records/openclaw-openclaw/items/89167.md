## Summary

Keep open: current main and the latest release still preserve a failed main session when its transcript exists, and the TUI history path still rehydrates the selected session without a failed-session guard. The overlapping recovery PR is open but unmerged and appears only partially aligned with this issue because it clears terminal state in place while this issue asks for rotation or unbinding.

## What This Changes

The issue reports that a failed session can remain bound as `agent:main:main`, causing the next TUI launch to rehydrate the failed session and crash.

## Best Possible Solution

Choose one shared terminal-session recovery policy for main/TUI and channel sessions, coordinate it with https://github.com/openclaw/openclaw/pull/89045, and add focused TUI/session regression proof for the chosen behavior.

## Reproduction Assessment

No, not for the exact TUI crash. Source and tests show that current main can keep a failed main session reusable when its transcript exists, and the TUI history path does not filter that failed entry.

## Solution Assessment

Unclear. The open recovery PR fixes related terminal-session behavior by clearing failed state in place, but maintainers still need to decide whether this TUI case should rotate, unbind, or preserve transcript continuity.
