## Summary

Keep open: current main still rebuilds array-backed message content and returns a new message object even when no directive text changes, and the related implementation PRs in the provided context are closed unmerged.

## What This Changes

Preserve message and text-part object identity when `stripInlineDirectiveTagsFromMessageForDisplay` has no inline directive tags to remove.

## Best Possible Solution

Update the helper and focused tests so unchanged text parts keep their references, the original message is returned when no directives are stripped, and existing directive stripping semantics remain stable.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: pass an array-backed message with plain text and the helper will still map the content, spread the text part, and return a new message object even though the lower-level stripper reports no change.

## Solution Assessment

Yes. The narrow maintainable fix is to use `stripInlineDirectiveTagsForDisplay(...).changed`, rebuild only changed text parts, and return the original message when no part changed.

## Repair Work Prompt

Fix this issue by preserving object identity in `stripInlineDirectiveTagsFromMessageForDisplay` when no inline directive tags are stripped. In `src/utils/directive-tags.ts`, use the existing `stripInlineDirectiveTagsForDisplay(...).changed` result so unchanged text parts are returned by reference, changed text parts are rebuilt with stripped text, non-text parts remain untouched, and the original message object is returned when no content part changed. Do not change directive parsing, text stripping output, the exported helper signature, or plugin SDK exports. Add focused regression tests in `src/utils/directive-tags.test.ts` for an unchanged array-backed message returning the same object and mixed content preserving unchanged part references while rebuilding only changed text parts. Because this is an exported plugin SDK helper, add a concise single-line `CHANGELOG.md` fix/perf entry crediting @shuofengzhang if the repo's active changelog section is required for this user-visible SDK behavior.
