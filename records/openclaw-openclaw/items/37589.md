## Summary

Keep open: current main still rebuilds array-backed message content and returns a new message object even when no directive text changes, the identity regression tests are still absent, and the focused implementation PRs are closed unmerged.

## What This Changes

The issue requests that the display directive-tag helper preserve message and text-part object identity when no inline directive text is stripped.

## Best Possible Solution

Update the helper to rebuild only changed text parts and return the original message when all parts are unchanged, with focused identity regression tests and stable stripping semantics.

## Reproduction Assessment

Yes, at source level. An array-backed message with a plain text part follows the unconditional map/spread/new-message path even though the lower-level stripper would report `changed: false`.

## Solution Assessment

Yes. The narrow maintainable fix is to consume the existing `changed` result, reuse unchanged text-part references, track whether any part changed, and keep the exported API and text output stable.

## Repair Work Prompt

Fix the source-reproducible object identity churn reported at https://github.com/openclaw/openclaw/issues/37589. In `src/utils/directive-tags.ts`, update `stripInlineDirectiveTagsFromMessageForDisplay` so it calls `stripInlineDirectiveTagsForDisplay` once per text part, reuses the original text part when `changed` is false, tracks whether any part changed, and returns the original message object when no array content changed. Add focused regression tests in `src/utils/directive-tags.test.ts` using `toBe` for an unchanged array-backed message and for unchanged parts in mixed content. Do not change directive stripping semantics, exported types, SDK subpath exports, or package metadata. No changelog entry is expected unless maintainers decide this SDK identity behavior is user-facing release-note material.
