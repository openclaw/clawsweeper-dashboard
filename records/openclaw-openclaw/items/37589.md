## Summary

Keep open: current main still rebuilds array-backed message content and returns a new message object even when directive stripping makes no text change, and the focused implementation PRs linked from the discussion are closed unmerged.

## What This Changes

The issue requests preserving message and text-part object identity when inline directive stripping makes no content changes.

## Best Possible Solution

Land a narrow helper and regression-test update that reuses unchanged text-part references, tracks whether any part changed, and returns the original message when no directives were stripped.

## Reproduction Assessment

Yes, at source level. Passing an array-backed message with plain text through the current helper follows the unconditional map/spread/new-message path even though the lower-level stripper would report `changed: false`.

## Solution Assessment

Yes. The narrow maintainable fix is to use `stripInlineDirectiveTagsForDisplay(...).changed`, rebuild only changed text parts, and return the original message when no part changed.

Update `src/utils/directive-tags.ts` to call `stripInlineDirectiveTagsForDisplay` once per text part, use its `.changed` flag, return the original part when unchanged, track whether any part changed, and return the original message when no part changed. Preserve the helper signature, existing stripped text output, parse/delivery semantics, and SDK exports. Add focused tests in `src/utils/directive-tags.test.ts` for unchanged array-backed messages returning the same object with `toBe`, mixed content preserving unchanged text/non-text part identity while recreating only changed text parts, and existing stripping behavior.
