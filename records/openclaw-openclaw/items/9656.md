## Summary

Keep open; current main still flattens Signal `editMessage` envelopes into ordinary inbound messages, and the linked Signal fix PR is closed unmerged.

## What This Changes

The issue requests Signal `editMessage` envelope handling that preserves `targetSentTimestamp`, marks or replaces edited context, and avoids duplicate responses.

## Best Possible Solution

Implement edit-aware handling inside the bundled Signal plugin with typed target timestamps, edit metadata or an `[edited]` marker, target-timestamp dedupe or replacement, and mention-gated reply behavior.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` follows the ordinary message path and keeps the fresh timestamp as `MessageSid`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `extensions/signal`, because core dedupe already keys off the plugin-supplied `MessageSid` and no broad core seam is needed for the basic edit behavior.
