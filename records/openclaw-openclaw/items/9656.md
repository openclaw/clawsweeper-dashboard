## Summary

Keep open: current main and the latest release still flatten Signal `editMessage` envelopes into the normal inbound path, ignore the edit target timestamp for message identity, and have no edit-aware transcript, dedupe, marker, or reply-suppression behavior. The related prior PR is closed and unmerged, so it remains useful history rather than shipped behavior.

## What This Changes

Handle Signal `editMessage` envelopes by preserving edit metadata, linking edits to the original message timestamp, and avoiding duplicate context or duplicate replies.

## Best Possible Solution

Implement an explicit Signal edit path in the bundled Signal plugin with typed edit metadata, target-timestamp-aware identity, transcript/context handling, mention-gated reply suppression, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal message path, and identity is derived from `envelope.timestamp` rather than the edit target timestamp. I did not replay a live signal-cli daemon edit event.

## Solution Assessment

Unclear for the exact implementation: the Signal plugin is the right owner boundary, but maintainers still need to choose transcript replacement, marker metadata, dedupe, and reply-trigger semantics.
