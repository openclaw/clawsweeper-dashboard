## Summary

Keep this open: current main still flattens Signal `editMessage.dataMessage` into the ordinary inbound path, does not type or preserve `editMessage.targetSentTimestamp`, and has no edit-aware marker, transcript replacement, dedupe identity, or reply-suppression behavior. The related Signal PR is closed and unmerged, so it remains useful prior art rather than shipped behavior.

## What This Changes

The issue requests Signal `editMessage` envelope handling so edited messages preserve their original timestamp relationship, avoid duplicate context, and do not trigger duplicate replies unless explicitly mentioned.

## Best Possible Solution

Implement this in the bundled Signal plugin after choosing the edit semantics: type `editMessage.targetSentTimestamp`, route edits explicitly, preserve edit metadata or an `[edited]` marker, use the target timestamp for dedupe or transcript replacement, suppress unmentioned group edit replies as intended, and add focused Signal tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: feed the handler an inbound Signal payload with `envelope.editMessage.dataMessage`; current code treats it as normal `dataMessage` and assigns identity from `envelope.timestamp` rather than `targetSentTimestamp`.

## Solution Assessment

Yes in direction, but the exact behavior still needs maintainer semantics. The maintainable fix belongs inside the Signal plugin inbound handler with typed edit metadata and regression tests; the closed prior PR is useful history but not complete shipped behavior.
