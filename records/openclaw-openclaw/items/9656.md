## Label Justifications

- `P2`: Signal edits can produce duplicate responses and stale context, but the confirmed impact is confined to one bundled channel.
- `impact:message-loss`: Corrected content is processed without its relationship to the original message, allowing stale content or duplicate handling to remain effective.
- `impact:session-state`: Original and edited text can become independent context entries, causing the agent transcript and conversational state to drift.

## Summary

Current main and v2026.6.5 still flatten Signal edit envelopes into ordinary inbound processing, omit the original-message target from the typed envelope, and use the edit envelope timestamp as message identity. The source gap is clear, but transcript mutation and reply-trigger semantics require maintainer product judgment; the closed, unmerged prior PR is useful history rather than a canonical implementation.

## What This Changes

Add edit-aware Signal inbound processing that associates corrected content with the original message and prevents duplicate context or responses.

## Best Possible Solution

Keep Signal-specific envelope parsing in the bundled Signal plugin, carry a typed edit target through inbound normalization, and implement a maintainer-selected append-with-metadata or canonical replacement model with explicit direct-message and group-mention trigger rules plus focused regression tests.

## Reproduction Assessment

Yes at source level with high confidence: a valid signal-cli edit envelope is flattened to dataMessage and enqueued under the new envelope timestamp. A live Signal edit was not executed, so this is source reproduction rather than a real-environment reproduction.

## Solution Assessment

Not yet: edit detection and target identity are necessary, but the issue proposes several materially different transcript and response policies. Maintainers should select the canonical behavior before implementation rather than merging a partial marker-and-dedupe workaround.

## Risks / Open Questions

- Replacing an earlier transcript entry requires a canonical session-state mutation contract; implementing it only inside Signal could create channel-specific transcript behavior.

- Reusing targetSentTimestamp directly as a normal dedupe key could suppress the corrected content instead of replacing or annotating the original entry unless edit-aware dedupe behavior is designed explicitly.
