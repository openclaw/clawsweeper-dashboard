## Label Justifications

- `P2`: This is a normal-priority Signal channel behavior gap with clear source evidence and limited blast radius to edited Signal messages.
- `impact:message-loss`: Edited Signal content is currently processed as a separate inbound message instead of being associated with the original corrected message.
- `impact:session-state`: The behavior can leave original and edited Signal content side by side in session context, causing transcript state drift.

## Summary

Keep open: current main and the latest release still flatten Signal edit envelopes into normal inbound messages, ignore the edit target timestamp, and lack edit-aware transcript, dedupe, marker, or reply-suppression behavior. The prior Signal edit PR was closed unmerged and remains useful context rather than shipped behavior.

## What This Changes

Request Signal plugin handling for signal-cli editMessage envelopes so edited content is associated with the original message instead of treated as a separate inbound message.

## Best Possible Solution

Implement explicit edit handling in the bundled Signal plugin with typed targetSentTimestamp, target-aware transcript or dedupe behavior, edit context metadata or marker, mention-gated group reply suppression, and focused Signal regression tests.

## Reproduction Assessment

Yes, at source level: an inbound payload with envelope.editMessage.dataMessage enters the normal Signal message path and message identity is derived from envelope.timestamp rather than the edit target timestamp. I did not replay a live signal-cli edit event.

## Solution Assessment

Unclear for exact semantics: the Signal plugin is the right boundary, but maintainers need to choose whether edits replace transcript entries, add marker or structured metadata, dedupe on target timestamp, and suppress replies unless mentioned.
