## Label Justifications

- `P2`: This is a clear Signal channel behavior gap with limited blast radius to edited Signal messages and session context handling.
- `impact:message-loss`: Signal edits are currently not associated with the original corrected message and can be processed as separate inbound content.
- `impact:session-state`: Original and edited Signal content can remain side by side in session context, causing transcript state drift.

## Summary

Keep open: current main still flattens Signal edit envelopes into the normal inbound path, does not type or use `editMessage.targetSentTimestamp`, and has no edit-aware transcript replacement, marker, dedupe, or reply suppression. The linked Signal PR was closed unmerged, and the Telegram edit work does not supersede this Signal-specific gap.

## What This Changes

This issue requests Signal `editMessage` envelope handling so edited Signal messages are associated with the original message, marked or replaced in context, and do not trigger duplicate responses unless explicitly mentioned.

## Best Possible Solution

Implement explicit edit handling inside the bundled Signal plugin with typed `targetSentTimestamp`, target-aware dedupe or transcript replacement, visible edit metadata or marker, mention-gated reply suppression, and focused Signal regression tests.

## Reproduction Assessment

Yes, at source level: an inbound Signal payload with `envelope.editMessage.dataMessage` enters the normal message path, and identity still comes from `envelope.timestamp` rather than the edit target timestamp. I did not replay a live signal-cli edit event.

## Solution Assessment

Unclear for exact semantics: the Signal plugin is the right boundary, but maintainers need to choose the durable transcript, marker or metadata, dedupe, and reply-trigger policy before a final implementation.
