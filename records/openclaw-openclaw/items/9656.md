## Summary

Keep open: current main still flattens Signal edit envelopes into the ordinary inbound path, uses the edit envelope timestamp as the message identity, and lacks edit-aware transcript, dedupe, marker, or reply-suppression behavior.

## What This Changes

Request Signal plugin support for signal-cli `editMessage` envelopes, including target timestamp handling, edited-message context, dedupe or replacement, and mention-gated reply behavior.

## Best Possible Solution

Implement an explicit Signal edit path in the bundled Signal plugin with typed edit metadata, target-timestamp-aware identity, transcript/context handling, mention-gated reply suppression, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` currently enters the normal Signal message path, and `messageId` is derived from `envelope.timestamp` rather than the edit target timestamp. I did not replay a live signal-cli daemon edit event.

## Solution Assessment

Unclear for implementation: the fix belongs in the Signal plugin, but a marker-only patch would not settle transcript replacement, dedupe, and non-mentioned group edit reply-suppression semantics.
