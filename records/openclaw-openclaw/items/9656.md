## Label Justifications

- `P2`: Edited Signal messages can duplicate context and trigger unwanted replies, but the impact is bounded to one bundled channel workflow.
- `impact:session-state`: Original and edited Signal versions can both enter conversation context, causing transcript and agent-state drift.
- `impact:message-loss`: Corrected Signal content is not linked to the original message and can be misrepresented as a separate inbound event.

## Summary

Keep open. Current main and the latest release still flatten Signal `editMessage.dataMessage` into the ordinary Signal inbound path, do not type or use `editMessage.targetSentTimestamp`, and do not implement edit-aware context replacement, marker, dedupe, or reply suppression; the linked Signal PR is closed unmerged.

## What This Changes

Request Signal plugin support for signal-cli `editMessage` envelopes so edited content is tied to the original message instead of creating duplicate context or replies.

## Best Possible Solution

Add a Signal-owned edit-message path with typed target timestamp, explicit edit metadata or marker, maintainer-approved transcript/dedupe semantics, mention-gated reply policy, and focused Signal tests.

## Reproduction Assessment

Yes, at source level. A Signal event with `envelope.editMessage.dataMessage` follows the normal inbound path while `targetSentTimestamp` is neither typed nor used for message identity.

## Solution Assessment

Unclear as a final design. A plugin-local edit path is the right foundation, but transcript replacement, dedupe, and reply-trigger semantics need maintainer choice before implementation.
