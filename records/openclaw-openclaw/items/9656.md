## Summary

Keep open; current main still flattens Signal `editMessage` envelopes into ordinary inbound messages, uses the fresh envelope timestamp as `MessageSid`, and the related Signal fix PR is closed unmerged.

## What This Changes

Request Signal `editMessage` envelopes to be detected, marked or deduped against `targetSentTimestamp`, and prevented from causing duplicate replies unless explicitly mentioned.

## Best Possible Solution

Implement edit-aware inbound normalization in the Signal plugin with tests for the typed target timestamp, edited context, dedupe identity, and group mention gating.

## Reproduction Assessment

Yes, source-level. A receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` follows the ordinary data-message path and uses the fresh timestamp as `MessageSid`.

## Solution Assessment

Yes. The narrow maintainable solution belongs in `extensions/signal` because the plugin owns Signal envelope typing, message identity, and group admission before core dedupe dispatch.
