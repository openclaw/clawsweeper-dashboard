## Label Justifications

- `P2`: This is a clear Signal channel behavior gap with limited blast radius to edited Signal messages and session context handling.
- `impact:message-loss`: Edited Signal messages can be processed as separate inbound messages and can trigger duplicate replies instead of preserving the corrected message relationship.
- `impact:session-state`: Original and edited Signal content can both remain in agent context without linkage, causing transcript/session state drift.

## Summary

Keep open: current main and the latest release still flatten Signal `editMessage` envelopes into the normal inbound path without typing `targetSentTimestamp`, using edit-aware message identity, replacing or marking transcript context, or suppressing edit-triggered replies; the related Signal PR is closed and unmerged.

## What This Changes

Handle Signal `editMessage` envelopes by preserving the target timestamp relationship, marking or replacing edited transcript content, and avoiding duplicate responses when appropriate.

## Best Possible Solution

Implement explicit Signal edit handling in the bundled Signal plugin, with typed edit metadata, target-timestamp identity, transcript marker or replacement semantics, non-mentioned group edit suppression, and focused Signal tests.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal path, and message identity still comes from `envelope.timestamp` rather than `targetSentTimestamp`. I did not replay a live signal-cli edit event.

## Solution Assessment

Unclear for exact semantics: the Signal plugin is the right boundary, but maintainers need to choose durable edit metadata, transcript replacement versus marker-only dedupe, and reply-trigger behavior before implementation.
