## Label Justifications

- `P2`: Edited Signal messages can duplicate context and trigger unwanted replies, but the impact is bounded to one bundled channel workflow.
- `impact:session-state`: Original and edited Signal versions can both enter conversation context, causing transcript and agent-state drift.
- `impact:message-loss`: Corrected Signal content is not linked to the original message and can be misrepresented as a separate inbound event.

## Summary

Keep open. Current main and the latest release still flatten Signal edit envelopes into the ordinary inbound path, so edit identity, transcript replacement/dedupe, edit marking, and edit-specific reply suppression remain unimplemented.

## What This Changes

Request Signal plugin support for signal-cli `editMessage` envelopes so edited Signal messages keep their original-message relationship and do not create duplicate context or responses.

## Best Possible Solution

Add a Signal-owned edit-message path with typed edit target timestamp, explicit edit metadata or marker, maintainer-approved transcript/dedupe semantics, mention-gated reply policy, and focused Signal tests.

## Reproduction Assessment

Yes, at source level. A Signal event containing `envelope.editMessage.dataMessage` follows the normal inbound path, while `targetSentTimestamp` is not typed or used for message identity.

## Solution Assessment

Unclear as a final design. Plugin-local edit typing and routing are the right foundation, but transcript replacement, dedupe, and reply-trigger behavior need maintainer decision before an automated fix is safe.
