## Summary

Keep open; current main still accepts Signal edit envelopes through the normal inbound path without modeling the edit target, edit marker, transcript replacement, or edit-specific reply suppression.

## What This Changes

Handle Signal `editMessage` envelopes by preserving the edited-message relationship and avoiding duplicate or outdated agent context.

## Best Possible Solution

Implement the behavior inside the bundled Signal plugin with typed edit metadata, stable target-timestamp identity or transcript replacement, an edited marker or structured context metadata, mention-gated reply suppression, and focused Signal regression tests.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` follows the normal data-message path and uses the fresh envelope timestamp as `MessageSid`.

## Solution Assessment

Yes. The narrow maintainable direction is to fix this in `extensions/signal`; core already provides generic dedupe and channel-turn seams once the plugin supplies edit-aware identity and admission behavior.
