## Summary

Keep open; current main still flattens Signal edit envelopes through ordinary inbound handling, and the only linked Signal fix PR is closed unmerged.

## What This Changes

The issue requests Signal `editMessage` handling that preserves the original target timestamp relationship, marks or replaces edited context, and avoids duplicate automatic replies.

## Best Possible Solution

Implement this in the bundled Signal plugin with typed edit metadata, target-timestamp identity or transcript replacement, edited context metadata/marker, mention-gated reply suppression, docs/changelog coverage, and focused Signal regression tests.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` follows the normal data-message path and uses the fresh envelope timestamp as `MessageSid`.

## Solution Assessment

Yes. The narrow maintainable solution belongs in `extensions/signal`, because core dedupe already keys off `MessageSid` once the plugin supplies edit-aware identity and admission behavior.
