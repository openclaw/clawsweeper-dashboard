## Summary

Keep open; current main still lacks Signal edit-aware handling. Source inspection shows edit envelopes are flattened into normal inbound messages, `targetSentTimestamp` is not modeled for edits, and prior Signal fix attempts are closed unmerged.

## What This Changes

The issue requests Signal `editMessage` envelope handling so edited messages preserve their original timestamp relationship, avoid duplicate context, and avoid unintended duplicate replies.

## Best Possible Solution

Implement edit-aware handling inside the bundled Signal plugin with typed target timestamps, explicit edit metadata or marker, target-timestamp dedupe or transcript replacement, and mention-gated reply suppression.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` follows the ordinary data-message path and becomes a new `MessageSid` from the envelope timestamp.

## Solution Assessment

Yes. The narrowest maintainable solution belongs in `extensions/signal`, because core inbound dedupe already keys off `MessageSid` once the plugin supplies edit-aware identity and admission behavior.
