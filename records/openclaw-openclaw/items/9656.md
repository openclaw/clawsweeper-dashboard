## Summary

Keep open: current `origin/main` still flattens Signal `editMessage.dataMessage` into the normal inbound path, does not model or use `editMessage.targetSentTimestamp`, and the only related Signal fix PR is closed unmerged. The remaining edit semantics affect transcript history and reply triggering, so this needs maintainer product judgment before an automated repair.

## What This Changes

Request Signal plugin handling for signal-cli `editMessage` envelopes, including target timestamp linkage, edited-message context, optional dedupe/replacement, and mention-gated reply behavior.

## Best Possible Solution

Implement edit-aware handling in `extensions/signal`, preserving `targetSentTimestamp` metadata, marking or replacing edited content consistently, deduping against the original message when appropriate, and covering mention-gated reply behavior with focused tests.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` still follows the ordinary inbound path and would use the fresh timestamp as `MessageSid`.

## Solution Assessment

Unclear as fully specified. The Signal plugin is the right owner boundary, but maintainers should choose the exact transcript replacement, marker, dedupe, and reply-suppression semantics before implementation.
