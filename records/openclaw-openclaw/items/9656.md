## Label Justifications

- `P2`: This is a clear Signal channel behavior gap with limited blast radius to edited Signal messages and session context handling.
- `impact:message-loss`: Edited Signal messages can be duplicated through the normal inbound path and can trigger duplicate replies instead of preserving the corrected message relationship.
- `impact:session-state`: Original and edited Signal content can both remain in agent context without linkage, causing transcript and conversation state drift.

## Summary

Keep open: current main and v2026.5.22 still flatten Signal `editMessage` envelopes into the ordinary inbound path, do not model `targetSentTimestamp`, and still key Signal message identity from the edit envelope timestamp rather than the edited message target.

## What This Changes

Add Signal plugin handling for `editMessage` envelopes so edited messages preserve their relationship to the original message, avoid duplicate context, and do not trigger stale replies unintentionally.

## Best Possible Solution

Implement explicit Signal edit handling inside the bundled Signal plugin after maintainers choose durable edit metadata and whether edits should replace transcript entries, mark context only, suppress replies, or combine those behaviors.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal path, and message identity still comes from `envelope.timestamp` rather than `targetSentTimestamp`. I did not replay a live signal-cli edit event.

## Solution Assessment

Unclear for exact semantics: the Signal plugin is the right boundary, but maintainers need to choose durable edit metadata, transcript replacement versus marker-only dedupe, and reply-trigger behavior before implementation.
