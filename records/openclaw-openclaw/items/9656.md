## Summary

Keep open. Current main still flattens Signal `editMessage` envelopes into the ordinary inbound message path, does not model or use `targetSentTimestamp` for edit identity, and the related Signal PR #9701 is closed unmerged.

## What This Changes

Handle Signal `editMessage` envelopes by preserving edit metadata or an edited marker, tying edits to the original target timestamp, and avoiding duplicate responses for non-mentioned group edits.

## Best Possible Solution

Implement the edit handling inside the bundled Signal plugin with typed edit metadata, target-timestamp identity, an edited marker or structured context metadata, mention-gated reply suppression, and focused Signal regression tests.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, a fresh `envelope.timestamp`, and `editMessage.targetSentTimestamp` currently follows the ordinary data-message path and enqueues `MessageSid` from the fresh timestamp.

## Solution Assessment

Yes. The narrow maintainable solution is a fresh repair in `extensions/signal`, using PR #9701 only as prior art because it is closed, unmerged, and targeted an older layout.
