## Summary

Keep open; current main still flattens Signal `editMessage` envelopes into the ordinary inbound path, does not type or use `editMessage.targetSentTimestamp`, and related Signal fix work at https://github.com/openclaw/openclaw/pull/9701 is closed unmerged.

## What This Changes

The issue requests Signal edit-message handling for `editMessage` envelopes, including preserving the target timestamp, marking or replacing edited content, and avoiding duplicate responses.

## Best Possible Solution

Implement edit-aware handling inside `extensions/signal`, with typed edit target timestamps, explicit edit metadata or marker behavior, target-timestamp dedupe or transcript replacement, and focused Signal tests for context and mention-gated dispatch.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` follows the normal inbound path and keeps the fresh envelope timestamp as `MessageSid`.

## Solution Assessment

Yes for the owner boundary, but not yet for automation. The narrow implementation belongs in the bundled Signal plugin, while marker versus replacement and reply-suppression semantics need maintainer product judgment first.
