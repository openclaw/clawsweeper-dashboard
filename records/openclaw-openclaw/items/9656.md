## Summary

Keep open; current main still flattens Signal edit envelopes into ordinary inbound messages, does not type the edit target timestamp, and the Signal edit PRs found during review are closed and unmerged.

## What This Changes

Handle Signal `editMessage` envelopes by preserving edit metadata, deduping or replacing edited content, and avoiding unsolicited duplicate replies.

## Best Possible Solution

Implement edit-aware handling inside `extensions/signal` with typed `targetSentTimestamp`, explicit edit metadata or marker, target-timestamp dedupe or transcript replacement, mention-gated reply suppression, and focused tests.

## Reproduction Assessment

Yes, source-level. A receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` is flattened into the ordinary inbound path and uses the fresh timestamp as `MessageSid`.

## Solution Assessment

Yes for the owner boundary, but the final behavior needs maintainer judgment. The maintainable fix belongs in the bundled Signal plugin unless maintainers choose a broader cross-channel edit API.
