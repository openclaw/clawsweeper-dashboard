## Summary

Keep open: current main still flattens Signal edit envelopes into normal inbound handling, omits the edit target timestamp from the Signal event types, and uses the new envelope timestamp as the inbound message id; the related Signal edit PR is closed and unmerged.

## What This Changes

Add Signal editMessage handling that preserves targetSentTimestamp, marks or replaces edited context, dedupes against the original message, and avoids unwanted replies.

## Best Possible Solution

Implement edit-aware handling inside the bundled Signal plugin after maintainers choose the transcript and reply-trigger semantics, with focused tests for edit envelopes, target timestamp identity, context history, and group mention behavior.

## Reproduction Assessment

Yes, source-level. A Signal receive event with envelope.editMessage.dataMessage, editMessage.targetSentTimestamp, and a fresh envelope.timestamp still follows the ordinary inbound path and exposes the fresh timestamp as MessageSid.

## Solution Assessment

Unclear as fully specified. The Signal plugin is the right owner boundary, but maintainers should settle marker, transcript replacement, dedupe, and reply-suppression semantics before implementation.
