## Summary

Keep open; current main still lacks Signal edit-aware handling, and the closed Signal PR is useful prior art but not shipped behavior.

## What This Changes

Handle Signal `editMessage` envelopes by preserving edit metadata, deduping against the original message timestamp, and avoiding duplicate reply behavior.

## Best Possible Solution

Implement this inside `extensions/signal` with typed edit metadata, stable target-timestamp identity or transcript replacement, edited context marking, mention-gated reply suppression, and focused Signal tests.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` follows the normal data-message path and uses the fresh envelope timestamp as `MessageSid`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the bundled Signal plugin; core already dedupes by `MessageSid` once the plugin supplies the correct edit target identity.
