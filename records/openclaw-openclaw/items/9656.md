## Label Justifications

- `P2`: This is a normal-priority Signal channel behavior gap with clear source evidence and limited blast radius to edited Signal messages.
- `impact:message-loss`: Edited Signal messages can be handled as separate inbound messages instead of being associated with the original message.
- `impact:session-state`: The current path can leave original and edited content as separate session context entries, causing transcript state drift.

## Summary

Keep open: current main still treats Signal `editMessage.dataMessage` as normal inbound content, keys the queued message by the new envelope timestamp, and lacks edit-aware transcript replacement, metadata, dedupe, or reply suppression. The prior Signal edit PR at https://github.com/openclaw/openclaw/pull/9701 is closed and unmerged, and no canonical merged replacement showed up in the search pass.

## What This Changes

Request Signal inbound handling for `editMessage` envelopes, including `targetSentTimestamp` tracking, edited-message context, dedupe or replacement, and mention-gated reply behavior.

## Best Possible Solution

Implement an explicit Signal edit path in the bundled Signal plugin with typed target timestamp metadata, target-aware identity or transcript handling, mention-gated reply suppression, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal message path and identity is derived from `envelope.timestamp` rather than the edit target timestamp. I did not replay a live signal-cli daemon edit event.

## Solution Assessment

Unclear for exact semantics: the Signal plugin is the right boundary, but maintainers need to decide whether edits replace transcript entries, add markers or structured metadata, and when edited group messages should trigger replies.
