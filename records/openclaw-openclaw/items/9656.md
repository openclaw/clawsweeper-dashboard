## Summary

Keep open: current main still handles Signal `editMessage.dataMessage` as ordinary inbound content, does not type or use `editMessage.targetSentTimestamp`, and the prior Signal fix PR at https://github.com/openclaw/openclaw/pull/9701 is closed unmerged. The remaining work changes edit, dedupe, transcript, and reply-trigger semantics, so it needs maintainer product judgment before implementation.

## What This Changes

Handle Signal `editMessage` envelopes by preserving edit metadata, reducing duplicate context, and avoiding unwanted edit-triggered replies.

## Best Possible Solution

Add edit-aware handling inside the Signal plugin: type and carry `targetSentTimestamp`, mark or replace edited content consistently, dedupe against the original message where appropriate, and cover mention-gated reply behavior with focused tests.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` still follows the ordinary inbound path and would use the fresh timestamp as `MessageSid`.

## Solution Assessment

Unclear as fully specified. The Signal plugin is the right owner boundary, but maintainers should choose the transcript replacement, marker, dedupe, and reply-suppression semantics before implementation.
