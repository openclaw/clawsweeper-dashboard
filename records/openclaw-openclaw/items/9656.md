## Summary

Keep open: current main still lacks explicit Signal edit handling. The Signal handler still treats `editMessage.dataMessage` as ordinary inbound content, does not type or use `editMessage.targetSentTimestamp`, and the related Signal edit PRs are closed and unmerged rather than shipped fixes.

## What This Changes

Request Signal inbound handling for `editMessage` envelopes so edited Signal messages are recognized, tied to their original timestamp, and do not create duplicate or stale context.

## Best Possible Solution

Implement edit-aware handling inside `extensions/signal`, with typed `editMessage.targetSentTimestamp`, structured edit metadata or an `[edited]` marker, target-timestamp dedupe or transcript replacement, and focused Signal tests for inbound context and mention-gated reply behavior.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` still follows the ordinary inbound path and would use the fresh timestamp as `MessageSid`.

## Solution Assessment

Yes for the owner boundary, but not fully specified. The maintainable location is the bundled Signal plugin unless maintainers choose a broader cross-channel edit contract, and the reply-suppression/transcript-replacement semantics need product judgment.
