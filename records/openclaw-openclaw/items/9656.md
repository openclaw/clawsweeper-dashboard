## Label Justifications

- `P2`: Edited Signal messages can produce duplicate or stale agent context, but the impact is bounded to one bundled channel workflow.
- `impact:message-loss`: Corrected Signal content is not linked to the original message and can be misrepresented as a separate inbound event.
- `impact:session-state`: Original and edited Signal versions can both accumulate in conversation context, causing transcript and agent-state drift.

## Summary

Keep open. Current main and v2026.6.6 still flatten Signal edit envelopes into ordinary inbound messages, omit the edit target timestamp from the typed envelope, and use the new envelope timestamp as message identity; the only Signal-specific PR is closed unmerged.

## What This Changes

Request Signal plugin handling for signal-cli editMessage envelopes so edited content is linked to the original message, represented in context, and does not trigger duplicate replies unless policy allows it.

## Best Possible Solution

Add a Signal edit mode in the bundled plugin with target timestamp, structured edited metadata or marker, defined transcript supersede/dedupe semantics, mention-gated reply policy, and focused tests; add a shared session-edit seam only if the plugin SDK cannot express the chosen behavior.

## Reproduction Assessment

Yes, at source level with high confidence. Current main routes editMessage.dataMessage through normal inbound processing, does not type targetSentTimestamp, and derives message identity from the new envelope timestamp.

## Solution Assessment

No, the issue's proposed options are not yet a complete safe implementation. Plugin-local edit typing and routing are the right foundation, but maintainers need to define transcript replacement, dedupe, and reply-trigger policy together.
