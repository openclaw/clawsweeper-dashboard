## Label Justifications

- `P2`: Edited Signal messages can create duplicate or stale agent context, but the impact is bounded to one bundled channel workflow.
- `impact:message-loss`: Corrected Signal content is not linked to the original message and can be represented as a separate inbound event.
- `impact:session-state`: Original and edited Signal versions can both accumulate in conversation context, causing transcript and agent-state drift.

## Summary

Keep open. Current main and v2026.6.8 still flatten Signal `editMessage` content into the ordinary inbound path without modeling or using `targetSentTimestamp`, so the requested edit-aware context, dedupe, and reply-suppression behavior remains missing.

## What This Changes

Handle signal-cli `editMessage` envelopes as Signal message edits instead of independent inbound messages.

## Best Possible Solution

Add a Signal edit mode in the bundled Signal plugin with typed target timestamp, explicit edit metadata or marker, chosen transcript supersede/dedupe semantics, mention-gated reply policy, and focused Signal tests; add a shared SDK/session-edit seam only if plugin-local routing cannot express the chosen behavior.

## Reproduction Assessment

Yes, source-level with high confidence. A Signal event containing `envelope.editMessage.dataMessage` follows the normal inbound path, while `targetSentTimestamp` is not typed or used for message identity.

## Solution Assessment

Unclear as a final design. Plugin-local edit typing and routing are the right foundation, but transcript replacement, dedupe, and reply-trigger behavior need a maintainer decision before an automated fix is safe.
