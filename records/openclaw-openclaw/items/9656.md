## Label Justifications

- `P2`: This is a clear Signal channel behavior gap with limited blast radius to edited Signal messages and session context handling.
- `impact:message-loss`: Signal edits can be processed as separate inbound messages and trigger duplicate replies instead of preserving the corrected message relationship.
- `impact:session-state`: Original and edited Signal content can both remain in agent context without linkage, causing transcript or session state drift.

## Summary

Keep open: current main and v2026.5.20 still flatten Signal `editMessage` envelopes into the normal inbound path without typed edit target timestamps, edit-aware identity, transcript update semantics, markers, or reply-suppression behavior; the only linked Signal PR is closed unmerged.

## What This Changes

Request Signal channel support for `editMessage` envelopes so edited messages are linked to their original timestamp, represented in context, and do not create duplicate stale turns.

## Best Possible Solution

Implement explicit Signal edit handling in the bundled Signal plugin with typed target timestamps, target-aware transcript or dedupe policy, visible edit metadata, mention-gated reply suppression, and focused regression tests.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal path, and message identity still comes from `envelope.timestamp` rather than `targetSentTimestamp`. I did not replay a live signal-cli edit event.

## Solution Assessment

Unclear for exact semantics: the Signal plugin is the right boundary, but maintainers need to choose transcript, marker or metadata, dedupe, and reply-trigger policy before implementation.
