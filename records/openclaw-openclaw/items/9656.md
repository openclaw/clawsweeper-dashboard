## Summary

Keep open: current main still flattens Signal `editMessage.dataMessage` into the normal inbound path, keeps the new envelope timestamp as `MessageSid`, and has no edit-aware marker, transcript replacement, dedupe, or reply-suppression behavior. The related Signal PR was closed unmerged, and the remaining behavior needs maintainer product judgment before automation.

## What This Changes

Add Signal inbound handling for signal-cli `editMessage` envelopes so edited messages retain the original target timestamp, context marker or replacement, and mention-gated reply behavior.

## Best Possible Solution

Implement this inside the bundled Signal plugin by typing `editMessage.targetSentTimestamp`, routing edit envelopes through an explicit edit path, preserving edit metadata or an `[edited]` marker, using the target timestamp for dedupe or transcript replacement, suppressing non-mentioned group edit replies, and adding focused Signal tests.

## Reproduction Assessment

Yes, at source level: an inbound Signal payload with `envelope.editMessage.dataMessage` currently flows through the normal message path, and `messageId` is derived from `envelope.timestamp` instead of the edit target timestamp. I did not replay a live signal-cli daemon edit event.

## Solution Assessment

Yes for keeping this open, but the implementation shape needs maintainer judgment. A marker-only PR would not cover the full requested transcript and reply-suppression behavior unless maintainers explicitly choose that smaller scope.
