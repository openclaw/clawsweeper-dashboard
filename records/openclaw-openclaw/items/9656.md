## Summary

Keep open: current main still flattens Signal `editMessage.dataMessage` into the ordinary inbound path, uses the new envelope timestamp as `MessageSid`, and has no edit-aware marker, transcript replacement, dedupe, or reply-suppression behavior. The related Signal PR is closed unmerged, and the remaining behavior needs maintainer product judgment.

## What This Changes

Request Signal `editMessage` envelope handling that preserves `targetSentTimestamp`, marks or replaces edited content, and avoids duplicate responses for non-mentioned group edits.

## Best Possible Solution

Implement this in the bundled Signal plugin with typed edit metadata, target-timestamp-aware identity, explicit transcript/context handling, mention-gated reply suppression, and focused Signal regression coverage.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` currently enters the normal Signal message path, and `messageId` is derived from `envelope.timestamp` instead of the edit target timestamp. I did not replay a live signal-cli daemon edit event.

## Solution Assessment

Yes for keeping this open, but not for automated implementation yet. A marker-only patch would not settle the requested transcript replacement, dedupe, and non-mentioned group edit reply-suppression semantics.
