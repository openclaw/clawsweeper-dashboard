## Summary

Keep open. Current main still flattens Signal `editMessage` envelopes into the normal inbound path, does not type or use `targetSentTimestamp`, and the prior Signal edit PR is closed and unmerged.

## What This Changes

Signal editMessage envelopes should be recognized as edits, tied to the original target timestamp, marked or deduplicated in context, and prevented from triggering duplicate replies unless explicitly mentioned.

## Best Possible Solution

Implement this in the bundled Signal plugin with typed edit metadata, target-timestamp identity, an edited marker or structured context metadata, mention-gated reply suppression, and focused regression tests.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, a fresh `envelope.timestamp`, and `editMessage.targetSentTimestamp` currently follows the ordinary data-message path and enqueues `MessageSid` from the fresh timestamp.

## Solution Assessment

Yes. The narrow maintainable path is a fresh Signal plugin repair in `extensions/signal`, using PR #9701 only as prior art because it is closed, unmerged, and targeted the old `src/signal` layout.
