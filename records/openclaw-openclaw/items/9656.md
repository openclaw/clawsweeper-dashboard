## Summary

Keep open. Current main still flattens Signal `editMessage` envelopes into the ordinary inbound Signal path, ignores `targetSentTimestamp` for message identity, and the related Signal PR #9701 is closed unmerged, so the requested edit-aware behavior remains missing.

## What This Changes

Handle Signal `editMessage` envelopes by preserving the original target timestamp relationship, marking or replacing edited context, and avoiding duplicate reply behavior.

## Best Possible Solution

Implement edit handling inside `extensions/signal` with typed edit metadata, target-timestamp identity, an edited marker or transcript replacement, mention-gated reply behavior, and focused Signal tests.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, a fresh `envelope.timestamp`, and `editMessage.targetSentTimestamp` currently follows the normal data-message path and enqueues `MessageSid` from the fresh timestamp.

## Solution Assessment

Yes. The narrow maintainable solution belongs in the bundled Signal plugin; PR #9701 is useful prior art but is closed, unmerged, and should not be treated as shipped behavior.
