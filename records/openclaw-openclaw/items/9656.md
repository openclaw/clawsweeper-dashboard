- `impact:message-loss`: Corrected content may be duplicated, ignored through naive deduplication, or processed without its relationship to superseded text.

## Summary

Keep open: current main and v2026.6.5 still flatten Signal edit envelopes into ordinary inbound processing, omit the original-message target from the typed envelope, and use the edit envelope timestamp as message identity. The source gap is clear, but transcript mutation and reply-trigger semantics need maintainer product judgment.

## What This Changes

Add explicit Signal edit-envelope handling so edited messages preserve their relationship to the original and avoid duplicate context or unintended responses.

## Best Possible Solution

Represent Signal edits as a distinct normalized plugin event carrying validated target identity, then apply one maintainer-approved policy for context amendment or annotation and mention-aware reply triggering, with focused regression coverage.

## Reproduction Assessment

Yes at source level: current main visibly flattens `editMessage.dataMessage` into ordinary Signal processing without target identity or edit semantics. A live Signal edit was not exercised during this read-only review.

## Solution Assessment

Partly: explicit plugin-local edit normalization is the narrowest maintainable boundary, but using the target timestamp for deduplication alone is unsafe; maintainers should first choose amendment, annotation, and response-trigger semantics.
