## Label Justifications

- `P2`: This is a normal-priority Signal channel behavior gap with clear source evidence and limited blast radius to edited Signal messages.
- `impact:message-loss`: Signal edits are currently processed as separate inbound messages instead of being associated with the original corrected message.
- `impact:session-state`: The reported behavior can leave original and edited Signal content side by side in session context, causing transcript state drift.

## Summary

Keep open: current main and the latest release still flatten Signal `editMessage.dataMessage` into the normal inbound Signal path, do not type or use `targetSentTimestamp` for edited messages, and have no edit-aware transcript replacement, marker, dedupe, or reply-suppression behavior. The related Signal edit PR is closed and unmerged, while the broader cross-channel recall/edit issue is not a replacement for this Signal-specific source gap.

## What This Changes

The issue requests Signal-channel handling for signal-cli `editMessage` envelopes so edits are associated with the original message, represented in context, and do not cause duplicate replies unless explicitly mentioned.

## Best Possible Solution

Implement explicit edit handling in the bundled Signal plugin with typed `targetSentTimestamp`, target-aware transcript or dedupe behavior, edit context metadata or marker, mention-gated reply suppression, and focused Signal regression tests.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal message path and message identity is derived from `envelope.timestamp` rather than the edit target timestamp. I did not replay a live signal-cli edit event.

## Solution Assessment

Unclear for exact semantics: the Signal plugin is the right boundary, but maintainers need to choose the transcript replacement, edit marker or metadata, dedupe identity, and group reply-trigger policy.
