## Label Justifications

- `P2`: This is a normal-priority Signal channel behavior gap with clear source evidence and limited blast radius to edited Signal messages.
- `impact:message-loss`: Edited Signal messages are currently handled as separate inbound messages instead of being associated with the original message.
- `impact:session-state`: The reported behavior can leave both original and edited content in session context, causing transcript state drift.

## Summary

Keep open: current main and the latest release still flatten Signal `editMessage.dataMessage` into the ordinary inbound path, use the new envelope timestamp as message identity, and lack edit-aware transcript replacement, marker, dedupe, or reply-suppression behavior. The only matching Signal edit PR is closed and unmerged.

## What This Changes

The issue requests Signal `editMessage` envelope handling so edited messages are associated with the original timestamp, marked or replaced in context, and do not trigger duplicate responses.

## Best Possible Solution

Implement explicit Signal edit handling in the bundled Signal plugin with typed target timestamp metadata, target-aware identity or transcript handling, mention-gated reply suppression, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal message path and identity is derived from `envelope.timestamp` rather than the edit target timestamp. I did not replay a live signal-cli daemon edit event.

## Solution Assessment

Unclear for exact semantics: the Signal plugin is the right boundary, but maintainers need to decide whether edits replace transcript entries, add markers or structured metadata, and when edited group messages should trigger replies.
