## Label Justifications

- `P2`: The issue describes a normal-priority Signal channel behavior gap with limited blast radius but real duplicate-context and duplicate-response impact.
- `impact:message-loss`: OpenClaw can keep the stale pre-edit content as a separate message and lose the semantic relationship to the corrected Signal message.
- `impact:session-state`: Treating edited Signal messages as fresh inbound messages can leave duplicate or outdated context in the agent session.

## Summary

Current main still treats Signal `editMessage` envelopes as ordinary inbound messages: it reads the edited `dataMessage`, but does not model `targetSentTimestamp`, replace or mark prior context, or suppress non-mentioned edit-triggered replies. The issue should stay open for maintainer-directed Signal plugin work.

## What This Changes

Handle Signal `editMessage` envelopes by preserving the original target timestamp relationship, marking or replacing edited content in context, and avoiding duplicate agent responses for non-mentioned edits.

## Best Possible Solution

Implement this in the bundled Signal plugin with explicit edit envelope typing, target-timestamp-based identity or transcript update behavior, visible edited metadata, and mention-gated reply suppression with focused Signal tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a Signal envelope with `editMessage.dataMessage` enters the normal inbound path, while `editMessage.targetSentTimestamp` is not typed or used for identity, dedupe, transcript replacement, or reply suppression.

## Solution Assessment

Unclear. Explicit Signal edit handling in the plugin is the right owner boundary, but maintainers should choose the product semantics for replacement versus marker-only context and when edited group messages should trigger replies.
