## Label Justifications

- `P2`: Edited Signal messages can create duplicate or stale agent context, but the impact is bounded to one bundled channel workflow.
- `impact:message-loss`: Corrected Signal content is not linked to the original message and can be misrepresented as a separate inbound event.
- `impact:session-state`: Original and edited Signal versions can both accumulate in conversation context, causing transcript and agent-state drift.

## Summary

Keep open. Current main and the latest release still flatten Signal edit envelopes into ordinary inbound messages, so the requested edit identity, transcript replacement/dedupe, marker, and reply-suppression behavior remains unimplemented.

## What This Changes

The issue requests Signal edit-message handling that preserves the target timestamp relationship and avoids duplicate context or duplicate agent replies.

## Best Possible Solution

Add a Signal-owned edit-message path with typed target timestamp, explicit edit metadata or marker, maintainer-approved transcript replacement/dedupe semantics, mention-gated reply policy, and focused Signal tests.

## Reproduction Assessment

Yes, source-level with high confidence. A Signal event containing envelope.editMessage.dataMessage follows the normal inbound path, while targetSentTimestamp is not typed or used for message identity.

## Solution Assessment

Unclear as a final design. Plugin-local edit typing and routing are the right foundation, but transcript replacement, dedupe, and reply-trigger behavior need maintainer decision before an automated fix is safe.
