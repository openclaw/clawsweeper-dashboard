## Label Justifications

- `P2`: This is a clear Signal channel behavior gap with limited blast radius to edited Signal messages and session context handling.
- `impact:message-loss`: Edited Signal messages can be processed as separate inbound messages and can trigger duplicate replies instead of preserving the corrected message relationship.
- `impact:session-state`: Original and edited Signal content can both remain in agent context without linkage, causing transcript/session state drift.

## Summary

Keep open: current main still flattens Signal `editMessage` envelopes into the normal inbound path, does not type or use `editMessage.targetSentTimestamp`, and has no edit-aware transcript replacement, marker, dedupe, or reply-suppression behavior. The earlier Signal PR was closed unmerged, while the Telegram edit work is channel-specific and does not supersede this Signal gap.

## What This Changes

The issue requests Signal `editMessage` envelope handling so edited messages stay linked to their original message and do not create duplicate context or replies.

## Best Possible Solution

Implement explicit Signal edit handling in the bundled Signal plugin with typed `targetSentTimestamp`, target-aware dedupe or transcript replacement, visible edit metadata or marker, mention-gated reply suppression, and focused Signal regression tests.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal message path, and message identity still comes from `envelope.timestamp` rather than the edit target timestamp. I did not replay a live signal-cli edit event.

## Solution Assessment

Unclear for exact semantics: the Signal plugin is the right boundary, but maintainers need to choose the durable transcript, marker or metadata, dedupe, and reply-trigger policy before implementation.
