## Summary

Keep open: current main still normalizes Signal `editMessage` envelopes as ordinary inbound messages, so the requested edit metadata, target timestamp relationship, transcript dedupe/replacement, and edit-specific reply behavior are not shipped. The linked Signal edit PR at https://github.com/openclaw/openclaw/pull/9701 is closed and unmerged, so it is useful prior art but not a fix on main.

## What This Changes

The issue requests Signal `editMessage` envelope handling with edited context metadata, target timestamp dedupe or replacement, and mention-gated reply suppression for edits.

## Best Possible Solution

Implement explicit Signal edit handling inside the bundled Signal plugin, with typed `targetSentTimestamp`, edit metadata or `[edited]` context, target timestamp based dedupe or replacement, mention-aware reply suppression, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: deliver a Signal receive payload whose envelope has `editMessage.targetSentTimestamp` and `dataMessage`; current main reads the edited data as normal content, keeps `MessageSid` from the new envelope timestamp, and runs the normal dispatch path.

## Solution Assessment

Partially. The Signal plugin is the right owner boundary, but a marker-only patch like https://github.com/openclaw/openclaw/pull/9701 is not enough until maintainers decide the transcript replacement and reply-suppression semantics.
