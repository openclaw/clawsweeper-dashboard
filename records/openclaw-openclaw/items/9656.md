## Summary

Keep open; current main still flattens Signal edit envelopes into ordinary inbound handling, so the requested edit metadata, dedupe/replacement, edited marker, and non-mentioned edit reply suppression are not implemented. Related PR #9701 is useful prior work but was closed unmerged.

## What This Changes

Request Signal plugin handling for signal-cli `editMessage` envelopes by modeling `targetSentTimestamp`, marking or replacing edited context, deduping against the original message, and suppressing non-mentioned edit replies.

## Best Possible Solution

Implement the behavior in the bundled Signal plugin with typed edit metadata, target-timestamp identity or transcript replacement, edited context metadata/marker, mention-gated reply suppression, docs/changelog coverage, and focused Signal regression tests.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` follows the normal data-message path and uses the fresh envelope timestamp as `MessageSid`.

## Solution Assessment

Yes. The narrow maintainable solution belongs in `extensions/signal`, because core dedupe already keys off `MessageSid` once the plugin supplies edit-aware identity and admission behavior.
