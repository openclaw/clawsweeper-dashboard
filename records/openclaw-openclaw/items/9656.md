## Summary

Keep open; current main still treats Signal `editMessage` envelopes as ordinary inbound messages, and the only Signal-specific fix PR found is closed unmerged.

## What This Changes

The issue requests Signal `editMessage` envelope handling with target timestamp tracking, edit marker or metadata, dedupe or transcript replacement, and mention-gated reply behavior.

## Best Possible Solution

Implement edit-aware handling inside the bundled Signal plugin, with typed target timestamps, structured edit metadata or an `[edited]` marker, target-timestamp dedupe or transcript replacement, and focused Signal tests.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` follows the ordinary Signal inbound path and keeps the fresh timestamp as `MessageSid`.

## Solution Assessment

Yes for the owner boundary, but the final behavior needs maintainer judgment. The narrow maintainable implementation belongs in `extensions/signal`; core dedupe already uses the plugin-supplied `MessageSid`, so a broad core seam is not needed for the basic edit behavior.
