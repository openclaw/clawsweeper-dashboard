## Summary

Keep open: current main still flattens Signal edit envelopes into ordinary inbound messages, drops the edit target timestamp at the typed boundaries, and the only related Signal edit PR is closed unmerged.

## What This Changes

The issue requests Signal `editMessage` envelope handling so edited messages preserve their original timestamp relationship, avoid duplicate context, and do not trigger unwanted replies.

## Best Possible Solution

Implement edit-aware handling inside the bundled Signal plugin after maintainers settle semantics: type the edit target timestamp, preserve edit metadata or a marker, dedupe or replace against the original timestamp, suppress non-mentioned group edit replies, and add focused Signal tests.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` still follows the ordinary inbound path and exposes the fresh timestamp as `MessageSid`.

## Solution Assessment

Unclear as fully specified. The Signal plugin is the right owner boundary, but the marker, transcript replacement, dedupe, and reply-trigger behavior need maintainer product decisions before implementation.
