## Summary

Keep open: current main still treats Signal `editMessage` envelopes as normal inbound messages, derives identity from the new envelope timestamp, and lacks edit-aware transcript, dedupe, marker, or reply-suppression behavior. The prior Signal edit PR is closed and unmerged, so it remains useful prior art rather than shipped behavior.

## What This Changes

The issue requests Signal `editMessage` envelope handling so edited messages keep their relationship to the original message, avoid duplicate context or responses, and can be marked or replaced in transcript state.

## Best Possible Solution

Implement an explicit Signal edit path in the bundled Signal plugin with typed edit metadata, target-timestamp-aware identity, transcript or context handling, mention-gated reply suppression, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal message path, and identity is derived from `envelope.timestamp` rather than the edit target timestamp. I did not replay a live signal-cli daemon edit event.

## Solution Assessment

Unclear for the exact implementation: the Signal plugin is the right owner boundary, but maintainers still need to choose transcript replacement, marker metadata, dedupe, and reply-trigger semantics.
