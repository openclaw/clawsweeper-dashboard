## Summary

Keep open: current main still flattens Signal edit envelopes into the normal inbound path, does not type or use the edit target timestamp, and the prior Signal edit PR is closed unmerged; the remaining behavior needs maintainer decisions around transcript, dedupe, and reply-trigger semantics.

## What This Changes

The issue requests Signal `editMessage` envelope handling that preserves the original target timestamp, marks or replaces edited content, and avoids duplicate responses for non-mentioned edits.

## Best Possible Solution

Implement edit-aware handling inside the Signal plugin with typed `targetSentTimestamp`, consistent edited-context metadata, dedupe or replacement against the original timestamp, mention-gated reply behavior, and focused Signal tests.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` still follows the ordinary inbound path and exposes the fresh timestamp as `MessageSid`.

## Solution Assessment

Unclear as fully specified. The Signal plugin is the right owner boundary, but maintainers should choose the transcript replacement, marker, dedupe, and reply-suppression semantics before implementation.
