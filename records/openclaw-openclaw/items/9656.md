## Summary

Keep open. Current main still treats Signal edit envelopes as ordinary inbound messages: it flattens `editMessage.dataMessage`, ignores `targetSentTimestamp`, uses the new envelope timestamp as `MessageSid`, and has no edit marker, transcript replacement, or edit-specific reply suppression; related PR #9701 is closed and unmerged prior art.

## What This Changes

Handle Signal `editMessage` envelopes by preserving the original target timestamp, marking or replacing edited content, deduping against the original message, and suppressing non-mentioned edit replies.

## Best Possible Solution

Implement this in the bundled Signal plugin with typed edit metadata, explicit edit normalization, target-timestamp identity, an `[edited]` marker or structured edit metadata, mention-gated reply suppression, and focused regression tests.

## Reproduction Assessment

Yes. A source-level reproduction can feed a Signal receive event with a new `envelope.timestamp` and an `editMessage.targetSentTimestamp` pointing at the original message; current main flattens the edited data message, sets `MessageSid` from the new timestamp, and dispatches normally.

## Solution Assessment

Yes. The narrow maintainable path is to fix the Signal plugin's inbound normalization and tests; no core API gap or cross-channel refactor is needed for the first repair.
