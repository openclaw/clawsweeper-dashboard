## Summary

Keep open; current main still lacks Signal edit-aware handling, and the Signal edit PRs found during review are closed unmerged rather than shipped.

## What This Changes

Handle Signal `editMessage` envelopes by relating edits to their target timestamp, marking or replacing edited content, and preventing duplicate edit-triggered replies.

## Best Possible Solution

Implement edit-aware handling in `extensions/signal` with typed `targetSentTimestamp`, structured edit metadata or an `[edited]` marker, target-timestamp dedupe or transcript replacement, and focused tests.

## Reproduction Assessment

Yes, source-level. A receive event containing `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` is flattened into the ordinary inbound path and uses the fresh timestamp as `MessageSid`.

## Solution Assessment

Yes for the owner boundary, but the final behavior needs maintainer judgment. The maintainable fix belongs in the bundled Signal plugin; broader core dedupe changes are not needed unless maintainers choose a cross-channel edit API.
