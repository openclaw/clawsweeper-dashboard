## Summary

Keep open. Current main still treats Signal `editMessage` envelopes as normal inbound messages: the typed envelope omits `targetSentTimestamp`, the handler flattens `editMessage.dataMessage` into the ordinary message path, message identity still comes from the new envelope timestamp, and there is no edit marker, transcript replacement, or edit-specific reply suppression. The related Signal PR #9701 is closed and unmerged, so it remains useful prior art rather than shipped behavior.

## What This Changes

The issue asks OpenClaw to recognize Signal `editMessage` envelopes, connect edited content to the original message through `targetSentTimestamp`, and avoid duplicate context or unwanted replies for edited messages.

## Best Possible Solution

Implement this in the bundled Signal plugin, not in generic core. The Signal handler should type `editMessage.targetSentTimestamp`, route edits through an explicit edit path, preserve `[edited]` or structured edit metadata, use the target timestamp for dedupe or transcript replacement, suppress non-mentioned group edits from starting a reply, and add focused Signal tests around edit envelopes, message identity, history context, and mention-gated behavior.

## Reproduction Assessment

Yes. A focused reproduction can feed a Signal receive event with `envelope.timestamp` set to a new edit timestamp and `editMessage.targetSentTimestamp` set to the original message timestamp; current main will flatten the edited `dataMessage`, set `MessageSid` from the new envelope timestamp, and dispatch through the normal reply path.

## Solution Assessment

Yes for keeping this open and fixing it locally in the Signal plugin. PR #9701 is useful history, but because it is closed and unmerged, the maintainable path is a fresh narrow implementation against current `extensions/signal` code and tests.
