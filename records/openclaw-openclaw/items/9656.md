## Label Justifications

- `P2`: This is a clear Signal channel behavior gap with limited blast radius to edited Signal messages and session context handling.
- `impact:message-loss`: Edited Signal messages can be processed as separate inbound messages and can trigger duplicate replies instead of preserving the corrected message relationship.
- `impact:session-state`: Original and edited Signal content can both remain in agent context without linkage, causing transcript/session state drift.

## Summary

Keep open: current main and v2026.5.22 still flatten Signal `editMessage` envelopes into the normal inbound path without modeling `targetSentTimestamp`, edit-aware message identity, transcript replacement/marker behavior, or edit-specific reply suppression; the related Signal PR is closed and unmerged.

## What This Changes

Request Signal plugin support for `editMessage` envelopes so edited Signal messages are linked to the original message, marked or replaced in context, and do not cause duplicate responses.

## Best Possible Solution

Implement explicit Signal edit handling inside the bundled Signal plugin: type `targetSentTimestamp`, preserve edit metadata or an `[edited]` marker, update or dedupe transcript context by the target timestamp, suppress non-mentioned group edit replies, and cover the behavior with focused Signal tests.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal path, and message identity still comes from `envelope.timestamp` rather than `targetSentTimestamp`. I did not replay a live signal-cli edit event.

## Solution Assessment

Unclear for exact semantics: the Signal plugin is the right boundary, but maintainers need to choose durable edit metadata, transcript replacement or dedupe policy, and reply-trigger behavior before implementation.
