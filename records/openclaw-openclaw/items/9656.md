## Summary

Keep open; current main still flattens Signal edit envelopes into normal inbound handling without modeling the edit target, marking edited context, replacing/deduping against the original message, or suppressing edit-triggered replies.

## What This Changes

The issue requests Signal `editMessage` envelope support so edited messages keep their original-message relationship and do not create duplicate context or responses.

## Best Possible Solution

Implement this in the bundled Signal plugin with typed edit metadata, target-timestamp identity or transcript replacement, an edited marker or structured context metadata, mention-gated reply suppression, docs/changelog coverage, and focused Signal regression tests.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` follows the normal data-message path and uses the fresh envelope timestamp as `MessageSid`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `extensions/signal`, because core already dedupes by provider route and `MessageSid` once the plugin supplies edit-aware identity and admission behavior.

## Repair Work Prompt

Implement Signal `editMessage` envelope handling for this issue in the bundled Signal plugin. Start in `extensions/signal/src/monitor/event-handler.types.ts` and `extensions/signal/src/monitor/event-handler.ts`; do not move Signal-specific policy into core unless a small generic seam is demonstrably required. Type `editMessage.targetSentTimestamp`, detect edit envelopes explicitly, preserve an `[edited]` marker or structured edit metadata in the inbound context, use the target timestamp for stable identity/dedupe or transcript replacement, and suppress automatic group replies for edits unless the edited content explicitly mentions the agent or contains an authorized control command. Add focused regression coverage for edit envelopes, `MessageSid`/target timestamp behavior, pending/history context, and mention-gated reply suppression. Use PR #9701 only as prior art, not as shipped code. Add a user-facing `CHANGELOG.md` entry because this changes Signal behavior; update `docs/channels/signal.md` if the visible behavior needs explanation. Validate with targeted Signal tests and a Testbox `pnpm check:changed` before handoff.
