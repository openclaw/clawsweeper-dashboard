## Label Justifications

- `P2`: This is a clear Signal channel behavior gap with limited blast radius to edited Signal messages and session context handling.
- `impact:message-loss`: Signal edits can be processed as separate inbound messages and trigger duplicate replies instead of preserving the corrected message relationship.
- `impact:session-state`: Original and edited Signal content can both remain in agent context without linkage, causing transcript or session state drift.

## Summary

Keep open: current main and v2026.5.20 still flatten Signal `editMessage` envelopes into normal inbound handling without typed target timestamps, edit-aware identity, transcript updates, markers, or reply-suppression behavior; the earlier Signal PR is closed unmerged and the broader cross-channel lifecycle issue does not replace this Signal-specific gap.

## What This Changes

Request to detect Signal `signal-cli` `editMessage` envelopes, preserve `targetSentTimestamp`, mark or replace edited context, and avoid duplicate replies for non-mentioned edits.

## Best Possible Solution

Implement explicit Signal edit handling in the bundled Signal plugin after choosing durable edit semantics: typed target timestamps, target-aware transcript or dedupe policy, visible edit metadata or marker, mention-gated reply suppression, and focused Signal regression tests.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal path, and message identity still comes from `envelope.timestamp` rather than `targetSentTimestamp`. I did not replay a live signal-cli edit event.

## Solution Assessment

Unclear for exact semantics: the Signal plugin is the right boundary, but maintainers need to choose transcript, marker or metadata, dedupe, and reply-trigger policy before implementation.
