## Label Justifications

- `P2`: This is a clear Signal channel behavior gap with limited blast radius to edited Signal messages and their context handling.
- `impact:message-loss`: Edited Signal messages can be processed as separate inbound messages and trigger duplicate or outdated responses instead of preserving the corrected message relationship.
- `impact:session-state`: Original and edited Signal content can both remain in agent context without linkage, causing transcript/session state drift.

## Summary

Keep open: current main and v2026.5.22 still flatten Signal `editMessage` envelopes into the normal inbound path, and the only Signal-specific implementation PR found is closed and unmerged at https://github.com/openclaw/openclaw/pull/9701; the remaining work needs a maintainer choice on edit semantics.

## What This Changes

The issue requests Signal `editMessage` envelope handling that links edits to `targetSentTimestamp`, marks or replaces transcript context, and suppresses duplicate replies unless mentioned.

## Best Possible Solution

Add explicit edit normalization inside the Signal plugin with typed edit metadata, target-timestamp identity, chosen transcript semantics, mention-gated dispatch, and focused tests.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal path, and message identity still comes from `envelope.timestamp` rather than `targetSentTimestamp`. I did not replay a live signal-cli edit event.

## Solution Assessment

Unclear for exact semantics: the Signal plugin is the right boundary, but maintainers need to choose durable edit metadata, transcript replacement versus marker-only dedupe, and reply-trigger behavior before implementation.
