## Label Justifications

- `P2`: This is a clear Signal channel behavior gap with limited blast radius to edited Signal messages and session context handling.
- `impact:message-loss`: Edited Signal messages can be processed as separate inbound messages and can trigger duplicate replies instead of preserving the corrected message relationship.
- `impact:session-state`: Original and edited Signal content can both remain in agent context without linkage, causing transcript/session state drift.

## Summary

Keep open: current main and the latest release still flatten Signal `editMessage` envelopes into ordinary inbound messages, do not type the edit target timestamp, and lack edit-aware transcript, dedupe, marker, or reply-suppression behavior. The earlier Signal PR is closed and unmerged, and the Telegram edit work is channel-specific rather than a Signal fix.

## What This Changes

The issue requests explicit Signal `editMessage` handling that relates edits to the original message, marks or replaces edited content, and avoids duplicate agent replies unless mentioned.

## Best Possible Solution

Implement explicit Signal edit handling in the bundled Signal plugin with typed `targetSentTimestamp`, target-aware transcript or dedupe policy, visible edit metadata or marker, mention-gated reply suppression, and focused Signal regression tests.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal message path, and message identity still comes from `envelope.timestamp` rather than `targetSentTimestamp`. I did not replay a live signal-cli edit event.

## Solution Assessment

Unclear for the exact semantics: the Signal plugin is the right boundary, but maintainers need to choose durable transcript, marker or metadata, dedupe, and reply-trigger policy before implementation.
