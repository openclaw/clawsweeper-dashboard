## Label Justifications

- `P2`: Edited Signal messages can produce stale context or duplicate responses, but the affected workflow is bounded to edits on one bundled channel.
- `impact:message-loss`: The corrected content is not linked to the original message and can be mishandled as a separate inbound event.
- `impact:session-state`: Original and edited versions can independently accumulate in conversation context and cause state drift.

## Summary

Keep this issue open. Current main still flattens Signal edit envelopes into ordinary inbound messages, omits the original-message target from the typed envelope, and uses the edit envelope’s timestamp as message identity; the only linked implementation PR closed unmerged and no canonical replacement exists.

## What This Changes

Handle Signal message edits as edits, preserving their relationship to the original message and preventing stale duplicate context or unintended responses.

## Best Possible Solution

Add a typed Signal edit mode carrying the target timestamp and edited metadata, then define coordinated, tested rules for context representation and response triggering; introduce a generic session-edit seam only if the plugin SDK cannot express the chosen behavior.

## Reproduction Assessment

Yes, at source level with high confidence. Current main routes `editMessage.dataMessage` through normal inbound processing, does not type the target timestamp, and derives message identity from the new envelope timestamp; a live Signal edit was not executed.

## Solution Assessment

No, the issue’s alternatives are not yet one complete safe implementation. Explicit plugin-local edit typing and routing is the right foundation, but marker, replacement, dedupe, and reply policy must be designed together.

- Reply suppression needs defined behavior for direct messages, mention-gated groups, and edits received after the agent already responded to the original message.
