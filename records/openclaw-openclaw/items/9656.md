## Summary

Keep open. Current main still flattens Signal `editMessage.dataMessage` into the ordinary inbound path, does not type or use `editMessage.targetSentTimestamp`, derives `MessageSid` from the new envelope timestamp, and has no edit marker, transcript replacement, or edit-specific reply suppression; the related Signal PR is closed and unmerged prior art.

## What This Changes

Request Signal `editMessage` envelopes to be recognized, linked to `targetSentTimestamp`, marked or deduped in context, and prevented from causing duplicate replies unless explicitly mentioned.

## Best Possible Solution

Implement this in the bundled Signal plugin with typed edit metadata, explicit edit routing, context marking or replacement, target-timestamp identity, mention-gated reply suppression, and focused Signal regression coverage.

## Reproduction Assessment

Yes. A focused source-level reproduction can feed a Signal receive event whose envelope has a new `timestamp` and an `editMessage.targetSentTimestamp` pointing at the original message; current main will flatten the edited `dataMessage`, set `MessageSid` from the new timestamp, and dispatch normally.

## Solution Assessment

Yes. The narrow maintainable path is to fix the Signal plugin's inbound normalization and tests; PR #9701 is useful prior art, but because it is closed and unmerged, it should not be treated as the shipped solution.
