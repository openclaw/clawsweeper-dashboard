## Label Justifications

- `P2`: Edited Signal messages can duplicate context and trigger unwanted agent responses, but the affected workflow is bounded to one bundled channel.
- `impact:session-state`: The original and edited Signal text can both enter session context without an edit relationship, causing transcript state drift.
- `impact:message-loss`: Corrected Signal content is not linked to the original message and can be treated as a separate inbound message.

## Summary

Keep open. Current main and the latest release still flatten Signal `editMessage.dataMessage` into the ordinary inbound path, so edit identity, transcript replacement or marking, dedupe, and reply-suppression behavior remain unimplemented; the only Signal-specific fix PR is closed unmerged and remains useful prior art only.

## What This Changes

Handle Signal `editMessage` envelopes so edited Signal messages preserve their relationship to the original message and avoid duplicate session context or unwanted replies.

## Best Possible Solution

Add a Signal-owned edit-message path with typed edit target metadata, maintainer-approved transcript replacement or marker semantics, edit-aware dedupe, mention-gated reply behavior, and focused Signal tests.

## Reproduction Assessment

Yes, source-level with high confidence. A Signal event containing `envelope.editMessage.dataMessage` follows the normal inbound path, while `targetSentTimestamp` is not typed or used for message identity.

## Solution Assessment

Unclear as a final design. Plugin-local edit parsing and metadata are the right foundation, but transcript replacement, dedupe, and reply-trigger behavior need maintainer direction before an automated fix is safe.
