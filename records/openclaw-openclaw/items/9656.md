## Summary

Keep open; current main still lacks Signal edit-aware handling, and the related Signal edit PRs found during review are closed and unmerged rather than shipped.

## What This Changes

The issue requests Signal `editMessage` envelope handling that preserves edit metadata, relates edits to the original timestamp, and avoids duplicate context or replies.

## Best Possible Solution

Implement edit-aware handling in `extensions/signal` with typed `targetSentTimestamp`, structured edit metadata or an `[edited]` marker, target-timestamp dedupe or transcript replacement, and focused Signal tests.

## Reproduction Assessment

Yes, source-level. A receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` is flattened into the ordinary inbound path and uses the fresh timestamp as `MessageSid`.

## Solution Assessment

Yes for the owner boundary, but the final behavior needs maintainer judgment. The maintainable fix belongs in the bundled Signal plugin unless maintainers choose a broader cross-channel edit API.
