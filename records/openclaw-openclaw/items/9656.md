## Label Justifications

- `P2`: Edited Signal messages can duplicate stale context and trigger duplicate responses, but the impact is bounded to one bundled channel workflow.
- `impact:message-loss`: Corrected Signal content is not linked to the original message and can be represented as a separate inbound event.
- `impact:session-state`: Original and edited Signal versions can both accumulate in conversation context, causing transcript and session-state drift.

## Summary

Keep open. Current main and the latest release still flatten Signal edit envelopes into ordinary inbound messages, so the requested edit identity, transcript update/dedupe, marker, and reply-suppression behavior remains unimplemented.

## What This Changes

Handle Signal `editMessage` envelopes by preserving the original target timestamp, marking or updating edited content in context, and avoiding duplicate responses for non-mentioned edits.

## Best Possible Solution

Add a Signal-owned edit-message path with typed targetSentTimestamp, explicit edit metadata or marker, maintainer-approved transcript replacement/dedupe semantics, mention-gated reply policy, and focused Signal tests.

## Reproduction Assessment

Yes, source-level with high confidence: a Signal event containing envelope.editMessage.dataMessage follows the normal inbound path, while targetSentTimestamp is not typed or used for message identity. I did not replay a live Signal edit event during this read-only review.

## Solution Assessment

Unclear as a final design. Plugin-local edit typing and routing are the right foundation, but transcript replacement, dedupe, and reply-trigger behavior need maintainer decision before an automated fix is safe.
