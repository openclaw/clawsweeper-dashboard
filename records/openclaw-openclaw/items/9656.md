## Label Justifications

- `P2`: Signal edits can create stale or duplicate context and unintended replies, but the impact is scoped to one channel and needs product design rather than emergency repair.
- `impact:message-loss`: Current normalization discards the relationship between corrected content and the original Signal message, allowing stale text to remain authoritative.
- `impact:session-state`: One logical Signal message can be represented as multiple history or session-context entries after an edit.

## Summary

Current main still treats Signal edit envelopes as ordinary inbound messages, losing the original-message identity and applying normal history and reply behavior; the only linked Signal implementation closed unmerged and did not resolve transcript replacement or reply policy, so the issue remains necessary.

## What This Changes

Add first-class Signal `editMessage` handling that preserves the original-message relationship and defines context-update and response-trigger behavior.

## Best Possible Solution

Define a closed edit-aware Signal inbound mode that carries `targetSentTimestamp`, then implement one coherent policy for marked context or canonical replacement plus explicit DM, group-mention, command, and passive-ingest dispatch behavior.

## Reproduction Assessment

Yes, source inspection provides a high-confidence path: an envelope containing `editMessage.dataMessage` is accepted as a normal inbound message, while the local type and downstream path omit `targetSentTimestamp`. A live Signal edit was not executed during this read-only review.

## Solution Assessment

No, the closed marker-and-messageId proposal is not the best complete solution because dedupe is not transcript replacement and could discard corrected content; identity, context mutation, and reply gating should be designed and tested together.
