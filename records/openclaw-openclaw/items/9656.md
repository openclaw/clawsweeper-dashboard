## Summary

Keep open; current main still flattens Signal `editMessage` envelopes into normal inbound messages, and the Signal fix PR linked from the issue was closed unmerged with no current replacement.

## What This Changes

The issue asks OpenClaw to recognize Signal `editMessage` envelopes, preserve the original-message timestamp relationship, mark or replace edited content, and avoid duplicate edit-triggered responses.

## Best Possible Solution

Implement edit-aware handling inside the bundled Signal plugin with typed target timestamps, edit metadata or an `[edited]` marker, target-timestamp dedupe or replacement, and mention-gated reply behavior.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` follows the ordinary inbound path and keeps the fresh timestamp as `MessageSid`.

## Solution Assessment

Yes, with a maintainer choice on semantics. The narrow maintainable implementation belongs in `extensions/signal`, because core dedupe already keys off the plugin-supplied `MessageSid` and no broad core seam is needed for the basic edit behavior.
