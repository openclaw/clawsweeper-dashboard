## Summary

Keep open: current main still flattens Signal `editMessage.dataMessage` into the normal inbound path, uses the edit envelope timestamp as `MessageSid`, and has no edit-aware marker, transcript replacement, dedupe, or reply-suppression behavior. The related Signal PR is closed unmerged, so it is prior art rather than shipped behavior.

## What This Changes

The issue asks Signal inbound handling to recognize `editMessage` envelopes, preserve the `targetSentTimestamp` relationship, mark or replace edited context, and avoid duplicate automatic replies.

## Best Possible Solution

Implement an explicit Signal edit path in the bundled Signal plugin with typed edit metadata, target-timestamp-aware identity, transcript/context handling, mention-gated reply suppression, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` currently enters the normal Signal message path, and `messageId` is derived from `envelope.timestamp` instead of the edit target timestamp. I did not replay a live signal-cli daemon edit event.

## Solution Assessment

Unclear for implementation: the fix belongs in the Signal plugin, but a marker-only patch would not settle transcript replacement, dedupe, and non-mentioned group edit reply-suppression semantics.
