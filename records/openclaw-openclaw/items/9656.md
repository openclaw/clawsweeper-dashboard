## Label Justifications

- `P2`: This is a clear Signal channel behavior gap with limited blast radius to edited Signal messages and session context handling.
- `impact:message-loss`: Edited Signal messages can be processed as separate inbound messages and can trigger duplicate replies instead of preserving the corrected message relationship.
- `impact:session-state`: Original and edited Signal content can both remain in agent context without linkage, causing transcript/session state drift.

## Summary

Keep open: current main and the latest release still flatten Signal `editMessage` envelopes into the ordinary inbound path, omit `editMessage.targetSentTimestamp` from the typed envelope, and derive message identity from the new envelope timestamp, so the reported session/context duplication behavior remains unresolved.

## What This Changes

The issue requests Signal edit-envelope handling that links edited content to the original message, marks or replaces transcript context, dedupes by `targetSentTimestamp`, and avoids duplicate replies for non-mentioned edits.

## Best Possible Solution

Implement explicit Signal edit handling in the bundled Signal plugin with typed edit metadata, target-timestamp identity, chosen transcript marker or replacement semantics, non-mentioned group edit suppression, and focused Signal tests.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal path, and message identity still comes from `envelope.timestamp` rather than `targetSentTimestamp`. I did not replay a live signal-cli edit event.

## Solution Assessment

Unclear for the exact semantics: the Signal plugin is the right boundary, but maintainers need to choose durable edit metadata, transcript replacement versus marker-only dedupe, and reply-trigger behavior before implementation.
