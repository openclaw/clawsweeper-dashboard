## Label Justifications

- `P2`: The issue is a normal-priority channel behavior gap with clear source evidence but limited blast radius to Signal edited messages.
- `impact:message-loss`: Edited Signal messages can be duplicated or handled as separate inbound messages instead of being associated with the original message.
- `impact:session-state`: The reported behavior can leave both original and edited content in session context, causing transcript state drift.

## Summary

Keep open: current main still flattens Signal `editMessage.dataMessage` into the ordinary inbound path, uses the new envelope timestamp as message identity, and has no edit-aware transcript replacement, marker, dedupe, or reply-suppression behavior. The only matching Signal edit PR is closed and unmerged.

## What This Changes

The issue requests Signal `editMessage` envelope handling so edited messages preserve their original-target relationship, avoid duplicate session context, and suppress non-mentioned edit replies.

## Best Possible Solution

Implement an explicit Signal edit path in the bundled Signal plugin with typed target timestamp metadata, target-aware identity or transcript handling, mention-gated reply suppression, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal message path and identity is derived from `envelope.timestamp` rather than the edit target timestamp. I did not replay a live signal-cli daemon edit event.

## Solution Assessment

Unclear for exact semantics: the Signal plugin is the right boundary, but maintainers need to decide whether edits replace transcript entries, add markers or structured metadata, and when edited group messages should trigger replies.
