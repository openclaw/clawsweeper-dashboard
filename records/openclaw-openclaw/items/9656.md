## Summary

Keep open; current main still flattens Signal edit envelopes into the normal inbound path, does not model `editMessage.targetSentTimestamp`, and the related Signal edit PRs are closed and unmerged.

## What This Changes

The issue asks OpenClaw to detect Signal `editMessage` envelopes, preserve the original target timestamp relationship, mark or replace edited content, and avoid duplicate responses for non-mentioned edits.

## Best Possible Solution

Implement edit-aware handling in `extensions/signal` with typed `targetSentTimestamp`, structured edit metadata or an `[edited]` marker, target-timestamp dedupe or transcript replacement, and focused Signal tests.

## Reproduction Assessment

Yes, source-level. A receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` is flattened into the ordinary inbound path and uses the fresh timestamp as `MessageSid`.

## Solution Assessment

Yes for the owner boundary, but the exact behavior needs maintainer judgment. The maintainable implementation belongs in the bundled Signal plugin unless maintainers choose a broader cross-channel edit API.
