## Label Justifications

- `P2`: This is a clear Signal channel behavior gap with limited blast radius to edited Signal messages and session context handling.
- `impact:message-loss`: Edited Signal content can be duplicated or treated as a separate inbound message instead of preserving the corrected message relationship.
- `impact:session-state`: Original and edited Signal content can both remain in agent context without linkage, causing transcript and conversation state drift.

## Summary

Keep open: current main and the latest release still flatten Signal `editMessage` envelopes into the ordinary inbound path, do not type the edit `targetSentTimestamp`, and still key Signal message identity from the edit envelope timestamp. The prior fix attempt is closed unmerged, so it remains useful history rather than shipped behavior.

## What This Changes

The issue requests Signal `editMessage` envelope handling so edited messages preserve their relationship to the original message, avoid duplicate context, and suppress unintended duplicate replies.

## Best Possible Solution

Implement explicit Signal edit handling inside the bundled Signal plugin, with typed `targetSentTimestamp`, edit-aware message identity, durable edit metadata or transcript replacement, and focused tests for edit envelopes and mention-gated reply behavior.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal path, and message identity still comes from `envelope.timestamp` rather than `targetSentTimestamp`. I did not replay a live signal-cli edit event.

## Solution Assessment

Unclear for exact semantics: the Signal plugin is the right boundary, but maintainers need to choose durable edit metadata, transcript replacement versus marker-only dedupe, and reply-trigger behavior before implementation.
