## Summary

Keep open; current main still handles Signal `editMessage` envelopes as ordinary inbound messages, so edit metadata, target-timestamp dedupe/replacement, edited context marking, and edit-specific reply suppression are not implemented. The prior Signal PR at https://github.com/openclaw/openclaw/pull/9701 is useful context but is closed and unmerged.

## What This Changes

Add Signal `editMessage` handling so edited messages preserve their target timestamp, avoid duplicate context/replies, and show edit-aware context.

## Best Possible Solution

Implement edit-aware handling inside the bundled Signal plugin, keeping core generic and adding focused Signal tests plus user-facing changelog/docs coverage.

## Reproduction Assessment

Yes, source-level. A Signal receive event with `envelope.editMessage.dataMessage`, `editMessage.targetSentTimestamp`, and a fresh `envelope.timestamp` follows the normal data-message path and uses the fresh envelope timestamp as `MessageSid`.

## Solution Assessment

Yes. The narrow maintainable path is in `extensions/signal`, because core dedupe already keys on plugin-supplied `MessageSid` and does not need Signal-specific edit policy.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/9656 in the bundled Signal plugin. Current main flattens `envelope.dataMessage ?? envelope.editMessage?.dataMessage` and uses the fresh `envelope.timestamp` as `messageId`, so Signal edits are treated as new inbound messages. Add typed `editMessage.targetSentTimestamp`, detect edit envelopes explicitly, preserve edit awareness in context with an `[edited]` marker or existing structured metadata, use the target timestamp for valid dedupe/transcript identity or replacement, and suppress automatic replies for non-mentioned group edits while preserving mention-gated behavior and silent ingest expectations. Keep the fix in `extensions/signal`; do not add a new config option, do not move Signal-specific policy into core, and do not change Telegram behavior. Add focused regression tests and a user-facing changelog entry; update Signal docs if the edit semantics become documented behavior.
