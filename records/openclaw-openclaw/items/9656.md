## Summary

Keep open: current main still treats Signal `editMessage` envelopes as ordinary inbound messages, derives identity from the edit envelope timestamp, and has no edit-aware transcript, dedupe, marker, or reply-suppression behavior. The related implementation attempts are closed and unmerged, so they remain prior art rather than shipped behavior.

## What This Changes

Request Signal `editMessage` envelope handling that preserves the original target timestamp relationship, marks or replaces edited context, and avoids duplicate automatic responses.

## Best Possible Solution

Implement an explicit Signal edit path in the bundled Signal plugin with typed edit metadata, target-timestamp-aware identity, transcript/context handling, mention-gated reply suppression, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: an inbound payload with `envelope.editMessage.dataMessage` enters the normal Signal message path, and `messageId` is derived from `envelope.timestamp` rather than the edit target timestamp. I did not replay a live signal-cli daemon edit event.

## Solution Assessment

Unclear for the exact implementation: the fix belongs in the Signal plugin, but a marker-only patch would not settle transcript replacement, dedupe, and non-mentioned group edit reply-suppression semantics.
