## Summary

Keep open: the Feishu message-management feature is useful and has live delete-path proof, but the PR currently exposes a sensitive message list/delete/recall surface too broadly and ignores per-account disablement in the native tool path.

## What This Changes

The PR adds a Feishu `feishu_message` tool, generic `delete`/`unsend` message actions, latest bot-message fallback storage, config/schema metadata, docs, and tests for message list/delete/recall/read receipts.

## Best Possible Solution

Ship Feishu message management with explicit opt-in or split read/destructive gates, enforce the resolved account's gate at execution time, and keep docs/tests aligned with that contract.

## Reproduction Assessment

Yes for the review findings: source inspection shows `feishu_message` registers from an aggregate account gate but executes against the resolved account without rechecking its `tools.messages` setting, and `tools.messages` defaults to true.

## Solution Assessment

No, not yet; the SDK endpoint choice and tests look aligned with Feishu's contract, but the account gate and default exposure need a safer maintainable shape before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor per-account message tool gates:** `extensions/feishu/src/message-tool.ts:348-352`
  - body: `registerFeishuMessageTools` only checks that some enabled account has `tools.messages`, then `execute` creates a client for whatever `accountId` or contextual account was resolved. In a multi-account config where account A disables message tools and account B enables them, an agent can still pass account A and list/delete/recall through this tool, bypassing the explicit account setting. Resolve the account first and reject the call when that account's `tools.messages` is false.
  - confidence: 0.91
- **[P1] Keep message management opt-in by default:** `extensions/feishu/src/tools-config.ts:15`
  - body: Defaulting `tools.messages` to true exposes a new agent-callable list/delete/recall/read-receipt surface to existing Feishu configs as soon as the app has the matching scopes. Because the upstream delete endpoint can recall bot messages and, for privileged group bots, other users' group messages, this should be explicit opt-in or split into safer read/destructive gates before existing users receive it on upgrade.
  - confidence: 0.84
