## Summary

Keep open: current main and the latest release still use Feishu contact-directory APIs by default for sender-name resolution and live user listing. The issue remains valid and privacy-sensitive, but it overlaps open protected PR https://github.com/openclaw/openclaw/pull/73399 and needs maintainer coordination instead of a competing cleanup close or repair job.

## What This Changes

Request to make basic Feishu sender attribution and directory behavior avoid the broad contact-directory permission by default.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, fall back quietly to stable sender ids, and gate contact-backed profile lookup plus live user enumeration behind explicit opt-in with docs, regression tests, and live Feishu/Lark proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the default inbound path still calls resolveFeishuSenderName, that resolver still calls contact.user.get, and live peer listing still calls contact.user.list; no live tenant replay was run.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally: removing default contact-directory calls is the right privacy boundary, but the maintainable fix should be coordinated through the Feishu plugin with verified event or chat-member metadata and explicit opt-in semantics.
