## Summary

Keep open: current main still uses Feishu contact-directory APIs on the default sender-name path and live peer-listing path, while overlapping Feishu display-name PRs remain open or only partially address the permission-noise symptom.

## What This Changes

The issue requests making basic Feishu sender attribution and directory behavior work without requiring broad contact-directory read permission by default.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, use verified low-scope event or chat-member metadata where available, fall back quietly to stable sender ids, and require explicit opt-in for contact-backed profile lookup plus live user enumeration.

## Reproduction Assessment

Yes at source level: current main defaults into resolveFeishuSenderName, that resolver calls contact.user.get, and live peer listing calls contact.user.list. No live tenant replay was run.

## Solution Assessment

Yes directionally: a contact-free default with explicit opt-in for contact-backed lookup and listing is the narrow maintainable end state. The exact low-scope metadata source still needs Feishu/Lark API validation and maintainer security review.
