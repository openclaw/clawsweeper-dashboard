## Label Justifications

- `P1`: The current default Feishu path pressures organizations toward broad contact-directory permission for ordinary inbound messages and has multiple user confirmations.
- `impact:security`: The issue is about avoiding broad organization contact-directory access for sender attribution.
- `impact:auth-provider`: The failure mode centers on Feishu/Lark permission scopes and authorization prompts during channel operation.

## Summary

Keep open: current main and the latest release still keep Feishu sender-name lookup contact-backed by default, and live peer listing still enumerates contact users. The issue remains valid and security-sensitive, but changing Feishu permission/default behavior overlaps assigned maintainer follow-up and needs maintainer/security review instead of a competing automated fix.

## What This Changes

The issue asks OpenClaw to stop requiring Feishu organization-wide contact-directory permission for basic sender-name resolution and to make contact-backed directory lookup opt-in.

## Best Possible Solution

Make basic inbound Feishu operation contact-free by default, quietly fall back to stable sender ids, and require explicit documented opt-in for contact-backed profile and live directory lookup with regression tests and live Feishu proof.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.5` default `resolveSenderNames` to true, call `contact.user.get` for sender names, and call `contact.user.list` for live peer listing.

## Solution Assessment

Yes directionally, but the exact policy needs maintainer review. The maintainable path is a Feishu-plugin-owned contact-free default with explicit opt-in contact lookup and tests/docs for upgrade behavior.
