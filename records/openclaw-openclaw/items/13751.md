## Label Justifications

- `P1`: The issue affects real Feishu users by prompting or requiring broad contact-directory permission during ordinary message handling.
- `impact:security`: The reported behavior asks operators to grant organization-wide contact-directory access for a basic attribution feature.
- `impact:auth-provider`: The issue is grounded in Feishu authorization scopes and permission-grant prompts for contact API access.

## Summary

Keep open: current main and the latest release still default Feishu sender-name resolution to a contact-backed lookup, and live peer listing still enumerates contact users. The issue is valid, security-sensitive, and overlaps an assigned maintainer Feishu PR, so the next step is maintainer/security direction rather than closing or queueing another repair branch.

## What This Changes

The issue asks for Feishu sender-name resolution and live directory listing to stop requiring broad organization contact-directory read access for basic bot use.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, keep any contact-backed profile or live user-directory lookup behind an explicit operator choice, and align docs plus regression tests with that security posture.

## Reproduction Assessment

Yes. Source inspection reproduces the current-main behavior: ordinary Feishu message handling calls the sender-name resolver by default, the resolver calls `contact.user.get`, and live peer listing calls `contact.user.list`; I did not run a live Feishu tenant.

## Solution Assessment

Yes on direction, unclear on exact implementation. A contact-free default with explicit opt-in for contact-backed lookup is the safest maintainable target, but the non-contact Feishu API path and upgrade behavior need maintainer/security direction.
