## Summary

Keep open: current main and v2026.6.8 still default Feishu sender-name lookup to the contact-backed path, and live user listing still calls the contact directory API. The issue remains valid, but the fix changes sensitive Feishu permission/default behavior and overlaps an assigned maintainer-owned PR, so it needs maintainer and security coordination rather than cleanup close or a competing repair job.

## What This Changes

The issue asks for Feishu basic sender-name resolution and directory behavior to stop depending on broad contact-directory permissions by default.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, use verified low-scope metadata when available, fall back quietly to a stable sender id, and require explicit opt-in for contact-backed profile lookup and live user-directory listing with docs and regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main defaults resolveSenderNames to true and then calls client.contact.user.get for inbound sender names; I did not run a live Feishu tenant in this read-only review.

## Solution Assessment

Yes on direction, but not yet on final implementation shape. The maintainable fix should stay inside the Feishu plugin, make contact-backed lookups explicit, and validate the low-scope replacement path before changing defaults.
