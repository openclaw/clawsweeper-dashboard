## Summary

Current main and the latest release still perform default-on contact-backed sender-name resolution, so the permission-reduction request remains necessary. The assigned Feishu owner’s overlapping PR only adds a narrower DM lookup before retaining contact fallback; it is still open, conflicting, and paused for human review, so this issue should remain open for coordinated product and security judgment rather than a competing repair.

## What This Changes

Make basic Feishu sender-name resolution work without organization-wide contact-directory access and reconsider automatic contact-backed live user listing.

## Best Possible Solution

Use verified chat-member metadata for sender names where Feishu supports it, quietly fall back to the stable sender ID without prompting for broad contact access, and keep contact-directory lookup or live user enumeration as an explicit operator-controlled capability with focused documentation and tests.

## Reproduction Assessment

Yes, from source with high confidence: current main defaults sender-name resolution on, calls the Feishu contact API, and preserves only an explicit opt-out. A live tenant reproduction was not performed during this read-only review.

## Solution Assessment

The contact-free inbound default is the best security direction, but the exact fallback contract and treatment of explicit live directory commands require maintainer judgment; the existing overlapping PR is only a partial solution because it retains contact fallback.
