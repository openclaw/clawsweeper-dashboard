## Summary

Keep open. Current main still uses Feishu contact APIs by default for sender-name lookup and live peer listing; existing fixes only suppress one stale-scope prompt, while the overlapping maintainer PR and docs PR do not show a shipped contact-free default.

## What This Changes

Request that the Feishu plugin stop requiring broad contact-directory permission for basic sender-name resolution and gate contact-backed directory lookup behind opt-in behavior.

## Best Possible Solution

Make basic inbound Feishu contact-free by default: prefer verified event or chat-member metadata when available, fall back quietly to the stable sender id, and put contact-backed sender/profile and live user-directory lookups behind explicit opt-in with docs and regression tests.

## Reproduction Assessment

Yes, from source inspection rather than a live Feishu replay. With default config, inbound handling calls `resolveFeishuSenderName`, which calls `contact.user.get`, and live user directory listing calls `contact.user.list`.

## Solution Assessment

No, current main is not the best final solution because it only suppresses one stale-scope notification while still performing contact-backed lookups by default. A maintainable fix should change the default sender metadata path and make contact-directory lookup explicit.
