## Label Justifications

- `P1`: The issue affects real Feishu users now by making basic channel use depend on a sensitive contact-directory permission and noisy authorization prompts.
- `impact:security`: The report is about reducing default access to organization contact-directory data for ordinary sender attribution.
- `impact:auth-provider`: The broken behavior is tied to Feishu authorization scopes and permission-grant handling for the channel integration.

## Summary

Keep open: current main and the latest release still default Feishu sender-name resolution to a contact-backed lookup, and live user listing still uses the contact directory; the open related PR is only a partial, human-review path rather than a merged fix.

## What This Changes

Remove or gate Feishu contact-directory API calls so basic sender attribution does not require broad contact-directory permission.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, fall back quietly to stable sender ids, and require explicit opt-in for contact-backed profile lookup and live user-directory listing with docs and regression tests.

## Reproduction Assessment

Yes. Source inspection shows current main defaults `resolveSenderNames` to true and then calls `client.contact.user.get` for inbound sender names; I did not run a live Feishu tenant in this read-only review.

## Solution Assessment

Unclear on the final implementation shape. The right owner boundary is the Feishu plugin, but maintainers still need to choose and verify the low-scope replacement path and the opt-in behavior for contact-backed directory lookup.
