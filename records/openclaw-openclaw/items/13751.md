## Label Justifications

- `P1`: The default Feishu path asks for broad contact-directory access during normal sender attribution, affecting real organizations with strict permission policies.
- `impact:security`: The issue is about reducing unnecessary broad organization-directory access for basic Feishu bot operation.
- `impact:auth-provider`: The user-visible failure mode is tied to Feishu permission scopes and authorization prompts for contact APIs.

## Summary

Keep open: current main and the latest release still use Feishu contact-directory APIs on the default sender-name path and live user listing path. Related open PRs overlap sender display-name behavior, but none has merged a contact-free default, so this remains valid maintainer/security/product work.

## What This Changes

Request to make Feishu sender-name attribution and live user listing avoid broad contact-directory permission by default, falling back to stable IDs or explicit opt-in when needed.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, use verified event or low-scope chat-member metadata when available, fall back quietly to stable sender IDs, and require explicit opt-in for contact-backed profile lookup plus live user enumeration with docs, tests, and live Feishu proof.

## Reproduction Assessment

Yes at source level: current main defaults into resolveFeishuSenderName, that resolver calls contact.user.get, and live peer listing calls contact.user.list. No live tenant replay was run.

## Solution Assessment

Yes directionally: a contact-free default with explicit opt-in for contact-backed lookup and listing is the narrow maintainable end state. The exact replacement metadata source still needs Feishu/Lark API validation and maintainer security review.
