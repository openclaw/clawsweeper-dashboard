## Summary

Keep open: current main and the latest release still make Feishu sender-name lookup contact-backed and default-on, while the overlapping maintainer PR is open, human-review gated, and only partially overlaps the requested contact-free default.

## What This Changes

The issue requests removing Feishu's broad contact-directory dependency from basic sender-name resolution and making live directory lookup explicitly opt-in.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default inside the Feishu plugin, keep contact-backed profile and live user-directory lookup as explicit operator-controlled behavior, and cover the new precedence/defaults in docs plus focused Feishu tests.

## Reproduction Assessment

Yes, at source level: current main defaults resolveSenderNames to true, calls resolveFeishuSenderName for inbound messages, and that helper calls client.contact.user.get. I did not run a live Feishu tenant, but the current code, tests, docs, and latest tag all show the default contact-backed dependency remains.

## Solution Assessment

Unclear on final implementation shape, but not solved yet. The maintainable direction is Feishu-plugin-owned: use verified low-scope sender metadata when available, fall back quietly to stable ids, and make any contact-backed lookup or live directory enumeration an explicit, documented choice.
