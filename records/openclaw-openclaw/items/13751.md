## Summary

Keep open: current `main` and `v2026.5.2` still use Feishu contact-directory APIs by default for sender labels and live peer listing, so the least-privilege request remains valid. The issue should stay with maintainer follow-up because open maintainer PR #73399 overlaps the Feishu display-name path and the default/opt-in behavior needs product coordination.

## What This Changes

The issue requests making basic Feishu sender-name handling and directory listing avoid the broad `contact:contact.base:readonly` contact-directory permission by default.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default inside the Feishu plugin, use only upstream-verified event or chat-member metadata when available, fall back quietly to the stable sender id, and require explicit opt-in for contact-backed profile lookup and live user-directory listing with docs and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with `channels.feishu.resolveSenderNames` unset, inbound message handling reaches `client.contact.user.get`; `channel-list` peer/all scopes reach `client.contact.user.list`.

## Solution Assessment

Yes, the requested direction is the right maintainable shape, but current `main` only suppresses one stale permission prompt and does not remove the default contact API dependency. The implementation should coordinate with #73399 and keep any contact-backed lookup as an explicit operator choice.
