## Label Justifications

- `P1`: The default Feishu channel path still pressures real organizations toward broad contact-directory permission for basic sender attribution.
- `impact:security`: The issue is about avoiding broad organization contact-directory access for a basic inbound message workflow.
- `impact:auth-provider`: The failure mode centers on Feishu/Lark permission scopes and authorization prompts for channel operation.

## Summary

Keep open: current main and the latest release still enable contact-backed Feishu sender-name lookup and live user-directory lookup by default. The stale `contact:contact.base:readonly` prompt is partly suppressed now, but the underlying default contact API dependency remains and the open related PRs do not fully supersede this issue.

## What This Changes

Requests removing Feishu sender-name resolution’s default contact-directory dependency and making live directory lookup opt-in or otherwise contact-free.

## Best Possible Solution

Make basic inbound Feishu contact-free by default: use verified event or chat-member metadata when safe, fall back quietly to stable sender IDs, and require explicit documented opt-in for contact-backed profile and live directory lookup.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.19` default `resolveSenderNames` to true, call `contact.user.get` for sender names, and still call `contact.user.list` for live peer listing.

## Solution Assessment

Yes directionally, but it needs maintainer policy and security review. The safest path is a Feishu-plugin-owned default that avoids contact directory access for basic inbound messages, with explicit opt-in for contact-backed lookup and tests/docs for upgrade behavior.
