## Summary

Keep open: current `main` and the latest release still default Feishu inbound messages into `contact.user.get`, and `channel-list` still uses `contact.user.list`; overlapping PRs do not yet resolve the contact-free default and the issue is already routed for maintainer/security review.

## What This Changes

The issue asks to make Feishu sender-name handling and directory listing avoid broad contact-directory permission by default.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, use verified event or chat-member metadata where available, fall back quietly to stable sender ids, and require explicit opt-in for contact-backed profile lookup plus live user enumeration.

## Reproduction Assessment

Yes at source level: current `main` defaults into `resolveFeishuSenderName`, that resolver calls `contact.user.get`, and live peer listing calls `contact.user.list`. No live tenant replay was run.

## Solution Assessment

Yes directionally: a contact-free default with explicit opt-in for contact-backed lookup and listing is the narrow maintainable end state. The exact replacement metadata source still needs Feishu/Lark API validation and maintainer security review.
