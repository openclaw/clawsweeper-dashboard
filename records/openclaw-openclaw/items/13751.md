## Summary

Keep open: current main and the latest release still use Feishu contact-directory APIs on the default sender-name path and live peer listing path, while the overlapping Feishu display-name PR is still open for maintainer/security follow-up.

## What This Changes

The issue asks to make basic Feishu sender attribution and directory behavior stop depending on broad contact-directory permissions by default.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, use verified low-scope event or chat-member metadata where available, fall back quietly to stable sender ids, and require explicit opt-in for contact-backed profile lookup plus live user enumeration.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main defaults into resolveFeishuSenderName, that resolver calls contact.user.get, and live peer listing calls contact.user.list; no live tenant replay was run.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally, but the maintainable fix needs Feishu plugin coordination around verified low-scope metadata, stable-id fallback, and explicit opt-in for contact-backed lookup/listing.
