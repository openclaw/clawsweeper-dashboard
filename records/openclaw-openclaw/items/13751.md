## Summary

Keep open: current main still resolves Feishu sender names through `contact.user.get` by default and live peer listing still calls `contact.user.list`; the shipped opt-out and stale-scope suppression reduce noise but do not remove the contact-directory dependency.

## What This Changes

The issue asks to make basic Feishu sender attribution and user listing work without default contact-directory permissions.

## Best Possible Solution

Make basic inbound Feishu contact-free by default, fall back quietly to stable sender ids, and gate contact-backed profile lookup plus live user enumeration behind explicit opt-in with docs and regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main's default inbound path calls `resolveFeishuSenderName`, that resolver calls `client.contact.user.get`, and live peer listing calls `client.contact.user.list`; no live tenant replay was run.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally: removing default contact-directory calls is the right boundary, but the maintainable fix should be coordinated through the Feishu plugin and the existing overlapping PR with API-verified metadata and explicit opt-in semantics.
