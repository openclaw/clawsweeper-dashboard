## Summary

Keep open: current main and the latest stable release still default to contact-backed Feishu sender-name lookup, and live peer listing still uses the contact directory. The issue remains valid, but it overlaps an assigned maintainer-owned Feishu PR, so the next step is maintainer/security coordination rather than a competing automated fix PR.

## What This Changes

The issue requests making Feishu sender-name resolution and related peer listing usable without requiring broad Feishu contact-directory permissions.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, keep any contact-backed profile or directory lookup behind an explicit operator choice, and align docs plus regression tests with that security posture.

## Reproduction Assessment

Yes, source inspection reproduces the current-main behavior: ordinary Feishu message handling calls the sender-name resolver by default, and the resolver calls `contact.user.get`. I did not run a live Feishu tenant, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear without maintainer/security direction; the safest target is a contact-free default with explicit opt-in for contact-backed lookup rather than another hidden fallback path.
