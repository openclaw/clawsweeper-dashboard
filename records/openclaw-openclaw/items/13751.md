## Label Justifications

- `P1`: The current default Feishu channel path pressures organizations toward broad contact-directory permission for ordinary inbound messages.
- `impact:security`: The issue is about avoiding broad organization contact-directory access for sender attribution.
- `impact:auth-provider`: The failure mode centers on Feishu/Lark permission scopes and authorization prompts for channel operation.

## Summary

Keep open: current main and `v2026.5.20` still use Feishu contact-directory APIs for default sender-name lookup and live peer listing, so the security/auth-provider concern is still real and not fully superseded by related open PRs.

## What This Changes

The issue requests making Feishu sender-name resolution and related directory listing avoid broad contact-directory permission by default.

## Best Possible Solution

Make basic inbound Feishu operation contact-free by default, quietly fall back to stable sender IDs, and require explicit documented opt-in for contact-backed profile and live directory lookup.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.20` default `resolveSenderNames` to true, call `contact.user.get` for sender names, and still call `contact.user.list` for live peer listing.

## Solution Assessment

Yes directionally, but it needs maintainer policy and security review. The maintainable path is a Feishu-plugin-owned default that avoids contact directory access for basic inbound messages, with opt-in contact lookup plus tests/docs for upgrade behavior.
