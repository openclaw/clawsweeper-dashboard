## Summary

Current main still makes contact-backed sender-name resolution the default and uses the contact directory for live user listing, so the requested permission reduction remains necessary. Keep this issue open because it is assigned to a Feishu area owner and overlaps https://github.com/openclaw/openclaw/pull/73399, an open maintainer-authored PR that adds a permission-lighter DM lookup but retains contact fallback and is currently conflicting; the permanent default and fallback policy requires coordinated maintainer judgment.

## What This Changes

Make basic Feishu message handling work without organization-wide contact-directory permission, with quiet sender-ID fallback and non-default contact-backed directory access.

## Best Possible Solution

Keep ordinary inbound messaging contact-free by default: prefer trustworthy event-provided names, then chat-member metadata where available, quietly use the stable sender ID otherwise, and reserve contact profile lookup and organization-wide live user enumeration for an explicit operator-selected capability with clear permission documentation.

## Reproduction Assessment

Yes, at source level with high confidence: on current main, leave channels.feishu.resolveSenderNames unset and process an inbound message; the handler invokes resolveFeishuSenderName, which calls contact.user.get. A live Feishu tenant was not exercised during this read-only review.

## Solution Assessment

The requested contact-free default is the best security direction, but simply deleting lookup is insufficient because current events can expose only open_id; the maintainable solution should establish a tested event/member/ID fallback order and separately decide how existing resolveSenderNames semantics upgrade.
