## Label Justifications

- `impact:security`: Routine sender attribution currently invokes an API permission that exposes organization-wide contact data.
- `impact:auth-provider`: The reported failures and user prompts arise from Feishu API authorization and permission-scope handling.

## Summary

Current main still performs contact-backed sender-name lookup by default, so the central security and authorization concern remains. The issue should stay open for the assigned Feishu owner because the overlapping maintainer PR improves DM display-name fallback but explicitly preserves contact fallback and therefore does not yet satisfy the contact-free requirement.

## What This Changes

Make routine Feishu sender attribution and basic bot operation independent of organization-wide contact-directory permission.

## Best Possible Solution

Make ordinary inbound Feishu attribution contact-free by preferring verified event or chat-member metadata and quietly falling back to the stable sender ID; keep contact-backed profile lookup and live directory enumeration as explicit, clearly documented privileged actions rather than basic-message defaults.

## Reproduction Assessment

Yes, from source with high confidence: current main defaults sender-name resolution on and calls `contact.user.get` after a sender cache miss; live directory listing separately calls `contact.user.list`. A live tenant is still needed to compare which contact-free metadata is consistently available.

## Solution Assessment

Yes on the requested direction, but the proposed alternatives are not equally sufficient: verified event/chat-member metadata plus a quiet stable-ID fallback is narrower and safer than merely suppressing errors, while the permanent role of privileged contact lookup needs maintainer judgment.
