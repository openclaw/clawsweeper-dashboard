## Summary

Keep open: current main and the latest release still default Feishu sender-name lookup to the contact-backed path, and live peer listing still calls the Feishu contact directory. The issue remains valid, security-sensitive, and overlaps an assigned open Feishu display-name PR, so the next step is maintainer coordination rather than cleanup closure.

## What This Changes

Request to make Feishu sender attribution and live peer listing stop depending on broad contact-directory permissions by default.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, fall back quietly to a stable sender id, and require explicit opt-in for contact-backed profile lookup and live user-directory listing with docs and regression coverage.

## Reproduction Assessment

Yes, source-level. Current main defaults resolveSenderNames to true, calls client.contact.user.get for inbound sender names, and still uses client.contact.user.list for live peer listing; I did not run a live Feishu tenant in this read-only review.

## Solution Assessment

Yes on direction, but the final patch shape still needs maintainer choice. The Feishu plugin is the right owner boundary, but maintainers should choose the low-scope sender metadata path and opt-in semantics before changing defaults.
