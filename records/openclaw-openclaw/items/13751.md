## Summary

Current main still enables contact-backed Feishu sender-name resolution by default, and the overlapping open PR only adds chat-member lookup before retaining the disputed contact fallback. The issue remains necessary, but the permanent least-privilege fallback and directory-capability boundary needs coordination with the assigned contributor and protected maintainer PR.

## What This Changes

Remove Feishu’s broad contact-directory dependency from routine sender-name resolution and avoid noisy permission prompts.

## Best Possible Solution

Use verified event or chat-member metadata for routine inbound attribution, quietly fall back to the stable sender ID, and make contact-backed profile enrichment and live directory enumeration explicit operator-selected capabilities with focused tests and documentation.

## Reproduction Assessment

Yes from current-main source: default-enabled inbound sender-name resolution calls the Feishu contact API, and the linked PR still retains contact fallback. This read-only review did not run a live Feishu tenant.

## Solution Assessment

No, the currently linked PR is not the complete solution because it improves DM lookup while preserving the disputed contact dependency. The maintainable solution is a contact-free default with explicit enrichment and directory boundaries.
