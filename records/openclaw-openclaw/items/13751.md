## Summary

Keep open: current main and the latest release still resolve Feishu sender names through the contact API by default, and the related maintainer PR is still open rather than merged.

## What This Changes

The issue requests removing Feishu basic inbound sender-name dependence on broad contact-directory permission and making contact-backed directory/profile lookup explicit rather than default behavior.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default: use verified event or chat-member metadata when available, fall back quietly to a stable sender id, and require explicit opt-in for contact-backed profile lookup and live user-directory listing with docs and focused regression tests.

## Reproduction Assessment

Yes at source level: current main still calls Feishu contact APIs for sender-name resolution by default and for live peer listing. I did not run a live Feishu tenant in this read-only review, but the source path, docs, and tests establish the default dependency.

## Solution Assessment

Yes on direction, unclear on final implementation shape. The maintainable fix should stay inside the Feishu plugin, make contact-backed lookup explicit, and validate the low-scope sender-name source before changing defaults.
