## Summary

Keep open: current main and v2026.5.12 still use Feishu contact-directory APIs on the default sender-name and live user-listing paths, while overlapping Feishu display-name PRs remain open and need maintainer sequencing.

## What This Changes

Request to make Feishu sender attribution and live directory behavior avoid broad contact-directory permissions by default.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, fall back quietly to stable sender ids, and gate contact-backed profile lookup plus live user enumeration behind explicit opt-in with docs, tests, and live Feishu/Lark proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main defaults into `resolveFeishuSenderName`, that resolver calls `contact.user.get`, and live peer listing calls `contact.user.list`; no live tenant replay was run.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally, but the maintainable fix should be coordinated in the Feishu plugin by using verified event/chat-member metadata where possible and making contact-backed lookups explicit opt-in behavior.
