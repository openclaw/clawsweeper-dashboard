## Summary

Keep open. Current main still defaults Feishu sender-name resolution to contact-backed lookup, live peer listing still enumerates contacts, and the overlapping maintainer PR at https://github.com/openclaw/openclaw/pull/73399 is open and conflicting rather than a shipped contact-free default.

## What This Changes

Request contact-free basic Feishu sender attribution by avoiding contact-directory sender lookup by default and making live user-directory lookup explicitly opt-in.

## Best Possible Solution

Make basic inbound Feishu contact-free by default, fall back quietly to stable sender ids, and gate contact-backed profile lookup plus live user-directory listing behind explicit opt-in with docs and regression tests.

## Reproduction Assessment

Yes, from source inspection rather than a live Feishu replay. With default config, inbound handling calls `resolveFeishuSenderName`, which calls `contact.user.get`, and live peer listing calls `contact.user.list`.

## Solution Assessment

No, current main is not the best final solution because stale-scope notice suppression and an opt-out flag still leave contact-backed lookup on by default. The safer path is to make contact access explicit and keep basic inbound messaging contact-free.
