## Summary

Keep open: current main and the latest release still use Feishu contact-directory APIs on the default sender-name path and live peer-listing path, and the overlapping Feishu display-name PR is still open for maintainer/security follow-up.

## What This Changes

The issue asks to remove the Feishu channel's default contact-directory dependency for sender display names and avoid broad live user-directory lookup unless explicitly chosen.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, use verified low-scope event or chat-member metadata where available, fall back quietly to stable sender ids, and require explicit opt-in for contact-backed profile lookup plus live user enumeration.

## Reproduction Assessment

Yes at source level: current main defaults into resolveFeishuSenderName, that resolver calls contact.user.get, and live peer listing calls contact.user.list. No live tenant replay was run.

## Solution Assessment

Yes directionally: a contact-free default with explicit opt-in for contact-backed lookup/listing is the narrow maintainable end state. The exact metadata source still needs Feishu/Lark API validation and maintainer/security review.
