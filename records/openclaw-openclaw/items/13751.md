## Label Justifications

- `P1`: The issue affects real Feishu users now by making basic channel use depend on a sensitive organization-wide contact permission and noisy authorization prompts.
- `impact:security`: The report is about reducing default access to sensitive organization contact-directory data for ordinary sender attribution.
- `impact:auth-provider`: The broken behavior is tied to Feishu authorization scopes and permission-grant handling for the channel integration.

## Summary

Keep open: current main and the latest release still attempt Feishu contact-directory lookups for default sender-name resolution, and live peer listing still uses the contact user list API. The report remains valid, but an assigned maintainer and overlapping human-review PR make this a coordination item rather than a new automated repair.

## What This Changes

Request to make Feishu sender-name resolution and live peer listing stop requiring broad contact-directory read permission by default.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default: use verified low-scope event or chat-member metadata when available, fall back quietly to the stable sender id, and require explicit opt-in for contact-backed profile lookup and live user-directory listing with docs and regression tests.

## Reproduction Assessment

Yes. Source inspection shows current main defaults `resolveSenderNames` to true and then calls `client.contact.user.get` for inbound sender names; I did not run a live Feishu tenant because credentials and external API behavior are outside this read-only review.

## Solution Assessment

Yes on direction, but not yet on final implementation shape. The maintainable fix should stay inside the Feishu plugin, make contact-backed lookups explicit, and validate the low-scope replacement path before changing defaults.
