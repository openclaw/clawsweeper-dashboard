## Summary

Keep open: current main and the latest release still use the contact-backed Feishu sender-name path by default, and live peer listing still calls the contact directory. The report remains valid, but the issue is assigned and an overlapping maintainer-authored Feishu display-name PR is open with human-review/security blockers, so this should stay in maintainer coordination rather than close or spawn a competing automated fix.

## What This Changes

The issue asks for Feishu sender attribution and related directory listing behavior to avoid requiring organization-wide contact directory read permission by default.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default: use verified event or chat-member metadata when available, fall back quietly to a stable sender id, and require explicit opt-in for contact-backed profile lookup and live user-directory listing with docs and regression tests.

## Reproduction Assessment

Yes, source-level: with a configured Feishu account and `resolveSenderNames` unset, current main takes the default-true branch and calls `client.contact.user.get` for inbound sender attribution. I did not run a live Feishu tenant test in this read-only review.

## Solution Assessment

Yes on direction, but not as an automatic repair yet. The maintainable fix is a maintainer-approved contact-free default plus explicit contact-backed opt-in and docs/tests, not only suppressing the permission error noise.
