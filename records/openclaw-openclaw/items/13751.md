## Summary

Keep open: current main and the latest release still default Feishu sender-name resolution to a contact-backed lookup, live peer listing still uses the Feishu contact directory, and the overlapping maintainer PR is open/conflicting and does not close or fully own the contact-free default.

## What This Changes

The issue requests making basic Feishu sender attribution work without the broad contact-directory permission and gating contact-backed directory lookup behind explicit operator choice.

## Best Possible Solution

Make default Feishu inbound sender attribution contact-free, fall back quietly to a stable sender id, and gate contact-backed profile and live user-directory lookup behind explicit operator opt-in with docs and regression tests.

## Reproduction Assessment

Yes, source-level: current main defaults sender-name resolution on and still calls Feishu contact user APIs for sender names and live peer listing. I did not run a live Feishu tenant in this read-only review.

## Solution Assessment

Yes on direction, but the exact patch still needs maintainer choice. The Feishu plugin is the right owner boundary, but maintainers should choose the low-scope sender metadata path and opt-in semantics before changing defaults.
