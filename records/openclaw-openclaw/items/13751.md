## Summary

Keep open: current main and latest release still default Feishu sender-name resolution to the contact-backed lookup and still use the live contact directory for peer listing. The issue is valid and security-sensitive, but the fix changes Feishu permission/default behavior and overlaps an assigned maintainer-owned PR, so it should stay in maintainer review rather than be closed or queued as a competing repair.

## What This Changes

The issue requests that the Feishu plugin stop requiring broad contact-directory access for basic sender-name resolution and live user-directory behavior.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, fall back quietly to a stable sender id, and require explicit opt-in for contact-backed profile lookup and live user-directory listing with docs and regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main defaults `resolveSenderNames` to true and then calls `client.contact.user.get` for inbound sender names; I did not run a live Feishu tenant in this read-only review.

## Solution Assessment

Yes on direction, but unclear on final patch shape. The Feishu plugin is the right owner boundary, but maintainers still need to choose the low-scope sender metadata path and opt-in contact-directory behavior before changing defaults.
