## Summary

Keep open: current main and the latest release still default Feishu sender-name lookup to the contact-backed path, and live peer listing still calls the contact directory. The report remains valid, security-sensitive, and overlaps open maintainer-owned Feishu display-name work, so cleanup should stay in maintainer review instead of being closed or queued as a competing repair.

## What This Changes

The issue asks to make Feishu sender-name resolution and live directory listing work without requiring broad contact-directory permission by default.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, fall back quietly to a stable sender id, and require explicit opt-in for contact-backed profile lookup and live user-directory listing with docs and regression coverage.

## Reproduction Assessment

Yes, source-level. Current main defaults `resolveSenderNames` to true, calls `client.contact.user.get` for inbound sender names, and still uses `client.contact.user.list` for live peer listing; I did not run a live Feishu tenant in this read-only review.

## Solution Assessment

Yes on direction, but the final patch shape still needs maintainer choice. The Feishu plugin is the right owner boundary, but maintainers should decide the low-scope sender metadata path and explicit opt-in behavior before changing defaults.
