## Label Justifications

- `P1`: The issue affects real Feishu users now by making basic channel use depend on sensitive organization contact access and noisy authorization behavior.
- `impact:security`: The report is about reducing default access to sensitive organization contact-directory data for ordinary sender attribution.
- `impact:auth-provider`: The behavior is tied to Feishu authorization scopes, permission-grant prompts, and channel auth expectations.

## Summary

Keep open: current main and the latest release still default Feishu sender-name resolution on and still use contact-directory APIs, so the sensitive-permission dependency remains. The work is valid, but changing Feishu permission defaults and contact-backed fallbacks needs maintainer/security coordination, especially with overlapping open work in https://github.com/openclaw/openclaw/pull/73399.

## What This Changes

Remove the Feishu plugin's default dependency on broad contact-directory permissions for sender display-name resolution and live user listing.

## Best Possible Solution

Make basic inbound Feishu sender attribution contact-free by default, use verified low-scope event or chat-member metadata when available, fall back quietly to a stable sender id, and require explicit opt-in for contact-backed profile lookup and live user-directory listing with docs and regression coverage.

## Reproduction Assessment

Yes. Source inspection of current main shows `resolveSenderNames` defaults to true and inbound Feishu messages call `client.contact.user.get`; I did not run a live Feishu tenant in this read-only review.

## Solution Assessment

Yes on direction, but the final implementation shape needs maintainer/security approval. The maintainable fix should stay inside the Feishu plugin, make contact-backed lookups explicit, and validate the low-scope replacement path before changing defaults.
