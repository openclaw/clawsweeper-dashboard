## Summary

Keep open: current main still performs contact-backed Feishu sender-name lookup by default and still exposes live contact-directory listing; repeated user reports plus an overlapping maintainer-labeled PR make this valid but coordination-owned.

## What This Changes

Remove Feishu basic sender attribution and default directory behavior from the broad contact-directory permission path, using contact-free metadata or quiet ID fallback unless contact lookup is explicitly enabled.

## Best Possible Solution

Make basic inbound Feishu contact-free by default, fall back quietly to stable sender ids, and gate contact-backed profile lookup plus live user enumeration behind explicit opt-in with docs and regression tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main's default inbound path calls `resolveFeishuSenderName`, that resolver calls `client.contact.user.get`, and live peer listing calls `client.contact.user.list`; no live tenant replay was run.

## Solution Assessment

Is this the best way to solve the issue? No for current main: the opt-out flag and stale-scope suppression reduce noise but keep contact-backed access on default paths; the maintainable fix is a Feishu-owned contact-free default with explicit opt-in for directory-backed lookups.
