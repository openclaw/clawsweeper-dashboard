## Summary

Keep open: current main still uses Feishu contact-directory APIs on the default sender-name path and on live peer listing, while the overlapping maintainer PR has not merged or clearly resolved the least-privilege default.

## What This Changes

Request to remove Feishu’s default dependency on organization-wide contact-directory access for sender-name resolution and live peer listing.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default inside the Feishu plugin, with quiet sender-id fallback, documented opt-in contact-backed lookup, and explicit gating for live user-directory listing.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with `channels.feishu.resolveSenderNames` unset, inbound handling reaches `client.contact.user.get`, and `channel-list` peers/all reaches `client.contact.user.list`.

## Solution Assessment

Yes for the requested direction, but current main is not there yet. The narrow maintainable path should coordinate with #73399 and preserve explicit opt-in behavior for operators who want contact-backed names or live user discovery.
