## Label Justifications

- `P1`: The current default Feishu path pressures organizations toward broad contact-directory permission for ordinary inbound messages.
- `impact:security`: The issue is about avoiding broad organization contact-directory access for sender attribution.
- `impact:auth-provider`: The failure mode centers on Feishu/Lark permission scopes and authorization prompts for channel operation.

## Summary

Keep open: current main and the latest shipped release still keep Feishu sender-name lookup and live peer listing on contact-directory APIs by default, while related open PRs cover adjacent display-name work without resolving the broad contact-permission dependency.

## What This Changes

The issue requests making Feishu sender-name resolution and live directory listing avoid Feishu contact-directory scope by default.

## Best Possible Solution

Make basic inbound Feishu operation contact-free by default, quietly fall back to stable sender IDs, and require explicit documented opt-in for contact-backed profile and live directory lookup.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.20` default `resolveSenderNames` to true, call `contact.user.get` for sender names, and call `contact.user.list` for live peer listing.

## Solution Assessment

Yes directionally, but the exact policy needs maintainer review. The maintainable path is a Feishu-plugin-owned contact-free default with explicit opt-in contact lookup and tests/docs for upgrade behavior.
