## Summary

Keep open: current main and the latest release still resolve Feishu sender names through contact-directory APIs by default, and live peer listing still uses contact user enumeration. The issue overlaps open protected PR https://github.com/openclaw/openclaw/pull/73399, which is paused for human review, so this should stay with maintainer coordination rather than be closed or queued as a competing repair.

## What This Changes

Request that the Feishu plugin stop using broad contact-directory APIs by default for sender-name resolution and live peer listing.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, fall back quietly to stable sender ids, and gate contact-backed profile lookup plus live user enumeration behind explicit opt-in with docs and regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main's default inbound path calls `resolveFeishuSenderName`, that resolver calls `client.contact.user.get`, and live peer listing calls `client.contact.user.list`; no live tenant replay was run.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally: removing default contact-directory calls is the right privacy boundary, but the maintainable fix should be coordinated through the Feishu plugin and the existing protected PR with API-verified metadata and explicit opt-in semantics.
