## Summary

Keep open: current main and the latest release still use Feishu contact-directory APIs for default sender-name lookup and live user listing, and an assigned overlapping Feishu display-name PR remains open for human review.

## What This Changes

Request to make basic Feishu sender attribution work without broad contact-directory permission and to gate contact-backed directory lookup behind explicit behavior.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default, use verified event or chat-member metadata where available, fall back quietly to stable sender ids, and require explicit opt-in for contact-backed profile lookup plus live user enumeration with docs, tests, and live proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main defaults into resolveFeishuSenderName, that resolver calls contact.user.get, and live peer listing calls contact.user.list; no live tenant replay was run.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally, but the maintainable fix should be coordinated in the Feishu plugin by using verified low-scope metadata where possible and making contact-backed lookups explicit opt-in behavior.
