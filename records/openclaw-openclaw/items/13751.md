## Summary

Keep open: current main still uses Feishu contact-directory APIs on the default sender-name path and on live peer listing; existing changes only add opt-out controls and suppress one stale permission prompt, so the least-privilege request remains unresolved and should be coordinated with the overlapping maintainer PR.

## What This Changes

Request to make Feishu basic sender-name handling and peer listing avoid the organization-wide contact-directory permission by default.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default inside the Feishu plugin, using verified event or chat-member metadata where available, quiet sender-id fallback, and explicit opt-in for contact-backed profile lookup and live directory listing.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with `channels.feishu.resolveSenderNames` unset, inbound message handling reaches `client.contact.user.get`, and `channel-list` peers/all reaches `client.contact.user.list`.

## Solution Assessment

Yes for the requested direction, but current main is not there yet. The maintainable fix should coordinate with #73399 and preserve explicit opt-in paths for operators who want contact-backed names or live user discovery.
