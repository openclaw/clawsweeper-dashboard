## Summary

Keep open: current main still has the reported least-privilege problem. Default Feishu sender-name resolution still reaches the contact user API, the config/docs/tests keep that lookup default-on, and live peer listing still enumerates contact users; the issue is assigned and overlaps an open maintainer-labeled Feishu PR, so this should stay in maintainer coordination rather than be closed or queued as a competing automated repair.

## What This Changes

The issue asks for Feishu basic sender-name resolution and live directory listing to stop depending on broad contact-directory permission by default.

## Best Possible Solution

Make ordinary inbound Feishu messages contact-free by default inside the Feishu plugin: use verified event or chat-member metadata when available, quietly fall back to a stable sender id, and put contact-backed profile lookup plus live user-directory listing behind explicit operator opt-in with docs and regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with `channels.feishu.resolveSenderNames` unset on a configured Feishu account, `handleFeishuMessage` calls `resolveFeishuSenderName`, which calls `client.contact.user.get`; live peer listing similarly calls `client.contact.user.list`.

## Solution Assessment

Yes for the requested direction, but not as a close decision. The narrow maintainable fix should coordinate with PR #73399 and change the default sender-label and live-directory behavior without removing explicit opt-in contact lookup support.
