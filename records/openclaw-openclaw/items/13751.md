## Summary

Keep open: current main has a partial mitigation for stale `contact:contact.base:readonly` prompts, but Feishu sender-name resolution and live peer listing still use contact-directory APIs by default, so the least-privilege request remains valid and needs maintainer coordination with the overlapping Feishu PR.

## What This Changes

Make Feishu inbound sender-name resolution and directory listing avoid broad contact-directory permissions by default, using quiet ID fallback or explicit opt-in for contact-backed lookup.

## Best Possible Solution

Make basic inbound Feishu messages contact-free by default inside the Feishu plugin, with verified event or chat-member metadata when available, stable sender-id fallback, and explicit operator opt-in for contact-backed profile lookup and live user-directory listing.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with `channels.feishu.resolveSenderNames` unset, inbound Feishu message handling reaches `client.contact.user.get`; `channel-list` peers/all similarly reaches `client.contact.user.list`.

## Solution Assessment

Yes for the requested direction, but current main is not there yet. The maintainable fix should coordinate with #73399 and change defaults without removing an explicit opt-in path for operators who want contact-backed names or live user discovery.
