## Summary

Keep open. Current main still keeps Feishu contact-backed sender-name lookup enabled by default and still uses live contact-directory enumeration for peer listing; the related open maintainer PR overlaps display-name behavior but does not clearly deliver the contact-free default requested here.

## What This Changes

The issue asks to make basic Feishu sender attribution and live peer discovery avoid broad contact-directory access by default.

## Best Possible Solution

Make basic inbound Feishu contact-free by default: prefer verified event or chat-member metadata when available, fall back quietly to the stable sender id, and put contact-backed profile lookup plus live user-directory listing behind explicit opt-in with docs and regression tests.

## Reproduction Assessment

Yes, from source inspection rather than a live Feishu replay. With default config, inbound handling calls `resolveFeishuSenderName`, which calls `contact.user.get`, and live user directory listing calls `contact.user.list`.

## Solution Assessment

No, current main is not the best final solution because it only suppresses one stale-scope notice while still performing contact-backed lookups by default. A maintainable fix should change the default sender metadata path and make contact-directory lookup explicit.
