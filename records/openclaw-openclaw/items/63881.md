## Summary

Keep open. The MCP hardening target is valid on current main, but this branch is stale against the active MCP handler split, lacks after-fix real MCP proof, and mixes security policy with dependency-release-age policy.

## What This Changes

The branch splits gateway dangerous-tool deny constants, filters the plugin-tools MCP server with the universal deny list and `gateway.tools` overrides, adds MCP boundary tests, and adds an axios release-age exclusion.

## Best Possible Solution

Rebase into the current MCP handler/shared-server path, enforce the universal dangerous-tool deny list at both list and call boundaries without losing `ownerOnly` or `before_tool_call` behavior, document/test MCP `gateway.tools` semantics, and keep axios policy in separate dependency work.

## Reproduction Assessment

Yes. A source-level reproduction is to instantiate `createPluginToolsMcpHandlers` on current main with an `exec` tool: `listTools` includes it and `callTool` dispatches it because the active handler only filters `ownerOnly` tools and wraps `before_tool_call`.

## Solution Assessment

No. The universal-versus-HTTP-only split is a useful direction, but this branch applies it to the stale monolithic MCP server shape and mixes in dependency policy; a handler-level fix on current main is the safer solution.
