## Summary

Keep this PR open. Current main still lacks dangerous-tool deny-list enforcement on the plugin-tools MCP path, so the security hardening remains useful, but the branch is dirty against the current MCP handler split and still carries a supply-chain-sensitive axios release-age exclusion that should not be bundled with the MCP authorization fix.

## What This Changes

The branch adds a universal versus HTTP-only dangerous-tool deny list, applies the universal set with gateway.tools overrides to the MCP plugin-tools server, adds MCP security-boundary tests, and adds a blanket axios minimumReleaseAge exclusion.

## Best Possible Solution

Rebase or replace the branch with a narrow MCP hardening change: add a universal dangerous-tool deny constant, enforce it in the active plugin-tools MCP handler/server path for both listTools and callTool, preserve ownerOnly and before_tool_call behavior, apply gateway.tools.allow and gateway.tools.deny consistently, document the broadened gateway.tools MCP semantics, and split any axios dependency policy into its own already-reviewed security/dependency path.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to inspect or instantiate the current createPluginToolsMcpHandlers path with a plugin tool named exec: listTools will include it and callTool will dispatch it because current main has no deny-set check there; I did not run tests because this review is read-only.

## Solution Assessment

No, not as currently shaped. The MCP deny-list direction is the right narrow fix, but the current branch needs a rebase into the active handler split and should drop or split the axios release-age policy change before it is the best maintainable solution.
