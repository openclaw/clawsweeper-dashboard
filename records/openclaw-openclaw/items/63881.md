## Summary

Keep open: the MCP hardening is still relevant, but this branch is stale against the active MCP handler path and still carries a supply-chain policy change that should not be coupled to the MCP fix.

## What This Changes

The PR splits dangerous-tool deny constants, applies a universal deny list plus `gateway.tools` overrides to the plugin-tools MCP server, adds MCP boundary tests, and adds an `axios` minimum-release-age exclusion.

## Best Possible Solution

Land a narrow MCP hardening change in `createPluginToolsMcpHandlers` or the shared MCP server layer, document `gateway.tools` MCP semantics, add handler-level tests, and leave axios release-age policy to separate dependency work.

## Reproduction Assessment

Yes. A source-level reproduction is to instantiate `createPluginToolsMcpHandlers` on current main with an `exec` tool: `listTools` includes it and `callTool` dispatches it because the handler only filters `ownerOnly` tools and wraps `before_tool_call`.

## Solution Assessment

No. The universal-versus-HTTP-only split is a useful direction, but this branch applies the policy to stale server code and mixes in dependency policy; a rebased handler-level fix is the safer solution.
