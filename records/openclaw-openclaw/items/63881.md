## Summary

Keep open. The security goal is valid and current main still lacks a dangerous-tool deny boundary on the active plugin-tools MCP handler, but this branch is stale/conflicting, applies policy to the old monolithic server shape, mixes in dependency-policy churn, and lacks external real MCP behavior proof.

## What This Changes

The PR splits gateway dangerous-tool deny constants, filters plugin-tools MCP tools with a universal deny list and `gateway.tools` overrides, adds MCP boundary tests, and adds an axios release-age exclusion.

## Best Possible Solution

Rebase onto current main, enforce the universal dangerous-tool deny list at the active MCP list and call boundaries while preserving `ownerOnly` and `before_tool_call` behavior, document/test MCP `gateway.tools` semantics, and keep axios policy in separate dependency work.

## Reproduction Assessment

Yes, source-level. On current main, constructing `createPluginToolsMcpHandlers` with an `exec` tool would list and dispatch it because the active handler only filters `ownerOnly` tools and wraps `before_tool_call`; I did not execute tests in this read-only review.

## Solution Assessment

No. Splitting universal versus HTTP-only denial is a useful direction, but this branch applies it to a stale server shape and bundles unrelated dependency policy; the maintainable fix belongs on the current shared handler/server path.
