## Summary

Keep open for human/security review. The MCP deny-list gap is source-reproducible on current main, but this branch is stale/conflicting and adds the boundary in the old monolithic server body instead of the current shared MCP handler path; contributor real-behavior proof is also still missing.

## What This Changes

The branch filters plugin-tools MCP exposure through a universal dangerous-tool deny list with gateway.tools overrides, splits HTTP-only from universal deny constants, adds MCP tests, and adds an axios release-age exclusion.

## Best Possible Solution

Rebase and enforce the universal dangerous-tool deny policy at the active createPluginToolsMcpHandlers/createToolsMcpServer boundary, keep HTTP-only exceptions separate, document MCP gateway.tools semantics, and land only with real MCP proof plus security review.

## Reproduction Assessment

Yes, source-level. On current main, createPluginToolsMcpHandlers filters ownerOnly tools but otherwise lists and dispatches all remaining tools, so a non-ownerOnly exec-style tool supplied to the MCP bridge would be exposed; no live MCP run was performed in this read-only review.

## Solution Assessment

No. Splitting universal versus HTTP-only denial is a plausible direction, but this patch targets a stale server shape, needs to move the policy to the active shared handler boundary, should document the expanded config semantics, and should keep dependency policy separate.
