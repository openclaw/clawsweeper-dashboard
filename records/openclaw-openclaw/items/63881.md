## Summary

Keep open. The MCP deny-list direction is valid, but current main has moved MCP list/call behavior into shared handlers, while this branch still implements the policy in the older monolithic server shape and carries an unrelated dependency-policy change.

## What This Changes

The PR splits the gateway dangerous-tool deny list, applies the universal subset plus gateway.tools overrides to the plugin-tools MCP server, adds MCP boundary tests, and adds an axios minimumReleaseAge exclusion.

## Best Possible Solution

Land a narrow MCP hardening change in the active createPluginToolsMcpHandlers/createToolsMcpServer path, preserving ownerOnly and before_tool_call behavior, documenting gateway.tools MCP semantics, adding handler-level tests, and leaving dependency policy changes to a separate dependency/security path.

## Reproduction Assessment

Yes. A source-level reproduction is to instantiate createPluginToolsMcpHandlers with a tool named exec: current main listTools includes it and callTool dispatches it because there is no deny-set check; I did not run tests because this review was constrained to read-only inspection and pnpm is unavailable.

## Solution Assessment

No. The universal-versus-HTTP-only split is the right direction, but this branch applies it to the stale server shape and mixes in dependency policy, so the maintainable fix is a rebased or replacement handler-level PR.
