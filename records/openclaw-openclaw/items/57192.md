## Summary

Keep open. This PR is a plausible path for the linked macOS allowlist problem, but it is incomplete: it default-allows `screen.record` in one policy table while setup deny-list seeding, dangerous-command audit/docs/tests, and the current security contract still treat it as explicit opt-in, and the PR still lacks real macOS behavior proof.

## What This Changes

The PR adds `screen.record` to macOS node command defaults with a gateway regression test and also omits empty `tools` arrays from an older OpenAI WebSocket warm-up path/tests.

## Best Possible Solution

Make one maintainer-approved policy update: either default-allow macOS `screen.record` across gateway policy, setup defaults, audit/docs/tests, and real macOS proof, or keep explicit opt-in and clarify the capability docs.

## Reproduction Assessment

Yes, from source inspection rather than a live macOS run. macOS nodes declare and implement `screen.record`, current main excludes it from the default macOS allowlist, and `node.invoke` emits the reported allowlist error when the declared command is not allowlisted.

## Solution Assessment

No. Adding only a macOS platform default is incomplete because setup-seeded `denyCommands`, dangerous-command audit behavior, docs, and tests still preserve explicit opt-in semantics.
