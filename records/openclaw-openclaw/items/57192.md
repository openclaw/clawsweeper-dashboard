## Summary

Keep open. This PR addresses a source-reproducible macOS screen-recording allowlist problem, but the proposed policy change is incomplete and privacy-sensitive: `screen.record` is still classified as dangerous, setup seeds it into `denyCommands`, docs describe explicit opt-in, and the PR lacks after-fix macOS behavior proof.

## What This Changes

The PR adds `screen.record` to macOS platform defaults with a gateway regression test and separately omits empty `tools` arrays from OpenAI WebSocket warm-up events/tests.

## Best Possible Solution

Make one maintainer-approved policy change: either default-allow macOS `screen.record` across gateway policy, setup defaults, audit checks, docs, tests, and real macOS proof, or keep explicit opt-in and clarify the macOS capability messaging.

## Reproduction Assessment

Yes, from source inspection rather than a live macOS run. Current main has macOS nodes declare `screen.record`, keeps it out of the default allowlist, and emits the reported allowlist error when the declared command is not allowed.

## Solution Assessment

No. Adding `screen.record` only to macOS platform defaults is not the complete maintainable fix because the dangerous-command contract, wizard deny-list, audit behavior, docs, and tests still preserve explicit opt-in semantics.
