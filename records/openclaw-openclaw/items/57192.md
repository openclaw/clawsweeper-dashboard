## Summary

Keep open. The linked behavior is source-reproducible, but this PR is incomplete because it default-allows a privacy-sensitive macOS screen-recording command without updating the dangerous-command, setup denylist, audit, docs, and test contract that still treats it as explicit opt-in.

## What This Changes

The PR branch adds `screen.record` to macOS node platform defaults with a regression test and also changes OpenAI WebSocket warm-up handling to omit `tools` when the array is empty.

## Best Possible Solution

Make one maintainer-approved policy change that either default-allows macOS `screen.record` across gateway policy, setup defaults, audit checks, docs, and tests, or keeps it explicit opt-in and clarifies the macOS capability messaging.

## Reproduction Assessment

Yes. Source inspection on current main shows macOS nodes declare `screen.record`, the gateway default allowlist excludes it, and `node.invoke` rejects declared commands that are not allowlisted; I did not run a live macOS recording in this read-only sweep.

## Solution Assessment

No. Adding `screen.record` only to macOS platform defaults is not the best complete fix because setup deny-lists, audit behavior, docs, tests, and the dangerous-command contract still preserve explicit opt-in semantics.
