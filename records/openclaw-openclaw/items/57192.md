## Summary

Keep open. The PR addresses a real source-reproducible macOS screen-recording mismatch, but the patch is incomplete because it changes only the platform default while setup deny-lists, dangerous-command audit behavior, docs, and current tests still preserve explicit opt-in semantics.

## What This Changes

The PR adds `screen.record` to macOS platform defaults with a gateway test and also includes an unrelated OpenAI WebSocket warm-up change to omit empty `tools` arrays.

## Best Possible Solution

Make one maintainer-approved policy change that either default-allows macOS `screen.record` across gateway policy, setup defaults, audit checks, docs, and tests, or keeps it explicit opt-in and clarifies the macOS capability messaging.

## Reproduction Assessment

Yes. Source inspection on current main shows macOS nodes declare and implement `screen.record`, the default gateway allowlist excludes it, and `node.invoke` rejects declared commands that are not allowlisted; I did not run a live macOS recording in this read-only review.

## Solution Assessment

No. Adding `screen.record` only to macOS platform defaults is not the complete fix because setup deny-lists, audit behavior, docs, tests, and the dangerous-command contract still preserve explicit opt-in semantics.

## Review Findings
