## Summary

Keep open: this PR is a plausible fix path for the linked macOS screen-recording report, but the patch is incomplete because current policy, setup defaults, audit/docs/tests, and security posture still treat `screen.record` as explicit opt-in, and the external PR lacks real macOS after-fix proof.

## What This Changes

The PR adds `screen.record` to macOS node command defaults with a gateway regression test and also changes an older OpenAI WebSocket warm-up path/tests to omit empty `tools` arrays.

## Best Possible Solution

Make one maintainer-approved policy update: either default-allow macOS `screen.record` across gateway policy, setup defaults, audit/docs/tests, and real macOS proof, or keep explicit opt-in and clarify capability docs.

## Reproduction Assessment

Yes, from source inspection rather than a live macOS run. macOS nodes declare and implement `screen.record`, current gateway policy excludes it from macOS defaults unless explicitly allowed, and `node.invoke` returns the reported allowlist error when policy blocks the command.

## Solution Assessment

No. Adding only a macOS platform default is incomplete because setup-seeded `denyCommands`, dangerous-command audit behavior, docs, and tests still preserve explicit opt-in semantics.
