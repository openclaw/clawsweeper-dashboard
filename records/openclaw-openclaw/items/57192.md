## Summary

Keep open: the linked macOS screen-recording bug is real, but this PR only changes the platform default and leaves the existing deny-list, audit, setup, and docs policy treating `screen.record` as explicit opt-in.

## What This Changes

This PR adds `screen.record` to macOS node command defaults with a gateway regression test and also includes an unrelated OpenAI WebSocket warm-up empty-tools cleanup.

## Best Possible Solution

Land a focused gateway/macOS policy change only after deciding the default screen-recording policy, updating the allowlist, dangerous-command classification, setup wizard defaults, audit behavior, docs, and regression tests together.

## Reproduction Assessment

Yes. Current main source and tests show `resolveNodeCommandAllowlist({}, { platform: "macOS 26.4.0", deviceFamily: "Mac" })` excludes `screen.record`, and wizard-generated configs would also deny it through `DEFAULT_DANGEROUS_NODE_COMMANDS`.

## Solution Assessment

No. Adding `screen.record` only to macOS defaults is not the narrowest maintainable fix because deny-list setup, audit, docs, and tests still define it as a dangerous explicit opt-in command.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep screen recording policy consistent:** `src/gateway/node-command-policy.ts:25`
  - body: This adds `screen.record` to macOS platform defaults, but the command still flows through `DEFAULT_DANGEROUS_NODE_COMMANDS`.
