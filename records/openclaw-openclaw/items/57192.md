## Summary

Keep open. The linked bug is still valid on current main, but this PR needs revision before merge because current policy, tests, docs, wizard defaults, and security audit behavior still deliberately gate `screen.record` rather than treating it as a macOS default.

## What This Changes

The PR adds `screen.record` to macOS platform defaults with a gateway allowlist regression test, and also changes OpenAI WebSocket warm-up payload construction/tests to omit empty `tools` arrays.

## Best Possible Solution

Choose one product/security policy for macOS screen recording and make the implementation consistent across gateway allowlist resolution, setup defaults, audit checks, docs, and regression tests. If the command stays opt-in, the docs/capability messaging should make that boundary clear despite the macOS node declaring the command.

## Reproduction Assessment

Yes. Current main gives a high-confidence static reproduction path: `resolveNodeCommandAllowlist({}, { platform: "macOS 26.4.0", deviceFamily: "Mac" })` uses the macOS defaults, which include `screen.snapshot` but not `screen.record`; wizard configs would also delete `screen.record` through `denyCommands`.

## Solution Assessment

No. Adding a macOS default entry alone is not the best fix because `denyCommands` and the security policy still classify `screen.record` as dangerous; the safer fix is a focused policy-consistent change, with the unrelated OpenAI WebSocket hunks removed or split.
