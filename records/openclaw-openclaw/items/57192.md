## Summary

Keep open. The PR is a plausible implementation candidate for the linked macOS node recording bug, but the current diff is not mergeable as-is because it default-allows a privacy-sensitive command while the wizard deny-list, audit policy, docs, and current tests still treat `screen.record` as explicit opt-in.

## What This Changes

This PR adds `screen.record` to macOS node command defaults, adds a gateway regression test, and includes an unrelated OpenAI WebSocket warm-up empty-tools cleanup.

## Best Possible Solution

Make one focused maintainer-approved policy change: either default-allow macOS `screen.record` and update dangerous classification, wizard defaults, audit messaging, docs, and tests together, or keep explicit opt-in and align the user-facing docs/CLI guidance around that contract.

## Reproduction Assessment

Yes. Current main source and tests give a high-confidence source reproduction: macOS defaults allow `screen.snapshot` but exclude `screen.record`, and wizard-created configs would deny `screen.record` through `DEFAULT_DANGEROUS_NODE_COMMANDS`; I did not run a live macOS node recording in this read-only pass.

## Solution Assessment

No. Adding `screen.record` only to macOS platform defaults is not the best fix because the setup wizard, audit checks, docs, and current tests still define the command as dangerous explicit opt-in.
