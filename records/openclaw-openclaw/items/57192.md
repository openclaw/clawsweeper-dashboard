## Summary

Keep open. The branch targets a real macOS node policy mismatch, but the proposed fix is incomplete because it default-allows `screen.record` while the same command remains in the dangerous deny-list path, and the PR still lacks after-fix real macOS recording proof.

## What This Changes

The PR adds `screen.record` to macOS gateway platform defaults, adds a gateway regression test for `macOS 26.4.0`, and includes an unrelated OpenAI WebSocket warm-up empty-tools cleanup.

## Best Possible Solution

Make one maintainer-approved policy change: either default-allow macOS `screen.record` and update dangerous classification, wizard defaults, audit behavior, docs, and tests together, or keep explicit opt-in and clarify the capability docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: macOS nodes declare and handle `screen.record`, current main excludes it from macOS defaults, and configured `denyCommands` remove commands after defaults; I did not run a live macOS recording in this read-only review.

## Solution Assessment

No. Adding `screen.record` only to macOS platform defaults is incomplete because wizard deny-lists, dangerous-command audit policy, docs, and tests still treat it as explicit opt-in.
