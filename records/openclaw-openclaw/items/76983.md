## Summary

Keep open: current main does not implement message editing, and the PR is useful but not mergeable because the latest diff still has compile, event-scoping, idempotency, rerun, response-id, docs, and changelog blockers.

## What This Changes

The PR adds a `chat.edit_user_message` Gateway write RPC with transcript branch rewinding, protocol and Swift schema updates, a gateway regression test, and a Control UI controller helper.

## Best Possible Solution

Land the edit-message feature only after the RPC compiles, scoped edit notifications match existing session subscription boundaries, retry/rerun semantics are explicit, docs and changelog are updated, and a Gateway owner approves the new public surface.

## Reproduction Assessment

Yes for the PR defects from source inspection: the latest diff calls an unimported `SessionManager`, broadly broadcasts `session.message`, discards the appended active id, and accepts idempotency/rerun fields without honoring them. I did not run tests because this review is read-only.

## Solution Assessment

No. Branching at the edited message's parent is consistent with the `SessionManager` contract, but the current patch is not the narrowest maintainable solution until its event delivery and API semantics are fixed and documented.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Import SessionManager before using it:** `src/gateway/server-methods/chat.ts:1773`
  - body: The new edit helper calls `SessionManager.open(...)`, but production `chat.ts` still imports only `CURRENT_SESSION_VERSION` from `@mariozechner/pi-coding-agent`. `tsgo` will fail before this handler can build.
  - confidence: 0.96
- **[P2] Scope edit events to session subscribers:** `src/gateway/server-methods/chat.ts:1842-1848`
  - body: `context.broadcast("session.message", ...)` sends the edit metadata to every read/write-scope operator client. Existing transcript `session.message` delivery is targeted through session-event or `sessions.messages.subscribe` recipients, so this bypasses the session subscription boundary.
  - confidence: 0.91
- **[P2] Return the active edited entry id:** `src/gateway/server-methods/chat.ts:1836-1838`
  - body: After branching, `appendMessage` creates a new active transcript entry, but the patch ignores that id and reports the original message id. The original entry is no longer on the active branch, so clients cannot reconcile the response with `chat.history` or use it for follow-up edits.
  - confidence: 0.9
- **[P2] Honor edit idempotency keys:** `src/gateway/server-methods/chat.ts:2054-2062`
  - body: The schema requires `idempotencyKey`, but the handler never checks or records it before mutating the transcript. A retry after a lost response can re-enter the edit path against a different active branch instead of replaying the first result.
  - confidence: 0.9
- **[P2] Handle or remove rerun requests:** `src/gateway/protocol/schema/logs-chat.ts:61`
  - body: The new schema accepts `rerun`, but the handler never reads it and always returns `rerunStarted: false`. Either implement and test rerun behavior or leave this request field out until the API supports it.
  - confidence: 0.86
- **[P3] Document the new chat edit RPC:** `src/gateway/server-methods-list.ts:157`
  - body: This adds a public Gateway write method and user-facing chat behavior, but the PR does not update protocol docs or the active changelog. Add the release note and request/response contract before merge.
  - confidence: 0.88
