## Summary

Keep open: current main does not implement this feature, and the PR still needs correctness fixes before merge because the edit RPC drops Gateway idempotency, returns an inactive message id after branching, accepts a no-op rerun request, and lacks the required changelog entry.

## What This Changes

The PR adds a `chat.edit_user_message` Gateway RPC with protocol/Swift model registration, transcript branch rewind logic, a gateway regression test, and a Control UI controller helper.

## Best Possible Solution

Land the edit-message feature only after the RPC has stable idempotent retry behavior, returns active-branch identifiers, resolves or removes rerun semantics, includes release-note coverage, and receives maintainer approval for the new protocol surface.

## Reproduction Assessment

Yes for the PR defects from source inspection: the patch validates fields that the handler ignores and discards the newly appended branch id. I did not execute tests because the review was read-only.

## Solution Assessment

No. Branching at the edited message parent is a plausible implementation direction, but the current PR is not the best fix until its retry, identifier, and optional rerun semantics are made explicit and tested.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Return the active edited entry id:** `src/gateway/server-methods/chat.ts:1765-1789`
  - body: The edit path branches to the original message's parent and appends a replacement, but it drops the id returned by `appendMessage` and reports `params.messageId`. The old entry is no longer on the active branch, so clients cannot reconcile the response with `chat.history` or use the returned id for a follow-up edit.
  - confidence: 0.88
- **[P2] Honor edit idempotency keys:** `src/gateway/server-methods/chat.ts:1983-1991`
  - body: `chat.edit_user_message` requires `idempotencyKey`, but the handler never checks or records it before mutating the transcript. If a response is lost and the client retries, the first call has already moved the active branch, so the retry can fail or apply different side effects instead of replaying the original result.
  - confidence: 0.91
- **[P2] Handle or remove rerun requests:** `src/gateway/server-methods/chat.ts:1993-1999`
  - body: The schema accepts `rerun`, but the handler never reads `p.rerun` and always returns `rerunStarted: false`. A client can request a rerun and receive a successful no-op, so the PR should either implement and test rerun behavior or remove that API field until it is supported.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/gateway/protocol/schema/logs-chat.ts:54-62`
  - body: This adds a user-facing Gateway RPC and Control UI chat capability, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-facing feature changes to include an active-version changelog entry before merge.
  - confidence: 0.9
