## Summary

Keep this PR open: current main does not already implement the edit-message feature, and the latest PR diff still has concrete compile, protocol, event-scoping, retry, rerun, docs, and changelog blockers before maintainer protocol review.

## What This Changes

The PR adds a `chat.edit_user_message` Gateway RPC, protocol/schema and Swift model entries, a gateway regression test, and a Control UI controller helper for editing a user message and rewinding later active-branch turns.

## Best Possible Solution

Land the edit-message feature only after the RPC compiles, emits through the existing session-scoped event path, returns active-branch identifiers, defines idempotent retry and rerun behavior, and updates protocol docs plus changelog.

## Reproduction Assessment

Yes for the PR defects from source inspection: the diff references `SessionManager` without importing it, broadcasts `session.message` outside the existing subscriber path, and accepts retry/rerun fields without honoring them. I did not execute tests because the review is read-only and `pnpm` is unavailable here.

## Solution Assessment

No. Branching at the edited message's parent is a plausible implementation direction, but this PR is not the best mergeable solution until the public Gateway contract, event delivery, retry/rerun behavior, docs, and changelog are made explicit and tested.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Import SessionManager before using it:** `src/gateway/server-methods/chat.ts:1773`
  - body: The new edit handler calls `SessionManager.open(...)`, but the PR does not import `SessionManager` from `@mariozechner/pi-coding-agent`. This will fail typechecking before the new RPC can build.
  - confidence: 0.96
- **[P2] Scope edit events to session subscribers:** `src/gateway/server-methods/chat.ts:1842-1848`
  - body: `session.message` transcript updates are currently routed only to session-event or per-session message subscribers. Broadcasting the edit event with `context.broadcast` sends the session key and edited/deleted ids to every read-scope operator client, bypassing that subscription boundary.
  - confidence: 0.91
- **[P2] Return the active edited entry id:** `src/gateway/server-methods/chat.ts:1836-1838`
  - body: After branching at the parent, `appendMessage` creates a new active transcript entry, but the patch discards that new id and reports the old `params.messageId`. Clients cannot reconcile the response with `chat.history` or use the returned id for a later edit.
  - confidence: 0.88
- **[P2] Honor edit idempotency keys:** `src/gateway/server-methods/chat.ts:2054-2062`
  - body: The schema requires `idempotencyKey`, but the handler never checks or records it before mutating the transcript. A retry after a lost response can fail or produce different side effects instead of replaying the original result.
  - confidence: 0.9
- **[P2] Handle or remove rerun requests:** `src/gateway/protocol/schema/logs-chat.ts:61`
  - body: The schema accepts `rerun`, but the handler never reads it and always returns `rerunStarted: false`. A client can request a rerun and get a successful no-op, so the PR should implement and test rerun behavior or leave the field out for now.
  - confidence: 0.86
- **[P3] Document the new chat edit RPC:** `src/gateway/protocol/schema/logs-chat.ts:55-65`
  - body: This adds a public Gateway write RPC and Control UI-facing capability without updating the protocol docs or active changelog. Add the release note and request/response contract before merge.
  - confidence: 0.87
