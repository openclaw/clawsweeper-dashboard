## Summary

Keep open: current main does not implement this feature, and the PR introduces a new chat-edit protocol surface with blocking API correctness gaps around idempotency and rerun behavior.

## What This Changes

The branch adds `chat.edit_user_message` protocol schemas and Swift models, gateway registration and handling, a gateway regression test, and a Control UI controller helper.

## Best Possible Solution

Define the edit-message API contract first, then land the gateway, client, UI, docs, changelog, and regression coverage with idempotent retries and explicit rerun semantics.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the handler requires but ignores `idempotencyKey`, appends a new branch entry, and never reads `rerun`. I did not execute the PR because this review was required to stay read-only.

## Solution Assessment

No: the append-only branch direction matches the Pi transcript model, but accepting a side-effecting RPC without honoring idempotency and silently ignoring `rerun` is not a maintainable protocol contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor the edit idempotency key before mutating:** `src/gateway/server-methods/chat.ts:1978-1985`
  - body: The new RPC requires `idempotencyKey`, but the handler never reads it before branching and appending the edited message. Because the edit removes the original message ID from the active branch, a normal client retry after a lost response can now fail with `Message not found` instead of returning the same successful result. Please persist or detect completed edit keys and return the original response without applying a second mutation.
  - confidence: 0.9
- **[P2] Do not silently ignore rerun requests:** `src/gateway/server-methods/chat.ts:1993`
  - body: The schema exposes `rerun`, but the server never checks `p.rerun` and always reports `rerunStarted: false`. A client asking to edit and rerun would receive a successful response even though no rerun was started, so either implement the rerun path or reject/remove the option until it is supported.
  - confidence: 0.86
