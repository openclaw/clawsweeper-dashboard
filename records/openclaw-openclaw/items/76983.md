## Summary

Keep open: the feature is useful, but the current PR is not mergeable because the latest diff has build-breaking, protocol-contract, event-scoping, and documentation/changelog blockers.

## What This Changes

The PR adds a `chat.edit_user_message` Gateway RPC, transcript branch rewind logic, generated Swift parameter models, a Gateway test, and a Control UI controller helper.

## Best Possible Solution

Land the feature only after the RPC compiles, scopes edit notifications through the existing session-message subscription model, returns stable active-branch identifiers, honors idempotent retries, defines or removes rerun semantics, and updates protocol docs plus the changelog.

## Reproduction Assessment

Yes for the PR defects from source inspection: the latest diff references an unimported `SessionManager`, broadly broadcasts `session.message`, returns the inactive original id, and accepts idempotency/rerun fields without implementing those contracts. I did not execute tests because this is a read-only review.

## Solution Assessment

No. Branching at the edited message's parent is a plausible implementation direction, but the current patch is not the narrowest maintainable solution until its compile, event delivery, retry, identifier, rerun, docs, and changelog contracts are made explicit and tested.

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Import SessionManager before using it:** `src/gateway/server-methods/chat.ts:1773`
  - body: The new edit handler calls `SessionManager.open(...)`, but this file still only imports `CURRENT_SESSION_VERSION` from `@mariozechner/pi-coding-agent`. `tsgo` will fail before the feature can build.
  - confidence: 0.96
- **[P2] Scope edit events to session subscribers:** `src/gateway/server-methods/chat.ts:1842-1848`
  - body: `session.message` is normally delivered through the transcript event path to session-event or `sessions.messages.subscribe` subscribers. Broadcasting this edit event with `context.broadcast` sends session keys and edited/deleted message ids to every read-scope connection, bypassing the existing subscription boundary.
  - confidence: 0.9
- **[P2] Return the new active message id:** `src/gateway/server-methods/chat.ts:1836-1838`
  - body: After branching, `appendMessage` creates a new active transcript entry, but the patch discards that id and reports the original `params.messageId`. The original entry is no longer on the active branch, so clients cannot reconcile the response with `chat.history` or use it for a follow-up edit.
  - confidence: 0.9
- **[P2] Honor edit idempotency keys:** `src/gateway/server-methods/chat.ts:2054-2062`
  - body: The schema requires `idempotencyKey`, but the handler never checks or records it before mutating the transcript. If the client retries after a lost response, the first call may already have moved the active branch, so the retry can fail or produce different side effects instead of replaying the original result.
  - confidence: 0.91
- **[P2] Handle or remove rerun requests:** `src/gateway/protocol/schema/logs-chat.ts:60`
  - body: The new schema accepts `rerun`, but the handler never reads it and always returns `rerunStarted: false`. A client can request a rerun and receive a successful no-op, so the PR should either implement and test rerun behavior or leave the field out until it is supported.
  - confidence: 0.86
- **[P3] Document the new chat edit RPC:** `src/gateway/protocol/schema/logs-chat.ts:55-65`
  - body: This adds a public Gateway write RPC and Control UI-facing capability, but the PR does not update the active changelog or protocol docs. Add the user-facing release note and document the request/response contract before merge.
  - confidence: 0.88
