## Label Justifications

- `P1`: The issue reports a real Feishu channel workflow where OpenClaw logs final delivery success but the user-visible final answer is lost.
- `impact:message-loss`: The affected behavior silently replaces the agent's final reply with a Feishu fallback prompt from the user's perspective.

## Summary

Keep open: current main still routes default Feishu final replies through the streaming CardKit path and treats accepted CardKit calls as visible delivery, while the reported fallback prompt is not detected or retried as plain text.

## What This Changes

Ensure Feishu final replies reported as dispatched are visible as the agent answer instead of a CardKit fallback/upgrade prompt.

## Best Possible Solution

Fix the Feishu plugin's streaming-card delivery path so CardKit updates follow the API contract and accepted-but-invisible card output falls back to a visible plain-text final reply, with docs for the existing `streaming: false` or `renderMode: raw` workaround.

## Reproduction Assessment

Yes for source-level reproduction, but not by a fresh live Feishu run in this review. The reporter provides live evidence, and current main still sends default final replies through CardKit streaming while treating HTTP OK as visible content.

## Solution Assessment

No, current main is not the best solution yet. The maintainable fix should live in the Feishu plugin's CardKit delivery path and should address the API update contract plus compatibility fallback behavior rather than only documenting the workaround.
