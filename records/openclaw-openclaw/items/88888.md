## Summary

Keep open: current main still treats default Feishu final replies as streaming-card delivery when streaming is enabled, and the plain post send path still has no transient retry or user-visible recovery when Feishu returns a send failure. The related docs PR documents a workaround, but it does not close the delivery-integrity bug.

## What This Changes

The issue reports that Feishu final replies can be hidden when default CardKit streaming renders only the client fallback prompt, or when a plain post final reply fails with a transient Feishu send error and is not recovered.

## Best Possible Solution

Keep this open for a Feishu-channel fix that guarantees final-answer visibility, likely combining explicit final-delivery semantics with transient post retries and a maintainer-approved CardKit fallback strategy.

## Reproduction Assessment

Yes for a source reproduction: current main still routes default Feishu final replies through CardKit streaming, and the plain post send path has no transient retry or recovery. No live Feishu tenant reproduction was run in this read-only pass.

## Solution Assessment

Unclear: the requested direction is valid, but the best fix needs a maintainer decision on whether to change default final rendering, detect CardKit fallback after acceptance, retry final posts, or record/report delivery failure state.
