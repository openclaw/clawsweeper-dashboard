## Summary

One correctness edge remains: the new hash is always non-empty, so calls with no session-owner fields never reach the advertised workspace fallback and instead share one global Claude queue across workspaces. The owner-key direction is sound, but that regression, the dirty branch, and missing direct proof block merge. Likely related people: @vincentkoc for the live-session owner boundary and @steipete for the original Claude CLI queueing path.

## What This Changes

The PR adds a shared Claude owner-key helper, uses it to queue fresh Claude CLI runs by session owner instead of workspace, and reuses it for the live-session map key.

## Best Possible Solution

Make owner-key construction return no key unless a real session-owner discriminator is present, retain workspace fallback otherwise, rebase the focused changes onto current main, and prove both independent-session concurrency and same-session serialization.

## Reproduction Assessment

Yes at source level. Current main and v2026.6.5 deterministically collapse fresh Claude CLI sessions in one workspace onto one strict queue key, and the linked issue provides observed serial timing.

## Solution Assessment

No, not yet. Reusing the live-session owner boundary is the right direction, but the helper must preserve workspace fallback when no meaningful owner identity exists and the rebased implementation needs direct runtime proof.
