## Summary

Keep open: the PR targets a real Telegram reasoning-preview orphan path, but the current patch still does not preserve or delete the prior reasoning preview id before starting the next split preview, and the supplied proof is mock-only rather than live Telegram behavior.

## What This Changes

The PR changes Telegram reasoning split handling to call `rotateLaneForNewMessage(reasoningLane)` and adds a regression test for `stop()` before `forceNewMessage()` across two reasoning segments.

## Best Possible Solution

Delete or otherwise retain the previous reasoning preview id before starting a new split reasoning preview, cover that deletion with a production-faithful regression test, then attach live Telegram proof or get an explicit maintainer override.

## Reproduction Assessment

Yes from source and reporter logs: current main can set `splitReasoningOnNextStream`, rotate the reasoning stream with `forceNewMessage()`, and lose the prior preview id before final cleanup. I did not run a live Telegram DM reproduction in this read-only review.

## Solution Assessment

No. Reusing the answer-lane helper gives stop-before-rotate ordering, but it still calls `forceNewMessage()` before the old reasoning preview is deleted or retained for later cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Delete the previous reasoning preview before rotating:** `extensions/telegram/src/bot-message-dispatch.ts:1441`
  - body: `rotateLaneForNewMessage` only calls `stop()` and then `forceNewMessage()`, whose production implementation clears `streamMessageId`. Final cleanup later calls `clear()` on the new stream state, so the first split reasoning preview id is still lost and can remain visible. The split path needs to delete the prior reasoning preview or keep its id for cleanup before starting the next preview.
  - confidence: 0.88
- **[P3] Use an id-clearing stream mock for this regression:** `extensions/telegram/src/bot-message-dispatch.test.ts:1398-1401`
  - body: The new test uses `setupDraftStreams`, which creates `createTestDraftStream({ messageId })`; that mock does not clear the message id on `forceNewMessage()` unless `clearMessageIdOnForceNew` is set. This means the test only proves call order and would pass without guarding the production id-loss behavior reported here.
  - confidence: 0.86
