## Summary

Keep open: the PR targets a real Telegram reasoning-stream bug, but the proposed fix still loses the prior reasoning preview id before any delete path can reach it, and the contributor proof is mock-only for visible Telegram streaming behavior.

## What This Changes

The PR replaces the reasoning split path's direct `forceNewMessage()`/lane reset with `rotateLaneForNewMessage()` and adds a regression test asserting `stop()` is called before `forceNewMessage()`.

## Best Possible Solution

Delete or otherwise register the prior reasoning preview for cleanup before resetting the reasoning lane, then cover the split-reasoning path with focused regression coverage plus real Telegram proof.

## Reproduction Assessment

Yes, from source inspection: a reasoning stream followed by `onReasoningEnd` and another reasoning stream can reset the stream state before cleanup can delete the prior preview id. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Reusing the answer-lane rotation helper finalizes and retains a message, but split reasoning previews need the previous preview deleted or explicitly tracked for cleanup before the stream state is reset.
