## Summary

Keep open: the PR identifies a real current-main Telegram reasoning-preview lifecycle bug, but the proposed helper reuse still finalizes rather than deletes the first reasoning preview before `forceNewMessage()` drops its id, so it likely leaves the stale preview in place. The supplied proof is also mock-only for a Telegram-visible behavior change.

## What This Changes

The branch replaces the Telegram reasoning split path's direct draft-stream rotation with `await rotateLaneForNewMessage(reasoningLane)` and adds a regression test asserting `stop()` runs before `forceNewMessage()`.

## Best Possible Solution

Use reasoning-specific rotation that deletes or otherwise preserves the old reasoning preview id for superseded cleanup before starting the next reasoning segment, then prove the behavior with focused regression coverage and live Telegram evidence.

## Reproduction Assessment

Yes from source and the linked report: current `main` rotates the reasoning draft stream with `forceNewMessage()` before cleanup, matching repeated reasoning `sendMessage` calls without edit/delete. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Reusing the answer-lane `stop()` rotation helper is not the best fix for reasoning previews because reasoning previews must be deleted or tracked for deletion, while answer-lane finalized messages are intended to remain visible.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Delete the old reasoning preview before rotating:** `extensions/telegram/src/bot-message-dispatch.ts:1441`
  - body: This uses the answer-lane rotation helper for reasoning splits, but that helper calls `stop()` and then `forceNewMessage()`. `stop()` only finalizes/flushes the draft, while production `forceNewMessage()` clears `streamMessageId`, so final cleanup can only clear the second segment's preview. A multi-segment reasoning turn still leaves the first preview in chat, which is the bug this PR is meant to fix; use reasoning-specific cleanup that deletes or tracks the old preview before starting the next message.
  - confidence: 0.91
