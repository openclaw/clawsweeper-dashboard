## Summary

Keep open: current main still turns a successful Telegram threadless retry into the same cron `delivered` state as a correctly routed topic delivery. The open sibling at https://github.com/openclaw/openclaw/issues/79408 and https://github.com/openclaw/openclaw/pull/79410 cover forum retry parity, not the cron status signal for this DM fallback case.

## What This Changes

Report Telegram DM topic threadless fallback separately from clean cron delivery instead of persisting it as ordinary `lastDeliveryStatus: "delivered"`.

## Best Possible Solution

Carry a structured Telegram threadless-fallback signal through outbound delivery results into cron state, run history, and CLI/UI display, with regression coverage, while preserving the existing content-delivery fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Telegram `to` value ending in `:topic:<id>` resolves a thread id, the Telegram DM retry removes that thread id on `message thread not found`, and cron persists the successful retry as ordinary `delivered`.

## Solution Assessment

Yes. Surfacing recoverable route fallback as delivery metadata/status is narrower than changing Telegram routing policy or disabling the fallback, which would risk losing content.
