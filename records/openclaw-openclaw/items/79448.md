## Summary

Close as implemented: current main no longer performs the Telegram DM-topic threadless retry that caused fallback-to-main sends to be persisted as ordinary `delivered`, and the fix shipped in `v2026.5.9-beta.1`.

## What This Changes

The issue asks Telegram cron delivery to stop recording a DM-topic threadless fallback as an indistinguishable successful delivery.

## Best Possible Solution

Keep the fail-closed Telegram implementation that preserves topic intent and surfaces Telegram rejection; revisit successful DM-topic delivery only if Telegram provides a reliable topic parameter contract.

## Reproduction Assessment

No current-main failure path remains for the reported silent fallback: source and tests show DM-topic thread rejection is now surfaced instead of retried as a plain DM. I did not run a live Telegram cron in this read-only review, but the merged PR included live before/after Telegram output.

## Solution Assessment

Yes. Failing the affected topic send loudly is narrower and safer than adding a new `delivered-fallback` cron state for a wrong-surface retry that current main no longer performs.
