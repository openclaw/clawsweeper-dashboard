## Summary

Current main already fixes the managed npm-root pruning behavior reported here: plugin installs now preserve the npm-root dependency manifest and install from that root, with regression coverage for keeping previously installed sibling plugins on disk. The fix is present after v2026.5.2 and is not yet contained in a release tag I could verify.

## What This Changes

The issue reports that sequentially installing @openclaw/feishu and @martian-engineering/lossless-claw on OpenClaw 2026.5.2 removes the previously installed plugin from the managed npm root.

## Best Possible Solution

Keep the current managed npm-root installer behavior and let the next release carry the fix; no new plugin-specific compatibility path is needed.

## Reproduction Assessment

Yes for the reported 2026.5.2 behavior: the old release used a single-spec npm install path and #76571 documents the same pruning failure. I did not run the mutating install commands in this read-only review; current main is source/test verified fixed.

## Solution Assessment

Yes. The current main fix is narrow and maintainable: keep the managed npm-root manifest authoritative, run npm install from that root, and cover sequential plugin installs without hard-coding Feishu or lossless-claw.
