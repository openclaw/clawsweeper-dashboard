## Summary

Keep open: current main still skips the configured-plugin install repair whenever the source config is invalid, even though the release repair helper and installer path exist for the missing external plugin case described here.

## What This Changes

The issue reports that upgrading from pre-externalization OpenClaw to 2026.5.12 can leave configured official plugins missing, while config validation blocks both startup and the doctor auto-install repair.

## Best Possible Solution

Keep this open and fix doctor to run the configured-plugin install repair for narrowly identified missing official external plugin validation failures, then revalidate without adding a broad invalid-config bypass.

## Reproduction Assessment

Yes, at source level: an invalid Brave/Slack-style missing external plugin config sets `sourceConfigValid` false, and the current doctor health contribution returns before the configured-plugin install repair can run. I did not execute the live package-upgrade scenario in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to relax or specialize the doctor gate for known missing official external plugin references and keep arbitrary invalid configs fail-closed. A broad `plugins install` bypass or registry-wide compatibility search would be larger follow-up work, not the core unblocker.
