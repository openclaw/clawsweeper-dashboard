## Summary

Keep open: current main still skips the configured-plugin install repair when the source config is invalid, and an open closing PR is already the right implementation path. The narrower catalog workaround in https://github.com/openclaw/openclaw/pull/82420 may help Brave/Slack installs, but it does not replace the doctor-gate fix tracked by https://github.com/openclaw/openclaw/pull/82361.

## What This Changes

The issue asks OpenClaw doctor to unblock stock-to-externalized plugin upgrades by repairing missing configured plugins even when the old config is invalid because those plugins are absent.

## Best Possible Solution

Land or replace the narrow doctor-gate fix in https://github.com/openclaw/openclaw/pull/82361, while treating https://github.com/openclaw/openclaw/pull/82420 as a short-term catalog workaround rather than the terminal fix.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: a missing installable Brave-style web search provider makes the source config invalid, and current doctor returns before the configured-plugin install repair can run. I did not execute the full live package-upgrade scenario in this read-only review.

## Solution Assessment

Yes, the linked doctor-gate direction is the best maintainable fix because it can stay scoped to repairable missing-plugin validation failures. A broad config-validation bypass or registry compatibility search would be larger follow-up work, not the core unblocker.
