## Summary

Keep open: current main still has the asymmetric MEDIA directive handling described in the report, and the open design follow-up tracks the broader channel.message contract rather than replacing this narrower duplicate-text bug.

## What This Changes

The issue reports duplicate visible text when a final reply containing text plus a MEDIA directive is split through channel.message/source-channel delivery.

## Best Possible Solution

Fix the shared reply payload/dedupe path so a text+MEDIA directive whose visible text was already sent delivers the attachment without reusing that text as a second caption, while leaving the broader channel.message payload contract to the linked design issue.

## Reproduction Assessment

Yes for source-level reproduction: current main still has block delivery that skips MEDIA extraction, final delivery that parses MEDIA, and outbound media delivery that uses remaining text as caption. I did not execute a live Bncr transport repro in this read-only review.

## Solution Assessment

Yes for the narrow bug: the maintainable fix belongs in shared reply payload construction or block/final dedupe before transport delivery, not in a Bncr-specific workaround. The broader choice to delegate logical text+media physical sending to plugins is separately tracked in https://github.com/openclaw/openclaw/issues/91867.
