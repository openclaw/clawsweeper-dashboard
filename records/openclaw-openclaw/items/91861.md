## Summary

Keep open: the checked-out main still has the asymmetric MEDIA directive handling described by the reporter, and the current tests cover nearby structured-media cases but not the exact source-channel/channel.message split that can re-use final text as a media caption.

## What This Changes

The issue asks OpenClaw to stop duplicating visible text when one final assistant reply contains text plus a legacy MEDIA directive and is split between block/intermediate delivery and final channel.message media delivery.

## Best Possible Solution

Fix core reply payload construction or block/final dedupe so a text+MEDIA directive whose visible text was already emitted delivers the attachment without reusing that text as a second caption, backed by an exact regression test.

## Reproduction Assessment

Yes for source-level reproduction: current main still has block delivery that does not extract MEDIA directives, final delivery that does, and outbound media captioning that reuses remaining text. I did not run a live Bncr transport repro in this read-only review.

## Solution Assessment

Yes: the best fix is in the shared reply payload/dedupe path before transport delivery, not in a Bncr-specific workaround or generic media caption removal. The repair should preserve normal captions while suppressing only the duplicate text from the same logical final reply.
