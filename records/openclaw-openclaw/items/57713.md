## Summary

Keep open: current `main` fixed the original default-image fallback, but the May 9 follow-up shows the same write/edit failure still occurs with custom slim sandbox images, and current source still surfaces the missing-Python helper failure without actionable image guidance. There are already open related PRs for this diagnostic surface, including a forked PR with closing syntax for this issue, so cleanup should wait for that work to land or be replaced.

## What This Changes

The issue requests that Docker sandbox write/edit either use a sandbox image containing Python for the mutation helper or fail with clear validation and recovery guidance when Python is missing.

## Best Possible Solution

Keep the shipped default-image guard, then land a current-main diagnostic fix that maps local and remote missing-Python mutation failures to actionable sandbox image guidance with focused tests and real Docker proof.

## Reproduction Assessment

Yes. Source-reproducible: a Docker sandbox image without `python3` or `python` reaches the pinned mutation helper and exits before write/edit can complete; the May 9 comment reports the same path with `node:22-bookworm-slim`.

## Solution Assessment

No. Closing as implemented is not the best outcome because current `main` fixed the default image path but not the custom/slim-image diagnostic path; the narrow maintainable fix is to land or replace the open guidance PR work.
