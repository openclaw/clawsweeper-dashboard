Close as implemented on main: current code no longer creates a default-tagged sandbox from plain Debian, the default sandbox Dockerfile installs python3, regression coverage locks the missing-image behavior, and the fix commit is contained in v2026.4.29.

## What This Changes

The issue asks OpenClaw to include python3 in the default Docker sandbox image or fail early with a clear diagnostic because Docker sandbox edit/write failed while read still worked.

## Best Possible Solution

Keep the shipped default-image contract: the default sandbox image is built from Dockerfile.sandbox with python3, and a missing default image fails early with recovery guidance rather than silently creating an incompatible Debian tag.

## Reproduction Assessment

Yes. The current regression test reproduces the missing-default-image path deterministically by mocking Docker image inspection and verifies the old pull/tag fallback no longer happens; the original live symptom is also explained by the mutation helper's python requirement.

## Solution Assessment

Yes. Failing fast with setup guidance is the narrow maintainable fix for the bad default-tag path, while Dockerfile.sandbox preserves the intended python3 dependency for edit/write helpers.
