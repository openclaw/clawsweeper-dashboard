## Summary

Keep open. The Feishu silent-token fix is a valid bug-fix direction, but this branch is not mergeable as-is: it imports the helper from the wrong SDK surface, misses media and streaming edge paths, lacks after-fix real behavior proof, and overlaps with but is not fully replaced by https://github.com/openclaw/openclaw/pull/56167 because that PR does not cover the reply dispatcher.

## What This Changes

The PR adds Feishu `NO_REPLY` suppression checks in outbound `sendText` and reply-dispatcher delivery before Feishu text API calls.

## Best Possible Solution

Land one Feishu-owned fix on current main that filters exact silent tokens at every Feishu text boundary, preserves media and streaming cleanup, adds focused outbound and dispatcher tests, and consolidates overlap with https://github.com/openclaw/openclaw/pull/56167.

## Reproduction Assessment

Yes, at source level. Current main still has Feishu outbound and dispatcher text paths that can call Feishu text send helpers without a Feishu-local exact silent-token guard, although I did not run a live Feishu API reproduction.

## Solution Assessment

No. The PR has the right direction, but the branch uses the wrong SDK import, suppresses only part of the Feishu surface, mishandles media plus silent text, and can bypass streaming cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use an exported SDK subpath:** `extensions/feishu/src/outbound.ts:3`
  - body: The root `openclaw/plugin-sdk` entrypoint on current main does not export `isSilentReplyText` or `SILENT_REPLY_TOKEN`; those helpers are available from a reply-runtime subpath. As written, the new value import prevents this Feishu fix from building reliably.
  - confidence: 0.93
- **[P1] Preserve streaming cleanup for silent finals:** `extensions/feishu/src/reply-dispatcher.ts:255-259`
  - body: In card streaming mode, `onReplyStart` can open a Feishu streaming card before the final payload arrives. Returning here for a text-only silent token skips the final-text/idle cleanup path, so an initial thinking card can be left behind instead of being closed cleanly.
  - confidence: 0.86
- **[P2] Suppress silent text while keeping media:** `extensions/feishu/src/reply-dispatcher.ts:255-259`
  - body: The `!hasMedia` condition means a payload with media plus `text === "NO_REPLY"` bypasses this guard and still enters the text-send path before media delivery. Treat the silent token as non-deliverable text while continuing to send the media.
  - confidence: 0.91
- **[P2] Guard the sendMedia text paths:** `extensions/feishu/src/outbound.ts:83-87`
  - body: Adding the guard only in `sendText` leaves `sendMedia` paths that call the text helper directly for captions, degraded voice fallback, failures, and no-media fallback. A direct `sendMedia` call with `text: "NO_REPLY"` can still send the token to Feishu.
  - confidence: 0.9
