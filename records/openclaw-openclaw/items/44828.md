## Summary

Keep open. The Feishu silent-token leak is still present on current main, but this PR is not landable as written because it imports helpers from the unsupported root SDK surface and only suppresses some text-only paths. The overlapping #56167 branch is a plausible consolidation path, so the next step is maintainer choice rather than closing this PR as implemented.

## What This Changes

This PR adds Feishu-local `NO_REPLY` checks that early-return in `feishuOutbound.sendText` and in the text-only reply-dispatcher delivery path before Feishu API sends.

## Best Possible Solution

Land one Feishu-owned fix that imports silent-reply helpers from a supported focused SDK subpath, suppresses exact silent tokens at every Feishu API boundary, preserves media delivery and streaming-card cleanup, and adds focused outbound plus reply-dispatcher regression tests. Maintainers should decide whether to fold the missing dispatcher work into #56167 or request a replacement/update for this PR.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists without live Feishu: direct `sendText({ text: "NO_REPLY" })`, `sendMedia({ text: "NO_REPLY", mediaUrl })`, and dispatcher `deliver({ text: "NO_REPLY", mediaUrl })` can still route text into Feishu send helpers on current main.

## Solution Assessment

No. The direction is valid, but this patch is not the best fix because it uses an unsupported root SDK import, misses `sendMedia` caption/fallback paths, suppresses only no-media dispatcher payloads, and can bypass streaming-card finalization behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Import silent helpers from a supported subpath:** `extensions/feishu/src/outbound.ts:3`
  - body: `openclaw/plugin-sdk` does not export `isSilentReplyText` or `SILENT_REPLY_TOKEN` on current main; the focused reply SDK subpaths do. This import pattern will fail when rebased, and the same issue applies to the new dispatcher import.
  - confidence: 0.92
- **[P1] Preserve streaming cleanup for silent finals:** `extensions/feishu/src/reply-dispatcher.ts:259-261`
  - body: In card streaming mode, `onReplyStart` may already create a streaming card before the final payload arrives. Returning here for text-only `NO_REPLY` skips the final-text path that updates the stream state, so idle close can leave the placeholder card content behind.
  - confidence: 0.86
- **[P2] Suppress silent text when media is present:** `extensions/feishu/src/reply-dispatcher.ts:259-261`
  - body: The `!hasMedia` condition means a payload with media plus `text === "NO_REPLY"` bypasses this guard and still enters the text-send block before media delivery. Treat the silent token as non-deliverable text while still allowing the media path to run.
  - confidence: 0.91
- **[P2] Guard sendMedia captions and fallback text:** `extensions/feishu/src/outbound.ts:83-87`
  - body: Adding the guard only in `sendText` leaves `sendMedia` paths that call `sendOutboundText` directly for captions and text-only fallback. A direct `sendMedia` call with `text: "NO_REPLY"` can still send the token to Feishu.
  - confidence: 0.89
- **[P3] Return the normal Feishu suppressed result shape:** `extensions/feishu/src/outbound.ts:85`
  - body: Normal Feishu sends return chat-scoped metadata such as `chatId`, but this suppressed result returns `channelId: to`. That can make downstream result formatting treat a suppressed Feishu send differently from successful Feishu sends.
  - confidence: 0.72
