## Summary

Keep open. Current main still has Feishu API-bound paths that can leak exact silent-reply text, but this PR is not landable as written: it imports helpers from the unsupported root SDK surface, leaves media and dispatcher mixed-payload cases uncovered, and can regress streaming-card cleanup. The overlapping #56167 branch is already the better consolidation lane for outbound work, with dispatcher behavior still needing maintainer direction.

## What This Changes

The PR adds Feishu-local exact `NO_REPLY` suppression in outbound `sendText` and the text-only reply-dispatcher path before Feishu API sends.

## Best Possible Solution

Land one Feishu-owned fix that imports silent-reply helpers from a focused SDK subpath, suppresses exact silent tokens at every Feishu text API boundary, preserves media delivery and streaming-card cleanup, adds focused outbound and reply-dispatcher regression tests, and includes the required user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists without live Feishu: direct `feishuOutbound.sendText({ text: "NO_REPLY" })`, direct `sendMedia({ text: "NO_REPLY", mediaUrl })`, and dispatcher delivery of `{ text: "NO_REPLY", mediaUrl }` exercise API-bound paths that current main or this PR still handles incorrectly.

## Solution Assessment

No. The direction is valid, but this patch is not the best implementation because it uses an unsupported SDK import, suppresses only part of the outbound surface, treats dispatcher media payloads incorrectly, and can skip streaming finalization behavior.

- **[P1] Import silent helpers from a supported subpath:** `extensions/feishu/src/outbound.ts:3`
  - body: `openclaw/plugin-sdk` does not export `isSilentReplyText` or `SILENT_REPLY_TOKEN` on current main; the focused reply SDK subpaths do. This import will fail type checking when rebased, and the same issue applies to the new dispatcher import.
  - confidence: 0.92
- **[P1] Preserve streaming cleanup for silent finals:** `extensions/feishu/src/reply-dispatcher.ts:259-261`
  - body: In card streaming mode, `onReplyStart` may already create a streaming card before the final payload arrives. Returning here for text-only `NO_REPLY` skips the final-text path that updates the stream state, so idle close can leave the placeholder card content behind.
  - confidence: 0.86
- **[P2] Suppress silent text when media is present:** `extensions/feishu/src/reply-dispatcher.ts:259-261`
  - body: The `!hasMedia` condition means a payload with media plus `text === "NO_REPLY"` bypasses this guard and still enters the text-send block before media delivery. Treat the silent token as non-deliverable text while still allowing the media path to run.
  - confidence: 0.91
- **[P2] Guard sendMedia captions and fallback text:** `extensions/feishu/src/outbound.ts:83-87`
  - body: Adding the guard only in `sendText` leaves `sendMedia` paths that call `sendOutboundText` directly for captions and text-only fallback. A direct `sendMedia` call with `text: "NO_REPLY"` can still send the token to Feishu.
  - confidence: 0.89
- **[P3] Return the normal Feishu suppressed result shape:** `extensions/feishu/src/outbound.ts:85`
  - body: Normal Feishu sends return chat-scoped metadata such as `chatId`, but this suppressed result returns `channelId: to`. Use `chatId: to` so downstream result formatting sees the same metadata shape as successful Feishu sends.
  - confidence: 0.72

  - body: This is a user-facing Feishu delivery fix, and the PR file list only changes Feishu source. OpenClaw policy requires a single-line `CHANGELOG.md` entry under the active fixes section before merge.
