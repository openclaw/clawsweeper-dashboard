## Summary

Keep open. Current main still lacks Feishu-local silent-token suppression at the relevant API boundaries, so the work is not obsolete, but this PR is not landable as written: it imports helpers from an unsupported root SDK surface, only guards text-only paths, leaves `sendMedia` and mixed media/text payloads able to forward `NO_REPLY`, and can bypass Feishu streaming-card closeout. The overlapping open #56167 is useful consolidation context but is not merged and does not fully supersede the reply-dispatcher side.

## What This Changes

This PR adds Feishu-local `NO_REPLY` suppression checks before outbound `sendText` and text-only reply-dispatcher delivery.

## Best Possible Solution

Land one canonical Feishu-owned fix that treats exact silent-token text as non-deliverable at every Feishu API boundary, preserves media delivery when the caption is silent, closes or avoids streaming placeholders correctly, imports helpers from a supported reply SDK subpath, and adds focused outbound plus reply-dispatcher regression tests.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by source inspection: direct `feishuOutbound.sendText({ text: "NO_REPLY" })`, `sendMedia({ text: "NO_REPLY", mediaUrl })`, and reply-dispatcher payloads with `NO_REPLY` all exercise Feishu API-boundary paths that current main does not guard locally.

## Solution Assessment

No. The direction is valid, but this patch is not the best solution because it only handles text-only paths, uses an unsupported root SDK import, and can skip streaming-card finalization; the safer path is a small Feishu-local boundary guard consolidated with the overlapping outbound PR and backed by tests.

  - body: The `!hasMedia` condition means a payload with media and `text === "NO_REPLY"` skips this guard and still sends the token through the text path before delivering media. Treat silent text as non-deliverable while still allowing media to proceed.
