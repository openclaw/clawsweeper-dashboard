## Summary

Keep open. The reported Feishu silent-token leak is real on current main, but this patch is incomplete and introduces a streaming-card cleanup regression before it can be merged.

## What This Changes

This PR adds Feishu `NO_REPLY` checks in `feishuOutbound.sendText` and the text-only reply-dispatcher delivery path.

## Best Possible Solution

Consolidate one Feishu-owned fix that suppresses exact silent tokens at every Feishu text API boundary, preserves media and streaming-card cleanup, adds focused regression tests, and includes the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is direct: call `sendText({ text: "NO_REPLY" })`, `sendMedia({ text: "NO_REPLY", mediaUrl })`, `sendMedia({ text: "NO_REPLY" })`, or dispatcher `deliver({ text: "NO_REPLY", mediaUrl })` with mocked Feishu send helpers and observe whether text reaches the Feishu helper.

## Solution Assessment

No. The direction is right, but the patch is not the best complete fix because it only covers one outbound text path and a no-media dispatcher case while leaving media captions/fallbacks, mixed payloads, streaming cleanup, SDK-boundary imports, and changelog coverage unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve streaming cleanup for silent finals:** `extensions/feishu/src/reply-dispatcher.ts:259-261`
  - body: In card streaming mode, `onReplyStart` may already create a Feishu streaming card before the final payload arrives. Returning here for text-only `NO_REPLY` skips the final-text path that updates stream state before idle close, which can leave the initial placeholder card content visible.
  - confidence: 0.86
- **[P2] Suppress silent text when media is present:** `extensions/feishu/src/reply-dispatcher.ts:259-261`
  - body: The `!hasMedia` condition means a payload with media plus `text === "NO_REPLY"` bypasses this guard and still enters the text-send block before media delivery. Treat the silent token as non-deliverable text while still allowing the media path to run.
  - confidence: 0.92
- **[P2] Guard sendMedia captions and fallback text:** `extensions/feishu/src/outbound.ts:83-87`
  - body: Adding the guard only in `sendText` leaves `sendMedia` paths that call `sendOutboundText` directly for captions and text-only fallback. A direct `sendMedia` call with `text: "NO_REPLY"` can still send the token to Feishu.
  - confidence: 0.9
- **[P3] Use a narrow SDK subpath for silent helpers:** `extensions/feishu/src/outbound.ts:3`
  - body: The Feishu plugin otherwise uses narrow `openclaw/plugin-sdk/*` entrypoints, and the scoped extension rules avoid the broad root SDK barrel in production code. Import these helpers from an existing reply subpath such as `openclaw/plugin-sdk/reply-runtime` or `reply-chunking`.
  - confidence: 0.76
- **[P3] Return the normal Feishu suppressed result shape:** `extensions/feishu/src/outbound.ts:85`
  - body: Normal Feishu sends return chat-scoped metadata such as `chatId`, but this suppressed result returns `channelId: to`. That can make downstream result formatting treat a suppressed Feishu send differently from successful Feishu sends.
  - confidence: 0.72
- **[P3] Add the required changelog entry:** `extensions/feishu/src/outbound.ts:83-87`
  - body: This is a user-facing Feishu delivery fix, and the PR only changes source files. OpenClaw policy requires a `CHANGELOG.md` entry for user-facing fixes before merge.
  - confidence: 0.82
