## Summary

Keep open: the patch is a focused fix for a real current-main WhatsApp media cap gap, and I found no blocking code-review findings, but this external PR still needs real WhatsApp/OpenClaw behavior proof before merge.

## What This Changes

The branch switches WhatsApp inbound and quoted media downloads from Baileys buffer mode to stream mode, enforces `mediaMaxMb` while reading, and adds regression tests.

## Best Possible Solution

Land the targeted stream-cap change after live WhatsApp/OpenClaw proof confirms over-limit inbound media stops during download and quoted fallback is skipped.

## Reproduction Assessment

Yes. Source inspection shows current main requests Baileys `"buffer"` mode and only passes the byte cap to `saveMediaBuffer` afterward, and Baileys buffer mode concatenates the stream before returning.

## Solution Assessment

Yes, the proposed stream read is the narrowest maintainable fix for early cap enforcement while keeping persistence unchanged. It still needs live behavior proof before merge.
