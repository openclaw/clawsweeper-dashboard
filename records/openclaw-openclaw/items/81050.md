## Summary

Keep open: the patch is a focused and correct-looking fix for WhatsApp inbound media cap enforcement, but this external PR still has only mocked test proof and needs redacted real WhatsApp/OpenClaw behavior proof before merge.

## What This Changes

The PR changes WhatsApp inbound media downloads from Baileys buffer mode to stream mode, enforces `mediaMaxMb` while reading, prevents quoted fallback after limit rejection, and adds regression tests.

## Best Possible Solution

Land the targeted stream-cap change after the contributor adds redacted real WhatsApp/OpenClaw proof showing over-limit inbound media stops during download and quoted fallback is skipped.

## Reproduction Assessment

Yes. Source inspection of current main shows `downloadInboundMedia` requests Baileys `"buffer"` mode and the monitor applies `mediaMaxMb` only after the buffer returns via `saveMediaBuffer`; Baileys buffer mode concatenates stream chunks before returning.

## Solution Assessment

Yes. Streaming the Baileys download and enforcing `maxBytes` during iteration is the narrow maintainable fix, and rethrowing the limit error prevents quoted-media fallback; it still needs real behavior proof before merge.
