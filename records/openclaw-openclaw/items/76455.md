## Summary

Keep open. The security hardening direction is valid, but the PR is not merge-ready because it tightens the official npm trust path without updating an existing npm-spec test table that still expects bare official packages with launch code to bypass the scanner, and it is missing the required changelog entry.

## What This Changes

The PR requires exact-version npm specs plus non-empty `expectedIntegrity` before official `@openclaw/*` npm installs can use the trusted scan bypass, and adds fixture coverage for those trust cases.

## Best Possible Solution

Update the PR so the existing npm-spec install tests and changelog match the stricter trust contract, then land the hardening after green plugin/install checks and maintainer security review.

## Reproduction Assessment

Yes. Source inspection shows current main can mark official npm installs as trusted using name and plugin id alone, and the existing npm-spec tests exercise bare `@openclaw/*` packages with dangerous launch-code fixtures.

## Solution Assessment

Mostly yes. Requiring exact-version specs plus `expectedIntegrity` at the trust helper is the narrowest fix for this bypass, but the PR also needs to update the existing npm-spec test expectations and changelog before merge.
