## Summary

Keep open. Current main still lacks the requested command-lane circuit breaker, but this PR should not merge as-is because it carries a stale timeout API and an advertised priority option that is not implemented.

## What This Changes

The PR adds command-lane circuit-breaker rejection options and default main-lane saturation rejection, plus command-queue tests, while also carrying older maxExecutionMs timeout changes.

## Best Possible Solution

Reimplement only the circuit-breaker portion on current main: add typed breaker options and the rejection error, check depth/wait before enqueue, keep the existing timeout API, add focused tests, and include the required changelog entry if the default main-lane behavior is accepted.

## Reproduction Assessment

Yes for the PR's target behavior: a unit test can block a lane, fill queued work to the configured threshold, and assert the next enqueue rejects immediately with the breaker error. I did not run tests because this review is read-only.

## Solution Assessment

No, this PR is not the best way to solve it as written. The safer path is a small current-main patch that keeps `taskTimeoutMs`, omits the unimplemented priority option, and applies the circuit-breaker checks directly to the existing queue implementation.
