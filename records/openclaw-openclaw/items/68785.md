## Summary

Keep this PR open as useful source work, but do not merge it as-is. Current main still lacks the requested command-lane circuit breaker, while the PR is dirty against main and has concrete correctness blockers around default coverage, stale timeout API shape, and an advertised but unused priority option.

## What This Changes

The PR adds command-lane circuit-breaker errors and options with default main-lane saturation rejection tests, while also carrying an older maxExecutionMs timeout patch in the command queue.

## Best Possible Solution

Land a small current-main implementation that adds circuit-breaker options to the shared command queue contract, applies default saturation rejection to the actual main run lane path, preserves the existing taskTimeoutMs timeout API, and covers the behavior with focused command-queue and embedded-run path tests.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a stability feature PR. The behavior can be verified with a focused unit scenario that blocks a lane, fills depth or wait thresholds, and asserts that the next enqueue rejects immediately with a retry hint.

## Solution Assessment

No, this branch is not currently the best way to solve the issue. The safer fix is to rebase only the circuit-breaker work onto current main's CommandQueueEnqueueOptions/taskTimeoutMs shape and ensure production main-lane enqueues receive the default breaker.
