- **[P1] Accept scoped tokens on lifecycle requests:** `src/gateway/mcp-http.request.ts:223-227`
  - body: The scoped-token lookup is only reached on POST. GET/SSE startup and DELETE teardown still compare only `ownerToken`/`nonOwnerToken`, while Codex sends the same bearer Authorization on those lifecycle requests, so a generated client can list/call over POST but get 401 when opening or closing the Streamable HTTP transport.
  - confidence: 0.96
- **[P1] Fingerprint every scoped-token field:** `src/agents/cli-runner/claude-live-session.ts:282-288`
  - body: The live-session fingerprint hashes only five fields, but the scoped token also binds session id, channel/thread/message ids, inbound audio, delivery mode, and explicit-target policy. Changing one of those omitted fields can reuse a warm Claude process with the previous scoped token and stale loopback context.
  - confidence: 0.94
- **[P2] Revoke scoped tokens on later prepare failures:** `src/agents/cli-runner/prepare.ts:442-457`
  - body: The token is registered before later preparation work runs, but only the bundle-MCP config catch revokes it. If `prepareExecution`, skill preparation, prompt tool resolution, or later session checks throw before cleanup is returned, the scoped bearer stays registered in the process token map.
  - confidence: 0.9
- **[P2] Remove obsolete scope headers from MCP config:** `src/gateway/mcp-http.loopback-runtime.ts:449-461`
  - body: The generated loopback config still emits session/account/channel/event/delivery/target context headers even though this fix is meant to make token scope authoritative. Keep the required Authorization and diagnostic capture plumbing, but remove the obsolete client-controlled scope channel to avoid drift around the auth boundary.
  - confidence: 0.88
