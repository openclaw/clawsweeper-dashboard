## Summary

Needs changes before merge. The token-bound scope direction is the right fix layer for the MCP loopback security issue, but the current branch still has scoped-token lifecycle, live-session reuse, cleanup, and stale header/env gaps on the security boundary.

## What This Changes

The PR changes MCP loopback auth to issue scoped bearer tokens, resolves POST tool context from token scope, updates CLI runner/live-probe MCP env handling, and adds regression tests for spoofed loopback scope headers.

## Best Possible Solution

Land a security-reviewed token-bound loopback implementation that covers every lifecycle method, fingerprints the full token scope, revokes tokens on all failure paths, removes obsolete scope header/env plumbing, and then resolves the canonical security issue.

Yes, source inspection gives a high-confidence reproduction path: current main and v2026.6.8 still thread mutable MCP loopback scope headers into tool context after bearer authentication. I did not run an end-to-end exploit in this read-only review.

## Solution Assessment

No, not as submitted. Token-bound scope is the right layer, but the branch needs full lifecycle-method auth, full live-session fingerprinting, token cleanup, and stale header/env removal before it is the best fix.

Full review comments:

- **[P1] Accept scoped tokens on all lifecycle methods:** `src/gateway/mcp-http.request.ts:223-227`
  - body: The new scoped-token lookup runs only after the GET and DELETE branches have already returned, so a Streamable HTTP client using the scoped bearer that POST accepts can still receive 401 on SSE startup or session teardown. Share the scoped-token auth path across POST, GET, and DELETE.
  - confidence: 0.91
- **[P1] Fingerprint the full scoped token context:** `src/agents/cli-runner/claude-live-session.ts:281-287`
  - body: The scoped bearer now binds session id, channel, thread, message, audio, delivery, explicit-target, and owner context, but Claude live-session reuse hashes only five fields. A warm process can keep an old OPENCLAW_MCP_TOKEN when one of the omitted scope fields changes.
  - confidence: 0.93
- **[P2] Revoke scoped tokens on later prepare failures:** `src/agents/cli-runner/prepare.ts:442-458`
  - body: The scoped token is issued before prepareExecution, auth-epoch resolution, reference-path work, and skills setup. If any later step throws before the returned cleanup is installed, the bearer remains registered in the loopback token table.
  - confidence: 0.87
- **[P2] Remove obsolete scope env from the generated MCP path:** `src/agents/cli-runner/prepare.ts:470-490`
  - body: The PR says the loopback client stops sending mutable scope context, but prepare still exports the old scope env placeholders used by the generated config. Keeping that client-controlled channel in a security boundary makes the new contract confusing and easier to regress.
  - confidence: 0.84
