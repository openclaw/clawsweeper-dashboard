## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Accept scoped tokens on GET and DELETE:** `src/gateway/mcp-http.request.ts:151-153`
  - body: `OPENCLAW_MCP_TOKEN` is now a scoped bearer, but the GET and DELETE branches still authenticate only the owner/non-owner runtime tokens. The MCP Streamable HTTP client reuses configured request headers for GET SSE startup, POST sends, and DELETE session termination, so generated clients can 401 on lifecycle methods even though POST works.
  - confidence: 0.92
- **[P1] Fingerprint every scoped token field:** `src/agents/cli-runner/claude-live-session.ts:279-287`
  - body: The scoped bearer binds channel, thread, message, audio, sessionId, reply-delivery mode, explicit-target policy, and owner state, but the live-session fingerprint hashes only five fields. A warm Claude process can therefore keep a previous `OPENCLAW_MCP_TOKEN` when one of the omitted fields changes.
  - confidence: 0.91
- **[P2] Revoke scoped tokens across all prepare failures:** `src/agents/cli-runner/prepare.ts:442-458`
  - body: The token is registered before later fallible preparation, but cleanup is not wrapped around every path before the final returned cleanup exists. If skills plugin preparation or another intervening step throws, the scoped bearer can remain registered until the loopback runtime is reset.
  - confidence: 0.84
- **[P2] Remove stale loopback scope headers:** `src/gateway/mcp-http.loopback-runtime.ts:447-461`
  - body: The PR says generated loopback config now sends only the bearer, but this config still emits session/account/channel/current-message/inbound-event scope headers. Even if POST no longer trusts them, keeping the old mutable context surface makes the token-bound contract ambiguous and easier to regress.
  - confidence: 0.86
