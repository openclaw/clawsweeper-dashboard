## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Accept scoped tokens on lifecycle requests:** `src/gateway/mcp-http.request.ts:222-227`
  - body: The scoped-token lookup is only reached on POST. Codex Streamable HTTP sends bearer Authorization on POST, DELETE teardown, and GET/SSE startup, so a generated client can list or call tools over POST but then get 401 when opening or closing the transport.
  - confidence: 0.96
- **[P1] Fingerprint every scoped-token field:** `src/agents/cli-runner/claude-live-session.ts:282-288`
  - body: The live-session fingerprint hashes only five fields, but the scoped token also binds session id, channel/thread/message ids, inbound audio, delivery mode, and explicit-target policy. Changing one of those omitted fields can reuse a warm Claude process with the previous scoped token and stale loopback context.
  - confidence: 0.94
- **[P2] Revoke scoped tokens on later prepare failures:** `src/agents/cli-runner/prepare.ts:442-457`
  - body: The token is registered before later preparation work runs, but the prepareExecution failure path only cleans up the backend and then rethrows. If that or later auth/skill preparation fails before cleanup is returned, the scoped bearer can stay registered in the process token map.
  - confidence: 0.9
- **[P2] Remove obsolete scope headers from MCP config:** `src/gateway/mcp-http.loopback-runtime.ts:449-462`
  - body: The generated loopback config still emits session/account/channel/event/delivery/target context headers even though this fix is meant to make token scope authoritative. Keep Authorization and required capture plumbing, but remove the obsolete client-controlled scope channel to avoid drift around the auth boundary.
  - confidence: 0.86
