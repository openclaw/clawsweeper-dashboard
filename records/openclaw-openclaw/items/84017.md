## Summary

Keep this PR open: token-bound MCP loopback scope is the right security direction, but this head still has merge-blocking lifecycle, live-session, and token-cleanup defects on the gateway/agent boundary.

## What This Changes

The PR issues scoped MCP loopback bearer tokens, resolves POST tool context from token scope, adjusts CLI-runner MCP env/logging, and adds spoofed-header regression tests.

## Best Possible Solution

Land a security-reviewed token-bound loopback implementation that authenticates scoped bearers consistently across POST, GET, and DELETE, fingerprints the full token scope, revokes tokens on every failure path, and removes obsolete scope context plumbing.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.8 still derive MCP loopback scope from mutable request headers after bearer authentication; I did not run an end-to-end exploit in this read-only review.

Full review comments:

- **[P1] Accept scoped tokens on lifecycle requests:** `src/gateway/mcp-http.request.ts:223-227`
  - body: The scoped-token lookup runs only after the GET and DELETE branches have already authenticated against the legacy owner/non-owner tokens and returned. A client using the scoped bearer accepted by POST can still get 401 on SSE startup or session teardown; share the scoped-token auth path across POST, GET, and DELETE.
  - confidence: 0.93
- **[P1] Fingerprint the full scoped token context:** `src/agents/cli-runner/claude-live-session.ts:279-287`
  - body: The scoped bearer now binds session id, channel, thread, message, audio, delivery, explicit-target, and owner context, but Claude live-session reuse hashes only accountId, inboundEventKind, messageProvider, senderIsOwner, and sessionKey. A warm process can keep an old OPENCLAW_MCP_TOKEN when one of the omitted scope fields changes.
  - confidence: 0.94
- **[P2] Revoke scoped tokens on all prepare failures:** `src/agents/cli-runner/prepare.ts:442-458`
  - body: The scoped token is issued before prepareExecution, auth-epoch resolution, reference-path work, and skills setup. If any later step throws before the returned cleanup is installed, the bearer remains registered in the loopback token table.
  - confidence: 0.89
