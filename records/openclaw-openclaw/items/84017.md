## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Authenticate scoped tokens on GET and DELETE:** `src/gateway/mcp-http.request.ts:223-227`
  - body: The PR mints `OPENCLAW_MCP_TOKEN` as a scoped bearer, but `validateMcpLoopbackRequest` only resolves scoped token context after the GET and DELETE branches have returned. The MCP SDK's Streamable HTTP transport opens GET SSE streams and sends DELETE teardown using the same configured headers, so generated loopback clients can 401 on those methods even though POST works.
  - confidence: 0.89
- **[P1] Include every token-bound field in the live fingerprint:** `src/agents/cli-runner/claude-live-session.ts:279-287`
  - body: The scoped token binds channel/thread/message/audio, sessionId, reply-delivery mode, and explicit-target policy, but the live-session fingerprint hashes only accountId, inboundEventKind, messageProvider, owner, and sessionKey. Reusing a warm Claude process can therefore keep the previous `OPENCLAW_MCP_TOKEN` when one of the omitted fields changes.
  - confidence: 0.92
- **[P2] Revoke scoped tokens on all prepare failure paths:** `src/agents/cli-runner/prepare.ts:431-433`
  - body: The cleanup guard only revokes when `prepareCliBundleMcpConfig` throws, but later preparation steps run before the returned cleanup exists. If any of those later steps throw, the scoped bearer stays registered until the loopback server closes.
  - confidence: 0.86
