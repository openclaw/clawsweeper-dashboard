## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Accept scoped tokens on lifecycle requests:** `src/gateway/mcp-http.request.ts:151-153`
  - body: The scoped-token lookup runs only after the GET and DELETE branches have already authenticated against the legacy owner/non-owner tokens and returned. Codex Streamable HTTP sends Authorization on POST, GET, and DELETE, so a generated client using the scoped `OPENCLAW_MCP_TOKEN` can list tools but fail SSE startup or teardown with 401.
  - confidence: 0.95
- **[P1] Fingerprint every scoped-token field:** `src/agents/cli-runner/claude-live-session.ts:281-287`
  - body: The issued scoped bearer now binds session id, current channel/thread/message, inbound audio, delivery mode, explicit-target policy, and owner context, but Claude live-session reuse hashes only five fields. A warm process can keep an old `OPENCLAW_MCP_TOKEN` when one of the omitted scope fields changes.
  - confidence: 0.94
- **[P2] Revoke issued tokens on prepare failures:** `src/agents/cli-runner/prepare.ts:442-457`
  - body: The scoped token is registered before prepareExecution, auth epoch, and skills setup work, but the later failure paths do not revoke it before throwing and before the returned cleanup is installed. Those failures can leave a bearer registered in the loopback token table.
  - confidence: 0.9
- **[P2] Remove obsolete scope headers from generated config:** `src/gateway/mcp-http.loopback-runtime.ts:447-461`
  - body: `createMcpLoopbackServerConfig` still emits `x-session-key` and other scope headers/env placeholders even though scope is supposed to come from token identity. That keeps a second client-controlled scope channel alive and contradicts the stated cleanup and test name.
  - confidence: 0.88
