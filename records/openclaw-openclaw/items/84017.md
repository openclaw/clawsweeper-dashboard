## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Accept scoped tokens on GET and DELETE:** `src/gateway/mcp-http.request.ts:151-153`
  - body: `OPENCLAW_MCP_TOKEN` is now a scoped bearer, but the GET and DELETE branches still authenticate only the owner/non-owner runtime tokens. The MCP Streamable HTTP client builds common headers from the configured request headers and uses them for GET SSE startup and DELETE termination, so generated clients can 401 on those lifecycle methods even though POST works.
  - confidence: 0.9
- **[P1] Fingerprint every scoped token field:** `src/agents/cli-runner/claude-live-session.ts:279-287`
  - body: The scoped bearer includes channel, thread, message, audio, sessionId, reply-delivery mode, explicit-target policy, and owner state, but the live-session fingerprint only hashes five fields. A warm Claude process can therefore keep a previous `OPENCLAW_MCP_TOKEN` when one of the omitted fields changes.
  - confidence: 0.92
- **[P2] Revoke scoped tokens after later prepare failures:** `src/agents/cli-runner/prepare.ts:442-458`
  - body: The token is registered before `prepareExecution`, auth-epoch resolution, reference-path lookup, and skills setup. If any later preparation step throws before the returned cleanup is installed, this path can leave the scoped bearer registered in the loopback token table.
  - confidence: 0.86
