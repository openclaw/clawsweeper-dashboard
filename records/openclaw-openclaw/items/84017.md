## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Accept scoped tokens on lifecycle requests:** `src/gateway/mcp-http.request.ts:223-227`
  - body: The scoped-token lookup runs only after the GET/SSE and DELETE branches have already authenticated only the legacy owner/non-owner tokens. A generated client using the new scoped `OPENCLAW_MCP_TOKEN` can POST successfully but receive 401 on notification-stream startup or teardown.
  - confidence: 0.95
- **[P1] Fingerprint every scoped-token field:** `src/agents/cli-runner/claude-live-session.ts:279-287`
  - body: The live-session fingerprint hashes only five scope fields, but the issued bearer also binds session id, channel/thread/message ids, inbound audio, delivery mode, and explicit-target policy. A warm Claude process can keep an old token when one of those omitted fields changes.
  - confidence: 0.94
- **[P2] Revoke scoped tokens on prepare failures:** `src/agents/cli-runner/prepare.ts:442-457`
  - body: The scoped token is registered before later preparation steps run, but failures after bundle MCP config preparation do not revoke it before throwing and before the returned cleanup is installed. Those paths can leave a bearer registered in the loopback token table.
  - confidence: 0.9
- **[P2] Remove obsolete scope headers from generated MCP config:** `src/gateway/mcp-http.loopback-runtime.ts:447-461`
  - body: The PR claims generated loopback config now sends only Authorization, but the config still emits session, account, channel, event, delivery, and target-policy header placeholders. Keeping that client-controlled context channel conflicts with the token-bound scope contract and leaks unnecessary run context to MCP clients.
  - confidence: 0.88
