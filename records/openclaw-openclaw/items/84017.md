## Summary

Keep open. The token-bound loopback scope direction matches the accepted security issue, but the current PR is still not merge-ready because scoped tokens are only recognized on POST, the Claude live-session fingerprint omits token-bound context, prepare failures can leave scoped tokens registered, and GitHub reports the branch as dirty against current main.

## What This Changes

The branch adds process-local scoped MCP loopback tokens, resolves loopback request context from token scope, updates CLI/probe token issuance, and adds gateway plus CLI runner regressions for spoofed scope headers.

## Best Possible Solution

Land a rebased, security-reviewed token-bound loopback implementation that authenticates scoped tokens on POST/GET/DELETE, fingerprints and revokes the complete scoped context, and records the intended compatibility contract for generated loopback clients.

## Reproduction Assessment

Yes. Source inspection on current main shows the loopback bearer is authenticated before mutable request headers are threaded into scoped tool resolution, and the PR body supplies after-fix runtime logs for the central spoofed-header path.

## Solution Assessment

No, not as submitted. Token-bound loopback scope is the right fix direction, but the patch must cover Streamable HTTP GET/DELETE, the full live-session scope fingerprint, token cleanup failure paths, and the current-main rebase first.

  - confidence: 0.9

  - body: The scoped bearer includes channel/thread/message/audio, sessionId, reply-delivery mode, explicit-target policy, and owner state, but the live-session fingerprint only hashes five fields. A warm Claude process can therefore keep a previous `OPENCLAW_MCP_TOKEN` when one of the omitted fields changes.
  - confidence: 0.92

  - body: This catch only covers `prepareCliBundleMcpConfig`; later preparation steps run before the returned cleanup exists. If those steps throw, the scoped bearer stays registered until the loopback runtime is reset.
  - confidence: 0.86
