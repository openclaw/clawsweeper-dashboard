## Label Justifications

- `P1`: The PR fixes a gateway/agent trust-boundary issue but also introduces a source-proven regression in an active CLI live-session workflow.
- `merge-risk: 🚨 compatibility`: Existing Claude live-session users would see a documented warm-process behavior silently replaced by per-turn restarts.
- `merge-risk: 🚨 session-state`: The changed token value participates in live-session identity and can force session teardown/recreation despite unchanged user context.

## Summary

This PR targets a real MCP loopback trust-boundary gap and supplies convincing after-fix logs, but it should stay open because the new per-run random token becomes part of the Claude live-session fingerprint and will restart same-context live CLI sessions every turn.

## What This Changes

The PR moves MCP loopback session/account/channel/event scope from request headers into server-issued scoped bearer tokens, updates CLI bundle env/config generation, and adds spoofed-header regression coverage.

## Best Possible Solution

Keep server-side token-bound scope, but make the scoped-token lifecycle or live-session fingerprint stable for unchanged scope and add regression coverage for same-scope reuse plus changed-scope isolation.

## Reproduction Assessment

Yes. From source, the PR issues a fresh random scoped token on each prepare and Claude live-session fingerprinting hashes env values, so same-context live MCP turns will not match the previous fingerprint.

## Solution Assessment

No, not yet. Moving scope off request headers is the right security direction, but the implementation should preserve unchanged-scope live-session reuse instead of making the random token itself part of the reuse decision.
