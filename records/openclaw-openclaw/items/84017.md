## Summary

Keep open. Current main still has the header-driven MCP loopback scope problem and this PR is a credible fix candidate, but the current head is not merge-ready because scoped tokens are not accepted on GET/DELETE, the Claude live-session fingerprint omits token-bound context, and scoped tokens can remain registered after later prepare failures.

## What This Changes

The PR adds process-local scoped MCP loopback bearer tokens, resolves loopback request context from token scope, updates CLI/probe token issuance, and adds gateway plus CLI-runner regressions for spoofed scope headers.

## Best Possible Solution

Land a security-reviewed token-bound loopback implementation that accepts scoped tokens on POST/GET/DELETE, fingerprints and revokes the complete scoped context, and states the intended compatibility contract for generated loopback clients.

## Reproduction Assessment

Yes. Current main source still shows mutable loopback headers feeding scoped tool context, and the canonical security issue provides a concrete curl-shaped source reproduction path.

## Solution Assessment

No, not as submitted. Token-bound loopback scope is the right layer, but the branch must cover all transport methods, full live-session scope fingerprinting, and token cleanup failure paths before it is the best fix.

  - body: `OPENCLAW_MCP_TOKEN` is now a scoped bearer, but the GET and DELETE branches still authenticate only the owner/non-owner runtime tokens.   - confidence: 0.9

  - body: The scoped bearer includes channel/thread/message/audio, sessionId, reply-delivery mode, explicit-target policy, and owner state, but the live-session fingerprint only hashes five fields.   - confidence: 0.92

  - body: The token is registered before `prepareExecution`, auth-epoch resolution, and skills setup. If any later preparation step throws, the returned cleanup is never installed and this path does not revoke the scoped bearer.
