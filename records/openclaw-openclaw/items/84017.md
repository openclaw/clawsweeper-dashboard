## Summary

Keep this PR open for maintainer review: the source change appears to close the MCP loopback scope-spoofing gap and the prior live-session fingerprint finding is addressed, with remaining risk limited to an intentional compatibility/session-context tradeoff.

## What This Changes

The PR moves MCP loopback session/account/channel/event scope from request headers into server-issued scoped bearer tokens and stabilizes Claude live-session fingerprints across same-scope token rotation.

## Best Possible Solution

Land the token-scoped implementation after maintainers explicitly accept the legacy loopback-header compatibility break and normal gates pass.

Yes from source inspection: current main authenticates the bearer and then reads mutable scope headers into loopback tool resolution. I did not run an executable repro in this read-only review, but the PR adds targeted regression coverage for the spoofed-header path.

Yes, the proposed direction is the maintainable fix: move the trust boundary to server-issued scoped tokens and keep live-session reuse stable by fingerprinting the token scope. The remaining question is maintainer acceptance of the compatibility break, not a code defect I can point to in the patch.
