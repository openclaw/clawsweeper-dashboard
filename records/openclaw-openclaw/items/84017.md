## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Accept scoped tokens on all lifecycle methods:** `src/gateway/mcp-http.request.ts:223-224`
  - body: The new scoped-token lookup only runs after GET and DELETE have already returned, so a generated Streamable HTTP client can present the same scoped bearer that POST accepts and still get 401 on SSE startup or session teardown. Move scoped-token validation before those method branches or share the auth helper across POST, GET, and DELETE.
  - confidence: 0.91
- **[P1] Fingerprint the full scoped token context:** `src/agents/cli-runner/claude-live-session.ts:279-288`
  - body: The scoped bearer now binds session, channel, thread, message, audio, delivery, explicit-target, and owner context, but Claude live-session reuse hashes only five fields. A warm process can keep an old OPENCLAW_MCP_TOKEN when one of the omitted fields changes.
  - confidence: 0.93
- **[P2] Revoke scoped tokens on later prepare failures:** `src/agents/cli-runner/prepare.ts:442-458`
  - body: The scoped token is issued before prepareExecution, auth-epoch resolution, reference-path work, and skills setup. If any later step throws before the returned cleanup is installed, the bearer remains registered in the loopback token table.
  - confidence: 0.87
- **[P2] Remove stale scope headers from generated config:** `src/gateway/mcp-http.loopback-runtime.ts:447-461`
  - body: The PR says the loopback client stops emitting mutable scope headers, but the generated server config still carries the scope context headers. The server may ignore them after the fix, but keeping that client-controlled channel in a security boundary makes the contract confusing and easier to regress.
  - confidence: 0.84
