- **[P1] Accept scoped tokens on GET and DELETE:** `src/gateway/mcp-http.request.ts:151-153`
  - body: `OPENCLAW_MCP_TOKEN` is now a scoped bearer, but the GET and DELETE branches still authenticate only the old owner/non-owner runtime tokens. The MCP Streamable HTTP client reuses configured headers for GET SSE startup and DELETE teardown, so generated loopback clients can fail lifecycle calls even though POST works.
  - confidence: 0.91
- **[P1] Fingerprint the full scoped token context:** `src/agents/cli-runner/claude-live-session.ts:279-287`
  - body: The scoped bearer binds session, channel, thread, message, audio, delivery, explicit-target, and owner context, but the Claude live-session fingerprint hashes only five fields. A warm process can therefore keep an old `OPENCLAW_MCP_TOKEN` when one of the omitted context fields changes.
  - confidence: 0.93
- **[P2] Revoke scoped tokens on later prepare failures:** `src/agents/cli-runner/prepare.ts:442-458`
  - body: The scoped token is registered before `prepareExecution`, auth-epoch resolution, reference-path work, and skills setup. If any later step throws before the returned cleanup is installed, the bearer remains registered in the loopback token table.
  - confidence: 0.87
- **[P2] Remove stale scope headers from generated config:** `src/agents/cli-runner/prepare.ts:474-489`
  - body: This PR says the loopback client stops emitting mutable scope headers, but the prepared backend env still carries the scope variables that feed those generated headers. The server may ignore them now, but keeping the client-controlled scope channel in the generated config leaves the security boundary confusing and easier to regress.
  - confidence: 0.82
