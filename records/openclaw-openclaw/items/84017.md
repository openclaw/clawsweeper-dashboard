## Summary

Keep open: the token-bound direction is still needed for the open security issue, but this PR is not merge-ready because scoped tokens only work on POST and the token context can still go stale or remain registered on failure paths.

## What This Changes

The PR changes MCP loopback auth to issue scoped bearer tokens, resolve POST tool context from token scope, adjust CLI runner/live-probe MCP env handling, and add spoofed-header regression tests.

## Best Possible Solution

Land a security-reviewed token-bound loopback implementation that authenticates scoped bearers consistently across POST, GET, and DELETE, fingerprints the full token scope, revokes tokens on every failure path, removes obsolete scope header/env plumbing, and then resolves the canonical security issue.

- **[P1] Accept scoped tokens on lifecycle requests:** `src/gateway/mcp-http.request.ts:223-227`
  - body: The scoped-token lookup runs only after the GET and DELETE branches have already authenticated against the legacy owner/non-owner tokens and returned. A Streamable HTTP client using the scoped bearer accepted by POST can still get 401 on SSE startup or session teardown; share the scoped-token auth path across POST, GET, and DELETE.
  - confidence: 0.93
- **[P1] Fingerprint the full scoped token context:** `src/agents/cli-runner/claude-live-session.ts:279-287`
  - body: The scoped bearer now binds session id, channel, thread, message, audio, delivery, explicit-target, and owner context, but Claude live-session reuse hashes only accountId, inboundEventKind, messageProvider, senderIsOwner, and sessionKey. A warm process can keep an old OPENCLAW_MCP_TOKEN when one of the omitted scope fields changes.
  - confidence: 0.94

  - body: The scoped token is issued before prepareExecution, auth-epoch resolution, reference-path work, and skills setup. If any later step throws before the returned cleanup is installed, the bearer remains registered in the loopback token table.
  - confidence: 0.89

  - confidence: 0.86
