## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Accept scoped tokens on lifecycle requests:** `src/gateway/mcp-http.request.ts:223-227`
  - body: The scoped-token lookup runs after the GET/SSE and DELETE branches have already authenticated only the legacy owner/non-owner tokens and returned. A generated client that receives the new scoped `OPENCLAW_MCP_TOKEN` can POST successfully but get 401 on lifecycle requests.
  - confidence: 0.95
- **[P1] Fingerprint every scoped-token field:** `src/agents/cli-runner/claude-live-session.ts:279-287`
  - body: The live-session fingerprint hashes only five scope fields, but the issued bearer also binds session id, channel/thread/message ids, inbound audio, delivery mode, and explicit-target policy. A warm Claude process can keep an old token when one of those omitted fields changes.
  - confidence: 0.94
- **[P2] Revoke issued tokens on prepare failures:** `src/agents/cli-runner/prepare.ts:442-457`
  - body: The scoped token is registered before later preparation steps run, but failures after `prepareCliBundleMcpConfig` do not revoke it before throwing and before the returned cleanup is installed. Those paths can leave a bearer registered in the loopback token table.
  - confidence: 0.9
- **[P2] Remove obsolete scope env plumbing:** `src/agents/cli-runner/prepare.ts:470-489`
  - body: After issuing a scoped token, this branch still populates the env values that feed the old mutable scope headers in the generated MCP config. Keeping that client-controlled context channel undermines the token-bound scope cleanup and conflicts with the PR's stated contract.
  - confidence: 0.88
