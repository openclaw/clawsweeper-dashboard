## Summary

Keep open. Current main and v2026.6.8 still need the security fix, but this PR is not merge-ready because the scoped-token implementation still splits lifecycle authentication, fingerprints only part of the token-bound context, and can leave scoped tokens registered after prepare failures.

## What This Changes

The PR binds MCP loopback request context to server-issued scoped bearer tokens, updates CLI-runner loopback env/session handling, and adds spoofed-header regression tests.

## Best Possible Solution

Land a security-reviewed token-bound loopback implementation that authenticates scoped bearers consistently across POST, GET, and DELETE, fingerprints the full token scope, revokes tokens on every failure path, and removes obsolete context plumbing.

## Reproduction Assessment

Yes. Source inspection of current main and v2026.6.8 shows mutable request headers still drive MCP loopback session/account/channel scope after bearer auth, and the PR body supplies real logs for the POST spoofing path; I did not run a live exploit in this read-only review.

## Solution Assessment

No. Token-bound scope is the right layer, but this patch is not the best fix until it covers all lifecycle methods, full token-scope fingerprinting, token cleanup, and generated-config cleanup.
