## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Authenticate scoped tokens on GET and DELETE:** `src/gateway/mcp-http.request.ts:223-227`
  - body: `OPENCLAW_MCP_TOKEN` is now issued as a scoped bearer, but the GET and DELETE branches return before `resolveScopedTokenContext` runs. The MCP Streamable HTTP transport uses the same configured Authorization header for GET notification streams and DELETE teardown, so generated clients can 401 on those methods even when POST works.
  - confidence: 0.91
- **[P1] Fingerprint the full scoped-token context:** `src/agents/cli-runner/claude-live-session.ts:279-287`
  - body: The scoped bearer binds sessionId, channel/thread/message/audio, reply-delivery mode, explicit-target policy, and owner state, but the live-session fingerprint hashes only five fields. A warm Claude process can therefore keep a previous `OPENCLAW_MCP_TOKEN` when one of the omitted fields changes.
  - confidence: 0.92
- **[P2] Remove stale scope headers from loopback config:** `src/gateway/mcp-http.loopback-runtime.ts:446-461`
  - body: `createMcpLoopbackServerConfig` still emits session/account/channel/delivery scope headers even though the PR is supposed to make the server-side scoped token authoritative. Keeping the old mutable scope channel in generated configs creates drift and leaves custom clients with a competing source of truth.
  - confidence: 0.86
- **[P2] Revoke scoped tokens after later prepare failures:** `src/agents/cli-runner/prepare.ts:524-530`
  - body: The returned cleanup revokes the token, but several later awaits can throw before that cleanup is returned, and the prepareExecution catch only runs backend cleanup. Those paths leave scoped bearer tokens registered until the loopback runtime resets.
  - confidence: 0.87
