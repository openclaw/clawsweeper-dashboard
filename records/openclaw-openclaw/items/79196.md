## Summary

Current main already prevents this from becoming a lost cron task by finalizing persisted running jobs as restart-interrupted and letting task maintenance recover from durable cron state before lost marking.

## What This Changes

The issue asks for running cron job identity to survive gateway restarts so in-flight cron tasks are not later marked lost.

## Best Possible Solution

Keep the current restart-interrupted finalization and durable recovery path rather than rehydrating process-local active IDs for work killed by a Gateway restart.

## Reproduction Assessment

No current-main failing path was found. Source and regression coverage show startup records persisted running cron jobs as interrupted, then task maintenance recovers them from durable state before lost marking.

## Solution Assessment

Yes. The current implementation uses persisted `runningAtMs`, run logs, and job state to produce a terminal cron task result, which is narrower and safer than treating a post-restart process-local active set as still authoritative.
