## Summary

Close as implemented: current main and the shipped v2026.4.25 release no longer rely on a restart-persistent activeJobIds set for this path; persisted runningAtMs jobs are recorded as interrupted failures on startup, and task maintenance recovers cron task records from durable run logs or job state before they can be marked lost.

## What This Changes

The issue asks for cron active job tracking to survive Gateway restarts so in-flight cron task records are not later marked lost with backing session missing.

## Best Possible Solution

Keep the shipped startup-interruption and durable task-recovery behavior; persisting activeJobIds would make dead post-restart work look live instead of recording a clear terminal interruption.

## Reproduction Assessment

No, not against current main. The old failure path is source-understandable, but current main records persisted runningAtMs jobs as interrupted failures and recovers cron task records before lost handling.

## Solution Assessment

Yes. The current solution is safer than persisting activeJobIds because a Gateway restart kills the in-process job; recording a visible interrupted failure and recovering the task ledger is the narrow maintainable behavior.
