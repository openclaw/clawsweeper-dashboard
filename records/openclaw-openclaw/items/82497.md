## Summary

Keep this PR open. It targets a valid auth-profile precedence bug, but the latest diff still broadens explicit selected-agent auth order and the real behavior proof is stale after the force-push.

## What This Changes

The PR changes auth-profile merge precedence and adds focused auth regression coverage, with additional cron persisted-shape and pairing test stabilization changes.

## Best Possible Solution

Keep the shared merge fix, but only synthesize local-first fallback order when the selected agent has no explicit order for that provider; refresh latest-head proof and split unrelated stabilization if it is not part of the auth repair.

## Reproduction Assessment

Yes. Current main clearly merges main and requested auth stores before resolveAuthProfileOrder honors stored provider order, and the PR body demonstrates the intended MiniMax local-first path with synthetic OAuth values.

## Solution Assessment

No. The shared merge boundary is the right place, but this implementation must preserve explicit selected-agent provider order instead of appending every selected-agent profile that was omitted from that order.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve explicit local auth order:** `src/agents/auth-profiles/persisted.ts:813`
  - body: When the selected-agent store already has an order entry for a provider, that order is authoritative: docs say omitted profiles must not be silently tried later. This appends every local profile for that provider after the explicit order, so a credential the agent intentionally left out can re-enter runtime auth selection after merging with main fallback. Only synthesize local-first fallback order when there is no selected-agent order entry for that provider.
  - confidence: 0.91
