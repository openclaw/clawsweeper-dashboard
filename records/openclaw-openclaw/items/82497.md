## Summary

Keep open: the PR targets a real source-reproducible auth precedence bug, but the current diff can break explicit selected-agent auth order by re-adding local profiles that the order intentionally omitted.

## What This Changes

The PR changes auth-profile store merging so selected-agent provider profiles and order entries precede inherited main-agent profiles, with a regression test for MiniMax same-provider precedence.

## Best Possible Solution

Keep the shared merge fix, but only synthesize local-first fallback order when the selected agent has no explicit order for that provider; explicit local order must remain authoritative.

## Reproduction Assessment

Yes. Source inspection shows current main merges main and requested auth stores before resolveAuthProfileOrder honors stored provider order; the PR body supplies terminal proof for the intended local-first path, and the review finding is reproducible from the new merge helper logic.

## Solution Assessment

No. The shared merge boundary is the right place, but this patch appends every local provider profile even when a local explicit order intentionally omitted it; preserve local explicit order and add coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve explicit local auth order:** `src/agents/auth-profiles/persisted.ts:813`
  - body: When the selected-agent store already has an order entry for a provider, that order is authoritative: docs say omitted profiles must not be silently tried later. This helper appends every local profile for that provider after the explicit order, so a profile the agent intentionally left out can re-enter runtime auth selection after merging with main fallback. Only synthesize local-first fallback order when there is no selected-agent order entry for that provider.
