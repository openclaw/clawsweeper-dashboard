## Summary

Keep open: the PR targets a real auth precedence bug, but latest head still has blocking correctness issues in auth order merging and an unrelated cron persisted-payload change, and its real behavior proof is tied to an older head.

## What This Changes

The PR changes auth-profile store merging to prefer selected-agent provider profiles, adds focused auth regression coverage, and includes cron persisted-shape plus pairing test stabilization changes.

## Best Possible Solution

Keep the shared auth-profile merge fix, but preserve explicit selected-agent order semantics and remove or separately justify the unrelated cron persisted-shape change before merge.

## Reproduction Assessment

Yes. Current main has a source-reproducible path: selected-agent stores are merged with main stores through mergeAuthProfileStores, and base-first profile/order insertion can leave inherited main same-provider profiles ahead of local profiles when IDs differ.

## Solution Assessment

No. The shared merge boundary is the right place, but this implementation must not append selected-agent profiles that explicit local order omitted, and the unrelated cron validation loosening should not ride along with the auth fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve explicit local auth order:** `src/agents/auth-profiles/persisted.ts:813`
  - body: When the selected-agent store has an order entry for a provider, that order is authoritative: omitted profiles must stay excluded. This appends every local profile for the provider after the explicit order, so a credential the agent intentionally left out can be tried at runtime; only synthesize local-first fallback order when no selected-agent order entry exists.
  - confidence: 0.91
- **[P2] Keep blank cron payloads invalid:** `src/cron/persisted-shape.ts:55-61`
  - body: Removing the trim checks lets persisted jobs with whitespace-only systemEvent text or agentTurn message pass getInvalidPersistedCronJobReason. The cron loader and doctor repair path rely on invalid-payload to skip or prune those malformed jobs, and current service coverage expects empty systemEvent jobs not to enqueue work.
  - confidence: 0.84
