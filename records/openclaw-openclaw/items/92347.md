## Label Justifications

- `P2`: The PR addresses a real scheduled-spend observability bug, but the current patch would still produce incorrect totals rather than disrupt agent execution.
- `merge-risk: 🚨 other`: Merging the current head would silently undercount cron total-token telemetry in bundled diagnostics exporters, a meaningful observability risk outside the specific merge-risk taxonomy.

## Summary

The missing cron diagnostic emission is valid and the chosen finalization point is appropriate, but the patch exports a prompt/context snapshot as total billed tokens, so bundled diagnostics would under-report every cron completion with output tokens; real after-fix runtime proof is also still missing.

## What This Changes

The PR adds trusted `model.usage` diagnostic events for nonzero isolated cron runs, including trace, session, usage, context, cost, and duration metadata, plus positive and zero-usage tests.

## Best Possible Solution

Keep the event at the existing nonzero-usage finalization point, derive its billing total as `usage.total ?? input + cacheRead + cacheWrite + output`, retain `deriveSessionTotalTokens` only for session-context accounting, assert the full emitted shape, and provide one redacted live cron-to-subscriber proof.

## Reproduction Assessment

Yes at source level: current main records nonzero cron usage but lacks the event consumed by bundled diagnostics exporters, and the linked report documents the same behavior on release `v2026.6.5`. This review did not independently execute the live OTel reproduction.

## Solution Assessment

No, not in its current form. The emission location and most fields are appropriate, but `usage.total` must use billing-token semantics rather than the session context snapshot before this is the narrowest correct solution.
