Keep open for now: current main has a terminal-prune path and an existing regression test, but the reported mark-lost control flow concern is real enough that the exact SQLite row state needs a focused current-main repro before closing or changing retention semantics.

## What This Changes

The issue reports that `openclaw tasks maintenance --apply` reconciles terminal task runs but leaves expired terminal records in the SQLite `task_runs` table.

## Best Possible Solution

Add a focused current-main SQLite regression seeded with an expired terminal `task_runs` row and fix the maintenance/store path only if that row survives `--apply`, while preserving the documented seven-day retention contract unless maintainers choose a separate force-prune feature.

## Reproduction Assessment

No high-confidence reproduction yet. Source confirms the mark-lost branch skips the later prune branch in the same pass, but current main also has a terminal prune implementation and an existing test that contradict the broader claim that expired terminal rows are never deleted.

## Solution Assessment

Unclear. A direct expired-row prune failure should be solved in `src/tasks` with SQLite-backed regression coverage; changing lost-task retention timing or adding `--prune-all-terminal` would need an explicit maintainer behavior decision.
