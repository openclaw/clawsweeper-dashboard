## Summary

Keep this PR open. Current main still lacks the PR's narrow `message_end` block-reply suppression for assistant `stopReason === "toolUse"` turns with structured tool-call blocks, while the current OpenAI Responses conversion can still produce exactly that text-plus-`toolCall` shape. The diff is focused and I found no blocking correctness or security finding, but it should be rebased and revalidated against current main before merge.

## What This Changes

The PR adds a guarded suppression path in `handleMessageEnd` plus regression tests so Telegram `message_end` block replies are skipped only for `toolUse` assistant turns containing structured tool-call blocks, while preserving normal `stop` turns and `toolUse` turns without tool-call blocks.

## Best Possible Solution

Land this PR or an equivalent rebased patch in the current `handleMessageEnd` implementation: suppress only `message_end` block replies for assistant turns with `stopReason === "toolUse"` and structured tool-call blocks, preserve `text_end`, normal `stop`, and toolUse-without-tool-block behavior, keep the regression tests, then run the targeted agent/Telegram checks and changed gate before merge.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to emit a `message_end` assistant message with `stopReason: "toolUse"`, text content, and a structured `toolCall` block through `subscribeEmbeddedPiSession` with `blockReplyBreak: "message_end"`; current main's code path still reaches `onBlockReply` for that shape.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix I found: it gates only the message-end block-reply path on the exact intermediate tool-use shape and adds tests for the preserved cases. The safer merge path is to rebase onto current main and apply the same guard around the newer `silentExpected` and deterministic-approval checks already present there.
