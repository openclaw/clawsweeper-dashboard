## What This Changes

The PR adds an embedded-Pi `message_end` guard that suppresses block replies for `toolUse` assistant turns with tool-call blocks, plus regression tests for suppressed and non-suppressed cases.

## Best Possible Solution

Refresh and land the narrow toolUse/message_end suppression guard with regression coverage and redacted real Telegram or bot-to-bot QA proof, coordinated with overlapping leak-boundary work.

## Reproduction Assessment

Yes, by source inspection. Current main can take visible text from a `message_end` assistant turn and emit it through `onBlockReply` when `blockReplyBreak` is `message_end`, while OpenAI transport produces `stopReason: "toolUse"` messages with `toolCall` blocks; I did not run tests because this review is read-only.

## Solution Assessment

Yes. The `stopReason` plus structured tool-call guard is the narrowest maintainable fix for this specific leak, but it should be rebased into the current `handleMessageEnd` shape and kept separate from broader sanitizer or session-routing policy changes.
