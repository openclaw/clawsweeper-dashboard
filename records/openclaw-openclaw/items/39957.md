## Summary

## What This Changes

The PR adds a `message_end` guard in the embedded subscribe handler to suppress block replies for `stopReason: "toolUse"` assistant turns with tool-call blocks, plus regression tests.

## Best Possible Solution

Land one refreshed suppression path that handles all repo-recognized tool-call block shapes for this `message_end` leak, with regression coverage and redacted Telegram/equivalent delivery proof, or fold it into an approved broader channel-leak fix.

## Reproduction Assessment

Yes. Current main can route an unphased assistant message with `stopReason: "toolUse"`, visible text, a tool-call block, and `blockReplyBreak: "message_end"` into `onBlockReply`.

## Solution Assessment

No, not as-is. The narrow guard matches the reported path, but it should use the repo’s full tool-block shape contract and still needs real Telegram/equivalent proof before merge.
