## Summary

Keep this open: current main still lacks a Slack progress visibility policy, the source-level disclosure path is clear, and the protected maintainer label plus security/default-config choices require human product/security direction.

## What This Changes

The issue requests a Slack-safe `streaming.progress` visibility mode that hides raw commands, paths, tool args, approval text, and patch filenames while preserving safe progress/status output.

## Best Possible Solution

Define one typed disclosure policy at the structured progress boundary, choose a conservative Slack default with explicit trusted debug opt-in, and cover native task-card plus text/rich Slack progress with upgrade-aware tests.

## Reproduction Assessment

Yes, at source level: current main routes progress detail and fallback text into Slack-visible task titles/fields, and the config schema has no visibility mode. I did not run a live Slack workspace reproduction.

## Solution Assessment

No, not as a ready implementation plan: the secure-by-default direction is right, but maintainers still need to choose the smallest durable disclosure model, Slack default, and backward mapping for `commandText`.
