## Label Justifications

- `P2`: This is a bounded but meaningful security-hardening and usability improvement for shared Slack conversations, without evidence of an emergency or active outage.
- `impact:security`: Current Slack progress can disclose commands, filesystem paths, tool arguments, and operational execution details to channel participants.

## Summary

Current main still exposes detailed progress metadata through Slack task cards and legacy progress rendering, with no existing visibility policy or canonical replacement item. This protected issue should remain open for maintainer product and security direction on the shared policy, default, and compatibility behavior.

## What This Changes

Add Slack progress visibility levels that retain safe status and reasoning while hiding commands, paths, arguments, approval strings, and tool internals by default.

## Best Possible Solution

Define a typed progress-sensitivity policy at the shared formatting boundary, apply it consistently to Slack native and legacy renderers, default shared Slack surfaces conservatively, and preserve detailed operational output only through explicit opt-in.

## Reproduction Assessment

Yes, at source level. Current main constructs sensitive tool and patch details, Slack renders them into visible task titles, and an existing test preserves a local path; a live Slack scenario was not run.

## Solution Assessment

The requested outcome is appropriate, but the exact Slack-only enum is not yet proven to be the best design. A shared typed sensitivity policy feeding both Slack rendering paths is the narrower and more maintainable boundary.
