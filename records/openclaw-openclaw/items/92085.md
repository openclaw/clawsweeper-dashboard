## Summary

Keep open: current main and the latest release still lack a Slack-safe progress visibility policy, and the issue has a protected `maintainer` label plus a security-sensitive config/default decision. The source path still carries progress details into Slack-visible native task titles and rich/text progress, so this remains necessary maintainer work.

## What This Changes

The issue requests a Slack progress visibility mode that defaults shared Slack progress/task-card UI to safe, high-level status instead of exposing commands, paths, tool args, or tool internals.

## Best Possible Solution

Define a typed Slack disclosure policy at the structured progress boundary, default Slack to safe high-level progress, keep detailed debug output explicitly opt-in, and cover native task cards plus rich/text progress with schema, docs, upgrade, and renderer tests.

## Reproduction Assessment

Yes. Source inspection on current main shows progress details and fallback text flowing into Slack-visible native task titles and rich/text progress while config has no visibility level; I did not run a live Slack workspace in this read-only review.

## Solution Assessment

Unclear until maintainers choose the permanent disclosure model, default, and compatibility behavior. The best implementation direction is a policy at the structured formatter boundary, not Slack renderer regex redaction after sensitive and safe text have already been mixed.
