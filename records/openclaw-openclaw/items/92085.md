## Summary

Current main exposes structured tool, patch, path, and fallback progress text through Slack-visible progress renderers, and no canonical issue or PR owns the gap. The request fits OpenClaw’s safe-default direction, but the protected maintainer signal and overlap with existing progress-detail controls require an explicit product and compatibility decision.

## What This Changes

Add a Slack progress-visibility policy that keeps useful status or reasoning updates while suppressing commands, paths, arguments, and tool internals unless detailed output is explicitly enabled.

## Best Possible Solution

Define one typed, structured progress-disclosure policy before Slack encoding, reuse or narrowly extend the existing detail contract, and apply it consistently to native cards and legacy renderers with detailed output as an explicit opt-in.

## Reproduction Assessment

Yes at the source level: current-main Slack task cards and legacy progress consume structured detail or fallback text, while existing redaction is command-specific. No live Slack run was performed, but the source and tests establish the disclosure path with high confidence.

## Solution Assessment

No, not yet as fully specified: the proposed four-level Slack enum may duplicate the agent-level detail policy. Maintainers should first define one disclosure contract and add only distinctions the current contract cannot express.
