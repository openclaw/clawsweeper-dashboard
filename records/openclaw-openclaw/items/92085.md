## Summary

The issue remains necessary: current main has no safe-but-visible Slack progress mode, and shared progress payloads can carry tool, command, approval, patch, and path details into Slack-native task titles. It requires maintainer judgment because the security policy introduces a compatibility-sensitive config and default surface.

## What This Changes

Add a configurable Slack progress visibility policy that preserves safe high-level status while suppressing commands, paths, arguments, and tool internals by default.

## Best Possible Solution

Define one typed disclosure policy at the shared progress-line boundary, let Slack select an explicitly approved upgrade-safe default, and make native task cards and legacy Slack progress consume the same sanitized lines.

## Reproduction Assessment

Yes, from source inspection at high confidence: progress-line construction can retain the reported operational details, and Slack native rendering turns those lines into visible task titles. No live Slack run was performed, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

The security goal is appropriate, but the proposed four-level enum is not yet proven to be the best shape. Maintainers should choose the smallest stable disclosure contract, its upgrade behavior, and the shared-versus-Slack ownership boundary first.
