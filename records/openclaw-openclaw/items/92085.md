## Summary

Keep this open: current main still has no Slack-safe progress visibility policy, the source-level disclosure path is clear, and the protected `maintainer` label requires explicit maintainer handling rather than cleanup closure.

## What This Changes

Slack progress rendering is requested to gain a conservative visibility mode that prevents commands, paths, tool args, and similar operational details from appearing in shared Slack task-card/progress UI.

## Best Possible Solution

Define one typed disclosure policy at the structured progress boundary, choose a conservative Slack default with explicit trusted debug opt-in, and cover native task-card plus text/rich Slack progress with fresh-install and upgrade tests.

## Reproduction Assessment

Yes, at source level: current main routes structured progress detail and fallback text into Slack progress drafts/native task titles, and the config schema has no visibility setting. I did not run a live Slack workspace reproduction.

## Solution Assessment

No, not as a ready implementation plan: the secure-by-default direction is right, but maintainers still need to choose the smallest durable disclosure model, Slack default, and backward mapping for `commandText`.
