## Summary

Current main has a concrete Slack-visible disclosure gap, no duplicate or existing complete safety mode, and the issue is protected for maintainer judgment; it should remain open while maintainers choose the canonical disclosure contract and upgrade-safe default.

## What This Changes

Requests a Slack-specific progress visibility policy that preserves safe status updates while suppressing commands, paths, arguments, and tool internals by default.

## Best Possible Solution

Define one typed disclosure policy at shared progress-line construction, let Slack select a conservative default for shared destinations, retain an explicit detailed opt-in, and verify native task cards plus legacy rich/text rendering consume the same sanitized lines.

## Reproduction Assessment

Yes at source level: approval commands and patch paths enter progress details, and Slack native task titles render those details directly; this review did not run a live Slack workspace reproduction.

## Solution Assessment

No, not exactly as proposed: the safety goal is correct, but a four-level Slack-only enum risks duplicating the existing explain/raw tool-detail control; maintainers should first choose one canonical prepared disclosure contract and then expose only the necessary Slack policy surface.
