## Summary

Keep open: current main still lacks a Slack-safe progress visibility policy, Slack rich/native progress still renders line details or fallback text into visible Slack UI, and the protected maintainer label plus security-sensitive default/config choice require maintainer review.

## What This Changes

The issue requests a Slack-specific progress visibility mode that keeps shared Slack progress/task-card output from exposing raw commands, paths, tool args, and operational details by default.

## Best Possible Solution

Define a typed disclosure policy at the structured progress boundary, make Slack safe by default with explicit debug opt-in, and cover native task cards plus rich/text progress with schema, docs, fresh-install, and upgrade tests.

## Reproduction Assessment

Yes, at source level: current main routes progress detail and fallback text into Slack-visible native task titles and rich/text progress while config has no visibility level. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation plan: a typed Slack disclosure mode is likely the right direction, but maintainers still need to choose the durable enum/default and backward mapping for existing `commandText` behavior.
