## Summary

Current main and v2026.6.6 still lack a Slack-safe visibility policy: non-command tool metadata remains eligible for progress output, and Slack native task cards intentionally turn line details—including local-looking paths—into user-visible titles. The issue uniquely tracks this unresolved gap and has a protected maintainer label, so it needs an explicit compatibility and security-default decision.

## What This Changes

Add a Slack-safe progress visibility mode that preserves high-level status or reasoning while suppressing commands, paths, arguments, approval strings, and tool internals unless explicitly enabled.

## Best Possible Solution

Define one canonical progress sanitization contract before Slack rendering, use a conservative Slack default, preserve detailed output behind an explicit trusted opt-in, map or migrate the existing command-detail setting deliberately, and prove native plus legacy Slack paths with focused disclosure tests and live redacted Slack output.

## Reproduction Assessment

Yes at source level: current main deterministically carries non-command tool metadata and progress details into Slack-visible rendering, and the native task-card test confirms local-looking paths are preserved. A live Slack workspace reproduction was not run.

## Solution Assessment

The secure-default objective is the right direction, but the proposed four-mode enum is not yet proven to be the best implementation. Maintainers should first choose the smallest canonical visibility contract, compatibility mapping, and Slack default that cover every rendering path.
