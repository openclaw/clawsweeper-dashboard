## Summary

Current main has a concrete Slack disclosure path: structured progress details can become native task-card titles, while existing controls only suppress command text or all tool progress. The safe-default goal fits project direction, but this protected item needs an explicit product and security decision on the canonical disclosure policy, default, and compatibility behavior.

## What This Changes

Add a Slack-safe progress visibility policy that keeps high-level status or reasoning visible while hiding commands, paths, arguments, and tool internals unless explicitly enabled.

## Best Possible Solution

Define one typed progress-disclosure policy that sanitizes structured progress lines before both Slack native and fallback rendering, with a conservative Slack default and an explicit detailed opt-in after maintainers decide compatibility behavior.

## Reproduction Assessment

Yes at source level: current main passes structured progress detail into Slack native task titles, and a focused test expects a local-looking path to remain visible. No live Slack reproduction was run.

## Solution Assessment

No final design is approved yet. The safe-default goal is sound, but the narrowest maintainable solution is a canonical structured disclosure policy shared by Slack native and fallback renderers, not an isolated task-card text filter.

## Risks / Open Questions

- A Slack-only visibility enum could duplicate generic progress-presentation policy and let native cards, rich blocks, and text fallback diverge.
- Changing the default from detailed to safe would intentionally alter existing Slack output for users who rely on visible tool details, so maintainers must decide upgrade and compatibility behavior.
- A string-redaction filter in the Slack renderer would be brittle; structured disclosure policy must prevent sensitive detail from entering every Slack-visible fallback path.
