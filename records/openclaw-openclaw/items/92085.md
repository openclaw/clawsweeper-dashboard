## Label Justifications

- `P2`: This is a bounded but meaningful Slack privacy and security improvement that needs design review rather than emergency response.
- `impact:security`: Slack-visible progress can disclose commands, local filesystem paths, tool arguments, and operational strategy to thread participants.

## Summary

Current main confirms the disclosure gap and no canonical item or shipped/current implementation supplies a Slack-safe visibility policy; this should remain open for maintainer product and security direction because it changes configuration and defaults.

## What This Changes

Add Slack progress visibility levels that retain safe status or reasoning updates while hiding commands, paths, arguments, and tool internals unless detailed output is explicitly enabled.

## Best Possible Solution

Define one typed disclosure policy at the shared structured-progress boundary, let all Slack renderers consume sanitized facts, and choose a conservative Slack default plus explicit trusted debug opt-in with fresh-install and upgrade coverage.

## Reproduction Assessment

Yes, at source level: current main has a direct path from non-command tool details and item metadata into Slack-visible progress titles, and tests preserve local-looking path text. This review did not run a live Slack workspace reproduction.

## Solution Assessment

No, not yet as specified: the security outcome is appropriate, but maintainers should choose the smallest canonical disclosure model, ownership boundary, default, and backward mapping instead of adopting the example four-level enum verbatim.
