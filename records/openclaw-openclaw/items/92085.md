## Label Justifications

- `P2`: The issue identifies a concrete but currently main-only Slack disclosure-hardening gap with bounded channel impact and no demonstrated emergency exposure.
- `impact:security`: Slack-visible progress can disclose local paths, command details, tool arguments, and operational strategy to thread participants.

## Summary

Current main confirms a real Slack disclosure surface: progress metadata can carry commands and changed-file paths into rich progress and native task titles, while existing controls either suppress all tool lines or only hide command text. The issue needs explicit maintainer design because it has a protected `maintainer` label and proposes a compatibility-sensitive configuration/default change; the four-mode enum is plausible, but a smaller canonical safe-versus-debug projection may be preferable.

## What This Changes

Add a Slack progress visibility setting that keeps high-level status visible while hiding commands, paths, arguments, approval strings, and tool internals by default.

## Best Possible Solution

Define a single typed safe progress projection before Slack rendering that preserves reasoning and coarse status while excluding commands, arguments, local or patch paths, approval strings, and tool internals; retain detailed output only through an explicit trusted/debug choice and map existing controls without creating parallel policy.

## Reproduction Assessment

Yes, at source level: patch filenames and formatted tool details flow into Slack native task titles and legacy progress rendering. This review did not perform a live Slack run, so the exact appearance of every claimed detail class remains unverified.

## Solution Assessment

Unclear for the exact proposed enum: the safety objective and affected paths are valid, but maintainers should first decide whether the narrowest durable contract is a Slack visibility setting or a stricter shared `explain` versus `raw` projection plus Slack line-class filtering.
