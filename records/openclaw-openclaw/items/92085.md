## Summary

Keep this open: current main and the latest release still render detailed progress metadata into Slack-visible progress, while the protected maintainer label and security-sensitive default/config choices require explicit product and security direction.

## What This Changes

Add a Slack progress visibility policy that preserves safe reasoning/status updates while suppressing commands, paths, tool arguments, and operational details unless explicitly enabled.

## Best Possible Solution

Define one Slack-owned visibility contract over typed progress facts, default shared Slack output to non-sensitive status summaries, preserve current detailed rendering through an explicit trusted opt-in, and apply the policy consistently to native task cards and legacy progress rendering with focused upgrade tests.

## Reproduction Assessment

Yes, at source level with high confidence: detailed progress fields flow directly into both Slack native task titles and legacy progress blocks, and current configuration provides no visibility classification.

## Solution Assessment

The safety-policy direction is appropriate, but the suggested mode set is not yet proven to be the narrowest solution; maintainers must choose the default, compatibility mapping, and Slack-versus-generic ownership boundary.
