## Summary

Keep open: current main still has the reported silent-success path because the gateway returns `restart.coalesced` metadata, but the Control UI treats an `ok` `update.run` response as success before surfacing any coalesced-restart state.

## What This Changes

The issue asks for the Control UI Update button to show progress or an actionable message when `update.run` coalesces with an already in-flight restart.

## Best Possible Solution

Surface `restart.coalesced` from `update.run` in the Control UI as an actionable pending or stalled restart banner, while leaving backend restart/handoff policy to the narrower update-recovery issues.

## Reproduction Assessment

Yes, source-reproducible: current main returns `restart.coalesced` from `update.run`, while `runUpdate` returns through the ok path before displaying that state. I did not run a live browser update click in this read-only review.

## Solution Assessment

No, the current UI path is not the best solution because it hides an available backend signal. The narrow maintainable fix is to render existing `restart.coalesced` metadata as update-in-progress or recovery guidance without changing restart semantics.
