## Summary

Keep open: current main still has the Control UI feedback gap. The gateway returns structured `restart.coalesced` data for a coalesced `update.run`, but `runUpdate` ignores `restart`, treats `status: "ok"` as success, and clears the busy state without a visible banner.

## What This Changes

The Control UI should visibly report an already in-flight `update.run` instead of leaving the Update button unchanged when the restart request is coalesced.

## Best Possible Solution

Teach the Control UI to treat `restart.coalesced` from `update.run` as a visible in-progress or informational state, with regression coverage, while leaving backend restart coalescing intact.

## Reproduction Assessment

Yes, source-reproducible: mock `update.run` returning `{ ok: true, result: { status: "ok" }, restart: { coalesced: true, delayMs: 0 } }` and current `runUpdate` returns before setting a banner, then clears `updateRunning`. The gateway path still returns that `coalesced: true` shape for an in-flight restart.

## Solution Assessment

Yes: the narrowest maintainable fix is in Control UI response handling because the gateway already exposes and logs the coalesced state. No backend restart policy, config option, or product decision is needed for basic user feedback.
