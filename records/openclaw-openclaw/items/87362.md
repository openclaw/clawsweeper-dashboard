## Summary

Keep open. Current main and v2026.6.8 still do not expose task-flow lifecycle events through plugin hooks, and the linked implementation PR remains open as the candidate path for maintainer API/security/proof review.

## What This Changes

The issue requests plugin-observable task flow lifecycle hook events, with optional diagnostic-event and metrics integration for task flow create, transition, and delete events.

## Best Possible Solution

Review the linked implementation PR as the canonical path, narrow the first landing to a stable sanitized task-flow hook contract, and split tags or diagnostics if those need separate API decisions.

## Reproduction Assessment

Yes, by source inspection: task-flow mutations emit only internal observer events, while plugin hook types expose no task event family and the plugin task-flow runtime is imperative CRUD only. This is a feature gap, not a broken shipped behavior path.

## Solution Assessment

Unclear. A plugin-facing hook or typed runtime event seam fits the direction, but maintainers still need to choose the stable payload, privacy boundary, delivery semantics, and whether diagnostics or tags belong in the same change.
