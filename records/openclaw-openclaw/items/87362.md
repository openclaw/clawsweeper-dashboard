## Summary

Keep open. Current main and the latest release still do not expose task-flow lifecycle events through plugin hooks, and the linked implementation PR is open, conflicting, and still needs maintainer API/product review before this issue can close.

## What This Changes

Request plugin-observable task-flow lifecycle hook events for created, status-transition, and deleted flow records, with optional diagnostics and tagging follow-up.

## Best Possible Solution

Review the linked implementation as the canonical path, narrow the first landing to a stable sanitized task-flow hook contract, and split tags or diagnostics if those need separate API decisions.

## Reproduction Assessment

Yes, by source inspection: task-flow mutations emit only internal observer events, while plugin APIs expose task-flow CRUD and mutation helpers without lifecycle subscription. This is a feature gap rather than a broken existing behavior path.

## Solution Assessment

Unclear. A plugin-facing hook or typed runtime event seam fits the direction, but maintainers still need to choose the stable payload, privacy boundary, delivery semantics, and whether diagnostics/tags belong in the same change.
