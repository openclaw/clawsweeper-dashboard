## Summary

Keep open. Current main and the latest release still lack a task-flow hook family for plugin observability, and the open linked PR remains the implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Request to expose task-flow created, transition, and deleted lifecycle events to plugins through hooks, with optional diagnostic-event and metadata support.

## Best Possible Solution

Review the linked implementation PR as the canonical path, narrow the first landing to a stable sanitized task-flow hook contract, and split tags or diagnostics if those need separate API decisions.

## Reproduction Assessment

Yes, by source inspection: task-flow mutations emit only internal observer events, the hook type system has no task family, and the plugin task-flow runtime has no listener API. This is a feature gap, not a broken shipped behavior path.

## Solution Assessment

Unclear. A plugin-facing hook or typed runtime event seam fits the direction, but maintainers still need to choose the stable payload, privacy boundary, delivery semantics, and whether diagnostics or tags land in the same change.
