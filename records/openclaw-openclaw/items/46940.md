## Summary

Keep open: the reported multi-channel exec-approval routing bug is still source-reproducible on current main, and this PR is useful source work, but the patch is not merge-ready because current outbound context plumbing, handler forwarding, queue replay, test imports, and changelog coverage are incomplete.

## What This Changes

The PR adds target-derived tool context to forwarded exec-approval deliveries and extends outbound delivery tests for forwarded request and resolved notices.

## Best Possible Solution

Land a rebased narrow fix that carries `ChannelThreadingToolContext` from resolved exec-approval targets through outbound delivery params, adapter context, queue storage/recovery replay, focused regression tests, and a single-line changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: current main resolves a forwarding target but delivers without `toolContext`, while channel selection falls back to `toolContext.currentChannelProvider` before throwing the multi-channel required-channel error.

## Solution Assessment

No. The PR direction is right, but it is not the best mergeable fix until it updates the current outbound context type, forwards the value into the channel handler, preserves it through queue replay, fixes the test imports, and adds the required changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Update the current outbound context type:** `src/channels/plugins/types.adapters.ts:100`
  - body: The PR adds `toolContext` to the old adapter-surface declaration, but current main defines `ChannelOutboundContext` in `src/channels/plugins/outbound.types.ts` and `types.adapters.ts` only re-exports it. After rebase, adapters still will not have typed `ctx.toolContext` unless the leaf type is updated there.
  - confidence: 0.9
- **[P1] Import the filesystem helpers:** `src/infra/exec-approval-forwarder.test.ts:382`
  - body: The added regression test calls `fs.mkdtempSync`, `path.join`, `os.tmpdir`, `fs.writeFileSync`, and `fs.rmSync` without adding `node:fs`, `node:path`, and `node:os` imports, so the test file will not typecheck or run.
  - confidence: 0.97
- **[P1] Pass toolContext into the channel handler:** `src/infra/outbound/deliver.ts:210`
  - body: Adding `toolContext` to delivery params is not enough because `deliverOutboundPayloadsCore` still constructs `createChannelHandler` without `toolContext: params.toolContext`. Adapter `sendText` and `sendPayload` contexts will keep receiving `undefined`, so forwarded approvals can still lose the routing context.
  - confidence: 0.95
- **[P2] Persist toolContext through queue replay:** `src/infra/exec-approval-forwarder.ts:345-350`
  - body: Forwarded deliveries are queued before send, and recovery rebuilds delivery params from the stored entry. Without adding `toolContext` to the queued payload, enqueue write, and recovery replay, crash or backoff retries can lose the target-derived channel identity and hit the same routing failure.
  - confidence: 0.9
