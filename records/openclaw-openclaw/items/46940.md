## What This Changes

## Best Possible Solution

Yes, source-reproducible: current main resolves turn-source targets but sends forwarded approval notices without `toolContext`, while downstream message-action routing uses that context to avoid ambiguous multi-channel resolution. I did not run a live multi-channel setup in this read-only review.

## Solution Assessment

No. Passing target-derived context is the right direction, but this branch is not the best mergeable fix until it updates the current type surface, forwards and persists the context, fixes test imports, rebases cleanly, and adds real behavior proof.

Without adding `toolContext` to the queued payload, enqueue write, and recovery replay, crash or retry paths can lose the target-derived channel identity and hit the same multi-channel routing failure.
  - confidence: 0.9
