## Summary

Keep open. The PR targets a real current-main routing gap, and the earlier review discussion correctly caught that the branch is only a partial fix: it does not fully thread `toolContext` through the current outbound adapter contract, handler construction, tests, or queued replay path.

## What This Changes

The PR attempts to add target-derived threading tool context to forwarded exec-approval outbound deliveries and adds regression coverage for forwarded pending and resolved notices.

## Best Possible Solution

Land a rebased narrow fix that carries `ChannelThreadingToolContext` from exec-approval target resolution through outbound delivery params, adapter contexts, and queued recovery replay, with current-main tests for live and recovered forwarded notices.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with multiple message channels configured, forwarded exec-approval delivery has a resolved target but no `toolContext.currentChannelProvider`, while channel selection relies on that fallback before throwing the multi-channel channel-required error.

## Solution Assessment

No. The PR direction is right, but it is not the best mergeable fix until it updates the current outbound type surface, imports/types the new contract, forwards the value into `createChannelHandler`, persists/replays it through the queue, and fixes the added test imports.
