## Summary

Keep this PR open: the reported multi-channel exec-approval routing bug is still valid on current main, but the proposed patch is stale and incomplete against the current outbound type, handler, queue replay, test, and changelog surfaces.

## What This Changes

The PR adds target-derived `toolContext` to forwarded exec-approval deliveries, extends outbound delivery context plumbing, and adds regression coverage for forwarded request and resolved notices.

## Best Possible Solution

Land a rebased narrow fix that carries `ChannelThreadingToolContext` from exec-approval target resolution through outbound delivery params, current adapter context types, handler construction, queued storage/recovery replay, focused tests, and the changelog.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: current main resolves a forwarding target but delivers without `toolContext`, while multi-channel routing relies on channel identity fallback before throwing the channel-required error.

## Solution Assessment

No. The PR direction is right, but it is not the best mergeable fix until it updates the current outbound type surface, forwards the value into the handler, preserves it through queue replay, fixes the added test imports, and adds the required changelog entry.

- **[P1] Update the current outbound context type:** `src/channels/plugins/types.adapters.ts:103`
  - body: The PR adds `toolContext` to the old adapter-surface declaration, but current main defines `ChannelOutboundContext` in `src/channels/plugins/outbound.types.ts` and `types.adapters.ts` only re-exports it.   - body: The added regression test calls `fs.mkdtempSync`, `path.join`, `os.tmpdir`, `fs.writeFileSync`, and `fs.rmSync` without adding `node:fs`, `node:path`, and `node:os` imports, so the test file will not typecheck or run.
  - confidence: 0.97
- **[P1] Pass toolContext into the channel handler:** `src/infra/outbound/deliver.ts:211`
  - body: Adding `toolContext` to delivery params is not enough because `deliverOutboundPayloadsCore` still constructs `createChannelHandler` without `toolContext: params.toolContext`. Adapter `sendText`/`sendPayload` contexts will keep receiving `undefined`, so the forwarded approval can still lose the routing context.
  - confidence: 0.95

  - body: Forwarded deliveries are queued before send, and recovery rebuilds delivery params from the stored entry. Without adding `toolContext` to the queued payload, enqueue write, and recovery replay, crash or backoff retries can lose the target-derived channel identity and hit the same routing failure.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/infra/exec-approval-forwarder.ts:348-351`
  - body: This is a user-facing fix for multi-channel exec-approval routing, so repo policy expects a single-line `CHANGELOG.md` entry. The PR currently changes runtime behavior and tests without documenting the fix.
  - confidence: 0.86
