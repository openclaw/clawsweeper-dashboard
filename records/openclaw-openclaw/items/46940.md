## Summary

Keep open. Current main still lacks the requested forwarded exec-approval tool context, and the PR remains useful prior art but is not mergeable as-is because it leaves adapter propagation and queue replay incomplete and has type/test compile issues.

## What This Changes

The PR adds target-derived tool context to forwarded exec-approval deliveries, extends outbound adapter context typing, and adds regression assertions for pending and resolved forwarded notices.

## Best Possible Solution

Preserve target-derived channel identity as a typed outbound delivery contract from exec-approval forwarding through adapter sends and queued recovery replay, with current-main tests covering live forwarded pending/resolved notices and recovered queued notices.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: in a multi-channel setup, forwarded approval delivery has a resolved target but no `toolContext`, while downstream channel selection relies on `toolContext.currentChannelProvider` before throwing the multi-channel channel-required error.

## Solution Assessment

No. The PR direction is right, but it is not the best mergeable fix until it updates the current outbound type surface, imports/types the new contract, passes the context into `createChannelHandler`, and persists/replays it through the delivery queue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

  - body: The PR adds `toolContext?: ChannelThreadingToolContext` to `ChannelOutboundContext`, but the PR-head file does not import or define `ChannelThreadingToolContext`, so strict TypeScript fails before the runtime fix can be used.
  - confidence: 0.96

  - confidence: 0.97
- **[P1] Forward toolContext into the channel handler:** `src/infra/outbound/deliver.ts:211`
  - body: Adding `toolContext` to delivery params is not enough: `deliverOutboundPayloadsCore` still constructs the channel handler without `toolContext: params.toolContext`, so adapter send contexts continue to receive `undefined`.
  - confidence: 0.95
- **[P2] Persist toolContext through queue replay:** `src/infra/exec-approval-forwarder.ts:348-351`
  - body: Forwarded deliveries are queued before send, and recovery rebuilds delivery params from the stored entry. Without adding `toolContext` to the queued payload, enqueue write, and recovery replay, crash/backoff retries can lose the target-derived channel identity and hit the same routing failure.
  - confidence: 0.9
