## Summary

Keep open: the reported multi-channel exec-approval routing bug still exists on current main, and this PR is the plausible fix direction, but the branch is stale/conflicting, incomplete, and still lacks contributor-owned real behavior proof.

## What This Changes

The PR threads target-derived `toolContext` through forwarded exec-approval outbound delivery and adds focused regression coverage for forwarded request/resolved notices.

## Best Possible Solution

Land a rebased patch that carries `ChannelThreadingToolContext` from resolved approval targets through the active outbound context type, handler params, durable queue storage/replay, and focused tests, with redacted live multi-channel proof.

## Reproduction Assessment

Yes, source-reproducible: current main resolves the forwarded approval target but sends outbound delivery without `toolContext`, while downstream message-action routing uses that context as its channel fallback before the multi-channel selection error. I did not run a live multi-channel setup in this read-only review.

## Solution Assessment

No. Passing target-derived context is the right direction, but this branch is not the best complete fix until it is rebased and carries the context through the active type, handler path, queue replay path, and proof path.

Full review comments:

- **[P1] Update the active outbound context type:** `src/channels/plugins/types.adapters.ts:103`
  - body: Current main defines `ChannelOutboundContext` in `src/channels/plugins/outbound.types.ts`, while `types.adapters.ts` only re-exports the channel types.   - confidence: 0.91

  - body: The added regression test calls `fs.mkdtempSync`, `path.join`, `os.tmpdir`, `fs.writeFileSync`, and `fs.rmSync`, but this file does not import `node:fs`, `node:path`, or `node:os`. The test will fail to typecheck or run as written.
  - confidence: 0.97
- **[P1] Forward toolContext into createChannelHandler:** `src/infra/outbound/deliver.ts:194`
  - body: Adding `toolContext` to the base context builder is not enough because the delivery path still constructs `createChannelHandler` without `toolContext: params.toolContext`. Adapter send contexts will continue receiving `undefined`, so forwarded approvals can still lose routing context.
  - confidence: 0.95
- **[P2] Persist toolContext through queue replay:** `src/infra/outbound/deliver.ts:214`
  - body: `toolContext` becomes part of delivery params, but the write-ahead queue payload, enqueue write, and recovery replay still omit it. Crash, retry, or backoff replay can resend forwarded exec-approval notices without the target-derived channel identity.
  - confidence: 0.9
