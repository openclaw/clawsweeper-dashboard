## Summary

Keep open: current main still lacks the forwarded exec-approval tool-context fix, but this PR is incomplete against current main and missing required real behavior proof.

## What This Changes

The PR adds target-derived `toolContext` to exec-approval forwarded deliveries, extends outbound delivery/channel context typing, and adds regression tests for forwarded request and resolved notices.

## Best Possible Solution

Land a rebased patch that carries `ChannelThreadingToolContext` from resolved approval targets through durable outbound delivery, adapter context, queue storage/replay, and focused tests, with redacted live multi-channel proof.

## Reproduction Assessment

Yes, source-reproducible: current main resolves turn-source approval targets and then calls outbound delivery without `toolContext`, while message-action routing uses that context as the channel/target fallback. I did not run a live multi-channel setup in this read-only review.

## Solution Assessment

No. Passing target-derived context is the right direction, but this PR is not the best mergeable fix until it is rebased and completes the active type, handler propagation, queue replay, and proof coverage.

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Update the active outbound context type:** `src/channels/plugins/types.adapters.ts:100`
  - body: Current main defines `ChannelOutboundContext` in `src/channels/plugins/outbound.types.ts` and `types.adapters.ts` only re-exports it. After rebase, adding `toolContext` to the old inline location will not type adapter `ctx.toolContext`, so the active type surface must be updated instead.

  - body: The added regression test calls `fs.mkdtempSync`, `path.join`, `os.tmpdir`, `fs.writeFileSync`, and `fs.rmSync` without importing `node:fs`, `node:path`, and `node:os`, so the test file will not typecheck or run.
  - confidence: 0.97
- **[P1] Pass toolContext into the channel handler:** `src/infra/outbound/deliver.ts:190`
  - body: Adding `toolContext` to the base context builder is not enough because `deliverOutboundPayloadsCore` still constructs `createChannelHandler` without `toolContext: params.toolContext`. Adapter `sendText` and `sendPayload` contexts will keep receiving `undefined`, so forwarded approvals can still lose the routing context.
  - confidence: 0.95
- **[P2] Persist toolContext through queue replay:** `src/infra/outbound/deliver.ts:210`
  - body: `toolContext` becomes part of delivery params, but the write-ahead queue payload, enqueue write, and recovery replay still omit it. Crash or retry paths can resend forwarded exec-approval notices without the target-derived channel identity and hit the same multi-channel routing failure.
  - confidence: 0.9
