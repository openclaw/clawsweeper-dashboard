## Summary

Keep open: current main still drops the context this PR is trying to preserve, but the external PR is not mergeable as-is because it is conflicting, has blocking propagation/test gaps, and has only test validation rather than real behavior proof.

## What This Changes

The PR adds target-derived `toolContext` to exec-approval forwarded deliveries, extends outbound delivery/context typing, and adds regression coverage for forwarded approval notices.

## Best Possible Solution

Land a rebased patch that carries `ChannelThreadingToolContext` from resolved approval targets through the active outbound type, handler params, durable queue storage/replay, and focused tests, with redacted live multi-channel proof.

## Reproduction Assessment

Yes, source-reproducible: current main resolves the approval target and calls outbound delivery without `toolContext`, while downstream routing uses that context as the channel fallback. I did not run a live multi-channel setup in this read-only review.

## Solution Assessment

No. Passing target-derived context is the right direction, but this PR is not the best mergeable fix until it is rebased and completes active type, handler, queue replay, test import, and real-proof coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Update the active outbound context type:** `src/channels/plugins/types.adapters.ts:100`
  - body: Current main defines `ChannelOutboundContext` in `src/channels/plugins/outbound.types.ts` and `types.adapters.ts` only re-exports it. After rebase, adding `toolContext` to this stale inline location will not type adapter `ctx.toolContext`, so the active type surface must be updated instead.
  - confidence: 0.91
- **[P1] Import the filesystem helpers:** `src/infra/exec-approval-forwarder.test.ts:382`
  - body: The added regression test calls `fs.mkdtempSync`, `path.join`, `os.tmpdir`, `fs.writeFileSync`, and `fs.rmSync` without importing `node:fs`, `node:path`, and `node:os`, so the test file will not typecheck or run.
  - confidence: 0.97
- **[P1] Pass toolContext into the channel handler:** `src/infra/outbound/deliver.ts:191`
  - body: Adding `toolContext` to the base context builder is not enough because `deliverOutboundPayloadsCore` still constructs `createChannelHandler` without `toolContext: params.toolContext`. Adapter `sendText` and `sendPayload` contexts will keep receiving `undefined`, so forwarded approvals can still lose the routing context.
  - confidence: 0.95
- **[P2] Persist toolContext through delivery queue replay:** `src/infra/outbound/deliver.ts:211`
  - body: `toolContext` becomes part of delivery params, but the write-ahead queue payload, enqueue write, and recovery replay still omit it. Crash or retry paths can resend forwarded exec-approval notices without the target-derived channel identity and hit the same multi-channel routing failure.
  - confidence: 0.9
