## Summary

Keep open: the reported multi-channel exec-approval follow-up bug is source-reproducible, but this PR is not mergeable as-is because it misses current-main type/context/queue plumbing, has a test import breakage, and still lacks contributor-owned real behavior proof.

## What This Changes

The PR adds target-derived `toolContext` to exec-approval forwarded deliveries, extends outbound delivery/channel context typing, and adds regression tests for forwarded request and resolved notices.

## Best Possible Solution

Land a rebased patch that carries `ChannelThreadingToolContext` from resolved approval targets through durable outbound delivery, adapter context, queue storage/replay, and focused tests, with redacted live multi-channel proof.

## Reproduction Assessment

Yes, source-reproducible: current main resolves turn-source approval targets and then calls outbound delivery without `toolContext`, while message-action routing uses that context as the channel/target fallback. I did not run a live multi-channel setup in this read-only review.

## Solution Assessment

No. Passing target-derived context is the right direction, but this PR is not the best mergeable fix until it is rebased and completes active type, handler, queue replay, and proof coverage.

  - body: Current main defines `ChannelOutboundContext` in `src/channels/plugins/outbound.types.ts` and `types.adapters.ts` only re-exports it. After rebase, adding `toolContext` to the old inline location will not type adapter `ctx.toolContext`, so the active type surface must be updated instead.

  - confidence: 0.97

  - confidence: 0.95

  - body: `toolContext` becomes part of delivery params, but the write-ahead queue payload, enqueue write, and recovery replay still omit it. Crash or retry paths can resend forwarded exec-approval notices without the target-derived channel identity and hit the same multi-channel routing failure.
  - confidence: 0.9
