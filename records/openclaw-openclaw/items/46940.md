## Summary

Keep this PR open: it targets a real source-reproducible multi-channel exec-approval routing bug, but the branch is not mergeable because it is stale against the active outbound type surface, does not fully propagate the new context, drops it on queue replay, and still lacks contributor-owned real behavior proof.

## What This Changes

The PR adds target-derived tool context to forwarded exec-approval outbound notices and adds focused regression coverage.

## Best Possible Solution

Land a rebased patch that carries `ChannelThreadingToolContext` from resolved exec-approval targets through delivery params, the active adapter context type, handler construction, durable queue storage/recovery, and focused tests, with redacted live multi-channel proof.

## Reproduction Assessment

Yes, source-reproducible: current main resolves a concrete turn-source exec-approval target, then sends forwarded notices without `toolContext`, while message-action routing uses that context to avoid ambiguous channel and target inference. I did not run a live multi-channel setup in this read-only review.

## Solution Assessment

No. Passing target-derived context is the right direction, but this PR is not the best mergeable fix until it updates the active type surface, forwards and persists the context, fixes the test imports, rebases cleanly, and adds real behavior proof.
