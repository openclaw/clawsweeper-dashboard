## What This Changes

The PR adds target-derived `toolContext` to forwarded exec-approval deliveries and adds focused regression tests for forwarded request/resolved notices.

## Best Possible Solution

Land a rebased fix that carries `ChannelThreadingToolContext` from target resolution through outbound params, adapter context, queue storage/recovery replay, focused tests, and maintainer-owned changelog handling.

## Reproduction Assessment

Yes. Current main resolves a concrete exec-approval forward target, then delivers without `toolContext`; downstream multi-channel selection has a `toolContext.currentChannelProvider` fallback before throwing the channel-required error.

## Solution Assessment

No. The PR direction is narrow, but it is not the best mergeable fix until it updates the current type surface, forwards and persists the value end to end, fixes the added test imports, and supplies real behavior proof.

Full review comments:

- **[P1] Update the current outbound context type:** `src/channels/plugins/types.adapters.ts:103`
  - body: The PR adds `toolContext` to the old adapter-surface declaration, but current main defines `ChannelOutboundContext` in `src/channels/plugins/outbound.types.ts` and `types.adapters.ts` only re-exports it. After rebase, adapters still will not have typed `ctx.toolContext` unless the actual leaf type is updated there.
  - confidence: 0.91

  - body: The added regression test calls `fs.mkdtempSync`, `path.join`, `os.tmpdir`, `fs.writeFileSync`, and `fs.rmSync` without adding `node:fs`, `node:path`, and `node:os` imports, so the test file will not typecheck or run.
  - confidence: 0.97
- **[P1] Pass toolContext into the channel handler:** `src/infra/outbound/deliver.ts:211`
  - body: Adding `toolContext` to delivery params is not enough because `deliverOutboundPayloadsCore` still constructs `createChannelHandler` without `toolContext: params.toolContext`. Adapter `sendText`/`sendPayload` contexts will keep receiving `undefined`, so forwarded approvals can still lose the routing context.
  - confidence: 0.95
- **[P2] Persist toolContext through queue replay:** `src/infra/exec-approval-forwarder.ts:348-351`
  - body: Forwarded deliveries are queued before send, and recovery rebuilds delivery params from the stored entry. Without adding `toolContext` to the queued payload, enqueue write, and recovery replay, crash or backoff retries can lose the target-derived channel identity and hit the same routing failure.
  - confidence: 0.9
