## Summary

Keep open: the multi-channel exec-approval routing bug is source-reproducible and the PR is directionally useful, but the branch is not mergeable because it misses active type, handler, and queue replay paths and lacks contributor-owned real behavior proof.

## What This Changes

The PR adds target-derived `toolContext` to forwarded exec-approval deliveries, extends outbound delivery/context plumbing, and adds focused forwarder regression tests.

## Best Possible Solution

Land a rebased patch that carries `ChannelThreadingToolContext` from resolved exec-approval targets through the active outbound context type, handler construction, durable queue storage/replay, and focused tests, with redacted live multi-channel proof.

## Reproduction Assessment

Yes, source-reproducible: current main resolves a concrete turn-source exec-approval target, then forwards notices without `toolContext`, while downstream message-action routing uses that context to infer channel and target. I did not run a live multi-channel setup in this read-only review.

## Solution Assessment

No. Passing target-derived context is the right direction, but this PR is not the best mergeable fix until it updates the active type surface, forwards and persists the context, fixes test imports, rebases cleanly, and adds real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Update the active outbound context type:** `src/channels/plugins/types.adapters.ts:100`
  - body: Current main defines `ChannelOutboundContext` in `src/channels/plugins/outbound.types.ts` and `types.adapters.ts` only re-exports it. After rebase, adding `toolContext` to the old inline location will not type adapter `ctx.toolContext`, so the active type surface must be updated instead.
  - confidence: 0.91
- **[P1] Import the filesystem helpers:** `src/infra/exec-approval-forwarder.test.ts:382`
  - body: The added regression test calls `fs.mkdtempSync`, `path.join`, `os.tmpdir`, `fs.writeFileSync`, and `fs.rmSync` without importing `node:fs`, `node:path`, and `node:os`, so the test file will not typecheck or run.
  - confidence: 0.97
- **[P1] Pass toolContext into the channel handler:** `src/infra/outbound/deliver.ts:190`
  - body: Adding `toolContext` to the context builder is not enough because `deliverOutboundPayloadsCore` still constructs `createChannelHandler` without `toolContext: params.toolContext`. Adapter `sendText` and `sendPayload` contexts will keep receiving `undefined`, so forwarded approvals can still lose the routing context.
  - confidence: 0.95
- **[P2] Persist toolContext through queue replay:** `src/infra/outbound/deliver.ts:210`
  - body: `toolContext` is now part of delivery params, but the write-ahead queue payload, enqueue write, and recovery replay still omit it. Crash or retry paths can resend forwarded exec-approval notices without the target-derived channel identity and hit the same multi-channel routing failure.
  - confidence: 0.9
