  - body: This adds `toolContext` to the old inline adapter declaration, but current main defines `ChannelOutboundContext` in `src/channels/plugins/outbound.types.ts` and `types.adapters.ts` only re-exports it.   - body: The added regression test calls `fs.mkdtempSync`, `path.join`, `os.tmpdir`, `fs.writeFileSync`, and `fs.rmSync` without importing `node:fs`, `node:path`, and `node:os`, so the test file will not typecheck or run.
  - confidence: 0.97
- **[P1] Pass toolContext into the channel handler:** `src/infra/outbound/deliver.ts:214`
  - body: Adding `toolContext` to delivery params is not enough because `deliverOutboundPayloadsCore` still constructs `createChannelHandler` without `toolContext: params.toolContext`. Adapter `sendText` and `sendPayload` contexts will keep receiving `undefined`, so forwarded approvals can still lose the routing context.
  - confidence: 0.95
- **[P2] Persist toolContext through queue replay:** `src/infra/exec-approval-forwarder.ts:348-351`
  - body: Forwarded deliveries are queued before send, and recovery rebuilds delivery params from the stored entry. Without adding `toolContext` to the queued payload, enqueue write, and recovery replay, crash or retry paths can lose the target-derived channel identity and hit the same multi-channel routing failure.
  - confidence: 0.9
