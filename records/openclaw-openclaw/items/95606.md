## Summary

Keep open: current main and v2026.6.10 still leave the memory deletion handoff incomplete, and https://github.com/openclaw/openclaw/pull/95929 is an open closing PR that should be reviewed or rejected before this issue is closed.

## What This Changes

The issue reports that stale or conflicting LanceDB memories cannot be precisely removed because recall output exposes display numbers while deletion requires UUIDs, and query-based forget can return no candidates.

## Best Possible Solution

Review and either land or close https://github.com/openclaw/openclaw/pull/95929 as the canonical deletion-handoff fix, while keeping automatic conflict replacement as a separate maintainer product decision.

## Reproduction Assessment

Yes for the source-level selector gap: current main formats recall as numbered lines while exact deletion rejects non-UUID IDs. I did not run a live LanceDB embedding repro for the exact query no-match symptom.

## Solution Assessment

Unclear for the full request because automatic replacement/deprecation needs product direction. For the narrow bug, a model-visible UUID in recall output plus a safe query fallback is more maintainable than accepting transient display indexes as deletion IDs.
