## Summary

Keep open: current main and the latest release resolve the older full-UUID candidate-list gap, but they do not fully solve this report because `memory_recall` still shows only numeric list items in model-visible content while targeted deletion requires a full UUID, and automatic replacement of conflicting facts remains undecided.

## What This Changes

Request to make long-term memory cleanup precise so stale or conflicting memories can be deleted, replaced, or listed with deletable identifiers.

## Best Possible Solution

Define the plugin-owned memory-lancedb cleanup contract, then implement one model-visible exact deletion or replace path with focused tests and matching docs.

## Reproduction Assessment

Unclear for the exact no-match query path: I did not run a live LanceDB repro, and that part depends on embeddings and scores. The visible-selector gap is source-reproducible because recall content lists numeric entries, details are stripped before model replay, and forget rejects numeric IDs.

## Solution Assessment

Unclear. Accepting recall indices is not obviously the best fix because it needs scoped result state; a model-visible deletion selector or maintainer-approved replace flow is likely safer, but the product choice should come first.
