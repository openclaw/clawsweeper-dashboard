## Summary

Keep open: current main and v2026.6.9 still leave the recall-to-forget handoff and conflict cleanup semantics unresolved. The merged candidate-list fix makes `memory_forget(query=...)` show full UUIDs, but `memory_recall` still shows display numbers while exact deletion requires a UUID, and automatic replacement/deprecation remains a memory product-contract decision.

## What This Changes

The issue asks for a reliable way to delete or supersede stale LanceDB-backed memories when newer conflicting memories are stored.

## Best Possible Solution

Define one memory cleanup contract in the active memory plugin: expose a model-visible exact deletion selector from recall, and handle automatic replacement/deprecation through the broader Memory MVP conflict resolver rather than an ad hoc LanceDB-only rule.

## Reproduction Assessment

Yes for the source-level recall-to-forget gap: current main formats recall output with display numbers and details-only UUIDs, while exact deletion rejects non-UUID IDs. The exact query no-match symptom was not live-reproduced against LanceDB.

## Solution Assessment

Unclear: accepting numeric recall indices is probably not the best durable API because indices are transient. A model-visible UUID or deletion handle plus a separate conflict/replacement resolver is safer, but maintainers need to choose the contract.
