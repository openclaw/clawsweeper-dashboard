## Summary

Keep open: current main still leaves a real memory cleanup gap. The prior fix made `memory_forget(query=...)` candidate UUIDs visible, but `memory_recall` still shows numbered entries while exact deletion requires a UUID, and automatic replacement of conflicting facts remains an unresolved product/API decision.

## What This Changes

The issue asks for stale or incorrect built-in LanceDB memories to be precisely deletable or automatically replaced when corrected.

## Best Possible Solution

Define one supported LanceDB memory cleanup contract: either expose a model-visible exact deletion selector from recall, or add deliberate non-destructive replacement/deprecation semantics with focused tests and docs.

## Reproduction Assessment

Yes for the visible-selector gap: current source shows recall output lists numbered memories while exact deletion rejects non-UUID IDs. No live LanceDB run was performed for the exact query no-match symptom.

## Solution Assessment

Unclear. Accepting numeric recall indices would require scoped result state, while exposing UUIDs in recall or adding replacement/deprecation semantics has different product and safety tradeoffs.
