## Summary

Keep open: current main and v2026.6.9 fixed the older `memory_forget` candidate-list UUID problem, but `memory_recall` still shows only numbered model-visible results while the deletable UUIDs are only in stripped `details`; conflicting stores are also only blocked as near-exact duplicates.

## What This Changes

The issue requests a reliable way to delete or supersede stale built-in LanceDB memories so conflicting memory entries do not persist after corrections.

## Best Possible Solution

Land a narrow plugin-owned change that puts a full `memoryId` or equivalent stable delete handle in `memory_recall` model-visible content, while leaving supersede/conflict lifecycle design to the open Memory MVP issues.

## Reproduction Assessment

Yes, by source inspection with medium confidence: `memory_recall` returns visible numbered entries but puts full UUIDs only in `details`, and OpenClaw docs say `details` are stripped before provider replay. I did not run a live LanceDB/tool invocation in this read-only checkout.

## Solution Assessment

Yes for the deletion half: making recall results include a model-visible deletion selector is the narrowest maintainable fix and reuses the existing `memory_forget(memoryId)` path. Automatic replacement, tombstones, and conflict resolution should stay in the broader Memory MVP lifecycle track.
