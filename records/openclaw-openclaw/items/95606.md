## Summary

Keep open: current main and the latest release fixed the older full-UUID candidate-list bug, but they still do not provide a settled way to correct or supersede conflicting memories from recall results. The remaining work is a memory lifecycle/product decision, not conservative cleanup.

## What This Changes

The issue asks for precise stale-memory deletion or replacement semantics when corrected facts leave conflicting memory entries behind.

## Best Possible Solution

Define one plugin-owned memory cleanup contract, likely through the Memory MVP lifecycle work, then implement deletion/edit/supersede behavior with model-visible handles, provenance, docs, and focused tests.

## Reproduction Assessment

Yes for the visible-selector gap: current source shows recall output lists numeric items while exact deletion accepts only UUID memory IDs. The exact query no-match symptom is still unclear without a live LanceDB embedding run against the reporter's data.

## Solution Assessment

Unclear: accepting numeric recall indices would require scoped result state, while exposing UUIDs or adding edit/supersede semantics has different safety and product tradeoffs. The narrowest maintainable solution should come from the memory lifecycle contract rather than another one-off deletion shortcut.
