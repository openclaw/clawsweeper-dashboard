## Summary

Keep open: current main and the latest release still leave the recall-to-forget handoff unresolved. The prior merged fix made `memory_forget(query=...)` candidate lists show full UUIDs, but `memory_recall` still shows numbered entries while exact deletion requires a UUID, and automatic replacement/deprecation of conflicting facts remains a product-contract choice.

## What This Changes

The issue asks for a reliable way to delete stale or conflicting memories, either by making recalled memories deletable, accepting recall selectors, listing deletable IDs, or defining replacement/deprecation semantics.

## Best Possible Solution

Define one memory cleanup contract in the owning memory plugin: expose a model-visible exact deletion handle from recall, add a deliberate list/delete selector flow, or separately design replacement/deprecation semantics with focused tests and docs.

## Reproduction Assessment

Yes for the source-level selector gap: current main shows numbered recall entries while exact deletion requires UUID-formatted IDs. The exact LanceDB semantic-query no-match symptom still needs live reproduction.

## Solution Assessment

Unclear. Exposing UUIDs in recall is the narrowest likely repair for precise deletion, but accepting numeric indices or auto-replacing conflicting facts would create broader state and product semantics.
