## Summary

Keep open: current main still leaves a concrete memory cleanup gap. The query-based forget path can show full UUID candidates, but recall still presents display numbers while exact deletion requires a UUID, and automatic replacement/deprecation of conflicting facts remains an unresolved memory API decision.

## What This Changes

The issue asks for stale or conflicting built-in LanceDB memory entries to be precisely deletable, replaced, or deprecated after a corrected fact is stored.

## Best Possible Solution

Expose a stable model-visible deletion handle from recall or add a supported list-with-IDs flow, while handling automatic replacement/deprecation through the broader memory review/write-pipeline product track.

## Reproduction Assessment

Yes for the source-level recall-to-forget gap: recall still returns numbered model-visible entries while exact forget expects UUID-shaped handles. No live provider reproduction was established for the exact query no-match symptom.

## Solution Assessment

Unclear as a complete fix: exposing stable recall IDs is a narrow maintainable repair, but numeric index deletion and automatic replacement/deprecation need maintainer agreement on memory semantics.
