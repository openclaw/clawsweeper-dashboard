## Summary

Keep open: current main still lacks one supported cleanup contract for this report. The previous full-UUID fix helps `memory_forget(query=...)` candidate selection, but `memory_recall` still shows numbered entries while targeted deletion requires UUIDs, and corrected-fact replacement remains an unresolved memory product decision.

## What This Changes

The issue asks for precise deletion or replacement of stale memory entries so corrected facts do not persist alongside old ones.

## Best Possible Solution

Define one supported memory cleanup contract, then implement it in the active memory plugin with model-visible exact deletion or replacement semantics, non-destructive conflict handling where appropriate, focused tests, and public docs.

## Reproduction Assessment

Yes for the source-level selector gap: current code shows recall output has numbered visible entries while precise deletion accepts only UUIDs. The exact query no-match example still needs live LanceDB/embedding reproduction.

## Solution Assessment

Unclear as a final product direction. Accepting numeric recall indices would require scoped result state, while exposing UUIDs, adding a list command, or adding replacement/tombstone semantics have different safety and UX tradeoffs.
