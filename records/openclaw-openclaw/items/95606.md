## Summary

Keep open: current main still leaves a real memory cleanup gap. The merged candidate-list fix makes `memory_forget(query=...)` show full UUIDs, but `memory_recall` still shows numbered entries while exact deletion requires a UUID, and automatic replacement or deprecation of conflicting facts remains an unresolved memory API/product decision.

## What This Changes

The issue requests precise deletion, replacement, or deprecation of stale/conflicting LanceDB memories after a corrected fact is stored.

## Best Possible Solution

Define one plugin-owned LanceDB memory cleanup contract: expose a model-visible exact deletion selector from recall or add deliberate replacement/deprecation/conflict semantics with focused tests and docs.

## Reproduction Assessment

Yes for the source-level selector gap: current recall output is numbered while exact deletion rejects non-UUID IDs. The exact query no-match symptom was not live-reproduced against LanceDB.

## Solution Assessment

Unclear. Accepting numeric recall indices would require scoped result state, while exposing UUIDs in recall or adding replacement/deprecation semantics has different product and safety tradeoffs.
