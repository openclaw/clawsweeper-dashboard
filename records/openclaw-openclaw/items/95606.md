## Summary

Keep open: current main and the latest release still leave the central cleanup UX unresolved. The prior full-UUID fix makes `memory_forget(query=...)` candidates usable, but `memory_recall` still shows only numbered visible entries while exact deletion requires a UUID, and automatic replacement of conflicting facts remains a product decision.

## What This Changes

The issue asks for a reliable way to remove or replace stale built-in memory facts so conflicting recall results do not persist.

## Best Possible Solution

Define one supported memory cleanup contract, then implement it in `memory-lancedb` with model-visible exact deletion or replacement semantics plus focused docs and tests.

## Reproduction Assessment

Yes for the visible-selector gap: current source shows recall output lists numeric items while precise deletion accepts only UUID memory IDs. No for the exact query no-match symptom because I did not run a live LanceDB embedding reproduction.

## Solution Assessment

Unclear. Accepting numeric recall indices would require scoped result state, while exposing UUIDs or adding an explicit replacement/refresh flow has different safety and compatibility tradeoffs.
