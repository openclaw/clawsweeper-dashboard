## Summary

Keep open: current main and v2026.6.10 still leave the reported memory cleanup gap, and an open linked PR with closing syntax is the candidate implementation rather than a reason to close the issue before merge.

## What This Changes

The issue requests a reliable way to delete stale built-in memory entries after recall or by query so corrected facts do not persist alongside older incorrect ones.

## Best Possible Solution

Review and either land or reject the open linked memory-lancedb fix PR; keep automatic replacement/deprecation semantics as a separate maintainer product decision.

## Reproduction Assessment

Yes at source level: current main shows recall numbers are not UUIDs, exact deletion rejects non-UUID IDs, and forget-by-query can return no match after embedding search. I did not run a live LanceDB agent scenario in this read-only review.

## Solution Assessment

Yes for the narrow selector gap: exposing a deletable UUID in model-visible recall content matches the existing UUID deletion contract. Replacement/deprecation of conflicting facts and broader text fallback behavior need maintainer product judgment.
