## Summary

Keep open: current main and v2026.6.9 still leave the core cleanup UX unresolved. The merged full-UUID candidate fix makes `memory_forget(query=...)` candidates usable, but `memory_recall` still shows numbered rows while exact deletion requires a UUID, and automatic replacement of conflicting facts remains a product decision.

## What This Changes

The issue requests reliable deletion or replacement of stale/conflicting long-term memories when corrected facts persist beside old facts.

## Best Possible Solution

Define one supported memory cleanup contract for the active memory plugin, then implement it with model-visible exact deletion or explicit replacement semantics plus focused docs and regression coverage.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only pass. Source inspection does reproduce the visible-selector gap: recall shows numeric rows, while targeted deletion requires a UUID that is not visible in recall content.

## Solution Assessment

Unclear. Exposing a UUID or opaque deletion handle in recall is a narrower fix for deletion, while semantic replacement or automatic deprecation is a broader product behavior that maintainers should choose deliberately.
