## Summary

Keep open: current main still leaves a real memory cleanup gap. The merged full-UUID candidate-list fix helps `memory_forget(query=...)`, but `memory_recall` still shows numbered model-visible entries while exact deletion requires a UUID, and replacing/deprecating conflicting facts remains an unresolved product/API decision.

## What This Changes

Request to let users reliably delete, replace, or deprecate stale/conflicting long-term memories created through the LanceDB memory tools.

## Best Possible Solution

Define one supported memory cleanup contract for LanceDB memory: a model-visible exact deletion selector and/or deliberate replace/deprecate semantics, with focused tool tests and docs.

## Reproduction Assessment

Yes at source level: current main shows recall entries with numeric labels while exact deletion rejects non-UUID IDs, and recall/forget query thresholds differ. I did not run a live LanceDB reproduction for the specific embedding no-match example.

## Solution Assessment

Unclear. Exposing UUIDs in recall is a narrow repair, but accepting recall indices or replacing/deprecating conflicting facts changes the memory API and needs maintainer direction.
