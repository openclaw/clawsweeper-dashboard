## Summary

Keep open: current main and the latest release still leave a real memory cleanup gap. The merged candidate-list fix makes `memory_forget(query=...)` candidates deletable after a successful query, but `memory_recall` still shows numbered results while exact deletion requires UUIDs, and replacement/deprecation semantics for corrected facts remain a maintainer product/API decision.

## What This Changes

The issue requests a reliable way to delete or replace stale/conflicting long-term memory entries after corrected facts are stored.

## Best Possible Solution

Define one supported memory cleanup contract: expose a model-visible exact deletion selector from recall or add deliberate replacement/deprecation semantics, then cover it with focused memory-lancedb tests and docs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the visible-selector gap: recall output is numbered while exact deletion rejects non-UUID IDs. The exact semantic-query no-match report still needs a live LanceDB run.

## Solution Assessment

Unclear as a final product direction. Accepting numeric recall indices would require scoped result state, while exposing UUIDs in recall or adding replacement/deprecation semantics has different safety and UX tradeoffs.
