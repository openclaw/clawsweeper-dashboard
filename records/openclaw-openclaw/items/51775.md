## Summary

Keep open: current main still has the reported WebChat Stop cleanup gap, but this PR cannot merge as-is because it turns targeted and owner-filtered aborts into session-wide cleanup and has only mocked/unit proof.

## What This Changes

The PR adds embedded Pi abort, session queue clearing, and subagent stopping to both `chat.abort` paths and adds mocked gateway tests.

## Best Possible Solution

Implement scoped `chat.abort` cleanup that only affects authorized aborted targets, preserves run-id and owner isolation, and proves WebChat Stop with focused tests plus redacted runtime evidence.

## Reproduction Assessment

Yes, at source level: WebChat Stop calls `chat.abort`, current main's handler lacks embedded Pi, queue, and subagent cleanup, and `/stop` uses those cleanup primitives. I did not run a live WebChat reproduction in this read-only review.

## Solution Assessment

No. The cleanup goal is valid, but copying session-wide `/stop` cleanup into the run-id and owner-filtered RPC branches is not the narrowest maintainable fix because those branches already have narrower contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep run-id cleanup scoped to the targeted run:** `src/gateway/server-methods/chat.ts:1227-1232`
  - body: After aborting one `runId`, this also aborts the embedded session and clears queues for the whole session. If another prompt is active or queued on the same session, canceling one run can drop unrelated work the caller did not target.
  - confidence: 0.88
- **[P1] Preserve owner isolation for session-key cleanup:** `src/gateway/server-methods/chat.ts:1176-1185`
  - body: The session-key path first aborts only runs authorized for the requesting client, but this cleanup still clears queues, aborts the embedded session, and stops subagents for the whole session key. That bypasses the mixed-owner behavior where another device's run remains active.
  - confidence: 0.86
