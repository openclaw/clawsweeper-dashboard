## Summary

Keep this PR open, but do not merge it as-is. The original stop-button bug is real, and current main already has narrower active-run signal cancellation for agent RPC runs, but this PR adds whole-session cleanup to `chat.abort` paths whose current contracts are run-scoped or owner-scoped.

## What This Changes

The PR adds `abortEmbeddedPiRun`, `clearSessionQueues`, and `stopSubagentsForRequester` calls to both `chat.abort` paths and adds tests for those cleanup calls.

## Best Possible Solution

Keep the current active-run signal cancellation, then add cleanup only behind a contract that proves all affected session work is authorized and intended to stop. For `runId`, cleanup should be run-scoped or limited to the last active run for that session; for sessionKey aborts, cleanup should only cover work owned by the requester or all work when the requester has admin authority, with regression tests for concurrent runs and mixed-owner sessions.

## Reproduction Assessment

Yes. A focused unit reproduction can create two active runs or queued follow-ups in the same session, abort only one run or one device owner's runs, and show that the PR's session-keyed cleanup would clear or stop work outside the authorized target.

## Solution Assessment

No. Mirroring `/stop` directly is not the best fix because `/stop` is a session-targeted command, while `chat.abort(runId)` and owner-filtered session aborts have narrower authorization and scoping semantics.
