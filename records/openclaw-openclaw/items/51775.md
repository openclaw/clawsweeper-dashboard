## Summary

Keep open: the bug is source-reproducible and current main still lacks the embedded Pi/sub-agent/queue cleanup, but this PR’s cleanup is broader than the existing run-id and owner-scoped abort contracts and it still lacks after-fix real behavior proof.

## What This Changes

The PR imports existing embedded Pi abort, sub-agent stop, and queue cleanup helpers into `chat.abort` and adds tests asserting they run for sessionKey and runId aborts.

## Best Possible Solution

Add scoped cleanup to `chat.abort` that stops embedded Pi, sub-agents, and queues only for the authorized aborted target(s), with regression coverage for mixed-owner and same-session concurrent runs plus real webchat/gateway proof.

## Reproduction Assessment

Yes, at source level: current UI/docs route Stop to `chat.abort`, current main lacks the cleanup calls there, and `/stop` uses the embedded Pi/queue/sub-agent cleanup path. I did not establish a live webchat reproduction in this read-only pass.

## Solution Assessment

No. The cleanup goal is valid, but directly mirroring `/stop` in both RPC paths ignores `chat.abort(runId)` scope and the owner-filtered session abort contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep run-id cleanup scoped to the targeted run:** `src/gateway/server-methods/chat.ts:1227-1230`
  - body: After a successful abort by `runId`, this calls `abortEmbeddedPiRun(active.sessionId)` and clears queues for the whole session. If another prompt is active or queued on the same session, canceling one run can abort or drop unrelated work the caller did not target.
  - confidence: 0.88
- **[P1] Preserve owner isolation for session-key cleanup:** `src/gateway/server-methods/chat.ts:1176-1185`
  - body: The session-key path first aborts only runs authorized for the requesting client, but this cleanup still aborts the embedded session, clears queues, and stops sub-agents for the whole session key. That bypasses the mixed-owner behavior where another device's run remains active.
  - confidence: 0.86
