## Summary

Keep open: the underlying Stop-button bug is plausible from source, but the current PR introduces session-wide cleanup side effects that can cross run and owner boundaries, and it still lacks real behavior proof.

## What This Changes

This PR adds embedded Pi abort, sub-agent stop, and session queue cleanup calls to both `chat.abort` RPC paths, with a focused test file for those calls.

## Best Possible Solution

Implement a scoped abort cleanup helper that preserves existing run and owner isolation, only clears session queues when the whole authorized session lane is being stopped, and proves the WebChat Stop flow against an embedded Pi run.

## Reproduction Assessment

Yes from source, though not from a live run in this review. Current main shows `chat.abort` only aborting controller entries while `/stop` also cleans the embedded Pi run, sub-agents, and queues.

## Solution Assessment

No. Mirroring `/stop` cleanup in both RPC paths is not scoped enough because `runId` aborts and mixed-owner session aborts have narrower contracts than a full session stop.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep runId abort cleanup run-scoped:** `src/gateway/server-methods/chat.ts:1232-1234`
  - body: When a caller aborts one `runId`, this added cleanup aborts the shared embedded session and clears the session queue lane. If another prompt is active or queued on the same session, that unrelated work can be stopped or dropped even though only one run was targeted.
  - confidence: 0.9
- **[P1] Preserve owner isolation for session abort cleanup:** `src/gateway/server-methods/chat.ts:1183-1186`
  - body: `abortChatRunsForSessionKeyWithPartials` can abort only the requester-authorized subset and leave other device-owned runs active, but this new block still aborts the embedded session and clears queues for the whole session once any owned run is aborted.
  - confidence: 0.9
