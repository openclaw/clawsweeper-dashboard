## Summary

Keep open: current main still lacks the PR's embedded Pi, queue, and subagent cleanup in `chat.abort`, so the underlying stop-button gap is not fully resolved. The proposed patch is not merge-ready because it applies session-wide cleanup after narrower run-scoped or owner-filtered aborts, which can interrupt unrelated active or queued work in the same session.

## What This Changes

The PR adds embedded Pi abort, session queue clearing, and subagent stopping to both `chat.abort` paths, plus a new Vitest file covering sessionKey cleanup, runId cleanup, and the no-active-run case.

## Best Possible Solution

Keep the active-run abort signal behavior, then add embedded Pi, queue, and subagent cleanup only where the handler can prove it is authorized for all affected session work. For `runId`, use a truly run-scoped cleanup path or gate broader cleanup on last-active-run semantics; for sessionKey aborts, preserve owner isolation or require an admin/all-runs contract.

## Reproduction Assessment

Yes. A focused unit reproduction can create two active runs or queued follow-ups under one session, or two owner devices sharing a session, then abort only one run or one owner; the PR's new cleanup would clear session-wide queues/subagents outside that target.

## Solution Assessment

No. Mirroring `/stop` directly is not the best fix because `/stop` is a session-level command, while `chat.abort(runId)` and owner-filtered session aborts have narrower scoping and authorization semantics.
