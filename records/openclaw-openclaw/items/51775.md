## Summary

Keep open: this is a plausible user-facing abort bug, but the current PR broadens cleanup beyond the targeted/authorized abort scope and lacks required after-fix real behavior proof, so it should not be merged as-is.

## What This Changes

The PR adds embedded Pi abort, sub-agent stop, and queue cleanup calls to both `chat.abort` paths and adds unit tests for those calls.

## Best Possible Solution

Implement scoped cleanup in `chat.abort` that only affects the authorized aborted run/session targets, preserves owner isolation, and includes regression coverage plus real WebChat Stop proof.

## Reproduction Assessment

Yes, at source level: the UI Stop button routes to `chat.abort`, current main lacks embedded Pi/queue/sub-agent cleanup there, and `/stop` uses that cleanup path. I did not run a live WebChat reproduction in this read-only review.

## Solution Assessment

No. The cleanup goal is valid, but the PR mirrors session-level `/stop` cleanup in both RPC paths and ignores the narrower run-id and owner-filtered abort contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep run-id cleanup scoped to the targeted run:** `src/gateway/server-methods/chat.ts:1227-1232`
  - body: After a successful abort by `runId`, this calls `abortEmbeddedPiRun(active.sessionId)` and clears queues for the whole session. If another prompt is active or queued on the same session, canceling one run can abort or drop unrelated work the caller did not target.
  - confidence: 0.88
- **[P1] Preserve owner isolation for session-key cleanup:** `src/gateway/server-methods/chat.ts:1176-1185`
  - body: The session-key path first aborts only runs authorized for the requesting client, but this cleanup still aborts the embedded session, clears queues, and stops sub-agents for the whole session key. That bypasses the mixed-owner behavior where another device's run remains active.
  - confidence: 0.86
