## Summary

Keep open. Current main and latest release still inject active-memory recall through `prependContext` into user-prompt space; the OpenAI-compatible transport uses `promptCacheKey` or `sessionId` rather than hashing the full request body, so the exact key-change claim still needs live provider proof, but the cache-boundary problem remains actionable.

## What This Changes

The issue asks OpenClaw to fix or document active-memory dynamic recall breaking prompt-cache behavior for OpenAI-compatible providers configured with `compat.supportsPromptCacheKey: true`.

## Best Possible Solution

Define an owner-approved cache contract for dynamic active-memory context, then either route recall through an existing cache-aware below-boundary path with tests and live proof or document the OpenAI-compatible limitation.

## Reproduction Assessment

No high-confidence live reproduction yet. Source inspection shows active-memory still returns `prependContext` and both main runners prepend it into user-prompt space, but the OpenAI-compatible cache-hit collapse and changing key value need a real provider trace.

## Solution Assessment

Unclear. A docs warning is the narrow safe mitigation, while any code fix needs a maintainer decision about whether dynamic memory belongs in user prompt space, system prompt space, or an explicit cache-aware contract.
