## Summary

Keep open. Current main still has generic WSL2 remote CDP guidance and generic browser CDP diagnostics, but it does not cover the reported `svchost`/`iphlpsvc` portproxy self-loop that makes port 9222 look alive while Chrome is not behind it; the related diagnostics PR at https://github.com/openclaw/openclaw/pull/42027 is broader, still open, and does not close this issue.

## What This Changes

The issue asks OpenClaw to detect or document the WSL2/Windows CDP failure where `netsh portproxy` self-loops cause `svchost`/`iphlpsvc` to own port 9222 and return empty CDP responses instead of Chrome JSON.

## Best Possible Solution

Add targeted browser doctor/status hints and WSL2 docs for `Empty reply from server` plus `svchost`/`iphlpsvc` portproxy self-loops, while leaving Windows `netsh` cleanup as an explicit user-run recovery step.

## Reproduction Assessment

Yes, for a high-confidence manual path: the report gives exact Windows and WSL2 commands plus before/after evidence, and current-main source/docs inspection confirms the missing diagnostic case. I did not live-run the Windows portproxy setup in this review environment.

## Solution Assessment

Yes, as a narrow diagnostics/docs fix. The maintainable path is targeted hints and regression coverage in the active browser plugin plus the WSL2 guide, not automatic Windows `netsh portproxy` repair.
