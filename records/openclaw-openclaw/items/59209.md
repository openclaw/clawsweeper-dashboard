## Summary

Keep open. Current main still has only generic WSL2 remote CDP troubleshooting and generic browser CDP diagnostics; it does not cover the reported `svchost`/`iphlpsvc` portproxy self-loop that makes port 9222 look alive while Chrome is not behind it.

## What This Changes

The issue requests targeted OpenClaw diagnostics and docs for WSL2 CDP failures caused by a Windows `netsh portproxy` self-loop where `svchost`/`iphlpsvc` owns port 9222 instead of Chrome.

## Best Possible Solution

Add targeted browser diagnostics and WSL2 remote CDP docs for empty `/json/version` responses where port 9222 is owned by `svchost`/`iphlpsvc`, while keeping Windows portproxy cleanup as an explicit user-run recovery step.

Yes for a high-confidence manual path: the report gives exact Windows and WSL2 commands plus before/after evidence. I did not live-run Windows/WSL2 here, but current source/docs inspection confirms OpenClaw still lacks this diagnostic case.

## Solution Assessment
