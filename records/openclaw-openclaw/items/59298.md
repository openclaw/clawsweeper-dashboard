## Summary

Keep this PR open, but it should not merge as-is: the cached `openclaw update hint` helper is plausible, while the user-visible promise depends on fenced shell blocks in `SKILL.md` being executed even though current skill delivery only reads or materializes skill files. The PR also still lacks inspectable real behavior proof for the claimed skill-invocation behavior.

## What This Changes

The PR adds an `openclaw update hint` CLI subcommand backed by cached update state and inserts fenced shell preambles into bundled maintainer skill files.

## Best Possible Solution

If maintainers want this feature, expose the update notice through one shared executed runtime or prelude hook, keeping the cached-state CLI helper only if it is part of that supported path.

## Reproduction Assessment

Yes, source-reproducible: current main's skills prompt and docs load `SKILL.md` by read-on-demand, and the Claude CLI path materializes skill directories as a plugin. Those paths do not execute fenced bash blocks before skill use.

## Solution Assessment

No. The cached helper may be useful, but per-skill Markdown fences are not the narrowest maintainable way to deliver an automatic runtime update notice.
