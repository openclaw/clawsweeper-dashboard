## Summary

Keep open: the PR targets a real ACP command-routing gap, but the current patch bypasses shared command gating in ACPX and lacks external real-behavior proof required before merge.

## What This Changes

The PR wraps ACPX `reply_dispatch` to skip ACP runtime dispatch for `/acp`, `/focus`, and `/unfocus`, adds ACPX unit tests, and adds a changelog entry.

## Best Possible Solution

Extend the shared `shouldBypassAcpDispatchForCommand` helper to recognize `/focus` under the existing local-command gate, keep ACPX on the shared hook, and add central regression coverage for `/focus` plus disabled text-command cases.

## Reproduction Assessment

Yes, at source level: current main routes ACPX through the shared ACP dispatch hook, and that shared bypass does not recognize documented `/focus` commands. I did not run a live ACP-bound chat session in this read-only review.

## Solution Assessment

No. The PR likely fixes the happy path, but the ACPX-local wrapper skips the shared `shouldHandleTextCommands` gate; the narrower maintainable fix is to update the shared bypass helper and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve command-routing gates in the bypass:** `extensions/acpx/index.ts:56-57`
  - body: This ACPX wrapper returns before the shared ACP hook can compute `shouldBypassAcpDispatchForCommand`. Add `/focus` to the shared bypass helper and keep the existing `allowTextCommands` gate for local commands.
