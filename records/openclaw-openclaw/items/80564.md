## Summary

Keep open: current main still has a real ACP command-routing gap for `/focus`, but this PR’s ACPX-local wrapper is not the safest merge shape and the external contributor proof gate is still missing.

## What This Changes

The PR wraps ACPX reply dispatch so `/acp`, `/focus`, and `/unfocus` command-shaped messages bypass ACP runtime handling while a session is active, with ACPX unit tests and a changelog entry.

## Best Possible Solution

Extend `shouldBypassAcpDispatchForCommand` to include `/focus` under the same local-command gate as `/unfocus`, keep ACPX registered to the shared hook, and add central regression coverage for `/focus` plus disabled text-command cases.

## Reproduction Assessment

Yes, at source level: current main routes ACPX through the shared ACP reply hook, and the shared bypass helper does not recognize documented `/focus` command messages. I did not run a live ACP-bound chat session in this read-only review.

## Solution Assessment

No. The PR likely fixes the happy path, but the ACPX-local wrapper bypasses the central command-routing gates; the narrower maintainable fix is to update the shared bypass helper and its tests.
