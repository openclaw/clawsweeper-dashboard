## Summary

Keep open: current main still leaves `/focus` out of ACP dispatch bypass, but this PR’s ACPX-local wrapper skips the shared text-command gate and the external contributor has not provided real behavior proof, so the patch is not merge-ready.

## What This Changes

The PR adds an ACPX-local reply dispatch wrapper for `/acp`, `/focus`, and `/unfocus`, updates ACPX unit tests, and adds a changelog entry.

## Best Possible Solution

Move the `/focus` addition into `shouldBypassAcpDispatchForCommand`, keep the existing `allowTextCommands` gate for local focus commands, and add shared regression coverage for `/focus` plus disabled text-command cases.

## Reproduction Assessment

Yes, at source level: current main routes ACPX through the shared ACP reply hook, and that helper does not recognize documented `/focus` commands. I did not run a live ACP-bound chat session in this read-only review.

## Solution Assessment

No. The PR likely fixes the happy path, but the maintainable fix is to update the shared bypass helper and tests rather than adding an ACPX-local wrapper that bypasses existing command-routing gates.
