## Summary

Keep open: this is a real source-level ACPX command-routing bug, but the PR is not merge-ready because it lacks required real behavior proof and the plugin-local wrapper bypasses existing cfg-aware command gating.

## What This Changes

The PR wraps ACPX `reply_dispatch` with a plugin-local bypass for `/acp`, `/focus`, and `/unfocus`, adds ACPX tests, and updates the changelog.

## Best Possible Solution

Centralize `/focus` in the shared ACP dispatch-bypass helper with regression tests that preserve text-command gating, then land after redacted real ACP chat proof is provided.

## Reproduction Assessment

Yes, at source level: ACPX registers the shared ACP reply hook, the shared helper bypasses `/acp` and `/unfocus` but not documented `/focus`, and the docs define `/focus` as an existing command. I did not run a live ACP-bound chat session.

## Solution Assessment

No. The narrower maintainable fix is to extend the shared `shouldBypassAcpDispatchForCommand` helper for `/focus` while preserving its cfg-aware command policy, instead of adding a second ACPX-only command classifier.
