## Summary

Keep open: the reported ACPX command-routing gap is source-reproducible, but this branch should not merge yet because the plugin-local bypass skips the existing command-routing gates and the external PR still has no real behavior proof.

## What This Changes

The PR wraps ACPX `reply_dispatch` with a plugin-local bypass for `/acp`, `/focus`, and `/unfocus`, adds ACPX unit tests, and updates the changelog.

## Best Possible Solution

Extend `shouldBypassAcpDispatchForCommand` to recognize `/focus` under the same `allowTextCommands` gate as `/unfocus`, keep ACPX on the shared hook, and add central regression coverage for `/focus` plus disabled text-command cases.

## Reproduction Assessment

Yes, at source level: current main routes ACPX through the shared ACP dispatch hook, and that shared bypass does not recognize documented `/focus` commands. I did not run a live ACP-bound chat session in this read-only review.

## Solution Assessment

No. The PR likely fixes the happy path, but the ACPX-local wrapper bypasses the central command-routing gates; the narrower maintainable fix is to update the shared bypass helper and its tests.
