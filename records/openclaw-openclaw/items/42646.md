## Summary

Keep open: current main still uses SQLite for retrieval/index storage and Markdown for durable memory, not the requested canonical typed SQLite record/revision/status schema; the related Memory MVP provenance, write-pipeline, and review-flow items are also still open.

## What This Changes

Define a v1 SQLite durable-memory schema with typed canonical records, revisions, statuses, timestamps, validity windows, and entity fields.

## Best Possible Solution

Design the durable record contract in the memory owner layer, then implement migrations/schema and focused tests alongside the provenance, write-pipeline, and review-flow decisions.

## Reproduction Assessment

Not applicable. This is a schema/product request rather than a runtime bug, and source/docs inspection shows current SQLite is a retrieval index over Markdown memory files, not a failing behavior path.

## Solution Assessment

Unclear. A conservative v1 SQLite schema is plausible, but it should be sequenced with the open provenance, write-pipeline, and review-flow items rather than landed as a standalone storage contract.
