## Summary

Keep open. Current main and v2026.6.8 still use Markdown as durable memory and SQLite mainly for indexing/plugin state, so the typed canonical SQLite memory record model remains unimplemented and needs maintainer data-model review.

## What This Changes

The issue requests a v1 SQLite-backed durable memory schema with typed canonical records, revision history, lifecycle statuses, timestamps, validity windows, and entity normalization.

## Best Possible Solution

Have maintainers choose the memory-plugin durable record contract and migration plan across the Memory MVP cluster, then implement the SQLite schema at the storage boundary with focused tests and docs while keeping embeddings retrieval-only.

## Reproduction Assessment

Not applicable; this is a feature and stored data-model request, not a current-main bug report. Source and docs inspection show the requested durable SQLite record model is missing rather than failing at runtime.

## Solution Assessment

Unclear until maintainers choose the product boundary. A typed SQLite model is plausible, but it should be designed together with the open provenance, write-pipeline, review/edit/forget, and CLI/skill follow-ups.
