## Summary

Keep open. Current main and the latest release still use SQLite for the memory search index and plugin state, while durable memory remains Markdown-based; the requested typed canonical record, revision, status, and entity schema is not implemented and needs maintainer product direction.

## What This Changes

The issue requests a v1 SQLite-backed canonical memory schema with typed durable records, revision history, status transitions, and entity normalization.

## Best Possible Solution

Keep this issue open as the schema/product anchor until maintainers choose the durable memory owner boundary, SQLite tables, migration path, and how it composes with the open provenance, write-pipeline, review-flow, and CLI-surface Memory MVP issues.

## Solution Assessment

No, current main is not the best solution for this issue because it only provides Markdown durable memory plus SQLite retrieval/plugin-state storage. The maintainable path is a maintainer-approved memory-core/plugin schema and migration design before code is queued.
