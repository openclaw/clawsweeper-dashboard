## Summary

Keep open. Current main and latest release still use Markdown as durable memory and SQLite mainly for indexes/plugin state; the requested typed canonical SQLite memory record model remains a product and stored-data design decision.

## What This Changes

Request a v1 local-first SQLite schema and canonical typed durable memory record model with revisions, lifecycle statuses, timestamps, and normalized subject/entity fields.

## Best Possible Solution

Have maintainers choose the memory-plugin durable record contract and migration plan across the Memory MVP issues, then implement it at the storage boundary with focused tests and docs while keeping embeddings retrieval-only.

## Reproduction Assessment

Not applicable; this is a feature and stored data-model request, not a reported current-main bug. Source and docs inspection show the requested durable SQLite record model is missing by design rather than failing at runtime.

## Solution Assessment

Unclear until maintainers choose the product boundary. A typed SQLite model is plausible, but it should be designed together with the open provenance, write-pipeline, review/edit/forget, and CLI/skill follow-ups.
