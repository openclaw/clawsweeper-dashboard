## Summary

Keep open. Current main and the latest release still treat SQLite as memory index/plugin-state storage while durable long-term memory remains Markdown; the requested typed canonical memory records, revision trail, lifecycle statuses, and entity model are still a product/data-model decision.

## What This Changes

The issue requests a v1 SQLite durable memory schema with typed records, canonical current state, revision history, lifecycle statuses, timestamps, validity windows, and entity normalization.

## Best Possible Solution

Decide the memory-core v1 durable record contract across schema, provenance, write, review/forget flows, then implement it at the memory plugin/storage boundary with migrations, docs, and focused tests while keeping embeddings retrieval-only.

## Reproduction Assessment

Not applicable; this is a feature and stored data-model request, not a reported failing current behavior. Source and docs inspection show current main lacks the requested durable record model rather than exposing a reproducible bug.

## Solution Assessment

Unclear until maintainers choose the memory product boundary. A typed SQLite durable record model is plausible, but it should be designed together with the open provenance, write-pipeline, review/edit/forget, and CLI/skill follow-ups.
