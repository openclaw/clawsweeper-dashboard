## Summary

Keep open. Current main and the latest release still use SQLite for memory indexes/plugin state and Markdown for durable memory, not the requested typed canonical memory record, revision, status, and entity model; the related Memory MVP follow-up issues remain open and this needs maintainer product direction before implementation.

## What This Changes

Request defines a v1 SQLite-backed typed memory record schema with revisions, explicit statuses, timestamps, validity windows, and entity normalization.

## Best Possible Solution

Decide the memory-core v1 durable record contract first, then implement the typed SQLite schema, migrations, docs, and focused tests at the memory plugin/storage boundary while keeping embeddings retrieval-only.

## Reproduction Assessment

Not applicable; this is a feature and data-model request, not a reported failing current behavior. Source inspection shows current main lacks the requested durable record model rather than exposing a reproducible bug.

## Solution Assessment

Unclear until maintainers choose the memory product boundary. A typed SQLite record model is plausible, but it should be designed together with the open provenance, write-pipeline, and review/edit/forget follow-ups.
