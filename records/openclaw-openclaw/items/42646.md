## Summary

Keep open: current main and the latest release still use Markdown files as the durable memory source and SQLite as a retrieval/index layer, so the requested canonical SQLite record model remains an unresolved Memory MVP storage-contract decision.

## What This Changes

The issue requests a v1 SQLite-backed durable memory schema with typed records, canonical current state, revision history, status transitions, timestamps, validity windows, and entity normalization.

## Best Possible Solution

Settle the durable-memory contract across memory-host SDK, memory-core promotion/write paths, provenance, review/forget flows, Markdown compatibility, migrations, and tests, then implement the v1 schema as a coordinated Memory MVP slice.

## Reproduction Assessment

Not applicable. This is a feature/storage-contract request, and source plus docs inspection shows current main uses Markdown durable storage with SQLite indexing rather than a failing behavior path.

## Solution Assessment

Unclear until maintainers decide the v1 durable-memory contract. The SQLite schema direction is plausible, but adding tables alone before migration, provenance, review/forget, and Markdown compatibility decisions would be the wrong slice.
