## Summary

Keep open. Current main and v2026.6.10 still use Markdown files as durable memory and SQLite as a retrieval/index store, so the requested typed canonical durable-memory record model remains missing and needs maintainer data-model direction.

## What This Changes

Define a v1 SQLite schema and canonical record model for local-first durable memories, including typed records, lifecycle status, revisions, entity fields, and validity windows.

## Best Possible Solution

Have maintainers define the plugin-owned durable memory data model and migration/doctor boundary across the Memory MVP cluster, then implement the conservative SQLite schema with embeddings kept retrieval-only.

## Reproduction Assessment

Not applicable; this is a stored data-model feature request, not a current-main bug report. Source and docs inspection show the requested durable SQLite record model is missing rather than failing at runtime.

## Solution Assessment

Unclear until maintainers choose the product and ownership boundary. A typed SQLite model is plausible, but it should be designed together with the open provenance, write-pipeline, review/edit/forget, and CLI/skill follow-ups.
