## Summary

Keep open. Current main still treats SQLite as a retrieval/indexing store over Markdown and session chunks, not as the requested durable typed memory source of truth with canonical records, revision history, normalized entities, lifecycle statuses, confidence, and validity windows.

## What This Changes

Define a v1 SQLite schema and canonical record model for local-first durable memory records, revisions, entities, statuses, timestamps, confidence, and validity windows.

## Best Possible Solution

Keep this as a maintainer-owned Memory MVP design item and land a coordinated versioned schema, migrations, exported record/revision/entity/status/provenance types, write-path integration, and docs that explain how canonical SQLite records coexist with Markdown memory, retrieval indexes, memory-wiki, LanceDB, and optional memory plugins.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a schema/product request. There is a high-confidence verification path by inspecting current source and docs: current main has no durable typed SQLite record schema and still documents Markdown plus retrieval indexes as the default memory model.

## Solution Assessment

Unclear as an autonomous fix. The requested direction is plausible, but the maintainable solution is a coordinated Memory MVP storage/API design rather than an opportunistic table addition without provenance, write-pipeline, and review/forget semantics.
