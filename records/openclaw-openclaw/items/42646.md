## Summary

Keep open. Current main and the latest release still treat Markdown as durable memory and SQLite as an index/plugin-state store, so the requested typed canonical durable-memory record model remains missing and needs maintainer data-model direction.

## What This Changes

Define a typed local-first SQLite durable memory schema with canonical records, revisions, statuses, validity windows, and entity normalization.

## Best Possible Solution

Have maintainers define the plugin-owned durable memory data model and migration/doctor boundary across the Memory MVP cluster, then implement the conservative SQLite schema and tests with embeddings remaining retrieval-only.

## Reproduction Assessment

Not applicable; this is a feature and stored data-model request, not a current-main bug report. Source and docs inspection show the requested durable SQLite record model is missing rather than failing at runtime.

## Solution Assessment

Unclear until maintainers choose the product boundary. A typed SQLite model is plausible, but it should be designed together with the open provenance, write-pipeline, review/edit/forget, and CLI/skill follow-ups.
