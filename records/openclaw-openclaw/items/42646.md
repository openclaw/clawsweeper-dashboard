## Summary

Keep open: current main still uses SQLite as a search/index database over Markdown memory files, not as a typed durable-memory source of truth with canonical records, revisions, statuses, entities, and validity windows. The linked Memory MVP provenance, write-pipeline, and review/forget issues are still open, so this needs maintainer sequencing rather than cleanup closure.

## What This Changes

Define a v1 SQLite schema and canonical record model for local-first durable memories.

## Best Possible Solution

Design the durable memory record contract together with provenance, resolver, review, edit, and forget semantics, then land schema/migration coverage in the memory owner layer.

## Reproduction Assessment

Not applicable. This is a schema/product request rather than a runtime bug, and source/docs inspection shows current SQLite storage is still an index over Markdown chunks.

## Solution Assessment

Unclear. A conservative v1 SQLite schema is plausible, but it should be designed with the open provenance, write-pipeline, review/forget, and owner-boundary issues rather than attempted as an isolated automated fix.
