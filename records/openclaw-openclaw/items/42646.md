## Summary

Keep open. Current main and the latest release still treat Markdown files as durable memory while SQLite supports indexing and plugin state, so the requested canonical typed SQLite memory record model remains a product and stored-data design decision.

## What This Changes

The issue requests a v1 SQLite schema and canonical typed record model for durable local memory records.

## Best Possible Solution

Have maintainers choose the memory-plugin durable record contract and migration plan across the Memory MVP cluster, then implement the SQLite schema at the storage boundary with focused tests and docs while keeping embeddings retrieval-only.

## Reproduction Assessment

Not applicable; this is a feature and stored data-model request, not a reported current-main bug. Source and docs inspection show the requested durable SQLite record model is missing rather than failing at runtime.

## Solution Assessment

Unclear until maintainers choose the product boundary. A typed SQLite model is plausible, but it should be designed together with the open provenance, write-pipeline, review/edit/forget, and CLI/skill follow-ups.
