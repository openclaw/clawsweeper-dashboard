## Summary

Keep open. Current main and the latest release still use Markdown as durable memory while SQLite is used for indexing/plugin state, so the requested typed canonical SQLite memory record model remains an unresolved product and stored-data design decision.

## What This Changes

Define a v1 local-first SQLite durable memory schema with typed records, canonical current state, revision history, lifecycle statuses, timestamps, validity windows, and entity normalization.

## Best Possible Solution

Have maintainers choose the memory-plugin durable record contract and migration plan across the Memory MVP cluster, then implement it at the storage boundary with focused tests and docs while keeping embeddings retrieval-only.

## Reproduction Assessment

Not applicable; this is a feature and stored data-model request, not a reported current-main bug. Current source and docs show the requested durable SQLite record model is missing by design rather than failing at runtime.

## Solution Assessment

Unclear until maintainers choose the product boundary. A typed SQLite model is plausible, but it should be designed together with the open provenance, write-pipeline, review/edit/forget, and CLI/skill follow-ups.
