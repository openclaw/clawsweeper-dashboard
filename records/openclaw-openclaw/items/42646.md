## Summary

Keep open: current main and the latest release still use SQLite as an index over Markdown memory chunks, while the requested typed durable record schema and canonical revision/status model remain unimplemented and tied to open Memory MVP design work.

## What This Changes

Define a v1 local-first SQLite memory schema with typed records, one canonical current state, revision history, explicit statuses, timestamps, validity windows, and normalized entity fields.

## Best Possible Solution

Design the durable memory record contract together with provenance, resolver/write-pipeline, review/edit/forget flows, then land schema, migration, and regression coverage in the chosen memory owner layer.

## Reproduction Assessment

Not applicable. This is a feature/schema request, and source inspection against current main shows the active SQLite path indexes Markdown chunks rather than exposing a failing runtime behavior.

## Solution Assessment

Unclear. A conservative SQLite v1 schema is plausible, but the maintainable path is coordinated memory design across the open provenance, write-pipeline, and review/forget work rather than an isolated automated schema patch.
