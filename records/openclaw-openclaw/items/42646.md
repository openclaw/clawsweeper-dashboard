## Summary

Current main does not implement the requested canonical SQLite record model, and no merged or open replacement owns it. The proposal remains a valid but broad product architecture decision because OpenClaw currently treats Markdown workspace files as durable memory while the memory plugin provides indexing, recall, and promotion around that contract; changing the source of truth must be coordinated with the linked provenance, write-resolution, and operator-flow issues.

## What This Changes

Define a v1 SQLite schema and canonical typed record model for durable local memory, including revisions, statuses, validity windows, and normalized entities.

## Best Possible Solution

Approve one coordinated memory architecture that selects the recommended memory plugin and database owner, defines a single canonical record and migration contract, and aligns provenance, write resolution, retrieval, review, edit, conflict, and forget flows before implementation.

## Reproduction Assessment

Not applicable: this issue proposes a new canonical memory architecture rather than reporting a failure in current behavior.

## Solution Assessment

No, not as an isolated schema-first task; the best solution must first settle the canonical plugin, database owner, migration from Markdown memory, and the linked provenance/write/review lifecycle contracts.
