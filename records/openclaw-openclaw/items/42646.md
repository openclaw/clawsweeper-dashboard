## Summary

Keep open: current main and the latest release still use Markdown files as durable memory and SQLite as an index/cache backend, so the requested v1 durable-record schema remains an unresolved Memory MVP storage/product contract. The related Memory MVP companion issues are still open, and the broad Memory v2 sidecar PR is related but does not close this work and still has maintainer-review blockers.

## What This Changes

Define a v1 SQLite durable-memory schema and canonical record model for typed records, revisions, statuses, timestamps, validity windows, and entity normalization.

## Best Possible Solution

Settle one coordinated durable-memory contract across memory-host SDK, memory-core, Memory Wiki/Markdown compatibility, migrations, provenance, review/forget flows, and tests before adding the v1 SQLite source-of-truth schema.

## Reproduction Assessment

Not applicable. This is a feature/storage-contract request, and current-main source plus latest-release inspection show Markdown durable storage with SQLite indexing rather than a failing bug path.

## Solution Assessment

Unclear until maintainers decide the v1 durable-memory contract. The SQLite schema direction is plausible, but adding tables alone before migration, provenance, review/forget semantics, and Markdown compatibility are settled would be the wrong slice.
