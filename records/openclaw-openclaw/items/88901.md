## Summary

Keep open. This PR is still useful: current main still builds Mattermost draft progress callbacks without opting them into suppressed-source delivery, while the branch wires the existing allow flag and adds focused inbound coverage. The broader shared room-event work in https://github.com/openclaw/openclaw/pull/92738 does not supersede this because it does not make Mattermost opt into the callback path.

## What This Changes

The PR adds Mattermost reply-option wiring for `allowProgressCallbacksWhenSourceDeliverySuppressed` when draft tool progress is enabled, plus a regression test for message-tool-only source replies.

## Best Possible Solution

Land the narrow Mattermost opt-in and regression test, while leaving broader room-event progress behavior to the separate shared-routing PR if maintainers want it.

## Reproduction Assessment

Yes. Source inspection shows Mattermost creates draft progress callbacks but current main suppresses them under `message_tool_only` unless the channel passes the existing allow flag; the linked issue also reports the same symptom on v2026.6.5.

## Solution Assessment

Yes. The PR uses the existing channel-owned progress callback contract only when Mattermost draft tool progress is enabled, which is narrower than changing final delivery defaults or forcing `visibleReplies: "automatic"`.
