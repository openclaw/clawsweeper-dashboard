## Summary

Keep open: current main still contains the Feishu streaming final-skip and pending-merge paths described, and an existing test asserts one of the lossy late-final behaviors, so this is a valid source-level bug rather than stale or implemented.

## What This Changes

The issue asks to fix Feishu/Lark streaming card delivery so block updates, throttled pending updates, idle close, and final replies do not lose, duplicate, or preserve stale text.

## Best Possible Solution

Patch the Feishu plugin dispatcher/session so final card content is flushed deterministically, block snapshots and final suffixes do not accumulate stale text, idle-close skips only already delivered finals, and regression tests cover the reported paths.

## Reproduction Assessment

Yes, at source level: current main has the exact skip, delta-append, and pending-merge paths described, and the existing test harness already demonstrates distinct late final text being dropped after streaming close. I did not run a live Feishu card scenario in this read-only review.

## Solution Assessment

Yes: a plugin-local fix in `reply-dispatcher.ts` and `streaming-card.ts` with focused Feishu regression tests is the narrowest maintainable route. It should not change core streaming contracts or other channels.
