## Summary

Keep open. Current main and the latest tagged release still model token/password gateway auth as a single shared operator secret, while the requested per-token identity, scopes, and session attribution would add a new security-sensitive public auth/config contract that needs maintainer product review.

## What This Changes

Request support for multiple `gateway.auth` bearer tokens with per-token labels/scopes so shared Control UI deployments can isolate users and sessions.

## Best Possible Solution

Design a cohesive gateway token-identity model covering config schema/types, docs, token storage/rotation/revocation, scope semantics, session attribution, and composition with trusted-proxy, Tailscale, and device pairing.

## Reproduction Assessment

Yes, as a source-level capability check: current main accepts and resolves one shared `gateway.auth.token`/password, token auth returns no user identity, and the strict schema has no `tokens` registry. This is a missing feature/config surface rather than a broken existing bug contract.

## Solution Assessment

Unclear. A token registry is plausible, but maintainers need to settle whether token-derived labels/scopes are authoritative, display-only, or bound to device/session identity before implementation.
