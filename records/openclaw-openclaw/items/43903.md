## Summary

Keep open. Current main still has one shared Gateway token/password path plus identity-bearing trusted-proxy/Tailscale paths; it does not support a `gateway.auth.tokens` map or a token-authenticated principal for Control UI/session isolation.

## What This Changes

Request adding multiple Gateway token entries with labels/scopes so shared Control UI access can map each token to an isolated user/session identity.

## Best Possible Solution

Define a backwards-compatible Gateway auth contract that preserves `gateway.auth.token`, optionally adds validated multi-token entries with labels/scopes, resolves the matched entry into a server-attested principal, and coordinates that principal with chat attribution work such as #68353 and broader auth separation in #69066.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source inspection of config types/schema, resolved auth, token authorization, connect schema, Control UI `chat.send`, and Gateway auth docs on current main.

## Solution Assessment

Unclear. The proposed `tokens` map is plausible, but the best solution needs maintainer/security design for token matching, scopes, device pairing, trusted-proxy/Tailscale interaction, and chat/session attribution.
