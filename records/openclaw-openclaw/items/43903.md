## Summary

Keep open. Current main still treats token/password auth as one shared operator credential, while identity-bearing behavior lives in trusted-proxy/Tailscale paths; the request remains a security-sensitive feature/config design item, not an implemented fix or duplicate.

## What This Changes

Add a gateway token registry that maps multiple token secrets to labels/scopes so team members can use the Control UI with per-user session isolation.

## Best Possible Solution

Define one cohesive gateway identity model for token-mode team access, including token registry schema, label/scope semantics, rotation/revocation, session attribution, docs, audit checks, and composition with trusted-proxy plus chat author attribution.

## Reproduction Assessment

Yes for the current limitation by source inspection: current main still resolves and compares one shared gateway.auth.token rather than a per-token identity registry. This is a missing feature/config surface rather than a broken existing bug contract.

## Solution Assessment

Unclear. A tokens registry is plausible, but maintainers should first choose the trust model for labels, scopes, revocation, rotation, and overlap with trusted-proxy plus chat.send.author attribution.
