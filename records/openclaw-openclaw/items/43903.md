## Summary

Keep open. Current main still has a single shared Gateway token path, while the issue asks for a new multi-token, token-derived identity and scope model for shared Control UI deployments; the discussion also shows real team-deployment demand, so this needs maintainer/security design rather than cleanup.

## What This Changes

Request support for a `gateway.auth.tokens` map that assigns labels and scopes to multiple Gateway tokens for isolated multi-user Control UI sessions.

## Best Possible Solution

Define a backward-compatible Gateway auth contract that preserves `gateway.auth.token`, adds validated multi-token entries only if maintainers accept the product direction, resolves the matched token into a server-attested principal, and coordinates that principal with shared-session attribution work such as #68353.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source inspection showing current main still accepts one shared token and has no token principal or `gateway.auth.tokens` config surface.

## Solution Assessment

Unclear. The proposed `tokens` map is plausible, but the best solution needs maintainer/security design for token matching, scope composition, trusted-proxy/Tailscale interaction, device pairing, and chat/session attribution.
