## Summary

Keep open. Current main still has one shared Gateway token/password path plus identity-bearing trusted-proxy/Tailscale paths; it does not have a `gateway.auth.tokens` map, per-token labels/scopes, or a server-attested token principal for Control UI attribution.

## What This Changes

Request support for multiple Gateway auth tokens with per-token labels/scopes so shared Control UI deployments can isolate and attribute users without trusted-proxy auth.

## Best Possible Solution

Define a backwards-compatible Gateway auth contract that preserves `gateway.auth.token`, adds validated multi-token entries if maintainers accept the product direction, resolves the matched token into a server-attested principal, and coordinates that principal with Control UI/session attribution work such as #68353.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source inspection of config types/schema, resolved auth, token authorization, connect schema, Control UI `chat.send`, and shared-secret docs on current main.

## Solution Assessment

Unclear. The proposed `tokens` map is plausible, but the best solution needs maintainer/security design for token matching, scope composition, device pairing, trusted-proxy/Tailscale interaction, and how token identity feeds chat/session attribution.
