## Label Justifications

- `P2`: The issue is a normal-priority Gateway auth/session isolation feature with meaningful security impact but no current outage or regression.
- `impact:session-state`: The requested behavior is explicitly about isolating shared Control UI users into distinct sessions/history.
- `impact:security`: Multiple bearer credentials with identity and scopes would change the Gateway trust boundary and operator credential blast radius.
- `impact:auth-provider`: The issue is about Gateway authentication credential modeling and how tokens resolve into caller identity/scopes.

## Summary

Keep open: current main and v2026.6.6 still implement Gateway token auth as one shared secret; trusted-proxy is the identity-bearing path, and the open related scoped-token and author-attribution issues do not fully replace this per-user Control UI token/session isolation request.

## What This Changes

Request support for multiple Gateway bearer tokens with per-token labels/scopes so shared Control UI users can get isolated session identity while retaining token/device-pairing workflows.

## Best Possible Solution

Keep this open for maintainer-led Gateway auth design that decides the core primitive for per-token identity/scopes and how it composes with trusted-proxy, device pairing, Control UI sessions, and related attribution work.

## Reproduction Assessment

Not a bug reproduction; source inspection confirms current main only supports singular gateway.auth.token/password plus trusted-proxy identity, and the strict config schema has no gateway.auth.tokens shape.

## Solution Assessment

Unclear as proposed: a tokens object is plausible, but the best solution needs a maintainer security/product decision across Gateway auth scopes, Control UI pairing, session attribution, and the related scoped-token/author proposals.
