## Summary

Keep open. Current main still has a single shared `gateway.auth.token` path; trusted-proxy auth and the related per-message author issue cover adjacent identity problems, but they do not implement per-token identity mapping for token-mode Control UI access.

## What This Changes

Request support for multiple gateway auth tokens with per-token labels/scopes so shared Control UI users can have isolated, attributable sessions.

## Best Possible Solution

Define a cohesive gateway identity model for token-mode team access, including token registry schema, label/scope semantics, session attribution, revocation/rotation behavior, and how it composes with trusted-proxy and first-class `chat.send` authorship.

## Reproduction Assessment

Yes for the current limitation, via source inspection rather than a runtime bug repro: current main only accepts and resolves one shared `gateway.auth.token`. The requested behavior is a new feature/config surface, not a broken existing contract.

## Solution Assessment

Unclear. Multiple token-mode identities are plausible, but the best solution needs maintainer/security/product direction on trust semantics and overlap with trusted-proxy auth and https://github.com/openclaw/openclaw/issues/68353.
