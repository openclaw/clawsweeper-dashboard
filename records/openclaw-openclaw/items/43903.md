## Summary

Keep open. Current main still models token auth as one shared bearer secret with no per-token identity, label, or scope mapping, and the related author-attribution issue only covers a narrower `chat.send` authorship path.

## What This Changes

The issue requests multiple gateway auth tokens with per-token labels/scopes so shared Control UI deployments can isolate and attribute users without trusted-proxy setup.

## Best Possible Solution

Define a first-class gateway auth identity contract, or explicitly choose trusted-proxy plus the author-attribution follow-up as the supported multi-user path; the chosen path should include config schema, auth resolution, scope semantics, Control UI propagation, docs, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main lacks the requested multi-token identity surface: config, auth resolution, token authorization, and Control UI send paths all carry one shared token and no per-token label.

## Solution Assessment

Unclear. The proposed `tokens` map is a plausible direction, but this is not a narrow bug fix because it changes gateway auth, operator scopes, session attribution, and security semantics; maintainers should choose the contract before implementation.
