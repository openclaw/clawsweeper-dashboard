- `impact:auth-provider`: The affected surface is Gateway authentication, token resolution, and SecretRef-compatible credential config.

## Summary

Keep open: current main and v2026.6.6 still model Gateway shared-secret auth as one token/password plus identity-bearing trusted-proxy and device-token alternatives, while this asks for a new multi-principal Gateway auth and session-attribution design.

## What This Changes

Add support for multiple Gateway bearer tokens with per-token labels/scopes so multiple Control UI users can get isolated sessions without trusted-proxy setup.

## Best Possible Solution

Design one additive multi-principal Gateway auth model, or explicitly keep recommending trusted-proxy/front-layer patterns such as Pinchy for multi-user deployments until token-principal security semantics are approved.

## Reproduction Assessment

Yes, source-level: current main accepts one gateway.auth.token and compares one presented token, with no tokens map/list or per-token identity reaching scopes or session metadata.

## Solution Assessment

Unclear: per-token identity is plausible, but the schema, scope vocabulary, attestation, audit/session attribution, and relationship to trusted-proxy/device-token/front-layer patterns need maintainer security and product design first.
