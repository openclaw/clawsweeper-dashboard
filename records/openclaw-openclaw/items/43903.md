## Label Justifications

- `P2`: This is a meaningful Gateway auth/session-isolation feature with security impact, but it is not a current outage or regression.
- `impact:session-state`: The requested behavior is explicitly about mapping different Control UI users to isolated session history.
- `impact:security`: Multiple bearer credentials with identity and scopes would change the Gateway trust boundary and credential blast radius.
- `impact:auth-provider`: The issue changes how Gateway auth credentials resolve into caller identity and operator scopes.

## Summary

Keep open: current main and v2026.6.6 still model Gateway token auth as one shared secret, while the identity-bearing path is trusted-proxy/device auth. The linked scoped-token, author-attribution, and org-deployment issues are complementary but do not replace this per-user Control UI token/session-isolation request.

## What This Changes

The issue requests multiple Gateway bearer tokens with per-token labels/scopes so shared Control UI users can map to isolated session history without using trusted-proxy.

## Best Possible Solution

Maintainers should design one canonical Gateway principal model for per-token identity/scopes and decide how it composes with trusted-proxy, device tokens, Control UI session keys, and author attribution before any implementation PR.

## Reproduction Assessment

Not a bug reproduction; source inspection gives a high-confidence capability check. Current main and v2026.6.6 support singular gateway.auth.token/password plus trusted-proxy identity, with no gateway.auth.tokens schema or per-token identity path.

## Solution Assessment

Unclear as proposed: a tokens map is plausible, but not yet proven as the best solution. The best fix needs maintainer security/product design across Gateway scopes, device pairing, trusted-proxy, Control UI session identity, and related author-attribution proposals.
