## Summary

Keep open. Current main and the latest release still model gateway token/password auth as a single shared operator credential, while identity-bearing behavior remains on trusted-proxy/Tailscale-style paths; this is a valid security-sensitive feature/config request that needs maintainer product review rather than cleanup closure.

## What This Changes

Add a `gateway.auth.tokens` registry that maps multiple bearer tokens to user labels/scopes so shared Control UI deployments can isolate and attribute users.

## Best Possible Solution

Design a cohesive gateway token identity model with schema, docs, tests, rotation/revocation semantics, session attribution, and clear composition with device pairing and trusted-proxy auth.

## Reproduction Assessment

Yes by source inspection: current main still accepts and compares one shared `gateway.auth.token`, and token auth returns no user identity. This is a missing feature/config surface rather than a broken existing bug contract.

## Solution Assessment

Unclear. A token registry is plausible, but maintainers should first choose the trust model for labels, scopes, revocation, rotation, and overlap with trusted-proxy plus chat attribution.
