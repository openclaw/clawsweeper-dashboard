## Summary

Keep open. Current main still models token/password auth as a single shared operator credential, while identity-bearing behavior is limited to trusted-proxy/Tailscale-style paths; this remains a security-sensitive feature/config design request, not an implemented fix or duplicate.

## What This Changes

Add multi-token gateway auth with per-token labels/scopes so shared Control UI deployments can isolate and attribute user sessions without trusted-proxy setup.

## Best Possible Solution

Define one cohesive gateway identity model for token-mode team access, covering token registry schema, label/scope semantics, rotation/revocation, session attribution, docs, audit checks, and composition with trusted-proxy plus chat author attribution.

## Reproduction Assessment

Yes for the current limitation by source inspection: current main still resolves and compares one shared gateway.auth.token rather than a per-token identity registry. This is a missing feature/config surface rather than a broken existing bug contract.

## Solution Assessment

Unclear. A tokens registry is plausible, but maintainers should first choose the trust model for labels, scopes, revocation, rotation, and overlap with trusted-proxy plus chat.send author attribution.
