## Summary

Keep open: current main and v2026.6.8 still model Gateway token/password auth as one shared operator secret, and the open related threads overlap without replacing this focused per-user Control UI session-isolation request.

## What This Changes

Requested native support for multiple Gateway bearer tokens with per-token identity/scopes so shared Control UI access can isolate users and sessions.

## Best Possible Solution

Keep this as the focused per-user Gateway token/session-isolation design item until maintainers choose a principal model that composes token identity, scoped credentials, trusted-proxy, device pairing, and author attribution.

## Reproduction Assessment

Yes, as a source-level capability check rather than a runtime bug repro. Current main and v2026.6.8 support singular shared token/password auth plus identity-bearing trusted-proxy paths, with no gateway.auth.tokens map/list or token-derived user session identity.

## Solution Assessment

Unclear as proposed. Gateway auth is a plausible owner boundary, but the token object shape, scope semantics, session binding, device/trusted-proxy interaction, and author-attribution model need maintainer security/product direction first.
