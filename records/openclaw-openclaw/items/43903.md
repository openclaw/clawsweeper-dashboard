## Summary

Keep open: current main still has single shared-token gateway auth, while the request asks for a new identity-bearing auth/config surface that overlaps but is not fully covered by the related per-message author-attribution issue.

## What This Changes

The issue requests multiple gateway authentication tokens with per-token identity metadata so shared Control UI access can have isolated, attributable sessions.

## Best Possible Solution

Design this as an explicit gateway auth and identity feature: either add token identities with clear scope/enforcement/docs, or steer shared-team deployments to trusted-proxy plus dmScope and separate author-attribution work.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main supports one shared token and has no token-to-user identity mapping.

## Solution Assessment

Unclear. The proposed shape is plausible, but maintainers should first decide whether this is attribution-only, scoped operator auth, or a trusted-proxy alternative because each has different security semantics.
