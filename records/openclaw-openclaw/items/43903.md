## Label Justifications

- `P2`: This is a concrete Gateway auth and session-isolation feature with security impact, but it is not a live outage or urgent regression.
- `impact:session-state`: The requested per-token identity mapping is meant to route different Control UI users into isolated session history.
- `impact:security`: Multiple bearer credentials with identity and scopes would change the Gateway trust boundary and credential blast radius.
- `impact:auth-provider`: The feature changes how Gateway auth credentials resolve into caller identity and operator scopes.

## Summary

Keep open: current main and v2026.6.6 still model Gateway shared-secret auth as one token/password, with trusted-proxy as the identity-bearing alternative. The linked scoped-token, author-attribution, team-auth, and org-deployment issues are complementary open work, and the Pinchy workaround shows a useful external layer rather than an OpenClaw core implementation.

## What This Changes

The issue requests multiple Gateway bearer tokens with per-token labels/scopes so webchat and Control UI users can get isolated sessions without setting up trusted-proxy auth.

## Best Possible Solution

Design one canonical Gateway principal model for per-token identity/scopes and decide how it composes with trusted-proxy, device pairing, Control UI session keys, and author attribution before implementation.

## Reproduction Assessment

Yes, as a source-level capability check: current main and v2026.6.6 have no gateway.auth.tokens schema or per-token identity path, and token auth compares one shared secret.

## Solution Assessment

Unclear as proposed: a tokens map is plausible, but the best fix needs maintainer security/product design across Gateway scopes, device pairing, trusted-proxy, Control UI session identity, and related author-attribution proposals.
