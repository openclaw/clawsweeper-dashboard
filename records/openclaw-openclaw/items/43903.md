## Summary

Keep open. Current main still has one shared `gateway.auth.token`/password path plus identity-bearing trusted-proxy/Tailscale modes, while the requested per-token labels, scopes, and Control UI session attribution would add a new public auth/config contract that needs maintainer product and security review.

## What This Changes

Request support for multiple gateway tokens mapped to user labels/scopes so shared Control UI deployments can isolate and attribute sessions per human user.

## Best Possible Solution

Design one cohesive gateway token-identity model covering config schema/types, token storage and rotation, scope semantics, session attribution, docs, tests, and composition with trusted-proxy, Tailscale, and device pairing.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a missing feature/config surface. Source inspection gives high confidence that current main accepts and resolves one shared gateway auth token/password and has no token registry.

## Solution Assessment

Unclear. A token registry is plausible, but maintainers need to decide whether token-derived labels/scopes are authoritative, display-only, or bound to device/session identity before implementation.

## Risks / Open Questions

- Per-token labels and scopes would change the gateway authentication and authorization boundary, so storage, rotation, revocation, audit semantics, and scope limits need explicit design.
- The current trust model treats shared-secret callers as trusted operators; partial per-token identity could imply stronger multi-user isolation than OpenClaw currently provides.
- Adjacent open issues cover token scopes, per-message authorship, service identity, and multi-tenancy, but none is a complete replacement for this exact token-identity registry request.
