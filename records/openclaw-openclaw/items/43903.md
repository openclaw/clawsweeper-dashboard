- `P2`: This is a meaningful Gateway auth/session improvement with real team-deployment demand, but it is a feature/design request rather than an urgent regression or outage.
- `impact:session-state`: The requested per-token identity would affect how shared Control UI users map to isolated session history and metadata.

- `impact:auth-provider`: The affected surface is Gateway bearer-token authentication, SecretRef-backed credentials, and caller identity/scope resolution.

## Summary

Keep open: current main and v2026.6.10 still implement Gateway token/password auth as one shared operator credential, and the related open issues plus Pinchy-style front-layer workaround do not replace this focused per-user token/session-isolation design.

## What This Changes

Request support for multiple Gateway bearer tokens with per-token labels/scopes so Control UI/webchat users can get isolated sessions without trusted-proxy.

## Best Possible Solution

Keep this focused design item open until maintainers choose a Gateway principal model that composes token identity, scoped credentials, trusted-proxy, device pairing, session keys, and author attribution.

## Reproduction Assessment

Yes, as a source-level capability check rather than a runtime bug repro. Current main and v2026.6.10 support singular shared token/password auth plus identity-bearing trusted-proxy paths, with no gateway.auth.tokens map/list or token-derived user session identity.

## Solution Assessment

Unclear as proposed. Gateway auth is a plausible owner boundary, but the token object shape, scope semantics, session binding, device/trusted-proxy interaction, and author-attribution model need maintainer security/product direction first.
