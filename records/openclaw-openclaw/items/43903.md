## Summary

Keep open: current main and v2026.6.6 still model Gateway token auth as one shared operator secret, while this issue asks for a new per-user Gateway principal/token identity model for Control UI session isolation. The Pinchy workaround linked in discussion is useful evidence that an external multi-user layer can work today, but it does not replace the native Gateway auth/config work requested here.

## What This Changes

The issue requests multi-token Gateway auth with per-token identity mapping so multiple Control UI users can keep separate sessions while retaining token-mode access.

## Best Possible Solution

Design one core Gateway principal model that covers per-user token identity, scoped credentials, session attribution, docs, tests, and upgrade behavior while keeping external front layers like Pinchy as supported deployment options.

## Reproduction Assessment

No bug reproduction is applicable, but there is a high-confidence source capability check. Current main and v2026.6.6 accept only singular gateway.auth.token/password plus trusted-proxy identity, with no gateway.auth.tokens or per-token identity mapping.

## Solution Assessment

Unclear as a final design. Gateway core is the right layer for native token identity, but maintainers need to choose the principal, scope, attribution, and upgrade semantics before implementation.
