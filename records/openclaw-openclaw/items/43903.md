## Summary

Keep open. Current main still implements gateway token/password auth as a single shared operator credential, while identity-bearing behavior remains on trusted-proxy/Tailscale-style paths; this is a concrete security-sensitive feature/config request that needs maintainer product review rather than cleanup closure.

## What This Changes

Add a multi-token gateway auth mode where each token can carry an operator label/scopes for multi-user Control UI session isolation.

## Best Possible Solution

Design a cohesive gateway token identity model with schema, docs, tests, rotation/revocation semantics, session attribution, and clear composition with device pairing, trusted-proxy auth, and per-message author attribution.

## Reproduction Assessment

Yes, by source inspection: current main still accepts and compares one shared gateway.auth.token, the strict schema has no tokens map, and successful token auth returns no user identity. This is a missing feature/config surface rather than a broken existing bug contract.

## Solution Assessment

Unclear. A token registry is plausible, but maintainers should first choose the trust model for labels, scopes, revocation, rotation, and overlap with trusted-proxy, device auth, and chat/session attribution.
