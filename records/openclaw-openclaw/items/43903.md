## Summary

Keep open. Current main still has a single shared gateway token/password model, while identity-bearing behavior is only available through trusted-proxy/Tailscale-style paths; this is a security-sensitive feature and config design request, not an implemented bug fix or duplicate.

## What This Changes

Add a multi-token gateway auth mode with per-token labels/scopes so shared Control UI deployments can attribute and isolate users without trusted-proxy setup.

## Best Possible Solution

Define one cohesive gateway token-identity contract covering schema, SecretRef storage, label/scope semantics, rotation/revocation, auth result identity, session attribution, docs, audit checks, and interaction with trusted-proxy plus per-message author work.

## Reproduction Assessment

Yes, by source inspection: current main exposes a single gateway.auth.token, resolves one token secret, and token auth returns only method token with no user identity. This proves the requested token registry is absent, though it is a missing feature rather than a failing bug contract.

## Solution Assessment

Unclear. A tokens registry is plausible, but maintainers should first choose the trust model for labels, scopes, revocation, rotation, audit attribution, and overlap with trusted-proxy plus chat author attribution.
