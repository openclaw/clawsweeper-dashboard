## Summary

Keep open. Current main still has a single shared-token auth path, while the requested named token principals/scopes would add a new security-sensitive gateway auth and attribution model; the supporting deployment comment reinforces that this is a real multi-user use case, not stale noise.

## What This Changes

Request support for multiple gateway auth tokens with per-token labels/scopes so shared Control UI users can get isolated identity and session handling.

## Best Possible Solution

Define an additive gateway auth contract for named token principals and scopes, then align config schema, auth resolution, Control UI handshake, session/author projection, docs, and tests around explicit trust semantics.

## Reproduction Assessment

Not applicable as a feature request rather than broken documented behavior. Source inspection gives high confidence the requested mode is absent because current config accepts one `gateway.auth.token` and token auth compares one presented token to that one value.

## Solution Assessment

Unclear. The proposed `tokens` object is plausible, but it should not be implemented until maintainers decide whether token labels/scopes are authoritative identities, display metadata, or compose with trusted-proxy/device pairing and https://github.com/openclaw/openclaw/issues/68353.
