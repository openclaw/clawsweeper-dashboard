## Summary

Keep open: current main and v2026.6.6 still model Gateway shared-secret auth as one token/password plus identity-bearing trusted-proxy/device-token alternatives, while this asks for a new multi-principal Gateway auth and session-attribution surface that needs maintainer security/product design.

## What This Changes

The issue requests multiple Gateway bearer tokens with per-token labels/scopes so shared Control UI or webchat deployments can isolate users and sessions without trusted-proxy setup.

## Best Possible Solution

Maintainers should design an additive multi-principal Gateway auth model that explicitly composes with operator scopes, device tokens, trusted-proxy identity, session key derivation, docs, and upgrade rules before adding the new config surface.

## Reproduction Assessment

Yes, as source-level capability proof: current main accepts one gateway.auth.token and compares one presented token, with no tokens map/list or per-token identity reaching scopes or session metadata.

## Solution Assessment

Unclear: per-token identity is plausible, but the schema, scope vocabulary, attestation, and relationship to trusted-proxy, device tokens, Pinchy-style front layers, and related author-attribution issues need maintainer security/product design first.
