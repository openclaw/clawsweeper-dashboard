- `impact:security`: The request changes Gateway authentication, authorization, token leak blast radius, and multi-user trust boundaries.

- `impact:auth-provider`: The issue is about Gateway auth token shape, credential routing, scopes, and device-pairing behavior.

## Summary

Keep open: current main and the latest release still model token-mode Gateway auth as one shared secret, while trusted-proxy or a private frontend layer such as Pinchy covers only part of the team-webchat use case and not the requested per-token Gateway identity/session mapping.

## What This Changes

Request adding multi-token Gateway auth with per-token labels/scopes so shared Control UI/webchat deployments can isolate users and sessions while retaining token-mode CLI and device-pairing behavior.

## Best Possible Solution

Maintainers should choose an explicit product/security direction: endorse private frontend or trusted-proxy patterns for team access, or design additive multi-principal Gateway auth across config schema, token issuance/rotation, scopes, session attribution, device pairing, docs, and tests.

## Reproduction Assessment

Yes, as a source-level capability check: current main and `v2026.6.5` expose one `gateway.auth.token` and one-token comparison, with no multi-token identity map. This is a missing feature, not a broken existing contract.

## Solution Assessment

Unclear. Per-token mapping is plausible, but the best solution depends on whether maintainers want Gateway to become multi-principal or keep that boundary in trusted-proxy/private frontend layers.
