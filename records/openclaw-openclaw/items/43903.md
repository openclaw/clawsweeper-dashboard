## Summary

Keep open. Current main still has single shared-secret token auth, while the existing trusted-proxy path is the only gateway auth mode that returns a user identity; the request is a real feature/design change rather than an implemented cleanup candidate.

## What This Changes

The issue requests multiple gateway bearer tokens with per-token identity and scopes so shared Control UI access can isolate users and attribute sessions separately.

## Best Possible Solution

Keep this as a core gateway auth feature/design item and decide whether token-bound identity should be added directly or whether shared-session attribution should first land through the broader author-attribution work in #68353.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main lacks the requested multi-token config and token-to-user identity mapping.

## Solution Assessment

Unclear. The proposed `tokens` object is plausible, but the best maintainable solution needs maintainer security/product design across gateway auth, Control UI device pairing, scopes, and the related author-attribution proposal.
