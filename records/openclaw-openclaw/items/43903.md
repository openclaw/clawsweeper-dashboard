## Summary

Keep open. Current main still has only one shared gateway token in token mode, and the requested per-token label/scope/session identity mapping would add a new gateway auth/config surface with security and product semantics to decide.

## What This Changes

The issue requests multiple gateway bearer tokens with per-token user identity so shared Control UI access can isolate and attribute sessions per user.

## Best Possible Solution

Make a maintainer-led design call on a multi-user gateway identity model, then implement a documented additive config/protocol path that maps authenticated identity to scopes, sessions, and author metadata without weakening device or trusted-proxy boundaries.

## Reproduction Assessment

Not applicable as a feature request. The source check is high-confidence that current `gateway.auth` exposes one shared token and the connect/auth path has no per-token label or scope map.

## Solution Assessment

Unclear. A `gateway.auth.tokens` map is plausible, but the best solution needs maintainer/security direction on trust semantics and how it coordinates with trusted-proxy, device tokens, and https://github.com/openclaw/openclaw/issues/68353.
