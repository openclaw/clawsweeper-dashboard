## Summary

Keep open. Current main still models token auth as one shared operator secret, and the related author-attribution issue is adjacent rather than a per-token identity registry, so this remains a feature/security/product design request rather than a closeable implemented or duplicate item.

## What This Changes

Request adding a `gateway.auth.tokens` registry that maps multiple token secrets to labels/scopes so token-mode Control UI sessions can isolate and attribute users.

## Best Possible Solution

Define one cohesive gateway identity model for token-mode team access, covering token registry schema, label/scope semantics, revocation/rotation, session attribution, docs, audit checks, and composition with trusted-proxy and per-message author attribution.

## Reproduction Assessment

Yes for the current limitation by source inspection: current main still resolves and compares one shared `gateway.auth.token`, not a per-token identity registry. This is a missing feature/config surface rather than a broken existing bug contract.

## Solution Assessment

Unclear. A `tokens` registry is plausible, but maintainers should first choose the trust model for labels, scopes, revocation, rotation, and overlap with trusted-proxy plus `chat.send.author` attribution.
