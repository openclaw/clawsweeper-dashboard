## Summary

Keep open. Current main still has single shared-token/password gateway auth, no `gateway.auth.tokens` map, no server-resolved token principal, and no per-token label/scope metadata that could isolate Control UI users. The request is valid but security-sensitive gateway auth/product work, so it needs maintainer design before implementation.

## What This Changes

Request support for configuring multiple gateway auth tokens with per-token labels/scopes so Control UI users can have isolated sessions while preserving CLI and device-pairing flows.

## Best Possible Solution

Define a backwards-compatible multi-token gateway auth contract that preserves the existing single `gateway.auth.token`, resolves the matched entry into a server-attested principal, composes scopes explicitly with device/bootstrap/trusted-proxy auth, and updates Control UI/session attribution only through documented protocol surfaces.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence verification path: the config types/schema, resolved auth object, token authorizer, and WebSocket auth schema all lack multi-token entries or token-principal metadata on current main.

## Solution Assessment

Unclear. The proposed `tokens` map is plausible, but the best implementation needs maintainer/security design across shared-secret matching, scope semantics, trusted-proxy identity, device pairing, and session attribution before coding.
