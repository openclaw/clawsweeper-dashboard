## Label Justifications

- `P2`: This is a normal-priority gateway auth/team-deployment feature with clear user value but no regression in an existing contract.
- `impact:session-state`: The request is specifically about isolating and attributing Control UI sessions per human user.
- `impact:security`: Multiple token identities and scopes would change the gateway authentication and authorization boundary.
- `impact:auth-provider`: The issue targets gateway auth token resolution and credential semantics for operator access.

## Summary

Keep open. Current main still supports one shared `gateway.auth.token`/password and uses trusted-proxy as the identity-bearing path; the requested per-token labels, scopes, and session attribution would add a new public auth/config contract that needs maintainer product and security review.

## What This Changes

Add a `gateway.auth.tokens` configuration surface that maps multiple bearer tokens to per-user labels/scopes for isolated and attributed multi-user Control UI sessions.

## Best Possible Solution

Design one cohesive gateway token-identity model covering config schema/types, token storage and rotation, scope semantics, session attribution, and composition with trusted-proxy, Tailscale, and device pairing.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a missing feature/config surface. Source inspection gives high confidence that current main accepts and resolves one shared gateway auth token/password, token auth returns no user identity, and the strict schema has no token registry.

## Solution Assessment

Unclear. A token registry is plausible, but maintainers need to decide whether token-derived labels/scopes are authoritative, display-only, or bound to device/session identity before implementation.
