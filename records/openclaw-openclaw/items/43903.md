## Summary

Keep open. The current Gateway auth path still supports one shared token/password or identity-bearing trusted-proxy auth, with no `gateway.auth.tokens` map, matched token principal, label/scope metadata, or Control UI user attribution backed by token identity. This is valid but security-sensitive Gateway auth/protocol work, so it needs maintainer design rather than conservative cleanup.

## What This Changes

Request support for multiple Gateway auth tokens with per-token labels/scopes so shared Control UI access can isolate users and attribution.

## Best Possible Solution

Define a backwards-compatible Gateway auth contract that preserves existing `gateway.auth.token`, adds validated multi-token entries, resolves the matched entry into a server-attested principal, and exposes attribution/scopes only through documented protocol and session surfaces.

## Reproduction Assessment

Not applicable as a feature request. The verification path is high-confidence source inspection: config type/schema, resolved auth, token authorization, connect auth schema, and chat send attribution all lack multi-token identity support on current main.

## Solution Assessment

Unclear. The proposed `tokens` map is plausible, but the best implementation needs maintainer/security design across matching semantics, scope composition, device pairing, trusted-proxy interaction, and Control UI/session attribution.
