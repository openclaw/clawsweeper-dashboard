## Summary

Keep open. Current main still models Gateway token auth as one shared secret, and the proposed `gateway.auth.tokens` map would be a new security-sensitive auth/config feature; the related #68353 attribution work is adjacent but does not replace per-token principals.

## What This Changes

Add multiple Gateway token entries with label/scope metadata so shared Control UI users can authenticate and be attributed separately while retaining token-mode CLI and pairing behavior.

## Best Possible Solution

Define a backwards-compatible Gateway auth principal contract that preserves `gateway.auth.token`, optionally adds validated multi-token entries, resolves the matched token into a server-attested principal, and coordinates that principal with Control UI/session attribution rather than relying on browser-local identity alone.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source inspection of the current config schema, resolved auth, token authorization, connect schema, chat send schema, transcript persistence, and Gateway auth docs.

## Solution Assessment

Unclear. The proposed `tokens` map is plausible, but the best solution needs maintainer/security design for token matching, scope composition, device pairing, trusted-proxy/Tailscale interaction, and how token identity feeds chat/session attribution.

## Risks / Open Questions

- A multi-token matcher could introduce token-entry enumeration or timing differences if matching, rate limiting, logging, and redaction are not designed uniformly.
- Token labels/scopes need an explicit authority model across shared tokens, device tokens, bootstrap tokens, trusted-proxy users, Tailscale identity, and HTTP scope headers.
- Per-token authentication alone may not solve user-visible Control UI/session attribution unless it is coordinated with the still-open #68353 `chat.send` author-attribution surface.
