- `P2`: This is a normal-priority team-deployment feature with clear user value but no regression in an existing supported contract.
- `impact:session-state`: The request is specifically about isolating and attributing Control UI sessions per human user.

## Summary

Keep open. Current main still has a single shared `gateway.auth.token`/password path plus separate identity-bearing trusted-proxy/Tailscale paths; the requested multi-token labels, scopes, and Control UI session attribution would add a new public gateway auth contract that needs maintainer, product, and security review.

## What This Changes

The issue requests multiple gateway auth tokens with per-token labels/scopes so shared Control UI access can isolate and attribute sessions per user.

## Best Possible Solution

Design one cohesive gateway token-identity model covering config schema/types, secure token storage and rotation, revocation/audit behavior, scope semantics, Control UI/session attribution, docs, tests, and composition with trusted-proxy, Tailscale, and device pairing.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a missing feature/config surface. Source inspection gives high confidence that current main accepts and resolves one shared gateway auth token/password and has no token registry.

## Solution Assessment

Unclear. A token registry is plausible, but maintainers need to decide whether token-derived labels/scopes are authoritative authorization facts, display metadata, or inputs to device/session identity before implementation.
