## Summary

Keep open: current main and v2026.6.5 still model token auth as one shared bearer secret, while trusted-proxy and the Pinchy external layer are useful workarounds rather than the requested token-mode per-user identity mapping. This remains a core gateway auth/session product and security design item, not stale cleanup.

## What This Changes

The issue requests multiple gateway bearer tokens with per-token user identity so shared Control UI deployments can isolate sessions per human user.

## Best Possible Solution

Make a maintainer product/security decision for a first-class gateway user identity model, then either add a compatible multi-token auth surface or explicitly route team identity to trusted-proxy or an external frontend layer while coordinating with https://github.com/openclaw/openclaw/issues/68353, https://github.com/openclaw/openclaw/issues/69066, and https://github.com/openclaw/openclaw/issues/89314.

## Reproduction Assessment

Yes, as a source-level limitation: current main has only gateway.auth.token in the config/schema and authorizeTokenAuth accepts one configured secret. I did not run a live Gateway repro because the missing capability is directly visible in the current auth and protocol contracts.

## Solution Assessment

Unclear: the proposed tokens object is plausible, but it is not yet proven as the best permanent design. It needs to be reconciled with trusted-proxy identity, device tokens, per-message author attribution, operator scopes, and the external Pinchy-style frontend pattern.
