## Label Justifications

- `P2`: This is a normal-priority gateway auth/team-deployment feature with clear user value but no current regression in an existing contract.
- `impact:session-state`: The request is specifically about isolating and attributing Control UI sessions per human user.
- `impact:security`: Multiple token identities and scopes would change the gateway authentication and authorization boundary.
- `impact:auth-provider`: The issue targets gateway auth token resolution and credential semantics for operator access.

## Summary

Keep open. Current main and v2026.5.18 still model token/password gateway auth as one shared operator secret, while the requested per-token identity, scopes, and session attribution would add a new security-sensitive public auth/config contract that needs maintainer product review.

## What This Changes

Request support for multiple gateway tokens with per-token labels/scopes so shared Control UI deployments can isolate and attribute sessions per user.

## Best Possible Solution

Design a cohesive gateway token-identity model covering config schema/types, token storage/rotation/revocation, scope semantics, session attribution, and composition with trusted-proxy, Tailscale, and device pairing.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a missing feature/config surface. Source inspection gives high confidence that current main accepts and resolves one shared gateway.auth.token/password, token auth returns no user identity, and the strict schema has no tokens registry.

## Solution Assessment

Unclear. A token registry is plausible, but maintainers need to settle whether token-derived labels/scopes are authoritative, display-only, or bound to device/session identity before implementation.
