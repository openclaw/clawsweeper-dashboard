- `impact:session-state`: The request is specifically about isolating and attributing Control UI sessions per human user.

## Summary

Keep open. Current main still has one shared `gateway.auth.token`/password path and separate identity-bearing trusted-proxy/Tailscale modes; the requested per-token labels, scopes, and Control UI session attribution would add a new public gateway auth/config contract that needs product and security review.

## What This Changes

The issue requests a `gateway.auth.tokens` registry that maps multiple bearer tokens to per-user labels/scopes for isolated multi-user Control UI sessions.

## Best Possible Solution

Design one cohesive gateway token-identity model covering config schema/types, secure token storage and rotation, scope semantics, Control UI/session attribution, docs, tests, and composition with trusted-proxy, Tailscale, and device pairing.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a missing feature/config surface. Source inspection gives high confidence that current main accepts and resolves one shared gateway auth token/password and has no token registry.

## Solution Assessment

Unclear. A token registry is plausible, but maintainers need to decide whether token-derived labels/scopes are authoritative, display-only, or bound to device/session identity before implementation.
