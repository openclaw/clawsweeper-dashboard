## Summary

Keep open. Current main and the latest release still model token auth as one shared operator secret, while the related author-attribution issue is adjacent and does not provide per-token identity mapping for token-mode Control UI sessions.

## What This Changes

The issue requests a multi-token `gateway.auth` configuration that maps each token to user identity and scopes for isolated multi-user Control UI sessions.

## Best Possible Solution

Define one cohesive gateway identity model for token-mode team access, covering token registry schema, label/scope semantics, revocation/rotation, session attribution, docs, audit checks, and composition with trusted-proxy and per-message author attribution.

## Reproduction Assessment

Yes for the current limitation by source inspection: current main and the latest release still resolve and compare one shared `gateway.auth.token`, not a per-token identity registry. This is a missing feature/config surface rather than a broken existing bug contract.

## Solution Assessment

Unclear. A `tokens` registry is plausible, but maintainers should first choose the trust model for labels, scopes, revocation, and overlap with trusted-proxy and `chat.send.author` attribution.
