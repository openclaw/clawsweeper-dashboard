## Summary

Keep open: current main and v2026.6.6 still implement Gateway shared-secret auth as one token/password plus identity-bearing trusted-proxy and device-token alternatives, while this asks for a new multi-principal Gateway auth/session-attribution design. The Pinchy workaround linked in discussion is useful evidence that front-layer identity can solve some deployments today, but it does not implement the requested core token-principal surface.

## What This Changes

Request support for multiple Gateway bearer tokens with per-token labels/scopes so Control UI/webchat users can keep separate identities and sessions.

## Best Possible Solution

Design one additive multi-principal Gateway auth model, or explicitly keep recommending trusted-proxy/front-layer patterns such as Pinchy for multi-user deployments until token-principal security semantics are approved.

## Reproduction Assessment

Yes for the current limitation by source inspection: current main accepts one gateway.auth.token and compares one presented token, with no tokens map/list or per-token identity reaching scopes or session metadata. This is a feature/security-design request, not a broken existing behavior report.

## Solution Assessment

Unclear: per-token identity is plausible, but the schema, scope vocabulary, attestation, audit/session attribution, and relationship to trusted-proxy/device-token/front-layer patterns need maintainer security and product design first.
