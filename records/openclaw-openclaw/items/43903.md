## Summary

Keep open: current main and v2026.6.6 still model Gateway token auth as one shared secret with no token-derived user identity, while the related scoped-token, author-attribution, and org-deployment issues are complementary rather than replacements.

## What This Changes

Requested support for multiple Gateway bearer tokens that map to per-user identity so shared Control UI deployments can isolate sessions without trusted-proxy auth.

## Best Possible Solution

Maintain this as the canonical per-user Control UI token/session-isolation design item, and have maintainers decide one Gateway principal model that composes token identity, scoped credentials, trusted-proxy, device pairing, and author attribution before implementation.

## Reproduction Assessment

Yes, as a source-level capability check. Current main and v2026.6.6 support singular shared token/password auth plus identity-bearing trusted-proxy/Tailscale paths, with no gateway.auth.tokens map/list or token-derived user label/session identity.

## Solution Assessment

Unclear as proposed. Per-token identity is a plausible Gateway-layer solution, but the exact token shape, scope semantics, session binding, device/trusted-proxy interaction, and author-attribution model need maintainer security/product direction first.
