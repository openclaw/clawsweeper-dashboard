## Summary

Keep open. Current main still models token auth as one shared gateway secret, while the identity-aware alternative is trusted-proxy; multiple named gateway tokens with labels/scopes would be a new auth/config feature with security and product semantics to define.

## What This Changes

The issue requests multiple gateway bearer tokens with per-token identity labels/scopes so shared Control UI users can have isolated or attributable sessions.

## Best Possible Solution

Define an additive gateway auth contract only after maintainer/security/product agreement: token identity mapping, scope semantics, local CLI and device-pairing behavior, Control UI session attribution, docs, and tests should line up with the existing trusted-proxy and session trust model.

## Reproduction Assessment

Yes. Source inspection is enough: current main has a single `gateway.auth.token`, resolves one shared token, compares one presented bearer token, and returns no per-token identity or scopes from token auth.

## Solution Assessment

Unclear. Multiple tokens may be a useful operator feature, but the exact proposed `tokens` object is not ready to implement until maintainers decide how it interacts with trusted-proxy identity, device pairing, scopes, and the documented non-isolation role of `sessionKey`.
