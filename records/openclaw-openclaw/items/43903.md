- `impact:session-state`: The requested token identity would affect how shared Control UI users map to isolated session history and metadata.

## Summary

Keep open: current main and v2026.6.8 still model token-mode Gateway auth as one shared bearer secret with no token-derived user label, scope binding, or session identity. The Pinchy/front-layer workaround is useful context, but it reinforces that this native Gateway identity primitive is still missing rather than replacing it.

## What This Changes

The issue requests multiple Gateway bearer tokens with per-token identity labels and scopes so shared Control UI/WebChat access can isolate sessions per user while staying in token auth mode.

## Best Possible Solution

Keep this as the canonical per-user Gateway token/session-isolation design item until maintainers choose one Gateway principal model that composes token identity, scoped credentials, trusted-proxy, device pairing, and author attribution.

## Reproduction Assessment

Yes, as a source-level capability check rather than a runtime bug repro. Current main and v2026.6.8 support singular shared token/password auth plus identity-bearing trusted-proxy/Tailscale paths, with no gateway.auth.tokens map/list or token-derived user label/session identity.

## Solution Assessment

Unclear as proposed. Per-token identity is a plausible Gateway-layer solution, but the exact token shape, scope semantics, session binding, device/trusted-proxy interaction, and author-attribution model need maintainer security/product direction first.
