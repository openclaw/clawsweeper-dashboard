## Summary

Keep open. Current main and the latest release still model token/password gateway auth as one shared operator secret, while identity-bearing behavior is handled by trusted-proxy/Tailscale/device-token paths; this is a valid security-sensitive feature/config request that needs maintainer product review, not cleanup closure.

## What This Changes

Request support for multiple gateway auth tokens with per-token labels/scopes so shared Control UI users can have separate identity and session attribution while retaining token-mode access.

## Best Possible Solution

Design a cohesive gateway token identity model with config schema/types, docs, tests, rotation/revocation semantics, scope rules, session attribution, and clear composition with trusted-proxy and device pairing.

## Reproduction Assessment

Yes, by source inspection: current main accepts and resolves one shared gateway.auth.token/password and token auth returns no user identity. The proposed tokens object would be rejected by the strict config schema.

## Solution Assessment

Unclear. A token registry is plausible, but maintainers need to settle the trust model for labels, scopes, rotation, revocation, and how token-derived identity should flow into chat/session attribution.
