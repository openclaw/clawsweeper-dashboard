## Summary

Keep open: current main and v2026.6.6 still model Gateway auth as one shared token/password plus identity-bearing trusted-proxy/device-token alternatives, while this asks for a new multi-principal gateway auth and session-attribution surface that needs maintainer security/product design.

## What This Changes

Add multiple Gateway token entries with per-token identity and scope mapping so Control UI/webchat users can be attributed and isolated by token.

## Best Possible Solution

Maintainers should design an additive multi-principal Gateway auth model that explicitly reuses or extends operator scopes, device tokens, trusted-proxy identity, session key derivation, docs, and upgrade rules before adding a new config surface.

## Reproduction Assessment

Yes, as source-level capability proof: current main accepts one `gateway.auth.token` and compares one presented token, with no `tokens` map/list or per-token identity reaching scopes or session metadata.

## Solution Assessment

Unclear: per-token identity is plausible, but the exact schema, scope vocabulary, attestation, and relationship to trusted-proxy, device tokens, Pinchy-style front layers, and related author-attribution issues need maintainer security/product design first.
