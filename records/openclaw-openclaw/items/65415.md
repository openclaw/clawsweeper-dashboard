## What This Changes

Adds an optional `agents.defaults.subagents.gatewayTimeoutMs` config field, applies it as a floor inside `callSubagentGateway`, and adds a focused subagent spawn timeout test.

## Best Possible Solution

Decide whether the public knob is wanted; if yes, rebase it as a raise-only floor over the existing subagent spawn/control timeout helpers and document/test the accepted config contract.

## Reproduction Assessment

Yes for the review findings: source inspection shows current main already has 60s/run-timeout-aware budgets and direct 10s control paths that the branch must preserve or cover. The original 40+ lane gateway timeout symptom was not live-reproduced in this review.

## Solution Assessment

No. The submitted wrapper-only approach is directionally related, but the maintainable version needs a product decision and must layer on top of current timeout helpers instead of replacing or bypassing them.
