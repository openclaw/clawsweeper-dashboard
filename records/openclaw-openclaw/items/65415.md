## Summary

Keep open for maintainer review: the branch is a new public config surface, current main has already changed the timeout and announce-delivery boundary, and the proposed wrapper still has correctness gaps plus missing real behavior proof.

## What This Changes

The PR adds an optional `agents.defaults.subagents.gatewayTimeoutMs` config/schema key, applies it as a timeout floor in `callSubagentGateway`, and adds a focused subagent spawn timeout unit test.

## Best Possible Solution

If maintainers want this knob, rework it as a documented raise-only floor over the current spawn/control timeout helpers and include the direct subagent-control paths; otherwise keep the current internal timeout and in-process announce fixes without adding a new config surface.

## Reproduction Assessment

Yes for the review findings: source inspection on current main shows 60s/run-timeout-aware spawn budgets and a direct 10s control path outside the wrapper. The original 40+ lane gateway timeout symptom was not live-reproduced in this read-only review.

## Solution Assessment

No. A wrapper-only timeout floor with a new public config key is not the best fix unless maintainers first approve the config surface and it is layered over current helpers, direct control calls, docs, and proof.
