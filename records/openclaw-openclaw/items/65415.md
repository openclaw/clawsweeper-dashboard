## Summary

Keep open for maintainer judgment: the PR is directionally related to real subagent timeout pain, but the patch is not correct against current main and still lacks contributor real behavior proof.

## What This Changes

Adds optional `agents.defaults.subagents.gatewayTimeoutMs` config/schema typing, applies it as a floor in `callSubagentGateway`, and adds a subagent spawn timeout test.

## Best Possible Solution

Decide whether a public gateway-timeout knob is wanted; if yes, rework it as a raise-only floor over current spawn/control timeout helpers, cover direct control paths, update docs/tests, and require real behavior proof.

## Reproduction Assessment

Yes for the review findings: source inspection on current main shows existing 60s/run-timeout-aware budgets and direct 10s control paths that the branch does not preserve or cover. The original 40+ lane gateway timeout symptom was not live-reproduced in this read-only review.

## Solution Assessment

No. The wrapper-only approach is related to the problem, but it should not replace current timeout helpers, and the public config surface needs maintainer approval before it is the right fix.
