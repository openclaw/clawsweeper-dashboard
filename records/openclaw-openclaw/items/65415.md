## Summary

Keep open: the reliability problem is plausible and current main still does not expose `gatewayTimeoutMs`, but the branch is stale against current 60s/run-timeout-aware gateway budgets and needs maintainer API judgment plus mechanical schema/changelog fixes before merge.

## What This Changes

The PR adds an optional `agents.defaults.subagents.gatewayTimeoutMs` config key, applies it as a floor in `callSubagentGateway`, and adds focused spawn timeout tests.

## Best Possible Solution

Decide whether this public operator knob is wanted; if yes, rebase it onto current timeout helpers so it can only raise existing spawn/control budgets and update generated schema, docs/changelog, and focused tests.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR adds a Zod config key without the generated schema baseline, and current main already has different timeout helper behavior the branch must preserve. The original 40+ lane gateway timeout was not live-reproduced in this read-only review.

## Solution Assessment

No. The submitted approach is directionally related, but it is not the best current solution because it is stale relative to current main's 60s/run-timeout-aware budgets and treats the new config as a replacement floor rather than a raise-only operator floor.
