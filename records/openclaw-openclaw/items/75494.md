## Summary

Current main shows the reported status and visible-reply behavior are expected configuration semantics rather than a Telegram routing defect: `groups:unmentioned` means unmentioned group messages are allowed, and group/channel final replies are private by default unless the agent uses the message tool or `messages.groupChat.visibleReplies` is set to `automatic`.

## What This Changes

The issue asks OpenClaw to reply visibly in a Telegram supergroup with `requireMention:false` and reports `channels status --probe` showing `groups:unmentioned`.

## Best Possible Solution

Keep the current message-tool-first group safety default, and have users who want DM-like visible group replies set `messages.groupChat.visibleReplies: "automatic"`; if a failure remains after that, it should be filed with sanitized config and a single-turn Telegram log showing the actual delivery path.

## Reproduction Assessment

Yes for the visible-reply symptom: current code and tests show a group turn can process with typing/reactions while normal final text is not posted under the default message-tool-only policy. No live Telegram container run was needed to explain the reported `groups:unmentioned` status or the missing final reply.

## Solution Assessment

Yes. The narrowest maintainable answer is configuration/docs guidance, because changing Telegram routing to auto-post all group finals would bypass the intentional all-channel group visibility policy.
