## Summary

Keep this PR open for maintainer review. Current main still has no visionOS app, no visionOS node docs, and no visionOS platform policy, but the proposed branch is not mergeable as-is because it lacks build integration and still disagrees with current Gateway node protocol contracts.

## What This Changes

The PR adds a new SwiftUI visionOS node app under `apps/visionos/` plus setup/command documentation for pairing an Apple Vision Pro with the OpenClaw Gateway as a spatial peripheral node.

## Best Possible Solution

If OpenClaw wants a first-party visionOS node, land it as a complete Apple app surface: add a buildable project or package target, resolve #55944 or equivalent policy support, align invoke/APNs/error handling with current Gateway contracts, update docs/labeler/changelog/navigation, and require maintainer security plus real-device validation before merge.

Yes for the source-level review findings: current main shows the Gateway sends `paramsJSON`, validates invoke errors with `message`, and handles APNs through `node.event`, while the PR uses incompatible shapes. Real-device runtime behavior remains unverified because this was a read-only source review.

## Solution Assessment

No. The proposed direction may be useful, but this branch is not the best mergeable solution until it includes build integration and matches the current node protocol, config, and platform-policy contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Add a buildable visionOS project target:** `apps/visionos/README.md:17`
  - body: The README tells users to open `visionOS-node/visionOS-node.xcodeproj`, but this PR does not add an Xcode project, Package.swift, or integration into an existing Apple build. A fresh checkout cannot build or run the app as documented.
  - confidence: 0.91
- **[P1] Decode paramsJSON before dispatching invokes:** `apps/visionos/Sources/NodeManager.swift:261`
  - body: Gateway `node.invoke.request` payloads carry arguments in `paramsJSON`; this handler only reads `payload["params"]`. Commands that rely on parameters will be dispatched with an empty dictionary against current Gateway behavior.
  - confidence: 0.94
- **[P1] Use message in invoke error payloads:** `apps/visionos/Sources/NodeManager.swift:160-162`
  - body: `node.invoke.result` errors are validated as `{ code, message }` with extra properties rejected. Sending `detail` makes error results invalid, so callers can see failed acknowledgements or timeouts instead of the intended error.
  - confidence: 0.93
- **[P2] Publish APNs registration through node.event:** `apps/visionos/Sources/APNsManager.swift:61`
  - body: The Gateway does not expose a `node.apns` method; APNs registration is handled as `node.event` with `event: "push.apns.register"`. This request will be rejected and APNs wake registration will not be stored.
  - confidence: 0.91
- **[P2] Use the gateway.nodes allowlist path in docs:** `docs/nodes/visionos.md:100-101`
  - body: This example puts `allowCommands` at the config root, but the current config contract uses `gateway.nodes.allowCommands`. Users following this snippet will not enable the spatial commands they need.
  - confidence: 0.88
- **[P2] Cancel stale reconnects before connecting:** `apps/visionos/Sources/NodeManager.swift:132-135`
  - body: A scheduled reconnect can still fire after a manual or successful reconnect and call `connect()` again. Since `GatewaySocket.connect()` starts by disconnecting, that stale task can drop a healthy session.
  - confidence: 0.82
- **[P2] Avoid closing a newer socket from an old disconnect task:** `apps/visionos/Sources/NodeManager.swift:117-119`
  - body: `disconnect()` waits asynchronously for the exit frame and then calls `socket.disconnect()` on the shared socket. If the user reconnects during that wait, the old task can close the new connection.
  - confidence: 0.8
