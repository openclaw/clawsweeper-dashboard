## Summary

Keep this PR open because current main still has no visionOS app or docs, but the branch is not mergeable as-is: it lacks a buildable Apple project target and has current Gateway protocol/config mismatches, with related platform-policy work still open in #55944.

## What This Changes

The PR adds a new SwiftUI `apps/visionos` node app and `docs/nodes/visionos.md` for pairing an Apple Vision Pro with the Gateway as a spatial peripheral.

## Best Possible Solution

Land a first-party visionOS node only after it has a buildable Apple target, current Gateway node protocol compatibility, platform-policy support, repo metadata/docs/changelog wiring, maintainer security review, and real-device validation.

## Reproduction Assessment

Yes for the source-level review findings: current main shows `paramsJSON`, `node.invoke.result` error schema, `push.apns.register`, and `gateway.nodes.allowCommands` contracts that the PR does not match. I did not run a Vision Pro/Xcode build in this read-only review.

## Solution Assessment

No. The first-party visionOS direction may be useful, but this branch is not the best mergeable solution until it includes a build target and matches current Gateway node protocol, config, policy, and repository integration contracts.

Full review comments:

- **[P1] Add a buildable visionOS project target:** `apps/visionos/README.md:17`
  - body: The README tells users to open `visionOS-node/visionOS-node.xcodeproj`, but this PR does not add an Xcode project, Package.swift, XcodeGen project, or other Apple build target. A fresh checkout cannot build or run the app as documented.
  - confidence: 0.92
- **[P1] Decode paramsJSON before dispatching invokes:** `apps/visionos/Sources/NodeManager.swift:261`
  - body: Gateway `node.invoke.request` payloads carry arguments in `paramsJSON`, but this handler only reads `payload["params"]`. Current invoke params are dispatched as `{}`, and the claimed smuggled-approval validation never sees fields delivered by the current Gateway.
  - confidence: 0.94
- **[P1] Use message in invoke error payloads:** `apps/visionos/Sources/NodeManager.swift:160-162`
  - body: `node.invoke.result` errors are validated as `{ code, message }` with extra properties rejected. Sending `detail` makes error results invalid, so callers can see failed acknowledgements or timeouts instead of the intended error.
  - confidence: 0.93
- **[P2] Publish APNs registration through node.event:** `apps/visionos/Sources/APNsManager.swift:61`
  - body: The Gateway does not expose a `node.apns` method; APNs registration is handled as `node.event` with `event: "push.apns.register"`. This request will be rejected and APNs wake registration will not be stored.
  - confidence: 0.91
- **[P2] Use the gateway.nodes allowlist path in docs:** `docs/nodes/visionos.md:100-101`
  - body: This example puts `allowCommands` at the config root, but the current config contract uses `gateway.nodes.allowCommands`. Users following this snippet will not enable the spatial or camera commands they need.
  - confidence: 0.9
- **[P2] Cancel stale reconnects before connecting:** `apps/visionos/Sources/NodeManager.swift:132-135`
  - body: A scheduled reconnect can still fire after a manual or successful reconnect and call `connect()` again. Since `GatewaySocket.connect()` starts by disconnecting, that stale task can drop a healthy session.
  - confidence: 0.82
- **[P2] Avoid closing a newer socket from an old disconnect task:** `apps/visionos/Sources/NodeManager.swift:117-119`
  - body: `disconnect()` waits asynchronously for the exit frame and then calls `socket.disconnect()` on the shared socket. If the user reconnects during that wait, the old task can close the new connection.
  - confidence: 0.8
- **[P2] Wire the new app into repo metadata:** `apps/visionos/README.md:1`
  - body: This adds a user-facing app and docs page, but does not update the required changelog, docs navigation, or `.github/labeler.yml` app label coverage. Release notes, docs discovery, and PR labeling will miss the new surface.
  - confidence: 0.86
- **[P3] Fix README links to the visionOS docs:** `apps/visionos/README.md:11`
  - body: From `apps/visionos/README.md`, `docs/nodes/visionos.md` resolves under `apps/visionos/docs/...`, which does not exist. Use the public docs URL or a correct repo-relative path.
  - confidence: 0.87
