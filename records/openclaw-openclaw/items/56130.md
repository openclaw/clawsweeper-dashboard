## Summary

Keep this PR open for maintainer review. Current main still has no visionOS node app or docs, the paired visionOS platform-policy work remains open, and the PR branch has concrete gateway protocol mismatches that would break invoke params, APNs registration, and error reporting.

## What This Changes

Adds a new native SwiftUI visionOS node app under `apps/visionos/` plus `docs/nodes/visionos.md` for Apple Vision Pro spatial, location, APNs, and Gateway WebSocket integration.

## Best Possible Solution

If OpenClaw wants a first-party visionOS node, land it as a complete maintained app surface: align the app with current node protocol contracts, resolve #55944 or fold equivalent platform policy into the stack, add docs navigation and labeler coverage, and require Xcode plus physical Apple Vision Pro validation before merge.

## Reproduction Assessment

Yes for the blocking protocol bugs: current main statically shows `node.invoke.request` uses `paramsJSON`, `node.invoke.result.error` accepts `message` only, and APNs registration is handled through `node.event`/`push.apns.register`. No live Vision Pro reproduction was available for the ARKit and native UI claims.

## Solution Assessment

No, this PR is not mergeable as-is because it implements a plausible app but does not yet match the current Gateway node wire contract. The maintainable path is to fix the protocol mismatches and platform-policy dependency first, then validate the native app on device.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Decode invoke paramsJSON before dispatching commands:** `apps/visionos/Sources/NodeManager.swift:261-263`
  - body: Current Gateway `node.invoke.request` events carry command arguments in `paramsJSON`, but this handler only reads `payload["params"]`. Any command with arguments will be dispatched with an empty params object, producing incorrect behavior against current main.
  - confidence: 0.97
- **[P2] Send invoke errors with message, not detail:** `apps/visionos/Sources/NodeManager.swift:158-170`
  - body: `node.invoke.result` errors are validated with `additionalProperties: false` and only allow `code` and `message`. Building errors with `detail` makes error responses invalid, so callers can see timeouts or opaque failures instead of the intended command error.
  - confidence: 0.96
- **[P2] Register APNs through the supported node event:** `apps/visionos/Sources/APNsManager.swift:59-62`
  - body: The gateway handles APNs registration as a `node.event` named `push.apns.register`; there is no supported `node.apns` RPC method. Sending the token through `node.apns` means APNs registration will be rejected or ignored, so wake/notification flows cannot work.
  - confidence: 0.95
- **[P2] Restrict failed response handling to connect responses:** `apps/visionos/Sources/GatewaySocket.swift:158-177`
  - body: This treats every `res` frame with `ok:false` as a handshake failure. Once connected, ordinary request failures from app-originated RPCs can incorrectly move the whole node to error state even though the socket is still healthy.
  - confidence: 0.88
- **[P2] Cancel pending reconnects when a connection succeeds:** `apps/visionos/Sources/NodeManager.swift:127-136`
  - body: A delayed reconnect task is not cleared by `connect()` or by a successful handshake. If the user reconnects manually before the delay expires, the old task can still fire and call `connect()`, whose socket path disconnects the healthy session first.
  - confidence: 0.84
- **[P2] Avoid closing a new socket from an old disconnect task:** `apps/visionos/Sources/NodeManager.swift:114-120`
  - body: `disconnect()` waits for `sendExitFrame()` in an async task before closing `socket`. If the user reconnects during that wait, the old task can later call `socket.disconnect()` on the newly opened connection.
  - confidence: 0.82
