## Summary

Keep open: this is a substantial first-party Vision Pro node feature, but the current branch still has source-reproducible protocol/build blockers, missing real-device proof for the changed behavior, and an open platform-policy dependency.

## What This Changes

Adds a SwiftUI visionOS node app under `apps/visionos` plus `docs/nodes/visionos.md` for Gateway pairing, spatial commands, location, camera placeholders, APNs, and setup.

## Best Possible Solution

Land a repo-buildable visionOS node that uses the current node.invoke, node.invoke.result, and node.event contracts, with docs aligned to gateway.nodes config and real Vision Pro plus Gateway proof.

## Reproduction Assessment

Yes for the source-level PR findings: current main shows `paramsJSON` invoke requests, `node.invoke.result` error `message` validation, `push.apns.register` event handling, and missing `visionos` platform policy. I did not run Xcode or Vision Pro hardware in this read-only review.

## Solution Assessment

No. A first-party Vision Pro node may be the right direction, but this branch needs build-target, protocol, APNs, policy, proof, and security fixes before it is a maintainable merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Add a buildable visionOS target:** `apps/visionos/README.md:17`
  - body: The README tells users to open `visionOS-node/visionOS-node.xcodeproj`, but the PR does not add an Xcode project, Swift package, XcodeGen spec, Tuist project, or other build target. A fresh checkout cannot build or run the app as documented.
  - confidence: 0.92
- **[P1] Decode paramsJSON before dispatching invokes:** `apps/visionos/Sources/NodeManager.swift:261`
  - body: Current Gateway `node.invoke.request` events carry command arguments in `paramsJSON`, but this handler only reads `payload["params"]`. Commands receive empty params, and the approval-smuggling validation never inspects the argument string delivered by current Gateway.
  - confidence: 0.94
- **[P1] Use message in invoke error payloads:** `apps/visionos/Sources/NodeManager.swift:158-161`
  - body: `sendError` emits `error.detail`, but the current `node.invoke.result` schema accepts only `error.code` and `error.message` and rejects additional properties. Failed command responses can therefore be invalid and surface as timeouts or opaque failures.
  - confidence: 0.91
- **[P2] Register APNs through node.event:** `apps/visionos/Sources/APNsManager.swift:59-61`
  - body: This sends the APNs token with method `node.apns`, which is not a supported Gateway method. Current Gateway handles APNs registration through `node.event` with `event: "push.apns.register"`, so this registration path will be rejected.
  - confidence: 0.88
- **[P2] Limit handshake failures to connect responses:** `apps/visionos/Sources/GatewaySocket.swift:165-176`
  - body: The socket treats every `ok:false` response as a handshake error. After the node is connected, normal request failures such as an unsupported APNs registration response can incorrectly flip the UI into a handshake error even though the WebSocket is still healthy.
  - confidence: 0.78
