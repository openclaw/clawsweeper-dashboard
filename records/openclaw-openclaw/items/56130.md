Full review comments:

- **[P1] Add a buildable visionOS target:** `apps/visionos/README.md:17`
  - body: The branch adds Swift sources and docs, but no Xcode project, workspace, `Package.swift`, or project file under `apps/visionos`. Since the README tells users to open `visionOS-node/visionOS-node.xcodeproj`, a fresh OpenClaw checkout cannot build or run the app being merged.
  - confidence: 0.94
- **[P1] Decode paramsJSON before dispatch:** `apps/visionos/Sources/NodeManager.swift:257-263`
  - body: Current Gateway invokes are delivered with `paramsJSON`, but this handler only reads `payload["params"]`. Commands that rely on arguments will run with empty/default params, and the smuggled-approval check will miss fields sent through the normal wire contract.
  - confidence: 0.96
- **[P1] Use message in invoke error payloads:** `apps/visionos/Sources/NodeManager.swift:158-161`
  - body: `NodeInvokeResultParamsSchema` accepts `error.code` and `error.message`; this sends `detail`, which is an additional property and makes error results invalid. Failing commands can then time out or surface opaque Gateway errors instead of the intended node error.
  - confidence: 0.95
- **[P1] Register APNs through node.event:** `apps/visionos/Sources/APNsManager.swift:59-61`
  - body: The Gateway handles APNs registration through `node.event` with `event: "push.apns.register"`; `node.apns` is not a supported request method. This path will be rejected, so APNs wake/notification registration cannot work as written.
  - confidence: 0.94
- **[P2] Do not treat RPC failures as handshake failures:** `apps/visionos/Sources/GatewaySocket.swift:165-176`
  - body: After connect, ordinary request responses can be `ok:false`; this parser forwards every such response to the handshake-error path. A rejected later request, including the unsupported APNs registration, can incorrectly put the node into an error state while the socket is otherwise healthy.
  - confidence: 0.86
- **[P2] Guard delayed reconnects before reconnecting:** `apps/visionos/Sources/NodeManager.swift:132-135`
  - body: A scheduled reconnect always calls `connect()` after sleeping. If the user or app has already reconnected, that delayed task can still fire and `GatewaySocket.connect()` starts by disconnecting the existing socket, dropping a healthy session.
  - confidence: 0.84
- **[P2] Avoid closing newer sockets from stale disconnect tasks:** `apps/visionos/Sources/NodeManager.swift:117-119`
  - body: `disconnect()` waits in an async task before calling `socket.disconnect()`. If a new connection is established during that wait, the old task can close the new socket and create intermittent disconnects after reconnect.
  - confidence: 0.82
