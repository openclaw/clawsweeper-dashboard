## What This Changes

The PR adds `compat.systemPromptPlacement` for OpenAI Responses models so configured proxies can move `context.systemPrompt` from `input` into top-level `instructions`, with docs and targeted tests.

## Best Possible Solution

Land this only after the compat key is wired through all OpenAI Responses request builders and catalog normalization, with default `input` behavior preserved and redacted real provider proof showing the stricter proxy succeeds.

## Reproduction Assessment

Yes, source-level: current main places non-Codex OpenAI Responses system prompts in `input` through `convertResponsesMessages`, so a strict proxy rejecting system/developer input items would still fail. I did not run a live proxy reproduction.

## Solution Assessment

No: moving the prompt into `instructions` is a plausible fix, but this implementation is incomplete because it does not cover the registered built-in Responses provider or catalog metadata path.
