## What This Changes

The PR adds `compat.systemPromptPlacement` so strict OpenAI Responses-compatible providers can receive `context.systemPrompt` as top-level `instructions` instead of `system` or `developer` items in `input`.

## Best Possible Solution

Land the opt-in compat switch after maintainer acceptance of the new config surface and normal CI/changed-gate proof, while keeping existing Responses models on the current `input` default.

## Reproduction Assessment

Yes for the source path: current main includes system/developer messages in Responses `input` by default, which matches the reported strict-proxy rejection mode when a proxy disallows those roles. I did not run the reporter's local gateway.

## Solution Assessment

Yes, assuming maintainers accept the new compat key: the opt-in switch is narrow, preserves current defaults, and matches the top-level instructions shape used by Codex and supported by the OpenAI SDK typings. A provider-specific hard default would be riskier because it would change existing configured models.
