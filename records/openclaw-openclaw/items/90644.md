## Summary

Keep open: this is a focused fix for a source-reproducible Telegram group native-command bug, but the current patch is too narrow because it only trusts account/global open group policy and still misses scoped group/topic `groupPolicy: "open"` overrides; it also lacks real Telegram behavior proof.

## What This Changes

The PR changes Telegram native command authorization so open group policy can authorize group commands when no explicit command sender allowlist is configured, and adds a regression test for `/status` in open groups.

## Best Possible Solution

Update the native-command auth path to honor the same resolved scoped group/topic policy as normal Telegram messages, keep `commands.allowFrom` as the explicit sole-source override when configured, and require redacted real Telegram proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main can pass Telegram group policy and still fail the later command ingress allowlist for native commands without sender allowlist entries. The PR also demonstrates the account-level open-policy case with a focused unit test, but not the scoped override case.

## Solution Assessment

No. The direction is right, but the patch is not the best complete fix because it reads a policy result that ignores group/topic overrides, while normal Telegram message routing honors those overrides.
