## Summary

Keep open: this PR still addresses a real Telegram group native-command gap that is present on current main and the latest release. The new Mantis Telegram Desktop proof resolves the previous real-behavior proof blocker, but the PR intentionally broadens native command authorization for open Telegram groups, so it still needs maintainer security-boundary acceptance before merge.

## What This Changes

The PR changes Telegram native command authorization to honor scoped open group/topic policy and treat resolved open group policy as sufficient when `commands.allowFrom` is unset, with regression coverage for `/status` in open groups.

## Best Possible Solution

Land the narrow Telegram-owned auth fix after maintainers accept that `groupPolicy: "open"` authorizes native commands when `commands.allowFrom` is unset and normal merge gates pass.

## Reproduction Assessment

Yes, source-level: current main can pass Telegram group policy and still fail the later command ingress allowlist gate when no command sender allowlist is configured. The Mantis run proves the PR-head Telegram Desktop behavior, while current main has the same relevant source shape as the tested baseline.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix because it reuses the resolved Telegram group policy after base/group checks and leaves `commands.allowFrom` as the sole command auth source when configured; changing the shared command ingress helper would risk broadening other text-command and event callers.
