## Summary

Keep this PR open, but do not merge it as-is. The targeted bug is real on current main, but the latest diff drops cron-originated wake targeting entirely in `scopedHeartbeatWakeOptions`, which can turn one agent's cron exec completion into an unscoped heartbeat wake across enabled agents.

## What This Changes

The PR changes heartbeat wake scoping and forced heartbeat session resolution so cron run session keys are not used as heartbeat targets, with focused tests for the session-key helper and `runHeartbeatOnce` fallback behavior.

## Best Possible Solution

Revise the patch so heartbeat never targets isolated cron run sessions while preserving the originating agent boundary, with multi-agent regression coverage for cron exec wake routing.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: with the PR diff, `scopedHeartbeatWakeOptions` on an agent cron session returns no `sessionKey` or `agentId`, and current wake dispatch treats that as an unscoped wake across enabled heartbeat agents.

## Solution Assessment

No. Blocking direct cron-session heartbeat execution is the right direction, but dropping all target context is not the narrowest maintainable fix; preserving the parsed agent target or mapping to that agent's main heartbeat session is safer.
