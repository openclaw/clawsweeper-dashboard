## Summary

Keep open: the PR targets a real source-reproducible heartbeat/cron bug, but the patch regresses targeted wakes by dropping both `sessionKey` and `agentId` for cron exec completions, which can fan out heartbeats to every enabled agent. It also lacks after-fix real behavior proof, so it is not merge-ready.

## What This Changes

The PR changes heartbeat wake scoping and heartbeat session resolution so cron run session keys fall back to an agent main heartbeat session, with focused regression tests.

## Best Possible Solution

Reject cron run session keys as heartbeat sessions while preserving the originating agent as the wake target, and add scheduler-level coverage for multi-agent cron exec completion wakes.

## Reproduction Assessment

Yes, at source level. Current main forwards cron run session keys from exec completion into heartbeat session resolution, and the PR diff can be traced to an unscoped wake when it removes that session key without adding `agentId`.

## Solution Assessment

No. The right direction is to keep heartbeat prompts out of cron run sessions, but the implementation must carry the parsed agent id forward so the wake falls back to that agent's main session instead of all agents.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve agent targeting when dropping cron session keys:** `src/routing/session-key.ts:43`

In `startHeartbeatRunner`, that skips the targeted branch and iterates every enabled agent, so a background exec exit in `agent:secondary:cron:...` can run unrelated heartbeats instead of the originating agent's main session.
