## Summary

Keep this PR open. The underlying cron/heartbeat bug still exists on current main, but the proposed patch is not merge-ready because dropping cron session keys from wake options also drops the originating agent target, turning a scoped exec-completion wake into an unscoped heartbeat wake.

## What This Changes

The PR changes heartbeat wake option construction and forced heartbeat session resolution so cron session keys are not used as heartbeat targets, with focused regression tests for session-key helpers and runHeartbeatOnce fallback behavior.

## Best Possible Solution

Revise the patch so heartbeat never runs inside isolated cron run sessions while preserving the originating agent boundary, likely by converting cron session keys into an agentId-scoped wake or that agent's main heartbeat session, with regression coverage for non-default and multi-agent cron exec wakes.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: with the PR diff, scopedHeartbeatWakeOptions on an agent cron session returns no sessionKey or agentId, and current wake dispatch treats that as an unscoped wake across enabled heartbeat agents.

## Solution Assessment

No. Blocking direct cron-session heartbeat execution is the right direction, but dropping all target context is not the narrowest maintainable fix; preserving the parsed agent target is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve agent scope when dropping cron keys:** `src/routing/session-key.ts:46`
  - body: Returning wakeOptions unchanged for cron session keys removes the only target data exec completion callers pass into requestHeartbeatNow. The resulting wake has neither sessionKey nor agentId, so startHeartbeatRunner can run every enabled heartbeat agent instead of the originating cron job's agent.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff has no dependency, workflow, or supply-chain changes, but it introduces a concrete cross-agent routing risk by turning cron-originated heartbeat wakes into unscoped wakes.

Concerns:

  - body: When the cron session key is dropped without an agentId or main-session replacement, a wake from one cron job can run unrelated agents' heartbeat prompts and sends outside the originating agent boundary.
