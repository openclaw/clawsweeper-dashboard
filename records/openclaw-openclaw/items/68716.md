## Summary

Keep open: this targets a real Discord authorization gap, but the current branch is merge-conflicting, lacks real Discord proof, and its runtime missing-sender guard would break trusted CLI/manual owner guild-admin actions that carry `senderIsOwner` without `requesterSenderId`.

## What This Changes

The PR adds Discord trusted-requester gating and guild-admin permission checks for privileged guild-admin message actions, plus focused regression tests.

## Best Possible Solution

Land a rebased Discord hardening that requires Discord requester identity and matching guild permissions for Discord-origin guild-admin tool calls while preserving trusted owner/manual execution through an explicit trust signal or scoped runtime check.

## Reproduction Assessment

Yes. Source inspection on current main shows `channel-delete` can reach `deleteChannelDiscord` without a trusted-requester hook or `senderUserId`, and the PR regression is source-reproducible through the CLI owner path that carries `senderIsOwner` but no `requesterSenderId`.

## Solution Assessment

No. The two-layer Discord requester-authz direction is right, but unconditional runtime rejection when `senderUserId` is absent is too broad for trusted owner/manual paths.
