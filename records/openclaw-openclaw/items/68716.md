## Summary

Keep open. The underlying Discord guild-admin authorization bug is valid and current main still lacks the requested protection, but this PR is not merge-ready because the new runtime guard also rejects trusted owner CLI/admin guild actions that do not have a Discord requester sender id.

## What This Changes

The PR adds trusted-requester gating for Discord guild-admin message actions, forwards `requesterSenderId` as `senderUserId`, adds guild-admin runtime permission checks, and adds authorization regression tests.

## Best Possible Solution

Keep the Discord guild-admin authorization hardening, but model the trust boundary explicitly: require server-injected requester identity and Discord permission checks for Discord-origin tool actions, while preserving trusted owner CLI/manual admin flows through a scoped owner/trusted-execution signal or an equivalent runtime contract. Add regression coverage for both unauthorized Discord requester rejection and owner CLI/direct runtime guild-admin success.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `openclaw message` sets `senderIsOwner` but does not pass `requesterSenderId`, then the PR maps missing `requesterSenderId` to missing `senderUserId` and the new runtime guard throws before guild-admin execution.

## Solution Assessment

No. The direction is right, but this implementation puts a mandatory Discord requester-id check in a runtime layer that cannot distinguish untrusted Discord-origin tool calls from trusted local owner/admin calls; the safer solution is to carry explicit execution trust context or scope the mandatory check to requester-driven paths.

## Review Findings

Overall correctness: patch is incorrect

Carry a trusted-owner signal through this path or limit the mandatory Discord requester check to tool-driven requester contexts.
