## Summary

Keep open: the Discord guild-admin authorization gap still exists on current main, but this PR is not merge-ready because the same reviewed head still lacks real Discord proof, is merge-conflicting, and introduces a trusted-owner/CLI regression.

## What This Changes

This PR adds Discord trusted-requester gating and guild-admin permission checks for privileged message actions, with focused regression tests.

## Best Possible Solution

Land a corrected Discord hardening that requires requester identity for Discord-origin tool actions, preserves trusted owner/manual execution with an explicit trust signal or scoped runtime checks, and includes focused tests plus real Discord proof.

## Reproduction Assessment

Yes. Source inspection shows current main routes guild-admin actions through Discord without requiresTrustedRequesterSender and then runs bot-credentialed guild mutations without requester permission checks; the PR regression is also source-reproducible through openclaw message paths that carry senderIsOwner but no requesterSenderId.

## Solution Assessment

No. The trusted-requester gate is the right direction, but the runtime-level missing-sender rejection is broader than the established moderation boundary; it should be scoped to requester-driven contexts or carry explicit trusted-owner context.
