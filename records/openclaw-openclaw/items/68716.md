## Summary

Keep open. The PR addresses a real Discord guild-admin authorization gap, but the current patch still blocks trusted local owner guild-admin flows and lacks after-fix real Discord behavior proof.

## What This Changes

The PR adds Discord guild-admin trusted-requester gating, forwards requesterSenderId as senderUserId, enforces guild permission checks in the guild runtime, and adds focused regression tests.

## Best Possible Solution

Keep the Discord hardening direction, but preserve trusted owner/manual execution with an explicit trusted execution signal or requester-context-only missing-sender failure, then add focused regression coverage and real Discord proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main lacks the Discord guild-admin trusted-requester hook and runtime permission check, and the PR regression is reached by local openclaw message flows that carry senderIsOwner but no requesterSenderId.

## Solution Assessment

No. The direction is right, but the runtime-level missing-sender rejection is broader than the established moderation trust boundary; the safer fix distinguishes untrusted Discord requester-driven calls from trusted owner/manual execution.
