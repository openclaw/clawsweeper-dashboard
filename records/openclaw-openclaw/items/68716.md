## Summary

Keep open: this PR targets a real Discord guild-admin requester-authorization gap on current main, but the current patch has a blocking trusted-owner regression and still lacks after-fix real Discord behavior proof.

## What This Changes

The PR adds trusted-requester gating, threads requesterSenderId as senderUserId into Discord guild-admin actions, verifies Discord permissions in the guild runtime, and adds authorization regression tests.

## Best Possible Solution

Keep the Discord hardening direction, but preserve trusted owner/manual execution by carrying an explicit trusted execution signal or limiting mandatory sender checks to requester-driven Discord contexts, then add focused regression coverage and redacted real Discord proof.

## Reproduction Assessment

Yes. Source inspection shows current main advertises Discord guild-admin message actions without a trusted-requester hook and the guild runtime performs bot-credentialed mutations after only feature gates; I did not run a live Discord repro in this read-only review.

## Solution Assessment

No. The PR’s direction is right, but unconditional runtime rejection when senderUserId is absent is broader than the established moderation and CLI trust boundary; the safer fix distinguishes requester-driven Discord calls from trusted owner/manual execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve trusted owner guild actions without sender IDs:** `extensions/discord/src/actions/runtime.guild.ts:184`
  - body: Do not reject every privileged guild action when `senderUserId` is absent. Local `openclaw message` calls default `senderIsOwner` to true but do not pass `requesterSenderId`, and this PR forwards only `requesterSenderId` as `senderUserId`, so trusted CLI role/channel/event admin commands now fail before Discord. Scope the missing-sender failure to requester-driven contexts or carry an explicit trusted-owner signal.
  - confidence: 0.91
