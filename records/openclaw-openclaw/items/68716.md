## Summary

Keep open. The PR targets a real Discord authorization gap, but the patch is not mergeable as-is because its runtime fail-closed guard also blocks trusted local owner CLI/manual guild-admin actions that do not carry a Discord requester id.

## What This Changes

The PR adds Discord trusted-requester gating, forwards requester sender identity into guild-admin actions, enforces Discord permission checks in the guild runtime, and adds authorization regression tests.

## Best Possible Solution

Keep the Discord guild-admin authorization hardening, but carry an explicit trusted execution signal or scope the mandatory sender check to tool-driven Discord-origin requests, with tests for both unauthorized requester rejection and trusted owner CLI/manual success.

## Reproduction Assessment

Yes. The PR regression has a source-level reproduction path: `openclaw message` sets `senderIsOwner` but does not pass `requesterSenderId`, while the PR requires `senderUserId` before privileged guild-admin runtime actions.

## Solution Assessment

No. The direction is correct, but the runtime-level missing-sender rejection is broader than the trust boundary; the safer fix distinguishes untrusted Discord requester-driven calls from trusted owner CLI/manual execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve trusted CLI guild-admin actions without sender IDs:** `extensions/discord/src/actions/runtime.guild.ts:234`
  - body: `openclaw message` defaults local CLI calls to `senderIsOwner: true` but does not pass `requesterSenderId`, so owner commands such as `role-add`, `emoji-upload`, `event-create`, and `channel-delete` now reach this runtime without `senderUserId` and fail with `Sender identity required...`. Limit the mandatory sender check to tool/requester-driven contexts or pass an explicit trusted-owner execution signal through this path.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch fixes a real Discord authorization gap but introduces an auth-boundary regression for trusted owner/admin execution.

  - body: The new runtime guard rejects privileged guild actions whenever `senderUserId` is missing, which disables trusted local CLI/manual owner paths that intentionally have `senderIsOwner` but no Discord requester id. The fix should preserve that trust boundary while still rejecting untrusted Discord-origin tool requests.
