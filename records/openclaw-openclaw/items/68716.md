## Summary

Keep this PR open, but do not merge it as-is. The Discord guild-admin authorization fix is addressing a real current-main bug, but the branch still blocks trusted local owner/admin flows that carry `senderIsOwner` without a Discord requester id, and the external PR is missing after-fix real Discord proof.

## What This Changes

The PR adds trusted-requester gating for Discord guild-admin message actions, forwards `requesterSenderId` as `senderUserId`, enforces Discord permission checks in the guild runtime, and adds regression tests.

## Best Possible Solution

Keep the Discord hardening, but preserve trusted owner/manual execution by carrying an explicit trusted execution signal or scoping mandatory sender checks to requester-driven contexts, with focused regression tests and real Discord proof.

## Reproduction Assessment

Yes. Source inspection on current main shows Discord guild-admin mutations can reach bot-credentialed runtime calls without requester authorization, and the PR regression is source-reproducible through local `openclaw message` flows that carry `senderIsOwner` but no `requesterSenderId`.

## Solution Assessment

No. The direction is right, but the runtime-level missing-sender rejection is broader than the established moderation trust boundary; the safer solution distinguishes untrusted Discord requester-driven calls from trusted owner/manual execution.
