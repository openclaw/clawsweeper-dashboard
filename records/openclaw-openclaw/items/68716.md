## Summary

Keep this PR open: it targets a real Discord authorization gap, but the current patch blocks trusted local owner guild-admin actions and still lacks after-fix real Discord proof.

## What This Changes

The PR adds Discord trusted-requester gating, forwards requester sender IDs into guild-admin actions, enforces guild permission checks in the guild runtime, and adds authorization regression tests.

## Best Possible Solution

Keep the Discord hardening, but preserve trusted owner/manual execution by carrying an explicit trusted execution signal or limiting mandatory sender checks to requester-driven contexts, with focused regression tests and real Discord proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main lacks the Discord guild-admin trusted-requester hook and runtime permission check, and the PR regression is reached by local openclaw message flows that carry senderIsOwner but no requesterSenderId.

## Solution Assessment

No. The direction is right, but the runtime-level missing-sender rejection is broader than the established moderation trust boundary; the safer fix distinguishes untrusted Discord requester-driven calls from trusted owner CLI/manual execution.
