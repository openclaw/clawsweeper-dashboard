## Label Justifications

## Summary

Keep open: the reported Discord authorization gap is real, but this PR is not merge-ready because the current guard breaks trusted local/CLI owner guild-admin execution, the branch is conflicting, and contributor real Discord proof is still missing.

## What This Changes

The PR adds Discord trusted-requester gating, forwards requester sender IDs into guild-admin actions, enforces guild permission checks for privileged Discord mutations, and adds focused regression tests.

## Best Possible Solution

Land a rebased Discord hardening that requires Discord-origin guild-admin tool calls to carry trusted requester identity and pass Discord permission checks while preserving trusted owner/manual execution through an explicit trust signal or scoped runtime rule.

## Reproduction Assessment

Yes. Source inspection on current main shows `channel-delete` can reach `deleteChannelDiscord` without a trusted-requester hook or `senderUserId`, and the PR regression is source-reproducible through the CLI owner path that carries `senderIsOwner` but no `requesterSenderId`.

## Solution Assessment

No. The two-layer Discord requester-authorization direction is right, but unconditional runtime rejection when `senderUserId` is absent is too broad for trusted owner/manual paths.
