## Summary

Keep open. Current main still has the linked Discord guild-admin authorization gap, but this PR is not merge-ready because its fail-closed runtime guard breaks trusted local/CLI owner guild-admin actions, the branch is conflicting, and real Discord proof is missing.

## What This Changes

This PR adds Discord trusted-requester gating, threads requester sender IDs into guild-admin actions, adds runtime permission checks for privileged Discord guild mutations, and adds focused tests.

## Best Possible Solution

Land a rebased Discord hardening that requires Discord requester identity and matching guild permissions for Discord-origin guild-admin tool calls while preserving trusted owner/manual execution through an explicit trust signal or scoped runtime check.

## Reproduction Assessment

Yes. Source inspection on current main shows channel-delete can reach deleteChannelDiscord without a trusted-requester hook or senderUserId, and the PR regression is source-reproducible through the CLI owner path that carries senderIsOwner but no requesterSenderId.

## Solution Assessment

No. The two-layer Discord requester-authorization direction is right, but unconditional runtime rejection when senderUserId is absent is too broad for trusted owner/manual paths.
