## Summary

Keep open: current main still has a source-reproducible path where Nextcloud Talk room replies use the shared `message_tool_only` suppression mode and can complete without visible source delivery. The linked implementation PR at https://github.com/openclaw/openclaw/pull/77020 uses closing syntax for this issue and is still open, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to avoid silently completing Nextcloud Talk replies when visible text is generated but no `message` tool send reaches the room.

## Best Possible Solution

Review and land or supersede https://github.com/openclaw/openclaw/pull/77020 for the diagnostic gap, while keeping broader auto-delivery fallback decisions on https://github.com/openclaw/openclaw/issues/77320.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: Nextcloud rooms enter the shared group/channel path, `message_tool_only` sets `suppressDelivery`, and current dispatch can complete after generated visible content without a successful source send. I did not run a live Nextcloud stack in this read-only review.

## Solution Assessment

Yes for the narrow issue: a shared dispatch diagnostic like the linked PR is the maintainable first fix because the invariant is core source-reply accounting, not a Nextcloud-specific workaround. Auto-promoting hidden final text should stay separate because it changes the documented group/channel delivery policy.
