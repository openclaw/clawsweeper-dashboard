## Summary

Keep open: current main still has a source-reproducible path where Nextcloud Talk room turns can suppress automatic delivery under `message_tool_only`, accept generated final/block text, and still record the turn as completed without a successful `message` tool send.

## What This Changes

The issue asks for Nextcloud Talk replies to either be delivered through the `message` tool or fail/log clearly instead of silently completing with no post.

## Best Possible Solution

Add a core dispatch invariant for `message_tool_only` runs: if visible reply text is produced but no successful messaging-tool delivery is recorded, do not report an ordinary completed visible reply; emit an actionable no-delivery diagnostic or narrow fallback while preserving duplicate suppression after real `message` sends.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, as a focused source/unit reproduction: a Nextcloud/group-style dispatch that resolves `message_tool_only`, emits visible final or block text, and records no `messagingToolSent*` evidence currently suppresses delivery and completes normally.

## Solution Assessment

Is this the best way to solve the issue? Yes: the maintainable fix should be in the shared source-reply/message-tool dispatch path, not a Nextcloud-only workaround, because the failing invariant is core delivery accounting after a direct channel dispatch.
