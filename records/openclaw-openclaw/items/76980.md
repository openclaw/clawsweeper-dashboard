## Summary

Keep open: current main still has a source-reproducible path where a Nextcloud Talk group/channel turn can resolve to `message_tool_only`, suppress automatic final/block delivery, and record the message as completed even when no message-tool delivery evidence exists.

## What This Changes

Ensure Nextcloud Talk direct-channel replies never report an ordinary completed reply without either a successful `message` tool send or an explicit no-delivery/no-reply outcome.

## Best Possible Solution

Add a shared dispatch/accounting invariant for `message_tool_only` runs so visible generated output without committed message-tool delivery cannot be reported as an ordinary completed visible reply.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, by source/unit reproduction: run a group/channel-style dispatch with `sourceReplyDeliveryMode: "message_tool_only"`, emit visible final or block text, record no message-tool delivery evidence, and observe suppressed delivery plus completed processing.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow fix belongs in shared reply dispatch or embedded-run delivery accounting, because Nextcloud Talk is using the common SDK dispatch path and a normal direct outbound adapter.
