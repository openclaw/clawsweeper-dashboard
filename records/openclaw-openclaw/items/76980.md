## Summary

Keep open: current main still has a source-reproducible path where a Nextcloud Talk room turn can resolve to `message_tool_only`, generate visible assistant text, suppress block/final delivery, and still record the turn as completed when no successful `message` tool send occurred.

## What This Changes

The issue reports that Nextcloud Talk direct-channel replies can complete without invoking the message tool or posting a response.

## Best Possible Solution

Add a shared dispatch invariant for `message_tool_only` turns: when visible assistant text is produced but no successful messaging-tool delivery is recorded, report an actionable no-delivery diagnostic or non-ordinary outcome while preserving duplicate suppression after real message sends.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, by source/unit path: a Nextcloud room dispatch resolves to `message_tool_only`, emits visible final or block text without `messagingToolSent*` evidence, suppresses delivery, and records `completed`.

## Solution Assessment

Is this the best way to solve the issue? Yes, if scoped to shared delivery accounting/diagnostics rather than a Nextcloud-only workaround or a default policy change.

Add a shared dispatch invariant so this no-delivery case emits an actionable diagnostic or non-ordinary delivery outcome without changing the default `messages.groupChat.visibleReplies` policy and without adding a new config option.
