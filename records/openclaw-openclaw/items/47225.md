## Summary

Keep this PR open for maintainer API review: the feature direction is useful, but the patch is currently incorrect, lacks required real behavior proof, and overlaps with a broader unmerged hook-based narrowing proposal.

## What This Changes

This PR adds an optional `toolsOverride` result to context-engine assembly and wraps embedded-runner stream calls to filter model-visible tool schemas by tool name.

## Best Possible Solution

Choose one documented dynamic tool-surface narrowing contract, make it derivable from host-provided inputs, define undefined versus empty-list behavior, and cover it with focused tests plus redacted real-run proof.

## Reproduction Assessment

Yes for the review findings: source inspection and the PR diff show that `assemble()` exposes only a tool-name set while the PR returns tool objects, and the proposed guard skips `[]`. This is a source-level PR review reproduction, not a live bug repro.

## Solution Assessment

No as written. The feature direction is useful, but the patch should expose a returnable name allow-list or pass real tool objects into `assemble()`, and it must define present-empty behavior before adding the public contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor present empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The new contract says that when `toolsOverride` is present, only those schemas are included, but this guard skips the wrapper for `[]`. A context engine trying to run a no-tool turn would silently send the full tool set; check for `undefined` instead or explicitly document passthrough semantics.
  - confidence: 0.92
- **[P2] Expose a returnable tool override contract:** `src/context-engine/types.ts:16`
  - body: The added result requires `AnyAgentTool[]`, but `assemble()` currently receives only `availableTools?: Set<string>`, and the PR body's `params.tools.filter` example is not part of the interface. Context engines cannot derive real tool objects from host-provided inputs, while the wrapper only uses names; pass tool objects in or make the result a name allow-list.
  - confidence: 0.88
