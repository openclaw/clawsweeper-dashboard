## Summary

Keep open for maintainer review: current main has not implemented context-engine-owned tool schema narrowing, so the feature is not obsolete, but this PR is incorrect as written and overlaps the open hook-based proposal at https://github.com/openclaw/openclaw/pull/68734.

## What This Changes

The PR adds an optional `AssembleResult.toolsOverride`, a stream wrapper that filters model-visible tool schemas by name, and embedded-runner wiring that applies the wrapper after context-engine assembly.

## Best Possible Solution

Choose one documented dynamic narrowing seam, make it derivable from host-provided inputs, define undefined versus empty-list behavior, and cover it with focused tests plus redacted real-run proof.

## Reproduction Assessment

Yes for the PR review findings: the PR diff shows the present-empty guard, and current main shows context-engine assembly receives tool names rather than live tool objects. This is source-level review reproduction, not a live product bug repro.

## Solution Assessment

No, not as written. The feature direction is plausible, but the patch should expose a returnable name allowlist or pass host-provided tool objects into `assemble()`, and it must define present-empty semantics before adding the public contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Honor present empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The new contract says that when `toolsOverride` is present, only those schemas are included, but this guard skips the wrapper for `[]`. A context engine trying to run a no-tool turn would silently send the full tool set; check for `undefined` instead or explicitly document passthrough semantics.
  - confidence: 0.92

Context engines cannot derive real tool objects from host-provided inputs while the wrapper only uses names; pass tool objects in or make the result a name allowlist.
  - confidence: 0.88
