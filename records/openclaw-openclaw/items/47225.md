## Summary

Keep open: current main does not implement this capability, and the PR is not ready because the new shared context-engine result field is only partially enforced, is not implementable from the current assemble inputs as advertised, and overlaps the broader open tool/skill narrowing proposal in #68734.

## What This Changes

The PR adds an optional `AssembleResult.toolsOverride` field and wraps PI embedded stream calls to filter model-visible tool schemas by override tool names.

## Best Possible Solution

Define one additive per-turn tool and skill visibility contract with explicit undefined-versus-empty semantics, usable selection inputs, consistent PI and Codex enforcement, focused tests, docs, and a changelog entry.

## Reproduction Assessment

Yes. Source and diff review reproduce the patch issues: the PR guard ignores explicit empty arrays, current `assemble()` provides only tool names, and Codex harness turns have no `toolsOverride` consumer.

## Solution Assessment

No. The direction is useful, but this patch is not the best complete solution until the shared tool-visibility contract is chosen and applied consistently across runtimes, especially relative to #68734.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor explicit empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The guard only wraps the stream when `toolsOverride.length > 0`, so an engine returning `toolsOverride: []` to hide all schemas still sends the full default tool set. Use field presence as the override signal, or document and warn that empty arrays mean passthrough.
  - confidence: 0.96
- **[P2] Apply tool overrides to Codex harness turns:** `src/context-engine/types.ts:10-16`
  - body: Adding `toolsOverride` to the shared `AssembleResult` lets any context engine return it, but this PR only consumes it in the PI runner. Codex app-server turns also call `assemble()` and continue to send the full `toolBridge.specs`, so the same engine would behave differently by runtime.
  - confidence: 0.9
- **[P2] Expose selectable tools or make the override name-based:** `src/context-engine/types.ts:16`
  - body: The new field requires engines to return `AnyAgentTool[]`, but `assemble()` only receives `availableTools?: Set<string>`. Plugin authors cannot implement the PR body's `params.tools.filter(...)` example without private schema access, so the API should either pass selectable tools or accept names.
  - confidence: 0.9
