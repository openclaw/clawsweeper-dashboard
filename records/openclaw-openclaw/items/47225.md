## Summary

Keep open for maintainer review: the feature is useful, but current main does not implement this context-engine result seam, the related hook proposal is still open, and this PR has source-confirmed contract issues plus missing real behavior proof.

## What This Changes

The PR adds an optional `AssembleResult.toolsOverride` field and wraps embedded-runner stream calls to filter model-visible tool schemas by returned tool names.

## Best Possible Solution

Choose one documented dynamic narrowing seam, make the allowed set derivable from host-provided data, define undefined versus empty-list semantics, coordinate with https://github.com/openclaw/openclaw/pull/68734, and prove it with focused tests plus redacted real-run evidence.

## Reproduction Assessment

Yes for the PR review findings: source inspection of the PR diff shows the present-empty guard and current main exposes only `availableTools?: Set<string>`, not live tool objects. This is a source-level PR review reproduction, not a live product bug reproduction.

## Solution Assessment

No, not as written. The feature direction is plausible, but the patch should expose a derivable name allow-list or pass host tool objects into `assemble()`, and maintainers should reconcile it with the hook-based proposal before adding a public contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor present empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2124`
  - body: The new contract says that when `toolsOverride` is present, only those schemas are included, but this guard skips the wrapper for `[]`. A context engine trying to run a no-tool turn would silently send the full tool set; check for `undefined` instead or explicitly document empty-array passthrough semantics.
  - confidence: 0.92
- **[P2] Expose a derivable tool override contract:** `src/context-engine/types.ts:16`
  - body: The added result requires `AnyAgentTool[]`, but `assemble()` currently receives only `availableTools?: Set<string>`, and the PR body's `params.tools.filter` example is not part of the interface. Context engines cannot derive live tool objects from host-provided inputs while the wrapper only uses names; pass tool objects in or make the result a name allow-list.
  - confidence: 0.88
