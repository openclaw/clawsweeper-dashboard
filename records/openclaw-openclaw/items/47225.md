## Summary

Keep open. The feature is still useful, but the branch is not merge-ready because it adds a shared context-engine result field while ignoring explicit empty overrides, leaving Codex harness turns on the full tool set, and giving engines only tool names rather than live schemas to select from.

## What This Changes

This PR adds optional `AssembleResult.toolsOverride`, a PI stream wrapper that filters model-visible tool schemas by override names, and PI runner wiring that applies non-empty overrides.

## Best Possible Solution

Define one additive per-turn tool visibility contract, document undefined-versus-empty semantics, give engines a practical live selection input or use a name-based allow list, apply it consistently in PI and Codex harnesses, and add focused contract tests plus plugin docs.

## Reproduction Assessment

Yes. Source and diff review are enough: the PR guard ignores `toolsOverride: []`, current Codex harness requests still use full `toolBridge.specs`, and current `assemble()` params expose only `availableTools?: Set<string>`.

## Solution Assessment

No. The direction is reasonable, but this patch is not the best complete solution because it changes a shared plugin/context-engine API without consistent runtime enforcement or a usable schema-selection input.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor explicit empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The guard applies the wrapper only when the override has at least one tool, so an engine returning `toolsOverride: []` to hide all schemas still sends the full default tool set. Use field presence for the override signal, or document and warn that empty arrays mean passthrough.
  - confidence: 0.96
- **[P2] Apply tool overrides to Codex harness requests:** `src/context-engine/types.ts:10-16`
  - body: `toolsOverride` is added to the shared context-engine result type, but this PR only consumes it in the PI runner. Codex app-server turns also call `assemble()` and continue to start or resume with full `toolBridge.specs`, so the same engine would behave differently by runtime.
  - confidence: 0.9
- **[P2] Expose selectable schemas or make the override name-based:** `src/context-engine/types.ts:16`
  - body: The new result requires engines to return `AnyAgentTool[]`, but `assemble()` only receives `availableTools?: Set<string>`. The documented `params.tools.filter(...)` usage cannot compile against the current contract, so plugins would need private tool-schema access or hard-coded objects.
  - confidence: 0.9
