## Summary

Keep open: the feature direction is plausible and still not implemented on current main, but this PR is not merge-ready because the proposed public contract cannot be implemented from the current assemble inputs, explicit empty overrides are ignored, and the external contributor has not provided real behavior proof.

## What This Changes

The PR adds a `toolsOverride` field to context-engine assembly results and wraps the embedded runner stream function to filter model-visible tool schemas by name.

## Best Possible Solution

Settle the maintained per-turn narrowing API, make the override returnable from host-provided inputs, define empty-list behavior, add focused regression coverage, and coordinate it with the open hook-based and broader tool-overhead trackers.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR ignores explicit empty overrides and current main only gives context engines tool names, not `AnyAgentTool` objects. I did not run tests because this was a read-only review.

## Solution Assessment

No as written. A per-turn narrowing seam is plausible, but this patch should either return a name allow-list or expose live tool objects to `assemble()`, and it must define present-empty behavior before becoming a public plugin contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2124`
  - body: The new guard only wraps the stream function when the override has at least one entry, so a context engine returning `toolsOverride: []` still sends the full default tool schema set. Since the new contract says a present override controls which schemas are included, check for `undefined` or explicitly document that an empty array is passthrough.
  - confidence: 0.92
- **[P2] Expose a returnable override contract:** `src/context-engine/types.ts:10-16`
  - body: `assemble()` on current main receives `availableTools?: Set<string>`, but this result field requires `AnyAgentTool[]`; a context engine cannot build that value from host-provided inputs without retaining or fabricating tool objects. Use a name allow-list result or pass the actual tool objects into `assemble()` so plugin authors can implement the API type-safely.
  - confidence: 0.86
