## Summary

Keep this PR open for maintainer review: the feature direction is useful, but the current patch has API/semantics gaps and lacks external real behavior proof before merge.

## What This Changes

The PR adds an optional context-engine `toolsOverride` result and wraps the embedded runner stream function to filter tool schemas sent to the model.

## Best Possible Solution

Keep the dynamic narrowing seam if maintainers want it, but make the contract name-based or pass full tool objects into `assemble()`, honor explicit empty overrides, add focused tests, and document how this differs from hook-based `toolsAllow`.

## Reproduction Assessment

Yes for the review findings: source inspection shows an explicit empty override is ignored, and current main only passes tool names into `assemble()`. I did not run tests because this was a read-only review.

## Solution Assessment

No as written. The approach is plausible, but the public API should either use tool names or provide actual tool objects, and present-but-empty overrides must have defined behavior before this is a maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The new guard only wraps the stream function when the override has at least one entry, so a context engine returning `toolsOverride: []` still sends the full tool schema set. Since the new contract says a present override controls which schemas are included, this silently defeats pure-chat or no-tool turns; check for `undefined` instead of non-empty length, or explicitly document/pass through the empty-array behavior.
  - confidence: 0.92
- **[P2] Expose an override contract engines can return:** `src/context-engine/types.ts:10-16`
  - body: `assemble()` currently receives `availableTools?: Set<string>` from the runner, but this result field requires `AnyAgentTool[]`. A context engine cannot build that value from the host-provided inputs without retaining or fabricating tool objects, and the wrapper only reads names anyway. Use a name allow-list result or pass the actual tool objects into `assemble()` so plugin authors can implement the API type-safely.
  - confidence: 0.86
