## Summary

Keep this PR open for human API review: the feature direction is useful, but the current patch adds a public context-engine contract that is not derivable from current `assemble()` inputs, has surprising empty-list behavior, and still lacks contributor real behavior proof.

## What This Changes

The PR adds an optional `toolsOverride` field to `AssembleResult` and wraps embedded-runner stream calls to filter model-visible tool schemas by tool name.

## Best Possible Solution

Choose one documented dynamic tool-surface narrowing seam, make it derivable from host-provided inputs, define undefined versus empty-list behavior, coordinate it with the hook proposal, and cover it with focused tests plus redacted real-run proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR diff shows the present-empty guard, and current main shows context-engine assembly receives a tool-name set rather than live tool objects. This is a source-level PR review reproduction, not a live product bug repro.

## Solution Assessment

No as written. The feature direction is plausible, but the patch should expose a returnable name allow-list or pass host-provided tool objects into `assemble()`, and it must define present-empty semantics before adding the public contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor present empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The new contract says that when `toolsOverride` is present, only those schemas are included, but this guard skips the wrapper for `[]`. A context engine trying to run a no-tool turn would silently send the full tool set; check for `undefined` instead or explicitly document passthrough semantics.
  - confidence: 0.92
- **[P2] Expose a derivable tool override contract:** `src/context-engine/types.ts:16`
  - body: The added result requires `AnyAgentTool[]`, but `assemble()` currently receives only `availableTools?: Set<string>`, and the PR body's `params.tools.filter` example is not part of the interface. Context engines cannot derive real tool objects from host-provided inputs while the wrapper only uses names; pass tool objects in or make the result a name allow-list.
  - confidence: 0.88
