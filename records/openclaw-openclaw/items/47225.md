## Summary

Keep open: current main still lacks this context-engine schema-filter seam, but the PR is not mergeable as-is because the proposed public contract is incomplete, explicit empty overrides are ignored, and the external PR still lacks real behavior proof. The related hook-based narrowing PR is active and relevant, but it is not merged and does not conclusively supersede this separate context-engine API choice.

## What This Changes

The PR adds an optional `toolsOverride` result to context-engine assembly and wraps the embedded runner stream function to send only matching tool schemas to the model.

## Best Possible Solution

Choose the maintained dynamic narrowing seam, make it name-based or pass real tool objects into `assemble()`, honor present empty allow lists, add focused regression coverage, and document how it relates to hook-based `toolsAllow`.

## Reproduction Assessment

Yes for the review findings: source inspection shows an explicit empty override is ignored, and current main only passes tool names into `assemble()`. I did not run tests because this was a read-only review.

## Solution Assessment

No as written. The feature direction is plausible, but the contract should be returnable from host-provided inputs and must define present-empty behavior before this becomes a maintainable public API.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The new guard only wraps the stream function when the override has at least one entry, so a context engine returning `toolsOverride: []` still sends the full tool schema set. Since the new contract says a present override controls which schemas are included, this silently defeats pure-chat or no-tool turns; check for `undefined` instead of non-empty length, or explicitly document the empty-array behavior.
  - confidence: 0.92
- **[P2] Expose a returnable override contract:** `src/context-engine/types.ts:10-16`
  - body: `assemble()` currently receives `availableTools?: Set<string>` from the runner, but this result field requires `AnyAgentTool[]`. A context engine cannot build that value from the host-provided inputs without retaining or fabricating tool objects, and the wrapper only reads names anyway. Use a name allow-list result or pass the actual tool objects into `assemble()` so plugin authors can implement the API type-safely.
  - confidence: 0.86
