## Summary

Keep open. The feature is useful and still absent from current main, but the proposed shared context-engine API is incomplete: the patch ignores explicit empty overrides, only applies the override in the PI runner, and asks engines to return tool schemas even though assemble inputs expose only tool names.

## What This Changes

The PR adds an optional `AssembleResult.toolsOverride`, a PI stream wrapper that filters `context.tools` by override names, and PI runner wiring that applies non-empty override arrays.

## Best Possible Solution

Define one additive per-turn tool visibility contract for context engines, document empty-subset behavior, give engines a practical live selection input or use name-based overrides, apply it consistently in PI and Codex harnesses, and add focused contract tests plus plugin docs while keeping executor authorization unchanged.

## Reproduction Assessment

Yes. Source review is enough: current main lacks `toolsOverride`, the PR diff would ignore `toolsOverride: []`, and the Codex harness still passes full `toolBridge.specs` to `dynamicTools` after context-engine assembly.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best complete solution because it changes a shared plugin API without a complete selection input or consistent enforcement across active harness consumers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor explicit empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The guard only wraps the stream when `toolsOverride.length > 0`, so a context engine cannot intentionally hide all tool schemas for a pure-chat turn. Check field presence instead, or document and warn that an empty array is a no-op.
  - confidence: 0.96
- **[P2] Apply tool overrides in Codex harness too:** `src/context-engine/types.ts:16`
  - body: `toolsOverride` is added to the shared context-engine result type, but only the PI runner consumes it. The Codex app-server harness also calls `assemble()` and still starts threads with full `toolBridge.specs`, so the same engine would behave differently there.
  - confidence: 0.9
- **[P2] Expose live tool schemas or use name-based overrides:** `src/context-engine/types.ts:16`
  - body: The result type asks plugins to return `AnyAgentTool[]`, but `assemble()` receives only `availableTools?: Set<string>`. Engines cannot build the advertised filtered schema list from the live registry without a separate private tool source.
  - confidence: 0.9
