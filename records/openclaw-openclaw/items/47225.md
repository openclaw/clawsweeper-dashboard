## Summary

Keep open. The PR targets a useful token-reduction problem, but the proposed shared context-engine API is incomplete: empty overrides are silently ignored, only the PI runner consumes the new field, Codex harness turns would still send the full tool catalog, and the API asks engines to return tool schemas without giving them schemas to choose from.

## What This Changes

This PR adds optional `AssembleResult.toolsOverride`, a PI stream wrapper that filters model-visible tool schemas by override names, and PI runner wiring that applies non-empty overrides.

## Best Possible Solution

Define one additive per-turn tool and skill visibility contract, document undefined-versus-empty semantics, give plugins a practical selection input, apply it consistently in PI and Codex harnesses, and add focused contract tests plus docs and changelog coverage.

## Reproduction Assessment

Yes. Source and diff review are enough: the PR guard ignores `toolsOverride: []`, current Codex harness requests still use full `toolBridge.specs`, and current `assemble()` params expose only `availableTools?: Set<string>`.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best complete solution because it changes a shared plugin/context-engine result type without consistent runtime enforcement or a usable schema-selection input.
