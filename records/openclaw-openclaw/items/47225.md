## Summary

Keep open. The feature direction is useful, but the PR is not merge-ready: it adds a shared context-engine result field that only PI consumes, ignores explicit empty override arrays, and asks plugins to return live tool schemas even though `assemble()` currently exposes only tool names. Current `main` does not already implement this context-engine override, and the related hook-side narrowing work in #68734 is overlapping but not a shipped replacement.

## What This Changes

The PR adds an optional `AssembleResult.toolsOverride`, a PI stream wrapper that filters model-visible tool schemas by override names, and PI runner wiring that applies non-empty overrides.

## Best Possible Solution

Define one additive per-turn tool visibility contract for context engines, document undefined-versus-empty behavior, provide a practical live selection input or use a name-based allow list, apply it consistently in PI and Codex harnesses, and keep executor authorization unchanged while adding focused contract tests and plugin docs.

## Reproduction Assessment

Yes. Source review is enough: current main lacks `toolsOverride`, the PR diff would ignore `toolsOverride: []`, and the Codex harness still passes full `toolBridge.specs` after context-engine assembly.

## Solution Assessment

No. The direction is reasonable, but this patch is not the best complete solution because it changes a shared plugin API without complete selection input, clear empty-subset semantics, or consistent enforcement across active harness consumers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor explicit empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The guard only applies the override when `toolsOverride.length > 0`, so an engine cannot intentionally hide all tool schemas for a pure-chat turn. Check field presence instead, or document and warn that an empty array is passthrough.
  - confidence: 0.96
- **[P2] Apply tool overrides to Codex harness requests:** `src/context-engine/types.ts:10-16`
  - body: `toolsOverride` is added to the shared context-engine result type, but only the PI runner consumes it. Codex app-server turns also call `assemble()` and would still start or resume threads with full `toolBridge.specs`, making the same engine behave differently by harness.
  - confidence: 0.9
- **[P2] Expose schemas to engines or make overrides name based:** `src/context-engine/types.ts:16`
  - body: The new result asks plugins to return `AnyAgentTool[]`, but `assemble()` only receives `availableTools?: Set<string>`. The advertised `params.tools.filter(...)` pattern is not available, so either pass live schemas or make the override a name-based subset.
  - confidence: 0.9
