## Summary

Keep open: the feature direction is useful, but the current PR still has API-contract bugs, no after-fix real behavior proof, and a related maintainer/API decision around the hook-based narrowing work.

## What This Changes

The branch adds a context-engine `toolsOverride` result and wraps the embedded runner stream function to filter model-visible tool schemas by tool name.

## Best Possible Solution

Keep dynamic narrowing if maintainers want a context-engine seam, but make the contract name-based or pass actual tool objects into `assemble()`, honor explicit empty overrides, document how it relates to hook-based `toolsAllow`, and add focused tests plus real proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR diff shows an explicit empty override is ignored, and current main only passes tool names into `assemble()` while the PR result requires tool objects. I did not run tests because this was a read-only review.

## Solution Assessment

No as written. The approach is plausible, but the public contract should either use tool names or provide actual tool objects, and present-empty overrides need defined behavior before this is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The new guard only wraps the stream function when the override has at least one entry, so a context engine returning `toolsOverride: []` still sends the full tool schema set. Since the new contract says a present override controls which schemas are included, this silently defeats pure-chat or no-tool turns; check for `undefined` instead of non-empty length, or explicitly document/pass through the empty-array behavior.
  - confidence: 0.92
- **[P2] Expose an override contract engines can return:** `src/context-engine/types.ts:10-16`
  - body: `assemble()` currently receives `availableTools?: Set<string>` from the runner, but this result field requires `AnyAgentTool[]`. A context engine cannot build that value from the host-provided inputs without retaining or fabricating tool objects, and the wrapper only reads names anyway; use a name allow-list result or pass actual tool objects into `assemble()`.
  - confidence: 0.86
