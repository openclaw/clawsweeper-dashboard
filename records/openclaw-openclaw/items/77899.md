## Summary

Keep open: the reported bug is real, but this PR is not merge-ready because it converts total supplement failure into a successful empty result and only supplies test/inline proof. The overlapping same-issue PR is still open with its own blockers, so it is not a safe superseding close target.

## What This Changes

The PR changes memory-core supplement search from fail-fast `Promise.all` to `Promise.allSettled` and drops rejected supplement outcomes while preserving fulfilled result sorting and slicing.

## Best Possible Solution

Preserve fulfilled sibling results while keeping unavailable/error behavior when every requested supplement fails, with committed tool-path regression tests and redacted real `memory_search corpus=all` proof.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.6` await `Promise.all` over supplement searches; registering two supplements and making one reject reproduces rejection of the full fan-out.

## Solution Assessment

No, not as-is. `Promise.allSettled` is the right direction for partial successes, but rejected outcomes need aggregate semantics rather than unconditional `[]` so total requested-supplement failure remains unavailable.
