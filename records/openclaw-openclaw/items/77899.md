## Summary

The PR targets a real current-main memory retrieval bug, but this branch is not merge-ready because it silently turns all rejected supplement searches into an empty successful result and still lacks committed production-path regression coverage. The canonical issue remains open, and https://github.com/openclaw/openclaw/pull/78035 is a stronger open candidate for the same root cause because it adds logging and committed tests, though it is not merged yet.

## What This Changes

The branch changes `searchMemoryCorpusSupplements` in memory-core from fail-fast `Promise.all` aggregation to `Promise.allSettled` and drops rejected supplement results while preserving fulfilled results.

## Best Possible Solution

Land one canonical memory-core fix that preserves sibling supplement results, keeps total supplement failure distinguishable from no hits, and commits regression coverage through the production memory tool path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register two memory corpus supplements, make one `search()` reject, and current main's `Promise.all` rejects the supplement fan-out for `memory_search` with `corpus="wiki"` or `corpus="all"`; I did not run tests because this review is read-only.

## Solution Assessment

No. `searchMemoryCorpusSupplements` is the right owner boundary for isolating partial supplement failures, but this implementation also turns total requested-supplement failure into a normal empty result and does not add committed regression coverage.
