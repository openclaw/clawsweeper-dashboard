## Summary

This should stay open: current main still has the misleading Pi-first runtime metadata, but the proposed patch has a concrete P2 correctness issue and the supplied real-behavior proof does not directly show the changed picker/provider-data ordering.

## What This Changes

The PR changes `buildModelsProviderData` and focused tests so model picker runtime choices are derived from effective harness policy, showing Codex before Pi for default official OpenAI routes.

## Best Possible Solution

Derive non-default provider runtime choices from provider-level policy and implicit OpenAI rules, keep exact-model overrides scoped to the active/default model case, add a regression for that ordering, and provide direct after-fix provider-data or picker proof.

## Reproduction Assessment

Yes. Current main source shows `buildModelsProviderData` returns Pi-first runtime choices for `openai`, while `resolveAgentHarnessPolicy` resolves official `openai/gpt-5.5` execution to Codex.

## Solution Assessment

No. The direction is appropriate, but the patch should not let an arbitrary first visible model or exact-model override decide provider-level runtime ordering for providers that are not the active default.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Derive non-default provider runtime without picking a model:** `src/auto-reply/reply/commands-models.ts:126-129`

If that model has an exact `agentRuntime` override, provider-level picker metadata can put a model-specific runtime first even though the provider's general policy would select something else; resolve provider-wide policy instead of using an arbitrary `Set` entry.
