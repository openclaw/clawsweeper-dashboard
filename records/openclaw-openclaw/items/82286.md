## Summary

Keep open: the reported main-branch bug is source-reproducible and the PR moves in the right direction, but the branch still has a P2 correctness issue for non-default providers and the supplied real-behavior proof does not directly show the changed picker/runtime metadata.

## What This Changes

The PR changes `buildModelsProviderData` and its tests so model picker runtime choices use effective harness policy, showing Codex before Pi for default official OpenAI routes.

## Best Possible Solution

Derive non-default provider runtime choices from provider-level policy and implicit OpenAI rules rather than an arbitrary visible model, add a regression for model-specific override ordering, and provide direct after-fix picker/provider-data proof.

## Reproduction Assessment

Yes. Source inspection shows current `buildModelsProviderData` puts Pi first while the execution policy resolves official `openai/gpt-5.5` to Codex; the focused reproduction is building provider data for that config and inspecting `runtimeChoicesByProvider.get("openai")`.

## Solution Assessment

No. The proposed direction is appropriate, but the implementation should not let a first visible model or exact-model override decide provider-level runtime ordering for providers that are not the active default.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Derive non-default provider runtime without picking a model:** `src/auto-reply/reply/commands-models.ts:131`
