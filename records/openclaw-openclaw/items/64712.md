## Summary

Keep open. Current main has not implemented the PR’s intended change, and the PR branch still has blocking correctness issues: the real Feishu helper can return a synthetic `default` ID, the new test fixture does not model that path and lacks imports, and the guard can hide explicit-but-unresolved channel configs. The linked #64707 report remains open, so this PR should be repaired or replaced rather than closed.

## What This Changes

The PR adds a `hasRealAccounts` guard around channel-summary construction and status-line emission, plus a regression test for a channel with no configured accounts.

## Best Possible Solution

Repair the status-summary logic so it suppresses only synthetic fallback accounts with no authored or persisted channel state, while preserving diagnostics for explicit default accounts and unresolved configured channels. Cover the real Feishu helper path and an explicit no-account visibility case in tests before landing.

## Reproduction Assessment

Yes. The source-level reproduction is clear: Feishu’s shared helper can produce `['default']` without configured accounts, and the PR’s guard treats that as real; the added fixture with `accountIds: []` does not reproduce the Feishu path and also references missing imports.

## Solution Assessment

No. The PR’s `accountIds.length > 0` heuristic is not the narrowest maintainable fix; the safer direction is to distinguish authored or persisted account/channel state from synthetic fallback IDs and keep unresolved explicit configs visible.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Distinguish synthetic Feishu defaults from real accounts:** `src/infra/channel-summary.ts:194`
  - body: `hasRealAccounts` is true whenever `accountIds` is non-empty, but Feishu’s `listFeishuAccountIds` comes from `createAccountListHelpers`, whose `listAccountIds` returns `default` when no configured account IDs exist. In that real Feishu path this PR still calls the status adapter and still emits a status line; the new test stubs `accountIds: []` and misses the linked behavior.
  - confidence: 0.91

  - body: The new test calls `setActivePluginRegistry` and `createTestRegistry`, but this file does not import either name and the PR diff adds no import hunk. The test file will fail before it can verify the regression.
  - confidence: 0.95
- **[P2] Keep unresolved configured channels visible:** `src/infra/channel-summary.ts:244-246`
  - body: The new status-line guard drops channels when both `accountIds` and `configuredEntries` are empty. That hides explicit-but-unresolved configs, such as Nostr config where `listAccountIds` returns `[]` unless `privateKey` is a literal string, turning a visible `not configured` diagnostic into no status output.
  - confidence: 0.84
