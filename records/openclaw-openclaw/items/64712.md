## Summary

Keep this PR open but not merge-ready: the patch still treats Feishu’s synthetic `default` account id as a real account, adds a test that is missing imports, drops some unresolved configured-channel diagnostics, and has no after-fix real behavior proof from the contributor.

## What This Changes

The PR adds a `hasRealAccounts` guard around channel-summary construction and status-line emission, plus a regression test for an empty Feishu/default account status case.

## Best Possible Solution

Repair the status-summary logic so it suppresses only synthetic fallback accounts with no authored or persisted channel state, preserves diagnostics for explicit unresolved configs, and adds Feishu-focused regression coverage for the real helper path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Feishu uses the shared account helper that can return `default` for empty account maps, and the PR’s `accountIds.length > 0` guard still treats that synthetic id as real; I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. The PR’s heuristic is too broad because it uses non-empty account ids as proof of real account state; the narrower maintainable fix is to distinguish authored or persisted account/channel state from synthetic fallback ids while keeping unresolved configured channels visible.
