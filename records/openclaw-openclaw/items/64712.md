## Summary

Keep this PR open, but it is not ready to merge: the patch targets a real channel-summary bug, yet the guard still treats Feishu's synthetic `default` id as a real account, can hide unresolved configured-channel diagnostics, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a `hasRealAccounts` guard around channel-summary status building and line emission, plus a regression test for the no-configured-accounts case.

## Best Possible Solution

Update or replace the PR with a narrow fix that distinguishes synthetic fallback ids from authored account state, preserves unresolved-config status output, and adds Feishu-focused regression plus real Feishu/status proof.

## Reproduction Assessment

Yes, source-reproducible: current main builds a fallback account id and pushes a status line, while Feishu's shared account helper can synthesize `default` for an empty account map. I did not run a live gateway restart because this review is read-only.

## Solution Assessment

No. The PR's returned-id heuristic is not the best way to solve this because it cannot distinguish authored accounts from synthetic defaults and can suppress diagnostics for explicit but unresolved configs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Distinguish synthetic defaults from real accounts:** `src/infra/channel-summary.ts:193-195`
  - body: `hasRealAccounts` is true for any returned account id, but Feishu's shared account helper can return `default` when no account map is configured. In the reported path this still lets the status adapter run and can emit the phantom default line the PR is trying to suppress.
  - confidence: 0.92
- **[P2] Exercise the real Feishu fallback path:** `src/infra/channel-summary.test.ts:373-377`
  - body: The new regression fixture forces `accountIds: []`, but real Feishu account listing can return the synthetic `default` id from the shared helper. This test can pass while the real Feishu no-account path still sets `hasRealAccounts` to true, so it should use real Feishu behavior or a fixture that returns the synthetic default.
  - confidence: 0.9
- **[P2] Keep unresolved configured channels visible:** `src/infra/channel-summary.ts:244-245`
  - body: Suppressing the whole status line when both account ids and configured entries are empty can hide channels that have explicit config but cannot resolve account ids yet. For example, Nostr SecretRef-style private keys can produce no listed ids until resolved; current main still shows a diagnostic line, but this guard drops it.
  - confidence: 0.84
