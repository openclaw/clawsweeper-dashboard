## Summary

Keep this PR open for author/maintainer follow-up: it targets a real source-reproducible Feishu/default-account status bug, but the current patch still treats Feishu's synthetic `default` id as a real account and can hide unresolved channel diagnostics; it also lacks after-fix real behavior proof.

## What This Changes

The PR adds a `hasRealAccounts` guard in `src/infra/channel-summary.ts` to skip channel status summary/line emission when no accounts are configured and adds an empty-account regression test.

## Best Possible Solution

Update or replace the PR with a narrow fix that distinguishes authored account state from synthetic fallback ids, preserves diagnostics for unresolved configured channels, and adds Feishu-focused restart/status regression coverage plus real behavior proof.

## Reproduction Assessment

Yes, source-reproducible: current main and the latest release build a default fallback status path, and Feishu's shared helper can synthesize `default` from an empty account map. I did not establish a live gateway-restart reproduction in this read-only review.

## Solution Assessment

No. The PR's `accountIds.length` heuristic is not the best fix because it treats Feishu's synthetic fallback id as real and can suppress diagnostics for configured channels whose account ids are unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Distinguish synthetic defaults from real accounts:** `src/infra/channel-summary.ts:193`
  - body: `hasRealAccounts` is true for any returned account id, but Feishu's shared account helper can return `default` when no account map is configured. In the reported path this still lets the status adapter run and can emit the phantom default line the PR is trying to suppress; the fix needs an authored-account signal rather than the returned-id count.
  - confidence: 0.93
- **[P2] Exercise the real Feishu fallback path:** `src/infra/channel-summary.test.ts:373-377`
  - body: The new regression fixture forces `accountIds: []`, but real Feishu listing can return the synthetic `default` id from the shared helper. This test can pass while the actual Feishu no-account path still sets `hasRealAccounts` to true, so it should use real Feishu behavior or a fixture that returns the synthetic default with no authored account.
  - confidence: 0.9
- **[P2] Keep unresolved channel configs visible:** `src/infra/channel-summary.ts:244-245`
  - body: Suppressing the whole status line when both `accountIds` and `configuredEntries` are empty can make configured-but-unresolved channel surfaces disappear from `channels status`. For example, SecretRef or partial-config paths can select a channel for read-only status while account ids are not listable; the fix should suppress only synthetic unconfigured defaults, not all diagnostics.
  - confidence: 0.84
