## Summary

Keep this PR open, but it is not ready to merge: the proposed `accountIds.length` guard still treats Feishu's synthetic `default` id as real, can hide unresolved configured-channel diagnostics, and the external PR still lacks after-fix real Feishu/status proof.

## What This Changes

The PR gates channel-summary status building and line emission behind `hasRealAccounts` and adds a no-accounts regression test for phantom default-account output.

## Best Possible Solution

Use an authored-account/config signal to suppress only synthetic unconfigured defaults, preserve diagnostics for configured but unresolved channels, and add Feishu-focused regression coverage plus redacted real status or restart proof.

## Reproduction Assessment

Yes for the source-level path: current main can synthesize `default` through the shared account helper and feed that snapshot into Feishu status output. No live Feishu restart reproduction was established, so the persistence part remains unverified.

## Solution Assessment

No. The PR's `accountIds.length` heuristic is not the best fix because returned ids can be synthetic, and broad status-line suppression can hide unresolved configured-channel diagnostics.
