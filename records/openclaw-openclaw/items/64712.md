## Summary

This PR should stay open: it targets a real Feishu/default-account status problem, but the current patch still conflates fallback-expanded account IDs with authored accounts and can hide useful diagnostics for configured channels whose account IDs are unresolved.

## What This Changes

The PR gates channel-summary status adapter calls and status-line output behind a `hasRealAccounts` check and adds a regression test for a no-Feishu-accounts case.

## Best Possible Solution

Use an explicit authored-account or persisted-config signal to suppress only synthetic unconfigured defaults while preserving diagnostics for configured-but-unresolved channels, then prove the Feishu status path with redacted real output.

## Reproduction Assessment

Yes for the source-level path: current main can synthesize a fallback `default` account ID before status rendering, matching the phantom-entry mechanism. I did not establish a live Feishu gateway restart reproduction in this read-only review.

## Solution Assessment

No. The current PR direction is plausible, but the implementation needs a real authored/configured account signal rather than using fallback-expanded `accountIds.length`.
