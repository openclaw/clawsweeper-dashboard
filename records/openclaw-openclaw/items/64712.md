## Summary

Keep open. The linked Feishu report remains unresolved, and this PR branch is still not ready because the guard treats synthetic fallback account IDs as real accounts, adds a test that would not compile as shown, and can hide diagnostics for explicitly configured channels whose account ID cannot be resolved yet.

## What This Changes

The PR adds a `hasRealAccounts` guard in `src/infra/channel-summary.ts` and a regression test intended to suppress no-account channel summary status lines.

## Best Possible Solution

Use or add a typed account-resolution signal that distinguishes authored/persisted account configuration from synthetic fallback IDs, keep status diagnostics visible for explicit-but-unresolved configs, and cover the real Feishu helper path plus an unresolved-config visibility case in tests.

## Reproduction Assessment

Yes. The source-level path is clear: current main falls back to a default account and pushes a status line, while Feishu's real account helper can return a synthetic `default` ID, so the PR's empty-array fixture does not reproduce the linked Feishu behavior accurately.

## Solution Assessment

No. The PR's `accountIds.length > 0` heuristic is not the best fix because some non-empty IDs are synthetic fallbacks and some explicit configs legitimately resolve to no account IDs; a typed real-account/configured-state distinction is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Distinguish synthetic Feishu defaults from real accounts:** `src/infra/channel-summary.ts:193-194`
  - body: `hasRealAccounts` is true whenever `accountIds` is non-empty, but Feishu gets its IDs from `createAccountListHelpers`, which returns `default` when no configured accounts exist. The linked Feishu phantom-default path can therefore still call the status adapter and push a status line, so the regression test's `accountIds: []` fixture misses the real behavior.
  - confidence: 0.91
- **[P2] Import the registry helpers used by the test:** `src/infra/channel-summary.test.ts:369-370`
  - body: The new test calls `setActivePluginRegistry` and `createTestRegistry`, but the file does not import either name and the PR diff adds no import hunk. The test will fail to typecheck before it can verify the regression.
  - confidence: 0.95
- **[P2] Keep unresolved configured channels visible:** `src/infra/channel-summary.ts:245-246`
  - body: Gating the status line on `hasRealAccounts` drops channels whose explicit config cannot resolve an account ID yet. For example, Nostr treats an unresolved SecretRef `privateKey` as no account IDs, so this change would hide the channel diagnostic instead of surfacing the configuration problem.
  - confidence: 0.86
