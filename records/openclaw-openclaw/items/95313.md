## Summary

Keep this PR open: it targets a real Slack name-allowlist regression, but the current patch still has a blocking authorization-source bug and lacks after-fix real Slack proof.

## What This Changes

The PR changes Slack read authorization to consider current or lazily resolved Slack channel names when read-like actions target a Slack channel ID.

## Best Possible Solution

Use a trusted Slack conversation name only when the producer actually has one, otherwise leave the name unset so the Slack-owned `conversations.info` fallback resolves it, then prove allowed and denied reads in a real Slack workspace.

## Reproduction Assessment

Yes at source level. Current main checks Slack read targets by channel ID only, and the PR head’s context shortcut still comes from a core field that carries `originProvider`, not a Slack channel name.

## Solution Assessment

No. The lazy `conversations.info` fallback is a good Slack-owned repair path, but the current shortcut should be removed or fed by a real Slack conversation name before it is used in the authorization boundary.
