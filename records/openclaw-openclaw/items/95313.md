## Summary

Keep this PR open but not merge-ready: the Slack-owned lazy conversations.info path is the right direction, but the PR still feeds currentChannelName from the core Channel field, which is the provider id on normal Slack turns, and the contributor has not added after-fix real Slack proof.

## What This Changes

The PR changes Slack read-target authorization to use an optional channel name, lazily resolve Slack conversation names with conversations.info, and adds Slack action/channel-type tests plus a shared threading-context field.

## Best Possible Solution

Populate currentChannelName only from a trusted Slack conversation name, or leave it unset so conversations.info resolves the name, then add core-built context coverage and redacted real Slack proof.

## Reproduction Assessment

Yes, at source level. Current main only checks Slack read targets by channel ID, while PR head wires currentChannelName from a field that core fills with the provider id, so the intended name-allowlist path and new tests do not match the real core-built context.

## Solution Assessment

No. The lazy conversations.info fallback is the right Slack-owned repair path, but the context shortcut is sourced from the wrong field and should be fixed or removed before merge.
