## Summary

Keep open: the branch usefully fixes the no-context Slack lookup path and includes redacted live proof, but the normal core-built Slack context still feeds the provider id as the channel name, so the authorization shortcut can leave the reported workflow broken and can mis-match name allowlists.

## What This Changes

The PR updates Slack read authorization to resolve Slack channel names for channel-ID read targets under name-based allowlists, adds a current channel-name field to threading context, and extends Slack action/channel-type tests.

## Best Possible Solution

Use one trusted authorization path: lazily resolve the Slack conversation name for ID-only read targets unless a verified conversation label is supplied, and cover the real core-built Slack threading context in regression tests.

## Reproduction Assessment

Yes from source inspection, without executing tests: current main checks Slack read targets by channel ID, while name allowlist entries need a channel name. The PR’s remaining failure is also source-reproducible because core passes `Channel: originProvider` into the Slack threading context.

## Solution Assessment

No. The lazy `conversations.info` lookup is the right repair layer, but the context shortcut must either be removed or fed from a trusted conversation label before it participates in authorization.
