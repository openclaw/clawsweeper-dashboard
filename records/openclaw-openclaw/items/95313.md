## Summary

Keep open: the PR addresses a real Slack read-authorization regression, but the current patch still feeds the Slack authorization check from an untrusted context field on the normal core-built path.

## What This Changes

The PR changes Slack read-like actions to resolve channel names for channel-ID targets under name-based Slack allowlists, adds a current channel-name context field, and extends Slack action/channel-type tests.

## Best Possible Solution

Remove the untrusted context-name shortcut or source it from a trusted ConversationLabel/channel label with focused tests, while keeping the Slack-owned lazy conversations.info fallback for ID-only reads.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.8 check Slack read targets by ID only, while PR head can still fail the core-built same-context path because Channel is the provider id and lazy lookup is skipped when any channelName is present.

## Solution Assessment

No. The lazy conversations.info fallback is the right Slack-owned repair path, but the context shortcut is sourced from the wrong field and should be removed or backed by a trusted conversation label before merge.
