## Summary

Keep this PR open: the bug report is valid, but the patch has a blocking source-level authorization bug because it treats the generic provider value as a Slack channel name, and the external PR still lacks real Slack proof.

## What This Changes

The PR changes Slack read authorization to use a current channel name or lazily resolved Slack conversation name when authorizing channel-ID read targets against name-enabled allowlists.

## Best Possible Solution

Pass a real Slack conversation label/name into the threading context or leave currentChannelName unset so conversations.info resolves the name, then add regression coverage for the core-built context and redacted real Slack proof.

## Reproduction Assessment

Yes. Source inspection shows current main only checks the channel ID for read authorization, while PR head wires currentChannelName from a field that core currently fills with the provider id, so the intended context-name path and new tests do not match the real core-built context.

## Solution Assessment

No. The lazy conversations.info fallback is the right Slack-owned repair path, but the context shortcut is sourced from the wrong field and should be fixed or removed before merge.
