## Label Justifications

## Summary

Keep open: the lazy `conversations.info` repair is useful and now has redacted live-output proof, but the PR still feeds the core `Channel` field into Slack read authorization as a channel name. Current main passes the provider there, so same-context name-allowlisted reads can still fail and the authorization check can compare the wrong name.

## What This Changes

This PR updates Slack read-target authorization to resolve channel names for ID-targeted reads, threads a current channel name through Slack context, and adds Slack regression tests.

## Best Possible Solution

Remove the context-name shortcut or plumb a trusted `ConversationLabel` through the channel contract with focused tests; keep the Slack-owned lazy lookup for ID-only reads and preserve denied-channel coverage.

## Reproduction Assessment

Yes, at source level. Current main rejects name-keyed allowlist reads by channel ID, and PR head can still fail the core-built same-context path because `Channel` is the provider id while lazy lookup is skipped when any `channelName` is present.

## Solution Assessment

No. The lazy `conversations.info` fallback is the right Slack-owned repair path, but the context shortcut is sourced from the wrong field and should be removed or backed by a trusted conversation label before merge.
