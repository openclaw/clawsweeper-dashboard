Keep this PR open but do not merge it as submitted; the proposed Slack name-allowlist path can weaken read-target authorization, the no-context fallback is unreachable, and the external PR still lacks real Slack behavior proof.

## What This Changes

The PR changes Slack read-like action authorization so channel-ID targets can use name-based Slack channel allowlists when dangerous name matching is enabled, adds channel names to Slack tool context, and returns cached conversation names from `conversations.info`.

## Best Possible Solution

Keep the Slack-owned fix direction, but bind any context channel name to the same target channel ID, allow the lazy `conversations.info` fallback before denying unmatched allowlist IDs, and require redacted real Slack proof before merge.

Yes. Source inspection of PR head shows a high-confidence path where `downloadFile` can pass a supplied target channel ID plus the current tool context name into the read guard, and another path where an unmatched name allowlist throws before the lazy lookup runs.

## Solution Assessment

No. The owner boundary is right, but the implementation must first bind context names to their channel IDs and reorder the fallback before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Bind the channel name to the target ID:** `extensions/slack/src/action-runtime.ts:525`
  - body: `currentChannelName` describes the current conversation, but this guard uses it to authorize whichever `channelId` the action supplied. `download-file` already passes `ctx.toolContext`, so a request from an allowed `#allowed-channel` can supply another channel ID and have that other target checked against the allowed current name. Only use the context name when the target channel ID matches `context.currentChannelId`/target context, or resolve the target's own name first.
  - confidence: 0.94
- **[P2] Let the lazy name lookup run before denying:** `extensions/slack/src/action-runtime.ts:214-215`
  - body: For an allowlist keyed as `#allowed-channel` and a channel-ID target with no `currentChannelName`, `resolveSlackReadTargetAllowance` returns `explicitlyDenied: true`, so this throw runs before the `conversations.info` fallback below. That leaves the channel-ID permalink/tool path still rejected unless the caller already has a name. Treat unmatched allowlist IDs as unresolved until the name lookup has a chance to match.
  - confidence: 0.97
