## Label Justifications

- `P1`: The PR addresses a recent user-facing Slack regression in read-like actions while touching a Slack authorization boundary.
- `merge-risk: 🚨 security-boundary`: Merging changes Slack read authorization by allowing channel-ID reads based on API-resolved, mutable room names.

## Summary

Keep open: the PR targets a real Slack regression and has sufficient redacted live proof, but the current patch adds Slack room names to a process-lifetime cache and then uses that cached name for read authorization, which can authorize with stale channel names after a rename.

## What This Changes

The PR updates Slack read-like actions to resolve a Slack channel ID through `conversations.info` for name-allowlisted channels and adds regression tests.

## Best Possible Solution

Keep the Slack-owned API lookup design, but make names used for read authorization fresh or explicitly invalidated before Slack/security owner acceptance and merge.

## Reproduction Assessment

Yes for the reported regression: current main checks Slack read targets by channel ID only, so a name-only allowlist entry is rejected before the PR's API name lookup can occur. The stale-cache blocker is also source-reproducible from the process cache plus the channel rename path lacking invalidation.

## Solution Assessment

No, not as currently written. The Slack plugin is the right owner and the API-derived lookup is a good shape, but names used for authorization should not come from a process-lifetime cache without invalidation or a documented owner decision.
