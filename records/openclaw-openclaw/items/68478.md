## Summary

Keep this open: current main has partial controls for agent final replies and cron task notifications, but it still has explicit command/tool success reply paths and no general policy for suppressing chat-visible system confirmations. The requested default or config behavior needs maintainer product judgment because it would change message delivery expectations across commands, skills, cron, and transports.

## What This Changes

The issue asks for system-generated chat confirmations such as cron creation, skill installation, and generic done replies to be silent, suppressible, or grouped.

## Best Possible Solution

Maintainers should choose an explicit confirmation policy, likely class-based and opt-in at first, that suppresses low-value success confirmations while preserving errors, approvals, and native command responses that users expect to see.

## Reproduction Assessment

Yes at source level, but not via a live Telegram run in this read-only review. Current main still has a visible `✅ Done.` fallback for inline skill/tool commands and visible skill-install success payloads, while existing visible-reply config does not generally suppress native command confirmations.

## Solution Assessment

No, not as an automatic small bug fix. The requested direction is plausible, but deciding silent-by-default versus an opt-in config and deciding which confirmation classes remain visible is a product/config decision.
