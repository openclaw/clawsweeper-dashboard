## Summary

Keep this open: current main still has visible command, skill-install, and cron creation confirmation paths, and the requested suppression policy would change user-facing delivery/config behavior that needs maintainer product review rather than cleanup closure.

## What This Changes

The issue requests silent-by-default or configurable suppression of automatic chat-visible system confirmations such as cron, skill, and generic done messages.

## Best Possible Solution

Maintainers should choose an explicit class-based confirmation policy, likely opt-in first, that quiets low-value success confirmations while preserving errors, approvals, native command acknowledgements, and security-relevant notices.

## Reproduction Assessment

Yes at source level: command and skill/tool paths still emit visible success confirmations, and the existing message config does not provide a general suppression class. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No for an automatic bug fix: the requested suppression is useful, but the best solution needs a maintainer choice about default visibility and which confirmation classes stay visible. A broad config/default change should not be queued as a narrow repair without that policy decision.
