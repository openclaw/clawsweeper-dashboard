## Summary

Keep open: current main still classifies Bedrock inference-profile-prefixed embedding IDs against the raw model string, so `global.cohere.embed-v4:0` misses the Cohere v4 path and can build the Titan request body reported in the issue. The related fix PR was closed unmerged, so there is no current implementation to close this against.

## What This Changes

The issue requests Bedrock embedding lookup and family detection to strip AWS inference-profile prefixes while preserving the original profile ID for invocation.

## Best Possible Solution

Apply a plugin-local normalization helper for Bedrock system inference-profile prefixes in catalog/family/dimension lookup only, preserve the configured model ID for Bedrock invocation, and add focused regression coverage plus a changelog fix entry.

## Reproduction Assessment

Yes. By source inspection, configuring Bedrock memory search with `global.cohere.embed-v4:0` makes current `resolveSpec()` and `inferFamily()` miss the Cohere v4 catalog/family path and fall back to Titan v1; the issue and closed PR also provide live/reporter proof of the same failure mode.

## Solution Assessment

Yes. The narrow maintainable fix is to normalize AWS system inference-profile prefixes only for local Bedrock embedding lookup/family selection while leaving the full configured model ID intact for `InvokeModelCommand.modelId`.
