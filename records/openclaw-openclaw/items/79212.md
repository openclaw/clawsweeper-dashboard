## Summary

Keep open: current main and the latest release still classify Bedrock embedding models from the raw prefixed id, so `global.cohere.embed-v4:0` misses the Cohere v4 family and falls back to the Titan request shape. The exact-fix PR was closed unmerged, so no current open or merged item owns the remaining fix.

## What This Changes

The issue reports Bedrock memory embeddings failing for inference-profile-prefixed Cohere model ids because local spec/family lookup uses the raw prefixed id.

## Best Possible Solution

Land a narrow Bedrock embedding-provider fix that normalizes AWS system inference-profile prefixes only for local catalog/family lookup while preserving the configured model id for Bedrock invocation.

## Reproduction Assessment

Yes: source inspection and a focused read-only expression reproduce the current-main classification path where `global.cohere.embed-v4:0` misses the Cohere branch and selects `titan-v1`. I did not perform a live Bedrock call, so the reproduction is source-level rather than end-to-end.

## Solution Assessment

Yes: the best fix is to strip AWS system inference-profile prefixes only for local spec/family matching while preserving the original id for `InvokeModelCommand.modelId`. The closed unmerged PR showed the right shape, but the landed fix should include focused regression coverage and avoid `CHANGELOG.md`.
