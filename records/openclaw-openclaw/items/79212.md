## Summary

Keep open: current main still appears to misclassify Bedrock inference-profile-prefixed embedding model IDs, and the failure is source-reproducible in the bundled Amazon Bedrock plugin.

## What This Changes

The issue asks for Bedrock memory embeddings to classify inference-profile-prefixed model IDs by the underlying foundation model while still invoking Bedrock with the prefixed profile ID.

## Best Possible Solution

Apply a plugin-local normalization helper for Bedrock system inference-profile prefixes in catalog/family/dimension lookup only, preserve the configured model ID for `InvokeModelCommand.modelId`, and add regression coverage plus a changelog fix entry.

## Reproduction Assessment

Yes, by source inspection: configure Bedrock memory search with `global.cohere.embed-v4:0` and current `resolveSpec()`/`inferFamily()` will miss the Cohere catalog/family path and fall back to Titan v1. I did not live-call AWS Bedrock in this review.

## Solution Assessment

Yes, the narrow maintainable fix is to strip inference-profile prefixes only for Bedrock embedding lookup/family classification while preserving the full configured model for the runtime call. No core API, config option, or product decision appears necessary.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79212. In the Amazon Bedrock memory embedding provider, inference-profile-prefixed model IDs such as `global.cohere.embed-v4:0` and `us.cohere.embed-v4:0` are used directly for catalog/family lookup, so Cohere v4 falls back to Titan v1 request bodies. Add a small plugin-local normalization helper for Bedrock system inference-profile prefixes and use it only for `resolveSpec()`, `inferFamily()`, and dimension validation. Preserve the configured normalized model string on `client.model` and `InvokeModelCommand.modelId` so Bedrock still receives the inference profile ID. Add focused regression tests proving prefixed Cohere v4 uses the Cohere v4 query/batch body shape, preserves the full `modelId`, carries max token/dimension metadata, and validates `outputDimensionality` against the bare model spec. Keep the change in `extensions/amazon-bedrock`; do not add a config option or move behavior into core. Add a user-facing `CHANGELOG.md` fix entry.
