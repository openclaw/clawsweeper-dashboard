## Summary

Keep open: the PR addresses a real WhatsApp outbound/inbound allowlist limitation that is still present on current main, but it adds a new public config/SDK contract and overlaps an open alternative PR, so it needs maintainer product review plus real WhatsApp behavior proof before merge.

## What This Changes

The PR adds optional WhatsApp `allowSendTo` root/account config and wires it through account resolution, outbound target authorization, cron delivery, status output, schema/docs, tests, and plugin SDK contracts.

## Best Possible Solution

Choose one outbound allowlist contract across the overlapping WhatsApp PRs, keep inbound `allowFrom` semantics unchanged, and land the selected implementation only after a real paired-session proof shows outbound send and inbound gating behavior.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection on current main does show the relevant path clearly: WhatsApp direct outbound resolution uses `allowFrom` only, so a target outside that list is denied by source behavior.

## Solution Assessment

Unclear until maintainers choose the public contract. The `allowSendTo` approach is coherent and better named than the open `allowTo` alternative, but it adds config and SDK surface and still needs real WhatsApp proof before merge.
