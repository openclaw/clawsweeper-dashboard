## Summary

Keep open. Current main appears to have the shared Copilot IDE-header path covered for OpenAI-family transports, but Gemini Copilot models still resolve to `openai-responses`, matching the reported unsupported Responses API failure.

## What This Changes

The issue requests a GitHub Copilot provider fix so Gemini/chat-completions-only models use `openai-completions` with Copilot IDE auth headers instead of falling through to paid fallback models.

## Best Possible Solution

Land a focused GitHub Copilot provider fix that routes Gemini/chat-completions-only Copilot IDs to `openai-completions`, preserves IDE-header injection, updates provider docs and changelog, and leaves broader fallback-cost warnings to a separate follow-up.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main maps bundled and dynamic Copilot Gemini IDs to `openai-responses`, while the reporter supplied direct Copilot API evidence that Gemini rejects Responses and succeeds through chat completions.

## Solution Assessment

Yes. The narrowest maintainable direction is provider-owned transport selection plus regression coverage; changing generic OpenAI transport behavior or bundling the fallback-spend warning would be wider than this bug requires.
