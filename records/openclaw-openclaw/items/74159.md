## Summary

Keep open. Current main still routes GitHub Copilot Gemini model IDs to `openai-responses` through both the bundled catalog and the dynamic resolver, so the central unsupported Responses API bug remains. The second reported header problem appears addressed on current main because the shared OpenAI completions client now injects Copilot IDE headers for `github-copilot` models.

## What This Changes

Route GitHub Copilot Gemini and other chat-completions-only Copilot models through `openai-completions` while preserving Copilot IDE auth headers so requests do not silently fall back to paid providers.

## Best Possible Solution

Implement a focused GitHub Copilot provider fix: route known and forward-compatible Gemini model IDs to `openai-completions`, keep Claude on Anthropic Messages and known OpenAI-family models on Responses, preserve the shared Copilot IDE-header injection, add regression tests for bundled/default/dynamic Gemini rows, and update the provider docs plus changelog. Keep the broader repeated-fallback cost warning as a separate follow-up.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: resolving bundled or dynamic `github-copilot/gemini-*` models on current main assigns `openai-responses`, while the reporter supplied live Copilot API evidence that Gemini rejects Responses and succeeds through chat completions.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the GitHub Copilot provider-owned model catalog/resolver plus tests/docs; generic OpenAI completions header injection is already covered, and changing fallback policy would be wider than needed for this bug.
