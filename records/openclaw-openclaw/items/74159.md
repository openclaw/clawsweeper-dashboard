## Summary

Keep open. Current main still routes built-in, generated, and synthetic GitHub Copilot Gemini IDs through OpenAI Responses, so the reported unsupported `/responses` failure remains; the separate `openai-completions` IDE-header bypass no longer appears present because the shared completions client injects Copilot IDE headers for `github-copilot` models.

## What This Changes

The issue asks OpenClaw to route GitHub Copilot Gemini/chat-completions-only models away from unsupported Responses requests while preserving Copilot IDE auth headers.

## Best Possible Solution

Land a provider-owned Copilot transport fix that routes Gemini/chat-completions-only ids to `openai-completions`, preserves shared IDE-header injection, and updates regression tests, docs, and changelog.

## Reproduction Assessment

Yes, at source level. Current main maps Copilot Gemini ids to `openai-responses`, and the reporter supplied direct Copilot API evidence that those Gemini ids reject `/responses` while accepting `/chat/completions`.

## Solution Assessment

Yes. The narrow maintainable fix is Copilot-specific transport selection plus focused coverage; the optional repeated-fallback spend warning is useful but should be tracked separately from this endpoint-routing bug.
