## Summary

Keep open. Current main has addressed the shared Copilot IDE-header path for OpenAI-family transports, but GitHub Copilot Gemini model IDs still resolve to `openai-responses`, matching the reporter's unsupported Responses API failure. The open auto-discovery PR does not supersede this because its stated inference still maps non-Claude models to Responses.

## What This Changes

The issue requests a GitHub Copilot provider fix so Gemini and other chat-completions-only Copilot models use `openai-completions` with Copilot IDE headers instead of failing through `/responses` and falling back to paid providers.

## Best Possible Solution

Land a focused GitHub Copilot provider fix that routes known and forward-compatible Gemini model IDs to `openai-completions`, keeps the shared Copilot IDE-header injection on that path, adds regression coverage for bundled/default/dynamic Gemini rows, updates the provider docs and changelog, and leaves repeated fallback-cost warnings for a separate follow-up.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main resolves bundled and dynamic GitHub Copilot Gemini IDs to `openai-responses`, while the reporter supplied direct Copilot API evidence that Gemini rejects Responses and succeeds through chat completions.

## Solution Assessment

Yes. The narrow maintainable solution is provider-owned Gemini transport selection plus tests/docs; changing generic OpenAI transport behavior or bundling the broader fallback-cost warning would be wider than needed for the reported regression.
