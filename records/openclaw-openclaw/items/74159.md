## Summary

Keep open. Current main still routes GitHub Copilot Gemini model IDs through the Responses transport, while the shared OpenAI transport now appears to add Copilot IDE headers for OpenAI-family requests; that leaves a valid, focused provider routing bug rather than a cleanup close.

## What This Changes

The issue requests that GitHub Copilot Gemini models use the chat/completions transport with Copilot IDE headers so they do not fail on unsupported Responses requests and silently fall back to paid models.

## Best Possible Solution

Land a focused GitHub Copilot provider fix that routes bundled and forward-compatible Gemini model IDs to openai-completions, keeps the shared Copilot IDE header injection covered by tests, updates provider docs and the changelog, and treats the broader repeated-fallback cost warning as a separate follow-up.

Yes. The source-level path is high confidence: current main resolves both bundled and dynamic GitHub Copilot Gemini IDs to openai-responses, and the reporter supplied direct Copilot API evidence that Gemini rejects that endpoint but works through chat/completions.

## Solution Assessment

Yes. The narrow maintainable path is provider-owned Gemini transport selection plus regression coverage; changing generic OpenAI transport policy or bundling the optional fallback-cost warning into this fix would be wider than needed.
