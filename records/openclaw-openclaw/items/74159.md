## Summary

Keep open. Current main still routes GitHub Copilot Gemini built-in, generated-default, and dynamic model definitions to OpenAI Responses while the reporter supplied direct Copilot API evidence that Gemini works on chat completions instead; the IDE-header side appears covered by the shared OpenAI client header path.

## What This Changes

The issue asks GitHub Copilot Gemini/chat-completions-only models to use `/chat/completions` with required IDE headers instead of failing on `/responses` and falling back.

## Best Possible Solution

Land a provider-owned Copilot transport fix that routes verified Gemini/chat-completions-only IDs to `openai-completions`, preserves shared IDE-header injection, updates docs and changelog, and leaves the fallback-spend warning for a separate product decision.

## Reproduction Assessment

Yes. Current main source maps Copilot Gemini IDs to `openai-responses`, and the reporter supplied direct Copilot API evidence that Gemini rejects `/responses` while accepting `/chat/completions`; I did not need a live token to confirm that routing path.

## Solution Assessment

Yes. The narrow maintainable fix is Copilot-specific transport metadata plus regression coverage; adding a repeated-fallback spend warning would be useful but is a separate feature/policy change.
