## Summary

Keep open. Current `origin/main` still routes GitHub Copilot Gemini IDs to `openai-responses` through the provider default, generated/default model builder, and dynamic resolver, so the central unsupported Responses API failure remains. The separate IDE-header complaint appears addressed on current main because the shared OpenAI client header helper now injects Copilot IDE headers for `github-copilot` models, including the chat-completions client path.

## What This Changes

Route GitHub Copilot Gemini and other chat-completions-only Copilot models through `openai-completions` while preserving Copilot IDE auth headers to avoid silent paid-provider fallback.

## Best Possible Solution

Land a focused GitHub Copilot provider fix that routes known and forward-compatible Gemini IDs to `openai-completions`, keeps Claude on Anthropic Messages and known OpenAI-family models on Responses, preserves the shared Copilot IDE-header injection, adds regression coverage for bundled/default/dynamic Gemini rows, and updates the provider docs plus changelog. Keep the broader repeated-fallback cost warning as a separate follow-up.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: resolving bundled or dynamic `github-copilot/gemini-*` models on current main assigns `openai-responses`, while the reporter supplied direct Copilot API evidence that Gemini rejects Responses and succeeds through chat completions.

## Solution Assessment

Yes. The narrow maintainable solution is provider-owned Gemini transport selection with tests/docs; generic OpenAI completions header injection already exists, and changing fallback policy would be wider than needed for this bug.
