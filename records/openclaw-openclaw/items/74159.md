## Summary

Keep open. Current main has addressed the shared Copilot IDE-header path for OpenAI-family transports, but GitHub Copilot Gemini model IDs still resolve to `openai-responses`, which matches the reporter's unsupported Responses API failure and leaves a narrow provider-owned bug to fix.

## What This Changes

The issue requests that GitHub Copilot Gemini/chat-completions-only models route to `openai-completions` with Copilot IDE auth headers so they do not fail and fall back to paid fallback models.

## Best Possible Solution

Land a focused GitHub Copilot provider fix that routes known and forward-compatible Gemini model IDs to `openai-completions`, keeps Copilot IDE headers on that path through the shared OpenAI transport, adds regression coverage for bundled/default/dynamic Gemini rows, updates the GitHub Copilot provider docs and changelog, and leaves repeated-fallback cost warnings for a separate follow-up.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main resolves bundled and dynamic GitHub Copilot Gemini IDs to `openai-responses`, while the reporter supplied direct Copilot API evidence that Gemini rejects Responses and succeeds through chat completions.

## Solution Assessment

Yes. The narrow maintainable fix is provider-owned Gemini transport selection with tests and docs; changing generic OpenAI transport behavior or bundling the broader fallback-cost warning would be wider than needed for the reported regression.
