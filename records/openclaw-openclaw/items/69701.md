## Summary

Keep open. Current main and latest release still lack the Windows `::1` skip, and the PR is a narrow patch for the gateway loopback bind path with focused tests; the remaining actionable issue is the required changelog entry before landing.

## What This Changes

The PR adds a win32 early return in `resolveGatewayListenHosts` so loopback binds only `127.0.0.1`, and adds unit tests for Windows skip and non-Windows dual-loopback behavior.

## Best Possible Solution

Land this narrow win32 guard with focused regression coverage and a changelog entry, or land an equivalent fix that prevents Windows from binding both `127.0.0.1` and `::1` on the same gateway port while preserving non-Windows dual-loopback behavior.

## Reproduction Assessment

Yes. The linked bug gives concrete Windows 11 gateway and `curl` steps, and current main statically maps the default loopback path to dual bind-hosts when `::1` is bindable; this read-only review did not run a live Windows repro.

## Solution Assessment

Yes, with one process cleanup. The PR is the narrowest maintainable code fix for the proposed Windows dual-bind failure mode; if maintainers require Windows `::1`, a separate IPv6-only socket design would be safer than mixing that into this fix.
