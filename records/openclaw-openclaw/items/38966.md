## Summary

Keep open: current main still lacks the TUI subscription and handler needed to react to an external reset of the current session, and open PR #40472 is already the paired implementation candidate.

## What This Changes

The issue requests that the TUI clear or reload stale chat, token, and run state when another client resets the same gateway session.

## Best Possible Solution

Review and land or replace #40472 so the TUI subscribes to `sessions.changed`, handles current-session `new`/`reset` events, refreshes session history/metadata, clears stale run state, and ignores unrelated sessions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the current TUI session from another gateway client; current main can emit `sessions.changed`, but the TUI neither subscribes to nor handles that event family.

## Solution Assessment

Yes. Reusing the existing `sessions.changed` subscription/event path is the narrow maintainable direction, and #40472 already follows that path, so this issue should stay open until that PR or a replacement merges.
