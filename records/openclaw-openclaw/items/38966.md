## Summary

Keep open: current main has the gateway-side `sessions.changed` reset signal, but the TUI still does not subscribe to or handle that event, so the reported stale TUI state remains. The linked open fix path is https://github.com/openclaw/openclaw/pull/40472, which uses closing syntax for this issue and should remain the canonical implementation path until it lands or is replaced.

## What This Changes

The issue asks for the TUI to refresh when the active session is reset externally from another client via `/new` or `/reset`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a TUI consumer for `sessions.subscribe` / `sessions.changed` reset and new events that refreshes the active session and clears stale run/token state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the active TUI session from another gateway client, and current main can emit `sessions.changed` while the TUI neither subscribes to nor handles that event family.

## Solution Assessment

Yes. Reusing the existing `sessions.subscribe` / `sessions.changed` lifecycle contract is the narrow maintainable fix; adding a new event family or polling loop would be broader than needed.
