## Summary

Current main still does not complete the external reset refresh path: the gateway emits subscriber-gated `sessions.changed` events, but the TUI does not subscribe to or dispatch that event family, and open PR #40472 is already the paired implementation candidate.

## What This Changes

The issue asks for the TUI to refresh its chat log, token counts, and stale run state when the current session is reset from another client.

## Best Possible Solution

Review and land or replace #40472 so the TUI subscribes to `sessions.changed`, handles current-session `new`/`reset` events, refreshes history/session metadata, clears stale run state, and ignores unrelated sessions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the current TUI session from another gateway client; current main can emit `sessions.changed`, but the TUI neither subscribes to nor handles that event family.

## Solution Assessment

Yes. Reusing the existing `sessions.changed` subscription/event path is the narrow maintainable fix; adding a second reset event family or polling loop would be broader than needed.
