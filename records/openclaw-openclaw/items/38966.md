## Label Justifications

## Summary

Keep open: current main now broadcasts reset state through `sessions.changed`, but the TUI still dispatches only chat and agent events, so an external reset can leave its transcript, token display, and run tracking stale. The open linked implementation remains the appropriate landing candidate, although it must be rebased and narrowed around the newer canonical event contract. Likely related people: Shakker for the recent session-lifecycle work, and wsyjh8 for the original focused implementation attempt.

## What This Changes

Ensure the TUI refreshes transcript, token, and in-memory run state when its active session is reset from another client.

## Best Possible Solution

Keep the canonical `sessions.changed` reset broadcast and update the TUI to match the active session, reload history and session metadata, reset displayed usage, and invalidate stale run/tool tracking without adding a parallel event type.

## Reproduction Assessment

Yes, from current source. Gateway reset emits `sessions.changed`, while the TUI event dispatcher has no handler for that event, directly explaining why its active view is not refreshed.

## Solution Assessment

No, the original suggestion to introduce a separate gateway `session` event is no longer the best solution because current main already provides the canonical `sessions.changed` reset notification; the remaining fix should consume that event in the TUI.
