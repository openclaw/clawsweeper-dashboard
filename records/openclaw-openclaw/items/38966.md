## Summary

Keep open: current main still has the TUI gap, because the gateway emits `sessions.changed` after reset but the TUI neither subscribes to that event family nor dispatches a session-change handler. The active fix path is https://github.com/openclaw/openclaw/pull/40472, which is open and uses closing syntax for this issue.

## What This Changes

The issue asks for the TUI to clear or reload the active session view and reset stale token/run state when another client resets that same session.

## Best Possible Solution

Review, rebase or replace, and land https://github.com/openclaw/openclaw/pull/40472 with the existing `sessions.subscribe` / `sessions.changed` approach, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the active TUI session from another gateway client, and current main can emit `sessions.changed` while the TUI neither subscribes to nor dispatches that event family.

## Solution Assessment

Yes. Reusing the existing `sessions.subscribe` / `sessions.changed` lifecycle contract is the narrow maintainable fix; adding a separate event family or polling loop would be broader than needed.
