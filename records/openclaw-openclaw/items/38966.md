## Summary

Keep open: current main still has the source-level stale-TUI path, and the open closing PR at https://github.com/openclaw/openclaw/pull/40472 remains the canonical implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue requests that the TUI refresh transcript, token, and run state when the active session is reset from another connected client such as Telegram.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that refreshes the active session and clears stale run/token state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the active TUI session from another gateway client, and current main can emit `sessions.changed` while the TUI neither subscribes to nor dispatches that event family.

## Solution Assessment

Yes. Reusing the existing `sessions.subscribe` / `sessions.changed` lifecycle contract is the narrow maintainable fix; adding a separate event family or polling loop would be broader than needed.
