## Summary

Keep open: current main and v2026.5.20 still have a gateway `sessions.changed` reset event source, but the TUI neither subscribes to that event family nor dispatches it to a reset handler. A closing implementation PR is already open at https://github.com/openclaw/openclaw/pull/40472, so this issue should wait for that PR to be rebased, proven, landed, or replaced.

## What This Changes

The issue asks for the TUI to refresh/clear stale transcript, token, and run state when another client resets the same gateway session.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that refreshes the active session and clears reset-sensitive run state.

## Reproduction Assessment

Yes, by source inspection. Current main and v2026.5.20 can emit subscription-gated `sessions.changed` reset/new events, while the TUI neither subscribes to nor handles that event family.

## Solution Assessment

Yes for the direction. Reusing the existing `sessions.subscribe` and `sessions.changed` contract is the narrow maintainable fix, but the open PR still needs rebase/proof or replacement before this issue closes.
