## Summary

Keep open: current main emits a subscription-gated `sessions.changed` event after `sessions.reset`, but the TUI still does not subscribe to or dispatch that event family, so the reported stale TUI session state remains source-reproducible. The open closing PR at https://github.com/openclaw/openclaw/pull/40472 is still the canonical implementation path, but it is not merged.

## What This Changes

The issue asks for the TUI to refresh history, token counts, and stale run state when another client resets the active session.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that refreshes the active session and clears stale run state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the active TUI session from another gateway client, and current main can emit `sessions.changed` while the TUI neither subscribes to nor dispatches that event family.

## Solution Assessment

Yes for the direction. Reusing the existing `sessions.subscribe` / `sessions.changed` lifecycle contract is the narrow maintainable fix; the issue should stay open until the linked PR or a replacement lands.
