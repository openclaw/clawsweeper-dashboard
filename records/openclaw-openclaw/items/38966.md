## Summary

Keep open: current main has a gateway-side `sessions.changed` reset event, but the TUI still neither subscribes to session lifecycle events nor dispatches them, so an external reset can leave stale TUI history and run state visible. The open closing PR at https://github.com/openclaw/openclaw/pull/40472 remains the canonical fix path and should be reviewed, rebased, or replaced before this issue closes.

## What This Changes

The issue requests that the TUI refresh and clear stale session, token, and run state when the active session is reset by another gateway client such as Telegram `/new`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that refreshes the active session and clears active, pending, local, watchdog, session, and finalized run state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the active TUI session from another gateway client, and current main can emit subscription-gated `sessions.changed` while the TUI neither subscribes to nor dispatches that event family.

## Solution Assessment

Yes for the direction. Reusing the existing `sessions.subscribe` and `sessions.changed` contract is the narrow maintainable fix; a parallel event family or polling loop would be broader than needed.
