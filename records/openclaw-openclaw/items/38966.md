## Summary

Current main still leaves the reported external-reset refresh path incomplete: the gateway emits reset changes through the subscriber-gated session-change event path, but the TUI client does not subscribe to those events and the TUI dispatcher/handler layer still ignores them. Because an open PR already targets this issue, it should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for the TUI to refresh history/session state, clear stale token/run state, and optionally show a notice when the active session is reset from another client such as Telegram.

## Best Possible Solution

Review and land #40472, or an equivalent narrow fix, so the TUI subscribes to session-change events, handles current-session `new`/`reset` changes, refreshes history/session metadata, clears stale run/stream/local-run state, and ignores unrelated sessions.
