## Summary

Keep open: current main still does not refresh the TUI after an external reset because the TUI neither subscribes to `sessions.changed` nor handles session lifecycle events. The active implementation path is https://github.com/openclaw/openclaw/pull/40472, which is still open and uses closing syntax for this issue.

## What This Changes

The issue requests that the TUI clear or reload session state when another client resets the same gateway session.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a fix that subscribes the TUI to session changes, handles current-session reset/new events, reloads history/session metadata, clears stale token/run state, and ignores unrelated sessions.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: reset the active TUI session from another gateway client, and current main can emit `sessions.changed` while the TUI neither subscribes to nor handles that event family. I did not run a live multi-client TUI/Telegram repro in this read-only sweep.

## Solution Assessment

Yes, reusing the existing `sessions.changed` subscription/event path is the narrow maintainable solution. Adding a separate reset event family or polling loop would be broader than needed.
