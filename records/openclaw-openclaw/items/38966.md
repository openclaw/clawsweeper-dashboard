## Summary

Keep this open: the source-level bug still exists on current main and v2026.6.5 because the gateway emits subscription-gated session-change events, but the TUI neither subscribes to nor handles the event family needed to refresh after an external reset. There is also an open conflicting linked fix PR, so this issue should remain the canonical tracker until that path is landed, replaced, or closed.

## What This Changes

The issue requests that the TUI refresh its chat history, token counts, and stale run state when the same session is reset from another client such as Telegram.

## Best Possible Solution

Reuse the existing `sessions.subscribe` plus `sessions.changed` contract: make TUI subscribe, handle matching current-session reset/new changes, clear stale run and token state, reload history, and prove the multi-client reset path before closing this issue.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: reset/new events can be emitted as subscription-gated `sessions.changed` events, while the TUI neither subscribes to nor handles that event family.

## Solution Assessment

Is this the best way to solve the issue? The issue direction is valid, but the best implementation should reuse current `sessions.changed` infrastructure rather than add a parallel `session` event.
