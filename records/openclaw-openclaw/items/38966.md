## Summary

Keep open: current main still has the reset signal on the gateway side, but the TUI does not subscribe to or dispatch `sessions.changed`, so an external reset cannot refresh the visible TUI session. The active implementation path is https://github.com/openclaw/openclaw/pull/40472, which is still open and uses closing syntax for this issue.

## What This Changes

The issue asks for the TUI to refresh the current session view, token state, and stale run state when another client resets the same session.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a fix that subscribes the TUI to `sessions.changed`, handles reset/new for the current session, reloads history/session metadata, clears stale token/run state, and ignores unrelated sessions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the active TUI session from another gateway client, and current main can emit `sessions.changed` while the TUI neither subscribes to nor handles that event family.

## Solution Assessment

Yes. Reusing the existing `sessions.subscribe` / `sessions.changed` lifecycle contract is the narrow maintainable fix; adding a separate event family or polling loop would be broader than needed.
