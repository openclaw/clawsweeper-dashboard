## Summary

Keep open: current main has the gateway-side `sessions.changed` reset event, but the TUI still does not subscribe to or dispatch that lifecycle event, so an external reset can leave stale TUI state visible. The open closing PR at https://github.com/openclaw/openclaw/pull/40472 remains the canonical implementation path and should be reviewed, repaired, or replaced before this issue closes.

## What This Changes

The issue asks the TUI to refresh its active session view, token counts, and stale run state when the same session is reset from another gateway client such as Telegram.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that refreshes the active session and clears active, pending, local, watchdog, session, and finalized run state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the active TUI session from another gateway client, and current main can emit subscription-gated `sessions.changed` while the TUI neither subscribes to nor dispatches that event family.

## Solution Assessment

Yes for the direction. Reusing the existing `sessions.subscribe` / `sessions.changed` lifecycle contract is the narrow maintainable fix; adding a parallel event family or polling loop would be broader than needed.
