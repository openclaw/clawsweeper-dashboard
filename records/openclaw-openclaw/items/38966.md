## Summary

Keep open: current main still has the stale-TUI path. The gateway emits subscription-gated `sessions.changed` reset/new events, but the TUI currently neither subscribes to nor dispatches that event family; the open closing PR at https://github.com/openclaw/openclaw/pull/40472 remains the canonical implementation path until it lands or is replaced.

## What This Changes

The issue asks for the TUI to refresh its active session view when another client resets the same gateway session.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that refreshes the active session and clears stale run/token state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the active TUI session from another gateway client, and current main can emit `sessions.changed` while the TUI neither subscribes to nor dispatches that event family.

## Solution Assessment

Yes. Reusing the existing `sessions.subscribe` and `sessions.changed` lifecycle contract is the narrow maintainable fix; adding a separate event family or polling loop would be broader than needed.
