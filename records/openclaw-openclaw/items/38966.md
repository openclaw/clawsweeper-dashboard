## Summary

Keep open: current main still leaves the TUI outside the existing session-change event path, so an external reset can leave stale TUI session state. The target-repo fix PR was closed unmerged, and the only open cross-repo PR reference is conflicting and not a merge-ready OpenClaw target path.

## What This Changes

Request that the TUI refresh or clear its visible session state when another client resets the same gateway session.

## Best Possible Solution

Use the existing `sessions.subscribe` and `sessions.changed` contract: have the TUI subscribe, handle matching reset/new changes for the current session, clear stale run/token state, reload history/session info, and prove the multi-client reset path.

## Reproduction Assessment

Yes for a source-level reproduction: the gateway emits reset updates through subscriber-gated `sessions.changed`, while the TUI neither subscribes to nor handles that event. I did not run a live TUI plus Telegram workflow in this read-only review.

## Solution Assessment

Yes on direction, but not as a new parallel `session` event. The narrow maintainable fix is to reuse the existing `sessions.changed` session lifecycle contract already used by Control UI and add TUI reset-state handling around it.
