## Label Justifications

- `P2`: This is a normal-priority user-visible TUI/session-state bug with a bounded gateway and TUI surface.
- `impact:session-state`: The reported behavior leaves stale transcript, token, and run state visible after another client resets the same session.

## Summary

Keep open: current main still emits reset/new session-change events, but the TUI does not subscribe to `sessions.changed` and only dispatches `chat`, `chat.side_result`, and `agent` events, so an external reset cannot refresh the active TUI view. The original target-repo PR is closed unmerged, and an open fork PR still references this issue, so this should wait for a current target PR or an explicit replacement path.

## What This Changes

The issue asks the TUI to refresh history, token display, and reset-sensitive run state when another client resets the active session.

## Best Possible Solution

Port, review, and land a current-main target-repo fix that subscribes the TUI to `sessions.changed`, filters to the active session, reloads history/session info, and clears reset-sensitive run state.

## Reproduction Assessment

Yes, by source inspection. Current main emits subscription-gated `sessions.changed` reset/new events, while the TUI neither subscribes to nor dispatches that event family.

## Solution Assessment

Yes for the direction. Reusing the existing `sessions.subscribe` and `sessions.changed` contract is the narrow maintainable solution; the implementation still needs to clear active, pending, local, finalized, and watchdog run state before refreshing.
