## Summary

Keep open: current main still leaves the TUI disconnected from external session-reset lifecycle updates. The linked implementation at https://github.com/BingqingLyu/openclaw/pull/206 is open and uses closing syntax, but it is unmerged and currently conflicting, so this issue remains the canonical tracker.

## What This Changes

The issue asks for the TUI to refresh session history, token counts, and stale run state when the same session is reset from another client such as Telegram.

## Best Possible Solution

Use the existing `sessions.changed` subscription contract: subscribe the gateway-backed TUI, match the active session and agent scope, then handle reset/new as an atomic transition that reloads history/session metadata and clears stale transcript, token, streaming, optimistic-message, and run projections.

## Reproduction Assessment

Yes. On current main and v2026.6.5, external reset can emit `sessions.changed` only to subscribed clients, while the TUI neither subscribes nor handles that event, so its local view cannot refresh from that reset path.

## Solution Assessment

Yes, the event-driven direction is the best narrow fix, but it should reuse the existing `sessions.changed` contract rather than add a parallel reset-only event. The linked PR is the current implementation candidate, but it needs conflict resolution and review before this issue can close.
