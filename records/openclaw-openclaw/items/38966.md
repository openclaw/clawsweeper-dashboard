## Summary

Keep open: current main emits the gateway-side session-change signal, but the TUI still does not subscribe to or dispatch `sessions.changed`, so an externally reset active session cannot refresh the visible TUI state. The linked open fix path is https://github.com/openclaw/openclaw/pull/40472, which uses closing syntax for this issue and should remain the canonical implementation path until it lands or is replaced.

## What This Changes

The issue requests that the TUI refresh transcript, token, and run state when the active session is reset from another client such as Telegram.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a fix that subscribes the TUI to `sessions.changed`, handles reset/new for the current session, reloads history/session metadata, clears stale run/token state, and ignores unrelated sessions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the active TUI session from another gateway client, and current main can emit `sessions.changed` while the TUI neither subscribes to nor handles that event family.

## Solution Assessment

Yes. Reusing the existing `sessions.subscribe` / `sessions.changed` lifecycle contract is the narrow maintainable fix; adding a new event family or polling loop would be broader than needed.
