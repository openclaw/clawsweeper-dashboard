## Summary

Keep open: current main and the latest release have the gateway-side `sessions.changed` reset event, but the TUI still does not subscribe to or dispatch that event family, so the reported stale TUI state remains source-reproducible. The open closing PR at https://github.com/openclaw/openclaw/pull/40472 is still the canonical fix path and should be reviewed, rebased, proven, or replaced before this issue closes.

## What This Changes

The issue requests that TUI detect session resets initiated by another client and refresh or clear stale transcript, token, and run state.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that refreshes the active session and clears reset-sensitive run state.

## Reproduction Assessment

Yes, source-level. On current main and latest release, another client can trigger subscription-gated `sessions.changed` reset events while the TUI neither subscribes to nor dispatches that event family.

## Solution Assessment

Yes for the direction. Reusing the existing `sessions.subscribe` and `sessions.changed` contract is the narrow maintainable solution; the open PR still needs to be rebased, proven, or replaced before this issue closes.
