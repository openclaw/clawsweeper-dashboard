## Summary

Keep open. Current main still has the TUI consumer gap: the gateway emits reset/new changes through the existing session-event subscription path, but the TUI neither subscribes to that path nor dispatches `sessions.changed`; the linked open PR is the active fix path and should not be bypassed before it lands.

## What This Changes

Refresh the TUI when the active session is reset from another client, clearing stale transcript, token, and run state.

## Best Possible Solution

Use the existing `sessions.subscribe` / `sessions.changed` contract in the TUI, clear active and pending run state on matching reset/new events, and land or replace the linked fix PR with real two-client proof.

## Reproduction Assessment

Yes, by source inspection. Current main emits `sessions.changed` for reset/new only to subscribed clients, while the TUI neither subscribes nor handles that event family, so a reset from another client leaves the active TUI view stale.

## Solution Assessment

Partly. Reusing the existing `sessions.subscribe` / `sessions.changed` path is the narrow maintainable solution; adding a separate new event family is less aligned with the current protocol contract.
