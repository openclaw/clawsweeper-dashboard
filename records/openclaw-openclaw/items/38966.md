## Summary

Keep open: current main has the gateway-side `sessions.changed` reset event, but the TUI still does not subscribe to or dispatch session lifecycle events, so the reported stale TUI state remains source-reproducible. The open closing PR at https://github.com/openclaw/openclaw/pull/40472 is still the canonical fix path and should be reviewed, rebased, proven, or replaced before this issue closes.

## What This Changes

The issue asks for the TUI to refresh its active session state when another client resets that same gateway session.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that refreshes the active session and clears active, pending, local, watchdog, session, and finalized run state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the active TUI session from another gateway client, and current main can emit subscription-gated `sessions.changed` while the TUI neither subscribes to nor dispatches that event family.

## Solution Assessment

Yes for the direction. Reusing the existing `sessions.subscribe` and `sessions.changed` contract is the narrow maintainable solution; the open PR still needs to be rebased/proven or replaced before this issue closes.
