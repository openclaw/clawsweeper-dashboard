## Summary

Current main still does not complete the external session reset refresh path for the TUI. The gateway emits subscriber-gated `sessions.changed` events after `sessions.reset`, but the TUI client does not subscribe to those events and the TUI dispatcher/event-handler layer still ignores them; the issue is also paired with open implementation PR #40472, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

Refresh the TUI when another client resets the current session, including clearing stale transcript, token, and run state.

## Best Possible Solution

Review, land, or close #40472. The implementation should make the TUI subscribe to `sessions.changed`, dispatch and handle current-session `new`/`reset` events, refresh history/session metadata, clear stale active-run/stream/local-run state, ignore unrelated sessions, and add focused regression coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: trigger `sessions.reset` from a non-TUI client while the TUI is connected to the same session; current main can emit `sessions.changed`, but the TUI neither subscribes to nor handles that event, so it cannot refresh from that external reset.

## Solution Assessment

Yes, the linked direction is the narrow maintainable fix. Reusing the existing `sessions.changed` subscription/event family and adding TUI handling is preferable to inventing a parallel reset event or polling loop.
