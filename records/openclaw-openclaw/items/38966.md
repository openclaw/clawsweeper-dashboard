## Summary

Keep open: current main still emits session lifecycle changes from the gateway, but the TUI does not subscribe to or dispatch those events, so the stale-session-state path remains. The open closing PR at https://github.com/openclaw/openclaw/pull/40472 is still the canonical implementation path and should be reviewed or replaced rather than closing this issue before merge.

## What This Changes

The issue asks for the TUI to refresh history, token counts, and run state when the active session is reset by another gateway client.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that refreshes the active session and clears active, pending, local, watchdog, session, and finalized run state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the active TUI session from another gateway client, and current main can emit subscription-gated `sessions.changed` while the TUI neither subscribes to nor dispatches that event family.

## Solution Assessment

Yes for the direction. Reusing the existing `sessions.subscribe` / `sessions.changed` lifecycle contract is the narrow maintainable fix; a new parallel event family or polling loop would be broader than needed.
