## Summary

Keep open: current main and the latest shipped tag still leave the TUI without a `sessions.subscribe` call or a `sessions.changed` handler, so the stale TUI state remains. The open closing PR at https://github.com/openclaw/openclaw/pull/40472 is still the canonical fix path and should be reviewed, rebased, landed, or replaced rather than closing this issue first.

## What This Changes

The issue reports stale TUI transcript, token, and run state when another gateway client resets the same session.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that refreshes the active session and clears stale run/token state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the active TUI session from another gateway client, and current main can emit `sessions.changed` while the TUI neither subscribes to nor handles that event family.

## Solution Assessment

Yes. Reusing the existing `sessions.subscribe` / `sessions.changed` lifecycle contract is the narrow maintainable fix; adding a separate event family or polling loop would be broader than needed.
