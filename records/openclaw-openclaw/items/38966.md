## Summary

Keep open: current main still emits `sessions.changed` for `sessions.reset`, but the TUI does not subscribe to that event family or dispatch it to a reset/history refresh handler. A linked fork PR is open but conflicting and not mergeable in the target repo as-is, so this issue should stay open until that work is revived, replaced, or landed.

## What This Changes

The issue reports stale TUI transcript, token, and run state after the same session is reset from another client such as Telegram.

## Best Possible Solution

Land or replace the linked fix path with a current-main TUI subscription to `sessions.subscribe`/`sessions.changed` that refreshes only the active session and clears reset-sensitive run state.

## Reproduction Assessment

Yes, by source inspection. Current main emits subscription-gated `sessions.changed` reset/new events, while the TUI neither subscribes to nor handles that event family, so an external reset cannot trigger `loadHistory()` or run-state cleanup.

## Solution Assessment

Yes for the direction. Reusing the existing `sessions.subscribe` and `sessions.changed` contract is the narrow maintainable fix; the current open fork PR still needs rebase/proof or a target-repo replacement before this issue can close.
