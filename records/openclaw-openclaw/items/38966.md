## Summary

Keep open: current main still emits reset-related `sessions.changed` events only to session-event subscribers, while the TUI neither subscribes to that event family nor dispatches a session-reset handler. The linked open PR #40472 remains the implementation path, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue asks for the TUI to refresh its chat history, token counts, and stale run state when the current session is reset from another client.

## Best Possible Solution

Review and land or replace #40472 so the TUI subscribes to `sessions.changed`, handles current-session `new`/`reset` events, refreshes history/session metadata, clears stale token/run state, and ignores unrelated sessions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the current TUI session from another gateway client; current main can emit `sessions.changed`, but the TUI neither subscribes to nor handles that event family.

## Solution Assessment

Yes. Reusing the existing `sessions.changed` subscription path is the narrow maintainable fix; adding a second reset event family or a polling loop would be broader than needed.
