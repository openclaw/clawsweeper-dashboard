## Label Justifications

- `P2`: This is a high-confidence, bounded cross-client synchronization bug that causes misleading TUI state without disabling the core runtime.
- `impact:session-state`: The TUI remains associated with transcript, usage, and run projections from the session incarnation that another client reset.

## Summary

Current main still leaves the TUI disconnected from external session-reset lifecycle updates: the gateway emits `sessions.changed` only to explicit subscribers, while the TUI neither subscribes nor handles that event. Keep this issue open and paired with the active fork PR; the earlier OpenClaw PR closed unmerged and is not a safe superseding implementation.

## What This Changes

Refresh the active TUI transcript, usage display, and in-memory run state when another client resets the same gateway session.

## Best Possible Solution

Use the existing `sessions.changed` subscription contract: subscribe the gateway-backed TUI, match the canonical active session and agent scope, and handle `new`/`reset` as one atomic transition that reloads history/session metadata and clears transcript, token, streaming, optimistic-message, and run projections.

## Reproduction Assessment

Yes. On current main, an external reset emits `sessions.changed` only to subscribed clients, but the TUI neither subscribes nor handles that event, so its local view cannot refresh from that reset path.

## Solution Assessment

The event-driven approach is correct, but the best implementation should reuse the existing canonical `sessions.changed` contract rather than add a parallel reset-only event, with focused matching and stale-run cleanup tests.
