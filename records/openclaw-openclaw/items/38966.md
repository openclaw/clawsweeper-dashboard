## Summary

Keep open: current main still exposes a gateway session-change event, but the TUI neither subscribes to nor dispatches `sessions.changed`, so an external reset can still leave stale TUI history and run state visible. The open closing PR at https://github.com/openclaw/openclaw/pull/40472 remains the canonical implementation path and should be reviewed, repaired, or replaced before this issue closes.

## What This Changes

The issue asks for the TUI to refresh and clear stale session state when another client resets the active session.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that reloads the active session and clears active, pending, local, watchdog, session, and finalized run state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the active TUI session from another gateway client, and current main can emit subscription-gated `sessions.changed` while the TUI neither subscribes to nor dispatches that event family.

## Solution Assessment

Yes for the direction. Reusing the existing `sessions.subscribe` and `sessions.changed` contract is the narrow maintainable fix; creating a parallel event family or polling loop would be broader than needed.
