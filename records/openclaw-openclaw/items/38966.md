## Summary

Keep open: current main emits `sessions.changed` after `sessions.reset`, but the TUI neither subscribes to that event family nor dispatches it to a reset/history refresh handler. A related same-repo PR is closed unmerged and an open fork PR carries the fix idea, so this issue should remain open until that path is landed, replaced, or closed.

## What This Changes

The issue asks for the TUI to clear or reload the active session and reset displayed state when another client runs `/new` or `/reset` for the same session.

## Best Possible Solution

Land or replace the linked fix path with a current-main TUI `sessions.subscribe`/`sessions.changed` handler that refreshes only the active session, clears reset-sensitive run state, and includes focused TUI coverage plus real multi-client proof.

## Reproduction Assessment

Yes, by source inspection. Current main emits subscription-gated `sessions.changed` reset/new events, while the TUI neither subscribes to nor handles that event family, so an external reset cannot trigger `loadHistory()` or run-state cleanup.

## Solution Assessment

Yes for the direction. Reusing the existing `sessions.subscribe` and `sessions.changed` contract is the narrow maintainable fix; the open fork PR still needs upstream landing or replacement plus proof before this issue can close.
