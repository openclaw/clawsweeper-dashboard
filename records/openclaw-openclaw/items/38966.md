## Summary

Keep open: current main emits `sessions.changed` from `sessions.reset`, but the TUI still does not subscribe to session lifecycle changes or dispatch `sessions.changed`, so an external reset can leave stale TUI session state. The open closing PR at https://github.com/openclaw/openclaw/pull/40472 is the canonical implementation path and should be reviewed or replaced before this issue closes.

## What This Changes

The issue reports that the TUI keeps showing stale transcript, token, and run state when the active session is reset from another gateway client such as Telegram.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/40472 with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that refreshes the active session and clears stale run state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reset the active TUI session from another gateway client, and current main can emit subscription-gated `sessions.changed` while the TUI neither subscribes to nor dispatches that event family.

## Solution Assessment

Yes for the direction. Reusing the existing `sessions.subscribe` / `sessions.changed` lifecycle contract is the narrow maintainable fix; adding a parallel event family or polling loop would be broader than needed.
