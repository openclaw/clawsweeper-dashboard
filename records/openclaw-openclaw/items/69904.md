## Summary

Keep open: the stale-notice cleanup is not implemented on current main, and the PR branch still has a P1 path that can consume durable non-completion exec notices without delivering them. The external PR also only provides Vitest output, so it still needs real behavior proof before merge.

## What This Changes

The PR adds a system-event stale policy, filters stale transient async notices after later session activity, adjusts exec-event heartbeat prompting, and adds regression tests.

## Best Possible Solution

Keep the stale-policy seam, but rebase and narrow heartbeat skipping/consumption so only marked transient completion or success notices are dropped while durable exec started, denied, failed, timeout, cron, and routed events remain deliverable.

## Reproduction Assessment

Yes. Source inspection of the PR head gives a high-confidence path: enqueue an Exec denied or Exec started event, wake heartbeat with reason exec-event, and the PR null-prompt path can consume that queued notice without a reply turn.

## Solution Assessment

No. The stale-policy direction is maintainable, but the current PR skip condition is too broad and should be limited to stale-policy-marked transient completions after session advance.
