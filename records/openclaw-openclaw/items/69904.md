## Summary

Keep open. The PR is not obsolete on current main, but the head still has a blocking exec-event delivery bug: an exec-event wake with only non-completion notices returns a null prompt, and that path consumes the inspected queue entries without delivering them.

## What This Changes

The PR adds transient system-event stale-policy metadata, filters stale queued async notices after later session activity, adjusts heartbeat exec-event prompting, and adds regression coverage for stale async notices and mixed cron queues.

## Best Possible Solution

Revise the branch so stale filtering discards only entries explicitly marked transient after session advance. Heartbeat should skip an exec-event wake only when stale filtering leaves no valid pending event to deliver; `Exec started`, `Exec denied`, failures, timeouts, blocked events, and route-bearing durable notices should remain queued or be prompted.

## Reproduction Assessment

Yes. On PR head, enqueue an `Exec denied` or `Exec started` event and run `runHeartbeatOnce` with reason `exec-event`; it has no exec completion or cron event, returns `prompt: null`, and then consumes the inspected event queue.

## Solution Assessment

No. The stale-policy seam is a maintainable direction, but the current heartbeat guard is too broad; the safer fix is to make the skip depend on the stale discard result or otherwise preserve non-completion exec notices.
