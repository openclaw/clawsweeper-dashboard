## Summary

Keep open: the stale-policy direction is useful and not already on current main, but the PR head can still suppress durable non-completion exec notices and it lacks real after-fix behavior proof.

## What This Changes

The PR adds stale-policy metadata for transient system events, filters stale queued entries after session activity, changes heartbeat exec-event prompting, and adds regression tests.

## Best Possible Solution

Keep the explicit stale-policy seam, but ensure heartbeat only drops marked transient completion/success entries after session advance while durable exec started/denied/failed/timeout, cron, and route-context-bearing events remain queued or are delivered.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enqueue an Exec denied or Exec started event, wake heartbeat with reason exec-event, and the PR guard can return prompt:null with those inspected entries selected for consumption.

## Solution Assessment

No. The stale-policy mechanism is a maintainable direction, but the current heartbeat skip condition is too broad and should be narrowed to stale-policy-marked transient completions only.
