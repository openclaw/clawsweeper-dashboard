## Summary

Keep this PR open. Current main still lacks the stale transient-notice policy, so the branch is relevant, but the proposed heartbeat skip path still drops valid non-completion exec notices such as `Exec denied` when an `exec-event` wake has no completion marker left.

## What This Changes

The PR adds stale-policy metadata to transient async success notices, filters queued system events after later session activity, adjusts heartbeat exec-event prompt handling, and adds regression tests for stale completions and mixed cron queues.

## Best Possible Solution

Revise the branch so only explicitly marked transient success/completion notices are dropped after session advance, and make the heartbeat prompt/consume path skip only when no valid pending event remains. Durable denied, started, failed, timeout, blocked, cron, and route-context-bearing events should remain queued or be processed normally.

## Reproduction Assessment

Yes. Enqueue an `Exec denied` or `Exec started` system event and wake heartbeat with reason `exec-event`: the PR guard sees no exec completion and no cron event, returns `prompt: null`, and the null-prompt path consumes the inspected queue entry as `no-tasks-due`.

## Solution Assessment

No. The explicit stale policy is the right direction, but the current PR guard is too broad because it treats all non-completion exec-event wakes as stale/no-op instead of preserving valid durable notices.
