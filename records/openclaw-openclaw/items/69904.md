## Summary

This PR should stay open because current main does not already implement the stale transient notice filtering, but the PR head still has a blocking exec-event notice-loss path and lacks real behavior proof.

## What This Changes

The PR adds an in-memory system-event stale policy, marks transient async completion notices as drop-on-session-advance, filters stale entries before prompt construction, and adds focused heartbeat/system-event regression tests.

## Best Possible Solution

Keep the stale-policy direction, but rebase and narrow heartbeat skipping/consumption so only marked stale transient completion notices are dropped while durable started, denied, failed, timeout, cron, and routed events stay deliverable.

## Reproduction Assessment

Yes. Source inspection of the PR head gives a high-confidence path: enqueue an Exec denied or Exec started event, wake heartbeat with reason exec-event, and the null-prompt branch can consume that queued notice without a reply turn.

## Solution Assessment

No. The stale-policy seam is a maintainable direction, but the current exec-event skip condition is too broad and should be limited to stale-policy-marked transient completions after session advance.
