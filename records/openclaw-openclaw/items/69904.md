## Summary

Keep open. Current main still lacks the stale-policy filtering this PR proposes, so the contribution is not obsolete; however, the current PR head still has a blocking exec-event regression that can consume durable `Exec started` and `Exec denied` notices without delivering them.

## What This Changes

The PR adds stale-policy metadata for transient async completion notices, filters stale queued system events after later session activity, adjusts heartbeat exec-event prompting, and adds regression coverage for stale completions and mixed heartbeat queues.

## Best Possible Solution

Revise the branch so stale filtering drops only entries explicitly marked as transient completion/success notices after session advance. Heartbeat should skip an `exec-event` wake only when stale filtering leaves no valid pending event to deliver; durable denied, started, failed, timeout, blocked, cron, and route-context-bearing events should stay queued or be processed normally.

## Reproduction Assessment

Yes. Enqueue an `Exec denied` or `Exec started` system event and wake heartbeat with reason `exec-event`; the PR guard sees no exec completion and no cron event, returns `prompt: null`, and the null-prompt path consumes the inspected queue entry as `no-tasks-due`.

## Solution Assessment

No. The explicit stale-policy seam is a maintainable direction, but the current heartbeat guard is too broad because it treats every non-completion exec-event wake as stale/no-op rather than preserving valid durable notices.
