## Summary

Keep open. The stale-policy direction is useful and the latest PR revision preserves tagged cron reminders, but the branch still short-circuits every `exec-event` wake that lacks a completion marker, which can consume valid `Exec started` and `Exec denied` notices before they reach a reply turn.

## What This Changes

The PR adds stale-policy metadata and queue filtering so transient async completion notices are dropped after newer session activity while durable notices remain deliverable.

## Best Possible Solution

Revise the branch so stale filtering only drops entries explicitly marked as transient completion or success notices after session advance. Heartbeat should skip an `exec-event` wake only when no valid pending event remains, while durable denied, started, failed, timeout, blocked, route-context-bearing, and cron events stay queued or get processed normally.

## Reproduction Assessment

Yes. On this branch, enqueue an `Exec denied` or `Exec started` system event and wake heartbeat with reason `exec-event`; the new guard sees no exec completion and no cron event, returns `prompt: null`, and the null-prompt path consumes the inspected queue entry as `no-tasks-due`.

## Solution Assessment

No. The explicit stale-policy seam is a maintainable approach, but the current heartbeat skip is broader than the stated goal because it treats non-completion exec notices as stale/no-op work.
