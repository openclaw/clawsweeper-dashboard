## Summary

Keep this PR open, but it should not merge as-is: the stale-policy direction is useful, while the current branch can suppress durable non-completion exec notices and only provides test output rather than real behavior proof.

## What This Changes

The PR adds stale-policy metadata for transient system events, filters stale queued notices after session activity, adjusts heartbeat exec-event handling, and adds regression coverage.

## Best Possible Solution

Keep the explicit stale-policy seam, but revise heartbeat so only marked transient success/completion entries are dropped after session advance while durable denied, started, failed, timeout, blocked, cron, and route-context-bearing events stay queued or are processed normally.

## Reproduction Assessment

Yes, source inspection gives a high-confidence PR-branch reproduction: enqueue an Exec denied or Exec started event and wake heartbeat with reason exec-event. The PR guard sees no exec completion and no cron event, returns prompt:null, and the null-prompt path can consume the inspected entry without delivery.

## Solution Assessment

No. The stale-policy mechanism is a maintainable direction, but the current heartbeat skip condition is too broad; it should skip only when stale filtering leaves no valid pending event to deliver.
