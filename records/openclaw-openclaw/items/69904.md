## Summary

Keep open: the stale transient completion cleanup is still a useful bug-fix direction, but this PR is not merge-ready because the submitted head can silently consume non-completion exec-event notices, is conflicting with current main, and only provides unit-test validation rather than real behavior proof.

## What This Changes

The PR adds a transient system-event stale policy so late async success/completion notices are dropped after newer session activity while durable failure, timeout, and blocked notices remain queued.

## Best Possible Solution

Rebase the idea onto current main's selected-consumption design: drop only explicitly stale transient success/completion notices after session advance, preserve exec started, denied, failed, timeout, blocked, and cron notices, and add real transport proof.

## Reproduction Assessment

## Solution Assessment

No as submitted. The stale transient completion filter is plausible, but the best fix must be rebased onto current main's selected-consumption path and must preserve non-stale exec, cron, blocked, error, and timeout notices.
