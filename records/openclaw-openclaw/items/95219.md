## Summary

Keep open: current main and v2026.6.8 still rebuild exact older-first aggregate tool-result reductions for request-local prompt projection, so the same historical tool result can change as newer tail content grows. The open candidate fix at https://github.com/openclaw/openclaw/pull/93186 targets this behavior, but it is not merged and still needs maintainer acceptance of the cache-stability tradeoff.

## What This Changes

The issue asks OpenClaw to keep historical tool-result projections byte-stable across adjacent model requests so prompt-cache prefixes remain reusable.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/93186 or an equivalent stable live-projection fix, with focused regression coverage that compares historical tool-result bytes across adjacent prompt builds.

## Reproduction Assessment

Yes at source level: current main rebuilds request-local tool-result projections from the evolving aggregate total and older-first reductions, so a focused test can compare the same historical tool result before and after appending tail results. I did not run a fresh live provider cache trace in this read-only review.

## Solution Assessment

Yes, conditionally: the right boundary is the live prompt projection and matching estimate path, not persisted transcript recovery. The open PR is a narrow candidate, but maintainers still need to accept the cache-stability versus retained-context policy before it becomes the solution.
