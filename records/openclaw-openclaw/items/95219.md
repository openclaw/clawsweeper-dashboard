## Summary

Keep open: current main and the latest release still use exact older-first aggregate tool-result reduction during live prompt projection, so the reported prompt-cache prefix instability is not fixed on main. Open candidate PRs now own the fix path, especially https://github.com/openclaw/openclaw/pull/95305, so this issue should stay open until one candidate lands.

## What This Changes

The issue reports that long channel sessions can re-project historical tool-result messages differently across adjacent prompt builds, reducing prompt-cache prefix reuse.

## Best Possible Solution

Land one narrow stable live prompt-projection fix, preferably the current-main-safe candidate at https://github.com/openclaw/openclaw/pull/95305 if maintainers accept its context tradeoff, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level: current main rebuilds aggregate tool-result projections from the evolving message set and spends exact reductions on older entries first, so appending newer tail output can change old model-facing bytes. I did not run a live provider cache trace in this read-only pass.

## Solution Assessment

Yes, conditionally: the live prompt-projection boundary is the right layer, but maintainers still need to choose the exact truncation policy and land one of the open candidate PRs. Spawning another parallel fix is not the best path while proof-positive candidates already exist.
