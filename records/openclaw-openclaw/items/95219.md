## Summary

Keep open: current main and latest v2026.6.9 still use exact older-first aggregate tool-result reduction during prompt projection, so the reported cache-prefix instability is not implemented away yet. The issue is already paired with open candidate fix PRs, especially https://github.com/openclaw/openclaw/pull/95305, so cleanup should wait for the chosen fix to merge.

## What This Changes

The issue reports that long channel sessions can re-project historical tool-result content differently across adjacent model requests, reducing prompt-cache prefix reuse.

## Best Possible Solution

Review and land the chosen stable live prompt-projection fix, likely https://github.com/openclaw/openclaw/pull/95305 if maintainers accept its bounded context tradeoff, then close this issue after the fix reaches main.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main rebuilds aggregate tool-result projections from the current message set and spends exact reductions on older entries first, so appending newer tail output can change old model-facing bytes.

## Solution Assessment

Is this the best way to solve the issue? Yes, the live prompt-projection boundary is the narrowest maintainable layer for the cache-prefix bug, but the exact truncation policy needs maintainer acceptance before closure.
