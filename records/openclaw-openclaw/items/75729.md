## Summary

Keep open. This protected maintainer PR has a remaining P1 correctness issue: the follow-up fixed the unknown-runtime and missing-service-definition cases, but the branch still lets a service-marked updater proceed after stopping the running Gateway service, which can kill the updater before package replacement/restart completes under the current service lifecycle contracts.

## What This Changes

The PR changes package-manager `openclaw update` to inspect managed Gateway runtime state before blocking inherited service-marker environments, strip Gateway service markers from the post-core update child process, add CLI tests, and add a changelog entry.

## Best Possible Solution

Keep the useful tri-state runtime inspection, conservative unknown-runtime fallback, marker stripping, and tests, but fail closed for service-marked package updates while the managed Gateway is running unless the update is handed off to a detached process outside the service control group. The changelog should describe that final behavior after the lifecycle decision is made.

## Reproduction Assessment

Yes. A high-confidence reproduction path is the linked #75691 shape: run a package-manager `openclaw update` from a Gateway-owned exec child with `OPENCLAW_SERVICE_MARKER=openclaw` and a running systemd-managed Gateway; current source shows that `systemctl stop` plus `KillMode=control-group` can terminate that child before update completion.

## Solution Assessment

No. The PR is close on runtime inspection and marker cleanup, but allowing `stopState.stopped` from a service-marked caller is not the narrowest safe fix because it assumes the updater survives stopping its own service tree. A fail-closed guard or an explicit detached handoff is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Block service-owned updates before stopping the Gateway:** `src/cli/update-cli/update-command.ts:290-291`
  - body: Returning `false` for `stopState.stopped` lets an updater launched from the Gateway service tree proceed after it calls `service.stop()`. Under the rendered systemd unit, children remain in the service cgroup, so stopping the service can kill the updater before package replacement/restart completes. Block service-marked package updates while the managed Gateway is running, or hand off to a detached process outside the service tree.
