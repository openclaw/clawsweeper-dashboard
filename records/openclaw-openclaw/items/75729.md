## Summary

Keep open. The latest PR head contains useful runtime-state fallback and marker-stripping work, but it still has a P1 service-lifecycle regression: a Gateway service-marked updater can proceed after stopping the running service that owns its process tree.

## What This Changes

The PR changes package-manager `openclaw update` to inspect managed Gateway runtime state before blocking inherited service-marker environments, strip service markers from the post-core update child, add CLI tests, and add a changelog entry.

## Best Possible Solution

Keep the tri-state runtime inspection, conservative unknown-runtime fallback, marker stripping, and tests, but either fail closed before stopping a service-marked running Gateway or hand off to an updater outside the service control group.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a package-manager `openclaw update` launched from a Gateway-owned exec child with Gateway service markers while the systemd-managed Gateway is running; the PR path stops that service before package replacement, while current systemd and child-spawn code keep that child in the stopped service lifecycle.

## Solution Assessment

No. The PR is close on runtime inspection and marker cleanup, but allowing `stopState.stopped` from a service-marked caller is not the narrowest safe fix because it assumes the updater survives stopping its own service tree.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Block service-owned updates before stopping the Gateway:** `src/cli/update-cli/update-command.ts:290-291`
  - body: Returning `false` for `stopState.stopped` lets a service-marked updater proceed after it stops the service that owns its process tree. Systemd units use `KillMode=control-group` and service-mode child spawns stay attached, so the updater can be killed before package replacement/restart completes; keep the guard fail-closed or hand off to a detached updater first.
