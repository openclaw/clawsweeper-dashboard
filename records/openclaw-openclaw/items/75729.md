## Summary

Keep open. The latest head fixes the earlier fail-open and changelog review points, but the proposed flow still allows a service-marked updater to proceed after calling `service.stop`; current service/runtime code shows systemd children stay in the service control group, so a gateway-owned updater can be killed before package replacement. The protected `maintainer` label and MEMBER author also rule out automatic cleanup closing.

## What This Changes

The PR changes `openclaw update` package-manager updates to inspect managed Gateway state before blocking inherited service-marker environments, strip service markers from the post-core update child process, and add CLI tests plus a changelog entry.

## Best Possible Solution

Preserve the useful tri-state runtime inspection, changelog, tests, and post-core marker stripping, but do not let a service-owned updater rely on surviving a managed-service stop. The maintainable fix is to fail closed for running service-marker callers or move the update/restart into a detached external handoff that is outside the Gateway service control group.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the linked #75691 scenario: run `openclaw update --tag <version> --yes` from a Gateway-owned exec child under a managed systemd service; the service env marker is inherited, and the current systemd unit keeps children in the stopped control group.

## Solution Assessment

No. The PR is close, and the fail-closed unknown-runtime follow-up is good, but allowing `stopState.stopped` from a service-marked caller is not the narrowest safe fix because the caller may be killed by the service stop before the package update can complete.
