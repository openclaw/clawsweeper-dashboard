## Summary

Keep open: current main only supports in-process reload for plugin config/policy changes, while plugin source/Jiti cache invalidation still has no public reload or cache-clear workflow and still requires Gateway restart.

## What This Changes

The issue requests a supported plugin development reload or cache-clear workflow so TypeScript plugin edits do not require a full OpenClaw process/container restart.

## Best Possible Solution

Design a supported plugin development reload lifecycle that invalidates loader/Jiti/module state, safely swaps active plugin registrations, disposes stale resources, exposes an additive CLI/RPC UX such as `openclaw plugins reload <id>` or `openclaw plugins clear-cache`, and updates the public plugin docs.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature gap. Current docs, CLI registration, reload-plan tests, and loader code give a high-confidence source-inspection path showing source-changing plugin work remains restart-backed and no public cache-clear/reload command exists.

## Solution Assessment

Unclear until maintainers choose the lifecycle design. A command that only deletes a guessed filesystem cache path would be too narrow; the maintainable fix needs coordinated loader/Jiti invalidation, active registry replacement, resource disposal, and Gateway/CLI UX.
