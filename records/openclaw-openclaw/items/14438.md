## Summary

Keep open: current main and the latest release still lack a supported TypeScript plugin source/Jiti cache reload or clear-cache workflow, and the live issue discussion already identifies this as Gateway/plugin-loader lifecycle design rather than cleanup-close material.

## What This Changes

The issue requests a plugin development workflow such as `openclaw plugins reload <id>`, a watcher-backed source reload, or a cache-clear command for Jiti-compiled TypeScript plugins.

## Best Possible Solution

Define a supported Gateway/API-backed plugin development reload contract that invalidates OpenClaw loader/module state and safely swaps plugin registrations, then expose it through plugin CLI UX or document restart-only behavior with precise cache guidance if live source reload remains unsupported.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature request. Source, docs, tests, latest-release inspection, upstream Jiti source, and live issue context give high-confidence evidence that plugin source/code changes still lack a reload or clear-cache workflow.

## Solution Assessment

Unclear as proposed. A bare cache-delete command is not enough unless paired with safe Gateway runtime replacement and resource cleanup; the maintainable path is a designed reload contract or explicit restart-only documentation.
