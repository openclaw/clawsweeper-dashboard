## Summary

Keep this issue open: release v2026.6.5 added config-driven plugin registry replacement, but current main still documents source edits as restart-only, exposes no reload/cache-clear CLI, and retains loaded module exports in process caches.

## What This Changes

Request a supported plugin reload or cache-clear workflow so TypeScript plugin changes can take effect without restarting the OpenClaw process or container.

## Best Possible Solution

Extend the existing Gateway plugin replacement lifecycle with an explicit development reload API and CLI that invalidates source-module caches and safely replaces all plugin-owned registrations; keep install and load-path changes restart-backed unless maintainers separately approve their upgrade semantics.

## Reproduction Assessment

Yes from current source and documentation: plugin source changes still require Gateway restart, no supported reload/cache-clear command exists, and module exports are cached in process. The June 2 report provides consistent real-world install-activation evidence, though this review did not execute a live container reproduction.

## Solution Assessment

No, deleting environment-specific `/tmp/jiti` files is not the best solution. Cache invalidation, disposal, registry replacement, and failure handling should be owned by the existing Gateway plugin lifecycle behind a supported development command or watcher.
