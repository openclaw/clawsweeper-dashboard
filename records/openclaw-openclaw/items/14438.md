## Summary

Keep open: current main partially hot-reloads plugin enable/config policy changes, but it still lacks a public `plugins reload` or cache-clear workflow and keeps plugin source/install/load-path changes restart-backed; the issue discussion also has explicit maintainer guidance to leave the lifecycle design open.

## What This Changes

The issue requests a supported plugin development workflow to reload TypeScript plugins or clear Jiti/plugin loader cache without restarting the OpenClaw container or Gateway.

## Best Possible Solution

Design and document a supported plugin development reload lifecycle that invalidates loader/Jiti/module state, safely swaps active registrations, disposes stale resources, and exposes an additive CLI/RPC workflow such as `openclaw plugins reload <id>` or `openclaw plugins clear-cache`.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature gap. Source, docs, tests, and issue discussion give a high-confidence inspection path showing plugin source/load-path changes remain restart-backed and no public reload/cache-clear command exists.

## Solution Assessment

Unclear until maintainers choose lifecycle semantics. Deleting a guessed Jiti cache path alone is not the best fix; the maintainable path needs coordinated loader invalidation, registry replacement, resource disposal, Gateway/CLI UX, docs, and security review.

- Live plugin reload needs explicit disposal and replacement semantics for hooks, tools, services, providers, HTTP routes, commands, background resources, channel runtimes, and Jiti/module state.
- A narrow cache-deletion command could reduce confusion but would still leave stale in-process registrations unless paired with a supported Gateway/runtime reload path.
