## Summary

Keep open: current main still has no supported plugin source hot-reload, cache-clear, or live plugin replacement command; partial hot-reload work covers plugin entry state while source/install changes remain restart-backed and docs still tell users to restart the active Gateway.

## What This Changes

Request a supported plugin development reload workflow that invalidates Jiti/module/plugin-loader state and refreshes plugin registrations without restarting the whole container.

## Best Possible Solution

Define a Gateway-owned development reload lifecycle that safely disposes old plugin runtime state, invalidates loader/module caches, rebuilds the registry, and exposes a documented CLI/API command only after upgrade and live-plugin proof.

## Reproduction Assessment

Yes, at source level. Current main lacks a public plugin reload/cache-clear path, classifies plugin source/install changes as restart-backed, and current docs still instruct users to restart the live Gateway after source changes.

## Solution Assessment

No, deleting Jiti cache files directly is not the best permanent solution. The safer design is a lifecycle-owned Gateway reload operation with explicit teardown, cache invalidation, registry replacement, and CLI/API semantics.
