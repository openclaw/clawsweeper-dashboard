## Summary

Keep open: current main still deliberately restarts for plugin source/install/load-path changes, has no public plugin reload or cache-clear command, and the latest maintainer discussion says the remaining Gateway/plugin-loader lifecycle design should stay open.

## What This Changes

The issue requests a supported plugin development workflow to reload TypeScript plugin code or clear Jiti/runtime caches without restarting the Gateway/container.

## Best Possible Solution

Design and document a supported plugin development reload lifecycle that invalidates loader/Jiti/module state, safely swaps active registrations, disposes stale resources, and exposes an additive CLI/RPC workflow such as `openclaw plugins reload <id>` or `openclaw plugins clear-cache`.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature gap. Source, docs, and tests give a high-confidence inspection path showing plugin source/load-path changes remain restart-backed and no public reload/cache-clear command exists.

## Solution Assessment

Unclear until maintainers choose the lifecycle semantics. A command that only deletes a guessed filesystem cache path would be too narrow; the maintainable solution needs coordinated loader/Jiti invalidation, registry replacement, disposal, Gateway/CLI UX, docs, and security review.
