## Summary

Keep open: current main and v2026.6.10 still lack a supported plugin source/Jiti reload or cache-clear command, and the remaining work crosses Gateway/plugin-loader lifecycle design rather than conservative cleanup closure.

## What This Changes

Request a supported plugin development hot-reload or cache-clear workflow so TypeScript plugin edits can take effect without manually deleting Jiti cache files and restarting the container.

## Best Possible Solution

Design a Gateway-owned plugin development reload lifecycle that safely disposes old runtime state, invalidates loader/Jiti/module caches, rebuilds and swaps the active registry, and exposes a documented CLI/API command; if live reload is rejected, document restart-only behavior and supported cache diagnostics precisely.

## Reproduction Assessment

Yes, at source level. Current main has no public plugin reload/cache-clear command, restart-backs plugin source/install paths, and documents Gateway restart after plugin code changes; I did not run a live container edit cycle in this read-only review.

## Solution Assessment

No, manually deleting dependency cache files is only a workaround. The maintainable solution is a lifecycle-owned Gateway reload operation with explicit teardown, cache invalidation, registry replacement, and CLI/API semantics.
