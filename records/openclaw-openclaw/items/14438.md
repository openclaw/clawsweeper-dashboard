## Summary

Keep open: current main only supports config-driven plugin reload paths and still lacks a supported plugin source/Jiti cache reload or cache-clear workflow.

## What This Changes

The issue requests a plugin development reload or cache-clear path so TypeScript plugin edits can take effect without manually clearing Jiti cache and restarting the Gateway/container.

## Best Possible Solution

Design a supported plugin development reload workflow that invalidates plugin module/Jiti cache state, disposes plugin-owned runtime resources, swaps registrations through the Gateway lifecycle, and exposes a documented CLI/RPC path; until then keep plugin code changes restart-backed in docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main has no public plugin reload/cache-clear CLI, docs still say plugin code lifecycle changes require Gateway restart, and tests keep plugin install/source changes restart-backed.

## Solution Assessment

Unclear as a final implementation path. A cache-clear command alone would not safely replace active plugin registrations, so the best fix needs a Gateway/plugin-loader lifecycle design rather than a narrow one-file command.
