## Summary

Keep open: current main partially supports plugin config hot reload, but it still lacks the requested developer workflow for reloading TypeScript plugin source or Jiti/module cache state without a process/container restart, and prior maintainer discussion leaves this as a product/lifecycle design item.

## What This Changes

Request a supported plugin development reload path, such as `openclaw plugins reload <id>`, cache clearing, or a source watcher, so TypeScript plugin edits take effect without restarting the container.

## Best Possible Solution

Design a supported developer reload workflow that distinguishes config hot reload from plugin source changes, then expose a Gateway-backed reload or cache-clear command, or document restart-only behavior if live source reload remains unsupported.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source and docs inspection give high confidence that current main lacks a supported plugin source/Jiti reload command or watcher while plugin install/source changes remain restart-backed.

## Solution Assessment

Unclear until maintainers choose the product boundary. A targeted Gateway-backed CLI reload or cache-clear command looks safer as a first surface than automatic source watching because it can make cache invalidation and runtime disposal explicit.
