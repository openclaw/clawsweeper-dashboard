## Summary

Keep open: current main only partially hot-reloads plugin management/config changes; it still has no public plugin reload or cache-clear workflow and still treats plugin source/load-path/install changes as restart-backed. Maintainer discussion also explicitly says this should stay open for plugin loader and Gateway lifecycle design.

## What This Changes

Request a supported plugin development reload or cache-clear workflow so TypeScript plugin code changes can apply without restarting the OpenClaw process or container.

## Best Possible Solution

Design and document a supported plugin development reload lifecycle that invalidates loader/Jiti/module state, safely swaps active registrations, disposes stale resources, and exposes an additive CLI/RPC workflow such as `openclaw plugins reload <id>` or `openclaw plugins clear-cache`.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature gap. Source, docs, tests, and issue discussion give a high-confidence inspection path showing plugin source/load-path changes remain restart-backed and no public reload/cache-clear command exists.

## Solution Assessment

Unclear until maintainers choose lifecycle semantics. Deleting a guessed Jiti cache path alone is not the best fix; the maintainable path needs coordinated loader invalidation, registry replacement, resource disposal, Gateway/CLI UX, docs, and security review.

## Risks / Open Questions

- Live plugin reload needs explicit cleanup and replacement semantics for hooks, tools, services, providers, HTTP routes, commands, background resources, channel runtimes, and module/Jiti state.
