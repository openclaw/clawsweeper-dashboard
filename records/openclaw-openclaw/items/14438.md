## Summary

Keep open: current main still lacks a public plugin reload or cache-clear workflow, keeps plugin source/install/load-path changes restart-backed, and maintainer discussion explicitly leaves the Gateway/plugin-loader lifecycle design open.

## What This Changes

The issue requests a supported plugin development reload path, such as a CLI reload/cache-clear command or watcher-backed reload, so TypeScript plugin edits can take effect without container restart and manual Jiti cache deletion.

## Best Possible Solution

Design and document a supported plugin development reload lifecycle that invalidates loader/Jiti/module state, safely swaps active registrations, disposes stale resources, and exposes an additive CLI/RPC workflow such as openclaw plugins reload <id> or openclaw plugins clear-cache.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature gap. Source, docs, tests, and issue discussion give a high-confidence inspection path showing plugin source/load-path changes remain restart-backed and no public reload/cache-clear command exists.

## Solution Assessment

Unclear until maintainers choose lifecycle semantics. Deleting a guessed Jiti cache path alone is not the best fix; the maintainable path needs coordinated loader invalidation, registry replacement, resource disposal, Gateway/CLI UX, docs, and security review.
