## Summary

Keep open: current main and v2026.6.10 still lack a supported plugin source/Jiti reload or cache-clear command, and the remaining work needs plugin-loader/Gateway lifecycle design rather than cleanup closure.

## What This Changes

The issue requests a supported plugin development reload or cache-clear workflow so TypeScript plugin edits do not require manually clearing Jiti cache and restarting OpenClaw.

## Best Possible Solution

Design a Gateway-owned plugin development reload lifecycle that safely disposes old runtime state, invalidates loader/Jiti/module caches, rebuilds and swaps the active registry, and exposes a documented CLI/API command; if live reload is rejected, document restart-only behavior and supported cache diagnostics precisely.

## Reproduction Assessment

Yes, at source level. Current main has no public plugin reload/cache-clear command, restart-backs plugin source/install paths, and documents Gateway restart after plugin code or source changes; I did not run a live container edit cycle in this read-only review.

## Solution Assessment

No, manually deleting dependency cache files is not the best permanent solution. The maintainable fix is a lifecycle-owned Gateway reload operation with explicit teardown, cache invalidation, registry replacement, and CLI/API semantics.
