## Summary

Keep open: current main and the latest shipped tag still keep plugin source/install/load-path changes restart-backed, and there is no public plugin reload or cache-clear command. The linked cache-dir PR and watcher debounce issue are related but do not replace the requested Gateway/plugin-loader lifecycle design.

## What This Changes

The issue asks for a supported plugin-development reload/cache invalidation workflow, such as `openclaw plugins reload`, `openclaw plugins clear-cache`, or source watching, so TypeScript plugin edits do not require a container restart.

## Best Possible Solution

Design a supported plugin development reload lifecycle that safely invalidates loader/Jiti/module state, swaps active registrations, disposes stale resources, and exposes an additive CLI/RPC workflow such as `openclaw plugins reload <id>` or `openclaw plugins clear-cache`.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source, docs, tests, release-tag inspection, and issue discussion give a high-confidence inspection path showing plugin source/load/install changes remain restart-backed and no public reload or cache-clear command exists.

## Solution Assessment

Unclear until maintainers choose lifecycle semantics. A standalone cache-clear command is not the best complete fix because it would not safely replace active registrations or dispose plugin-owned resources.
