## Summary

Keep open: current main and the latest release still lack a supported plugin source/Jiti reload or cache-clear command, and source/install changes remain restart-backed even though some plugin entry config changes can hot-reload.

## What This Changes

The issue requests a supported plugin reload or cache-clear workflow so TypeScript plugin code changes can take effect without restarting OpenClaw.

## Best Possible Solution

Design a Gateway-owned plugin development reload lifecycle that safely disposes old runtime state, invalidates loader/module caches, rebuilds and swaps the active registry, and exposes a documented CLI/API command only after maintainer approval and live proof.

## Reproduction Assessment

Yes, at source level. Current main lacks a public plugin reload/cache-clear path, classifies plugin source/install changes as restart-backed, and documents Gateway restart after source changes; I did not run a live container plugin edit cycle in this read-only review.

## Solution Assessment

No, deleting dependency cache files directly is not the best permanent solution. The maintainable path is a lifecycle-owned Gateway reload operation with explicit teardown, cache invalidation, registry replacement, and CLI/API semantics.
