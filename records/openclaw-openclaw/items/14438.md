## Summary

Keep open. Current main still documents and implements restart-only behavior for native plugin runtime/config changes, has no plugin reload or cache-clear CLI, and the latest maintainer discussion explicitly kept this as plugin-loader and Gateway lifecycle design work.

## What This Changes

The issue requests a supported development workflow for reloading TypeScript plugin code or clearing plugin/Jiti caches without restarting the OpenClaw process or container.

## Best Possible Solution

Define a supported plugin development reload lifecycle in the loader and Gateway: invalidate Jiti/module/runtime caches, safely replace active plugin registrations, dispose stale resources, expose an additive CLI/RPC UX such as `openclaw plugins reload <id>` or `openclaw plugins clear-cache`, and update public plugin docs. If live runtime reload remains unsupported, keep restart-only behavior documented with accurate cache-location guidance.

## Reproduction Assessment

Not applicable as a normal bug reproduction, but the feature gap is high-confidence. Current docs, CLI registration, Gateway reload rules, tests, and loader code all show restart-only behavior for native plugin runtime/config changes and no supported cache-clear path.

## Solution Assessment

Unclear until maintainers choose the lifecycle design. A command that only deletes a guessed filesystem cache path would be too narrow; the maintainable solution needs loader/Jiti cache invalidation plus safe active registry replacement and Gateway/CLI UX.
