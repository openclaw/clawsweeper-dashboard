## Summary

Current main still lacks a supported runtime plugin reload. The newer `reloadDeferredGatewayPlugins` helper only completes deferred loading during startup, plugin install/load changes remain restart-classified, and fresh user evidence shows plugin activation still waits for a Gateway restart; this needs maintainer lifecycle design.

## What This Changes

Add a supported plugin development workflow to reload changed or newly installed plugin code without restarting the OpenClaw process or container.

## Best Possible Solution

Define a Gateway-owned development reload lifecycle that disposes the old plugin runtime, invalidates loader and module state, rebuilds the registry, and atomically activates the replacement, then expose it through a CLI/API command with live installation and source-edit proof.

## Reproduction Assessment

Yes at source level: current main has no public plugin reload path, treats plugin install/load changes as restart-required, and the June 2, 2026 report confirms the activation gap in a real setup. This read-only review did not perform a live plugin installation.

## Solution Assessment

No, deleting a Jiti cache file is not the best solution. The maintainable path is a lifecycle-owned Gateway operation that safely tears down and replaces all plugin registrations while controlling dependency cache invalidation internally.
