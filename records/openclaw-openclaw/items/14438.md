## Summary

Keep open: current main still lacks a supported plugin source hot-reload or cache-clear command, and plugin source/install path changes remain restart-backed even though plugin entry config can now hot-reload.

## What This Changes

Request a supported plugin development reload or cache-clear workflow that invalidates Jiti/plugin-loader state without restarting the OpenClaw container.

## Best Possible Solution

Design a lifecycle-owned Gateway/API reload path plus CLI UX that safely invalidates loader/Jiti/module state and replaces plugin registrations, or explicitly document restart-only plugin source reload as the supported contract.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main has no public plugin reload/cache-clear CLI and keeps plugin source/install path changes restart-backed. I did not run a live Docker repro in this read-only review.

## Solution Assessment

Unclear as a final design: a cache-clear command alone would not be the best fix because stale runtime registrations and plugin-owned resources also need safe replacement semantics. The better path is an explicit Gateway-owned reload lifecycle or documented restart-only contract.
