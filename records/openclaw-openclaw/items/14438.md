## Summary

Keep open: current main has config/policy plugin reloads, but source-changing plugin work still requires Gateway restart and there is no public plugin reload or cache-clear command; the maintainer discussion also explicitly kept this for lifecycle design.

## What This Changes

The issue requests a supported plugin hot-reload or cache-clear workflow so TypeScript plugin changes do not require container restart and manual Jiti cache cleanup.

## Best Possible Solution

Design a supported plugin development reload lifecycle that invalidates loader/Jiti/module state, safely swaps active plugin registrations, disposes stale resources, exposes an additive CLI/RPC UX such as `openclaw plugins reload <id>` or `openclaw plugins clear-cache`, and updates the public plugin docs.

## Reproduction Assessment

Not applicable as a normal bug reproduction, but the feature gap is high-confidence. Current docs, CLI registration, reload-plan tests, and loader code show source-changing plugin operations still require restart and no public cache-clear/reload command exists.

## Solution Assessment

Unclear until maintainers choose the lifecycle design. A command that only deletes a guessed filesystem cache path would be too narrow; the maintainable solution needs loader/Jiti invalidation plus safe active registry replacement and Gateway/CLI UX.
