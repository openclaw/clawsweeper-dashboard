## Summary

Current main supports hot reload for selected plugin management/config changes but deliberately keeps plugin source and install-path changes restart-backed. The requested source/Jiti reload workflow therefore remains necessary and needs an explicit lifecycle design rather than cleanup closure.

## What This Changes

Add a supported command or watcher that reloads changed plugin source and invalidates relevant loader/Jiti state without requiring a full process or container restart.

## Best Possible Solution

Implement a Gateway-owned scoped reload transaction that disposes every affected plugin resource, invalidates module and loader state, reconstructs registries, and returns an explicit restart-required result for plugin types that cannot be safely replaced live.

## Reproduction Assessment

Yes, at source level. Current reload rules, regression tests, and documentation explicitly keep source-changing plugin updates restart-backed, and the June 2026 Windows report supplies an observed plugin-not-active-until-restart case.

## Solution Assessment

No, direct Jiti cache deletion alone is not the best solution because it does not safely replace active plugin registrations and resources. The narrow maintainable direction is a lifecycle-owned Gateway reload operation with an explicit restart fallback.
