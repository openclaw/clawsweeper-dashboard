## Summary

Keep open: current main still deliberately keeps plugin source/install/load-path changes restart-backed, exposes no public plugin reload or cache-clear command, and the maintainer discussion says the remaining Gateway/plugin-loader lifecycle design should stay open. The related merged work at https://github.com/openclaw/openclaw/pull/75976 is useful but partial.

## What This Changes

The issue requests a supported plugin development reload or cache-clear workflow so TypeScript plugin changes do not require manually clearing Jiti cache and restarting the container.

## Best Possible Solution

Design and document a supported plugin development reload lifecycle that invalidates loader/Jiti/module state, safely swaps active registrations, disposes stale resources, and exposes an additive CLI/RPC workflow such as `openclaw plugins reload <id>` or `openclaw plugins clear-cache`.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature gap. Source, docs, tests, and issue discussion give a high-confidence inspection path showing plugin source/load-path changes remain restart-backed and no public reload/cache-clear command exists.

## Solution Assessment

Unclear until maintainers choose the lifecycle semantics. A command that only deletes a guessed filesystem cache path would be too narrow; the maintainable solution needs coordinated loader/Jiti invalidation, registry replacement, disposal, Gateway/CLI UX, docs, and security review.
