## Summary

Keep open: current main partially supports config/policy-driven plugin reload, but plugin source/Jiti/module cache reload and a public plugin reload or cache-clear workflow remain absent. The issue discussion also explicitly leaves the remaining work for Gateway/plugin-loader lifecycle design.

## What This Changes

Request a supported plugin development workflow to invalidate Jiti/module/plugin loader cache and reload plugin runtime registrations without restarting the Gateway or container.

## Best Possible Solution

Design and document a supported plugin development reload lifecycle that safely invalidates loader/Jiti/module state, swaps active registrations, disposes stale resources, and exposes an additive CLI/RPC workflow such as `openclaw plugins reload <id>` or `openclaw plugins clear-cache`.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source, docs, tests, release docs, and issue discussion give a high-confidence inspection path showing plugin source/load/install changes remain restart-backed and no public reload or cache-clear command exists.

## Solution Assessment

Unclear until maintainers choose lifecycle semantics. A standalone cache-clear command is not the best complete fix because it would not safely replace active registrations or dispose plugin-owned resources.
