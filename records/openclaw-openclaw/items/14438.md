## Summary

Keep open: current main has config-only plugin reload support, but source-changing plugin development still routes to restart behavior and there is no public plugin reload or cache-clear command; the maintainer discussion also explicitly kept this for lifecycle design.

## What This Changes

Add a supported plugin development workflow so TypeScript plugin code changes can invalidate loader/Jiti/module caches and take effect without restarting the container.

## Best Possible Solution

Design a supported plugin development reload lifecycle that invalidates loader/Jiti/module state, safely swaps active plugin registrations, disposes stale resources, exposes an additive CLI/RPC UX, and updates public plugin docs.

## Reproduction Assessment

Not applicable as a normal bug reproduction. The feature gap is high-confidence from current docs, CLI registration, reload-plan tests, and loader code showing source-changing plugin operations still require restart and no public cache-clear/reload command exists.

## Solution Assessment

Unclear until maintainers choose the lifecycle design. A command that only deletes a guessed filesystem cache path would be too narrow; the maintainable solution needs loader/Jiti invalidation plus safe active registry replacement and Gateway/CLI UX.
