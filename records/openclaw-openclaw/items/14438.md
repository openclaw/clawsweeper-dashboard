## Summary

Keep open: current main and the latest release still keep plugin source/install changes restart-backed, the plugin CLI has no reload or cache-clear command, and the docs still tell users to restart the active Gateway after source changes. The merged management hot-reload work is useful but only partial overlap, not a fix for the requested source/Jiti reload workflow.

## What This Changes

The issue requests a supported plugin development reload or cache-clear workflow so TypeScript plugin edits can take effect without restarting the OpenClaw process or container.

## Best Possible Solution

Define a Gateway-owned development reload lifecycle that safely disposes old plugin runtime state, invalidates loader/module caches, rebuilds the registry, and exposes a documented CLI/API command only after upgrade and live-plugin proof.

## Reproduction Assessment

Yes, at source level. Current main lacks a public plugin reload/cache-clear path, classifies plugin source/install changes as restart-backed, and docs instruct users to restart the live Gateway after source changes; I did not run a live container/plugin edit cycle in this read-only review.

## Solution Assessment

No, deleting dependency cache files directly is not the best permanent solution. The safer path is a lifecycle-owned Gateway reload operation with explicit teardown, cache invalidation, registry replacement, and CLI/API semantics.
