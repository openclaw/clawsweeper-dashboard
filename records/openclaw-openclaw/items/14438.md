## Summary

Keep open: current main partially hot-reloads plugin enable/config policy changes, but it still lacks a supported plugin source/Jiti cache reload or cache-clear workflow, and maintainer discussion explicitly left the Gateway/plugin-loader lifecycle design open.

## What This Changes

The issue requests a supported plugin development reload workflow that invalidates plugin/Jiti caches without restarting the OpenClaw container or Gateway.

## Best Possible Solution

Design and document a supported plugin development reload lifecycle that invalidates loader/Jiti/module state, safely swaps active registrations, disposes stale resources, and exposes an additive CLI/RPC workflow such as `openclaw plugins reload <id>` or `openclaw plugins clear-cache`.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature request. Source, docs, tests, and issue discussion give a high-confidence inspection path showing plugin source/load/install changes remain restart-backed and no public reload/cache-clear command exists.

## Solution Assessment

Unclear until maintainers choose lifecycle semantics. Deleting a guessed Jiti cache path alone is not the best fix because it would not safely replace active registrations or dispose plugin-owned resources.
