## Summary

Keep open: current main still lacks a supported workflow to reload TypeScript plugin source or clear Jiti/plugin-loader state without a Gateway/container restart. The merged plugin-management hot-reload work deliberately keeps source-changing plugin installs restart-backed, and the live discussion/labels still point to maintainer lifecycle design rather than cleanup closure.

## What This Changes

Request a supported developer workflow, CLI command, or watcher to invalidate Jiti/plugin loader state and reload TypeScript plugin changes without restarting the OpenClaw process or container.

## Best Possible Solution

Design a Gateway-backed plugin source reload or cache-clear workflow with explicit cache invalidation, runtime disposal/replacement, channel restart semantics, and docs; if live source reload remains unsupported, document the restart-only boundary precisely.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source and docs inspection give high confidence that current main lacks a supported plugin source/Jiti reload command or watcher while source-changing plugin paths remain restart-backed.

## Solution Assessment

Unclear until maintainers choose the lifecycle boundary. A targeted Gateway-backed plugins reload or clear-cache command looks safer as a first surface than automatic source watching because it can make cache invalidation and runtime disposal explicit.
