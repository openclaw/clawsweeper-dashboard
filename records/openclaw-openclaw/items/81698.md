## Summary

Keep open: current main fixes one related getMe timeout cause via sticky Telegram probe fallback, but it still does not aggregate sustained Telegram `getMe`/probe failures into the channel health restart policy for a still-connected account.

## What This Changes

The issue requests a frozen-channel health trigger that restarts Telegram after sustained `getMe` fetch-timeout failures while the provider remains nominally connected.

## Best Possible Solution

Add a Telegram-owned liveness failure signal for sustained Bot API probe/fetch failures and let the existing generic health policy restart the account without hardcoding Telegram probes in core.

## Reproduction Assessment

No live high-confidence reproduction was established. Source inspection does show the current-main gap: probe failures can be observed in status refreshes, but no consecutive-failure state reaches `evaluateChannelHealth` or the background restart loop.

## Solution Assessment

Unclear as written: repeated auth-probe failures are the right signal, but the maintainable fix should keep Telegram API probing in the Telegram/plugin lifecycle and pass generic liveness facts into health evaluation instead of adding provider-specific probing to core.
