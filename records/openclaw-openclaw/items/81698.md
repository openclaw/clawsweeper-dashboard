## Summary

Keep open: current main has Telegram polling stall and stale-transport recovery, but the background health monitor still has no mechanism to aggregate sustained Telegram getMe/fetch-timeout probe failures for a still-connected account.

## What This Changes

The issue requests a health-monitor restart trigger for sustained Telegram getMe/auth-probe failures while the provider remains nominally connected.

## Best Possible Solution

Add a plugin-owned Telegram liveness failure signal for repeated Bot API probe/fetch timeouts and feed that generic fact into the existing health policy restart path without hardcoding Telegram probes in core.

## Reproduction Assessment

Yes at source level, but not as a live incident reproduction. Current main has no path that stores or aggregates repeated getMe probe failures into the background health-monitor verdict for a connected account.

## Solution Assessment

No, not exactly as proposed if it means core should run provider-specific auth probes. The narrow maintainable fix is for Telegram to publish bounded liveness-failure facts that the generic health policy can restart on.
