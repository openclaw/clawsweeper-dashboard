## Summary

Keep open: current main restarts channels for stopped, disconnected, stale transport, and Telegram getUpdates stalls, but it does not aggregate sustained Telegram getMe/fetch-timeout failures into a health verdict for a still-connected account.

## What This Changes

Request to restart a nominally connected Telegram channel when repeated getMe fetch timeouts show the underlying HTTP client is frozen.

## Best Possible Solution

Implement a plugin-owned liveness failure signal that lets the existing health policy restart Telegram after sustained Bot API probe/fetch timeouts, while preserving current stale-socket and polling-stall behavior.

## Reproduction Assessment

No, not as a live high-confidence reproduction. Source inspection shows the current health path has no probe-failure aggregation, but the reported wedged Node HTTP client was not recreated.

## Solution Assessment

Unclear as written: repeated probe failures are the right signal, but the maintainable fix should keep Telegram API probing in the Telegram/plugin lifecycle and pass generic liveness facts into core health evaluation instead of hardcoding provider probes in core.
