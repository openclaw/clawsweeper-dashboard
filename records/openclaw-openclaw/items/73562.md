## Summary

Keep this open. Current main still has the direct `readline.question()` await in the shared confirmation helper, and the non-forced `plugins uninstall` path still reaches that helper before mutation. The linked closing PR remains open and unmerged, so this issue should stay as the tracker until that PR or an equivalent fix lands.

## What This Changes

Make non-forced `plugins uninstall` fail cleanly when confirmation stdin closes, while preserving piped yes/no answers and `--force`.

## Best Possible Solution

Review and land PR #73566, or an equivalent narrow fix, so the shared prompt primitive settles on readline close/EOF and `plugins uninstall` maps that condition to exit 1 with interactive or `--force` guidance. Keep the separate stale-entry cleanup tracked independently in #73551 unless maintainers intentionally combine the work.

## Reproduction Assessment

Yes. The issue gives deterministic closed-stdin steps, current source still reaches the direct readline await, and a minimal readline reproduction in this review exited 13 under `/dev/null`.

## Solution Assessment

Yes. The narrow maintainable fix is to make the shared prompt helper EOF-aware while preserving piped answers, then handle that specific prompt failure in `plugins uninstall`; a blanket non-TTY gate would regress valid `echo y` / `echo n` usage.
