## Summary

Keep open. Current main still lacks the proposed per-agent env scoping, and the PR branch is not merge-ready because the fetched diff has a compile-time import mismatch plus unresolved exec-environment isolation bugs.

## What This Changes

This PR adds an `agents.list[].env` scope/vars/allow config surface, schema validation, a `resolveAgentEnvVars()` helper, and plumbing into the exec tool defaults.

## Best Possible Solution

Keep the PR open and rework it around the current agent config helper split with an explicit env-scope contract, no owner identity leakage in scoped exec, and focused tests for `all`, `own`, and `none`.

## Reproduction Assessment

Yes. A static review of the fetched PR diff against current main reproduces the main failures: an unimported called symbol, skipped `env.vars` for default/all scope, and owner host env values overlaid into scoped exec runs.

## Solution Assessment

No. The product direction is plausible, but the patch is not the narrowest maintainable solution until the env source, host baseline, current helper placement, and tests are aligned as a security contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Import the env helper that is actually called:** `src/agents/agent-scope.ts:4`
  - body: The new resolver calls `collectConfigRuntimeEnvVars(cfg)`, but this file imports `collectConfigEnvVars`. That leaves the called symbol unbound, so the branch should fail TypeScript before the feature can build.
  - confidence: 0.97
- **[P1] Do not re-add owner HOME to scoped exec env:** `src/agents/bash-tools.exec.ts:1397-1404`
  - body: When `defaults.agentEnvVars` is set, the scoped branch overlays `HOME` and other host values from `process.env` after the agent-scoped vars. A collaborator configured with `scope: "none"` still receives the owner's home path, and an agent-provided `HOME` cannot win.
  - confidence: 0.92
- **[P2] Apply agent vars when scope defaults to all:** `src/agents/pi-tools.ts:470-474`
  - body: `agentEnvVars` is only resolved when `env.scope` is present and not `"all"`, so an agent with `env.vars` and omitted or explicit `all` scope falls back to raw `process.env` and silently loses those per-agent overrides.
  - confidence: 0.93
- **[P2] Allowlist real runtime env values for own scope:** `src/agents/agent-scope.ts:386-398`
  - body: `scope: "own"` filters only config-defined runtime env from `collectConfigRuntimeEnvVars(cfg)`, not the actual runtime `process.env`. That means allowlisting shell or CI-provided API keys cannot work even though the feature is meant to allow controlled global env inheritance.
  - confidence: 0.83
