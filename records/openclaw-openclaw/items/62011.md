## Summary

Keep this PR open for maintainer handling: current main still has no per-agent env scoping, but the branch remains security-sensitive, has unresolved correctness findings, has requested changes from a repository member, and still lacks real behavior proof.

## What This Changes

The PR adds an `agents.list[].env` scope/vars/allow config surface, a resolver, schema typing, and exec-tool plumbing for per-agent environment inheritance.

## Best Possible Solution

Define the scoped-env contract, rebase onto current main, route one tested resolver through exec without skipped vars or owner env leakage, and update config schema/help/docs plus focused tests.

## Reproduction Assessment

Yes for the PR defects: static source review at `ac226ce9ae8a2f8752e32b9f4890e857bca75186` shows scoped exec re-adds selected owner env values and skips `env.vars` for default/all scope. No after-fix runtime proof has been provided for the intended behavior.

## Solution Assessment

No. The direction is useful, but this implementation is not the best mergeable solution until the env baseline is explicit, the current resolver layout is used, focused security tests cover the contract, and real behavior proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped exec from re-inheriting owner env:** `src/agents/bash-tools.exec.ts:1395-1404`
  - body: The scoped branch spreads `defaults.agentEnvVars` and then overlays host values like `HOME`, `SHELL`, and `USER`. A collaborator configured with `scope: "none"` still receives owner identity/env baseline values, and an agent-provided `HOME` or `PATH` cannot win, contradicting the isolation contract this PR adds.
  - confidence: 0.92
- **[P2] Apply agent vars for default all scope:** `src/agents/pi-tools.ts:463-473`
  - body: `agentEnvVars` is only computed when `env.scope` is present and not `"all"`. If a config sets `env.vars` with omitted or explicit `all` scope, exec falls back to raw `process.env` and silently drops the agent-level overrides even though the new type comment says vars are always included.
  - confidence: 0.93
- **[P2] Allowlist values from the runtime environment:** `src/agents/agent-scope.ts:387-400`
  - body: `scope: "own"` filters only `collectConfigRuntimeEnvVars(cfg)`, which contains config-defined env entries, not the actual shell or CI `process.env`. That means allowlisting runtime API keys cannot work unless users duplicate those keys into config, undermining the stated controlled global inheritance mode.
  - confidence: 0.84
