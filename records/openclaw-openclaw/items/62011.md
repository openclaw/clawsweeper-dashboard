## Summary

Keep open. Current main does not implement per-agent env scoping, and the PR is not ready to merge because the diff has a compile-time helper mismatch plus security-boundary regressions in scoped exec env composition.

## What This Changes

The PR adds an `agents.list[].env` config block, schema validation, a per-agent env resolver, and exec-tool plumbing for `all`/`own`/`none` environment inheritance scopes.

## Best Possible Solution

Rebase onto current main, define the env-scope contract in the split agent config/schema/docs layout, route one canonical resolver into exec without reintroducing owner HOME, and cover all scopes with focused tests plus a user-facing changelog entry before merge.

## Reproduction Assessment

Yes. A static diff review against current main reproduces the blocking issues: the added resolver calls an unimported helper, default/all-scope `env.vars` is never passed to exec, and `scope: none` still receives host HOME through the exec env overlay.

## Solution Assessment

No. The feature direction is plausible, but this patch is not the best implementation until it is rebased onto the current resolver split and treats scoped exec env as a real security boundary with documented baseline inheritance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import the env helper that is actually called:** `src/agents/agent-scope.ts:387`
  - body: The new resolver imports `collectConfigEnvVars` but calls `collectConfigRuntimeEnvVars(cfg)`, leaving the called symbol undefined. This will fail TypeScript before the feature can build.
  - confidence: 0.95
- **[P1] Stop re-adding owner HOME to scoped exec env:** `src/agents/bash-tools.exec.ts:1398`
  - body: For scoped agents, the exec env branch overlays `HOME: process.env.HOME` after `defaults.agentEnvVars`. A `scope: "none"` agent still receives the owner's home directory, and an agent-provided HOME cannot take priority, which undermines the isolation this PR is adding.
  - confidence: 0.92
- **[P2] Apply agent vars when scope defaults to all:** `src/agents/pi-tools.ts:470-472`
  - body: `agentEnvVars` is only built when `env.scope` is explicitly set and not `"all"`, so an agent with `env.vars` and omitted/default or explicit `all` scope falls back to raw `process.env` and silently loses its agent-specific overrides.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff introduces a security boundary for collaborator exec runs but still leaks owner host env values into supposedly scoped executions.

Concerns:

- **[medium] Scoped exec still inherits owner HOME:** `src/agents/bash-tools.exec.ts:1398`
  - body: The scoped env branch overlays host `HOME` after `defaults.agentEnvVars`; current host env sanitization preserves inherited HOME, so `scope: "none"` does not mean zero global inheritance and cannot override HOME with an agent-specific value.
  - confidence: 0.9

  - body: The same branch re-adds selected host values such as PATH, USER, LANG, and TERM while the new config comments promise no global env for `scope: "none"`. The allowed baseline needs an explicit contract and regression tests before this is a reliable security boundary.
  - confidence: 0.78
