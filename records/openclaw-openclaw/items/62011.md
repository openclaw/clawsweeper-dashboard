## Summary

Keep open: this is a plausible security feature, but the current PR head still has blocking env-isolation defects, lacks after-fix real behavior proof, and has an active maintainer changes-requested review rather than evidence that main already supersedes it.

## What This Changes

The PR adds `agents.list[].env` scope/vars/allow config, a resolver, schema validation, and exec-tool plumbing for per-agent environment inheritance.

## Best Possible Solution

Define the scoped-env contract explicitly, rebase onto current main's agent config layout, route a single tested resolver into exec without skipped vars or owner env leakage, and update schema/help/docs coverage.

## Reproduction Assessment

Yes for the PR defects: static review of PR head ac226ce9ae8a shows default/all-scope `env.vars` is skipped, `scope: "own"` allowlisting is config-only, and scoped exec re-adds owner host env values. No runtime proof exists for the intended after-fix behavior.

## Solution Assessment

No. The direction is reasonable, but the current patch is not the narrow maintainable solution until the env baseline is explicit, the current-main resolver layout is used, and the security contract has focused tests plus real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped exec from re-inheriting owner env:** `src/agents/bash-tools.exec.ts:1393-1404`
  - body: The scoped branch spreads `defaults.agentEnvVars` and then overlays `PATH`, `HOME`, `SHELL`, `USER`, `LANG`, and `TERM` from `process.env`. That gives `scope: "none"` agents owner host values and makes agent-provided `HOME` or `PATH` lose, contradicting the isolation contract this PR adds.
  - confidence: 0.91
- **[P2] Apply agent vars when scope is all or omitted:** `src/agents/pi-tools.ts:463-473`
  - body: `agentEnvVars` is only computed when `env.scope` is present and not `"all"`. A config with `env.vars` and omitted or explicit `all` scope falls back to raw `process.env`, so the agent-specific overrides promised by the new type comments are silently ignored.
  - confidence: 0.93
- **[P2] Allowlist values from the runtime environment:** `src/agents/agent-scope.ts:387-400`
  - body: `scope: "own"` filters `collectConfigRuntimeEnvVars(cfg)`, which contains config-defined env entries rather than the shell or CI `process.env`. Allowlisting API keys that exist only in the runtime environment will not work unless users duplicate them into config, undermining the intended controlled inheritance mode.
  - confidence: 0.82
