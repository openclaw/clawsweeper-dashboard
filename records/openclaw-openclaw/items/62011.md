## Summary

Keep open: this is not obsolete, current main does not implement the scoped-env surface, and the current PR head still has blocking correctness/security issues plus missing real behavior proof.

## What This Changes

The PR adds `agents.list[].env` scoping (`all`/`own`/`none`), schema/types, `resolveAgentEnvVars()`, and `agentEnvVars` plumbing into exec tool creation.

## Best Possible Solution

Define a documented scoped-env contract, implement one canonical resolver through agent config and exec env construction, add focused config/exec tests and docs/help updates, then require redacted real-run proof.

## Reproduction Assessment

Yes for the PR defects: PR-head source inspection shows the scoped exec overlay, skipped default/all `env.vars`, and `own` allowlist source issue. The feature itself still lacks real runtime proof.

## Solution Assessment

No. The feature direction is plausible, but the current patch is not the narrowest maintainable solution until the host-env baseline and `all`/`own` semantics are explicit, tested, documented, and proven in a real run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped exec from re-inheriting owner env:** `src/agents/bash-tools.exec.ts:1393-1404`
  - body: The scoped branch spreads `defaults.agentEnvVars` and then overlays selected `process.env` values. A collaborator configured with `scope: "none"` still receives host identity/baseline values, and agent-provided `HOME` cannot win, contradicting the new isolation contract.
  - confidence: 0.91
- **[P2] Apply agent vars when scope defaults to all:** `src/agents/pi-tools.ts:469-472`
  - body: `agentEnvVars` is only computed when `env.scope` is present and not `"all"`. If a config sets `env.vars` with omitted or explicit `all` scope, exec falls back to raw `process.env` and silently drops the agent-level overrides even though the new type comment says vars are always included.
  - confidence: 0.93
- **[P2] Allowlist runtime env values for own scope:** `src/agents/agent-scope.ts:387-400`
  - body: `scope: "own"` filters only `collectConfigRuntimeEnvVars(cfg)`, which is config-defined env, not the actual shell or CI `process.env`. Allowlisting runtime API keys will not work unless users duplicate those keys into config, undermining the stated controlled inheritance mode.
  - confidence: 0.82

## Security Review
