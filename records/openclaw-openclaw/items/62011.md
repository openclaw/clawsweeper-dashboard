## Summary

Keep this PR open: the feature is not on current main, and the proposed security-sensitive env isolation path still has concrete correctness/security defects plus missing real behavior proof.

## What This Changes

This PR adds an `agents.list[].env` scope/vars/allow config and threads a resolved per-agent environment into host exec creation.

## Best Possible Solution

Define the per-agent env isolation contract, implement it through the existing env sanitization/config helpers with focused tests and docs, then require redacted real-run proof before merge.

## Reproduction Assessment

Yes for the PR defects: static inspection of PR head shows host-env overlay, skipped default/all env.vars, and config-only own-scope allowlisting. The feature itself still lacks real runtime proof.

## Solution Assessment

No. The direction is plausible, but this patch needs an explicit baseline-env policy, correct runtime env sourcing, focused tests/docs, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Prevent scoped exec from re-inheriting host identity env:** `src/agents/bash-tools.exec.ts:1393-1403`
  - body: When agentEnvVars is set, this branch spreads the scoped env and then overlays PATH, HOME, SHELL, USER, LANG, and TERM from the owner host. A scope:"none" agent still receives host baseline values, and agent-provided HOME cannot override them, contradicting the isolation contract this PR adds.
  - confidence: 0.91
- **[P2] Honor env.vars when scope defaults to all:** `src/agents/pi-tools.ts:468-472`
  - body: agentEnvVars is only computed when env.scope is explicitly present and not "all". Configuring env.vars with omitted or explicit all scope falls back to raw process.env and silently drops the agent-level vars even though the new type comment says they are always included.
  - confidence: 0.93
- **[P2] Allowlist values from the runtime environment:** `src/agents/agent-scope.ts:387-400`
  - body: scope:"own" filters only collectConfigRuntimeEnvVars(cfg), which is config-defined env, not the actual shell or CI runtime environment. Allowlisting an existing runtime API key will not work unless users duplicate that key into config, undermining the intended controlled inheritance mode.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch changes the exec/env security boundary and still leaks or misses scoped environment values in ways that need correction before merge.

Concerns:

- **[medium] Scoped exec still inherits owner host env:** `src/agents/bash-tools.exec.ts:1393`
  - body: When agentEnvVars is set, exec overlays selected process.env values after the scoped env, so scope:"none" is not limited to agent vars and cannot safely override baseline keys such as HOME.
  - confidence: 0.91
- **[medium] Own-scope allowlist misses runtime secrets:** `src/agents/agent-scope.ts:387`
  - body: The allowlist is applied only to config-defined env values, so administrators cannot allowlist already-present shell or CI API keys without duplicating them into config.
  - confidence: 0.82
- **[medium] Scoped env baseline is undefined:** `src/agents/bash-tools.exec.ts:1396`
  - body: The new config comments promise no global env for scope:"none", but the implementation forwards host baseline variables without a documented and tested policy for which values are safe or necessary.
  - confidence: 0.78
