## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Stop scoped exec from re-inheriting host identity env:** `src/agents/bash-tools.exec.ts:1393-1404`
  - body: The scoped branch spreads `defaults.agentEnvVars` and then overlays host `PATH`, `HOME`, `SHELL`, `USER`, `LANG`, and `TERM`. A collaborator configured with `scope: "none"` still receives host baseline identity env, and agent-provided baseline keys cannot win, so the new isolation contract is not enforced.
  - confidence: 0.91
- **[P2] Honor env.vars when scope defaults to all:** `src/agents/pi-tools.ts:469-472`
  - body: `agentEnvVars` is only computed when `env.scope` is present and not `"all"`. If config sets `env.vars` with omitted or explicit all scope, exec falls back to raw `process.env` and silently drops agent-level overrides even though the new type comment says vars are always included.
  - confidence: 0.93
- **[P2] Allow own scope to read allowed runtime env:** `src/agents/agent-scope.ts:387-400`
  - body: `scope: "own"` filters only `collectConfigRuntimeEnvVars(cfg)`, which is config-defined env, not the actual shell or CI `process.env`. Allowlisted runtime API keys will not work unless users duplicate those keys into config, undermining controlled inheritance.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch introduces a collaborator env-isolation boundary, but the scoped exec path still forwards host baseline env and own-scope allowlisting misses runtime credentials.

Concerns:

- **[medium] Scoped exec still inherits host baseline env:** `src/agents/bash-tools.exec.ts:1393`
  - body: When `agentEnvVars` is set, exec overlays selected `process.env` values after scoped vars, so `scope: "none"` is not limited to agent-specific vars and cannot override baseline keys safely.
  - confidence: 0.91
- **[medium] Scoped env baseline is not specified:** `src/config/types.agents.ts:104`
  - body: Some host baseline values may be operationally necessary, but the new config comments promise no global env for `none` while the implementation forwards host values without a documented and tested policy.
  - confidence: 0.78
- **[medium] Own-scope allowlist misses runtime secrets:** `src/agents/agent-scope.ts:387`
  - body: The allowlist is applied only to config-defined env, so administrators cannot allowlist already-present shell or CI API keys without duplicating them into config.
  - confidence: 0.82
