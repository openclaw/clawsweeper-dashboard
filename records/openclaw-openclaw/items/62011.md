## Summary

Keep open. Current main does not have the requested per-agent env scope surface, and the PR is useful but not mergeable because it currently has a compile-time helper mismatch plus security-boundary defects in scoped exec env composition.

## What This Changes

This PR adds an `agents.list[].env` config block, schema validation, a per-agent env resolver, and exec-tool plumbing for `all`/`own`/`none` environment inheritance scopes.

## Best Possible Solution

Define the env-scope contract in the current agent config/schema layout, feed exec through one canonical resolver, preserve the scoped security boundary without undocumented owner env inheritance, and cover all three scopes before merging.

## Reproduction Assessment

Yes. Static diff review against current main reproduces the main failures: the added resolver calls/imports the wrong env helper, default/all-scope `env.vars` never reaches exec, and `scope: "none"` still receives selected host env values.

## Solution Assessment

No. The owner-module direction is plausible, but this patch is not the best implementation until it is rebased onto the current split agent config helper and the exec env baseline is specified and tested as a real security boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import the env helper that is actually called:** `src/agents/agent-scope.ts:387`
  - body: The resolver calls `collectConfigRuntimeEnvVars(cfg)`, but the file imports `collectConfigEnvVars` from `../config/env-vars.js`. Current main's barrel exports `collectConfigRuntimeEnvVars`, so this will fail typechecking before the feature can run.
  - confidence: 0.95
- **[P1] Do not re-add owner HOME to scoped exec env:** `src/agents/bash-tools.exec.ts:1398`
  - body: When `defaults.agentEnvVars` is set, the new branch spreads the scoped vars and then overlays `HOME` from `process.env`. A `scope: "none"` agent still receives the owner's home directory and cannot override it, which breaks the promised zero-global-inheritance boundary.
  - confidence: 0.91
- **[P2] Apply agent vars for the default scope:** `src/agents/pi-tools.ts:470-472`
  - body: `agentEnvVars` is only computed when `env.scope` is present and not `"all"`. An agent with `env.vars` and omitted/default `all` scope falls back to raw `process.env`, so agent-specific overrides never reach exec despite the new contract saying they always apply.
  - confidence: 0.93
- **[P2] Allowlist the actual runtime env:** `src/agents/agent-scope.ts:387`
  - body: `scope: "own"` builds its global source from `collectConfigRuntimeEnvVars(cfg)`, which only includes config-defined env. That means allowlisted keys that exist only in the inherited shell or CI `process.env` are unavailable, including the API-key inheritance scenario this PR is meant to make selective.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/config/types.agents.ts:97`
  - body: This adds a user-facing agents config surface and exec behavior, but the PR does not touch `CHANGELOG.md`. OpenClaw policy requires user-facing `feat` changes to include a changelog entry.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff adds an agent env isolation boundary but still forwards selected owner host env values into scoped collaborator exec runs.

Concerns:

- **[medium] Scoped exec still inherits owner HOME:** `src/agents/bash-tools.exec.ts:1398`
  - body: The scoped env branch overlays host `HOME` after `defaults.agentEnvVars`, so `scope: "none"` does not provide zero global inheritance and cannot override `HOME` with an agent-specific value.
  - confidence: 0.91
- **[medium] Scoped exec reintroduces selected host env by default:** `src/agents/bash-tools.exec.ts:1396`
  - body: The same block re-adds `PATH`, `SHELL`, `USER`, `LANG`, and `TERM` for scoped agents. Some baseline may be needed, but the security contract and tests need to define it explicitly instead of claiming no global inheritance.
  - confidence: 0.8
- **[low] Allowlisted own scope does not use inherited process env:** `src/agents/agent-scope.ts:387`
  - body: The `own` resolver uses config env as the allowlist source, not the actual inherited runtime env, so operators cannot selectively pass shell/CI keys through this security boundary as described.
  - confidence: 0.78
