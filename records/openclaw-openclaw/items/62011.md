## Summary

## What This Changes

The branch adds `agents.list[].env` config with `all`/`own`/`none`, a resolver, schema validation, and `agentEnvVars` plumbing into exec environment creation.

## Best Possible Solution

Define and document a tested scoped-env security contract, route one resolver through canonical agent config and exec paths, update config docs/schema/help, and require redacted real-run proof before merge.

## Reproduction Assessment

Yes for the PR defects: static inspection of PR head `ac226ce9ae8a2f8752e32b9f4890e857bca75186` shows the scoped exec overlay, skipped default/all `env.vars`, and `own` allowlist source issue. The feature itself still needs real runtime proof.

## Solution Assessment

No. The feature direction may be valid, but the current patch is not the narrowest maintainable solution until the host-env baseline is explicit, tested, documented, and proven in a real run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped exec from re-inheriting owner env:** `src/agents/bash-tools.exec.ts:1396-1403`
  - body: The scoped branch spreads `defaults.agentEnvVars` and then overlays host `PATH`, `HOME`, `SHELL`, `USER`, `LANG`, and `TERM`. A collaborator configured with `scope: "none"` still receives owner host identity/baseline values, and agent-provided `HOME` cannot win, contradicting the isolation contract this PR adds.
  - confidence: 0.91
- **[P2] Apply agent vars for default all scope:** `src/agents/pi-tools.ts:469-472`
  - body: `agentEnvVars` is only computed when `env.scope` is present and not `"all"`. If a config sets `env.vars` with omitted or explicit `all` scope, exec falls back to raw `process.env` and silently drops the agent-level overrides even though the new type comment says vars are always included.
  - confidence: 0.93
- **[P2] Allowlist values from the runtime environment:** `src/agents/agent-scope.ts:387-400`
  - body: `scope: "own"` filters only `collectConfigRuntimeEnvVars(cfg)`, which is config-defined env, not the actual shell or CI `process.env`. That means allowlisting runtime API keys will not work unless users duplicate those keys into config, undermining the stated controlled global inheritance mode.
  - confidence: 0.82
