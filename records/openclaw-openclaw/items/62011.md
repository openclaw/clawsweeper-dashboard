## Summary

Keep open. Current main does not implement per-agent env scoping, and the current PR diff is still not merge-ready: it is stale against the current agent config split, has a compile-time helper import mismatch, and still weakens the proposed collaborator isolation boundary. Prior Greptile and ClawSweeper comments correctly identified overlapping correctness/security issues that remain relevant.

## What This Changes

The PR adds an `agents.list[].env` config block, schema validation, a per-agent env resolver, and exec-tool plumbing for `all`/`own`/`none` env inheritance scopes.

## Best Possible Solution

Define the env-scope contract in the current agent config/schema/docs layout, make one canonical resolver feed exec without reintroducing owner HOME or skipped agent vars, and cover all three scopes with focused tests before landing.

## Reproduction Assessment

Yes. A static diff review is enough to reproduce the main failures: the added resolver calls an unimported symbol, default/all-scope `env.vars` never reaches exec, and a `scope="none"` agent still receives host HOME through the exec env overlay.

## Solution Assessment

No. The direction is plausible, but this patch is not the best implementation until it is rebased onto the current split config resolver and treats scoped env as a real security boundary instead of layering selected owner env back in.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import the env helper that is actually called:** `src/agents/agent-scope.ts:387`
  - body: The new resolver calls `collectConfigRuntimeEnvVars(cfg)`, but the PR imports `collectConfigEnvVars` from `../config/env-vars.js`. That leaves `collectConfigRuntimeEnvVars` undefined and will fail TypeScript before the feature can build.
  - confidence: 0.94
- **[P1] Do not re-add owner HOME to scoped exec env:** `src/agents/bash-tools.exec.ts:1398`
  - body: For scoped agents, the exec base env spreads `defaults.agentEnvVars` and then overlays host values including `HOME`. A collaborator configured with `scope: "none"` still receives the owner home directory, and an agent-provided HOME cannot take priority, which undermines the isolation this PR is adding.
  - confidence: 0.9
- **[P2] Apply agent vars when scope defaults to all:** `src/agents/pi-tools.ts:470-472`
  - body: `agentEnvVars` is only built when `env.scope` is set and not `"all"`, so an agent with `env.vars` and the default or explicit `all` scope falls back to raw `process.env` and silently loses its agent-specific overrides.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff adds a security boundary but still leaks owner host identity/env values into supposedly scoped collaborator exec runs.

Concerns:

- **[medium] Scoped exec still inherits owner HOME:** `src/agents/bash-tools.exec.ts:1398`
  - body: The scoped env branch overlays host `HOME` after `defaults.agentEnvVars`, so `scope: "none"` does not actually mean zero global inheritance and cannot override HOME with an agent-specific value.
  - confidence: 0.9
- **[medium] Scoped exec reintroduces selected host env by default:** `src/agents/bash-tools.exec.ts:1396`
  - body: The same block re-adds host `PATH`, `SHELL`, `USER`, `LANG`, and `TERM` for scoped agents. Some of these may be operationally necessary, but the contract and tests need to define them explicitly instead of claiming no global inheritance.
  - confidence: 0.78
