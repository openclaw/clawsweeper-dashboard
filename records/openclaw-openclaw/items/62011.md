## Summary

Keep open. Current main does not implement per-agent env scoping, but this branch is not merge-ready because the proposed security boundary has concrete correctness and isolation defects against the current exec/config code.

## What This Changes

The PR adds an `agents.list[].env` config block with `scope`, `vars`, and `allow`, resolves agent-scoped env vars, and passes them into exec tool defaults.

## Best Possible Solution

Rebase onto current main, define the agent env-scope contract in config/docs, resolve through canonical agent config helpers, preserve explicit agent vars, avoid owner HOME inheritance for isolated agents, and add focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the PR diff shows `scope=none` still receives owner `HOME`, default/all scope skips `env.vars`, empty string vars are filtered, and the added env helper import is stale against current main.

## Solution Assessment

No. The direction is plausible, but this patch is not the best current implementation because it bypasses canonical lookup, weakens the isolation promise, misses default-scope vars, and is stale against the current helper exports.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the current runtime env helper:** `src/agents/agent-scope.ts:4`
  - body: On current main, `../config/env-vars.js` exports `collectConfigRuntimeEnvVars` but not `collectConfigEnvVars`, so this added import will fail typechecking after rebase. Import and use the current runtime helper instead.
  - confidence: 0.86
- **[P1] Do not overwrite scoped HOME with owner HOME:** `src/agents/bash-tools.exec.ts:1395`
  - body: For scoped agents, the exec base env spreads `defaults.agentEnvVars` and then overlays `HOME: process.env.HOME`. A `scope=none` collaborator still gets the owner's home directory, and an agent-specific HOME cannot win, which undermines this isolation boundary.
  - confidence: 0.9
- **[P2] Apply agent vars when scope defaults to all:** `src/agents/pi-tools.ts:470-472`
  - body: `agentEnvVars` is only built when `env.scope` is set and not `"all"`, so an agent with `env.vars` and omitted/default scope silently falls back to raw `process.env` and never applies those agent-specific variables.
  - confidence: 0.91
- **[P2] Preserve empty agent env values:** `src/agents/agent-scope.ts:412-414`
  - body: The resolver copies agent vars only when `if (value)` is truthy, so `vars: { SOME_FLAG: "" }` is dropped even though the type and schema accept strings. Empty env values are valid intentional overrides.
  - confidence: 0.9
- **[P2] Use canonical agent lookup before applying scope:** `src/agents/pi-tools.ts:465-468`
  - body: The inline lowercase `agents.list.find` check does not use the same normalization as `resolveAgentConfig`. If an id only matches after canonical normalization, this gate can miss `env.scope` and leave full process env inheritance enabled.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `src/config/types.agents.ts:97`
  - body: This adds a user-facing config feature, but the PR does not update `CHANGELOG.md`. Add one single-line entry under the active version before this ships.
  - confidence: 0.82
