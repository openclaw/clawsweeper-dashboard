## Summary

Keep open. The PR is a plausible security feature, but the latest diff still has blocking env-isolation defects, lacks after-fix real behavior proof, and needs maintainer/security agreement on the scoped exec baseline.

## What This Changes

Adds `agents.list[].env` scope/vars/allow config, a resolver, schema validation, and exec-tool plumbing for per-agent environment inheritance.

## Best Possible Solution

Rebase onto current main, define and document the scoped-env contract, route one tested resolver into exec without skipped vars or owner env leakage, and update focused tests, docs/schema help, and changelog.

## Reproduction Assessment

Yes for the PR defects: static diff review at head `ac226ce9ae8a2f8752e32b9f4890e857bca75186` shows default/all-scope `env.vars` is skipped and scoped exec re-adds owner host env values. No runtime proof has been provided for the intended after-fix behavior.

## Solution Assessment

No. The direction is reasonable, but this patch is not the narrow maintainable solution until the env baseline is explicit, current-main resolver layout is used, and the security contract has focused tests and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped exec from re-inheriting owner env:** `src/agents/bash-tools.exec.ts:1396-1404`
  - body: The scoped branch spreads `defaults.agentEnvVars` and then overlays host values like `HOME`, `SHELL`, and `USER`. A collaborator configured with `scope: "none"` still receives owner identity/env baseline values, and an agent-provided `HOME` cannot win, which contradicts the isolation contract this PR adds.
  - confidence: 0.91
- **[P2] Apply agent vars for default all scope:** `src/agents/pi-tools.ts:463-474`
  - body: `agentEnvVars` is only computed when `env.scope` is present and not `"all"`. If a config sets `env.vars` with omitted or explicit `all` scope, exec falls back to raw `process.env` and silently drops the agent-level overrides even though the new type comment says vars are always included.
  - confidence: 0.93
- **[P2] Allowlist values from the runtime environment:** `src/agents/agent-scope.ts:387-398`
  - body: `scope: "own"` filters only `collectConfigRuntimeEnvVars(cfg)`, which is config-defined env, not the actual shell/CI `process.env`. That means allowlisting runtime API keys cannot work unless users duplicate those keys into config, undermining the stated controlled global inheritance mode.
  - confidence: 0.82
