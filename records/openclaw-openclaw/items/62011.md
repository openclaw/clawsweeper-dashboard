## Summary

Keep open: current main does not implement per-agent env scoping, but this PR is dirty and still has build, behavior, and security-boundary blockers before it can merge.

## What This Changes

The PR adds per-agent `env` scope/vars/allow config and wires a resolved env map into exec tool creation.

## Best Possible Solution

Rebase onto current main and implement one canonical agent-env resolver through the current config/exec path, with secops-reviewed baseline semantics and tests for `all`, `own`, and `none`.

## Reproduction Assessment

Yes. Static diff review against current main is enough: the final patch calls an unimported helper, skips `env.vars` for default/all scope, and overlays owner host env values for scoped exec runs.

## Solution Assessment

No. The direction is plausible, but this is not the best implementation until it is rebased onto the current split config helpers and treats scoped env inheritance as an explicit security contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import the env helper that is actually called:** `src/agents/agent-scope.ts:4`
  - body: The final patch calls `collectConfigRuntimeEnvVars(cfg)`, but the import added at the top is still `collectConfigEnvVars`. That leaves the called symbol unbound and should fail TypeScript before the feature can build.
  - confidence: 0.96
- **[P1] Do not re-add owner HOME to scoped exec env:** `src/agents/bash-tools.exec.ts:1398`
  - body: When `agentEnvVars` is set, the exec base env spreads scoped vars and then overlays host values including `HOME`. A collaborator configured with `scope: "none"` still receives the owner's home directory, and an agent-provided `HOME` cannot override it.
  - confidence: 0.91
- **[P2] Apply agent vars when scope defaults to all:** `src/agents/pi-tools.ts:469-470`
  - body: `agentEnvVars` is only resolved when `env.scope` is present and not `"all"`, so an agent with `env.vars` and omitted or explicit all scope falls back to raw `process.env` and silently loses those per-agent overrides.
  - confidence: 0.93
- **[P2] Allowlist real runtime env values for own scope:** `src/agents/agent-scope.ts:387`
  - body: `scope: "own"` filters only config-defined env from `collectConfigRuntimeEnvVars(cfg)`, not the runtime `process.env`. Allowlisting shell or CI-provided API keys therefore cannot work even though the feature is meant to allow controlled global env inheritance.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch adds a security-sensitive collaborator env isolation boundary, but scoped exec still inherits selected owner host env values.

Concerns:

- **[medium] Scoped exec still inherits owner HOME:** `src/agents/bash-tools.exec.ts:1398`
  - body: The scoped env branch overlays host `HOME` after `defaults.agentEnvVars`, so `scope: "none"` does not actually mean only agent-specific env and cannot override `HOME` safely.
  - confidence: 0.91
- **[medium] Scoped exec baseline policy is undefined:** `src/agents/bash-tools.exec.ts:1397`
  - body: The same block re-adds host `PATH`, `SHELL`, `USER`, `LANG`, and `TERM`. Some values may be necessary, but the isolation contract and tests need to define that baseline before this can be treated as a security boundary.
  - confidence: 0.78
