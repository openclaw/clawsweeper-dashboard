## Summary

Keep open: current main does not implement per-agent env isolation, and the PR still has blocking correctness/security issues plus missing real behavior proof for a security-sensitive exec boundary.

## What This Changes

The PR adds `agents.list[].env` scope/vars/allow config, a scoped env resolver, schema/type support, and exec-tool plumbing for per-agent command environments.

## Best Possible Solution

Land a revised env-isolation design that documents the baseline policy, applies agent vars consistently, filters a clearly defined runtime/config env source, and proves the exec boundary with focused tests plus redacted real-run evidence.

## Reproduction Assessment

Yes for the PR defects: PR-head source inspection shows host-env overlay, skipped default/all `env.vars`, and config-only own-scope allowlisting. I did not run tests because this review is constrained to read-only inspection.

## Solution Assessment

No. The feature direction may be valid, but this patch is not the safest implementation until the security baseline, env source semantics, docs/schema, tests, and real behavior proof are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Stop scoped exec from re-inheriting host identity env:** `src/agents/bash-tools.exec.ts:1396-1403`
  - body: The scoped branch spreads `defaults.agentEnvVars` and then overlays host `PATH`, `HOME`, `SHELL`, `USER`, `LANG`, and `TERM`. A collaborator configured with `scope: "none"` still receives owner host baseline values, and agent-provided `HOME`/`PATH` cannot win, so the new isolation contract is not enforced.
  - confidence: 0.91
- **[P2] Honor env.vars when scope defaults to all:** `src/agents/pi-tools.ts:469-472`
  - body: `agentEnvVars` is only computed when `env.scope` is present and not `"all"`. If a config sets `env.vars` with omitted or explicit `all` scope, exec falls back to raw `process.env` and silently drops the agent-level overrides even though the new type comment says vars are always included.
  - confidence: 0.93
- **[P2] Allow own scope to read allowed runtime env:** `src/agents/agent-scope.ts:387-400`
  - body: `scope: "own"` filters only `collectConfigRuntimeEnvVars(cfg)`, which is config-defined env, not the actual shell or CI `process.env`. That means allowlisting runtime API keys will not work unless users duplicate those keys into config, undermining the controlled global-inheritance mode described by the PR.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch introduces a collaborator env-isolation boundary, but scoped exec still overlays host identity env and own-scope allowlisting misses runtime env values.

  - body: When `agentEnvVars` is set, exec overlays selected `process.env` values after scoped vars, so `scope: "none"` is not limited to agent-specific vars and cannot override baseline keys safely.

  - confidence: 0.78
