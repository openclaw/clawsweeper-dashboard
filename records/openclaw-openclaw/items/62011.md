## Summary

Keep open: current main does not implement per-agent env scoping, and this external PR still has security-sensitive env-isolation defects plus missing real behavior proof before it can be merged.

## What This Changes

The PR adds `agents.list[].env` scope/vars/allow config, a scoped env resolver, schema/type support, and exec-tool plumbing for per-agent command environments.

## Best Possible Solution

Define and document a scoped-env security contract, thread it through canonical agent config and exec env construction, cover env composition in focused tests, update schema/help/docs, and require redacted real-run proof.

## Reproduction Assessment

Yes for the PR defects: static inspection of PR head shows host-env overlay, skipped default/all `env.vars`, and config-only own-scope allowlisting. No real runtime proof has been provided for the feature behavior itself.

## Solution Assessment

No. The feature direction may be valid, but this patch is not ready until the security baseline policy is explicit, tested, documented, and proven in a real run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Prevent scoped exec from re-inheriting host identity env:** `src/agents/bash-tools.exec.ts:1396-1403`
  - body: The scoped branch spreads `defaults.agentEnvVars` and then overlays host `PATH`, `HOME`, `SHELL`, `USER`, `LANG`, and `TERM`. A collaborator configured with `scope: "none"` still receives owner host identity/baseline values, and agent-provided `HOME` cannot win, contradicting the isolation contract this PR adds.
  - confidence: 0.91
- **[P2] Honor env.vars when scope defaults to all:** `src/agents/pi-tools.ts:469-472`
  - body: `agentEnvVars` is only computed when `env.scope` is present and not `"all"`. If a config sets `env.vars` with omitted or explicit `all` scope, exec falls back to raw `process.env` and silently drops the agent-level overrides even though the new type comment says vars are always included.
  - confidence: 0.93
- **[P2] Allowlist values from the runtime environment:** `src/agents/agent-scope.ts:387-400`
  - body: `scope: "own"` filters only `collectConfigRuntimeEnvVars(cfg)`, which is config-defined env, not the actual shell or CI `process.env`. That means allowlisting runtime API keys will not work unless users duplicate those keys into config, undermining the stated controlled global inheritance mode.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch introduces a collaborator env-isolation boundary, but scoped exec still overlays selected owner host env values and own-scope allowlisting misses runtime env values.

  - confidence: 0.78
