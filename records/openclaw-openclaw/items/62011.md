## Summary

Keep open. Current main does not implement per-agent env scoping, and the PR is still dirty against main with concrete compile, behavior, and security-boundary defects in the proposed patch.

## What This Changes

The PR adds `agents.list[].env` scope/vars/allow config, schema validation, a per-agent env resolver, and exec-tool plumbing for `all`/`own`/`none` inheritance.

## Best Possible Solution

Define the env-scope contract in the current split agent config/schema/docs layout, feed one canonical resolver into exec without reintroducing owner identity or skipped agent vars, and cover all three scopes with focused tests.

## Reproduction Assessment

Yes. A static diff review is enough to reproduce the main failures: the added resolver calls an unimported symbol, default/all-scope `env.vars` never reaches exec, and scoped exec overlays owner host env values.

## Solution Assessment

No. The direction is plausible, but this patch is not the best implementation until it is rebased onto current main and treats scoped env inheritance as an explicit security contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import the env helper that is actually called:** `src/agents/agent-scope.ts:4`
  - body: The new resolver calls `collectConfigRuntimeEnvVars(cfg)`, but the diff imports `collectConfigEnvVars` from `../config/env-vars.js`. That leaves the called symbol unbound and should fail TypeScript before the feature can build.
  - confidence: 0.96
- **[P1] Do not re-add owner HOME to scoped exec env:** `src/agents/bash-tools.exec.ts:1397-1404`
  - body: When `defaults.agentEnvVars` is set, the exec base env spreads agent-scoped vars and then overlays host values including `HOME`. A collaborator configured with `scope: "none"` still receives the owner home directory, and an agent-provided `HOME` cannot win.
  - confidence: 0.91
- **[P2] Apply agent vars when scope defaults to all:** `src/agents/pi-tools.ts:470-474`
  - body: `agentEnvVars` is only resolved when `env.scope` is present and not `"all"`, so an agent with `env.vars` and omitted or explicit `all` scope falls back to raw `process.env` and silently loses those per-agent overrides.
  - confidence: 0.93
- **[P2] Allowlist real runtime env values for own scope:** `src/agents/agent-scope.ts:387-398`
  - body: `scope: "own"` filters only `collectConfigRuntimeEnvVars(cfg)`, which is config-defined env, not the actual runtime `process.env`. That means allowlisting shell or CI-provided API keys cannot work even though the feature is meant to allow controlled global env inheritance.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR adds a collaborator env isolation boundary but still leaks selected owner host env values into supposedly scoped exec runs.

Concerns:

- **[medium] Scoped exec still inherits owner HOME:** `src/agents/bash-tools.exec.ts:1397`
  - body: The scoped env branch overlays host `HOME` after `defaults.agentEnvVars`, so `scope: "none"` does not actually mean only agent-specific env and cannot override `HOME` safely.
  - confidence: 0.91
- **[medium] Scoped exec baseline is undefined security policy:** `src/agents/bash-tools.exec.ts:1397`
  - body: The same block re-adds host `PATH`, `SHELL`, `USER`, `LANG`, and `TERM` for scoped agents. Some values may be operationally necessary, but the contract and tests need to define them before this can be treated as isolation.
  - confidence: 0.78
