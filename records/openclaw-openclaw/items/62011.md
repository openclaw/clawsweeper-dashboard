## Summary

This PR should remain open because the requested env-isolation capability is not present on current main, but the current patch still has blocking correctness and security issues and lacks required real behavior proof.

## What This Changes

The PR adds `agents.list[].env` scope/vars/allow configuration, a `resolveAgentEnvVars()` resolver, Zod validation, and exec-tool plumbing for per-agent environment inheritance.

## Best Possible Solution

Define the scoped environment contract, implement one tested resolver that preserves agent overrides while excluding unapproved owner env, and require focused tests plus redacted runtime proof before merge.

## Reproduction Assessment

Yes for the PR defects: static review of PR head `ac226ce9ae8a2f8752e32b9f4890e857bca75186` shows scoped exec re-adds host identity env and default/all-scope agent vars are skipped. There is no real runtime proof for the intended after-fix behavior.

## Solution Assessment

No. The feature direction is plausible, but this patch is not the best mergeable solution until the env baseline is explicit, allowlisting reads the intended source, and the behavior is covered by tests and live proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped exec from re-inheriting owner env:** `src/agents/bash-tools.exec.ts:1397-1405`
  - body: When `agentEnvVars` is present, this branch applies the scoped env and then overlays host values like `HOME`, `SHELL`, and `USER`. A `scope: "none"` agent still receives owner identity/baseline values and cannot override `HOME`, contradicting the isolation contract this PR adds.
  - confidence: 0.91
- **[P2] Apply agent vars for the default all scope:** `src/agents/pi-tools.ts:463-474`
  - body: `agentEnvVars` is only resolved when `env.scope` is present and not `"all"`. Configuring `env.vars` with omitted or explicit `all` scope falls back to raw `process.env` and silently drops those agent-level overrides, despite the new contract saying agent vars are always included.
  - confidence: 0.93
- **[P2] Allowlist values from the intended runtime source:** `src/agents/agent-scope.ts:384-398`
  - body: `scope: "own"` filters `collectConfigRuntimeEnvVars(cfg)`, which is config-defined env rather than the live shell/CI env this feature is meant to control. Allowlisting an existing runtime API key will not work unless users duplicate it into config.
  - confidence: 0.82
