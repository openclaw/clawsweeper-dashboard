## Summary

Keep open: current main does not implement this new per-agent env isolation surface, and the PR still changes a security-sensitive config/exec boundary with blocking correctness/security findings plus missing real behavior proof.

## What This Changes

The PR adds `agents.list[].env` scoping (`all`/`own`/`none`), a resolver, schema/type support, and exec-tool plumbing for per-agent environment variables.

## Best Possible Solution

Define a documented and tested scoped-env security contract, route env resolution through the canonical agent config and exec paths, and require redacted real-run proof before merge.

## Reproduction Assessment

Yes for the PR defects: static inspection of PR head `ac226ce9ae8a2f8752e32b9f4890e857bca75186` shows the host-env overlay, skipped default/all `env.vars`, and config-only own-scope allowlist. No real runtime proof has been provided for the feature behavior itself.

## Solution Assessment

No. The feature direction may be valid, but the current patch is not the narrowest maintainable solution until the env baseline policy is explicit, tested, documented, and proven in a real run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Prevent scoped exec from re-inheriting host identity env:** `src/agents/bash-tools.exec.ts:1396-1403`
  - body: The scoped branch spreads `defaults.agentEnvVars` and then overlays host `PATH`, `HOME`, `SHELL`, `USER`, `LANG`, and `TERM`. A collaborator configured with `scope: "none"` still receives owner host identity/baseline values, and agent-provided `HOME` cannot win, contradicting the isolation contract this PR adds.
  - confidence: 0.91
- **[P2] Honor env.vars when scope defaults to all:** `src/agents/pi-tools.ts:469-472`
  - body: `agentEnvVars` is only computed when `env.scope` is present and not `"all"`. If a config sets `env.vars` with omitted or explicit `all` scope, exec falls back to raw `process.env` and silently drops the agent-level overrides even though the new type comment says vars are always included.
  - confidence: 0.93
- **[P2] Allowlist values from the runtime environment:** `src/agents/agent-scope.ts:387-400`
  - body: `scope: "own"` filters only `collectConfigRuntimeEnvVars(cfg)`, which is config-defined env, not the actual shell or CI `process.env`. That means allowlisting runtime API keys will not work unless users duplicate those keys into config, undermining the stated controlled global inheritance mode.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch introduces a collaborator env-isolation boundary, but scoped exec still overlays selected owner host env values and own-scope allowlisting misses existing runtime env values.

Concerns:

- **[medium] Scoped exec still inherits owner host env:** `src/agents/bash-tools.exec.ts:1396`
  - body: When `agentEnvVars` is set, the exec env overlays selected `process.env` values after scoped vars, so `scope: "none"` is not limited to agent-specific vars and cannot override `HOME` safely.
  - confidence: 0.91
- **[medium] Scoped env baseline is undefined:** `src/agents/bash-tools.exec.ts:1396`
  - body: Some host baseline values may be operationally necessary, but the new config comments promise no global env for `none` while the implementation forwards host values without a documented, tested policy.
  - confidence: 0.78
- **[medium] Own-scope allowlist misses runtime secrets:** `src/agents/agent-scope.ts:387`
  - body: The allowlist is applied only to config-defined env, so administrators cannot allowlist already-present shell or CI API keys without duplicating them into config, weakening the intended controlled-inheritance mode.
  - confidence: 0.82
