## Summary

Keep open. Current main does not implement per-agent environment scoping, and the PR is useful source work, but the proposed patch is not merge-ready because it skips the new `env.vars` contract in the default scope, can reintroduce the owner `HOME` into isolated exec environments, and drops intentional empty-string env values. The earlier #61696 context supports this being a replacement attempt rather than a duplicate to close.

## What This Changes

The PR adds an `agents.list[].env` config block with `scope`, `vars`, and `allow`, resolves per-agent env vars, validates the schema, and wires the result into exec tool defaults.

## Best Possible Solution

Rebase the feature onto current main, define the public env-scope contract in config types/schema/docs, resolve env through canonical agent config helpers, preserve agent vars including empty strings and safe HOME overrides, avoid reintroducing owner-derived env for isolated agents, add focused tests for all three scopes across exec paths, and include a changelog entry if the feature remains user-facing.

## Reproduction Assessment

Yes. From the PR diff alone, a config with `env.vars` and omitted/default `scope` never reaches exec because `agentEnvVars` stays undefined; a `scope=none` config still receives `process.env.HOME`; and an empty string in `env.vars` is skipped by `if (value)`.

## Solution Assessment

No. The direction is plausible, but this patch is not the narrowest maintainable security fix because it bypasses canonical agent lookup, weakens the isolation promise with owner HOME inheritance, and leaves key behavior untested; a rebased implementation should make the env resolver the single source of truth and then pass its output through the existing exec sanitization boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not overwrite scoped HOME with owner HOME:** `src/agents/bash-tools.exec.ts:1395`
  - body: For scoped agents, the exec base env spreads `defaults.agentEnvVars` and then overlays `HOME: process.env.HOME`. Since HOME is allowed as an inherited host env, a `scope=none` collaborator still sees the owner's home directory and an agent-specific HOME cannot take priority, which undermines the isolation this PR is adding.
  - confidence: 0.87
- **[P2] Apply agent vars when scope defaults to all:** `src/agents/pi-tools.ts:470-472`
  - body: `agentEnvVars` is only built when `env.scope` is set and not `"all"`, so an agent configured with `env.vars` and an omitted/default scope falls back to raw `process.env` and never applies those agent-specific variables. This contradicts the new contract that agent vars are always included and makes the most backward-compatible configuration silently ineffective.
  - confidence: 0.91
- **[P2] Preserve empty agent env values:** `src/agents/agent-scope.ts:412-414`
  - body: The resolver copies agent vars only when `if (value)` is truthy, so `vars: { SOME_FLAG: "" }` is dropped even though the type and schema accept strings. Empty env values are valid and can be intentional overrides, so the copy should not filter them out.
  - confidence: 0.9
- **[P2] Use canonical agent lookup before deciding isolation:** `src/agents/pi-tools.ts:465-468`
  - body: The inline `agents.list.find(a => a.id.toLowerCase() === agentId.toLowerCase())` check does not use the same normalization as `resolveAgentConfig`. If a configured id needs trimming or normalization, the gate can miss `env.scope` and leave the exec tool on full `process.env` inheritance instead of applying the requested isolation.
  - confidence: 0.78
