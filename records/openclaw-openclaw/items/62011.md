## Summary

Keep open: current main does not implement per-agent env scoping, but the PR is not mergeable as-is because the latest head still has scoped-env correctness/security blockers and no real behavior proof.

## What This Changes

The PR adds `agents.list[].env` scope/vars/allow config and threads resolved agent environment variables into exec tool creation.

## Best Possible Solution

Define and document the scoped-env contract, route one tested resolver through the current agent-scope/config layout, and prove exec runs without skipped vars or owner env leakage.

## Reproduction Assessment

Yes for the PR defects: static source review at head `ac226ce9ae8a2f8752e32b9f4890e857bca75186` shows scoped exec re-adds host env values, default/all-scope `env.vars` is skipped, and `own` allowlisting reads config env rather than live runtime env. No runtime proof has been provided for the intended after-fix behavior.

## Solution Assessment

No. The feature direction is plausible, but this implementation is not the narrow maintainable solution until the env baseline is explicit, the current-main resolver layout is used, and focused security tests plus real behavior proof exist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped exec from re-inheriting owner env:** `src/agents/bash-tools.exec.ts:1395-1404`
  - body: The scoped branch spreads `defaults.agentEnvVars` and then overlays host values like `PATH`, `HOME`, `SHELL`, and `USER`. A collaborator configured with `scope: "none"` still receives owner host env baseline values, and agent-provided values for those keys cannot win, contradicting the isolation contract this PR adds.
  - confidence: 0.91
- **[P2] Apply agent vars for default all scope:** `src/agents/pi-tools.ts:468-472`
  - body: `agentEnvVars` is only computed when `env.scope` is present and not `"all"`. If a config sets `env.vars` with omitted or explicit `all` scope, exec falls back to raw `process.env` and silently drops the agent-level overrides even though the new type comment says vars are always included.
  - confidence: 0.93
- **[P2] Allowlist values from the runtime environment:** `src/agents/agent-scope.ts:387-400`
  - body: `scope: "own"` filters only `collectConfigRuntimeEnvVars(cfg)`, which returns config-defined env entries rather than the actual shell or service runtime env. Allowlisting runtime API keys therefore cannot work unless users duplicate those keys into config.
  - confidence: 0.82
