## Summary

Keep open: the feature direction is plausible, but the latest head still contradicts its scoped-env security contract, lacks after-fix real behavior proof, and has a maintainer changes-requested review.

## What This Changes

Adds `agents.list[].env` scope/vars/allow config plus resolver, schema validation, and exec-tool plumbing for per-agent environment inheritance.

## Best Possible Solution

Define and document the scoped-env contract, rebase onto the current agent config layout, route one tested resolver into exec without owner env leakage or skipped vars, and update the config help/docs/changelog surfaces.

## Reproduction Assessment

Yes for the PR defects: source review of head `ac226ce9ae8a2f8752e32b9f4890e857bca75186` shows scoped exec re-adds owner process values and default/all-scope `env.vars` is skipped. No real runtime proof has been provided for the intended after-fix behavior.

## Solution Assessment

No. The requested direction is reasonable, but this patch is not the best maintainable solution until the env baseline is explicit, current-main resolver layout is used, and the security contract has focused tests plus real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped exec from re-inheriting owner env:** `src/agents/bash-tools.exec.ts:1393-1404`
  - body: For scoped agents, this spreads `defaults.agentEnvVars` and then overwrites it with `PATH`, `HOME`, `SHELL`, `USER`, `LANG`, and `TERM` from the owner's process. That means `scope: "none"` still receives owner host identity/baseline values and agent-provided `HOME` cannot win, contradicting the isolation contract.
  - confidence: 0.91
- **[P2] Apply agent vars when scope defaults to all:** `src/agents/pi-tools.ts:463-474`
  - body: Because `agentEnvVars` is only computed when `env.scope` is present and not `"all"`, a config with `env.vars` and omitted or explicit all scope falls back to raw `process.env`; the new per-agent vars are never applied even though the type comment says they are always included.
  - confidence: 0.93
- **[P2] Allowlist values from the runtime environment:** `src/agents/agent-scope.ts:387-400`
  - body: `scope: "own"` allowlists only values returned from `collectConfigRuntimeEnvVars(cfg)`, which contains config-declared env entries, not the actual shell/CI runtime env. Allowlisting a key from `process.env` will not work unless the user duplicates it into config, undermining the controlled global inheritance mode.
  - confidence: 0.84
