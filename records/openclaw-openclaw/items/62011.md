## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Stop scope none from inheriting the host home:** `src/agents/bash-tools.exec.ts:1403`
  - body: `scope: "none"` is documented as no global env inheritance, but the exec path still injects `HOME` from `process.env` whenever scoped env is enabled. Many CLIs discover owner credentials through `$HOME`, so an external collaborator agent can still reach owner-local auth/config locations despite the new zero-global claim.
  - confidence: 0.86
- **[P2] Read allowlisted values from the runtime env:** `src/agents/agent-scope.ts:401`
  - body: `resolveAgentEnvVars()` builds `globalVars` from `collectConfigRuntimeEnvVars(cfg)`, which only contains config-defined env entries. With `scope: "own", allow: ["OPENAI_API_KEY"]`, an API key supplied by the shell or service process is never considered, so the allowlist mode cannot provide the credentials it is meant to expose.
  - confidence: 0.9
- **[P2] Apply agent vars when scope is all:** `src/agents/pi-tools.ts:474`
  - body: `createOpenClawCodingTools()` only resolves `agentEnvVars` when `env.scope` is explicitly set and not `"all"`. If a user sets `agents.list[].env.vars` with the default scope or with `scope: "all"`, those agent-specific values are silently ignored even though the new contract says they always override.
  - confidence: 0.9
