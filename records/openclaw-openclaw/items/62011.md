## Summary

Keep this PR open for maintainer/security review rather than cleanup-close: current main does not implement per-agent env scoping, but the latest PR head still has correctness and security blockers plus missing real behavior proof.

## What This Changes

The PR adds `agents.list[].env` scope/vars/allow config, schema validation, a resolver, and exec-tool plumbing for per-agent environment inheritance.

## Best Possible Solution

Define and document the scoped-env contract, route one tested resolver into exec without skipped vars or owner-env leakage, and add focused schema/resolver/exec tests plus redacted real behavior proof.

## Reproduction Assessment

Yes for the PR defects: source review at head `ac226ce9ae8a2f8752e32b9f4890e857bca75186` shows default/all-scope `env.vars` is skipped and scoped exec re-adds selected owner host env values. No after-fix runtime proof has been provided for the intended behavior.

## Solution Assessment

No. The direction is plausible, but the current patch is not the narrow maintainable solution until the env baseline is explicit, process/config env semantics are settled, and the security boundary has tests and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped exec from re-inheriting owner env:** `src/agents/bash-tools.exec.ts:1393-1404`
  - body: For scoped agents, this branch spreads `defaults.agentEnvVars` and then overlays `PATH`, `HOME`, `SHELL`, `USER`, `LANG`, and `TERM` from `process.env`. A `scope: "none"` agent therefore still receives owner host values, and an agent-provided `HOME` cannot win, contradicting the isolation contract this PR adds.
  - confidence: 0.91
- **[P2] Apply agent vars when scope is all or omitted:** `src/agents/pi-tools.ts:463-474`
  - body: `agentEnvVars` is only resolved when `env.scope` is present and not `"all"`. If a config sets `env.vars` with omitted or explicit `all` scope, exec falls back to raw `process.env` and silently drops the agent-level overrides even though the new type comment says agent vars are always included.
  - confidence: 0.93
- **[P2] Allowlist values from the runtime environment:** `src/agents/agent-scope.ts:387-400`
  - body: `scope: "own"` filters only `collectConfigRuntimeEnvVars(cfg)`, which is config-defined env, not the actual shell/CI `process.env`. That means allowlisting runtime API keys cannot work unless users duplicate those keys into config, undermining the stated controlled global inheritance mode.
  - confidence: 0.82
