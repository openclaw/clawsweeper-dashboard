## Summary

Keep open. Current main does not implement the requested per-agent env scope, and the PR still has compile-time and security-boundary defects that should block merge.

## What This Changes

The PR adds an `agents.list[].env` scope/vars/allow config surface, a per-agent env resolver, and exec-tool plumbing for scoped environment inheritance.

## Best Possible Solution

Define the env-scope contract in the current config/schema/docs layout, route exec through one canonical resolver, avoid reintroducing owner env for scoped agents, and cover all three scopes with focused tests before landing.

## Reproduction Assessment

Yes. Static diff review is enough to reproduce the failures: the added resolver calls an unimported symbol, default/all-scope `env.vars` is skipped, and scoped exec overlays host env values.

## Solution Assessment

No. The direction is plausible, but the patch is not the best implementation until it is rebased onto the current agent config split and treats scoped env as an explicit security boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import the env helper that is actually called:** `src/agents/agent-scope.ts:387`
  - body: The new resolver calls `collectConfigRuntimeEnvVars(cfg)`, but this file imports `collectConfigEnvVars`. That leaves the called symbol unbound and makes the PR fail TypeScript before the feature can build.
  - confidence: 0.94
- **[P1] Do not re-add owner HOME to scoped exec env:** `src/agents/bash-tools.exec.ts:1398`
  - body: For scoped agents, the base env spreads `defaults.agentEnvVars` and then overlays `process.env.HOME`. A `scope: "none"` collaborator still receives the owner home directory, and an agent-provided `HOME` cannot take priority.
  - confidence: 0.9
- **[P2] Apply agent vars when scope defaults to all:** `src/agents/pi-tools.ts:470-472`
  - body: `agentEnvVars` is only built when `env.scope` is explicitly set and not `"all"`, so an agent with `env.vars` and omitted/default `all` scope silently loses its configured overrides.
  - confidence: 0.92
- **[P2] Allowlist values from the actual runtime env:** `src/agents/agent-scope.ts:387`
  - body: `scope: "own"` uses `collectConfigRuntimeEnvVars(cfg)`, which only returns config-defined env values. It cannot allowlist API keys supplied through the shell or CI env even though the feature is described as controlling global env inheritance.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/config/types.agents.ts:97`
  - body: This adds a user-facing `feat` config and exec behavior surface, but the PR does not update `CHANGELOG.md`, which OpenClaw policy requires for user-facing feature changes.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds a security boundary but still leaks selected owner host env values into scoped collaborator exec runs.

Concerns:

- **[medium] Scoped exec still inherits owner HOME:** `src/agents/bash-tools.exec.ts:1398`
  - body: The scoped env branch overlays `process.env.HOME` after agent-scoped env vars, so `scope: "none"` does not mean zero global inheritance and cannot override HOME with an agent-specific value.
  - confidence: 0.9

  - body: The same branch forwards host `PATH`, `SHELL`, `USER`, `LANG`, and `TERM`. Some keys may be operationally necessary, but the security contract and tests must define them instead of claiming no global inheritance.
