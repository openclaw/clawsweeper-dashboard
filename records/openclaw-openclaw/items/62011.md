## Summary

Keep open: current main does not implement per-agent env scoping, and the PR is still a security-sensitive config/exec change with changes requested, a conflicting head, unresolved env-isolation defects, and no contributor real behavior proof.

## What This Changes

Adds an `agents.list[].env` scope/vars/allow config, a resolver, schema validation, and exec-tool plumbing for per-agent environment inheritance.

## Best Possible Solution

Define and document the scoped-env security contract, route one tested resolver through current agent config and exec paths, update schema/help/docs, and require redacted runtime proof before merge.

## Reproduction Assessment

Yes for the PR defects: static inspection of PR head `ac226ce9ae8a2f8752e32b9f4890e857bca75186` shows skipped default/all-scope vars and selected owner host env re-inheritance. No after-fix real setup proof has been provided for the intended feature.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best mergeable solution until the env baseline is explicit, security-contract tests exist, config docs/schema are updated, and real behavior proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped exec from re-inheriting owner env:** `src/agents/bash-tools.exec.ts:1396-1404`
  - body: The scoped branch spreads `defaults.agentEnvVars` and then overlays host `PATH`, `HOME`, `SHELL`, `USER`, `LANG`, and `TERM`. A collaborator configured with `scope: "none"` still receives owner host identity/env baseline values, and agent-provided `HOME` cannot win, contradicting the isolation contract this PR adds.
  - confidence: 0.91
- **[P2] Apply agent vars for default all scope:** `src/agents/pi-tools.ts:466-470`
  - body: `agentEnvVars` is only computed when `env.scope` is present and not `"all"`. If a config sets `env.vars` with omitted or explicit `all` scope, exec falls back to raw `process.env` and silently drops the agent-level overrides even though the new type comment says vars are always included.
  - confidence: 0.93
- **[P2] Allowlist values from the runtime environment:** `src/agents/agent-scope.ts:384-398`
  - body: `scope: "own"` filters only `collectConfigRuntimeEnvVars(cfg)`, which is config-defined env, not the actual shell or CI `process.env`. That means allowlisting runtime API keys will not work unless users duplicate those keys into config, undermining the stated controlled global inheritance mode.
  - confidence: 0.82
