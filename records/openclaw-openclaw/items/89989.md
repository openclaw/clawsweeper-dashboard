## Summary

Keep this PR open, but it is not merge-ready: the central mitigation is useful, yet the patch keeps one global retry timer, so mixed pending wake batches can still use the wrong retry delay, and the PR still lacks real channel behavior proof.

## What This Changes

The PR changes heartbeat wake retries so `requests-in-flight` busy skips wait 60 seconds while cron/lane busy skips keep the 1 second retry path, with wake and scheduler tests updated.

## Best Possible Solution

Preserve retry timing per queued wake target, confirm or adjust the 60 second policy, add mixed-order tests, and demonstrate real message flow during heartbeat contention before merge.

## Reproduction Assessment

Yes, at source level: current main routes retryable busy heartbeat skips through one shared 1 second wake retry path, and the PR changes that path. I did not run a live Telegram, Feishu, or WebChat stall reproduction in this read-only review.

## Solution Assessment

No. The 60 second delay is a plausible mitigation, but this is not the best implementation until retry timing is preserved per pending target and the fixed policy has live channel proof.
