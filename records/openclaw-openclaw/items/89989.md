## Summary

Keep open: current main and the latest release still have the aggressive shared 1 second heartbeat retry behavior, and this PR is the open candidate for the canonical bug, but it is not merge-ready because mixed pending wake batches still share one retry timer and real channel proof is missing.

## What This Changes

The PR changes heartbeat wake retries so `requests-in-flight` busy skips wait 60 seconds while cron/lane busy skips keep the 1 second retry path, with wake and scheduler tests updated.

## Best Possible Solution

Preserve retry timing per pending wake target, confirm the 60 second `requests-in-flight` policy, add mixed-order regression coverage, and require redacted live channel proof before merge.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.9 route retryable busy heartbeat skips through the shared 1 second wake retry path, and adjacent tests assert that cadence. I did not run a live Telegram, Feishu, or WebChat stall reproduction in this read-only review.

## Solution Assessment

No. The 60 second delay is a plausible narrow mitigation, but this is not the best implementation until retry timing is preserved per pending target and the policy has live channel proof.
