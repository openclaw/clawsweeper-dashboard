## Summary

Keep open: the PR targets a real heartbeat/session contention bug, but the current patch still has a mixed-batch retry timing defect and lacks after-fix real Telegram or equivalent behavior proof.

## What This Changes

The PR changes heartbeat wake retry scheduling so `requests-in-flight` busy skips retry after 60 seconds while cron-in-progress and lanes-busy skips keep the 1 second retry path, with focused wake and scheduler tests updated.

## Best Possible Solution

Preserve retry timing per pending wake target or retry bucket, keep the intended 60 second requests-in-flight policy only after maintainer acceptance, and add redacted real transport proof before merge.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 still route retryable busy heartbeat skips through the shared 1 second wake retry path, and the patch-level mixed-batch defect follows from one retry timer serving a whole pending batch. I did not establish a live Telegram, Feishu, or WebChat stall reproduction.

## Solution Assessment

No. A 60 second delay is a plausible mitigation, but this patch is not the best implementation until retry timing belongs to each queued wake target or equivalent mixed-batch logic and real channel proof is provided.
