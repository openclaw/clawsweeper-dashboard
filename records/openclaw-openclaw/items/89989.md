## Summary

Keep open: current main and v2026.6.10 still have the shared 1 second requests-in-flight retry, but this PR is not merge-ready because mixed pending wake batches can still use the first scheduled retry delay and real channel proof is missing.

## What This Changes

This PR changes heartbeat wake retries so requests-in-flight busy skips wait 60 seconds while cron/lane busy skips keep the 1 second retry path, with wake and scheduler tests updated.

## Best Possible Solution

Preserve retry timing per pending wake target, add mixed-order regression tests, confirm the 60 second policy, and add redacted live channel proof before merge.

## Reproduction Assessment

Yes at source level: current main and v2026.6.10 route requests-in-flight skips through the shared 1 second DEFAULT_RETRY_MS wake retry path. I did not run a live Telegram, Feishu, or WebChat stall reproduction in this read-only review.

## Solution Assessment

No. The 60 second delay is a plausible narrow mitigation, but this implementation is not the best fix until retry timing is preserved per pending wake target and the policy has live channel proof.
