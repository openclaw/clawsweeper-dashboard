## Summary

Keep open: this is a focused candidate for the heartbeat retry regression, but it is not merge-ready because the global retry timer can still mix delays for batched pending wakes and live channel proof is missing.

## What This Changes

The PR changes heartbeat wake retries so `requests-in-flight` busy skips wait 60 seconds while cron/lane busy skips keep the 1 second path, with wake and scheduler tests updated.

## Best Possible Solution

Preserve retry timing per pending wake target, confirm the 60 second `requests-in-flight` policy, add mixed-order regression coverage, and require redacted live channel proof before merge.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 route retryable busy heartbeat skips through the shared 1 second wake retry path, and the adjacent tests encode that cadence. I did not run a live Telegram, Feishu, or WebChat stall reproduction in this read-only review.

## Solution Assessment

No. A 60 second delay is a plausible narrow mitigation, but this implementation is not the best fix until retry timing is preserved per pending target and the policy has live channel proof.
