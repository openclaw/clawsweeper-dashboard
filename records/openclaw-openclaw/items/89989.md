## Summary

Keep open and not merge-ready: the PR addresses a real heartbeat/message-delivery bug, but the proposed scheduler change still uses one process-global retry timer, so mixed pending wake batches can lose the intended per-reason delay; the PR also still lacks after-fix live channel proof.

## What This Changes

The PR adds a 60 second retry delay for requests-in-flight heartbeat skips while keeping other retryable busy skips on the existing 1 second delay and updates focused heartbeat tests.

## Best Possible Solution

Keep the heartbeat debounce fix, but store retry timing per queued wake target or otherwise preserve 60s requests-in-flight retries and 1s cron/lane retries in mixed batches, then add redacted live channel proof before merge.

## Reproduction Assessment

Source-reproducible, but not live-reproduced here: current main delegates requests-in-flight skips to the wake layer, which retries with DEFAULT_RETRY_MS, matching the linked report's root cause. The PR author also did not run a live Telegram/Feishu/WebChat stall reproduction.

## Solution Assessment

No. A reason-specific delay is a plausible fix direction, but this patch is not the best implementation until retry delay is attached to the queued wake target or equivalent mixed-batch logic and proven in a real channel run.
