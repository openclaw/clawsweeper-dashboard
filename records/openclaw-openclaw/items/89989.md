- `P1`: The linked bug reports active channel conversations being blocked for minutes by heartbeat retry timing.

## Summary

Keep this PR open as the active candidate for the canonical heartbeat retry bug, but it is not merge-ready: the patch still uses one global retry timer for mixed pending wake batches and the PR has no real channel behavior proof.

## What This Changes

The PR changes heartbeat wake retries so `requests-in-flight` busy skips wait 60 seconds while cron/lane busy skips keep the 1 second retry path, with wake and scheduler tests updated.

## Best Possible Solution

Store retry due time per pending wake target, keep each busy reason's intended cadence, and land only after maintainers accept the 60 second policy with redacted live channel proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.9 route retryable busy heartbeat skips through the shared 1 second wake retry path, and adjacent tests assert that cadence. I did not run a live Telegram, Feishu, or WebChat stall reproduction in this read-only review.

## Solution Assessment

No. The 60 second delay is a plausible narrow mitigation, but this is not the best implementation until retry timing is preserved per pending target and the fixed policy has live channel proof.
