## Summary

Keep open: this PR is a focused candidate for the canonical heartbeat retry regression, but it is not merge-ready because mixed pending wake batches still share one retry timer and the PR lacks after-fix real channel proof.

## What This Changes

This PR changes heartbeat wake scheduling so `requests-in-flight` busy skips retry after 60 seconds while cron/lane-busy skips stay on the existing 1 second path, with wake and scheduler tests updated.

## Best Possible Solution

Preserve retry timing per pending wake target, keep the 60 second requests-in-flight policy only after maintainer confirmation, and require redacted live channel proof before merge.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 still route retryable busy heartbeat skips through the shared 1 second wake retry path. I did not establish a live Telegram, Feishu, or WebChat stall reproduction during this read-only review.

## Solution Assessment

No. The 60 second delay is a plausible mitigation, but this patch is not the best implementation until retry timing belongs to each queued wake target or equivalent mixed-batch logic and real channel proof is provided.
