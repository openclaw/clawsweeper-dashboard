- `P1`: The linked bug reports active channel conversations being blocked for minutes by heartbeat retry timing, which is an urgent real-user workflow regression.

## Summary

Keep open: this PR targets a valid heartbeat/session contention bug, but the patch still has a mixed-batch retry timing defect and lacks real Telegram, Feishu, WebChat, or equivalent after-fix proof.

## What This Changes

This PR makes `requests-in-flight` heartbeat busy skips retry after 60 seconds instead of the shared 1 second delay and updates heartbeat wake/scheduler tests.

## Best Possible Solution

Preserve retry timing per pending wake target, keep the 60 second requests-in-flight policy only after maintainer confirmation, and add redacted live channel proof before merge.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 still route retryable busy heartbeat skips through the shared 1 second wake retry path, and the patch-level mixed-batch defect follows from `schedule()` having one retry timer. I did not establish a live Telegram, Feishu, or WebChat stall reproduction.

## Solution Assessment

No. The 60 second delay is a plausible mitigation, but this patch is not the best implementation until retry timing belongs to each queued wake target or equivalent mixed-batch logic and real channel proof is provided.
