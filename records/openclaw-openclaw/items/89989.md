## Label Justifications

- `P1`: The linked bug reports active channel conversations being blocked for minutes by heartbeat retry timing, which is an urgent real-user workflow regression.
- `merge-risk: 🚨 message-delivery`: Incorrect heartbeat retry timing can continue blocking or delaying chat messages during active channel conversations.
- `merge-risk: 🚨 session-state`: The change alters when heartbeat runs acquire the same session/runtime locks as user turns, so a bad retry schedule can preserve session-lock contention.

## Summary

Keep this PR open for repair rather than cleanup-close: the narrow timing idea is useful and current main still has the 1 second requests-in-flight retry, but the patch keeps a single global retry timer so mixed busy-reason batches can still get the wrong delay, and the PR still lacks real channel behavior proof.

## What This Changes

This PR changes heartbeat wake retry timing so `requests-in-flight` skips wait 60 seconds while cron/lane-busy skips stay on the 1 second retry path, with wake and scheduler tests updated.

## Best Possible Solution

Preserve retry timing per queued wake target, keep the 60 second requests-in-flight policy only after maintainer confirmation, and add redacted live channel proof before merge.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 still route retryable busy heartbeat skips through the shared 1 second wake retry path, and the patch-level mixed-batch defect follows from `schedule()` having one retry timer. I did not establish a live Telegram, Feishu, or WebChat stall reproduction.

## Solution Assessment

No. The 60 second delay is a plausible mitigation, but this patch is not the best implementation until retry timing belongs to each queued wake target or equivalent mixed-batch logic and real channel proof is provided.
