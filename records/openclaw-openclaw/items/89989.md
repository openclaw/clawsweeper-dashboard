## Summary

Keep open: the PR targets a valid P1 heartbeat/session contention bug that is still present on current main and v2026.6.9, but the patch is not merge-ready because mixed pending wake batches can still lose the intended per-reason retry delay and the PR has no real channel behavior proof.

## What This Changes

The PR changes heartbeat wake retries so `requests-in-flight` skips wait 60 seconds while cron/lane busy skips keep the 1 second retry path, with wake and scheduler tests updated.

## Best Possible Solution

Preserve retry timing per pending wake target, confirm the 60 second requests-in-flight policy, add mixed-order coverage, and collect redacted live channel proof before merge.

## Reproduction Assessment

Yes at source level: current main and v2026.6.9 route retryable busy heartbeat skips through the shared 1 second wake retry path, and adjacent tests assert that cadence. I did not run a live Telegram, Feishu, or WebChat stall reproduction in this read-only review.

## Solution Assessment

No. The 60 second delay is a plausible narrow mitigation, but the implementation is not the best fix until retry timing is preserved per pending target and the timing policy has live channel proof.
