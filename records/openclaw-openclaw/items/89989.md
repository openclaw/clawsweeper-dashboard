## Summary

Keep this PR open, but it is not merge-ready: it targets a real heartbeat retry regression, yet the patch still applies one global retry timer to mixed pending wake batches and the contributor proof is source/test-only.

## Best Possible Solution

Preserve retry timing per pending wake target, confirm the 60-second requests-in-flight policy, add mixed-order tests, and require redacted live channel proof before merge.

## Reproduction Assessment

Yes at source level: current main routes retryable heartbeat busy skips through one shared 1-second wake retry path, and adjacent tests assert that cadence. I did not run a live Telegram, Feishu, or WebChat stall reproduction.

## Solution Assessment

No. A longer delay is a plausible mitigation, but this is not the best implementation until retry timing is preserved per pending wake target and the fixed policy has live channel proof.
