## Label Justifications

- `P1`: The linked bug reports active channel conversations being blocked for minutes by heartbeat retry timing, which is an urgent real-user workflow regression.
- `merge-risk: 🚨 message-delivery`: Incorrect heartbeat retry timing can continue blocking or delaying chat messages during active channel conversations.
- `merge-risk: 🚨 session-state`: The change alters when heartbeat runs acquire the same session/runtime locks as user turns, so a bad retry schedule can preserve session-lock contention.

## Summary

Keep open: this is a plausible fix for the linked P1 heartbeat/channel regression, but the patch is not merge-ready because mixed busy-reason batches can still inherit the wrong retry delay and the PR has no after-fix real channel proof.

## What This Changes

The PR changes heartbeat wake retries so `requests-in-flight` skips wait 60 seconds while other retryable busy skips keep the 1 second retry, with focused scheduler tests.

## Best Possible Solution

Keep the retry-delay direction, but store retry timing per queued wake target or select the next retry from per-target metadata, cover both mixed busy-reason orders, then require redacted live channel proof before merge.

## Reproduction Assessment

Yes, source-level: current main returns retryable busy skips and the wake layer retries them through the shared 1s retry path. I did not establish a live Telegram/Feishu/WebChat stall reproduction during this read-only review.

## Solution Assessment

No. The fixed 60s delay is a plausible mitigation, but this patch is not the best implementation until retry timing belongs to each queued wake target or equivalent mixed-batch logic and both mixed orders are covered.
