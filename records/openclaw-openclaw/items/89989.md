## Summary

Keep this PR open: it targets a valid canonical heartbeat regression, but the branch still has a blocking mixed-target retry timing bug and lacks real transport proof for the user-visible channel stall path.

## What This Changes

The PR adds a hard-coded 60-second retry delay for heartbeat `requests-in-flight` busy skips and updates wake/scheduler tests while leaving other retryable busy skips on the 1-second delay.

## Best Possible Solution

Store retry timing or due time per pending wake target, keep cron/lane-busy retries on the 1-second cadence, apply the accepted requests-in-flight cadence only to that target, cover mixed orders, and add redacted live channel proof before merge.

## Reproduction Assessment

Yes at source level: current main and v2026.6.9 route retryable heartbeat busy skips through the shared 1-second wake retry path, and the linked issue includes user-visible channel stall reports. I did not run a live Telegram, Feishu, or WebChat reproduction.

## Solution Assessment

No. A longer requests-in-flight delay is a plausible mitigation, but this implementation is not the best fix until retry timing is preserved per pending target and the 60-second policy has real channel proof.
