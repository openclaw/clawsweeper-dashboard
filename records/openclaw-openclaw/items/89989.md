## Summary

Keep open: the linked heartbeat/session contention bug is still present on current main and in v2026.6.8, and this PR is still the open narrow fix candidate, but the patch is not merge-ready because mixed pending wake batches can lose the intended per-reason retry delay and the contributor has not provided real channel proof.

## What This Changes

The branch adds a 60 second heartbeat wake retry delay for requests-in-flight skips and updates wake and scheduler tests while leaving other retryable busy skip reasons on the 1 second retry path.

## Best Possible Solution

Keep the retry-delay direction, but preserve retry timing per queued wake target, cover both mixed busy-reason orders, and require redacted live channel proof before merge.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 still route retryable busy heartbeat skips through the shared 1 second wake retry path. I did not establish a live Telegram, Feishu, or WebChat stall reproduction during this read-only review.

## Solution Assessment

No. The fixed 60 second delay is a plausible mitigation, but this patch is not the best implementation until retry timing belongs to each queued wake target or equivalent mixed-batch logic and real channel proof is provided.
