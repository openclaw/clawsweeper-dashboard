## Label Justifications

- `P1`: The linked bug reports active channel conversations being blocked for minutes by heartbeat retry timing, which is an urgent real-user workflow regression.

## Summary

Keep this PR open: it targets a valid P1 heartbeat/session contention bug, but the current patch still has a blocking mixed-batch retry-timing defect and lacks after-fix real channel proof.

## What This Changes

This PR changes heartbeat wake retries so `requests-in-flight` skips wait 60 seconds while cron-in-progress and lanes-busy retries stay on the 1 second delay, with wake and scheduler tests updated.

## Best Possible Solution

Keep the retry-delay direction, but preserve retry timing per queued wake target, cover both mixed busy-reason orders, and require redacted live channel proof before merge.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 still route retryable busy heartbeat skips through the shared 1 second wake retry path. I did not establish a live Telegram, Feishu, or WebChat stall reproduction during this read-only review.

## Solution Assessment

No. The fixed 60 second delay is a plausible mitigation, but this patch is not the best implementation until retry timing belongs to each queued wake target or equivalent mixed-batch logic and real channel proof is provided.
