## Summary

Keep open: the PR addresses a real source-reproducible heartbeat regression, but it is not merge-ready because the new per-reason retry delay is still governed by one global retry timer and the external PR only provides source/test proof, not real behavior proof.

## What This Changes

Adds a 60s retry delay for requests-in-flight heartbeat busy skips and updates heartbeat wake/scheduler tests while keeping 1s retries for other retryable busy skips.

## Best Possible Solution

Make the retry cooldown target-aware or otherwise preserve per-target busy-skip deadlines, then add real channel proof that active conversation messages are not blocked when a due heartbeat sees requests-in-flight.

## Reproduction Assessment

Yes, source-level: current main routes retryable busy heartbeat skips through one DEFAULT_RETRY_MS wake retry path, and the linked issue includes concrete channel symptoms. I did not establish a live Telegram/Feishu/WebChat reproduction during this read-only review.

## Solution Assessment

No, not as submitted: a reason-specific delay in the existing global retry timer is too narrow because mixed pending targets can inherit the wrong cooldown. The best fix keeps retry timing per target or per queued wake while preserving the drift fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve per-target retry delays in mixed batches:** `src/infra/heartbeat-wake.ts:256`
  - body: `schedule()` treats any existing retry timer as a hard minimum and returns while it is active, so calling it once per busy skip makes the first skipped target in `pendingBatch` control the retry delay for every queued target. In a mixed batch, cron/lanes-busy first means requests-in-flight still retries after 1s; requests-in-flight first means the other busy skips wait 60s, contradicting the PR's guarantee. Keep retry deadlines target-aware or otherwise split the queued retries so one skip reason cannot collapse or lengthen another target's cooldown.
  - confidence: 0.89
