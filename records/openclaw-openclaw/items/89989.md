## Summary

Keep open: the linked heartbeat regression is still present on current main, but this PR is not merge-ready because the single global retry timer can still apply the first skipped target's delay to a mixed retry batch and the branch has no live channel proof.

## What This Changes

The PR adds a fixed 60-second retry delay for `requests-in-flight` heartbeat busy skips and updates wake/scheduler tests while keeping other retryable busy reasons on the 1-second path.

## Best Possible Solution

Preserve retry timing per pending wake target, confirm the 60-second requests-in-flight policy, add mixed-order tests, and require redacted live channel proof before merge.

## Reproduction Assessment

Yes at source level: current main schedules retryable heartbeat busy skips through one shared 1-second wake retry path, and adjacent tests assert that cadence. I did not run a live Telegram, Feishu, or WebChat stall reproduction.

## Solution Assessment

No. The 60-second delay is a plausible mitigation, but this is not the best implementation until retry timing is preserved per pending target and the fixed policy has live channel proof.
