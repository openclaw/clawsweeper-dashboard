## Summary

Keep open: the linked WebChat overlap bug is valid, but this PR does not fully fix the current DOM/CSS path. The changed padding selectors still depend on a `has-copy` class that the production renderer does not emit, and the media query still matches coarse-primary hybrid devices instead of excluding all hover-capable hybrids.

## What This Changes

The PR changes `ui/src/styles/chat/grouped.css` to apply message action padding on hover and to make action icons always visible only under `hover: none` plus `pointer: coarse`.

## Best Possible Solution

Make the action-present padding selector match real rendered assistant bubbles, use a media query that preserves hover behavior on hybrid devices, and add focused WebChat regression coverage plus the required user-facing changelog entry.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: render an assistant message with copy/expand actions and the bubble gets `.chat-bubble-actions` but not `.has-copy`, so the PR's padding rules cannot match. I did not run live browser/device tests in this read-only review.

## Solution Assessment

No. A CSS-only change to `.chat-bubble.has-copy` is not sufficient while the renderer omits that class, and the touch-only media query should exclude hover-capable hybrids more directly.
