## Summary

Keep open: current main still lets `sessions_spawn.model` drive native subagent model selection without a requester-scoped model allowlist, so the central request is not implemented. The request is plausible as a cost/model-routing guardrail, but it adds a new config and policy surface that needs maintainer product judgment first.

## What This Changes

Add `agents.defaults.subagents.allowedModels` and `agents.list[].subagents.allowedModels` to constrain explicit native `sessions_spawn` model overrides by the dispatching agent.

## Best Possible Solution

Approve or reject the native policy shape first; if accepted, implement a requester-scoped allowlist with canonical provider/model normalization, enforce only explicit model overrides before child session creation, and document backward-compatible unset or wildcard behavior.

## Reproduction Assessment

Yes for the current source behavior, but not for the reported billing incident. Current code passes `sessions_spawn.model` into native model selection before configured defaults, and tests assert that explicit override is applied as a user-sourced model override.

## Solution Assessment

Unclear until maintainers choose the config policy. The proposed requester-scoped allowlist matches existing `allowAgents` and plugin `allowedModels` patterns, but the exact semantics should be decided before code work starts.
