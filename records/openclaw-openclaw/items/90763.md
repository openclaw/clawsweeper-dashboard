## Summary

Keep open: current `main` and the latest release still allow explicit native `sessions_spawn.model` overrides without a requester-scoped model allowlist. The request is credible as a model-routing and cost-control guardrail, but it adds a new canonical config/policy surface, so it needs maintainer product judgment before implementation.

## What This Changes

Request a native `agents.defaults/list[].subagents.allowedModels` allowlist so a requester agent cannot pass arbitrary `sessions_spawn.model` overrides.

## Best Possible Solution

Decide the native policy first; if accepted, implement a requester-scoped allowlist with canonical provider/model normalization before child session creation, keep unset/wildcard behavior backward-compatible, and document how it differs from `allowAgents` and plugin `allowedModels`.

## Reproduction Assessment

Yes for the source behavior: current `main` accepts `sessions_spawn.model`, resolves it before configured subagent defaults, and records explicit model input as a user-sourced override. The real billing incident was not replayed live.

## Solution Assessment

Unclear until maintainers choose the config policy. The proposed requester-scoped allowlist matches existing `allowAgents` and plugin `allowedModels` patterns, but the exact native semantics should be approved before implementation.
