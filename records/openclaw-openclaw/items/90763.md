## Summary

Keep open: current main and v2026.6.8 still lack a native requester-scoped `subagents.allowedModels` policy, while explicit `sessions_spawn.model` remains the first model-selection input. The request is a plausible model-routing and cost-control guardrail, but it adds a new config/default enforcement surface that needs maintainer product approval before implementation.

## What This Changes

Add a native `agents.defaults.subagents.allowedModels` and/or `agents.list[].subagents.allowedModels` config surface to restrict explicit `sessions_spawn.model` overrides by requester agent.

## Best Possible Solution

Maintainers should decide the native config contract; if approved, enforce a requester-scoped model allowlist before child session creation, reuse existing model alias normalization, preserve wildcard/unset compatibility, and document the upgrade behavior.

## Reproduction Assessment

Yes for the current source behavior, but not for the reported billing incident. Current main accepts an explicit `sessions_spawn.model` before configured subagent defaults and has no native `allowedModels` config surface to reject it.

## Solution Assessment

Unclear until maintainers choose the product policy. The requester-scoped allowlist matches existing `allowAgents` and plugin `allowedModels` patterns, but the native config semantics should be approved before code work starts.
