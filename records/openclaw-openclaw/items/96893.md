## Summary

This PR addresses a real unbounded provider-response read, but it is not merge-ready because the generic 16 MiB JSON cap can reject valid inline base64 image responses and the existing provider-http test mocks are not updated for the new import.

## What This Changes

This PR replaces three image-generation success-path `response.json()` reads with `readProviderJsonResponse` in the shared OpenAI-compatible factory, OpenAI direct provider, and Microsoft Foundry provider.

## Best Possible Solution

Keep the bounded-reader hardening, but pass an image-generation-sized JSON cap for inline base64 response paths and update the affected provider tests to cover both valid large inline payloads and oversized rejection.

## Reproduction Assessment

Yes at source level. Current main has raw `response.json()` reads on provider-controlled image-generation success bodies, and the PR body includes loopback `node:http` proof showing bounded reads stop oversized streamed bodies while raw `response.json()` buffers the full payload.

## Solution Assessment

No. Reusing the shared bounded reader is the right layer, but the PR should pass an image-appropriate cap and update the affected test harnesses before merge.
