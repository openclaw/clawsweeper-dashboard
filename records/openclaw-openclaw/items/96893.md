## Summary

Keep this PR open because the OOM hardening is real, but the proposed patch is not merge-ready: it applies the generic 16 MiB JSON cap to inline image payloads and leaves existing provider-http test mocks out of sync with the new import.

## What This Changes

This PR replaces three image-generation success-path `response.json()` reads with `readProviderJsonResponse` in the shared OpenAI-compatible factory, OpenAI direct image provider, and Microsoft Foundry image provider.

## Best Possible Solution

Keep the bounded-reader hardening, but size the JSON cap for inline image payloads/counts and update the affected tests before merging; use the broader related PR only as reference while it remains conflicting.

## Reproduction Assessment

Source-reproducible: current main still has the unbounded reads, and the PR-introduced cap/test-mock issues are visible from the changed imports, helper default, provider capabilities, and existing test mocks. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

No. Using the shared bounded reader is the right layer, but this PR should pass an image-appropriate cap and update the affected test harnesses before merge.
