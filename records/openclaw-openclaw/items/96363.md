## Label Justifications

- `P1`: A current-release Telegram direct-chat `/status` command can complete internally but drop the final user-visible reply when rich messages are enabled.
- `impact:message-loss`: The reported failure loses the final Telegram slash-command response after Telegram rejects the rich payload.

## Summary

Keep open: current main and the latest release still lack a Telegram rich-message email/entity guard or rich-send fallback for this `/status` failure, while https://github.com/openclaw/openclaw/pull/95900 is an open proof-positive candidate fix for the same behavior and should be reviewed or landed before closing this issue.

## What This Changes

Telegram `/status` should still deliver a reply when `channels.telegram.richMessages=true` and Telegram rejects an email-like rich-message entity.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/95900 or an equivalent central Telegram rich-message email guard with focused tests and live Telegram proof, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level and from reporter logs: `/status` status text can include OAuth profile/email labels, current rich delivery sends them through `sendRichMessage`, and non-quote rich-send failures propagate without a plain fallback. I did not run an independent live Telegram reproduction in this read-only pass.

## Solution Assessment

Yes: the best narrow fix is a shared Telegram rich-message builder guard for email-like tokens, as in the open candidate PR, because durable sends, drafts, and reply delivery all pass through that boundary. A broader plain-text fallback can be considered separately but is not required to keep this exact invalid email entity from dropping `/status`.
