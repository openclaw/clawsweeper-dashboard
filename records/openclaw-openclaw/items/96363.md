## Summary

Keep open: current main and v2026.6.10 still lack the Telegram rich-message email/entity guard or shared fallback for this `/status` delivery failure, and open candidate PRs already cover the fix path. The issue should stay open until https://github.com/openclaw/openclaw/pull/95900, https://github.com/openclaw/openclaw/pull/96376, or an equivalent central fix lands.

## What This Changes

The issue reports that Telegram `/status` final replies are dropped when `channels.telegram.richMessages = true` because `sendRichMessage` rejects an email-like rich entity with `RICH_MESSAGE_EMAIL_INVALID`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/95900 or an equivalent central Telegram rich-message email guard, or revise https://github.com/openclaw/openclaw/pull/96376 until the shared reply-delivery path is covered, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes at source level and from logs: `/status` reply text can include OAuth/profile email labels, current rich delivery sends that text through `sendRichMessage`, and non-quote rich-send failures propagate without a plain fallback. I did not run an independent live Telegram reproduction in this read-only review.

## Solution Assessment

Yes for the narrow fix direction: a shared Telegram rich-message builder guard for email-like tokens is the best owner-boundary fix because normal sends, drafts, and reply delivery already route through that boundary. A generic rich-invalid plain fallback may still be useful, but it should be shared rather than caller-specific.
