## Summary

Current main now fixes the reported Telegram `/status` rich-message email-entity drop through the merged shared rich-message builder guard from https://github.com/openclaw/openclaw/pull/95900; the fix is not proven in a stable release yet, so this is a current-main-only implementation close.

## What This Changes

Telegram `/status` replies with `channels.telegram.richMessages = true` can be dropped when Telegram rejects rich email entity detection with `RICH_MESSAGE_EMAIL_INVALID`.

## Best Possible Solution

Keep the merged shared rich-message email guard as the canonical fix; treat any generic rich-send validation fallback as separate follow-up work rather than keeping this issue open.

## Reproduction Assessment

Yes. The stable-release report and independent confirmation show the `/status` rich send failing with `RICH_MESSAGE_EMAIL_INVALID`, and the source path explains why provider-prefixed email-like labels reached `sendRichMessage`; current main should no longer reproduce after the guard.

## Solution Assessment

Yes. The merged shared rich-message builder guard is the narrowest maintainable fix for this exact failure because normal sends, drafts, and reply delivery all pass through it; a broader plain-text fallback can be evaluated separately.
