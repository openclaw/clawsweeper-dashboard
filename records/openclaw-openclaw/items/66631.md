## Summary

Keep this open: current main still routes Feishu topic sessions by `thread_id` but final topic/config reply delivery falls back to the inbound message id when `root_id` is absent, so the reported thread_id-only topic reply case is not fixed. There are related open Feishu anchor/session PRs, so this should stay in maintainer review rather than queueing another competing fix.

## What This Changes

The issue requests Feishu topic-group replies to resolve the topic root when Feishu supplies `thread_id` but no `root_id`, so bot replies stay in the existing topic.

## Best Possible Solution

Review, land, or explicitly replace the canonical Feishu anchor-routing fix so thread_id-only topic messages resolve a safe final reply anchor while preserving normal group replies, typing targets, reactions, and session keys.

## Reproduction Assessment

Yes at source level: a Feishu topic/config reply-in-thread message with `thread_id` and no `root_id` reaches the final reply-anchor code and falls back to the inbound message id. I did not run a live Feishu/Lark topic-group repro in this read-only review.

## Solution Assessment

No for current main: session routing now understands `thread_id`, but final reply delivery still lacks a root-anchor resolution for the thread_id-only topic case. The best path is to review or refresh the existing Feishu anchor-routing PR rather than start a competing automated fix.
