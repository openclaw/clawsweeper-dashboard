## Label Justifications

- `P2`: This is a normal-priority Feishu channel bug with bounded message-routing blast radius.
- `impact:message-loss`: Replies can land in a different native topic than the user expects, making the bot response easy to miss.
- `impact:session-state`: The same missing root anchor can split Feishu topic conversation state across different thread/session identifiers.

## Summary

Keep this issue open: current main still handles `thread_id` as a Feishu topic session key, but reply anchoring still falls back to the inbound `message_id` when `root_id` is absent, which matches the reported topic-splitting behavior.

## What This Changes

The issue requests that Feishu/Lark topic-group bot replies stay inside the existing native topic when events provide `thread_id` but omit `root_id`.

## Best Possible Solution

Keep this open and fix the Feishu plugin by resolving a native topic root from `thread_id` before reply-anchor selection, while preserving the P2P behavior from https://github.com/openclaw/openclaw/pull/92136 and normal group reply behavior.

## Reproduction Assessment

Yes at source level: a Feishu topic event with `thread_id` but no `root_id` routes by `thread_id`, then current main chooses the inbound `message_id` as the reply target. I did not run a live Feishu/Lark topic-group repro in this read-only review.

## Solution Assessment

Unclear until Feishu API proof is attached: the reporter's fetch-root direction is plausible and matches existing `listFeishuThreadMessages` infrastructure, but the final fix should reuse plugin-local helpers and preserve existing P2P and normal-group routing.
