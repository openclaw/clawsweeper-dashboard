## Summary

Keep open: current main still accepts `thread_id`-only Feishu topic-group events as topic sessions but normal and ACP reply dispatch fall back to `ctx.messageId`; the adjacent open Feishu PRs I checked do not yet land an exact `omt_*` topic container to `om_*` root-message lookup.

## What This Changes

The requested fix is for the Feishu plugin to reply inside the existing topic when a topic-group event has `thread_id` but no `root_id` or `parent_id`.

## Best Possible Solution

Resolve `omt_*` topic containers to the root `om_*` message inside the Feishu plugin before ACP failure, normal, and broadcast reply dispatch, with regression coverage that preserves normal-group reply targeting.

## Reproduction Assessment

Yes, source-level: a Feishu topic-group event with `thread_id: "omt_*"` and no `root_id`, `parent_id`, or `reply_target_message_id` under `group_topic`, `group_topic_sender`, or `replyInThread: "enabled"` resolves the reply target to `ctx.messageId` on current main. I did not run a live Feishu tenant replay.

## Solution Assessment

Yes. The narrow maintainable fix is plugin-local root resolution from the topic container before reply dispatch, reusing existing Feishu message/thread API seams and preserving documented normal-group safeguards.
