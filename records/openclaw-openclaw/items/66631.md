## Summary

Keep open: current main still routes Feishu topic-group `thread_id` events but does not resolve a `thread_id`-only topic event to the root `om_*` message before choosing reply targets. The related open PR #73972 is explicitly scoped to P2P direct-message threads and leaves this group-topic case out of scope.

## What This Changes

The issue asks Feishu topic-group bot replies with only `thread_id` to resolve the topic root message before replying, so replies stay in the existing topic.

## Best Possible Solution

Resolve `thread_id` to the root `om_*` message inside the Feishu plugin before ACP failure replies and normal/broadcast reply dispatch, with regression coverage that preserves normal-group #32980 behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Feishu group/topic event with `thread_id: "omt_*"` and no `root_id`, `parent_id`, or `reply_target_message_id` under `groupSessionScope: "group_topic"`, `groupSessionScope: "group_topic_sender"`, or `replyInThread: "enabled"`; current main selects `ctx.messageId` as the reply target.

## Solution Assessment

Yes. Resolving the topic root in the Feishu plugin before reply dispatch is the narrowest maintainable fix; it can reuse or extend the existing thread-list helper without changing core plugin contracts.

## Repair Work Prompt

Fix Feishu topic-group reply anchoring for this issue. Current main preserves `thread_id` for `group_topic`/`group_topic_sender` session keys and thread history, but when a group/topic event has `ctx.threadId` (`omt_*`) and no `ctx.rootId`, `ctx.parentId`, or `ctx.replyTargetMessageId`, `extensions/feishu/src/bot.ts` computes `replyTargetMessageId` as `ctx.messageId`; the configured-ACP init failure path has the same `ctx.rootId ?? ctx.messageId` fallback. Implement this inside the Feishu plugin only: resolve `ctx.threadId` to the root `om_*` message before both the ACP failure reply and normal/broadcast reply dispatcher creation, preferably by adding or reusing a `send.ts` thread-list/root helper that calls Feishu `im.message.list` with `container_id_type: "thread"`. Do not use `omt_*` as `replyToMessageId` or `MessageThreadId`. Preserve normal group behavior from #32980 and keep PR #73972's P2P scope separate. Add focused regression coverage for `thread_id`-only topic-group events under `groupSessionScope: "group_topic"`, `groupSessionScope: "group_topic_sender"`, and `replyInThread: "enabled"`; cover the ACP failure path if the existing lifecycle support makes that practical. Add a one-line user-facing changelog entry. Do not change core plugin contracts or unrelated Feishu routing behavior.
