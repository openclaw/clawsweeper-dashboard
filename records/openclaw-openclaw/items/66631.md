## Label Justifications

- `P2`: This is a bounded Feishu channel bug where replies can land in the wrong native topic, but it does not make the core runtime unusable.
- `impact:message-loss`: A bot reply that starts a different native topic can be missed by users following the original Feishu topic.
- `impact:session-state`: The same missing root anchor can split a topic conversation between the correct thread session and a reply thread rooted at the inbound message.

## Summary

Keep this issue open: current main and the latest release still route Feishu topic sessions by `thread_id` but choose the inbound `message_id` as the reply anchor when `root_id` is absent, which matches the reported topic-splitting behavior. The merged P2P thread fix explicitly left group/topic routing unchanged.

## What This Changes

The issue requests fixing Feishu topic-group replies so thread-id-only topic messages reply inside the existing native topic instead of creating a new topic.

## Best Possible Solution

Fix the Feishu plugin by resolving the native topic root from `thread_id` before reply-anchor selection, while preserving P2P direct-thread behavior, normal group routing, and the current topic session-key semantics.

## Reproduction Assessment

Yes at source level: a Feishu topic event with `thread_id` but no `root_id` routes as a topic session, then current main uses the inbound `message_id` as the reply target. I did not run a live Feishu/Lark topic-group repro in this read-only review.

## Solution Assessment

Unclear until live Feishu API proof is attached: the reporter's root-fetch direction is plausible and matches nearby thread-list infrastructure, but the final fix should reuse plugin-local helpers and preserve P2P and normal-group routing.
