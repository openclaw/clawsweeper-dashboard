## Summary

Keep open: current main still routes Feishu topic-group events with `thread_id` but no `root_id` into topic sessions, then falls back to the triggering `messageId` for reply dispatch. The linked P2P repair is explicitly out of scope, and the closest open root/thread PR is unmerged and does not resolve the missing topic-root lookup.

## What This Changes

Fix Feishu topic-group replies so rootless `thread_id` events reply inside the existing topic instead of creating a new topic.

## Best Possible Solution

Land one canonical Feishu plugin fix that resolves `omt_*` topic containers to the correct `om_*` reply anchor before ACP failure, normal, and broadcast reply dispatch while preserving normal-group `root_id` safeguards.

## Reproduction Assessment

Yes, at source level: a Feishu topic-group event with `thread_id`, no `root_id`, no `parent_id`, and no `reply_target_message_id` under topic-scoped routing resolves the reply target to `ctx.messageId`. I did not run a live Feishu tenant replay.

## Solution Assessment

Yes, the best direction is a narrow Feishu plugin fix that resolves the topic root before reply dispatch and reuses existing send/list/get helpers. It should not move channel-specific policy into core or break the normal-group safeguard that replies to the triggering message.
