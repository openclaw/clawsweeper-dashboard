## Summary

Keep open: current main still has a source-level path where Feishu topic-group events with `thread_id` but no `root_id`, `parent_id`, or `reply_target_message_id` are routed as thread/topic replies but use the triggering message as the reply target, matching the reported new-topic behavior.

## What This Changes

The issue asks the Feishu plugin to keep bot replies inside an existing topic when Feishu sends only `thread_id` for a topic-group message.

## Best Possible Solution

Resolve a Feishu topic `thread_id` to the topic root `om_*` message before ACP failure replies and normal or broadcast reply dispatch, while preserving the established normal-group `root_id` behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Feishu group/topic event with `thread_id: "omt_*"` and no `root_id`, `parent_id`, or `reply_target_message_id` under `groupSessionScope: "group_topic"` or configured `replyInThread`; current main selects `ctx.messageId` as the reply target.

## Solution Assessment

Yes. A Feishu-plugin root-resolution fix before reply dispatch is the narrow maintainable direction; it should reuse the existing thread-list/get-message helpers after checking the Feishu API contract.
