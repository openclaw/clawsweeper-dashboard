## Summary

Keep this open. Current main still preserves Feishu `thread_id` for topic session routing and thread-history lookup, but it does not resolve a `thread_id`-only topic event to the root `om_*` message before choosing outbound reply targets; both normal replies and the configured-ACP failure notice still fall back to the triggering message. The only open cross-reference, #73972, is explicitly scoped to P2P direct-message threads and says this topic-group case is out of scope.

## What This Changes

The issue asks Feishu topic-group replies to resolve `thread_id`-only topic events to the topic root message so bot replies stay in the existing topic instead of creating a new one.

## Best Possible Solution

Implement this in the Feishu plugin: when a topic-mode or explicit `replyInThread` group event has `ctx.threadId` but lacks a usable `ctx.rootId`, `ctx.replyTargetMessageId`, and `ctx.parentId`, resolve the `omt_*` thread container to the root `om_*` message before the ACP failure notice and normal/broadcast dispatchers are created. Preserve #32980 normal-group behavior, keep `omt_*` IDs out of reply-target fields, and add focused regression coverage for the `thread_id`-only topic event.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Feishu group/topic event with `thread_id: "omt_*"` and no `root_id`, `parent_id`, or `reply_target_message_id` under `groupSessionScope: "group_topic"` or `replyInThread: "enabled"`; current main selects `ctx.messageId` as `replyTargetMessageId`.

## Solution Assessment

Yes. Resolving the thread root inside the Feishu plugin before reply dispatch is the narrowest maintainable solution; core/plugin APIs do not need to change, and the fix can reuse or extend the existing thread-list helper with a bounded cache after verifying the Feishu API contract.
