## Summary

Keep this open. Current main still preserves Feishu `thread_id` for topic session routing, but it does not resolve a `thread_id`-only topic-group event to the root `om_*` message before choosing outbound reply targets; both normal replies and the configured-ACP failure notice can still fall back to the triggering message. The related open PR #73972 is explicitly scoped to P2P direct-message threads and says this topic-group case is out of scope.

## What This Changes

The issue requests fixing Feishu topic-group replies so `thread_id`-only events resolve the topic root `om_*` message before normal and ACP-failure replies choose their `replyToMessageId`.

## Best Possible Solution

Implement this inside the Feishu plugin by resolving `ctx.threadId` to the root `om_*` message when topic-mode or explicit `replyInThread` handling sees a group event with no usable `ctx.rootId`, `ctx.replyTargetMessageId`, or `ctx.parentId`. Apply that resolved root before both the ACP failure reply and normal/broadcast reply dispatcher creation, keep `omt_*` IDs out of reply-target fields, preserve normal-group #32980 behavior, and add focused regression coverage for the `thread_id`-only topic event.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Feishu group/topic event with `thread_id: "omt_*"` and no `root_id`, `parent_id`, or `reply_target_message_id` under `groupSessionScope: "group_topic"`/`"group_topic_sender"` or `replyInThread: "enabled"`; current main selects `ctx.messageId` as the reply target.

## Solution Assessment

Yes. Resolving the topic root inside the Feishu plugin before reply dispatch is the narrowest maintainable fix; it can reuse or extend the existing thread-list API helper without changing core plugin contracts.
