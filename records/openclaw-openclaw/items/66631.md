## Label Justifications

- `P2`: This is a bounded Feishu channel bug with visible topic reply misrouting but no core runtime outage.
- `impact:message-loss`: Replies can land in a different Feishu topic than the one the user is watching, making the bot response effectively missed.

## Summary

Keep this issue open: current main and the latest release still compute Feishu topic replies from `ctx.rootId ?? defaultReplyTargetMessageId`, so a topic event with `thread_id` but no `root_id` can still reply to the inbound message instead of the topic root. Several open PRs overlap the fix, but they are unmerged and need maintainer coordination rather than issue cleanup.

## What This Changes

The issue reports that Feishu topic-group bot replies for `thread_id`-only events create a new topic instead of replying inside the existing topic.

## Best Possible Solution

Land or repair a Feishu plugin fix that resolves the proper topic-root send anchor for `thread_id`-only topic events, preserves P2P and normal-group semantics, and then close this issue from the merged PR.

## Reproduction Assessment

Yes, at source level: a group-topic event with `thread_id` but no `root_id` keeps a topic session key but computes the outgoing reply target from the inbound message. I did not run a live Feishu/Lark replay in this read-only review.

## Solution Assessment

Unclear for the exact implementation: fetching or deriving the topic root is plausible, but the best fix should be a consolidated Feishu-local anchor repair that preserves existing P2P, normal-group, message-tool, and reaction behavior.
