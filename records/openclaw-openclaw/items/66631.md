## Summary

Keep this issue open: current main still routes Feishu topic sessions by `thread_id` but normal topic replies fall back to the inbound `message_id` when `root_id` is absent, matching the reported topic-splitting behavior. Open PRs overlap the fix, but none is merged and the active candidates still need maintainer handling.

## What This Changes

The issue reports that Feishu topic-group bot replies create new topics instead of staying in the existing topic when events include `thread_id` but no `root_id`.

## Best Possible Solution

Land a narrow Feishu plugin fix that resolves the native topic reply root for `thread_id`-only topic events, preserves P2P and normal-group behavior, and includes focused tests plus redacted Feishu/Lark proof.

## Reproduction Assessment

Yes, at source level: current main routes missing-`root_id` topic events by `thread_id`, then normal replies choose `ctx.rootId ?? defaultReplyTargetMessageId`, which becomes the inbound `message_id`. I did not run live Feishu/Lark proof in this read-only review.

## Solution Assessment

Unclear for the exact implementation: fetching or deriving the root from Feishu thread data is plausible and matches existing thread-list infrastructure, but maintainers should choose the narrow root-anchor fix that preserves existing P2P and normal-group semantics.
