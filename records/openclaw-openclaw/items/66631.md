## Summary

Keep open: current main still routes Feishu topic-group events that have `thread_id` but no `root_id` as topic sessions, then falls back to the triggering `message_id` for reply dispatch. The related open Feishu PRs cover adjacent reply-anchor problems, but I did not find an exact merged or open fix for resolving an `omt_*` topic container to its root `om_*` message before bot replies.

## What This Changes

Feishu topic-group messages with `thread_id` but no `root_id` should resolve the topic root before replying so bot replies stay in the existing topic.

## Best Possible Solution

Resolve `omt_*` topic containers to the root `om_*` message inside the Feishu plugin after admission and before ACP failure, normal, and broadcast reply dispatch, while preserving normal-group reply targeting.

## Reproduction Assessment

Yes, source-level: a Feishu topic-group event with `thread_id: "omt_*"` and no `root_id`, `parent_id`, or `reply_target_message_id` under `group_topic`, `group_topic_sender`, or `replyInThread: "enabled"` currently resolves the reply target to `ctx.messageId`. I did not run a live Feishu tenant replay.

## Solution Assessment

Yes. The narrow maintainable fix is plugin-local root resolution before reply dispatch, reusing the existing Feishu message/thread API seams and preserving documented normal-group safeguards.
