## Summary

Keep open: current main still routes Feishu `thread_id`-only topic-group events as topic sessions, but reply-target selection falls back to the triggering message, so the reported new-topic behavior can still occur. The related open PR is explicitly P2P-only and leaves this group-topic case out of scope.

## What This Changes

The issue requests resolving Feishu topic-group `thread_id` events without `root_id` to the topic root message before sending bot replies.

## Best Possible Solution

Resolve the Feishu `thread_id` to the root `om_*` message inside the Feishu plugin before ACP failure replies and normal or broadcast reply dispatch, while preserving normal-group reply targeting behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Feishu group/topic event with `thread_id: "omt_*"` and no `root_id`, `parent_id`, or `reply_target_message_id` under `groupSessionScope: "group_topic"`/`"group_topic_sender"` or `replyInThread: "enabled"`; current main selects `ctx.messageId` as the reply target.

## Solution Assessment

Yes. Resolving the topic root inside the Feishu plugin before reply dispatch is the narrowest maintainable fix; it should reuse or extend the existing thread-list helper after confirming the Feishu API contract.
