## Summary

Keep open: current main still routes Feishu topic-group events that have `thread_id` but no `root_id` as topic sessions, yet reply dispatch falls back to the triggering `message_id`, so the reported new-topic behavior remains source-reproducible.

## What This Changes

Resolve Feishu topic `thread_id`-only messages to the topic root message before sending bot replies so replies stay in the existing topic.

## Best Possible Solution

Resolve `omt_*` topic containers to the root `om_*` message inside the Feishu plugin after admission and before ACP failure, normal, and broadcast reply dispatch, without changing normal-group reply targeting.

## Reproduction Assessment

Yes, source-level: a Feishu group/topic event with `thread_id: "omt_*"` and no `root_id`, `parent_id`, or `reply_target_message_id` under `group_topic`, `group_topic_sender`, or `replyInThread: "enabled"` currently resolves the reply target to `ctx.messageId`. I did not run a live Feishu tenant replay.

## Solution Assessment

Yes. The narrow maintainable fix is to resolve the Feishu topic root in the Feishu plugin before reply dispatch, reusing the existing thread-list/get-message seams and preserving normal-group root/message targeting behavior.
