## Label Justifications

- `P2`: This is a real Feishu channel bug with a clear source path, but it is limited to configured Feishu topic-group reply behavior.
- `impact:message-loss`: Bot replies are delivered to a new topic instead of the existing topic, which is message misrouting.
- `impact:session-state`: The same Feishu topic can fragment into the wrong reply/session context when rootless topic events are handled.

## Summary

Keep open: current main still routes Feishu topic-group events that have `thread_id` but no `root_id` as topic sessions, then replies to the triggering `messageId` when no root or reply target is available. Several open Feishu PRs overlap this area, but none is merged or explicitly closes this issue.

## What This Changes

The issue requests that Feishu topic-group messages with a topic `thread_id` but no `root_id` reply inside the original topic instead of creating a new topic.

## Best Possible Solution

Land one canonical Feishu plugin fix that resolves `omt_*` topic containers to the correct `om_*` reply anchor before ACP failure, normal, and broadcast reply dispatch, while preserving normal-group `root_id` safeguards.

## Reproduction Assessment

Yes, at source level: a Feishu topic-group event with `thread_id: "omt_*"` and no `root_id`, `parent_id`, or `reply_target_message_id` under topic-scoped routing resolves the reply target to `ctx.messageId` on current main. I did not run a live Feishu tenant replay.

## Solution Assessment

Yes, the narrow maintainable solution is plugin-local root resolution from the topic container before reply dispatch. The implementation should preserve current normal-group safeguards and avoid broad pre-admission API lookups.
