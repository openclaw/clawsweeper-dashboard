## Summary

Keep open: current main still treats `thread_id`-only Feishu topic-group events as topic sessions, but reply dispatch falls back to `ctx.messageId` when `root_id` and `reply_target_message_id` are missing. I did not find a merged or linked PR that resolves the required `omt_*` topic container to the `om_*` root message before reply dispatch.

## What This Changes

The issue asks Feishu topic-group bot replies with only `thread_id` to stay in the existing topic instead of creating a new topic.

## Best Possible Solution

Resolve `omt_*` topic containers to the root `om_*` message inside the Feishu plugin before ACP failure, normal, and broadcast reply dispatch, with regression coverage that preserves normal-group reply targeting.

## Reproduction Assessment

Yes, at source level: a Feishu topic-group event with `thread_id: "omt_*"` and no `root_id`, `parent_id`, or `reply_target_message_id` under topic-scoped routing resolves the reply target to `ctx.messageId` on current main. I did not run a live Feishu tenant replay.

## Solution Assessment

Yes. The narrow maintainable fix is plugin-local root resolution from the topic container before reply dispatch, reusing existing Feishu message/thread API seams while preserving the documented normal-group safeguards.
