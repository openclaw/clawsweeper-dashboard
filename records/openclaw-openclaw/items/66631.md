## Summary

Keep open: current main and v2026.5.18 still route the reported Feishu `thread_id`-only topic event to a topic session, but reply dispatch falls back to the triggering `messageId` instead of resolving the `omt_*` topic container to the root `om_*` message. The related open P2P PR explicitly excludes this issue, and the nearby topic-session PR addresses session-key stability rather than this reply-target lookup.

## What This Changes

The issue asks OpenClaw to keep Feishu bot replies inside an existing topic when events include `thread_id` but omit `root_id` and `parent_id`.

## Best Possible Solution

Resolve `omt_*` topic containers to the root `om_*` message inside the Feishu plugin before ACP failure and normal reply dispatch, with regression coverage that preserves normal-group and P2P routing semantics.

## Reproduction Assessment

Yes, at source level: a Feishu topic-group event with `thread_id: "omt_*"` and no `root_id`, `parent_id`, or `reply_target_message_id` under topic-scoped routing resolves the reply target to `ctx.messageId` on current main. I did not run a live Feishu tenant replay.

## Solution Assessment

Yes. The maintainable fix is plugin-local root resolution from the Feishu topic container before reply dispatch, reusing existing Feishu message/thread API seams and keeping current normal-group safeguards intact.
