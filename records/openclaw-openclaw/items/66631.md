## Summary

Keep open: current main and v2026.6.6 still fall back to the inbound Feishu message ID as the visible reply target when a topic event has `thread_id` but no `root_id`, while several open Feishu PRs overlap the repair.

## What This Changes

Fix Feishu topic-group replies so thread-id-only topic messages reply within the existing topic instead of creating a new topic.

## Best Possible Solution

Review, land, or narrow the canonical Feishu anchor-routing fix so thread-id-only topic events resolve a safe reply root while preserving normal group and DM semantics.

## Reproduction Assessment

Yes at source level: a Feishu topic/config reply-in-thread event with `thread_id` and no `root_id` follows the current fallback to the inbound `messageId` as the reply target. I did not run a live Feishu/Lark topic-group repro during this read-only review.

## Solution Assessment

No for current main: it preserves thread-id session routing but not the visible reply root for the reporter's event shape. The best fix should stay in the Feishu plugin and keep session keys, typing/reaction anchors, and final reply anchors separate.
