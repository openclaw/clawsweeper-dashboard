## Summary

Keep this issue open: current main and v2026.6.10 still preserve Feishu topic sessions by `thread_id` while selecting `root_id` or the inbound `message_id` as the reply API target, so a thread-id-only topic event can still split visible replies. The strongest related fix candidate is still open and conflicting, so it is evidence of active work rather than proof that this issue can close.

## What This Changes

The issue requests a Feishu/Lark topic-group reply fix so bot replies stay inside the existing topic when Feishu sends `thread_id` but omits `root_id`.

## Best Possible Solution

Finish, repair, or replace the Feishu topic-anchor fix with a narrow plugin-owned change that resolves a sendable topic root for thread_id-only events while preserving the existing normal-group and P2P invariants.

## Reproduction Assessment

Yes, at source level: a Feishu topic event with `thread_id` but no `root_id` routes by `thread_id`, then replies through `ctx.rootId ?? message_id`. I did not run a live Feishu/Lark tenant reproduction in this read-only review.

## Solution Assessment

Unclear for the exact helper proposed in the issue. The best fix should resolve the missing topic send anchor near Feishu dispatch/outbound routing without globally redefining `rootId` or regressing normal-group and P2P behavior.
