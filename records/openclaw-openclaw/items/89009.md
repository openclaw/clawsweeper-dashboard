## Summary

Keep open: current main still lets a later `final` payload update the active Feishu streaming card before idle close, so the reported visible-final overwrite is still a valid delivery-integrity bug.

## What This Changes

The issue reports that Feishu streaming card delivery can generate the correct final answer but overwrite the visible card with a later final-shaped diagnostic before the stream is closed.

## Best Possible Solution

Add a Feishu dispatcher committed-final state so the first real streaming final owns the active card, exact duplicates are skipped, and distinct later final-shaped diagnostics are delivered outside that card.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: with Feishu streaming enabled, deliver final text A and then final-shaped text B before `onIdle`; current source lets B replace `streamText` on the active card. I did not execute a live Feishu repro in this read-only review.

## Solution Assessment

Yes, the suggested fix direction is the right owner boundary: handle the ordering/state in `extensions/feishu/src/reply-dispatcher.ts`, not by adding config or changing core dispatch semantics.
