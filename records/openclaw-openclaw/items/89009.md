## Summary

Keep open: current main and the latest v2026.6.6 release still allow a second pre-idle `final` payload to mutate the active Feishu streaming card, and the two linked closing PRs are still open rather than merged.

## What This Changes

The issue reports that Feishu streaming cards can lose the real final answer when a later final-shaped diagnostic reuses the same mutable card.

## Best Possible Solution

Land one Feishu-owned dispatcher fix that marks the first visible streaming final as committed, skips exact duplicates, closes or preserves that card before later diagnostics, and routes distinct later final-shaped diagnostics separately with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows that delivering final text A and then distinct final-shaped text B before `onIdle` lets B replace `streamText` on the active Feishu card.

## Solution Assessment

Yes. The best fix direction is a narrow Feishu dispatcher state repair, not a core dispatch change or new config option; the linked PRs still need maintainer review and proof before this issue can close.
