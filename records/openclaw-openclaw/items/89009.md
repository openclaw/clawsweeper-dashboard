## Summary

Keep open: current main only partially mitigates this Feishu final-delivery bug by appending `isError` finals into the active streaming card, while the requested committed-final separation is still owned by an open closing PR.

## What This Changes

The issue reports that Feishu `renderMode=auto` streaming can let a later final-shaped diagnostic mutate or overwrite the visible final answer card before the stream closes.

## Best Possible Solution

Land or replace the linked Feishu dispatcher fix so the first committed streaming final owns the active card, exact duplicates are skipped, and distinct later final-shaped diagnostics are delivered separately without mutating that card.

## Reproduction Assessment

Yes, at source level. The issue includes a concrete Feishu DM repro with live logs, and current dispatcher code still lets final-shaped payloads update the active streaming card before idle close except for a narrower `isError` append mitigation.

## Solution Assessment

Yes, the Feishu dispatcher is the right owner boundary for the fix. The current implementation is only partial; the best path is to finish the committed-final guard in the linked PR or a clean replacement and prove it with focused tests plus real Feishu evidence.
