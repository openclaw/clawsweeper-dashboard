## Summary

Keep open: this targets a real Telegram ingress stall, but the patch requeues the timed-out claim before the original handler has settled, so the same update can be retried while the old handler may still complete. The PR also still has only unit-test output, not real behavior proof from a Telegram/gateway/SQLite recovery path.

## What This Changes

The PR reduces the Telegram isolated ingress handler timeout, changes timed-out spooled claims from failed to pending retry with `last_error`, and updates related polling-session tests.

## Best Possible Solution

Keep the timeout recovery direction, but make the claim retryable only after the old handler has settled or after another explicitly safe terminal policy is chosen, then cover late completion and provide real recovery proof.

## Reproduction Assessment

Yes at source level. The PR requeues the claim before handler settlement, while the existing token-guarded cleanup will no-op after the row is pending, leaving the same update available for a later retry.

## Solution Assessment

No. Requeue-on-timeout is plausible, but the safer fix must preserve the old claim or a non-retryable terminal state until the original handler can no longer complete successfully.
