## Summary

Keep open: this is an active Telegram ingress recovery PR, but the current patch is not merge-ready because it releases the timed-out claimed update back to retry before the original handler has stopped, which can replay the same update while the old handler may still complete.

## What This Changes

The PR changes Telegram isolated-ingress timeout recovery from dead-lettering a timed-out claimed update to releasing it back to pending retry with `last_error`, reduces the backlog stall timeout, and updates Telegram spool tests for retry behavior.

## Best Possible Solution

Keep the timed-out update out of retry until the original handler has stopped, or move it into an explicit safe terminal state, then add late-completion regression coverage and redacted recovery proof.

## Reproduction Assessment

Yes, at source level. PR head releases the row before handler settlement, while current queue cleanup is token/status guarded, so a late original completion can leave the same update available for retry; I did not run a live Telegram timeout reproduction in this read-only review.

## Solution Assessment

No. Requeue-on-timeout is a plausible recovery direction, but this ordering is not the best fix because it exposes duplicate processing unless the old handler has stopped or the row is moved to an explicitly safe terminal state.
