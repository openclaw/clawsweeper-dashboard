## Summary

Keep open: this targets a real P1 Telegram ingress stall, but current head is not merge-ready because it makes a timed-out update retryable before the original handler has stopped and the supplied proof still does not show that recovery path in a real or fault-injected setup.

## What This Changes

The PR changes Telegram isolated-ingress timeout recovery to reduce the handler timeout, release timed-out claimed updates back to pending retry with `last_error`, and update Telegram spool tests for retry behavior.

## Best Possible Solution

Keep timed-out claims non-retryable until the original handler has stopped, or move them into an explicit safe terminal state, then add late-completion regression coverage and redacted SQLite/log recovery proof.

## Reproduction Assessment

Yes at source level: PR head releases the row before handler settlement, while current queue cleanup is token/status guarded, so a late original completion can leave the same update available for retry. I did not run a live Telegram timeout reproduction in this read-only review.

## Solution Assessment

No. Requeue-on-timeout is a plausible recovery direction, but this ordering is not the best fix because it exposes duplicate processing unless the old handler has stopped or the row is moved to an explicitly safe terminal state.
