## Summary

Keep open: this PR targets a real Telegram ingress stall, but the patch makes the timed-out update retryable before the original handler has stopped, so the same update can be processed twice; the updated proof also still does not show the stuck-claim recovery transition.

## What This Changes

The PR changes Telegram polling-session timeout recovery to shorten the spooled handler timeout, release timed-out claimed updates back to pending retry with `last_error`, and update Telegram spool tests for retry behavior.

## Best Possible Solution

Keep timed-out claims out of retry until the old handler has stopped, or move them into an explicit safe terminal state, then add late-completion regression coverage and redacted SQLite/log recovery proof.

## Reproduction Assessment

Yes, at source level. PR head releases the claim before handler settlement, while current queue cleanup is token/status guarded, so a late original handler completion can leave the same update available for retry.

## Solution Assessment

No. Requeue-on-timeout is plausible, but this ordering is not the best fix because it exposes duplicate processing unless the old handler has stopped or the row is moved to an explicitly safe terminal state.
