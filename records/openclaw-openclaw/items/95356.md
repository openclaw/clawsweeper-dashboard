## Summary

Keep open: this is a focused Telegram bug-fix attempt, but it is not merge-ready because the PR requeues the timed-out update before the original handler has settled, so a late successful completion can leave the same update pending for a later retry; the PR also still has only unit-test proof.

## What This Changes

The PR changes Telegram isolated ingress timeout recovery to release timed-out spooled update claims back to pending with `last_error`, lowers the default handler timeout to five minutes, and updates polling-session tests.

## Best Possible Solution

Keep the retry direction only if the timed-out claim remains non-retryable until the old handler has settled or is moved to a safe terminal state, with a late-completion regression test and real recovery proof.

## Reproduction Assessment

Yes, source-reproducible. The PR releases the claim before handler settlement, while the queue release makes the row pending and token-guarded late cleanup will not remove that pending row.

## Solution Assessment

No. Requeue-on-timeout is plausible, but the implementation releases too early; the safer narrow fix is to delay retry or use a safe terminal state until the old handler cannot complete successfully.
