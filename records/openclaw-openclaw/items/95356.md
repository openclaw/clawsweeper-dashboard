## Summary

Keep open: the PR still makes a timed-out Telegram update retryable before the original handler is known to have stopped, so a late successful handler can leave the same update pending for duplicate processing; contributor proof is still unit-test-only.

## What This Changes

The PR reduces the Telegram isolated ingress handler timeout, releases timed-out spooled claims back to pending retry with `last_error`, and updates Telegram polling-session/spool tests.

## Best Possible Solution

Keep the retry direction only if the timed-out claim remains non-retryable until the old handler has settled or an equivalent safe terminal state prevents duplicate handling, then add late-completion coverage and real recovery proof.

## Reproduction Assessment

Yes, at source level. The PR requeues the claim before handler settlement, while current queue cleanup is token-guarded and can leave the same update available for a later retry.

## Solution Assessment

No. Requeue-on-timeout is plausible, but the safer fix must preserve the claim or another non-retryable terminal state until the original handler can no longer complete successfully.
