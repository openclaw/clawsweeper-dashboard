## Summary

Keep open: this targets a real Telegram ingress stall, but the patch is not merge-ready because it can put a timed-out update back in pending while the original handler is still able to complete, allowing duplicate handling. The PR also lacks real after-fix Telegram or queue-runtime proof beyond unit-test output.

## What This Changes

The PR changes Telegram isolated ingress timeout recovery to use a shorter timeout, release timed-out spooled claims back to pending with `last_error`, and update the related tests.

## Best Possible Solution

Keep the timeout recovery, but only make the claim retryable after the old handler has actually stopped or after another explicitly safe terminal policy is chosen and covered by a late-completion regression test.

## Reproduction Assessment

Yes at source level: the PR requeues the claim before handler settlement, while the existing token-guarded cleanup will no-op after the row is pending, leaving it available for a later retry. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Requeue-on-timeout is a plausible direction, but the safest implementation must preserve the old claim or a non-retryable terminal state until the original handler can no longer complete successfully.
