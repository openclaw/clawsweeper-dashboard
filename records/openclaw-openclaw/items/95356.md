## Summary

Keep open: this is a focused Telegram bugfix, but it is not merge-ready because it requeues the timed-out update before the original handler is known to have stopped, creating a duplicate Telegram update handling risk, and the supplied proof still does not show the timeout recovery path.

## What This Changes

The PR reduces the Telegram spooled handler timeout from 25 minutes to 5 minutes, releases timed-out claimed updates back to pending retry with `last_error`, and updates Telegram polling/spool tests.

## Best Possible Solution

Keep the retry direction only after the timed-out claim remains non-retryable until the original handler settles, or after maintainers choose an explicit safe terminal state, then prove the recovery path with focused tests and redacted real or fault-injected evidence.

## Reproduction Assessment

Yes, at source level. The PR requeues the claim before handler settlement, while the shared queue delete path is token/status guarded, so a late original handler completion can leave the same update available for a retry.

## Solution Assessment

No. Requeue-on-timeout is plausible, but this ordering is not the best fix because it exposes duplicate processing unless the old handler has stopped or the row is moved to an explicitly safe terminal state.
