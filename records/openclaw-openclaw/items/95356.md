- `P1`: The PR targets a Telegram ingress stall that can block real inbound message delivery until operator recovery or restart.
- `merge-risk: 🚨 compatibility`: The diff changes the default spooled handler timeout from 25 minutes to 5 minutes, which can alter upgrade behavior for long-running Telegram work.

## Summary

Keep open: the PR targets a real P1 Telegram ingress stall, but it is not merge-ready because the patch makes the timed-out update retryable while its original handler can still complete, and the only after-fix proof is unit-test output. Current main and v2026.6.9 still use fail-on-timeout with the 25-minute default, so the requested retry behavior is not already implemented or superseded.

## What This Changes

The PR changes Telegram spooled-handler timeout recovery from dead-lettering to releasing the claim for retry with `last_error`, reduces the default handler timeout from 25 minutes to 5 minutes, and updates Telegram spool/polling tests.

## Best Possible Solution

Keep retry-on-timeout only after the timeout recovery path prevents duplicate processing, then require maintainer acceptance of the 5-minute default and redacted runtime or SQLite proof for the recovery path.

## Reproduction Assessment

Yes, source-reproducible. The PR releases the token-guarded claim to `pending` before the old handler settles, while successful cleanup only deletes a still-claimed matching token, so late completion can leave the same update pending for retry.

## Solution Assessment

No. Requeue-on-timeout is plausible, but this implementation makes retry visible too early; the safer fix keeps the claim non-retryable until handler settlement or uses an equivalent duplicate-prevention state.
