## Summary

Keep open: this PR targets a real P1 Telegram ingress stall, but the current patch can make a timed-out update retryable while its original handler may still finish, and the supplied proof is still unit-test-only.

## What This Changes

The PR changes Telegram isolated-ingress timeout recovery to release timed-out spooled claims back to pending with `last_error`, lowers the default handler timeout from 25 minutes to 5 minutes, and updates Telegram spool/polling tests.

## Best Possible Solution

Keep timeout retry only if the timed-out update cannot be retried until the original handler has stopped, or use an equivalent guarded state that prevents duplicate handling.

## Reproduction Assessment

Yes. Source inspection shows the PR releases the token-guarded claim to `pending` before the old handler settles, while successful cleanup only deletes a still-claimed matching token.

## Solution Assessment

No. Requeue-on-timeout is plausible, but this implementation makes retry visible too early; the safer fix keeps the claim non-retryable until handler settlement or uses another duplicate-prevention state.
