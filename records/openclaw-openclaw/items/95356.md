## Summary

Keep open: this is a plausible Telegram ingress recovery direction, but the current patch can make the timed-out update retryable while the original handler may still complete, creating a duplicate-processing path; contributor proof is still unit-test-only.

## What This Changes

The PR changes Telegram spooled-handler timeout recovery to release timed-out claims back to pending with `last_error`, shortens the default handler timeout from 25 minutes to 5 minutes, and updates Telegram spool/polling tests.

## Best Possible Solution

Keep the retry direction only after timeout recovery prevents duplicate processing, either by delaying retry until the original handler settles or by using an equivalent guarded state, then validate it with redacted runtime or SQLite proof and an explicit timeout-policy decision.

## Reproduction Assessment

Yes. Source inspection shows the PR releases the token-guarded claim to `pending` before the old handler settles, while successful cleanup only deletes a still-claimed matching token, so late completion can leave the same update pending for retry.

## Solution Assessment

No. Requeue-on-timeout is plausible, but this implementation makes retry visible too early; the safer fix keeps the claim non-retryable until handler settlement or uses another duplicate-prevention state.
