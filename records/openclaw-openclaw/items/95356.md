## Summary

Keep open: current main does not implement the linked Telegram stuck-claim recovery, but this PR is not safe as-is because it makes the timed-out update retryable before the original handler has stopped, creating a duplicate-processing path. It also still has only unit-test proof for a real Telegram/runtime recovery problem.

## What This Changes

The PR changes Telegram isolated-ingress timeout recovery to release timed-out spooled claims for retry with `last_error`, lowers the handler timeout, and updates Telegram polling/spool tests.

## Best Possible Solution

Keep the retry direction only if timeout recovery prevents duplicate processing, either by delaying retry until the original handler settles or by adding an equivalent guarded state, then validate the recovery with real runtime or SQLite proof.

## Reproduction Assessment

Yes, source-reproducible. PR head releases the token-guarded claim to `pending` before the old handler settles, while successful cleanup only deletes a still-claimed matching token; I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Requeue-on-timeout is plausible, but this implementation makes retry visible too early; the safer fix keeps the claim non-retryable until handler settlement or uses another duplicate-prevention state.
