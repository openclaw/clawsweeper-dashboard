## Summary

Keep open: the PR targets a real Telegram ingress stall, but the current branch is not merge-ready because it requeues a timed-out update while the original handler can still complete, creating duplicate Telegram update processing risk. The proof is unit-test-only, and the broader live-worker recovery policy remains tracked by https://github.com/openclaw/openclaw/issues/95248.

## What This Changes

The branch changes Telegram isolated-ingress timeout recovery to release timed-out spooled claims back to pending with `last_error`, reduces the default handler timeout from 25 minutes to 5 minutes, and updates Telegram spool/polling tests.

## Best Possible Solution

Keep the retry direction only if timeout recovery prevents duplicate processing, either by delaying retry until the original handler settles or by adding an equivalent guarded state, then validate the behavior with real runtime or SQLite proof.

## Reproduction Assessment

Yes, source-reproducible. The PR releases the token-guarded claim to `pending` before the old handler settles, while successful cleanup only deletes a still-claimed matching token, so late completion can leave the same update pending for retry.

## Solution Assessment

No. Requeue-on-timeout is plausible, but this implementation makes retry visible too early; the safer fix keeps the claim non-retryable until handler settlement or uses another duplicate-prevention state.
