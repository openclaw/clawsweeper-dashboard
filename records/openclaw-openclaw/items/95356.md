## Summary

Keep open: the PR addresses a real Telegram ingress stall, but the current patch makes the timed-out update retryable before the original handler is known to have stopped, so it can duplicate Telegram update handling.

## What This Changes

The PR reduces Telegram spooled-handler timeout from 25 minutes to 5 minutes, releases timed-out claims back to pending retry with `last_error`, and updates Telegram polling/spool tests.

## Best Possible Solution

Keep the retry direction only after the timed-out claim remains non-retryable until the original handler settles, or after maintainers choose an explicit safe terminal state, then prove the real recovery path.

## Reproduction Assessment

Yes, at source level. The PR requeues the claim before handler settlement, while current main's token-guarded delete will no-op after that row is pending, leaving the same update available for a later retry.

## Solution Assessment

No. Requeue-on-timeout is plausible, but this ordering is not the best fix because it exposes duplicate processing unless the old handler has stopped or the row is moved to an explicitly safe terminal state.
