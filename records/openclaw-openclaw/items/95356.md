## Summary

Keep open: this PR targets a real Telegram ingress stall, but the patch is not merge-ready because it makes a timed-out update retryable before the original handler has stopped, which can duplicate processing or stale same-lane state.

## What This Changes

The branch shortens Telegram isolated-ingress timeout recovery and changes timed-out claimed updates from failed/dead-lettered to pending retry with `last_error`, with updated Telegram spool tests.

## Best Possible Solution

Keep timed-out updates out of retry until the original handler has stopped or entered an explicit safe terminal state, then add late-completion regression coverage and redacted recovery proof.

## Reproduction Assessment

Yes, at source level. PR head releases the claimed row before handler settlement, while current queue cleanup is token/status guarded, so a late original completion can leave the same update available for retry.

## Solution Assessment

No. Requeue-on-timeout is a plausible recovery direction, but this ordering is not the best fix because it exposes duplicate processing unless the old handler has stopped or the row is moved to an explicit safe terminal state.
