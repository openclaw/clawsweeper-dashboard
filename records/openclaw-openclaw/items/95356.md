## Summary

Keep open: this PR still has a blocking Telegram delivery/session-state issue because it requeues a timed-out claimed update before proving the original handler has stopped, and the contributor proof does not show the stuck-claim recovery transition.

## What This Changes

The branch changes Telegram isolated-ingress timeout recovery to reduce the backlog stall timeout, release timed-out claimed updates back to pending retry with `last_error`, and update Telegram spool tests.

## Best Possible Solution

Keep the timed-out update out of retry until the original handler has stopped or is moved to an explicit safe terminal state, then add late-completion regression coverage and redacted recovery proof.

## Reproduction Assessment

Yes, at source level. PR head releases the row before handler settlement, while current queue cleanup is token/status guarded, so a late original completion can leave the same update available for retry.

## Solution Assessment

No. Requeue-on-timeout is a plausible recovery direction, but this ordering is not the best fix because it exposes duplicate processing unless the old handler has stopped or the row is moved to an explicit safe terminal state.
