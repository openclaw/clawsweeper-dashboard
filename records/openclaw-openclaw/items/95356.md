## Summary

Keep open: this PR targets a real P1 Telegram ingress stall, but it releases the durable claim back to pending before the original handler has stopped, so the same update can be retried while the old handler may still complete; the submitted proof also does not show that timeout-recovery transition in a real or fault-injected setup.

## What This Changes

The branch changes Telegram isolated-ingress timeout recovery to release timed-out spooled claims for retry with `last_error` and lowers the backlog stall timeout.

## Best Possible Solution

Keep the timed-out update out of retry until the original handler has stopped, or move it into an explicit safe terminal state, then add late-completion regression coverage and redacted recovery proof.

## Reproduction Assessment

Yes, at source level: PR head releases the row before handler settlement, while the current queue cleanup is token/status guarded, so a late original completion can leave the same update available for retry. I did not run a live Telegram timeout reproduction in this read-only review.

## Solution Assessment

No. Requeue-on-timeout is a plausible recovery direction, but this ordering is not the best fix because it exposes duplicate processing unless the old handler has stopped or the row is moved to an explicitly safe terminal state.
