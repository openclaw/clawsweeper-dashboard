## Summary

Keep this PR open: it targets a real P1 Telegram ingress stall, but the current patch can make the same timed-out update retryable before the original handler has stopped, and the contributor proof is unit-test-only.

## What This Changes

The branch changes Telegram spooled-handler timeout recovery to release timed-out claims back to pending with `last_error`, shortens the default handler timeout from 25 minutes to 5 minutes, and updates Telegram spool/polling tests.

## Best Possible Solution

Keep the retry direction only after timeout recovery prevents duplicate processing, either by delaying retry until the original handler settles or by using an equivalent guarded state, then validate the behavior with redacted runtime or SQLite proof and an explicit timeout-policy decision.

## Reproduction Assessment

Yes, source-reproducible. The PR releases the token-guarded claim to `pending` before the old handler settles, while successful cleanup only deletes a still-claimed matching token, so late completion can leave the same update pending for retry.

## Solution Assessment

No. Requeue-on-timeout is plausible, but this implementation makes retry visible too early; the safer fix keeps the claim non-retryable until handler settlement or uses another duplicate-prevention state.
