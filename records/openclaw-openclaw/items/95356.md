## Summary

Keep open: the PR targets a real P1 Telegram ingress stall, but the current patch makes the timed-out update retryable while its original handler may still complete, creating a duplicate-processing path. It also still has only unit-test output, not real Telegram/runtime/SQLite proof for the changed recovery behavior.

## What This Changes

The PR changes Telegram isolated-ingress timeout recovery to release timed-out spooled claims for retry with `last_error` instead of dead-lettering them, lowers the timeout from 25 minutes to 5 minutes, and updates Telegram polling/spool tests.

## Best Possible Solution

Keep the retry direction only if timeout recovery prevents duplicate processing, either by delaying retry until the original handler settles or by adding an equivalent guarded state, then validate the recovery with real runtime or SQLite proof.

## Reproduction Assessment

Yes, source-reproducible. The PR releases the token-guarded claim to `pending` before the old handler settles, while successful cleanup only deletes a still-claimed matching token, so late completion can leave the same update pending for retry.

## Solution Assessment

No. Requeue-on-timeout is plausible, but this implementation makes retry visible too early; the safer fix keeps the claim non-retryable until handler settlement or uses another duplicate-prevention state.
