## Summary

Keep open: current main and v2026.6.8 have meaningful fallback hardening, but the live discussion still shows interactive embedded runs bypassing model fallback under provider quota/auth failure, and current source still has supported paths where effective fallback availability becomes false despite configured defaults or related user/session override state.

## What This Changes

The issue reports that embedded interactive runs can retry or surface provider quota/auth/timeout failures instead of entering the configured model fallback chain, sometimes ending in EmbeddedAttemptSessionTakeoverError.

## Best Possible Solution

Normalize interactive embedded provider/auth/quota failures into the same model fallback path used by working background lanes, decide the exact user-selection fallback contract in the related policy issue, and keep the session-write takeover race tracked separately.

## Reproduction Assessment

Yes, source-reproducible with high confidence from current code and issue logs: fallbackConfigured can resolve false and block escalation, while the latest live comment shows the interactive lane bypassing fallback where a background lane succeeds. I did not run a fresh live quota or Telegram reproduction in this read-only review.

## Solution Assessment

Unclear as a single automatic fix: normalizing interactive provider failures into FailoverError is the right bug-fix direction for configured fallbacks, but changing explicit user-selection fallback semantics should be decided in the related policy thread before code changes land.
