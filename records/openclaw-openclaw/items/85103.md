- `impact:session-state`: The reported failure class can surface as `EmbeddedAttemptSessionTakeoverError` and can leave embedded session state inconsistent or blocked.

- `impact:message-loss`: Telegram and cron delivery reports show failed or delayed responses when fallback is skipped.

## Summary

Keep open. Current main has partial rate-limit and fallback hardening, but the issue remains necessary because current source still has fallback-source paths that can resolve to no fallbacks for embedded/session override runs, session-takeover errors still abort outer fallback, and later reports on 2026.6.1 describe primary-only attempts despite configured fallback chains.

## What This Changes

Configured model fallback chains should advance on provider quota, rate-limit, auth, and timeout failures instead of retrying the same model until an embedded session-takeover or single-candidate failure aborts the run.

## Best Possible Solution

Keep this issue as the canonical provider-failure fallback tracker, land narrow proof or fixes for fallback inheritance and Codex usage-limit handling, and keep the pure delivery-mirror session race on https://github.com/openclaw/openclaw/issues/84583.
