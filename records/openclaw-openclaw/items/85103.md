## Summary

Keep open. Current main has meaningful fallback hardening, but the live issue discussion still shows interactive embedded runs bypassing the model-fallback engine under provider quota/auth failure, and current source still has documented cases where model overrides resolve to an empty fallback override despite configured defaults.

## What This Changes

The issue requests that embedded and interactive runs escalate provider quota, auth, overload, and idle-timeout failures into the configured model fallback chain instead of retrying the same model or aborting with session takeover.

## Best Possible Solution

Normalize interactive embedded provider/auth/quota failures into the same FailoverError path used by background lanes, preserve documented fallback inheritance where it applies, and keep #84583 focused on local session-write races.

## Reproduction Assessment

Yes, source-reproducible with high confidence from current code plus issue logs, but not personally live-reproduced; fallbackConfigured can resolve false under explicit override paths and the latest comment demonstrates the interactive lane bypassing fallback while a background lane succeeds in the same process.

## Solution Assessment

No close yet; the best fix is a bounded owner-path repair in embedded fallback/error normalization and fallback inheritance, not closing under the existing partial rate-limit hardening or folding the provider-fallback problem into the separate #84583 session race.
