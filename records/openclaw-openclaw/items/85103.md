## Label Justifications

- `P1`: Recent reports show configured fallback chains failing for live interactive channel workflows when the primary provider is quota-exhausted or unavailable.
- `impact:message-loss`: Interactive channel replies can fail or send provider error text instead of retrying on a working configured fallback model.

- `impact:auth-provider`: The central failure is provider quota/auth errors not routing through configured model fallback choices.

## Summary

Keep open: current main and latest tagged release still have supported embedded paths where configured fallbacks are suppressed or auth/quota failures surface as plain errors, and recent issue comments show the interactive Telegram lane still bypassing model fallback while background lanes recover.

## What This Changes

Ensure embedded/interactive runs escalate provider-wide quota, auth, rate-limit, and timeout failures into configured model fallbacks instead of retrying or surfacing the primary model failure, while keeping session takeover symptoms separated.

## Best Possible Solution

Normalize interactive embedded provider/auth/quota failures into the same fallback path used by background lanes when configured fallbacks are eligible, decide explicit user-selection fallback semantics in the related policy issue, and leave session-write takeover races to their separate tracker.

## Reproduction Assessment

Yes for source-level current-main behavior and issue logs: fallbackConfigured can resolve false and block escalation, while the latest live comment shows the interactive lane bypassing fallback where a background lane succeeds; no fresh live quota run was performed.

## Solution Assessment

Unclear as a single automatic fix: normalizing provider/auth quota failures into FailoverError for fallback-eligible runs is the likely bug fix, but changing user or explicit model override strictness requires maintainer product decision.
