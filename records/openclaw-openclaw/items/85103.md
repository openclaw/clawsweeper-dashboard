The issue reports that embedded OpenAI Codex quota exhaustion retries the same model until a session takeover error instead of advancing through configured model fallbacks.

## Best Possible Solution

Add a focused repro for the OpenAI Codex `usage_limit_reached` embedded path, prove whether the configured chain is enabled for that session, and keep session-takeover handling separate from provider/model fallback.

No. The report has useful logs, but current main already has source and tests for the generic rate-limit fallback path, and the missing session/config/fallback-decision logs are needed to reproduce the exact failure.

## Solution Assessment

Unclear. The maintainable path is not another broad fallback layer; it is a narrow repro that proves whether `usage_limit_reached` bypasses classification, whether fallback was disabled by session policy, or whether the lock race is the real terminal failure.
