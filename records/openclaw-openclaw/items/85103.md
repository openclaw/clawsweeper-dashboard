## Label Justifications

- `P1`: A configured model fallback chain allegedly fails during provider quota exhaustion and can leave active agent sessions without a reply across multiple production reports.
- `impact:session-state`: The terminal `EmbeddedAttemptSessionTakeoverError` and reported need for `/new` involve session/transcript state coordination.
- `impact:auth-provider`: The core report is about provider quota classification, auth-profile rotation, and model fallback routing.
- `impact:message-loss`: The embedded run can fail before producing an assistant reply even though fallback models are configured.

## Summary

Keep open. Current main has generic rate-limit and overload fallback handling, but this recent report has multiple production signals for `usage_limit_reached`/overload retry loops ending in `EmbeddedAttemptSessionTakeoverError`, and the exact OpenAI Codex path is not proven fixed or duplicated by the older takeover/fallback issues.

## What This Changes

Issue reports that embedded runs keep retrying the same provider/model after provider quota or overload errors, then can terminate with `EmbeddedAttemptSessionTakeoverError` instead of using configured model fallbacks.

## Best Possible Solution

Capture a focused OpenAI Codex `usage_limit_reached` embedded trace with session entry, auth profile state, fallback source, and `model_fallback_decision`/trajectory records; fix the rate-limit handoff if it bypasses the configured chain, while routing the transcript-race side through https://github.com/openclaw/openclaw/issues/84583.

## Reproduction Assessment

No. The report has concrete production logs and current source shows the relevant fallback/session paths, but I do not have a high-confidence current-main reproduction without the session entry, auth profile state, configured fallback source, and full `model_fallback_decision` or trajectory logs.

## Solution Assessment

Unclear. A broad fallback rewrite would be risky; first prove whether `usage_limit_reached` is losing `lastRetryFailoverReason`, is blocked by strict session/agent selection policy, or is being preempted by the session takeover race.
