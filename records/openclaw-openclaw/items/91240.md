This PR updates Codex-harness `/status` auth labeling to consult a per-agent Codex app-server auth home before OpenAI env fallback, guards Codex `auth.json` API-key mode, and adds regression coverage.

## Best Possible Solution

Make `/status` auth labeling use the active Codex app-server home/transport source, and avoid reporting a local Codex OAuth label when the active app-server account is remote or unknown.

Yes, source-level reproduction is clear: current main routes OpenAI-on-Codex status labels through the OpenAI env path without the per-agent Codex home. The PR body also supplies after-fix terminal output for the default local Codex-home case.

## Solution Assessment

No, not yet: the default-case fix is the right layer, but hard-coding `agentDir/codex-home` is too narrow for supported explicit-home and WebSocket app-server configurations.
