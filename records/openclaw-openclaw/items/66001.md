## Label Justifications

- `P1`: The PR affects agent process lifecycle and can unexpectedly kill legitimate background exec jobs in real session-keyed agent flows.
- `merge-risk: 🚨 compatibility`: The new fail-closed reconciliation behavior changes existing process lifecycle semantics and can break existing background exec workflows.

- `merge-risk: 🚨 availability`: Unexpected cancellation of long-running process trees can interrupt active agent work and leave users with failed or missing background jobs.

## Summary

Keep open: the PR is aimed at a real process-supervisor gap, but it is not merge-ready because the new reconciliation predicate can misclassify valid session-key-owned exec runs as orphans and terminate them, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes the process supervisor from a no-op orphan reconciliation hook to a caller-supplied reconciliation pass, calls it from exec/process-tool paths, exports the new types, and adds focused tests.

## Best Possible Solution

Land a corrected in-process reconciliation mitigation only after it checks the same ownership identifiers the registry actually stores, adds session-key regression coverage, rebases cleanly, and leaves durable restart-orphan handling tracked by the linked issue.

## Reproduction Assessment

Yes, at source level: current exec runs can store `opts.sessionKey` in the supervisor record while the bash registry is keyed by generated `session.id`, and the PR's new predicate only checks that id map. I did not run live process proof because this review is read-only.

## Solution Assessment

No, not as written. The in-process reconciliation layer is a reasonable mitigation, but it must use session-key-aware ownership checks and provide real behavior proof before it is the best fix for this PR.
