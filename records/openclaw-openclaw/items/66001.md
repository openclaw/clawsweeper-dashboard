## Summary

Keep open: current main has not implemented orphan reconciliation, but this PR is not merge-ready because the new reconciliation predicate can cancel legitimate sessionKey-owned background runs, the branch is conflicting, and the PR body only lists tests rather than real behavior proof.

## What This Changes

The PR adds a `ProcessSupervisor.reconcileOrphans` API, calls it from exec/process tool paths, and adds tests for canceling untracked supervisor runs while preserving tracked ones.

## Best Possible Solution

Repair the PR so orphan reconciliation uses one consistent ownership key across supervisor records and bash process sessions, add sessionKey regression coverage plus real behavior proof, and keep durable restart kill-versus-reattach policy tracked in the canonical issue.

## Reproduction Assessment

Yes, at source level: start an exec with a non-empty `sessionKey`, then run reconciliation; the supervisor record is owned by the sessionKey while the bash registry is keyed by generated `session.id`, so `hasSession` returns false.

## Solution Assessment

No. In-process reconciliation is a plausible mitigation, but the current implementation must first use consistent owner keys and prove both orphan cancellation and sessionKey-owned run preservation; durable restart cleanup remains the canonical issue's product decision.

  - confidence: 0.95

  - body: `ReconcileOrphansParams.reason` accepts natural exit reasons like `exit`, `signal`, and `spawn-error`. If a caller passes one of those, a force-canceled orphan can be recorded as a natural exit, so this API should accept only cancellation reasons or omit the override until it is needed.
