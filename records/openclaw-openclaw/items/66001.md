## Summary

## What This Changes

## Best Possible Solution

Rebase on main, make reconciliation use canonical session ownership that preserves sessionKey-backed runs, add regression coverage, and provide redacted live runtime proof while leaving durable restart reaping tracked in the linked issue.

Yes, source-reproducible for the PR defect: `runExecProcess` stores registry sessions under `session.id` while supervisor records use `opts.sessionKey` when present, and the new checker only queries `runningSessions.has(id)`. I did not run live process proof in this read-only review.

## Solution Assessment

No as-is. In-process reconciliation is a reasonable partial mitigation, but it needs sessionKey-aware ownership, a rebase, and real behavior proof; durable restart orphan reaping remains a separate maintainer decision.

  - confidence: 0.95

  - body: `reconcileOrphans` always force-cancels active runs, but this exported type lets callers pass natural terminal reasons like `exit`, `signal`, or `spawn-error`. That would record a forced orphan cleanup as a natural exit and make process diagnostics misleading; the type should only allow forced-cancellation reasons.
