## Best Possible Solution

## Reproduction Assessment

Yes, source-level: a background exec with a non-empty `sessionKey` records supervisor ownership under that key while the bash registry is keyed by the generated session id, so the PR's `hasSession` check marks it untracked. I did not run a live process proof in this read-only review.

## Solution Assessment

No. In-process reconciliation is a plausible narrow mitigation, but this implementation needs a session-key-aware ownership contract and a current-main refresh before it is the best fix.

- **[P1] Preserve session-key-owned runs during reconciliation:** `src/agents/bash-process-registry.ts:97-100`
  - body: `hasSession` only checks the bash registry by generated session id, but `runExecProcess` stores supervisor ownership as `opts.sessionKey?.trim() || sessionId`. Any active run started with a non-empty `sessionKey` will be treated as untracked and cancelled on the next reconciliation, so the ownership check needs to recognize both the registry id and session-key-owned run before this can merge.
  - confidence: 0.94
- **[P3] Restrict orphan cancellation reasons:** `src/process/supervisor/types.ts:11-14`
  - body: `reconcileOrphans` always cancels active processes, but the new params type accepts natural terminal reasons such as `exit`, `signal`, and `spawn-error`. That lets a future caller record a forced orphan kill as a natural exit, so narrow this to cancellation/timeout reasons or remove the override.
  - confidence: 0.78
