## Summary

Keep open: this PR targets a real process/session lifecycle bug, but it is currently conflicting with main, lacks real behavior proof, and introduces a source-proven regression where session-keyed exec runs can be cancelled as false orphans.

## What This Changes

The branch adds a `ProcessSupervisor.reconcileOrphans` contract, calls it from exec/process tool paths, exports new supervisor types, and adds unit coverage for cancelling untracked runs while preserving tracked runs.

## Best Possible Solution

Keep this PR open as a useful partial fix, but require a rebase, session-key-aware reconciliation with regression coverage, and redacted real behavior proof; leave https://github.com/openclaw/openclaw/issues/65983 open for durable restart orphan policy.

## Reproduction Assessment

Yes, at source level: exec stores running sessions under a generated `session.id`, while supervisor records can store `opts.sessionKey` as `record.sessionId`; the new `hasSession(record.sessionId)` check returns false for those live runs. I did not run a live process proof in this read-only review.

## Solution Assessment

No. Reconciliation is a reasonable boundary, but the best fix must align the supervisor and bash registry ownership identifiers or pass a session-key-aware predicate, with coverage for session-key-owned runs; durable restart orphan reaping remains separate.
