## Label Justifications

- `P1`: The PR targets a high-impact process/session bug and the current patch can break real background exec workflows by cancelling valid runs.
- `merge-risk: 🚨 compatibility`: Existing users with sessionKey-owned background exec runs could see those runs cancelled after upgrade if this reconciliation ships unchanged.
- `merge-risk: 🚨 session-state`: The patch changes how supervisor run state is matched to bash process registry state and can mis-associate a valid sessionKey-owned run as orphaned.
- `merge-risk: 🚨 availability`: A false orphan classification can kill long-running background worker processes that users still depend on.

## Summary

This PR is still aimed at a real open process/session lifecycle bug, but it should not merge as-is because the new reconciliation path can cancel legitimate background exec runs that are owned by a non-empty sessionKey.

## What This Changes

The branch adds a ProcessSupervisor orphan-reconciliation API, calls it from exec/process-tool paths, exports its types, and adds focused supervisor/process-tool tests.

## Best Possible Solution

Keep the in-process orphan mitigation direction, but reconcile against the process session/run identity or a typed ownership shape that preserves sessionKey-owned runs, then rebase and prove the behavior in a real exec/process flow while leaving cold-start reaping tracked by the linked issue.

## Reproduction Assessment

Yes for the PR-introduced regression: current source shows supervisor records may store sessionKey in record.sessionId while the bash registry is keyed by process session slug, so the new hasSession check misclassifies valid runs. The original linked cold-start orphan leak still needs live reproduction and remains open.

## Solution Assessment

No. The mitigation is plausible, but the best fix must preserve the existing process session/run id contract and add sessionKey coverage before maintainers consider the branch mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Do not treat sessionKey-owned runs as orphans:** `src/agents/bash-process-registry.ts:100`
  - body: `runExecProcess` stores the supervisor run under the process session slug but sets the supervisor record's `sessionId` to `opts.sessionKey` when present. The new reconciliation passes that value to `hasSession`, which only checks the bash registry's slug-keyed `runningSessions`, so any valid sessionKey-owned background run can be cancelled on the next exec or process-tool call. Reconcile against the process session/run id or pass a typed ownership shape, and cover a non-empty `sessionKey`.
  - confidence: 0.93
- **[P3] Restrict orphan cancellation reasons:** `src/process/supervisor/types.ts:14`
  - body: `ReconcileOrphansParams.reason` accepts the full `TerminationReason` union, including natural outcomes like `exit`, `signal`, and `spawn-error`. That lets a forced orphan cancellation be recorded as a natural process outcome; narrow the type to forced-cancellation reasons or drop the option until there is a real caller.
  - confidence: 0.79
