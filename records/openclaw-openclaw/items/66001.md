## Label Justifications

- `P1`: The PR touches active exec/process lifecycle behavior and currently has a blocking path that can kill valid background jobs.
- `merge-risk: 🚨 compatibility`: Existing session-key-owned background exec workflows can stop working because the new reconciliation check uses a different ownership key.
- `merge-risk: 🚨 session-state`: The patch can misclassify a live supervisor run as orphaned when registry session ids and supervisor session keys diverge.
- `merge-risk: 🚨 availability`: The new cancellation path can terminate active long-running processes during normal process-tool or exec requests.

## Summary

Keep open. The PR targets a real process-supervisor cleanup gap, but the current patch can cancel legitimate session-key-owned background runs, is currently conflicting, and still lacks contributor real behavior proof.

## What This Changes

The PR adds `ProcessSupervisor.reconcileOrphans(...)`, calls it from exec/process paths, exports the new types, and adds focused orphan-cancellation tests.

## Best Possible Solution

Revise this PR or replace it with ownership-aware in-process reconciliation that preserves `sessionKey`-owned runs, fixes the mock/type issues, and leaves cold-start stale PTY policy tracked by the canonical issue until maintainers decide it.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the blocking PR regression: start a background exec with a non-empty `sessionKey`, then trigger reconciliation by starting another exec or invoking the process tool. The supervisor record is owned by `sessionKey`, while the PR predicate checks only generated registry ids.

## Solution Assessment

No. In-process reconciliation is a reasonable smaller fix direction, but this patch compares the wrong ownership key, leaves the cancellation reason contract too broad, and should not close the broader restart-orphan issue without maintainer scope agreement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use ownership-aware session checks:** `src/agents/bash-process-registry.ts:100-102`
  - body: `hasSession` only checks `runningSessions` by generated `ProcessSession.id`, but `runExecProcess` stores supervisor ownership as `opts.sessionKey?.trim() || sessionId`. A background exec launched with a non-empty `sessionKey` can be misclassified as orphaned and canceled the next time reconciliation runs.
  - confidence: 0.94
- **[P2] Update the exec-runtime supervisor mock:** `src/agents/bash-tools.exec-runtime.ts:557-559`
  - body: The unconditional `reconcileOrphans` call means `bash-tools.exec-runtime.test.ts` cases that mock `getProcessSupervisor()` with only `spawn` can throw before reaching the behavior under test. Add the mocked method and reset it with the existing supervisor mock.
  - confidence: 0.86
- **[P3] Restrict reconciliation reasons to forced cancels:** `src/process/supervisor/types.ts:14`
  - body: `ReconcileOrphansParams.reason` accepts natural terminal reasons such as `exit`, `signal`, and `spawn-error`, but reconciliation force-cancels live runs. Narrow the type so callers cannot record a killed orphan as a natural exit.
  - confidence: 0.8
