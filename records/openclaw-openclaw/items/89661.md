## Summary

Keep open. Current main still has a source-reproducible Control UI bug: Skills rows are rendered without stable keyed identity and the checkbox uses a normal property binding, so a user-clicked checked DOM state can carry to the next row after the toggled skill leaves a filtered tab.

## What This Changes

The issue reports that toggling a skill out of a filtered Skills tab leaves the next skill's toggle visually checked.

## Best Possible Solution

Land a narrow Control UI fix that keeps each skill toggle's checked state tied to the current skill identity or live DOM value, with a regression test for toggling the first disabled or ready skill out of a filtered list.

## Reproduction Assessment

Yes, from source: render a filtered Skills tab, click an unchecked row toggle, then let the skill leave the filtered list. The unkeyed list plus normal .checked binding can preserve the clicked DOM checkbox state for the next row; I did not run a live browser repro in this read-only review.

## Solution Assessment

Yes, a small UI rendering fix is the right shape. Prefer keying rows by skill.skillKey or using Lit's live checked binding, plus a regression test, without changing skill mutation or config semantics.
