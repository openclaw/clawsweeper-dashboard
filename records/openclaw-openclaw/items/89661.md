## Summary

Keep open. Current main still has a source-reproducible Skills UI row-state bug: toggling a skill out of a filtered tab can leave the reused checkbox DOM in the clicked state for the next skill.

## What This Changes

The issue reports that toggling a skill out of a Skills tab leaves the next row's toggle visually carrying the previous row's state.

## Best Possible Solution

Keep the fix in the Skills view: make list row or checkbox state follow `skill.skillKey` across filtered renders, then add a focused regression test for toggling a skill out of a filtered tab.

## Reproduction Assessment

Yes from source inspection: in a filtered Skills tab, toggling a row can remove that skill while Lit reuses the same checkbox DOM position for the next skill, and the checkbox property binding can skip resetting after the user click changed the DOM state.

## Solution Assessment

Yes. The narrowest maintainable fix is in `ui/src/ui/views/skills.ts`, by keying/remounting skill row identity or forcing the checkbox property to reflect the live skill state; backend skill status behavior does not need to change.

## Repair Work Prompt

Fix the Control dashboard Skills tab bug reported at https://github.com/openclaw/openclaw/issues/89661. Current main renders filtered skill rows in `ui/src/ui/views/skills.ts` without stable skill identity, and the checkbox `.checked` property can keep the clicked DOM state when the toggled skill leaves the current Ready/Disabled tab and the next skill reuses that row. Keep the repair in the Skills view layer; do not change backend `skills.update` semantics or add config. Prefer a small Lit-native fix that ties row or checkbox state to `skill.skillKey` across filtered renders, then add a focused regression test in `ui/src/ui/views/skills.test.ts` that renders two disabled skills, simulates the first toggle becoming checked, rerenders with the first skill removed, and asserts the next disabled skill's toggle is not checked. No CHANGELOG.md edit.
