The PR adds Telegram `network.httpTimeoutMs` handling/tests, changes transcript migration ID preservation, adds context-engine cache snapshot/restore, and adjusts WebChat tool-card output extraction, overflow CSS, and tests.

## Best Possible Solution

Split or replace this PR with narrow fixes: keep the current-main WebChat implementation for #77054, fix #77012 in the Codex transcript mirror idempotency path with regression coverage, and land Telegram/context-engine changes only with generated metadata, docs, changelog, and focused tests.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Fix the mirror dedupe path for transcript loss:** `src/config/sessions/transcript-append.ts:164-166`
  - body: This migration-only change does not address the reported WebChat transcript overwrite path. `mirrorCodexAppServerTranscript` still replays the full `messagesSnapshot` with a new `codex-app-server:${threadId}:${turnId}` scope each turn, and migration is skipped once parent-linked records exist, so previous messages can still be appended again under fresh keys. Please move the fix and regression coverage to the Codex mirror/idempotency path before using this PR to close that bug.
  - confidence: 0.9
- **[P2] Make the timeout tests observe aborts:** `extensions/telegram/src/probe.test.ts:310-319`
  - body: The new fake `fetch` promises only check `signal.aborted` at call time and never add an abort listener. `fetchWithTimeout` aborts the signal but relies on the fetch implementation to reject, so after advancing fake timers these `probePromise`s can stay pending and hang the test file. Use the abort-listener pattern from the existing timeout-budget test for both new mocks.
  - confidence: 0.94
- **[P2] Regenerate Telegram config metadata for the new key:** `src/config/zod-schema.providers-core.ts:279-287`
  - body: Adding `network.httpTimeoutMs` to the public Telegram config schema needs the generated channel metadata/docs baseline updated as well. Current generated metadata exposes the neighboring Telegram network keys but not this one, so config UI/docs surfaces will be stale and the repo's generated-config checks are expected to fail.
  - confidence: 0.86
