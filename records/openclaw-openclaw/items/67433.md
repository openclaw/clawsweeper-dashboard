## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve idempotency for wait-mode retries:** `src/gateway/server/hooks-request-handler.ts:318-320`
  - body: This disables the existing hook replay cache for `waitForResult` requests. If an HTTP client times out after the agent run starts and retries with the same idempotency key, the retry can launch a second agent run and duplicate side effects, which breaks the endpoint's retry guard exactly where synchronous callers need it most. Preserve/cache the wait-mode result or reject and document idempotency keys for wait mode.
  - confidence: 0.88
- **[P2] Add announceToMain to the replay scope:** `src/gateway/server/hooks-request-handler.ts:324-337`
  - body: `announceToMain` changes whether the hook result is posted into the resolved main session, but the async replay key still scopes only the older dispatch fields. Reusing an idempotency key with a different `announceToMain` value can return the cached run id while preserving the first request's announcement behavior, so this field needs to be part of the replay scope or otherwise rejected as a changed request.
  - confidence: 0.82
- **[P2] Document the new hook contract:** `src/gateway/hooks.ts:229-238`
  - body: These new fields and the expanded `{status, sessionKey, result, error}` response shape change the public `/hooks/agent` API, but the gateway configuration reference still documents only the old request fields and no result-return response. Please update the public docs with the default async behavior, wait-mode response, error/timeout behavior, and retry/idempotency contract.
  - confidence: 0.8
