## Summary

Keep open: current main still lacks a result-return mode for `POST /hooks/agent`, and this PR is the active implementation candidate for #49019. The latest diff appears to address the earlier error propagation and hook trust/sanitization concerns, but it should not merge as-is because `waitForResult` bypasses the established hook idempotency contract and the new public API contract is not documented or reflected in the changelog.

## What This Changes

This PR adds `waitForResult` and `announceToMain` fields to `POST /hooks/agent`, makes direct hook dispatch await isolated agent completion when requested, and returns status, session, result, or error metadata.

## Best Possible Solution

Ship a maintainer-approved `/hooks/agent` result-return contract that preserves authenticated hook idempotency, documents request/response/error/retry semantics, and covers waiting, failure, replay, and announcement behavior with focused tests.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection of current main shows the requested result-return mode is absent: payload normalization ignores `waitForResult`/`announceToMain`, and the handler returns only a run id.

## Solution Assessment

Unclear. Awaiting the isolated run is a plausible implementation and the branch appears to preserve hook trust protections, but bypassing established idempotency and shipping new response fields without docs/changelog leaves the public API contract unsettled.

  - body: The direct `/hooks/agent` path sets `replayKey` to `undefined` whenever `waitForResult` is true, so retrying a timed-out request with the same idempotency key starts a second agent run. Current hook delivery behavior dedupes retries by idempotency key; keep an in-flight or completed replay entry for wait-for-result requests instead of bypassing the cache.
  - confidence: 0.86

  - body: Fresh `/hooks/agent` responses now include `status` and `sessionKey`, but the replay branch still returns only `{ ok, runId }`. An idempotent retry should not observe a narrower response shape than the original call, so cache or reconstruct the added metadata with the replay entry.
  - confidence: 0.78

Please align docs, changelog, and inline contract before merging.
