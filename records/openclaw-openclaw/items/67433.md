## Summary

Keep open: current main still lacks the requested waitForResult mode, so this is a real implementation candidate, but the PR is not mergeable as-is because it would regress hook event owner-downgrade metadata and still needs API decisions around synchronous retry/error-announcement semantics.

## What This Changes

The PR adds waitForResult and announceToMain request fields to POST /hooks/agent, changes hook dispatch to optionally await an isolated agent turn and return result/status/sessionKey, updates hook test harness signatures, and declares Slack's typebox runtime dependency.

## Best Possible Solution

Ship a rebased and documented /hooks/agent result-return contract that preserves current hook trust metadata, explicitly supports or rejects idempotent synchronous retries, and keeps machine-only runs from leaking unwanted main-session artifacts.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new Gateway API mode. Source inspection of current main shows /hooks/agent is still async and runId-only, while the PR body supplies live after-fix proof for the branch behavior.

## Solution Assessment

No. Awaiting the isolated agent turn is a plausible implementation, but the patch should first preserve current hook authority metadata and settle retry, error-announcement, docs, and coverage expectations for the public API.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve hook event owner downgrades:** `src/gateway/server/hooks.ts:171-174`
  - body: Current main passes `forceSenderIsOwnerFalse: true` with hook system events so webhook-originated content cannot carry owner authority into the next run. The PR head's hook enqueue calls only set `trusted: false`, so keeping this branch version during rebase would regress `senderIsOwner`/`traceAuthorized` handling; add the owner-downgrade metadata to all hook wake/result/error system events.
  - confidence: 0.89
- **[P2] Honor announceToMain on thrown hook failures:** `src/gateway/server/hooks.ts:232`
  - body: In `waitForResult` mode, non-throwing results respect `announceToMain: false`, but thrown import/runtime failures still call `handleRunError`, which enqueues a `Hook ... (error)` system event and heartbeat. A machine callback that requested no main-session artifact can still inject an error artifact even though the HTTP response already carries the failure.
  - confidence: 0.86
- **[P2] Do not bypass idempotency for waitForResult:** `src/gateway/server/hooks-request-handler.ts:293-295`
  - body: Setting `replayKey` to `undefined` whenever `waitForResult` is true means a retry with the same idempotency key starts another agent run. Existing hook calls dedupe retries by idempotency key, and synchronous callers are likely to retry after network loss; store a replayable result or return a clear unsupported-idempotency error instead of duplicating side effects.
  - confidence: 0.84
- **[P3] Align the waitForResult response contract:** `src/gateway/hooks.ts:222-224`
  - body: The field comment promises a `text` response and cache-bypass behavior, while the handler returns `result` and public hook docs still list only the old async contract. Update the inline and public contract so API consumers do not implement against the wrong field name or retry behavior.
  - confidence: 0.9
