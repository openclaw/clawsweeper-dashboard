## Summary

Keep open: current main still lacks the requested result-return hook mode, and this active PR is the implementation path. The branch has useful live proof and no concrete security regression, but it is not merge-ready because the latest diff still silently bypasses existing hook idempotency for waitForResult and leaves the public API contract/docs inconsistent.

## What This Changes

The PR adds waitForResult and announceToMain handling to POST /hooks/agent, changes hook dispatch to return status/session/result data, updates hook-handler fixtures, and declares Slack's typebox dependency.

## Best Possible Solution

Ship a documented opt-in /hooks/agent result-return contract with explicit idempotency semantics, focused regression tests, and preserved hook trust/session protections.

## Reproduction Assessment

Not applicable; this is a feature/API PR, not a bug report. Source inspection of current main confirms /hooks/agent is still async and runId-only, while the PR branch adds the requested wait path.

## Solution Assessment

No. Awaiting the isolated run is a plausible implementation direction, but silently bypassing idempotent retries and exposing mismatched docs/comments is not the narrowest maintainable API shape yet.
