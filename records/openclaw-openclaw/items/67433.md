## Summary

Keep open: current `main` still lacks the requested result-return hook mode, and this PR is the active implementation path, but the current patch is not merge-ready because it regresses current hook authority metadata and leaves retry/announcement/docs semantics unresolved.

## What This Changes

The PR adds `waitForResult` and `announceToMain` handling to `POST /hooks/agent`, changes hook dispatch/response shapes, updates hook test harnesses, and declares Slack's `typebox` runtime dependency.

## Best Possible Solution

Ship a rebased, documented `/hooks/agent` result-return contract that preserves current hook authority metadata, defines retry/error/timeout behavior, and keeps machine-only runs from creating unwanted session artifacts.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new Gateway API mode. Source and release inspection show current `main` and v2026.5.12 still expose only async, runId-returning `/hooks/agent` behavior, while the PR body supplies real after-fix branch proof.

## Solution Assessment

No, not yet. Awaiting the isolated agent turn is a plausible direction, but this branch must preserve owner-downgrade metadata, define retry semantics, suppress thrown-error artifacts when requested, and document the public contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve hook event owner downgrades:** `src/gateway/server/hooks.ts:171-174`
  - body: Current `main` passes `forceSenderIsOwnerFalse: true` on hook system events so webhook-originated artifacts cannot preserve owner authority in later prompts. The PR's new result/error enqueue paths only set `trusted: false`, which drops the structured downgrade metadata current hook trust tests protect.
  - confidence: 0.91
- **[P2] Honor announceToMain on thrown hook failures:** `src/gateway/server/hooks.ts:232`
  - body: `waitForResult` non-throwing results respect `announceToMain: false`, but the catch path still calls `handleRunError`, which enqueues a hook error system event and heartbeat. Machine callbacks that requested no main-session artifact can still inject one when import/runtime failures throw.
  - confidence: 0.86
- **[P2] Do not bypass idempotency for waitForResult:** `src/gateway/server/hooks-request-handler.ts:293-295`
  - body: Setting `replayKey` to `undefined` for every `waitForResult` request means a retry with the same idempotency key starts another agent run. Existing hook calls dedupe retries by idempotency key, and synchronous callers are especially likely to retry after a lost response, so store/replay the result or reject idempotency explicitly instead of duplicating side effects.
  - confidence: 0.84
- **[P3] Align the waitForResult response contract:** `src/gateway/hooks.ts:222-225`
  - body: The field comment promises a `text` response and cache-bypass behavior, while the handler returns `result` and the public hook docs still describe only the old async contract. Update the code comment and docs so API consumers implement against the shipped response and retry semantics.
  - confidence: 0.88
