## Summary

Keep open: current main still lacks a result-return mode for `/hooks/agent`, and the latest branch preserves the recent hook trust/routing protections, but it still needs maintainer API judgment and a blocking-mode idempotency fix before merge.

## What This Changes

The PR adds a `blocking` flag to agent hook payloads, makes direct `/hooks/agent` calls await isolated agent completion when requested, returns text or errors in HTTP responses, and adjusts hook run-id idempotency handling.

## Best Possible Solution

Ship a maintainer-approved result-return contract for `/hooks/agent` that preserves hook event isolation, keeps retry/idempotency semantics explicit, and updates tests, docs, and changelog with the accepted behavior.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection of current main shows the requested blocking/result-return capability is absent: payload normalization ignores it and the handler responds with runId only.

## Solution Assessment

Unclear. Awaiting the isolated run is a plausible implementation, but the `blocking`/`text`/500/idempotency-bypass contract should be settled against #49019 before this becomes the shipped API.

Full review comments:

- **[P2] Keep hook idempotency active in blocking mode:** `src/gateway/server/hooks-request-handler.ts:290-293`
  - body: Existing `/hooks/agent` idempotency prevents duplicate agent runs on retries, but this branch disables the replay key whenever `blocking` is true. If a client times out and retries the same blocking request with the same key, the gateway starts another isolated run and can repeat side effects; keep an in-flight/completed replay entry for blocking requests instead of bypassing the cache.
  - confidence: 0.8

Please document the accepted `blocking`, `text`/error, timeout, and idempotency behavior and add the required user-facing changelog entry.
