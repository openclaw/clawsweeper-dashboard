## Summary

Keep open: this is the active implementation for the still-missing result-return hook feature, but the current PR head is stale/conflicting and has correctness and security blockers around hook authority metadata, retry semantics, and the public API contract.

## What This Changes

The PR adds `waitForResult` and `announceToMain` fields to `POST /hooks/agent`, changes hook dispatch to optionally await an isolated agent turn and return status/session/result data, updates hook test harnesses, and includes a Slack `typebox` dependency update.

## Best Possible Solution

Ship a rebased and documented `/hooks/agent` result-return contract that preserves hook owner-downgrade metadata, defines idempotent retry/error/timeout behavior, and keeps machine-only runs from producing unwanted main-session artifacts.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new Gateway API mode. Source inspection confirms current main is still async and runId-only, while the PR body supplies real after-fix proof for the proposed branch behavior.

## Solution Assessment

No, not yet. Awaiting the isolated agent turn is a plausible implementation, but the patch needs a rebase plus explicit fixes for owner-downgrade metadata, idempotent retries, thrown-error announcement suppression, tests, and docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve hook event owner downgrades:** `src/gateway/server/hooks.ts:171-174`
  - body: Current main passes `forceSenderIsOwnerFalse: true` with hook system events so webhook-originated content cannot carry owner authority into the next run. The PR head's result/error enqueue calls only set `trusted: false`, so rebasing this branch as-is would regress the structured owner-downgrade contract and fail the current hook trust coverage.
  - confidence: 0.9
- **[P2] Honor announceToMain on thrown hook failures:** `src/gateway/server/hooks.ts:232`
  - body: In `waitForResult` mode, non-throwing results respect `announceToMain: false`, but thrown import/runtime failures still call `handleRunError`, which enqueues a `Hook ... (error)` system event and heartbeat. Machine callbacks that requested no main-session artifact can still inject an error artifact even though the HTTP response already carries the failure.
  - confidence: 0.86
- **[P2] Do not bypass idempotency for waitForResult:** `src/gateway/server/hooks-request-handler.ts:293-295`
  - body: Setting `replayKey` to `undefined` for every `waitForResult` request means a retry with the same idempotency key starts another agent run. Existing hook calls dedupe retries by idempotency key, and synchronous callers are likely to retry after a lost response; store/replay the result or reject idempotency clearly instead of duplicating side effects.
  - confidence: 0.84
- **[P3] Align the waitForResult response contract:** `src/gateway/hooks.ts:221-225`
  - body: The inline field comment promises a `text` response and cache-bypass behavior, while the handler returns `result` and the public hook docs still describe only the old async contract. Update the code comment and docs so API consumers do not implement against the wrong response field or retry behavior.
  - confidence: 0.88
