## Summary

Keep open. This is a useful implementation candidate with supplied real behavior proof, but the current patch still has correctness/API blockers around `announceToMain:false` on thrown failures and `waitForResult` retry semantics; current `main` does not implement the requested result-return mode.

## What This Changes

The PR adds `waitForResult` and `announceToMain` handling to `/hooks/agent`, changes hook dispatch to optionally await isolated agent turns and return status/sessionKey/result, updates handler mocks, and declares Slack's `typebox` runtime dependency.

## Best Possible Solution

Ship a documented `/hooks/agent` result-return contract that preserves hook security boundaries, defines idempotent retry/error/timeout semantics, honors `announceToMain:false` for all completion and failure paths, and includes focused Gateway tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature PR. Source inspection shows current `main` is still async and runId-only, while the PR source-proves the remaining review findings.

## Solution Assessment

No. Awaiting the isolated agent turn is a plausible implementation direction, but the patch should first settle idempotent retry semantics, honor announcement suppression for thrown failures, align the public/inline contract, and add regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor announceToMain on thrown hook failures:** `src/gateway/server/hooks.ts:232`
  - body: In `waitForResult` mode, non-throwing results respect `announceToMain:false`, but thrown/import/runtime failures still call `handleRunError`, which enqueues a `Hook ... (error)` system event and heartbeat. A machine callback that requested no main-session artifact still gets one even though the HTTP response already carries the failure.
  - confidence: 0.9
- **[P2] Preserve waitForResult idempotency semantics:** `src/gateway/server/hooks-request-handler.ts:293-295`
  - body: The direct `/hooks/agent` path disables the replay key whenever `waitForResult` is true, so retrying a timed-out request with the same idempotency key starts another agent run. Existing `/hooks/agent` calls dedupe by idempotency key, and result-returning callers are especially likely to retry after network loss, so this can duplicate side effects unless the API stores a replayable result or explicitly rejects idempotency for this mode.
  - confidence: 0.87
- **[P3] Align the waitForResult response contract:** `src/gateway/hooks.ts:222-224`
  - body: The new field comment promises a `text` response field, while the handler sends `result` and the public hook docs still list only the old request/response contract. Align the inline and public contract so API consumers do not implement against the wrong field name or retry behavior.
  - confidence: 0.9
