## Summary

Keep open. Current main still lacks a blocking/result-return mode for `/hooks/agent`, and this PR is a live implementation candidate tied to the broader open API tracker #49019, but it should not merge as-is because the branch is heavily diverged and must preserve current hook routing, announcement, trust, docs, and changelog contracts during rebase.

## What This Changes

The PR adds an optional `blocking` field to agent hook payloads, makes `/hooks/agent` await dispatch and return text or errors, adjusts idempotency handling, and adds Gateway hook tests.

## Best Possible Solution

Rebase the implementation onto current `src/gateway/server/hooks-request-handler.ts` and `src/gateway/server/hooks.ts`, preserve current hook security and announcement policies, document and changelog the accepted response contract, and align the field names with the maintainer decision for #49019 before merge.

## Reproduction Assessment

Yes. Against current main, a `POST /hooks/agent` request with `blocking: true` is normalized without that field, dispatched fire-and-forget, and returns only `{ ok: true, runId }`; the evidence is from source and docs inspection, not a live Gateway run.

## Solution Assessment

Unclear as currently proposed. Awaiting the isolated agent run and returning text is a plausible narrow implementation, but the branch is not the best merge path until it is rebased and the API, idempotency, timeout, docs, and hook-announcement semantics are settled.
