## Summary

Keep open: current main still lacks the requested Gateway result-return mode, and the active PR has concrete API-contract issues plus a live conflict that need maintainer review before merge.

## What This Changes

The PR adds a synchronous `waitForResult` response mode and `announceToMain` suppression for `POST /hooks/agent`, updates hook dispatch fixtures, and declares Slack's `typebox` runtime dependency.

## Best Possible Solution

Ship a documented `/hooks/agent` result-return contract after settling retry and failure-announcement semantics, adding focused Gateway coverage, and dropping the redundant Slack dependency diff during rebase.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new Gateway API mode. Source and docs inspection of current main show `/hooks/agent` is still async and runId-only today, while the PR body supplies live proof for the branch behavior.

## Solution Assessment

No. Awaiting the isolated agent turn is a plausible implementation, but the PR should first make announcement suppression consistent, explicitly preserve or reject idempotent retries, and align inline/public docs with the actual `result` response contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor announceToMain on thrown hook failures:** `src/gateway/server/hooks.ts:232`
  - body: In `waitForResult` mode, non-throwing results respect `announceToMain: false`, but thrown/import/runtime failures still call `handleRunError`, which enqueues a `Hook ... (error)` system event and heartbeat. A machine callback that requested no main-session artifact can still inject an error artifact even though the HTTP response already carries the failure.
  - confidence: 0.88
- **[P2] Do not silently bypass idempotency for waitForResult:** `src/gateway/server/hooks-request-handler.ts:293-295`
  - body: Setting `replayKey` to `undefined` whenever `waitForResult` is true means a retry with the same idempotency key starts another agent run. Existing hook calls dedupe retries by idempotency key, and synchronous callers are likely to retry after network loss; store a replayable result or return a clear unsupported-idempotency error instead of duplicating side effects.
  - confidence: 0.84
- **[P3] Align the waitForResult response contract:** `src/gateway/hooks.ts:222-224`
  - body: The field comment promises a `text` response and cache-bypass behavior, while the handler returns `result` and the public hook docs still list only the old async contract. Update the inline and public contract so API consumers do not implement against the wrong field name or retry behavior.
  - confidence: 0.9
