## Summary

Keep open. Current main still lacks a result-return mode for agent hooks, and this PR remains a live implementation candidate for #49019, but the branch is stale against current Gateway hook routing and announcement policy and should not merge before maintainer API/security review.

## What This Changes

The branch adds an optional `blocking` field for agent hook payloads, changes `/hooks/agent` dispatch to return text or errors, adjusts idempotency handling, and adds Gateway hook tests.

## Best Possible Solution

Rebase the implementation onto `src/gateway/server/hooks-request-handler.ts` and `src/gateway/server/hooks.ts`, preserve current hook event routing and announcement policy, and document/changelog the maintainer-approved result-return contract for #49019.

Yes. Source inspection of current main shows `blocking` is not normalized, `/hooks/agent` dispatch remains fire-and-forget, and the HTTP response is still `{ ok: true, runId }`; I did not run a live Gateway because this was a read-only review.

## Solution Assessment

Unclear. Awaiting the isolated agent run and returning text is a plausible narrow implementation, but the best merge path must first preserve current hook trust/routing policies and settle the public response contract with #49019.
