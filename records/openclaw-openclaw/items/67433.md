## Summary

Keep open: current main still lacks the requested result-return hook mode and the related feature request is still open, but this PR is not ready because the latest head conflicts with main and still has retry and announcement contract bugs.

## What This Changes

The branch adds `waitForResult` and `announceToMain` handling to `POST /hooks/agent`, makes agent hook dispatch awaitable with result/error responses, updates hook handler fixtures, and adds Slack's `typebox` dependency.

## Best Possible Solution

Land a rebased hook API change that documents the result contract, preserves or explicitly rejects idempotent retry semantics, honors `announceToMain` on all success and error paths, and includes focused gateway tests for success, failure, suppression, timeout, and retry behavior.

## Reproduction Assessment

Not applicable; this is a feature PR rather than a bug report. Source inspection of `origin/main` shows `/hooks/agent` is still async and runId-only, so the requested capability remains absent on current main.

## Solution Assessment

No. Awaiting the isolated agent turn is a plausible implementation direction, but the branch should settle idempotency semantics, honor announcement suppression on thrown failures, and align docs before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor announcement suppression on thrown failures:** `src/gateway/server/hooks.ts:232`
  - body: In `waitForResult` mode, normal and non-ok results pass through the `value.announceToMain !== false` gate, but thrown/import/runtime failures always call `handleRunError`, which enqueues a `Hook ... (error)` system event. A machine callback using `announceToMain: false` can still inject an error artifact into a session even though the HTTP response already returns the error.
  - confidence: 0.88
- **[P2] Keep waitForResult retries idempotent:** `src/gateway/server/hooks-request-handler.ts:293-295`
  - body: This branch sets `replayKey` to `undefined` whenever `waitForResult` is true, so retrying a timed-out request with the same `Idempotency-Key` starts another agent run. Either store/replay the completed result, report an in-flight/conflict response, or reject idempotency for this mode instead of silently ignoring the existing retry contract.
  - confidence: 0.86
- **[P3] Align the documented result field:** `src/gateway/hooks.ts:221-225`
  - body: The new payload comment says the response includes `text`, but the handler sends `result`, and the public hook docs still list only the old request shape. Align the inline and public docs for request fields, response shape, errors, timeout, and retry behavior before exposing this API.
  - confidence: 0.9
