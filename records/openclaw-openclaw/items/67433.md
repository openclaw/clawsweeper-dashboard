## Summary

Keep this PR open: it is an active feature implementation for the open /hooks/agent result-return request, but it still needs real behavior proof and maintainer agreement on idempotency and public contract semantics before merge.

## What This Changes

The branch adds waitForResult and announceToMain fields to POST /hooks/agent, makes agent hook dispatch awaitable, returns status/sessionKey/result/error metadata, and updates hook handler test harness mocks.

## Best Possible Solution

Ship a documented /hooks/agent result-return contract that has explicit idempotency semantics, preserves hook trust/session protections, includes focused regression coverage, and is backed by redacted real run proof.

## Reproduction Assessment

Not applicable; this is a feature PR, not a bug report. Current main source shows waitForResult and announceToMain are absent and /hooks/agent is async runId-only today.

## Solution Assessment

No. Awaiting the isolated run is a plausible direction, but silently disabling idempotent retries and leaving the public contract mismatched is not ready as the maintainable API shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep waitForResult retries idempotent:** `src/gateway/server/hooks-request-handler.ts:293-295`
  - body: The direct /hooks/agent path sets replayKey to undefined when waitForResult is true, so retrying a timed-out request with the same idempotency key starts another agent run. Current /hooks/agent dedupes idempotent retries; preserve an in-flight/replay response or explicitly reject idempotency for this mode instead of silently disabling it.
  - confidence: 0.86

  - body: The new field comment says waitForResult returns a text field, but the handler sends result and the public docs are not updated for request fields, response shape, errors, timeout, or retry behavior. Keep the inline and public contract aligned before exposing this API.
  - confidence: 0.9
