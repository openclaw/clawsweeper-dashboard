## Label Justifications

- `P2`: This is a bounded but meaningful gateway API feature with compatibility, session, and security review blockers rather than an urgent runtime outage.
- `merge-risk: 🚨 compatibility`: The PR changes the public `/hooks/agent` response contract and leaves idempotent cached responses with a different shape from fresh async responses.
- `merge-risk: 🚨 session-state`: The new wait mode bypasses existing hook idempotency replay, so retries can create duplicate isolated agent turns unless maintainers accept that semantic change.
- `merge-risk: 🚨 security-boundary`: The new `announceToMain:false` option can suppress main-session visibility for external hook-token-triggered agent runs, including non-throwing failures.

## Summary

Keep open: this is the live implementation candidate for the requested hook result mode, but the current patch is not merge-ready because it can hide non-ok async hook outcomes when announcements are suppressed, returns inconsistent response shapes for idempotent async replays, and still needs docs/tests plus current-head packed or live dispatcher proof.

## What This Changes

Adds `waitForResult` and `announceToMain` to `POST /hooks/agent`, changes hook dispatch to optionally await agent completion, and returns richer result/error/sessionKey data from the gateway hook handler.

## Best Possible Solution

Land this PR only after it preserves failure/audit visibility, makes replayed async responses match the new contract, documents and tests the public API, and includes current-head packed or live proof through the real dispatcher path.

## Reproduction Assessment

Yes, for the review findings: source inspection shows cached async responses omit the new fields, and non-ok results are gated by `announceToMain !== false`. Current-head proof also states the dispatcher was stubbed, so full real behavior remains unproven.

## Solution Assessment

No, not yet. Extending `/hooks/agent` is a plausible narrow solution, but this patch needs safer failure visibility, consistent replay semantics, docs/tests, and current-head live dispatcher proof before it is the best implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep failed hook runs visible when announcements are suppressed:** `src/gateway/server/hooks.ts:166-168`
  - body: `announceToMain !== false` now gates every `shouldAnnounceHookRunResult` outcome. For async requests, a non-throwing `status: "error"` run with `announceToMain:false` has already returned `200 accepted` and will enqueue no system event, regressing the current invariant that non-ok `deliver:false` hook outcomes stay visible. Keep non-ok/error visibility, or add an explicit maintainer-approved audit path.
  - confidence: 0.86
- **[P2] Keep idempotent async responses equivalent:** `src/gateway/server/hooks-request-handler.ts:387-392`
  - body: Fresh async requests now return `status` and `sessionKey`, but the replay cache still stores only `runId`, so an idempotent retry gets a different response shape from the original request. Store and replay the resolved `sessionKey`/`accepted` status, or keep the async response contract unchanged.
  - confidence: 0.84
- **[P2] Document and cover the new hook contract:** `src/gateway/hooks.ts:476-477`
  - body: The PR adds public request fields and response statuses, but the rebased diff only updates harness signatures; PR-head docs still list `/hooks/agent` only through `timeoutSeconds`, and branch tests do not cover `waitForResult` or `announceToMain`. Add docs and focused tests for success, non-ok/error, timeout, idempotency, and announcement behavior.
  - confidence: 0.8
