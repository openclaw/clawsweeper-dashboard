## Summary

Keep open. This is still an active feature PR for a missing Gateway API mode, but the current branch needs maintainer API review and fixes for retry/error-announcement semantics before merge.

## What This Changes

The PR adds a synchronous `waitForResult` mode and `announceToMain` suppression to `/hooks/agent`, updates gateway hook dispatch/response test fixtures, and declares Slack's `typebox` runtime dependency.

## Best Possible Solution

Ship a documented `/hooks/agent` result-return contract that has explicit retry semantics, consistent announcement suppression, untrusted hook event handling, and focused Gateway tests/docs; keep the Slack dependency fix from main or split it out if needed.

## Reproduction Assessment

Not applicable; this is a feature PR rather than a bug report. Source and docs inspection of current main show `/hooks/agent` is still async and runId-only today.

## Solution Assessment

No. Awaiting the isolated agent turn is a plausible implementation direction, but the PR should preserve or explicitly reject idempotent retries, honor announcement suppression on failure paths, and align the public/inline contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor announceToMain on thrown hook failures:** `src/gateway/server/hooks.ts:232`
  - body: In `waitForResult` mode, non-throwing results respect `announceToMain: false`, but thrown/import/runtime failures still call `handleRunError`, which enqueues a `Hook ... (error)` system event and heartbeat. A machine callback that requested no main-session artifact can still inject an error artifact into the target session even though the HTTP response already carries the failure.
  - confidence: 0.88
- **[P2] Preserve waitForResult idempotency semantics:** `src/gateway/server/hooks-request-handler.ts:293-295`
  - body: The direct `/hooks/agent` path disables the replay key whenever `waitForResult` is true, so retrying a timed-out request with the same idempotency key starts another agent run. Existing `/hooks/agent` calls dedupe by idempotency key; synchronous callers are especially likely to retry after network loss, so this can duplicate side effects unless the API stores a replayable result or explicitly rejects idempotency for this mode.
  - confidence: 0.87
- **[P3] Align the waitForResult response contract:** `src/gateway/hooks.ts:222-224`
  - body: The new field comment promises a `text` response field and cache-bypass behavior, while the handler sends `result` and the public hook docs still list only the old request/response contract. Align the inline and public contract so API consumers do not implement against the wrong field name or retry behavior.
  - confidence: 0.9
