## Summary

Keep open: current main still lacks a blocking/result-return mode for `/hooks/agent`, and this PR remains the live implementation path for the related open feature request. The latest branch addresses earlier error/trust feedback in principle, but it still needs maintainer agreement on blocking idempotency semantics plus a small test/docs cleanup before merge.

## What This Changes

The PR adds a `blocking` flag to direct agent hook payloads, makes `/hooks/agent` await isolated agent completion, and returns response text or errors while changing hook dispatch run-id handling.

## Best Possible Solution

Ship a maintainer-approved `/hooks/agent` result-return contract that keeps retry semantics explicit, preserves hook isolation/trust protections, and includes regression tests, public docs, and a changelog entry.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection of current main shows the requested blocking/result-return capability is absent: payload normalization ignores it and the handler responds with runId only.

## Solution Assessment

No, not yet. Awaiting the isolated run is a plausible implementation, but the branch should not merge while blocking mode bypasses the existing idempotency contract and leaves docs/changelog plus test harnesses out of sync.

## Review Findings

Overall confidence: 0.86

Full review comments:

- **[P2] Update hook dispatcher mocks for the async result:** `src/gateway/server/hooks-request-handler.ts:49-51`
  - body: Changing the dispatcher contract to `Promise<{ runId, outputText?, agentError? }>` leaves the existing `createHooksHandler` default and timeout-test mocks returning a bare string. With this handler awaiting and destructuring the result, those tests either fail type checking or return `{ ok: true }` without a run id; update the harness/tests to return the new result object.
  - confidence: 0.82
- **[P2] Keep hook idempotency active in blocking mode:** `src/gateway/server/hooks-request-handler.ts:293-296`
  - body: Existing `/hooks/agent` idempotency prevents duplicate agent runs on retries, but this branch disables the replay key whenever `blocking` is true. If a client times out and retries the same blocking request with the same key, the gateway starts another isolated run and can repeat side effects; keep an in-flight/completed replay entry for blocking requests instead of bypassing the cache.
  - confidence: 0.84
- **[P3] Document the blocking hook response contract:** `src/gateway/hooks.ts:221-227`
  - body: This adds a public payload flag and new success/error response shapes, while the public webhook docs and changelog still describe only the existing async fields. Please document the accepted `blocking`, `text`/error, timeout, and idempotency behavior and add the required user-facing changelog entry.
  - confidence: 0.9
