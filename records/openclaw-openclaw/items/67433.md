## Summary

Keep open: current main still lacks the result-return hook mode, and this PR is a relevant implementation candidate for #49019, but the latest diff still bypasses hook idempotency for waitForResult and leaves the public contract undocumented.

## What This Changes

This PR adds waitForResult and announceToMain fields to POST /hooks/agent, makes direct agent hook dispatch await isolated run completion when requested, and returns status, session, result, or error metadata.

## Best Possible Solution

Ship a maintainer-approved /hooks/agent result-return contract that preserves idempotent retries, maintains hook trust/session protections, and documents the request and response semantics.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection of current main shows the requested result-return mode is absent: payload normalization ignores waitForResult/announceToMain and the handler returns only a runId.

## Solution Assessment

Unclear. Awaiting the isolated run is a plausible implementation, but bypassing established hook idempotency and shipping new response fields without docs/changelog is not the narrowest maintainable public API path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep waitForResult retries idempotent:** `src/gateway/server/hooks-request-handler.ts:290-293`
  - body: The direct /hooks/agent path sets replayKey to undefined whenever waitForResult is true, so retrying a timed-out request with the same idempotency key starts a second agent run. Current main dedupes /hooks/agent retries, and duplicating a blocking automation can repeat side effects; keep a replay or in-flight entry for waitForResult requests instead of bypassing the cache.
  - confidence: 0.82

Please align docs/changelog and the inline contract before merging.
