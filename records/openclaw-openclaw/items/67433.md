## Summary

Keep open: current main still lacks a result-returning hook mode and this PR is a plausible implementation for https://github.com/openclaw/openclaw/issues/49019, but the patch still has blocking contract issues around announcement suppression, idempotent retry semantics, and response/docs alignment.

## What This Changes

The PR adds opt-in `waitForResult` and `announceToMain` fields to `/hooks/agent`, makes gateway hook dispatch return result/error data, updates hook test fixtures, and adds Slack's `typebox` runtime dependency.

## Best Possible Solution

Ship a documented and tested `/hooks/agent` result-return contract that has explicit retry semantics, consistent `announceToMain` behavior across success and failure paths, and untrusted hook event handling.

## Reproduction Assessment

Not applicable; this is a feature PR rather than a bug report. Source inspection of current main shows `/hooks/agent` remains async and runId-only today.

## Solution Assessment

No. Awaiting the isolated agent turn is a reasonable implementation direction, but the PR should settle retry semantics, suppress all requested machine-only artifacts, and align docs plus inline contracts before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor announceToMain on thrown hook failures:** `src/gateway/server/hooks.ts:232`
  - body: In `waitForResult` mode, non-throwing results respect `announceToMain: false`, but thrown/import/runtime failures still call `handleRunError`, which enqueues a `Hook ... (error)` system event and heartbeat. A machine callback that requested no main-session artifact can still inject an error artifact into the target session even though the HTTP response already carries the failure.
  - confidence: 0.88
- **[P2] Preserve waitForResult idempotency semantics:** `src/gateway/server/hooks-request-handler.ts:293-295`
  - body: The direct `/hooks/agent` path disables the replay key whenever `waitForResult` is true, so retrying a timed-out request with the same idempotency key starts another agent run. Existing `/hooks/agent` calls dedupe by idempotency key, and synchronous callers are especially likely to retry after network loss, so this can duplicate side effects unless the API stores a replayable result or explicitly rejects idempotency for this mode.
  - confidence: 0.86
- **[P3] Align the waitForResult response contract:** `src/gateway/hooks.ts:222-224`
  - body: The new field comment promises a `text` response field and cache-bypass behavior, while the handler sends `result` and the public hook docs still list only the old request fields. Align the inline and public contract so API consumers do not implement against the wrong field name or retry behavior.
  - confidence: 0.9
