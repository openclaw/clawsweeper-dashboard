## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not suppress async failure announcements:** `src/gateway/server/hooks.ts:166-168`
  - body: `announceToMain: false` is applied before `shouldAnnounceHookRunResult`, so an async hook can return `accepted` and then suppress the session-visible event for a non-ok agent result. Thrown failures still enqueue an error event, but `result.status === "error"` does not; keep failure announcements independent of success-summary suppression unless the caller is actually waiting for the error response.
  - confidence: 0.88
- **[P2] Keep replay responses aligned with accepted responses:** `src/gateway/server/hooks-request-handler.ts:340-341`
  - body: Fresh async requests now return `status` and `sessionKey`, but this replay branch still returns only `{ ok, runId }`. A retried idempotent request can therefore see a different schema from the original accepted response, breaking clients that branch on `status` or need the resolved `sessionKey`; cache enough response data or keep the async response contract consistent.
  - confidence: 0.9
- **[P2] Document the hook result-return contract:** `src/gateway/hooks.ts:229-238`
  - body: This adds a public `/hooks/agent` mode, but the published gateway docs still list only the old fields and do not describe `waitForResult`, `announceToMain`, result/error responses, timeout behavior, or idempotency semantics. Update the user-facing hook docs with the API change so operators can use it safely.
  - confidence: 0.84
