## Summary

Keep this open: `--gpus` support is already implemented and shipped, but current `main` still has no host IPC or generic Docker-args sandbox surface. The remaining `--ipc=host` request is a security/product decision, not a safe cleanup close.

## What This Changes

Request to pass custom Docker sandbox flags such as `--gpus all` and `--ipc=host` for GPU/ML workloads.

## Best Possible Solution

Keep the shipped `sandbox.docker.gpus` support and, if maintainers approve host IPC, add a narrow dangerous IPC setting with schema validation, audit/guard visibility, docs, and Docker create-args tests instead of arbitrary Docker flag passthrough.

## Reproduction Assessment

Yes, at the source level: current main and `v2026.6.8` show typed GPU support, while the strict Docker sandbox schema and create-args path have no IPC field or `--ipc` emission. This is a missing feature capability, not a broken existing contract.

## Solution Assessment

No, the generic custom Docker-flag passthrough proposed in the issue is not the best shape because it would bypass typed config and security guardrails. The maintainable path is a dedicated, explicitly dangerous IPC option if maintainers accept the tradeoff.
