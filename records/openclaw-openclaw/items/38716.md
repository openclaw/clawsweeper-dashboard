## Label Justifications

- `P2`: This is a normal-priority Slack plugin feature with meaningful product and security decisions but limited blast radius to Slack search behavior.
- `impact:security`: RTS can expose workspace messages, files, channels, and users, so the issue is about sensitive data and privacy boundaries.

## Summary

Keep open. Current main still lacks Slack RTS support: there is no `assistant.search.context` call path, no `action_token` propagation into Slack turn/action context, and the adjacent Slack `search.messages` PRs do not satisfy Slack's RTS action-token contract.

## What This Changes

The issue asks the Slack plugin to capture Slack RTS action tokens and expose `assistant.search.context` workspace search to agents.

## Best Possible Solution

Implement Slack-plugin-owned RTS support with ephemeral per-turn action-token handling, a typed read-only `assistant.search.context` action, documented scope/privacy limits, and deliberate separation or consolidation with legacy `search.messages`.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes for the source-level gap: current main lacks RTS symbols, action-token propagation, and a Slack search action. The focused source checks above provide high confidence that the feature is still missing.

## Solution Assessment

Unclear until maintainers define the Slack RTS product and security contract. The requested direction is useful, but the implementation should follow Slack's current action-token source from inbound `message`/`app_mention` events and avoid treating `search.messages` as an RTS substitute.
