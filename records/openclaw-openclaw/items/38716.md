## Summary

Keep open. Current main still lacks Slack RTS/action-token support: there is no `assistant.search.context` call path, no `action_token` capture or propagation, and no Slack `search` message action; the related `search.messages` work is adjacent user-token search, not the requested assistant action-token RTS flow.

## What This Changes

Add Slack Real-Time Search support by capturing Slack assistant action tokens and exposing `assistant.search.context` through Slack message tooling.

## Best Possible Solution

Implement the feature in the bundled Slack plugin with a scoped RTS contract: capture ephemeral Slack action tokens from supported inbound events, pass them only through the active assistant/message context, expose a typed read-only RTS search action, document required AI/search scopes and privacy limits, and keep the design distinct from user-token `search.messages` unless maintainers choose to consolidate them.

## Reproduction Assessment

Not applicable as a traditional bug reproduction. The gap is high-confidence from static current-main verification: the Slack plugin has no `assistant.search.context`, `action_token`, or Slack `search` action path to exercise.

## Solution Assessment

Unclear. Supporting Slack RTS is plausible, but the requested path needs maintainer design around event source, ephemeral token scope, user-token overlap, permissions, privacy, and rate limits before a narrow implementation is clearly best.
