## Summary

Keep open. Current main has Slack assistant-thread lifecycle handling, but still has no Slack RTS implementation: no `action_token` capture, no `assistant.search.context` call path, and no Slack RTS/search message action; the related `search.messages` PRs are adjacent user-token search work, not the event-scoped RTS contract requested here.

## What This Changes

Request support for Slack Real-Time Search by extracting assistant event action tokens and exposing `assistant.search.context` through the Slack tool/action surface.

## Best Possible Solution

Implement a Slack-plugin-owned RTS design that captures event-scoped action tokens only for the active turn, exposes a typed read-only `assistant.search.context` action, documents required Slack AI/search scopes and privacy limits, and coordinates naming with the separate `search.messages` work.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection provides a high-confidence current-main verification path: RTS symbols, action-token propagation, and Slack RTS action dispatch are absent.

## Solution Assessment

Unclear until maintainers define the Slack RTS product and security contract. The maintainable direction is event-scoped `assistant.search.context` support in the Slack plugin, distinct from user-token `search.messages` unless maintainers deliberately unify them.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because a future implementation would process ephemeral Slack action tokens and potentially broad workspace search results.

Concerns:
