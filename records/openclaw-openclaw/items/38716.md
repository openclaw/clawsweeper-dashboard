## Summary

Keep open. Current main still lacks the requested Slack RTS path: the Slack plugin does not preserve an event-scoped `action_token`, does not call `assistant.search.context`, and does not advertise or dispatch a Slack `search` action. The linked `search.messages` work is adjacent but does not cover bot-token RTS/action-token behavior.

## What This Changes

Request Slack plugin support for Real-Time Search by carrying Slack event action tokens and exposing `assistant.search.context` through a message action or dedicated tool.

## Best Possible Solution

Implement the remaining capability inside the bundled Slack plugin after maintainer product/security review: capture the event-scoped Slack action token from the inbound event that authorized the query, keep it confined to the originating team/user/channel/thread/request context, call `assistant.search.context` through a typed read-only RTS action, and document the current Slack AI/search scopes and privacy limits.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime regression, but yes for feature absence: current-main source and docs inspection shows no RTS method call, no action-token field in Slack event handling, and no Slack search action dispatch path.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer product/security review settles the contract; the feature belongs in the Slack plugin, but the token source should follow Slack's current `message` / `app_mention` RTS docs rather than assuming assistant-thread events carry the action token.
