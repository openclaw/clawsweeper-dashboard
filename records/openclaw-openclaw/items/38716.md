## Summary

Keep open. Current main still lacks Slack RTS/action-token support, and the adjacent Slack search work covers the older user-token `search.messages` path rather than `assistant.search.context` with event-scoped action tokens.

## What This Changes

Request Slack Real-Time Search support by capturing Slack assistant action tokens and exposing `assistant.search.context` through the Slack message/action surface.

## Best Possible Solution

Implement Slack-plugin-owned RTS support with event-scoped action-token handling, a typed read-only search action, focused docs, and privacy/scope tests kept distinct from the user-token `search.messages` path unless maintainers intentionally consolidate them.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main has no Slack RTS symbols, no action-token context, and no Slack search action dispatch.

## Solution Assessment

Unclear until maintainers decide the Slack RTS API and security shape. The maintainable direction is Slack-plugin-owned `assistant.search.context` support; treating the adjacent `search.messages` PR as the fix would leave the requested action-token behavior missing.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would handle ephemeral Slack action tokens and workspace-wide search results.

Concerns:

- **[medium] Define Slack RTS action-token boundaries:**
  - body: A future implementation should avoid persisting or logging `action_token` values, bind each token to the triggering Slack turn, and keep it out of generic tool output and stored transcripts.
  - confidence: 0.88
- **[medium] Gate workspace-wide search results:**
  - body: Slack RTS can expose broad workspace context depending on event authorization and scopes, so the design needs explicit consent, privacy, scope, and rate-limit handling before automation should implement it.
