## Summary

Keep open. Current main still lacks Slack RTS/action-token search: no Slack code reads `action_token` or calls `assistant.search.context`, and the Slack message actions only expose the existing send/read/edit/delete/reaction/pin/member/emoji/file paths. Slack's current RTS docs say bot-token RTS calls require an `action_token` from `message` or `app_mention` events, while `assistant_thread_started` provides thread context rather than the token. citeturn1view0turn1view2

## What This Changes

Request Slack plugin support for Real-Time Search by capturing Slack action tokens and exposing `assistant.search.context` through a read-only message/search tool action.

## Best Possible Solution

Implement RTS in the bundled Slack plugin using Slack's current contract: carry the event-scoped action token from the triggering inbound message/app mention into the assistant turn, expose a typed read-only `assistant.search.context` action, document required scopes and privacy limits, and keep it distinct from user-token `search.messages` unless maintainers deliberately unify them.

## Reproduction Assessment

Yes. A current-main source check reproduces the gap: Slack never advertises or dispatches `search`, no code handles `action_token`, and the existing read path calls Slack conversation history/replies APIs rather than RTS.

## Solution Assessment

Unclear as written. The product direction is valid, but the exact requested extraction from `assistant_thread_started`/context-changed should be corrected to Slack's current `message`/`app_mention` action-token contract before implementation.
