## Summary

Keep open. Current main and the latest release still lack a Slack RTS path: the Slack plugin stores assistant thread context but not `action_token`, has no `assistant.search.context` call, and does not advertise or dispatch a Slack `search` message action; the related Slack search PRs are either closed unmerged or open/conflicting and use user-token `search.messages`, not the assistant action-token RTS flow.

## What This Changes

The issue requests Slack RTS support by capturing Slack assistant action tokens and using them for workspace search through a message action or dedicated tool.

## Best Possible Solution

Keep this issue open until maintainers approve a Slack-plugin-owned RTS design that privately consumes the action token, documents scope/rate-limit behavior, and proves live Slack search without exposing raw credentials to generic tool state.

## Reproduction Assessment

Yes, by source inspection. Current main handles Slack assistant thread events and Slack message actions, but targeted searches show no `action_token` model, no `assistant.search.context` call, and no advertised or dispatched Slack `search` action.

## Solution Assessment

Unclear as phrased. A typed Slack RTS search action may be the right endpoint, but exposing or persisting the raw action token outside the Slack plugin boundary would not be the safest solution.
