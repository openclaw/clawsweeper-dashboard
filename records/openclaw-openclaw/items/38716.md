## Summary

Keep open. Current main still lacks Slack RTS/action-token support, and the adjacent Slack search PRs implement user-token `search.messages` rather than `assistant.search.context` with ephemeral Slack action tokens.

## What This Changes

Support Slack Real-Time Search by capturing Slack assistant action tokens and exposing `assistant.search.context` through a Slack message action or dedicated tool.

## Best Possible Solution

Implement Slack-plugin-owned RTS support with a typed read-only action that captures the inbound Slack `action_token`, keeps it scoped to the triggering turn, calls `assistant.search.context`, and documents scopes, privacy limits, and rate-limit behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives a high-confidence verification path: no action-token field, RTS helper, `assistant.search.context` call, or Slack RTS search action exists.

## Solution Assessment

Unclear until maintainers decide the security and API shape. The maintainable direction is Slack-plugin-owned RTS support; treating user-token `search.messages` work as the fix would not cover the requested action-token API.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would process ephemeral Slack action tokens and workspace-wide search results.

Concerns:

- **[medium] Define Slack RTS token boundaries:**
  - body: A future implementation should avoid persisting or logging `action_token` values, bind each token to the triggering Slack turn, and ensure the token is not exposed through generic tool output or stored transcripts.
  - confidence: 0.88
- **[medium] Gate workspace-wide search results:**
  - body: Slack RTS can return broad workspace context depending on scopes and channel type, so the design needs explicit scope, consent, rate-limit, and privacy handling before this is a safe automated fix.
