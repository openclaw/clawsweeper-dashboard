## Summary

Keep open. Current main and the latest release still lack the requested Slack RTS/action-token path: assistant-thread handlers keep Slack thread context but not `action_token`, Slack actions do not call `assistant.search.context`, and Slack message actions do not advertise or dispatch a `search` action. The related Slack search PRs are adjacent `search.messages` user-token work, not a merged or safe canonical implementation of assistant action-token RTS.

## What This Changes

The issue requests Slack assistant action-token support so OpenClaw can call Slack Real-Time Search through assistant thread events.

## Best Possible Solution

Keep this open until maintainers approve a Slack-plugin-owned RTS design that captures the action token narrowly, calls `assistant.search.context` through a typed read-only action, documents scopes/privacy/rate limits, and proves live Slack behavior without credential leakage.

## Reproduction Assessment

Yes, by source inspection. Current main handles Slack assistant thread events and Slack message actions, but targeted searches show no `action_token` model, no `assistant.search.context` call, and no advertised or dispatched Slack `search` action.

## Solution Assessment

Unclear as phrased. A typed Slack RTS action may be the right endpoint, but exposing or persisting the raw action token outside the Slack plugin boundary would not be the safest solution.
