## Summary

Keep open. Current main still lacks Slack RTS support: there is no `assistant.search.context` call path, no captured Slack `action_token` on inbound message/app mention events, and no Slack `search` message action; the related open Slack search PR targets `search.messages`, which is useful adjacent work but not the requested event-scoped RTS contract.

## What This Changes

Request support in the bundled Slack plugin for Slack Real-Time Search via event-scoped assistant action tokens.

## Best Possible Solution

Implement a Slack-plugin-owned RTS path that captures event-scoped action tokens, exposes a typed read-only `assistant.search.context` action, documents privacy/scope limits, and coordinates but does not conflate it with user-token `search.messages`.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main lacks RTS symbols, event action-token context, and Slack RTS action dispatch.

## Solution Assessment

Unclear until maintainers define the Slack RTS product and security contract. The likely maintainable direction is a Slack-plugin-owned, event-scoped `assistant.search.context` action rather than treating user-token `search.messages` as the same feature.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would handle ephemeral Slack action tokens and organization-wide search results.

  - body: A future implementation should avoid persisting or logging `action_token` values, bind each token to the triggering Slack turn, and keep tokens out of generic tool output and stored transcripts.
