## Summary

Keep this open. Current main and the latest release still lack Slack RTS/action-token support: the Slack plugin handles assistant-thread lifecycle and thread context, but it does not model `action_token`, call `assistant.search.context`, or expose a Slack search message action. The related closed PRs are adjacent `search.messages` user-token attempts and do not supersede this event-scoped RTS request.

## What This Changes

Request support for Slack Real-Time Search by extracting assistant action tokens from Slack assistant events and exposing an RTS-backed search action or tool.

## Best Possible Solution

Add a Slack-plugin-owned RTS search operation that captures the action token only from the relevant assistant/user event, keeps it in memory, binds it to the Slack account/thread/user context, enforces scopes/rate/privacy boundaries, documents setup, and proves behavior against a live Slack workspace.

## Reproduction Assessment

Yes. The gap is source-reproducible on current main: Slack assistant events are handled, but no `action_token` field, RTS method call, or Slack search message action exists.

## Solution Assessment

No, not exactly as proposed. Exposing a generic token would be the wrong boundary; the maintainable solution is a typed Slack search operation that privately consumes the short-lived, thread-bound token.
