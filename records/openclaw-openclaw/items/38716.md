## Label Justifications

- `impact:security`: Slack RTS can expose workspace messages, files, channels, and users, so the issue is about sensitive data and privacy boundaries.

## Summary

Keep open. Current main and the latest release still do not implement Slack RTS: there is no `assistant.search.context` call path, no `action_token` propagation into Slack message or assistant-thread context, and the active Slack search PRs are `search.messages` work rather than the requested action-token RTS behavior.

## What This Changes

The issue requests Slack Real-Time Search support by capturing Slack assistant action tokens and exposing `assistant.search.context` through a Slack message action or dedicated tool.

## Best Possible Solution

Implement Slack-plugin-owned RTS as a deliberately scoped read-only search action that keeps ephemeral action tokens out of storage/logs, documents scope and privacy limits, and either separates or explicitly consolidates it with the existing `search.messages` work.

## Reproduction Assessment

Not applicable as a bug reproduction; yes as a source-level feature-gap check. Current main lacks RTS symbols, action-token propagation, and a Slack search action, while the latest release has the same gap.

## Solution Assessment

Unclear until maintainers define the Slack RTS product and security contract. The requested direction is useful, but `search.messages` should not be treated as an RTS substitute without an explicit coexistence or consolidation decision.
