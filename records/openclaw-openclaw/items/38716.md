## Summary

The requested Slack RTS capability is still necessary: current main handles assistant-thread lifecycle and context but neither models the event-issued action token nor invokes `assistant.search.context`; the closed related PRs used the different `search.messages` user-token flow and do not supersede this request. Keep this open for a maintainer product/security decision because the raw ephemeral credential must not become generic persisted message-tool state.

Likely related people
- Shakker — high confidence; introduced the current assistant-event and thread-context path.
- Peter Steinberger — medium confidence; broad historical contributor across the Slack plugin.

## What This Changes

Add Slack Real-Time Search by consuming assistant-event action tokens and exposing workspace search to agents.

## Best Possible Solution

Keep the raw token inside an in-memory Slack-plugin-owned assistant-thread capability, expose only a typed RTS search operation, enforce account/thread binding and expiry, and require live Slack proof that results follow the initiating user's visibility without logging or persisting the token.

## Reproduction Assessment

Yes for the reported capability gap: current main handles both named Slack assistant events, but the event model omits the action token and no RTS method is called. This is source-verifiable missing functionality rather than a reproduction of broken established behavior.

## Solution Assessment

No, not exactly as proposed. A tool-facing search operation is appropriate, but making the credential generically available would weaken its boundary; the Slack plugin should privately consume a short-lived, thread-bound token.

## Risks / Open Questions

- The event-issued action token grants sensitive workspace-search authority; persisting it in assistant metadata, logs, transcripts, or shared session state could leak the credential beyond its intended lifetime.
- A generic message action needs an explicit binding between the active Slack account, assistant channel/thread, event token, and search invocation; otherwise an agent could use a stale token or search under the wrong thread context.
- The prior user-token search proposals do not establish the correct scope or upgrade behavior for RTS, so reusing that static-token design would create a competing and potentially confusing authorization path.
