## Summary

Keep open. Current main still lacks Slack RTS support: there is no `action_token` capture, no `assistant.search.context` call path, and no Slack `search` message action; the related Slack `search.messages` PRs are adjacent but do not satisfy the requested assistant action-token RTS behavior.

## What This Changes

The issue requests Slack Real-Time Search support by extracting Slack assistant action tokens and exposing an RTS-backed search action or tool.

## Best Possible Solution

Implement this in the bundled Slack plugin as a distinct, least-privilege RTS path that captures ephemeral action tokens from authorized inbound Slack events, does not persist them beyond the relevant turn, exposes a typed read-only search action, documents scopes/rate limits, and keeps the behavior separate from user-token `search.messages` unless maintainers intentionally consolidate them.

## Reproduction Assessment

Yes for the source-level gap: current main has assistant thread event handling, Slack read actions, and Slack docs, but no `action_token`, `assistant.search.context`, or Slack search action path. I did not run a live Slack RTS call because this read-only review has no workspace token and the missing implementation is visible in source.

## Solution Assessment

Unclear until maintainers choose the product/security boundary. The narrow implementation should live in the Slack plugin, but it needs an explicit RTS token lifecycle and permission contract rather than reusing the older `search.messages` user-token proposals blindly.
