## Summary

Keep open. Current main still lacks Slack RTS support: there is no `action_token` propagation, no `assistant.search.context` call path, and no Slack `search` action on the current Slack message-action surface.

## What This Changes

The issue requests Slack RTS support by extracting assistant action tokens and exposing `assistant.search.context` as a Slack search action.

## Best Possible Solution

Implement Slack-plugin-owned RTS support with an explicit ephemeral-token contract, a typed read-only `assistant.search.context` action, documented `search:read.*` scope limits, and clear separation or deliberate consolidation with user-token `search.messages` work.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection provides a high-confidence current-main gap check: RTS symbols, event action-token propagation, and Slack RTS action dispatch are absent.

## Solution Assessment

Unclear until maintainers define the Slack RTS product and security contract. The maintainable direction is Slack-plugin-owned RTS support based on Slack's current `assistant.search.context` contract, not treating legacy `search.messages` as a full substitute.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would handle ephemeral Slack action tokens and broad workspace search results.

Concerns:

- **[medium] Define RTS action-token boundaries:**
  - body: A future implementation should avoid persisting or logging Slack `action_token` values, bind each token to the triggering Slack turn, and keep tokens out of generic tool output and stored transcripts.
  - confidence: 0.9
- **[medium] Gate workspace-wide search exposure:**
  - body: RTS can return messages, files, users, and channels based on token type and scopes, so maintainers should define consent, redaction, rate-limit, and scope behavior before landing the feature.
  - confidence: 0.86
