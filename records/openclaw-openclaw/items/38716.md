## Summary

Keep open. Current main still has no Slack RTS/action-token implementation, and the related Slack search PRs cover the older user-token `search.messages` path rather than `assistant.search.context` with event-scoped action tokens.

## What This Changes

Request support for Slack Real-Time Search by capturing Slack assistant action tokens and exposing an `assistant.search.context` search action or tool.

## Best Possible Solution

Implement Slack-plugin-owned RTS support with an event-scoped action-token path, typed read-only search action, focused docs, and privacy/scope tests distinct from `search.messages`.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main has no Slack RTS symbols, no action-token context, and no Slack search action dispatch.

## Solution Assessment

Unclear until maintainers decide the security and API shape. The maintainable direction is Slack-plugin-owned `assistant.search.context` support; treating the open `search.messages` PR as the fix would leave the requested action-token RTS behavior missing.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would handle ephemeral Slack action tokens and workspace-wide search results.

Concerns:

- **[medium] Define Slack RTS action-token boundaries:**
  - body: A future implementation should avoid persisting or logging `action_token` values, bind each token to the triggering Slack turn, and keep it out of generic tool output and stored transcripts.
  - confidence: 0.88
- **[medium] Gate workspace-wide search results:**
  - body: Slack RTS can return broad workspace context depending on scopes and channel type, so the design needs explicit consent, privacy, scope, and rate-limit handling before automation should implement it.
  - confidence: 0.84
