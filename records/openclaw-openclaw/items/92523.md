## Summary

Keep this issue open: current main and the latest release still leave `waiting` TaskFlows nonterminal, and task-flow maintenance only prunes terminal rows. The linked repair PR is still open and previously blocked on wait-abandonment semantics plus gateway/runtime proof, so there is no merged implementation or safe duplicate close path yet.

## What This Changes

The issue asks OpenClaw to recover orphaned managed TaskFlow records stuck in `waiting` so they no longer contribute to persistent heartbeat `requests-in-flight` deadlocks.

## Best Possible Solution

Keep the issue open until https://github.com/openclaw/openclaw/pull/92648 or a replacement defines safe TaskFlow wait-abandonment semantics and proves gateway startup/runtime recovery for the actual busy-state owner.

## Reproduction Assessment

No high-confidence end-to-end reproduction of the full heartbeat deadlock is present. Source inspection does verify that `waiting` TaskFlows can persist indefinitely and maintenance will not expire them, but the direct `flow_runs` to `requests-in-flight` link remains unproven.

## Solution Assessment

No, the current open PR’s blanket maintenance TTL is only a plausible mitigation. The better fix is explicit controller-owned wait abandonment or deadline semantics plus startup/runtime recovery tied to the proven busy-state resource.
