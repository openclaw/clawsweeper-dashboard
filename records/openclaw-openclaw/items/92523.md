## Summary

Keep this issue open: current main and the latest release still allow managed TaskFlows to remain in `waiting` indefinitely, and the only current implementation path is the still-open linked PR with unresolved proof and semantics concerns.

## What This Changes

The issue reports that orphaned TaskFlows left in `waiting` status can keep sessions marked busy and indefinitely skip agent heartbeats.

## Best Possible Solution

Review or revise https://github.com/openclaw/openclaw/pull/92648 into an explicit TaskFlow wait-abandonment and gateway/runtime recovery fix, then close this issue only after that fix merges.

## Reproduction Assessment

No high-confidence end-to-end reproduction is present. Source inspection supports that `waiting` flows can persist indefinitely and maintenance does not touch them, but the direct heartbeat deadlock path still needs live or focused gateway proof.

## Solution Assessment

No, not yet. The active PR’s blanket age-based expiry is a plausible mitigation, but the best solution should define wait-abandonment semantics and recover through the gateway/runtime owner that actually holds the busy state.
