## Label Justifications

- `P1`: The report describes a persistent real deployment failure where scheduled and event-triggered agent work can stop until manual database and gateway intervention.
- `impact:crash-loop`: The affected agent becomes functionally unavailable, with heartbeats repeatedly skipped until operator recovery.
- `impact:message-loss`: The follow-up reports accepted wake or voice events remaining queued and unconsumed while heartbeat dispatch stays blocked.
- `impact:session-state`: Persisted TaskFlow wait state, runtime busy state, and pending delivery state allegedly fail to converge across restarts.

## Summary

Keep open: current main and latest release still leave `waiting` TaskFlows nonterminal indefinitely, and the open fix PR at https://github.com/openclaw/openclaw/pull/92648 is not yet a safe or merged implementation path. The exact heartbeat busy-state link still needs gateway-level proof, but the core TaskFlow lifecycle gap is concrete enough for maintainer follow-up.

## What This Changes

The issue requests core recovery for managed TaskFlows that remain in `waiting` after abandoned callbacks so they do not keep agent heartbeats blocked.

## Best Possible Solution

Define controller-owned wait abandonment or deadline semantics, wire recovery into the gateway/runtime owner that actually releases the busy resource, and keep pending final-delivery cleanup tracked through https://github.com/openclaw/openclaw/issues/83184 or https://github.com/openclaw/openclaw/issues/85743.

## Reproduction Assessment

No high-confidence end-to-end reproduction of the heartbeat deadlock is established. Source inspection does show `waiting` TaskFlows can persist indefinitely and maintenance only warns or prunes terminal flows, while the actual heartbeat busy-state source still needs runtime proof.

## Solution Assessment

No, not as a simple TTL sweep. The safer solution needs explicit wait-abandonment semantics and proof that the selected gateway/runtime path clears the busy resource without breaking legitimate long waits.
