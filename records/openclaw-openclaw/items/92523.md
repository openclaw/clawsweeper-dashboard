## Label Justifications

- `P1`: The report describes an unattended deployment where accepted wake events and scheduled heartbeats can stop producing agent turns until manual database and gateway intervention.
- `impact:crash-loop`: The affected agent can become functionally unavailable in a persistent busy/deadlocked state until operator recovery.
- `impact:message-loss`: The follow-up says voice and system events can be accepted but remain unconsumed while heartbeat dispatch stays blocked.
- `impact:session-state`: The suspected failure crosses persisted TaskFlow rows, runtime busy state, and pending final delivery state across restarts.

## Summary

Keep this issue open: current main still allows durable TaskFlows to remain in `waiting` indefinitely, and the same user-visible deadlock is already being attempted in an open but not-yet-mergeable implementation PR. The full heartbeat deadlock path still needs proof because heartbeat busy gating does not directly read `flow_runs`.

## What This Changes

The issue asks OpenClaw core to recover TaskFlows stuck in `waiting` so abandoned callback waits cannot keep an agent heartbeat-blocked forever.

## Best Possible Solution

Review or replace the linked implementation with a fix that defines explicit wait-abandonment semantics and clears the actual gateway/runtime busy resource, with upgrade-safe tests for legitimate long waits.

## Reproduction Assessment

No, not for the full deadlock. Source inspection confirms waiting TaskFlows can persist indefinitely and maintenance only warns, but the exact current-main path from a waiting row to persistent heartbeat `requests-in-flight` still needs a focused reproduction.

## Solution Assessment

No, a bare TTL sweep is only a mitigation. The better fix is to define controller-owned wait deadlines or orphan detection and recover the actual runtime busy state without breaking legitimate durable waits.
