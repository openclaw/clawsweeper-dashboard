## Summary

Keep this issue open: current main still has no safe abandonment or gateway recovery path for stale `waiting` TaskFlows, and the only direct fix attempt was closed unmerged after review found blanket timeout semantics unsafe. The report is credible, but the exact `flow_runs` to `requests-in-flight` owner still needs runtime diagnosis before a narrow repair is safe.

## What This Changes

The issue asks OpenClaw core to prevent orphaned TaskFlows left in `waiting` from indefinitely blocking heartbeat dispatch for the owning agent session.

## Best Possible Solution

Define explicit controller-owned wait abandonment or deadline semantics, then wire recovery into the gateway/runtime owner that actually holds `requests-in-flight`, with proof that legitimate durable waits are preserved.

## Reproduction Assessment

No high-confidence live current-main reproduction of the full heartbeat deadlock is established here. Source inspection does show `waiting` TaskFlows can persist indefinitely and audit only warns, but the exact `requests-in-flight` owner still needs runtime diagnostics.

## Solution Assessment

No, a generic timeout is only a plausible mitigation. The safer fix needs explicit wait-abandonment semantics and gateway/runtime recovery tied to the concrete busy-state owner.
