## Label Justifications

- `P1`: The report describes a current-release availability failure where unattended agents can stop producing heartbeat turns until manual intervention.
- `impact:crash-loop`: The affected agent is reported to become functionally unavailable even though wake calls are accepted.
- `impact:message-loss`: Accepted wake events and voice transcripts can remain queued without dispatch while the session is treated as busy.
- `impact:session-state`: The issue centers on persisted TaskFlow state and runtime busy-state failing to converge after missed callbacks or restarts.

## Summary

Current main and the latest shipped release still leave stale waiting TaskFlows as nonterminal records that maintenance can audit but not expire, so the reported core cleanup need remains open. The issue now has an open implementation candidate, so the next maintainer action is to review that PR rather than queue a second repair lane.

## What This Changes

The issue requests a core TaskFlow cleanup path so abandoned waiting flows from missed callbacks cannot keep an agent session effectively busy forever.

## Best Possible Solution

Define a core-owned abandonment deadline or controller contract for stale waiting TaskFlows, prove the heartbeat busy-state resource it releases, and land the open fix PR or a narrower replacement with upgrade-safe tests.

## Reproduction Assessment

No high-confidence live reproduction is present for the full heartbeat deadlock. Source inspection does confirm that old `waiting` TaskFlows are detectable but not expired by current maintenance, and the reporter supplied concrete production observations.

## Solution Assessment

Unclear as a final fix until maintainers decide the allowed waiting-flow deadline and prove the busy-state release path. The open PR is the active implementation candidate, but its semantics and proof need review before this issue can close.
