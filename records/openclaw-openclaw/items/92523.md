## Summary

Keep this issue open: current `main` still does not expire stale managed TaskFlows in `waiting`, and the active implementation path is the open fix PR at https://github.com/openclaw/openclaw/pull/92648. The exact heartbeat deadlock still needs gateway-level proof because current heartbeat busy checks do not directly query `flow_runs`.

## What This Changes

The issue requests a core TaskFlow maintenance or timeout path that expires orphaned managed TaskFlows left in `waiting` so agent heartbeats cannot stay blocked indefinitely.

## Best Possible Solution

Review and either land or replace https://github.com/openclaw/openclaw/pull/92648 with a core-owned waiting-flow abandonment path that uses a canonical terminal state, preserves legitimate waits, and proves the heartbeat busy state actually clears.

## Reproduction Assessment

Partial/source only. Current source clearly allows old managed `waiting` TaskFlows to persist and only warns about them, but the full heartbeat deadlock still needs a current-main gateway reproduction because `runHeartbeatOnce` does not directly read `flow_runs`.

## Solution Assessment

Unclear. Expiring stale waits is a plausible fix direction, but the best solution must define safe abandonment semantics and prove the actual `requests-in-flight` resource is released; the open PR is the current place to validate that.
