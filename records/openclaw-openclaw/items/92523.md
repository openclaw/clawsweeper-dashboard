## Summary

Keep this issue open: current main and v2026.6.6 still leave managed TaskFlows in `waiting` as nonterminal state, and maintenance only warns about stale waits instead of recovering them. https://github.com/openclaw/openclaw/pull/92648 is the active repair attempt, but it has unresolved proof and lifecycle-semantics blockers, so the issue should remain the canonical report for maintainer follow-up.

## What This Changes

The issue reports orphaned managed TaskFlows stuck in `waiting` state allegedly keeping heartbeat work in `requests-in-flight` and blocking unattended agents.

## Best Possible Solution

Define controller-owned wait abandonment or deadline semantics, prove the actual heartbeat busy-state owner with a gateway-level reproduction, and land a recovery path that preserves legitimate long waits.

## Reproduction Assessment

No high-confidence full deadlock reproduction is available yet. Source inspection shows current main and v2026.6.6 can retain `waiting` TaskFlows indefinitely and only warn on stale waits, but the direct heartbeat busy-state owner still needs diagnostic reproduction.

## Solution Assessment

No, a generic N-minute timeout is only a plausible mitigation. The best fix needs an explicit wait-abandonment contract and gateway/runtime proof that the actual blocking resource is released.
