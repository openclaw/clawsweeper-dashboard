## Summary

Keep this issue open: current and shipped code allow managed TaskFlows to remain `waiting` indefinitely, but heartbeat admission is controlled by separate reply-run, command-lane, and final-delivery state, so the reported direct causal link still needs a correlated reproduction before implementing expiry.

## What This Changes

Prevent abandoned TaskFlows from leaving agents permanently unable to dispatch scheduled heartbeats.

## Best Possible Solution

Capture one stuck session with correlated flow ID, controller ID, linked tasks, reply-run registry keys, command-lane snapshots, session final-delivery state, and restart behavior; then add a supported controller-owned abandonment transition that releases the proven busy resource and atomically moves the durable flow to an existing terminal state or an explicitly approved new state.

## Reproduction Assessment

No—not yet. Current source proves that `waiting` TaskFlows can persist indefinitely and separately identifies all heartbeat busy gates, but it does not establish which reply run, lane, or delivery resource remains active because of a persisted flow row.

## Solution Assessment

No. Adding `waiting` to retention pruning or writing an unsupported `expired` value would alter durable state without proving or releasing the runtime resource responsible for `requests-in-flight`; the controller deadline and terminal transition need an explicit lifecycle decision.
