## Label Justifications

- `P1`: The report describes a persistent unattended-agent availability failure where accepted wake events stop producing agent turns.
- `impact:crash-loop`: The affected agent can become functionally unavailable until manual database intervention and gateway restart.
- `impact:message-loss`: The comments describe voice/system events being accepted but not consumed while heartbeat dispatch remains blocked.
- `impact:session-state`: The bug involves persisted TaskFlow wait state, runtime busy state, and pending delivery state failing to converge across restarts.

## Summary

Keep this issue open: current main and the latest release still let `waiting` TaskFlows remain nonterminal, and the only targeted implementation path is an open PR that has not merged and still needs maintainer decisions around wait-abandonment semantics and gateway-level proof.

## What This Changes

The issue requests recovery for orphaned TaskFlows stuck in `waiting` so they cannot permanently block heartbeat dispatch.

## Best Possible Solution

Define controller-owned wait abandonment or deadline semantics, then recover the actual gateway/runtime busy-state owner with upgrade-safe tests and review the existing open implementation PR against that contract.

## Reproduction Assessment

No high-confidence full reproduction of the heartbeat deadlock is present. Source inspection does prove the stale-`waiting` TaskFlow half of the report, but the exact `requests-in-flight` owner still needs gateway-level reproduction.

## Solution Assessment

No. A generic timeout is only a plausible mitigation; the best fix should define wait ownership/deadline semantics and release the concrete runtime or delivery resource that keeps heartbeat busy.
