## Label Justifications

- `P1`: The issue reports an unattended agent becoming permanently unavailable for scheduled and event-triggered work.
- `impact:crash-loop`: The reported agent stops producing heartbeats until manual database intervention and gateway restart.
- `impact:message-loss`: The follow-up says accepted voice/system events can remain queued and unconsumed while heartbeat dispatch is blocked.
- `impact:session-state`: The report centers on persisted TaskFlow state, in-memory busy state, and pending delivery state failing to converge.

## Summary

Keep this issue open: current main still allows managed TaskFlows to remain in `waiting` indefinitely, and neither maintenance nor heartbeat recovery defines safe abandonment for those waits. The direct `flow_runs` to `requests-in-flight` link is still not fully proven, so the remaining work needs maintainer-guided runtime diagnosis and a TaskFlow deadline/abandonment decision rather than cleanup closure.

## What This Changes

The issue requests expiry or cleanup for TaskFlows stuck in `waiting` so orphaned callback waits cannot permanently block agent heartbeats.

## Best Possible Solution

Define explicit controller-owned wait abandonment or deadline semantics, then wire recovery into the gateway/runtime owner that actually holds `requests-in-flight` and prove legitimate long waits are preserved.

## Reproduction Assessment

No high-confidence current-main reproduction of the full deadlock was established in this read-only review. Source inspection does show `waiting` TaskFlows can persist indefinitely and audit only warns, but the specific `requests-in-flight` owner still needs runtime diagnostics.

## Solution Assessment

No. A generic age-based expiry is only a plausible mitigation; the safer fix needs explicit wait-abandonment semantics and gateway/runtime recovery tied to the concrete busy-state owner.
