## Label Justifications

- `P1`: The report describes a credible unattended deployment failure where heartbeats and event-triggered agent work stop until manual database and gateway intervention.
- `impact:crash-loop`: The affected agent is described as functionally unavailable until manual recovery, which is a process-availability class failure even without a literal crash loop.
- `impact:message-loss`: Queued wake and voice events can be accepted but not consumed while heartbeat dispatch remains blocked.
- `impact:session-state`: The suspected failure spans persisted TaskFlow rows, in-memory busy state, and pending final-delivery state that do not converge across restart.

## Summary

Keep this issue open: current main and the latest release still allow managed TaskFlows to remain in `waiting` indefinitely, while the reported heartbeat deadlock needs maintainer-guided reproduction before a safe deadline or abandonment policy is chosen.

## What This Changes

The issue requests recovery for orphaned TaskFlows stuck in `waiting` so they cannot permanently block future agent heartbeats.

## Best Possible Solution

Define explicit controller-owned wait abandonment or deadline semantics, then wire recovery into the gateway startup/runtime owner that actually retains busy state and prove legitimate long waits plus failed final delivery are handled separately.

## Reproduction Assessment

No. Source inspection shows `waiting` TaskFlows can persist indefinitely and maintenance only warns, but I did not establish a current-main end-to-end path where those rows alone hold a heartbeat `requests-in-flight` busy state.

## Solution Assessment

No. A generic N-minute expiry is only a plausible mitigation; the safer fix is explicit wait-abandonment semantics tied to the actual gateway/runtime resource that blocks heartbeat dispatch.
