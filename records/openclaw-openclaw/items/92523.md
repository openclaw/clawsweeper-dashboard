## Summary

Keep this issue open: it reports a credible persistent availability failure on the latest release, and no merged fix or canonical issue covers abandoned managed TaskFlows. Current source does not support the claimed direct `flow_runs`-to-heartbeat counter relationship, so the exact busy-state owner needs reproduction before implementing the proposed TTL.

## What This Changes

Prevent abandoned managed TaskFlows from leaving agent sessions permanently unable to process scheduled or event-triggered heartbeats.

## Best Possible Solution

Reproduce the abandoned callback with diagnostics for TaskFlow state, linked tasks, active reply/embedded operations, command-lane snapshots, session delivery state, and restart behavior; then make the owning controller resolve or cancel the actual runtime resource and atomically transition the flow using a defined deadline or abandonment signal.

## Reproduction Assessment

No, not yet. The production observations are detailed and current source permits indefinite waiting records, but the inspected heartbeat code does not derive busy state directly from `flow_runs`, so a diagnostic current-main reproduction must identify the actual retained runtime resource.

## Solution Assessment

No. A generic N-minute expiry is only a plausible mitigation; the maintainable fix is to define controller abandonment semantics and release the concrete active-run or delivery resource that causes heartbeat deferral.
