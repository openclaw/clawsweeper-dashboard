## Summary

Keep this issue open: current `main` and the latest release still leave `waiting` TaskFlows nonterminal, and the linked implementation PR is open with unresolved proof and compatibility blockers. The production report is credible, but the exact heartbeat busy-state resource still needs a gateway-level reproduction before maintainers accept a default expiry semantics change.

## What This Changes

The issue asks core to recover orphaned TaskFlow `waiting` rows so abandoned callback waits cannot permanently block agent heartbeats.

## Best Possible Solution

Define controller-owned wait abandonment or deadline semantics, then land or replace https://github.com/openclaw/openclaw/pull/92648 with gateway-level proof that the actual busy-state owner is released without breaking legitimate long waits.

## Reproduction Assessment

No high-confidence full live reproduction of the heartbeat deadlock is established. Source inspection does prove current main can retain stale `waiting` TaskFlows indefinitely, but the exact `requests-in-flight` owner still needs diagnostic reproduction.

## Solution Assessment

No, a generic N-minute expiry is only a plausible mitigation. The best fix should define explicit wait-abandonment semantics and prove the gateway/startup/runtime path releases the actual busy resource.
