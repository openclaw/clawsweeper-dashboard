## Summary

Keep this issue open. Current main detects stale waiting TaskFlows but does not expire or reconcile them, and no merged fix or canonical issue owns that gap. However, the inspected source does not support a direct `flow_runs`-to-heartbeat counter relationship, so the retained live resource must be identified before implementing a safe repair.

## What This Changes

Prevent abandoned waiting TaskFlows from indefinitely blocking heartbeat dispatch and agent wake processing.

## Best Possible Solution

Define controller-owned wait deadlines and abandonment semantics, reconcile expired waits atomically through the existing TaskFlow maintenance owner, and add a regression proving that an abandoned callback releases the exact live activity state observed by heartbeat admission.

## Reproduction Assessment

No, not end to end. Current source proves that waiting flows can remain stale and heartbeats can skip for live activity, but it does not show that a `flow_runs` row directly owns that activity; diagnostics must capture command lanes, active reply/embedded runs, linked tasks, and pending final delivery before and after restart.

## Solution Assessment

No. A generic N-minute startup sweep is only a mitigation; the maintainable solution is controller-specific abandonment semantics plus cleanup of the concrete runtime resource causing heartbeat deferral.
