## Summary

Keep this issue open: current `main` still has no recovery path for stale `waiting` TaskFlows, but the safe fix needs maintainer judgment because the direct heartbeat busy-state owner is not proven and the prior blanket-timeout PR was closed unmerged for compatibility/proof reasons.

## What This Changes

The issue asks OpenClaw core to prevent orphaned TaskFlows stuck in `waiting` from permanently blocking future heartbeat dispatch.

## Best Possible Solution

Define explicit controller-owned wait abandonment or deadline semantics, then wire recovery into the gateway/runtime owner that actually releases the stale busy-state resource and prove legitimate long waits are preserved.

## Reproduction Assessment

Partly: current source shows stale `waiting` TaskFlows are audited but not expired, and current tests lock that diagnostic behavior. I did not establish a high-confidence current-main live reproduction showing `flow_runs` rows alone keep `runHeartbeatOnce` returning `requests-in-flight`.

## Solution Assessment

No. A generic N-minute maintenance expiry is only a plausible mitigation; the better fix needs explicit wait-abandonment semantics and gateway/runtime recovery tied to the actual busy-state owner.
