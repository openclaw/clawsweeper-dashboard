- `impact:session-state`: The issue centers on persisted TaskFlow wait state and runtime busy/session state failing to converge across restarts.

## Summary

Keep this issue open: current main and the latest shipped tag still leave `waiting` TaskFlows non-terminal with no expiry path, and the only linked repair is an open PR that has not merged and still needs maintainer review of wait-abandonment semantics and gateway-level proof.

## What This Changes

The issue requests a core recovery path for orphaned `waiting` TaskFlows so abandoned callback waits do not leave agent heartbeats stuck behind `requests-in-flight`.

## Best Possible Solution

Define explicit TaskFlow wait-abandonment semantics, then wire startup or runtime recovery into the actual gateway busy-state owner with tests and real gateway proof before closing this issue.

## Reproduction Assessment

No high-confidence full heartbeat-deadlock reproduction is present. Source inspection shows current main can retain `waiting` TaskFlows indefinitely and maintenance does not touch them, but the direct `requests-in-flight` owner still needs gateway-level proof.

## Solution Assessment

No, the issue's suggested generic TTL is only a plausible mitigation. The better fix is explicit controller-owned deadline or abandonment semantics plus recovery in the gateway/runtime path that actually blocks heartbeat dispatch.
