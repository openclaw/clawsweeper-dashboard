## Label Justifications

- `P1`: The report describes a persistent unattended-agent heartbeat deadlock on the latest release, with accepted wake events not producing turns.
- `impact:crash-loop`: The affected agent can become functionally unavailable until manual database intervention and a gateway restart.
- `impact:message-loss`: Queued wake or voice-input events can remain unconsumed while heartbeat dispatch stays blocked.
- `impact:session-state`: Persisted TaskFlow wait state, runtime busy state, and pending delivery state allegedly fail to converge across restarts.

## Summary

Keep this issue open: current main and v2026.6.6 still retain old `waiting` TaskFlows without expiring them, and the linked fix PR is open, unmerged, and already flagged for proof plus wait-semantics review.

## What This Changes

Requests a core recovery path for orphaned managed TaskFlows stuck in `waiting` so they do not leave agent heartbeats permanently skipped as `requests-in-flight`.

## Best Possible Solution

Review or replace the linked PR with a fix that defines controller-owned wait abandonment and wires recovery into the actual gateway startup/runtime busy-state owner, with tests preserving legitimate long waits.

## Reproduction Assessment

Partially: source inspection shows stale `waiting` TaskFlows are retained and only audited, but there is no high-confidence current-main live reproduction tying a `flow_runs` row to permanent heartbeat busy state.

## Solution Assessment

No: a blanket TTL is only a plausible mitigation; the safer fix needs explicit wait-abandonment semantics and recovery tied to the actual busy-state owner.
