## Summary

Keep this issue open: current main and v2026.6.6 still leave stale `waiting` TaskFlows as durable nonterminal state, and the only open repair PR is blocked on proof and lifecycle semantics. Source does not prove the issue’s direct `flow_runs` to heartbeat-counter claim, but it does support the core maintenance gap and the linked PR is not merged.

## What This Changes

Issue reports that TaskFlow rows stuck in `waiting` status can permanently keep an agent session busy and prevent heartbeat dispatch.

## Best Possible Solution

Define core-owned TaskFlow wait abandonment semantics, then land a gateway or maintenance recovery path that preserves valid long waits and proves heartbeat admission is unblocked after expiry.

## Reproduction Assessment

Partly yes from source: current main and v2026.6.6 can retain `waiting` TaskFlows indefinitely and maintenance only warns about stale waits. The full deadlock from those rows to heartbeat `requests-in-flight` is not high-confidence reproduced because the inspected heartbeat path does not directly read `flow_runs`.

## Solution Assessment

Unclear. Expiring abandoned waits is the right area, but the best fix needs explicit wait-deadline or orphan semantics plus proof against the actual heartbeat busy-state owner.
