## Summary

Keep open: current main and v2026.6.6 still have the source-backed idle-by-default state, and the remaining work is a maintainer product/UX decision rather than an already-implemented fix or a narrow strict bug repair.

## What This Changes

The issue asks OpenClaw to make the default memory-idle state explicit or produce memory artifacts sooner when dreaming is disabled and compaction thresholds are not reached.

## Best Possible Solution

Define one maintainer-approved memory UX contract that explains idle state in status/onboarding and chooses whether reset/daily flushing or threshold policy should create useful artifacts sooner, without adding competing memory behaviors.

## Reproduction Assessment

Yes, at source level: current main still gates pre-compaction memory flush by token/transcript thresholds and dreaming remains opt-in, while status surfaces do not explain that idle state as an expected condition. I did not run a live memory session because this cleanup review is read-only.

## Solution Assessment

Unclear: diagnostics/onboarding, reset flushing, threshold scaling, and default behavior changes are all plausible, but maintainers need to choose the product contract before implementation. A narrow one-branch patch would risk creating another memory behavior without resolving the UX policy.
