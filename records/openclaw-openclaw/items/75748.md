## Summary

Keep this draft PR open. The proposed direction matches the reported provider-default overwrite bug, but the new auth-patch preservation helper still lets provider patches add fallback models when the user explicitly configured an empty fallback list, so the patch is not yet correct.

## What This Changes

The PR updates provider preset helpers, provider auth patch merging, API-key setup, OpenAI/Moonshot defaults, docs, changelog, and targeted tests so provider setup registers models additively unless an explicit set-default path runs.

## Best Possible Solution

Keep the additive provider setup behavior, but preserve the existing fallback field whenever it is present, including an explicit empty array, and add a regression test for auth config patches with `fallbacks: []`.

## Reproduction Assessment

Yes. Apply a provider auth config patch with fallback defaults to a config whose existing `agents.defaults.model` is `{ primary: "custom/model", fallbacks: [] }`; the PR helper keeps the primary but adopts the provider fallback list.

## Solution Assessment

No. The proposed approach is otherwise maintainable, but the preservation helper must distinguish an absent fallback field from an explicit empty fallback selection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve explicit empty fallback lists:** `src/plugins/provider-auth-choice-helpers.ts:127-129`
  - body: When `preserveExistingDefaultModel` is true, this helper only restores existing fallbacks if `existingFallbacks.length > 0`. A user who explicitly configured `fallbacks: []` will keep their primary but still inherit the provider patch's fallback list, which is the same silent default-selection drift this PR is meant to prevent. Preserve the fallback field whenever it is present, including an empty array, and cover that case in `provider-auth-choice-helpers.test.ts`.
  - confidence: 0.9
