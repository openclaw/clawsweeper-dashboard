The PR changes provider preset and provider-auth config patch handling so provider defaults register additively, updates OpenAI/Moonshot behavior, and adds tests, docs, and changelog notes.

## Best Possible Solution

Keep the additive provider-default approach, but make preservation explicitly option-gated, keep generic core tests provider-neutral, remove the duplicate release note, and land only after targeted and changed-lane checks pass.

## Reproduction Assessment

Yes. The original bug reproduces from current-main helper paths by applying an OpenAI/Moonshot preset or provider `configPatch` over a config with a custom primary/fallback; the PR defects are directly visible from PR-head source and the boundary test contract.

## Solution Assessment

No, not as implemented. The desired direction is maintainable, but the raw merge helper now preserves defaults unconditionally, so explicit replacement and migration paths can keep stale primary/fallback data.
