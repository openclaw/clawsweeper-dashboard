Keep open: the linked bug remains source-reproducible on current main for provider preset onboarding, and this PR is a plausible fix direction but still has blocking implementation issues, a main-branch conflict, and no after-fix real behavior proof from the contributor.

## What This Changes

The PR changes provider onboarding/auth config helpers, OpenAI/Moonshot defaults, docs, changelog, and tests so provider setup adds models without replacing existing default model primary/fallbacks unless an explicit default-setting path is used.

## Best Possible Solution

Rebase and land an equivalent narrow fix that preserves existing primary/fallbacks for provider presets and non-explicit auth patches while keeping explicit `models set` and `models auth login --set-default` replacement semantics and provider-neutral tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main by passing an existing `agents.defaults.model.primary` through `applyProviderConfigWithDefaultModelPreset`, `applyOpenAIConfig`, or `applyMoonshotConfig` and observing the call to `applyAgentDefaultModelPrimary`; I did not run tests because this review is read-only.

## Solution Assessment

No, not as currently implemented. The direction is right, but preservation needs to be gated at the call site instead of baked into raw merge behavior, the generic fixture needs provider-neutral model refs, and the duplicate changelog entry should be removed.
