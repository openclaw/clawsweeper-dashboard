## Summary

Keep this PR open, but do not merge it as-is. Current main still has the reported provider-default overwrite paths, and the PR moves in the right direction with docs, changelog, and targeted tests, but the head patch still lets provider auth patches add fallback models when the existing default selection has `fallbacks: []` or no fallback field.

## What This Changes

The PR adds preserve-existing-default-model options across provider preset/auth helpers, updates OpenAI and Moonshot setup behavior, and documents/tests that provider auth setup should not replace an existing default model unless an explicit default-setting path is used.

## Best Possible Solution

Keep the additive provider-default behavior, but change the preservation helper so existing default model selection is preserved exactly enough that provider patches cannot add or replace fallback behavior when a user already has a configured primary, including empty fallback and primary-only configs. Keep explicit replacement limited to `openclaw models set`, `openclaw models auth login --set-default`, and opt-in migration replacement paths.

## Reproduction Assessment

Yes. Current main can be reproduced at helper level with provider preset/auth config patches against an existing `agents.defaults.model`; the PR-specific bug is reproducible by applying a provider patch with fallbacks over an existing `{ primary, fallbacks: [] }` or primary-only config.

## Solution Assessment

No, not as-is. The PR direction is the narrow maintainable fix, but the auth-patch preservation logic should restore the existing fallback field or absence before the patch is mergeable.
