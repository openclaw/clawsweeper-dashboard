## Summary

Keep open: the cleanup intent is understandable, but the patch changes a live doctor write-decision signal. Current code uses `pendingChanges` to make repair-mode config mutations return `shouldWriteConfig: true`, so clearing it in repair mode can make repaired config stay in memory without being persisted by callers that rely on `loadAndMaybeMigrateDoctorConfig`.

## What This Changes

The PR changes doctor config mutation state so repair-mode mutations set `pendingChanges` to false and updates related test expectations.

## Best Possible Solution

Separate the semantic state cleanup from the persistence decision, or keep `pendingChanges` as the write-needed signal until a replacement flag is threaded through `finalizeDoctorConfigFlow` and covered at the `loadAndMaybeMigrateDoctorConfig` boundary.

## Reproduction Assessment

Yes. Source inspection shows a repair-mode mutation flows through `pendingChanges` into `finalizeDoctorConfigFlow`, and that value is what makes `loadAndMaybeMigrateDoctorConfig` request a config write.

## Solution Assessment

No. The proposed one-line change matches part of the field comment, but it is not the best fix because it changes the persistence signal without replacing it elsewhere.
