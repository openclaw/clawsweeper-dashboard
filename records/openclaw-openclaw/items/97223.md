## Summary

Keep this PR open, but do not merge it as-is: the changed flag is also the repair-mode writeback signal, so clearing it after repair would make some `openclaw doctor --fix` config repairs stop being persisted.

## What This Changes

The branch changes `applyDoctorConfigMutation` to clear `pendingChanges` in repair mode and updates doctor repair sequencing tests to expect repaired mutations to have no pending changes.

## Best Possible Solution

Keep repair mutations marked as needing writeback, or introduce a separate internal flag only if the naming needs cleanup, with config-flow coverage proving repair-mode writes still happen.

## Reproduction Assessment

Yes, from source inspection: the patched false value flows into `finalizeDoctorConfigFlow`, which writes repaired config only when `shouldRepair && pendingChanges`. Existing config-flow coverage also expects a repair routed through this helper to keep `shouldWriteConfig: true`.

## Solution Assessment

No. The patch treats `pendingChanges` as only a preview marker, but current code uses it as the final persisted-write signal; the safer fix is to keep that signal true or split the state names with full config-flow coverage.
