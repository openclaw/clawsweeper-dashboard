## Summary

Keep open: current main and v2026.6.10 still generate the Windows update restart helper with a manual argument joiner assigned to `ProcessStartInfo.Arguments`. The current task-name inputs are guarded, so this is not a proven live exploit, but the requested command-line hardening remains unsolved and belongs with maintainer security triage.

## What This Changes

The issue requests hardening the generated Windows update restart helper so schtasks arguments cannot be malformed by embedded double quotes in raw command-line strings.

## Best Possible Solution

Keep this issue open and harden the owning Windows restart helper with a compatibility-proven argument-passing fix plus focused regression coverage for embedded quotes.

## Reproduction Assessment

Yes for the helper hardening gap from source inspection: current main still joins schtasks arguments into a raw string and assigns it to `ProcessStartInfo.Arguments`. No high-confidence live Windows exploit path is established because current task-name input is guarded before the helper runs.

## Solution Assessment

Unclear as a final repair choice: the Windows restart helper is the right owner boundary, but maintainers should choose between `ArgumentList` and explicit raw-string escaping only after proving compatibility with the Windows PowerShell runtime used by the generated script.
