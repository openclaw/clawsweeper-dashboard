## Summary

Keep open: current main still has the raw ProcessStartInfo.Arguments joiner, and the quoting concern remains plausible, but the current user-controlled task-name path is guarded, so this should be handled as bounded Windows restart hardening with maintainer security review.

## What This Changes

The issue asks OpenClaw to harden the Windows update restart helper so embedded double quotes cannot corrupt or inject schtasks.exe arguments.

## Best Possible Solution

Keep the existing task-name validation and harden the generated Windows restart helper with contract-proven argument escaping or a compatible ArgumentList implementation, plus focused regression coverage for embedded quotes and the existing safe-task-name behavior.

## Reproduction Assessment

No, not as a high-confidence current-user exploit path. Source inspection shows the helper would mishandle embedded double quotes if such an argument reached it, while current generated calls pass fixed flags and a task name filtered by isWindowsTaskNameSafe.

## Solution Assessment

Unclear until maintainers choose the Windows compatibility target. Escaping embedded quotes in the current raw string is likely the narrowest repair for powershell.exe compatibility; ArgumentList is cleaner only if the generated helper can rely on that runtime API.
