## Summary

Keep open: current `main` and `v2026.6.8` still build the Windows restart helper's `schtasks.exe` launch through a raw `ProcessStartInfo.Arguments` string. The task-name source is guarded, but the helper remains security-sensitive hardening rather than implemented cleanup.

## What This Changes

The issue reports a Windows update restart-helper hardening gap where embedded double-quotes are not escaped when constructing `ProcessStartInfo.Arguments`.

## Best Possible Solution

Keep this in maintainer security triage and either switch the PowerShell helper to an argument-list API with focused regression coverage, or explicitly document why the guarded task-name-only call sites are an acceptable boundary.

## Reproduction Assessment

Yes from source inspection, but not from a live Windows run. Current `main` still emits a helper that turns a string array into one raw `ProcessStartInfo.Arguments` string, while current call sites feed only a validated scheduled-task name.

## Solution Assessment

Unclear as a final security decision: using `ArgumentList` is the cleanest hardening direction supported by .NET docs, but maintainers should decide whether the current guarded call-site boundary is sufficient before changing the shipped restart helper.
