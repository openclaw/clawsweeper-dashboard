## Summary

Keep open: current main still builds a raw `ProcessStartInfo.Arguments` string without escaping embedded double quotes, but the current user-controlled task name is guarded, so this is bounded Windows restart security hardening rather than a proven live exploit path.

## What This Changes

Harden the generated Windows update restart script so embedded double quotes cannot malformedly split or inject into the `schtasks.exe` command line.

## Best Possible Solution

Keep the task-name guard and harden the generated PowerShell helper with proven-compatible argument construction plus focused regression coverage for embedded quotes and the existing safe-task-name path.

## Reproduction Assessment

No high-confidence current-user exploit reproduction is established. Source inspection shows an embedded quote would be mishandled if it reached the helper, but current generated calls pass fixed flags and a task name filtered by `isWindowsTaskNameSafe`.

## Solution Assessment

Unclear until maintainers choose the compatibility target. Escaping embedded quotes in the existing raw string is likely the narrowest Windows PowerShell-compatible repair; `ArgumentList` is cleaner only if runtime availability is proven.
