## Summary

Keep open: current main still has the raw `ProcessStartInfo.Arguments` helper, and the argument-quoting concern is plausible, but the current user path validates the Windows task name before the helper is called. This should be treated as bounded Windows restart hardening, not auto-closed and not an automatic security fix without maintainer review.

## What This Changes

The issue asks to harden the Windows update restart `schtasks` launcher so embedded double quotes in generated process arguments cannot malformedly split or inject arguments.

## Best Possible Solution

Keep the safe task-name gate and either implement correct Windows command-line escaping in the generated PowerShell helper or switch to an argument-list API only after proving that the generated script runtime supports it, with focused regression coverage for embedded quotes and existing safe-task-name behavior.

## Reproduction Assessment

No, not as a high-confidence current-user exploit path. Source inspection shows the helper would mishandle embedded double quotes, but the current generated calls only pass fixed flags and a task name filtered by `isWindowsTaskNameSafe`.

## Solution Assessment

Unclear until a maintainer chooses the compatibility target. Escaping embedded quotes in the existing raw string is the narrowest likely fix for `powershell.exe`; using `ArgumentList` is cleaner only if runtime availability is proven for the generated Windows helper.
