## Summary

Keep open: current main still contains the raw `ProcessStartInfo.Arguments` construction, and the embedded-quote concern remains plausible. The current user path is guarded by task-name validation, so this is bounded Windows restart hardening that needs maintainer security review rather than auto-close or automatic repair.

## What This Changes

The issue asks OpenClaw to harden the generated Windows update restart helper so `schtasks.exe` arguments cannot be malformed by embedded double quotes.

## Best Possible Solution

Keep the task-name validation and add focused hardening in the generated Windows restart helper, with regression coverage for embedded quotes and current safe-task-name behavior.

## Reproduction Assessment

No, not as a high-confidence current-user exploit path. Source inspection shows the helper would mishandle embedded quotes if such an argument reached it, but the current generated calls only pass fixed flags and a validated task name.

## Solution Assessment

Unclear until maintainers choose the compatibility target. Escaping embedded quotes in the existing raw string is likely the narrowest Windows PowerShell-compatible fix, while `ArgumentList` is cleaner if runtime support is proven.
