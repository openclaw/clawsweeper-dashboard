## Summary

Keep open: current main and v2026.6.8 still generate a Windows restart helper that manually joins schtasks arguments into ProcessStartInfo.Arguments. The present task-name path is guarded, so this is security hardening rather than a proven live exploit, but the requested hardening is neither implemented nor superseded.

## What This Changes

The issue reports that the Windows restart helper manually joins schtasks arguments into ProcessStartInfo.Arguments without escaping embedded double quotes.

## Best Possible Solution

Keep the issue open for maintainer security triage and harden the generated Windows restart helper with compatibility-proven structured argument passing or correct raw-string escaping plus focused regression coverage.

## Reproduction Assessment

Yes from source inspection for the helper itself: current main still joins a raw schtasks argument string through Join-OpenClawProcessArguments and assigns it to ProcessStartInfo.Arguments. No high-confidence live Windows user exploit path is established because the current task-name input is guarded.

## Solution Assessment

Unclear as a final repair choice: the narrow owner boundary is the generated Windows restart helper, but maintainers should choose between ArgumentList and correct raw-string escaping only after confirming Windows PowerShell runtime compatibility.
