## Summary

Keep open: current main still builds a manually joined raw string for `ProcessStartInfo.Arguments`. The current user-controlled task name is filtered before reaching that helper, so this is bounded Windows restart hardening rather than a proven current-user exploit, but the source-level concern remains real.

## What This Changes

The issue asks OpenClaw to harden the generated Windows update restart PowerShell so `schtasks.exe` arguments with embedded double quotes cannot malformedly split or inject into `ProcessStartInfo.Arguments`.

## Best Possible Solution

Keep the task-name safety gate and add quote-safe argument construction for the generated Windows helper, with regression coverage for embedded double quotes plus the existing safe-task-name path.

## Reproduction Assessment

No, not as a high-confidence current-user exploit path. Source inspection shows the helper would mishandle embedded double quotes, but the current generated calls pass fixed flags plus a task name rejected if it contains quotes.

## Solution Assessment

Unclear until maintainers choose the compatibility target. The narrowest likely fix is to escape embedded quotes in the existing raw string unless `ArgumentList` is proven safe for the generated Windows PowerShell runtime.
