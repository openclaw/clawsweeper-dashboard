## Summary

Keep open: current main and the latest release still generate the Windows PowerShell helper with a manually joined raw `ProcessStartInfo.Arguments` string. The current task-name path is guarded, so this remains security hardening rather than a proven live exploit, but the requested hardening is not implemented or superseded.

## What This Changes

The issue requests hardening Windows restart-helper schtasks argument construction so embedded double quotes cannot corrupt or inject into the raw `ProcessStartInfo.Arguments` command line.

## Best Possible Solution

Keep the task-name validation and harden the generated Windows helper with compatibility-proven safe argument handling, plus regression coverage for embedded double quotes and a focused Windows restart proof.

## Reproduction Assessment

No, not as a high-confidence current-user exploit path. Source inspection shows the helper would mishandle embedded double quotes if an unguarded argument reached it, but current generated calls pass fixed flags and a task name filtered by `isWindowsTaskNameSafe`.

## Solution Assessment

Unclear. The owner boundary is the restart helper, but maintainers should choose between correct raw-string escaping and a proven-compatible argument-list API after checking the Windows PowerShell runtime target.
