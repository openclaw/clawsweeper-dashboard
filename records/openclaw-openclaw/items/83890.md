## Summary

Keep open: current main and the latest stable release still generate a Windows update restart helper that manually joins schtasks arguments into ProcessStartInfo.Arguments without escaping embedded double quotes. The current task-name path is guarded, so this remains a security-hardening issue rather than a proven live exploit, but the requested hardening is neither implemented nor superseded.

## What This Changes

The issue requests hardening the Windows update restart helper so schtasks arguments are passed without unsafe raw command-line string concatenation.

## Best Possible Solution

Move the generated Windows helper from raw ProcessStartInfo.Arguments string assembly to ArgumentList, then add focused script-content coverage for embedded double quotes while preserving the current task-name validation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for a live user path: source inspection shows the raw ProcessStartInfo.Arguments joiner can mishandle embedded double quotes if such an argument reaches it, but current main guards the task-name input and I did not establish a live Windows update exploit path.

## Solution Assessment

Is this the best way to solve the issue? Yes: ArgumentList is the narrowest maintainable repair because it removes custom command-line quoting from the generated PowerShell helper instead of trying to preserve a bespoke escaping function.
