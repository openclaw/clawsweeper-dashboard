## Summary

Keep open: current main still builds a raw `ProcessStartInfo.Arguments` string in the generated Windows restart helper, and the quote-escaping concern remains source-reproducible. Current task-name validation limits the present user-controlled path, so this should be handled as bounded security hardening with maintainer review rather than auto-closed or automatically repaired.

## What This Changes

The issue requests hardening the Windows update restart script so schtasks arguments with embedded double quotes cannot malformedly change the raw process command line.

## Best Possible Solution

Keep the task-name guard, add correct embedded double-quote handling or a proven compatible argument-list path in the generated PowerShell helper, and cover both safe task names and embedded quote cases in `restart-helper.test.ts`.

## Reproduction Assessment

No, not as a high-confidence current-user exploit path. Source inspection shows the helper would mishandle an embedded double quote if one reached it, but current generated calls pass fixed flags and a task name filtered by `isWindowsTaskNameSafe`.

## Solution Assessment

Unclear until maintainers choose the compatibility target. Escaping embedded quotes in the existing raw string is likely the narrowest Windows PowerShell-compatible fix, while `ArgumentList` needs runtime compatibility proof before replacing `Arguments`.
