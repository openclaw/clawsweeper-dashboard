## Summary

Keep open: current main still builds a raw `ProcessStartInfo.Arguments` string through a helper that quotes whitespace but not embedded double quotes. The present user-controlled task name is guarded, so this is bounded Windows restart hardening rather than a proven current exploit path.

## What This Changes

The issue requests hardening the Windows update restart helper's PowerShell `schtasks.exe` argument construction so embedded double quotes cannot break or inject arguments.

## Best Possible Solution

Keep the task-name guard and harden the generated PowerShell process invocation with current-compatible quote escaping or a runtime-proven `ArgumentList` implementation plus focused regression coverage.

## Reproduction Assessment

No, not as a high-confidence current-user exploit path. Source inspection shows the helper would mishandle embedded double quotes, but current generated calls pass only fixed flags and a task name filtered by `isWindowsTaskNameSafe`.

## Solution Assessment

Unclear until the Windows PowerShell runtime target is confirmed. Escaping embedded quotes in the existing raw string is likely the narrowest compatible fix, while `ArgumentList` is preferable only if runtime availability is proven.
