## Summary

Keep open: current main and the latest release still build the Windows schtasks command through a raw `ProcessStartInfo.Arguments` string, while the current user-controlled task-name path is guarded, so this remains bounded security hardening rather than an implemented or directly reproducible exploit.

## What This Changes

The issue asks OpenClaw to harden the Windows update restart helper so embedded double quotes cannot malform schtasks process arguments.

## Best Possible Solution

Keep the task-name safety gate and harden the generated Windows helper with a proven-compatible argument-list API or correct raw command-line escaping, plus regression coverage for embedded double quotes.

## Reproduction Assessment

No, not as a high-confidence current-user exploit path. Source inspection shows the helper would mishandle embedded double quotes if an unguarded argument reached it, but the current generated calls pass fixed flags and a task name filtered by `isWindowsTaskNameSafe`.

## Solution Assessment

Unclear. The desired end state is clear, but maintainers should choose between raw-string escaping and `ArgumentList` after confirming the generated Windows helper's runtime compatibility target.
