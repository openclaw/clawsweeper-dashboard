## Summary

Keep open: current main still supports plugin-side Gateway method registration but does not expose a plugin-callable Gateway method/runtime dispatcher, and the remaining work is a public SDK/security design decision rather than a cleanup close.

## What This Changes

The issue requests a plugin SDK/runtime API that lets one plugin call Gateway methods registered by another plugin.

## Best Possible Solution

Design a narrow, documented plugin Gateway-call seam or service-registry alternative with explicit method/capability policy, plugin caller attribution, SDK baseline/docs updates, and authorization tests.

## Reproduction Assessment

Yes. Source inspection on current main is a high-confidence way to reproduce the API gap: plugins can register Gateway methods, but no callGatewayMethod or api.runtime.gateway caller exists.

## Solution Assessment

Unclear for the exact requested API. A plugin-to-Gateway call seam may be the right direction, but a raw arbitrary dispatcher is not clearly the best solution without allowlisting/capability checks and caller identity propagation.

## Security Review

Status: needs_attention

Summary: The requested public runtime surface is security-sensitive because it could let plugin code invoke Gateway RPCs that currently sit behind operator scopes.

Concerns:

- **[medium] Gate plugin Gateway dispatch:** `src/gateway/method-scopes.ts:286`
  - body: Any plugin-facing dispatcher needs an explicit allowlist or capability contract; Gateway method authorization currently maps unknown methods to admin scope, and exposing a raw dispatcher would make that policy part of the plugin trust boundary.
  - confidence: 0.88
- **[medium] Preserve plugin caller attribution:** `src/plugins/registry.ts:2263`
  - body: The current per-plugin runtime proxy wraps subagent and LLM calls with plugin identity, while other runtime properties pass through unchanged. A new Gateway namespace would need equivalent identity propagation for ownership checks and auditability.
  - confidence: 0.84
