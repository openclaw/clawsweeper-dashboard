## Summary

Keep open: current main and the latest release still let plugins register Gateway methods but do not expose a general plugin-side caller for another plugin's registered method. The existing dispatch helper is intentionally reserved for authenticated plugin HTTP routes, so this remains a valid SDK capability request that needs maintainer API and security design.

## What This Changes

The issue requests a public plugin API for invoking Gateway methods registered by other plugins, such as a proposed `api.callGatewayMethod(method, params)` helper.

## Best Possible Solution

Design a narrow documented plugin interop or Gateway-call runtime seam with caller attribution, method allowlisting or scope enforcement, plugin-handler routing, and focused SDK/security regression coverage.

## Reproduction Assessment

Yes for the capability gap. Source inspection on current main shows `registerGatewayMethod` exists, `PluginRuntime` lacks a Gateway caller namespace, and the only public dispatch helper rejects callers outside an entitled plugin HTTP request scope.

## Solution Assessment

Unclear for the exact requested API. A generic `api.callGatewayMethod` is plausible, but the safer path needs maintainer-approved SDK shape, caller attribution, method policy, and security tests before implementation.
