## Summary

Keep open: current main still exposes plugin-side gateway method registration but no plugin-side gateway method caller, and the open implementation candidate needs maintainer API and authorization review before this public runtime surface changes.

## What This Changes

The issue requests a plugin SDK method for one plugin to invoke gateway methods registered by another plugin.

## Best Possible Solution

Resolve this through maintainer SDK/runtime design review, either by landing a revised #46485 or a narrower documented dispatcher seam with API baseline updates and authorization tests.

## Reproduction Assessment

Yes. Source inspection on current main is a high-confidence reproduction path for this API gap: registration exists, callGatewayMethod has no matches, and PluginRuntime has no gateway namespace.

## Solution Assessment

Unclear for the exact requested API name. A runtime gateway dispatcher may be the narrowest maintainable seam, but caller identity, scopes, docs, API baselines, and tests need maintainer review first.
