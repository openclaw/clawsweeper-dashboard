## Summary

Keep open: current main and the latest release still lack a plugin-side Gateway caller, while the existing bridge is intentionally limited to entitled plugin HTTP routes. The closed raw-dispatch PR does not solve the security/API design, and the newer host-read RPC issue overlaps but does not replace this inter-plugin registered-method request.

## What This Changes

Requested a public plugin SDK/runtime API that lets one plugin call registered Gateway methods or host Gateway RPCs from plugin code.

## Best Possible Solution

Design a narrow documented plugin interop or Gateway-call seam with caller identity, method allowlisting/scope policy, plugin-handler routing, and SDK/docs/regression coverage, coordinated with the host-read RPC design.

## Reproduction Assessment

Yes for the capability gap: source inspection on current main shows plugin code can register Gateway methods but has no public plugin-side caller. The only current dispatch helper rejects callers outside an entitled authenticated plugin HTTP route.

## Solution Assessment

Unclear for the exact `api.callGatewayMethod` shape. A caller may be useful, but the maintainable solution needs maintainer-approved SDK shape, authorization policy, caller attribution, and allowlisting first.

Status: needs_attention
