## Summary

Keep open. Current main still has plugin-side gateway method registration but no plugin-side caller, and the related implementation candidate remains open rather than merged. The remaining work is a public Plugin SDK/Gateway runtime design question, not conservative cleanup.

## What This Changes

Add a plugin-side API such as `api.callGatewayMethod(...)` so one plugin can invoke gateway methods registered by another plugin.

## Best Possible Solution

Add a documented, tested plugin-side request/response seam that preserves plugin isolation while allowing intentional cross-plugin capability calls, with explicit caller identity, scope behavior, error semantics, and contract coverage. That could be a revised `api.runtime.gateway.dispatchMethod` or a narrower typed service/capability API if maintainers prefer not to expose the full Gateway RPC bus.
