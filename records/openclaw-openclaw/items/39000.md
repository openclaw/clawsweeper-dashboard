## Summary

Keep open: current main still lacks a public plugin-side API for invoking Gateway methods from plugin code, while the only public dispatch helper is deliberately limited to authenticated plugin HTTP request scope. This remains a valid SDK capability gap that needs product and security design before implementation.

## What This Changes

The issue requests a plugin API such as `api.callGatewayMethod()` so one plugin can invoke Gateway methods registered by another plugin.

## Best Possible Solution

Define a narrow documented plugin interop seam with explicit caller attribution, entitlement policy, allowed method scopes, plugin-handler routing behavior, and regression coverage for both allowed and denied calls.

## Reproduction Assessment

Yes. Source inspection on current main confirms plugins can register Gateway methods and can use a reserved authenticated-route dispatch helper, but there is no public plugin-side `api.callGatewayMethod()`, `runtime.gateway.dispatchMethod()`, or equivalent caller for the reported inter-plugin use case.

## Solution Assessment

Unclear for the exact proposed API. The best implementation should be narrower than arbitrary Gateway RPC access and must define plugin identity, scope enforcement, handler routing, and allowed method policy before code changes.

## Security Review

Status: needs_attention

Summary: The requested SDK capability is security-sensitive because plugin-side Gateway dispatch would need explicit authorization, caller attribution, and method-scope boundaries.

- **[medium] Avoid arbitrary Gateway RPC access from plugins:** `src/gateway/method-scopes.ts:175`
  - body: Gateway authorization defaults methods without a resolved scope to `operator.admin`, so exposing a generic plugin caller without an allowlist or entitlement policy could let plugin code attempt methods it should not invoke.
