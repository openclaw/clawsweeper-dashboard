## Summary

Keep open: current main still lacks a supported plugin-side caller for plugin-registered Gateway methods, and the related raw dispatcher PR was closed unmerged in favor of this canonical API/security discussion.

## What This Changes

Request a new plugin SDK/runtime API, proposed as `api.callGatewayMethod()`, so one plugin can invoke Gateway methods registered by another plugin.

## Best Possible Solution

Define a narrow documented plugin interop/Gateway-call seam with explicit method policy, plugin caller attribution, handler routing for plugin-owned methods, SDK baseline/docs updates, and focused regression tests.

## Reproduction Assessment

Yes for the API gap: current main exposes `registerGatewayMethod`, but source inspection and targeted search show no supported `callGatewayMethod` or `runtime.gateway` caller. This is a feature request rather than a broken existing contract.

## Solution Assessment

Unclear for the exact requested shape. A maintainable solution should be narrower than arbitrary Gateway RPC access and must include capability policy, plugin caller identity, and plugin handler routing.
