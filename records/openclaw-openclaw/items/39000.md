## Summary

Keep open: current main and v2026.6.10 still do not expose a general plugin-to-plugin Gateway method caller. The existing in-process dispatcher is deliberately limited to authenticated plugin HTTP routes, and this remains a public plugin SDK plus Gateway security-boundary decision.

## What This Changes

Request to add a plugin API so one plugin can invoke Gateway methods registered by another plugin from plugin code.

## Best Possible Solution

A maintainer-approved plugin SDK contract that exposes only intentional Gateway or inter-plugin capabilities with caller identity, authorization, lifecycle semantics, docs, and contract tests.

## Reproduction Assessment

Yes. Source inspection on current main shows plugins can register Gateway methods, but ordinary plugin runtime code has no matching caller; the only SDK dispatcher rejects callers outside an entitled HTTP-route scope.

## Solution Assessment

No for the raw proposed shape. A direct arbitrary callGatewayMethod API is too broad without method exposure policy, caller identity, authorization, lifecycle handling, and contract tests.
