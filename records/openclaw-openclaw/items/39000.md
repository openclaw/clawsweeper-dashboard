## Summary

Keep open: current main still exposes plugin-side Gateway method registration but no supported plugin-side caller, and the related implementation PR is still open with SDK authorization and caller-identity concerns unresolved.

## What This Changes

Request a plugin-facing API to call registered Gateway RPC methods from another plugin.

## Best Possible Solution

Define a narrow documented plugin interop/Gateway-call seam with explicit method policy, plugin caller attribution, SDK type/baseline updates, docs, and focused regression tests, then land or supersede the related PR.

## Reproduction Assessment

Yes for the API gap: source inspection on current main shows registerGatewayMethod exists, while callGatewayMethod and runtime.gateway dispatch are absent. This is a feature request rather than a broken existing contract.

## Solution Assessment

Unclear for the exact requested shape. A maintainable solution should be narrower than arbitrary Gateway RPC access and should include capability policy, allowlisting or scoped helpers, and plugin identity propagation.
