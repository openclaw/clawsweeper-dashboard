## Label Justifications

- `P2`: This is a valid normal-priority plugin composition gap, but it needs bounded architecture and security design rather than urgent regression repair.
- `impact:security`: The requested surface would change which privileged gateway operations loaded plugin code can invoke and how those calls are authorized and attributed.

## Summary

Current main has strengthened internal in-process gateway dispatch, but it remains deliberately private and the public plugin runtime still exposes only narrow subagent/node methods. The requested cross-plugin capability is still necessary, while the unrestricted proposed API needs maintainer product and security design because plugin identity, method policy, lifecycle, and typed contracts remain unresolved.

## What This Changes

Requests a supported plugin API for one plugin to invoke gateway methods registered by another plugin.

## Best Possible Solution

Keep this issue as the canonical design item and add a typed or capability-scoped inter-plugin request/response contract with caller plugin identity, explicit method exposure, lifecycle semantics, authorization, and focused SDK documentation and tests; keep unrestricted raw gateway dispatch internal.

## Reproduction Assessment

Yes at source level: current main publicly supports registering plugin gateway methods but provides no matching public plugin caller or discoverable request/response service seam. The closed implementation attempt further confirms the gap, although no live runtime reproduction is needed for this feature request.

## Solution Assessment

No, the unrestricted proposed `callGatewayMethod(method, params)` signature is not yet the best solution because it omits caller identity, method authorization, lifecycle availability, stable typing, and an explicit exposure contract. A narrower capability or typed service seam is safer and more maintainable.
