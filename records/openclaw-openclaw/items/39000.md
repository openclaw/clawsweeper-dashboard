## Summary

Keep open: current main and v2026.6.10 still do not expose a general plugin-to-plugin Gateway method caller. The closed raw-dispatch PR is superseded by this design item, and the newer host-RPC accessor issue is a narrower partial overlap rather than a replacement.

## What This Changes

The issue requests a public plugin API surface that lets one plugin call Gateway methods registered by another plugin.

## Best Possible Solution

Keep this as the canonical inter-plugin API design item; if maintainers approve it, add a narrow SDK contract with caller attribution, explicit method exposure or allowlisting, lifecycle semantics, docs, and contract tests while keeping raw Gateway dispatch internal.

## Reproduction Assessment

Yes. Source inspection on current main shows plugins can register Gateway methods, but ordinary plugin runtime/hooks/services have no matching general caller; the only SDK dispatcher is restricted to authenticated plugin HTTP routes with a manifest contract.

## Solution Assessment

No for the raw callGatewayMethod(method, params) shape as proposed. A safer solution needs caller identity, explicit method exposure or allowlisting, authorization rules, lifecycle handling, docs, and contract tests.
