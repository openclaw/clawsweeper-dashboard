## Summary

Keep open: current main and the latest release still do not expose a general plugin-to-plugin Gateway method caller. The closed raw-dispatch PR was not merged, the newer host-RPC accessor issue is only a partial overlap, and this remains a public plugin API plus security-boundary decision.

## What This Changes

The issue requests a plugin API such as `api.callGatewayMethod()` so one plugin can invoke Gateway methods registered by another plugin.

## Best Possible Solution

Keep this as the canonical inter-plugin API design item and, if maintainers approve the direction, add a narrow SDK contract with caller attribution, explicit exposure or allowlisting, lifecycle semantics, docs, and contract tests while keeping raw Gateway dispatch internal.

## Reproduction Assessment

Yes. Source inspection shows plugins can register Gateway methods, but ordinary plugin runtime/hooks/services have no matching general caller; the only SDK dispatcher is restricted to authenticated plugin HTTP routes with a manifest contract.

## Solution Assessment

No. A raw `callGatewayMethod(method, params)` shape is too broad without caller identity, explicit method exposure, authorization, lifecycle handling, and typed contract tests; a scoped capability or service/request seam is safer.
