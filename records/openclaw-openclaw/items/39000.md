## Summary

Keep open: current main still lacks a plugin-side caller for Gateway methods, and the existing bridge is intentionally limited to entitled plugin HTTP routes. The related raw-dispatch PR was closed unmerged, while the newer host-read RPC issue overlaps but does not replace this inter-plugin registered-method request.

## What This Changes

Request a plugin SDK/runtime API that lets one plugin call Gateway methods registered by another plugin.

## Best Possible Solution

Design a narrow documented plugin interop or Gateway-call seam with caller identity, method allowlisting/scope policy, plugin-handler routing, and focused SDK regression coverage.

## Reproduction Assessment

Yes for the capability gap: source inspection on current main shows plugin code can register Gateway methods but has no plugin-side caller, and the only dispatch helper throws outside an entitled HTTP route.

## Solution Assessment

Unclear for the exact `api.callGatewayMethod` shape. A caller may be useful, but the safer solution needs maintainer-approved SDK shape, authorization policy, caller attribution, and allowlisting before code work.
