## Summary

Keep open: current main and the latest release only provide a restricted Gateway dispatch helper for authenticated plugin HTTP routes, while the reported event/service-time inter-plugin communication use case still lacks a maintainer-approved API and security model.

## What This Changes

The issue requests a plugin-facing API for one plugin to call another plugin's registered Gateway method instead of relying on scoped API object property sharing or parsing message text.

## Best Possible Solution

Design a narrow, documented inter-plugin request/response contract that carries caller identity, authorization policy, handler routing, and tests, or explicitly document that this use case must stay outside core for now.

## Reproduction Assessment

Yes, source-level: current main exposes registerGatewayMethod but no general callGatewayMethod or runtime.gateway API, and the only dispatch helper is blocked outside authenticated plugin HTTP route scopes.

## Solution Assessment

No, not as proposed without more design: a flat api.callGatewayMethod(method, params) is plausible but too broad unless it carries caller identity, authorization policy, method routing, and tests; the current HTTP-route-only helper is safer but does not solve the event/service inter-plugin use case.
