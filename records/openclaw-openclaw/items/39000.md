## Summary

Keep open: current main still has plugin-side Gateway method registration but no plugin-side caller, and the existing implementation candidate needs maintainer SDK and authorization design before this becomes a public runtime surface.

## What This Changes

Request a plugin-side API that lets one plugin invoke Gateway RPC methods registered by another plugin.

## Best Possible Solution

Define a narrow documented plugin Gateway-call seam with explicit method policy, caller attribution, SDK type/baseline updates, docs, and regression tests before landing implementation.

## Reproduction Assessment

Yes for the API gap: source inspection on current main shows registerGatewayMethod exists, callGatewayMethod has no matches, and PluginRuntime has no gateway namespace. This is a feature request rather than a broken existing contract.

## Solution Assessment

Unclear for the exact requested shape. A maintained solution should be narrower than exposing the raw Gateway bus unless it includes explicit authorization, allowlisting, and plugin identity propagation.
