## Summary

Keep open: current main and v2026.6.6 still do not expose a general plugin-to-plugin Gateway method caller. The only current SDK dispatcher is deliberately limited to authenticated plugin HTTP routes, the raw-dispatch PR closed unmerged, and this remains a security-sensitive plugin API design item.

## What This Changes

The issue requests a public plugin API for one plugin to call Gateway methods registered by another plugin.

## Best Possible Solution

Keep this as the canonical inter-plugin API design item and, if maintainers approve the direction, add a narrow SDK contract with caller attribution, explicit exposure or allowlisting, lifecycle semantics, docs, and contract tests while keeping raw Gateway dispatch internal.

## Reproduction Assessment

Yes. Source inspection shows plugins can register Gateway methods, but ordinary plugin runtime/hooks/services have no matching general caller; the only dispatcher is restricted to authenticated plugin HTTP routes with a manifest contract.

## Solution Assessment

No. The proposed raw `callGatewayMethod(method, params)` shape is too broad without caller identity, explicit method exposure, authorization, lifecycle handling, and typed contract tests; a scoped capability or service/request seam is safer.
