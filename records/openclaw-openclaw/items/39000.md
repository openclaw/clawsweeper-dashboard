## Summary

Keep open: current main still lacks a general plugin-side Gateway caller, and the existing helper is intentionally limited to authenticated plugin HTTP routes. The related raw-dispatcher PR was closed in favor of this issue remaining the canonical SDK/security design discussion.

## What This Changes

Request to add a plugin-side `api.callGatewayMethod()` API so one plugin can invoke Gateway methods registered by another plugin.

## Best Possible Solution

Define a narrow documented plugin interop/Gateway-call seam with explicit entitlement, caller attribution, method-scope policy, plugin-handler routing, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `registerGatewayMethod` and a reserved authenticated-route dispatch helper, but no general `api.callGatewayMethod()` or plugin runtime caller for hooks/services like the reported use case.

## Solution Assessment

Unclear for the exact requested API. The maintainable solution should be narrower than arbitrary Gateway RPC access and carry plugin identity, caller scope, and an explicit entitlement policy.
