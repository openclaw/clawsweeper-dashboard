## Summary

Keep open: current main has a restricted Gateway dispatch helper for authenticated plugin HTTP routes, but it still does not provide the requested general plugin-side caller for hooks, services, or cross-plugin use. The remaining work is an SDK/security product decision, not conservative cleanup.

## What This Changes

Request to add a plugin-side API that lets one plugin call Gateway methods registered by another plugin.

## Best Possible Solution

Define a narrow, documented plugin interop/Gateway-call seam with explicit entitlement, caller attribution, handler routing, docs, and regression coverage while keeping the current HTTP-route bridge limited.

## Reproduction Assessment

Yes. Source inspection on current main shows `registerGatewayMethod` and a reserved HTTP-route dispatch helper, but no general `api.callGatewayMethod()` or plugin runtime caller for hooks/services like the reported use case.

## Solution Assessment

Unclear for the exact requested API. A maintainable solution should be narrower than arbitrary Gateway RPC access and should carry plugin identity, caller scopes, and an explicit entitlement policy.
