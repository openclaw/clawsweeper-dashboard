## Summary

Keep this issue open. Current `main` still exposes plugin gateway-method registration but no plugin-side `api.callGatewayMethod()` or `api.runtime.gateway.dispatchMethod()` caller, and the related implementation candidate, #46485, is still open rather than merged.

## What This Changes

The issue requests a plugin-side API for invoking registered gateway methods so plugins can communicate through a stable request/response seam.

## Best Possible Solution

Resolve this through Plugin SDK/runtime design review, likely by reviewing, revising, or replacing #46485 with a documented and tested request/response seam that preserves plugin identity and authorization semantics instead of exposing the whole Gateway bus casually.

## Reproduction Assessment

Yes. For the reported API gap, a source/docs inspection against current `main` is a high-confidence reproduction path: the registration API exists, but no plugin-side call API or runtime gateway namespace is present.

## Solution Assessment

Unclear for the exact requested `api.callGatewayMethod()` shape. The maintainable solution should be an explicit, documented runtime seam, or a narrower typed service/capability contract, with tests for caller identity, scope handling, unavailable-runtime behavior, and external plugin compatibility.
