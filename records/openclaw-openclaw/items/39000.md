## Summary

Keep open: current main still lacks a general plugin-side caller for registered Gateway methods, and the existing dispatch helper is deliberately limited to entitled authenticated plugin HTTP routes. The request is a real SDK capability gap, but it needs maintainer API and security design before implementation.

## What This Changes

Feature request to add a plugin API/runtime method for one plugin to call another plugin’s registered Gateway method.

## Best Possible Solution

Define a narrow documented plugin Gateway-call or interop seam with explicit entitlement, caller attribution, plugin-handler routing, method-scope policy, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows registerGatewayMethod and a reserved authenticated-route dispatch helper, but no general api.callGatewayMethod(), runtime.gateway.dispatchMethod, or equivalent plugin caller for hooks/services like the reported use case.

## Solution Assessment

Unclear for the exact requested API. The maintainable solution should be narrower than arbitrary Gateway RPC access and must carry plugin identity, caller scope, plugin-handler routing, and explicit entitlement policy.
