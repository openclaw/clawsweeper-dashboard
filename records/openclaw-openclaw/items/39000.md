## Summary

Keep open: current main still has only the registration side of plugin Gateway RPCs, and the related raw-dispatch PR was closed unmerged with this issue left as the canonical API-design discussion. The requested capability is plausible, but it is a new public plugin/runtime feature with authorization, caller attribution, and plugin-handler routing choices that need maintainer design.

## What This Changes

Request a plugin-facing `api.callGatewayMethod()` or equivalent runtime API so one plugin can invoke Gateway methods registered by another plugin.

## Best Possible Solution

Design a narrow plugin runtime Gateway-call contract with explicit plugin handler routing, caller attribution, scope or allowlist policy, docs, and regression coverage.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main supports registering plugin Gateway methods but does not expose a corresponding plugin caller API.

## Solution Assessment

No, not as a raw arbitrary dispatcher. The best path is a maintainer-approved SDK/runtime seam that preserves plugin method lookup and makes authorization and caller identity explicit.
