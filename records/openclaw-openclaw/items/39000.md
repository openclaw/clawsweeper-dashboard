## Summary

Keep open: current main still has no general plugin-side caller for registered Gateway methods, and the existing exported dispatch helper is deliberately reserved for authenticated plugin HTTP routes. The related raw-dispatch PR was closed unmerged in favor of this issue as the canonical API/security design discussion, while https://github.com/openclaw/openclaw/issues/76756 is related but narrower host-read RPC work rather than a full inter-plugin communication replacement.

## What This Changes

The issue requests a plugin-side method such as `api.callGatewayMethod()` so one plugin can invoke Gateway methods registered by another plugin.

## Best Possible Solution

Design a narrow documented plugin interop or Gateway-call seam with caller attribution, entitlement checks, method-scope policy, plugin-handler routing, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows plugins can register Gateway methods and import a request-scoped bridge, but neither `OpenClawPluginApi` nor `PluginRuntime` exposes `callGatewayMethod`, `runtime.gateway.dispatchMethod`, or an equivalent general plugin caller.

## Solution Assessment

Unclear for the exact requested API. The real gap should be solved through a maintainer-approved, security-reviewed SDK seam rather than exposing arbitrary Gateway RPC strings without caller policy.
