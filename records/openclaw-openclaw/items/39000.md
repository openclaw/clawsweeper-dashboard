## Summary

Keep open: current main still lacks a general plugin-side caller for registered Gateway methods, and the existing dispatch bridge is intentionally limited to entitled authenticated plugin HTTP routes. The closed raw-dispatch PR was superseded by this issue as the broader SDK/security design discussion, while https://github.com/openclaw/openclaw/issues/76756 is related but narrower host-read RPC work.

## What This Changes

Add a public plugin SDK/runtime API for one plugin to call Gateway methods registered by another plugin.

## Best Possible Solution

Design a narrow documented plugin interop or Gateway-call seam with caller attribution, entitlement checks, method-scope policy, plugin-handler routing, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows plugins can register Gateway methods and use an authenticated-route-only dispatch bridge, but neither OpenClawPluginApi nor PluginRuntime exposes a general callGatewayMethod, runtime.gateway.dispatchMethod, or equivalent caller.

## Solution Assessment

Unclear for the exact requested API. The real gap should be solved through a maintainer-approved, security-reviewed SDK seam rather than exposing arbitrary Gateway RPC strings without caller policy.

## Security Review

Status: needs_attention

Summary: Plugin-side Gateway dispatch is security-sensitive because it would expose control-plane method access across plugin trust boundaries.

  - confidence: 0.9
