## Summary

Keep open: current main and v2026.5.22 still lack a general plugin-side Gateway caller, while the only current dispatch helper is intentionally reserved for entitled authenticated plugin HTTP routes. The closed raw-dispatch pull request is not merged and the overlapping host-read RPC issue does not replace this plugin-owned registered-method request.

## What This Changes

The issue requests a public plugin API for one plugin to invoke Gateway methods registered by another plugin.

## Best Possible Solution

Design a narrow documented plugin interop or Gateway-call seam with caller identity, method allowlisting or scope policy, plugin-handler routing, SDK docs, and regression coverage, coordinated with the host-read RPC design.

## Reproduction Assessment

Yes for the capability gap: source inspection on current main shows plugin code can register Gateway methods but has no public plugin-side caller. The only current dispatch helper rejects callers outside an entitled authenticated plugin HTTP route.

## Solution Assessment

Unclear for the exact api.callGatewayMethod shape. A caller may be useful, but the maintainable solution needs maintainer-approved SDK shape, authorization policy, caller attribution, and allowlisting first.

## Security Review

Status: needs_attention

Summary: Plugin-side Gateway dispatch is security-sensitive because it would expose control-plane or plugin-owned method access across plugin trust boundaries.

Concerns:

- **[medium] Avoid arbitrary Gateway RPC access from plugins:** `src/gateway/method-scopes.ts:175`
  - body: Gateway authorization defaults unresolved non-dynamic methods to admin scope, so a public plugin caller needs an explicit allowlist, method-scope mapping, or entitlement policy before implementation.
  - confidence: 0.9

  - body: The current bridge only runs inside an authenticated plugin request scope that opted into Gateway method dispatch; a broader caller must define how plugin identity and scopes are carried outside that scope.
