## Summary

Keep open: current main still lets plugins register Gateway methods but does not expose a general plugin-side caller, and the only dispatch helper is reserved for authenticated plugin HTTP routes. The request remains valid, but it needs maintainer SDK and security design before implementation.

## What This Changes

Request adding a plugin-side API such as `api.callGatewayMethod()` so one plugin can invoke Gateway methods registered by another plugin.

## Best Possible Solution

Design a narrow documented plugin interop or Gateway-call runtime seam with caller attribution, allowlisted method-scope policy, plugin-handler routing, and focused regression coverage.

## Reproduction Assessment

Yes for the capability gap. Source inspection on current main shows `OpenClawPluginApi` has `registerGatewayMethod` but no caller, `PluginRuntime` has no Gateway namespace, and the only dispatch helper throws outside an entitled plugin HTTP route.

## Solution Assessment

Unclear for the exact requested API. A generic `api.callGatewayMethod` is plausible, but the safer path needs maintainer-approved SDK shape, authorization and caller policy, and method allowlisting.
