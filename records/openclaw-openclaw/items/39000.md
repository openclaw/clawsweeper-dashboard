## Summary

Keep open: current main added a deliberately scoped Gateway dispatcher for authenticated plugin HTTP routes, but background plugin-to-plugin request/response remains unsupported; the broader API needs an explicit product and security design rather than ambient access to all Gateway RPCs.

## What This Changes

Requests a supported plugin API for invoking another plugin’s registered Gateway method from plugin services, hooks, events, or timers.

## Best Possible Solution

Define an explicit inter-plugin request/response contract—preferably typed services rather than general Gateway RPCs—with manifest-declared providers and consumers, authorization and lifecycle rules, and focused tests; retain Gateway dispatch only for authenticated HTTP routes.

## Reproduction Assessment

Yes, at source level. Current main explicitly rejects Gateway dispatch outside an entitled authenticated plugin HTTP request, so plugin services, hooks, events, and timers cannot use it for the reported interaction.

## Solution Assessment

No, the proposed unrestricted `api.callGatewayMethod()` is not the safest maintainable solution because it would bypass the scoped trust model. A declared inter-plugin service contract would solve the request/response use case without making every Gateway RPC ambiently callable.
