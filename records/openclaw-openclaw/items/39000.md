## Summary

Keep open: current main still lacks a general plugin-side caller for registered or host Gateway methods, and the existing dispatch bridge is deliberately limited to authenticated plugin HTTP routes. The linked closed PR was superseded by this issue, while the newer host-RPC issue is related but narrower, so neither is a safe close target.

## What This Changes

The issue requests a plugin-side API for invoking registered or host Gateway methods from another plugin.

## Best Possible Solution

Design a narrow documented plugin interop or Gateway-call seam with caller attribution, entitlement checks, method-scope policy, plugin-handler routing, redaction rules, and focused regression coverage.

## Reproduction Assessment

Yes for the capability gap. Source inspection on current main shows plugins can register Gateway methods and use a request-scope-only dispatch bridge, but neither OpenClawPluginApi nor PluginRuntime exposes a general plugin-side caller.

## Solution Assessment

Unclear for the exact requested API. The real gap should be solved through a maintainer-approved, security-reviewed SDK seam rather than exposing arbitrary Gateway method strings without caller policy.

## Security Review

Status: needs_attention

Summary: Plugin-side Gateway dispatch is security-sensitive because it would expose control-plane method access across plugin trust boundaries.

Concerns:

- **[medium] Avoid arbitrary Gateway RPC access from plugins:** `src/gateway/method-scopes.ts:175`
  - body: Gateway authorization lets admin scope call any method and treats unclassified methods as admin-class, so a public plugin caller needs an explicit allowlist, method-scope mapping, or entitlement policy before implementation.
  - confidence: 0.9
- **[medium] Preserve plugin caller identity outside HTTP routes:** `src/plugin-sdk/gateway-method-runtime.ts:32`
  - body: The current bridge only runs inside an authenticated plugin request scope that opted into Gateway method dispatch; a broader caller must define how plugin identity and scopes are carried outside that scope.
  - confidence: 0.88
- **[medium] Protect sensitive status data:** `src/gateway/server-methods/health.ts:145`
  - body: The status handler includes sensitive fields only for operator.admin callers, so any plugin runtime accessor must preserve or replace that redaction boundary intentionally.
  - confidence: 0.86
