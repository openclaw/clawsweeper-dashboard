## Summary

Keep open: current main and v2026.6.10 still do not expose a general plugin-to-plugin Gateway method caller. The existing dispatch helper is deliberately restricted to authenticated plugin HTTP routes, the raw-dispatch PR was closed unmerged, and the newer host-RPC accessor issue is only a narrower partial overlap.

## What This Changes

The issue requests an SDK call side, `api.callGatewayMethod(...)`, so plugins can invoke Gateway methods registered by other plugins instead of parsing message context.

## Best Possible Solution

Keep this as the canonical inter-plugin API design item; if maintainers approve the direction, add a narrow SDK contract with caller attribution, explicit method exposure or allowlisting, lifecycle semantics, docs, and contract tests while keeping raw Gateway dispatch internal.

## Reproduction Assessment

Yes. Source inspection on current main shows plugins can register Gateway methods, but ordinary plugin runtime/hooks/services have no matching general caller; the only SDK dispatcher is restricted to authenticated plugin HTTP routes with a manifest contract.

## Solution Assessment

No for the raw `callGatewayMethod(method, params)` shape as proposed. A safer solution needs caller identity, explicit method exposure or allowlisting, authorization rules, lifecycle handling, docs, and contract tests.
