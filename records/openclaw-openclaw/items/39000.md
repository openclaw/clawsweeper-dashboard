## Summary

Keep open: current main still has plugin-side Gateway method registration but no plugin-side caller, and the linked implementation PR is still open pending SDK and authorization design.

## What This Changes

Add a plugin API/runtime method that lets one plugin invoke registered Gateway methods from another plugin instead of relying on text parsing or disk coupling.

## Best Possible Solution

Define a narrow documented plugin Gateway-call seam with explicit method policy, caller attribution, SDK type and baseline updates, docs, and regression tests, then resolve this issue by landing or superseding the linked implementation PR.

## Reproduction Assessment

Yes for the API gap: source inspection on current main shows registerGatewayMethod exists, while callGatewayMethod and runtime.gateway dispatch are absent. This is a feature request rather than a broken existing contract.

## Solution Assessment

Unclear for the exact requested shape. A maintainable solution should be narrower than exposing arbitrary Gateway RPC access, with explicit authorization, allowlisting, and plugin caller identity propagation.
