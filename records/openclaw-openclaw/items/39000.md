## Summary

Keep open: current main still lacks a plugin-side API for invoking Gateway methods, while the existing in-process dispatch helper is limited to authenticated plugin HTTP request scope. This remains a valid SDK capability gap that needs maintainer API and security design before implementation.

## What This Changes

The issue requests a public plugin API such as api.callGatewayMethod() so one plugin can invoke Gateway methods registered by another plugin.

## Best Possible Solution

Define a narrow documented plugin interop or Gateway-call seam with explicit caller attribution, entitlement checks, method-scope policy, plugin-handler routing, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows registerGatewayMethod and a reserved authenticated-route dispatch helper, but no general api.callGatewayMethod(), runtime.gateway.dispatchMethod, or equivalent plugin caller for the reported plugin-to-plugin use case.

## Solution Assessment

Unclear for the exact requested API. The maintainable path should be narrower than arbitrary Gateway RPC access and must carry plugin identity, caller scope, plugin-handler routing, and explicit entitlement policy.
