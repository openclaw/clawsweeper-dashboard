## Summary

Keep open: current main and v2026.6.10 still do not provide an ordinary plugin-to-plugin Gateway method caller; the existing helper is public but deliberately reserved for authenticated plugin HTTP routes, so this remains a plugin SDK product and security-boundary decision.

## What This Changes

The issue requests a plugin SDK/Gateway API that lets one plugin invoke Gateway methods registered by another plugin.

## Best Possible Solution

Keep this as the canonical inter-plugin API design item and, if maintainers approve it, add a narrow documented SDK contract with caller attribution, explicit exposure or allowlisting, lifecycle semantics, docs, and contract tests while keeping raw Gateway dispatch internal.

## Reproduction Assessment

Yes. Source inspection shows plugins can register Gateway methods, but ordinary plugin runtime/hooks/services have no matching general caller; the only SDK dispatcher is restricted to authenticated plugin HTTP routes with a manifest contract.

## Solution Assessment

No, not as a raw callGatewayMethod(method, params) surface. A safer solution needs explicit caller identity, method exposure or allowlisting, authorization, lifecycle behavior, and public contract tests.
