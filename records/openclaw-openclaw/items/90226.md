- **[P1] Do not flip the shipped thread reset default without an upgrade path:** `src/config/sessions/reset-policy.ts:56-59`
  - body: Current main and v2026.6.1 use the daily DEFAULT_RESET_MODE when no reset policy is configured. This branch changes only unconfigured thread sessions to idle with idleMinutes=0, so existing topic/thread conversations can keep the same session id and transcript state past the daily boundary without opt-in or migration; this needs an explicit maintainer-approved compatibility path before merge.
  - confidence: 0.92
- **[P1] Restore the generic run-context callback contract:** `src/plugins/host-hooks.ts:168`
  - body: PluginAgentEventSubscriptionRegistration is exported through the plugin SDK, and current main lets handlers call ctx.getRunContext<T>(...) for namespace-typed JSON. Narrowing this callback to PluginJsonValue | undefined can break external plugin typechecks on upgrade even though runtime behavior is unchanged; keep the generic signature or add an explicit compatibility/deprecation path.
  - confidence: 0.91
- **[P2] Keep the include-permission audit test platform-stable:** `src/security/audit-config-include-perms.test.ts:17`
  - body: Replacing the mocked inspectPathPermissions result with fs.chmodSync(0o644) stops this test from proving collectIncludeFilePermFindings passes env/platform/exec into the permission inspector, and it can depend on host POSIX chmod behavior instead of the Windows ACL path. Keep the permission facade mocked here or add separate platform-targeted coverage.
  - confidence: 0.86
- **[P2] Update the session lifecycle deep dive too:** `docs/concepts/session.md:86-90`
  - body: The PR updates the overview/config docs for the new thread default, but docs/reference/session-management-compaction.md still says daily reset is the default without the thread exception.
