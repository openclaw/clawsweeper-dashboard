## Summary

This should stay open as a narrow source-backed bug: current main still derives the URL fallback filename from the raw `URL.pathname`, so ordinary escaped spaces remain visible as `%20`. The related Feishu items are useful context, but they cover plugin-specific upload/display handling rather than this generic remote-media fallback.

## What This Changes

The issue requests decoding valid percent escapes in remote media URL basename fallback filenames when no response filename or caller hint is available.

## Best Possible Solution

Decode the URL-derived basename in `src/media/fetch.ts` with safe fallback behavior, while keeping header and caller-hint precedence unchanged and adding focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main passes `new URL(finalUrl).pathname` directly to the basename helper, and a Node probe shows that path remains `/files/My%20Report.pdf` with basename `My%20Report.pdf`. I did not run the OpenClaw Vitest lane in this read-only review.

## Solution Assessment

Yes, the maintainable fix is narrow: decode the URL fallback basename after extracting the basename, fall back to the raw basename on malformed escapes, and add tests for both `saveRemoteMedia` and `readRemoteMediaBuffer` fallback filenames.
