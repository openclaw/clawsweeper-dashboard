## Summary

This should stay open: current main and the latest release still derive the remote-media fallback filename from the raw `URL.pathname`, and there is an open linked fix PR that should be reviewed or landed before the issue closes.

## What This Changes

The issue asks for remote-media fallback filenames derived from URL path basenames to decode valid percent escapes before being exposed.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84052 if it safely decodes the URL-derived basename while preserving header and caller-hint precedence.

## Reproduction Assessment

Yes, at source level: current main passes `new URL(finalUrl).pathname` directly to the basename helper, and a Node probe shows that path remains `/files/My%20Report.pdf` with basename `My%20Report.pdf`. I did not run the OpenClaw Vitest lane in this read-only review.

## Solution Assessment

Yes. The maintainable fix is narrow: decode the URL fallback basename after extracting the basename, fall back to the raw basename on malformed escapes, and keep existing header and caller-hint precedence unchanged.
