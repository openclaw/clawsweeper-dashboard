## Summary

Keep open: current main still has the reported Telegram DM-topic session-key split, where outbound proactive sends mirror under a topic-only suffix while inbound replies use a chatId-plus-topic suffix.

## What This Changes

Requests that cron/message-tool Telegram DM-topic sends create or update the same routed session key that inbound replies compute, preserving context for cron-initiated conversations.

## Best Possible Solution

Align Telegram direct DM-topic outbound session-route computation with inbound routing by using a `chatId:threadId` session suffix while preserving the numeric Telegram thread id for delivery.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a proactive Telegram direct-topic send appends only the topic id, while an inbound reply in the same configured DM topic appends `chatId:topicId` on the same base key; I did not run live Telegram or cron in this read-only review.

## Solution Assessment

Yes. Option A, outbound session-key alignment, is the narrowest maintainable fix; a cron `sessionKey` override or generic aliasing system would be broader than needed for this concrete Telegram routing mismatch.
