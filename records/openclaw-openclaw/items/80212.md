## Summary

Keep this open: current main still computes different Telegram DM-topic session keys for proactive outbound sends and inbound replies, so the reported cron-initiated conversation split remains source-reproducible.

## What This Changes

The issue asks OpenClaw to let cron-initiated Telegram DM-topic messages create or use the same session key that inbound DM replies will route to.

## Best Possible Solution

Align Telegram direct-topic outbound session-route computation with inbound DM-topic routing so proactive sends mirror under the same `:thread:<chatId>:<topicId>` key that replies use, while keeping any broader cron session-key override out of scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: proactive outbound direct-topic sends append only the topic id, while inbound DM-topic replies append `chatId:topicId` on the same per-account direct base.

## Solution Assessment

Yes. Option A is the narrowest maintainable fix because it aligns an existing Telegram routing contract; cron session-key overrides or general aliasing would add broader product surface than this bug requires.
