## Summary

Keep open: current main still computes proactive Telegram DM-topic outbound mirror sessions with `:thread:<topicId>` while inbound DM-topic replies use `:thread:<chatId>:<topicId>`, so the reported cron-initiated conversation split remains source-proven.

## What This Changes

Allow cron/proactive Telegram DM-topic sends to create or mirror into the same session key that inbound replies use for that DM topic.

## Best Possible Solution

Align Telegram direct-topic outbound session route computation with inbound DM-topic routing by using `<chatId>:<topicId>` for the session suffix while preserving the numeric topic id for Telegram send parameters.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: proactive Telegram DM-topic sends append only the topic id, while inbound replies append `chatId:topicId` on the same per-account DM base. I did not run live Telegram or cron in this read-only review.

## Solution Assessment

Yes: outbound session-key alignment is the narrowest maintainable fix for the concrete mismatch. A cron session-key override or generic aliasing layer would be broader and should not be needed for this routing bug.
