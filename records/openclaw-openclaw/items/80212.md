## Summary

Keep open: current main still has a source-level mismatch between proactive Telegram DM-topic outbound mirror sessions and inbound reply sessions, so the reported cron-initiated conversation split remains valid.

## What This Changes

The issue requests that cron-initiated Telegram DM-topic sessions use a session key compatible with inbound DM reply routing so replies continue the same conversation.

## Best Possible Solution

Align Telegram direct DM-topic outbound session-route computation with inbound routing while keeping the numeric topic id for Telegram delivery and avoiding broader cron session-key overrides or generic session aliasing.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a proactive Telegram DM-topic send appends only the topic id, while an inbound reply appends `chatId:topicId` on the same base key; I did not run a live Telegram cron repro in this read-only review.

## Solution Assessment

Yes. Option A, outbound session key alignment, is the narrowest maintainable fix; a cron `sessionKey` override or generic session-key alias system would be broader than needed for this Telegram routing mismatch.

## Repair Work Prompt

Fix the current-main Telegram DM-topic session-key mismatch reported in https://github.com/openclaw/openclaw/issues/80212. A proactive cron/message-tool send to a Telegram DM topic currently mirrors into `...:thread:<topicId>`, while inbound replies in the same DM topic route to `...:thread:<chatId>:<topicId>` when DM-topic sessions are enabled. Implement the narrow Telegram outbound route alignment in the active `extensions/telegram/src` plugin path so direct DM-topic outbound routes use the inbound-compatible session suffix while preserving the numeric topic id used for actual Telegram delivery. Do not add a new cron `sessionKey` override, generic aliasing system, broad session migration, or cross-channel behavior change unless the focused fix proves impossible. Add or update regression coverage for explicit proactive DM-topic targets with isolated `dmScope` and for existing inbound/current-session behavior. Include a `CHANGELOG.md` fix entry because this is user-visible.
