## Summary

This should stay open: current main still computes different Telegram DM-topic session keys for proactive outbound sends and inbound replies, so the cron-initiated continuity break remains source-reproducible.

## What This Changes

Allow cron-initiated Telegram DM-topic conversations to use the same session key that inbound replies compute.

## Best Possible Solution

Align Telegram direct DM-topic outbound session-route computation with inbound routing, keeping the Telegram send thread id numeric and avoiding a new cron session-key override or aliasing system.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a proactive Telegram DM-topic send appends only the topic id, while the inbound reply appends `chatId:topicId` on the same direct base session key.

## Solution Assessment

Yes, the narrowest maintainable fix is Option A: align outbound Telegram DM-topic session keys with inbound keys. A generic cron `sessionKey` override or alias system would be broader than this concrete routing bug requires.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/80212. For Telegram direct DM topics with `dm.threadReplies: "inbound"`, per-DM `threadReplies`, `requireTopic`, or topic config, proactive outbound message-tool sends from cron must mirror transcript/session metadata under the same session key that an inbound reply in that DM topic computes. Preserve the numeric Telegram `threadId` used for `message_thread_id`; only the session-key suffix should become chat-scoped, e.g. `agent:finance:telegram:finance:direct:104506878:thread:104506878:174872` for target `104506878:topic:174872` under `dmScope: "per-account-channel-peer"`. Do not add a generic cron `sessionKey` override, new config knob, or aliasing system. Preserve group-topic behavior and the existing shared `dmScope: "main"` safeguards. Add focused regression coverage and a changelog fix entry.
