## Summary

Keep open: current main still has the reported Telegram DM-topic session-key split. Outbound proactive sends mirror under a `:thread:<topicId>` suffix while inbound DM-topic replies use `:thread:<chatId>:<topicId>`, and I found no merged canonical fix for this specific mismatch.

## What This Changes

Request that cron-initiated Telegram DM-topic messages use or allow the same session key that inbound replies compute so replies continue the proactive cron conversation.

## Best Possible Solution

Align Telegram direct DM-topic outbound session-route computation with inbound routing, keep the numeric Telegram thread id for delivery, and add focused regression coverage instead of adding a generic cron session-key override or alias system.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: a proactive Telegram DM-topic send appends only the topic id, while the inbound reply path appends `chatId:topicId` to the same base key. I did not run live Telegram or cron in this read-only review.

## Solution Assessment

Yes, Option A is the narrowest maintainable direction: align outbound Telegram DM-topic session routing with inbound routing. A cron `sessionKey` override or broad aliasing feature would be larger than this concrete routing bug requires.
