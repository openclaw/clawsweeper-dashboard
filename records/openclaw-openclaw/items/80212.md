## Summary

This should stay open: current main still computes different Telegram DM-topic session keys for proactive outbound sends and inbound replies, so the cron-initiated continuity problem remains in both main and the latest release.

## What This Changes

The issue requests cron-initiated Telegram DM-topic messages to use the same session key that later inbound replies from that DM topic use.

## Best Possible Solution

Align Telegram outbound DM-topic mirror session routing with the inbound DM-topic session-key calculation, while preserving group/forum topic behavior and default flat DM sessions.

## Reproduction Assessment

Yes, at source level: current main computes outbound and inbound Telegram DM-topic session suffixes differently for the same chat/topic inputs. I did not run a live Telegram cron repro in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to make outbound Telegram DM-topic session routing use the same scoped thread id that inbound routing already uses, rather than adding a cron-only override or alias system.
