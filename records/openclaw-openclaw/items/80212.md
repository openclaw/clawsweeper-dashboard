## Summary

Keep this open: current main still has a source-level Telegram DM-topic session-key mismatch between proactive outbound sends and inbound replies, and the latest release appears to carry the same mismatch.

## What This Changes

Request to make cron-initiated Telegram DM-topic messages use the same routing session key as inbound replies so proactive conversations keep context.

## Best Possible Solution

Align Telegram direct DM-topic outbound session-route computation with inbound routing, keeping the send thread id numeric while mirroring session metadata under the inbound-compatible key.

## Reproduction Assessment

Yes, at source level: a proactive Telegram DM-topic send appends only the topic id, while an inbound reply appends chatId:topicId on the same per-account direct base. I did not run live Telegram or cron in this read-only review.

## Solution Assessment

Yes: Option A, outbound session-key alignment, is the narrow maintainable fix. A cron session-key override or generic aliasing system would be broader than needed for this concrete Telegram routing mismatch.

## Repair Work Prompt

Fix Telegram DM-topic outbound session-key alignment for proactive `message` tool or cron sends. On current main, `extensions/telegram/src/channel.ts` passes numeric `resolvedThreadId` directly to `buildThreadAwareOutboundSessionRoute`, producing `:thread:<topicId>`, while inbound DM-topic handling in `extensions/telegram/src/bot-message-context.ts` uses `:thread:<chatId>:<topicId>`. Update the Telegram outbound route so direct DM-topic session keys match inbound keys for `dmScope` values such as `per-account-channel-peer`, but preserve the numeric `route.threadId`/Telegram `message_thread_id` used for actual sends. Avoid broad cron session-key overrides, generic aliasing, or changing group-topic routing. Add regression coverage for a direct Telegram target with account id and `session.dmScope: "per-account-channel-peer"`, plus any focused recovery case needed to avoid double-prefixing a recovered `chatId:topicId` thread suffix. Note any one-time legacy session-key transition in the changelog if the behavior is user-visible.
