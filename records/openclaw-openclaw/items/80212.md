## Summary

Keep open: current main still has a high-confidence source-reproducible Telegram DM-topic session-key mismatch between proactive outbound delivery and inbound reply routing, and no merged/current canonical item owns the remaining fix.

## What This Changes

The issue requests session continuity for cron-initiated Telegram DM-topic conversations by aligning proactive outbound session keys with later inbound reply routing.

## Best Possible Solution

Align Telegram outbound direct-topic session-key construction with inbound routing by using the composite `<chatId>:<topicId>` session suffix while keeping the transport thread id numeric and leaving group topics unchanged.

## Reproduction Assessment

Yes, from source inspection. Current main's outbound Telegram direct-topic path builds a bare topic-id session suffix while inbound DM-topic routing builds a composite chatId:topicId suffix, and existing tests encode both shapes.

## Solution Assessment

Yes for Option A. The narrow maintainable fix is to normalize outbound Telegram DM-topic session keys to the existing inbound format, not to add a cron session-key override or aliasing feature first.

Current main lets outbound direct Telegram topic sends persist a session suffix like `thread:<topicId>` while inbound DM-topic replies route to `thread:<chatId>:<topicId>`, especially visible with `session.dmScope: "per-account-channel-peer"`. Implement the narrow Option A fix: adjust the active plugin path in `extensions/telegram/src/channel.ts` so direct-topic outbound session keys use the same composite `<chatId>:<topicId>` suffix as inbound routing while preserving the numeric Telegram transport `threadId` and leaving group topic keys unchanged. Do not add a new cron `sessionKey` override, aliasing system, or config option. Add regression coverage for direct Telegram topic targets under default/main and `per-account-channel-peer` or named-account DM scope, plus message-tool/outbound session persistence if needed. Keep related ACP-bound DM-topic behavior and unbound DM-topic isolation intact. Changelog is expected because this is user-visible session continuity behavior.
