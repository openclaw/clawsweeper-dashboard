## Summary

Keep open: current main still has the reported Telegram DM-topic session-key split. A proactive outbound send mirrors into a `:thread:<topicId>` session, while an inbound reply in the same DM topic routes to `:thread:<chatId>:<topicId>`, so cron-started conversations can lose context.

## What This Changes

Allow cron-initiated Telegram DM-topic messages to seed the same session key that inbound replies use, so proactive conversations keep context.

## Best Possible Solution

Fix Telegram plugin outbound direct-topic session routing to use the inbound-compatible `<chatId>:<topicId>` session suffix for DM topics while preserving the numeric Telegram `threadId` used for sending.

## Reproduction Assessment

Yes, at source level. A proactive Telegram DM-topic send appends only the topic id, while an inbound reply appends `chatId:topicId` on the same base key; I did not run a live Telegram cron repro in this read-only review.

## Solution Assessment

Yes: outbound/inbound key alignment is the narrowest maintainable fix. A cron session-key override or generic aliasing system would be broader than needed for this concrete Telegram routing mismatch.
