## Label Justifications

- `impact:message-loss`: The affected `sessions_send` announce flow can duplicate or misroute cross-session messages.

## Summary

This issue should stay open: current `main` still passes the configured A2A ping-pong count into `sessions_send` announce flow, and the two linked closing PRs are still open candidate fixes rather than merged provenance.

## What This Changes

The issue reports that fire-and-forget `sessions_send` from an isolated cron session can inject the target agent's reply context back into the cron requester, causing unsolicited follow-up sends.

## Best Possible Solution

Land the narrowest linked fix that suppresses requester ping-pong only for isolated cron `sessions_send` while preserving target announce delivery and normal requester behavior.

## Reproduction Assessment

Yes. Source inspection shows current `main` can still step the requester session in the A2A ping-pong loop when `maxPingPongTurns` is nonzero; I did not run a live cron/channel reproduction in this read-only review.

## Solution Assessment

Yes. The best fix direction is a narrow isolated-cron requester gate before A2A ping-pong, not a timeout-based or whole-A2A skip, because target announce delivery and normal requester behavior must remain intact.
