## Summary

Keep this issue open: current main and the latest release still allow timeout-zero cross-session `sessions_send` to start the A2A announce flow with the configured ping-pong turn count, and the two linked fixing PRs are still open rather than merged.

## What This Changes

The issue reports that fire-and-forget `sessions_send` from an isolated cron run can inject the target agent's reply context back into the cron requester and trigger repeated follow-up messages.

## Best Possible Solution

Land one canonical fix that detects canonical isolated cron requesters, captures a target baseline for fire-and-forget cross-session cron sends, sets ping-pong turns to zero only for those requesters, preserves target announce delivery, and covers normal requester behavior.

## Reproduction Assessment

Yes, at source level: current main can enter the A2A requester ping-pong loop for a timeout-zero cross-session send with an isolated cron requester, and the issue includes a real gateway trace. I did not run a fresh live cron reproduction in this read-only pass.

## Solution Assessment

No merged solution exists yet. The best path is to review and land one canonical PR that suppresses only isolated-cron requester ping-pong while preserving announce delivery and baseline-protecting target replies.
