## Label Justifications

- `P1`: A real isolated-cron agent workflow can generate unsolicited cross-session messages and a feedback loop affecting user-visible agent behavior.
- `impact:message-loss`: The issue is about `sessions_send`/announce delivery misrouting and duplicating inter-session messages into the wrong requester path.
- `impact:session-state`: The target agent's reply context can be injected into an isolated cron session, causing session state drift and follow-up turns from the wrong context.

## Summary

Keep open: current main still has the source-level path that can step an isolated cron requester during A2A announce follow-up, and the linked closing PR is open rather than merged. The issue should remain the canonical tracker until https://github.com/openclaw/openclaw/pull/92283 or an equivalent fix lands.

## What This Changes

Report that fire-and-forget `sessions_send` from an isolated cron session can receive A2A reply turns from the target session, polluting the cron run and causing a feedback loop.

## Best Possible Solution

Land or reject https://github.com/openclaw/openclaw/pull/92283 after PR review; the durable fix should suppress requester ping-pong only for isolated cron requesters while preserving normal A2A fire-and-forget behavior.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main creates run-scoped isolated cron session keys, passes the default nonzero A2A turn count into timeout-zero sends, and the A2A loop first steps the requester session. I did not execute a live cron replay in this read-only review.

## Solution Assessment

Yes, the best direction is a narrow isolated-cron requester gate rather than a broad `timeoutSeconds === 0` behavior change. The open PR appears to target that shape, but the issue should stay open until the PR or equivalent fix merges.
