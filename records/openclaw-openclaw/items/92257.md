## Label Justifications

- `P1`: The report describes a real agent/session workflow producing repeated unintended inter-session sends and context injection in a scheduled cron path.
- `impact:session-state`: The bug is about target-agent reply context being injected into an isolated cron requester session, corrupting that run's state.
- `impact:message-loss`: The observed failure produced duplicate and misrouted `sessions_send` messages from the cron session to the main agent.

## Summary

Keep this issue open: current main still has the isolated-cron `sessions_send` A2A feedback path, and https://github.com/openclaw/openclaw/pull/92283 is an open closing PR for the same bug. Closing before that PR lands would drop the tracking issue for a source-reproducible session/message routing defect.

## What This Changes

The issue reports that fire-and-forget `sessions_send` from an isolated cron session can feed the target agent's reply context back into the cron requester and trigger repeated cross-session sends.

## Best Possible Solution

Review and either land or replace https://github.com/openclaw/openclaw/pull/92283 so isolated cron requesters cannot receive A2A ping-pong turns while normal requester behavior remains covered.

## Reproduction Assessment

Yes at source level, though I did not run a live cron/channel reproduction in this read-only review. Current main starts the A2A flow for fire-and-forget sends and the callee steps the requester first when ping-pong is enabled, matching the reporter's isolated-cron feedback-loop logs.

## Solution Assessment

Yes, the linked PR's direction is the narrowest visible fix path: gate the A2A ping-pong loop for isolated cron requesters rather than disabling normal fire-and-forget A2A behavior. The issue should stay open until that PR is reviewed and merged or superseded.
