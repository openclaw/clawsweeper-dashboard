## Label Justifications

## Summary

Keep this open: current main and the latest release still allow the A2A ping-pong path to step a different requester session, and the direct closing fix is still the open PR at https://github.com/openclaw/openclaw/pull/92283.

## What This Changes

Reports that fire-and-forget `sessions_send` from an isolated cron session can inject target-agent reply context back into the cron run and trigger repeated inter-session messages.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/92283 with an equivalent narrow fix that prevents isolated cron requesters from entering requester-side A2A ping-pong while preserving normal cross-session behavior.

## Reproduction Assessment

Yes at source level: current main passes the configured A2A ping-pong count into `runSessionsSendA2AFlow`, which steps the requester when it differs from the target. I did not run a live cron/Telegram reproduction in this read-only review.

## Solution Assessment

Yes, the linked PR's direction is the narrow maintainable fix: gate requester-side A2A ping-pong for isolated-cron requesters, not for all timeout-zero sends. This issue should remain open until that PR or an equivalent fix merges.
