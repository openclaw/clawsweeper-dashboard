## Label Justifications

- `P1`: The defect can create an active agent feedback loop with repeated unsolicited cross-session messages in a real cron workflow.
- `impact:message-loss`: The reported loop duplicates and misroutes inter-session messages beyond the single intended send.
- `impact:session-state`: Reply context from the target agent is injected into and changes the behavior of an isolated cron session.

## Summary

Current main still has the requester-side A2A ping-pong behavior described by the report, and the open, mergeable closing PR is the active repair path, so this issue remains necessary until that PR lands or is replaced.

## What This Changes

Prevent announce-mode sessions_send replies from being injected into isolated cron sessions and triggering feedback loops.

## Best Possible Solution

Land the linked narrow fix so isolated cron requesters are never stepped by the A2A reply loop, while target announce delivery and normal non-cron ping-pong remain unchanged.

## Reproduction Assessment

Yes at source level. Current main shows the A2A loop can step the requester, and the report provides concrete session keys, timestamps, and resulting messages; the linked PR also supplies before-and-after runtime output.

## Solution Assessment

Yes in direction. Suppressing requester stepping only for isolated cron runs is narrower than changing every `timeoutSeconds: 0` send, which would break the established non-cron ping-pong behavior.
