## Label Justifications

- `P1`: The reported release regression made a real WhatsApp channel unusable after upgrade by selecting the wrong runtime/auth path.
- `impact:message-loss`: The reported WhatsApp replies failed with error text instead of delivering normal agent responses.
- `impact:session-state`: The report centers on existing channel session/runtime binding changing across upgrade while the user config stayed unchanged.
- `impact:auth-provider`: The failure replaced intended Claude CLI OAuth routing with Anthropic API-key behavior.

## Summary

Close as implemented on current main: the legacy Claude CLI OAuth/runtime upgrade path now backfills model-scoped `claude-cli` runtime policy, and WhatsApp uses the shared inbound runner that consumes that policy, so the reported PI fallback, tool loss, and misleading Anthropic API-key/billing path are fixed on main. The fix is not in the latest tagged release `v2026.5.12`.

## What This Changes

The issue reports a v2026.5.12 WhatsApp regression where legacy Claude CLI runtime config fell through to PI, removing tools and surfacing Anthropic API billing/auth errors.

## Best Possible Solution

Keep the current model-scoped Claude CLI runtime backfill, ship it in the next release, and track any remaining stale session-pin cleanup under https://github.com/openclaw/openclaw/issues/83098.

## Reproduction Assessment

Yes for the release regression: the merged fix PR reproduced the pre-fix migration/resolver failure, and current tests cover the legacy Claude CLI OAuth config. I did not run live WhatsApp in this read-only review.

## Solution Assessment

Yes. Writing provider/model-scoped `agentRuntime: { id: "claude-cli" }` during Anthropic migration/defaulting preserves the documented runtime contract without reviving ignored whole-agent runtime pins.
