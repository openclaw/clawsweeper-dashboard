## Summary

Keep open. The prior ClawSweeper assessment still holds on current main: `/v1/models` is intentionally agent-first, `models.list` already has the provider-model catalog over Gateway RPC, and exposing that catalog over HTTP still needs a maintainer API, auth, and compatibility decision.

## What This Changes

The issue requests an HTTP surface that returns configured or reachable provider model IDs instead of only OpenClaw agent-target IDs from `/v1/models`.

## Best Possible Solution

Preserve the documented agent-first `/v1/models` contract and add an additive read-only HTTP provider-model discovery surface backed by `models.list`, unless maintainers intentionally revise the compatibility API.

## Reproduction Assessment

Yes, for the current behavior. Source inspection shows `/v1/models` maps `loadAgentModelIds()` into the response, and the e2e test asserts the returned ids are OpenClaw agent targets.

## Solution Assessment

Unclear. Replacing `/v1/models` is not clearly the best fix because docs and tests intentionally make it agent-first; an additive endpoint or explicit opt-in view is the safer maintainable direction after maintainer API review.
