## Summary

Keep open. Current main still implements and documents `/v1/models` as an OpenClaw agent-target list, while provider-model discovery exists through Gateway RPC `models.list`; exposing that catalog over HTTP remains a valid feature request that needs an API compatibility, auth, and response-shape decision.

## What This Changes

Request an HTTP API surface that exposes routable provider model IDs for remote clients instead of only agent targets from `/v1/models`.

## Best Possible Solution

Preserve the documented agent-first `/v1/models` contract and add an additive read-only HTTP provider-model discovery surface backed by `models.list`, unless maintainers intentionally revise the compatibility API.

## Reproduction Assessment

Yes for the current behavior. Source inspection on current main shows `/v1/models` returns `loadAgentModelIds()` results, and the e2e test asserts the returned IDs are OpenClaw agent targets.

## Solution Assessment

Unclear. Replacing `/v1/models` is not clearly the best fix because current docs and tests intentionally make it agent-first; an additive endpoint or explicit opt-in view is the safer maintainable direction after maintainer API review.
