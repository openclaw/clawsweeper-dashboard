## Summary

Keep open. Current main still intentionally exposes OpenClaw agent targets from the OpenAI-compatible HTTP `GET /v1/models` endpoint, while provider/model IDs are available only through the Gateway `models.list` RPC and UI-backed configured model view. The requested remote HTTP discovery surface for actual provider model IDs is still unresolved and should not be closed as implemented or duplicate.

## What This Changes

The issue asks for the Gateway OpenAI-compatible model discovery surface to return provider/model IDs, or to add a new endpoint for them, so third-party clients can list actual AI models without reading `openclaw.json`.

## Best Possible Solution

Add an additive read-only provider-model discovery surface, such as a new HTTP endpoint or explicit `view` parameter backed by the existing `models.list` configured/all catalog, while preserving `/v1/models` as the documented agent-target list and documenting auth, reachability, and compatibility semantics.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with OpenAI-compatible HTTP enabled, `GET /v1/models` reaches `handleOpenAiModelsHttpRequest()`, which returns `loadAgentModelIds()`, and the e2e test asserts all returned ids start with `openclaw`.

## Solution Assessment

Unclear. The user problem is valid, but replacing `/v1/models` is probably not the best solution because current main intentionally documents and tests it as agent-first; an additive endpoint or explicit opt-in view is the safer maintainable direction.
