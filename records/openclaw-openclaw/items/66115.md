## Summary

Keep open. Current main still intentionally exposes `/v1/models` as an agent-target compatibility list, while provider-model discovery exists only on the Gateway RPC `models.list` surface; adding that catalog to HTTP still needs an API-shape, compatibility, and auth decision.

## What This Changes

The issue asks for an HTTP discovery surface that returns backend provider model IDs available to the gateway instead of only OpenClaw agent targets.

## Best Possible Solution

Preserve the documented agent-first `/v1/models` contract and add an additive read-only HTTP provider-model discovery surface backed by `models.list` if maintainers approve the public API contract.

## Reproduction Assessment

Yes, for the current behavior. Source inspection shows `/v1/models` maps `loadAgentModelIds()` into the response, and the e2e test asserts the returned ids are OpenClaw agent targets.

## Solution Assessment

Unclear. Replacing `/v1/models` is not the safest path because docs and tests intentionally make it agent-first; an additive endpoint or explicit opt-in view would be more maintainable after maintainer API review.
