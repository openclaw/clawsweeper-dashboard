## Summary

Keep open. Current main still returns OpenClaw agent targets from the OpenAI-compatible HTTP `/v1/models` route, while configured/provider model catalog discovery is only exposed through the Gateway RPC `models.list` surface.

## What This Changes

Expose configured or reachable provider model IDs through a remote HTTP discovery surface instead of only agent-target IDs from `/v1/models`.

## Best Possible Solution

Preserve the documented agent-first `/v1/models` contract and add an additive read-only HTTP provider-model discovery surface backed by `models.list`, unless maintainers intentionally revise the compatibility API.

## Reproduction Assessment

Yes. The source-level path is high-confidence: `/v1/models` reaches `handleOpenAiModelsHttpRequest()`, which returns `loadAgentModelIds()`, and the e2e test asserts all returned IDs start with `openclaw`.

## Solution Assessment

Unclear. The user problem is valid, but replacing `/v1/models` is likely not the safest fix because current docs and tests intentionally make it agent-first; an additive endpoint or opt-in view is the safer direction.
