## Summary

Keep open. Current main and the latest release still intentionally expose `/v1/models` as an OpenClaw agent-target list; provider model discovery exists through `models.list` and optional admin HTTP RPC, but the requested standard HTTP discovery surface still needs an API, compatibility, and auth decision.

## What This Changes

Request that OpenClaw expose actual routable provider model IDs to third-party HTTP clients instead of only OpenClaw agent targets from `GET /v1/models`.

## Best Possible Solution

Preserve the documented agent-first `/v1/models` contract and add an additive read-only provider-model HTTP discovery surface backed by `models.list` after maintainer API and auth review.

## Reproduction Assessment

Yes for the current behavior, by source and test inspection: `/v1/models` returns `loadAgentModelIds()` and the e2e test asserts all returned ids start with `openclaw`. I did not run a live Gateway in this read-only review.

## Solution Assessment

Unclear. Replacing `/v1/models` is not the safest solution because docs and tests intentionally make it agent-first; an additive endpoint or explicit provider-catalog view is more maintainable after API/auth review.
