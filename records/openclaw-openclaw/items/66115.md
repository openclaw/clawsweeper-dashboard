## Summary

Keep open. Current main still exposes agent targets from the OpenAI-compatible HTTP `/v1/models` route, while provider/model discovery exists only on the Gateway RPC surface, so the requested remote HTTP model discovery remains unresolved.

## What This Changes

Expose actual provider/model IDs through a Gateway HTTP discovery surface so remote clients can populate model pickers without reading local config files.

## Best Possible Solution

Add an additive read-only HTTP discovery surface, or an explicit opt-in view, backed by `models.list` while preserving the documented agent-first `/v1/models` contract.

## Reproduction Assessment

Yes. The source-level path is high-confidence: `GET /v1/models` reaches `handleOpenAiModelsHttpRequest()`, which returns `loadAgentModelIds()`, and the e2e test asserts all ids start with `openclaw`.

## Solution Assessment

Unclear. The user problem is valid, but replacing `/v1/models` is likely not the safest fix because current docs and tests intentionally make it agent-first; an additive endpoint or opt-in view is the safer design.
