## Summary

Keep open. Current main still documents and implements `/v1/models` as an OpenClaw agent-target list, while provider-model discovery is available through Gateway RPC `models.list`; exposing that catalog over HTTP needs maintainer API compatibility, auth, and view-shape decisions.

## What This Changes

Request that the OpenAI-compatible `/v1/models` HTTP surface expose routable backend provider model IDs for third-party model pickers instead of only OpenClaw agent targets.

## Best Possible Solution

Preserve the documented agent-first `/v1/models` contract and add an additive read-only HTTP provider-model discovery surface backed by `models.list`, unless maintainers intentionally revise the compatibility API.

## Reproduction Assessment

Yes. Source inspection on current main shows `/v1/models` reaches `handleOpenAiModelsHttpRequest()`, which sends `loadAgentModelIds()` results; the e2e test asserts all returned IDs start with `openclaw`.

## Solution Assessment

Unclear. The user problem is valid, but replacing `/v1/models` is not clearly the best fix because docs and tests intentionally make it agent-first; an additive endpoint or opt-in view is safer.
