## Summary

Keep open. Current main still serves only OpenClaw agent targets from the OpenAI-compatible HTTP `/v1/models` route, while actual provider/model discovery is available through the Gateway RPC `models.list` surface, so the requested remote HTTP discovery path remains unresolved.

## What This Changes

The issue requests HTTP discovery of actual provider/model IDs through `/v1/models` or an additive endpoint, instead of only returning OpenClaw agent targets.

## Best Possible Solution

Add an additive read-only HTTP discovery surface, or an explicit opt-in model-catalog view, backed by `models.list` while preserving the documented agent-first `/v1/models` default.

## Reproduction Assessment

Yes. The source path is high-confidence: `GET /v1/models` reaches `handleOpenAiModelsHttpRequest()`, which returns `loadAgentModelIds()`, and the e2e test asserts the returned IDs start with `openclaw`.

## Solution Assessment

Unclear. The user problem is valid, but replacing `/v1/models` is probably not the safest fix because current docs and tests intentionally make it agent-first; an additive endpoint or opt-in view is the safer maintainable direction.
