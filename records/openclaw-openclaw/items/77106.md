## Summary

Close as implemented on current `main`: the upgrade/plugin repair path now detects missing install payloads, reinstalls configured downloadable plugins, and uses a managed npm root so installing one official plugin does not prune sibling plugins. The `plugins.allow` warning remains passive when no restrictive allowlist is configured, so it is not the routing blocker.

## What This Changes

The issue reports that upgrading from 2026.4.29 to 2026.5.2 leaves non-bundled plugins marked installed without files on disk and does not populate `plugins.allow`, breaking Discord routing and Brave web search.

## Best Possible Solution

Keep the current main repair path, ship it in the next patch release, and leave `plugins.allow` as an optional restrictive trust list rather than making it mandatory for open-allow configs.

## Reproduction Assessment

Yes, from source inspection rather than a live macOS upgrade: the reported state maps to install records whose package directories are absent plus configured Discord/Brave surfaces. Current main has explicit repair code and regression coverage for that state.

## Solution Assessment

Yes. Repairing the managed install ledger/payloads in update and doctor is the narrow maintainable fix, and it avoids Discord- or Brave-specific startup policy in core while preserving the existing `plugins.allow` trust semantics.
