## Summary

Keep open: current main partially covers standard `message_sending` on Telegram reply delivery, but it still lacks a typed `InputRichMessage`-level transformation seam for rich-message plugins, and the related generic hook-bypass work only partially overlaps this request.

## What This Changes

The issue asks for Telegram rich-message outbound delivery to expose a plugin interception point so markdown can be transformed into Telegram Rich Messages before send.

## Best Possible Solution

Define a narrow typed Telegram/channel outbound seam for rich-message shaping, coordinated with the generic hook-bypass work so standard hooks run exactly once and rich payload plugins do not reach into private internals.

## Reproduction Assessment

Yes, at source level for the missing rich-message object seam: current hook types cannot return `InputRichMessage`, and the direct rich send helper does not run hook dispatch. The broader claim that no rich Telegram path runs `message_sending` is not fully supported because `deliverReplies` already does.

## Solution Assessment

Unclear as filed. Reusing standard string hooks may help part of the report, but the best durable fix is likely a maintainer-approved typed SDK/channel seam rather than an ad hoc Telegram-only hook name.
