## Summary

Keep open: current main still has no plugin-facing way to transform Telegram rich-message payloads into `InputRichMessage`, and the overlapping generic `message_sending` bypass is only covered by open candidate PRs, not a merged fix.

## What This Changes

The issue asks for Telegram rich-message delivery to expose a hook or seam so plugins can convert outbound Markdown into Telegram `InputRichMessage` payloads before send.

## Best Possible Solution

Define a narrow typed Telegram or channel outbound rich-payload seam, coordinated with the generic exactly-once `message_sending` fix so rich-message plugins do not reach into private Telegram internals.

## Reproduction Assessment

Yes, at source level for the missing rich-payload seam: current hook result types cannot return `InputRichMessage`, and the rich send helper builds `rich_message` internally. I did not run the reporter's live Telegram plugin setup, and normal Telegram reply delivery already runs `message_sending` for part of the surface.

## Solution Assessment

Unclear as filed. Reusing existing string hooks may fix part of the bypass story, but the maintainable end state is a maintainer-approved typed SDK/channel seam rather than an ad hoc Telegram-only hook name.
