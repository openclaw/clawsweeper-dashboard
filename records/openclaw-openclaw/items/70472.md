## Summary

Keep open. Current main still has source evidence that overflow and compaction-failure recovery can call the reset path that rewrites the active session id, the checkpoint sentinel still only warns when the session store row is missing, and the linked closing PR #70479 remains open and unmerged.

## What This Changes

Request to prevent context-overflow or compaction recovery from silently rotating the active WebChat `agent:main:main` session mapping and orphaning prior transcripts.

## Best Possible Solution

Preserve the existing logical session mapping through compaction failure unless the user intentionally requests a reset, and make any unavoidable recovery rotation explicit and durable. The fix should also guard the related missing-transcript state so `sessions.json` never points an active session key at a nonexistent transcript.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make the embedded runner return a context-overflow error payload or throw a compaction-failure error while the active session store exists; current main then calls `resetSessionAfterCompactionFailure`, and the reset unit test asserts the active store row rotates to a generated session id.

## Solution Assessment

Yes for keeping this issue open. The narrow maintainable solution is to review and land #70479 or an equivalent patch that preserves the active session id; the current reset-and-replay behavior is only a partial mitigation because it still changes the active mapping.
