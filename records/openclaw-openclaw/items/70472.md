## Summary

Keep open. Current main still handles embedded context-overflow payloads and thrown compaction failures by calling the session-reset path, which writes a new session id into the active session store row. Because #70479 is an open, unmerged closing PR for this issue, the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that context overflow or compaction failure can orphan the active WebChat `agent:main:main` mapping and silently remap it to a new session id while the prior transcript remains on disk.

## Best Possible Solution

Preserve the existing logical session mapping when overflow or compaction recovery fails, and only rotate to a new transcript through an explicit, durable, auditable path. The fix should cover WebChat direct sessions and the related missing-transcript variant in #75151 if it shares the same reset-store ordering behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to drive the embedded runner through a context-overflow error payload or thrown compaction-failure error while `resetSessionAfterCompactionFailure` succeeds; current main then executes `resetReplyRunSession`, and the existing reset unit test asserts the active store row rotates to a generated session id.

## Solution Assessment

Yes for keeping this issue open. The current main reset path is not the best fix because it preserves only a replayed continuity slice while changing the active session id; the best path is to review and land #70479 or an equivalent narrow fix that preserves the active mapping unless rotation is explicit.
