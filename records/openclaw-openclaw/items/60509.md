## Summary

Keep open. Current main still does not expose the incoming prompt or user message on the pre-injection `agent:bootstrap` context; newer context-engine hooks can see `prompt`, and `contextInjection: "never"` can disable bootstrap injection, but that is a different extension point and does not let an `agent:bootstrap` hook query-condition `bootstrapFiles` before they become injected context.

## What This Changes

Request to add an optional incoming user message or query field to the `agent:bootstrap` hook context for query-conditioned bootstrap file compression.

## Best Possible Solution

Add a backward-compatible optional `userMessage` field to `AgentBootstrapHookContext`, thread it only from user-triggered run paths where the current prompt is already available, leave it undefined for raw/background runs, and document how it differs from context-engine assembly and `before_prompt_build`.

## Reproduction Assessment

Yes. The API gap is reproducible by source inspection on current main: the bootstrap context type, bootstrap resolver params, and runner call site contain no query field, while prompt-aware APIs run later or through a different context-engine surface.

## Solution Assessment

Yes. A narrow optional context-field extension is the most maintainable way to satisfy this specific hook use case; relying only on context engines would help broader prompt ownership but would not let existing bootstrap hooks mutate `bootstrapFiles` based on the current user turn.

## Repair Work Prompt

Implement issue https://github.com/openclaw/openclaw/issues/60509 as a focused API extension. Add an optional `userMessage?: string` field to `AgentBootstrapHookContext`, thread it through `applyBootstrapHookOverrides`, `resolveBootstrapFilesForRun`, and `resolveBootstrapContextForRun`, and pass it only from user-triggered CLI/embedded run paths where the current prompt is already available. Keep it undefined for raw model runs, heartbeats/background/generated runs, and paths without a user prompt. Do not move SMELT-style compression into core and do not replace the context-engine APIs. Add focused tests proving the field reaches `agent:bootstrap` before bootstrap files are converted to context files, and proving it is omitted where appropriate. Update `docs/automation/hooks.md` and any relevant system-prompt/context docs to explain the difference between pre-injection `agent:bootstrap`, context-engine `assemble(prompt)`, and `before_prompt_build`. Add a single-line `CHANGELOG.md` entry under the active version because this is a user-facing hook API change.
