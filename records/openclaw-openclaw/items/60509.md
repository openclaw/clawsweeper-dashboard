## Summary

Keep open. Current main and the latest release still do not expose the triggering user prompt on `agent:bootstrap`; the prompt-aware `before_prompt_build` and `ContextEngine.assemble()` paths exist, but they run after bootstrap-file selection or target a different mutable surface, so this remains a real hook API/product decision.

## What This Changes

Request to add an optional incoming user message or query field to the `agent:bootstrap` hook context so hooks can do query-conditioned bootstrap-file compression.

## Best Possible Solution

Add a backward-compatible optional current-prompt field only if maintainers confirm `agent:bootstrap` should support query-conditioned bootstrap-file mutation, then document its lifecycle and how it differs from prompt-build hooks and context engines.

## Reproduction Assessment

Yes, source-level. Current main exposes no current-prompt field on `AgentBootstrapHookContext` or bootstrap resolver parameters, while later prompt-aware surfaces receive `params.prompt`.

## Solution Assessment

Unclear until maintainers make the API decision. A narrow optional field is the clean shape if bootstrap-file mutation is desired, but `ContextEngine.assemble({ prompt })` already covers some retrieval-oriented compression and may be the preferred direction for non-bootstrap use cases.

## Risks / Open Questions

- Maintainers still need to choose the public field name and semantics: raw prompt versus enriched prompt, first turn versus continuation turns, and when the field is intentionally undefined.

- Some retrieval use cases are already served by `ContextEngine.assemble({ prompt })`; adding a parallel bootstrap-hook prompt field should avoid creating two competing query-compression APIs.
