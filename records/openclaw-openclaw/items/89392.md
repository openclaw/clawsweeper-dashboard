## Label Justifications

- `P1`: The report describes Gateway-wide timer, WebSocket, sidecar, and chat.history stalls lasting seconds to minutes during a real long-streaming session.

## Summary

Keep open: current main and v2026.6.8 include the relevant cooperative stream-yield and lightweight text-delta mitigations, but there is still no current-main or latest-release proof that a DeepSeek-size Gateway stream keeps timers, WebSocket RPCs, and chat history responsive.

## What This Changes

The issue reports Gateway event-loop starvation during very long streaming model responses, especially high-output DeepSeek/OpenAI-compatible streams.

## Best Possible Solution

Run a current-main or v2026.6.8 long-output DeepSeek or synthetic OpenAI-compatible Gateway scenario with concurrent timer, WebSocket, and chat.history probes; if it still starves, fix the proven provider/agent/Gateway drain rather than landing a broad worker-thread rewrite.

## Reproduction Assessment

No. The original diagnostics are credible, but no current-main or v2026.6.8 live/synthetic DeepSeek-size Gateway run proves the exact starvation still reproduces after the stream-yield and dense-delta fixes.

## Solution Assessment

Unclear. The shipped stream fixes are a plausible partial solution, but the best final fix needs current profiling across the provider, agent-loop, and Gateway broadcast path; the open worker-thread PR is not the best fix as reviewed.
