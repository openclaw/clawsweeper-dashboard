## Summary

Keep this open. Current main still strips successfully transcribed audio from inbound media notes and has no typed raw audio/video prompt payload or config surface; the related implementation PR is open, unmerged, and has review blockers around schema, session-key cleanup, prompt-media typing, and raw-media egress policy.

## What This Changes

The issue requests an option to forward raw inbound audio to multimodal models instead of, or in addition to, the current transcription-only path.

## Best Possible Solution

Keep this as the product tracker and resolve it through https://github.com/openclaw/openclaw/pull/48900 or a successor that adds typed audio/video prompt media, schema-visible config, capability-aware provider behavior, raw-media egress docs, limits, and regression coverage.

## Reproduction Assessment

Yes. Current-main source shows inbound audio is converted to `audio.transcription` text, the corresponding audio note is filtered after successful transcription, and the embedded runner prompt path only forwards text plus image blocks; I did not run a live Telegram or Signal probe.

## Solution Assessment

Yes for keeping this issue open as the tracker. The maintainable implementation should be a typed, capability-gated prompt-media contract rather than an automated repair that bypasses the open PR’s schema, cleanup, provider, and egress-policy questions.
