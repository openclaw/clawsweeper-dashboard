## Summary

Keep this issue open. Current main still converts inbound audio into transcription/media-understanding text and only forwards text plus image prompt attachments to the embedded agent, so native raw-audio forwarding is not implemented; the directly related implementation candidate, #48900, is still open and unmerged in the provided GitHub context.

## What This Changes

Request to let inbound voice/audio attachments reach multimodal reply models as raw audio, optionally alongside the existing STT transcript, with config and model-capability fallbacks.

## Best Possible Solution

Keep this issue as the product tracker for native inbound audio forwarding, and resolve it by reviewing/hardening #48900 or a successor that adds a typed prompt-media contract, capability-aware provider adapters, transcript-only versus audio-plus-transcript config, size/token limits, docs, and regression coverage for Telegram/Signal-style voice notes.

## Reproduction Assessment

Yes. Current-main source and regression tests give a high-confidence reproduction path: an inbound audio attachment with a successful audio.transcription output is stripped from prompt media notes, and the embedded agent prompt path only accepts text/images.

## Solution Assessment

Unclear for the open implementation candidate because it is not merged and its own context calls out a companion pi-ai mapping dependency. The maintainable fix should be typed and capability-gated across the prompt contract, config schema, provider adapters, docs, and tests rather than only passing opaque raw media blocks through an image-shaped path.
