## Best Possible Solution

Keep the explicit `--tls`/`wss://` Serve node path as the supported behavior; any broader reachability wording can be folded into normal docs polish rather than tracking this as a gateway bug.

## Reproduction Assessment

Yes for the source-level failure shape: without `--tls`, current code builds `ws://host:443`, which explains the 400 from a Tailscale Serve HTTPS endpoint. I did not run a live tailnet reproduction, but the reporter confirmed the `--tls` command fixes the real setup.

## Solution Assessment

Yes: the narrow supported solution is to use the existing TLS WebSocket flag for Serve endpoints, not to change gateway pairing. Current docs and code already carry that contract well enough to close this as implemented.
