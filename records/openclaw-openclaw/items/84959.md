## Summary

This issue should stay open, but the evidence points first to a Tailscale Serve invocation/docs gap rather than a proven gateway pairing regression: the reported command omits the TLS flag, and current node-host code only builds a `wss://` URL when TLS is explicitly enabled. A maintainer should decide whether the node CLI should infer TLS for Serve-style `*.ts.net:443` targets or whether the docs and diagnostics should make the required `--tls` path explicit.

## What This Changes

Remote node pairing over a Tailscale Serve URL fails with HTTP 400 when the gateway is loopback-bound and the node is run against the tailnet host on port 443.

## Best Possible Solution

Make the supported Serve node path unambiguous, either by documenting and diagnosing `openclaw node run --tls --host <magicdns> --port 443` or by intentionally adding tested TLS inference for Serve-style targets.

## Reproduction Assessment

Partially: the source clearly shows the reported command builds a plaintext `ws://` URL because `--tls` is absent, which plausibly explains the 400 from a Serve endpoint. I did not run a real tailnet reproduction, so the full gateway/Tailscale failure is not locally reproduced.

## Solution Assessment

Unclear: the narrowest safe fix may be documentation and a targeted diagnostic, but auto-inferring TLS for `.ts.net:443` could be a better UX if maintainers want that behavior. The current code path should not be closed as implemented until one of those choices is made and verified.
