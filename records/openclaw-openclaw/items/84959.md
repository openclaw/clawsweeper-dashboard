## Summary

Keep open: current main explains the reported failure path, but does not settle the intended UX. The node CLI only uses `wss://` when TLS is explicitly requested, while the Tailscale docs say Serve can work for node access, so maintainers should choose between clearer docs/diagnostics and intentional TLS inference for Serve-style targets.

## What This Changes

The issue asks for remote node pairing through a Tailscale Serve URL to work when the gateway is loopback-bound, instead of only working through an SSH tunnel.

## Best Possible Solution

Make the supported Serve node path unambiguous: either document and diagnose `openclaw node run --tls --host <magicdns> --port 443`, or add a deliberate, tested TLS inference rule for Serve-style targets.

## Reproduction Assessment

Partially: source inspection shows the reported command builds `ws://<magicdns>:443` because `--tls` is absent, which plausibly explains a Serve endpoint returning 400. A real tailnet run is still needed to prove the full failure and the `--tls` comparison.

## Solution Assessment

Unclear: a docs/diagnostic fix is the narrowest low-risk path, while TLS inference for `.ts.net:443` may be the better product experience. Maintainers should choose that direction before this becomes an automated fix PR.
