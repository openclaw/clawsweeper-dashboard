## Summary

Keep open: current `main` and the latest release still parse `tasks audit --limit` with numeric-prefix parsing, and the open linked fix PR is the correct path to resolve this before the issue closes.

## What This Changes

The issue asks for `tasks audit --limit` to reject malformed positive-integer values such as `1abc`, `1.5`, `0`, and `-1` instead of silently truncating numeric prefixes.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/84901; once that PR merges, close this issue as implemented by the merged fix.

## Reproduction Assessment

Yes. Source inspection on current `main` shows both the registered command path and lean JSON path still use `Number.parseInt`-based limit parsing, which accepts `1abc` as `1`; I did not run the CLI because this was a read-only review.

## Solution Assessment

Yes. The best current solution is the linked focused PR, which targets both affected parser entry points and adds regression coverage instead of changing unrelated limit behavior broadly.
