## Summary

Keep open: current main and v2026.5.19 still parse `tasks audit --limit` through prefix-based integer parsing, and the linked closing PR is open rather than merged.

## What This Changes

Require `tasks audit --limit` to reject non-positive, fractional, or partially numeric tokens before the audit runs.

## Best Possible Solution

Review and land or reject `https://github.com/openclaw/openclaw/pull/84901`; if it merges, close this issue as implemented by that PR.

## Reproduction Assessment

Yes. Current main routes both `tasks audit` entry paths through `Number.parseInt`-based positive-integer parsing, so a token like `1abc` is source-reproducibly accepted as `1`; I did not execute the CLI because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to validate this option with a strict positive-integer parser in both the Commander route and lean JSON route, rather than changing the shared lenient helper globally.
