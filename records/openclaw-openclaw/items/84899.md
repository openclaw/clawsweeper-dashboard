## Summary

Keep open: current main still routes `tasks audit --limit` through partial integer parsing, and the linked fixing PR is open with closing syntax, so the issue should remain open until that PR lands or is closed.

## What This Changes

The issue asks OpenClaw to reject malformed `tasks audit --limit` tokens such as `1abc`, `1.5`, `0`, and `-1` instead of silently truncating them.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84901, or fold its strict-parser change into a maintainer-approved validation PR with coverage for both the Commander route and lean JSON route.

## Reproduction Assessment

Yes. The issue body supplies a concrete current-main CLI command, and source inspection shows both `tasks audit` entry paths still use `Number.parseInt`-based parsing; I did not run the CLI during this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to use strict positive-integer parsing only for `tasks audit --limit` in both route paths, with focused regressions, while leaving broader shared-helper behavior to a separate maintainer decision.
