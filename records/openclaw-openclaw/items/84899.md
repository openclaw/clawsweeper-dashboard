## Summary

Keep open: current main and the latest shipped tag still accept a numeric prefix for `tasks audit --limit`, and the linked fixing PR is open rather than merged.

## What This Changes

The issue asks for `openclaw tasks audit --limit` to reject malformed positive-integer tokens such as `1abc`, `1.5`, `0`, and `-1` before running the audit.

## Best Possible Solution

Review and land the linked fix PR if it remains clean, using a strict positive-integer parser for both the registered Commander path and the lean JSON route with focused regression tests.

## Reproduction Assessment

Yes. The issue includes source-entry CLI proof, and current source still sends both `tasks audit` entry paths through `Number.parseInt`-based positive-integer parsing; I did not run the CLI to keep the target checkout read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to use strict positive-integer parsing for `tasks audit --limit` in both CLI routes and add focused regressions, rather than changing the shared loose helper globally without an explicit broader contract change.
