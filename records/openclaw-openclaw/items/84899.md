## Summary

Keep open: current main still accepts numeric prefixes for `tasks audit --limit`, and the linked open implementation at https://github.com/openclaw/openclaw/pull/84901 should be reviewed and landed before this issue is closed.

## What This Changes

The issue asks `tasks audit --limit` to reject malformed, fractional, zero, and negative values instead of accepting numeric prefixes.

## Best Possible Solution

Review and land the focused fix in https://github.com/openclaw/openclaw/pull/84901, then close this issue from that merge; if that PR closes, apply the same strict positive-integer parsing to both tasks-audit entry paths.

## Reproduction Assessment

Yes. Current source still sends both `tasks audit` entry paths through `Number.parseInt`-based parsing, so `--limit 1abc` is source-reproducible as `1`; I did not execute the CLI in this read-only review.

## Solution Assessment

Yes. The best fix is localized strict positive-integer parsing for the registered Commander path and lean JSON route, preferably reusing the existing central strict parser instead of changing unrelated command contracts.
