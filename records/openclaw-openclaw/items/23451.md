## Summary

Keep open. Current main has host exec approvals and plugin-controlled `before_tool_call.requireApproval` from the merged approval-hook work, but it still lacks the requested built-in `tools.confirmation` risk-tier config and same-response assistant-text gate before the first tool executes.

## What This Changes

Request a configurable core confirmation gate that classifies tool calls by risk and can pause execution for explicit user approval.

## Best Possible Solution

Design a backward-compatible core `tools.confirmation` config/schema/docs surface that classifies tools deterministically, flushes assistant text before holding execution, and defines approve, deny, timeout, and transcript behavior while preserving plugin `before_tool_call.requireApproval` as a lower-level hook.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence evidence that current main lacks the requested config/schema surface and response-level confirmation gate.

## Solution Assessment

Unclear until maintainers settle the product and safety policy. The direction is plausible, but the maintainable implementation should be a core gate that composes with existing plugin and exec approvals rather than replacing those primitives.
