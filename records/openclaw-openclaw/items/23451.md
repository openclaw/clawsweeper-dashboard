## Summary

Keep open: current main and v2026.6.6 provide an enforceable plugin-controlled pre-tool approval pipeline, but they still lack a built-in confirmation policy, configuration surface, and response-level context for detecting same-response confirmation text. The direct implementation attempt at https://github.com/openclaw/openclaw/pull/85872 closed unmerged and is not a viable canonical replacement because its focused change was mixed with extensive unrelated branch churn.

## What This Changes

Add a configurable human confirmation gate that pauses selected tool calls for explicit approval before execution.

## Best Possible Solution

Keep one canonical pre-execution approval pipeline, pass it prepared response-level facts when needed, and define an explicit maintainer-approved consent contract covering deterministic tool classification, authorized approvers, deny-by-default timeouts, parallel calls, channel delivery, and transcript outcomes.

## Reproduction Assessment

Yes, at source level: completed assistant responses proceed into same-turn tool execution unless an existing policy blocks or requires approval, and the tool-scoped hook lacks the assistant text required by the proposed auto mode. No runtime test was executed during this read-only review.

## Solution Assessment

No, the proposed risk tiers and natural-language question detection are not yet established as the safest permanent design. The narrower architecture is to compose a deterministic consent policy with the existing approval manager rather than introduce a parallel execution path.
