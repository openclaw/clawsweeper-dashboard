## Label Justifications

- `P1`: The request addresses a high-impact agent safety gap where destructive or sensitive tools can execute before the user can answer same-turn confirmation text.
- `impact:data-loss`: The proposed gate is intended to prevent unintended file edits, deletes, config changes, or other destructive tool effects.
- `impact:security`: The issue asks for a deterministic runtime consent boundary around risky tool execution rather than relying only on model behavior.

## Summary

Keep open: current main and the latest release still lack the requested built-in `tools.confirmation` config/schema and same-response assistant-text-aware execution gate. Existing plugin `before_tool_call.requireApproval` is useful lower-level infrastructure, but the directly related implementation PR was closed unmerged as an unmergeable stacked branch.

## What This Changes

Request a configurable built-in tool confirmation gate that can pause risky same-response tool calls until user approval.

## Best Possible Solution

Land a reviewed, backward-compatible core confirmation contract that preserves existing plugin `before_tool_call.requireApproval`, trusted tool policies, host exec approvals, and gateway authorization semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/design request rather than broken existing documented behavior. Source inspection gives high confidence that current main lacks the requested `tools.confirmation` config/schema and response-level gate.

## Solution Assessment

Yes: keeping this open for core product/security design is the right path. Plugin-only, MCP-only, or exec-only approvals cover useful subsets but do not provide the requested built-in same-response gate across risky tools and chat channels.
