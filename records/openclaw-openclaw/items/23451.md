## Label Justifications

- `P1`: The request addresses a high-impact agent safety gap where destructive or sensitive tools can execute before the user can answer same-turn confirmation text.

## Summary

Keep open: current main and the latest release still lack the requested built-in `tools.confirmation` config, risk-tier classifier, and same-response assistant-text-aware gate before first tool execution; the related implementation PR was closed unmerged as an unmergeable stacked branch rather than a fix.

## What This Changes

The issue requests a configurable tool-level confirmation gate that can hold risky tool calls until explicit human approval before execution.

## Best Possible Solution

Land a reviewed, backward-compatible core confirmation contract that composes with existing plugin approvals and host exec approvals instead of replacing them.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/design request rather than broken existing documented behavior. Source inspection gives high confidence that current main and `v2026.5.22` lack the requested `tools.confirmation` config/schema and response-level gate.

## Solution Assessment

Yes: keeping this issue open for core product/security design is the right path. Plugin-only, MCP-only, or exec-only approvals cover useful subsets but do not provide the requested built-in same-response gate across risky tools and chat channels.
