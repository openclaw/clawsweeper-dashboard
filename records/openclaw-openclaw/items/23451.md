## Summary

Keep open. Current main still has lower-level approval primitives, including plugin `before_tool_call.requireApproval`, exec approvals, and a Codex native permission relay, but it does not implement the requested built-in `tools.confirmation` gate with configurable risk tiers and same-response assistant-text inspection before the first tool executes.

## What This Changes

Request a configurable core tool confirmation gate that can pause tool execution for human approval based on tool risk and same-response assistant confirmation text.

## Best Possible Solution

Add a backward-compatible core confirmation feature with config/schema/help, deterministic tool risk classification, same-response assistant-text access before first execution, visible text flushing, explicit approve/deny/timeout handling, skipped-call transcript semantics, and regression coverage proving it composes with existing approval paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: the proposed `tools.confirmation` config is absent from strict schemas, and the current pre-tool event lacks assistant text, so current main cannot express the requested same-response confirmation gate.

## Solution Assessment

No. The existing plugin and exec approval primitives are useful infrastructure, but they are not the complete or narrowest solution for this request; the maintainable path is a core response-level gate that composes with them.
