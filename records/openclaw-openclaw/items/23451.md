## Summary

Keep open. Current main has exec approvals and plugin-controlled `before_tool_call.requireApproval`, but still lacks the requested built-in `tools.confirmation` risk-tier config and same-response assistant-text-aware gate before the first tool executes.

## What This Changes

The issue requests a configurable tool-level confirmation gate that pauses tool execution for human approval based on risk level and same-response assistant text.

## Best Possible Solution

Design a backward-compatible core `tools.confirmation` config/schema/docs surface that classifies tool risk, flushes assistant text before holding execution, and defines approve, deny, timeout, and transcript behavior while preserving existing plugin and exec approval primitives.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence evidence that current main lacks the requested config/schema surface and response-level confirmation gate.

## Solution Assessment

Unclear until maintainers settle the product and safety policy. The requested core gate is plausible, but the maintainable implementation should compose with plugin `requireApproval` and exec approvals rather than replacing those primitives.
