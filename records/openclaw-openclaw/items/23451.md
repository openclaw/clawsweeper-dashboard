## Summary

Keep open: current main has plugin-triggered pre-tool approvals and host exec approvals, but not the requested built-in, configurable confirmation gate. The config/schema surface has no `tools.confirmation` block, and the existing `before_tool_call` decision point sees tool name and params, not the same-response assistant text needed for the proposed auto-confirm behavior.

## What This Changes

Request a backward-compatible tool confirmation gate that classifies tools by risk and can pause execution after streaming assistant text for explicit user approval.

## Best Possible Solution

Design a core, documented `tools.confirmation` policy that is disabled or backwards-compatible by default, classifies tools deterministically, can inspect same-response assistant text before the first tool executes, flushes user-visible text before holding execution, handles approve/deny/timeout transcript outcomes, and composes cleanly with existing plugin approvals and host exec approvals.
