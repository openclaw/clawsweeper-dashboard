## Label Justifications

- `P1`: The issue describes a broad user-facing safety gap where risky tool calls can execute before the user can answer the assistant's confirmation question.
- `impact:data-loss`: The requested gate is meant to prevent unintended writes, edits, deletes, configuration changes, or other persistent mutations before approval.
- `impact:security`: The issue concerns a hard runtime authorization boundary for side-effecting tools instead of relying on model text or prompt compliance.

## Summary

Keep open. Current main and v2026.6.6 still lack the requested configurable, assistant-text-aware tool confirmation gate; existing plugin `before_tool_call.requireApproval` and exec approvals are useful lower-level primitives but do not provide a core risk-tier policy that can hold the first same-response tool call before execution.

## What This Changes

Request a disabled-by-default core `tools.confirmation` policy that can pause risky tool calls for user approval before execution when the assistant text asks for confirmation.

## Best Possible Solution

Define a documented, disabled-by-default core `tools.confirmation` contract with deterministic risk tiers, same-response assistant-text access, explicit approve/deny/timeout behavior, transcript-safe skipped-tool results, and regression coverage against existing approval hooks.

## Reproduction Assessment

Yes, at source level. Current main has no `tools.confirmation` config/schema, the pre-tool event lacks same-response `assistantTexts`, and the tool adapter executes after the existing hook path when no lower-level policy blocks; I did not run a live destructive repro.

## Solution Assessment

Unclear as a final design. A core execution gate is the right layer for the first same-response tool call, but risk tiers, timeout behavior, approval routing, transcript semantics, and channel UX need maintainer product/security review.
