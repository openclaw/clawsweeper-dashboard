## Label Justifications

- `P1`: The issue describes a broken safety workflow where chat users can see a confirmation question only after the same response has already executed side-effecting tools.
- `impact:data-loss`: The requested gate is meant to prevent unintended file edits, writes, config changes, and other persistent side effects before user approval.
- `impact:security`: The issue concerns a hard execution authorization boundary for high-risk tools and human approval.

## Summary

Keep open: current main still lacks the requested built-in configurable tool-level confirmation gate, while existing plugin approvals and exec approvals are adjacent lower-level mechanisms that do not inspect same-response assistant text before the first tool executes.

## What This Changes

The issue requests a configurable, risk-tiered confirmation gate that can pause tool execution before same-response tool calls run.

## Best Possible Solution

Define a backward-compatible core confirmation policy with typed config/schema/help, deterministic tool-risk classification, same-response assistant-text access before execution, approve/deny/timeout semantics, transcript handling for skipped tools, and regression coverage preserving existing plugin approvals.

## Reproduction Assessment

Yes, source-level: current main has no tools.confirmation config/schema, no plugin pre-tool assistant text, and no built-in risk-tier gate before the first same-response tool executes.

## Solution Assessment

Unclear as a final design: a core gate is the right layer for the first same-response tool call, but the config shape, risk tiers, routing, timeouts, and transcript semantics need maintainer product/security review.
