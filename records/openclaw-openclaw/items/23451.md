## Label Justifications

- `P1`: The request concerns sensitive tool calls executing before a user can answer an apparent confirmation prompt in real agent workflows.
- `impact:data-loss`: Write, edit, exec, and configuration tools can modify or delete user state before consent is mechanically enforced.
- `impact:security`: The issue asks for a host authorization boundary between model-proposed tool calls and execution.

## Summary

Keep open: current main and v2026.6.5 still have plugin-driven pre-tool approval and exec approvals, but no built-in configurable tool confirmation policy, risk tiers, or same-response assistant-text-aware gate. This remains valid core safety work and needs maintainer product/security design before implementation.

## What This Changes

The issue requests a configurable host-enforced tool confirmation gate that can pause risky tool calls before execution when model text asks for confirmation.

## Best Possible Solution

Define one canonical host approval state machine that composes with existing plugin approvals, uses deterministic tool/action classification, returns explicit denial or timeout results, preserves transcript shape, and documents backward-compatible defaults.

## Reproduction Assessment

Yes, at the source level: current strict schemas have no general confirmation policy, and the before-tool hook event does not include same-response assistant text. I did not run a live destructive model-output scenario, which is unnecessary for proving the missing capability.

## Solution Assessment

No, not as the exact proposed shape. The host-enforced boundary is the right layer, but free-form question detection and a parallel approval system should be replaced by deterministic policy on one canonical approval primitive.
