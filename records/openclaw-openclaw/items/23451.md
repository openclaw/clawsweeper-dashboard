## Summary

Keep open. Current main has lower-level exec and plugin approval primitives, including `before_tool_call.requireApproval`, but it still lacks the requested built-in `tools.confirmation` config/schema surface and response-level gate that can inspect same-response assistant text before the first tool executes.

## What This Changes

Request a configurable runtime confirmation gate that can pause high-risk tool calls before execution when assistant text and tool calls are emitted in the same model response.

## Best Possible Solution

Add a documented, disabled-by-default core confirmation policy that composes with existing exec and plugin approval paths, with deterministic risk classification, explicit approve/deny/timeout results, and regression coverage for same-response text plus tool calls.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main lacks the requested config/schema and response-level confirmation gate.

## Solution Assessment

Unclear until maintainers choose the policy shape. The direction is plausible, but the maintainable solution needs explicit product/security decisions around risk tiers, assistant-text matching, timeout defaults, denial results, and compatibility with existing approval hooks.
