## Summary

Keep open: current main still has exec approvals and plugin `before_tool_call.requireApproval`, but not the requested built-in, configurable response-level confirmation gate that can inspect same-response assistant text before the first risky tool executes.

## What This Changes

The issue requests a configurable runtime confirmation gate that can pause risky tool calls before execution when the model emits assistant text and tool calls together.

## Best Possible Solution

Design a core, backward-compatible `tools.confirmation` contract with deterministic risk tiers, same-response assistant-text awareness, visible text flush before holding tools, explicit approve/deny/timeout semantics, and regression coverage across agent, config, plugin, and channel approval paths.

## Reproduction Assessment

Not applicable as a feature/design request. Current-main source inspection gives a high-confidence state check that the requested config/schema surface and response-level gate are absent.

## Solution Assessment

Yes, keeping this open for core design is the best current path. A plugin-only or exec-only solution would not cover the requested built-in gate across risky tools and chat channels.
