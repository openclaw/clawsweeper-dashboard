## Summary

Keep this issue open: current main has exec approvals and plugin-driven `before_tool_call.requireApproval`, but it still lacks the requested built-in, configurable response-level confirmation gate for all risky tool calls.

## What This Changes

Requests a disabled-by-default configurable policy that can hold risky tool calls for human confirmation before execution, especially when assistant text asks for permission in the same model response.

## Best Possible Solution

Keep this issue open for maintainer security/product design of a core confirmation policy that composes cleanly with plugin `before_tool_call.requireApproval`, exec approvals, trusted tool policies, transcript behavior, and channel approval UX.

## Reproduction Assessment

Yes from source inspection, not from a live scenario: current main has no accepted `tools.confirmation` config/schema and the pre-tool path only blocks when existing lower-level policies or plugins request it.

## Solution Assessment

Unclear until maintainers choose the security model; a core gate is a plausible layer, but the final design must avoid duplicating or bypassing exec approvals, plugin approval hooks, and trusted policy registration.
