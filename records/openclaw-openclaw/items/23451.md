## Summary

Keep open: current main and the latest release still have lower-level exec/plugin approval primitives, but not the requested built-in, configurable response-level confirmation gate for risky tool calls.

## What This Changes

The issue requests a configurable tool confirmation gate that can pause risky tool calls for human approval before execution.

## Best Possible Solution

Keep this issue open for a maintainer-designed, disabled-by-default core confirmation policy with deterministic risk/intent semantics, schema/docs coverage, and regression tests around tool execution ordering and existing approval primitives.

## Reproduction Assessment

Yes, from source inspection: current main has no tools.confirmation config/schema surface, and the pre-tool path only blocks when existing lower-level policies or plugins request it. I did not run a live destructive-tool scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security model. A core gate is the likely right layer for same-response tool calls, but the final design must avoid duplicating exec approvals, plugin approval hooks, and trusted policy registration.
