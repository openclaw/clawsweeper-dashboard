## Label Justifications

- `P1`: The issue describes a broken safety boundary where agents can perform risky tool actions before users can answer a same-response confirmation prompt.
- `impact:data-loss`: The requested gate is meant to prevent unintended file edits, deletes, config changes, and other persistent mutations.
- `impact:security`: The issue is about enforcing human authorization before sensitive tool execution rather than relying on model text.

## Summary

Keep open: current main still lacks a built-in, configurable tool-level confirmation gate before execution, while existing plugin approvals and exec approvals are adjacent lower-level primitives rather than the requested core policy.

## What This Changes

Request a configurable tool confirmation gate that can pause risky tool calls, flush assistant text, and wait for explicit human approval before execution.

## Best Possible Solution

Define a backward-compatible core tool-confirmation policy seam, then implement config/schema/help, deterministic risk classification, approve/deny/timeout behavior, transcript handling for skipped tools, and channel-visible proof while preserving existing plugin and exec approval paths.

## Reproduction Assessment

Yes, at source level. Current main has no tools.confirmation config/schema surface, and the agent loop proceeds from assistant tool calls into execution through only existing beforeToolCall/plugin approval primitives.

## Solution Assessment

Unclear as proposed: the core direction is valid, but recent comments correctly note that parsing natural-language questions is fragile. The best fix should be a deterministic intent/risk execution gate rather than only a text-pattern heuristic.
