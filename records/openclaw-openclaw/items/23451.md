## Label Justifications

- `P1`: The issue describes a safety gap where agents can perform file, message, gateway, or exec actions before the user can answer the assistant's own confirmation question.
- `impact:data-loss`: The missing gate can allow unintended file, config, or persistent side-effect operations before user consent.

## Summary

Keep open: current main and v2026.6.6 still provide lower-level exec/plugin approval primitives, but not the requested runtime-owned configurable gate that can inspect same-response assistant text before the first tool executes.

## What This Changes

Request a disabled-by-default configurable tool confirmation gate that can pause risky tool calls for user approval before execution.

## Best Possible Solution

Design a backward-compatible runtime-owned confirmation gate with schema/docs, deterministic risk classification, same-response assistant-text access, explicit approve/deny/timeout outcomes, and regression coverage around existing approval primitives.

## Reproduction Assessment

Yes, at source level. Current main has no tools.confirmation config/schema and the agent loop still proceeds from assistant tool calls to execution without a response-level confirmation gate.

## Solution Assessment

Yes on the layer, unclear on the exact config shape. A runtime gate between parsed tool calls and execution is the durable boundary, but maintainers need to settle risk tiers, timeout semantics, channel UX, and transcript handling.
