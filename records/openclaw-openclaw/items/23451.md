## Summary

Keep open: current main still has lower-level exec/plugin approval primitives, but not the requested built-in, configurable response-level confirmation gate that can inspect same-response assistant text before the first risky tool executes.

## What This Changes

Request a configurable, disabled-by-default tool confirmation gate that classifies tool calls by risk and pauses execution for explicit user approval before risky tools run.

## Best Possible Solution

Keep this open for a maintainer-designed core confirmation policy with deterministic risk/intent semantics, schema/docs coverage, and regression tests around tool execution ordering and existing approval primitives.

## Reproduction Assessment

Yes, at source level: current main has no tools.confirmation config/schema surface and the pre-tool path only blocks when existing lower-level policies or plugins request it. I did not run a live destructive-tool scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security model. A core gate is the likely right layer for same-response tool calls, but it must not duplicate or bypass existing exec approval and plugin approval seams.
