## Summary

Keep open: current main and the latest release still have only lower-level plugin and exec approval primitives, not the requested built-in `tools.confirmation` config, risk-tier classifier, or same-response assistant-text gate before first tool execution. The recent discussion usefully confirms this is an execution-boundary design issue, so it needs maintainer product and security review rather than cleanup closure.

## What This Changes

Requests a configurable core tool confirmation gate that can pause risky tool calls based on risk tiers and same-response assistant text before execution.

## Best Possible Solution

Design a backward-compatible core `tools.confirmation` contract with deterministic risk tiers, same-response assistant-text awareness, visible text flush before holding tools, explicit approve/deny/timeout semantics, and regression coverage across agent, config, plugin, and channel approval paths.

## Reproduction Assessment

Not applicable as a feature/design request. Source inspection gives a high-confidence state check that the requested config/schema surface and response-level gate are absent on current main and the latest release.

## Solution Assessment

Yes, keeping this open for core design is the best current path. A plugin-only or exec-only solution would not cover the requested built-in gate across risky tools and chat channels.
