## Summary

Keep open: current `origin/main` still has lower-level exec approvals and plugin `before_tool_call.requireApproval`, but not the requested built-in response-level confirmation gate that can inspect same-response assistant text before the first risky tool executes.

## What This Changes

The issue requests a configurable tool confirmation gate that can pause risky tool execution after an LLM response is parsed but before the first tool call runs.

## Best Possible Solution

Design a core, backward-compatible `tools.confirmation` contract with deterministic risk tiers, same-response assistant-text awareness, visible text flush before holding tools, explicit approve/deny/timeout semantics, and regression coverage across agent, config, plugin, and channel approval paths.

## Reproduction Assessment

Not applicable as a feature/design request. Current-main source inspection gives a high-confidence state check that the requested config/schema surface and response-level gate are absent.

## Solution Assessment

Partly: the deterministic runtime gate is the right boundary, but the exact risk tiers, auto-confirm heuristic, timeout behavior, and channel UX need maintainer product/security design before implementation.
