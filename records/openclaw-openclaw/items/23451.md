## Label Justifications

- `P1`: The issue concerns sensitive tool actions executing before a user can answer an apparent confirmation request, affecting real agent workflows across channels.
- `impact:data-loss`: Write, edit, exec, and configuration tools can modify or delete user state before consent is mechanically enforced.
- `impact:security`: The requested capability establishes a human authorization boundary between a model-proposed tool call and host execution.

## Summary

Current main and v2026.6.5 provide a generic plugin-driven pre-tool approval primitive, but they still do not provide the requested built-in confirmation policy, risk tiers, or same-response text-aware gate. The issue remains necessary, while its security semantics, defaults, ownership boundary, channel interaction, and transcript behavior require explicit maintainer product review.

## What This Changes

Add a configurable host-enforced confirmation policy that can pause selected tool calls for explicit human approval before execution.

## Best Possible Solution

Define one canonical host approval state machine that reuses the existing approval manager and plugin hook, then add any core policy only with deterministic tool/action classification, explicit fail-closed denial and timeout semantics, valid skipped-tool transcript results, channel-authorized approval routing, and a backward-compatible documented default.

## Reproduction Assessment

Yes at the source level: current strict schemas have no general confirmation policy, and the before-tool decision event has no same-response assistant text while allowed calls proceed to execution. This establishes the missing capability without claiming a deterministic model-output reproduction.

## Solution Assessment

No, not as fully specified: the host-enforced gate is the right boundary, but free-form question detection and a parallel approval system are unsafe. The maintainable design should extend the existing approval primitive with deterministic policy and one canonical state machine.

## Risks / Open Questions

- A core classifier could create a second source of truth beside plugin `before_tool_call.requireApproval`, producing conflicting or duplicated prompts.
- The proposed `auto` mode depends on interpreting assistant language; false negatives could permit destructive actions and false positives could stall ordinary workflows.
- Approval identity, channel routing, timeout defaults, cancellation, denial results, multi-tool ordering, and transcript persistence are security-sensitive product contracts that are not yet specified.
- Adding another config surface increases compatibility and operator complexity; fresh-install and upgrade behavior would need explicit default and documentation decisions.
