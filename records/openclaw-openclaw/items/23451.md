## Label Justifications

- `P1`: The issue describes a safety gap where chat-originated agents can perform file, message, gateway, or exec actions before a user can answer the assistant's own confirmation question.
- `impact:data-loss`: The reported missing gate can allow unintended file/config modifications or other persistent side effects before user consent.
- `impact:security`: The requested change is about enforcing a runtime authorization boundary for risky tool execution instead of relying on model text compliance.

## Summary

Keep open: current main and the latest release still have plugin/exec approval primitives, but not the requested runtime-owned, configurable tool confirmation gate that can inspect same-response assistant text before the first tool executes.

## What This Changes

Request to add a configurable tool-level confirmation gate that can pause risky tool calls for human approval before execution.

## Best Possible Solution

Design a backward-compatible, runtime-owned confirmation gate before tool execution, with schema/docs, deterministic tool-risk classification, same-response assistant-text access, explicit approval outcomes, and regression coverage around existing plugin and exec approvals.

## Reproduction Assessment

Yes, at source level. Current main has no `tools.confirmation` config/schema and the agent loop still proceeds from assistant tool calls to tool execution without a response-level confirmation gate.

## Solution Assessment

Yes on the layer, unclear on the exact config shape. A runtime gate between parsed tool calls and execution is the right durable boundary, but the risk tiers, timeout semantics, and channel approval UX need maintainer design before code work.
