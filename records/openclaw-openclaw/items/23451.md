## Summary

Keep open: current main still has lower-level exec and plugin approval primitives, but not the requested built-in, configurable response-level confirmation gate for risky tool calls.

## What This Changes

Request a configurable tool confirmation policy that can pause risky tool execution and wait for human approval before execution.

## Best Possible Solution

Design a core, backward-compatible confirmation policy that preserves existing approval primitives while adding deterministic risk classification, text/intent handling, approve/deny/timeout semantics, and regression coverage.

## Reproduction Assessment

Yes, from source inspection: current main has no `tools.confirmation` config/schema surface, and the pre-tool path only blocks when existing lower-level policies or plugins request it. I did not run a live destructive-tool scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security model. A core gate is the right likely layer for same-response tool calls, but the final design must avoid duplicating exec approvals, plugin approval hooks, and trusted policy registration.
