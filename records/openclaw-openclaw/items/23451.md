## Label Justifications

- `P1`: The issue describes a missing hard execution gate for high-risk tool calls that can affect real users before they can respond.
- `impact:data-loss`: Unconfirmed write/edit/exec tool calls can modify or delete local user data before human approval.
- `impact:security`: The requested behavior is about execution authorization and approval boundaries for destructive or credential-bearing actions.

## Summary

Keep open: current main and v2026.6.6 still lack a built-in, configurable tool-level confirmation gate; existing plugin and exec approval primitives are useful lower-level pieces but do not solve the same-response first-tool execution problem.

## What This Changes

The issue requests a configurable core confirmation gate that can pause risky tool calls before execution when assistant text asks for confirmation in the same response.

## Best Possible Solution

Design a default-off core confirmation gate with config/schema/docs, deterministic tool/action risk classification, executor-level approve/deny/timeout behavior, transcript-safe skipped tool results, and regression tests that preserve existing plugin and exec approval paths.

## Reproduction Assessment

Yes, from source inspection. Current main has strict tool config schemas without tools.confirmation, and the current pre-tool path only gates when existing policies or hooks request approval; I did not run a live destructive-agent scenario because this review is read-only.

## Solution Assessment

Unclear until maintainer design review. A core executor-level gate is the right layer for same-response first-tool holds, but the exact policy, defaults, timeout behavior, and approval composition are product/security decisions.
