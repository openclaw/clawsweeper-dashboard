## Summary

Keep open. Current main has lower-level approval primitives, plugin `before_tool_call.requireApproval`, trusted tool policies, and host exec approvals, but it still does not implement the requested built-in configurable tool confirmation gate before risky same-response tool execution.

## What This Changes

Request a configurable runtime confirmation system that prevents risky exec, file, or config-changing tool calls from running before the user has actually approved them.

## Best Possible Solution

Keep this open for a maintainer-approved core design covering config/schema/help, deterministic tool risk tiers, same-response assistant-text access, user-visible text flush before holding tools, approve/deny/timeout behavior, transcript handling for skipped calls, and regression tests that preserve plugin `before_tool_call.requireApproval` as a lower-level hook.

## Reproduction Assessment

Yes, from source inspection: current main still has no `tools.confirmation` config/schema, and the before-tool-call event lacks same-response assistant text needed for the proposed auto gate. I did not run a live destructive repro in this read-only review.

## Solution Assessment

Unclear as final design: a core execution gate is the right layer for preventing the first same-response risky tool call, but risk tiers, approval routing, timeout behavior, channel UX, and transcript semantics need maintainer product/security direction.
