## Summary

Keep open. Current main still lacks the requested built-in, configurable tool confirmation gate; merged approval work provides lower-level plugin `before_tool_call.requireApproval`, but not a core risk-tier classifier, `tools.confirmation` config/schema surface, or response-level hold that can inspect same-response assistant text before the first tool executes.

## What This Changes

Add a configurable core gate that can pause tool execution for human approval based on tool risk and same-response confirmation intent.

## Best Possible Solution

Design a backward-compatible core `tools.confirmation` surface with deterministic risk classification, assistant-text-aware hold behavior, explicit approve/deny/timeout semantics, and regression coverage that preserves plugin approvals as the lower-level policy hook.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main confirms the requested config and response-level gate are absent, while only lower-level hook/plugin approval primitives exist.

## Solution Assessment

Unclear until maintainers settle the product contract. The requested direction is plausible, but the best implementation should be a core execution-boundary design layered over the existing hook and gateway approval primitives rather than a plugin-only workaround.
