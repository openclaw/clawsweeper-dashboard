## Summary

Keep open: current main and v2026.5.12 still have exec approvals and plugin-driven `before_tool_call.requireApproval`, but they do not implement the requested built-in `tools.confirmation` config, risk-tier classifier, or same-response assistant-text gate before the first tool executes. This remains a core consent-boundary feature that needs product and security review.

## What This Changes

Request a configurable, risk-tiered tool confirmation gate that can pause tool execution for explicit human approval before risky tools run.

## Best Possible Solution

Design a backward-compatible core `tools.confirmation` contract with deterministic risk tiers, same-response assistant-text awareness, visible text flush before holding tools, explicit approve/deny/timeout semantics, and regression coverage across agent, config, plugin, and channel approval paths.

## Reproduction Assessment

Not applicable as a feature/design request. Source inspection gives high-confidence evidence that the requested config/schema surface and response-level gate are absent on current main and v2026.5.12.

## Solution Assessment

Yes, keeping this open for core design is the best current path. A plugin-only, MCP-only, or exec-only solution would not cover the requested built-in gate across risky tools and chat channels.
