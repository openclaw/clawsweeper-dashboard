## Summary

Keep open: current main still has exec and plugin approval primitives, but not the requested built-in, configurable response-level confirmation gate for risky tool calls.

## What This Changes

The issue requests a configurable tool-level confirmation gate before risky tool execution.

## Best Possible Solution

Keep this issue as the canonical design thread for a backward-compatible core confirmation policy that preserves existing exec approvals, plugin `before_tool_call.requireApproval`, and trusted tool policies.

## Reproduction Assessment

Yes, from source inspection: current main has no `tools.confirmation` config/schema surface, and the pre-tool path only blocks when existing lower-level policies or plugins request it. I did not run a live destructive-tool scenario in this read-only review.

## Solution Assessment

No for closure: exec approvals and plugin approval hooks are useful lower-level primitives, but they do not fully solve same-response first-tool confirmation. The best fix needs maintainer security/product design before code work.
