## Label Justifications

- `P1`: The missing gate can allow high-risk tool calls to execute before a user can answer a same-response confirmation question, affecting real agent/channel workflows.
- `impact:data-loss`: The requested gate is meant to prevent unintended file edits, config mutations, or other destructive tool effects before user consent.
- `impact:security`: The issue concerns a hard human-approval boundary for sensitive tool execution, credentials, messaging, and host operations.

## Summary

Keep open: current main and v2026.6.6 still lack the requested built-in, configurable tool confirmation gate. Existing exec and plugin approval paths are useful lower-level primitives, but they do not provide the requested core risk-tier classifier, `tools.confirmation` config/schema surface, or same-response assistant-text-aware gate before the first tool executes.

## What This Changes

Request to add a configurable core tool-level confirmation gate that can pause risky tool calls for human approval before execution.

## Best Possible Solution

Add a maintainer-approved, disabled-by-default core confirmation policy with documented config/schema/help, deterministic tool-risk classification, same-response assistant-text handling, explicit approve/deny/timeout outcomes, transcript-safe skipped tool results, and tests that preserve existing plugin approval behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction of the missing capability: current main has no `tools.confirmation` config/schema field and the pre-tool path only blocks when existing policies or plugin hooks require it. I did not run a live model scenario in this read-only review.

## Solution Assessment

Unclear as a final design: a core execution gate is the right layer for the first same-response tool call, but the exact config shape, risk tiers, approval routing, timeout behavior, and transcript semantics need maintainer product/security review.
