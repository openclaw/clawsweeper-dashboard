## Label Justifications

- `P1`: The request addresses a high-impact agent safety gap where destructive or sensitive tools can execute before the user can answer same-turn confirmation text.
- `impact:data-loss`: The proposed gate is intended to prevent unintended file edits, deletes, config changes, or other destructive tool effects.

## Summary

Keep open: current main still lacks the requested built-in `tools.confirmation` config, risk-tier classifier, and same-response assistant-text-aware execution gate, and the new related implementation PR is still open rather than merged.

## What This Changes

The issue requests a disabled-by-default, configurable runtime gate that can pause risky tool calls before execution when assistant text asks for human confirmation.

## Best Possible Solution

Land a reviewed, backward-compatible core confirmation contract that preserves existing plugin `before_tool_call.requireApproval`, trusted tool policies, host exec approvals, and gateway approval authorization semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/design request rather than broken existing documented behavior. Source inspection gives high confidence that current main lacks the requested `tools.confirmation` config/schema and response-level gate.

## Solution Assessment

Yes: keeping this open for core product/security design and review of the related implementation PR is the right path. Plugin-only, MCP-only, or exec-only approvals cover useful subsets but do not provide the requested built-in same-response gate across risky tools and chat channels.
