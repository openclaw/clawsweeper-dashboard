## Summary

Keep open. Current main still lacks the requested built-in, configurable tool confirmation gate; the merged plugin approval work is useful adjacent infrastructure, but it does not provide a core risk-tier classifier, `tools.confirmation` config/schema, or a response-level hold that can inspect same-response assistant text before the first tool executes. The latest discussion correctly frames this as an execution-boundary design problem, not a natural-language prompt convention.

## What This Changes

Add a configurable, risk-based confirmation gate that can pause risky tool calls before execution and wait for explicit user approval.

## Best Possible Solution

Design a backward-compatible core `tools.confirmation` surface with deterministic risk classification, assistant-text-aware hold behavior, explicit approve/deny/timeout semantics, and regression coverage that preserves plugin approvals as the lower-level policy hook.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main confirms the requested config and response-level gate are absent, while only lower-level hook/plugin approval primitives exist.

## Solution Assessment

Unclear until maintainers settle the product and security contract. The requested direction is plausible, but the maintainable solution should be a core execution-boundary design layered over existing hook and gateway approval primitives rather than a plugin-only workaround.
