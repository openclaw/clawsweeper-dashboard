## Summary

Keep open. Current main has useful lower-level approval primitives, especially plugin `before_tool_call.requireApproval` from #55339 and host exec approvals, but it still lacks the requested built-in configurable tool confirmation gate with risk tiers and same-response assistant-text inspection before the first tool executes.

## What This Changes

Request a configurable core tool confirmation gate that can pause tool execution for human approval based on tool risk and whether the assistant text is asking for confirmation.

## Best Possible Solution

Add a backward-compatible core confirmation feature: documented config/schema/help for `tools.confirmation`, deterministic tool risk classification, access to same-response assistant text before the first tool executes, visible text flush before holding tools, explicit approve/deny/timeout behavior, transcript handling for skipped calls, and regression coverage proving it composes with plugin `before_tool_call.requireApproval` and exec approvals.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: the proposed `tools.confirmation` config is absent from strict config schemas, and the current pre-tool event lacks `assistantTexts`, so the requested same-response confirmation gate cannot be expressed on current main.

## Solution Assessment

No. The current plugin approval hook is valuable infrastructure but is not the complete or narrowest solution for this request; the maintainable fix is a core response-level gate that composes with plugin and exec approvals.
