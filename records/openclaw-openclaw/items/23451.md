## Label Justifications

- `P1`: The issue covers a high-impact agent safety gap where destructive or sensitive tools can run before a user can answer a same-turn confirmation question.
- `impact:data-loss`: The requested gate is meant to prevent unintended file edits, deletes, config changes, or other destructive tool effects.
- `impact:security`: The issue is about moving risky tool execution from model self-discipline to a deterministic runtime approval boundary.

## Summary

Keep open: current main still has exec approvals and plugin-level `before_tool_call.requireApproval`, but it does not have the requested built-in `tools.confirmation` config, risk-tier classifier, or same-response assistant-text-aware gate before the first tool executes.

## What This Changes

Add a configurable, risk-tiered confirmation gate that can pause risky tool execution until the user explicitly approves or denies it.

## Best Possible Solution

Design a backward-compatible core `tools.confirmation` contract with deterministic risk tiers, same-response assistant-text awareness, visible text flush before holding tools, explicit approve/deny/timeout semantics, and regression coverage across agent, config, plugin, and channel approval paths.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/design request rather than a broken existing contract. Source inspection gives high confidence current main lacks the requested `tools.confirmation` config/schema and response-level gate.

## Solution Assessment

Yes: keeping this open for core design is the best current path. Plugin-only, MCP-only, or exec-only approvals do not cover the requested built-in gate across risky tools and chat channels.
