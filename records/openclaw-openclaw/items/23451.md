## Summary

Keep this open: current main still has lower-level plugin and exec approval primitives, but not the requested configurable, response-level confirmation gate that can inspect same-response assistant text before the first risky tool executes.

## What This Changes

Request a disabled-by-default, configurable tool confirmation gate that can pause risky tool calls before execution based on risk tier and assistant confirmation text.

## Best Possible Solution

Design a core, disabled-by-default confirmation gate with documented config/schema/help, deterministic risk tiers, assistant-text-aware hold behavior, approve/deny/timeout transcript semantics, and regression coverage against existing plugin and exec approval paths.

## Reproduction Assessment

Yes, from source inspection: current main has no `tools.confirmation` config/schema, no assistant-text-aware pre-tool decision event, and no response-level risk gate before `executeToolCalls`. I did not run a live model repro in this read-only review.

## Solution Assessment

Unclear as a final design: a core execution gate is the right layer for the first same-response tool call, but the exact risk tiers, config shape, approval routing, timeout behavior, and transcript semantics need maintainer product/security review.
