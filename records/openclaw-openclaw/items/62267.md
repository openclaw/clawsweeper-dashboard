## Summary

Keep open. Current main has the expected Ollama tuning and mitigation paths, but the maintainer discussion already narrowed this to a data-dependent question: whether OpenClaw is exceeding the configured prompt budget or Ollama is spending a valid capped prompt in slow prompt evaluation. The issue is recent, lacks the requested current-main config/log/`prompt_eval_*` evidence, and the current code/docs controls do not prove the qwen3:14b fallback behavior is fixed.

## What This Changes

The issue asks OpenClaw to avoid impractically slow local Ollama qwen3:14b long-context prompt ingestion and cloud fallback by respecting effective context caps and handling slow prompt evaluation separately from provider failure.

## Best Possible Solution

Keep the issue open for the requested current-main retest. The end state should be evidence-driven: compare OpenClaw's submitted prompt/context size with configured `contextWindow` or `contextTokens` and Ollama `prompt_eval_count`; fix prompt budgeting or compaction if OpenClaw exceeds the cap, otherwise improve local-model diagnostics/docs around `params.num_ctx`, `timeoutSeconds`, `localModelLean`, model sizing, and fallback interpretation.

## Reproduction Assessment

Unclear. A maintainer reproduced the underlying native Ollama prompt-eval slowness, but the issue does not yet provide enough current-main OpenClaw config, fallback log, and matching Ollama `prompt_eval_*` metrics to reproduce the OpenClaw-specific failure with high confidence.

## Solution Assessment

Unclear. There is no proposed patch, and the narrowest maintainable solution depends on the missing data: runtime prompt-budgeting/compaction should change only if OpenClaw exceeds the configured cap; otherwise this is better handled through config guidance and diagnostics.
