## What This Changes

The issue asks OpenClaw to resolve MiniMax portal auth for a selected agent from that agent's auth-profiles.json instead of effectively resolving through the main/default agent store.

## Best Possible Solution

Land a focused auth-runtime fix that keys active auth snapshots by stable runtime identity, keeps default/main fallback explicit, and adds a regression covering distinct main and selected-agent minimax-portal profiles when OPENCLAW_AGENT_DIR points at the selected agent.

## Solution Assessment

Yes. The narrowest maintainable solution is to fix runtime snapshot identity/resolution in the shared auth-profile path and preserve existing read-through semantics, rather than adding MiniMax-specific behavior or copying credentials across agents.

## Repair Work Prompt

Fix the agent-specific MiniMax auth runtime snapshot collision reported at https://github.com/openclaw/openclaw/issues/64274. Current main keys active auth profile runtime snapshots by resolved auth-store paths, so when OPENCLAW_AGENT_DIR points at a selected agent, the default and selected-agent snapshot identities can collapse and MiniMax portal auth may resolve from the wrong store. Use https://github.com/openclaw/openclaw/pull/64320 as source context, but rebase the idea onto current main.

Do not copy OAuth credentials between agents, do not broaden OAuth portability, do not change MiniMax provider ids or model catalog semantics, and do not add core special cases for MiniMax beyond regression coverage. Treat https://github.com/openclaw/openclaw/issues/44571 as adjacent context only unless the same narrow fix naturally improves it.
