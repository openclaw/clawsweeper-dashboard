## Summary

Keep open: current main and the latest shipped tag still merge the main auth store ahead of the selected agent store, so a selected-agent MiniMax OAuth profile can lose to an inherited main MiniMax profile when the profile IDs differ and no explicit order overrides it.

## What This Changes

The issue asks for a selected agent's MiniMax auth to resolve from that agent's auth-profiles.json instead of the main agent store.

## Best Possible Solution

Land a focused shared auth-profile precedence fix so selected-agent profiles for a provider override inherited main profiles for that provider, while retaining main fallback for providers absent from the selected agent.

## Reproduction Assessment

Yes by source inspection, though not by executing a live MiniMax account flow. Create main and selected-agent stores with different eligible `minimax-portal` OAuth profile IDs and no explicit auth order; the current merge/order path can try the inherited main profile before the selected-agent profile.

## Solution Assessment

Yes. The narrow maintainable fix belongs in shared auth-profile merge or ordering semantics, not in MiniMax-specific credential handling or OAuth token copying.
