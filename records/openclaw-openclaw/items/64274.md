## Summary

Keep open: current main still has the auth-profile runtime snapshot identity path that can collapse the selected-agent store with the active/default store when `OPENCLAW_AGENT_DIR` points at that selected agent, and the direct fix PR is closed unmerged.

## What This Changes

Resolve MiniMax portal auth for selected agents from the selected agent's `auth-profiles.json` instead of effectively using the main or active-agent auth store.

## Best Possible Solution

Land a focused auth-runtime fix that keys active snapshots by stable runtime identity, preserves explicit default/main fallback semantics, and adds regression coverage for distinct main and selected-agent `minimax-portal` profiles.

## Solution Assessment

Yes. The narrow maintainable solution is in the shared auth-profile runtime snapshot identity/resolution path, not a MiniMax-specific special case or credential-copying behavior.

## Repair Work Prompt

Fix the agent-specific MiniMax auth runtime snapshot collision reported at https://github.com/openclaw/openclaw/issues/64274. Current main keys active auth profile runtime snapshots by resolved auth-store paths, so when OPENCLAW_AGENT_DIR points at a selected agent, the default/main and selected-agent snapshot identities can collapse and MiniMax portal auth may resolve from the wrong store. Use https://github.com/openclaw/openclaw/pull/64320 as source context, but rebase the idea onto current main.

Expected boundary: update the auth-profile runtime snapshot identity/resolution path so default/main and explicitly requested agent snapshots remain distinct, while preserving documented read-through fallback from selected agents to the real default/main store when no local profile exists. Do not copy OAuth credentials between agents, do not broaden OAuth portability, do not change MiniMax provider ids or model catalog semantics, and do not add core special cases for MiniMax beyond regression coverage. Treat https://github.com/openclaw/openclaw/issues/44571 as adjacent context only unless the same narrow fix naturally improves it.

Add a regression that sets up distinct main and selected-agent auth stores with OPENCLAW_AGENT_DIR pointing at the selected agent and proves the selected agent's `minimax-portal` profile is used without breaking main fallback/read-through behavior. Add a single-line `CHANGELOG.md` Fixes entry with `Thanks @blaspat` if the active changelog policy requires one for this user-facing regression.
