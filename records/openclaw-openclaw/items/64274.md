## Label Justifications

- `P1`: This is a reported regression that can break a real agent's MiniMax dashboard/chat auth by selecting the wrong stored credential.
- `impact:security`: The issue concerns credential selection across main-agent and selected-agent auth scopes.
- `impact:auth-provider`: The affected behavior is provider auth profile resolution for MiniMax OAuth through the shared model auth store.

## Summary

Keep open: current main and v2026.5.18 still preserve the main-first auth-store merge/order path, and the linked closing PR is open rather than merged.

## What This Changes

The issue reports that a selected agent's MiniMax OAuth should resolve from its own `auth-profiles.json` before inherited main-agent credentials.

## Best Possible Solution

Land a focused shared auth-profile precedence fix that tries selected-agent same-provider profiles before inherited main fallback while preserving explicit order exclusions and main read-through fallback.

## Reproduction Assessment

Yes, by source inspection rather than a live MiniMax OAuth run. A main store with `minimax-portal:cli` plus a selected-agent store with `minimax-portal:default` and no selected-agent order can resolve the inherited main profile first on current main.

## Solution Assessment

Yes, the right fix boundary is shared auth-profile merge/order semantics. The issue should remain open until https://github.com/openclaw/openclaw/pull/82497 or an equivalent focused fix lands and preserves fallback plus explicit order behavior.
