## Label Justifications

- `P1`: This is a reported regression that can break a real agent's MiniMax dashboard and chat auth flow by selecting the wrong stored credential.
- `impact:security`: The issue is about cross-agent credential selection and data-scope boundaries between main-agent and selected-agent auth profiles.
- `impact:auth-provider`: The affected behavior is provider auth profile resolution for MiniMax OAuth and the shared model auth store.

## Summary

Keep open: current main and the latest release still merge inherited main-agent auth profiles ahead of selected-agent profiles, and the open closing PR has not merged yet.

## What This Changes

The issue reports that a non-main agent with local MiniMax OAuth credentials can still resolve MiniMax auth from the main agent auth store.

## Best Possible Solution

Land a focused shared auth-profile precedence fix that tries selected-agent same-provider profiles before inherited main profiles while preserving main fallback and explicit auth-order exclusions.

## Reproduction Assessment

Yes by source inspection, though not by a live MiniMax OAuth run. A main store with `minimax-portal:cli` plus a selected-agent store with `minimax-portal:default` and no explicit local order can resolve the inherited main profile first on current main.

## Solution Assessment

Yes: the narrow maintainable fix belongs in shared auth-profile merge/order semantics, not in MiniMax-specific token copying or provider special cases. The open linked PR is the right path only after its review findings are addressed.
