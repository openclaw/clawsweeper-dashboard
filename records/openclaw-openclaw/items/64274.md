## Summary

Keep open. Current main fixed the active-agent auth path collision, but the shared auth-store merge/order path can still keep inherited main MiniMax profiles ahead of the selected agent's own MiniMax profile when the profile IDs differ, which matches the remaining reported failure shape.

## What This Changes

The issue asks for MiniMax auth for an explicit agent to resolve from that agent's auth-profiles.json instead of the main agent's store.

## Best Possible Solution

Add shared auth-store precedence so a selected agent's local profile for a provider shadows inherited main profiles for that same provider while preserving main-agent fallback for providers the agent does not define.

## Reproduction Assessment

Yes, by source inspection: main `minimax-portal:minimax-cli` plus agent `minimax-portal:default` is still merged with the main profile first, and default profile ordering can select the inherited main credential. I did not run a live MiniMax OAuth flow in this read-only review.

## Solution Assessment

No for closing as implemented. The current resolver change is necessary but not sufficient; the maintainable fix belongs in shared auth-profile merge/order semantics, not in MiniMax-specific credential handling.
