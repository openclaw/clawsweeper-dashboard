## Summary

Keep this PR open: the security hardening goal is valid and current main still lacks dynamic module-load detection, but this branch’s first-character regex approach has concrete bypasses and false-positive install blockers, and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds two critical `LINE_RULES` entries in `src/security/skill-scanner.ts` to flag non-literal `require()` and `import()` calls during skill/plugin code scanning.

## Best Possible Solution

Use a tested module-load expression classifier or parser-backed scanner path that distinguishes static literals from runtime-resolved specifiers before adding critical install blockers.

## Reproduction Assessment

Yes for the proposed patch defects: `scanSource` applies each line rule to raw individual lines with `rule.pattern.exec(line)`, and the PR patterns only inspect the first relevant characters around the call. The bypasses and false positives follow directly from that source path.

## Solution Assessment

No. The security hardening direction is useful, but first-character regex line rules are not the best install-blocking policy because they miss runtime-resolved specifiers and can block static literal forms.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Detect concatenated module specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The new patterns skip calls whose first argument starts with a quote, so `require("./" + name)` and `import("./" + name)` still evade the critical rule even though the target resolves at runtime.
  - confidence: 0.9
- **[P1] Handle comments before module-load parens:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript allows comments between the callee and `(`, but these regexes only accept whitespace there, so `require/*x*/(target)` and `import/*x*/(target)` bypass the new checks.
  - confidence: 0.86
- **[P2] Skip static literal import forms:** `src/security/skill-scanner.ts:185`
  - body: The import rule flags forms like `import(/* webpackChunkName: "x" */ "./mod")` and line-broken literal imports because it only checks the first character after `(`, turning static specifiers into critical install blockers.
  - confidence: 0.84
- **[P2] Treat static template literals as static:** `src/security/skill-scanner.ts:178-185`
  - body: Because only single and double quotes are excluded, fixed template literals such as `import(`./fixed.js`)` are classified as dynamic even without interpolation.
  - confidence: 0.88
