## Summary

Keep open: current main still lacks this dynamic-load scanner coverage, but the PR’s two-regex implementation has blocking classifier gaps and no observable after-fix proof.

## What This Changes

The branch adds two critical `LINE_RULES` in `src/security/skill-scanner.ts` to report non-literal `require()` and `import()` calls.

## Best Possible Solution

Replace the two first-character regexes with a tested lightweight JS load-expression classifier that flags runtime expressions while preserving literal specifiers and the scanner’s low-dependency boundary.

## Reproduction Assessment

Yes by source inspection. Current main has no dedicated dynamic `require()` or `import()` rule, and the PR regex behavior is directly checkable against examples like `require("./" + name)`, `require/*x*/(target)`, and `import(`./fixed.js`)`; I did not run tests because this review is read-only.

## Solution Assessment

No. The current PR is a useful direction, but two line regexes are not the best fix because they miss valid dynamic syntax and can classify static literals as critical install blockers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Detect concatenated module specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The new patterns only inspect the first non-whitespace character after `(`, so `require("./" + name)` and `import("./" + name)` begin with a quote and are skipped. Those specifiers still resolve at runtime and leave the scanner gap this PR is meant to close.
  - confidence: 0.9
- **[P1] Handle comments between dynamic-load tokens:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript allows comments between the callee and `(`, but these regexes require only whitespace there. Calls like `require/*x*/(target)` or `import/*x*/(target)` bypass the new critical checks while still dynamically loading modules.
  - confidence: 0.86
- **[P2] Skip commented static literal specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The rule checks only the first character after `(`, so `import(/* webpackChunkName: "x" */ "./mod")` starts with `/` and is reported as dynamic even though the module specifier is a static literal. Because this is critical severity, that can block safe installs.
  - confidence: 0.84
- **[P2] Treat static template literals as literals:** `src/security/skill-scanner.ts:178-185`
  - body: Backtick specifiers without interpolation, such as `import(`./fixed-module`)`, are static, but the negative lookahead excludes only single and double quotes. These get reported as critical even when there is no runtime module selection.
  - confidence: 0.88
