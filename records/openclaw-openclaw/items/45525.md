## Summary

Keep open: current main does not already implement dynamic require/import scanner rules, but this PR is not merge-ready because the regex-only approach leaves bypasses and false positives in install-blocking security behavior, and contributor real behavior proof is still missing.

## What This Changes

The PR adds two critical `LINE_RULES` to `src/security/skill-scanner.ts` for `require()` and `import()` calls whose first argument does not start with a single or double quote.

## Best Possible Solution

Replace the two first-character regexes with a tested module-load expression classifier or parser-backed scanner helper, then get security-owner review and redacted real scan proof.

## Reproduction Assessment

Yes for the proposed patch defects: source inspection shows `scanSource` applies raw per-line regexes, so concatenated specifiers, comment-separated calls, commented static imports, and static template literals follow from the pattern behavior. I did not run tests because this was a read-only review.

## Solution Assessment

No. The hardening direction is useful, but first-character regexes are not the narrowest maintainable solution for an install-blocking security scanner.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Detect concatenated module specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The new patterns skip calls whose first argument starts with a quote, so `require("./" + name)` and `import("./" + name)` still evade the critical rule even though the module target is resolved at runtime.
  - confidence: 0.9
- **[P1] Handle comments before dynamic-load parens:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript permits comments between the callee and `(`, but these regexes only allow whitespace there. Calls like `require/*x*/(target)` or `import/*x*/(target)` bypass the new checks.
  - confidence: 0.86
- **[P2] Skip commented or multiline static imports:** `src/security/skill-scanner.ts:185`
  - body: The import rule checks only the first character after `(`, so `import(/* webpackChunkName: "x" */ "./mod")` and line-broken static literal imports can be reported as dynamic even though the specifier is static.
  - confidence: 0.84
- **[P2] Treat static template literals as static:** `src/security/skill-scanner.ts:178-185`
  - body: Backtick specifiers without interpolation, such as a fixed-path `import()` template literal, are constant literals, but the new patterns classify them as critical dynamic loads because only single and double quotes are excluded.
  - confidence: 0.88
