## Summary

Keep open: current main does not yet detect dynamic `require()` or `import()` in the skill scanner, so the PR targets a real security-scanner gap, but the proposed regex-only patch still has concrete false negatives, false positives, no targeted tests, and no attached real behavior proof.

## What This Changes

The PR adds two critical `LINE_RULES` to `src/security/skill-scanner.ts` for dynamic `require()` and `import()` calls.

## Best Possible Solution

Implement a tested dynamic-load classifier that distinguishes literal specifiers from runtime expressions, covers common JavaScript syntax edge cases, and preserves the scanner's low-dependency security-audit boundary.

## Reproduction Assessment

Yes by source inspection: current main has no dynamic `require()` or `import()` rule in `LINE_RULES`, so a dedicated finding for `require(target)` or `import(target)` cannot be produced today. I did not run tests because this review is read-only.

## Solution Assessment

No. Adding two first-character line regexes is not the best fix because they miss concatenated and comment-split dynamic loads while treating static template literals as critical; a small parser/tokenizer-style helper plus regression tests would be more maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Detect concatenated dynamic module specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The new rules only inspect the first non-whitespace character after `(`, so `require("./" + name)` and `import("./" + name)` start with a quote and are treated as static. Those specifiers still resolve at runtime and leave the scanner gap this PR is meant to close.
  - confidence: 0.9
- **[P1] Handle comments between dynamic-load tokens:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript allows comments between the callee and `(`, but these patterns require only whitespace there. A payload like `require/*x*/(target)` or `import/*x*/(target)` bypasses the added critical checks even though it still performs dynamic module loading.
  - confidence: 0.86
- **[P2] Avoid blocking static template literal specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The negative lookahead excludes only single and double quotes, so `import(`./fixed.js`)` and `require(`./fixed.js`)` are reported as critical even when there is no `${}` interpolation. Critical findings can block skill installation, so static template literals need to be treated as literals.
  - confidence: 0.88
