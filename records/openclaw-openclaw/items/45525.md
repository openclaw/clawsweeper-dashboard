## Summary

Keep this PR open, but do not merge the current patch. It targets a real skill-scan security gap, yet the added regex rules leave bypasses, introduce critical false positives, and lack inspectable after-fix real behavior proof.

## What This Changes

The PR adds two critical skill-scanner line rules intended to flag dynamic `require()` and `import()` calls.

## Best Possible Solution

Add tested specifier-expression classification for `require()` and dynamic `import()` that flags non-literal module targets, handles comments, allows static literals, and includes real install/audit proof.

## Reproduction Assessment

Yes. Source inspection and a direct regex probe reproduce the PR-introduced false negatives and false positives; no full install test was run because this review is read-only.

## Solution Assessment

No. The current regex additions aim at the right security gap, but checking only the first character after `(` is not the narrowest maintainable solution for module specifier safety.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Detect string-built dynamic specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The new patterns skip any call whose first argument starts with a quote, so `require("./" + name)` and `import("./" + name)` produce no finding even though the resolved module target is still runtime-controlled. Classify the first argument expression, not only its first character.
  - confidence: 0.88
- **[P1] Allow comments around dynamic load calls:** `src/security/skill-scanner.ts:178-185`
  - body: Only `\s*` is allowed between `require`/`import` and `(`, so valid evasions like `require/*x*/(name)` bypass both new critical rules. A scanner rule for hostile skill code needs to strip or accept comments here, or use a token-aware check.
  - confidence: 0.84
- **[P2] Avoid critical findings for static template literals:** `src/security/skill-scanner.ts:178-185`
  - body: Backtick literals are now treated as dynamic without checking for `${...}`, so `import(`./fixed-module`)` and the same `require` shape become critical findings even when the specifier is constant. Since critical findings block installs, static template literals need to be distinguished from interpolated ones.
  - confidence: 0.86
- **[P2] Skip comments before checking literal specifiers:** `src/security/skill-scanner.ts:185`
  - body: A static call such as `import(/* webpackChunkName: "x" */ "./mod")` starts with `/` after the parenthesis, so this rule reports a critical dynamic import even though the specifier is literal. Strip comments before deciding whether the argument is static.
  - confidence: 0.82
