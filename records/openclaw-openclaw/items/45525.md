## Summary

Keep open: current main still lacks dynamic require/import detection, but this PR's two regex rules are too brittle for an install-blocking security scanner and still need contributor-provided real behavior proof.

## What This Changes

The PR adds two critical `LINE_RULES` in `src/security/skill-scanner.ts` to flag non-literal `require()` and `import()` calls.

## Best Possible Solution

Replace the first-character regexes with a tested lightweight JavaScript load-expression classifier that flags runtime module specifier expressions while preserving static literal specifiers and the scanner's low-dependency boundary.

## Reproduction Assessment

Yes for the proposed patch defects: source inspection and a direct regex probe reproduce the misses and false positives. Current main also shows the requested dynamic-load detection is not implemented there.

## Solution Assessment

No. The hardening direction is useful, but two first-character line regexes are not the narrowest maintainable solution for an install-blocking security scanner.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Detect concatenated module specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The new patterns skip calls whose first argument starts with a quote, so `require("./" + name)` and `import("./" + name)` still evade the critical rule even though the module target is resolved at runtime.
  - confidence: 0.9
- **[P1] Handle comments between the callee and paren:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript permits comments between the callee and `(`, but these regexes only allow whitespace there. Calls like `require/*x*/(target)` or `import/*x*/(target)` bypass the new checks.
  - confidence: 0.86
- **[P2] Skip commented static literal specifiers:** `src/security/skill-scanner.ts:185`
  - body: The rule checks only the first character after `(`, so `import(/* webpackChunkName: "x" */ "./mod")` is reported as dynamic even though the specifier is a static literal.
  - confidence: 0.84
- **[P2] Treat static template literals as static specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: Backtick specifiers without interpolation, such as `import(`./fixed-module`)`, are static, but the new patterns classify them as critical dynamic loads.
  - confidence: 0.88
