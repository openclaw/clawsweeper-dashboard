## Summary

Keep open: the scanner hardening target is valid, but this branch is not merge-ready because the two raw-line regexes leave concrete bypasses, introduce critical false positives, and still lack contributor-provided real behavior proof.

## What This Changes

The PR adds two critical line rules to `src/security/skill-scanner.ts` for non-literal `require()` and `import()` calls.

## Best Possible Solution

Replace the two first-character regexes with a tested lightweight JavaScript load-expression classifier that flags runtime expressions while preserving literal specifiers and the scanner’s low-dependency boundary.

## Reproduction Assessment

Yes, by source inspection for the patch review. Current main lacks dedicated dynamic `require()`/`import()` rules, and the PR regex behavior is directly checkable against examples like `require("./" + name)`, `require/*x*/(target)`, and `import(`./fixed.js`)`.

## Solution Assessment

No. The requested scanner hardening is useful, but two raw line regexes are not the best fix because they miss valid dynamic syntax and classify some static literal specifiers as critical install blockers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Detect concatenated module specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The new patterns skip calls whose first argument starts with a quote, so `require("./" + name)` and `import("./" + name)` still evade detection even though the module target is resolved at runtime.
  - confidence: 0.9
- **[P1] Handle comments between the callee and paren:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript permits comments between the callee and `(`, but these regexes only allow whitespace there. Calls like `require/*x*/(target)` or `import/*x*/(target)` bypass the new critical checks.
  - confidence: 0.86
- **[P2] Skip commented static literal specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The rule checks only the first character after `(`, so `import(/* webpackChunkName: "x" */ "./mod")` starts with `/` and is reported as dynamic even though the specifier is a static literal.
  - confidence: 0.84
- **[P2] Treat static template literals as literals:** `src/security/skill-scanner.ts:178-185`
  - body: Backtick specifiers without interpolation, such as `import(`./fixed-module`)`, are static, but the negative lookahead excludes only single and double quotes, so they are reported as critical dynamic loads.
  - confidence: 0.88
