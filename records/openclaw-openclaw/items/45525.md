## Summary

Keep open: current main still lacks dynamic require/import scanner rules, but this PR should not merge as-is because the first-character regexes leave bypasses and false positives in an install-blocking security scanner, and contributor real behavior proof is missing.

## What This Changes

The PR adds two critical `LINE_RULES` to `src/security/skill-scanner.ts` for `require()` and `import()` calls whose first argument does not start with a single or double quote.

## Best Possible Solution

Implement a tested JavaScript module-load expression classifier that distinguishes runtime expressions from static literals before emitting critical install-blocking findings.

## Reproduction Assessment

Yes for the proposed patch defects: source inspection shows the scanner runs raw one-line regexes, so concatenated specifiers, inline comments, commented static imports, and static template literals follow directly from the pattern behavior. I did not run tests because this was a read-only review.

## Solution Assessment

No. The hardening direction is useful, but two first-character regexes are not the narrowest maintainable solution for a CODEOWNERS-protected install-blocking security scanner.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Detect concatenated module specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The new patterns skip calls whose first argument starts with a quote, so `require("./" + name)` and `import("./" + name)` still evade the critical rule even though the module target is resolved at runtime.
  - confidence: 0.9
- **[P1] Handle comments before dynamic-load parens:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript permits comments between the callee and `(`, but these regexes only allow whitespace there. Calls like `require/*x*/(target)` or `import/*x*/(target)` bypass the new checks.
  - confidence: 0.86
- **[P2] Skip commented static literal imports:** `src/security/skill-scanner.ts:185`
  - body: The import rule checks only the first character after `(`, so `import(/* webpackChunkName: "x" */ "./mod")` is reported as dynamic even though the specifier is a static literal.
  - confidence: 0.84
- **[P2] Treat static template literals as static:** `src/security/skill-scanner.ts:178-185`
  - body: Backtick specifiers without interpolation, such as `import(`./fixed-module`)`, are static, but the new patterns classify them as critical dynamic loads.
  - confidence: 0.88
