## Summary

Keep open; current main still lacks dynamic `require()`/`import()` scanner rules, but this PR's first-token regex classifier leaves verified bypasses and introduces critical false-positive install blocks in a security-sensitive scanner.

## What This Changes

The PR adds two critical `LINE_RULES` in `src/security/skill-scanner.ts` to flag non-literal `require()` and `import()` calls during skill/plugin code scanning.

## Best Possible Solution

Replace the first-token regexes with an argument-aware classifier or small tokenizer, then cover variables, concatenation, comments, static and interpolated templates, multiline calls, install blocking, and changelog coverage before merge.

## Reproduction Assessment

Yes. A read-only probe of the PR regexes reproduced missed concatenated specifiers and comment-separated calls, plus false positives for static template literals and commented literal imports.

## Solution Assessment

No. A single first-token line regex is not the best maintainable fix for an install-blocking security classifier; an argument-aware helper or tokenizer is safer and more testable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Classify the full module specifier:** `src/security/skill-scanner.ts:181-188`
  - body: The regex only checks the first non-whitespace character after `(`, so `require("./" + name)` and `import("./" + name)` avoid the critical finding even though the module target is still resolved dynamically at runtime.
  - confidence: 0.93
- **[P1] Handle comments before the call parenthesis:** `src/security/skill-scanner.ts:181-188`
  - body: JavaScript permits comments between the callee and `(`, but these patterns only allow whitespace, so `require/*x*/(name)` and the matching `import` form bypass the intended critical rule.
  - confidence: 0.9
- **[P2] Skip commented literal import specifiers:** `src/security/skill-scanner.ts:188`
  - body: A literal import with a leading comment, such as `import(/* webpackChunkName: "x" */ "./mod")`, starts with `/` after the parenthesis and is reported as critical even though the module specifier is static.
  - confidence: 0.87
- **[P2] Treat static templates as constants:** `src/security/skill-scanner.ts:181-188`
  - body: The patterns exclude only single and double quotes, so `import(`./fixed.js`)` and the matching `require` form are classified as critical even when the template contains no interpolation.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `src/security/skill-scanner.ts:181`
  - body: This user-facing scanner change affects plugin and skill install/audit behavior, but the PR only changes scanner source. Add a `CHANGELOG.md` entry before merge.
  - confidence: 0.82
