## Summary

Keep open: the security hardening direction is valid and current main still lacks dynamic `require()`/`import()` scanner rules, but this PR is not merge-ready because the added first-character regexes leave bypasses, introduce critical false positives, and still lack real behavior proof.

## What This Changes

The PR adds two critical line rules to `src/security/skill-scanner.ts` to flag non-literal `require()` and `import()` calls.

## Best Possible Solution

Replace the first-character regexes with a tested module-load expression classifier or parser-backed scanner that flags runtime-resolved specifiers while allowing static literals, then prove it through focused install-scan output.

## Reproduction Assessment

Yes for the proposed patch defects: source inspection shows `scanSource` applies each line rule regex to raw source lines, and the PR patterns only inspect the first token after the module-load paren. The bypasses and false positives are source-reproducible without running the full install flow.

## Solution Assessment

No. The hardening belongs in the scanner, but first-character regexes are too brittle for critical install-blocking policy; a tested expression classifier or parser-backed path is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Detect concatenated module specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The rules treat any quote-starting argument as static, so `require("./" + name)` and `import("./" + name)` evade the critical finding even though the target is resolved at runtime.
  - confidence: 0.9
- **[P1] Handle comments before module-load parens:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript permits comments between the callee and `(`, but these regexes only allow whitespace, so `require/*x*/(target)` and `import/*x*/(target)` bypass the added checks.
  - confidence: 0.86
- **[P2] Skip commented literal import forms:** `src/security/skill-scanner.ts:185`
  - body: `import(/* webpackChunkName: "x" */ "./mod")` is a static literal import, but this rule sees `/` as the first token after `(` and can turn benign code into a critical finding.
  - confidence: 0.84
- **[P2] Treat static template literals as static:** `src/security/skill-scanner.ts:178-185`
  - body: Fixed template literals such as `import(`./fixed.js`)` are classified as dynamic even without interpolation, which can create critical false positives for static module specifiers.
  - confidence: 0.88
