## Summary

Keep this PR open but do not merge it as-is: the dynamic module-load gap is real, current main still lacks these rules, and the proposed first-character regexes leave bypasses and false-positive install blockers while the external PR still lacks real behavior proof.

## What This Changes

The PR adds two critical `src/security/skill-scanner.ts` line rules for non-literal `require()` and `import()` calls.

## Best Possible Solution

Replace the first-character regexes with a tested module-load expression classifier, parser-backed if feasible, that distinguishes static literals from runtime-resolved specifiers and includes positive and false-positive scanner coverage.

## Reproduction Assessment

Yes for the proposed patch defects: `scanSource` runs raw per-line regex rules, and the PR patterns only check the first relevant characters around each call. The bypasses and false positives follow from that source path without needing a full install run.

## Solution Assessment

No. The hardening direction belongs in the scanner, but these first-character line regexes miss runtime-resolved specifiers and can block static literal forms; a small classifier/parser path with regression coverage is the safer solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Detect concatenated module specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The patterns skip any call whose first argument starts with a quote, so `require("./" + name)` and `import("./" + name)` still evade the critical rule even though the target resolves at runtime.
  - confidence: 0.9
- **[P1] Handle comments before module-load parens:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript permits comments between the callee and `(`, but these regexes only allow whitespace, so `require/*x*/(target)` and `import/*x*/(target)` bypass the added checks.
  - confidence: 0.86
- **[P2] Skip static literal import forms:** `src/security/skill-scanner.ts:185`
  - body: The import rule can flag static forms such as `import(/* webpackChunkName: "x" */ "./mod")` because it only checks the first character after `(`, turning literal imports into critical install blockers.
  - confidence: 0.84
- **[P2] Treat static template literals as static:** `src/security/skill-scanner.ts:178-185`
  - body: Because only single and double quotes are excluded, fixed template literals such as `import(`./fixed.js`)` are classified as dynamic even without interpolation.
  - confidence: 0.88
