## Summary

Keep open. The hardening direction is useful, but the current regex rules are too shallow for a security scanner: they miss common dynamic specifier forms, can be bypassed with comments, and also create critical false positives that would block safe skill installs.

## What This Changes

Adds two critical skill-scanner line rules intended to flag dynamic `require()` and `import()` calls with non-literal arguments.

## Best Possible Solution

Implement token-aware or parser-backed dynamic load detection that distinguishes literal specifiers from expressions, covers concatenation, template interpolation, and comment whitespace, and adds focused scanner tests for both detections and false positives.

## Reproduction Assessment

Yes, source reproducible. The PR regexes can be checked directly against representative strings, showing missed concatenated/comment-split dynamic loads and false positives for static template/commented literal imports.

## Solution Assessment

No. The requested hardening is reasonable, but the current regex-only approach is not the best mergeable solution because it misses common JavaScript syntax and would turn safe literal imports into critical install blockers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Detect expression specifiers that start with a literal:** `src/security/skill-scanner.ts:183-190`
  - body: These rules only inspect the first non-whitespace character after `(`. Calls like `require("./" + userInput)` and `import("./" + name)` are still non-literal runtime module resolution, but both start with a quote and bypass the new critical rule. Please inspect the full argument expression instead of only the first character.
  - confidence: 0.92
- **[P1] Handle comments between the callee and call:** `src/security/skill-scanner.ts:183-190`
  - body: JavaScript treats comments as whitespace, so `require/*x*/(userInput)` and `import/*x*/(userInput)` are valid calls. The new patterns require `\s*` between the callee and `(`, so a skill can evade the critical finding by inserting an inline comment. Strip/tokenize comments before matching or use a parser-aware check.
  - confidence: 0.88
- **[P2] Do not flag static literal specifiers as critical:** `src/security/skill-scanner.ts:190`
  - body: The negative lookahead excludes only single and double quotes, so static template literals such as `import(`./fixed.js`)` and literal imports with leading bundler comments are reported as critical. Since critical scanner findings block skill installs by default, this would reject safe code rather than only dynamic loading.
  - confidence: 0.9
