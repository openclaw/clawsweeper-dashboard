## Summary

Keep this PR open: the security gap is real and current main still lacks dynamic module-load detection, but this patch is not merge-ready because the two regex rules leave concrete bypasses, introduce critical false positives, and still lack contributor-provided real behavior proof.

## What This Changes

The PR adds two critical `LINE_RULES` in `src/security/skill-scanner.ts` to flag non-literal `require()` and `import()` calls.

## Best Possible Solution

Replace the two first-character regexes with a tested lightweight JS load-expression classifier that flags runtime expressions while preserving static literal specifiers and the scanner’s low-dependency boundary.

## Reproduction Assessment

Yes, by source inspection for the review findings. Current main lacks dedicated dynamic `require()`/`import()` rules, and the PR regex behavior is directly checkable against inputs like `require("./" + name)`, `require/*x*/(target)`, and `import(`./fixed.js`)`; I did not run tests because this review is read-only.

## Solution Assessment

No. The security-hardening direction is useful, but raw line regexes that only inspect the first non-whitespace character after `(` are not the narrowest maintainable fix for this scanner behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Detect concatenated module specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The new patterns only inspect the first non-whitespace character after `(`, so calls like `require("./" + name)` and `import("./" + name)` start with a quote and are skipped even though the module target is selected at runtime.
  - confidence: 0.9
- **[P1] Handle comments between dynamic-load tokens:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript allows comments between the callee and `(`, but these regexes allow only whitespace there. `require/*x*/(target)` and `import/*x*/(target)` bypass the new critical rules while still dynamically loading modules.
  - confidence: 0.87
- **[P2] Skip commented static literal specifiers:** `src/security/skill-scanner.ts:185`
  - body: The rule checks only the first character after `(`, so `import(/* webpackChunkName: "x" */ "./mod")` starts with `/` and is reported as dynamic even though the specifier is a static literal; at critical severity that can block safe installs.
  - confidence: 0.84
- **[P2] Treat static template literals as literals:** `src/security/skill-scanner.ts:178-185`
  - body: Backtick specifiers without interpolation, such as `import(`./fixed-module`)`, are static, but the negative lookahead excludes only single and double quotes. These get reported as critical even when there is no runtime module selection.
  - confidence: 0.88
