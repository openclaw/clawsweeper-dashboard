## Summary

Keep this PR open: the scanner hardening is useful, but the current patch is not merge-ready because the regexes miss valid dynamic-load syntax, add critical false positives, and still lack contributor-provided real behavior proof.

## What This Changes

The PR adds critical `dynamic-require` and `dynamic-import` line rules to `src/security/skill-scanner.ts` for non-literal module loads.

## Best Possible Solution

Replace the two first-character regexes with a tested lightweight JavaScript load-expression classifier that flags runtime module specifier expressions while preserving static literal specifiers and the scanner’s low-dependency boundary.

## Reproduction Assessment

Yes, by source inspection and a direct regex probe of the proposed patterns. The key cases are `require("./" + name)`, `require/*x*/(target)`, `import(/* comment */ "./mod")`, and ``import(`./fixed-module`)``.

## Solution Assessment

No. The hardening direction is useful, but two first-character line regexes are not the narrowest maintainable fix because critical scanner findings block installs and the syntax boundary needs tested classification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Detect concatenated module specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The new patterns skip calls whose first argument starts with a quote, so `require("./" + name)` and `import("./" + name)` still evade the critical rule even though the module target is resolved at runtime.
  - confidence: 0.9
- **[P1] Handle comments between the callee and paren:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript permits comments between the callee and `(`, but these regexes only allow whitespace there. Calls like `require/*x*/(target)` or `import/*x*/(target)` bypass the new checks.
  - confidence: 0.86
- **[P2] Skip commented static literal specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The rule checks only the first character after `(`, so `import(/* webpackChunkName: "x" */ "./mod")` is reported as dynamic even though the specifier is a static literal.
  - confidence: 0.84
- **[P2] Treat static template literals as literals:** `src/security/skill-scanner.ts:178-185`
  - body: Backtick specifiers without interpolation, such as ``import(`./fixed-module`)``, are static, but the new patterns classify them as critical dynamic loads.
  - confidence: 0.88
