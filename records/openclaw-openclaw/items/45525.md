## Summary

Keep this PR open, but do not merge it as-is. The requested dynamic module-load detection is still useful on current main, yet the proposed first-character regexes leave concrete bypasses and introduce critical false positives in a scanner whose critical findings block plugin and skill installs.

## What This Changes

The PR adds critical `dynamic-require` and `dynamic-import` `LINE_RULES` to `src/security/skill-scanner.ts` for `require()` and `import()` calls whose first argument does not start with a single or double quote.

## Best Possible Solution

Implement an argument-aware module specifier classifier for `require()` and dynamic `import()` instead of a first-character line regex, then cover variables, concatenation, comments between tokens, commented static import specifiers, static versus interpolated template literals, multiline calls, and normal literal imports in `src/security/skill-scanner.test.ts`.

## Reproduction Assessment

Yes. A read-only probe against the exact proposed regexes reproduces the missed dynamic forms and false-positive static forms, and the current scanner code confirms those regexes would be applied line-by-line.

## Solution Assessment

No. The proposed regex approach is not the best way to solve a security scanner gap because it only inspects the first token after `(`; a small tokenizer or parser-backed helper is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Classify the full module specifier:** `src/security/skill-scanner.ts:178-185`
  - body: The regex skips any call whose first argument token is a quote, so `require("./" + userInput)` and `import("./" + name)` avoid the critical finding even though the module target is resolved dynamically at runtime.
  - confidence: 0.93
- **[P1] Detect calls split by comments:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript permits comments between the callee and `(`, but these patterns only allow whitespace, so `require/*x*/(userInput)` and the matching `import` form bypass the intended critical rule.
  - confidence: 0.9
- **[P2] Skip commented literal import specifiers:** `src/security/skill-scanner.ts:185`
  - body: A literal import with a leading comment, such as `import(/* webpackChunkName: "x" */ "./mod")`, starts with `/` after the paren and is reported as critical even though the module specifier is static.
  - confidence: 0.87
- **[P2] Treat static template literals as constants:** `src/security/skill-scanner.ts:178-185`
  - body: The patterns exclude only single and double quotes, so `import(`./fixed.js`)` and the matching `require` form are classified as critical even when the template contains no interpolation.
  - confidence: 0.88
