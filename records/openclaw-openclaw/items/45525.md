## Summary

Keep open: the hardening goal is still useful and not implemented on current main, but this PR is not merge-ready because the first-character regex approach leaves concrete scanner bypasses, creates critical false positives, and still lacks after-fix real behavior proof.

## What This Changes

The PR modifies `src/security/skill-scanner.ts` by adding two critical `LINE_RULES` for non-literal `require()` and `import()` arguments.

## Best Possible Solution

Replace the two first-character regexes with a tested module-load expression classifier or parser-backed scanner path, then get security-owner review and redacted real scan proof.

## Reproduction Assessment

Yes for the proposed patch defects: `scanSource` applies each `LineRule` to raw individual lines with `rule.pattern.exec(line)`, and the PR patterns only inspect the first argument character. The bypasses and false positives follow directly from that source path.

## Solution Assessment

No. The hardening direction is useful, but critical install-blocking detection should use a classifier or parser-backed scanner with explicit coverage for variables, concatenation, comments, multiline literals, dynamic templates, and static templates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Detect concatenated module specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The new patterns skip any call whose first argument starts with a quote, so `require("./" + name)` and `import("./" + name)` still evade the critical rule even though the target is resolved at runtime.
  - confidence: 0.9
- **[P1] Handle comments before dynamic-load parens:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript allows comments between the callee and `(`, but the regex only accepts whitespace there, so `require/*x*/(target)` and `import/*x*/(target)` bypass the new checks.
  - confidence: 0.86
- **[P2] Skip static literal import forms:** `src/security/skill-scanner.ts:185`
  - body: The import rule flags `import(/* webpackChunkName: "x" */ "./mod")` and line-broken literal imports because it only checks the first character after `(`; these static specifiers can become critical install blockers.
  - confidence: 0.84
- **[P2] Treat static template literals as static:** `src/security/skill-scanner.ts:178-185`
  - body: Because only single and double quotes are excluded, fixed template literals such as `import(`./fixed.js`)` are classified as dynamic even without interpolation.
  - confidence: 0.88
