## Summary

Keep open: source review supports a real diagnostic-lifecycle bug where stale persisted installed-index diagnostics can be replayed as gateway config warnings even after current bundled plugin hint logic no longer generates this mismatch.

## What This Changes

The issue reports noisy gateway startup warnings for bundled plugin manifest IDs whose package-derived entry hints include `-plugin` or `-provider` suffixes.

## Best Possible Solution

Invalidate or filter obsolete persisted installed-index `manifest uses ... entry hints ...` diagnostics while preserving current real plugin id mismatch diagnostics and regenerating valid plugin metadata.

## Reproduction Assessment

Yes by source inspection: a persisted installed-index diagnostic with a plugin id and still-existing source path is accepted, copied into the manifest registry, and surfaced as a config warning. I did not run the gateway in this read-only sweep.

## Solution Assessment

No for manifest-id renaming; the safer fix is to stop replaying obsolete persisted entry-hint mismatch diagnostics or force a registry refresh for that stale diagnostic shape.

## Repair Work Prompt

Fix the gateway startup log-noise bug where obsolete persisted installed-plugin-index diagnostics of the form `plugin id mismatch (manifest uses ..., entry hints ...)` can be replayed for bundled plugins whose current manifests and package names are valid. Keep the fix bounded to plugin registry persistence/diagnostic handling; do not rename manifest ids or package names, and do not suppress genuine current loader/export id mismatch diagnostics. Likely approach: make `plugin-registry-snapshot` treat this obsolete persisted diagnostic shape as stale and derive a fresh registry, or filter that stale diagnostic when loading from the installed index. Add a changelog fixes entry because this changes user-visible startup diagnostics.
