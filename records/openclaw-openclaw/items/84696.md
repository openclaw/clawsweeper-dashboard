## Summary

Keep open: current main still routes workspace-only host write/edit through a lexical relative path into the pinned `@openclaw/fs-safe` write path, where the package source can reject symlinked directory parents with the reported error. The existing discussion is useful and JulyanXu’s comment already sketches the right focused pass, but there is no linked closing PR or merged fix yet.

## What This Changes

The issue requests deterministic host `write`/`edit` behavior through symlinked in-workspace directory parents while preserving workspace-boundary enforcement.

## Best Possible Solution

Canonicalize in-root symlink parents before pinned host write/edit operations, preferably by fixing or bumping `@openclaw/fs-safe`, and add OpenClaw regressions for write/edit through in-workspace symlink parents plus outside-root rejection.

## Reproduction Assessment

Yes, source-level: a workspace-only host write to a path like `memory/file.md` where `memory` is a symlinked directory still reaches fs-safe 0.2.7 with a lexical symlink parent, whose guard rejects symlinks with the reported message. I did not run the intermittent live session path.

## Solution Assessment

Yes, the best fix is a narrow canonicalization or fs-safe dependency repair with regression coverage, not a new user-facing setting. The implementation must keep existing hardlink/symlink escape protections intact.
