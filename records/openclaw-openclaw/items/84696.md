## Summary

Keep open: current main still routes the host write tool through a root-bounded fs-safe write path that uses the lexical symlink parent, matching the reported failure. The existing sandbox bridge has symlink-parent coverage, but the host write path does not.

## What This Changes

The issue reports intermittent host write-tool failures when a workspace path traverses an in-root symlinked directory.

## Best Possible Solution

Canonicalize in-root symlink parents before pinned host write operations, preferably in fs-safe with an OpenClaw dependency bump, and add host write regression coverage while preserving outside-root symlink rejection.

## Reproduction Assessment

Yes. Source inspection shows a workspace-only host write through a symlinked parent reaches fs-safe 0.2.7 with the lexical parent and can hit the exact directory-guard error, though I did not run the live intermittent session path.

## Solution Assessment

Yes, directionally. The narrow maintainable fix is to canonicalize in-root symlink parents for host writes while keeping escape rejection; an unconditional policy change or broad filesystem fallback would be riskier.

Observable bug: with tools.fs.workspaceOnly enabled, the host write tool can fail on memory/file.md when memory is an in-workspace symlinked directory, throwing "directory component must be a directory".
