## Summary

Keep open: current main still routes workspace-only host write/edit through a lexical symlink parent into `@openclaw/fs-safe` 0.2.7, whose directory guard rejects symlink parents with the reported error; open linked fix PR https://github.com/openclaw/openclaw/pull/85818 should be reviewed or landed before this issue is closed.

## What This Changes

The issue asks the host write/edit tools to follow in-workspace symlink directory parents deterministically while preserving workspace boundary checks.

## Best Possible Solution

Land one narrow fix, likely the current focused linked PR if review passes, that canonicalizes existing in-workspace symlink parent directories before fs-safe host writes while keeping outside-root, hardlink, and final-symlink rejection covered by regression tests.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: with `tools.fs.workspaceOnly=true`, a host write to `memory/file.md` where `memory` is an in-workspace symlink reaches fs-safe 0.2.7 as a lexical symlink parent and can throw the reported guard error. I did not run a live model session with context compaction.

## Solution Assessment

Yes, the narrow maintainable repair is to canonicalize only existing symlink parents that resolve inside the workspace before fs-safe root writes, with regression tests for write/edit success and escape rejection. The open linked PR should be reviewed against that boundary before this issue closes.
