## Summary

Keep open: current main and the latest shipped tag still route workspace-only host write/edit through a lexical workspace path into `@openclaw/fs-safe` 0.2.7, where symlink directory components can produce the reported error. The issue is already paired with open closing PRs, especially https://github.com/openclaw/openclaw/pull/85818, so it should stay open until one canonical fix lands.

## What This Changes

The issue asks for workspace-only host `write` to follow symlinked directory parents that resolve inside the workspace instead of rejecting them as non-directories.

## Best Possible Solution

Land one narrow fix, preferably the focused path in https://github.com/openclaw/openclaw/pull/85818 or an equivalent maintainer-approved patch, that canonicalizes in-workspace symlink parents before fs-safe pinned writes while preserving outside-workspace and final-file symlink rejection.

## Reproduction Assessment

Yes, from source inspection: a workspace-only host write to `memory/file.md` where `memory` is an in-workspace symlink reaches fs-safe 0.2.7 with lexical parent `memory`, and that dependency lstat-rejects symlink directories with the reported message. I did not live-run the intermittent cold-cache session path in this read-only review.

## Solution Assessment

Yes, the maintainable fix is a bounded canonicalization of existing symlink directory parents before host write/edit delegates to fs-safe, with tests preserving outside-workspace and final-file symlink rejection. Current main is not yet that solution; the linked focused PR is the right review target.
