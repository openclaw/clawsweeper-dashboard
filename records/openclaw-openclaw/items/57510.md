## Summary

Keep open: current main does not implement a `pre_route` hook, and this PR still contains useful work, but the patch is not mergeable as-is because it targets stale hook/reply option surfaces, weakens in-flight dedupe ordering, leaves ACP agent override behavior inconsistent, lacks real behavior proof, and needs explicit API/security review.

## What This Changes

The PR adds a blocking `pre_route` plugin hook before reply dispatch with route/session/account/agent overrides, timeout fail-open behavior, docs, changelog text, and dispatch/hook tests.

## Best Possible Solution

Define the pre-routing contract in the current hook surface, preserve in-flight dedupe, thread overrides consistently through dispatch, ACP, and reply execution, document trusted-plugin semantics, and land only after focused tests plus redacted live proof.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this PR proposes a new public plugin routing capability. Source inspection confirms current main has no `pre_route` hook, so the requested behavior cannot occur through an existing contract.

## Solution Assessment

No as-is. A blocking pre-routing hook may be reasonable, but this patch must be rebased to current hook contracts, preserve in-flight dedupe, handle ACP agent overrides consistently, and receive API/security review plus real proof.

  - confidence: 0.88

  - body: This early duplicate check uses only the completed-message cache before `pre_route` runs. Concurrent duplicate inbound messages can both execute `pre_route` and its side effects before either turn commits, so the new hook should keep the current `claimInboundDedupe` in-flight protection ahead of side effects.
  - confidence: 0.9

  - body: `pre_route` can set `routeOverride.agentId`, but ACP dispatch runs before the normal reply resolver and receives only `sessionKey`. ACP-bound turns whose session key does not encode the override agent can still resolve or run on the wrong path.
  - confidence: 0.82
