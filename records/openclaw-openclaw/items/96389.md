## Summary

Keep open: current main and the latest release still use the reported batch image fallback, and clawpdf 0.3.0 source confirms that batch image extraction spends one shared pixel budget across selected pages.

## What This Changes

The issue reports that document-extract's image fallback can render later scanned PDF pages as invalid tiny images because clawpdf batch image extraction shares a 4M pixel budget across pages.

## Best Possible Solution

Fix the included document-extract fallback so selected pages render without sharing one cross-page pixel budget, either by using clawpdf per-page rendering or by upgrading to a dependency version that fixes the batch behavior.

## Reproduction Assessment

Yes at source level: current main uses the reported batch image fallback, and clawpdf 0.3.0 source shows the shared remaining pixel budget across pages. I did not run a live multi-page scanned PDF through OpenClaw in this read-only review.

## Solution Assessment

Yes, the suggested per-page rendering direction is a narrow maintainable fix because clawpdf exposes `pdf.page(n).png(...)` and the current failure comes from the batch extractor's cross-page budget accounting.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/96389. Current `extensions/document-extract/document-extractor.ts` renders PDF image fallback pages with one batch `pdf.extract({ mode: "images", image: { maxPixels: request.maxPixels } })`, while clawpdf 0.3.0 spends that pixel budget across all selected pages. Change the included document-extract fallback so multi-page scanned PDFs render valid page images without sharing a single cross-page pixel budget, using clawpdf's supported per-page `pdf.page(pageNum).png(...)` API or a dependency upgrade if that dependency behavior is fixed. Keep the plugin boundary intact; do not add config options, do not change unrelated PDF/native-provider behavior, and do not edit CHANGELOG.md. Add or update focused tests in `extensions/document-extract/document-extractor.test.ts` that prove selected pages/maxPages are preserved and fallback image rendering does not collapse later pages because an earlier page used the pixel budget. Validate with `node scripts/run-vitest.mjs extensions/document-extract/document-extractor.test.ts` in this Codex-style checkout, or the repo-approved equivalent in a normal checkout.
