## Summary

Keep open: current main and v2026.6.10 still use the reported batch `clawpdf` image fallback, while https://github.com/openclaw/openclaw/pull/96390 is an open closing fix candidate and should be reviewed or replaced before this issue closes.

## What This Changes

The issue reports that scanned multi-page PDFs in the bundled document-extract fallback can render later pages as 1x1 images because one batch `clawpdf` image extraction shares the pixel budget across pages.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96390 or an equivalent bundled document-extract fix that resets the render budget per selected page and updates the focused regression test.

## Reproduction Assessment

Yes at source level: current main uses the reported batch `pdf.extract({ mode: "images" })` fallback, and `clawpdf` 0.3.0 spends one shared `remainingPixels` budget across pages. I did not run a live scanned PDF through OpenClaw during this read-only review.

## Solution Assessment

Yes, the per-page rendering direction is the narrowest maintainable fix: it keeps the owner boundary in the bundled document-extract plugin and avoids reimplementing `clawpdf`'s render planning. An upstream `clawpdf` release that changes batch budget semantics would also be acceptable if paired with the same OpenClaw regression coverage.
