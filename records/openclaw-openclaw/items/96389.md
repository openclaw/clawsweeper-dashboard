## Summary

Keep open: current main and the latest release still batch-render fallback PDF images through clawpdf 0.3.0, whose shared pixel budget can collapse later pages; no open PR owns the fix.

## What This Changes

Fix document-extract's scanned-PDF image fallback so multi-page renders do not share one clawpdf pixel budget and produce invalid 1x1 images.

## Best Possible Solution

Render PDF fallback images per selected page through clawpdf's page API or update clawpdf to per-page budget semantics, preserving current limits and adding regression coverage.

## Reproduction Assessment

Yes, at source level: current main uses the reported batch image fallback, and clawpdf 0.3.0 source shows one shared remaining pixel budget across pages. I did not run a live scanned PDF through OpenClaw in this read-only review.

## Solution Assessment

Yes. The per-page rendering direction is the narrowest maintainable fix because clawpdf already exposes `pdf.page(n).png(...)` and the current failure comes from the batch extractor's cross-page budget accounting.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/96389. In the bundled document-extract plugin, the low/no-text PDF fallback currently calls clawpdf `pdf.extract({ mode: "images" })` once for all selected pages, so clawpdf 0.3.0 spends one shared `maxPixels` budget and can collapse later scanned pages to 1x1 images. Change the fallback to render selected pages independently, preferably via `pdf.page(pageNumber).png(...)`, while preserving existing page selection, maxPages, password handling, forms rendering, max dimension/pixel intent, result shape, and error callback behavior. Do not add config flags or change provider routing. Update focused regression tests so multi-page image fallback no longer depends on a single batch image extraction call; update PDF docs only if the pixel-budget wording becomes per-page. Validate with the focused document-extract/media tests and git diff checks.
