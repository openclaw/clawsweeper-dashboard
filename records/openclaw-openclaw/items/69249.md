## Label Justifications

- `P1`: A gateway restart can silently interrupt an active Discord user turn and leave the user without recovery feedback.
- `impact:message-loss`: The reported failure mode drops the expected Discord reply or recovery notice from the channel.
- `impact:session-state`: The issue centers on restart-interrupted in-flight turn state and whether recovery is visible to the original session.

## Summary

Keep open: current main has more restart-recovery machinery than the originally reported version, but the remaining Discord-visible gap is still present. Marked restart-aborted main sessions are resumed internally with `deliver: false`, and the Discord abort path still returns without posting a same-channel recovery notice.

## What This Changes

The issue asks OpenClaw to resume or visibly notify the original Discord channel when a gateway restart interrupts an in-flight Discord-backed turn.

## Best Possible Solution

Add a generic, idempotent post-restart interruption notice for externally routable main sessions using persisted delivery context or last-route metadata, while preserving hidden resume for internal or non-routable sessions.

## Reproduction Assessment

Yes, at source level: a stale lock can mark a running main session for recovery, recovery calls the agent with `deliver: false`, and the Discord abort branch returns without a channel notice. I did not run a live Discord restart E2E in this read-only review.

## Solution Assessment

Yes, if solved channel-agnostically. The maintainable fix is to reuse persisted delivery context or last-route metadata for a same-channel recovery notice, not to add a Discord-only workaround.
