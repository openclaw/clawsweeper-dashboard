## Label Justifications

- `P1`: A gateway restart can silently interrupt an active Discord user turn and leave the channel without any recovery feedback.
- `impact:message-loss`: The reported failure drops the expected Discord reply or recovery notice from the user-visible channel.
- `impact:session-state`: The failure depends on restart-interrupted main-session state and whether recovery reaches the original session route.

## Summary

Keep open: current main and v2026.5.20 still recover restart-aborted main sessions through a hidden `deliver: false` turn, so the recovered answer can land only in the transcript while the original Discord channel sees no reply or recovery notice. The latest user comment reports the same failure mode on 2026.5.19 and confirms that visible delivery made the local hot patch reach the Discord DM.

## What This Changes

The issue asks OpenClaw to avoid silent Discord ghosting when a gateway restart interrupts an in-flight turn by either safely resuming the turn or posting a visible recovery notice.

## Best Possible Solution

Add a channel-agnostic, idempotent restart-interruption delivery path that uses persisted route or delivery context for externally routable main sessions, while preserving session-only hidden recovery for internal and non-routable sessions.

## Reproduction Assessment

Yes, at source level: a restart-aborted Discord main session is recovered through `deliver: false`, and the shared delivery path skips durable channel send when delivery is disabled. I did not run a live Discord restart E2E.

## Solution Assessment

Yes, if solved narrowly and channel-agnostically. The maintainable repair should use persisted route or delivery context for an idempotent interruption notice or recovered final delivery, not blindly enable delivery for every hidden recovery turn.

## Repair Work Prompt

Fix the source-reproducible Discord-visible gateway restart recovery bug without broad restart-manifest or config redesign. In current main, `src/agents/main-session-restart-recovery.ts` resumes restart-aborted main sessions with `deliver: false`, so recovered output can be written only to the transcript; Discord inbound code also returns on abort without a same-channel notice. Implement a narrow, channel-agnostic, idempotent path for externally routable main sessions: use persisted `deliveryContext`/last-route metadata or pending final delivery context to either deliver the recovered final answer best-effort or post a concise interruption/recovery notice to the original route. Preserve hidden `deliver: false` behavior for internal, route-less, subagent, cron, and non-routable sessions; do not add a new config option or plugin API; do not blindly set `deliver: true` for every recovery. Add regression coverage for a Discord-backed main session with stored route metadata proving visible delivery/notice is requested, for internal sessions proving they remain session-only, and for abort cleanup if the fix touches the Discord handler. Add a user-facing changelog entry because this fixes visible channel behavior.
