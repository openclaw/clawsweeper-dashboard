## Label Justifications

- `P1`: A gateway restart can silently interrupt an active Discord user turn and leave the original channel without a reply or recovery feedback.
- `impact:message-loss`: The reported failure drops the expected Discord final reply or recovery notice from the user-visible channel.
- `impact:session-state`: The failure depends on restart-interrupted in-flight turn state and whether recovery reaches the original session route.

## Summary

Keep open: current main and latest release still resume restart-aborted main sessions through a hidden `deliver: false` recovery turn, while Discord abort handling exits without a same-channel recovery notice. The new 2026.5.19 user report strengthens the source-level evidence that recovery can write the answer to the transcript without delivering it to Discord.

## What This Changes

The issue requests automatic recovery or a visible same-channel Discord notice when a gateway restart interrupts an in-flight turn.

## Best Possible Solution

Add a channel-agnostic, idempotent post-restart interruption notice or safe best-effort delivery resume for externally routable main sessions using persisted route or delivery context, while preserving hidden recovery for internal or non-routable sessions.

## Reproduction Assessment

Yes, at source level: a restart-aborted Discord main session is recovered through `deliver: false`, the shared delivery path skips durable channel send in that mode, and the Discord abort branch returns without a notice. I did not run a live Discord restart E2E.

## Solution Assessment

Yes, if solved at the shared recovery/delivery boundary. The maintainable path is to reuse persisted delivery or last-route metadata for an idempotent interruption notice or best-effort visible resume, not add a Discord-only fallback branch.
