## Summary

Keep open. Current main still has the channel-visible recovery gap: Discord records enough route metadata to know where the interrupted turn came from, but the abort path exits without posting a notice, the main-session restart recovery path resumes internally with `deliver: false`, and restart-sentinel outbound notices only run when a sentinel file already exists. Related Feishu and broader restart-recovery reports support a generic recovery-notice path rather than closing this as Discord-only or already implemented.

## What This Changes

A Discord-backed gateway restart should either safely resume an interrupted in-flight turn or post a user-visible recovery message to the original channel.

## Best Possible Solution

Implement a channel-agnostic post-restart interruption notice path: when startup identifies a restart-aborted or recently abandoned channel-originated main session, resolve the persisted delivery context or last route and enqueue one safe same-channel recovery notice. Keep actual automatic resume limited to transcript tails that are explicitly resumable, and reuse restart-sentinel/outbound delivery machinery where practical.

## Reproduction Assessment

Yes. The source-level path is high confidence: Discord records route metadata, abort handling returns before any recovery delivery, main-session recovery uses `deliver: false`, and sentinel notices require a sentinel file. No live Discord restart was run because this review was read-only.

## Solution Assessment

Yes. A generic channel-visible recovery notice is the narrow maintainable fix; a Discord-only branch would miss the Feishu cluster, while hidden synthetic replay for channel-backed sessions is riskier than an explicit recovery notice.
