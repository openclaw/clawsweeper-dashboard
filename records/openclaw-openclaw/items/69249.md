## Summary

Keep open: current main still has the Discord-visible restart interruption gap. Restart-aborted main sessions are marked and retried internally, but the recovery turn uses hidden delivery and the Discord abort path still exits without posting a same-channel recovery notice.

## What This Changes

Gateway restart should automatically resume or visibly notify the original Discord channel when an in-flight turn is interrupted.

## Best Possible Solution

Add one generic, idempotent post-restart interruption notice for externally delivered main sessions using persisted delivery context, while preserving hidden automatic resume only for safe local/internal sessions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: interrupt a Discord-backed main turn during gateway restart, startup marks the stale locked session for recovery, recovery runs with deliver: false, and the Discord abort branch returns without a channel notice.

## Solution Assessment

Yes, if implemented channel-agnostically. The narrow maintainable fix is to reuse persisted delivery context or last-route metadata for an idempotent recovery notice, not to add a Discord-only workaround or keep the current hidden resume as the only recovery path.
