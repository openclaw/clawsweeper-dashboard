## Label Justifications

- `P1`: A gateway restart can silently interrupt an active Discord user turn and leave the channel without recovery feedback.
- `impact:message-loss`: The reported failure drops the expected Discord reply or recovery notice from the user-visible channel.
- `impact:session-state`: The failure depends on restart-interrupted in-flight turn state and whether recovery reaches the original session.

## Summary

Keep open: current main and the latest stable tag still run restart-aborted main-session recovery as a hidden `deliver: false` agent turn, while the Discord abort branch exits without a same-channel recovery notice. Related broad restart-recovery issues and dashboard PRs do not ship this focused Discord-visible fix.

## What This Changes

Request automatic same-channel recovery or resume feedback when a Discord-backed turn is interrupted by a gateway restart.

## Best Possible Solution

Add a generic, idempotent post-restart interruption notice for externally routable main sessions using persisted delivery or last-route context, while preserving hidden resume only for internal or non-routable sessions.

## Reproduction Assessment

Yes, at source level: a marked restart-aborted main session is resumed with `deliver: false`, the shared delivery path returns before channel send, and the Discord abort branch returns without a notice. I did not run a live Discord restart E2E.

## Solution Assessment

Yes, if solved channel-agnostically. The maintainable fix should reuse persisted delivery or last-route metadata for a same-channel interruption notice rather than adding a Discord-only workaround.
