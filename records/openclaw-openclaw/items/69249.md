## Label Justifications

- `P1`: A gateway restart can silently interrupt an active Discord user turn and leave the channel without recovery feedback.
- `impact:message-loss`: The reported failure drops the expected Discord reply or recovery notice from the user-visible channel.
- `impact:session-state`: The failure depends on restart-interrupted in-flight turn state and whether recovery reaches the original session route.

## Summary

Keep open: current main still recovers restart-aborted main sessions through a hidden `deliver: false` turn, and the Discord abort branch exits without posting a same-channel recovery notice. The related open PRs cover adjacent restart or dashboard behavior but do not yet ship this focused Discord-visible recovery path.

## What This Changes

The issue asks OpenClaw to resume or visibly notify the original Discord channel when a gateway restart interrupts an in-flight turn.

## Best Possible Solution

Add a generic, idempotent post-restart interruption notice for externally routable main sessions using persisted delivery or last-route context, while preserving hidden resume only for internal or non-routable sessions.

## Reproduction Assessment

Yes, at source level: a marked restart-aborted main session is recovered through a hidden `deliver: false` gateway call, and the Discord abort branch returns before any same-channel recovery send. I did not run a live Discord restart E2E in this read-only review.

## Solution Assessment

Yes, if solved channel-agnostically. The maintainable fix should reuse persisted delivery or last-route metadata for an idempotent interruption notice rather than adding a Discord-only fallback branch.
