## Summary

Keep open: current main still has the reported channel-visible recovery gap for Discord-backed turns interrupted by a gateway restart, and the related open restart-recovery work is not merged proof that users get a same-channel notice.

## What This Changes

The issue asks for automatic resume or a same-channel recovery notice when a gateway restart aborts an in-flight Discord-backed turn.

## Best Possible Solution

Implement one generic, deduplicated post-restart interruption notice for channel-originated main sessions using persisted delivery context or last-route metadata, while keeping automatic resume limited to explicitly safe non-channel cases.

## Reproduction Assessment

Yes. The source-level path is high confidence: Discord records route metadata, abort handling returns silently, main-session recovery uses `deliver: false`, and sentinel notices require a preexisting sentinel.

## Solution Assessment

Yes for the requested direction. A channel-agnostic recovery notice is narrower and safer than hidden replay for external channels, while a Discord-only branch would miss the Feishu-related cluster.
