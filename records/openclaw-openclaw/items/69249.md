## Summary

Keep open: current main still has the Discord-visible restart interruption gap. Startup can mark and internally resume restart-aborted main sessions, but the recovery path is still hidden delivery and the Discord abort branch still exits without a same-channel notice.

## What This Changes

The issue asks OpenClaw to recover or visibly notify users when a gateway restart interrupts an in-flight Discord-backed turn.

## Best Possible Solution

Add a generic, idempotent post-restart interruption notice for routable externally delivered main sessions using persisted delivery context or last-route metadata, while preserving hidden resume for internal/non-routable sessions.

## Reproduction Assessment

Yes, source-level: interrupting a Discord-backed main turn can leave a stale lock that startup marks for recovery, and current recovery resumes with `deliver: false` while Discord’s abort path returns without a channel notice.

## Solution Assessment

Yes, if implemented channel-agnostically. The narrow maintainable fix is to reuse persisted delivery context or route metadata for an idempotent recovery notice, not to add a Discord-only workaround.
