## Summary

Keep open: current main still has the Discord-visible recovery gap after gateway restart interruption, and the related restart-recovery PRs are open, unmerged, or aimed at adjacent behavior rather than proof of a same-channel recovery notice.

## What This Changes

Requests that Discord-backed in-flight turns interrupted by a gateway restart either resume safely or post a user-visible recovery message in the original channel.

## Best Possible Solution

Implement one generic, deduplicated post-restart interruption notice for external channel-originated main sessions using persisted delivery context or last-route metadata, while keeping automatic hidden resume limited to safe local/internal sessions.

## Reproduction Assessment

Yes. The source path is high-confidence: Discord persists route metadata, the Discord abort path returns silently, and current main-session restart recovery uses `deliver: false` instead of notifying the channel.

## Solution Assessment

Yes for the requested direction. A channel-agnostic recovery notice is narrower than hidden replay and safer than a Discord-only branch because the same symptom is already clustered with Feishu and restart-sentinel routing work.
