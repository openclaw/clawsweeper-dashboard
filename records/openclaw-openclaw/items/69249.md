## Label Justifications

- `P1`: A gateway restart can silently interrupt an active Discord user turn and leave the channel without recovery feedback.
- `impact:message-loss`: The reported failure drops the expected Discord reply or recovery notice from the user-visible channel.
- `impact:session-state`: The failure depends on restart-interrupted in-flight turn state and whether recovery reaches the original session route.

## Summary

Keep open: current main and v2026.5.18 still recover restart-aborted main sessions through a hidden `deliver: false` turn, while the Discord abort path exits without a same-channel recovery notice. Related restart-recovery PRs are open and do not yet close this focused Discord-visible gap.

## What This Changes

Add automatic resume or a same-channel recovery notice when a gateway restart aborts an in-flight Discord turn.

## Best Possible Solution

Add a channel-agnostic, idempotent post-restart interruption notice for externally routable main sessions using persisted route or delivery context, while preserving hidden resume for internal or non-routable sessions.

## Reproduction Assessment

Yes, at source level: a restart-aborted Discord main session is resumed with `deliver: false`, the shared delivery path skips durable channel send, and the Discord abort branch returns without a notice. I did not run a live Discord restart E2E.

## Solution Assessment

Yes, if solved channel-agnostically. The maintainable fix should reuse persisted delivery or last-route metadata for an idempotent interruption notice rather than adding a Discord-only fallback branch.
