## Label Justifications

- `P1`: A gateway restart can silently interrupt an active Discord turn and leave the user-visible channel without the final reply or recovery feedback.
- `impact:message-loss`: The reported failure drops the expected Discord reply or recovery notice from the channel.
- `impact:session-state`: The failure depends on restart-interrupted in-flight session state and whether recovery reaches the original route.

## Summary

Keep open: current main still resumes restart-aborted main sessions through a hidden `deliver: false` gateway call, so recovered Discord output can be written to the transcript without reaching the channel. The latest user comment reports the same shape on OpenClaw 2026.5.19, and the related open PRs do not yet close this focused Discord-visible recovery gap.

## What This Changes

The issue asks OpenClaw to resume or visibly notify users when a gateway restart interrupts an in-flight Discord-backed turn.

## Best Possible Solution

Add a channel-agnostic, idempotent post-restart recovery path that reuses persisted delivery or last-route metadata to either deliver the recovered final answer best-effort or post a visible interruption notice, while preserving hidden recovery for internal/non-routable sessions.

## Reproduction Assessment

Yes, at source level: a restart-aborted Discord main session can be marked for recovery, then `resumeMainSession` calls `agent` with `deliver: false`, and the delivery path skips channel send. I did not run a live Discord restart E2E in this read-only review.

## Solution Assessment

Yes, if solved through the existing recovery and delivery contracts rather than a Discord-only special case. The safer repair is to reuse persisted delivery/last-route metadata and keep internal sessions hidden unless a visible route exists.
