- `impact:message-loss`: The reported failure drops the expected Discord reply or recovery notice from the user-visible channel.

## Summary

Keep open: current main and the latest shipped release still resume restart-aborted main sessions through a hidden `deliver: false` agent turn, while Discord has persisted route metadata that could support a visible recovery path. The newer user comment matches the current source: recovery can write the answer into the session without delivering it back to the Discord DM.

## What This Changes

The issue requests automatic recovery or a same-channel user-visible notice when a gateway restart interrupts an in-flight Discord-backed turn.

## Best Possible Solution

Add a channel-agnostic, idempotent post-restart recovery delivery or interruption notice for externally routable main sessions using persisted route or delivery context, while keeping hidden resume for internal or non-routable sessions.

## Reproduction Assessment

Yes, at source level: restart-aborted main-session recovery calls `agent` with `deliver: false`, and the shared delivery path returns before durable channel send when delivery is disabled. I did not run a live Discord restart E2E.

## Solution Assessment

Yes, the maintainable repair should be channel-agnostic and route through persisted delivery metadata rather than adding a Discord-only apology branch.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/69249. The observable bug is that restart-aborted Discord-backed main sessions can recover into the transcript without delivering a visible reply or recovery notice to the original channel because `src/agents/main-session-restart-recovery.ts` calls `agent` with `deliver: false`. Implement a bounded, channel-agnostic repair: for externally routable main sessions, reuse persisted `deliveryContext` or last-route metadata to either deliver the recovered final reply best-effort or post an idempotent interruption/recovery notice; preserve hidden `deliver: false` behavior for internal/non-routable sessions. Coordinate with the policy direction in https://github.com/openclaw/openclaw/pull/74687 and do not add a Discord-only fallback if a generic delivery-context path is available. Add focused regression coverage in restart recovery and delivery/Discord route tests, and add a user-facing changelog fix entry unless maintainers direct otherwise.
