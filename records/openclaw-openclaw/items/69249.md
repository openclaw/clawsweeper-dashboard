## Summary

Keep open: current main still has the Discord-visible restart interruption gap. Restart-aborted main sessions are marked and resumed internally, but the recovery turn still uses hidden delivery and the Discord abort path still returns without a same-channel recovery notice.

## What This Changes

The issue requests automatic user-visible recovery when a gateway restart aborts an in-flight Discord-backed turn.

## Best Possible Solution

Add a generic, idempotent post-restart interruption notice for externally delivered main sessions using persisted delivery context or last-route metadata, while preserving hidden automatic resume only for safe internal sessions.

## Reproduction Assessment

Yes, source-level. Interrupting a Discord-backed main turn during gateway restart can leave a stale lock that startup marks for recovery, and current recovery resumes with `deliver: false` while Discord’s abort branch returns without a channel notice.

## Solution Assessment

Yes, if implemented channel-agnostically. The narrow maintainable fix is to reuse persisted delivery context or route metadata for an idempotent recovery notice, not to add a Discord-only workaround.
