## Label Justifications

- `P1`: A gateway restart can silently interrupt an active Discord user turn and leave the channel without recovery feedback.
- `impact:message-loss`: The reported failure drops the expected Discord reply or recovery notice from the user-visible channel.
- `impact:session-state`: The failure depends on restart-interrupted in-flight main-session state and whether recovery reaches the original route.

## Summary

Keep open: current main and v2026.5.20 still resume restart-aborted main sessions through a hidden `deliver: false` turn, so a recovered Discord reply can be written to the transcript without reaching the original channel. The newer user comment matches the current source, and the related open PRs do not close this focused Discord-visible recovery gap.

## What This Changes

The issue asks OpenClaw to send or safely resume a user-visible Discord recovery response when a gateway restart interrupts an in-flight turn.

## Best Possible Solution

Add an idempotent, channel-agnostic recovery notice or best-effort final delivery for externally routable main sessions using persisted delivery or last-route context, while keeping hidden resume for internal or non-routable sessions.

## Reproduction Assessment

Yes, at source level: restart recovery calls `agent` with `deliver: false`, the delivery layer returns before channel send, and the Discord abort branch exits without a visible notice. I did not run a live Discord restart E2E.

## Solution Assessment

Yes, if solved generically. The maintainable path is to reuse persisted delivery or last-route metadata for an idempotent interruption notice or best-effort final delivery rather than adding a Discord-only fallback branch.
