## Summary

Keep open: current main still has the Discord-visible restart interruption gap. The code persists enough Discord route metadata to notify the original channel, but the abort path returns silently and the main-session restart recovery path still resumes with hidden `deliver: false` instead of posting a same-channel recovery notice.

## What This Changes

Request that gateway restarts interrupting Discord-backed turns either safely resume or post a same-channel recovery message instead of leaving the user with silence.

## Best Possible Solution

Implement one generic, idempotent post-restart interruption notice for external channel-originated main sessions using persisted delivery context or last-route metadata, while keeping hidden automatic resume limited to safe local/internal sessions.

## Reproduction Assessment

Yes, source-level with high confidence. Discord persists the route, the Discord abort branch returns silently, and current restart recovery resumes with `deliver: false` instead of notifying the originating channel.

## Solution Assessment

Yes for the requested direction. A channel-agnostic recovery notice is narrower than a Discord-only fallback and safer than hidden replay for channel-backed sessions, but it should be reconciled with https://github.com/openclaw/openclaw/pull/74687.
