- `impact:message-loss`: The reported failure drops the expected Discord reply or recovery notice from the user-visible channel.
- `impact:session-state`: The failure depends on restart-interrupted in-flight turn state and whether recovery reaches the original session route.

## Summary

Keep open: current main still has the reported source-level failure shape, with restart-aborted main-session recovery using a hidden `deliver: false` agent turn and the Discord abort branch returning without a same-channel recovery notice. Related closed Feishu/Discord reports were routed to this issue, while the open dashboard and channel-recovery PRs do not yet ship this focused Discord-visible recovery behavior.

## What This Changes

The issue requests automatic visible recovery when a Discord-backed turn is interrupted by a gateway restart.

## Best Possible Solution

Add a generic, idempotent post-restart interruption notice for externally routable main sessions using persisted delivery or last-route context, while preserving hidden resume only for internal or non-routable sessions.

## Reproduction Assessment

Yes, at source level: a marked restart-aborted main session is recovered through a hidden `deliver: false` gateway call, and the shared delivery path returns before any channel send. I did not run a live Discord restart E2E in this read-only review.

## Solution Assessment

Yes, if solved channel-agnostically. The maintainable fix should reuse persisted delivery or last-route metadata for an idempotent same-channel interruption notice rather than adding a Discord-only workaround.
