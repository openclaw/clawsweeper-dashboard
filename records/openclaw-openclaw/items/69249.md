## Label Justifications

- `P1`: A gateway restart can silently abort an active Discord-backed turn and leave the user with no channel feedback until they manually ping again.
- `impact:message-loss`: The reported failure is a lost or suppressed user-visible Discord reply/recovery message after an in-flight turn aborts.
- `impact:session-state`: The remaining gap is in restart-aborted main-session state recovery and route metadata reuse after gateway startup.

## Summary

Keep open: current main still has the Discord-visible restart interruption gap. Startup can mark and internally resume restart-aborted main sessions, but that recovery still runs with hidden delivery while the Discord abort path exits without a same-channel recovery notice.

## What This Changes

Request a user-visible same-channel recovery or resume notice when a Discord-backed main turn is interrupted by gateway restart.

## Best Possible Solution

Implement an idempotent, channel-agnostic restart-interruption notice for routable externally delivered main sessions using persisted delivery context or last-route metadata, while preserving hidden resume for internal or non-routable sessions.

## Reproduction Assessment

Yes, source-level. A Discord-backed main turn can be marked for restart recovery, current recovery resumes with `deliver: false`, and the Discord abort branch returns without a same-channel notice; I did not run a live Discord restart in this read-only review.

## Solution Assessment

Yes. The maintainable fix is channel-agnostic and should reuse persisted delivery context or last-route metadata for an idempotent notice, not add a Discord-only workaround.
