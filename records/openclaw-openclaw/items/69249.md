## Summary

Keep open: current main still has the Discord-visible restart interruption gap. Restart-aborted main sessions are detected, but the built-in recovery path resumes internally with `deliver: false`, and the Discord abort path returns after cleanup without posting a channel recovery message.

## What This Changes

The issue requests that interrupted Discord-backed turns after a gateway restart either resume safely or produce a visible recovery message in the original channel.

## Best Possible Solution

Implement one generic, idempotent post-restart interruption notice for external channel-originated main sessions using persisted delivery context or last-route metadata, while keeping hidden automatic resume limited to safe local/internal sessions.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: interrupt a Discord-backed main turn during gateway restart, then startup marks the session and schedules recovery that runs with `deliver: false` while the Discord abort branch returns silently.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if implemented channel-agnostically. A Discord-only fallback or the current open PR's manual-failure behavior would still leave users without an automatic same-channel recovery notice.
