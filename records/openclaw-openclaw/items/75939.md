## Summary

Close as implemented on current main. The LM Studio JIT/TTL problem is now handled by the maintainer-landed `models.providers.lmstudio.params.preload: false` opt-out, with docs, changelog, and a focused regression test, so the stale `sendLoadApi` PR should not be merged as a second config surface.

## What This Changes

This PR adds a top-level LM Studio `sendLoadApi` provider option and uses it to skip native `/api/v1/models/load` calls during inference preload and embedding-provider warmup.

## Best Possible Solution

Keep the current-main `params.preload` contract as the single LM Studio native-preload opt-out and close this older PR to avoid adding a second user-facing knob.

## Reproduction Assessment

Yes. The pre-fix path was deterministic because LM Studio inference called `ensureLmstudioModelLoaded`; current main has a focused test showing `params.preload: false` skips that call.

## Solution Assessment

Yes. Reusing provider `params` is narrower than adding a new top-level config key, and current main already includes the docs, changelog, and test coverage this stale branch lacked.
