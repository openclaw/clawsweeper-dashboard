## Summary

This should stay open: current main and the latest release still have the reported ClickClack token resolver behavior, where configured non-env SecretRefs produce an empty token and make the channel look unconfigured at gateway/runtime use sites.

## What This Changes

The issue asks OpenClaw to make ClickClack token SecretRefs from env, file, and exec providers behave consistently across schema validation, secrets audit, gateway startup, status, and outbound delivery.

## Best Possible Solution

Add ClickClack-owned SecretRef runtime support for top-level and account token fields, following sibling channel secret-contract patterns, and update gateway/outbound/account tests so resolved env, file, and exec refs configure the channel without exposing token values.

## Reproduction Assessment

Yes, source inspection gives a high-confidence repro path: configure a ClickClack token as a resolvable exec SecretRef, then gateway/status uses resolveClickClackAccount, which returns an empty token for non-env configured refs. I did not run the live gateway repro because this review is read-only.

## Solution Assessment

Yes, the best fix is in the ClickClack plugin's token resolution and channel secret contract, not a new config option or wrapper workaround. The durable shape should reuse the existing SecretRef runtime materialization/contract pattern used by sibling channels.
