## Summary

Keep open: current main and v2026.6.6 still expose only `screen_record` as the file-backed screen media action, while `screen.snapshot` remains a default-allowed raw invoke path. The linked workaround plugin is useful evidence that the request is practical, but the existing core nodes media tool is the right place to add parity with `screen_record`.

## What This Changes

The issue requests a first-class `screen_snapshot` nodes action that saves `screen.snapshot` image output to a file/media reference instead of returning inline base64.

## Best Possible Solution

Add a dedicated file-backed `screen_snapshot` action in the core nodes tool and CLI/docs, while preserving direct `screen.snapshot` invoke behavior unless maintainers explicitly choose a breaking policy change.

## Reproduction Assessment

Yes, source-reproducible: current main has `screen_record` only, the media redirect map lacks `screen.snapshot`, and the gateway policy still allows raw `screen.snapshot` by default. I did not run a live paired-node/channel flow in this read-only review.

## Solution Assessment

Yes for the action-only part: mirroring `screen_record` in the nodes media tool is the narrow owner-boundary fix. The optional direct-invoke block is not the best default without maintainer approval because it would change documented behavior.
