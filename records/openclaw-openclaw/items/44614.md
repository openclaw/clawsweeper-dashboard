## Summary

Keep open. Current main still leaves the Windows Gateway/Control UI update path running the package replacement from inside the live Gateway process, so the user-facing EBUSY problem is not implemented away; however, this PR is not safe to merge as-is because it introduces blocking Windows process-lifecycle and command-execution regressions.

## What This Changes

The PR adds a Windows-only detached `.cmd` package-update helper, wires `runGatewayUpdate` to delegate win32 global updates to it, and makes `update.run` respond before exiting the Gateway.

## Best Possible Solution

Keep this PR open for maintainer review or a revised implementation. The desired end state is a current-main Windows `update.run` package-update flow that avoids replacing the live package tree while the Gateway is running, preserves the existing package-update hardening and restart/status contracts, gates hard exits on a verified restart path, escapes command execution safely, and reconciles detached results on startup before reporting success.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists from the provided #40540 Windows EBUSY logs plus current-main source: invoke `update.run` for a Windows global npm install from the live Gateway/Control UI, where the handler calls the shared package-update steps and those steps still rename the live package tree.

## Solution Assessment

No. The detached-helper direction is plausible, but this patch is not the best current solution because it bypasses current package-update safeguards, reports success before the install completes, loses prepared environment handling, relies on unsafely built cmd strings, and can hard-exit without proving a supervised restart path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Default the detached exit delay before clamping:** `src/gateway/server-methods/update.ts:119`
  - body: `restartDelayMs` is optional, so `Math.max(restartDelayMs, 2000)` becomes `NaN` when callers use the default parameters. Node treats that as an immediate timer, which can exit the Gateway before the `update.run` response flushes.
  - confidence: 0.95
- **[P1] Require restart supervision before exiting:** `src/gateway/server-methods/update.ts:115-117`
  - body: This path hard-exits the Gateway after delegation, but the helper can only relaunch through `schtasks`. In an unsupervised Windows Gateway run, the API reports success and then leaves the Gateway down instead of using the existing restart fallback behavior.
  - confidence: 0.88
- **[P1] Pass the prepared install environment to the helper:** `src/infra/update-detached-win32.ts:137-141`
  - body: `spawnDetachedUpdate` accepts the `globalInstallEnv`, but the `spawn()` options omit `env`, so the `.cmd` helper runs with `process.env` instead of the Windows npm settings and portable-Git PATH prepared for package updates.
  - confidence: 0.92
- **[P1] Escape install argv before embedding it in cmd:** `src/infra/update-detached-win32.ts:126-128`
  - body: The helper joins argv into a batch command by quoting only arguments with spaces. Cmd metacharacters such as `&`, `|`, `%`, `!`, and `^` in a package spec or path will be interpreted by the shell, unlike the previous argv-based execution.
  - confidence: 0.9
- **[P2] Preserve npm's omit-optional retry:** `src/infra/update-runner.ts:896-900`
  - body: The Windows branch delegates only the primary `globalInstallArgs(...)` command and returns before the existing fallback path can run. Hosts where the first npm install fails but `--omit=optional` succeeds will regress to a failed detached update.
  - confidence: 0.86
- **[P2] Do not mark delegated installs as complete:** `src/infra/update-runner.ts:930-937`
  - body: The PR returns `status: "ok"` immediately after spawning the helper, before npm has installed anything. If the helper later records `install-failed` or `pid-wait-timeout`, the sentinel and API response still look like a successful update.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The patch introduces a concrete Windows command-execution risk in the detached helper.

Concerns:

- **[medium] Generated cmd script does not escape argv metacharacters:** `src/infra/update-detached-win32.ts:126`
  - body: `installArgv` is converted to a single cmd command with only space-based quoting, so shell metacharacters in package specs or paths can change the command that runs. This is a supply-chain-sensitive update path and should preserve argv semantics or use a hardened cmd/PowerShell quoting helper.
  - confidence: 0.9
