## Summary

Keep this PR open, but do not merge it as-is. Current main still has a real Windows package-update gap for live Gateway `update.run`, while the proposed detached helper introduces lifecycle, observability, retry, environment, and command-execution regressions.

## What This Changes

The PR adds a Windows-only detached `.cmd` updater helper, delegates win32 global package updates to it, exits the Gateway after responding to `update.run`, and adds unit coverage for helper spawning/result parsing.

## Best Possible Solution

Revise the Windows `update.run` package-update path around the current shared package-update contract: run the package replacement outside the live Gateway process, preserve prepared npm environment and fallback/verification behavior, use argv-safe command execution, exit only when restart supervision is verified, and reconcile detached results through startup/update status before reporting success.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists from the related #40540 Windows EBUSY logs plus current-main source: trigger a global npm package update from the live Gateway/Control UI on Windows, where the handler still reaches the live package-tree swap path.

## Solution Assessment

No. The detached-helper direction is plausible, but this patch is not the best way to solve it because it bypasses current package-update safeguards, reports success before installation finishes, loses environment handling, builds an unsafe cmd command, and can hard-exit without proving the Gateway can restart.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Default the detached exit delay before clamping:** `src/gateway/server-methods/update.ts:119`
  - body: `restartDelayMs` is optional, so `Math.max(restartDelayMs, 2000)` becomes `NaN` for the default request. Node coerces that timer to a near-immediate timeout, so this can exit the Gateway before the `update.run` response has flushed.
  - confidence: 0.95
- **[P1] Require restart supervision before exiting:** `src/gateway/server-methods/update.ts:117`
  - body: This branch reports a successful detached restart and then calls `process.exit(0)`, but the helper only relaunches through `schtasks`. On unsupervised Windows runs or missing task installs, `update.run` can return success and leave the Gateway down.
  - confidence: 0.88
- **[P1] Pass the prepared install environment to the helper:** `src/infra/update-detached-win32.ts:137-141`
  - body: `spawnDetachedUpdate` receives `globalInstallEnv`, but the `spawn()` options omit `env`, so the cmd helper runs with the ambient process environment instead of the Windows npm/Corepack/portable-Git settings prepared for package updates.
  - confidence: 0.92
- **[P1] Escape install argv before embedding it in cmd:** `src/infra/update-detached-win32.ts:126-128`
  - body: The helper converts argv to a batch command by only quoting arguments with spaces. Cmd metacharacters such as `&`, `|`, `%`, `!`, and `^` in a package spec or path will be interpreted by the shell, unlike the previous argv-based execution.
  - confidence: 0.9
- **[P2] Preserve npm's omit-optional retry:** `src/infra/update-runner.ts:895`
  - body: The Windows branch delegates only the primary `globalInstallArgs(...)` command and returns before the existing `globalInstallFallbackArgs` retry can run. Machines where the first npm install fails but the `--omit=optional` retry succeeds will regress to failed detached updates.
  - confidence: 0.86
- **[P2] Do not mark delegated installs as complete:** `src/infra/update-runner.ts:930-937`
  - body: This returns `status: "ok"` immediately after spawning the helper, before npm has installed anything. If the helper later records `install-failed` or `pid-wait-timeout`, the API response and restart sentinel still look like a successful update.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete Windows command-execution risk in the detached update helper.

Concerns:

- **[medium] Generated cmd script does not escape argv metacharacters:** `src/infra/update-detached-win32.ts:126`
  - body: `installArgv` is converted to a single cmd command with only space-based quoting, so shell metacharacters in package specs or paths can alter the command that runs. This is a supply-chain-sensitive update path and should preserve argv semantics or use a hardened cmd/PowerShell quoting helper.
  - confidence: 0.9
