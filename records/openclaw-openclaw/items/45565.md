## Summary

Keep open: current main and the latest shipped release still expose only suppression-oriented warning controls, with no configured target for lifecycle or internal warning delivery; the remaining work is a public config and routing-policy decision rather than conservative cleanup.

## What This Changes

The issue requests a messages config option to route or suppress gateway lifecycle and internal tool-warning notifications away from the active conversation channel.

## Best Possible Solution

Define a documented, validated warning-routing policy in core messages config with category defaults, route/suppress semantics, target authorization, fallback behavior, and tests/docs for lifecycle and tool-error delivery.

Not applicable as a feature request. Source inspection shows the proposed routing keys are absent and current warning payloads continue through normal reply delivery.

## Solution Assessment

Unclear for the exact key names. The maintainable direction is a shared category-aware core warning-routing policy, coordinated with existing tool-warning suppression work, rather than a Discord-specific special case.
