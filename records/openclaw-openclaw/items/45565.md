## Summary

Current main has shipped targeted suppression for recovered Discord tool warnings and merged Telegram post-final noise suppression, but it does not provide the requested dedicated operational destination or a unified lifecycle-notification contract. The remaining feature is meaningful and not owned by the closed cron report or the Telegram-only open PR, so it should stay open for a narrow product decision rather than stale cleanup.

## What This Changes

Add configurable routing or suppression for gateway lifecycle and internal tool-failure notifications so they do not appear in unrelated active conversation channels.

## Best Possible Solution

Define a small typed operator-notification category above transport adapters, keep recovered failures suppressed and actionable terminal errors contextual, then support an optional account-aware destination only for that diagnostic category with explicit fallback behavior.

## Reproduction Assessment

No. The historical OpenClaw v2026.3.12 examples are concrete, but current main now suppresses several recovered and post-final tool-warning paths, and no fresh current-main Discord lifecycle reproduction establishes what noise remains.

## Solution Assessment

No. A broad catch-all or per-category config map is not yet the best solution; maintainers should first define the notification taxonomy and ownership boundary, then add only the narrow destination contract that remains necessary.
