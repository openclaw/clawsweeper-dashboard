## Summary

Keep open: current main has partial warning suppression and cron-specific alert routing, but it still lacks a supported config surface for routing gateway lifecycle or internal warning categories to a dedicated channel. Because the request adds cross-channel config/default behavior, it needs maintainer product review rather than an automated fix PR.

## What This Changes

Request a `messages` config surface to route or suppress gateway lifecycle and internal warning messages instead of posting them in the active conversation channel.

## Best Possible Solution

Define one maintainer-approved warning-routing policy, reuse existing outbound target parsing/delivery helpers where possible, then add schema/docs/tests for default compatibility, tool-error warnings, gateway/system notices, and cron interactions.

## Reproduction Assessment

Yes at source level: current config/schema expose suppression and visible-reply controls but no dedicated warning target, and warning payloads flow through the normal reply dispatch path. I did not run a live Discord restart repro in this read-only review.

## Solution Assessment

Unclear as a concrete implementation shape. Dedicated routing is plausible, but maintainers need to choose categories, defaults, and overlap with `messages.suppressToolErrors`, cron alerts, and raw provider-error policy before implementation.
