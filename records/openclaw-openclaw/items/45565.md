## Summary

Keep open: current main still lacks a supported destination/category routing surface for lifecycle or internal warning messages; existing `messages.suppressToolErrors` and `cron.failureDestination` cover narrower suppression or cron-only cases, so the remaining work is a maintainer product/config decision.

## What This Changes

The issue requests a config surface to route gateway lifecycle and internal warning messages to a dedicated destination instead of the active conversation channel.

## Best Possible Solution

Define one canonical warning-routing policy with safe defaults, category semantics, config/schema/docs alignment, and regression coverage across the shared warning delivery path plus cron-specific behavior.

## Reproduction Assessment

Yes, at source level. Current main lacks the requested routing keys, still builds warning payloads without a destination/category target, and tests preserve user-visible mutating tool warnings after success-looking replies unless broad suppression is enabled.

## Solution Assessment

No, the one-off `messages.lifecycleTarget` shape is not proven to be the best final design. The maintainable path is a maintainer-approved warning-routing taxonomy that composes with broad tool-error suppression and cron failure destinations.
