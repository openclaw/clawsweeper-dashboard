## Summary

Keep open: current main still has only partial warning suppression, not a configurable dedicated route for gateway lifecycle or internal warning messages. The merged `messages.suppressToolErrors` work from #16620 and the open #39406 discussion are useful adjacent context, but they do not supersede the routing request.

## What This Changes

Add a public config option to route or suppress gateway lifecycle/internal warning messages instead of posting them to the active conversation channel.

## Best Possible Solution

Add a generic category-aware internal warning delivery policy in core config, with validated channel-route targets, backwards-compatible active-channel fallback, docs/schema/help updates, and regression coverage across lifecycle, tool-error, and system-notice delivery.

## Reproduction Assessment

Not applicable as a feature request, but the source-level verification is high-confidence: proposed routing keys are absent/rejected, and current tool warnings still become `isError` payloads sent through the normal reply path.

## Solution Assessment

Unclear for the exact requested key names because the public API has not been chosen. The maintainable direction is a shared routing policy with category-aware targets, not a Discord-only special case or silent suppression by default.
