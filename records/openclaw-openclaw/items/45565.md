## Summary

Keep open. Current main still lacks a configurable dedicated target for gateway lifecycle/internal warning delivery; it only has partial suppression for some tool errors, and unresolved mutating tool failures still produce normal user-facing error payloads through the existing conversation delivery path. The expanded discussion and related #39406 remain valid rather than obsolete.

## What This Changes

The issue requests a public config option to route or suppress gateway lifecycle and internal warning messages separately from the active conversation channel.

## Best Possible Solution

Add a generic core message-delivery policy for internal warning categories under the public config contract, with strict schema/types/help/docs, validated target resolution through the existing channel-route layer, backward-compatible active-channel fallback, and regression coverage for lifecycle, tool-error, and system-notice delivery alongside existing suppression behavior.

## Reproduction Assessment

Not applicable as a feature request, but the source-level check is high-confidence: current main has no proposed routing keys, strict messages schema would reject them, and unresolved mutating tool warnings still become normal `isError` reply payloads.

## Solution Assessment

Unclear for the exact requested key names because maintainers have not chosen the public API. The maintainable direction is a shared category-aware routing policy with validated targets, not a Discord-only special case or broad silent suppression by default.
