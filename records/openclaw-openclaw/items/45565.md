## Summary

Keep open: current main and v2026.6.6 still lack a supported target for routing lifecycle or internal warning categories; only coarse tool-error suppression and cron-specific failure destinations exist, so the remaining work is a maintainer product/config decision.

## What This Changes

Request adds a messages config surface to route gateway lifecycle and internal warning messages to a dedicated channel or suppress them instead of sending them to the active conversation.

## Best Possible Solution

Define one canonical warning-routing policy in the shared warning delivery path, with safe defaults, category semantics, config/schema/docs alignment, and regression coverage for both fresh config and upgrade behavior.

## Reproduction Assessment

Yes, at source level. Current main lacks the requested routing keys, builds warning payloads without category targets, and sends them through the existing reply/dispatcher route; I did not run a live Discord replay.

## Solution Assessment

No, the proposed one-off lifecycleTarget key is not clearly the best final shape. The maintainable path is a maintainer-approved routing taxonomy that composes with existing suppression and cron failure-destination behavior.
