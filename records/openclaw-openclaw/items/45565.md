## Summary

Keep open: current main and the latest release have partial suppression and cron-specific alert routing, but they do not provide a supported config surface for routing gateway lifecycle or internal warning categories to a dedicated channel. The remaining work is a core message-routing/config design decision, not a stale or already-implemented request.

## What This Changes

The issue requests a configuration option to route gateway lifecycle and internal warning messages away from the active conversation channel and into a dedicated ops/debug target.

## Best Possible Solution

Maintainers should decide one canonical warning-routing policy, then implement it through the shared warning delivery path, config schema, docs, and focused upgrade tests while preserving current default behavior.

## Reproduction Assessment

Yes at source level, but not as a live Discord replay. Current schema lacks the requested routing keys, and warning payloads still flow through the existing reply/dispatcher target.

## Solution Assessment

No, a one-off lifecycleTarget key is not clearly the best final shape yet. The safer path is a maintainer-approved routing taxonomy for lifecycle, tool-error, and system-notice warnings that composes with existing suppression and cron-specific destination behavior.
