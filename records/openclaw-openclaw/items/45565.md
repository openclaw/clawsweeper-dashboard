## Summary

Keep open: current main still only has suppression controls for tool warnings, not a configurable route for gateway lifecycle or internal warning messages, and the remaining work is a new public config/routing policy rather than conservative cleanup.

## What This Changes

The issue requests a config option to route gateway lifecycle and internal warning messages to a dedicated channel instead of the active conversation channel.

## Best Possible Solution

Add a generic, category-aware internal-warning delivery policy in core config with validated route targets, backwards-compatible active-channel fallback, docs/schema/help updates, and regression coverage for lifecycle, tool-error, and system-notice delivery.

## Reproduction Assessment

Not applicable as a feature request. Source inspection of current main confirms the proposed routing keys are absent/rejected and warning payloads still flow through ordinary reply delivery.

## Solution Assessment

Unclear for the exact key names because the public API has not been chosen. The maintainable direction is a shared category-aware routing policy, not a Discord-only special case or silent suppression by default.
