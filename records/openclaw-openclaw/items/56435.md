## Summary

Keep open. Current main still lacks the Feishu `update` card-action flow, and the earlier send-vs-update review point is addressed at PR head, but this draft/conflicting branch still lacks inspectable real Feishu proof and has concrete P2/security issues around card targeting, broadcast delivery, reply-size fallback, and a removed Signal SDK subpath.

## What This Changes

The branch adds a Feishu `update` card-action flow with a pending update registry, in-place processing/final card patches, duplicate text suppression, tests/docs, and Signal SDK entrypoint/runtime-api changes.

## Best Possible Solution

Rebase or replace this with a focused Feishu plugin implementation that uses callback-provided card ids, threads card-update context through every dispatch path, preserves existing reply fallback behavior, avoids removed Signal SDK subpaths, coordinates with the open Feishu card-update threads, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes, source-level: reviewing PR head against current main shows the metadata-derived patch target, missing broadcast `cardUpdate` wiring, direct final-card patching before chunk fallback, and stale Signal SDK subpath. I did not run live Feishu.

## Solution Assessment

No. The feature direction is plausible, but this branch is not the narrowest maintainable solution until it is rebased or replaced to use callback ids, preserve existing delivery safeguards, stay on current SDK seams, and supply real Feishu proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use callback message ids for card updates:** `extensions/feishu/src/card-action.ts:310-314`
  - body: The update path trusts `envelope.m.messageId` and rejects callbacks without that metadata, even though parsed callbacks already expose the clicked card id through `open_message_id`. Patch the callback-provided id so stale or crafted card metadata cannot redirect the update to a different card.
  - confidence: 0.89
- **[P2] Thread card-update context through broadcast dispatch:** `extensions/feishu/src/bot.ts:1023-1037`
  - body: The broadcast active-agent dispatcher is created without `cardUpdate`, so update actions in broadcast-enabled groups still use normal reply delivery instead of patching the source card. Pass the same card-update context into this dispatcher path.
  - confidence: 0.92
- **[P2] Preserve reply-size fallback before final card patches:** `extensions/feishu/src/reply-dispatcher.ts:404-410`
  - body: Card-update finals send the full model text through one `updateCardFeishu` payload, bypassing the existing chunk/fallback path. Long outputs can make the patch fail and leave the card stuck in the processing state.
  - confidence: 0.86
- **[P2] Rebase off the removed signal-core SDK subpath:** `scripts/lib/plugin-sdk-entrypoints.json:188`
  - body: Current main no longer exposes `openclaw/plugin-sdk/signal-core` as a public SDK subpath, but this branch re-adds that entrypoint and imports it from Signal runtime API. Use the current supported SDK/local Signal seams instead of restoring the removed surface.
  - confidence: 0.84
