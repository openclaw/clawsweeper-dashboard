## Summary

Keep open. Current main still lacks the PR's Feishu `update` card-action flow, so the branch is not obsolete, but the latest reviewed diff still has merge-blocking issues in the card patch target, broadcast dispatch propagation, final patch size handling, and current-main Signal SDK contract alignment.

## What This Changes

The PR adds a Feishu interactive-card `update` action flow with pending update state, in-place processing/final card patches, duplicate final-text suppression, tests/docs, and Signal runtime API/plugin SDK entrypoint adjustments.

## Best Possible Solution

Keep the work Feishu-owned, rebase onto current main, use the callback-provided card message id as the patch target, propagate card-update context through both single-agent and broadcast dispatch, reuse existing Feishu chunk/size fallback policy for final updates, and keep the Signal build fix aligned with current SDK subpaths rather than restoring `signal-core`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: construct a Feishu `update` callback that provides `open_message_id` but no trusted metadata `messageId`, or run the same flow in a broadcast-enabled group, and the reviewed PR path either targets the wrong id source or drops card-update context.

## Solution Assessment

No. The direction is useful, but this diff is not the best final fix because it trusts metadata for message mutation, misses the broadcast path, bypasses existing size handling, and reintroduces a Signal SDK subpath that current main no longer exposes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use callback message ids for card updates:** `extensions/feishu/src/card-action.ts:310-314`
  - body: The update path reads the patch target from envelope metadata, but Feishu callbacks already provide the clicked card's `open_message_id`. Newly built card payloads cannot reliably know their future message id, and metadata can be stale or crafted, so patch the callback message id after validation instead.
  - confidence: 0.89
- **[P2] Pass card-update context through broadcast dispatch:** `extensions/feishu/src/bot.ts:1023-1037`
  - body: The broadcast active-agent dispatcher is still created without `cardUpdate`, so update actions in broadcast-enabled Feishu groups fall back to normal reply delivery instead of patching the source card. This can reintroduce duplicate follow-up replies for the same click.
  - confidence: 0.93
- **[P2] Reuse size handling before final card patches:** `extensions/feishu/src/reply-dispatcher.ts:404-410`
  - body: Final card updates put the entire model text into one `updateCardFeishu` payload instead of going through the existing Feishu chunk/limit fallback used by normal replies. Long outputs can make the patch fail and leave the card stuck in the processing state.
  - confidence: 0.86
- **[P2] Drop the retired signal-core SDK entrypoint:** `scripts/lib/plugin-sdk-entrypoints.json:188`
  - body: Current main no longer has a `src/plugin-sdk/signal-core.ts` subpath, and Signal runtime exports now use supported SDK subpaths plus local seams. Adding `signal-core` back to the public entrypoint list conflicts with the current SDK contract after rebase.
  - confidence: 0.84
- **[P3] Add the required Feishu changelog entry:** `extensions/feishu/src/card-action.ts:304`
  - body: This is a user-facing Feishu fix, but the PR does not add an active-version `CHANGELOG.md` entry. Add a single-line fix entry with appropriate non-bot contributor credit before merge.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no concrete supply-chain change, but the new Feishu update path introduces a message-mutation authorization concern.

- **[medium] Card patch target comes from action metadata:** `extensions/feishu/src/card-action.ts:310`
