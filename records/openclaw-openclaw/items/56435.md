## Summary

Keep open. Current main still lacks Feishu interactive-card `update` action support, and the PR contains useful progress, including fixing the earlier send-vs-update concern by calling `updateCardFeishu` for the processing state. The latest PR head is still not merge-ready because it targets card patches from action metadata, misses the broadcast dispatcher path, bypasses existing Feishu size fallback for final patches, and reintroduces a retired `signal-core` SDK subpath.

## What This Changes

The PR branch adds a Feishu `update` card action with a pending update registry, in-place processing and final card patches, duplicate final-text suppression, Feishu tests/docs, and Signal runtime API/plugin SDK entrypoint adjustments.

## Best Possible Solution

Rebase the PR onto current main, keep the implementation inside the Feishu plugin, use the callback `open_message_id` as the validated patch target, pass `cardUpdate` through every dispatcher path including broadcast, reuse existing Feishu size/chunk fallback handling for final updates, and align Signal runtime exports with the current public SDK subpath contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on the PR head: an `update` callback that provides the callback `open_message_id` but no `m.messageId` is rejected, and a broadcast-enabled group creates the active dispatcher without `cardUpdate`, so final replies return to normal message delivery.

## Solution Assessment

No. The overall direction is maintainable, but this patch is not the best current fix because it trusts the wrong update target source, misses broadcast dispatch, bypasses existing Feishu size handling, and conflicts with the current SDK contract for `signal-core`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use callback message ids for card updates:** `extensions/feishu/src/card-action.ts:310-314`
  - body: The update branch only accepts `metadata.messageId`, but current Feishu callback parsing already preserves the clicked card's `open_message_id`. Use and validate `event.context.open_message_id` or `event.open_message_id` as the patch target so normal callbacks do not fail and stale or crafted card metadata cannot redirect the patch.
  - confidence: 0.89
- **[P2] Pass card-update context through broadcast dispatch:** `extensions/feishu/src/bot.ts:1023-1037`
  - body: The broadcast active-agent dispatcher is still created without `cardUpdate`, so `k:"update"` actions in broadcast-enabled groups fall back to normal reply delivery instead of patching the source card and can reintroduce duplicate follow-up replies.
  - confidence: 0.93
- **[P2] Guard final card patches with size handling:** `extensions/feishu/src/reply-dispatcher.ts:404-410`
  - body: Card-update finals put the entire model text into one `updateCardFeishu` payload, bypassing the existing Feishu `textChunkLimit` and chunk fallback used by normal replies. Long outputs can make the patch fail and leave the card stuck in the processing state.
  - confidence: 0.86
- **[P2] Drop the retired signal-core SDK entrypoint:** `scripts/lib/plugin-sdk-entrypoints.json:188`
  - body: Current main explicitly denies `signal-core` as a public plugin SDK subpath and no longer has `src/plugin-sdk/signal-core.ts`. Adding it back to `plugin-sdk-entrypoints.json` conflicts with the current SDK export policy after rebase.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff has no dependency or workflow supply-chain change, but the new Feishu update path introduces a message-mutation authorization concern.

Concerns:

- **[medium] Card patch target comes from action metadata:** `extensions/feishu/src/card-action.ts:310`
  - body: The PR patches the message id supplied in card metadata rather than the Feishu callback's `open_message_id`, so a stale or crafted card payload can direct the bot to patch a different Feishu card than the one clicked. Use and validate the callback message id before registering a pending update.
  - confidence: 0.78
