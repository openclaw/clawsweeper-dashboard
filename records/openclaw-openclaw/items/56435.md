## Summary

Keep open. Current main still does not implement this Feishu card-update action, and the PR contains useful work, but the draft branch is not mergeable as-is because the remaining blockers affect card mutation targeting, broadcast delivery, long-output handling, and the current plugin SDK subpath contract.

## What This Changes

The PR adds a Feishu `update` card action and pending-update registry, routes final replies back into Feishu cards, updates Feishu tests/docs, and changes the Signal runtime API plus plugin SDK entrypoint list.

## Best Possible Solution

Keep the feature in the Feishu plugin, rebase it onto current main, target callback `open_message_id`, pass update context through every reply-dispatch route, preserve Feishu chunk/fallback behavior, and leave Signal on supported SDK subpaths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on the PR head: an update callback with callback `open_message_id` but no metadata `messageId` is rejected, broadcast-enabled groups lose card-update context, and long final text bypasses existing chunking before patch.

## Solution Assessment

No. The implementation direction is useful, but this is not the best mergeable solution until it uses the callback card id, covers broadcast dispatch, guards oversized final patches, and aligns with the current SDK contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use callback message ids for card updates:** `extensions/feishu/src/card-action.ts:310-314`
  - body: The update path reads the patch target from card metadata and rejects the action when that field is absent. Feishu callbacks already provide the clicked card's `open_message_id`; use that callback-provided id after validation so normal callbacks work and stale or crafted metadata cannot redirect the patch.
  - confidence: 0.89
- **[P2] Pass card-update context through broadcast dispatch:** `extensions/feishu/src/bot.ts:1023-1037`
  - body: The broadcast active-agent dispatcher is still created without `cardUpdate`, so update actions in broadcast-enabled groups fall back to normal reply delivery instead of patching the source card. Thread the same card-update context into this dispatcher path.
  - confidence: 0.93
- **[P2] Reuse size fallback before final card patches:** `extensions/feishu/src/reply-dispatcher.ts:404-410`
  - body: Card-update finals send the full model text through one `updateCardFeishu` payload, bypassing the existing Feishu text chunk limit and fallback path. Long outputs can make the patch fail and leave the interaction card stuck in the processing state.
  - confidence: 0.86
- **[P2] Drop the retired signal-core SDK entrypoint:** `scripts/lib/plugin-sdk-entrypoints.json:188`
  - body: Current main explicitly denies `signal-core` as a public plugin SDK subpath and the Signal runtime API no longer imports it. Re-adding it to the entrypoint list conflicts with the current SDK contract after rebase.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Security review needs attention because the new Feishu update path can patch a message id supplied by card metadata instead of the callback card id.

Concerns:

  - body: The PR patches the message id stored in the card payload rather than the Feishu callback's `open_message_id`, so stale or crafted card metadata could direct the bot to mutate a different Feishu card than the one clicked.
