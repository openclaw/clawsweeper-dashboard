## Summary

Keep open. Current main still does not implement the Feishu `update` card action, and this draft PR contains useful Feishu-plugin work, but the latest head remains dirty against main and has blocking correctness findings in the card-update target selection, broadcast dispatch, final card size handling, and the retired `signal-core` SDK subpath.

## What This Changes

The PR adds a Feishu interactive-card `update` action flow with a pending update registry, in-place processing and final card patches, duplicate final-text suppression, tests/docs, and Signal runtime API/plugin SDK entrypoint adjustments.

## Best Possible Solution

Keep the work inside the Feishu plugin, rebase onto current main, use the callback-provided card message id as the update target with appropriate validation, carry card-update context through both single-agent and broadcast dispatch, reuse the existing Feishu size/chunk fallback policy for final updates, and align the Signal build fix with the current SDK contract instead of restoring `signal-core`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on the PR head: a normal Feishu update callback that only provides `open_message_id` is rejected because the update path requires `metadata.messageId`, and a broadcast-enabled group loses `cardUpdate` before final reply dispatch.

## Solution Assessment

No. The implementation direction is maintainable, but this patch is incomplete because it trusts the wrong update target source, misses the broadcast path, bypasses existing Feishu size handling, and reintroduces a retired SDK subpath.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use callback message ids for card updates:** `extensions/feishu/src/card-action.ts:310-314`
  - body: The update path only accepts `metadata.messageId`, but Feishu callbacks already carry the clicked card's `open_message_id` and newly sent cards cannot know their eventual message id when button payloads are built. Use `event.context.open_message_id`/`event.open_message_id` as the update target and validate it instead of requiring metadata to name the message to patch.
  - confidence: 0.89
- **[P2] Pass card-update context through broadcast dispatch:** `extensions/feishu/src/bot.ts:1023-1037`
  - body: The broadcast active-agent dispatcher is still created without `cardUpdate`, so update actions in broadcast-enabled Feishu groups fall back to normal reply delivery instead of patching the source card and can reintroduce duplicate follow-up replies.
  - confidence: 0.93
- **[P2] Guard final card patches with Feishu size handling:** `extensions/feishu/src/reply-dispatcher.ts:404-410`
  - body: Card-update finals put the entire model text into one `updateCardFeishu` payload, bypassing the existing `textChunkLimit`/chunk fallback used by normal Feishu replies. Long outputs can make the patch fail and leave the card stuck in the processing state.
  - confidence: 0.86
- **[P2] Drop the retired signal-core SDK entrypoint:** `scripts/lib/plugin-sdk-entrypoints.json:188`
  - body: Current main explicitly denies `signal-core` as a public plugin SDK subpath and no longer has `src/plugin-sdk/signal-core.ts`. Adding it back to `plugin-sdk-entrypoints.json` conflicts with the current SDK export policy after rebase.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but the new Feishu update path introduces a message-mutation authorization concern.

Concerns:

- **[medium] Card patch target comes from action metadata:** `extensions/feishu/src/card-action.ts:310`
  - body: The PR patches the message id supplied in card metadata rather than the Feishu callback's `open_message_id`, so a crafted or stale card payload can direct the bot to patch a different Feishu card than the one clicked. Use and validate the callback message id before registering a pending update.
  - confidence: 0.78
