## Summary

Keep open: current main does not implement Feishu `update` card actions, but this PR is not mergeable as-is because the latest diff still has source-level blockers and a security-sensitive patch-target issue; the author did address the earlier send-vs-patch review note.

## What This Changes

The PR adds Feishu update-kind card interactions with a pending update registry, in-place processing/final card patches, duplicate reply suppression, tests/docs, and a Signal runtime API/SDK entrypoint adjustment.

## Best Possible Solution

Keep the feature in the Feishu plugin, rebase onto current main, use the callback card id as the patch target, carry card-update context through every reply route, preserve Feishu size fallback behavior, and keep Signal on supported SDK subpaths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists from the PR diff and current main: update callbacks can rely on metadata for the patch target, broadcast dispatch lacks card-update wiring, long final text bypasses existing chunking, and `signal-core` is denied by the current SDK contract.

## Solution Assessment

No. The implementation direction is reasonable, but the mergeable solution needs callback-provided message ids, broadcast dispatch support, Feishu text-size fallback preservation, and removal of the retired SDK subpath change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use callback message ids for card updates:** `extensions/feishu/src/card-action.ts:306-310`
  - body: The update path reads the patch target from card metadata and rejects the action when that field is absent. Feishu callbacks already provide the clicked card's `open_message_id`; use that callback-provided id after validation so stale or crafted metadata cannot redirect the patch.
  - confidence: 0.89
- **[P2] Pass card-update context through broadcast dispatch:** `extensions/feishu/src/bot.ts:1138`
  - body: The PR wires `cardUpdate` only through the single-agent dispatcher path. In broadcast-enabled groups, the active dispatcher still falls back to normal reply delivery, so the source card is not updated in place and duplicate follow-up replies can reappear.
  - confidence: 0.91
- **[P2] Reuse size fallback before final card patches:** `extensions/feishu/src/reply-dispatcher.ts:401-410`
  - body: Card-update finals send the full model text through one `updateCardFeishu` payload, bypassing the existing Feishu chunk limit and fallback path. Long outputs can make the patch fail and leave the interaction card stuck in the processing state.
  - confidence: 0.86
- **[P2] Drop the retired signal-core SDK entrypoint:** `scripts/lib/plugin-sdk-entrypoints.json:188`
  - body: Current main explicitly denies `signal-core` as a public plugin SDK subpath and the Signal runtime API no longer imports it. Re-adding it to the entrypoint list conflicts with the current SDK contract after rebase.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The Feishu update path has a concrete patch-target concern because it can patch a message id supplied by card metadata instead of the callback card id.

Concerns:

- **[medium] Card patch target comes from action metadata:** `extensions/feishu/src/card-action.ts:306`
  - body: The PR patches the message id stored in the card payload rather than the Feishu callback's `open_message_id`, so stale or crafted card metadata could direct the bot to mutate a different Feishu card than the one clicked.
  - confidence: 0.78
