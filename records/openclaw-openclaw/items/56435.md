## Summary

Keep open: current main does not implement the requested Feishu interactive card update flow, but this PR head is not mergeable as-is because source review still shows patch-target, broadcast, reply-size, and stale Signal seam blockers, and it lacks inspectable live Feishu proof.

## What This Changes

The PR adds a Feishu `update` card-action path with a pending update registry, in-place processing/final card patches, duplicate final-text suppression, Feishu docs/tests, and a Signal runtime/API entrypoint adjustment.

## Best Possible Solution

A focused Feishu plugin implementation should use callback-provided card ids, preserve existing reply delivery safeguards, reconcile with https://github.com/openclaw/openclaw/pull/64027 and https://github.com/openclaw/openclaw/issues/71677, and include redacted live Feishu proof before merge.

## Reproduction Assessment

Yes, source-level. PR head shows metadata-derived patch targeting, missing broadcast `cardUpdate` wiring, direct final card patching before the chunk/fallback helper, and stale Signal SDK entrypoint drift; I did not run live Feishu.

## Solution Assessment

No. The feature direction is plausible, but this branch should be rebased or replaced with a narrower implementation that uses callback ids, threads all dispatch paths, preserves fallback behavior, and stays on current SDK seams.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use callback message ids for card updates:** `extensions/feishu/src/card-action.ts:310-314`
  - body: The update path takes the patch target from `envelope.m.messageId` and rejects callbacks without that metadata field. Current callbacks already expose the clicked card id through `open_message_id`; use that parsed callback id so stale or crafted card metadata cannot redirect the patch.
  - confidence: 0.89
- **[P2] Thread card-update context through broadcast dispatch:** `extensions/feishu/src/bot.ts:1023-1037`
  - body: The broadcast active-agent dispatcher is created without `cardUpdate`, so update actions in broadcast-enabled groups still use normal reply delivery instead of patching the source card. Pass the same card-update context into this dispatcher path.
  - confidence: 0.92
- **[P2] Preserve reply-size fallback before final card patches:** `extensions/feishu/src/reply-dispatcher.ts:404-410`
  - body: Card-update finals send the full model text through one `updateCardFeishu` payload, bypassing the existing chunk/fallback path. Long outputs can make the patch fail and leave the card stuck in the processing state.
  - confidence: 0.86
- **[P2] Rebase off the removed signal-core SDK subpath:** `scripts/lib/plugin-sdk-entrypoints.json:188`
  - body: Current main no longer exposes `openclaw/plugin-sdk/signal-core`, but this branch still adds that entrypoint and imports it from Signal runtime API. Rebase the Signal build fix onto current supported subpaths instead of reintroducing the stale SDK surface.
  - confidence: 0.84
