## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the callback message id for card patches:** `extensions/feishu/src/card-action.ts:310-341`
  - body: This path reads `messageId` from `envelope.m` and then uses it for both the pending update and `updateCardFeishu`. Current main already carries the clicked card's `open_message_id` through the callback event, so stale or crafted card metadata can patch a different message than the one the user clicked; resolve the target from the callback event and cover that shape in tests.
  - confidence: 0.89
- **[P1] Propagate cardUpdate through broadcast replies:** `extensions/feishu/src/bot.ts:1023-1037`
  - body: `cardUpdateContext` is only passed to the single-agent dispatcher. In groups with `broadcastAgents`, the active dispatcher is created without card-update context, so `k:"update"` actions fall back to normal Feishu replies and can reintroduce duplicate follow-up messages instead of patching the original card.
  - confidence: 0.88
- **[P2] Keep final card updates on the existing fallback path:** `extensions/feishu/src/reply-dispatcher.ts:404-410`
  - body: The card-update final path sends the whole final text as one `updateCardFeishu` payload instead of using the existing chunking/fallback behavior. Long outputs that exceed a single card patch can fail and leave the card stuck in Processing while the final text is suppressed.
  - confidence: 0.85
- **[P2] Drop the stale signal-core SDK entrypoint:** `scripts/lib/plugin-sdk-entrypoints.json:188`
  - body: Current main no longer exposes `openclaw/plugin-sdk/signal-core` through the Signal runtime path, but this branch adds the public SDK entrypoint back as part of an unrelated build fix. Rebase and remove that SDK-surface change unless maintainers explicitly want to restore the old subpath with compatibility tests and docs.
  - confidence: 0.82
