## Summary

Keep open. Current main does not implement Feishu `update` card actions, so this PR is not obsolete, but the PR head still has merge-blocking issues in the Feishu update target, broadcast reply path, long final-result handling, and the Signal SDK subpath contract.

## What This Changes

The PR adds Feishu interactive-card update actions with a pending update registry, in-place processing/final card patches, duplicate final-text suppression, tests/docs, and Signal runtime API/plugin SDK entrypoint adjustments.

## Best Possible Solution

Keep the work in the Feishu plugin, but rebase onto current main and make the update target come from the callback `open_message_id` with validation. The final implementation should pass card-update context through both single-agent and broadcast dispatch, reuse existing Feishu size/chunk fallback behavior for final results, and align the Signal build fix with the current SDK contract rather than restoring `signal-core`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on the PR head: a normal Feishu update callback with `open_message_id` but no `m.messageId` is rejected, a broadcast-enabled group loses `cardUpdate` before the reply dispatcher, and a long final text goes into a single card patch.

## Solution Assessment

No. The implementation direction is useful, but this patch is not the best mergeable solution because it trusts the wrong card id source, misses the broadcast route, bypasses established size handling, and reintroduces a retired SDK subpath.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use callback message ids for card updates:** `extensions/feishu/src/card-action.ts:310-314`
  - body: The update flow reads the patch target from `envelope.m.messageId` and rejects the action if that metadata is absent. Feishu callbacks already carry the clicked card's `open_message_id`, and card payload metadata can be stale or crafted, so register and patch the callback-provided id after validation instead of trusting the envelope.
  - confidence: 0.89
- **[P2] Pass card-update context through broadcast dispatch:** `extensions/feishu/src/bot.ts:1023-1037`
  - body: The broadcast active-agent dispatcher is still created without `cardUpdate`, so update actions in broadcast-enabled groups fall back to normal Feishu reply delivery instead of patching the source card. Thread the same `cardUpdateContext` used by the single-agent path into this dispatcher.
  - confidence: 0.93
- **[P2] Guard final card patches with Feishu size handling:** `extensions/feishu/src/reply-dispatcher.ts:404-410`
  - body: Card-update finals send the entire model text through one `updateCardFeishu` payload, bypassing the existing `textChunkLimit` and chunk fallback used for normal replies. Long outputs can make the patch fail and leave the card stuck in the processing state.
  - confidence: 0.86
- **[P2] Drop the retired signal-core SDK entrypoint:** `scripts/lib/plugin-sdk-entrypoints.json:188`
  - body: Current main explicitly denies `signal-core` as a public plugin SDK subpath and no longer exposes that facade. Re-adding it to `plugin-sdk-entrypoints.json` conflicts with the current SDK contract after rebase; keep the Signal runtime API on current local or supported SDK exports instead.
  - confidence: 0.84
