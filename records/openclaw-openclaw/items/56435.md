## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Resolve card patches from the callback message id:** `extensions/feishu/src/card-action.ts:310-341`
  - body: This path reads `messageId` from `envelope.m` and then uses it for `updateCardFeishu`. Current main already carries the clicked card id through `open_message_id` / `context.open_message_id`, so stale or crafted card metadata can patch a different message than the one the user clicked; resolve the patch target from the callback event and cover those shapes in tests.
  - confidence: 0.89
- **[P1] Pass cardUpdate to broadcast dispatchers:** `extensions/feishu/src/bot.ts:1023-1037`
  - body: `cardUpdateContext` is only passed to the single-agent dispatcher. In groups with `broadcastAgents`, the active dispatcher is created without card-update context, so `k:"update"` actions can fall back to normal Feishu replies and reintroduce duplicate follow-up messages instead of patching the original card.
  - confidence: 0.88
- **[P2] Preserve reply fallback for card-update finals:** `extensions/feishu/src/reply-dispatcher.ts:404-410`
  - body: The card-update final path sends one `updateCardFeishu` patch and returns before the existing chunking/fallback path. Long model outputs or rejected card payloads can therefore leave the interaction card stuck in processing and drop the final text; keep the existing chunk/fallback behavior for failed or oversized final updates.
  - confidence: 0.84
- **[P1] Remove the stale signal-core SDK surface:** `scripts/lib/plugin-sdk-entrypoints.json:188`
  - body: This PR adds `signal-core` as a plugin SDK entrypoint while current main and the latest release have Signal using narrower SDK/local seams without that public subpath. Reintroducing this unrelated SDK surface from a Feishu PR creates a conflicting plugin API contract; drop the Signal entrypoint/runtime churn or get explicit maintainer approval for the API expansion.
  - confidence: 0.82
