## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Target card updates from the callback id:** `extensions/feishu/src/card-action.ts:310`
  - body: The update flow trusts `envelope.m.messageId` and patches that message directly, but current main already carries the clicked card's `open_message_id` from the Feishu callback. A stale or crafted card value can therefore patch the wrong card; derive the update target from the callback id and reject mismatches instead.
  - confidence: 0.9
- **[P1] Pass card-update context through broadcast dispatch:** `extensions/feishu/src/bot.ts:1141`
  - body: `cardUpdate` is only passed to the single-agent dispatcher, so groups configured with `broadcastAgents` still build the active dispatcher without update context. In that mode the original card is not updated in place and normal duplicate follow-up replies can reappear.
  - confidence: 0.89
- **[P2] Keep final card updates on the Feishu size fallback path:** `extensions/feishu/src/reply-dispatcher.ts:405-410`
  - body: Card-update mode patches the final text as one card payload instead of using the existing `sendChunkedTextReply`/`textChunkLimit` path. Long model output can make `im.message.patch` fail, leaving the card stuck in the processing state while the final text is suppressed.
  - confidence: 0.85
- **[P2] Do not restore the removed signal-core SDK subpath:** `scripts/lib/plugin-sdk-entrypoints.json:188`
  - body: Current main routes Signal runtime exports through narrower SDK/local seams and has no `signal-core` entrypoint. Re-adding that public subpath in this Feishu PR reopens a stale SDK boundary and can drift from package export checks after rebase.
  - confidence: 0.84
