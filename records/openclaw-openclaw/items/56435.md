## Summary

Keep this draft PR open: the Feishu card-update direction is useful, but the PR head still has merge blockers around the patch target, broadcast dispatch, long final text handling, and current plugin SDK subpath contracts.

## What This Changes

The PR adds a Feishu `update` card-action flow with a pending update registry, in-place card patching, reply suppression, tests/docs, and a Signal runtime API build adjustment.

## Best Possible Solution

Keep the feature in the Feishu plugin, rebase onto current main, patch the callback card id, pass card-update context through every reply route, preserve chunk/fallback behavior, and keep Signal on supported SDK subpaths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists from the PR diff and current main: update callbacks can rely on metadata for the patch target, broadcast groups have a separate dispatcher path, long final text bypasses existing chunking, and `signal-core` conflicts with the current SDK contract.

## Solution Assessment

No. The implementation direction is reasonable, but it is not the best mergeable solution until it uses callback-provided message ids, covers broadcast dispatch, reuses Feishu text-size fallback behavior, and removes the retired SDK subpath change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use callback message ids for card updates:** `extensions/feishu/src/card-action.ts:310-314`
  - body: The update path reads the patch target from card metadata and rejects the action when that field is absent. Feishu callbacks already provide the clicked card's `open_message_id`; use that callback-provided id after validation so normal callbacks work and stale or crafted metadata cannot redirect the patch.
  - confidence: 0.89
- **[P2] Pass card-update context through broadcast dispatch:** `extensions/feishu/src/bot.ts:1023-1037`
  - body: The broadcast active-agent dispatcher is created without the card-update context, so update actions in broadcast-enabled groups still fall back to normal reply delivery instead of patching the source card. Thread the same context into this dispatcher path.
  - confidence: 0.93
- **[P2] Reuse size fallback before final card patches:** `extensions/feishu/src/reply-dispatcher.ts:404-410`
  - body: Card-update finals send the full model text through one `updateCardFeishu` payload, bypassing the existing Feishu chunk limit and fallback path. Long outputs can make the patch fail and leave the interaction card stuck in the processing state.
  - confidence: 0.86
- **[P2] Drop the retired signal-core SDK entrypoint:** `scripts/lib/plugin-sdk-entrypoints.json:188`
  - body: Current main explicitly denies `signal-core` as a public plugin SDK subpath and the Signal runtime API no longer imports it. Re-adding it to the entrypoint list conflicts with the current SDK contract after rebase.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The new Feishu update path has a concrete patch-target concern because it can patch a message id supplied by card metadata instead of the callback card id.

Concerns:

- **[medium] Card patch target comes from action metadata:** `extensions/feishu/src/card-action.ts:310`
  - body: The PR patches the message id stored in the card payload rather than the Feishu callback's `open_message_id`, so stale or crafted card metadata could direct the bot to mutate a different Feishu card than the one clicked.
  - confidence: 0.78
