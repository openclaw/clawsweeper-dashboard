## Label Justifications

- `impact:message-loss`: The reported failure can suppress or replace useful child result text with `(no output)` before the parent sees it.
- `impact:session-state`: The same child can disagree across notification delivery, wait_agent status, task rows, and diagnostic lookup state.

## Summary

Keep open: current main has a Codex-native completion monitor, but it still collapses null or empty Codex subagent completion payloads to `(no output)` and treats that parsed completion as terminal before a later transcript or wait result can improve it. The related open reports are adjacent, but they do not fully cover this released 2026.6.1 multi_agent_v1/Codex-native mismatch.

## What This Changes

Ensure Codex native subagent completion notifications, wait_agent results, and diagnostics agree when child output is present or absent.

## Best Possible Solution

Use one durable result source for Codex-native child completion, preserve final text before terminal delivery, represent no-final as a typed reason, and expose stable codex-thread diagnostics through existing task/subagent/session tooling.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current code shows null or empty native completions become `(no output)`, upstream Codex allows completed-without-final-text, and the monitor finalizes on the first parsed completion; a live Telegram repro was not run in this read-only pass.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the durable result and diagnostic contract; the requested direction is sound, but the fix should stay in the Codex-native monitor/task runtime path with regression coverage and no new config knob.
