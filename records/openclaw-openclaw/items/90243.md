## Summary

Keep open: current main and v2026.6.8 still do not expose a supported way to decouple the configured/catalog model id from the Google Vertex model id sent upstream, and the related Flex tracker only partially covers this broader provider-model contract.

## What This Changes

The issue requests Google Vertex physical-model aliasing so a virtual model id can carry Priority/Flex headers while the provider request uses a real Gemini model id.

## Best Possible Solution

Have maintainers approve one explicit provider-facing model-id contract, coordinate only the shared tier semantics with https://github.com/openclaw/openclaw/issues/69102, and avoid suffix guessing.

## Reproduction Assessment

Yes. Source inspection shows configured model headers can be merged, but both Google Vertex URL construction and the shared Google SDK path still use the selected `model.id` as the upstream model string.

## Solution Assessment

Unclear until maintainers choose the public API shape. The suffix-stripping workaround is not the best long-term fix; the maintainable path is an explicit provider-facing model-id or tier contract with tests and docs.
