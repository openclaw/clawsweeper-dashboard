## Summary

Keep open: current main and latest release still lack a supported virtual-model-to-physical-model mapping for Google Vertex, while users can already attach model headers. The remaining work is a provider/model config contract decision, not a stale or duplicate cleanup item.

## What This Changes

The issue requests a way to map a virtual Google Vertex catalog model id to a different physical provider model id so Priority/Flex PayGo headers can be selected per model.

## Best Possible Solution

Design one canonical physical/provider model-id contract, wire it through Google Vertex and shared model catalog/config resolution, and cover model headers plus endpoint model selection with focused tests and docs.

## Reproduction Assessment

Yes, from source: configure a google-vertex model with a virtual id and model headers, and current main will merge the headers but still build the provider endpoint from that virtual model id. I did not run a live Vertex request in this read-only review.

## Solution Assessment

No to the suffix-stripping workaround; the maintainable path is a deliberate physical/provider model-id contract that flows through config, catalog, runtime model construction, Google transport, tests, and docs.
