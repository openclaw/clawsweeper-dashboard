## Summary

Keep open: current `main` and latest release still send the OpenClaw catalog model id as the Google Vertex physical model id, while the requested alias/physical-id split is not implemented and needs a deliberate provider/model API decision.

## What This Changes

Request support for Google Vertex model aliases that keep a virtual OpenClaw model id for routing, pricing, and headers while sending a separate physical Gemini model id to Vertex.

## Best Possible Solution

Design one canonical model-catalog/config contract for provider-facing physical model ids, wire it through Google Vertex and adjacent model resolution paths with focused tests/docs, and avoid hardcoded suffix stripping.

## Reproduction Assessment

Yes from source: current main sends `model.id` as the Google Vertex endpoint model id while model-level headers are merged separately. I did not run a live Vertex request in this read-only cleanup review.

## Solution Assessment

No to the suffix-stripping approach; the best fix is a deliberate physical/API model id contract that flows through config, catalog normalization, runtime model construction, Google transport, tests, and docs.
