## Summary

Keep open: current main and the latest stable release still lack a supported Google Vertex virtual-alias to physical-model mapping, and the related direct Gemini Flex tracker only partially overlaps this provider/model-id contract problem.

## What This Changes

Request a supported way for Google Vertex catalog aliases to attach tier headers/pricing while sending a separate physical model id to the provider API.

## Best Possible Solution

Approve one canonical provider-facing model-id contract, coordinated with https://github.com/openclaw/openclaw/issues/69102 where service-tier semantics overlap, then wire it through config/catalog/runtime, Google Vertex transport, tests, and docs without suffix guessing.

## Reproduction Assessment

Yes. Source inspection shows a google-vertex model can carry configured headers, but both current Google request paths still use the selected model id as the provider-facing model string; I did not run a live paid Vertex request in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. The suffix-stripping workaround is not the best long-term fix; the maintainable path is an explicit physical/provider model-id or tier contract with tests and docs.
