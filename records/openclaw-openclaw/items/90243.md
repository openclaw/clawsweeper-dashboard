## Summary

Keep open: current main and v2026.6.9 still send the configured Google Vertex model id directly to Vertex, so the requested virtual model-to-physical model mapping is still necessary and needs maintainer API/product direction.

## What This Changes

Requested support for mapping a user-facing Google Vertex/Gemini catalog alias to a physical provider model id so Flex/Priority tier headers can be used without sending a non-existent model path.

## Best Possible Solution

Have maintainers approve the provider/model contract, then implement a Google Vertex physical-model mapping that preserves the configured OpenClaw model ref while sending the approved physical id and tier headers to Vertex.

## Reproduction Assessment

Yes, source-reproducible: the current Vertex URL builder sends `model.id` as the publisher model path while headers are merged independently, so a virtual alias would still be sent as a non-existent Vertex model id. I did not run a live paid Vertex tier request in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. A provider-owned explicit physical-model mapping or a carefully scoped generic model contract is maintainable; suffix stripping would be a brittle workaround.
