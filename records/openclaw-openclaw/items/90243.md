## Summary

Keep open: current main and the latest release still pass the configured Google Vertex model id directly into the Vertex model endpoint, so the requested virtual catalog id to physical provider model mapping is still missing. The related Gemini Flex tracker overlaps on service-tier motivation, but it does not replace this Vertex aliasing/config-contract decision.

## What This Changes

Request support for Google Vertex virtual model ids or physical model aliases so per-model tier headers can route to Flex or Priority PayGo without sending a non-existent model id to Vertex.

## Best Possible Solution

Have maintainers choose a Google-provider-owned alias/tier contract, then implement the physical model id mapping with focused config/schema, Google transport, docs, and regression coverage while coordinating service-tier semantics with https://github.com/openclaw/openclaw/issues/69102.

## Reproduction Assessment

Yes, source-reproducible: current main constructs the Vertex endpoint from the selected model id and has no physical model id override, so a virtual alias would still be sent as the provider model path. I did not run a live paid Vertex tier request in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. A provider-owned physical model mapping or explicit params contract is likely safer than suffix stripping, which would hardcode naming policy into the transport.
