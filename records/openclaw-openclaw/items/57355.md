## Summary

Keep open: current main and the latest release still lack the requested Plugin SDK `registerMcpServer` surface; existing MCP support covers static bundle and user-config paths, and the related implementation PR is closed unmerged. The remaining work is a public SDK plus plugin-managed stdio process-launch security/product decision.

## What This Changes

Add a native Plugin SDK `registerMcpServer` API so plugins can register managed stdio MCP servers with OpenClaw-owned lifecycle and embedded OpenAI/Codex runtime wiring.

## Best Possible Solution

Design and land an additive, maintainer-reviewed Plugin SDK MCP registration contract that reuses existing MCP validation/projection paths, preserves user/operator precedence, and makes process-launch security boundaries explicit.

## Reproduction Assessment

Not applicable. This is a feature/API request, and source inspection confirms current main and v2026.6.6 do not expose `registerMcpServer` while existing MCP paths are bundle/user-config based.

## Solution Assessment

Unclear. A generic Plugin SDK seam is plausibly the right layer, but maintainers still need to decide the permanent API shape and the security model for plugin-managed stdio launches before implementation.
