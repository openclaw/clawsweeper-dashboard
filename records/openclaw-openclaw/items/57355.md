## Summary

Keep open: current main and the latest release still lack the requested native `registerMcpServer` Plugin SDK surface. Existing bundle/config MCP support covers static MCP packaging and saved server definitions, but not dynamic code-plugin lifecycle registration, and the closed prototype did not merge.

## What This Changes

The issue requests a Plugin SDK API for plugins to register managed MCP servers and inject them into embedded OpenAI-backed runs.

## Best Possible Solution

Design a narrow additive SDK contract for managed plugin MCP registration only if maintainers accept the execution-surface tradeoff, with explicit validation, ownership metadata, user-config precedence, workspace scoping, docs, and focused registry/runtime tests.

Unclear as an implementation path; the requested native SDK seam is plausible, but the best solution requires maintainer product and security approval because existing bundle/config MCP paths only cover part of the use case.
