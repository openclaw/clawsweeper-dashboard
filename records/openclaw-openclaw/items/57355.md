## Summary

Keep open: current main and v2026.6.10 still lack a native `registerMcpServer` Plugin SDK API, and the only implementation PR is closed unmerged. The request remains a valid feature/design item, but it needs maintainer product and security approval because it would add a public plugin process-launching surface.

## What This Changes

The issue requests a native Plugin SDK API for registering managed MCP servers so plugins can expose dynamic MCP-backed capabilities into embedded OpenAI-backed runs.

## Best Possible Solution

A maintainer-sponsored design should decide on a narrow, additive, validated Plugin SDK MCP registration contract, or explicitly keep dynamic MCP exposure outside core until the security model is settled.

## Reproduction Assessment

Not applicable; this is a feature request, not a report of broken existing behavior. Source inspection shows the requested SDK method is still absent from both current main and v2026.6.10 while existing MCP support is bundle/config based.

## Solution Assessment

Unclear until maintainer product and security review. A native Plugin SDK seam may be the right solution if existing bundle/config MCP cannot express dynamic account-state inventories, but the execution boundary needs explicit owner approval.
