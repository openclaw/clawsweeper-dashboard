## Summary

Keep this open: current main and the latest release still lack a native Plugin SDK `registerMcpServer` surface, and the only in-repo implementation attempt was closed unmerged. The request is clear and fits the plugin API direction, but it needs maintainer API, product, and security review because it would let native plugins register stdio MCP processes that OpenClaw launches for embedded runs.

## What This Changes

Request to add a native Plugin SDK `registerMcpServer` API for managed stdio MCP servers that can be injected into embedded OpenAI-backed runs.

## Best Possible Solution

Land a maintainer-approved additive SDK design that validates managed stdio MCP config, preserves user/operator precedence and workspace scoping, wires lifecycle cleanup through the registry/runtime, documents the security boundary, and adds focused SDK, loader, status, and embedded runtime coverage.

## Reproduction Assessment

Not applicable as a feature/API request rather than a bug. Source inspection is high-confidence that current main does not expose `registerMcpServer` in the Plugin SDK or embedded MCP registry path.

## Solution Assessment

Unclear until maintainer API and security review. A generic SDK seam is plausible for dynamic runtime-dependent MCP inventory, but the final design must avoid duplicating bundle MCP and must keep process launching explicit, validated, and owner-controlled.
