## Summary

Keep open: the timeout boundary is relevant to the linked non-TTY status regression, but the current branch can warn on normal empty-auth results and still needs the required changelog entry.

## What This Changes

The PR wraps provider auth resolution for `status --usage --json` in a timeout, returns empty providers on timeout, logs a warning, and adds unit coverage for a hanging auth resolver.

## Best Possible Solution

Land the auth-resolution timeout with an explicit timeout sentinel or result object, keep normal empty-auth results quiet, cover both paths in tests, and add the active-release changelog entry.

## Reproduction Assessment

Yes. The linked report gives a concrete `spawnSync` non-TTY reproduction, current main shows the unbounded auth await, and the PR-introduced false warning is reproducible by having `resolveProviderAuths` resolve to `[]`.

## Solution Assessment

No. Wrapping auth resolution is the right narrow boundary, but the warning must be driven by the actual timeout path rather than inferred after the promise has resolved.
