## Summary

Keep open: the PR addresses a real unbounded auth-resolution path, but the current timeout-warning inference mislabels legitimate empty-auth results and the user-facing CLI fix still needs a changelog entry.

## What This Changes

The PR wraps provider usage auth resolution in `withTimeout`, emits a timeout warning when it infers auth resolution timed out, and adds a unit test for a never-settling auth resolver.

## Best Possible Solution

Land the auth-resolution timeout with an explicit timeout sentinel or boolean, keep normal empty-auth results quiet, cover both paths in tests, and add the active-release changelog entry.

## Reproduction Assessment

Yes. The linked report gives a concrete `spawnSync` non-TTY reproduction for the original hang, current main shows the unbounded auth await, and the PR-introduced warning bug is reproducible by resolving auths to `[]` and observing `console.warn`.

## Solution Assessment

No. The timeout boundary is the narrow maintainable fix, but the diagnostic should track the timeout outcome directly instead of inferring it after the promise has resolved.
