## Summary

Keep this PR open. Current main still has the unbounded provider-auth phase that can block `status --usage --json`, but the PR head has a concrete correctness bug in its timeout-warning detection and is missing the required user-facing changelog entry.

## What This Changes

The PR wraps provider-usage auth resolution in a timeout, logs when it infers auth resolution timed out, and adds a unit test for a never-settling auth resolver.

## Best Possible Solution

Land the auth-resolution timeout, but track the timer outcome explicitly with a sentinel or boolean, keep normal empty-auth results quiet, cover both timeout and resolved-empty paths in tests, and add the 2026.4.30 changelog fix entry credited to the reporter/contributor.

## Reproduction Assessment

Yes. The linked report gives a concrete `spawnSync` reproduction for the original non-TTY hang, current main shows the auth await is unbounded, and the PR regression is reproducible with a minimal Promise-race check plus the existing empty-auth test expectation.

## Solution Assessment

No. The timeout boundary is the right narrow fix, but inferring timeout after `withTimeout` returns is not reliable; an explicit timeout sentinel or timer-branch boolean is the safer implementation.

## Review Findings
