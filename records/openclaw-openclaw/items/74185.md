## Summary

Keep this PR open: current main still awaits provider auth resolution without a timeout, so the reported status usage hang remains plausible, but the latest PR head introduces a false timeout warning for normal empty-auth results and lacks the required user-facing changelog entry.

## What This Changes

The PR wraps provider usage auth discovery in a timeout, logs when it infers auth resolution timed out, and adds a unit test for a never-settling auth resolver.

## Best Possible Solution

Land a narrow provider-auth timeout for status usage with reliable timeout classification, keeping normal empty-provider results quiet and adding the required changelog entry for the CLI regression fix.

## Reproduction Assessment

Yes. The linked report gives a concrete spawnSync reproduction for the original non-TTY status command hang, and current main shows the auth phase is still awaited without a timeout. The PR-specific regression is also reproducible by the resolved-empty auth path and the Promise.race ordering in the patch.

## Solution Assessment

No. Wrapping auth resolution in a timeout is the right direction, but this implementation infers timeout after the fact with a race that also flags normal empty auth resolution. A safer fix is an explicit timeout sentinel or timer-branch boolean plus tests for both timeout and resolved-empty paths.
