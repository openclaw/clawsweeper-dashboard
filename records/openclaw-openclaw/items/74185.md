## Summary

Keep open: the PR addresses the earlier review comments with a warning, tests, and changelog entry, but the timeout only races the auth promise and does not cancel auth work that can keep the CLI process alive.

## What This Changes

The branch wraps provider usage auth resolution in a timeout, warns and returns empty providers on auth timeout, adds regression tests, and adds a changelog fix entry.

## Best Possible Solution

Add cancellation-aware provider auth resolution, or avoid starting handle-owning auth refresh on this status path, and prove the CLI process exits within the usage timeout for active-handle OAuth or plugin auth hangs.

## Reproduction Assessment

Yes, at source level. The linked report gives a concrete non-TTY spawnSync reproduction, current main awaits resolveProviderAuths before any timeout, and the proposed patch still leaves a source-visible active-handle timeout path in OAuth refresh.

## Solution Assessment

No. Racing resolveProviderAuths with withTimeout is the right surface, but the implementation is incomplete because it returns before cancelling the underlying auth work that can keep the Node process alive.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Abort timed-out auth work before returning:** `src/infra/provider-usage.load.ts:106-110`
  - body: The timeout races `resolveProviderAuths`, but the auth work continues after the fallback. In the reported OpenAI Codex OAuth path, auth resolution can enter the OAuth manager's 120s refresh timer and pi-ai 0.73.0 refreshes with `fetch()` and no `AbortSignal`; since the CLI does not force `process.exit` after a successful status command, those handles can keep `status --usage --json` alive past the caller's subprocess timeout. Please cancel/abort the underlying auth work, or avoid starting handle-owning auth refresh on this path, and cover it with an active-handle/subprocess-style regression test.
  - confidence: 0.87
