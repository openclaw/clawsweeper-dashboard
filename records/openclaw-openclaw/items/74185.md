## Summary

Keep this PR open: the auth-resolution timeout is a useful fix for the non-TTY status hang, but the branch also changes status usage to suppress OAuth refreshes, which can hide expired-but-refreshable OAuth-backed usage and needs repair or explicit provider/auth maintainer acceptance before merge.

## What This Changes

The PR bounds provider usage auth resolution for status usage, preserves scoped agent lookup, and changes status usage auth to skip plugin auth without credential sources and avoid OAuth refreshes.

## Best Possible Solution

Land the auth-resolution timeout while preserving existing OAuth refresh behavior by default, or have provider/auth owners explicitly accept and document cached-only status usage with focused expired-token tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection confirms current main has no timeout around resolveProviderAuths and the PR proof shows a non-TTY status command completing after the patch, but I did not run a live expired-OAuth subprocess reproduction in this checkout.

## Solution Assessment

Is this the best way to solve the issue? No: wrapping auth resolution in the existing timeout is the right layer, but suppressing OAuth refresh for status usage is broader than needed and can remove existing OAuth usage visibility.
