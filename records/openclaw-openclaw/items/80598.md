## Summary

Keep open: this PR targets a valid backup temp-file leak, but the latest head still does not run the intended signal cleanup, exits interrupted backups as success, can break archive publication across filesystems, and lacks sufficient real interrupted-run proof.

## What This Changes

The PR moves backup temp archive creation into the staging temp directory, wires SIGINT/SIGTERM cleanup through `backupCreateCommand`, adds a temp-path regression test, and updates backup docs/changelog.

## Best Possible Solution

Keep the temp archive out of the output directory only with cross-device-safe publication, register a real synchronous cleanup closure at the CLI boundary, preserve signal exit codes, add focused regression coverage, and require redacted real interrupted-CLI proof.

## Reproduction Assessment

Yes. Source-backed reproduction is to run `openclaw backup create` long enough to enter the tar write, then terminate it before normal unwinding; current main writes an adjacent temp archive and only removes it in `finally`.

## Solution Assessment

No. Moving the temp archive under the staging temp root is directionally useful, but this patch needs a real cleanup closure, signal-derived exit codes, cross-device publication fallback, and latest-head real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass a real cleanup closure to onCleanup:** `src/infra/backup-create.ts:504`
  - body: `opts.onCleanup` is called with a setter-like callback instead of the synchronous cleanup function the CLI stores and invokes. On SIGINT/SIGTERM, `signalCleanup?.()` calls that setter with no handler, so neither `tempArchivePath` nor `tempDir` is removed before `process.exit`; the signal-cleanup part of the fix still does not work.
  - confidence: 0.97
- **[P2] Handle cross-device publication before moving temp archives:** `src/infra/backup-create.ts:499`
  - body: Moving `tempArchivePath` under the system temp root means it may be on a different filesystem than the requested output. `publishTempArchive` still starts with `fs.link` and does not fall back on cross-device link failures, so backups to an external mount or separate volume can now fail where the adjacent temp file path stayed on the output filesystem.
  - confidence: 0.88
- **[P2] Return signal exit codes from interrupted backups:** `src/commands/backup.ts:30`
  - body: The signal handler exits with status 0 for both SIGINT and SIGTERM, so an interrupted backup can look successful to scripts or supervisors even though no archive was completed. Match the existing CLI shutdown convention by returning signal-derived codes such as 130 for SIGINT and 143 for SIGTERM.
  - confidence: 0.9
