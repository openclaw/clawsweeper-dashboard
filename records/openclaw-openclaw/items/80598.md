## Summary

Keep open: the linked backup leak is valid and the temp-dir direction is plausible, but current PR head still has blocking cleanup, exit-code, and cross-device publication defects, and the supplied proof does not show latest-head interrupted CLI cleanup.

## What This Changes

The PR moves `openclaw backup create` temp archive creation into the staging temp directory, adds SIGINT/SIGTERM cleanup plumbing, updates backup docs/changelog text, and adds a temp-path regression test.

## Best Possible Solution

Revise the PR so backup creation keeps temp archive lifecycle in the staging root, passes a real synchronous cleanup closure, preserves SIGINT/SIGTERM exit semantics, handles cross-device publication fallback, and proves an interrupted CLI run with redacted real output.

## Reproduction Assessment

Yes, source-backed: current main writes the active tar output beside `outputPath` and removes it only during normal `finally` unwinding. Running `openclaw backup create` long enough to enter `tar.c`, then terminating it before unwinding, matches the linked orphaned `.tmp` report, though I did not perform a live signal run in this read-only review.

## Solution Assessment

No. Moving the temp archive out of the output directory is a reasonable direction, but this patch miswires signal cleanup, exits successfully on interruption, and can break publication when `/tmp` and the destination are on different filesystems.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass a real cleanup function to onCleanup:** `src/infra/backup-create.ts:504`
  - body: `onCleanup` receives the function the CLI will invoke on SIGINT/SIGTERM, but this passes a setter-style callback instead. The CLI stores that callback as `signalCleanup`; calling it with no argument does not remove `tempArchivePath` or `tempDir`, and the new `rmSync` import remains unused under `noUnusedLocals`.
  - confidence: 0.97
- **[P2] Handle cross-device publication after moving the archive:** `src/infra/backup-create.ts:499`
  - body: Now that `tempArchivePath` is under the staging temp directory instead of beside `outputPath`, `fs.link` can cross filesystems and fail with `EXDEV` for mounted backup destinations. `publishTempArchive` rethrows anything outside the current unsupported-link list, so the backup fails instead of falling back to the exclusive copy path.
  - confidence: 0.84
- **[P2] Return signal exit codes on interruption:** `src/commands/backup.ts:30`
  - body: The backup signal handler exits 0 for SIGINT/SIGTERM, so scripts and supervisors can treat an interrupted backup as successful even though no archive was produced. Use signal-derived codes such as 130 and 143, matching the existing CLI proxy shutdown path.
  - confidence: 0.9
