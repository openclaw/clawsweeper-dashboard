## Summary

Keep open: the PR addresses the right heartbeat/message-tool bug, but current head still applies same-route delivery metadata one payload at a time, so a multi-visible heartbeat result can suppress fallback text that was not proven delivered by the message tool.

## What This Changes

The PR threads route-scoped generic message-tool delivery evidence into heartbeat reply payloads and suppresses heartbeat fallback when same-route message-tool delivery is present, with focused auto-reply and heartbeat tests.

## Best Possible Solution

Keep the route-scoped suppression, but mark delivery evidence only after whole-result filtering proves there is exactly one visible plain fallback; multi-visible, rich, media, no-route, wrong-route, and wrong-thread cases should keep fallback delivery.

## Reproduction Assessment

Yes. The linked issue has a high-confidence source-level reproduction for duplicate heartbeat fallback after generic message-tool delivery, and the current PR defect is source-reproducible from the per-payload marking loop plus the runner's metadata-based suppression.

## Solution Assessment

No, not yet. Route-scoped suppression is the right fix direction, but the metadata decision needs to move to whole-result scope before merge so the suppression cannot hide unrelated visible fallback content.
