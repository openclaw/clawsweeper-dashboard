## Summary

This PR is still useful and should stay open, but the current head still has a message-delivery blocker: same-route evidence is attached per payload before the heartbeat runner makes the whole-result decision, so a multi-visible heartbeat reply can suppress legitimate fallback content.

## What This Changes

The PR threads route-scoped generic message-tool delivery evidence through heartbeat reply payload metadata and suppresses duplicate heartbeat fallback sends when that evidence is present.

## Best Possible Solution

Keep the route-scoped evidence idea, but make suppression a whole-result decision so only the single duplicate fallback is skipped and unrelated visible fallback payloads still deliver.

## Reproduction Assessment

Yes. Source inspection of the current PR head shows the remaining failure path: each visible heartbeat payload can be marked with same-route delivery evidence, and the runner suppresses delivery based on the last selected payload rather than the whole result.

## Solution Assessment

No, not yet. Route-scoped evidence is the right boundary, but per-payload marking is too early; the safer fix is to decide suppression after the whole heartbeat result is known.
