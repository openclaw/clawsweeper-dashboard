## Summary

Keep this PR open: it is a narrow fix for a real P1 heartbeat duplicate-delivery bug, but the rebuilt head introduces a blocking message-delivery regression for multi-payload heartbeat replies.

## What This Changes

The PR threads route-scoped generic message-tool delivery evidence through auto-reply payload metadata and uses it to suppress duplicate heartbeat fallback delivery.

## Best Possible Solution

Keep the route-scoped suppression design, but compute the heartbeat message-tool delivery marker against the whole heartbeat payload set so only the intended single plain fallback bypasses text matching.

## Reproduction Assessment

Yes for the review finding: source inspection shows the new branch marks each payload independently inside the dedupe loop, and runHeartbeatOnce suppresses the resolved heartbeat payload from that metadata. I did not run a live repro in this read-only review.

## Solution Assessment

No: the route-scoped evidence design is the right layer, but the current implementation applies the different-text shortcut per payload instead of across the whole heartbeat result. The safer fix is to preserve ordinary text dedupe whenever there is more than one outbound-capable heartbeat payload.
