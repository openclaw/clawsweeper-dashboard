## Summary

Keep open: the fix targets the right heartbeat duplicate-delivery path, but the proposed route evidence currently ignores the heartbeat thread/topic component, which can suppress fallback delivery for the wrong Telegram topic or threaded channel route.

## What This Changes

The PR adds route-scoped message-tool delivery metadata to heartbeat reply payloads and suppresses heartbeat fallback narration when a generic message tool delivery matched the reply route.

## Best Possible Solution

Keep the metadata-based suppression shape, but include thread/topic identity in the existing route matcher before suppressing fallback, then cover same-chat different-thread and matching-thread cases with focused tests.

## Reproduction Assessment

Yes, source-level: heartbeat delivery can carry threadId separately from OriginatingTo, while the PR's route evidence check ignores MessageThreadId and the new heartbeat branch returns before fallback delivery. A focused same-chat/different-thread test should reproduce the bad suppression without a live Telegram run.

## Solution Assessment

No, not yet. The metadata handoff is the maintainable direction, but the route check must include thread/topic identity before it is safe to treat generic message-tool delivery as owning the heartbeat route.
