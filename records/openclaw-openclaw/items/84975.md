## Summary

Keep open: the route-scoped evidence direction is right, but the current head still marks heartbeat delivery evidence inside a per-payload loop, so multi-visible fallback results can be suppressed wholesale instead of delivering legitimate remaining content.

## What This Changes

The branch threads route-scoped message-tool delivery metadata through heartbeat reply payloads and suppresses heartbeat fallback text when that metadata is present, with focused heartbeat/payload tests.

## Best Possible Solution

Keep the route-scoped evidence idea, but make suppression a whole-result decision so only a single duplicate fallback is skipped and unrelated visible fallback payloads still deliver.

## Reproduction Assessment

Yes. Source inspection of the current PR head shows the remaining failure path: each visible heartbeat payload can be marked with same-route delivery evidence, and the runner suppresses delivery based on the last selected payload rather than the whole result.

## Solution Assessment

No, not yet. Route-scoped evidence is the right boundary, but per-payload marking is too early; the safer fix is to decide suppression after the whole heartbeat result is known.
