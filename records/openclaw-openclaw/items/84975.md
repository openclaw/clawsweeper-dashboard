## Summary

Keep open: this PR is still the active candidate for the heartbeat duplicate-fallback bug, and it has real Telegram proof for the central same-route case, but the current head still marks delivery evidence per payload instead of across the whole heartbeat result, so multi-visible fallback content can be suppressed before merge.

## What This Changes

The branch adds route-scoped message-tool delivery metadata to heartbeat reply payloads and suppresses heartbeat fallback sends when the generic message tool already delivered to the same route, with focused regression coverage.

## Best Possible Solution

Keep the route-scoped metadata approach, but decide marker eligibility across the whole heartbeat result: only the single plain visible fallback case should suppress by route evidence, while multi-payload results should fall back to existing route-scoped text/media dedupe.

## Reproduction Assessment

## Solution Assessment

No as currently implemented. The route-scoped suppression direction is the right fix, but the marker must be computed against the whole heartbeat result rather than each one-payload loop iteration.
