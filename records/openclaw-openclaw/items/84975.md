## Summary

Keep open. Current main still routes heartbeat fallback text through the normal send path when no heartbeat tool response is present, while this PR adds route-scoped message-tool delivery evidence, focused tests, and sufficient Telegram proof for the linked duplicate-delivery bug at https://github.com/openclaw/openclaw/issues/84217; the remaining question is maintainer acceptance of the message-delivery contract.

## What This Changes

The PR threads route-scoped generic message-tool delivery evidence into heartbeat reply payloads so duplicate heartbeat fallback text is suppressed only for the same provider/account/target/thread route, with regression coverage and minor formatting-only changes.

## Best Possible Solution

Land the route-scoped suppression only if maintainers accept the same-route generic message-tool delivery contract, preserving no-route, wrong-route, wrong-thread, media-only, and multi-payload fallback coverage.

## Reproduction Assessment

Yes, by source inspection and linked production evidence: current main still sends heartbeat fallback text when a generic message tool delivered but no heartbeat tool response exists, matching the reported duplicate Telegram behavior. I did not run a live current-main reproduction in this read-only review.

## Solution Assessment

Yes, this is an acceptable narrow fix: it threads delivery evidence from the generic message-tool send through reply-payload metadata and keeps suppression route-scoped rather than globally dropping heartbeat text. The only remaining choice is whether maintainers accept that route-scoped generic message-tool text delivery should count as heartbeat delivery evidence.
