## Summary

Keep open: the PR is the right route-scoped fix direction for the heartbeat/message-tool duplicate fallback bug, but the current head still marks delivery evidence inside the per-payload loop, so a multi-visible-payload heartbeat result can be wrongly suppressed.

## What This Changes

The PR threads route-scoped generic message-tool delivery evidence through heartbeat reply payload metadata and uses it to suppress duplicate heartbeat fallback sends, with focused regression coverage.

## Best Possible Solution

Keep the route-scoped suppression design, but move the marker decision to whole-result scope so only one eligible visible fallback payload can be suppressed.

## Reproduction Assessment

Yes. From source inspection, a same-route generic message-tool send followed by two non-reasoning visible heartbeat fallback payloads causes each payload to be marked independently and lets the runner suppress the main fallback send.

## Solution Assessment

No, not yet. The route-scoped suppression direction is right, but the implementation should evaluate the whole reply result before marking delivery evidence.
