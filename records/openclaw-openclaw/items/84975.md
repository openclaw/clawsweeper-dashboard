## Summary

Keep this PR open: the route-scoped heartbeat suppression direction is useful and the central Telegram duplicate-fallback behavior has strong proof, but the current branch still has a blocking message-delivery bug when a heartbeat result contains multiple visible non-reasoning text payloads.

## What This Changes

The PR threads same-route generic message-tool delivery evidence through reply payload metadata and suppresses heartbeat fallback sends when that evidence is present.

## Best Possible Solution

Repair the PR so same-route message-tool evidence suppresses only the single visible fallback payload proven to be covered, while multi-visible heartbeat results continue to use text-match filtering for unrelated fallback content.

## Reproduction Assessment

Yes. A source-level reproduction is a heartbeat result with same-route message-tool text evidence plus two visible non-reasoning text payloads where only one matches the tool body; the PR marks the unrelated selected fallback and suppresses it.

## Solution Assessment

No. Route-scoped suppression is the right direction, but this implementation applies the marker inside the per-payload loop instead of deciding eligibility across the full heartbeat reply result, so it is not yet the safest fix.
