## Summary

Keep open: the PR is the active fix for a real duplicate Telegram heartbeat delivery bug, but the current patch can suppress a legitimate heartbeat payload when a heartbeat result has more than one text payload.

## What This Changes

The PR adds route-scoped message-tool delivery metadata to heartbeat reply payloads and suppresses heartbeat fallback delivery when that metadata reaches `runHeartbeatOnce`.

## Best Possible Solution

Keep the route-scoped evidence approach, but preserve normal text dedupe for multi-payload heartbeat results and only suppress fallback when the selected heartbeat payload is the single plain text fallback already covered by same-route message-tool delivery.

## Reproduction Assessment

Yes. The linked issue has production Telegram evidence, and source inspection shows current main still sends fallback text when generic message-tool delivery leaves no heartbeat-tool response.

## Solution Assessment

No. The route-scoped metadata direction is right, but this implementation marks each payload before preserving multi-payload text dedupe, so it is not yet the safest fix.
