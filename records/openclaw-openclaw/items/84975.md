## Summary

Keep open: this PR still owns the active fix for the open heartbeat duplicate-message issue, but the current patch should not merge because route-delivery evidence is applied per payload and can suppress legitimate heartbeat fallback content when more than one visible fallback payload remains.

## What This Changes

The PR threads route-scoped generic message-tool delivery metadata through heartbeat reply payloads and suppresses heartbeat fallback delivery when that evidence is present, with focused heartbeat and auto-reply tests.

## Best Possible Solution

Move heartbeat message-tool delivery marking to whole-result scope so same-route tool delivery suppresses only the single duplicate plain fallback while preserving any additional visible fallback content.

## Reproduction Assessment

Yes. The linked issue gives a production Telegram trace, and current-main source still shows generic message-tool output can leave `heartbeatToolResponse` absent while fallback text proceeds to `sendDurableMessageBatch`.

## Solution Assessment

No, not yet. The metadata-based route-scoped direction is right, but the current implementation marks evidence inside a per-payload loop instead of after whole-result filtering, which leaves a concrete message-drop edge case.
