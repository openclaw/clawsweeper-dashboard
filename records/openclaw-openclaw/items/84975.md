## Summary

Keep open: this active PR targets a real P1 heartbeat message-delivery bug, but the live head is not merge-ready because it only suppresses text-duplicate fallback and still leaves a generic route-dedupe path that can lazy-load bundled channel plugins.

## What This Changes

The PR threads route-scoped generic message-tool delivery metadata through reply payload construction and heartbeat dispatch, with tests, to suppress heartbeat fallback after matching message-tool sends.

## Best Possible Solution

Mark heartbeat fallback as delivered based on successful same-route generic message-tool delivery evidence rather than text equality, preserve wrong-route/no-route fallback behavior with focused tests, and keep generic route dedupe on loaded-channel or pure helpers.

## Reproduction Assessment

Yes, source-level. The linked issue provides production Telegram logs and current main shows the fallback text path still runs when a generic message tool, rather than heartbeat_respond, owns visible delivery.

## Solution Assessment

No. Route-scoped evidence is the right direction, but this PR gates suppression on text duplication and leaves a bundled-plugin fallback in generic route dedupe, so it is not yet the narrow maintainable fix.

Use the loaded-channel parser or a pure route-string fallback here.
