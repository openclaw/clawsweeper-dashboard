## What This Changes

This PR makes reply-capable chat delivery contexts emit empty successful background exec completion notifications by default while preserving explicit opt-out and non-chat silence.

## Best Possible Solution

Land a rebased, chat-scoped default that preserves explicit false and non-chat silence, with docs/help text that describes the context-dependent behavior and real transport proof of the resulting chat notification.

## Reproduction Assessment

Yes, at source level. Current main sets empty-success notification only for an explicit true value and the runtime skips empty successful completions otherwise, while the PR changes that default for routed chat contexts.

## Solution Assessment

Mostly yes, but incomplete. The narrow chat-channel default with explicit false preserved is a maintainable shape, but it needs conflict resolution, docs/config alignment, and real chat transport proof.
