## Summary

Keep open: current main still uses the explicit-only empty-success notification default, and this PR is a plausible narrow fix, but it still needs real chat/background-exec proof and docs/help alignment before merge.

## What This Changes

The PR makes reply-capable chat delivery contexts default `notifyOnExitEmptySuccess` on for empty successful background exec completions, adds regression cases, and adds a changelog entry.

## Best Possible Solution

Merge a rebased update only after documenting the chat-only default and explicit opt-out, with redacted real behavior proof from a reply-capable chat background exec run.

## Reproduction Assessment

Yes, source-level. Current main sets `notifyOnExitEmptySuccess` only when explicitly true and the runtime drops completed empty-output background exec notifications otherwise; the related discussion also includes a stable chat-session repro.

## Solution Assessment

Mostly yes if maintainers accept the chat-only notification policy. The PR is narrower than the broader default-on alternative because it keys off reply-capable delivery context, preserves explicit false, and keeps non-chat provider labels silent; the remaining safer path is docs/help alignment plus real proof.
