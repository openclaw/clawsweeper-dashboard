## Summary

Keep open: current main still suppresses empty-output successful background exec completion events unless `notifyOnExitEmptySuccess` is explicitly true, and this PR still carries a meaningful narrow fix for reply-capable chat contexts. It needs real behavior proof, docs/help alignment, and a rebase before it is merge-ready; the broader https://github.com/openclaw/openclaw/pull/46926 is related but not a merged superseding fix.

## What This Changes

The PR defaults empty-success background exec notifications on for reply-capable chat delivery contexts, preserves explicit opt-out and non-chat silence, adds regression tests, and updates the changelog.

## Best Possible Solution

Merge a rebased update that documents the conditional chat default, keeps explicit false as the opt-out, preserves non-chat silence, and includes redacted real behavior proof from a chat background exec run.

## Reproduction Assessment

Yes, by source inspection and related stable-version repro. Current main only enables empty-success notifications for explicit true, and the runtime drops completed empty-output events otherwise; the linked umbrella discussion includes a real chat repro for the same quiet-success gap.

## Solution Assessment

Mostly yes. The PR is narrower than defaulting all exec completions on because it keys the new behavior to reply-capable delivery context and preserves explicit false, but it still needs docs/help alignment, rebase, and real behavior proof.
