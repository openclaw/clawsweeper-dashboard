## Summary

Keep this PR open: current `main` still suppresses empty successful background exec completions unless `notifyOnExitEmptySuccess` is explicitly true, and the latest PR head appears to implement the narrowed chat-delivery default while addressing earlier opt-out and non-chat-provider review comments. The remaining concrete gap is user-facing docs/config help, which still state the default is always false.

## What This Changes

The PR changes the exec tool so reply-capable chat delivery contexts notify on quiet successful background completion by default, adds regression coverage for chat/non-chat/explicit opt-out cases, and adds a changelog entry.

## Best Possible Solution

Land the narrowed PR after updating the background exec docs and config help to explain that explicit false stays silent, non-chat contexts stay silent by default, and reply-capable chat delivery contexts wake on quiet success by default.

## Reproduction Assessment

Yes. On current main, a background `exit 0` or no-op command with chat delivery context but no explicit `notifyOnExitEmptySuccess` still stores the session flag as false and the runtime suppresses the empty completion event; the code and existing test helper show that path.

## Solution Assessment

Mostly yes. The latest PR branch is the narrow maintainable implementation because it uses normalized delivery context, requires a real gateway message channel plus target, preserves explicit false, and keeps generic provider labels silent; the safer completion is to align docs/config help before merge.
