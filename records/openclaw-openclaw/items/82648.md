## Summary

Keep open. The reported image-ref failure is source-reproducible and the PR includes real after-fix proof, but the branch is draft, the author is explicitly waiting for contract direction, and the current patch bakes Lossless/LCM storage into core while missing image formats already accepted by current media paths.

## What This Changes

The branch adds a `src/media` LCM `file_<16hex>` resolver, wires prompt image loading and the image tool to read `OPENCLAW_STATE_DIR/lcm-files`, and adds focused tests.

## Best Possible Solution

Keep the PR open and route the fix through a maintainer-approved generic or plugin-owned managed-media reference contract that prompt image loading and the image tool can consume while sharing the existing image format policy.

## Reproduction Assessment

Yes, source-reproducible. Current main has only `path` and `media-uri` prompt refs and resolves opaque non-URL tool inputs against the workspace, while the PR body shows after-fix terminal output for the normal loader path with a managed `lcm-files` image.

## Solution Assessment

No, not yet. The branch proves the failure, but hard-coding the Lossless/LCM `lcm-files` layout in shared core media resolution is not the narrowest maintainable contract without maintainer direction.

## Review Findings

- **[P2] Keep LCM storage out of core media resolution:** `src/media/lcm-file-reference.ts:25`
