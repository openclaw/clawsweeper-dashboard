## Summary

Keep open: the PR targets a plausible real LCM image-reference failure and includes after-fix terminal proof, but the current patch hard-codes Lossless/LCM storage into core media paths and misses some image formats OpenClaw already supports.

## What This Changes

The PR adds an LCM `file_<16hex>` resolver under `src/media` and wires prompt image loading plus the `image` tool to load matching images from `OPENCLAW_STATE_DIR/lcm-files`, with focused tests.

## Best Possible Solution

Resolve externalized LCM images through a generic or plugin-owned managed-media reference contract that prompt image loading and tools can consume, while reusing OpenClaw's existing image type policy and keeping Lossless storage conventions out of core.

## Reproduction Assessment

Yes, source-reproducible. Current main has no `file_<16hex>` managed-media resolver, so those tokens fall through to workspace/path handling, and the PR body shows a local after-fix loader run that succeeds with a managed `lcm-files` image.

## Solution Assessment

No. The branch demonstrates the failing path, but hard-coding `lcm-files` and LCM token parsing in core is not the narrowest maintainable boundary when context engines and hosted media resolution are plugin-owned surfaces.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep LCM storage out of core media resolution:** `src/media/lcm-file-reference.ts:25`
  - body: This adds a core resolver rooted at `stateDir/lcm-files`, which bakes the Lossless/LCM storage layout into the shared media pipeline. Context engines are plugin-registered and core already has a hosted-media resolver seam for plugin-owned media; this should be routed through a generic/plugin-owned media reference contract instead of adding a Lossless-specific core path.
  - confidence: 0.83

LCM refs for those already-supported uploads would still miss the managed resolver and fall back to the broken workspace lookup.
  - confidence: 0.78
