## Summary

Keep this PR open: it addresses a real, source-reproducible image-ref failure and includes usable after-fix proof, but the current implementation has blocking solution-fit issues around core/plugin ownership and duplicated image-format policy.

## What This Changes

The branch adds a core LCM `file_<16hex>` image resolver under `src/media`, wires prompt image loading and the `image` tool to use it before workspace fallback, and adds focused tests.

## Best Possible Solution

Normalize externalized LCM images into an existing or generic managed-media reference consumed by prompt loading and tools, with plugin-owned storage knowledge kept out of core and image format policy reused from the central media code.

## Reproduction Assessment

Yes, source-reproducible. Current main has no `file_<16hex>` managed-media resolver, so those refs do not match the prompt image contract and the image tool falls through to workspace-relative loading.

## Solution Assessment

No. The PR demonstrates the failing path, but hard-coding `lcm-files` in core and maintaining a private image extension list is not the narrowest maintainable solution.

## Review Findings

  - body: This resolver roots shared media loading at `stateDir/lcm-files`, which bakes a Lossless/LCM storage layout into core.   - confidence: 0.83

LCM refs backed by those supported image formats would still miss this resolver and fall back to the broken workspace lookup, so the resolver should reuse the central media/image policy.
  - confidence: 0.78
