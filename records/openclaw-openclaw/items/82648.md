## Summary

Keep open; the PR addresses a real source-reproducible LCM image-ref failure, but the current patch is not merge-ready because it adds an unscoped all-conversation managed-file lookup and bakes Lossless storage policy into core media resolution.

## What This Changes

The branch adds an LCM `file_<16hex>` resolver, wires prompt image loading and the image tool to read matching files from `OPENCLAW_STATE_DIR/lcm-files`, and adds focused tests.

## Best Possible Solution

Normalize LCM externalized images into a scoped managed-media reference while conversation context is still available, or route them through a plugin-owned resolver with enough scope, while reusing the shared image format policy.

## Reproduction Assessment

Yes, source-reproducible. Current `main` has no managed resolver for bare `file_<16hex>` refs, and the PR body provides terminal after-fix output through the normal prompt image loader with a temporary state directory.

## Solution Assessment

No. The proposed core resolver proves the failure mode, but a global `lcm-files` scan is not the safest maintainable contract because it lacks active conversation scoping and duplicates plugin/media boundaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Bind LCM refs to the active conversation:** `src/media/lcm-file-reference.ts:49-64`
  - body: The resolver walks every directory under `stateDir/lcm-files` and returns a unique matching `file_<16hex>` without any current conversation or session constraint. That means any prompt/tool input containing a known opaque ref can load an image from another LCM conversation while bypassing `workspaceOnly`; normalize to a scoped managed-media ref earlier or pass a caller-scoped allowed root instead.
  - confidence: 0.88
- **[P2] Keep Lossless storage out of core media resolution:** `src/media/lcm-file-reference.ts:25`
  - body: This hard-codes the Lossless/LCM `lcm-files` layout into shared core media code. Since context engines are plugin-registered and core already has hosted-media resolver plumbing, this should go through a generic or plugin-owned media-reference contract before adding a Lossless-specific core path.
  - confidence: 0.84
- **[P2] Reuse the shared image format policy:** `src/media/lcm-file-reference.ts:7`
  - body: The new resolver uses its own extension list, omitting currently supported prompt image formats like BMP, TIFF, HEIC, and HEIF while adding SVG. LCM refs for supported uploads would still miss this resolver, and unsupported formats get a different path than normal prompt images.
  - confidence: 0.8
- **[P3] Use toSorted for deterministic traversal:** `src/media/lcm-file-reference.ts:49`
  - body: The latest CI lint job fails on the added `entries.sort(...)` because this repo requires non-mutating `Array#toSorted()`. The PR head cannot pass checks until the traversal uses the accepted form.
  - confidence: 0.9
