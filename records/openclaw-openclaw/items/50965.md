## Summary

Keep this PR open. Current main still has the cron-remove false-success behavior, and the PR is a useful but incomplete implementation because it fixes only the service payload while leaving the user-visible CLI exit-status path unchanged.

## What This Changes

The PR changes cron service removal so missing stores or job ids return failure payloads, and successful removals persist, rearm, emit, and return the removed job name only after a matching id is found.

## Best Possible Solution

Finish the fix end to end: make cron removal return a typed not-found failure without persisting, rearming, or emitting on a no-op; update the exported result contract and relevant gateway/tool consumers; make `cron rm/remove/delete` exit non-zero on `!ok` or `!removed`; and add focused service plus CLI regression tests. If name-based removal is desired, add exact-name lookup deliberately and document it.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call cron removal with an id that is not in the store: current main returns `{ ok: true, removed: false }`, the gateway resolves the RPC frame as successful, and the CLI remove action prints the payload without setting a failing exit code.

## Solution Assessment

No, this PR is not the best complete fix as written. The service-layer direction is useful, but the narrow maintainable solution must also update the typed result contract, CLI exit behavior, and focused tests so the observable `cron remove` command no longer fails silently.
