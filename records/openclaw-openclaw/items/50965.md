## Summary

This PR should stay open because the proposed service-layer change does not yet fix the end-to-end silent-failure path and it changes a shared cleanup contract without updating the caller. The PR also still lacks after-fix real behavior proof from a real OpenClaw CLI/gateway run.

## What This Changes

The PR changes `src/cron/service/ops.ts` so `cron.remove` returns explicit failure payloads for unavailable stores and missing jobs and includes `jobName` on successful removal.

## Best Possible Solution

Complete the end-to-end cron removal contract: make missing-job removal observable to users and scripts, preserve or deliberately update scheduled cleanup idempotency, and cover the service, CLI, gateway, and scheduled-turn paths with focused tests.

## Reproduction Assessment

Yes, from source inspection. Current main returns `{ ok: true, removed: false }` for a loaded store with a missing id, and this branch would make the service payload fail while leaving CLI exit behavior and scheduled cleanup semantics unresolved.

## Solution Assessment

No. Returning richer service payloads is useful, but the proposed patch is incomplete until the CLI consumes failure payloads and scheduled cleanup semantics are preserved or intentionally changed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Propagate remove failures to the CLI exit code:** `src/cron/service/ops.ts:330-331`
  - body: The branch makes missing jobs a payload-level failure, but `cron.remove` is still returned in a successful gateway frame and the CLI only prints the payload. `openclaw cron rm/remove/delete` will still exit 0 for a missing job unless this command checks `ok`/`removed` and exits nonzero.
  - confidence: 0.94
- **[P2] Preserve scheduled cleanup idempotency:** `src/cron/service/ops.ts:330-331`
  - body: This changes an already-missing job from `{ ok: true, removed: false }` to `{ ok: false, reason: "job not found" }`, but scheduled-turn cleanup currently treats the old payload as successful cleanup. Update that caller or contract deliberately so already-cleaned scheduled turns do not become failures.
  - confidence: 0.86
