## Summary

Keep open: current main still has the cron remove false-success path, but this PR is incomplete because it changes only the service result and leaves the CLI exit status, focused coverage, and changelog unresolved. The prior Greptile and ClawSweeper comments identified the same end-to-end gap.

## What This Changes

The PR changes `src/cron/service/ops.ts` so `remove()` returns missing-store or missing-job failure payloads, skips timer/event work when no job is found, and returns the removed job name on success.

## Best Possible Solution

Complete end-to-end cron removal semantics: typed missing-job/store failures from the service, nonzero CLI exits for `!ok || !removed`, regression coverage across service/gateway/CLI behavior, and a changelog bullet.

## Reproduction Assessment

Yes. A high-confidence source-level path is to call `cron.remove` or `openclaw cron rm` with an id absent from the loaded store; current main can return a successful no-op payload and the CLI has no nonzero exit path for that payload.

## Solution Assessment

No. Returning richer service payloads is the right direction, but the PR is not the narrow complete fix until the CLI exit status, public result contract, regression tests, and changelog are updated.
