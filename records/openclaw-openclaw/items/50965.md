## Summary

Keep open: the PR targets a real cron remove bug, but the branch is incomplete because the CLI still would not exit nonzero for payload-level remove failures, scheduled-turn cleanup idempotency is not preserved, the typed contract is not updated, and after-fix real behavior proof is missing.

## What This Changes

The PR changes cron service removal to return failure payloads for missing or unavailable jobs and to only persist, rearm, and emit after finding a job.

## Best Possible Solution

Complete the cron removal contract end to end: distinguish user-requested missing removals, make the CLI exit nonzero on failed removals, preserve scheduled-cleanup idempotency, update the exported result type, and add focused coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: remove() can return ok:true with removed:false, the gateway frames that as a successful RPC, and the CLI prints the payload without setting a nonzero exit status.

## Solution Assessment

No. The service payload change is useful, but the maintainable fix needs the CLI exit behavior, scheduled-cleanup contract, exported result type, and focused tests to move together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Propagate remove failures to the CLI exit code:** `src/cron/service/ops.ts:330`
  - body: Returning ok:false here will not make openclaw cron rm/remove/delete fail at the process level because the gateway still sends a successful RPC frame and the CLI only prints the payload. Scripts checking command status will keep treating missing-job removal as success unless the CLI exits nonzero when ok or removed is false.
  - confidence: 0.94
- **[P2] Preserve scheduled-turn cleanup idempotency:** `src/cron/service/ops.ts:330`
  - body: This changes already-missing cron jobs from ok:true, removed:false to ok:false, but scheduled-turn cleanup currently treats the old payload as successful cleanup and has a contract test for that behavior. Update that caller or contract intentionally so already-cleaned scheduled turns do not become failures.
  - confidence: 0.88
- **[P3] Expose the new remove payload fields in the contract:** `src/cron/service/ops.ts:326-336`
  - body: The implementation now returns reason and jobName, but CronRemoveResult still omits those fields. Typed callers through CronServiceContract cannot rely on the new payload shape unless the exported result type is updated with the implementation.
  - confidence: 0.78
