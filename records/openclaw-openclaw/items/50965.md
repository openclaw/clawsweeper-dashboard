## Summary

Keep open because the reported failure is still source-reproducible on current main and this branch does not yet complete the user-facing removal contract.

## What This Changes

The PR changes `cron.remove` service results so unavailable stores and nonexistent jobs return explicit failure payloads and successful removals include `jobName`.

## Best Possible Solution

Complete the end-to-end cron removal contract: service reports missing jobs distinctly, the CLI exits nonzero for `!ok || !removed`, internal scheduled cleanup preserves intended idempotency, and focused tests cover each path.

## Reproduction Assessment

Yes, from source inspection. A missing id in a loaded store returns `{ ok: true, removed: false }`, the gateway wraps that payload in a successful RPC frame, and the CLI prints it without setting a nonzero exit code.

## Solution Assessment

No. Returning richer service payloads is useful, but the patch is incomplete until the CLI consumes failure payloads and scheduled-turn cleanup semantics are preserved or intentionally changed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Propagate remove failures to the CLI exit code:** `src/cron/service/ops.ts:330-331`
  - body: The branch makes missing jobs a payload-level failure, but `cron.remove` is still sent in a successful gateway frame and the CLI only prints the payload. Scripts will still exit 0 for missing-job removals until `cron rm/remove/delete` checks `ok`/`removed` and exits nonzero.
  - confidence: 0.94
