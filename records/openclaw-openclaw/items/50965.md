## Summary

Keep this PR open but not merge-ready: the service change addresses part of the false-success bug, but payload failures still do not affect the CLI exit code, scheduled-turn cleanup needs deliberate compatibility handling, and real behavior proof is missing.

## What This Changes

This PR changes `src/cron/service/ops.ts` so `cron.remove` returns failure payloads for unavailable stores or missing jobs, skips timer/event work when nothing is removed, and includes `jobName` on successful removal.

## Best Possible Solution

Complete the end-to-end cron removal contract: service reports missing jobs distinctly, the CLI exits nonzero for `!ok || !removed`, scheduled-turn cleanup preserves its intended already-missing behavior, and focused tests cover those paths.

## Reproduction Assessment

Yes, from source inspection. An absent id in a loaded store returns `{ ok: true, removed: false }` on current main, the gateway sends it in a successful response frame, and the CLI prints it without setting a nonzero exit code.

## Solution Assessment

No. Returning richer service payloads is useful, but the patch is incomplete until the CLI consumes payload failures and shared cleanup semantics/tests are updated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Propagate remove failures to the CLI exit code:** `src/cron/service/ops.ts:330-331`
  - body: This changes `remove()` to return `{ ok: false }` for a missing job, but `cron.remove` is still delivered in a successful gateway frame and the CLI command only prints the payload. Without checking `ok`/`removed` and exiting nonzero, scripts still treat missing-job removals as success.
  - confidence: 0.94
- **[P2] Preserve idempotent scheduled-turn cleanup:** `src/cron/service/ops.ts:330-331`
  - body: Returning `ok: false` for an already-missing job also changes the shared service contract used by scheduled-turn cleanup, where current code treats cleanup as successful only when the remove result has `ok: true` and existing tests document `{ ok: true, removed: false }` as the already-missing case. Keep that cleanup path idempotent, for example by handling a distinct `job not found` reason there, or update the contract deliberately.
  - confidence: 0.86
