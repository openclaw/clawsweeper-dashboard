## Summary

Keep open: current main still has the false-success cron removal path, but this PR only changes the service payload and leaves CLI exit behavior, scheduled-turn cleanup compatibility, typed contracts, merge conflicts, and real behavior proof unresolved.

## What This Changes

The PR changes `src/cron/service/ops.ts` so `remove()` returns `ok:false` with reasons when the store is unavailable or the job id is missing, and returns a job name on successful removal.

## Best Possible Solution

Complete cron removal end to end: make user-requested missing removals fail visibly, preserve idempotent scheduled cleanup, update the exported result type, add focused regression coverage, and include redacted terminal proof.

## Reproduction Assessment

Yes, at source level: current main's `remove()` returns `ok:true, removed:false` for a loaded-store miss, the gateway sends an ok RPC frame, and the CLI prints the payload without exiting nonzero. I did not run the CLI because this review was read-only.

## Solution Assessment

No. The service payload direction is useful, but the best fix must also carry failure semantics through CLI exit behavior, preserve scheduled-turn cleanup idempotency, update `CronRemoveResult`, and add regression/proof coverage.

  - confidence: 0.94
