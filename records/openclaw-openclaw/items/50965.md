## Summary

Keep this PR open: current main still has a source-reproducible cron remove false-success path, and the branch addresses part of it, but the patch is incomplete because CLI process status and scheduled-cleanup semantics are not updated and contributor real-behavior proof is missing.

## What This Changes

The PR changes `src/cron/service/ops.ts` so cron removal returns false payloads for unavailable or missing jobs and only persists, rearms, and emits after finding a job.

## Best Possible Solution

Complete the end-to-end cron removal contract by distinguishing user-requested missing removals, making the CLI exit nonzero for failed removals, preserving scheduled cleanup idempotency, aligning the exported result type, and adding focused coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: `remove()` returns a successful payload with `removed: false`, the gateway wraps it in a successful RPC frame, and the CLI prints it without setting a nonzero exit status.

## Solution Assessment

No. Returning richer service payloads is a useful piece, but the maintainable fix needs the CLI exit behavior, scheduled-cleanup contract, exported result type, and focused tests to move together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Propagate remove failures to the CLI exit code:** `src/cron/service/ops.ts:330`
  - body: Returning `{ ok: false }` from the service does not make `openclaw cron rm/remove/delete` fail at the process level: the gateway still sends a successful RPC frame and the CLI only prints the payload. Scripts checking command status will keep seeing success unless this command exits nonzero when `ok` or `removed` is false.
  - confidence: 0.94
- **[P2] Preserve scheduled cleanup idempotency:** `src/cron/service/ops.ts:330`
  - body: This changes an already-missing job from `{ ok: true, removed: false }` to `{ ok: false, reason: "job not found" }`, but scheduled-turn cleanup currently treats the old payload as successful cleanup and has a contract test for that behavior. Update that caller or contract intentionally so already-cleaned scheduled turns do not become failures.
  - confidence: 0.87
- **[P3] Expose the new remove result fields in the contract:** `src/cron/service/ops.ts:326-336`
  - body: The PR returns `reason` and `jobName`, but `CronRemoveResult` still omits those fields. Typed callers through `CronServiceContract` cannot rely on the new payload shape unless the exported result type is updated with the implementation.
  - confidence: 0.78
