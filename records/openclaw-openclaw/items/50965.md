## Summary

Keep open: the service-layer change is useful, but the PR still leaves the user-visible `cron rm/remove/delete` silent-failure path exiting successfully, and the external PR has no after-fix real behavior proof.

## What This Changes

The PR changes `src/cron/service/ops.ts` so cron removal returns missing-store or missing-job failure payloads and includes the removed job name on success.

## Best Possible Solution

Complete the end-to-end cron removal contract: service failure payloads for missing jobs, nonzero CLI exits for `!ok || !removed`, typed result coverage, and focused regression tests; treat name-based removal as a separate product choice if maintainers want it.

## Reproduction Assessment

Yes. A high-confidence source path is an absent cron job id in a loaded store: current main returns `{ ok: true, removed: false }`, the gateway sends a successful response frame, and the CLI prints the payload without exiting nonzero.

## Solution Assessment

No. Returning richer service payloads is the right direction, but the PR is not the narrow complete fix until CLI exit handling, the public result contract, and regressions are covered.
