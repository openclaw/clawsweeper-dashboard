## Summary

Keep open. The cron removal bug is still present on current main and the PR moves the service layer in the right direction, but it does not complete the user-visible fix because the CLI still prints the payload without exiting nonzero for `ok: false` / `removed: false`, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes `src/cron/service/ops.ts` so cron removal returns failure payloads for unloaded stores or missing jobs, skips timer/event work when nothing is removed, and includes the removed job name on success.

## Best Possible Solution

Complete the end-to-end cron removal contract: explicit missing-job failure from the service, nonzero CLI exits for `!ok || !removed`, updated public result typing, and focused service/CLI regression tests.

## Reproduction Assessment

Yes, from source inspection. An absent cron job id in a loaded store returns `{ ok: true, removed: false }` on current main, the gateway sends it in a successful response frame, and the CLI prints it without setting a nonzero exit code.

## Solution Assessment

No. Returning richer service payloads is the right direction, but the PR is not the narrow complete fix until the CLI propagates removal failures and the result contract/regression tests cover that behavior.
