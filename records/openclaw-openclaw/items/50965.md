## Summary

Keep open. The service-layer direction is useful, but this PR is not merge-ready because it does not make the CLI fail at the process level, changes an idempotent scheduled-turn cleanup contract, leaves the exported result type behind, lacks real behavior proof, and is currently conflicting.

## What This Changes

The branch changes `src/cron/service/ops.ts` so cron removal returns payload-level failure reasons for missing store/job cases and includes `jobName` on successful removal.

## Best Possible Solution

Complete cron removal end to end: make user-requested missing removals fail visibly, preserve idempotent internal cleanup, update the exported result contract and focused tests, and require redacted real CLI proof.

## Reproduction Assessment

Yes, at source level: current main's `remove()` returns `ok:true, removed:false` for a loaded-store miss, the gateway sends an ok RPC frame, and the CLI prints that payload without exiting nonzero. I did not run the CLI because this review was read-only.

## Solution Assessment

No. The service payload direction is useful, but the best fix must also carry failure semantics through CLI exit behavior, preserve scheduled-turn cleanup idempotency, update `CronRemoveResult`, and add regression/proof coverage.
