## Label Justifications

- `P2`: This is a real cron management bug with limited surface area, but it is not currently proven as an urgent release-blocking regression.
- `merge-risk: 🚨 compatibility`: The PR changes the meaning of missing cron removals from an idempotent ok payload to a failure payload without updating all existing callers.
- `merge-risk: 🚨 session-state`: Scheduled session-turn cleanup currently depends on already-missing cron jobs being cleanup-successful, so the PR can leave scheduler bookkeeping in a failed state.

## Summary

Keep open: the cron removal bug is valid on current main, but this PR is not merge-ready because it only changes the service payload while leaving the CLI false-success path, scheduled-turn cleanup semantics, exported contract type, and real behavior proof unresolved.

## What This Changes

The PR changes `src/cron/service/ops.ts` so `remove()` returns explicit failure payloads for unavailable stores or missing jobs and includes `jobName` on successful removal.

## Best Possible Solution

Complete the cron removal contract end to end: distinguish user-requested missing removals from cleanup idempotency, make the CLI exit nonzero for failed user removals, update the exported result type, and add focused regression coverage plus redacted terminal proof.

## Reproduction Assessment

Yes, by source inspection on current main: a loaded-store miss returns `ok:true, removed:false`, the gateway sends that as an ok RPC frame, and the CLI prints it without setting a failing exit code. I did not run the CLI because this review was constrained to read-only inspection.

## Solution Assessment

No. The service-layer payload change is useful, but it is not the best complete fix until the CLI exit behavior, scheduled cleanup contract, exported result type, and regression tests move with it.

Full review comments:

- **[P1] Propagate remove failures to the CLI exit code:** `src/cron/service/ops.ts:331`
  - body: Returning `ok:false` here does not make `openclaw cron rm/remove/delete` fail at the process level: the gateway still sends an ok RPC frame and the CLI only prints the payload. Scripts checking `$?` will still see success unless the CLI exits nonzero when `ok` or `removed` is false.
  - confidence: 0.94
- **[P2] Preserve scheduled cleanup idempotency:** `src/cron/service/ops.ts:331`
  - body: Scheduled-turn cleanup currently treats `ok:true, removed:false` as successful already-missing cleanup and has a contract test for that behavior. Returning `ok:false` for every missing job will make already-cleaned scheduled turns report failures unless this caller or contract distinguishes cleanup from user removal.
  - confidence: 0.88
- **[P3] Expose the new remove payload fields in the contract:** `src/cron/service/ops.ts:327-337`
  - body: The implementation adds `reason` and `jobName`, but `CronRemoveResult` still only exposes `ok` and `removed`. Typed callers through `CronServiceContract` cannot rely on the new payload shape unless the exported result type is updated with the implementation.
  - confidence: 0.78
