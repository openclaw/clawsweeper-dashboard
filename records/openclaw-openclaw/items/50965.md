## Summary

Keep this PR open, but it is not merge-ready: the service-layer change is useful, yet current main still has the reported behavior and the branch does not complete the CLI, cleanup-contract, type, test, or real-proof work needed for a safe fix.

## What This Changes

The PR changes `src/cron/service/ops.ts` so `remove()` returns explicit failure payloads for unavailable stores or missing jobs and includes a job name on successful removal.

## Best Possible Solution

Complete the cron removal contract end to end: make user-requested missing removals fail visibly, preserve idempotent scheduled cleanup, update the exported result type, add focused regression coverage, and include redacted terminal proof.

## Reproduction Assessment

Yes, by source inspection on current main: a loaded-store miss returns `ok:true, removed:false`, the gateway sends that as an ok RPC frame, and the CLI prints it without setting a failing exit code. I did not run the CLI because this review was constrained to read-only inspection.

## Solution Assessment

No. The service-layer payload change is useful, but it is not the best complete fix until CLI exit behavior, scheduled cleanup semantics, exported result types, regression tests, and real behavior proof move with it.

Full review comments:

- **[P1] Propagate remove failures to the CLI exit code:** `src/cron/service/ops.ts:331-333`
  - body: Returning `ok:false` here does not make `openclaw cron rm/remove/delete` fail at the process level: the gateway still sends an ok RPC frame and the CLI only prints the payload. Scripts checking `$?` will still see success unless the CLI exits nonzero when `ok` or `removed` is false.
  - confidence: 0.94
- **[P1] Preserve cleanup idempotency for scheduled turns:** `src/cron/service/ops.ts:331-333`
  - body: Scheduled-turn cleanup currently treats `ok:true, removed:false` as successful already-missing cleanup and has a contract test for that behavior. Returning `ok:false` for every missing job will make already-cleaned scheduled turns report failures unless this caller or contract distinguishes cleanup from user removal.
  - confidence: 0.9
- **[P3] Expose the new remove payload fields in the contract:** `src/cron/service/ops.ts:327-337`
  - body: The implementation adds `reason` and `jobName`, but `CronRemoveResult` still only exposes `ok` and `removed`. Typed callers through `CronServiceContract` cannot rely on the new payload shape unless the exported result type is updated with the implementation.
  - confidence: 0.78
