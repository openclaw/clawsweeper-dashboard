## Summary

Keep open. The service-layer change is useful, but the PR does not finish the observable `cron rm/remove/delete` failure path because the CLI still exits successfully for a payload-level remove failure, and the typed contract, regression tests, and required changelog entry are still missing.

## What This Changes

The PR changes `src/cron/service/ops.ts` so `remove()` returns not-found/store-unavailable failure payloads and only persists, rearms, emits, and returns a job name after finding a job.

## Best Possible Solution

Complete an end-to-end cron remove fix: typed missing-job/store failures in the service, nonzero CLI exits for `!ok` or `!removed`, preserved removed-event job snapshots, focused regression coverage, and a changelog bullet.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `cron.remove` or `openclaw cron rm` with an id absent from the store: current main returns a payload-level success/no-op and the CLI has no nonzero exit path for that payload.

## Solution Assessment

No. Returning richer service payloads is the right start, but the PR is not the narrow complete fix until the CLI exit status, result contract, regression tests, and changelog are updated.
