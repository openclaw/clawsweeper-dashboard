## Summary

Keep open: current main still has the service/CLI false-success path, but this PR is incomplete because it only changes the service payload and leaves the CLI exit code, scheduled cleanup semantics, exported result type, and real-behavior proof unresolved.

## What This Changes

The PR changes `src/cron/service/ops.ts` so cron removal returns explicit failure payloads for unavailable stores or missing jobs and includes the removed job name on success.

## Best Possible Solution

Complete the cron removal contract end to end: distinguish user-requested missing removals from cleanup idempotency, make the CLI exit nonzero on failed user removals, update the exported result type, and add focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: a loaded-store miss returns `ok:true, removed:false`, the gateway sends that as an ok RPC frame, and the CLI prints it without setting a failing exit code.

## Solution Assessment

No. The service-layer payload change is useful, but it is not the best complete fix until the CLI exit behavior, scheduled cleanup contract, exported result type, and regression tests move with it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Propagate remove failures to the CLI exit code:** `src/cron/service/ops.ts:331`
  - body: Returning `ok:false` for missing jobs will not make `openclaw cron rm/remove/delete` fail at the process level because the gateway still sends a successful RPC frame and the CLI only prints the payload. Scripts checking `$?` will continue to see success unless the CLI exits nonzero when `ok` or `removed` is false.
  - confidence: 0.94
- **[P2] Preserve scheduled cleanup idempotency:** `src/cron/service/ops.ts:331`
  - body: Scheduled-turn cleanup currently treats `ok:true, removed:false` as a successful already-missing cleanup and has a contract test for that behavior. Returning `ok:false` for every missing job will make already-cleaned scheduled turns report failures unless that caller or contract distinguishes cleanup from user removal.
  - confidence: 0.88
- **[P3] Expose the new remove payload fields in the contract:** `src/cron/service/ops.ts:327-337`
  - body: The implementation adds `reason` and `jobName`, but `CronRemoveResult` still only exposes `ok` and `removed`. Typed callers through `CronServiceContract` cannot rely on the new payload shape unless the exported result type is updated with the implementation.
  - confidence: 0.78
