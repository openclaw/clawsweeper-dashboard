## Summary

Keep open. The branch targets a real cron remove bug, but the patch is incomplete: the CLI still treats missing-job removal as a successful process, the new service failure changes scheduled-turn cleanup semantics, the typed contract is not updated, the head is conflicting, and no real behavior proof is present.

## What This Changes

The PR changes `src/cron/service/ops.ts` so `remove()` returns failure reasons when the store or job is missing and includes `jobName` only when a job is removed.

## Best Possible Solution

Complete cron removal end to end: make user-requested missing removals fail visibly, preserve internal idempotent cleanup, update result contracts and focused tests, refresh the branch, and add redacted real CLI proof.

## Reproduction Assessment

Yes, at source level: current main's `remove()` returns `ok:true, removed:false` for a loaded-store miss, the gateway sends an ok RPC frame, and the CLI prints that payload without exiting nonzero. I did not run the CLI because this review is read-only.

## Solution Assessment

No. The service payload direction is useful, but the best fix must also carry failure semantics through CLI exit behavior, preserve scheduled-turn cleanup idempotency, update `CronRemoveResult`, and add regression/proof coverage.
