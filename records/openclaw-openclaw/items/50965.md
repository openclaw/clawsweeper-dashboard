## Summary

Keep open; current main still has the silent missing-removal path, and this branch does not address the CLI exit status that makes the failure observable to scripts.

## What This Changes

The PR changes `src/cron/service/ops.ts` so `remove()` returns failure payloads for missing stores or job ids and only persists, rearms, emits, and returns a job name after finding a matching job id.

## Best Possible Solution

Finish the fix end to end: return a typed not-found failure without persisting, rearming, or emitting on no-op removal; make `cron rm/remove/delete` exit nonzero for `!ok` or `!removed`; preserve current removed-event job snapshots; and add focused service, CLI, and gateway regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `cron.remove` or `openclaw cron rm` with an id absent from the store: current main returns a successful RPC payload with `removed:false`, and the CLI remove action has no nonzero exit path for that payload.

## Solution Assessment

No. The service-layer direction is useful, but the PR is incomplete because the observable CLI path still reports shell success and the typed contract/tests are not updated.
