## Summary

Keep this PR open, but it should not merge as-is: the service-layer change is useful, yet the user-visible CLI exit path and scheduled-turn cleanup contract remain incomplete, and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The branch changes `src/cron/service/ops.ts` so `remove()` returns explicit false payloads for unavailable or missing jobs and only persists, arms, and emits after finding a job.

## Best Possible Solution

Complete the end-to-end cron removal contract: distinguish missing user-requested removals, make the CLI exit nonzero for failed removals, preserve scheduled cleanup idempotency, align the exported result type, and add focused service/CLI/contract coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: `remove()` returns `{ ok: true, removed: false }` for a missing job in a loaded store, the gateway wraps that payload in a successful RPC frame, and the CLI prints it without changing exit status.

## Solution Assessment

No. Returning richer service payloads is a useful piece, but the maintainable fix needs the CLI, scheduled cleanup contract, exported result type, and tests to move together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Propagate remove failures to the CLI exit code:** `src/cron/service/ops.ts:330-331`
  - body: Returning `{ ok: false }` from the service does not make `openclaw cron rm/remove/delete` fail at the process level: the gateway still sends a successful RPC frame and the CLI only prints the payload. Scripts checking command status will keep seeing success unless this command exits nonzero when `ok` or `removed` is false.
  - confidence: 0.94
- **[P2] Preserve scheduled cleanup idempotency:** `src/cron/service/ops.ts:330-331`
  - body: This changes an already-missing job from `{ ok: true, removed: false }` to `{ ok: false, reason: "job not found" }`, but scheduled-turn cleanup currently treats the old payload as successful cleanup and has a contract test for that behavior. Update that caller or contract intentionally so already-cleaned scheduled turns do not become failures.
  - confidence: 0.86
