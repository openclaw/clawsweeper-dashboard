## Summary

Keep open: the PR addresses a real cron removal failure path, but the service-only patch does not complete the user-visible fix and would regress an existing scheduled-turn cleanup contract.

## What This Changes

This PR changes `cron.remove` service behavior to return explicit failure payloads for missing or unavailable jobs and include `jobName` on successful removal.

## Best Possible Solution

Complete the end-to-end cron removal contract: make missing user-requested removals observable to users and scripts, preserve scheduled cleanup idempotency, and cover the service, CLI, gateway, and scheduled-turn paths with focused tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main returns `{ ok: true, removed: false }` for a missing job in a loaded store, and the CLI prints that payload without changing exit status. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. Returning richer service payloads is useful, but the patch is incomplete until the CLI consumes failures and scheduled cleanup either preserves or intentionally updates its idempotent missing-job behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Propagate remove failures to the CLI exit code:** `src/cron/service/ops.ts:330-331`
  - body: Returning `{ ok: false }` from the service does not make `openclaw cron rm/remove/delete` fail at the process level: the gateway still sends a successful RPC frame and the CLI only prints the payload. Scripts checking the command status will keep seeing success for a missing job unless this command exits nonzero when `ok`/`removed` is false.
  - confidence: 0.94
- **[P2] Preserve scheduled cleanup idempotency:** `src/cron/service/ops.ts:330-331`
  - body: This changes an already-missing job from `{ ok: true, removed: false }` to `{ ok: false, reason: "job not found" }`, but scheduled-turn cleanup currently treats the old payload as successful cleanup and has a contract test for that behavior. Update that caller or contract intentionally so already-cleaned scheduled turns do not become failures.
  - confidence: 0.86
