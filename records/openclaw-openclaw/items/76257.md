## Summary

Close as implemented on main: native Ollama dispatch no longer hardcodes the default provider URL; current code resolves `models.providers.<selectedProvider>.baseUrl`, tests cover multi-provider routing, and the fix shipped in `v2026.4.7`.

## What This Changes

Honor each custom Ollama provider's `baseUrl` when fallback dispatch selects provider-prefixed models such as `ollama-spark/qwen3:32b`.

## Best Possible Solution

Keep the shipped per-provider routing path; if this still reproduces, a fresh report should include the current OpenClaw version, effective provider config, selected provider id, and gateway dispatch logs.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction shape: configure two `api: "ollama"` providers with different `baseUrl` values and select or fall back to the second provider; current tests model this and verify the second URL is used.

## Solution Assessment

Yes. The current implementation is the narrowest maintainable fix because the Ollama plugin resolves the active provider's config at dispatch time and leaves loopback only as a fallback when no provider or model URL is configured.
