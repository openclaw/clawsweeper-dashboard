## Summary

Keep open. The PR addresses a real cron routing gap on current main, but the proposed rewrite is too broad: it would stamp any configured CLI backend over unrelated resolved providers, and the user-facing cron routing/cost fix is missing the required changelog entry.

## What This Changes

The PR changes cron model selection to stamp a configured CLI runtime provider for cron sessions and adds cron regression tests for CLI runtime provider stamping.

## Best Possible Solution

Land a narrowed version of this PR that applies the same provider/runtime compatibility rule used by interactive agent execution: use the CLI runtime stamp only for compatible provider selections, preserve unrelated provider overrides, cover the mismatch in tests, and add a single-line changelog fix entry.

## Reproduction Assessment

Yes. Current main can be reproduced by resolving a cron turn with `agentRuntime.id: "claude-cli"` and an Anthropic model: `resolveCronModelSelection` returns `provider: "anthropic"`, and the cron executor only dispatches CLI runs when the stamped provider is a CLI provider. The PR-specific regression can be reproduced with `agentRuntime.id: "claude-cli"` plus an `openai/gpt-5.5` cron payload, which the new block would stamp as `claude-cli`.

## Solution Assessment

No. The direction is right for Anthropic plus `claude-cli`, but the implementation should be provider/runtime-pair aware, for example by reusing `resolveCliRuntimeExecutionProvider` or equivalent mapping, rather than replacing every resolved provider with any configured CLI backend id.
