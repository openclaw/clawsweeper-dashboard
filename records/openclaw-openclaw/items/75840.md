## Summary

Keep open. The reported cron routing bug is real on current main, but this PR should not merge as-is because its provider rewrite is not provider/runtime-pair aware and it omits the required changelog entry for a user-facing fix.

## What This Changes

The PR changes cron model selection to stamp a configured CLI runtime provider for cron sessions and adds regression tests for CLI runtime provider stamping.

## Best Possible Solution

Keep the PR open and narrow the implementation so cron uses the same provider/runtime compatibility rule as interactive runs: stamp `claude-cli` only for compatible Anthropic selections, preserve unrelated provider overrides, cover the mismatch in tests, and add the changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `agentRuntime.id: "claude-cli"` with an Anthropic cron model and current main resolves `provider: "anthropic"`, so the executor does not enter the CLI path; the proposed patch also needs a mismatched `openai/gpt-5.5` override reproduction.

## Solution Assessment

No. The direction is right for Anthropic plus `claude-cli`, but the implementation should be provider/runtime-pair aware rather than replacing every resolved provider with any configured CLI backend id.
