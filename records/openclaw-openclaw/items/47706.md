Full review comments:

- **[P1] Use the existing memory concurrency config path:** `src/memory/manager-embedding-ops.ts:40`
  - body: Current main already exposes `agents.defaults.memorySearch.remote.nonBatchConcurrency` for non-batch memory indexing, so adding `OPENCLAW_EMBEDDING_CONCURRENCY` creates a second source of truth with unclear precedence. Port this through the existing config path or get an explicit maintainer decision for a new env API.
  - confidence: 0.84
- **[P2] Restore the global process stub after the ARM test:** `src/infra/platform-profile.test.ts:34`
  - body: `vi.stubGlobal("process", ...)` is not undone by `vi.restoreAllMocks()`, so later tests in the same worker can still see the mocked ARM process. Add `vi.unstubAllGlobals()` in teardown or avoid replacing the whole process object.
  - confidence: 0.93
- **[P2] Document the ARM low-profile threshold accurately:** `docs/platforms/raspberry-pi.md:255-257`
  - body: The implementation classifies ARM devices with less than 4 GB as `low`, but the docs say `low` is only `< 2GB` and `standard` is `2-8GB`. Update the table so Pi users can predict when reduced defaults and systemd limits apply.
  - confidence: 0.88
- **[P2] Clamp the low-profile resize grid before the first encode:** `src/media/image-ops.ts:30`
  - body: The low-profile grid still starts with `sideStart`, so a 2000px image still attempts the expensive 2000px encode before trying reduced sizes. Clamp the first low-profile entry to the low ceiling, such as `Math.min(sideStart, 1200)`, so the optimization actually starts lower.
  - confidence: 0.82
- **[P3] Validate systemd memory limit values before rendering:** `src/daemon/systemd-unit.ts:55-58`
  - body: The new `resourceLimits` strings are appended to the unit without the CR/LF checks used for description, args, env, and EnvironmentFile values. Keep these values closed over or validate them before rendering so future callers cannot inject extra unit lines.
  - confidence: 0.78
