## Summary

Keep open: current main and latest release v2026.6.10 still reuse one shared GitHub Copilot API token cache without tying the cache entry to the requesting GitHub credential, and the linked same-repository repair PR remains open, unmerged, and conflicting.

## What This Changes

The issue requests credential-aware GitHub Copilot API token caching so profile rotation after rate limits exchanges a token for the newly selected profile instead of reusing another account's cached token.

## Best Possible Solution

Land a focused fix in the canonical Copilot token helper that binds cached Copilot API tokens to the requesting GitHub credential, with regression coverage for profile switches and legacy cache misses.

## Reproduction Assessment

Yes. Source inspection shows resolveCopilotApiToken returns a usable shared cache entry before consulting the newly supplied githubToken, so two different profiles can receive the same cached Copilot API token.

## Solution Assessment

Yes. Credential-aware caching in src/plugin-sdk/provider-auth.ts is the narrowest maintainable fix; the linked PR’s approach should be refreshed against that current helper surface before landing.
