## Summary

Keep open. Current main still returns any usable shared Copilot token before using the selected GitHub token, and same-author PR #43659 remains open with closing syntax, so the issue should wait for that PR to be reviewed, landed, or closed.

## What This Changes

Request that GitHub Copilot API token cache entries be bound to the auth profile or GitHub token that produced them so profile rotation after rate limits exchanges a distinct Copilot token.

## Best Possible Solution

Add a profile-safe cache binding in `src/plugin-sdk/provider-auth.ts`, treat legacy unbound cache entries as misses, keep facades re-exporting the shared helper, and cover the cross-profile and legacy-cache cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `resolveCopilotApiToken` with a usable cached token, then call it with a different `githubToken`; current main returns the cached Copilot token before the new GitHub token can be exchanged.

## Solution Assessment

Yes. Binding cache reuse to a non-secret fingerprint or equivalent profile-safe key in the shared helper is the narrow maintainable fix; PR #43659 appears to target that direction but is still open.
