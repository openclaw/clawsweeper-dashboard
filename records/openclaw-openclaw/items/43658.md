## Summary

Keep open. Current origin/main and the latest release still use a single unowned GitHub Copilot API-token cache, so profile rotation can reuse a token minted from a different GitHub PAT; the open linked PR is the current fix path.

## What This Changes

The issue reports that GitHub Copilot auth-profile rotation reuses one shared Copilot API-token cache instead of minting a token for the newly selected GitHub PAT.

## Best Possible Solution

Land or replace the linked provider-auth fix so cached Copilot API tokens are bound to the selected GitHub token/profile, legacy unbound cache entries miss once, and focused regression tests cover same-profile hits plus cross-profile misses.

## Reproduction Assessment

Yes. Source inspection on current origin/main shows a valid shared cache entry in src/plugin-sdk/provider-auth.ts returns before params.githubToken is used, while the GitHub Copilot provider passes the selected ctx.apiKey into that helper.

## Solution Assessment

Yes. The narrow maintainable fix is to make the centralized provider-auth Copilot cache owner-aware while preserving integrationId and Copilot-Integration-Id behavior; the linked PR should be reviewed, ported, landed, or replaced rather than closing this issue early.
