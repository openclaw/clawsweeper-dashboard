- `impact:auth-provider`: The issue directly affects GitHub Copilot token selection and auth-profile rotation after rate limits.

## Summary

Keep open: current main and v2026.5.20 still use one ownerless GitHub Copilot API-token cache entry before the rotated profile's GitHub token is considered, and the linked fixing PR is still open rather than merged.

## What This Changes

Bind GitHub Copilot's cached API token to the selected auth profile or GitHub token so rate-limit rotation actually uses the next account.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/43659 with an owner-bound cache fix in `src/plugin-sdk/provider-auth.ts` that covers same-profile hits, cross-profile misses, legacy unowned cache entries, and preserves current Copilot integration-id/header semantics.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: configure two GitHub Copilot profiles, let profile A populate the shared cache, then select profile B while the cache is still usable; current main can return A's cached Copilot API token before using B's GitHub token for exchange.

## Solution Assessment

Yes for the requested direction. The narrow maintainable fix is to bind the centralized Copilot token cache to the selected GitHub token or profile, but this issue should stay open until the linked implementation merges or is replaced.
