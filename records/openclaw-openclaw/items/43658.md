## Summary

Keep open. Current main and the latest release still use a shared, unowned GitHub Copilot API-token cache, and the open linked fix PR is the active path rather than a reason to close this issue early.

## What This Changes

Report that GitHub Copilot auth-profile rotation is defeated because all profiles can reuse one cached Copilot API token that is not bound to the selected GitHub token/profile.

## Best Possible Solution

Land or replace the linked provider-auth fix so cached Copilot API tokens are bound to the selected GitHub token/profile, legacy unbound cache entries miss once, and focused regression coverage checks same-profile hits plus cross-profile misses.

## Reproduction Assessment

Yes, with source-level confidence: a valid shared Copilot cache entry in src/plugin-sdk/provider-auth.ts is returned before the rotated params.githubToken is used. I did not establish a live quota/rate-limit run.

## Solution Assessment

Yes, the narrow maintainable fix is to make the centralized Copilot token cache owner-aware while preserving the existing integrationId and Copilot-Integration-Id behavior. The open linked PR should be reviewed, ported, landed, or replaced rather than closing this issue early.
