## Summary

Keep open. Current main and the latest release still use a single unowned GitHub Copilot API-token cache, and an open linked PR with closing syntax is the active fix path rather than a reason to close the issue early.

## What This Changes

The issue reports that GitHub Copilot auth-profile rotation reuses one cached Copilot API token across distinct GitHub PAT profiles, nullifying rate-limit failover.

## Best Possible Solution

Land or replace the linked provider-auth fix so cached Copilot API tokens are bound to the selected GitHub token/profile, legacy unbound cache entries miss once, and focused regression tests cover same-profile hits plus cross-profile misses.

## Reproduction Assessment

Yes, with source-level confidence: a valid shared Copilot cache entry in src/plugin-sdk/provider-auth.ts is returned before the rotated params.githubToken is used. I did not establish a live quota run.

## Solution Assessment

Yes. The narrow maintainable fix is to make the centralized Copilot token cache owner-aware while preserving integrationId and Copilot-Integration-Id behavior; the open linked PR should be reviewed, ported, landed, or replaced.
