## Summary

Keep open. Current main still has the reported GitHub Copilot cache behavior: the shared Copilot token helper reuses any valid cached Copilot API token before checking the currently selected GitHub token/profile, and this issue is paired with open same-author PR #43659.

## What This Changes

Bind cached GitHub Copilot API tokens to the selected GitHub auth profile/token so profile rotation after rate limits exchanges a distinct Copilot token.

## Best Possible Solution

Update the shared Copilot token helper so a cache hit requires a binding to the currently selected GitHub token/profile, treat legacy unbound cache entries as refresh misses, and add focused regression coverage through the active GitHub Copilot plugin path before closing this issue.

## Reproduction Assessment

Yes. A high-confidence focused reproduction is to call `resolveCopilotApiToken` with a valid cached Copilot token produced for one GitHub token and then pass a different `params.githubToken`; current main returns the cached token without exchanging the second token.

## Solution Assessment

Yes. Binding the cache to a non-secret fingerprint or equivalent profile-safe key in the shared helper is the narrowest maintainable fix; PR #43659 appears to target that direction, but it still needs maintainer review before this issue should close.
