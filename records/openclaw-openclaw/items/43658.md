## Summary

Keep open. Current main and the latest release still share one GitHub Copilot API-token cache entry without a profile or GitHub-token ownership check, and the linked open fix PR remains the active path rather than a reason to close this issue before merge.

## What This Changes

The issue reports that rotating between multiple GitHub Copilot token auth profiles still reuses the first profile's cached Copilot API token.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/43659 with an owner-aware cache fix in src/plugin-sdk/provider-auth.ts, preserving integrationId and Copilot-Integration-Id behavior while adding same-profile hit, cross-profile miss, and legacy-cache miss regression coverage.

## Reproduction Assessment

Yes, source-level: src/plugin-sdk/provider-auth.ts returns a valid shared cached Copilot token before using params.githubToken, while the GitHub Copilot provider passes the selected ctx.apiKey into that helper. I did not run a live multi-profile quota/rate-limit rotation scenario.

## Solution Assessment

Yes, the narrow maintainable fix is to bind the centralized Copilot token cache to the selected GitHub token or profile, treat legacy unowned cache entries as misses, and keep the current Copilot integration identity contract.
