## Summary

Keep open: current main and v2026.6.6 still use a single integration-scoped Copilot token cache with no GitHub-account fingerprint, so the reported rotation bug remains. The open closing PR is the fix path, not a reason to close this issue before merge.

## What This Changes

The issue reports that rotating between multiple GitHub Copilot auth profiles still reuses one shared cached Copilot API token, binding all profiles to the rate-limited account.

## Best Possible Solution

Patch the canonical `src/plugin-sdk/provider-auth.ts` helper so Copilot token cache entries are scoped to the GitHub token or account, then update or replace the linked PR with regression coverage for same-token cache hits and different-token cache misses.

## Reproduction Assessment

Yes at source level: current main returns a valid cached Copilot API token before consulting `params.githubToken`, while session auth-profile code can rotate to another profile. I did not perform a live two-account Copilot rate-limit run.

## Solution Assessment

Yes, scoping the Copilot API-token cache by a non-secret GitHub-token or account fingerprint at the canonical helper is the narrow maintainable fix. Changing profile rotation alone would leave the token-exchange cache account-blind.
