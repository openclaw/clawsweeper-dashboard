## Label Justifications

- `P2`: The defect defeats multi-profile Copilot rate-limit rotation but has limited blast radius and does not make single-profile Copilot usage universally unavailable.
- `impact:auth-provider`: The issue concerns provider credential selection and reuse of a cached token belonging to a different auth profile.

## Summary

The reported bug remains in current main and the latest release: Copilot cache validity is tied to the integration id and expiry, not the selected GitHub credential. The issue should remain open while the linked repair PR is updated for the consolidated provider-auth implementation.

## What This Changes

Ensure GitHub Copilot token caching respects auth-profile rotation instead of reusing one account’s cached API token for another profile.

## Best Possible Solution

Update the canonical helper in `src/plugin-sdk/provider-auth.ts` so cached Copilot tokens are reusable only for the credential that produced them, add focused same-credential, different-credential, and legacy-cache tests, and retarget the linked PR without recreating duplicate helpers.

## Reproduction Assessment

Yes from source: populate the shared cache with one GitHub credential, then call the helper with another credential while the entry remains usable; the second call returns from cache before using the new credential. A live two-account rate-limit test was not performed.

## Solution Assessment

Yes, credential-bound cache identity is the narrow maintainable fix, but it must be implemented once in the consolidated provider-auth helper rather than in the obsolete duplicate files currently changed by the linked PR.
