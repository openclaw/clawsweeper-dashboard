## Label Justifications

- `P2`: The defect defeats multi-profile Copilot rate-limit rotation but has limited blast radius and does not make single-profile Copilot usage universally unavailable.
- `impact:auth-provider`: The issue concerns provider credential selection and reuse of a cached token belonging to a different auth profile.

## Summary

Keep open: current main and v2026.6.6 still use a single Copilot API-token cache entry keyed only by path, integration id, and expiry, so a rotated GitHub Copilot auth profile can still reuse the prior account's cached Copilot token. The open closing PR is the active fix path, so this issue should stay open until that PR is rebased/ported or replaced and merged.

## What This Changes

The issue requests that GitHub Copilot API-token caching respect the selected auth profile so rate-limit rotation uses the next account's token.

## Best Possible Solution

Implement credential-bound Copilot token cache reuse once in `src/plugin-sdk/provider-auth.ts`, keep wrappers thin, add cross-profile and legacy-cache regression coverage, then land or replace the linked PR.

## Reproduction Assessment

Yes at source level: current main returns a usable cached Copilot API token before account identity from `params.githubToken` affects the exchange path. I did not run a live two-account Copilot rate-limit test.

## Solution Assessment

Yes: scoping the canonical Copilot token cache by a non-secret GitHub-token or account identity is the narrow maintainable fix. Patching only the rotation layer would leave the token-exchange cache account-blind.
