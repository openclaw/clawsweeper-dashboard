## Summary

Keep open. Current main and the latest release still use a shared, profile-unowned GitHub Copilot API-token cache, and the linked open fix PR is the active path rather than a reason to close this issue early.

## What This Changes

The issue reports that GitHub Copilot auth-profile rotation after rate limits still reuses one cached Copilot API token across profiles.

## Best Possible Solution

Land or replace the linked provider-auth fix so cached Copilot API tokens are bound to the selected GitHub token/profile, legacy unowned cache entries miss once, and focused regression coverage checks same-profile hits plus cross-profile misses.

## Reproduction Assessment

Yes, with source-level confidence: a valid shared Copilot cache entry in src/plugin-sdk/provider-auth.ts is returned before the rotated params.githubToken is used. I did not establish a live quota/rate-limit rotation run.

## Solution Assessment

Yes, the narrow maintainable fix is to make the centralized Copilot token cache owner-aware while preserving the existing integrationId and Copilot-Integration-Id behavior. The open linked PR should be reviewed, ported, landed, or replaced rather than closing this issue early.
