- `impact:auth-provider`: The issue concerns GitHub Copilot auth profile routing and cached token reuse across accounts.

## Summary

Keep open: current main and v2026.6.6 still use a single account-blind Copilot API token cache, while the closing fix at https://github.com/openclaw/openclaw/pull/43659 is still open and conflicting.

## What This Changes

The issue reports that rotating between multiple GitHub Copilot auth profiles still reuses one cached Copilot API token, defeating rate-limit failover.

## Best Possible Solution

Land or replace the linked fix with credential-bound cache reuse in src/plugin-sdk/provider-auth.ts, keep wrappers thin, add regression coverage, and require redacted real two-profile proof.

## Reproduction Assessment

Yes at source level: a valid cached Copilot API token is returned before params.githubToken is used, so a different selected profile can receive the prior profile's token. I did not run a live two-account quota scenario.

## Solution Assessment

Yes on direction: credential-bound cache identity in the canonical provider-auth helper is the narrow maintainable fix. The linked PR has the right root-cause direction but must be reconciled with current main and proved before this issue closes.
