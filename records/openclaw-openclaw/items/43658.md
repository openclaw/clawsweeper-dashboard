- `impact:auth-provider`: The failure is in GitHub Copilot auth-profile token exchange and provider credential routing.

## Summary

Keep open: current main and the latest release still reuse one GitHub Copilot token cache entry without tying it to the requesting GitHub credential, and https://github.com/openclaw/openclaw/pull/43659 is the linked open repair candidate.

## What This Changes

The issue asks OpenClaw to make GitHub Copilot auth-profile rotation exchange and cache Copilot API tokens per selected GitHub credential instead of reusing one shared token.

## Best Possible Solution

Land a focused Copilot token-cache fix that scopes cached Copilot API tokens to the requesting GitHub credential or profile, then close this issue through the linked repair PR or a narrow replacement.

## Reproduction Assessment

Yes at source level: current main returns a usable shared Copilot token cache entry before using `params.githubToken`, so a rotated profile can receive the previous profile's Copilot API token. I did not run a live two-account Copilot scenario in this read-only review.

## Solution Assessment

Yes: the right fix is in the Copilot token exchange cache because rotation already passes a selected GitHub token into `resolveCopilotApiToken`. Changing only the rotation layer would still let a valid shared cache entry bypass the selected credential.
