## Summary

Keep open. Current main and the latest release still use one shared, unowned GitHub Copilot API-token cache, and the open closing PR at https://github.com/openclaw/openclaw/pull/43659 is the active fix path rather than a reason to close the issue early.

## What This Changes

The issue asks OpenClaw to make GitHub Copilot API-token caching respect auth-profile rotation so a rotated profile exchanges its own GitHub token instead of reusing another profile's cached Copilot token.

## Best Possible Solution

Land or replace the linked provider-auth fix so cached Copilot API tokens are bound to the selected GitHub token/profile, legacy unbound cache entries miss once, and focused regression coverage checks same-profile hits plus cross-profile misses.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a valid shared cache entry in src/plugin-sdk/provider-auth.ts is returned before the rotated params.githubToken is used. I did not establish a live quota/rate-limit run.

## Solution Assessment

Yes. The narrow maintainable fix is to make the centralized Copilot token cache owner-aware while preserving integrationId and Copilot-Integration-Id behavior; the linked PR should be reviewed, rebased/ported, landed, or replaced.
