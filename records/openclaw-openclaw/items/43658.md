## Summary

Keep open: current main and v2026.6.6 still use a single Copilot API token cache that is not tied to the selected GitHub credential, and the linked closing PR is still open and conflicting.

## What This Changes

The issue reports that GitHub Copilot auth profile rotation is ineffective because all profiles can reuse one cached Copilot API token.

## Best Possible Solution

Land or replace the linked fix so the Copilot token cache is scoped to the GitHub credential that produced it across the current provider-auth helper and its compatibility barrels.

## Reproduction Assessment

Yes, at source level: a valid cached Copilot token is returned before the newly selected params.githubToken is used, so a rotated profile can receive the previous profile's Copilot API token. I did not run a live two-account Copilot quota scenario.

## Solution Assessment

Yes: fixing cache ownership at resolveCopilotApiToken is the narrow maintainable layer because every current Copilot runtime path goes through that helper or its barrels. The linked PR needs to be rebased/reviewed against the current provider-auth location before merge.
