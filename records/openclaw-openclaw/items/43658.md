## Label Justifications

- `P2`: This is a real provider-specific multi-profile auth bug with limited blast radius, not a core runtime outage.
- `impact:auth-provider`: The issue directly affects GitHub Copilot token selection and auth-profile rotation after rate limits.

## Summary

Keep open. Current main and v2026.5.20 still use one shared GitHub Copilot token cache that can be returned before the rotated profile's GitHub token is used, and the active closing implementation is still the open https://github.com/openclaw/openclaw/pull/43659.

## What This Changes

The issue reports that GitHub Copilot multi-profile rotation reuses one cached Copilot API token, so a rotated profile can keep using the rate-limited account's token.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/43659 with an owner-bound Copilot token cache fix in src/plugin-sdk/provider-auth.ts, covering same-profile cache hits, cross-profile misses, and legacy unowned cache entries.

## Reproduction Assessment

Yes, source-level: current main can return a usable shared Copilot token before using the rotated params.githubToken. I did not run a live multi-account rate-limit scenario.

## Solution Assessment

Yes. The narrow maintainable fix is to bind the centralized Copilot token cache to the selected GitHub token or profile and keep this issue open until the linked implementation path merges or is replaced.
