- `impact:auth-provider`: The failure is in Copilot credential-to-token routing after auth profile rotation.

## Summary

Keep open: current main and latest release v2026.6.10 still return a usable shared GitHub Copilot API-token cache before checking the requesting GitHub token, and the open same-repository repair PR is unmerged and conflicting.

## What This Changes

The issue asks for GitHub Copilot API-token caching to be scoped to the selected auth profile so rotation after rate limiting uses the next account's token.

## Best Possible Solution

Land or replace the focused fix in src/plugin-sdk/provider-auth.ts so cached Copilot API tokens are bound to the requesting GitHub credential, with regression coverage for same-profile hits, profile switches, and legacy cache misses.

## Reproduction Assessment

Yes at source level: current main returns a usable shared Copilot token cache entry before using the newly selected githubToken, so two profiles can receive the same cached Copilot API token. I did not run a live two-account Copilot flow in this read-only pass.

## Solution Assessment

Yes. Credential-aware cache validation in the canonical provider-auth helper is the narrowest maintainable fix; the linked repair PR should be refreshed against that current helper surface before landing.
