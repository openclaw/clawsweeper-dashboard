- `impact:auth-provider`: The issue directly affects token selection and auth-profile rotation for the GitHub Copilot provider.

## Summary

Keep open. Current main and the latest release still reuse one GitHub Copilot API-token cache entry without binding it to the selected auth profile or GitHub token, and the open linked fix PR is the implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue asks for GitHub Copilot API-token caching to respect rotated auth profiles so each GitHub token receives its own Copilot API token.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/43659 with an owner-aware cache fix in src/plugin-sdk/provider-auth.ts, preserving the Copilot integration identity behavior and adding same-profile hit, cross-profile miss, and legacy-cache miss regression coverage.

## Reproduction Assessment

Yes, source-level: resolveCopilotApiToken checks and returns the shared cached Copilot token before using the selected ctx.apiKey from the GitHub Copilot provider. I did not run a live multi-profile rate-limit scenario.

## Solution Assessment

Yes. The narrow maintainable fix is to bind the centralized Copilot token cache to the selected GitHub token or profile, treat legacy unowned cache entries as misses, and keep the current Copilot integration identity contract.
