## Summary

Keep open: current main and v2026.6.6 still return a usable shared Copilot API token before checking which GitHub credential selected profile rotation, and the open closing PR is still the active fix path rather than merged provenance.

## What This Changes

Ensure GitHub Copilot token caching is bound to the selected auth profile or GitHub PAT so rate-limit rotation exchanges a distinct Copilot API token.

## Best Possible Solution

Patch the canonical provider-auth resolver to reject cache hits from a different GitHub credential, keep the current wrappers thin, add cross-profile and legacy-cache regression coverage, and land or replace the linked fix PR.

## Reproduction Assessment

Yes, at source level: the current resolver returns a valid shared cached Copilot token before using the newly selected params.githubToken. I did not run a live two-account Copilot quota scenario.

## Solution Assessment

Yes, the best fix is in the canonical provider-auth token resolver because all current Copilot token wrappers route through it. The open PR's intended credential-bound cache direction is right, but it must be current-main compatible and proved in a real setup before merge.
