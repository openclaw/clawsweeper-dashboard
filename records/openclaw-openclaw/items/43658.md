## Summary

Keep open. Current main still returns a valid shared GitHub Copilot API-token cache entry before checking the rotated GitHub token, and the linked open fix PR is the active path rather than a reason to close this issue before merge.

## What This Changes

The issue asks for GitHub Copilot API-token caching to be bound to the selected auth profile so profile rotation after rate limits uses the next account’s token.

## Best Possible Solution

Land or replace the linked provider-auth fix so cached Copilot API tokens are bound to the selected GitHub token/profile, legacy unowned cache entries miss once, and regression coverage checks same-profile hits plus cross-profile misses.

## Reproduction Assessment

Yes, with source-level confidence: a valid shared Copilot cache entry in src/plugin-sdk/provider-auth.ts is returned before the rotated params.githubToken is used. I did not establish a live quota/rate-limit rotation run.

## Solution Assessment

Yes, the narrow maintainable fix is to make the centralized Copilot token cache owner-aware while preserving the existing integrationId and Copilot-Integration-Id behavior. The open linked PR should be reviewed, ported, landed, or replaced rather than closing this issue early.
