## Summary

Keep open: current main still has a source-provable path where dry-run validates the requested config object, but the write path persists a merge patch against a runtime-defaulted config and then reports the requested operation count as success, so default-valued writes can be dropped silently.

## What This Changes

Request that `config set` and `config patch` stop reporting success when default-valued writes are normalized out of the persisted config.

## Best Possible Solution

Make the config writer honor explicit default-valued user writes or fail with a clear diagnostic before success, and add focused regression coverage for bundled channel defaults.

## Reproduction Assessment

Yes. The issue has a concrete CLI reproduction, and current source explains the failure path: default-valued requested writes disappear when persistence diffs against runtime defaults; I did not perform live config writes because this was a read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is in the config mutation/persistence path, not in Telegram alone, because any schema-defaulted field can hit the same success-without-persist behavior.
