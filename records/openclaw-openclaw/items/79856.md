## Summary

Keep open: current main still has a source-provable path where default-valued config writes can be validated, passed to the writer, stripped during persistence against runtime defaults, and then reported as successful. The related open persisted-config PRs are useful context, but they focus on API response state and do not by themselves prove the CLI silent-discard behavior is fixed.

## What This Changes

Make `config set` and `config patch` stop reporting success when requested default-valued writes are not persisted, and clarify or align the validation behavior around those writes.

## Best Possible Solution

Fix the shared config mutation/persistence path so explicit default-valued user writes either persist or produce a clear non-success diagnostic, with regression coverage for Telegram `dmPolicy` and `groupPolicy` defaults.

## Reproduction Assessment

Yes. The issue has concrete CLI steps, and current source explains the same path: requested default-valued writes are applied to `snapshot.resolved`, then persistence diffs against runtime-defaulted config and can drop them before success is logged.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared config write/projection or post-write verification path, not in Telegram-specific schema defaults; it should account for the related persisted-config PRs before changing files.
