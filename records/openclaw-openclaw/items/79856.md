## Summary

Keep open: current main still has the source-level path where an explicit config write can be accepted, persisted through a diff against runtime-defaulted config, and then reported as successful even when the authored file does not retain the default-valued field.

## What This Changes

The issue reports that `openclaw config set` and `openclaw config patch` can report successful writes while omitting explicitly written values that equal schema defaults.

## Best Possible Solution

Preserve explicitly written default-valued config paths during persistence, or reject/warn before reporting success, with regression coverage around schema-defaulted channel settings.

## Reproduction Assessment

Yes. The issue includes concrete CLI steps, and current source shows the same mechanism by diffing the requested config against a runtime-defaulted snapshot before writing.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the config mutation/persistence path by carrying explicit operation paths into the write projection or carefully comparing against the authored source shape without persisting unrelated runtime-only defaults.
