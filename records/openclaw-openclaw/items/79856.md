## Summary

Keep open: current main still has a source-provable CLI path where default-valued `config set`/`config patch` writes can be omitted from the persisted file while the CLI reports the requested operation count as success. The related open PR addresses persisted response plumbing for another config-write mismatch, but it does not fix the CLI silent-discard behavior.

## What This Changes

The issue asks OpenClaw to stop reporting success when `config set` or `config patch` drops explicit writes whose values match schema defaults.

## Best Possible Solution

Make explicit default-valued user writes either persist or fail with a clear diagnostic, while preserving the existing protection against leaking unrelated runtime defaults.

## Reproduction Assessment

Yes, source-level. The issue has a concrete CLI reproduction, and current main still builds a source-shaped next config, diffs persistence against runtime defaults, and reports requested operation count rather than persisted requested paths.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared config mutation/persistence path so explicit operation paths are honored or rejected before success; the related persistedConfig response PR alone is not enough.
