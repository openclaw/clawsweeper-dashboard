## Summary

Keep open: current main still has a source-provable path where default-valued config writes validate and report success, but persistence computes the on-disk patch against runtime-defaulted config, so explicit writes equal to schema defaults can disappear.

## What This Changes

The issue reports that `config set` and `config patch` silently discard explicitly written values when those values equal schema defaults.

## Best Possible Solution

Preserve explicitly requested config paths through the persistence projection, including default-valued paths, and add regression coverage that proves the written file contains those explicit CLI writes.

## Reproduction Assessment

Yes. The issue has concrete CLI steps, and current main source shows the requested default-valued paths can be removed by diffing `nextConfig` against runtime-defaulted `snapshot.config`; I did not run live writes because this was a read-only review and `pnpm` is unavailable in this checkout.

## Solution Assessment

Yes. The narrow maintainable fix is to carry explicit operation paths into the config write projection or otherwise force-include them, rather than changing Telegram defaults or broad validator semantics first.
