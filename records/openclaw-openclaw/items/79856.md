## Summary

Keep open: current main still has a source-provable path where default-valued config writes validate and report success but can be omitted from the authored config because persistence diffs against runtime-defaulted config. The related open PR addresses response/readback divergence for config RPCs, but it does not fix explicit default-valued writes being dropped on disk.

## What This Changes

The issue asks `config set` and `config patch` to persist explicit values that match schema defaults, or fail/warn instead of reporting success after dropping them.

## Best Possible Solution

Preserve explicit user-written paths through the config persistence projection, or use an equivalent authored-source diff, so default-valued writes survive without persisting unrelated runtime defaults; add focused regression coverage for the CLI and write-preparation helper.

Yes. The reporter's CLI steps are concrete, and current main source shows the same failure path: the persistence patch is computed from runtime defaults, so default-equal explicit writes create no persisted delta; this review did not run write commands because the checkout was read-only.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79856. Current main can report success for `openclaw config set` / `config patch` writes whose values equal schema defaults, while `resolvePersistCandidateForWrite` drops those paths because it diffs `runtimeConfig` with defaults against `nextConfig`. Thread the explicitly written operation paths from `config set`/`config patch` through `replaceConfigFile`/`writeConfigFile` into `resolvePersistCandidateForWrite`, or implement an equivalent authored-source projection, so explicit writes such as `channels.telegram.dmPolicy=pairing` and `channels.telegram.groupPolicy=allowlist` persist on disk. Keep runtime defaults from leaking during unrelated writes, preserve `$include` protections, and do not make this a Telegram-only schema workaround. Coordinate with https://github.com/openclaw/openclaw/pull/77558 and https://github.com/openclaw/openclaw/issues/77455 if they land first, but note they cover response/readback divergence rather than this explicit default-write persistence bug. Add regression tests and a changelog entry for the user-facing data-loss fix.
