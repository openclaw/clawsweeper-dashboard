## Summary

Keep open: current main still has a source-provable path where default-valued writes are validated and reported as successful, but persistence diffs against a runtime-defaulted config so those explicit values can disappear before reaching disk. The related open PR covers returned persisted config from write handlers, but it does not thread explicit operation paths through this persistence reducer.

## What This Changes

The issue asks for `config set` and `config patch` to stop reporting successful writes when schema-default-valued properties are discarded from the persisted config.

## Best Possible Solution

Thread explicit user-written paths from `config set` and `config patch` into generic config persistence so default-valued writes are either preserved on disk or rejected before success, with regression coverage for bundled channel defaults.

## Reproduction Assessment

Yes. The reporter provided a concrete CLI transcript, and current main still shows the same source path: explicit default-valued writes can become zero-delta merge patches against runtime defaults before the CLI reports success.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared config mutation/persistence path, not in Telegram-specific schema handling; coordinate with https://github.com/openclaw/openclaw/pull/77558 but do not rely on response-plumbing alone.
