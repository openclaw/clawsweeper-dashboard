## Summary

Keep open: current main still has a source-level path where default-valued explicit writes can be reduced away while the CLI reports success, and no merged PR currently fixes this exact behavior.

## What This Changes

The issue reports that `openclaw config set` and `openclaw config patch` claim successful writes while schema-default-valued properties are stripped from the saved config.

## Best Possible Solution

Preserve explicit touched paths through the write reducer, or fail with a clear diagnostic before reporting success when a requested write is reduced away.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: start with a source config missing a defaulted channel field, set it to the schema default, and the runtime-to-next merge patch emits no persisted change. I did not run a live CLI write because this review was constrained to read-only inspection and `pnpm` is unavailable in the environment.

## Solution Assessment

Yes for keeping this open: the narrow maintainable fix belongs in the core config mutation/write path, with regression coverage proving explicit default-valued `config set` and `config patch` writes either persist or fail visibly.
