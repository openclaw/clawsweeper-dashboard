## Summary

This should stay open: current main still treats a queued subagent completion wake as delivered immediately, and the registry then records completionAnnouncedAt from that queue-accept result before collect-mode drain commits the message to the parent transcript.

## What This Changes

The issue asks OpenClaw to make queued subagent completion announcement timestamps reflect transcript delivery rather than queue acceptance.

## Best Possible Solution

Keep the issue open and change the subagent completion delivery contract so queue acceptance records enqueue state while completionAnnouncedAt is written only after the queued parent-session turn has drained or the direct gateway commit has resolved.

## Reproduction Assessment

Yes, from source inspection: an active requester completion wake returns delivered as soon as the queue accepts it, and registry cleanup immediately timestamps completionAnnouncedAt from that boolean. I did not execute the live timing scenario, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes, the proposed direction is the right narrow fix: split queue acceptance from transcript delivery and update the registry from the queue drain completion point. I would treat completionEnqueuedAt as useful observability, but the core correctness fix is deferring completionAnnouncedAt.
