## Summary

Keep open. Current main still has a source-level path where a subagent completion wake is treated as delivered once the active requester queue accepts it, and the registry then stamps `completionAnnouncedAt`, so the reported state drift remains plausible and actionable.

## What This Changes

Defer subagent `completionAnnouncedAt` until the parent transcript actually receives the completion message, with an optional enqueue timestamp for queue/steer acceptance.

## Best Possible Solution

The registry should distinguish enqueue acceptance from transcript delivery and only mark `completionAnnouncedAt` and delivered status after the queued or steered parent message has committed.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only pass. Source inspection gives a medium-confidence reproduction path: an accepted queued/steered completion wake can set `completionAnnouncedAt` before the collect queue's debounce-delayed drain commits the parent message.

## Solution Assessment

Yes, the proposed semantic split is the right direction. The narrowest maintainable fix is to preserve direct-path behavior, record queue acceptance separately if useful, and stamp announced/delivered state from a queue drain or transcript-commit completion signal.
