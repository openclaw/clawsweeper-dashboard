## Summary

Keep open: current main and v2026.6.9 still do not expose a native outbound mention contract for the shared message tool or QQBot send path, and DingTalk/WeCom are not in-tree implementations, so the remaining decision is the portable versus plugin-owned API shape.

## What This Changes

The issue requests native outbound @mention support for DingTalk, QQ Bot, and WeCom group messages via platform-specific fields or a portable mentions object.

## Best Possible Solution

Choose a portable mention contract or plugin-owned message-tool schema pattern, then implement QQBot mapping in-tree and coordinate DingTalk/WeCom mappings in their owning plugins with focused tests and platform proof.

## Reproduction Assessment

Yes for source-level verification: current main has no shared outbound mention fields and QQBot builds outbound payloads without native mention metadata. No live DingTalk, QQBot, or WeCom transport reproduction was attempted.

## Solution Assessment

Unclear: native outbound mentions are a valid capability, but the proposed platform-specific ChannelOutboundContext fields are not clearly the best contract. Maintainers should choose the portable versus plugin-owned schema shape first.
