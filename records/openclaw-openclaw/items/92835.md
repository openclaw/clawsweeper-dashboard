## Summary

Keep open: current main still lacks the requested outbound native mention contract, and the latest release has no matching mention fields either. The request is useful, but it needs maintainer direction on whether mentions become a portable message-tool contract or plugin-owned schema/payload behavior, especially because QQBot is in this repo while WeCom is an external channel plugin and DingTalk source is not present here.

## What This Changes

The issue requests `message(action="send")` support for @mentioning users or everyone in DingTalk, QQ Bot, and WeCom group messages.

## Best Possible Solution

Maintainers should choose a portable mention model or a plugin-owned message-tool schema pattern first, then implement per-channel payload mapping and tests in the owning channel packages.

## Reproduction Assessment

Yes for source-level verification: current main lacks shared outbound mention fields and QQBot's send path does not carry native mention metadata. No live DingTalk, QQBot, or WeCom transport reproduction was attempted.

## Solution Assessment

No clear best solution yet: the requested behavior is valid, but the proposed `atMobiles`/`atUserIds`/`isAtAll` fields would make shared core carry platform-specific fields. The existing plugin schema contribution seam is a plausible alternative, so maintainers should choose the contract first.
