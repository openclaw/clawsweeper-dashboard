## Summary

Keep this PR open: the runtime diagnostic approach is useful and well-proven, but the current patch leaks an internal diagnostics payload into the public plugin SDK surface and exact-head CI is failing the plugin SDK surface budget.

## What This Changes

The branch adds runtime plugin load-progress snapshots, threads them through embedded-agent and isolated-cron execution phase reporting, and formats attempted/completed/in-flight plugin details in cron timeout errors.

## Best Possible Solution

Keep the diagnostic-only runtime behavior, but make the progress payload internal to cron/embedded watchdog plumbing and restore the plugin SDK surface check before maintainer merge review.

## Reproduction Assessment

## Solution Assessment

No, not in its current shape. The loader-to-watchdog diagnostic path is the right narrow fix for the first ask, but exposing the internal progress payload through the public plugin SDK is not the best boundary.
