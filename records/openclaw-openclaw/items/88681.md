## Summary

Keep this PR open: it targets the right runtime-plugin stall diagnostic gap, but the patch can preserve stale runtime plugin details across later same-phase watchdog updates, so timeout text may name plugins that are no longer in flight. External real-behavior proof is also still missing.

## What This Changes

The PR adds runtime plugin load progress diagnostics, forwards them through embedded-agent and isolated cron phase reporting, and appends attempted/completed/in-flight plugin ids to cron timeout messages.

## Best Possible Solution

Keep the diagnostic-only loader progress path, but make phase state replacement clear stale runtimePlugins data and prove the behavior with focused regression coverage plus redacted real timeout output from a slow or hung plugin.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: emit runtime plugin progress, then emit a later runtime_plugins phase update without runtimePlugins; the watchdog merge keeps the old diagnostic payload and the formatter still accepts it because the phase matches.

## Solution Assessment

Is this the best way to solve the issue? Not yet; adding loader progress is the right boundary, but the state model must clear stale runtimePlugins on later phase updates and the PR needs real after-fix diagnostic proof.
