## Summary

Keep open. Current main has Telegram approval delivery, owner-derived Telegram exec approvers, group allowlists, sender-scoped group tool policy, and the session-level `/elevated full` approval skip, but it still lacks the requested opt-in trusted-owner/group/tool-class pre-approval bypass with audit labeling and destructive-operation fallback.

## What This Changes

Request an opt-in Telegram group policy that lets explicitly allowlisted owners execute selected approval-gated admin actions without a second approval prompt, with auditability and safety limits.

## Best Possible Solution

Design this as a narrow, explicit Telegram plugin/config policy with numeric owner IDs, allowlisted group/topic scope, allowed tool classes or tool names, an auditable bypass reason, and fail-closed destructive-operation behavior; add a generic approval seam only if the host approval pipeline lacks the necessary hook.

## Reproduction Assessment

Not applicable as a live bug reproduction; this is a feature gap. Source/schema/docs inspection gives a high-confidence verification path: the requested config and pre-authorization hook are absent on current main.

## Solution Assessment

Unclear as a final implementation proposal. The direction is plausible, but the maintainable solution should be a narrowly scoped, audited policy rather than a broad approval disable.
