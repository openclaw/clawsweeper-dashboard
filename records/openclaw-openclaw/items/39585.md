## Summary

Keep open. Current main still supports Telegram exec approval routing through explicit approvers or command-owner IDs, plus the broad existing `/elevated full` bypass; the requested trusted-owner, group-scoped, tool-class approval bypass is still a security-sensitive product/config feature.

## What This Changes

Add an opt-in Telegram group mode that lets explicitly trusted owners bypass approval prompts for configured host/tool actions with audit semantics.

## Best Possible Solution

Maintainers should either document elevated-full as the intended break-glass path, or approve a narrow audited Telegram policy scoped by numeric sender ID, allowlisted group/topic, tool class, and destructive-operation rules.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a request for a new opt-in approval-bypass capability. Source, docs, and tests give high confidence that the requested config and policy path are absent on current main.

## Solution Assessment

Unclear. The direction is plausible, but a direct boolean bypass is not the best safe shape until maintainers decide how it interacts with elevated-full mode, approval routing, tool-class metadata, and destructive-operation gates.
