## Summary

Keep open. Current main still provides Telegram-native approval routing and the broader `/elevated full` no-approval path, but it does not implement the requested group-scoped trusted-owner approval bypass with sender, group, tool-class, and audit controls.

## What This Changes

The issue requests an opt-in Telegram group policy that lets explicitly trusted owners bypass approval prompts for configured administrative tool classes with audit coverage.

## Best Possible Solution

Maintainers should either document the existing `/elevated full` escape hatch as sufficient, or approve a narrow audited Telegram/core policy scoped by sender ID, allowlisted group/topic, tool class, and destructive-operation rules.

## Reproduction Assessment

Not applicable as a bug reproduction: this issue asks for a new opt-in bypass mode. Source and test inspection do confirm the current behavior shape: `on`/`ask` preserves approvals, while only the broader `full` mode skips them.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact policy shape needs maintainer product and security review before implementation because the existing no-approval path is intentionally broader than the requested Telegram owner/group/tool-class mode.
