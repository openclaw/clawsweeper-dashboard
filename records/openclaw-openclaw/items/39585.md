## Summary

Keep open. Current main routes Telegram exec approval prompts and derives approvers from owner config, but the requested trusted owner/group/tool-class pre-approval bypass is still absent and needs security/product design before implementation.

## What This Changes

Request an opt-in Telegram policy that lets explicitly allowlisted owners in allowed groups bypass approval prompts for selected tool classes, with audit labeling and destructive-operation fallback.

## Best Possible Solution

Design a narrow Telegram-owned policy using numeric owner IDs, explicit group/topic scope, allowed tool names or classes, auditable bypass metadata, and fail-closed destructive-operation handling; add a generic approval seam only if the host approval pipeline lacks the required hook.

## Reproduction Assessment

Not applicable as a live bug reproduction because this is a new approval-bypass mode. Source, schema, and docs inspection give a high-confidence verification path that current main does not expose the requested config or pre-authorization hook.

## Solution Assessment

Unclear as a final implementation proposal. The direction is plausible, but the maintainable solution should be a narrowly scoped, audited policy rather than a broad approval disable from Telegram groups.
