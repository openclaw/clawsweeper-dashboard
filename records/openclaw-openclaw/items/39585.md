## Summary

Keep open. Current main supports Telegram exec approval routing, owner-derived approval authorization, same-chat/native approval UX, and elevated-full no-approval mode, but it does not implement the requested trusted Telegram group-owner bypass; that remains a security-sensitive product/config decision.

## What This Changes

The issue requests an opt-in Telegram group mode where explicitly allowlisted owners can run approval-gated runtime/filesystem actions without a second manual approval, with group, tool-class, audit, and destructive-operation limits.

## Best Possible Solution

Design this as a maintainer-approved shared approval/elevated policy extension with explicit owner identity, group allowlist, tool-class gates, destructive-operation handling, audit records, docs, and Telegram regression proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this asks for a new opt-in trusted-owner mode. Current source and docs provide high-confidence evidence that the requested bypass is absent rather than broken.

## Solution Assessment

No, not as a bare Telegram boolean. The safer solution is a shared approval-policy design that preserves explicit security boundaries and avoids a one-off channel shortcut.
