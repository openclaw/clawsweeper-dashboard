## Summary

Keep open. Current main and v2026.5.12 still have Telegram-native approval routing and broad no-approval modes, but not the requested audited trusted-owner bypass scoped by sender, allowlisted group/topic, and tool class; this is a security/product design change, not stale cleanup.

## What This Changes

The issue requests an opt-in Telegram group mode that lets explicitly trusted owners bypass approval-gated admin actions by sender, group, and tool class, with audit behavior.

## Best Possible Solution

Maintainers should either document `/elevated full` and YOLO as the intended break-glass path, or approve a narrow audited Telegram policy scoped by numeric sender ID, allowlisted group/topic, tool class, and destructive-operation rules.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a request for a new opt-in approval-bypass capability. Source, docs, tests, and v2026.5.12 inspection give high confidence that the requested config and policy path are absent.

## Solution Assessment

Unclear. The direction is plausible, but a direct boolean bypass would be too security-sensitive without designing it against existing `/elevated full`, YOLO, exec-approval routing, and future tool-class metadata.
