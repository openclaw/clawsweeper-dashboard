## Summary

Keep open. Current main has Telegram-native exec approval routing and broad no-approval paths such as `/elevated full`, but not the requested audited trusted-owner bypass scoped by sender, allowlisted group/topic, and tool class; that changes approval/security semantics and needs maintainer product/security review.

## What This Changes

Request an opt-in Telegram group policy that lets explicitly allowlisted owners bypass approval-gated runtime/filesystem actions with audit and tool-class/destructive-operation limits.

## Best Possible Solution

Maintainers should either document `/elevated full`/YOLO as the intended break-glass flow or approve a narrow audited Telegram policy scoped by numeric sender ID, allowlisted group/topic, tool class, and destructive-operation rules.

## Reproduction Assessment

## Solution Assessment

Unclear. The direction is plausible, but a new approval-bypass mode is not a narrow bug fix and should be designed against the existing `/elevated full`, YOLO, and exec-policy mechanisms before implementation.
