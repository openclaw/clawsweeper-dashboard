## Summary

Keep open. Current main has Telegram-native approval routing and a broader `/elevated full` escape hatch, but it does not implement the requested Telegram group owner bypass policy with group/tool-class scoping and audit semantics.

## What This Changes

The issue requests an opt-in Telegram group mode that lets explicitly allowlisted owners bypass approval gates for selected tool classes with audit logging.

## Best Possible Solution

Maintainers should decide whether to add a narrow Telegram/core approval policy or document the existing `/elevated full` and `agents.defaults.elevatedDefault="full"` path; any new policy should be explicit, group- and sender-scoped, audited, and covered by Telegram plus exec approval regression tests.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the reported shape is plausible because allowlisted elevated senders default to approval-preserving `on`/`ask`, while only `full` or `ask=off` suppresses exec approvals.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact bypass policy needs maintainer security/product judgment because the existing full-elevated path is broader than the requested Telegram owner/group/tool-class behavior.
