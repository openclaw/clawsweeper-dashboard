## Summary

Keep open. Current main has Telegram group allowlists, sender-scoped group tool policy, owner-derived Telegram exec approvers, native Telegram approval delivery, and the explicit `/elevated full` approval skip, but it still does not provide the requested opt-in Telegram trusted-owner/group/tool-class pre-approval bypass with audit labeling and destructive-operation fallback.

## What This Changes

Add an opt-in Telegram trusted-owner execution mode that lets explicitly allowlisted owners in allowlisted groups bypass approvals for selected tool classes, with audit labeling and safety fallbacks.

## Best Possible Solution

Keep this open and design the feature as a narrow opt-in Telegram plugin/config policy. A complete implementation should require resolved numeric owner IDs, an allowlisted group/topic scope, explicit tool classes or tool names, an audit reason such as trusted-owner-bypass, destructive-operation fallback behavior, schema/docs updates, and focused Telegram approval-path tests; add a generic approval-policy seam only if the host approval pipeline needs it.

## Reproduction Assessment

Not applicable as a live reproduction; this is a current feature gap rather than a crash. The high-confidence verification path is source/schema/docs inspection: the proposed config is absent, strict schema would reject it, and approval handling still creates or routes prompts instead of pre-authorizing owner-originated group execution.

## Solution Assessment

Unclear as an implementation proposal. The requested direction is plausible, but the best maintainable fix is not a broad approval disable; it needs a narrowly scoped, audited Telegram policy with generic core changes only where the approval pipeline lacks the necessary hook.
