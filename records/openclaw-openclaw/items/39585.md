## Summary

Keep open. Current main and the latest release have Telegram native exec-approval routing and the broad `/elevated full` break-glass path, but they do not have a group-scoped trusted-owner/tool-class approval-bypass contract; this is a security-sensitive feature request that needs maintainer product/security review before implementation.

## What This Changes

Request an opt-in Telegram policy that lets explicitly trusted owners in allowlisted groups bypass approval prompts for configured high-power tool classes with audit and destructive-operation controls.

## Best Possible Solution

Maintainers should either document `/elevated full` as the intended break-glass route or approve a narrow audited Telegram policy scoped by numeric owner ID, allowlisted group/topic, tool class, and destructive-operation rules.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new opt-in approval-bypass capability request. Source inspection confirms the requested config and policy path is absent on current main and in `v2026.5.20`.

## Solution Assessment

Unclear. The direction is plausible, but a direct boolean bypass is likely too coarse until maintainers decide how it composes with native approvals, `/elevated full`, tool-class metadata, destructive-operation handling, and audit logging.
