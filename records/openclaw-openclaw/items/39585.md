## Summary

Keep open. Current main supports Telegram owner-derived exec approvers, group allowlists, group tool policy, and native approval prompt delivery, but it still lacks the requested opt-in trusted-owner/group/tool-class pre-approval bypass with audit labeling and destructive-operation fallback.

## What This Changes

Request an opt-in Telegram policy that lets explicitly allowlisted owners in allowlisted groups bypass selected approval-gated runtime/filesystem actions with audit markers and safeguards.

## Best Possible Solution

Design this as a narrow audited Telegram policy with numeric owner IDs, allowlisted group/topic scope, allowed tool classes or tool names, and fail-closed destructive-operation behavior; add a generic approval seam only if the host approval pipeline lacks the needed hook.

## Reproduction Assessment

Not applicable as a live bug reproduction; this is a feature request. Source inspection on current main gives high-confidence verification that the requested config key and pre-authorization hook are absent.

## Solution Assessment

Unclear as a final implementation proposal. The direction is plausible, but the maintainable solution should be a narrowly scoped, audited policy rather than a broad approval disable.
