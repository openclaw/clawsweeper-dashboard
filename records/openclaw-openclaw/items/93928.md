## Summary

Keep open: current main and the latest release still have first-page-only Feishu drive `list`/`info` behavior, and live GitHub shows open linked PRs that already target the fix.

## What This Changes

The issue requests pagination support for Feishu drive `list` and `info` so files beyond the first Lark API page remain accessible.

## Best Possible Solution

Land one plugin-local Feishu drive pagination fix that wires cursor inputs for `list`, drains `has_more`/`next_page_token` for `info`, adds focused regression tests, and updates Feishu drive skill docs if the tool input shape changes.

## Reproduction Assessment

Yes, source-reproducible: current main passes only `folder_token` to `drive.file.list` for the affected paths and `getFileInfo` searches only the first response page. The issue also provides a mocked two-page repro, but I did not run a live Feishu tenant test.

## Solution Assessment

Yes, the maintainable fix belongs inside the Feishu plugin: use the Lark SDK cursor contract for `list` and `info`, with bounded pagination behavior and tests. The open PRs are candidate implementations, so the issue should track their review instead of spawning a separate repair lane.
