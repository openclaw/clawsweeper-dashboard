## Summary

Keep open: current main and the latest release still have first-page-only Feishu drive `list`/`info` behavior, and https://github.com/openclaw/openclaw/pull/94000 plus https://github.com/openclaw/openclaw/pull/94042 are open fix candidates. This issue should stay open until one candidate lands or maintainers choose a different fix path.

## What This Changes

The issue reports that Feishu drive `list`/`info` ignore Lark pagination, making files beyond the first API page unreachable and causing false `File not found` errors.

## Best Possible Solution

Land one plugin-local Feishu drive pagination fix that wires cursor inputs for `list`, drains `has_more`/`next_page_token` for `info`, adds focused regression tests, and updates the drive skill docs if the tool input shape changes.

## Reproduction Assessment

Yes, source-reproducible: current main passes only `folder_token` to `drive.file.list` for the affected paths and `getFileInfo` searches only the first response page. The issue also provides a mocked two-page repro, but I did not run a live Feishu tenant test.

## Solution Assessment

Yes, the maintainable fix belongs inside the Feishu plugin: use the Lark SDK cursor contract for `list` and `info`, with bounded pagination behavior and tests. The open PRs are candidate implementations, so the issue should track their review instead of spawning a separate repair lane.
