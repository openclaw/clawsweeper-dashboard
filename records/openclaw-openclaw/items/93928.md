## Summary

Keep open: current main and v2026.6.8 still have first-page-only Feishu drive list/info behavior, and two open PRs already target this bug with closing syntax.

## What This Changes

The issue reports that Feishu drive list/info only inspect the first Lark Drive file-list page, making files beyond page 1 unreachable and causing false File not found results.

## Best Possible Solution

Land one plugin-local Feishu drive pagination fix with explicit cursor support for `list`, bounded page-draining `info`, and focused regression coverage; then close the duplicate candidate PR.

## Reproduction Assessment

Yes, source-reproducible: current main calls `drive.file.list` only once for the affected `list` and `info` paths, and the schema cannot accept a list cursor. I did not run a live Feishu tenant test.

## Solution Assessment

Yes: the maintainable fix belongs in the Feishu plugin by wiring the Lark SDK cursor contract for `list` and paginating `info` until the file is found or pages are exhausted. Because open linked PRs already target the issue, the best path is to review and land one candidate rather than queue another fix.
