## Label Justifications

- `P2`: This useful feature has several bounded but merge-blocking lifecycle defects plus a product and security boundary decision.
- `merge-risk: 🚨 compatibility`: The new persisted schedule kind re-arms at startup, has incorrect same-ID update behavior, and introduces rollback considerations.

## Summary

The happy path is convincingly demonstrated, but three high-confidence watcher lifecycle defects can run deleted, obsolete, or repeated commands; after those repairs, explicit cron/gateway owner approval is still needed for the persistent shell-execution boundary.

## What This Changes

Adds a gateway-owned `on-exit` cron schedule that persists a watched shell command and wakes the originating session when the command exits.

## Best Possible Solution

Implement a lifecycle-owned watcher state machine keyed by job ID and schedule identity, reserve state before spawning, cancel stale generations, replace changed schedules, require durable one-shot completion before waking, and obtain explicit owner approval and public documentation for the new execution contract.

## Reproduction Assessment

Yes. Deterministic source-level reproductions are available by delaying `spawn()` and removing the job, updating command or cwd under the same ID, or rejecting `persistCompletion()` and restarting with the still-enabled stored job.

## Solution Assessment

No. Reusing the origin-aware wake is appropriate, but the watcher lifecycle is unsafe and the PR does not yet establish why a separate persistent execution model is preferable to extending the existing cron command surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.99

Full review comments:

- **[P1] Track pending spawns so removal can cancel them:** `src/gateway/cron-exit-watchers.ts:87-99`
  - body: `active` is populated only after `spawn()` resolves. If the job is removed or disabled while that promise is pending, `reconcile()` sees no entry to cancel; the spawn then completes, becomes active, and can fire a deleted job. Reserve a generation/state entry before awaiting spawn and discard or cancel stale completions.
  - confidence: 0.99
- **[P1] Replace the watcher when command or cwd changes:** `src/gateway/cron-exit-watchers.ts:149-155`
  - body: Reconciliation skips any job whose ID is already present without comparing its schedule. Updating an armed job therefore leaves the old process running and eventually wakes from the obsolete command while persistence shows the new definition. Track schedule identity and cancel and re-arm changed jobs.
  - confidence: 0.99
- **[P1] Fail closed when persisting completion fails:** `src/gateway/cron-exit-watchers.ts:118-127`
  - body: If `persistCompletion()` rejects, this path logs that restart may replay and then still calls `fireOnExit()`. The stored job remains enabled, so restart can re-arm and execute the command again despite the one-shot contract. Do not wake until terminal state is durable, and cover the failed-write and restart path.
  - confidence: 0.99
