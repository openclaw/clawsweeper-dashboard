## Label Justifications

- `P2`: The useful new cron capability has a normal-priority but merge-blocking lifecycle defect that can repeat user commands after restart.
- `merge-risk: 🚨 compatibility`: Persisted enabled jobs replay their command after gateway restart because one-shot completion is not durable.

## Summary

Keep this PR open, but do not merge the current head: its advertised one-shot watcher state is process-local, so restarting the gateway after completion can execute the persisted command and wake the session again. The live proof establishes the happy path, and the remaining defect has a focused repair.

## What This Changes

The PR adds an `on-exit` cron schedule that runs a gateway-owned shell command watcher and wakes the originating session when the command exits.

## Best Possible Solution

Atomically record terminal completion or disable/remove the job before firing, make every reconciliation path skip completed jobs, and keep command execution governed by a durable gateway task/process lifecycle contract.

## Reproduction Assessment

Yes, from source: allow an enabled `on-exit` watcher to finish, restart the gateway, and startup reconciliation sees the still-enabled persisted job without any durable completion marker and launches its command again.

## Solution Assessment

No, not in its current form. Gateway ownership solves per-turn teardown, but durable terminal state and an explicitly accepted execution-authority contract are required for this to be the narrowest safe implementation.
