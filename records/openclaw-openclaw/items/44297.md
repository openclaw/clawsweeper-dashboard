## Summary

Keep open: current main still preserves Slack startup by falling back to static argument menus, but the downgrade is only recorded through verbose logging and is not visible in channel status, doctor output, or Slack docs.

## What This Changes

The issue asks for Slack external argument-menu registration fallback to remain non-fatal while surfacing a visible operator health or diagnostics signal when external menus are disabled.

## Best Possible Solution

Add Slack plugin-owned runtime status/status-issue output and a visible non-verbose warning for external arg-menu fallback, preserving the non-fatal static-menu behavior and documenting the diagnostic path.

## Reproduction Assessment

Yes, at source level. The existing Slack slash test can make `app.options()` throw, and current main then falls back to `static_select` while only logging through `logVerbose`.

## Solution Assessment

Yes. A narrow Slack plugin change that reports degraded argument-menu capability through existing runtime/status conventions is the maintainable fix; a core health-model redesign is not needed.

## Repair Work Prompt

Fix this issue by adding a visible Slack plugin-owned degraded-mode signal when external argument-menu options registration fails. Preserve the existing non-fatal fallback to static menus, do not change slash command rendering thresholds, and avoid adding a new core health model unless an existing plugin SDK/status seam already requires it. Prefer threading a Slack-owned status/status-issue field and a non-verbose runtime warning from `registerSlackMonitorSlashCommands` through `monitorSlackProvider`/`setStatus`, then include regression coverage proving `app.options()` failure keeps startup alive, falls back to `static_select`, and exposes the degraded signal. Update Slack docs and `CHANGELOG.md` because this is operator-visible behavior. Related refs: current issue, #28702, #18496.
