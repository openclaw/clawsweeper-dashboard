## Summary

Keep open. Current main still falls back from Slack external argument menus to static menus non-fatally, but the failure is only recorded through verbose logging; there is no visible runtime warning, account status field, health signal, doctor diagnostic, or Slack docs guidance for operators.

## What This Changes

Request that Slack external argument-menu registration failures remain non-fatal while becoming visible to operators through a warning, status, health, doctor, or diagnostics signal.

## Best Possible Solution

Add a Slack plugin-local degraded-mode signal for external argument-menu registration failure: keep static menu fallback, emit a visible runtime warning, persist a scoped status/detail on the Slack account snapshot, and document the meaning in Slack troubleshooting if exposed to users.

## Repair Work Prompt

Fix openclaw/openclaw#44297 in the bundled Slack plugin. When `registerSlackMonitorSlashCommands()` catches an `app.options()` registration failure for external argument menus, preserve the current non-fatal fallback to static menus, but also emit a visible runtime warning and publish a scoped degraded status/detail that `openclaw channels status --probe --json` can expose for the affected Slack account. Keep the signal plugin-local and do not mark the whole Slack account disconnected solely because external argument menus are unavailable. Add focused regression tests for the fallback, visible warning/status detail, and unchanged static-menu behavior. Update Slack troubleshooting docs and CHANGELOG.md if the new signal is user-facing. Do not change unrelated Slack command rendering, Slack auth, or core channel status contracts beyond any generic field already accepted by account snapshots.
