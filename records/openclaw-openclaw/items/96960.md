## Summary

Keep open: this is a focused fix for a real auth-provider regression, but the patch is not merge-ready because the new fallback depends on a production Qwen auth alias that current metadata does not declare.

## What This Changes

The PR changes `resolveApiKeyForProvider` so an explicit alias-based profile ID can retry the same profile suffix under a canonical provider ID before throwing.

## Best Possible Solution

Make the Qwen/modelstudio compatibility contract real in production metadata or a narrow migration path, then keep the resolver fallback covered by tests that do not mock `modelstudio` directly to `qwen`.

## Reproduction Assessment

Yes, source inspection gives a medium-confidence reproduction path: current `main` throws when `resolveApiKeyForProvider` receives `profileId: "modelstudio:default"` while the store only has `qwen:default`. I did not run a live Qwen upgrade repro in this read-only pass.

## Solution Assessment

No, this is not yet the best complete fix because the new retry depends on `resolveProviderIdForAuth("modelstudio")` returning `qwen`, but current Qwen manifest metadata does not declare that auth alias. The safer path is to repair the production auth-alias contract or encode the legacy profile migration path and test that exact path.
