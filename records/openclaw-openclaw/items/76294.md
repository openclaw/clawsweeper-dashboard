## Summary

Keep open: the PR fixes the original array double-splice path, but the current guard is type-blind and would make numeric object-key unsets such as `agents.defaults.models.30` silently no-op when write preservation restores the deleted key; it also still needs the required changelog entry.

## What This Changes

The branch changes `runConfigUnset` to omit write-time `unsetPaths` for numeric leaf paths and adds a CLI regression test for `agents.list[1]`.

## Best Possible Solution

Make the array-index decision type-aware by checking the resolved parent container, keep `unsetPaths` for object-key deletion, add focused regressions for numeric object keys and nested array leaves, then land with a single Unreleased Fixes changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main splices the array in `unsetAtPath`, then write preparation can apply the same unset path and splice again.

## Solution Assessment

No as written. Omitting `unsetPaths` only because the final segment looks numeric is not the narrowest safe fix; the condition needs to depend on whether the parent value is actually an array.
