## Summary

Keep open. The PR targets a real current-main data-loss bug, but the proposed fix is type-blind: it treats any numeric-looking final segment as an array index, which would skip explicit unset handling for numeric object keys and can silently preserve entries that should be deleted. The PR also still needs the required changelog entry.

## What This Changes

The PR changes `runConfigUnset` to omit write-time `unsetPaths` for numeric-looking leaf paths and adds a CLI regression test for removing `agents.list[1]`.

## Best Possible Solution

Repair the PR so `unsetPaths` is omitted only when the resolved parent is an actual array, preserve explicit unset handling for numeric object keys, add targeted regression tests, and add one Unreleased Fixes changelog line before landing.

## Reproduction Assessment

Yes. Source inspection on current main shows a high-confidence path: `runConfigUnset` splices the array in memory, then the write preparation applies the same numeric unset path again.

## Solution Assessment

No, not yet. The focused boundary is right, but the implementation must test the resolved parent type instead of only the final segment text, and it still needs the required changelog entry.
