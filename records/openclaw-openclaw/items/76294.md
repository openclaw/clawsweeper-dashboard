## Summary

Keep open: the PR targets a real current-main data-loss bug, but the current fix is type-blind and can turn numeric-string object-key unsets into no-ops; the later reviewer comment about `agents.defaults.models.30` is supported by the current write-preservation code. It also still needs the required changelog entry.

## What This Changes

The PR changes `runConfigUnset` to suppress write-time `unsetPaths` for numeric leaf paths and adds a regression test for unsetting `agents.list[1]`.

## Best Possible Solution

Land a focused fix that distinguishes actual array parents from numeric object keys, covers both cases with regression tests, and adds a single Unreleased Fixes changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows `unsetAtPath` splices `agents.list[1]` before `replaceConfigFile`, then `writeOptions.unsetPaths` sends the same path to `unsetPathForWriteAt`, which can splice again.

## Solution Assessment

No. The proposed direction is narrow, but the current implementation should inspect the resolved parent container and omit write-time `unsetPaths` only for actual array leaf removals.
