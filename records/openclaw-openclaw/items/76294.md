## Summary

Keep open. The PR fixes the reported array double-splice path, but the current diff still needs changes because it drops `unsetPaths` based only on a numeric leaf segment and lacks the required changelog entry.

## What This Changes

The PR changes `runConfigUnset` to omit write-time `unsetPaths` for numeric leaf paths and adds a regression test for `agents.list[1]`.

## Best Possible Solution

Land a narrow fix that suppresses `unsetPaths` only when the resolved parent is actually an array, adds regression coverage for numeric object keys and the reported array case, and includes one Unreleased Fixes changelog line.

## Reproduction Assessment

Yes. Source inspection on current main shows `runConfigUnset` splices `agents.list[1]` in memory and then sends the same numeric path to write preparation, which can splice the array a second time.

## Solution Assessment

No. The direction is right for array leaves, but the implementation should check the actual parent value rather than treating every numeric final segment as an array index, and the user-facing fix still needs a changelog entry.
