## Summary

Keep open. Current main has persistent same-ID Feishu dedupe and WebSocket recovery, but it still has no Feishu reconnect/offline replay policy for stale messages replayed with fresh message IDs.

## What This Changes

The issue asks Feishu WebSocket reconnects and Gateway restarts to avoid reprocessing replayed messages that arrive with new message IDs despite dedupe being enabled.

## Best Possible Solution

Fix this inside the bundled Feishu plugin with an explicit per-account reconnect/offline replay policy that preserves same-ID persistent dedupe while suppressing stale fresh-ID replays using create_time or another stable Feishu-owned signal.

## Reproduction Assessment

Yes, with a source-level path: deliver the same stale Feishu logical message twice after reconnect using different message_id values, and current main treats both as fresh. A live Feishu account would be needed to prove the SDK replay behavior end to end.

## Solution Assessment

No. Current main handles same-ID dedupe and WebSocket recovery, but the narrow maintainable fix still needs a Feishu-owned replay/offline policy instead of a broad core Gateway workaround.
