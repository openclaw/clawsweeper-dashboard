## Summary

Keep open. Current main has persistent same-ID Feishu dedupe and WebSocket recovery, but the fresh-message-ID replay path described here still has no connection-start, timestamp/content, or offline-message policy that would stop stale replayed messages from reaching the agent.

## What This Changes

The issue asks for Feishu WebSocket reconnect/restart replay handling so old messages replayed with fresh message IDs do not trigger duplicate agent responses.

## Best Possible Solution

Fix this inside the bundled Feishu plugin with an explicit per-account reconnect/offline replay policy that preserves current same-ID persistent dedupe while suppressing stale fresh-ID replays using message create_time and focused tests.

## Reproduction Assessment

Yes, with a source-level path: deliver a stale Feishu message event twice with different message_id values after reconnect and current main keys dedupe on the fresh ID. A live Feishu account would be needed to prove the external SDK replay behavior end to end.

## Solution Assessment

No, current main is not a complete solution because it restores and checks same-ID dedupe only. The narrow maintainable direction is a Feishu-owned replay/offline policy, not a core gateway workaround.
