## Summary

Keep open. Current main restores and persists Feishu dedupe for the same message ID and claims same-ID events before dispatch, but the reported fresh-ID WebSocket replay path still bypasses those guards because there is no connection-start, replay-window, or offline-message policy in the Feishu receive path.

## What This Changes

The issue requests suppressing duplicate Feishu agent responses after WebSocket reconnects when old replayed messages arrive with new message IDs and bypass the current dedupe cache.

## Best Possible Solution

Fix this in the bundled Feishu plugin by adding an explicit per-account WebSocket reconnect replay/offline policy that uses message.create_time relative to connection start or reconnect start, preserves the current same-ID persistent and in-flight dedupe, and updates config schema, docs, and focused Feishu tests for stale fresh-ID replay suppression versus fresh post-connect dispatch.

## Reproduction Assessment

Yes, with a live Feishu WebSocket account: send one or more messages, restart the gateway or trigger the WebSocket reconnect, and observe Feishu replaying recent events. The current-main code path shows why fresh IDs bypass the existing same-ID guards even without live credentials in this review.

## Solution Assessment

No, current main is not the complete solution: it solves same-message-ID replay but not stale replay under new IDs. The maintainable fix should stay in the Feishu plugin and be policy/config-aware rather than adding a core migration or generic gateway workaround.
