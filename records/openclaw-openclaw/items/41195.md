## Summary

Keep open. Current main restores and persists same-ID Feishu dedupe, but it still has no Feishu reconnect/offline replay policy for stale logical messages replayed with fresh message IDs; the related Feishu concatenation report was closed into this issue as the canonical tracker.

## What This Changes

The issue asks Feishu WebSocket reconnect handling to prevent replayed history messages with fresh IDs from being processed as new inbound messages after gateway restart.

## Best Possible Solution

Implement a Feishu-owned reconnect/offline replay guard before debounce/dispatch, using create_time or another stable Feishu-owned signal while preserving persistent same-ID dedupe and documenting any default or config policy.

## Reproduction Assessment

Yes, source-level: deliver the same stale Feishu logical message twice after reconnect with different message_id values, and current main treats both as fresh. A live Feishu account is still needed to prove the SDK new-ID replay behavior end to end.

## Solution Assessment

Unclear. The requested direction is valid, but a hard-coded grace or timestamp drop is not safely mergeable until maintainers decide the Feishu offline/reconnect policy and whether users need a knob.
