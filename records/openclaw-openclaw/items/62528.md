## Summary

Keep open. The feature is useful and matches the linked request, but the current PR still introduces a mapped-hook compatibility regression, drops structured session-create error status, and is missing the required changelog entry.

## What This Changes

The PR adds a generic `POST /hooks/message` webhook ingress with shared hook auth/body/idempotency handling, docs/tests, and narrower `/hooks/agent`/cron delivery mirroring behavior.

## Best Possible Solution

Land a rebased implementation that preserves existing mapped-hook names, keeps structured gateway error status through the new ingress path, and documents the user-facing endpoint in the changelog.

## Reproduction Assessment

Yes. The mapped-hook regression is reproducible by configuring `hooks.mappings[].match.path` to `message` and posting the existing mapped payload; the error-status regression is reproducible by making `sessions.create` return `INVALID_REQUEST` or `UNAVAILABLE` and observing `/hooks/message` map it to a generic 500.

## Solution Assessment

No, not as written. The endpoint direction is maintainable, but it should either let configured mappings for `message` win or provide an explicit compatibility path, and it should preserve structured `sessions.create` errors before merge.
