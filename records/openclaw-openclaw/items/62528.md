## Summary

Keep open. Current main still does not provide the requested `/hooks/message` ingress, but this PR is not merge-ready because it introduces two concrete Gateway behavior regressions, lacks contributor-supplied real behavior proof, and needs maintainer judgment on reserving a currently valid mapped-hook subpath.

## What This Changes

This PR adds a Gateway `POST /hooks/message` ingress, changes `/hooks/agent` and cron delivery mirroring semantics, and updates related Gateway/cron docs and tests.

## Best Possible Solution

Land a rebased implementation that preserves mapped-hook compatibility, propagates structured Gateway method errors, includes required scoped-instruction sibling links, and shows redacted real Gateway runtime proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a mapped hook with `match.path: "message"`, or make `sessions.create` return `INVALID_REQUEST`/`UNAVAILABLE`; the PR intercepts the mapped route and wraps the structured session error as a generic failure.

## Solution Assessment

No. The endpoint direction is plausible, but the current patch is not the narrowest maintainable solution because it reserves an existing mapped-hook name, drops structured error metadata, misses scoped-instruction sibling coverage, and lacks real runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

  - body: This branch handles `subPath === "message"` before evaluating `hooks.mappings`, so an existing custom hook with `hooks.mappings[].match.path: "message"` becomes unreachable and gets the new strict message validation instead of its configured mapping.

  - confidence: 0.9
