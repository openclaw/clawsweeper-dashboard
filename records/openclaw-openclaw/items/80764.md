## Summary

This issue should stay open because current main still has the Discord REST raw `response.text()`/`JSON.parse` path, and the linked unmerged fix PR with closing syntax is the right resolution path: https://github.com/openclaw/openclaw/pull/80788.

## What This Changes

Discord message read, search, and channel-info should parse gzipped Discord REST JSON instead of surfacing compressed binary or a `.map` error.

## Best Possible Solution

Land the focused Discord plugin fix in https://github.com/openclaw/openclaw/pull/80788, then close this issue as implemented by that merged PR.

## Reproduction Assessment

Yes, at source level: raw gzip bytes can reach `coerceResponseBody` as a string, fail JSON parsing, and then hit read/search callers that trust the parsed shape. I did not run a live Discord account repro in this review.

## Solution Assessment

Yes. The narrow maintainable solution is plugin-local decoding in the Discord REST client plus response-shape guards for read/search, which is exactly what the linked open PR is attempting.
