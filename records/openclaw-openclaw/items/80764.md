## Summary

## What This Changes

The issue reports Discord message read/search/channel-info receiving gzip-compressed response bytes and surfacing binary text or a `.map` failure instead of parsed JSON.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80788 or an equivalent Discord-plugin fix that decodes raw gzip before JSON parsing and validates read/search response shapes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for a live Discord account path; source inspection plus the focused Node gzip proof gives a clear current-main failure condition if raw gzip bytes reach the parser.

## Solution Assessment

Is this the best way to solve the issue? Yes, the linked plugin-local PR direction is the narrowest maintainable fix because it handles decoding at the shared Discord REST boundary and keeps malformed response guards near read/search contracts.
