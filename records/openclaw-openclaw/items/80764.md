## What This Changes

Report that Discord message read, search, and channel-info can surface raw gzip-compressed REST response bytes instead of parsed JSON.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/80788; if it lands, close this issue as implemented by that merge.

## Reproduction Assessment

No live high-confidence Discord reproduction was run here. Source inspection plus a focused Node `Response` proof shows the failing condition: raw gzip bytes become a non-JSON string, which current main then either maps as an array for read or returns through search.

## Solution Assessment

Yes, the narrow maintainable fix is plugin-local: decode raw gzip bodies in the Discord REST client before JSON parsing and add response-shape guards for message read/search. The linked PR appears aimed at that solution and should be reviewed rather than duplicating work from this issue.
