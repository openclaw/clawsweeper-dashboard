## Summary

Keep open: current main still has the reported Discord REST response boundary, and the downstream message actions assume parsed JSON values. The report is recent, not duplicated, and the narrow fix belongs in the Discord plugin REST client plus focused shape/regression coverage.

## What This Changes

The issue asks for Discord `read`, `search`, and `channel-info` actions to return parsed JSON instead of raw gzip-compressed binary strings.

## Best Possible Solution

Fix the Discord plugin REST response boundary to safely detect and decode encoded JSON before returning parsed values, then add shape guards and focused regression tests for read/search/channel-info.
