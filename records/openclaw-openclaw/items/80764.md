## Summary

Keep open: current main still parses Discord REST responses with `response.text()` plus JSON coercion and trusts the read/search response shapes. The focused fix PR at https://github.com/openclaw/openclaw/pull/80788 references this issue with closing syntax, but it is still open and unmerged, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Discord REST message read/search/channel-info should decode gzip-compressed API responses into parsed JSON instead of returning raw compressed text.

## Best Possible Solution

Land the focused Discord plugin REST decoding and response-shape guard fix in https://github.com/openclaw/openclaw/pull/80788, then close this issue as implemented with commit and release proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a clear failure condition: if raw gzip bytes reach `coerceResponseBody`, current main returns a string, `readMessages` calls `.map` on it, and `searchMessages` can surface it as results.

## Solution Assessment

Yes. The linked focused PR is the narrow maintainable path because it fixes decoding at the Discord REST client boundary and adds read/search shape guards without changing endpoints, auth, or core message-action contracts.
