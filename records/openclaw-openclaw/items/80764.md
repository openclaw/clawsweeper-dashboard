## Summary

Keep open: current main still has the Discord REST parsing path described in the report, and the open closing PR is the right place to finish the fix before this issue is closed.

## What This Changes

Discord message read/search/channel-info should decode gzip-compressed REST responses as JSON instead of surfacing binary text or type errors.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80788 if it remains correct; after merge, close this issue as implemented.

## Reproduction Assessment

No live Discord high-confidence reproduction was run here. The source path is clear: if raw gzip bytes reach RequestClient, coerceResponseBody returns a string and readMessages then calls .map on that string.

## Solution Assessment

Yes. A plugin-local decode in the Discord REST client plus read/search response-shape guards is the narrow maintainable fix, and the open closing PR follows that direction.
