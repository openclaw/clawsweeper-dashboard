## Summary

This looks like a valid, narrow Discord plugin bug rather than cleanup-close material: current main still parses Discord REST responses via `response.text()` plus `JSON.parse`, and the read/search/channel-info callers trust the parsed shape.

## What This Changes

The issue reports Discord message read, search, and channel-info returning gzip-compressed response bytes instead of parsed JSON.

## Best Possible Solution

Handle compressed Discord REST bodies in the plugin-owned REST client and add clear response-shape validation for message read/search so malformed upstream responses fail with actionable errors.

## Reproduction Assessment

No live high-confidence Discord reproduction was run. Source inspection gives a clear reproduction condition: if gzip bytes reach `coerceResponseBody`, JSON parsing falls back to a string and `readMessages` then calls `.map` on that string.

## Solution Assessment

Yes, a defensive decode in the Discord REST client plus targeted response-shape guards is the narrowest maintainable fix. This should stay plugin-local rather than adding a config flag or changing core message-action contracts.
