## Summary

Keep open: current main still has the Discord REST gzip parsing gap, and an open focused fix PR already references this issue with closing syntax, so the issue should close only after that PR lands.

## What This Changes

The issue reports that Discord message read, search, and channel-info actions should parse gzip-compressed REST responses as JSON instead of returning binary text.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80788, or an equivalent plugin-local fix that decodes raw gzip bodies in the Discord REST client and validates read/search response shapes.

## Reproduction Assessment

No live high-confidence Discord reproduction was run. Source inspection on current main shows raw gzip bytes can fall through as a string and then fail `.map` in `readMessages`, and a synthetic gzip-body check matches that failure condition.

## Solution Assessment

Yes. The narrow maintainable fix is plugin-local decompression before JSON coercion plus read/search response-shape guards; the existing open PR follows that direction, so this issue should stay open until it merges.
