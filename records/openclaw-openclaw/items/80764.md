## Summary

Keep open: current main still lacks gzip decoding in the Discord REST client, and the issue is already paired with the open closing fix at https://github.com/openclaw/openclaw/pull/80788. The merged read-message hardening at https://github.com/openclaw/openclaw/pull/82276 reduces one crash mode, but it does not parse raw gzip bodies or cover search/channel-info.

## What This Changes

Discord message read, search, and channel-info should parse gzip-compressed Discord REST responses as JSON instead of surfacing raw compressed bytes or shape errors.

## Best Possible Solution

Review and land the plugin-local fix in https://github.com/openclaw/openclaw/pull/80788, or an equivalent Discord REST-client decode with focused read/search shape coverage.

## Reproduction Assessment

No live high-confidence Discord reproduction was run. Source inspection gives a clear current-main reproduction condition: if raw gzip bytes reach RequestClient, coerceResponseBody returns a string, search/channel-info can surface it, and read now fails with a boundary error instead of parsing JSON.

## Solution Assessment

Yes. A Discord plugin-local raw gzip decode in the REST client plus targeted read/search shape guards is the narrowest maintainable solution and avoids changing core message-action contracts.
