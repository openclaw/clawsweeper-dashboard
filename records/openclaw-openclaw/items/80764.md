## Summary

Keep open: current main still has the raw Discord REST text parsing path and trusted read/search response shapes, while the focused fixing PR is open and linked to close this issue after merge.

## What This Changes

The issue asks Discord message read, search, and channel-info responses to parse gzip-compressed REST bodies as JSON instead of surfacing binary text or type errors.

## Best Possible Solution

Review and land the focused Discord plugin fix in https://github.com/openclaw/openclaw/pull/80788, then close this issue after the merge verifies gzip decoding and read/search shape guards on main.

## Reproduction Assessment

Yes for the source-level failure condition: current main lets raw gzip bytes fall through as a string and then read/search consume that value as structured Discord data. I did not run a live Discord account reproduction in this read-only review.

## Solution Assessment

Yes. A plugin-local decode before JSON coercion in the Discord `RequestClient`, plus narrow read/search shape guards, is the smallest maintainable fix and is already represented by the open linked PR.
