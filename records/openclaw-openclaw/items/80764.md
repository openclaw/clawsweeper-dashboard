## What This Changes

The issue reports Discord message read/search/channel-info returning raw gzip-compressed response bodies instead of parsed JSON.

## Best Possible Solution

Review and land the focused Discord plugin fix in https://github.com/openclaw/openclaw/pull/80788, keeping gzip decoding in the plugin REST client and shape validation at the message helper boundary.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live Discord credential flow was reproduced here, but source inspection gives a clear reproduction condition: raw gzip bytes reaching `coerceResponseBody` are returned as a string, after which read/search callers fail or surface the string.

## Solution Assessment

Is this the best way to solve the issue? Yes: a Discord-plugin-local decode in `RequestClient` plus response-shape guards for read/search is the narrowest maintainable fix, and the linked PR follows that boundary.
