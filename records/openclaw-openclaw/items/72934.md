## Summary

Keep open. Current main still leaves the reported browser snapshot timeout path unresolved: the browser tool exposes `timeoutMs`, but the snapshot action, client, plan, route, Chrome MCP call, and Playwright helpers do not carry one timeout budget end to end. Related PR #75702 is open with closing syntax for this issue, so this should remain open until that PR lands, is replaced, or is closed.

## What This Changes

Fix a browser automation regression where `browser action=snapshot` times out after successful open or navigate operations by honoring snapshot timeout handling.

## Best Possible Solution

Review and land, replace, or close PR #75702. The desired end state is one browser plugin fix that normalizes a snapshot timeout budget and threads it through `executeSnapshotAction`, the host client/proxy request, `/snapshot` plan and route, Chrome MCP snapshot/screenshot calls, Playwright snapshot helpers, focused regression tests, and the required changelog entry.

## Reproduction Assessment

Yes for triage. The issue gives a precise `open` then `snapshot` sequence, and current-main source inspection independently confirms that the timeout budget cannot reach several snapshot operations that may hang.

## Solution Assessment

Yes. Threading the existing `timeoutMs` through the browser plugin snapshot path is the narrow maintainable direction; closing this issue before the linked open implementation lands would be premature.
