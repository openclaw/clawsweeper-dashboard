## Summary

Keep open. Current main still exposes browser snapshots as a supported product surface and accepts a top-level `timeoutMs`, but the snapshot path drops that timeout before the host/node proxy call and the `/snapshot` route does not propagate a route-level timeout into Chrome MCP, Playwright AI/role/ARIA snapshots, or label side paths. The existing ClawSweeper comment and related #72653 context remain relevant, and I did not find an open PR directly tied to this issue.

## What This Changes

The issue reports that browser `open` and `navigate` succeed but `snapshot` consistently times out on OpenClaw 2026.4.25 on macOS.

## Best Possible Solution

Repair the browser plugin snapshot timeout path so one normalized snapshot timeout is propagated from the agent tool through host and node proxy calls, `browserSnapshot`, `/snapshot`, Chrome MCP snapshot/screenshot/URL collection, Playwright AI/role/ARIA snapshot helpers, and snapshot label side paths, with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes for triage: the issue provides a concrete open-then-snapshot sequence, affected version, platform, gateway status, and repeated failures across URLs. Current-main source inspection independently confirms that snapshot timeout budgets cannot reach several work paths that can hang.

## Solution Assessment

Yes: a narrow browser plugin timeout propagation fix is the best maintainable path. A broad core gateway rewrite or browser profile migration would be larger than the evidence supports.
