## Summary

Keep open. Current main still has a source-backed browser snapshot timeout propagation gap: `timeoutMs` is exposed at the browser tool/API boundary and outer gateway/node proxy layers can time out, but the snapshot action, direct host client, `/snapshot` plan, Chrome MCP snapshot call, and Playwright snapshot branches still do not consistently pass the budget into the work that can hang.

## What This Changes

The issue reports that browser `open`/`navigate` succeeds but `snapshot` consistently times out on macOS OpenClaw 2026.4.25.

## Best Possible Solution

Make snapshot timeout handling a first-class browser plugin contract: normalize one timeout budget at the browser tool/client/CLI boundary, include it in snapshot query/proxy/body as appropriate, add it to the snapshot plan, and pass it through Chrome MCP, Playwright AI/role/ARIA snapshot, label screenshot, URL collection, and ref-storage side paths where the underlying dependency supports cancellation or timeout.

## Reproduction Assessment

Yes for triage: the issue gives a precise open-then-snapshot sequence with version, platform, gateway status, and repeated cross-URL failures. Current-main source inspection independently confirms that snapshot timeout budgets cannot reach several parts of the snapshot stack.

## Solution Assessment

Yes: a narrow browser plugin snapshot timeout propagation fix is the best maintainable path. A broad core gateway rewrite or browser profile migration would be larger than the evidence supports.

## Repair Work Prompt

Fix openclaw/openclaw#72934 by repairing browser snapshot timeout propagation in the bundled browser plugin. Observable bug: browser open/navigate can succeed, but `browser({ action: "snapshot", timeoutMs: ... })` and default snapshot calls time out because the timeout budget is not consistently carried from the tool/client/proxy/CLI boundary into `/snapshot`, Chrome MCP, and Playwright snapshot work. Keep the fix in `extensions/browser`; do not add bundled browser ids or policy to core, do not broaden unrelated browser behavior, and do not relax SSRF checks. Expected implementation boundary: normalize snapshot timeout from top-level tool input and CLI/default caller timeout; pass it through `executeSnapshotAction`, node proxy params, `browserSnapshot`, `/snapshot` query/plan, Chrome MCP `take_snapshot`/label paths where callTool supports timeout options, Playwright AI/role/ARIA snapshot helpers, label screenshot, URL collection/ref storage where feasible, and ensure route-level cancellation/signal handling is coherent with existing browser route patterns. Add focused tests proving direct host snapshot and node proxy snapshot forward caller/default `timeoutMs`, the client serializes it, `/snapshot` parses it, Chrome MCP snapshot receives it, Playwright snapshot helpers receive or enforce it, and existing refs fallback preserves the same timeout. Include a single-line user-facing CHANGELOG.md entry under the active release section with appropriate `Thanks @wkomers-hash` credit, and mention related #72653 only if the fix actually covers that broader path.
