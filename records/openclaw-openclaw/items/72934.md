## Summary

Keep open. Current main still has a source-backed browser snapshot timeout propagation gap: the browser tool exposes `timeoutMs` and the gateway/node proxy can honor it when supplied, but the snapshot action, host client, `/snapshot` route, and snapshot engines do not consistently carry a timeout budget. The related #72653 is broader, while this issue has a distinct snapshot-only macOS reproduction that is useful to keep until a focused browser plugin fix lands.

## What This Changes

The issue reports that browser open/navigate succeed but `browser` snapshot consistently times out after returning a targetId on OpenClaw 2026.4.25.

## Best Possible Solution

Repair the browser plugin snapshot path so a caller-provided or default snapshot timeout is normalized once and propagated from the agent tool through the host client, gateway/node proxy, `/snapshot` route, Chrome MCP snapshot calls, Playwright snapshot helpers, and CDP fallback helpers, with focused regression coverage for both host and node-proxy paths.

## Reproduction Assessment

Yes for triage: the issue provides a precise open-then-snapshot sequence with version/platform details, and current-main source inspection independently confirms that `timeoutMs` cannot affect the snapshot request or its inner snapshot calls. A live macOS run was not performed in this read-only cleanup review.

## Solution Assessment

Yes: the narrow maintainable fix is timeout propagation inside the browser plugin snapshot stack. A core gateway rewrite or broad browser profile migration would be larger than the evidence supports.
