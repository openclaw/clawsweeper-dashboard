## Summary

Keep open. Current main still leaves the reported browser snapshot timeout path unresolved: the browser tool exposes `timeoutMs`, but snapshot calls do not carry it through the tool action, host client, `/snapshot` route plan, Chrome MCP snapshot call, or several Playwright snapshot helpers that can block.

## What This Changes

The issue reports that browser `open`/`navigate` succeeds but `snapshot` consistently times out on macOS OpenClaw 2026.4.25 instead of returning page content or an accessibility tree.

## Best Possible Solution

Implement one browser plugin fix that normalizes a snapshot timeout budget and carries it end-to-end through the agent tool, node proxy/host client, `/snapshot` planning, Chrome MCP snapshot and label side paths, and Playwright AI/role/ARIA snapshot helpers, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes for triage: the issue gives a precise `open` then `snapshot` sequence, platform/version, gateway status, and repeated cross-URL failures. Current-main source inspection independently confirms the timeout budget cannot reach several snapshot operations that may hang.

## Solution Assessment

Yes: a narrow browser plugin timeout propagation repair is the best maintainable path. A gateway rewrite, browser profile migration, or broad CDP transport change would be larger than the evidence supports.
