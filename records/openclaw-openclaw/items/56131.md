## Summary

Current main still does not expose reset archives through `sessions_history` or `sessions_list`. The tool schemas have no archive selector, the Gateway history path reads only active transcript candidates, and reset still archives the old transcript after replacing the session row. The duplicate #57139 was already closed in favor of this issue, while #60409 remains an open partial implementation candidate rather than a complete fix for archive discovery and reset-chain recovery.

## What This Changes

Make reset-archived session fragments discoverable and readable through supported session tools so agents can recover context after a reset.

## Best Possible Solution

Keep this issue as the canonical product/API thread and implement gateway-mediated reset-archive discovery and reading under the existing session visibility and redaction model. Coordinate the narrow fallback in #60409 with the broader #45003 reset-archive roadmap, and decide whether exec completion events also need durable origin metadata tied to the pre-reset session/run.

## Reproduction Assessment

Yes. A source-level reproduction is clear: after reset archives the old transcript and the current session points at a new file, `sessions_history` can only call `chat.history` for the current entry, and `readSessionMessages` returns `[]` when the active transcript candidates are absent. I did not run a runtime repro because this review was explicitly read-only.

## Solution Assessment

Unclear for the full issue. The open #60409 approach is a useful partial read-path fallback, but the complete fix should add explicit archive/chain discovery or metadata to the supported session surfaces rather than only falling back when the primary transcript is missing.
