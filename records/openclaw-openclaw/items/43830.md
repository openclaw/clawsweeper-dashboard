## Summary

Keep open. Current `main` and the latest release have partial redaction for logs, transcripts, tool-detail surfaces, and internal formatting artifacts, but they do not implement the requested automatic sensitive-data sanitization for all agent outputs.

## What This Changes

The issue requests automatic sanitization/redaction of sensitive data in agent/tool outputs via core configuration or middleware.

## Best Possible Solution

Define the core security policy first, then implement a narrow central output-boundary redaction path that reuses existing redaction primitives and coordinates with the concrete file-read/exec leak issues instead of landing another broad unreviewed privacy layer.

## Reproduction Assessment

Yes, partially from source: current `main` has log/transcript/tool-detail redaction, but no general `outputSanitization` config and the user-facing reply sanitizer does not invoke sensitive-data redaction. I did not run a live agent/channel reproduction with a synthetic secret.

## Solution Assessment

Unclear as proposed: the goal fits security priorities, but the proposed broad middleware/config needs maintainer decisions on defaults, false positives, scope, and interaction with existing transcript/tool redaction. A smaller central boundary using existing redaction helpers is the safer direction once policy is decided.
