## Summary

Keep open: current main has log/transcript redaction and plugin hook seams, but it does not implement the requested automatic sanitizer for sensitive user-visible agent output; the remaining work needs maintainer security/product direction rather than a narrow autonomous bug fix.

## What This Changes

The issue asks for configurable automatic redaction of API keys, tokens, passwords, PII, and other sensitive values from agent/tool outputs before they reach users.

## Best Possible Solution

Maintainers should decide the security boundary first, then implement either a narrow core/plugin hook-based sanitizer with explicit defaults or route the capability to a vetted plugin if current hooks are sufficient.

## Reproduction Assessment

Yes, at source level: current delivery strips internal scaffolding and supports rewrite/cancel hooks, but it does not automatically apply sensitive-value redaction to all outbound channel text. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Unclear. The requested protection is useful, but the best solution depends on a maintainer decision about whether this should be a core safety default, an opt-in config surface, or a plugin-owned policy using the existing hook seams.
