## Summary

Keep open: current main has scoped log/transcript/tool/UI redaction and several plugin/provider hook seams, but it still does not implement the requested automatic sensitive-data sanitizer across agent responses, tool outputs, channel delivery, and provider-bound content; related privacy-filter PRs are closed unmerged, so a maintainer security/product decision remains necessary.

## What This Changes

Request automatic configurable sanitization for sensitive values in agent responses, tool outputs, and related output paths.

## Best Possible Solution

Define separate owner-approved boundaries for channel-output redaction and provider-egress/tool-result privacy, then add the narrow core or plugin contract needed for the missing boundary with safe defaults, docs, and regression proof.

## Reproduction Assessment

Yes, at source level: current main exposes scoped redaction and hook seams but no configured `outputSanitization` or universal sensitive-data filter for all agent output and provider-bound paths.

## Solution Assessment

No, the proposed single middleware/config block is not the best safe implementation by itself because it combines channel display, provider egress, tool payloads, reversible state, custom regex, and reporting; the safer path separates those boundaries first.

## Risks / Open Questions

- A global regex sanitizer can corrupt legitimate code, identifiers, quoted evidence, or operational output while still missing unknown secret formats.
- Provider-egress replacement and inbound restoration can change tool arguments or multi-turn state; incorrect mappings could mis-associate secrets or expose originals.
- Custom regexes, per-agent modes, reporting, and defaults create a security-sensitive config surface with ReDoS, upgrade, and operator-expectation implications.
- Channel-only redaction can make displayed output look safe after sensitive content has already reached an external model provider.
