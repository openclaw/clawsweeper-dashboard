## Summary

Keep open: current main and the latest release still provide scoped log, transcript, tool-payload, display, and plugin-hook redaction seams, but not automatic sensitive-data sanitization for all agent outputs or provider-bound content. The remaining work is security-sensitive and needs maintainer product direction before implementation.

## What This Changes

Request automatic output sanitization to prevent API keys, tokens, passwords, and PII from leaking through agent responses, tool output, and optional provider-bound content.

## Best Possible Solution

Define a narrow guardrail architecture with separate channel-delivery filtering, provider-egress protection, explicit state storage, defaults, migration, reporting, and plugin/API boundaries before any implementation.

## Reproduction Assessment

Yes at source level. Current main exposes scoped logging/transcript/tool/display redaction and plugin hooks, but no automatic all-output or provider-egress sanitizer config or implementation; I did not run a live leak transcript in this read-only review.

## Solution Assessment

No, the proposed single middleware/config block is not the best solution as written because it mixes channel redaction, provider egress, reversible state, reporting, regex policy, and defaults. The safer path is to split those trust boundaries and design the model-boundary protection explicitly.

## Risks / Open Questions

- A delivery-only sanitizer could make channel output appear safe after sensitive content has already reached an external model provider.
- A global regex sanitizer can corrupt legitimate code, identifiers, quoted evidence, or operational output while still missing unknown secret formats.
- Reversible provider-egress replacement would create sensitive session state and could mis-associate secrets, tool arguments, or restored values if lifecycle rules are wrong.
- New defaults, custom regex patterns, reporting, and per-agent modes would add security-sensitive config and upgrade behavior that maintainers need to define first.
