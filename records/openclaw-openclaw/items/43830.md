## Label Justifications

- `P1`: The issue covers a credible credential and PII exposure class in common agent workflows, but resolving it safely requires explicit security and product design rather than an emergency patch.
- `impact:security`: The requested feature controls whether secrets and personal data cross channel and model-provider trust boundaries.

## Summary

Current main still lacks a universal sensitive-data sanitizer: existing redaction protects logs, diagnostics, configuration displays, and selected tool/UI payloads, while assistant-visible sanitizers remove protocol artifacts rather than arbitrary credentials or PII. A channel-output sanitizer can already be implemented through the public `message_sending` plugin hook, but preventing sensitive prompts, system context, and tool results from reaching model providers requires a separate security-sensitive runtime boundary and product policy. The related reversible-filter PRs closed unmerged and do not provide a viable canonical implementation, so explicit maintainer security/product direction remains necessary.

## What This Changes

Requests configurable automatic sanitization of credentials, tokens, passwords, PII, and similar sensitive values in agent responses and tool output.

## Best Possible Solution

Treat channel delivery and provider egress as separate products: publish channel-output redaction through the existing `message_sending` plugin seam, and only add a narrow generic model-boundary hook or core guardrail after maintainers define the threat model, state ownership, defaults, and failure behavior.

## Reproduction Assessment

Yes at the source level: current main has scoped redaction and protocol sanitization but no universal configured secret/PII filter for provider-bound content or every agent output path. No live leak transcript was provided, so this is not a runtime reproduction.

## Solution Assessment

No. The proposed single middleware/config block combines channel redaction, provider-egress privacy, tool transformation, reporting, and reversible session state; the safer design separates those boundaries and uses the existing plugin hook where it already suffices.

## Risks / Open Questions

- A global regex sanitizer can silently corrupt legitimate code, identifiers, quoted evidence, or operational output while still missing unknown secret formats.
- Provider-egress replacement plus inbound restoration changes tool arguments and multi-turn state; incorrect mapping can mis-associate secrets, expose originals, or execute tools with altered values.
- User-defined regexes, per-agent modes, reporting, and defaults create a large security-sensitive configuration surface with ReDoS, upgrade, and operator-expectation implications.
- Channel-delivery filtering can make displayed output appear safe even though the sensitive content has already reached an external model provider.
