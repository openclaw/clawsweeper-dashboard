## Summary

Keep open. Current main has scoped redaction for logs, persisted transcripts, tool-detail storage, diagnostics, and internal runtime scaffolding, but it still lacks the requested automatic sensitive-value sanitizer for all agent-visible output and a dedicated output-sanitization policy.

## What This Changes

The issue requests configurable automatic sanitization of API keys, tokens, passwords, and PII in agent responses, tool outputs, and delivery paths.

## Best Possible Solution

Define the maintainer-approved security boundary first, then implement a bounded sanitizer path that reuses the existing redaction engine where appropriate and proves fresh-install, upgrade, delivery, transcript, and tool-output behavior.

Do we have a high-confidence way to reproduce the issue? Partially: source inspection shows no current automatic output-sanitization config or central response sanitizer, but no live end-to-end leak reproduction was run in this read-only review.
