## Summary

Keep open. Current main has best-effort redaction for logs, transcripts, diagnostics, tool payloads, and some user-facing formatting cleanup, but it does not implement the requested configurable automatic sanitization layer for final agent/channel outputs; the overlapping privacy/security PRs need maintainer direction.

## What This Changes

Request configurable automatic sanitization of sensitive data in agent responses, tool results, and related output paths.

## Best Possible Solution

Keep this as the tracking issue until maintainers choose one coherent privacy architecture and either land one of the active PRs or split a smaller core/plugin API with config, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows final replies are normalized and delivered without a secret-pattern redaction layer, while existing redaction is scoped to logging, transcripts, diagnostics, and tool/UI payloads.

## Solution Assessment

Unclear. The request identifies a real security gap, but the best fix should be the maintainer-selected architecture from the active privacy/security PRs rather than an isolated regex-only final-response filter.
