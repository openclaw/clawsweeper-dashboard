## Summary

Keep open: current main has scoped log, transcript, tool, and display redaction plus plugin hook seams, but not a universal sensitive-data sanitizer for all agent outputs or provider-bound content. Related implementation PRs are closed unmerged, and the remaining work needs maintainer security and product direction.

## What This Changes

The issue requests configurable automatic sanitization of secrets and PII across agent outputs, tool results, and final responses.

## Best Possible Solution

Define a narrow guardrail architecture: channel-delivery filtering through message hooks, provider-egress protection through an approved model-boundary/plugin API, and explicit state, default, migration, and failure-mode rules before implementation.

## Reproduction Assessment

Yes, source inspection is high-confidence: current main has scoped logging/transcript/tool/display redaction and hook seams, but no automatic all-output or provider-egress sanitizer. I did not run a live leak transcript in this read-only review.

## Solution Assessment

No, the proposed single middleware/config block is not the best solution as written because it mixes channel redaction, provider egress, reversible state, reporting, regex policy, and defaults. The safer path is to split those trust boundaries and design the model-boundary protection explicitly.

## Risks / Open Questions

- A delivery-only sanitizer can make channel output appear safe after sensitive content has already reached an external model provider.
- A global regex sanitizer can corrupt legitimate code, identifiers, quoted evidence, or operational output while still missing unknown secret formats.
- Reversible provider-egress replacement creates sensitive session state and can mis-associate secrets or tool arguments if the mapping lifecycle is wrong.
- User-defined patterns, defaults, reporting, and per-agent modes would add a security-sensitive config surface that needs upgrade and operator-expectation review.
