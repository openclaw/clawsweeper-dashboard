## Summary

Keep open. Current main has meaningful partial redaction for logs, persisted transcripts, tool-detail/UI payloads, diagnostics, and internal runtime scaffolding, but it still does not provide the requested automatic sensitive-value sanitizer for all agent-visible outputs or a dedicated output-sanitization config policy.

## What This Changes

The issue requests automatic sanitization of sensitive data in agent outputs, including configurable coverage for responses, tool outputs, redaction patterns, exceptions, and reporting.

## Best Possible Solution

Keep this open for a maintainer-approved security design that reuses the existing redaction engine where appropriate, defines the exact final-output/tool-output boundary, and adds focused coverage for delivery, transcripts, config, docs, and upgrade behavior.

## Reproduction Assessment

No live reproduction was run, but source inspection gives a clear reproduction basis: final delivery strips internal scaffolding while current secret redaction is scoped to logs, transcripts, tool details, diagnostics, and related safety surfaces. The requested automatic sensitive-data filtering for all agent outputs remains absent in current main and v2026.6.8.

## Solution Assessment

No, the issue body is a plausible direction rather than a ready best fix. The best path is a maintainer/security-approved policy that decides default behavior and scope before changing the core delivery/config surface.
