## Summary

Keep open: current main and v2026.6.6 still expose plugin identity and hook-time `ctx.agentId`, but not register-time agent identity on `OpenClawPluginApi`. This remains a core plugin SDK design request, not a ClawHub/plugin-close item, and no related PR appears to own it.

## What This Changes

The issue requests `agentId` and preferably `agentName` on the `OpenClawPluginApi` object passed to plugin `register()` for per-agent diagnostics and behavior.

## Best Possible Solution

Define a narrow additive registration-context contract for full runtime plugin registration, explicitly document what setup/discovery modes receive, and update SDK type, builder, baseline, docs, and focused loader/runtime tests together.

## Reproduction Assessment

Yes. Source inspection on current main shows `OpenClawPluginApi` and `buildPluginApi` lack register-time agent identity, while hook callbacks already have `ctx.agentId`; I did not run `openclaw doctor`.

## Solution Assessment

Unclear. Adding agent identity is plausibly the right direction, but the best fix is a maintainer-chosen SDK contract rather than blindly adding flat fields without registration-mode semantics.
