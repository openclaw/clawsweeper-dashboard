## Summary

Keep open: current main and v2026.6.8 still lack register-time agent identity in `OpenClawPluginApi`, while hook contexts already carry `agentId`; this remains a public Plugin SDK design decision rather than conservative cleanup.

## What This Changes

Request to add `agentId` and preferably `agentName` to the plugin API passed to `register()` for multi-agent diagnostics and per-agent plugin behavior.

## Best Possible Solution

Define an additive, documented Plugin SDK registration context with explicit absent semantics for non-agent modes, plus focused SDK contract tests and docs.

## Reproduction Assessment

Yes. Source inspection shows current `OpenClawPluginApi`, `buildPluginApi`, and runtime plugin loading lack register-time `agentId` or `agentName`, while hook contexts already expose `ctx.agentId`; I did not run `openclaw doctor`.

## Solution Assessment

Unclear. Exposing agent identity may be the right direction, but the best SDK shape needs maintainer design because registration modes are not all agent-specific.
