## Summary

Keep open: current main still does not expose register-time agent identity on `OpenClawPluginApi`, while hook callbacks do receive `agentId`; the request remains a valid plugin SDK product/API decision rather than a cleanup close.

## What This Changes

Expose the current agent id and ideally agent name on the plugin API object passed to plugin `register()` so per-agent plugin initialization can log or branch by agent context.

## Best Possible Solution

Choose a stable plugin SDK registration-context contract, then add focused implementation and tests proving per-agent plugin registration receives the intended identity without making plugins depend on unstable runtime internals.

## Reproduction Assessment

Yes from source: `OpenClawPluginApi` has no `agentId` or `agentName`, while hook contexts already expose `agentId`, so the reported register-time gap is clear without running `openclaw doctor`. I did not execute a runtime doctor scenario because this read-only review is enough to establish the API absence.

## Solution Assessment

Unclear: exposing identity on `OpenClawPluginApi` is plausible, but the best permanent contract may be a separate registration context or core-owned diagnostic labeling. Because this is public SDK shape, maintainers should choose the API before implementation.
