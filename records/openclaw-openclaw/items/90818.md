Request to expose the active agent ID and name to plugin `register(api)` calls for per-agent diagnostics and initialization behavior.

## Best Possible Solution

Define a stable registration-context contract for agent identity, then update the API builder, loader/registry path, API baseline, docs, and focused SDK tests together.

## Reproduction Assessment

Yes. Source inspection on current main shows `OpenClawPluginApi`, `buildPluginApi`, and the loader register path have no register-time `agentId` or `agentName`, while hook contexts already expose `agentId`.

## Solution Assessment

Unclear. Exposing fields on `OpenClawPluginApi` is plausible, but maintainers should first decide whether permanent identity belongs on API fields, a separate registration context, or core-owned diagnostics.
