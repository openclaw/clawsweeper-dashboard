## Summary

Keep open: current main still lacks register-time agent identity on `OpenClawPluginApi`, while hook contexts already carry `agentId`; this remains a public Plugin SDK API decision rather than conservative cleanup.

## What This Changes

The issue requests adding `agentId` and preferably `agentName` to the `OpenClawPluginApi` object passed to plugin `register()` so initialization logs and per-agent setup can identify the current agent.

## Best Possible Solution

Keep this issue open until maintainers decide whether register-time agent identity belongs directly on `OpenClawPluginApi`, in a separate registration context, or in core-owned diagnostic labeling.

## Reproduction Assessment

Yes from source inspection: `OpenClawPluginApi` and `buildPluginApi` do not include `agentId` or `agentName`, while `PluginHookAgentContext` already includes `agentId`. I did not run `openclaw doctor`, so the evidence is source-reproducible rather than a live runtime reproduction.

## Solution Assessment

Unclear: exposing identity on `OpenClawPluginApi` is plausible, but the best permanent contract may be a separate registration context or core-owned diagnostics. Because this is a public Plugin SDK shape, maintainer API direction should come first.
