## Summary

Keep open: current main and the latest release still lack register-time agent identity in `OpenClawPluginApi`; hook callbacks already carry `agentId`, so the remaining work is a plugin SDK design decision rather than stale cleanup.

## What This Changes

Expose the current agent id and optionally display name on `OpenClawPluginApi` during plugin `register()` so plugin initialization can identify the active agent.

## Best Possible Solution

Define an additive, documented Plugin SDK context shape for full runtime registration, with explicit absence semantics for setup/CLI metadata modes and focused contract tests/docs.

## Reproduction Assessment

Yes. Source inspection shows `OpenClawPluginApi` and `buildPluginApi` lack register-time `agentId`/`agentName`, while hook contexts already expose `ctx.agentId`; I did not run `openclaw doctor`.

## Solution Assessment

Unclear. Exposing agent identity may be the right direction, but the best SDK shape needs maintainer design because registration modes are not all agent-specific.
