## Summary

Keep open: this remains a valid plugin SDK product/API request. Current main exposes plugin identity and hook-time agent context, but register-time `OpenClawPluginApi` still does not carry `agentId` or `agentName`, so plugin initialization cannot identify which agent instance is registering without a new SDK decision.

## What This Changes

Expose the current agent identity on the `OpenClawPluginApi` object passed to plugin `register()` so plugin initialization can log or branch by agent.

## Best Possible Solution

Define the public SDK contract for register-time agent context, update the API type and loader construction path consistently, and document whether `agentId` and `agentName` are available for every plugin load mode or only agent-scoped loads.

## Reproduction Assessment

Yes for source reproduction: current main's plugin API type and loader registration path show no register-time `agentId` or `agentName`, while hook dispatch does pass agent context later. I did not run `openclaw doctor`, because the requested gap is directly visible in source.

## Solution Assessment

Unclear until maintainers choose the SDK shape. Exposing agent identity on `OpenClawPluginApi` is plausible, but the best solution needs a compatibility decision about availability and semantics across non-agent and agent-scoped plugin registration.
