## Summary

Keep open: current main and v2026.6.8 still do not expose register-time `agentId` or `agentName` on `OpenClawPluginApi`. The remaining work is a public plugin SDK design decision because registration modes are not always tied to one agent.

## What This Changes

The issue requests adding `agentId` and preferably `agentName` to `OpenClawPluginApi` during plugin `register()` for per-agent initialization logs and behavior.

## Best Possible Solution

Define an additive, documented Plugin SDK context shape for full agent-scoped runtime registration, with explicit absence semantics for non-agent modes and focused contract tests/docs.

## Reproduction Assessment

Yes, at source level. Current `OpenClawPluginApi` and `buildPluginApi` lack register-time `agentId` or `agentName`, while hook contexts already expose `ctx.agentId`; I did not run `openclaw doctor`.

## Solution Assessment

Unclear. Exposing agent identity may be the right direction, but the best SDK shape needs maintainer design because registration modes are not all agent-specific.
