## Summary

Keep this open: current `main` still has the requested host shapes only on internal surfaces, while the plugin SDK public surface and docs do not expose canonical status/model response types for plugin authors.

## What This Changes

The issue requests exporting existing host status and model catalog response types from the plugin SDK public API so plugins can bind to gateway response shapes.

## Best Possible Solution

Add a narrow, documented SDK type surface for `StatusSummary`, `SessionStatus`, and `ModelCatalogEntry`, preferably on an existing focused gateway/model subpath, with API baseline/export coverage kept in sync.

## Reproduction Assessment

Yes. Source inspection on current `main` shows the requested types exist internally and are used by gateway status/model paths, but no plugin SDK public re-export exists.

## Solution Assessment

Yes on the direction, with one implementation caveat: this should be a focused SDK contract update rather than only growing the broad root barrel. Keeping docs and API baseline aligned is the maintainable path.
