## Summary

Keep open: current main still does not provide the requested canonical SDK type surface for gateway status/model data. The underlying host types exist, and `ModelCatalogEntry` is reachable indirectly through `plugin-sdk/agent-runtime`, but `StatusSummary` and `SessionStatus` are not exposed from a plugin-facing gateway/status contract, and choosing that public surface should be coordinated with the still-open gateway RPC request.

## What This Changes

Expose canonical `StatusSummary`, `SessionStatus`, and `ModelCatalogEntry` types through the public plugin SDK for plugins consuming host status and model RPC data.

## Best Possible Solution

Promote these as a deliberate, documented, type-only SDK contract on the narrow gateway/runtime surface used by the future host RPC accessors, with API-baseline/export coverage.

## Reproduction Assessment

Not applicable as a feature request. Source inspection does confirm the gap: the host types exist internally, while the current plugin-sdk root and gateway-runtime surfaces do not expose the requested status/session types.

## Solution Assessment

No, not exactly as proposed. A simple root-barrel re-export is not clearly the best fit because the local SDK boundary guidance prefers narrow documented subpaths; the safer solution is to attach the types to the gateway access surface being designed in #76756.
