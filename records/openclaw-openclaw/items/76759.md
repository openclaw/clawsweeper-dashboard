## Summary

Keep open. Current main still lacks the requested status type exports from the plugin SDK public surface, while the related open PR #77184 appears to be the right implementation track rather than a reason to close this issue before merge.

## What This Changes

Expose canonical `StatusSummary`, `SessionStatus`, and `ModelCatalogEntry` types through supported plugin SDK imports so plugins do not copy host response shapes.

## Best Possible Solution

Land a narrow additive SDK change that exports the status payload types from a focused subpath, updates API baseline/docs/changelog coverage, and documents or aliases the existing model catalog import path as maintainers prefer.

## Reproduction Assessment

Yes, at source level. Current main exposes the `command-status` subpath but that file only exports command/help builders, while the requested status types exist in `src/commands/status.types.ts`; `ModelCatalogEntry` is already reachable through `plugin-sdk/agent-runtime`.

## Solution Assessment

Yes, the narrow additive re-export direction is the maintainable fix if maintainers accept the public API contract. Copying these host shapes into plugins would keep the drift risk that this issue documents.
