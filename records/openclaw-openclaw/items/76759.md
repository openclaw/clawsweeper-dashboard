## Summary

Keep open. Current main still lacks the requested public plugin-sdk type exports, and this is a valid SDK feature request, but the exact public API shape needs maintainer review because it promotes host-internal status/model shapes and is tied to the still-open gateway accessor work in #76756.

## What This Changes

Request to export StatusSummary, SessionStatus, and ModelCatalogEntry from the public plugin-sdk surface so plugins can type gateway status and model catalog responses.

## Best Possible Solution

Define a narrow, documented plugin-sdk type contract for status and model-list responses, preferably aligned with the curated read-only gateway accessors from #76756, with API baseline, export-map, docs, and regression coverage updated together.

## Reproduction Assessment

Not applicable as a runtime reproduction; this is an SDK API surface request. Source inspection does give high confidence that the named core types exist but are not exported from the current public plugin-sdk surface.

## Solution Assessment

No, not exactly as proposed. A simple root-barrel re-export is probably not the best default; a narrow documented type subpath or gateway-runtime contract aligned with #76756 is the safer maintainable direction.
