## Summary

Keep this PR open, but it is not merge-ready: the credentialed-URL guard is useful security hardening, yet the shared cron webhook normalizer also controls persisted runtime delivery and doctor migration, so maintainers need an explicit upgrade policy plus regression coverage before merge.

## What This Changes

The PR changes `normalizeHttpWebhookUrl` so HTTP(S) cron webhook URLs with parsed username or password userinfo return `null`.

## Best Possible Solution

Land the guard only after maintainers choose fail-closed, warning/doctor repair, or migration behavior for existing credential-bearing cron webhooks and add focused regression coverage.

## Reproduction Assessment

Yes. Current main and the latest release accept credential-bearing HTTP(S) cron webhook URLs in the shared helper, and the PR screenshot shows the patched helper rejecting one while accepting a normal webhook URL.

## Solution Assessment

No. The shared helper is a reasonable enforcement point, but unconditional rejection is only a plausible hardening step until stored-config upgrade behavior and doctor/runtime handling are explicitly covered.
