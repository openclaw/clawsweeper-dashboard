## Summary

Keep this PR open: the security hardening is plausible and the screenshot proves the helper behavior, but the shared helper change intentionally breaks a URL form accepted by current main and the latest release, so maintainer-approved upgrade policy is needed before merge.

## Best Possible Solution

Ship cron webhook userinfo rejection only after maintainers choose the upgrade policy and the patch covers create/update validation, stored dispatch, failure destinations, and legacy notify doctor migration.

## Reproduction Assessment

Yes, source-level. Current main and `v2026.6.8` accept credential-bearing HTTP(S) cron webhook URLs, and the PR screenshot shows the patched helper returning `null` for that shape.

## Solution Assessment

No, not yet. Helper-level rejection is a reasonable hardening point, but it is not the best complete fix until maintainers approve the upgrade behavior for stored webhook jobs and the patch covers those paths.
