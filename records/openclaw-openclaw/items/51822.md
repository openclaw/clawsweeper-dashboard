## Summary

Keep this PR open. Current main still accepts credential-bearing cron webhook URLs and the PR targets the right shared normalizer, but the branch is not safe to merge because the new rejection path can expose stored credentials in an existing raw warning and does not handle legacy fallback or doctor migration behavior.

## What This Changes

The PR adds a username/password check in the cron HTTP webhook URL normalizer so credential-bearing HTTP(S) webhook URLs return `null`.

## Best Possible Solution

Adopt userinfo rejection in the shared cron webhook normalizer together with redacted invalid-target diagnostics, legacy fallback warnings, doctor migration validation, regression coverage, and explicit migration/release handling.

## Reproduction Assessment

Yes. Source inspection of current main shows `normalizeHttpWebhookUrl` accepts credential-bearing HTTP(S) URLs, and the PR body includes terminal screenshot proof that the patched helper rejects one while accepting a plain URL.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch alone can leak credentials through existing invalid-target logging and break legacy/doctor migration paths without a clear warning or validation step.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:19-21`
  - body: When this guard makes an existing `delivery.to` with URL userinfo invalid, the scheduler falls into the invalid-webhook warning and logs `params.job.delivery.to` raw as `deliveryTo`. That can expose the credential-bearing URL in gateway logs, so redact or omit the target before landing the rejection path.
  - confidence: 0.91
- **[P2] Warn on rejected legacy cron.webhook targets:** `src/cron/webhook-url.ts:19-21`
  - body: Legacy `notify: true` jobs resolve global `cron.webhook` through this helper. With a credential-bearing legacy URL rejected, `resolveCronWebhookTarget` returns null, but the invalid-target warning only covers explicit webhook delivery jobs, so callbacks can stop with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:19-21`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` and deletes `notify` without using this normalizer. After this guard, doctor can migrate a legacy fallback into an invalid per-job webhook, so it should validate or warn instead.
  - confidence: 0.84
