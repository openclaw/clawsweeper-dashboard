## Summary

Keep open. The PR targets the right shared cron webhook normalizer and the attached terminal proof shows the helper behavior, but the current diff leaves source-reproducible runtime and migration side effects that can leak or silently strand credential-bearing legacy webhook configuration.

## What This Changes

The PR adds a username/password check to `normalizeHttpWebhookUrl` so credential-bearing HTTP(S) cron webhook URLs return `null`.

## Best Possible Solution

Reject URL userinfo through the shared normalizer while redacting invalid-target logs, surfacing legacy fallback diagnostics, guarding doctor migration, and adding focused tests plus release notes.

## Reproduction Assessment

Yes. Source inspection shows current main accepts credential-bearing http(s) webhook URLs, and the PR screenshot shows the proposed helper rejection; the remaining blockers are also source-reproducible in the runtime and doctor paths.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch is incomplete without redaction, legacy diagnostics, doctor safeguards, regression tests, and release notes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:20-21`
  - body: When this guard makes an existing credential-bearing `delivery.to` invalid, finished-run dispatch reaches the skipped-delivery warning and logs `params.job.delivery.to` as `deliveryTo`. That exposes the same raw userinfo URL on every run, so the invalid-target path needs to redact or omit the target before this hardening can merge.
  - confidence: 0.91
- **[P2] Warn on invalid legacy cron.webhook fallback:** `src/cron/webhook-url.ts:20-21`
  - body: Legacy `notify: true` jobs resolve the global `cron.webhook` through this normalizer. With this guard, a credentialed legacy URL becomes `null`, but the only invalid-target warning is for explicit `delivery.mode === "webhook"` jobs, so those callbacks silently stop instead of telling users how to migrate.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:20-21`
  - body: Doctor still migrates legacy notify fallback jobs by copying `cron.webhook` into `delivery.to` and deleting `notify`. If that global webhook contains userinfo, this patch turns the repaired job into an invalid per-job webhook that the scheduler skips.
  - confidence: 0.84
