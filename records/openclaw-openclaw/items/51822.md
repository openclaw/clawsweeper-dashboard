## Summary

Keep open: this is useful security hardening, but current main still accepts cron webhook URL userinfo and the PR needs fixes before merge because rejection currently interacts badly with logging, legacy fallback, and doctor migration paths.

## What This Changes

The PR adds a guard in `normalizeHttpWebhookUrl` so parsed HTTP(S) cron webhook URLs with `URL.username` or `URL.password` return `null`.

## Best Possible Solution

Adopt userinfo rejection in the shared cron webhook normalizer together with redacted invalid-target diagnostics, explicit legacy fallback warnings, doctor migration validation or refusal, focused regression tests, and release/migration notes.

## Reproduction Assessment

Yes for source-level review. Current main accepts HTTP(S) cron webhook URLs without checking `URL.username` or `URL.password`, and the PR screenshot shows the patched helper rejecting a credential-bearing URL while accepting a plain one.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch needs redaction, legacy fallback handling, doctor migration validation, tests, and an explicit migration policy before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact stored webhook targets before rejecting them:** `src/cron/webhook-url.ts:20`
  - body: When this guard rejects an existing stored `delivery.mode = "webhook"` target containing URL userinfo, `dispatchGatewayCronFinishedNotifications` still logs `params.job.delivery.to` as `deliveryTo`. That can put the full credential-bearing URL in gateway logs on every finished run, so redact or omit the invalid target before this rejection ships.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook is rejected:** `src/cron/webhook-url.ts:20`
  - body: Legacy `notify: true` jobs use the global `cron.webhook` through this helper, but the only invalid-target warning is behind explicit webhook delivery. After this guard, a credentialed legacy URL becomes a silent no-op, so surface a redacted warning or migration hint for that path.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` and deletes `notify` without normalizing it. With this guard, a credentialed legacy webhook can be migrated into an invalid per-job target that the scheduler skips; validate or refuse the migration first.
  - confidence: 0.84
