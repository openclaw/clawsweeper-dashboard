## Summary

Keep open: this is useful security hardening, but the patch remains incomplete because rejected stored webhook targets can still be logged raw, legacy fallback rejection is silent, and doctor migration does not validate the new policy.

## What This Changes

The PR adds a guard in the cron webhook URL normalizer so HTTP(S) URLs with parsed username or password fields are rejected.

## Best Possible Solution

Ship URL userinfo rejection through the shared cron webhook normalizer together with redacted invalid-target diagnostics, explicit legacy fallback handling, doctor validation or refusal, focused regression tests, and migration notes.

## Reproduction Assessment

Yes. Current main and v2026.5.12 accept credential-bearing HTTP(S) cron webhook URLs from source inspection, the PR proof shows the patched helper rejecting them, and the logging plus doctor-migration blockers are source-reproducible from the current call paths.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch is not complete until rejected targets are redacted, legacy cron.webhook rejection is surfaced, doctor avoids unsafe migration, and regression coverage is added.

Overall correctness: patch is incorrect

Full review comments:

  - confidence: 0.91

  - body: Legacy notify:true jobs pass cfg.cron.webhook through this normalizer, but the invalid-target warning only covers explicit webhook delivery. A credentialed legacy webhook would stop delivering finished-run callbacks without a redacted operator warning or migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:19-20`
  - body: openclaw doctor --fix copies cfg.cron.webhook into delivery.to and removes notify without using this normalizer. Once credentialed URLs are rejected, doctor can migrate a legacy fallback into per-job webhook metadata that the scheduler later skips.
  - confidence: 0.84
