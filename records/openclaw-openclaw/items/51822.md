## Summary

Keep open: the credentialed-URL guard is a useful security hardening direction, but the PR changes shipped cron webhook acceptance behavior without maintainer-approved upgrade, runtime-delivery, and doctor-migration handling for existing credential-bearing cron configs.

## What This Changes

Adds a username/password check to `src/cron/webhook-url.ts` so `normalizeHttpWebhookUrl` returns `null` for HTTP(S) cron webhook URLs with URL userinfo.

## Best Possible Solution

Land the guard only after maintainers choose fail-closed, migration, warning, or compatibility behavior for existing credential-bearing cron webhooks and add focused regression coverage across the shared call sites.

## Reproduction Assessment

Yes. Current main and `v2026.6.9` accept credential-bearing HTTP(S) cron webhook URLs in the shared helper, and the inspected terminal screenshot shows the patched helper rejecting one while accepting a normal webhook URL.

## Solution Assessment

No. The shared helper is a plausible enforcement point, but unconditional rejection is not the best fix until stored-config upgrade behavior and doctor/runtime handling are explicitly chosen and tested.
