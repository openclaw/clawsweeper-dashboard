## Summary

Keep open: the hardening targets a real credential-leak vector, but the one-line normalizer change is incomplete because the current invalid-target and migration paths can leak credentials or silently break legacy cron webhooks.

## What This Changes

Adds a username/password check to `normalizeHttpWebhookUrl` so HTTP(S) cron webhook URLs with parsed URL userinfo return `null`.

## Best Possible Solution

Ship the userinfo rejection with redacted invalid-target diagnostics, explicit legacy fallback warnings, doctor validation or refusal, focused regression tests, and migration or release notes.

## Reproduction Assessment

Yes, at source level. Current main accepts credential-bearing HTTP(S) webhook URLs after only a protocol check, and the PR proof shows the patched helper rejecting one; I did not run an end-to-end cron job in this read-only review.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch is incomplete until rejected targets are redacted, legacy fallback rejection is surfaced, doctor avoids unsafe migration, and regression coverage is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:19-20`
  - body: When this guard rejects an existing `delivery.mode = "webhook"` target containing userinfo, `dispatchGatewayCronFinishedNotifications` falls into the invalid-target warning and logs `params.job.delivery.to` as `deliveryTo`. That can write the credential-bearing URL to logs on every run, so redact or omit the rejected target before merging.
  - confidence: 0.91
- **[P2] Surface rejected legacy cron.webhook targets:** `src/cron/webhook-url.ts:19-20`
  - body: Legacy `notify: true` jobs resolve `cfg.cron.webhook` through this helper, but the invalid-target warning only runs for explicit `delivery.mode = "webhook"`. After this new `null` result, credentialed legacy fallback deliveries become silent no-ops with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:19-20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` for legacy notify jobs without validating it. With this rejection in place, doctor can convert a working legacy fallback into stored metadata the scheduler later rejects.
  - confidence: 0.84
