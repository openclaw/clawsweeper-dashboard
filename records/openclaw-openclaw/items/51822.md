## Summary

Keep open: the hardening direction is valid, but the patch is incomplete because the new invalid-URL path can expose existing credential-bearing cron webhook URLs in logs and can break legacy fallback/migration paths without useful diagnostics.

## What This Changes

The PR adds a username/password check to cron's shared HTTP webhook URL normalizer so credential-bearing webhook URLs return `null`.

## Best Possible Solution

Reject webhook URL userinfo through the shared normalizer while redacting rejected targets, warning or refusing invalid legacy fallback migration, adding focused regression tests, and documenting the breaking security hardening in the changelog.

## Reproduction Assessment

Yes, source-reproducible. Current main accepts `https://user:pass@example.invalid/hook` by source inspection, and the proposed `null` path flows into stored delivery, legacy fallback, and doctor migration code paths that are visible in current main.

## Solution Assessment

No. The shared normalizer is the right place to reject userinfo, but the patch needs companion runtime redaction, legacy diagnostics, doctor safeguards, tests, and release notes before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact rejected stored webhook URLs:** `src/cron/webhook-url.ts:19-21`
  - body: When this guard makes an existing credential-bearing `delivery.to` invalid, finished-run notification dispatch falls into the skipped-delivery warning and logs the raw `params.job.delivery.to` as `deliveryTo`. That can write URL userinfo to logs on every run, so this rejection path needs to redact or omit the target before merge.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook becomes invalid:** `src/cron/webhook-url.ts:19-21`
  - body: Legacy `notify: true` jobs resolve the global `cron.webhook` through this same normalizer. With this guard, a credentialed legacy webhook becomes `null`, but the invalid-target warning only covers explicit webhook-mode jobs, so those callbacks stop without a migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:19-21`
  - body: Doctor still migrates `notify: true` jobs by copying `cron.webhook` into `delivery.to` and deleting `notify`. If that global webhook contains userinfo, this PR makes the migrated job invalid after `doctor --fix`, converting a working legacy fallback into a skipped per-job webhook.
  - confidence: 0.84
