## Summary

Keep this PR open: the hardening idea is useful, but current main and the latest release accept stored credential-bearing cron webhook URLs, and this helper-level rejection changes create/update, runtime delivery, failure notification, and doctor migration behavior without a maintainer-approved rollout policy or enough consumer proof.

## What This Changes

The branch adds a username/password check to `normalizeHttpWebhookUrl` so credential-bearing cron HTTP(S) webhook URLs return `null`.

## Best Possible Solution

Adopt credential-userinfo rejection only after maintainers define the rollout policy for existing stored cron webhook configs and the PR covers each affected webhook consumer with focused tests and real behavior proof.

## Reproduction Assessment

Yes, source-level: current main and `v2026.6.10` accept credential-bearing HTTP(S) cron webhook URLs through the shared normalizer, while the PR returns `null`. Live cron create/update and stored delivery behavior are not proven by the PR evidence.

## Solution Assessment

No, not yet. Helper-level rejection is a plausible guardrail, but the best merge shape needs an explicit upgrade policy plus coverage for create/update validation, runtime completion delivery, failure destinations, failure alerts, and doctor migration.
