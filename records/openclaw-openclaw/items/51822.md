## Summary

The guard is useful, but this PR should stay open because current main still accepts credential-bearing cron webhook URLs and the proposed rejection path has unresolved security and migration problems. The branch is not obsolete or already implemented on main.

## What This Changes

The PR adds a guard in `normalizeHttpWebhookUrl` so parsed HTTP(S) cron webhook URLs with a username or password return `null`.

## Best Possible Solution

Ship the userinfo rejection through the shared cron webhook normalizer together with redacted diagnostics, explicit legacy fallback warnings, doctor migration validation or refusal, focused regression tests, and migration/release notes.

## Reproduction Assessment

Yes. Source inspection on current main shows the normalizer accepts HTTP(S) URLs with userinfo, and the PR body terminal proof shows the patched helper rejecting a credential URL while accepting a plain URL.

## Solution Assessment

No. The shared helper is the right place for this policy, but the current patch needs redacted invalid-target diagnostics, legacy fallback handling, doctor migration validation or refusal, and regression coverage before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact stored webhook targets before rejecting them:** `src/cron/webhook-url.ts:20`
  - body: With this new `null` path, an existing stored `delivery.to` containing URL userinfo reaches the invalid-target warning in `dispatchGatewayCronFinishedNotifications`, which logs the raw value as `deliveryTo`. Redact or omit that target before making credential-bearing URLs invalid.
  - confidence: 0.91
- **[P2] Warn when rejected legacy cron.webhook disables delivery:** `src/cron/webhook-url.ts:20`
  - body: Legacy `notify: true` jobs resolve the global `cron.webhook` through this helper, but the invalid-target warning only runs for explicit `delivery.mode === "webhook"` jobs. If this guard rejects a credentialed legacy URL, finished-run callbacks stop without a migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` and deletes `notify` without using this normalizer. After this guard lands, a credentialed legacy fallback can be migrated into an invalid per-job webhook that the scheduler skips.
  - confidence: 0.84
