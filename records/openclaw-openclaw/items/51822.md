## Summary

Keep open. Current main still accepts credential-bearing cron webhook URLs, but this PR’s narrow guard introduces unsafe invalid-target behavior around raw logging and legacy migration, so it needs maintainer/security handling before merge.

## What This Changes

The PR adds a `URL.username`/`URL.password` rejection to `src/cron/webhook-url.ts` for cron HTTP(S) webhook URLs.

## Best Possible Solution

Ship userinfo rejection through the shared cron webhook normalizer together with redacted diagnostics, explicit legacy fallback warnings, doctor migration validation or refusal, focused tests, and release/migration notes.

## Reproduction Assessment

Yes. Source inspection on current main shows the normalizer accepts HTTP(S) URLs with userinfo, and the PR body terminal screenshot shows the patched helper rejecting a credential URL while accepting a plain URL.

## Solution Assessment

No. The shared helper is the right seam, but this patch needs redacted invalid-target diagnostics, legacy fallback handling, doctor migration validation or refusal, and regression coverage before the policy is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact stored webhook targets before rejecting them:** `src/cron/webhook-url.ts:20`
  - body: When this guard rejects an existing stored `delivery.to` with URL userinfo, `dispatchGatewayCronFinishedNotifications` logs the raw `params.job.delivery.to` as `deliveryTo`. That can write the credential-bearing URL to gateway logs on every finished run, so redact or omit the invalid target before shipping the rejection.
  - confidence: 0.91
- **[P2] Warn when rejected legacy cron.webhook disables delivery:** `src/cron/webhook-url.ts:20`
  - body: Legacy `notify: true` jobs run the global `cron.webhook` through this helper, but the invalid-target warning only covers explicit webhook delivery jobs. If the new guard rejects a credentialed legacy URL, finished-run callbacks stop without a migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` and deletes `notify` without using this normalizer. With this guard, a credentialed legacy fallback can be migrated into an invalid per-job webhook that the scheduler skips.
  - confidence: 0.84
