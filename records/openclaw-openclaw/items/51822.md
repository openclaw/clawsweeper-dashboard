## Summary

Keep open: the guard is useful security hardening, but the shared normalizer also controls stored cron delivery and doctor migration, so existing credential-bearing webhook configs can stop delivery after upgrade without an approved migration or fail-closed policy.

## What This Changes

The PR adds a parsed URL username/password rejection to the shared cron webhook URL normalizer.

## Best Possible Solution

Land the guard only after maintainers choose fail-closed, migration, warning, or compatibility behavior for existing credential-bearing cron webhooks and add focused regression coverage across the shared call sites.

## Reproduction Assessment

Yes, source-level. Current main and `v2026.6.9` accept credential-bearing HTTP(S) cron webhook URLs in the shared helper, and the inspected terminal screenshot shows the patched helper rejecting one while accepting a normal webhook URL.

## Solution Assessment

No. The shared helper is a plausible enforcement point, but unconditional rejection is not the best fix until stored-config upgrade behavior and doctor/runtime handling are explicitly chosen and tested.
