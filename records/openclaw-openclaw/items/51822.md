## Summary

Keep this PR open for maintainer review: it is a focused cron security hardening, but it changes shipped stored-webhook behavior by making credential-bearing HTTP(S) webhook URLs invalid across validation and runtime delivery paths.

## What This Changes

Adds a userinfo guard to `normalizeHttpWebhookUrl` so cron webhook URLs with a parsed username or password return `null`.

## Best Possible Solution

Ship cron webhook userinfo rejection only after maintainers approve the fail-closed upgrade policy and the patch covers create/update validation, stored runtime delivery, failure destinations, and doctor legacy notify migration behavior.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.8` return credential-bearing HTTP(S) URLs from the helper, while the PR diff and inspected terminal screenshot show the helper returning `null` after the new guard; I did not run an end-to-end cron delivery scenario.

## Solution Assessment

No, not yet. Helper-level rejection is a plausible hardening point, but it is not the best complete fix until the upgrade behavior for stored webhook jobs is approved and covered.
