## Summary

Keep open: the security direction is useful, but the current patch creates a raw-credential logging path and leaves legacy/doctor cron migration behavior unsafe, so it should not merge as-is.

## What This Changes

The PR changes `normalizeHttpWebhookUrl` to return `null` for parsed HTTP(S) webhook URLs that contain a username or password.

## Best Possible Solution

Reject URL userinfo through the shared normalizer after adding redacted invalid-target diagnostics, legacy fallback warnings, doctor validation, focused regression coverage, and maintainer-owned release notes.

## Reproduction Assessment

Yes, at source level. Current main accepts HTTP(S) webhook URLs after only protocol validation, and the PR body’s terminal proof shows the patched helper rejecting a credential URL.

## Solution Assessment

No. The shared normalizer is the right policy point, but the patch needs adjacent redaction, legacy diagnostics, doctor migration guards, tests, and maintainer approval for the backward-incompatible security policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact skipped credential webhook targets:** `src/cron/webhook-url.ts:20-22`
  - body: When this guard makes a stored `delivery.to` like `https://user:pass@host` invalid, the finished-notification invalid-target branch logs `params.job.delivery.to` as `deliveryTo`. That writes the credential-bearing URL to logs on every skipped delivery, so the rejection path needs to redact or omit the target before this lands.
  - confidence: 0.91
- **[P2] Warn on invalid legacy webhook fallback:** `src/cron/webhook-url.ts:20-22`
  - body: Legacy `notify: true` jobs still pass global `cron.webhook` through this normalizer. Once userinfo URLs return `null`, `resolveCronWebhookTarget` has no target and the only invalid-target warning is gated to explicit per-job webhook delivery, so legacy callbacks silently stop without a migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:20-22`
  - body: `openclaw doctor --fix` copies global `cron.webhook` into `delivery.to` when migrating `notify: true`. With this new rejection, a credentialed legacy webhook is rewritten into an invalid per-job webhook that the scheduler skips; doctor should validate and warn instead of writing the unusable target.
  - confidence: 0.84
