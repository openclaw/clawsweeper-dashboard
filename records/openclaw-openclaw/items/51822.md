## Summary

Keep this PR open: current main still accepts credential-bearing cron webhook URLs, so the hardening is not obsolete, but the patch is not merge-ready because the shared helper also controls stored runtime delivery and doctor legacy migration paths that need an explicit upgrade policy and focused coverage.

## What This Changes

The PR makes `normalizeHttpWebhookUrl` reject HTTP(S) cron webhook URLs when the parsed URL contains a username or password.

## Best Possible Solution

Land the credentialed-webhook guard only after maintainers choose the stored-config upgrade behavior and the PR covers the affected validation, runtime delivery, and doctor migration paths.

## Reproduction Assessment

Yes. Source inspection shows current main accepts credential-bearing cron webhook URLs in the shared helper, and the inspected terminal screenshot shows the PR branch returning `null` for the same helper case.

## Solution Assessment

No. The shared helper is a good central guard location, but unconditional fail-closed behavior is only a plausible hardening step until stored webhook configs and doctor migration have explicit upgrade handling.
