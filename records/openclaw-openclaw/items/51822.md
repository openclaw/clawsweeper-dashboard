## Summary

Keep open. The hardening is useful, but the current patch makes credential-bearing stored webhook URLs hit a raw invalid-target log path and does not handle legacy fallback or doctor migration behavior for this backward-incompatible policy change.

## What This Changes

The PR adds a username/password check to cron's shared HTTP webhook URL normalizer.

## Best Possible Solution

Adopt userinfo rejection in the shared cron webhook normalizer with redacted diagnostics, legacy fallback warnings, doctor migration validation or refusal, regression coverage, and explicit migration/release handling.

## Reproduction Assessment

Yes. Source inspection on current main shows `normalizeHttpWebhookUrl` accepts `https://user:pass@...` because it only checks protocol, and the PR body terminal screenshot shows the patched helper rejecting that input while accepting a plain HTTPS URL.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch should also redact invalid-target logs and handle legacy `cron.webhook` plus doctor migration before shipping.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:20`
  - body: When this guard makes an existing `delivery.mode = "webhook"` target with URL userinfo invalid, `dispatchGatewayCronFinishedNotifications` logs `deliveryTo: params.job.delivery.to` raw. That can write the credential-bearing URL to gateway logs on every run, so the invalid-target diagnostic needs to redact or omit the target before this rejection ships.
  - confidence: 0.91
- **[P2] Warn on rejected legacy cron.webhook:** `src/cron/webhook-url.ts:20`
  - body: Legacy `notify: true` jobs run global `cron.webhook` through this helper. If this new guard rejects that legacy URL, `resolveCronWebhookTarget` returns null and no warning fires because the invalid-target warning only checks explicit webhook delivery jobs, so callbacks stop without migration guidance.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` and deletes `notify` without this normalizer. With the new guard, a credentialed legacy fallback is migrated into an invalid per-job webhook that the scheduler skips; doctor should warn or refuse that migration.
  - confidence: 0.84
