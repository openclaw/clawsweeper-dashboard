## Summary

Keep open: this is useful cron security hardening, but it is not merge-ready because the shared webhook helper also controls stored cron delivery and doctor migration, so existing credential-bearing webhook configs can stop delivering without a maintainer-approved upgrade policy and focused tests.

## What This Changes

The PR changes `normalizeHttpWebhookUrl` so HTTP(S) cron webhook URLs with parsed username or password userinfo are rejected.

## Best Possible Solution

Land the credentialed-webhook guard only with maintainer-approved upgrade behavior for existing stored cron webhook configs and regression tests across validation, runtime delivery, failure destinations, and doctor migration.

## Reproduction Assessment

Yes. Current main source shows credential-bearing HTTP(S) cron webhook URLs are accepted by the shared helper, and the inspected PR screenshot shows the patched helper rejecting one.

## Solution Assessment

No. The shared helper is a reasonable guard point, but unconditional fail-closed behavior is only a plausible hardening step until stored-config upgrade behavior and doctor/runtime handling are explicitly covered.
