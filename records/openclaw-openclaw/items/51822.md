## Summary

This PR should stay open: the security hardening direction is valid, and the helper-level proof is sufficient, but the current patch makes existing credential-bearing cron webhook configs fail through unsafe upgrade paths that can leak secrets or silently stop delivery.

## What This Changes

The PR adds a guard to `normalizeHttpWebhookUrl` so cron webhook URLs with parsed username or password are rejected.

## Best Possible Solution

Ship the userinfo rejection with redacted invalid-target diagnostics, invalid legacy-fallback warnings, doctor validation/refusal for unsafe migrations, focused regression tests, and maintainer-owned release-note migration guidance.

## Reproduction Assessment

Yes, at source level. Current main accepts valid HTTP(S) cron webhook URLs, and the PR's new guard clearly makes credential-bearing URLs normalize to `null`, which flows into the explicit delivery, legacy fallback, and doctor migration paths inspected above.

## Solution Assessment

No, not yet. Rejecting embedded credentials is a sound direction, but the patch is not the best complete solution until the upgrade, logging, legacy fallback, and doctor migration paths are made safe and tested.
