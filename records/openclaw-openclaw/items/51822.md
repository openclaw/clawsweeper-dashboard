## Summary

Keep open: current main still accepts credential-bearing cron webhook URLs, and the PR targets the right shared helper, but the proposed patch creates security and migration gaps around stored jobs, legacy fallback, and doctor repair.

## What This Changes

The branch adds a username/password rejection guard to cron's shared HTTP webhook URL normalizer.

## Best Possible Solution

Adopt userinfo rejection in the shared cron webhook normalizer only with redacted invalid-target diagnostics, legacy fallback warnings, doctor migration validation or refusal, regression coverage, and explicit release/migration handling.

## Reproduction Assessment

Yes. Source inspection on current main shows normalizeHttpWebhookUrl accepts http(s) URLs without checking username/password, and the PR body terminal screenshot shows the patched helper rejecting a credential-bearing URL while accepting a plain URL.

## Solution Assessment

No. The shared normalizer is the right seam, but the patch should also redact invalid stored webhook diagnostics and handle legacy cron.webhook plus doctor migration before shipping.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:20`
  - body: When this guard rejects an existing stored delivery.to with URL userinfo, dispatchGatewayCronFinishedNotifications logs params.job.delivery.to as deliveryTo without redaction. That can write the credential-bearing URL to gateway logs on every finished run, so redact or omit the invalid target before this rejection ships.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook is rejected:** `src/cron/webhook-url.ts:20`
  - body: Legacy notify jobs run the global cron.webhook through this helper. If the new guard rejects that URL, resolveCronWebhookTarget returns null and the existing invalid-target warning does not fire for legacy jobs, so callbacks stop without a migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:20`
  - body: openclaw doctor --fix copies cron.webhook into delivery.to and deletes notify without using this normalizer. With the new guard, a credentialed legacy fallback can be migrated into an invalid per-job webhook that the scheduler skips, so doctor should warn or refuse that migration.
  - confidence: 0.84
