## Summary

Keep open: the cron webhook hardening is useful and current main plus v2026.6.6 still accept embedded URL credentials, but the PR is not merge-ready because the shared normalizer is used by persisted cron delivery and doctor migration paths, so fail-closed upgrade behavior can stop existing webhook callbacks without maintainer-approved migration or coverage.

## What This Changes

The PR tightens cron's shared HTTP(S) webhook URL normalizer to return null when the parsed URL contains a username or password.

## Best Possible Solution

Land the hardening only with maintainer-approved upgrade behavior and focused coverage across create/edit validation, runtime delivery, failure destinations, and doctor legacy notify migration.

## Reproduction Assessment

## Solution Assessment

No, not as-is. The helper-level rejection is a plausible hardening point, but the best fix needs an explicit upgrade policy and tests for persisted cron delivery and doctor migration paths before fail-closing existing jobs.
