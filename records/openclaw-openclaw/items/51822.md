## Summary

This PR should remain open: it hardens the right cron webhook URL normalizer, but the current patch still leaves concrete safety and migration gaps around raw rejected-target logging, legacy fallback behavior, and doctor migration.

## What This Changes

Adds a username/password check to `normalizeHttpWebhookUrl` so parsed HTTP(S) cron webhook URLs with embedded credentials return `null`.

## Best Possible Solution

Ship the userinfo rejection through the shared cron webhook normalizer together with redacted diagnostics, explicit legacy fallback warnings, doctor validation or refusal, focused tests, and release or migration notes.

## Reproduction Assessment

Yes, at source level. Current main accepts credential-bearing HTTP(S) cron webhook URLs through `normalizeHttpWebhookUrl`, and the PR body shows the patched helper rejecting that input while preserving a plain URL; I did not run an end-to-end cron job in this read-only review.

## Solution Assessment

No. The shared normalizer is the right place to reject userinfo, but the patch is incomplete without redacted invalid-target handling, legacy fallback visibility, doctor migration safeguards, and regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:19-20`
  - body: When this guard rejects an existing `delivery.mode = "webhook"` target containing userinfo, `dispatchGatewayCronFinishedNotifications` reaches the invalid-target warning and logs `params.job.delivery.to` as `deliveryTo`. That can write the credential-bearing URL to logs on every run, so redact or omit the rejected target before merging.
  - confidence: 0.91
- **[P2] Surface rejected legacy cron.webhook targets:** `src/cron/webhook-url.ts:19-20`
  - body: Legacy `notify: true` jobs resolve `cfg.cron.webhook` through this helper, but the invalid-target warning only runs for explicit `delivery.mode = "webhook"`. After this new `null` result, credentialed legacy fallback deliveries become silent no-ops with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:19-20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` for legacy notify jobs without validating it. With this rejection in place, doctor can convert a working legacy fallback into stored metadata the scheduler later rejects.
  - confidence: 0.84
