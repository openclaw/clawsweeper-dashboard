## Summary

Keep open: the hardening target is useful, but the current PR only adds the userinfo rejection and leaves credential-redaction, legacy fallback, and doctor migration paths unsafe or confusing for existing cron webhook users.

## What This Changes

The PR adds a username/password check to `normalizeHttpWebhookUrl` so HTTP(S) cron webhook URLs with URL userinfo are rejected.

## Best Possible Solution

Ship the userinfo rejection with redacted invalid-target diagnostics, explicit legacy fallback warnings, doctor validation or refusal, focused regression tests, and migration or release notes.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main accepts credential-bearing HTTP(S) webhook URLs after only a protocol check, and the PR proof shows the patched helper rejecting one; I did not run an end-to-end cron job in this read-only review.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch is incomplete until rejected targets are redacted, legacy fallback rejection is surfaced, doctor avoids unsafe migration, and regression coverage is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:20`
  - body: When this guard rejects a stored `delivery.mode="webhook"` URL with userinfo, `dispatchGatewayCronFinishedNotifications` falls into the invalid-target warning and logs `params.job.delivery.to` as `deliveryTo`. That can write the credential-bearing URL to logs on every run, so the rejection path needs to redact or omit the target before merge.
  - confidence: 0.91
- **[P2] Surface rejected legacy cron.webhook targets:** `src/cron/webhook-url.ts:20`
  - body: Legacy `notify: true` jobs resolve `cfg.cron.webhook` through this helper, but the invalid-target warning only runs for explicit `delivery.mode="webhook"`. After this new `null` result, credentialed legacy fallback deliveries become silent no-ops with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` for legacy notify jobs without validating it. With this rejection in place, doctor can convert a working legacy fallback into stored metadata the scheduler later rejects.
  - confidence: 0.84
