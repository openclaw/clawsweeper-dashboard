## Summary

Keep this PR open: the security hardening goal is valid and the helper-level proof is sufficient, but the current patch changes a shared cron webhook normalizer without an approved upgrade policy for stored credential-bearing webhook configs or regression coverage across its runtime and doctor fanout.

## What This Changes

The PR adds a username/password userinfo check to the shared cron HTTP webhook URL normalizer so credential-bearing HTTP(S) webhook URLs return null.

## Best Possible Solution

Land the credentialed-webhook guard only with maintainer-approved upgrade behavior for existing stored cron webhook configs and focused tests across validation, runtime delivery, failure destinations, and doctor migration.

## Reproduction Assessment

Yes for the helper-level behavior by source inspection and attached proof: current main accepts credential-bearing HTTP(S) webhook URLs, while the PR screenshot shows the patched helper returning null. I did not run an end-to-end cron delivery repro in this read-only review.

## Solution Assessment

No. The shared helper is a reasonable enforcement point, but unconditional rejection is only a plausible hardening step until stored-config upgrade behavior and doctor/runtime handling are explicitly covered.
