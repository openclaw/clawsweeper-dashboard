## Summary

Keep open: the security direction is useful, but the branch still introduces raw logging and legacy migration regressions when credentialed cron webhook URLs are rejected, so it needs human security/compatibility handling before merge.

## What This Changes

The branch makes `normalizeHttpWebhookUrl` reject http(s) webhook URLs whose parsed URL includes a username or password.

## Best Possible Solution

Ship userinfo rejection through the shared cron webhook normalizer together with redacted diagnostics, explicit legacy fallback handling, doctor validation or refusal, focused regression tests, and migration notes.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.5.12 accept credential-bearing HTTP(S) cron webhook URLs, the PR proof shows the patched helper rejecting them, and the logging plus doctor-migration blockers are traceable from current call paths.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch is not complete until rejected targets are redacted, legacy `cron.webhook` rejection is surfaced, doctor avoids unsafe migration, and regression coverage is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:20`
  - body: After this guard returns `null` for an existing stored `delivery.mode: "webhook"` URL with userinfo, `dispatchGatewayCronFinishedNotifications` logs the raw `params.job.delivery.to` as `deliveryTo`. Redact or omit that value before rejecting URL userinfo, otherwise the hardening can write the credential-bearing URL to logs.
  - confidence: 0.91
- **[P2] Warn on rejected legacy cron.webhook targets:** `src/cron/webhook-url.ts:20`
  - body: Legacy `notify: true` jobs pass `cfg.cron.webhook` through this normalizer, but the invalid-target warning only runs for explicit webhook delivery. Rejecting a credentialed legacy webhook would turn finished-run delivery into a silent no-op, so emit a redacted warning or migration hint for that path too.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:20`
  - body: `openclaw doctor --fix` copies `cfg.cron.webhook` into `delivery.to` and removes `notify` without using this normalizer. Once credentialed URLs are rejected, doctor can rewrite a working legacy fallback into a per-job webhook that the scheduler later skips.
  - confidence: 0.84
