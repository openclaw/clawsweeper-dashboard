## Summary

Keep this PR open: the security hardening direction is valid, but the patch still needs maintainer-approved upgrade behavior because it rejects a URL shape that current main and the latest release accept for stored cron webhook delivery.

## What This Changes

The PR adds a username/password check to the shared cron `normalizeHttpWebhookUrl` helper so credential-bearing HTTP(S) webhook URLs return `null`.

## Best Possible Solution

Ship cron webhook userinfo rejection only after maintainers choose the upgrade policy and the patch covers create/update validation, stored dispatch, failure destinations, and legacy notify doctor migration.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.6.8` shows credential-bearing HTTP(S) cron webhook URLs are accepted by the helper today; the PR screenshot shows the patched helper rejects that shape, but I did not run a full cron E2E.

## Solution Assessment

No, not yet. Helper-level rejection is a reasonable hardening point, but it is not the best complete fix until maintainers approve the upgrade behavior for stored webhook jobs and the patch covers those paths.
