## Summary

Keep open: the PR targets a real cron webhook hardening gap and the shared helper is the right seam, but the patch is not safe to merge as-is because rejected stored URLs can hit an existing raw logging path and legacy migration paths can silently break.

## What This Changes

The PR adds a username/password check to `src/cron/webhook-url.ts` so `normalizeHttpWebhookUrl` returns `null` for HTTP(S) cron webhook URLs with URL userinfo.

## Best Possible Solution

Adopt userinfo rejection in the shared cron webhook normalizer together with redacted invalid-target diagnostics, legacy fallback warnings, doctor migration validation, focused regression coverage, and explicit release/migration handling.

## Reproduction Assessment

Yes. Source inspection on current main shows the shared normalizer accepts credential-bearing HTTP(S) URLs, and the PR body terminal screenshot shows the patched helper rejecting a `user:pass@` URL while accepting a plain URL.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch omits redaction, legacy warnings, doctor migration validation, and tests around the backward-incompatible policy change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:19-21`
  - body: When this guard turns an existing `delivery.to` containing URL userinfo into `null`, `dispatchGatewayCronFinishedNotifications` logs `params.job.delivery.to` raw in the invalid webhook warning. That can write the credential-bearing URL to logs on every run, so redact or omit the target before landing this guard.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook is rejected:** `src/cron/webhook-url.ts:19-21`
  - body: Legacy `notify: true` jobs resolve global `cron.webhook` through this normalizer. With the new guard, credential-bearing legacy URLs become `null`, but only explicit `delivery.mode === "webhook"` jobs get an invalid-target warning, so callbacks can stop with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:19-21`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` and deletes `notify` without validating through this normalizer. After this guard, doctor can migrate a working legacy fallback into an invalid per-job webhook and remove the fallback.
  - confidence: 0.84
