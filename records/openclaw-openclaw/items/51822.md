## Summary

Keep this PR open: current main does not implement the credentialed cron webhook rejection, but the branch is not merge-ready until maintainers choose the upgrade policy for existing stored webhook configs and the shared call paths are covered.

## What This Changes

The branch adds a username/password check to `normalizeHttpWebhookUrl` so credential-bearing cron HTTP(S) webhook URLs return `null`.

## Best Possible Solution

Land the credentialed-webhook guard only with maintainer-approved upgrade behavior and regression coverage across validation, runtime delivery, failure destinations, and doctor migration.

## Solution Assessment

No. The shared helper is a reasonable guard location, but unconditional rejection is only a plausible hardening step until stored-config upgrade behavior and doctor/runtime handling are explicitly covered.
