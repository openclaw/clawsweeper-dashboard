## Summary

Keep open. Current main still lacks cron webhook userinfo rejection, but this PR is not mergeable as submitted because the new rejection path would turn existing credential-bearing stored webhooks into invalid targets that are logged raw, and it does not handle legacy cron.webhook warning or doctor migration behavior.

## What This Changes

This PR adds a guard to `normalizeHttpWebhookUrl` so parsed http(s) cron webhook URLs with a username or password return `null`.

## Best Possible Solution

Land cron URL userinfo rejection through the shared normalizer, but pair it with redacted invalid-target logging, explicit legacy `cron.webhook` warning or migration guidance, doctor validation that refuses or warns instead of copying invalid credential URLs, targeted tests for create/edit/runtime/doctor paths, and a user-facing changelog entry for the breaking security hardening.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction: `normalizeHttpWebhookUrl("https://user:pass@example.invalid/hook")` passes because only protocol is checked, and the PR’s new null path can be traced to the raw invalid-delivery warning for stored webhook jobs.

## Solution Assessment

No. The shared normalizer is the right seam for rejecting URL userinfo, but this patch alone is not the best fix because it leaves stored-job logging and legacy doctor migration unsafe under the new invalid-URL behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Redact rejected stored webhook URLs:** `src/cron/webhook-url.ts:19-21`
  - body: When this guard returns `null` for an already stored credential-bearing `delivery.to`, `dispatchGatewayCronFinishedNotifications` falls into the skipped-delivery warning and logs `params.job.delivery.to` as `deliveryTo`. That writes the full userinfo URL to logs on every finished run, so the rejection needs a redacted invalid-target path before merge.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook is invalid:** `src/cron/webhook-url.ts:19-21`
  - body: Legacy `notify: true` jobs resolve the global `cron.webhook` through this same normalizer. With this guard, a credentialed legacy webhook becomes `null`, but the invalid-target warning only runs for explicit `delivery.mode === "webhook"` jobs, so legacy callbacks stop delivering with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:19-21`
  - body: Doctor still migrates `notify: true` jobs by copying `cron.webhook` into `delivery.to`. If that global webhook contains userinfo, this PR makes the repaired job invalid after `doctor --fix`, converting a working legacy fallback into a skipped per-job webhook.
  - confidence: 0.84
