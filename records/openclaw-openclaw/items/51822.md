## Summary

Keep open: the shared URL normalizer is the right place for this hardening, but the patch is incomplete because the new invalid path can expose stored webhook credentials in logs and leaves legacy cron webhook migration paths unsafe or silent.

## What This Changes

The PR adds a username/password check to the cron HTTP webhook URL normalizer so URL userinfo returns null.

## Best Possible Solution

Land the userinfo rejection together with redacted invalid-target diagnostics, legacy cron.webhook warnings, doctor safeguards, focused regression coverage, and migration wording.

## Reproduction Assessment

Yes. Source inspection shows current main accepts credential-bearing HTTP(S) cron webhook URLs, and the PR terminal screenshot shows the helper rejecting them after the change; the remaining blockers are source-visible in logging and doctor migration paths.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch needs redacted diagnostics and legacy migration handling before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:20-21`
  - body: With this guard, an existing `delivery.mode: "webhook"` URL containing userinfo now resolves to null, and `dispatchGatewayCronFinishedNotifications` logs `params.job.delivery.to` as `deliveryTo`. That writes raw credentials to logs on every run until the target is redacted or omitted.
  - confidence: 0.91
- **[P2] Warn on invalid legacy cron.webhook:** `src/cron/webhook-url.ts:20-21`
  - body: Legacy `notify: true` jobs resolve the global `cron.webhook` through this normalizer. Once userinfo URLs return null, those callbacks stop, but the invalid-target warning only covers explicit per-job webhook delivery, leaving operators without a migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:20-21`
  - body: `openclaw doctor --fix` copies the global `cron.webhook` into per-job `delivery.to` and removes `notify`. If that legacy URL contains userinfo, the repaired job becomes invalid under this guard, so doctor should validate and warn instead of writing a skipped webhook.
  - confidence: 0.84
