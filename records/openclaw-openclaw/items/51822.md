## Summary

Keep this PR open: the security hardening is useful, but the current patch fail-closes a shared cron webhook normalizer without a maintainer-approved upgrade policy for previously accepted stored webhook URLs.

## What This Changes

The PR makes `normalizeHttpWebhookUrl` return `null` for HTTP(S) cron webhook URLs whose parsed URL contains a username or password.

## Best Possible Solution

Ship cron webhook userinfo rejection only after maintainers choose the upgrade policy and the patch covers create/update validation, stored dispatch, failure destinations, and legacy notify doctor migration.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.8` accept credential-bearing HTTP(S) cron webhook URLs, and the PR screenshot shows the patched helper rejecting that shape; I did not run a full cron E2E in this read-only review.

## Solution Assessment

No, not yet. Helper-level rejection is a reasonable hardening point, but it is not the best complete fix until maintainers approve the upgrade behavior for stored webhook jobs and the patch covers those paths.
