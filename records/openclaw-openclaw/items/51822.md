## Summary

Keep this PR open: the security hardening target is valid, but the current diff only rejects URL userinfo in the shared normalizer and leaves source-proven logging, legacy fallback, and doctor migration paths unsafe for existing cron webhook users.

## What This Changes

The PR adds a username/password check to `normalizeHttpWebhookUrl` so cron webhook URL validation rejects HTTP(S) URLs with parsed URL userinfo.

## Best Possible Solution

Ship the userinfo rejection together with redacted invalid-target diagnostics, legacy fallback warnings, doctor migration validation or refusal, and focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main returns credential-bearing HTTP(S) URLs from `normalizeHttpWebhookUrl`, and the inspected PR proof shows the patched helper rejecting one while accepting a plain HTTPS URL.

## Solution Assessment

No. The shared normalizer is the right seam, but the proposed implementation needs safe diagnostics, legacy fallback handling, doctor migration validation, and regression coverage before it is the best fix.
