## Summary

Keep open: the security hardening direction is useful, but the current patch is not merge-ready because the new rejection path can expose stored credential URLs in logs, silently stop legacy webhook callbacks, and make doctor migrate legacy jobs into invalid webhook delivery.

## What This Changes

The PR makes `normalizeHttpWebhookUrl` reject HTTP(S) webhook URLs when the parsed URL contains a username or password.

## Best Possible Solution

Ship the userinfo rejection with redacted invalid-target diagnostics, invalid legacy-fallback warnings, doctor validation or refusal for unsafe migrations, focused regression tests, and maintainer-owned migration notes.

## Reproduction Assessment

Yes, at source level. Current main accepts valid HTTP(S) cron webhook URLs, and the PR's new guard clearly makes credential-bearing URLs normalize to `null`, which flows into the explicit delivery, legacy fallback, and doctor migration paths inspected above.

## Solution Assessment

No, not yet. Rejecting embedded credentials is a sound direction, but the patch is not the best complete solution until the upgrade, logging, legacy fallback, and doctor migration paths are made safe and tested.
