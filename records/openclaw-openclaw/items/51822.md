## Summary

Keep open: the userinfo rejection is a useful security hardening primitive, but the PR is not merge-ready because existing cron upgrade paths can still leak or silently drop credential-bearing webhook URLs.

## What This Changes

The PR changes `src/cron/webhook-url.ts` so cron HTTP(S) webhook URL normalization returns `null` when the parsed URL has a username or password.

## Best Possible Solution

Ship the userinfo rejection together with redacted invalid-target diagnostics, invalid legacy fallback warnings, doctor migration validation, and focused regression coverage for fresh and upgraded cron webhook configurations.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.19` accept credential-bearing HTTP(S) webhook URLs, and applying the PR's helper behavior makes the explicit, legacy, and doctor paths follow the unsafe branches shown in source.

## Solution Assessment

No, not as currently proposed. The helper guard is a sound primitive, but it needs redacted diagnostics, invalid legacy fallback handling, doctor validation, and tests before it is the maintainable fix.
