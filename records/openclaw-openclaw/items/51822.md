## Summary

Keep this PR open, but it is not merge-ready: the credentialed-URL guard is useful security hardening, yet the shared helper also controls stored cron delivery and doctor migration, so maintainers need to choose the upgrade policy for existing credential-bearing webhook configs before merge.

## What This Changes

The PR adds a username/password rejection to the shared cron `normalizeHttpWebhookUrl` helper.

## Best Possible Solution

Land the credentialed-webhook guard only after maintainers choose the stored-config upgrade behavior and add focused coverage for validation, runtime dispatch, and doctor migration.

No. The shared helper is the right central location, but unconditional fail-closed behavior is only a plausible hardening step until stored webhook configs and doctor migration have an explicit upgrade policy.
