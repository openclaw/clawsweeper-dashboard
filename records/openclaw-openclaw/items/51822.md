## Summary

Keep open: the security hardening target is valid and the contributor's terminal proof is sufficient, but the patch still creates an unredacted stored-webhook logging path and misses legacy fallback and doctor migration behavior. Because it changes accepted cron webhook URLs, it also needs maintainer/security approval before merge.

## What This Changes

The PR adds a guard in `src/cron/webhook-url.ts` that rejects parsed HTTP(S) cron webhook URLs with username or password userinfo.

## Best Possible Solution

Adopt userinfo rejection in the shared cron webhook normalizer together with redacted invalid-target diagnostics, explicit legacy fallback warnings, doctor migration validation or refusal, focused regression tests, and release/migration notes.

## Reproduction Assessment

Yes. Current main source shows HTTP(S) cron webhook URLs are accepted without checking `URL.username` or `URL.password`, and the PR body terminal screenshot shows the patched helper rejecting a credential-bearing URL while accepting a plain URL.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch needs redaction, legacy fallback handling, doctor migration validation, tests, and an explicit migration policy before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact stored webhook targets before rejecting them:** `src/cron/webhook-url.ts:20`
  - body: When this guard rejects an existing stored `delivery.to` with URL userinfo, `dispatchGatewayCronFinishedNotifications` logs the raw `params.job.delivery.to` as `deliveryTo`. That can write the credential-bearing URL to gateway logs on every finished run, so redact or omit the invalid target before shipping the rejection.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook is rejected:** `src/cron/webhook-url.ts:20`
  - body: Legacy `notify: true` jobs run the global `cron.webhook` through this helper, but the invalid-target warning only covers explicit webhook delivery jobs. If the new guard rejects a credentialed legacy URL, finished-run callbacks stop without a migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` and deletes `notify` without using this normalizer. With this guard, a credentialed legacy fallback can be migrated into an invalid per-job webhook that the scheduler skips.
  - confidence: 0.84
