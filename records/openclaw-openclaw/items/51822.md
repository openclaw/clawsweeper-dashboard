## Summary

Keep open: the PR targets the right shared cron webhook normalizer, but the current one-line rejection leaves source-visible security and migration blockers in invalid-target logging, legacy notify fallback diagnostics, and doctor repair.

## What This Changes

The PR changes `normalizeHttpWebhookUrl` so HTTP(S) cron webhook URLs with parsed username or password userinfo return `null`.

## Best Possible Solution

Reject URL userinfo through the shared normalizer, redact and diagnose every invalid cron webhook path, guard doctor legacy migration, and add focused regression coverage plus release notes.

## Reproduction Assessment

Yes. Source inspection shows current main accepts credential-bearing HTTP(S) cron webhook URLs, and the PR attachment shows the changed helper rejecting one; the merge blockers are also source-visible in logging and doctor migration paths.

## Solution Assessment

No. The shared normalizer is the right seam, but the current patch is incomplete without redacted invalid-target logging, legacy fallback diagnostics, doctor safeguards, and focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:20-21`
  - body: When this guard makes an existing credential-bearing `delivery.to` invalid, finished-run dispatch logs `params.job.delivery.to` as `deliveryTo` without redaction. That exposes the raw userinfo URL on each skipped delivery, so the invalid-target path needs to redact or omit the target before this guard lands.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook becomes invalid:** `src/cron/webhook-url.ts:20-21`
  - body: Legacy `notify: true` jobs resolve global `cron.webhook` through this normalizer. With userinfo rejected, a credentialed legacy URL becomes `null`, but the invalid-target warning only runs for explicit `delivery.mode === "webhook"`, so callbacks stop with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:20-21`
  - body: `openclaw doctor --fix` copies global `cron.webhook` into per-job `delivery.to` and deletes `notify`. If that URL contains userinfo, the repaired job is now invalid and the scheduler skips it, so doctor should validate and warn instead of writing an unusable webhook.
  - confidence: 0.84
