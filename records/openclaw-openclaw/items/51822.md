## Summary

Keep this PR open but not merge-ready: the security hardening is valid, yet changing the shared cron webhook normalizer makes already-stored credential-bearing webhook destinations fail closed without a maintainer-approved upgrade policy or focused coverage.

## What This Changes

The PR makes cron's shared HTTP(S) webhook URL normalizer return null when the parsed URL contains a username or password.

## Best Possible Solution

Land the credentialed-webhook guard only after maintainers choose the stored-config upgrade behavior and the PR covers the affected validation, runtime delivery, and doctor migration paths.

## Solution Assessment

No. The shared helper is a good central guard location, but unconditional fail-closed behavior is only a plausible hardening step until stored webhook configs and doctor migration have explicit upgrade handling.
