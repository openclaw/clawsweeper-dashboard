## Summary

Keep open: the guard is a plausible security hardening point, but it changes shipped cron webhook URL acceptance in a shared helper that also controls stored runtime delivery and doctor migration, so maintainers need an upgrade policy and focused coverage before merge.

## What This Changes

The PR adds a username/password check to `normalizeHttpWebhookUrl` so cron HTTP(S) webhook URLs with URL userinfo return `null`.

## Best Possible Solution

Ship cron webhook userinfo rejection only after maintainers choose the upgrade policy and the patch covers create/update validation, stored delivery, failure destinations, and legacy notify doctor migration.

## Reproduction Assessment

Yes, source-level. Current main and `v2026.6.8` accept credential-bearing HTTP(S) cron webhook URLs, and the PR screenshot shows the patched helper returning `null` for that shape; I did not run a full cron E2E in this read-only review.

## Solution Assessment

No, not yet. Helper-level rejection is a reasonable hardening point, but it is not the best complete fix until maintainers approve the upgrade behavior for stored webhook jobs and the patch covers those paths.
