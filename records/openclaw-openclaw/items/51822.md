## Summary

Keep open: the PR is a focused and useful cron webhook hardening change, but current main still accepts credential-bearing cron webhook URLs and the proposed shared-helper rejection would affect stored runtime delivery and doctor migration without a confirmed upgrade policy or coverage.

## What This Changes

The PR adds a credential check to the cron webhook URL normalizer so HTTP(S) webhook URLs with parsed username or password fields are rejected.

## Best Possible Solution

Land the credentialed-webhook guard only with maintainer-approved upgrade behavior for existing stored cron webhook configs and focused regression coverage across validation, runtime delivery, failure destinations, and doctor migration.

## Reproduction Assessment

Yes. Current main source shows credential-bearing HTTP(S) cron webhook URLs are accepted because the shared helper returns the trimmed URL after only protocol validation; the PR proof shows the patched helper returning null for that input.

## Solution Assessment

No. The shared helper is a reasonable guard location, but unconditional rejection is only a plausible hardening step until upgrade and doctor/runtime behavior are explicitly covered.
