## Summary

Keep open. The hardening is useful, but the patch changes shipped cron webhook behavior from accepting credential-bearing HTTP(S) URLs to treating them as invalid, and that needs explicit upgrade policy plus focused coverage before merge.

## What This Changes

The PR adds a username/password userinfo guard to `normalizeHttpWebhookUrl` so cron webhook URLs with embedded credentials return `null`.

## Best Possible Solution

Ship credential-bearing webhook rejection only after maintainers confirm the fail-closed upgrade policy and the PR covers validation, runtime warning, and doctor migration paths; otherwise use a warning or doctor migration-first path.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.6` return credential-bearing HTTP(S) URLs from the helper, while the PR patch and inspected terminal screenshot show the helper returning `null` after the new guard.

## Solution Assessment

No, not yet. Helper-level rejection is a plausible hardening direction, but it is not the best complete fix until maintainers choose the upgrade policy and the affected runtime and doctor paths are covered.
