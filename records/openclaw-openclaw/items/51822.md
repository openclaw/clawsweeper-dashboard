## Summary

Keep open: the PR is useful cron webhook security hardening, but the current diff only changes the shared URL normalizer and leaves existing cron paths that can log credential-bearing URLs or silently break legacy fallback/migration behavior.

## What This Changes

Adds a username/password check to `normalizeHttpWebhookUrl` so cron webhook URL validation rejects HTTP(S) URLs with URL userinfo.

## Best Possible Solution

Ship the userinfo rejection together with redacted invalid-target diagnostics, legacy fallback warnings, doctor migration validation/refusal, and focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main checks only protocol and returns credential-bearing HTTP(S) URLs, while the PR proof shows the patched helper rejecting one and accepting a plain HTTPS URL.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch needs the surrounding logging, legacy fallback, doctor migration, and regression-test repairs before it is the safest implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected webhook targets:** `src/cron/webhook-url.ts:19-20`
  - body: Rejecting userinfo here makes stored `delivery.mode = "webhook"` values with credentials return `null`, and `dispatchGatewayCronFinishedNotifications` then logs `params.job.delivery.to` as `deliveryTo` when normalization fails. That writes the raw credential-bearing URL to logs on every finished run, so redact or omit the rejected target before adding this rejection.
  - confidence: 0.91
- **[P2] Warn on invalid legacy cron.webhook fallback:** `src/cron/webhook-url.ts:19-20`
  - body: Legacy `notify: true` jobs resolve `cron.webhook` through this helper, but the invalid-target warning only runs for explicit webhook delivery. After this new `null` result, credentialed legacy fallback deliveries become silent no-ops with no migration hint.
  - confidence: 0.86
- **[P2] Validate cron.webhook before doctor migration:** `src/cron/webhook-url.ts:19-20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` for legacy notify jobs without validating it. With this rejection in place, doctor can convert a previously delivering legacy fallback into stored metadata the scheduler later rejects.
  - confidence: 0.84
