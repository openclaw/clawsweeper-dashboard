## Summary

Keep open: the hardening goal is valid, but the patch is not merge-ready because the shared cron webhook normalizer also controls stored runtime delivery, failure destinations, and doctor migration, so existing credential-bearing cron webhook configs can stop delivering without a maintainer-approved upgrade policy.

## What This Changes

The PR adds a username/password userinfo check to `normalizeHttpWebhookUrl` so cron HTTP(S) webhook URLs with embedded credentials normalize to null.

## Best Possible Solution

Land the credentialed URL guard only with maintainer-approved upgrade behavior and focused regression tests for create/update validation, runtime delivery, failure destinations, and doctor legacy migration.

## Reproduction Assessment

Yes. Current main and `v2026.6.9` accept credential-bearing HTTP(S) webhook URLs in the shared helper, and the inspected PR screenshot shows the patched helper rejecting that shape directly.

## Solution Assessment

No. The shared helper is a reasonable guard location for new inputs, but unconditional rejection is not the best merge shape until stored-config upgrade behavior and doctor/runtime handling are explicitly chosen and tested.
