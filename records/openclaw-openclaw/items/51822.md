## Summary

Keep open: the cron webhook hardening is useful and current main still accepts credential-bearing HTTP(S) webhook URLs, but this patch is not safe to merge as-is because the new rejection path can expose stored credentials in logs and can break legacy cron webhook migration without clear diagnostics.

## What This Changes

The PR changes `normalizeHttpWebhookUrl` to reject HTTP(S) cron webhook URLs whose parsed URL contains a username or password.

## Best Possible Solution

Ship userinfo rejection through the shared cron webhook normalizer together with redacted diagnostics, explicit legacy fallback warnings, doctor migration validation or refusal, focused regression coverage, and migration notes.

## Reproduction Assessment

Yes. Source inspection shows current main accepts HTTP(S) cron webhook URLs with URL userinfo, and the PR body terminal screenshot shows the patched helper rejecting a credential-bearing URL while accepting a plain HTTPS URL.

## Solution Assessment

No. The shared helper is the right location, but the current patch is not the best complete fix because it does not redact the new invalid-target path or protect legacy cron webhook migration behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact stored webhook targets before rejecting them:** `src/cron/webhook-url.ts:20`
  - body: With this new `null` path, an existing stored `delivery.to` containing URL userinfo reaches the invalid-target warning in `dispatchGatewayCronFinishedNotifications`, which logs the raw value as `deliveryTo`. Redact or omit that target before making credential-bearing URLs invalid.
  - confidence: 0.91
- **[P2] Warn when rejected legacy cron.webhook disables delivery:** `src/cron/webhook-url.ts:20`
  - body: Legacy `notify: true` jobs resolve the global `cron.webhook` through this helper, but the invalid-target warning only runs for explicit `delivery.mode === "webhook"` jobs. If this guard rejects a credentialed legacy URL, finished-run callbacks stop without a migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` and deletes `notify` without using this normalizer. After this guard lands, a credentialed legacy fallback can be migrated into an invalid per-job webhook that the scheduler skips.
  - confidence: 0.84
