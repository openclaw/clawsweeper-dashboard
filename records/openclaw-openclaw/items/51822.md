## Summary

Keep open: current main still accepts cron webhook URL userinfo and the PR touches the right shared helper, but the patch is not merge-ready because it makes credential-bearing stored webhooks invalid while one invalid-target warning still logs the raw URL, and it does not handle legacy fallback or doctor migration paths.

## What This Changes

The PR adds a username/password check to `normalizeHttpWebhookUrl` so cron HTTP(S) webhook URLs with URL userinfo are rejected.

## Best Possible Solution

Adopt userinfo rejection in the shared cron webhook normalizer together with redacted invalid-target diagnostics, legacy fallback warnings, doctor migration validation, regression coverage, and explicit migration/release handling.

## Reproduction Assessment

Yes. Current main's shared normalizer accepts credential-bearing HTTP(S) URLs by returning the trimmed input, and the inspected PR body terminal screenshot shows the patched helper rejecting one while accepting a plain URL.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch alone is not the best fix until it redacts rejected targets and handles legacy fallback and doctor migration paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:20`
  - body: When this guard makes an existing `delivery.mode = "webhook"` target with URL userinfo invalid, `dispatchGatewayCronFinishedNotifications` logs `deliveryTo: params.job.delivery.to` raw. That can expose the credential-bearing URL in gateway logs until the warning redacts or omits the target.
  - confidence: 0.91
- **[P2] Warn on rejected legacy cron.webhook:** `src/cron/webhook-url.ts:20`
  - body: Legacy `notify: true` jobs run global `cron.webhook` through this helper. If the new guard rejects that legacy URL, `resolveCronWebhookTarget` returns null and no warning fires because the invalid-target warning only checks explicit webhook delivery jobs, so callbacks stop without migration guidance.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` and deletes `notify` without this normalizer. With the new guard, a credentialed legacy fallback is migrated into an invalid per-job webhook that the scheduler skips; doctor should warn or refuse that migration.
  - confidence: 0.84
