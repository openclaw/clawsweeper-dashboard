## Summary

Keep open: the security hardening goal is valid and current main still accepts credential-bearing cron webhook URLs, but the PR changes a shared cron URL normalizer in a way that can break stored webhook delivery and doctor migration without an explicit upgrade policy or focused coverage.

## What This Changes

The branch adds a username/password check to the cron HTTP webhook URL normalizer so credential-bearing HTTP(S) webhook URLs return null.

## Best Possible Solution

Land the credentialed-webhook guard only with maintainer-approved upgrade behavior for existing stored cron webhook configs and regression tests across validation, runtime delivery, failure destinations, and doctor migration.

## Reproduction Assessment

Yes. Current main source shows credential-bearing HTTP(S) cron webhook URLs are accepted by the shared helper, and the PR screenshot shows the patched helper returning null for a credential-bearing URL.

## Solution Assessment

No. The shared helper is a reasonable guard location, but unconditional rejection is only a plausible hardening step until stored-config upgrade behavior and doctor/runtime handling are explicitly covered.
