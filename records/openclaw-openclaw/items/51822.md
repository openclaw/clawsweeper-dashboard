## Summary

Keep open: the security hardening remains useful and current main/latest release do not implement it, but the patch is not merge-ready because the shared helper also controls stored cron delivery and doctor migration, so existing credential-bearing webhook jobs need maintainer-approved upgrade behavior and tests.

## What This Changes

The PR adds a username/password check to the shared cron webhook URL normalizer so HTTP(S) webhook URLs with URL userinfo are rejected.

## Best Possible Solution

Land a cron URL-userinfo guard only with explicit upgrade handling for persisted/legacy webhooks and focused coverage across validator, runtime, and doctor paths.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.9` accept credential-bearing HTTP(S) cron webhook URLs, and the PR screenshot shows the patched helper rejecting one.

## Solution Assessment

No. The shared helper is a plausible enforcement point, but unconditional rejection is not the best fix until stored-config upgrade behavior and doctor/runtime handling are explicitly chosen and tested.
