## Summary

Keep open: the shared cron webhook normalizer is the right hardening point, but the PR is not merge-ready because stored credential-bearing URLs can still be logged raw after the new rejection path, legacy fallback jobs lose diagnostics, and doctor can migrate legacy jobs into invalid webhook delivery.

## What This Changes

The PR adds a username/password check to `normalizeHttpWebhookUrl` so parsed HTTP(S) cron webhook URLs with URL userinfo return `null`.

## Best Possible Solution

Reject URL userinfo through the shared cron webhook normalizer, redact invalid-target diagnostics, warn on invalid legacy fallback, guard doctor migration, and add focused regression coverage plus release notes.

## Reproduction Assessment

Yes. Source inspection on current main shows a credential-bearing HTTP(S) webhook URL passes the shared normalizer, and the PR body provides terminal proof that the patched helper rejects that URL shape.

## Solution Assessment

No. The shared normalizer is the narrowest policy point, but the patch needs adjacent log redaction, legacy fallback diagnostics, doctor migration validation, and tests before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:19-21`
  - body: When this guard makes an existing credential-bearing `delivery.to` invalid, finished-run dispatch logs `params.job.delivery.to` as `deliveryTo` without redaction. That exposes the raw userinfo URL on each skipped delivery, so the invalid-target path needs to redact or omit the target before this guard lands.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook becomes invalid:** `src/cron/webhook-url.ts:19-21`
  - body: Legacy `notify: true` jobs resolve global `cron.webhook` through this normalizer. With userinfo rejected, a credentialed legacy URL becomes `null`, but the invalid-target warning only runs for explicit `delivery.mode === "webhook"`, so callbacks stop with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:19-21`
  - body: `openclaw doctor --fix` copies global `cron.webhook` into per-job `delivery.to` and deletes `notify`. If that URL contains userinfo, the repaired job is now invalid and the scheduler skips it, so doctor should validate and warn instead of writing an unusable webhook.
  - confidence: 0.84
