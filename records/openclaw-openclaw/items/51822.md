## Summary

Keep open. The hardening is useful and the shared cron webhook normalizer is the right seam, but the PR currently leaves credential-bearing legacy and stored webhook paths with logging, migration, and diagnostic regressions that should be handled before merge.

## What This Changes

The PR adds a username/password rejection to `normalizeHttpWebhookUrl` so cron webhook URLs with URL userinfo return `null`.

## Best Possible Solution

Reject URL userinfo through the shared cron webhook normalizer while redacting invalid-target diagnostics, warning for invalid legacy fallback, guarding doctor migration, and covering the cases with focused cron tests.

## Reproduction Assessment

Yes. Source inspection on current main shows credential-bearing HTTP(S) cron webhook URLs pass the shared normalizer, and the PR body includes after-fix terminal proof for the helper behavior.

## Solution Assessment

No. The shared normalizer is the right policy seam, but the current patch needs adjacent log redaction, legacy fallback diagnostics, doctor migration validation, and regression coverage before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:19-21`
  - body: When this guard makes an existing `delivery.to` with URL userinfo invalid, finished-run dispatch logs `deliveryTo: params.job.delivery.to` on the skipped-delivery path. That writes the raw credential-bearing URL to logs, so the warning should redact or omit the target before this guard lands.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook is rejected:** `src/cron/webhook-url.ts:19-21`
  - body: Legacy `notify: true` jobs resolve global `cron.webhook` through this normalizer. If userinfo is rejected, the target becomes `null`, but the invalid-target warning only covers explicit `delivery.mode === "webhook"`, so callbacks stop without a migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:19-21`
  - body: `openclaw doctor --fix` copies global `cron.webhook` into `delivery.to` and deletes `notify`. With this normalizer rejecting userinfo, doctor can rewrite a working legacy fallback into an invalid per-job webhook, so it should validate and warn instead.
  - confidence: 0.84
