## Summary

Keep open: the security hardening is useful, but shared helper-level rejection changes shipped cron webhook delivery and doctor migration behavior without maintainer-approved upgrade policy or focused coverage.

## What This Changes

The PR adds a username/password guard to `normalizeHttpWebhookUrl` so cron webhook URLs with URL userinfo return `null`.

## Best Possible Solution

Ship cron webhook userinfo rejection only with maintainer-approved upgrade policy and focused coverage for create/update validation, stored runtime delivery, failure destinations, and doctor legacy notify migration.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.8` accept credential-bearing HTTP(S) cron webhook URLs, and the attached terminal screenshot shows the PR helper rejecting that shape after the patch.

## Solution Assessment

No, not yet. Helper-level rejection is a plausible hardening point, but it is not the best complete fix until maintainers approve the upgrade behavior for stored webhook jobs and the patch covers those paths.
