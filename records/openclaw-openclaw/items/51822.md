## Summary

Keep this PR open: the shared normalizer is the right seam for rejecting URL userinfo, but the patch is incomplete because existing stored credential-bearing webhook URLs can be logged raw and legacy cron webhook migration paths can fail without a clear operator warning.

## What This Changes

The PR adds a username/password check to `normalizeHttpWebhookUrl` so credential-bearing HTTP(S) cron webhook URLs return `null`.

## Best Possible Solution

Reject URL userinfo through the shared normalizer, redact invalid-target diagnostics, warn on invalid legacy fallback, guard doctor migration, and add focused regression coverage plus release notes.

## Reproduction Assessment

Yes. Source inspection shows current main accepts credential-bearing HTTP(S) cron webhook URLs, and the PR body includes terminal proof that the patched helper rejects one; the remaining blockers are also source-visible in logging and doctor migration paths.

## Solution Assessment

No. The shared normalizer is the right place for the guard, but the patch needs the adjacent redaction, legacy diagnostic, doctor migration, and test coverage work before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:20-22`
  - body: When this guard makes an existing credential-bearing `delivery.to` invalid, finished-run dispatch logs `params.job.delivery.to` as `deliveryTo` without redaction. That exposes the raw userinfo URL on each skipped delivery, so the invalid-target path needs to redact or omit the target before this guard lands.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook becomes invalid:** `src/cron/webhook-url.ts:20-22`
  - body: Legacy `notify: true` jobs resolve global `cron.webhook` through this normalizer. With userinfo rejected, a credentialed legacy URL becomes `null`, but the invalid-target warning only runs for explicit `delivery.mode === "webhook"`, so callbacks stop with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:20-22`
  - body: `openclaw doctor --fix` copies global `cron.webhook` into per-job `delivery.to` and deletes `notify`. If that URL contains userinfo, the repaired job is now invalid and the scheduler skips it, so doctor should validate and warn instead of writing an unusable webhook.
  - confidence: 0.84
