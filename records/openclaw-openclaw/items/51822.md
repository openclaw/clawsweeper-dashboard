## Summary

Keep open: the credential-rejection guard is a useful security direction, but the current patch is incomplete because it can expose rejected stored webhook URLs in logs and leaves legacy cron migration paths unsafe or silent.

## What This Changes

The PR adds a username/password check to the cron HTTP webhook URL normalizer so credential-bearing HTTP(S) cron webhook URLs are rejected.

## Best Possible Solution

Adopt userinfo rejection in the shared cron webhook normalizer together with redacted invalid-target diagnostics, legacy fallback warnings, doctor migration validation, regression coverage, and explicit migration/release handling.

## Reproduction Assessment

Yes. Source inspection shows current main accepts credential-bearing HTTP(S) cron webhook URLs, and the PR body terminal screenshot shows the patched helper rejecting a `user:pass@` URL while accepting a plain URL.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch omits redaction, legacy diagnostics, doctor validation, and tests around the backward-incompatible policy change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:19-21`
  - body: When this guard makes an existing `delivery.to` with URL userinfo invalid, `dispatchGatewayCronFinishedNotifications` logs `params.job.delivery.to` raw in the invalid webhook warning. That can write the credential-bearing URL to logs on every run, so redact or omit the target before landing the guard.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook is rejected:** `src/cron/webhook-url.ts:19-21`
  - body: Legacy `notify: true` jobs resolve global `cron.webhook` through this normalizer. With credential-bearing legacy URLs rejected, `resolveCronWebhookTarget` returns `null`, but the invalid-target warning only covers explicit webhook delivery jobs, so callbacks can stop with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:19-21`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` and deletes `notify` without using this normalizer. After this guard, doctor can migrate a legacy fallback into an invalid per-job webhook; validate or warn instead.
  - confidence: 0.84
