## Summary

Keep this PR open: the security direction is useful, but the shared cron URL normalizer also gates stored runtime delivery and doctor migration paths, so rejecting credential-bearing URLs needs a maintainer-approved upgrade policy and broader proof before merge.

## What This Changes

The PR changes `normalizeHttpWebhookUrl` so cron webhook URLs with parsed username or password userinfo return `null`.

## Best Possible Solution

Adopt credential-userinfo rejection only with explicit upgrade handling or maintainer approval plus focused coverage for stored cron delivery, failure paths, and doctor legacy notify migration.

## Reproduction Assessment

No, not yet. Rejecting embedded credentials is a good safety direction, but changing the shared normalizer alone is not the best merge shape until persisted config upgrade behavior and sibling cron paths are covered.
