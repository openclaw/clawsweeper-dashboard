## Label Justifications

- `P2`: This is a focused security hardening PR for cron webhooks, but the blast radius is limited to webhook delivery and migration behavior.
- `merge-risk: 🚨 compatibility`: Merging this as-is can make existing stored credential-bearing webhook URLs invalid without an upgrade-safe migration path.
- `merge-risk: 🚨 message-delivery`: Legacy notify fallback jobs can stop sending finished-run webhook callbacks without a warning when `cron.webhook` becomes invalid.
- `merge-risk: 🚨 security-boundary`: The new rejection path can route rejected credential-bearing explicit webhook URLs into raw skipped-delivery logging.

## Summary

Keep open: the security goal is valid and the helper-level proof is sufficient, but the one-line fail-closed rejection still creates raw-secret logging, silent legacy webhook delivery loss, and an unsafe doctor migration path for existing cron jobs.

## What This Changes

The PR adds a `URL.username`/`URL.password` guard to `normalizeHttpWebhookUrl` so cron HTTP(S) webhook URLs with embedded credentials normalize to `null`.

## Best Possible Solution

Ship the userinfo rejection together with redacted invalid-target diagnostics, invalid legacy fallback warnings, doctor migration validation, and focused regression coverage for fresh and upgraded cron webhook configurations.

## Reproduction Assessment

Yes, at source level. Current main accepts any valid HTTP(S) URL in `normalizeHttpWebhookUrl`, the PR’s terminal screenshot shows the patched helper rejecting a credential-bearing URL, and source inspection traces the downstream skipped-delivery and doctor migration paths.

## Solution Assessment

No, not as currently proposed. The helper rejection is the right security primitive, but it needs redacted diagnostics, legacy fallback handling, doctor validation, and tests before it is the maintainable fix.
