## Label Justifications

- `P2`: This is focused security hardening for cron webhooks with limited blast radius, but it needs upgrade-safe handling before merge.
- `merge-risk: 🚨 compatibility`: Merging this as-is can make existing stored credential-bearing webhook URLs invalid without a safe migration path.
- `merge-risk: 🚨 message-delivery`: Legacy notify fallback jobs can stop sending finished-run webhook callbacks without a warning when `cron.webhook` becomes invalid.
- `merge-risk: 🚨 security-boundary`: The new rejection path can send rejected credential-bearing explicit webhook URLs into raw skipped-delivery logging.

## Summary

Keep open. The security primitive is useful, but this patch is not merge-ready because rejected stored cron webhook URLs still flow through raw logging, silent legacy fallback loss, and unsafe doctor migration paths.

## What This Changes

The PR adds a username/password guard to `normalizeHttpWebhookUrl` so cron HTTP(S) webhook URLs with embedded credentials normalize to `null`.

## Best Possible Solution

Ship the userinfo rejection together with redacted invalid-target diagnostics, invalid legacy fallback warnings, doctor validation or refusal, and regression coverage for fresh and upgraded cron webhook configurations.

## Reproduction Assessment

Yes, at source level. Current main accepts any valid HTTP(S) URL in `normalizeHttpWebhookUrl`, and the inspected terminal screenshot shows the patched helper rejecting a credential-bearing URL while accepting a plain HTTPS URL.

## Solution Assessment

No, not as currently proposed. The helper rejection is the right security primitive, but the safe fix also needs redacted diagnostics, legacy fallback handling, doctor migration validation, and tests.
