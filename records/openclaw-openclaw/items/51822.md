## Summary

Keep open. Current main still accepts credential-bearing cron webhook URLs, but this PR should not merge as submitted because the new rejection path can expose existing stored credentials in invalid-target logs and leaves legacy notify/doctor migration behavior unsafe.

## What This Changes

The PR adds a username/password check to `normalizeHttpWebhookUrl` so parsed cron webhook URLs with URL userinfo return `null`.

## Best Possible Solution

Land cron URL userinfo rejection through the shared normalizer only with redacted invalid-target logging, explicit legacy `cron.webhook` diagnostics and doctor handling, focused regression tests, and a changelog entry for the breaking security hardening.

## Reproduction Assessment

Yes, from source inspection. Current main accepts `https://user:pass@example.invalid/hook` because `normalizeHttpWebhookUrl` only checks parseability and http(s), and the PR’s new `null` path flows into the raw invalid-delivery warning for stored webhook jobs.

## Solution Assessment

No. The shared normalizer is the right seam for rejecting URL userinfo, but this patch alone is not the best solution because it changes stored/legacy runtime behavior without redacted logging, legacy warnings, or doctor migration safeguards.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Redact rejected stored webhook URLs:** `src/cron/webhook-url.ts:19-21`
  - body: When this guard makes an existing credential-bearing `delivery.to` invalid, `dispatchGatewayCronFinishedNotifications` falls into the skipped-delivery warning and logs the raw `params.job.delivery.to` as `deliveryTo`. That would write the full userinfo URL to logs on every finished run, so the invalid-target path needs to redact or omit the target before this lands.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook becomes invalid:** `src/cron/webhook-url.ts:19-21`
  - body: Legacy `notify: true` jobs resolve the global `cron.webhook` through this same normalizer. With this guard, a credentialed legacy webhook becomes `null`, but the invalid-target warning only covers explicit `delivery.mode === "webhook"` jobs, so those callbacks stop with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:19-21`
  - body: Doctor currently migrates `notify: true` jobs by copying `cron.webhook` into `delivery.to`. If that global webhook contains userinfo, this PR makes the migrated job invalid after `doctor --fix`, converting a working legacy fallback into a skipped per-job webhook instead of warning or refusing the rewrite.
  - confidence: 0.84
