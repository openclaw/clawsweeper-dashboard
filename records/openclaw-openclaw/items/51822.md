## Summary

Keep open: this is a focused cron webhook security hardening PR, but the current patch is not safe to merge because rejecting stored credential-bearing URLs routes them into unredacted logging and leaves legacy notify and doctor migration paths without a safe upgrade story.

## What This Changes

The PR adds a username/password rejection guard to cron's shared HTTP webhook URL normalizer.

## Best Possible Solution

Adopt userinfo rejection in the shared cron webhook normalizer only with redacted invalid-target diagnostics, explicit invalid legacy `cron.webhook` warnings, doctor migration validation or refusal, regression coverage, and release/migration notes.

## Reproduction Assessment

Yes. Current main source shows http(s) webhook URLs are accepted without checking `URL.username` or `URL.password`, and the PR body terminal proof shows the patched helper rejecting a credential-bearing URL while accepting a plain URL.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch needs redaction, legacy fallback handling, doctor migration validation, tests, and explicit migration policy before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact stored webhook targets before rejecting them:** `src/cron/webhook-url.ts:20`
  - body: When this guard rejects an existing stored `delivery.to` with URL userinfo, `dispatchGatewayCronFinishedNotifications` logs the raw `params.job.delivery.to` as `deliveryTo`. That can write the credential-bearing URL to gateway logs on every finished run, so redact or omit the invalid target before shipping the rejection.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook is rejected:** `src/cron/webhook-url.ts:20`
  - body: Legacy `notify: true` jobs run the global `cron.webhook` through this helper, but the invalid-target warning only covers explicit webhook delivery jobs. If the new guard rejects a legacy credentialed URL, finished-run callbacks stop without a migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` and deletes `notify` without using this normalizer. With this guard, a credentialed legacy fallback can be migrated into an invalid per-job webhook that the scheduler skips, so doctor should warn or refuse that migration.
  - confidence: 0.84
