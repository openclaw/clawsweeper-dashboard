## Summary

Keep open. The security hardening is useful and targets the right helper, but the branch still creates unsafe rejected-target paths for existing credential-bearing cron webhook config and leaves legacy fallback/doctor migration behavior unhandled.

## What This Changes

The PR adds a username/password check to `normalizeHttpWebhookUrl` so cron webhook URLs with URL userinfo are rejected.

## Best Possible Solution

Ship the shared userinfo rejection with redacted invalid-target diagnostics, explicit legacy fallback warnings, doctor migration validation or refusal, and focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main accepts credential-bearing HTTP(S) cron webhook URLs after only the protocol check, and the PR proof shows the patched helper rejecting one.

## Solution Assessment

No. The shared normalizer is the right seam, but the patch should also repair the rejected-target logging, legacy fallback visibility, doctor migration safety, and regression coverage before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected webhook targets:** `src/cron/webhook-url.ts:19-20`
  - body: Rejecting userinfo here makes stored `delivery.mode = "webhook"` values with credentials return `null`, and `dispatchGatewayCronFinishedNotifications` logs `params.job.delivery.to` as `deliveryTo` when normalization fails. That writes the raw credential-bearing URL to logs on every finished run, so redact or omit the rejected target before adding this rejection.
  - confidence: 0.91
- **[P2] Surface rejected legacy cron.webhook targets:** `src/cron/webhook-url.ts:19-20`
  - body: Legacy `notify: true` jobs resolve `cron.webhook` through this helper, but the invalid-target warning only runs for explicit `delivery.mode = "webhook"`. After this new `null` result, credentialed legacy fallback deliveries become silent no-ops with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:19-20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` for legacy notify jobs without validating it. With this rejection in place, doctor can convert a previously delivering legacy fallback into stored metadata the scheduler later rejects.
  - confidence: 0.84
