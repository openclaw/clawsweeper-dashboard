## Summary

Keep this PR open: the shared normalizer is the right place for the guard, but the current diff is incomplete for a security hardening because it can make stored credential-bearing webhook targets invalid while leaving raw logging, legacy fallback, and doctor migration paths unsafe.

## What This Changes

The PR adds a guard to cron webhook URL normalization so parsed HTTP(S) URLs with `username` or `password` are rejected.

## Best Possible Solution

Reject URL userinfo through the shared normalizer, redact invalid-target diagnostics, warn on invalid legacy fallback, guard doctor migration, and add focused regression coverage plus release-note handling for the breaking security hardening.

## Reproduction Assessment

Yes, by source inspection. Current main accepts credential-bearing HTTP(S) cron webhook URLs through `normalizeHttpWebhookUrl`, and the PR screenshot shows the proposed helper-level rejection after the patch.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch is not the best complete fix until the associated redaction, legacy fallback diagnostics, doctor migration guard, and regression tests are handled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:20-21`
  - body: When this guard makes an existing `delivery.to` invalid, finished-run dispatch logs `params.job.delivery.to` as `deliveryTo` without redaction. That can expose the same userinfo URL this hardening is trying to block, so the invalid-target path needs to redact or omit the target before merge.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook becomes invalid:** `src/cron/webhook-url.ts:20-21`
  - body: Legacy `notify` jobs resolve the global `cron.webhook` through this normalizer. With this guard, a credentialed legacy URL becomes `null`, but the invalid-target warning only covers explicit `delivery.mode="webhook"` jobs, so those callbacks silently stop instead of telling users how to migrate.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:20-21`
  - body: Doctor still migrates legacy notify fallback jobs by copying `cron.webhook` into `delivery.to` and deleting `notify`. If that global webhook contains userinfo, this patch turns the repaired job into an invalid per-job webhook that the scheduler skips.
  - confidence: 0.84
