## Summary

Keep this PR open: the hardening target is valid, but the current diff is incomplete because the new rejection path can expose raw stored webhook credentials in logs and can break legacy notify/doctor migration paths without an operator hint.

## What This Changes

The PR adds a username/password check to `normalizeHttpWebhookUrl` so cron webhook URLs with URL userinfo return `null`.

## Best Possible Solution

Reject URL userinfo through the shared cron webhook normalizer while adding redacted diagnostics, legacy fallback warnings, doctor migration validation, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows credential-bearing HTTP(S) cron webhook URLs pass the shared normalizer, and the PR body terminal screenshot shows the patched helper rejecting that input and accepting a plain URL.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch needs adjacent log redaction, legacy fallback diagnostics, doctor migration validation, and tests before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:19-21`
  - body: When this guard makes an existing `delivery.to` with URL userinfo invalid, finished-run dispatch logs the raw `delivery.to` on the skipped-delivery path. That writes the credential-bearing URL to logs, so the warning should redact or omit the target before this guard lands.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook is rejected:** `src/cron/webhook-url.ts:19-21`
  - body: Legacy `notify: true` jobs resolve global `cron.webhook` through this normalizer. If userinfo is rejected, the target becomes `null`, but the invalid-target warning only covers explicit webhook delivery, so callbacks stop without a migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:19-21`
  - body: `openclaw doctor --fix` copies global `cron.webhook` into `delivery.to` and deletes `notify`. With this normalizer rejecting userinfo, doctor can rewrite a working legacy fallback into an invalid per-job webhook, so it should validate and warn instead.
  - confidence: 0.84
