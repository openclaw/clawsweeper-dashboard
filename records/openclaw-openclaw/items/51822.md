## Summary

Keep this PR open: the hardening targets the right shared cron webhook URL normalizer, but the current patch leaves credential-bearing stored targets on paths that can log raw URLs, silently drop legacy fallback delivery, and migrate invalid legacy webhooks into rejected job metadata.

## What This Changes

Adds a username/password check to `normalizeHttpWebhookUrl` so parsed HTTP(S) cron webhook URLs with URL userinfo return `null`.

## Best Possible Solution

Ship the userinfo rejection with redacted invalid-target diagnostics, explicit legacy fallback warnings, doctor validation or refusal, focused regression tests, and migration or release notes.

## Reproduction Assessment

Yes, at source level. Current main accepts credential-bearing HTTP(S) cron webhook URLs after only a protocol check, and the PR proof shows the patched helper rejecting one; I did not establish an end-to-end cron run in this read-only review.

## Solution Assessment

No. The shared normalizer is the right seam, but the patch should also handle redacted diagnostics, rejected legacy fallback visibility, doctor migration safety, and regression coverage before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:19-20`
  - body: When this guard rejects an existing `delivery.mode = "webhook"` target containing userinfo, `dispatchGatewayCronFinishedNotifications` reaches the invalid-target warning and logs `params.job.delivery.to` as `deliveryTo`. That can write credentials to logs on every finished run, so redact or omit the rejected target before adding this rejection.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook is rejected:** `src/cron/webhook-url.ts:19-20`
  - body: Legacy `notify: true` jobs resolve `cron.webhook` through this helper, but the invalid-target warning only runs for explicit `delivery.mode = "webhook"`. After this new `null` result, credentialed legacy fallback deliveries become silent no-ops with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:19-20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` for legacy notify jobs without validating it. With this rejection in place, doctor can convert a working legacy fallback into stored metadata the scheduler later rejects.
  - confidence: 0.84
