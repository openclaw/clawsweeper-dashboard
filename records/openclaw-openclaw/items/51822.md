## Summary

Keep this PR open. The hardening direction is useful, but the current one-guard patch is not merge-ready because it can send stored credential-bearing webhook URLs into raw logging and leaves legacy fallback and doctor migration paths without a safe, redacted upgrade behavior.

## What This Changes

The PR adds a username/password guard to `normalizeHttpWebhookUrl` so cron HTTP(S) webhook URLs with URL userinfo return `null`.

## Best Possible Solution

Ship URL userinfo rejection through the shared cron webhook normalizer together with redacted diagnostics, explicit legacy fallback handling, doctor validation or refusal, focused regression tests, and migration notes.

## Reproduction Assessment

Yes for the narrow behavior. Source and a Node URL probe show current main and the latest release accept credential-bearing HTTP(S) webhook URLs, and the inspected terminal proof shows the patched helper rejecting that form while accepting plain HTTPS.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch is not the best complete fix until it also redacts rejected stored targets, handles legacy fallback and doctor migration safely, and adds focused regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:20`
  - body: After this guard returns `null` for an existing stored `delivery.mode: "webhook"` URL with userinfo, `dispatchGatewayCronFinishedNotifications` logs `params.job.delivery.to` as `deliveryTo`. Redact or omit that value before rejecting URL userinfo, otherwise the hardening writes the raw credential URL to logs.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook is rejected:** `src/cron/webhook-url.ts:20`
  - body: Legacy `notify: true` jobs pass `cfg.cron.webhook` through this same normalizer, but the invalid-target warning only runs for explicit webhook delivery. Rejecting a credentialed legacy webhook would turn finished-run delivery into a silent no-op, so emit a redacted warning or migration hint for that path too.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:20`
  - body: `openclaw doctor --fix` copies `cfg.cron.webhook` into `delivery.to` and removes `notify` without using the normalizer. Once credentialed URLs are rejected, doctor can rewrite a working legacy fallback into a per-job webhook that the scheduler later skips.
  - confidence: 0.84
