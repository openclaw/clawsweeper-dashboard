## Summary

Keep this PR open: the security hardening is useful, but it is not merge-ready because the shared cron webhook normalizer also gates stored delivery, completion/failure webhooks, and doctor legacy notify migration, so rejecting userinfo URLs changes shipped upgrade behavior.

## What This Changes

The PR adds a username/password userinfo rejection check to the cron webhook URL normalizer.

## Best Possible Solution

Ship credential-bearing webhook rejection only with maintainer-approved upgrade policy and focused validation, runtime warning, and doctor migration coverage; otherwise use a warning or doctor migration path before fail-closed runtime rejection.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 return credential-bearing HTTP(S) URLs from the helper, while the PR screenshot shows the patched helper returning null for that shape.

## Solution Assessment

No, not yet. Helper-level rejection is a plausible hardening direction, but the compatibility-sensitive fail-closed change needs maintainer approval and upgrade-path coverage before it is the best complete fix.
