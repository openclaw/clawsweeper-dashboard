## Summary

Keep open: the hardening is useful, but the shared validator now rejects a URL form that current main and the latest release accept, so existing stored cron webhook deliveries can stop posting callbacks without maintainer-approved upgrade handling and focused coverage.

## What This Changes

The branch adds a username/password check to the shared cron HTTP(S) webhook URL normalizer.

## Best Possible Solution

Land a maintainer-approved cron credential policy that rejects new credential-bearing webhooks while proving stored-job, failure-destination, and legacy doctor behavior with focused upgrade tests and redacted warnings.

## Reproduction Assessment

Yes by source inspection: current main and v2026.6.8 accept credential-bearing http(s) webhook URLs, and the PR screenshot shows the helper rejecting them after the patch. I did not run an end-to-end cron add/run flow in this read-only review.

## Solution Assessment

No. The helper guard is a plausible security hardening, but because the helper gates stored runtime delivery and doctor migration, the best fix needs an explicit upgrade policy and focused coverage before fail-closing existing webhook URLs.
