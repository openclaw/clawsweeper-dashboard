## Summary

Keep open: the security direction is reasonable, but the patch changes a shared cron webhook validator that current releases accept, so stored primary, completion, and failure webhook deliveries can stop after upgrade without a maintainer-approved rollout policy and stronger cron-path proof.

## What This Changes

The PR changes the cron webhook URL normalizer to return `null` when a parsed HTTP(S) URL contains username or password userinfo.

## Best Possible Solution

Adopt credential-userinfo rejection only after maintainers choose the upgrade policy and the PR covers stored cron delivery, failure paths, and doctor migration behavior.

## Reproduction Assessment

Yes, from source inspection: current main returns credential-bearing HTTP(S) cron webhook URLs from the helper, while the PR returns `null`; the shared consumers then reject or skip those targets. I did not run a full cron job, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No, not yet. Rejecting embedded credentials is a good safety direction, but changing the shared normalizer alone is not the best merge shape until upgrade behavior and sibling cron paths are covered.
