## Summary

Keep this PR open: the userinfo rejection is a useful security hardening, but the current patch still needs upgrade-safe handling for stored and legacy cron webhook configurations before merge.

## What This Changes

The PR changes `normalizeHttpWebhookUrl` so HTTP(S) cron webhook URLs with parsed `username` or `password` are rejected.

## Best Possible Solution

Ship the userinfo rejection only with redacted invalid-target diagnostics, invalid legacy fallback warnings, doctor migration validation, and focused regression coverage for explicit and legacy cron webhook paths.

## Reproduction Assessment

Yes at source level. Current main accepts any valid HTTP(S) URL in `normalizeHttpWebhookUrl`, and the PR's guard clearly makes credential-bearing cron webhook URLs normalize to `null`, which flows into the inspected explicit, legacy, and doctor migration paths.

## Solution Assessment

No, not yet. Rejecting embedded credentials is the right direction, but the implementation needs upgrade-safe logging, legacy fallback diagnostics, and doctor migration handling before it is the maintainable fix.
