## Summary

Keep open: the PR targets the right shared cron webhook URL normalizer, but the patch is not ready because rejecting existing credential-bearing webhook URLs can expose raw targets in logs, silently disable legacy fallback callbacks, and let doctor migrate legacy config into invalid webhook delivery.

## What This Changes

The PR adds a username/password check to `normalizeHttpWebhookUrl` so cron HTTP(S) webhook URLs with URL userinfo are rejected.

## Best Possible Solution

Adopt userinfo rejection in the shared cron webhook normalizer together with redacted invalid-target diagnostics, legacy fallback warnings, doctor migration validation or refusal, regression coverage, and explicit migration/release handling.

## Reproduction Assessment

Yes. Source inspection shows current main accepts `https://user:pass@...` in `normalizeHttpWebhookUrl`, and the inspected PR body terminal screenshot shows the patched helper rejecting that form while accepting a plain HTTPS URL.

## Solution Assessment

No. The shared helper is the right seam, but this patch alone is not the best fix until rejected-target logging, legacy fallback diagnostics, and doctor migration handling are covered.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:20`
  - body: When this guard makes an existing `delivery.mode = "webhook"` target with URL userinfo invalid, `dispatchGatewayCronFinishedNotifications` logs `deliveryTo: params.job.delivery.to` raw. That can write the credential-bearing URL to gateway logs on every run, so the invalid-target diagnostic needs to redact or omit the target before this rejection ships.
  - confidence: 0.91
- **[P2] Warn on rejected legacy cron.webhook:** `src/cron/webhook-url.ts:20`
  - body: Legacy `notify: true` jobs run global `cron.webhook` through this helper. If this new guard rejects that legacy URL, `resolveCronWebhookTarget` returns null and no warning fires because the invalid-target warning only checks explicit webhook delivery jobs, so callbacks stop without migration guidance.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` and deletes `notify` without this normalizer. With the new guard, a credentialed legacy fallback is migrated into an invalid per-job webhook that the scheduler skips; doctor should warn or refuse that migration.
  - confidence: 0.84
