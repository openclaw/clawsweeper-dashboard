## Summary

## What This Changes

The PR adds a `normalizeHttpWebhookUrl` guard that rejects HTTP(S) cron webhook URLs whose parsed URL has a username or password.

## Best Possible Solution

Land the hardening only after maintainer approval and focused tests prove cron validation, delivery warnings, and doctor legacy notify migration handle stored credential URLs safely.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.6` return credential-bearing HTTP(S) URLs from the helper, while the PR and terminal proof show that the proposed guard returns `null`.

## Solution Assessment

No, not yet. Helper-level rejection is a plausible hardening direction, but it is not the best complete fix until maintainers accept the fail-closed upgrade behavior and tests cover the stored cron paths.
