## Summary

Keep this PR open: the cron credentialed-URL guard is useful security hardening, but the submitted patch is not merge-ready because it turns currently accepted stored cron webhook URLs into invalid delivery targets without a maintainer-confirmed upgrade policy.

## What This Changes

The PR adds a username/password userinfo rejection to the shared cron HTTP(S) webhook URL normalizer.

## Best Possible Solution

Land the guard only after the cron stored-config upgrade behavior is chosen and covered across validation, runtime delivery, failure delivery, and doctor migration.

## Reproduction Assessment

Yes. Current main returns trimmed HTTP(S) webhook URLs without checking parsed username/password, and the PR diff makes those same values invalid through the shared helper.

## Solution Assessment

No. The shared helper is the right central enforcement point, but unconditional fail-closed behavior is only a plausible hardening step until stored webhook configs and doctor migration have explicit upgrade policy.
