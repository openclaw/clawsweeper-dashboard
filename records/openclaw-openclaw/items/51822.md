## Summary

Keep this PR open: the credentialed cron webhook guard is useful security hardening, but it is not merge-ready because the shared helper also governs stored webhook delivery and legacy doctor migration without a maintainer-approved upgrade policy or focused boundary coverage.

## What This Changes

The PR changes `normalizeHttpWebhookUrl` so cron webhook URLs with a parsed username or password return `null`.

## Best Possible Solution

Land the central credentialed-webhook guard only with a maintainer-approved upgrade policy and focused tests across create/update validation, runtime completion/failure dispatch, failure alerts, and legacy notify doctor migration.

## Reproduction Assessment

Yes. Current main accepts credential-bearing HTTP(S) cron webhook URLs because the shared helper only checks protocol, and the PR screenshot shows the patched helper rejecting `https://user:pass@...` while accepting a normal webhook URL.

## Solution Assessment

No. The shared helper is the right central location, but this is only a plausible hardening step until upgrade behavior and regression coverage are added for stored jobs and doctor migration.
