## Summary

Keep this PR open. Current main still accepts credential-bearing cron webhook URLs, so the hardening target is real, but the submitted patch is not merge-ready because the new rejection path can log already stored credential URLs raw and does not handle legacy notify/doctor migration paths.

## What This Changes

The PR adds a guard to `normalizeHttpWebhookUrl` so parsed http(s) cron webhook URLs with a username or password return `null`.

## Best Possible Solution

Land URL-userinfo rejection in the shared cron webhook normalizer together with redacted invalid-target logging, explicit legacy `cron.webhook` warning or migration guidance, doctor handling that refuses or warns on credentialed legacy webhooks, focused tests across those paths, and a user-facing changelog/release-note entry for the intentional compatibility break.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a stored cron job with `delivery.mode = "webhook"` and `delivery.to = "https://user:pass@example.invalid/hook"`; with the PR guard, the normalizer returns `null` and the finished-notification warning logs the raw `delivery.to`.

## Solution Assessment

No. The shared normalizer is the right seam for rejecting URL userinfo, but this PR is incomplete until the invalid runtime log path is redacted and the legacy notify/doctor migration paths are handled with tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Redact rejected stored webhook URLs:** `src/cron/webhook-url.ts:19-21`
  - body: When this guard rejects an already stored `delivery.mode = "webhook"` URL containing userinfo, `dispatchGatewayCronFinishedNotifications` takes the invalid-delivery branch and logs `params.job.delivery.to` as `deliveryTo`. That prints the full credential-bearing URL on every run, so this rejection needs to land with redaction for that path.
  - confidence: 0.91
- **[P2] Warn on invalid legacy cron.webhook targets:** `src/cron/webhook-url.ts:19-21`
  - body: Legacy `notify: true` jobs resolve the global `cron.webhook` through this same normalizer. After this guard rejects a credentialed global webhook, `resolveCronWebhookTarget` returns `null`, but the invalid-target warning only runs for explicit `delivery.mode = "webhook"` jobs, so those legacy callbacks stop delivering with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:19-21`
  - body: Doctor copies `cfg.cron.webhook` into `delivery.to` for simple `notify: true` jobs. With this new normalizer, a credentialed global webhook can be migrated into an explicit webhook job that runtime then rejects, converting a working legacy fallback into skipped delivery.
  - confidence: 0.84

## Security Review

Concerns:
