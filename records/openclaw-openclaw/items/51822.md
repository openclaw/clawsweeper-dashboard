## Summary

Keep open: this is a focused security-hardening PR, but it is not merge-ready because the shared cron webhook normalizer also affects stored and migrated cron delivery targets that current main and the latest release accept.

## What This Changes

The PR changes `src/cron/webhook-url.ts` so `normalizeHttpWebhookUrl` returns `null` when a parsed HTTP(S) URL contains a username or password.

## Best Possible Solution

Adopt credential-userinfo rejection only after maintainers define the rollout policy for existing cron webhook configs and the PR covers each affected webhook consumer with focused proof.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.10` return credential-bearing HTTP(S) cron webhook URLs from `normalizeHttpWebhookUrl`, while the PR would return `null`. A live cron create/update and stored delivery path has not been proven in the PR evidence.

## Solution Assessment

No, not yet. Helper-level rejection is a plausible guardrail, but the best merge shape needs an explicit upgrade policy plus coverage for create/update validation, runtime delivery, failure destinations, and doctor migration.
