## Summary

Keep open: the security hardening is useful, but the current patch is not merge-ready because the new rejection path can expose stored credential-bearing webhook URLs in logs and leaves legacy cron migration paths incomplete.

## What This Changes

Adds a username/password guard to `src/cron/webhook-url.ts` so `normalizeHttpWebhookUrl` returns `null` for HTTP(S) cron webhook URLs with URL userinfo.

## Best Possible Solution

Ship URL userinfo rejection through the shared cron webhook normalizer together with redacted diagnostics, explicit legacy fallback handling, doctor validation or refusal, focused regression tests, and migration notes.

## Reproduction Assessment

Yes for the narrow behavior: current main and `v2026.5.12` accept credential-bearing HTTP(S) webhook URLs by source inspection, and the inspected terminal proof shows the patched helper rejecting that form. The logging and doctor-migration blockers are source-reproducible from the current call paths.

## Solution Assessment

No. The shared normalizer is the right seam, but this patch is incomplete until rejected stored targets are redacted, rejected legacy `cron.webhook` values warn clearly, doctor refuses or safely migrates invalid legacy webhooks, and focused coverage is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:20`
  - body: After this guard returns `null` for an existing stored `delivery.mode: "webhook"` URL with userinfo, `dispatchGatewayCronFinishedNotifications` logs the raw `params.job.delivery.to` as `deliveryTo`. Redact or omit that value before rejecting URL userinfo, otherwise the hardening can write the credential-bearing URL to logs.
  - confidence: 0.91
- **[P2] Warn on rejected legacy cron.webhook:** `src/cron/webhook-url.ts:20`
  - body: Legacy `notify: true` jobs pass `cfg.cron.webhook` through this normalizer, but the invalid-target warning only runs for explicit webhook delivery. Rejecting a credentialed legacy webhook would turn finished-run delivery into a silent no-op, so emit a redacted warning or migration hint for that path too.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:20`
  - body: `openclaw doctor --fix` copies `cfg.cron.webhook` into `delivery.to` and removes `notify` without using this normalizer. Once credentialed URLs are rejected, doctor can rewrite a legacy fallback into a per-job webhook that the scheduler later skips.
  - confidence: 0.84
