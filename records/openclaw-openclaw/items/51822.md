## Summary

Keep open: the hardening is focused and useful, but it changes shipped cron webhook behavior from accepting credentialed HTTP(S) targets to failing closed, so it still needs maintainer policy approval and upgrade-path coverage before merge.

## What This Changes

The PR adds a guard to `normalizeHttpWebhookUrl` that returns `null` when the parsed URL contains username or password userinfo.

## Best Possible Solution

Ship credential-bearing webhook rejection only with explicit maintainer approval and focused upgrade tests for validation, runtime warnings, and doctor migration; otherwise choose a warning/migration-first path.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.6` return credential-bearing HTTP(S) URLs from the helper, while the PR patch and terminal screenshot show the helper returning `null` after the new guard.

## Solution Assessment

No, not yet. Helper-level rejection is a plausible hardening direction, but the compatibility-sensitive fail-closed change needs maintainer approval and upgrade-path coverage before it is the best complete fix.
