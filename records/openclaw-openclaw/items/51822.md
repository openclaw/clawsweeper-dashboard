## Summary

Keep open: current main still accepts URL userinfo and the PR targets the right shared cron webhook seam, but merging this branch as-is would introduce raw credential logging plus legacy and doctor migration gaps that need maintainer/security judgment.

## What This Changes

The PR adds a username/password check in `normalizeHttpWebhookUrl` so HTTP(S) cron webhook URLs with URL userinfo return `null`.

## Best Possible Solution

Adopt the userinfo rejection in the shared normalizer together with redacted invalid-target diagnostics, legacy fallback warnings, doctor migration validation, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the shared normalizer accepts credential-bearing HTTP(S) URLs, and the PR body terminal screenshot shows the patched helper rejecting that input and accepting a plain URL.

## Solution Assessment

No. The shared normalizer is the right seam, but the patch needs adjacent redaction, legacy fallback diagnostics, doctor migration validation, and tests before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:19-21`
  - body: When this guard turns an existing `delivery.to` with URL userinfo into `null`, `dispatchGatewayCronFinishedNotifications` takes the invalid webhook branch and logs `params.job.delivery.to` raw. That can write the credential-bearing URL to logs on every run, so the warning needs to omit or redact the target before this guard lands.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook is rejected:** `src/cron/webhook-url.ts:19-21`
  - body: Legacy `notify: true` jobs resolve global `cron.webhook` through this normalizer. With this guard, credential-bearing legacy URLs become `null`, but the invalid-target warning only covers explicit `delivery.mode === "webhook"`, so callbacks stop without a migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhook before doctor migration:** `src/cron/webhook-url.ts:19-21`
  - body: Doctor still copies `cron.webhook` into `delivery.to` and deletes `notify` without using the normalizer. After this guard, `openclaw doctor --fix` can convert a working legacy fallback into an invalid per-job webhook and remove the fallback.
  - confidence: 0.84
