## Summary

Keep open: the security goal is valid and the helper-level proof is sufficient, but the current diff is not merge-ready because the new rejection path can expose stored credential-bearing webhook URLs in logs and break legacy cron webhook delivery/migration without safe diagnostics.

## What This Changes

The branch adds a guard in `src/cron/webhook-url.ts` so `normalizeHttpWebhookUrl` returns `null` when the parsed URL has a username or password.

## Best Possible Solution

Ship the userinfo rejection together with redacted invalid-target diagnostics, invalid legacy fallback warnings, doctor migration validation, and focused regression coverage for fresh and upgraded cron webhook configurations.

## Reproduction Assessment

Yes, at source level. Current main returns trimmed HTTP(S) URLs from `normalizeHttpWebhookUrl` without checking userinfo, the PR proof terminal screenshot shows the patched helper returning `null` for userinfo, and source inspection traces the resulting skipped-delivery and doctor migration paths.

## Solution Assessment

No, not as currently proposed. The helper rejection is the right primitive, but the maintainable fix also needs redacted diagnostics, legacy fallback handling, doctor validation, and tests before merge.
