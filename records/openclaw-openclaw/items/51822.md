## Summary

Keep this PR open: the security hardening is useful, but the current patch changes shipped cron webhook acceptance in a shared helper without an explicit upgrade policy for existing stored jobs and legacy doctor migration.

## What This Changes

The PR adds a username/password check to the shared cron HTTP webhook URL normalizer so credential-bearing webhook URLs are rejected.

## Best Possible Solution

Land the guard only after the cron stored-config upgrade behavior is chosen and covered across validation, runtime delivery, failure delivery, and doctor migration.

## Reproduction Assessment

Yes. Current main returns the trimmed HTTP(S) URL without checking parsed `username` or `password`, and the PR's terminal screenshot shows the patched helper returning `null` for a credentialed webhook URL.

## Solution Assessment

No. The shared helper is the right central location, but unconditional fail-closed behavior is only a plausible hardening step until stored webhook configs and doctor migration have an explicit upgrade policy.
