## Summary

Keep open. The hardening is targeted and the terminal screenshot proves the helper behavior, but current main and the latest release accept credential-bearing cron webhook URLs, so this PR needs maintainer-approved upgrade handling before it can safely land.

## What This Changes

The PR adds a username/password guard in `normalizeHttpWebhookUrl` so HTTP(S) cron webhook URLs with URL userinfo return `null`.

## Best Possible Solution

Ship credential-bearing cron webhook rejection only after maintainers approve the fail-closed upgrade policy and the patch covers create/update validation, stored runtime delivery, failure destinations, and doctor legacy notify migration behavior.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.8` return credential-bearing HTTP(S) URLs from the helper, while the PR diff and inspected terminal screenshot show the helper returning `null` after the new guard. I did not run an end-to-end cron delivery scenario in this read-only review.

## Solution Assessment

No, not yet. Helper-level rejection is a plausible hardening point, but it is not the best complete fix until the upgrade behavior for stored webhook jobs and doctor legacy fallback migration is approved and covered.
