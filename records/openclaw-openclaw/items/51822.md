## Summary

The PR addresses a valid cron webhook hardening gap, but it is not merge-ready because the current diff rejects stored credential-bearing webhook URLs without making the scheduler and doctor paths safe for upgrades or logs.

## What This Changes

The branch adds a username/password check to `normalizeHttpWebhookUrl` so credential-bearing HTTP(S) cron webhook URLs normalize to `null`.

## Best Possible Solution

Ship the userinfo rejection together with redacted invalid-target diagnostics, legacy fallback warnings, doctor migration validation, and regression coverage for fresh and upgraded cron webhook configurations.

## Reproduction Assessment

Yes at source level. Current main accepts userinfo in `normalizeHttpWebhookUrl`, and the inspected terminal screenshot shows the patched helper rejecting a credential-bearing URL while accepting a plain HTTPS URL.

## Solution Assessment

No, not as currently shaped. The shared normalizer is the right enforcement point, but the change also needs safe scheduler diagnostics and doctor migration handling before it is maintainable.
