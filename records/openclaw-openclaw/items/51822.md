## Summary

Keep open: the hardening target is valid and current main still accepts URL userinfo, but the one-line rejection is not mergeable because it can log rejected stored credentials and breaks legacy fallback/doctor migration without warnings.

## What This Changes

Adds a username/password check to `normalizeHttpWebhookUrl` so HTTP(S) cron webhook URLs with URL userinfo return `null`.

## Best Possible Solution

Ship the userinfo rejection with redacted invalid-target diagnostics, visible legacy fallback warnings or refusal, doctor validation, focused regression tests, and migration/release notes.

## Reproduction Assessment

Yes. Current main accepts credential-bearing HTTP(S) cron webhook URLs at the shared helper, and the PR proof shows the patched helper rejecting one; I did not run an end-to-end cron job in this read-only review.

## Solution Assessment

No. The shared normalizer is the right seam, but the patch is incomplete until rejected targets are redacted, legacy fallback rejection is surfaced, doctor avoids unsafe migration, and tests cover the policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact rejected stored webhook targets:** `src/cron/webhook-url.ts:19-20`
  - body: When this guard rejects a stored `delivery.mode="webhook"` URL with userinfo, finished-run notification handling falls into the invalid-target warning and logs `params.job.delivery.to` as `deliveryTo`. That can write the credential-bearing URL to logs on every run, so the rejection path needs to redact or omit the target before merge.
  - confidence: 0.91
- **[P2] Warn when legacy cron.webhook becomes invalid:** `src/cron/webhook-url.ts:19-20`
  - body: Legacy `notify: true` jobs resolve `cfg.cron.webhook` through this helper, but the invalid-target warning only runs for explicit `delivery.mode="webhook"`. After this new `null` result, credentialed legacy fallback deliveries become silent no-ops with no migration hint.
  - confidence: 0.86
- **[P2] Validate legacy webhooks before doctor migration:** `src/cron/webhook-url.ts:19-20`
  - body: `openclaw doctor --fix` copies `cron.webhook` into `delivery.to` for legacy notify jobs without validating it. With this rejection in place, doctor can convert a working legacy fallback into stored metadata the scheduler later rejects.
  - confidence: 0.84
