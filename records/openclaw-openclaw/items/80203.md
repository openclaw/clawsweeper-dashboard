## Summary

Keep open: this PR is a real public SDK feature that current main does not implement, but it should not merge yet because the new auth-profile gate misses profiles embedded in model refs and the external PR lacks real behavior proof.

## What This Changes

The branch adds `api.runtime.llm.completeStructured(...)`, structured LLM request/result types, auth-profile override config/docs, context-engine exposure, tests, and generated SDK baseline updates.

## Best Possible Solution

Land a maintainer-approved structured `runtime.llm` API only after profile selection is gated consistently for both explicit profile fields and model-ref suffixes, with redacted real runtime proof.

## Reproduction Assessment

Yes for the review finding: source inspection shows `model: "provider/model@profile"` reaches `resolveSimpleCompletionSelectionForAgent` and then `selection.profileId` without the new profile gate. The original PR is a feature/API addition, so current-main bug reproduction is not applicable.

## Solution Assessment

No: the overall direction matches the linked follow-up, but this patch is not mergeable until embedded auth-profile suffixes are gated consistently and contributor real behavior proof is supplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate profiles embedded in model refs:** `src/plugins/runtime/runtime-llm.runtime.ts:545-546`
  - body: The new profile gate checks only `params.profile`, but `prepareRuntimeCall` still resolves `params.model` through `resolveSimpleCompletionSelectionForAgent`, which splits a trailing `@profile` and passes it into auth prep. A plugin trusted for model override but not profile override can call `completeStructured` with `model: "openai/gpt-5.5@work"` and steer stored credentials, so split or normalize the model ref before the gate and reject embedded profiles unless profile override is trusted.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff introduces an auth-profile override policy, but model-ref profile suffixes can still bypass that policy.

Concerns:

- **[medium] Auth-profile override bypass through model refs:** `src/plugins/runtime/runtime-llm.runtime.ts:545`
  - body: Because the new gate checks only the explicit `profile` field, a model override containing a trailing auth-profile suffix can still flow into auth resolution through the existing model selection path. That lets a plugin steer which stored credential/account the host uses without `allowProfileOverride`.
  - confidence: 0.91
