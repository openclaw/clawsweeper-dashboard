## Label Justifications

- `P1`: The PR addresses a security-sensitive provider-key exposure path, but the current patch has an upgrade-breaking auth risk.
- `merge-risk: 🚨 compatibility`: The diff changes documented merge-mode behavior by removing existing plaintext `models.json` credentials that currently win.

## Summary

This is a plausible security fix, but it is not merge-ready because it can remove the only persisted credential for existing custom-provider setups and the PR only provides test-level proof.

## What This Changes

The PR strips provider `apiKey` values from generated `models.json` output and adds regression coverage for plaintext config keys, merge-mode plaintext keys, and SecretRef markers.

## Best Possible Solution

Keep the secret-stripping direction, but pair it with an upgrade-safe credential path or explicit fail-closed migration guidance before removing existing `models.json`-only keys.

## Reproduction Assessment

Yes, from source: current `main` serializes merged provider entries into `models.json`, and a plaintext configured or existing `apiKey` can flow into `plan.contents`. I did not run the regression test locally because this review is read-only.

## Solution Assessment

No, not yet: stripping prompt-visible secrets is the right direction, but removing existing merge-mode credentials without migration or an explicit fail-closed operator path is not the safest maintainable fix.
