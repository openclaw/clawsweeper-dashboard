## Summary

Close: this PR adds a finite native allowlist exception that current main, shipped docs, and the canonical maintainer decision intentionally reject; restricted Codex `toolsAllow` remains fail-closed unless omitted or wildcard.

## What This Changes

This PR enables Codex native Code Mode for finite full-native `toolsAllow` lists and decouples user MCP server projection for those native-only turns.

## Best Possible Solution

Keep finite `toolsAllow` fail-closed and use omitted `toolsAllow` or `toolsAllow: ["*"]` for Codex native shell/file access unless maintainers later approve exact per-native-tool enforcement.

## Reproduction Assessment

Yes for source/current behavior: current main and v2026.6.10 only enable native Code Mode for omitted or wildcard `toolsAllow`, and the canonical issue includes live isolated-cron proof for omitted, finite, and wildcard cases.

## Solution Assessment

No; enabling finite `exec/read/write/edit` to grant the broad Code Mode surface is contrary to the accepted fail-closed security boundary, so the safer path is omitted/wildcard access or a future exact enforcement design.
