## Summary

Keep open: the allowlist fix is useful, but the patch currently lets an explicit native tools allowlist enable the broader Codex native surface that also projects user MCP servers. That is a security-boundary issue, and the supplied proof is structured but still does not show the real app-server or cron wake path after the patch.

## What This Changes

The PR changes the Codex app-server runtime to enable native code mode when `toolsAllow` explicitly includes `exec`, `read`, `write`, and `edit`, with focused run-attempt tests for full, partial, and dynamic-only allowlists.

## Best Possible Solution

Keep the explicit `exec`/`read`/`write`/`edit` native code-mode support, but gate user MCP projection separately and add a regression test covering `mcp.servers` with a full native allowlist.

## Reproduction Assessment

Yes. Source inspection on current main shows a full explicit allowlist still returns false unless it contains a wildcard, and the PR adds a targeted regression path for `exec/read/write/edit`.

## Solution Assessment

No. The direction is right, but using the same native-mode decision to enable user MCP projection widens a restricted allowlist; the safer solution separates those gates.
