## Summary

Keep open: current main has plugin `session_start`, `before_prompt_build`, Active Memory, memory-lancedb, and `/new`/`/reset` startupContext, but it does not yet provide the requested automatic recall of bundled `session-memory` Markdown artifacts for naturally fresh first user turns. The remaining work is a product/API decision because it adds startup trigger semantics and memory-injection policy.

## What This Changes

The issue requests a `session:start` or equivalent startup hook that automatically injects recent `workspace/memory/*.md` session-memory files into new sessions.

## Best Possible Solution

Define one startup memory contract that specifies when natural first-turn sessions receive runtime-owned memory context, how it relates to startupContext and plugin hooks, and how session-memory artifacts are sanitized or summarized before automatic recall.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature request. Source inspection does show the requested gap: session-memory writes artifacts, startupContext only preloads them on bare `/new` or `/reset`, and no first-message session-memory recall path exists.

## Solution Assessment

Unclear pending maintainer design. A first-turn startup hook or startupContext expansion is plausible, but automatic recall policy must be coordinated with active-memory, memory-lancedb, and the open session-memory trust-boundary work.
