## Summary

Keep open. The discussion includes maintainer triage saying this Android/Termux stdin parent-death failure is separate from the merged Chrome MCP command/args work, and current main still routes the built-in existing-session browser profile through Chrome MCP using the upstream stdio transport directly.

## What This Changes

The issue reports that `profile="user"` existing-session browser control fails on Android Termux because the Chrome DevTools MCP subprocess sees stdin EOF, treats its parent as dead, and exits.

## Best Possible Solution

Keep the issue open and fix the browser Chrome MCP lifecycle path directly: preserve the Chrome MCP child stdin on Android/Termux, reuse or expose an OpenClaw-owned stdio/process transport through a boundary-safe seam, and add a focused regression test for the existing-session status/actions path when the MCP subprocess closes stdin or exits early.

## Reproduction Assessment

Yes. The issue gives concrete Termux setup steps and logs for `openclaw browser --browser-profile user status`, and current main still sends that command through the same Chrome MCP stdio attach path; live device reproduction was not run in this read-only pass.

## Solution Assessment

Unclear until a patch exists. The proposed direction, keeping child stdin open or supporting a non-stdio attach transport, matches the failure mode, but the maintainable fix should avoid a browser-plugin-to-core-internals import and preserve current Chrome MCP command/args behavior.

## Repair Work Prompt

Fix openclaw/openclaw#48867: `profile="user"` / `driver: "existing-session"` fails on Android Termux because the spawned `chrome-devtools-mcp` server sees stdin EOF and exits with parent-death detection. Current main constructs upstream `StdioClientTransport` directly in `extensions/browser/src/browser/chrome-mcp.ts`, while configured MCP servers use OpenClaw's owned stdio/process transport in `src/agents/mcp-stdio-transport.ts`.

Implement a focused browser Chrome MCP lifecycle fix that preserves child stdin until the session is intentionally closed and cleans up the child deterministically. Do not import core `src/**` internals directly from extension production code; either add/reuse a boundary-safe SDK/shared seam or keep a browser-private helper that follows the same behavior. Preserve existing Chrome MCP command/args resolution, `mcpCommand`/`mcpArgs`, stderr capture, pid reporting, ephemeral probe close behavior, and session cache/reconnect semantics. Do not change remote CDP or managed `openclaw` profile semantics.

Add focused tests that mock the transport/spawn layer and cover: Chrome MCP session creation uses open stdin/pipe semantics; an EOF/close/early child exit during `profile="user"` availability produces a clear `BrowserProfileUnavailableError` and does not poison later sessions; existing ephemeral status probes still close their temporary client; normal command/args tests remain unchanged. Add a user-facing `CHANGELOG.md` fix entry crediting `Thanks @fii6` if behavior changes. Do not credit bots.
