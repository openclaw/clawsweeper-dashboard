## Summary

Keep open. Maintainer discussion explicitly separates this Termux Chrome MCP stdin parent-death failure from the merged existing-session command/args fix, and current `origin/main` still routes `profile=user` through the browser plugin's direct SDK stdio transport path.

## What This Changes

The issue reports that `profile=user` existing-session browser attach fails on Android Termux because the spawned Chrome DevTools MCP subprocess sees closed stdin and exits.

## Best Possible Solution

Route existing-session Chrome MCP through a boundary-safe transport or lifecycle path that keeps stdio open, preserves the existing command/args endpoint behavior, and adds regression coverage for early subprocess stdin or close failures.

## Reproduction Assessment

Yes. The issue gives concrete Android 15/Termux steps and logs for `openclaw browser --browser-profile user status`, and current source inspection shows that path still reaches Chrome MCP stdio; live Termux reproduction was not run in this read-only pass.

## Solution Assessment

Unclear until a patch exists.
