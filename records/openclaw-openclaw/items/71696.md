## Summary

Keep open: current main still lacks Telegram chat-type reply mode, and the PR is useful, but the latest head still changes fallback precedence by defaulting `replyToModeByChatType` during schema parsing before runtime/global reply-mode fallbacks can apply.

## What This Changes

The PR adds Telegram per-chat-type reply mode config/schema/metadata, applies it to message and native command delivery, and passes Telegram message IDs into plugin command contexts.

## Best Possible Solution

Land a repaired version that keeps `replyToModeByChatType` optional, applies direct/group defaults only after explicit per-chat, runtime, and account-level fallbacks are absent, regenerates metadata, and documents the Telegram change in the active changelog.

## Reproduction Assessment

Yes. On the PR head, parsing a Telegram config with only `replyToMode` inserts `replyToModeByChatType`, and the resolver returns that inserted map before the runtime/account fallback.

## Solution Assessment

No. The PR touches the right Telegram/config/plugin surfaces, but schema-level defaults are the wrong layer for this fallback; resolver-owned final defaults are the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve replyToMode fallback before per-chat defaults:** `src/config/zod-schema.providers-core.ts:237-240`
