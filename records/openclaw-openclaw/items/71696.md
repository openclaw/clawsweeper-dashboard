## Summary

Keep this PR open. The feature is useful and current main does not already implement Telegram chat-type reply modes, but the current PR head still has a merge-blocking schema/default precedence regression and is missing the required user-facing changelog entry; the earlier type gap noted by Greptile appears addressed in the latest diff.

## What This Changes

This PR adds Telegram `replyToModeByChatType` config/schema/type/metadata, resolves reply mode per chat type in Telegram message and native-command delivery, and passes Telegram message ids through plugin command contexts.

## Best Possible Solution

Land a repaired version that keeps the Telegram per-chat option explicit, preserves existing global/runtime reply-mode behavior, applies direct/group defaults only at the resolver's final fallback layer, and documents the user-facing Telegram change in the active changelog.

## Reproduction Assessment

Yes. A source-level reproduction is to validate a Telegram config with `replyToMode: "first"` and no `replyToModeByChatType` on the PR branch; the schema inserts the per-chat default map, and the resolver then returns that map before the global/runtime fallback.

## Solution Assessment

No. The PR touches the right Telegram/config/plugin surfaces, but defaulting `replyToModeByChatType` in Zod is not the best fix because it changes precedence for existing validated configs; the safer alternative is an optional schema plus resolver-owned defaults after explicit fallbacks are absent.
