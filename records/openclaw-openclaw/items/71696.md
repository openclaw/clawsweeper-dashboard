## Summary

Keep this PR open. Current main does not already implement Telegram chat-type reply mode, and the PR is aimed at the right Telegram/config/plugin-command surfaces, but the current head is not merge-ready because the schema default makes absent per-chat config override existing replyToMode fallbacks and the user-facing changelog entry is missing.

## What This Changes

The PR adds Telegram chat-type-based reply mode config/schema/types/metadata, resolves it in message and native-command delivery, restores explicit reply_to_current/reply_to_id threading, and passes Telegram message IDs into plugin command contexts.

## Best Possible Solution

Keep the implementation direction, but make replyToModeByChatType optional in schema and metadata, preserve runtime/account replyToMode precedence when the per-chat map is absent, add regression coverage for validated configs, and document the user-facing Telegram behavior in the Unreleased changelog before merge.

## Reproduction Assessment

Yes. A source-level reproduction is to validate a Telegram config with replyToMode set and no replyToModeByChatType; the PR schema inserts the direct/group default and the resolver then returns that per-chat value before the fallback/account mode.

## Solution Assessment

No. The PR touches the right surfaces, but defaulting the per-chat map in Zod is not the narrowest maintainable solution; resolver-owned defaults after explicit per-chat, runtime fallback, and account replyToMode are safer.
