## What This Changes

The PR replaces the public npm update-check fallback `res.json()` read with a 512 KiB `readResponseWithLimit` read and adds oversized-response regression coverage.

## Best Possible Solution

Merge the bounded-reader path after redacted real output confirms the update-check/public-registry fallback still succeeds against normal registry metadata and maintainers accept the 512 KiB cutoff.

## Reproduction Assessment

Yes from source: current main calls `res.json()` on successful public-registry fallback responses, so an oversized body would be buffered before parsing. I did not run a memory-growth reproduction for an oversized response in this read-only review.

## Solution Assessment

Yes for the code shape: reusing `readResponseWithLimit` at the response boundary is the narrowest maintainable fix and avoids new config surface. The remaining blocker is real behavior proof and maintainer acceptance of the cap, not a different implementation layer.
