Overall confidence: 0.92

Full review comments:

- **[P1] Remove the core-owned Hermes provider path:** `src/migrations/registry.ts:4`
  - body: Current main resolves migration providers through the plugin registry and already has a merged Hermes provider under `extensions/migrate-hermes`. Registering Hermes directly in a new core `src/migrations` registry creates a second implementation path with different semantics and conflicts with the owner boundary.
  - confidence: 0.94
- **[P1] Require confirmation outside TTY too:** `src/cli/program/register.migrate.ts:129-132`
  - body: This guard only blocks apply without `--yes` when stdout is a TTY. In CI, cron, or piped output, stdout is often non-TTY, so the command can copy files and write config without explicit confirmation; current main rejects non-interactive apply without `--yes`.
  - confidence: 0.92
- **[P1] Quarantine executable MCP server imports:** `src/migrations/providers/hermes.ts:546-552`
  - body: Hermes config can supply `command`, `args`, `env`, and `cwd` through `mapMcpServers`, and this action writes those values directly into live `mcp.servers`. Importing an untrusted agent home can persist a stdio command that later runs when MCP connects, so executable servers need manual review or a disabled quarantine path.
  - confidence: 0.9
- **[P1] Redact nested secrets structurally:** `src/migrations/report.ts:13-15`
  - body: `mergeConfig` redaction stringifies the action and applies value regexes, while the Markdown report writes reasons verbatim. Nested secret-looking keys such as headers, cookies, credentials, or arbitrary tokens can survive into plan/report artifacts; use key-aware recursive redaction for all report fields.
  - confidence: 0.86
