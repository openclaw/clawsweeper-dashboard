  - body: Current main already resolves `openclaw migrate` providers through the plugin registry and ships Hermes as `extensions/migrate-hermes`. Adding a hard-coded core migration registry creates a second Hermes implementation with separate CLI shape, reports, config writes, and safety checks.

  - body: This guard only refuses when `process.stdout.isTTY` is true, so CI, scripts, and piped output can apply migrations without `--yes`. Non-interactive mutation should require explicit confirmation, as the current plugin-owned command does.
  - confidence: 0.97
- **[P1] Reject unsafe config keys before merging:** `src/migrations/config-merge.ts:28-29`
  - body: Hermes YAML-derived provider, MCP, and skill names flow into this merge, but the setter assigns arbitrary keys into normal objects. Keys like `__proto__`, `constructor`, or `prototype` can pollute prototypes before config is written.
  - confidence: 0.9
- **[P1] Redact migration reports structurally:** `src/migrations/report.ts:13-15`
  - body: Regex redaction over serialized `mergeConfig` actions misses secret-looking keys such as headers, cookies, and nested env values, while report rows write action reasons without redaction. Migration plan/report artifacts can persist imported credentials.
  - confidence: 0.88
- **[P1] Constrain archive targets before writing:** `src/migrations/apply.ts:221-229`
  - body: `archivePath` is joined directly with the report directory, so absolute or traversal paths can escape the report root; copy actions also trust plan targets without an apply-time base-directory check.
  - confidence: 0.88
- **[P1] Quarantine executable MCP imports:** `src/migrations/providers/hermes.ts:330-343`
  - body: The importer copies `command`, `args`, `env`, and `cwd` into live `mcp.servers`, which can persist executable stdio definitions from an imported agent home. Executable MCP entries need explicit review or a disabled/manual quarantine path.
  - confidence: 0.9
