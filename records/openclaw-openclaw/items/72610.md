  - body: Current main already resolves migration providers through plugin manifests/SDK, and `extensions/migrate-hermes` already declares the Hermes provider.   - body: Hermes-derived provider, MCP, and skill names can flow into this recursive merge and assign keys such as `__proto__`, `constructor`, or `prototype` on normal objects. Use the existing blocked-key guard or a null-prototype/safe map path before writing imported config.
  - confidence: 0.9
- **[P1] Require --yes for every non-dry-run apply:** `src/cli/program/register.migrate.ts:129-130`
  - body: This guard only refuses missing `--yes` when stdout is a TTY, so CI, scripts, or piped invocations can mutate state without explicit confirmation. Non-dry-run apply should require `--yes` for noninteractive use and prompt separately for interactive use.
  - confidence: 0.95
- **[P1] Redact migration reports structurally:** `src/migrations/report.ts:13-14`
  - body: Merged config is redacted by JSON-string regex and report reasons are written unredacted, so nested secret-looking keys such as headers, cookies, or tokens can survive in `plan.json` or `report.md`. Walk action objects structurally and redact all report fields by key and value before writing.
  - confidence: 0.88
- **[P1] Quarantine imported stdio MCP commands:** `src/migrations/providers/hermes.ts:544-551`
  - body: The importer writes MCP `command`, `args`, `env`, and `cwd` from another agent home into live `mcp.servers` config. Those entries can later spawn local processes, so stdio definitions should be skipped, quarantined, or require explicit review instead of being auto-enabled.
  - confidence: 0.86
