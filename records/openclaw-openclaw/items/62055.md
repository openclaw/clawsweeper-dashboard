## Summary

Keep open. Current main still does not contain the reported Windows stack mitigation: the CLI respawn planner exits early on `win32`, the unit test still locks in no Windows respawn, and a targeted search found no `--stack-size` or equivalent maximum-call-stack mitigation. The issue remains concrete, recent, and supported by two Windows reports plus a related open Windows tracker, not a fixing PR.

## What This Changes

The issue requests a Windows-specific fix for CLI startup crashes where heavy commands overflow V8’s default stack or hit heap OOM while evaluating OpenClaw’s large ESM module graph.

## Best Possible Solution

Add a narrow Windows startup mitigation in the existing CLI respawn planner, or an equally focused lazy-loading fix that prevents affected native-Windows commands from evaluating the oversized graph before startup. The end state should preserve existing skip guards, avoid broad plugin/runtime refactors, and include focused unit coverage plus Windows package smoke proof for the reported commands.

## Reproduction Assessment

Yes for the affected releases: the issue includes concrete Windows install/run steps, crash signatures, and stack-size controlled experiments, with a second user confirming related commands on a later 2026.4.x release. Current-main reproduction is not live-verified and should be checked on a Windows runner after a patch.

## Solution Assessment

Unclear but likely: the proposed `buildCliRespawnPlan()` stack-size change is the narrowest direct mitigation for the reported V8 stack limit, but the existing `win32` respawn short-circuit means the patch must prove Windows respawn is safe. Import-graph slimming is a reasonable follow-up if the startup flag is insufficient.
