## Summary

Keep open: current main still short-circuits CLI respawn on Windows before any startup flags can be added, has no `--stack-size` mitigation, and still documents native Windows CLI flows that include affected command families.

## What This Changes

The issue reports native Windows CLI startup crashes on heavy commands due to a large ESM graph exceeding V8's default stack.

## Best Possible Solution

Add a narrow Windows startup mitigation in the CLI respawn planner, with unit coverage for the preserved skip paths and native Windows package smoke for the reported commands.

## Reproduction Assessment

Yes for the affected releases: the issue provides install/run steps, crash signatures, controlled `--stack-size` experiments, and a second Windows confirmation. Current-main runtime reproduction still needs a native Windows runner, but source inspection shows the proposed mitigation is not present.

## Solution Assessment

Likely yes: adding a Windows-specific `--stack-size` respawn through `buildCliRespawnPlan()` is the narrowest direct fix for the reported V8 stack limit. Import-graph slimming should remain a follow-up if Windows package smoke still shows heap pressure.
