## Summary

Current main still does not implement the requested local-managed Playwright navigation interception. The browser plugin continues to reserve persistent Playwright tab creation for remote CDP profiles, while local-managed profiles fall into the raw CDP path and fail closed for explicit strict HTTP(S) navigation that needs redirect-hop inspection.

## What This Changes

The issue requests Playwright-backed navigation interception for local-managed browser profiles so strict SSRF policy can inspect redirect hops like remote CDP mode.

## Best Possible Solution

Implement a focused browser plugin change that routes local-managed profiles through the existing Playwright-backed createPageViaPlaywright/gotoPageWithNavigationGuard path when explicit strict SSRF policy requires redirect-hop inspection, while preserving Chrome MCP existing-session behavior and raw CDP paths for safe cases such as about:blank or non-strict navigation. Add local-managed strict-mode regression tests for IP-literal navigation and private redirect blocking, and update browser docs if the profile capability matrix or operator-facing behavior changes.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with a local-managed profile and browser.ssrfPolicy.dangerouslyAllowPrivateNetwork=false, openTab sees usesPersistentPlaywright=false, requiresInspectableBrowserNavigationRedirectsForUrl returns true for HTTP(S), and the code throws before any Playwright redirect inspection can run.

## Solution Assessment

Unclear as a blanket capability flip, but the issue direction is valid. The safer maintainable path is a targeted local-managed Playwright navigation path only when redirect-hop inspection is required, with explicit tests around fallback and lifecycle behavior.
