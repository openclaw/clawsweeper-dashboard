## Summary

Keep open: current main still has GitHub Copilot token and catalog paths that parse JSON directly, and the Copilot stream wrapper adds no compression guard; adjacent gzip fixes/reports do not cover this provider-specific auth and LLM path.

## What This Changes

GitHub Copilot provider should handle gzip-compressed Copilot API/token responses instead of surfacing raw gzip bytes to JSON or SSE parsers.

## Best Possible Solution

Add a narrow Copilot-owned compressed-response strategy for token refresh, catalog/embedding JSON, and model-stream requests, with byte-guarded tests and no global fetch behavior change unless shared transport proof demands it.

## Reproduction Assessment

No, there is not a high-confidence current-main live reproduction here. The issue provides concrete gzip-byte logs, and source inspection shows the matching Copilot JSON/SSE parse points, but I did not run real Copilot credentials on the affected Node/macOS setup.

## Solution Assessment

Unclear for the exact final patch: generic Fetch normally decodes gzip, so a broad core decompression wrapper is not the safest first move. The maintainable direction is a focused Copilot provider fix that either requests identity encoding where allowed or byte-detects/decompresses raw gzip only at Copilot-owned parse boundaries.
