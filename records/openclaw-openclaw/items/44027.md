## Summary

Keep open: the reported `/verbose` group-chat bug is valid, but this PR is stale against current main and would replace the current shared progress-delivery policy with an older group/native gate that regresses Slack, forum-topic, and native-command behavior.

## What This Changes

The PR updates `dispatchReplyFromConfig` to normalize the stored session verbose level and allow group-chat tool summaries when it is `on` or `full` while continuing to suppress native command-source sessions.

## Best Possible Solution

Layer explicit verbose `on` or `full` into current main's shared progress/tool-summary delivery policy, with focused regression tests and a changelog entry, while preserving the existing safety gates.

## Reproduction Assessment

Yes. A focused unit test can seed an ordinary group session with `verboseLevel: "on"`, trigger `opts.onToolResult({ text: "..." })`, and observe that current main still drops the text payload because `shouldSendToolSummaries` is false for non-forum groups.

## Solution Assessment

No. The PR patches an outdated gate; the safer solution is to extend the current shared policy so verbose opt-in affects ordinary group delivery without changing Slack non-direct, forum-topic, native, preview, media, or exec-approval behavior.
