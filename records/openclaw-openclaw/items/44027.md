## Summary

Keep open: the DingTalk/card-mode context makes the user-visible gap plausible, but this branch is conflicting, lacks required real behavior proof, changes privacy-sensitive group delivery semantics, and would suppress native command-source tool summaries that current main now expects to deliver.

## What This Changes

The PR changes reply dispatch so stored `/verbose on` or `/verbose full` allows tool summaries in group chats while preserving default group suppression and adding native command-source suppression.

## Best Possible Solution

Define a fail-closed trusted-room verbose-progress policy with typed config, docs warnings, and regression coverage, then allow group `/verbose` tool output only when that policy permits it.

## Reproduction Assessment

Yes, at source level. Current main blocks ordinary group verbose/tool summaries through `shouldSendVerboseProgressMessages`, drops text-only tool payloads on that path, and has regression coverage asserting the suppression baseline.

## Solution Assessment

No. A session-level `/verbose` override addresses the symptom, but the branch is stale against current main, regresses native command-source delivery, and bypasses the needed trusted-room policy decision for group output.
