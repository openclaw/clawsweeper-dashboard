## Summary

Keep open: current main already has the provider-auth framework and several real OAuth/subscription-style routes, but the broader request still needs provider-by-provider entitlement, terms, billing-mode, and UI/status clarity decisions before it can be treated as implemented or safely automated.

## What This Changes

The issue asks OpenClaw to support official provider OAuth or subscription-backed model access where providers make that available, while clearly distinguishing API credits from OAuth/subscription routes.

## Best Possible Solution

Keep the provider-owned auth boundary, avoid web-app scraping, and split remaining work into narrow provider or UX issues that define entitlement, billing-mode, setup labels, status output, and live proof for each official route.

## Reproduction Assessment

Not applicable: this is a feature and product-scope request, not a report of broken existing behavior. Source and docs inspection show partial current support plus provider-specific open gaps.

## Solution Assessment

Unclear as a single implementation: the existing plugin/provider auth boundary is the maintainable path, but each official subscription route needs provider-specific contract and policy review. The safer direction is incremental provider-owned support and clearer status/billing metadata, not a generic bypass layer.
