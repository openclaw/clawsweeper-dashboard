## Label Justifications

- `impact:auth-provider`: The issue directly affects provider/model override provenance and effective model selection for native child agents.

## Summary

Current `main` still contains the reported provenance collision: an implicit configured subagent model is stored as origin-less `auto`, then child execution clears it as legacy fallback state before model selection. The issue is valid, narrowly repairable, and not owned by another canonical report; related https://github.com/openclaw/openclaw/issues/58822 concerns precedence earlier in selection rather than this later persistence/cleanup defect.

## What This Changes

Correct native subagent spawning so a model selected from `agents.defaults.subagents.model` or per-agent subagent configuration survives session persistence and is used by the child run.

## Best Possible Solution

Represent implicitly configured subagent models as exact configured selections distinct from runtime failover state, then add regression coverage carrying the initial session patch through child model resolution while retaining cleanup of genuine legacy origin-less auto fallbacks.

## Reproduction Assessment

Yes at source level: current `main` deterministically creates the origin-less `auto` override and clears it before applying stored model selection. A focused test can reproduce the failure without external provider access.

## Solution Assessment

Yes in direction: correcting provenance at the spawn boundary is narrower and safer than weakening legacy fallback cleanup. The repair should prove both configured-model retention and continued cleanup of true legacy fallback entries.
