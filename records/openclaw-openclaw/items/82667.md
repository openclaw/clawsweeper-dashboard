## Summary

Keep open: the current-main sanitizer gap is real and this PR targets the right central path, but the proposed chat_window parser can drop real user text when the message body starts with a #<number> line after the generated context block.

## What This Changes

The PR extends inbound metadata stripping for message-tool Delivery hints and chat_window chronological context, exports the Delivery hint constant, and adds sanitizer regression tests plus a media-only test mock adjustment.

## Best Possible Solution

Land the central sanitizer fix after tightening the chat_window row boundary so generated metadata is removed without deleting blank-separated user bodies that look like issue or message ids.

## Reproduction Assessment

Yes. Source inspection shows current main emits Delivery and chat_window blocks, then historical user turns pass through stripInboundMetadata, whose current sentinel parser does not cover those shapes.

## Solution Assessment

No, not as written. The central sanitizer is the right place to fix this, but the chat_window parser needs a stricter generated-row boundary before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve # messages after chat_window context:** `src/auto-reply/reply/strip-inbound-meta.ts:274`
  - body: After confirming a chat_window header, this loop keeps consuming blank lines and every following line that starts with `#<digits>`. The generated context block is followed by a blank separator and then the actual user body, so a real message like `#1234 is the bug number...` immediately after chat_window context is dropped as if it were another context row. Tighten the row matcher to the generated chat-window line shape or stop at the body separator, and add that combined regression.
  - confidence: 0.9
