## Summary

Keep open: this is a valid replay-sanitizer bug fix with sufficient real Telegram prompt-dump proof, but the current parser still drops a real blank-separated user body that starts with a `#<id>` token after a generated chat_window block.

## What This Changes

The PR extends inbound metadata stripping for message-tool Delivery hints and chat_window context blocks, exports the Delivery hint constant, and adds sanitizer plus media-only test coverage.

## Best Possible Solution

Land the sanitizer after tightening chat_window consumption so it removes only generated context rows and preserves blank-separated user bodies.

## Reproduction Assessment

Yes. Source inspection shows current main emits Delivery and chat_window blocks, then historical user turns pass through `stripInboundMetadata`; I did not run tests because this was a read-only sweep.

## Solution Assessment

No, not as written. The central sanitizer is the right place to fix the replay bug, but the chat_window parser must stop at the generated block boundary or match generated rows more tightly before merge.

## Review Findings

Overall confidence: 0.9

- **[P2] Preserve # bodies after chat_window context:** `src/auto-reply/reply/strip-inbound-meta.ts:274-275`

Stop at the body separator or use a stricter generated-row matcher, and add that combined regression.
