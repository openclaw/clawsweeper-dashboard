## Summary

Keep open: this targets a real replay-metadata gap and includes adequate real behavior proof, but the new chat_window parser can strip legitimate user text that starts with a #<digits> line after the envelope.

## What This Changes

The PR extends `stripInboundMetadata` and its tests to remove the 2026.5.12 message-tool Delivery hint and chronological chat_window Conversation context block from replayed user messages.

## Best Possible Solution

Keep the replay-boundary sanitization approach, but stop chat_window stripping at the generated block boundary or validate the full generated transcript-line shape so real user body lines are preserved.

## Reproduction Assessment

Yes. Source inspection shows current main emits the two missing envelope shapes and the PR's new loop would consume an actual post-envelope user body line such as `#1234 is the bug...`.

## Solution Assessment

No. Extending the existing helper is the right direction, but the chat_window parser needs a safer delimiter before this is the best maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve user body lines starting with #:** `src/auto-reply/reply/strip-inbound-meta.ts:251`
  - body: The chat_window stripper keeps consuming through blank lines and any following `#<digits>` line. Because the generated prefix is separated from the actual user body by a blank line, a valid message like `#1234 is the bug I want to fix` after the envelope is stripped as if it were chat context; the added false-positive test only covers text without an envelope. Stop at the separator or require the full generated chat-window line shape before consuming it.
  - confidence: 0.91
