## Summary

Keep this PR open: current main still strips historical replay metadata through the central sanitizer, but that sanitizer does not recognize the 2026.5.12 Delivery hint or chronological chat_window block. The proposed boundary is appropriate, with one low-priority false-positive guard to tighten before merge.

## What This Changes

This PR adds Delivery and chronological chat_window sentinel handling to `stripInboundMetadata` and expands focused tests for replay sanitization.

## Best Possible Solution

Land the central sanitizer fix after tightening the chat_window false-positive guard, keeping replay cleanup channel-agnostic at `normalizeMessagesForLlmBoundary`.

## Reproduction Assessment

Yes, from source inspection: current main emits the Delivery and chat_window text blocks, then historical user turns pass through `stripInboundMetadata`, whose current sentinel set does not include either new shape. A focused call using that generated envelope would leave those blocks attached on current main.

## Solution Assessment

Mostly yes: extending the existing central sanitizer is the narrow maintainable fix for replay bloat. The chat_window branch should require at least one generated chronological row before consuming the header so exact user text is not stripped accidentally.
