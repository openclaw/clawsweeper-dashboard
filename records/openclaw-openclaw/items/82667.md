## Summary

Keep open: the PR addresses a real current-main sanitizer gap and has sufficient live proof, but the latest head still has a blocking false-positive in the chat_window parser that can delete a real user body beginning with a hash-number token after the metadata separator.

## What This Changes

The PR extends `stripInboundMetadata` to strip the message-tool Delivery hint and chat_window chronological context blocks, exports the Delivery hint constant, and updates focused sanitizer plus media-only tests.

## Best Possible Solution

Land the sanitizer fix after tightening the chat_window row boundary so generated context is removed without deleting separator-separated user text.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: a chat_window header, one generated row, a blank separator, and a real body beginning with a hash-number token will still be consumed by the PR head loop.

## Solution Assessment

No, not as written. The central sanitizer is the right place to fix the bug, but the chat_window parser must stop at the generated block boundary or match a stricter generated-row shape before merge.
