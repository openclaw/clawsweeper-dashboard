## Summary

Keep open: current main and the latest release still compute sandboxed `pty: true` as `usePty=false` without adding a sandbox-specific warning, so the reported silent-disable behavior remains. The issue is valid, but existing maintainer/security-review labels and an open fork fix candidate make this a maintainer-review path rather than a new autonomous fix lane.

## What This Changes

The issue asks OpenClaw to warn or error when `pty: true` is requested for the exec tool while sandbox mode disables PTY.

## Best Possible Solution

Keep PTY disabled under sandbox, but emit a clear warning through the existing exec warning path, add focused regression coverage, and document the sandbox limitation in the exec tool docs.

## Reproduction Assessment

Yes. Source inspection on current main shows sandboxed exec still computes `usePty` from `params.pty === true && !sandbox` and has no sandbox-specific warning branch before execution.

## Solution Assessment

Yes, warning rather than enabling sandbox PTY is the narrowest maintainable fix. It explains the ignored explicit request without weakening the sandbox boundary; throwing should be a maintainer choice because it would stop workflows that currently continue.
