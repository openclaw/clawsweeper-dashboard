## Summary

Keep open: current main and v2026.6.10 still suppress explicit `pty: true` whenever sandbox execution is selected, and the checked warning/docs paths do not tell the user that the PTY request was ignored. The direct target-repo fix attempts are closed unmerged, while the only open closing PR is a cross-repository candidate rather than a merged fix.

## What This Changes

The issue reports that explicit `pty: true` exec requests are silently ignored under sandbox execution instead of warning or failing.

## Best Possible Solution

Keep PTY disabled under sandbox unless maintainers choose broader sandbox PTY support, but emit a clear existing-path warning and document the sandbox limitation.

## Reproduction Assessment

Yes from source inspection: current main still computes sandboxed explicit PTY requests as `usePty=false`, and the checked warning path only covers host PTY spawn fallback. I did not run a live sandboxed exec command in this read-only review.

## Solution Assessment

Yes for a warning-only repair: using the existing warnings path explains the ignored explicit request without weakening the sandbox boundary. Enabling sandbox PTY or hard-failing the call would be a broader product/security decision.
