## Summary

Keep open: current main and the latest release still suppress `pty: true` whenever sandbox execution is selected, and the checked warning/docs paths do not tell the user that the explicit PTY request was ignored. The same-repo fix attempts are closed unmerged, while the only open closing PR is a fork candidate rather than a merged fix.

## What This Changes

The issue reports that exec requests with `pty: true` are silently downgraded when sandboxing is active, without warning the user.

## Best Possible Solution

Keep PTY disabled under sandbox unless maintainers intentionally choose broader sandbox PTY support, but emit a clear warning through the existing exec warning path and document the sandbox limitation.

## Reproduction Assessment

Yes from source inspection: current main computes sandboxed explicit PTY requests as `usePty=false` and no sandbox-specific warning branch was found before execution. I did not run a live sandbox command in this read-only review.

## Solution Assessment

Yes for a warning-only repair: adding a warning through the existing warnings path is the narrowest fix because it explains the ignored explicit request without weakening the sandbox boundary. Enabling PTY inside sandbox or hard-failing the call would be a broader security/product choice.
