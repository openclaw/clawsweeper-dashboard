## Summary

Keep this issue open. Current main still forces explicit `pty:true` off whenever exec resolves to a sandbox runtime, no sandbox-specific warning or rejection is emitted, the user-facing docs/schema still omit that limitation, and open PR #39349 still serves as the active linked implementation candidate.

## What This Changes

The issue requests that exec stop silently ignoring explicit `pty:true` when sandbox mode is active, either by warning or rejecting and by documenting the limitation.

## Best Possible Solution

Keep this issue open until #39349 or an equivalent narrow fix lands. The desired end state is an explicit sandbox `pty:true` contract, focused regression coverage, and aligned public docs/schema/tool-description text.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: select a sandbox runtime with `host=auto` or `host=sandbox`, pass `pty:true`, and current `createExecTool` computes `usePty` as false without adding a sandbox-specific warning.

## Solution Assessment

Unclear. Warning or rejection would both address the silent failure; the narrowest compatibility-preserving path is likely a warning plus docs/schema/tests unless maintainers decide sandbox PTY should be supported or fail closed.
