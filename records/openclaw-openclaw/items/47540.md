## Summary

Keep open. Current main still does not implement the requested pricing-threshold split or config key, but this PR is not merge-ready because it is stale against current Control UI auth/freshness work and introduces concrete regressions in bootstrap authorization, pricing-warning accuracy, stacked notice styling, and generated config docs artifacts.

## What This Changes

The PR adds `gateway.controlUi.chat.showPricingThreshold`, wires it through Gateway bootstrap and Control UI state, and changes chat context notices, styles, tests, and generated config docs to separate pricing-threshold hints from model context-limit warnings.

## Best Possible Solution

Refresh or replace the branch on current main, add the chat setting inside the already-authorized bootstrap response, implement the threshold split in the current `ui/src/ui/chat/context-notice.ts` view-model path, base pricing hints on a fresh request/context usage signal, and update only the current config-doc hash artifact.

## Reproduction Assessment

Yes. The auth regression is reproducible by comparing PR-head `src/gateway/control-ui.ts:347-376` with current main `src/gateway/control-ui.ts:759-808`; the pricing false positive follows from a session with high cumulative `inputTokens` and lower fresh `totalTokens`; the CSS issue follows from the PR-head import order and later `.context-notice` rule.

## Solution Assessment

No. The direction is useful, but this branch is not the best current solution because it applies the behavior to stale Gateway/UI paths, bases a billing-threshold claim on aggregate session input, and carries obsolete generated docs artifacts instead of integrating with current main’s auth, freshness, and config-doc workflows.
