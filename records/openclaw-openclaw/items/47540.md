## Summary

Keep open: current main does not have the threshold split, but this PR cannot merge safely because it is stale against the authenticated bootstrap path, uses aggregate session input for pricing notices, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a Control UI chat setting and bootstrap/UI wiring to display model context and higher-rate pricing threshold notices separately, with config schema/docs baselines and UI tests.

## Best Possible Solution

Rebuild the threshold split on current main in the existing context-notice model, keep bootstrap auth and fresh-total behavior intact, use a provider-owned or fresh-context threshold signal, align config docs generation, and require redacted real Control UI proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head against current main shows the stale bootstrap-auth path, aggregate-token pricing check, and CSS cascade issue. I did not run tests because this was a read-only review.

## Solution Assessment

No, not as-is. The direction may be useful, but the implementation needs a current-main rebase, a clearer pricing-threshold contract, and real behavior proof before it is the maintainable solution.
