## Summary

Keep open: current main still rejects the reported SSH-transport remote URL shape, while this PR’s proposed fix introduces a security-sensitive fail-open path and is not yet aligned with the existing SSH transport contract.

## What This Changes

The PR rewrites `gateway.remote.transport=ssh` remote gateway URLs to a loopback tunnel endpoint before `buildGatewayConnectionDetails` runs the WebSocket security check, with one gateway call regression test.

## Best Possible Solution

Keep the gateway URL validator fail-closed and align SSH transport endpoint resolution, docs, tests, and changelog around one explicit config contract.

## Reproduction Assessment

Yes. On current main, `gateway.mode=remote`, `gateway.remote.transport=ssh`, and `gateway.remote.url=ws://203.0.113.10:18789` make `buildGatewayConnectionDetails()` select the non-loopback URL and the current security check rejects it before any tunnel endpoint is considered.

## Solution Assessment

No as written. Rewriting to an effective loopback endpoint may be viable, but malformed SSH remote URLs must not fall back to a trusted local default and the endpoint shape/docs contract need to stay consistent.
