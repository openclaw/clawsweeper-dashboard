## Summary

Keep open. Current main does not contain this PR’s URL rewrite, and the proposed patch still has a blocking fail-closed regression in a security-sensitive gateway guardrail: malformed SSH remote URLs are converted to a trusted loopback endpoint before validation. The surrounding docs and macOS config also currently treat `gateway.remote.url` as the loopback tunnel URL for SSH mode, so maintainers should decide that product contract before accepting core normalization of remote-host URLs.

## What This Changes

The PR rewrites `gateway.remote.transport=ssh` gateway URLs to a `127.0.0.1` tunnel endpoint inside `buildGatewayConnectionDetails` and adds one gateway call test for that mapping.

## Best Possible Solution

Maintainers should first choose the SSH remote URL contract. If core should accept remote-host URLs for SSH transport, implement it as a fail-closed effective-endpoint derivation that preserves URL shape, does not mask malformed config, keeps direct-transport plaintext rejection unchanged, and includes tests for malformed, bare-host, explicit-path, and client/bootstrap callers. If the current loopback-url contract is preferred, close or redirect the PR and strengthen docs/doctor/config repair around `gateway.remote.url` plus `gateway.remote.sshTarget`.

## Reproduction Assessment

Yes. Current main can be checked by setting `gateway.mode=remote`, `gateway.remote.transport=ssh`, and a non-loopback plaintext `gateway.remote.url`, which still reaches the existing WebSocket security rejection; the PR-introduced malformed-URL bug is reproducible from the proposed helper logic with an invalid nonempty URL.

## Solution Assessment

No as written. Rewriting SSH remote URLs may be a viable direction, but this patch silently converts invalid config to a trusted default and changes a documented security-sensitive remote access contract without enough runtime coverage.
