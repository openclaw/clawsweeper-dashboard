## Summary

Keep this PR open. The underlying gateway SSH/plaintext guardrail problem is still not implemented on current main, but this patch needs revision because it turns malformed SSH remote URLs into a trusted loopback endpoint before the existing security validator can fail closed.

## What This Changes

The PR rewrites SSH-transport gateway remote URLs to a 127.0.0.1 tunnel endpoint in `buildGatewayConnectionDetails` and adds a gateway call test for that mapping.

## Best Possible Solution

If maintainers want core to support remote-host URLs for SSH transport, revise the patch to parse and validate the configured URL first, derive only the effective loopback endpoint from a valid URL, preserve port/path semantics including bare URLs, keep direct-transport plaintext rejection unchanged, and add regression coverage for malformed, bare-host, and explicit-path SSH cases while coordinating with #26250.

## Reproduction Assessment

Yes. A focused reproduction for the PR-introduced bug is `gateway.mode=remote`, `gateway.remote.transport=ssh`, no URL override, and a malformed nonempty `gateway.remote.url`; the patch catches the parse failure and returns `ws://127.0.0.1:18789` instead of failing closed.

## Solution Assessment

No as written. Mapping SSH transport to the effective loopback endpoint may be acceptable, but this implementation should not silently replace invalid remote config with a trusted destination and should be reconciled with the documented `remote.url` loopback contract.
