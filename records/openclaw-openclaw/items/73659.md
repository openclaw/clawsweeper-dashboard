## Summary

Keep open. Current main still validates `gateway.remote.url` directly before the WebSocket security check, so the underlying SSH-transport rejection remains plausible, but this PR is not mergeable as written because it changes a security-sensitive guardrail and converts malformed SSH remote URLs into a trusted loopback default.

## What This Changes

The PR adds an SSH-transport URL rewrite in `buildGatewayConnectionDetails` from a configured remote gateway URL to a `127.0.0.1` tunnel endpoint and adds one gateway call test for that mapping.

## Best Possible Solution

Maintainers should first settle the SSH remote-url contract. If core should accept remote-host URLs for `transport=ssh`, the implementation should parse and validate the configured URL fail-closed, derive only the effective loopback tunnel endpoint, preserve port/path/search/hash semantics, keep direct-transport plaintext rejection unchanged, and update tests, docs if the contract changes, and the changelog.

## Reproduction Assessment

Yes. Current main can be reasoned through with `gateway.mode=remote`, `gateway.remote.transport=ssh`, and `gateway.remote.url=ws://203.0.113.10:18789`: the helper selects that non-loopback URL and the security check rejects it before an SSH tunnel endpoint is used. The PR-introduced failure is also reproducible by passing a malformed nonempty SSH `remote.url` to the new helper.

## Solution Assessment

No as written. Rewriting SSH transport to an effective loopback endpoint may be the right direction, but the proposed patch should not silently replace invalid config with `ws://127.0.0.1:18789`, and it should preserve pathless URL shape before merge.
