## Summary

Keep this PR open. Current main still validates `gateway.remote.url` directly and has no `gateway.remote.transport === "ssh"` branch, so the proposed SSH-loopback mapping is not obsolete; however the patch changes a security boundary and currently fails open on malformed SSH remote URLs.

## What This Changes

The PR rewrites `gateway.remote.transport=ssh` remote gateway URLs to a loopback tunnel endpoint in `buildGatewayConnectionDetails` and adds a gateway call test for that mapping.

## Best Possible Solution

If maintainers want SSH transport to accept remote-host `gateway.remote.url` values, the core fix should validate and parse that URL fail-closed, derive only the effective loopback endpoint while preserving scheme/port/path semantics, keep direct-transport plaintext rejection unchanged, and add focused coverage for malformed, bare-host, explicit-path, and client-start cases.

## Reproduction Assessment

Yes. Current main can be checked with `gateway.mode=remote`, `gateway.remote.transport=ssh`, and `gateway.remote.url=ws://203.0.113.10:18789`, which still reaches the non-loopback WebSocket rejection; the PR-introduced bug is reproducible by giving the helper a malformed nonempty SSH remote URL.

## Solution Assessment

No as written. Mapping SSH transport to an effective loopback URL may be the right direction, but the patch should not silently replace invalid config with a trusted default and should preserve pathless URL shape before it is considered mergeable.
