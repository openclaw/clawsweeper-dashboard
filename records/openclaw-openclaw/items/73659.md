## Summary

Keep open. Current main does not implement the core behavior this PR proposes, and the related gateway report remains open, but the proposed patch has a concrete security regression and overlaps the documented SSH loopback-url contract. Maintainer review should settle that contract before merge.

## What This Changes

The PR adds an SSH-transport branch in gateway connection resolution to rewrite configured remote gateway URLs to a 127.0.0.1 tunnel endpoint and adds one gateway call test for a `/ws` SSH remote URL.

## Best Possible Solution

The end state should keep the plaintext non-loopback guard fail-closed while giving SSH-tunnel users one documented source of truth: either normalize/repair configs so `gateway.remote.url` is loopback and `sshTarget` stores the host, or add a carefully validated core mapper with fail-closed parse behavior, path/search/hash preservation, tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. Current main can be traced with `gateway.mode=remote`, `gateway.remote.transport=ssh`, and `gateway.remote.url=ws://203.0.113.10:18789`: the helper selects that non-loopback URL and the security check rejects it. The PR-introduced malformed-URL and pathless-URL behaviors are also reproducible from the proposed helper logic and Node URL parsing.

## Solution Assessment

No as written. Rewriting SSH transport to an effective loopback endpoint may be a valid direction, but this patch should not silently replace invalid config with `ws://127.0.0.1:18789`, should preserve pathless URL shape, and needs maintainer confirmation against the documented loopback-url contract.

Full review comments:

- **[P2] Fail closed on malformed SSH remote URLs:** `src/gateway/connection-details.ts:27-29`
  - body: `resolveSshTunnelLoopbackUrl` catches any parse error and returns `ws://127.0.0.1:18789`, so a malformed nonempty `gateway.remote.url` no longer reaches the existing validation path and can connect to a different local gateway with remote credentials. Please throw a config error or leave the original URL to the validator, and add a regression test for malformed SSH remote URLs.
  - confidence: 0.89
