## Summary

Keep open. The branch targets a real current-main failure path, but it is not mergeable as written because the new SSH URL rewrite can turn malformed config into a trusted loopback endpoint before the security validator runs; the existing discussion also correctly flagged path-shape drift for bare URLs and the missing changelog entry.

## What This Changes

The PR rewrites SSH-transport remote gateway URLs to loopback tunnel endpoints in `buildGatewayConnectionDetails` and adds a gateway call test for that mapping.

## Best Possible Solution

If maintainers want this SSH convenience behavior, implement it as a fail-closed URL normalization seam: parse and reject malformed SSH remote URLs, preserve the original scheme/port/path/query/hash semantics while replacing only the effective host with loopback, keep direct-transport plaintext rejection unchanged, add coverage for malformed, bare-host, explicit-path, and override cases, and update the changelog/docs contract accordingly.

## Reproduction Assessment

Yes. Current main can be checked with `gateway.mode=remote`, `gateway.remote.transport=ssh`, and a non-loopback `ws://` `gateway.remote.url`, which follows the existing `buildGatewayConnectionDetails()` rejection path; the PR-introduced bug is reproducible by passing a malformed nonempty SSH remote URL into the new helper.

## Solution Assessment

No as written. The direction may be acceptable after maintainer approval, but the current patch silently trusts a default loopback endpoint on parse failure and changes pathless URL shape, so it is not the narrowest safe fix.
