## Summary

Keep open. The PR is useful source work for the task-maintenance performance report, but the current head is not ready to merge against current main because it reintroduces the heavyweight chat-type module that main now forbids in task maintenance, loses current wedged-subagent recovery semantics, and lacks the required user-facing changelog entry.

## What This Changes

The PR adds a pass-local backing-session lookup cache in task-registry maintenance plus focused tests for reusing session-store reads and CLI chat-type derivation during one maintenance pass.

## Best Possible Solution

Land an equivalent per-maintenance-pass session lookup cache on current main that caches session-store reads without losing entry-level wedged-subagent handling, keeps CLI chat checks on `session-chat-type-shared.js`, retains focused lookup-count regression tests, adds the required changelog entry, and lets the linked performance issue close only after merge.

## Reproduction Assessment

Yes. Current main's source shows one `loadSessionStore` path per stale subagent/CLI task check, and the linked report provides CPU-profile stacks through that path; the PR regressions are also reproducible by static boundary/test comparison against current main.

## Solution Assessment

No. The pass-local cache is the right direction, but this PR head is not the best mergeable form because it conflicts with current-main parser boundaries and wedged subagent recovery semantics; a rebased or replacement patch should preserve both.
