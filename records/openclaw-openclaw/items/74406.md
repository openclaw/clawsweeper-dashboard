## Summary

Keep open. The performance problem is real and current main still lacks a pass-local session-store lookup cache, but this PR is not mergeable as-is against current main because it reintroduces the heavyweight chat-type module, drops current wedged-subagent recovery semantics, and lacks the required changelog entry.

## What This Changes

The PR adds a pass-local task-registry maintenance backing-session lookup cache plus harness injection and lookup-count tests for stale subagent and CLI task reconciliation.

## Best Possible Solution

Land a rebased equivalent pass-local cache that reduces repeated session-store loads while keeping current-main lightweight chat parsing, wedged-subagent lost-task semantics, focused regression tests, and a user-facing changelog entry; then let the linked performance issue close through the merge reference.

## Reproduction Assessment

Yes. Current main still has the per-task loadSessionStore path for stale subagent/CLI checks, and the linked report provides CPU-profile stacks through that path; the PR regressions are reproducible by static comparison against current-main boundary and wedged-subagent tests.

## Solution Assessment

No. The pass-local cache is the right direction, but this PR is not the best mergeable form because it undoes current parser and recovery semantics and omits required release-note coverage.
