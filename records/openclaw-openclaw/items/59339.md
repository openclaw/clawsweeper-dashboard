## Summary

Keep open. Current main still maps `sessions_send(timeoutSeconds=0)` to a hardcoded 30s A2A announce budget, and the downstream A2A wait/agent-step paths still cap waits at 60s, so the reported fire-and-forget reply loss remains source-backed. Related #40961 is useful overlapping work, but it does not cover the 30s fire-and-forget default and has its own timeout-policy concerns.

## What This Changes

Request to make the `sessions_send` A2A announce timeout configurable or derived from the tool timeout so asynchronous replies are not dropped after the current hardcoded budget.

## Best Possible Solution

Land a narrow `sessions_send` A2A timeout policy: honor positive `timeoutSeconds` consistently through the initial wait and nested announce/reply steps within a bounded maximum, use a documented configurable or generous default for fire-and-forget mode, and keep broader gateway connection/plugin-init timeout design in the separate gateway timeout discussion.

Yes. By source inspection, `sessions_send(timeoutSeconds=0)` returns accepted while the background A2A flow waits on a 30,000 ms announce budget and can drop the reply if the target run completes later; I did not run live tests because this review was explicitly read-only.

## Solution Assessment

No. Current main is not the best solution because sub-agent announce configurability does not cover this `sessions_send` path, and the related PR only partially addresses the cap without changing the fire-and-forget budget.

## Repair Work Prompt

Fix the reported `sessions_send(timeoutSeconds=0)` A2A announce timeout behavior from https://github.com/openclaw/openclaw/issues/59339. Current main sets `announceTimeoutMs = 30_000` for fire-and-forget sends, and `runSessionsSendA2AFlow` / `runAgentStep` cap waits at 60s, causing replies from slow or queued target sessions to be lost.
