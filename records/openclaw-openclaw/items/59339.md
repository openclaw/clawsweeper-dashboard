## Summary

Keep open: current main and v2026.6.8 still map `sessions_send(timeoutSeconds=0)` to a 30s A2A announce budget, while the existing 120s configurable announce timeout is subagent-scoped and does not cover this tool path.

## What This Changes

Request to make the `sessions_send` A2A announce timeout configurable or otherwise safer because the hardcoded 30s fire-and-forget wait can drop delayed replies.

## Best Possible Solution

Define one canonical `sessions_send` A2A timeout contract, then update the tool behavior, bounded wait policy, focused tests, and docs/schema together.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection on current main shows `timeoutSeconds=0` still creates a 30s A2A announce budget and can return before a delayed target reply is available.

## Solution Assessment

Unclear. A raw timeout knob may be useful, but maintainers should first choose whether the right contract is a longer default, reuse of the subagent announce timeout, an A2A-specific option, or a push-result pattern for fire-and-forget sends.
