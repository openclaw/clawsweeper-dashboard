## Summary

Keep open: current main and v2026.6.6 still hardcode a 30s fire-and-forget `sessions_send` announce window, so the central A2A reply-loss concern remains. ## What This Changes

The issue requests configurable or safer `sessions_send` announce timeout behavior because `timeoutSeconds=0` can drop late A2A replies after the hardcoded 30s announce wait.

## Best Possible Solution

Define the canonical `sessions_send` A2A timeout contract, preserve fire-and-forget replies with a tested default or explicit config path, update docs/schema/tests together, and resolve the related conflicting PR against that decision.

## Reproduction Assessment

No live reproduction was established in this read-only review. Source inspection on current main shows the reported `timeoutSeconds=0` path still uses a 30s announce wait and exits the A2A flow when the reply is not available before that wait.

## Solution Assessment

Unclear. A raw new timeout knob is not automatically the best fix; maintainers should first choose whether fire-and-forget A2A should use a longer default, reuse an existing subagent timeout concept, add a new A2A-specific config/parameter, or skip announce in favor of an explicit push-result pattern.
