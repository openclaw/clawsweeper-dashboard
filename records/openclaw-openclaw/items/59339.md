## Summary

Keep open. Current main still maps `sessions_send(timeoutSeconds=0)` to a 30s A2A announce budget and still caps the delayed wait paths at 60s, while the merged partial fix only keeps soft wait-window replies alive.

## What This Changes

Request to make the `sessions_send` A2A announce timeout configurable or otherwise prevent fire-and-forget replies from being dropped after the hardcoded timeout.

## Best Possible Solution

Define one bounded `sessions_send` A2A timeout policy that covers initial waits, fire-and-forget, reply turns, and announce turns consistently while leaving broader gateway startup timeout work to the separate gateway tracker.

## Reproduction Assessment

Yes, by source inspection. `sessions_send(timeoutSeconds=0)` still starts background A2A delivery with a 30,000 ms announce budget, and downstream delayed wait paths still cap at 60,000 ms, so replies completed after those windows can still be lost.

## Solution Assessment

No. Current main is only partially improved by the merged soft wait-window fix; it still needs a bounded timeout policy rather than the existing hardcoded fire-and-forget budget and scattered 60s caps.
