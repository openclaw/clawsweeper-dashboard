## Summary

Keep open: current main and the latest release still keep the reported `sessions_send(timeoutSeconds=0)` A2A announce path on a 30s budget, and the related PRs that attempted timeout changes are closed unmerged rather than safe canonical fixes.

## What This Changes

Request configurable or otherwise safer A2A announce timeout behavior so fire-and-forget `sessions_send` replies are not dropped after the hardcoded announce wait.

## Best Possible Solution

Define one canonical `sessions_send` A2A timeout contract, then update the tool behavior, bounded wait policy, regression tests, and docs/schema together so fire-and-forget replies are either delivered with a safe bounded wait or intentionally handled through an explicit push-result pattern.

## Reproduction Assessment

No live reproduction was run. Source inspection on current main shows the reported `timeoutSeconds=0` path still uses a 30s A2A announce wait and can return before a delayed target reply is available.

## Solution Assessment

Unclear. A raw new timeout knob may be useful, but maintainers should first choose whether the best contract is a longer default, reusing the existing subagent announce timeout, a new A2A-specific parameter, or skipping announce for fire-and-forget sends.
