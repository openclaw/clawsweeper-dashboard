## Label Justifications

- `P2`: This is a source-backed A2A delivery reliability issue with limited surface, but the fix requires timeout policy review rather than an urgent one-line regression fix.
- `impact:message-loss`: The reported failure mode is that generated A2A replies are not delivered back to the requester after the hardcoded wait budget expires.
- `impact:session-state`: The requester session can miss the target agent's generated reply, leaving session transcript/context state out of sync with the completed target work.

## Summary

Keep open. Current main and v2026.5.18 still have the reported hardcoded `sessions_send` fire-and-forget announce budget plus 60s delayed/nested A2A caps, and the related open PRs are not merged replacements for the remaining timeout-policy decision.

## What This Changes

Request to make `sessions_send` A2A announce timing configurable or tied to `timeoutSeconds` so fire-and-forget replies are not dropped after hardcoded wait budgets.

## Best Possible Solution

Define one bounded `sessions_send` A2A timeout contract that covers fire-and-forget, target run, delayed wait, reply-back, final announce, docs, and tests without unbounded model-controlled waits.

## Reproduction Assessment

Yes, source-reproducible: `timeoutSeconds=0` still computes `announceTimeoutMs` as 30,000 ms, while delayed A2A and nested agent-step waits remain capped at 60,000 ms. I did not run a live Windows slow-agent reproduction.

## Solution Assessment

No, a raw configurable timeout alone is not clearly the best fix. The safer direction is a bounded A2A timeout policy that preserves compatibility and aligns caller wait, target run, delayed reply, and announce behavior.
