## Summary

Keep open. Current main and v2026.6.5 still give `sessions_send(timeoutSeconds=0)` a fixed 30s announce budget, so the reported A2A reply-loss path is not implemented away; related work is either open or closed unmerged.

## What This Changes

The issue requests configurable or longer A2A announce timeout behavior so asynchronous `sessions_send(timeoutSeconds=0)` replies are not dropped after a fixed timeout.

## Best Possible Solution

Define one canonical A2A timeout policy for fire-and-forget sends, then update `sessions_send`, the A2A flow, regression tests, and docs so generated replies are not lost behind the fixed 30s budget.

## Reproduction Assessment

Yes, from source at medium confidence: current `sessions_send(timeoutSeconds=0)` still maps A2A announce delivery to a 30s budget, so a target run that replies after that window can miss the follow-up flow. I did not run a live three-agent Windows reproduction.

## Solution Assessment

Unclear: a configurable timeout is plausible, but maintainers should first choose whether fire-and-forget uses a longer default, the caller timeout, the existing subagent announce timeout, or a separate sessions_send/A2A policy.
