## Label Justifications

- `P2`: This is a source-backed A2A delivery reliability issue with limited surface, but the fix needs timeout-policy review rather than emergency handling.
- `impact:message-loss`: The reported failure mode is generated A2A replies not being delivered back to the requester after the fixed wait budget expires.
- `impact:session-state`: The requester session can miss a target agent's generated reply, leaving transcript and conversational context out of sync with completed target work.

## Summary

Keep open. Current main and the latest release still route `sessions_send(timeoutSeconds=0)` through a fixed 30s A2A announce budget, with separate 60s caps in delayed wait and nested reply/announce steps, so the reported reply-loss scenario is not fully solved by current code or any merged related PR.

## What This Changes

Request configurable or otherwise policy-owned A2A announce timeouts so fire-and-forget `sessions_send` replies are not lost after the hardcoded 30s budget.

## Best Possible Solution

Define one bounded `sessions_send` A2A timeout contract covering fire-and-forget, target wait, delayed reply, ping-pong turns, final announce, docs, and regression tests without unbounded model-controlled waits.

## Reproduction Assessment

Yes, source-reproducible: current main still maps `sessions_send(timeoutSeconds=0)` to a 30,000 ms announce budget, with additional 60,000 ms caps in the delayed A2A path. I did not run a live Windows slow-agent reproduction in this read-only review.

## Solution Assessment

Unclear as requested: making the timeout configurable may be part of the fix, but a raw knob alone is not clearly the safest contract. Maintainers should first choose the bounded A2A timeout semantics for zero-timeout sends, delayed waits, ping-pong replies, and final announce delivery.
