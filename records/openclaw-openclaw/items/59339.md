## Label Justifications

- `P2`: This is a source-backed A2A delivery reliability issue with limited surface, but the fix needs timeout-policy review rather than an emergency one-line patch.
- `impact:message-loss`: The reported failure mode is that generated A2A replies are not delivered back to the requester after the fixed wait budget expires.
- `impact:session-state`: The requester session can miss the target agent's generated reply, leaving transcript/context state out of sync with completed target work.

## Summary

Keep open. Current main and v2026.5.19 still contain the reported `sessions_send(timeoutSeconds=0)` 30s announce budget plus separate 60s delayed/nested A2A caps, and the related open PR has unresolved proof and correctness blockers rather than being a merged replacement.

## What This Changes

Request to make the `sessions_send` A2A announce timeout configurable or otherwise long enough that fire-and-forget replies are not lost after slow target-agent processing.

## Best Possible Solution

Define one bounded `sessions_send` A2A timeout contract that covers fire-and-forget, initial target wait, delayed wait, ping-pong replies, final announce, docs, and tests without unbounded model-controlled waits.

## Reproduction Assessment

Yes, source-reproducible: `timeoutSeconds=0` still computes a 30,000 ms announce budget, while delayed A2A and nested agent-step waits remain capped at 60,000 ms. I did not run a live Windows slow-agent reproduction.

## Solution Assessment

No, a raw configurable timeout alone is not clearly the best fix. The safer direction is a bounded A2A timeout policy that preserves compatibility and aligns caller wait, target run, delayed reply, and announce behavior.
