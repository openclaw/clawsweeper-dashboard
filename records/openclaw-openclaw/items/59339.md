## Summary

Keep open: current main and v2026.6.10 still use a 30s `sessions_send(timeoutSeconds=0)` A2A announce budget, and the remaining question is the public timeout/default/config contract rather than conservative cleanup.

## What This Changes

Request to make `sessions_send` A2A announce timeout behavior configurable or otherwise avoid losing delayed fire-and-forget replies after the current 30s announce window.

## Best Possible Solution

Define one canonical A2A delivery timeout contract, then update the tool behavior, focused tests, and public docs/schema together.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows `timeoutSeconds=0` still uses a 30s announce budget and starts asynchronous A2A follow-up, so the reported timing shape is source-reproducible but not live-proven here.

## Solution Assessment

Unclear. A configurable timeout is plausible, but maintainers should first choose whether the canonical contract is a longer default, reuse of subagent announce timeout, a new A2A option, or durable/push-result delivery.
