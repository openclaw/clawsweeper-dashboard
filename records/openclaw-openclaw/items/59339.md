## Summary

Keep open: current main still hardcodes a 30s A2A announce budget for `sessions_send(timeoutSeconds=0)`, and neither the shipped release nor the adjacent open timeout-compensation PR covers this fire-and-forget timeout contract.

## What This Changes

The issue requests a configurable or otherwise safer announce timeout for A2A `sessions_send` so fire-and-forget replies are not lost after a hardcoded 30s announce wait.

## Best Possible Solution

Define one canonical `sessions_send` A2A timeout contract, then update the tool behavior, schema/docs if a config knob is chosen, focused tests, and live proof together.

## Reproduction Assessment

Yes for source-level reproduction: current main still maps `sessions_send(timeoutSeconds=0)` to a 30s A2A announce budget and the A2A helper stops if the target reply is not available inside that wait. No live Windows/provider reproduction was run.

## Solution Assessment

Unclear. A raw timeout knob may be useful, but maintainers should first choose whether the right contract is a longer default, reuse of the subagent announce timeout, an A2A-specific option, or a push-result pattern for fire-and-forget sends.
