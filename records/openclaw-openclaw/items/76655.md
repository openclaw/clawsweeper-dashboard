## Summary

Keep open: the latest head still misses the workspace-scoped current plugin metadata snapshot in the normal runtime-skipped Gateway startup path, so the proposed caches can fall back to repeated metadata scans in the hot sessions list path.

## What This Changes

The PR caches manifest model-id normalization and bundled setup CLI backend metadata, threads manifest plugin normalization through model/pricing helpers, and adds focused tests plus a changelog entry.

## Best Possible Solution

Keep the snapshot-keyed cache design, but route the Gateway default workspace or lookup-table metadata into both hot-path helpers during runtime-skipped startup and cover that case with regression tests.

## Reproduction Assessment

Yes, source-level. The runtime-skipped Gateway startup stores a workspace-scoped current metadata snapshot while the new helpers only recover workspace from active runtime state, so getCurrentPluginMetadataSnapshot rejects the snapshot and the hot path can reload metadata.

## Solution Assessment

No. Snapshot-keyed caching is the right direction, but the patch needs to use the Gateway workspace-compatible snapshot in the runtime-skipped startup path before it is the narrow maintainable fix.
