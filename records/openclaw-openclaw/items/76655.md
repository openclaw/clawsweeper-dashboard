## Summary

Keep open. The latest PR addresses the earlier stale-cache and changelog concerns, but the new manifest-normalization fallback now rebuilds full plugin metadata in no-current-snapshot callers and exact PR-head CI reproduces model-pricing-cache failures from those extra registry passes.

## What This Changes

The PR caches manifest model-id normalization policies and bundled setup CLI fallback metadata against the active plugin metadata snapshot, with regression tests and a changelog entry.

## Best Possible Solution

Keep the snapshot-keyed cache for Gateway-owned metadata, but preserve a lightweight no-current-snapshot fallback or thread an existing metadata view through callers so unrelated Gateway cache refreshes do not rebuild manifest metadata per model.

## Reproduction Assessment

Yes. Exact PR-head CI reproduces the blocker in `src/gateway/model-pricing-cache.test.ts`, and source inspection points to the new no-current-snapshot fallback that loads full plugin metadata during model normalization.

## Solution Assessment

No. Caching against the active Gateway snapshot is the right direction, but replacing the fallback raw-manifest path with repeated full metadata snapshot loads is not the narrow maintainable fix.
