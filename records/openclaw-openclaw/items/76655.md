## Summary

Keep this PR open: the latest head addresses the earlier changelog credit and active-runtime workspace case, but the normal Gateway startup path still publishes a workspace-scoped metadata snapshot while leaving active plugin runtime workspace unset, so both new caches can still miss the hot-path snapshot and fall back to cold metadata loads.

## What This Changes

The PR caches manifest model-id normalization and bundled setup CLI backend lookups against plugin metadata snapshots, threads manifest plugins through model normalization/pricing, and adds focused tests plus a changelog entry.

## Best Possible Solution

Keep the snapshot-keyed cache design, but make the hot-path helpers consume the actual Gateway workspace-compatible metadata snapshot in runtime-skipped startup and cover that case with regression tests.

## Reproduction Assessment

Yes, source-level. Current Gateway startup builds and publishes a workspace-scoped lookup table with runtime plugins skipped, while the PR helpers only recover workspace from active runtime state, so `getCurrentPluginMetadataSnapshot` rejects the current snapshot in that path.

## Solution Assessment

No. Caching against the active metadata snapshot is the right direction, but the PR must pass or preserve the Gateway default workspace for the runtime-skipped startup path instead of relying only on active plugin runtime state.
