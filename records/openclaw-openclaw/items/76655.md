## Summary

Keep open. The performance root cause is source-visible, but the current patch adds unkeyed metadata caches that can go stale across manifest edits, state-dir changes, and plugin metadata reloads, and it still needs the required changelog entry.

## What This Changes

The PR adds module-level caches for manifest model-id normalization policies and bundled setup CLI backend lookup, and updates the corresponding plugin tests.

## Best Possible Solution

Tie the performance cache to the existing plugin metadata snapshot or control-plane fingerprint lifecycle, keep freshness regression coverage, and add one Unreleased changelog entry for the sessions.list polling improvement.

## Reproduction Assessment

Source-level yes: current main loops through session rows and reaches manifest normalization and CLI backend metadata lookup paths that rescan plugin metadata. I did not rerun the live large-history CPU profile.

## Solution Assessment

No, not as currently proposed. Caching is the right direction, but unkeyed module-level caches with test-only invalidation are not the narrow maintainable solution for metadata that already has a snapshot/reload lifecycle.
