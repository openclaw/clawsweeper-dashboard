## Summary

Keep open. The updated PR addresses the earlier stale-cache and changelog concerns, but the new cache fast path still misses the actual Gateway snapshot because it calls the current-snapshot getter without the workspace directory that Gateway uses.

## What This Changes

The PR caches manifest model-id normalization policies and bundled setup CLI backend metadata against plugin metadata snapshot fingerprints, with regression tests and a changelog entry.

## Best Possible Solution

Thread or derive the workspace-scoped current metadata snapshot used by Gateway, keep the cold path fresh when no compatible snapshot exists, and cover the workspaceDir case in the cache tests.

## Reproduction Assessment

Yes, source-level. The `sessions.list` path builds rows in a loop, and the PR's no-workspace current-snapshot lookup cannot match the workspace-scoped Gateway lookup table, so repeated calls fall back to cold metadata loads.

## Solution Assessment

No. Caching against the active metadata snapshot is the right direction, but the current diff must use the same workspace-scoped snapshot that Gateway actually publishes.

## Review Findings

Overall confidence: 0.87

- **[P2] Use the workspace-scoped snapshot for model-id caching:** `src/plugins/manifest-model-id-normalization.ts:33`
  - body: Gateway creates the current plugin lookup table with `defaultWorkspaceDir`, and `getCurrentPluginMetadataSnapshot` returns undefined when a stored snapshot has a workspace but the caller omits it. This new call therefore misses the active Gateway snapshot during `sessions.list` and falls back to `loadPluginMetadataSnapshot` on every normalization lookup, leaving the reported manifest-scan hot path in place. Please pass or derive the Gateway workspace/snapshot here and add a regression with a workspace-scoped current snapshot.
  - confidence: 0.89

  - body: The bundled setup CLI cache has the same workspace mismatch: the current Gateway snapshot is workspace-scoped, but this lookup asks for only `env`, so `isCliProvider` still falls back to a cold metadata snapshot for each unresolved backend check. That keeps the `sessions.list` CLI-backend scan cost on the row hot path. Please make this helper consume the workspace-compatible current snapshot as well.
  - confidence: 0.87
