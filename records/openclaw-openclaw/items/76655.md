## Summary

Keep open. The source supports the reported sessions.list performance concern, but the proposed caches are not wired into existing plugin metadata invalidation/reload paths and the PR is missing the required changelog entry.

## What This Changes

The PR caches manifest model-id normalization policies and fallback bundled setup CLI backend metadata, with tests adjusted for manual cache invalidation and reduced snapshot calls.

## Best Possible Solution

Keep the narrow metadata caches, but key or invalidate them at plugin metadata lifecycle boundaries, add regression coverage for stale-cache cases, and add a single-line Unreleased changelog entry.

## Reproduction Assessment

Source-level yes, live profiling not run: current main loops over every session row and reaches model/plugin lookup helpers that synchronously scan manifest metadata. The PR's large-history Control UI polling steps are plausible, but this review did not execute the profile.

## Solution Assessment

No, not as currently proposed. Caching is the right direction, but the implementation needs lifecycle-aware invalidation/fingerprinting and the required changelog entry before it is the narrowest maintainable fix.
