## Summary

Keep open. Current main still has the macOS lsof command-name filter bug, and this PR is the right narrow direction, but the PR head is not merge-ready because the two new poll-path `ps` verifier calls use the default 2s timeout instead of the existing 400ms poll timeout; it also needs the required changelog entry and currently reports a dirty merge state.

## What This Changes

The PR changes Unix stale-gateway cleanup to parse raw lsof listener PIDs, verify candidate gateways with ps argv inspection, and update stale-PID/restart tests for macOS lsof p_comm behavior.

## Best Possible Solution

Land this PR or an equivalent narrow fix after bounding the two poll-path verifier calls with `POLL_SPAWN_TIMEOUT_MS`, adding the required Unreleased changelog entry, and resolving the branch's merge state while preserving raw lsof PID parsing, argv-based gateway verification, self/ancestor exclusions, Windows behavior, and focused macOS `cnode` regression coverage.

## Reproduction Assessment

Yes. The linked issue gives concrete macOS launchd reproduction steps, and the current-main source path can be reproduced by feeding lsof output like `p<PID>\ncnode` for a gateway process into the existing parser; the PR-specific timeout defect is directly visible in the diff.

## Solution Assessment

No, not yet. The argv-verifier direction is the narrowest maintainable fix and matches the Windows architecture, but the poll verifier calls must use the short poll timeout before this is safe to merge.
