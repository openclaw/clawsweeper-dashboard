## Summary

Keep open: the branch is useful but still has a source-level heartbeat drain hole and its real behavior proof does not convincingly cover the final PR head.

## What This Changes

The PR adds a `SystemEvent.audience` field, wraps internal system events in runtime-context delimiters at the generic drain, filters heartbeat selectors around that lane, and marks isolated cron awareness events internal.

## Best Possible Solution

Land the audience primitive only after internal events cannot be drained by heartbeat reply runs, coverage exercises the real reply path, and current-head live proof is refreshed; keep further producer migrations under https://github.com/openclaw/openclaw/issues/69492.

## Reproduction Assessment

Yes, at source level: current main queues cron awareness as an ordinary trusted-false system event, and the PR head can consume the new internal event during a real non-isolated heartbeat reply path before a normal user turn sees it. I did not run live cron/channel reproduction in this read-only review.

## Solution Assessment

No, not yet: wrap-on-drain is a plausible narrow direction, but the branch must make the drain heartbeat-aware or otherwise leave internal awareness queued until a normal reply turn. A safer fix is to add a regression that runs the real heartbeat reply path instead of stubbing `getReplyFromConfig`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid draining internal events during heartbeat replies:** `src/auto-reply/reply/session-system-events.ts:130`
  - body: `selectGenericSystemEvents` now selects every `audience: "internal"` entry, and this generic drain consumes those entries before building any reply. A non-isolated heartbeat calls the real `getReplyFromConfig` with the main session key, so an internal cron-awareness event can be wrapped into the heartbeat prompt and removed from the queue; if the heartbeat returns OK/silent/no-target, `prompt-prelude.ts` keeps `systemEventBlocks` out of `transcriptCommandBody`, and the next normal turn never receives the awareness. Please leave internal events queued for heartbeat runs or add a heartbeat-aware drain path, with coverage that does not stub `getReplyFromConfig`.
  - confidence: 0.9
