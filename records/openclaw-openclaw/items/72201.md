## Summary

Keep open: the latest PR head still leaves a source-level owner-auth regression where hidden internal cron-awareness events can downgrade heartbeat turns before the drain-time wrapper is applied.

## What This Changes

The PR adds optional `SystemEvent.audience`, wraps internal system events as runtime context, migrates cron-awareness events to that lane, updates owner-auth scanning, adds tests, and updates the changelog.

## Best Possible Solution

Make heartbeat's pending-event owner-auth scan ignore `audience: "internal"` while preserving the downgrade for user-facing untrusted events, then add a heartbeat regression before maintainer approval.

## Reproduction Assessment

Yes for the review finding: source inspection of PR head shows the migrated cron-awareness event is hidden but still `trusted: false`, while heartbeat preflight still checks only `event.trusted === false`. I did not run a live heartbeat scenario because this review is read-only.

## Solution Assessment

No, not as-is. The direct reply strip is useful, but the narrower maintainable fix also needs the earlier heartbeat pending-event scan to honor the same internal-audience carve-out.
