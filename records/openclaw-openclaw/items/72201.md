## Summary

Keep open: the patch still has a source-proven owner-auth regression because hidden internal cron-awareness events remain `trusted: false` and the heartbeat path ignores `audience` when setting `ForceSenderIsOwnerFalse`.

## What This Changes

The branch adds optional `SystemEvent.audience`, wraps internal-audience drained system events as runtime context, marks cron-awareness delivery events internal, adjusts the owner-auth scan, and adds tests plus a changelog entry.

## Best Possible Solution

Teach the heartbeat pending-event owner-auth scan the same internal-audience carve-out, then add a heartbeat regression proving internal cron-awareness events do not strip owner-only tools while user-facing untrusted events still do.

## Reproduction Assessment

Yes for the review finding: source inspection on PR head shows the migrated cron-awareness event is `audience: "internal"` and `trusted: false`, while `heartbeat-runner.ts` still downgrades ownership on any pending `trusted === false` event before the drain/wrap path runs.

## Solution Assessment

No, not as-is. The new `get-reply-run.ts` strip fixes the drained-block scan, but the same hidden event can still trip heartbeat's earlier pending-event scan; the safer fix is to make the heartbeat scan audience-aware and cover it with a heartbeat regression.
