## Summary

Keep open: the PR has a plausible direction and supplied runtime proof, but the current diff still lets hidden internal events trip the existing untrusted-system-event owner-auth guard, and it is missing the required changelog coverage.

## What This Changes

The branch adds optional `SystemEvent.audience`, wraps internal-audience drained events as runtime context, marks isolated cron awareness events internal, and adds focused tests.

## Best Possible Solution

Keep the hidden runtime-context lane narrow, but make internal wrapped events invisible to the untrusted-system-event owner-auth scan or avoid the visible untrusted prefix inside wrapped blocks, then add changelog coverage before maintainer API review.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR puts `System (untrusted):` lines inside an internal runtime-context block, and current main scans the full drained block to force `senderIsOwner=false`. This is not a standalone current-main bug reproduction; it is a PR-introduced regression path.

## Solution Assessment

No, not as-is. The audience primitive may be maintainable, but the current implementation needs to separate hidden runtime context from the existing owner-auth downgrade signal and add the required changelog entry.
