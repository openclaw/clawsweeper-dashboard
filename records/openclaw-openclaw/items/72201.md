## What This Changes

The PR adds `SystemEvent.audience`, wraps internal-audience system events as hidden runtime context, routes heartbeat/collect-mode consumers around that lane, and marks isolated cron awareness events internal.

## Best Possible Solution

Land the audience primitive only after maintainers accept the hidden-runtime-context boundary and the current head has green required checks, then keep future producer migrations case-by-case under the canonical system-event leakage issue.

Yes at source level: current main has no `audience` field and formats generic queued system events as visible `System:` lines, while the cron awareness producer enqueues without an internal lane. I did not run a live cron transport reproduction in this read-only review.

## Solution Assessment

Yes directionally: a producer-level audience field plus wrap-on-drain fits the existing internal-runtime-context contract and avoids per-surface display filters. The remaining question is maintainer acceptance of the new core contract and upgrade behavior, not a narrower code defect found in this pass.
