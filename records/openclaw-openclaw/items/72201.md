## Summary

Keep open: current main does not implement this audience primitive, and the active PR still has a concrete heartbeat-drain correctness issue plus insufficient current-head real behavior proof.

## What This Changes

The PR adds a SystemEvent audience field, wraps internal-audience system events as runtime context, migrates cron awareness events to that lane, and adds focused regression tests.

## Best Possible Solution

Keep the narrow audience primitive only if heartbeat replies cannot consume internal awareness before a normal turn, then require refreshed current-head proof or an explicit maintainer proof waiver before merge.

## Reproduction Assessment

Yes, at source level: PR head allows non-isolated heartbeat runs to call getReplyFromConfig on the main SessionKey, and that call drains internal events before the next normal user turn. I did not run a live cron/channel reproduction in this read-only review.

## Solution Assessment

No, not yet: wrap-on-drain is a plausible narrow direction, but the current patch needs a heartbeat-aware guard or durable preservation path for internal awareness events before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep heartbeat replies from draining internal events:** `src/auto-reply/reply/session-system-events.ts:128-130`
