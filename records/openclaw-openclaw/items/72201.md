## Summary

Keep open: the latest branch preserves current-main exec-completion filtering, but still introduces a blocking owner-auth regression for hidden untrusted events, lacks the required changelog entry, and has no real behavior proof.

## What This Changes

The PR adds optional `SystemEvent.audience`, wraps internal-audience drained events in internal runtime-context delimiters, marks isolated cron awareness events internal, and adds focused tests.

## Best Possible Solution

Keep the hidden runtime-context lane narrow, but make hidden internal events invisible to the untrusted-system-event owner-auth scan or avoid the visible untrusted prefix inside wrapped blocks, then add changelog coverage and real behavior proof before maintainer API review.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR creates an internal runtime-context block containing `System (untrusted):` lines, and current main scans the whole drained block to downgrade owner auth. Not applicable as a standalone user bug because this is a feature/API PR.

## Solution Assessment

No, not as-is. The audience primitive may be a reasonable direction, but the implementation must keep hidden context from tripping the existing untrusted-event auth guard and must include changelog and proof follow-through.
