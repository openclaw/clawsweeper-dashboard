## Summary

Keep open. The PR is a plausible implementation slice, but it is not merge-ready because internal events can still be consumed by a non-isolated heartbeat reply before the next normal user turn, and the real behavior proof has not been refreshed for the final PR head.

## What This Changes

The PR adds a `SystemEvent.audience` field, wraps internal system events in runtime-context delimiters, updates heartbeat selectors, marks cron awareness events internal, and adds tests plus a changelog entry.

## Best Possible Solution

Make the drain heartbeat-aware, or otherwise leave internal events queued during heartbeat reply runs, then add coverage for the real `runHeartbeatOnce -> getReplyFromConfig -> drainFormattedSystemEvents` path and refresh current-head real behavior proof.

## Reproduction Assessment

Yes, at source level. Enqueue an `audience: "internal"` event, run a non-isolated heartbeat through the real reply runner, and the generic drain can consume that event before a normal user turn while the heartbeat transcript path does not preserve the system-event block.

## Solution Assessment

No, not yet. Wrap-on-drain is a plausible narrow direction, but the current PR must keep internal events out of heartbeat reply drains or explicitly handle that path before it is the maintainable fix.
