## Summary

Keep this PR open, but it should not merge yet: the branch still lets a non-isolated heartbeat reply drain internal runtime-context events before the next normal user turn, and the real behavior proof does not demonstrate the final rebased head.

## What This Changes

The PR adds a `SystemEvent.audience` field, wraps internal-audience events in runtime-context delimiters at the generic drain, filters heartbeat selectors around that lane, marks cron awareness events internal, and updates tests plus changelog.

## Best Possible Solution

Land the audience primitive only after heartbeat reply runs cannot consume internal awareness events prematurely, with coverage through the real heartbeat reply path and refreshed final-head live proof.

## Reproduction Assessment

Yes, at source level: enqueue an `audience: "internal"` event, then run a non-isolated heartbeat through the real reply runner; the generic drain consumes the event before a normal user turn can receive the wrapped awareness. I did not run a live cron/channel reproduction in this read-only review.

## Solution Assessment

No, not yet: wrap-on-drain is a plausible direction, but the generic drain needs to be heartbeat-aware or otherwise leave internal events queued until a normal reply turn. The safer fix is to add a regression that exercises `runHeartbeatOnce -> getReplyFromConfig -> drainFormattedSystemEvents`, not only a stubbed heartbeat context.
