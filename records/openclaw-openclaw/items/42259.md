## Summary

Keep open. Current main still lacks configurable image-tool compression, and this PR is the canonical thread for the closed feature request, but the patch is not correct as-is because its fallback can violate the new bounds and it still lacks real behavior proof plus a settled config contract.

## What This Changes

This PR adds two `agents.defaults` image compression settings, threads them through image-tool JPEG media loading, adds config schema/help/types/tests, and includes unrelated README/docs/gateway recovery copy changes.

## Best Possible Solution

Land a focused compression configuration feature after maintainers settle the config shape, the optimizer preserves configured bounds in every fallback, and real image-tool proof shows the document-quality use case.

## Reproduction Assessment

Yes for the PR defects: source inspection of the PR head shows `minQuality: 95` and `maxSide: 600` can empty the filtered ladders and fall back to `40` or `800`. The original OCR/detail-loss motivation still lacks a real image-tool reproduction run.

## Solution Assessment

No. The feature direction may be useful, but this branch needs an agreed config contract, bound-preserving fallback candidates, focused tests, and real behavior proof before it is the maintainable fix.
