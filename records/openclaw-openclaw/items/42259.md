## Summary

Keep this PR open. Current main still lacks configurable image-tool JPEG compression, and this PR is a plausible implementation path, but the patch is not merge-ready because its fallback logic violates the proposed constraints for out-of-ladder values and the branch also carries unrelated docs/gateway cleanup that should be split or deliberately accepted.

## What This Changes

The PR adds two `agents.defaults` image compression config fields, threads `minQuality` and `maxSide` into image-tool media loading and JPEG optimization, adds focused tests, and also changes README, provider-link, doctor, gateway, and debugging docs text.

## Best Possible Solution

Ship a focused implementation on current main that deliberately chooses the public config contract, adds type/schema/help/docs coverage under `agents.defaults`, threads compression options through the current image-tool and `src/media/web-media.ts` path without weakening media access or SSRF guards, and adds regression tests for out-of-ladder values such as quality `95` and max side `600`. Split unrelated README/provider-link/doctor/gateway wording changes into separate PRs unless maintainers explicitly want them here.

## Reproduction Assessment

Yes for the PR defect: at PR head, `minQuality: 95` or `maxSide: 600` deterministically filters the candidate ladders to empty and the fallback then selects `40` or `800`. The broader OCR-quality feature request has no high-confidence real-image reproduction attached.

## Solution Assessment

No, this PR is not the best solution as written because the fallback logic breaks the proposed contract and the branch mixes unrelated changes. The safer path is a rebased, scoped patch that uses the configured constraint itself when no ladder candidate qualifies and adds tests for those exact edge cases.
