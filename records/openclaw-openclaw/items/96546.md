## Label Justifications

- `P1`: The default daily reset path can split an active turn's transcript and disconnect normal session history from live output.
- `impact:data-loss`: The output can remain on disk but become effectively lost from normal session access because it is split into archived and orphaned files.
- `impact:message-loss`: Post-boundary output can be written to an orphan transcript instead of the session history users expect to contain the running turn.
- `impact:session-state`: The session row rotates while the active run continues writing through the old transcript path, creating session/transcript state drift.

## Summary

Keep open: current main still makes the inbound daily-stale decision inside auto-reply session initialization and archives the previous transcript before the later active-run admission path runs, so the reported live-transcript split remains a valid session-state bug.

## What This Changes

The issue reports that an inbound daily session reset can archive a live transcript while an active turn is still writing, splitting one turn's output across an archived file and a recreated orphan.

## Best Possible Solution

Add a narrow auto-reply session rollover guard so implicit stale daily/idle rollover cannot rename a transcript while an active run is still writing, with regression coverage proving the active turn remains in one referenced transcript.

## Reproduction Assessment

Yes at source level: current main still evaluates daily staleness and archives the previous transcript inside initSessionState before runPreparedReply reaches active-run admission. I did not run a full live gateway/provider rollover repro in this read-only review.

## Solution Assessment

Yes: the best fix is a focused guard on the auto-reply inbound rollover path before destructive transcript archive, matching the sibling compaction invariant. A scheduler-only change would not cover this inbound archive path.
