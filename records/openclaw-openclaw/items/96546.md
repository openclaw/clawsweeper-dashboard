## Label Justifications

- `P1`: The default daily reset path can split an in-flight reply transcript and disconnect normal session history from live output.
- `impact:data-loss`: The running turn output can remain only in archived or orphaned transcript files that the resulting session no longer references.
- `impact:message-loss`: Post-boundary reply output can be written outside the session history users expect to contain the running turn.
- `impact:session-state`: The session row can rotate while the active run continues writing through the old transcript path, creating session/transcript state drift.

## Summary

Keep open: current main still archives stale daily-reset transcripts inside auto-reply session initialization before the later active-run gate, and the open closing PR is the right landing path rather than a reason to close early.

## What This Changes

The issue asks for implicit daily rollover to avoid renaming or orphaning the live transcript while a reply run is still active.

## Best Possible Solution

Review and land the focused active-run stale-rollover guard in https://github.com/openclaw/openclaw/pull/96715, then close this issue after the fix is merged.

## Reproduction Assessment

Yes at source level: current main still evaluates daily staleness and archives the previous transcript inside initSessionState before runPreparedReply reaches active-run admission. I did not run a live wall-clock provider rollover in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to defer implicit stale rollover while a matching active reply run owns the session, which is the approach in the open closing PR. The issue should stay open until that PR or an equivalent fix merges.
