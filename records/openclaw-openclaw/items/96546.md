## Label Justifications

- `P1`: The default daily reset path can split an in-flight reply transcript and disconnect live output from normal session history.
- `impact:data-loss`: Part of the running turn can become effectively lost from normal session access when output is split into archived and orphaned files.
- `impact:message-loss`: Post-boundary output can be written to an orphan transcript instead of the referenced session history users expect.
- `impact:session-state`: The session row can rotate while the active run continues writing through the old transcript path, creating session/transcript state drift.

## Summary

Keep open: current main still evaluates stale daily rollover and captures `previousSessionEntry` before the later active-run admission path, so the live-transcript split is still a valid bug. https://github.com/openclaw/openclaw/pull/96715 is an open, clean candidate fix with closing syntax, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that implicit daily session rollover can archive a live auto-reply transcript mid-turn, splitting output across archived and orphaned JSONL files.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96715 or an equivalent focused guard that defers implicit stale rollover while the matching active reply run owns the session, then close this issue after merge.

## Reproduction Assessment

Yes at source level: current main evaluates daily staleness, captures `previousSessionEntry`, and archives before the later active-run admission path. I did not rerun a full wall-clock provider stream in this read-only review.

## Solution Assessment

Yes: deferring implicit rollover before the destructive archive decision when the same session has an active reply run is the narrow owner-boundary fix. A scheduler-only change would not cover this inbound auto-reply archive path.
