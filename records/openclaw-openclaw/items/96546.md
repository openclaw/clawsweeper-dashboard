## Label Justifications

- `P1`: The default daily reset path can split an active turn's transcript and disconnect the running output from normal session history.
- `impact:data-loss`: The output can become effectively lost to normal session access because it is split between archived and orphaned transcript files.
- `impact:message-loss`: Post-boundary assistant output can be written to an orphan transcript instead of the session history users expect.
- `impact:session-state`: The session row can rotate while the active run continues writing through the old transcript path, creating session/transcript state drift.

## Summary

Keep open: current main still performs inbound daily-stale rollover inside auto-reply session initialization before the later active-run admission gate, so the reported live transcript split remains a valid session-state bug. The issue is already paired with an open closing fix PR, so cleanup should wait for that PR to land or be rejected.

## What This Changes

The issue reports that default daily session rollover can rename a live transcript while a reply run is still writing, splitting one turn across archived and orphaned JSONL files.

## Best Possible Solution

Land or revise the open closing fix so implicit daily/idle rollover reuses or defers the current session while an active reply run owns it, with regression coverage proving the transcript remains in one referenced file.

## Reproduction Assessment

Yes at source level: current main still evaluates daily staleness and archives the previous transcript inside initSessionState before runPreparedReply reaches active-run admission. I did not execute the wall-clock/provider repro in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow guard on the auto-reply inbound rollover path so implicit stale rollover cannot archive or mint away from a session while an active run owns it. A scheduler-only change is adjacent and would not cover this inbound archive path.
