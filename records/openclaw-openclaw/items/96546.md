## Label Justifications

- `P1`: The default daily reset path can split an active reply transcript and break a live session workflow.
- `impact:session-state`: The issue is about session row and transcript ownership drifting while a reply run is active.
- `impact:data-loss`: Turn output can remain only in archived/orphaned files that the resulting session does not reference.
- `impact:message-loss`: Post-boundary assistant output can be disconnected from the normal session transcript read path.

## Summary

Keep open: current main still evaluates stale daily rollover inside auto-reply session initialization and can archive the previous transcript before the later active-run gate runs. The linked fix PR is open and appears to own the remaining work, so this issue should close only after that or an equivalent fix lands.

## What This Changes

Prevent implicit daily/idle session rollover from archiving or rotating a transcript while the matching reply run is still active.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/96715 or an equivalent auto-reply guard that preserves the active session entry and transcript path until the active run finishes, then close this issue as implemented.

## Reproduction Assessment

Yes, at source level: current main still evaluates daily staleness and archives the previous transcript inside `initSessionState` before `runPreparedReply` reaches active-run admission. The reporter also supplied a deterministic function-level repro, but I did not execute it in this read-only review.

## Solution Assessment

Yes: deferring the whole implicit rollover while the matching active reply run owns the session is the narrowest maintainable fix. The archive-only approach in https://github.com/openclaw/openclaw/pull/96552 is weaker because it can still mint a new canonical session while the active run writes the old transcript.
