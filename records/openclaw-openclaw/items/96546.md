## Label Justifications

- `P1`: The default daily reset path can split an active turn's transcript and disconnect normal session history from live output.
- `impact:data-loss`: Output can become effectively lost from normal session access when it is split between an archived file and an orphaned recreated file.
- `impact:message-loss`: Post-boundary turn output can be written outside the session transcript users expect to contain the running conversation.
- `impact:session-state`: The session row can rotate while the active run continues writing through the old transcript path, creating session/transcript state drift.

## Summary

Keep open: current main still evaluates the inbound daily-stale rollover in auto-reply session initialization and archives the previous transcript before the later active-run admission path checks whether the same session is running. The issue is now paired with the open closing PR https://github.com/openclaw/openclaw/pull/96715, so the right cleanup path is to review and land or reject that PR before closing this issue.

## What This Changes

Prevent daily session rollover from renaming a live transcript while an active reply run for that session is still writing.

## Best Possible Solution

Land a focused auto-reply session rollover fix that reuses the existing session entry while that entry's reply run is active, with regression coverage for active and inactive stale daily sessions.

## Reproduction Assessment

Yes at source level: current main still marks the session stale in initSessionState and passes the previous entry for archival before runPreparedReply reaches the active-run gate. I did not run a full wall-clock/provider rollover in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow auto-reply session initialization guard like https://github.com/openclaw/openclaw/pull/96715 that defers implicit rollover while the existing session has an active reply run. A gateway scheduler-only change is adjacent and does not cover this inbound archive path.
