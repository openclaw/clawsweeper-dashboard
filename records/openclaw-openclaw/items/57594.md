## Summary

Keep open: current main still wires Zalo proactive target resolution to a numeric-only predicate, so the reported hex-like `chat_id` values can still fall through to `Unknown target` before reaching the Zalo send helper that accepts string chat IDs.

## What This Changes

The issue requests that Zalo proactive outbound sends accept valid non-numeric Bot API `chat_id` values.

## Best Possible Solution

Add a Zalo-owned `looksLikeId` predicate that accepts real non-empty normalized Zalo `chat_id` strings, cover the reported hex-like IDs and prefixes, and leave the shared numeric helper unchanged.

## Reproduction Assessment

Yes. Source inspection against current main shows a reported hex-like target normalizes correctly, fails Zalo’s numeric-only predicate, and can return `unknownTargetError` before the string-accepting Zalo send helper is called.

## Solution Assessment

Yes. A Zalo-local target predicate is the narrowest maintainable fix; relaxing `isNumericTargetId` globally would change target semantics for unrelated channels.
