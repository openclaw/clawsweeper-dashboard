## Label Justifications

- `P2`: This is a normal-priority channel bug with limited blast radius: proactive Zalo sends can fail even though in-session replies still work.
- `impact:message-loss`: The issue blocks scheduled or scripted outbound Zalo messages from being delivered to valid chat IDs.

## Summary

Keep open: current main and latest release still use a numeric-only Zalo target predicate before proactive send dispatch, while the Zalo send layer already accepts string `chat_id` values and the reporter provided upstream-success examples for hex-like IDs.

## What This Changes

Allow Zalo proactive outbound sends to accept non-numeric Bot API `chat_id` strings instead of rejecting them during CLI target resolution.

## Best Possible Solution

Fix this in the Zalo plugin by relaxing the Zalo target ID predicate to accept valid nonblank Bot API chat IDs, then add focused coverage for hex-like CLI targets and update the Zalo delivery-target docs.

## Reproduction Assessment

Yes, from source inspection: a hex-like Zalo target reaches `resolveChannelTarget`, Zalo's `looksLikeId` returns false, no directory match is found, and the CLI throws the unknown-target error. I did not run a live Zalo credentialed send in this read-only review.

## Solution Assessment

Yes, the best fix is plugin-local target recognition, because `sendMessageZalo` already accepts string `chat_id` values and a core resolver change would be broader than necessary.
