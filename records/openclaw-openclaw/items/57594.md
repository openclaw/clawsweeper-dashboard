## Summary

Keep open: current main still uses a numeric-only Zalo target predicate for proactive sends, while the Zalo send/API layer accepts string chat_id values, and no duplicate issue or fix PR was found.

## What This Changes

Allow Zalo proactive sends to accept valid non-numeric Bot API chat_id targets instead of rejecting them as unknown targets.

## Best Possible Solution

Update the Zalo plugin’s targetResolver predicate to accept real non-empty normalized Zalo chat_id strings, add regression coverage for the reported hex-like IDs and prefixes, and leave the shared numeric helper unchanged.

## Reproduction Assessment

Yes. Source inspection shows a reported hex-like target normalizes correctly, fails Zalo’s numeric-only looksLikeId check, and can hit unknownTargetError before the Zalo send helper that accepts string chat_id values.

## Solution Assessment

Yes. A Zalo-owned looksLikeId predicate is the narrowest maintainable fix; relaxing isNumericTargetId globally would change target semantics for unrelated channels.
