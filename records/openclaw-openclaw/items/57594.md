## Summary

Keep this open: current main and the latest release still use the numeric-only Zalo target predicate, so the reported non-numeric chat IDs can still fall through to `Unknown target`; the only related fix PR was closed unmerged and is not a viable canonical fix path.

## What This Changes

The issue reports that Zalo proactive outbound sends reject valid non-numeric `chat_id` targets even though replies and direct Bot API sends can use them.

## Best Possible Solution

Land a plugin-local Zalo targetResolver predicate that evaluates normalized Zalo targets, accepts the reported non-numeric `chat_id` form while preserving numeric IDs, covers `zalo:` and `zl:` prefixes, and leaves the generic plugin SDK helper unchanged.

## Reproduction Assessment

Yes, at source level. Current main routes Zalo target checks through a numeric-only predicate before the unknown-target path, although I did not run a live Zalo CLI send in this read-only review.

## Solution Assessment

Yes. The best fix is narrow and plugin-local: update Zalo’s target predicate to accept the real `chat_id` shapes after plugin normalization, with focused tests for bare and provider-prefixed targets.
