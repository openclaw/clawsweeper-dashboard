## Summary

Keep open: current main and the latest release still route Zalo proactive targets through a numeric-only predicate, so valid hex-like Bot API chat_id values can be rejected before the Zalo send layer.

## What This Changes

The issue asks OpenClaw to accept valid non-numeric Zalo Bot API chat_id values for proactive outbound sends via `openclaw message send --channel zalo --target`.

## Best Possible Solution

Add a Zalo-local target predicate that accepts plausible non-empty Bot API chat_id strings after prefix stripping, keep the shared numeric predicate unchanged, and cover proactive Zalo target resolution with focused tests.

## Reproduction Assessment

Yes, as a high-confidence source-level path: a hex-like Zalo target normalizes, fails the numeric-only `looksLikeId` predicate, and can fall through to `unknownTargetError` before `sendMessageZalo` runs. I did not perform a live Zalo API send in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is Zalo-local target recognition for real Bot API chat_id strings; relaxing `isNumericTargetId` globally would change target semantics for unrelated channels.
