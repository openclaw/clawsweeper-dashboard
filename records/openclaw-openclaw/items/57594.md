## Summary

Keep open: current main and latest release v2026.5.19 still route Zalo proactive sends through a numeric-only target predicate, so the reported hex-like chat_id values can be rejected before the Zalo send layer receives them.

## What This Changes

The issue requests that Zalo proactive outbound sends accept valid non-numeric Bot API chat_id values instead of rejecting them as unknown targets.

## Best Possible Solution

Add a Zalo-local target predicate that accepts real non-empty Bot API chat_id strings after prefix normalization, keep `isNumericTargetId` unchanged for other channels, and cover resolver plus send behavior with focused tests.

## Reproduction Assessment

Yes, as a high-confidence source-level path: a hex-like Zalo target normalizes, fails the numeric-only `looksLikeId` predicate, and can fall through to `unknownTargetError` before `sendMessageZalo` runs. I did not perform a live Zalo API send in this read-only review.

## Solution Assessment

Yes. The maintainable fix is a Zalo-local target recognizer for real Bot API chat_id strings; relaxing the shared numeric predicate globally would change target semantics for unrelated channels.
