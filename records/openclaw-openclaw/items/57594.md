## Summary

Keep open: current main still wires Zalo proactive target recognition to a numeric-only predicate, so the reported hex-like Bot API chat IDs can still be rejected before the Zalo send layer receives them.

## What This Changes

The issue asks Zalo proactive outbound sends to accept valid non-numeric Bot API chat_id strings instead of treating them as unknown targets.

## Best Possible Solution

Fix the Zalo plugin target resolver to accept real string chat_id values while preserving the shared numeric helper for other channels, then cover CLI/cron proactive sends with regression tests and docs.

## Reproduction Assessment

Yes. Source inspection on current main shows the reported hex-like target normalizes, fails Zalo's numeric-only looksLikeId predicate, and can return unknownTargetError before the string-accepting send helper runs.

## Solution Assessment

Yes. A Zalo-local looksLikeId predicate for non-empty normalized Bot API chat_id strings is the narrow maintainable fix; relaxing isNumericTargetId globally would change target semantics for unrelated channels.
