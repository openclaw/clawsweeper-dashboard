## Summary

Keep this open: current main and the latest release still route Zalo proactive target validation through the numeric-only helper, and the only related fix PR at https://github.com/openclaw/openclaw/pull/90859 was closed unmerged.

## What This Changes

The issue reports that Zalo proactive outbound sends reject valid non-numeric hex-like `chat_id` targets before the message reaches the Zalo Bot API send path.

## Best Possible Solution

Land a plugin-local Zalo targetResolver predicate that accepts normalized numeric and hex-like `chat_id` values, covers bare and provider-prefixed targets, and leaves core/plugin SDK behavior unchanged.

## Reproduction Assessment

Yes, at source level. Current main routes Zalo target checks through a numeric-only predicate before the unknown-target path, although I did not run a live Zalo CLI send in this read-only review.

## Solution Assessment

Yes. The best fix is narrow and plugin-local: validate Zalo's real `chat_id` shape using the normalized target value so both bare and `zalo:`/`zl:` targets work.
