## What This Changes

The issue asks OpenClaw to let Zalo proactive outbound sends accept valid nonnumeric Bot API chat_id targets instead of rejecting them before send.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/93689 if maintainers accept its Zalo-local opaque chat_id behavior, then close this issue with merge and release provenance.

## Reproduction Assessment

Yes. Source inspection shows current main's Zalo target resolver uses isNumericTargetId, and that helper rejects the reported hex-like chat_id values before the resolver falls through to unknown-target handling.

## Solution Assessment

Yes. A plugin-local Zalo chat_id predicate is the right fix boundary because Zalo owns its target grammar and the shared numeric helper can remain unchanged for other channels.
