## Summary

Keep open. Current main still has the reported Zalo proactive-send mismatch: Zalo strips optional prefixes but still uses the shared numeric-only target predicate, so hex-like Bot Platform chat IDs can be rejected before the Zalo send helper runs.

## What This Changes

Accept non-numeric Zalo Bot Platform chat_id strings for proactive outbound sends via `openclaw message send --channel zalo --target <chat_id>`.

## Best Possible Solution

Fix this in the Zalo plugin by accepting real, non-empty Zalo Bot Platform chat ID strings after optional `zalo:`/`zl:` normalization, add focused regression coverage for a hex-like chat ID through target resolution or message send, and update target docs/examples. Do not relax the shared numeric helper globally because other channels may depend on that distinction.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call Zalo proactive send or `resolveChannelTarget` with a configured Zalo channel and `3becaa50ae12474c1e03`; current main treats it as non-ID-like, misses directory lookup, and returns the reported unknown-target error.

## Solution Assessment

Yes, a Zalo-plugin-owned target predicate is the narrowest maintainable fix. Changing `isNumericTargetId` globally would risk altering target resolution semantics for unrelated channels.
