- `impact:message-loss`: Valid Zalo outbound messages can be rejected before delivery when the chat_id is non-numeric.

## Summary

Keep open: current main and the latest release still use the shared numeric-only target predicate for Zalo proactive sends, so hex-like Bot API chat_id values can be rejected before the Zalo send layer, which already accepts string chat IDs.

## What This Changes

Allow Zalo proactive outbound sends to treat valid non-numeric Bot API chat_id strings as explicit delivery targets.

## Best Possible Solution

Add a Zalo-local target predicate that accepts plausible non-empty normalized Bot API chat_id strings after prefix stripping, keep the shared numeric predicate unchanged for other channels, and cover proactive Zalo target resolution with focused tests.

## Reproduction Assessment

Yes, as a high-confidence source-level path: a hex-like Zalo target normalizes, fails the numeric-only looksLikeId predicate, and can fall through to unknownTargetError before sendMessageZalo runs. I did not perform a live Zalo API send in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is Zalo-local target recognition for real Bot API chat_id strings; relaxing isNumericTargetId globally would change target semantics for unrelated channels.
