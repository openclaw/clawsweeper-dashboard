## What This Changes

The issue asks OpenClaw to let Zalo proactive outbound sends accept valid non-numeric Bot API `chat_id` strings instead of rejecting them as unknown targets.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/93689 if maintainers accept its Zalo-local hex chat-id heuristic, then close this issue with the merge and release provenance.

## Reproduction Assessment

Yes by source inspection. Current `main` normalizes Zalo targets but delegates direct-id detection to a numeric-only helper, so the reported hex-like chat ids cannot enter the direct id-like path before unknown-target handling.

## Solution Assessment

Yes. A plugin-local Zalo chat-id predicate is the right boundary because the Zalo plugin owns its target grammar and the generic numeric helper remains appropriate for other channels.
