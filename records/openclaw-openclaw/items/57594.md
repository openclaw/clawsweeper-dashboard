## Summary

Keep open: current main and the latest shipped tag still route Zalo proactive target recognition through a numeric-only predicate, so valid hex-like Bot API chat_id values can still be rejected before the Zalo send layer is reached.

## What This Changes

The issue requests that Zalo proactive outbound sends accept valid non-numeric Bot API chat_id strings instead of rejecting them as unknown targets.

## Best Possible Solution

Add a Zalo-local target predicate that accepts non-empty normalized Bot API chat_id strings, preserve stricter numeric behavior for unrelated channels, and cover proactive send resolution with focused Zalo/outbound tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the reported hex-like target normalizes, fails Zalo's numeric-only looksLikeId predicate, and can return unknownTargetError before the string-accepting send helper runs.

## Solution Assessment

Yes. A Zalo-local looksLikeId predicate for non-empty Bot API chat_id strings is the narrow maintainable fix; relaxing isNumericTargetId globally would change target semantics for unrelated channels.
