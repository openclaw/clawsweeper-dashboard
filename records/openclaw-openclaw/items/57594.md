## Summary

Keep open: current main and the latest shipped release still route Zalo proactive target recognition through the shared numeric-only predicate, so the reported hex-like Bot API chat_id values can be rejected before the Zalo send layer is reached.

## What This Changes

The issue requests that Zalo proactive outbound sends accept valid non-numeric Bot API chat_id strings instead of rejecting them as unknown targets.

## Best Possible Solution

Add a Zalo-local target predicate that accepts non-empty normalized Bot API chat_id strings, keep shared numeric heuristics unchanged for other channels, and cover proactive Zalo target resolution with focused tests.

## Reproduction Assessment

Yes, as a source-level reproduction path: a hex-like Zalo target normalizes successfully, fails the numeric-only looksLikeId predicate, and can fall through to unknownTargetError before sendMessageZalo runs. I did not perform a live Zalo API send in this read-only review.

## Solution Assessment

Yes. A Zalo-local predicate for non-empty Bot API chat_id strings is the narrow maintainable fix; relaxing isNumericTargetId globally would change target semantics for unrelated channels.
