## Summary

Keep open: current main still wires Zalo proactive-send target validation to a numeric-only predicate, while the Zalo send layer accepts string chat_id values, so the reported hex-like IDs can still be rejected before dispatch. No duplicate issue or fix PR was found.

## What This Changes

Accept valid non-numeric Zalo Bot Platform chat_id strings for proactive outbound sends through the Zalo channel.

## Best Possible Solution

Update the Zalo plugin’s own target predicate to accept real non-empty Zalo chat_id strings after prefix normalization, add focused regression coverage and docs/examples, and leave the shared numeric helper unchanged for other channels.

## Reproduction Assessment

Yes. Source inspection against current main shows a target like 3becaa50ae12474c1e03 fails Zalo’s numeric-only looksLikeId check and reaches the existing unknown-target path before the Zalo send helper can accept it.

## Solution Assessment

Yes. A Zalo-plugin-owned predicate is the narrowest maintainable fix; changing isNumericTargetId globally would alter target-resolution semantics for unrelated channels.
