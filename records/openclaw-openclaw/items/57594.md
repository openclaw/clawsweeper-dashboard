## What This Changes

The issue requests that Zalo proactive outbound sends accept valid non-numeric Zalo Bot Platform chat_id values instead of rejecting them before delivery.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/93689 if maintainers accept its Zalo-local string chat_id behavior, then close this issue with merge and release provenance.

## Reproduction Assessment

Yes. Current main normalizes Zalo targets but still delegates direct-id detection to the numeric-only helper, so reported hex-like chat_id values cannot enter the direct target path before unknown-target handling.

## Solution Assessment

Yes. A plugin-local Zalo chat_id predicate is the right fix boundary because Zalo owns its target grammar and the shared numeric helper can remain unchanged for other channels.
