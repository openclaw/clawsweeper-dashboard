## Summary

Keep open: current `main` and the latest release still have the reported hostless `isPortBusy` preflight, but this PR is not merge-ready because it collapses `checkPortInUse`'s `unknown` status to `false` and the supplied proof does not exercise the patched path.

## What This Changes

The PR exports `checkPortInUse` from `src/infra/ports-inspect.ts` and changes `isPortBusy` in `src/cli/ports.ts` to return whether that multi-address probe reports `busy`.

## Best Possible Solution

Land one canonical shared-helper fix that detects IPv4-only listeners, preserves indeterminate probe failures as an error path, adds focused force-path coverage, and includes redacted real listener proof.

## Reproduction Assessment

Yes at source level for the PR defect: `checkPortInUse` can return `unknown`, and this patch maps that to false so `forceFreePortAndWait` treats the port as free. I did not run a live reproduction because this review is read-only.

## Solution Assessment

No as submitted. Reusing `checkPortInUse` is the right maintainable direction, but the caller must preserve `unknown` as an error or fail-closed state instead of treating it as `free`.
