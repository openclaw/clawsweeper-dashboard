## Summary

Keep open: the PR targets a real gateway port-cleanup bug, but the patch still collapses an indeterminate multi-host probe into the same false result as a free port, and the supplied terminal proof does not actually exercise the patched `isPortBusy` or `gateway --force` path.

## What This Changes

The PR exports `checkPortInUse` and changes `isPortBusy` to use that shared multi-address port probe for the `gateway --force` busy check.

## Best Possible Solution

Land one canonical shared-helper fix that detects IPv4-only listeners, preserves indeterminate probe failures as an error path, adds focused force-path coverage, and includes redacted real listener proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level for the PR defect: `checkPortInUse` can return `unknown`, and this patch maps that to false so `forceFreePortAndWait` treats the port as free.

## Solution Assessment

Is this the best way to solve the issue? No as submitted. Reusing `checkPortInUse` is the right maintainable direction, but the caller must preserve `unknown` as an error or fail-closed state instead of treating it as `free`.
