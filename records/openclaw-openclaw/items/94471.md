## Summary

Keep open: the PR targets a valid `gateway --force` port-detection bug, but as submitted it collapses `checkPortInUse`'s `unknown` result into `false` and can make the force path skip cleanup when probing is indeterminate. ## What This Changes

The PR exports `checkPortInUse` from `src/infra/ports-inspect.ts` and changes `src/cli/ports.ts` so `isPortBusy` delegates to that multi-address port probe.

## Best Possible Solution

Land one canonical shared-helper fix that detects IPv4-only listeners, preserves indeterminate probe failures as an error path, adds focused force-path coverage, and includes consistent redacted real listener proof.

## Reproduction Assessment

Yes at source level: force `checkPortInUse` to return `unknown` and the patched `isPortBusy` returns `false`, causing `forceFreePortAndWait` to skip cleanup. I did not run a live reproduction because this review is read-only.

## Solution Assessment

No as submitted. Reusing `checkPortInUse` is the right maintainable direction, but the caller must preserve `unknown` as an error or fail-closed state instead of treating it as `free`.
