## What This Changes

The PR exports `checkPortInUse` and changes CLI `isPortBusy` to call that shared multi-address-family probe, throwing when the probe result is `unknown`.

## Best Possible Solution

Land one canonical `isPortBusy` fix that reuses the shared multi-family probe, preserves `unknown` as fail-closed, updates force-path coverage for multi-host probes, and includes changed-path real listener proof.

## Reproduction Assessment

Yes for the review blocker by source inspection: the PR makes each `isPortBusy` check call `checkPortInUse`, which can consume up to four `tryListenOnPort` results while the force-path test still queues one free result for a full status check. I did not run a live IPv4-only listener reproduction in this read-only review.

## Solution Assessment

No as submitted. Reusing `checkPortInUse` and throwing on `unknown` is the right maintainable direction, but the PR still needs force-path test updates and changed-path real proof before it is the best merge candidate.
