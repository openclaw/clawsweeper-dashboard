## Summary

Keep this PR open: the convergence direction is sound, but latest head still has a blocking validation gap where the new smoke check can accept escaping extension entries that the runtime loader rejects, so it can still restart with an unloadable plugin. The current head is also merge-conflicting and needs current-head real behavior proof for the latest restart/validation changes.

## What This Changes

This PR adds a mandatory post-core plugin convergence step to `openclaw update`, static plugin payload validation, doctor phase helpers, tests, docs, and changelog coverage.

## Best Possible Solution

Keep the post-core convergence design, but harden the static payload validator to reuse or mirror the runtime package-entry boundary checks, add regression coverage for escaping `openclaw.extensions` entries, resolve conflicts, and refresh current-head terminal proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main skips invalid configs and lacks manifest-entry validation, and the PR body provides before/after terminal output for those paths; the new PR regression is also source-reproducible by comparing the smoke check's `path.join` with runtime package-boundary validation.

## Solution Assessment

No, not yet. The convergence approach is the right narrow fix, but the current smoke check must enforce the same package-root boundary as runtime loading before it can reliably decide whether restart is safe.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Keep smoke-checked extension entries inside the package:** `src/cli/update-cli/plugin-payload-validation.ts:105-107`
