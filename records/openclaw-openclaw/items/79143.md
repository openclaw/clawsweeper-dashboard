## Summary

Keep open: the PR targets a real update/plugin repair gap, but the draft patch has blocking correctness issues in the new smoke check and lacks required real behavior proof.

## What This Changes

The PR adds post-core update phase helpers, a plugin convergence orchestrator and static payload smoke check, update-command wiring, and related doctor/update tests to block gateway restart when plugin repair or validation fails.

## Best Possible Solution

Revise the PR so convergence reuses existing OpenClaw plugin entrypoint and enable-state logic, then add a changelog entry and redacted proof from an actual update run before maintainer review.

## Reproduction Assessment

Yes, from source inspection: a configured package-tracked plugin with a missing payload can reach post-core update handling, produce warning/error outcomes, and still avoid the `status: "error"` gate on current main. I did not run a live package update scenario.

## Solution Assessment

No. Mandatory post-core convergence is a reasonable boundary, but this implementation validates the wrong package entrypoint contract and should be narrowed to configured/enabled payloads using existing helpers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate OpenClaw plugin entrypoints:** `src/cli/update-cli/plugin-payload-validation.ts:83-85`
  - body: OpenClaw plugin packages are installed and loaded through `package.json` `openclaw.extensions`, not `package.json` `main`. A valid package with only `openclaw: { extensions: ["./dist/index.js"] }` and no `main`/root `index.js` will be reported as `missing-main-entry`, causing `openclaw update` to fail before restart even though the plugin is structurally valid. Reuse the existing package entrypoint resolution/validation helpers instead of checking `main`.
  - confidence: 0.91
- **[P2] Scope convergence checks to active payloads:** `src/cli/update-cli/post-core-plugin-convergence.ts:48-52`
  - body: This reloads every tracked install record and smoke-checks all of them. Existing post-core payload repair passes config with `skipDisabledPlugins`, so disabled non-official records do not block an update; with this change, a stale disabled npm/git/marketplace record missing `package.json` makes the update exit before restart even though the gateway would not load that plugin. Filter through the same enabled/configured logic or downgrade unrelated records.
  - confidence: 0.83
- **[P3] Add the required changelog entry:** `src/cli/update-cli/update-command.ts:1292-1306`
  - body: This is a user-facing update/doctor behavior fix, but the PR does not touch `CHANGELOG.md`. Repo policy requires user-facing fix/feat/perf changes to add an Unreleased changelog entry.
  - confidence: 0.78
