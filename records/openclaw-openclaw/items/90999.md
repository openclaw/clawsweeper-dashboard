## Label Justifications

- `P2`: This is a deterministic Gateway race with bounded but real client-visible impact on aborted `chat.send` runs.
- `impact:session-state`: The `chat:<runId>` dedupe cache can store the wrong terminal state for same-run retries and waiters.
- `impact:message-loss`: A late error broadcast can replace or contradict the correct terminal abort state clients should observe.

## Summary

Keep open: current main and the latest release still have the post-dispatch `chat.send` rejection path that can publish an error after an abort, and the linked fix remains an open PR rather than merged.

## What This Changes

The issue requests preserving the terminal `chat.abort` result when a later same-run `chat.send` dispatch rejection arrives, while keeping dispatch-error-first ordering terminal.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/91000, or replace it with an equivalent Gateway fix that preserves abort-first and error-first terminal precedence.

## Reproduction Assessment

Yes. Source inspection shows the abort state is recorded, then the post-dispatch rejection handler can still write an error terminal; the reporter also provided a targeted Vitest repro, though this read-only review did not run it.

## Solution Assessment

Yes. A focused fix in the `chat.send` post-dispatch catch path with regressions for abort-first and error-first ordering is the narrow maintainable solution; closing before the linked PR lands would be premature.
