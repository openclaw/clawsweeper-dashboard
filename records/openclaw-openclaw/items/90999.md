## Label Justifications

- `P2`: This is a deterministic Gateway race with bounded but real client-visible impact on aborted `chat.send` runs.
- `impact:session-state`: The `chat:<runId>` dedupe cache can store the wrong terminal state for same-run retries.
- `impact:message-loss`: A late error broadcast can replace or contradict the correct terminal aborted chat state.

## Summary

Keep open: current main and the latest release still have a post-dispatch `chat.send` rejection path that can overwrite an already-recorded abort with an error result, and the linked fix PR is still open.

## What This Changes

Preserve terminal `chat.abort` outcomes when a same-run `chat.send` dispatch rejects after abort, while keeping error-first late-abort ordering terminal as an error.

## Best Possible Solution

Review and land the linked Gateway race fix, or replace it with an equivalent focused fix that preserves both abort-first and error-first terminal precedence.

## Reproduction Assessment

Yes. Source inspection on current main shows abort records `chatAbortedRuns`, but the post-dispatch catch can still overwrite `chat:<runId>` as an error; the reporter also supplied a deterministic focused Vitest repro, though I did not run it in this read-only review.

## Solution Assessment

Yes. A focused Gateway fix in the post-dispatch `chat.send` catch path with regressions for both race orderings is the narrow maintainable solution.
