## Summary

Keep open: current main and the latest release still have the post-dispatch `chat.send` rejection path that can replace an already-recorded abort with an error result, and the linked fixing PR is open rather than merged.

## What This Changes

The issue reports that a late `chat.send` dispatch rejection can overwrite an already-terminal `chat.abort` result with an error status and late error broadcast.

## Best Possible Solution

Land the linked Gateway race fix, or replace it with an equivalent fix that preserves both abort-first and error-first terminal ordering in the `chat.send` post-dispatch path.

## Reproduction Assessment

Yes. Source inspection shows the abort tombstone can be recorded before the later post-dispatch rejection catch writes an error result; the issue also includes a deterministic targeted Vitest repro, though this read-only review did not run tests.

## Solution Assessment

Yes. A focused fix in the `chat.send` post-dispatch error path plus regressions for abort-first and error-first orderings is the narrow maintainable solution.
