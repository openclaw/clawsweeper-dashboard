## Label Justifications

- `P2`: This is a deterministic Gateway race with real but bounded client-visible impact on aborted `chat.send` runs.
- `impact:message-loss`: A late error broadcast can replace the correct terminal abort message clients should observe.
- `impact:session-state`: The `chat:<runId>` dedupe cache can store the wrong terminal state for same-run retries.

## Summary

Keep open: current main still has the post-dispatch `chat.send` catch path that can overwrite an already-recorded abort with an error result, and the linked fixing PR is open rather than merged.

## What This Changes

Reports a Gateway race where a post-dispatch `chat.send` rejection can replace an already-terminal `chat.abort` result with a cached error.

## Best Possible Solution

Review and land the linked Gateway race fix, or replace it with an equivalent fix that preserves abort-first and error-first terminal precedence in the `chat.send` post-dispatch path.

## Reproduction Assessment

Yes. Source inspection shows the abort tombstone is recorded, then the post-dispatch rejection handler can still unconditionally write an error result; the reporter also supplied a deterministic targeted Vitest repro, though I did not run it in this read-only review.

## Solution Assessment

Yes. A focused fix in the `chat.send` post-dispatch catch path plus regressions for both abort-first and error-first orderings is the narrow maintainable solution.
