## Summary

Keep open: current main still has the post-dispatch `chat.send` catch path that can overwrite an already-terminal abort with an error, and https://github.com/openclaw/openclaw/pull/91000 is the open closing PR that owns the focused fix.

## What This Changes

The issue reports a Gateway race where a post-dispatch `chat.send` rejection can overwrite an already-terminal abort result.

## Best Possible Solution

Land the targeted Gateway fix from https://github.com/openclaw/openclaw/pull/91000, or an equivalent patch, so abort-first keeps the terminal abort and error-first keeps the terminal error without stale abort replay.

## Reproduction Assessment

Yes, at source level. Current main records abort state and broadcasts abort, but the later post-dispatch catch path can still write an error dedupe entry and error broadcast; the reporter also supplied a deterministic targeted Vitest repro, though I did not run tests in this read-only review.

## Solution Assessment

Yes. A focused `chat.send` post-dispatch catch fix with regressions for both abort-first and error-first ordering is the narrow maintainable solution, and the open linked PR follows that shape.
