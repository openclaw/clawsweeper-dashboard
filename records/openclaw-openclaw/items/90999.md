## Label Justifications

- `P2`: This is a deterministic Gateway race with bounded but real client-visible impact on aborted `chat.send` runs.
- `impact:session-state`: The `chat:<runId>` dedupe cache can store the wrong terminal state for same-run retries and waiters.
- `impact:message-loss`: A late error broadcast can replace the abort terminal message clients should observe for the run.

## Summary

Keep open: current main and the latest tagged release still have a post-dispatch `chat.send` catch path that can overwrite terminal abort state with an error, and the linked fix at https://github.com/openclaw/openclaw/pull/91000 is still open rather than merged.

## What This Changes

The issue reports a Gateway race where a `chat.send` dispatch rejection after `chat.abort` can replace the cached terminal abort outcome with `status:"error"`.

## Best Possible Solution

Review and land the linked Gateway race fix, or replace it with an equivalent focused fix that preserves abort-first and error-first terminal precedence in `chat.send`.

## Reproduction Assessment

Yes. Source inspection shows the abort marker can be recorded, then the post-dispatch rejection handler can still write an error result; I did not run the supplied Vitest repro because this review is read-only.

## Solution Assessment

Yes. A focused fix in the `chat.send` post-dispatch rejection path with regressions for both abort-first and error-first orderings is the narrow maintainable solution.
