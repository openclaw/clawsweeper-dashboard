## Label Justifications

- `P2`: Gateway abort/result ordering is a normal-priority correctness issue with limited blast radius.
- `impact:session-state`: The `chat:<runId>` dedupe cache can store the wrong terminal state for same-run retries and waiters.
- `impact:message-loss`: A late error broadcast can replace the correct terminal abort state that clients should observe.

## Summary

Keep open: current `main` still has the post-dispatch `chat.send` catch path that can overwrite an abort terminal result, and the linked fix PR is open rather than merged.

## What This Changes

The issue reports a Gateway race where a `chat.send` dispatch rejection can overwrite an already-terminal `chat.abort` result.

## Best Possible Solution

Review and land the linked Gateway fix, or replace it with an equivalent focused fix that preserves abort-first and error-first terminal precedence.

## Reproduction Assessment

Yes. Source inspection shows `chat.abort` records the abort tombstone, while the later post-dispatch `chat.send` rejection handler can still write an error result; I did not run the reporter's regression because this review is read-only.

## Solution Assessment

Yes. A focused fix in the Gateway `chat.send` post-dispatch rejection path plus regressions for both abort-first and error-first orderings is the narrow maintainable solution.
