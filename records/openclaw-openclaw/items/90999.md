## Label Justifications

- `P2`: This is a deterministic Gateway race with bounded but real client-visible impact on aborted `chat.send` runs.
- `impact:message-loss`: A late dispatch error can produce an incorrect error chat broadcast after the terminal abort state was already emitted.
- `impact:session-state`: The `chat:<runId>` dedupe cache can store the wrong terminal status for same-run retry and waiter observations.

## Summary

Keep open: current main and the latest release still have the unguarded `chat.send` post-dispatch rejection path, and https://github.com/openclaw/openclaw/pull/91000 is the open closing PR that owns the fix path.

## What This Changes

The issue asks OpenClaw to preserve terminal `chat.abort` state when a later same-run `chat.send` dispatch rejection arrives, while keeping error-first ordering terminal as an error.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/91000 after review, or merge an equivalent Gateway fix that preserves abort-first and error-first terminal precedence, then close this issue.

## Reproduction Assessment

Yes at source level. Current main records abort state and has a pre-dispatch retry guard, but the post-dispatch rejection catch still writes an error dedupe and broadcasts an error; the reporter also supplied a deterministic targeted Vitest repro that was not run in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is in the `chat.send` post-dispatch error path, with regressions for abort-first and error-first orderings; the linked PR is the active implementation path.
