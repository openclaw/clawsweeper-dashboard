## Summary

This should stay open because current main has ordinary Telegram access/routing for bot-authored updates but not the explicit sender policy, loop guard, peer delegation primitive, or doctor diagnostics requested here.

## What This Changes

Request explicit Telegram bot-sender policy, bot-to-bot loop controls, inter-agent delegation routing, and doctor diagnostics for Telegram's May 7 2026 guest/bot-to-bot features.

## Best Possible Solution

Design this as a Telegram-owned feature: add an explicit safe bot-sender policy and loop protection first, then add peer/delegation routing and doctor diagnostics once the public config and upstream Bot API limits are clear.

## Reproduction Assessment

Yes, source inspection gives a high-confidence code path for the reported gap: current main skips only self-authored Telegram messages and otherwise sends allowed bot-sourced messages through normal Telegram authorization. I did not run a live BotFather bot-to-bot setup, so the live platform path is source-reproducible rather than locally reproduced.

## Solution Assessment

No, this is not a safe small bug-fix lane as written because it creates new public config and routing semantics. The maintainable path is a maintainer-reviewed Telegram design that starts with explicit bot-sender gating and rate limits before adding inter-agent delegation behavior.
