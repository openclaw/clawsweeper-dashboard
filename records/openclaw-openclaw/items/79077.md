## Summary

Keep open: current main still lacks the requested Telegram sender-type policy, peer/delegation routing, per-sender loop protection, guest-bot handling, and doctor diagnostics. Current code can process bot-authored Telegram messages through existing allowlist/group routing, so this is a real Telegram feature and safety design item rather than a stale cleanup candidate.

## What This Changes

The issue requests Telegram bot-to-bot and guest-bot support with explicit sender policy, rate limits, peer delegation, and doctor diagnostics.

## Best Possible Solution

Design an opt-in Telegram-owned bot-sender policy and bot-pair loop guard first, then add peer/delegation routing and guest-bot doctor/runtime support once the public config and upstream API limits are confirmed.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes as a source-level feature gap: current main exposes Telegram sender IDs and standard allowlist routing, but no Telegram bot-sender policy, per-sender rate limit, peer/delegation config, guest-mode handler, or doctor toggle diagnostics.

## Solution Assessment

Is this the best way to solve the issue? Not as a narrow automated patch. The direction is plausible, but the maintainable path needs maintainer-reviewed Telegram API/config design that starts with safe intake and loop protection before delegation behavior.
