## Summary

Keep open: current main and v2026.5.12 still lack the requested Telegram bot-sender policy, bot-pair loop facts, peer/delegation routing, Guest Bot handling, and doctor diagnostics. The comments and upstream docs support the use case, but this is new Telegram product/config design rather than conservative cleanup.

## What This Changes

Request adds Telegram bot-to-bot and guest-bot support with sender-type policy, peer mappings, rate limits, loop guards, inter-agent delegation, and doctor diagnostics.

## Best Possible Solution

Design an opt-in Telegram bot-sender policy first, wire accepted bot updates into the shared bot-pair loop guard, then add peer/delegation routing and doctor warnings behind documented config.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a failing bug repro, but yes as a source-level feature gap: Telegram exposes sender is_bot data, while current OpenClaw only self-filters the receiving bot and lacks the requested policy, loop, peer, and doctor surfaces.

## Solution Assessment

Is this the best way to solve the issue? Not as a single small fix lane; the maintainable path is a Telegram-owned design that starts with safe bot-sender gating and loop protection before adding delegation semantics.
