## Summary

Keep open: current main partially recognizes allowlisted Telegram bot-authored updates, but it still lacks the requested explicit bot-sender policy, peer/delegation routing, per-sender loop guard, and doctor diagnostics. The related Telegram allowBots and multi-account routing issues now point back to this issue as the canonical thread rather than superseding it.

## What This Changes

Request support for Telegram Guest Mode and Bot-to-Bot Communication with explicit Telegram bot-sender policy, peer mapping, inbound rate limits, delegation routing, and doctor diagnostics.

## Best Possible Solution

Design this as Telegram-owned public config and runtime behavior: safe bot-sender intake and loop protection first, then peer/delegation routing and doctor diagnostics once the Bot API surface is confirmed.

## Reproduction Assessment

Yes for the source-level gap: current main can accept allowlisted bot-authored Telegram updates through the normal path, and source inspection shows no explicit peer policy, limiter, or delegation branch. I did not run a live two-bot BotFather setup in this read-only review.

## Solution Assessment

Unclear as written: the requested direction is valid, but the exact config and routing contract needs maintainer design. The safer first step is explicit bot-sender authorization and loop protection before adding inter-agent delegation semantics.
