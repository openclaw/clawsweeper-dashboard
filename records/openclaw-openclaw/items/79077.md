## Summary

Keep open. Current main has basic Telegram bot-authored update handling and shared bot-loop infrastructure for other channels, but Telegram still lacks the requested sender-type policy, peer/delegation routing, per-sender loop guard wiring, Guest Bot diagnostics, and doctor checks.

## What This Changes

Requests Telegram support for Guest Bot and Bot-to-Bot modes with explicit bot-sender policy, per-sender rate limits, peer delegation routing, and doctor diagnostics.

## Best Possible Solution

Keep this as the canonical Telegram design thread: first define safe bot-sender intake and loop protection, then add peer/delegation routing and doctor diagnostics once the config and upstream API contract are settled.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the reported gap: Telegram currently skips only self-authored messages and otherwise uses normal sender allowlists without peer-bot policy or Telegram loop facts. I did not run a live BotFather two-bot or Guest Bot setup.

## Solution Assessment

No, this is not a small bug-fix lane as written. The maintainable solution needs a maintainer-approved Telegram product/API design for sender policy, loop limits, peer trust, session semantics, and doctor diagnostics.
