## Summary

Keep open: current main still lacks Telegram-specific bot sender policy, peer/delegation routing, per-sender loop protection, Guest Bot handling, and doctor diagnostics. The discussion and official Telegram docs make the use case real, but this is new Telegram plugin/API design rather than conservative cleanup.

## What This Changes

Add Telegram Guest Bot and bot-to-bot support with explicit sender policy, peer routing, rate limits, loop guards, and doctor diagnostics.

## Best Possible Solution

Design an opt-in Telegram-owned bot-sender policy and loop guard first, then add peer/delegation routing and Guest Bot diagnostics once the public config contract and upstream API limits are confirmed.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a broken-behavior bug, but yes as a source-level feature gap: current main has sender IDs and normal Telegram routing, while the requested bot-sender policy, peer routing, rate-limit, loop-protection, Guest Bot, and doctor surfaces are absent.

## Solution Assessment

Is this the best way to solve the issue? The direction is reasonable, but not as a small automated fix; the maintainable path is a maintainer-reviewed Telegram config/API design with safe intake and loop controls before delegation behavior.
