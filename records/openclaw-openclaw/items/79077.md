## Summary

Keep open. Current main still lacks the requested Telegram sender-type policy, peer/delegation routing, inbound rate or circuit breaker, and doctor diagnostics; the related Telegram allowBots/routing reports were closed as superseded to this canonical issue, while the remaining Discord loop issue is separate.

## What This Changes

Add Telegram support for bot-originated and guest-bot updates with explicit account sender policy, peer delegation routing, rate limits/loop guards, and doctor diagnostics.

## Best Possible Solution

Design this as an opt-in Telegram feature: fail closed for bot senders by default, add explicit peer and rate-limit policy, then layer delegation context and doctor diagnostics once the public config contract is clear.

## Reproduction Assessment

Yes for the source-level gap: current main only self-skips Telegram bot-authored messages before normal allowlist/group handling, and the requested config fields are absent from the type/schema. I did not run a live two-bot BotFather setup in this read-only review.

## Solution Assessment

No as a small repair lane; the requested direction is useful but it defines new public Telegram policy, routing, and safety semantics. The safer path is maintainer-reviewed design that ships bot-sender gating and loop protection before broad delegation behavior.
