## Summary

Keep open. Current main still lacks Telegram-specific bot/guest sender policy, peer/delegation routing, guest-message handling, Telegram loop-guard wiring, and doctor diagnostics; the related older Telegram bot-to-bot issues now point back here rather than superseding it.

## What This Changes

The issue requests support for Telegram Guest Bots and Bot-to-Bot Communication with explicit sender policy, per-sender limits, peer delegation routing, and doctor diagnostics.

## Best Possible Solution

Design this as a staged Telegram feature: fail-closed bot/guest intake and peer authorization first, map accepted bot pairs into loop protection, then add delegation/session semantics and doctor checks where Telegram exposes reliable API signals.

## Reproduction Assessment

Yes for the source-level gap: current main can be inspected to show non-self bot-authored Telegram messages flow through normal sender authorization, while the config/schema has no requested bot-sender policy or guest surfaces. I did not run a live two-bot BotFather setup in this read-only review.

## Solution Assessment

Unclear as an exact implementation, because the issue is a good product problem statement but its config, routing, guest-mode, and doctor details need maintainer design against Telegram's current Bot API. The safer path is staged intake and loop safety before delegation semantics.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because accepting bot-authored Telegram traffic changes authorization, delegation, and loop-risk boundaries.

- **[medium] Define peer-bot authorization before accepting bot senders:** `extensions/telegram/src/bot-handlers.runtime.ts:2651`
  - body: Current Telegram handling authorizes by sender id/username and only skips the same bot id, so adding bot-to-bot support without an explicit peer trust model could let one bot drive another account or agent through ordinary allowlists.
  - confidence: 0.88
