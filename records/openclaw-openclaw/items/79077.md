## Summary

Keep open. Current main handles basic Telegram bot-authored group updates, but it does not implement the requested sender-type policy, peer/delegation model, inbound rate or loop guard, guest-bot handling, or doctor diagnostics, so this remains valid feature and safety design work.

## What This Changes

The issue requests Telegram support for May 7 2026 bot-to-bot and guest-bot modes with explicit sender policy, peer routing, rate limiting, loop prevention, and doctor diagnostics.

## Best Possible Solution

Define an opt-in Telegram-owned policy that defaults to human senders, explicitly allows trusted peer bots, rate-limits bot senders before dispatch, preserves agent/account identity during delegation, adds guest-bot update handling if supported by the Bot API, and documents/validates the behavior through doctor and live QA.

## Reproduction Assessment

Yes for the source-level gap: current main only skips Telegram self-echoes and otherwise sends bot-sourced updates through the normal human-style inbound path, while the requested policy/config keys are absent. A live Telegram May 7 bot-to-bot or guest-bot setup was not run in this read-only review.

## Solution Assessment

Unclear. The requested primitives are directionally reasonable, but the best maintainable shape needs maintainer decisions on config names, peer trust semantics, BotFather toggle detection, and whether any generic channel loop-guard seam should be shared.
