## Summary

Keep open: current main and the latest shipped release still lack Telegram-specific bot sender policy, per-sender loop protection, peer/delegation routing, Guest Bot handling, and doctor diagnostics. The related discussion strengthens the request, but it also shows this needs maintainer API/product design rather than conservative cleanup.

## What This Changes

Request support for Telegram's new Guest Bot and Bot-to-Bot modes with explicit sender policy, inbound rate limits, peer delegation, and doctor diagnostics.

## Best Possible Solution

Design an opt-in Telegram-owned bot-sender policy and loop guard first, then add peer/delegation routing and doctor diagnostics once the public config and upstream API limits are confirmed.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug repro, but yes as a source-level feature gap: current main has the Telegram sender data and normal allowlist routing, while the requested policy, rate-limit, peer, and doctor surfaces are absent.

## Solution Assessment

Is this the best way to solve the issue? Not as a small fix lane. The maintainable path is a maintainer-reviewed Telegram design that starts with safe bot-sender gating and loop protection before adding delegation behavior.
