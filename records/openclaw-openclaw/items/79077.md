## Summary

Keep open: current main has partial Telegram bot-to-bot support for mentioned group replies and channel posts, but it still lacks the requested Guest Mode handling, explicit bot-sender policy, peer/delegation routing, per-sender loop guard, and doctor diagnostics. Related Telegram issues were closed toward this issue as the canonical thread, and the open paired report about unaudited Telegram allowlist drift reinforces that this needs maintainer design rather than cleanup closure.

## What This Changes

Request support for Telegram bot-to-bot and guest-bot modes with sender-type policy, peer mapping, rate limiting, delegation routing, and doctor diagnostics.

## Best Possible Solution

Design a Telegram-owned opt-in bot-sender and peer-delegation contract with loop protection and doctor checks, then cover it with focused unit tests plus the existing live Telegram QA lane.

## Reproduction Assessment

Yes, source-level: the current Telegram handler forwards non-self bot-authored messages through the normal sender allowlist path and the config/schema lack the requested policy and rate-limit fields. I did not run a live BotFather Guest Mode or two-bot setup in this read-only review.

## Solution Assessment

No, not as a small repair: the requested direction is plausible, but it defines new public config, authorization, routing, loop-safety, and doctor semantics. The maintainable path is maintainer-reviewed design before implementation.
