## Summary

Keep open: current main still buffers cron due-job outcomes until the whole worker batch finishes, while this active PR contains the focused incremental persistence fix, regression coverage, sufficient real behavior proof, and green latest-head checks. The remaining blocker is merge handling because GitHub currently reports the branch as conflicting against current main, not a correctness issue in the proposed cron fix.

## What This Changes

The PR moves cron due-job outcome persistence from the end of a due batch into a per-completed-job locked flush, preserves current-batch running markers during maintenance recompute, and adds focused regression coverage plus small docs/gateway test updates.

## Best Possible Solution

Resolve the branch conflicts while keeping the cron change limited to immediate due-batch completion accounting, then land the focused fix with its regression coverage after normal maintainer gates.

## Reproduction Assessment

Yes. Current-main source inspection shows due jobs are persisted as running, worker results are buffered, and outcomes are not applied until Promise.all(workers) completes; the PR adds a direct regression scenario with a completed deleteAfterRun job while a later same-batch job blocks.

## Solution Assessment

Yes. The proposed fix reuses the existing locked force-reload, applyOutcomeToStoredJob, maintenance recompute, and persist path, moving only the application point earlier and preserving same-batch running markers during that recompute.
