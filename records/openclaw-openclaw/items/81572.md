Keep open: current main and v2026.6.8 still buffer ordinary due-job outcomes until the worker batch drains, while this PR is a useful, proven candidate fix that now needs maintainer rebase and review against newer cron state-machine changes.

## What This Changes

The PR moves normal cron due-job outcome persistence from an end-of-batch flush to per-completed-job application under the cron lock, adds cron regression coverage, and adjusts two gateway sessions_send timing assertions.

## Best Possible Solution

Rebase the incremental due-job finalization onto current main, preserve setup-timeout, active-marker, reservation-release, and SQLite cron invariants, then land or replace it as the focused candidate fix for the canonical cron issue.

## Reproduction Assessment

Yes, at source level. Current main reserves due jobs, buffers ordinary outcomes, waits for Promise.all(workers), and only then finalizes completedResults, so an earlier completed deleteAfterRun job can remain persisted as running while later same-batch work blocks.

## Solution Assessment

Yes, with a rebase caveat. Applying each completed outcome under the existing cron lock is the right owner-boundary fix, but the final patch must be resolved against current main's setup-timeout and persisted-state paths.
