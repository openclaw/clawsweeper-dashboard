The PR moves cron due-job outcome persistence from end-of-batch to per-completed job, adds regression tests for one-shot cron finalization, and adjusts a gateway sessions_send test for delayed calls.

## Best Possible Solution

Rebase the incremental due-job finalization through current `finalizeCompletedResults` flow, preserve the newer cleanup invariants, and land it with refreshed focused cron validation.

## Reproduction Assessment

Yes. Current `main` still buffers ordinary due-job outcomes until all workers finish, so a two-job due batch where the first deleteAfterRun job completes and the second blocks can leave persisted state stale until batch drain.

## Solution Assessment

Yes directionally, but not as the current branch. The maintainable fix is to move ordinary completed outcomes through the existing locked finalization path per result, not to land this old-base diff without rebasing current cleanup behavior.
