## Summary

Keep open. Current main still explicitly filters the cron reaper to run-scoped keys, current tests assert that base cron rows remain, and #76545 is an open closing PR rather than a merged fix.

## What This Changes

The issue asks OpenClaw to stop leaving stable cron base session rows unswept when cron retention prunes old isolated cron session state.

## Best Possible Solution

Keep this issue open while #76545 is reviewed, and land a fix that explicitly defines cron base-row and boot-row lifecycle, preserves intended metadata, prunes transcript-heavy stale state, and updates docs/tests together.

## Reproduction Assessment

Yes for the key-selection behavior: current source and tests show the reaper skips base `agent:...:cron:<jobId>` rows and only prunes `:run:` rows. The production CPU/RAM growth and the later `boot-*` race were not live-reproduced here.

## Solution Assessment

Unclear. #76545's filter change is a plausible implementation candidate, but the safer maintainable fix should first encode what stable base rows are allowed to retain and whether boot-session cleanup belongs in the same path.
