## Summary

Keep open. Current main still sweeps only cron `:run:` session keys and has tests that preserve the stable base cron row, but changing that row's lifecycle needs maintainer judgment because it also carries selected cron preferences and current docs describe the retention setting as run-session retention.

## What This Changes

The issue asks for cron session retention to include stable base cron session rows instead of only `:run:` sub-session keys.

## Best Possible Solution

Define the stable cron base-row lifecycle, then prune or sanitize transcript-heavy completed-run state while preserving intentional preferences, with docs and regression tests updated together.

## Reproduction Assessment

Yes for the key-selection behavior: source and tests show the reaper skips base cron keys and prunes only `...:run:` keys. The reported production CPU/RAM blowup was not live-reproduced here.

## Solution Assessment

Unclear. The proposed one-line helper swap is not clearly the best fix because stable base rows also carry preserved preferences; the safer path is to decide and encode the base-row lifecycle explicitly.
