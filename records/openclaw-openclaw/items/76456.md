## Summary

Keep open. Current main still filters the cron reaper with `isCronRunSessionKey`, and existing tests explicitly keep the base `agent:...:cron:<jobId>` entry, so the reported key-selection behavior is real; however, a one-line switch to all cron keys needs maintainer judgment because base entries also preserve cron-session preferences and current docs describe `cron.sessionRetention` as run-session retention.

## What This Changes

The issue asks for `cron.sessionRetention` to sweep base `agent:...:cron:<jobId>` cron sessions as well as `...:run:<runId>` sub-sessions so isolated cron state cannot remain unpruned.

## Best Possible Solution

Define the intended lifecycle for stable cron base rows, then implement pruning or sanitization that removes stale transcript-heavy state without losing the current run or preserved user preferences, with docs and regression tests updated together.

## Reproduction Assessment

Yes for the key-selection behavior: current source and tests show the reaper skips base cron keys and prunes only `...:run:` keys. The reported production token/CPU blowup was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. The proposed one-line filter swap is probably too broad because base cron rows are also used as stable metadata/preference carriers; a safer fix needs an explicit base-session lifecycle decision first.
