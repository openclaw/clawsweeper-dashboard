## Summary

Keep open. Current main still sweeps only cron `:run:` keys, and an open unmerged PR already references this issue with closing syntax, so the issue should stay open until that PR lands or maintainers choose a different lifecycle fix.

## What This Changes

The issue reports that cron retention pruning skips stable base cron session keys and only removes `:run:` cron sub-session keys.

## Best Possible Solution

Review PR #76545 against the intended base-session lifecycle, add regression coverage for transcript-heavy base rows and any separate boot orphan path, then land or replace it with docs kept in sync.

## Reproduction Assessment

Yes for the cron key-selection behavior: a store fixture with an expired `agent:main:cron:<jobId>` base key will not be touched because current main filters with `isCronRunSessionKey`. I did not live-reproduce the reported CPU/RAM impact or the later boot-session symptom.

## Solution Assessment

Unclear. The proposed one-line switch in the linked PR addresses the direct filter miss, but maintainers still need to confirm how to prune or sanitize base rows without dropping safe preferences and whether `boot-*` orphans need a parallel path.
