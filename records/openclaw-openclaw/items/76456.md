## Summary

Keep open. Current main still filters the cron reaper with `isCronRunSessionKey`, tests preserve the stable base cron row, and #76545 is an open closing PR for this issue, so the issue should remain open until that PR or an equivalent lifecycle-safe fix lands.

## What This Changes

The issue asks for cron session retention to sweep stable `agent:...:cron:<jobId>` base sessions, not only `...:run:<runId>` entries.

## Best Possible Solution

Define the lifecycle for stable cron base rows, then land #76545 or an equivalent fix that prunes transcript-heavy stale state without losing preserved cron preferences; split the `boot-*` orphan cleanup if source review confirms a distinct recovery bug.

## Reproduction Assessment

Yes for the cron key-selection behavior: current source and tests show the reaper prunes only `...:run:` keys and preserves `agent:...:cron:<jobId>` base rows. The reported production resource blowup and the newer `boot-*` orphan scenario were not live-reproduced here.

## Solution Assessment

Unclear. The linked one-line direction may cover cron base rows, but current docs/tests intentionally preserve base rows, and the `boot-*` evidence likely needs separate source-level analysis before folding it into the cron reaper.
