## Summary

Keep open: current main and the latest release still sweep only cron `:run:` session keys, while isolated cron writes durable base keys that skip retention; active PRs already cover the repair path, so this should stay open for maintainer lifecycle review rather than cleanup closure.

## What This Changes

The issue reports that cron session retention skips base isolated cron session keys and only sweeps `:run:` sub-session keys.

## Best Possible Solution

Land the guarded lifecycle fix in https://github.com/openclaw/openclaw/pull/90194 or an equivalent design that prunes stale orphaned base cron rows while preserving live cron base metadata, then track any remaining boot-session orphan behavior separately.

## Reproduction Assessment

Yes. Source inspection on current main shows isolated cron jobs persist stable base keys while the reaper only accepts `:run:` keys; I did not run a mutating live gateway repro during this read-only review.

## Solution Assessment

No for the simple filter swap proposed in the issue, because it can make durable base cron state retention-deletable. The safer path is the guarded ownership/lifecycle design in https://github.com/openclaw/openclaw/pull/90194 or an equivalent maintainer-approved cleanup boundary.
