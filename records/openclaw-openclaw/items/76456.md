## Summary

Keep open. Current main still filters the cron reaper with `isCronRunSessionKey`, and tests assert the stable `agent:...:cron:<jobId>` row is retained; open PR #76545 already uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for cron retention to also prune or sanitize stable cron base sessions because they can accumulate transcript-heavy state while only `:run:` sub-sessions are swept.

## Best Possible Solution

Review and land or revise PR #76545 so stale transcript-heavy base cron state is pruned or sanitized without losing intended stable preferences, and split boot orphan cleanup if it proves to be a separate lifecycle bug.

## Reproduction Assessment

Yes for the key-selection bug: current source and tests show base cron keys are skipped while `:run:` keys are pruned. The production CPU/RAM growth and the later `boot-*` restart race were not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. The linked one-line filter direction addresses the missed key family, but current code, tests, and docs intentionally preserve stable base rows, so maintainers should choose prune versus sanitize before merging.
