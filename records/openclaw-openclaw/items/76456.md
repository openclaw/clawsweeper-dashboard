## Label Justifications

- `P1`: The report describes cron session accumulation causing high CPU, event-loop stalls, RAM exhaustion, and broken scheduled-agent workflows on real installs.
- `impact:session-state`: The issue is about stale cron session rows, transcript pointers, and retained cron metadata not following the intended retention lifecycle.
- `impact:crash-loop`: The reported bloated session state can stall the gateway process and exhaust memory, making this an availability-impacting cleanup bug.

## Summary

Keep open: current main and the latest release still prune only `:run:` cron session keys, while isolated cron runs persist the active state under the stable base cron key. The original linked closing PR is still open/conflicting, and the newer canonical fix PR is also open, so this issue should remain open until one safe fix lands.

## What This Changes

The issue asks for cron session retention to sweep base cron session entries, not only `:run:` sub-session entries.

## Best Possible Solution

Land the safer cron-owned retention fix, likely through https://github.com/openclaw/openclaw/pull/90194 or an equivalent replacement, preserving protected base metadata while pruning expired isolated/deleted-job artifacts.

## Reproduction Assessment

Yes. Source inspection on current main shows an expired `agent:<id>:cron:<jobId>` base row fails `isCronRunSessionKey` and is skipped before the retention cutoff, while an expired `agent:<id>:cron:<jobId>:run:<runId>` row is eligible; no live mutating cron run was performed in this read-only review.

## Solution Assessment

No, the issue's simple filter swap is not the best complete fix by itself because base cron rows carry safe preferences and user-selected model/auth overrides. The best direction is a cron-owned retention path that protects persistent/base metadata and prunes only stale isolated or deleted-job artifacts.
