## Summary

Keep open: current main and the latest release still skip base cron session keys in the cron retention sweep, and an open linked PR plus a newer related PR are still unresolved.

## What This Changes

Fix cron session retention so base isolated cron session entries are swept instead of only `:run:` sub-session entries.

## Best Possible Solution

Land a cron-owned retention fix, likely via https://github.com/openclaw/openclaw/pull/90194 or a repaired linked PR, then close this issue after the merge ships through main.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main persists isolated cron state under the base cron key while the reaper filters only `:run:` keys. I did not run a live gateway cron scenario in this read-only review.

## Solution Assessment

No for the simplest proposed one-line filter swap; the better fix is a cron-owned retention path that prunes stale isolated base entries while protecting active or persistent named cron sessions.
