## Summary

Keep open. Current main still filters the cron reaper with `isCronRunSessionKey`, and the focused test still asserts that the stable base cron row remains, so the reported key-selection behavior is real; the remaining decision is the intended lifecycle for base cron rows because they also carry preserved cron preferences.

## What This Changes

The issue asks for `cron.sessionRetention` to sweep stable `agent:...:cron:<jobId>` base cron sessions as well as `...:run:<runId>` sub-sessions.

## Best Possible Solution

Define the intended lifecycle for stable cron base rows, then prune or sanitize stale transcript-heavy state without losing the current run metadata or preserved user preferences, with docs and regression tests updated together.

## Reproduction Assessment

Yes for the key-selection behavior: current source and tests show the reaper skips base cron keys and prunes only `...:run:` keys. The reported production token/CPU blowup was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. The proposed one-line filter swap is probably too broad because base cron rows are also stable preference and metadata carriers; a safer fix needs an explicit base-session lifecycle decision first.
