## Summary

Keep open: current main and v2026.6.10 still skip isolated cron base session keys, and although https://github.com/openclaw/openclaw/issues/89666 plus https://github.com/openclaw/openclaw/pull/90194 cover the cron-retention root cause, this issue also contains adjacent boot-session orphan evidence that is not clearly owned elsewhere.

## What This Changes

The issue reports that cron retention only sweeps `:run:` session keys and leaves base cron sessions, plus a related boot-session orphan scenario, to accumulate across runs or restarts.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/90194 or an equivalent guarded base-session lifecycle fix for cron, then split or explicitly track the boot-session orphan path if it remains reproducible after the cron repair.

## Reproduction Assessment

Yes. Source inspection on current main shows isolated cron jobs persist stable `agent:<id>:cron:<jobId>` base keys while the reaper only accepts `:run:` keys, so those base rows skip the retention cutoff; I did not run a live mutating gateway repro in this read-only review.

## Solution Assessment

No. The diagnosis is solid, but the broad predicate swap proposed in the issue is not the best fix because live base rows can carry durable cron state; the safer path is guarded orphan-only cleanup plus a separate decision on the boot-session orphan case.
