Keep open: current main and the latest release still skip base cron session keys, the guarded repair path is open but unmerged, and this issue also contains boot-session orphan evidence that does not appear to have a separate tracker.

## What This Changes

The issue reports that cron session retention only sweeps `:run:` cron session keys and leaves isolated/base cron sessions accumulating.

## Best Possible Solution

Land the guarded orphan-only cron cleanup if maintainers accept the lifecycle semantics, then explicitly split or close the boot-session orphan evidence based on whether the landed fix covers it.

## Reproduction Assessment

Yes. Source inspection on current main shows isolated cron jobs can persist `agent:<agentId>:cron:<jobId>` base rows while the reaper only accepts `:run:` keys, so those base rows skip the retention cutoff; I did not run a mutating live gateway repro in this read-only review.

## Solution Assessment

No. The issue's simple predicate-swap fix is too broad because base cron rows can carry durable session preferences; the better solution is guarded cleanup that preserves live-owned base state and prunes stale orphan rows only after maintainer acceptance.
