## Label Justifications

- `P1`: The reported cron session accumulation can stall gateway workers, exhaust memory, and break real scheduled-agent workflows.
- `impact:session-state`: The issue concerns stale cron session rows, transcript pointers, and preserved cron metadata that do not follow the intended retention lifecycle.
- `impact:crash-loop`: The report includes high CPU, event-loop stalls, and RAM exhaustion from bloated session state, which is a process availability problem.

## Summary

Keep open: current main and the latest release still sweep only `:run:` cron session keys, while isolated cron runs persist state under the stable base cron key. The linked fix PR at https://github.com/openclaw/openclaw/pull/76545 explicitly closes this issue but is still open, conflicting, and has compatibility/proof blockers, so the issue should stay open until that PR or a replacement lands.

## What This Changes

The issue asks OpenClaw to make cron session retention sweep stable base cron sessions, not only `:run:` sub-session keys.

## Best Possible Solution

Update or replace the linked PR with a compatibility-safe cron retention fix that prunes expired heavy base/run artifacts and deleted-job leftovers while preserving safe base metadata, then close this issue after the fix merges.

## Reproduction Assessment

Yes for a source-level reproduction: an expired `agent:main:cron:<jobId>` base row fails `isCronRunSessionKey` and is skipped before the retention cutoff is checked, while an expired `agent:main:cron:<jobId>:run:<runId>` row is eligible. I did not run a mutating live cron setup in this read-only review.

## Solution Assessment

No, the simple filter swap is not the best complete fix by itself because base cron rows carry safe preferences and user-selected model/auth overrides. The safer solution is a cron-owned cleanup path that removes expired heavy transcript/session state without deleting preserved base metadata unintentionally.
