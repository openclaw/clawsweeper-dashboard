## Summary

Keep open: current main and v2026.6.8 still only sweep cron keys with a `:run:` segment, while the safer canonical repair is open but currently conflicting and the boot-session comment may need separate follow-up.

## What This Changes

The issue requests retention cleanup for stale base cron session entries that are skipped by the current per-run cron session reaper.

## Best Possible Solution

Land a protected-base-key cron retention fix such as the open canonical PR, then separately track boot-session restart orphan cleanup if that path is not covered.

## Reproduction Assessment

Yes. Source inspection on current main shows stale `agent:<agentId>:cron:<jobId>` base entries are skipped because the reaper only accepts `:run:` keys; I did not run a mutating live gateway repro in this read-only review.

## Solution Assessment

No, the simple filter swap proposed in the issue is not the best final shape because it can prune persistent cron base sessions. The best fix is a guarded reaper that prunes stale isolated or orphaned base keys while preserving known persistent cron session keys.
