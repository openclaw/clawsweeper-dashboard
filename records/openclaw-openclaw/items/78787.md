## Summary

Keep open: current main still treats `mcp.servers` as a global MCP registry with only general tool-policy filtering after MCP tools are materialized, so the requested per-server agent ACL is not implemented and needs maintainer product/security direction.

## What This Changes

The issue requests a new optional `agents` list on each `mcp.servers.<name>` entry to restrict runtime MCP server exposure to selected agents and validate those IDs.

## Best Possible Solution

If maintainers accept the feature, add a reviewed core MCP ACL contract across types, schema, help, docs, runtime filtering before connection/catalog exposure, subagent semantics, and doctor validation against `agents.list`.

## Reproduction Assessment

Not applicable; this is a feature/config request, not a broken existing behavior report. Source inspection shows current main has no first-class `mcp.servers.<name>.agents` path to reproduce as failing behavior.

## Solution Assessment

Unclear until maintainers define the product and security contract. A per-server ACL may be the right core seam, but the maintainable solution needs explicit rules for subagents, server connection reuse, catalog visibility, and doctor diagnostics.
