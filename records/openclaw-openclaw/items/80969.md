## Summary

Keep open for normal review and landing: current main still retains a successful repair backup on every repair, this PR implements the narrow delete-after-success contract with regression and cleanup-failure coverage, and I found no blocking functional or security findings.

## What This Changes

The PR deletes successful transient session-repair `.bak-<pid>-<ts>` backups after atomic replacement, returns `backupPath` only when cleanup fails, and updates tests, transcript hygiene docs, and changelog text.

## Best Possible Solution

Land the focused delete-after-success contract with the included regression coverage, while keeping broader ended-session reachability or SQLite migration cleanup as separate work if still reproducible.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a repairable session JSONL on current main writes and returns a fresh retained `.bak-*` on each successful repair, and both run attempts and compaction can invoke that helper repeatedly.

## Solution Assessment

Yes. Deleting the backup only after `replaceFileAtomic` succeeds preserves the crash-safety window while stopping success-path disk growth; the latest head also covers the cleanup-failure retention branch.
