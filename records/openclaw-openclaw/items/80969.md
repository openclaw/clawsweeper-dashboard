## Summary

Keep this PR open for normal maintainer review and landing. The patch is a focused bug fix for current main's retained session-repair backups, includes source-aligned regression coverage plus a cleanup-failure test, and I found no blocking correctness or security issue.

## What This Changes

The PR updates `repairSessionFileIfNeeded` to remove successful transient `.bak-<pid>-<ts>` backups, return `backupPath` only when cleanup fails, and align tests, transcript hygiene docs, and changelog text.

## Best Possible Solution

Land the focused delete-after-success backup cleanup with the included regression coverage, while keeping stale-session reachability and future SQLite migration cleanup as separate work.

## Reproduction Assessment

Yes. Current main has a clear source path: repeated calls to `repairSessionFileIfNeeded` on a repairable JSONL file write and retain a fresh `.bak-*` after each successful repair, and the run-attempt plus compaction callers can hit that helper repeatedly.

## Solution Assessment

Yes. Deleting the backup only after `replaceFileAtomic` succeeds preserves the crash-safety window while stopping success-path disk growth; adding a rotation policy or config knob would be broader than this bug requires.
