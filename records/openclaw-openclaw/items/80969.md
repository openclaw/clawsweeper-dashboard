## What This Changes

The PR deletes successful session-file repair `.bak-<pid>-<ts>` backups after atomic replacement, returns `backupPath` only if cleanup fails, and updates tests, transcript hygiene docs, and the changelog.

## Best Possible Solution

Land this focused delete-after-success contract with the included regression coverage and keep broader SQLite migration or stale-session reachability work separate.

## Reproduction Assessment

Yes. Source inspection shows a valid session JSONL that repeatedly triggers `repairSessionFileIfNeeded` on current main will retain a unique `.bak-*` backup after each successful repair, and the PR body provides baseline-versus-patched terminal proof.

## Solution Assessment

Yes. Deleting the backup only after `replaceFileAtomic` succeeds preserves the crash-safety window while removing success-path disk growth; adding a retention config or broader migration cleanup would be larger than this bug requires.
