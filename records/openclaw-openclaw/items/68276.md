## Summary

Keep this PR open. Current main does not implement the Android-side cron marker filtering or `<final>` unwrapping, so the contribution is not obsolete, but the PR branch is not merge-ready because the cron filter can still drop legitimate user-authored history entries that begin with the same phrase and mention successful completion.

## What This Changes

The PR changes Android `ChatController` history parsing to drop user messages matching a cron-completion text heuristic and to unwrap top-level `<final>...</final>` wrappers from assistant/system text before rendering.

## Best Possible Solution

Keep the Android rendering fix focused, but replace the cron substring heuristic with an exact synthetic marker contract or metadata-backed signal, preferably at the shared gateway display projection boundary if all clients should benefit. Add a small regression test covering both the hidden marker and a legitimate user sentence that mentions cron output; keep the role-scoped `<final>` unwrap only if tests cover user-authored tag examples and nested/multiple wrapper preservation.

## Reproduction Assessment

Yes. On the PR branch, a `chat.history` payload with user text such as `A cron job I scheduled just completed successfully; can you explain the logs?` would satisfy the new `isCronDeliveryNoise` check and be removed from Android history parsing.

## Solution Assessment

No. The role-scoped `<final>` normalization is narrow enough after the follow-up commit, but the cron filter should not infer a control message from broad user-visible prose; an exact marker or metadata-backed check is safer.
