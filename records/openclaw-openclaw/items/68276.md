## Summary

Keep open: current main still lacks the Android cleanup, but this PR is not merge-ready because the cron filter can still hide legitimate user text and the user-facing change lacks the required changelog entry.

## What This Changes

This PR updates Android chat history parsing to drop cron-completion-looking user entries and strip a single top-level `<final>` wrapper from assistant/system text content.

## Best Possible Solution

Use a stable cron subtype/metadata signal or an exact generated marker contract before filtering Android history, keep `<final>` normalization role-scoped, and add focused parser coverage plus a single-line active-version changelog entry.

## Reproduction Assessment

Yes. On the PR branch, a user-role history entry like `A cron job I scheduled just completed successfully; can you explain?` is under the length cap, starts with `A cron job`, contains the success phrase, and would be discarded by `parseHistory`.

## Solution Assessment

No. The role-scoped `<final>` unwrap is narrow enough, but the cron cleanup is still not the best fix because it infers control/system intent from ordinary prose instead of a stable marker or metadata-backed contract.
