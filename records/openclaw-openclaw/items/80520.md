## Label Justifications

- `P1`: The issue covered an active Telegram regression where final replies could disappear or never reach users after generation.
- `impact:message-loss`: The reported failure mode is a generated reply not being delivered visibly in Telegram.
- `impact:session-state`: Follow-up evidence showed the assistant reply could be saved to session JSONL while Telegram delivery state did not produce a stable visible reply.

## Summary

Close: current main now fixes the reported Telegram preview/final-delivery loss by removing runner-level preview-text suppression and keeping final payloads available for channel delivery; the fix is merged via https://github.com/openclaw/openclaw/pull/83468 but is not in a tagged release yet.

## What This Changes

Issue reports Telegram replies being generated and saved while the visible Telegram reply disappears or is never sent/logged as sendMessage.

## Best Possible Solution

Keep the merged current-main fix from https://github.com/openclaw/openclaw/pull/83468, close this issue as implemented, and treat https://github.com/openclaw/openclaw/pull/83161 as superseded unless maintainers want a separate dedupe follow-up.

## Reproduction Assessment

Yes, by source and regression-test evidence: the prior implementation treated partial preview text as delivery evidence and could suppress matching final payloads, while current main keeps those final blocks for delivery. I did not run a live Telegram repro during this read-only review, but the linked reports and merged PR proof align with the source path.

## Solution Assessment

Yes. Removing preview-text-based final suppression from the core reply runner is the narrow maintainable fix because Telegram can then finalize the existing preview or fall back through its normal delivery path instead of losing the only final payload.
