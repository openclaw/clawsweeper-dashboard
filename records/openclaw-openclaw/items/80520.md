## Summary

This looks like the documented visible-replies policy path rather than an unlogged Telegram transport drop: in message-tool-only source delivery, a normal final answer can be stored in the session while no Telegram sendMessage is attempted. The report lacks config/session evidence showing this happened with automatic visible replies or after a successful message-tool send, so there is no actionable source-level bug to keep open from this thread.

## What This Changes

Investigate Telegram direct-message replies that are generated by the gateway but never result in a Telegram sendMessage call or user-visible reply.

## Best Possible Solution

Keep the documented message-tool-only behavior and use `messages.visibleReplies: "automatic"` or a model/runtime that calls the message tool for visible Telegram replies; open a new issue only with config and session/log evidence excluding that policy path.

## Reproduction Assessment

Yes for the documented policy path: current source and tests show a direct Telegram turn in message-tool-only mode can finish without any sendMessage call. No high-confidence reproduction is provided for an unexplained transport drop outside that mode.

## Solution Assessment

Yes for triage: this should be handled as documented configuration/model-tool behavior, not a source fix, unless new logs prove automatic visible replies or message-tool delivery failed.
