## Summary

## What This Changes

The PR replaces the OpenAI-compatible embedding provider's successful JSON parsing with a 16 MiB capped `readResponseWithLimit` read before `JSON.parse`.

## Best Possible Solution

Land a narrow provider fix that uses the shared bounded provider JSON reader with focused normal, malformed, and oversized response coverage plus contributor runtime proof of the embedding path.

## Reproduction Assessment

No high-confidence runtime reproduction was established. Source inspection shows current main uses `response.json()` on the successful embedding response path, but I did not run an oversized OpenAI-compatible embedding server against current main.

## Solution Assessment

No. The cap is the right direction, but the best fix is to reuse `readProviderJsonResponse` and add focused oversized success-response coverage plus real behavior proof rather than introducing a second local JSON reader.
