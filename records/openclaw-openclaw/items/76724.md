## Summary

Keep open: current main still has a source-reproducible gap between runtime MCP discovery and the Agent Tools inventory path. The reporter's `mcporter.json` location is not the documented OpenClaw MCP config source, but the supported `mcp.servers` path is also omitted from `tools.effective`, so this is not support-only.

## What This Changes

The issue requests that configured SSE MCP server tools be discovered with `tools/list` and shown in the Agent tools dashboard after a successful MCP initialize handshake.

## Best Possible Solution

Make `tools.effective` and the Tools panel include discovered bundle-MCP tools from the supported `mcp.servers` runtime path, while keeping `mcporter.json` unsupported and surfacing clear status or error notices for failed MCP discovery.

## Reproduction Assessment

Yes, source-reproducible: the UI asks `tools.effective`, and that resolver does not call the bundle-MCP runtime/materializer that performs `tools/list`. The exact external Flask SSE server was not live-run, but the reported missing dashboard discovery follows directly from current main.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse the existing session MCP runtime/materialization and bundled-tool policy in the `tools.effective` path. Adding a new `mcporter.json` convention or a separate dashboard-only MCP client would widen the contract unnecessarily.
