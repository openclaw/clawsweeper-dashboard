## Summary

Keep open: current main still has a source-proven gap in the Agent Tools dashboard path, even though the agent-run path has MCP materialization coverage. The related outbound-provider issue at https://github.com/openclaw/openclaw/issues/80909 is useful context, but it does not fully replace this dashboard/listTools symptom.

## What This Changes

Configured MCP server tools should be discovered with `tools/list` and shown in the Agent Tools dashboard after a successful MCP handshake.

## Best Possible Solution

Have `tools.effective` reuse the session MCP materialization and final policy path used by embedded Pi runs so configured MCP tools appear in the dashboard, while keeping final provider-body serialization tracked with the broader linked issue if it remains distinct.

## Reproduction Assessment

Yes from source for the dashboard path: `tools.effective` calls a synchronous inventory resolver that only builds `createOpenClawCodingTools(...)` output and never reaches the MCP runtime that performs `client.listTools()`. I did not live-run the reporter’s SSE server.

## Solution Assessment

Yes, the narrow maintainable fix is to route effective inventory through the same configured-MCP materialization and policy path used by embedded Pi runs. Closing this as implemented or fully duplicated would leave the dashboard-specific `tools/list` gap untracked.
