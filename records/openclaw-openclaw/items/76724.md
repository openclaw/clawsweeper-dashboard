## Summary

Keep open: current main still has a source-reproducible split between the documented/runtime MCP tool discovery path and the Agent Tools dashboard inventory. The reporter's `mcporter.json` location is not the documented OpenClaw MCP registry, but the current `tools.effective` path still cannot materialize configured bundle-MCP tools for the panel.

## What This Changes

Make configured SSE MCP server tools appear in the Agent Tools dashboard after the MCP initialize handshake succeeds.

## Best Possible Solution

Have `tools.effective` reuse the existing session MCP runtime/materializer and policy path so the panel reports discovered bundle-MCP tools while keeping `mcp.servers` as the supported config source.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: the panel requests `tools.effective`, and that resolver does not call the MCP runtime path that performs `tools/list`; the exact SSE server was not live-run.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is to feed bundle-MCP materialized tools into the runtime-effective inventory instead of adding a dashboard-only MCP client or supporting the reporter's undocumented `mcporter.json` path.
