## Summary

Keep open: current main still separates the Agent Tools `tools.effective` inventory path from the bundle-MCP discovery path, so configured MCP tools can be materialized for an agent run while the Tools panel remains at the built-in count. The reporter's manually created `mcporter.json` is not the documented OpenClaw MCP registry, but the dashboard/API gap is source-reproducible and should be fixed.

## What This Changes

The issue reports that a remote SSE MCP server initializes successfully but the Agent Tools dashboard and reload actions never send `tools/list` or show the server's tools.

## Best Possible Solution

Make `tools.effective` and the Tools panel include discovered bundle-MCP tools from supported `mcp.servers`, reusing the session MCP runtime/materialization and final policy path without adding a new config convention.

## Reproduction Assessment

Source-reproducible: yes, the current `tools.effective` path only resolves `createOpenClawCodingTools` and never calls the bundle-MCP catalog/materializer that sends `tools/list`. I did not live-run the reporter's SSE server setup in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to thread existing session MCP runtime/materialization into the effective inventory path and apply the same bundled-tool policy used by agent runs. A separate dashboard-only MCP client or `mcporter.json` convention would be the wrong direction.
