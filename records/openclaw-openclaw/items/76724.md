## Summary

Keep open: current main still has a plausible dashboard bug. Runtime MCP discovery calls `tools/list`, but the Control UI inventory path is fed by `tools.effective`/`tools.catalog` and does not materialize configured `bundle-mcp` servers, so the reported 33/33 dashboard symptom can still happen even when an MCP `initialize` succeeds.

## What This Changes

The issue requests that configured remote SSE MCP tools be discovered with `tools/list` and shown in the Agent Tools dashboard after a successful MCP `initialize` handshake.

## Best Possible Solution

Define one clear MCP availability contract for the Tools panel: either materialize and display `bundle-mcp` tools with status/errors, or explicitly separate configured MCP servers from runtime-discovered tools and document that distinction.

## Reproduction Assessment

Yes for the dashboard symptom by source inspection: the UI uses catalog/effective inventory, and that path does not call the bundle-MCP materializer that issues `tools/list`. The exact remote SSE server setup is not fully reproduced without the user's `mcp.servers` config and logs.

## Solution Assessment

Unclear. The source boundary is concrete, but the narrowest maintainable fix depends on whether maintainers want live remote MCP discovery in `tools.effective` or a UI/docs split between configured MCP servers and runtime-discovered tools.
