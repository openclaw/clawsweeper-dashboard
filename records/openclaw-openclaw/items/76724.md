## Summary

Keep open: current main still has runtime MCP discovery during embedded agent attempts, but the Control UI `tools.effective` inventory path remains built from `createOpenClawCodingTools` and does not materialize configured bundle MCP servers, so the reported dashboard count symptom is still plausible.

## What This Changes

The issue asks for configured remote MCP/SSE server tools to be discovered with `tools/list` and shown in the Agent Tools dashboard instead of staying at the built-in tool count.

## Best Possible Solution

Define one MCP availability contract for the Control UI: either make the live inventory safely display discovered `bundle-mcp` tools with status/errors, or show configured MCP servers separately and document that remote discovery happens only during agent runtime materialization.

## Reproduction Assessment

Yes for the dashboard symptom by source inspection: `tools.effective` does not materialize bundle MCP servers, so reloads of that panel would not trigger `tools/list`. The exact remote SSE setup is not fully reproduced because the issue lacks the OpenClaw `mcp.servers` config and logs.

## Solution Assessment

Unclear. Adding remote MCP discovery to `tools.effective` is the direct dashboard fix, but maintainers should first decide whether that endpoint may perform network discovery or whether the UI should separate configured MCP server status from runtime-discovered tools.
