## Summary

Keep open: current main still has a source-reproducible gap between runtime MCP tool materialization and the Control UI tool inventory. Agent turns materialize configured MCP servers and call `listTools()`, but `tools.effective` builds the dashboard inventory from the core/plugin/channel tool factory without the bundle-MCP materialization path; the reporter's exact `mcporter.json` setup still needs config confirmation.

## What This Changes

The issue reports that a remote SSE MCP server handshakes successfully, but Agent tools stay at the built-in count and no discovered MCP tools appear in the dashboard.

## Best Possible Solution

Define one MCP availability contract for the Tools panel: either materialize and cache configured bundle-MCP tools in `tools.effective` with visible status/errors, or show configured MCP servers separately and document that discovery occurs only during agent turns.

## Reproduction Assessment

Yes for the dashboard symptom by source inspection: `tools.effective` omits the bundle-MCP materialization path that agent turns use to call `listTools()`. The exact reported SSE setup is not fully reproduced because the issue does not include an OpenClaw `mcp.servers` config or logs, and it references a non-documented `mcporter.json` path.

## Solution Assessment

Unclear. The narrow code fix is to include bundle-MCP materialization in `tools.effective`, but maintainers should first decide whether a UI refresh is allowed to perform remote MCP discovery or should instead show configured MCP server status/guidance.
