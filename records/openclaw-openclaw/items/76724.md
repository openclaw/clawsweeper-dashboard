## Summary

Keep open: current main still discovers configured MCP tools only in the embedded agent run path, while the Control UI tool inventory/catalog path does not materialize bundle-MCP tools, so the reported 33/33 dashboard symptom is source-plausible. The exact SSE setup still needs config/log confirmation because the report mentions `mcporter.json` rather than OpenClaw's documented `mcp.servers` registry.

## What This Changes

The issue reports that an SSE MCP server completes initialize successfully but the Agent Tools dashboard never sends `tools/list` or shows the server's tools.

## Best Possible Solution

Define one MCP dashboard contract: either show materialized `bundle-mcp` tools with bounded refresh/status/error handling in `tools.effective`, or explicitly separate configured MCP servers from runtime-discovered tools in UI/docs.

## Reproduction Assessment

Yes for the dashboard symptom by source inspection: the UI calls `tools.catalog`/`tools.effective`, and that path never reaches the bundle-MCP materializer that calls `tools/list`. The exact remote SSE setup is not fully reproduced because the issue lacks the OpenClaw `mcp.servers` config and logs.

## Solution Assessment

Unclear. The source-supported gap is real, but the narrowest maintainable fix depends on whether maintainers want live remote MCP discovery in the dashboard inventory path or a clearer UI/docs separation between configured servers and per-run tool materialization.
