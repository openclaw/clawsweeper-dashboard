## Summary

Keep open: current main has runtime MCP discovery for actual agent turns, but the Control UI inventory path behind Available Right Now builds only the existing core/plugin/channel tool inventory and does not call the bundle-MCP materializer, so the reported dashboard symptom remains plausible. The issue's `mcporter.json` detail does not match the documented OpenClaw `mcp.servers` registry, so maintainers should confirm config/logs before choosing the UI/runtime contract.

## What This Changes

The issue asks for configured SSE MCP server tools to be discovered with `tools/list` and shown in the Agent Tools dashboard after a successful initialize/reload.

## Best Possible Solution

Define one dashboard MCP availability contract: either make `tools.effective` surface configured bundle-MCP tools with bounded discovery, caching, status, and errors, or make the UI/docs clearly separate configured MCP servers from runtime-only tool materialization.

## Reproduction Assessment

Yes for the dashboard symptom by source inspection: agent turns call MCP `listTools` through bundle-MCP materialization, while the `tools.effective` dashboard path never reaches that materializer. The exact reported SSE setup still needs config/log confirmation because the issue names a `mcporter.json` path rather than documented `mcp.servers`.

## Solution Assessment

Unclear. A narrow code fix could reuse the session MCP runtime cache in `tools.effective`, but maintainers should first decide whether the dashboard endpoint is allowed to perform remote MCP discovery or should instead show configured MCP server status separately.
