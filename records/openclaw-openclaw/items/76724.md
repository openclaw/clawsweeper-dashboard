## Summary

Keep open: current main still has a source-reproducible gap where agent runs can materialize configured bundle-MCP tools, but the `tools.effective` path used by the Agent Tools panel only inventories core/plugin/channel tools and does not call the MCP catalog/materialization path. The reporter's `mcporter.json` path is not the documented OpenClaw MCP registry, but the dashboard/API behavior remains a real OpenClaw bug when tools are configured under `mcp.servers`.

## What This Changes

The issue asks for remote SSE MCP tools to be discovered after a successful initialize handshake and shown in the Agent Tools dashboard via `tools/list`.

## Best Possible Solution

Make `tools.effective` and the Agent Tools panel include discovered bundle-MCP tools from the existing session MCP runtime, with clear status/error handling while keeping `mcp.servers` as the supported config source.

## Reproduction Assessment

Yes for source-level reproduction: `tools.effective` resolves inventory from `createOpenClawCodingTools` without calling the MCP runtime/materializer, while normal agent runs do call the MCP catalog path that performs `listAllTools`. No live SSE server was run.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse the existing session MCP runtime/materialization and bundled-tool policy in the `tools.effective` path, not add support for the reporter's ad hoc `mcporter.json` file.
