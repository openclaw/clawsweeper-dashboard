## Summary

Keep open: current main still splits runtime MCP discovery from the `tools.effective`/Agent Tools inventory path, so configured bundle-MCP tools can be usable during an agent run while the dashboard stays at the built-in count. The reporter's `mcporter.json` path is not the documented OpenClaw MCP registry, but the dashboard/API gap is source-reproducible and should not be closed as support-only.

## What This Changes

The issue asks for configured remote MCP tools to be discovered and shown in the Agent Tools dashboard after a successful SSE initialize handshake.

## Best Possible Solution

Make the runtime-effective tools endpoint and Tools panel include discovered bundle-MCP tools, with clear status/error handling and docs that keep mcp.servers as the supported config source.

## Reproduction Assessment

Source-reproducible: the dashboard path uses tools.effective, and that resolver does not materialize bundle-MCP tools or call tools/list. The exact remote SSE server setup was not live-reproduced because the issue lacks OpenClaw mcp.servers config and logs.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse the existing session MCP runtime/materialization and bundled-tool policy in the tools.effective path, not add a new config file convention or separate dashboard-only MCP implementation.
