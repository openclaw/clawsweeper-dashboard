## Summary

Keep open: current main still has runtime bundle-MCP discovery, but the Control UI inventory path does not materialize configured MCP servers, so the reported dashboard behavior remains plausible. The exact SSE setup still needs config/log confirmation because the report references a mcporter config path rather than the documented OpenClaw MCP registry.

## What This Changes

The issue asks for MCP server tools discovered over SSE to appear in the Agent Tools dashboard and trigger tools/list after a successful initialize handshake.

## Best Possible Solution

Define one MCP availability contract: either make the Tools panel show discovered bundle-MCP tools with status/errors, or clearly separate configured MCP servers from runtime-only tool discovery in UI and docs.

## Reproduction Assessment

Yes for the dashboard symptom by source inspection: tools.effective uses resolveEffectiveToolInventory, which builds from createOpenClawCodingTools and never materializes bundle-MCP tools. The exact Flask/SSE setup is not live-reproduced because the issue lacks the actual OpenClaw mcp.servers config and logs.

## Solution Assessment

Unclear. The narrowest maintainable fix depends on whether maintainers want remote MCP discovery inside the Control UI inventory endpoint or a UI/docs contract that says discovery occurs during agent runtime materialization.
