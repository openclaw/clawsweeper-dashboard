## Summary

Keep open: current main still leaves the Agent Tools dashboard `tools.effective` path on a static inventory that does not materialize configured MCP tools, and the related repair PR is open rather than merged. The sandbox allowlist finding in https://github.com/openclaw/openclaw/issues/80909 is useful context, but it does not by itself close this dashboard/listTools symptom.

## What This Changes

The issue reports an SSE MCP server that initializes successfully but whose tools never appear in the Agent Tools dashboard or trigger `tools/list` discovery.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/84096 for the dashboard effective-inventory path, then close this issue only after merged proof shows configured MCP tools are discoverable or the remaining work is moved to a narrower docs/diagnostics issue.

## Reproduction Assessment

Yes from source for the dashboard path: current `tools.effective` calls a static inventory resolver that never reaches the session MCP runtime that performs `client.listTools()`. I did not live-run the reporter's SSE server.

## Solution Assessment

Yes, keeping this open while the related repair PR is reviewed is the best current outcome. The safer implementation direction is an explicit refresh/materialization path that avoids making ordinary read-scoped dashboard loads start configured MCP transports.
