## Summary

Keep open: current main still has a source-reproducible gap between runtime MCP tool materialization and the `tools.effective` inventory used by the Agent Tools panel. The reporter's `mcporter.json` path is not the documented OpenClaw MCP registry, but documented `mcp.servers` tools can be available to embedded Pi runs while the dashboard path never performs MCP `tools/list` discovery.

## What This Changes

The issue asks for configured remote SSE MCP tools to be discovered with `tools/list` and shown in the Agent Tools dashboard after a successful MCP initialize handshake.

## Best Possible Solution

Make `tools.effective` and the Agent Tools panel include discovered bundle-MCP tools from the existing session MCP runtime/materializer, while keeping `mcp.servers` as the supported config source and reporting discovery failures without blocking built-in tools.

## Reproduction Assessment

Yes, source-reproducible: the dashboard path calls `tools.effective`, whose resolver builds only OpenClaw coding tools and never calls the MCP runtime that performs `tools/list`. The exact Flask SSE setup was not live-reproduced because the issue lacks a documented `mcp.servers` config and logs.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse the existing session MCP runtime/materialization and final tool policy in the `tools.effective` inventory path, not add a new `mcporter.json` convention or separate dashboard-only MCP implementation.
