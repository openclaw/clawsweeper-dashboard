## Summary

Keep open: current main still has runtime MCP discovery for embedded Pi agent turns, but the Control UI tool inventory path does not materialize configured bundle-MCP servers, so the reported dashboard symptom remains plausible from source inspection. The reported `workspace/config/mcporter.json` path is not the documented OpenClaw MCP registry, so the exact live SSE setup still needs config/log confirmation.

## What This Changes

The issue asks OpenClaw to discover and display configured SSE MCP server tools in the Agent tool inventory after a successful initialize handshake.

## Best Possible Solution

Define one MCP availability contract: either make the Tools panel show discovered `bundle-mcp` tools with status/errors, or show configured MCP servers separately and document that remote tool discovery happens during agent runtime materialization.

## Reproduction Assessment

Unclear for the exact live SSE setup because the issue lacks the OpenClaw `mcp.servers` config and logs. Source inspection does show the dashboard inventory path does not call the bundle-MCP materializer that issues `tools/list`, while the runtime agent-turn path does.

## Solution Assessment

Unclear. The narrowest maintainable fix depends on whether maintainers want live MCP discovery in `tools.effective`, with caching and failure handling, or a UI/docs change that separates configured MCP servers from runtime-discovered tools.
