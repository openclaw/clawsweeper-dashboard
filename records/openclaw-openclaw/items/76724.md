## Summary

Keep open: current main still has a plausible dashboard inventory gap for configured MCP tools. Runtime agent turns can materialize bundle-MCP tools, but the Control UI `tools.effective` path does not call that materialization path, so the reported “no tools/list from Reload/Tools panel” symptom remains source-reproducible.

## What This Changes

The issue requests that configured SSE MCP server tools be discovered with `tools/list` and shown in the Agent Tools dashboard after a successful MCP `initialize` handshake.

## Best Possible Solution

Define one effective-tools contract: either make `tools.effective` show discovered `bundle-mcp` tools with status/errors, or expose configured MCP server status separately and document that actual tool discovery happens during agent runtime materialization.

## Reproduction Assessment

Yes for the dashboard symptom by source inspection: with a valid `mcp.servers` entry, runtime materialization calls `listTools`, but `tools.effective` only resolves built-in/plugin/channel tools through `createOpenClawCodingTools`. The reporter's exact `mcporter.json` setup remains unverified.

## Solution Assessment

Unclear. The narrowest code path is to add session-scoped bundle-MCP materialization to `tools.effective`, but maintainers should first decide whether dashboard refresh is allowed to start remote MCP discovery or should present a separate MCP status surface.
