## Summary

Keep open: current main and the latest release still have a source-reproducible split where the Agent Tools panel reads `tools.effective`, but that resolver only inventories static OpenClaw tools and never uses the MCP runtime path that calls `tools/list`. The reporter's `mcporter.json` path is not the supported OpenClaw MCP registry, but the same panel gap remains for documented `mcp.servers` entries.

## What This Changes

MCP tools discovered through the runtime should appear in the Agent Tools dashboard after a successful MCP initialize handshake.

## Best Possible Solution

Have `tools.effective` reuse the existing session MCP runtime/materializer and final policy path for supported `mcp.servers`, with bounded caching and errors, while leaving mcporter registry UX to https://github.com/openclaw/openclaw/issues/65209.

## Reproduction Assessment

Yes from source: configure a supported `mcp.servers` entry and request the Agent Tools dashboard; the panel calls `tools.effective`, whose resolver cannot call the MCP runtime that sends `tools/list`. The exact reporter SSE server was not live-run.

## Solution Assessment

Yes: the narrow maintainable fix is to feed materialized bundle-MCP tools into the runtime-effective inventory instead of adding a dashboard-only MCP client or supporting the reporter's undocumented mcporter config path.
