## Summary

Keep open: current main still separates runtime MCP discovery from the `tools.effective` inventory path that feeds the Agent Tools dashboard, so configured MCP tools can be available to an agent run while the dashboard remains at the built-in count. The reporter's `mcporter.json` path is not the documented registry, but the source-backed dashboard/API gap is valid.

## What This Changes

The issue asks for configured SSE MCP server tools to be discovered with `tools/list` and shown in the Agent Tools dashboard after a successful initialize handshake.

## Best Possible Solution

Make `tools.effective` and the Tools panel include discovered bundle-MCP tools by sharing the existing session MCP runtime/materialization path, with clear status/error handling and no new config file convention.

## Reproduction Assessment

Yes, source-reproducible: current main documents `tools.effective` as the live runtime inventory, but that resolver only uses `createOpenClawCodingTools` while agent runs separately materialize MCP tools through the session MCP runtime. I did not live-run the reporter's Flask SSE server.

## Solution Assessment

Yes: the narrow maintainable solution is to reuse the existing bundle-MCP runtime/materializer and policy path in `tools.effective`, not create a dashboard-only MCP client or accept `mcporter.json` as a new registry.
