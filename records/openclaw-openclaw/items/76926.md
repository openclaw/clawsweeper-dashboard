## What This Changes

The PR adds a default-off BlueBubbles textFormatting config flag, parser/tests, macOS/private-API gating, outbound payload passthrough, generated config metadata, and a changelog entry.

## Best Possible Solution

Keep the feature in the BlueBubbles plugin, but align the parser with current `stripMarkdown` output and wait for the upstream server/helper support to settle before maintainer review.

## Reproduction Assessment

Yes for the review finding: source inspection shows `_foo_bar_` would close the `_` span at the inner underscore, while current `stripMarkdown` preserves `foo_bar` as one span. The feature itself is not a bug report, so no end-user current-main failure reproduction applies.

## Solution Assessment

No. The opt-in plugin-local direction is maintainable, but the parser should preserve the existing markdown-stripped body semantics before the draft can be ready; maintainer review should also wait on the upstream BlueBubbles releases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Respect underscore close boundaries:** `extensions/bluebubbles/src/text-formatting.ts:104-108`
  - body: The tokenizer closes an active `_` span on the first underscore it sees. With the flag enabled, `_foo_bar_` becomes `foobar_` with only `foo` formatted, while the existing `stripMarkdown` rule treats the trailing underscore as the close and sends `foo_bar`. Please apply the same close-side word-boundary rule here and cover this case so the new payload does not alter message text.
