## Summary

Keep open: the default OpenAI double-greeting fix is plausible, but the PR still leaves the `autoRespondToAudio: false` path unwired to the OpenAI bridge and the supplied proof is only tests/CLI smoke rather than real voice-call behavior.

## What This Changes

The PR changes the voice-call realtime handler to put outbound initial greetings into OpenAI session instructions and skip `triggerGreeting()` when OpenAI auto-response is expected, with handler tests for default and disabled-auto-response modes.

## Best Possible Solution

Land one consolidated fix that passes the auto-response decision through to the OpenAI bridge, keeps regression coverage for default and opt-out modes, and includes redacted real Twilio/OpenAI behavior proof showing a single first greeting.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main queues an initial greeting, triggers `triggerGreeting()` on ready, and OpenAI server VAD defaults `create_response` to true. I did not run a live phone call in this read-only review.

## Solution Assessment

No. The default-path direction is reasonable, but the disabled-auto-response branch is incomplete until the value is passed into the OpenAI bridge and proven with real behavior evidence.
