## Summary

Keep this PR open, but it is not merge-ready: the default OpenAI greeting direction is useful, while the `autoRespondToAudio: false` path still is not wired into the OpenAI bridge and the submitted proof is local harness/mock proof rather than real voice-call behavior.

## What This Changes

The PR changes the voice-call realtime handler so OpenAI outbound initial greetings are folded into session instructions and `triggerGreeting()` is suppressed when OpenAI auto-response is expected, with regression tests for default and opt-out paths.

## Best Possible Solution

Land one consolidated voice-call/OpenAI fix that carries the auto-response decision through the bridge, covers default and opt-out modes, and includes redacted live-call or provider-log proof of a single first greeting.
