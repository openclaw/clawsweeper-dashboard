## Summary

Keep open: current main and v2026.6.6 still allow Mattermost native slash registration to proceed with an HTTP-derived callback, and this now-narrow PR addresses that remaining security gap. It still needs real behavior proof and maintainer acceptance of the fail-closed upgrade impact before merge.

## What This Changes

This PR refuses non-HTTPS Mattermost native slash callback URLs at registration and updates the Mattermost docs plus monitor-slash tests.

## Best Possible Solution

Merge a narrowly scoped Mattermost slash-command transport hardening change only after maintainers accept the fail-closed upgrade impact and the PR includes redacted real runtime proof.

## Reproduction Assessment

Yes from source inspection: current main still lets `resolveCallbackUrl()` return `http://...` and `registerMattermostMonitorSlashCommands()` proceeds to registration after only a loopback warning. I did not run a live Mattermost setup in this read-only review.

## Solution Assessment

Yes as a narrow mitigation, if maintainers accept the policy choice to fail closed for all non-HTTPS native slash callbacks. A compatibility-preserving default or opt-in strict mode would be the safer alternative if existing HTTP localhost/internal deployments must keep working by default.
