## Summary

Keep open: current main and the latest release still allow Mattermost native slash registration to use a derived `http://...` callback, and this PR is the live fix candidate for that remaining transport security gap. The branch is not merge-ready because it intentionally fails closed for existing HTTP callback deployments and still needs maintainer/security acceptance plus redacted real Mattermost proof.

## What This Changes

The PR changes the Mattermost plugin to refuse native slash-command registration when the resolved callback URL is not HTTPS, and updates Mattermost docs and monitor tests.

## Best Possible Solution

Land this or a replacement Mattermost transport gate after maintainers accept the fail-closed upgrade behavior, the contributor adds redacted real setup proof, and the PR body matches the narrowed scope.

## Reproduction Assessment

Yes, source-reproducible: current main returns an `http://` callback when `commands.callbackUrl` is omitted and continues to registration after only a warning. I did not run a live Mattermost network capture in this read-only review.

## Solution Assessment

Likely yes for the remaining transport gap: the gate belongs in the Mattermost registration path before token-bearing callbacks are registered over cleartext. The best landing path still needs maintainer/security acceptance of the breaking upgrade behavior and real setup proof.
