## Label Justifications

- `P1`: The linked Mattermost slash callback issue is security-sensitive and current main still allows cleartext derived callback registration for a protected command surface.
- `merge-risk: 🚨 compatibility`: Merging this PR can stop existing derived or explicit HTTP Mattermost slash callbacks from registering until operators configure HTTPS.
- `merge-risk: 🚨 security-boundary`: The PR changes the Mattermost slash callback transport/auth boundary by failing closed before command registration.

## Summary

Keep open: this PR still contains the remaining Mattermost slash-callback transport hardening that current main lacks, but it needs maintainer security/compatibility acceptance and real Mattermost behavior proof before merge. The token-replay half was already superseded by merged current-main validation work, so the useful remaining branch scope is HTTPS fail-closed registration.

## What This Changes

This PR refuses non-HTTPS Mattermost native slash-command callback URLs before registration and updates the Mattermost docs and monitor-slash tests for that behavior.

## Best Possible Solution

Adopt an explicit HTTPS-only policy for Mattermost native slash callbacks with upgrade-facing docs and real behavior proof, or deliberately narrow the gate if maintainers want a supported local HTTP exception.

## Reproduction Assessment

Yes, at source level. Current main still derives http:// callbacks when callbackUrl is omitted and the monitor proceeds to registration after only a warning; I did not run a live Mattermost reproduction in this read-only review.

## Solution Assessment

Yes if maintainers choose HTTPS-only as the slash-command policy; otherwise the safer path is to narrow the gate to the derived/public HTTP cases and document any explicit local exception.
