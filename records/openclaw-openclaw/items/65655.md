## Summary

Keep open: this PR is the live fix candidate for the remaining Mattermost slash-command callback transport gap, but it is not merge-ready because it intentionally stops HTTP callback deployments and still needs redacted real Mattermost proof plus maintainer/security acceptance. Current main and v2026.6.10 still derive and register HTTP callbacks when `commands.callbackUrl` is omitted, while the older token-to-command replay portion is already covered by https://github.com/openclaw/openclaw/pull/72923.

## What This Changes

This PR adds a Mattermost native slash-command registration guard that refuses non-HTTPS callback URLs, updates the Mattermost docs, and adds focused monitor tests for derived and explicit HTTP callbacks.

## Best Possible Solution

Land this or a replacement Mattermost plugin transport gate after maintainers accept the fail-closed upgrade behavior, the contributor adds redacted real setup proof, and the PR body matches the narrowed scope.

## Reproduction Assessment

Yes. Current main and v2026.6.10 derive an `http://` Mattermost slash callback when `commands.callbackUrl` is omitted and proceed toward registration after only a warning; I did not run a live Mattermost network capture.

## Solution Assessment

Yes for the remaining transport gap if maintainers accept the compatibility break: the guard belongs in the Mattermost registration path before token-bearing callbacks are registered over cleartext. If preserving HTTP deployments wins, the safer alternative is a maintainer-approved staged or opt-in strict mode.
