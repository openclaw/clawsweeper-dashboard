## Summary

Keep open: current main and the latest release still allow Mattermost native slash command registration with a derived `http://` callback, and this PR is a useful narrow transport hardening patch, but it still needs real Mattermost behavior proof and explicit security/upgrade acceptance before merge.

## What This Changes

The PR adds a Mattermost monitor-side HTTPS check that refuses derived or explicit non-HTTPS native slash callback URLs before command registration and updates the Mattermost docs and monitor tests.

## Best Possible Solution

Adopt an explicit HTTPS-only Mattermost slash callback policy with upgrade-facing docs and real behavior proof, or deliberately narrow the gate if maintainers want a local HTTP exception.

## Reproduction Assessment

Yes, at source level. Enable Mattermost native slash commands, omit `channels.mattermost.commands.callbackUrl`, and current main derives `http://...` while the monitor continues to registration after only a warning.

## Solution Assessment

Yes if maintainers choose HTTPS-only as the Mattermost slash callback policy. If same-host HTTP remains supported, the safer solution is a narrower gate with matching docs and tests.
