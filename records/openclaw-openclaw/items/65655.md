## Label Justifications

- `P1`: This is an auth-sensitive Mattermost slash-command security fix with a remaining cleartext callback token exposure path.
- `merge-risk: 🚨 compatibility`: Failing closed on callback URLs can break existing Mattermost slash-command setups that currently rely on omitted or HTTP callback URLs.
- `merge-risk: 🚨 security-boundary`: The diff changes slash callback authentication and transport checks, and an unsafe rebase could weaken the stronger validation now on main.

## Summary

Keep open for maintainer rework rather than cleanup-close: current main already supersedes the token-replay half via stronger Mattermost command validation, but the remaining non-HTTPS callback transport gap is still real and this PR is conflicting, too narrow, and missing real behavior proof.

## What This Changes

The branch changes Mattermost slash-command docs, registration, callback state, HTTP auth, and tests to fail closed on omitted non-HTTPS derived callback URLs and bind callback tokens to slash triggers.

## Best Possible Solution

Rebase or replace this with a narrow Mattermost plugin fix that preserves current-command validation, rejects the agreed non-HTTPS callback cases before registration, updates docs/tests, and includes redacted real Mattermost/gateway proof.

## Reproduction Assessment

Yes, at source level. Enable Mattermost native slash commands without channels.mattermost.commands.callbackUrl; current main derives an http:// callback URL and proceeds into registration after only the warning path.

## Solution Assessment

No, not as submitted. The token-binding half is superseded by stronger current-main validation, and the remaining transport fix is too narrow because explicit http:// callback URLs still bypass it.
