## Summary

Keep open: current main and v2026.6.6 still allow Mattermost native slash registration to proceed with a derived HTTP callback, while this PR remains the targeted transport-policy fix. The command-token replay portion is already handled on main by the merged validation refresh, so the remaining blocker is maintainer acceptance of the HTTPS-only startup behavior plus real Mattermost/runtime proof.

## What This Changes

The PR adds a Mattermost monitor-side HTTPS gate for native slash callback URLs and updates Mattermost docs and monitor tests for the fail-closed behavior.

## Best Possible Solution

Adopt an explicit HTTPS-only Mattermost slash callback policy with upgrade-facing docs and real behavior proof, or deliberately narrow the gate if maintainers want a local HTTP exception.

## Reproduction Assessment

Yes, at source level: enable Mattermost native slash commands, omit channels.mattermost.commands.callbackUrl, and current main derives http:// while the monitor continues to registration after only a warning. I did not run a live Mattermost roundtrip in this read-only review.

## Solution Assessment

Yes if maintainers choose HTTPS-only as the Mattermost slash callback policy. If same-host HTTP remains supported, the safer solution is a narrower gate with matching docs and tests.
