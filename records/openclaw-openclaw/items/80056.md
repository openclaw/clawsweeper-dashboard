## Summary

Keep open. This is a draft, protected maintainer-labeled feature PR, and source review found a blocking correctness issue: policy check selection currently enables unrelated checks by default, so a channel-only policy can fail on tool metadata.

## What This Changes

The PR adds a bundled policy plugin, doctor lint health checks, oc-path document helpers, docs, and an optional trusted-tool runtime policy for policy.jsonc and TOOLS.md conformance.

## Best Possible Solution

Keep the PR open for maintainer review; narrow the policy-check toggle semantics, then validate the stacked doctor, policy, and runtime flow with focused tests and updated proof.

## Reproduction Assessment

Yes. Source inspection shows policyRuleEnabledFromSnapshot defaults every check to enabled whenever any policy section or plugin enabled flag exists; a policy with only channel checks plus a TOOLS.md entry without metadata would trigger tool findings.

## Solution Assessment

No. The PR should keep check selection tied to each explicit setting or matching policy section so channel policy does not impose tool metadata requirements.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Honor each policy check toggle independently:** `extensions/policy/src/doctor/register.ts:571-572`
  - body: `policyRuleEnabledFromSnapshot` returns true whenever the policy has any `channels` or `tools` object, or when `plugins.entries.policy.config.enabled` is true. That makes a channel-only policy also run `requireRisk` and `requireSensitivity`, so `policy check`/`doctor --lint` can fail on unrelated `TOOLS.md` declarations even though those tool metadata requirements were never enabled. Please only enable the specific check when that setting is true, or default from the matching policy section/rules.
  - confidence: 0.89
