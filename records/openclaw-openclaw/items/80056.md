## Summary

Keep this PR open for maintainer review. The proposed policy/tool conformance feature is not on current main and the PR is an active protected draft; the linked real behavior proof is sufficient, but source inspection still finds a P2 false-positive path in the tool metadata checks.

## What This Changes

The PR adds a bundled policy plugin surface for TOOLS.md metadata conformance, doctor lint integration, policy check attestations, oc-path document helpers, docs, tests, package exports, and labeler metadata.

## Best Possible Solution

Keep the draft open until the doctor/channel policy stack is accepted, then land tool metadata checks with tools-specific activation and regression coverage for non-tool policy files.

## Reproduction Assessment

Yes for the review finding by source inspection: a channels-only policy makes policyAreaExists true, and policyRuleEnabled can enable requireRisk or requireSensitivity without tools.settings opt-in. I did not run tests because this review is read-only.

## Solution Assessment

No, not as-is. Tool metadata findings should activate only from tools-specific settings or explicit plugin config, while maintainers decide the broader policy/doctor sequencing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate tool checks on tool policy settings:** `extensions/policy/src/doctor/register.ts:450`
  - body: When policy.jsonc contains only channel policy, policyRuleEnabled(ctx, "requireRisk") and "requireSensitivity" still fall through to policyAreaExists, which is true for the channels area. A workspace with channel conformance plus TOOLS.md would start emitting tool metadata errors even though the authored policy never required tool metadata, so gate these checks on tools settings or explicit plugin config.
  - confidence: 0.9
- **[P3] Reject invalid policy severity values:** `extensions/policy/src/cli.ts:100`
  - body: policy check --severity-min currently treats any unrecognized value as info because parseHealthFindingSeverity(...) falls back with ?? "info". That differs from doctor --lint and makes a typo silently run with the broadest threshold, so reject invalid values before producing an attestation.
  - confidence: 0.78
