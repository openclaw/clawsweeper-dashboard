## Summary

Keep open: this is a draft PR with the protected maintainer label, and latest-head source inspection still shows policy doctor checks are not activated through `openclaw doctor` and malformed deny-rule entries can throw in the runtime evaluator.

## What This Changes

This PR adds a bundled policy plugin, policy generate/check commands, doctor lint/fix health-check plumbing, `policy.jsonc` trusted-tool runtime guardrails, an oc-path package API seam, docs, tests, labeler, lockfile, and SDK export updates.

## Best Possible Solution

Keep the PR open for maintainer review, wire doctor lint/fix to the intended policy health contributor path, validate the full policy JSON shape before runtime matching, and require redacted after-fix proof that covers the real doctor and runtime paths before merge.

## Reproduction Assessment

Yes for the review findings by source inspection: `doctor` is configured not to load plugins while doctor lint registers only core checks, and malformed deny-rule values are not validated before matching. These are PR-introduced paths, not current-main bugs.

## Solution Assessment

No. The PR should keep policy health checks active through a narrow doctor-owned or plugin-owned registration path and reject malformed deny-rule entries during snapshot validation before runtime matching.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Load policy health checks for doctor:** `src/cli/command-catalog.ts:265`
  - body: The docs and manifest say policy contributes to `doctor --lint` and `doctor --fix`, but this command policy keeps the plugin registry from loading for `openclaw doctor`. `runDoctorLintCli` only registers core checks, so policy drift/risk checks and repairs are not available through the advertised doctor path; wire a narrow policy health registration path or load the intended contributors for doctor.
  - confidence: 0.88
- **[P2] Validate deny-rule values before matching:** `extensions/policy/src/runtime.ts:286-287`
  - body: `validatePolicySnapshot` checks that deny-rule containers are arrays, but not that each rule and `when` value is an object with string match fields. A syntactically valid policy like `denyRules: [null]` or `when: null` reaches `rule.when`/`when.tool` and turns an invalid policy into a generic hook failure instead of a deterministic invalid-policy deny.
  - confidence: 0.84
