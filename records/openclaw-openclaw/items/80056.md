## Summary

Keep open: this is a protected draft PR for a new security-sensitive policy surface, and the latest head still has blocking source-reproducible issues in the advertised doctor path and runtime policy validation.

## What This Changes

The PR adds doctor lint/fix health infrastructure, a bundled policy plugin with policy generate/check commands, doctor-backed policy checks, policy.jsonc runtime tool-call guardrails, oc-path document APIs, docs, tests, labeler updates, and SDK export metadata.

## Best Possible Solution

Keep this as a maintainer-reviewed security/product PR, wire doctor lint/fix to the intended policy health contributor path without broad arbitrary plugin preload, validate the full policy JSON shape before runtime matching, and require redacted after-fix proof for the real doctor and runtime paths.

## Reproduction Assessment

Yes for the review findings by source inspection: doctor is configured not to load plugins while doctor lint registers only core checks, and malformed deny-rule entries can reach rule.when without validation.

## Solution Assessment

No. The PR should keep policy health checks active through a narrow doctor-owned or plugin-owned registration path and reject malformed deny-rule entries during snapshot validation before runtime matching.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Load policy health checks for doctor:** `src/cli/command-catalog.ts:265`
  - body: The docs and manifest say policy contributes to openclaw doctor --lint and doctor --fix, but the doctor command policy sets loadPlugins: "never" and runDoctorLintCli only registers core checks. A real doctor lint/fix invocation therefore cannot see the policy drift/risk checks or repairs; wire a narrow policy health registration path or load the intended contributors for doctor.
  - confidence: 0.9
- **[P2] Validate deny-rule entries before matching:** `extensions/policy/src/runtime.ts:342-343`
  - body: validatePolicySnapshot only checks that denyRules containers are arrays, then matchDenyRule dereferences rule.when. A syntactically valid policy with denyRules: [null] or when: null can throw through the evaluator instead of producing the documented deterministic invalid-policy deny.
  - confidence: 0.86
