## Summary

Keep open: this is a protected draft feature PR, and the latest head still has a blocking runtime activation issue in the new trusted-tool policy path.

## What This Changes

The branch adds a bundled policy plugin, doctor lint health checks, oc-path document API exports, policy docs, and an opt-in runtime trusted-tool policy for TOOLS.md metadata conformance.

## Best Possible Solution

Land a plugin-owned policy layer only after the runtime gate follows the documented plugin/config activation contract, the stacked doctor seam is accepted, and focused tests cover the activation and fail-closed paths.

## Reproduction Assessment

Yes. Source inspection of the latest PR head shows evaluatePolicyTrustedToolCall exits unless nested config.enabled is true, while current main's plugin enable path only sets plugins.entries.<id>.enabled=true.

## Solution Assessment

No. The runtime gate should either honor plugin entry enablement plus runtimeToolPolicy, or the docs/schema/tests should consistently require config.enabled so operators cannot enable a policy gate that silently no-ops.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor plugin enablement for the runtime policy:** `extensions/policy/src/runtime-tool-policy.ts:84-85`
  - body: `openclaw plugins enable policy` writes `plugins.entries.policy.enabled=true`, and the docs say `runtimeToolPolicy` enables the gate when the policy extension is enabled. This guard returns unless the separate nested `config.enabled` is also true, so a workspace can set `runtimeToolPolicy=true` and still run governed tools with no trusted-policy check. Please make the runtime gate follow the plugin entry enablement contract, or make the documented activation path require `config.enabled` and cover it in tests.
  - confidence: 0.89
