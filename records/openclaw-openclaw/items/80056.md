## What This Changes

Adds a bundled policy plugin, policy generate/check commands, doctor lint/fix health-check plumbing, policy.jsonc trusted-tool runtime guardrails, an oc-path document API seam, docs, tests, labeler, lockfile, and SDK export updates.

## Best Possible Solution

Keep the PR open, wire doctor lint to load or register plugin-owned policy health contributors through the intended activation path, validate policy JSON shapes before matching, then add redacted real proof for doctor --fix and a runtime guard decision.

## Reproduction Assessment

Yes for the review findings by source inspection: doctor is configured not to load plugins while doctor lint only registers core checks, and malformed deny-rule entries pass validation before matchDenyRule dereferences them. These are PR-introduced paths, not current-main bugs.

## Solution Assessment

No. The PR should keep policy health checks active in doctor lint without broad arbitrary plugin preload, and should reject malformed deny-rule entries during snapshot validation before runtime matching.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Load policy health checks for doctor lint:** `src/cli/command-catalog.ts:265`
  - body: The policy plugin declares `activation.onCommands: ["doctor", "policy"]` and the docs say `doctor --lint` reports policy findings, but this command policy prevents the plugin registry from loading for `openclaw doctor --lint`. `runDoctorLintCli` only calls `registerCoreHealthChecks()`, so a workspace with `policy.jsonc` will not run the new policy drift/risk checks through doctor; only `openclaw policy check` registers them manually. Keep the doctor lint path loading or registering the intended policy health contributors.
  - confidence: 0.88
- **[P2] Validate deny-rule entries before matching:** `extensions/policy/src/runtime.ts:151-152`
  - body: `validatePolicySnapshot` now validates tool entries and deny-rule containers, but it never checks each deny-rule value or `when` object. A syntactically valid policy with `denyRules: [null]` or `when: null` reaches `matchDenyRule` and throws while reading `rule.when`, turning invalid policy into a generic hook failure instead of the documented deterministic deny.
  - confidence: 0.84
