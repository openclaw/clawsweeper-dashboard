## Summary

Keep open: this is a draft PR with the protected `maintainer` label, and the latest head still has a runtime-policy activation bug that should block merge until fixed and reviewed.

## What This Changes

The PR adds bundled policy tool metadata checks, doctor lint integration, oc-path document APIs, policy attestations, and an optional trusted-tool runtime gate.

## Best Possible Solution

Fix the runtime activation contract to honor the same enabled policy state as the health/doctor path, add a regression test for plugin-enabled plus `runtimeToolPolicy`, then continue maintainer architecture and security review.

## Reproduction Assessment

Yes, source-reproducible. With `plugins.entries.policy.enabled=true` and `plugins.entries.policy.config.runtimeToolPolicy=true` but no `config.enabled`, `evaluatePolicyTrustedToolCall` returns before loading policy even though the documented enable flow and health registration treat the plugin as enabled.

## Solution Assessment

No. The proposed runtime hook is the right narrow surface, but this implementation should align enablement with the plugin activation contract before it can be trusted as an opt-in enforcement gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor plugin enablement for runtime policy:** `extensions/policy/src/runtime-tool-policy.ts:83-84`
  - body: This guard requires `plugins.entries.policy.config.enabled === true` before it even checks `runtimeToolPolicy`. The documented setup is `openclaw plugins enable policy`, and the doctor health path treats `plugins.entries.policy.enabled === true` as enabled, so an operator can enable the plugin and set `runtimeToolPolicy: true` while runtime calls silently skip this gate. Please use the same enabled-state contract as the health registration and cover that config shape in the runtime tests.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a trusted runtime policy gate, and the current activation check can silently leave that gate unenforced for a normally enabled policy plugin.

Concerns:

- **[medium] Runtime policy can be silently disabled:** `extensions/policy/src/runtime-tool-policy.ts:83`
  - body: Because the evaluator returns unless `config.enabled` is true, a workspace that enables the policy plugin and sets `runtimeToolPolicy: true` can miss the expected fail-closed trusted-tool enforcement path.
