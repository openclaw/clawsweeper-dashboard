## Summary

Keep open. The PR is not safe to merge because it changes generated gateway-token persistence to save `gateway.auth.mode: "none"`, making the generated token ineffective and conflicting with current main's documented default of token auth unless no-auth is explicitly chosen.

## What This Changes

The PR changes `resolveGatewayInstallToken` persistence so an auto-generated gateway token is saved with `gateway.auth.mode: "none"` when no auth mode exists, and updates the unit test to expect that mode.

## Best Possible Solution

Keep generated-token persistence in token mode. If maintainers want a local no-auth path, make it an explicit exposure-aware policy that proves loopback-only operation, rejects or warns for Tailscale/private ingress and non-loopback binds, and updates the gateway docs and regression tests together.

## Reproduction Assessment

Yes for the PR-introduced behavior. The focused seam is `resolveGatewayInstallToken()` with empty `gateway.auth`, `autoGenerateWhenMissing: true`, and `persistGeneratedToken: true`; inspect the `replaceConfigFile` payload to see whether the generated token is persisted under token or none mode.

## Solution Assessment

No. Persisting `mode: "none"` from the generated-token path is not the narrowest maintainable fix, because the branch exists to satisfy a required token; a safer alternative is to preserve token mode unless a separate exposure-aware no-auth policy explicitly opts in.

## Review Findings

- **[P1] Preserve token auth for generated install tokens:** `src/commands/gateway-install-token.ts:62`
