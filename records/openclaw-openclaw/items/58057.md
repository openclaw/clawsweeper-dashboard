## Summary

Keep open: current main and v2026.6.8 still expose only static/pairing/open/disabled channel admission policies, and the iMessage ingress path blocks non-allowlisted senders before normal dispatch/message hooks can authorize them. This is a valid security-sensitive feature request, but it needs maintainer product/API review before implementation.

## What This Changes

The issue requests dynamic sender and group authorization for channel allowlists, including resolver or hook-based identity lookup and optional unknown-sender routing.

## Best Possible Solution

Add a maintainer-approved shared authorization or dynamic access-group seam that preserves current safe defaults, fails closed with useful diagnostics, and lets iMessage identity lookup plug into the existing channel ingress graph.

## Reproduction Assessment

Yes. Source inspection shows current main has no dynamic policy value and iMessage sender access is resolved from static allowlists plus pairing-store entries before dispatch.

## Solution Assessment

Unclear. The issue identifies a real gap, but HTTP resolver, pre-allowlist hook, unknown-sender routing, and dynamic access-group membership are different security/API shapes that need maintainer selection.
