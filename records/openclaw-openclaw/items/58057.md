## Summary

Keep open. Current main still only accepts static DM/group policy values and static allowlist/access-group config for iMessage; the remaining work is a new authorization/config surface that needs product and security design, not conservative cleanup closure.

## What This Changes

Request a dynamic DM/group authorization mode for message-channel allowlists, including resolver or hook support, unknown-sender routing, and iMessage/Supabase identity resolution.

## Best Possible Solution

Define an additive generic dynamic authorization or access-group contract with explicit fail-closed security semantics, then wire it through iMessage/BlueBubbles with docs and focused ingress tests while preserving static defaults.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Current-main source and docs checks show the requested dynamic policy modes, resolver config, pre-allowlist authorization hook, and unknown-sender routing are absent.

## Solution Assessment

Unclear. The proposed HTTP resolver and message authorization hook are plausible, but the safer maintainable path is a generic channel ingress or dynamic access-group contract with explicit fail-closed semantics rather than an ad hoc iMessage-only resolver.
