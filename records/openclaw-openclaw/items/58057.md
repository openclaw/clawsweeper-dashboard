## Summary

Keeping this open: current main still only supports pairing/allowlist/open/disabled policies, while iMessage sender admission returns before existing message hooks can authorize or reroute the sender. The request is a real authorization/config feature that needs maintainer product and security review, not a stale cleanup close.

## What This Changes

Request a dynamic DM/group authorization mode for channel allowlists, including an HTTP/user resolver or pre-allowlist authorization hook plus optional unknown-sender onboarding.

## Best Possible Solution

A generic, fail-closed channel-ingress/plugin API for dynamic sender membership, with iMessage coverage and security docs, is preferable to a channel-local HTTP resolver bolted into steady-state config.

## Reproduction Assessment

Yes at source level: current schemas exclude `dynamic`, iMessage normalizes unknown policies to pairing/allowlist, and denied sender decisions return before dispatch hooks. I did not run a live BlueBubbles/iMessage setup.

## Solution Assessment

Unclear as written: the need is valid, but literal `dmPolicy: "dynamic"` plus arbitrary HTTP resolver is not obviously the best permanent shape. Extending the existing channel ingress/access-group contract may keep the authorization boundary generic and fail-closed.
