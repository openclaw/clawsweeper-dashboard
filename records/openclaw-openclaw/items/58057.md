## Summary

Keeping this open: current main and the latest release still only support static DM/group policy values plus existing access-group plumbing, and iMessage does not wire a dynamic membership resolver before it drops unauthorized senders. The request is security-sensitive product/API work, not cleanup closure.

## What This Changes

Request to add dynamic sender identity resolution for DM/group allowlists, including resolver or authorization-hook support and optional handling for unknown senders.

## Best Possible Solution

Keep this as a maintainer product/security design item and decide whether dynamic sender authorization should extend the existing channel-ingress/access-group contract, rather than adding ad hoc per-channel resolver config.

## Reproduction Assessment

Yes. Source and release-tag checks show iMessage currently authorizes senders through static configured allowlists plus pairing-store entries and drops unauthorized senders before normal dispatch; no live BlueBubbles setup was run.

## Solution Assessment

Unclear. A dynamic resolver may be useful, but the best shape needs maintainer security/API judgment because current ingress already has a narrower dynamic access-group seam and this proposal would add new authorization/config behavior.
