## Summary

Keep open. Current main still only supports the existing DM/group policy set and static or pairing-store allowlists for iMessage; the requested dynamic resolver/authorization hook would add a new access-control and config surface that needs maintainer product/security design.

## What This Changes

Requests dynamic runtime identity resolution for DM and group allowlists, including dynamic policy modes, an HTTP or hook resolver, and optional unknown-sender onboarding for iMessage/BlueBubbles multi-user deployments.

## Best Possible Solution

Design a generic channel ingress authorization or access-group resolver contract that supports dynamic identity backends with clear cache, fail-closed, and audit semantics, then wire iMessage and docs/tests through that contract.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new dynamic authorization mode. Current source and docs do confirm the requested mode is absent and the existing behavior is static allowlists plus pairing-store approvals.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact HTTP resolver, hook timing, and unknown-sender routing shape should be settled as a generic access-control contract before implementation.
