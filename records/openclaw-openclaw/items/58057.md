## Summary

Keep open: current main and v2026.6.10 still do not provide `dmPolicy`/`groupPolicy: "dynamic"`, a generic HTTP-backed sender resolver, or an iMessage dynamic identity lookup. The merged access-groups work is useful partial progress, but the remaining request is a security-sensitive product/API decision rather than cleanup-close material.

## What This Changes

The issue requests dynamic sender and group authorization for allowlists, including `dmPolicy`/`groupPolicy: "dynamic"`, HTTP or hook-based identity resolution, unknown-sender routing, and iMessage/BlueBubbles identity lookup.

## Best Possible Solution

Define a maintainer-approved dynamic authorization seam, likely through channel ingress or access groups, with safe defaults preserved and iMessage able to use identity lookup without a parallel ad hoc policy path.

## Reproduction Assessment

Yes. Source inspection shows current main has no `dynamic` DM/group policy value and iMessage sender admission still resolves through static policy plus allowlists/access groups before dispatch.

## Solution Assessment

Unclear. The need is valid, but the proposed HTTP resolver, pre-allowlist hook, unknown-sender routing, and dynamic access-group membership are different security/API shapes; the narrowest maintainable path likely extends shared channel ingress or access groups instead of adding a parallel policy source.
