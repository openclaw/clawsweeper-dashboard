## Summary

Keeping this open: current main and the latest release still expose only static DM/group admission policies, and the iMessage path drops denied senders before normal dispatch or message hooks can authorize them. The request is a real security-sensitive channel authorization feature that needs maintainer product/API review, not cleanup closure.

## What This Changes

The issue requests dynamic DM/group sender authorization, optional resolver or pre-allowlist hook support, and an unknown-sender handling path for multi-user deployments.

## Best Possible Solution

Add a generic, fail-closed channel-ingress or plugin authorization seam with iMessage coverage, focused tests, and security docs, rather than bolting an arbitrary channel-local HTTP resolver into steady-state config.

## Reproduction Assessment

Yes, at source level: current config/schema and shared ingress types exclude `dynamic`, and iMessage returns on denied admission before dispatch. I did not run a live BlueBubbles/iMessage deployment.

## Solution Assessment

Unclear as written: the need is valid, but literal `dmPolicy: "dynamic"` plus arbitrary HTTP resolver config is not obviously the best permanent API. A generic channel ingress or plugin authorization contract would better match OpenClaw's owner boundaries and security posture.
