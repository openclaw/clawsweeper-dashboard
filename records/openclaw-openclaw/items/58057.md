## Summary

Keep open. Current main and the latest release still expose static DM/group policy values plus static or access-group allowlist inputs; the requested dynamic resolver, pre-allowlist authorization hook, and unknown-sender routing remain a new inbound authorization surface that needs product and security design.

## What This Changes

The issue requests dynamic DM and group allowlist authorization through runtime identity resolution, a pre-allowlist authorization hook, and optional unknown-sender routing.

## Best Possible Solution

Design an additive generic ingress authorization or access-group extension with explicit fail-closed, timeout, cache, and redaction semantics, then wire iMessage through it while preserving existing static defaults.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source and docs checks on current main verify that dynamic policy values, resolver fields, and a pre-allowlist authorization hook are absent.

## Solution Assessment

Unclear. The HTTP resolver and authorization-hook proposals are plausible, but the maintainable path is likely a generic ingress/access-group authorization contract rather than an ad hoc iMessage-only resolver.
