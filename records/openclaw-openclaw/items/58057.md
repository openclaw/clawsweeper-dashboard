## Summary

Keep open. Current main still only accepts static DM/group policy values and static or access-group allowlist inputs; the requested dynamic resolver, pre-allowlist authorization hook, and unknown-sender routing are a new inbound authorization surface that needs maintainer product and security design.

## What This Changes

Request adds dynamic DM/group identity resolution, an optional pre-allowlist authorization hook or HTTP resolver, and unknown-sender routing for allowlist-based channel ingress.

## Best Possible Solution

Define an additive generic ingress authorization or access-group contract with explicit fail-closed, timeout, cache, and redaction semantics, then wire iMessage through it while preserving existing static defaults.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source and docs checks on current main are enough to verify that dynamic policy values, resolver fields, and a pre-allowlist authorization hook are absent.

## Solution Assessment

Unclear. The HTTP resolver and authorization-hook proposals are plausible, but the maintainable path is likely a generic ingress/access-group authorization contract rather than an ad hoc iMessage-only resolver.
