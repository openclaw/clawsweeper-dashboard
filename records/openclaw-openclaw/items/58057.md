## Summary

Keep open. Current main and v2026.5.22 still restrict DM/group policy to static policy enums; access groups provide a partial dynamic Discord-backed membership path, but not the requested general HTTP resolver, pre-allowlist authorization hook, or unknown-sender route. This remains product and security design work, not a strict bug fix.

## What This Changes

Requested dynamic inbound identity resolution for DM and group allowlists via a new policy mode, resolver or authorization hook, and optional unknown-sender routing.

## Best Possible Solution

Define a generic, fail-closed dynamic sender-authorization contract, likely through access groups or a narrow plugin SDK ingress seam, then wire iMessage/BlueBubbles through it with security docs and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection confirms current main does not accept the proposed dynamic policy or resolver config, and shipped v2026.5.22 has the same policy shape.

## Solution Assessment

Unclear. The exact dmPolicy="dynamic" HTTP resolver proposal is plausible but not proven to be the best permanent API; a generic dynamic access-group or plugin SDK ingress contract may fit OpenClaw’s boundaries better.
