## Summary

Keep this open. Current main has centralized channel ingress and access groups, but it still lacks the requested generic dynamic identity resolver for iMessage/BlueBubbles DM and group allowlists; the remaining work is a security-sensitive product/API design, not conservative cleanup.

## What This Changes

The issue requests dynamic pre-allowlist identity authorization for DM and group senders, including resolver or hook support and unknown-sender routing.

## Best Possible Solution

Define an additive generic channel authorization or dynamic access-group contract, keep static allowlist/pairing defaults, fail closed on resolver errors, then wire it through iMessage with docs and focused ingress tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection confirms current main has no dynamic dmPolicy/groupPolicy or resolver field, and iMessage still drops or pairs denied senders before normal routing.

## Solution Assessment

Unclear. The reported HTTP resolver and message authorization hook are plausible, but the safer path is to settle the generic channel ingress/access-group API and security semantics before adding a public resolver mode.
