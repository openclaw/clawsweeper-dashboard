## Summary

Keep open. Current main still lacks a generic dynamic `dmPolicy`/`groupPolicy` resolver or pre-allowlist authorization hook; existing access groups cover static sender lists and Discord channel audiences, so this is a security-sensitive product/API design rather than conservative cleanup.

## What This Changes

Request adding dynamic inbound identity resolution for DM and group allowlists, with HTTP or hook-based authorization and optional unknown-sender routing.

## Best Possible Solution

Define an additive generic channel authorization or dynamic access-group contract, keep static allowlist/pairing defaults, fail closed on resolver errors, then wire it through iMessage with docs and focused ingress tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main lacks dynamic policy modes and iMessage denies or pairs senders before normal routing.

## Solution Assessment

Unclear. The HTTP resolver and authorization hook are plausible, but the safer solution is a generic channel authorization or dynamic access-group contract with fail-closed security semantics before adding iMessage-specific config.
