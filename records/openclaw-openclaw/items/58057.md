## Summary

Keep open. Current main still supports static allowlists, pairing, open, disabled, and a narrow Discord dynamic access-group type, but not the requested dynamic DM/group identity resolver, pre-allowlist authorization hook, or unknown-sender onboarding route.

## What This Changes

The issue requests dynamic DM/group identity authorization for channel allowlists, including resolver or hook options and an unknown-sender onboarding path.

## Best Possible Solution

Design a generic, fail-closed ingress authorization or access-group/SDK seam with explicit trust, caching, audit, and docs/tests, then wire iMessage and similar channels through that shared contract.

## Reproduction Assessment

Yes at source level: current schemas, docs, and iMessage ingress only accept the existing policy values and block or pair before dispatch. I did not run a live BlueBubbles/iMessage repro.

## Solution Assessment

Unclear: the issue identifies a real gap, but choosing raw HTTP resolver, pre-allowlist hook, access-group resolver, or trusted plugin SDK seam needs maintainer product and security design first.
