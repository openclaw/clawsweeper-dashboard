## Summary

Keep this open. Current main still has only static DM/group access policies and no pre-allowlist dynamic resolver or authorization hook for BlueBubbles/iMessage; the prior ClawSweeper comment remains accurate after rechecking current source. This also should not close as ClawHub-only because the missing piece is a generic, security-sensitive channel authorization seam rather than an optional plugin that can be built against today’s public API.

## What This Changes

This feature request asks for dynamic DM and group authorization, such as `dmPolicy`/`groupPolicy: "dynamic"`, HTTP identity resolution, a pre-allowlist authorization hook, and optional unknown-sender routing for multi-user BlueBubbles/iMessage deployments.

## Best Possible Solution

Define an additive generic channel authorization contract that runs before static allowlist denial, preserves existing static and pairing defaults, fails closed on resolver or hook errors, records denial reasons safely, and then wire BlueBubbles to that contract with focused docs and tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: set the requested `dmPolicy: "dynamic"` or `groupPolicy: "dynamic"` and current schema/types have no such enum, while BlueBubbles still blocks non-allowlisted senders before route dispatch or message hooks can run.

## Solution Assessment

Unclear. The proposed HTTP resolver and authorization hook are plausible, but the maintainable path should settle a generic channel/plugin authorization API and security semantics before adding a BlueBubbles-specific resolver.
