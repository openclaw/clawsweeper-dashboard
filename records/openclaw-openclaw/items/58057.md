## Summary

Keep open. Current main still lacks the requested dynamic `dmPolicy`/`groupPolicy` modes, resolver config, pre-allowlist authorization hook, and unknown-sender routing; the adjacent access-group work does not cover iMessage/Supabase-style identity resolution, and this changes an inbound authorization boundary that needs maintainer product and security design.

## What This Changes

The issue requests resolver-backed dynamic DM/group allowlist authorization, plus optional unknown-sender routing, for scalable multi-user channel deployments such as BlueBubbles/iMessage.

## Best Possible Solution

Design an additive generic channel authorization or resolver-backed access-group contract that fails closed, documents timeout/cache/redaction behavior, then wire iMessage through it with schema, docs, and focused ingress tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request rather than a broken existing contract. Source inspection gives high confidence that current main lacks the requested dynamic policy modes and resolver/hook contract.

## Solution Assessment

Unclear. The HTTP resolver and authorization hook sketches are plausible, but the safer maintainable path is a generic channel authorization or dynamic access-group contract with explicit fail-closed security semantics before adding iMessage-specific config.
