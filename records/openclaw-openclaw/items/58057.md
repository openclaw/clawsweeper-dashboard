## Summary

Keep open. Current main has adjacent access-group work, including Discord dynamic channel audiences, but it still does not provide the requested generic dynamic `dmPolicy`/`groupPolicy` mode, HTTP resolver, pre-allowlist authorization hook, or iMessage/Supabase-style identity resolution; this is an authorization-boundary feature that needs maintainer product and security design.

## What This Changes

Request dynamic inbound identity resolution for DM/group allowlists, including resolver or authorization-hook config plus an unknown-sender handling path.

## Best Possible Solution

Define an additive generic channel authorization or dynamic access-group contract with fail-closed behavior, keep static allowlist/pairing defaults, then wire it through iMessage with docs and focused ingress tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main lacks the requested dynamic policy modes or pre-allowlist authorization hook, and iMessage still evaluates static ingress policy inputs.

## Solution Assessment

Unclear. The proposed HTTP resolver and authorization hook are plausible, but the safer maintainable path is a generic channel authorization or dynamic access-group contract with explicit fail-closed security semantics before iMessage-specific config is added.
