## Summary

Keep open: the startup sweep is a plausible fix for the SQLite ingress deadlock, but the current patch can still release a freshly re-claimed row and the external PR only supplies deterministic test output, not live crash/restart proof.

## What This Changes

The PR adds a gateway startup sweep that resets stale claimed `channel_ingress_events` rows to `pending` and adds ingress-queue/startup sweep tests.

## Best Possible Solution

Keep the startup sweep approach, but guard each release with the original selected claim identity, count actual affected rows, add a re-claim race regression, and require redacted live recovery proof before merge.

## Reproduction Assessment

Yes for the patch defect via source inspection: a selected stale row can be released and re-claimed before this sweep's update, and `status = claimed` still matches. The original reported bug is also independently reproduced in the linked issue, but I did not run a live gateway repro in this read-only review.

## Solution Assessment

No. The startup sweep is the right layer for the reported deadlock, but the current implementation needs compare-and-set guards around the original claim identity before it is the best safe fix.
