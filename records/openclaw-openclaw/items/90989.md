## Summary

Keep this PR open: current main still lacks a startup sweep for stale SQLite channel ingress claims, but the proposed PID-only liveness guard misses the container/Kubernetes PID-reuse cases documented on the linked canonical issue, so it should not land as-is.

## What This Changes

The PR adds a gateway startup sweep that releases stale claimed `channel_ingress_events` rows with dead owners, wires it into gateway bootstrap, and adds queue/startup tests.

## Best Possible Solution

Make the startup recovery process-instance-aware, or otherwise PID-reuse-safe, while preserving the age gate and exact `claim_token` guard before landing the shared gateway sweep.

## Reproduction Assessment

Yes, source inspection and linked reports give a high-confidence reproduction path, though I did not execute it in this read-only pass. Current main lacks a startup sweep, and the canonical issue includes Docker/Kubernetes `claim_owner=1:<uuid>` rows that survive restarts.

## Solution Assessment

No, not as currently implemented. The shared startup sweep is the right owner boundary, but the PID-only liveness check must handle process/PID reuse before this is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Recover claims owned by the restarted gateway PID:** `src/channels/message/ingress-queue.ts:906`
  - body: `isIngressClaimProcessAlive` treats any live PID as an active owner, so a stale row with `claim_owner = "1:<old uuid>"` is skipped after a Docker or Kubernetes restart where the new gateway is PID 1. The linked canonical issue has multiple reports of exactly that state, so this startup sweep would still leave those users deadlocked; the owner check needs to distinguish the current/restarted process instance from a true sibling owner.
  - confidence: 0.92
- **[P3] Reject malformed PID prefixes before liveness checks:** `src/channels/message/ingress-queue.ts:965`
  - body: `Number.parseInt` accepts leading digits in malformed owners such as `${process.pid}abc`, which can make a non-supported owner string look live and keep a stale row claimed. Require either a fully numeric owner or a numeric prefix followed by `:` before calling `process.kill`.
  - confidence: 0.86
