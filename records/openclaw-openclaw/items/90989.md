## Summary

Keep open: the startup sweep is a useful fix direction and has solid dead-PID proof, but the current owner-liveness predicate still preserves any non-current live PID, which leaves the documented unrelated-PID-reuse deadlock unresolved.

## What This Changes

The PR adds a gateway startup sweep that releases stale claimed `channel_ingress_events` rows back to `pending` and adds ingress/gateway regression tests.

## Best Possible Solution

Land the startup sweep only after owner-liveness semantics recover PID-reused stale claims without stealing genuine live sibling claims, or keep the remaining PID-reuse work explicitly tracked before closing the linked SQLite issue.

## Reproduction Assessment

Yes, at source level. Current main lacks a startup-wide sweep for already-claimed SQLite ingress rows, and linked reports provide concrete claimed/pending queue states plus a non-current PID-reuse case the PR still preserves.

## Solution Assessment

No. The startup sweep is the right boundary for the crash/restart gap, but the live-PID predicate is still too broad to close the reported stale-claim family safely.

- **[P1] Do not treat every live non-current PID as a sibling owner:** `src/channels/message/ingress-queue.ts:968-969`
  - body: This returns `true` for any positive non-current PID where `process.kill(pid, 0)` succeeds. https://github.com/openclaw/openclaw/issues/94546 reports a stale claimed Telegram row whose recorded PID had been reused by an unrelated macOS process, so this sweep would still skip the row and leave later messages blocked.
