## Summary

Keep this PR open: it targets a real channel deadlock bug, but the new startup sweep can release a freshly re-claimed ingress row in a concurrent gateway recovery race, and the PR still lacks live gateway or Telegram recovery proof.

## What This Changes

The PR adds a gateway startup sweep for stale SQLite channel ingress claims, wires it into plugin bootstrap, and adds queue/startup sweep tests.

## Best Possible Solution

Keep the startup sweep approach, but guard each release with the original selected claim identity, count actual affected rows, add a race regression test, and require redacted live recovery proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence race path: select a stale claimed row, let another gateway release and re-claim it, then this sweep still matches `status = claimed` and can release the fresh claim.

## Solution Assessment

No. The startup sweep is the right layer for the reported deadlock, but the current implementation needs compare-and-set guards before it is the best safe fix.
