## Summary

Keep open: the startup sweep is the right boundary for crash-left SQLite ingress claims, but the current owner-liveness rule still leaves the linked live-PID reuse deadlock unresolved for stale claims whose old PID now belongs to another process.

## What This Changes

The PR adds a best-effort gateway startup sweep that releases stale claimed `channel_ingress_events` rows before channel startup, plus regression tests for dead, current, malformed, and sibling owner cases.

## Best Possible Solution

Land the startup sweep only after owner-liveness semantics recover PID-reused stale claims without stealing genuine live sibling claims, or narrow the PR so the remaining PID-reuse work stays explicitly tracked by the canonical Telegram stale-claim issue.

## Reproduction Assessment

Yes, at source level. Current main lacks a startup-wide recovery pass for already-claimed SQLite ingress rows, and linked reports provide concrete `claimed` plus `pending` queue states that survive restarts.

## Solution Assessment

No. The startup sweep is the right owner boundary, but the proposed live-PID predicate still leaves a documented PID-reuse deadlock unresolved and needs either a stronger identity rule or an explicit partial-fix scope.
