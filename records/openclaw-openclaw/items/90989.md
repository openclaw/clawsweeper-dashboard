## Label Justifications

- `P1`: The PR targets a real channel deadlock where inbound Telegram messages can remain undelivered after gateway crashes or restarts.
- `merge-risk: 🚨 message-delivery`: The startup sweep decides whether persisted inbound channel events are requeued for delivery, so an incorrect owner boundary can duplicate or disrupt messages.

## Summary

Keep open: the latest head fixes the dead-PID and PID-reuse cases from the earlier review, but it now treats age alone as the recovery boundary. That can requeue an old claimed event during an in-process gateway restart while the previous channel handler is still settling, which can duplicate or disrupt inbound message delivery.

## What This Changes

The PR adds a gateway startup sweep that releases old claimed `channel_ingress_events` rows back to `pending` before channel workers start.

## Best Possible Solution

Keep the startup recovery, but gate it with a lifecycle-safe owner rule or fresh-process-only boundary and add a regression for an old active claim during in-process restart shutdown.

## Reproduction Assessment

Yes. Current main lacks a startup-wide sweep for already-claimed SQLite ingress rows, and the linked issue/comments provide concrete claimed/pending queue states on released builds.

## Solution Assessment

No. The startup boundary is a good fit for the crash/restart gap, but age-only recovery on every bootstrap is not the safest fix because in-process restarts can leave old handlers alive past the shutdown grace window.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not requeue old claims while prior handlers can still be running:** `src/channels/message/ingress-queue.ts:893`
  - body: This age-only filter will recover any claim older than `staleMs` on the next bootstrap. During an in-process restart, channel shutdown can continue after a five-second stop timeout while Telegram only best-effort waits for spooled handlers, so a long-running claimed update can be put back to `pending` while its previous handler still owns work. That can produce duplicate processing or make the old completion lose its claim token; keep a lifecycle/owner guard or skip this sweep when the prior lifecycle has not fully stopped.
  - confidence: 0.86
