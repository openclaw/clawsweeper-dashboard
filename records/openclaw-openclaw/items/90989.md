## Summary

Keep open: this fixes a real SQLite ingress recovery gap, but the latest head still makes age the only ownership boundary for startup requeueing, which can duplicate or disrupt inbound message handling during SIGUSR1 in-process restarts.

## What This Changes

The branch adds a gateway startup sweep that releases old claimed `channel_ingress_events` rows back to `pending` and adds channel/gateway tests for stale-claim recovery.

## Best Possible Solution

Keep the startup recovery direction, but require a lifecycle-safe owner boundary, boot identity, or lease model that proves the previous claimant is gone before releasing old claims.

## Reproduction Assessment

Yes. Current main lacks a startup-wide sweep for already-claimed SQLite ingress rows, and the linked issue plus comments provide concrete released-build `claimed` and `pending` queue states after crash/restart.

## Solution Assessment

No. Startup recovery is a plausible layer for the crash gap, but age-only recovery across all queues is not the safest fix because SIGUSR1 restarts can start a new lifecycle while old handlers are still settling.

- **[P1] Do not requeue old claims while prior handlers can still run:** `src/channels/message/ingress-queue.ts:893`
  - body: The sweep now treats age as the only owner boundary, so any claimed row older than `staleMs` is reset to `pending` on the next bootstrap. SIGUSR1 restarts are in-process: shutdown can continue after channel stop grace, and Telegram handler draining is also bounded, so an old handler can still be settling when the new startup sweep requeues its claim. That can duplicate inbound processing or make the old completion lose its claim token; gate recovery on a lifecycle owner/lease that proves the previous handler is gone.
  - confidence: 0.86
