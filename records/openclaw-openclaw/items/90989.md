## Summary

Keep this PR open: the startup recovery idea addresses a real stale-claim gap, but the proposed cross-queue sweep would incorrectly recover current Telegram spool claims whose live owner IDs use the existing `pid:uuid` format.

## What This Changes

Adds a gateway startup sweep that scans SQLite channel ingress claims and releases rows whose owner process is considered dead, with queue and startup-runner unit tests.

## Best Possible Solution

Keep the startup recovery approach only after it preserves current claim-owner contracts, guards cross-process races, and is backed by focused tests plus redacted live gateway/Telegram restart proof.

## Solution Assessment

No. Startup recovery is the right layer, but this implementation is not the best fix until it reuses or parses existing owner contracts instead of treating non-numeric owners as dead.

Full review comments:

- **[P1] Preserve live compound claim owners:** `src/channels/message/ingress-queue.ts:925-927`
  - body: The sweep runs for every `channel_ingress_events` claim at gateway startup, but this helper parses the owner with `Number(claimOwner)` and returns false for non-numeric owners. Existing Telegram spool claims are stored as `pid:uuid` and the current Telegram recovery path parses the PID prefix before preserving another live process's claim. With this PR, a gateway startup that overlaps such a live claim will release it back to pending, allowing duplicate or concurrent handling. Parse the existing owner formats or scope the sweep before releasing.
