## Best Possible Solution

Yes, this is an acceptable narrow fix: it reuses the existing textChunkLimit, message chunker, static-card sender, and preview discard path rather than adding a new config surface. The remaining question is maintainer acceptance of the visible fallback behavior, not a concrete code defect.
