## Summary

This PR addresses a real Codex harness compaction gap, but the current patch should not merge as-is because the new core-side successor transcript can become the active session without preserving the Codex app-server thread binding needed on the next turn.

## What This Changes

The PR rotates to a synthetic successor transcript after successful harness compaction that did not return its own successor transcript, with focused regression coverage.

## Best Possible Solution

Keep the harness transcript rotation, but make the session handoff preserve or deliberately invalidate harness-owned state such as Codex app-server thread bindings before returning a new active session file.

## Reproduction Assessment

Yes. Source inspection shows current `main` returns Codex harness compaction success without an OpenClaw successor transcript, and the PR's changed path can be followed to the active session-file update and Codex binding lookup.

## Solution Assessment

No. The central rotation direction is maintainable, but returning a core-created successor without carrying the Codex binding breaks the harness-owned thread lifecycle after native compaction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve Codex bindings when returning a rotated session:** `src/agents/pi-embedded-runner/compact.queued.ts:370-371`
  - body: When this path returns `rotation.sessionFile`, the gateway stores that successor as the active session file. Codex app-server bindings are stored next to the session file and the next Codex run reads only the active file's binding, but this core rotation neither moves/copies the native thread binding nor gives the Codex harness a chance to do it. A native compaction would abandon the compacted Codex thread on the next turn and leave the stale binding on the archived parent.
  - confidence: 0.84
