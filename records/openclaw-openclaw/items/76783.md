## Summary

Current main and the shipped v2026.5.2 tree already keep these replay entries out of Gemini payloads through the shared transport transform and Google converter, and the proposed branch should not land as-is because its new zero-usage check ignores `totalTokens`.

## What This Changes

The PR adds Google replay-context sanitization before message conversion, filtering runtime-context custom messages and selected poisoned assistant turns, plus regression coverage.

## Best Possible Solution

Close this redundant PR and keep the shipped shared transport filtering; if maintainers want extra coverage, add test-only regression coverage without a provider-local sanitizer that changes usage semantics.

## Reproduction Assessment

No, I did not find a failing current-main path. Source and release inspection show current Google replay already drops failed assistant turns and does not serialize runtime-context custom entries or empty assistant turns into Gemini payloads.

## Solution Assessment

No, this branch is not the best fix to land because the behavior is already present and the new helper can misclassify valid silent replies by ignoring `usage.totalTokens`. The safer direction is to close it or keep only narrowly targeted regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve nonzero totalTokens silent replies:** `extensions/google/transport-stream.ts:412`
  - body: `isZeroUsageEmptyStopAssistantTurn` treats a replayed empty `stop` assistant turn as zero-usage when the component counters are missing or zero, but the transport usage shape uses `totalTokens`. A valid silent reply with a positive `totalTokens` value could be dropped from history; include `totalTokens` in this check or reuse the existing usage shape before filtering.
  - confidence: 0.87
