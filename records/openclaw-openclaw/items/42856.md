## Summary

Current `main` still misclassifies the supported read-only `config.schema.lookup` action as mutating, which feeds the failed-tool warning policy and can expose an internal schema path in Telegram. The original repository fixes closed unmerged, while the surviving fork copies are not safe canonical landing paths: one is stale/unstable with a broader fail-open classification concern and no live proof, and the other conflicts with current code.

## What This Changes

Prevent missing `config.schema.lookup` paths from producing user-facing channel warnings while preserving the structured error returned to the agent.

## Best Possible Solution

Classify supported compound gateway actions from the canonical gateway method-scope contract or an equally narrow explicit mapping, keep unknown methods fail-closed, add focused classifier and warning-policy coverage, and provide redacted Telegram proof that the structured lookup error remains agent-visible without entering chat.

## Reproduction Assessment

Yes at source level: current `main` deterministically marks `gateway` action `config.schema.lookup` as mutating, and failed mutating actions enter the separate user-visible warning policy. This read-only review did not perform a live Telegram run.

## Solution Assessment

The shared classifier is the correct root-cause boundary, but the surviving PR's unrestricted leaf-verb fallback is not the best final implementation because it can classify unsupported methods as read-only; maintainers should choose a contract-backed narrow mapping.
