## Summary

Keep open. The PR is a plausible direction, but it is not mergeable as-is because it only wires the callback source message id through one old fallback callback path while current main has other generic callback paths that still use `callback.id`, and the PR has no real Telegram proof.

## What This Changes

The branch adds a Telegram callback source-message-id field through Telegram context options, model-visible inbound metadata, `MsgContext`, and threading tool context.

## Best Possible Solution

Land a rebased equivalent that carries `callbackMessage.message_id` through every generic Telegram callback turn via the current turn-context, metadata, and threading surfaces, with focused regression coverage and real Telegram proof.

## Reproduction Assessment

Yes, source-level. On current main, a Telegram `callback_query` with `id: "cbq-1"` and `message.message_id: 10` reaches generic callback paths that pass `messageIdOverride: callback.id`, and current context types have no separate callback source message-id carrier.

## Solution Assessment

No, not as-is. The additive metadata direction is narrow, but the PR must be rebased to carry the source message id through multi-select, single-select, and fallback/native generic callback turns while preserving `callback.id` for callback answering and dedupe.
