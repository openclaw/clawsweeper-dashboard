## Summary

Keep open. The PR targets a real Telegram callback gap, but the branch only wires one old fallback path, current main still lacks the callback source-message-id carrier, and required real Telegram proof is missing.

## What This Changes

The PR adds a Telegram callback source-message-id field through Telegram context options, inbound metadata, model-visible `callback_message_id`, `MsgContext`, and threading tool context.

## Best Possible Solution

Land a rebased equivalent that carries `callbackMessage.message_id` through every generic Telegram callback turn via the current turn-context, metadata, and threading surfaces, with focused regression coverage and real Telegram proof.

## Reproduction Assessment

Yes, source-level. On current main, a Telegram `callback_query` with `id: "cbq-1"` and `message.message_id: 10` reaches generic callback paths that set `messageIdOverride` from `callback.id`, and there is no separate callback source message-id carrier.

## Solution Assessment

No, not as-is. The additive metadata direction is narrow, but this branch only wires one old fallback path and needs rebase onto the current `buildChannelTurnContext` flow covering multi-select, single-select, and fallback/native generic callback turns.
