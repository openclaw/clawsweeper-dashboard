## Summary

Keep open. The PR is a plausible direction, but it is incomplete against current main because it wires the callback source message id through only one generic callback dispatch while current multi-select, single-select, and fallback/native callback paths still use `callback.id`; it also lacks required real Telegram proof and is currently conflicting.

## What This Changes

The PR adds Telegram callback source-message-id propagation into inbound context, model-visible metadata, `MsgContext`, and reply-threading context.

## Best Possible Solution

Land a rebased equivalent that carries `callbackMessage.message_id` through every generic Telegram callback turn, preserves `callback.id` for callback answering and dedupe, adds focused regression coverage, and includes real Telegram proof.

## Reproduction Assessment

Yes, source-level. A Telegram `callback_query` with `id: "cbq-1"` and `message.message_id: 10` reaches generic callback paths that currently use `callback.id`, and existing fixtures exercise that shape without asserting callback message-id metadata.

## Solution Assessment

No, not as-is. The additive metadata direction is plausible, but the branch only wires one older fallback path and needs current multi-select, single-select, and fallback/native callback coverage plus real Telegram proof.
