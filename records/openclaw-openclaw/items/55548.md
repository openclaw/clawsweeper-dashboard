## Summary

Keep this PR open. Current main and v2026.4.29 still expose generic Telegram callback turns without the source callback message id, while this PR is a small relevant implementation for #55462 and its latest head addresses the earlier review feedback by propagating the value into model-visible metadata and threading context.

## What This Changes

The PR adds Telegram callback source message id metadata through `callbackMessageId`/`CallbackMessageId`/`callback_message_id` and uses it in inbound metadata plus message-tool threading context.

## Best Possible Solution

Land this PR or an equivalent Telegram callback metadata change after targeted coverage for callback context, inbound metadata, and threading context; then close #55462 as implemented and keep #45638 separate unless maintainers explicitly want the broader `MessageSid` semantics change here.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Telegram `callback_query` where `callback.id` differs from `callback.message.message_id`; current main passes the callback id as `messageIdOverride` and exposes no separate callback source id.

## Solution Assessment

Yes for #55462: the additive callback source id is a narrow, backward-compatible way to expose the message id needed for `editMessage`. It is not the full #45638 fix, so maintainers should decide whether that older `MessageSid` behavior belongs in this PR or a separate follow-up.
