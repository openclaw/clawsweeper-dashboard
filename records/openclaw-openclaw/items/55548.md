## Summary

Keep open. The bug is valid, but this external PR is not mergeable as-is because it lacks real Telegram proof and only propagates the source message id through one old callback path while current main has additional generic callback paths.

## What This Changes

The PR adds a Telegram callback source-message-id field to inbound context, model-visible metadata, `MsgContext`, and reply threading.

## Best Possible Solution

Land a rebased equivalent that carries `callbackMessage.message_id` through every generic Telegram callback turn, preserves `callback.id` for callback answering and dedupe, adds focused regression coverage, and includes real Telegram proof.

## Reproduction Assessment

Yes, source-level. A Telegram `callback_query` with `id: "cbq-1"` and `message.message_id: 10` reaches generic callback code that currently sets `messageIdOverride` from `callback.id`, and the existing callback test fixture exercises that shape without asserting the message id metadata.

## Solution Assessment

No, not as-is. The additive metadata direction is plausible, but this branch only wires one old fallback path and needs rebase coverage for current multi-select, single-select, and fallback/native generic callback turns.
