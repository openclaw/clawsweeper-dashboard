## Summary

Keep open. Current main still lacks a callback source-message-id carrier, and this PR is a plausible direction, but the branch needs adaptation to current Telegram callback/context flow and after-fix real Telegram proof before merge.

## What This Changes

The PR adds a Telegram callback source-message-id carrier through Telegram inbound context, auto-reply metadata, `MsgContext`, and message-tool threading context.

## Best Possible Solution

Land a rebased equivalent that carries `callbackMessage.message_id` through every generic Telegram callback turn via the current turn-context, metadata, and threading surfaces, with focused regression coverage and real Telegram proof.

## Reproduction Assessment

Yes, source-level. On current main, a Telegram `callback_query` with `id: "cbq-1"` and `message.message_id: 10` reaches callback paths that set `messageIdOverride` from `callback.id`, and there is no `CallbackMessageId` or `callback_message_id` carrier.

## Solution Assessment

No, not as-is. The additive metadata direction is narrow and maintainable, but this branch should be rebased onto the current `buildChannelTurnContext` flow and cover all agent-routed callback paths before merge.
