## Summary

Keep open. Current main and v2026.5.19 still render ordinary Slack markdown through the mrkdwn/text send path, while outbound MarkdownIR-to-rich_text conversion remains unimplemented and the RFC still needs maintainer decisions on defaulting, config, streaming, chunking, and explicit-block precedence.

## What This Changes

Request automatic conversion of Slack agent markdown replies into Slack rich_text blocks using the existing MarkdownIR parser.

## Best Possible Solution

Keep this as a focused Slack rich-output RFC, decide default/config, streaming/final delivery, chunking, fallback text, and explicit-block precedence, then implement a Slack-owned MarkdownIR-to-rich_text renderer with focused tests and redacted real Slack proof.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature RFC. Source inspection gives a high-confidence verification path: ordinary Slack text still renders through mrkdwn/text chunks and no outbound rich_text converter exists on current main or v2026.5.19.

## Solution Assessment

Unclear until maintainers answer the RFC choices. A Slack-local renderer appears to be the likely boundary because MarkdownIR is already available to the Slack plugin, but default/config, streaming, chunking, and precedence semantics need a product decision first.
