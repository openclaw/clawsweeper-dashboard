## Summary

Keep open. Current main still lacks the requested Markdown-to-Slack `rich_text` auto-conversion: ordinary Slack outbound text renders through the existing `mrkdwn` path, while Block Kit is only used for explicit Slack blocks, shared presentation blocks, or interactive replies. The broader #12602 remains open, and this issue is a narrower RFC with product decisions still unresolved.

## What This Changes

The issue proposes automatically converting Slack agent Markdown output into Slack `rich_text` Block Kit blocks using the existing Markdown IR instead of rendering ordinary text as Slack `mrkdwn`.

## Best Possible Solution

Keep this as a focused Slack plugin follow-up under #12602: implement a Slack-owned `MarkdownIR` to `rich_text` renderer if maintainers approve the direction, preserve explicit block precedence, define opt-in/default behavior, and cover normal sends, streaming/final updates, chunking, lists, quotes, links, inline styles, fenced code, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction. Current-main source inspection provides a high-confidence absence check: ordinary Slack sends still route through `markdownToSlackMrkdwnChunks()`, and no outbound `rich_text` renderer exists.

## Solution Assessment

Unclear until maintainers answer the RFC’s product questions. A Slack plugin-local renderer using the existing plugin SDK text runtime looks like the narrowest technical direction; moving it into the shared SDK should wait until another plugin needs the same target renderer.
