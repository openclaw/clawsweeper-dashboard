## Summary

Keep open: the PR targets a real auth/model-cache churn bug, but the proposed no-save path is not merge-ready because it makes steady-state `lastUsed` updates memory-only while current auth-profile ordering uses persisted `lastUsed` for round-robin provider routing.

## What This Changes

The PR changes `markAuthProfileSuccess` to skip saving `auth-profiles.json` when a success only updates `lastUsed`, and adds mtime-based regression tests.

## Best Possible Solution

Keep durable auth usage semantics and fix cache invalidation by making the models-json readiness fingerprint ignore `lastUsed`-only churn or by moving volatile usage metadata out of the auth fingerprint path, with coverage for both cache reuse and multi-profile rotation.

## Reproduction Assessment

Yes for source-level reproduction: current main saves `auth-profiles.json` on every successful profile use and the models-json cache fingerprints that file's mtime. I did not run the live multi-provider latency scenario, so the exact runtime latency impact is not independently reproduced here.

## Solution Assessment

No. The PR fixes mtime churn by dropping durable `lastUsed` writes, but current source and tests show `lastUsed` is part of auth-profile rotation; the safer fix is to preserve that state and adjust the cache invalidation boundary.
