## Summary

The report matches the documented group/channel visible-reply policy: Feishu dispatched the turn, but current OpenClaw keeps normal group finals private unless the agent uses the message tool or the operator opts back into automatic group replies.

## What This Changes

The reporter asks why a Feishu group turn for one agent shows `queuedFinal=false, replies=0` after dispatch while DMs and other group sessions reply.

## Best Possible Solution

Keep the existing group visible-reply contract and route this as configuration/session diagnosis: inspect the session transcript for message-tool delivery evidence, then either use a tool-calling model or opt that group back into automatic final replies.

## Reproduction Assessment

Yes for the observable no-final path: current source and tests show group/channel turns default to `message_tool_only`, which produces `queuedFinal=false` and no `sendFinalReply` when the agent does not call the message tool. I did not run a live Feishu account because the issue is explained by the documented dispatch contract.

## Solution Assessment

Yes. The narrow maintainable answer is to use the existing `messages.groupChat.visibleReplies` setting or a more reliable tool-calling model, not to change Feishu transport delivery for one agent/session report without a failing send error.
