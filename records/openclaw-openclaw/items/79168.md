## Summary

Keep open: current main and v2026.6.10 have wrapping, special-token sanitization, logged-only suspicious-pattern detection, and a trusted pre-model tool-result middleware seam, but they still do not provide the requested built-in or official scanner policy with block/warn/redact semantics.

## What This Changes

The issue requests content-based prompt-injection scanning or an officially supported scanner path for tool output before model input.

## Best Possible Solution

Define the security contract first, then implement a trusted tool-result scanner or official reference plugin with documented defaults, severity thresholds, and validation.

## Reproduction Assessment

Yes. Source inspection shows suspicious content is detected and logged while current main lacks a contentScanning/tool_result_filter-style scanner policy or block/warn/redact implementation.

## Solution Assessment

Unclear until maintainers decide the security contract. The trusted tool-result middleware is the likely owner boundary, but default action, severity semantics, config posture, and core-versus-plugin ownership need maintainer review before implementation.
