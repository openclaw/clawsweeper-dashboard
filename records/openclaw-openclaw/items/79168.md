## Summary

Keep open: current main still lacks a configurable pre-model tool-output prompt-injection scanner/blocker, and the latest reporter follow-up correctly frames the remaining work as a security/product contract decision rather than a ready mechanical fix.

## What This Changes

The issue requests opt-in content-based prompt-injection scanning and blocking for tool outputs before those outputs reach the model.

## Best Possible Solution

Have maintainers define the tool-result security boundary, then implement either a trusted opt-in bundled scanner/middleware or a narrow pre-model tool-result gate with documented policy, config, validation, and regression coverage.

## Reproduction Assessment

Not applicable: this is a security-hardening feature request, not a broken existing contract. Source inspection shows current main has wrapping, logging, observation hooks, and bundled-only middleware, but no configurable pre-model tool-output scan/block policy.

## Solution Assessment

Unclear: the requested direction is valid, but the exact implementation needs maintainer security/product judgment. A trusted bundled middleware or narrowly privileged pre-model tool-result gate is safer than relying on observation-only after_tool_call.
