## Summary

Keep open: current main still lacks the requested configurable, pre-model tool-output prompt-injection scanner/blocker; existing wrapping, suspicious-pattern logging, before-tool hooks, after-tool observation, and bundled-only middleware do not close the policy/API gap.

## What This Changes

The issue requests configurable content-based prompt-injection scanning and blocking for tool output before that output is sent back to the model.

## Best Possible Solution

Have maintainers choose the security boundary, then implement either an opt-in trusted scanner/middleware with documented severity policy or a narrow pre-model tool-result gate for vetted plugins.

## Reproduction Assessment

Not applicable: this is a security hardening feature request, not broken existing behavior. Current-main source inspection gives high-confidence evidence that the requested configurable tool-output scan/block policy is absent.

## Solution Assessment

Unclear: the direction is valid, but the exact fix needs security/product judgment. after_tool_call alone cannot block result content; trusted bundled middleware or a narrow pre-model tool-result gate is the safer implementation point.
