## Summary

Keep open: current main and v2026.6.8 have external-content wrapping, a monitor-only suspicious-pattern detector, and a pre-model tool-result middleware seam, but they still do not provide the requested built-in or officially supported block/warn/redact scanner for suspicious tool output.

## What This Changes

The issue requests built-in or officially supported content-based prompt-injection scanning for tool output, with pre-model block, warn, or redact policy instead of monitor-only logging.

## Best Possible Solution

Keep this as the canonical security-design issue, define the tool-result scanning contract, then implement the chosen bundled middleware or vetted plugin/API path with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.8 shows monitor-only suspicious-pattern detection and no built-in or officially supported blocking scanner for suspicious tool output.

## Solution Assessment

No, not as an autonomous patch yet. The best path depends on maintainer security decisions for default action, threshold semantics, false-positive handling, and whether ownership is bundled middleware or a trusted plugin/API surface.
