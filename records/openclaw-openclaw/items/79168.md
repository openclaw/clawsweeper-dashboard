## Summary

Keep open: current main still lacks the requested configurable pre-model tool-output prompt-injection scanner/blocker; existing wrapping, hooks, and bundled-only middleware are useful building blocks but leave a security policy and plugin/API decision unresolved.

## What This Changes

The issue requests built-in or officially supported content-based prompt-injection scanning and blocking for tool output before model submission.

## Best Possible Solution

Have maintainers choose the security boundary, then implement either a trusted opt-in bundled scanner/middleware with documented severity policy or a narrow versioned pre-model tool-result gate for vetted plugins.

## Reproduction Assessment

Not applicable: this is a security hardening feature request, not a broken existing contract. The source-level check is high-confidence: current main has wrappers and observation hooks, but no configurable pre-model tool-output scan/block policy.

## Solution Assessment

Unclear until maintainer design. The requested direction is valid, but after_tool_call is observation-only and the safest implementation point is either trusted bundled middleware or a deliberately designed pre-model tool-result gate.
