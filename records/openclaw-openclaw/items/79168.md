## Summary

Keep open: current main has external-content wrapping, special-token sanitization, hook observation, and bundled-only tool-result middleware, but it still lacks the requested configurable pre-model scan/block policy for tool output.

## What This Changes

Request built-in or officially supported content scanning for tool outputs with pattern matching, Unicode normalization, severity handling, and opt-in configuration.

## Best Possible Solution

Define the trusted tool-result gating boundary first, then ship either an opt-in bundled/reference scanner or a narrow plugin API extension with documented severity behavior.

## Reproduction Assessment

Not applicable: this is a security hardening feature request, not a regression. Source inspection gives high confidence that current main does not provide the requested configurable scanner/blocker.

## Solution Assessment

Unclear: the requested hardening is valid, but after_tool_call alone is not the best enforcement point because it is observation-only. A trusted pre-model middleware or new narrow gating seam would be more maintainable.
