## Summary

Keep open: current main still lacks the requested configurable pre-model tool-output prompt-injection scanner/blocker. Existing wrapping, pattern logging, pre-execution blocking hooks, and bundled-only result middleware are useful, but they leave the security policy and plugin/API boundary unresolved, and the reporter added production context confirming they are waiting for a maintainer-defined contract.

## What This Changes

Request an opt-in content-based scanner for tool outputs that can warn, block, or redact prompt-injection patterns before model submission, or an official reference plugin if built-in scanning is out of scope.

## Best Possible Solution

Have maintainers define the security contract, then implement either a trusted opt-in bundled scanner/middleware with documented severity semantics or a versioned pre-model result gate that can host a reference scanner plugin.

## Reproduction Assessment

Not applicable: this is a security hardening feature request, not a broken existing contract. Source inspection shows the requested configurable pre-model tool-output scan/block policy is absent on current main.

## Solution Assessment

Unclear: the direction is valid, but the exact fix needs maintainer security/product judgment. after_tool_call alone cannot block results, while bundled-only middleware would require either a trusted built-in scanner or a new contract for external scanners.
