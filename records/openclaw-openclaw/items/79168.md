## Summary

Keep open: current main has external-content wrapping, marker/special-token sanitization, and limited suspicious-pattern logging for cron hook payloads, but it does not provide the requested configurable pre-model tool-output scanning or block/warn policy across exec, web, file, browser, and MCP outputs.

## What This Changes

Request adds built-in or official opt-in prompt-injection scanning for tool output, with pattern matching, Unicode normalization, severity handling, and blocking or warning controls.

## Best Possible Solution

Decide the security product boundary first, then either add a trusted opt-in bundled middleware plugin with documented severity policy or expand a narrow plugin API for pre-model tool-result gating before publishing a reference scanner.

## Reproduction Assessment

Not applicable: this is a security feature request, not a reported regression. Source inspection gives high confidence that the requested configurable tool-output scanner/blocker is not present on current main.

## Solution Assessment

Unclear: the requested direction is valid hardening, but after_tool_call alone is not the best implementation point because it is observation-only; the maintainable path is likely trusted tool-result middleware or a new narrow pre-model gating seam.
