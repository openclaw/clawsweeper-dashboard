## Summary

Keep open: current main still lacks the requested configurable pre-model tool-output prompt-injection scanner/blocker. The existing wrapping, suspicious-pattern logging, before/after tool hooks, and bundled-only tool-result middleware are useful, but this remains a security/product policy decision rather than a conservative cleanup close.

## What This Changes

The issue requests built-in or officially supported content-based prompt-injection scanning for tool outputs, with configurable warn/block severity before results reach the model.

## Best Possible Solution

Have maintainers choose the security boundary, then implement either a trusted opt-in bundled scanner/middleware with documented severity policy or a narrow pre-model tool-result gate for vetted plugins.

## Reproduction Assessment

Not applicable: this is a security hardening feature request, not broken existing behavior. Source inspection shows current main does not have the requested configurable tool-output scanner/blocker.

## Solution Assessment

Unclear: the direction is valid, but the exact solution needs maintainer security/product judgment. after_tool_call alone is observation-only; trusted bundled middleware or a narrow pre-model result gate is the maintainable implementation point.
