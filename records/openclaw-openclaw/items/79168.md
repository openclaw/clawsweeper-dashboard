## Summary

Keep open: current main still lacks a configurable pre-model scanner or blocker for prompt-injection patterns in tool output. Existing suspicious-pattern detection, external-content wrapping, `before_tool_call`, observation-only `after_tool_call`, and bundled-only tool-result middleware are useful pieces, but the remaining work is a security/product contract decision rather than a cleanup-close candidate.

## What This Changes

The issue requests configurable content-based prompt-injection scanning or blocking for tool outputs before they reach the model.

## Best Possible Solution

Define the security contract first, then implement either a tested opt-in bundled scanner or a narrow versioned pre-model tool-result gate with documented policy, SDK boundaries, and regression coverage.

## Reproduction Assessment

Not applicable: this is a security hardening feature request, not a broken existing contract. Source inspection on current main verifies that no configurable pre-model tool-output scanner or blocking gate exists.

## Solution Assessment

Unclear: the direction is valid, but the implementation should wait for maintainer security/product decisions. Current bundled middleware is the safer existing pre-model seam; exposing or extending it needs a trust contract rather than a mechanical patch.
