## Summary

Keep open: current main and the latest release have external-content wrapping, suspicious-pattern logging, special-token sanitization, and a trusted pre-model tool-result middleware seam, but they still do not provide the requested built-in or officially supported prompt-injection scanner, severity policy, or block/warn/redact contract.

## What This Changes

Add configurable content-based prompt-injection scanning for tool output before model input, or an official reference plugin using the current tool-result seam.

## Best Possible Solution

Define the security contract first, then implement the scanner as trusted tool-result middleware or an official reference plugin using that seam with documented defaults, severity thresholds, and validation.

## Reproduction Assessment

Yes. Source inspection shows suspicious content is detected and logged, then still wrapped and passed forward, while current main has no contentScanning/tool_result_filter-style scanner policy or block/warn/redact implementation.

## Solution Assessment

Unclear until maintainers decide the security contract. The shipped tool-result middleware is the likely owner boundary, but default action, severity semantics, config posture, and core-versus-plugin ownership need maintainer review before implementation.
