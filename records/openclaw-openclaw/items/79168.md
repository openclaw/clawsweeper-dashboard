## Summary

Keep open: current main still lacks a configurable pre-model tool-output prompt-injection scanner or blocker, and the latest reporter comment confirms this is waiting on a maintainer-defined security contract rather than an implementation PR. The related broad hook PR at https://github.com/openclaw/openclaw/pull/20802 was closed as superseded, while https://github.com/openclaw/openclaw/pull/61485 only covers mutable `llm_input`/`llm_output`, not the tool-result gate requested here.

## What This Changes

The issue requests a configurable content-based prompt-injection scanner or blocking gate for tool output before it reaches the model.

## Best Possible Solution

Have maintainers define the security boundary, then implement either a trusted opt-in bundled tool-result scanner with documented severity policy or a narrow validated pre-model tool-result gate for approved plugins.

## Reproduction Assessment

Not applicable: this is a security hardening feature request, not a broken existing contract. Source inspection still verifies the gap because current main has wrapping/logging and bundled-only middleware, but no configurable tool-output scan/block policy.

## Solution Assessment

Unclear: the requested layer is valid, but the best implementation depends on maintainer security policy. `after_tool_call` alerting alone is insufficient; a trusted bundled middleware or narrow pre-model tool-result gate is the safer direction.
