## Summary

Keep open: current main still lacks the requested configurable, pre-model tool-output prompt-injection scanner or blocker; existing wrapping, logging, hooks, and bundled-only middleware are useful but leave a security contract and API/product decision unresolved.

## What This Changes

Request configurable content-based scanning and possible blocking/redaction of tool outputs before model submission, with a reference plugin path if built-in scanning is out of scope.

## Best Possible Solution

Have maintainers define the security boundary, then implement a trusted opt-in scanner or pre-model result gate with documented severity policy, failure behavior, shape validation, tests, and public docs.

## Reproduction Assessment

Not applicable: this is a security hardening feature request, not a broken existing contract. Source inspection confirms the requested configurable pre-model scanner/blocker is absent while current mitigation is wrapping, logging, and bundled-only middleware.

## Solution Assessment

Unclear: the direction is valid, but the exact fix needs maintainer security/product judgment. after_tool_call is observation-only, and the existing pre-model middleware seam is intentionally bundled-only, so the safest path is a contract decision before code.
