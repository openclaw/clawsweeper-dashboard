## Summary

Keep open: current main still lacks a configurable tool-output prompt-injection scanner or public pre-model tool-result blocking gate, and the remaining work is a security/product contract decision rather than conservative cleanup.

## What This Changes

The issue requests an opt-in content scanner or pre-model gate that can warn, redact, or block prompt-injection patterns in tool output before model submission.

## Best Possible Solution

Have maintainers define the security contract, then implement either a tested opt-in bundled scanner or a versioned pre-model tool-result gate with documented policy, SDK boundaries, and Pi/Codex coverage.

## Risks / Open Questions

- Maintainers still need to choose block vs. warn vs. redact defaults, severity thresholds, and operator configurability.

- False positives, Unicode/encoding evasions, and performance costs make the built-in scanner policy non-mechanical.
