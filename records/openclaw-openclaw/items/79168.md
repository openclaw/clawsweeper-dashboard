## Summary

Keep open: current main still lacks the requested configurable pre-model tool-output prompt-injection scanner/blocker. Existing external-content wrapping, suspicious-pattern logging, special-token sanitization, and bundled-only tool-result middleware are useful hardening, but the remaining security contract is a maintainer decision.

## What This Changes

Request built-in or officially supported content-based prompt-injection scanning for tool outputs, with configurable severity and an opt-in block/warn/redact policy.

## Best Possible Solution

Have maintainers define the security boundary, then implement a documented opt-in scanner as trusted bundled middleware or a narrow pre-model tool-result gate with validation, severity semantics, and tests.

## Reproduction Assessment

Not applicable: this is a security hardening feature request, not a broken existing contract. Source inspection shows the requested configurable tool-output scan/block policy is still absent on current main.

## Solution Assessment

Unclear: the requested direction is valid, but the exact solution needs maintainer security/product judgment. after_tool_call alone is observation-only; trusted bundled middleware or a purpose-built pre-model tool-result gate is the safer implementation point.
