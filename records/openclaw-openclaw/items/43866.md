## Label Justifications

- `P1`: The server hook path can return success for a multi-account delivery request while losing the account selector needed to deliver the message.
- `impact:message-loss`: The reported failure mode is an accepted hook request whose outbound message is not delivered.
- `impact:auth-provider`: The missing `accountId` controls which channel account and credentials are used for outbound delivery.

## Summary

Keep this open: current main still drops `accountId` from `/hooks/agent` before cron delivery, and the active closing PR is open but conflicting rather than merged.

## What This Changes

The issue asks for `/hooks/agent` to preserve `accountId` into cron delivery so multi-account channels can route outbound messages correctly.

## Best Possible Solution

Land the linked PR or a fresh replacement that threads trimmed `accountId` through hook normalization, replay scope, `CronDelivery`, docs, and focused regression tests while preserving current sanitization and source-delivery behavior.

## Reproduction Assessment

Yes, at source level: a `/hooks/agent` payload can include `accountId`, but current normalization and dispatch omit it before constructing `CronDelivery`. I did not run a live WhatsApp multi-account setup in this read-only review.

## Solution Assessment

Yes in direction, but it is not solved on main. The maintainable fix is to preserve trimmed `accountId` through the hook-to-cron contract and keep the existing sanitized-name, replay, source-delivery, docs, and regression-test coverage intact.
