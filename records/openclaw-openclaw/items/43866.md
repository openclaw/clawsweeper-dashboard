- `P1`: The issue reports a broken hook-to-channel workflow where callers can receive success while multi-account delivery is not sent.

## Summary

Keep open: current main and v2026.6.10 still omit `/hooks/agent` `accountId` before cron delivery resolution, and the open linked PR is the exact implementation candidate.

## What This Changes

The issue requests preserving flat `accountId` from `POST /hooks/agent` into `CronDelivery` so multi-account channel delivery can resolve the intended account.

## Best Possible Solution

Land or replace the focused linked PR so `/hooks/agent` reuses canonical cron delivery normalization, preserves `accountId`, and includes regression coverage plus redacted real multi-account delivery proof.

## Reproduction Assessment

Yes, at source level: current main’s hook normalizer omits `accountId`, dispatch builds delivery with only `mode`/`channel`/`to`, and the handler returns a successful run id after dispatch. I did not run a live WhatsApp setup in this read-only review.

## Solution Assessment

Yes, reusing the canonical cron delivery normalization path is the narrow maintainable fix direction. This issue should remain open until the linked PR lands or is replaced by an equivalent focused fix.
