## Summary

Keep this issue open: current main still drops `accountId` on the `/hooks/agent` path, and there is an open linked fix PR that uses closing syntax but has not merged.

## What This Changes

Preserve `/hooks/agent` `accountId` through hook normalization and dispatch so multi-account channel delivery can resolve the intended account.

## Best Possible Solution

Land a narrow fix that parses and trims hook `accountId`, includes it in replay scope and dispatch, carries it into `CronDelivery.accountId`, documents the hook trust contract, and preserves current cron announce tool policy.

## Reproduction Assessment

Yes. Source inspection on current main shows `/hooks/agent` normalization and dispatch omit `accountId`, so a payload containing it cannot reach `resolveDeliveryTarget()` through `CronDelivery.accountId`.

## Solution Assessment

Yes in direction, but the open implementation needs maintainer review before landing. The maintainable fix is to repair the hook-to-cron boundary and documentation without regressing the current announce message-tool policy.
