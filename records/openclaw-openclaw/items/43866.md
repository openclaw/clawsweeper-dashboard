## Label Justifications

- `P1`: The hook endpoint can accept a multi-account delivery request and return success while losing the account selector needed for delivery.
- `impact:message-loss`: The reported and source-supported failure mode is an accepted hook run whose outbound channel message is not delivered.
- `impact:auth-provider`: The missing `accountId` selects the channel account and credentials used for multi-account outbound delivery.

## Summary

Keep this open: current main still discards `accountId` before `/hooks/agent` can construct cron delivery, and the linked closing PR is still open and conflicting rather than merged.

## What This Changes

The issue requests preserving `/hooks/agent` `accountId` through hook normalization, replay scoping, cron delivery construction, and multi-account delivery resolution.

## Best Possible Solution

A single merged fix should add trimmed `accountId` to the hook payload, mapping, replay, and cron delivery boundary, with regression coverage for direct and mapped hooks plus updated user-facing docs.

## Reproduction Assessment

Yes, at source level: `normalizeAgentPayload()` and `dispatchAgentHook()` omit `accountId` even though downstream cron delivery supports it. I did not run a live WhatsApp multi-account setup in this read-only review.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix: preserve the existing cron account routing contract by feeding it from the hook boundary. The issue should remain open until the linked PR or an equivalent replacement is merged.
