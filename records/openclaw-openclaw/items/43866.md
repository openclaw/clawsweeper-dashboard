## Summary

Keep this open: current main still drops the hook payload accountId before cron delivery, and an open closing PR already tracks the repair but has not landed.

## What This Changes

The issue asks for /hooks/agent to preserve accountId through hook dispatch into CronDelivery for multi-account channel delivery.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/43951 or a narrow replacement that threads a trimmed accountId through the active hook handler, replay scope, and CronDelivery while preserving hook-name sanitization and current cron message-tool policy.

## Reproduction Assessment

Yes. Source inspection on current main is enough: normalizeAgentPayload never reads accountId and dispatchAgentHook builds delivery with only mode, channel, and to, so a hook request with accountId cannot produce job.delivery.accountId.

## Solution Assessment

Yes in direction, but not yet landed. The maintainable fix is to carry accountId through the active hook request handler and cron delivery contract, with focused tests and docs; the existing PR needs rebase and proof before merge.
