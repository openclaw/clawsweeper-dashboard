## Summary

This issue is still necessary: current main and v2026.6.8 still drop flat `/hooks/agent` `accountId` before cron delivery resolution. The linked implementation candidate at https://github.com/openclaw/openclaw/pull/43951 is open and explicitly targets this issue, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests preserving `/hooks/agent` `accountId` through hook payload normalization and cron delivery dispatch for multi-account channel routing.

## Best Possible Solution

Land or replace the focused linked PR so `/hooks/agent` reuses canonical cron delivery normalization, preserves `accountId`, and has regression coverage plus real behavior proof for multi-account delivery.

## Reproduction Assessment

Yes. Source inspection on current main shows `normalizeAgentPayload` does not preserve `accountId`, `dispatchAgentHook` builds delivery without it, and the HTTP handler still returns a successful run id after dispatch; I did not run a live WhatsApp scenario in this read-only review.

## Solution Assessment

Yes. Reusing the canonical cron delivery normalization and target-resolution path is the narrow maintainable fix direction; the open linked PR still needs normal review and proof before this issue can close.
