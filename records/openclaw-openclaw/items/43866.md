## Label Justifications

- `P1`: The issue reports a broken hook-to-channel workflow where callers can receive success while multi-account delivery is not sent.
- `impact:message-loss`: The central failure mode is silent non-delivery after `/hooks/agent` accepts the request.
- `impact:auth-provider`: The missing `accountId` affects multi-account provider/channel routing and credential selection.

## Summary

This issue is still necessary: current `main` and the latest release both keep the `/hooks/agent` path on a hook-specific payload/job builder that does not preserve `accountId`. It should stay open because https://github.com/openclaw/openclaw/pull/43951 is an open linked implementation candidate that explicitly closes it, but that PR has not landed.

## What This Changes

`POST /hooks/agent` should preserve `accountId` through hook normalization and cron-style delivery dispatch so multi-account channel delivery resolves the intended account.

## Best Possible Solution

Land or replace the linked focused fix so `/hooks/agent` uses the same canonical cron delivery normalization and account-target resolution path as cron jobs, with regression coverage for preserved `accountId` and duplicate-delivery behavior.

## Reproduction Assessment

Yes from source inspection: current `main` still drops `accountId` in `normalizeAgentPayload` and constructs the hook `CronJob` without the cron delivery normalizer that preserves `delivery.accountId`. I did not run a live WhatsApp hook scenario in this read-only review.

## Solution Assessment

Yes, the requested fix direction is the narrow maintainable shape: reuse the canonical cron normalization and delivery-target path instead of adding a separate hook-only account shim. The open linked PR still needs normal PR review and behavior proof before this issue can close.
