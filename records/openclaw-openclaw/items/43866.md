## Summary

Keep this issue open: current `main` and the latest release still drop `/hooks/agent` `accountId`, and the linked closing PR is open, unmerged, and currently conflicting.

## What This Changes

The issue requests preserving `/hooks/agent` `accountId` through hook payload normalization and cron delivery dispatch so multi-account channels can resolve the correct account.

## Best Possible Solution

Land or replace the linked hook normalization fix so `accountId` reaches `CronDelivery.accountId` with focused regression coverage for hook payload normalization, dispatch, and delivery/tool policy.

## Reproduction Assessment

Yes, at source level: `normalizeAgentPayload()` never reads `accountId`, and `dispatchAgentHook()` builds cron delivery without it. I did not run a live multi-account WhatsApp repro in this read-only review.

## Solution Assessment

Yes, the maintainable direction is to carry hook dispatch through the existing cron delivery contract instead of adding a separate hook-only routing path. The open linked PR is the current implementation candidate, but it needs conflict repair and review before this issue can close.
