## Label Justifications

- `P1`: The hook endpoint can accept a multi-account delivery request and return success while losing the selector needed to deliver the message.
- `impact:message-loss`: The source-supported failure mode is an accepted hook run whose outbound channel message is not delivered.
- `impact:auth-provider`: The missing `accountId` selects the channel account and credentials used for multi-account outbound delivery.

## Summary

Keep this open: current main and the latest release still omit `accountId` from the `/hooks/agent` payload and dispatch path, while downstream cron delivery already supports the field; https://github.com/openclaw/openclaw/pull/43951 is still open and conflicting rather than merged.

## What This Changes

The issue requests preserving `accountId` from `/hooks/agent` payloads through hook normalization, dispatch, replay scoping, and cron delivery so multi-account channels can deliver via the selected account.

## Best Possible Solution

Review and land a rebased https://github.com/openclaw/openclaw/pull/43951, or a narrow replacement that threads trimmed `accountId` through hook normalization, dispatch, replay scope, cron delivery construction, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level: `normalizeAgentPayload()` and `dispatchAgentHook()` omit `accountId` while cron delivery already supports it. I did not run a live WhatsApp multi-account setup in this read-only review.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: feed the existing cron account-routing contract from the hook boundary. This issue should stay open until the linked PR or an equivalent replacement merges.
