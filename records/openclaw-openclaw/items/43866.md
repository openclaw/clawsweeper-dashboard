## Summary

Keep open. Current main still drops direct `/hooks/agent` `accountId` before cron delivery, and https://github.com/openclaw/openclaw/pull/43951 is an open closing PR that should be reviewed, landed, or replaced rather than closing this issue early.

## What This Changes

The issue requests that `/hooks/agent` preserve `accountId` through hook normalization, replay scoping, and cron delivery so multi-account channel dispatch can target the correct account.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/43951 with a narrow gateway fix that threads `accountId` through payload normalization, replay scoping, dispatch, tests, and docs while preserving cron-owned account resolution and current cron tool policy.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a `/hooks/agent` body with `accountId` reaches `normalizeAgentPayload()`, which never reads it, and `dispatchAgentHook()` builds `CronJob.delivery` without it even though cron delivery supports `accountId`.

## Solution Assessment

Yes in direction, but not closed yet. The best fix is the narrow generic hook-to-cron `accountId` propagation already represented by the open closing PR, with replay/docs/tests included and unrelated cron policy changes avoided.
