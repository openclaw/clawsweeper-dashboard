## Label Justifications

- `P1`: The hook endpoint can accept a multi-account delivery request and return success while losing the account selector required for delivery.
- `impact:message-loss`: The reported and source-supported failure mode is an accepted hook run whose outbound channel message is not delivered.
- `impact:auth-provider`: The missing `accountId` selects the channel account and credentials used for multi-account outbound delivery.

## Summary

Keep this open: current `main` still drops `accountId` before `/hooks/agent` can build cron delivery, and the linked closing PR at https://github.com/openclaw/openclaw/pull/43951 is still open and conflicting rather than merged.

## What This Changes

The issue requests preserving `accountId` from `/hooks/agent` and mapped hook payloads through cron delivery so multi-account channel delivery selects the intended account.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/43951 with a narrow fix that threads trimmed `accountId` through direct hooks, mapped hooks, replay scope, and `CronDelivery`, with focused regression coverage and documented hook fields.

## Reproduction Assessment

Yes, at source level: `normalizeAgentPayload()` and `dispatchAgentHook()` omit `accountId` even though downstream cron delivery supports it. I did not run a live WhatsApp multi-account setup in this read-only review.

## Solution Assessment

Yes, the narrow maintainable solution is to feed the existing cron account routing contract from the hook boundary. The issue should remain open until the linked PR or an equivalent replacement is merged.
