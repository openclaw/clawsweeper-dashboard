## Summary

Keep this open: current main still cannot carry request `accountId` through `/hooks/agent`, and the closing PR is open with review/proof blockers rather than merged.

## What This Changes

The issue asks for `/hooks/agent` to preserve `accountId` through hook normalization and dispatch into cron delivery for multi-account channel routing.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/43951 or a narrow rebased replacement that threads a trimmed `accountId` through hook payload normalization, replay scope, and `CronDelivery` while preserving hook-name sanitization, current cron message-tool policy, and docs.

## Reproduction Assessment

Yes. Source inspection on current main is enough: `normalizeAgentPayload()` never reads `accountId`, and hook-created delivery only carries `mode`, `channel`, and `to`, so a hook request with `accountId` cannot produce `job.delivery.accountId`.

## Solution Assessment

Yes in direction, but it is not landed. The maintainable fix is to carry `accountId` through the existing hook and cron delivery contracts, with docs and real multi-account proof before merge.
