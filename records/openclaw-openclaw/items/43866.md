## Summary

Keep open. Current main still drops `accountId` before hook-triggered cron delivery, and open PR #43951 explicitly closes this issue, so this should stay open until maintainers land or replace that PR.

## What This Changes

Preserve `/hooks/agent` `accountId` through hook normalization, replay scoping, and cron delivery dispatch so multi-account channel delivery resolves the intended account.

## Best Possible Solution

Land or replace PR #43951 with a narrow gateway-owned fix that normalizes, replays, dispatches, tests, and documents `accountId` for `/hooks/agent` while leaving account-aware delivery resolution in cron.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a `/hooks/agent` body containing `accountId` reaches `normalizeAgentPayload()`, the field is never read, and `dispatchAgentHook()` creates `job.delivery` without it even though cron delivery can carry `accountId`.

## Solution Assessment

Yes in direction. The narrow maintainable fix is generic hook-to-cron account propagation plus replay/tests/docs; a WhatsApp-specific workaround would leave the shared hook contract broken, while PR #43951 still needs maintainer review.
