## Label Justifications

- `P1`: The hook endpoint can accept a multi-account delivery request and return success while losing the selector needed to deliver the message.
- `impact:message-loss`: The source-supported failure mode is an accepted hook run whose outbound channel message is not delivered.
- `impact:auth-provider`: The missing `accountId` selects the channel account and credentials used for multi-account outbound delivery.

## Summary

Keep this open: current main still drops `accountId` at the `/hooks/agent` boundary before cron delivery can use it, and the open closing PR is conflicting rather than merged.

## What This Changes

The issue asks OpenClaw to preserve `/hooks/agent` `accountId` through hook normalization, replay scoping, and cron delivery so multi-account channel delivery can target the right account.

## Best Possible Solution

Land a rebased fix or narrow replacement that threads trimmed `accountId` through direct and mapped hook payload normalization, replay scope, CronDelivery construction, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: `normalizeAgentPayload()` and `dispatchAgentHook()` omit `accountId` while cron delivery already supports it. I did not run a live WhatsApp multi-account setup in this read-only review.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix: feed the existing cron account routing contract from the hook boundary. The issue should stay open until https://github.com/openclaw/openclaw/pull/43951 or an equivalent replacement merges.
