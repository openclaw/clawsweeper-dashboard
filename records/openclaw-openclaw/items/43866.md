## Summary

Keep open. Current main still has the core bug: direct `/hooks/agent` normalization and dispatch do not preserve `accountId`, while cron delivery downstream already supports it. The issue is also paired with open closing PR #43951, so this should remain open until maintainers review, land, or replace that PR.

## What This Changes

The issue requests preserving `accountId` from `/hooks/agent` payloads through hook normalization and cron delivery dispatch so multi-account channel delivery can resolve the intended account.

## Best Possible Solution

Land or replace PR #43951 with a narrow gateway-owned fix that normalizes, trims, replays, dispatches, and documents `accountId` for `/hooks/agent` without adding channel-specific WhatsApp routing logic; cron should remain the owner of account-aware delivery resolution.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a `/hooks/agent` body containing `accountId` reaches `normalizeAgentPayload()`, the field is never read, and `dispatchAgentHook()` creates `job.delivery` without it even though cron delivery can carry `accountId`.

## Solution Assessment

Yes in direction. The maintainable solution is generic hook-to-cron account propagation plus replay/docs/tests; a WhatsApp-only workaround would leave the shared hook contract broken, while the exact paired PR should still be reviewed for accepted hook shapes and replay scope.
