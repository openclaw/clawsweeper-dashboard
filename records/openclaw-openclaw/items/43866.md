## Label Justifications

- `P1`: The hook endpoint can accept a multi-account delivery request and return success while losing the selector needed to deliver the message.
- `impact:message-loss`: The reported and source-supported failure mode is an accepted hook run whose outbound channel message is not delivered.
- `impact:auth-provider`: The missing `accountId` selects the channel account and credentials used for multi-account outbound delivery.

## Summary

Keep this open: current main still drops `accountId` at the `/hooks/agent` boundary before cron delivery can use it, and the linked closing PR remains open and conflicting rather than merged.

## What This Changes

The issue reports that `/hooks/agent` should preserve `accountId` into cron delivery so multi-account channel routing can select the correct account.

## Best Possible Solution

Land one narrow fix that threads trimmed `accountId` through direct and mapped hook payload normalization, replay scope, and cron delivery construction, with regression coverage and docs for the hook contract.

## Reproduction Assessment

Yes, at source level: `normalizeAgentPayload()` and `dispatchAgentHook()` omit `accountId` while cron delivery already supports it. I did not run a live WhatsApp multi-account setup in this read-only review.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix: preserve the existing cron account routing contract by feeding it from the hook boundary. The issue should remain open until the linked PR or an equivalent replacement is merged.
