## Summary

Keep this open: current main still drops `accountId` from `/hooks/agent` before cron delivery, and the live closing PR has not merged.

## What This Changes

`POST /hooks/agent` should preserve a request `accountId` into `CronDelivery.accountId` so multi-account channel delivery can resolve the intended account.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/43951 or a narrow replacement that threads a trimmed `accountId` through the active hook handler, replay scope, and cron delivery while preserving hook-name sanitization and current cron message-tool policy.

## Reproduction Assessment

Yes. Source inspection on current main is enough: `normalizeAgentPayload()` never reads `accountId`, and hook-created delivery only carries `mode`, `channel`, and `to`, so a request `accountId` cannot reach `job.delivery.accountId`.

## Solution Assessment

Yes in direction, but not yet landed. The maintainable fix is to carry `accountId` through the active hook request path into the existing cron delivery contract, with docs and real multi-account proof before merge.
