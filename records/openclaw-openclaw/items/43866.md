## Summary

Keep open. Current main still drops a direct `/hooks/agent` `accountId` before building cron delivery, and open PR #43951 explicitly closes this issue, so the issue should stay open until that PR or a replacement lands.

## What This Changes

The issue requests preserving `accountId` from direct `/hooks/agent` payloads into cron delivery so multi-account channel routing can select the intended account.

## Best Possible Solution

Land or replace PR #43951 with a narrow gateway-owned fix that threads `accountId` through hook normalization, replay scoping, dispatch, tests, and docs while preserving cron-owned account resolution.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a `/hooks/agent` body with `accountId` reaches `normalizeAgentPayload()`, which never reads it, and `dispatchAgentHook()` builds `CronJob.delivery` without it even though cron delivery supports `accountId`.

## Solution Assessment

Yes in direction. The narrow maintainable solution is generic hook-to-cron `accountId` propagation with replay/tests/docs, not a WhatsApp-specific workaround; the issue should remain open until PR #43951 or a replacement lands.
