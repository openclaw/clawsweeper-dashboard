## Summary

Keep this open: current main still drops `accountId` on the `/hooks/agent` ingress path before cron delivery can route a multi-account channel, while the linked closing PR remains open and unmerged.

## What This Changes

The issue asks for `/hooks/agent` to preserve a request `accountId` through hook normalization, replay scope, and cron delivery so multi-account channel sends route to the requested account.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/43951, or a narrow rebased replacement, that threads trimmed `accountId` through hook normalization, replay scope, cron delivery, tests, and public hook docs while preserving sanitized hook names and current cron message-tool policy.

## Reproduction Assessment

Yes. Source inspection on current main shows `/hooks/agent` never normalizes `payload.accountId`, hook-created delivery only carries `mode`, `channel`, and `to`, and cron can only route by `accountId` when that field reaches `CronDelivery`.

## Solution Assessment

Yes in direction, but not complete yet. The maintainable fix is a narrow hook-to-cron contract update with replay-key, tests, and docs coverage; this issue should stay open until the linked PR or a replacement merges.
