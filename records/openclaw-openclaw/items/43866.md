## Summary

Keep open. Current main still omits `accountId` from the direct `/hooks/agent` normalization and dispatch path, and open PR #43951 explicitly closes this issue, so the issue should wait for that PR or a replacement instead of being closed early.

## What This Changes

The issue asks for `/hooks/agent` to preserve an incoming `accountId` through hook dispatch into cron delivery for multi-account channel routing.

## Best Possible Solution

Land or replace PR #43951 with a narrow gateway-owned fix that threads `accountId` through hook normalization, replay scoping, dispatch, tests, and docs while preserving cron-owned account resolution.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: a `/hooks/agent` body with `accountId` reaches `normalizeAgentPayload()`, which never reads it, and `dispatchAgentHook()` builds `CronJob.delivery` without it even though cron delivery supports `accountId`.

## Solution Assessment

Yes in direction. The narrow maintainable solution is generic hook-to-cron `accountId` propagation with replay/tests/docs, not a WhatsApp-specific workaround; the issue should stay open until PR #43951 or a replacement lands.
