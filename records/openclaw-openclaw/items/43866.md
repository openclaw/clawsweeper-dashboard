## Summary

Keep open. Current main still drops a direct `/hooks/agent` `accountId` before cron delivery, and open PR #43951 explicitly closes this issue, so the right path is to review, land, or replace that PR rather than close this early.

## What This Changes

This issue requests that `/hooks/agent` preserve `accountId` into cron delivery for multi-account channel routing.

## Best Possible Solution

Land or replace PR #43951 with a narrow gateway fix that threads `accountId` through hook normalization, replay scoping, dispatch, tests, and docs while preserving the cron-owned account resolution contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a `/hooks/agent` body with `accountId` reaches `normalizeAgentPayload()`, which never reads it, and `dispatchAgentHook()` builds `CronJob.delivery` without it even though cron delivery supports `accountId`.

## Solution Assessment

Yes in direction. The narrow maintainable fix is generic hook-to-cron `accountId` propagation with replay/docs/tests, not WhatsApp-specific logic; PR #43951 should be reviewed against current main before landing.
