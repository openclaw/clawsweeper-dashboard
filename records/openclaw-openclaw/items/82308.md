## Summary

Keep this issue open because current main still has the redundant discovery call pattern, and the issue is explicitly paired with an open implementation PR that closes it. The right cleanup path is to review and land or reject that linked PR rather than closing the issue early.

## What This Changes

The issue requests avoiding repeated plugin discovery filesystem walks during startup by threading a function-scoped discovery result through plugin subsystem calls.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/75451 as the implementation candidate, preserving stateless discovery while allowing callers in the same startup flow to reuse an explicit discovery result.

## Reproduction Assessment

Yes, at source level: current main shows the named helpers still call `discoverOpenClawPlugins` independently, and the retry loop can invoke that path twice. I did not run a runtime benchmark because this review is read-only.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix: thread an explicit discovery result through the affected functions while leaving `discoverOpenClawPlugins` stateless. The linked PR should be reviewed for correctness and proof before the issue is closed.
