## Summary

Keep this issue open because current main still has the redundant plugin discovery call pattern, and the issue is paired with the open implementation PR at https://github.com/openclaw/openclaw/pull/75451. The right cleanup path is to review, land, or reject that PR rather than closing the issue before its implementation is merged.

## What This Changes

The issue asks to avoid redundant startup filesystem walks by threading an explicit `PluginDiscoveryResult` through plugin discovery callers while keeping `discoverOpenClawPlugins()` stateless.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/75451 as the implementation candidate, preserving stateless discovery while allowing callers in the same startup flow to reuse an explicit discovery result.

## Reproduction Assessment

Yes, at source level: current main still shows the named helpers independently invoking `discoverOpenClawPlugins`, and the contracts retry loop can re-enter that path. I did not run a runtime benchmark because this review is read-only.

## Solution Assessment

Yes, the explicit function-scoped discovery threading proposed by the linked PR matches the plugin cache boundary and avoids a hidden global cache. The implementation should be reviewed and merged or rejected before this issue is closed.
