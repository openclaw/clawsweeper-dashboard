## Best Possible Solution

Yes for the PR defects from source inspection: current main forwards abort/accepted callbacks and stops each client, while the PR returns pooled clients with one-shot callbacks and no teardown. No live upgraded doctor reproduction or after-fix proof was provided for the original regression.

No; this is not the best way to solve the issue because a global `GatewayClient` pool does not preserve the existing connection, auth, and request lifecycle contract. A scoped reusable transport or caller-level throttling fix with tests and live proof would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Do not reuse a settled one-shot GatewayClient:** `src/gateway/call.ts:58-59`
  - body: `GatewayClient` stores `onHelloOk` in constructor options, so returning an existing live client reuses the callback closure from the first call. A later same-key `callGateway` promise gets no fresh request callback; `start()` will open another socket on the same instance and the new wrapper can only time out instead of sending its method.
  - confidence: 0.94
- **[P1] Include the full gateway context in the pool key:** `src/gateway/call.ts:53-54`
  - body: The key only includes method/token/password, but the client is constructed with URL, TLS fingerprint, scopes, device identity, approval runtime token, client mode/name, instance id, and protocol bounds. Reusing across any of those differences can route a call through the wrong target or authorization context.
  - confidence: 0.93
- **[P1] Preserve abort and accepted callbacks:** `src/gateway/call.ts:711-714`
  - body: Current main forwards `signal` and `onAccepted` into `GatewayClient.request`, and `agent-via-gateway` relies on that to track accepted runs and abort them on process signals. The PR drops those options, so interrupted accepted gateway runs can keep running.
  - confidence: 0.91
- **[P1] Do not leave per-call CLI clients open indefinitely:** `src/gateway/call.ts:74-79`
  - body: `releasePooledClient` no longer stops the client after the wrapper promise settles. Current main waits for teardown before resolving; with this no-op release, one-shot CLI calls can leave WebSockets, tick timers, and reconnect loops alive after the command should be done.
  - confidence: 0.9
- **[P2] Actually enforce the pool size cap:** `src/gateway/call.ts:62-71`
  - body: When the map is at capacity and all entries are still live, this loop deletes nothing and `clientPool.set()` adds another key anyway. Diverse method/auth combinations can therefore grow past the advertised cap.
  - confidence: 0.88
