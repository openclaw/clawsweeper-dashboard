## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Include the full gateway context in the pool key:** `src/gateway/call.ts:52`
  - body: Pooling by only method/token/password can reuse a client that was created for a different URL, TLS fingerprint, scopes, device identity, approval runtime token, client mode, or protocol bounds. A later call can then authenticate or route through the wrong gateway context, so this needs either a complete key or no cross-call client reuse.
  - confidence: 0.93
- **[P1] Do not restart a live pooled client:** `src/gateway/call.ts:58-59`
  - body: `acquirePooledClient` can return an existing live client, but the call path still invokes `startGatewayClientWhenEventLoopReady(client)`. `GatewayClient.start()` opens a new WebSocket without closing an existing one, so repeated calls can still create overlapping sockets and orphan the previous connection.
  - confidence: 0.91
- **[P1] Cancel stale release timers before reusing clients:** `src/gateway/call.ts:66-76`
  - body: Every release schedules a new 10s cleanup timer, but reuse does not clear or extend older timers. A timer from an earlier call can delete and stop the same client while a later request is in flight, reintroducing connection churn and request failures.
  - confidence: 0.89
- **[P1] Keep one-shot gateway calls from leaving sockets open:** `src/gateway/call.ts:717`
  - body: The old path waited for `stopGatewayClient` before resolving the request. This change resolves immediately and leaves the WebSocket alive until the pool TTL, so short-lived CLI commands can remain running for the TTL after a successful gateway call.
  - confidence: 0.86
- **[P2] Remove the unused PoolKey alias:** `src/gateway/call.ts:51`
  - body: `PoolKey` is declared but never used in production code. Core typechecking enables `noUnusedLocals`, so this will fail the prod tsgo lane even before runtime behavior is considered.
  - confidence: 0.88
