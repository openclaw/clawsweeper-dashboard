## Summary

Keep open: this is a plausible focused bug fix for a source-reproducible cron hot-reload race, and the linked stale issue should be resolved by reviewing this implementation candidate rather than closing the PR. It should not merge yet because the external PR still lacks real behavior proof and has two current-main correctness blockers.

## What This Changes

The PR adds `CronService.stopGraceful()`, tracks in-flight timer/manual cron runs, switches gateway hot reload to await the graceful stop, and adds cron regression tests.

## Best Possible Solution

Land a rebased graceful cron replacement that updates the contract/lazy/unavailable implementations, bounds the drain with a fallback warning, preserves regression coverage, and includes redacted real `config.patch` proof.

## Reproduction Assessment

Yes, source-reproducible: current main stops and immediately rebuilds cron while timer/manual cron execution can run outside `locked()` and later re-enter it to persist. I did not run a live `config.patch` reproduction in this read-only review.

## Solution Assessment

No, not yet. Graceful draining is the right direction, but this branch needs a current-main contract/lazy-wrapper rebase and a bounded drain before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Add graceful stop to the cron contract:** `src/gateway/server-reload-handlers.ts:137`
  - body: Current main routes gateway cron through `CronServiceContract` and may install the lazy or unavailable cron implementation, but this patch only adds `stopGraceful()` to the concrete service before calling it from hot reload. Rebase the method through the contract and every implementation so this call is valid on all gateway cron states.
  - confidence: 0.9
- **[P2] Bound the graceful drain before restarting cron:** `src/cron/service/ops.ts:179`
  - body: `stopGraceful()` awaits every tracked run with no deadline before cron hot reload can rebuild the service. Because cron agent turns can explicitly disable the watchdog with `timeoutSeconds <= 0`, one stuck run can wedge config reload forever; add a bounded wait and fallback warning before allowing reload to continue.
  - confidence: 0.88
