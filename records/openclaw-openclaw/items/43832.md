## Summary

## What This Changes

The PR adds a graceful cron stop path, tracks in-flight cron runs, and awaits that drain during gateway hot reload before replacing the cron service.

## Best Possible Solution

Land a current-main-compatible graceful cron replacement path that preserves persisted job state without unbounded reload stalls, with focused regression tests and redacted real `config.patch` hot-reload proof.

## Reproduction Assessment

Yes, source-reproducible: current main stops and rebuilds cron synchronously during hot reload while timer and manual cron executions can still perform final persists after outside-lock worker phases. I did not run a live `config.patch` repro because this review is read-only.

## Solution Assessment

No. The graceful-drain direction is plausible, but this PR is not yet the best mergeable solution because it must be rebased through the current cron contract/lazy facade, bound the drain, and show real hot-reload behavior proof.
