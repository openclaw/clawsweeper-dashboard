## Summary

Keep open: current main still has the cron hot-reload stale-write path, but this PR is not mergeable as-is because the new graceful stop is not exposed through the cron contract/lazy implementation, the reload wait is unbounded, and the external PR lacks real after-fix behavior proof.

## What This Changes

The branch adds a graceful cron stop path that tracks in-flight timer/manual runs, awaits it during cron hot reload, and adds regression tests.

## Best Possible Solution

Land a corrected cron lifecycle fix that drains old-service persists before replacement, exposes the graceful stop through every cron contract implementation, bounds the hot-reload wait, and keeps focused regression coverage.

## Reproduction Assessment

Yes, from source inspection. Current main stops cron synchronously during hot reload while timer/manual cron executions can continue outside the lock and later persist through the old service.

## Solution Assessment

No, not as-is. Graceful draining is the right boundary, but this PR needs the contract/lazy surface and a bounded reload wait before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Expose graceful stop on the cron contract:** `src/gateway/server-reload-handlers.ts:137`
  - body: `state.cronState.cron` is a `CronServiceContract`, but the PR only adds `stopGraceful()` to the concrete `CronService`. The hot-reload call will not be valid through the contract and lazy cron wrapper until those surfaces expose the new method too.
  - confidence: 0.94
- **[P2] Bound the graceful cron drain before reload:** `src/gateway/server-reload-handlers.ts:137`
  - body: The hot-reload path now awaits a drain that waits on tracked run promises without a deadline. Since agent-turn cron jobs can configure `timeoutSeconds <= 0` and run without the outer timeout, one stuck run can keep `cron.*` config changes from ever applying.
  - confidence: 0.9
