## Summary

Keep open: current main still has the cron hot-reload race, but this PR is not mergeable as-is because it misses the current cron service contract/lazy wrapper, can block hot reload indefinitely, and lacks external real behavior proof.

## What This Changes

The PR adds `CronService.stopGraceful`, tracks in-flight cron runs, switches hot cron reload to await it, and adds regression tests for locked, manual, and timer-run drains.

## Best Possible Solution

Keep the fix direction, but rebase it onto current main, add the graceful-stop method through the cron contract and lazy cron wrapper, bound the drain, and require focused regression plus real hot-reload proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main's hot reload calls synchronous `cron.stop()`, while timer and manual cron runs can execute outside `locked()` and later persist. I did not run a live `config.patch` scenario because this was a read-only review.

## Solution Assessment

No, not yet. The overall approach is pointed at the right race, but the current PR needs contract/lazy-wrapper updates, a bounded drain behavior, and real after-fix proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Add stopGraceful to the cron contract boundary:** `src/gateway/server-reload-handlers.ts:137`
  - body: `state.cronState.cron` is still a `CronServiceContract` on current main, and that contract plus the lazy cron wrapper expose only `stop()`. After rebasing, this call either fails typechecking or reaches a lazy cron object without `stopGraceful()`, so add the method to `CronServiceContract` and implement it in `createLazyGatewayCronState` and the affected test doubles before switching reload to it.
  - confidence: 0.93
- **[P2] Bound the graceful-stop drain before hot reload:** `src/cron/service/ops.ts:177`
  - body: `stopGraceful()` awaits every tracked run with no deadline, and hot reload awaits this before rebuilding cron. Cron agent jobs can explicitly disable their timeout with `timeoutSeconds <= 0`, so a hung/no-timeout run can now prevent cron config changes from applying indefinitely; use a bounded drain or fallback that still prevents post-reload stale writes.
  - confidence: 0.86
