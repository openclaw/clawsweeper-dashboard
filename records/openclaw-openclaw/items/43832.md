## Summary

Keep open: the PR addresses a real cron hot-reload state race and incorporates useful drain coverage, but the current diff still needs a bounded graceful-stop fallback and the required changelog entry before merge.

## What This Changes

The PR adds `CronService.stopGraceful()`, tracks in-flight cron timer/manual run promises, awaits graceful stop during cron hot reload, and adds regression tests.

## Best Possible Solution

Land a rebased cron-owned fix that drains old cron writes before replacement, applies the existing gateway reload timeout/fallback policy, preserves `jobs.json` semantics, and includes the user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main stops cron synchronously while timer and manual cron executions can still persist after outside-lock work; the PR blocker is reproducible by applying a `cron.*` hot reload during an in-flight cron run with `timeoutSeconds <= 0`.

## Solution Assessment

No, not as-is. Graceful draining is the right direction, but the hot-reload caller needs a bounded wait or fallback using the existing reload deferral policy before replacing cron.
