## Summary

Keep open: current main does not implement the central fix, and this PR targets a valid source-reproducible cron hot-reload race, but it still needs real behavior proof plus contract/lazy-wrapper and bounded-drain fixes before merge.

## What This Changes

The branch adds `CronService.stopGraceful()`, tracks in-flight cron timer/manual runs, awaits it during gateway cron hot reload, and adds stop-graceful regression tests.

## Best Possible Solution

Implement a contract-safe, lazy-wrapper-aware graceful cron replacement with a bounded drain/fallback, focused regression coverage, and redacted live `config.patch` proof.

## Reproduction Assessment

Yes, source-reproducible: current main synchronously stops and replaces cron during config reload while timer/manual cron runs can execute outside the lock and later persist. I did not run a live repro because this review is read-only.

## Solution Assessment

No, not yet. The direction is maintainable, but the current patch must update the `CronServiceContract` and lazy wrapper boundary and add a bounded drain/fallback before it is the narrow safe fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Add graceful stop to the cron contract:** `src/gateway/server-reload-handlers.ts:137`
  - body: Gateway reload holds cron as a `CronServiceContract`, and the lazy cron wrapper implements only `stop()`. This PR calls `stopGraceful()` through that handle but only adds it to the concrete service, so a rebase onto current main leaves the contract/lazy path without the method; add it to the contract and lazy wrapper, or route through a helper that handles both shapes.
  - confidence: 0.88
- **[P2] Bound the graceful drain before restart:** `src/cron/service/ops.ts:177`
  - body: `stopGraceful()` waits for every pending run with no deadline, and hot reload awaits it before applying the new cron config. Cron agent jobs can disable the outer timeout with `timeoutSeconds <= 0`, so a hung run can wedge config reload indefinitely; add a bounded wait/fallback with a warning so reload can proceed.
  - confidence: 0.85
