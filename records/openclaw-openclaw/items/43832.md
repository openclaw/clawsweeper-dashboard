## Summary

This PR should stay open: it targets a real cron hot-reload stale-write race and the latest commits address prior manual-run and timer-tick drain feedback, but the current head still has merge-blocking contract/type coverage, an unbounded reload wait, and the required changelog entry is missing.

## What This Changes

The PR adds a graceful cron stop path, tracks in-flight timer/manual cron runs, awaits the graceful stop during cron hot reload, and adds regression tests.

## Best Possible Solution

Land a rebased cron-owned fix that drains old cron writes before replacement, exposes the graceful stop through all cron contract implementations, bounds the hot-reload wait with existing reload fallback semantics, preserves `jobs.json` state, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main stops cron synchronously during hot reload while timer/manual cron executions can continue outside the lock and later persist through the old service.

## Solution Assessment

No, not as-is. Graceful draining is the right boundary, but the PR needs the contract/lazy/test surface update, a bounded drain fallback, and changelog coverage before merge.
