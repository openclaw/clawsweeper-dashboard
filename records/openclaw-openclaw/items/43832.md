## Summary

Keep open: the reported cron hot-reload race is still source-reproducible on current main, and this PR is pointed at the right bug, but the patch is not merge-ready because it misses the current cron contract/lazy wrapper shape, can block reload indefinitely, and still lacks external real behavior proof.

## What This Changes

This PR adds `CronService.stopGraceful()`, tracks in-flight cron timer/manual runs, awaits it during gateway config hot reload, and adds focused cron regression tests.

## Best Possible Solution

Implement graceful cron replacement on current main's cron contract and lazy wrapper, with a bounded drain/fallback, focused regression tests, and redacted live `config.patch` proof.

## Reproduction Assessment

Yes. Source inspection shows current main stops cron synchronously while timer/manual runs can persist afterward, and the linked report gives a concrete cron API update plus `config.patch` sequence; I did not run it because this review is read-only.

## Solution Assessment

No. The direction is maintainable, but this patch needs current-main contract/lazy-wrapper support and a bounded graceful drain before it is the narrow safe fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Implement graceful stop on the cron contract:** `src/gateway/server-reload-handlers.ts:137`
  - body: On current main, gateway reload holds `cron` as a `CronServiceContract` and it can be backed by the lazy cron wrapper. This PR only adds `stopGraceful()` to the concrete `CronService`, so after rebasing this call has no contract/lazy-wrapper implementation and will either fail type checking or miss the real runtime shape. Add the method to the contract and lazy wrapper, or route through a helper that works for both concrete and lazy cron states.
  - confidence: 0.88
- **[P2] Bound the graceful drain before reload:** `src/cron/service/ops.ts:179`
  - body: `stopGraceful()` waits for every pending run with no deadline, and hot reload awaits it before applying the new cron config. Cron agent jobs can disable timeout with `timeoutSeconds <= 0`, so a hung run can wedge config reload indefinitely. Add a bounded wait/fallback with a warning so reload can still proceed when a run never settles.
  - confidence: 0.84
