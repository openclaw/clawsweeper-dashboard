## Summary

Keep open: the PR targets a real cron hot-reload race that is still present on current main, but the proposed branch needs focused fixes before it can merge.

## What This Changes

The PR adds `CronService.stopGraceful()`, tracks in-flight cron timer/manual runs, awaits graceful cron shutdown during gateway hot reload, and adds stop-graceful regression tests.

## Best Possible Solution

Land a rebased cron-owned fix that drains old cron writes before replacement, exposes graceful shutdown through the cron contract and lazy/test implementations, bounds reload waiting with fallback semantics, preserves `jobs.json` state, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main stops cron synchronously during hot reload while timer/manual cron executions can continue outside the lock and later persist through the old service.

## Solution Assessment

No, not as-is. Graceful draining is the right boundary, but the PR needs the cron contract/lazy/test surface update, a bounded drain fallback, and changelog coverage before merge.
