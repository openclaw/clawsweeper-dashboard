## Summary

Keep open: current main still has the cron hot-reload stale-write path, and this PR is directionally useful, but the branch still has merge-blocking contract, reload-bounding, and changelog gaps.

## What This Changes

The PR adds a graceful CronService stop path, tracks in-flight cron timer/manual runs, awaits graceful cron shutdown during hot reload, and adds regression tests.

## Best Possible Solution

Land a rebased cron/gateway fix that drains old cron writes before replacement, exposes graceful stop through the cron contract and lazy implementations, bounds reload waiting, preserves jobs.json state, and records the fix in the changelog.

## Reproduction Assessment

Yes. Source inspection shows current main stops cron synchronously during hot reload while timer/manual cron executions can continue outside the lock and later persist through the old service.

## Solution Assessment

No, not as-is. Graceful draining is the right boundary, but the PR needs contract/lazy surface coverage, a bounded drain fallback, and changelog coverage before merge.
