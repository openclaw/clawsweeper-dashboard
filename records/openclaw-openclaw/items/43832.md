## Summary

Keep open: current main still has a source-reproducible cron hot-reload race, but this PR needs revision before merge because the new drain can wedge hot reloads, the branch is stale against the current cron contract, and external real behavior proof is missing.

## What This Changes

The PR adds `CronService.stopGraceful()`, tracks in-flight cron runs, uses it during gateway hot config reload, and adds cron state persistence regression tests.

## Best Possible Solution

Land a rebased fix that quiesces old cron instances without unbounded hot-reload waits, updates the cron service contract and implementations, and proves the `config.patch` cron state flow in a real gateway setup.

## Reproduction Assessment

Yes. The current-main source path is clear: hot reload calls synchronous `cron.stop()`, `stop()` only clears the timer, and timer/manual runs can continue outside the lock before a later persist.

## Solution Assessment

No, not yet. The PR's drain direction targets the right race, but the current implementation can block hot reload indefinitely and needs to be rebased across the current `CronServiceContract` boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound the graceful-stop drain before reload:** `src/cron/service/ops.ts:178-179`
  - body: `stopGraceful()` now waits for every tracked run with no deadline before the reload handler constructs the replacement cron service. Cron agent jobs can explicitly disable the execution timeout with `timeoutSeconds <= 0`, so one hung or intentionally long-lived run can keep `cron.*` hot reload from applying forever. Add a bounded wait, abort/interrupted-run path, or other fallback so reload cannot be wedged by a single in-flight run.
  - confidence: 0.9
- **[P2] Expose graceful stop through the cron contract:** `src/gateway/server-reload-handlers.ts:137`
  - body: The changed reload call targets `state.cronState.cron`, but current main types that field as `CronServiceContract`, and the PR head still adds `stopGraceful()` only to the concrete `CronService` class. After rebasing onto current main, this call will not type-check and alternate contract implementations will lack the method. Add `stopGraceful()` to the contract and implementations, or keep the reload state typed to an object that actually provides it.
  - confidence: 0.82
