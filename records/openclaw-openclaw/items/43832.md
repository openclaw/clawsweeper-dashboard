## Summary

Keep open: current main still has the cron hot-reload stale-persist race, and this branch addresses much of it, but the proposed reload path can wait unbounded on in-flight cron runs and still needs the required changelog entry.

## What This Changes

The branch adds a graceful cron stop path, tracks in-flight timer/manual cron runs, awaits it during cron hot reload, and adds drain regression tests.

## Best Possible Solution

Land a rebased cron-owned fix that drains old cron writes before replacement while applying the existing gateway reload timeout/fallback policy and preserving `jobs.json` semantics.

## Reproduction Assessment

Yes. Source inspection shows current main stops cron synchronously while timer and manual cron executions can still persist after outside-lock work; the PR blocker is reproducible by applying a `cron.*` hot reload during a no-timeout or hung in-flight cron run.

## Solution Assessment

No, not as-is. Graceful draining is the right direction, but the hot-reload caller needs a bounded wait or fallback using the existing reload deferral policy before replacing cron.
