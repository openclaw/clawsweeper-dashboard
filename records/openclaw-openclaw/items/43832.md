## Summary

Keep open: current main still restarts cron synchronously, and this PR targets a plausible source-reproducible hot-reload race, but the branch is not merge-ready because it lacks real behavior proof, conflicts with the current cron contract/lazy wrapper shape, and adds an unbounded reload-blocking drain.

## What This Changes

The PR adds a graceful cron stop path for gateway hot reload, tracks in-flight timer/manual cron runs, and adds regression tests for draining those runs before replacing the service.

## Best Possible Solution

Land a rebased graceful cron replacement that updates the contract and lazy/unavailable implementations, bounds the drain with a fallback warning, preserves the regression coverage, and includes redacted real `config.patch` proof.

## Reproduction Assessment

Yes, source-reproducible: current main stops and replaces cron synchronously while timer/manual cron execution can run outside `locked()` and later re-enter it to persist. I did not run a live `config.patch` reproduction in this read-only review.

## Solution Assessment

No, not yet. Graceful draining is the right direction, but the branch needs a current-main contract/lazy-wrapper rebase and a bounded drain before it is the narrow maintainable fix.
