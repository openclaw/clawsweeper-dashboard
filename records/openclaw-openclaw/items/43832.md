## Summary

Keep open: the PR addresses a real current-main cron hot-reload stale-write race, but the latest patch still has merge-blocking contract coverage, reload liveness, and changelog gaps.

## What This Changes

The PR adds a graceful cron stop path, tracks in-flight timer/manual cron runs, awaits the drain during cron hot reload, and adds regression tests.

## Best Possible Solution

Land a rebased cron-owned hot-reload fix that drains old cron writes before replacement while preserving reload progress guarantees and repository changelog policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main stops cron synchronously during hot reload while timer/manual cron executions can continue outside the lock and later persist through the old service.

## Solution Assessment

No, not as-is. The graceful drain boundary is the right direction, but it needs contract/lazy coverage, a bounded reload fallback, and changelog coverage before merge.
