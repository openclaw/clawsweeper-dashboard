## Summary

Keep open: this is a plausible fix for a source-reproducible cron hot-reload race that still exists on current main, but the PR is not merge-ready because it is stale against the current cron contract/lazy wrapper, can block reload indefinitely, and lacks contributor real behavior proof.

## What This Changes

The PR adds `CronService.stopGraceful()`, tracks in-flight timer/manual cron runs, awaits it during gateway cron hot reload, and adds regression tests.

## Best Possible Solution

Land a rebased graceful cron replacement that updates the contract and lazy wrapper, drains timer/manual paths with a bounded fallback warning, preserves regression coverage, and includes redacted `config.patch` proof.

## Reproduction Assessment

Yes, by source inspection: current main stops and replaces cron synchronously while timer/manual cron execution can run outside `locked()` and later re-enter it to persist. I did not run a live `config.patch` reproduction in this read-only review.

## Solution Assessment

No, not yet. Graceful draining is the right direction, but this branch must be rebased across the current `CronServiceContract`/lazy wrapper and must bound the drain so hot reload cannot wait forever.
