## Summary

Keep open: the PR targets a real current-main cron hot-reload race, but it is not merge-ready because it misses the current cron contract/lazy-wrapper surface, can block hot reload indefinitely, and lacks required after-fix real behavior proof.

## What This Changes

The PR adds `CronService.stopGraceful()`, tracks in-flight cron timer/manual runs, awaits graceful stop during gateway hot reload, and adds cron regression tests.

## Best Possible Solution

Land a rebased graceful cron replacement that covers the contract/lazy/local implementations, bounds the drain with fallback logging, preserves focused regression tests, and includes redacted real `config.patch` proof.

## Reproduction Assessment

Yes, source-reproducible: current main stops and immediately rebuilds cron while timer/manual cron execution can run outside `locked()` and later re-enter it to persist. I did not run a live `config.patch` reproduction in this read-only review.

## Solution Assessment

No, not yet. Graceful draining is the right direction, but this branch needs a current-main contract/lazy/local rebase and a bounded drain before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Add graceful stop to the cron contract:** `src/gateway/server-reload-handlers.ts:137`
  - body: Current main routes gateway cron through `CronServiceContract` and may install the lazy or local unavailable cron implementation, but this patch only adds `stopGraceful()` to the concrete service before calling it from hot reload. Rebase the method through the contract and every implementation so this call is valid on all gateway cron states.
  - confidence: 0.9
- **[P2] Bound the graceful drain before restarting cron:** `src/cron/service/ops.ts:179`
  - body: `stopGraceful()` awaits every tracked run with no deadline before cron hot reload can rebuild the service. Because cron agent turns can explicitly disable the watchdog with `timeoutSeconds <= 0`, one stuck run can block config reload indefinitely; add a bounded wait and fallback warning before allowing reload to continue.
  - confidence: 0.88
