## Summary

Keep open: this PR targets a valid current-main cron hot-reload race, but the proposed patch still needs contract/lazy-surface coverage, a bounded drain fallback, and the required changelog entry before merge.

## What This Changes

The PR adds a graceful cron stop path, tracks in-flight timer/manual cron runs, awaits it during cron hot reload, and adds stop-graceful regression tests.

## Best Possible Solution

Land a rebased cron-owned fix that drains old cron writes before replacement, exposes the graceful stop through all cron contract implementations, bounds the hot-reload wait with existing reload fallback semantics, preserves `jobs.json` state, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Current main calls `state.cronState.cron.stop()` during `restartCron` hot reload, while timer and manual cron executions can continue outside the lock and later persist through the old service.

## Solution Assessment

No, not as-is. Graceful draining is the right boundary, but the PR needs the contract/lazy/test surface update, a bounded drain fallback, and changelog coverage before merge.
