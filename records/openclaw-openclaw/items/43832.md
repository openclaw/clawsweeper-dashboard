## Summary

Keep open. The PR targets a real cron hot-reload race that current main still has, but the proposed hot-reload path can now wait forever on tracked cron runs, so it needs a bounded drain or fallback before merge.

## What This Changes

The branch adds a graceful cron stop path, tracks in-flight cron timer/manual runs, switches cron hot reload to await that path, and adds regression tests for the drain behavior.

## Best Possible Solution

Land a rebased cron-owned fix that prevents stale post-reload cron persists while bounding the graceful-stop wait with the existing gateway reload policy, preserving `jobs.json`/`jobs-state.json` semantics and covering locked operations, manual runs, timer ticks, and timeout fallback in tests.

## Reproduction Assessment

Yes. The original race is reproducible by triggering a `cron.*` hot reload while a timer or manual cron run is suspended in its outside-lock worker phase; the PR's new blocker follows from the same path with an active run whose `timeoutSeconds <= 0`.

## Solution Assessment

No, not as-is. Gracefully draining old cron work before replacement is the right direction, but the reload path must have a finite wait or fallback so one long-lived cron run cannot wedge config reload forever.
