## Summary

Keep open: the PR targets a real source-reproducible cron hot-reload race, but the branch still needs current-main contract/lazy-cron reconciliation, a bounded graceful drain, and contributor real behavior proof before merge.

## What This Changes

The PR adds `CronService.stopGraceful()` to drain in-flight cron timer/manual runs during gateway hot reload and adds regression tests.

## Best Possible Solution

Land a rebased graceful cron replacement that updates the contract and lazy wrapper, bounds the drain with a fallback warning, preserves regression coverage, and includes redacted `config.patch` proof.

## Reproduction Assessment

Yes, by source inspection. Current main stops and replaces cron synchronously while timer/manual cron execution can continue outside the service lock and later re-enter it to persist state.

## Solution Assessment

No, not yet. The proposed direction fits the bug, but the branch needs current-main contract/lazy-wrapper support and a bounded drain before it is the narrow maintainable fix.
