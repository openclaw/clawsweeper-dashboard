## Summary

Keep open: current main still has the source-level cron hot-reload race, and this PR is a useful fix direction, but the branch still needs current-main contract/lazy-wrapper rebase work, a bounded drain, and required real behavior proof before merge.

## What This Changes

The PR adds a graceful cron stop path, tracks in-flight cron timer/manual runs, awaits it during gateway hot reload, and adds cron regression tests.

## Best Possible Solution

Land a rebased graceful cron replacement that updates the contract and all cron implementations, bounds the drain with fallback logging, preserves focused regression tests, and includes redacted real config.patch proof.

## Reproduction Assessment

Yes, source-reproducible: current main stops and immediately rebuilds cron while timer/manual cron execution can run outside locked() and later re-enter it to persist. I did not run a live config.patch reproduction in this read-only review.

## Solution Assessment

No, not yet. Graceful draining is the right direction, but this branch needs a current-main contract/lazy/local rebase, a bounded drain, and real behavior proof before it is the narrow maintainable fix.

- **[P2] Add graceful stop to the cron contract:** `src/gateway/server-reload-handlers.ts:137`
  - body: Current main routes gateway cron through CronServiceContract and has a lazy gateway cron wrapper, but this patch only adds stopGraceful() to the concrete service before calling it from hot reload.   - confidence: 0.9

Because cron agent turns can explicitly disable the watchdog with timeoutSeconds <= 0, one stuck run can block config reload indefinitely; add a bounded wait and fallback warning before allowing reload to continue.
  - confidence: 0.88
