## Summary

Keep open. The cron hot-reload race is still present on current main and this PR is pointed at the right bug, but it currently has a type-surface break, an unbounded reload wait, and a missing changelog entry, so it should not merge as-is.

## What This Changes

The branch adds a graceful cron stop path, tracks in-flight timer/manual cron runs, awaits graceful stop during cron hot reload, and adds regression tests.

## Best Possible Solution

Land a rebased cron-owned fix that drains old cron writes before replacement, exposes graceful stop through the cron contract, bounds the reload wait with existing deferral semantics, preserves `jobs.json` state, and records the fix in the changelog.

## Reproduction Assessment

Yes. Source inspection shows current main stops cron synchronously while timer and manual cron executions can still persist after outside-lock work; the PR blocker is also reproducible from source because `CronServiceContract` does not expose the new method.

## Solution Assessment

No, not as-is. Graceful draining is the right direction, but the branch needs the contract update, a bounded reload fallback for stuck/no-timeout cron runs, and the required changelog entry.
