## Summary

Keep open: this PR targets a real cron hot-reload race, but current main does not already implement the fix and the branch still has contract/lazy-wrapper, bounded-drain, merge-dirty, and real behavior proof blockers before merge.

## What This Changes

The PR adds a graceful cron shutdown path, tracks in-flight timer/manual cron runs, awaits it during gateway cron hot reload, and adds regression tests.

## Best Possible Solution

Land a rebased, contract-safe graceful cron replacement that drains old writes with a bounded fallback and includes regression coverage plus redacted config.patch proof.

## Reproduction Assessment

Yes. The race is source-reproducible: current main stops and replaces cron synchronously while timer/manual cron execution can run outside the service lock and later re-enter the lock to persist state.

## Solution Assessment

No, not yet. The approach is pointed at the right bug, but it needs the current `CronServiceContract`/lazy wrapper boundary and a bounded drain before it is the narrow safe fix.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Add graceful stop to the cron contract:** `src/gateway/server-reload-handlers.ts:137`
  - body: Gateway reload holds cron through `CronServiceContract`, and current main also has a lazy cron wrapper that only implements `stop()`. Calling `stopGraceful()` here without adding it to the contract and lazy wrapper leaves the rebased gateway path without a valid method.
  - confidence: 0.88

  - confidence: 0.86
