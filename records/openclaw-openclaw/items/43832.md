## Summary

Keep open. The bug is still source-reproducible on current main and this PR is aimed at the right cron hot-reload race, but it is not merge-ready because it misses the current CronServiceContract/lazy wrapper boundary and its graceful drain can wedge config reload behind unbounded cron runs; external real behavior proof is still missing.

## What This Changes

The PR adds a CronService.stopGraceful drain for in-flight timer/manual cron runs, calls it during gateway cron hot reload, and adds stop-graceful regression tests.

## Best Possible Solution

Implement graceful cron replacement through the current cron contract and lazy wrapper with a bounded drain/fallback, focused regression tests, and redacted live config.patch proof.

## Reproduction Assessment

Yes, source-reproducible: current main stops cron synchronously during hot reload while timer and manual cron runs can execute outside the lock and persist afterward. I did not run a live config.patch reproduction because this review is read-only.

## Solution Assessment

No. The direction is maintainable, but the current patch must be rebased onto the CronServiceContract/lazy-wrapper shape and bound its drain before it is the narrow safe fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Implement graceful stop on the cron contract:** `src/gateway/server-reload-handlers.ts:137`
  - body: On current main, gateway reload holds `cron` as a `CronServiceContract`, and the lazy cron wrapper implements only `stop()`. This PR calls `stopGraceful()` during reload but only adds it to the concrete service, so after rebase the call has no contract/lazy-wrapper implementation and will fail the boundary. Add it to the contract and lazy wrapper, or route through a helper that works for both shapes.
  - confidence: 0.88
- **[P2] Bound the graceful drain before restarting cron:** `src/cron/service/ops.ts:179`
  - body: `stopGraceful()` waits for every pending run with no deadline, and hot reload awaits it before applying the new cron config. Cron agent jobs can disable the outer timeout with `timeoutSeconds <= 0`, so a hung run can wedge config reload indefinitely. Add a bounded wait/fallback with a warning so reload can still proceed.
  - confidence: 0.85
