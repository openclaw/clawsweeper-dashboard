## Summary

Keep open: this PR targets a real, source-reproducible cron hot-reload state overwrite, but the current head is not merge-ready because it is stale/conflicting against current main, misses the current cron contract/lazy wrapper surface, has an unbounded drain in the reload path, and lacks required real behavior proof.

## What This Changes

Adds a CronService graceful-stop drain, tracks in-flight cron timer/manual runs, awaits it during gateway hot reload, and adds cron regression tests.

## Best Possible Solution

Land a rebased graceful cron replacement that extends CronServiceContract and all implementations, bounds the drain with fallback logging, preserves focused regression coverage, and includes redacted config.patch proof.

## Reproduction Assessment

Yes, source-reproducible: current main stops and immediately rebuilds cron while timer/manual cron execution can run outside locked() and later re-enter it to persist. I did not run a live config.patch reproduction in this read-only review.

## Solution Assessment

No, not yet. Graceful draining is the right direction, but this branch needs a current-main contract/lazy-wrapper rebase, a bounded drain, and real behavior proof before it is the narrow maintainable fix.

## Review Findings

Full review comments:

- **[P2] Add graceful stop to the cron contract:** `src/gateway/server-reload-handlers.ts:137`
  - body: Current main routes gateway cron through `CronServiceContract` and a lazy gateway cron wrapper, but this patch calls `stopGraceful()` from hot reload without adding that method to the contract or every implementation. Rebase the method through the contract, lazy wrapper, and unavailable/mock cron surfaces so this call is valid on all gateway cron states.
  - confidence: 0.9
- **[P2] Bound the graceful drain before restarting cron:** `src/cron/service/ops.ts:179`
  - body: `stopGraceful()` awaits every tracked run with no deadline before cron hot reload can rebuild the service. Because cron agent turns can explicitly disable the watchdog with `timeoutSeconds <= 0`, one stuck run can block config reload indefinitely; add a bounded wait and fallback warning before allowing reload to continue.
  - confidence: 0.88
