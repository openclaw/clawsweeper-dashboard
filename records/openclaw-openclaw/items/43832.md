## Summary

Keep open: the PR targets a real current-main cron hot-reload race, but it is not mergeable as-is because the graceful stop method is not wired through the cron contract/lazy gateway cron boundary, the drain is unbounded, and external real behavior proof is still missing.

## What This Changes

This PR adds a graceful cron shutdown path, tracks in-flight cron runs, switches hot config reload to await that drain, and adds regression tests.

## Best Possible Solution

Keep the fix direction, but rebase it onto current main, expose graceful stop through the cron contract and lazy cron wrapper, add a bounded stale-write-safe drain, and require real hot-reload proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main stops cron synchronously during hot reload, while timer and manual cron runs can execute outside locked() and later persist. I did not run a live config.patch scenario because this review was read-only.

## Solution Assessment

No, not yet. The approach targets the right race, but the current branch needs the contract/lazy-wrapper updates, a bounded drain strategy, and real after-fix proof before it is the maintainable fix.

- **[P1] Expose stopGraceful through the cron contract:** `src/gateway/server-reload-handlers.ts:137`
  - body: `state.cronState.cron` is a `CronServiceContract` on current main, but the PR only adds `stopGraceful()` to `CronService`.   - confidence: 0.93

Since `timeoutSeconds <= 0` disables the cron job timeout, a hung no-timeout run can now keep cron config changes from applying indefinitely; add a bounded drain or fallback that still prevents stale post-reload writes.
  - confidence: 0.86
