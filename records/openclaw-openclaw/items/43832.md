## Summary

Keep open: the underlying cron hot-reload race is valid and the PR moves in the right direction, but it still needs real behavior proof and code changes for current main's lazy cron contract plus a bounded drain before it is merge-ready.

## What This Changes

The PR adds `CronService.stopGraceful()`, tracks in-flight cron runs, and awaits graceful stop before rebuilding cron during gateway config reload.

## Best Possible Solution

Implement graceful cron replacement on current main's cron contract and lazy wrapper with a bounded drain/fallback, then land it with focused regression tests and redacted live config.patch proof.

## Reproduction Assessment

Yes. Source inspection shows current main stops cron synchronously while timer/manual runs can persist later, and the linked bug report gives a concrete config.patch sequence; I did not run it because this review is read-only.

## Solution Assessment

No. The direction is maintainable, but the PR needs to be updated for current main's lazy cron contract and should not let one hung or no-timeout cron run block config reload forever.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Implement graceful stop on the current cron contract:** `src/gateway/server-reload-handlers.ts:137`
  - body: On current main, gateway reload holds `cron` as a `CronServiceContract` and may be backed by the lazy cron wrapper. This PR only adds `stopGraceful()` to the concrete `CronService`, so after rebasing this call has no contract/lazy-wrapper implementation and will either fail type checking or bypass the real runtime shape. Add the method to the contract and lazy cron wrapper, or route through a helper that works for both concrete and lazy cron states.
  - confidence: 0.88
- **[P2] Bound the graceful drain before applying reload:** `src/cron/service/ops.ts:179`
  - body: `stopGraceful()` waits for every pending run with no deadline, and hot reload awaits it before applying the new cron config. Cron agent jobs can disable timeout with `timeoutSeconds <= 0`, so a hung run can wedge config reload indefinitely. Add a bounded wait/fallback with a warning so reload can still proceed when a run never settles.
  - confidence: 0.84
