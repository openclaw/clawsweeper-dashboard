## Summary

Keep open: current main and the latest release still use the built-in 12 KiB reseed-history cap at the production CLI reseed call site and expose no per-backend config key. There is an open linked PR with closing syntax for this issue, so the issue should remain open while maintainers review the config contract and that implementation path.

## What This Changes

Expose a per-CLI-backend config option to control the maximum characters included in CLI session reseed history.

## Best Possible Solution

Add a documented optional per-backend reseed-history cap on the current raw-reseed call path, keep the 12 KiB fallback for unset configs, and cover both configured and default behavior with focused tests.

## Reproduction Assessment

Yes for the source-level behavior: current main has a hardcoded fallback cap, the production call site does not pass a configured value, and the backend config schema has no matching field. The private live Telegram/Claude JSONL case was not rerun.

## Solution Assessment

Unclear until maintainers choose the public config contract. The proposed per-backend knob is a maintainable direction because it preserves the default and matches nearby backend tuning patterns, but it is still a new user-facing config surface.
