## Summary

Keep open: current main still uses the built-in 12 KiB reseed-history cap at the production CLI reseed call site, and the backend config contract has no per-backend cap field. A linked open PR already closes this issue, so the issue should stay open until that PR lands or maintainers choose a different config shape.

## What This Changes

The issue asks to add an optional per-backend CLI reseed-history character cap and plumb it into `buildCliSessionHistoryPrompt` while preserving the 12 KiB default.

## Best Possible Solution

Review and land the linked PR, or an equivalent additive config implementation, with schema/docs/tests proving both configured larger caps and the unchanged unset fallback.

## Reproduction Assessment

Yes for the source-level behavior: current main has a hardcoded fallback cap and the production call site does not pass a configured value. The live Telegram/Claude JSONL scenario was not rerun during this read-only review.

## Solution Assessment

Unclear until maintainers choose the public config contract. The proposed per-backend knob is a maintainable direction because it preserves the default and follows nearby CLI backend tuning patterns, but it is still a new user-facing config surface.
