## Summary

Keep open: current main and the latest shipped release still use the built-in 12 KiB CLI reseed-history cap at the production call site, and the backend config contract has no per-backend cap field. The linked implementation PR is still open, so this issue should stay open until that PR is reviewed, landed, or deliberately closed.

## What This Changes

Request to expose the CLI reseed history character cap as an optional per-backend config value instead of relying on the current hardcoded 12 KiB fallback.

## Best Possible Solution

Review the linked PR or a replacement that adds a documented optional per-backend cap, preserves the 12 KiB default when unset, threads the value through the current raw-reseed call path, and covers configured plus fallback behavior with focused tests.

## Reproduction Assessment

Yes for the source-level behavior: current main has a hardcoded fallback cap and the production call site does not pass a configured value. The live Telegram/Claude JSONL scenario was not rerun in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public config contract. The proposed per-backend knob is a maintainable direction because it preserves the default and follows nearby backend tuning patterns, but it still adds a new user-facing config surface.
