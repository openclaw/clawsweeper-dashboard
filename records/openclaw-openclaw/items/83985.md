## Summary

Keep open: current main and v2026.5.18 still have only the built-in 12 KiB CLI reseed-history cap, no per-backend config/schema field, and a same-author closing PR is open but unresolved.

## What This Changes

The issue requests an optional per-backend CLI reseed history cap so long-context CLI backends can raise the current hardcoded default without changing smaller backends.

## Best Possible Solution

Review or rebase the linked PR, add a documented optional per-backend cap through the current CLI backend type/schema/prepare path, and cover both unset fallback and configured larger-cap behavior with focused tests.

## Reproduction Assessment

Yes for the source-level behavior: current main hardcodes the fallback cap, the production call site does not pass a configured value, and the strict backend schema has no requested field. I did not rerun the reporter's live Telegram/Claude JSONL scenario.

## Solution Assessment

Unclear until maintainers choose the public config contract. A per-backend knob is a maintainable direction because it preserves the default and follows existing backend tuning patterns, but it is still a new user-facing config surface.
