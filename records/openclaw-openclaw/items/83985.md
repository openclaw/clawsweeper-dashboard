## Summary

Keep open: current main and the latest release still use the built-in 12 KiB reseed-history cap at the CLI reseed call site, and the backend config contract has no per-backend cap field. The issue also has an open linked PR with closing syntax, so the right cleanup is to review or replace that PR rather than close the issue early.

## What This Changes

The issue requests a per-backend CLI reseed-history cap so `buildCliSessionHistoryPrompt` is not always limited by the 12 KiB default.

## Best Possible Solution

Review or rebase the linked implementation so it adds a documented optional per-backend cap, preserves the 12 KiB default and current raw-reseed safety contract, and covers configured and unset behavior with focused tests.

## Reproduction Assessment

Yes for the source-level behavior: current main exposes a helper parameter but the production call site passes no cap, and the config type/schema lack the requested field. The live Telegram/Claude JSONL scenario was not rerun during this read-only review.

## Solution Assessment

Unclear until maintainers choose the public config contract. The proposed optional per-backend knob is a maintainable direction because it preserves the default and follows nearby backend tuning patterns, but it is still a new user-facing config surface.
