- `P2`: The issue asks for a bounded CLI session-recovery configuration improvement that affects memory quality but does not make the core runtime unusable.
- `impact:session-state`: The reported hard cap limits how much prior conversation context survives a CLI session reseed.

## Summary

Current main still has the hardcoded reseed-history cap with no per-backend override, but this issue is already paired with an open closing PR, so it should stay open until that implementation PR is reviewed and either merged or closed.

## What This Changes

The issue requests a new optional `agents.defaults.cliBackends.<id>.maxReseedHistoryChars` setting that is passed into `buildCliSessionHistoryPrompt` instead of always using the 12 KiB fallback.

## Best Possible Solution

Land a reviewed version of the linked PR that adds the optional per-backend cap, schema/type/docs coverage, and regression tests while preserving the existing 12 KiB default when unset.

## Reproduction Assessment

Yes. Source inspection on current main shows the hardcoded fallback exists, the production call site does not pass `maxHistoryChars`, and the backend config type/schema do not expose the requested field.

## Solution Assessment

Yes, pending normal PR review. An optional per-backend config field is the narrowest maintainable path because it preserves existing defaults while letting long-context backends opt into a larger reseed history window.
