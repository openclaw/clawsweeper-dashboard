## Summary

Keep open: current main still uses the built-in 12 KiB reseed-history cap at the production CLI reseed call site, and the backend config schema has no per-backend cap field. The related open slice-direction issue is orthogonal, so this remains a valid config/API request rather than a duplicate or implemented cleanup.

## What This Changes

The issue requests a per-CLI-backend setting to control the maximum rendered CLI reseed history characters while preserving the existing default when unset.

## Best Possible Solution

Add a documented optional per-backend reseed-history cap, merge it through the existing CLI backend config path, pass it into `buildCliSessionHistoryPrompt`, and cover both unset fallback and configured larger-cap behavior with focused tests.

## Reproduction Assessment

Yes for the source-level behavior: current main has a hardcoded fallback cap and the production call site does not pass a configured value. The live Telegram/Claude JSONL scenario was not rerun during this read-only review.

## Solution Assessment

Unclear until maintainers choose the public config contract. The proposed per-backend knob is a maintainable direction because it preserves the default and follows nearby CLI backend tuning patterns, but it is still a new user-facing config surface.
