## Summary

Keep open: the PR fixes the right Gateway boundary for fresh agent-prefixed session keys, but the current patch introduces a blocking compatibility regression by rejecting legacy `agent:main:main`-style aliases before canonical session-key remapping can route them to a configured non-`main` default agent.

## What This Changes

The branch changes Gateway agent dispatch to infer `agentId` from agent-prefixed explicit session keys, reject unknown or mismatched embedded agent IDs, and add regression tests for fresh CLI and cron-style sessions.

## Best Possible Solution

Preserve existing canonical legacy main-alias behavior while still inferring configured owners for fresh explicit keys and rejecting truly unknown non-alias agent-prefixed keys before dispatch.

## Reproduction Assessment

Yes, source-reproducible rather than locally rerun: current main only infers `agentId` for parsed session keys that canonicalize to `global`, while CLI Gateway sends agent-shaped keys through unchanged and dispatch can use an unset owner for non-global fresh keys. The PR body supplies real CLI/Gateway proof for the configured `main` path.

## Solution Assessment

No, not as currently patched: inferring the owner before session load is the right boundary, but rejecting raw parsed unknown agents before canonical alias remapping breaks existing legacy main-session behavior. The safer fix is to classify aliases through the existing session-store canonicalization path before deciding an embedded agent is truly unknown.
