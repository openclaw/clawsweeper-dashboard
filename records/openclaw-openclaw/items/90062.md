## Summary

Keep open. The patch looks like the right Gateway boundary fix and has convincing fresh-session proof, but this draft PR has a protected maintainer label and intentionally changes session/auth routing behavior, so it needs maintainer acceptance rather than cleanup closure.

## What This Changes

The branch changes Gateway agent handling to infer `agentId` from configured explicit `agent:<id>:...` session keys, reject unknown or mismatched embedded ids, and add focused regression tests.

## Best Possible Solution

Land the narrow Gateway fix once a maintainer accepts the fail-closed invalid-key behavior; otherwise adjust the unknown-agent policy before merge.

## Reproduction Assessment

Yes, source-reproducible rather than locally rerun: current main only infers `agentId` for agent keys that canonicalize to `global`, so a fresh `agent:main:probe-*` key can reach dispatch without the embedded owner. The linked issue and PR proof provide concrete CLI/Gateway evidence for the user-visible failure and after-fix behavior.

## Solution Assessment

Yes, this is the best implementation boundary: infer or reject the owner before `loadSessionEntry` and `dispatchAgentRunFromGateway`, rather than adding a later provider fallback. The only remaining decision is whether maintainers accept the intentional fail-closed behavior for unknown embedded agent ids.
