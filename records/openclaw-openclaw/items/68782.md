## Summary

Keep open. Current main has partial runtime and packaging improvements for plugin startup overhead, including allow/deny configuration, bundled discovery policy, plugin status/install commands, and lazy CLI/runtime paths, but it still lacks the core requested product surface for install-time selection of plugins/channels/skills and strict user-selected discovery/loading semantics. The prior ClawSweeper comment already captured the same direction, and later source/history checks show the remaining work is still a broad packaging/setup design item rather than a narrow bug fix.

## What This Changes

The issue requests user-selectable installation and loading of plugins, skills, and channels to reduce install size, startup overhead, dependency exposure, and log noise.

## Best Possible Solution

Keep this as a maintainer-owned design item and split implementation into focused tracks for install composition, setup/onboarding UX, strict discovery semantics, plugin status reporting, and skill/channel dependency policy.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that install-time channel/plugin/skill selection and strict discovery semantics are still missing from current main.

## Solution Assessment

Unclear as a single implementation. The existing runtime allowlist and bundledDiscovery work are useful foundations, but the requested install-selection behavior needs maintainer product and packaging design before code should be written.
