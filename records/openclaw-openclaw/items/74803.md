## Summary

Keep open. Current main still preserves exact explicit isolated cron `agentTurn` timeout behavior and only documents the `lightContext`/`toolsAllow` knobs; it does not add the requested cold-start warning, default-timeout adjustment, or prominent timeout guidance. Open PR #74893 already references this issue with closing syntax and remains the active implementation candidate, so this should not be closed or queued as a competing repair job before that PR is reviewed.

## What This Changes

The issue requests a warning, timeout default adjustment, or prominent docs guidance for short isolated `agentTurn` cron jobs that omit lightweight context and tool allowlists, because cold-start bootstrap can consume the configured timeout before useful work begins.

## Best Possible Solution

Review and land PR #74893, or replace it with a narrow guardrail that warns and documents short explicit isolated cron `agentTurn` jobs without lightweight context or a tool allowlist. Preserve explicit timeout semantics unless the broader wall-time enforcement problem tracked by #74681 is intentionally addressed separately.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: create an isolated `agentTurn` cron job with a short positive `timeoutSeconds`, no `lightContext`, and no `toolsAllow`; current main applies that timeout directly and starts it before later embedded-run bootstrap/context work completes.

## Solution Assessment

Unclear until PR #74893 is reviewed. A targeted warning plus docs guidance is the narrowest maintainable fix for this issue, while changing timeout enforcement or defaults would overlap the broader timeout semantics tracked in #74681.
