## Summary

Keep open. Current `main` still has the reported guardrail gap: explicit isolated `agentTurn` timeouts are preserved, `lightContext` and `toolsAllow` remain optional mitigations, the cron docs do not prominently warn about bootstrap/pre-first-call budget loss, and the paired implementation PR is still open.

## What This Changes

The issue requests a warning, timeout-default adjustment, or prominent docs guidance for isolated cron `agentTurn` jobs that can spend their timeout budget before the first tool or model call.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74893 or an equivalent narrow warning/docs guardrail, then track broader hard wall-time and pre-first-call stall behavior through https://github.com/openclaw/openclaw/issues/74681 or a narrower follow-up.

## Reproduction Assessment

Yes, at source level. Current `main` still accepts an isolated `agentTurn` cron payload with a short explicit `timeoutSeconds`, no `lightContext`, and no `toolsAllow`, preserves that timeout, and emits only the generic timeout error when the cron timer fires after execution starts.

## Solution Assessment

Unclear. A targeted warning plus docs guidance is the narrowest current mitigation for the original report, but the newer 2026.5.7 evidence may require separate diagnostics or a soft bootstrap timeout decision rather than only warning about `lightContext` and `toolsAllow`.
