## Summary

This PR should stay open for maintainer handling: it targets a real agent/subagent result-delivery gap, but the branch is not a correct current-main fix because it misses structured result envelopes, does not port the fallback through the current run-manager path, can still return a result-less lifecycle snapshot first, and lacks required after-fix real behavior proof.

## What This Changes

The PR threads terminal `agent.wait` result payloads through gateway wait snapshots/responses and reuses that result text as a subagent announce/registry fallback, with focused regression tests.

## Best Possible Solution

Port a consolidated current-main fix that carries result payloads through dedupe, `agent.wait`, `waitForAgentRun`, and `createSubagentRunManager`, decodes normal `{ payloads, meta }` results, and keeps chat-history captures authoritative.

## Reproduction Assessment

Yes, at source level. Current main stores completed agent output as `payload.result`, but the dedupe snapshot, `agent.wait` response, wait wrapper, and current run-manager completion path do not preserve that result into subagent fallback delivery.

## Solution Assessment

No. The direction is useful, but this branch is not the best mergeable solution until it is rebased onto the current run-manager path, decodes `{ payloads, meta }`, avoids returning a result-less lifecycle snapshot first, and includes real behavior proof.
