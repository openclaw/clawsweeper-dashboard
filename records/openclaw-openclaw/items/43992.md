## Summary

Keep open: current main still has the source-level result-propagation gap and this PR is useful direction, but the branch cannot merge as-is because it misses current run-manager/result-envelope/race paths and lacks required after-fix real behavior proof.

## What This Changes

The PR threads terminal `agent.wait` result payloads through gateway dedupe/wait responses and uses them as fallback text for subagent completion announces, with focused regression tests.

## Best Possible Solution

Port the fix onto current main so result payloads flow through dedupe, `agent.wait`, `waitForAgentRun`, and `createSubagentRunManager`, while decoding normal `{ payloads, meta }` results and keeping chat-history captures authoritative.

## Reproduction Assessment

Yes, at source level. Current main stores completed agent output as `payload.result`, but the current wait snapshot, `agent.wait` response, `waitForAgentRun`, and run-manager completion paths do not carry that result into subagent completion fallback delivery.

## Solution Assessment

No. The direction is useful, but this branch is not the best mergeable solution until it is ported to current run-manager/wait paths, decodes `{ payloads, meta }`, and avoids returning lifecycle snapshots before the result-bearing dedupe payload is available.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Parse result payload envelopes:** `src/agents/subagent-announce.ts:345-347`
  - body: `extractAgentWaitResultText` checks scalar fields and passes `result` directly to the text extractor, but gateway agent runs normally store `{ payloads, meta }`. That leaves `payloads[*].text` invisible during the history catch-up window, so the fallback can still announce no completion text.
  - confidence: 0.91
- **[P1] Port fallback through the run manager:** `src/agents/subagent-registry.ts:1261-1264`
  - body: This writes fallback text into the older inline registry wait path, while current main completes registered subagent waits through `createSubagentRunManager` and `waitForAgentRun`. A current-main port would still complete without preserving the wait result unless that path carries and freezes the result text too.
  - confidence: 0.9
- **[P2] Wait for the result-bearing dedupe snapshot:** `src/gateway/server-methods/agent.ts:772`
  - body: The live `agent.wait` path can return the lifecycle snapshot first and abort the dedupe waiter, but lifecycle snapshots do not carry `result`. Normal runs can emit lifecycle end before the gateway writes the result-bearing dedupe payload, so this can still return `undefined` in the target catch-up window.
  - confidence: 0.85
