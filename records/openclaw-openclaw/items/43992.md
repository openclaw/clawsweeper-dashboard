## Summary

Keep open: the PR targets a real current-main subagent completion gap, but the branch is not mergeable as-is because it lacks contributor real behavior proof and still misses the current run-manager/result-envelope paths.

## What This Changes

The PR threads terminal `agent.wait` result payloads through gateway wait snapshots/responses and reuses that text as a fallback for subagent announce and registry completion flows, with focused tests.

## Best Possible Solution

Port a consolidated current-main fix so result payloads flow through dedupe, `agent.wait`, `waitForAgentRun`, and `createSubagentRunManager`, while decoding normal `{ payloads, meta }` results and keeping chat-history captures authoritative.

## Reproduction Assessment

Yes, at source level. Current main stores completed agent output as `payload.result`, but the wait snapshot, `agent.wait` response, wait wrapper, and run-manager completion path do not carry that result into fallback delivery.

## Solution Assessment

No. The direction is useful, but this branch is not the best mergeable solution until it is rebased onto the current run-manager path, decodes `{ payloads, meta }`, avoids returning result-less lifecycle snapshots first, and includes real behavior proof.
