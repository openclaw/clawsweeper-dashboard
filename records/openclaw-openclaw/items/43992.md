## Summary

Keep open: current main still drops `agent.wait` result payloads before they can be used for subagent completion, and this PR is not merge-ready because it misses the current run-manager path, normal result-envelope parsing, and a lifecycle/dedupe race.

## What This Changes

The PR preserves terminal `agent.wait` result payloads in gateway dedupe responses and reuses them as subagent announce/registry fallback text, with gateway and subagent regression tests.

## Best Possible Solution

Land a current-main port that carries result payloads through dedupe, `agent.wait`, and `waitForAgentRun`, parses `{ payloads, meta }` result envelopes, freezes fallback text in the active run manager only when chat-history capture has not already won, and includes focused regressions plus the required changelog entry.

## Reproduction Assessment

Yes. Current main stores the agent command return value as `payload.result`, then drops it from terminal snapshots, `agent.wait` responses, the shared run-wait helper, and the active subagent run-manager completion path.

## Solution Assessment

No. The PR is directionally useful, but it is not the best current fix because it targets an obsolete registry wait path, misses normal `{ payloads, meta }` envelope parsing, and can still return a lifecycle snapshot before the result-bearing dedupe snapshot is available.
