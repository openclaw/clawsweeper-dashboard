## Summary

Keep open: current main still drops the completed `agent.wait` result before subagent fallback delivery, and this PR addresses a real path, but the branch is not merge-ready because it lacks real behavior proof, is merge-conflicting, and needs to be ported through the newer run-manager/result-envelope path.

## What This Changes

The branch threads gateway `agent.wait` result payloads into wait snapshots/responses and uses them as subagent announce/registry fallback text, with focused regression tests.

## Best Possible Solution

Port a consolidated current-main fix that carries result payloads through dedupe, `agent.wait`, `waitForAgentRun`, and `createSubagentRunManager`, decodes normal `{ payloads, meta }` results, and keeps chat-history captures authoritative.

## Reproduction Assessment

Yes, at source level. Current main stores completed agent output as `payload.result`, but dedupe snapshots, `agent.wait` responses, the wait wrapper, and the current run-manager completion path do not preserve that result into subagent fallback delivery.

## Solution Assessment

No. The direction is useful, but this branch is not the best mergeable solution until it is rebased onto the current run-manager path, decodes `{ payloads, meta }`, avoids returning a result-less lifecycle snapshot first, and includes real behavior proof.
