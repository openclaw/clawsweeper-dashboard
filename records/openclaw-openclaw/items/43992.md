## Summary

Keep open: the PR is useful source work for a real current-main bug, but it should not merge as-is because it is stale against the active subagent run-manager path and still misses normal result envelopes plus a lifecycle/dedupe race.

## What This Changes

The PR adds `agent.wait` result payload forwarding through gateway snapshots/responses, reuses that text as subagent announce/registry fallback content, and adds gateway/subagent regression coverage.

## Best Possible Solution

Land a current-main port that carries result payloads through dedupe, `agent.wait`, and `waitForAgentRun`, parses `{ payloads, meta }` envelopes, freezes fallback text in the active run manager only when chat-history capture has not already won, and includes focused regressions plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main writes completed agent command output to `payload.result`, then drops it from terminal snapshots, `agent.wait` responses, `waitForAgentRun`, and the active subagent run-manager completion path.

## Solution Assessment

No. The PR is directionally useful, but it is not the best current fix because it targets an obsolete registry wait path, misses normal `{ payloads, meta }` envelope parsing, and can still return a lifecycle snapshot before the result-bearing dedupe snapshot is available.
