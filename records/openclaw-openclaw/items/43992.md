## Summary

Keep open: the underlying result-propagation bug is still present on current main, but this PR is not merge-ready because its branch is stale/conflicting, it misses active current-main paths, and it lacks real after-fix behavior proof.

## What This Changes

The PR forwards `agent.wait` result payloads through gateway wait snapshots/responses and reuses them as fallback text for subagent completion announces, with regression tests.

## Best Possible Solution

Land a current-main port that carries result payloads through dedupe, `agent.wait`, and `waitForAgentRun`, parses `{ payloads, meta }` envelopes, and freezes fallback text in the active run manager only when chat-history capture has not already won.

## Reproduction Assessment

Yes. Source inspection on current main shows the completed agent output is stored as `payload.result` and then dropped by the terminal snapshot, `agent.wait`, `waitForAgentRun`, and active subagent run-manager paths.

## Solution Assessment

No. The PR is directionally useful, but it is not the best current fix because it targets an obsolete registry path, misses normal `{ payloads, meta }` result envelopes, and can still return a lifecycle snapshot before the result-bearing dedupe snapshot is available.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Parse agent.wait result payload envelopes:** `src/agents/subagent-announce.ts:345-347`
  - body: `extractAgentWaitResultText` only checks direct scalar fields before delegating to `extractToolResultText`, but normal agent runs return `{ payloads, meta }`. In the history-catch-up case this still drops `result.payloads[*].text`, so the fallback can miss the completion text it is meant to preserve.
  - confidence: 0.91
- **[P1] Port the fallback into the run manager:** `src/agents/subagent-registry.ts:1262-1264`
  - body: This fallback writes to the older inline registry wait path, while current main completes registered subagent waits through `createSubagentRunManager` and `waitForAgentRun`. After rebase, the active path would still not freeze or announce the `agent.wait` result.
  - confidence: 0.9
- **[P2] Wait for the result-bearing dedupe snapshot:** `src/gateway/server-methods/agent.ts:775`
  - body: The live `agent.wait` path can return the lifecycle snapshot first and abort the dedupe waiter. Because lifecycle end can fire before the gateway writes the dedupe payload that carries `result`, this can still return `undefined` during the catch-up window this PR is trying to fix.
  - confidence: 0.85
