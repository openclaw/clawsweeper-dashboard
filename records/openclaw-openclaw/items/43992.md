## Summary

Keep open. The underlying `agent.wait` result propagation bug is still present on current `main`, but this PR is not merge-ready because it misses the normal `{ payloads, meta }` result envelope, applies the subagent fallback to an older registry path instead of the active run-manager path, and can still return before the result-bearing dedupe snapshot is available.

## What This Changes

The PR threads `agent.wait` result payloads through gateway dedupe snapshots and responses, uses that text as a subagent announce/registry fallback, and adds gateway plus subagent regression tests.

## Best Possible Solution

Land a current-main implementation that carries `agent.wait` result payloads end to end, parses normal `{ payloads, meta }` envelopes, waits or merges carefully when lifecycle and dedupe terminal signals arrive in different orders, and freezes subagent result text only as a fallback before completion delivery.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main stores `payload.result` but drops it from dedupe snapshots, `agent.wait` responses, `waitForAgentRun`, and the active subagent run-manager completion path.

## Solution Assessment

No. The PR is directionally useful, but it is not the best current fix because it targets an obsolete registry wait path, misses the normal result envelope shape, and does not guarantee a result-bearing snapshot when lifecycle wins the live wait race.

  - confidence: 0.9

  - body: The live `agent.wait` path returns the first terminal snapshot and aborts the other waiter. If lifecycle wins before the gateway dedupe entry writes `payload.result`, the added `result` field can still be `undefined`, so callers miss the completion payload.
  - confidence: 0.85
