## Summary

Keep this PR open, but it is not merge-ready: the bug is real and the direction is useful, yet the branch is conflicting, lacks real behavior proof, and misses current-main result propagation paths.

## What This Changes

The PR forwards `agent.wait` result payloads through gateway dedupe/responses, reuses them as fallback subagent announce text, and adds gateway/subagent regression tests.

## Best Possible Solution

Land a rebased fix that carries terminal results through gateway snapshots, `agent.wait`, `waitForAgentRun`, and the active subagent run manager while parsing normal payload envelopes and keeping chat-history capture authoritative.

## Reproduction Assessment

Yes, at source level. Current main stores completed agent output in `payload.result`, then drops it from terminal snapshots, `agent.wait` responses, `waitForAgentRun`, and the active subagent completion path.

## Solution Assessment

No. The PR is directionally useful, but the current diff misses current-main run-manager plumbing, normal `{ payloads, meta }` result-envelope parsing, and the lifecycle-before-dedupe race.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Parse agent.wait result payload envelopes:** `src/agents/subagent-announce.ts:345-347`
  - body: `extractAgentWaitResultText` passes `result` directly to `extractToolResultText`, but normal agent runs return `result: { payloads, meta }`. That leaves `payloads[*].text` invisible during the history catch-up window, so the fallback can still produce no completion reply.
  - confidence: 0.91
- **[P1] Port the fallback into the run manager:** `src/agents/subagent-registry.ts:1262-1264`
  - body: This writes fallback text into the older inline registry wait path, while current main completes registered subagent waits through `createSubagentRunManager` and `waitForAgentRun`. A current-main port would still complete and announce without the preserved wait result.
  - confidence: 0.9
- **[P2] Wait for the result-bearing dedupe snapshot:** `src/gateway/server-methods/agent.ts:775`
  - body: The live `agent.wait` path can return the lifecycle snapshot first and abort the dedupe waiter, but lifecycle snapshots do not carry `result`. Since normal runs can emit lifecycle end before writing the dedupe payload, this still returns `undefined` in the target catch-up window.
  - confidence: 0.85
