## Summary

Keep this PR open: the bug is still source-reproducible on current main, and the branch is useful but not mergeable as-is because it misses current run-manager plumbing, normal result envelopes, the lifecycle-vs-dedupe race, and contributor real behavior proof.

## What This Changes

The branch preserves terminal `agent.wait` result payloads through gateway wait snapshots/responses and tries to reuse them as fallback subagent completion text, with focused gateway and subagent regression tests.

## Best Possible Solution

Port a consolidated fix onto current main so result payloads flow through dedupe, `agent.wait`, `waitForAgentRun`, and `createSubagentRunManager`, while decoding normal `{ payloads, meta }` results and keeping chat-history captures authoritative.

## Reproduction Assessment

Yes, at source level. Current main stores completed agent output as `payload.result`, but the wait snapshot, `agent.wait` response, wait wrapper, and run-manager completion path do not carry that result into fallback delivery.

## Solution Assessment

No. The direction is useful, but this branch is not the best mergeable solution until it is rebased onto the current run-manager path, decodes `{ payloads, meta }`, avoids returning result-less lifecycle snapshots first, and includes real behavior proof.
