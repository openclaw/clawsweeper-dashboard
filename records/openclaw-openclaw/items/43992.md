## Summary

Keep open for maintainer/contributor follow-up: the bug is source-reproducible and the direction is useful, but this PR is not mergeable as-is because it misses current-main run-manager/result-envelope paths, can still lose result-bearing dedupe payloads, and lacks required real behavior proof.

## What This Changes

The PR threads terminal `agent.wait` result payloads through gateway wait snapshots/responses and reuses them as fallback subagent completion text with focused tests.

## Best Possible Solution

Port a consolidated fix onto current main so result payloads flow through dedupe, `agent.wait`, `waitForAgentRun`, and `createSubagentRunManager`, while decoding normal `{ payloads, meta }` results and keeping chat-history captures authoritative.

## Reproduction Assessment

Yes, at source level. Current main stores completed agent output as `payload.result`, but the current wait snapshot, `agent.wait` response, `waitForAgentRun`, and run-manager completion paths do not carry that result into subagent completion fallback delivery.

## Solution Assessment

No. The PR direction is useful, but this branch is not the best mergeable solution until it is ported to the current run-manager/wait paths, decodes `{ payloads, meta }`, avoids returning lifecycle snapshots before the result-bearing dedupe payload is available, and includes real behavior proof.

  - confidence: 0.85
