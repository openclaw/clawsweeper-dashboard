## Summary

Keep open. Current main still drops the stored agent result before it reaches `agent.wait`, `waitForAgentRun`, and active subagent completion, so the bug is real, but this PR is not merge-ready because it targets stale registry plumbing and misses normal result envelopes plus a live-wait ordering race.

## What This Changes

The PR threads `agent.wait` result payloads through gateway dedupe snapshots/responses, reuses that text as a subagent fallback, and adds gateway plus subagent regression coverage.

## Best Possible Solution

Land a current-main port that carries result payloads through dedupe, `agent.wait`, and `waitForAgentRun`, parses `{ payloads, meta }` envelopes, and freezes fallback text in the active run manager only when chat-history capture has not already won.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main stores `payload.result` but drops it from terminal snapshots, `agent.wait` responses, the run-wait helper, and the active subagent completion path.

## Solution Assessment

No. The PR is directionally useful, but it is not the best current fix because it targets an obsolete registry wait path, misses normal result-envelope parsing, and still lets lifecycle completion beat the result-bearing dedupe snapshot.
