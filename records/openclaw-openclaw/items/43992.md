## Summary

Keep open. Current main still drops the result payload before it reaches `agent.wait`, `waitForAgentRun`, and the active subagent run-manager completion path, but this PR is not merge-ready because it needs a current-main port, fixes for the reviewed edge cases, and real behavior proof from the contributor.

## What This Changes

The PR threads `agent.wait` result payloads through gateway dedupe/responses, uses that text as a subagent announce/registry fallback, and adds gateway/subagent regression tests.

## Best Possible Solution

Land a current-main port that carries result payloads through dedupe, `agent.wait`, and `waitForAgentRun`, parses `{ payloads, meta }` envelopes, freezes fallback text in the active run manager only when chat-history capture has not already won, and coordinates with the remaining native subagent delivery work.

## Reproduction Assessment

Yes, at source level. Current main writes completed agent output into `payload.result`, then drops it from terminal snapshots, `agent.wait` responses, `waitForAgentRun`, and the active subagent completion path.

## Solution Assessment

No. The PR is directionally useful, but it is not the best current fix because it misses current-main run-manager plumbing, normal result-envelope parsing, and the lifecycle-before-dedupe race.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Parse result payload envelopes:** `src/agents/subagent-announce.ts:345-347`
  - body: `extractAgentWaitResultText` only checks direct scalar fields before delegating to tool-result text extraction, but normal agent runs return `{ payloads, meta }`. In the history-catch-up case this still drops `result.payloads[*].text`, so the fallback can miss the completion text it is meant to preserve.
  - confidence: 0.91
- **[P1] Port fallback into the run manager:** `src/agents/subagent-registry.ts:1262-1264`
  - body: This fallback writes to the older inline registry wait path, while current main completes registered subagent waits through `createSubagentRunManager` and `waitForAgentRun`. After rebase, the active path would still not freeze or announce the `agent.wait` result.
  - confidence: 0.9
- **[P2] Wait for result-bearing snapshots:** `src/gateway/server-methods/agent.ts:772`
  - body: The live `agent.wait` path can return the lifecycle snapshot first and abort the dedupe waiter. Since lifecycle end can fire before the gateway writes the dedupe payload that carries `result`, this can still return `undefined` during the catch-up window this PR is trying to fix.
  - confidence: 0.85
