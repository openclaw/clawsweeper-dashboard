## Summary

Keep open: the underlying bug is still present on current main, but this PR is not merge-ready because it misses current-main result plumbing, does not parse normal result envelopes, can still lose results when lifecycle wins the wait race, and lacks real behavior proof.

## What This Changes

The branch preserves `agent.wait` result payloads in gateway wait snapshots/responses and reuses them as fallback subagent announce/registry completion text with regression tests.

## Best Possible Solution

A rebased fix should make result payloads part of the wait contract through dedupe, `agent.wait`, `waitForAgentRun`, and `createSubagentRunManager`, decode normal payload envelopes, and leave chat-history capture authoritative.

## Reproduction Assessment

Yes, at source level. Current main writes completed gateway agent output into `payload.result`, then drops it from terminal snapshots, `agent.wait`, `waitForAgentRun`, and the active subagent run-manager completion path.

## Solution Assessment

No. The PR is directionally useful, but the current diff is based on an older inline registry path and misses the current run-manager plumbing, normal `{ payloads, meta }` parsing, and the lifecycle-before-dedupe race.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Parse agent.wait result payload envelopes:** `src/agents/subagent-announce.ts:345-347`
  - body: `extractAgentWaitResultText` passes `result` directly to `extractToolResultText`, but normal gateway agent runs return `{ payloads, meta }`. That leaves `payloads[*].text` invisible during the history catch-up window, so the fallback can still produce no completion reply.
  - confidence: 0.91
- **[P1] Port the fallback into the run manager:** `src/agents/subagent-registry.ts:1262-1264`
  - body: This writes fallback text into the older inline registry wait path, while current main completes registered subagent waits through `createSubagentRunManager` and `waitForAgentRun`. A current-main port would still complete and announce without the preserved wait result.
  - confidence: 0.9
- **[P2] Wait for the result-bearing dedupe snapshot:** `src/gateway/server-methods/agent.ts:772`
  - body: The live `agent.wait` path can return the lifecycle snapshot first and abort the dedupe waiter, but lifecycle snapshots do not carry `result`. Normal runs can emit lifecycle end before the gateway writes the result-bearing dedupe payload, so this still returns `undefined` in the target catch-up window.
  - confidence: 0.85
