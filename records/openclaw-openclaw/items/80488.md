## Summary

Keep open: this is a plausible bug fix for a defined per-model `api` contract, but the current PR is draft, still misses the normal runtime auth entry point, and lacks after-fix real behavior proof.

## What This Changes

The PR threads an optional `modelApi` into provider synthetic-auth hook lookup and adds tests plus model-resolution changes for per-model `api` and `baseUrl` overrides.

## Best Possible Solution

Carry the resolved model's `api` into all provider-auth hook discovery paths, add production-entry regression coverage, and prove the patched runtime for https://github.com/openclaw/openclaw/issues/80487 with private details redacted.

## Reproduction Assessment

Yes, source-reproducible: current main accepts per-model `api` and `baseUrl`, but the auth hook-ref and `getApiKeyForModel` path still use only the provider-level API. I did not run live tests because this was a read-only review.

## Solution Assessment

No. Passing `modelApi` into the lower helper is the right direction, but the PR must also thread `params.model.api` from `getApiKeyForModel` so the normal runtime path exercises the fix.
