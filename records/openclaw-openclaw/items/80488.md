## Summary

Keep this PR open: it targets a real per-model `api`/`baseUrl` bug, but the latest diff is still incomplete in the production auth path and has focused test gaps that would let the broken path survive.

## What This Changes

The PR adds a `modelApi` argument for provider synthetic-auth hook discovery, applies configured model `api`/`baseUrl` in selected model transport paths, and adds auth/model regression tests.

## Best Possible Solution

Finish the narrow bug fix by carrying the effective model `api` and `baseUrl` through production auth and every configured model resolution path, with regression tests that exercise `getApiKeyForModel`, inline models, fallback models, and discovered models using realistic local model ids.

## Reproduction Assessment

Yes, source-reproducible: current main accepts and documents per-model `api`/`baseUrl`, while auth hook refs and the runtime `getApiKeyForModel` entry still use the provider-level API. I did not run the Docker/router flow in this read-only pass.

## Solution Assessment

No. The direction is right, but the patch still misses the normal auth entry point, leaves plugin deferral on provider-level API, and does not prove or fix the inline configured model path for model-level `baseUrl`.
