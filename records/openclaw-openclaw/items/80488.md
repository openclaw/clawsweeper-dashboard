## Summary

Keep open. The reported bug is valid, but this draft PR does not yet fix the real model-auth path because the runtime model resolver still drops `model.api`, and the external PR also lacks after-fix real behavior proof.

## What This Changes

The branch adds a `modelApi` parameter to synthetic provider auth hook resolution and a unit test for an Ollama model API override under an OpenAI-compatible custom provider.

## Best Possible Solution

Thread the effective `model.api` from `getApiKeyForModel` through provider synthetic auth lookup, including profile deferral, and cover that real model path with a focused regression test plus redacted runtime proof.

## Reproduction Assessment

Yes, by source inspection: model-level `api` is accepted and preserved on runtime models, but current auth lookup passes only `model.provider` into provider auth and hook refs use only provider-level `api`. I did not run the reproduction because this review is read-only.

## Solution Assessment

No. Adding `modelApi` to the provider resolver is the right direction, but the PR must pass the effective model API from the real model-auth caller and prove that path, otherwise the reported runtime flow remains broken.
