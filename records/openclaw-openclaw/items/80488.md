## Summary

Keep open. This is a plausible focused bug fix with sufficient after-fix router logs, but the latest diff still has one discovered-model edge case that can regress existing `models-add` stale metadata handling, and the PR is currently draft.

## What This Changes

The PR threads per-model `api` through provider auth lookup and makes model-level `api`/`baseUrl` override provider-level transport selection in configured model resolution.

## Best Possible Solution

Finish the narrow fix by honoring per-model transport/auth overrides for normal configured rows while preserving the `models-add` rule that official runtime metadata can override stale generated rows.

## Reproduction Assessment

Yes. Current main accepts per-model `api`/`baseUrl`, while source inspection shows auth hook lookup and several model-resolution paths still use provider-level values; the PR body also includes runtime router logs for the reported mixed-provider setup.

## Solution Assessment

No. The approach is the right narrow owner-boundary fix, but the discovered-model override should not let `metadataSource: "models-add"` rows override runtime/discovered transport metadata.
