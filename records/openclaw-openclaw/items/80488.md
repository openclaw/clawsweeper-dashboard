## Summary

Keep this PR open, but it should not merge as-is: the reported auth bug is valid, yet the patch only wires `modelApi` into the lower-level provider helper while the real runtime entry point still drops `model.api`. The PR is also draft and its body explicitly says the patched behavior was not verified at runtime.

## What This Changes

The branch threads an optional `modelApi` into provider synthetic-auth hook lookup, adds regression coverage, and updates fallback model transport selection to prefer per-model `api` and `baseUrl` overrides.

## Best Possible Solution

Amend the patch so `getApiKeyForModel` and any synthetic-profile deferral path carry `model.api` into provider auth hook discovery, then prove the real model-auth/chat path with a patched build or redacted runtime output.

## Reproduction Assessment

Yes, from source inspection rather than a live run: current main accepts per-model `api`, but `getApiKeyForModel` does not pass that API into the provider auth resolver. That makes the reported failure path source-reproducible for normal runtime auth.

## Solution Assessment

No. Adding `modelApi` to the lower-level helper is the right direction, but this PR misses the production caller and tests only the manually supplied helper path.
