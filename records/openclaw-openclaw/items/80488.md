## Summary

Keep open: the bug is plausible, but this PR is not mergeable as-is because the normal model auth path still never supplies the new `modelApi` value, and the external after-fix runtime proof is missing.

## What This Changes

The PR adds `modelApi` plumbing to synthetic provider auth hook-reference resolution and a regression test for resolving Ollama auth under a mixed custom provider.

## Best Possible Solution

Forward the selected `model.api` from `getApiKeyForModel` into provider auth resolution, thread it through profile deferral where needed, and prove the mixed-provider Ollama path with wrapper-level tests plus redacted real runtime output.

## Reproduction Assessment

Yes, at source level: current main resolves and streams with the model-level API, but auth hook refs are still based on provider-level API and the normal auth wrapper drops `model.api`. I did not run a live Docker/Ollama reproduction in this read-only review.

## Solution Assessment

No. The direction is right, but this PR only makes the lower-level provider helper accept `modelApi`; the production path enters through `getApiKeyForModel`, which still does not forward the selected model's effective API.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Forward the model API from getApiKeyForModel:** `src/agents/model-auth.ts:545`
  - body: The new `modelApi` parameter is never populated by the normal model auth path. Chat and other runtime callers pass a full model into `getApiKeyForModel`, but that wrapper still forwards only `params.model.provider` to `resolveApiKeyForProvider`, so a model with `api: "ollama"` under an `openai-completions` provider still reaches synthetic auth with `modelApi` undefined. Please pass `params.model.api` through this wrapper and cover that caller path rather than only testing `resolveApiKeyForProvider` directly.
