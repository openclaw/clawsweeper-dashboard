## Summary

Keep open: the reported bug is source-reproducible and this draft PR is still incomplete because the normal model-auth entry point does not pass the effective model `api` into the new hook lookup path, and the PR still lacks after-fix real runtime proof.

## What This Changes

The PR adds a `modelApi` parameter to provider synthetic-auth hook lookup, threads it through provider auth helpers, and updates fallback/discovered model transport resolution tests for per-model `api` and `baseUrl` overrides.

## Best Possible Solution

Thread the effective `model.api` through `getApiKeyForModel` into provider synthetic-auth hook discovery, cover that production auth path with a regression test, and require redacted patched-runtime proof before merge.

## Reproduction Assessment

Yes, at source level: current main accepts per-model `api`, resolves model/stream paths with the effective model API, but auth hook refs only see the provider-level API. I did not run a live Docker/Ollama reproduction in this read-only review.

## Solution Assessment

No. Adding `modelApi` to the lower-level helper is the right direction, but this PR still misses `getApiKeyForModel`, so the production runtime auth path can continue to drop the model-level override.
