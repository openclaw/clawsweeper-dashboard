## Summary

Keep open: the reported bug appears valid, but this PR is not mergeable yet because the real model-auth path still drops the model-level API override and the supplied proof explicitly lacks after-fix runtime verification.

## What This Changes

The PR adds an optional model-level API override to provider synthetic-auth hook resolution and adds a regression test for mixed provider/model API configs.

## Best Possible Solution

Thread the effective resolved model API through the real model-auth callers and synthetic profile deferral, cover the getApiKeyForModel runtime path, and require redacted after-fix Docker/Ollama or equivalent runtime proof before merge.

## Reproduction Assessment

Yes by source inspection: a resolved model can carry a model-level api, but getApiKeyForModel currently calls resolveApiKeyForProvider without that value while hook refs read only providerConfig.api. I did not run the live Docker/Ollama path in this read-only review.

## Solution Assessment

No. Using the effective model API for hook discovery is the right direction, but the PR currently exposes a low-level optional parameter without wiring it from real model-auth callers or a valid regression test.
