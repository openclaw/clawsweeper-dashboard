## Summary

Keep open: the patch is aimed at the right auth/plugin boundary, but it does not fix the reported runtime path because model-based callers still drop the effective `model.api`; after-fix real behavior proof is also absent.

## What This Changes

The branch adds an optional `modelApi` parameter to synthetic provider-auth hook resolution and a regression test for an Ollama API override under an OpenAI-compatible custom provider.

## Best Possible Solution

Thread the effective `params.model.api` from `getApiKeyForModel` into synthetic auth hook reference resolution, cover that production wrapper path with a focused regression test, and require real after-fix runtime proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main supports per-model `api` in config and stream registration, but auth hook refs only see `providerConfig.api`. I did not run a live Docker/Ollama reproduction in this read-only review.

## Solution Assessment

No. Passing the effective model API into hook discovery is the right direction, but this patch stops short of the production `getApiKeyForModel` entry point used by chat and simple-completion flows.
