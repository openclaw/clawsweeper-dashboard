## Summary

Keep open: the bug report is credible, but the patch is not mergeable because the new modelApi value is only added to the lower-level resolver and is not passed from getApiKeyForModel, which is what runtime request paths use. The external PR also lacks after-fix real behavior proof.

## What This Changes

The PR adds a modelApi override parameter to provider synthetic-auth hook resolution and a unit test for a custom provider whose model overrides api to ollama.

## Best Possible Solution

Thread the effective model api from getApiKeyForModel into the provider auth and deferral hook lookup path, cover that wrapper/runtime path with a regression test, then add redacted after-fix runtime proof.

## Reproduction Assessment

Yes, from source inspection rather than a live run: current main accepts per-model api in config and resolves it onto the model, while the auth wrapper drops model.api before provider hook discovery. I did not run a Docker/Telegram repro in this read-only review.

## Solution Assessment

No. The lower-level hook-ref change is in the right area, but the PR does not connect it to getApiKeyForModel, so actual runtime requests still miss the model-level api override.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Thread model API through the model auth wrapper:** `src/agents/model-auth.ts:545`
  - body: The new modelApi parameter is only supplied by the direct resolveApiKeyForProvider test path. Real requests call getApiKeyForModel with the resolved model, and that wrapper still forwards only model.provider into resolveApiKeyForProvider, so a model whose effective api is ollama under an openai-completions provider will still reach hook discovery without the override. Pass params.model.api through the wrapper and cover that path.
  - confidence: 0.92
