## Summary

Keep open. The report matches OpenClaw's documented silent-reply contract and describes a privacy-impacting Telegram leak, but this read-only sweep did not prove the exact current-main branch that turns a successful empty agent result into a Telegram message containing the whole gateway run envelope.

## What This Changes

The issue asks OpenClaw to suppress Telegram delivery when an agent run's final assistant message is exactly `NO_REPLY` instead of sending the raw run-result envelope JSON.

## Best Possible Solution

Keep the documented silent-reply behavior and fix the agent/gateway-to-Telegram boundary so only reply payloads are deliverable; successful empty or silent runs must never fall back to channel-sending run metadata.

## Reproduction Assessment

Unclear. The issue includes concrete live evidence from OpenClaw 2026.5.6, and source confirms both the silent-reply contract and the gateway envelope shape, but I did not establish the exact current-main path that hands that envelope to Telegram delivery.

## Solution Assessment

Unclear because there is no proposed patch. The narrow maintainable fix should be at the delivery boundary that treats an agent RPC response as user text, not a global change to `NO_REPLY` policy.
