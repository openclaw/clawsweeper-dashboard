## Summary

Close as implemented on main: the agent delivery path now stops after `NO_REPLY` normalizes to an empty delivery plan, so an empty/silent result cannot fall through to channel delivery as raw run metadata.

## What This Changes

The issue asks for Telegram agent turns whose final assistant text is exactly `NO_REPLY` to be suppressed instead of delivering the gateway run envelope JSON.

## Best Possible Solution

Keep the current delivery-boundary fix: projected empty/silent agent results should be treated as terminal suppressed delivery, with no channel fallback to run metadata.

## Reproduction Assessment

Yes for the reported older path: the issue includes concrete live Telegram evidence, and `v2026.5.7` source shows delivery proceeded based on the original payload list rather than the projected deliverable payloads. Current main no longer reproduces that path because the projected empty plan returns before channel delivery.

## Solution Assessment

Yes. The fix is at the narrow agent delivery boundary where payloads are projected for outbound delivery, so `NO_REPLY` suppression is preserved without changing Telegram policy or the gateway envelope contract.
