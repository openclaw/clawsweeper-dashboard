## Summary

Keep open: the report describes broken documented silent-reply behavior, and current source still contains a credible path where a silent `NO_REPLY` run normalizes to empty payloads while the gateway `agent` RPC terminal envelope remains available as structured output. The exact Telegram caller that forwarded that envelope as channel text still needs maintainer review before patching.

## What This Changes

The issue reports that a final assistant message of exactly `NO_REPLY` should suppress Telegram delivery, but a full run-result envelope JSON was delivered instead.

## Best Possible Solution

Keep the structured `agent` RPC response for API clients, but add a Telegram/gateway delivery guard so completed runs with empty normalized payloads are treated as silent and the run envelope is never serialized into channel text.

## Reproduction Assessment

No live Telegram reproduction was established in this read-only review. Source inspection gives a credible current-main reproduction path because the reported JSON matches the gateway `agent` terminal envelope while `NO_REPLY` delivery paths normalize to empty payloads.

## Solution Assessment

No proposed patch is present. The maintainable fix should be at the boundary that forwards or interprets the run result for Telegram delivery, not by removing structured terminal results from the gateway API.
