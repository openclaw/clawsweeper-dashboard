## Summary

Keep this PR open, but do not merge it in its current form: the SQLite single-owner direction fits repository policy, yet the replacement run cannot take over the interrupted run’s same-session route, so a second restart can lose the delivery target. The branch is also currently conflicted with main and the PR body provides harness-based validation rather than real restart and doctor proof.

## What This Changes

The PR moves restart-recovery delivery route metadata from session JSON into the shared SQLite state database, strips the legacy runtime fields, and adds doctor migration plus focused regression tests.

## Best Possible Solution

Keep SQLite as the sole runtime owner, but make recovery dispatch atomically transfer the interrupted same-session route to the replacement run, retain doctor as the one legacy migration owner, and verify a legacy upgrade plus two consecutive gateway restarts in a real process flow before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence sequence: retain the interrupted run’s row, dispatch the replacement run with a new run ID, observe its claim rejected, clear the old row, and restart again while the replacement turn is active.

## Solution Assessment

No, not in its current form. SQLite plus doctor migration is the correct ownership direction, but the replacement-run takeover must be atomic and the upgrade/runtime behavior needs real proof.
