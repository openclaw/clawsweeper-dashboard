## What This Changes

The issue reports Windows Codex harness startup failures in v2026.5.18 after legacy model/runtime config and a malformed `appServer.command` override, and asks for clearer validation, doctor checks, and recovery guidance.

## Best Possible Solution

Keep managed Codex startup as the default and add focused validation, doctor linting, and error/docs guidance that tells users to remove `appServer.command` or configure executable plus `args` correctly on Windows.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does confirm that current main keeps `command` and `args` separate, validates `command` as a string, and does not reject shell-like Windows command strings before app-server spawn.

## Solution Assessment

Yes, a narrow validation and diagnostics fix is the best direction. Splitting arbitrary command strings would expand shell parsing semantics; rejecting or doctor-warning malformed `command` values while preserving `args` is safer.
