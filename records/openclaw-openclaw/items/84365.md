## Summary

Keep open: current main has the managed Codex startup path and legacy Codex CLI migration, but the malformed Windows `appServer.command` path is still source-reproducible and no doctor/runtime diagnostic catches the combined `node C:\...\codex.js` shape before startup fails.

## What This Changes

The issue requests clearer Codex app-server override validation and doctor guidance for Windows startup failures when `appServer.command` is misconfigured.

## Best Possible Solution

Add targeted Codex `appServer.command` validation and doctor lint that flags executable-plus-arguments command strings, points users to managed startup or `appServer.args`, and keeps managed Codex startup as the default.

## Reproduction Assessment

Yes for the malformed command path by source inspection: a combined `node C:\...\codex.js` string can pass through config, reach the Windows spawn resolver, and be treated as one `.js` entrypoint. No live Windows/npm reproduction was established for the managed-binary discovery part.

## Solution Assessment

Yes, a narrow diagnostic fix is the best path: reject or warn about combined command strings before runtime and direct users to managed startup or `appServer.args`, rather than adding broad shell splitting heuristics.
