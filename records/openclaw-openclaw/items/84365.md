## Best Possible Solution

Add targeted Codex `appServer.command` validation and doctor lint for executable-plus-args command strings, improve the managed-binary recovery guidance, and keep managed startup plus separate `appServer.args` as the supported path.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/84365. On current main, Codex `plugins.entries.codex.config.appServer.command` accepts a raw string like `node C:\\Users\\<user>\\.openclaw\\npm\\node_modules\\@openai\\codex\\bin\\codex.js`; on Windows the shared spawn resolver can treat that whole string as one `.js` entrypoint and fail with `Cannot find module ...node C:\\...codex.js`. Add a narrow validation/doctor diagnostic that flags executable-plus-args command strings for Codex app-server overrides, points users to remove the override or put extra argv in `appServer.args`, and improves managed-binary missing guidance so it does not lead Windows users into the same bad shape. Keep OpenClaw's managed Codex startup as the default, do not add shell splitting or broad command parsing, do not add a new config option, and preserve existing explicit executable-only overrides. Add focused regression tests for config/doctor warning behavior and the Windows spawn path as needed; update Codex harness docs only if the implemented diagnostic changes user-visible wording. Changelog is expected if the fix changes user-facing diagnostics.
