## Summary

This PR is still necessary and targets a valid Discord workflow, but it should not merge yet: deletion cancellation starts only after a queue job exists, so a message deleted during inbound debounce can still be processed, and the external contribution has no real Discord behavior proof.

## What This Changes

The branch adds Discord message-update reprocessing, source-message-scoped cancellation for deletes and superseding edits, listener registration, revision-aware replay keys, focused tests, and Discord docs.

## Best Possible Solution

Use one source-message lifecycle that cancels matching buffered entries through the existing debouncer and queued or active jobs through the run queue, preserves replay and typing cleanup, and is verified with a real Discord edit/delete roundtrip.

## Reproduction Assessment

Yes for the introduced defect: current source shows debounce precedes queue insertion, while the branch exposes deletion cancellation only through the run queue. The PR body also explicitly reports the same debounce-window behavior.

## Solution Assessment

No. The overall source-message key design is reasonable, but queue-only cancellation is not the narrowest complete solution because the existing debouncer already provides keyed cancellation for pending work.
