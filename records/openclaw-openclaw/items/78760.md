## Label Justifications

- `P2`: This is a normal-priority cost-trust and provider-auth semantics issue with real user confusion but no confirmed outage, data loss, or security bypass.
- `impact:auth-provider`: The reported behavior depends on distinguishing OpenAI/Codex OAuth subscription access from OpenAI Platform API-key billing for the same model family.
- `impact:other`: The issue affects maintainer-visible cost accounting and operator trust in session logs, which does not fit the more specific impact labels.

## Summary

Keep open: current main improved the adjacent unpriced-model `$0` case and docs now call displayed costs estimated, but the core issue remains because persisted usage still exposes a required numeric `usage.cost.total` without a billing mode or authoritativeness signal, and gateway/session cost workflows still consume that field as cost data.

## What This Changes

The issue requests clearer Codex OAuth billing semantics so session logs and COST_LOG-style workflows do not present subscription-style usage as authoritative per-token API spend.

## Best Possible Solution

Define a narrow persisted usage contract that keeps useful local cost estimates but marks Codex ChatGPT/OAuth usage as estimated, non-authoritative, subscription-backed, or otherwise distinguishable from OpenAI Platform API-key spend across transcripts, `/usage cost`, and status surfaces.

## Reproduction Assessment

Yes. Source inspection shows OpenClaw persists numeric `usage.cost.total`, computes estimates from local pricing, and aggregates transcript cost totals, while the Codex SDK and docs distinguish token usage from ChatGPT subscription versus API-key billing.

## Solution Assessment

Unclear until maintainers choose the contract. The issue's direction is sound, but the best implementation needs a deliberate decision between adding billing mode metadata, renaming/splitting estimated cost, or documenting a stable non-authoritative cost interpretation.
