## Summary

Keep open: the PR targets a real log-forging issue, but the current head still leaves `channel` unsanitized, adds a weaker duplicate sanitizer, can regress JavaScript plugin callers, and lacks required real behavior proof.

## What This Changes

The PR adds a channel-local log sanitizer for `error`, `reason`, and `target` values in shared channel logging helpers, plus a new Vitest suite for control-character log injection cases.

## Best Possible Solution

Use a small string-coercing wrapper around the shared terminal `sanitizeForLog` for `channel`, `target`, `reason`, and `error`, with regression tests for channel values, ANSI/OSC/C1/DEL controls, and JavaScript-style non-string inputs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main and the PR head interpolate raw channel logging fields, and a value containing a newline or control character can produce forged log output.

## Solution Assessment

No. The hardening direction is right, but this patch should reuse the existing shared sanitizer and sanitize every interpolated field without breaking loosely typed plugin callers.

## Review Findings

  - body: The patch still emits `params.channel` directly at the start of each log message. A channel value containing a newline or control character can still forge a separate log line, so sanitize `channel` in each helper and cover it in tests.

  - confidence: 0.88

  - body: This adds a second sanitizer that handles only C0 controls and DEL, while the existing terminal sanitizer also strips ANSI/OSC sequences and C1 controls. Delegate through the shared helper so log-safety behavior stays consistent.
  - confidence: 0.84
