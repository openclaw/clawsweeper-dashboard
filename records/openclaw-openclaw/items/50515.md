## Summary

Keep open: current main still has the channel log-forging surface, but this PR is not merge-ready because it leaves `channel` raw, duplicates a weaker sanitizer, can regress JavaScript plugin callers, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a channel-local sanitizer for `reason`, `target`, and `error` values in shared channel logging helpers, plus a new Vitest suite for control-character log injection cases.

## Best Possible Solution

Use a string-coercing wrapper around the shared terminal sanitizer for every interpolated channel logging field, with regression coverage for `channel`, ANSI/OSC/C1/DEL controls, and non-string JavaScript inputs.

## Reproduction Assessment

Yes. Source inspection on current main shows raw channel logging fields are interpolated into log messages, so caller-provided newline or control characters can forge log output.

## Solution Assessment

No. The hardening direction is right, but the branch should reuse the shared sanitizer, coerce values before sanitizing, and cover `params.channel` too.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Sanitize channel before emitting log lines:** `src/channels/logging.ts:16`
  - body: The patch still emits `params.channel` directly at the start of each log message. A channel value containing a newline or control character can still forge a separate log line, so sanitize `channel` in each helper and cover it in tests.
  - confidence: 0.93
- **[P2] Coerce values before replacing characters:** `src/channels/logging.ts:4-6`
  - body: `sanitizeForLog()` calls `.replace` on its input. These helpers are public plugin SDK surface, and JavaScript callers that pass numeric or object `target`/`reason` values would now throw where template interpolation previously stringified them.
  - confidence: 0.88
- **[P2] Reuse the shared terminal sanitizer:** `src/channels/logging.ts:4-6`
  - body: This adds a second sanitizer that handles only C0 controls and DEL, while the existing terminal sanitizer also strips ANSI/OSC sequences and C1 controls. Delegate through the shared helper so log-safety behavior stays consistent.
  - confidence: 0.84
