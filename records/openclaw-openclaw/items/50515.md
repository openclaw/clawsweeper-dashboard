- `merge-risk: 🚨 compatibility`: The PR changes public SDK runtime behavior by calling `.replace` on `target` and `reason` values that previously stringified through template interpolation.

## Summary

Keep open: the PR addresses a real source-reproducible log-forging problem, but it is not merge-ready because it leaves `channel` raw, duplicates a weaker sanitizer than the shared terminal helper, changes public SDK runtime coercion behavior, and lacks real behavior proof.

## What This Changes

The PR adds a local channel log sanitizer, applies it to `reason`, `target`, and `error` in shared channel logging helpers, and adds Vitest coverage for control-character log injection.

## Best Possible Solution

Use a string-coercing wrapper around the shared terminal `sanitizeForLog` for every interpolated channel logging field, with regression coverage for `channel`, ANSI/OSC/C1/DEL controls, and JavaScript non-string inputs.

## Reproduction Assessment

Yes. Source inspection on current main shows caller-provided channel logging fields are directly interpolated into log lines, so newline or control-character inputs can forge output; I did not run tests in this read-only review.

## Solution Assessment

No. The hardening direction is right, but the maintainable fix should reuse the shared sanitizer, coerce values before sanitizing, and cover every interpolated field including `channel`.
