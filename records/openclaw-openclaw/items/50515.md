## Summary

Keep open. Current main still has the unsanitized shared channel logging helpers, so the PR targets a real hardening gap, but the proposed patch is not merge-ready because it leaves the `channel` field log-forgeable, duplicates a weaker sanitizer than the existing shared log sanitizer, and changes loose JavaScript caller behavior from stringification to possible `TypeError`.

## What This Changes

The PR adds a local sanitizer in `src/channels/logging.ts`, applies it to `reason`, `target`, and `error` fields in channel logging helpers, and adds `src/channels/logging.test.ts` coverage for control-character log injection cases.

## Best Possible Solution

Revise the PR to use the shared terminal log sanitizer through a small channel-local wrapper that coerces values with `String(...)`, sanitize every interpolated field in these helpers, and add tests for channel injection, DEL/C1/ANSI stripping, and non-string runtime inputs.

## Reproduction Assessment

Yes. By inspection, `logInboundDrop({ channel: "x\nFAKE", reason: "drop", log })` still emits a forged second line on the PR branch, and a JavaScript caller passing a numeric `target` would hit `.replace` on a non-string value.

## Solution Assessment

No. The best maintainable fix is to centralize on the existing shared sanitizer, preserve previous runtime stringification behavior, and cover all fields that enter the log message rather than adding a weaker parallel helper.

  - body: The new sanitizer is not applied to `params.channel`, but that value is still interpolated into every helper output. A channel value containing a newline or other control character can still forge log entries after this patch, so sanitize a local `channel` value before composing each message.
  - confidence: 0.93
- **[P2] Coerce values before calling replace:** `src/channels/logging.ts:6`
  - body: `sanitizeForLog` accepts a string and immediately calls `.replace`. These helpers are public plugin SDK exports, and JavaScript plugins can pass loose runtime values; this patch changes previous template-literal stringification into a `TypeError` for values such as numeric `target` or `reason`. Coerce inside the sanitizer wrapper.
  - confidence: 0.88
- **[P2] Reuse the shared log sanitizer:** `src/channels/logging.ts:6`
  - body: This local helper only strips C0 controls and DEL, while current main already has a shared log sanitizer that also strips ANSI/OSC sequences and C1 controls. Reuse the shared helper so channel logs follow the repository-wide log-safety contract.
  - confidence: 0.84
