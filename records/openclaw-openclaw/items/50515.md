## Summary

Keep open: the PR targets a real current-main log-injection surface, but the proposed patch is incomplete and adds inconsistent sanitization behavior.

## What This Changes

The PR adds a local sanitizer for channel logging helper values and a new Vitest suite for control-character log injection cases.

## Best Possible Solution

Revise the PR to use a string-coercing wrapper around the shared terminal log sanitizer for every interpolated channel-log field, with regression tests and a changelog entry if this ships as user-facing hardening.

## Reproduction Assessment

Yes. Source inspection shows `logInboundDrop({ channel: "x\nFAKE", reason: "drop", log })` would still emit a forged second line on the PR branch because `channel` remains unsanitized; current `main` also directly interpolates these values.

## Solution Assessment

No. The direction is right, but the patch is not the best fix because it partially sanitizes the log line and duplicates a weaker sanitizer instead of reusing the repository-wide log-safety helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Sanitize the channel field before logging:** `src/channels/logging.ts:16`
  - body: The patch sanitizes `reason`, `target`, and `error`, but every helper still interpolates `params.channel` directly. A channel string containing a newline or other control character can still forge log lines, so sanitize the channel value before building each message and cover it in tests.
  - confidence: 0.93
- **[P2] Coerce values before calling replace:** `src/channels/logging.ts:6`
  - body: `sanitizeForLog` calls `.replace` on its input. These helpers are exported through plugin SDK subpaths and can be called by loose JavaScript plugins; before this patch template literals stringified values like numeric targets, while the new helper can throw.
  - confidence: 0.88
- **[P2] Reuse the shared log sanitizer:** `src/channels/logging.ts:4-6`
  - body: The new local sanitizer strips only C0 controls plus DEL, while the existing terminal sanitizer also strips ANSI/OSC and C1 controls. Reuse the shared helper through a small channel wrapper so channel logs follow the same log-safety contract as the rest of the repo.
  - confidence: 0.84
