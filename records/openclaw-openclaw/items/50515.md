## Summary

Keep open: this is a plausible security hardening PR, but the branch is still incomplete because it leaves the channel prefix unsanitized, adds a weaker duplicate sanitizer, can throw for non-string JavaScript callers, and has no real behavior proof.

## What This Changes

The PR adds a local sanitizer to channel logging helpers for error, reason, and target values, plus Vitest coverage for control-character log injection cases.

## Best Possible Solution

Use a string-coercing wrapper around the existing shared terminal sanitizer for every interpolated channel log value, then cover channel, DEL/C1/ANSI, and non-string runtime cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main logs channel logging fields through template interpolation, and the PR branch still logs params.channel directly; a value containing a newline would split the log line.

## Solution Assessment

No. The direction is right, but the maintainable fix is to reuse the existing shared sanitizer with explicit string coercion and apply it consistently to channel, target, reason, and error values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Sanitize the channel field before logging:** `src/channels/logging.ts:16`
  - body: The patch sanitizes reason, target, and error, but every helper still interpolates params.channel directly. A channel string containing a newline or other control character can still forge log lines, so sanitize the channel value before building each message and cover it in tests.
  - confidence: 0.93
- **[P2] Coerce values before sanitizing:** `src/channels/logging.ts:4-6`
  - body: sanitizeForLog calls .replace on its input. These helpers are public plugin SDK exports, and JavaScript plugin callers could pass numeric targets or other values that previously stringified through template literals; this patch would throw instead of logging.
  - confidence: 0.88
- **[P2] Reuse the shared log sanitizer:** `src/channels/logging.ts:4-6`
  - body: The new local sanitizer strips only C0 controls and DEL, while the existing terminal sanitizer also strips ANSI/OSC sequences and C1 controls. Reuse the shared helper through a small channel wrapper so channel logs follow the same log-safety contract as the rest of the repo.
  - confidence: 0.84
