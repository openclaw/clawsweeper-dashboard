## Summary

Keep open: this is useful security hardening, but the PR is incomplete because it leaves channel values unsanitized, adds a weaker duplicate sanitizer, can regress loose JavaScript callers, and still lacks real behavior proof.

## What This Changes

The PR adds a local sanitizer to channel logging helpers for error, reason, and target values, plus a new Vitest suite for log-injection cases.

## Best Possible Solution

Use a small string-coercing wrapper around the shared terminal sanitizeForLog for every interpolated channel log value, with coverage for channel, DEL/C1/ANSI, and non-string JavaScript callers.

## Reproduction Assessment

Yes. Source inspection shows current main and the PR branch template-interpolate channel log values, so a newline in params.channel still splits the emitted log line; the helper also has a clear TypeError path for non-string JS inputs.

## Solution Assessment

No. The direction is right, but the narrow maintainable fix is to reuse the shared terminal sanitizer with explicit string coercion and apply it consistently to channel, target, reason, and error values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Sanitize the channel field before logging:** `src/channels/logging.ts:16`
  - body: The patch still interpolates params.channel directly, so a channel value containing a newline or control character can forge log lines even after the new sanitizer runs on other fields. Sanitize the channel value before building each message and cover it in tests.
  - confidence: 0.93
- **[P2] Coerce values before sanitizing:** `src/channels/logging.ts:4-6`
  - body: sanitizeForLog calls .replace on its input. These helpers are exported through the plugin SDK, and JavaScript plugin callers that previously stringified numeric or object values through template literals would now throw instead of logging.
  - confidence: 0.88
- **[P2] Reuse the shared log sanitizer:** `src/channels/logging.ts:4-6`
  - body: This adds a second sanitizer that only handles C0 controls and DEL, while the existing shared sanitizer also strips ANSI/OSC sequences and C1 controls. Reuse the shared helper through a channel-specific wrapper so log-safety behavior stays consistent.
  - confidence: 0.84
