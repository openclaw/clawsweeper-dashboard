## Summary

Keep open. The hardening target is valid, but the branch is not merge-ready because it leaves `channel` unsanitized, adds a narrower duplicate sanitizer instead of reusing the shared terminal sanitizer, can regress JavaScript plugin callers by calling `.replace` on non-string values, and still lacks after-fix real behavior proof.

## What This Changes

The PR adds a channel logging sanitizer in `src/channels/logging.ts`, applies it to `reason`, `target`, and `error`, and adds Vitest coverage for control-character log injection.

## Best Possible Solution

Land this hardening only after every interpolated channel logging field goes through a string-coercing wrapper around the shared terminal sanitizer, with focused regression coverage and redacted runtime/log proof.

## Reproduction Assessment

Yes. Current main directly interpolates caller-provided channel logging fields, so a focused helper call with newline/control-character inputs would forge log output; I verified the path from source rather than running tests in the read-only checkout.

## Solution Assessment

No. The direction is right, but the maintainable fix should reuse the shared terminal sanitizer, coerce values before sanitizing, and cover every interpolated field including `channel`.

- **[P1] Reuse the shared log sanitizer:** `src/channels/logging.ts:4-6`
  - body: This adds a second `sanitizeForLog` that strips only C0 controls and DEL, while `src/terminal/ansi.ts` already handles ANSI/OSC, C1 controls, and DEL for log-safe interpolation. Reuse the shared helper through a small channel wrapper so this security behavior does not drift.
  - confidence: 0.88

  - body: The patch still prefixes each log message with raw `params.channel`. A crafted channel value containing a newline or control character can still forge another log line, so sanitize `channel` in all three helpers and cover it in tests.
  - confidence: 0.93
