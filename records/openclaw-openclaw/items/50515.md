## Summary

This PR should stay open because the hardening target is valid, but the current patch is not merge-ready: it duplicates a weaker sanitizer, leaves `channel` unsanitized, can throw for JavaScript plugin callers that previously stringified values, and still lacks contributor-provided real behavior proof.

## What This Changes

The PR adds a local sanitizer to shared channel logging helpers and new Vitest coverage for control-character log injection cases.

## Best Possible Solution

Use a small string-coercing wrapper around the shared terminal `sanitizeForLog`, apply it to every interpolated channel logging field including `channel`, and land it with focused regression coverage plus redacted terminal/log proof.

## Reproduction Assessment

Yes. From current-main source inspection, calling the shared channel logging helpers with newline or control-character values will interpolate those values directly into log output; I did not run tests in the read-only checkout.

## Solution Assessment

No. The hardening direction is right, but the maintainable fix should reuse the shared terminal sanitizer, coerce values before sanitizing, and cover every interpolated field including `channel`.
