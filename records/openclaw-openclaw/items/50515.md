## Summary

Keep open: the PR addresses a real log-forging concern, but it is still incomplete for a security-sensitive surface, has compatibility regressions for public SDK callers, and lacks required real behavior proof.

## What This Changes

The PR adds a local channel log sanitizer, applies it to error/reason/target values in shared channel logging helpers, and adds a new Vitest suite for control-character log injection cases.

## Best Possible Solution

Revise the PR to use a string-coercing wrapper around the shared terminal sanitizeForLog for channel, target, reason, and error, with tests for channel, ANSI/OSC/C1/DEL, and non-string JavaScript callers.

## Reproduction Assessment

Yes. Source inspection shows current main logs unsanitized channel helper values, and the PR diff still interpolates params.channel while adding a raw .replace sanitizer for other fields.

## Solution Assessment

No. The hardening direction is right, but the maintainable fix is to reuse the existing shared sanitizer and coerce values before sanitizing across every interpolated field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Sanitize the channel field before logging:** `src/channels/logging.ts:16`
  - body: The patch still interpolates params.channel directly, so a channel value containing a newline or control character can forge log lines even after the new sanitizer handles other fields. Sanitize the channel value before building each message and cover it in tests.
  - confidence: 0.93
- **[P2] Coerce values before sanitizing:** `src/channels/logging.ts:4-6`
  - body: sanitizeForLog calls .replace on its input. These helpers are exported through the plugin SDK, and JavaScript plugin callers that previously stringified numeric or object values through template literals would now throw instead of logging.
  - confidence: 0.88
- **[P2] Reuse the shared log sanitizer:** `src/channels/logging.ts:4-6`
  - body: This adds a second sanitizer that handles only C0 controls and DEL, while the existing shared sanitizer also strips ANSI/OSC sequences and C1 controls. Reuse the shared helper through a channel-specific wrapper so log-safety behavior stays consistent.
  - confidence: 0.84
