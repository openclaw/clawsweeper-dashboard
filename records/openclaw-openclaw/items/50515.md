## Summary

Keep open: the hardening direction is valid, but this PR still leaves a forgeable channel field, duplicates a weaker sanitizer than the shared log-safety helper, and can regress loose JavaScript plugin callers that previously relied on template-literal string coercion. Prior review comments in the PR discussion flag the same core gaps, so the branch needs revision before merge.

## What This Changes

The PR adds a local channel log sanitizer, applies it to reason/target/error fields in shared channel logging helpers, and adds Vitest coverage for control-character log injection cases.

## Best Possible Solution

Revise the PR to run every interpolated channel-log field through a string-coercing wrapper around the shared terminal sanitizeForLog helper, cover channel/ANSI/C1/non-string cases in tests, and add a changelog entry if this ships as user-facing hardening.

## Reproduction Assessment

Yes. Source inspection shows logInboundDrop({ channel: "x\nFAKE", reason: "drop", log }) would still emit params.channel directly on the PR branch, and current main directly interpolates the same logging fields.

## Solution Assessment

No. The PR points in the right direction, but the narrow maintainable fix is to reuse the repository-wide log sanitizer with explicit string coercion and apply it consistently to channel, target, reason, and error values.

  - body: The patch sanitizes reason, target, and error, but every helper still interpolates params.channel directly. A channel string containing a newline or other control character can still forge log lines, so sanitize the channel value before building each message and cover it in tests.
  - confidence: 0.93
- **[P2] Coerce values before sanitizing:** `src/channels/logging.ts:6`
  - body: sanitizeForLog calls .replace on its input. These helpers are public plugin SDK exports, and loose JavaScript plugin callers could pass numeric targets or other values that previously stringified through template literals; this patch would throw instead of logging.
  - confidence: 0.88
- **[P2] Reuse the shared log sanitizer:** `src/channels/logging.ts:4-6`
  - body: The new local sanitizer strips only C0 controls and DEL, while the existing terminal sanitizer also strips ANSI/OSC and C1 controls. Reuse the shared helper through a small channel wrapper so channel logs follow the same log-safety contract as the rest of the repo.
  - confidence: 0.84
