## Summary

Keep this PR open but not merge-ready: current main still has the raw channel logging surface, but this branch leaves `channel` unsanitized, duplicates a weaker sanitizer than the shared terminal helper, changes public SDK stringification behavior, and lacks after-fix real behavior proof.

## What This Changes

Adds a local channel log sanitizer, applies it to `reason`, `target`, and `error` in shared channel logging helpers, and adds a new Vitest suite for control-character log injection.

## Best Possible Solution

Land this hardening only after every interpolated channel logging field is sanitized through a string-coercing wrapper around the shared terminal helper, with regression coverage and redacted runtime/log proof.

## Reproduction Assessment

Yes, from source inspection: current main directly interpolates caller-provided channel logging fields, so newline or control-character inputs can forge output. I did not run runtime tests because this was a read-only review.

## Solution Assessment

No. The direction is right, but the maintainable fix should reuse the shared terminal sanitizer, coerce values before sanitizing, and cover every interpolated field including `channel`.

## Review Findings

Full review comments:

- **[P1] Reuse the shared log sanitizer:** `src/channels/logging.ts:4-6`
  - body: This adds a second `sanitizeForLog` that strips only C0 controls and DEL, while `src/terminal/ansi.ts` already handles ANSI/OSC, C1 controls, and DEL for log-safe interpolation. Reuse the shared helper through a small channel wrapper so this security behavior does not drift.
  - confidence: 0.88
- **[P1] Coerce values before sanitizing:** `src/channels/logging.ts:4-6`
  - body: `sanitizeForLog` calls `.replace` directly on `target` and `reason`. Public SDK callers written in JavaScript can pass numeric IDs or objects that previously stringified in template literals, so this can turn logging into a runtime TypeError.
  - confidence: 0.86
- **[P2] Sanitize channel before emitting log lines:** `src/channels/logging.ts:16`
  - body: The patch still prefixes each log message with raw `params.channel`. A crafted channel value containing a newline or control character can still forge another log line, so sanitize `channel` in all three helpers and cover it in tests.
  - confidence: 0.93
