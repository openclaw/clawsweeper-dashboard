## Summary

This focused security hardening should stay open, but it is not merge-ready: the PR duplicates a weaker sanitizer, leaves the channel prefix unsanitized, can regress JavaScript plugin callers that previously relied on template-literal stringification, and still lacks after-fix real behavior proof.

## What This Changes

The PR adds a local channel-log sanitizer to `src/channels/logging.ts` and a new Vitest file covering control-character log injection cases for channel logging helpers.

## Best Possible Solution

Use a small string-coercing wrapper around the shared terminal `sanitizeForLog`, apply it to every interpolated channel logging field including `channel`, and land it with focused regression coverage plus redacted runtime/log proof.

## Reproduction Assessment

Yes. From current-main source inspection, calling the shared channel logging helpers with newline or control-character values will interpolate those values directly into log output; I did not run tests in the read-only checkout.

## Solution Assessment

No. The hardening direction is right, but the maintainable fix should reuse the shared terminal sanitizer, coerce values before sanitizing, and cover every interpolated field including `channel`.

  - body: This adds a second `sanitizeForLog` that strips only C0 controls and DEL, while `src/terminal/ansi.ts` already handles ANSI/OSC, C1 controls, and DEL for log-safe interpolation. Reuse the shared helper through a small channel wrapper so this security behavior does not drift.

  - confidence: 0.93
