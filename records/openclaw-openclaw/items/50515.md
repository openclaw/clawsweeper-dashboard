## Summary

Keep open: this is a plausible security hardening PR, but the branch is not merge-ready because it leaves one interpolated field unsanitized, duplicates a weaker sanitizer, changes public SDK runtime behavior for JavaScript callers, and still lacks real behavior proof.

## What This Changes

The PR adds a local channel-log sanitizer for `reason`, `target`, and `error` values and adds Vitest coverage for control-character log injection cases.

## Best Possible Solution

Use a small string-coercing wrapper around the shared terminal `sanitizeForLog`, apply it to every interpolated channel logging field including `channel`, and land it with focused regression coverage plus redacted runtime/log proof.

## Reproduction Assessment

Yes. From current-main source inspection, calling the shared channel logging helpers with newline or control-character values will interpolate those values directly into log output; I did not run tests in the read-only checkout.

## Solution Assessment

No. The hardening direction is right, but the maintainable fix should reuse the shared terminal sanitizer, coerce values before sanitizing, and cover every interpolated field including `channel`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reuse the shared log sanitizer:** `src/channels/logging.ts:4-6`
  - body: This adds a second `sanitizeForLog` that strips only C0 controls and DEL, while `src/terminal/ansi.ts` already handles ANSI/OSC, C1 controls, and DEL for log-safe interpolation. Reuse the shared helper through a small channel wrapper so this security behavior does not drift.
  - confidence: 0.9
- **[P1] Coerce values before sanitizing:** `src/channels/logging.ts:4-6`
  - body: `sanitizeForLog` calls `.replace` directly on `target` and `reason`. Public SDK callers written in JavaScript can pass numeric IDs or objects that previously stringified in template literals, so this can turn logging into a runtime TypeError.
  - confidence: 0.86
- **[P2] Sanitize channel before emitting logs:** `src/channels/logging.ts:16`
  - body: The patch still prefixes each log message with raw `params.channel`. A crafted channel value containing a newline or control character can still forge another log line, so sanitize `channel` in all three helpers and cover it in tests.
  - confidence: 0.93
