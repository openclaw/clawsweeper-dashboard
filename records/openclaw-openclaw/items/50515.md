## Summary

Keep open. The PR addresses a real channel logging hardening gap, but it is not merge-ready: the patch still leaves `channel` interpolated unsanitized, changes public helper runtime behavior for loose JavaScript callers, and adds a weaker parallel sanitizer instead of using the existing shared log sanitizer. Existing review discussion already flagged the same channel and sanitizer concerns, so this should be revised rather than closed.

## What This Changes

The PR adds a local channel log sanitizer, applies it to `reason`, `target`, and `error` fields in shared channel logging helpers, and adds channel logging sanitization tests.

## Best Possible Solution

Revise the PR to reuse the shared `src/terminal/ansi.ts` sanitizer through a small wrapper that applies `String(...)`, sanitize every interpolated field in these helpers including `channel`, `reason`, `target`, `action`, and `error`, add tests for channel injection, DEL/C1/ANSI stripping, and loose non-string runtime values, and include the required user-facing changelog entry if the hardened behavior ships.

## Reproduction Assessment

Yes. By source and PR-diff inspection, `logInboundDrop({ channel: "x\nFAKE", reason: "drop", log })` would still emit a forged second line on the PR branch because `channel` remains interpolated directly. A loose JavaScript caller passing a numeric `target` or `reason` would also regress from implicit template-literal stringification to `.replace` on a non-string value.

## Solution Assessment

No. The intent is right, but the current patch is not the narrowest maintainable solution because it adds a weaker duplicate sanitizer and only sanitizes part of the log line. The safer path is to centralize on the existing log sanitizer and coerce values before sanitizing all interpolated fields.

  - body: The patch sanitizes `reason`, `target`, and `error`, but every helper still interpolates `params.channel` directly. Before this patch, template literals stringified `target` and `reason`; after it, non-string runtime values can throw instead of logging, so coerce inside the sanitizer wrapper.
  - confidence: 0.88

  - body: The new local sanitizer duplicates the existing terminal `sanitizeForLog` but strips only C0 controls plus DEL. Reuse the shared helper so channel logs also strip ANSI/OSC and C1 controls under the repository-wide log-safety contract.
