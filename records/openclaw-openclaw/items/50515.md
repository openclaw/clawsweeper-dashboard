## Summary

Keep open: the PR addresses a real security hardening gap, but the current branch still leaves a log-forging path through `channel`, duplicates a weaker sanitizer instead of the shared terminal sanitizer, can regress loosely typed JS plugin callers, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a local channel log sanitizer for `reason`, `target`, and `error` values in `src/channels/logging.ts` plus Vitest coverage for C0 control-character log injection cases.

## Best Possible Solution

Use a string-coercing wrapper around the shared terminal `sanitizeForLog` for every interpolated channel logging field, with regression coverage for `channel`, ANSI/OSC/C1/DEL controls, and non-string JS inputs.

## Reproduction Assessment

Yes. Source inspection on current main shows raw channel logging fields are interpolated into log messages, so a caller-provided newline or control character can forge log output.

## Solution Assessment

No. The hardening direction is right, but this branch should reuse the shared sanitizer and sanitize all interpolated fields while preserving plugin SDK runtime compatibility.

## Review Findings

- **[P2] Sanitize channel before emitting log lines:** `src/channels/logging.ts:16`
  - body: The patch still emits `params.channel` directly at the start of each log message. A channel value containing a newline or control character can still forge a separate log line, so sanitize `channel` in each helper and cover it in tests.
  - confidence: 0.93
- **[P2] Coerce log values before replacing characters:** `src/channels/logging.ts:4-6`
  - body: `sanitizeForLog()` calls `.replace` on its input. These helpers are public plugin SDK surface, and JavaScript callers that pass numeric or object `target`/`reason` values would now throw where template interpolation previously stringified them.
  - confidence: 0.88
- **[P2] Reuse the shared terminal sanitizer:** `src/channels/logging.ts:4-6`
  - body: This adds a second sanitizer that handles only C0 controls and DEL, while the existing terminal sanitizer also strips ANSI/OSC sequences and C1 controls. Delegate through the shared helper so log-safety behavior stays consistent.
  - confidence: 0.84
