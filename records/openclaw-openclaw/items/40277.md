## Summary

Keep open. Current main still has the user-visible gap: Gateway chat lifecycle errors can reach the Control UI, but the UI controller only stores them in transient error state and error-only assistant history still extracts as empty. The PR is a useful direction, but the current patch would not render persisted assistant error-only messages and would regress structured Gateway connect-error copy.

## What This Changes

The branch adds Control UI extraction, rendering, styling, controller updates, and tests so model failures appear as inline assistant error bubbles in web chat.

## Best Possible Solution

Land a focused Control UI fix that preserves `formatConnectError`, appends one structured assistant error message for send/live failures, routes persisted assistant error-only history through assistant extraction/normalization, renders grouped error bubbles for detailed and empty-detail cases, adds regression tests, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `state: "error"` chat event only updates `lastError`, and an assistant history entry with `stopReason: "error"`, `errorMessage`, and `content: []` still produces no visible markdown through current-main extraction and grouping.

## Solution Assessment

No. The PR's direction is appropriate, but this patch is not yet the best implementation because the error fallback is wired into a path assistant messages bypass, and the send-failure rewrite drops existing structured Gateway connect-error formatting.
