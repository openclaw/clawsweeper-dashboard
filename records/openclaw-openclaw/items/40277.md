## Summary

Keep open. Current main still lacks the inline web-chat model-error UI behavior, and this PR is still relevant, but the proposed patch has blocking correctness issues: the assistant error fallback is placed on a path assistant messages do not use, and send failures would lose existing structured connect-error formatting. Open #55923 overlaps only part of the live error append behavior and does not supersede this PR's persisted-history and grouped-rendering scope.

## What This Changes

The PR adds Control UI chat error bubbles by converting send/live run failures into assistant `stopReason: "error"` messages, extracting persisted assistant `errorMessage` text, styling grouped bubbles, and adding controller/extraction/view tests.

## Best Possible Solution

Land a focused UI fix that preserves current structured connect-error formatting, appends structured assistant error messages for live/send failures, makes assistant error-only history extract visible text through the actual assistant extraction path, and renders grouped error bubbles with tests for detailed and empty-detail cases.

## Reproduction Assessment

Yes. The original missing behavior is visible from current-main code paths for `payload.state: "error"` and persisted assistant `{ stopReason: "error", errorMessage, content: [] }` entries, and the PR defects are reproducible by following `extractText`'s assistant branch plus the existing structured send-failure test.

## Solution Assessment

No. The PR's product direction is narrow and appropriate, but this patch is not yet the best implementation because it places the history fallback where assistant extraction will not call it and replaces the existing connect-error formatter; preserving `formatConnectError` and moving the fallback into the assistant extraction path is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route assistant error fallback through extractText:** `ui/src/ui/chat/message-extract.ts:123`
  - body: The new fallback is added at the end of `extractRawText`, but `extractText` sends assistant messages through `extractSharedAssistantVisibleText` instead. Error-only assistant history with `content: []` still produces `null`, so the new tests and grouped error bubbles will not work until the fallback is wired into the assistant branch.
  - confidence: 0.94
- **[P2] Keep structured connect-error formatting:** `ui/src/ui/controllers/chat.ts:263`
  - body: Replacing `formatConnectError(err)` with plain normalization drops detail-code mapping for `GatewayRequestError`; the existing chat-send test for `CONTROL_UI_ORIGIN_NOT_ALLOWED` would surface `Fetch failed` instead of `origin not allowed`. Reuse `formatConnectError` and put that formatted string into the structured assistant error message.
  - confidence: 0.9
