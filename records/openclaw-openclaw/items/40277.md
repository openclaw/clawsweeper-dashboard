## Summary

Keep open. Current main still hides live chat error events in Control UI state and drops assistant error-only history entries, so the PR targets a real remaining UI bug. The branch is not ready to merge because the assistant error fallback is wired into an extraction path assistant messages bypass, and the send-failure change regresses existing structured Gateway connect-error formatting.

## What This Changes

The PR updates Control UI chat extraction, rendering, styling, controller behavior, and tests so assistant model errors from history and live chat events appear as inline highlighted assistant error bubbles.

## Best Possible Solution

Land a focused Control UI fix that preserves `formatConnectError`, appends one structured assistant error message for send/live failures, routes persisted assistant error-only history through assistant extraction before returning null, renders grouped error bubbles even when details are empty, adds regression tests, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `state: "error"` chat event only updates `lastError` on current main, and an assistant history entry with `stopReason: "error"`, `errorMessage`, and `content: []` produces no visible assistant text through the current extraction path.

## Solution Assessment

No. The PR's direction is appropriate, but this patch is not the best implementation because the fallback is attached to `extractRawText`, which assistant messages bypass, and the send-failure rewrite drops structured Gateway connect-error formatting. The safer path is to wire the fallback into the assistant branch and keep `formatConnectError` for request failures.
