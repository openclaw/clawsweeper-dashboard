## Summary

Keep open: current main still has a plausible memory-core reliability bug where synchronous SQLite reindex/temp-db work can block the gateway event loop, leave `main.sqlite.tmp-*` sidecars after process termination, and indirectly break Slack Socket Mode recovery.

## What This Changes

The issue reports Slack Socket Mode disconnects caused by long synchronous SQLite maintenance and proposes off-main-thread maintenance, orphan temp cleanup, cache bounds, and reconnect hardening.

## Best Possible Solution

Move heavy builtin memory SQLite reindex/maintenance off the gateway event loop, safely sweep stale memory temp sidecars on open/startup, enforce a real embedding-cache bound, and keep Slack recovery self-healing under transient memory stalls.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the temp-db and synchronous `DatabaseSync` paths needed to explain the orphan files and event-loop starvation, but a real bloated DB plus Slack Socket Mode repro is still missing.

## Solution Assessment

Unclear as a single patch: stale-temp cleanup and cache bounding are narrow, but the best durable fix likely needs an owner-approved memory execution boundary so SQLite work cannot starve channel heartbeats.
