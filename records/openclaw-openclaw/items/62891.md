## Summary

Keep open. Current main has partial startup deferral for scheduled services and missed isolated cron jobs, but it still lacks a bounded channel transport readiness gate for cron startup/reload, so the Discord delivery race described in the comments remains source-reproducible.

## What This Changes

The issue requests reliable cron behavior across startup and SIGUSR1/config.patch reloads so LLM-backed Discord cron deliveries do not run before session/channel readiness.

## Best Possible Solution

Add a bounded generic cron delivery-readiness gate for initial startup and hot reload that waits for configured delivery-capable channel transports to connect or times out with clear diagnostics.

## Reproduction Assessment

Yes, at source level. An enabled Discord channel plus a due isolated cron job can run after channel startup handoff but before Discord sets connected=true, and config.patch hot reload starts cron before restarting affected channels.

## Solution Assessment

No, current main is not the complete fix. The narrow maintainable repair is to gate cron delivery startup/reload on bounded channel transport readiness instead of relying on sidecar completion or a fixed missed-job delay.

## Repair Work Prompt

Fix the cron/channel startup race tracked by https://github.com/openclaw/openclaw/issues/62891. Add a bounded generic readiness gate so cron deliveries that depend on configured delivery-capable channels do not start on initial gateway startup or config.patch/SIGUSR1 hot reload until the relevant channel transports are connected, or until a timeout logs clear diagnostics and proceeds/fails in the existing style. Keep the fix in core gateway/cron/channel readiness seams; do not add a new user-facing config option unless an existing contract forces it, and do not add Discord-specific policy to generic core beyond using existing channel status. Preserve webhook-only and non-channel cron jobs so they are not unnecessarily blocked. Add/update regression tests for initial startup ordering and hot reload ordering, plus a cron missed-job or delivery-target test if the gate lives in cron. Add a single-line CHANGELOG.md fix entry because this is user-facing reliability behavior.
