## Summary

Keep open. Current main includes partial cron startup deferral and lazy-loading work, but cron can still start after channel handoff rather than after channel transport readiness, and hot reload still restarts cron before affected channels reconnect.

## What This Changes

Ensure startup and SIGUSR1/config.patch reload do not allow cron jobs that use LLM or channel delivery to run before Discord/channel transports are ready.

## Best Possible Solution

Add a bounded generic cron delivery-readiness gate for initial startup and hot reload that waits for configured delivery-capable channel transports or times out with clear diagnostics.

## Reproduction Assessment

Yes, at source level. An enabled Discord channel plus a due isolated cron job with announce/channel delivery can run after channel startup handoff but before Discord reports connected, and config.patch reload starts cron before restarting affected channels.

## Solution Assessment

No, current main is not the complete fix. Deferring cron until sidecars return reduces the startup window, but the narrow maintainable repair is a bounded delivery-readiness gate at the gateway/cron boundary with matching hot-reload ordering.

## Repair Work Prompt

Fix the cron/channel readiness race tracked by https://github.com/openclaw/openclaw/issues/62891. Add a bounded, generic readiness gate so initial cron startup and config.patch/SIGUSR1 hot reload do not execute due cron jobs with channel delivery until configured delivery-capable channel transports are ready or a clear timeout/warning occurs. Keep core extension-agnostic: do not hard-code Discord IDs or provider-specific policy into core; add a typed generic channel readiness seam if needed. Preserve existing cron missed-job stagger behavior and do not add a new user-facing config option unless the existing config surface already has an appropriate timeout. Cover startup and hot reload ordering with regression tests, including a channel whose startChannel returns before connected status flips true. Add a changelog entry because this is user-facing reliability behavior.
