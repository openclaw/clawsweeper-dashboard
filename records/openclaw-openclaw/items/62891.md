## Summary

Keep open. Current main still starts or restarts cron without waiting for delivery-capable channel transport readiness, so the Discord/SIGUSR1 cron race remains source-reproducible despite the existing sidecar and missed-agent-job startup deferrals.

## What This Changes

The issue requests that cron startup and config.patch/SIGUSR1 hot reload avoid firing Discord/LLM cron work before the relevant channel transport is ready.

## Best Possible Solution

Add a bounded generic cron delivery-readiness gate for initial startup and hot reload that waits for configured delivery-capable channel transports to connect, or times out with clear diagnostics.

## Reproduction Assessment

Yes, at source level. An enabled Discord channel plus a due isolated cron job can run after post-ready/sidecar activation but before Discord sets connected=true, and config.patch hot reload starts cron before restarting affected channels.

## Solution Assessment

Yes. A bounded generic delivery-readiness gate is the narrow maintainable fix; relying on session-store lock timing, a fixed sleep, or a config-only delay would be brittle.
