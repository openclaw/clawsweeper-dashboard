## Summary

Keep open. Current main still defers cron only until sidecar/post-attach handoff, and hot reload still starts cron before channel restarts, so the Discord transport readiness race remains source-reproducible.

## What This Changes

Ensure cron/LLM scheduled work does not run on startup or SIGUSR1/config.patch reload before configured channel transports, especially Discord, are ready.

## Best Possible Solution

Add a bounded generic cron delivery-readiness gate for initial startup and hot reload that waits for configured delivery-capable channel transports to connect or times out with clear diagnostics.

## Reproduction Assessment

Yes, at source level. An enabled Discord channel plus a due isolated cron job can run after channel startup handoff but before Discord sets connected=true, and config.patch hot reload starts cron before restarting affected channels.

## Solution Assessment

No, current main is not the complete fix. The narrow maintainable repair is to gate cron delivery startup/reload on bounded channel transport readiness instead of relying on sidecar completion, session-store lock timing, or a fixed sleep.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/62891. The observable bug is that cron scheduled jobs can run after gateway sidecar handoff but before delivery-capable channel transports, especially Discord, are connected; config.patch hot reload also restarts cron before restarting affected channels. Implement a bounded, plugin-agnostic cron delivery-readiness gate for initial startup and cron hot reload using channel manager runtime snapshots/connected status. Wait for enabled and configured delivery-capable channels to connect, or time out with clear diagnostics; skip disabled, unconfigured, intentionally skipped, or non-delivery channels. Do not add a new config knob unless a maintainer explicitly asks, do not rely on fixed sleeps or session-store locks, and keep core independent of Discord-specific policy. Add regression tests for initial startup deferral and hot reload ordering/readiness wait. Before handoff, run the focused gateway/cron tests and oxfmt on touched files; live Discord restart proof is recommended before landing.
