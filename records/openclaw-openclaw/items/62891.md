## Summary

Keep open. Current main still has a real cron/channel readiness gap: startup has a readiness checker, but cron activation and hot reload do not consume it, and the existing mitigation only delays overdue startup agent-turn catch-up rather than normal due ticks or reload-time cron activation before Discord reconnects.

## What This Changes

The issue requests that gateway startup and SIGUSR1/config.patch cron scheduling wait for channel readiness so Discord-delivered cron/LLM work does not exhaust model fallbacks before the Discord WebSocket is connected.

## Best Possible Solution

Add a generic, bounded readiness gate at the gateway/cron boundary for initial activation and hot reload. It should wait only for cron work that can deliver to configured channel transports, leave non-channel jobs alone, time out with clear logs, and avoid hard-coding Discord-specific behavior in core.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an enabled Discord channel plus a due isolated cron job with announce delivery, then startup or SIGUSR1/config.patch reload before Discord READY; current main can start or restart cron before channel transport readiness is consulted.

## Solution Assessment

Yes for the remaining issue direction, if implemented as a bounded delivery-aware gate rather than a blanket sleep or Discord-only workaround. The current main mitigation is incomplete because it only defers missed startup agent-turn catch-up.
