## Label Justifications

- `P1`: The issue describes a real restart/reload window where scheduled Discord/LLM work can fail before the configured channel transport is delivery-ready.
- `impact:message-loss`: Cron jobs that send Discord messages can run before the Discord gateway is connected, causing delivery failure in the reported window.
- `impact:auth-provider`: The observable failure includes all configured LLM fallbacks timing out during the premature cron run.

## Summary

Keep open: current main still has a source-visible race where cron can start or restart without waiting for delivery-capable channel transport readiness, so the Discord cron/SIGUSR1 failure window remains actionable.

## What This Changes

The issue reports cron or LLM work firing during startup or SIGUSR1/config reload before Discord transport readiness, causing fallback timeouts and failed Discord delivery.

## Best Possible Solution

Add a bounded delivery-readiness gate around initial post-ready cron activation and hot-reload cron restart, using channel runtime connected status with clear timeout diagnostics and focused regression tests.

## Reproduction Assessment

Yes, at source level. A due cron job can be started from post-ready maintenance or config reload before Discord has published connected status, although I did not run a live Discord timing reproduction.

## Solution Assessment

Yes. A bounded generic channel-readiness wait around cron activation/reload is the narrow maintainable fix; fixed sleeps or a new config-only delay would be brittle.
