## Summary

Keep open. Current main has partial startup and lazy-cron mitigations, but source inspection still shows cron can start from channel-start handoff rather than transport readiness, and hot reload still restarts cron before affected channels reconnect.

## What This Changes

The issue asks OpenClaw to prevent startup and SIGUSR1/config.patch cron runs from firing before Discord/channel transport readiness, which can exhaust model fallbacks and fail channel delivery.

## Best Possible Solution

Add a bounded generic cron delivery-readiness gate for startup and hot reload that waits for configured delivery-capable channel transports to connect or times out with clear diagnostics.

## Reproduction Assessment

Yes, at source level. An enabled Discord channel plus a due isolated cron job with announce/channel delivery can run after channel startup handoff but before Discord reports connected, and config.patch hot reload starts cron before restarting affected channels.

## Solution Assessment

No, current main is not yet the complete fix. Deferring cron until sidecar startup reduces one window, but the narrow maintainable repair is a bounded delivery-readiness gate at the gateway/cron boundary with matching hot-reload ordering.

## Repair Work Prompt

Fix the cron/channel readiness race tracked by https://github.com/openclaw/openclaw/issues/62891. Current main starts cron after channel startup handoff, but not after delivery-capable channel transports are connected, and applyHotReload starts cron before restarting affected channels. Add a bounded generic readiness gate at the gateway/cron boundary for initial startup and hot reload so due isolated cron jobs with channel/announce delivery do not run until configured delivery-capable channel accounts are connected or a timeout emits clear diagnostics. Prefer existing generic channel account snapshot/status fields such as connected/running; do not add a new user-facing config option unless the existing contracts make an internal bounded wait impossible. Keep owner-specific Discord details in the Discord plugin and use core only for generic readiness orchestration. Add regression tests for startup ordering and config.patch/hot-reload ordering, update CHANGELOG.md with a single-line user-facing fix entry, and do not change unrelated cron scheduling semantics or provider-specific delivery target parsing.
