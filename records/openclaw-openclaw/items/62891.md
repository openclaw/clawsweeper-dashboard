## Summary

Keep open. Current main still lacks a delivery-readiness gate around cron startup and config hot reload, so the reported Discord cron race remains source-reproducible rather than implemented.

## What This Changes

The issue asks OpenClaw to prevent scheduled cron/LLM work from firing during startup or SIGUSR1 reload before configured Discord/channel transports are ready.

## Best Possible Solution

Add a bounded generic cron delivery-readiness gate for initial startup and hot reload that uses existing channel account status, skips non-delivery-ready exceptions, and continues with diagnostics on timeout.

## Reproduction Assessment

Yes, at source level. A configured Discord channel can be marked running and handed off before connected=true, while due cron starts through post-ready maintenance or hot reload without requiring that connected state.

## Solution Assessment

No, current main is not the complete fix. The maintainable path is a bounded readiness gate around cron activation/reload rather than relying on session-store lock timing, sidecar completion, or fixed sleeps.
