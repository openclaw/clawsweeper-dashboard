## Summary

Keep open. Current main has partial startup mitigation, but cron can still begin after channel startup handoff rather than after delivery transports report connected, and hot reload can restart cron before restarting affected channels.

## What This Changes

The issue requests startup and SIGUSR1/config.patch restart ordering that prevents cron-triggered LLM/Discord work from firing before session reconciliation and Discord channel readiness.

## Best Possible Solution

Add a bounded, generic cron delivery-readiness gate for initial startup and config.patch/SIGUSR1 reload that waits for configured delivery-capable channel transports or times out with clear diagnostics.

## Reproduction Assessment

Yes, at source level. An enabled Discord channel plus a due isolated cron job with announce/channel delivery can start during startup or config.patch reload before Discord reports connected, because current main gates on channel handoff rather than transport readiness.

## Solution Assessment

No, current main is only a partial mitigation. The narrow maintainable fix is to gate cron delivery on a bounded generic channel transport readiness signal and preserve that ordering during hot reload.

## Repair Work Prompt

Fix the cron/channel readiness race described by https://github.com/openclaw/openclaw/issues/62891. On initial startup and config.patch/SIGUSR1 hot reload, cron-triggered delivery work must not fire before configured delivery-capable channel transports, especially Discord, have either reported delivery readiness or hit a bounded timeout with diagnostics. Keep the fix generic in gateway/cron/channel readiness contracts; do not hard-code Discord IDs, add a new config option, or change provider auth/catalog behavior. Prefer an existing readiness helper such as transport-ready semantics where it fits. Add or update focused regression coverage for startup ordering and hot-reload ordering, including the case where channel startup handoff completes before the provider transport is ready. Update CHANGELOG.md because this is user-facing reliability behavior.
