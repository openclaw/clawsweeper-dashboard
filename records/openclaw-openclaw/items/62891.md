## Summary

Keep open. Current main still has the cron/channel-readiness race described in the issue: startup cron is only deferred until sidecars return, and config.patch hot reload restarts cron before affected channels are restarted, while Discord transport readiness is recorded later through the channel status path.

## What This Changes

The issue reports that startup or SIGUSR1/config.patch reload can let cron/LLM work run before Discord channel readiness, causing fallback-chain timeouts.

## Best Possible Solution

Implement a bounded generic cron delivery-readiness gate for initial startup and hot reload that waits for configured delivery-capable channel transports to connect or times out with clear diagnostics.

## Reproduction Assessment

Yes, at source level. An enabled Discord channel plus a due isolated cron job can run after sidecar/post-ready handoff but before Discord sets connected=true, and config.patch hot reload starts cron before restarting affected channels.

## Solution Assessment

Yes. A bounded generic delivery-readiness gate is the narrow maintainable fix; relying on session-store lock timing, a fixed sleep, or a config-only delay would be brittle.
