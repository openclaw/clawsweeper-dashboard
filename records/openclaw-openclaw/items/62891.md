## Summary

Keep open. Current main has useful startup mitigations, but cron can still run before a delivery-capable channel transport is actually ready, especially on hot reload where cron restarts before channel restarts.

## What This Changes

The issue requests that cron startup and reload wait for channel transport readiness so scheduled LLM/channel jobs do not fire before Discord is connected.

## Best Possible Solution

Add a bounded, generic delivery-readiness gate at the gateway/cron boundary for initial activation and hot reload, applying it only to jobs that can deliver through configured channel transports and timing out with clear diagnostics.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an enabled Discord channel plus a due isolated cron job with channel delivery during startup or config.patch reload before Discord READY; current main starts or restarts cron without waiting for that transport-ready state.

## Solution Assessment

No. The current mitigation defers scheduled services until sidecars return and staggers missed agent jobs, but it does not prove channel delivery readiness and does not cover hot cron reload ordering.
