## Summary

Keep open. Current main still starts or restarts cron without a bounded wait for delivery-capable channel transport readiness, so the Discord/config.patch cron race remains source-reproducible; the existing startup deferral only covers missed agent-turn catch-up jobs.

## What This Changes

The issue asks for startup and SIGUSR1/config.patch cron scheduling to wait until Discord/channel transports are ready instead of firing cron jobs during the reconnect window.

## Best Possible Solution

Add a bounded generic delivery-readiness gate for initial startup and hot reload that waits for configured delivery-capable channel transports to connect, skips non-applicable channels, and logs timeout diagnostics.

## Reproduction Assessment

Yes, at source level. An enabled Discord channel plus a due cron job can run after post-ready cron activation or config.patch cron restart while the channel snapshot is running but not yet connected; I did not run a live Discord timing reproduction in this read-only pass.

## Solution Assessment

Yes. A bounded generic channel-readiness wait around cron activation and reload is the narrow maintainable fix; fixed sleeps or a new config-only delay would be brittle and broader fallback-provider changes belong elsewhere.

Fix the cron/channel readiness race reported in https://github.com/openclaw/openclaw/issues/62891. Current main can start cron in post-ready maintenance and restart cron during config.patch/SIGUSR1 before configured delivery-capable channel transports, especially Discord, have reported connected=true. Add a bounded generic readiness gate around initial cron activation and hot-reload cron restart that waits for configured delivery-capable channel accounts to become connected, skips disabled/unconfigured/skipped/unmanaged channels, times out with clear diagnostics, and does not add a new config option unless the existing architecture already has an appropriate policy knob.
