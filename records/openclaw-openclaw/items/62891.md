## Summary

Keep open. Current main has partial startup mitigation, including main-only cron deferral work, but cron still starts after channel startup handoff rather than after transport readiness, and hot reload still restarts cron before restarting affected channels.

## What This Changes

The issue asks OpenClaw to prevent startup and SIGUSR1/config.patch cron/LLM work from firing before Discord/channel delivery is ready.

## Best Possible Solution

Add a bounded, generic cron delivery-readiness gate for initial startup and hot reload that waits for configured delivery-capable channel transports or times out with clear diagnostics.

## Reproduction Assessment

Yes, at source level. An enabled Discord channel plus a due isolated cron job with announce/channel delivery can start during startup or config.patch reload before Discord reports connected, because current main gates on channel handoff rather than transport readiness.

## Solution Assessment

No, current main is not the complete solution. Deferring cron until sidecars return reduces the startup window, but the narrow maintainable fix is a bounded delivery-readiness gate at the gateway/cron boundary and matching hot-reload ordering.

## Repair Work Prompt

Fix #62891 as a narrow OpenClaw reliability bug. Current main defers cron until sidecars return, but channel startup only hands off long-running provider tasks, and hot reload starts the rebuilt cron service before affected channels restart. Add a bounded generic delivery-readiness gate at the gateway/cron boundary for initial cron activation and config.patch/SIGUSR1 cron restart, using existing channel runtime/status data rather than Discord-specific checks. Apply it only where cron jobs may deliver through configured channel transports; disabled/unconfigured channels must not block startup. Time out with clear gateway/cron diagnostics rather than hanging indefinitely. Do not add a new config option unless absolutely necessary, do not turn this into a Discord-only workaround, and do not try to solve the separate inbound-message replay problem. Add focused regression coverage for startup ordering and hot reload ordering, update user-facing changelog if behavior changes, and keep docs changes minimal unless a new diagnostic or behavior contract needs documenting.
