## Summary

Keep open. Current main has partial startup and lazy-cron mitigations, but cron is still started without a delivery transport-readiness gate on startup and is restarted before affected channels on hot reload, so the reporter's SIGUSR1/Discord race is still source-reproducible.

## What This Changes

Cron startup and reload should not let scheduled LLM or Discord-delivery jobs fire before session recovery and delivery-capable channel transports are ready.

## Best Possible Solution

Add a bounded generic cron delivery-readiness gate for initial startup and config.patch/SIGUSR1 reload that waits for configured delivery-capable channel transports, then times out with clear diagnostics.

## Reproduction Assessment

Yes, at source level. An enabled Discord channel plus a due isolated cron job can run after channel startup handoff but before Discord reports connected, and config.patch reload starts cron before restarting affected channels.

## Solution Assessment

No, current main is not the complete fix. The narrow maintainable repair is a generic bounded readiness gate at the gateway/cron boundary plus matching hot-reload ordering, not a Discord-specific sleep or new config knob by default.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/62891. Add a bounded generic cron delivery-readiness gate so scheduled cron jobs that can deliver to chat channels do not run before configured delivery-capable channel transports are ready on initial startup or config.patch/SIGUSR1 hot reload. Keep the fix generic in core; do not special-case Discord IDs or add a new config knob unless maintainers explicitly request it. Reuse existing channel runtime/status/health contracts where possible. On startup, cron should remain unarmed or due execution should pause until sidecar channel transport readiness is satisfied or a bounded timeout logs clear diagnostics. On hot reload, avoid restarting/arming cron before affected channels have been restarted and passed the same readiness boundary, or make the cron service defer due delivery until that boundary. Add focused regression tests for initial startup handoff vs connected readiness, and hot reload with restartCron plus restartChannels. Update CHANGELOG.md because this is user-facing reliability. Do not move owner-specific Discord lifecycle logic into core beyond a generic readiness seam.
