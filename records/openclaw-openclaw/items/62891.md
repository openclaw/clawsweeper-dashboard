## Label Justifications

- `P1`: The issue describes a broken real cron/channel startup workflow where scheduled LLM work and Discord delivery fail during restart windows.
- `impact:message-loss`: Cron-triggered Discord delivery can fire before the Discord gateway is available, producing failed or lost outbound messages.
- `impact:session-state`: The reported timing depends on session-store reconciliation and cron/session state across restart paths.

## Summary

Keep open. Current main still has a source-visible cron/channel readiness race: startup cron activation and hot-reload cron restart do not wait for delivery-capable channel transport readiness, while Discord only reports connected after its gateway reaches READY.

## What This Changes

The issue asks OpenClaw to prevent cron-triggered LLM/message work from firing during startup or SIGUSR1/config.patch reload before Discord/channel transport is ready.

## Best Possible Solution

Add a bounded generic cron delivery-readiness gate for startup and hot reload that waits for configured delivery-capable channel transports to become ready, skips non-applicable channels, and logs timeout diagnostics.

## Reproduction Assessment

Yes, at source level. An enabled Discord channel plus a due cron job can run after cron activation or cron hot-reload restart while the channel task is handed off but not yet connected; I did not run a live Discord timing reproduction.

## Solution Assessment

Yes. A bounded readiness wait around cron activation and reload is the narrow maintainable fix; fixed sleeps or a config-only delay would be brittle, and broader fallback/provider timeout changes belong to separate reports.
