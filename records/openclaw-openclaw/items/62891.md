## Summary

Keep open. Current main and the latest shipped tag still start or restart cron without waiting for delivery-capable channel transport readiness, so the Discord/SIGUSR1 cron race remains source-reproducible.

## What This Changes

The issue reports cron/LLM work firing during startup or SIGUSR1/config.patch reload before Discord transport readiness, causing model fallback timeouts and failed Discord delivery.

## Best Possible Solution

Add a bounded generic cron delivery-readiness gate for initial startup and hot reload that waits for configured delivery-capable channel transports to connect or times out with clear diagnostics.

## Reproduction Assessment

Yes, at source level. An enabled Discord channel plus a due isolated cron job can run after sidecar/post-ready handoff but before Discord sets connected=true, and config.patch hot reload starts cron before restarting affected channels.

## Solution Assessment

Yes. A bounded generic delivery-readiness gate is the narrow maintainable fix; relying on session-store lock timing, a fixed sleep, or a config-only delay would be brittle.
