## Label Justifications

- `P1`: The PR addresses a persistent authorization-list clobber path that can silently remove previously paired senders and lock users out after a permissions failure.
- `merge-risk: 🚨 compatibility`: Existing broken-permission setups will now fail closed with EACCES during allowFrom mutations instead of returning a successful but unsafe rewrite.

## Summary

Keep open: current main still has the allowFrom mutator read/fallback gap, and this PR is a focused fix with sufficient real behavior proof and no blocking findings.

## What This Changes

The PR changes the pairing allowFrom mutator to use the fail-closed allowFrom file reader before full-file rewrites and adds a regression test for unreadable base files.

## Best Possible Solution

Land the narrow mutator-reader fix if maintainers accept the fail-closed EACCES behavior, while leaving the broader sudo/config ownership policy tracked in https://github.com/openclaw/openclaw/issues/78493.

## Reproduction Assessment

Yes at source/proof level: current main can collapse an unreadable existing allowFrom store to the empty fallback before rewrite, and the PR body includes before/after filesystem output. I did not run the OS-level reproduction in this read-only pass.

## Solution Assessment

Yes: routing the rewrite pre-read through the same fail-closed allowFrom reader as enforcement is the narrow owner-boundary fix for the data-loss path. A cache-warm EACCES regression test would further improve confidence but is not a blocking correctness issue from this review.
