## Summary

Keep open. Current main and the latest release still show regular auto-compaction ending with `willRetry=false` and only resolving the compaction wait, while the issue has multiple concrete user reports of lost or undelivered in-flight replies across TUI, WhatsApp, and Telegram; no merged current-main proof or canonical open item fully owns the whole resumption/delivery problem.

## What This Changes

The issue asks OpenClaw to preserve and resume an in-flight agent turn after automatic compaction so the user receives the intended final reply instead of a silent abandonment, `NO_REPLY`, or truncated delivery.

## Best Possible Solution

Define one maintainer-owned compaction recovery contract that preserves the active turn, reply obligation, and original delivery context across compaction, then prove it with focused tests plus a live long-context channel/TUI reproduction.

## Reproduction Assessment

No high-confidence current-main live reproduction was established. The source path and multiple production reports make the failure class credible, but the deterministic long-context/channel replay still needs to be run.

## Solution Assessment

Unclear. The suggested resume options point in the right direction, but the best fix is likely a first-class turn/reply-obligation and delivery-context contract rather than only calling `continue()` or filtering one symptom.
