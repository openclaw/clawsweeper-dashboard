## Summary

Keep open. Current main has adjacent compaction-retry, liveness, and mid-turn precheck safeguards, but it still does not prove that a post-compaction turn has a durable reply obligation or that retry-generated output is flushed through the active gateway/TUI delivery path; related #47335 and #66126 remain open for overlapping delivery and turn-state gaps.

## What This Changes

The issue requests that automatic compaction during an active turn preserve or resume the interrupted task and deliver the final assistant response to the user.

## Best Possible Solution

Land a regression-backed compaction-resume and delivery fix that preserves active turn identity across compaction, tracks whether a user-visible reply is still owed, and proves retry-produced tool events plus final assistant output traverse the normal gateway/WebSocket/TUI delivery path.

## Reproduction Assessment

Unclear for current main. The issue and PlaidJohn's comment provide credible logs plus a deterministic high-context vLLM trigger, but this read-only review did not run that scenario and the in-repo tests cover adjacent retry and flush mechanics rather than the active-client disappearance.

## Solution Assessment

No, the current implementation is not enough to close this. A universal resume or delayed-compaction change alone may miss the reported variant where work continues but delivery disappears; the safer solution is a turn/reply-obligation and delivery-barrier fix with an end-to-end regression.
