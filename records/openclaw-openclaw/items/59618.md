## Summary

Keep open. Fresh OpenClaw 2026.5.7 transcript evidence and current main source both show this is still an active compaction resume/delivery problem, with only partial mitigations for continuation, retry loops, and mid-turn prechecks; related https://github.com/openclaw/openclaw/issues/47335 and https://github.com/openclaw/openclaw/issues/66126 remain open for overlapping delivery and retry-state gaps.

## What This Changes

Ensure automatic compaction during an active tool-based turn resumes the same task and delivers the final user-visible response instead of silently ending or emitting runtime-only continuation artifacts.

## Best Possible Solution

Land a regression-backed compaction turn-state and delivery fix that preserves active turn identity across Pi compaction, tracks whether a user-visible reply is still owed, handles runtime-only continuation prompts and NO_REPLY safely, and proves retry-produced tool events plus final output reach gateway/WebSocket/channel clients.

## Reproduction Assessment

No high-confidence current-main live reproduction was run. The report has credible v2026.4.9 and v2026.5.7 logs plus a deterministic high-context trigger, and source inspection shows the relevant retry/delivery contracts remain only partially guarded.

## Solution Assessment

No, the current implementation is not enough to close this. A universal resume alone would miss the reported delivery-loss variant, while current main's continuation retry and loop guard do not prove a durable reply obligation for toolful compaction retries.
