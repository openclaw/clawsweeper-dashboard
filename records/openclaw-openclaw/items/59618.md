## Summary

Keep open. Current main still has source-level gaps in compaction retry delivery and turn-state handling, and the fresh OpenClaw 2026.5.7 WhatsApp reproduction shows the issue family survived the prior stale-replay fix; the remaining work overlaps https://github.com/openclaw/openclaw/issues/47335 and https://github.com/openclaw/openclaw/issues/66126 rather than being safely closeable.

## What This Changes

The issue reports that automatic compaction during an active turn can lose or hide the final user-visible response instead of resuming the original task.

## Best Possible Solution

Define and land a shared compaction-resume contract that preserves active turn identity and guarantees retry-produced tool events plus final assistant output drain through the normal channel delivery path before the turn can appear complete.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction in this review. The issue has credible release-era logs and current main remains source-reproducible around compaction wait resolution, async flush ordering, and missing stable turn-state tracking.

## Solution Assessment

No, the current implementation is not enough. A retry-specific delivery barrier may be the narrow fix for https://github.com/openclaw/openclaw/issues/47335, but this issue also needs stable active-turn/resume semantics so compaction cannot turn the original task into a runtime-event `NO_REPLY` path.
