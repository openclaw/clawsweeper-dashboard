## Summary

Keep open. Current `main` has adjacent compaction-retry, liveness, and opt-in mid-turn precheck safeguards, but it still does not prove that post-compaction work has a durable reply obligation or that retry-produced output reaches the active client; related #47335 and #66126 remain open for overlapping delivery and turn-state gaps.

## What This Changes

Ensure auto-compaction during an active turn resumes the task and delivers the final output instead of leaving the user with no visible response.

## Best Possible Solution

Land a regression-backed compaction-resume and delivery fix that preserves active turn identity across compaction, tracks whether a user-visible reply is still owed, and proves retry-produced tool events plus final assistant output traverse the normal gateway/WebSocket/TUI path.

## Reproduction Assessment

Unclear for current `main`. The issue and PlaidJohn's comment provide credible logs and a deterministic high-context trigger, but this read-only review did not run that scenario and current tests cover adjacent retry/flush ordering rather than active-client disappearance.

## Solution Assessment

No, the current implementation is not enough to solve this. A universal resume or delayed-compaction change alone may miss the reported variant where work continues but delivery disappears; the safer path is a turn/reply-obligation plus delivery-barrier fix.
