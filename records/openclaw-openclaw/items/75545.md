## Summary

Keep this PR open. Current main still sends an installed `gateway start` through the service-manager restart path, and the related issue's login-shell repro matches that behavior. The proposed branch is a narrow lifecycle fix with focused CLI and unit coverage, and I did not find a blocking correctness or security issue in the diff.

## What This Changes

The PR adds an `already-running` gateway service start outcome, skips service-manager restart when the managed runtime is already running, emits that result from the CLI, updates lifecycle tests, and adds a changelog entry for #60087.

## Best Possible Solution

Land a narrow gateway lifecycle change that makes `openclaw gateway start` behave as ensure-running for already-active managed services, while preserving `openclaw gateway restart` as the explicit restart operation and keeping the regression covered at both service and CLI layers.

## Reproduction Assessment

Yes. The #60087 repro path is high-confidence from source: a login shell can invoke `openclaw gateway start`, and current main then calls the platform restart action for an installed service even when runtime state is already running.

## Solution Assessment

Yes. The proposed service-layer idempotency guard is the narrowest maintainable fix because it addresses the command contract directly instead of changing exec shell startup behavior or adding profile-specific workarounds.
