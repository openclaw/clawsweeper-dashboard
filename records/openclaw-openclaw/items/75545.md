## Summary

Keep open. Current main still routes `gateway start` through the restart path when the service is installed/loaded, while this PR adds the narrow service-layer no-op behavior, CLI result branch, regression tests, and changelog entry needed for #60087. I found no discrete correctness or security finding in the diff, but the PR should remain open for maintainer review and CI resolution before merge.

## What This Changes

The PR adds an `already-running` gateway service start outcome, skips service restart when runtime state is already running, surfaces that result in CLI output, updates lifecycle tests, and adds a changelog fix entry for #60087.

## Best Possible Solution

Land the service-layer idempotency behavior with the CLI result branch, focused regression tests, and changelog entry after maintainer review and green changed-gate validation, while keeping `openclaw gateway restart` as the explicit forced-cycle command and leaving #62086 as a separate SIGTERM path.

## Reproduction Assessment

Yes. The source-backed reproduction is high-confidence: exec PATH discovery can run `bash -l -c env -0`, a profile can invoke `openclaw gateway start`, and current main then reaches `service.restart` for an already-running managed service.

## Solution Assessment

Yes. The PR solves the reported behavior at the shared gateway service lifecycle boundary instead of adding an exec-tool-specific workaround, and it preserves explicit forced restarts through the existing `gateway restart` command.
