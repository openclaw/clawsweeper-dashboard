## Summary

Keep open: the PR targets a valid gateway restart bug, but the current patch returns before the existing repair-required checks, so stale or broken loaded service definitions no longer get repaired during `gateway start`.

## What This Changes

Adds an `already-running` gateway service start result with CLI output, tests, and a changelog entry, plus a few formatting-only test changes.

## Best Possible Solution

Keep the idempotent start behavior for healthy running services, but preserve the existing repair-required path before returning `already-running`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the exec PATH probe uses a login shell, current `gateway start` reaches `service.restart()` even when the runtime is running, and systemd restart sends the implicated SIGTERM; I did not run a live systemd repro in this read-only review.

## Solution Assessment

No. The PR direction is right, but the no-op must run after the existing repair checks, or only when no repair issue exists.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve service repair before the already-running no-op:** `src/daemon/service.ts:201-202`
  - body: This early return runs before `collectGatewayServiceStartRepairIssues`, so a loaded running service installed by an older OpenClaw version or pointing at a temporary/missing binary now reports `already-running` and skips the existing `repairLoadedService` path. Move the no-op after the repair checks, or only return it when no repair issue exists.
