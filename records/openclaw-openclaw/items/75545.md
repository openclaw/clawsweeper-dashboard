## Summary

Keep open: the PR targets a real gateway idempotency bug, but the current ordering would skip the existing stale-service repair path whenever the service is already running.

## What This Changes

The PR adds an `already-running` gateway start outcome, skips service-manager restart for running services, updates CLI reporting, tests, and changelog text.

## Best Possible Solution

Keep the idempotent start behavior, but preserve repair-required precedence by moving the already-running no-op after stale service repair checks and validating the lifecycle tests.

## Reproduction Assessment

Yes. Source inspection shows a login-shell env probe can source `~/.profile`, invoke `openclaw gateway start`, and currently reach `service.restart()` even when runtime state is running; the PR regression is also source-reproducible from the running default in the lifecycle harness plus the stale-service repair test.

## Solution Assessment

No, not as currently ordered. The shared `startGatewayService` fix is the right narrow surface, but the already-running return must not bypass existing stale service definition repair.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve stale-service repair before no-op start:** `src/daemon/service.ts:201-206`
  - body: This early return runs before `collectGatewayServiceStartRepairIssues`, so a running service installed by an old OpenClaw version, pointing at a missing binary, or pointing at a temporary installer path now reports `already-running` and never reaches `repairLoadedService`. Current main deliberately repairs those loaded-service definitions during `gateway start`, and the lifecycle harness defaults runtime to `running`, so move the no-op guard after the repair-required check or otherwise preserve that precedence.
  - confidence: 0.9
