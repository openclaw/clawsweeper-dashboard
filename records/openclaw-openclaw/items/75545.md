## Summary

Keep open: the PR targets a real current-main gateway restart bug, but the current diff would regress the existing repair-on-start path for running services with stale or broken service definitions.

## What This Changes

The PR adds an `already-running` gateway service start result, CLI output, tests, and a changelog entry so `openclaw gateway start` no-ops when the managed service is already running.

## Best Possible Solution

Keep the idempotent start behavior, but only return `already-running` after the existing loaded-service repair checks have had a chance to surface `repair-required`.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: exec performs a login-shell PATH probe, current-main managed start can reach the platform restart path, and systemd restart sends the reported SIGTERM. I did not run a live systemd/profile reproduction in this read-only review.

## Solution Assessment

No. The PR's direction is narrow and appropriate, but the guard is in the wrong place because it bypasses the existing repair-required checks for running services.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve service repair before the running no-op:** `src/daemon/service.ts:202`
  - body: This early return runs before `collectGatewayServiceStartRepairIssues`, so a loaded running service installed by an older OpenClaw version or pointing at a temporary/missing binary now reports `already-running` and skips `repairLoadedService`. Move the no-op after the repair checks, or only return it when no repair issue exists.
