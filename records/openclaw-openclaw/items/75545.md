## Summary

Keep this PR open for normal maintainer review. Current main still has the non-idempotent gateway start behavior described by the linked bug, and the proposed service-layer no-op for an already-running gateway is a narrow fix with matching CLI output, tests, and changelog coverage; I found no blocking review findings in the provided diff.

## What This Changes

The PR adds an `already-running` gateway start result, makes `startGatewayService` skip service-manager restart when runtime state is already running, updates CLI output and lifecycle tests, and adds a changelog fix entry for #60087.

## Best Possible Solution

Land the service-layer idempotency behavior with the CLI result branch, focused regression tests, and changelog entry after maintainer review and Testbox changed-gate validation, while keeping `openclaw gateway restart` as the explicit forced-cycle command.

## Reproduction Assessment

Yes. The reproduction path is source-backed: an exec login-shell environment probe can source a profile that runs `openclaw gateway start`, and current main then reaches `service.restart` even when service runtime state reports `running`.

## Solution Assessment

Yes. The proposed fix is the narrowest maintainable solution because it changes the shared service start contract instead of adding an exec-tool or shell-profile workaround, and it preserves forced restarts through the existing `gateway restart` command.
