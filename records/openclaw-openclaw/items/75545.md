## Summary

Keep open: the PR addresses a real source-backed gateway restart loop, but its `already-running` guard is currently placed before the existing repair-required checks, so running stale or broken service definitions can skip repair.

## What This Changes

This PR adds an `already-running` gateway service start outcome, CLI reporting, focused lifecycle tests, a changelog entry, and small formatting-only test changes.

## Best Possible Solution

Keep the idempotent start behavior, but return `already-running` only after the existing stale-version, temporary-program, and missing-program repair checks have had a chance to produce `repair-required`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: exec probes use a login shell, `gateway start` on current main reaches `service.restart()` for a running service, and the systemd adapter issues `systemctl --user restart`.

## Solution Assessment

No. The PR’s direction is the narrow maintainable fix, but the guard should run after the existing repair checks or be conditioned on there being no repair issues.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve repair checks before returning already-running:** `src/daemon/service.ts:202`
  - body: This early return runs before `collectGatewayServiceStartRepairIssues`, so a loaded running service installed by an older OpenClaw version or pointing at a temporary/missing binary now reports `already-running` and skips the existing repair path. Move the no-op after the repair checks, or only return it when no repair issue exists.
