## Summary

Keep open: the PR targets a valid gateway restart bug, but the current diff regresses the existing repair-on-start path by returning `already-running` before stale loaded service definitions can be repaired.

## What This Changes

The PR adds an `already-running` gateway service start outcome, CLI reporting, lifecycle tests, a changelog entry, and a few formatting-only changes outside the gateway path.

## Best Possible Solution

Keep the idempotent start behavior, but run the existing repair checks first and only return `already-running` for healthy running service definitions with regression coverage for both paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the exec probe runs a login shell, current main lets `gateway start` reach the service restart action, and the PR guard placement can be traced directly from the provided diff.

## Solution Assessment

No. The PR's direction is right, but the guard should move after `collectGatewayServiceStartRepairIssues` or otherwise require no repair issues before returning `already-running`.

Overall correctness: patch is incorrect
