## Label Justifications

- `P1`: The PR targets a high-impact gateway restart loop that drops WebSocket sessions on normal exec tool calls for affected managed-service users.
- `merge-risk: 🚨 compatibility`: The new early return can skip current loaded-service repair behavior for upgraded users with stale service definitions.
- `merge-risk: 🚨 availability`: The affected code controls Gateway service restarts, so a wrong merge can keep restart loops or stale service failures in an availability-sensitive path.

## Summary

Current main still has the central bug: `startGatewayService()` reaches `service.restart(...)` after repair checks with no already-running guard. The PR is useful but not merge-ready because the proposed guard is placed before the existing `repair-required` path, which would skip stale-service repair for running services, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds an `already-running` `GatewayServiceStartResult`, returns it from `startGatewayService()` when the managed service is already running, handles that outcome in daemon lifecycle CLI output, and adds focused unit coverage plus some unrelated formatting churn.

## Best Possible Solution

Keep the idempotent-start behavior, but move it after the existing repair-required decision, refresh the branch against current main, and add real systemd/profile exec proof that `gateway start` no longer restarts a running service.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: profile-sourced `openclaw gateway start` reaches `startGatewayService()`, and current main still calls `service.restart(...)` when the service is already running. I did not run a live systemd/profile exec scenario in this read-only review.

## Solution Assessment

No, not in the current patch shape. The idempotent-start fix belongs in `startGatewayService()`, but the new already-running guard must preserve the existing stale-service repair decision before returning success.
