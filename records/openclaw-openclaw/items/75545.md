## Summary

Keep this PR open: it targets a real current-main gateway lifecycle bug, but the patch appears to leave one existing config-guard test stale after changing `start` to no-op for already-running services.

## What This Changes

The PR adds an `already-running` gateway service start outcome, skips restart when service state is already running, reports that outcome through the CLI, updates focused lifecycle tests, and adds a changelog entry for the linked exec-triggered gateway restart bug.

## Best Possible Solution

Keep the service-layer idempotency behavior, CLI result branch, focused regression coverage, and changelog entry, then update the stale config-guard test so the changed gate can pass before maintainer review lands the PR.

## Reproduction Assessment

Yes. The underlying bug has a source-backed path through the exec login-shell environment probe into `openclaw gateway start`, and the PR test blocker is reproducible with the existing `runServiceStart` config-guard test setup because its mock service still reports `running`.

## Solution Assessment

Mostly yes, but not as submitted. The product fix belongs in the shared service start contract, but the PR should also update the config-guard test to set a stopped runtime when it expects a restart, or assert the new already-running no-op when that is the intended branch.
