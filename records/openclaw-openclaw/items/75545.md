## Summary

Keep this PR open: the bug and fix direction are valid, but the current head still skips the existing repair-on-start path for running stale service definitions and CI shows that regression.

## What This Changes

The PR adds an `already-running` gateway start outcome so running services no-op, updates CLI output/tests, adds a changelog entry, and includes a few formatting-only reflows.

## Best Possible Solution

Keep the idempotent `gateway start` behavior, but only no-op after the existing repair checks prove the installed service definition is healthy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for the linked bug: exec probes use a login shell, profile startup can call `openclaw gateway start`, and current main still reaches the service-manager restart path when the service is already running.

## Solution Assessment

No. The PR's direction is right, but the no-op guard needs to move after `collectGatewayServiceStartRepairIssues` or be conditioned on a healthy loaded service definition.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve repair before the running-service no-op:** `src/daemon/service.ts:201-204`
  - body: This early return runs before `collectGatewayServiceStartRepairIssues`, so a running service installed by an older OpenClaw version or pointing at a temporary/missing binary now reports `already-running` and never reaches the existing CLI repair path. Current main deliberately repairs those loaded services during `gateway start`, and this PR's CI shows the stale-service start tests failing because `repairLoadedService` is not called; move the no-op after the repair checks or only return it when no repair issue exists.
  - confidence: 0.96
