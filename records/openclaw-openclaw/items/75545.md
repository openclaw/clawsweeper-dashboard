## Summary

Keep open: this PR targets a real source-reproducible gateway restart bug, but the current diff returns `already-running` before the existing repair-on-start checks and would skip stale or broken service-definition repair when the service is already running.

## What This Changes

The PR adds an `already-running` gateway start outcome, surfaces it in the daemon CLI, updates focused lifecycle tests, and adds a changelog entry plus minor formatting-only test changes.

## Best Possible Solution

Move the idempotent no-op behind repair issue collection, or only return `already-running` when the loaded service definition has no repair issues.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: exec uses a login-shell env probe, `~/.profile` can run `openclaw gateway start`, current main start reaches `service.restart()`, and systemd restart sends the reported SIGTERM.

## Solution Assessment

No. The direction is narrow and appropriate, but the PR should preserve repair-on-start before treating an already-running service as a no-op.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve repair before the already-running no-op:** `src/daemon/service.ts:201`
  - body: This early return runs before `collectGatewayServiceStartRepairIssues`, so a loaded running service installed by an older OpenClaw version or pointing at a temporary/missing binary now reports `already-running` and skips the existing repair path. Move the no-op after the repair checks, or only return it when no repair issue exists.
