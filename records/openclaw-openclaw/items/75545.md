## Summary

Keep this PR open for normal maintainer review/merge. Current main still restarts an installed or loaded gateway service from `startGatewayService` without checking whether the runtime is already running, while the PR adds a narrow idempotency guard, CLI result handling, regression tests, and a changelog entry for the linked gateway/exec crash path.

## What This Changes

The PR adds an `already-running` `GatewayServiceStartResult`, skips `service.restart()` when gateway service state is already running, surfaces that outcome through `openclaw gateway start`, and adds focused lifecycle/service tests plus a changelog entry.

## Best Possible Solution

Land this PR after the normal changed-surface gate if maintainers agree with `gateway start` meaning "ensure running" for managed services, and keep any broader exec shell-profile side-effect work or #62086 behavior separate.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current main's service start path reaches `service.restart()` for installed or loaded services, and the linked issue shows `~/.profile` invoking `openclaw gateway start` during exec login-shell probing. I did not run the live systemd reproduction in this read-only review.

## Solution Assessment

Yes. The proposed `state.running` early return is the narrowest maintainable fix for the gateway service restart loop, preserves restart behavior for stopped services, and leaves forced cycling to `openclaw gateway restart`.
