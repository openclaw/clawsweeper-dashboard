## Summary

Keep open: the linked gateway restart bug is valid and this PR is aimed at the right shared lifecycle path, but the current patch regresses the existing service-repair path and the PR head has a failing CLI check from that behavior.

## What This Changes

The PR adds an `already-running` gateway service start outcome, CLI output, focused lifecycle tests, a changelog entry, and some formatting-only changes in unrelated tests.

## Best Possible Solution

Adjust the idempotent start guard so healthy running services no-op, while loaded services that need definition repair still return `repair-required` and run the existing repair path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the exec environment probe uses a login shell, `gateway start` on current main reaches `service.restart()` even when the service runtime is running, and systemd restart sends the SIGTERM reported in the linked issue.

## Solution Assessment

No. The PR's direction is right, but the guard should run after the existing repair-required checks, or otherwise only return `already-running` when the installed service definition is healthy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve service repair before the already-running no-op:** `src/daemon/service.ts:202`
  - body: This early return runs before `collectGatewayServiceStartRepairIssues`, so a loaded running service installed by an older OpenClaw version or pointing at a temporary/missing binary now reports `already-running` and skips `repairLoadedService`. Current main deliberately repairs those loaded services during `gateway start`, and the PR head is already failing the stale-service CLI test because the repair callback is no longer called. Move this no-op after the repair checks, or only return it when no repair issue exists.
  - confidence: 0.94
