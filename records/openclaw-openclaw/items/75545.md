## Summary

Keep open: the PR targets a real gateway-start idempotency bug, but the current patch returns `already-running` before the existing repair checks and would skip stale service repair during `gateway start`.

## What This Changes

The PR adds an `already-running` gateway service start result, CLI output, lifecycle tests, a changelog entry, and a few unrelated formatting-only test changes.

## Best Possible Solution

Keep the idempotent `start` behavior for healthy running services, but only return `already-running` after repair issues are ruled out or when the loaded definition is healthy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the exec environment probe uses a login shell, current main reaches `service.restart()` for a running managed service, and systemd activation uses `restart`.

## Solution Assessment

No. The PR's direction is right, but the guard should run after `collectGatewayServiceStartRepairIssues` or be conditioned on there being no repair issue, so existing repair-required behavior is preserved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve repair before the already-running no-op:** `src/daemon/service.ts:202`
  - body: This early return runs before `collectGatewayServiceStartRepairIssues`, so a loaded running service installed by an older OpenClaw version or pointing at a temporary/missing binary now reports `already-running` and skips `repairLoadedService`. Current main deliberately repairs those loaded services during `gateway start`; move this no-op after the repair checks, or only return it when no repair issue exists.
  - confidence: 0.94
