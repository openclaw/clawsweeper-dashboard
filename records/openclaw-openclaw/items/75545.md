## Summary

Keep open. The PR targets a real gateway exec restart bug, but the current head still returns `already-running` before the existing stale-service repair contract and CI confirms that regression; the external PR also lacks real behavior proof from the affected runtime path.

## What This Changes

The PR adds an `already-running` gateway service start outcome, CLI output and tests for no-oping an already-running service, a changelog entry, and a few formatting-only reflows.

## Best Possible Solution

Keep the idempotent `gateway start` behavior, but only return `already-running` after the existing repair checks prove the loaded service definition is healthy, then add redacted real runtime proof for the profile/exec/systemd path.

## Reproduction Assessment

Yes, from source inspection: the exec env probe uses a login shell, current main's `gateway start` path still reaches `service.restart`, and the systemd adapter implements that as `systemctl --user restart`. I did not run a live systemd reproduction in this read-only review.

## Solution Assessment

No. The PR direction is right, but the no-op guard needs to move after `collectGatewayServiceStartRepairIssues` or be conditioned on no repair issues so stale loaded services still repair before success.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve repair before the running-service no-op:** `src/daemon/service.ts:201-205`
  - body: This guard returns before `collectGatewayServiceStartRepairIssues`, so a running service installed by an older CLI or pointing at a temporary/missing binary now reports success and bypasses the existing repair path. Current main routes those cases through `repair-required`, and the PR CI shows the stale-service start tests failing because repair is no longer called; move the no-op after repair collection or only use it when no repair issue exists.
  - confidence: 0.96
