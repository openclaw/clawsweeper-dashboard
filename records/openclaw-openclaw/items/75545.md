## Summary

Keep open: current main still has the non-idempotent service-start path behind the linked gateway restart bug, and the proposed fix targets the shared lifecycle helper rather than a one-off exec workaround. I found no blocking correctness or security issue in the diff from the provided context and current main inspection.

## What This Changes

The PR adds an `already-running` gateway service start outcome, skips service restarts when runtime state is already running, surfaces that result in CLI output, updates focused lifecycle tests, and adds a changelog fix entry for #60087.

## Best Possible Solution

Land the idempotent start behavior after maintainer review and changed-gate validation, preserving `openclaw gateway restart` as the explicit forced-cycle command while making profile-driven `gateway start` safe as an ensure-running operation.

## Reproduction Assessment

Yes. Current main shows a high-confidence source reproduction path: a login-shell env probe can source profile startup, invoke `openclaw gateway start`, and then reach `service.restart()` even when service runtime state is already `running`.

## Solution Assessment

Yes. Fixing this in `startGatewayService` is the narrowest maintainable solution because gateway and node service entrypoints inherit the same ensure-running semantics without adding exec-tool or shell-profile special cases.
