## Summary

Keep open: current main does not implement CaMeL, and this PR still changes security-sensitive agent/tool execution with blocking taint propagation, config-resolution, and post-orchestration replay issues that need maintainer/security review.

## What This Changes

This PR adds an opt-in CaMeL prompt-injection defense under `src/agents/camel`, wires it into embedded agent execution and before-tool-call checks, extends config and exec approval surfaces, updates `SECURITY.md`, and adds tests.

## Best Possible Solution

Keep the PR open for explicit maintainer/security design review or replace it with a narrower prompt-injection boundary design aligned with #62939; any core CaMeL implementation should preserve provenance end-to-end, use one shared config resolver, and synthesize final replies without tool access after side effects run.

## Reproduction Assessment

Yes, source-reproducible. A static plan such as `web_fetch -> trusted transform/no-side-effect tool -> message.send` loses taint at `orchestrator.ts:153`, defaults-only config skips the hook path, and trusted side-effect plans can re-enter the tool-enabled session during synthesis.

## Solution Assessment

No. The CaMeL direction may be useful, but this patch is not mergeable until provenance propagation, shared config resolution, no-tool synthesis, and the security model are reviewed and fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve taint through intermediate tool results:** `src/agents/camel/orchestrator.ts:153`
  - body: The orchestrator wraps each result using only the destination tool name, so output from a trusted or no-side-effect intermediate tool is re-labeled as trusted even when its inputs were tainted. A `web_fetch -> transform/read-only tool -> message.send` chain can therefore clear provenance before the final side-effect policy check; derive or merge the wrapped input dependencies into the result.
  - confidence: 0.91
- **[P1] Disable tools for the synthesis turn:** `src/agents/pi-embedded-runner/run/attempt.ts:2178-2182`
  - body: After CaMeL has already executed the side-effect plan, this calls the normal tool-enabled session again and relies on prompt text telling the model not to rerun tools. For trusted-input side effects there may be no taint state for the hook to block, so the model can duplicate `write`, `exec`, or messaging actions before summarizing; synthesize with tools disabled or persist completed tool calls as non-executable context.
  - confidence: 0.86
- **[P2] Merge defaults into the hook CaMeL config:** `src/agents/pi-tools.before-tool-call.ts:187-195`
  - body: The embedded runner resolves `agents.defaults.camel`, `agents.camel`, and per-agent overrides, but the before-tool-call fallback only reads `agents.camel` and per-agent `camel`. A deployment that enables CaMeL under defaults parses and reaches the orchestrator path, but fallback/direct tool calls skip the hook-level CaMeL guard.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a security-sensitive prompt-injection defense, but concrete bypass and false-assurance paths remain.

Concerns:

- **[high] Taint can be laundered through intermediate tools:** `src/agents/camel/orchestrator.ts:153`
  - body: `CaMeLOrchestrator` rewraps tool results using only the callee name, and `TaintTracker` labels non-untrusted tools as trusted. This can drop provenance from tainted inputs before a later side-effect policy check.
  - confidence: 0.91
- **[high] Post-orchestration tools can replay side effects:** `src/agents/pi-embedded-runner/run/attempt.ts:2178`
  - body: The synthesis turn runs through the normal tool-enabled session after side effects already completed. Prompt text is not an enforcement boundary, and clean trusted-input actions may be duplicated without a CaMeL taint block.
  - confidence: 0.86
- **[medium] Defaults-only CaMeL config skips hook fallback:** `src/agents/pi-tools.before-tool-call.ts:187`
  - body: The orchestrator path reads `agents.defaults.camel`, but the before-tool-call path does not, so documented defaults-level enablement can leave fallback or direct tool calls without the advertised CaMeL policy check.
  - confidence: 0.86
