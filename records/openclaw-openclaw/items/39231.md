## Summary

Keep open. Current main still has no CaMeL implementation, so this PR is not obsolete, but the proposed patch is not merge-ready: the PR branch still has a concrete taint-laundering path through intermediate tools and a config-resolution mismatch between the orchestrator path and hook fallback. The PR is security-sensitive and should stay with maintainer/secops review rather than automated cleanup. https://github.com/openclaw/openclaw/pull/39231

## What This Changes

This PR adds an opt-in CaMeL prompt-injection defense under `src/agents/camel`, wires it into embedded agent execution and before-tool-call checks, extends agent config and exec approval schemas, updates `SECURITY.md`, and adds focused tests.

## Best Possible Solution

Keep the PR open for explicit maintainer and security review. If OpenClaw accepts CaMeL in core, the implementation should preserve provenance through every derived tool result, resolve CaMeL config through one shared contract for orchestrator and hook paths, align docs/schema/tests with that contract, and then run the appropriate changed gate.

## Reproduction Assessment

Yes. A static plan such as `web_fetch -> trusted transform/no-side-effect tool -> message.send` shows the taint-loss path because the intermediate result is wrapped from the callee name instead of its tainted inputs; a config using only `agents.defaults.camel` reproduces the hook fallback mismatch.

## Solution Assessment

No. The CaMeL direction may be valuable, but this patch is not the best mergeable form until the taint propagation and config-resolution defects are fixed and the security design is reviewed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve taint when wrapping tool results:** `src/agents/camel/orchestrator.ts:153`
  - body: The orchestrator wraps each result only from the destination tool name, so output from an intermediate trusted/no-side-effect tool becomes `TrustedTool` even when its arguments came from tainted data. A `web_fetch -> transform/read-only tool -> message.send` plan can therefore clear provenance before the final side-effect policy check; wrap results with the input dependencies/capabilities as well.
  - confidence: 0.91
- **[P2] Merge defaults into the hook CaMeL config:** `src/agents/pi-tools.before-tool-call.ts:187-195`
  - body: The runtime orchestrator merges `agents.defaults.camel`, but the before-tool-call fallback only reads `agents.camel` and per-agent overrides. A config that enables CaMeL under defaults parses and reaches the orchestrator, but any fallback to the normal tool loop skips the hook-level CaMeL guard, leaving direct tool calls unprotected.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces a security-sensitive prompt-injection defense, but the diff still has concrete bypass and false-assurance risks.

Concerns:

- **[high] Taint can be laundered through intermediate tools:** `src/agents/camel/orchestrator.ts:153`
  - body: `CaMeLOrchestrator` rewraps tool results using only the callee name, and `TaintTracker` labels non-untrusted tools as trusted. This can drop provenance from tainted inputs before a later side-effect policy check.
  - confidence: 0.91
- **[medium] Defaults-only CaMeL config skips hook fallback:** `src/agents/pi-tools.before-tool-call.ts:187`
  - body: The orchestrator path reads `agents.defaults.camel`, but the before-tool-call path does not. Documented defaults-level enablement can therefore leave fallback or direct tool calls without the advertised CaMeL policy check.
  - confidence: 0.86
