## Summary

Keep open: current main does not implement CaMeL, and this PR remains the canonical implementation candidate, but the proposed security boundary is not mergeable yet because source inspection confirms false-assurance paths and the contributor has not provided real behavior proof.

## What This Changes

The PR adds an opt-in CaMeL prompt-injection defense with taint-tracked values, P-LLM/Q-LLM orchestration, before-tool-call enforcement, config schema, exec-approval routing, SECURITY.md docs, and tests.

## Best Possible Solution

Land only a maintainer/secops-approved prompt-injection boundary that preserves provenance end to end, uses one enforcement path, prevents side-effect replay, derives approval trust internally, coordinates with https://github.com/openclaw/openclaw/issues/62939, and includes redacted real runtime proof.

## Reproduction Assessment

Yes for the PR review findings. Source inspection of the latest PR head gives high-confidence paths for taint laundering through intermediate results, post-orchestration tool replay, partial-failure replay, defaults-only hook bypass, hook rewrite bypass, and client-supplied approval-source bypass.

## Solution Assessment

No. CaMeL-style provenance may be useful, but this implementation is not the best mergeable form until the security contract, taint propagation, retry behavior, hook/config ordering, approval trust boundary, and runtime proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve taint when wrapping tool results:** `src/agents/camel/orchestrator.ts:153`
  - body: This wraps each tool result using only the executed tool name, so outputs from trusted or no-side-effect intermediate tools can lose the tainted argument provenance that produced them. Carry the wrapped argument dependencies into the result before later side-effect policy checks.
  - confidence: 0.92
- **[P1] Disable tools during final synthesis:** `src/agents/pi-embedded-runner/run/attempt.ts:2178-2182`
  - body: After CaMeL has already executed side-effect steps, this prompts the normal tool-enabled session and relies on prompt text asking the model not to re-run tools. Use a no-tool synthesis path or persist completed calls as non-executable context before prompting.
  - confidence: 0.88
- **[P1] Set the replay guard before side effects can fail:** `src/agents/pi-embedded-runner/run/attempt.ts:2154`
  - body: The replay guard is set only after orchestrator.execute returns. If one mutating step succeeds and a later step throws, the outer retry loop can enter CaMeL again and replay the already-completed side effect.
  - confidence: 0.84
- **[P2] Merge defaults into hook CaMeL config:** `src/agents/pi-tools.before-tool-call.ts:187-195`
  - body: The embedded runner resolves agents.defaults.camel, agents.camel, and per-agent overrides, but this hook fallback omits defaults. Defaults-only enablement can therefore fall back to the normal tool loop without the configured CaMeL guard.
  - confidence: 0.87
- **[P2] Re-check CaMeL after hook param rewrites:** `src/agents/pi-tools.before-tool-call.ts:395-397`
  - body: CaMeL policy runs before before_tool_call hooks can merge replacement params, so the arguments that actually execute may differ from the approved arguments. Re-run the CaMeL decision after hook rewrites or move the check after final params are known.
  - confidence: 0.78
- **[P2] Derive the CaMeL approval source internally:** `src/infra/exec-approval-forwarder.ts:71-75`
  - body: The new source field is accepted from exec.approval.request, and source="camel" bypasses the forwarding enabled gate. Derive this marker inside trusted CaMeL approval code instead of letting any approvals client force session approval forwarding while it is disabled.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a core security defense but leaves concrete false-assurance paths in taint propagation, side-effect replay, hook ordering, and approval-source trust.

Concerns:

- **[high] Taint can be laundered through intermediate tools:** `src/agents/camel/orchestrator.ts:153`
  - body: The orchestrator wraps tool results using only the executed tool name while non-untrusted tool outputs can be treated as trusted, so derived values can lose untrusted provenance before a later side-effect policy check.
  - confidence: 0.92
- **[high] Post-orchestration tools can replay side effects:** `src/agents/pi-embedded-runner/run/attempt.ts:2178`
  - body: The final synthesis turn uses the normal active session after side effects have completed, replacing enforcement with prompt text that asks the model not to call tools again.
  - confidence: 0.88
- **[high] Partial orchestration failures can replay side effects:** `src/agents/pi-embedded-runner/run/attempt.ts:2154`
  - body: The side-effect replay guard is set after orchestration returns, leaving completed mutating steps replayable if a later step throws and the outer run loop retries.
  - confidence: 0.84
- **[medium] Hook path can skip defaults-only CaMeL enablement:** `src/agents/pi-tools.before-tool-call.ts:187`
  - body: The before-tool-call fallback omits agents.defaults.camel, so a deployment that enables CaMeL through defaults can enter the normal tool loop without the configured guard.
  - confidence: 0.87
- **[medium] Hook rewrites can change approved arguments:** `src/agents/pi-tools.before-tool-call.ts:395`
  - body: CaMeL checks the original params before plugin hooks can rewrite them, so the executed arguments can differ from the policy decision.
  - confidence: 0.78
- **[medium] Approval forwarding trusts a client-supplied source marker:** `src/infra/exec-approval-forwarder.ts:71`
  - body: Any exec approval client can submit source="camel" and bypass the forwarding enabled gate; this trust bit should be derived inside the trusted CaMeL approval path.
  - confidence: 0.82
