## Summary

Keep open: current main does not include this CaMeL feature, and the PR remains a broad security-sensitive change with source-reproducible enforcement gaps that should block merge until maintainer/security design review.

## What This Changes

This PR adds an opt-in CaMeL prompt-injection defense across agent execution, before-tool-call checks, config schemas, gateway exec approvals, SECURITY.md, and new tests.

## Best Possible Solution

Keep this open for maintainer/security review or replace it with a narrower prompt-injection boundary design aligned with #62939 that preserves provenance end-to-end, resolves config through one path, and synthesizes final replies without tool access after side effects run.

## Reproduction Assessment

Yes. Source inspection against the PR head gives high-confidence paths: a web_fetch to trusted transform to message.send plan can lose taint at orchestrator.ts:153, defaults-only config can skip the hook path, and post-orchestration synthesis can run with normal tools still enabled.

## Solution Assessment

No. The CaMeL direction may be useful, but this patch is not the best mergeable implementation until provenance propagation, shared config resolution, no-tool synthesis, and the security boundary are reviewed and fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve taint through intermediate tool results:** `src/agents/camel/orchestrator.ts:153`
  - body: The orchestrator rewraps each result using only the callee name, so a tainted value can pass through a trusted or no-side-effect transform and become trusted before a later side-effect policy check. Derive the result capabilities from the wrapped input dependencies so provenance survives multi-step plans.
  - confidence: 0.91
- **[P1] Disable tools for the synthesis turn:** `src/agents/pi-embedded-runner/run/attempt.ts:2178-2182`
  - body: After CaMeL executes the side-effect plan, this prompts the normal tool-enabled session again and relies on text telling the model not to rerun tools. Clean trusted-input actions may be duplicated before the summary; synthesize with tools disabled or persist completed tool calls as non-executable context.
  - confidence: 0.86
- **[P2] Merge defaults into the hook CaMeL config:** `src/agents/pi-tools.before-tool-call.ts:187-195`
  - body: The embedded runner resolves agents.defaults.camel, agents.camel, and per-agent overrides, but the before-tool-call fallback only reads agents.camel and per-agent camel. A deployment enabling CaMeL under defaults can parse and reach the orchestrator path while fallback/direct tool calls skip the hook guard.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a security-sensitive prompt-injection defense, but concrete bypass and false-assurance paths remain.

Concerns:

- **[high] Taint can be laundered through intermediate tools:** `src/agents/camel/orchestrator.ts:153`
  - body: CaMeLOrchestrator rewraps tool results using only the callee name, and TaintTracker labels non-untrusted tools as trusted. This can drop provenance from tainted inputs before a later side-effect policy check.
  - confidence: 0.91
- **[high] Post-orchestration tools can replay side effects:** `src/agents/pi-embedded-runner/run/attempt.ts:2178`
  - body: The synthesis turn runs through the normal tool-enabled session after side effects already completed. Prompt text is not an enforcement boundary, and clean trusted-input actions may be duplicated without a CaMeL taint block.
  - confidence: 0.86
- **[medium] Defaults-only CaMeL config skips hook fallback:** `src/agents/pi-tools.before-tool-call.ts:187`
  - body: The orchestrator path reads agents.defaults.camel, but the before-tool-call path does not, so documented defaults-level enablement can leave fallback or direct tool calls without the advertised CaMeL policy check.
