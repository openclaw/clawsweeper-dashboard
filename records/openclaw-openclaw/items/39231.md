## Summary

Keep open for maintainer/security review. Current main still has no CaMeL runtime/config surface, and the PR head still has blocking false-assurance paths in taint propagation, side-effect replay, config resolution, and approval-routing trust, plus no real behavior proof.

## What This Changes

Adds an optional CaMeL-style prompt-injection defense with taint-tracked values, P-LLM/Q-LLM orchestration, before-tool-call enforcement, config schema, gateway exec approvals, SECURITY.md, and tests.

## Best Possible Solution

Land only a maintainer/secops-approved prompt-injection boundary design that preserves provenance end to end, prevents side-effect replay, resolves config and hook ordering through one path, and keeps approval-source trust internal.

## Reproduction Assessment

Yes for the PR review findings. PR-head source inspection gives high-confidence paths for taint laundering through intermediate results, tool-enabled synthesis after side effects, partial-failure replay, defaults-only hook bypass, hook rewrite bypass, and client-supplied approval-source bypass.

## Solution Assessment

No. CaMeL-style provenance may be a useful direction, but this PR is not the best mergeable implementation until the security contract, taint propagation, retry behavior, config resolution, approval-routing trust boundary, and real behavior proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve taint dependencies when wrapping tool results:** `src/agents/camel/orchestrator.ts:153`
  - body: This wraps each tool result using only the executed tool name, so output from a trusted or no-side-effect intermediate tool can lose the tainted argument provenance that produced it. Carry the wrapped argument dependencies into the result value before any later side-effect check.
  - confidence: 0.92
- **[P1] Disable tools during post-orchestration synthesis:** `src/agents/pi-embedded-runner/run/attempt.ts:2178-2182`
  - body: After CaMeL has already run side-effect steps, this prompts the normal tool-enabled session and relies on prompt text telling the model not to re-run tools. Use a no-tool synthesis path or persist completed calls as non-executable context before prompting.
  - confidence: 0.88
- **[P1] Set the replay guard before side effects can fail:** `src/agents/pi-embedded-runner/run/attempt.ts:2154`
  - body: The retry guard is set only after orchestrator.execute returns. If one mutating step succeeds and a later step throws, the outer retry loop can enter CaMeL again and replay the already-completed side effect.
  - confidence: 0.84
- **[P2] Merge defaults into hook CaMeL config:** `src/agents/pi-tools.before-tool-call.ts:187-195`
  - body: The embedded runner resolves agents.defaults.camel, agents.camel, and per-agent overrides, but this hook fallback omits defaults. Defaults-only enablement can therefore fall back to the normal tool loop without the configured CaMeL guard.
  - confidence: 0.87
- **[P2] Recheck CaMeL after hook parameter rewrites:** `src/agents/pi-tools.before-tool-call.ts:395-397`
  - body: CaMeL policy runs before before_tool_call hooks can merge replacement params, so the arguments that actually execute may differ from the approved arguments. Re-run the CaMeL decision after hook rewrites or move the check after final params are known.
  - confidence: 0.78
- **[P2] Do not trust client-supplied approval source:** `src/infra/exec-approval-forwarder.ts:71-75`
  - body: The new source field is accepted from exec.approval.request and source="camel" bypasses the forwarding enabled gate. Derive this marker internally for CaMeL requests instead of letting any operator.approvals client force session approval forwarding while it is disabled.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a core prompt-injection defense, but concrete taint-laundering, replay, config-bypass, hook-ordering, and approval-routing false-assurance paths remain.

Concerns:

- **[high] Taint can be laundered through intermediate tools:** `src/agents/camel/orchestrator.ts:153`
  - body: The orchestrator wraps tool results using only the executed tool name while non-untrusted tool outputs can be treated as trusted, so derived values can lose untrusted provenance before a later side-effect policy check.
  - confidence: 0.92
- **[high] Post-orchestration tools can replay side effects:** `src/agents/pi-embedded-runner/run/attempt.ts:2178`
  - body: The final synthesis turn uses the normal activeSession.prompt path after side effects have completed, so enforcement is replaced by prompt text asking the model not to call tools again.
  - confidence: 0.88
- **[high] Partial orchestration failures can replay side effects:** `src/agents/pi-embedded-runner/run/attempt.ts:2154`
  - body: The side-effect replay guard is set after orchestrator.execute returns, leaving completed mutating steps replayable if a later step fails before the guard is assigned.
  - confidence: 0.84
- **[medium] Approval-source trust crosses the client boundary:** `src/infra/exec-approval-forwarder.ts:71`
  - body: Any exec approval requester can supply source="camel" and bypass the forwarding enabled gate, so the trusted CaMeL marker should be derived server-side rather than accepted from protocol input.
  - confidence: 0.82
