## Summary

Keep open: current main does not implement the CaMeL engine/config surface, and this PR is a security-sensitive new runtime with blocking provenance, replay, and config-resolution defects plus missing real behavior proof.

## What This Changes

The PR adds an opt-in CaMeL prompt-injection defense with a new agent taint/provenance engine, runner and before-tool-call integration, config schema, gateway approval metadata, SECURITY.md docs, and tests.

## Best Possible Solution

Keep this open for maintainer/security review or replace it with a narrower prompt-injection boundary design that preserves provenance end-to-end, resolves CaMeL config through one shared path, and synthesizes final replies without tool access after side effects run.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence on the PR head: a tainted value can pass through an intermediate trusted/no-side-effect tool, defaults-only config can skip fallback hook enforcement, and post-orchestration synthesis can still call normal tools. I did not run a live runtime scenario because this is a read-only review.

## Solution Assessment

No. CaMeL-style provenance is directionally useful, but this patch is not the best mergeable implementation until provenance propagation, shared config resolution, no-tool synthesis, and the security boundary are fixed and reviewed.

  - body: The orchestrator rewraps each result using only the callee name, so a tainted value can pass through a trusted or no-side-effect transform and become trusted before a later side-effect policy check.   - confidence: 0.91

  - body: After CaMeL executes the side-effect plan, this prompts the normal tool-enabled session again and relies on text telling the model not to rerun tools. Clean trusted-input actions may be duplicated before the summary; synthesize with tools disabled or persist completed tool calls as non-executable context.
  - confidence: 0.86
- **[P2] Merge defaults into the hook CaMeL config:** `src/agents/pi-tools.before-tool-call.ts:187-195`
  - body: The embedded runner resolves agents.defaults.camel, agents.camel, and per-agent overrides, but the before-tool-call fallback only reads agents.camel and per-agent camel. A deployment enabling CaMeL under defaults can parse and reach the orchestrator path while fallback or direct tool calls skip the hook guard.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a security-sensitive prompt-injection defense, but concrete bypass and false-assurance paths remain.

Concerns:

- **[high] Taint can be laundered through intermediate tools:** `src/agents/camel/orchestrator.ts:153`
  - body: CaMeLOrchestrator rewraps tool results using only the callee name, and TaintTracker labels non-untrusted tools as trusted. This can drop provenance from tainted inputs before a later side-effect policy check.
  - confidence: 0.91
- **[high] Post-orchestration tools can replay side effects:** `src/agents/pi-embedded-runner/run/attempt.ts:2178`
  - body: The synthesis turn runs through the normal tool-enabled session after side effects already completed. Prompt text is not an enforcement boundary, and trusted-input actions may be duplicated without a CaMeL taint block.
  - confidence: 0.86
- **[medium] Defaults-only CaMeL config skips hook fallback:** `src/agents/pi-tools.before-tool-call.ts:187`
  - body: The orchestrator path reads agents.defaults.camel, but the before-tool-call path does not, so documented defaults-level enablement can leave fallback or direct tool calls without the advertised CaMeL policy check.
