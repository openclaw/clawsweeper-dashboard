## Summary

Keep open. Current main does not implement the CaMeL surface, so the PR is not obsolete, but the branch is not merge-ready: the proposed security boundary still has concrete taint-provenance and config-resolution gaps that could give operators false assurance.

## What This Changes

This PR adds an opt-in CaMeL prompt-injection defense under `src/agents/camel`, wires it into embedded agent execution, before-tool-call checks, agent config schemas, exec approval forwarding, SECURITY.md, and focused tests.

## Best Possible Solution

Keep the PR open for explicit maintainer and secops review. If OpenClaw accepts CaMeL in core, the implementation should preserve taint through every tool result, apply one config-resolution contract across orchestrator and hook paths, prove isolated planner/extractor sessions and replay-safe side effects, align docs/schema/tests, and then run the appropriate changed gate.

## Reproduction Assessment

Yes for the blocking PR defects: a static plan such as `web_fetch -> trusted transform -> message.send` shows that `wrapToolResult` reclassifies the transform output from the callee name instead of the tainted inputs. The defaults mismatch is also directly reproducible by configuring only `agents.defaults.camel` and observing that the hook fallback does not read it.

## Solution Assessment

No. The CaMeL direction may be valuable, but this diff is not the best mergeable form until the taint propagation and config contract are corrected and reviewed as a security design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve taint when wrapping tool results:** `src/agents/camel/orchestrator.ts:153`
  - body: The orchestrator wraps each result only from the destination tool name, so output from an intermediate trusted/no-side-effect tool becomes `TrustedTool` even when its arguments came from tainted data. A `web_fetch -> transform/read-only tool -> message.send` plan can therefore clear provenance before the final side-effect policy check; wrap results with the input dependencies/capabilities as well.
  - confidence: 0.91
- **[P2] Merge defaults into the hook CaMeL config:** `src/agents/pi-tools.before-tool-call.ts:187-195`
  - body: The runtime orchestrator merges `agents.defaults.camel`, but the before-tool-call fallback only reads `agents.camel` and per-agent overrides. A config that enables CaMeL under defaults parses and reaches the orchestrator, but any fallback to the normal tool loop skips the hook-level CaMeL guard, leaving direct tool calls unprotected.
  - confidence: 0.86

## Security Review

Summary: The PR introduces a security-sensitive prompt-injection defense, but the diff still has concrete bypass/false-assurance risks.

  - body: `CaMeLOrchestrator` rewraps tool results using only the callee name, and `TaintTracker` labels non-untrusted tools as trusted. This can drop provenance from tainted inputs before a later side-effect policy check.
