## Summary

Keep open. Current main and the latest release still treat plugin slots as scalar plugin-id selectors, while this RFC asks for a new public config/plugin contract with owner provenance and overwrite policy; the broader memory composition discussion at https://github.com/openclaw/openclaw/issues/60572 overlaps but does not supersede this narrower ownership contract.

## What This Changes

The issue requests a backward-compatible `plugins.slots.<slot>` owner record with provenance fields and an overwrite invariant instead of bare string slot selectors.

## Best Possible Solution

Maintainers should choose a minimal backward-compatible ownership contract first, then implement it through config types/schema, normalization, docs/tests, canonical writes, and explicit override semantics.

## Reproduction Assessment

Not applicable: this is an RFC for a public config/plugin contract, not a bug report. Source, docs, tests, and release inspection give high-confidence proof that current behavior remains scalar slot selection.

## Solution Assessment

Unclear until maintainer API review. The staged rollout is plausible, but field names, provenance guarantees, migration behavior, and overwrite policy are public API choices rather than a safe automated cleanup fix.
