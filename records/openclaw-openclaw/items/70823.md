## Summary

Keep open. Current main still models plugin slots as bare string selectors, while this RFC asks for a public config/schema/plugin contract change with owner/provenance metadata and overwrite policy that needs maintainer API judgment.

## What This Changes

The issue requests a backward-compatible `plugins.slots.<slot>` owner record with optional provenance fields so plugins and core can distinguish slot ownership from a bare selected plugin id.

## Best Possible Solution

Have maintainers decide the minimal slot ownership contract first, then land a staged backward-compatible implementation covering read normalization, schema/docs/tests, canonical writes, and explicit overwrite behavior if accepted.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC. The source-level verification is high-confidence because current main still accepts, normalizes, validates, writes, tests, and documents slot values as strings rather than ownership records.

## Solution Assessment

Unclear until maintainer design review. The proposed staged rollout is plausible, but field names, provenance guarantees, migration behavior, and overwrite enforcement are public API choices rather than a safe autonomous cleanup.
