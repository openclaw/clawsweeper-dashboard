## Summary

Keep open. Current main still treats plugin slots as string selectors, while this RFC asks for a new public config/plugin contract with owner provenance and overwrite policy that needs maintainer API judgment.

## What This Changes

Request to evolve `plugins.slots.<slot>` from a bare plugin id string into a normalized owner/provenance record with safer overwrite semantics.

## Best Possible Solution

Have maintainers decide the minimal public slot ownership contract first, then stage any accepted implementation through types/schema, read normalization, write paths, docs, generated schema, and regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new public config/schema contract. The source check is high-confidence because current main still types, validates, normalizes, writes, tests, and documents slot values as strings.

## Solution Assessment

Unclear until maintainer design review. The staged rollout in the RFC is plausible, but the field set, provenance guarantees, migration behavior, and overwrite policy are public API choices rather than a safe cleanup fix.
