## Summary

Keep open. Current main and the latest release still model `plugins.slots.*` as scalar plugin-id selectors, while this RFC asks for a new public config/plugin contract with owner provenance and overwrite policy that needs maintainer API review.

## What This Changes

Request a backward-compatible `plugins.slots.<slot>` owner-record shape with provenance fields and overwrite policy, while continuing to accept existing string slot values.

## Best Possible Solution

Have maintainers choose the minimal public slot ownership contract first, then stage accepted work through schema/types, read normalization, docs/tests, canonical writes, and explicit override semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new public config/plugin contract. Source, docs, tests, and release inspection give high-confidence proof that current behavior is still scalar slot selection.

## Solution Assessment

Unclear until maintainer design review. The staged rollout is plausible, but the field set, migration behavior, provenance guarantees, and overwrite policy are public API choices rather than a safe automated cleanup fix.
