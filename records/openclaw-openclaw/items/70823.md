## Summary

Keep open. Current main still treats plugin slots as bare string selectors, and this RFC asks for a new public config/schema/plugin contract with provenance fields and overwrite policy that needs maintainer API/product judgment.

## What This Changes

The issue proposes evolving `plugins.slots.<slot>` from a bare plugin-id string into a backward-compatible owner/provenance record such as `{ owner, claimed_at, claimed_by_version }`.

## Best Possible Solution

Have maintainers decide the minimal slot ownership contract first, then stage any accepted implementation through schema/types, read normalization, docs/tests, canonical writes, and explicit overwrite semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new public config/plugin contract. Source inspection gives high-confidence proof that current main still uses bare string slot selectors.

## Solution Assessment

Unclear until maintainer design review. The staged rollout is plausible, but the field set, provenance guarantees, migration behavior, and overwrite policy are public API choices rather than a safe automated cleanup fix.
