## Summary

Keep open. Current main still treats plugin slots as bare string selectors, while this RFC asks for a public config/schema/plugin contract change with owner/provenance metadata and overwrite policy that needs maintainer API judgment.

## What This Changes

Request a backward-compatible extension so `plugins.slots.<slot>` can expose machine-readable ownership metadata such as `owner`, `claimed_at`, and `claimed_by_version`.

## Best Possible Solution

Have maintainers decide the minimal slot ownership contract first, then stage any accepted implementation through read normalization, schema/docs/tests, canonical writes, and explicit overwrite semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new public config/schema contract. The source check is high-confidence because current main still types, validates, normalizes, writes, tests, and documents slot values as strings.

## Solution Assessment

Unclear until maintainer design review. The staged rollout in the issue is plausible, but the field set, provenance guarantees, and overwrite policy are public API choices rather than a safe cleanup fix.
