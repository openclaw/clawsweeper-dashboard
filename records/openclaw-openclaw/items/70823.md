## Summary

Keep open. Current main still treats plugin slots as scalar plugin-id selectors, while this RFC asks for a new public config/plugin contract with provenance, migration, and overwrite-policy semantics that need maintainer design.

## What This Changes

The issue requests evolving `plugins.slots.<slot>` from a string selector into a backward-compatible owner record with provenance fields and explicit overwrite rules.

## Best Possible Solution

Have maintainers decide the minimal versioned slot ownership contract first, then stage any accepted work through schema/types, read normalization, docs/tests, canonical writers, and explicit overwrite semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new public config/plugin contract. Source, docs, and tests give high-confidence proof that current main still uses bare string slot selectors.

## Solution Assessment

Unclear until maintainer design review. The staged rollout is plausible, but the field set, provenance guarantees, migration behavior, and overwrite policy are public API choices rather than a safe automated cleanup fix.
