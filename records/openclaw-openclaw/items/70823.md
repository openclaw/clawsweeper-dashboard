## Summary

Keep open. Current main still treats plugin slots as string selectors, while this RFC asks for a public config/schema/plugin contract change with owner/provenance metadata and overwrite policy that needs maintainer API judgment.

## What This Changes

The issue proposes evolving `plugins.slots.<slot>` from a bare plugin-id string into a backward-compatible ownership record with `owner`, `claimed_at`, and `claimed_by_version`.

## Best Possible Solution

Have maintainers decide the minimal slot ownership contract first, then land a staged backward-compatible implementation: read normalization and docs/tests, canonical writes with provenance if accepted, and explicit overwrite/force behavior only after compatibility rules are settled.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC. The source-level check is high-confidence because current main still accepts, normalizes, validates, writes, tests, and documents slot values as strings.

## Solution Assessment

Unclear until maintainer design review. The proposed staged rollout is plausible, but field names, provenance guarantees, and overwrite enforcement are public API choices rather than a safe cleanup decision.
