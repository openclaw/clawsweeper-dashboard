## Summary

Keep open. Current main and the latest release still expose `plugins.slots.*` as scalar plugin-id selectors, while this RFC asks for a new public config/plugin contract with provenance fields and overwrite semantics that needs maintainer API/product review.

## What This Changes

Add a backward-compatible owner/provenance object form for `plugins.slots.<slot>` and eventually enforce explicit overwrite policy for slot claims.

## Best Possible Solution

Have maintainers choose the minimal public slot ownership contract first, then stage accepted work through schema/types, read normalization, docs/tests, canonical writes, and explicit override semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new public config/plugin contract. Source, docs, and release inspection give high-confidence proof that current behavior is still scalar slot selection.

## Solution Assessment

Unclear until maintainer design review. The staged rollout is plausible, but the field set, migration behavior, provenance guarantees, and overwrite policy are public API choices rather than a safe automated cleanup fix.
