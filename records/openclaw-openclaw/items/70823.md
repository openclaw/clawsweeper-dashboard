## Summary

Keep open. Current main still treats `plugins.slots.*` as scalar plugin-id selectors, while this RFC asks for a new public config/plugin contract with provenance fields, migration behavior, and overwrite policy that needs maintainer API/product design.

## What This Changes

The RFC asks OpenClaw to evolve `plugins.slots.<slot>` from a bare plugin-id string into a backward-compatible owner record with provenance and overwrite semantics.

## Best Possible Solution

Have maintainers decide the minimal slot ownership contract first, then stage any accepted work through schema/types, read normalization, docs/tests, canonical writes, and explicit overwrite semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new public config/plugin contract. Source, docs, and tests give high-confidence proof that current main still uses bare string slot selectors.

## Solution Assessment

Unclear until maintainer design review. The staged rollout is plausible, but the field set, provenance guarantees, migration behavior, and overwrite policy are public API choices rather than a safe automated cleanup fix.
