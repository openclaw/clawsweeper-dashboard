## Summary

Keep open. Current main still models plugin slots as string selectors across the public config type, runtime schema, normalization, validation, install selection writes, generated schema, and docs. The external plur-ai/plur context is a useful motivator, but the requested owner/provenance object and overwrite policy are public plugin/config API decisions that need maintainer review before implementation.

## What This Changes

The issue requests a backward-compatible `plugins.slots.<slot>` ownership record with an `owner` field and optional provenance metadata so plugins and core can distinguish slot ownership from a bare selected plugin id.

## Best Possible Solution

Have maintainers decide the minimal slot ownership contract first. If accepted, land a backward-compatible read model with schema/docs/tests, then canonical object writes with provenance metadata, and only after that add an explicit overwrite/force invariant once compatibility behavior is settled.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC. The source-level verification is high-confidence: current main still accepts, normalizes, validates, writes, and documents slot values as strings rather than ownership records.

## Solution Assessment

Unclear until maintainer design review. The proposed staged rollout is plausible, but the field set, provenance semantics, migration behavior, and overwrite policy are public API decisions rather than a safe autonomous cleanup.
