## Summary

Keep open. The PR changes a behavior-sensitive service lifecycle contract: current main keeps DBus/user-bus `is-enabled` transport failures on the error path, install already has a scoped non-fatal probe path, and other lifecycle callers consume `false` as a successful not-loaded state.

## What This Changes

The PR changes `isSystemdServiceEnabled` so DBus/user-bus unavailable `systemctl is-enabled` failures return `false` instead of throwing.

## Best Possible Solution

Keep the install-specific non-fatal handling unless maintainers intentionally redesign the broader service-loaded contract. If they accept the broader contract change, update lifecycle callers so disabled, missing, and unknown transport states remain distinct, then add targeted install, status, reset, and uninstall coverage.

## Reproduction Assessment

Yes. Existing mocked systemd tests model direct and exhausted machine-scope DBus/user-bus failures and assert rejection; caller inspection shows a `false` result is consumed as a successful not-loaded state in uninstall.

## Solution Assessment

No. The one-line change is not the narrowest maintainable fix because it collapses transport failure into disabled/not-loaded state; the safer path is the existing install-scoped helper or an explicit unknown transport state with coordinated caller updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve DBus probe failures as errors:** `src/daemon/systemd.ts:635`
  - body: Returning `false` here turns an exhausted DBus/user-bus transport failure into "service not loaded." Callers such as `openclaw uninstall --service` treat `false` as a successful not-loaded state and skip stop/uninstall, so an installed service can remain after an unavailable bus. Keep this as an error path or update lifecycle callers/tests to distinguish unknown from not loaded.
  - confidence: 0.9
