## Summary

Keep open. The PR is a targeted one-line change, but it is not mergeable as-is because it turns a DBus transport failure into a false loaded-state result that lifecycle callers treat as successful not-loaded cleanup.

## What This Changes

The PR adds `isSystemctlBusUnavailable(detail)` to `isSystemdServiceEnabled` so DBus/user-bus `systemctl is-enabled` failures return `false` instead of throwing.

## Best Possible Solution

Keep the generic `isSystemdServiceEnabled` transport-failure path distinct from disabled/not-loaded; if broader non-fatal behavior is desired, introduce an explicit unknown state or update lifecycle callers and tests together.

## Reproduction Assessment

Yes. The current mocked systemd tests reproduce direct and exhausted machine-scope DBus/user-bus failures and assert rejection; applying the PR would change those paths to a `false` loaded-state result.

## Solution Assessment

No. This is not the narrowest maintainable fix because it collapses transport failure into disabled/not-loaded state; the safer existing path is the install-scoped non-fatal helper unless maintainers intentionally redesign the lifecycle contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve DBus probe failures as errors:** `src/daemon/systemd.ts:635`
  - body: Returning `false` here turns an exhausted DBus/user-bus transport failure into "service not loaded." Callers such as `openclaw uninstall --service` treat `false` as success and skip stop/uninstall, so an installed service can remain when the bus is unavailable. Keep this as an error path or update lifecycle callers/tests to distinguish unknown from not loaded.
  - confidence: 0.9
