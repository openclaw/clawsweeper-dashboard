## Summary

Keep open: current main still routes Feishu P2P thread events through the non-thread direct-message path, and the linked fix PR is open but not merged.

## What This Changes

The issue reports that Feishu direct-message thread replies are sent as standalone messages instead of staying inside the thread.

## Best Possible Solution

Refresh and land https://github.com/openclaw/openclaw/pull/73972, or replace it with an equivalent mergeable Feishu plugin patch that treats direct messages carrying thread_id as thread replies while preserving root-id-only quote replies and group behavior.

## Reproduction Assessment

Yes, source-level. A Feishu p2p event with root_id and thread_id reaches ctx.rootId and ctx.threadId, then current main leaves direct-message replyInThread/threadReply false and drops reply metadata before send.

## Solution Assessment

Yes at the issue level. The narrow Feishu plugin routing fix is the maintainable owner-boundary path because the send layer already supports reply_in_thread and the remaining work is selecting the correct P2P thread reply target.
