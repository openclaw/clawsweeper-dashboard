## Label Justifications

- `P1`: The issue describes an active regression that blocks over-threshold Claude CLI runtime channel turns before model dispatch.
- `impact:message-loss`: The failure replaces the expected assistant reply with a missing-harness warning on inbound channel turns.
- `impact:session-state`: The failed preflight prevents compaction and forces users toward `/new` or `/reset`, losing live conversation continuity.
- `impact:auth-provider`: The failure is tied to model-scoped Claude CLI runtime/auth routing being treated as an embedded harness id.

## Summary

Keep open: current main still sends model-scoped `claude-cli` runtime policy through embedded harness selection during compaction preflight, so the reported missing-harness failure remains source-reproducible.

## What This Changes

The issue asks OpenClaw to bypass embedded harness selection for CLI runtime aliases during compaction preflight so long Claude CLI runtime sessions can continue replying and compacting.

## Best Possible Solution

Land a corrected version of https://github.com/openclaw/openclaw/pull/83552 or an equivalent narrow fix that bypasses embedded harness selection only for provider-owned CLI runtime aliases while preserving context-engine compaction.

## Reproduction Assessment

Yes at source level: current main still sends explicit `claude-cli` runtime policy into `selectAgentHarness` during compaction preflight, where non-harness runtime ids throw `MissingAgentHarnessError`. I did not run a live over-threshold Claude CLI channel session in this read-only review.

## Solution Assessment

Yes, the requested direction is the right narrow fix, but the implementation should validate that the CLI alias belongs to the selected provider/model. The open PR already covers the central path and needs that ownership guard before landing.
