## Label Justifications

- `P1`: The issue describes an active regression that can block Claude CLI-backed channel turns once compaction preflight crosses threshold.
- `impact:message-loss`: The reported failure replaces the expected assistant response with a missing-harness warning on inbound channel turns.
- `impact:session-state`: The failure prevents compaction and leaves long-running sessions growing toward overflow or requiring a reset.
- `impact:auth-provider`: The affected path is model-scoped Claude CLI runtime/auth routing being misread as an embedded harness id.

## Summary

Keep open: current main still has the reported CLI-runtime versus embedded-harness mismatch in the compaction preflight path. The related open fix PR at https://github.com/openclaw/openclaw/pull/83552 targets the same bug, but it is unmerged and currently has a routing-safety review finding.

## What This Changes

The issue requests fixing compaction preflight so model-scoped `agentRuntime.id: "claude-cli"` is treated as a CLI backend alias rather than an embedded harness id.

## Best Possible Solution

Land a narrow fix that bypasses embedded harness selection only for provider-owned CLI runtime aliases, preserves `MissingAgentHarnessError` for mismatched aliases, and keeps context-engine compaction running with focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main still resolves the model-scoped `claude-cli` runtime through `selectAgentHarness` during compaction preflight, and that selector still throws for explicit non-harness runtimes. I did not run a live over-threshold Claude CLI session in this read-only review.

## Solution Assessment

Yes, the requested direction is the right fix boundary: make compaction preflight respect provider-owned CLI runtime aliases instead of treating them as embedded harness ids. The implementation should preserve context-engine compaction and avoid the over-broad alias fallback currently flagged on the related PR.
