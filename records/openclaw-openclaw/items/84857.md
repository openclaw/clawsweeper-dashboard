- `impact:message-loss`: The reported failure replaces the expected assistant response with the missing-harness warning on inbound channel turns.

- `impact:auth-provider`: The affected path is the Claude CLI runtime/auth selection created by model-scoped runtime policy.

## Summary

Keep open: current main still routes the model-scoped `claude-cli` runtime into embedded harness selection during compaction preflight, where it can throw the reported missing-harness error. An open closing PR already targets this bug, so the issue should stay open until a fix is merged.

## What This Changes

The issue asks OpenClaw to make compaction preflight respect CLI runtime aliases such as `claude-cli` instead of treating them as embedded harness IDs.

## Best Possible Solution

Review and land one focused fix PR, preserving context-engine compaction while bypassing embedded harness selection for CLI runtime aliases, then close this issue after the merge.

## Reproduction Assessment

Yes at source level: current main resolves model-scoped `claude-cli` runtime policy and then passes it into harness selection, which throws for unmatched explicit non-harness runtimes. I did not execute a live over-threshold Claude CLI session in this read-only review.

## Solution Assessment

Yes: the maintainable fix is to recognize CLI runtime aliases before embedded harness selection and let the non-harness compaction path continue. Maintainers still need to choose between the linked narrow fix and the broader alternate PR shape.
