## Summary

Keep open: current main and the latest release still generate cron job IDs internally, while the current open implementation candidate at https://github.com/openclaw/openclaw/pull/75065 is conflicting and still needs maintainer/product review.

## What This Changes

Request support for an optional human-readable `--id` when creating cron jobs, while keeping generated UUIDs by default and rejecting unsafe or duplicate IDs.

## Best Possible Solution

Confirm the public cron ID contract, then land a rebased narrow implementation that adds `--id` through CLI, Gateway protocol, service creation, agent-tool behavior if desired, docs, generated clients, and focused duplicate/validation tests while preserving UUID defaults.

## Reproduction Assessment

Yes. Source inspection shows current main has no `cron add --id` option, the public `CronAddParamsSchema` rejects unknown `id`, and service creation always assigns a UUID; no write-producing test run was needed for this read-only review.

## Solution Assessment

Unclear until maintainers confirm the public ID contract. If accepted, the narrow maintainable solution is an end-to-end create-path update with validation, duplicate rejection, generated-client/docs alignment, and default UUID behavior preserved.
