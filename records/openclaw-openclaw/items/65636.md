## Summary

Keep open. Current main still lacks a supported `cron add --id` path, and an open implementation PR with closing syntax still targets this issue; overlapping PR work should be reviewed or closed before this issue is resolved.

## What This Changes

The issue requests optional human-readable cron job IDs via `openclaw cron add --id`, with UUIDs preserved by default, safe validation, and duplicate rejection.

## Best Possible Solution

Review and repair one active implementation PR, such as https://github.com/openclaw/openclaw/pull/66023 or https://github.com/openclaw/openclaw/pull/75065, so a validated unique create-time ID flows through CLI, Gateway schema, service creation, clients, docs, and changelog while UUID generation remains the default.

## Reproduction Assessment

Yes. Source inspection on current main shows no `--id` option, a Gateway add schema that rejects top-level `id`, a create type that omits `id`, and a service create path that always calls `crypto.randomUUID()`.

## Solution Assessment

Yes, the requested direction is a narrow maintainable feature if implemented end-to-end with conservative ID validation and duplicate rejection. The best path is to review or repair the existing PR work instead of closing this issue early.
