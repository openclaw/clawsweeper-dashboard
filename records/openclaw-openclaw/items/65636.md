## Summary

Keep open. Current main and the latest tagged release still lack a supported create-time custom cron job ID path, and the overlapping open implementation is not a clean/proof-positive landing path yet.

## What This Changes

Request support for an optional create-time cron job ID, such as `openclaw cron add --id daily-brief`, while preserving UUID generation when omitted.

## Best Possible Solution

Land one canonical rebased implementation that threads an optional validated ID through CLI, Gateway protocol, cron service/types, docs/generated clients as needed, and focused tests while keeping UUIDs as the default.

## Reproduction Assessment

Yes for the current behavior claim: source inspection shows no `cron add --id` option, no cron.add protocol `id` field, `CronJobCreate` omits `id`, and service creation still assigns a UUID. I did not run live CLI proof because this review is read-only.

## Solution Assessment

Yes on the product direction, but the landing path is not ready yet. The maintainable fix is an end-to-end optional ID through CLI, Gateway protocol, and cron service; editing `jobs.json` directly should remain unsupported.
