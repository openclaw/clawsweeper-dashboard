## Summary

Keep open: current main and the latest release still generate cron job IDs internally and do not expose a supported custom `cron add --id` create path. The issue is now paired with an open implementation PR, but that PR still needs maintainer/product review rather than closing this request early.

## What This Changes

Request support for an optional custom human-readable cron job ID when creating jobs with `openclaw cron add --id`.

## Best Possible Solution

Land a maintainer-approved current-main implementation that threads an optional safe custom id through CLI, Gateway protocol, service creation, generated clients/docs, and relevant agent-tool surfaces while preserving UUID defaults.

## Reproduction Assessment

Yes, source-level. Current main has no cron add `--id` flag, `CronJobCreate` omits `id`, the protocol schema rejects extra create fields, and `createJob` always generates a UUID.

## Solution Assessment

Unclear until maintainers choose the public ID contract. The requested direction is bounded and plausible, but the best implementation needs a current-main port plus explicit validation, collision, generated-client, and agent-tool decisions.
