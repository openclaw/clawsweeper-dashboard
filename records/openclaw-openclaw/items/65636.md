## Summary

Keep open. Current main still lacks the requested `cron add --id` creation path, and open PR #66023 explicitly targets this issue with closing syntax, so the issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

Add an optional `--id` flag to `openclaw cron add` so users can create path-safe, human-readable cron job IDs instead of always receiving UUIDs.

## Best Possible Solution

Review and either land or close PR #66023. The desired end state is an end-to-end, backward-compatible cron create contract that accepts an optional validated unique ID through the CLI, protocol schema, normalization, service create path, docs, and focused tests while preserving UUID generation when omitted.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current main has no `cron add --id` flag, `CronAddParamsSchema` rejects an `id` field, `CronJobCreate` omits `id`, and `createJob` always calls `crypto.randomUUID()`.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if implemented end-to-end with conservative ID validation and duplicate rejection; because PR #66023 is already the active linked implementation candidate, the best next step is review/land or close that PR rather than close this issue early.
