## What This Changes

The issue requests `openclaw cron add --id <id>` for optional, human-readable cron job IDs with validation and duplicate rejection.

## Best Possible Solution

Land or close one canonical implementation PR that adds a validated optional ID through CLI, Gateway schema, service creation, generated clients, docs, and focused tests while keeping UUID generation as the default.

## Reproduction Assessment

Yes. Source inspection on current main shows no CLI `--id` option, no Gateway `cron.add` `id` field, `CronJobCreate` still omits `id`, and service creation still generates UUIDs unconditionally.

## Solution Assessment

Yes, the requested direction is a narrow maintainable feature if implemented end-to-end with conservative validation and duplicate rejection. The best path is to review or repair the existing PR work rather than close the issue early or queue a separate repair lane.
