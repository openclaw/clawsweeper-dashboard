## Summary

Keep open: the PR targets a real shipped default-prewarm regression and now has live startup proof, but the current merge result drops auth-failure invalidation when default prewarm is disabled and leaves the new test using a stale mock name.

## What This Changes

The PR changes gateway post-attach startup to schedule provider-auth prewarming only when `providerAuthPrewarm.enabled` is true and adds a regression test for the omitted-enabled path.

## Best Possible Solution

Keep startup prewarming opt-in, but register auth-failure invalidation independently of the expensive warm sidecar and cover both default-disabled startup and stale-state clearing in tests.

## Reproduction Assessment

No full live CPU reproduction was run here, but the default scheduling path is source-reproducible: current main and `v2026.5.27` pass `providerAuthPrewarm` without `enabled`, and the guard treats that as enabled. The contributor added after-fix live startup proof for the PR head.

## Solution Assessment

No, not yet. Disabling startup warm by default is plausible, but the maintainable fix needs to separate auth-failure invalidation from warm scheduling and update the regression test for the current off-main-thread warm mock.
