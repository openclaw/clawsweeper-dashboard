## Summary

Keep this PR open: current main and the latest release still run provider-auth startup prewarm by default, and the diff is a narrow, useful fix with no blocking code findings. The maintainer-authored sibling has stronger Windows/WSL proof, but it is currently draft and conflicting, so it is not a safe superseding landing target for cleanup closure.

## What This Changes

The PR makes gateway provider-auth startup warming explicit opt-in while keeping the auth-profile failure rewarm hook installed, and adds a regression test for the hook-only default path.

## Best Possible Solution

Land one rebased canonical hook-only startup implementation after maintainer sign-off on the lazy provider-auth default, while keeping the broader Windows/WSL/Telegram performance cluster tracked separately.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source for the central scheduler bug: current main's normal gateway caller supplies `providerAuthPrewarm` without `enabled`, and post-attach startup still schedules warm work for that omitted value.

## Solution Assessment

Is this the best way to solve the issue? Yes, conditionally: the patch targets the implicated startup scheduler, preserves explicit startup warming and auth-failure rewarm, and adds focused coverage, but maintainer sign-off is needed for the shipped-default tradeoff.
