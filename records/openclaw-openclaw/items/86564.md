## Summary

Keep open: current main and v2026.6.6 still run the normal gateway provider-auth startup warm by default, and this PR contains a focused fix with no concrete correctness finding. It still needs maintainer handling because the branch is conflicting and the same central change is also present in the open maintainer draft at https://github.com/openclaw/openclaw/pull/88514 with stronger platform proof.

## What This Changes

The PR makes gateway provider-auth startup warming explicit opt-in while keeping auth-profile failure rewarm installed and adding regression coverage for the hook-only default path.

## Best Possible Solution

Land one current-main-compatible, maintainer-approved hook-only startup implementation that skips unsolicited provider-auth startup warm work while preserving explicit opt-in warming and auth-profile failure rewarm.

## Reproduction Assessment

Yes by source plus linked operational reports, not by a fresh live rerun: current main's normal gateway caller supplies providerAuthPrewarm without enabled and the post-attach runtime schedules the startup timer. I did not reproduce the exact CPU/WSL2 latency symptom during this read-only review.

## Solution Assessment

Yes, if maintainers accept the compatibility tradeoff: the patch targets the implicated startup scheduler, keeps explicit startup warming and failure rewarm, and adds focused coverage. The safer landing path is to choose one rebased canonical branch rather than carrying both this PR and https://github.com/openclaw/openclaw/pull/88514.
