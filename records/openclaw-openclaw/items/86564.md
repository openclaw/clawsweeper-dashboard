## Summary

Keep open: current main and the latest release still run the provider-auth startup warm by default, and this branch contains a meaningful focused fix. ## What This Changes

The PR makes gateway provider-auth startup warming explicit opt-in while keeping the auth-profile failure rewarm sidecar and adding focused regression coverage.

## Best Possible Solution

Consolidate on one mergeable hook-only/default-off provider-auth startup branch, preserve explicit startup warming and failure-hook rewarm coverage, and track broader startup/event-loop symptoms separately.

Yes for the central startup-scheduling defect by source inspection: current main's normal gateway caller supplies `providerAuthPrewarm` without `enabled`, and the post-attach sidecar treats omitted `enabled` as startup warm enabled. ## Solution Assessment
