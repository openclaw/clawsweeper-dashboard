## Summary

Keep open: the PR targets a real gateway availability regression, but the current patch makes provider-auth prewarm opt-in for normal startup, drops the auth-failure invalidation hook with it, and lacks real gateway proof before merge.

## What This Changes

The PR changes gateway post-attach startup so provider-auth prewarm runs only when `providerAuthPrewarm.enabled` is explicitly true and adds a regression test for the disabled-by-default path.

## Best Possible Solution

Preserve provider-auth freshness and the model-listing fast path while giving maintainers a deliberate policy: either keep prewarm enabled by default with a documented operator opt-out, or move warm work off the gateway event loop with real startup and `/models` proof.

## Reproduction Assessment

Yes at source level: current gateway startup passes no `enabled` flag, so the PR guard skips the sidecar and its auth-failure hook for normal boots. I did not run a live macOS CPU reproduction in this read-only review.

## Solution Assessment

No: the patch is a plausible mitigation direction, but the current shape flips the default without preserving the auth-failure hook or proving the model-listing/startup trade-off. A safer solution is an explicit operator policy or off-main-thread warm path with real gateway proof.

- **[P1] Keep auth-failure invalidation when skipping startup warm:** `src/gateway/server-startup-post-attach.ts:975`
  - body: Changing this guard to `enabled === true` means the normal gateway boot from `src/gateway/server.impl.ts` no longer calls `scheduleProviderAuthStatePrewarm`, because that caller passes only `{ getConfig }`. Please split the failure-hook lifecycle from the startup warm or gate all direct rewarm paths under one policy before flipping the default.
