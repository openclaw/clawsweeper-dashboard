## Summary

Keep this PR open but blocked: the reported gateway CPU regression is real enough to pursue and the contributor added acceptable live startup proof, but the patch currently drops provider-auth failure invalidation on the default startup path and has a stale test assertion on current main.

## What This Changes

The PR changes gateway post-attach startup so provider-auth prewarming only schedules when `providerAuthPrewarm.enabled` is true and adds a regression test for the omitted-enabled path.

## Best Possible Solution

Keep startup warming opt-in if maintainers accept the default change, but register auth-failure invalidation independently of expensive warm scheduling and cover both default-disabled startup and stale-state clearing in tests.

## Reproduction Assessment

No full live CPU reproduction was run here, but the default scheduling path is source-reproducible: current main and `v2026.5.27` pass `providerAuthPrewarm` without `enabled`, and the guard treats that as enabled. The contributor supplied after-fix live startup proof for the PR head.

## Solution Assessment

No, not yet. Disabling startup warm by default is plausible, but the maintainable fix needs to separate auth-failure invalidation from warm scheduling and update the regression test for the current off-main-thread warm mock.
