## Summary

Keep open: the underlying Telegram thread-bound subagent spawn problem is still plausible on current main, but this PR is not ready to merge because it targets pre-refactor entrypoints, has a concrete Telegram target normalization bug, and lacks after-fix real Telegram proof.

## What This Changes

The PR adds Telegram subagent lifecycle hooks, session-resume routing hydration, shared session-binding state, and tests for topic-bound `sessions_spawn(..., thread=true, mode="session")`.

## Best Possible Solution

Rework the fix on current main as a lazy Telegram `subagent-hooks-api.ts` registered from `registerFull`, use the existing SDK conversation-runtime and Telegram target parsing surfaces, add focused regression tests, and provide redacted live Telegram proof.

## Reproduction Assessment

Yes, at source level: current main still has no Telegram `subagent_spawning` registration, and the subagent spawn path errors when that hook is absent. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The proposed direction is useful, but this exact patch is not the best solution because it bypasses the current bundled-channel hook pattern, deep-imports old core paths, and mishandles supported Telegram target prefixes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Normalize Telegram route prefixes before binding:** `extensions/telegram/src/subagent-hooks.ts:55-59`
  - body: This resolver strips only `channel:` and `telegram:` before storing the conversation ID. Supported targets such as `telegram:group:-100123:topic:55` and `tg:group:-100123` are left as `group:-100123:topic:55`, which later becomes `channel:group:-100123` and will not resolve to the real Telegram chat. Reuse the current Telegram target parser/normalizer before binding.
  - confidence: 0.88
- **[P2] Use the current bundled-channel hook boundary:** `extensions/telegram/src/subagent-hooks.ts:7-9`
  - body: The new Telegram hook module imports core internals through `../../../src/...` and is wired into the old direct `register()` shape. Current bundled channels register subagent hooks from `registerFull` through a lazy `*-hooks-api.ts` barrel and SDK exports, so this needs to be rebased to that boundary before it can work on current main.
  - confidence: 0.83
- **[P3] Initialize the lazy global service before asserting it:** `src/infra/outbound/session-binding-service.test.ts:219-223`
  - body: The added test reads `globalThis.__openclawSessionBindingService` before calling `getSessionBindingService()`, but the PR initializes that global lazily in the getter. If this test runs first or in isolation, `globalService` is `undefined` and the final assertion fails.
  - confidence: 0.86
