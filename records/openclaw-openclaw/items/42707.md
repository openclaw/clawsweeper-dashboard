## Summary

Keep open. Current main still lacks Telegram native subagent hook wiring for thread-bound session spawns, so the user-visible request is not implemented; the source PR is useful but stale and incorrect against the current bundled-channel entrypoint and extension-boundary rules, with #42986 still open as the matching tracker.

## What This Changes

The PR adds Telegram subagent lifecycle hook registration and handlers, session-resume routing hydration, global session-binding singleton changes, a Telegram thread-binding prewarm helper, and targeted tests.

## Best Possible Solution

Deliver a refreshed Telegram-local implementation that mirrors the current Discord registerFull lazy hook bridge, uses SDK conversation/thread-binding facades plus Telegram-local target parsing, hydrates resumed agent runs from persisted delivery context, and updates native subagent docs/tests once support is actually enabled.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a Telegram requester calling sessions_spawn with thread=true and mode="session" has no Telegram subagent_spawning hook, so src/agents/subagent-spawn.ts returns the hook-unavailable path before binding can be prepared.

## Solution Assessment

No. The PR's direction is useful, but the patch is not the best current fix because it imports core internals from a plugin, is stale against the current bundled entry layout, and mishandles supported Telegram target prefixes; a refreshed boundary-compliant replacement is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use SDK seams for Telegram hook wiring:** `extensions/telegram/src/subagent-hooks.ts:7-9`
  - body: The new Telegram production hook module imports host internals through ../../../src, including a stale ../../../src/telegram/thread-bindings.js path that is not present on current main. Bundled plugins are expected to cross this boundary through plugin SDK exports or Telegram-local barrels, so this should mirror Discord's registerFull bridge and use openclaw/plugin-sdk/conversation-runtime or narrow local APIs.
  - confidence: 0.9
- **[P2] Normalize Telegram targets before binding:** `extensions/telegram/src/subagent-hooks.ts:55-59`
  - body: resolveTelegramConversationId only strips channel: and telegram:, so valid targets such as telegram:group:-100123:topic:55 or tg:-100123:topic:55 become malformed conversation IDs. Those malformed IDs later produce delivery targets the Telegram send path cannot resolve.
  - confidence: 0.88
- **[P3] Initialize the global service before asserting it:** `src/infra/outbound/session-binding-service.test.ts:215-223`
  - body: This test reads globalThis.__openclawSessionBindingService before getSessionBindingService() has populated the lazy global. If the test runs in isolation, globalService is undefined and the final assertion fails.
  - confidence: 0.86
