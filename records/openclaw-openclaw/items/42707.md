## Summary

Keep open: current main still has the Telegram hook gap, but this PR cannot merge as-is because it violates extension boundaries, mishandles supported Telegram target forms, and has no real Telegram proof.

## What This Changes

The PR adds Telegram subagent lifecycle hook registration, hook handlers and tests, session-resume routing hydration, shared session-binding state, and a Telegram binding-manager prewarm helper.

## Best Possible Solution

Refresh this as a current-main Telegram `registerFull` hook implementation using plugin-sdk/local barrels and existing Telegram target parsing, then verify with focused tests and redacted real Telegram proof while keeping https://github.com/openclaw/openclaw/issues/42986 as the tracker.

## Reproduction Assessment

Yes, at source level: current main rejects `thread:true` native subagent spawns unless a channel registers `subagent_spawning`, and the Telegram entry still does not register that hook. I did not run live Telegram or tests because this review is read-only.

## Solution Assessment

No. The direction is useful, but this PR is not the best current implementation because it reaches into core internals from extension code, bypasses existing Telegram target parsing, and targets stale pre-refactor paths.
