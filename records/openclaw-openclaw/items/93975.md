## Summary

Keep open: the PR is aimed at the right plugin registry failure mode and includes real Windows proof for newly generated marker rows, but it misses the upgrade case that created the reported stale SQLite diagnostics because current and released rows were written without `code`.

## What This Changes

The PR adds an `orphan-source-path` plugin diagnostic code, persists diagnostic `code` through the installed-plugin SQLite schema, and changes registry snapshot stale-diagnostic detection and tests to rederive on that marker.

## Best Possible Solution

Keep the explicit marker for new diagnostics, but add a narrow legacy persisted-row path or migration for no-pluginId `plugin path not found` diagnostics with a missing absolute source, backed by a SQLite round-trip regression test.

## Reproduction Assessment

Yes, source-reproducible without executing: current main writes no `code` on `plugin path not found`, the persisted schema has no `code`, and stale detection only catches diagnostics with `pluginId`. A no-pluginId persisted row can therefore still be reused.

## Solution Assessment

No. The explicit marker is a good future-proof shape, but the best fix must also handle already persisted no-code rows from current/shipped OpenClaw so the reported upgrade failure is actually repaired.
