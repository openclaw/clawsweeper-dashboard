## What This Changes

The issue reports that WhatsApp replies send text but drop images produced by the read tool, leaving outbound logs with hasMedia false.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81415, or an equivalent shared read-tool media bridge, so read image bytes are persisted into inbound media and delivered through existing outbound media URLs.

## Reproduction Assessment

Yes. From source inspection, a read result with image content but no MEDIA token and no details.path produces no media artifact, so the pending-media queue is not populated and WhatsApp receives no primary media URL.

## Solution Assessment

Yes. The linked PR’s shared read-tool media bridge is the narrow maintainable fix because it feeds the existing outbound media URL path instead of adding a WhatsApp-only workaround.
