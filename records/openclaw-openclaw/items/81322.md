## Summary

Keep open: current main still has the source-visible read-tool image media gap, and https://github.com/openclaw/openclaw/pull/81415 is an open closing PR for the same bug. This issue should close after that PR, or an equivalent shared media bridge, lands.

## What This Changes

The issue reports that WhatsApp outbound replies lose images read by the `read` tool, sending only text and logging `hasMedia: false` despite image processing.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81415, or an equivalent shared read-tool image bridge, so base64 image blocks become trusted media URLs for outbound delivery.

## Reproduction Assessment

Yes. Source inspection on current main shows a read result with image content but no `MEDIA:` token or `details.path` produces no media artifact, so the pending-media queue stays empty and WhatsApp receives no primary media URL.

## Solution Assessment

Yes. The linked PR’s shared read-tool media bridge is the best current direction because it feeds the existing outbound media URL path instead of adding a WhatsApp-specific workaround.
