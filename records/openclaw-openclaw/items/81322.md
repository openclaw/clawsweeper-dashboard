## Summary

Keep open: current main still drops read-tool image blocks that have no `MEDIA:` token or `details.path`, and the issue is paired with an open closing PR that targets that shared media bridge.

## What This Changes

Fix WhatsApp replies so images read by the `read` tool are attached to outbound messages instead of producing text-only sends with `hasMedia: false`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81415, or an equivalent shared read-tool media bridge, so image bytes become media URLs before outbound channel delivery.

## Reproduction Assessment

Yes, at source level: a `read` tool result with an image block but no `MEDIA:` token or `details.path` currently produces no media URL, so WhatsApp later sends text only. I did not run a live WhatsApp device/provider-chain reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is to persist trusted read-tool image bytes into the shared media queue before channel delivery, which is exactly the boundary targeted by https://github.com/openclaw/openclaw/pull/81415.
