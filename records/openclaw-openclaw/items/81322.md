## Summary

Keep open: current main still drops read-tool image content when it has no `MEDIA:` token or `details.path`, and the open closing PR is the active implementation candidate for the same bug.

## What This Changes

The issue reports that WhatsApp replies lose image attachments produced by the read-tool path, leaving outbound sends text-only with `hasMedia: false`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81415, or an equivalent shared media bridge, so read-tool image blocks become trusted inbound media URLs for outbound delivery.

## Reproduction Assessment

Yes. Source inspection shows a read result with image content but no `MEDIA:` token or `details.path` produces no media artifact, so the pending-media queue stays empty and WhatsApp receives no primary media URL.

## Solution Assessment

Yes. The linked PR’s shared read-tool media bridge is the best current direction because it feeds the existing outbound media URL path instead of adding a WhatsApp-specific workaround.
