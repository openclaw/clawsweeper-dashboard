## Summary

Keep open: current main still has the source-visible read-tool image media gap, and the issue is paired with the open closing pull request https://github.com/openclaw/openclaw/pull/81415. This should close only after that PR, or an equivalent shared media bridge, lands.

## What This Changes

The issue reports that WhatsApp outbound replies resize read-tool image content but send text-only messages because no media URL reaches the shared outbound path.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81415, or an equivalent shared read-tool media bridge, so read image bytes are persisted into inbound media and delivered through existing outbound media URLs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path. A read result with image content but no `MEDIA:` token and no `details.path` produces no media artifact, so the pending-media queue is not populated and WhatsApp receives no primary media URL.

## Solution Assessment

Yes. The paired PR's shared read-tool media bridge is the narrow maintainable fix because it feeds the existing outbound media URL path instead of adding a WhatsApp-only workaround.
