## Summary

Keep open: current main still has the source-visible read-tool image gap, and the open closing PR that targets this bug has not merged.

## What This Changes

The issue reports that WhatsApp replies drop images produced by the read tool: image resizing succeeds, but outbound WhatsApp logging shows no attached media.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81415, or an equivalent shared read-tool media bridge, so read image bytes are persisted into inbound media and delivered through existing outbound media URLs.

## Reproduction Assessment

Yes. Source inspection against origin/main shows a read result with image content but no MEDIA token and no details.path produces no media artifact, so the pending-media queue is not populated and WhatsApp receives no primary media URL.

## Solution Assessment

Yes. The linked PR’s shared read-tool media bridge is the narrow maintainable direction because it feeds the existing outbound media URL path instead of adding a WhatsApp-only workaround.
