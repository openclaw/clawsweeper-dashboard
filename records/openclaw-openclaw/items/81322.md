## Summary

Keep open: current main still has the read-tool image media promotion gap, and the direct closing PR for this WhatsApp report is open rather than merged. This should close after that PR, or an equivalent shared media bridge, lands.

## What This Changes

The issue reports that WhatsApp replies send text only after a read-tool image is processed, because the processed image is not attached to the outbound message.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81415, or fold the same inbound-media bridge into the broader read-image media fix, so pathless read image blocks become trusted media URLs before outbound delivery.

## Reproduction Assessment

Yes. Source inspection on current main shows a read result with image content but no `MEDIA:` token or `details.path` produces no media artifact, so the pending-media queue stays empty and WhatsApp receives no primary media URL.

## Solution Assessment

Yes. The linked PR’s shared read-tool image bridge is the best current direction because it feeds the existing media URL delivery path instead of adding a WhatsApp-specific workaround.
