## Summary

Keep open: current main still shows the reported bridge gap. WhatsApp sends media when a media URL reaches the outbound path, but read-tool image results with only base64 image content still produce no attachment URL for the pending tool-media queue.

## What This Changes

The issue reports that WhatsApp replies send text only after an image read-tool result is resized, with outbound logs showing `hasMedia:false`.

## Best Possible Solution

Bridge trusted read-tool image outputs into the existing pending tool-media/outbound media pipeline with scoped local-media handling and regression coverage, without adding WhatsApp-specific sender logic.

## Reproduction Assessment

No; I did not run a live WhatsApp reproduction. Source inspection provides a clear reproduction path: a read-image result with base64 image content but no `MEDIA:` marker or `details.path` yields no queued media URL, so WhatsApp sends text with `hasMedia:false`.

## Solution Assessment

Yes; the narrow maintainable fix is in the shared read-tool media bridge. The WhatsApp sender already handles images correctly once a media URL reaches `sendMessageWhatsApp`.
