## What This Changes

The issue reports WhatsApp read-tool image replies being processed but sent as text-only outbound messages with `hasMedia: false`.

## Best Possible Solution

Land a rebased version of https://github.com/openclaw/openclaw/pull/81415, or an equivalent current-file fix, that persists trusted read-tool image blocks into inbound media and queues `media://inbound` URLs with focused agent and WhatsApp outbound tests.

## Reproduction Assessment

Yes, at source level: current main returns inline read-tool image content without a path source, and the extractor explicitly returns no media when image content lacks `details.media` or `details.path`. I did not run a live WhatsApp device reproduction in this read-only review.

## Solution Assessment

Yes in direction: the maintainable fix belongs in the agent read-tool media bridge, not the WhatsApp send loop, because WhatsApp already sends media when `mediaUrls` are present. The open PR still needs rebase or replacement before it is the best landing vehicle.
