## What This Changes

Read-tool image results sent through WhatsApp should attach processed images instead of producing text-only replies with `hasMedia: false`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81415, or an equivalent shared read-tool media bridge, so read image bytes are persisted and delivered through the existing outbound media pipeline.

## Reproduction Assessment

No live end-to-end reproduction was run. Source inspection and existing tests show the current-main failure condition: read image content without `MEDIA:` or `details.path` yields no media artifact, so WhatsApp receives no media URL.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared read-tool media bridge, and the linked PR targets that boundary rather than adding WhatsApp-specific workaround logic.
