## Summary

Keep open: current main still has a source-visible gap between `read` tool image content and queued outbound media, and the linked open fixing PR uses closing syntax for this issue. Repository policy says the issue should stay open until that PR merges or is replaced by an equivalent fix.

## What This Changes

The issue reports that WhatsApp replies send text but drop images after the `read` tool processes an image, leaving outbound `hasMedia:false`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81415, or an equivalent shared agent media bridge that persists trusted read-tool image content and queues it for outbound replies, then close this issue after merge.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection explains the failure path: base64 `read` image output without `details.path` produces no media artifact, so no pending media URL reaches WhatsApp.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared read-tool media bridge, not in WhatsApp-specific send code, because WhatsApp already attaches media when a media URL reaches its sender.
