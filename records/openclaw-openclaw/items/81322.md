## Summary

Keep open: current main still has the source-visible read-tool image media gap, and https://github.com/openclaw/openclaw/pull/81415 is an open closing PR for the same bug. This issue should close after that PR, or an equivalent shared media bridge, lands.

## What This Changes

WhatsApp image replies generated from read-tool image output should attach and deliver media instead of sending text-only messages with `hasMedia:false`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81415, or an equivalent shared bridge, so read image blocks become trusted inbound media URLs for outbound delivery.

## Reproduction Assessment

Yes. Source inspection shows a read result with image content but no `MEDIA:` token or `details.path` produces no media artifact, so the pending-media queue stays empty and WhatsApp receives no primary media URL.

## Solution Assessment

Yes. The linked PR’s shared read-tool media bridge is the best current direction because it feeds the existing outbound media URL path instead of adding a WhatsApp-specific workaround.
