## Summary

Keep open. Current main and the latest release still expose explicit `--session-id` but no `--new-session` or generated-session JSON contract; the linked draft PR was closed unmerged, and the broader resume issue is related but not a complete replacement for this narrower request.

## What This Changes

The issue requests an `openclaw agent --new-session` flag that generates a fresh session id, uses it for the invocation, and returns it to JSON callers.

## Best Possible Solution

If maintainers want this CLI UX, add a backward-compatible flag that generates one UUID before dispatch, rejects conflicting explicit session ids, returns the generated id to callers, updates docs, and coordinates the behavior with https://github.com/openclaw/openclaw/issues/71417.

Not applicable as a runtime bug reproduction because this is a feature request. Source, docs, release, and related-PR inspection give a high-confidence way to verify that the requested flag is still absent.

## Solution Assessment

Unclear pending maintainer UX direction. A CLI-owned generated `sessionId` routed through existing session-id handling is narrow, but the flag name, aliases, resume disclosure, and JSON contract should be decided with the broader session-resume issue.
