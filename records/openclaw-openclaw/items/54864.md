## Summary

Keep open. Current main has explicit `--session-id` isolation improvements, but the requested `openclaw agent --new-session` flag, typed option plumbing, generated session id handling, and documented JSON return contract are still absent. The related #69108 proposed this exact feature but is closed and unmerged in the provided GitHub context.

## What This Changes

Request add `--new-session` to `openclaw agent` so the CLI generates a fresh isolated session id for one invocation and returns that id in JSON output.

## Best Possible Solution

Implement a narrow core CLI feature: register `--new-session`, reject it with `--session-id`, generate one `crypto.randomUUID()` in `agentCliCommand`, pass the effective `sessionId` through gateway and embedded paths, return the generated id in JSON output, preserve default agent-main-session reuse when the flag is absent, and document the scripting workflow.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. Current-main inspection is high-confidence: the CLI parser, command option type, JSON envelope, docs, and tests do not contain `--new-session` or `newSession` plumbing.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix because `openclaw agent` already owns session selection and JSON output for scripted one-shot runs. The safer implementation boundary is to add only a CLI-owned generated effective `sessionId`, not a broader session-store migration or startup-time behavior change.
