## Summary

Keep open. Current main has explicit `--session-id` scoping improvements, but the requested `openclaw agent --new-session` flag, typed option plumbing, CLI-owned generated session id, JSON return contract, and docs/tests are still absent. The closest prior implementation, #69108, is closed and unmerged in the provided GitHub context.

## What This Changes

Request adding `openclaw agent --new-session` so scripted callers can ask the CLI to generate a fresh session id and receive it back in JSON output for later reconnects.

## Best Possible Solution

Implement the feature as a narrow CLI-owned session selection addition: add `--new-session`, reject it with `--session-id`, generate one effective UUID before gateway/local dispatch, preserve current default main-session reuse, expose the generated id in JSON output, and update focused tests, docs, and changelog.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable solution because `openclaw agent` already owns session selection and structured output for scripted one-shot runs. The safer boundary is CLI pre-dispatch generation plus existing `sessionId` plumbing, not a broader session-store migration.
