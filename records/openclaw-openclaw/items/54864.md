## Summary

Keep open. Current main and v2026.5.20 still expose only explicit `--session-id`; https://github.com/openclaw/openclaw/pull/69108 was closed unmerged after the session-id isolation portion shipped, so the generated `--new-session` flag and JSON contract remain a product/UX feature request.

## What This Changes

The issue requests a native `openclaw agent --new-session` flag that generates an isolated session ID and reports it in JSON output.

## Best Possible Solution

If maintainers want this CLI UX, add a backward-compatible flag that generates one UUID before dispatch, flows through local/Gateway/fallback paths, returns the generated id in JSON, documents the reconnect path, and coordinates semantics with https://github.com/openclaw/openclaw/issues/71417.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. Source, docs, protocol, release, and related-PR inspection give a high-confidence way to verify that the requested flag is still absent.

## Solution Assessment

Unclear pending maintainer UX direction. A CLI-owned generated `sessionId` routed through existing `sessionId` handling is narrow, but the flag name, aliases, resume disclosure, and JSON contract should be decided with the broader session-resume issue.
