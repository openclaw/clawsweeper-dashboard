## Summary

Keep open: current main and the latest release still reject Google Chat Add-on lifecycle-only payloads, and this PR is a plausible plugin-local fix, but it still lacks inspectable real behavior proof and focused lifecycle regression tests.

## What This Changes

The PR changes `extensions/googlechat/src/monitor-webhook.ts` to normalize Google Workspace Add-on `messagePayload`, `addedToSpacePayload`, and `removedFromSpacePayload` envelopes into Google Chat event types.

## Best Possible Solution

Land a plugin-local parser fix with lifecycle regression tests and redacted real Google Chat proof, while keeping broader Google Chat parser/policy/threading work tracked under https://github.com/openclaw/openclaw/issues/65007.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.19` only unwrap Add-on `messagePayload`, so an envelope containing only `chat.addedToSpacePayload` or `chat.removedFromSpacePayload` lacks a root event type and reaches the HTTP 400 `invalid payload` branch.

## Solution Assessment

Yes for the narrow lifecycle parser gap: plugin-local normalization is the right boundary. The merge-ready path still needs focused tests, redacted real Google Chat proof, and a maintainer decision on whether to consolidate broader Add-on union handling with the open Google Chat cluster.
