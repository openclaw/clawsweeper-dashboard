## Summary

Keep open. The duplicate iMessage watcher bug is still present on current main, and this PR remains incorrect because it filters `listEnabledIMessageAccounts()` while gateway startup still enumerates raw iMessage account IDs; it also has disabled-account and normalized-default edge regressions.

## What This Changes

The PR adds fallback-default suppression to `listEnabledIMessageAccounts()` and extends iMessage account tests for fallback-only, distinct-runtime, and inherited-processing default configurations.

## Best Possible Solution

Land one iMessage-owned fix that dedupes only fallback-only `default` watchers on the actual startup path, after resolving enabled accounts and normalized default keys, while preserving distinct backend runtimes and covering the reported config with focused regression tests.

## Reproduction Assessment

Yes. The linked issue provides a concrete config, and current main deterministically enumerates both `default` and the named account through `plugin.config.listAccountIds(cfg)` before starting each account.

## Solution Assessment

No. The proposed direction is narrow, but this implementation is not the best fix because it changes a helper startup does not call and computes default suppression before enabled-state and normalized-key handling are correct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Wire the filtered account list into startup:** `extensions/imessage/src/accounts.ts:80`
  - body: `listEnabledIMessageAccounts()` now contains the fallback-default filtering, but gateway startup still calls `plugin.config.listAccountIds(cfg)`, and the iMessage adapter still wires that to `listIMessageAccountIds`. The reported config will still start both `default` and the named account unless the startup enumeration path uses the deduped list or equivalent `isEnabled` logic.
  - confidence: 0.94
- **[P1] Ignore disabled named accounts before dropping default:** `extensions/imessage/src/accounts.ts:82-85`
  - body: `hasNamedAccounts` is computed from every configured ID before resolving whether those accounts are enabled. A disabled named account plus an enabled fallback `default` would drop `default`, then the later enabled filter drops the named account too, leaving no iMessage monitor running.
  - confidence: 0.9
- **[P2] Resolve default through normalized account lookup:** `extensions/imessage/src/accounts.ts:68`
  - body: `hasDistinctDefaultIMessageRuntime` reads `accounts.default` directly, while account resolution accepts normalized and case-insensitive keys. A config using `Default` with a distinct runtime can be misclassified as fallback-only and dropped when another named account exists.
  - confidence: 0.86
