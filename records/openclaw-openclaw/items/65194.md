## Summary

Keep open. This targets a real iMessage duplicate-watcher bug, but the branch is not mergeable because the new filtering is added to a helper that gateway startup does not consume, and the PR also lacks after-fix real behavior proof.

## What This Changes

The PR adds fallback-only `default` suppression to `listEnabledIMessageAccounts()` and extends iMessage account tests for fallback, distinct-runtime, and inherited-processing configurations.

## Best Possible Solution

Land one scoped iMessage startup fix that deduplicates fallback-only `default` watchers on the actual account-start path after enabled-state and normalized-key resolution, while preserving distinct `cliPath`/`dbPath` runtimes.

## Reproduction Assessment

Yes. The linked issue supplies concrete config, process-list, duplicate-message, and workaround evidence, and current main statically shows both account IDs flow from `plugin.config.listAccountIds(cfg)` into independent startup tasks.

## Solution Assessment

No. The PR direction is narrow, but this implementation filters a helper that startup does not call and computes default suppression before enabled-state and normalized default-key handling are correct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Wire the default filter into startup enumeration:** `extensions/imessage/src/accounts.ts:88-91`
  - body: `listEnabledIMessageAccounts()` gets the fallback-default filter, but gateway startup still obtains IDs from `plugin.config.listAccountIds(cfg)`, and the iMessage adapter still points that to `listIMessageAccountIds`. The reported `default` plus named-account config will still start both accounts unless the startup path consumes the deduped list or an equivalent startup-only gate.
  - confidence: 0.94
- **[P1] Filter enabled named accounts before dropping default:** `extensions/imessage/src/accounts.ts:82-86`
  - body: `hasNamedAccounts` is computed from configured IDs before resolving whether those accounts are enabled. A disabled named account plus an enabled fallback `default` would remove `default`, and the final enabled filter would then remove the named account too, leaving no iMessage monitor running.
  - confidence: 0.9
- **[P2] Resolve the default account through normalized lookup:** `extensions/imessage/src/accounts.ts:68`
  - body: `hasDistinctDefaultIMessageRuntime()` reads `accounts.default` directly even though account IDs are normalized elsewhere. A `Default` entry with its own `cliPath` or `dbPath` can be misclassified as fallback-only and dropped when another named account exists.
  - confidence: 0.86
