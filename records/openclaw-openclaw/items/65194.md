## Summary

Keep open. The underlying duplicate iMessage monitor bug is still real on current main, but this PR is not merge-ready because the new filtering is not wired into the gateway startup enumeration and it introduces enabled/default-key edge-case regressions.

## What This Changes

The PR adds fallback-only `default` suppression to `listEnabledIMessageAccounts()` and expands iMessage account tests for fallback, distinct-runtime, and inherited-processing default-account configurations.

## Best Possible Solution

Land one iMessage-owned fix that changes the account list actually used for gateway monitor startup, dedupes only fallback-only `default` watchers covered by an enabled named account on the same backend, preserves distinct default runtimes, handles normalized account keys, and includes focused regression tests for the reported config plus disabled-placeholder and distinct-runtime cases.

## Reproduction Assessment

Yes. The linked issue gives a concrete config, and current main deterministically enumerates both `default` and the named account before starting one monitor per listed account; this read-only review verified that path statically rather than running tests.

## Solution Assessment

No. The proposed direction is narrow, but this implementation is not the best fix yet because it filters a helper that startup does not call and computes default suppression before enabled-account and normalized-key behavior are fully respected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Wire the filtered account list into startup:** `extensions/imessage/src/accounts.ts:80`
  - body: `listEnabledIMessageAccounts()` now contains the fallback-`default` filtering, but gateway startup still calls `plugin.config.listAccountIds(cfg)`, and the iMessage adapter still wires that to `listIMessageAccountIds`. The reported config will therefore still start both `default` and the named account monitors unless the filtered/deduped list is used by startup.
  - confidence: 0.94
- **[P1] Filter after resolving enabled named accounts:** `extensions/imessage/src/accounts.ts:82-85`
  - body: `hasNamedAccounts` is computed from every configured account id, before resolving whether those accounts are enabled. A disabled named account plus an enabled fallback `default` would cause the filter to drop `default`, then the later enabled filter drops the disabled account too, leaving no iMessage monitor running.
  - confidence: 0.9
- **[P2] Resolve default runtime through normalized lookup:** `extensions/imessage/src/accounts.ts:68`
  - body: `hasDistinctDefaultIMessageRuntime` reads `accounts.default` directly, while account resolution accepts normalized/case-insensitive keys. A config using `Default` with a distinct runtime can be misclassified as fallback-only and the monitor can be dropped when another named account exists.
  - confidence: 0.86
