## Best Possible Solution

Either land `parallel-free` as an explicit provider first, or record maintainer approval for the zero-config default switch and align MCP-specific schema/session handling before merge.

## Solution Assessment

No, not as-is. The provider implementation is plausible, but replacing the implicit keyless default needs an explicit maintainer product decision and the free MCP path should not reuse the paid REST session contract unchanged.

  - body: With `requiresCredential: false` and `autoDetectOrder: 76`, this provider becomes the first keyless fallback because the runtime defers keyless providers until credentialed candidates fail and then returns the first one by auto-detect order.   - confidence: 0.89

Please give `parallel-free` its own schema/normalizer, or cap only the MCP path, while leaving the paid REST contract intact.
