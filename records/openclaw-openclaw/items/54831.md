## What This Changes

The PR adds hooks.enabled/hooks.token cross-field validation during shared config validation and updates the hooks runtime error text.

## Best Possible Solution

Land a corrected version that rejects merged config writes with hooks.enabled=true and no non-empty hooks.token, keeps resolveHooksConfig as the runtime backstop, uses supported CLI/RPC remediation syntax, and adds focused regression coverage.

## Reproduction Assessment

Yes. By source inspection, current main allows hooks.enabled and hooks.token to be optional, config.patch validates the merged config through validateConfigObjectWithPlugins, and resolveHooksConfig is the first place that requires a non-empty token.

## Solution Assessment

No, not as written. Cross-field validation plus a runtime fallback is the narrow maintainable fix, but the PR's remediation text uses unsupported `openclaw config.patch` syntax and still needs real behavior proof.
