## Summary

Keep open: current main still lacks write-time hooks.enabled/hooks.token validation, so this PR remains the canonical implementation candidate, but it is not merge-ready because the remediation text uses an unsupported command, encourages inline token handling, lacks real behavior proof, and is currently conflicting.

## What This Changes

The PR adds hooks.enabled/hooks.token cross-field validation during shared config validation and expands the hooks startup error with remediation text.

## Best Possible Solution

Refresh and land a tested shared validation rule that rejects hooks.enabled=true without a non-empty hooks.token during config writes, keeps resolveHooksConfig as the runtime backstop, and uses supported redacted `config patch --stdin`/`--file` or RPC guidance.

## Reproduction Assessment

Yes. Source inspection on current main shows config.patch validates the merged config before writing, while the schema accepts hooks.enabled without hooks.token and resolveHooksConfig only rejects the missing token later during runtime setup.

## Solution Assessment

No, not as written. Shared cross-field validation plus a runtime fallback is the right narrow solution, but this PR must replace the unsupported remediation command, avoid inline secret guidance, add focused tests, refresh conflicts, and provide real behavior proof.
