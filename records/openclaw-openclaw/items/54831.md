## Summary

Keep open: the underlying hooks token validation bug is still source-reproducible on current main and this PR targets the right boundary, but it is merge-conflicting, lacks real behavior proof, and adds unsupported/unsafe token remediation text that should block merge.

## What This Changes

The PR adds a hooks.enabled/hooks.token cross-field validation check in shared config validation and changes the hooks startup error text to include remediation guidance.

## Best Possible Solution

Refresh and land a tested shared validation rule that rejects hooks.enabled=true without a non-empty hooks.token during config writes, keeps resolveHooksConfig as the runtime backstop, and uses supported redacted config patch --stdin/--file or RPC guidance.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main lets hooks.enabled and hooks.token validate as independent optional fields, config.patch runs shared validation before writing, and resolveHooksConfig only rejects the missing token later at runtime.

## Solution Assessment

No, not as written. Shared cross-field validation plus a runtime fallback is the right narrow solution, but this PR needs supported redacted remediation text, focused tests, conflict resolution, and real behavior proof before merge.
