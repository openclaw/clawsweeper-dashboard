## Summary

Keep open: the underlying bug is real and current main still lacks config-time rejection for `hooks.enabled` without `hooks.token`, but this PR should not merge as-is because its new remediation text points users at an invalid dotted CLI command.

## What This Changes

The PR adds cross-field hook token validation in `src/config/validation.ts` and expands the `resolveHooksConfig` missing-token startup error in `src/gateway/hooks.ts`.

## Best Possible Solution

Keep the cross-field validation approach, replace both remediation strings with a documented real command, add focused validation/runtime tests, then rebase or replace the stale PR branch for merge.

## Reproduction Assessment

Yes from source inspection, though not by executing a live gateway. Current `config.patch` validates the merged config through `validateConfigObjectWithPlugins`, which lacks the hook token dependency check, while startup `resolveHooksConfig` throws when hooks are enabled without a token.

## Solution Assessment

No as written. The validation boundary is maintainable, but the new guidance uses an invalid `openclaw config.patch` command and should be paired with focused regression tests before merge.
