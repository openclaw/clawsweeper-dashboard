## Summary

## What This Changes

The PR adds a hooks.enabled/hooks.token cross-field check in shared config validation and updates the hooks startup error message with remediation guidance.

## Best Possible Solution

Refresh and land a tested shared validation rule that rejects hooks.enabled=true without a non-empty hooks.token during config writes, keeps resolveHooksConfig as the runtime backstop, and uses supported redacted --stdin/--file or gateway RPC guidance.

## Reproduction Assessment

Yes. Source inspection shows current main accepts hooks.enabled and hooks.token as independent optional schema fields, config.patch relies on shared validation before persistence, and resolveHooksConfig only rejects the missing token later at runtime.

## Solution Assessment

No, not as written. The shared validation plus runtime backstop is the right narrow direction, but the remediation strings need supported redacted guidance and the branch needs focused tests plus real behavior proof before merge.
