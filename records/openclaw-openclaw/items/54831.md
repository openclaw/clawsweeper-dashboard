## Summary

Keep open: current main still lacks config-time hooks.enabled/hooks.token validation, and this PR remains the canonical implementation candidate, but the patch has a blocking remediation-command bug and still lacks external real behavior proof.

## What This Changes

The PR adds hooks.enabled/hooks.token cross-field validation during shared config validation and expands the runtime hooks error with a remediation command.

## Best Possible Solution

Land a corrected version that rejects merged config writes with hooks.enabled=true and no non-empty hooks.token, keeps resolveHooksConfig as the runtime backstop, uses supported CLI/RPC remediation syntax, adds focused regression coverage, and includes redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main permits optional hooks.enabled and hooks.token in the schema, config.patch validates the merged config through validateConfigObjectWithPlugins, and only resolveHooksConfig throws later at startup.

## Solution Assessment

No, not as written. Cross-field validation plus the runtime fallback is the narrow maintainable fix, but the PR's remediation text uses unsupported openclaw config.patch syntax and still needs proof and focused tests.
