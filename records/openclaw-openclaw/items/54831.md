## Summary

Keep open: current main still has the hooks.enabled/hooks.token validation gap and this PR is the canonical linked fix path, but the patch is not merge-ready because it adds unsupported inline token guidance, lacks real behavior proof, and is currently conflicting.

## What This Changes

The PR adds shared config validation for hooks.enabled=true without hooks.token and changes resolveHooksConfig's startup error text to include remediation guidance.

## Best Possible Solution

Refresh onto current main, add a tested shared validation rule using normalizeOptionalString, keep resolveHooksConfig as the runtime backstop, and replace remediation with supported redacted `config patch --stdin/--file` or gateway RPC guidance.

Yes by source inspection. ## Solution Assessment

No, not as written. The shared validation plus runtime backstop is the right narrow direction, but the branch needs current-main refresh, supported redacted guidance, focused tests, and real behavior proof before merge.
