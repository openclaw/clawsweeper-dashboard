## Summary

Keep open. Current main still lacks config-time validation for `hooks.enabled` without `hooks.token`, so the PR addresses a real remaining problem, but the patch is not merge-ready because its remediation text points users at an unsupported top-level command and the user-facing fix is missing required changelog coverage.

## What This Changes

The PR adds a `hooks.enabled`/`hooks.token` cross-field validation check to config validation and expands the runtime hooks startup error with remediation guidance.

## Best Possible Solution

Land a corrected version that rejects `hooks.enabled: true` with a missing or blank token during merged config write validation, preserves the runtime guard as a final safety net, uses supported CLI/RPC remediation syntax, and includes focused tests plus a single-line active changelog entry.

Yes. Current-main source gives a high-confidence reproduction path: a merged config with `hooks.enabled: true` and no non-empty `hooks.token` passes the current schema/config validation path but throws from `resolveHooksConfig` when hooks are resolved.
