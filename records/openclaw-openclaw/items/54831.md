## Summary

Keep open. Current main still has the source-reproducible hooks config validation gap, but this PR cannot merge as written because its remediation text uses an unsupported command, the external PR lacks real behavior proof, and the branch is currently dirty/conflicting.

## What This Changes

The PR adds hooks.enabled/hooks.token cross-field validation in shared config validation and expands the hooks startup error with remediation text.

## Best Possible Solution

Land a refreshed patch that rejects hooks.enabled=true without a non-empty hooks.token during shared config writes, keeps resolveHooksConfig as the runtime backstop, uses supported CLI/RPC remediation text, and adds focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows config.patch validates the merged config through validateConfigObjectWithPlugins, while hooks.enabled/hooks.token remain optional in schema validation and the non-empty token requirement is only enforced later by resolveHooksConfig.

## Solution Assessment

No, not as written. Shared cross-field validation plus a runtime fallback is the right narrow approach, but the PR's remediation command is unsupported and the branch still needs proof, refresh, and focused tests.
