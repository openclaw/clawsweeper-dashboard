## Summary

Keep this PR open because the underlying Telegram media retrieval gap is still real on current main, but the PR should not merge as-is. The branch targets deleted legacy Telegram paths, while current main owns Telegram actions and media handling under the plugin; its new download path also bypasses the existing guarded Telegram transport and media-store path.

## What This Changes

The PR adds Telegram media `file_id` values to inbound media placeholders, adds a Telegram `download-file` message action backed by Bot API `getFile`, and extracts a shared inbound media file-id helper.

## Best Possible Solution

Refresh or replace the implementation on the current Telegram plugin surface: add `download-file` through `extensions/telegram/src/channel-actions.ts` and `extensions/telegram/src/action-runtime.ts`, reuse or refactor the existing guarded media-download helper, expose a parser-safe `file_id` history contract, update exact-placeholder consumers such as inbound-audio detection, and add focused plugin tests plus docs/changelog coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on current main, Telegram `message(action="download-file", fileId="...")` is not mapped or advertised and `handleAction` rejects unmapped actions; current placeholders are bare tags with only a limited sidecar `[file_id:...]` fallback.

## Solution Assessment

No. The PR addresses a valid gap, but this is not the best current solution because it edits removed legacy files and adds a credentialed download path outside the current Telegram plugin transport/media-store contracts.
