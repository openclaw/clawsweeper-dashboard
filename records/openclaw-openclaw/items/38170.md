## Summary

Close as mostly implemented: current main and shipped releases already cover the central Telegram media retrieval use case through the guarded inbound media store, while the remaining branch diff is stale legacy-path work with direct-download security concerns.

## What This Changes

The PR adds Telegram `file_id` attributes to media placeholders, a `download-file` message action, and helper/test updates in the old Telegram channel paths.

## Best Possible Solution

Keep the shipped media-store implementation; if maintainers still want a Telegram `download-file` action, design it fresh on the plugin action surface and route it through the existing guarded downloader.

## Reproduction Assessment

Yes for the PR review findings: source inspection on current main shows `download-file` is not on the Telegram plugin action surface, while the PR targets removed legacy files and direct download code. The original user problem is already covered by current media-store behavior and the closed linked issue.

## Solution Assessment

No. The useful user outcome is already shipped through local `MediaPath(s)` and guarded Telegram media downloads; this branch is not the best implementation because it bypasses that path and would need a rewrite onto `extensions/telegram`.

Full review comments:

- **[P2] Port download-file onto the Telegram plugin surface:** `src/channels/plugins/actions/telegram.ts:107-108`
  - body: The added discovery and dispatch live in the legacy channel-action shim, but current main owns Telegram message actions in `extensions/telegram/src/channel-actions.ts`. Rebase any accepted action onto the plugin surface so it is actually advertised and handled.
  - confidence: 0.93

  - body: This path calls Telegram URLs directly, buffers the whole response, and writes to `/tmp` instead of using the existing transport-aware, size-limited, SSRF-checked media downloader and media store. That can break proxy/apiRoot deployments and bypass media protections.
  - confidence: 0.91
- **[P2] Preserve audio classification for attributed placeholders:** `src/telegram/bot-message-context.ts:470`
  - body: Emitting placeholders like `<media:audio file_id="...">` requires updating the shared inbound-audio parser too; it still matches only bare `<media:audio>` when media type facts are absent, so inbound-audio-conditioned dispatch can regress after download failures.
  - confidence: 0.87
- **[P3] Avoid deterministic temporary path overwrites:** `src/agents/tools/telegram-actions.ts:500`
  - body: The output path uses only the first 16 characters of `fileId`, so different files sharing that prefix can overwrite each other. Use the media store or a collision-resistant sanitized name.
  - confidence: 0.79
