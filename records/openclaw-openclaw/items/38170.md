## Summary

Keep this PR open, but do not merge it as-is. The requested Telegram file retrieval capability is still not fully present on current main, but the branch targets removed legacy paths, misses the current plugin-owned action surface, and introduces a credentialed download path that bypasses existing Telegram media-download guards.

## What This Changes

The PR adds Telegram `file_id` values to inbound media placeholders, introduces a Telegram `download-file` message action using Bot API `getFile`, and extracts inbound media file-id resolution into a helper.

## Best Possible Solution

Refresh or replace the implementation on the current Telegram plugin surface: add `download-file` through `extensions/telegram/src/channel-actions.ts` and `extensions/telegram/src/action-runtime.ts`, reuse/refactor the guarded Telegram media downloader and media store, expose a parser-safe `file_id` history contract, update audio placeholder detection, and cover the behavior with focused Telegram plugin tests plus docs/changelog as needed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main does not advertise or dispatch Telegram `download-file`, and the current bare-placeholder path can still omit a `file_id` for no-text undownloaded media.

## Solution Assessment

No. The proposed direction addresses a real gap, but this branch is not the best current solution because it edits removed legacy files and adds direct Bot API downloads outside the guarded Telegram transport/media-store pipeline.
