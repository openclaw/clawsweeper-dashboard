## Summary

Keep this PR open for maintainer follow-up. Current main still does not provide the proposed Telegram `download-file` action or parser-safe `file_id` placeholder attribute, but the PR diff is not mergeable as-is because it targets removed pre-plugin paths and introduces blocking issues around Telegram transport/media-guard reuse and inbound-audio detection.

## What This Changes

The PR adds a Telegram `download-file` action, embeds Telegram `file_id` values in inbound media placeholder tags, extracts a shared inbound media file-id helper, and relaxes audio transcript placeholder checks.

## Best Possible Solution

Refresh or replace the implementation on the current Telegram plugin surface: add `download-file` through `extensions/telegram/src/channel-actions.ts` and `extensions/telegram/src/action-runtime.ts`, reuse the existing guarded Telegram media-download/store helpers, expose `file_id` through a parser-safe inbound history/placeholder contract, update exact-placeholder consumers, and cover the action/body/history/docs paths with focused tests.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: a Telegram-scoped `message(action="download-file", fileId="...")` is not advertised or mapped, and inbound media placeholders are still bare tags or sidecar `[file_id:...]` strings rather than placeholder attributes.

## Solution Assessment

No. The PR solves a real remaining gap, but this diff is not the best current fix because it targets removed paths and bypasses the Telegram plugin's existing transport and guarded media-download contracts; the safer path is a current-plugin port that reuses those helpers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reuse the Telegram media transport for downloads:** `src/agents/tools/telegram-actions.ts:478-493`
  - body: The new download path hard-codes `api.telegram.org` and calls global `fetch`, so it ignores per-account `apiRoot`, configured Telegram proxy/fallback transport, and the existing guarded media-download path. In proxy or custom Bot API deployments, `download-file` can fail even while normal Telegram media works.
  - confidence: 0.9
- **[P2] Preserve inbound audio detection:** `src/telegram/bot-message-context.ts:470`
  - body: Embedding `file_id` changes audio bodies to `<media:audio file_id="...">`, but current inbound-audio detection only matches exact `<media:audio>` when media types are unavailable. Audio messages that fail media download would stop triggering inbound-audio TTS/dispatch behavior.
  - confidence: 0.87
- **[P3] Avoid deterministic tmp-path collisions:** `src/agents/tools/telegram-actions.ts:500`
  - body: Using only the first 16 characters of `fileId` for `/tmp/tg_<prefix><ext>` can overwrite another downloaded file with the same prefix. Use a sanitized full ID or a unique media-store path instead.
  - confidence: 0.8
