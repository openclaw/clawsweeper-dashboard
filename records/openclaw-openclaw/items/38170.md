## Summary

Keep open for maintainer review: current main still does not implement Telegram `download-file`, but this PR is stale against the plugin-owned Telegram surface and introduces an unsafe direct download path.

## What This Changes

The PR adds Telegram `file_id` media placeholders, a `download-file` action that downloads Bot API files to `/tmp`, and helper/test updates in legacy Telegram paths.

## Best Possible Solution

Implement the feature, if accepted, as a Telegram plugin action that exposes a parser-safe file_id contract and routes downloads through the existing transport-aware media store path with focused tests, docs, and changelog.

## Reproduction Assessment

Yes. Source inspection against current main is enough: Telegram action discovery/dispatch omits `download-file`, while the PR diff targets legacy paths and direct download code.

## Solution Assessment

No. The direction is useful, but this patch is not the best implementation because it edits removed legacy files and bypasses the existing guarded Telegram media downloader.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Port download-file onto the Telegram plugin surface:** `src/channels/plugins/actions/telegram.ts:107-108`
  - body: The added discovery/dispatch lives in the legacy channel-action shim, but current main owns Telegram message actions in `extensions/telegram/src/channel-actions.ts`. Rebase this onto the plugin surface so `download-file` is actually advertised and handled.
  - confidence: 0.93
- **[P1] Use the guarded Telegram media downloader:** `src/agents/tools/telegram-actions.ts:478-493`
  - body: This path calls Telegram URLs directly, buffers the whole response, and writes to `/tmp` instead of using the existing transport-aware, size-limited, SSRF-checked media downloader and media store. That can break proxy/apiRoot deployments and bypass media protections.
  - confidence: 0.91
- **[P2] Keep audio detection compatible with attributed placeholders:** `src/telegram/bot-message-context.ts:470`
  - body: Emitting placeholders like `<media:audio file_id="...">` requires updating the shared inbound-audio parser too; it still matches only bare `<media:audio>` when media type facts are absent, so inbound-audio-conditioned dispatch can regress after download failures.
  - confidence: 0.87
- **[P3] Avoid deterministic temporary-path overwrites:** `src/agents/tools/telegram-actions.ts:500`
  - body: The output path uses only the first 16 characters of `fileId`, so different files sharing that prefix can overwrite each other. Use the media store or a collision-resistant sanitized name.
  - confidence: 0.79
