## Summary

Keep open for maintainer review: the PR addresses a real Telegram media-file gap, but it is stale against current main and its download path bypasses the guarded Telegram media pipeline.

## What This Changes

The PR branch adds Telegram file IDs to media placeholders, exposes a Telegram `download-file` message action, and moves inbound file-id lookup into a helper.

## Best Possible Solution

Replace the stale branch with a Telegram plugin-surface implementation that exposes a parser-safe file_id history contract and routes downloads through the existing guarded downloader/media store with focused plugin tests and user-facing docs/changelog if accepted.

## Reproduction Assessment

Yes. Source inspection is enough: current main does not advertise or dispatch Telegram `download-file`, and its placeholder helpers still emit bare media tags except for a limited sidecar path.

## Solution Assessment

No. The product direction may be useful, but this PR is not the best implementation because it edits removed legacy files and adds direct credentialed Bot API downloads outside the guarded Telegram media path.

  - confidence: 0.87

  - body: The output path uses only the first 16 characters of `fileId`, so distinct files with the same prefix can overwrite each other. Use the media store or a collision-resistant sanitized name.
  - confidence: 0.79
