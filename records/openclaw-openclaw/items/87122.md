## Summary

Keep open: this PR is not merge-ready because it removes persisted overload cooldown/probe state that current main, tests, docs, and the canonical retry tracker still rely on, and it lacks real overload/fallback proof.

## What This Changes

The PR changes auth-profile failure handling so provider `overloaded` failures record metadata without setting `cooldownUntil`, and updates fallback tests to expect no persisted overload cooldown.

## Best Possible Solution

Keep overload retry/backoff design on https://github.com/openclaw/openclaw/issues/49376; a replacement should preserve the cooldown/probe contract or explicitly replace it with tested, maintainer-approved behavior.

No. Removing persisted overload cooldown state is not the best fix; the maintainable path is a maintainer-approved retry/backoff contract that either preserves or deliberately replaces the current cooldown/probe behavior.
