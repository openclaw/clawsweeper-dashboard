## Summary

Keep open: the branch is a meaningful overload/fallback behavior change, but it is not merge-ready because it can clear existing active cooldown state and still lacks after-fix real behavior proof while the broader retry/backoff policy remains on the canonical tracker.

## What This Changes

The PR changes auth-profile failure accounting so provider `overloaded` failures record failure metadata without persisting `cooldownUntil`, with matching auth-profile and fallback tests.

## Best Possible Solution

Preserve existing active cooldown/probe state unless maintainers explicitly approve replacing it, then align overload retry/backoff behavior with the canonical tracker and add real behavior proof.

## Reproduction Assessment

Yes for the review blocker: source inspection shows the new `overloaded` branch clears `cooldownUntil`, so an existing active cooldown can be removed by a later overload mark. I did not run live overload traffic in this read-only review.

## Solution Assessment

No. Clearing `cooldownUntil` for overloads is too broad because it can erase existing active cooldown windows; the safer path is to preserve those windows and settle overload retry/backoff policy explicitly.
