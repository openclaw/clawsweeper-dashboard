## Summary

Keep open: the direction is useful, but the patch currently clears an existing non-overload cooldown when a later overload is recorded, and the external PR still lacks real behavior proof.

## What This Changes

The branch stops overloaded auth-profile failures from persisting `cooldownUntil` and updates two fallback/auth-profile tests to expect repeated overload attempts.

## Best Possible Solution

Preserve active non-overload cooldown windows while skipping creation of new cooldown windows for fresh overloaded failures, then prove the real fallback path with redacted runtime output.

## Reproduction Assessment

Yes for the introduced defect: current main preserves active cooldown windows through `keepActiveWindowOrRecompute`, while the PR diff would clear `cooldownUntil` for any later overloaded failure before checking the existing active window.

## Solution Assessment

No as-is. The maintainable fix is to avoid creating new overloaded cooldowns without deleting active cooldowns created by rate-limit, auth, or other non-overload failures.
