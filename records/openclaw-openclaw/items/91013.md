## What This Changes

The PR changes Gateway chat abort state from timestamp-only markers to ordered markers so stale aborts do not suppress fresh chat events while current aborts still do.

## Best Possible Solution

Merge the Gateway state-level fix with one shared ordering source for run registrations and abort markers, while keeping chat.send replay and TTL cleanup timestamp-compatible.

## Reproduction Assessment

Yes from source, without running tests: current main checks only abort-marker presence in `server-chat.ts`, so a stale marker for the client run id can suppress fresh assistant and lifecycle projection. The PR body also reports a current-main red regression for that case.

## Solution Assessment

Yes, this appears to be the right fix boundary: Gateway chat-run state owns both the registry entry and abort marker, while provider routing, UI, chat.send replay, and maintenance only need their existing contracts preserved.
