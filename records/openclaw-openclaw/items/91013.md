## Summary

Keep open for maintainer review: current main and the latest release still use presence-only abort markers, while this PR adds the ordered marker check and focused coverage without a blocking correctness finding. The remaining concern is Gateway message-delivery semantics, especially confirming that retained abort markers should still drive chat.send replay while stale markers stop suppressing fresh event projection.

## What This Changes

The PR changes Gateway chat abort marker state from numeric presence-only timestamps to ordered markers so stale aborts no longer suppress fresh chat delta/final projection while current aborts still suppress active output.

## Best Possible Solution

Land the ordered-marker implementation after Gateway maintainers confirm the replay contract and the normal required checks for the clean head remain green.

## Reproduction Assessment

Yes. Current main and v2026.6.6 use presence-only abort-marker checks, and the PR body includes a red/green production-source Gateway harness showing stale markers suppress on main and emit delta/final after the patch.

## Solution Assessment

Yes for the projection bug: putting ordering metadata in the Gateway chat state module and using it at both projection checks is the narrow owner-boundary fix. The preserved presence-based chat.send replay behavior is a maintainer contract check, not a blocking patch defect by itself.
