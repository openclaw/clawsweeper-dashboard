## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve the existing init CLI contract:** `src/cli/plugins-cli.ts:239`
  - body: Current main and the public docs support `openclaw plugins init <id> --name "Display Name"`, but this patch removes `--name` and redefines the positional argument as the display name. Existing scripts and docs snippets will now fail with an unknown option or silently get different display-name behavior, so keep the old form as a tested compatibility path while adding `--type provider`.
  - confidence: 0.95
- **[P2] Register generated providers in the model catalog:** `src/cli/plugins-authoring-command.ts:637`
  - body: The provider scaffold only calls `api.registerProvider(...)`. Current provider docs say `registerModelCatalogProvider` is the control-plane catalog surface for list/help/picker UI, and `defineSingleProviderPluginEntry` wires both surfaces together. As generated, a provider can pass the scaffold test and ClawHub validation while still being absent from model browsing surfaces.
  - confidence: 0.87
- **[P1] Avoid unpinned lifecycle installs in PR validation:** `scripts/validate-plugin-init-provider-scaffold.ts:64`
  - body: This script is run by a new `pull_request` workflow and executes `npm install` inside the generated package. That package uses `clawhub: "latest"`, `openclaw: "latest"`, and semver-ranged dev dependencies without a lockfile, so CI will execute registry lifecycle scripts and resolve moving versions outside the repo lockfile. Pin/pack the inputs or disable lifecycle scripts before making this a PR gate.
  - confidence: 0.86
