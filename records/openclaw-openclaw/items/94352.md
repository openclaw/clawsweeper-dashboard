## Summary

Keep open: the provider scaffold idea is useful, but the patch should not merge as-is because it removes the shipped `openclaw plugins init <id> --name <name>` command shape instead of adding provider scaffolding compatibly.

## What This Changes

This PR changes `openclaw plugins init` to derive plugin ids from a display-name positional argument, adds `--type provider`, and adds provider-scaffold docs, tests, and CI validation.

## Best Possible Solution

Add provider scaffolding as an additive `--type provider` path while preserving the shipped `<id> --name <name>` invocation with regression coverage for both command forms.

## Reproduction Assessment

Yes. Source and release-doc inspection show the shipped command accepts `openclaw plugins init <id> --name <name>`, while the PR removes `--name` from Commander registration.

## Solution Assessment

No. Provider scaffolding is a reasonable init feature, but the best fix is additive: keep the old id/name command form and add provider support beside it.
