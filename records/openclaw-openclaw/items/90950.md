## Summary

Keep open: the ACOS-controlled mode idea is directionally aligned with OpenClaw's security goals, but this PR is not merge-ready because public Gateway callers can self-supply ACOS provenance and diagnostic mode still allows mutating actions when valid provenance is present. The submitted proof is limited to tests/import/lint claims, not real controlled-mode behavior across the guarded entry points.

## What This Changes

The PR adds an opt-in ACOS-controlled execution mode using new environment flags and provenance metadata across agent, Gateway, channel, cron, shell, and apply_patch paths, plus docs and protocol/Swift model fields.

## Best Possible Solution

Land this only after ACOS authority is derived from a trusted server/control-plane boundary, diagnostic mode fails closed for every mutating path, and real Gateway/channel/cron/tool proof validates the contract.

## Reproduction Assessment

Yes, source-reproducible for the PR findings: the PR schema accepts `acosProvenance`, the Gateway forwards it to ingress, and the validator lets diagnostic-mode mutating actions continue when provenance exists. I did not run a live controlled-mode Gateway or Telegram setup in this read-only review.

## Solution Assessment

No, this is not the best merge shape yet. A safer design would keep ACOS provenance server-side or restricted to trusted backend/control-plane callers, then make diagnostic mode reject every mutating action before relying on runtime proof.
