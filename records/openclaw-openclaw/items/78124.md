## Summary

Keep open: the PR is a plausible fix direction, but it still misses Open-WebUI's actual default forwarded chat-id header, changes the documented `user` precedence despite the PR body claiming otherwise, and lacks after-fix real Open-WebUI proof.

## What This Changes

The PR changes OpenAI-compatible gateway session resolution to derive deterministic session keys from `x-chat-id` or `x-thread-id` and adds request-context unit tests.

## Best Possible Solution

Define and implement a documented stable-session contract for OpenAI-compatible clients: explicit OpenClaw session key first, existing `user` behavior preserved or deliberately changed with tests, Open-WebUI's default forwarded chat id supported, and real multi-turn proof attached.

## Reproduction Assessment

Yes for source-level reproduction: current main creates a random OpenAI-compatible session when no explicit session key or `user` is present, and upstream Open-WebUI forwards its stable chat id as `X-OpenWebUI-Chat-Id` when configured. I did not establish after-fix live multi-turn proof for this PR.

## Solution Assessment

No. Header-based correlation is a reasonable direction, but this patch reads the wrong Open-WebUI default header, changes the documented `user` precedence, and lacks docs plus real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Read Open-WebUI's default chat header:** `src/gateway/http-utils.ts:140-142`
  - body: Open-WebUI currently forwards `metadata.chat_id` as `X-OpenWebUI-Chat-Id` when forwarded user headers are enabled, but this patch only reads `x-chat-id` and `x-thread-id`. Default Open-WebUI requests would still fall through to `randomUUID()`, so the linked session-reuse problem remains.
  - confidence: 0.94
- **[P3] Preserve the documented user precedence:** `src/gateway/http-utils.ts:144-149`
  - body: The PR body says the OpenAI `user` path still has higher precedence, but the new header lookup runs before `params.user`. Requests that already rely on the documented `user`-based stable session can be silently rerouted when a chat/thread header is also present; either keep `user` ahead or document and test the intentional precedence change.
  - confidence: 0.82
