## Summary

Keep open: this PR targets a real session-continuity bug that is still present on current main and the latest release, but the branch is not merge-ready because it misses Open-WebUI's default forwarded chat-id header, changes the documented `user` precedence, broadens shared resolver behavior, lacks real behavior proof, and is currently dirty against `main`.

## What This Changes

The PR adds `x-chat-id` / `x-thread-id` derived stable session keys to the shared OpenAI-compatible gateway request context and adds request-context tests for those headers.

## Best Possible Solution

Land a narrowed stable-session contract that reads the actual Open-WebUI header, preserves or explicitly documents precedence, scopes or documents sibling endpoints, and includes live multi-turn proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 fall back to `randomUUID()` when no explicit OpenClaw session key or OpenAI `user` is present. I did not run a live Open-WebUI multi-turn setup in this read-only review.

## Solution Assessment

No. Header-based correlation is a plausible direction, but this patch misses Open-WebUI's default header, changes `user` precedence, and broadens shared resolver behavior without docs, endpoint tests, or live proof.

Full review comments:

- **[P2] Read Open-WebUI's forwarded chat-id header:** `src/gateway/http-utils.ts:145-146`
  - body: Open-WebUI's forwarded chat id defaults to `X-OpenWebUI-Chat-Id`, but this branch only reads `x-chat-id` and `x-thread-id`. A stock Open-WebUI setup with forwarding enabled can still fall through to the random UUID fallback, leaving the reported session proliferation unresolved.
  - confidence: 0.94
- **[P1] Keep user ahead of new header hints:** `src/gateway/http-utils.ts:145-148`
  - body: Current behavior and docs make OpenAI `user` the stable session source after `x-openclaw-session-key`. This lookup runs before `params.user`, so clients that already send `user` plus a chat/thread header can be silently moved to a different session key.
  - confidence: 0.89
- **[P2] Scope the hints before changing the shared resolver:** `src/gateway/http-utils.ts:145-148`
  - body: `resolveGatewayRequestContext` is also used by `/v1/responses`, so adding these headers here changes Responses session routing without docs or tests for that endpoint. Gate the hints to the intended endpoint or cover the shared contract explicitly.
  - confidence: 0.86
