## Summary

Keep open: current main still has the no-user random OpenAI-compatible session fallback, but this PR is not merge-ready because it misses Open-WebUI's default forwarded chat header, changes session precedence, broadens the shared resolver to Responses without coverage, and lacks real Open-WebUI-to-OpenClaw proof.

## What This Changes

The PR adds `x-chat-id` and `x-thread-id` header-derived stable session keys to the shared Gateway request-context resolver and covers those cases in resolver tests.

## Best Possible Solution

Repair this branch or replace it with a narrow Gateway session-correlation contract that preserves established precedence, supports the actual Open-WebUI forwarded header, scopes the shared resolver behavior, and includes redacted real multi-turn proof.

## Solution Assessment

No, not as-is: header-based correlation is plausible, but this patch changes precedence, misses Open-WebUI's default forwarded header, and broadens the shared resolver without enough endpoint coverage.

- **[P1] Keep user ahead of inferred session headers:** `src/gateway/http-utils.ts:144-148`
  - body: Current docs and tests define OpenAI `user` as the stable session source after `x-openclaw-session-key`, but this branch resolves chat/thread headers before `params.user`.   - confidence: 0.9
- **[P2] Read Open-WebUI's forwarded chat header:** `src/gateway/http-utils.ts:145-146`
  - body: Open-WebUI source forwards the chat id through `FORWARD_SESSION_INFO_HEADER_CHAT_ID`, defaulting to `X-OpenWebUI-Chat-Id`, when forwarding is enabled. Reading only `x-chat-id` and `x-thread-id` can leave the reported Open-WebUI flow on the random UUID fallback.
  - confidence: 0.92
