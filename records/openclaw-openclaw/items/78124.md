## Summary

Keep open: the PR is aimed at a real OpenAI-compatible session-continuity problem, but the current patch still misses Open-WebUI's default forwarded chat header, changes the documented `user` precedence despite saying it preserves it, and has only test output rather than after-fix real behavior proof.

## What This Changes

This PR derives Gateway OpenAI-compatible session keys from `x-chat-id` or `x-thread-id` headers and adds request-context tests for that routing.

## Best Possible Solution

Define and document a stable-session contract for OpenAI-compatible clients: full OpenClaw session override first, OpenAI `user` next unless intentionally changed, Open-WebUI's forwarded `x-openwebui-chat-id` when enabled, deliberate generic aliases, regression tests, and real multi-turn proof.

## Reproduction Assessment

Yes, at source level. Current main falls back to `randomUUID()` when no explicit OpenClaw session key or OpenAI `user` is present, and upstream Open-WebUI source shows the stable chat id is forwarded as `X-OpenWebUI-Chat-Id` when that forwarding option is enabled; I did not run a live multi-turn Open-WebUI setup.

## Solution Assessment

No. Header-based correlation is the right direction, but this patch misses Open-WebUI's default header and puts chat/thread headers ahead of the existing `user` path without docs or tests for that precedence change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Read Open-WebUI's default chat header:** `src/gateway/http-utils.ts:144-147`
  - body: Open-WebUI defines and forwards the chat id as `X-OpenWebUI-Chat-Id` when forwarding is enabled, but this patch only reads `x-chat-id` and `x-thread-id`. Default Open-WebUI requests would still miss the stable id and fall through to `randomUUID()`, leaving the reported session proliferation unresolved.
  - confidence: 0.94
- **[P3] Preserve the documented user precedence:** `src/gateway/http-utils.ts:144-147`
  - body: The PR body says the OpenAI `user` path still has higher precedence, but the new header lookup runs before `params.user`. Requests that already rely on the documented `user`-based stable session can be silently rerouted when a chat/thread header is also present; keep `user` ahead or document and test the intentional precedence change.
  - confidence: 0.84
