## Label Justifications

- `P1`: The linked bug can silently lose a completed Telegram final reply while session state appears healthy.
- `merge-risk: 🚨 compatibility`: The diff changes the existing contract where intentional suppressed sends can be terminal successes.
- `merge-risk: 🚨 message-delivery`: Hook-cancelled final replies could be retried or surfaced as failed delivery instead of staying intentionally suppressed.
- `merge-risk: 🚨 session-state`: The patch can preserve `pendingFinalDelivery` with an error for suppression states that current code treats as complete.

## Summary

Keep open: this is a focused reliability fix for a credible Telegram final-delivery bug, but the patch is not merge-ready because it turns every durable `suppressed` send into a failed route, which conflicts with existing hook-cancelled suppression semantics, and the PR only provides mocked test proof.

## What This Changes

The PR changes final-reply routing and restart recovery so no-result final deliveries preserve pending-final state and record non-null retry errors, with focused regression tests.

## Best Possible Solution

Narrow failure accounting to unproven/no-identity delivery outcomes while preserving intentional hook suppression, then add redacted real runtime proof that the no-send path records a non-null pending-final error.

## Reproduction Assessment

Yes at source level, but not with a live Telegram repro here. Current main can treat a no-result routed final send and a `deliver: false` resume as clean pending-final accounting, and the linked issue supplies production Telegram evidence.

## Solution Assessment

No, not as written. The direction is right for no-send-proof accounting, but the route-level fix should distinguish accidental no-result delivery from intentional hook suppression.
