## Summary

Keep open: current main and v2026.6.10 still route post-compaction session context through `limitHistoryTurns`, which slices away a leading `compactionSummary` when the retained tail exceeds the configured user-turn limit.

## What This Changes

The issue reports that configured `historyLimit` / `dmHistoryLimit` trimming silently removes the post-compaction summary from later model context.

## Best Possible Solution

Patch the shared history limiter to preserve a leading `compactionSummary` before the trimmed tail, with focused regression coverage and no new config or channel-specific shim.

## Reproduction Assessment

Yes. Current source shows `buildSessionContext` prepends a `compactionSummary`, then `limitHistoryTurns` can slice from a later user-turn index and drop that prefix; I did not run a live channel/provider reproduction in this read-only review.

## Solution Assessment

Yes. The best fix is in the shared limiter because both the normal attempt path and the direct compaction path already funnel through it.
