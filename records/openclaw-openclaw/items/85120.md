- `impact:message-loss`: The reported failure disconnects the active webchat/tool session before the update command can return a durable result.

## Summary

Keep open: current main has update safety checks, but the macOS ancestor helper still only records the immediate parent, so an in-band chat/tool subprocess deeper under the LaunchAgent gateway can plausibly miss the refusal and reach the stop/restart path. The adjacent systemd and managed macOS handoff issues are related but do not fully own this manual macOS LaunchAgent variant.

## What This Changes

The issue asks for macOS LaunchAgent `openclaw update` invocations from a live OpenClaw session to either hand off to a durable launchd-owned updater or refuse before stopping the gateway.

## Best Possible Solution

Harden the existing update safety boundary so any macOS LaunchAgent-owned child or grandchild `openclaw update` either refuses before stopping the gateway or transfers to a launchd-owned worker with durable status/result reporting.

## Reproduction Assessment

No live macOS reproduction was run here. The source path is reproducible by inspection: the CLI relies on an ancestor set that stops at `process.ppid` on macOS, which can miss a gateway grandparent for in-band chat/tool command runners.

## Solution Assessment

No, current main is not yet the best solution for this issue. The safer narrow fix is to extend the existing refusal boundary for macOS LaunchAgent-owned process trees, or explicitly route this path through a launchd-owned handoff before any gateway stop/restart.
