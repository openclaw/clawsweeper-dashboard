## Summary

Keep open: current main still blocks in-band package updates only when the managed gateway PID is present in `getSelfAndAncestorPidsSync()`, and that helper still stops at the immediate parent on macOS. The linked landed fixes addressed adjacent cwd and lazy-import restart failures, but the maintainer comment and current source both leave the nested LaunchAgent ownership case unresolved.

## What This Changes

The issue asks OpenClaw to make in-band macOS LaunchAgent `openclaw update` either hand off to launchd-owned supervision or refuse before stopping the gateway.

## Best Possible Solution

Extend the existing safety contract so nested macOS in-band updater descendants reliably refuse before any LaunchAgent stop, or replace that path with a launchd-owned handoff that records durable status after restart.

## Reproduction Assessment

No high-confidence reproduction was run in this review. Source inspection plus reporter-supplied macOS logs give a medium-confidence path: invoke `openclaw update` from a nested child of a LaunchAgent-managed gateway so the gateway is not the immediate parent on macOS.

## Solution Assessment

No, the current implementation is not sufficient because the guard depends on a non-Linux ancestor set that stops at `process.ppid`. The narrow maintainable fix is to detect nested macOS ancestry before `service.stop(...)` or perform a durable launchd-owned handoff before stopping the gateway.
