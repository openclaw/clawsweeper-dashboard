## Summary

Keep open: current main still has only a partial in-band update guard for macOS, and the recent landed update fixes address adjacent cwd and lazy-import restart failures rather than the nested LaunchAgent-supervised update path reported here.

## What This Changes

The issue reports that `openclaw update` launched from a live macOS LaunchAgent-backed OpenClaw session can stop the gateway that is supervising the command before a durable handoff exists.

## Best Possible Solution

Make the macOS in-band package update path reliably refuse before `service.stop(...)` when the active gateway is anywhere in the caller's process ancestry, or hand off to a launchd-owned updater with durable status before stopping the LaunchAgent.

## Reproduction Assessment

No high-confidence current-main live repro was run here. Source inspection plus the reporter's redacted macOS logs support a medium-confidence path: launch `openclaw update` from a nested child where the LaunchAgent gateway is an ancestor but not `process.ppid`.

## Solution Assessment

No, the current partial guard is not enough for the reported macOS/manual in-band shape. The maintainable fix is to detect nested macOS ancestry before service stop or perform a durable launchd-owned handoff first.
