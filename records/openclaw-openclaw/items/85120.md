## Label Justifications

- `P1`: A real in-band update path can drop the active OpenClaw control session while performing a gateway restart on macOS LaunchAgent installs.
- `impact:message-loss`: The initiating webchat/tool session can lose the update command result when the gateway is stopped underneath it.
- `impact:session-state`: The failure is about an in-band session losing the runtime that supervises the self-referential update.

## Summary

Keep open: current main still has the ancestry-dependent refusal, but macOS only includes `process.ppid`, so a nested OpenClaw tool/chat-launched updater can miss the LaunchAgent gateway ancestor and enter the managed service stop path.

## What This Changes

The issue requests preventing in-band `openclaw update` on macOS LaunchAgent gateways from killing the gateway/session that owns the updater.

## Best Possible Solution

Preserve the existing safety contract by making nested macOS in-band updater processes reliably refuse before stopping the LaunchAgent, or by using a launchd-owned durable handoff before any gateway stop.

## Reproduction Assessment

No live current-main macOS repro was run in this read-only review. Source inspection plus the reporter's v2026.5.20 logs give a clear medium-confidence path: invoke `openclaw update` from a nested OpenClaw-owned child where the LaunchAgent gateway is not the immediate parent.

## Solution Assessment

Yes for a narrow fix that preserves the current refusal contract: extend the macOS ancestry check or equivalent guard so it catches nested gateway descendants before `service.stop`. A launchd-owned durable handoff is also acceptable, but it is larger than the minimum repair.
