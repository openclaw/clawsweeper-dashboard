## Summary

Keep open: current main still lacks proof or code for the specific nested macOS LaunchAgent in-band updater case, and a recent maintainer comment explicitly says the landed update hardening does not close this shape.

## What This Changes

The issue reports that running `openclaw update` from an in-band macOS LaunchAgent-backed OpenClaw session can stop the gateway/session that is supervising the update.

## Best Possible Solution

Extend the existing in-band update safety contract so nested macOS LaunchAgent descendants reliably refuse before service stop, or provide an equivalent launchd-owned durable handoff with status that survives the gateway restart.

## Reproduction Assessment

No, not with high confidence from this review: I did not run a live macOS LaunchAgent reproduction. Source inspection gives a medium-confidence path because non-Linux ancestry stops at `process.ppid`, so a nested OpenClaw-launched updater can miss the gateway ancestor and proceed to `stopLaunchAgent`.

## Solution Assessment

Yes for the refusal direction: extending the existing gateway-process-tree refusal to nested macOS descendants is the narrowest maintainable fix. A launchd-owned durable handoff would also satisfy the issue, but it is broader and should not be required to preserve the current safety contract.
