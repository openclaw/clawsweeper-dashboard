## Summary

Keep this PR open. The linked bug is still present on current main and v2026.6.10, and this branch is the direct candidate fix; the remaining blocker is maintainer acceptance of the auth/session-state durability tradeoff from moving success bookkeeping into background work.

## What This Changes

The PR moves post-run auth-profile success bookkeeping in the embedded agent runner off the awaited reply path, adds slow/failure logging with redacted profile IDs, and adds a regression test for nonblocking return ordering.

## Best Possible Solution

Land the latency fix after maintainers either accept the bounded stale-auth-metadata risk or require the background write to be tracked by a drainable lifecycle without re-blocking reply delivery.

## Reproduction Assessment

Yes at source level: current main and v2026.6.10 still await `markAuthProfileSuccess` immediately after `embedded run done`, and lane completion waits for the embedded task to resolve. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

Acceptable mitigation, with one maintainer decision remaining. Moving the auth-success write off the reply path targets the shared embedded-runner layer, but maintainers must decide whether fire-and-forget persistence is acceptable or should be made drainable.
