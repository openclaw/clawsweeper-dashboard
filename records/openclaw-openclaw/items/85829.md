## Summary

Keep open: the PR targets a credible source-level cause for the linked Discord reply delay, but it is not merge-ready because real behavior proof is absent and the new slow/failure logs expose auth profile IDs without the existing redaction pattern.

## What This Changes

The PR moves successful auth-profile bookkeeping in `runEmbeddedPiAgent` into a fire-and-forget post-run helper with slow/failure logging and adds a runner regression test for nonblocking return.

## Best Possible Solution

Keep post-run auth success bookkeeping off the reply-blocking lane, but redact profile identifiers in the new logs and require a real Discord or gateway proof showing prompt reply delivery while slow bookkeeping is logged after the run.

## Reproduction Assessment

No live high-confidence reproduction was run in this review. The linked issue gives concrete v2026.5.20 Discord timing logs, and source inspection shows current main awaits `markAuthProfileSuccess` on the lane after `embedded run done`.

## Solution Assessment

Partly: moving best-effort success bookkeeping off the reply-blocking path is a plausible narrow fix. It still needs redacted logging and real Discord/gateway proof before it is the best merge-ready solution.
