## Summary

Keep open for maintainer review. Current main and v2026.6.9 still await `markAuthProfileSuccess` after embedded run completion, and PR head fixes the reply-ordering path with strong proof, but merge still needs an explicit auth/session-state durability decision.

## What This Changes

The PR moves post-run auth-profile success bookkeeping in the embedded agent runner from an awaited call to background logging and adds a regression test for nonblocking return ordering.

## Best Possible Solution

Land this latency fix only after maintainers explicitly accept the bounded stale-auth-metadata risk or require the background write to be tracked by a drainable lifecycle without re-blocking delivery.

## Reproduction Assessment

Yes at source level: current main and v2026.6.9 still await `markAuthProfileSuccess` immediately after embedded run completion, and lane completion waits for the embedded task to resolve. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

Acceptable mitigation, with one maintainer decision remaining. Moving the auth-success write off the reply path targets the shared embedded-runner layer, but maintainers must decide whether fire-and-forget persistence is acceptable or should be made drainable.
