- `P1`: The PR targets a reported real-user channel workflow where completed embedded runs could hold replies for tens of seconds.
- `merge-risk: 🚨 auth-provider`: The diff changes when auth-profile last-good and usage metadata are persisted relative to successful run completion.

## Summary

Keep open. The PR still addresses a current-main reply-blocking path, and the patch looks clean, but the remaining merge decision is whether maintainers accept moving auth-success metadata persistence after reply delivery.

## What This Changes

The branch moves post-run auth-profile success bookkeeping in the embedded runner to a caught background promise with redacted slow/failure logging and adds a regression test proving the run returns before that bookkeeping promise settles.

## Best Possible Solution

Land the nonblocking reply-path change only after maintainers accept the durability tradeoff, or route the write through a lifecycle-owned background task if shutdown durability is required.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main awaits `markAuthProfileSuccess` after `embedded run done` before returning the result, and the PR proof injects delayed bookkeeping to show the desired ordering.

## Solution Assessment

Yes, as an acceptable latency mitigation, but it needs maintainer acceptance of the durability tradeoff. A drainable background owner is the safer shape if owners require shutdown-time durability for this metadata.
