## Label Justifications

- `P1`: The PR targets a reported real-user Discord reply delay where completed embedded runs could hold delivery for tens of seconds.
- `merge-risk: 🚨 auth-provider`: The diff changes when auth-profile last-good and usage metadata are persisted relative to successful run completion.
- `merge-risk: 🚨 session-state`: The changed ordering can leave successful-run metadata stale if the process exits before the background write settles.

## Summary

Keep open for maintainer review: current main still awaits post-run auth-profile success bookkeeping after `embedded run done`, while this PR correctly moves that write out of the reply-blocking path. The patch has sufficient real behavior proof and no concrete line-level defect, but it intentionally changes auth/session-state durability timing and needs maintainer acceptance before merge.

## What This Changes

The PR makes embedded-run auth-profile success bookkeeping asynchronous after run completion, adds redacted slow/failure diagnostics, and adds regression coverage that the runner returns before that bookkeeping settles.

## Best Possible Solution

Land the nonblocking bookkeeping change after maintainers accept the auth-provider/session-state durability tradeoff, or route the write through a lifecycle-owned background task if shutdown durability is required.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main awaits post-run auth-success bookkeeping after embedded run completion, and delaying that call keeps the lane from finishing. The linked production report confirms the same reply-path shape with Discord delivery waiting tens of seconds after `embedded run done`.

## Solution Assessment

Yes, this is an acceptable latency mitigation because auth-success bookkeeping is non-critical to delivering the completed reply. If maintainers require shutdown-time durability for that metadata, the safer version is a drainable background owner rather than an untracked promise.
