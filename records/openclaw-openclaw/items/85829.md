## Label Justifications

- `P1`: The PR targets a reported real-user channel workflow where completed embedded runs could hold replies for tens of seconds.
- `merge-risk: 🚨 auth-provider`: The diff changes when auth-profile last-good and usage metadata are persisted relative to successful run completion.
- `merge-risk: 🚨 session-state`: The changed ordering can leave successful-run metadata stale if the process exits before the background write settles.

## Summary

Keep this PR open. Current main and the latest release still await post-run auth-profile success bookkeeping before the embedded runner returns payloads, and the merged stale-lock fix addresses an adjacent lock-recovery cause rather than this PR's reply-path ordering change.

## What This Changes

The PR makes post-run auth-profile success bookkeeping asynchronous after embedded run completion, adds redacted slow/failure diagnostics, and adds regression coverage for returning before that bookkeeping settles.

## Best Possible Solution

Keep the PR open for maintainer review and either accept the post-reply auth-success write or replace it with an equivalent nonblocking, drainable background owner.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path because `runEmbeddedAgent` awaits `markAuthProfileSuccess` after `embedded run done`, and the linked issue plus PR proof show that delayed bookkeeping can retain reply delivery.

## Solution Assessment

Yes for the reply-latency bug, with a product tradeoff. The stronger lifecycle design would keep the write nonblocking while making the background auth-state update drainable or otherwise owned during shutdown.
