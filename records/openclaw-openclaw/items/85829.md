- `P1`: The PR targets a real agent/channel workflow where completed embedded runs could hold replies for tens of seconds after model completion.
- `merge-risk: 🚨 auth-provider`: The diff changes when auth-profile last-good and usage metadata are persisted relative to successful run completion.

## Summary

Keep open for maintainer review: the PR is a focused, well-proven fix for reply-blocking post-run auth bookkeeping, but it intentionally changes auth-success metadata from awaited durability to background work.

## What This Changes

This PR moves embedded-runner post-run auth-profile success bookkeeping off the awaited reply path, adds slow/failure logging with redacted profile IDs, and adds regression coverage for nonblocking return ordering.

## Best Possible Solution

Land after maintainers accept the post-reply auth-success durability tradeoff, or revise it into lifecycle-owned nonblocking bookkeeping with observable drain/shutdown behavior.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path because runEmbeddedAgent awaits markAuthProfileSuccess after embedded run completion, and the issue comments plus PR proof show delayed bookkeeping can retain reply delivery.

## Solution Assessment

Unclear as a final merge shape. This is the narrowest latency fix for the reply-blocking bug, but lifecycle-owned background bookkeeping is safer if maintainers require auth-success metadata to be drainable before shutdown.
