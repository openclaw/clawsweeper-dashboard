## Summary

Keep open: current main and the latest release still await post-run auth-profile success bookkeeping, and this PR remains the direct candidate fix for the linked reply-latency bug. The patch looks correct, but it intentionally changes auth/session-state durability timing, so it should stay in maintainer review rather than be auto-cleaned up.

## What This Changes

The PR moves post-run auth-profile success bookkeeping in the embedded-agent runner off the awaited reply path, adds slow/failure logging with redacted profile IDs, and adds regression coverage for nonblocking return ordering.

## Best Possible Solution

Land the latency fix after maintainers either accept the bounded stale-auth-metadata risk or require the background auth-success write to be tracked by a drainable lifecycle without re-blocking reply delivery.

## Reproduction Assessment

Yes at source level: current main and v2026.6.10 await `markAuthProfileSuccess` immediately after `embedded run done`, and the contributor proof shows qa-channel delivery before delayed success bookkeeping finishes. I did not run a fresh live Discord reproduction in this read-only review.

## Solution Assessment

Acceptable mitigation, with one maintainer decision remaining. Targeting the shared embedded-runner auth bookkeeping is better than a Discord-only workaround, but maintainers must decide whether fire-and-forget persistence is acceptable or should be made drainable.
