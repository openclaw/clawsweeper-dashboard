## Label Justifications

- `P1`: The PR addresses a reported real-user Discord reply delay where completed turns can hold the lane for roughly 45-55 seconds before delivery.
- `merge-risk: 🚨 auth-provider`: The patch changes when auth-profile `lastGood` and usage stats are persisted after a successful provider run.
- `merge-risk: 🚨 session-state`: Backgrounding post-run bookkeeping can briefly leave session-adjacent auth state stale after the reply has already been returned.

## Summary

Keep open. This is a focused fix for a real reply-latency bug: current main still awaits auth-profile success bookkeeping after the embedded run, while the PR moves that bookkeeping off the reply path, catches/logs slow or failed writes, and now includes gateway-backed ordering proof.

## What This Changes

The PR changes `runEmbeddedPiAgent` to run post-success auth-profile bookkeeping asynchronously with redacted slow/failure diagnostics and adds regression coverage for returning before that bookkeeping resolves.

## Best Possible Solution

Land the focused async bookkeeping change after maintainers explicitly accept the short best-effort auth-state window and normal CI remains green.

## Reproduction Assessment

No live current-main reproduction was run in this review. Source inspection shows current main still awaits the locked auth-success write after `embedded run done`, and the linked report provides concrete v2026.5.20 timing logs for the observed lane retention.

## Solution Assessment

Yes. Moving the success write off the reply-blocking path is the narrow maintainable fix for this code path; a bounded drain is an optional durability addition if maintainers do not want best-effort post-reply auth-state persistence.
