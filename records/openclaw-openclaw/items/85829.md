## Label Justifications

- `P1`: The PR targets a reported real-user channel workflow where completed embedded runs could hold replies for tens of seconds.
- `merge-risk: 🚨 auth-provider`: The diff changes when auth-profile last-good and usage metadata are persisted relative to successful run completion.

## Summary

Keep open for maintainer review: the PR targets a real P1 reply-latency bug and the patch/proof are strong, but merging changes auth/session metadata durability timing, so maintainers need to accept that tradeoff or require a lifecycle-owned background write.

## What This Changes

The branch changes `runEmbeddedAgent` so post-run auth-profile success bookkeeping runs asynchronously with redacted slow/failure logging, and adds a regression test that the embedded run returns before the success promise resolves.

## Best Possible Solution

Land the nonblocking reply-path change only after maintainers accept post-reply auth-success bookkeeping, or route the write through a lifecycle-owned background task if shutdown durability is required.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path because `runEmbeddedAgent` awaits `markAuthProfileSuccess` after `embedded run done`, and the PR proof injects delayed bookkeeping to show delivery can complete first after the patch.

## Solution Assessment

Yes, as an acceptable latency mitigation, but it is not a no-tradeoff fix. A drainable background owner is the safer shape if maintainers require shutdown-time durability for this auth metadata.
