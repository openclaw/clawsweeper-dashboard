## Summary

Keep open. Current main and the latest release still await post-run auth-profile success bookkeeping after embedded-run completion, while this PR directly removes that reply-path wait with convincing gateway-backed proof; the remaining question is maintainer acceptance of the auth/session-state durability tradeoff.

## What This Changes

This PR moves embedded-runner post-run auth-profile success bookkeeping into background logging, redacts the new auth-profile diagnostics, and adds regression coverage for nonblocking return ordering.

## Best Possible Solution

Land this latency fix only after maintainers either accept the bounded stale-auth-metadata risk or require the background success write to be tracked by a restart/shutdown drain without re-blocking reply delivery.

## Reproduction Assessment

Yes at source level: current main and v2026.6.9 still await markAuthProfileSuccess immediately after embedded run completion, and command-queue completion waits for that task to resolve. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

Acceptable mitigation, with one maintainer decision remaining. Moving the auth-success write off the reply path targets the right shared runner layer, but maintainers must decide whether fire-and-forget persistence is acceptable or should be made drainable.
