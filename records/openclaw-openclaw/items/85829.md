## Label Justifications

- `P1`: The PR targets a real channel workflow where completed embedded runs could hold replies for tens of seconds after model completion.
- `merge-risk: 🚨 auth-provider`: The diff changes when auth-profile last-good and usage metadata are persisted relative to successful run completion.
- `merge-risk: 🚨 session-state`: The changed ordering can leave successful-run metadata stale if the process exits before the background write settles.

## Summary

Keep open for maintainer review: the patch is a strong narrow fix for the reply-blocking latency shape, but it intentionally moves auth-success durability after reply availability, so the remaining decision is an auth/session-state ordering tradeoff rather than a contributor-facing defect.

## What This Changes

The PR moves embedded-runner post-run auth-profile success bookkeeping from an awaited call to fire-and-forget work with slow/failure logging and regression coverage.

## Best Possible Solution

Land after maintainers accept the post-reply auth-success durability tradeoff, or revise it into a lifecycle-owned nonblocking background task with observable drain/shutdown behavior.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path because `runEmbeddedAgent` awaits `markAuthProfileSuccess` after embedded run completion, and the issue comments plus PR proof show delayed bookkeeping can retain reply delivery.

## Solution Assessment

Unclear as a final merge shape. This is the narrowest latency fix for the reply-blocking bug, but a lifecycle-owned background task is safer if maintainers require auth-success metadata to be drainable before shutdown.
