## Label Justifications

- `P1`: The PR targets a reported real-user Discord reply delay where completed embedded runs could hold delivery for tens of seconds.
- `merge-risk: 🚨 auth-provider`: The diff changes when auth-profile last-good and usage metadata are persisted relative to successful run completion.
- `merge-risk: 🚨 session-state`: The changed ordering can leave successful-run metadata stale if the process exits before the background write settles.

## Summary

Keep open for maintainer review: current main still awaits post-run auth-success bookkeeping before returning the embedded-run result, while this branch removes that reply-path wait and has credible proof. The remaining issue is an explicit auth-provider/session-state durability tradeoff, not a close reason or an automated repair task.

## What This Changes

The PR moves embedded-run auth-profile success bookkeeping out of the reply-blocking path, adds redacted slow/failure diagnostics, and adds regression coverage.

## Best Possible Solution

Land the small nonblocking bookkeeping change only after maintainers accept the auth-provider/session-state durability tradeoff, or route the write through a drainable background owner if shutdown durability is required.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main awaits post-run auth-success bookkeeping after embedded run completion, and the linked production report shows that lock-backed bookkeeping can hold Discord delivery for tens of seconds. I did not run tests because this review is read-only.

## Solution Assessment

Yes, as an acceptable latency mitigation, but it needs maintainer acceptance of the durability tradeoff. A drainable background task is the safer shape if owners require shutdown-time durability for this metadata.
