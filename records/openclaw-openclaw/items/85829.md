## Label Justifications

- `P1`: The PR targets a reported real-user channel workflow where completed embedded runs could hold replies for tens of seconds.
- `merge-risk: 🚨 auth-provider`: The diff changes when auth-profile last-good and usage metadata are persisted relative to successful run completion.

## Summary

Keep open: current main still awaits auth-success bookkeeping on the successful embedded-run path, so the linked reply-latency bug remains unsolved until this PR or an equivalent fix lands. The patch is focused and well-proven, but it needs maintainer acceptance of the auth/session metadata durability tradeoff before merge.

## What This Changes

The branch moves post-run auth-profile success bookkeeping off the embedded-run reply path, adds redacted slow/failure diagnostics, and adds regression coverage for nonblocking return behavior.

## Best Possible Solution

Land the nonblocking reply-path change only after maintainers accept post-reply auth-success bookkeeping, or route the write through a lifecycle-owned background task if shutdown durability is required.

## Reproduction Assessment

Yes. Current main awaits `markAuthProfileSuccess` after `embedded run done` before the result can reach `deliverAgentCommandResult`, and the PR proof injects delayed bookkeeping to show delivery can complete first.

## Solution Assessment

Yes, as an acceptable latency mitigation, but it is not a pure no-tradeoff fix. A drainable background owner is the safer shape if maintainers require shutdown-time durability for this auth metadata.
