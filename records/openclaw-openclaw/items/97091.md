## Label Justifications

- `P1`: The issue reports an agent-core workflow where a length-truncated assistant turn launched unintended side-effectful subagent work for nearly eight minutes.
- `impact:session-state`: The bug can commit and replay invalid assistant tool-call state, including dangling tool calls without matching intended tool results.
- `impact:other`: The issue also has concrete maintainer-visible runtime/cost impact from unintended tool execution that is not fully captured by the specific session-state label.

## Summary

Keep open: current main and v2026.6.10 still allow a final assistant message with toolCall blocks to reach execution without requiring stopReason === "toolUse", and the protected maintainer-labeled issue is already paired with an open fix PR.

## What This Changes

Prevent tool calls from length-truncated streamed assistant turns, such as partial sessions_spawn calls, from being committed, executed, or replayed as dangling tool state.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/97092, or an equivalent agent-core guard that executes tools only for toolUse turns and strips non-executable toolCall blocks before commit, message_end, and replay.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: return a final assistant message with stopReason "length" and a toolCall block, and current main routes it into executeToolCalls.

## Solution Assessment

Yes at the issue level. Agent-core owns tool execution and replay context, so the durable fix belongs there; provider-only scrubbing would be narrower and could miss other adapters that produce the same terminal shape.
