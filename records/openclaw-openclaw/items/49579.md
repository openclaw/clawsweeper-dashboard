## Summary

Keep open: the noise-suppression direction is useful, but the branch suppresses every WhatsApp disconnect banner, including the one current main queues for terminal close decisions; the PR also still lacks real WhatsApp behavior proof.

## What This Changes

The branch adds a `drainFormattedSystemEvents` prefix filter for WhatsApp connected/disconnected gateway banners and tests for suppressing those banners while preserving another WhatsApp system event.

## Best Possible Solution

Suppress only classified routine WhatsApp transport flaps, preferably at or near the WhatsApp close-decision source, while keeping logged-out, conflict, and max-attempt disconnect context prompt-visible and proving it in a real WhatsApp run.

## Reproduction Assessment

Yes by source inspection for the PR regression: the branch filters all `WhatsApp gateway disconnected` lines, while current main queues that line for every WhatsApp close decision before the detailed runtime-error branch. I did not run a live WhatsApp session in this read-only review.

## Solution Assessment

No. The proposed core prefix filter is too broad; the safer fix is to suppress only routine retryable flaps or source-specific connected banners while preserving terminal disconnect system events.
