## Summary

Keep open. The PR targets a real prompt-noise problem, but the current patch suppresses every WhatsApp disconnect banner even though current main queues that generic disconnect before it knows whether the close is retryable or terminal; terminal guidance is emitted through runtime errors rather than queued prompt context.

## What This Changes

The PR adds a `drainFormattedSystemEvents` filter for WhatsApp gateway connected/disconnected banners and focused tests for suppressing routine banners while preserving a separately queued WhatsApp logout event.

## Best Possible Solution

Keep the valid noise-suppression goal, but preserve terminal channel-health context: suppress routine connected banners, and either suppress disconnects only when the WhatsApp monitor has classified them as retryable/routine or enqueue distinct terminal system events before filtering the generic banner. Treat the broader system-event policy knob in #72043 as separate product work.

Yes. The source path is high-confidence: current main queues a generic WhatsApp disconnect before terminal-vs-retry handling, and the PR filter drops every line with that prefix, so logged-out/conflict/max-attempt-stop prompt context can be hidden.

## Solution Assessment

No. A blanket formatter-level disconnect filter is not the best fix because the formatter lacks the WhatsApp close decision; the safer fix belongs at the WhatsApp event source or should carry enough event classification to filter only routine transport flaps.
