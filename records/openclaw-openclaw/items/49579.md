## Summary

Keep open: the PR targets real WhatsApp prompt noise, but the current patch is not mergeable because it suppresses every WhatsApp disconnect system event, including terminal logged-out/conflict/max-attempt signals, and it has only test/typecheck validation rather than real behavior proof.

## What This Changes

The PR adds a `drainFormattedSystemEvents` filter for WhatsApp connected/disconnected gateway banners and focused tests for suppression/preservation behavior.

## Best Possible Solution

Keep the useful connected-banner suppression direction, but narrow disconnect suppression so terminal or recovery-significant WhatsApp outages still reach prompt system events, with regression coverage and real WhatsApp behavior proof.

## Reproduction Assessment

Yes for the PR regression by source inspection: the diff filters all `WhatsApp gateway disconnected` lines, while current main queues that exact line for every WhatsApp close decision before emitting detailed `runtime.error` guidance. I did not run a real WhatsApp session in this read-only review.

## Solution Assessment

No. The proposed core prefix filter is too broad; the safer fix is to classify only routine retryable flaps, or move/narrow the suppression near the WhatsApp close-decision source so critical disconnects remain prompt-visible.
