## Summary

Keep open. Current main still has the routine WhatsApp banner noise this PR is trying to address, but the proposed formatter filter is too broad because it suppresses all `WhatsApp gateway disconnected...` prompt events, including terminal close states whose detailed recovery guidance is currently logged through `runtime.error` rather than queued as system events. The PR head is also currently reported unmergeable against main, so it needs a narrower update before it can land.

## What This Changes

The PR adds a `drainFormattedSystemEvents` filter for WhatsApp gateway connected/disconnected banners and tests suppression of those banners while preserving a separately queued WhatsApp logged-out event.

## Best Possible Solution

Keep this work in the WhatsApp owner boundary: suppress routine connected banners, and suppress disconnected banners only when the monitor has proven the close is retryable/routine, or first enqueue distinct prompt-visible terminal events for logged-out, conflict, and max-attempt-stop paths. Add coverage for routine reconnect flaps and terminal outage paths, and treat #72043 as separate broader system-event policy work.

## Reproduction Assessment

Yes. On current main, the WhatsApp monitor queues `WhatsApp gateway disconnected...` and `drainFormattedSystemEvents` returns it as prompt-visible `System:` text; on the PR branch, the new `startsWith` filter drops that same line. The terminal close paths are reproducible from the code path where the generic disconnect is enqueued before `decision.action === "stop"`, while detailed guidance goes through `runtime.error`.

## Solution Assessment

No. The connected-banner suppression direction is narrow, but the proposed blanket disconnected-banner filter is not the best fix because it removes terminal outage context from prompt assembly. A safer implementation should filter only retryable/routine disconnects or add distinct queued terminal events before suppressing the generic banner.
