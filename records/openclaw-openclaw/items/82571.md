## Summary

Keep this PR open, but it is not mergeable as written: the proposed startup installer throws on the progress callback used by the shared install routine, and the gateway startup path can still filter the newly enabled codex plugin out for restrictive allowlist configs. The PR also lacks after-fix real gateway startup proof, so it needs contributor or maintainer follow-up before merge.

## What This Changes

The PR adds a gateway-startup helper that installs and enables the `codex` runtime plugin when configured OpenAI model refs require it, then calls that helper before gateway plugin loading.

## Best Possible Solution

Keep the startup install boundary, but make the silent prompter compatible with the shared installer and carry codex enablement into the activation source used for plugin planning/loading.

## Reproduction Assessment

Yes from source inspection: the missing-plugin path calls `ensureOnboardingPluginInstalled`, whose npm branch calls `prompter.progress`, while the new startup prompter throws for `progress`. The allowlist failure is also source-reproducible through `server.impl.ts` passing raw source config and `gateway-startup-plugin-ids.ts` enforcing activation-source allowlists.

## Solution Assessment

No; installing before plugin loading is the right place to solve the upgrade bug, but this implementation does not complete the install path and does not preserve codex activation for restrictive source configs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Implement a non-throwing startup progress prompter:** `src/commands/codex-runtime-plugin-install.ts:178`
  - body: The startup install path passes this prompter into `ensureOnboardingPluginInstalled`, whose npm install branch always calls `prompter.progress(...)` before downloading the package. Because this implementation throws for `progress`, the exact missing-plugin startup path this PR is trying to fix aborts before installing `@openclaw/codex`.
  - confidence: 0.96
- **[P2] Update the activation source after enabling codex:** `src/gateway/server-startup-plugins.ts:90`
  - body: Real gateway startup passes a raw `activationSourceConfig`, so this fallback keeps using the pre-install allowlist even after `cfgAtStart` has `codex` enabled. `canStartRequiredAgentHarnessPlugin` rejects required harness plugins missing from `activationSource.plugins.allow`, so allowlisted installs can still fail to load the codex harness and hit the original registered-harness error.
  - confidence: 0.92
