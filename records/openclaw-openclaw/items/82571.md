## Summary

Keep open: the PR addresses a real source-reproducible Codex harness startup gap and keeps Codex plugin-owned, but the latest head still misses activation-source policy in the pre-install check and lacks real gateway startup proof.

## What This Changes

The PR adds a gateway startup path that installs and enables the Codex runtime plugin when configured OpenAI/OpenAI Codex model refs require the Codex harness.

## Best Possible Solution

Fix the activation-source policy preflight, add regression coverage for source-config deny/disable cases, and land only after redacted real gateway startup proof shows Codex installs and loads successfully.

## Reproduction Assessment

Yes, source-reproducible. Current main has OpenAI/OpenAI Codex model refs routed toward the Codex harness while the installer is only reached from onboarding/model selection, and the PR-introduced activation-source policy gap follows directly from the startup/bootstrap contracts.

## Solution Assessment

No, not as currently patched. The intended startup-install direction is plausible, but it must honor both runtime and activation-source plugin policy before any install side effect and needs real gateway proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Check activation-source policy before installing Codex:** `src/commands/codex-runtime-plugin-install.ts:135`
  - body: Gateway startup passes the authored source config as a separate `activationSourceConfig`, and the startup planner enforces that source for `plugins.enabled`, `plugins.deny`, and `plugins.allow`. This helper only checks `params.cfg` before loading install records and potentially running the npm installer, so a runtime-overridden config can still install `@openclaw/codex` even when the activation source explicitly blocks Codex and will refuse to load it. Check both configs before any install side effect and add coverage for activation-source deny/disable cases.
  - confidence: 0.87
