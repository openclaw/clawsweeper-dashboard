## Summary

Keep open, but this PR is not mergeable as written: the new startup install path throws before it can install anything, can reuse a pre-install plugin metadata snapshot, and lacks after-fix real gateway proof. The broader startup auto-install direction also needs maintainer confirmation because related Codex runtime work has recently moved toward doctor repair and PI fallback instead of changing gateway startup behavior.

## What This Changes

The PR adds a gateway-startup helper that scans configured model refs, installs/enables `@openclaw/codex` when required, and calls it before gateway plugin loading.

## Best Possible Solution

Use the existing doctor/repair and runtime fallback direction unless maintainers explicitly approve startup auto-install; if approved, the implementation should provide a non-interactive progress reporter, invalidate/reload plugin metadata after install, and prove a real gateway no longer hits the missing-harness path.

## Reproduction Assessment

Yes for the PR defects by source inspection: the installer calls `prompter.progress(...)` before npm install, and the PR's progress implementation throws. The broader original 5.12 missing-harness path is partly superseded by merged doctor/fallback work, so I did not live-replay it in this read-only review.

## Solution Assessment

No. The PR chooses a startup auto-install strategy that is not yet product-confirmed, and the current implementation would fail before install or miss the newly installed plugin during the same startup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Return a real startup progress reporter:** `src/commands/codex-runtime-plugin-install.ts:179`
  - body: `ensureOnboardingPluginInstalled` reaches `installPluginFromNpmSpecWithProgress`, which calls `prompter.progress(...)` before invoking npm. This implementation throws for `progress`, so the first gateway start that actually needs to install `@openclaw/codex` will abort with the unexpected-prompt error instead of installing the plugin. Return a non-interactive `WizardProgress` that logs or ignores updates and supports `stop()`.
  - confidence: 0.96
- **[P2] Reload plugin metadata after installing Codex:** `src/gateway/server-startup-plugins.ts:91`
  - body: Normal gateway startup passes a pre-install `activationSourceConfig` and `pluginMetadataSnapshot` into this function. Because this line keeps the old activation source when it is provided, `loadPluginLookUpTable` can treat the old snapshot as compatible and build the startup plan without the newly installed Codex plugin, leaving the harness unregistered until a later restart. After installing/enabling Codex, use the updated activation config and force a fresh metadata/index lookup.
  - confidence: 0.88
