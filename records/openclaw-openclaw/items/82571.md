## Summary

Keep this PR open: it targets a real P1 gateway regression, but the current head still has two source-backed blockers and no after-fix real gateway proof.

## What This Changes

The PR adds a gateway startup path that auto-installs and enables the `codex` runtime plugin for configured OpenAI/OpenAI Codex model refs, plus focused mocked tests.

## Best Possible Solution

Keep Codex as a plugin-owned native harness, but make gateway startup install/enable it only when runtime and activation-source policy both allow activation, thread the adjusted activation source through deferred loads, and add redacted real startup proof.

## Reproduction Assessment

Yes, source-reproducible. Current main requires the codex harness for official OpenAI/OpenAI Codex refs but only installs it through onboarding/model-selection paths, and the PR blockers follow from the source paths for explicit plugin policy and deferred runtime loading.

## Solution Assessment

No, not yet. Startup installation before plugin loading is the right direction, but the patch must avoid package installs when policy blocks codex and must pass the adjusted activation source into the real deferred load path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Skip install when codex is policy-blocked:** `src/commands/codex-runtime-plugin-install.ts:132-145`
  - body: When `plugins.enabled` is false, `plugins.deny` contains `codex`, or the activation source blocks codex, `ensureCodexEnabledInStartupConfig` only logs and returns the original config. The caller still reaches the missing-record branch and can install `@openclaw/codex` at gateway startup even though the harness cannot activate. Check both config surfaces and return before loading/installing when explicit policy blocks codex.
  - confidence: 0.93
- **[P2] Thread the adjusted activation source into deferred loads:** `src/gateway/server-startup-plugins.ts:101`
  - body: The adjusted `activationSourceConfig` only affects planning inside `prepareGatewayPluginBootstrap`. The real server calls this with `loadRuntimePlugins: false` and later loads runtime plugins with the raw `startupActivationSourceConfig`, so an allowlist like `["telegram"]` can still disable codex and leave the harness unregistered. Return the adjusted activation source and use it for deferred load/reload paths.
  - confidence: 0.9
