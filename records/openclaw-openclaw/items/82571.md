## Summary

Keep open: the PR targets a real gateway regression, but the current patch still has source-reproducible blockers in the startup policy path and the real deferred plugin-load path, and it still lacks after-fix real gateway behavior proof.

## What This Changes

The PR adds a gateway startup helper that auto-installs and enables `@openclaw/codex` for configured OpenAI/OpenAI Codex model refs, wires it into gateway plugin bootstrap, and adds focused mocked tests.

## Best Possible Solution

Keep the startup-install boundary, but skip npm install when explicit plugin policy prevents codex activation, return/thread the codex-adjusted activation source into every later gateway plugin load, then add redacted real startup proof.

## Reproduction Assessment

Yes, source-reproducible. Current main requires the codex harness for official OpenAI/OpenAI Codex refs but only installs it through onboarding, and the PR's remaining failures follow from the source paths for explicit plugin policy and deferred runtime loading with the raw activation source.

## Solution Assessment

No. Installing before gateway plugin loading is the right shape, but this patch must respect explicit plugin disable/deny before any npm install and must carry the updated activation source through the real deferred loader.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Skip install when codex cannot be activated:** `src/commands/codex-runtime-plugin-install.ts:132-140`
  - body: When `plugins.enabled` is false or `plugins.deny` contains `codex`, `ensureCodexEnabledInStartupConfig` only logs and returns the original config, but the caller still falls through to the missing-record npm install. That can download/write `@openclaw/codex` at gateway startup even though enablement will fail and the harness cannot register. Return early before invoking the installer when explicit plugin policy blocks codex.
  - confidence: 0.9
- **[P2] Thread the updated activation source into runtime load:** `src/gateway/server-startup-plugins.ts:101`
  - body: This codex-adjusted `activationSourceConfig` only affects bootstrap planning when the real server passes `loadRuntimePlugins: false`. The later `loadStartupPlugins` callback still passes the raw startup source config, and `prepareGatewayPluginLoad` merges that raw `plugins.allow` back over the runtime config, so an allowlist like `["telegram"]` can still disable codex and leave the harness unregistered. Return the updated activation source and use it for the deferred load/reload paths.
  - confidence: 0.88
