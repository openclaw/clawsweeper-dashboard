## Summary

Keep this PR open. Current main does not already implement the proposed Bedrock auth/config surface, and the PR still has blocking schema/runtime alignment issues plus a credential-handling security concern before it is safe to merge.

## What This Changes

The PR adds Amazon Bedrock auth config types and normalization tests, expands the Bedrock plugin manifest config schema/uiHints, re-exports setup types, updates Bedrock docs and changelog, and bumps the bundled plugin version.

## Best Possible Solution

Land a coordinated Bedrock auth/config contract where manifest schema, raw uiHints, setup/onboarding behavior, docs, helper types, runtime consumption, and the pi-ai dependency all describe the same supported auth modes and fields.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: normalizeBedrockAuthConfig emits awsBedrockCustomSelected, while the PR manifest omits that property under additionalProperties:false and current plugin config validation rejects additional fields.

## Solution Assessment

No. The Bedrock plugin boundary is the right place for provider-owned config, but the current patch exposes and documents fields before the shipped runtime resolver consumes them and before the manifest contract safely round-trips secrets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Add the custom model fields to the manifest schema:** `extensions/amazon-bedrock/openclaw.plugin.json:61-62`
  - body: `BedrockAuthConfig` includes `awsBedrockCustomSelected` and `normalizeBedrockAuthConfig({})` always emits it, but the manifest keeps `additionalProperties: false` and omits both `awsBedrockCustomSelected` and `awsBedrockCustomModelBaseId`. A normalized `BedrockSetupOptions` value will fail plugin config validation on round-trip.
  - confidence: 0.91
- **[P1] Mark Bedrock credential hints as sensitive:** `extensions/amazon-bedrock/openclaw.plugin.json:104-123`
  - body: The new `awsBedrockApiKey`, `awsSecretKey`, and `awsSessionToken` hints describe credentials but omit `sensitive: true`. Some setup paths read raw plugin `configUiHints` and only skip unmasked prompts when this flag is explicit, so these fields can be prompted or handled like ordinary text.
  - confidence: 0.86
- **[P2] Do not advertise auth modes before runtime support exists:** `docs/providers/bedrock.md:356-365`
  - body: The docs say users can select `apikey`, `profile`, and `credentials`, but this PR does not bump the pi-ai package or wire those plugin config fields into OpenClaw's current Bedrock runtime path. Users configuring these manifest fields would see setup/docs promise modes that inference does not consume yet.
  - confidence: 0.83
- **[P3] Preserve explicit empty regions during normalization:** `extensions/amazon-bedrock/bedrock-auth-config.ts:91`
  - body: `awsRegion` uses `||` while the other defaults use `??`, so an explicit empty string is replaced with `us-east-1` instead of preserving the caller-provided value. Use nullish coalescing here for the same defaulting contract as the rest of the helper.
  - confidence: 0.72
