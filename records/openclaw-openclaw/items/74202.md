## Summary

Keep this PR open. Current main does not already implement the proposed Bedrock config surface, and the PR still has merge-blocking issues: its helper/schema contract does not round-trip, credential fields are surfaced without explicit sensitive hints, and the docs advertise auth modes before the shipped runtime/dependency path consumes them.

## What This Changes

The PR adds Bedrock auth config types and a normalizer, expands the Amazon Bedrock plugin manifest schema and UI hints, re-exports setup types, updates Bedrock docs and changelog, and bumps the bundled plugin version.

## Best Possible Solution

Land a coordinated Bedrock auth/config contract where the manifest schema, uiHints, setup/onboarding types, docs, runtime consumption, and pi-ai dependency all describe the same supported auth modes and fields.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: normalizeBedrockAuthConfig emits awsBedrockCustomSelected, while the PR manifest omits that property under additionalProperties:false and current plugin config validation rejects additional fields.

## Solution Assessment

No. The Bedrock plugin is the right ownership boundary, but this patch is not the narrowest maintainable solution until the schema, sensitive handling, docs, and runtime dependency support are aligned.

  - confidence: 0.86

  - body: The docs say users can select apikey, profile, and credentials modes, but this PR does not bump pi-ai or wire these plugin config fields into OpenClaw's current Bedrock runtime path. Users would see documented/setup-visible modes that inference does not consume yet.
  - confidence: 0.83
- **[P3] Preserve explicit empty regions during normalization:** `extensions/amazon-bedrock/bedrock-auth-config.ts:91`
  - body: awsRegion uses || while the other defaults use ??, so an explicit empty string is replaced with us-east-1 instead of preserving the caller-provided value. Use nullish coalescing for the same defaulting contract as the rest of the helper.
