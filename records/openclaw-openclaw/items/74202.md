## Summary

Keep open. Current main still lacks the proposed Bedrock auth-config helper and manifest surface, and the PR is not merge-ready because the helper/schema contract is inconsistent, credential fields are not marked sensitive, and the docs expose auth modes before the runtime/pi-ai resolver support is shipped.

## What This Changes

The PR adds Amazon Bedrock plugin config schema/uiHints for Cline-style auth options, a BedrockAuthConfig normalization helper with tests, setup-api type re-exports, Bedrock docs/changelog entries, and a private plugin version bump.

## Best Possible Solution

Land a coordinated Bedrock auth/config contract after the shipped pi-ai resolver is available: manifest schema, uiHints, helper types, docs, secret handling, runtime consumption, and tests should all describe the same supported fields and modes.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: normalizeBedrockAuthConfig emits awsBedrockCustomSelected, while the PR manifest keeps additionalProperties:false and omits that property, and current plugin config validation rejects additional properties.

## Solution Assessment

No. The PR is in the right Bedrock plugin boundary, but the current patch needs schema/type alignment, sensitive credential hints, and runtime/dependency coordination before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Add the custom model fields to the manifest schema:** `extensions/amazon-bedrock/openclaw.plugin.json:61-62`
  - body: `BedrockAuthConfig` makes `awsBedrockCustomSelected` a normal output field and `normalizeBedrockAuthConfig({})` always emits it, but the manifest still has `additionalProperties: false` and omits both `awsBedrockCustomSelected` and `awsBedrockCustomModelBaseId`. A normalized `BedrockSetupOptions` round-trip through plugin config validation is rejected as an additional property.
  - confidence: 0.91
- **[P1] Mark Bedrock credential hints as sensitive:** `extensions/amazon-bedrock/openclaw.plugin.json:104-123`
  - body: The new `awsBedrockApiKey`, `awsSecretKey`, and `awsSessionToken` uiHints describe credentials but do not set `sensitive: true`. Current setup/config UI behavior relies on that hint to avoid unmasked prompts and redact sensitive config values, so these fields can be handled as ordinary text.
  - confidence: 0.88
- **[P2] Gate the new auth modes on runtime support:** `docs/providers/bedrock.md:356-365`
  - body: The added docs say callers can choose `apikey`, `profile`, and `credentials`, but this PR does not bump the `@mariozechner/pi-ai` pin or wire these plugin config fields into OpenClaw's current Bedrock runtime/auth path, which still resolves auth from AWS env/default-chain auth. Users configuring these manifest fields would see setup/docs promise modes that current runtime does not consume.
  - confidence: 0.82
