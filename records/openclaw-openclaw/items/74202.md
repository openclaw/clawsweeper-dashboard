## Summary

Keep open; the PR is not merge-ready because the manifest schema cannot round-trip the new helper output, credential fields are exposed without sensitive hints, and the new auth modes are advertised before current runtime/pi-ai support is landed.

## What This Changes

The branch adds Bedrock auth-config types and normalization tests, expands the Amazon Bedrock plugin manifest schema/UI hints, re-exports setup types, updates Bedrock docs/changelog, and bumps the bundled plugin version.

## Best Possible Solution

Land a coordinated Bedrock auth contract where manifest schema, raw uiHints, setup/onboarding behavior, docs, helper types, runtime consumption, and the pi-ai dependency all describe the same supported auth modes and fields.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: normalizeBedrockAuthConfig emits awsBedrockCustomSelected, while the PR manifest omits that property under additionalProperties:false and current plugin config validation rejects additional fields.

## Solution Assessment

No. The Bedrock plugin boundary is the right place for provider-owned config, but this patch should wait for schema parity, credential-sensitive hints, and runtime/dependency support or an explicit maintainer decision to stage inert metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Add the custom model fields to the manifest schema:** `extensions/amazon-bedrock/openclaw.plugin.json:61`
  - body: `BedrockAuthConfig` includes `awsBedrockCustomSelected` and `normalizeBedrockAuthConfig({})` always emits it, but the manifest keeps `additionalProperties: false` and omits both `awsBedrockCustomSelected` and `awsBedrockCustomModelBaseId`. A normalized `BedrockSetupOptions` value will fail plugin config validation on round-trip.
  - confidence: 0.91
- **[P1] Mark Bedrock credential hints as sensitive:** `extensions/amazon-bedrock/openclaw.plugin.json:104-123`
  - body: The new `awsBedrockApiKey`, `awsAccessKey`, `awsSecretKey`, and `awsSessionToken` hints describe credential-bearing values but omit `sensitive: true`. Setup paths read raw plugin `configUiHints` and only skip unmasked prompts when this flag is explicit, so these can be handled like ordinary text.
  - confidence: 0.86
- **[P2] Do not expose auth modes before runtime support lands:** `extensions/amazon-bedrock/openclaw.plugin.json:41-45`
  - body: Adding `awsAuthentication` to the manifest makes setup/onboarding advertise `apikey`, `profile`, and `credentials`, but current main still resolves Amazon Bedrock through the AWS SDK default chain and this PR does not bump or wire the pending pi-ai resolver. Users can store config values that inference will not consume.
  - confidence: 0.83
- **[P3] Preserve explicit empty regions during normalization:** `extensions/amazon-bedrock/bedrock-auth-config.ts:91`
  - body: `awsRegion` uses `||` while the other defaulted fields use `??`, so an explicit empty string is replaced with `us-east-1` instead of preserving the caller-provided value. Use nullish coalescing for the same defaulting contract as the rest of the helper.
  - confidence: 0.72
