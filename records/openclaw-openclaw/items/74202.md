## Summary

Keep open. The PR is useful source work, and the earlier schema concern is real, but the patch is not mergeable yet because the proposed manifest/helper contract is internally inconsistent, the new credential fields are exposed to the unmasked setup wizard, and the referenced pi-ai resolver PR is closed unmerged while current pi-ai still lacks these option names.

## What This Changes

The PR adds Bedrock auth normalization types/tests, setup-api type exports, manifest configSchema/uiHints, docs/changelog entries, and a private amazon-bedrock plugin version bump for Cline-style Bedrock auth fields.

## Best Possible Solution

Land this only as a coordinated Bedrock contract: the manifest schema, helper output, setup uiHints, docs, tests, and the shipped pi-ai resolver should agree on the same fields and secret handling, or the user-facing docs/config surface should stay hidden until runtime support ships.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: normalizeBedrockAuthConfig emits awsBedrockCustomSelected, the PR manifest omits it while additionalProperties is false, and current plugin config validation rejects undeclared properties.

## Solution Assessment

No. The owner boundary is right for plugin-owned metadata, but this patch needs schema/helper alignment, sensitive prompt handling, and a real shipped pi-ai runtime contract before exposing these auth modes to users.
