## Summary

Keep open. Current main does not implement this Bedrock config surface, and the PR still has blocking contract, security, and runtime-support gaps: the normalizer output does not validate against the strict manifest schema, credential fields are not declared sensitive in the manifest metadata, and the docs describe auth modes that current OpenClaw runtime and the pinned pi-ai dependency do not yet consume.

## What This Changes

The PR adds Bedrock auth config types and a normalizer with tests, expands the amazon-bedrock manifest schema and uiHints plus docs for Cline-style auth fields, re-exports BedrockSetupOptions, adds changelog entries, and bumps the private plugin package version.

## Best Possible Solution

Land a coordinated Bedrock plugin contract only when schema, uiHints, helper types, docs, tests, and the shipped runtime dependency all describe the same supported auth modes and fields. If the OpenClaw side must land first, keep it explicitly schema-only and do not document the modes as usable runtime auth until the pi-ai resolver is available.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: normalizeBedrockAuthConfig({}) emits awsBedrockCustomSelected, while the PR manifest has additionalProperties:false and omits that property, and current plugin validation rejects additional properties.

## Solution Assessment

No. The PR is in the right amazon-bedrock plugin boundary, but the current patch is not the narrow maintainable solution until the manifest schema matches the exported type, credential metadata is sensitive, and docs/runtime support are synchronized with the pi-ai resolver.
