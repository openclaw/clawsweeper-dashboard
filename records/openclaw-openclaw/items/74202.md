## Summary

Keep open. The PR is in the right Bedrock plugin boundary, but the head should not merge as-is: the helper/schema contract is inconsistent, credential config hints are not marked sensitive for setup prompts, and the docs expose auth modes that current OpenClaw/pi-ai runtime does not yet consume.

## What This Changes

The PR adds Amazon Bedrock plugin config schema and UI hints for Cline-style auth options, a BedrockAuthConfig normalization helper with tests, setup-api type re-exports, Bedrock docs, changelog entries, and a private plugin version bump.

## Best Possible Solution

Land a coordinated Bedrock auth/config contract only after schema, uiHints, secret handling, docs, runtime consumption, and the pi-ai dependency agree on the same supported fields and modes.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: normalizeBedrockAuthConfig emits awsBedrockCustomSelected, while the PR manifest keeps additionalProperties:false and omits that property, and current plugin config validation rejects additional properties.

## Solution Assessment

No. The PR belongs in the Bedrock plugin area, but the current patch should either stay type-only and non-user-facing or wait until the runtime/dependency resolver is shipped and the manifest/secret UX matches it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Add the custom model fields to the manifest schema:** `extensions/amazon-bedrock/openclaw.plugin.json:61`
  - body: `BedrockAuthConfig` includes `awsBedrockCustomSelected` and `awsBedrockCustomModelBaseId`, and `normalizeBedrockAuthConfig({})` always emits `awsBedrockCustomSelected`. Since the manifest still has `additionalProperties: false` and omits both fields, a normalized `BedrockSetupOptions` object fails plugin config validation on round-trip.
  - confidence: 0.91
- **[P1] Mark Bedrock credential hints as sensitive:** `extensions/amazon-bedrock/openclaw.plugin.json:104-123`
  - body: The new `awsBedrockApiKey`, `awsSecretKey`, and `awsSessionToken` fields are credential-bearing, but their `uiHints` omit `sensitive: true`. The setup wizard only skips plain text prompts for fields marked sensitive, so these credentials can be prompted as ordinary text.
  - confidence: 0.88
- **[P2] Gate the new auth modes on runtime support:** `docs/providers/bedrock.md:355-365`
  - body: The docs say users can choose the new Bedrock auth modes, but this PR does not wire these plugin config keys into the current Bedrock runtime, and the pinned `@mariozechner/pi-ai` package does not expose the proposed `awsAuthentication`/static-credentials resolver contract. Users setting these fields would get accepted config that current Bedrock calls ignore.
  - confidence: 0.84
