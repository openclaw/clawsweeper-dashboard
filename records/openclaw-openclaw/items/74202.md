## Summary

Keep open. The PR is useful Bedrock owner-surface work, but the proposed branch still has a manifest/type mismatch and documents selectable auth modes that current OpenClaw runtime and the pinned pi-ai dependency do not consume yet.

## What This Changes

The PR adds Bedrock auth config types/normalization tests, expands the amazon-bedrock plugin manifest and UI hints, exports setup types, updates Bedrock docs/changelog, and bumps the plugin package version.

## Best Possible Solution

Land a coordinated Bedrock contract where manifest schema, setup types, docs, runtime consumption, tests, and the pi-ai dependency all describe the same supported auth modes and fields.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: normalize a Bedrock config at PR head, then validate that result against the PR manifest schema; `awsBedrockCustomSelected` is emitted but not allowed by the strict schema.

## Solution Assessment

No. The Bedrock plugin is the right ownership boundary, but this patch is not the best merge shape until schema parity and runtime/docs support are aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Add the custom model fields to the schema:** `extensions/amazon-bedrock/openclaw.plugin.json:61-62`
  - body: `BedrockAuthConfig` includes `awsBedrockCustomSelected` and `awsBedrockCustomModelBaseId`, and `normalizeBedrockAuthConfig({})` always emits `awsBedrockCustomSelected`. Because the manifest keeps `additionalProperties:false` but omits both properties, a normalized `BedrockSetupOptions` value fails plugin config validation on round-trip.
  - confidence: 0.93
- **[P2] Do not document auth modes before runtime support exists:** `docs/providers/bedrock.md:356-365`
  - body: The new docs say users can pick `apikey`, `profile`, and `credentials` plugin auth modes, but this PR does not bump pi-ai or wire these plugin config fields into OpenClaw's current Bedrock runtime path. Users would see documented/setup-visible modes that inference still ignores.
  - confidence: 0.86
- **[P3] Preserve explicit empty regions during normalization:** `extensions/amazon-bedrock/bedrock-auth-config.ts:91`
  - body: `awsRegion` uses `||` while the other defaults use `??`, so an explicit empty string is replaced with `us-east-1` instead of preserving the caller-provided value. Use nullish coalescing for the same defaulting contract as the rest of the helper.
  - confidence: 0.72
