## Summary

Keep open. The branch is useful Bedrock plugin work, but current head still has merge-blocking schema parity, credential-sensitivity, and runtime/docs alignment issues confirmed against current main and the PR diff.

## What This Changes

The PR adds Amazon Bedrock auth config types and normalization, manifest schema/UI hints, setup type exports, docs/changelog entries, tests, and a bundled plugin version bump.

## Best Possible Solution

Land a coordinated Bedrock auth/config contract where manifest schema, uiHints or secret contracts, setup types, docs, runtime consumption, pi-ai dependency support, and tests all describe the same supported fields.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: normalizeBedrockAuthConfig emits awsBedrockCustomSelected, the PR manifest omits that property under additionalProperties:false, and current plugin config validation uses AJV without removing additional fields.

## Solution Assessment

No. The Bedrock plugin is the right ownership boundary, but this patch is not the best mergeable solution until schema parity, credential sensitivity, documentation, and runtime dependency support are aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Add the custom model fields to the manifest schema:** `extensions/amazon-bedrock/openclaw.plugin.json:61`
  - body: BedrockAuthConfig includes awsBedrockCustomSelected and awsBedrockCustomModelBaseId, and normalizeBedrockAuthConfig({}) always returns awsBedrockCustomSelected. Since the manifest still has additionalProperties:false and omits both keys, any normalized BedrockSetupOptions round-trip through plugin config validation is rejected.
  - confidence: 0.92
- **[P1] Mark Bedrock credential hints as sensitive:** `extensions/amazon-bedrock/openclaw.plugin.json:104-122`
  - body: The new awsBedrockApiKey, awsSecretKey, and awsSessionToken uiHints are plain text hints. The setup wizard skips masked prompting only when hint.sensitive is true, and UI raw-diff redaction also consults sensitive hints, so these credential fields need explicit sensitive handling before they are exposed.
  - confidence: 0.87
- **[P2] Do not advertise auth modes before runtime support exists:** `docs/providers/bedrock.md:355-365`
  - body: The new section tells users and downstream callers they can select apikey, profile, and credentials modes, but this patch does not bump pi-ai or wire plugin config into current Bedrock runtime auth. Current main still treats Bedrock as AWS SDK default-chain auth, so these documented modes are setup-visible but not consumed by inference.
  - confidence: 0.84
- **[P3] Preserve explicit empty regions during normalization:** `extensions/amazon-bedrock/bedrock-auth-config.ts:91`
  - body: awsRegion is the only defaulted field using ||, so an explicitly supplied empty string is rewritten to us-east-1 while neighboring defaults preserve explicit values via ??. Use nullish coalescing if the helper's contract is to default only nullish inputs.
  - confidence: 0.68

## Security Review

Status: needs_attention

Summary: The diff introduces AWS credential-bearing config fields without explicit sensitive handling in the manifest.

Concerns:

- **[medium] Credential fields are not marked sensitive:** `extensions/amazon-bedrock/openclaw.plugin.json:104`
  - body: awsBedrockApiKey, awsSecretKey, and awsSessionToken can carry bearer tokens, AWS secret keys, and STS session tokens, but their new uiHints omit sensitive:true; current setup and config reveal/redaction paths use that hint to avoid treating secrets as ordinary text.
