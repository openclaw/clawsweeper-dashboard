## Summary

Keep open: the intended missing-cwd recovery is valid and has real log proof, but the patch still passes the persisted cwd through the outer dispatch catch, so an `acp.allowedAgents` policy rejection can delete bound-conversation state before runtime init. A narrow repair is needed before maintainers decide on the remaining session-state semantics.

## What This Changes

The PR adds a filesystem-backed missing-cwd predicate to ACP dispatch stale-session cleanup and tests that generic ACP init failures unbind only when the persisted runtime cwd is gone.

## Best Possible Solution

Keep the missing-cwd recovery in ACP dispatch, restrict it to runtime init failures, and preserve policy rejections as non-destructive before maintainer acceptance of targetSessionKey-wide cleanup.

## Reproduction Assessment

Yes. Source inspection shows current main lacks the cwd predicate, and the provided staging ACP log shows the after-fix generic init-failure path removing a stale binding; I did not rerun tests because this was a read-only review.

## Solution Assessment

No. The fix belongs in dispatch stale-binding cleanup, but cwd-based unbind must be gated to runtime init or `runTurn` failures before the remaining session-state semantics are safe for maintainer acceptance.
