## Summary

Keep open: the missing-cwd recovery is useful and has real behavior proof, but the patch is still incorrect because the new cwd-based unbind can also fire for pre-runtime `acp.allowedAgents` policy rejection before any runtime init was attempted.

## What This Changes

The PR extends ACP dispatch stale-binding cleanup so a generic ACP session init failure can unbind a bound conversation when the persisted session cwd is missing.

## Best Possible Solution

Land a narrowed dispatch-only fix that uses the missing persisted cwd signal only for actual ACP runtime initialization failures, preserves policy-rejection bindings, and has maintainer acceptance for target-session-wide cleanup semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence introduced failure path: a ready ACP session with missing persisted cwd and a disallowed `acp.allowedAgents` policy throws `ACP_SESSION_INIT_FAILED` before `runTurn`, then the PR's broad catch can unbind it.

## Solution Assessment

No. The missing-cwd recovery is the right direction, but the cwd signal should be limited to actual runtime initialization failures rather than all `ACP_SESSION_INIT_FAILED` errors caught by dispatch.
