## Summary

Keep open: the bug is real and the branch has useful proof, but the current patch can remove persisted bindings for an `acp.allowedAgents` policy rejection before any ACP runtime init, so it needs a narrow repair and maintainer acceptance of the remaining session-state semantics.

## What This Changes

The PR extends ACP dispatch stale-binding cleanup so a generic `ACP_SESSION_INIT_FAILED` can unbind a bound conversation when the persisted ACP runtime `cwd` is filesystem-confirmed missing, with dispatch regression tests.

## Best Possible Solution

Keep the missing-cwd recovery in ACP dispatch, restrict cwd-based unbinding to actual runtime init or `runTurn` failures, and then get maintainer acceptance for targetSessionKey-wide stale cleanup.

## Reproduction Assessment

Yes, from source. Current main lacks the missing-cwd predicate while the PR body shows an after-fix staging ACP run where a generic init failure removed the stale binding; I did not run tests because this review is read-only.

## Solution Assessment

No. The dispatch cleanup is the right general owner boundary, but the current patch passes the cwd signal through an outer catch that also handles pre-runtime policy failures; the safe fix gates cwd-based unbind to runtime init or `runTurn` failures and adds the missing regression.
