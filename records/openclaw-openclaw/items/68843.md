## Summary

Keep open: the missing-cwd recovery is useful, but the current patch can delete bound-conversation state for a pre-runtime policy rejection rather than a stale runtime session, and the remaining deletion semantics need maintainer acceptance.

## What This Changes

The branch adds a persisted ACP session cwd existence check to dispatch stale-session cleanup and tests generic init failures with missing or existing cwd metadata.

## Best Possible Solution

Land a narrowed dispatch-only fix that uses the missing-cwd signal only for actual ACP runtime initialization failures, preserves policy-rejection bindings, and records maintainer acceptance for the remaining session-state semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence PR-introduced path: a ready ACP session with a missing persisted cwd and disallowed acp.allowedAgents throws ACP_SESSION_INIT_FAILED before runTurn, then the PR catch can unbind it.

## Solution Assessment

No. The missing-cwd recovery is the right direction, but the cwd signal should be limited to actual runtime initialization failures and backed by a policy-rejection negative regression.
