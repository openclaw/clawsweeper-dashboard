## Summary

Keep open. The intended missing-cwd recovery is useful and has real after-fix proof, but the current patch can still delete persisted bound-conversation state for a pre-runtime `acp.allowedAgents` policy rejection because that path also throws `ACP_SESSION_INIT_FAILED` before `runTurn` starts.

## What This Changes

The PR adds a missing persisted ACP cwd check to dispatch stale-session cleanup and tests generic ACP init failures with missing versus existing cwd.

## Best Possible Solution

Land a narrowed dispatch-only fix that uses missing persisted cwd only for actual ACP runtime init failures, preserves policy-rejection bindings, and has maintainer acceptance for target-session-wide cleanup semantics.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main lacks the missing-cwd stale predicate, and the PR proof shows the intended after-fix generic init-failure cleanup. The introduced blocker is also source-reproducible with a ready ACP session whose persisted cwd is missing and `cfg.acp.allowedAgents` rejects the agent before `runTurn`.

## Solution Assessment

No, this is not the best fix as-is. The cwd probe should be limited to actual ACP runtime initialization failures rather than the whole dispatch catch block, then maintainers can decide whether the remaining session-state cleanup semantics are acceptable.
