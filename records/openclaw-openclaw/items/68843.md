## Summary

Keep open: current main still lacks the persisted-cwd stale-session signal, and this PR is a narrow, source-supported fix with sufficient live proof; the remaining blocker is maintainer acceptance of automatic persisted binding removal.

## What This Changes

The PR changes ACP auto-reply dispatch to unbind bound conversations after a generic ACP session-init failure when the persisted session cwd is missing on disk, with focused regression tests.

## Best Possible Solution

Land the narrow dispatch cleanup after an owner accepts the session-state tradeoff, or revise the cleanup policy to add a retry/grace period or narrower removal scope if that risk is not acceptable.

## Reproduction Assessment

Yes by source and supplied proof: a bound ACP session with a missing persisted cwd can hit generic ACP_SESSION_INIT_FAILED, and current main does not inspect cwd in the cleanup classifier. I did not run a local live gateway reproduction during this read-only review.

## Solution Assessment

Yes on implementation shape if maintainers accept the semantics: it reuses the existing dispatch stale-binding cleanup and the existing ACP cwd resolver rather than adding a parallel cleanup path. The only unresolved question is policy, not a concrete code defect.
