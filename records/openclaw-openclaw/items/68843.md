## Summary

Keep open. Current main still lacks the generic ACP init-failure plus persisted-cwd stale-binding cleanup, and the rebased PR now applies that narrow fix cleanly; the remaining gate is explicit maintainer acceptance of the session-state semantics, not a code blocker.

## What This Changes

The PR expands ACP dispatch stale-binding cleanup so a generic ACP_SESSION_INIT_FAILED unbinds a bound conversation when the persisted ACP session cwd is filesystem-confirmed missing, with regression tests for missing and existing cwd cases.

## Best Possible Solution

Merge the narrow dispatch-side missing-cwd cleanup only after a maintainer accepts the session-state semantics; otherwise narrow the stale-session signal or binding-removal scope before merge.

## Reproduction Assessment

Yes, at source level: current main only classifies selected ACP init-failure messages as stale and does not inspect persisted cwd on a generic ACP_SESSION_INIT_FAILED. The PR also supplies live after-fix proof showing the stale binding count drop to 0 after the generic init failure with missing cwd.

## Solution Assessment

Yes, with maintainer acceptance: the dispatch catch is the directly implicated recovery point, and using the existing ACP-core cwd resolver keeps the fix narrow. A manager-layer cleanup would be broader because generic init failures also participate in backend handle creation and retry behavior.
