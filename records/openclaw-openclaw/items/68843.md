## Summary

Keep open: the PR addresses a real ACP stale-bound-session bug and current main does not already contain the missing-cwd recovery, but the branch is not merge-ready because the new cwd-based stale classifier can delete bindings for pre-runtime policy failures.

## What This Changes

The PR extends ACP dispatch cleanup to unbind stale bound conversations on generic ACP session init failure when the persisted ACP cwd is missing, and adds regression tests for missing and existing cwd cases.

## Best Possible Solution

Keep the missing-cwd recovery, but confine cwd-based unbinding to runtime-init failures after the runtime path is actually attempted and preserve bindings on policy/config rejections.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the PR-introduced blocker: ready ACP session metadata with a missing cwd plus disallowed cfg.acp.allowedAgents throws ACP_SESSION_INIT_FAILED before runTurn and then reaches the new cwd-based cleanup.

## Solution Assessment

No. The recovery direction is right, but the current catch makes the cwd signal too broad; the best fix is to limit it to actual runtime-init/runTurn failures and add the policy-rejection negative test.
