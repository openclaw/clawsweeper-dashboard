## Summary

Keep open: the intended missing-cwd recovery is useful, but the current patch can delete bound-conversation state for a pre-runtime ACP agent-policy rejection when old session metadata has a missing cwd.

## What This Changes

The PR broadens ACP dispatch stale-session cleanup to unbind a bound session after generic ACP init failure when the persisted session cwd is filesystem-confirmed missing, with regression tests for missing and existing cwd cases.

## Best Possible Solution

Keep the dispatch-only missing-cwd recovery, but gate the cwd-based stale classification to actual ACP runtime-init/runTurn failures, preserve policy-rejection bindings, and add the missing negative regression.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the PR-introduced blocker: ready ACP session metadata with a missing cwd plus disallowed cfg.acp.allowedAgents throws ACP_SESSION_INIT_FAILED before runTurn and then reaches the new cwd-based cleanup.

## Solution Assessment

No. The recovery direction is right, but the current catch makes the cwd signal too broad; the best fix is to limit it to actual runtime-init/runTurn failures and add the policy-rejection negative test.
