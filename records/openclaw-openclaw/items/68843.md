## Summary

Keep open for maintainer review: current main and the latest release still lack the missing-cwd stale-binding classification, and the patch, tests, and linked proof support the fix, but merging intentionally changes persisted session-binding cleanup semantics.

## What This Changes

The branch extends ACP dispatch stale-session cleanup to unbind bound conversations after a generic ACP_SESSION_INIT_FAILED when the persisted ACP runtime cwd is filesystem-confirmed missing, with focused regression tests.

## Best Possible Solution

Land this after an ACP/session owner explicitly accepts the missing-cwd unbind semantics and targetSessionKey-wide cleanup scope, or narrow the cleanup if that state-loss tradeoff is not acceptable.

## Reproduction Assessment

Yes, at source/proof level: current main and v2026.6.8 classify stale ACP init failures by message only, while the linked staging log shows the patched path removing the stale binding after a generic ACP_SESSION_INIT_FAILED. I did not rerun the gateway locally because this was a read-only review.

## Solution Assessment

Yes for the code shape: dispatch already owns bound-conversation cleanup, and resolveAcpSessionCwd is the canonical persisted-cwd helper. The safer alternative, if maintainers reject the state tradeoff, is to narrow the unbind scope rather than keep message-only classification.
