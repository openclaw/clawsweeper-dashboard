## Summary

Keep open: the missing-cwd recovery is a useful bug fix with real proof, but the current diff passes the cwd stale signal through a broad catch that also handles pre-runtime `ACP_SESSION_INIT_FAILED` policy errors. That can remove persisted bindings for a policy/config failure rather than a dead ACP runtime.

## What This Changes

The PR adds a persisted-cwd filesystem check to ACP dispatch stale-session cleanup and tests generic `ACP_SESSION_INIT_FAILED` with missing versus existing cwd metadata.

## Best Possible Solution

Keep the missing-cwd recovery, but activate it only for actual ACP runtime initialization failures after dispatch has reached `runTurn`; preserve bindings for policy/config failures and cover both paths with focused tests.

## Reproduction Assessment

Yes. Source inspection shows current main only message-classifies stale ACP init errors, and the supplied redacted live log shows the intended missing-cwd path removing the stale binding after a generic `ACP_SESSION_INIT_FAILED`; I did not run local tests in this read-only review.

## Solution Assessment

No, not yet. The fix direction is right, but the implementation is too broad because it lets pre-runtime policy errors use the missing-cwd cleanup path; the safer solution is a runtime-init-only gate plus a policy-denial regression.
