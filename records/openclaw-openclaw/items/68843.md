## Summary

This PR should stay open, but it needs changes before merge: the intended missing-cwd ACP recovery is useful, yet the current catch can delete persisted bindings for a pre-runtime policy rejection rather than a stale runtime session.

## What This Changes

The PR extends ACP dispatch stale-session cleanup to remove bound conversations after generic `ACP_SESSION_INIT_FAILED` when the persisted ACP session cwd is missing, with regression tests for missing and existing cwd metadata.

## Best Possible Solution

Keep the dispatch-only missing-cwd recovery, but gate cwd-based stale cleanup to actual ACP runtime-init or `runTurn` failures, preserve policy-rejection bindings, and add the missing negative regression.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the PR-introduced blocker: ready ACP session metadata with a missing cwd plus disallowed `cfg.acp.allowedAgents` throws `ACP_SESSION_INIT_FAILED` before `runTurn` and then reaches the new cwd-based cleanup.

## Solution Assessment

No. The recovery direction is right, but the current catch makes the cwd signal too broad; the best fix is to limit it to actual runtime-init or `runTurn` failures and add the policy-rejection negative test.
