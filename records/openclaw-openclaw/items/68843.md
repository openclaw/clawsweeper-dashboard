## Summary

Keep open: the PR fixes a real ACP stale-binding gap and has sufficient live proof for the intended missing-cwd path, but the current patch is still too broad because the new cwd-based unbind signal is evaluated in the outer catch that also handles pre-runtime policy failures.

## What This Changes

The PR extends ACP dispatch stale-binding cleanup to unbind a bound conversation when a generic `ACP_SESSION_INIT_FAILED` coincides with a missing persisted runtime cwd, with regression tests for missing and existing cwd.

## Best Possible Solution

Keep the missing-cwd recovery, but limit cwd-based unbinding to errors from `acpManager.runTurn` or runtime initialization and require maintainer acceptance of the remaining target-session-wide unbind semantics.

## Reproduction Assessment

Yes. Source inspection shows current main only message-classifies stale ACP init errors, and the supplied live staging log shows the intended generic init-failure plus missing-cwd path; the remaining over-unbind bug is source-reproducible from the shared outer catch and policy error code.

## Solution Assessment

No, not yet. The fix direction is right, but the implementation is too broad because pre-runtime policy errors can use the missing-cwd cleanup path; the safer solution is a runtime-init-only gate plus a policy-denial regression.
