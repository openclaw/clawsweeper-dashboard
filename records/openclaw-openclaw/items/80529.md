## Summary

## What This Changes

The PR relaxes Telegram's grammY network-request regex so bare `Network request for ... failed!` errors become recoverable in send context and updates the classifier test expectation.

## Best Possible Solution

Keep the safe-send boundary and land a narrower fix only when it keys recovery to nested `HttpError.error`, cause, or code evidence, or when maintainers explicitly approve broader retry semantics with real Telegram proof.

## Reproduction Assessment

Yes for the source-level classifier path: current main explicitly expects a bare grammY network envelope to be false in send context. I did not run a live low-resource Telegram transport reproduction in this read-only review.

## Solution Assessment

No; the regex broadening is broader than the current safe-send boundary and does not target the non-idempotent final send wrapper. The safer fix is to key recovery to nested `HttpError.error`, cause, or code evidence, or get explicit maintainer approval for broader retry semantics.
