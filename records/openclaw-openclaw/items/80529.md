## Summary

## What This Changes

The PR relaxes Telegram's grammY network-request regex so bare `Network request for ... failed!` errors become recoverable in send context and updates the classifier tests.

## Best Possible Solution

Keep the safe-send boundary and land a narrow fix only when it uses dependency-backed nested pre-connect evidence or maintainer-approved broader retry semantics, with redacted live Telegram proof.

## Reproduction Assessment

Yes for the source-level classifier path: current main explicitly expects a bare grammY network envelope to be false in send context. I did not run a live low-resource Telegram transport reproduction in this read-only review.

## Solution Assessment

No; the regex broadening is broader than the safest existing boundary. The better fix is to keep plain bare envelopes guarded unless nested `HttpError.error`, cause, or code evidence proves a pre-connect failure, with live Telegram proof.
