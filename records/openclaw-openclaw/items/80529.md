Keep open: the PR targets a real Telegram retry gap, but the patch broadens the generic send-context regex past the current strict safe-send boundary and still lacks real behavior proof.

## What This Changes

The PR relaxes Telegram's grammY network-request regex and updates the classifier test so bare `Network request for ... failed!` errors become recoverable in send context.

## Best Possible Solution

Use a maintainer-approved narrow retry boundary, preferably nested pre-connect evidence in `isSafeToRetrySendError`, and require redacted real Telegram or runtime-log proof before merge.

## Reproduction Assessment

Yes for the source-level classifier path: current main tests and source show a bare grammY network envelope returns false in send context. I did not run a live Telegram transport reproduction in this read-only review.

## Solution Assessment

No; broadening the generic recoverable regex is too broad for non-idempotent send safety and does not directly update the strict final-send helper. The safer path is nested pre-connect evidence in the strict send boundary unless maintainers explicitly approve broader retry semantics.
