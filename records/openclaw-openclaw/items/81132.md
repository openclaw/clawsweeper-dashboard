## Summary

Keep open. Current main has a post-report Telegram startup-probe limiter, but the reported message-loss path is not fully fixed because Telegram polling watchdogs, Telegram request timeouts, diagnostics, and stuck-session recovery still depend on the same gateway Node event loop, and the related fix PR remains open.

## What This Changes

The issue asks OpenClaw to keep Telegram polling and recovery reliable when gateway or agent work saturates the Node event loop so inbound Telegram messages are not lost.

## Best Possible Solution

Keep this open until maintainers choose and verify a resilience design that either isolates Telegram polling from gateway event-loop starvation or reliably aborts and recovers the wedged agent work before Telegram delivery expires.

## Reproduction Assessment

No high-confidence live reproduction is established. The reporter's incident logs plus current source make the starvation path source-reproducible, but the steps do not provide a deterministic current-main stuck-agent trigger.

## Solution Assessment

Unclear. The current startup-probe limiter is a useful partial mitigation, but the maintainable fix needs a maintainer decision between polling isolation, stronger gateway backpressure, and safer stuck-run recovery rather than only adding more same-process timers.
