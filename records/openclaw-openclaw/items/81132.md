## Summary

Keep open. Current main has partial Telegram startup and polling-liveness mitigations after the report, but it still does not prove that Telegram inbound polling survives or recovers from full gateway event-loop starvation and stuck agent work.

## What This Changes

Ensure Telegram polling continues receiving or safely recovers messages when gateway event-loop saturation delays `getUpdates`, timers, and stuck-session recovery.

## Best Possible Solution

Land a maintainer-owned resilience fix with live Telegram proof, either by isolating or durably controlling Telegram polling/startup work or by reliably aborting/recovering wedged agent work before channel timers starve.

## Reproduction Assessment

No, not as a high-confidence executable current-main reproduction. The incident logs and current source make the failure path plausible, but the report does not provide a deterministic stuck-agent trigger or live Telegram proof against current main.

## Solution Assessment

Unclear. Current same-process watchdogs and the merged startup limiter are useful partial mitigations, but the maintainable fix needs a product/architecture choice around polling isolation, startup throttling, and stuck-run recovery.
