## Summary

Keep open. Current main still relies on same-process timers and aborts for Telegram polling, Telegram request timeouts, and stuck-session diagnostics, so the reporter's event-loop starvation logs describe a plausible unresolved failure mode rather than a cleanup-close candidate.

## What This Changes

The issue reports that Telegram long polling can stall and eventually miss inbound messages when an agent session saturates the gateway event loop.

## Best Possible Solution

Keep this open until maintainers choose and verify a resilience design that either isolates Telegram polling from wedged agent work or reliably aborts and recovers the stuck run before it can starve channel processing.

## Reproduction Assessment

No, not as a high-confidence executable reproduction. The incident logs and current source make the starvation path plausible, but the steps do not identify a deterministic stuck-agent trigger on current main.

## Solution Assessment

Unclear. More same-process timers or polling retries alone are unlikely to solve full event-loop starvation; the safer direction is a maintainer decision between polling isolation, durable channel backpressure, and stronger stuck-run recovery.
