## Summary

Keep open. The report is recent and the supplied logs align with current source: Telegram polling watchdogs, Telegram fetch timeouts, and stuck-session diagnostics all run in the same Node event loop, so full event-loop saturation can delay both detection and recovery instead of giving Telegram independent liveness.

## What This Changes

The issue reports Telegram long polling losing inbound messages when a stuck agent session saturates the gateway event loop and diagnostic recovery returns no-op.

## Best Possible Solution

Pick a maintainer-owned resilience design that either isolates or durably controls Telegram polling, or reliably aborts and recovers the wedged embedded run before it can starve gateway channel work.

## Reproduction Assessment

No, not as a high-confidence executable repro. The incident logs plus current source make the starvation path plausible, but the report does not provide a deterministic stuck-agent trigger that was reproduced on current main.

## Solution Assessment

Unclear. Adding more same-process timers or retries is unlikely to solve full event-loop starvation; maintainers should choose between polling isolation/durability and stronger stuck-run recovery before implementation.
