## Summary

Keep open. Current main still relies on same-process timers for Telegram polling, Telegram request timeouts, and diagnostic recovery, so the reporter's event-loop-starvation logs describe a plausible unresolved failure mode rather than an obsolete or already-shipped fix.

## What This Changes

The issue reports that Telegram long polling and stuck-session recovery can stall when one agent saturates the Node event loop, causing inbound Telegram messages to remain undelivered.

## Best Possible Solution

Choose a maintainer-owned resilience design that keeps Telegram inbound work observable and recoverable under event-loop starvation, either by isolating polling or by reliably aborting the wedged embedded run and applying backpressure before channel work is starved.

## Reproduction Assessment

No. The incident logs are detailed and current source makes the same-loop starvation path plausible, but I did not establish a deterministic current-main stuck-agent trigger that reproduces lost Telegram delivery.

## Solution Assessment

Unclear. More same-process watchdog retries are unlikely to survive 100% event-loop starvation; the safer direction is a maintainer decision between polling isolation, durable recovery signaling, and stronger stuck-run abort/backpressure.
