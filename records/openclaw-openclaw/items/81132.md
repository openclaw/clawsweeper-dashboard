Keep open. The incident evidence and current source still line up with an unresolved starvation path: Telegram polling, Telegram request aborts, and stuck-session recovery all depend on same-process timers, and no duplicate or merged fix was found.

## What This Changes

The issue reports that Telegram inbound polling can stall and lose messages when a stuck agent session saturates the gateway event loop.

## Best Possible Solution

Make Telegram receive liveness resilient to a saturated gateway loop and make wedged embedded-run recovery capable of unblocking channel work, with focused regression coverage and live Telegram proof.

## Reproduction Assessment

No, not as a high-confidence executable repro. The incident logs and current source make the starvation path plausible, but the steps do not identify a deterministic stuck-agent trigger on current main.

## Solution Assessment

Unclear. More same-process retries or a circuit breaker alone are unlikely to solve full event-loop starvation; maintainers should choose between receive-path isolation and stronger stuck-run recovery.
