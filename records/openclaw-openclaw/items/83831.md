## Label Justifications

- `P1`: The report describes a current stable Discord workflow losing the generated final assistant reply after a tool failure warning.
- `impact:message-loss`: The user-visible failure is that Discord no longer shows the assistant's final response and leaves only the warning message.

## Summary

Keep open: current main still has the source-supported Discord delivery path that can clear a finalized streamed reply when a later failed-tool warning final is delivered. The issue is also paired with open fix PRs that use closing references, so this should stay open until one of those PRs lands or is rejected.

## What This Changes

The issue asks OpenClaw to preserve Discord streamed assistant replies when a later failed-tool warning is emitted in the same turn.

## Best Possible Solution

Land one focused Discord delivery fix that keeps later error finals out of the finalized preview cleanup path while still delivering the warning separately and preserving stale-draft cleanup for standalone failures.

## Reproduction Assessment

Yes, source-level: Pi can emit a normal assistant reply followed by an isError warning, and Discord then routes the warning through a finalizer path that can clear the attached draft. I did not run live Discord credentials in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow Discord preview-finalization guard, not global suppression of tool warnings. The warning should remain visible without cancelling a durable assistant answer.
