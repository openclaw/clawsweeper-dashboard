## Summary

Keep open. Current main still lacks a host-wide auth migration or active-profile command; the implemented auth surfaces are per-agent, and the related open #8190 tracks session auth-cache reset rather than atomic auth-state migration across agent stores.

## What This Changes

Request a CLI path to atomically switch a provider's active auth profile across all agent auth stores, with optional stale-profile cleanup and reporting.

## Best Possible Solution

Add an owner-reviewed auth management path that enumerates configured and discoverable agent stores, validates provider/profile availability per target, updates current `auth-state.json` order/last-good state through existing locks, supports dry-run and JSON reporting, and coordinates with session-cache reset work tracked in #8190.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence: source and docs show per-agent auth order control only, while all-agent targeting exists for sessions rather than auth.

## Solution Assessment

Unclear until auth/security-owner design review. The proposed `set-active` or `migrate` direction is plausible, but the maintainable fix should be based on the current auth-state split, single-store locks, target discovery rules, partial-failure semantics, and #8190 session-cache coordination.
