## Summary

Keep open. Current main still exposes only per-agent `models auth order` writes, while all-agent discovery/maintenance is wired for sessions rather than auth-state migration; the related #8190 tracks session auth-cache reset, not atomic cross-agent auth profile switching.

## What This Changes

Request a host-wide CLI command to switch or migrate a provider's active auth profile across all agent-scoped auth stores atomically.

## Best Possible Solution

Implement an explicit, documented auth management command that enumerates configured and discoverable agent stores, validates the requested provider/profile per target, updates current `auth-state.json` state through existing locks with dry-run/JSON reporting, and coordinates with the separate session-reset design where needed.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence: source and docs show per-agent auth order control only, with all-agent targeting limited to sessions.

## Solution Assessment

Unclear until auth/security-owner review. The proposed `set-active` or `migrate` direction is plausible, but the durable design should use the current `auth-state.json` split, existing per-store locks, target discovery, and session-cache coordination boundaries.
