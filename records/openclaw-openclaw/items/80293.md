## What This Changes

The PR applies auto thread-id resolution before targeted plugin-owned message action dispatch, adds a focused threaded plugin-action regression, and declares Slack's runtime typebox dependency.

## Best Possible Solution

Land a focused core dispatcher fix that reuses the existing thread helper before plugin action dispatch, with regression coverage for a thread-aware plugin action and the existing dependency contract kept green.

## Reproduction Assessment

Yes, from source inspection: current main resolves threadId for send and poll but not for plugin-owned actions, so a targeted sticker-style plugin action can dispatch without the current threadId. I did not run the test locally because this review was read-only.

## Solution Assessment

Yes. Reusing resolveAndApplyOutboundThreadId at the plugin-action boundary is the narrowest maintainable fix because it matches existing send/poll behavior before both gateway and local dispatch.
