## Summary

Keep open. This is a focused bug-fix PR for a real current-main gap: send and poll already apply outbound thread resolution, while plugin-owned actions dispatch without it; I found no blocking functional or security finding in the proposed approach.

## What This Changes

The PR applies existing outbound thread-id resolution before targeted plugin-owned message action dispatch and adds a threaded plugin-action regression test, plus a Slack typebox dependency manifest update.

## Best Possible Solution

Land a focused dispatcher fix that reuses the existing thread helper before plugin action dispatch, keeps the regression test, and refreshes the already-main Slack dependency change during normal review/landing.

## Reproduction Assessment

Yes. Source inspection on current main shows threadId resolution happens for send and poll but not for plugin-owned actions before gateway/local dispatch, so a targeted sticker-style action can miss the current forum/topic threadId.

## Solution Assessment

Yes. Reusing resolveAndApplyOutboundThreadId at the plugin-action boundary is the narrowest maintainable fix because it matches existing send/poll behavior and runs before both gateway and local plugin dispatch.
