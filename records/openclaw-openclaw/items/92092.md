## Label Justifications

- `P1`: Current main can silently drop assistant-authored narration in a supported Claude CLI agent and channel workflow.

## Summary

This is a valid, focused repair for a newly introduced output-loss regression. Current main can classify Claude CLI narration without wiring its delivery callback, while the proposed patch aligns both runner gates and adds a safe parser fallback; no blocking correctness or security finding remains, and the supplied live proof is convincing.

## What This Changes

The PR prevents Claude CLI narration from entering an unwired commentary lane and falls back to normal assistant deltas whenever classified text has no commentary consumer.

## Best Possible Solution

Keep one invariant across immediate and followup CLI execution: classify narration only when commentary delivery is active, while the parser guarantees that model-authored text is never silently discarded if a caller omits the consumer.

## Reproduction Assessment

Yes, from source with high confidence. On current main, a false-but-defined commentary option activates classification without its delivery callback, and a subsequent Claude tool-use boundary clears the pending narration.

## Solution Assessment

Yes. Aligning classification with the actual delivery condition fixes the ownership mismatch at both runner entry points, and reusing the normal assistant flush is the narrowest safe defense against future callback mismatches.
