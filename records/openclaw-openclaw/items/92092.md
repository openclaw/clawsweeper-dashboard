## Label Justifications

- `P1`: Current main can silently suppress model-authored narration in real Claude CLI agent turns when commentary delivery is configured but off.

## Summary

The PR fixes a source-confirmed regression introduced by the recently merged verbose-commentary work: current main can classify Claude CLI narration while no commentary consumer is wired, silently discarding model-authored text. The two-layer repair is the best fix because it aligns both runner gates with actual delivery and makes the parser degrade safely instead of losing output; focused tests, real gateway proof, and inspected screenshots support the behavior.

## What This Changes

The PR disables Claude CLI commentary classification unless standalone commentary delivery is active and falls back to assistant deltas when no commentary callback is wired.

## Best Possible Solution

Merge the gate alignment and parser fallback together so every CLI runner classifies narration only for an active consumer and no future caller mismatch can silently lose assistant text.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence reproduction path: set commentaryProgressEnabled to false for a Claude CLI run, which enables classification without wiring onCommentaryText, then emit assistant text before a tool_use block; the buffered narration is cleared without an assistant delta.

## Solution Assessment

Yes. Aligning classification with the actual delivery predicate fixes both runner paths, and the parser fallback is the narrowest safe invariant against future caller mistakes without changing verbose-on behavior.
