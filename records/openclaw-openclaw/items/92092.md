## Summary

The patch correctly addresses a source-confirmed regression introduced by the recent verbose-commentary work: current main can classify Claude CLI narration without wiring a commentary consumer, silently dropping assistant text. The implementation is focused and no blocking code defect was found, but this external PR still needs after-fix proof from a real Claude CLI-backed gateway or channel run before merge.

Likely related people
- Vincent Koc — high confidence; introduced the affected classification gates in both runner paths.
- Ayaan Zaidi — high confidence; recently refactored the surrounding verbose-commentary delivery path.

## What This Changes

The PR enables Claude CLI commentary classification only when an item-event consumer is active and falls back to normal assistant deltas if classified narration has no commentary consumer.

## Best Possible Solution

Keep one shared invariant across both runners and the parser: classify narration only when commentary delivery is active, never discard pending model text, and demonstrate both verbose-off preservation and verbose-on separation in a real Claude CLI-backed run.

## Reproduction Assessment

Yes at source level. Current main deterministically enables classification without a delivery consumer when commentaryProgressEnabled is defined but false, and the parser then drops pending narration.

## Solution Assessment

Yes. Aligning the runner gates removes the known invalid state, while the parser fallback is a narrow defense that preserves model output if another caller violates the invariant later.
