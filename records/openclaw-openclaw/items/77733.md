## Summary

Keep open. Current main still short-circuits bare `/new` and `/reset` before the model/startup path, but docs and tests now intentionally encode that no-model acknowledgement behavior, so this needs maintainer product/UX direction rather than conservative auto-close.

## What This Changes

The issue asks to restore or make configurable the old behavior where bare `/new` and `/reset` trigger the persona/startup greeting instead of only returning canned acknowledgements.

## Best Possible Solution

Maintainers should choose the bare reset contract, then align reset handling, docs, e2e coverage, and live proof around either restored startup-turn fallthrough or an explicit opt-in mode/hook.

## Reproduction Assessment

Yes, source-level: current main returns `shouldContinue: false` for empty reset tails before the prepared reply path, while reporter comments confirm the same behavior in Feishu and WebChat installs. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Unclear. Removing the short-circuit is the narrow code direction for the reported greeting regression, but current docs/tests intentionally assert no-model acknowledgements, so maintainers need to decide whether to restore old behavior or expose a configurable opt-in.
