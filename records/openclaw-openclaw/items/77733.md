## Summary

Keep open. Current main still short-circuits bare `/new` and `/reset` before the startup/model path, the issue has multiple matching real-world confirmations, and an open closing PR now owns the narrow fallthrough fix, so this issue should not be closed before that PR is reviewed and merged or rejected.

## What This Changes

The issue asks to restore bare `/new` and `/reset` so reset hooks run and the request then reaches the startup/model turn instead of returning only a canned acknowledgement.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/81939, then align reset handling, docs, tests, and live-channel proof around the chosen bare reset contract.

## Reproduction Assessment

Yes. Source-level inspection on current main shows bare `/new` and `/reset` return `shouldContinue: false` before `get-reply-run.ts` can build the bare reset startup prompt; reporters also confirmed the same behavior in Feishu, WebChat, and Telegram installs.

## Solution Assessment

Unclear. Restoring fallthrough as in https://github.com/openclaw/openclaw/pull/81939 is the narrow code direction for the reported regression, but current docs/tests and https://github.com/openclaw/openclaw/pull/78968 encode acknowledgement-only behavior that maintainers need to reconcile.
