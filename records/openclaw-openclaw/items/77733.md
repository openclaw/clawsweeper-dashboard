## Summary

Keep open. Current main still short-circuits bare `/new` and `/reset` before the model/startup path, while current docs and tests intentionally assert that no-model acknowledgement behavior; the related cleanup comments treat this issue as the canonical tracker for the broader reset/new startup contract, so this needs maintainer product/UX direction rather than conservative closure.

## What This Changes

Bare `/new` and `/reset` should trigger the agent startup/persona greeting, or provide an opt-in mode, instead of stopping at a canned acknowledgement.

## Best Possible Solution

Maintainers should define the bare reset/new contract, then align reset handling, docs, tests, and live channel behavior around either restored startup-turn fallthrough or an explicit acknowledgement-only/configurable mode.

## Reproduction Assessment

Yes, source-level: current main returns `shouldContinue: false` for empty reset tails before the prepared reply/model path can run, and multiple reporters confirmed the same result in Feishu, WebChat, and Telegram installs. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Unclear. Removing the short-circuit is the narrow code direction for restoring the historical greeting behavior, but current docs/tests intentionally assert no-model acknowledgements, so maintainers need to choose whether to restore fallthrough or expose a configurable policy.
