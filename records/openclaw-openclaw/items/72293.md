## Summary

Keep this open. Current main still preserves `stopReason: "aborted"` but classifies a lifecycle `end` without `aborted: true` as successful in `agent.wait` and subagent completion; the adjacent recovery PR at https://github.com/openclaw/openclaw/pull/75462 is still open and does not cover this stop-reason classification path.

## What This Changes

The issue asks OpenClaw to surface aborted empty-output subagent runs as failed or aborted instead of silently treating them as successful completion.

## Best Possible Solution

Normalize terminal `stopReason: "aborted"` into the existing non-success lifecycle, `agent.wait`, subagent completion, and parent announcement paths, with focused regression tests and no new public status or retry policy.

## Reproduction Assessment

Yes, at source level. Emit a lifecycle `end` carrying `stopReason: "aborted"` without `aborted: true`; current main preserves the stop reason but `agent.wait` and the subagent registry still classify the run as successful.

## Solution Assessment

Yes. The narrow fix is to map the existing abort signal into the current failure/timeout/killed completion paths; auto-retry or a new public `aborted` status would be broader product work.
