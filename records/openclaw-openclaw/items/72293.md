## Summary

Keeping this open: current main still has a source-level path where a lifecycle `end` carrying `stopReason: "aborted"` but no `aborted: true` is classified as successful by `agent.wait` and the subagent registry, so the parent completion path can still receive an `ok` outcome instead of a failure/timeout signal.

## What This Changes

The issue requests that aborted-empty subagent terminal states be reported as non-success and parent-visible instead of being marked done and silently dropped.

## Best Possible Solution

Normalize terminal `stopReason: "aborted"` into the existing non-success lifecycle, `agent.wait`, subagent completion, and parent announcement paths, with focused regression tests and without adding auto-retry or a new public status unless maintainers choose that separately.

## Reproduction Assessment

Yes, at source level. Emit a lifecycle `end` event with `stopReason: "aborted"` and no `aborted: true`; current main preserves the stop reason but reports success through `agent.wait` and subagent completion.

## Solution Assessment

Yes. The narrow maintainable fix is to map the existing abort signal into the current failure/timeout/killed completion paths and add regression coverage; auto-retry or a new public `aborted` status would be broader product work.
