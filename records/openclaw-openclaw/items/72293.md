## Summary

Keep open: current main still has a source-level path where a lifecycle end carrying `stopReason: "aborted"` but no `aborted: true` is classified as successful by `agent.wait` and subagent completion. Recent main may produce a no-output completion in some paths, but the abort is still not normalized into a parent-visible non-success outcome.

## What This Changes

The issue asks OpenClaw to treat empty aborted subagent terminal turns as non-success and send a parent-visible failure or abort completion signal.

## Best Possible Solution

Normalize `stopReason: "aborted"` into the existing non-success lifecycle, `agent.wait`, subagent completion, and parent announcement paths, without adding auto-retry or a new public status unless maintainers choose that separately.

## Reproduction Assessment

Yes, at source level. Emit a lifecycle `end` event with `stopReason: "aborted"` and no `aborted: true`; current main preserves the stop reason but reports success through `agent.wait` and subagent completion.

## Solution Assessment

Yes. The narrow maintainable fix is to map the existing abort signal into current failure, timeout, or killed completion paths and add regression coverage; auto-retry or a new public `aborted` status would be broader product work.
