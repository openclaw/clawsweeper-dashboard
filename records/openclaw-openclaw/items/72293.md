## Summary

Keep this open. Current main still has the stop-reason-only abort gap: the lifecycle producer, `agent.wait`, and subagent registry can preserve `stopReason: "aborted"` without converting it into a parent-visible failure, and the issue is already cross-referenced by open fix work that should be reviewed or replaced before closure.

## What This Changes

The issue requests that empty-content aborted subagent terminal turns be classified as aborted or failed and announced to the parent instead of being marked done with no parent-visible completion.

## Best Possible Solution

Land or replace a narrow core fix that normalizes final assistant `stopReason: "aborted"` into a parent-visible failed or killed outcome across lifecycle emission, `agent.wait`, subagent registry completion, announce formatting, and focused tests. Keep retry policy or a new public `aborted` status out of the minimal fix unless maintainers explicitly choose that larger product behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit a terminal lifecycle `phase: "end"` carrying `stopReason: "aborted"` without `aborted: true`; current main preserves the stop reason but `agent.wait` and the subagent registry still classify that terminal event as successful.

## Solution Assessment

Yes. The best direction is the narrow classification fix already outlined by the linked fix work; auto-retry or a new status union would be broader policy and should not be bundled into the minimal bug fix by default.
