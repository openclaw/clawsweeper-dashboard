## Summary

Keeping this open: current main still has a concrete source-level path where a terminal assistant message with `stopReason: "aborted"` can be emitted as a lifecycle `end` without `aborted: true`, and the `agent.wait` plus subagent completion consumers still classify that event as successful. The adjacent recovery PR at https://github.com/openclaw/openclaw/pull/75462 only changes a swallowed catch path and does not resolve this stop-reason classification bug.

## What This Changes

The issue requests that aborted subagent terminal states with empty content be surfaced to the parent instead of being marked successful and silently dropped.

## Best Possible Solution

Normalize terminal `stopReason: "aborted"` into the existing non-success lifecycle, `agent.wait`, subagent completion, and parent announcement paths, with focused regression tests and without adding a new retry policy or public status unless maintainers choose that separately.

## Reproduction Assessment

Yes, at source level. Emit a lifecycle `end` event with `stopReason: "aborted"` and no `aborted: true`; current main preserves the stop reason but reports success through `agent.wait` and subagent completion.

## Solution Assessment

Yes. The narrow maintainable fix is to map the existing abort signal into the current failure/timeout/killed completion paths; auto-retry or a new public `aborted` status would be broader product work.
