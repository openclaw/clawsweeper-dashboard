## Summary

Keep this open. Current main still has the core stop-reason-only abort gap: the embedded runner can carry `stopReason: "aborted"` into terminal lifecycle metadata, but the lifecycle producer, `agent.wait`, and subagent registry do not consistently classify that shape as a failed/killed parent-visible outcome. The issue is also paired with an open linked fix PR, so it should remain open until that PR or an equivalent target-repo patch lands.

## What This Changes

The issue reports that aborted subagent runs with an empty final assistant message are classified as successful and never announce a parent-visible failure.

## Best Possible Solution

Land or replace a narrow core fix that normalizes a terminal assistant `stopReason: "aborted"` into a failed/killed parent-visible outcome across lifecycle emission, `agent.wait`, subagent registry completion, announce formatting, and focused regression tests. Keep provider retry policy and any new public status union out of scope unless maintainers explicitly choose that broader design.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit a terminal lifecycle `end` carrying `stopReason: "aborted"` without `aborted: true`; current main persists that shape as killed in one helper while `agent.wait` and the subagent registry still classify it as successful.

## Solution Assessment

Yes. The best path is the narrow lifecycle/classification fix described by the linked PR: propagate aborted terminal metadata into an existing error/killed/timeout-facing outcome and test both the producer and consumer paths, rather than adding an automatic retry or a new public runtime status in this bug fix.
