## Summary

Keep open: current main still has a source-level path where a terminal lifecycle event with `stopReason: "aborted"` but no `aborted: true` is treated as successful by `agent.wait` and subagent completion, even though the session lifecycle mapper already recognizes that stop reason as killed.

## What This Changes

The issue requests that aborted subagent runs with empty output become parent-visible non-success completions instead of successful or silently missed terminal states.

## Best Possible Solution

Normalize `stopReason: "aborted"` into the existing non-success lifecycle, `agent.wait`, subagent completion, and parent announcement paths, while leaving auto-retry or a new public status to separate product work.

## Reproduction Assessment

Yes, at source level. Emit a lifecycle `end` event with `stopReason: "aborted"` and no `aborted: true`; current main preserves the stop reason but reports success through `agent.wait` and subagent completion.

## Solution Assessment

Yes. The narrow maintainable fix is to map the existing abort signal into current failure, timeout, or killed completion paths and add regression coverage; auto-retry or a new public `aborted` status would be broader product work.
