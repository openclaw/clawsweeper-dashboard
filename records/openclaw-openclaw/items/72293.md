## Summary

Keep this open. Current main still has the reported stopReason-only abort classification gap in the lifecycle, agent.wait, and subagent registry paths, and the issue is already tied to open related fix work that should be reviewed or replaced before cleanup.

## What This Changes

The issue requests that aborted, empty-output subagent runs surface as aborted or failed and notify the parent instead of being treated as done with no announcement.

## Best Possible Solution

Normalize terminal stopReason: "aborted" into the existing non-ok lifecycle/wait/registry outcomes, preserve sanitized error text, ensure a parent-visible failure announcement is generated for empty-output terminal aborts, and add focused regression coverage across handleAgentEnd, agent.wait, and subagent completion.

## Reproduction Assessment

Yes. A deterministic source-level reproduction is to emit a lifecycle phase "end" event with stopReason: "aborted" and no aborted: true; current main preserves the stop reason but still classifies agent.wait and subagent completion as ok.

## Solution Assessment

Yes. The narrowest maintainable fix is to propagate the existing abort classification through current lifecycle consumers and tests; automatic provider retry or a new public status union would be broader product behavior.
