## Summary

Keep this open. Current main still treats a lifecycle `end` event with only `stopReason: "aborted"` as successful in `agent.wait` and subagent completion, and the issue is already linked to open fix work that should be reviewed, landed, or deliberately replaced before cleanup.

## What This Changes

The issue reports that aborted subagent runs with empty assistant content are marked as successful completion and fail to deliver a parent-visible failure announcement.

## Best Possible Solution

The desired end state is a narrow fix that normalizes terminal `stopReason: "aborted"` into existing non-success lifecycle, `agent.wait`, and subagent completion outcomes, preserves sanitized error text, keeps transient `aborted: true` retry grace intact, and adds focused regression coverage for producer and consumer paths.

## Reproduction Assessment

Yes. Emit a lifecycle `end` event with `stopReason: "aborted"` and no `aborted: true`; current main preserves the stop reason but maps `agent.wait` to `ok` and the subagent registry to an `ok` completion.

## Solution Assessment

Partly. Propagating `stopReason: "aborted"` through existing non-success outcomes is the narrowest maintainable fix; automatic provider retry or a new public status union should be a separate product decision.
