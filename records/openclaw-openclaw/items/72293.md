## Summary

Keep this open. Current main still has the reported classification gap: lifecycle end events can carry `stopReason: "aborted"`, but the parent-visible wait path and subagent registry still only treat `phase: "error"` or `aborted: true` as non-ok outcomes. The issue is also paired with open related fix work, so it should stay open until a fix PR lands or is deliberately replaced.

## What This Changes

The issue requests that empty terminal assistant turns with `stopReason: "aborted"` be classified and announced as aborted/failed instead of silently completing as done/ok.

## Best Possible Solution

Normalize terminal `stopReason: "aborted"` into a parent-visible failed or killed outcome across lifecycle emission, `agent.wait`, subagent registry completion, and announce formatting, with focused regression tests. Keep provider auto-retry or a new public status union out of the minimal fix unless maintainers choose that larger product behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit lifecycle `phase: "end"` with `stopReason: "aborted"` and no `aborted: true`; current main preserves the stop reason but `agent.wait` and the subagent registry still classify that terminal event as successful.

## Solution Assessment

Yes. The narrowest maintainable fix is to propagate the existing abort classification through the current lifecycle/wait/registry consumers and tests; auto-retry is a broader policy decision and should not be required for this bug fix.
