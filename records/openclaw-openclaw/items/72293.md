## Summary

Keep this open. Current main still has the stopReason-only abort classification gap: an `end` lifecycle event carrying `stopReason: "aborted"` is preserved but still becomes success in `agent.wait` and subagent completion unless `aborted: true` or `phase: "error"` is also present; linked open fix work should be reviewed or replaced before cleanup.

## What This Changes

The issue reports aborted subagent terminal turns with empty content being classified as successful completion and not reliably announced to the parent.

## Best Possible Solution

Review and land, replace, or close the open fix work with a narrow change that normalizes terminal `stopReason: "aborted"` into existing non-success lifecycle, `agent.wait`, and subagent completion outcomes, with sanitized error text and focused regression tests.

## Reproduction Assessment

Yes. A deterministic reproduction is to emit a lifecycle `phase: "end"` event with `stopReason: "aborted"` and no `aborted: true`; current main preserves the stop reason but maps `agent.wait` and subagent completion to success.

## Solution Assessment

Yes, the narrowest maintainable fix is to propagate the existing abort signal through current lifecycle consumers and tests. Auto-retry or a new public status union would be broader product behavior and should not be required for this bug fix.
