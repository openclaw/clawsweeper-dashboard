## Summary

Keep this open. Current main still maps a terminal lifecycle `end` carrying only `stopReason: "aborted"` to success in both `agent.wait` and subagent completion, while the documented subagent contract expects push-based completion and an open linked fix PR has not merged.

## What This Changes

The issue reports aborted subagent terminal turns with empty output being classified as done/ok and not announced to the parent.

## Best Possible Solution

Review and land, replace, or close the linked fix work with a narrow change that normalizes terminal `stopReason: "aborted"` into existing non-success lifecycle, `agent.wait`, and subagent completion outcomes, with focused regression tests and no new public status union unless maintainers explicitly want that broader product change.

## Reproduction Assessment

Yes. A deterministic reproduction is to emit a lifecycle `phase: "end"` event with `stopReason: "aborted"` and no `aborted: true`; current main preserves the stop reason but maps `agent.wait` and subagent completion to success.

## Solution Assessment

Yes. The narrowest maintainable fix is to propagate the existing abort signal through current lifecycle consumers and completion announcement handling; auto-retry or a new public `aborted` status would be broader than required for this bug.
