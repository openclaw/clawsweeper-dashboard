## Summary

Keep this open. Current main still carries an aborted assistant `stopReason` as normal lifecycle metadata, while the `agent.wait` and subagent completion consumers only treat lifecycle `error` or `aborted: true` as non-success; the open fix work has not merged.

## What This Changes

The issue requests that aborted empty subagent terminal turns surface as failed/aborted parent-visible completions instead of successful silent completion.

## Best Possible Solution

Land or replace the existing narrow fix so terminal `stopReason: "aborted"` flows into the existing non-success lifecycle, `agent.wait`, subagent completion, and announce paths without adding a new retry policy or public status.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a lifecycle `end` event carrying `stopReason: "aborted"` without `aborted: true`; current main preserves the stop reason but classifies `agent.wait` and subagent completion as success.

## Solution Assessment

Yes. The narrow maintainable solution is to normalize the existing aborted terminal signal into current failure/timeout handling; auto-retry or a new public `aborted` status would be broader product work.
