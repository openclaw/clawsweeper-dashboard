## Summary

Keep this open. Current main still treats a lifecycle `end` with `stopReason: "aborted"` as successful in both `agent.wait` and subagent completion unless another abort/error flag is present, and linked open fix work should be reviewed before cleanup. https://github.com/openclaw/openclaw/issues/72293

## What This Changes

The issue reports aborted subagent terminal turns with empty content being classified as successful completion and not reliably announced to the parent.

## Best Possible Solution

Normalize terminal `stopReason: "aborted"` into the existing non-success lifecycle, `agent.wait`, and subagent completion outcomes, with sanitized error text and focused regression tests.

## Reproduction Assessment

Yes. A deterministic source-level reproduction is to emit a lifecycle `phase: "end"` event with `stopReason: "aborted"` and no `aborted: true`; current main preserves the stop reason but maps `agent.wait` and subagent completion to success.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate the existing abort signal through current lifecycle consumers; auto-retry or a new public status union would be broader product behavior and should not be required for this bug.
