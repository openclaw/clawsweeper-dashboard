## Summary

Keep this open. Current main still has the stopReason-only aborted path classified as success in `agent.wait` and subagent completion, and an open PR with closing syntax already targets the issue.

## What This Changes

Reports that subagent runs ending with an empty aborted assistant turn are marked successful and do not announce a failure to the parent.

## Best Possible Solution

Normalize terminal `stopReason: "aborted"` into the existing non-success lifecycle, wait, and subagent completion outcomes with focused regression tests, without adding a broader public status union unless maintainers choose that separately.

## Reproduction Assessment

Yes. Emit a lifecycle `phase: "end"` event with `stopReason: "aborted"` and no `aborted: true`; current main preserves the stop reason but classifies `agent.wait` and subagent completion as success.

## Solution Assessment

Yes. The narrowest maintainable fix is to propagate the existing abort signal through current lifecycle consumers and completion announcement handling; auto-retry would be broader than needed for this bug.
