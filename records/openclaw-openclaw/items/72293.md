## Summary

Keep this open. Current main still preserves `stopReason: "aborted"` but classifies a lifecycle `end` without `aborted: true` as successful for `agent.wait` and subagent completion, while the related open PR #75462 is adjacent recovery work and has not merged.

## What This Changes

The issue reports that aborted subagent terminal messages with empty content are treated as successful done runs and can fail to notify the parent.

## Best Possible Solution

Normalize terminal `stopReason: "aborted"` into the existing non-success lifecycle, `agent.wait`, subagent completion, and parent announcement paths with focused regression tests; avoid adding auto-retry or a new public status unless maintainers choose that product change.

## Reproduction Assessment

Yes, source-level. Emit a lifecycle `end` with `stopReason: "aborted"` and no `aborted: true`; current main preserves the stop reason but classifies `agent.wait` and subagent completion as success.

## Solution Assessment

Yes for the narrow fix direction. Reusing the existing timeout/error/aborted lifecycle paths is more maintainable than adding auto-retry or a new public status for this bug.
