## Summary

Keep this open. Current main still preserves `stopReason: "aborted"` metadata but can classify a terminal lifecycle `end` without `aborted: true` as successful for `agent.wait` and subagent completion, matching the reported parent black-hole behavior; related PR #75462 is open but does not prove this issue is fixed on main.

## What This Changes

The issue requests that aborted subagent terminal messages with empty content be surfaced as aborted or failed instead of being marked done and silently dropped from parent announcement.

## Best Possible Solution

Normalize terminal `stopReason: "aborted"` into the existing aborted/failure lifecycle, `agent.wait`, subagent completion, and parent announcement paths, with focused regression coverage and no new retry policy unless maintainers choose that separately.

## Reproduction Assessment

Yes, at source level. Emit a lifecycle `end` with `data.stopReason: "aborted"` and no `data.aborted`; current main preserves that stop reason but classifies `agent.wait` and subagent completion as success.

## Solution Assessment

Yes. The narrow maintainable fix is to map the existing abort signal into current non-success completion paths; auto-retry or a new public `aborted` status would be broader product work.
