## Summary

Keep this open. Current main still has the stopReason-only abort classification gap: `stopReason: "aborted"` is preserved on lifecycle `end` events, but `agent.wait` and subagent completion still treat that path as success unless `aborted: true` or `phase: "error"` is also present; linked open fix work should be reviewed or replaced before cleanup.

## What This Changes

The issue reports that aborted, empty subagent terminal messages are classified as done/ok and silently dropped instead of producing a parent-visible failure announcement.

## Best Possible Solution

Review and land, replace, or close the linked fix work with a narrow change that normalizes terminal `stopReason: "aborted"` into existing non-success lifecycle, `agent.wait`, and subagent completion outcomes with focused regression tests.

## Reproduction Assessment

Yes. A deterministic reproduction is to emit a lifecycle `phase: "end"` event with `stopReason: "aborted"` and no `aborted: true`; current main preserves the stop reason but maps `agent.wait` and subagent completion to success.

## Solution Assessment

Yes. The narrowest maintainable fix is to propagate the existing abort signal through current lifecycle consumers; auto-retry or a new public status union would be broader product behavior and should not be required for this bug fix.
