## Summary

Keeping this open: current main still has a concrete source-level path where `stopReason: "aborted"` is carried on a lifecycle `end` event, but `agent.wait` and the subagent registry classify that event as successful unless `aborted: true` is also set. The adjacent open recovery PR at https://github.com/openclaw/openclaw/pull/75462 does not supersede this because it only targets swallowed completion-handler errors, not stop-reason classification.

## What This Changes

The issue requests that aborted subagent terminal messages with empty content be surfaced as aborted or failed instead of completed successfully and silently dropped.

## Best Possible Solution

Normalize terminal `stopReason: "aborted"` into the existing non-success lifecycle, `agent.wait`, subagent completion, and parent announcement paths, with focused regression tests and no new retry policy or public status unless maintainers choose that separately.

## Reproduction Assessment

Yes, at source level. Emit a lifecycle `end` event with `stopReason: "aborted"` and no `aborted: true`; current main preserves the stop reason but reports success through `agent.wait` and subagent completion.

## Solution Assessment

Yes. The narrow maintainable fix is to map the existing abort signal into current failed/killed/timeout-style completion paths; auto-retry or a new public `aborted` status would be broader product work.
