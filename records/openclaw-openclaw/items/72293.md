## Summary

Keep this open. Current main still has the reported abort-classification gap: a lifecycle `end` with `stopReason: "aborted"` can be preserved as metadata but still becomes success in `agent.wait` and subagent completion unless `aborted: true` or `phase: "error"` is also present; linked open fix work should be reviewed, landed, or replaced before cleanup.

## What This Changes

Treat aborted empty subagent terminal turns as failed or aborted and deliver a parent-visible completion/error instead of silently marking the run done.

## Best Possible Solution

Normalize terminal `stopReason: "aborted"` into the existing non-success lifecycle, `agent.wait`, subagent registry, and announce outcomes with focused regression tests; defer auto-retry or a new public status union as broader product work.

## Reproduction Assessment

Yes. Emit a lifecycle `phase: "end"` event with `stopReason: "aborted"` and no `aborted: true`; current main preserves the stop reason but derives success in both `agent.wait` and subagent completion.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate the existing abort signal through current lifecycle consumers; adding provider retry policy is not required to fix the parent black-hole behavior.
