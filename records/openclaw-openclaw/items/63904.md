## Summary

Keep open. Current main now implements the explicit outbound Slack `replyBroadcast` path, but the original inbound detection and automatic mirroring policy remain unresolved, and a recent Slack-area comment explicitly left this issue open for that remaining scope.

## What This Changes

Request to support Slack `reply_broadcast` end to end by detecting inbound broadcast thread replies and letting agents mirror that visibility on outbound Slack thread replies.

## Best Possible Solution

Add Slack-owned inbound broadcast visibility metadata and an explicit mirroring policy that only sets `replyBroadcast` when agent/user intent is clear, while preserving the current explicit outbound opt-in behavior.

## Reproduction Assessment

Yes for the remaining feature gap. Source inspection on current main shows explicit outbound `replyBroadcast` is present, but inbound `thread_broadcast` is not surfaced as structured broadcast visibility that an agent can mirror.

## Solution Assessment

Unclear for the remaining work. The landed outbound opt-in is a good narrow start, but automatic mirroring needs a Slack visibility policy decision and should probably be driven by explicit structured inbound metadata rather than implicit generic reply behavior.
