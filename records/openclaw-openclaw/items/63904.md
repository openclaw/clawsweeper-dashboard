## Summary

Keep open. Current main still lacks Slack `reply_broadcast` outbound plumbing and does not expose inbound `thread_broadcast` as a visibility flag that an agent can mirror. The related open PR #64365 is still an implementation candidate rather than shipped behavior, and the issue is recent with a concrete Slack Socket Mode reproduction path.

## What This Changes

Request support for Slack thread reply broadcasts by detecting inbound `thread_broadcast`/`reply_broadcast` visibility and mirroring it on outbound threaded replies.

## Best Possible Solution

Keep the issue open and land PR #64365 or a successor Slack-plugin patch that adds typed `replyBroadcast` plumbing through Slack send/action/message-tool paths, passes Slack `reply_broadcast` only with a valid `threadTs`, exposes inbound broadcast visibility in finalized/tool context, and covers the behavior with focused Slack tests and docs.

## Reproduction Assessment

Yes. A static current-main reproduction is high-confidence: `replyBroadcast` cannot be supplied to Slack send paths, and a `thread_broadcast` inbound event is accepted but finalized without a structured broadcast visibility flag.

## Solution Assessment

Unclear for the open PR, because the provided PR description mainly covers outbound sending while this issue also requires inbound mirroring metadata. The maintainable direction is Slack-scoped and additive, with generic schema exposure only if it is discoverable as Slack-specific capability.
