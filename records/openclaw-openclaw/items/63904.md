## Summary

Keep open. Current main still has Slack thread reply plumbing based on `thread_ts`, but it does not expose or send Slack `reply_broadcast`, and inbound `thread_broadcast` events are accepted without a structured visibility flag an agent can mirror. The related PR #64365 is still open source work, not merged behavior.

## What This Changes

Request Slack support for detecting inbound `thread_broadcast` visibility and sending threaded replies with `reply_broadcast` so agents can mirror Slack's "Also send to channel" behavior.

## Best Possible Solution

Land PR #64365 or a successor Slack-plugin patch that adds typed `replyBroadcast` support through Slack send/action/message-tool dispatch, passes Slack `reply_broadcast` only when a valid thread timestamp is present, exposes inbound `thread_broadcast` as structured visibility metadata, and updates focused Slack tests plus `docs/channels/slack.md`.

## Reproduction Assessment

Yes. A static current-main reproduction is high-confidence: source inspection shows no way to pass `reply_broadcast` outbound, while inbound `thread_broadcast` events are accepted but finalized without a structured broadcast flag.

## Solution Assessment

Unclear for the open PR because the issue needs both outbound broadcast sending and inbound visibility mirroring. The maintainable solution is Slack-scoped and additive, with no core migration unless the shared message-tool schema needs an explicit channel-specific option.
