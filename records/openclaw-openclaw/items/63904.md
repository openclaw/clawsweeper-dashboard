## Summary

Keep open. Current main has the explicit outbound Slack `replyBroadcast` primitive, but inbound broadcast-visibility metadata and any automatic mirroring policy remain unresolved and were explicitly left open in the issue discussion.

## What This Changes

Support Slack thread reply broadcasts by detecting inbound `thread_broadcast` visibility and mirroring replies with Slack `reply_broadcast` when appropriate.

## Best Possible Solution

Add Slack-owned structured inbound broadcast visibility metadata, then implement mirroring only through an explicit maintainer-approved policy that avoids accidental parent-channel broadcasts.

## Reproduction Assessment

Yes, at source level. `thread_broadcast` events pass through the Slack message handler, but prepared agent context exposes normal thread fields without a structured broadcast/subtype visibility flag.

## Solution Assessment

Unclear. The explicit outbound opt-in is already the narrow primitive, but automatic mirroring needs a Slack-scoped inbound metadata contract and maintainer policy decision first.
