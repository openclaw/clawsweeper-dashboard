## Summary

Keep open. Current main still lacks outbound Slack `reply_broadcast` plumbing and only accepts inbound `thread_broadcast` as a normal user message without structured broadcast-visibility metadata; related PR #64365 is open and unmerged work, not implemented behavior.

## What This Changes

The issue requests Slack support for mirroring "Also send to channel" by exposing inbound thread-broadcast visibility and adding outbound `reply_broadcast` send plumbing.

## Best Possible Solution

Land PR #64365 or a successor Slack-plugin patch that adds typed outbound `replyBroadcast`, surfaces inbound broadcast visibility, covers text/block/file paths where appropriate, and updates focused docs/tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the Slack send path can pass `thread_ts` but not `reply_broadcast`, while inbound `thread_broadcast` is accepted without a structured visibility flag an agent can mirror.

## Solution Assessment

Unclear for the current related PR because the issue asks for both outbound sending and inbound detection. The maintainable direction is an additive Slack-scoped capability with regression tests and a clear metadata/tool schema decision.
