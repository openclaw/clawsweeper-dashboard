## Summary

Keep open. Current main covers explicit outbound Slack `replyBroadcast`, and prior work processes inbound `thread_broadcast` messages, but the remaining request to expose inbound broadcast visibility and define mirroring policy is still unresolved and was explicitly left open by maintainer discussion.

## What This Changes

Request support for detecting Slack thread-broadcast replies and mirroring that visibility when OpenClaw replies in the same Slack thread.

## Best Possible Solution

Add Slack-owned structured inbound broadcast visibility metadata, then implement any mirroring behavior only behind an explicit maintainer-approved policy that avoids accidental parent-channel broadcasts.

## Reproduction Assessment

Yes. Source inspection shows `thread_broadcast` is accepted, but the prepared agent context exposes only normal thread metadata and no structured broadcast/subtype visibility flag to mirror.

## Solution Assessment

Unclear. Explicit outbound opt-in is already the right narrow primitive, but automatic mirroring needs a Slack-scoped inbound metadata contract and maintainer policy decision before code should be added.
