## Summary

Keep open. Current main handles Slack `thread_ts` and accepts `thread_broadcast` as a user message, but it still lacks outbound `reply_broadcast` plumbing and does not expose structured inbound broadcast visibility for agents to mirror; related PR #64365 remains open source work.

## What This Changes

Add Slack `replyBroadcast` support so agents can detect inbound thread broadcasts and mirror replies with Slack `reply_broadcast` when responding in a thread.

## Best Possible Solution

Land PR #64365 or a successor Slack-plugin patch that adds typed `replyBroadcast` through message tool dispatch, Slack action runtime, and send payloads, gates `reply_broadcast` on `threadTs`, surfaces inbound `thread_broadcast` visibility in structured context, and updates focused Slack tests plus `docs/channels/slack.md`.

## Reproduction Assessment

Yes. A static current-main reproduction is high-confidence: source inspection shows no outbound parameter or payload field for `reply_broadcast`, while inbound `thread_broadcast` reaches the handler but is finalized without structured broadcast visibility.

## Solution Assessment

Yes for the requested direction: a Slack-scoped additive option and inbound metadata field is the narrow maintainable fix. It is not solved by the current tree or by an outbound-only change.
