## Summary

Keep open. Current main still lacks outbound Slack `reply_broadcast` plumbing and does not surface inbound `thread_broadcast` as structured visibility metadata; related PR #64365 is open work, not merged behavior, and appears to need maintainer review for the inbound half.

## What This Changes

Request support for mirroring Slack thread replies that were also sent to the channel, covering both inbound detection and outbound sends.

## Best Possible Solution

Land PR #64365 or a successor Slack-plugin patch that adds typed outbound `replyBroadcast`, exposes inbound broadcast visibility, covers text/block/file paths where appropriate, and documents the Slack-only behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows no outbound path for `reply_broadcast`, while inbound `thread_broadcast` is accepted without a structured visibility flag that an agent can mirror.

## Solution Assessment

Unclear for the current open PR because the issue asks for both outbound sending and inbound detection. The maintainable solution is an additive Slack-scoped capability with focused tests and docs, plus care around explicit channel-broadcast visibility.
