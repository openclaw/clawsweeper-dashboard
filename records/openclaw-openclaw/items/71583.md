## Summary

Keep open. Current main still posts QQBot access-token requests without a plugin-owned timeout/abort path and QQBot still lacks a status probe, so the reporter's token-stage failure is not implemented or external-only.

## What This Changes

The issue requests an OpenClaw-specific fix or diagnostic path for QQBot access-token fetches to bots.qq.com timing out while the core gateway remains healthy.

## Best Possible Solution

QQBot should bound the access-token POST, map token-stage timeout/network failures into status/probe output, and document bots.qq.com:443 egress diagnostics.

## Reproduction Assessment

Yes for the repo-owned failure mode: stub `fetch` for the QQBot token endpoint so it never resolves, then `getAccessToken()` remains pending because the token path has no abort signal while gateway startup awaits it. The exact A/B uplink timeout remains live-network-specific.

## Solution Assessment

Yes. A QQBot-local timeout, structured token-stage status/probe result, and troubleshooting note is the narrow maintainable fix because core already has a generic probe seam.
