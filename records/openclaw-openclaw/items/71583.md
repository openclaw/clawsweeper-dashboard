## Summary

Keep open. Current main still leaves the QQBot token-fetch stage unbounded and weakly diagnosable: the token POST to `bots.qq.com` has no abort signal or timeout, gateway connection setup waits on that token before WebSocket startup, QQBot does not provide a live `probeAccount` status path, and the docs still only cover generic credential checks.

## What This Changes

The issue reports QQBot disconnects because `https://bots.qq.com/app/getAppAccessToken` times out across two uplinks while the OpenClaw gateway/dashboard stays healthy.

## Best Possible Solution

Keep the repair inside the QQBot plugin: add a bounded abortable token request, surface token-stage failures in runtime status, add a QQBot `probeAccount` path for `channels status --probe`, and document `bots.qq.com:443` token-egress checks without adding Tencent-specific policy to core gateway code.

## Reproduction Assessment

Yes for the repo-owned failure mode: stub `fetch()` in `TokenManager.getAccessToken()` so it never resolves, and current main has no token-request timer or probe path to settle that condition while `GatewayConnection.connect()` awaits the token. The reporter's exact A/B uplink timeout remains a live network/environment reproduction.

## Solution Assessment

Yes. A QQBot-owned timeout/status/probe/docs fix is the narrowest maintainable solution because core already has a generic channel status seam and should stay plugin-agnostic.
