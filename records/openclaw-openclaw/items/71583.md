## Summary

Keep open. Current main still has the QQBot-owned token-stage failure mode described by the report: the `bots.qq.com` access-token POST has no request timeout or abort signal, gateway connection setup awaits that token before WebSocket setup, and QQBot still lacks a live `probeAccount` or token-egress troubleshooting docs for `channels status --probe`.

## What This Changes

The issue requests OpenClaw-specific diagnostics, workarounds, or a fix for QQBot staying disconnected when `https://bots.qq.com/app/getAppAccessToken` times out while the core gateway and dashboard remain healthy.

## Best Possible Solution

Keep the fix inside the QQBot plugin: add a bounded abortable token fetch, surface token-stage failures into runtime status, provide a QQBot `probeAccount` path for `channels status --probe`, and document `bots.qq.com:443` egress checks without adding Tencent-specific policy to core gateway code.

## Reproduction Assessment

Yes for the repo-owned failure mode: stub `fetch()` in `TokenManager.getAccessToken()` so it never resolves, then `GatewayConnection.connect()` awaits that promise before WebSocket setup and current main has no token-fetch timer to settle the path. The reporter's exact network timeout remains environment-specific.

## Solution Assessment

Yes. A QQBot-owned timeout, live probe/status update, focused tests, and docs are the narrowest maintainable repair; core should remain plugin-agnostic and should not special-case `bots.qq.com`.
