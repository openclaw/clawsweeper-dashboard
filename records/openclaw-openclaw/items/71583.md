## Summary

Keep open. Current main still leaves the QQBot access-token fetch to `bots.qq.com` without a timeout or abort signal, gateway connection setup still awaits that token before WebSocket startup, and QQBot has no live `channels status --probe` token-stage check or troubleshooting docs for this failure.

## What This Changes

The issue requests OpenClaw-specific handling and diagnostics for QQBot access-token fetches to `bots.qq.com` timing out while the core gateway remains healthy.

## Best Possible Solution

Keep the fix inside the QQBot plugin: make token acquisition abortable and bounded, surface token-stage failures through status/probe output, add focused hung-fetch regression tests, and document the `bots.qq.com:443` access-token egress check without adding Tencent-specific policy to core gateway code.

## Reproduction Assessment

Yes for the repo-owned failure mode: stub `fetch()` in `TokenManager.getAccessToken()` so it never resolves and observe that current main has no token-request timer while gateway connection setup awaits the promise. The reporter's A/B uplink timeout remains an external network reproduction.

## Solution Assessment

Yes. A QQBot-owned timeout, probe/status, and docs update is the narrowest maintainable direction because core already exposes a generic `probeAccount` seam and should remain plugin-agnostic.
