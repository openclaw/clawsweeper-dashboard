## Summary

Keep open. Current main still leaves QQBot access-token acquisition without a plugin-owned timeout, token-stage probe, or troubleshooting docs, so the reporter's isolated `bots.qq.com` timeout remains actionable in the QQBot plugin rather than obsolete or external-only.

## What This Changes

The issue asks for QQBot token-fetch diagnostics or workarounds when `https://bots.qq.com/app/getAppAccessToken` times out while the core gateway remains healthy.

## Best Possible Solution

Keep the repair in QQBot: bound access-token fetches with an abortable budget, expose token-stage failures through `status --probe`, and document the `bots.qq.com:443` egress check with a changelog entry.

## Reproduction Assessment

Yes for the repo-owned failure mode: stub QQBot token fetch to hang or abort and observe that gateway connection waits before WebSocket setup while `channels status --probe` has no QQBot token probe. The reporter's exact A/B uplink timeout remains a live network condition.

## Solution Assessment

Yes. A QQBot-owned timeout, structured token-stage error/probe, and docs note is the narrow maintainable fix because core already provides a generic channel probe seam and other QQBot API paths already use abortable request budgets.
