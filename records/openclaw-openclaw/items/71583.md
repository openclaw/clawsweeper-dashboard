## Summary

Keep open. Current main still has an unbounded QQBot access-token POST and no QQBot-owned live token probe, so the reporter's isolated `bots.qq.com` timeout remains a concrete plugin bug rather than an implemented or external-only support report.

## What This Changes

The issue requests diagnostics or a workaround for QQBot access-token fetches to `bots.qq.com` timing out while the OpenClaw gateway remains healthy.

## Best Possible Solution

Keep the repair in QQBot: add an abortable access-token timeout, surface token-stage status/probe errors without leaking secrets, and document the `bots.qq.com:443` egress check.

## Reproduction Assessment

Yes. The repo-owned failure mode can be reproduced deterministically by mocking the QQBot token fetch so it never resolves or aborts, then observing that current main has no token-stage timeout/probe; the exact A/B uplink timeout remains a live network condition.

## Solution Assessment

Yes. A QQBot-owned timeout, structured probe result, and docs note is the narrow maintainable fix because core already passes a bounded `timeoutMs` to channel `probeAccount` implementations.
