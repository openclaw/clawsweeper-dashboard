## Summary

Keep open. Current main still matches the repo-owned part of the report: QQBot's access-token POST can wait indefinitely, gateway startup waits on that token before WebSocket setup, and QQBot has no live token-stage probe or token-egress troubleshooting doc even though the core gateway can remain healthy.

## What This Changes

This issue reports QQBot access-token fetches to `bots.qq.com` timing out across two network paths while core health remains good, and asks for known OpenClaw diagnostics or workarounds.

## Best Possible Solution

Keep the fix inside the QQBot plugin: bound the access-token fetch with an abortable timeout, surface token-stage failures in runtime status/probe output, and document `bots.qq.com:443` egress checks without adding Tencent-specific policy to core gateway code.

## Reproduction Assessment

Yes for the repo-owned failure mode: stub `fetch()` in `TokenManager.getAccessToken()` so it never resolves, then current main has no token-fetch timer or probe to settle that path while `GatewayConnection` awaits the token. The reporter's exact A/B uplink timeout remains environment-specific.

## Solution Assessment

Yes. A QQBot-owned timeout, status/probe update, focused tests, and docs are the narrowest maintainable fix; core should stay plugin-agnostic and should not special-case `bots.qq.com`.
