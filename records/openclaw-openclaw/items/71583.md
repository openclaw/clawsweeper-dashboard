## Summary

Keep open. Current main still has a QQBot-owned token-stage failure mode that matches the report: the `bots.qq.com` token POST has no abort signal or timeout, gateway startup awaits that token before WebSocket setup, and QQBot still lacks a live account probe or specific token-egress troubleshooting docs.

## What This Changes

The issue requests diagnostics or a workaround for QQBot disconnects when fetching an access token from `https://bots.qq.com/app/getAppAccessToken` times out while the core gateway remains healthy.

## Best Possible Solution

Keep the fix inside the QQBot plugin: add a bounded abortable token fetch, surface token-stage failures in runtime status, provide a QQBot `probeAccount` path for `channels status --probe`, and document `bots.qq.com:443` egress checks without adding Tencent-specific policy to core gateway code.

## Reproduction Assessment

Yes for the repo-owned failure mode: stub `fetch()` in `TokenManager.getAccessToken()` so it never resolves, and current main has no token-fetch timer or probe to settle that path while `GatewayConnection` awaits the token. The exact A/B uplink timeout remains environment-specific.

## Solution Assessment

Yes. A QQBot-owned timeout, status/probe update, focused tests, and docs are the narrowest maintainable solution; core should stay plugin-agnostic and should not special-case `bots.qq.com`.
