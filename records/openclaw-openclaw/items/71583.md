## Summary

Keep open. Current main still has an unbounded QQBot access-token fetch and no QQBot `probeAccount` implementation, so the repo-owned timeout and diagnostics gap remains even though the exact A/B uplink failure is network-specific.

## What This Changes

The issue requests a fix or OpenClaw-specific diagnostics for QQBot access-token requests to `bots.qq.com` timing out while the core gateway remains healthy.

## Best Possible Solution

Add a QQBot-local access-token timeout/abort path, surface token-stage failures through the existing channel status probe contract, and document `bots.qq.com:443` egress diagnostics.

## Reproduction Assessment

Yes, for the source-level OpenClaw failure mode: stub `fetch` for the QQBot token endpoint so it never resolves, and `TokenManager.getAccessToken()` remains pending because `doFetchToken()` passes no abort signal or timeout. The exact A/B uplink timeout remains live-network-specific.

## Solution Assessment

Yes. A QQBot-local timeout, structured probe/status result, and focused troubleshooting note is the narrow maintainable fix because core already has the generic `probeAccount` timeout seam.
