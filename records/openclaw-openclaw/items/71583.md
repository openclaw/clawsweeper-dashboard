## Summary

Keep open. Current main still has the QQBot access-token request as an unbounded `fetch()` to `bots.qq.com`, and gateway startup still awaits that token before opening the WebSocket, so the reporter's timeout stage remains a concrete plugin-owned failure mode rather than an obsolete or external-only report.

## What This Changes

The issue reports that QQBot stays disconnected because fetching an access token from `https://bots.qq.com/app/getAppAccessToken` times out even while OpenClaw core gateway and dashboard health checks pass.

## Best Possible Solution

Keep the fix in the QQBot plugin: make access-token acquisition abortable with a clear timeout error, prevent QQBot gateway startup from hanging indefinitely at the token stage, surface token-egress failures through the existing status/probe contract, and add a short QQBot troubleshooting note for `bots.qq.com:443` without adding Tencent-specific policy to core.

## Reproduction Assessment

Yes for the repo-owned failure mode: stub `fetch()` in `TokenManager.getAccessToken()` so it never resolves, then observe that current main has no token-request timer while QQBot gateway setup awaits that promise. The reporter's external A/B uplink timeout is a live network condition rather than a checkout-local reproduction.

## Solution Assessment

Yes. A QQBot-owned timeout, structured token-stage error, live probe/status hook, and docs update is the narrowest maintainable solution because core already has a generic probe seam and should remain plugin-agnostic.
