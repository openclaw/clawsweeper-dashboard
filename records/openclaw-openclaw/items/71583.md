## Summary

Keep open. Current main still leaves the QQBot access-token POST to bots.qq.com without a request timeout or abort signal, gateway startup still waits on that token before WebSocket setup, and QQBot does not expose a live probe path or token-egress troubleshooting docs for the reported failure stage.

## What This Changes

The issue asks for OpenClaw-specific diagnostics or a workaround for QQBot account connection failures when fetching the QQ access token from bots.qq.com times out while core gateway health remains good.

## Best Possible Solution

Keep the repair inside the QQBot plugin: make token acquisition abortable and bounded, surface token-stage failures in runtime status/probe output, add focused tests for hung token fetches, and document the bots.qq.com:443 access-token egress check without adding Tencent-specific behavior to core gateway code.

## Reproduction Assessment

Yes for the repo-owned failure mode: stub `fetch()` in `TokenManager.getAccessToken()` so it never resolves, then observe that current main has no token-request timer while gateway connection setup awaits that promise. The reporter's A/B uplink timeout remains a live network reproduction rather than something this checkout can fully replay.

## Solution Assessment

Yes. A QQBot-owned timeout, probe/status, and docs update is the narrowest maintainable fix because core already exposes a generic `probeAccount` seam and should remain plugin-agnostic.
