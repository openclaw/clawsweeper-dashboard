## Summary

Keep open for maintainer handling: the duplicate-listener problem is source-reproducible, but this branch is merge-conflicting, lacks real behavior proof, and its state-key refactor would regress the shipped global handler-registry contract.

## What This Changes

The PR adds a globalThis-backed unhandled-rejection state object and uses it to make installUnhandledRejectionHandler return early after the first process listener install.

## Best Possible Solution

Add a narrow main-based install guard that preserves HANDLERS_GLOBAL_KEY, covers duplicate installer calls and plugin handler sharing, and includes redacted runtime proof.

## Reproduction Assessment

Yes at source level: current main can call installUnhandledRejectionHandler() from both the direct entrypoint and runCli(), while the installer still unconditionally attaches a process listener. I did not run a listener-count repro because this review was read-only.

## Solution Assessment

No, not as currently branched: the install-once guard is the right narrow direction, but replacing the shipped handler registry key is not compatibility-safe. The safer fix is a separate install guard that leaves the handler Set contract intact.
