## Summary

Keep open. The patch has no blocking code finding, and the idempotent install guard is still unique work not present on current main; the handler-registry globalization half has since landed, so this needs rebase and real behavior proof rather than cleanup closure.

## What This Changes

This PR moves unhandled-rejection handler state onto globalThis and makes installUnhandledRejectionHandler skip repeated process listener registration.

## Best Possible Solution

Rebase onto current main, preserve the existing global handler registries, keep the installed guard narrow, and add focused proof or coverage for repeated install registering only one listener.

## Reproduction Assessment

Yes from source inspection: current main's installUnhandledRejectionHandler calls process.on unconditionally, so calling it twice would attach two unhandledRejection listeners. I did not run a runtime repro because this review is read-only.

## Solution Assessment

Yes, the installed guard is the narrow maintainable fix for duplicate process listeners. The global registry part should be reconciled with the current main implementation that already anchors handler Sets on globalThis.
