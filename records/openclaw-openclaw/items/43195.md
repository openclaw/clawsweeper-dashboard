## Summary

Keep open: current main already shares the handler registries globally, but it still does not make installUnhandledRejectionHandler idempotent, so the central guard remains useful. The PR needs a current-main rebase, focused listener-count coverage, and external real behavior proof before merge.

## What This Changes

The PR moves unhandled-rejection handler state into a Symbol.for globalThis slot and makes installUnhandledRejectionHandler return early after the first process listener install.

## Best Possible Solution

Rebase into a minimal current-main patch that preserves existing global handler registry keys, adds a shared installed guard, and covers listener-count idempotency.

## Reproduction Assessment

Yes, at source level: current main's installUnhandledRejectionHandler unconditionally calls process.on("unhandledRejection", ...), so repeated calls would register repeated listeners. I did not run a live repro because this review is read-only.

## Solution Assessment

No, not as-is. The direction is reasonable, but current main already globalizes the registries with a different shape, so the maintainable fix is a rebased install guard plus focused regression coverage and proof.
