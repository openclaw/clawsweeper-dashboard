## Summary

Keep this PR open: current `main` still lacks an install-once guard, so the useful fix is not implemented, but the branch is not merge-ready because it conflicts with `main`, would bypass the shipped global handler registry key added by https://github.com/openclaw/openclaw/pull/71668, and lacks required real behavior proof.

## What This Changes

The PR moves unhandled-rejection install and handler state into a `globalThis` object and skips repeated `installUnhandledRejectionHandler()` calls.

## Best Possible Solution

Rebase a minimal install-once guard onto current `main`, preserve or migrate the existing handler registry key, add focused listener-count coverage, and provide redacted runtime proof.

## Reproduction Assessment

Yes, at source level: current `main` has startup call sites in `src/index.ts:91` and `src/cli/run-main.ts:677`, while `installUnhandledRejectionHandler()` unconditionally attaches a process listener at `src/infra/unhandled-rejections.ts:513`. I did not run a live repro because this review is read-only.

## Solution Assessment

No, not as currently branched. The install-once guard is the right narrow direction, but the rebased patch must preserve or migrate from current `main`'s shipped `openclaw.unhandledRejection.handlers` registry key.
