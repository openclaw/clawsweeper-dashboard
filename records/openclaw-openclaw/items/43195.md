## Summary

Keep open: the installed guard is useful, but the PR is stale and currently unsafe because it moves the handler registry away from the shipped shared plugin-runtime key, which can split suppressors across module copies.

## What This Changes

The PR moves unhandled-rejection install and handler state into a Symbol.for-backed global object and skips repeated process.on("unhandledRejection") installation.

## Best Possible Solution

Land a rebased version that adds idempotent install state while retaining the existing handler registry contract and current fatal/transient classifications.

## Reproduction Assessment

Yes. Current main has an unconditional installUnhandledRejectionHandler() listener path, and a read-only Node probe confirmed duplicate unhandledRejection registrations are additive.

## Solution Assessment

No as written. The installed guard is the right direction, but the branch should preserve or bridge Symbol.for("openclaw.unhandledRejection.handlers") and add focused regression coverage after rebasing.
