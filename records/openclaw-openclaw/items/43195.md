## Summary

Keep open: current main still installs the unhandled-rejection listener unconditionally, so the install-once part is not implemented. The PR is not mergeable as-is because it conflicts with current main's shipped global handler registry key and lacks required after-fix runtime proof.

## What This Changes

The PR moves unhandled-rejection install and handler state into a shared globalThis object and makes installUnhandledRejectionHandler return after the first listener install.

## Best Possible Solution

Rebase a minimal install-once guard onto current main, preserve or migrate the existing handler registry key, add focused listener-count coverage, and provide redacted runtime proof.

## Reproduction Assessment

Yes, at source level: calling installUnhandledRejectionHandler() repeatedly on current main reaches process.on("unhandledRejection", ...) each time because there is no installed-state guard. I did not run a live repro because this review is read-only.

## Solution Assessment

No, not as currently branched. The install-once guard is the right narrow direction, but the rebased patch must preserve or migrate current main's shipped openclaw.unhandledRejection.handlers registry key.
