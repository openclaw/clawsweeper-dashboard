## Summary

Keep open: the underlying install-idempotency problem still appears valid on current main, but this branch is conflicting and would need to preserve the shipped global handler registry contract before it can be merged. It also lacks required after-fix real behavior proof from the contributor.

## What This Changes

This PR changes `src/infra/unhandled-rejections.ts` to store unhandled-rejection install and handler state on `globalThis` and return early after the first `installUnhandledRejectionHandler()` call.

## Best Possible Solution

Rebase a minimal install-once guard onto current main, preserve or migrate from the existing handler registry key, add focused listener-idempotency coverage, and provide redacted runtime proof.

## Reproduction Assessment

Yes, at source level: current main has startup call sites in `src/index.ts:91` and `src/cli/run-main.ts:677`, while `installUnhandledRejectionHandler()` unconditionally attaches a process listener at `src/infra/unhandled-rejections.ts:513`. I did not run a live repro because this review is read-only.

## Solution Assessment

No, not as currently branched. The install-once guard is the right narrow fix, but the rebased patch must preserve or migrate from current main's shipped `openclaw.unhandledRejection.handlers` registry key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the shipped handler registry key:** `src/infra/unhandled-rejections.ts:10`
  - body: Current main already shares plugin rejection handlers through `Symbol.for("openclaw.unhandledRejection.handlers")`. Keep that Set key, or migrate from it, and add a separate global install flag.
