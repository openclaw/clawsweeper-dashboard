## Summary

Keep open: the install-once fix is still useful because current main still attaches a new `unhandledRejection` listener on every call, but this branch is dirty, conflicts with the shipped handler-registry key from the merged Bonjour/runtime-copy fix, and still lacks real behavior proof.

## What This Changes

The PR changes `src/infra/unhandled-rejections.ts` to store unhandled-rejection state on `globalThis` and skip duplicate `process.on("unhandledRejection")` installs.

## Best Possible Solution

Rebase a minimal install-once guard onto current main, preserve or migrate from the shipped global handler Set key, add focused listener-idempotency coverage, and provide redacted runtime proof.

## Reproduction Assessment

Yes, at source level: `src/index.ts` and `src/cli/run-main.ts` both call `installUnhandledRejectionHandler()`, while current main unconditionally attaches a new listener. I did not execute a live repro because this review is read-only.

## Solution Assessment

No, not as currently branched. The install-once guard is the right narrow fix, but the rebased patch must preserve current main’s shipped registry-sharing contract from https://github.com/openclaw/openclaw/pull/71668.
