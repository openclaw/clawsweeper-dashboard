## Summary

Keep open for maintainer handling: the duplicate-listener problem is still source-reproducible on current main, but this branch is merge-conflicting, lacks real behavior proof, and would replace a shipped global handler-registry key that plugin runtime copies rely on.

## What This Changes

The PR moves unhandled-rejection install and handler state into a shared globalThis object and makes installUnhandledRejectionHandler return early after the first install.

## Best Possible Solution

Land an updated narrow patch that preserves the shipped handler registry key, adds a separate idempotent listener-install guard, and covers both repeated install and staged plugin runtime registration.

## Reproduction Assessment

Yes at source level: current main calls installUnhandledRejectionHandler() from both the direct entrypoint and runCli(), while the installer still unconditionally attaches a process listener. I did not run a listener-count repro because this review was read-only.

## Solution Assessment

No, not as currently branched: the install-once guard is the right narrow fix, but replacing the shipped handler registry key is not compatibility-safe. The safer path is a separate install guard that leaves the handler Set contract intact.
