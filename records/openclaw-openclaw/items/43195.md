## Summary

Keep this PR open: current main still has the duplicate-listener problem, but the registry-sharing part is already superseded by merged work and the branch now needs a careful rebase plus real behavior proof before merge.

## What This Changes

The PR changes `src/infra/unhandled-rejections.ts` to store unhandled-rejection handler state on `globalThis` and skip duplicate `process.on("unhandledRejection")` installs.

## Best Possible Solution

Rebase a minimal install-once guard onto current main, preserve or migrate from the shipped global handler Set key, add focused listener-idempotency coverage, and include redacted runtime proof.

## Reproduction Assessment

Yes, at source level: `src/index.ts` and `src/cli/run-main.ts` both call `installUnhandledRejectionHandler()`, while current `src/infra/unhandled-rejections.ts` always attaches a new `unhandledRejection` listener. I did not execute a live repro because the review is read-only.

## Solution Assessment

No, not as currently branched. The install-once guard is the right narrow fix, but the rebase must preserve current main’s shipped registry-sharing contract from https://github.com/openclaw/openclaw/pull/71668.
