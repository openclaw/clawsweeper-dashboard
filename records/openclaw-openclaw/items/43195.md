## Summary

Keep open. Current main still lacks an idempotent installer guard, so the PR addresses a real duplicate-listener path, but the branch is stale against current global handler-registry work and should be repaired before merge.

## What This Changes

The PR moves unhandled-rejection handler state into a Symbol.for-backed globalThis object and adds an installed flag so installUnhandledRejectionHandler() returns after the first process-level listener is registered.

## Best Possible Solution

Land a current-main repair that adds a shared installed flag around the process listener while preserving the existing handler registries, then cover repeated installs with a focused regression test.

## Reproduction Assessment

Yes. Current main has two startup call sites and an unconditional installer, and a Node probe confirms repeated process.on("unhandledRejection") registrations add duplicate listeners.

## Solution Assessment

No as written against current main. The installed flag is the right mechanism, but the rebased fix should preserve the existing global handler Set keys instead of moving handlers behind a new incompatible state key.

## Review Findings

Overall correctness: patch is incorrect
