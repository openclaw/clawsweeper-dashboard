## Summary

Keep open: current main still registers a new `unhandledRejection` listener on every `installUnhandledRejectionHandler()` call, so the install-idempotency part remains meaningful. Current main already globalizes handler registries in a different shape, so this branch needs rebase/proof rather than cleanup closure.

## What This Changes

The branch moves unhandled-rejection handler state into a shared `globalThis` object and adds an installed flag so repeated installs do not add duplicate process listeners.

## Best Possible Solution

Rebase a minimal install guard onto current main, preserve the existing global handler registries, add focused listener-idempotency coverage, and include redacted runtime proof.

## Reproduction Assessment

Yes, at source level: current main's `installUnhandledRejectionHandler()` unconditionally calls `process.on("unhandledRejection", ...)`, so two calls attach two listeners. I did not run a live repro because this review is read-only.

## Solution Assessment

No, not as-is. The install-once direction is narrow and reasonable, but current main already globalized the registries differently, so the maintainable merge path is a rebased guard with focused coverage and proof.
