## Summary

Keep this PR open. Current main still has the double-install behavior because the shared handler registries are globalized but the installer itself has no shared installed flag or early return, so the PR’s central fix remains useful after rebase and focused regression coverage.

## What This Changes

The PR moves unhandled-rejection handler state into a Symbol-backed globalThis object and makes installUnhandledRejectionHandler() return after the first process-level listener is installed.

## Best Possible Solution

Land the idempotent installer guard on current main with shared installer state, keep the existing global handler registries and fatal-error behavior intact, and add a test proving repeated installUnhandledRejectionHandler() calls do not add more unhandledRejection listeners.

## Reproduction Assessment

Yes. Current main has two startup call sites for installUnhandledRejectionHandler(), and a read-only Node probe shows duplicate process.on("unhandledRejection") calls increase listenerCount.

## Solution Assessment

Yes. A shared installed flag checked before process.on(...) is the narrowest maintainable fix; the safer landing path is a current-main rebase that preserves the existing uncaught-exception registry and adds regression coverage.
