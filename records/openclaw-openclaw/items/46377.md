## Summary

Keep open: the security hardening target is real, but this branch applies strict guarded fetch directly to provider POSTs, which can break or bypass provider proxy/dispatcher policy; current main already has the Gemini header-auth part, and the PR still lacks real behavior proof.

## What This Changes

The PR wraps Anthropic/Gemini native PDF calls in strict guarded fetch, moves Gemini auth to `x-goog-api-key`, and releases guarded-fetch dispatchers.

## Best Possible Solution

Route native PDF provider POSTs through a provider-transport-aware guarded-fetch helper that preserves proxy/dispatcher policy, timeouts, Gemini header auth, Google base URL normalization, audit context, and focused SSRF/proxy regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still passes model base URLs into the native PDF helpers and both helpers issue raw `fetch` calls to derived URLs; I did not run live provider credentials or private-IP probes.

## Solution Assessment

No. The security direction is right, but the PR should reuse or add a provider-transport-aware guarded-fetch path instead of applying strict guarded fetch directly to provider POSTs.
