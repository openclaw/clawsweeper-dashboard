## Summary

Keep open: current main still has source-level SSRF exposure in the native Anthropic/Gemini PDF fetch path, but this PR is not merge-ready because its strict guarded-fetch integration can bypass the provider transport env-proxy path and the contributor has not provided real after-fix behavior proof.

## What This Changes

The PR wraps Anthropic and Gemini native PDF HTTP calls with the SSRF guard, moves Gemini API authentication to `x-goog-api-key`, and adds guarded-fetch release cleanup.

## Best Possible Solution

Rework or replace the branch with a provider transport-aware guarded-fetch integration that preserves Gemini header auth, timeout behavior, proxy policy, audit context, and focused SSRF regression coverage.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: pass a private/internal `baseUrl` into the native Anthropic or Gemini PDF provider path and the URL is fed into raw `fetch` without `fetchWithSsrFGuard`; I did not run a live provider repro in this read-only review.

## Solution Assessment

No. The security direction is right, but hardcoding `withStrictGuardedFetchMode` is not the best implementation because the provider transport path already has proxy-aware SSRF handling that this code should preserve.
