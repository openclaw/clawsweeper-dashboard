## Summary

Keep open: the underlying SSRF hardening is still valid and not fully implemented on current main, but this PR cannot merge as-is because strict guarded fetch drops provider proxy/dispatcher policy and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR wraps Anthropic/Gemini native PDF HTTP calls in strict guarded fetch, moves Gemini auth from a query param to `x-goog-api-key`, and releases guarded-fetch dispatchers.

## Best Possible Solution

Route native PDF provider POSTs through a provider-transport-aware guarded-fetch helper that preserves proxy/dispatcher policy, timeouts, Gemini header auth, Google base URL normalization, audit context, and focused SSRF/proxy regression coverage.

## Reproduction Assessment

Yes, at source level. Current main passes `model.baseUrl` into `anthropicAnalyzePdf` and `geminiAnalyzePdf`, and both provider functions still call raw `fetch` on derived URLs; I did not run live provider credentials.

## Solution Assessment

No. The security direction is right, but `withStrictGuardedFetchMode` is not the best implementation because provider traffic must preserve trusted env-proxy and explicit dispatcher policy where configured.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve provider proxy routing:** `src/agents/tools/pdf-native-providers.ts:68-69`
  - body: Using `withStrictGuardedFetchMode` here forces native PDF provider POSTs into strict mode, which only uses env proxy when the managed OpenClaw proxy lifecycle is active. Model-provider fetches currently switch to trusted env-proxy mode when `HTTP_PROXY`/`HTTPS_PROXY` applies and no explicit dispatcher policy is configured, so proxy-only or proxy-audited deployments can fail or bypass expected egress routing; the same issue applies to the Gemini wrapper later in this diff.
  - confidence: 0.88
