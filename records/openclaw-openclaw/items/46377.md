## Summary

Keep this PR open, but do not merge it as-is: current main still has the native-PDF SSRF gap, while the Gemini query-string key portion has already been handled on main and the proposed strict guard path risks breaking proxy-governed provider traffic.

## What This Changes

The PR wraps Anthropic and Gemini native PDF HTTP calls in `fetchWithSsrFGuard(withStrictGuardedFetchMode(...))`, moves Gemini auth to `x-goog-api-key`, and adds guarded-fetch release cleanup.

## Best Possible Solution

Rebase onto current main and implement native-PDF SSRF hardening with the provider transport-aware guarded-fetch pattern, preserving Gemini header auth, timeout behavior, proxy policy, audit context, and focused regression coverage.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: pass an internal/private `baseUrl` into the native Anthropic or Gemini PDF provider path and it is fed into raw `fetch` without `fetchWithSsrFGuard`.

## Solution Assessment

No. The security direction is right, but this implementation hardcodes strict guarded fetch without matching the provider transport proxy/timeout pattern and is stale against current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

- **[P1] Preserve proxy-aware provider routing:** `src/agents/tools/pdf-native-providers.ts:68`
