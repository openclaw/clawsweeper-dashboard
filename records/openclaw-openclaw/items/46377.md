## Summary

Keep open: the PR targets a real native-PDF SSRF gap that still exists on current main, but the proposed strict guarded-fetch integration can regress env-proxy-aware provider transport and the external contributor has not provided real after-fix proof.

## What This Changes

The branch wraps Anthropic and Gemini native PDF HTTP calls in the SSRF guard, moves Gemini API authentication to `x-goog-api-key`, and adds guarded-fetch release cleanup.

## Best Possible Solution

Rework the native PDF path to use a provider transport-aware guarded-fetch integration that preserves Gemini header auth, timeout behavior, Google base URL normalization, audit context, and focused SSRF/proxy regression coverage.

## Reproduction Assessment

Yes, at source level. Current main builds Anthropic and Gemini native-PDF request URLs from provider/base URL input and sends them through raw `fetch`; I did not run a live provider repro in this read-only review.

## Solution Assessment

No. The security direction is right, but hard-coding `withStrictGuardedFetchMode` is not the best implementation because the provider transport path already has proxy-aware guarded-fetch behavior that native PDF calls should preserve.
