## Summary

Keep this PR open: it addresses a real native-PDF SSRF gap that still exists on current main, but the branch is stale and the proposed guarded-fetch calls would regress proxy-only provider deployments and drop the current native PDF request timeout.

## What This Changes

The PR updates the native Anthropic and Gemini PDF provider calls to use the repository SSRF guard, moves the Gemini API key from a query parameter to a header, and adds release cleanup around the guarded fetches.

## Best Possible Solution

Land a refreshed patch on current main that moves both native PDF provider calls onto guarded fetches while preserving timeout, Gemini transport normalization, and the existing proxy-aware guarded-fetch contract.

## Reproduction Assessment

Yes, source-reproducible. In a proxy-only environment with HTTPS_PROXY/HTTP_PROXY required, the PR's withStrictGuardedFetchMode path still performs local DNS/pinned direct dispatch, and the diff also visibly removes the AbortSignal timeout currently asserted by native PDF provider tests.

## Solution Assessment

No. The SSRF hardening direction is correct, but the maintainable fix should use the same proxy-aware guarded-fetch mode selection as other provider calls and preserve current main's timeout and Gemini transport/header fixes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve proxy-aware guarded fetch mode:** `src/agents/tools/pdf-native-providers.ts:67-68`
  - body: This wraps the native PDF POSTs in withStrictGuardedFetchMode, which forces fetchWithSsrFGuard through local DNS pinning and a direct dispatcher. Current provider HTTP paths switch to trusted_env_proxy when HTTP(S)_PROXY applies because strict mode fails in proxy-only deployments, so native Anthropic/Gemini PDF calls would regress there. Use the same shouldUseEnvHttpProxyForUrl/withTrustedEnvProxyGuardedFetchMode pattern before merging.
  - confidence: 0.9
- **[P2] Restore the native PDF request timeout:** `src/agents/tools/pdf-native-providers.ts:69-85`
  - body: Both current fetch calls pass AbortSignal.timeout(NATIVE_PDF_PROVIDER_FETCH_TIMEOUT_MS), but the guarded-fetch replacement does not pass signal or timeoutMs. That drops the bounded native PDF request behavior added on current main; thread timeoutMs or the abort signal through fetchWithSsrFGuard for both providers.
  - confidence: 0.88
