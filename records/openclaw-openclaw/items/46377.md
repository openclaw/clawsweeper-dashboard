## Summary

Keep open: the reported native-PDF SSRF gap is still relevant on current main, but this branch should not merge as-is because its strict guarded-fetch integration would regress provider proxy routing and it lacks required real behavior proof.

## What This Changes

The branch modifies `src/agents/tools/pdf-native-providers.ts` to wrap Anthropic and Gemini native PDF POSTs in `fetchWithSsrFGuard(withStrictGuardedFetchMode(...))`, move Gemini auth to `x-goog-api-key`, and call guarded-fetch `release()` in `finally` blocks.

## Best Possible Solution

Route native PDF provider POSTs through a provider transport-aware guarded-fetch helper that preserves env-proxy behavior, explicit dispatcher policy, timeout handling, Gemini header auth, Google base URL normalization, audit context, and focused SSRF/proxy regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still builds Anthropic and Gemini native-PDF request URLs from provider/base URL input and sends them through raw `fetch`; I did not run live provider credentials in this read-only review.

## Solution Assessment

No. The security direction is right, but hard-coding `withStrictGuardedFetchMode` is not the best implementation because the provider transport path already has proxy-aware guarded-fetch behavior that native PDF calls should preserve.
