## Summary

Keep this PR open, but do not merge it as-is. It targets a real native PDF provider SSRF surface, yet the patch routes provider traffic through direct strict guarded fetch instead of the provider transport policy, and the contributor has not provided after-fix real behavior proof.

## What This Changes

The PR changes native Anthropic and Gemini PDF analysis calls to use strict SSRF-guarded fetches, moves Gemini authentication from a query parameter to an `x-goog-api-key` header, and adds guarded-fetch cleanup.

## Best Possible Solution

Add a provider-transport-aware guarded fetch path for native PDF analysis that preserves SSRF blocking, provider proxy/dispatcher policy, native PDF timeouts, Gemini header auth, audit context, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main passes `model.baseUrl` into the native Anthropic/Gemini PDF helpers, and those helpers perform raw `fetch` calls against derived provider URLs; I did not run live provider credentials.

## Solution Assessment

No. The security direction is right, but direct `withStrictGuardedFetchMode` is not the narrow maintainable fix because native PDF provider traffic needs the same provider transport policy as the rest of model traffic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve provider transport routing:** `src/agents/tools/pdf-native-providers.ts:68-69`
  - body: Using `withStrictGuardedFetchMode` directly here sends native PDF provider POSTs outside the provider transport path. Provider fetches intentionally switch to trusted env-proxy mode when env proxy routing applies and no explicit dispatcher policy is set; this patch would pin direct dispatch for these PDF calls, breaking or bypassing controlled egress in proxy-only deployments. The Gemini call has the same issue later in the file.
  - confidence: 0.89
