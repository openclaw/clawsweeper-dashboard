## Summary

Keep this PR open: the underlying SSRF hardening is still relevant on current main, but the branch is stale and its strict guarded-fetch wiring would regress proxy-backed provider environments.

## What This Changes

The PR wraps native Anthropic and Gemini PDF HTTP calls with `fetchWithSsrFGuard`, moves Gemini credentials from the URL query to `x-goog-api-key`, and adds guarded-fetch cleanup.

## Best Possible Solution

Rework the current-main PDF provider calls through the proxy-aware guarded provider HTTP helper, preserving Gemini header auth, transport normalization, timeouts, provider-specific audit context, tests, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows both native PDF provider functions still accept a configurable `baseUrl` and dispatch with raw `fetch()`, so the SSRF exposure is clear without executing a network request.

## Solution Assessment

No. The PR points in the right direction, but forcing `withStrictGuardedFetchMode()` is not the best current fix because existing provider HTTP helpers deliberately auto-select trusted env proxy mode for proxy-backed deployments.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve proxy-aware guarded provider routing:** `src/agents/tools/pdf-native-providers.ts:68-70`
  - body: Using `withStrictGuardedFetchMode()` here forces native PDF requests onto strict pinned-DNS dispatch. Existing guarded-fetch tests assert strict mode ignores env proxy by default, while provider HTTP helpers now auto-upgrade to trusted env proxy mode for `HTTP_PROXY`/`HTTPS_PROXY` deployments. This would make Anthropic/Gemini PDF calls fail or bypass operator proxy routing where other provider calls still work; route these through the shared proxy-aware guarded provider helper instead.
  - confidence: 0.91

  - confidence: 0.86
