## Summary

Keep open. Current main still only partially satisfies the requested recovery guidance: direct Gateway startup names `openclaw doctor --fix` but omits the `openclaw.json.bak` fallback, and daemon start/restart preflight still uses generic `openclaw doctor` wording. The latest release tag has the same gap, and the linked implementation PRs are closed unmerged.

## What This Changes

Gateway invalid-config startup errors should suggest `openclaw doctor --fix` and mention `openclaw.json.bak` recovery when repair is still blocked.

## Best Possible Solution

Centralize the invalid-config recovery hint for direct Gateway startup validation and daemon lifecycle preflight, include both the direct repair command and a cautious `openclaw.json.bak` restore fallback, and cover both user-facing surfaces with focused tests without adding automatic rollback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current direct startup recovery tests assert the incomplete message, and daemon start/restart preflight constructs the generic doctor hint when `readConfigFileSnapshot()` returns an invalid snapshot.

## Solution Assessment

Yes. The narrowest maintainable fix is a shared message/helper update plus targeted assertions for direct startup and daemon preflight; this should remain an error-text recovery guidance fix, not a change to config rollback policy.
