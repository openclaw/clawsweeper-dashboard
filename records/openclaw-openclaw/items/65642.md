## Summary

Keep this PR open. Current main still lacks the requested OpenResponses internal-tool SSE behavior, so the contribution is not obsolete; however, the proposed patch needs maintainer/security review because it bypasses the existing sanitized tool-result event payload and streams raw tool results through `/v1/responses`.

## What This Changes

The PR adds OpenResponses SSE handling for internal tool start/result events, a raw-result side channel on agent events, `function_call_output` output schema support, AI SDK-style input normalization, and focused gateway/agent-event tests.

## Best Possible Solution

Land this PR or an equivalent Gateway change after tightening the raw-output contract: bridge internal `stream: "tool"` events into stable OpenResponses output items, keep terminal snapshots consistent, preserve scoped/redacted behavior for non-full-trust HTTP callers, and update the public OpenResponses docs plus changelog.

## Reproduction Assessment

Yes. A focused gateway test can mock an agent run that emits `stream: "tool"` start/result events; current main will stream only assistant/lifecycle output because the OpenResponses listener ignores tool events, while the PR adds coverage for that path.

## Solution Assessment

No, not as currently proposed. The event-shaping approach is the right narrow area, but raw `function_call_output` should be gated to an explicit full-trust/secret-capable caller or fall back to the already sanitized tool result before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate raw tool output before streaming it:** `src/gateway/openresponses-http.ts:1077-1078`
  - body: The `/v1/responses` endpoint is authorized as `chat.send`, which can be only `operator.write` in identity-bearing HTTP modes, but this path prefers the raw tool result over the sanitized `evt.data.result`. Current tool-result tests explicitly redact API keys and bearer tokens before event emission, so this would stream secrets in `function_call_output` to callers that previously only saw redacted tool payloads. Gate raw output to an explicit full-trust/secret scope or use the sanitized result here.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete raw-tool-result exposure path on an authenticated HTTP endpoint whose identity-bearing mode can run with narrowed scopes.

Concerns:

- **[high] Raw tool results bypass existing redaction:** `src/gateway/openresponses-http.ts:1077`
  - body: Current main sanitizes tool args/results before emitting shared tool events and has regression coverage for credential redaction. The PR adds a hidden raw-result side channel and streams that raw value as OpenResponses `function_call_output`; this should be explicitly scoped or otherwise redacted before merge.
  - confidence: 0.86
