## Summary

Keep open: the feature is still useful, but current head is not merge-ready because it streams raw internal tool results over `/v1/responses` despite the existing sanitized tool-event boundary, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds OpenResponses SSE output items for internal tool calls/results, adds `function_call_output` output schema support, normalizes response-shaped inputs, and carries raw tool results through agent events for HTTP streaming.

## Best Possible Solution

Implement internal tool SSE output using sanitized tool results by default, or add an explicit reviewed trust gate for raw output, then update coverage/docs as needed and require redacted real OpenResponses runtime proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature PR. Source inspection confirms current main lacks internal tool-call SSE output behavior and lacks `function_call_output` in the response output schema.

## Solution Assessment

No, not as currently proposed. The Gateway SSE listener is the right surface, but the result output should use sanitized event data by default or an explicit reviewed trust gate before exposing raw tool payloads to HTTP clients.

## Review Findings

Overall correctness: patch is incorrect

- **[P1] Gate raw tool output before streaming it:** `src/gateway/openresponses-http.ts:1082-1084`
