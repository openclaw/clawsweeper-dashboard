## Summary

Keep open. Current main still lacks the requested OpenResponses internal-tool SSE bridge, and the PR is useful, but it should not merge while it streams raw internal tool results to `/v1/responses` callers that can be scoped to `operator.write`; current shared tool events deliberately carry sanitized results, and the Gateway HTTP auth contract distinguishes full shared-secret access from narrowed trusted-identity scopes.

## What This Changes

The PR adds OpenResponses SSE output items for internal tool start/result events, a non-enumerable raw-tool-result side channel on agent events, `function_call_output` output schema support, AI SDK-style input normalization, and gateway/agent-event regression tests.

## Best Possible Solution

Land this PR or an equivalent Gateway change after tightening the output contract: bridge internal `stream: "tool"` start/result events into stable OpenResponses output items and terminal snapshots, but default to sanitized tool results unless the caller is explicitly full-trust or otherwise authorized for raw/secret-bearing tool payloads; update the public OpenResponses docs and changelog before merge.

## Reproduction Assessment

Yes. A focused gateway test can mock an agent run that emits `stream: "tool"` start/result events while streaming `/v1/responses`; current main's listener only handles assistant/lifecycle events, while the PR adds coverage for internal tool events.

## Solution Assessment

No, not as currently proposed. The event-shaping location is the narrow maintainable place to solve the feature, but preferring raw tool results before a full-trust or secret-capable authorization check is not the safest contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate raw tool output before streaming it:** `src/gateway/openresponses-http.ts:1077-1078`
  - body: The `/v1/responses` endpoint is authorized as `chat.send`, which trusted identity-bearing HTTP callers can satisfy with only `operator.write`, but this path prefers `getAgentEventToolRawResult(evt)` over the sanitized `evt.data.result` for `function_call_output`. Current tool event production redacts API keys, bearer tokens, and nested credential fields before publishing shared tool results, so this would stream secrets to callers that previously only received redacted payloads. Gate raw output to a full-trust/secret-capable caller or use the sanitized result here.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces a raw-tool-result exposure path on an authenticated HTTP endpoint whose trusted identity-bearing mode can run with narrowed scopes.
