## Summary

Keep open: current main still lacks the requested OpenResponses internal-tool SSE output path, and this PR remains the canonical implementation candidate, but it should not merge while raw tool results can be streamed without an explicit authorization/redaction contract.

## What This Changes

The PR adds OpenResponses SSE output items for internal tool start/result events, a raw-tool-result side channel, `function_call_output` output schema support, AI SDK-style input normalization, and gateway/agent-event regression tests.

## Best Possible Solution

Land this PR or an equivalent Gateway change after defaulting internal tool output to sanitized results unless the caller is explicitly full-trust or secret-capable, with OpenResponses docs and changelog updated for the chosen contract.

## Reproduction Assessment

Not applicable as a bug reproduction. The feature gap is source-proven on current main because the OpenResponses SSE listener lacks an internal tool-event branch and the output schema lacks `function_call_output`.

## Solution Assessment

No, not as currently proposed. The Gateway SSE listener is the right narrow place to shape tool events, but preferring raw tool results before an authorization or redaction gate is not the safest maintainable contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate raw tool output before streaming it:** `src/gateway/openresponses-http.ts:1082-1083`
  - body: The `/v1/responses` endpoint is authorized through `chat.send`, and trusted identity-bearing callers can satisfy that with narrowed `operator.write` scopes. This branch prefers `getAgentEventToolRawResult(evt)` over the already sanitized `evt.data.result`, so `function_call_output` can expose secrets that current tool-event redaction intentionally removes. Use the sanitized result by default or gate raw output to a full-trust/secret-capable caller.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR exposes a raw tool-result side channel through OpenResponses SSE without an authorization or redaction gate.

Concerns:

- **[high] Raw tool results bypass existing redaction:** `src/gateway/openresponses-http.ts:1082`
  - body: Current main emits sanitized tool results on shared agent events and has coverage for credential redaction, but the PR streams the non-enumerable raw result as `function_call_output`; this can expose API keys, bearer tokens, or nested credential fields to callers that previously received redacted payloads.
  - confidence: 0.86
