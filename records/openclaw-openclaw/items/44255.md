## Summary

Keep open. Current main does not implement outbound-message redaction, but this PR is not safe to merge because it leaves the primary plugin delivery path unredacted and adds redaction logic with concrete leak and false-positive risks. The broader security guardrail PR #72733 is related but still open, so it does not supersede this PR yet.

## What This Changes

The PR adds runtime sensitive-value registration from config and environment variables, initializes that redaction during CLI startup, applies it to outbound send and poll fallback content, and adds docs/tests for outbound message redaction.

## Best Possible Solution

Keep this in a maintainer/security lane and rework the design around a central sanitized action-param boundary before plugin dispatch and fallback delivery. Preserve existing logging redaction semantics and tests, use narrowly reviewed secret metadata or schema hints for discovery, and align with the broader guardrail direction before merging.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a message send or poll through Slack, Discord, or Telegram containing a known secret: the PR computes redacted fallback variables, but plugin dispatch receives unchanged `ctx.params`, and bundled plugin handlers read message and poll text from those raw params. I did not run tests because this review is read-only.

## Solution Assessment

No. The proposed implementation redacts fallback delivery and transcript mirroring but misses the dominant plugin delivery path; the safer fix is to sanitize cloned action params once before plugin dispatch and then feed the same sanitized values into fallback delivery with targeted plugin regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass redacted params to plugin send actions:** `src/infra/outbound/outbound-send-service.ts:108-110`
  - body: `redactedMessage` is computed here, but `tryHandleWithPluginAction` still receives the original `ctx.params`. Slack, Discord, Telegram, and other plugin handlers read `message` from those params, so plugin-based sends still deliver the raw secret.
  - confidence: 0.95
- **[P1] Route redacted poll params through plugins:** `src/infra/outbound/outbound-send-service.ts:184-186`
  - body: The new redacted poll question/options are only used by the core fallback. Plugin poll handlers run first and read `pollQuestion`/`pollOption` from the unchanged action params, so secrets in poll text still leak whenever a plugin handles the poll.
  - confidence: 0.94
- **[P1] Use a function replacer for dynamic masks:** `src/logging/redact.ts:112`
  - body: Passing `masked` as a string replacement lets `$&`, `$'`, and similar sequences in the preserved prefix/suffix be interpreted by JavaScript. A discovered secret containing those characters can reinsert the full match; use a function replacer so the mask is emitted literally.
  - confidence: 0.91
- **[P2] Tighten auth key matching:** `src/logging/redact-init.ts:20`
  - body: The `/auth/i` key pattern also matches benign keys like `author`, `authenticate`, and `authorization`. Long non-secret values under those keys become globally redacted from outbound messages, which can silently corrupt legitimate content.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Security review needs attention: the patch is intended as hardening but still leaks secrets through plugin delivery and adds dynamic-redaction edge cases.

Concerns:

- **[high] Plugin delivery bypasses outbound redaction:** `src/infra/outbound/outbound-send-service.ts:108`
  - body: The PR computes redacted send/poll text but does not sanitize `ctx.params` before plugin dispatch; bundled plugin handlers read raw params and can send secrets to external services.
  - confidence: 0.95
- **[high] Dynamic replacement can reinsert matched secrets:** `src/logging/redact.ts:112`
  - body: Using a string replacement for a mask that may contain `$` replacement metacharacters can expand back to the full matched secret instead of the intended mask.
  - confidence: 0.91
- **[medium] Broad secret discovery can redact benign data:** `src/logging/redact-init.ts:20`
  - body: The `/auth/i` scan treats author-like metadata and other non-secret auth substrings as sensitive, adding their values to a global redaction set and altering user-visible outbound content.
  - confidence: 0.86
