## Summary

Keep open. Current main does not implement outbound channel redaction, and this PR’s security goal is valid, but the proposed patch is not safe to merge: it leaves the primary plugin send/poll paths unredacted, introduces a replacement-string leak in dynamic redaction, and uses broad config/env scanning that can redact benign values. The related security guardrail PR #72733 is still open and unmerged, so it does not supersede this work yet.

## What This Changes

The PR adds dynamic logging redaction APIs and config/env secret discovery, initializes them from CLI startup, applies redaction to outbound send and poll content, and adds security redaction docs and tests.

## Best Possible Solution

Keep this PR open for maintainer/security rework or explicit replacement. A viable implementation should sanitize cloned send and poll action params before gateway plugin dispatch, direct plugin dispatch, and core fallback delivery; preserve existing logging redaction semantics and tests; and use reviewed secret metadata or narrowly scoped patterns rather than broad key-name scanning.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is message(action=send) or poll through Slack, Telegram, or Discord with a known secret: the PR computes redacted local variables but plugin dispatch still receives ctx.params unchanged, and plugin handlers read those raw fields. I did not run tests because this review is read-only.

## Solution Assessment

No. The proposed patch redacts only values used by the core fallback and some transcript mirroring, so it misses the dominant plugin delivery paths; the safer approach is a central sanitized action-param boundary with regression tests for bundled plugin sends and polls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass redacted params to plugin send actions:** `src/infra/outbound/outbound-send-service.ts:108-110`
  - body: `redactedMessage` is computed here, but the plugin path still receives the original `ctx.params`. Slack, Telegram, Discord, and other plugin handlers read message text from those params, so plugin-based sends still deliver the raw secret and the security fix is ineffective for the primary path.
  - confidence: 0.95
- **[P1] Route redacted poll params through plugins:** `src/infra/outbound/outbound-send-service.ts:184-186`
  - body: The new `redactedQuestion` and `redactedOptions` are only used by the core fallback. Plugin poll handlers run first and read `pollQuestion`/`pollOption` from the unchanged action params, so secrets in poll text still leak whenever a plugin handles the poll.
  - confidence: 0.94
- **[P1] Use a function replacer for dynamic masks:** `src/logging/redact.ts:112`
  - body: Passing `masked` as a string replacement lets `$&`, `$'`, and similar sequences in the preserved prefix/suffix be interpreted by JavaScript. A discovered secret containing those characters can reinsert the full match; use a function replacer so the mask is emitted literally.
  - confidence: 0.91
- **[P2] Tighten auth key matching:** `src/logging/redact-init.ts:12-21`
  - body: The `/auth/i` key pattern also matches benign keys like `author`, `authenticate`, and `authorization`. Any long string under those keys becomes globally redacted from outbound messages, which can silently corrupt legitimate content.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch is intended as security hardening but still leaks secrets through plugin delivery and introduces a dynamic replacement edge case that can re-expose a matched secret.

Concerns:

- **[high] Plugin delivery bypasses outbound redaction:** `src/infra/outbound/outbound-send-service.ts:108`
  - body: The PR computes redacted send/poll text but does not sanitize ctx.params before plugin dispatch; bundled plugin handlers read raw params and can send secrets to external services.
  - confidence: 0.95
- **[high] Dynamic replacement can reinsert matched secrets:** `src/logging/redact.ts:112`
  - body: Using a string replacement for a mask that may contain `$` replacement metacharacters can expand back to the full matched secret instead of the intended mask.
  - confidence: 0.91
- **[medium] Broad secret discovery can redact benign data:** `src/logging/redact-init.ts:20`
  - body: The `/auth/i` key scan treats non-secret fields such as author-like metadata as sensitive and stores their values in the global redaction set, which can alter outbound user-visible content.
  - confidence: 0.86
