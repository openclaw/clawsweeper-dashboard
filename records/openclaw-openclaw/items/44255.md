## Summary

## What This Changes

The branch adds dynamic secret discovery/redaction APIs, initializes them during CLI startup, applies redaction to outbound send/poll fallback paths, and adds docs/tests.

## Best Possible Solution

Keep the remaining delivery-layer sanitizer work on https://github.com/openclaw/openclaw/pull/51668, with focused review, tests, and live channel proof before merge.

## Reproduction Assessment

Yes, at source level: plugin-handled send and poll actions still receive raw `ctx.params` while this PR only redacts fallback values. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No. The branch redacts too late and only part of the delivery path; the safer direction is a gateway-layer sanitizer before channel delivery, as tracked by https://github.com/openclaw/openclaw/pull/51668.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass redacted params into plugin sends:** `src/infra/outbound/outbound-send-service.ts:106`
  - body: `redactedMessage` is computed here, but the plugin send path below still dispatches the original `ctx.params`. Telegram, Discord, and Slack handlers read message text from those params, so raw secrets can still be sent externally.
  - confidence: 0.95
- **[P1] Route redacted poll params through plugins:** `src/infra/outbound/outbound-send-service.ts:181-182`
  - body: The redacted poll question/options are only used by fallback delivery. Plugin poll handlers run first with unchanged params, so secrets in poll text can still leak through plugin channels.
  - confidence: 0.94
- **[P1] Emit dynamic masks with a function replacer:** `src/logging/redact.ts:114`
  - body: Passing `masked` as a string replacement lets `$&`, `$'`, and related replacement tokens in preserved secret prefixes or suffixes expand. Use a function replacer so the mask is emitted literally.
  - confidence: 0.91
- **[P2] Tighten the auth key pattern:** `src/logging/redact-init.ts:20`
  - body: The `/auth/i` discovery rule also matches benign keys such as `author` and `authorization`, which can register non-secret strings and unexpectedly redact normal outbound content.
  - confidence: 0.88
- **[P2] Use one redaction length threshold for env discovery:** `src/logging/redact-init.ts:67`
  - body: Environment scanning still requires length 18 while the dynamic redactor accepts length 12, so env-backed custom secrets from 12 to 17 characters can be skipped and sent unredacted.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch is intended as security hardening, but the diff leaves plugin delivery unprotected and adds dynamic-redaction edge cases.

Concerns:

- **[high] Plugin delivery bypasses outbound redaction:** `src/infra/outbound/outbound-send-service.ts:106`
  - body: The PR redacts fallback send/poll values but does not sanitize `ctx.params` before plugin dispatch; bundled plugin handlers read raw params and can send secrets to external services.
  - confidence: 0.95
- **[high] Dynamic replacement can reinsert matched secrets:** `src/logging/redact.ts:114`
  - body: Using a string replacement for a mask derived from secret content lets JavaScript replacement metacharacters expand instead of being emitted literally.
  - confidence: 0.91
- **[medium] Broad discovery can redact benign content:** `src/logging/redact-init.ts:20`
  - body: The broad auth-key scan can register non-secret config values and alter outbound messages unexpectedly.
  - confidence: 0.86
