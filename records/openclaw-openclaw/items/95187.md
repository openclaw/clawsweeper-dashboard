## Summary

Keep open: this PR targets a real Anthropic OAuth stale-header bug, but it is not merge-ready because the fallback can still emit a fabricated `claude-cli/0.0.0` when package metadata is not resolvable, and the PR does not prove a patched OpenClaw request path.

## What This Changes

The PR replaces the hardcoded Claude Code version in both Anthropic OAuth header builders with duplicated `createRequire` lookups of `@anthropic-ai/claude-code/package.json` and a `0.0.0` fallback.

## Best Possible Solution

Use one shared version resolver that matches OpenClaw's supported external Claude CLI packaging, never emits fabricated identity headers, covers both builders with focused tests, and includes redacted OpenClaw runtime or network proof.

## Reproduction Assessment

Yes for the source-level stale-header path: current main and v2026.6.8 still construct both Anthropic OAuth headers with `claude-cli/2.1.75`. I did not live-test Anthropic's credentialed rejection in this read-only review.

## Solution Assessment

No. Dynamic version resolution is the right direction, but this implementation duplicates resolver logic, silently falls back to `0.0.0`, and lacks patched OpenClaw request proof.
