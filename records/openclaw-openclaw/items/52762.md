## Summary

Keep this PR open: it targets a real Discord inbound dedupe hot path, but the submitted patch still leaves saturated caches paying full prune scans and changes shared map-pruning semantics in a way that can weaken webhook guard state.

## What This Changes

The PR changes `createDedupeCache().check()` to prune only after the cache exceeds `maxSize` and rewrites `pruneMapToMaxSize()` as a counted forward-pass deletion helper.

## Best Possible Solution

Land or replace this with a bounded pruning strategy that avoids full scans in both normal and saturated cache states, preserves or deliberately validates invalid numeric limits, and proves the shared helper plus Discord replay path with focused tests and changed-gate validation.

## Reproduction Assessment

Yes. A high-confidence static reproduction fills the proposed 5000-entry cache with unique keys and shows full-prune scans after saturation; a read-only Node comparison also reproduces the `NaN` pruning semantic change.

## Solution Assessment

No. The direction is useful, but this patch is not the best fix because it delays the full scan until saturation and changes shared helper behavior instead of making pruning bounded while keeping guard semantics stable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid full-cache prune once the cache saturates:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's five-minute, 5000-entry replay guard, a saturated unique-message stream still rescans roughly the whole cache on every new key, so the delay this PR targets can continue on busy servers.
  - confidence: 0.91
- **[P2] Preserve invalid-limit pruning behavior:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN` and the loop never breaks, deleting every entry. Current behavior leaves the map untouched for `NaN`, and SDK webhook guard callers can propagate `NaN` after numeric normalization, so this can make guard state forget active keys.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff does not change CI, dependencies, secrets, or package resolution, but the shared map-pruning rewrite can clear webhook guard maps for invalid numeric limits.

Concerns:

- **[medium] Preserve webhook guard state for invalid limits:** `src/infra/map-size.ts:10`
  - body: The counted loop deletes every tracked entry when `maxSize` is `NaN`; in-repo webhook in-flight and rate-limit helpers can normalize malformed `maxTrackedKeys` input to `NaN` before pruning, which can clear active request or rate-limit tracking.
  - confidence: 0.84
