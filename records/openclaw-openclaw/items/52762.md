## Summary

Keep open. Current main and the latest release still have the eager prune path, so this PR is not obsolete, but the branch still has blocking correctness and security-relevant findings and lacks after-fix real behavior proof.

## What This Changes

The PR changes `createDedupeCache().check()` to prune only after exceeding `maxSize` and rewrites `pruneMapToMaxSize()` with a counted forward pass.

## Best Possible Solution

Revise or replace this with a bounded shared-helper fix that removes per-message full scans after saturation, preserves explicit invalid-limit semantics, updates focused infra and SDK regression tests, and includes redacted real Discord/performance proof.

## Reproduction Assessment

Yes for the PR defects: current source and read-only Node simulations show the proposed saturated guard still performs repeated full scans and the map rewrite clears `NaN`-limited maps. I did not establish a live Discord E2E replay of the original 33-122s delay.

## Solution Assessment

No. The direction is useful, but this patch shifts the full scan to saturated caches and changes shared helper semantics; a bounded prune strategy with explicit invalid-limit handling is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid saturated-cache full scans:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's 5-minute, 5000-entry replay guard, once unique inbound messages keep the cache saturated, every new key still scans about the whole cache, so the delay this PR targets can continue on the busiest servers.
  - confidence: 0.91
- **[P2] Preserve invalid-limit pruning semantics:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` is also `NaN`, so `i++ >= deleteCount` never becomes true and the loop deletes every entry. Current behavior leaves maps untouched for `NaN`, and webhook guard callers can pass `NaN` after numeric normalization, so active guard state can be cleared.
  - confidence: 0.88
