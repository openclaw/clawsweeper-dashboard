## Summary

Keep open: the PR targets a real Discord hot-path concern, but the current branch still has blocking functional and security-relevant findings, and current main has not implemented the lazy-prune behavior.

## What This Changes

The PR changes `createDedupeCache().check()` to prune only when the cache exceeds `maxSize` and rewrites `pruneMapToMaxSize()` to delete a counted number of oldest entries.

## Best Possible Solution

Land or replace this with a focused helper fix that avoids per-message full scans in saturated cache states, preserves invalid numeric limit behavior, and proves the Discord hot path with targeted tests plus redacted real behavior evidence.

## Reproduction Assessment

Yes for the PR defects: source inspection and read-only Node simulations show the saturated-cache full-scan path and the invalid-limit semantic change. I did not establish a live Discord replay of the original 33-122 second delay.

## Solution Assessment

No. The direction is useful, but this patch is not the best fix because it keeps a full prune path in saturated caches and changes shared helper behavior for invalid numeric limits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid full-cache pruning after saturation:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's five-minute, 5000-entry replay guard, a busy server can keep the cache saturated, so each new message still scans roughly the whole cache and the targeted delay can continue in the busiest case.
  - confidence: 0.91
- **[P2] Preserve invalid-limit pruning behavior:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN`, so the counted loop never breaks and deletes every entry. Current behavior leaves maps untouched for `NaN`, and webhook guard callers can pass `NaN` after numeric normalization, so active guard state can be cleared.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff does not touch dependencies or CI, but the shared map-size rewrite can clear webhook guard state for invalid numeric limits.
