## Summary

Keep open: the performance problem is still present on current main and the latest release, but this PR has blocking correctness/security findings and lacks required real behavior proof from the external contributor.

## What This Changes

The PR changes `createDedupeCache().check()` to prune only when the map exceeds `maxSize` and rewrites `pruneMapToMaxSize()` to delete old entries with a counted forward pass.

## Best Possible Solution

Replace or revise this with a focused shared-helper fix that avoids full per-message scans in saturated cache states, preserves invalid numeric limit behavior, and proves the Discord hot path plus SDK guard behavior with targeted tests and real runtime evidence.

## Reproduction Assessment

Yes for the PR defects: source inspection of the proposed guard and map-size loop shows the saturated-cache full-prune path and the invalid-limit semantic change. I did not establish a live Discord replay of the original 33-122 second delay.

## Solution Assessment

No. The direction is useful, but this patch is not the best fix because it leaves the expensive saturated-cache path and changes shared helper semantics for invalid numeric limits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid full-cache prune once the cache saturates:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's five-minute, 5000-entry replay guard, a busy server can keep the cache saturated, so each new unique message still scans roughly the whole cache and the targeted delay can continue.
  - confidence: 0.91
- **[P2] Preserve invalid-limit behavior in map pruning:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN` and the loop never breaks, deleting every entry. Current behavior leaves maps untouched for `NaN`, and webhook guard callers can pass `NaN` after numeric normalization, so active guard state can be cleared.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff does not touch dependencies or CI, but the shared map-size rewrite can clear webhook guard state for invalid numeric limits.

Concerns:

- **[medium] Preserve webhook guard state for invalid limits:** `src/infra/map-size.ts:10`
  - body: The counted delete loop clears every entry when `maxSize` is `NaN`; webhook in-flight, rate-limit, and anomaly helpers can normalize malformed numeric config to `NaN` before pruning, which can make active guard maps forget tracked keys.
  - confidence: 0.84
