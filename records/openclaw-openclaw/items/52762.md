## Summary

Keep open. Current main and the latest release still run the eager dedupe prune path, so the underlying performance work is not obsolete, but this PR should not merge as-is because the saturated-cache scan, invalid-limit semantic change, and missing real behavior proof remain unresolved.

## What This Changes

The PR changes `createDedupeCache().check()` to prune only after exceeding `maxSize` and rewrites `pruneMapToMaxSize()` with a counted forward pass.

## Best Possible Solution

Land or replace this with a bounded shared-helper fix that removes per-message full scans in both normal and saturated cache states, preserves or explicitly validates invalid numeric limits, keeps public SDK helper semantics stable, and proves the helper plus Discord replay path with focused tests and real behavior evidence.

## Reproduction Assessment

Yes for the PR defects. Source inspection plus read-only Node simulations reproduce the saturated-cache full scan and the `NaN` map-clearing regression; I did not run a live Discord E2E performance replay.

## Solution Assessment

No. The direction is useful, but this patch delays the O(n) scan until saturation and changes shared map-pruning semantics instead of making pruning bounded and stable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid full-cache prune once the cache saturates:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's five-minute, 5000-entry replay guard, a saturated unique-message stream still scans about the whole cache on every new key, so the delay this PR targets can continue on busy servers.
  - confidence: 0.91
- **[P2] Preserve invalid-limit pruning semantics:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN` and this loop never breaks, deleting every entry. Current behavior is effectively a no-op for `NaN`, and SDK guard callers can pass `NaN` after numeric normalization, so active guard state can be cleared.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff does not touch CI, dependencies, secrets, or package resolution, but the shared map-size rewrite can clear webhook guard state for invalid numeric limits.

Concerns:
