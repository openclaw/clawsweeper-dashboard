## Summary

## What This Changes

The branch changes the shared dedupe cache to prune only when size exceeds `maxSize` and rewrites map pruning to delete oldest entries with a counted forward pass.

## Best Possible Solution

Land or replace this with a focused shared-helper fix that removes per-message full scans in normal and saturated cache states, preserves or explicitly validates invalid numeric limit handling, and proves the helper plus Discord replay path with targeted tests and real runtime evidence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid full-cache pruning once saturated:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever a new key pushes the map over `maxSize`. With Discord's five-minute, 5000-entry replay guard, a busy server can keep the cache saturated, so each new unique delivery still scans roughly the whole map and the targeted delay can continue.
  - confidence: 0.91
- **[P2] Preserve invalid-limit pruning behavior:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN` and `i++ >= deleteCount` never becomes true, so the loop deletes every key. Current behavior is a no-op for `NaN`, and webhook guard callers can normalize malformed numeric config to `NaN` before pruning.
  - confidence: 0.88
