## Summary

Keep this PR open, but do not merge it as-is: current main still has the eager dedupe prune path, while the PR leaves a saturated-cache O(n) scan and changes shared map-size semantics for invalid limits.

## What This Changes

The PR makes `createDedupeCache().check()` prune only after the map exceeds `maxSize` and rewrites `pruneMapToMaxSize()` to delete old entries with a counted forward pass.

## Best Possible Solution

Land or replace this with a focused shared-helper fix that avoids full per-message scans in normal and saturated cache states, preserves invalid numeric limit behavior, and proves the Discord replay path with targeted tests plus real runtime evidence.

## Reproduction Assessment

Yes for the PR defects: source inspection of the proposed guard and counted loop shows both the saturated-cache full-prune path and the invalid-limit semantic change. I did not establish a live Discord replay of the original 33-122 second delay.

## Solution Assessment

No. The direction is useful, but this patch is not the best fix because it leaves the expensive saturated-cache path and changes shared helper behavior for invalid numeric limits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid full-cache pruning after the cache saturates:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's five-minute, 5000-entry replay guard, a busy server can keep the cache at the cap, so each new unique delivery still scans roughly the whole map and the targeted delay can continue.
  - confidence: 0.91
- **[P2] Preserve invalid-limit pruning behavior:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` is `NaN` and `i++ >= deleteCount` never becomes true, so the loop deletes every key. Current behavior is a no-op for `NaN`, and webhook guard callers can normalize malformed numeric config to `NaN` before pruning.
  - confidence: 0.88

## Security Review
