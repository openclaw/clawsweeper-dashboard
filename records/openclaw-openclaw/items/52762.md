## Summary

This PR should stay open for normal review rather than cleanup closure: current `main` still has the eager dedupe prune behavior, but the proposed fix still has blocking correctness issues and lacks required real behavior proof for an external contributor PR.

## What This Changes

The PR changes `createDedupeCache().check()` to prune only after the cache exceeds `maxSize` and rewrites `pruneMapToMaxSize()` to delete older entries with a counted forward pass.

## Best Possible Solution

Revise or replace this with a bounded shared-helper fix that avoids per-message full scans in saturated caches, preserves explicit invalid-limit behavior, and proves the Discord hot path with focused tests plus redacted real behavior evidence.

## Reproduction Assessment

Yes for the PR defects: source inspection and read-only Node simulations reproduce the saturated-cache full-scan path and the invalid-limit semantic change. I did not establish a live Discord E2E replay of the original 33-122 second delay.

## Solution Assessment

No. The direction is useful, but this patch is not the best fix because it keeps the full scan in the saturated state and changes a shared helper's invalid numeric limit behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid saturated-cache full scans:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's 5-minute, 5000-entry replay guard, a busy server can keep the cache saturated so each new message still scans roughly the whole cache, leaving the targeted delay regression in place for the busiest cases.
  - confidence: 0.91
- **[P2] Preserve invalid-limit pruning semantics:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` is also `NaN`, so `i++ >= deleteCount` never becomes true and the loop deletes every entry. Current behavior leaves maps untouched for `NaN`, and webhook guard callers can pass `NaN` after numeric normalization, so active guard state can be cleared.
  - confidence: 0.88
