## Summary

Keep this PR open, but do not merge it as-is: current main and the latest release still have the eager dedupe prune behavior, while the proposed patch leaves the saturated Discord hot path scanning the full cache and changes shared map-pruning semantics for invalid numeric limits.

## What This Changes

The PR changes `createDedupeCache().check()` to prune only after exceeding `maxSize` and rewrites `pruneMapToMaxSize()` to delete old entries with a counted forward pass.

## Best Possible Solution

Revise or replace this with a bounded pruning strategy that avoids per-message full scans in saturated caches, preserves or explicitly validates invalid numeric limit handling, adds focused helper and Discord replay tests, and includes the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by filling the proposed 5000-entry replay cache with unique keys and observing full-prune scans after saturation; a read-only Node comparison also reproduces the `NaN` pruning semantic change.

## Solution Assessment

No. The direction is useful, but this patch is not the best fix because it only delays the full scan until saturation and changes shared helper behavior instead of making pruning bounded and preserving guard semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid full-cache prune once the cache saturates:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's five-minute, 5000-entry replay guard, a saturated unique-message stream still rescans roughly the whole cache on every new key, so the delay this PR targets can continue on busy servers.
  - confidence: 0.91
- **[P2] Preserve invalid-limit pruning behavior:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN` and the loop never breaks, deleting every entry. Current behavior leaves the map untouched for `NaN`, and SDK webhook guard callers can propagate `NaN` after numeric normalization, so this can make guard state forget active keys.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff does not change CI, dependencies, secrets, or package resolution, but the shared map-pruning rewrite can weaken webhook guard state for invalid numeric limits.

Concerns:

  - confidence: 0.84
