## Summary

Keep this PR open. The performance direction is valid and the linked discussion already identified the Discord replay-cache hot path, but the current patch still rescans the full cache once it is saturated and also changes shared map-pruning semantics in a way that can clear webhook guard state for invalid numeric limits.

## What This Changes

The PR changes `createDedupeCache().check()` to prune only after the cache exceeds `maxSize` and rewrites `pruneMapToMaxSize()` to delete old entries with a counted forward pass.

## Best Possible Solution

Revise or replace this with a bounded pruning strategy that avoids per-message full scans in both normal and saturated cache states, preserves or explicitly validates invalid numeric limit handling for shared helpers, adds focused helper and Discord replay tests, and includes the required changelog entry for the user-facing fix.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by filling the 5000-entry Discord replay cache with unique keys and observing the PR still enters full pruning on every overflow; a read-only Node comparison also reproduces the `NaN` map-pruning semantic change.

## Solution Assessment

No. The direction is useful, but this patch is not the best fix because it only delays the full scan until saturation and changes a public helper's invalid-limit behavior instead of making pruning bounded and preserving existing guard semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid full-cache prune once the cache saturates:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's five-minute, 5000-entry replay guard, a saturated unique-message stream overflows on every new key and still rescans roughly the whole cache, so the delay this PR targets can continue on busy servers.
  - confidence: 0.91
- **[P2] Preserve invalid-limit pruning behavior:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN` and the loop never breaks, deleting every entry. Current behavior leaves the map untouched for `NaN`, and webhook guard callers can propagate `NaN` after numeric normalization, so this can make in-flight or rate-limit guards forget active keys.
  - confidence: 0.88

## Security Review
