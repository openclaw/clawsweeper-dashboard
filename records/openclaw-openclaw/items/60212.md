## Summary

Keep open: the underlying cron empty-output bug is valid and the PR’s main repair direction is useful, but the current patch still leaves a concrete runtime-boundary bug when a CLI-origin repair falls back to an embedded candidate.

## What This Changes

The PR adds a one-pass cron empty-output repair flow, explicit fail-loud handling, CLI bundled-MCP suppression/session reuse, sender ownership propagation, descendant-output handling, and focused regression tests.

## Best Possible Solution

Keep the one-pass repair design, but derive repair constraints inside the fallback callback so embedded candidates always get disableTools=true and CLI candidates only disable OpenClaw bundled MCP while reusing the matching CLI session.

## Reproduction Assessment

Yes. Current main’s silent empty-output behavior is source-reproducible from delivery dispatch returning ok/delivered=false for empty normalized payloads, and the PR-head blocker is source-reproducible from shared repair options crossing fallback candidates.

## Solution Assessment

No, not yet. The repair flow is the right shape, but the implementation must make the repair constraints runtime-specific before it is safe to merge.
