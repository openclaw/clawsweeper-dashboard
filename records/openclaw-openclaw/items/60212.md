## Summary

Keep open: the cron empty-output bug is still source-reproducible on current main and the PR direction is useful, but this branch can lose CLI repair context and still lacks real behavior proof.

## What This Changes

The PR adds a cron no-tools retry for empty sanitized final replies, returns an explicit delivery error when repair still has no text, threads `disableTools` through agent runs, and adds targeted regression tests plus a changelog entry.

## Best Possible Solution

Port one no-tools empty-output repair retry into the current cron executor/delivery split, preserve CLI session context or restate the original task safely, fail explicitly only after repair still has no deliverable text, and leave intentional `NO_REPLY` plus https://github.com/openclaw/openclaw/issues/60204 unchanged.

## Reproduction Assessment

Yes. Source inspection on current main shows empty payloads can resolve without `synthesizedText`, `finalizeTextDelivery` returns `null` for that state, and the executor only retries interim acknowledgement text; no tests were run because this review is read-only.

## Solution Assessment

No, not as proposed. The repair direction is narrow, but the branch changes CLI session-reuse inputs and needs a current-main port that keeps context available for the no-tools retry.
