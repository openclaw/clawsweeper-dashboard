## Summary

The PR is useful and addresses a real cron empty-output failure mode, but it is not ready to merge because the repair pass still chooses tool constraints before the fallback candidate is known, allowing a CLI-origin repair fallback to embedded execution with tools enabled.

## What This Changes

The branch adds cron empty-sanitized-output repair/fail-loud handling, CLI bundled-MCP suppression and session reuse, descendant-output recovery, sender-ownership coverage, and focused regression tests.

## Best Possible Solution

Keep the cron repair/fail-loud behavior, but derive embedded no-tools and CLI bundled-MCP constraints inside each fallback attempt after resolving that candidate's execution provider.

## Reproduction Assessment

Yes. The current-main bug is source-reproducible from empty synthesized text returning null in delivery finalization, and the PR-head blocker is source-reproducible from CLI-origin repair options being reused across fallback candidates.

## Solution Assessment

No, not yet. The direction is narrow and maintainable, but the repair constraint selection must move into the per-candidate fallback path after the execution provider is known.
