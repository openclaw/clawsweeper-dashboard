## Summary

Keep this PR open, but it still needs one correctness/security repair before merge: the empty-output repair options are computed once from the original repair provider and can be reused across fallback candidates with the wrong tool boundary.

## What This Changes

The PR adds a one-pass cron empty-output repair flow, explicit fail-loud handling, CLI bundled-MCP suppression/session reuse, descendant-output handling, and focused regression tests.

## Best Possible Solution

Keep the one-pass repair design, but compute repair constraints inside the fallback callback so embedded candidates always get disableTools=true and CLI candidates only disable bundled MCP while reusing the matching CLI session.

## Reproduction Assessment

Yes. Current main's silent empty-output behavior is source-reproducible in delivery dispatch, and the PR-head mixed-fallback blocker is source-reproducible from the shared repair options crossing fallback candidates.

## Solution Assessment

No, not yet. The overall repair shape is appropriate, but the implementation must make repair constraints runtime-specific before it is safe to merge.
