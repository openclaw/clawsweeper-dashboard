## Summary

Keep open: current main still has the silent cron empty-output gap and this PR addresses the core problem with proof, but the current head still risks intercepting already-deliverable structured or threaded direct delivery by running descendant finalization before direct delivery selection.

## What This Changes

The PR adds cron empty-final-output repair and fail-loud behavior, CLI repair session reuse with bundled MCP suppression, sender-ownership coverage, shared silent-reply normalization, a small Telegram optional-config spread, docs/baseline updates, and regression tests.

## Best Possible Solution

Land the cron repair after gating descendant fallback and fail-loud handling to empty/no-deliverable parent output or the explicit fresh-descendant empty-output signal, preserving current direct delivery for non-empty structured and threaded payloads.

## Reproduction Assessment

Yes. Current main is source-reproducible: cron execution only retries interim acknowledgements, and silent direct delivery can return ok with delivered=false when payloads normalize to empty text.

## Solution Assessment

No. The repair approach is appropriate, but the PR should not run descendant finalization ahead of every structured or threaded direct delivery when an already-deliverable payload exists.
