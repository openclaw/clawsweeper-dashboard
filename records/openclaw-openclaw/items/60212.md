## Summary

Keep open: the core repair design is useful, but the current head still has a merge-blocking mixed-runtime tool boundary bug in the repair fallback path.

## What This Changes

The PR adds cron empty-output repair and fail-loud handling, plus CLI repair session reuse and bundled-MCP suppression for CLI-backed cron runs.

## Best Possible Solution

Keep the one-pass repair behavior, but compute repair constraints inside the fallback candidate callback so CLI candidates suppress bundled MCP and embedded candidates always receive `disableTools=true`.

## Reproduction Assessment

Yes. Current main’s empty sanitized output path is source-reproducible in delivery finalization, and the PR-head blocker is source-reproducible from fallback candidates sharing precomputed repair options.

## Solution Assessment

No, not yet. The proposed repair shape is maintainable, but the runtime-specific tool constraints need to move into the per-candidate execution path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Constrain repair options per fallback runtime:** `src/cron/isolated-agent/run-executor.ts:575-577`
  - body: The repair retry derives `disableTools` and `disableBundleMcpForCli` from the provider that produced the empty reply, then reuses those options for the whole fallback run. If a CLI repair falls through to an embedded candidate, embedded execution receives `disableTools` as `undefined` and can still run OpenClaw tools despite the repair prompt forbidding tool use. Compute these constraints inside the per-candidate callback after resolving `executionProvider`.
  - confidence: 0.88
