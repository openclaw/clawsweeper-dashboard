## Summary

Keep open: the current-main bug is real and the PR direction is useful, but the current PR head still accepts a non-deliverable repair result such as `NO_REPLY` as recovered output before delivery normalizes it back to a silent ok/not-delivered result.

## What This Changes

The PR adds a cron final-output repair pass, embedded/CLI repair constraints, descendant-output fallback handling, and regression coverage for empty sanitized cron replies.

## Best Possible Solution

Keep the one-repair-pass approach, but classify repaired output through the same silent-token deliverability rules used by delivery before marking the cron run recovered.
