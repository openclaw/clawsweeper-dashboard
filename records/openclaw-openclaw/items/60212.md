## Summary

Keep this PR open: the cron empty-output repair direction is useful and backed by tests/proof, but the current patch still accepts a repair response like `NO_REPLY` as recovered text and lets delivery finish silently.

## What This Changes

The PR adds cron empty-output final-reply repair and fail-loud handling, CLI bundled-MCP disablement for repair mode, sender-owner propagation, descendant-output fallback handling, and focused regression tests.

## Best Possible Solution

Keep the PR's repair approach, but classify repaired output with the same silent-token/non-deliverable rules before accepting it, and add a regression test for a repair response of `NO_REPLY`.

## Reproduction Assessment

Yes. The current-main bug is source-reproducible from the missing empty-output repair path, and the PR-head blocker is source-reproducible with a repair payload of `NO_REPLY`.

## Solution Assessment

No, not yet. The overall repair design is narrow, but the acceptance check must use the existing silent-reply deliverability contract before the run can finish ok.

- **[P2] Classify silent repair output as failed repair:** `src/cron/isolated-agent/run-executor.ts:614-620`
  - body: The repair pass only fails when the repaired text trims to an empty string.
