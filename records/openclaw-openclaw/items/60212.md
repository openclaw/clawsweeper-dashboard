## Summary

This PR is still a plausible bug fix, but the latest head leaves one fail-loud gap in the descendant branch: if the parent output is empty and descendants produce no usable final/fallback text, the run can still finish as ok with no delivery instead of recording an explicit cron error.

## What This Changes

The PR adds empty-output repair for cron isolated-agent runs, adjusts CLI repair/session and bundled MCP handling, recovers descendant replies, and adds regression tests/docs/changelog coverage.

## Best Possible Solution

Keep the one-pass repair design, but make the descendant-empty-output path deliver descendant text when available and otherwise record the same explicit cron error instead of finishing as an empty ok run.

## Reproduction Assessment

Yes. Source inspection against the PR head/merge shows executeCronRun skips repair when descendants exist, and delivery dispatch can fall through to finishSilentReplyDelivery when descendant follow-up produces no text.

## Solution Assessment

No, not yet. The proposed approach is narrow and mostly right, but the descendant branch still needs the same fail-loud guarantee when no descendant output can be recovered.
