## Summary

Keep this PR open: the cron empty-output repair is useful and the PR includes real behavior proof, but the latest head still leaves a direct/thread delivery path that can turn an empty descendant follow-up into silent ok/delivered=false instead of the explicit error this fix is meant to add.

## What This Changes

The PR adds cron empty-output repair, CLI bundled-MCP repair handling, shared silent-reply normalization, fail-loud delivery handling, and focused regression tests.

## Best Possible Solution

Keep the one-repair design, but run the descendant fallback and empty-output fail-loud contract before both direct/thread delivery and text delivery so every delivery mode shares the same result semantics.

## Reproduction Assessment

Yes. The original main-branch bug is source-reproducible from the missing final empty-output repair path, and the PR-head blocker is source-reproducible by following useDirectDelivery before finalizeTextDelivery when descendant output remains empty.

## Solution Assessment

No, not yet. The repair approach is narrow and maintainable, but the delivery-side finalization needs to run before both direct/thread and text delivery branches.
