## Summary

Keep open: the bug and repair direction are valid, but the current PR head still has a merge-blocking mixed-runtime repair path where a CLI-origin empty-output repair can fall back to an embedded candidate with tools enabled.

## What This Changes

The PR adds cron empty-output repair and fail-loud handling, CLI bundled-MCP suppression/session reuse, descendant-output recovery, sender-ownership propagation, and focused regression tests.

## Best Possible Solution

Keep the cron repair design, but choose repair constraints inside each fallback candidate after resolving whether that candidate is CLI-backed or embedded.

## Reproduction Assessment

Yes. The current-main bug is source-reproducible from empty synthesized text returning null in delivery finalization, and the PR-head blocker is source-reproducible from shared repair options crossing per-candidate fallback execution.

## Solution Assessment

No, not yet. The approach is narrow and maintainable, but repair constraints must be derived per fallback candidate after the execution provider is known.
