## Summary

Keep this PR open: it addresses a real cron delivery bug, but the current head still has a blocking delivery-order regression where descendant finalization runs before already-deliverable structured or threaded direct delivery.

## What This Changes

This PR adds cron empty-final-output repair, explicit fail-loud handling, CLI repair session reuse with bundled-MCP suppression, sender-ownership coverage, shared silent-reply normalization, and small docs/Telegram/UI baseline updates.

## Best Possible Solution

Land the cron repair after gating descendant fallback and fail-loud handling to empty/no-deliverable parent output or the explicit fresh-descendant empty-output signal, while preserving current direct delivery for non-empty structured and threaded payloads.

## Reproduction Assessment

Yes. Current main is source-reproducible: cron execution only retries interim acknowledgements, and delivery can finish ok/not-delivered when normalized payloads contain no deliverable text.

## Solution Assessment

No. The repair direction is appropriate, but the current PR head should not run descendant finalization ahead of every structured or threaded direct-delivery path when a deliverable payload already exists.
