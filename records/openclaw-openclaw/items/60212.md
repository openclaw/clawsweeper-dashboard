## Summary

Keep open. The bug is valid and the repair direction is useful, but the PR still changes direct delivery semantics by running descendant finalization before every structured or threaded direct delivery, which can suppress or replace already-deliverable payloads.

## What This Changes

The PR adds cron empty-final-output repair and fail-loud behavior, CLI bundled-MCP disablement/session reuse for repair, shared silent-reply normalization, regression tests, and a small Telegram polling optional-config spread.

## Best Possible Solution

Land the cron repair after gating descendant fallback and fail-loud handling to empty/no-deliverable parent output or the explicit fresh-descendant empty-output signal, preserving current direct delivery for non-empty structured and threaded payloads.

## Reproduction Assessment

Yes. Current main is source-reproducible: cron execution only retries interim acknowledgements, and silent direct delivery can return ok with delivered=false when payloads normalize to empty text.

## Solution Assessment

No. The repair approach is appropriate, but this PR should not run descendant finalization ahead of every structured or threaded direct delivery when an already-deliverable payload exists.
