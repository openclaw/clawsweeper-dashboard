## Summary

Keep open: the repair design is useful, and the PR includes convincing live proof, but the current head still shares CLI-origin repair options across all fallback candidates, so an embedded fallback can regain tools during a supposedly no-tools repair pass.

## What This Changes

The PR adds one cron empty-output repair pass, explicit fail-loud handling for unrecovered text, CLI repair mode with bundled MCP disabled and session reuse, descendant-output recovery, and focused cron/CLI regression tests.

## Best Possible Solution

Keep the one-pass repair design, but derive repair constraints inside the fallback candidate callback so CLI candidates suppress bundled MCP and embedded candidates always receive disableTools=true.

## Reproduction Assessment

Yes. Current main’s empty synthesizedText path is source-reproducible in finalizeTextDelivery, and the PR-head blocker is source-reproducible from runWithModelFallback invoking the callback per candidate while repair options are computed once.

## Solution Assessment

No, not yet. The proposed repair shape is maintainable, but the runtime-specific repair constraints need to move into the per-candidate execution path before merge.
