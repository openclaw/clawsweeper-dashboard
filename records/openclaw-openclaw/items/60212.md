## Summary

Keep open. The bug this PR targets still appears valid on current main, but the proposed patch is not safe to merge as written because the CLI-backed repair pass can lose the original cron context, and the branch is now stale against the current cron executor refactor.

## What This Changes

The PR adds a no-tools retry for cron runs whose sanitized final text is empty, turns unrecovered empty delivery into an explicit error, disables bundled MCP for that retry, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Implement the empty sanitized-output repair on current main in the cron executor, preserving the original run context for CLI-backed providers while keeping the repair no-tools/MCP-free, then make delivery dispatch fail loudly only after that single repair still yields no deliverable text.

## Reproduction Assessment

Yes. The PR body gives focused mocked tests for embedded repair success, repair failure, and CLI bundled-MCP disablement; the CLI-session defect is also reproducible from source by following `extraSystemPromptHash` into `resolveCliSessionReuse`. I did not run tests because this review was read-only.

## Solution Assessment

No, not as currently proposed. The repair idea is narrow and maintainable, but the PR's CLI path can run the repair without the original task context and the implementation now needs to be moved into the current `run-executor.ts` structure.
