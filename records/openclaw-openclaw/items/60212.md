## Summary

Keep open; the PR fixes the right cron seams and has sufficient real behavior proof, but one repair-mode edge still needs a code change because a CLI repair that falls through model fallback to embedded execution loses the enforced no-tools constraint.

## What This Changes

The PR adds a cron empty-output repair pass, fail-loud delivery handling, CLI bundled-MCP suppression/session reuse, ownership and descendant edge-case tests, and a changelog entry.

## Best Possible Solution

Keep the one-pass cron repair design, but make repair constraints runtime-specific so embedded primary or fallback candidates are tool-disabled while CLI candidates resume the captured session with bundled MCP disabled.

## Reproduction Assessment

Yes. Source inspection on current main shows empty sanitized outputs have no repair path and can finish without an explicit delivery failure; the new fallback concern is also source-reproducible through `runWithModelFallback` reusing the repair options across candidates.

## Solution Assessment

No, not quite. The overall fix is the right shape, but the repair options need to be applied per execution runtime so embedded fallback remains no-tools while CLI keeps session reuse and bundled-MCP suppression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep embedded fallback repairs tool-disabled:** `src/cron/isolated-agent/run-executor.ts:543`
  - body: When the first empty-output run used a CLI provider, this passes `disableTools: undefined` into `runPrompt`. That same options object is reused for every model-fallback candidate, so if the CLI repair falls back to a non-CLI embedded runtime, the embedded call also receives `disableTools: undefined` and can run tools despite this being a constrained finalization pass. Please decide the repair constraints per `executionProvider` so embedded fallback candidates stay tool-disabled while CLI candidates only disable bundled MCP.
  - confidence: 0.84
