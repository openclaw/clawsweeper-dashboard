## Summary

Keep this PR open. It addresses a real, source-reproducible cron empty-output gap, but the latest head weakens the promised no-tools repair boundary for CLI-backed cron runs and still has only mocked/test-runner proof rather than real after-fix behavior proof.

## What This Changes

The PR adds an empty sanitized-output repair retry for isolated cron runs, explicit failure when repair remains empty, CLI bundled-MCP suppression for repair, regression tests, and a changelog entry.

## Best Possible Solution

Preserve the empty-output repair, but keep the repair fail-closed: embedded repair can use disableTools, while CLI repair should either enforce a real backend-specific no-tools mode or skip/fail for native tool-capable backends, then add real cron/Telegram proof.

## Reproduction Assessment

Yes. Source inspection on current main shows empty sanitized output can leave no synthesized delivery text, and the delivery path returns null for missing synthesizedText; I did not run tests because this review is read-only.

## Solution Assessment

No. The repair direction is right, but the latest CLI mapping replaces enforceable disableTools with bundled-MCP-only suppression, so native tool-capable CLI backends can still use tools during the supposedly no-tools repair.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep CLI repairs fail-closed for tools:** `src/cron/isolated-agent/run-executor.ts:199`
  - body: Line 199 converts the repair pass's `disableTools` request into `disableBundleMcp` for CLI runs, so `prepareCliRunContext` never sees `disableTools: true`. Because bundled CLI backends are marked `nativeToolMode: "always-on"` and that prepare path only fails closed when `disableTools` is true, the supposedly no-tools cron repair can still run a native tool-capable CLI. Please keep this boundary enforceable, or fail/skip repair for native-tool CLI backends instead of relying on prompt text.
  - confidence: 0.93
