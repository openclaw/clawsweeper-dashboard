## Summary

Keep open: the repair direction is useful and has live proof, but current head still lets a CLI-origin empty-output repair fall back to embedded execution without enforcing the no-tools repair boundary.

## What This Changes

The PR adds a cron empty-output repair pass, explicit fail-loud handling, CLI bundled-MCP suppression/session reuse, sender ownership propagation, descendant follow-up handling, and focused regression tests.

## Best Possible Solution

Keep the one-pass repair design, but compute repair constraints inside each fallback candidate so embedded repair always disables tools and CLI repair only disables OpenClaw bundled MCP while reusing the matching CLI session.

## Reproduction Assessment

Yes. Current main's silent empty-output behavior is source-reproducible from delivery dispatch returning silent completion when normalized payloads are empty, and the PR-head blocker is source-reproducible from `runWithModelFallback` reusing one repair options object across candidates.

## Solution Assessment

No, not yet. The overall repair path is the right shape, but runtime constraints must be derived per fallback candidate before this is safe to merge.
