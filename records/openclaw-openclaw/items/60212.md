## Summary

Keep open: the cron empty-output repair is a valid bug fix direction and the PR has real behavior proof, but current head still leaves a mixed CLI-to-embedded repair fallback path with tools enabled despite the no-tools repair prompt.

## What This Changes

The PR adds one repair pass for empty sanitized cron replies, fail-loud handling when no final text is recovered, CLI repair session reuse with bundled MCP suppressed, and focused cron/CLI regression coverage.

## Best Possible Solution

Keep the one-pass repair and fail-loud behavior, but derive repair constraints per fallback candidate after resolving executionProvider and add mixed CLI-to-embedded regression coverage.

## Reproduction Assessment

Yes. Current main’s empty synthesized text path is source-reproducible in delivery finalization, and the PR-head blocker is source-reproducible from shared repair options being applied across fallback candidates.

## Solution Assessment

No, not yet. The proposed repair shape is maintainable, but CLI candidates and embedded candidates need separate repair constraints inside the per-candidate fallback callback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Constrain repair options per fallback candidate:** `src/cron/isolated-agent/run-executor.ts:575-577`
