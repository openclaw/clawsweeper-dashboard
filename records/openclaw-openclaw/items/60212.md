## Summary

Keep this PR open: the empty-output repair direction is valid and the supplied live proof clears the real-behavior gate, but the current CLI repair path still has blocking correctness/security issues around session continuity and tool-disable semantics.

## What This Changes

The PR adds one cron empty-output repair retry, explicit error finalization when repair still has no deliverable text, CLI bundled-MCP suppression for repair runs, regression tests, and a changelog entry.

## Best Possible Solution

Keep the empty-output repair, but make CLI repair resume the first run's returned CLI session when needed and keep the tool-disable contract explicit instead of silently downgrading it to bundled-MCP-only.

## Reproduction Assessment

Yes. Source inspection on current main shows empty sanitized cron output can leave no synthesizedText, the existing retry only handles interim acknowledgements, and text delivery returns null when synthesizedText is missing.

## Solution Assessment

No. The repair approach is directionally right, but the proposed CLI path should preserve first-run session context and should not describe or route a bundled-MCP-only retry as a tool-disabled repair.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P2] Reuse the first CLI run when repairing empty output:** `src/cron/isolated-agent/run-executor.ts:514-516`
  - body: The repair is invoked before `finalizeCronRun` saves the first run's returned CLI session id, and `runPrompt` only supplies `cliSessionId` when `cronSession.isNewSession` is false. Isolated cron forces a new session, so this repair can start a fresh CLI thread that only sees the restated task, not the first run's native tool/output context. Carry the first run's returned CLI session binding into the repair call so an empty sanitized reply can repair the actual run.
  - confidence: 0.87
- **[P2] Do not downgrade disableTools to bundled MCP only:** `src/cron/isolated-agent/run-executor.ts:199`
  - body: Mapping the repair option to `disableBundleMcp` leaves native CLI tools enabled for built-in CLI backends marked `nativeToolMode: "always-on"`, bypassing the existing fail-closed contract for callers that require no tools. That means the automatic repair turn can still run shell/filesystem/native tools despite being logged and prompted as a no-tools pass; either enforce `disableTools` where supported or make this a separate explicitly non-tool-free mode with matching tests and wording.
  - confidence: 0.84
