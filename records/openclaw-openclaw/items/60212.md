## Summary

Keep this open: the cron empty-output bug is still present on current main, but this PR is not mergeable as-is because the CLI repair retry can lose the session that contains the original cron task and the external real-behavior proof gate is still unmet.

## What This Changes

The PR adds a cron empty-output repair retry, fail-loud delivery handling, CLI disableTools MCP gating, regression tests, and a changelog entry.

## Best Possible Solution

Port one no-tools empty-output repair retry into the current cron executor/delivery split, preserve CLI session context or safely restate the original task, and fail explicitly only after the repair still has no deliverable text.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: empty sanitized payloads can leave synthesizedText undefined, current main only retries interim acknowledgements, and the delivery path returns no explicit error for missing text; I did not run tests because this review is read-only.

## Solution Assessment

No, not as proposed. The repair direction is narrow, but the branch should preserve the existing CLI session binding or include the original cron task context instead of changing the session-reuse prompt hash for the retry.

## Review Findings
