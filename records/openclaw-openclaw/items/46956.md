## Summary

Keep this PR open for maintainer/security review: current main does not implement trust windows, but the proposed implementation still changes a security-sensitive exec approval bypass and has source-reproducible authority, hard-deny, shared-auth, audit-accounting, and proof gaps.

## What This Changes

The PR adds CLI trust, untrust, and trust-status commands with Gateway RPC/protocol and Swift model updates, in-memory trust-window state, JSONL audit logging, exec-host enforcement, and tests for time-bounded exec approval bypass.

## Best Possible Solution

Define one maintainer-approved timed exec approval model with explicit authority, scope, expiry, hard-deny behavior, audit guarantees, protocol/docs, regression coverage, and real-runtime proof before landing either implementation.

## Reproduction Assessment

Yes for review purposes. The blockers are source-reproducible from the PR diff against current main: the trust caller gate, local clientIp handling, CLI device identity defaults, host deny policy, and audit predicates are all inspectable without running the PR branch.

## Solution Assessment

No. The timed trust concept is useful, but this patch is not the best mergeable form until Gateway-owned authority, shared-auth behavior, hard-deny semantics, audit accounting, and real behavior proof are settled explicitly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Enforce trust authority on the Gateway:** `src/gateway/server-methods/exec-approvals.ts:127`
  - body: The trust RPC accepts any CLI connection with device identity, while the non-agent restriction is only a client-side env-var check. Current CLI calls identify as CLI and load device identity by default, so an agent-spawned CLI can self-grant the approval bypass unless the Gateway has a stronger authority/scope check.
  - confidence: 0.86
- **[P1] Keep shared localhost auth usable for trust:** `src/gateway/server-methods/exec-approvals.ts:134`
  - body: This requires `client.clientIp` to be loopback for token/password auth, but current WebSocket handling clears the reported client IP for local or loopback clients before storing it on the Gateway client. The no-device shared-auth path described by the PR will be rejected.
  - confidence: 0.88
- **[P1] Preserve deny policies as a hard ceiling:** `src/infra/exec-approvals.ts:624-626`
  - body: When a trust window is active, the resolver promotes the agent policy to `security=full` and `ask=off`. Current host policy treats `security=deny` as an exec block, so this silently bypasses an explicit deny policy unless maintainers intentionally redefine deny semantics.
  - confidence: 0.87
- **[P2] Record commands that started under trust:** `src/agents/bash-tools.exec.ts:570-574`
  - body: The audit append is gated on trust being active both when the command starts and when it finishes. A command that starts during the window but finishes after expiry or revoke ran under bypassed approvals but is omitted from the trust audit trail.
  - confidence: 0.9
- **[P3] Evict expired trust windows lazily:** `src/infra/exec-approvals.ts:200-201`
  - body: Expired entries remain in the module-level trustWindowCache unless explicitly revoked or the gateway restarts. A long-running gateway with many transient agent IDs can accumulate stale trust records for the process lifetime.
  - confidence: 0.74
