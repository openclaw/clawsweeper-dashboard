## Summary

Keep this PR open: current main does not implement trust windows, and the PR still changes security-sensitive exec approval semantics with source-reproducible authorization, hard-deny, audit, and proof gaps.

## What This Changes

The PR adds CLI trust-window commands, Gateway RPC and protocol models, in-memory trust state, JSONL audit logging, exec-host enforcement, Swift model updates, and related tests for time-bounded exec approval bypass.

## Best Possible Solution

Define one maintainer-approved timed exec approval model with explicit authority, scope, hard-deny semantics, audit guarantees, expiry behavior, UI/CLI surfaces, protocol docs, and regression coverage before landing code.

## Reproduction Assessment

Yes for review purposes. The blocking issues are source-reproducible by inspecting the PR trust RPC gate, current WebSocket client IP reporting, current hard-deny host policy, and the PR audit append predicates.

## Solution Assessment

No. The direction is useful, but this PR is not the best mergeable solution until Gateway-owned authorization, localhost auth behavior, hard-deny semantics, audit accounting, and real behavior proof are settled explicitly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Enforce trust authority on the Gateway:** `src/gateway/server-methods/exec-approvals.ts:127-130`
  - body: The trust RPC accepts a CLI client with device identity while the non-agent block is only the new CLI env check. Current main does not propagate agent-session identity into this gate, so an agent-launched CLI can bypass that guard; the grant boundary needs server-verifiable session or origin metadata.
  - confidence: 0.84
- **[P1] Keep shared localhost auth usable for trust:** `src/gateway/server-methods/exec-approvals.ts:136`
  - body: The PR says loopback token/password auth is supported, but the gate requires `isLoopbackAddress(client?.clientIp)`. Current WebSocket handling clears local reported client IPs to `undefined`, so valid local shared-auth CLI calls without device identity are rejected.
  - confidence: 0.88
- **[P1] Preserve deny policies as a hard ceiling:** `src/infra/exec-approvals.ts:624-628`
  - body: When a trust window is active, the resolver promotes the agent policy to the trust-window security value. Current host-context behavior treats `security=deny` as a hard exec block, so this silently bypasses an explicit deny policy.
  - confidence: 0.86
- **[P2] Record commands that start under trust:** `src/agents/bash-tools.exec.ts:570-574`
  - body: The audit append is gated on trust being active at both command start and completion. A command that starts during the window but finishes after expiry or revoke is omitted from the audit even though it ran under bypassed approval policy.
  - confidence: 0.9
- **[P3] Evict expired trust windows from the cache:** `src/infra/exec-approvals.ts:200-201`
  - body: Expired entries remain in the module-level trust-window cache unless explicitly revoked or the gateway restarts. A long-running gateway with many transient agent IDs can accumulate stale entries for the process lifetime.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: The patch introduces a security-sensitive exec approval bypass with concrete authorization, hard-deny, and audit-accounting concerns.

Concerns:

- **[high] Agent-spawned CLI can self-grant trust:** `src/gateway/server-methods/exec-approvals.ts:127`
  - body: The server-side trust gate accepts CLI device identity while the non-agent restriction is enforced only by client-side environment variables, which is not a reliable boundary for a feature that disables approval prompts.
  - confidence: 0.84
- **[high] Trust windows override explicit exec deny policy:** `src/infra/exec-approvals.ts:624`
  - body: The resolver promotes a trusted agent to full execution, contradicting current hard-deny behavior and allowing a trust grant to bypass a configured host-exec block.
  - confidence: 0.86
- **[medium] Audit can miss trusted commands after expiry or revoke:** `src/agents/bash-tools.exec.ts:570`
  - body: Commands that start while trusted but finish after the window is no longer active are dropped from the JSONL audit trail, weakening the feature's accountability mitigation.
  - confidence: 0.9
