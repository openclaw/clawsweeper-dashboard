## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Enforce trust authority on the Gateway:** `src/gateway/server-methods/exec-approvals.ts:127`
  - body: The trust RPC accepts CLI device identity, while the non-agent restriction is only enforced by client-side environment variables. An agent-spawned CLI with a TTY can still present as CLI and open the approval bypass unless the Gateway verifies a server-side operator/session authority for this action.
  - confidence: 0.86
- **[P1] Keep shared localhost auth usable for trust:** `src/gateway/server-methods/exec-approvals.ts:136`
  - body: This path requires `isLoopbackAddress(client?.clientIp)` for token/password auth, but the WebSocket handler stores `clientIp` as `undefined` for local or loopback clients. The no-device localhost auth path described by the PR is therefore rejected.
  - confidence: 0.88
- **[P1] Preserve deny policies as a hard ceiling:** `src/infra/exec-approvals.ts:624-626`
  - body: When a trust window is active, the resolver promotes the agent policy to `security=full` and `ask=off`. Current host policy treats `security=deny` as an execution block, so this silently bypasses an explicit deny policy unless maintainers intentionally redefine deny semantics.
  - confidence: 0.87
- **[P2] Record commands that started under trust:** `src/agents/bash-tools.exec.ts:570-574`
  - body: The audit append is gated on trust being active both when the command starts and when it finishes. A command that starts during the window but finishes after expiry or revoke ran under bypassed approvals but is omitted from the trust audit trail.
  - confidence: 0.9
- **[P3] Evict expired trust windows lazily:** `src/infra/exec-approvals.ts:171`
  - body: Expired entries remain in the module-level trust cache unless explicitly revoked or the gateway restarts. A long-running gateway with many transient agent IDs can accumulate stale trust records for the process lifetime.
  - confidence: 0.74
