## Summary

Keep open: current main does not implement trust windows, and the PR is not mergeable yet because the proposed approval bypass still has concrete Gateway authorization, hard-deny policy, and audit-accounting problems.

## What This Changes

The PR adds CLI trust, untrust, and trust-status commands plus Gateway RPC/protocol, Swift models, in-memory trust-window enforcement, JSONL audit logging, exec integration, and tests for time-bounded exec approval bypasses.

## Best Possible Solution

Land a consolidated trust-window design that enforces grant authority at the Gateway, preserves or explicitly redefines deny-policy semantics, records trusted-command audit entries from start-time trust state, and updates CLI, protocol, Swift clients, docs, and regression tests together.

## Reproduction Assessment

Yes for review purposes. The blocking issues are source-reproducible from the provided PR diff plus current main: local WebSocket client IP is cleared, current host policy treats deny as a hard ceiling, and the non-agent restriction is only a client-side environment check.

## Solution Assessment

No. The feature direction is useful, but this PR should not merge until the Gateway owns the authorization boundary and the deny/audit semantics are settled explicitly with secops.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Enforce non-agent trust grants on the Gateway:** `src/gateway/server-methods/exec-approvals.ts:127-130`
  - body: The trust RPC accepts a CLI client with device identity while the non-agent guard is implemented only as client-side environment checks. Current main has no repo-wide producer for those env vars, so an agent-run CLI can still present as CLI and grant its own trust window; this boundary needs server-verifiable origin or session metadata.
  - confidence: 0.84
- **[P1] Keep localhost shared auth usable for trust:** `src/gateway/server-methods/exec-approvals.ts:136`
  - body: The PR says loopback token/password auth is supported, but this gate requires client.clientIp to be loopback. Current WebSocket handling clears reported client IPs for local clients, so valid local shared-auth CLI calls without device identity are rejected.
  - confidence: 0.88
- **[P1] Preserve deny policies as a hard ceiling:** `src/infra/exec-approvals.ts:624-628`
  - body: When a trust window is active, the resolver promotes the agent policy to full/off. Current host-context behavior treats security=deny as an execution block, so this silently bypasses an explicit deny policy.
  - confidence: 0.86
- **[P2] Record commands that start under trust:** `src/agents/bash-tools.exec.ts:570-574`
  - body: The audit append is gated on trust being active both at command start and at completion. A command that starts during the window but finishes after expiry or revoke is omitted from the audit even though it ran under bypassed approval policy.
  - confidence: 0.9
- **[P3] Evict expired trust windows from the cache:** `src/infra/exec-approvals.ts:200-201`
  - body: Expired entries remain in the module-level trust window cache unless explicitly revoked or the gateway restarts. A long-running gateway with many transient agent IDs can accumulate stale entries for the process lifetime.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: The patch introduces a security-sensitive exec approval bypass with concrete authorization, hard-deny, and audit-accounting concerns.

Concerns:

- **[high] Agent-spawned CLI can self-grant trust:** `src/gateway/server-methods/exec-approvals.ts:127`
  - body: The server-side trust gate accepts CLI device identity while the non-agent restriction is enforced only by client-side environment variables, which is not a reliable boundary for a feature that disables approval prompts.
  - confidence: 0.84
- **[high] Trust windows override explicit exec deny policy:** `src/infra/exec-approvals.ts:624`
  - body: The resolver promotes a trusted agent to full execution, contradicting current hard-deny behavior and allowing a trust grant to bypass a configured host-exec block.
  - confidence: 0.86
- **[medium] Audit can miss trusted commands after expiry or revoke:** `src/agents/bash-tools.exec.ts:570`
  - body: Commands that start while trusted but finish after the window is no longer active are dropped from the JSONL audit trail, weakening the feature's accountability mitigation.
  - confidence: 0.9
