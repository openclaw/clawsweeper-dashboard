## Summary

Keep open. Current main does not implement trust windows, and the PR still changes a security-sensitive exec approval boundary with source-proven authorization, deny-policy, audit-accounting, and real-behavior-proof blockers.

## What This Changes

Adds CLI trust, untrust, and trust-status commands plus Gateway RPC/protocol models, in-memory trust-window state, JSONL auditing, exec-host enforcement, Swift models, and tests for time-bounded exec approval bypass.

## Best Possible Solution

Define one maintainer-approved timed exec approval model with explicit authority, scope, hard-deny semantics, audit guarantees, expiry behavior, UI/CLI surfaces, protocol docs, and regression plus real-runtime coverage.

## Reproduction Assessment

Yes for review purposes. The blocking issues are source-reproducible by inspecting the PR trust RPC gate, current WebSocket client IP reporting, current hard-deny host policy, and the PR audit append predicates.

## Solution Assessment

No. The timed trust direction is useful, but this PR is not the best mergeable implementation until Gateway-owned authorization, localhost auth behavior, hard-deny semantics, audit accounting, and real behavior proof are settled explicitly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Enforce trust authority on the Gateway:** `src/gateway/server-methods/exec-approvals.ts:127-130`
  - body: The trust RPC accepts a CLI client with device identity while the non-agent block is enforced only by the new CLI env check. Current main does not set those env vars for agent-run commands, so an agent-spawned CLI can still self-grant trust; this boundary needs server-verifiable session or origin metadata.
  - confidence: 0.85
- **[P1] Keep shared localhost auth usable for trust:** `src/gateway/server-methods/exec-approvals.ts:136`
  - body: The PR says loopback token/password auth is supported, but this gate requires clientIp to be loopback. Current WebSocket handling clears reported local and loopback IPs to undefined, so valid local shared-auth CLI calls without device identity are rejected.
  - confidence: 0.88
- **[P1] Preserve deny policies as a hard ceiling:** `src/infra/exec-approvals.ts:624-628`
  - body: When a trust window is active, the resolver promotes the resolved agent policy to the trust-window security value. Current host-context behavior treats security=deny as a hard exec block, so this silently bypasses an explicit deny policy.
  - confidence: 0.87
- **[P2] Record commands that started under trust:** `src/agents/bash-tools.exec.ts:570-574`
  - body: The audit append is gated on trust being active at both command start and completion. A command that starts during the window but finishes after expiry or revoke is omitted from the audit even though it ran under bypassed approval policy.
  - confidence: 0.9
- **[P3] Evict expired trust windows from the cache:** `src/infra/exec-approvals.ts:200-201`
  - body: Expired entries remain in the module-level trust-window cache unless explicitly revoked or the gateway restarts. A long-running gateway with many transient agent IDs can accumulate stale entries for the process lifetime.
  - confidence: 0.74
