## Summary

Keep open: the feature is not implemented on current main, and the PR still has source-reproducible security and policy blockers around trust-grant authority, deny-policy semantics, audit accounting, docs, and changelog coverage.

## What This Changes

The PR adds CLI trust, untrust, and trust-status commands plus Gateway RPC/protocol, Swift models, in-memory trust-window enforcement, JSONL audit logging, and related tests for time-bounded exec approval bypasses.

## Best Possible Solution

Land one consolidated trust-window design that enforces grant authority at the Gateway, preserves or explicitly redefines deny-policy semantics, records commands that started under trust, updates protocol clients/docs/changelog, and supersedes overlapping trust-window PRs/issues only after merge.

## Reproduction Assessment

Yes for review purposes. The blocking issues are source-reproducible by comparing current main's exec approval and WebSocket client metadata behavior with the PR head's trust RPC, trust resolver, and audit predicates.

## Solution Assessment

No. The feature direction is useful, but this PR is not the best mergeable solution until Gateway-owned authorization, hard-deny behavior, audit semantics, stale-main drift, docs, and changelog are resolved together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Enforce non-agent trust grants on the Gateway:** `src/gateway/server-methods/exec-approvals.ts:127-130`
  - body: The trust RPC accepts a CLI client with device identity before the Gateway can prove the request did not originate from an agent-run CLI. The non-agent guard is only the CLI-side `OPENCLAW_SESSION_KEY`/`OPENCLAW_AGENT_ID` check, and current main has no production source setting those env vars, so this privilege-escalation boundary needs server-verifiable origin or session metadata.
  - confidence: 0.84
- **[P1] Keep localhost shared auth usable for trust:** `src/gateway/server-methods/exec-approvals.ts:136`
  - body: The PR says loopback token/password auth is supported, but this requires `isLoopbackAddress(client?.clientIp)`. The WebSocket handshake stores local direct clients with `clientIp: undefined`, so local shared-auth CLI calls without device identity are rejected despite being a documented grant path.
  - confidence: 0.88
- **[P1] Preserve deny policies as a hard ceiling:** `src/agents/bash-tools.exec-host-shared.ts:184-186`
  - body: When a trust window is active, host approval resolution uses the trusted approval values instead of the current `minSecurity(params.security, approvals.agent.security)` cap. That lets a trust grant broaden an explicit deny policy that current host-context code treats as a hard exec block.
  - confidence: 0.86
- **[P2] Record commands that start under trust:** `src/agents/bash-tools.exec.ts:570-574`
  - body: The audit append is gated on trust being active at both command start and completion. A command that starts during the window but finishes after expiry or revoke is omitted from the audit even though it ran under bypassed approval policy.
  - confidence: 0.9
- **[P3] Evict expired trust windows from the cache:** `src/infra/exec-approvals.ts:200-201`
  - body: Expired entries remain in the module-level trust window cache unless explicitly revoked or the gateway restarts. A long-running gateway with many transient agent IDs can accumulate stale entries for the process lifetime.
  - confidence: 0.72
- **[P2] Add docs and changelog for the new trust surface:** `src/gateway/server-methods-list.ts:29-31`
  - body: This adds user-facing CLI commands and reserved Gateway protocol methods, but the PR changes no docs or `CHANGELOG.md`. Protocol changes should update the operator docs/client contract, and repo policy requires a changelog entry for user-facing feature work.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch introduces a security-sensitive exec approval bypass with concrete authorization, hard-deny, and audit-accounting concerns.

Concerns:

- **[high] Agent-spawned CLI can self-grant trust:** `src/gateway/server-methods/exec-approvals.ts:127`
  - body: The server-side trust gate accepts CLI device identity while the non-agent restriction is enforced only by client-side environment variables, which is not a reliable boundary for a feature that disables approval prompts.
  - confidence: 0.84
- **[high] Trust windows override explicit exec deny policy:** `src/agents/bash-tools.exec-host-shared.ts:184`
  - body: The trust path promotes a trusted agent to full/off behavior, contradicting current hard-deny host behavior and allowing a trust grant to bypass a configured host-exec block.
  - confidence: 0.86
- **[medium] Audit can miss trusted commands after expiry or revoke:** `src/agents/bash-tools.exec.ts:570`
  - body: Commands that start while trusted but finish after the window is no longer active are dropped from the JSONL audit trail, weakening the feature's accountability mitigation.
  - confidence: 0.9
