## Summary

Keep open. The timed-trust direction is useful, but this PR still has source-proven security and behavior blockers, lacks required real behavior proof, and overlaps related timed-approval proposals that need one maintainer-approved model.

## What This Changes

The PR adds CLI trust, untrust, and trust-status commands plus Gateway RPC/protocol and Swift models, in-memory trust-window state, JSONL audit logging, exec-host enforcement, and tests for time-bounded exec approval bypass.

## Best Possible Solution

Define one maintainer-approved timed exec approval/trust model with explicit authority, scope, expiry, audit guarantees, hard-deny semantics, UI/CLI/protocol docs, regression coverage, and real-runtime proof, then land a single implementation.

## Reproduction Assessment

Yes for review purposes. The blockers are source-reproducible by inspecting the PR trust RPC gate, current CLI/device identity behavior, current WebSocket client IP reporting, current hard-deny host policy, and the PR audit append predicates.

## Solution Assessment

No. The timed trust concept is useful, but this implementation is not the best mergeable form until Gateway-owned authority, shared-auth behavior, hard-deny semantics, audit accounting, and real behavior proof are settled explicitly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Enforce trust authority on the Gateway:** `src/gateway/server-methods/exec-approvals.ts:127-130`
  - body: The trust RPC treats any CLI connection with device identity as authorized, while the non-agent restriction is only the new CLI env-var check. Current CLI calls identify as CLI and load device identity by default, and current main has no production OPENCLAW_SESSION_KEY/OPENCLAW_AGENT_ID marker, so an agent-spawned CLI can self-grant the approval bypass.
  - confidence: 0.86
- **[P1] Keep shared localhost auth usable for trust:** `src/gateway/server-methods/exec-approvals.ts:136`
  - body: This requires client.clientIp to be loopback for token/password auth, but current WebSocket handling clears reportedClientIp for local or loopback clients and stores that cleared value on the Gateway client. The supported local shared-auth path without device identity will be rejected.
  - confidence: 0.88
- **[P1] Preserve deny policies as a hard ceiling:** `src/infra/exec-approvals.ts:624-628`
  - body: When a trust window is active, the resolver promotes the resolved policy to security=full and ask=off. Current host resolution treats security=deny as a hard exec block, so this silently bypasses an explicit deny policy.
  - confidence: 0.87
- **[P2] Record commands that started under trust:** `src/agents/bash-tools.exec.ts:570-574`
  - body: The audit append is gated on trust being active at both command start and completion. A command that starts during the window but finishes after expiry or revoke is omitted from the audit even though it ran under bypassed approval policy.
  - confidence: 0.9
- **[P3] Evict expired trust windows lazily:** `src/infra/exec-approvals.ts:200-201`
  - body: Expired entries remain in the module-level trustWindowCache unless explicitly revoked or the gateway restarts. A long-running gateway with many transient agent IDs can accumulate stale entries for the process lifetime.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The patch introduces a security-sensitive exec approval bypass with concrete authority, hard-deny, and audit-accounting concerns.

Concerns:

- **[high] Agent-spawned CLI can self-grant trust:** `src/gateway/server-methods/exec-approvals.ts:127`
  - body: The server-side trust gate accepts CLI device identity while the non-agent restriction is enforced only by client-side environment variables, which is not a reliable boundary for a feature that disables approval prompts.
  - confidence: 0.86

  - confidence: 0.87
