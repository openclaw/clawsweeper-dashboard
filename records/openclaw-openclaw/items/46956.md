## Summary

Keep open: current main does not implement trust windows, and this security-sensitive PR still has source-reproducible authorization, hard-deny, and audit-accounting blockers before it is mergeable.

## What This Changes

This PR adds `openclaw approvals trust`, `untrust`, and `trust-status` commands, Gateway trust RPC/protocol/Swift surfaces, in-memory trust-window enforcement, JSONL audit logging, and related tests.

## Best Possible Solution

Land a consolidated trust-window design that enforces grant authority at the Gateway, preserves or explicitly redefines deny-policy semantics, audits commands from start-time trust state, and updates CLI, protocol, Swift clients, docs, and regression tests together.

## Reproduction Assessment

Yes for review purposes. The blockers are source-reproducible from current main plus the PR diff by inspecting the trust RPC gate, WebSocket local client IP handling, hard-deny policy behavior, and audit append predicates.

## Solution Assessment

No. The feature direction is useful, but this PR is not the best mergeable solution until the Gateway owns the authorization boundary and the deny/audit semantics are settled explicitly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Enforce trust grants on a server-verifiable boundary:** `src/gateway/server-methods/exec-approvals.ts:127-130`
  - body: The trust RPC treats a CLI client with device identity as sufficient, while the PR's non-agent protection is the local CLI env/TTY check. For an approval-bypass feature, the Gateway needs server-verifiable origin or session metadata so an agent-controlled process cannot mint a trust window by presenting as the CLI.
  - confidence: 0.84
- **[P1] Preserve loopback shared auth for trust commands:** `src/gateway/server-methods/exec-approvals.ts:136`
  - body: The PR says loopback token/password auth can grant trust, but `isLoopbackAddress(client?.clientIp)` is false for local WebSocket clients because the current handshake stores `reportedClientIp` as undefined for local clients. That rejects the supported no-device-identity CLI path.
  - confidence: 0.88
- **[P1] Keep deny policy as a hard ceiling:** `src/infra/exec-approvals.ts:624-628`
  - body: When trust is active, the resolver replaces `security: "deny"` with the trust window's `full`, and the shared host context then bypasses the usual `minSecurity` cap. Current docs and host context treat `deny` as blocking host exec, so this silently weakens explicit deny configs.
  - confidence: 0.86
- **[P2] Audit commands based on start-time trust:** `src/agents/bash-tools.exec.ts:570-574`
  - body: The audit append is gated on trust being active both at command start and completion. A command that starts during the window but finishes after expiry or revoke is omitted from the audit even though it ran under bypassed approval policy.
  - confidence: 0.9
- **[P3] Evict expired trust windows lazily:** `src/infra/exec-approvals.ts:200-201`
  - body: Expired entries remain in the module-level trust window cache unless explicitly revoked or the gateway restarts. A long-running gateway with many transient agent IDs can accumulate stale entries for the process lifetime.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: The patch introduces a security-sensitive exec approval bypass with concrete authorization, hard-deny, and audit-accounting concerns.

Concerns:

- **[high] Trust grants lack a server-verifiable non-agent boundary:** `src/gateway/server-methods/exec-approvals.ts:127`
  - body: The Gateway accepts CLI identity/device metadata while the non-agent restriction is enforced by client-side environment and TTY checks, which is not a sufficient boundary for disabling approval prompts.
  - confidence: 0.84
- **[high] Trust windows override explicit exec deny policy:** `src/infra/exec-approvals.ts:624`
  - body: The resolver promotes a trusted agent to full execution, contradicting current hard-deny behavior and allowing a trust grant to bypass a configured host-exec block.
  - confidence: 0.86
- **[medium] Audit can miss trusted commands after expiry or revoke:** `src/agents/bash-tools.exec.ts:570`
  - body: Commands that start while trusted but finish after the window is no longer active are dropped from the JSONL audit trail, weakening the feature's accountability mitigation.
  - confidence: 0.9
