## Summary

Keep open. Current main does not implement trust windows, and the PR still needs maintainer judgment because it changes exec privilege semantics with unresolved authority, hard-deny, localhost-auth, audit-accounting, and real-runtime proof issues.

## What This Changes

Adds CLI `approvals trust`, `untrust`, and `trust-status` commands with Gateway RPC/protocol and Swift model updates, in-memory trust-window state, JSONL auditing, exec enforcement changes, and tests.

## Best Possible Solution

Define one maintainer-approved timed exec approval model with Gateway-owned grant authority, explicit scope and expiry, hard-deny semantics, auditable accounting, protocol/docs coverage, focused tests, and redacted real-runtime proof.

## Reproduction Assessment

Yes for source-level PR review: the trust caller gate, local-client IP mismatch, deny-policy override, and audit predicate issues are visible in PR head and current main source. I did not run live runtime proof in this read-only review.

## Solution Assessment

No as-is. The timed-trust concept may be useful, but this patch is not the best mergeable form until the authority model, hard-deny behavior, audit guarantees, and real behavior proof are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enforce trust authority on the Gateway:** `src/gateway/server-methods/exec-approvals.ts:127-130`
  - body: The trust RPC accepts any CLI client with a device identity, while the non-agent restriction is only a client-side environment-variable check in the CLI. An agent-spawned CLI with operator credentials can therefore self-grant the approval bypass, so the grant needs Gateway-owned operator/session authority before opening trust.
  - confidence: 0.86
- **[P1] Keep shared localhost auth usable for trust:** `src/gateway/server-methods/exec-approvals.ts:136`
  - body: The shared-auth fallback requires `isLoopbackAddress(client?.clientIp)`, but local WebSocket clients store `clientIp` as `undefined`. That rejects the token/password localhost path the PR explicitly advertises for CLI sessions without device identity.
  - confidence: 0.88
- **[P1] Preserve deny policies as a hard ceiling:** `src/infra/exec-approvals.ts:624-628`
  - body: When trust is active, the resolver rewrites the agent policy to `full/off`. Current host resolution treats `security=deny` as a hard execution block, so this lets a trust grant bypass an explicit deny policy unless maintainers intentionally redefine deny semantics.
  - confidence: 0.87
- **[P2] Record commands that started under trust:** `src/agents/bash-tools.exec.ts:570-574`
  - body: The audit append checks that trust is still active after the command finishes. A command that started under the bypass but completes after expiry or revoke can be omitted from the trust audit trail, weakening the feature's accountability story.
  - confidence: 0.9
- **[P3] Evict expired trust windows lazily:** `src/infra/exec-approvals.ts:200-201`
  - body: Expired entries stay in the module-level cache until explicit revoke or gateway restart. A long-running gateway that sees many transient agent IDs can accumulate stale trust records for the process lifetime; `getTrustWindow` should delete expired entries before returning.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The patch introduces a security-sensitive exec approval bypass with concrete authority, hard-deny, and audit-accounting concerns.

Concerns:

- **[high] Agent-spawned CLI can self-grant trust:** `src/gateway/server-methods/exec-approvals.ts:127`
  - body: The server-side trust gate accepts CLI device identity while the non-agent restriction is enforced only by client-side environment variables, which is not a reliable security boundary for disabling approval prompts.
  - confidence: 0.86
- **[high] Trust windows override explicit exec deny policy:** `src/infra/exec-approvals.ts:624`
  - body: The resolver promotes a trusted agent to full execution, contradicting current hard-deny behavior and allowing a trust grant to bypass a configured host-exec block.
  - confidence: 0.87
- **[medium] Audit can miss trusted commands after expiry or revoke:** `src/agents/bash-tools.exec.ts:570`
  - body: Commands that start while trusted but finish after the window is no longer active are dropped from the JSONL audit trail, weakening the feature's accountability mitigation.
  - confidence: 0.9
