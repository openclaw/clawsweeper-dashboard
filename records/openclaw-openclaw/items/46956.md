## Summary

Keep open. This PR proposes a useful approval-fatigue feature, but current main does not implement trust windows and the PR still changes a security boundary with unresolved Gateway authorization, hard-deny policy, and audit-accounting problems.

## What This Changes

This PR adds CLI trust, untrust, and trust-status commands, Gateway RPC/protocol and Swift model surfaces, an in-memory exec trust-window override, JSONL audit logging, and focused tests.

## Best Possible Solution

Consolidate the trust-window work into a maintainer-approved security design that uses Gateway-owned authorization/provenance, preserves explicit deny policies unless deliberately redefined in code and docs, records commands based on start-time trust state, and updates CLI, Gateway protocol, Swift clients, docs, and tests together.

## Reproduction Assessment

Yes. The blocking issues have high-confidence static reproduction paths from current main and the PR diff: compare the trust RPC gate, CLI env guard, local WebSocket clientIp clearing, hard-deny code/docs, and audit append predicates.

## Solution Assessment

No. The feature direction is useful, but this PR is not the best mergeable solution until the Gateway owns the non-agent authorization boundary, localhost shared auth matches the advertised behavior, hard-deny semantics are settled, and audit accounting follows command start time.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Enforce non-agent trust grants on the Gateway:** `src/gateway/server-methods/exec-approvals.ts:127-130`
  - body: The trust RPC accepts a CLI client with device identity before proving the request did not originate from an agent-run CLI. The non-agent guard is client-side env checking, so this privilege-escalation boundary needs server-verifiable origin/session metadata.
  - confidence: 0.84
- **[P1] Keep localhost shared auth usable for trust:** `src/gateway/server-methods/exec-approvals.ts:136`
  - body: The PR says loopback token/password auth is supported, but this requires `isLoopbackAddress(client?.clientIp)`. Current WebSocket handling stores `undefined` for local reported client IPs, so local shared-auth CLI calls without device identity are rejected.
  - confidence: 0.88
- **[P1] Preserve deny policies as a hard ceiling:** `src/infra/exec-approvals.ts:624-628`
  - body: When a trust window is active, the resolver promotes the agent policy to the trust window's full/off values. Current docs and host-context code treat `security=deny` as a hard exec block, so this silently bypasses an explicit deny policy.
  - confidence: 0.86
- **[P2] Record commands that start under trust:** `src/agents/bash-tools.exec.ts:570-574`
  - body: The audit append is gated on trust being active at both command start and completion. A command that starts during the window but finishes after expiry or revoke is omitted from the audit even though it ran under bypassed approval policy.
  - confidence: 0.9
- **[P3] Evict expired trust windows from the cache:** `src/infra/exec-approvals.ts:200-201`
  - body: Expired entries remain in the module-level trust window cache unless explicitly revoked or the gateway restarts. A long-running gateway with many transient agent IDs can accumulate stale entries for the process lifetime.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: The patch introduces a security-sensitive exec approval bypass with concrete authorization, hard-deny, and audit-accounting gaps.

Concerns:

- **[high] Agent-spawned CLI can self-grant trust:** `src/gateway/server-methods/exec-approvals.ts:127`
  - body: The server-side trust gate accepts CLI device identity while the non-agent restriction is enforced only by client-side environment variables. That is not a reliable boundary for a feature that disables approval prompts.
  - confidence: 0.84
- **[high] Trust windows override explicit exec deny policy:** `src/infra/exec-approvals.ts:624`
  - body: The resolver promotes a trusted agent to full execution, contradicting current hard-deny behavior and allowing a trust grant to bypass a configured host-exec block.
  - confidence: 0.86
- **[medium] Audit can miss trusted commands after expiry or revoke:** `src/agents/bash-tools.exec.ts:570`
  - body: Commands that start while trusted but finish after the window is no longer active are dropped from the JSONL audit trail, weakening the feature's accountability mitigation.
  - confidence: 0.9
