## Summary

Keep open: current main still does not implement the requested `allowedExternalPaths` config surface, and this PR remains a security-sensitive feature branch with concrete correctness blockers and missing real behavior proof. The related maintainer PR covers a narrower bootstrap-symlink path, but it does not fully supersede this PR's broader read/write allowlist proposal.

## What This Changes

Adds a default/per-agent `workspaceConfig.allowedExternalPaths` allowlist for symlinked agent workspace/bootstrap files, wires it into agent file APIs and bootstrap loading, and documents/tests the behavior.

## Best Possible Solution

Decide the workspace symlink trust model first, then either land/supersede via the scoped bootstrap-file PR or rework this allowlist with cache, config metadata, symlink-alias fixes, focused tests, and redacted real setup proof.

## Solution Assessment

No, not in this branch shape. The allowlist direction may be useful, but the patch needs maintainer trust-model approval and must preserve existing cache/config/symlink safety contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject case-only symlink aliases on writes:** `src/gateway/server-methods/agents.ts:882`
  - body: The write guard lowercases both paths, so on a case-sensitive workspace a symlink such as `MEMORY.md -> memory.md` can pass as a casing-only match and write a different file. That bypasses the PR's documented rule that in-workspace symlink aliases are read-only.
  - confidence: 0.88
- **[P2] Refresh bootstrap cache when allowlist changes:** `src/agents/bootstrap-cache.ts:10-12`
  - body: This returns a cached bootstrap snapshot for the session before reloading, while bootstrap content now depends on `allowedExternalPaths`. Removing an allowlisted directory can therefore keep previously cached external context injected into long-lived sessions.
  - confidence: 0.9
- **[P2] Add metadata for the new config key:** `src/config/zod-schema.agent-defaults.ts:39-44`
  - body: Adding `workspaceConfig` to the defaults schema makes it a public config key, but the PR does not add matching labels/help metadata for `agents.defaults.workspaceConfig.allowedExternalPaths`. Gateway config schema and UI/help surfaces will miss or poorly display the new setting until that contract is updated.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch broadens workspace file access and still has concrete symlink-alias and allowlist-revocation safety gaps.

Concerns:

- **[medium] Case-only symlink aliases can be written:** `src/gateway/server-methods/agents.ts:882`
  - body: The write guard lowercases the resolved and requested paths, so a case-only symlink alias inside a case-sensitive workspace can bypass the intended read-only alias policy and write another workspace file.
  - confidence: 0.88
- **[medium] Allowlist revocation can stay cached:** `src/agents/bootstrap-cache.ts:10`
  - body: Bootstrap content depends on `allowedExternalPaths`, but the PR returns a cached snapshot by session key before reloading or comparing allowlist state, so removed external trust can continue affecting an existing session.
  - confidence: 0.9
