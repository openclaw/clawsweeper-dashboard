## Summary

Keep this open. Current main still treats tool policy as named-tool availability: denying `write`/`edit` blocks filesystem tools and `apply_patch`, but it does not make an explicitly allowed `exec` read-only, and the docs still contain a no-file-modification example that allows `exec`.

## What This Changes

The issue requests that read-only tool policies account for write-capable `exec` usage, either by enforcement or by documenting that `exec` must be denied for no-file-modification agents.

## Best Possible Solution

Define one explicit read-only contract across code and docs: either make read-only exec enforceable through sandbox/host filesystem policy, or document that no-file-modification agents must deny `exec` and `process` alongside filesystem tools.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: use the documented `Safe execution (no file modifications)` policy, observe that `exec` remains allowed by the matcher, then run a shell write through the normal `exec` process path wherever the selected host/sandbox permits it.

## Solution Assessment

Unclear. The report is valid, but the best fix needs maintainer/security direction because enforcement, sandbox defaults, and docs-only correction have different compatibility and trust-model tradeoffs.
