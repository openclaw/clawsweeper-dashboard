## Summary

Keep open. Current main still treats tool policy as named-tool availability: denying `write`/`edit` blocks filesystem tools and `apply_patch`, but an explicitly allowed `exec` remains available and can run shell commands with filesystem side effects wherever the selected exec host permits them.

## What This Changes

The issue asks OpenClaw to prevent or clearly document `exec`-based file writes when an agent policy denies `write` and `edit`.

## Best Possible Solution

Define one explicit read-only contract across code and docs: either enforce read-only exec through sandbox/host filesystem policy, or document that no-file-modification agents must deny `exec` and `process` alongside filesystem tools.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: configure the documented no-file-modification profile, observe that policy filtering still keeps `exec`, then run a shell write through the selected exec host where filesystem permissions allow it.

## Solution Assessment

Unclear. The report is valid, but the best fix needs maintainer/security direction because enforcement, sandbox defaults, command inspection, and docs-only correction have different compatibility costs.
