## Summary

Keep open: current main and v2026.6.10 still exact-match plugin `contracts.tools`, while the active wildcard/dynamic-contract PR is open and still needs maintainer API/security-boundary acceptance plus real behavior proof.

## What This Changes

Request wildcard or glob-style entries in plugin `contracts.tools` so dynamic plugin tool namespaces do not require every runtime tool name to be enumerated.

## Best Possible Solution

Review and land or revise the active dynamic-tool contract PR with bounded prefix semantics, concrete allowlist filtering, docs/tests, and redacted real Gateway or plugin proof.

## Reproduction Assessment

Yes. Current main normalizes `contracts.tools` as literal strings and checks exact membership in registration, runtime tool loading, and manifest tool-owner resolution.

## Solution Assessment

Unclear as a final implementation. Bounded terminal-wildcard ownership is plausible, but the best fix must preserve manifest-first discovery and concrete allowlist filtering; unrestricted glob semantics would be too broad for a callable-tool exposure contract.
