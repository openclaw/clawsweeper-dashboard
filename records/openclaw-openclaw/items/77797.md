## Summary

Keep open: current main still treats `contracts.tools` as exact static tool names, and wildcard support remains an unmerged plugin-contract feature with an active implementation candidate at https://github.com/openclaw/openclaw/pull/78084. This needs maintainer API/design acceptance rather than cleanup closure.

## What This Changes

Request support for glob or wildcard entries in plugin manifest `contracts.tools` so dynamically registered plugin tools can be owned without enumerating every runtime tool name.

## Best Possible Solution

Define and land a bounded dynamic-tool ownership contract that preserves exact-match compatibility, undeclared-tool diagnostics, optional metadata behavior, explicit allowlist filtering, docs, and regression coverage.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken documented behavior. Source inspection confirms current main uses exact `contracts.tools` matching, which explains why wildcard entries are not supported today.

## Solution Assessment

Unclear; wildcard support is plausible, but it changes the public plugin manifest contract and tool ownership policy. The safer path is maintainer review of the bounded approach in https://github.com/openclaw/openclaw/pull/78084 with exact allowlist filtering and real behavior proof.
