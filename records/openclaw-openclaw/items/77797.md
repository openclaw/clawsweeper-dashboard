Keep open: current main and the latest release still exact-match plugin `contracts.tools`, while the active implementation path remains an open PR with maintainer API/security-boundary and proof questions unresolved.

## What This Changes

Requested support for glob or wildcard entries in plugin manifest `contracts.tools` so dynamically registered tool namespaces do not require every tool name to be enumerated.

## Best Possible Solution

Review and land or revise the linked dynamic-tool contract PR so OpenClaw has one documented, tested, bounded wildcard contract that preserves manifest-first loading and concrete allowlist filtering.

## Reproduction Assessment

Yes, source-level: current main exact-matches `contracts.tools` through the contract helper and registration gate. Because this is a feature request, a failing runtime bug repro is not the deciding gate.

## Solution Assessment

Unclear: bounded terminal namespace ownership is plausible, but unrestricted glob semantics would be too broad for a callable-tool exposure contract. The best fix needs maintainer-approved API semantics, concrete allowlist filtering, docs, tests, and real runtime proof.
