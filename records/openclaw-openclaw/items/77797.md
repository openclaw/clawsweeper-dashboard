## Summary

Keep open: current main still treats `contracts.tools` as exact static tool names, and wildcard support remains an unmerged plugin-contract feature with an open implementation candidate that needs maintainer acceptance and real behavior proof.

## What This Changes

The issue requests wildcard or glob entries in `contracts.tools` so dynamically registered plugin tools can be declared by namespace rather than enumerated one by one.

## Best Possible Solution

Define and land a bounded dynamic-tool ownership contract that preserves manifest-first discovery, exact-match compatibility, undeclared-tool diagnostics, tool metadata behavior, and plugin allow/deny policy.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken documented behavior. Source inspection shows current main uses exact `contracts.tools` matching, which explains why wildcard entries are not supported today.

## Solution Assessment

Unclear; wildcard support is plausible, but the exact semantics need maintainer approval. The safest path is to review the bounded wildcard approach in https://github.com/openclaw/openclaw/pull/78084 with real behavior proof before changing the public manifest contract.
