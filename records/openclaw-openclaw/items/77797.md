## Summary

Keep open: current `main` and `v2026.6.10` still exact-match plugin `contracts.tools`, and the active wildcard/dynamic-contract PR is open with API, security-boundary, and proof blockers rather than a merged fix.

## What This Changes

The issue requests glob or wildcard support in plugin manifest `contracts.tools` so dynamically registered tool namespaces do not require every tool name to be enumerated.

## Best Possible Solution

Review and land or revise the active dynamic-tool contract PR so OpenClaw has one documented, tested, bounded wildcard contract that preserves manifest-first loading and concrete allowlist filtering.

## Reproduction Assessment

Yes. Current main normalizes `contracts.tools` as literal strings and checks exact membership in registration, runtime tool loading, and manifest tool-owner resolution.

## Solution Assessment

Unclear as a final implementation. Bounded wildcard ownership is plausible, but unrestricted glob semantics would be too broad for a callable-tool exposure contract, so the linked PR needs maintainer-approved API semantics, docs, tests, and real runtime proof.
