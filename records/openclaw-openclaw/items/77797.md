## Summary

Keep open: current main treats `contracts.tools` as exact static tool names used for metadata-first discovery and runtime enforcement, so wildcard support would be a new manifest/plugin-contract design rather than a cleanup-close candidate.

## What This Changes

The issue requests glob or wildcard support in `contracts.tools` for plugins whose runtime tool catalog is fetched dynamically.

## Best Possible Solution

Define an additive, documented dynamic-tool ownership contract that preserves metadata-first discovery, undeclared-tool diagnostics, tool metadata behavior, and plugin allow/deny semantics.

## Reproduction Assessment

Not applicable; this is a feature request, not a bug report. Source inspection shows current main treats `contracts.tools` as exact static names, but there is no broken existing behavior to reproduce.

## Solution Assessment

Unclear; wildcard support may be useful, but adding glob semantics directly to `contracts.tools` needs maintainer design first. A safer direction may be an explicit pattern field or tightly specified glob semantics with contract tests.
