## Summary

Keep open: current main still treats plugin `contracts.tools` as exact static names, while wildcard ownership is an unmerged public manifest feature with an active implementation path at https://github.com/openclaw/openclaw/pull/78084. The remaining work needs maintainer API/security review and real runtime proof rather than conservative cleanup closure.

## What This Changes

The issue requests wildcard or glob entries in plugin `contracts.tools` so dynamic tool-owning plugins do not have to enumerate every runtime tool name.

## Best Possible Solution

Review and land a bounded dynamic-tool ownership contract through https://github.com/openclaw/openclaw/pull/78084 or an equivalent branch, with exact-match compatibility, concrete allowlist filtering, docs, regression coverage, and real runtime proof.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken documented behavior. Source inspection gives high confidence that current main uses exact `contracts.tools` matching, which explains why wildcard entries are unsupported today.

## Solution Assessment

Unclear; wildcard support is a plausible direction, but it changes the public plugin manifest contract and tool ownership policy. The safer path is maintainer review of the bounded approach in https://github.com/openclaw/openclaw/pull/78084 with allowlist filtering and real behavior proof.
