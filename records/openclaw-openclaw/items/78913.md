## Summary

Keep open: current main still returns the reported `sessions_spawn` accepted note for ordinary run-mode subagent spawns, and that note conflicts with the newer push-based guidance in the subagent system prompt and docs. The related https://github.com/openclaw/openclaw/issues/77426 is still open but covers a different `sessions_yield` MCP runtime failure, not this post-spawn guidance.

## What This Changes

The issue asks for `sessions_spawn` to stop recommending `sessions_yield` as the default wait step in push-delivered CLI/chat flows.

## Best Possible Solution

Align the normal run-mode accepted note with push-based auto-announce behavior, while preserving `sessions_yield` for explicit next-turn wait contexts and any runtime path that truly requires it.

## Reproduction Assessment

Yes. Source inspection shows a normal non-cron `sessions_spawn` run reaches `resolveSubagentSpawnAcceptedNote` and returns the constant that recommends `sessions_yield`; the reporter also provided a matching live tool result.

## Solution Assessment

Yes. The narrow maintainable fix is note-generation and focused test coverage, not changing subagent delivery or `sessions_yield` behavior globally.
