## Summary

Keep open: this is a focused security fix with useful proof, but the patch still leaves the initial running/background exec result able to return raw session tail data in `details.tail`. That means the same secret-shaped stdout class can still reach an exec tool-result surface before `process poll/log/list` redaction gets a chance to run.

## What This Changes

The PR redacts foreground exec, node-host exec, and process list/poll/log result text/details and adds focused redaction regression tests.

## Best Possible Solution

Use the existing redaction helpers at every exec/process result-emission boundary that can expose stdout, stderr, aggregated output, or session tails, while preserving raw process registry buffers for later bookkeeping.

## Reproduction Assessment

Yes, at source level: current main returns raw exec/process output through foreground exec, process poll/log/list, and the running exec result `details.tail`. A command that prints an API-key-shaped value before yielding/backgrounding would still expose that value in the running exec result.

## Solution Assessment

No. Result-emission redaction with `redactToolPayloadText` and `redactSecrets` is the right direction, but this PR is incomplete until the running/background exec result tail, and any model-visible update payloads that share that tail boundary, are redacted too.
