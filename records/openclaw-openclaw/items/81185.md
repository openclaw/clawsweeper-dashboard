## Label Justifications

- `P1`: The PR is tied to an active security issue where exec/process tool results can expose secret-shaped command output to the agent runtime.
- `merge-risk: 🚨 compatibility`: The diff intentionally changes returned exec/process output from exact command output to masked text, which can affect workflows relying on exact stdout/stderr.
- `merge-risk: 🚨 security-boundary`: The patch sits on the tool-result redaction boundary and still leaves a command-derived process-name path that can expose secret fragments.

## Summary

Keep open: the PR is directionally right and has strong fake-canary runtime proof, but two process display-name paths still derive names from the raw command before redaction, which can leak fragments of long standalone token-shaped secrets. Maintainers also need to accept the compatibility tradeoff that exec/process tool results now return masked output instead of exact stdout/stderr.

## What This Changes

The branch redacts secret-shaped exec, process, node-host, and live-update tool-result text/details, adds fake-canary regression coverage/proof, and skips provider wildcard model entries while building model catalog metadata.

## Best Possible Solution

Keep the tool-result redaction boundary, derive process display names from redacted commands before truncation, add long standalone token regression coverage, then have maintainers explicitly accept the masked-output contract before merge.

## Reproduction Assessment

Yes for the remaining blocker at source level: deriveSessionName truncates the raw command before redactSecrets runs, and the two changed process-name paths still use that order. The submitted live proof covers env-style canary surfaces but not a long standalone token command-name case.

## Solution Assessment

No as submitted: returned tool-result redaction is the right direction, but process names need pre-derivation redaction and the masked-output contract needs maintainer acceptance before merge.
