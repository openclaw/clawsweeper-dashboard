## Summary

Keep open: the PR targets a real browser-agent download visibility bug, but the patch is not merge-ready because it samples asynchronous download state too early and still lacks real click-to-download proof.

## What This Changes

The PR adds recent download tracking to the Browser Playwright page state and threads a `downloads` payload through `/act` responses for click/evaluate/batch actions.

## Best Possible Solution

Wire click-triggered downloads through an awaited per-action download observer or the existing managed download helpers, update proxy/path handling for the chosen response shape, and prove a real click-to-download flow before merge.

## Reproduction Assessment

Yes, at source level. The current and PR code show a click-triggered download is saved asynchronously while `/act` can return before that save is recorded, so the reported missing confirmation can still happen.

## Solution Assessment

No. The narrow direction is useful, but reading a background recent-download array after the action is too brittle; a per-action awaited download observer or the existing managed download helper path is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wait for the triggered download before returning:** `extensions/browser/src/browser/pw-tools-core.interactions.ts:1747-1755`
  - body: The PR appends to `recentDownloads` only after the async auto-save finishes, but `/act` reads that array immediately after `executeSingleAction`. A normal click can therefore still return `{ ok: true }` without `downloads`, which is the bug this PR is meant to fix. Arm and await a per-action download event/save, or reuse the existing `downloadViaPlaywright`/waiter path for this response.
  - confidence: 0.91
- **[P2] Use a monotonic cursor for recent downloads:** `extensions/browser/src/browser/pw-tools-core.interactions.ts:1715`
  - body: `downloadCountBefore` is the array length, but `recentDownloads` is capped by `shift()`. Once the list already has 20 entries, saving a new download keeps the length at 20, so `slice(downloadCountBefore)` returns an empty array and later downloads are hidden. Track a sequence/timestamp or collect downloads for the action directly instead of using the capped array length as a cursor.
  - confidence: 0.88
- **[P2] Transfer returned downloads through the node proxy:** `extensions/browser/src/browser/pw-tools-core.interactions.ts:1750`
  - body: The new payload returns `downloads.recent[].savedPath`, but the node-host browser proxy only collects and rewrites top-level `path`, `imagePath`, and `download.path`. For node-backed browser runs, the agent would receive a remote node path and no persisted local file for this new `/act` download result. Extend proxy collection/rewrite for the new shape or reuse the existing `download.path` contract.
  - confidence: 0.84
