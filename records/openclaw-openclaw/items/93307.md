## Summary

Keep open: the browser download visibility gap is real, but this PR is not merge-ready because it samples completed downloads after the action instead of waiting for the async save, so the click response can still omit the new field; the supplied proof is unit-test only.

## What This Changes

The branch records auto-saved browser downloads in page state and adds an optional `downloads` payload to Playwright-backed `/act` responses.

## Best Possible Solution

Tie `/act` download reporting to a promise or event record armed before the action and resolved after `saveAs` completes, or land a maintained explicit download/wait-for-download action surface with real managed-browser proof.

## Reproduction Assessment

Yes, at source level. Current main auto-saves unmanaged downloads asynchronously, while `/act` click responses return success without any download completion payload.

## Solution Assessment

No. Sampling a bounded recent-download list after the action is weaker than arming and awaiting a download/save record around the action or exposing the existing explicit managed download actions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Wait for the save before reporting downloads:** `extensions/browser/src/browser/pw-tools-core.interactions.ts:1731-1736`
  - body: `recentDownloads` is only pushed after the background handler finishes `download.saveAs`, but this code slices the list immediately after `executeSingleAction` or the batch returns. For a real click-triggered download, the click can resolve while the save is still in progress, so the `/act` response can still omit `downloads` for the exact flow this PR is meant to fix. Arm a per-action download record before the action and await/settle the saved result before building the response.
  - confidence: 0.88
- **[P2] Do not use the bounded list length as a cursor:** `extensions/browser/src/browser/pw-tools-core.interactions.ts:1715`
  - body: This records `downloadCountBefore` as the current list length, but `recentDownloads` is capped at 20 by shifting the oldest entry. Once the list is full, a new download keeps the length at 20, so `slice(downloadCountBefore)` returns an empty array and every later download is missed. Track a sequence/timestamp cursor or pending action-owned records instead.
  - confidence: 0.86
