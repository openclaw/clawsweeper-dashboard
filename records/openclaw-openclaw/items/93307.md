## Summary

Keep open: the PR targets a real browser-agent download feedback bug, but this patch is not merge-ready because it samples an asynchronous, bounded download list and still lacks real click-to-download proof.

## What This Changes

The branch records recent auto-saved browser downloads in page state, computes a before/after delta around Playwright `/act`, and adds an optional `downloads` payload to `/act` responses.

## Best Possible Solution

Tie `/act` download reporting to an action-scoped waiter or promise that is armed before the click and awaited through saved-file completion, or land one explicit download/waitfordownload action PR as the canonical path with real managed-browser proof.

## Reproduction Assessment

Yes, at source level: current main auto-saves unmanaged downloads asynchronously while `/act` click responses return success without a download completion payload. I did not run a live browser download in this read-only review.

## Solution Assessment

No. Sampling a bounded recent-download list after the action is weaker than arming and awaiting an action-scoped download/save record, and maintainers still need to choose the canonical path among overlapping download PRs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wait for the download save before reporting it:** `extensions/browser/src/browser/pw-tools-core.interactions.ts:1731-1736`
  - body: `recentDownloads` is pushed only after the background handler finishes `download.saveAs`, but this code slices the list immediately after the action returns. A real click can resolve while the save is still in progress, so `/act` can still omit `downloads` for the flow this PR is meant to fix; arm and await an action-owned saved-download record instead.
  - confidence: 0.88
- **[P2] Do not use the bounded list length as a cursor:** `extensions/browser/src/browser/pw-tools-core.interactions.ts:1715`
  - body: `downloadCountBefore` stores the current list length, but `recentDownloads` is capped by shifting the oldest entry. Once the list is full, a new download keeps the length unchanged, so `slice(downloadCountBefore)` returns an empty array and later downloads are missed; track a sequence/timestamp cursor or action-owned records.
  - confidence: 0.86
