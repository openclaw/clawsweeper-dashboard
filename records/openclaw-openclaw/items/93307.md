## Summary

Keep open: the underlying browser download feedback bug is real, but this branch is not merge-ready because it can still miss click-triggered downloads after asynchronous saves or once the recent-download buffer is full, and it lacks real browser proof.

## What This Changes

The branch records auto-saved browser downloads in page state and adds optional download metadata to Playwright `/act` responses for batch, evaluate, click, clickCoords, and other actions.

## Best Possible Solution

Tie `/act` download reporting to an action-scoped waiter or monotonic sequence that waits through saved-file completion, or land one explicit `download`/`waitfordownload` action PR as the canonical path with real managed-browser proof.

## Reproduction Assessment

Yes. Source inspection shows current main auto-saves unmanaged page downloads while `/act` click responses return success without a download completion payload; I did not run a live browser download in this read-only review.

## Solution Assessment

No. Sampling a bounded shared list after an action is weaker than an action-scoped waiter or monotonic saved-download record, and maintainers still need to choose the canonical API shape among overlapping browser-download PRs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wait for the saved download before reporting it:** `extensions/browser/src/browser/pw-tools-core.interactions.ts:1747-1751`
  - body: `recentDownloads` is pushed only after the background handler finishes `download.saveAs`, but this code slices the list immediately after the action returns. A real click can resolve while the save is still in progress, so `/act` can still omit `downloads` for the flow this PR is meant to fix; arm and await an action-owned saved-download record instead.
  - confidence: 0.9
- **[P2] Track downloads with a monotonic cursor:** `extensions/browser/src/browser/pw-tools-core.interactions.ts:1715`
  - body: `downloadCountBefore` stores the current bounded-list length, but `recentDownloads` is capped by shifting the oldest entry. Once the list is full, a new download keeps the length unchanged, so `slice(downloadCountBefore)` returns an empty array and later downloads are missed; use a sequence/timestamp cursor or action-owned records.
  - confidence: 0.88
