## Summary

Keep open: the PR targets a real browser download feedback gap, but the patch can still miss click-triggered downloads and only supplies unit-test proof. It also overlaps the open explicit download-action approach, so maintainers still need to choose the canonical API shape before landing this branch.

## What This Changes

The PR records background auto-saved browser downloads in page state and returns optional `downloads` metadata from managed `/act` responses.

## Best Possible Solution

Use an action-scoped download waiter or monotonic saved-download cursor that waits through saved-file completion, or land a proof-positive explicit download-action API as the canonical browser download path.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.9` auto-save unmanaged downloads but `/act` click responses return no download payload, and the browser tool schema has no explicit download action. I did not run a live browser click-to-download scenario in this read-only review.

## Solution Assessment

No. Sampling a bounded shared list after an action is weaker than the existing waiter pattern, and the canonical browser download API shape still needs maintainer choice.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wait for the saved download before reporting it:** `extensions/browser/src/browser/pw-tools-core.interactions.ts:1747-1751`
  - body: `recentDownloads` is appended only after the background handler finishes `download.saveAs`, but this code slices the list immediately after the action returns. A click can resolve while the save is still in progress, so `/act` can still omit `downloads` for the flow this PR is intended to fix; arm and await an action-owned saved-download record instead.
  - confidence: 0.9
- **[P2] Track downloads with a monotonic cursor:** `extensions/browser/src/browser/pw-tools-core.interactions.ts:1715`
  - body: `downloadCountBefore` stores a bounded-list length. Once `recentDownloads` is full, a new download shifts the old entry and keeps the length at 20, so `slice(downloadCountBefore)` is empty and later downloads are missed; use a monotonic sequence/timestamp cursor or action-owned records.
  - confidence: 0.88
