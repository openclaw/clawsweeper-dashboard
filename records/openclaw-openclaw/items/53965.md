## Summary

Keep open: the PR is not mergeable because it adds an unused helper while current main still writes the affected secrets/auth files through the unguarded active writer. Related #62099/#67077 are useful Windows EPERM context, but they cover a later auth-profile bookkeeping failure mode and do not supersede this patch.

## What This Changes

Adds `src/shared/fs-atomic-fix.ts` with a synchronous temp-file writer that catches `chmodSync` failures before renaming.

## Best Possible Solution

Replace the orphan helper with a focused change on the active secrets persistence writer, preserving parent-directory creation and secure permissions while adding targeted regression coverage and a changelog entry.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the new helper has no callers, while `runSecretsApply` still reaches `writeTextFileAtomic` with an unguarded `chmodSync`. A focused regression test can mock `fs.chmodSync` throwing `EPERM` on that active path without a live Windows/Docker mount.

## Solution Assessment

No. Adding a disconnected helper is not the narrow maintainable fix; the guarded behavior needs to live in the writer that `secrets apply` actually calls, with the existing directory and permission semantics preserved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Wire the helper into the active writers:** `src/shared/fs-atomic-fix.ts:7`
  - body: This adds `writeTextFileAtomicSync`, but nothing imports or calls it. Current secrets apply still uses `writeTextFileAtomic` in `src/secrets/shared.ts`, whose `chmodSync` remains unguarded, so merging this leaves the reported Windows/Docker crash unchanged.
  - confidence: 0.98
- **[P2] Preserve parent directory creation:** `src/shared/fs-atomic-fix.ts:8-9`
  - body: The new writer creates the temp file without first ensuring the destination directory exists. The existing writer calls `ensureDirForFile(pathname)`, so using this helper would regress new auth/env writes to `ENOENT` when the parent directory is missing.
  - confidence: 0.94
- **[P2] Narrow the chmod fallback:** `src/shared/fs-atomic-fix.ts:14-15`
  - body: The catch suppresses every `chmodSync` failure. If this helper is used for auth/config writes, unexpected permission hardening failures can leave credential-bearing files written with default permissions while the caller sees success.
  - confidence: 0.84
