## Summary

The PR replaces MiniMax image, OAuth, TTS, and video success-path JSON reads with bounded `readResponseWithLimit` parsing and adds video/proof tests.

## Best Possible Solution

Rebase and narrow the branch: preserve merged TTS behavior, let the open video PR own video if it lands first, and fix the remaining MiniMax OAuth/image gaps with shared bounded JSON helpers and an image-sized cap.

## Reproduction Assessment

Yes, source-reproducible. Current main still has raw `response.json()` on MiniMax image, OAuth authorization, and video success/control responses, and the PR body includes loopback terminal proof for the bounded reader behavior.

## Solution Assessment

No. Bounding these reads is the right direction, but this branch is too broad and dirty; the best fix should use `readProviderJsonResponse`, preserve current-main TTS, coordinate with the video PR, and use an image-sized cap for inline image JSON.
