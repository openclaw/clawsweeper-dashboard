## Summary

Keep this PR open because it targets a real MiniMax unbounded-response hardening gap, but it is not merge-ready: the patch duplicates the shared provider JSON helper with a new 1 MiB contract, leaves the video tests out of sync with body-based reads, overlaps the focused MiniMax video PR, and the proof/body are stale for the current head.

## What This Changes

The PR changes MiniMax OAuth authorization and video success/control JSON reads from `response.json()` to manual bounded reads using `readResponseWithLimit` plus `JSON.parse`.

## Best Possible Solution

Narrow this branch to the MiniMax OAuth authorization-code success read, implement it with `readProviderJsonResponse`, and let https://github.com/openclaw/openclaw/pull/96889 own MiniMax video hardening.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.10` still have raw `response.json()` on MiniMax OAuth authorization and video submit, poll, and metadata success bodies. I did not run a live MiniMax endpoint or local streaming harness in this read-only review.

## Solution Assessment

No. Bounded JSON is the right direction, but the current patch is not the best fix because it duplicates `readProviderJsonResponse`, changes cap semantics, leaves video tests with non-Response mocks, and overlaps the focused MiniMax video PR.
