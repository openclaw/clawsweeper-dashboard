## Summary

Keep open. The MiniMax oversized-success-response problem still exists on current main and the proof is strong, but this PR is not merge-ready because it bypasses the shared provider JSON helper, introduces a smaller MiniMax-specific cap, and leaves the video test fixtures mismatched with the new body-reader behavior.

## What This Changes

The branch changes MiniMax OAuth authorization and video submit, poll, and metadata success JSON reads from `response.json()` to manual `readResponseWithLimit` plus `JSON.parse` with a 1 MiB cap.

## Best Possible Solution

Use `readProviderJsonResponse` for the MiniMax OAuth success read, and either let https://github.com/openclaw/openclaw/pull/96889 own MiniMax video or align this branch with its shared-helper and real-Response test shape before merge.

No. Bounded reads are the right direction, but this branch is not the best fix because it bypasses the shared provider JSON helper, changes cap semantics, and leaves the video tests incompatible with body-based reads.
