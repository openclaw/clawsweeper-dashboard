## Summary

Keep open: the MiniMax bounded-read hardening is still useful because current main and the latest release still have unbounded OAuth/video success JSON reads, but this branch is not merge-ready because it creates a parallel 1 MiB JSON cap instead of using the shared provider helper and leaves the video tests on json-only mock responses.

## What This Changes

The branch replaces MiniMax OAuth authorization and video submit, poll, and metadata success `response.json()` calls with manual `readResponseWithLimit` plus `JSON.parse` using a 1 MiB cap.

## Best Possible Solution

Use `readProviderJsonResponse` for the remaining MiniMax OAuth success read, and either let https://github.com/openclaw/openclaw/pull/96889 own video or update this branch to use the same shared-helper and real-Response test shape for all MiniMax video reads.

## Reproduction Assessment

Yes. Current main and `v2026.6.10` still call `response.json()` for the MiniMax OAuth authorization-code response and the three MiniMax video success/control responses; the PR body also provides loopback proof that the bounded reader cancels oversized chunked bodies.

## Solution Assessment

No. Bounded reads are the right direction, but this is not the best fix because it bypasses the shared provider JSON helper, changes cap semantics, and leaves the video test fixtures incompatible with body-based reads.
