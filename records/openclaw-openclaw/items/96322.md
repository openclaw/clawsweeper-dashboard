## Summary

Keep open: the bounded-read hardening is valid in principle and the proof shows the shared reader cancels oversized streams, but this branch is not merge-ready because it lowers MiniMax TTS to a 1 MiB cap, caps inline base64 image JSON at 1 MiB, overlaps focused response-limit PRs, and currently fails lint.

## What This Changes

The PR replaces six MiniMax success JSON reads in image generation, OAuth authorization, TTS, and video generation with fixed 1 MiB `readResponseWithLimit` calls, updates video tests to use real `Response` bodies, and adds a standalone proof script.

## Best Possible Solution

Keep the response-limit hardening, but narrow this branch to the unique MiniMax OAuth authorization success-read gap or rework media-bearing paths through the shared domain-sized helpers owned by the focused response-limit PRs.

## Reproduction Assessment

Yes, source-reproducible: current main still has raw MiniMax image, OAuth authorization, and video success `response.json()` reads, and the PR body gives loopback terminal proof that the bounded reader cancels oversized streamed JSON. The cap regressions are also source-backed by the current inline media payload code and CI lint logs.

## Solution Assessment

No. The OAuth hardening is useful, but applying a generic 1 MiB cap to media-bearing image and TTS success bodies is not the safest fix when https://github.com/openclaw/openclaw/pull/96495, https://github.com/openclaw/openclaw/pull/96889, and the merged TTS work already provide narrower ownership paths.
