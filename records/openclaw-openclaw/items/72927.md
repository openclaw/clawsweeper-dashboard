## Summary

Keep open. Current main still lacks Xiaomi MiMo VoiceDesign support: the Xiaomi speech provider only exposes preset voice/style config, the request serializer always sends `audio.voice`, and the docs describe only built-in voice configuration. Xiaomi’s current public TTS docs define `mimo-v2.5-tts-voicedesign` as a separate supported model whose `user` message is the voice description, so this remains a real provider-local feature gap.

## What This Changes

Add Xiaomi MiMo v2.5 TTS VoiceDesign support to OpenClaw’s Xiaomi TTS provider configuration while preserving existing preset `voice` behavior.

## Best Possible Solution

Extend only the bundled Xiaomi plugin with a documented VoiceDesign config shape that maps to Xiaomi’s official `mimo-v2.5-tts-voicedesign` contract: use the configured prompt as the required `user` message, keep the target text as the `assistant` message, omit preset `audio.voice`, and preserve existing `voice`, `style`, `format`, `mimo-v2.5-tts`, and `mimo-v2-tts` behavior.

## Reproduction Assessment

Yes. Static inspection of current main reproduces the gap: there is no VoiceDesign config key or model metadata, and the Xiaomi request body always includes preset `audio.voice`.

## Solution Assessment

Yes. A provider-local Xiaomi implementation is the narrowest maintainable solution; a generic core `extraBody` passthrough or new SDK seam would be broader than this provider-specific API shape requires.
