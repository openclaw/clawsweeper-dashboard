## Summary

Keep open: current main and the latest release still lack the requested `sessions_spawn(runtime="acp", mode="persistent", affinityKey)` contract for cron/orchestrator-owned persistent ACP reuse, and the remaining work needs ACP API, ownership, delivery, and security review rather than cleanup closure.

## What This Changes

Request support for non-thread ACP persistent affinity so cron and orchestrator callers can deterministically reuse ACP sessions by owner, backend, cwd, and affinity key.

## Best Possible Solution

Add a reviewed upstream ACP persistent-affinity contract scoped by trusted requester owner, target backend/agent, cwd, and affinity key, with explicit authorization, delivery/lifecycle cleanup semantics, docs, and regression coverage.

## Reproduction Assessment

Yes, source-level. Current main accepts only `run`/`session`, has no `affinityKey`, rejects non-thread ACP session mode, and creates random ACP child session keys.

## Solution Assessment

Unclear as a direct implementation path. The downstream `persistent` plus `affinityKey` shape is useful prior art, but upstream needs maintainer approval for requester identity, authorization, deterministic key scoping, delivery, and cleanup semantics.
