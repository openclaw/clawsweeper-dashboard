## Summary

Keep open. Current main and v2026.6.8 still expose ACP `sessions_spawn` as `run`/`session`, still reject ACP `mode="session"` without `thread=true`, and still have no `affinityKey` or cron/hook persistent-affinity contract; the active hub-delegated PR explicitly leaves cron/hook affinity out of scope.

## What This Changes

The issue requests a first-class non-thread ACP persistence mode for cron/orchestrator sessions, with deterministic affinity reuse scoped by requester identity, backend, cwd, and an affinity key.

## Best Possible Solution

Define an explicit cron/hook/orchestrator ACP affinity contract scoped by requester, backend, cwd, and affinity key, then update schema, runtime, docs, and regression tests while preserving the thread-bound `mode="session"` contract.

## Reproduction Assessment

Yes, source-level. Current main only accepts ACP `run`/`session`, rejects `mode="session"` without `thread=true`, creates random ACP session keys, and exact search found no `affinityKey` path.

## Solution Assessment

Unclear. The downstream `mode="persistent"` plus `affinityKey` shape is plausible, but maintainers still need to choose the scoping, lifecycle, and relationship to hub-delegated ACP workers before implementation.
