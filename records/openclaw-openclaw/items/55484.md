## Summary

Keep open: current main and the latest release still lack the requested `sessions_spawn(runtime="acp", mode="persistent", affinityKey)` contract for cron/orchestrator-owned ACP reuse, and the remaining work is an ACP API, delivery, ownership, and security decision rather than conservative cleanup.

## What This Changes

The issue requests a supported ACP `sessions_spawn` persistent-affinity mode for cron/orchestrator sessions that is not tied to channel thread binding.

## Best Possible Solution

Add a reviewed upstream ACP persistent-affinity contract scoped by trusted requester owner, target backend/agent, cwd, and affinity key, with explicit authorization, delivery/lifecycle cleanup semantics, docs, and regression coverage.

## Reproduction Assessment

Yes, source-level. Current main accepts only `run` and `session`, has no `affinityKey`, rejects non-thread ACP session mode, and creates random ACP child session keys.

## Solution Assessment

Unclear as a direct implementation path. The downstream `persistent` plus `affinityKey` shape is useful prior art, but upstream needs maintainer approval for requester identity, authorization, deterministic key scoping, delivery, and cleanup semantics.
