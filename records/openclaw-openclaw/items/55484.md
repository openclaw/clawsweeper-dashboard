## Summary

Keep open. Current main and v2026.6.10 still lack an ACP `affinityKey` or `mode="persistent"` `sessions_spawn` contract, still expose ACP spawns as `run`/`session`, and still reject unthreaded ACP `mode="session"`; related cron and hub-delegated work does not implement the cron/hook affinity request.

## What This Changes

The issue requests ACP `sessions_spawn` support for non-thread persistent affinity, especially cron/hook or orchestrator sessions that need reusable ACP workers without channel thread binding.

## Best Possible Solution

Define an explicit cron/hook/orchestrator ACP affinity contract scoped by requester identity, backend, cwd, and affinity key, then update schema, runtime, docs, and regression tests while keeping thread-bound `mode="session"` intact.

## Reproduction Assessment

Yes, source-level: current main has no `affinityKey` schema or runtime path, accepts only `run`/`session`, and rejects ACP `mode="session"` without `thread=true`. I did not execute a live ACP cron/hook flow because the missing contract is directly visible in source and docs.

## Solution Assessment

Unclear. The downstream `mode="persistent"` plus `affinityKey` shape is plausible, but maintainers still need to choose ownership, lifecycle, security scope, and relation to cron ACP or hub-delegated work before implementation.
