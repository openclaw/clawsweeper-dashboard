## Summary

Keep open. Current main still lacks the requested non-thread ACP persistent-affinity contract: `sessions_spawn` for ACP only accepts `run` or thread-bound `session`, has no `affinityKey` input, and still creates fresh random ACP child session keys for non-thread spawns. The downstream WuKongAI-CMU/openclaw#3 implementation linked in the discussion remains useful prior art, but it has not landed upstream.

## What This Changes

Add a non-thread ACP `sessions_spawn` persistent-affinity mode with `affinityKey` for trusted cron, hook, or orchestrator sessions so recurring orchestrations can reuse deterministic ACP sessions without channel thread binding.

## Best Possible Solution

Keep this as an upstream tracker and design the ACP contract before implementation: add an explicit `mode: "persistent"` plus `affinityKey` path for trusted non-thread callers, scope deterministic session keys by requester owner identity, backend, cwd, and affinity key, preserve existing affinity-backed sessions when a later dispatch fails, and document/test the authorization boundary.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: `sessions_spawn({ runtime: "acp", mode: "persistent", affinityKey: "..." })` has no schema/parser support, while `mode: "session"` without `thread=true` returns `thread_required` and non-thread ACP spawns use a random session key.

## Solution Assessment

Unclear. The downstream implementation is strong prior art and the separate persistent-affinity mode is a plausible shape, but the best upstream solution needs maintainer review of requester identity, authorization, cleanup, and session-key scoping before a repair lane changes core ACP behavior.
