## Label Justifications

- `impact:session-state`: The request is about preserving and reusing ACP session context across cron/orchestrator invocations without a thread binding.
- `impact:security`: Caller-controlled persistent affinity can cross security boundaries if requester, workspace, backend, or ownership scoping is wrong.

## Summary

Keep open. Current main and the latest release still only expose `run`/`session` ACP spawn modes, still reject ACP `mode="session"` without `thread=true`, and do not contain the requested `affinityKey`/non-thread persistent reuse path; the linked hub-delegated PR explicitly excludes cron/hook affinity.

## What This Changes

Request ACP sessions spawned by cron or orchestrator flows to reuse a non-thread persistent session through deterministic affinity.

## Best Possible Solution

Define a first-class ACP non-thread persistence contract for cron/hook/orchestrator ownership, scoped by requester, backend, cwd, and affinity key, then implement schema, runtime, docs, and regression tests without accidentally relaxing thread-bound session behavior.

## Reproduction Assessment

Yes, source-level. Current main rejects `sessions_spawn({ runtime: "acp", mode: "session" })` without `thread=true`, exposes no `affinityKey`, and documents ACP session mode as thread-bound; I did not run a live ACP/cron reproduction in this read-only review.

## Solution Assessment

Unclear. A separate persistent-affinity path is plausibly safer than simply allowing unthreaded `mode="session"`, but maintainers need to decide the permanent contract and how it relates to https://github.com/openclaw/openclaw/issues/53548 and https://github.com/openclaw/openclaw/pull/91093.
