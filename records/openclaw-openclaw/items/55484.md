## Summary

Keep open. Current main and v2026.6.8 still expose ACP `sessions_spawn` as `run`/`session`, still reject ACP `mode="session"` without `thread=true`, and still have no `affinityKey` or cron/hook persistent-affinity contract; the open hub-delegated PR explicitly leaves cron/hook affinity reuse out of scope.

## What This Changes

Request support for a non-thread ACP persistence mode that lets cron, hook, or orchestrator callers reuse a deterministic ACP session by affinity key.

## Best Possible Solution

Define an explicit cron/hook/orchestrator ACP affinity contract scoped by requester, backend, cwd, and affinity key, then update schema, runtime, docs, and regression tests while preserving the thread-bound `mode="session"` contract.

## Reproduction Assessment

Yes, source-level. Current main only accepts ACP `run`/`session`, rejects `mode="session"` without `thread=true`, creates random ACP session keys, and exact source search found no `affinityKey` path.

## Solution Assessment

Unclear. The downstream `mode="persistent"` plus `affinityKey` shape is plausible, but maintainers still need to choose scoping, lifecycle, and how this relates to hub-delegated ACP workers before implementation.
