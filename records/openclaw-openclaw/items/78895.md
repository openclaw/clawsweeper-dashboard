## Summary

Keep open: current main has an opt-in global media cleanup path and manual QQBot storage cleanup, but QQBot docs still do not make those controls discoverable for this QQBot storage problem. The latest maintainer comment also supports documenting the existing `media.ttlHours` and `/bot-clear-storage` path before adding a QQBot-specific retention setting or default deletion policy.

## What This Changes

Request automatic or configurable cleanup for stale QQBot downloaded media accumulating under the QQBot downloads directory.

## Best Possible Solution

Document QQBot storage cleanup around the existing global `media.ttlHours` setting and `/bot-clear-storage`, while keeping default deletion opt-in and leaving per-channel retention or legacy doctor cleanup for a separate maintainer decision if users need it.

## Reproduction Assessment

Not applicable as a feature/docs request. Source inspection confirms the current behavior: QQBot media uses the shared media tree, global cleanup only runs when `media.ttlHours` is configured, and QQBot docs do not yet explain that cleanup path.

## Solution Assessment

Yes for the first maintainer-supported step: documenting the existing global retention knob is narrower and safer than adding a QQBot-specific config key or surprising default deletion. A per-channel setting should wait until there is evidence that global media retention is insufficient.
