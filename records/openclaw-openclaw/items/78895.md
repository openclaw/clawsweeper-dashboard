## Summary

Keep open: current main still only has manual QQBot storage cleanup plus opt-in global `media.ttlHours`; the requested QQBot/default retention behavior remains a product/config decision, and the follow-up comment says a PR may come but no linked/open PR was found.

## What This Changes

Request automatic cleanup for stale QQBot downloaded media, potentially via a configurable `channels.qqbot.downloadRetentionDays` retention window and startup or periodic pruning.

## Best Possible Solution

Decide whether the supported path is QQBot documentation for existing `media.ttlHours`, a plugin-owned retention/default cleanup feature, or a deliberate legacy-path repair rule, then implement that chosen policy with focused tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the current behavior: QQBot downloads into the media tree, `/bot-clear-storage` is manual, and automatic recursive media cleanup only runs when the global `media.ttlHours` path supplies a TTL.

## Solution Assessment

Unclear. A QQBot-specific retention setting is not obviously better than documenting or extending the existing global media TTL, and the legacy-path/default-deletion policy needs maintainer direction before implementation.
