## Summary

Keep open: the request is valid product/docs follow-up, but current main only has opt-in global media retention plus manual QQBot cleanup, not QQBot-specific/default retention or legacy-path guidance. A later contributor comment also treats the request as valid and prefers surfacing the existing global retention knob before adding a new QQBot setting.

## What This Changes

The issue requests configurable or default automatic cleanup for stale QQBot downloaded media.

## Best Possible Solution

Keep automatic deletion opt-in, document QQBot storage cleanup around `media.ttlHours` and `/bot-clear-storage`, and separately decide whether legacy QQBot download directories should be surfaced by doctor/checkup.

## Reproduction Assessment

Not applicable as a feature request rather than broken defined behavior. Source inspection confirms current-main behavior: QQBot writes downloads into the shared media tree, and automatic cleanup only starts when `media.ttlHours` is configured.

## Solution Assessment

Unclear. A QQBot-specific `downloadRetentionDays` default is not clearly the best first solution because `media.ttlHours` already covers the current shared media tree; docs/setup discoverability plus an optional legacy-path doctor hint looks narrower.
