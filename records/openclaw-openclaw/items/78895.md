## Summary

Current main now solves the central stale QQBot media problem well enough: QQBot media lands under the shared media tree, and configured `media.ttlHours` drives startup plus hourly recursive cleanup. The full cleanup loop is current-main-only and is not in `v2026.6.6`.

## What This Changes

The issue requests automatic or configurable cleanup for stale QQBot downloaded media under the OpenClaw user data directory.

## Best Possible Solution

Keep the current opt-in global media retention path instead of adding a QQBot-specific retention setting or default deletion policy.

## Reproduction Assessment

Yes from source for the reported old behavior: older/shipped paths lacked the current configured startup/hourly media cleanup for QQBot’s shared media tree. Current main now has a source-backed cleanup path through `media.ttlHours`.

## Solution Assessment

Yes. Reusing the global `media.ttlHours` cleanup path is the cleaner solution than adding a QQBot-specific retention key, because it avoids extra config surface and matches the shared media-root ownership.
