## Summary

Keep open. Current main still treats inbound Slack message content as text-first and does not recover full agent body text from top-level Slack `blocks` or rich-text payloads when `message.text` is only a preview; the open system-event preview PR is adjacent but does not solve that body-recovery gap.

## What This Changes

The issue requests that Slack DM inbound messages longer than 160 characters be delivered to the agent in full instead of being cut at the preview boundary.

## Best Possible Solution

Fix the Slack plugin inbound content path: type and preserve top-level message `blocks`, add a focused Block Kit/rich-text-to-text resolver for inbound message bodies, prefer the block-derived body when `message.text` is missing or only a prefix/preview, and keep the system-event preview behavior separated from agent body delivery.

## Reproduction Assessment

Unclear. The issue has concrete live Slack DM steps and current code can be statically checked with a synthetic text-preview-plus-blocks payload, but this review did not have a representative live Slack Socket Mode DM event to confirm the exact Slack field carrying the tail.

## Solution Assessment

Yes. A Slack plugin-local inbound-body fix is the narrowest maintainable path; removing the 160-character system-event preview alone, as in #67761, is useful for #67503 but does not recover full Slack DM content when the event `text` field is already shortened.
