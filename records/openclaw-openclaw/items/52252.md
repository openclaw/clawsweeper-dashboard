## Summary

Keep open: the PR targets a real daemon path fragility, but the submitted resolver is still narrower than OpenClaw’s Volta support and lacks required real behavior proof.

## What This Changes

The PR adds Volta tools/image-to-shim canonicalization to `resolveStableNodePath` plus unit tests for the default Unix `~/.volta` layout and missing-shim fallback.

## Best Possible Solution

Update the resolver to derive Volta home from any `<VOLTA_HOME>/tools/image/node/<version>` root, handle Unix and Windows image/shim layouts, preserve missing-shim fallback, and attach redacted real Volta gateway/install proof.

## Reproduction Assessment

Yes, source-level. Current main returns Volta tools/image Node paths unchanged, and the PR branch still leaves custom `VOLTA_HOME` and Windows Volta image layouts version-pinned despite accessible stable shims.

## Solution Assessment

No, not as submitted. Resolver-level normalization is the right boundary, but the matcher should parse Volta home generically and account for upstream Windows layout before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Match custom Volta homes:** `src/infra/stable-node-path.ts:19`
  - body: The matcher only accepts paths containing `/.volta/tools/image/...`, but OpenClaw already treats env-configured `VOLTA_HOME/bin` as a supported service PATH input. A path such as `/opt/volta/tools/image/node/24.14.0/bin/node` still falls through unchanged, so the installed service remains version-pinned after this fix.
  - confidence: 0.94
- **[P2] Handle Windows Volta image paths:** `src/infra/stable-node-path.ts:19-25`
  - body: The Windows branch still expects a `.volta` home plus `bin/node.exe`, but upstream Volta defaults to `%LOCALAPPDATA%\Volta`, places the Node image binary directly under the version directory on Windows, and uses `.cmd` shims. Those installs will not be canonicalized by this PR.
  - confidence: 0.88
