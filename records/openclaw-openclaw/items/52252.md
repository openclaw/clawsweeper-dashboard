## What This Changes

The branch adds Volta tools/image matching in `resolveStableNodePath` plus unit tests for default Unix shim resolution and missing-shim fallback.

## Best Possible Solution

Update the resolver to derive Volta home from any Volta image root, handle Unix and Windows layouts, preserve missing-shim fallback, and attach redacted real Volta gateway/install proof.

## Reproduction Assessment

Yes, source-level. Current main returns Volta tools/image Node paths unchanged, and the PR branch still leaves custom `VOLTA_HOME` and Windows Volta layouts version-pinned despite accessible stable shims.

## Solution Assessment

No, not as submitted. Resolver-level normalization is the right boundary, but the matcher should parse Volta home generically and follow Volta's Unix and Windows image/shim layout before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Match custom Volta homes:** `src/infra/stable-node-path.ts:19`
  - body: The matcher only recognizes paths containing `/.volta/tools/image/...`, but OpenClaw already supports env-configured `VOLTA_HOME/bin` in service PATH construction. With `VOLTA_HOME=/opt/volta`, `/opt/volta/tools/image/node/<version>/bin/node` still returns unchanged, leaving the service version-pinned after this patch.
  - confidence: 0.94
- **[P2] Handle Windows Volta image paths:** `src/infra/stable-node-path.ts:19-25`
  - body: The Windows branch still expects a `.volta` directory and `bin/node.exe`, but upstream Volta defaults home to `%LOCALAPPDATA%\Volta`, places Node image executables at the version root on Windows, and exposes shims as `.cmd` scripts. Native Windows Volta installs therefore fall through unchanged.
  - confidence: 0.88
