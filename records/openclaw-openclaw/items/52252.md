## Summary

Keep open: the PR addresses a real resolver gap, but it is not merge-ready because the matcher only handles the default Unix `.volta` layout, misses custom `VOLTA_HOME` and Windows Volta layouts, and includes only test output rather than real behavior proof.

## What This Changes

The branch adds Volta tools/image path normalization in `resolveStableNodePath` and two unit tests for the default Unix `~/.volta` shim and missing-shim fallback.

## Best Possible Solution

Update the resolver to derive Volta home from any `<VOLTA_HOME>/tools/image/node/<version>` root, handle Unix and Windows image/shim layouts, preserve missing-shim fallback, and attach redacted real Volta gateway/install proof.

## Reproduction Assessment

Yes, source-level. Current main returns Volta tools/image Node paths unchanged, and the PR branch still leaves custom `VOLTA_HOME` and Windows Volta image layouts version-pinned despite accessible stable shims.

## Solution Assessment

No, not as submitted. Resolver-level normalization is the right boundary, but the matcher should parse Volta home generically and account for upstream Windows layout before merge.

## Review Findings

  - body: The matcher only recognizes paths containing `/.volta/tools/image/...`, but adjacent service PATH code already treats env-configured `VOLTA_HOME/bin` as supported. Native Windows Volta installs therefore fall through unchanged.
