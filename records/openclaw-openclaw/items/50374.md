## Summary

Keep open. Current main still forwards OpenAI Chat Completions message objects from the pinned converter without a provider/model-level key policy, so strict OpenAI-compatible backends can still receive tool replay fields such as `tool_calls`, `tool_call_id`, `reasoning_details`, or `name`. The KeepMy Claw workaround mentioned in the discussion is useful external routing context, but it does not fix direct OpenClaw provider calls.

## What This Changes

The issue requests sanitizing outbound `openai-completions` message objects for strict OpenAI-compatible providers that reject extra message keys.

## Best Possible Solution

Add an explicit provider/model-level outbound Chat Completions serialization policy that runs after conversion and before dispatch in both direct and embedded payload paths, preserves stored transcripts, and documents how it interacts with `supportsTools`, `requiresStringContent`, and tool-result replay.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a direct or embedded `openai-completions` request with prior assistant tool-call and tool-result history; current source plus the pinned converter show those outgoing messages can include the strict-provider-rejected keys described in the report.

## Solution Assessment

Yes, for the remaining work the narrow maintainable solution is an opt-in or compatibility-scoped serialization policy rather than automatically stripping every `openai-completions` payload, because standard-compatible providers may rely on tool replay fields.
