## Summary

Keep open. Current main still sends upstream OpenAI Chat Completions message objects without a strict per-provider message-key policy, so strict OpenAI-compatible providers can still reject replayed tool-call/result keys; the KeepMy Claw workaround in the discussion is useful external routing context but not an OpenClaw source fix.

## What This Changes

The issue requests sanitizing outgoing `openai-completions` message objects for strict OpenAI-compatible providers such as Infomaniak.

## Best Possible Solution

Add an explicit provider/model compat serialization policy that runs after message conversion and before dispatch in both direct and embedded paths, with docs and regression coverage for prior assistant tool calls plus tool-result history.

## Reproduction Assessment

Yes. A high-confidence source reproduction path is an `openai-completions` model with prior assistant tool-call and tool-result history; current main shows those converted messages can reach direct and embedded payloads without strict key sanitization.

## Solution Assessment

Partly. Sanitizing at the outbound payload boundary is the right fix shape, but automatically stripping every `openai-completions` payload would be too broad; an explicit provider/model compat policy is safer.
