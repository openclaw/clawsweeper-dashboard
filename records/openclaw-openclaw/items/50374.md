## Summary

Keep open. Current main still sends OpenAI-compatible Chat Completions messages from the upstream converter without a provider/model-level message-key policy, so strict OpenAI-compatible providers can still reject tool-call and reasoning replay keys; the KeepMy Claw workaround in the discussion is useful external routing context but not an in-repo fix.

## What This Changes

The issue requests sanitizing outbound `openai-completions` message objects so strict OpenAI-compatible providers do not receive unsupported extra keys.

## Best Possible Solution

Add an opt-in provider/model Chat Completions serialization policy that runs after message conversion and before dispatch in both direct and embedded paths, preserving stored transcripts and documenting the interaction with `supportsTools` and `requiresStringContent`.

## Reproduction Assessment

Yes. A high-confidence reproduction path is an `openai-completions` model with prior assistant tool-call and tool-result history; current source shows those converted messages can reach direct and embedded request payloads without a strict key sanitizer.

## Solution Assessment

Yes. The best maintainable fix is an explicit provider/model serialization policy rather than automatically stripping every `openai-completions` payload, because some compatible providers support standard tool replay fields.
