## Summary

Keep open: this PR targets a real Gateway history metadata gap, but it is not merge-ready because the live SSE path still depends on `update.message` already carrying a key that real `chat.send` updates do not provide, and the external PR has no real behavior proof.

## What This Changes

The PR copies persisted `idempotencyKey` values into `__openclaw` metadata during session-history hydration and inline session-history projection, with focused tests.

## Best Possible Solution

Carry the client idempotency key at the canonical `chat.send` transcript update/write point, then expose it as `__openclaw.idempotencyKey` in full history, recent history, and live session-history events with targeted protocol/runtime coverage.

## Reproduction Assessment

Yes, by source inspection. A transcript row with `message.idempotencyKey` reaches the current main hydration helper, which only attaches `id` and `seq`, and the PR’s live SSE path still relies on a real `chat.send` update shape that does not include the key.

## Solution Assessment

No. The history hydration part is the right direction, but the inline SSE fix should pass the client key into the canonical `chat.send` transcript update instead of reading a field that real updates do not populate.
