## Summary

Keep open: the PR targets a valid Gateway correlation gap, but the patch is incomplete because live `chat.send` transcript updates do not carry `idempotencyKey` into `update.message`, and the external PR still lacks real behavior proof.

## What This Changes

The PR copies persisted `idempotencyKey` values into `__openclaw` metadata for `chat.history` hydration and inline session-history events, and adds focused tests.

## Best Possible Solution

Carry the client idempotency key at the canonical `chat.send` transcript update/write point, then expose it as `__openclaw.idempotencyKey` in full history, recent history, and live session-history events with focused tests and real Gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a transcript row with `message.idempotencyKey` reaches current main hydration with only `id` and `seq`, and the PR’s live SSE path reads a field that real `chat.send` updates do not populate.

## Solution Assessment

No. The history hydration part is the right direction, but the inline SSE/session-message fix should pass the client key into the canonical `chat.send` transcript update instead of reading it from an update object that lacks it.
