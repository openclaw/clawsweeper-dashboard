## Summary

Keep open: the persisted-history hydration part is aligned with the linked report, but the live `chat.send` SSE path still appears to build an inline update message without `idempotencyKey`, and the external PR only lists tests/type/lint proof rather than real behavior proof.

## What This Changes

The PR copies persisted `idempotencyKey` values into `__openclaw` metadata during gateway session-history hydration and inline session-history event projection, with focused tests.

## Best Possible Solution

Carry `chat.send`'s client idempotency key at the canonical user transcript update/write point, then hydrate it into `__openclaw.idempotencyKey` for full history, recent history, and live session-history events with protocol-level coverage.

## Reproduction Assessment

Yes, by source inspection. A transcript row with `message.idempotencyKey` currently reaches `parsedSessionEntryToMessage`, which attaches only `id` and `seq`, and the PR's live SSE path still depends on `update.message` carrying a key that the current chat.send inline update builder does not include.

## Solution Assessment

No. Hoisting from persisted messages is the right narrow direction for history reads, but the live inline path should source the key from the canonical chat.send transcript update instead of assuming `update.message` already contains it.
