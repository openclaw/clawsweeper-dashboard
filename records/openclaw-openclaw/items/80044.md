## Summary

Keep open: the history hydration direction is useful, but the live session-history SSE path still misses the real `chat.send` key and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR hoists persisted `idempotencyKey` values into `__openclaw` during Gateway history hydration, inline session-history SSE projection, broadcast projection, and focused tests.

## Best Possible Solution

Carry the `chat.send` client key at the canonical user transcript update/write point, then hydrate it into `__openclaw.idempotencyKey` for full history, recent history, broadcast messages, and live session-history SSE with protocol-level coverage.

## Reproduction Assessment

Yes, by source inspection. A transcript row with `message.idempotencyKey` currently reaches `parsedSessionEntryToMessage`, which attaches only `id` and `seq`, and the PR's SSE path still depends on `update.message` carrying a key that real `chat.send` does not include.

## Solution Assessment

No. Hoisting persisted messages during history hydration is the right narrow direction, but the live SSE fix should source the key from the canonical `chat.send` transcript update instead of only copying it when the update message already has it.
