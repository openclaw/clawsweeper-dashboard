## Summary

Keep this open: current main still has no OpenAI Codex active-workspace validation or recovery path, and the related open PR only covers a narrow failover-classification mitigation. The report is now corroborated by multiple users and remains distinct from older accountId extraction and token-refresh fixes.

## What This Changes

Ensure OpenAI Codex OAuth does not persist or reuse a deactivated ChatGPT workspace when an account also has an active workspace.

## Best Possible Solution

Implement an OpenAI-plugin-owned active-workspace path: validate and persist a usable accountId after OAuth when possible, detect deactivated_workspace at runtime, retry or re-discover once, persist the recovered selection, and surface a clear re-login/doctor path when recovery is impossible.

## Reproduction Assessment

Yes. The issue gives a concrete account-state reproduction path with one active and one deactivated ChatGPT workspace, and two independent commenters report the same failure; this review did not live-run it because that account state is not available in the checkout.

## Solution Assessment

Unclear. The requested behavior is right, but the best implementation depends on whether OpenClaw should call a ChatGPT account metadata API directly or rely on @mariozechner/pi-ai to select an active workspace; #55893 is only a partial matcher fix.
