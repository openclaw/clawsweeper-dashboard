## Summary

Keep open: current main and v2026.6.10 classify `deactivated_workspace` as an auth/failover signal, but they still do not validate, select, or recover an active ChatGPT workspace for multi-workspace OpenAI OAuth users.

## What This Changes

The issue reports that OpenAI Codex OAuth can persist or reuse a deactivated ChatGPT workspace when the account also has an active workspace.

## Best Possible Solution

Define active ChatGPT workspace selection and recovery policy at the OpenAI/Codex auth boundary, then implement it with focused tests plus live multi-workspace proof.

## Reproduction Assessment

Yes for source reproduction: current main extracts and stores one token-derived account id, reuses it in app-server and discovery paths, and drops the upstream Codex `previous_account_id` refresh hint. A live multi-workspace account has not been exercised in this read-only review.

## Solution Assessment

Unclear as an implementation: upstream provides account-session/workspace metadata and refresh hints, but maintainers still need to choose whether OpenClaw should auto-select, retry, persist, or only guide remediation.
