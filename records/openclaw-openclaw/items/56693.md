## Summary

Keep this open: current main still does not validate or recover from OpenAI Codex OAuth selecting a deactivated ChatGPT workspace, and the open related PR only handles one failover-classification slice. The report is corroborated by a second user on v2026.4.8 and remains distinct from older accountId extraction and persistence fixes.

## What This Changes

The issue asks OpenClaw to avoid selecting or continuing to use a deactivated ChatGPT workspace for OpenAI Codex OAuth accounts that have multiple workspaces.

## Best Possible Solution

Keep the issue open and implement an OpenAI-plugin-owned active workspace path: after browser or device-code OAuth, persist a verified usable accountId when possible, and at runtime detect deactivated_workspace, refresh or re-discover a usable active workspace once, persist that selection, and surface a clear doctor/login remediation when recovery is impossible.

## Reproduction Assessment

Yes. The issue gives a specific reproduction path using one active and one deactivated ChatGPT workspace, and a second user independently reported the same deactivated_workspace payload on v2026.4.8; this review did not live-run it because that account state is not available in the checkout.

## Solution Assessment

Unclear. The requested behavior is correct, but the best implementation depends on the account/workspace API boundary; related #55893 is a useful partial mitigation, not a complete fix for workspace selection or recovery.
