## Summary

Keep open: current main includes the merged `deactivated_workspace` failover-classifier mitigation, but the reported active-workspace selection and recovery gap still remains in the OpenAI Codex OAuth path.

## What This Changes

The issue requests that OpenAI Codex OAuth avoid or recover from binding to a deactivated ChatGPT workspace when the same account has an active workspace.

## Best Possible Solution

Add an OpenAI-plugin-owned or upstream-backed active workspace path that verifies a usable accountId after OAuth, recovers once on `deactivated_workspace`, persists the recovered accountId, and gives clear re-login or doctor remediation when no active workspace can be selected.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: use an OpenAI account with one active and one deactivated ChatGPT workspace, complete Codex OAuth, then hit runtime paths that reuse the token-derived accountId. I did not live-run it because that external account state is unavailable here.

## Solution Assessment

Unclear. The active-workspace recovery end state is right, but maintainers need to decide whether OpenClaw’s OpenAI plugin or the upstream OAuth helper owns workspace enumeration and selection.
