## Summary

Keep open: current main and the latest release classify `deactivated_workspace` as an auth/failover signal, but they still do not validate, select, or recover an active ChatGPT workspace for multi-workspace OpenAI OAuth users.

## What This Changes

The issue asks OpenClaw to avoid binding OpenAI ChatGPT OAuth to deactivated workspaces and to recover or guide users when an active workspace exists.

## Best Possible Solution

Define the active-workspace selection and recovery policy at the OpenAI/Codex auth boundary, then implement it with focused source tests and live multi-workspace proof.

## Reproduction Assessment

Yes, source-reproducible: current login/refresh extracts one `chatgpt_account_id` from the token and downstream runtime paths reuse it, with no active-workspace enumeration or deactivated-workspace recovery path. A live multi-workspace account was not exercised in this read-only review.

## Solution Assessment

Unclear as an implementation: the right layer is the OpenAI/Codex auth and app-server bridge boundary, but maintainers need to choose the product policy for active workspace selection, retry, and persistence.
