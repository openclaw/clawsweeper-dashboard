## Label Justifications

- `P2`: This is a real OpenAI Codex OAuth/provider failure for multi-workspace users, but the blast radius is limited to that provider/account state.
- `impact:auth-provider`: The issue is about OAuth workspace/account selection, persisted accountId metadata, and provider auth recovery.

## Summary

Keep open: current main and the latest release still persist OpenAI Codex account identity passively from OAuth tokens and do not validate, select, or recover an active ChatGPT workspace; the linked PR is only a failover-classifier mitigation.

## What This Changes

The issue requests OpenAI Codex OAuth/runtime handling that avoids binding to a deactivated ChatGPT workspace when an active workspace is available.

## Best Possible Solution

Add an OpenAI-plugin-owned or upstream-backed active workspace path that verifies a usable accountId after OAuth, recovers once on `deactivated_workspace`, persists the recovered accountId, and gives clear re-login or doctor remediation when recovery is impossible.

## Reproduction Assessment

Yes, source inspection plus user reports give a high-confidence reproduction path: use an OpenAI account with one active and one deactivated ChatGPT workspace, complete Codex OAuth, then hit runtime paths that reuse the token-derived accountId. I did not live-run it because that external account state is not available in the checkout.

## Solution Assessment

Unclear. The requested active-workspace recovery is the right end state, but the durable implementation needs a boundary decision on whether OpenClaw’s OpenAI plugin or the upstream OAuth helper owns workspace enumeration and selection.
