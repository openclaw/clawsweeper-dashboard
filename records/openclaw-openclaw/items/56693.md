## Summary

Keep open: current main and the latest release only classify `deactivated_workspace` as a permanent auth/failover signal; they still do not validate, select, or recover an active ChatGPT workspace for multi-workspace OpenAI OAuth users.

## What This Changes

The issue requests OpenAI ChatGPT OAuth workspace selection or recovery so OpenClaw does not keep using a deactivated workspace when an active workspace is available.

## Best Possible Solution

Define the OpenAI ChatGPT workspace selection and recovery policy, then implement it at the OpenAI provider auth boundary with tests for login, refresh, and runtime `deactivated_workspace` failures.

## Reproduction Assessment

Yes at source level, with user logs: OAuth stores the token-derived accountId and runtime sends it, while no current OpenClaw code validates active workspace or discovers a replacement. I did not run a live multi-workspace OpenAI account.

## Solution Assessment

No complete solution is implemented yet; the merged classifier mitigation is useful but too narrow because it explicitly excludes provider login, auth storage, active workspace selection, and network recovery.
