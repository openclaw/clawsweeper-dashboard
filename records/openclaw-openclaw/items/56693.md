## Summary

Keep open: current main still stores OpenAI Codex workspace/account identity from the OAuth token and has no active/deactivated workspace validation or recovery path; the open related PR only addresses failover classification for the error marker.

## What This Changes

The issue requests OpenAI Codex OAuth and runtime handling that avoids binding to a deactivated ChatGPT workspace when an active workspace is available.

## Best Possible Solution

Implement an OpenAI-plugin-owned active workspace path: verify and persist a usable accountId after browser or device-code OAuth when possible, detect `deactivated_workspace` at runtime, refresh or rediscover one usable active workspace once, persist the recovered accountId, and surface a clear re-login/doctor remediation when recovery is impossible.

## Reproduction Assessment

Yes. The issue gives a concrete active-plus-deactivated ChatGPT workspace reproduction path, two users corroborated the same payload, and current source shows no active-workspace validation; I did not live-run it because that account state is not available in the checkout.

## Solution Assessment

Unclear. The requested behavior is correct, but the durable implementation depends on the provider/upstream boundary for discovering active ChatGPT workspaces; the related PR is only a narrow classifier mitigation, not the full fix.
