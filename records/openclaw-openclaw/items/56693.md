## Summary

Keep open: current main and the latest release only classify `deactivated_workspace` for failover, while the reported active-workspace selection and persisted accountId recovery gap remains in the OpenAI Codex OAuth path.

## What This Changes

The issue requests OpenAI Codex OAuth/runtime to avoid or recover from selecting a deactivated ChatGPT workspace when an account also has an active workspace.

## Best Possible Solution

Add an OpenAI-plugin-owned or upstream-backed active workspace validation/recovery path that persists a usable accountId and gives clear remediation when no active workspace can be selected.

## Reproduction Assessment

Yes, at source level: use an OpenAI account with one active and one deactivated ChatGPT workspace, complete Codex OAuth, then exercise a runtime path that reuses the token-derived accountId. I did not live-run it because that external account state is unavailable here.

## Solution Assessment

Unclear. The active-workspace recovery end state is right, but maintainers need to decide whether OpenClaw's OpenAI plugin or the upstream OAuth helper owns workspace enumeration and selection.
