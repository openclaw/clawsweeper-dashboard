## Summary

Keep open: current main and the latest release classify `deactivated_workspace` for failover, but they still do not select, validate, or recover an active ChatGPT workspace during OpenAI OAuth, refresh, or app-server auth bridging.

## What This Changes

Request to prevent OpenAI/ChatGPT OAuth profiles from binding to a deactivated workspace when the same account also has an active workspace.

## Best Possible Solution

Define the provider-owned workspace selection policy, then implement OpenAI OAuth/refresh/runtime recovery with regression tests and live multi-workspace proof.

## Reproduction Assessment

Yes, at source level with supporting user logs: OpenClaw stores a token-derived accountId and sends it on later requests, while no current path validates or recovers an active workspace. I did not run a live multi-workspace OpenAI account.

## Solution Assessment

No complete solution is implemented yet. The merged classifier mitigation is useful, but the best fix needs a maintainer-approved workspace selection and recovery policy in the OpenAI provider/auth boundary.
