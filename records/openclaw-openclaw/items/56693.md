## Summary

Keep open: the merged failover-classifier mitigation on current main recognizes `deactivated_workspace`, but source and dependency inspection still show no active workspace validation, alternate ChatGPT account selection, or persisted recovery when Codex OAuth binds to a deactivated workspace.

## What This Changes

The issue requests that OpenAI Codex OAuth avoid or recover from binding stored credentials to a deactivated ChatGPT workspace when another active workspace exists.

## Best Possible Solution

Add an OpenAI-plugin-owned or upstream-backed active workspace path that verifies a usable accountId after OAuth, recovers once on `deactivated_workspace`, persists the recovered accountId, and gives clear re-login or doctor remediation when no active workspace can be selected.

## Reproduction Assessment

Yes. Source inspection plus multiple user reports give a high-confidence reproduction path: use an OpenAI account with one active and one deactivated ChatGPT workspace, complete Codex OAuth, then hit runtime paths that reuse the token-derived accountId; I did not live-run it because that external account state is unavailable here.

## Solution Assessment

Unclear. The desired active-workspace recovery is the right end state, but maintainers need to decide whether OpenClaw’s OpenAI plugin or the upstream OAuth helper owns workspace enumeration and selection.
