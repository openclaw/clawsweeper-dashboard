## Summary

Keep open: current main and the latest release only classify `deactivated_workspace` as an auth-permanent failover signal; they still do not validate, select, or recover an active ChatGPT workspace during OpenAI OAuth or token refresh.

## What This Changes

Request OpenAI ChatGPT/Codex OAuth to avoid or recover from deactivated workspace bindings for multi-workspace accounts.

## Best Possible Solution

Define the product-owned workspace selection policy, then implement it in the OpenAI provider/OAuth boundary with regression coverage for login, refresh, and runtime deactivated-workspace failures.

## Reproduction Assessment

Yes at source level, with user logs: OAuth stores the token-derived accountId and runtime sends it, while no current OpenClaw code validates active workspace or discovers a replacement; I did not run a live multi-workspace OpenAI account.

## Solution Assessment

No complete solution is implemented yet; PR #55893 is a useful classifier mitigation, but its own scope excludes provider login, auth storage, active workspace selection, and network recovery.
