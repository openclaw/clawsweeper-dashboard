## Label Justifications

- `P2`: This is a real OpenAI Codex OAuth/provider failure for multi-workspace users, with blast radius limited to that provider/account state.
- `impact:auth-provider`: The issue is about OAuth workspace/account selection, persisted accountId metadata, and provider auth recovery.

## Summary

Keep open: current main and v2026.5.19 still rely on token-derived Codex account IDs and do not validate, select, or recover an active ChatGPT workspace; the linked failover PR is a useful mitigation but does not solve workspace binding.

## What This Changes

Prevent OpenAI Codex OAuth from persisting or reusing a deactivated ChatGPT workspace when an active workspace is available.

## Best Possible Solution

Add an OpenAI-plugin-owned or upstream-backed active workspace path that verifies a usable accountId after OAuth, recovers once on `deactivated_workspace`, persists the recovered accountId, and gives clear re-login or doctor remediation when recovery is impossible.

## Reproduction Assessment

Yes. Source inspection plus multiple user reports give a high-confidence reproduction path: use an OpenAI account with one active and one deactivated ChatGPT workspace, complete Codex OAuth, then hit runtime paths that reuse the token-derived accountId; I did not live-run it because that external account state is unavailable here.

## Solution Assessment

Unclear. The desired active-workspace recovery is the right end state, but maintainers need to decide whether OpenClaw’s OpenAI plugin or the upstream OAuth helper owns workspace enumeration and selection.
