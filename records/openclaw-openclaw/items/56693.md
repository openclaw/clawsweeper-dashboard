## Summary

Keep open: current main still passively persists token-derived ChatGPT account identity for OpenAI Codex OAuth and has no active/deactivated workspace validation or recovery path; the open related PR only classifies the error marker for failover.

## What This Changes

Ensure OpenAI Codex OAuth selects, persists, or recovers an active ChatGPT workspace instead of continuing with a deactivated workspace for multi-workspace accounts.

## Best Possible Solution

Add an OpenAI-plugin-owned or upstream-backed active workspace path that verifies a usable accountId after OAuth, recovers once on `deactivated_workspace`, persists the recovered accountId, and gives a clear re-login or doctor remediation when recovery is impossible.

## Reproduction Assessment

Yes. The issue gives concrete active-plus-deactivated workspace steps, two later users corroborate the same payload, and current source shows no active-workspace validation; I did not live-run it because that account state is not available in the checkout.

## Solution Assessment

Unclear. The requested active-workspace recovery is the right end state, but the durable implementation depends on whether OpenClaw’s OpenAI plugin or the upstream OAuth helper should own workspace enumeration and selection.
