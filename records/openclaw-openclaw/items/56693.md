## Label Justifications

- `P2`: The issue affects OpenAI OAuth users with multiple workspaces and can make that provider unusable, but the blast radius is limited to a specific account/workspace state.
- `impact:auth-provider`: The issue is about OAuth workspace/account binding, provider credential metadata, and OpenAI Codex request routing.

## Summary

Keep open: the merged failover mitigation only classifies `deactivated_workspace` as an auth-permanent failure, while current main and the latest release still extract and persist the ChatGPT workspace/account id from token claims and then send that id without active-workspace validation or recovery. Choosing automatic workspace selection or retry behavior crosses provider/auth policy, so this still needs maintainer product direction.

## What This Changes

The issue requests stronger OpenAI Codex OAuth workspace selection or recovery so multi-workspace ChatGPT accounts do not keep using a deactivated workspace when an active one exists.

## Best Possible Solution

Keep the classifier mitigation from the merged PR and add a maintainer-approved OpenAI plugin/Codex auth policy for validating or recovering an active workspace, with focused tests for login, refresh, and runtime request behavior.

## Reproduction Assessment

Yes, source-level: current main can be traced from OAuth token account-id extraction through credential persistence to runtime `chatgpt-account-id` headers, with no active-workspace validation. A live reproduction still requires a ChatGPT account that has both an active and deactivated workspace.

## Solution Assessment

Unclear as an implementation choice: the merged classifier is a good mitigation, but it is not the full fix for active workspace selection or recovery. The narrow maintainable solution needs maintainer-approved policy before changing login, refresh, or retry behavior.
