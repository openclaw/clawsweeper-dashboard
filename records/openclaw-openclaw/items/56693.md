## Summary

Keep open: current main and the latest release classify `deactivated_workspace` for failover, but they still persist and reuse a token-derived OpenAI Codex accountId without active workspace validation or recovery.

## What This Changes

The issue asks OpenClaw to avoid binding OpenAI Codex OAuth credentials to a deactivated ChatGPT workspace when the same account also has an active workspace.

## Best Possible Solution

Add an owner-approved active workspace path that verifies a usable accountId after OAuth, recovers once on `deactivated_workspace`, persists the recovered accountId, and gives clear re-login or doctor remediation when no active workspace can be selected.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: use an OpenAI account with one active and one deactivated ChatGPT workspace, complete Codex OAuth, then hit runtime paths that reuse the token-derived accountId; I did not live-run it because that external account state is unavailable here.

## Solution Assessment

Unclear. The active-workspace recovery end state is right, but maintainers need to decide whether OpenClaw’s OpenAI plugin or the upstream OAuth helper owns workspace enumeration and selection.
