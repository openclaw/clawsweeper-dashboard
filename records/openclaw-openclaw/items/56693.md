## Summary

Keep open: current main and v2026.6.8 only classify `deactivated_workspace` as a permanent auth/failover signal; they still do not validate, select, or recover an active ChatGPT workspace for multi-workspace OpenAI OAuth users.

## What This Changes

The issue reports that OpenAI Codex OAuth can bind to a deactivated ChatGPT workspace when the same ChatGPT account also has an active workspace.

## Best Possible Solution

Define the product policy for active ChatGPT workspace selection/recovery, then implement it at the OpenAI/Codex auth and app-server bridge boundary with focused source tests plus live multi-workspace proof.

## Reproduction Assessment

Yes, source-reproducible: current login/refresh extracts one `accountId` from the token and downstream runtime paths reuse it, with no active-workspace enumeration or retry path. A live multi-workspace account was not used in this read-only review.

## Solution Assessment

No, the merged classifier mitigation is not the full solution because it only turns `deactivated_workspace` into an auth/failover signal. The maintainable fix needs an explicit active-workspace selection or recovery policy at the auth/bridge boundary.
