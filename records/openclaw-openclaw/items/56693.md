## Summary

Keep open: current main and v2026.6.6 still derive and reuse the ChatGPT account/workspace id from OAuth tokens without validating or recovering an active workspace; the merged deactivated-workspace PR only improves failover classification.

## What This Changes

The issue requests that OpenAI ChatGPT/Codex OAuth avoid binding to a deactivated ChatGPT workspace when the account also has an active workspace.

## Best Possible Solution

Define a provider-owned active-workspace selection/recovery policy and implement it in OpenAI OAuth, refresh, and runtime paths with focused tests plus live multi-workspace proof.

## Reproduction Assessment

Yes at source level: OAuth stores the token-derived chatgpt_account_id and runtime sends it as ChatGPT-Account-ID, while user comments report deactivated_workspace failures. I did not run a live multi-workspace OpenAI account.

## Solution Assessment

No complete solution is implemented yet. The merged classifier mitigation is useful, but it explicitly leaves provider login, auth storage, active workspace selection, and network recovery unchanged.
