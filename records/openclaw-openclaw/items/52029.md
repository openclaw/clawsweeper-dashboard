## Summary

Keep open: current main still has no heartbeat-scoped tool toggle, and the existing no-tools behavior only applies to due-commitment delivery, not ordinary heartbeat turns. The request is valid, but it is a new security-sensitive config surface rather than a broken documented behavior.

## What This Changes

Add a heartbeat config option that lets operators disable tools for heartbeat turns while preserving the current tool-capable default.

## Best Possible Solution

Add documented `heartbeat.tools: false` support for defaults and per-agent heartbeat config, map it to the existing no-tools runtime path, and keep today's tool-capable heartbeat behavior as the default.

## Reproduction Assessment

Yes for the config gap: current main has no strict-schema heartbeat tool key, and ordinary heartbeat reply options do not disable tools. I did not live-reproduce the specific local-model tool loop.

## Solution Assessment

Yes as a first implementation shape: wiring a boolean heartbeat config into the existing `disableTools`/empty `skillFilter` seam is narrow and maintainable. Maintainers should still decide whether the initial public surface is a boolean only or should reserve room for a later allowlist.
