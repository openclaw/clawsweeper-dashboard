- `P2`: This is a normal-priority feature/config improvement with concrete safety and availability impact for local-model heartbeat deployments, but it is not a confirmed core regression.
- `impact:security`: The issue is about preventing heartbeat models from invoking available tools such as exec when the operator wants a no-tool background turn.
- `impact:crash-loop`: The report includes heartbeat tool loops and gateway hangs that can block agent turns and affect process availability.

## Summary

Keep open: current main still lacks a heartbeat-scoped config option to disable tools, while ordinary heartbeat turns only pass `disableTools` for due-commitment delivery. The request is a narrow feature/config addition with clear runtime, schema, docs, and test surfaces.

## What This Changes

Request a `heartbeat.tools` option that omits tool definitions during heartbeat runs for small or local models.

## Best Possible Solution

Add documented `heartbeat.tools: false` support for default and per-agent heartbeat config, map it to the existing no-tools runner path, and keep current tool-capable heartbeat behavior as the default.

## Reproduction Assessment

Yes for the config gap: current-main source shows a strict heartbeat schema with no `tools` key, and ordinary heartbeat reply options do not set `disableTools` except for due commitments. I did not live-reproduce the reported local-model tool loop.

## Solution Assessment

Yes. Plumbing a boolean heartbeat config into the existing `disableTools`/empty-skill-filter seam is the narrowest maintainable solution; a richer allowlist can be a later extension if maintainers want it.
