## Summary

Keep open. Current main still lacks the requested heartbeat-scoped tool-disable config: the heartbeat schema, public config type, docs, and config-honor inventory enumerate existing heartbeat knobs without a tool policy, and the runner only passes `disableTools` for due inferred-commitment heartbeat turns, not ordinary heartbeat runs.

## What This Changes

Request to add a heartbeat-scoped option such as `agents.defaults.heartbeat.tools: false` so heartbeat turns can omit all tool definitions for small or local models.

## Best Possible Solution

Add a heartbeat-scoped tool policy to the existing heartbeat config surface, conservatively starting with `tools: false` that maps to the existing run-level `disableTools` seam for heartbeat turns only. Keep normal user turns and existing global/per-agent tool policies unchanged, and document/tests the config so local-model heartbeat runs can be made text-only without relying on prompt obedience.

## Reproduction Assessment

Yes. A static current-main check is sufficient: `HeartbeatSchema` is strict and lacks the requested field, while `runHeartbeatOnce` has no general config branch that sets `disableTools` for ordinary heartbeats.

## Solution Assessment

Yes, plumbing the existing `disableTools` run option from heartbeat config is the narrowest maintainable solution. A future allowlist can be added later, but a boolean `tools: false` directly addresses the safety and latency problem without redesigning tool policy.
