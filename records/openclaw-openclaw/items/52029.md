## Label Justifications

- `impact:security`: The report is about execution-capable tools being exposed to heartbeat models that may ignore prompt-only no-tool instructions.
- `impact:crash-loop`: The report includes tool-call loops and gateway hangs during heartbeat turns, which are process availability failure modes.

## Summary

Keep open: current main and v2026.5.19 still lack a heartbeat-scoped tool toggle, and normal heartbeat turns only disable tools for due-commitment delivery. The existing `disableTools` runner seam makes the request feasible, but the new config/security semantics need maintainer review before implementation.

## What This Changes

Request a heartbeat-scoped config option to omit tool definitions during heartbeat turns, mainly to make weak local heartbeat models safer and cheaper.

## Best Possible Solution

Add a documented opt-in heartbeat tool policy for defaults and per-agent heartbeat config, map the no-tools mode to `disableTools: true` with an empty skill filter, preserve tool-capable heartbeat behavior by default, and cover schema, docs, and runner tests.

## Reproduction Assessment

Yes for the config gap: current-main source shows a strict heartbeat schema with no tool-policy key, and ordinary heartbeat reply options do not set `disableTools` except for due commitments. I did not live-reproduce the reported local-model tool loop.

## Solution Assessment

Yes on the general direction, but maintainer semantics are still needed. Plumbing an opt-in heartbeat no-tools setting into the existing `disableTools` seam is the narrow implementation path; an allowlist can be a later extension if desired.
