## Label Justifications

- `P2`: This is a focused heartbeat safety and local-model reliability enhancement with limited blast radius but real user impact.
- `impact:security`: The report is about execution-capable tools being exposed to heartbeat models that may ignore prompt-only no-tool instructions.
- `impact:crash-loop`: The report includes tool-call loops and gateway hangs during heartbeat turns, which are process availability failure modes.

## Summary

Keep open: current main and v2026.5.20 still lack a heartbeat-scoped tool toggle, and ordinary heartbeat runs only disable tools for due-commitment delivery. A dedicated no-tools model entry using `compat.supportsTools: false` is a useful workaround, but it is model-scoped rather than the requested heartbeat policy.

## What This Changes

Request adds a heartbeat config option to omit tool definitions during heartbeat turns so local/small heartbeat models cannot call tools.

## Best Possible Solution

Add a documented opt-in heartbeat tool policy for defaults and per-agent heartbeat config, preserve current tool-capable behavior by default, and wire the no-tools mode through the existing `disableTools` path with schema, docs, and heartbeat runner coverage.

## Reproduction Assessment

Yes for the config gap: current-main source shows a strict heartbeat schema with no heartbeat-scoped tool-policy key, and ordinary heartbeat reply options do not set `disableTools` except for due commitments. I did not live-run the reported Ollama/local-model tool loop.

## Solution Assessment

Unclear until maintainers choose the public semantics. The narrow implementation path is to plumb an opt-in heartbeat setting into the existing `disableTools` seam, while documenting how that differs from the current model-level `compat.supportsTools: false` workaround.
