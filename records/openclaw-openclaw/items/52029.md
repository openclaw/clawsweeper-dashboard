## Summary

Keeping this open: current main and the latest release still lack a heartbeat-specific tool policy, while the existing no-tools runtime seam is not exposed through heartbeat config and has backend-specific caveats.

## What This Changes

Request a heartbeat configuration option to omit tool definitions from heartbeat model requests when small local models ignore prompt-only no-tool instructions.

## Best Possible Solution

Define a heartbeat tool policy that preserves current defaults, explicitly handles backends that cannot disable native tools, and adds schema/docs/tests for both tool-enabled and configured no-tool heartbeat runs.

## Reproduction Assessment

Yes for the source-level feature gap: current main and v2026.5.22 do not expose a heartbeat tools policy, while heartbeat runs still route through the normal reply/tool path. No for the full live failure: I did not reproduce the reported Windows/Ollama dangerous exec loop.

## Solution Assessment

Unclear: the existing disableTools seam makes a narrow implementation plausible, but maintainers should first choose boolean versus allowlist semantics and how unsupported native-tool backends should fail or warn.
