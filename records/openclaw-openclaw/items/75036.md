## Summary

Keep open. Current main still lacks terminal `lost` cron-row recovery, and this PR is a focused implementation candidate; the diff looks correct, the current-head real behavior proof is sufficient, and I found no blocking code or security findings.

## What This Changes

The PR lets task maintenance recover terminal `lost` cron rows from exact durable cron run evidence when the lost state looks like a backing-session-missing false positive, clears stale errors on recovered success, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Land this recovery or an equivalent patch after maintainer merge handling resolves the false-negative proof gate, keeping recovery limited to exact durable cron evidence for backing-session false positives.

## Reproduction Assessment

Yes, source-level. Current main only calls durable cron recovery for active cron rows, so a terminal `lost` cron task with a matching `cron:<jobId>:<startedAt>` run id and durable finished run evidence is skipped and remains an audit finding.

## Solution Assessment

Yes. Reusing the existing exact durable run-log/job-state recovery path and narrowing terminal eligibility to backing-session-missing lost rows is more maintainable than broad pruning or job-id-only inference.
