## Summary

Keep open. The PR updates task-registry maintenance to recover selected terminal `lost` cron rows from durable cron run history, clears stale terminal errors when recovery has no error, and adds regression coverage.

## Best Possible Solution

Yes. A high-confidence reproduction is a `runtime: "cron"`, `status: "lost"`, `error: "backing session missing"` task whose `runId` matches a durable `finished`/`ok` cron run log; current main skips durable recovery because the row is no longer active.

## Solution Assessment

Yes, with the changelog and validation caveat. Reusing existing durable cron run history and limiting terminal recovery to empty or backing-session-missing lost rows is narrower than pruning or rewriting all lost rows.
