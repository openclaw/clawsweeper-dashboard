## Summary

Keep open: the reported regression still matches current `main`. `/stop` is documented as aborting the current run, and the current fast-abort path can still fall through to normal active-run queuing when text `/stop` does not pass the same authorization branch as a native command turn; I did not find a merged exact fix or a canonical duplicate that owns this narrow regression.

## What This Changes

The issue reports that Telegram text `/stop` commands are queued behind an active run instead of immediately interrupting it.

## Best Possible Solution

Add a focused regression for authorized text `/stop` during an active run, then make fast abort authorize permitted text stop requests without requiring a native command turn.

## Reproduction Assessment

Yes, source-reproducible with high confidence: an active run plus a text `/stop` that misses the fast-abort authorization branch can fall through into the follow-up queue. I did not execute a live Telegram run in this read-only review.

## Solution Assessment

Yes, a narrow fast-abort authorization fix with regression coverage is the best path. The broader interruptibility tracker should not be used as a substitute for restoring the existing `/stop` emergency brake.
