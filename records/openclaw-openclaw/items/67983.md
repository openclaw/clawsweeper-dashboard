## Summary

Keep this PR open, but do not merge it as-is. The feature addresses a real plugin observability gap, but the current diff passes live mutable tool schema objects into an observation hook and leaves the new public hook contract under-documented.

## What This Changes

The PR adds optional `tools` and `clientTools` descriptors to `PluginHookLlmInputEvent`, populates them from the embedded runner's effective tool lists, updates the wired hook test fixture, and updates the plugin SDK API baseline hash.

## Best Possible Solution

Revise this into a narrow, documented plugin SDK contract change that emits copied and normalized tool descriptors after effective tool selection, keeps hook telemetry from mutating request state, and adds focused coverage for both built-in tools and hosted client tools.

## Reproduction Assessment

Yes for the blocking review issue: register an `llm_input` handler that mutates `event.tools[0].parameters` and verify the model/session tool definition does not change. Static inspection already shows the PR passes the same parameter object references into the hook payload.

## Solution Assessment

No. Exposing tool descriptors on `llm_input` is a reasonable direction, but the current implementation should use a shared descriptor mapper that clones or freezes parameters, normalizes descriptions to match the request path, and documents the new public fields.
