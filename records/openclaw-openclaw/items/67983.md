## Summary

Keep open: current main still lacks the requested hook fields, and the PR branch still has a concrete hook-boundary integrity defect plus missing real behavior proof. The branch is useful, but it needs branch changes and maintainer API review before merge.

## What This Changes

This PR adds optional `tools` and `clientTools` descriptors to `PluginHookLlmInputEvent`, emits them from the embedded runner, updates the hook runner test fixture, and refreshes the plugin SDK API hash.

## Best Possible Solution

Ship the hook API only after mapping tool descriptors through an isolated, normalized helper, documenting `tools` and `clientTools`, covering populated client tools and mutation isolation, and posting real behavior proof from a subscribing plugin.

## Reproduction Assessment

Yes for the blocking patch defect: source inspection shows PR head passes `tool.parameters` and `tool.function.parameters` by reference into `runLlmInput`, and a focused test can mutate event parameters and assert the request definitions remain unchanged. This is source-reproducible without a live provider run.

## Solution Assessment

No. Exposing tool descriptors on `llm_input` is a reasonable API shape, but this implementation should isolate parameters, normalize descriptions to match request adapters, document the API, and cover populated `clientTools` before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Clone tool parameters before emitting llm_input:** `src/agents/pi-embedded-runner/run/attempt.ts:2048-2053`
  - body: The new event maps `parameters` directly from `effectiveTools` and `clientTools`. `runVoidHook` invokes handlers with the shared event object before prompt submission continues, while request tool definitions retain those parameter object references, so a tracing or redaction hook can mutate the schema the model receives in this turn.
  - confidence: 0.92
- **[P3] Normalize descriptions in the hook payload:** `src/agents/pi-embedded-runner/run/attempt.ts:2047-2052`
  - body: The request adapters normalize missing built-in and client tool descriptions to `""`, but this event emits raw optional descriptions. Plugins that compare or hash the hook payload against the real request can see false drift whenever a tool omits its description.
  - confidence: 0.84
- **[P3] Document the new llm_input fields:** `src/plugins/hook-types.ts:176-177`
  - body: This adds public plugin hook contract fields, but `docs/plugins/hooks.md` still describes `llm_input` as prompt and history only. Plugin authors need the new `tools` and `clientTools` payload fields documented before relying on the SDK contract.
  - confidence: 0.82
