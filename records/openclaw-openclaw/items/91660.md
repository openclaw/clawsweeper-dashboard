## Summary

Keep open: current main still returns an adapter-supplied empty `provider.model` unchanged from `createWithAdapter()`, and this PR is the focused candidate fix for that provider-identity gap. The patch looks correct, but it intentionally changes the model identity used by persisted memory metadata, cache keys, chunk tags, and vector search filters, so it still needs maintainer review and exact-head validation before merge.

## What This Changes

The PR backfills an empty memory embedding `provider.model` with the resolved adapter model in `createWithAdapter()` and adds regression tests for empty and already-populated provider models.

## Best Possible Solution

Land the focused provider-creation backfill after maintainer review and exact-head validation, while keeping the broader memory provider-upgrade and reindex issues open for separate decisions.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to register a memory embedding adapter with a `defaultModel` whose `create()` returns `provider.model = ""`; current main passes the resolved model into the adapter but returns that empty provider model unchanged.

## Solution Assessment

Yes for the narrow bug. Normalizing `provider.model` in `createWithAdapter()` is the clean owner-boundary fix compared with adding fallbacks at each downstream identity, write, cache, and search consumer.
