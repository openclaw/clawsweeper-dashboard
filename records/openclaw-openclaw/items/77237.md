## Summary

Keep open: the underlying managed npm rollback regression is real and the snapshot direction is useful, but the latest head still has blocking install-flow defects, a security-scan regression, failing checks, and real behavior proof tied to an older head.

## What This Changes

The branch snapshots and restores the managed npm plugin root on failed npm-backed installs, adds rollback regression coverage, and includes small provider/session test-stabilization changes.

## Best Possible Solution

Keep a single managed npm install flow that snapshots the root before any npm mutation, restores it on every validation failure, preserves hoisted scan roots, and refreshes real behavior proof against the fixed head.

## Reproduction Assessment

Yes, source-level. Current main still uses destructive npm-uninstall rollback after npm mutates the managed root, and the PR body documents a targeted blocked-update regression that fails on main; I did not run tests in this read-only review.

## Solution Assessment

No. The snapshot rollback approach is the right direction, but the current diff duplicates the install path, uses an undeclared snapshot variable, and drops hoisted dependency scan roots.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Declare and initialize the rollback snapshot before use:** `src/plugins/install.ts:776`
  - body: The new rollback path passes `rollbackSnapshot` into rollback and cleanup, but the latest head has no local declaration or snapshot creation before these uses. This will fail type/build checks before the rollback regression can run.
  - confidence: 0.98
- **[P1] Move the snapshot into the existing npm install flow:** `src/plugins/install.ts:729-747`
  - body: The patch leaves the existing npm install block in place and then starts a second install/validation block. That means npm can mutate the managed root before the intended snapshot protection exists, and a successful install would run npm twice once the compile issue is fixed.
  - confidence: 0.94
- **[P1] Pass new root packages into the install scanner:** `src/plugins/install.ts:864-867`
  - body: The replacement validation call no longer passes `additionalDependencyPackageDirs`. Current main computes newly installed root package dirs and passes them to the scanner, so omitting this can skip newly hoisted packages during post-install security validation.
  - confidence: 0.9

## Security Review
