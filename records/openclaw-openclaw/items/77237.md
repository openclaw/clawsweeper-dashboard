## Label Justifications

- `P2`: This is a normal-priority managed plugin installer reliability fix with limited blast radius but real user impact when a blocked update corrupts the active root.
- `impact:data-loss`: The affected failure mode can remove or corrupt the active managed plugin root, lockfile, manifest, and hoisted dependency state after a rejected candidate install.
- `merge-risk: 🚨 availability`: The PR intentionally adds full `node_modules` snapshot I/O before managed npm installs, which can increase latency, temporary disk usage, and failure modes for large roots.

## Summary

Keep open: current main still uses best-effort managed npm rollback, while this PR adds focused snapshot restore coverage with sufficient live proof and no blocking code findings. Merge should wait for maintainer acceptance of the full-root snapshot cost and the current-head CI failure to be cleared.

## What This Changes

The PR snapshots `package.json`, `package-lock.json`, and `node_modules` before managed npm plugin installs, restores that snapshot on blocked installs or updates, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Land the snapshot-based rollback once required checks are green and maintainers accept the full-root snapshot as the right atomicity tradeoff for blocked managed npm installs.

## Reproduction Assessment

Yes. Source inspection shows current main mutates the managed npm root before later validation failures call rollback without a full-root snapshot, and the PR body supplies after-fix live output for the blocked update path.

## Solution Assessment

Yes, with the explicit availability tradeoff. Capturing and restoring the previous managed npm root is the narrowest reliable fix for npm mutations that can affect package files, lockfiles, plugin directories, and hoisted dependencies before validation rejects the candidate.
