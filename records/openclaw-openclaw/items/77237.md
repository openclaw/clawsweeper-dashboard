## Summary

Keep open for normal maintainer review: current main still lacks atomic managed-npm-root rollback, this PR addresses that source-level failure path with a focused snapshot/restore change and regression coverage, and I found no blocking correctness or security finding.

## What This Changes

The PR snapshots and restores managed npm `package.json`, `package-lock.json`, and `node_modules` around plugin installs or updates, adds rollback regression coverage, and records a changelog entry.

## Best Possible Solution

Land the snapshot-based rollback after required CI and maintainer gates pass, keeping the change scoped to managed npm install/update rollback paths.

## Reproduction Assessment

Yes, source-level. Current main lets npm mutate the managed root before later validation and then rolls back with uninstall/removal rather than restoring the prior manifest, lockfile, and hoisted dependency state.

## Solution Assessment

Yes. Capturing the previous local root before npm mutation and restoring it on later validation failure is the narrow maintainable fix for npm hoisting and lockfile mutations; rebuild-from-tree rollback would depend on npm or registry behavior during an error path.
