## Summary

Keep open: the requested rollback fix is valid, but the latest PR head is not mergeable because the managed npm install path references an undeclared rollback snapshot, duplicates the install flow, and drops hoisted dependency scan roots.

## What This Changes

The PR adds snapshot/restore rollback around managed npm plugin installs with regression coverage, plus small provider-registry and session-chat-type test-stabilization changes.

## Best Possible Solution

Integrate the snapshot into the single existing managed npm install flow before any root mutation, restore it on every validation failure, preserve additionalDependencyPackageDirs for scanner coverage, and refresh proof on the repaired head.

## Reproduction Assessment

Yes, source-level. Current main uses destructive rollback after npm can mutate the managed root, and the PR body documents a targeted blocked-update regression that fails on main; I did not run tests in this read-only review.

## Solution Assessment

No. Snapshot rollback is the right direction, but the current PR head needs repair before merge because it duplicates the install flow, never initializes the snapshot variable, and omits hoisted dependency scan roots.
