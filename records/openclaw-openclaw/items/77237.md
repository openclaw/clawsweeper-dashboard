## Summary

Keep open: current main still uses the destructive managed npm rollback path, while this PR adds a meaningful atomic rollback fix with focused coverage. The patch is review-clean from the diff I inspected, but the full per-plugin `node_modules` snapshot/restore is an availability tradeoff maintainers should explicitly accept before merge.

## What This Changes

The PR snapshots the managed npm per-plugin project root, restores `package.json`, `package-lock.json`, and `node_modules` after blocked validation, and adds npm installer regression coverage.

## Best Possible Solution

Land an atomic per-plugin rollback once maintainers accept the full-tree snapshot availability tradeoff, or replace it with an equally proven narrower rollback design before merge.

## Reproduction Assessment

Yes. Current main rollback source shows the old uninstall/remove cleanup path, and the PR body provides live output where a blocked update preserves plugin `1.0.0` after npm attempted `2.0.0`; I did not rerun the live command in this read-only review.

## Solution Assessment

Unclear as a final merge choice. The exact snapshot is the strongest atomic rollback design, but maintainers should decide whether the full-tree I/O cost is acceptable for managed npm installs.

## Risks / Open Questions

- Merging this intentionally copies and may restore a full per-plugin `node_modules` tree around managed npm installs, so large plugin projects or filesystems without cheap copy-on-write can see extra disk use, I/O, or install stalls.
