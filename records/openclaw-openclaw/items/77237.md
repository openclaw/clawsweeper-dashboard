## Summary

Keep open: the underlying plugin rollback bug is valid and the snapshot direction is useful, but the latest PR head is not mergeable because it references an undeclared snapshot, duplicates the npm install flow, and weakens the hoisted dependency scan path.

## What This Changes

The PR snapshots and restores the managed npm plugin root around npm-backed install/update validation failures, with regression coverage and small provider/session test-stabilization changes.

## Best Possible Solution

Repair src/plugins/install.ts so one install flow creates the rollback snapshot before npm-root mutations, restores it on every validation failure, preserves new-root package scanning, and refreshes real behavior proof against the fixed head.

## Reproduction Assessment

Yes, source-level. Current main uses destructive npm-uninstall rollback after npm mutates the managed root, and the PR body documents a targeted blocked-update regression that fails on main; I did not run tests in this read-only review.

## Solution Assessment

No. The snapshot approach is the right direction, but the current PR head accidentally duplicates the install path, references an undeclared snapshot, and drops hoisted dependency scan roots.
