## Summary

Keep open; the rollback goal is valid, but the latest PR head is unbuildable because `src/plugins/install.ts` has a dangling `finally` and the install flow is no longer inside the intended outer rollback block.

## What This Changes

The PR changes managed npm plugin installs to snapshot and restore the npm root on rejected installs or updates, adds npm rollback tests and a changelog entry, and includes later provider-registry/session stabilization changes.

## Best Possible Solution

Reapply the snapshot rollback around the existing managed npm install and validation flow, pass the snapshot to every failure rollback path, keep cleanup in a valid outer `finally`, and leave unrelated stabilization out unless it is required and explained.

## Reproduction Assessment

Yes. Current main’s rollback path is source-reproducible as destructive after npm mutation, and the PR body supplies a focused main-failing regression; I did not run tests because this was a read-only review.

## Solution Assessment

No, not in the current head. The snapshot approach is the right direction, but the latest branch malformed the control flow; the maintainable fix is to wrap the existing install path rather than relocate it.
