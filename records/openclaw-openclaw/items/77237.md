## Summary

Keep open for maintainer review: the patch addresses a real managed npm rollback bug that is not fixed on current main, and I found no blocking correctness or security finding. The remaining decision is whether maintainers accept the full managed-root snapshot disk/I/O tradeoff on failed install/update paths.

## What This Changes

This PR adds managed npm install rollback snapshots/restores for `package.json`, `package-lock.json`, and `node_modules`, preserves managed-root npm config scope, and adds installer regression tests plus a changelog entry.

## Best Possible Solution

Merge the exact snapshot rollback if maintainers accept the availability tradeoff; otherwise require a lighter rollback design that still atomically restores package metadata, lockfile, plugin files, and hoisted dependencies.

## Reproduction Assessment

Yes. Current main's source path is enough to reproduce conceptually: after npm mutates the managed root, a post-install validation failure rolls back by uninstalling/removing the target instead of restoring the previous package metadata, lockfile, and `node_modules`.

## Solution Assessment

Yes, with a maintainer-owned tradeoff. Exact snapshot restore is the narrowest reliable fix for arbitrary npm mutations across plugin files and hoisted dependencies; a lighter design is only better if maintainers reject the disk/I/O cost.
