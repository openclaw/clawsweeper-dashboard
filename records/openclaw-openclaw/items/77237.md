## Summary

Keep open: current main still has the destructive managed npm rollback behavior, while this PR provides a focused snapshot/restore fix with regression coverage and green head checks; I found no blocking functional or security findings.

## What This Changes

The branch snapshots the managed npm plugin root before npm-backed installs and restores package metadata, lockfile, and node_modules when install/update validation rejects a candidate, with npm regression tests and a changelog entry.

## Best Possible Solution

Land the snapshot-based rollback after maintainer review so rejected npm install/update candidates restore the exact previously trusted managed root state.

## Reproduction Assessment

Yes, source-reproducible: current main mutates the managed npm root before validation and rolls back by uninstall/removal rather than restoring package metadata, lockfile, and hoisted dependencies. I did not run a live install scenario during this read-only review, but the code path and added regression are clear.

## Solution Assessment

Yes. Snapshotting package.json, package-lock.json, and node_modules before npm mutates the root is the narrow maintainable fix because npm can rewrite hoisted shared dependencies and lockfile entries that cannot be reliably inferred after rejection.
