## Summary

Keep open: current main still uses destructive rollback after npm mutates the managed plugin root, while this PR adds a focused snapshot restore with regression coverage and a changelog entry. I found no concrete correctness or security finding in the diff, but the head is currently dirty/unmergeable and needs rebase plus normal validation before landing.

## What This Changes

The PR snapshots package.json, package-lock.json, and node_modules around npm-backed plugin installs, restores that snapshot on rejected installs or updates, and adds rollback regression tests plus a changelog note.

## Best Possible Solution

Rebase the branch onto current main, keep the snapshot-based rollback approach, then land it after the targeted npm install test and changed gate pass.

## Reproduction Assessment

Yes, from source inspection. Current main mutates the managed npm root before validation and rolls rejected installs back destructively, and the PR adds a focused blocked-update test that exercises that path.

## Solution Assessment

Yes. Snapshotting the exact managed npm root before mutation and restoring it on rejection is the narrowest maintainable fix because npm can alter package metadata, lockfile entries, plugin directories, and hoisted dependencies together.
