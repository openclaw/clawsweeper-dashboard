## Label Justifications

- `P1`: A documented authenticated Feishu configuration can hide all native Feishu document/chat tools from real DM sessions, blocking the main Feishu document-reading workflow.
- `impact:auth-provider`: The failure is tied to credential/config availability checks for an authenticated Feishu plugin integration.

## Summary

Keep open: current main and v2026.5.19 still gate Feishu tools on top-level credentials only, while the documented and runtime-supported named-account Feishu config can have credentials only under `channels.feishu.accounts.<id>`. The reporter's follow-up live hotpatch confirmation strengthens the source repro, and the open closing PR addresses published dist import guards rather than the confirmed account-scoped manifest availability gap.

## What This Changes

Ensure Feishu plugin tools such as `feishu_doc`, `feishu_drive`, `feishu_perm`, and `feishu_wiki` appear in Feishu DM agent tool lists for documented account-scoped Feishu configurations.

## Best Possible Solution

Make manifest tool availability express top-level-or-any-named-account Feishu credentials, then update Feishu tool metadata and regression tests so account-scoped configs expose tools without weakening no-credential filtering.

## Reproduction Assessment

Yes, source-reproducible: a config with only `channels.feishu.accounts.<id>.appId/appSecret` follows the docs/runtime account model but fails the current top-level-only manifest tool availability check. The reporter also confirmed that adding account-scoped manifest signals made the tools callable in live Feishu DMs.

## Solution Assessment

Yes, the maintainable fix is account-aware manifest availability rather than another DM-specific injection path. The embedded runner already forwards Feishu channel/account context, so a separate runtime injection workaround would duplicate the wrong layer.
