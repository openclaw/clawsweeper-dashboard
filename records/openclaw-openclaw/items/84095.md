## Summary

Keep open: current main still appears to filter Feishu plugin tools out for documented named-account Feishu configurations, and the reporter confirmed the same failure in the latest v2026.5.19 release before applying a local manifest workaround. The open linked PR hardens a packaging guard but does not address the account-scoped manifest availability mismatch that now has source and deployment evidence.

## What This Changes

The issue asks OpenClaw to expose Feishu plugin tools such as feishu_doc in Feishu DM agent sessions when Feishu is configured with documented named accounts and the agent tool profile allows them.

## Best Possible Solution

Make Feishu plugin tool availability account-aware for both top-level and named-account credentials, preferably through a generic manifest map/wildcard or plugin availability predicate, while preserving policy filtering when no usable Feishu credentials exist.

## Reproduction Assessment

Yes, from source with a named-account Feishu config: the documented channels.feishu.accounts.<id>.appId/appSecret shape does not satisfy the current top-level-only manifest configSignals, so Feishu tools are filtered before the agent tool list. I did not independently run a live Feishu DM session.

## Solution Assessment

Yes, the best fix is to align Feishu tool availability with the existing documented and runtime-supported named-account config contract rather than add a DM-specific injection path. The current linked packaging-guard PR may be useful separately but is not sufficient for this root cause.
