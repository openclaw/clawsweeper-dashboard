## Summary

Close as implemented on current main: the same Weixin QR-login fetch failure class was fixed by the merged runtime lazy-loading change, and current main contains source and regression-test coverage that prevent no-proxy plugin SDK fetch imports from initializing Undici dispatcher state.

## What This Changes

The issue reports that the external Weixin channel plugin fails QR login because its Tencent iLink fetch rejects inside the OpenClaw Gateway process while the same request works outside the Gateway.

## Best Possible Solution

Keep the current-main lazy-loading fix and ship it in the next release so external channel plugins can use plain fetch on clean no-proxy Gateway startup without inheriting OpenClaw's packaged Undici dispatcher state.

## Reproduction Assessment

Yes for the affected release class: the earlier matching report and merged PR include before/after Weixin login output, and the external package code confirms QR login uses POST `fetch()` to the same Tencent iLink endpoint. No current-main failure path remains for the fixed no-proxy dispatcher side effect.

## Solution Assessment

Yes. Lazy-loading the existing Undici/proxy dependencies only when proxy or dispatcher work is actually requested is the narrow maintainable fix because it preserves managed proxy behavior without adding a Weixin-specific bypass.
