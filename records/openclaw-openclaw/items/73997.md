## Summary

Keep open. Current `main` still does not provide a mechanically read-only WhatsApp DM observe mode: `dmPolicy` cannot be set to `observe`, blocked DMs are dropped before any observation store, and admitted DMs still enter the normal session and agent-turn path. The earlier ClawSweeper comment and related read-only/send-policy reports are useful context, but those adjacent mechanisms still process inbound through agent/session machinery and do not satisfy this issue’s explicit no-session/no-turn requirement.

## What This Changes

The issue requests a WhatsApp `dmPolicy: "observe"` mode that logs incoming DMs for read-only agent queries without creating sessions, triggering agent turns, or exposing a reply path.

## Best Possible Solution

Define the privacy/product contract first, then implement a WhatsApp-owned pre-routing observe branch: accept the config value, append bounded observation records before route/session/turn resolution, skip hooks/session recording/agent dispatch/reply delivery for observed DMs, and expose a guarded read tool with account, sender, time-range, retention, deletion, and redaction controls.

## Reproduction Assessment

Yes for source-level verification, not as a live WhatsApp repro. Current `main` rejects `dmPolicy: "observe"`, drops denied DMs before any observation store, and sends admitted DMs through `runInboundReplyTurn`.

## Solution Assessment

Unclear until maintainers approve the privacy model. The narrow maintainable direction is a WhatsApp pre-routing observe policy; existing `sendPolicy` or generic `observeOnly` do not satisfy the no-session/no-turn requirement.
