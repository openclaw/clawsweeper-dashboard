## Summary

Keep open. Current `main` still lacks the requested mechanically read-only WhatsApp DM path: `dmPolicy` does not accept `observe`, blocked DMs return before any observation/persistence branch, and admitted DMs still enter the normal session and agent-turn dispatcher. The related send-policy and generic observe-only work is adjacent but does not satisfy the issue’s explicit no-session/no-turn requirement.

## What This Changes

The issue requests a WhatsApp `dmPolicy: "observe"` mode that records inbound DMs in a queryable read-only log while preventing session creation, agent turns, and reply capability.

## Best Possible Solution

Define the privacy/product contract first, then implement a WhatsApp-owned pre-routing observe policy: accept the config value, append bounded observation records before session resolution, skip hooks/route/session/agent dispatch for observed DMs, and expose a guarded read tool with account, sender, time-range, retention, and redaction controls.

## Reproduction Assessment

Yes for source-level verification, not as a live WhatsApp repro. Current `main` rejects `dmPolicy: "observe"`, drops blocked DMs before any persistence branch, and sends admitted DMs through `runInboundReplyTurn`.

## Solution Assessment

Unclear until maintainers approve the privacy model. The narrow maintainable direction is a WhatsApp pre-routing observe branch; existing `sendPolicy` or generic `observeOnly` would not meet the no-session/no-turn requirement.
