## Summary

Keep open. Current main still does not implement the requested WhatsApp read-only DM observe policy: `observe` is not accepted by the config contract, blocked or unauthorized DMs return before any observation/persistence branch, and admitted DMs still enter the normal hook/session/agent-turn path. The older sendPolicy and channel `observeOnly` primitives are adjacent but not equivalent because they still process inbound turns and/or record session state.

## What This Changes

The issue requests a WhatsApp `dmPolicy: "observe"` mode that logs incoming DMs for read-only agent access while preventing session creation, agent turns, and reply paths.

## Best Possible Solution

Define the privacy and product contract first, then implement a WhatsApp-owned pre-routing observe policy: accept the config value, append bounded observation records before session/route/agent dispatch, skip hooks and reply-capable turn handling for observed DMs, expose a guarded read tool with retention/redaction/account/sender/time filters, and update docs, schema metadata, tests, and changelog.

## Reproduction Assessment

Yes for source-level verification, not as a live WhatsApp repro. Current main rejects `dmPolicy: "observe"`, drops blocked DMs before any persistence branch, and sends admitted DMs through `runInboundReplyTurn`.

## Solution Assessment

Unclear until maintainers approve the privacy model. A WhatsApp pre-routing observe branch is the narrowest direction that matches the issue; `session.sendPolicy` and generic `observeOnly` are not sufficient because they still process inbound turns and/or record session state.

## Security Review

Status: needs_attention

Summary: The requested feature would persist and expose WhatsApp DM content, so it needs an explicit privacy and access-control design before implementation.

Concerns:

- **[medium] Define observation retention and access policy:** `docs/channels/whatsapp.md:166`
  - body: A queryable observation log would store personal message text, phone numbers, sender names, timestamps, account ids, and possibly group/session metadata. The design needs retention, deletion, redaction, account scoping, and agent tool-access rules before this is safe to expose.
  - confidence: 0.9
- **[medium] Keep observed DMs out of the turn pipeline:** `extensions/whatsapp/src/auto-reply/monitor/process-message.ts:435`
  - body: Current WhatsApp processing emits hooks, updates route state, and calls `runInboundReplyTurn` once access is allowed. An observe implementation must intercept before those steps so observed DMs cannot trigger agent turns or reply-capable paths.
  - confidence: 0.86
