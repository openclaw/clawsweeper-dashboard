## Summary

Keep this PR open: the SIGTERM hardening is useful, but the SIGUSR1 recovery path still leaves an availability failure when the lifecycle import itself is what fails, and the contributor proof does not exercise that path.

## What This Changes

The PR wraps the gateway SIGTERM and SIGUSR1 async signal handlers in `.catch()` blocks and adds run-loop tests for error logging, SIGTERM stop fallback, and SIGUSR1 restart-token clearing.

## Best Possible Solution

Land the catch-based hardening only after SIGUSR1 either clears the restart token or fails closed when lifecycle recovery cannot load, with focused regression coverage and redacted live proof for that fault path.

## Reproduction Assessment

Yes for source-level reproduction: current main discards the async signal-handler promises, and the PR's residual SIGUSR1 bug follows from the restart-token contract when lifecycle import recovery also fails. I did not run a live fault-injection test in this read-only review.

## Solution Assessment

No. Adding `.catch()` is the right direction, but the SIGUSR1 path must fail closed or clear state even when the lifecycle module import itself is unavailable; the current branch only handles errors after a successful recovery import.
