## Summary

Keep this PR open: the SIGTERM catch is a focused availability improvement and now has real SIGTERM fault-injection proof, but the SIGUSR1 catch currently logs and returns after restart state may already be marked in-flight, which can strand future gateway restart requests.

## What This Changes

The PR adds `.catch()` handlers around the gateway SIGTERM and SIGUSR1 async signal handlers and adds run-loop tests for handler error logging and fallback behavior.

## Best Possible Solution

Keep the SIGTERM fail-closed catch, but make SIGUSR1 failures either stop/fail closed or explicitly clear the restart token, add regression coverage for a later restart request, and provide redacted live proof for the SIGUSR1 catch path.

## Reproduction Assessment

Yes for source-level reproduction: current main discards async signal-handler promises after loading lifecycle runtime, and the PR branch's SIGUSR1 catch logs without clearing or consuming restart state. I did not run a live fault-injection test in this read-only review.

## Solution Assessment

No as written: the SIGTERM fallback is the right narrow boundary, but SIGUSR1 should not swallow a failed restart after the restart token can already be in-flight. The safer repair is fail-closed stop behavior or explicit restart-state rollback/handling plus coverage.
