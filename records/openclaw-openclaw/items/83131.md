## Summary

Keep open: the PR is a focused gateway bug fix, but the SIGUSR1 failure path only logs and returns, which can leave an authorized restart cycle marked in-flight; the supplied proof is also a standalone Node simulation rather than real patched-gateway behavior.

## What This Changes

Adds `.catch()` handlers to the gateway SIGTERM and SIGUSR1 async signal-handler IIFEs, with SIGTERM logging and falling back to `request("stop", "SIGTERM")` while SIGUSR1 logs import failures.

## Best Possible Solution

Keep the small signal-boundary fix, but make the SIGUSR1 import-failure path consume or roll back restart state, then prove the actual run-loop behavior with focused coverage or real gateway output.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for source reproduction: current main awaits the lifecycle loader inside discarded SIGTERM/SIGUSR1 async handlers, so a rejected import skips `request(...)`; I did not run a live gateway repro in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No, not yet: the SIGTERM catch is the right shape, but the SIGUSR1 catch should also clear or roll back the restart cycle instead of only logging.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Clear the SIGUSR1 restart token on failure:** `src/cli/gateway-cli/run-loop.ts:748-750`
