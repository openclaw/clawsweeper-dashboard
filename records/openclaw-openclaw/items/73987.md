## Summary

Keep open. Current main still scopes Control UI tokens by exact gateway URL, but this PR should not merge as-is because its positive test claims cross-host dashboard reuse with one mocked sessionStorage object, and the credential-scope expansion needs maintainer/security sign-off.

## What This Changes

The PR adds Control UI loopback host token-scope normalization, storage tests for reuse and isolation, and a changelog entry crediting the earlier #43254 work.

## Best Possible Solution

Define the approved loopback credential boundary, then either narrow the change to same-origin `gatewayUrl` alias edits or add an explicit canonical-origin/handoff design for true cross-host dashboard links, with tests and changelog wording that match the chosen behavior.

## Reproduction Assessment

Yes. Current main's exact-host token key is visible in source, and the PR finding is source-reproducible because the added test changes page hosts while retaining a single mocked sessionStorage object.

## Solution Assessment

No. Storage-key normalization may be a narrow solution for same-origin gateway URL alias edits, but it is not sufficient for browser address changes across localhost, 127/8, and IPv6 origins without a canonical-origin or handoff design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P2] Model browser storage per origin:** `ui/src/ui/storage.node.test.ts:284-287`
  - body: This positive case switches the page host from `127.0.0.1` to `localhost`, another 127/8 address, and `[::1]` while keeping the same mocked `sessionStorage`. Real browser sessionStorage is scoped by origin and tab, so a token saved under one loopback host will not be available after opening the dashboard under another host. Please either narrow this to same-origin `gatewayUrl` alias edits or add a real cross-origin handoff/canonical-host design before claiming dashboard-link reuse.
  - confidence: 0.86

## Security Review
