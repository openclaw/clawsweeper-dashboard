## Summary

Keep this PR open, but do not merge it as-is. Current main still scopes Control UI gateway tokens by exact parsed host, so the underlying loopback-alias problem is not already implemented; however, this PR’s added cross-host test models `localhost`, `127.0.0.1`, and `[::1]` with one shared `sessionStorage`, which does not match browser origin storage boundaries, and the 127/8 token-scope expansion needs maintainer security/product approval.

## What This Changes

The PR adds Control UI loopback host normalization for gateway token/storage scopes, adds storage tests for same-port/path loopback reuse and isolation, and adds a changelog entry crediting the prior focused PR.

## Best Possible Solution

Keep the focused loopback-token work alive, but first decide the intended boundary: for same-origin `gatewayUrl` alias normalization, narrow the tests/changelog to that behavior and approve or reduce the 127/8 scope; for true cross-origin dashboard-link reuse, use a canonical host redirect or server-mediated handoff instead of relying on browser storage continuity.

## Reproduction Assessment

Yes. The review finding is reproducible from the PR diff itself: the added test changes `location.host` while keeping one mocked `sessionStorage`, whereas the current docs/code intentionally keep tokens in sessionStorage for a selected gateway URL and browser tab.

## Solution Assessment

No. Storage-key normalization is a plausible narrow fix for same-origin gateway URL edits, but it is not the best complete solution for cross-host dashboard navigation; a canonical-origin or handoff design is safer if that broader behavior is required.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Do not claim host alias reuse with one sessionStorage:** `ui/src/ui/storage.node.test.ts:284-287`
  - body: This test switches `location.host` from `127.0.0.1` to `localhost` and `[::1]` while keeping the same mocked `sessionStorage`. Real browsers partition sessionStorage by origin, so a token saved before navigating to a different loopback host is not available after the host change. Please narrow this PR to same-origin `gatewayUrl` alias edits or add a real cross-origin handoff/canonical-host design before claiming dashboard link reuse.
  - confidence: 0.86
