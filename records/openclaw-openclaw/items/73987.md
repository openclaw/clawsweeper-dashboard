## Summary

Keep open: the PR carries useful loopback-token-scope work from #43254, but it is not merge-ready because it changes a bearer-token reuse boundary and the positive test overstates cross-host dashboard reuse under browser sessionStorage origin partitioning.

## What This Changes

The PR adds Control UI loopback host normalization for gateway token/storage scopes, adds storage tests for reuse and isolation, and adds a changelog entry crediting the earlier #43254 work.

## Best Possible Solution

Define the desired loopback-token boundary first; if the goal is same-origin gatewayUrl alias edits, narrow the tests/changelog to that behavior and approve the limited normalization, while true cross-host dashboard navigation should use a canonical-origin redirect or explicit handoff design.

## Reproduction Assessment

Yes. The review finding is reproducible from the PR diff and current docs: the added test changes page hosts while retaining one mocked sessionStorage, but browser sessionStorage is origin-partitioned and the docs describe current-tab storage for a selected gateway URL.

## Solution Assessment

No. Storage-key normalization may be a narrow fix for same-origin gatewayUrl edits, but it is not the best complete solution for cross-host dashboard links; that needs either a narrower claim or a separate canonical-origin/handoff design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not claim cross-host reuse with one sessionStorage:** `ui/src/ui/storage.node.test.ts:284-287`
  - body: This test switches `location.host` from `127.0.0.1` to `localhost` and `[::1]` while keeping the same mocked `sessionStorage`. Real browser sessionStorage is origin-partitioned, so a token saved before navigating to a different loopback host is not available after the host change; please narrow this to same-origin gatewayUrl alias edits or add an explicit handoff/canonical-host design before claiming dashboard link reuse.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has no dependency, workflow, package, or install-script changes, but it broadens loopback bearer-token reuse boundaries in Control UI storage.

Concerns:

- **[low] Loopback alias scope broadens bearer-token reuse:** `ui/src/ui/storage.ts:33`
  - body: Mapping numeric 127/8 addresses, localhost, and IPv6 loopback to one token-scope host lets one same-origin Control UI storage bucket reuse a gateway token across those aliases when protocol, port, and path match. That may be acceptable, but it is a security-sensitive boundary change that should be explicitly approved or narrowed.
