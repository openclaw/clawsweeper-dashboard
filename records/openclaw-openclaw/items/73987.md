## Summary

Keep open. Current main still scopes Control UI tokens by exact gateway host, so the PR is not obsolete, and the prior #43254 work is useful context. The proposed branch is not merge-ready because its positive test models changing the page host while keeping one mocked sessionStorage, which does not prove real cross-origin dashboard-link token reuse; the bearer-token scope expansion across loopback aliases also needs maintainer security/product approval.

## What This Changes

The PR adds Control UI gateway token-scope normalization for localhost, numeric 127/8 loopback addresses, and bracketed IPv6 loopback, adds storage tests for loopback reuse and isolation, and adds a changelog entry crediting #43254.

## Best Possible Solution

Keep the focused loopback-token work alive, but narrow the behavior and tests to same-origin Control UI sessions where gatewayUrl aliases change inside one browser storage partition. If maintainers want cross-origin dashboard-link reuse, use a canonical dashboard origin or explicit handoff design instead of relying on sessionStorage continuity.

## Reproduction Assessment

Yes. The review finding is reproducible from the PR diff and current contract: the added test changes the mocked page host while reusing one sessionStorage, while current code/docs keep the token in sessionStorage for the current tab and selected gateway URL.

## Solution Assessment

No. The storage-key normalization is a reasonable narrow mechanism for same-origin gatewayUrl alias edits, but it is not the best complete solution for cross-host dashboard navigation; the safer path is to narrow the PR or design a canonical-origin/server-mediated handoff.
