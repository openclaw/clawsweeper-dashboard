## Summary

Keep open. Current main still scopes Control UI tokens by exact gateway host/path, but this patch is not ready because its positive test and changelog imply cross-host dashboard token reuse even though sessionStorage is origin-partitioned, and the implementation broadens loopback bearer-token reuse.

## What This Changes

The PR adds Control UI loopback host normalization for gateway token/storage scopes, storage tests for reuse and isolation, and a changelog entry crediting the earlier #43254 work.

## Best Possible Solution

Keep the focused loopback-token work alive, but define the intended scope first: same-origin gatewayUrl alias edits can use narrowed storage-key normalization, while true cross-host dashboard navigation needs a canonical-origin redirect or explicit handoff design.

## Reproduction Assessment

Yes. Current main exact-host behavior is visible in `ui/src/ui/storage.ts`, and the PR finding is reproducible from the diff because the added test changes host aliases while retaining one mocked sessionStorage instance despite browser origin partitioning.

## Solution Assessment

No. The proposed normalization may be useful for same-origin gateway URL edits, but it is not the best complete solution for cross-host dashboard link reuse and should not broaden 127/8 token reuse without maintainer/security approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P2] Stop modeling cross-origin host reuse with one storage:** `ui/src/ui/storage.node.test.ts:284-287`
  - body: This test switches `location.host` from `127.0.0.1` to `localhost`, another 127/8 address, and `[::1]` while keeping the same mocked `sessionStorage`. Real browser sessionStorage is partitioned by origin and tab, so a token saved before navigating to another loopback host will not be available in that new origin. Please narrow this to same-origin `gatewayUrl` alias edits or add a real cross-origin handoff/canonical-host design before claiming dashboard link reuse.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: No dependency, workflow, package, or install-script changes were found, but the PR changes bearer-token reuse boundaries for loopback gateway URLs.

Concerns:

- **[low] Loopback alias scope broadens bearer-token reuse:** `ui/src/ui/storage.ts:33`
  - body: Mapping all numeric 127/8 addresses plus localhost and IPv6 loopback to one token-scope host means a token saved for one loopback address can be reused by another same-port/path local gateway within the same sessionStorage origin. That may be acceptable, but it broadens the existing exact-host credential boundary and should be explicitly approved or narrowed.
  - confidence: 0.78
