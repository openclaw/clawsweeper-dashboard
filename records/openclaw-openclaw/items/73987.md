## Summary

Keep open. Current main still scopes Control UI gateway tokens by the exact parsed host, so this PR is not obsolete; however, the patch should not merge as-is because its positive test proves token-key normalization inside one mocked storage area rather than browser behavior after switching loopback page origins.

## What This Changes

The PR normalizes Control UI gateway token/storage scope for loopback aliases, adds same-port/path isolation tests, and adds a changelog entry crediting the prior loopback-token PR.

## Best Possible Solution

Keep the focused loopback-token work alive, but settle the intended boundary first: if this is only for same-origin `gatewayUrl` alias edits, narrow the tests and changelog to that behavior and approve or reduce the loopback scope; if true cross-origin dashboard-link reuse is required, use a canonical-origin redirect or server-mediated handoff rather than relying on shared browser storage.

## Reproduction Assessment

Yes. The review finding is reproducible from the PR diff: the test changes mocked `location.host` across loopback aliases while keeping one mocked sessionStorage instance, which does not model a real browser origin change.

## Solution Assessment

No. Storage-key normalization is a plausible narrow fix for same-origin gateway URL edits, but it is not the best complete solution for cross-host dashboard navigation without a canonical-origin or handoff design and explicit credential-scope approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Do not model cross-origin token reuse with one storage:** `ui/src/ui/storage.node.test.ts:284-287`
  - body: This test switches `location.host` from `127.0.0.1` to `localhost` and `[::1]` while keeping the same mocked `sessionStorage`. Real browsers partition sessionStorage by origin, so a token saved before navigating to a different loopback host is not available after the host change. Please narrow this to same-origin `gatewayUrl` alias edits or add a real cross-origin handoff/canonical-host design before claiming dashboard-link reuse.
  - confidence: 0.86
