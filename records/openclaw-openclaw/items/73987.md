## Summary

This PR should stay open because the loopback token-scope change is not on current main, and the proposed branch still needs maintainer/security judgment on the credential boundary. The main blocker is that the new positive test models cross-host loopback navigation with one shared sessionStorage object, while browser sessionStorage is partitioned by origin and tab.

## What This Changes

The PR normalizes Control UI gateway token storage scopes across loopback aliases, adds storage regression tests, and adds a changelog entry.

## Best Possible Solution

Define the local credential boundary first, then land a focused storage-key change and regression tests that match browser origin behavior while preserving protocol, port, and path isolation.

## Reproduction Assessment

Yes for source-level behavior: current main uses exact gateway host scopes, and the PR diff shows how the proposed alias reuse is modeled. No live browser proof establishes cross-host loopback navigation because sessionStorage is origin-partitioned by origin and tab. ([developer.mozilla.org](https://developer.mozilla.org/en-US/docs/Web/API/Window/sessionStorage))

## Solution Assessment

No. The normalization may be useful for same-origin gatewayUrl alias edits, but the PR should not claim cross-host dashboard token reuse or broaden 127/8 bearer-token scope without explicit maintainer/security approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P2] Stop modeling cross-origin host reuse with one storage:** `ui/src/ui/storage.node.test.ts:281-287`
  - body: This test switches location.host from 127.0.0.1 to localhost, another 127/8 address, and [::1] while keeping the same mocked sessionStorage. Real browser sessionStorage is partitioned by origin and tab, so a token saved before navigating to another loopback host will not be available in that new origin. Please narrow this to same-origin gatewayUrl alias edits or add a real cross-origin handoff/canonical-host design before claiming dashboard link reuse. ([developer.mozilla.org](https://developer.mozilla.org/en-US/docs/Web/API/Window/sessionStorage))
  - confidence: 0.86
