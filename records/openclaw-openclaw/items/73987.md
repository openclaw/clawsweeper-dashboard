## Summary

Keep open. The PR is aimed at real Control UI loopback token-scope fragmentation documented in the source PR and related issue comments, but the proposed patch still needs explicit maintainer/security judgment because it changes bearer-token reuse boundaries and its positive test models cross-origin alias reuse with one mocked sessionStorage even though browser sessionStorage is partitioned by origin and tab. ([developer.mozilla.org](https://developer.mozilla.org/en-US/docs/Web/API/Window/sessionStorage))

## What This Changes

The PR normalizes Control UI gateway token storage scopes across localhost, numeric IPv4 127/8, and bracketed IPv6 loopback aliases, adds node storage tests, and adds a changelog entry.

## Best Possible Solution

Define the local Control UI credential boundary, then land a focused token-scope change with browser-origin-accurate tests that preserves sessionStorage and protocol, port, and path isolation.

## Reproduction Assessment

Yes for source-level token-scope fragmentation: current main keys Control UI tokens by exact parsed gateway host, and the related discussion documents localhost versus 127.0.0.1 token-missing cases. No for the PR's claimed cross-origin browser reuse, because browser sessionStorage is origin/tab partitioned. ([developer.mozilla.org](https://developer.mozilla.org/en-US/docs/Web/API/Window/sessionStorage))

## Solution Assessment

No. Storage-key normalization may be the right narrow fix for same-origin gatewayUrl alias edits, but the PR should not claim cross-host dashboard token reuse or broaden bearer-token scope without maintainer/security approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P2] Stop modeling cross-origin token reuse with one storage:** `ui/src/ui/storage.node.test.ts:281-287`
  - body: This test switches location.host from 127.0.0.1 to localhost, another 127/8 address, and [::1] while keeping the same mocked sessionStorage. Browsers partition sessionStorage by origin and tab, so a token saved before opening another loopback host will not be available in that new origin; narrow this to same-origin gatewayUrl alias edits or add a real canonical-host/handoff design. ([developer.mozilla.org](https://developer.mozilla.org/en-US/docs/Web/API/Window/sessionStorage))
  - confidence: 0.86
