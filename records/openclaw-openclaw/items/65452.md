## Summary

Keep open: the PR targets a real current-main SQL DDL risk, but the patch is not correct or merge-ready because it edits a now-deleted source-side schema file, rewrites valid quoted/dotted SQLite identifiers, and lacks auditable real behavior proof.

## What This Changes

The PR adds SQLite identifier quoting helpers to memory schema DDL in the package implementation and an older source-side memory schema file.

## Best Possible Solution

Land a revised package-owned SQL identifier hardening that defines preserve-or-reject behavior for schema-qualified and quoted names, adds focused regression tests, and keeps the current SDK bridge layout intact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call exported ensureMemoryIndexSchema with cacheEnabled or ftsEnabled and a crafted embeddingCacheTable or ftsTable; current main interpolates those strings directly into db.exec DDL.

## Solution Assessment

No. Identifier quoting is the right mitigation direction, but this PR should be rebased onto the package-only layout and should either parse/preserve SQLite quoted identifiers or reject unsupported forms explicitly with tests.
