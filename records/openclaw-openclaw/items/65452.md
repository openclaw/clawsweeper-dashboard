## Summary

Keep open: current main and v2026.5.7 still expose the unquoted memory-schema DDL path, but this PR is not mergeable as-is because it touches a deleted source-side copy, changes SQLite dotted-name semantics, and has no auditable real behavior proof.

## What This Changes

The PR adds SQLite identifier-quoting helpers around memory schema table and column DDL in the memory host SDK and the old source-side memory schema implementation.

## Best Possible Solution

Land a revised package-owned SQL identifier hardening that defines preserve-or-reject behavior for SQLite identifiers, adds focused regression tests, and keeps the current SDK bridge layout intact.

## Reproduction Assessment

Yes. Current main source still interpolates embeddingCacheTable and ftsTable into db.exec DDL, and a focused node:sqlite in-memory check confirms a crafted identifier string can terminate a statement and drop files.

## Solution Assessment

No. Identifier quoting is the right mitigation direction, but this PR should be rebased onto the package-only layout and should parse/preserve or explicitly reject quoted dotted SQLite identifiers with tests.
