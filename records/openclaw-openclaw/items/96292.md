## Summary

Keep open: current main standardizes where skills are installed and loaded, but it does not provide a skill-owned writable data directory, a `metadata.openclaw` data-path field, or a `{{skill_data}}` prompt/runtime placeholder. This is useful ecosystem cleanup, but it needs maintainer direction because it creates a public skill metadata and storage contract.

## What This Changes

Request to add a standard writable data directory convention for skills, such as `~/.openclaw/data/skills/<slug>`, and optionally a `metadata.openclaw` declaration or placeholder for skill authors.

## Best Possible Solution

Keep this open for maintainer design of a narrow skill-owned data contract, with storage semantics, docs, tests, and migration/deprecation guidance for legacy `configPaths` rather than treating legacy metadata as executable behavior.

## Reproduction Assessment

Not applicable; this is a feature request, not a broken existing behavior. Source inspection confirms current main has skill source roots and metadata parsing, but no supported data-directory placeholder or metadata field to reproduce as a bug.

## Solution Assessment

Unclear as proposed; the report identifies a real missing convention, but maintainers need to choose filesystem versus SQLite/plugin KV and whether the contract belongs in `metadata.openclaw`, a capability manifest, or an install/runtime API.
