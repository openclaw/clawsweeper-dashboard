## Summary

Keep open. Current main and the latest release do not provide a first-class per-skill writable runtime-data directory; the existing `~/.openclaw/skills` path is for managed skill packages, and `~/.openclaw/tools/<skillKey>` is for installer/downloaded tool artifacts, so this remains a product/API storage-boundary decision rather than cleanup-close material.

## What This Changes

The issue requests a standardized OpenClaw-owned per-skill data directory convention under `~/.openclaw/`, plus optional native metadata and install-time directory creation.

## Best Possible Solution

Maintainers should choose one durable skill-data contract, then align SKILL.md metadata, install behavior, docs, and any migration or doctor expectations around that convention.

Not applicable; this is a feature/API convention request, and source inspection confirms the requested data-directory contract is not currently present.
