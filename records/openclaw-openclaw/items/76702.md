## Summary

Keep open. Current main still places the reported Feishu DM timing window inside shared channel turn and agent/session processing, and I found no post-2026.5.2 fix that clearly bounds large-context per-turn latency or resolves this Windows status variant.

## What This Changes

The issue reports a Windows Feishu DM regression after 2026.5.2 where long-lived session/context growth appears to make agent dispatch take 30-60 seconds while `gateway status` shows `Runtime: unknown`.

## Best Possible Solution

Profile a current-main Windows Feishu DM turn with transcript/context-size diagnostics, then fix the smallest shared agent/session compaction or Windows status parsing gap instead of adding Feishu-only policy.

## Reproduction Assessment

No high-confidence live reproduction. Source inspection maps the reported timestamps to Feishu dispatch awaiting the shared channel turn and agent/session path, but the 30-60s Windows Feishu latency was not reproduced on current main.

## Solution Assessment

Unclear. Existing compaction and transcript-byte guards cover some long-session cases, but they are threshold/config driven and CLI compaction runs after CLI turns, so the maintainable fix needs profiling before changing defaults or adding channel-specific behavior.
