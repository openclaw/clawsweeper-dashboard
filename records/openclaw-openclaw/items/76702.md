## Summary

Keep open. Current main still routes the reported Feishu timing window through shared dispatch and agent/session processing, and I did not find a current-main or v2026.5.3 change that clearly bounds this large-context Feishu DM latency regression by default.

## What This Changes

The issue reports a Windows Feishu DM latency regression after upgrading to 2026.5.2, apparently during agent/session processing with large long-lived session context and an additional `Runtime: unknown` gateway status symptom.

## Best Possible Solution

Profile a current-main Windows Feishu DM turn with per-phase timing for dispatch, transcript load, prompt/context construction, compaction, model execution, and `schtasks` parsing, then fix the smallest shared compaction or Windows status gap with focused Feishu coverage.

## Reproduction Assessment

No high-confidence live reproduction. Source inspection confirms the reported log window wraps `core.channel.turn.run` through shared dispatch, but reproducing the 30-60s slowdown needs the reporter's Windows Feishu session state, logs, and raw status output.

## Solution Assessment

Unclear. Existing auto-compaction and the optional transcript byte guard cover some long-session cases, but profiling should decide whether the fix belongs in shared compaction defaults, CLI/session preparation, Feishu instrumentation, or Windows status parsing.
