## Summary

Keep open. Current main still routes the reported Feishu timestamp gap through the shared channel turn and agent/session dispatch path, and the existing compaction controls do not prove that the reported Windows Feishu long-session latency is fixed by default.

## What This Changes

The issue reports a Windows Feishu DM regression in 2026.5.2 where large long-lived sessions make replies take roughly 30-60 seconds between dispatch start and dispatch completion.

## Best Possible Solution

Profile a current-main Windows Feishu DM turn with a similarly large session, then fix the smallest shared agent/session compaction or Windows status gap rather than adding Feishu-only policy first.

## Reproduction Assessment

No high-confidence live reproduction. Source inspection maps the reported log window to Feishu -> shared channel turn -> agent dispatch, but I did not reproduce a 30-60s Windows Feishu turn on current main.

## Solution Assessment

Unclear. Existing auto-compaction and optional transcript byte guards cover some long-session cases, but changing default compaction behavior or Windows status handling needs profiling evidence first.
