Keep open. Current main still routes the reported Feishu timestamps through shared turn dispatch and agent/session processing, and I did not find a current-main change that clearly bounds this large-context latency regression by default.

## What This Changes

The issue requests investigation of a Windows Feishu DM latency regression after v2026.5.2 where long-lived session/context growth appears to make agent dispatch take 30-60 seconds and `gateway status` reports `Runtime: unknown`.

## Best Possible Solution

Profile a current-main Windows Feishu DM turn with per-phase dispatch, transcript, token, compaction, and `schtasks` diagnostics, then fix the smallest shared agent/session compaction or Windows status parser gap rather than adding Feishu-only policy first.

## Reproduction Assessment

No high-confidence live reproduction. Source inspection confirms the reported log window wraps `core.channel.turn.run` through `dispatchReplyFromConfig`, but reproducing the 30-60s slowdown needs the reporter's Windows Feishu session state and logs.

## Solution Assessment

Unclear. Existing auto-compaction and the optional `maxActiveTranscriptBytes` guard cover some long-session cases, but the default, CLI-backed, and Windows-status paths need profiling before changing global compaction behavior.
