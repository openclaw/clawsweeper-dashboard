## Summary

Keep open. Current main still routes the reported Feishu dispatch window through shared turn dispatch and agent/session processing, while the large-context and Windows status subpaths remain conditional enough that a live Windows Feishu profile is needed before a safe fix.

## What This Changes

The issue reports a Windows Feishu DM latency regression after upgrading to 2026.5.2, with slow turns appearing between Feishu dispatch start and dispatch completion and possibly amplified by large session context plus a `Runtime: unknown` gateway status.

## Best Possible Solution

Profile a current-main Windows Feishu DM turn with per-phase dispatch, transcript-size, token, compaction, provider/CLI, and raw `schtasks` diagnostics, then fix the smallest shared session/compaction or Windows status gap without adding Feishu-only policy first.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection maps the reported timestamps to Feishu `core.channel.turn.run` and shared `runDispatch`, but reproducing the 30-60s slowdown needs the reporter's Windows Feishu session state, logs, and raw service status output.

## Solution Assessment

Unclear. Existing compaction and transcript-byte guards cover some long-session cases, but they are conditional and CLI compaction is post-run, so maintainers should profile before changing global defaults or adding channel-specific behavior.
