Keep open. Current main still routes the reported Feishu `dispatching to agent` to `dispatch complete` interval through shared agent/session work, and the current compaction and Windows status code do not prove the large-context latency regression is bounded by default.

## What This Changes

Investigate a Windows Feishu DM latency regression after v2026.5.2 where long-lived large sessions appear to make agent/session processing take 30-60s, with a possible `Runtime: unknown` daemon-status symptom.

## Best Possible Solution

Profile a current-main Windows Feishu DM turn with per-phase dispatch, transcript/token, compaction, and raw `schtasks` diagnostics, then fix the smallest shared session-compaction or Windows-status gap.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection confirms the reported log interval wraps shared dispatch and agent/session processing, but reproducing the latency needs the reporter's Windows Feishu session state and raw logs.

## Solution Assessment

Unclear. Existing auto-compaction and the optional active-transcript byte guard cover some long-session cases, but the default, CLI/runtime, and Windows-status paths need profiling before changing shared behavior.
