## Summary

Keep open. Current main still routes the reported Feishu delay window through shared turn dispatch and agent/session processing, and I did not find a direct current-main or v2026.5.3 fix that bounds long-lived Windows Feishu DM context growth by default.

## What This Changes

The issue reports a Windows Feishu DM latency regression after upgrading to 2026.5.2, with slow turns apparently occurring between Feishu dispatch start and dispatch completion when the reused session context is large.

## Best Possible Solution

Profile a current-main Windows Feishu DM turn with per-phase dispatch, transcript/token, compaction, plugin-tool-prep, provider latency, and raw schtasks diagnostics, then fix the smallest shared agent/session or Windows status gap that the profile proves.

## Reproduction Assessment

No high-confidence live reproduction. Source inspection traces the reported log gap through Feishu dispatch, core turn dispatch, and runReplyAgent, but reproducing the 30-60s slowdown needs the reporter's Windows Feishu session state and timing logs.

## Solution Assessment

Unclear. Current main has opt-in transcript-byte and token-based compaction guards plus adjacent performance fixes, but a maintainable fix needs profiling before changing shared compaction/session defaults or adding Feishu-only policy.
