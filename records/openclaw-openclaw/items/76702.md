## Summary

Keep open. Current main still routes the reported Feishu log window through shared turn dispatch and agent/session processing, and I did not find a current-main or v2026.5.3 change that clearly bounds the reported large-context latency or explains the Windows `Runtime: unknown` symptom.

## What This Changes

The issue reports a Windows Feishu DM latency regression after 2026.5.2, with dispatch-to-agent time growing as long-lived session context and token counts grow.

## Best Possible Solution

Profile a current-main Windows Feishu DM turn with per-phase dispatch, transcript, token, compaction, hook, provider, and `schtasks` diagnostics, then patch the smallest shared agent/session or Windows status gap with regression coverage.

## Reproduction Assessment

No high-confidence live reproduction. Source inspection maps the reported timestamps to `core.channel.turn.run` and `dispatchReplyFromConfig`, but the 30-60s slowdown needs the reporter's Windows Feishu session state, large transcript, and raw status output.

## Solution Assessment

Unclear. Existing auto-compaction plus optional `maxActiveTranscriptBytes` and `midTurnPrecheck` cover some long-session cases, but profiling is needed before changing global defaults or adding Feishu-specific policy.
