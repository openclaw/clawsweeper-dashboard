## Summary

Keep open: current main still has the reverse-scan fallback that can mispair sequential same-name tool results, and the PR head is a plausible narrow fix, but the external PR still lacks real Control UI proof and is currently conflicting against main.

## What This Changes

This PR changes Control UI tool-result matching to prefer exact tool-call ID matches before falling back to the earliest unmatched same-name card, and adds a changelog entry.

## Best Possible Solution

Land the exact-ID-first matcher after redacted real Control UI proof and conflict resolution; add focused sequential same-name regression coverage if maintainers want a stronger guard.

## Reproduction Assessment

Yes, by source inspection against current main. Two same-name tool calls followed by their results can make the reverse scan attach a result to the newest unmatched card, while expanded rendering depends on the matched card's `outputText`.

## Solution Assessment

Yes for the code fix: exact-ID-first matching with earliest-unmatched name fallback is the narrowest maintainable change because it preserves explicit ID pairing while correcting sequential same-name fallback ordering. The merge path still needs real behavior proof and conflict resolution.
