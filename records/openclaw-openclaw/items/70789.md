## Summary

Keep open. Current main still has the reverse-scan tool-result matcher that can mispair sequential same-name tool calls, while this PR is a narrow fix that preserves exact-id priority and changes the fallback to first unmatched same-name card. I found no functional or security blocker in the code diff; the remaining mechanical issue is the required changelog entry for a user-facing Control UI fix.

## What This Changes

The PR changes Control UI tool-card extraction so tool results first match exact card IDs and otherwise attach to the first unmatched same-name tool call.

## Best Possible Solution

Add the required changelog entry, then land this PR or an equivalent narrow patch in `ui/src/ui/chat/tool-cards.ts`; a focused sequential pairing test would make the regression guard stronger without changing the product boundary.

## Reproduction Assessment

Yes. A message content array with two same-name `tool_use` blocks followed by two same-name `tool_result` blocks reproduces the issue by source inspection on current main because the matcher scans backward and selects the newest unmatched card first.

## Solution Assessment

Yes. Exact-id-first matching with first-unmatched same-name fallback is the narrow maintainable fix for this renderer helper; the safer follow-up is changelog plus focused regression coverage, not a broader UI rewrite.
