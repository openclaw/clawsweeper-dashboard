Keep this PR open. Current main still re-sanitizes session-history images and decodes metadata in `resizeImageBase64IfNeeded` without a cache, while this PR is a focused implementation with no concrete correctness or security finding from this review.

## What This Changes

The PR adds a SHA-256-keyed, byte-bounded resize-result cache in `src/agents/tool-images.ts`, regression tests, a log-test cache reset, and a changelog entry.

## Best Possible Solution

Land or adapt the full-payload-keyed, bounded cache in `tool-images`, keep the regression coverage, and validate the changed lanes before merge.

## Reproduction Assessment

Yes, source-backed rather than live-run. Current main routes replayed session messages through image sanitization, and the resize helper decodes metadata without a cache on each pass.

## Solution Assessment

Yes. Caching at the resize helper is the narrowest maintainable fix for repeated historical image sanitization, and this PR avoids the prior prefix-collision and unbounded-cache problems from #64514.
