## Summary

Keep open: current main still re-sanitizes historical images without a resize-result cache, and this PR is the active canonical fix path, but it needs a small correctness fix because the memoization key omits an input that can affect the returned MIME type.

## What This Changes

The PR adds a SHA-256-keyed, byte-bounded image resize-result cache, cache reset/stats test hooks, regression tests, and a changelog entry for repeated session-history image resizing.

## Best Possible Solution

Include every return-affecting input in the cache key, keep the full-payload byte-bounded cache and regression tests, add acceptable real behavior proof, then run the changed gates before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: session history calls sanitizeSessionMessagesImages, which routes historical image content through sanitizeContentBlocksImages, which currently calls resizeImageBase64IfNeeded without a cache.

## Solution Assessment

Mostly, but not as-is. Caching at the resize helper is the narrowest maintainable fix for repeated historical image sanitization, but the cache key should include the declared or normalized MIME type because the no-resize return can depend on it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Include MIME type in the resize cache key:** `src/agents/tool-images.ts:200-203`
  - body: `resizeImageBase64IfNeeded` can return `params.mimeType` unchanged on the no-resize path, but the new cache key only includes the base64 payload and size limits. If the same bytes are sanitized with different declared MIME types, the second call can receive the first call's MIME type from the cache; this is especially reachable for formats not corrected by `inferMimeTypeFromBase64`, such as WebP, which the current metadata reader supports. Add the normalized MIME type to the cache key before memoizing the result.
  - confidence: 0.86
