## Summary

Keep this PR open: current main and the latest release still lack resize-result memoization, while this branch is the live canonical fix candidate with focused code, regression coverage, and sufficient terminal proof. ## What This Changes

This PR adds a fixed 64 MiB SHA-256-keyed in-process LRU cache around tool image resize results and resets/tests that cache in adjacent image sanitizer tests.

## Best Possible Solution

Land or adapt the fixed-cap, full-payload resize cache after maintainer session-state review, preserving MIME/no-op behavior, limit-sensitive keys, bounded eviction, backend-unavailable handling, and existing sanitization semantics.

## Reproduction Assessment

Yes. Source inspection shows current main still routes session-history images through `sanitizeSessionMessagesImages` and `sanitizeContentBlocksImages` without a cache, and the PR body supplies terminal proof exercising the actual sanitizer with repeated image input.

## Solution Assessment

Yes. The cache is placed at the shared sanitizer helper used by replay and tool result paths, with full-payload plus limit keying and a fixed byte cap; broader session-store or channel-specific rewrites would be higher risk for this bug.
