## Summary

Keep this PR open, but not merge-ready as-is. The resize-cache approach is focused and well supported by terminal proof, yet the cache key omits an input that can affect no-op output MIME, so a narrow cache-identity fix is needed before maintainer landing.

## What This Changes

The PR adds a SHA-256-keyed, byte-bounded resize-result cache in `src/agents/tool-images.ts`, cache regression tests, log-test cache resets, and a changelog entry.

## Best Possible Solution

Merge a full-payload, byte-bounded resize cache after fixing the MIME-sensitive cache hit and rerunning current-head image sanitizer validation.

## Reproduction Assessment

Yes. Current main source shows session-history messages flow through `sanitizeSessionMessagesImages` and `sanitizeContentBlocksImages`, and `resizeImageBase64IfNeeded` performs metadata decoding before any unchanged-image return; I did not run a live chat repro in this read-only review.

## Solution Assessment

No, not as-is. Caching at the resize helper is the right maintainable shape, but the cache identity/output contract needs to account for effective MIME before this PR is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Include MIME in the resize cache contract:** `src/agents/tool-images.ts:260`
  - body: The cache key only includes base64 and size limits, but the cached value includes `mimeType` and no-op results return the caller's effective MIME. For formats that `inferMimeTypeFromBase64` does not canonicalize but `getImageMetadata` supports, such as WebP/HEIC, sanitizing the same base64 once with one declared MIME and later with another will return the first call's MIME on the cache hit; current main would preserve the second call's MIME. Include `params.mimeType` in the key or keep no-op hits using the current MIME, and add a regression test.
  - confidence: 0.86
