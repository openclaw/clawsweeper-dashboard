## Summary

Keep this PR open: current main still re-sanitizes session-history images without a resize-result cache, and the PR remains a focused candidate with sufficient direct sanitizer proof, but maintainers should explicitly accept the process-local session-image cache semantics before merge.

## What This Changes

Adds a SHA-256-keyed, byte-bounded in-process cache for tool image resize results with regression coverage for cache hits, eviction, keying, MIME preservation, and log-test isolation.

## Best Possible Solution

Merge or adapt the bounded full-payload resize cache after maintainer acceptance of the session-history cache invariant and current gates.

## Reproduction Assessment

Yes. Source inspection of current main shows active session messages are sanitized each model turn and historical image blocks flow through `resizeImageBase64IfNeeded`, which still has no cache before metadata/resize work; I did not run a live chat.

## Solution Assessment

Yes. Caching inside `resizeImageBase64IfNeeded` covers user, toolResult, and assistant replay callers without changing stored session data or adding operator config; a higher-level per-message cache would need stable attachment IDs across more transcript shapes.
