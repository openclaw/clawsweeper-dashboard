## Summary

Keep this PR open: current main still re-sanitizes session-history images without resize-result memoization, while the current PR head cleanly adds a full-payload SHA-256, byte-bounded cache and focused regression coverage. I found no blocking line-level defect, but maintainers should explicitly accept the new process-level session image cache invariant before merge.

## What This Changes

The PR adds a fixed 64 MiB SHA-256-keyed LRU cache around tool image resize results, test hooks, and regression tests for repeated session-history image sanitization.

## Best Possible Solution

Land the PR after maintainer acceptance of the bounded in-memory cache invariant and exact-head gates, keeping the cache cap internal rather than adding a new operator configuration surface.

## Reproduction Assessment

Yes from source inspection and PR proof, though I did not execute a local chat repro. Current main replays historical image blocks through sanitizeContentBlocksImages on each session-history rebuild with no resize cache, and the PR proof shows repeated calls changing from repeated resize work to cache hits.

## Solution Assessment

Yes, this appears to be the best bounded fix for the reported hot path. Caching at resizeImageBase64IfNeeded covers all sanitizer callers while preserving the current header-only pass-through, option normalization, full-payload identity, and no new operator config surface.
