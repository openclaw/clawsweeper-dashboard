## Summary

Keep this PR open. Current main still replays historical image blocks through `sanitizeSessionMessagesImages` and `resizeImageBase64IfNeeded`, where `getImageMetadata` runs before any early return and no resize cache exists; the PR remains a focused implementation of the linked performance fix and I found no blocking correctness or security issue in the diff.

## What This Changes

The PR adds a SHA-256-keyed, byte-bounded resize-result cache in `src/agents/tool-images.ts`, test hooks and regression coverage, a log-test cache reset, and a changelog entry for the session-history image resize bug.

## Best Possible Solution

Land or adapt the PR's full-payload-keyed, byte-bounded cache in `tool-images`, keep the regression tests for repeated calls, collision safety, cache limits, env parsing, eviction, and log isolation, then run the normal changed gate before merge.

## Solution Assessment

Yes. Caching at the resize helper is the narrowest maintainable fix for repeated historical image sanitization, and this PR avoids the prior prefix-collision and unbounded-cache concerns from #64514.
