## Summary

Keep open: current main still lacks a resize-result cache and continues routing historical image blocks through the same metadata/resize path, while this PR is a focused implementation candidate with no blocking review finding from this pass.

## What This Changes

The PR adds a SHA-256-keyed, byte-bounded resize-result cache in `src/agents/tool-images.ts`, cache regression tests, log-test cache reset, and a changelog entry.

## Best Possible Solution

Land or adapt the PR's full-payload-keyed, byte-bounded cache in `tool-images`, keep the regression coverage, and rerun the normal changed gate after rebase or mergeability confirmation.

## Reproduction Assessment

Yes. The current-main reproduction path is source-backed: session history calls `sanitizeSessionMessagesImages`, which routes historical image blocks through `sanitizeContentBlocksImages`, which still calls `getImageMetadata` in `resizeImageBase64IfNeeded` without a cache.

## Solution Assessment

Yes. Caching at the resize helper is the narrowest maintainable fix for repeated historical image sanitization, and the PR avoids the prior prefix-collision and unbounded-cache problems from #64514.
