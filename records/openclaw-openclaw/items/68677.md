## Summary

Keep open: current main and the latest release still route replayed image blocks through the resize helper with no resize-result cache, while this PR remains a focused implementation candidate with sufficient sanitizer-level proof and no blocking code finding. The remaining merge decision is maintainer acceptance of the new process-local cache semantics for model-visible session replay.

## What This Changes

The PR adds a SHA-256-keyed, fixed 64 MiB byte-bounded resize-result cache around tool image sanitization plus regression tests for reuse, keying, eviction, MIME preservation, and log-test isolation.

## Best Possible Solution

The desired end state is one bounded resize cache in the shared sanitizer, with full-payload keys, no new public config surface, regression coverage, and maintainer-accepted cache lifecycle semantics.

## Reproduction Assessment

Yes: current-main source shows session history is sanitized on every replay and each historical image reaches resizeImageBase64IfNeeded with no cache. I did not run a live provider chat in this read-only pass, but the PR body supplies sanitizer-level terminal proof for the same path.

## Solution Assessment

Yes: caching at the shared resize helper is the narrowest maintainable fix because session history, prompt images, and tool results already converge there. Earlier alternatives with prefix keys or count-only caps were weaker; this branch uses full-payload keys and a byte bound.
