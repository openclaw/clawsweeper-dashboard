## Summary

Keep open: current main still handles structured OpenAI tool calls and adjacent Kimi text-marker repairs, but it does not normalize the reported Qwen/ModelStudio-style tagged tool-call sections when they appear inside thinking blocks. A fix needs maintainer product/security review and live provider proof because promoting reasoning text into executable tool calls can cross a security boundary.

## What This Changes

The issue requests fixing premature subagent completion when Qwen/ModelStudio-style models emit tagged tool-call sections inside thinking blocks instead of structured toolCall blocks.

## Best Possible Solution

Add provider-owned Qwen/ModelStudio normalization or a narrow shared OpenAI-compatible repair helper that promotes only validated standalone marker sections before stop-reason and announce logic, with live replay proof and explicit allowed-tool gating.

## Reproduction Assessment

No, not as a high-confidence live repro: source inspection shows marker text inside thinking blocks bypasses current promotion, but the provider-specific stochastic behavior has not been replayed against current main.

## Solution Assessment

Unclear until maintainers choose the security boundary: delaying announce alone would not be the best fix, while narrowly canonicalizing validated provider-owned tool markers before announce is the more maintainable direction if live proof confirms the behavior.
