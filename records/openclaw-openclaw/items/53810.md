## Label Justifications

- `P2`: This is a normal-priority provider/runtime bug with bounded affected model routes and no evidence of a core-wide outage.
- `impact:message-loss`: Affected subagent runs can return truncated or empty output because model-emitted tool calls are not executed.
- `impact:session-state`: The stored assistant turn can retain stopReason stop with marker text in thinking content instead of canonical structured tool-call state.

## Summary

Keep open: current main and v2026.6.8 still handle Kimi tagged markers only when they arrive as text, while the reported ModelStudio/Qwen marker-in-thinking path is not normalized into executable tool calls. Related merged fixes cover adjacent text-block and parallel-tool-call cases, but not this central failure mode.

## What This Changes

The issue asks OpenClaw to normalize provider/model-specific tagged tool-call markers inside thinking or reasoning content so subagent runs do not announce completion before executing the tools.

## Best Possible Solution

Add conservative provider-owned Qwen/ModelStudio normalization, or a narrowly shared OpenAI-compatible helper, before terminal stopReason decisions so only validated standalone tagged sections become structured tool calls.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows the reported marker-in-thinking shape still bypasses current promotion because current parsers use structured tool_calls or text blocks, not thinking-block marker text.

## Solution Assessment

Yes, the best fix is provider/shared normalization before stopReason and subagent completion handling. Delaying subagent announcements after a run is already classified as stopped would treat the symptom rather than restoring canonical tool-call state.
