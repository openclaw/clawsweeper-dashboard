## Summary

Keep open. Current main still lacks the provider/model-specific normalization described in the report: Qwen/ModelStudio only patches request-side thinking flags, the generic OpenAI-compatible stream path only promotes structured `delta.tool_calls`, and the Kimi tagged-tool parser still ignores markers carried in thinking/reasoning blocks.

## What This Changes

Normalize non-standard tagged tool-call markers emitted inside thinking/reasoning content so affected subagents keep executing tools instead of announcing a stopped completion with missing work.

## Best Possible Solution

Add provider-owned Qwen/ModelStudio response normalization, or a narrowly gated shared OpenAI-compatible helper reused by Qwen and Kimi, that extracts only validated standalone tagged tool-call sections from response text/thinking/reasoning content, emits structured `toolCall` blocks, and changes `stopReason` to `toolUse` only after successful extraction.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to feed the OpenAI-compatible stream a `reasoning_content` or `reasoning_details` delta containing the reported tagged section: current code records it as thinking text, never creates a `toolCall`, and leaves or resets the final stop reason to `stop`.

## Solution Assessment

No. Current main only handles structured provider tool-call deltas and Kimi standalone text-block markers; the safer fix is a provider-scoped or shared helper with strict standalone-section parsing and focused regression tests.
