## Label Justifications

- `P2`: This is a normal-priority provider/runtime bug with bounded affected model routes and no evidence of a core-wide outage.
- `impact:message-loss`: Affected subagent runs can return truncated or empty output because model-emitted tool calls are not executed.
- `impact:session-state`: The stored assistant turn can retain `stopReason: "stop"` with marker text in thinking content instead of canonical structured tool-call state.

## Summary

Keep open: current main and the latest release still have only text-block/structured-tool-call repair paths, while the reported tagged marker inside thinking/reasoning content still has no source or release proof of being normalized into executable tool calls.

## What This Changes

The issue reports subagents announcing successful completion before model-emitted tagged tool calls inside thinking text are parsed or executed.

## Best Possible Solution

Add provider-owned Qwen/ModelStudio normalization, or a narrowly gated shared OpenAI-compatible helper, before terminal stopReason decisions and subagent completion announcements.

## Reproduction Assessment

No live high-confidence reproduction was run in this pass. Source inspection shows the reported marker-in-thinking shape still bypasses current promotion because current paths parse structured tool calls or text blocks, not thinking-block marker sections.

## Solution Assessment

Yes, the maintainable fix should happen in provider/shared normalization before terminal tool-use decisions. Delaying subagent announcements after a run is already classified as stopped would treat the symptom rather than restoring canonical tool-call state.
