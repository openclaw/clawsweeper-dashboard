## Summary

Keep open: current main and the latest inspected release still do not normalize the reported Qwen/ModelStudio-style tagged tool-call sections when they appear inside `thinking` blocks, so the central subagent premature-completion problem remains plausible and not superseded by the adjacent Kimi text-marker and OpenAI parallel-tool-call fixes.

## What This Changes

The issue asks OpenClaw to handle model-specific malformed tool-call markers in subagent thinking output so runs do not announce completion before intended tools execute.

## Best Possible Solution

Add a conservative provider-owned Qwen/ModelStudio normalization path, or a narrowly shared OpenAI-compatible helper, that validates standalone tagged tool-call sections before stop-reason classification and leaves subagent announce logic consuming canonical `toolCall` state.

## Reproduction Assessment

No, not as a high-confidence live repro. Source inspection shows the reported marker text inside a `thinking` block bypasses current promotion paths, but the specific provider/model behavior still needs live ModelStudio/Qwen confirmation.

## Solution Assessment

Yes, the best fix is before announce handling: normalize only validated provider marker sections into structured tool calls before terminal stop-reason decisions. Delaying or suppressing announce after a stopped turn would treat the symptom while leaving tool-call state non-canonical.
