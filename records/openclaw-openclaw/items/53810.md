## Summary

Keep open: current main still only handles the reported marker family in a Kimi text-block wrapper, while Qwen/ModelStudio reasoning or thinking deltas still flow through the OpenAI-compatible transport as thinking text and can finish with no structured tool calls.

## What This Changes

The issue reports provider/model-specific subagent completion announcements caused by tagged tool-call markers being emitted inside thinking or reasoning content instead of structured tool-call blocks.

## Best Possible Solution

Add conservative provider-owned Qwen/ModelStudio normalization, or a narrowly shared OpenAI-compatible helper, that promotes only validated standalone tagged tool-call sections from reasoning/thinking content into structured toolCall blocks before subagent completion is evaluated.

## Reproduction Assessment

No live high-confidence reproduction yet; the reported path needs affected provider credentials and stochastic model output. Source inspection does show the current-main gap: Qwen/ModelStudio reasoning text is not parsed into structured tool calls, and `toolUse` can be reset to `stop` when no structured toolCall block exists.

## Solution Assessment

Yes, the best fix is at the provider/OpenAI-compatible stream normalization boundary rather than the subagent announce queue. The announce flow is reacting to the final assistant message shape, so the maintainable fix is to normalize validated provider tool-call markup before completion state is computed.
