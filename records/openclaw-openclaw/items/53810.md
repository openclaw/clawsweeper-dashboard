## Summary

Keep open: current main still does not normalize the reported tagged tool-call markers when they arrive in Qwen/ModelStudio reasoning or thinking content, so subagent completion can still see a plain stop response with no structured tool calls.

## What This Changes

The issue asks OpenClaw to handle model/provider-specific tagged tool-call text so subagents do not announce completion before required tool calls run.

## Best Possible Solution

Add conservative provider-owned Qwen/ModelStudio normalization, or a narrowly shared OpenAI-compatible helper, that extracts only validated standalone tagged tool-call sections from reasoning/thinking content and emits structured toolCall blocks with stopReason toolUse only after successful extraction.

## Reproduction Assessment

No live current-main reproduction was run because this needs provider credentials and stochastic model output. Source inspection does reproduce the code path: Qwen/ModelStudio reasoning text is preserved as thinking, not parsed into toolCall blocks, and stopReason falls back to stop when no structured tool calls exist.

## Solution Assessment

Yes, the narrowest maintainable fix is provider-owned or shared stream normalization at the Qwen/ModelStudio/OpenAI-compatible boundary rather than changing the announce queue, because the queue is reacting to an already-final assistant message with no structured pending tools.
