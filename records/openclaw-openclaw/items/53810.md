## Summary

Keep open: current main still does not normalize the reported Qwen/ModelStudio-style tagged tool-call markers when they arrive through reasoning/thinking content, so the agent can still finish a turn with `stopReason: "stop"` and no structured tool calls. The Kimi parser added since the report helps a related text-block path, but it does not cover Qwen/ModelStudio reasoning deltas or thinking blocks.

## What This Changes

The issue reports provider/model-specific tagged tool-call markers inside thinking content causing subagents to announce completion before structured tool calls are executed.

## Best Possible Solution

Add conservative provider-owned Qwen/ModelStudio normalization, or a narrowly shared OpenAI-compatible helper, that extracts only validated standalone tagged tool-call sections from reasoning/thinking content and emits structured `toolCall` blocks with `stopReason: "toolUse"` only after successful extraction.

## Reproduction Assessment

No, not as a live high-confidence reproduction in this review; the reported path needs provider credentials and stochastic model output. Source inspection does show the current-main path remains source-reproducible: Qwen/ModelStudio reasoning text is not parsed into tool calls, and `toolUse` is reset to `stop` if no structured tool-call block exists.

## Solution Assessment

Yes, the best fix is at the provider/OpenAI-compatible stream normalization boundary rather than the subagent announce queue. The announce flow is reacting to an already-final assistant message with no structured pending tool calls, so parsing validated provider markup earlier is the narrowest maintainable path.
