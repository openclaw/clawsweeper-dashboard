## Summary

Keep open: current main still lacks provider/model-specific normalization for tagged tool-call markers in Qwen/ModelStudio reasoning text, and the newer finish-reason safety discussion means maintainers should decide the trust boundary before an automated fix changes tool execution behavior.

## What This Changes

The issue reports provider/model-specific tagged tool calls embedded in thinking text causing subagents to announce completion before tool execution.

## Best Possible Solution

Make an explicit maintainer decision on the OpenAI-compatible finish-reason trust boundary, then either add narrowly allowlisted provider normalization for validated standalone tagged tool-call sections with regression and live proof, or document the affected model behavior as unsupported.

## Reproduction Assessment

No live high-confidence reproduction was established because the reported path needs affected provider credentials and stochastic model output. Source inspection does show the current-main path: tagged markers in Qwen/OpenAI-compatible reasoning content are not promoted to structured toolCall blocks, and Kimi's parser only handles text blocks.

## Solution Assessment

Unclear until maintainers decide the finish-reason and reasoning-marker trust boundary. The narrow technical location is provider/OpenAI-compatible stream normalization, not the subagent announce queue, but the safe behavior now overlaps a tool-execution safety decision.
