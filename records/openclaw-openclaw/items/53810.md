## Summary

Keep open: current `origin/main` still does not normalize the reported ModelStudio/Qwen tagged tool-call format when it arrives in reasoning/thinking content. Kimi has a provider-local text-block parser, but the reported OpenAI-compatible ModelStudio path still turns reasoning deltas into `thinking` blocks and only creates tool calls from structured `choice.delta.tool_calls`.

## What This Changes

The issue requests provider/model-specific normalization so tagged tool calls emitted in thinking/reasoning text become structured `toolCall` blocks before subagent completion announces.

## Best Possible Solution

Add a conservative provider-owned Qwen/ModelStudio stream normalization layer, or a narrowly gated shared OpenAI-compatible helper, that extracts validated standalone tagged tool-call sections from reasoning/thinking content and only then emits structured `toolCall` blocks and `toolUse`.

## Reproduction Assessment

No live high-confidence reproduction was established in this review because it needs affected provider credentials and a stochastic model path. It is source-reproducible if the provider emits the marker payload shown in the issue: current `origin/main` preserves that payload as thinking/text unless it is already a structured `choice.delta.tool_calls` entry.

## Solution Assessment

Yes, the best fix is a provider-gated parser in the Qwen/ModelStudio stream path or a shared OpenAI-compatible helper with the same narrow guards. Delaying subagent announce would mask the symptom while still leaving the tool call unexecuted.
