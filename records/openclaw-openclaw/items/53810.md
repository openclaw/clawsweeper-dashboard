## Summary

Keep open: current main still lacks the provider/model-specific tagged tool-call normalization described in the report. Qwen/ModelStudio reasoning text still flows through the OpenAI-compatible stream as thinking/text, while the existing Kimi parser only rewrites tagged markers from text blocks, so the reported premature completion remains source-reproducible.

## What This Changes

The issue reports that subagents can announce successful completion when affected OpenAI-compatible provider models emit tagged tool-call markup inside reasoning/thinking text instead of structured tool calls.

## Best Possible Solution

Normalize only validated standalone tagged tool-call sections at the provider/OpenAI-compatible stream boundary, emit structured toolCall blocks, and set toolUse only after successful extraction with focused regression coverage.

## Reproduction Assessment

No live high-confidence reproduction was established because the reported path needs affected provider credentials and stochastic output. Source inspection does show the current-main path: tagged markers in reasoning/thinking content are not promoted to structured toolCall blocks.

## Solution Assessment

Yes, the best fix is provider or OpenAI-compatible stream normalization rather than changing the subagent announce queue. The announce flow should react to a correctly normalized assistant message shape.
