## Summary

Keep open: current main still lacks the reported provider/model-specific tagged tool-call normalization. Qwen/ModelStudio output still flows through the OpenAI-compatible stream path, the Qwen wrapper only patches request-side thinking fields, and marker text inside reasoning/thinking content is not promoted into structured tool calls.

## What This Changes

The issue reports provider/model-specific tagged tool calls embedded in thinking text causing subagents to announce completion before tools run.

## Best Possible Solution

Add conservative provider-owned Qwen/ModelStudio normalization, or a narrowly gated shared OpenAI-compatible helper, that extracts only validated standalone tagged tool-call sections from reasoning/thinking content and emits structured `toolCall` blocks before subagent completion observes the turn.

## Reproduction Assessment

No, not as a live high-confidence reproduction; the affected providers require credentials and the report says output varies by model and endpoint. Source inspection does show the current-main path: tagged markers in reasoning/thinking content are not promoted to structured `toolCall` blocks.

## Solution Assessment

Yes, the best fix is provider or OpenAI-compatible stream normalization rather than changing the subagent announce queue. The announce flow should react to correctly normalized assistant message content.
