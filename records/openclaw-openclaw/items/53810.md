## Summary

Keep open. Current main still handles Qwen/ModelStudio reasoning text and structured OpenAI-compatible tool-call deltas separately, so tagged tool-call markup inside reasoning/thinking content is still not promoted into executable tool calls and the subagent can complete with `stopReason: "stop"`.

## What This Changes

The issue reports that some Alibaba/ModelStudio-compatible models emit tagged tool-call markup inside thinking/reasoning content, causing subagents to announce successful completion before the intended tool calls execute.

## Best Possible Solution

Add provider-owned Qwen/ModelStudio response normalization, or a narrowly gated shared OpenAI-compatible helper, that extracts only validated standalone tagged tool-call sections from reasoning/thinking content and emits structured `toolCall` blocks with `stopReason: "toolUse"` only after successful parsing.

## Reproduction Assessment

Yes, by source inspection: feed the OpenAI-compatible completions stream a `reasoning_content` or `reasoning_details` delta containing the reported tagged section and a final `finish_reason: "stop"`; current code records thinking text, creates no `toolCall`, and leaves the final stop reason as `stop`.

## Solution Assessment

No for current main: it handles structured provider tool-call deltas and Kimi standalone text-block markers, but not tagged tool calls embedded in reasoning/thinking content. The safer fix is a provider-scoped or shared strict parser with focused regression tests.
