## Summary

Keep open. Current main still has the reported provider/model-specific gap: Qwen/ModelStudio streams only patch request-side thinking flags, the generic OpenAI-compatible stream path only promotes structured `delta.tool_calls`, and the Kimi tagged-marker parser remains limited to standalone text blocks rather than markers embedded in thinking/reasoning content.

## What This Changes

The issue reports that some Qwen/ModelStudio-compatible models emit tool-call markers inside thinking text, causing subagent completion announces to report success before pending tool work is executed.

## Best Possible Solution

Add provider-owned Qwen/ModelStudio response normalization, or a narrowly gated shared OpenAI-compatible helper reused by Qwen/Kimi, that extracts only validated standalone tagged tool-call sections from response text/thinking/reasoning content, emits structured `toolCall` blocks, and changes `stopReason` to `toolUse` only after successful extraction.

## Reproduction Assessment

Yes for a source-level synthetic reproduction: feed the OpenAI-compatible stream a chunk whose reasoning text contains the reported Kimi-style markers and whose finish reason is `stop`; current main will retain text/thinking content and produce no structured `toolCall`. The live provider reproduction is plausible from the report but stochastic and credential-dependent.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrow maintainable fix is provider-scoped normalization with strict validation, not an announce-layer guess that arbitrary thinking text means pending tool work.
