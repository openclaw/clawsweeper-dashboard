## Summary

Keep open: current main still has the reported gap. ModelStudio/Qwen uses the OpenAI-compatible stream path, while current marker repair only promotes structured `tool_calls` or text-block tool-call markup; markers inside `thinking` content remain unexecuted and can still leave the turn classified as `stop`.

## What This Changes

The issue reports premature subagent completion announcements when specific provider/model routes emit tool-call marker text inside thinking content instead of structured tool-call blocks.

## Best Possible Solution

Add conservative provider-owned Qwen/ModelStudio normalization, or a narrowly shared OpenAI-compatible helper, before terminal stop-reason decisions so only validated standalone marker sections become structured tool calls.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows a thinking block containing the reported marker text bypasses current promotion paths, because they parse structured `tool_calls` or `text` blocks rather than `thinking` content.

## Solution Assessment

Yes, the maintainable fix belongs before stop-reason and subagent completion handling, in provider/shared normalization. Delaying announce after a stopped turn would treat the symptom while leaving tool-call state non-canonical.
