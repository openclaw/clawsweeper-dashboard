## Summary

Keep open: current main and v2026.5.18 still do not promote the reported tagged tool-call markers when they arrive inside Qwen/ModelStudio reasoning or Kimi thinking content, so the agent can still end with no structured tool calls. The merged Kimi parser is useful partial coverage for standalone text blocks, but it does not cover the reported reasoning/thinking-block failure mode.

## What This Changes

The issue reports provider/model-specific subagent completion announcements when tagged tool-call text is emitted inside reasoning or thinking content instead of structured tool calls.

## Best Possible Solution

Add provider-owned Qwen/ModelStudio normalization, or a narrowly shared OpenAI-compatible helper, that extracts only validated standalone tagged tool-call sections from reasoning/thinking content and emits structured toolCall blocks with stopReason toolUse only after successful extraction.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review because the path needs provider credentials and stochastic affected model output. Source inspection still shows the reported marker-in-reasoning path is not promoted to structured tool calls on current main or v2026.5.18.

## Solution Assessment

Yes: the best fix is at the provider/OpenAI-compatible stream normalization boundary, not the subagent announce queue. The announce flow is reacting to an already-final assistant message with no structured pending tool calls, so parsing validated provider markup earlier is the maintainable repair.
