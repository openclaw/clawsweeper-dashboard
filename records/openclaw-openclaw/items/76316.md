## Summary

Keep open: the proposed OpenAI fix is ineffective on the PR base because `gpt-5.4-mini` matches the broader `gpt-5.[2-9]` branch before the new guard can run, and the PR also carries unrelated WhatsApp session-routing and subagent-policy changes that need their own rationale and coverage.

## What This Changes

The PR adds a `gpt-5.4` reasoning-effort exclusion, passes configured subagent IDs into target-policy checks, and forces WhatsApp inbound DM routing to `per-channel-peer`.

## Best Possible Solution

Fix the OpenAI resolver or payload policy at the point that actually controls chat-completions requests, add a focused regression test for `gpt-5.4-mini`, and split or separately document/test the unrelated WhatsApp and subagent changes.

## Reproduction Assessment

Yes. A focused source-level reproduction is to evaluate `resolveOpenAISupportedReasoningEfforts({ provider: "openai", id: "gpt-5.4-mini" })` after this patch; it still hits the earlier `gpt-5.[2-9]` branch and can still lead `buildOpenAICompletionsParams` to emit `reasoning_effort`.

## Solution Assessment

No. The intended `gpt-5.4` exclusion is placed after a broader earlier match, so it is not the narrowest effective fix; a payload-level guard may be safer if the unsupported case is specifically chat completions with tools.
