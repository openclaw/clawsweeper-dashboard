## Summary

This should stay open: the PR is maintainer-labeled and currently conflicts with current main, and the proposed existing-QR path has a concrete stale-QR correctness blocker that needs repair before merge.

## What This Changes

The PR adds optional gateway QR preflight/existing-result hooks, wires WhatsApp to reuse an active QR before preflight, preserves non-QR result codes in the WhatsApp login tool, and changes WhatsApp status issue handling for unknown linked state.

## Best Possible Solution

Rebase this work onto the current QR state-sync implementation, keep the existing/preflight hooks additive and documented, and make QR reuse conditional on the active login still being pending with a current QR.

## Reproduction Assessment

## Solution Assessment

No as currently proposed: moving existing QR reuse ahead of preflight is a plausible fix, but the best fix must preserve current main's QR state-sync behavior and only reuse pending active QR sessions.
