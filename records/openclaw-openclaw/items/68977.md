## Summary

Keep open. Current main still exposes an ephemeral, boolean-returning `enqueueSystemEvent` with only consecutive-text duplicate suppression, and the adjacent durable mechanisms do not provide the requested scoped system-event idempotency contract.

## What This Changes

Request adding scoped durable idempotency-key support to the plugin-facing `enqueueSystemEvent` API so plugin resume flows can retry without duplicate system events.

## Best Possible Solution

Define an additive plugin SDK/core contract that either extends `enqueueSystemEvent` with durable scoped idempotency without breaking existing callers, or explicitly introduces a new durable system-event helper. The design should document scope, retention, cleanup, payload mismatch behavior, heartbeat/session wake semantics, and include focused tests for cross-session keys, duplicate results, restart persistence, and post-consumption cleanup.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the missing API: callers cannot pass `idempotencyKey` to `enqueueSystemEvent`, and the implementation remains an in-memory queue returning a boolean; the crash window itself is inferred from that documented ephemeral contract rather than live-crash tested.

## Solution Assessment

Unclear. The requested direction is plausible, but the best maintainable fix needs maintainer API design because existing adjacent primitives solve related durable-context and internal-delivery problems without preserving the public system-event semantics requested here.
