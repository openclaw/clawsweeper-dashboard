## What This Changes

The PR suppresses `Conversation info` metadata for `heartbeat`, `cron-event`, and `exec-event` provider contexts in `buildInboundUserContextPrefix` and adds helper-level regression tests.

## Best Possible Solution

Land one canonical caller-owned heartbeat metadata suppression, with real heartbeat proof, that skips inbound user-context prefixing for synthetic heartbeat turns while preserving normal inbound metadata for real channel messages.

## Reproduction Assessment

Yes for source-level reproduction: current main builds synthetic heartbeat contexts with `OriginatingTo`, then passes the session context through `buildInboundUserContextPrefix`, which can emit `Conversation info.chat_id`. I did not live-run the downstream QQ/M3 warning path in this read-only review.

## Solution Assessment

No: suppressing synthetic heartbeat metadata is the right end state, but this branch puts provider-string policy inside the generic metadata helper instead of using the caller-owned `isHeartbeat` boundary. The safer fix is to skip inbound user-context prefix construction for heartbeat runs at `runPreparedReply`, or otherwise reuse the central synthetic-provider classifier.
