## Summary

Keep this PR open: the shared registry location is a good fit, but the current patch sources `/model` menu choices from all configured provider catalog rows instead of the existing `/model` visibility and allowlist contract, so existing allowlisted setups can see choices that the switch path rejects.

## What This Changes

The PR adds dynamic configured-model choices and an argument menu to the native `/model` command, plus focused command-registry tests and a catalog display-name type field.

## Best Possible Solution

Keep the shared registry-driven menu, but build `/model` choices from the same model visibility and allowlist contract used by the existing picker and switch path, with focused coverage for allowlisted subsets and provider wildcards.

## Reproduction Assessment

Yes, by source inspection: configure broader `models.providers` rows but a narrower `agents.defaults.models` allowlist; the PR menu lists provider catalog rows while the existing switch path enforces `allowedModelKeys`. I did not run a live Slack workspace reproduction.

## Solution Assessment

No, not as-is. The shared registry is the right layer for keeping Slack generic, but the choice provider should reuse the existing model visibility and allowlist contract instead of reading the full configured catalog directly.
