## Summary

Keep this PR open: the shared `/model` menu idea is useful and the supplied proof shows the basic behavior, but the latest head still introduces a Telegram compatibility blocker where valid long configured model choices are silently hidden from the new bare `/model` picker.

## What This Changes

The PR adds configured-model choices and argument-menu metadata for `/model`, then adapts Discord and Telegram native command handling and tests around that new menu surface.

## Best Possible Solution

Keep the shared `/model` choice registry for Slack/Discord, but route Telegram bare `/model` through the existing compact/paginated model picker or fall back when any generic callback cannot fit so the picker never hides configured choices.

## Reproduction Assessment

Yes. Source inspection shows over-limit Telegram callbacks are dropped before the menu is sent, and the added test asserts that a long configured model disappears from the `/model` reply markup.

## Solution Assessment

No. The registry change is a good shared input, but Telegram needs the existing compact/paginated model picker path or an all-or-fallback guard instead of a partial generic menu.
