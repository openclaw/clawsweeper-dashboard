## Summary

Current main still has the text-first final-mode TTS path, while Telegram already has the transport pieces needed for a captioned voice reply. This should stay open as a fresh, actionable delivery improvement rather than be swept as stale or already implemented.

## What This Changes

The issue requests deferring visible text delivery for final-mode TTS on caption-capable channels so Telegram can send one voice message with a caption instead of text followed by audio.

## Best Possible Solution

Add a narrow captioned-voice delivery capability, defer `settleVisibleText()` only for final-mode TTS when that capability is present, and keep the existing text flush when TTS synthesis or media delivery fails.

## Reproduction Assessment

Yes from source inspection: on current main and the latest release, ACP finalization settles visible text before final TTS is attempted, so Telegram cannot turn that first visible response into a voice caption. I did not run a live Telegram/TTS scenario in this read-only review.

## Solution Assessment

Yes, the proposed defer-then-fallback shape is the best fit if it is guarded by a captioned-voice channel capability. A Telegram-only special case or an unconditional defer would be more brittle.
