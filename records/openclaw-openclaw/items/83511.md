## Summary

Keep this issue open: current main and the latest release still settle visible ACP text before final-mode TTS, and the issue is already paired with an open closing PR that owns the implementation path.

## What This Changes

The issue requests deferring final-mode Telegram TTS text delivery so a final response can arrive as one captioned voice note instead of visible text followed by audio.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/83988 or a maintainer-approved replacement that defers text only for caption-capable final TTS, keeps synthesis-failure text fallback, and preserves non-text media delivery.

## Reproduction Assessment

Yes. Source inspection on current main shows visible text is settled before final-mode TTS in the ACP path, with the synthesized voice marked as a supplement after text delivery; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes, the best fix shape is a capability-gated deferral that only suppresses early text when the channel can carry the final TTS text as a voice caption and still falls back to text on synthesis or delivery failure.
