## Summary

Keep this open: current main and the latest release still settle visible ACP text before final-mode TTS, and the linked closing PR is open rather than merged. The useful next step is to review, fix, and land that PR or a narrow replacement, then close this issue after the implementation ships.

## What This Changes

The issue requests final-mode TTS delivery to avoid sending visible Telegram text before a captioned voice reply.

## Best Possible Solution

Land a guarded implementation that defers text settlement only when the channel can deliver captioned voice media, preserves the text fallback when synthesis or media send fails, and then close this issue from the merged PR.

## Reproduction Assessment

Yes from source inspection: current main and `v2026.5.18` settle visible ACP text before final-mode TTS synthesis, so Telegram cannot use the later voice send as the first captioned visible reply. I did not run a live Telegram/TTS scenario in this read-only review.

## Solution Assessment

Yes for the issue direction, with one important constraint: the defer-then-fallback design should be guarded by a captioned-voice channel capability or equivalent proven contract. The current linked PR is the right active path, but it still needs that guard and real Telegram proof.
