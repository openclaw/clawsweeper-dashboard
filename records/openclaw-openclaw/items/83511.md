## Summary

Keep this issue open: current main and the latest release still deliver visible text before final-mode TTS audio, and https://github.com/openclaw/openclaw/pull/83988 is the open closing PR that owns the fix path.

## What This Changes

The issue asks OpenClaw to defer Telegram-visible final-mode TTS text until audio synthesis can send a single voice note with caption, falling back to text if TTS fails.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/83988 or a maintainer-approved replacement that defers text only for caption-capable final TTS, preserves synthesis-failure text fallback, and does not drop non-text media blocks.

## Reproduction Assessment

Yes. Source inspection on current main shows ACP visible text is settled before final-mode TTS synthesis, and the generic path also sends visible block text before the accumulated TTS-only supplement; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes. The best fix shape is a capability-gated deferral that suppresses early text only when the channel can carry the final TTS text as a voice caption and still falls back to text on TTS failure.
