- `P2`: The issue affects a real user-visible Telegram delivery workflow but is bounded to final-mode TTS/channel behavior.
- `impact:message-loss`: The reported behavior creates duplicate and replacement churn in Telegram message delivery.

## Summary

Current main still has the text-first final-mode TTS path, and the issue is already paired with an open, mergeable PR that uses closing syntax for the same behavior. Keep this open until that PR lands or maintainers choose a replacement path.

## What This Changes

The issue requests deferring final-mode TTS visible text delivery for Telegram so a successful synthesis can send one voice message with a caption instead of a text message followed by audio.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83988 if it safely defers text only for caption-capable final TTS, preserves text fallback on synthesis failure, and preserves non-text media delivery.

## Reproduction Assessment

Yes. Source inspection shows that with Telegram plus `messages.tts.mode = "final"`, streamed block text is delivered before the later accumulated TTS voice payload, which matches the reported churn.

## Solution Assessment

Yes, a capability-gated deferral is the best fix shape: it changes the delivery ordering only for channels that can carry final TTS text as a voice caption and keeps existing text fallback behavior when synthesis fails.
