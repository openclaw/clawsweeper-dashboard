## Summary

Keep this issue open: current main and the latest release still flush visible text before final-mode TTS, and the issue is already paired with an open, clean closing PR that owns the implementation path.

## What This Changes

The issue asks OpenClaw to defer Telegram visible text during final-mode TTS so successful voice synthesis can deliver one voice note with a caption instead of a text message followed by audio.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83988, or an equivalent narrow fix, then close this issue once main defers text until final TTS succeeds and preserves text fallback on TTS failure.

## Reproduction Assessment

Yes, from source inspection. Current main calls `settleVisibleText()` before final TTS synthesis in ACP and sends a post-block TTS-only supplement in the non-ACP path; no live run was attempted in this read-only review.

## Solution Assessment

Yes, the best fix is to move the defer/combined-payload decision into shared dispatch with a channel capability, not a Telegram-layer delete-and-resend workaround. The open linked PR is the current candidate for that shape.
