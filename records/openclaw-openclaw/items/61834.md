## Summary

Keep open. Current main still lacks a QMD no-rerank config or request surface, and the existing direct CLI and mcporter/MCP query paths do not pass QMD's upstream no-rerank controls.

## What This Changes

Expose a no-rerank toggle for QMD query-mode memory search so OpenClaw can use hybrid QMD query results without the reranking step.

## Best Possible Solution

Add an opt-in `memory.qmd.noRerank` boolean for `searchMode: "query"`, propagate it to direct QMD CLI as `--no-rerank` and to the unified MCP query call as `rerank: false`, then update config docs, generated config help, tests, and changelog.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: the strict config schema rejects the proposed key, the `memory_search` tool schema has no rerank field, and both QMD query builders omit the upstream no-rerank controls.

## Solution Assessment

Yes for a config-level implementation. That path is the narrowest maintainable fix because it stays within the existing QMD backend contract while leaving per-request model-controlled rerank toggling for a separate product decision.
