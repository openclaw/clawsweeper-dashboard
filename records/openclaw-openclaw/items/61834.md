## Summary

Keep open. Current main and v2026.5.19 still lack a QMD no-rerank config field, memory_search request parameter, direct CLI flag propagation, and MCP rerank:false propagation, while upstream QMD already exposes both controls.

## What This Changes

Expose QMD's no-rerank toggle through OpenClaw's QMD memory query path, either as backend config or a memory_search request option.

## Best Possible Solution

Add an opt-in memory.qmd.noRerank boolean for QMD query mode, pass it as --no-rerank in direct CLI mode and rerank:false for unified MCP query calls, and treat request-level control as a separate maintainer decision.

## Reproduction Assessment

Yes, as a source-level capability gap. Current main lacks the proposed config field, the memory_search schema has no rerank parameter, and both direct and MCP QMD query paths omit upstream no-rerank controls.

## Solution Assessment

Yes for the config-level path, unclear for request-level control. A QMD-only config key is the narrowest maintainable surface, while a model-controlled memory_search argument needs maintainer product approval.
