## Summary

Keep open. Current main still exposes QMD `searchMode` and `searchTool`, but not a QMD `noRerank` config field or a request-level `memory_search.rerank` parameter; the direct CLI path builds `qmd query` without `--no-rerank`, and the MCP `qmd.query` path omits `rerank: false`. Upstream QMD v2.1.0 documents the requested CLI and MCP capability, so this is a narrow valid feature request rather than an obsolete report. 

## What This Changes

The issue asks OpenClaw to expose QMD query-mode rerank disabling, either as `memory.qmd.noRerank` or a request-level `memory_search.rerank` toggle.

## Best Possible Solution

Add an opt-in `memory.qmd.noRerank` boolean for configured `searchMode: "query"`, propagate it to direct CLI as `--no-rerank` and to the QMD MCP `query` tool as `rerank: false`, and document the QMD version expectation. Leave a per-request tool parameter for a separate product decision unless maintainers explicitly want model-controlled rerank toggling now.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main's strict config schemas, `memory_search` tool schema, CLI argument builder, and MCP call builder all lack the requested fields/arguments.

## Solution Assessment

Yes for a config-level fix. It is the narrowest maintainable path because it stays in the existing QMD backend configuration contract and avoids expanding the public `memory_search` tool schema before maintainers decide whether per-call agent control is desirable.
