## Summary

Keep this open: current main and the latest release still have the reported cron observability gap, while the only linked fix PR is closed unmerged and was explicitly closed for PR-limit capacity rather than because the issue was resolved.

## What This Changes

The issue asks for isolated cron agent runs with nonzero model usage to emit the same `model.usage` diagnostic event that interactive auto-reply runs already emit.

## Best Possible Solution

Add a narrow cron-side trusted `model.usage` emit with the existing auto-reply event semantics, assert the event shape in cron tests, and keep billed usage totals separate from context occupancy.

## Reproduction Assessment

Yes. Current-main source shows cron finalization has nonzero usage data but no `model.usage` emitter, while the diagnostic subscribers only see emitted `model.usage` events; I did not run a live OTel backend in this read-only review.

## Solution Assessment

Yes. Emitting from `finalizeCronRun` is the narrow owner-boundary fix because the needed usage, session, provider/model, cost, context, and duration facts are already in scope there; the repair should reuse the auto-reply contract rather than introduce a new event shape.
