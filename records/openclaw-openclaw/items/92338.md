## Summary

Keep open: current main and the latest release still record cron usage without emitting the canonical model.usage event, and the open linked PR is the implementation path rather than a reason to close before merge.

## What This Changes

Cron-isolated agent runs should emit model.usage diagnostic events so diagnostics-otel and OTel cost metrics include scheduled agent usage.

## Best Possible Solution

OpenClaw core should emit one canonical model.usage event from cron finalization when nonzero usage is persisted, matching auto-reply billing/context semantics and regression tests, then close the issue after the fix merges.

## Reproduction Assessment

Yes, source-level: current main records nonzero isolated-cron usage and the diagnostics-otel subscriber consumes model.usage, but cron never emits that event while auto-reply does. I did not run the live Prometheus/OTLP setup from the report.

## Solution Assessment

Yes, emitting from finalizeCronRun is the narrow owner-boundary fix because usage, model, provider, session, cost, and duration are finalized there. The event should reuse the established auto-reply semantics and avoid mixing billed usage.total with the session context snapshot.
