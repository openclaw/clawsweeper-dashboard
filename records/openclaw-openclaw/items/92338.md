## What This Changes

The issue reports that cron-isolated agent runs persist usage but do not emit `model.usage` diagnostic events, leaving scheduled model spend invisible to OTel subscribers.

## Best Possible Solution

Land a focused cron-owned diagnostic emission with regression tests for nonzero and zero usage, using the auto-reply event contract while keeping billed aggregate totals separate from context occupancy.

## Reproduction Assessment

Yes at the source level: current main computes and persists cron usage, while exact searches show no cron `model.usage` emitter and the sibling auto-reply path does emit it. I did not run a live OTel backend in this read-only review.

## Solution Assessment

Yes, with one caveat: emitting from the existing cron nonzero-usage finalization branch is the narrowest maintainable fix, but the event `usage.total` should use the billed aggregate rather than `deriveSessionTotalTokens`' context snapshot.
