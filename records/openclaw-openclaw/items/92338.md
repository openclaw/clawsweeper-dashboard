## Summary

The report is valid on both v2026.6.5 and current main: isolated cron runs persist normalized usage and cost but never publish the existing model usage diagnostic event, so diagnostics subscribers cannot observe scheduled spend. The prior broad attempt at https://github.com/openclaw/openclaw/pull/43552 closed unmerged and targeted a shared CLI path rather than the now-distinct cron completion boundary, leaving this focused core bug necessary.

## What This Changes

Make isolated cron agent runs emit model usage diagnostic events so scheduled token and cost telemetry reaches observability subscribers.

## Best Possible Solution

Emit one trusted model usage event from the canonical isolated-cron completion path after final provider, model, session identity, normalized usage, cost, and duration are known, and extend the existing cron diagnostics test for enabled, disabled, and zero-usage behavior.

## Reproduction Assessment

Yes. Current-main source provides a high-confidence reproduction: a successful isolated cron run with nonzero usage persists telemetry at `src/cron/isolated-agent/run.ts:985`, but that module has no model usage emitter, unlike the interactive sibling.

## Solution Assessment

Yes, with a narrower boundary than the prior proposal. The best fix is a focused emission in isolated cron completion using the established trusted event contract, not a new metrics API or a generic CLI wrapper that may miss cron-specific session adoption and accounting.
