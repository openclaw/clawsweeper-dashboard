## Summary

Keep open: current main and the latest release still record cron usage without emitting a trusted `model.usage` diagnostic event, and the only linked fix PR is closed unmerged.

## What This Changes

The issue asks OpenClaw to emit `model.usage` diagnostic events for cron-isolated agent runs so OTel subscribers see scheduled model spend.

## Best Possible Solution

Add the cron emit at the existing nonzero-usage accounting point, matching the auto-reply event shape while keeping session prompt-token snapshots separate from billed usage totals.

## Reproduction Assessment

Yes. Current-main source inspection shows nonzero cron usage is persisted and returned as telemetry without any cron `model.usage` diagnostic emit, while the sibling auto-reply path does emit that event.

## Solution Assessment

Yes. Emitting from the cron finalization/accounting point is the narrowest maintainable fix, provided the event uses the billed usage aggregate for `usage.total` and regression tests cover the zero-usage and disabled-diagnostics cases.
