## Summary

Keep open: current main still scopes the agentic memory flush to normal reply/compaction flow, while manual reset commands and daily/idle rollover replace or archive the outgoing session without a pre-reset agentic turn. Related PRs and issues cover pieces of the space, but none has merged or fully superseded this canonical reset-memory thread.

## What This Changes

The issue requests a silent agentic memory flush before `/new`, `/reset`, daily reset, and idle rollover destroy the outgoing session.

## Best Possible Solution

Keep this as the canonical reset-time curated-memory thread, then land a bounded implementation covering manual, daily, and idle reset with explicit guards, schema/docs/tests, timeout/cancellation, and session-memory preserved as a raw audit fallback.

## Reproduction Assessment

Yes for the current behavior, by source trace rather than a live run. Manual reset commands and daily/idle rollover replace or archive the outgoing session while the only production `runMemoryFlushIfNeeded` call remains in the normal reply runner.

## Solution Assessment

Unclear for the final implementation. Reusing the existing memory-flush primitive is plausible, but the maintainable solution needs explicit reset-reason coverage, config/default policy, timeout/cancellation, auditability, and tool/sandbox policy before landing.
