## Summary

Keep open: current main still rotates explicit and automatic reset sessions without running the agentic memory-flush path first. The slugged-file visibility subproblem is now partly addressed by startup-context loading, but the central reset-time agentic flush remains a product/security decision.

## What This Changes

Request to run an agentic memory-preservation flush before `/new`, `/reset`, daily reset, and idle reset destroy the current session.

## Best Possible Solution

Design a maintainer-approved reset-lifecycle memory preservation path that covers explicit and automatic rollovers while keeping latency, consent, sandbox, and plugin boundaries explicit.

## Reproduction Assessment

Yes from source inspection: reset rollover and memory-flush execution are separate current-main paths, so explicit or stale reset can rotate the session before any agentic pre-reset write is invoked. I did not run an end-to-end live reset because this review is read-only.

## Solution Assessment

Unclear: reusing the existing memory-flush turn is plausible, but blocking `/new`, running background cleanup, copying existing compaction summaries, or extending lifecycle hooks have different latency and security tradeoffs.
