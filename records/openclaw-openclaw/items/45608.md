- `impact:data-loss`: The report is about session context being silently lost across reset boundaries when no curated memory save occurs first.
- `impact:session-state`: The requested behavior directly changes session, transcript, memory, and cross-session context continuity.
- `impact:security`: The proposal adds silent tool-capable persistence and rehydration of conversation content, touching privacy and trust boundaries.

## Summary

Keep open: current main and the latest shipped tag still reset or archive outgoing sessions without an awaited, tool-capable agentic memory-flush turn against the outgoing session. The existing session-memory hook, startup memory prelude, and related open PRs are useful partial continuity paths, but this request still needs maintainer product and security decisions before implementation.

## What This Changes

Request a silent pre-reset agentic memory flush for manual `/new`/`/reset` and scheduled daily or idle session resets, matching the curated pre-compaction memory-save behavior.

## Best Possible Solution

Define one guarded reset-memory contract for manual, daily, and idle rollovers, including defaults/config, blocking versus background behavior, timeout and failure semantics, sandbox/tool policy, retention/redaction rules, docs, and focused tests coordinated with the related PRs.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main and v2026.5.19 reset or archive outgoing sessions without an agentic memory-flush turn against the outgoing session.

## Solution Assessment

Unclear as a final implementation. Reusing the existing memory-flush primitive is plausible, but maintainers still need to decide blocking versus background behavior, config shape, reset coverage, tool/sandbox policy, and retention semantics.
