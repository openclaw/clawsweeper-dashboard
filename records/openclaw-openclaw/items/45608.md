## Summary

Keep open: current main still has a real reset-memory lifecycle gap. The curated memory flush remains scoped to the normal reply/compaction path, while manual reset commands and stale daily/idle rollovers replace or archive the outgoing session without an awaited agentic flush.

## What This Changes

The issue requests a silent curated memory flush before manual `/new`/`/reset` and daily or idle session resets destroy the outgoing session.

## Best Possible Solution

Keep this as the canonical reset-time curated-memory thread, then land a bounded implementation covering manual, daily, and idle reset with explicit guards, schema/docs/tests, timeout/cancellation, and the existing session-memory hook preserved as a raw audit fallback.

## Reproduction Assessment

Yes for the current behavior, by source trace rather than a live run. Manual reset commands and daily/idle rollover can replace or archive the outgoing session while the only production `runMemoryFlushIfNeeded` call remains in the normal reply runner.

## Solution Assessment

Unclear for the final implementation. Reusing the existing memory-flush primitive is plausible, but the maintainable solution needs explicit reset-reason coverage, config/default policy, timeout/cancellation, auditability, and tool/sandbox policy before landing.
