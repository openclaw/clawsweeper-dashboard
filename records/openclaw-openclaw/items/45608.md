## Label Justifications

- `P2`: This is a normal-priority memory/session continuity improvement with meaningful but bounded blast radius.
- `impact:session-state`: The issue concerns reset and rollover paths that replace active session context without an agentic continuity handoff.
- `impact:data-loss`: Important learned context can be silently lost at reset unless the agent already wrote it to durable memory.
- `impact:security`: The proposed silent teardown turn would affect tool execution, sandboxing, auditability, and private memory handling.

## Summary

Keep open: current main still rotates or resets the outgoing session before any agentic memory flush; compaction uses the silent flush primitive, while reset paths only have mechanical/background hooks and startup-context loading afterward.

## What This Changes

Request to run a silent agentic memory flush before manual, daily, and idle session resets destroy or rotate the outgoing session.

## Best Possible Solution

Define one explicit reset-memory lifecycle contract covering manual, daily, and idle resets, with timeout, sandbox/tool policy, audit/error behavior, config/default decision, and focused upgrade tests.

## Reproduction Assessment

Yes, at source level. Reset/stale rollover through initSessionState rotates the session before any agentic memory flush, while the only production runMemoryFlushIfNeeded call remains in the post-init reply runner.

## Solution Assessment

Unclear until maintainers choose the lifecycle contract. Reusing the existing memory-flush primitive is plausible, but blocking vs background execution, config shape, and security/audit bounds need a product decision.
