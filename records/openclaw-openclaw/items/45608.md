## Summary

Keep open: current main has startup-context and session-memory mitigations, but it does not run a silent agentic memory flush before manual, daily, or idle reset; related open work overlaps but does not supersede this canonical design thread.

## What This Changes

Request a silent agentic memory flush before `/new`, `/reset`, daily reset, and related rollover paths destroy or archive the outgoing session.

## Best Possible Solution

Define a guarded reset-time memory-flush contract on the session lifecycle, covering manual, daily, and idle rollovers with explicit config/defaults, timeout/failure behavior, tool/sandbox policy, docs, and focused regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main archives or replaces reset sessions without invoking the existing agentic memory-flush turn on the outgoing session.

## Solution Assessment

Unclear as a final implementation. Reusing the existing memory-flush primitive is plausible, but reset ordering, blocking versus background behavior, config shape, and silent tool side effects need maintainer product/security design before code should land.

## Risks / Open Questions

- A silent teardown agent turn with tools can create hidden side effects unless timeout, cancellation, sandbox, approval, and logging policy are explicit.

- Blocking `/new`, `/reset`, or first-message-after-daily-reset on a model call could regress reset latency and reliability unless the ordering and failure fallback are designed.
- The open session-memory synthesis PR overlaps only part of the problem and is not merged, so it cannot be treated as a canonical fix.
