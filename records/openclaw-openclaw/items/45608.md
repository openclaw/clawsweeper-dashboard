## Label Justifications

- `P2`: This is a normal-priority memory-continuity improvement with real impact but nontrivial product and security design work.
- `impact:data-loss`: The issue is about session context being silently lost across reset boundaries when no curated memory save occurs first.
- `impact:session-state`: The requested behavior directly changes session, transcript, memory, and cross-session context continuity.
- `impact:security`: The proposal adds silent tool-capable persistence and rehydration of conversation content, which touches privacy and trust boundaries.

## Summary

Keep open: current main and the latest release still replace/archive outgoing sessions without a silent agentic memory-flush turn; startup memory loading and the bundled session-memory hook are useful partial mitigations, but they do not implement the requested reset-time curated save path.

## What This Changes

The issue requests a silent agentic memory flush before manual, daily, and idle session resets destroy or replace the outgoing session.

## Best Possible Solution

Define a guarded reset-time memory-flush contract for manual, daily, and idle rollovers, with explicit config/defaults, timeout and failure behavior, sandbox/tool policy, retention/redaction rules, docs, and focused regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main and v2026.5.12 replace/archive outgoing sessions without invoking an agentic memory-flush turn first.

## Solution Assessment

Unclear as a final implementation. Reusing the existing memory-flush primitive is plausible, but maintainers still need to choose blocking versus background behavior, config shape, reset coverage, tool/sandbox policy, and retention semantics.

## Risks / Open Questions

- A silent teardown agent turn with tool access can create hidden file or external-service side effects unless timeout, cancellation, sandbox, approval, and audit behavior are explicit.
- Reset-time memory persistence can carry private or prompt-hostile content into future sessions without a clear retention, redaction, and untrusted-note model.
- Blocking /new, /reset, or the first post-daily-reset message on an LLM call could regress reset latency and reliability unless ordering and fallback semantics are chosen by maintainers.
