## Summary

Keep open. Current main has reset-time preservation mitigations, including transcript archival, reset hooks, and startup loading for slugged session-memory artifacts, but it still does not run an outgoing agentic memory-flush turn before manual or stale session reset replaces the session.

## What This Changes

Run the existing silent agentic memory flush before `/new`, `/reset`, daily reset, and idle reset teardown so durable memories can be curated before sessions are replaced.

## Best Possible Solution

Keep this as the canonical maintainer design thread for reset-time curated memory persistence. The desired end state is a bounded reset-flush path that reuses the existing memory-flush capability against the outgoing session when policy allows, while keeping session-memory as a raw audit fallback.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: follow `/new` or `/reset` through `initSessionState` and `maybeHandleResetCommand`, or stale daily/idle rollover through `evaluateSessionFreshness`; the outgoing session is replaced or archived without a call to `runMemoryFlushIfNeeded`.

## Solution Assessment

Unclear because there is no patch yet. Reusing the existing memory-flush primitive looks plausible, but maintainers still need to choose default coverage, blocking versus background execution, timeout/failure behavior, and config shape.
