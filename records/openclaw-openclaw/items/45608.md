## Summary

Keep open: current main has reset archiving, session_end hooks, startup-context loading, and the mechanical session-memory hook, but it still does not run a silent agentic memory flush on the outgoing session before manual, daily, or idle reset.

## What This Changes

Request a reset-lifecycle memory flush that runs the existing silent agentic memory-write turn before `/new`, `/reset`, daily reset, and idle rollover destroy the outgoing session.

## Best Possible Solution

Define a guarded reset-time memory-flush contract covering manual, daily, and idle rollovers with explicit config/defaults, timeout and failure behavior, tool/sandbox policy, docs, and focused regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main archives or replaces reset sessions without invoking the existing agentic memory-flush turn on the outgoing session.

## Solution Assessment

Unclear as a final implementation. Reusing the existing memory-flush primitive is plausible, but reset ordering, blocking versus background behavior, config shape, and silent tool side effects need maintainer product/security design before code should land.

## Risks / Open Questions

- A silent teardown agent turn with tools can create hidden file or external-service side effects unless timeout, cancellation, sandbox, approval, and audit behavior are explicit.
- Reset-time memory persistence can carry private or prompt-hostile conversation content into future sessions without a clear retention, redaction, and trust-boundary model.
- Blocking `/new`, `/reset`, or first-message-after-daily-reset on an LLM call could regress reset latency and reliability unless ordering and fallback behavior are designed.
- The open session-memory synthesis PR overlaps only part of the problem and is not merged, so it cannot be treated as a superseding implementation.
