## Summary

Keep open: current main still has reset archiving, startup memory loading, and the mechanical session-memory hook, but it does not run a silent agentic memory-flush turn against the outgoing session before manual, daily, or idle reset. This remains a product/security-sensitive feature request rather than a stale or implemented item.

## What This Changes

Request a silent agentic memory flush before `/new`, `/reset`, daily reset, and related reset rollovers destroy the current session.

## Best Possible Solution

Define a guarded reset-time memory-flush contract for manual, daily, and idle rollovers with explicit config/defaults, timeout and failure behavior, sandbox/tool policy, retention/redaction rules, docs, and focused regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main replaces or archives reset sessions without invoking the agentic memory-flush turn on the outgoing session.

## Solution Assessment

Unclear as a final implementation. Reusing the existing memory-flush primitive is plausible, but reset ordering, blocking versus background behavior, config shape, tool access, and retention policy need maintainer product/security design before code lands.
