## Label Justifications

- `P2`: This is a normal-priority session continuity and memory preservation feature with meaningful but bounded blast radius.
- `impact:session-state`: The issue concerns reset and rollover paths that replace active session context without an agentic continuity handoff.
- `impact:data-loss`: Important learned context can be silently lost at reset unless the agent already wrote it to durable memory.
- `impact:security`: The proposed silent teardown turn would affect tool execution, sandboxing, auditability, and private memory handling.

## Summary

Keep open: current main still has compaction-time agentic memory flush and reset-time mechanical/background capture, but no `/new`, `/reset`, daily, or idle rollover path gives the outgoing session a silent tool-enabled agent turn before teardown. Related cleanup already closed narrower duplicates into this issue as the canonical reset-memory design thread.

## What This Changes

Request a pre-reset agentic memory flush before `/new`, `/reset`, daily reset, and related reset paths destroy or archive the outgoing session.

## Best Possible Solution

Define one explicit reset-memory lifecycle policy, likely at the session lifecycle plus memory-plugin boundary, covering manual, daily, idle, and gateway resets with bounded execution, delivery suppression, auditability, and focused fresh-install plus upgrade tests.

## Reproduction Assessment

Yes at source level: seeding a stale daily or idle session, or an explicit reset, reaches `initSessionState`/gateway reset paths that mint or commit a new session and archive the previous transcript without invoking `runMemoryFlushIfNeeded` on the outgoing session.

## Solution Assessment

Unclear until maintainers choose the lifecycle contract. Reusing the existing memory-flush primitive is plausible, but blocking versus background execution, config shape, tool policy, and security/audit bounds need product review first.
