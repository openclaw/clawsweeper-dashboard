## Summary

Keep open. Current main has useful reset-time preservation mitigations, including transcript archival, session lifecycle hooks, and startup loading for slugged session-memory artifacts, but it still does not run an outgoing agentic memory-flush turn before manual, daily, or idle reset replaces the session.

## What This Changes

Run a silent agentic memory flush before `/new`, `/reset`, daily reset, and idle reset teardown so durable memories can be curated before sessions are replaced.

## Best Possible Solution

Keep this as the canonical design thread for reset-time curated memory persistence, then implement a bounded reset-flush path that reuses the existing memory-flush capability against the outgoing session while preserving session-memory as the raw audit fallback.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to trace `/new` or `/reset` through `maybeHandleResetCommand`, or daily/idle staleness through `initSessionState`; the outgoing session is archived/replaced without any reset-time call to `runMemoryFlushIfNeeded`.

## Solution Assessment

Unclear because there is no acceptable patch yet. Reusing the existing memory-flush primitive is plausible, but maintainers still need to choose the policy boundary for reset reasons, blocking/background execution, timeout, failure visibility, and security controls.
