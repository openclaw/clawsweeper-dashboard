## Summary

Keep open: current main and the latest release still replace/archive reset sessions and provide mechanical reset-memory capture, but they do not run a silent agentic memory-flush turn against the outgoing session before manual, daily, or idle reset. This remains a clear product/security-sensitive feature request, not stale or implemented.

## What This Changes

The issue requests a silent agentic memory flush before `/new`, `/reset`, daily reset, and idle rollover destroy the outgoing session.

## Best Possible Solution

Define a guarded reset-time memory-flush contract for manual, daily, and idle rollovers with explicit config/defaults, timeout and failure behavior, sandbox/tool policy, retention/redaction rules, docs, and focused regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main replaces or archives reset sessions without invoking an agentic memory-flush turn on the outgoing session.

## Solution Assessment

Unclear as a final implementation. Reusing the existing memory-flush primitive is plausible, but reset ordering, blocking versus background behavior, config shape, tool access, and retention policy need maintainer product/security design before code lands.

## Risks / Open Questions

- A silent teardown agent turn with tool access can create hidden file or external-service side effects unless timeout, cancellation, sandbox, approval, and audit behavior are explicit.
- Reset-time memory persistence can carry private or prompt-hostile content into future sessions without a clear retention, redaction, and trust-boundary model.
- Blocking `/new`, `/reset`, or the first post-daily-reset message on an LLM call could regress reset latency and reliability without a maintainer-approved ordering and fallback contract.
- Open overlapping items and PRs cover parts of the space, but none has landed as the canonical implementation.
