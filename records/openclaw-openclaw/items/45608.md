## Summary

Keep open: current main and v2026.5.19 still roll reset/stale sessions to a new session id and archive the outgoing transcript without an agentic pre-reset memory flush. The existing session-memory hook and startup-memory loader are useful mitigations, but they do not implement the requested curated, silent teardown turn and the remaining design has product and security implications.

## What This Changes

The issue requests a silent agentic memory flush before manual, daily, and related session resets destroy or archive the outgoing session.

## Best Possible Solution

Define a guarded reset-time memory-flush contract for manual, daily, and idle rollovers, with explicit config/defaults, timeout and failure behavior, sandbox/tool policy, retention/redaction rules, docs, and focused regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main and v2026.5.19 replace/archive outgoing sessions without invoking an agentic memory-flush turn first.

## Solution Assessment

Unclear as a final implementation. Reusing the existing memory-flush primitive is plausible, but maintainers still need to decide blocking versus background behavior, config shape, reset coverage, tool/sandbox policy, and retention semantics.
