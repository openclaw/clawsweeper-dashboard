## Summary

Keep open: current main has reset-time archival, fire-and-forget lifecycle hooks, startup hydration for recent memory files, and a mechanical session-memory fallback, but it still does not run an outgoing agentic memory-flush turn before manual, daily, or idle reset replaces the session.

## What This Changes

The issue requests that OpenClaw run the existing silent agentic memory-flush mechanism before `/new`, `/reset`, daily reset, and idle reset destroy the outgoing session.

## Best Possible Solution

Keep this as the canonical reset-time curated-memory thread, then land a bounded implementation that covers manual, daily, and idle reset with explicit guards, schema/docs/tests, and session-memory preserved as a raw audit fallback.

## Reproduction Assessment

Yes. A source-level trace shows reset and rollover replace/archive the outgoing session through initSessionState and reset command handling, while the only production runMemoryFlushIfNeeded call remains in the normal reply runner.

## Solution Assessment

Unclear for the final implementation. Reusing the existing memory-flush primitive is plausible, but the maintainable fix needs explicit reset-reason coverage, config/defaults, timeout/cancellation, auditability, and tool/sandbox policy before landing.
