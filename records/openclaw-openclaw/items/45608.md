## Label Justifications

- `P2`: This is a normal-priority session continuity and memory preservation feature with meaningful but bounded blast radius.
- `impact:data-loss`: Important learned context can be silently lost at reset unless the agent already wrote it to durable memory.
- `impact:session-state`: The issue concerns reset and rollover paths that replace active session context without an agentic continuity handoff.
- `impact:security`: The proposed silent teardown turn would affect tool execution, sandboxing, auditability, and private memory handling.

## Summary

Keep open: current main and v2026.6.6 still run the agentic memory flush from the normal reply/compaction path, while reset and rollover paths replace/archive the outgoing session and rely on hooks or raw background capture. The remaining work is a security-sensitive session-memory lifecycle decision, not conservative cleanup.

## What This Changes

Request to run a silent, tool-enabled memory flush before manual and automatic session resets destroy the outgoing session.

## Best Possible Solution

Define one reset-memory lifecycle contract at the session plus memory-plugin boundary covering manual, daily, idle, and gateway resets with bounded execution, auditability, startup hydration, and focused fresh-install plus upgrade tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: seed a stale daily or idle session, or send an explicit reset, and current initSessionState/gateway reset paths mint or commit a new session and archive the previous transcript without invoking runMemoryFlushIfNeeded on the outgoing session.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the lifecycle contract; reusing the existing memory-flush primitive is plausible, but blocking/background behavior, config shape, tool policy, and security bounds need product review first.
