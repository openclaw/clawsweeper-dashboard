## Label Justifications

- `P2`: This is a normal-priority session continuity and memory preservation feature with meaningful but bounded blast radius.
- `impact:data-loss`: Important learned context can be silently lost at reset unless the agent already wrote it to durable memory.
- `impact:session-state`: The issue concerns reset and rollover paths that replace active session context without an agentic continuity handoff.
- `impact:security`: The requested silent teardown turn would affect tool execution, sandboxing, auditability, and private memory handling.

## Summary

Keep open: current main and v2026.6.6 still do not run the agentic memory flush on reset or stale rollover, and related duplicates have been closed into this canonical design thread. The remaining work is a product and security decision around silent teardown-time tool execution, not conservative cleanup.

## What This Changes

The issue requests an agentic memory flush before manual and automatic session resets so curated memory survives session teardown.

## Best Possible Solution

Define one maintainer-approved reset-memory lifecycle contract covering manual, daily, idle, and gateway resets with bounded execution, auditability, tool/sandbox policy, startup hydration, and focused fresh-install plus upgrade tests.

## Reproduction Assessment

Yes, at source level: seed a stale daily or idle session or send an explicit reset, and current initSessionState creates a new session and archives the previous transcript without invoking runMemoryFlushIfNeeded on the outgoing session. I did not run a live reset scenario because this cleanup review is read-only.

## Solution Assessment

Unclear: reusing the existing memory-flush primitive is plausible, but it is not proven as the best fix until maintainers choose blocking/background behavior, config shape, tool policy, auditability, and reset ordering. A lifecycle-level contract is safer than a one-branch call-site patch.

## Risks / Open Questions

- A silent teardown-time agent turn can run file or external-service tools during reset unless timeout, cancellation, sandbox, allowlist, and audit behavior are explicit.
- Blocking reset on a model call can add visible latency or strand reset paths, while background execution can race transcript archival and new-session startup.
- The request overlaps raw-tail continuity and flush-then-reset proposals, so maintainers should choose one lifecycle contract before adding another reset-specific path.
