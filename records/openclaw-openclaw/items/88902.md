## Summary

Keep open: current main and the latest release still choose the initial `/btw` harness before canonical OpenAI runtime resolution, so stale `openai-codex` route state can miss the Codex side-question hook. The open draft PR at https://github.com/openclaw/openclaw/pull/93183 now owns the fix, and this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that `/btw` can route a Codex OAuth side question through generic OpenAI Responses after `/new`, producing a missing `api.responses.write` scope error instead of using the Codex app-server side-thread transport.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/93183 if its runtime-reselect patch is correct, or keep this issue as the canonical bug until an equivalent focused fix and regression test land.

## Reproduction Assessment

Yes for source reproduction: current main can miss the Codex side-question hook when stale `openai-codex` provider state enters `/btw`, then continue toward direct provider fallback. I did not run a live Telegram OAuth flow in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to resolve the canonical runtime model before committing to the side-question route, then reselect the harness using the canonical provider/model. The open draft PR appears to target that shape, so the issue should remain open while that PR is reviewed.
