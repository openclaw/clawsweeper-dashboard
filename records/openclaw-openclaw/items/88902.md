## Summary

Keep open: current main still selects the `/btw` harness from the incoming provider/model before resolving the canonical OpenAI runtime model, and the only focused fix for the reported stale `openai-codex` route state was closed unmerged. The latest release shows the same ordering, so there is not enough evidence to close this as implemented.

## What This Changes

The issue reports that `/btw` after `/new` can route Codex OAuth side questions through public OpenAI Responses instead of the Codex side-thread transport.

## Best Possible Solution

Reintroduce the focused reselect-after-runtime-resolution fix for `/btw`, add regression coverage for legacy `openai-codex` route state resolving to canonical `openai/gpt-5.5`, and keep doctor as the long-term migration path for stale config.

## Reproduction Assessment

Yes for source reproduction: current main can miss the Codex side-question hook when stale `openai-codex` provider state enters `/btw`, then continue toward direct provider fallback. I did not run a live Telegram OAuth flow in this read-only review.

## Solution Assessment

Yes, the narrow best fix is to resolve the runtime model once, reselect the harness using the canonical provider/model, and only then decide between the Codex side-question hook and direct fallback. That keeps the existing native Codex side-thread boundary and avoids adding a long-lived legacy runtime shim.
