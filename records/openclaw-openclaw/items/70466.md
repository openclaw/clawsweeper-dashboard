## Summary

Keep open. The PR is aimed at a real gateway restart failure, but it is incomplete against current main: update.run restarts now take an earlier supervised branch that still exits 0, so the main scenario described by the PR remains broken for systemd units using Restart=on-failure.

## What This Changes

This PR splits the generic spawned and supervised gateway SIGUSR1 restart branch so spawned handoffs exit 0 while supervised restarts exit 1, and updates the launchd supervised restart test to expect exit code 1.

## Best Possible Solution

Update this PR or replace it with a narrow patch that makes both generic and update.run supervised systemd restarts exit non-zero, keeps detached spawned handoffs exiting 0, preserves the launchd delay semantics, and adds focused tests for non-launchd supervised restart behavior.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to drive runGatewayLoop with restartReason "update.run", systemd supervisor markers, and respawnGatewayProcessForUpdate returning mode "supervised"; current main reaches exitProcess(0). The generic SIGUSR1 supervised branch can be reproduced the same way through restartGatewayProcessWithFreshPid.

## Solution Assessment

No, not as currently proposed. The non-zero supervised-exit direction is the right narrow fix, but changing only the generic branch misses the current update.run branch, so a safer implementation should handle both branches or share one supervised-exit helper.
