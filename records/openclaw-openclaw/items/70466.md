## Summary

Keep open. The PR fixes only the generic supervised SIGUSR1 branch, but the stated update.run path still reaches the update-specific supervised branch on current main and exits 0, so the reported systemd Restart=on-failure failure remains possible.

## What This Changes

The PR splits generic spawned and supervised gateway restart handling so spawned restarts exit 0 while supervised restarts exit 1, and updates the launchd supervised restart test to expect exit code 1.

## Best Possible Solution

Update this PR or replace it with a narrow patch that gives supervised systemd update.run and generic SIGUSR1 restarts non-zero exit semantics, preserves spawned handoff and launchd delay behavior, and locks the behavior with focused non-launchd/systemd tests plus a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: schedule an update.run restart under Linux systemd markers, let respawnGatewayProcessForUpdate return supervised, and current main exits 0 from the update-specific branch; systemd Restart=on-failure treats that as a clean stop.

## Solution Assessment

No. The proposed change is directionally right for the generic branch, but it is not the best complete fix because it leaves the update.run branch unchanged and does not add the systemd/non-launchd assertions needed to protect the reported path.
