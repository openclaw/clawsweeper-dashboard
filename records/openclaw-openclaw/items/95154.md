- `P1`: The PR targets a user-facing WebChat restart path where an active dashboard session can be left failed and blank without a recovery notice.
- `merge-risk: 🚨 compatibility`: The patch can change an existing session's persisted transcript target during recovery, affecting what custom or forked sessions display after upgrade.

## Summary

Keep open: this is a focused candidate fix for the WebChat restart-recovery notice gap, but the current head still has a blocking session transcript compatibility defect because it can rebind valid custom or forked session files to canonical history during failed recovery.

## What This Changes

Adds a restart-recovery fallback that writes the unresumable notice into the transcript for non-deliverable WebChat/internal sessions, plus an explicit transcript append target and regression tests.

## Best Possible Solution

Keep the WebChat transcript fallback, but choose the same visible transcript target that chat.history would read for valid non-stale sessionFile entries and use canonical history only for missing or stale generated paths.

## Reproduction Assessment

Yes, source-reproducible: current main rejects WebChat as a deliverable channel and has no transcript fallback. The PR-introduced sessionFile rebind also follows from the canonical write plus touchSessionEntry behavior and reader resolution.

## Solution Assessment

No. The transcript fallback is the right layer for the missing WebChat notice, but always writing canonical history is not the safest fix because it can diverge from chat.history's visible transcript target.
