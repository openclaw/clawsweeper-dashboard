## Summary

Keep open. The PR addresses a real WebChat restart-recovery notice gap, but the current head bypasses the session-rebound guard when writing to an explicit transcript path, which can mis-associate a newly rebound session with an old recovery transcript in a startup-recovery race.

## What This Changes

The PR writes the existing unresumable restart-recovery notice into the canonical session transcript when normal channel delivery is unavailable, extends the transcript append helper with an explicit file parameter, and adds WebChat/stale-sessionFile regression tests.

## Best Possible Solution

Keep the transcript-visible notice fallback, but preserve expectedSessionId/session-rebound protection for explicit transcript writes before merge; then let maintainers decide whether the broader linked WebChat restart-loss issue needs follow-up beyond the notice.

## Reproduction Assessment

Yes by source inspection: current main treats WebChat as internal/non-deliverable, so the unresumable notice send path returns false and the recovery path marks the session failed without a transcript-visible notice. I did not run a live WebChat plus managed gateway restart reproduction in this read-only review.

## Solution Assessment

No as submitted: writing the existing notice to the transcript is the right narrow fix shape, but the explicit-file append path must keep the existing session-rebound guard. After that repair, the PR is an acceptable mitigation while broader lost-prompt or restart-deferral behavior stays with the linked issue if maintainers want more than a notice.
