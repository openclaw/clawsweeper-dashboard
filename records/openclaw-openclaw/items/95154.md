## Label Justifications

- `P1`: The PR targets a user-facing WebChat restart path where an active dashboard session can be left failed and blank without a recovery notice.
- `merge-risk: 🚨 compatibility`: The patch can change an existing session's persisted transcript target during recovery, affecting what custom or forked sessions display after upgrade.
- `merge-risk: 🚨 session-state`: The patch writes restart-recovery notice state into persisted session transcripts and can update session metadata.
- `merge-risk: 🚨 message-delivery`: The patch changes fallback behavior for recovery notices when normal outbound delivery is unavailable.

## Summary

Keep open: the WebChat transcript fallback is needed, but the branch should not merge as-is because it can rebind a valid non-stale transcript target to canonical history during failed restart recovery.

## What This Changes

The PR adds a WebChat/internal restart-recovery transcript notice fallback, an explicit transcript append target parameter, and focused restart-recovery tests.

## Best Possible Solution

Keep the WebChat transcript fallback, but choose the same visible transcript target that `chat.history` would read for valid non-stale `sessionFile` entries and use canonical history only for missing or stale generated paths.

## Reproduction Assessment

Yes, source-reproducible. Current main rejects WebChat as a deliverable channel and has no transcript fallback after an unresumable recovery, while the PR-introduced target rebind follows from the canonical write plus `touchSessionEntry` behavior and reader ordering.

## Solution Assessment

No. The transcript fallback is the right layer for the missing WebChat notice, but always writing canonical history is not the safest fix because it can diverge from `chat.history`'s visible transcript target.
