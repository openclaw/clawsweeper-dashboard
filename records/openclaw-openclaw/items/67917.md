## Summary

Keep this open: current main still hard-blocks OpenAI Codex browser OAuth on certificate-class preflight failures, while the merged runtime transport work only addresses a narrower bearer-injection path.

## What This Changes

The issue requests that OpenAI Codex OAuth precheck and runtime requests honor a valid custom CA/TLS setup instead of failing with certificate and fetch errors.

## Best Possible Solution

Define one owner-approved custom-CA/proxy/TLS story for OpenAI Codex OAuth and runtime, then adjust the preflight so valid custom-CA setups are not false hard-blocked while preserving actionable diagnostics and regression coverage.

## Reproduction Assessment

Partially yes. The OAuth blocker is reproducible from current source and unit coverage that mocks `UNABLE_TO_GET_ISSUER_CERT_LOCALLY`; the exact macOS custom-CA runtime fetch failure still needs targeted or live proof.

## Solution Assessment

No. The current hard-blocking preflight is not the best outcome for a valid custom-CA browser OAuth setup, and the existing runtime transport fix only solves a narrower bearer-injection failure.
