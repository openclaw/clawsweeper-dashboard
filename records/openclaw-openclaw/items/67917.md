## Summary

Keep this open: current main still hard-blocks OpenAI Codex browser OAuth on `tls-cert` preflight failures, and the merged runtime transport work only covers a narrower OAuth bearer/payload path rather than proving the reported custom-CA TLS/runtime failure is fixed.

## What This Changes

The issue reports OpenAI Codex OAuth TLS precheck failures and runtime fetch failures on macOS despite a valid custom CA setup.

## Best Possible Solution

Define one owner-approved custom-CA/proxy/TLS behavior for OpenAI Codex OAuth and runtime, then make browser OAuth avoid false hard blocks for valid custom-CA setups while preserving clear diagnostics and regression coverage.

## Reproduction Assessment

Yes for the OAuth blocker from source: the current helper rejects before upstream OAuth when `runOpenAIOAuthTlsPreflight()` returns a `tls-cert` result, and the unit test preserves that behavior. The exact macOS custom-CA runtime fetch failure is still not live-reproduced, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

No. The current hard-blocking preflight is not the best outcome for a valid custom-CA browser OAuth setup, and #75111 only proves a narrower runtime OAuth bearer/payload fix; the safer path is an owner-approved advisory or custom-CA-aware preflight plus targeted runtime proof.
