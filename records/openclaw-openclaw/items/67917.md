## Summary

Keep this open: current main still hard-blocks OpenAI Codex browser OAuth on certificate-class preflight failures, even though the changelog records that this probe should be advisory, and the merged runtime transport fix does not prove the custom-CA/runtime-fetch part is resolved.

## What This Changes

The issue reports OpenAI Codex OAuth TLS precheck failures and runtime `fetch failed` errors on macOS with a custom CA setup.

## Best Possible Solution

Restore advisory TLS-preflight behavior for OpenAI Codex browser OAuth while preserving clear diagnostics, then validate whether the reporter's macOS custom-CA runtime fetch failure still exists after that narrower fix.

## Reproduction Assessment

Yes for the OAuth blocker from source: current main classifies `UNABLE_TO_GET_ISSUER_CERT_LOCALLY` as `tls-cert` and throws before browser OAuth starts. No live reproduction was established for the separate macOS runtime `fetch failed` path.

## Solution Assessment

No, the current hard-blocking preflight is not the best solution because the changelog records the intended behavior as advisory. The narrow maintainable fix is to warn with diagnostics but continue into the real OAuth flow, without changing TLS verification itself.
