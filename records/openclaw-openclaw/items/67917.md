## Summary

Keep this open. Current main still hard-blocks normal OpenAI Codex browser OAuth when the TLS preflight classifies a failure as certificate-related, and the in-tree regression test preserves that behavior. The merged #75111 transport fix is useful for the runtime fetch side, but it does not address the reported OAuth precheck blocker or prove the macOS custom-CA path end to end.

## What This Changes

The issue requests fixing OpenAI Codex OAuth/TLS handling so NODE_EXTRA_CA_CERTS-backed macOS setups can complete browser OAuth and run openai-codex requests without fetch failures.

## Best Possible Solution

Keep the issue open until OpenAI Codex browser OAuth and runtime transport have one coherent custom-CA/proxy/env story. The likely end state is an owner-approved advisory preflight or effective custom-CA validation path, plus targeted coverage for certificate-class preflight behavior and Codex runtime transport under custom CA/proxy conditions.

## Reproduction Assessment

Partially yes. The OAuth blocker is reproducible from current unit coverage that mocks UNABLE_TO_GET_ISSUER_CERT_LOCALLY and verifies the upstream OAuth helper is never called; the exact macOS custom-CA runtime fetch failure still needs live or targeted transport reproduction.

## Solution Assessment

No. Current main is not the best solution because it hard-blocks browser OAuth on a speculative TLS preflight despite prior changelog guidance that this probe should be advisory; #75111 is a useful runtime transport fix but not a complete fix for this report.
