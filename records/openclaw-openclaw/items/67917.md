## Summary

Keep this issue open. Current main still hard-blocks OpenAI Codex browser OAuth when the TLS preflight classifies the failure as certificate-related, and the in-tree regression test preserves that exact failure mode; the merged runtime transport fix for #75111 is useful but only addresses the bearer-injection/runtime side, not the reported OAuth precheck blocker.

## What This Changes

The issue asks OpenClaw to stop failing OpenAI Codex browser OAuth and runtime requests on macOS custom-CA setups where direct Node TLS checks work.

## Best Possible Solution

Keep the issue open until OpenAI Codex browser OAuth and runtime transport share one owner-approved custom-CA/proxy/env story. The likely end state is an advisory preflight with actionable diagnostics or a targeted custom-CA validation path, backed by regression coverage for certificate-class preflight behavior and Codex runtime dispatch.

## Reproduction Assessment

Partially yes. The OAuth hard block is reproducible from current source and the unit test that mocks `UNABLE_TO_GET_ISSUER_CERT_LOCALLY` and verifies OAuth never starts; the exact macOS custom-CA runtime fetch failure still needs live or targeted transport proof.

## Solution Assessment

No. The current hard-blocking preflight is not the best solution for the reported valid-custom-CA browser OAuth path, and #75111 only fixes a narrower runtime bearer-injection issue rather than the precheck blocker.
