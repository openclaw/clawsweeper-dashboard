## Summary

Keep open. Current main still keeps RTL RLI/PDI isolation inside the TUI formatter, while shared assistant-visible, user-facing, gateway live projection, and outbound delivery paths pass Hebrew/Arabic text without a shared bidi-isolation stage.

## What This Changes

The issue requests applying existing RTL bidi isolation to non-TUI gateway and outbound replies so Hebrew/Arabic punctuation renders correctly in external channels.

## Best Possible Solution

Add a shared RTL isolation helper under src/shared/text, reuse it from the TUI formatter, and apply it late to user-visible outbound and live projection text with focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by passing Hebrew or Arabic text through sanitizeAssistantVisibleText, sanitizeUserFacingText, sanitizeForPlainText, or the deliverOutboundPayloads text path and observing that current main returns or sends raw text without RLI/PDI wrapping.

## Solution Assessment

Yes. Extracting the existing TUI helper into a shared text module and applying it at the user-visible outbound boundary is the narrowest maintainable fix; channel-specific workarounds would duplicate behavior and miss shared paths.
