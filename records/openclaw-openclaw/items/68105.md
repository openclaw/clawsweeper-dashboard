## Summary

Keep open. Current main still confines RTL RLI/PDI isolation to the TUI formatter, while the shared assistant-visible/user-facing sanitizers, gateway live projection, and outbound delivery paths pass Hebrew/Arabic text without a shared bidi-isolation stage.

## What This Changes

Apply the existing TUI RTL bidi isolation behavior to non-TUI gateway/outbound reply text so Hebrew/Arabic punctuation renders correctly.

## Best Possible Solution

Ship a shared RTL isolation helper used by TUI and channel-visible final/projection text, with regression tests for mixed LTR/RTL lines and existing bidi controls.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by passing Hebrew or Arabic text through sanitizeAssistantVisibleText, sanitizeUserFacingText, sanitizeForPlainText, or deliverOutboundPayloads and observing that current main returns or sends raw text without RLI/PDI wrapping.

## Solution Assessment

Yes. Extracting the TUI helper into src/shared/text and applying it late at the user-visible outbound boundary is narrower and more maintainable than channel-specific workarounds.
