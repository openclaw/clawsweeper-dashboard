## Summary

Keep open. Current main still keeps RTL RLI/PDI isolation private to the TUI formatter, while the shared assistant-visible sanitizer, plain-text outbound sanitizer, gateway live projection, and final outbound delivery path still pass Hebrew/Arabic text without a shared bidi-isolation stage. The related TUI and Telegram-preview reports are useful precedent, but they do not implement this non-TUI outbound boundary fix.

## What This Changes

The issue requests shared RTL bidi isolation for non-TUI outbound replies so Hebrew/Arabic punctuation renders correctly in channel clients.

## Best Possible Solution

Add a small shared RTL isolation helper under src/shared/text, reuse it from the TUI formatter, and apply it at the canonical user-visible outbound/display boundary before formatted text or text chunks are sent. Keep the helper out of internal prompt/transcript sanitization unless a separate contract explicitly requires it.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: pass Hebrew or Arabic text through sanitizeAssistantVisibleText, sanitizeForPlainText, or deliverOutboundPayloads and current main forwards the raw text to channel delivery without U+2067/U+2069 wrapping; live client rendering would only confirm the visual symptom.

## Solution Assessment

Yes. Extracting the existing TUI helper into a shared text module and applying it late in outbound/user-visible projection is the narrowest maintainable fix, provided tests lock no-double-wrap behavior and protect internal transcript paths.
