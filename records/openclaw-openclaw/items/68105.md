## Summary

Keep open. Current main still keeps the RTL RLI/PDI isolation helper private to the TUI formatter, while shared assistant-visible text, PI user-facing text, plain-text outbound sanitization, gateway live projection, and final outbound delivery continue to pass Hebrew/Arabic reply text without a shared bidi-isolation stage. The linked TUI and Telegram-preview items are useful precedent, and the merged WhatsApp sanitizer PR is adjacent, but none implements this non-TUI outbound boundary fix.

## What This Changes

Request to move the existing TUI RTL isolation behavior into a shared outbound/user-visible sanitizer so Hebrew/Arabic replies sent through non-TUI channels get Unicode direction isolation marks.

## Best Possible Solution

Add a small shared RTL isolation helper under src/shared/text, reuse it from the TUI formatter, and apply it at the canonical user-visible outbound/display boundary after internal scaffolding sanitization and before channel chunking/sending or live projection. Add focused regression tests for shared helper behavior, TUI reuse, assistant-visible/user-facing sanitizers, outbound delivery/plain-text paths, and no double wrapping when bidi controls are already present.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: pass Hebrew or Arabic text through sanitizeAssistantVisibleText, sanitizeUserFacingText, sanitizeForPlainText, or deliverOutboundPayloads and current main has no RTL isolation call before returning or sending the text. Live channel testing would only confirm the visual punctuation symptom in specific clients.

## Solution Assessment

Yes. Extracting the existing TUI helper into a shared text module and applying it late in outbound/user-visible projection is the narrowest maintainable fix; a broader channel-specific workaround would duplicate logic and still miss shared delivery paths.
