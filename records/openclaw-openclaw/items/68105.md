## Summary

Keep open. Current main still confines RLI/PDI RTL isolation to the TUI formatter, while shared assistant-visible text, user-facing sanitization, plain-text channel sanitization, and outbound delivery send paths do not apply a shared bidi-isolation stage.

## What This Changes

Request to reuse the existing TUI RTL bidi isolation behavior at shared/gateway/outbound reply boundaries so Hebrew/Arabic punctuation renders correctly in non-TUI channels.

## Best Possible Solution

Add a shared RTL isolation helper, reuse it from the TUI formatter, and apply it at the channel-visible outbound/final text boundary with regression coverage for mixed LTR/RTL lines and existing bidi controls.

## Reproduction Assessment

Yes, at source level. Passing Hebrew or Arabic text through sanitizeAssistantVisibleText, sanitizeUserFacingText, sanitizeForPlainText, or outbound delivery planning on current main has no RLI/PDI wrapping path, while the TUI formatter does wrap the same class of text.

## Solution Assessment

Yes. Extracting the existing TUI helper into shared text code and applying it late at the user-visible outbound boundary is narrower than channel-specific workarounds and avoids inventing a new config surface.
