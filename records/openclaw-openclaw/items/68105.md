## Summary

Keep open. Current main and the latest release still keep RLI/PDI wrapping inside the TUI formatter, while the shared assistant-visible, user-facing, plain-text outbound, gateway live projection, and delivery paths still return or send RTL text without a shared bidi-isolation stage. The closed TUI report at https://github.com/openclaw/openclaw/issues/21936 provides the implementation to extract; https://github.com/openclaw/openclaw/issues/32852 is Telegram-draft-specific, and https://github.com/openclaw/openclaw/pull/71830 was adjacent sanitizer work rather than a fix for this boundary.

## What This Changes

The issue requests shared RTL bidi isolation for Hebrew/Arabic outbound replies beyond the existing TUI-only sanitizer.

## Best Possible Solution

Move the existing RTL isolation helper into shared text code, reuse it from TUI, and apply it once at the user-visible outbound/final projection boundary with focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main wraps RTL text only through the TUI sanitizeRenderableText path, while shared user-visible/outbound paths return or send text without RLI/PDI isolation. I did not run a live channel client visual repro.

## Solution Assessment

Yes. Extracting the existing TUI helper into shared text code and applying it late at the user-visible outbound boundary is narrower than channel-specific workarounds and does not need a new config or product policy surface.
