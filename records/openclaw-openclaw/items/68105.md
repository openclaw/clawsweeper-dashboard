## Summary

Keep open. Current main still keeps RTL RLI/PDI isolation in the TUI formatter only; shared assistant-visible sanitization, user-facing text cleanup, gateway live projection, plain-text channel sanitization, and outbound delivery still have no shared bidi-isolation stage.

## What This Changes

The issue requests applying the existing TUI RTL bidi isolation behavior to gateway and outbound channel replies so Hebrew/Arabic punctuation renders correctly in non-TUI clients.

## Best Possible Solution

Add a shared RTL isolation helper under shared text code, reuse it from the TUI formatter, and apply it at the user-visible outbound/final projection boundary with focused regression coverage.

## Reproduction Assessment

Yes, at source level. Passing Hebrew or Arabic text through the current shared assistant-visible, user-facing, plain-text channel, gateway live projection, or outbound delivery paths has no RLI/PDI wrapping stage, while the TUI sanitizer wraps the same class of text.

## Solution Assessment

Yes. Extracting the existing TUI helper into shared text code and applying it late at the user-visible outbound boundary is narrower than channel-specific workarounds and avoids adding a config or product policy surface.
