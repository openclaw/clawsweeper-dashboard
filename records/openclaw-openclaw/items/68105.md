## Summary

Keep open. Current main still confines Unicode RTL isolation to the TUI rendering formatter, while shared user-visible sanitizers, gateway live projection, plain-text outbound sanitization, and the core outbound delivery path still pass Hebrew/Arabic text without a shared bidi-isolation stage. The related TUI fix and Telegram preview report are useful precedent but do not implement this outbound boundary behavior.

## What This Changes

The issue requests extracting the existing TUI RTL bidi isolation behavior into a shared helper and applying it to non-TUI user-visible outbound replies so Hebrew/Arabic punctuation renders correctly.

## Best Possible Solution

Add a shared RTL isolation helper under `src/shared/text`, reuse it from the TUI formatter, and apply it at the canonical user-visible outbound boundary before formatted/chunked channel sends. Cover no-double-wrap behavior and include gateway live projection only if that surface is intentionally under the same user-visible contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: Hebrew or Arabic text can pass through the current shared sanitizers and outbound delivery path to `sendFormattedText` or `sendTextChunks` without RLI/PDI wrapping, while the TUI formatter wraps the same class of text.

## Solution Assessment

Yes. Extracting the existing TUI helper and applying it at final user-visible delivery/projection boundaries is the narrowest maintainable fix; channel-specific hacks would leave other outbound surfaces inconsistent.
