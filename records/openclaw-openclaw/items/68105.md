## Summary

Keep open. Current main still confines RTL RLI/PDI isolation to the TUI formatter, while shared assistant-visible/user-facing sanitizers, gateway live chat projection, plain-text outbound sanitization, and outbound delivery still pass Hebrew/Arabic text without a shared bidi-isolation stage; the earlier TUI fix at https://github.com/openclaw/openclaw/issues/21936 is useful source work but not a core outbound fix, and no open paired PR was found.

## What This Changes

Request shared RTL bidi isolation for Hebrew/Arabic assistant replies sent through gateway/outbound non-TUI channels so punctuation renders on the correct side.

## Best Possible Solution

Move the RTL isolation helper into shared text code, reuse it from TUI, and apply it once at the user-visible outbound/final projection boundary with focused coverage for existing bidi controls, chunks, and media captions.

## Reproduction Assessment

Yes, at source level. Current main wraps RTL text only in sanitizeRenderableText, while assistant-visible, user-facing, gateway projection, plain-text outbound, and delivery paths return or send text without RLI/PDI isolation.

## Solution Assessment

Yes. Extracting the existing TUI helper into shared text code and applying it late at the user-visible outbound boundary is narrower than channel-specific workarounds and does not require a new config or product policy surface.
