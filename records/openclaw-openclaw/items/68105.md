## Summary

Keep open. Current main still keeps RTL RLI/PDI isolation inside the TUI formatter, while shared assistant-visible text, user-facing sanitization, plain-text outbound sanitization, gateway live projection, and outbound delivery still pass RTL message bodies without a shared bidi-isolation stage.

## What This Changes

Add shared RTL bidi isolation for non-TUI outbound replies so Hebrew/Arabic punctuation renders correctly in channel clients.

## Best Possible Solution

Move the existing RTL isolation helper into shared text code, reuse it from TUI, and apply it once at the user-visible outbound/final projection boundary with focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main wraps RTL text only through the TUI sanitizeRenderableText path, while shared user-visible/outbound paths return or send text without RLI/PDI isolation. I did not run a live channel client visual repro.

## Solution Assessment

Yes. Extracting the existing TUI helper into shared text code and applying it late at the user-visible outbound boundary is narrower than channel-specific workarounds and does not need a new config or product policy surface.
