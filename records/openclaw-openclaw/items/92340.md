## Summary

This PR should stay open because its plugin-local reuse of Feishu DM ingress is directionally sound, but the new handler clears dedupe state after every dispatch failure, unlike established Feishu synthetic-event handlers, so a terminal failure followed by platform redelivery can repeat the invite turn or meeting-join attempt. The PR also needs maintainer confirmation that subscribing to the Feishu VC event is sufficient consent for automatic join turns, and its claimed live result is not backed by inspectable evidence on the current head.

## What This Changes

Adds Feishu VC meeting-invite event handling that creates an inbound meeting-join turn, preserves inviter DM routing, adds focused tests, and documents the required event subscription.

## Best Possible Solution

Keep the feature plugin-local and on the canonical Feishu DM ingress path, but align failure handling with the existing retryable synthetic-event contract, prove duplicate redelivery behavior, attach inspectable current-head live evidence, obtain an explicit decision on the automatic-join opt-in boundary, and retain only one canonical PR.

## Reproduction Assessment

Yes for the introduced defect: source inspection shows every rejected dispatch clears the dedupe marker, so the identical event can be claimed again. A live Feishu redelivery was not performed during this read-only review.

## Solution Assessment

No as written. Reusing the plugin-local DM ingress, binding, session, and reply dispatcher is the right architectural direction, but retry classification must match sibling handlers and the automatic-join consent model needs explicit maintainer approval.
