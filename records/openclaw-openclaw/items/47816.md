## Summary

Keep open. Current main still lacks a VirtualBox/Vagrant install surface, and the PR head has unresolved functional and security blockers in the installer, VM wrapper, status handling, and gateway-access docs, so it is not a conservative close or merge candidate.

## What This Changes

The PR adds a VirtualBox/Vagrant install guide plus `scripts/virtualbox` installer, Vagrantfile, CLI wrapper, and guest provisioning scripts for running OpenClaw in a Linux VM.

## Best Possible Solution

If OpenClaw wants official VirtualBox support, land it as a maintained installer plus documented gateway/networking model after fixing the shell defects, security issues, docs navigation, and VM validation. If maintainers do not want to own that surface, keep core docs focused on supported install paths and redirect this to an explicitly unofficial community guide.

## Reproduction Assessment

Yes. The blocking PR defects have high-confidence static and focused shell reproductions: piped Bash leaves `BASH_SOURCE[0]` empty, the first-start success path hits `set -u` on an unset variable, and sourcing the generated `.env` evaluates password text.

## Solution Assessment

No. The proposed direction may be useful, but this branch is not the best mergeable form because official install docs should not publish an installer, LAN access path, or host status command that fails or evaluates user-controlled shell syntax.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Make the piped installer install the checked-in scripts:** `scripts/virtualbox/install.sh:149`
  - body: The guide's primary install command pipes this script into `bash`, but in that mode `BASH_SOURCE[0]` is empty, `LOCAL_DIR` stays empty, and this branch aborts. Users following the documented quick path cannot install `openclaw-vm`.
  - confidence: 0.96
- **[P1] Initialize vagrant_status before checking it:** `scripts/virtualbox/openclaw-vm:320-323`
  - body: On the first-create success path, `vagrant up | track_provision_progress` never assigns `vagrant_status`. With `set -u`, the later `[[ -n "$vagrant_status" ]]` exits with an unbound variable immediately after successful provisioning.
  - confidence: 0.95
- **[P1] Stop evaluating the VM password file in status:** `scripts/virtualbox/openclaw-vm:396`
  - body: `write_vm_env` stores the user-entered password verbatim, and `status` later `source`s that file. A password containing shell syntax such as command substitution will execute on the host when the user checks status; parse the known keys instead of sourcing the file.
  - confidence: 0.96
- **[P2] Report vagrant startup failures:** `scripts/virtualbox/openclaw-vm:300-307`
  - body: Wrapping `vagrant up --no-provision` in `|| true` makes `PIPESTATUS[0]` report success for the group even when Vagrant failed, so `start` can still print that the VM is running after a boot failure.
  - confidence: 0.88
- **[P2] Forward ssh subcommand arguments:** `scripts/virtualbox/openclaw-vm:570`
  - body: `cmd_ssh` supports arguments, but the dispatcher drops them here. The documented `openclaw-vm ssh -c "openclaw gateway"` example will open an interactive SSH session instead of running the command.
  - confidence: 0.97
- **[P2] Propagate reprovision failures:** `scripts/virtualbox/openclaw-vm:442-445`
  - body: `provision` also masks `vagrant provision` failures with `|| true` and then prints success. Failed package setup, firewall changes, or user provisioning can be hidden from users and automation.
  - confidence: 0.86
- **[P2] Configure gateway access before promising host WebChat:** `docs/install/vm.md:134`
  - body: The guide tells users to run the default `openclaw gateway` inside the VM and open `http://localhost:18789` on the host, but OpenClaw defaults to loopback and non-loopback Control UI access requires explicit auth and allowed origins. Add the needed bind/tunnel/auth instructions before advertising host browser access.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR adds executable setup and network exposure, and the current diff has concrete host-side command-evaluation and LAN SSH concerns.

Concerns:

- **[high] Raw .env sourcing executes password text:** `scripts/virtualbox/openclaw-vm:396`
  - body: The VM password is written verbatim to `.env`, then `openclaw-vm status` sources that file. A normal password containing shell syntax can execute commands on the host, so the status path should parse expected keys without shell evaluation.
  - confidence: 0.96
- **[medium] SSH is forwarded on all host interfaces:** `scripts/virtualbox/Vagrantfile:50`
  - body: The Vagrantfile exposes guest SSH on host `0.0.0.0:2222` while the docs present LAN password access as a normal workflow. That may be acceptable only after maintainer security review and clear defaults/threat-model wording.
  - confidence: 0.78
- **[medium] Provisioning executes external install scripts:** `scripts/virtualbox/scripts/provision.sh:17`
  - body: The VM provisioner pipes NodeSource and Ollama installer scripts into a root shell. This adds a supply-chain and repeatability surface that should be explicitly accepted, pinned, or documented before becoming official OpenClaw setup.
  - confidence: 0.78
