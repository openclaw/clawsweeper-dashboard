## Summary

Keep open. Current main still has no VirtualBox/Vagrant install surface, and this PR is a real feature proposal, but the branch adds official docs plus executable host/guest setup scripts with source-proven install, runtime, and security blockers and no real setup proof.

## What This Changes

The branch adds a VirtualBox/Vagrant install guide plus host installer, VM wrapper, Vagrantfile, and provisioning scripts under `docs/install/vm.md` and `scripts/virtualbox/`.

## Best Possible Solution

Either rework this into owner-approved official VirtualBox support with fixed installer, provisioning, gateway access, navigation, security handling, and clean-machine proof, or keep the VM helper as an external/community tool until OpenClaw chooses to own it.

## Reproduction Assessment

Yes. Source inspection of the PR head gives high-confidence reproduction paths for the piped installer failure, unbound-variable exits, raw env sourcing, dropped SSH arguments, masked provisioning/config import failures, and gateway bind mismatch.

## Solution Assessment

No. The direction may be useful, but bundling official VM scripts and docs is not the best mergeable shape until the ownership/security model is decided and the implementation matches current installer and gateway contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Make the documented piped installer work:** `scripts/virtualbox/install.sh:149`
  - body: The guide tells users to run this script with `curl ... | bash`, but in that mode `BASH_SOURCE[0]` is not a local checkout containing `openclaw-vm` and `Vagrantfile`, so the installer reaches this failure path instead of installing the VM files.
  - confidence: 0.96
- **[P2] Enter the cloned checkout before local install:** `docs/install/vm.md:60`
  - body: The fallback path clones `openclaw/openclaw` and then immediately runs `cd scripts/virtualbox` from the parent directory. That directory only exists after entering the cloned `openclaw` checkout, so the documented fallback install fails.
  - confidence: 0.94
- **[P1] Initialize vagrant_status before testing it:** `scripts/virtualbox/openclaw-vm:323`
  - body: On the first-create path, a successful `vagrant up | track_provision_progress` leaves `vagrant_status` unset. Because the wrapper uses `set -u`, the later expansion exits with an unbound-variable error after successful provisioning.
  - confidence: 0.96
- **[P2] Initialize halt_status before testing it:** `scripts/virtualbox/openclaw-vm:357`
  - body: A successful `vagrant halt` leaves `halt_status` unset, then this check expands it under `set -u`. A normal successful stop can therefore fail before printing success.
  - confidence: 0.91
- **[P1] Parse VM config without sourcing password text:** `scripts/virtualbox/openclaw-vm:396`
  - body: The setup flow writes `VM_PASSWORD` into `.env`, and `status` later sources that file. A password containing shell syntax or unmatched quotes can execute on the host or break parsing, so status should parse known keys without shell evaluation.
  - confidence: 0.97
- **[P2] Report config import failures:** `scripts/virtualbox/openclaw-vm:187-188`
  - body: `push_openclaw_config` swallows every `vagrant ssh` or write failure and always returns success. `openclaw-vm start --config` can therefore tell users their OpenClaw config was imported even when the VM never received it.
  - confidence: 0.88
- **[P2] Propagate reprovision failures:** `scripts/virtualbox/openclaw-vm:442-445`
  - body: `provision` masks `vagrant provision` failures with `|| true` and always prints success. Failed package setup, firewall changes, or SSH provisioning can be hidden from users and automation.
  - confidence: 0.9
- **[P2] Forward ssh subcommand arguments:** `scripts/virtualbox/openclaw-vm:570`
  - body: `cmd_ssh` accepts arguments, but the dispatcher drops them. The documented `openclaw-vm ssh -c "openclaw gateway"` example opens an interactive SSH session instead of running the requested command.
  - confidence: 0.97
- **[P2] Configure gateway access before promising WebChat:** `docs/install/vm.md:134`
  - body: The guide starts the default gateway inside the VM and then tells users to open host `localhost:18789`. Current gateway defaults bind to loopback, so Vagrant forwarding will not reach it without an explicit non-loopback bind or SSH tunnel guidance.
  - confidence: 0.86
- **[P3] Register the install page in navigation:** `docs/install/vm.md:1`
  - body: The PR adds a new install page but does not register it in `docs/docs.json`. The page would be published without an Install navigation entry, so readers would need to guess the URL.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces official host/guest setup scripts, LAN SSH exposure, and raw host-side env sourcing, so it needs security review before merge.

Concerns:

- **[high] Raw .env sourcing executes password text:** `scripts/virtualbox/openclaw-vm:396`
  - body: `VM_PASSWORD` is written from user input into `.env`, then `openclaw-vm status` sources that file. A password containing shell syntax can execute on the host, so status should parse known keys without shell evaluation.
  - confidence: 0.97
- **[medium] SSH is forwarded on all host interfaces:** `scripts/virtualbox/Vagrantfile:50`
  - body: The Vagrantfile exposes guest SSH on host port 2222 with `host_ip: "0.0.0.0"`, and provisioning can enable password auth for the `vagrant` user. Official scripts need explicit owner-approved defaults and threat-model guidance for that LAN exposure.
  - confidence: 0.88
