## Summary

This PR should stay open because it adds an official VirtualBox/Vagrant setup surface with host/guest execution scripts, but the branch still has merge-blocking install/runtime defects, security-sensitive password parsing, and no end-to-end VirtualBox proof.

## What This Changes

The PR adds a VirtualBox install guide plus `scripts/virtualbox/**` installer, wrapper CLI, Vagrantfile, and provisioning scripts for running OpenClaw in a Vagrant-managed VM.

## Best Possible Solution

Decide whether OpenClaw should own official VirtualBox support; if yes, land hardened scripts and docs with safe install, correct gateway access, no shell-evaluated secrets, navigation, and clean-machine proof.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection and focused Bash probes reproduce the piped-installer, unbound-variable, masked-failure, password-sourcing, and SSH-argument problems. No full VirtualBox clean-machine lifecycle was run, and the PR body says that path was not verified.

## Solution Assessment

No. The direction may be useful, but this branch is not a mergeable solution until maintainers decide to own official VirtualBox support and the implementation matches current install, gateway, docs, and security contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Make the documented piped installer work:** `scripts/virtualbox/install.sh:149`
  - body: The guide tells users to run this script with `curl ... | bash`, but in that mode `BASH_SOURCE[0]` does not point at a local checkout containing `openclaw-vm` and `Vagrantfile`, so the installer reaches this failure path instead of installing the VM files.
  - confidence: 0.97
- **[P2] Enter the cloned checkout before local install:** `docs/install/vm.md:60`
  - body: The fallback path clones `openclaw/openclaw` and then immediately runs `cd scripts/virtualbox` from the parent directory. That directory only exists after entering the cloned `openclaw` checkout, so the documented fallback install fails.
  - confidence: 0.95
- **[P2] Preserve vagrant resume failures:** `scripts/virtualbox/openclaw-vm:300`
  - body: The resume path wraps `vagrant up --no-provision` in `|| true` before reading `PIPESTATUS`, so a failed resume is reported as success. Users can be told the VM is ready even when Vagrant failed to boot it.
  - confidence: 0.88
- **[P1] Initialize vagrant_status before testing it:** `scripts/virtualbox/openclaw-vm:323`
  - body: On the first-create path, a successful `vagrant up | track_provision_progress` never assigns `vagrant_status`. Because the wrapper uses `set -u`, the later expansion exits with an unbound-variable error after successful provisioning.
  - confidence: 0.96
- **[P2] Initialize halt_status before testing it:** `scripts/virtualbox/openclaw-vm:357`
  - body: A successful `vagrant halt` can leave `halt_status` unset, then this check expands it under `set -u`. A normal successful stop can therefore fail before printing success.
  - confidence: 0.91
- **[P1] Parse VM config without sourcing password text:** `scripts/virtualbox/openclaw-vm:396`
  - body: The setup flow writes `VM_PASSWORD` into `.env`, and `status` later sources that file. A password containing shell syntax or unmatched quotes can execute on the host or break parsing, so status should parse known keys without shell evaluation.
  - confidence: 0.98
- **[P2] Report config import failures:** `scripts/virtualbox/openclaw-vm:187-188`
  - body: `push_openclaw_config` swallows every `vagrant ssh` or write failure and always returns success. `openclaw-vm start --config` can therefore tell users their OpenClaw config was imported even when the VM never received it.
  - confidence: 0.9
- **[P2] Propagate reprovision failures:** `scripts/virtualbox/openclaw-vm:442`
  - body: `provision` masks `vagrant provision` failures with `|| true` and always prints success. Failed package setup, firewall changes, or SSH provisioning can be hidden from users and automation.
  - confidence: 0.91
- **[P2] Forward ssh subcommand arguments:** `scripts/virtualbox/openclaw-vm:570`
  - body: The headless-mode docs use `openclaw-vm ssh -c "openclaw gateway"`, but the dispatcher calls `cmd_ssh` without `"$@"`. The wrapper opens an interactive SSH session instead of running the requested command.
  - confidence: 0.97
- **[P2] Configure gateway access before promising WebChat:** `docs/install/vm.md:134`
  - body: The guide starts the default gateway inside the VM and then tells users to open host `localhost:18789`. Current gateway docs say the default bind is loopback, so the VM instructions need a bind change or SSH tunnel guidance before promising host browser access.
  - confidence: 0.88
- **[P3] Register the install page in navigation:** `docs/install/vm.md:1`
  - body: The PR adds a new install page but does not register it in `docs/docs.json`. The page would be published without an Install navigation entry, so readers would need to guess the URL.
  - confidence: 0.9
- **[P3] Fail closed in the documented curl commands:** `docs/install/vm.md:53`
  - body: The install examples use `curl -sSL ... | bash` without `-f`, so HTTP error bodies can be piped to the shell. The examples should use the repo's safer `curl -fsSL` pattern.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff adds security-sensitive host/guest setup scripts, LAN SSH exposure, and host-side env sourcing that need attention before merge.

Concerns:

- **[high] Raw .env sourcing executes password text:** `scripts/virtualbox/openclaw-vm:396`
  - body: `VM_PASSWORD` is written from user input into `.env`, then `openclaw-vm status` sources that file. A password containing shell syntax can execute on the host, so the wrapper should parse known keys without shell evaluation.
  - confidence: 0.98
- **[medium] SSH is forwarded on all host interfaces:** `scripts/virtualbox/Vagrantfile:50`
  - body: The Vagrantfile exposes guest SSH on host port 2222 with `host_ip: "0.0.0.0"`, and provisioning can enable password auth for the `vagrant` user. Official scripts need owner-approved defaults and threat-model guidance for that LAN exposure.
  - confidence: 0.89
- **[medium] Provisioning runs remote install scripts:** `scripts/virtualbox/scripts/provision.sh:17`
  - body: The provisioning scripts pipe NodeSource and OpenClaw installer scripts into a shell inside the VM. That may be acceptable for an owned setup path, but it expands the supply-chain surface and needs explicit proof and docs around trust boundaries.
  - confidence: 0.86
