## Summary

Keep open. Current main does not include the proposed VirtualBox/Vagrant install surface, but this PR is not merge-ready because the installer and wrapper have blocking source-reproducible failures and the new SSH/provisioning surface needs maintainer security/product review.

## What This Changes

The branch adds a VirtualBox/Vagrant install guide plus `scripts/virtualbox` installer, Vagrantfile, CLI wrapper, and guest provisioning scripts for running OpenClaw in a Linux VM.

## Best Possible Solution

If OpenClaw wants official VirtualBox support, land it as an owned installer with working install paths, safe env parsing, failure propagation, documented gateway networking/auth, nav/changelog updates, and VM validation; otherwise keep this as an unofficial community guide outside core.

## Reproduction Assessment

Yes. Source inspection of PR head gives high-confidence reproduction paths for the blocking defects: the documented piped installer has no local companion directory, first-create success reads an unset `vagrant_status`, and `status` sources a password-bearing `.env` file.

## Solution Assessment

No. The proposed direction may be useful, but this is not the best mergeable form until the official ownership/security model and the script defects are addressed; the safer path is an OpenClaw-owned installer with auditable dependencies and docs that match Gateway bind/auth behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Make the piped installer install the checked-in scripts:** `scripts/virtualbox/install.sh:149`
  - body: The documented quick path pipes this script into `bash`, but in that mode `BASH_SOURCE[0]` is not a local `scripts/virtualbox` file. `LOCAL_DIR` stays empty and this branch aborts, so users following the docs cannot install `openclaw-vm`.
  - confidence: 0.96
- **[P2] Enter the cloned repository before local install:** `docs/install/vm.md:60`
  - body: The fallback instructions clone `openclaw/openclaw` and then run `cd scripts/virtualbox` from the parent directory. That path does not exist until the user first enters the `openclaw` checkout, so the documented fallback install also fails.
  - confidence: 0.94
- **[P1] Initialize vagrant_status before checking it:** `scripts/virtualbox/openclaw-vm:320-323`
  - body: On the first-create success path, `vagrant up | track_provision_progress` never assigns `vagrant_status`. Because the wrapper runs with `set -u`, the later `[[ -n "$vagrant_status" ]]` exits with an unbound variable immediately after provisioning succeeds.
  - confidence: 0.95
- **[P1] Stop evaluating the VM password file in status:** `scripts/virtualbox/openclaw-vm:396`
  - body: `write_vm_env` stores the password verbatim, and `status` later `source`s that file. A password containing shell syntax or quotes can execute commands or break parsing on the host when the user checks VM status.
  - confidence: 0.96
- **[P2] Report vagrant startup failures:** `scripts/virtualbox/openclaw-vm:300-307`
  - body: Wrapping `vagrant up --no-provision` in `|| true` makes the pipeline's first command report success even when Vagrant failed. `start` can then continue and print that the VM is running after a boot failure.
  - confidence: 0.88
- **[P2] Forward ssh subcommand arguments:** `scripts/virtualbox/openclaw-vm:570`
  - body: `cmd_ssh` accepts arguments, but the dispatcher drops them here. The documented `openclaw-vm ssh -c "openclaw gateway"` example opens an interactive SSH session instead of running the command.
  - confidence: 0.97
- **[P2] Propagate reprovision failures:** `scripts/virtualbox/openclaw-vm:442-445`
  - body: `provision` masks `vagrant provision` failures with `|| true` and then prints success. Failed package setup, firewall changes, or SSH provisioning can be hidden from users and automation.
  - confidence: 0.86
- **[P2] Configure gateway access before promising host WebChat:** `docs/install/vm.md:134`
  - body: The guide tells users to run the default `openclaw gateway` inside the VM and browse `http://localhost:18789` on the host. Current Gateway docs say the default bind is loopback and VM-style forwarding needs an explicit LAN/custom bind plus auth/origin configuration or a tunnel.
  - confidence: 0.82
- **[P3] Register the guide in install navigation:** `docs/install/vm.md:1`
  - body: Adding a new docs page without updating `docs/docs.json` leaves it out of the Install navigation. Readers would need to guess the URL even after the page is merged.
  - confidence: 0.9
- **[P3] Add a changelog entry for the VM installer:** `docs/install/vm.md:1`
  - body: This PR adds a user-facing `openclaw-vm` installer and lifecycle command surface, but it does not update `CHANGELOG.md`. Repo policy expects user-facing feature changes to be recorded before merge.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR adds executable host/guest setup, LAN SSH exposure, and raw env sourcing in an official install path.

Concerns:

- **[high] Raw .env sourcing executes password text:** `scripts/virtualbox/openclaw-vm:396`
  - body: `VM_PASSWORD` is written verbatim to `.env`, then `openclaw-vm status` sources that file. A password containing shell syntax can execute on the host, so the status path should parse known keys without shell evaluation.
  - confidence: 0.96
- **[medium] SSH is forwarded on all host interfaces:** `scripts/virtualbox/Vagrantfile:50`
  - body: The Vagrantfile exposes guest SSH on `0.0.0.0:2222`, while the docs present LAN password access as a normal workflow. That may be acceptable only after maintainer security review and clear default/threat-model wording.
  - confidence: 0.78
- **[medium] Provisioning executes external install scripts as root:** `scripts/virtualbox/scripts/provision.sh:17`
  - body: The VM provisioner pipes NodeSource and Ollama installer scripts into root shells. This adds supply-chain and repeatability risk that should be explicitly accepted, pinned, or documented before becoming an official setup path.
  - confidence: 0.78
- **[low] Piped installer commands omit curl fail-fast:** `docs/install/vm.md:53`
  - body: The docs use `curl -sSL ... | bash` for executable install commands. Without `-f`, HTTP error responses can be piped into the shell instead of failing cleanly.
  - confidence: 0.82
