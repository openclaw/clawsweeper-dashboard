## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Make the documented piped installer work:** `scripts/virtualbox/install.sh:149`
  - body: The guide tells users to run this script with `curl ... | bash`, but in that mode `BASH_SOURCE[0]` is empty and this branch always fails instead of installing the VM files.
  - confidence: 0.97
- **[P2] Enter the cloned checkout before local install:** `docs/install/vm.md:60`
  - body: The fallback path clones `openclaw/openclaw` and then immediately runs `cd scripts/virtualbox` from the parent directory. That directory only exists after entering the cloned `openclaw` checkout.
  - confidence: 0.95
- **[P2] Preserve Vagrant resume failures:** `scripts/virtualbox/openclaw-vm:300`
  - body: The resume path wraps `vagrant up --no-provision` in `|| true` before reading `PIPESTATUS`, so a failed resume is reported as success and users can be told the VM is ready when it did not boot.
  - confidence: 0.9
- **[P2] Report config import failures:** `scripts/virtualbox/openclaw-vm:187-188`
  - body: `push_openclaw_config` swallows every `vagrant ssh` or write failure and always returns success. `openclaw-vm start --config` can therefore claim the config was imported when the VM never received it.
  - confidence: 0.9
- **[P1] Parse VM config without sourcing password text:** `scripts/virtualbox/openclaw-vm:396`
  - body: The setup flow writes `VM_PASSWORD` into `.env`, and `status` later sources that file. A password containing shell syntax or unmatched quotes can execute on the host or break parsing.
  - confidence: 0.98
- **[P2] Propagate reprovision failures:** `scripts/virtualbox/openclaw-vm:442`
  - body: `provision` masks `vagrant provision` failures with `|| true` and always prints success, hiding failed package setup, firewall changes, or SSH provisioning from users and automation.
  - confidence: 0.91
- **[P2] Forward ssh subcommand arguments:** `scripts/virtualbox/openclaw-vm:570`
  - body: The headless-mode docs use `openclaw-vm ssh -c "openclaw gateway"`, but the dispatcher calls `cmd_ssh` without `"$@"`, so it opens an interactive SSH session instead of running the requested command.
  - confidence: 0.97
- **[P2] Configure gateway access before promising WebChat:** `docs/install/vm.md:134`
  - body: The guide starts the default gateway inside the VM and then tells users to open host `localhost:18789`. Current Gateway docs say the default bind is loopback, so the VM instructions need a bind change or SSH tunnel before promising host browser access.
  - confidence: 0.88
- **[P3] Register the install page in navigation:** `docs/install/vm.md:1`
  - body: The PR adds a new install page but does not register it in `docs/docs.json`. The page would be published without an Install navigation entry, so readers would need to guess the URL.
  - confidence: 0.9
- **[P3] Fail closed in curl install examples:** `docs/install/vm.md:53`
  - body: The install examples use `curl -sSL ... | bash` without `-f`, so HTTP error bodies can be piped to the shell. Use the repo's safer `curl -fsSL` pattern.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Needs attention because the PR adds host/guest setup scripts, all-interface SSH forwarding, and host-side shell sourcing of user password config.

Concerns:

- **[high] Raw .env sourcing executes password text:** `scripts/virtualbox/openclaw-vm:396`
  - body: `VM_PASSWORD` is written from user input into `.env`, then `openclaw-vm status` sources that file. Password text containing shell syntax can execute on the host, so the wrapper should parse known keys without shell evaluation.
  - confidence: 0.98
- **[medium] SSH is forwarded on all host interfaces:** `scripts/virtualbox/Vagrantfile:50`
  - body: The Vagrantfile exposes guest SSH on host port 2222 with `host_ip: "0.0.0.0"`, and provisioning can enable password auth for the `vagrant` user. Official scripts need owner-approved defaults and threat-model guidance for that LAN exposure.
  - confidence: 0.89
