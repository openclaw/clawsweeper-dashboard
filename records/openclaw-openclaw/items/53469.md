## Summary

Keep open. Current main only partially satisfies the request: SearXNG web_search can use trusted private HTTP targets and the shared SSRF guard supports private-network policy flags, but web_fetch still exposes only the narrower RFC2544/IPv6-ULA exceptions and Gateway OpenResponses input_file/input_image URL fetches still hardcode private-network access off.

## What This Changes

Request private/local network access support for web tools and Gateway URL fetch surfaces, including self-hosted search and local-first deployments.

## Best Possible Solution

Keep this broader tracker open until maintainers decide the explicit default-off policy shape for web_fetch and a separate Gateway OpenResponses URL-input policy, with docs and tests covering redirects, DNS/private ranges, allowlists, and cache isolation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure the requested web_fetch private-network key and the strict schema has no field for it, while Gateway input_file/input_image URL fetches pass allowPrivateNetwork: false in current main.

## Solution Assessment

Unclear as proposed. A default-off, explicitly named opt-in is the maintainable direction, but automatically enabling private-network access in local Gateway mode needs maintainer security/product review rather than an automated cleanup decision.
