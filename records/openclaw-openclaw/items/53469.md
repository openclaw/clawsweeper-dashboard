## Summary

Keep open. Current main only partially satisfies the request: SearXNG/web_search and the shared SSRF guard have private-network support, but web_fetch, image/media URL references that reuse web_fetch policy, and Gateway OpenResponses input_file/input_image URL fetches still lack a broad private-network opt-in.

## What This Changes

The issue requests explicit private/local network access support for web tools and Gateway URL-fetch surfaces so self-hosted search and local-first deployments can reach trusted private services.

## Best Possible Solution

Keep this as the broader tracker until maintainers choose a default-off public policy shape for web_fetch/media references and a separate documented Gateway OpenResponses URL-input policy, with tests for direct hosts, DNS-to-private, redirects, allowlists, cache isolation, and docs/changelog updates.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the requested web_fetch private-network config key is rejected by strict schema/runtime surfaces, and Gateway input_file/input_image URL fetches explicitly pass allowPrivateNetwork: false.

## Solution Assessment

Unclear as proposed. A default-off, explicitly named opt-in is the maintainable direction, but automatically enabling private-network access in local Gateway mode and widening Gateway URL-input behavior needs maintainer security/product review.
