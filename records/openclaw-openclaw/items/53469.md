## Summary

Keep open. Current main covers the SearXNG/web_search part and the shared SSRF guard already has private-network policy bits, but web_fetch, media URL references that inherit web_fetch policy, and Gateway OpenResponses URL inputs still do not expose the requested broad private-network opt-in; the open web_fetch PR is only a partial implementation.

## What This Changes

Request a private-network access opt-in for web_fetch, web_search/image/media tools, and Gateway URL fetches so trusted local or self-hosted endpoints such as SearXNG can be reached.

## Best Possible Solution

Keep the current SearXNG and provider-request support, review the narrow web_fetch opt-in separately, and define a documented default-off Gateway/media URL policy with regression coverage before broadening private-network access.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: the strict web_fetch schema rejects the requested private-network key, the runtime never forwards it, and Gateway input_file/input_image URL fetches hard-code allowPrivateNetwork: false.

## Solution Assessment

Unclear as proposed. A default-off explicit opt-in is the maintainable direction, but automatic local Gateway enablement and broadening OpenResponses URL inputs need maintainer security/product review.
