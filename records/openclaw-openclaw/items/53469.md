## Summary

Keep open: current main still lacks the requested broad private-network opt-in for `web_fetch`, image/media fetches, and gateway-local behavior, while related work only partially covers SearXNG, model-provider, and fake-IP cases. This is a valid feature/config request, but it changes the SSRF trust boundary and needs maintainer API and security-policy approval before implementation.

## What This Changes

Add configurable private-network access for web tools and gateway/local deployments so self-hosted services like SearXNG and local storage nodes can be reached.

## Best Possible Solution

Keep this as the canonical maintainer discussion until a default-off, documented private-network policy lands under an approved SSRF config shape, with separate follow-through for image/media and gateway scope that remains.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source inspection gives high confidence in the current-main gap because `web_fetch` schema/runtime still expose only the narrow fake-IP SSRF flags while docs and tests preserve private-host blocking.

## Solution Assessment

No, the broad proposed automatic/local-mode direction is not settled enough for automated implementation. The safer path is a maintainer-approved, default-off nested SSRF policy with explicit metadata/link-local behavior, docs, cache partitioning, and focused tests.
