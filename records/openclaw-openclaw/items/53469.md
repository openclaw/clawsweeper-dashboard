## Summary

Keep open: current main partially supports self-hosted search and lower-level SSRF policy flags, but the requested broad private-network opt-in for `web_fetch`, image media fetches, and gateway-related behavior is still unresolved and security-sensitive.

## What This Changes

Request a default-off private/local network access opt-in for web tools and gateway deployments so self-hosted services such as SearXNG can be reached.

## Best Possible Solution

Keep this as the canonical maintainer discussion until a default-off, documented private-network policy lands under the approved `tools.web.fetch.ssrfPolicy` shape, with explicit metadata/link-local policy and follow-ups for image or gateway scope that remains.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source inspection gives high confidence in the current-main gap because strict `web_fetch` config and runtime forwarding still expose only the narrow fake-IP SSRF flags.

## Solution Assessment

No, the broad proposed direction is not settled enough to implement automatically. The safer path is a maintainer-approved nested SSRF policy with documented metadata/link-local behavior, cache partitioning, generated config metadata, and focused SSRF tests.
