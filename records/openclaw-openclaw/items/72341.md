## Summary

Keep open. Current main still has the source path that can render the already-committed pre-tool assistant stream again after a tool boundary, the latest issue comment reports the same symptom on 2026.5.12, and the UI-side fix path remains open at https://github.com/openclaw/openclaw/pull/74498.

## What This Changes

The issue reports cumulative duplicate assistant text cards in Control UI/WebChat when one assistant turn streams text before and after tool calls.

## Best Possible Solution

Land a rebased UI-side committed-prefix projection, such as https://github.com/openclaw/openclaw/pull/74498 or an equivalent patch, with focused regression tests while preserving full-snapshot gateway deltas for ACP, TUI, and generic subscribers.

## Reproduction Assessment

Yes. Current main is source-reproducible with high confidence: full-snapshot assistant text is assigned to chatStream, committed into chatStreamSegments on tool start, and rendered again as the next active stream.

## Solution Assessment

Yes for the current direction. A Control UI committed-prefix slice is narrower than changing gateway delta semantics, because gateway tests and non-UI consumers rely on monotonic full-snapshot chat payloads.
