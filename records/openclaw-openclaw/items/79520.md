## Summary

Keep open: current main and v2026.6.6 still have the reported write-side 0o600 new-file path and the read-side still rethrows non-FsSafe permission errors, while the matching fix PR closed unmerged.

## What This Changes

Issue reports outbound channel media uploads failing with a generic 500 when sandbox or daemon-inherited write paths create local media files with owner-only permissions.

## Best Possible Solution

Keep local-path outbound media support, preserve LaunchAgent umask hardening, scope readable media/user-output modes deliberately, and map EACCES/EPERM to an actionable LocalMediaAccessError.

## Reproduction Assessment

Yes, at source level: current main still creates first-time sandbox bridge writes as 0o600 and still rethrows raw non-FsSafe permission errors from the local media reader. I did not run the full macOS Discord upload path in this read-only review.

## Solution Assessment

Unclear as a final product decision: typed EACCES/EPERM mapping is a narrow fix, but the write-side chmod policy needs maintainer approval or scoping so generic sandbox writes are not unintentionally loosened.
