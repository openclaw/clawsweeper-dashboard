## Summary

Keep open: current main still has the source-backed first-time sandbox write path that leaves new files at 0600 and the outbound media read path still rethrows raw non-FsSafe permission errors instead of a typed local-media error.

## What This Changes

The issue requests fixing outbound channel media sends so owner-only local media files created by sandbox/browser/exec paths are either written with the intended media permissions or rejected with an actionable error.

## Best Possible Solution

Keep local-path outbound media support, preserve LaunchAgent umask hardening, deliberately scope readable media/user-output modes, and map EACCES/EPERM to an actionable LocalMediaAccessError with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: first-time sandbox bridge writes still keep the 0600 temp-file mode, and outbound media reads still let raw EACCES/EPERM escape this boundary; I did not live-run the macOS Discord upload path in this read-only review.

## Solution Assessment

Unclear as a final product decision. Typed EACCES/EPERM mapping is the narrow read-side fix, but maintainers should decide whether the write-side chmod is scoped to media/user-facing files or accepted for generic sandbox bridge writes.
