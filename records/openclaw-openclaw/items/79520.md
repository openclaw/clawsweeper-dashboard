## Summary

Keep open: current main still has the source-backed sandbox write-mode gap and the local media reader can still rethrow raw permission errors into generic gateway error handling; the closed companion PR was not merged and no open replacement owns the fix.

## What This Changes

The issue asks OpenClaw to make outbound channel uploads of local media from restrictive `0o600` files reliable and debuggable by fixing the write-side media mode leak, typing read-side permission failures, and documenting the permission contract.

## Best Possible Solution

Keep the issue open, preserve the LaunchAgent umask hardening, make supported OpenClaw-managed outbound media artifacts readable under a clearly scoped contract, and surface local media permission failures as an actionable typed error.

## Reproduction Assessment

Yes, source-reproducible: current main still creates first-time sandbox fs-bridge writes at `0o600`, exposes `upload-file`/`filePath` as an outbound path, and can rethrow raw local media read permission failures. I did not live-run the macOS Discord upload path in this read-only review.

## Solution Assessment

Unclear whether the broad generic-helper chmod is the best final boundary. The maintainable direction is a media/workspace-scoped permission contract plus typed local-media permission errors, with the optional config knob deferred.
