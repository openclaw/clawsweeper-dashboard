## Summary

Keep open: current main still has the reported first-time 0600 sandbox write path and still lets raw EACCES/EPERM escape the outbound media reader, while the direct fix PR closed unmerged and the merged related PR only covers validated text-document sends.

## What This Changes

The issue reports outbound channel media uploads failing with a bare 500 when files created with restrictive 0600 permissions are sent through `media`/`path`/`filePath`.

## Best Possible Solution

Keep local-path outbound media support, preserve LaunchAgent umask hardening, scope readable media/user-output modes deliberately, and map EACCES/EPERM to an actionable local media error.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `origin/main` still creates first-time sandbox bridge writes at 0600 and the media reader still rethrows raw EACCES/EPERM from `@openclaw/fs-safe`; I did not live-run the macOS Discord upload path in this read-only review.

## Solution Assessment

Unclear as a final product decision. Typed EACCES/EPERM mapping is the narrow read-side fix, but the write-side chmod policy should be maintainer-approved or scoped so generic sandbox writes are not unintentionally loosened.
