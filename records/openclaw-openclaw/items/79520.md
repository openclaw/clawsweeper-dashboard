## Summary

Keep open: current main and the latest release still have the source-level 0600 write gap and the read-side permission error is still not mapped to an actionable local-media error. The linked fix PR was closed unmerged, and the remaining repair crosses a file-permission policy boundary that needs maintainer direction before automation changes generic sandbox write modes.

## What This Changes

The issue asks OpenClaw to make outbound media uploads from owner-only local files actionable and to stop OpenClaw-managed media writers from leaving sendable files at mode 0600.

## Best Possible Solution

Preserve the LaunchAgent umask hardening, make supported OpenClaw-managed outbound media artifacts readable under the established media contract, and convert local EACCES/EPERM media reads into an actionable typed error without broadening unrelated secret or state file writes.

## Reproduction Assessment

Yes for the source-level bug: current main still creates new sandbox fs-bridge files at 0600, Discord upload-file/path remains supported, and the media read path still rethrows raw non-FsSafe permission errors. I did not run the live macOS Discord upload flow in this read-only review.

## Solution Assessment

Unclear whether the broad fchmod proposal is the best final boundary. The typed error mapping is narrow, but the write-side fix should either be media-scoped or explicitly approved as generic workspace write policy.
