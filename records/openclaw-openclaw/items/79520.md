## Summary

Keep open: current `main` still has the source-backed write-side 0600 drift and read-side raw permission-error path, while the only linked fix PR was closed unmerged.

## What This Changes

The issue reports outbound channel media uploads failing with a generic 500 when local media files are mode 0600, and asks for write-side media permissions plus typed read-side errors.

## Best Possible Solution

Preserve the LaunchAgent umask hardening, make supported OpenClaw-managed outbound media artifacts satisfy the 0644 media contract, and map local media EACCES/EPERM failures to an actionable typed error; defer the optional config knob.

## Reproduction Assessment

Yes from source inspection: first-time sandbox helper writes keep 0600, and documented `upload-file`/`filePath` media reads still have a path where raw EACCES/EPERM can escape. I did not run the live Discord upload flow in this read-only pass.

## Solution Assessment

Unclear for the broad chmod proposal because it changes a permission boundary for generic sandbox writes. The typed media error mapping is narrow; the write-side repair should be scoped to outbound media policy or approved as a workspace-file mode change.
