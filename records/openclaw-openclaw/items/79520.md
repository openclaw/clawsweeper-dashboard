## Summary

Keep this open: current main and the latest release still have the source-backed write-side 0600 drift and the read-side raw EACCES/EPERM escape, while the directly related fix PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to make supported outbound media uploads work for gateway-created 0600 files and surface an actionable permissions error instead of a bare 500.

## Best Possible Solution

Keep the issue open and land a focused fix that preserves LaunchAgent umask hardening, makes supported outbound media artifacts readable under the existing media contract, maps local media permission failures to a typed actionable error, and defers any operator config knob.

## Reproduction Assessment

Yes. Source inspection on current main shows first-time sandbox fs-bridge writes still publish 0600 files and the local media reader can still rethrow raw permission errors; I did not live-run the Discord upload path.

## Solution Assessment

Unclear for the broad write-side proposal. Fchmoding before atomic replace and typed error mapping are the right shape, but the chmod scope should be media/artifact-policy aware or explicitly approved rather than silently changing every generic sandbox new-file write.
