## Summary

Keep open: current main and the latest release still do not satisfy the JSON-only OTLP collector request; Docker keeps the protobuf fallback, the bundled diagnostics-otel runtime/schema/docs exclude `http/json`, and the linked default-flip PR was closed unmerged after the author agreed it was the wrong scope.

## What This Changes

The issue requests a durable gateway telemetry protocol/default contract so JSON-only OTLP/HTTP collectors do not require manual edits that service regeneration can overwrite.

## Best Possible Solution

Choose a maintainer-approved telemetry protocol contract, then either implement end-to-end `http/json` support or document and test a durable operator override/preservation path that survives gateway service regeneration without breaking protobuf collectors.

## Reproduction Assessment

Yes. Source inspection shows current main still defaults Docker OTLP protocol to `http/protobuf` and current diagnostics-otel config/runtime/docs still do not support `http/json`; I did not run a live JSON-only collector in this read-only review.

## Solution Assessment

No. The linked default-only PR is not the best fix because it would set a protocol the bundled exporter, schema, and docs do not support; the safer path is a deliberate override/preservation contract or complete `http/json` support.
