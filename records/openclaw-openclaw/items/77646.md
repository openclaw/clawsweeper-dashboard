## Summary

This should stay open: current main still documents and implements only OTLP/HTTP protobuf for the bundled diagnostics exporter, and the linked open fix PR only flips the Docker Compose default to a protocol the exporter rejects. The remaining work needs maintainer direction on durable operator override versus full `http/json` exporter support.

## What This Changes

Request durable support for JSON-only OTLP/HTTP collectors by changing or preserving gateway OTEL protocol settings across service-env regeneration.

## Best Possible Solution

Add deliberate support for JSON-only collectors through a durable per-profile/env override that survives install regeneration, or add end-to-end `http/json` exporter support; keep the default unchanged until compatibility is decided.

## Reproduction Assessment

Yes, source-level: current main defaults the Docker fallback to `http/protobuf`, rejects any diagnostics-otel protocol other than `http/protobuf`, and documents that limit. I did not live-run the exact JSON-only collector rejection.

## Solution Assessment

No for the linked one-line default flip: it would feed `http/json` into a bundled exporter path that currently rejects it. The maintainable fix is a deliberate durable override or full exporter support with schema, docs, and tests.
