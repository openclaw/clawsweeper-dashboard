## Summary

Keep open: current main intentionally supports only OTLP/HTTP protobuf, while this report asks for JSON OTLP/HTTP compatibility and a durable operator override. The source, schema, package dependencies, and docs all show that `http/json` is not currently implemented.

## What This Changes

Request support for JSON OTLP/HTTP telemetry export, plus a persistent way for gateway profiles to pin the OTLP protocol without editing generated service-env files.

## Best Possible Solution

Extend the diagnostics-otel plugin, config schema, docs, and focused tests to support `http/json` as an explicit protocol first, then decide separately whether the default should change.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show that `http/json` is rejected before exporters start, and the current config schema cannot accept that protocol.

## Solution Assessment

Unclear. The narrow maintainable path is to add explicit `http/json` support through the existing `diagnostics.otel.protocol` contract; changing the default or preserving manual generated-file edits should be a separate maintainer decision.
