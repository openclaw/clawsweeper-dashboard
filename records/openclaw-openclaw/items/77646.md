## Summary

Current main still only supports `http/protobuf` for the `diagnostics-otel` exporter, while the requested JSON OTLP path and regeneration behavior need a maintainer decision rather than conservative cleanup closure.

## What This Changes

Request support for OTLP/HTTP JSON or a durable per-profile protocol override so JSON-only collectors do not need post-install service-env edits.

## Best Possible Solution

Add `http/json` support through the existing `diagnostics.otel.protocol`/`OTEL_EXPORTER_OTLP_PROTOCOL` surface, update schema/docs/tests, and make a deliberate compatibility decision on whether the default should change.

## Reproduction Assessment

Yes, source-level: current main rejects any diagnostics OTEL protocol other than `http/protobuf`, so configuring `http/json` follows the unsupported path. The live collector rejection rate was not replayed locally.

## Solution Assessment

Unclear as written: adding `http/json` behind the existing protocol setting is the narrowest maintainable path, while changing the default requires compatibility review.
