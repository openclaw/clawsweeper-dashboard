## Summary

Keep open: current main still rejects `http/json` for diagnostics OTLP export, while changing the supported protocol/default and regeneration warnings needs maintainer compatibility judgment rather than cleanup closure.

## What This Changes

Support JSON OTLP/HTTP collectors or a durable protocol override so generated gateway service-env files do not require manual edits.

## Best Possible Solution

Add `http/json` support through the existing `diagnostics.otel.protocol` and `OTEL_EXPORTER_OTLP_PROTOCOL` surfaces, update schema/docs/tests, and make an explicit compatibility decision on the default and regeneration warning behavior.

Yes, source-level: current main returns `unsupported_protocol` for any diagnostics OTEL protocol other than `http/protobuf`, and the schema rejects `http/json` as a config value. I did not live-run a JSON-only collector.

## Solution Assessment

Unclear as an automatic fix: adding `http/json` behind the existing protocol setting is the narrow maintainable path, but changing the default or warning policy needs a maintainer compatibility decision.
