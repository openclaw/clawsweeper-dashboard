## Summary

This should stay open: current main still defaults and documents the supported OpenTelemetry path as OTLP/HTTP protobuf, while JSON-only collector support needs a deliberate protocol/config decision rather than cleanup closure.

## What This Changes

The issue requests durable OpenClaw gateway OTEL protocol handling so JSON-only OTLP/HTTP collectors do not require manual edits to regenerated service env files.

## Best Possible Solution

Add a documented durable per-profile or environment protocol override that survives regeneration, or implement end-to-end http/json exporter support before changing defaults.

## Reproduction Assessment

Yes, source-level: current main still defaults Docker to http/protobuf, rejects non-http/protobuf in diagnostics-otel, and documents that only http/protobuf is honored. I did not live-run the exact JSON-only collector rejection.

## Solution Assessment

Unclear as a product decision: the default-only PR is not the best fix because it feeds an unsupported protocol into current exporter code. The maintainable path is either a durable override/preservation contract or full http/json exporter support with schema, docs, and tests.
