## Summary

Keep open: current main and v2026.6.10 still default Docker OTLP protocol to `http/protobuf`, still reject `http/json` in the bundled diagnostics-otel runtime/config path, and the linked default-flip PR closed unmerged after the author agreed it was the wrong scope.

## What This Changes

The issue asks for JSON-only OTLP collectors to work without manually editing regenerated gateway service environment files, with a durable way to pin the telemetry protocol.

## Best Possible Solution

Choose a maintainer-approved telemetry protocol contract, then implement either end-to-end `http/json` support or a documented durable override/preservation path that survives service-env regeneration without breaking protobuf collectors.

## Reproduction Assessment

Yes for source reproduction: current main still defaults Docker OTLP protocol to `http/protobuf`, rejects non-`http/protobuf` in diagnostics-otel, and documents only protobuf as honored. I did not run a live JSON-only collector in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. The linked default-only PR is not the best fix; the durable fix needs either complete `http/json` support or a documented override/preservation path.
