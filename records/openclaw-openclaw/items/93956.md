## Summary

Keep this PR open: current main still auto-discovers hosted `models.providers.ollama.baseUrl: "https://ollama.com"` through `/api/tags`, while the PR adds a targeted hosted-Ollama guard with focused tests and sufficient terminal proof. I found no discrete patch defect, but merging still needs maintainer acceptance of the compatibility/auth-provider tradeoff for existing hosted Ollama configs.

## What This Changes

The PR changes the Ollama bundled plugin so hosted `*.ollama.com` provider configs without explicit model rows skip live `/api/tags` discovery, preserves explicit/local/self-hosted paths, adds discovery tests, widens Ollama config input typing, and ignores `cross-spawn` in Knip dependency metadata.

## Best Possible Solution

Land only after maintainers accept that hosted `models.providers.ollama` configs now require explicit models, with `ollama-cloud` remaining the first-class hosted live-catalog path and local/self-hosted discovery preserved.

## Reproduction Assessment

Yes, source-level: current main reaches `buildProvider` for configured hosted Ollama base URLs, and a live unauthenticated probe confirms the hosted `/api/tags` endpoint returns a catalog. I did not run an authenticated OpenClaw model-list flow in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the compatibility tradeoff: the guard is hosted-Ollama-specific, keeps local and remote self-hosted discovery, and preserves explicit model rows. The safer alternative is to require an upgrade note or design an explicit compatibility path before merge.
