## Summary

Keep open: the current PR head addresses the earlier over-broad remote-host blocker by limiting the skip to hosted Ollama Cloud, preserves local and remote self-hosted discovery, and has sufficient terminal proof for the reported hosted-cloud behavior. The remaining concern is an intentional compatibility change for existing hosted-cloud configs, not a line-level defect.

## What This Changes

The PR narrows Ollama provider catalog discovery so hosted `ollama.com` configs without explicit models skip `/api/tags`, while local and remote self-hosted Ollama discovery remain enabled.

## Best Possible Solution

Keep the hosted-cloud skip narrowly scoped and make the supported upgrade path explicit for affected cloud users while preserving local and remote self-hosted discovery.

## Reproduction Assessment

Yes for the code path, with medium confidence: current main can discover hosted `models.providers.ollama.baseUrl: "https://ollama.com"` when auth/config make discovery meaningful, while the PR head returns before `/api/tags` for that hosted-cloud/no-explicit-models case. The PR body also supplies redacted before/after terminal output, but I did not run a live paid cloud account in this read-only review.

## Solution Assessment

Yes. The current head is the narrow maintainable fix because it targets hosted Ollama Cloud while preserving the current self-hosted remote discovery contract; changing only the `ollama-cloud` provider would not fix users who configured hosted cloud through `models.providers.ollama`.
