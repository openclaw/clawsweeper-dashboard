## Summary

Keep open: the patch appears correct for the hosted Ollama discovery bug, but it changes model availability for existing hosted `models.providers.ollama` configs, so maintainer acceptance of that compatibility tradeoff is still required.

## What This Changes

The PR skips `ollama` provider `/api/tags` discovery for hosted `*.ollama.com` base URLs without explicit models, widens Ollama provider config input typing, adds discovery tests, and updates Knip dependency-ignore metadata.

## Best Possible Solution

Land only after maintainers accept that hosted `models.providers.ollama` configs require explicit models or the first-class `ollama-cloud` provider, while preserving remote self-hosted discovery.

## Reproduction Assessment

Yes, source-level: current main passes configured hosted Ollama base URLs into `buildProvider`, which calls `/api/tags`. The PR body supplies redacted before/after terminal output, but I did not run an authenticated Ollama Cloud account.

## Solution Assessment

Yes, with maintainer acceptance of the compatibility tradeoff: the guard is hosted-Ollama-specific, preserves remote self-hosted and local discovery, and matches the documented explicit-model boundary.
