## Summary

Keep this PR open for maintainer compatibility acceptance: the patch is a narrow Ollama hosted-cloud discovery fix with good proof and no concrete correctness finding, but it intentionally changes model availability for existing hosted `models.providers.ollama` setups that relied on live `/api/tags` discovery.

## What This Changes

The PR changes the bundled Ollama provider discovery path so hosted `*.ollama.com` base URLs without explicit model rows skip live catalog discovery, while preserving explicit models, local/self-hosted discovery, tests, and Knip metadata.

## Best Possible Solution

Land only after maintainers accept that hosted `models.providers.ollama` configs now require explicit model rows, with `ollama-cloud` remaining the first-class hosted live-catalog path and local/self-hosted discovery preserved.

## Reproduction Assessment

Yes, source-level: current main reaches `buildProvider` for configured hosted Ollama base URLs, and a live unauthenticated probe confirms the hosted `/api/tags` endpoint returns a catalog. I did not run an authenticated OpenClaw model-list flow in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the compatibility tradeoff: the guard is hosted-Ollama-specific, keeps local and remote self-hosted discovery, and preserves explicit model rows. The safer alternative is to require an upgrade note or design an explicit compatibility path before merge.
