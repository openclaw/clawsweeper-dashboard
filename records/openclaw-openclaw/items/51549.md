## Summary

Keep this open. Current main has the gateway-backed history reload path, and it now preserves queued messages across in-page session switches, but queued-message and draft state still are not persisted across a full browser refresh. The issue is also paired with open PR #73154, so it should remain open until that PR is reviewed, merged, replaced, or closed.

## What This Changes

The issue asks WebChat/Control UI to preserve queued messages, restored conversation history, unsent draft text, and reconnect feedback across browser refreshes.

## Best Possible Solution

Keep the issue open while maintainers review PR #73154 or a replacement. The desired end state is a tested Control UI path that restores gateway history through chat.history, persists/drains queued messages across refresh/reconnect/session switch, and either persists drafts or explicitly tracks that remaining UX gap in a narrower follow-up before this issue closes.

## Reproduction Assessment

Yes. The report has concrete steps, and current source still keeps queued messages and drafts in browser memory only; a full refresh rebuilds App state with chatQueue and chatMessage empty while history reload is separately handled through chat.history.

## Solution Assessment

Unclear. The current chat.history reload is the right contract for conversation history, and localStorage queue persistence is a plausible narrow fix for refresh loss, but PR #73154 does not settle draft persistence or the broader ACK/retry design raised by #45952.
