## Summary

Keep open: current main and v2026.5.22 reload transcript history through `chat.history`, but queued messages and the draft composer still live in page-local state, and the open linked implementation PR needs maintainer review before this issue can close.

## What This Changes

The issue asks WebChat to preserve queued messages, visible conversation history, and draft input across browser refresh or reconnect.

## Best Possible Solution

Keep this issue open while maintainers land or replace https://github.com/openclaw/openclaw/pull/73154 with a scoped design that restores history, drafts, and queued messages under approved retention, attachment, and replay-ordering semantics.

## Reproduction Assessment

Yes, at source level: current main reconstructs `chatMessage` and `chatQueue` as empty page-local state on reload, while `loadChatHistory` only restores transcript messages through `chat.history`. I did not run a live browser F5 repro in this read-only review.

## Solution Assessment

Unclear as a final implementation: gateway-backed transcript history reload is the right contract, but queue and draft persistence still need maintainer-approved retention, attachment recovery, and active-run replay ordering before a fix is safe to land.
