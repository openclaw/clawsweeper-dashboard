## Summary

Keep open: current main and v2026.5.22 still keep WebChat queued messages and draft text in page-local state, while history reload is only partly addressed and open implementation PRs still need maintainer review.

## What This Changes

The issue asks WebChat to preserve queued messages, gateway-backed conversation history, draft text, and reconnect feedback across browser refresh or tab reopen.

## Best Possible Solution

Land or replace the existing linked PRs with a scoped WebChat refresh-state design that restores history, drafts, and queued messages under an approved retention, attachment, and replay-ordering policy.

## Reproduction Assessment

Yes, at source level: current main restores visible history through chat.history, but chatMessage and chatQueue are initialized and mutated in memory with no durable restore path for a full page refresh. I did not run a live browser F5 repro in this read-only review.

## Solution Assessment

Unclear as a final implementation: gateway-backed history reload is the right contract, but queue and draft persistence still need maintainer-approved retention, attachment, and active-run replay semantics before an implementation is safe to land.
