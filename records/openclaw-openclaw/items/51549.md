## Summary

Keep open. Current main restores WebChat history from the gateway, but queued messages and draft text still rebuild from in-memory UI state on refresh, and the open closing PR at https://github.com/openclaw/openclaw/pull/73154 is an implementation candidate rather than a reason to close this issue early.

## What This Changes

The issue asks WebChat to preserve queued messages, visible conversation history, and draft input across browser refresh or tab reopen.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/73154 or a replacement lands reviewed queue persistence through the shared WebChat session lifecycle, with draft persistence either included deliberately or split into a clearly linked follow-up.

## Reproduction Assessment

Yes, at source level. Current main restores visible history from chat.history, but chatMessage and chatQueue are initialized and mutated in component memory with no current browser-storage or gateway restore path for a full page refresh.

## Solution Assessment

Unclear. Gateway-backed history reload is the right contract, but the remaining queue and draft persistence need reviewed lifecycle, attachment, and retention semantics before the linked PR or a replacement can be considered the best fix.
