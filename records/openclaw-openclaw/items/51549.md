## Summary

Keep open: current main and v2026.5.22 still keep WebChat queued messages and drafts in browser component memory, while history reload is only partly solved and the linked fix PR remains open, conflicting, and review-blocked.

## What This Changes

The issue asks WebChat to restore queued messages, visible conversation history, and unsent draft text after a browser refresh or reconnect.

## Best Possible Solution

Land or replace the linked fix with a scoped WebChat refresh-state design that keeps gateway-backed history reload, restores drafts and queued messages under an approved retention policy, and defines safe attachment and active-run replay behavior.

## Reproduction Assessment

Yes, at source level: current main restores visible history through chat.history, but chatMessage and chatQueue are initialized and mutated in memory with no durable restore path for a full page refresh. I did not run a live browser F5 repro in this read-only review.

## Solution Assessment

Unclear as a final implementation: gateway-backed history reload is the right contract, but queue and draft persistence still need maintainer-approved retention, attachment, and active-run replay semantics before the linked PR is safe to land.
