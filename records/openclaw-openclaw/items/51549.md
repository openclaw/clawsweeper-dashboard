## Summary

Keep open: current main and the latest release still keep WebChat queued messages and draft text in page-local memory, while history reload is only partly addressed and related implementation PRs are still open for maintainer review.

## What This Changes

WebChat should preserve pending messages, visible conversation history, and draft input across browser refresh, tab close/reopen, and reconnect.

## Best Possible Solution

Review and land a scoped implementation, likely the newer composer/history approach in https://github.com/openclaw/openclaw/pull/83992 or a replacement for https://github.com/openclaw/openclaw/pull/73154, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level: current main restores visible history through chat.history, but chatMessage and chatQueue are initialized and mutated in memory with no durable restore path for a full page refresh. I did not run a live browser F5 repro in this read-only review.

## Solution Assessment

Unclear as a final implementation: gateway-backed history reload is the right contract, but queue and draft persistence still need maintainer-approved retention, attachment, and active-run replay semantics before an implementation is safe to land.
