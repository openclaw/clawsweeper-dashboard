## Summary

Keep open: current main and v2026.5.22 restore visible history through `chat.history`, but queued WebChat messages and draft text are still page-local state. The linked implementation at https://github.com/openclaw/openclaw/pull/73154 is still open and needs maintainer decisions before this issue can close.

## What This Changes

The issue requests that WebChat preserve queued messages, visible conversation history, and draft input across browser refresh or reconnect.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/73154 with a scoped WebChat refresh-state design that restores history, drafts, and queued messages under approved retention, attachment, and replay-ordering semantics.

## Reproduction Assessment

Yes, at source level: current main reloads visible history through `chat.history`, but `chatMessage` and `chatQueue` are initialized and mutated in memory with no durable restore path for a full page refresh. I did not run a live browser F5 repro in this read-only review.

## Solution Assessment

Unclear as a final implementation: gateway-backed history reload is the right contract, but queue and draft persistence still need maintainer-approved retention, attachment, and active-run replay semantics before an implementation is safe to land.
