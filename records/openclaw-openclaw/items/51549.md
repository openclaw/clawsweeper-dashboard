## Summary

Keep open. Current main has restored the gateway-backed WebChat history path, but queued messages and draft text are still held in browser component memory, and the open linked implementation candidate has not merged.

## What This Changes

WebChat should preserve queued messages, restored transcript history, and draft text across browser refresh for the active session.

## Best Possible Solution

Keep the gateway-backed history reload, then land a reviewed queue persistence path through the shared WebChat session lifecycle; decide draft retention explicitly or split it into a linked follow-up.

## Reproduction Assessment

Yes, at source level. Current main restores visible history from `chat.history`, but `chatMessage` and `chatQueue` are initialized and mutated in component memory with no current browser-storage or gateway restore path for a full page refresh.

## Solution Assessment

Unclear. Gateway-backed history reload is the right contract, but queue and draft persistence still need reviewed retention, attachment, and active-run replay semantics before the linked pull request or a replacement is the best fix.
