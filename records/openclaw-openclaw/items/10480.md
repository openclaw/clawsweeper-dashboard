## Summary

Keep this open. Current main still configures the bundled Cloudflare AI Gateway onboarding path as Anthropic-only and does not expose a Cloudflare-specific Workers AI model selector, endpoint mode, or model list population path for non-interactive onboarding.

## What This Changes

The issue requests Workers AI model selection during `openclaw onboard --non-interactive --auth-choice cloudflare-ai-gateway-api-key`, so users can configure `@cf/...` Workers AI models without hand-editing config.

## Best Possible Solution

Add a provider-owned Workers AI setup path in `extensions/cloudflare-ai-gateway` that preserves the existing Anthropic default, lets non-interactive onboarding select a Workers AI `@cf/...` model, writes an OpenAI-compatible Cloudflare AI Gateway provider config, and documents the exact scripted command. If maintainers prefer the unified `/compat` route instead of a provider-specific Workers endpoint, the docs and tests should make that contract explicit.

## Reproduction Assessment

Yes. A high-confidence reproduction is code inspection of current main: run the proposed non-interactive Cloudflare auth choice mentally through `runNonInteractive`, which only consumes account ID, gateway ID, and API key before writing Anthropic config. No live onboarding command was run because this review is read-only.

## Solution Assessment

No. Current main's custom-provider workaround can express an OpenAI-compatible endpoint, but it does not solve the bundled `cloudflare-ai-gateway-api-key` onboarding behavior requested here; the narrow maintainable fix belongs in the Cloudflare provider plugin with focused docs and tests.
