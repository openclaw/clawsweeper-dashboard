## Summary

The feature remains necessary: current main’s bundled Cloudflare onboarding still generates only an Anthropic Messages provider with a fixed Claude default, while Workers AI requires the separate generic custom-provider path and is not documented as the supported Cloudflare flow. The permanent interface needs maintainer product judgment because it changes public non-interactive onboarding and generated provider configuration.

## What This Changes

Add non-interactive onboarding support for selecting a Cloudflare Workers AI endpoint and `@cf/...` model during Cloudflare AI Gateway setup.

## Best Possible Solution

Define one provider-owned Workers AI mode that generates the `/workers-ai/v1` OpenAI-compatible provider and selected `@cf/...` model, or explicitly adopt and document the generic custom-provider command as the canonical supported flow.

## Reproduction Assessment

Yes at source level: the bundled non-interactive Cloudflare flow has no Workers AI endpoint or model input and always applies its fixed Anthropic provider configuration.

## Solution Assessment

Unclear: a provider-owned Workers AI mode is the cleanest first-class design, but maintainers must decide whether to add that public onboarding contract or formally support the existing generic custom-provider route.
