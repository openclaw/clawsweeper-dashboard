## Summary

Keep open: current main still clears Dashboard chat run state on reconnect hello, the gateway hello snapshot still has no active-run recovery payload, and the linked recovery PR is open/conflicting rather than a landed fix.

## What This Changes

The issue reports that Gateway Dashboard chat composer controls can revert from Stop/Queue to New session/Send after tab/view return or reconnect while the same session still has an active run.

## Best Possible Solution

Land a current-main-compatible active chat-run recovery fix that exposes authoritative active-run state to Dashboard reconnect/hello handling and keeps Stop/Queue visible until the active run really ends.

## Reproduction Assessment

Yes, at source level: the current reconnect hello path clears chatRunId and run indicators, and the current test expects that even with a hasActiveRun session row. I did not run a live browser tab-switch or reconnect reproduction.

## Solution Assessment

Unclear for the existing linked PR because it is open, conflicting, and has outstanding review/proof blockers. The maintainable fix direction is clear: recover from authoritative gateway active-run state instead of clearing Dashboard busy state on hello.
