## Label Justifications

- `impact:session-state`: The failure is stale or lost chat run state across Dashboard tab/reconnect recovery.

## Summary

Keep open. Current main and the latest release still clear local active chat-run state on reconnect hello and do not expose an active-run recovery payload in the hello snapshot; the linked fix path at https://github.com/openclaw/openclaw/pull/41067 is open, conflicting, and proof-gated rather than merged.

## What This Changes

Gateway Dashboard chat should keep the selected session in Stop/Queue busy mode after returning to chat while a run is still active.

## Best Possible Solution

Land or replace the linked recovery PR with a current-main fix that exposes reliable active-run state on hello/tab return and keeps Dashboard controls in Stop/Queue while the selected session is still abortable.

## Reproduction Assessment

Yes, at source level. Current main clears local chatRunId/chatStream on reconnect hello, the test suite expects that interruption behavior, and the hello snapshot lacks active-run recovery data; I did not run a live browser reproduction.

## Solution Assessment

No, this is not solved on main yet. The activeChatRuns direction in the linked PR is plausible, but the current candidate needs rebase/repair/proof or a narrow replacement before it becomes the best landed fix.
