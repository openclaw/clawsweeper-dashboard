## Label Justifications

- `P1`: The issue affects an active Dashboard chat workflow where reconnect/tab return can expose controls that interrupt or replace a still-running turn.
- `impact:session-state`: The current reconnect path discards active run identity and projects the session as killed even when server-side run state may still exist.
- `impact:message-loss`: Losing the busy/run state can hide or interrupt in-progress assistant output and make the operator send over the active turn.

## Summary

Keep open: current main and the latest release still clear the Dashboard chat run state after reconnect instead of restoring the active run, and the open linked recovery PR is not merged yet. The issue should stay tied to https://github.com/openclaw/openclaw/pull/41067 or a narrowed replacement until that work lands.

## What This Changes

The issue asks Gateway Dashboard chat to preserve Stop/Queue busy controls for the same active session after tab return or reconnect instead of reverting to New session/Send.

## Best Possible Solution

Restore Dashboard busy state from authoritative gateway in-flight/active-run data after reconnect or tab return, with focused UI/Gateway tests and real Dashboard proof before closing this issue.

## Reproduction Assessment

Yes at source level. Current main has a deterministic reconnect hello path that clears the active run and a focused test that expects `chatRunId` to become null, although this review did not run a live Dashboard browser reproduction.

## Solution Assessment

No merged solution exists yet. The best direction is gateway-authoritative active-run reconciliation, but the open linked PR needs rebase, proof, and possibly narrowing before it is the best landing path.
