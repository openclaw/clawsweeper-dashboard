## Summary

Keep open. Current main still clears Gateway Dashboard chat run state on hello/reconnect and has no active chat-run data in the hello snapshot to restore `chatRunId`; the composer still derives Stop/Queue from `chatRunId`. The issue is also paired with open, unmerged PR #41067 using closing syntax, so it should stay open until maintainers land, replace, or close that PR.

## What This Changes

Keep the Gateway Dashboard chat composer in the same session's Stop/Queue busy state after returning to chat or reconnecting while a run is still active.

## Best Possible Solution

Land or replace PR #41067 with a narrow additive protocol fix: include active chat-run metadata in the gateway hello snapshot from server active-run state, restore matching UI `chatRunId` and busy controls for the selected session on reconnect/tab return, preserve abort/queue behavior, and cover the gateway protocol/server plus Control UI reconnect tests before closing this issue.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: set an active `chatRunId`, trigger the Control UI hello/reconnect path, and current main clears `chatRunId`/stream state, after which the render path no longer passes `canAbort` and the buttons fall back from Stop/Queue.

## Solution Assessment

Yes at the design level: server-owned active-run snapshot recovery is the narrow maintainable fix because local UI state alone cannot reliably know whether the run is still abortable after reconnect. Current main is not sufficient, and the linked PR should be reviewed or replaced before this issue closes.
