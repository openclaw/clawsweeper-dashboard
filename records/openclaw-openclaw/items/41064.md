## Summary

Keep open: current main still has a source-level reconnect path that clears the local chat busy signal without a Gateway hello active-run recovery payload, and the linked implementation PR is still open rather than merged.

## What This Changes

The issue reports Gateway Dashboard chat controls reverting from Stop/Queue to New session/Send after returning to an active chat run.

## Best Possible Solution

Review, rebase, and land or replace https://github.com/openclaw/openclaw/pull/41067 with an additive active-run recovery contract that keeps Gateway hello/reconnect state and Dashboard chat controls aligned.

## Reproduction Assessment

Yes, source-reproducible: current main clears chatRunId/chatStream on Gateway hello while the hello snapshot has no active-run recovery payload. I did not run a live browser session in this read-only review.

## Solution Assessment

Yes for keeping this open: the narrow maintainable fix is to carry active abortable chat-run state through Gateway hello/reconnect and reconcile the Dashboard controls from that source. The linked PR is the active implementation lane, but it still needs rebase/proof/review before merge.
