## Summary

Keep this issue open: the linked closing PR is still open, and current main only has a partial shipped mitigation. Source still shows the UI can lose the concrete active `chatRunId` after reconnect/tab-return paths, and send/queue logic still treats only `chatRunId` as busy.

## What This Changes

The issue reports Gateway Dashboard chat controls reverting from `Stop` / `Queue` to `New session` / `Send` after returning to an active chat session.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/41067 with a current-main-aware fix that restores active run identity or consistently treats session-level active-run state as busy for both controls and send/queue behavior.

## Reproduction Assessment

Yes at source level: current main clears `chatRunId` after reconnect and the busy helper used by send logic ignores session-level `hasActiveRun`. I did not run a browser E2E, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No, current main is only a partial fix. The best fix should align hello/session active-run recovery, composer controls, and send/queue decisions so a recovered active session remains busy without relying on stale local state.
