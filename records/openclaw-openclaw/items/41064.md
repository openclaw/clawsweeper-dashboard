## Label Justifications

- `P1`: Current main can still expose active Dashboard chat runs as idle to send/queue logic after local run identity is lost.
- `impact:session-state`: The issue is about losing or misrepresenting active session run state after tab return or reconnect.
- `impact:message-loss`: A stale idle state can route a new send instead of queueing behind an in-flight response.

## Summary

Keep open: current main has partial Stop recovery from session rows, but send and queue decisions can still treat the selected chat as idle after local run identity is lost, and the open linked PR remains the candidate fix path.

## What This Changes

Gateway Dashboard chat should preserve Stop/Queue busy state after switching away from and back to an active chat session.

## Best Possible Solution

Make one authoritative active-run state drive chat rendering, queueing, aborts, stream ownership, and terminal cleanup, then prove the Dashboard tab-return/reconnect flow on a real setup.

## Reproduction Assessment

Yes, at source level: after `chatRunId` is cleared on reconnect, current main can render Stop from `sessionsResult.hasActiveRun` while send/queue logic only checks `isChatBusy`, which ignores that state. I did not run a live Dashboard tab-switch scenario.

## Solution Assessment

No merged solution fully solves this yet. The open linked PR is the right candidate path to review, repair, or replace before this issue closes.
