## Summary

Current main already fixes the Dashboard chat busy-state recovery problem: the UI now derives abortability from session active-run state after reconnect/reload, and the merged fix shipped in the 2026.5.2 release train.

## What This Changes

Keep Gateway Dashboard chat controls in Stop/Queue busy state after returning to a session while its run remains active.

## Best Possible Solution

Keep the shipped session-scoped active-run recovery path; evaluate any remaining restart-notice work from the older linked PR separately.

## Reproduction Assessment

Yes. The historical failure path is source-backed: Stop/Queue used to depend on local `chatRunId`, while current main now recovers abortability from session active-run state; I did not need a live browser repro to verify the shipped source fix.

## Solution Assessment

Yes. The shipped fix is the best current shape because it reuses `sessions.list`/`chat.history` session active-run state and session-scoped `chat.abort` instead of adding a separate hello-only active-run snapshot path.
