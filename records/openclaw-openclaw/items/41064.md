## Label Justifications

- `P1`: The original issue affected operator control of active Dashboard chat runs during normal navigation/reconnect workflows.
- `impact:session-state`: The bug was about losing active-run state in the Dashboard UI after returning to the chat session.
- `impact:message-loss`: The stale controls could let an operator interrupt or replace an in-flight run, risking lost assistant output.

## Summary

Current main already solves the central Dashboard busy-state problem: Control UI now derives abortability from server-reported active session runs, keeps Stop/Queue available when local run id is lost, and sends session-scoped aborts after reconnect; the fix shipped in v2026.5.2.

## What This Changes

The issue reports that Gateway Dashboard chat can lose active-run busy controls after tab/view return and expose `New session` / `Send` while the same session is still running.

## Best Possible Solution

Keep the shipped session-row active-run recovery path from v2026.5.2 and handle any remaining restart-notice or snapshot-shape work through a narrower follow-up rather than this issue.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? The original failure is source-reproducible from the pre-fix path that rendered Stop only from local `chatRunId`; I did not run a live browser repro, and current main now has source and test coverage for the recovered active-run path.

## Solution Assessment

Is this the best way to solve the issue? Yes: using existing Gateway session state plus session-scoped abort is the narrower shipped fix for the user-visible Stop/Queue recovery problem, without requiring a new hello snapshot field for this issue.
