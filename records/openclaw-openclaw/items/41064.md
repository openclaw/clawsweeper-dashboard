## Summary

Keep open: current main still has a source-level path where reconnect/session restoration can drop `chatRunId`, and this issue is already paired with open closing PR #41067, so it should remain open until that PR or a replacement lands.

## What This Changes

Keep Gateway Dashboard chat composer controls in `Stop` / `Queue` mode when returning to a session with an active run.

## Best Possible Solution

Land a reviewed fix that restores active run identity from gateway state on reconnect/session restore, keeps `Stop` / `Queue` visible, and makes the send path queue instead of starting a replacement run.

## Reproduction Assessment

Yes, at source level: keep an active run in `chatAbortControllers`, reconnect the Control UI, and current main clears `chatRunId`; the next send path no longer treats the session as busy. I did not run a live browser repro.

## Solution Assessment

Yes, the narrow maintainable direction is to recover active run state from gateway-owned state and align the queue/send logic with that recovered state. The open PR is the current implementation candidate and should be reviewed or replaced before this issue closes.
