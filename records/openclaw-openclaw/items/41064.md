## Summary

Keep open: current main partially recovers Stop visibility from session active-run flags, but the send path still treats a recovered active run as idle when `chatRunId` is gone, so the Queue button can send immediately instead of queueing. The linked implementation PR remains open with closing syntax and should be reviewed, rebased, or replaced rather than closing this issue first.

## What This Changes

The issue asks for Gateway Dashboard chat to preserve Stop/Queue busy state after returning to a session with an active run and prevent accidental run interruption or replacement.

## Best Possible Solution

Review, rebase, and land or replace https://github.com/openclaw/openclaw/pull/41067 with a narrow fix that treats recovered abortable session runs as busy for both rendering and send/queue behavior while preserving the shipped session-list recovery.

## Reproduction Assessment

Yes, source-reproducible: current main can recover `canAbort` from `sessionsResult.hasActiveRun`, but `handleSendChat` still queues only when `isChatBusy` sees `chatRunId`. I did not run a live Dashboard session.

## Solution Assessment

Yes for keeping this issue open: the narrow maintainable fix is to align the recovered active-run state with both controls and send/queue behavior. The existing linked PR is the right review lane, but it may need rebase, narrowing, or replacement.
