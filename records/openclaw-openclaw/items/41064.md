## Label Justifications

- `P1`: A shipped Dashboard workflow exposes incorrect controls that can interrupt an active user run.
- `impact:session-state`: The UI’s chat-run identity and busy state diverge from gateway-owned session state after reconnect or navigation.
- `impact:message-loss`: The incorrect controls can terminate, replace, or suppress completion of an in-progress response.

## Summary

Current main and v2026.6.5 still clear an in-progress Dashboard chat run during reconnect without checking gateway-owned active-run state, and the open same-author PR remains the canonical implementation candidate.

## What This Changes

Preserve `Stop` and `Queue` busy controls for an active Dashboard chat after tab return, refresh, or reconnect.

## Best Possible Solution

Add gateway-authoritative active-run recovery, restore only the selected session’s run identity on hello or tab return, and retain orphan cleanup only when the gateway confirms the run has ended.

## Reproduction Assessment

Yes at source level. The current reconnect hello path deterministically marks any tracked run interrupted and clears it without checking whether that session still has an active gateway run, matching the reported control-state loss.

## Solution Assessment

Yes in direction, but not necessarily in the linked PR’s full scope. Gateway-authoritative active-run reconciliation is the narrow maintainable solution; restart-notice and navigation changes should be separated or independently justified.
