## Label Justifications

- `P1`: A shipped Dashboard workflow can expose idle controls during an active run and let users interrupt or replace the in-flight response.
- `impact:session-state`: The UI’s chat-run identity and busy state can diverge from gateway-owned session state after reconnect or navigation.
- `impact:message-loss`: The wrong Send/New session controls can terminate, replace, or suppress completion of an in-progress chat response.

## Summary

Keep this open: current main and the latest release still clear Dashboard chat run state on reconnect without a gateway-authoritative active-run restore path, and the same-author implementation PR at https://github.com/openclaw/openclaw/pull/41067 is still open and conflicting rather than merged.

## What This Changes

Gateway Dashboard chat should keep Stop/Queue busy controls when returning to a session that still has an active run.

## Best Possible Solution

Add gateway-authoritative active-run recovery, restore the selected session’s run identity and busy controls on hello or tab return, and keep orphan cleanup only when the gateway confirms the run has ended.

## Reproduction Assessment

Yes, at source level. Current main’s reconnect hello path clears `chatRunId`, and the snapshot schema has no active-run field to restore it; I did not run a live Dashboard tab-switch reproduction in this read-only review.

## Solution Assessment

Yes in direction, but the merged end state is still missing. Gateway-authoritative active-run reconciliation is the narrow maintainable fix; the existing open PR should be reviewed, narrowed, rebased, or replaced before this issue closes.
