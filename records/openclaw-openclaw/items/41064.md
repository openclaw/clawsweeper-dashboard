## Summary

Keep open: current main still has a source-level path where reconnect or tab-return state can lose the chat busy signal, and the open linked implementation PR remains unmerged.

## What This Changes

The issue asks for Gateway Dashboard chat to preserve Stop/Queue busy controls when returning to an active chat session after a tab/view switch or reconnect.

## Best Possible Solution

Review, rebase, and land or replace https://github.com/openclaw/openclaw/pull/41067 with an additive active-run recovery contract that keeps Gateway hello/reconnect state and Dashboard chat controls aligned.

## Reproduction Assessment

Yes, source-reproducible: current main clears chatRunId/chatStream on Gateway hello while the hello snapshot has no active-run recovery field, and chat-tab refresh does not force a fresh sessions.list recovery. I did not run a live browser session in this read-only review.

## Solution Assessment

Yes for keeping this issue open: the narrow maintainable fix is to carry active abortable chat-run state through the Gateway hello/reconnect path and derive Stop/Queue from that recovered source. The existing linked PR is the right review lane, but it still needs maintainer review, rebase, and proof.
