- `impact:session-state`: The reported failure is loss of active chat run identity after reconnect or tab return.

## Summary

Keep open: current main still has a source-level busy-state mismatch after reconnect/tab return. The UI can partially recover abortability from session rows, but the exact chat run id is still cleared and queue/send logic still uses that local run id, while the linked open PR remains the implementation lane rather than a reason to close this issue early.

## What This Changes

The issue reports Gateway Dashboard chat controls reverting from Stop/Queue to New session/Send after returning to a session that still has an active run.

## Best Possible Solution

Review, land, or replace https://github.com/openclaw/openclaw/pull/41067 with an additive active-run recovery contract that keeps Stop, Queue, abort, and send behavior on one source of truth.

## Reproduction Assessment

Yes, at source level. Seed chatRunId/chatStream and process a Gateway hello or session switch path: current main clears local run identity, while queue/send still keys off chatRunId instead of the recovered session active-run flag.

## Solution Assessment

Yes for keeping this issue open. The maintainable fix is to align Gateway active-run recovery and UI queue/send/abort state, and the linked open PR is the current implementation candidate rather than a reason to close early.
