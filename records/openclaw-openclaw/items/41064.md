## Summary

Keep open: current main still has a source-level path for losing chat busy state, and the issue is paired with an open same-author PR that explicitly closes it, so cleanup should wait for that PR to land, be replaced, or be closed.

## What This Changes

Gateway Dashboard chat should preserve the same session's Stop/Queue busy state after tab/view switches or reconnects while a run is still active.

## Best Possible Solution

Review and land or replace the linked open PR with a focused fix that restores session-scoped active run identity on reconnect/tab return and makes compose actions queue while a run is still abortable.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: reconnect can clear chatRunId, the hello snapshot cannot restore it, and handleSendChat only queues when chatRunId/chatSending is set. I did not live-run the browser scenario during this read-only pass.

## Solution Assessment

Yes, the linked PR's stated direction is the right class of fix: recover active run state instead of relying only on local chatRunId. The safest maintainer path is to review that PR or replace it with a narrower equivalent if the broader restart-notice work is too large.
