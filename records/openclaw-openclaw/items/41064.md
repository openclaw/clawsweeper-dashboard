## Summary

Keep open: current main still has a source-level path where Gateway hello clears local chat run identity while the hello snapshot has no active-run recovery field, and the open closing PR at https://github.com/openclaw/openclaw/pull/41067 remains the active implementation lane.

## What This Changes

The issue reports Gateway Dashboard chat composer controls reverting from Stop/Queue to New session/Send after returning to chat while the same session still has an active run.

## Best Possible Solution

Review, rebase, and land or replace https://github.com/openclaw/openclaw/pull/41067 with an additive active-run recovery contract that keeps run identity, Stop/Queue rendering, and send/queue behavior aligned after reconnect or tab return.

## Reproduction Assessment

Yes, at source level. Seed chatRunId/chatStream, process a Gateway hello after reconnect, and current main clears the run while the hello snapshot lacks an active-run recovery field; I did not run a live browser tab-switch session.

## Solution Assessment

Yes for keeping this issue open. The narrow maintainable fix is to carry active run identity through Gateway recovery or make send/queue decisions use the same recovered active-run source as the Stop/Queue controls, then close only after the linked PR or replacement lands.
