## Summary

Keep open: current main still has the source-level path that clears Dashboard chat run identity after Gateway hello, and the open closing PR is the active implementation lane rather than a reason to close the issue early.

## What This Changes

The issue reports that Gateway Dashboard chat can lose Stop/Queue busy state after returning to a chat tab while the same session still has an active run.

## Best Possible Solution

Review and land or replace the open closing PR with an additive Gateway active-run recovery contract that keeps run identity, Stop/Queue rendering, and send/queue behavior aligned after tab return or reconnect.

## Reproduction Assessment

Yes, at source level. Seed chatRunId/chatStream and process a Gateway hello; current main clears local run identity while the hello snapshot has no active-run recovery field.

## Solution Assessment

Yes for keeping this issue open. The narrow maintainable fix is to carry active chat run identity through Gateway recovery or make composer decisions use the same recovered active-run source as the visible controls.
