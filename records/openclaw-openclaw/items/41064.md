## Summary

Keep open: current main still has a source-level path where the Dashboard can recover Stop/Queue visually from session active-run flags while the send/queue path still treats the chat as idle after `chatRunId` is cleared, and the open closing PR remains the active implementation lane.

## What This Changes

The issue requests that Gateway Dashboard chat preserve active-run busy state and avoid replacing an in-flight run after tab return, reconnect, or session restoration.

## Best Possible Solution

Review, rebase, and land or replace https://github.com/openclaw/openclaw/pull/41067 with an additive active-run recovery contract that keeps run identity, Stop/Queue rendering, and send/queue behavior aligned after reconnect or tab return.

## Reproduction Assessment

Yes. Source inspection on current main shows hello clears `chatRunId`, the snapshot has no active-run recovery field, and `handleSendChat` still queues only when `isChatBusy` sees a local `chatRunId`.

## Solution Assessment

Yes for keeping this issue open. The narrow maintainable fix is to carry active run identity through Gateway recovery or otherwise make send/queue decisions use the same recovered active-run source as the Stop/Queue controls.
