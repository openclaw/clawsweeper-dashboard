## Summary

Keep open. Current main still hides fallback and fallback-cleared reply notices when verbose mode is off, and this PR is a plausible Telegram-specific implementation, but it needs maintainer policy choice against the broader opt-in config path plus inspectable real Telegram proof before merge.

## What This Changes

The PR imports `resolveMessageChannel`, widens the fallback and fallback-cleared notice gate to verbose mode or Telegram chats, and adds reply-runner e2e coverage for Telegram verbose-off notices.

## Best Possible Solution

Choose one fallback-notice UX and land a single coherent implementation that preserves the existing fallback lifecycle/state handling, once-per-state behavior, focused tests, docs, and release notes.

## Reproduction Assessment

Yes. Source inspection of current main shows the notice guards are still verbose-only, and existing e2e cases mock fallback transitions while asserting verbose-off replies omit both fallback and cleared notice text.

## Solution Assessment

Unclear. The code change is narrow and likely correct for a Telegram-specific policy, but maintainers need to decide whether that policy is preferable to the default-false cross-channel config in https://github.com/openclaw/openclaw/pull/69399.
