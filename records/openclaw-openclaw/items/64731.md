## Summary

Keep open for maintainer review. Current main still suppresses model fallback and fallback-cleared notices outside verbose mode, so this PR is not implemented-on-main; the follow-up commit also appears to address the earlier missing-import review. The remaining blocker is product direction and freshness: this Telegram-specific default overlaps with the broader opt-in config proposed in #69399, and the PR branch needs refresh plus fresh validation against current main before merge.

## What This Changes

The PR changes the reply runner to prepend model fallback and fallback-cleared notices for Telegram chats even when verbose mode is off, while adding e2e helper parameters and Telegram/non-Telegram coverage for that behavior.

## Best Possible Solution

Pick one fallback-notice UX policy, then land the corresponding refreshed implementation. If Telegram should always surface these notices, keep this patch narrow and preserve the existing fallback lifecycle/state semantics; if the project prefers operator control across channels, use the #69399 config approach and close this PR after the chosen path lands.

## Reproduction Assessment

Yes. Current-main e2e coverage simulates `runWithModelFallback`; the verbose-off cases assert no `Model Fallback:` or `Model Fallback cleared:` text, matching the source guard that only prepends those notices under `verboseEnabled`.

## Solution Assessment

Unclear. This PR is a narrow implementation for the Telegram-specific request, but #69399 may be the more maintainable solution if maintainers want an explicit cross-channel opt-in instead of a channel-specific default.
