## Summary

Keep open. Current main still suppresses fallback and fallback-cleared notices outside verbose mode, and this PR remains a plausible Telegram-specific implementation, but it needs maintainer policy choice against the broader config-backed path and inspectable real Telegram proof before merge.

## What This Changes

The PR imports `resolveMessageChannel`, surfaces fallback notices when the effective channel is Telegram even if verbose is off, and adds e2e expectations for Telegram verbose-off fallback and fallback-cleared notices.

## Best Possible Solution

Land one canonical fallback-notice UX that preserves the existing fallback lifecycle state and once-per-state behavior, with focused tests, docs/changelog as needed, and redacted real Telegram proof.

## Reproduction Assessment

Yes. Source inspection of current main shows the notice guards are still verbose-only, and current e2e tests mock fallback transitions while asserting verbose-off replies omit both fallback and fallback-cleared notice text.

## Solution Assessment

Unclear. The patch is narrow and I found no discrete correctness bug in the diff, but maintainers need to decide whether Telegram-only always-visible notices are preferable to the open config-backed design in https://github.com/openclaw/openclaw/pull/69399.
