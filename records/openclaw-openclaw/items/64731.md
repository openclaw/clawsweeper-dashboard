## Summary

Keep open. Current main still gates user-facing fallback and fallback-cleared notices behind verbose mode, while this PR is a plausible Telegram-specific implementation that still needs maintainer UX policy choice and inspectable real Telegram proof before merge.

## What This Changes

The PR imports `resolveMessageChannel`, treats Telegram chats as eligible for fallback and fallback-cleared notices even when verbose is off, and updates reply-runner e2e coverage.

## Best Possible Solution

Choose and land one coherent fallback-notice UX that preserves the existing lifecycle state, once-per-state behavior, focused tests, docs, release notes, and real Telegram proof.

## Reproduction Assessment

Yes. Source inspection of current main shows the notice guards are still verbose-only, and existing e2e cases mock fallback transitions while asserting verbose-off replies omit both fallback and cleared notice text.

## Solution Assessment

Unclear. The PR is narrow and appears reasonable for a Telegram-specific policy, but maintainers need to decide whether that is preferable to the broader default-false config proposed in https://github.com/openclaw/openclaw/pull/69399.
