## Summary

Keep open. Current main and the latest release still have a source-supported path where a runtime-only event becomes the literal prompt text and is submitted through the normal user-message path, while display and memory export filters do not identify that plain user-role marker as internal runtime provenance.

## What This Changes

The issue reports that compaction/runtime continuation text can be persisted or displayed as if the human user authored it.

## Best Possible Solution

Define and persist first-class runtime-only event provenance so continuation context remains model-visible when needed but is never treated as human-authored content in WebChat, TUI, export, or memory ingestion.

## Reproduction Assessment

Yes, source-level with medium confidence: current main still generates the exact runtime-only marker and submits it through the normal prompt path. No live WebGUI/TUI compaction reproduction was run in this read-only review.

## Solution Assessment

Unclear. The expected outcome is right, but the best fix needs a runtime-only provenance and audience contract across submission, persistence, display/export projection, and memory ingestion rather than a literal marker filter.
