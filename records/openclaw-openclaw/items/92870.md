## Summary

Keep open. Current main still has a source-supported path where empty-transcript runtime context is converted to the literal marker as a user prompt, submitted through the normal prompt path, and not proven hidden from WebChat/export/memory projections by provenance rather than string matching.

## What This Changes

The issue reports that auto-compaction injected `Continue the OpenClaw runtime event.` into the canonical transcript as user-authored WebGUI/TUI text.

## Best Possible Solution

Model runtime-only continuation as first-class hidden runtime/session provenance, then make WebChat, transcript export, and memory ingestion honor that provenance without relying on the visible marker string.

## Reproduction Assessment

Yes, with medium confidence from source: the marker is still generated as runtime-only, submitted through the normal prompt path, and not tied to a hidden provenance check in the inspected projections. I did not run a live compaction/WebGUI reproduction.

## Solution Assessment

Unclear. The issue identifies the right boundary, but the best fix is likely a provenance-backed runtime/session contract rather than a narrow WebChat or memory string filter.
