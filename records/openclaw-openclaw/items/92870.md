## Summary

Keep open. Current main and the latest release still have a source-supported path where a runtime-only continuation uses the literal marker as the submitted prompt, and the visible chat projection does not hide that marker as non-user-authored content; related work is overlapping but not a merged complete fix.

## What This Changes

The issue reports that `Continue the OpenClaw runtime event.` can appear in compacted WebGUI/TUI transcript history as if the human user authored it.

## Best Possible Solution

Model runtime-only continuation events with first-class non-user provenance across prompt submission, session persistence, WebChat/export projection, and memory ingestion, then prove the compaction/restart path live.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live current-main reproduction; source inspection gives a credible current path from runtime-only marker generation to user-role transcript projection, but the WebGUI/TUI compaction scenario still needs live proof.

## Solution Assessment

Is this the best way to solve the issue? The issue's requested boundary is correct, but the best fix is likely a first-class runtime-only event/provenance contract rather than filtering one literal marker in WebChat.
