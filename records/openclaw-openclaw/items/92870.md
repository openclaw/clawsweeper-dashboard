## Summary

Keep open. Current main and the latest release still have a source-supported path where runtime-only continuation text is submitted through normal user-message persistence, while visible-history and memory-export consumers do not prove durable non-user provenance for that marker.

## What This Changes

The issue reports that runtime-generated continuation text can be persisted and shown as a human-authored user message after compaction.

## Best Possible Solution

Define runtime-only continuation as first-class non-user provenance through prompt submission, transcript persistence, UI projection, export, and memory ingestion while preserving model-visible resumption context.

## Reproduction Assessment

Yes, at source level: current main builds the runtime marker, submits it through normal session prompt handling, and lacks durable non-user provenance in the checked projection/export paths. No live WebGUI/TUI compaction reproduction was run in this read-only pass.

## Solution Assessment

The requested outcome is correct, but the best fix is not a literal downstream filter. Maintainers should choose a durable runtime-only provenance contract because the boundary crosses persistence, UI, export, memory, and security semantics.
