## Summary

Keep open: current main still has a source-level Feishu group session collision path because account-aware routing is called, but the canonical non-DM session key omits the Feishu account/app identity.

## What This Changes

The issue asks Feishu group sessions to distinguish multiple bot apps in the same chat so @claw, @code, and @xiaoxian do not share one session.

## Best Possible Solution

Scope Feishu group conversation/session identity by account/app for all group session modes, preserve practical compatibility for existing sessions, and add a regression with two Feishu accounts in the same group.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: two Feishu accounts routed to the same agent and group peer produce the same non-DM session key because `accountId` is dropped for group peers. I did not live-run Feishu events in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is in the Feishu group-routing/session-key boundary, with compatibility coverage for existing sessions. A broad core rewrite should be avoided unless another channel needs the same account-scoped group behavior.
