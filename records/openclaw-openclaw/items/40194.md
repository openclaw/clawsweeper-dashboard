## Summary

Keep open. Current main still builds Feishu group session keys from agent, channel, peer kind, and group peer only; the separate Feishu mention-target fix prevents non-target bots from replying, but it does not isolate multiple Feishu accounts that route to the same agent in the same group.

## What This Changes

The issue asks Feishu group sessions to distinguish multiple bot apps/accounts in the same group instead of sharing one group-only session key.

## Best Possible Solution

Design an account-aware Feishu group session boundary that preserves single-account behavior, chooses an accountId/appId migration story, and adds regression coverage for multiple Feishu accounts in one group.

## Reproduction Assessment

Yes at source level, but not via a live Feishu run. Two Feishu accounts that resolve to the same agent and default group peer still produce the same group session key because accountId is not included for non-direct peers.

## Solution Assessment

Unclear. The reported appId suffix captures the needed isolation, but the maintainable fix should choose the OpenClaw account/app-aware key shape and compatibility behavior before changing persisted group session keys.
