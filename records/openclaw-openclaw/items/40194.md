## Summary

Keep open. Current main fixes adjacent Feishu account-scoped binding paths, but the reported same-group multi-bot case still has an accountless default group peer and accountless non-direct session key, so separate Feishu bot accounts can still collide in session and last-route state.

## What This Changes

The issue reports multiple Feishu bot applications in one group sharing one session and replying through the wrong bot identity.

## Best Possible Solution

Adopt an account-aware Feishu group-session boundary, or a migration-compatible equivalent, with regression coverage and redacted same-group multi-account Feishu proof.

## Reproduction Assessment

Yes, at source level. Two Feishu accounts resolving to the same agent and default group peer still construct the same group session key because accountId is passed into routing but omitted from non-direct session keys.

## Solution Assessment

Unclear. The requested account/app suffix is directionally right, but maintainers need to choose the persisted key shape and migration behavior before changing existing group sessions.
