## Summary

Keep open: current main still routes normal Feishu group sessions through a chat-id-only group session key, so multiple Feishu app accounts in the same group can share one session bucket.

## What This Changes

The issue asks Feishu group routing to distinguish multiple bot apps in the same group by app/account identity instead of only by group chat id.

## Best Possible Solution

Add an account-aware Feishu group session-key path for multi-account group routes while preserving peer matching by plain group id and avoiding a repo-wide group-key migration unless a generic seam is needed.

## Reproduction Assessment

Yes, by source tracing. A Feishu group event for two accounts with the same `chatId` reaches `resolveAgentRoute` with different `accountId` values but still builds the same non-direct `agent:main:feishu:group:<chatId>` session key.

## Solution Assessment

Yes. The narrow maintainable fix is an account-aware Feishu group session-key/routing seam that does not change group binding semantics or rewrite all channel group keys.

## Repair Work Prompt

Fix Feishu multi-account group session collisions for https://github.com/openclaw/openclaw/issues/40194. Current main passes `account.accountId` into routing, but normal Feishu group sessions still use `agent:<agentId>:feishu:group:<chatId>`, so multiple Feishu apps in the same group can share one session. Implement an account-aware session-key path for Feishu group routes while preserving existing route binding semantics by plain group id (`peer: { kind: "group", id: "oc_xxx" }`) and preserving legacy/default-account continuity where practical. Prefer a narrow Feishu-owned change or an additive generic routing seam only if needed; do not change all channel group key shapes broadly. Add regression coverage for two Feishu accounts receiving the same group chat id producing distinct session keys, update docs for the Feishu/channel routing behavior, and add a user-facing changelog fix entry crediting the reporter if appropriate.
