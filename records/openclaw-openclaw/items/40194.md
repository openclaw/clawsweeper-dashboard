## Summary

Keep open: current main still passes the Feishu account through route resolution, but the non-direct session key used for Feishu groups remains account-agnostic, so multiple Feishu apps in the same group can still collide when they route to the same agent/session bucket.

## What This Changes

The issue asks OpenClaw to isolate Feishu group sessions by group plus app/account identity so multiple Feishu bots in one group do not share the same session or reply identity.

## Best Possible Solution

Feishu multi-account group routing should produce distinct session keys per normalized account/app for non-default multi-account group routes, preserve legacy default-account continuity where practical, and document/test the new behavior.

## Reproduction Assessment

Yes. The reporter gives concrete same-group multi-app steps, and current-main static tracing shows the same Feishu `chatId` becomes the group peer id while the non-direct session-key builder ignores accountId.

## Solution Assessment

Yes, the narrow maintainable direction is an account-aware Feishu group session fix or additive routing/session-key option, not a broad ungated rewrite of all channel group keys. The fix should include a regression test with two Feishu accounts in the same group routed to the same agent.

## Repair Work Prompt

Fix the Feishu multi-bot group session collision reported in https://github.com/openclaw/openclaw/issues/40194. Current main passes `account.accountId` into `resolveAgentRoute`, but `buildAgentPeerSessionKey` omits accountId for non-direct peers and `resolveFeishuGroupSession` defaults group peer ids to the Feishu chatId, so two Feishu accounts in the same group can share `agent:<agentId>:feishu:group:<chatId>` when routed to the same agent. Implement a conservative account-aware Feishu group isolation path, preferably in the Feishu plugin or via an additive routing/session-key option, while preserving legacy default-account/single-account continuity. Do not make an ungated global group-key migration for every channel. Check pending history and broadcast session helpers for matching account-aware behavior. Add a regression test with two Feishu accounts/apps in the same group routed to the same agent that proves their session keys differ, and keep existing topic/group-sender behavior intact. Update Feishu/channel-routing docs if the public session-key behavior changes. Add a CHANGELOG.md fix entry with reporter credit `Thanks @tontonbinary`.
