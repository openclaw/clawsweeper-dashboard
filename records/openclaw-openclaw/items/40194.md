## Summary

Current main still has a source-level Feishu multi-account group-session collision: inbound messages carry the Feishu account into routing, but the default group peer and final group session key are still group-only, so two bot accounts can share one agent/group session.

## What This Changes

Feishu group messages from multiple configured bot accounts in the same group should keep separate bot identity instead of sharing the group-only session key.

## Best Possible Solution

Design an account-aware Feishu group session boundary that preserves single-account behavior, picks an accountId/appId migration story, and adds regression coverage for multiple Feishu accounts in one group.

## Reproduction Assessment

Yes at source level, but not via a live Feishu run. Two Feishu accounts that resolve to the same agent and group peer still build the same group session key because accountId is not included for non-direct peers.

## Solution Assessment

Unclear. The reported appId suffix captures the needed isolation, but the maintainable fix should choose the OpenClaw account/app-aware key shape and compatibility behavior before changing persisted group session keys.
