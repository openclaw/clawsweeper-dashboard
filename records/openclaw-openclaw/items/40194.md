## Label Justifications

- `P1`: The issue reports a broken multi-bot Feishu group workflow where mentions can reuse the wrong bot/session identity for real users.
- `impact:session-state`: The central defect is shared Feishu group session identity across distinct bot applications/accounts.
- `impact:security`: Wrong-account bot identity in a shared group can cross intended application/account authorization boundaries.
- `impact:message-loss`: Misrouting a mention to the wrong bot/session can suppress the intended bot reply or deliver it from the wrong identity.

## Summary

Keep open: current main and the latest release still build Feishu group session identity from the group/topic/sender conversation id without the Feishu account or app identity. Existing multi-account routing and the open hook-context PR help adjacent surfaces, but they do not fix persisted group session keys or the migration/security decision this report needs.

## What This Changes

Make Feishu group routing distinguish multiple bot apps/accounts in the same group so each mention uses the correct bot identity and session.

## Best Possible Solution

Add an account-scoped canonical Feishu group conversation/session identity, with regression coverage for multiple accounts in one group and a maintainer-approved migration or documented break for existing group session state.

## Reproduction Assessment

Yes, at source level rather than live Feishu proof. Current main builds Feishu group peer ids from chat/topic/sender only, core group session keys ignore accountId, and tests assert an accountless Feishu group key.

## Solution Assessment

No, current main is not the best final state. Account-scoped bindings and hook-context accountId are useful adjacent work, but the durable fix needs account-scoped group session identity plus migration handling.
