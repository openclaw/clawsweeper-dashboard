## Label Justifications

- `P1`: The issue describes a real multi-bot channel workflow where one bot can reuse another bot's session identity in a shared Feishu group.
- `impact:session-state`: The central failure is durable session identity collision between multiple Feishu bots in one group.
- `impact:security`: Bot/account boundaries are mixed, so replies may occur under the wrong configured application identity.
- `impact:message-loss`: Messages addressed to one bot can be routed through another bot's session and produce the wrong response target or identity.

## Summary

Keep open: current main and the latest checked release still build Feishu group session identity from chat/topic/sender without a bot app or account component, so multiple Feishu apps in one group can still collide on the same durable session boundary.

## What This Changes

The issue requests Feishu group sessions to distinguish multiple bot applications in the same group instead of sharing one group-scoped session key.

## Best Possible Solution

Add a maintainer-approved Feishu group session identity that includes a stable bot/account boundary, with doctor migration, focused routing tests, and live multi-bot Feishu proof.

## Reproduction Assessment

Yes from source inspection, but not live reproduced: current main still constructs Feishu group session keys without account/app scope while the reported setup uses multiple apps in one group.

## Solution Assessment

No implemented fix is present yet; the best fix requires maintainer choice of the stable bot discriminator and an upgrade-safe session migration plan.
