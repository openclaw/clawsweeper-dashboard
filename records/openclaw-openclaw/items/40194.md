## Summary

Keep open: current main still builds Feishu group session identity from chat/topic/sender only while the shared group session-key builder omits accountId/appId for non-direct peers, so the reported multi-bot identity collision remains source-reproducible. The fix shape is clear but changes durable session identity and needs maintainer approval for accountId versus appId plus migration/security handling.

## What This Changes

The issue asks OpenClaw to isolate Feishu group sessions by bot app/account identity so multiple Feishu robots in the same group do not share one session or reply as the wrong identity.

## Best Possible Solution

Add an account/app-scoped canonical Feishu group session identity with regression coverage for multiple enabled accounts in one group and a maintainer-approved migration or explicit compatibility decision for existing group sessions.

## Reproduction Assessment

Yes, at source level rather than live Feishu proof: Feishu group peer IDs are chat/topic/sender only, and the shared group session-key builder ignores accountId. I did not run a live multi-app Feishu tenant reproduction in this read-only review.

## Solution Assessment

No, current main is not the best final state. Account-aware Feishu routing exists for some adjacent surfaces, but the durable group session key still needs account/app identity plus migration policy.
