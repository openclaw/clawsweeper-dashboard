- `impact:message-loss`: The reported behavior can route visible replies through the wrong Feishu bot identity.

## Summary

Keep open. Current main and the latest shipped tag still pass the active Feishu account into routing, but default Feishu group sessions and core non-direct session keys omit account or app identity, so multiple Feishu bot accounts in the same group can still share session and last-route state.

## What This Changes

The issue asks OpenClaw to distinguish Feishu group sessions by group plus bot application/account identity so different mentioned bots in one group do not share one session.

## Best Possible Solution

Adopt an account-aware Feishu group-session boundary, or a migration-compatible equivalent, with regression coverage and redacted same-group multi-account Feishu proof.

## Reproduction Assessment

Yes, at source level. Two Feishu accounts resolving to the same agent and default group peer still construct the same group session key because accountId reaches routing but is omitted from non-direct session keys.

## Solution Assessment

Unclear. The requested account/app suffix is directionally right, but maintainers need to choose the persisted key shape and migration behavior before changing existing group sessions.
