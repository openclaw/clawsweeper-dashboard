## Label Justifications

- `P1`: Multiple configured bots in one live group can share context and answer under the wrong identity, breaking a real channel workflow.
- `impact:session-state`: Messages received through different Feishu bot accounts converge on one persisted group-session context.
- `impact:message-loss`: The intended bot response can be suppressed, misattributed, or replaced by output using another bot account's context.
- `impact:security`: Cross-bot session sharing can expose conversation context and capabilities across identities that operators configured separately.

## Summary

Keep open. Current main and v2026.6.5 still discard the selected Feishu bot account when constructing non-direct session keys, so multiple bot accounts receiving messages from the same group can share session state and reply context. The related subagent proposal addresses per-user specialization rather than this account-isolation defect.

## What This Changes

Ensure multiple Feishu bot accounts in the same group do not share one conversation session or reply identity.

## Best Possible Solution

Add account-aware non-direct session identity at the shared routing boundary, preserve the intended single-account experience, cover sibling channels, and provide an explicit doctor migration or cleanup path for previously shared persisted sessions.

## Reproduction Assessment

Yes, from current-main source with high confidence: resolving the same Feishu group peer through two account IDs produces the same non-direct session key because accountId is omitted by the key builder. A live three-bot Feishu run was not performed during this read-only review.

## Solution Assessment

No, appending a Feishu appId directly is not the best solution. Isolation should use the normalized account identity in the shared routing/session contract, with regression coverage and a deliberate persisted-session upgrade path.
