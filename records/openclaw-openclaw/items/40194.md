## Label Justifications

- `P1`: The source-reproducible bug can route real Feishu group mentions through the wrong bot/session identity in multi-app groups.
- `impact:session-state`: Group session keys can be shared across distinct Feishu bot accounts/apps in the same group.
- `impact:security`: Wrong-account Feishu replies can cross bot/application boundaries in a shared group and expose the wrong agent context.
- `impact:message-loss`: A mention for one Feishu bot can be handled by a different session identity, suppressing or misrouting the intended reply.

## Summary

Keep open: current main still leaves Feishu group session keys scoped by group/topic/sender only, so multiple Feishu bot apps in the same group can share durable session identity even though account-scoped routing exists elsewhere.

## What This Changes

The issue asks OpenClaw to distinguish Feishu group sessions by bot application/account as well as group identity so @mentions to different Feishu bots do not reuse one session.

## Best Possible Solution

Add an account/app-scoped canonical Feishu group session identity with regression coverage for multiple enabled accounts in one group and a maintainer-approved migration or compatibility decision for existing group sessions.

## Reproduction Assessment

Yes, at source level. Current main builds Feishu group peer IDs from chat/topic/sender only, and the shared group session-key builder ignores accountId; I did not run a live multi-app Feishu tenant reproduction.

## Solution Assessment

No current implementation is not the final fix. Account-aware routing exists for bindings and direct-session scope, but durable Feishu group session identity still needs account/app scoping plus migration policy.
