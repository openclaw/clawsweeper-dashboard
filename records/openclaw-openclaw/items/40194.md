## Summary

Keep open: current main and v2026.6.8 still build Feishu group session identity from group/topic/sender only, while accountId/appId is passed separately and then dropped by the shared non-direct session-key path. The reported multi-bot group identity collision remains source-reproducible and needs maintainer review because the fix changes durable session identity and security boundaries.

## What This Changes

The issue reports that multiple Feishu bots in one group share a session identity, causing replies to come from the wrong bot identity.

## Best Possible Solution

Add an account/app-scoped canonical Feishu group session identity with regression coverage for multiple enabled accounts in one group and a maintainer-approved migration or explicit compatibility decision for existing group sessions.

## Reproduction Assessment

Yes, at source level: Feishu group peer IDs are built from chat/topic/sender only, and the shared group session-key builder ignores accountId for non-direct peers. I did not run a live multi-app Feishu tenant reproduction in this read-only review.

## Solution Assessment

No current implementation is not the final fix. Account-aware Feishu routing exists for some adjacent surfaces, but durable group session identity still needs account/app identity plus migration/security policy.
