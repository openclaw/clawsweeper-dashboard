## Summary

Keep open: current main still routes Feishu group sessions through an accountless group key, so multiple Feishu apps in the same group can converge on one session bucket even though the accountId reaches routing.

## What This Changes

The issue asks Feishu group sessions to distinguish multiple app/account identities in the same group so mentions route to the correct bot session.

## Best Possible Solution

Make Feishu group-session routing account-aware for multi-account group routes, preserve legacy default-account continuity where practical, and add focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. The report gives concrete same-group multi-app steps, and current-main source shows the same Feishu chat id becomes the group peer id while non-direct session keys ignore accountId.

## Solution Assessment

Yes. The narrow maintainable fix is account-aware Feishu group session routing, with compatibility for existing group keys, rather than a broad unversioned rewrite of every channel group key.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/40194. Current main passes Feishu accountId into routing, but Feishu group sessions still build accountless keys such as `agent:main:feishu:group:<chatId>`, so multiple Feishu apps in one group can share one session. Make Feishu non-direct group routing account-aware for multi-account group routes, or add the smallest generic seam needed for Feishu to do so. Preserve legacy default-account group session continuity where practical, do not broadly change unrelated channel group-key behavior without compatibility, and keep broadcast/topic/group_sender behavior covered. Add focused regression tests for same chat id with different Feishu accountIds producing isolated sessions and update docs/changelog for the user-facing fix.
