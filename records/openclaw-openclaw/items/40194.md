## Summary

Keep this open. Current main still routes Feishu group messages through an account-aware route lookup, but the canonical non-DM session key drops the account identity, so multiple Feishu accounts in the same group can collide on the same session.

## What This Changes

The issue reports that multiple Feishu bots in the same group share one OpenClaw session because the session key is based on group ID instead of group ID plus Feishu app identity.

## Best Possible Solution

Scope Feishu non-DM session identity by the receiving account using the existing accountId contract, while keeping user-facing group peer IDs and bindings stable.

## Reproduction Assessment

Yes, at source level. Configure two Feishu accounts that receive the same group chat id and the current route path passes different accountId values into resolveAgentRoute, but the non-direct session-key builder returns the same agent:<agent>:feishu:group:<chat> key for both.

## Solution Assessment

The reported appId suffix is directionally right but not the best exact contract. A maintainable fix should use OpenClaw accountId for session isolation, preserve plain group ids for bindings, and avoid exposing Feishu app ids in session keys unless maintainers explicitly prefer that shape.
