## Summary

Keep this open. Current main still has an account-aware Feishu route but an account-blind non-direct session key, so multiple Feishu apps in the same group can collapse onto one agent/group session despite the documented multi-account isolation model.

## What This Changes

The issue reports that multiple Feishu bots in one group share the same group-scoped session, causing replies to use the wrong bot identity.

## Best Possible Solution

Add a Feishu-owned account-scoped group-session route wrapper, patterned after WhatsApp, that preserves default-account keys and separates named-account group sessions with regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence. Feishu passes accountId into routing, but the non-direct session-key builder ignores accountId, so two Feishu accounts using the same agent and group peer resolve to the same session key.

## Solution Assessment

Yes, a narrow Feishu session-route wrapper is the best first fix. It should avoid a global session-key contract change unless maintainers explicitly choose that broader migration.

## Repair Work Prompt

Fix the Feishu multi-account group session collision reported in https://github.com/openclaw/openclaw/issues/40194. Current Feishu inbound routing passes `account.accountId` into `resolveAgentRoute`, but group session keys still collapse to `agent:<agentId>:feishu:group:<chatId>` because shared non-direct session keys ignore accountId. Prefer a Feishu-owned helper patterned after `extensions/whatsapp/src/group-session-key.ts`: keep default-account group routes unchanged, scope non-default/named-account Feishu group routes with a deterministic account-specific suffix, and preserve existing peer matching, `groupSessionScope` topic/sender peer ids, parentPeer behavior, and broadcast routing. Do not change the global `buildAgentPeerSessionKey` contract unless the patch proves that is necessary. Add regression tests showing two Feishu accounts in the same group resolve to distinct session keys while the default account remains backward-compatible. Update user-visible docs/changelog if the resulting behavior or migration note is user-facing.
