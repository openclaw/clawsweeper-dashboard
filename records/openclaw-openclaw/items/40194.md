## Label Justifications

- `P1`: Wrong Feishu bot identity in a live group can misroute user-visible channel replies and cross session/account boundaries.
- `impact:session-state`: The core symptom is multiple bot accounts reusing one group session key and therefore sharing session context.
- `impact:message-loss`: Mentions to one Feishu bot can be handled by another bot/session, which is a concrete misrouting of channel messages.
- `impact:security`: Using another bot's session/persona can expose or apply context across account boundaries in a multi-bot workspace.

## Summary

Keep open: current main and the latest release still build Feishu group session keys without account identity, so the reported multi-bot identity/session collision remains source-reproducible and needs a maintainer-reviewed fix path.

## What This Changes

Fix Feishu group multi-bot sessions so messages to different app/account bots in the same group do not reuse one group session/account identity.

## Best Possible Solution

Make Feishu group session identity account/app-aware at the canonical routing/session-key boundary, preserve intentional broadcast dedupe, and add upgrade/doctor migration plus focused Feishu multi-account tests.

## Reproduction Assessment

Yes at source level: current main routes Feishu group messages with accountId but builds non-direct session keys from channel, group kind, and peer id only. I did not run a live Feishu group with multiple app credentials.

## Solution Assessment

Unclear as an implementation detail: account/app-scoped group session identity is the right direction, but the exact key shape, migration, and broadcast behavior need maintainer approval before code should land.
