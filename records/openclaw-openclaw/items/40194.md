## Label Justifications

- `P2`: The report is a concrete Feishu multi-account group routing/session bug with limited channel scope and no core-runtime outage.
- `impact:message-loss`: The reported behavior can route replies through the wrong Feishu bot identity in a group.
- `impact:session-state`: The shared group session key can mix transcript and last-route state between Feishu bot accounts.
- `impact:auth-provider`: The failure depends on Feishu account/app identity being omitted from the persisted group session boundary.

## Summary

Keep open. Current main still carries Feishu account identity into routing, but default Feishu group peer IDs and non-direct session keys remain accountless, so multiple Feishu bot accounts in the same group can still share session and last-route state.

## What This Changes

Feishu group sessions should distinguish bot account or app identity so multiple Feishu robots in the same group do not share one session or reply as the wrong bot.

## Best Possible Solution

Adopt an account-aware Feishu group-session boundary, or a migration-compatible equivalent, with regression coverage and redacted same-group multi-account Feishu proof.

## Reproduction Assessment

Yes, at source level. Two Feishu accounts that resolve to the same agent and default group peer still build the same group session key because accountId reaches routing but is omitted from non-direct session keys.

## Solution Assessment

Unclear. The requested app/account suffix is directionally right, but the maintainable fix needs an explicit key shape and compatibility plan before changing persisted group sessions.
