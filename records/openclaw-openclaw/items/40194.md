## Summary

Keep open. Current main still carries the Feishu account id into routing, but the default group peer and non-direct session key remain group-only, so multiple Feishu bot accounts in the same group can still collide on the same agent session.

## What This Changes

The issue requests account/app-aware Feishu group session identity so multiple bot accounts in one group do not share the same group session.

## Best Possible Solution

Design an account-aware Feishu group session boundary that preserves single-account behavior, chooses an accountId/appId key and migration story, and adds regression coverage for multiple Feishu accounts in one group.

## Reproduction Assessment

Yes, at source level. Two Feishu accounts that resolve to the same agent and default group peer still build the same group session key because accountId reaches route resolution but is not included in non-direct session keys.

## Solution Assessment

Unclear. The reported appId/account suffix is directionally right, but the maintainable fix needs an explicit account-aware key shape and compatibility plan before changing persisted group sessions.
