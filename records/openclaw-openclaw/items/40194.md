## Summary

Keep open. Current main and the latest release still pass Feishu account identity into routing, but the shared group/channel session-key builder drops account identity for non-direct peers, so multiple Feishu bot accounts in the same group can still converge on the same group session.

## What This Changes

The issue reports that multiple Feishu bot applications in the same group share one OpenClaw session because group routing keys do not include the Feishu application/account identity.

## Best Possible Solution

Add account-aware Feishu group session identity at the routing/session-key boundary, with an explicit migration or fork policy for existing accountless group keys and regression coverage for same group plus multiple Feishu accountIds.

## Reproduction Assessment

Yes from source inspection, but not from a live Feishu group in this read-only review. Feishu passes `account.accountId` into route resolution, while the core group/channel session key shape ignores account id for non-direct peers.

## Solution Assessment

No final fix shape is proven yet. The requested `group + appId` isolation is directionally right, but OpenClaw’s internal contract is accountId-based, so maintainers should choose an accountId-scoped key plus an upgrade/migration policy rather than hard-coding App ID into runtime session keys.
