## Summary

Keep open. Current main still passes the Feishu account into routing, but default Feishu group sessions and shared non-direct session keys remain accountless, so same-group multi-bot traffic that resolves to the same agent can still share session state and last-route account identity.

## What This Changes

The issue reports multiple Feishu bot applications in the same group sharing one session/account identity instead of isolating replies by the mentioned bot account or app.

## Best Possible Solution

Design an account-aware Feishu group-session boundary that preserves existing single-account behavior, chooses a stable account/app key and migration story, and adds regression plus live Feishu proof.

## Reproduction Assessment

Yes, at source level. Two Feishu accounts that resolve to the same agent and default group peer still build the same group session key because accountId reaches routing but is omitted from non-direct session keys.

## Solution Assessment

Unclear. The requested app/account suffix is directionally right, but the maintainable fix needs an explicit account-aware key shape and compatibility plan before changing persisted group sessions.
