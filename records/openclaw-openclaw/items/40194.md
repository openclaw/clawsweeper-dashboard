## Summary

Keep open: current main still has a source-visible Feishu multi-account group-session collision path, and I did not find a merged fix, duplicate, protected-label reason, or paired implementation PR that would justify cleanup closure.

## What This Changes

The issue asks OpenClaw to keep Feishu group sessions distinct per bot app/account when multiple Feishu apps are installed in the same group.

## Best Possible Solution

Add a backwards-compatible Feishu group-session identity dimension for the receiving account/app, then cover two Feishu accounts in the same chat id with a regression test that proves distinct route session keys and outbound account selection.

## Reproduction Assessment

Yes from source inspection, not from live Feishu. Two Feishu accounts handling the same group chat under the same routed agent still converge on `agent:<agent>:feishu:group:<chatId>` because the group session key ignores account/app identity.

## Solution Assessment

Unclear. The reporter’s proposed `:appId:` suffix captures the missing dimension, but the maintainable fix should decide whether to key by normalized account id, app id, or a new SDK-supported account-scoped group route with migration handling.
