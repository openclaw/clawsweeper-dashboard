## Label Justifications

- `P2`: The bug is concrete and source-reproducible but scoped to Feishu multi-account group routing rather than a core-wide outage.
- `impact:session-state`: The accountless group session key can mix transcript and last-route state between Feishu bot accounts.
- `impact:message-loss`: The reported behavior can route visible replies or later delivery context through the wrong Feishu bot identity.
- `impact:auth-provider`: The failure depends on Feishu account/application identity not being part of the persisted group session boundary.

## Summary

Keep open. Current main and the latest release still pass the Feishu account id into routing but build default group session keys without account/app identity, so same-agent same-group bot applications can share transcript and last-route state.

## What This Changes

The issue requests Feishu group sessions to distinguish multiple bot applications/accounts in the same group so each mentioned bot keeps its own account/session identity.

## Best Possible Solution

Adopt an account-aware Feishu group-session boundary, or a migration-compatible equivalent, with regression coverage and redacted same-group multi-account Feishu proof.

## Reproduction Assessment

Yes, at source level. Two Feishu accounts that resolve to the same agent and default group peer still construct the same group session key because accountId is passed into routing but omitted from non-direct session keys.

## Solution Assessment

Unclear. The requested account/app suffix is directionally right, but maintainers need to choose the persisted key shape and migration behavior before changing existing group sessions.
