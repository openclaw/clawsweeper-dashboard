## Label Justifications

- `P1`: The issue can make real Feishu multi-bot group workflows use the wrong session and bot identity, affecting active channel behavior.
- `impact:session-state`: The central failure is shared persisted session identity across distinct Feishu bot accounts in the same group.
- `impact:security`: Wrong app/account session routing can expose or apply one bot identity's context and authorization boundary to another bot's interaction.
- `impact:message-loss`: Messages addressed to one Feishu bot can be handled or answered through another bot/session, which is misrouted delivery.

## Summary

Keep open: current main still builds Feishu group session keys from channel + group peer only, so two Feishu bot accounts in the same group can converge on the same session even though Feishu passes the account through routing. The report is source-reproducible and needs maintainer-approved session-key compatibility and security handling before a fix PR is safe.

## What This Changes

The issue asks Feishu group sessions with multiple bot apps in one group to be isolated by bot app/account identity instead of by group id alone.

## Best Possible Solution

Add account-scoped Feishu group session routing with a maintainer-approved migration or compatibility plan, likely using a Feishu-local pattern similar to WhatsApp's named-account group suffix, plus regression coverage for two Feishu accounts in the same group.

## Reproduction Assessment

Yes, at source level. A Feishu group event for account A and account B in the same chat passes different accountId values into routing, but current group session-key construction emits the same agent:main:feishu:group:<chatId> key; I did not run a live Feishu setup.

## Solution Assessment

No current implementation is not the final fix. The narrow maintainable solution is an account-scoped Feishu group-session shape with explicit migration/compatibility handling, not leaving generic group keys shared across Feishu bot apps.
