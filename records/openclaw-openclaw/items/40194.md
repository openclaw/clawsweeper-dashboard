## Summary

Keep open: current main still builds Feishu group session keys from the group peer id without the Feishu account/app identity, so multiple Feishu apps in the same group can still converge on one agent session bucket.

## What This Changes

Feishu group routing should keep separate identities/sessions when multiple Feishu apps are added to the same group and mentioned independently.

## Best Possible Solution

Make Feishu non-direct group session routing account-aware for multi-account group routes, preserve legacy default-account continuity where practical, and cover same-chat multi-app collisions with focused tests and docs.

## Solution Assessment

Yes. The narrow maintainable fix is an account-aware Feishu group session key or additive routing option for this channel path, not a broad rewrite of all channel group keys.

## Repair Work Prompt

Fix Feishu multi-account group session collisions from https://github.com/openclaw/openclaw/issues/40194. On current main, Feishu group routing passes accountId into `resolveAgentRoute`, but `buildAgentPeerSessionKey` omits accountId for non-direct peers, so multiple Feishu apps in the same group can share `agent:<agentId>:feishu:group:<chatId>`. Implement the narrowest account-aware Feishu group session behavior, preserving legacy/default-account continuity where practical. Prefer an additive routing/session-key seam or Feishu-owned peer/session shaping; do not broadly change unrelated channel group key behavior unless the generic seam is required and backwards-compatible. Add regression coverage for two Feishu accounts/apps in the same group routed to the same agent producing distinct session buckets, and update docs/changelog if user-facing key behavior changes.
