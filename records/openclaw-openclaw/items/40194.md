- `impact:message-loss`: Messages addressed to one Feishu bot can be handled and answered by another session.

## Summary

Keep open: current main and the latest release still build Feishu group session identity from chat/topic/sender only, while the shared non-direct session-key builder omits accountId/appId, so the reported multi-bot group identity collision remains source-reproducible.

## What This Changes

The issue reports Feishu group chat sessions colliding across multiple bot applications because the group session key is scoped by group but not by app/account identity.

## Best Possible Solution

Add an account/app-scoped canonical Feishu group session identity with regression coverage for multiple enabled accounts in one group and a maintainer-approved migration or compatibility decision for existing group sessions.

## Reproduction Assessment

Yes, at source level: current Feishu group peer ids are chat/topic/sender scoped and current non-direct session keys omit accountId/appId. I did not run a live multi-app Feishu tenant reproduction in this read-only review.

## Solution Assessment

No, current main is not the best final state. The maintainable fix belongs in Feishu group session identity/routing plus migration policy, not as a new config workaround.
