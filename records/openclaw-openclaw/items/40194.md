## Summary

Keep open: current main still builds Feishu group session identity from the group chat peer while the shared non-direct session-key path omits accountId, so multiple Feishu apps in the same group can still collide on one group session despite account-scoped routing support.

## What This Changes

Separate Feishu group sessions by both group chat and receiving Feishu account/app so multiple bots in one group do not share identity or context.

## Best Possible Solution

Implement a Feishu-owned account-aware group routing/session key path with regression tests, docs, and changelog while preserving legacy single-account behavior where feasible.

## Reproduction Assessment

Yes, source-backed. The reporter's same-group multi-app steps map directly to current main: Feishu group peer IDs are chat-scoped, and non-direct session-key construction ignores accountId.

## Solution Assessment

Yes. The narrowest maintainable fix is to make Feishu group routing/session identity account-aware in the Feishu plugin path rather than changing every channel's group session-key shape.
