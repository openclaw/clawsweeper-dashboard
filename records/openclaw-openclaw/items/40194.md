## Label Justifications

- `P1`: The report describes a broken Feishu group workflow where messages addressed to different bot apps can converge on one shared session.
- `impact:message-loss`: Messages addressed to one Feishu bot may be routed through or answered from another bot's session context.
- `impact:session-state`: The current group session key can conflate multiple Feishu account/app identities in the same group chat.
- `impact:security`: Cross-bot identity confusion can apply the wrong assistant identity, permissions, and context in a shared group.

## Summary

Current main still carries Feishu account identity as route metadata but omits it from group session keys, so this issue remains necessary. The next decision is compatibility-sensitive: changing persisted group session keys needs an explicit migration or documented break for existing group history, last-route records, and bindings.

## What This Changes

The issue reports Feishu group chats routing multiple bot apps into one shared session because group sessions are keyed only by group id instead of group plus app identity.

## Best Possible Solution

Design an account-aware Feishu group session-key and migration path that preserves or explicitly migrates existing group history, last-route records, and bindings.

## Reproduction Assessment

Yes at source level: Feishu group messages pass accountId to routing, but current group session-key construction omits accountId and current tests preserve that key shape. I did not run live Feishu multi-app traffic in this read-only review.

## Solution Assessment

Unclear until maintainers choose the upgrade contract. Adding account identity to Feishu group session keys is the likely fix direction, but persisted session history and binding compatibility need an explicit migration or documented break.
