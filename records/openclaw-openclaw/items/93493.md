## Summary

Keep open: current main and v2026.6.9 implement the main Telegram rich-message feature, opt-in config, docs, limits, and agent capability gating, but the requested unavailable-rich-method fallback is still missing from rich send/edit/draft paths. The remaining work is now narrower than the original feature request, with https://github.com/openclaw/openclaw/pull/92946 as an open candidate that still needs maintainer review before this issue can close.

## What This Changes

The issue requests Telegram Bot API rich-message delivery with account opt-in, legacy sendMessage fallback, and agent-facing rich output capability for richer tables, media, formulas, and longer messages.

## Best Possible Solution

Keep the shipped opt-in rich-message implementation, then land or explicitly reject one narrow Telegram-plugin fallback contract covering rich sends, edits, draft previews, long-message rechunking, and non-rich-error rethrow behavior.

## Reproduction Assessment

Yes. At source level, enable Telegram richMessages and use a Bot API root that rejects sendRichMessage or rich edit; current main routes rich sends, edits, and draft previews directly to raw rich methods without an unavailable-method fallback.

## Solution Assessment

No. The broad request is only partially solved on current main; the best remaining solution is a narrower maintainer-approved Telegram-plugin fallback path or an explicit decision that lagging/custom Bot API roots are unsupported for rich mode.
