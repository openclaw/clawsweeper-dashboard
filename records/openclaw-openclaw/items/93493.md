## Summary

Keep open: current main and v2026.6.9 already include the main Telegram rich-message feature, opt-in config, docs, limits, tests, and agent capability gating, but the requested automatic fallback when rich Bot API methods are unavailable is still missing across rich send/edit/draft paths. The remaining work is now a scoped Telegram-plugin fallback decision, with https://github.com/openclaw/openclaw/pull/92946 as the open candidate rather than a merged/safe superseding fix.

## What This Changes

The issue requests Telegram rich-message delivery support with richer tables/media/math/long-message handling, opt-in configuration, fallback to legacy text delivery, and agent-visible rich-output guidance.

## Best Possible Solution

Keep the shipped opt-in rich-message implementation, then land or explicitly reject one narrow Telegram-plugin fallback path that covers rich sends, edits, draft previews, long-message rechunking, and non-rich-error rethrow behavior.

## Reproduction Assessment

Yes at source level for the remaining gap: with `richMessages` enabled, current main calls raw `sendRichMessage` and rich edit/draft methods directly without an unavailable-method fallback. I did not run a live Telegram/Bot API proxy reproduction in this read-only review.

## Solution Assessment

No, current main is only the best partial solution. The maintainable end state is the existing opt-in rich path plus a carefully classified fallback, or a maintainer decision that custom/lagging Bot API roots are unsupported for rich mode.
