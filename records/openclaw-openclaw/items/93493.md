## Summary

Keep open: current main and v2026.6.9 implement the main Telegram rich-message feature, opt-in config, docs, limits, and prompt gating, but the requested unavailable-rich-method fallback is still missing in current source and the open fallback PRs are not merged safe replacements yet.

## What This Changes

The issue requests Telegram Bot API rich-message delivery with tables, inline media, math, 32k text, account-level opt-in, legacy fallback, and agent-facing rich output capability.

## Best Possible Solution

Keep standard Telegram text as the default and richMessages as explicit opt-in, then land or explicitly reject one narrow Telegram-plugin fallback covering rich sends, edits, draft previews, long-message rechunking, and non-rich-error rethrow behavior.

## Reproduction Assessment

Yes at source level: enable richMessages and use a custom or lagging Bot API root where sendRichMessage or rich edit is unavailable; current main calls the raw rich methods directly without a legacy fallback. I did not run a live Telegram proxy reproduction in this read-only review.

## Solution Assessment

No, current main is only the best partial solution: it implements opt-in rich messages and safe defaults, but not the requested unavailable-method fallback. The narrow maintainable solution is one classified Telegram-plugin fallback path, or an explicit maintainer decision that custom or lagging rich Bot API roots are unsupported for rich mode.
