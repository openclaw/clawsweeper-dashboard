## Summary

Keep open: the linked Telegram throttling report is source-backed and the PR targets the right owner boundary, but the patch currently changes fetch semantics by dropping headers from Request inputs and still lacks after-fix proof from a real OpenClaw Telegram setup.

## What This Changes

The PR wraps Telegram transport fetches to add an `OpenClawBot/<version>` User-Agent while preserving caller-provided User-Agent headers.

## Best Possible Solution

Keep the transport-level User-Agent fix, but preserve Request and init headers exactly, add focused regression coverage for that contract, and provide real OpenClaw Telegram transport proof.

## Reproduction Assessment

No for the ISP throttling end-to-end; this review cannot recreate a UK ISP layer-7 throttle. Yes for the source path: current main does not stamp Telegram transport requests, and undici adds its default User-Agent when the header is absent.

## Solution Assessment

No as written: stamping at the Telegram transport boundary is the right direction, but the wrapper must not replace headers already carried by a Request input. The safer fix is to seed from `init.headers` when provided, otherwise from `input.headers` for Request inputs, then add the User-Agent only if absent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve Request headers when stamping the UA:** `extensions/telegram/src/fetch.ts:390`
  - body: `stampUserAgent` rebuilds headers from only `init?.headers`. For `fetch(new Request(url, { headers }))` calls without an explicit `init.headers`, passing `{ headers }` to the underlying fetch replaces the Request's original header list, so existing headers are dropped while adding the UA. Seed the wrapper from the Request headers when `init.headers` is absent and add coverage for that case.
  - confidence: 0.87
