## Summary

Keep open: the branch targets a valid source-backed Telegram transport issue, but current head still loses a caller-provided User-Agent when the input is a Request, and the supplied terminal proof does not exercise the patched transport.

## What This Changes

The branch wraps Telegram transport `sourceFetch` with `OpenClawBot/<OPENCLAW_VERSION>` User-Agent stamping, adds fetch tests for default and custom init headers, and includes an unrelated image temp-dir lint test tweak.

## Best Possible Solution

Land a Telegram-owned fetch wrapper that stamps a stable versioned OpenClaw User-Agent across Bot API and media paths while preserving Request and init header semantics, with targeted regression coverage and real transport or Telegram proof.

## Reproduction Assessment

No for the ISP throttling end-to-end because it depends on a specific network path. The source path is high-confidence: current main sends Telegram transport requests without adding User-Agent, and undici 8.2.0 supplies its default when the header is absent.

## Solution Assessment

No as currently written. Stamping at the Telegram transport boundary is the right direction, but the wrapper should preserve User-Agent from Request inputs and the proof should run the patched transport or a live Telegram path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve Request-input User-Agent headers:** `extensions/telegram/src/fetch.ts:390-394`
  - body: `stampUserAgent` only copies `init.headers`. If a caller passes a `Request` that already carries a User-Agent and no init headers, this wrapper supplies a new `init.headers` object, and undici replaces the Request headers with that object; the promised custom UA preservation is lost. Build from the Request headers before stamping.
  - confidence: 0.86
