## Summary

Keep open: this is a plausible narrow Telegram bug fix, but the patch has a fetch-contract bug for Request inputs and the supplied proof does not exercise the patched transport. The transport boundary is the right place for this change, so it should be revised rather than closed.

## What This Changes

The PR wraps the Telegram transport's selected fetch with a User-Agent stamper and adds fetch tests for default and custom User-Agent headers.

## Best Possible Solution

Land a small Telegram transport wrapper that stamps a stable versioned User-Agent across Bot API and media fetches while preserving existing fetch header semantics, with targeted tests and real transport proof.

## Reproduction Assessment

No for the ISP throttling end-to-end because it depends on a specific network path. The source path is reproducible: current main sends Telegram transport requests without a User-Agent, and undici v8.2.0 supplies its default when the header is absent.

## Solution Assessment

No as currently written. Stamping at the Telegram transport boundary is the right direction, but the wrapper should preserve User-Agent on Request inputs and be proven through the patched transport or a live Telegram path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve User-Agent from Request inputs:** `extensions/telegram/src/fetch.ts:390`

Build from the Request headers when present before stamping.
