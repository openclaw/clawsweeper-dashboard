## Summary

Keep open: current main still lacks Telegram User-Agent stamping and this PR targets the right transport boundary, but the branch still needs a managed-service version-source fix plus real proof through the patched transport path.

## What This Changes

The PR wraps Telegram `sourceFetch` with an `OpenClawBot/<version>` User-Agent stamper, adds transport header tests, and makes a small lint-only image temp-dir test change.

## Best Possible Solution

Land a Telegram transport wrapper that stamps a real versioned OpenClaw User-Agent from the same runtime version sources managed services use, preserves Request/init header semantics, and is proven through the patched transport or a live Telegram probe.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the ISP throttling end-to-end because it depends on a specific network path; yes at source level because current main sends Telegram transport requests without a User-Agent and undici supplies its default when absent.

## Solution Assessment

Is this the best way to solve the issue? No as currently written; stamping at the Telegram transport boundary is the right direction, but the version source should match managed runtime environments and proof should exercise the patched path.

## Review Findings

Overall correctness: patch is incorrect
