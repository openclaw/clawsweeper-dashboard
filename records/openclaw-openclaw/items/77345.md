## Summary

Keep open: current main and v2026.6.10 still have the Vertex template-baseUrl versus concrete request-origin mismatch, so built-in Google Vertex calls do not reach the host-scoped fake-IP SSRF policy in the reported path.

## What This Changes

The issue reports that built-in Google Vertex model calls still miss fake-IP DNS SSRF allowlist handling because the resolved Vertex request origin does not match the provider `baseUrl` template.

## Best Possible Solution

Update the Google Vertex transport path so the guarded model fetch sees the resolved Vertex origin for the current request, then add focused regression coverage for global, regional, and `eu`/`us` Vertex hosts under the existing fake-IP policy.

## Reproduction Assessment

Yes, source-level: the built-in Vertex catalog keeps a `{location}` baseUrl template while requests are sent to concrete Vertex origins, so the fake-IP policy origin match fails. I did not run a live fake-IP DNS setup in this read-only review.

## Solution Assessment

Yes, the best fix direction is to feed the resolved Vertex origin into the existing host-scoped model transport policy. A gateway-wide SSRF knob is broader product work tracked separately and should not be required for this provider-specific bug.
