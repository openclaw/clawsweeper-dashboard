## Summary

Keep open: current main still has the reported source-level gap. Google Vertex builds the request URL from its resolved Vertex origin, but the shared guarded-fetch SSRF fake-IP policy is still derived only from model.baseUrl, so an unset or templated baseUrl can miss the host-scoped fake-IP allowance for aiplatform.googleapis.com.

## What This Changes

The issue asks OpenClaw to let built-in google-vertex requests work under fake-IP DNS when model.baseUrl is unset or not the concrete Vertex host.

## Best Possible Solution

Feed the resolved Google Vertex base origin into the guarded model fetch policy for the Vertex request, preserve the fake-IP-range-only hostnameAllowlist behavior, and add regression coverage for google-vertex with GOOGLE_CLOUD_LOCATION=global and no concrete model.baseUrl.

## Reproduction Assessment

Yes, at source level: a google-vertex model with no concrete model.baseUrl and GOOGLE_CLOUD_LOCATION=global builds a request to aiplatform.googleapis.com, while the guarded fetch policy still looks only at model.baseUrl. I did not run a live fake-IP proxy environment.

## Solution Assessment

No to the broad allowedHostnames suggestion; it would skip private-network checks beyond the intended fake-IP allowance. The narrower maintainable fix is to provide the resolved Vertex origin to the existing fake-IP hostname policy and keep the SSRF exception scoped.
