## Summary

Keep open: current main still builds the Google Vertex request URL from the resolved Vertex origin while the shared model SSRF fake-IP allowance is created only from `model.baseUrl`. The related fake-IP PRs are useful context, but they do not cover the unset built-in Vertex `baseUrl` path.

## What This Changes

Allow built-in `google-vertex` model requests to work behind fake-IP DNS when the derived Vertex host resolves into RFC2544 or IPv6 ULA proxy ranges.

## Best Possible Solution

Feed the resolved Google Vertex base origin into the existing host-scoped fake-IP SSRF policy path, keep the exception limited to fake-IP ranges, and add regression coverage for unset `model.baseUrl` with `GOOGLE_CLOUD_LOCATION=global`.

## Reproduction Assessment

Yes, at source level: a `google-vertex` model with no `model.baseUrl` and `GOOGLE_CLOUD_LOCATION=global` derives `aiplatform.googleapis.com`, but the guard policy sees no base URL and therefore no fake-IP allowance. I did not run a live fake-IP proxy setup.

## Solution Assessment

Yes for the narrow direction: derive or pass the concrete Vertex origin into the existing fake-IP host policy. The broader `allowedHostnames` suggestion is not the best default because it can skip private-network checks beyond the intended fake-IP ranges.
