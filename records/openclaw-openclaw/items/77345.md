- `impact:security`: The affected behavior is inside the SSRF guard's provider-host trust boundary.
- `impact:auth-provider`: The failure blocks Google Vertex model requests because provider transport metadata does not match the actual Vertex host.

## Summary

Keep open: current main still has the Vertex template-baseUrl versus resolved-request-origin mismatch, so built-in Google Vertex requests do not get the host-scoped fake-IP SSRF policy in the reported path.

## What This Changes

The issue requests fixing Google Vertex provider requests under fake-IP DNS so the SSRF guard trusts the resolved Vertex provider host without broad private-network bypass.

## Best Possible Solution

Update the Google Vertex transport path so the guarded model fetch sees the resolved Vertex base origin for the current request, then add focused regression coverage for global, regional, and `eu`/`us` Vertex hosts under the existing host-scoped fake-IP policy.

## Reproduction Assessment

Yes, at source level: built-in Vertex models carry a `{location}` baseUrl template while the request is sent to a concrete Vertex origin, so the fake-IP policy origin match fails. I did not run a live fake-IP DNS setup, so this is not marked fully reproduced.

## Solution Assessment

Yes, the best fix direction is to feed the resolved Vertex origin into the existing host-scoped transport policy. Broad private-network trust or metadata-capable hostname exceptions would be the wrong security boundary.
