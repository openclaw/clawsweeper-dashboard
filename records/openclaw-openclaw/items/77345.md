## Summary

Keep open: current main and v2026.6.10 still require the model base origin to match the request origin before applying the fake-IP SSRF policy, while built-in Google Vertex uses a `{location}` baseUrl template and sends requests to concrete Vertex origins.

## What This Changes

The issue requests a Google Vertex transport fix so built-in Vertex model requests on fake-IP DNS proxies reach the existing host-scoped fake-IP SSRF policy.

## Best Possible Solution

Feed the concrete resolved Google Vertex origin for the current request into the existing model transport SSRF policy, then cover global, regional, and `eu`/`us` Vertex fake-IP cases with focused regression tests.

## Reproduction Assessment

Yes, source-level: the built-in Vertex catalog stores a `{location}` baseUrl template while the runtime sends requests to concrete Vertex origins, so the fake-IP policy origin match fails. I did not run a live fake-IP DNS setup in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable direction is to pass the resolved Vertex request origin into the existing host-scoped SSRF policy. A gateway-wide SSRF configuration is broader product work and should not be required for this provider-specific bug.
