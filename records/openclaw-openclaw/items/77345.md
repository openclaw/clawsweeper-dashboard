## Summary

Keep open: current main still has the reported Google Vertex fake-IP SSRF gap because the provider transport creates the fake-IP policy only from a matching `model.baseUrl`, while the Google Vertex request host is resolved later from the location-specific Vertex origin.

## What This Changes

Allow built-in `google-vertex` model requests to use the existing host-scoped fake-IP SSRF allowance when Vertex DNS resolves to RFC2544 or IPv6 ULA fake-IP ranges.

## Best Possible Solution

Pass the resolved Google Vertex base origin into the existing hostname-scoped fake-IP SSRF policy path and add focused regression coverage for global and regional Vertex hosts.

## Reproduction Assessment

Yes, at source level: a global Vertex request resolves to `aiplatform.googleapis.com`, while the guarded model fetch still derives fake-IP allowance only from a matching raw `model.baseUrl`. I did not run a live fake-IP proxy setup.

## Solution Assessment

No to the broad `allowedHostnames` suggestion; the safer fix is to supply the resolved Vertex origin to the existing `hostnameAllowlist` fake-IP helper so only fake-IP ranges for that host are exempted.
