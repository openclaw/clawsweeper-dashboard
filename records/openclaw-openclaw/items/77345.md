## Summary

Keep open: current main still constructs Google Vertex request URLs from the resolved location while the shared provider SSRF fake-IP policy is created only from `model.baseUrl`, so the built-in Vertex path can fall back to the default block for fake-IP DNS answers.

## What This Changes

The issue asks for Google Vertex model calls to allow fake-IP DNS ranges for the derived `aiplatform.googleapis.com` host when the model has no concrete matching `baseUrl`.

## Best Possible Solution

Have the Google Vertex transport feed its resolved Vertex base origin into the existing host-scoped fake-IP SSRF policy, with regression coverage for unset or placeholder `model.baseUrl` and `GOOGLE_CLOUD_LOCATION=global`.

## Reproduction Assessment

Yes, at source level: a `google-vertex` model with no concrete matching `model.baseUrl` and `GOOGLE_CLOUD_LOCATION=global` builds an `aiplatform.googleapis.com` request URL, but the guarded fetch receives no fake-IP policy. I did not run a live fake-IP DNS proxy locally.

## Solution Assessment

Yes for the derived-origin approach: it is the narrowest maintainable fix because it reuses the existing fake-IP-range-only policy for the actual Vertex host. The `allowedHostnames` alternative is not the best default because it can skip private-network checks too broadly.
