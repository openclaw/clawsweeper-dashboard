## Summary

Keep open: current main still has a source-level Google Vertex gap. The shared provider transport only builds the fake-IP SSRF allowance from a matching `model.baseUrl`, while the Vertex transport derives `aiplatform.googleapis.com` at request time and still passes the original model to the guarded fetch.

## What This Changes

The issue asks OpenClaw to let built-in Google Vertex model calls work with Surge/Clash/sing-box fake-IP DNS without broadening SSRF protection.

## Best Possible Solution

Make the Google Vertex transport supply the resolved Vertex base origin to the existing host-scoped fake-IP SSRF policy, then add a regression test for unset `model.baseUrl` with global and regional Vertex hosts.

## Reproduction Assessment

Yes, at source level: a `google-vertex` request with no concrete `model.baseUrl` and `GOOGLE_CLOUD_LOCATION=global` derives `https://aiplatform.googleapis.com`, but the guard only creates the fake-IP policy from `model.baseUrl`. I did not run a live Shadowrocket/Clash/Surge setup.

## Solution Assessment

Yes, the narrow maintainable fix is to feed the resolved Vertex origin into the existing fake-IP hostname allowlist path. The broader `allowedHostnames` suggestion is not the best default because it can bypass private-network checks beyond the intended fake-IP ranges.
