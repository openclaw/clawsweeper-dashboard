## Summary

Keep open: current main still has the reported Google Vertex fake-IP gap. The shared provider SSRF policy only derives the fake-IP allowance from `model.baseUrl`, while the Google Vertex transport can derive `aiplatform.googleapis.com` dynamically and pass the original model into `buildGuardedModelFetch`.

## What This Changes

The issue asks OpenClaw to let built-in Google Vertex model calls work through fake-IP DNS without broadening the SSRF guard beyond the resolved Vertex host.

## Best Possible Solution

Have the Google Vertex transport pass the resolved Vertex base origin into the existing provider-host fake-IP policy path, with regression coverage for `GOOGLE_CLOUD_LOCATION=global` and an unset concrete `model.baseUrl`.

## Reproduction Assessment

Yes, at source level: use a `google-vertex` model whose concrete `model.baseUrl` is unset, `GOOGLE_CLOUD_LOCATION=global`, and fake-IP DNS resolving `aiplatform.googleapis.com` to `198.18.0.34`; current main derives the URL but does not derive a matching fake-IP policy from it.

## Solution Assessment

Yes for the first suggested direction: feeding the resolved Vertex origin into the existing fake-IP-only policy is the narrowest maintainable fix. The broader `allowedHostnames` suggestion is not the best default because it can bypass private-network checks beyond the intended fake-IP ranges.
