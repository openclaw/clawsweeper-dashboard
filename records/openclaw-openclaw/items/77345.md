## Summary

Keep open: current main still has a source-level gap where Google Vertex builds an `aiplatform.googleapis.com` request URL dynamically but the shared model SSRF policy only creates the fake-IP allowance from `model.baseUrl`. The linked fake-IP PRs cover adjacent paths, but they do not prove this built-in Vertex path is fixed.

## What This Changes

Ensure built-in `google-vertex` model calls get the provider-host fake-IP DNS SSRF allowance even when `model.baseUrl` is unset and the Vertex origin is derived from `GOOGLE_CLOUD_LOCATION`.

## Best Possible Solution

Make the Google Vertex transport supply the same resolved Vertex base origin to the guarded model fetch policy, preserving the existing fake-IP-range-only allowance and adding a regression test for unset `model.baseUrl` with `GOOGLE_CLOUD_LOCATION=global`.

## Reproduction Assessment

Yes, at source level: use a `google-vertex` model with no `model.baseUrl`, `GOOGLE_CLOUD_LOCATION=global`, and fake-IP DNS resolving `aiplatform.googleapis.com` to `198.18.0.34`; current main derives the request URL but passes no fake-IP policy to the guard. I did not run a live Shadowrocket/Clash/Surge setup here.

## Solution Assessment

The narrower solution is to feed the resolved Google Vertex origin into the existing provider-host fake-IP policy. The issue's broader `allowedHostnames` suggestion is not the best default because it can skip private-network checks beyond the RFC2544/fc00 fake-IP ranges.
