## Label Justifications

- `P2`: This is a provider transport bug with limited blast radius to Google Vertex users on fake-IP DNS or proxy setups.
- `impact:auth-provider`: The reported failure blocks Google Vertex model requests because provider transport metadata does not match the actual Vertex host.
- `impact:security`: The affected behavior is the SSRF guard's provider-host fake-IP exemption, which is part of OpenClaw's network security boundary.

## Summary

Keep open: current main still appears to miss the central Google Vertex fake-IP path because the built-in Vertex model carries a `{location}` baseUrl template while the actual request is sent to a resolved Vertex host, so the shared SSRF fake-IP policy is not created for the request host. The latest release inspected, `v2026.6.6`, has the same shape, and the related merged fake-IP and Vertex-host PRs do not cover this exact dynamic-origin mismatch.

## What This Changes

Fix Google Vertex model transport so fake-IP DNS addresses for the resolved Vertex API host are accepted by the existing host-scoped SSRF fake-IP policy.

## Best Possible Solution

Update the Google Vertex transport path so the guarded model fetch sees the resolved Vertex base origin for the current request, then add focused regression coverage for global, regional, and `eu`/`us` Vertex hosts under the existing host-scoped fake-IP policy.

## Reproduction Assessment

Yes, at source level: built-in Google Vertex models inherit a `{location}` baseUrl template, while the request URL is built from a concrete Vertex origin, so the shared SSRF fake-IP policy's origin match fails. I did not run a live fake-IP DNS setup, so this is not marked fully reproduced.

## Solution Assessment

Yes, the best fix direction is to feed the resolved Vertex origin into the existing host-scoped model transport policy. Broadening `allowedHostnames` or private-network trust would be a worse security-boundary fix than keeping the exemption tied to the actual provider request host.
