## Summary

Keep open: current main still derives the Google Vertex request host at send time but creates provider fake-IP SSRF policy only from `model.baseUrl`, so the reported fake-IP DNS path remains source-reproducible.

## What This Changes

The issue asks OpenClaw to allow trusted fake-IP DNS ranges for built-in Google Vertex requests whose Vertex host is derived dynamically rather than stored in `model.baseUrl`.

## Best Possible Solution

Have the Google Vertex transport supply the resolved Vertex base origin to the existing host-scoped fake-IP SSRF policy, with regression coverage for unset or templated `model.baseUrl` and `GOOGLE_CLOUD_LOCATION=global`.

## Reproduction Assessment

Yes, at source level: a `google-vertex` model with no concrete `model.baseUrl` and `GOOGLE_CLOUD_LOCATION=global` builds an `aiplatform.googleapis.com` request URL, but the shared provider SSRF policy cannot create the fake-IP allowance from that model. I did not run a live fake-IP proxy setup.

## Solution Assessment

Partly: deriving and passing the concrete Vertex origin into the existing fake-IP policy is the narrowest maintainable direction. The broader `allowedHostnames` suggestion is not the best default because it can skip private-network checks beyond the intended fake-IP ranges.
