## Summary

Keep open: current main still has the source-level gap described here. The shared provider transport only creates the fake-IP allowance when the request host matches `model.baseUrl`, while Google Vertex builds `aiplatform.googleapis.com` or a regional Vertex host dynamically before calling the guarded fetch; the related fake-IP work covers adjacent provider/web-tool paths but does not prove this Vertex dynamic-origin path is fixed.

## What This Changes

Reports that Google Vertex requests under fake-IP DNS still hit SSRF blocks because the dynamically resolved Vertex host is not available to the shared model transport fake-IP policy.

## Best Possible Solution

Pass the resolved Google Vertex origin used for the request into the existing fake-IP-range-only provider policy and add regression coverage for `GOOGLE_CLOUD_LOCATION=global` with an unset or placeholder `model.baseUrl`.

## Reproduction Assessment

Yes, at source level: use a `google-vertex` model whose `model.baseUrl` is unset or a `{location}` placeholder, set `GOOGLE_CLOUD_LOCATION=global`, and resolve `aiplatform.googleapis.com` to `198.18.0.34`; current main derives the request URL but does not derive a matching fake-IP SSRF policy. I did not run a live fake-IP DNS proxy here.

## Solution Assessment

The narrow fix is to feed the derived Vertex origin into the existing provider-host fake-IP policy. The broader `allowedHostnames` suggestion is not the best default because it can skip private-network checks beyond the intended fake-IP ranges.
