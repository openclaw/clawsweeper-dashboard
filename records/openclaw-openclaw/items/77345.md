## Summary

Keep open: current main still has a source-level gap where Google Vertex derives the `aiplatform.googleapis.com` request URL dynamically, while the shared model SSRF fake-IP allowance is only created from a matching `model.baseUrl`. The related fake-IP PRs are adjacent context, but they do not cover this built-in Vertex dynamic-origin path.

## What This Changes

Ensure built-in Google Vertex model requests get the fake-IP DNS SSRF allowance when the model lacks a matching `baseUrl`.

## Best Possible Solution

Have the Google Vertex transport feed the resolved Vertex origin into the existing host-scoped fake-IP policy and add regression coverage for unset or non-matching `model.baseUrl` with `GOOGLE_CLOUD_LOCATION=global`.

## Reproduction Assessment

Yes, at source level: a `google-vertex` request with `GOOGLE_CLOUD_LOCATION=global` builds an `aiplatform.googleapis.com` URL after `buildGuardedModelFetch(model)` captures a model whose `baseUrl` may be missing or non-matching, so the RFC2544 fake-IP policy is not attached. I did not run a live fake-IP DNS proxy setup.

## Solution Assessment

The narrowest maintainable fix is to supply the resolved Google Vertex base origin to the existing fake-IP-range policy path. The issue's broader `allowedHostnames` suggestion is not the best default because it can skip private-network checks beyond the intended fake-IP ranges.
