## Summary

Keep open: current main still has a source-level gap for the reported Google Vertex fake-IP path. The Vertex transport derives the request origin dynamically, but the shared model transport fake-IP SSRF allowance is still keyed off `model.baseUrl`, so a Vertex model with an unset or placeholder base URL will not receive the host-scoped fake-IP policy.

## What This Changes

The issue asks OpenClaw to stop blocking Google Vertex model API calls on fake-IP DNS setups when the built-in provider does not provide a concrete `model.baseUrl`.

## Best Possible Solution

Feed the resolved Google Vertex base origin into the existing guarded model fetch policy, keep the exception scoped to the derived Vertex host and fake-IP ranges, and add regression coverage for the unset-`model.baseUrl` Vertex path.

## Reproduction Assessment

Yes, at source level: a `google-vertex` model with no concrete `model.baseUrl` builds an `aiplatform.googleapis.com` request URL, but `resolveModelTransportSsrFPolicy` cannot create the host-scoped fake-IP policy. I did not run a live Shadowrocket/Clash/Surge DNS setup in this read-only review.

## Solution Assessment

No current-main fix is present. The narrowest maintainable direction is to pass the resolved Vertex origin into the existing fake-IP policy path, not to add a broad hostname bypass that skips private-network checks wholesale.
