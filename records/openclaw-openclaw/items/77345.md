## Summary

Keep open: current main still has a source-level gap where Google Vertex derives the `aiplatform.googleapis.com` request origin, but the shared model SSRF fake-IP allowance is created only from `model.baseUrl`. The related fake-IP work covers configured provider base URLs and web tools, not this derived Vertex host path.

## What This Changes

The issue requests fake-IP DNS SSRF allowance for built-in Google Vertex model calls whose request host is derived instead of stored in `model.baseUrl`.

## Best Possible Solution

Feed the resolved Google Vertex origin into the existing host-scoped fake-IP SSRF policy and add regression coverage for unset `model.baseUrl` with `GOOGLE_CLOUD_LOCATION=global`.

## Reproduction Assessment

Yes, at source level: a `google-vertex` model without `model.baseUrl` and `GOOGLE_CLOUD_LOCATION=global` builds an `aiplatform.googleapis.com` URL, but the shared guard receives no fake-IP policy because it only derives that policy from `model.baseUrl`. I did not perform live fake-IP DNS validation.

## Solution Assessment

Yes, the narrow maintainable direction is to supply the resolved Vertex base origin to the existing fake-IP-range-only provider policy. The broader `allowedHostnames` suggestion is riskier because it can skip private-network checks beyond the intended fake-IP ranges.
