## Summary

Keep open: current main still has the reported Google Vertex fake-IP SSRF gap. The Vertex transport derives `aiplatform.googleapis.com` after constructing the guarded fetch, while the provider fake-IP policy still depends on `model.baseUrl` matching the request origin.

## What This Changes

The issue requests that built-in Google Vertex model requests work with fake-IP DNS proxies instead of being blocked when the Vertex origin is derived dynamically.

## Best Possible Solution

Thread the resolved Google Vertex base origin into the existing host-scoped fake-IP SSRF helper for the Vertex request path, then add focused global and regional Vertex regression coverage.

## Reproduction Assessment

Yes, at source level: a `google-vertex` request with `GOOGLE_CLOUD_LOCATION=global` resolves to `aiplatform.googleapis.com`, while the guarded fetch policy still derives fake-IP allowance from `model.baseUrl`. I did not run a live fake-IP proxy setup.

## Solution Assessment

No to the broad `allowedHostnames` suggestion because it would bypass private-network checks beyond the intended fake-IP exception. The narrower maintainable fix is to pass the resolved Vertex origin into the existing `hostnameAllowlist`-based fake-IP policy.
