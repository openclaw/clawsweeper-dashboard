## Summary

Keep open: current main still has the reported source-level gap. Google Vertex derives the request host after constructing the guarded fetch, while the provider fake-IP SSRF policy still reads only the original model.baseUrl, so a templated or unset Vertex baseUrl can miss the aiplatform.googleapis.com allowance.

## What This Changes

Ensure built-in Google Vertex model requests on fake-IP DNS use the resolved Vertex API host for the provider SSRF allowance when model.baseUrl is templated or unset.

## Best Possible Solution

Pass the resolved Google Vertex base origin into the existing host-scoped fake-IP SSRF helper for the Vertex request path and add focused regression coverage for global and regional Vertex hosts.

## Reproduction Assessment

Yes, at source level: a google-vertex request for GOOGLE_CLOUD_LOCATION=global resolves to aiplatform.googleapis.com, while the guarded fetch policy still derives fake-IP allowance only from model.baseUrl. I did not run a live fake-IP proxy setup.

## Solution Assessment

No to the broad allowedHostnames suggestion; it would skip private-network checks beyond the intended fake-IP exception. The best fix is to supply the resolved Vertex origin to the existing hostnameAllowlist-based fake-IP policy.
