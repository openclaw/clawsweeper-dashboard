## Summary

Keep open: current main still has a source-level gap where Google Vertex derives the `aiplatform.googleapis.com` request URL after the shared model SSRF policy has only the original model `baseUrl` to compare against. The adjacent fake-IP PRs cover provider calls with concrete `model.baseUrl` values, but they do not cover this Vertex dynamic-origin path.

## What This Changes

Fix Google Vertex model calls so fake-IP DNS answers for the derived Vertex endpoint receive the provider-host fake-IP SSRF allowance when the model lacks a concrete matching `baseUrl`.

## Best Possible Solution

Have the Google Vertex transport supply the resolved Vertex origin to the existing fake-IP hostname policy, with regression coverage for global and regional locations and without broad private-network bypasses.

## Reproduction Assessment

Yes, at source level. A `google-vertex` model with no concrete matching `baseUrl` or a `{location}` template builds a request to `aiplatform.googleapis.com`, but `buildGuardedModelFetch` cannot create the fake-IP SSRF policy because the model hostname does not match the request hostname.

## Solution Assessment

Unclear for the issue’s suggested alternatives: setting or passing the resolved Vertex base origin is the narrow maintainable fix. Extending the helper to set `allowedHostnames` would be broader than needed because it can skip private-network checks beyond the intended fake-IP ranges.
