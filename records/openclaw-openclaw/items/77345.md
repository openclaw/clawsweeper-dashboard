- `impact:auth-provider`: The failure blocks Google Vertex model requests because provider transport metadata does not match the actual Vertex host.

## Summary

Keep open: current main and v2026.6.8 still have the Vertex template-baseUrl versus concrete request-origin mismatch, so built-in Google Vertex calls do not get the host-scoped fake-IP SSRF policy in the reported path.

## What This Changes

The issue requests fixing Google Vertex model transport so fake-IP DNS resolutions for the built-in `google-vertex` provider are allowed only for the resolved Vertex provider host.

## Best Possible Solution

Update the Google Vertex transport path so the guarded model fetch sees the resolved Vertex origin for the current request, then add focused regression coverage for global, regional, and `eu`/`us` Vertex hosts under the existing host-scoped fake-IP policy.

## Reproduction Assessment

Yes, at source level: built-in Vertex models carry a `{location}` baseUrl template while the request is sent to a concrete Vertex origin, so the fake-IP policy origin match fails. I did not run a live fake-IP DNS setup, so this is not marked fully reproduced.

## Solution Assessment

Yes, the best fix direction is to feed the resolved Vertex origin into the existing host-scoped model transport policy. A gateway-wide SSRF knob is broader product work tracked separately and should not be required for this provider-specific bug.
