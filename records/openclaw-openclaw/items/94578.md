## Summary

Keep this PR open for normal maintainer review: current main and v2026.6.9 still use the 512-byte recent-session header probe, while this focused patch replaces it with a bounded first-line reader, adds regression coverage, and has sufficient local real-behavior proof. I found no blocking code or security findings; the remaining merge risk is that `check-shrinkwrap` and `checks-node-core-tooling` are currently failing and should be rerun or confirmed unrelated before merge.

## What This Changes

The PR replaces the 512-byte recent-session header probe with a bounded first-line reader and adds regression tests for long and oversized session headers.

## Best Possible Solution

Land this bounded first-line session-header reader with its regression tests after maintainer review and after the failing checks are resolved or confirmed unrelated.

## Reproduction Assessment

Yes. Source inspection on current main shows `loadEntriesFromFile` reads the full JSONL transcript, while `findMostRecentSession` rejects a candidate after parsing only the first 512 bytes, so a header longer than 512 bytes can be loadable but skipped before `continueRecent` resumes it.

## Solution Assessment

Yes. The bounded first-line scanner fixes the validator in the discovery path without loading whole transcripts during directory scans, and the added tests cover both long valid headers and oversized-header fallback.
