## Summary

Keep open. Current main already removed the plain-audit plugin-runtime collector cost, but the hidden backup-directory scanner boundary reported here still exists in source and was also called out by the maintainer update in the issue discussion.

## What This Changes

The issue requests that `openclaw security audit --json` remain bounded/read-only by avoiding plugin runtime dependency staging and by not treating hidden or backup extension directories as installed plugins during audit scans.

## Best Possible Solution

Keep the current plain-audit runtime collector behavior, then add a small audit-side installed-plugin directory filter shared by the code-safety and plugin-trust scanners so hidden, backup, disabled, and `node_modules` entries under `stateDir/extensions` are ignored with focused regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is clear: place `.openclaw-install-backups` or `node_modules` under `stateDir/extensions` and the current `listInstalledPluginDirs` helpers will return it as a plugin candidate; the plain-audit runtime-staging half is already covered by the read-only scope test.

## Solution Assessment

Yes. The narrow maintainable fix is to filter scanner candidates at the installed-plugin directory boundary, preferably with one shared helper, without reworking the plugin runtime loader again.
