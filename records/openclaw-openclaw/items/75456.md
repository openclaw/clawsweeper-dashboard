## Summary

Keep open. Current main has addressed the plain-audit plugin-runtime staging half, but the installed-plugin audit directory boundary still treats hidden backup directories as plugin candidates in both trust and deep code-safety scans.

## What This Changes

Make `openclaw security audit --json` complete predictably by avoiding unnecessary plugin runtime work and excluding hidden or backup extension directories from installed-plugin audit scans.

## Best Possible Solution

Keep the runtime-collector fix, then add a shared audit-side installed-plugin directory filter aligned with plugin discovery and doctor cleanup, with regression tests for dotdirs, `.openclaw-install-backups`, `node_modules`, backup, and disabled directory names.

## Reproduction Assessment

Yes. On current main, a direct child such as `stateDir/extensions/.openclaw-install-backups` is returned by both audit `listInstalledPluginDirs` helpers because they only check `entry.isDirectory()` and `filter(Boolean)`; plain audit reaches the trust scanner, and deep audit reaches code safety too.

## Solution Assessment

Yes for the remaining work. Filtering at the audit installed-plugin enumeration boundary is narrower than changing plugin loading again, while preserving the already-fixed runtime collector behavior.
