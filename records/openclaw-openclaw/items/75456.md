## Summary

Keep open. Current main still matches the reported audit-boundary bug: the CLI audit path leaves plugin security collector loading enabled, collector discovery reaches the metadata snapshot loader without disabling bundled runtime-deps staging, and the audit-side installed-plugin scanners still accept hidden or backup directories under the extensions state directory. The later cross-reference #75508 is related but broader; #75032 and #75429 do not supersede this narrower collector-staging plus backup-directory scan issue.

## What This Changes

The issue reports that `openclaw security audit --json` can hang because audit collector discovery stages plugin runtime dependencies and hidden backup extension directories are scanned as installed plugins.

## Best Possible Solution

Make `openclaw security audit` genuinely read-only at this boundary: collector discovery should avoid bundled runtime-deps staging while still running intended collectors, and installed-plugin audit scans should ignore hidden, backup, `node_modules`, and non-plugin directories under the extensions state directory.

## Reproduction Assessment

Yes for the core issue. Current main still has a source-level path from `security audit --json` to plugin collector discovery with runtime-deps staging enabled by default, and both audit installed-plugin directory helpers still include hidden backup directories as candidates.

## Solution Assessment

Yes. A narrow audit-scoped fix is the best path: add or pass an `installBundledRuntimeDeps: false` style option for collector discovery, add installed-plugin directory filtering to both audit scanners, and cover both with targeted tests rather than waiting for the broader public runtime-deps opt-out work.

## Repair Work Prompt

Fix openclaw/openclaw#75456. The observable bug is that `openclaw security audit --json` can hang or run for a long time because the read-only security audit collector discovery path can stage bundled plugin runtime dependencies, and the audit scanners treat hidden install-backup directories under `stateDir/extensions` as installed plugins. Keep the fix narrow: do not change the repo-wide bundled runtime-deps policy, dependency versions, package-manager behavior, or plugin activation semantics beyond the audit path. In `src/security/audit.ts` and/or `src/plugins/runtime/metadata-registry-loader.ts`, add/pass an explicit audit-safe option equivalent to `installBundledRuntimeDeps: false` so plugin security collector discovery does not trigger bundled runtime-deps staging, while preserving intended collector execution for configured plugins. In `src/security/audit-extra.async.ts` and `src/security/audit-plugins-trust.ts`, filter installed plugin directory candidates to ignore hidden directories, `node_modules`, `.openclaw-install-backups`, and existing backup/disabled naming patterns; reuse or mirror the existing plugin discovery ignore semantics where appropriate without creating a core/plugin ownership leak. Add focused regression tests in `src/security/audit-plugin-readonly-scope.test.ts`, `src/plugins/runtime/metadata-registry-loader.test.ts`, `src/security/audit-extra.async.test.ts`, and `src/security/audit-plugins-trust.test.ts`. Add a single-line active `CHANGELOG.md` fix entry crediting `Thanks @liushuaiiu`. Before handoff, run the targeted tests and formatter check locally, then run `pnpm check:changed` in Testbox per repo policy.
