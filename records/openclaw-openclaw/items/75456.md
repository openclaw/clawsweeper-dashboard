## Summary

Keep open. Current main addresses the plugin-runtime dependency staging half for plain security audits, and the maintainer update agrees that part is obsolete, but the hidden backup-directory boundary still remains in source: both audit helpers still turn every direct directory under `stateDir/extensions` into a plugin candidate.

## What This Changes

The issue asks for `openclaw security audit --json` to stay bounded/read-only by avoiding plugin runtime dependency staging and by not scanning hidden install-backup directories as installed plugins.

## Best Possible Solution

Keep the merged plain-audit runtime behavior, then add a shared audit-side installed-plugin directory filter so hidden directories, `.openclaw-install-backups`, and `node_modules` under `stateDir/extensions` are not treated as installed plugins, with focused regression tests in both scanner paths and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to place `.openclaw-install-backups` under `stateDir/extensions`; current `listInstalledPluginDirs` helpers include any direct directory name and both consumers iterate that list as plugin candidates.

## Solution Assessment

Yes. The narrow maintainable solution is to filter installed-plugin candidates at the audit directory boundary and add tests, rather than changing the global plugin loader or disabling deep plugin scanning behavior.
