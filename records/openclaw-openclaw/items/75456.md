## Summary

Keep open. Current main has resolved the plain `security audit` plugin-runtime collector/staging half, but both installed-plugin audit scanners still enumerate hidden backup directories under `stateDir/extensions`, so the report is only partially fixed.

## What This Changes

`openclaw security audit --json` should stay bounded/read-only by avoiding plugin runtime dependency staging and by not treating hidden backup directories under `stateDir/extensions` as installed plugins.

## Best Possible Solution

Keep the shipped runtime-collector behavior, then add a shared audit-side installed-plugin directory filter with regression coverage for dotdirs, `.openclaw-install-backups`, `node_modules`, backup, and disabled entries.

## Reproduction Assessment

Yes. On current main, a `stateDir/extensions/.openclaw-install-backups` directory would pass both audit helpers because they only require `entry.isDirectory()`, after which the code-safety and trust scanners treat it as an installed plugin candidate.

## Solution Assessment

Yes for the remaining work. Filtering at the audit installed-plugin directory boundary is narrower than changing global plugin loading and matches existing plugin discovery/doctor cleanup semantics.
