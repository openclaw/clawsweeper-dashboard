## Summary

Keep open. Current main still shows the reported audit-boundary behavior in source: routine `security audit --json` does not disable plugin security collector loading, collector fallback can still reach metadata snapshot loading without a bundled-runtime-deps opt-out, and audit-side installed-plugin directory scans still accept hidden/backup directories. The related open PR #75613 is a relevant implementation candidate for the plain-audit runtime path, but this issue should remain open until that lands and the backup-directory scan boundary is also covered or intentionally split.

## What This Changes

The issue requests that `openclaw security audit --json` complete predictably by avoiding plugin runtime dependency staging during routine audit collector discovery and by excluding hidden backup extension directories from installed-plugin scan candidates.

## Best Possible Solution

Keep this issue open until routine `security audit --json` stays off plugin runtime collector imports by default and the audit installed-plugin scanners reject hidden, backup, and non-plugin directories. The clean end state is either #75613 expanded to cover both boundaries or #75613 merged for the runtime half plus a narrow follow-up for the scan filters, with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction. Current main traces from the CLI into default-enabled collector discovery and from both audit installed-plugin scan helpers into unfiltered directory listings; the live timeout/orphan-process symptom was not reproduced in this read-only review.

## Solution Assessment

Yes, the requested narrow audit-scoped direction is the best maintainable fix: keep routine audit collector discovery read-only/off plugin runtimes unless deep or explicitly requested, and filter hidden/backup/non-plugin directories in the audit scanners rather than changing global plugin loader behavior.
