## Summary

Keep open. Current main fixes the plain `security audit --json` plugin-runtime collector path, but the hidden backup-directory scanner boundary still exists in the audit code that enumerates installed plugin directories.

## What This Changes

The issue reports that `openclaw security audit --json` can hang because audit discovery may stage plugin runtime dependencies and audit scanners may treat hidden extension backup directories as installed plugins.

## Best Possible Solution

Keep the merged plain-audit runtime behavior, then add a narrow audit-side installed-plugin directory filter shared by both scanner helpers so hidden, backup/disabled, and `node_modules` entries under `stateDir/extensions` are not treated as installed plugins, with focused regression tests and a changelog fix entry.

## Reproduction Assessment

Yes for the remaining issue: current main still maps every direct directory under `stateDir/extensions` in both audit helpers, so a `.openclaw-install-backups` fixture would be scanned or counted as a plugin candidate. Live timeout reproduction is not necessary to prove that boundary bug.

## Solution Assessment

Yes, the narrowest maintainable fix is to filter candidates at the audit installed-plugin directory boundary, ideally consistent with plugin discovery’s existing backup/disabled ignore policy. Changing the global plugin loader or reworking runtime collector behavior would be broader than the remaining problem.
