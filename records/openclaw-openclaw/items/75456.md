## Summary

Keep open. Current main still has both reported boundaries: the plain security audit path does not opt out of plugin security collector loading, and the audit-side installed-plugin directory scans still accept hidden extension directories. The open #75613 PR is a relevant implementation candidate for the runtime-loading half, but it is not merged in current main and the hidden backup-directory scan still needs coverage.

## What This Changes

The issue asks for `openclaw security audit --json` to avoid plugin runtime dependency staging and hidden backup extension scans so the audit completes predictably.

## Best Possible Solution

Keep this open until routine JSON audits stay on a read-only, non-runtime plugin discovery path and the audit plugin scanners reject hidden, backup, and non-plugin directories. The clean path is to expand #75613 to cover both boundaries or merge it for the runtime half and land a narrow follow-up for the scan filters.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path: trace `security audit --json` into `runSecurityAudit`, the default-enabled collector discovery, and the unfiltered audit directory listings; I did not run a live timeout/orphan-process reproduction during this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to disable plugin collector runtime loading for plain audits, keep deep or explicit internal calls able to load collectors, and filter hidden/backup/non-plugin directories in the audit scan helpers rather than changing global plugin loader behavior.
