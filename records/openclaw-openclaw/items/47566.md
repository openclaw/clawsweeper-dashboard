## Summary

Keep open: current main still has no plugin scaffold command, so this PR is not obsolete, but the branch is not merge-ready because the patch conflicts with the scoped plugin-id contract, omits docs-aligned activation metadata, lacks docs for the new public command, and still needs contributor real behavior proof.

## What This Changes

The PR adds an `openclaw plugins init [directory]` CLI command that scaffolds `openclaw.plugin.json`, `index.ts`, and `package.json` with flags and unit tests.

## Best Possible Solution

Land a rebased plugin-init command that reuses canonical plugin-id validation, generates activation-aware starter manifests, updates the plugin CLI and authoring docs, and includes redacted after-fix scaffold/install proof.

## Reproduction Assessment

Yes. Static inspection of the PR diff reproduces the review issues: the PR rejects scoped ids that current main accepts, and the generated manifest lacks activation metadata current docs ask authors to set intentionally.

## Solution Assessment

No. The feature direction is reasonable, but the patch should be rebased, reuse the existing plugin-id contract, generate activation-aware manifests, document the public CLI surface, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Allow scoped plugin ids during scaffolding:** `src/cli/plugins-init.ts:30-31`
  - body: `plugins init` rejects any id containing `/`, so valid scoped ids such as `@scope/name` fail even though the current install path accepts and tests that shape. Reuse the canonical plugin-id validator or mirror that contract so scaffolded ids match installable ids.
  - confidence: 0.95
- **[P2] Emit explicit startup activation metadata:** `src/cli/plugins-init.ts:72-80`
  - body: The scaffolded manifest omits `activation`, but current plugin docs say every plugin should choose `activation.onStartup` intentionally and omission no longer startup-loads. Generate an explicit activation block, likely `onStartup: false` by default or via an option, so new plugins start from the documented contract.
  - confidence: 0.87
- **[P3] Document the plugins init command:** `src/cli/plugins-cli.ts:828-836`
  - body: This registers a new public `openclaw plugins init` subcommand without updating the CLI or plugin-authoring docs, so users cannot discover the command, options, generated files, or intended install flow from the published reference.
  - confidence: 0.82
