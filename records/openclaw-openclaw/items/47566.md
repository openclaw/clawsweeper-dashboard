## Summary

Keep open: current main still has no plugin scaffold command, but this PR is not merge-ready because it conflicts with current main, rejects scoped plugin ids that main accepts, emits a starter manifest that does not follow current activation guidance, lacks docs, and has no contributor-supplied real behavior proof.

## What This Changes

The branch adds an `openclaw plugins init [directory]` CLI subcommand, a scaffold generator for `openclaw.plugin.json`, `index.ts`, and `package.json`, and unit tests for generation, validation, and overwrite behavior.

## Best Possible Solution

Land a rebased plugin-init command that reuses the canonical plugin-id validator, emits docs-aligned activation-aware starter manifests, updates the plugin CLI/authoring docs, and includes redacted real behavior proof.

## Reproduction Assessment

Yes. Static inspection is enough: the PR rejects scoped ids that current main accepts, and it generates manifests without the activation metadata current docs ask plugin authors to set intentionally.

## Solution Assessment

No. The feature direction is reasonable, but the patch should be rebased, reuse the existing plugin-id contract, generate activation-aware manifests, and document the new public CLI surface before merge.
