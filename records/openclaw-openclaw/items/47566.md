## Summary

Keep open. Current main still lacks a plugin scaffold command, and this PR is a useful DX feature, but the branch still conflicts with current plugin-id and manifest contracts and needs real behavior proof before merge.

## What This Changes

The PR adds `openclaw plugins init [directory]` to scaffold `openclaw.plugin.json`, `index.ts`, and `package.json`, plus unit tests for the new command.

## Best Possible Solution

Land a rebased plugin-init command that reuses canonical plugin-id validation, emits a docs-aligned starter manifest, and documents the public CLI surface.

## Reproduction Assessment

Yes. Static inspection is enough: the PR validator rejects `/` while current main accepts scoped plugin ids, and the generated manifest omits activation metadata that current docs ask plugin authors to set intentionally.

## Solution Assessment

No. The feature direction is reasonable, but the patch should reuse the existing install-path contract, generate activation-aware manifests, and add user-facing docs before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Allow scoped plugin ids during scaffolding:** `src/cli/plugins-init.ts:29-31`
  - body: Current install validation accepts scoped ids such as `@scope/name` and has coverage for them, but this new validator rejects every `/`. Users can install scoped plugin ids that the scaffold command cannot create; reuse the canonical validator from `src/plugins/install-paths.ts` or mirror that contract.
  - confidence: 0.95
- **[P2] Emit explicit activation metadata in the scaffold:** `src/cli/plugins-init.ts:69-80`
  - body: The generated manifest only writes id, optional fields, and `configSchema`, but current docs say every plugin should set `activation.onStartup` intentionally and omission no longer startup-loads a plugin. Include explicit activation metadata, likely `onStartup: false` by default or through an option.
  - confidence: 0.87
- **[P3] Document the new plugins init command:** `src/cli/plugins-cli.ts:824-836`
  - body: This adds a public CLI subcommand, but the plugin CLI reference does not list or explain it. Add docs so users can discover the scaffold flow and understand its generated files/options.
  - confidence: 0.82
