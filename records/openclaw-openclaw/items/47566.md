## Summary

Close as ClawHub/plugin-ecosystem tooling scope: the requested scaffold is useful authoring DX, but current project policy keeps optional plugin authoring/discovery/promotion machinery outside core unless a missing core extension API is identified, and this PR does not identify such a gap.

## What This Changes

The PR adds an `openclaw plugins init [directory]` CLI scaffold command, generator, and unit tests that write a plugin manifest, entrypoint, and package metadata.

## Best Possible Solution

Keep plugin scaffolding as separate ClawHub/npm authoring tooling that consumes the documented SDK, manifest, package-contract, and snippet sources of truth; open a core issue only if that external tool exposes a missing plugin API.

## Reproduction Assessment

Yes. Static inspection against current main reproduces the PR-level issues: the branch adds a new core CLI command, rejects scoped plugin IDs that current install accepts, and generates files missing current activation and package metadata contracts.

## Solution Assessment

No. The feature direction is understandable, but adding a one-off core scaffold command is not the best fit under current plugin/ClawHub scope; an external ClawHub/npm scaffolder should reuse the existing docs snippets and package contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reuse canonical plugin ID validation:** `src/cli/plugins-init.ts:30-31`
  - body: `plugins init` rejects every `/`, so valid scoped plugin IDs such as `@team/my-plugin` fail even though current install and install-dir validation support `@scope/name`. Reuse the canonical validator or keep this command exactly aligned with it.
  - confidence: 0.95
- **[P2] Generate ClawHub-ready package metadata:** `src/cli/plugins-init.ts:82-88`
  - body: The generated `package.json` only writes `openclaw.extensions`, but current external code plugin validation requires compatibility/build metadata such as `openclaw.compat.pluginApi` and `openclaw.build.openclawVersion`. A fresh scaffold would fail the documented publish contract.
  - confidence: 0.9
- **[P2] Emit explicit startup activation metadata:** `src/cli/plugins-init.ts:72-80`
  - body: The generated manifest omits `activation`, while current docs say every plugin should choose `activation.onStartup` intentionally and omission no longer means implicit startup loading. New scaffolds should start with the current manifest contract.
  - confidence: 0.88
- **[P3] Generate the canonical entry helper:** `src/cli/plugins-init.ts:36-40`
  - body: The scaffold emits a bare default register function instead of the documented `definePluginEntry` shape with entry metadata. New plugin projects should start from the canonical SDK helper rather than a compatibility-shaped export.
  - confidence: 0.78
