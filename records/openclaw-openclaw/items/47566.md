## What This Changes

The PR adds an `openclaw plugins init [directory]` CLI scaffold command plus tests to generate a plugin manifest, entry file, and package metadata.

## Best Possible Solution

Land a rebased plugin-init command that reuses canonical plugin ID validation, emits current activation and `definePluginEntry` starter metadata, documents the command, and includes redacted real CLI proof.

## Reproduction Assessment

Yes. Static inspection of the PR diff against current-main plugin contracts reproduces the patch defects: scoped IDs are rejected in the PR validator while current install accepts `@scope/name`, and the scaffold omits current activation and entry helper conventions.

## Solution Assessment

No. The feature direction is reasonable, but this patch should be rebased and aligned with current plugin ID, activation, entrypoint, and documentation contracts before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Allow scoped plugin IDs during scaffolding:** `src/cli/plugins-init.ts:29-31`
  - body: `plugins init` rejects any ID containing `/`, so valid scoped IDs like `@team/my-plugin` fail even though the install path accepts `@scope/name`. Reuse the canonical plugin ID validator or keep the scaffold validator aligned with it.
  - confidence: 0.95
- **[P2] Emit explicit startup activation metadata:** `src/cli/plugins-init.ts:67-77`
  - body: The generated manifest omits `activation`, but current plugin docs say every plugin should choose `activation.onStartup` intentionally and omission no longer startup-loads implicitly. New scaffolded plugins should start from the current manifest contract, likely with `onStartup: false` unless an option says otherwise.
  - confidence: 0.88
- **[P3] Generate the canonical plugin entry helper:** `src/cli/plugins-init.ts:36-40`
  - body: The scaffold emits a bare default register function, while current authoring docs use `definePluginEntry` with `id`, `name`, `description`, and `register`. New projects should start from the canonical helper rather than the compatibility-shaped function export.
  - confidence: 0.78
- **[P3] Document the plugins init command:** `src/cli/plugins-cli.ts:824-836`
  - body: This adds a new public `openclaw plugins init` subcommand, but the CLI and plugin authoring docs are unchanged. Add docs for the command, options, generated files, and intended install flow so the public surface is discoverable.
  - confidence: 0.84
