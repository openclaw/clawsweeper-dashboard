## Summary

Keep open. Current main still does not implement an `openclaw plugins init` command, so the PR is not obsolete, but the proposed patch should not merge as-is because it diverges from current plugin-id validation and scaffolds a template that is not aligned with current activation/docs policy.

## What This Changes

This PR adds an `openclaw plugins init [directory]` CLI subcommand that scaffolds `openclaw.plugin.json`, `index.ts`, and `package.json` with options, validation, overwrite handling, and unit tests.

## Best Possible Solution

Land a rebased version that reuses the canonical plugin-id validation contract, generates a scaffold consistent with current plugin entry and activation guidance, and updates the plugin CLI docs plus changelog for the new public command.

## Reproduction Assessment

Yes. Static inspection is enough: the PR validator rejects `@scope/name`, while current main accepts and tests that scoped ID through `resolvePluginInstallDir`. The generated manifest also omits activation metadata that current status code warns about.

## Solution Assessment

No. The feature direction is useful, but this patch duplicates a stricter validator and emits an outdated scaffold; the safer fix is to share `src/plugins/install-paths.ts` validation and align the template/docs with current plugin contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Allow scoped plugin ids during scaffolding:** `src/cli/plugins-init.ts:30-31`
  - body: Current install validation accepts scoped ids such as `@scope/name` and has regression coverage for that path, but this new validator rejects any `/`. That means `openclaw plugins init --id @scope/name` cannot scaffold a plugin id the rest of the plugin system supports; reuse the install-path validator or mirror its scoped-id rules.
  - confidence: 0.95
- **[P2] Emit explicit activation metadata in the scaffold:** `src/cli/plugins-init.ts:67-77`
  - body: The generated manifest has no `activation.onStartup`, so a new plugin with no static capability metadata immediately relies on the deprecated implicit startup sidecar path and triggers current status warnings. Set an explicit activation default, or expose it through the scaffold options/docs, so generated plugins follow the current authoring contract.
  - confidence: 0.86
- **[P3] Document the new plugins init command:** `src/cli/plugins-cli.ts:824-836`
  - body: This adds a user-facing CLI subcommand, but the PR does not update the plugins CLI docs or changelog. Add the command to the public plugin docs and add the required user-facing feature changelog entry before merge.
  - confidence: 0.84
