## Summary

Keep open: current main does not implement plugin scaffolding, and this useful PR is not merge-ready because its validator conflicts with current scoped-id support, its manifest scaffold omits activation metadata required by docs, and the public command lacks docs/changelog coverage.

## What This Changes

The PR adds an `openclaw plugins init [directory]` CLI command that scaffolds `openclaw.plugin.json`, `index.ts`, and `package.json`, plus unit tests.

## Best Possible Solution

Land a rebased plugin-init command that reuses canonical plugin-id validation, emits a docs-aligned starter manifest, and updates docs/changelog.

## Reproduction Assessment

Yes. Static inspection is enough: the PR validator rejects `/` while current main accepts and tests `@scope/name`, and the generated manifest omits activation metadata that current docs require authors to set intentionally.

## Solution Assessment

No. The feature direction is useful, but the current patch should reuse the existing install-path contract, generate activation-aware scaffolds, and document the new public command before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Allow scoped plugin ids during scaffolding:** `src/cli/plugins-init.ts:30-31`
  - body: Current install validation accepts scoped ids such as `@scope/name` and has coverage through `resolvePluginInstallDir`, but this new validator rejects every `/`. Users can install scoped plugin ids that the scaffold command cannot create; reuse `validatePluginId` from `src/plugins/install-paths.ts` or mirror its scoped-id rules.
  - confidence: 0.95
- **[P2] Emit explicit activation metadata in the scaffold:** `src/cli/plugins-init.ts:72-80`
  - body: The generated manifest omits `activation.onStartup`, while current docs say every plugin should set it intentionally and omission no longer startup-loads a plugin. Include explicit activation metadata, likely `onStartup: false` by default or an option, so new projects follow the current manifest contract.
  - confidence: 0.86
- **[P3] Document the new plugins init command:** `src/cli/plugins-cli.ts:829-836`
  - body: This adds a public CLI subcommand, but the PR does not update the plugin CLI docs, the authoring guide if this becomes the canonical scaffold path, or `CHANGELOG.md`. Add those before merge so users can discover the command and the user-facing feature is recorded.
  - confidence: 0.84
