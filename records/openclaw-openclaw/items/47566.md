## Summary

Keep open: current main still lacks the scaffold command, so the feature is not obsolete, but the PR should not merge as-is because it conflicts with current scoped-plugin-id and activation contracts and misses required public docs/changelog coverage.

## What This Changes

The PR adds an `openclaw plugins init [directory]` CLI subcommand with scaffold generation for `openclaw.plugin.json`, `index.ts`, `package.json`, and unit tests.

## Best Possible Solution

Land a rebased version that reuses the canonical plugin id validator, generates a manifest aligned with current activation guidance, and documents/changelogs the new public command.

## Reproduction Assessment

Yes. Static inspection is enough: the PR validator rejects `/`, while current main accepts and tests `@scope/name`; the generated manifest also omits activation metadata that current docs require authors to set intentionally.

## Solution Assessment

No. The feature direction is useful, but the current patch duplicates an incompatible validator and emits an outdated scaffold; the safer path is to reuse the existing install-path contract and update docs/changelog with the command.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Allow scoped plugin ids during scaffolding:** `src/cli/plugins-init.ts:30-31`
  - body: Current install validation accepts scoped ids such as `@scope/name` and has coverage through `resolvePluginInstallDir`, but this new validator rejects every `/`. Users can install scoped plugin ids that the scaffold command cannot create; reuse `validatePluginId` from `src/plugins/install-paths.ts` or mirror its scoped-id rules.
  - confidence: 0.95
- **[P2] Emit explicit activation metadata in the scaffold:** `src/cli/plugins-init.ts:67-77`
  - body: The generated manifest omits `activation.onStartup`, while current docs say every plugin should set it intentionally and omission no longer startup-loads a plugin. Include explicit activation metadata, likely `onStartup: false` by default or an option, so new projects follow the current manifest contract.
  - confidence: 0.86
- **[P3] Document the new plugins init command:** `src/cli/plugins-cli.ts:824-836`
  - body: This adds a public CLI subcommand, but the PR does not update the plugin CLI docs, authoring guide if the scaffold becomes canonical, or `CHANGELOG.md`. Add those before merge so users can discover the command and the user-facing feature is recorded.
  - confidence: 0.84
