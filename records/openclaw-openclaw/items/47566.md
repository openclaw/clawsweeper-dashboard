## Best Possible Solution

Land a rebased plugin-init command that reuses canonical plugin ID validation, emits current activation and `definePluginEntry` starter metadata, documents the command, and includes redacted real CLI proof.

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Allow scoped plugin IDs during scaffolding:** `src/cli/plugins-init.ts:30-31`
  - body: `plugins init` rejects any ID containing `/`, so valid scoped IDs like `@team/my-plugin` fail even though the canonical install validator accepts `@scope/name`. Reuse the canonical plugin ID validator or keep this validator exactly aligned with it.
  - confidence: 0.95
- **[P2] Emit explicit startup activation metadata:** `src/cli/plugins-init.ts:72-80`
  - body: The generated manifest omits `activation`, but current docs say every plugin should set `activation.onStartup` intentionally and omission no longer startup-loads implicitly. New scaffolds should include current activation metadata, likely `onStartup: false` unless an option says otherwise.
  - confidence: 0.88
- **[P3] Generate the canonical plugin entry helper:** `src/cli/plugins-init.ts:36-40`
  - body: The scaffold emits a bare default register function instead of `definePluginEntry`, so new plugins start from a compatibility-shaped entrypoint rather than the current SDK helper and docs pattern. Generate the canonical helper for non-channel plugins.
  - confidence: 0.8
- **[P3] Document the plugins init command:** `src/cli/plugins-cli.ts:828-836`
  - body: This adds a new public `openclaw plugins init` subcommand, but the CLI and plugin authoring docs are unchanged. Add docs for the command, options, generated files, and intended install flow so the surface is discoverable.
  - confidence: 0.84
