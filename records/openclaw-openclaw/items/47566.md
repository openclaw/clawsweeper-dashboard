## Summary

Keep open: the feature is useful, but the current patch still conflicts with current plugin contracts by rejecting scoped ids that install supports, omitting explicit activation metadata from new manifests, and adding a public CLI command without docs or changelog coverage.

## What This Changes

The PR adds an `openclaw plugins init [directory]` command that scaffolds `openclaw.plugin.json`, `index.ts`, and `package.json` with unit tests.

## Best Possible Solution

Land a rebased plugin-init command that delegates to canonical plugin-id validation, emits a manifest matching current activation guidance, and documents/changelogs the public CLI surface.

## Reproduction Assessment

Yes. Static inspection is high-confidence: the PR validator rejects `/` while current main accepts and tests `@scope/name`, and the scaffolded manifest lacks `activation.onStartup` despite current docs requiring an intentional setting.

## Solution Assessment

No. The feature direction is reasonable, but the patch should reuse the existing install-path validator, generate activation-aware scaffolds, and update docs/changelog before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Allow scoped plugin ids during scaffolding:** `src/cli/plugins-init.ts:30-31`
  - body: Current install validation accepts scoped ids such as `@scope/name`, and current tests assert that `resolvePluginInstallDir("@scope/name")` is valid. This new validator rejects every `/`, so users cannot scaffold plugin ids that the rest of the plugin system supports; reuse `validatePluginId` from `src/plugins/install-paths.ts` or mirror its scoped-id rules.
  - confidence: 0.95
- **[P2] Emit explicit activation metadata in the scaffold:** `src/cli/plugins-init.ts:72-80`
  - body: The generated manifest omits `activation.onStartup`, while current docs say every plugin should set it intentionally and omission no longer startup-loads a plugin implicitly. Include explicit activation metadata, likely `onStartup: false` by default for the empty starter or a documented option, so new projects follow the current manifest contract.
  - confidence: 0.88
- **[P3] Document the new plugins init command:** `src/cli/plugins-cli.ts:824-836`
  - body: This registers a public `openclaw plugins init` subcommand, but the PR does not update the plugin CLI docs, the authoring guide if this becomes the canonical bootstrap path, or `CHANGELOG.md`. Add those before merge so users can discover the new command and the user-facing feature is recorded.
  - confidence: 0.86
- **[P3] Handle target directory creation failures:** `src/cli/plugins-init.ts:63`
  - body: `fs.mkdirSync(targetDir, { recursive: true })` runs before the friendly write-error handling, so a target that is an existing file or otherwise cannot be created will still surface as a raw exception. Wrap directory creation in the same CLI error path used for file writes.
  - confidence: 0.78
