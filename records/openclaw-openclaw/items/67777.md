## Summary

Keep open. Current main has partial pending-final-delivery recovery, but it still does not implement the requested durable registry-owned completion spool and still has source-visible loss windows around suspended delivery, orphan pruning, and message-tool-only completion handoff failures.

## What This Changes

The issue requests a durable subagent completion inbox/spool so successful child results survive direct announce failures, drain/restart gaps, and orphan pruning.

## Best Possible Solution

Define a registry-owned completion spool contract with self-contained payloads, stable idempotency, pending/delivered/failed state, restart retry, and explicit orphan retention or expiry behavior.

## Reproduction Assessment

Yes, source-level and shipped-version evidence are strong: current main still has orphan-prune and suspended-delivery paths matching the reported loss windows, and the discussion includes a `2026.5.20` Discord repro. I did not run a live restart/drain reproduction in this read-only review.

## Solution Assessment

Yes in direction, not as an automated patch. A durable registry-level spool is the right fix target, but maintainers should settle the persisted state, idempotency, expiry, retention, and task-status contract before implementation.
