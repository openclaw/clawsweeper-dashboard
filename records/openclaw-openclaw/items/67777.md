## Summary

Keep open. Current main has partial durable delivery state and the queued completion handoff from https://github.com/openclaw/openclaw/pull/88613, but the restore/resume orphan path still prunes missing-session runs before replaying pending completion payloads, so the requested durable spool/orphan-preservation contract is not fully implemented.

## What This Changes

The issue requests a durable subagent completion delivery inbox/spool that survives direct-announce failure, drain/restart, and restore/orphan pruning.

## Best Possible Solution

Keep this as the canonical planning thread until the registry-level delivery state preserves or retries pending completion payloads before restore/resume orphan pruning, with focused restart/orphan tests proving the new policy.

## Reproduction Assessment

Yes, source-reproducible: current main persists pending delivery payloads, but resume/restore can call orphan reconciliation before announce replay, and that helper deletes the run without checking pending completion payload. I did not run a live restart/drain scenario in this read-only review.

## Solution Assessment

No, current main is only a partial solution. The best fix is a narrow core registry delivery/orphan policy change, not more channel-specific fallback code.
