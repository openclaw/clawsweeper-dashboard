## Label Justifications

- `P1`: Successful subagent work can still fail to reach the requester in active agent/channel workflows, which breaks a real user-facing orchestration path.
- `impact:message-loss`: The issue is about completed subagent result messages being dropped, withheld, or never surfaced to the requester.
- `impact:session-state`: The remaining failure modes depend on persisted run/session state, pending-delivery markers, restart recovery, and orphan reconciliation.

## Summary

Keep open. Current main and the latest shipped release still only have partial pending-final-delivery handling: suspended successful completions are marked failed and skipped on resume, orphan reconciliation can still prune the run, and the active repair PR still has lifecycle guardrail findings.

## What This Changes

The issue requests durable subagent completion delivery so direct-announce timeout, drain/restart, or orphan pruning cannot hide a successful child result from the requester.

## Best Possible Solution

Define a registry-owned durable completion spool with self-contained payloads, idempotency, pending/delivered/failed state, restart retry, orphan retention, expiry/discard policy, and task-ledger alignment, then repair or land the narrow PR that implements that contract.

## Reproduction Assessment

Yes, source-level. Current main and v2026.5.20 still show suspended pending completions skipped on resume and orphan runs pruned without a durable retry spool, and the latest comments provide Discord and Telegram runtime examples.

## Solution Assessment

Yes in direction, but not as an automatic patch. A durable registry-level spool is the maintainable target, while retry, expiry, duplicate-prevention, task-status, and orphan-retention semantics need maintainer agreement before landing.
