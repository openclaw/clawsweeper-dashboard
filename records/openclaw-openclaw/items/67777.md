## Summary

Keep open. Current main still matches the issue discussion's partial-fix state: subagent completions have direct-first delivery plus conditional queue fallback, no persisted completion spool, terminal cleanup give-up behavior, and restore/resume orphan pruning that can remove a run before durable replay exists.

## What This Changes

Request durable subagent completion inbox/spool behavior so successful child completions survive direct-announce timeouts, drain/restart gaps, and orphan pruning before user-visible delivery.

## Best Possible Solution

Keep this as the core planning thread for a durable registry-level completion spool: persist a self-contained pending completion before cleanup can give up, have direct and queued delivery consume and update that state, keep task delivery pending while replay is possible, and make resume/orphan reconciliation retry or intentionally expire pending completions instead of pruning them silently.

## Reproduction Assessment

Yes. Current source and existing tests give a high-confidence reproduction path: completion-mode direct delivery fails, queue fallback returns none, and current dispatch/cleanup records failure rather than preserving a durable replay item; restored missing child-session runs are also pruned by reconcileOrphanedRun.

## Solution Assessment

Not applicable as a patch review because this is an issue, not a PR. The durable spool direction is the right target, but the persisted schema, idempotency key, retry exhaustion, and orphan-preservation rules need maintainer design approval before implementation.
