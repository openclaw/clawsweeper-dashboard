## Label Justifications

- `P1`: Completed subagent work can still fail to reach the requester in real agent/channel workflows, which is an urgent broken workflow.
- `impact:message-loss`: The issue is about completed subagent result messages being dropped, never surfaced, or stuck behind failed completion handoff.
- `impact:session-state`: The remaining failure path depends on persisted run state, pending delivery state, restart recovery, and orphan reconciliation.

## Summary

Keep open. Current main has meaningful partial hardening, but it still does not implement the requested durable completion spool/retry contract, and the latest discussion adds a v2026.5.20 reproduction for the message-tool-only completion handoff failure path.

## What This Changes

The issue asks for durable subagent completion delivery so completed child results survive direct-announce failure, drain/restart, and orphan pruning.

## Best Possible Solution

Define a registry-owned durable completion delivery contract with self-contained payloads, idempotent replay, explicit pending/delivered/failed states, restart retry, orphan retention or expiry, and clear external-delivery semantics.

## Reproduction Assessment

Yes, at source level. Current main still has finite announce give-up, suspended/failed pending-final-delivery states, and orphan-prune paths that match the reported loss windows; I did not run a live restart reproduction.

## Solution Assessment

Yes in direction, but not as an automated patch. A durable registry-level spool is the right target, but maintainers need to settle idempotency, expiry, duplicate prevention, task-status semantics, and whether direct external fallback is allowed for message-tool-only routes.
