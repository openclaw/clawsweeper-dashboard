## Summary

Keep open. Current main has partial pending-final-delivery and stale-announce recovery, but the requested durable registry spool and orphan-preservation contract are still not fully implemented and need maintainer/product decisions before coding.

## What This Changes

The issue requests a durable registry-backed subagent completion spool so completed child results survive direct-announce failure, drain/restart, and orphan reconciliation.

## Best Possible Solution

Define and implement a registry-owned completion spool contract with self-contained payloads, idempotency, pending/delivered/failed state, restart retry, and explicit orphan retention or expiry behavior.

## Reproduction Assessment

Yes, source-level. Current main still has an orphan-prune path without a pending-delivery guard and a partial pending-delivery model whose failure/suspension semantics match the reported loss windows, though I did not run a live restart reproduction.

## Solution Assessment

Yes in direction, not as an automated patch. A durable registry-level spool is the right fix target, but the persisted state, idempotency, expiry, retention, and task-status contract should be settled by maintainers first.
