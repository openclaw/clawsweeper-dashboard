## Summary

Keep open. Current main still has only partial pending-final-delivery state, while the requested durable completion spool is missing across the direct-failure, give-up cleanup, restart, and orphan-prune paths; recent discussion also confirms the partial Apr 26 fix did not cover the broader survivable inbox design.

## What This Changes

The issue requests a durable registry-level completion spool/inbox so subagent completion delivery survives direct announce timeout, drain/restart, and orphan pruning.

## Best Possible Solution

Keep this as the focused core planning thread for a registry-backed completion spool: persist self-contained completion payload and idempotency state before cleanup can give up, replay on resume/restart, and preserve or explicitly expire pending state during orphan reconciliation.

## Reproduction Assessment

Yes, source-level. Current main still exposes the direct-failure plus fallback-none path, terminal give-up cleanup, process-local fallback queue, and orphan-prune behavior that match the reported loss windows.

## Solution Assessment

Yes in direction, not as a ready patch. A durable registry-level spool is the maintainable fix target, but the exact persisted status, idempotency, retry, expiry, and retention contract should be settled before implementation.
