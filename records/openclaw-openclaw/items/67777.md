## Summary

Keep open. Current main has partial pending-final-delivery state, but the requested durable completion spool is still missing: completion dispatch can return an undelivered direct failure, cleanup can terminally clear pending delivery on give-up, and orphan reconciliation still deletes the run record before pending completion state can be retried.

## What This Changes

The issue requests a durable subagent completion spool/inbox so completion delivery survives direct-announce timeout, drain/restart, and orphan pruning.

## Best Possible Solution

Keep this as the focused core planning thread for a registry-backed completion spool: persist self-contained pending payload and idempotency state before cleanup can give up, replay on resume/restart, and preserve or explicitly expire pending state during orphan reconciliation.

## Reproduction Assessment

Yes, source-level. Current main still exposes the direct-failure plus fallback-none path, terminal give-up cleanup, and orphan-prune behavior that match the reported loss windows.

## Solution Assessment

Yes in direction, not as a ready patch. A durable registry-level spool is the maintainable fix target, but the exact persisted status, idempotency, retry, and expiry contract should be settled before implementation.
