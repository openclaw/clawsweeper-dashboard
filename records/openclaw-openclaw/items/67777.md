## Summary

Keep open. Current main has partial pending-final-delivery hardening and a recent completion-handoff simplification, but the durable registry spool/orphan-preservation contract requested here is still not implemented and source/tests still show remaining loss/failure windows.

## What This Changes

The issue requests a durable registry-owned subagent completion spool/inbox that survives direct-announce failure, drain/restart, and orphan reconciliation.

## Best Possible Solution

Settle one registry-owned completion spool or delivery-claim contract with self-contained payloads, idempotency, restart retry, orphan retention, and explicit expiry/failure semantics, then land one focused implementation with source tests and live restart/channel proof.

## Reproduction Assessment

Yes, source-level. Current main still has orphan-prune and retry/give-up paths that can remove or fail completion delivery before a durable spool exists, and the current tests preserve a message-tool failure case; I did not run a live restart reproduction.

## Solution Assessment

Yes in direction, not as an automated one-shot. A durable registry-level spool or delivery-claim is the right end state, but maintainers need to choose the retry, expiry, idempotency, orphan-retention, and parent-mediated versus direct-fallback policy.
