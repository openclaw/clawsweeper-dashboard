## Summary

Keep open. Current main still lacks the requested durable completion spool/inbox: completion-mode delivery remains direct-first with only conditional in-memory queue fallback, cleanup can mark delivery failed after retry/expiry give-up, and restore/resume orphan reconciliation still prunes missing child-session runs before any durable completion replay can happen.

## What This Changes

The issue requests a durable subagent completion inbox/spool so completion delivery survives direct-announce timeout, drain/restart, and orphan-prune paths.

## Best Possible Solution

Keep this as the planning thread for a core subagent lifecycle change: persist a self-contained pending completion record before cleanup can give up, have direct and queued delivery consume and update that state, keep task delivery pending while the spool is pending, and make resume/orphan reconciliation retry or deliberately expire pending completions instead of pruning them silently. The eventual fix should include focused registry/dispatch/restart tests, docs updates, and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists on current main: make completion-mode direct delivery fail while queue fallback returns none, or remove the child session before restore/resume cleanup; current tests and code show failure/prune behavior instead of durable pending replay.

## Solution Assessment

Not applicable as a patch review because this issue has no implementation attached. The durable spool direction is the right target, but the persisted data model, retry expiry, and orphan-preservation rules need maintainer confirmation before implementation.
