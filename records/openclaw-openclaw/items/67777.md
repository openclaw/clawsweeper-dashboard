## Summary

Keep open. Current main has important partial fixes for subagent completion delivery, but the broader durable completion spool/restart/orphan-prune request is still not fully implemented and still has active related work.

## What This Changes

The issue asks for a durable subagent completion inbox/spool so successful child results survive direct-announce timeout, drain/restart, and orphan pruning before becoming user-visible.

## Best Possible Solution

Keep this as the canonical planning thread until main has a registry-backed completion delivery contract that preserves self-contained payloads through restart/orphan paths, retries or expires them deliberately, and aligns task delivery status with the persisted state.

## Reproduction Assessment

Yes, source-level. Current main still has tests and code paths where restored orphaned runs are pruned without replay, and the issue discussion includes production observations plus partial-fix scope notes.

## Solution Assessment

Yes in direction, not as a ready mechanical patch. A durable registry-level completion delivery contract is the right owner boundary, but maintainers need to settle retry, expiry, duplicate-prevention, and orphan-prune semantics across the active related PRs.
