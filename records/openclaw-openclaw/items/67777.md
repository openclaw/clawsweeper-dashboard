## Label Justifications

- `P1`: Successful subagent work can fail to reach the requester in real orchestrator and Discord/message-tool workflows.
- `impact:message-loss`: The issue is about completed subagent result messages being dropped, suspended, or never surfaced to the requester.
- `impact:session-state`: The remaining failure depends on persisted run state, restart recovery, pending delivery state, and orphan reconciliation.

## Summary

Keep open: current main still has only partial pending-final-delivery behavior, while orphan reconciliation can still prune ended runs before durable completion redelivery and message-tool-only completion failures can still leave delivery suspended.

## What This Changes

The issue requests a durable subagent completion inbox/spool so successful child results survive direct-announce failure, restart/drain gaps, and orphan-prune recovery.

## Best Possible Solution

Define a registry-owned completion spool with self-contained payloads, idempotent retry, pending/delivered/failed state, restart retry, orphan retention or expiry, and task delivery status aligned to that state.

## Reproduction Assessment

Yes, source-level: current main still has direct completion failure paths, suspended pending delivery, and orphan pruning before announce cleanup; the latest issue comment adds a concrete 2026.5.20 Discord/message_tool reproduction. I did not run a live restart/drain reproduction in this read-only review.

## Solution Assessment

Yes in direction, not as an automated patch: the durable registry spool is the maintainable target, but retention, idempotency, expiry, duplicate prevention, and task-status semantics need maintainer agreement before implementation.
