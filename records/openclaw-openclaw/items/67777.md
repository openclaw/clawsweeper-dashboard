## Summary

Keep open. Current main has a durable delivery-state/pending-payload layer, but the restore/resume orphan path can still prune an ended run before replaying a pending or suspended completion payload, so the requested survivable completion inbox/orphan-preservation contract is not fully implemented.

## What This Changes

The issue requests a durable subagent completion spool/inbox that survives direct-announce failures, drain/restart, and orphan-prune recovery paths.

## Best Possible Solution

Keep the issue open until the registry restore/resume path preserves pending or suspended completion payloads through orphan handling, retries or visibly tombstones them under a maintainer-approved expiry policy, and has focused source plus live channel proof.

## Reproduction Assessment

Yes, at source level: current main can persist pending/suspended completion delivery state, then prune an ended run on missing-session orphan detection before starting delivery replay. I did not run a live current-main channel reproduction in this read-only review.

## Solution Assessment

No final solution is on main yet. The best fix is the durable registry-owned delivery/replay contract described in the issue, with explicit orphan-preservation and expiry policy rather than another narrow direct-announce fallback.
