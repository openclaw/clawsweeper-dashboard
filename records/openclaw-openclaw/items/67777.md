## Summary

Keep open. Current main still lacks the durable registry-level completion spool requested here, and the issue discussion explicitly says the landed commits only covered the immediate direct-fallback loss window while restart/orphan-prune durability remains unfinished.

## What This Changes

Request durable subagent completion delivery across direct-announce failure, drain/restart gaps, and restored/orphan cleanup by adding a persisted completion inbox/spool and replay semantics.

## Best Possible Solution

Keep this as the core planning thread for a persisted subagent completion spool: store a self-contained pending completion before cleanup can clear payloads, have direct and queued delivery update that state, keep task delivery pending while replay is possible, and make resume/orphan reconciliation retry or intentionally expire pending completions rather than pruning them silently.

## Reproduction Assessment

Yes. Current source and tests give a high-confidence reproduction path: completion-mode direct delivery can fail, fallback queue can return none, cleanup records failure, and restored orphan handling can prune the run without any durable replay item.

## Solution Assessment

Not applicable as a patch review because this is an issue, not a PR. The durable spool direction is the right target, but the persisted schema, idempotency, retry exhaustion, and orphan-preservation rules need maintainer design review before code is written.
