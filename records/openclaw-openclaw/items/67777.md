## Summary

Keep open. Current main and v2026.6.10 still have finite retry/give-up and orphan-prune paths that can remove or discard pending completion delivery before durable replay, while overlapping fix PRs remain unmerged.

## What This Changes

The issue requests a durable subagent completion inbox/spool so successful child work is not lost across direct-announce failures, drains, restarts, or orphan pruning.

## Best Possible Solution

Keep this issue open until maintainers land a registry-owned durable completion delivery path that preserves or tombstones pending/suspended payloads through resume/orphan handling, credits delivery exactly once, and updates tests, docs, and live channel proof.

## Reproduction Assessment

Yes, at source level: current main can preserve pending/suspended completion delivery state and still prune an ended missing-session run before replay. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

The durable registry-spool direction is the right owner boundary, but current main is only a partial implementation. The best remaining fix needs maintainer-approved orphan retention, expiry, and duplicate-delivery semantics rather than another narrow direct-announce fallback.
