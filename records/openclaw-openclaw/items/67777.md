## Summary

Keep open. Current main has partial pending-final-delivery retention, but the remaining durable completion-spool/redelivery contract is still unresolved, and the latest release discussion plus open follow-up PRs show this is still active product/agent reliability work rather than cleanup.

## What This Changes

The issue requests durable subagent completion delivery so completed subagent results survive direct-announce failure, drain/restart, and orphan reconciliation.

## Best Possible Solution

Settle one maintainer-approved registry-owned completion delivery contract, then land a canonical implementation that keeps the completion payload self-contained, idempotent, retryable after restart, and protected from orphan pruning until delivered, explicitly failed, or expired by policy.

## Reproduction Assessment

Yes, source-level. Current main still has suspended pending-final-delivery and orphan-prune paths that match the reported loss windows, and the issue has a fresh 2026.5.20 runtime reproduction variant, though I did not run a live restart/drain repro in this read-only pass.

## Solution Assessment

Yes in direction, but not as an automated one-off patch. A durable registry-owned completion spool or equivalent canonical delivery state is the right target, while the exact direct-fallback, expiry, and orphan-retention semantics need maintainer approval before coding.
