## Label Justifications

- `P1`: Successful subagent results can still be hidden from the requester across restart/orphan paths, which breaks a core agent workflow for affected users.
- `impact:message-loss`: The issue is specifically about completed subagent output not reaching the user-visible requester session.

## Summary

Keep open. Current main has partial durable delivery state and retry/suspension handling, but the requested restore/orphan-preservation contract is not fully satisfied: the current restore/resume orphan path can still prune an ended run before replaying a pending completion payload.

## What This Changes

The issue requests a durable subagent completion inbox/spool so successful child results survive direct-announce failures, restart/drain gaps, and orphan-prune races.

## Best Possible Solution

Keep this as the canonical planning thread until the registry-level delivery state preserves or retries pending completion payloads before restore/resume orphan pruning, with focused restart/orphan tests proving that behavior.

## Reproduction Assessment

Yes, by source inspection rather than live repro: current main's restore/resume orphan paths can call reconcileOrphanedRun before delivery replay, and that helper deletes the run without checking pending completion payload. I did not run a live restart/drain scenario.

## Solution Assessment

No, current main is only a partial solution. The maintainable fix should make pending/suspended completion delivery a durable obligation that orphan reconciliation preserves or retries before pruning.
