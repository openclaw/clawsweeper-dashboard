## Summary

Keep open. Current main still has no durable registry-level completion spool, and the remaining loss windows described in the discussion are still visible in source: direct-first completion dispatch can fail without a durable replay item, cleanup can terminally give up, and orphan reconciliation can prune restored runs.

## What This Changes

The issue requests a durable subagent completion inbox/spool that survives direct-announce failure, restart/drain gaps, and restore/orphan pruning.

## Best Possible Solution

Keep this as the core planning thread for a persisted registry-level completion spool: persist a self-contained pending completion before cleanup can give up, make direct and queued delivery consume that state, keep task delivery pending while replay is possible, and have resume/orphan reconciliation retry or intentionally expire pending completions.

## Reproduction Assessment

Yes, by source-level inspection rather than a live run: completion-mode direct delivery can fail, queue fallback can return none, tests assert that failure, cleanup can mark delivery failed, and orphan reconciliation can delete the run with no durable replay state.

## Solution Assessment

Not applicable as a patch review because this is an issue, not a PR. The durable spool direction is the right target, but the persisted schema, replay idempotency, expiry, and orphan-preservation rules need maintainer design approval before implementation.
