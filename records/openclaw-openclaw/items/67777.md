## Summary

Keep open. Current main has partial pending-final-delivery recovery, but it still does not implement the requested durable completion spool/orphan-preservation contract, and the latest issue discussion includes a 2026.5.20 Discord repro after the partial fixes.

## What This Changes

The issue requests a durable subagent completion inbox/spool so successful subagent results cannot be lost across direct-announce failures, drain/restart gaps, or orphan pruning.

## Best Possible Solution

Define a registry-owned durable completion spool with self-contained payloads, idempotency, pending/delivered/failed state, restart retry, expiry/discard policy, task-status alignment, and orphan retention before landing narrow announce fallbacks as permanent behavior.

## Reproduction Assessment

Yes, source-level. Current main still has failure paths where completion delivery is pending/suspended or pruned without a durable retry spool, and the latest discussion reports a 2026.5.20 Discord repro; I did not run a live restart reproduction.

## Solution Assessment

Yes in direction, but not as an automated patch. A durable registry-level spool is the maintainable fix target, while idempotency, expiry, retention, and task-status semantics need maintainer agreement before implementation.
