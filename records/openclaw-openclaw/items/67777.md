## Summary

Keep open. Current main has partial pending-final-delivery hardening, but it still lacks the requested durable completion spool/restart/orphan-preservation contract, and recent latest-release evidence reports the same retry-limit suspension failure mode.

## What This Changes

The issue requests a durable registry-owned subagent completion inbox/spool so completed child results survive direct-announce failures, gateway restarts, drains, and orphan pruning.

## Best Possible Solution

Define a registry-owned completion spool with self-contained payloads, stable idempotency, pending/delivered/failed state, restart retry, and explicit orphan retention or expiry behavior, then land a focused implementation that preserves the parent/requester delivery policy.

## Reproduction Assessment

Yes, source-level. Current main still has orphan-prune and retry-limit suspension paths that can leave a completed child result undelivered, and the latest-release Discord report matches those fields without requiring a speculative code path.

## Solution Assessment

Yes in direction, not as an automated patch. A durable registry-level spool is the right end state, but the persistent state shape, idempotency, expiry, orphan retention, and message-tool fallback policy need maintainer agreement first.
