## Summary

Keep open. Current main has meaningful pending-final-delivery hardening, but it still falls short of the requested durable completion spool/orphan-preservation contract, and fresh Discord/Telegram evidence on 2026.5.20 reports the same completion-loss class after the partial fixes.

## What This Changes

The issue asks for durable subagent completion delivery so completed child work cannot be lost across direct-announce failure, restart/drain, or orphan pruning.

## Best Possible Solution

Define one registry-owned durable completion spool or delivery-intent state with self-contained payloads, idempotent replay, restart retry, orphan retention, expiry/discard policy, and task-status alignment, then land focused fixes against that contract.

## Reproduction Assessment

Yes, at source level. Current main still has finite give-up, suspended pending-delivery states that do not retry on resume, and orphan pruning without a pending-delivery guard; I did not run a live transport restart repro.

## Solution Assessment

Yes in direction, but not as an automated one-shot patch. A durable registry-level spool is the maintainable target, while idempotency, expiry/discard, retry, visible-delivery, and task-status semantics need maintainer agreement.
