## Summary

Keep open. This draft PR adds a new privileged plugin-loading tier, but source review found blocking issues: the user-controlled system-dir override undermines machine-wide enforcement, the expanded origin union leaves exhaustive origin maps incomplete, and duplicate-id precedence would not actually make system plugins highest precedence.

## What This Changes

The PR adds a `system` plugin origin discovered from a machine-wide directory, bypasses normal plugin activation controls, grants implicit sensitive hook permissions, gives system hooks a high priority band, and adds focused plugin tests.

## Best Possible Solution

Define an explicit admin-controlled system-plugin trust contract, update all origin precedence/type/docs/changelog surfaces, and require security review before merge.

## Reproduction Assessment

Yes for the blocking review findings: source inspection shows the type/precedence gaps and the env-override trust bypass on current main plus the PR diff. I did not need a live run to establish those paths.

## Solution Assessment

No. A machine-wide policy tier should not be bypassable or user-promotable through ordinary process env, and the implementation must update all origin-ranking contracts before it can be considered the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not let user env promote plugins to system tier:** `src/plugins/roots.ts:30-32`
  - body: `OPENCLAW_SYSTEM_PLUGINS_DIR` is read from the normal process environment, then discovered as `origin: "system"`. Since the existing path-safety gate allows current-user-owned non-bundled directories, any user can point this at their own plugin directory to bypass plugin config and receive implicit prompt/conversation hook access; keep the enforced tier on admin-owned non-overridable roots or make the override test-only/admin-gated.
  - confidence: 0.9
- **[P1] Update every origin precedence map:** `src/plugins/plugin-origin.types.ts:1`
  - body: Adding `"system"` to `PluginOrigin` leaves existing `Readonly<Record<PluginOrigin, number>>` maps incomplete in provider auth aliases, provider auth choices, config metadata, manifest registry, provider install catalog, and channel catalog. Strict typecheck will reject this branch until each map gives `system` explicit handling.
  - confidence: 0.93
- **[P2] Give system candidates duplicate precedence:** `src/plugins/discovery.ts:967-969`
  - body: Scanning system candidates first is not enough because manifest registry duplicate resolution later calls `resolveDuplicatePrecedenceRank`; unknown origins fall through to rank 4, so a bundled duplicate can replace a system plugin. Add explicit system precedence where duplicate ids are resolved.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `CHANGELOG.md:11`
  - body: This is a user-facing plugin discovery and trust-model feature, but the PR does not add an `Unreleased` changelog entry. Add a single-line `### Changes` entry before merge.
  - confidence: 0.86
