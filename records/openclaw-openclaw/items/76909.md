## Summary

Keep this PR open, but it is not merge-ready. The latest branch appears to have addressed the earlier exhaustive-origin-map breakage, but it still introduces an un-disableable, in-process machine-wide plugin tier without an approved/enforced admin trust boundary, and the proposed hook priority floor does not actually reserve ordering.

## What This Changes

This PR adds a `system` plugin origin discovered from machine-wide roots, bypasses normal plugin activation controls, grants implicit hook permissions, floors system hook priority, updates source display/cache handling, and adds tests.

## Best Possible Solution

Define and approve the operator-managed system-plugin trust boundary first, then land a smaller implementation with platform ownership/ACL validation, origin-aware reserved hook ordering, docs, changelog, and focused regression tests.

## Reproduction Assessment

Yes. Source inspection of the PR diff plus current main shows system plugins bypass activation controls and the hook runner sorts only by raw priority, which is enough to reproduce the review blockers without running the full suite.

## Solution Assessment

No. A privileged machine-wide plugin tier is not best solved by bypassing user controls before the admin trust boundary, ACL model, ordering semantics, and operator documentation are defined.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Enforce admin-owned roots before system bypass:** `src/plugins/config-activation-shared.ts:141-150`
  - body: This branch enables `system` plugins before the global toggle, denylist, allowlist, and per-entry disable checks, then grants privileged hook access later. Because native plugins run in-process, the machine-wide root needs an approved and enforced admin ownership/ACL boundary before it can bypass user controls.
  - confidence: 0.91
- **[P2] Reserve system hooks in runner ordering:** `src/plugins/registry.ts:2075-2080`
  - body: Clamping system hooks to `>= 10000` does not guarantee they run first because the hook runner still sorts every origin by the same raw numeric priority. A user or bundled hook with a larger priority can preempt the intended system policy hook.
  - confidence: 0.88
- **[P3] Document the system plugin trust tier:** `src/plugins/roots.ts:31-37`
  - body: The PR adds implicit `/etc/openclaw/plugins` and `%ProgramData%\OpenClaw\plugins` roots that bypass normal plugin controls, but it does not update operator docs or the changelog. Operators need supported guidance for discovery order, trust, auditing, and disable/repair expectations before this ships.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a privileged, un-disableable in-process plugin trust tier without an enforced admin ownership model.

Concerns:

- **[high] Unapproved privileged plugin trust tier:** `src/plugins/config-activation-shared.ts:141`
  - body: System-origin plugins bypass normal activation controls and receive implicit prompt/conversation hook access. That can only be safe after OpenClaw defines how the operator-managed root is authenticated, owned, audited, and documented.
  - confidence: 0.92

  - body: The PR adds `%ProgramData%\OpenClaw\plugins` as a privileged root, while current discovery permission checks skip validation on Windows. A privileged un-disableable root needs platform-specific admin ACL validation.
  - confidence: 0.8
- **[medium] Policy hooks can be preempted:** `src/plugins/registry.ts:2075`
  - body: The proposed priority floor does not reserve ordering because non-system hooks are still sorted by the same raw priority field. A plugin with a higher number can run before the intended system policy hook.
