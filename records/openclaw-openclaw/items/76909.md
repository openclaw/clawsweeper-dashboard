## Summary

Keep this draft PR open, but it is not merge-ready: it introduces a privileged machine-wide plugin trust tier without the required product/security boundary and leaves concrete compile and ordering blockers.

## What This Changes

The PR adds a `system` plugin origin discovered from machine-wide directories, bypasses user plugin activation controls, grants implicit hook permissions, floors system hook priority, updates source display/cache handling, and adds system-plugin tests.

## Best Possible Solution

Define and approve the operator-managed system-plugin trust boundary first, then land a smaller implementation with exhaustive origin plumbing, platform ownership/ACL validation, reserved hook ordering, docs, changelog, and focused regression tests.

## Reproduction Assessment

Yes. Source inspection is enough: expanding `PluginOrigin` leaves current exhaustive origin maps incomplete, and the hook runner currently sorts only by raw numeric priority.

## Solution Assessment

No. A privileged machine-wide policy tier should not bypass existing plugin policy before the admin-owned trust boundary, ownership checks, and ordering semantics are designed and enforced.

  - body: Adding `system` to `PluginOrigin` leaves existing `Record<PluginOrigin, number>` maps in provider auth choices, provider install catalog, manifest registry, and channel catalog without a `system` member. Under strict TypeScript, the new union member needs to be ranked everywhere before this can compile cleanly.
  - confidence: 0.96

  - body: This branch makes system plugins ignore the global plugin toggle, denylist, allowlist, and per-entry disables while later granting privileged hook access. Because native plugins run in-process from a machine-wide root, this needs an approved and enforced admin-owned trust boundary before the bypass can ship.
  - confidence: 0.92
- **[P2] Reserve the system hook tier in ordering:** `src/plugins/registry.ts:2075-2080`
  - body: Clamping system hooks to `>= 10000` does not guarantee they run first because non-system registrations can still pass larger priorities and the runner sorts only by numeric priority. Cap non-system priorities or add an origin tier to the sort.
  - confidence: 0.89

## Security Review

Summary: The diff introduces an un-disableable, in-process machine-wide plugin trust tier without an approved or enforced admin ownership model.

Concerns:

  - body: System-origin plugins bypass normal activation controls and receive implicit prompt/conversation hook access. That can only be safe after OpenClaw defines how the operator-managed root is authenticated, owned, audited, and documented.

  - body: The PR adds `%ProgramData%\OpenClaw\plugins` as a privileged root, while current discovery permission checks skip validation on Windows. A privileged un-disableable root needs platform-specific admin ACL validation.
