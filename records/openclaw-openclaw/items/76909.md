## Summary

Keep open: this draft PR is a security-sensitive feature and the current diff is not mergeable because it introduces a PluginOrigin exhaustiveness break and a user-controlled path into an un-disableable privileged plugin tier.

## What This Changes

The PR adds a new system plugin origin, machine-wide discovery roots, activation-policy bypasses, privileged hook permissions, priority flooring, source display support, and new plugin tests.

## Best Possible Solution

Define and approve the operator-managed plugin trust model first, then implement it with non-user-controlled root resolution, exhaustive origin handling, documented security semantics, and regression tests for all policy boundaries.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows `PluginOrigin` gains `system` while exhaustive origin maps remain unchanged, which is a strict TypeScript compile break. The privileged path concern is also source-reproducible from the env override plus activation and hook-policy bypasses.

## Solution Assessment

No, this is not the best way to solve the issue as proposed. A privileged plugin tier needs a documented admin trust boundary, exhaustive origin plumbing, and a real reserved hook ordering mechanism before the feature can be safely reviewed for merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Update every exhaustive PluginOrigin map:** `src/plugins/plugin-origin.types.ts:1`
  - body: Adding `system` to `PluginOrigin` makes the existing `Record<PluginOrigin, number>` initializers exhaustive. Several current maps still only list config/workspace/global/bundled, so strict TypeScript will fail before the new tests can run.
  - confidence: 0.96
- **[P1] Do not trust an env-selected system plugin root:** `src/plugins/roots.ts:27-30`
  - body: `OPENCLAW_SYSTEM_PLUGINS_DIR` is controlled by the launching environment, but everything discovered there is later treated as un-disableable and granted prompt/conversation hook access. That lets a user-selected path bypass normal plugin policy instead of proving administrator control.
  - confidence: 0.91
- **[P2] Enforce the reserved hook tier:** `src/plugins/registry.ts:2057-2062`
  - body: Clamping system hooks to `>= 10000` does not guarantee they run before all other hooks because the SDK still allows non-system plugins to register priorities above that value. Cap non-system priorities or sort the system origin as a separate tier.
  - confidence: 0.86
