## Summary

Keep open for maintainer security/product review: this recent draft proposes a new privileged core trust tier, but the current diff still has blocking trust-boundary, Windows ACL, ordering, and duplicate-precedence issues plus no real behavior proof.

## What This Changes

The PR adds a new `system` plugin origin discovered from machine-wide directories, bypasses normal plugin activation controls, grants implicit prompt/conversation hook permissions, floors hook priority, updates origin ranks, and adds tests.

## Best Possible Solution

Define and approve the operator-managed system-plugin trust boundary first, then implement a smaller core change with enforced ownership/ACL checks, reserved origin ordering, docs, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection of the PR head and current main is enough: the branch bypasses activation controls, current Windows permission validation returns without checks, hook execution sorts by raw priority, and duplicate precedence does not reserve the new origin.

## Solution Assessment

No. The proposed path makes the privileged tier unconditional before the trust boundary is defined; the safer solution is a product/security-approved system-plugin model with platform permission enforcement and origin-tier ordering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Define admin trust before bypassing plugin controls:** `src/plugins/config-activation-shared.ts:141-151`
  - body: This always enables system-origin plugins ahead of the global toggle, denylist, allowlist, and per-entry disables, while the same branch grants implicit prompt/conversation access. Because native plugins run in-process, this needs an approved and enforced machine-wide ownership/ACL trust boundary before user controls are bypassed.
  - confidence: 0.93
- **[P1] Validate Windows system roots before trusting them:** `src/plugins/roots.ts:35-37`
  - body: The new ProgramData root is privileged, but current discovery permission validation returns without checking ownership or ACLs on Windows. Add platform-specific admin ACL validation or keep the system tier disabled on Windows until that exists.
  - confidence: 0.87
- **[P2] Reserve ordering by origin, not only numeric priority:** `src/plugins/registry.ts:2078-2083`
  - body: Clamping system hooks to >=10000 does not guarantee they run first because hook execution sorts only by raw priority and non-system hooks can register a larger number. Use an origin tier or cap non-system priorities for protected policy hooks.
  - confidence: 0.91
- **[P2] Give system origins top duplicate precedence:** `src/plugins/manifest-registry.ts:97`
  - body: The branch adds system as rank 4 in the origin rank while duplicate precedence still falls through to 4, so config/global/bundled/workspace duplicates can override the system record despite the PR's non-overridable claim. System needs explicit highest precedence in both same-root canonicalization and duplicate resolution.
  - confidence: 0.9
