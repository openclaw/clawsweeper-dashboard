## Summary

This PR should stay open: current main does not implement the requested system plugin tier, and the latest head still has blocking correctness and security issues. The force-push appears to have removed the earlier user-env root override concern, but it still leaves PluginOrigin exhaustiveness broken and introduces an unapproved privileged plugin trust tier.

## What This Changes

The PR adds a machine-wide "system" plugin origin with new discovery roots, activation bypasses, implicit hook permissions, priority flooring, source display support, and tests.

## Best Possible Solution

Define and approve the operator-managed trust boundary first, then land a smaller implementation with exhaustive origin plumbing, platform-specific admin ownership or ACL validation, enforced hook tiers, docs, changelog, and focused regression tests.

## Reproduction Assessment

Yes. Source inspection is enough for the PR blockers: adding "system" to PluginOrigin leaves current exhaustive origin maps incomplete, and the current hook runner sorts only by raw numeric priority.

## Solution Assessment

No. A privileged machine-wide plugin tier is not best solved by bypassing existing plugin policy first; the maintainable path is to define the admin trust model and enforce it across discovery, activation, hook ordering, docs, and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Update exhaustive PluginOrigin maps:** `src/plugins/plugin-origin.types.ts:1`
  - body: Adding `system` to `PluginOrigin` leaves existing `Record<PluginOrigin, number>` maps incomplete in config metadata, provider auth choices/aliases, manifest registry, provider install catalog, and channel catalog. Under strict TypeScript those exhaustive records fail before the runtime behavior can be reviewed.
  - confidence: 0.96
- **[P1] Define admin trust before bypassing plugin config:** `src/plugins/config-activation-shared.ts:141-150`
  - body: This new branch makes `system` plugins ignore the global plugin toggle, denylist, allowlist, and per-entry disables. Because that same origin is treated as privileged later, the implementation needs an approved and enforced admin-owned trust boundary before this bypass can ship.
  - confidence: 0.9
- **[P2] Reserve the hook priority tier for system hooks:** `src/plugins/registry.ts:2057-2062`
  - body: Clamping system hooks to `>= 10000` does not make them run before all user or bundled hooks, because non-system registrations can still pass priorities above that value and the runner sorts only by numeric priority. Cap non-system priorities or sort system hooks as a separate tier.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces a privileged machine-wide plugin trust tier that bypasses user controls and hook safeguards before the trust model is approved and enforced.

Concerns:

- **[high] Unapproved privileged plugin trust tier:** `src/plugins/config-activation-shared.ts:141`
  - body: System-origin plugins bypass normal activation controls and receive implicit prompt/conversation hook access. That may be valid only after OpenClaw defines how an operator-administered plugin root is authenticated, owned, audited, and documented.
  - confidence: 0.91
- **[high] Windows system root lacks ACL validation:** `src/plugins/roots.ts:36`
  - body: The PR adds a `%ProgramData%\OpenClaw\plugins` system root, while current discovery permission checks return without validation on win32. A privileged, un-disableable root needs platform-specific admin ACL validation rather than relying on the path name.
  - confidence: 0.78
- **[medium] Policy hooks can be preempted by higher user priorities:** `src/plugins/registry.ts:2057`
  - body: The proposed priority floor does not reserve ordering because non-system hooks are still sorted by the same raw priority field. A user or bundled plugin with a higher number can run before the intended system policy hook.
  - confidence: 0.86
