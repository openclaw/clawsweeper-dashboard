## Summary

Keep open: current main does not implement system-tier plugin discovery, and the PR needs maintainer security/product approval plus correctness repairs before it can merge.

## What This Changes

The PR adds a `system` plugin origin discovered from machine-wide directories, bypasses normal activation controls, grants implicit hook permissions, floors system hook priority, and adds plugin tests.

## Best Possible Solution

Define the operator-managed system-plugin trust boundary first, then land a smaller implementation with exhaustive origin plumbing, platform ownership/ACL validation, reserved hook ordering, docs, changelog, and focused tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for the PR blockers: the new `system` union member leaves existing exhaustive origin maps incomplete, and current hook ordering is numeric priority only.

## Solution Assessment

No. The current diff bypasses user controls before defining and enforcing the admin trust boundary; the safer path is explicit maintainer approval plus platform ownership checks and complete origin/hook/docs plumbing.

## Security Review

Status: needs_attention

Summary: The PR introduces a privileged, un-disableable in-process plugin tier without an approved or enforced machine-admin trust model.

Concerns:

- **[high] Unapproved privileged plugin trust tier:** `src/plugins/config-activation-shared.ts:141`
  - body: System-origin plugins bypass normal activation controls and receive implicit prompt/conversation hook access. That can only be safe after OpenClaw defines how the operator-managed root is authenticated, owned, audited, and documented.
  - confidence: 0.92
- **[high] Windows system root lacks ACL validation:** `src/plugins/roots.ts:30`
  - body: The PR adds `%ProgramData%\OpenClaw\plugins` as a privileged root, while current discovery permission checks skip validation on Windows. A privileged un-disableable root needs platform-specific admin ACL validation.
  - confidence: 0.8
- **[medium] Policy hooks can be preempted:** `src/plugins/registry.ts:2075`
  - body: The proposed priority floor does not reserve ordering because non-system hooks are still sorted by the same raw priority field. A user or bundled plugin with a higher number can run before the intended system policy hook.
  - confidence: 0.88
