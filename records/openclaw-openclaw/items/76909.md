## Summary

Keep open; this draft adds a privileged, un-disableable plugin tier and still needs an approved trust boundary plus enforceable ordering, precedence, and platform permission guarantees before merge.

## What This Changes

The PR adds a `system` plugin origin discovered from machine-wide directories, bypasses user plugin controls for it, grants implicit prompt/conversation hook permissions, floors hook priority, updates origin ordering maps, and adds tests.

## Best Possible Solution

Define and approve the operator-managed system-plugin trust boundary first, then land a smaller implementation with enforced ownership/ACL validation, reserved precedence/order, docs, changelog, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection of the PR diff against current main shows the activation bypass, Windows permission-check gap, duplicate precedence fallback, and raw priority ordering without needing a live install.

## Solution Assessment

No. Adding an always-on privileged code tier before defining and enforcing trust, ACL, and ordering semantics is not the narrowest maintainable solution; the trust model should come first and then be enforced across discovery, activation, registry, docs, and tests.

  - confidence: 0.91

  - body: The PR says system plugins cannot be overridden, but duplicate precedence still lacks a system-origin branch and falls unknown origins to the lowest rank for different plugin roots. System origin needs explicit top precedence wherever duplicate/plugin-origin ordering is resolved.
  - confidence: 0.9
