## Summary

Keep this draft PR open: it proposes a security-sensitive new system plugin trust tier, but the current diff still has compile-time origin-plumbing gaps, an unenforced hook-priority guarantee, and an unapproved policy bypass. The existing ClawSweeper comment already called out the same main blockers, and current source/docs support that review.

## What This Changes

The PR adds a new `system` plugin origin discovered from machine-wide directories, bypasses user plugin activation controls, grants implicit hook permissions, floors system hook priority, updates source display/cache key handling, and adds system-plugin tests.

## Best Possible Solution

Define and approve the operator-managed system-plugin trust boundary first, then land a smaller implementation with exhaustive origin plumbing, platform-specific ownership/ACL validation, enforced hook tiers, docs, changelog, and focused regression tests.

## Reproduction Assessment

Yes. Source inspection of the PR diff and current main is enough: expanding `PluginOrigin` leaves existing exhaustive origin maps incomplete, and the hook runner currently sorts only by raw numeric priority.

## Solution Assessment

No. A privileged machine-wide plugin tier is not best solved by bypassing existing plugin policy first; the safer path is to design and enforce the admin trust boundary across discovery, activation, hook ordering, docs, and tests.

  - body: Adding `system` to `PluginOrigin` makes existing `Record<PluginOrigin, number>` maps incomplete in config metadata, provider auth choices, manifest registry, provider install catalog, and channel catalog.   - confidence: 0.96

  - body: This branch makes system plugins ignore the global plugin toggle, denylist, allowlist, and per-entry disables while later granting privileged hook access. Because native plugins are in-process code, this needs an approved and enforced admin-owned trust boundary before the bypass can ship.
  - confidence: 0.91
- **[P2] Reserve the system hook tier in ordering:** `src/plugins/registry.ts:2075-2080`
  - body: Clamping system hooks to `>= 10000` does not guarantee they run first, because non-system registrations can still pass larger priorities and the runner sorts only by the numeric priority. Cap non-system priorities or add an origin tier to the sort.
  - confidence: 0.88

## Security Review

Concerns:

- **[high] Unapproved privileged plugin trust tier:** `src/plugins/config-activation-shared.ts:141`
  - body: System-origin plugins bypass normal activation controls and receive implicit prompt/conversation hook access. That can only be safe after OpenClaw defines how the operator-managed root is authenticated, owned, audited, and documented.
