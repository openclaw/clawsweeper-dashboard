## Summary

This draft PR should stay open: it proposes a privileged system plugin trust tier, but the current diff has blocking type exhaustiveness, ordering, and security-boundary problems that need maintainer product and security review before implementation repair.

## What This Changes

The PR adds a `system` plugin origin with machine-wide discovery roots, activation bypasses, implicit hook permissions, hook priority flooring, source display support, and new plugin tests.

## Best Possible Solution

Define and approve the operator-managed plugin trust model first, then implement it with non-user-controlled root resolution, exhaustive origin plumbing, explicit reserved hook ordering, docs/changelog, and regression tests for each policy boundary.

## Reproduction Assessment

Yes. Source inspection of the PR diff and current main shows the `PluginOrigin` union expansion conflicts with existing exhaustive maps, and the privileged path concern follows from the env override plus activation and hook-policy bypasses.

## Solution Assessment

No. A privileged machine-wide plugin tier is not best solved by a user-environment override plus config bypasses; it needs an approved administrator trust boundary and exhaustive contract updates before code-level repair.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Update every exhaustive PluginOrigin map:** `src/plugins/plugin-origin.types.ts:1`
  - body: Adding `system` to `PluginOrigin` makes the existing `Record<PluginOrigin, number>` initializers exhaustive. Several current maps still only list config/workspace/global/bundled, so strict TypeScript will fail before the new tests can run.
  - confidence: 0.96
- **[P1] Do not trust an env-selected system plugin root:** `src/plugins/roots.ts:27-30`
  - body: `OPENCLAW_SYSTEM_PLUGINS_DIR` is controlled by the launching environment, but everything discovered there is later treated as un-disableable and granted prompt/conversation hook access. That lets a user-selected path bypass normal plugin policy instead of proving administrator control.
  - confidence: 0.91
- **[P2] Enforce the reserved hook tier:** `src/plugins/registry.ts:2057-2062`
  - body: Clamping system hooks to `>= 10000` does not guarantee they run before all other hooks because the SDK still allows non-system plugins to register priorities above that value. Cap non-system priorities or sort the system origin as a separate tier.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a privileged plugin trust tier with user-environment root override, config bypasses, and implicit raw conversation/prompt hook access.

Concerns:

- **[high] User env can select privileged system plugins:** `src/plugins/roots.ts:27`
  - body: The proposed `OPENCLAW_SYSTEM_PLUGINS_DIR` override feeds the new system discovery tier, while system plugins bypass deny/allow/global disable and get implicit prompt/conversation access. Without an administrator-only root and ownership model, the trust boundary is user-controlled.
