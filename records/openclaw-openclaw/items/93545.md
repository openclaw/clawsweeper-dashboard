## Summary

Keep open: the fallback idea is useful, but current head is not merge-ready because the classifier treats any rich-message 400 as unsupported, the send fallback can replay already-delivered rich chunks, sibling rich delivery paths remain raw, and the supplied proof is only a standalone simulation.

The branch adds a Telegram rich-message unsupported-error classifier and retries durable rich sends/edits through legacy HTML send/edit paths when the classifier matches.

## Best Possible Solution

Land one shared, narrow unsupported-rich fallback across durable send/edit, bot delivery, and draft streaming that avoids partial replay, keeps the standard HTML default, adds regression tests, and includes redacted live Telegram/Bot API proof.
