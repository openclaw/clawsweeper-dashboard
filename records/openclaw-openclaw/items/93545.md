## Summary

Keep open: the fallback direction is relevant, but the current PR is not merge-ready because it still classifies every rich-message 400 as unsupported, only wraps one rich delivery surface, and supplies synthetic rather than real Telegram or version-pinned Bot API proof.

## What This Changes

The PR modifies Telegram send/edit handling in `extensions/telegram/src/send.ts` to retry legacy HTML send/edit after classified rich send/edit failures.

## Best Possible Solution

Use one shared, narrow rich-unsupported classifier across `send.ts`, `bot/delivery.send.ts`, and `draft-stream.ts`, add focused regression tests, and require redacted real Telegram or version-pinned Bot API proof before merge.

## Reproduction Assessment

Yes at source level: current main sends raw rich messages when `richMessages` is enabled, and unsupported rich API errors would propagate on several paths. I did not run a live Telegram or version-pinned Bot API reproduction in this read-only review.

## Solution Assessment

No; this PR is too broad in error matching and too narrow across delivery surfaces. The safer shape is a shared classifier with cross-surface tests and real Telegram proof.
