## Summary

Keep open: this is a focused Telegram fallback PR, but it is not merge-ready because the classifier still treats generic rich-message 400s as unsupported, the fallback is not applied across the sibling rich delivery and draft-stream surfaces, and the supplied proof is a synthetic script rather than real OpenClaw Telegram behavior.

## What This Changes

The PR changes `extensions/telegram/src/send.ts` so selected `sendRichMessage` and rich edit failures retry through legacy HTML `sendMessage` or `editMessageText`.

## Best Possible Solution

Land one shared, narrow rich-unsupported fallback across `send.ts`, `bot/delivery.send.ts`, and `draft-stream.ts`, with focused regression tests and redacted live Telegram or version-pinned Bot API proof.

## Reproduction Assessment

Yes at source level: current main sends raw rich messages when `richMessages` is enabled, and unsupported rich API errors would propagate through several paths. No live Telegram or version-pinned Bot API reproduction was run in this read-only review.

## Solution Assessment

No; this is only a plausible partial mitigation as submitted. The safer fix is a shared narrow classifier and fallback wrapper that preserves delivery semantics across every rich send/edit surface, plus real Telegram proof.
