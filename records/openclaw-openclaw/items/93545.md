## Best Possible Solution

Move the fallback into a shared, classified rich-transport path that only retries before side effects, covers durable sends, bot reply delivery, and rich preview/edit paths, and includes focused tests plus real Bot API proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Prevent legacy replay after partial rich sends:** `extensions/telegram/src/send.ts:815-821`
  - body: The catch wraps the whole rich chunk helper. That helper sends and records each rich chunk as it goes, so if chunk 2 fails after chunk 1 was accepted, this fallback sends the entire raw text again through legacy `sendMessage`, duplicating already-visible content. Restrict fallback to a classified unsupported-rich-method failure before any chunk lands, or have the rich helper prove no side effects before retrying.
  - confidence: 0.91
- **[P2] Cover the bot reply rich send path:** `extensions/telegram/src/send.ts:813-814`
  - body: This fallback only wraps `sendMessageTelegram`, but normal agent replies flow through `delivery.replies.ts` into `sendTelegramText`, which still calls raw `sendRichMessage` directly when `richMessages` is true. Users on an older/self-hosted Bot API would still see normal bot replies fail, so the linked fallback behavior remains incomplete.
  - confidence: 0.86
