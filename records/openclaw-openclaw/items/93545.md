## Summary

Keep open: current main still lacks the requested unsupported-rich fallback, and this PR is a plausible candidate, but it is not merge-ready because the classifier is still too broad, the fallback only covers part of the Telegram rich-delivery surface, and the supplied proof is synthetic.

## What This Changes

The PR wraps Telegram rich send/edit paths in `extensions/telegram/src/send.ts` with a fallback to legacy HTML send/edit when a new unsupported-rich classifier matches the error.

## Best Possible Solution

Use one shared, narrow unsupported-rich classifier across `send.ts`, `bot/delivery.send.ts`, and `draft-stream.ts`, add focused regression coverage, and require redacted real Telegram or version-pinned Bot API proof before merge.

## Reproduction Assessment

Yes at source level: current main calls raw rich Telegram send/edit methods when `richMessages` is enabled, so a Bot API root without `sendRichMessage` would propagate an error on several paths. I did not run a live Telegram or version-pinned Bot API reproduction in this read-only review.

## Solution Assessment

No; this PR is too broad in error matching and too narrow across delivery surfaces. The better fix is a shared classifier/fallback with cross-surface tests and real Telegram proof.
