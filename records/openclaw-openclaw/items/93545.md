## Summary

Keep open: this is a useful Telegram fallback direction, but the patch is not merge-ready because the unsupported classifier still catches unrelated 400 errors, the fallback is only applied to part of the rich-message delivery surface, and the supplied proof is a standalone simulation rather than real Telegram behavior.

## What This Changes

The branch modifies Telegram `send.ts` to catch classified rich send/edit failures and retry legacy HTML send/edit.

## Best Possible Solution

Use one shared, narrow unsupported-rich classifier across every Telegram rich send/edit path, preserve routing and presentation context, add focused regression tests, and require redacted real Telegram or version-pinned Bot API proof before merge.

## Reproduction Assessment

Yes at source level: current main sends raw rich messages on several `richMessages: true` paths, and the PR's broad matcher and partial coverage are visible in source. I did not establish a live Telegram or version-pinned Bot API reproduction in this read-only review.

## Solution Assessment

No; runtime fallback is a plausible mitigation for opt-in rich messages, but this implementation is too broad in error matching and too narrow across delivery surfaces. The safer solution is a shared narrow classifier with cross-surface tests and real Telegram proof.
