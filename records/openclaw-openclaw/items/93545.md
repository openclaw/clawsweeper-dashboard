## Summary

Keep this PR open, but it is not merge-ready: the proposed classifier still treats any rich-message `400 Bad Request` as unsupported, the fallback is only wired into one Telegram send/edit surface while other rich-enabled delivery paths remain raw, and the supplied proof is a synthetic Node script rather than real Telegram or version-pinned Bot API behavior.

## What This Changes

The PR changes `extensions/telegram/src/send.ts` to retry legacy HTML text sends/edits after classified `sendRichMessage` or rich edit failures.

## Best Possible Solution

Use one shared, narrow rich-unsupported classifier across `send.ts`, `bot/delivery.send.ts`, and `draft-stream.ts`, add focused regression tests for unsupported fallback and unrelated 400 rethrow, and require redacted real Telegram or version-pinned Bot API proof before merge.

## Reproduction Assessment

Yes at source level: current main sends raw rich messages when `richMessages` is enabled, and unsupported rich API errors would propagate on several paths. I did not run a live Telegram or version-pinned Bot API reproduction in this read-only review.

## Solution Assessment

No; this PR is too broad in error matching and too narrow across delivery surfaces. The safer shape is a shared classifier with cross-surface tests and real Telegram proof.
