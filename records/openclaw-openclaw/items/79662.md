## Summary

Current main now covers the reported bypass: OpenAI/Codex agent turns select the Codex app-server harness, skip PI's generic `openai-codex-responses` auth bootstrap, and pass the selected `openai-codex` auth profile plus stored account id into the app-server refresh path. This appears fixed on main after v2026.5.7 and is not in a shipped tag yet.

## What This Changes

The issue asks for Codex/OpenAI Codex harness calls after OAuth refresh to use OpenClaw's stored accountId instead of failing inside pi-ai's JWT accountId extraction.

## Best Possible Solution

Keep the current main harness/profile bridge behavior and ship it in the next release; any remaining legacy `openai-codex/*` PI usage should continue through doctor migration instead of reintroducing PI transport for Codex harness turns.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No failing current-main reproduction: source and regression tests show the reported `codex/*` and forced `openai/*` Codex harness paths now avoid the pi-ai `extractAccountId` call and forward the stored profile/account id instead.

## Solution Assessment

Is this the best way to solve the issue? Yes. The current-main approach is narrower than changing pi-ai's public API: keep Codex harness transport in the Codex plugin, forward `openai-codex` profile identity, and let app-server auth receive the stored account id.
