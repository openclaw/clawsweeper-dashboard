## Summary

Keep open: the PR is draft and has the protected `maintainer` label, and source review found two P2 correctness blockers in the proposed doctor cleanup path.

## What This Changes

This PR adds a plugin-backed doctor session-state repair seam and Codex-owned cleanup for stale auto-created Codex session model, runtime, auth, and CLI state, with tests, docs, and a changelog entry.

## Best Possible Solution

Keep the doctor cleanup, but expose it through a control-plane repair contract that can run for disabled bundled owners and split Codex provider/auth cleanup from runtime, harness, and CLI cleanup.

## Reproduction Assessment

Yes. Source inspection gives high confidence: the linked disabled-Codex config cannot register the plugin-owned provider, and a PI `openai-codex/*` route with stale `agentHarnessId: "codex"` or `codex-cli` state is skipped by the single `configuredAllowsCodexState` guard.

## Solution Assessment

No. The repair direction is appropriate, but the current implementation is not the narrowest maintainable fix because it misses explicitly disabled Codex plugin configs and does not preserve the documented provider/runtime split.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Load the cleanup provider for disabled Codex configs:** `src/commands/doctor-session-state-providers.ts:91`
  - body: The linked stale-route report uses `plugins.entries.codex.enabled=false`, but this runner only gets providers from normal migration-provider registration. Current plugin loading stops before `register` for an explicitly disabled plugin, so the Codex `doctorSessionState` provider is never registered and `doctor --fix` will not scan or repair the stale sessions this change is meant to handle.
  - confidence: 0.84
- **[P2] Keep provider and runtime cleanup independent:** `extensions/codex/src/doctor-session-state.ts:142`
  - body: `configuredAllowsCodexState` becomes true for any configured `openai-codex/*` model ref, then suppresses cleanup of `agentHarnessId: "codex"`, `codex-cli` bindings, and runtime model state. A PI Codex OAuth provider route is valid without native Codex runtime, so this leaves stale runtime pins active even though current docs define those surfaces as independent.
  - confidence: 0.87
