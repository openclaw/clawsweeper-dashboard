## Summary

Keep open. This is a draft PR with the protected `maintainer` label, and source review found a correctness issue where the new scan treats any configured Codex provider route as permission to preserve stale Codex runtime and CLI state.

## What This Changes

This PR adds a doctor session-store scan and `doctor --fix` repair for stale auto-created Codex session route state, with docs, tests, and a changelog entry for related stale-routing reports.

## Best Possible Solution

Keep the doctor cleanup, but classify stale provider/auth state separately from stale runtime and CLI state so a valid `openai-codex/*` PI route does not preserve a stale native Codex runtime pin.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed a session with `agentHarnessId: "codex"` or a `codex-cli` binding while the config uses `openai-codex/*` without `agentRuntime.id: "codex"`, and the new scan returns no repair because `usesCodexProvider` suppresses runtime cleanup.

## Solution Assessment

No. The repair direction is appropriate, but the implementation is not the narrowest maintainable fix until it respects the documented split between Codex provider/auth routes and Codex runtime/CLI routes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep provider and runtime cleanup independent:** `src/commands/doctor-session-codex-state.ts:143`
  - body: The scan currently treats any configured Codex provider route as permission to keep every Codex session field. With an `openai-codex/*` PI route, a stale `agentHarnessId: "codex"` or `codex-cli` binding is still outside the configured runtime path, but this branch skips it and `doctor --fix` reports nothing. Split the checks so provider/auth state is compared to configured model providers while runtime and CLI pins are compared to the configured runtime.
  - confidence: 0.86
