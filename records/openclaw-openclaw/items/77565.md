## Summary

Keep open: latest head still misses the two core cleanup paths needed for the linked stale Codex routing bug, so this draft is not ready to merge.

## What This Changes

This PR adds a doctor session-state repair provider contract, registers Codex-owned stale session cleanup through the Codex plugin, and updates tests, docs, and changelog text.

## Best Possible Solution

Keep the plugin-owned doctor cleanup direction, but load the doctor provider for disabled bundled owners through a bounded repair-only path and split Codex provider/auth cleanup from native runtime, harness, and CLI cleanup.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the #68615 config with `plugins.entries.codex.enabled=false` cannot register this plugin-owned provider, and an `openai-codex/*` PI route causes stale runtime/harness/CLI fields to be skipped.

## Solution Assessment

No. The repair direction is appropriate, but the current patch is not the narrowest maintainable fix until disabled owner discovery works and provider routes are separated from native runtime state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Load cleanup providers for disabled bundled owners:** `src/plugins/doctor-session-state-provider-runtime.ts:20-23`
  - body: The #68615 configuration explicitly has `plugins.entries.codex.enabled=false`, but this scoped load still passes through `withBundledPluginEnablementCompat`, which preserves existing disabled entries. The loader then returns no registration plan before `register(api)`, so `doctor --fix` will never load the Codex cleanup provider for the disabled-plugin case this PR is meant to repair.
  - confidence: 0.88
- **[P2] Separate provider routes from runtime cleanup:** `extensions/codex/src/doctor-session-state.ts:71`
  - body: `routeAllowsCodexState` treats configured `openai-codex/*` refs as allowing all Codex session state. That leaves `agentHarnessId: "codex"`, Codex CLI bindings, and native runtime model state uncleared even though the docs define `openai-codex/*` as the PI OAuth provider route, independent from the native Codex runtime.
  - confidence: 0.9
