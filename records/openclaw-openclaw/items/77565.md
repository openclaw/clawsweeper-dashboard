## Summary

Keep open: the PR is useful but not merge-ready because source inspection confirms the same two blockers already raised in discussion: disabled Codex configs from #68615 cannot register the cleanup provider, and the cleanup guard conflates `openai-codex/*` provider state with native Codex runtime state. The PR is also draft with a protected `maintainer` label, so it needs maintainer-handled branch updates before merge review continues.

## What This Changes

The PR adds a doctor session-state repair provider contract, registers Codex-owned stale session cleanup through the Codex plugin, updates tests/docs, and adds a changelog entry.

## Best Possible Solution

Keep the plugin-owned doctor cleanup direction, but make cleanup discovery work for explicitly disabled bundled owners and split Codex provider/auth cleanup from native runtime, harness, and CLI cleanup.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: the linked #68615 config with `plugins.entries.codex.enabled=false` cannot register the plugin-owned provider, and an `openai-codex/*` PI route sets the single guard true while stale runtime/harness/CLI fields remain uncleared.

## Solution Assessment

No. The repair direction is appropriate, but the current diff is not the narrowest maintainable fix until disabled owner discovery and provider/runtime cleanup are separated.
