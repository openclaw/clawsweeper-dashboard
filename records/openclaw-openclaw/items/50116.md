## Summary

Keep open: the PR targets a valid Windows-host/POSIX-container session path bug, but the current branch can misroute another agent's transcript by collapsing Windows absolute paths to a basename and it still lacks required real behavior proof.

## What This Changes

The PR adds POSIX detection for Windows drive-letter `sessionFile` values, maps them into the local sessions directory by basename, adds one regression test, and updates the changelog.

## Best Possible Solution

Parse Windows absolute paths with Windows semantics, preserve valid `agents/<agentId>/sessions/<file>` structure through the existing resolver fallback rules, reject unusable basenames, and prove the Windows-host/POSIX-container upgrade path.

## Reproduction Assessment

Yes, at source level: current main uses POSIX `path.isAbsolute`, and a POSIX Node probe shows a Windows drive-letter `sessionFile` is appended literally under the sessions directory. I did not run the full Windows-host/POSIX-container flow.

## Solution Assessment

No. The PR fixes the simplest same-agent symptom but bypasses existing realpath, containment, sibling-agent, and structural fallback logic, so the safer fix should translate Windows paths into that resolver flow instead of collapsing to basename.
