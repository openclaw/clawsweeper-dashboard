## Summary

Keep open: the PR targets a real source-reproducible Windows/POSIX session metadata bug, but the current basename rewrite is not mergeable because it bypasses existing agent/root-preserving path resolution and can persist a wrong transcript alias; real behavior proof is also missing.

## What This Changes

The PR adds a POSIX-only Windows drive-letter path detector, maps persisted Windows absolute `sessionFile` values to the local sessions directory by basename, adds one regression test, and updates the changelog.

## Best Possible Solution

Parse Windows absolute session paths with Windows semantics, preserve canonical agent/root structure through the existing containment and fallback logic, and add same-agent plus cross-agent Windows-path regression coverage.

## Reproduction Assessment

Yes, by source inspection: on POSIX, `path.isAbsolute` does not recognize Windows drive-letter values, so current main treats those stored `sessionFile` values as relative under `sessionsDir`. I did not run the full Windows-host/POSIX-container flow in a real setup.

## Solution Assessment

No. The basename-only rewrite fixes the simplest malformed path but is not the best maintainable fix because it bypasses existing containment, symlink, and agent-preservation logic.

## Review Findings

Overall correctness: patch is incorrect
