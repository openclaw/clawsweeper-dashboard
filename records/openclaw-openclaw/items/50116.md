## Summary

Keep this PR open, but it is not merge-ready: it targets a valid Windows/POSIX session path bug, while the current patch can alias another agent's Windows-authored transcript into the active agent's sessions directory and still lacks after-fix real environment proof.

## What This Changes

The PR adds POSIX detection for Windows drive-letter sessionFile paths, maps them to the local sessions directory by basename, adds one regression test, and updates the changelog.

## Best Possible Solution

Parse Windows absolute paths with Windows semantics, preserve valid agents/<agentId>/sessions/<file> structure through the existing realpath and fallback rules, reject unusable basenames, and prove the Windows-host/POSIX-container upgrade path.

## Reproduction Assessment

Yes, at source level: current main uses POSIX path.isAbsolute and a POSIX Node probe shows a Windows drive-letter sessionFile is appended literally under the sessions directory. I did not run the full Windows-host/POSIX-container flow.

## Solution Assessment

No. The PR fixes the simplest same-agent symptom but bypasses existing realpath, containment, sibling-agent, and structural fallback logic, so the safer fix should translate Windows paths into that resolver flow instead of collapsing to basename.
