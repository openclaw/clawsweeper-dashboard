## Summary

Keep this PR open, but do not merge it as-is. The underlying Windows-authored session path bug is still present on current main, while the PR's basename-only early return can bypass existing agent-aware and realpath containment logic and persist a wrong transcript alias.

## What This Changes

The PR adds a POSIX-only Windows drive-letter detector and basename extractor in `resolvePathWithinSessionsDir` so Windows-authored `sessionFile` values resolve under the current sessions directory.

## Best Possible Solution

Revise this PR or replace it with a narrow fix that recognizes Windows absolute paths on POSIX, normalizes them through shared/path.win32-aware logic into the existing resolver flow, preserves agent/root fallback behavior, rejects degenerate path shapes, and adds focused regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. A focused POSIX Node check reproduces the path-classification problem, and current main uses that same `path.isAbsolute` branch before resolving the session path.

## Solution Assessment

No. The submitted basename extraction fixes the simplest local filename case, but it is not the best maintainable solution because it discards agent/root context and bypasses existing containment and fallback protections.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve agent context for Windows session paths:** `src/config/sessions/paths.ts:211`
  - body: This early return collapses every Windows absolute `sessionFile` to `<sessionsDir>/<basename>` before the existing cross-agent fallback can inspect `agents/<agentId>/sessions`. If a Windows-authored path points at another agent and the basename exists locally, `resolveAndPersistSessionFile()` can persist the current agent's transcript path and later writes can attach to the wrong session. Normalize the Windows path into the existing containment/fallback flow instead of returning the basename directly.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: No dependency, workflow, lockfile, script, or secret-handling change is present, but the patch introduces a concrete session data-isolation concern.

Concerns:

- **[medium] Session path rewrite can alias another agent transcript:** `src/config/sessions/paths.ts:211`
  - body: The basename-only return can turn a Windows-authored cross-agent transcript path into the current agent's local transcript basename, and that resolved path is persisted in the session store. That is a data-integrity and agent-isolation risk.
