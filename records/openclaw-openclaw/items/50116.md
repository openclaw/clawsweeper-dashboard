- `merge-risk: 🚨 security-boundary`: Misrouting another agent's transcript into the current agent's sessions directory can expose or mix private session history.

## Summary

Keep open: current main still has the POSIX handling bug, but this PR is not ready because the basename-only rewrite can alias cross-agent transcripts and lacks real Windows-host/POSIX-container behavior proof.

## What This Changes

The PR adds POSIX-only detection for Windows drive-letter sessionFile values, maps them to the local sessions directory by basename, adds a regression test, and updates the changelog.

## Best Possible Solution

Parse Windows absolute paths with Windows semantics, preserve valid agents/<agentId>/sessions/<file> structure through the existing containment and fallback rules, and prove the Windows-host/POSIX-container upgrade path.

## Reproduction Assessment

Yes, at source level: current main uses POSIX path.isAbsolute and Node on Linux reports false for a Windows drive-letter sessionFile, so the resolver can append that value under the current sessions directory. I did not run the full Windows-host/POSIX-container flow.

## Solution Assessment

No. The branch fixes the simplest same-agent symptom but bypasses existing realpath, containment, and agent-preservation logic, so the safer fix should translate Windows paths into the existing resolver flow rather than collapse to basename.
