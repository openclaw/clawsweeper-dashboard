## Summary

Keep open: the PR targets a real current-main session path bug, but the proposed basename rewrite can alias Windows-authored cross-agent transcripts into the active agent's sessions directory and still lacks real Windows-host/POSIX-container behavior proof.

## What This Changes

The PR adds POSIX detection for Windows drive-letter sessionFile values, maps them into the local sessions directory by basename, and adds a focused test plus changelog entry.

## Best Possible Solution

Parse Windows absolute paths with Windows semantics, preserve valid agents/<agentId>/sessions/<file> structure through the existing containment and fallback rules, and add same-agent plus cross-agent regression coverage with real behavior proof.

## Reproduction Assessment

Yes, at source level: current main uses POSIX path.isAbsolute, and a POSIX Node probe confirms Windows drive-letter sessionFile values are treated as relative before resolution. I did not run the full Windows-host/POSIX-container flow.

## Solution Assessment

No. The branch fixes the simplest malformed-path symptom but bypasses existing realpath, containment, and agent-preservation logic, so the safer fix should preserve path structure rather than collapsing to basename.

## Review Findings

  - body: This return collapses any Windows absolute sessionFile to <sessionsDir>/<basename>. A path from another agent or state root can be rewritten into the current agent's transcript path and then persisted, bypassing the existing cross-agent fallback behavior.
  - confidence: 0.9

  - confidence: 0.78
