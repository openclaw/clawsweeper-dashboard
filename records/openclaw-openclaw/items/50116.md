## Summary

Keep open: the Windows-authored session path bug is source-reproducible on current main, but this PR is not merge-ready because the basename-only rewrite can break existing cross-agent transcript isolation and the external PR still lacks real Windows-host/POSIX-container behavior proof.

## What This Changes

The branch detects Windows drive-letter sessionFile paths on POSIX, resolves them into the local sessions directory by basename, adds one focused regression test, and adds a changelog entry.

## Best Possible Solution

Parse Windows absolute paths with Windows semantics, preserve valid agents/<agentId>/sessions/<file> structure through the existing containment and fallback rules, and add same-agent, cross-agent, forward-slash, trailing-separator, and real behavior proof coverage.

## Reproduction Assessment

Yes, at source level: current main uses POSIX path.isAbsolute, and a Node probe shows C:\... sessionFile values are treated as relative and resolved under the current sessions directory. I did not establish the full Windows-host/POSIX-container flow with live files.

## Solution Assessment

No. The PR fixes the simplest malformed-path symptom but bypasses current realpath, containment, and agent-preservation behavior, so the safer fix should parse Windows paths while preserving the existing session ownership structure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve agent context when rewriting Windows paths:** `src/config/sessions/paths.ts:201`
  - body: This return collapses every Windows absolute sessionFile to <sessionsDir>/<basename>. A stored transcript path for another agent or state root can be rewritten into the current agent's sessions directory and then persisted, bypassing the existing cross-agent fallback behavior.
  - confidence: 0.9
- **[P3] Reject unusable Windows basenames before falling through:** `src/config/sessions/paths.ts:198-203`
  - body: If the Windows path ends with a slash or backslash, extractBasename returns an empty string and this guard falls through to the POSIX path logic, recreating the literal drive-letter concatenation. Use Windows path semantics or throw when no usable transcript filename exists.
  - confidence: 0.78
