## Summary

Keep open. The reported bug is plausible and current main has not implemented the Windows-drive session path handling, but this PR still collapses Windows absolute paths to a basename before the existing agent-preserving containment logic and lacks required real behavior proof.

## What This Changes

The PR adds POSIX-only detection for Windows drive-letter sessionFile paths, rewrites them to the local sessions directory by basename, adds one regression test, and updates the changelog.

## Best Possible Solution

Parse Windows absolute session paths with Windows semantics, preserve valid agents/<agentId>/sessions/<file> structure through the existing containment and fallback rules, and cover same-agent plus cross-agent Windows-path cases.

## Reproduction Assessment

Yes, at source level: on Linux, Node treats a Windows drive-letter sessionFile as non-absolute and appends it under the current sessionsDir. The focused probe also shows this PR's basename rewrite can route a bot2 Windows transcript path into bot1's sessions directory; I did not run the full Windows-host/POSIX-container flow.

## Solution Assessment

No. The branch fixes the simplest malformed-path symptom but bypasses existing realpath, containment, and agent-preservation logic, so the safer fix should preserve path structure instead of collapsing to basename.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve agent context when rewriting Windows paths:** `src/config/sessions/paths.ts:201`
  - body: This early return collapses every Windows absolute sessionFile to `<sessionsDir>/<basename>`. A Windows-authored path for another agent or state root can then resolve to the current agent's transcript name and be persisted, so the fix should parse with Windows path semantics and preserve valid `agents/<agentId>/sessions/<file>` structure instead.
  - confidence: 0.9
