## Summary

Keep open: the bug class is valid on current main, but this PR still has a blocking session-isolation regression and lacks required after-fix real behavior proof.

## What This Changes

The PR adds POSIX-only detection for Windows drive-letter sessionFile paths, resolves those paths by local basename, adds a focused session test, and updates the changelog.

## Best Possible Solution

Recognize Windows absolute sessionFile paths on POSIX while preserving the existing realpath and agent fallback semantics, reject degenerate Windows path shapes, and include focused regression coverage plus redacted real import proof.

## Reproduction Assessment

Yes, at source level. On POSIX, Node treats Windows drive-letter paths as non-absolute, and current main feeds persisted sessionFile values through that branch before resolving non-absolute values under the local sessions directory.

## Solution Assessment

No. The PR fixes the simplest same-agent filename case, but the early basename return bypasses existing realpath and cross-agent fallback behavior; the safer fix is to normalize Windows absolute paths into the existing resolver flow and throw on unusable path shapes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve agent context for Windows session paths:** `src/config/sessions/paths.ts:201`
  - body: This return collapses every Windows absolute sessionFile to the current sessionsDir plus basename before the existing sibling-agent and extracted-agent fallbacks can run. A Windows-authored path for another agent with the same basename can be persisted as the current agent's transcript path, so the Windows-path handling needs to preserve agent/root context instead of returning here.
  - confidence: 0.89
- **[P2] Reject degenerate Windows session paths:** `src/config/sessions/paths.ts:198-203`
  - body: When the Windows-style path ends with a slash or backslash, extractBasename returns an empty string and this guard falls through to the POSIX relative-path branch. That reproduces the literal drive-letter concatenation this PR is trying to prevent, so the Windows branch should reject or safely normalize paths without a usable filename.
  - confidence: 0.82
