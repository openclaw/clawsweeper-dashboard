## Summary

Keep this PR open, but do not merge as-is: the bug is plausible and source-reproducible, yet the patch collapses Windows-authored session paths to a basename before the existing agent-aware path logic runs, and the PR still lacks real behavior proof.

## What This Changes

The PR detects Windows drive-letter sessionFile paths on POSIX, maps them to the local sessions directory by basename, adds a focused regression test, and adds a changelog entry.

## Best Possible Solution

Normalize Windows-authored session paths inside `resolvePathWithinSessionsDir` while preserving the existing realpath, agent, and structural fallback semantics, then cover same-agent, cross-agent, forward-slash, degenerate/trailing-separator, and symlink cases.

## Reproduction Assessment

Yes for source-level reproduction: on POSIX, Node treats `C:\...` as non-absolute and current main resolves it as a literal drive-letter filename under the sessions directory. A full real Windows-host/Linux-container reproduction has not been posted.

## Solution Assessment

No. The fix should parse or normalize Windows path structure into the existing agent-aware resolver path rather than returning the basename before containment and cross-agent checks run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve agent context when mapping Windows paths:** `src/config/sessions/paths.ts:201`
  - body: This early return rewrites any Windows drive path to `<sessionsDir>/<basename>` before the existing agent/root fallback logic runs. If bot1 reads a Windows-authored `sessionFile` for `...\agents\bot2\sessions\sess-1.jsonl`, this maps it to bot1's `sess-1.jsonl`, and `resolveAndPersistSessionFile` can persist that alias. Keep the Windows-path fix inside the existing agent-aware resolution path, or reject/fallback when the agent/root cannot be proven.
  - confidence: 0.86
- **[P3] Handle Windows paths without a usable filename:** `src/config/sessions/paths.ts:198-203`
  - body: When the detected Windows path ends in a separator, `extractBasename` returns an empty string and the guard falls through to the old POSIX relative-path behavior. That preserves the malformed drive-letter path under the local sessions directory for inputs the new branch explicitly recognized as Windows absolute; reject these paths or fall back to the derived transcript path instead.
  - confidence: 0.76
