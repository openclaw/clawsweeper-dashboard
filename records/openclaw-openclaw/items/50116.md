## Summary

Keep this PR open, but do not merge the current head: it targets a real Windows-host/POSIX-container session path bug, but the basename rewrite can mis-associate transcripts across agents and the external PR still lacks real behavior proof.

## What This Changes

The PR adds POSIX-only Windows drive-letter detection for persisted sessionFile paths, maps them to the local sessions directory by basename, adds one regression test, and updates the changelog.

## Best Possible Solution

Parse Windows absolute paths with Windows semantics, preserve valid agents/<agentId>/sessions/<file> structure through the existing containment and fallback rules, and prove the Windows-host/POSIX-container upgrade path.

## Reproduction Assessment

Yes, at source level: current main uses POSIX path.isAbsolute, and a POSIX Node probe shows a Windows drive-letter sessionFile is treated as relative and appended under the sessions directory. I did not run the full Windows-host/POSIX-container flow.

## Solution Assessment

No. The branch fixes the simplest same-agent symptom but bypasses existing realpath, containment, sibling-agent, and structural fallback logic, so the safer fix should translate Windows paths into that resolver flow instead of collapsing to basename.
