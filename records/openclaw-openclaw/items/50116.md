## Summary

Keep open: this PR targets a real Windows-on-POSIX session path bug, but the branch is not merge-ready because it collapses Windows absolute transcript paths to a basename, bypassing the current agent/root preservation contract, and the external PR still lacks real behavior proof.

## What This Changes

The branch adds a POSIX-only Windows drive-letter sessionFile detector, maps such paths to the local sessions directory by basename, adds a focused regression test, and updates the changelog.

## Best Possible Solution

Parse Windows absolute session paths with Windows semantics, preserve valid agents/<agentId>/sessions/<file> structure through the existing containment and fallback rules, and cover same-agent plus cross-agent Windows-path cases.

## Reproduction Assessment

Yes. Source inspection plus a focused Node probe show POSIX path.isAbsolute does not recognize Windows drive-letter sessionFile values, and the PR's basename rewrite can route a cross-agent Windows path into the current sessions directory; I did not run the full Windows-host/POSIX-container flow.

## Solution Assessment

No. The branch fixes the simplest malformed-path symptom but bypasses existing realpath, containment, and agent-preservation logic, so the safer fix should preserve path structure instead of collapsing to basename.
