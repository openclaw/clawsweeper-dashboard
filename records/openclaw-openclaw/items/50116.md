## Summary

Keep open: the Windows-on-POSIX path problem is real by source inspection, but this branch should not merge because the basename rewrite bypasses current cross-agent/cross-root session path rules and can alias another agent's transcript; it also lacks required real behavior proof.

## What This Changes

The PR adds a POSIX guard for Windows drive-letter `sessionFile` values, maps them to the local sessions directory by basename, adds a regression test, and updates the changelog.

## Best Possible Solution

Parse Windows absolute paths with Windows path semantics, preserve canonical `agents/<agentId>/sessions/<file>` structure through the existing containment/fallback rules, and fall back to the derived transcript path when the structure is unsafe.

## Reproduction Assessment

Yes, by source inspection. On POSIX, current `path.isAbsolute` does not recognize `C:\...`, and the PR branch is also source-reproducible for the cross-agent aliasing case; I did not run tests because this review is read-only.

## Solution Assessment

No. The basename-only rewrite fixes the simplest malformed path but is not the best maintainable fix because it bypasses existing containment and agent-preservation logic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve agent context when rewriting Windows paths:** `src/config/sessions/paths.ts:201`
  - body: This return collapses every Windows absolute `sessionFile` to `<sessionsDir>/<basename>`. If the stored path points at another agent or state root, the PR aliases it to the current agent's transcript name, and `resolveAndPersistSessionFile()` can persist that wrong path back to the store; please parse the Windows path structure and preserve valid `agents/<agentId>/sessions/<file>` paths through the existing containment rules.
  - confidence: 0.9
