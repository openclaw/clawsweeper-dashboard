## Summary

This PR should stay open: the core ordinal-tracking idea is useful, but the latest head still has a blocking ACK-classifier false positive and a route-matching gap that can add misleading Discord checkpoints.

## What This Changes

The PR tracks post-message-tool activity in embedded agent runs and adds a Discord-only terminal checkpoint when an acknowledged longwork turn ends without a final visible payload.

## Best Possible Solution

Keep the completed-tool ordinal tracking, but only synthesize the checkpoint for genuine future/progress ACK text delivered to the same Discord route, then verify that path with redacted Discord-visible proof.

## Reproduction Assessment

Yes for the blocking finding: a direct regex probe against the PR predicate shows final first-person completion messages still classify as ACK/progress. The original Discord silence is source-reproducible from the current empty-payload path, but I did not live-reproduce it in Discord.

## Solution Assessment

No, not yet. The ordinal-tracking direction is maintainable, but the checkpoint decision needs stricter completion-text exclusion and same-route message-tool evidence before it is safe to merge.
