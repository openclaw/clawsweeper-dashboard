## Summary

Keep open: the direction is plausible, but the ACK classifier is too broad and can synthesize a misleading Discord checkpoint after a final-looking first-person message, and the external PR only provides mocked/test proof rather than real Discord behavior evidence.

## What This Changes

The PR tracks post-message-tool tool activity through the embedded runner and adds a Discord-only synthesized terminal checkpoint plus regression coverage for ACK/progress turns that would otherwise end silently.

## Best Possible Solution

Keep the ordinal-based post-message activity signal, but tighten the ACK/progress text predicate with negative coverage for final-looking first-person messages and require redacted real Discord proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence mocked reproduction for the review finding: a Discord run with no payloads, `toolActivityAfterMessagingToolDelivery: true`, and message-tool text like “I'm done checking; the patch is ready” passes the new regex and would return the terminal checkpoint.

## Solution Assessment

No, not yet; the ordinal tracking is a maintainable shape, but the text classifier needs tightening so final-message-plus-cleanup cases stay suppressed as the PR intends.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Narrow the ACK text classifier:** `src/auto-reply/reply/agent-runner.ts:176-178`
