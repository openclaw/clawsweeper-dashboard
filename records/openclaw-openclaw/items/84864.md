## Summary

Keep open: the ordinal-based post-message activity signal is useful, but the new ACK classifier can still misclassify first-person final results and the supplied proof is isolated test output rather than live Discord behavior.

## What This Changes

The PR adds completed-tool ordinal tracking through the embedded runner and a Discord-only terminal checkpoint when an ACK/progress message is followed by later tool activity with no final visible reply.

## Best Possible Solution

Keep the ordinal-based post-message activity signal, tighten the ACK/progress predicate to exclude final-result phrasing, add the missing negative coverage, and require redacted real Discord proof before merge.

## Reproduction Assessment

Yes for the review finding: source inspection shows that a Discord run with no payloads, `toolActivityAfterMessagingToolDelivery: true`, and first-person final text such as `I'm done checking; the patch is ready` passes the new regex and returns the checkpoint. I did not live-reproduce the original Discord silence on current main in this read-only review.

## Solution Assessment

No, not yet; the ordinal tracking is a maintainable direction, but the ACK classifier still needs to avoid final-result messages and the visible Discord behavior needs real transport proof.
