## Summary

Keep open: current main still does not treat `shouldStripStore` as a replay-off signal, and the linked open fix PR at https://github.com/openclaw/openclaw/pull/89128 owns the focused repair and proof.

## What This Changes

The issue asks OpenClaw to stop replaying stale OpenAI Responses item IDs when the effective Responses request is non-persistent.

## Best Possible Solution

Land the linked PR or an equivalent guard that treats `shouldStripStore` as non-persistent for replay while preserving store-enabled replay behavior.

## Reproduction Assessment

Yes. Source inspection shows current main sets `shouldStripStore` for `supportsStore=false` routes but still allows replay IDs because the replay gate only checks `explicitStore !== false`.

## Solution Assessment

Yes. Reusing the existing `shouldStripStore` policy at the wrapper and transport-builder gates is the narrow maintainable fix; a new config option or provider-specific special case would duplicate existing policy.
