## Summary

Keep open: current main still prepends full fenced Conversation info and Sender JSON blocks to direct Telegram turns, and #72749 is an open unmerged closing PR rather than a shipped fix.

## What This Changes

Request compact or structured handling for Telegram inbound user-message metadata so model-facing text is not dominated by repeated inline JSON blocks.

## Best Possible Solution

Review and land #72749 or an equivalent narrow inbound prompt-builder change that compacts direct-chat metadata while preserving full untrusted context for groups, replies, forwards, locations, structured context, and history.

## Reproduction Assessment

Yes. A direct Telegram inbound context sets ChatType direct with Provider/Surface/OriginatingChannel telegram, then buildInboundUserContextPrefix emits the full metadata blocks and get-reply-run prepends them to the user body.

## Solution Assessment

Unclear. The issue is valid on current main, but the maintainable fix should be reviewed through #72749 or an equivalent focused change; provider-specific structured metadata fields are not portable across all supported model backends.
