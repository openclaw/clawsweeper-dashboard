## Summary

Keep open. Current main still prepends full fenced Conversation info and Sender JSON blocks for direct non-webchat channel messages, including Telegram, and open PR #72749 is already the explicit closing implementation candidate. Closing this issue before that PR lands would be premature.

## What This Changes

The issue requests reducing Telegram/direct-chat prompt noise by replacing repeated inline JSON metadata blocks with structured or compact inbound metadata.

## Best Possible Solution

Review and land #72749, or an equivalent narrow prompt-builder/config change, so direct Telegram-style chats can opt into compact metadata while groups, replies, forwards, locations, and richer history contexts keep the fuller untrusted context where it is still useful.

## Reproduction Assessment

Yes. A Telegram DM finalizes inbound context with ChatType direct, telegram provider/surface, message id, destination, and sender fields; current prompt assembly then prepends buildInboundUserContextPrefix output before the user body, and that builder emits full fenced Conversation info and Sender blocks.

## Solution Assessment

Yes for triage: keeping this issue open while the linked closing PR is reviewed is the right handling. The implementation should stay narrow in inbound prompt construction/config and avoid dropping richer metadata from group or contextual message flows.
