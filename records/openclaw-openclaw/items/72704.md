## Summary

Keep open: current main still emits full fenced Conversation info and Sender metadata for direct Telegram/external-channel turns, and the linked compact-metadata PR is open and unmerged rather than implemented behavior.

## What This Changes

Request reducing or moving Telegram/direct inbound metadata out of user message text so short messages are not preceded by large repeated JSON blocks.

## Best Possible Solution

Review, repair, land, or close https://github.com/openclaw/openclaw/pull/72749; if that path is abandoned, consolidate the remaining design work under https://github.com/openclaw/openclaw/issues/62077.

## Reproduction Assessment

Yes for the observable prompt-shape complaint: source inspection shows Telegram DMs produce direct Telegram context, the inbound prefix builder emits full metadata blocks, and the runner carries that context into the model turn. The claimed model-comprehension degradation was not live-measured in this read-only review.

## Solution Assessment

Unclear. A compact direct metadata mode is plausible, but maintainers still need to choose the default versus opt-in config shape and preserve attribution/sanitization contracts before this should be considered solved.
