## Summary

Keep open: current main still sends full fenced Conversation info and Sender metadata blocks into the model prompt for direct Telegram/external-channel turns, and https://github.com/openclaw/openclaw/pull/72749 is still an open implementation candidate rather than a merged fix.

## What This Changes

The issue requests reducing or restructuring verbose inline Telegram inbound metadata that is prepended to short user messages.

## Best Possible Solution

Review and land, repair, or close https://github.com/openclaw/openclaw/pull/72749; if that path is abandoned, consolidate the remaining product decision with https://github.com/openclaw/openclaw/issues/62077.

Yes. Source inspection gives a high-confidence path: Telegram DMs build direct Telegram context, buildInboundUserContextPrefix emits full untrusted metadata blocks for direct external channels, and the runner prepends that context to the model prompt.

## Solution Assessment

Unclear. A compact direct metadata mode is plausible, but maintainers still need to choose the default versus opt-in config shape and ensure sender attribution, rich context, and provider portability remain intact.
