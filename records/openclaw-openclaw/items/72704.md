## Summary

Keep open. Current main still prepends full fenced Conversation info and Sender metadata blocks for direct Telegram-style external-channel messages, and the open linked PR remains the active closing implementation candidate rather than a reason to close this issue early.

## What This Changes

The issue requests a less noisy Telegram inbound prompt format so repeated conversation/sender metadata is not injected as large inline JSON blocks before short user messages.

## Best Possible Solution

Review and land #72749, or an equivalent narrow config-backed prompt-builder and sanitizer change, so direct Telegram-style chats can use compact metadata while groups, replies, forwards, locations, and richer history surfaces keep the fuller untrusted context.

## Reproduction Assessment

Yes. A direct Telegram message context with Telegram provider/surface, message id, destination, and sender fields flows into buildInboundUserContextPrefix, which currently emits fenced Conversation info and Sender blocks that get prepended before the user body.

## Solution Assessment

Yes for triage: keeping this issue open while #72749 is reviewed is the right handling. The implementation should stay in inbound prompt construction/config and avoid dropping richer metadata where it is still needed.
