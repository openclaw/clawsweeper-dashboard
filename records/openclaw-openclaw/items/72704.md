## Summary

Keep open. Current main still prepends full fenced Conversation info and Sender JSON blocks to direct non-webchat channel messages, including Telegram DMs, and PR #72749 is still the explicit open implementation candidate with closing syntax for this issue.

## What This Changes

Request compacting or moving Telegram inbound metadata out of inline user-message content so short direct messages are not preceded by full Conversation info and Sender JSON blocks.

## Best Possible Solution

Land PR #72749 or an equivalent narrow prompt-builder/config change so direct Telegram-style chats can opt into compact metadata while groups, replies, forwards, locations, and richer history contexts keep the fuller untrusted context.

## Reproduction Assessment

Yes. A Telegram DM finalizes inbound context with ChatType direct, Provider/Surface telegram, sender fields, and a message id; current prompt assembly then prepends buildInboundUserContextPrefix output before the user body, and that builder emits full fenced Conversation info and Sender blocks.

## Solution Assessment

Yes for triage: keeping this issue open while the linked closing PR is reviewed is the right handling. The implementation should stay narrow in inbound prompt construction/config and avoid dropping richer metadata from group or contextual message flows.
