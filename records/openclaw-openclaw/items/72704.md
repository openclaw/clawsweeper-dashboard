## Summary

Keep open. Current main still prepends full fenced Conversation info and Sender JSON blocks for direct Telegram-style inbound messages, and #72749 remains the open implementation candidate that explicitly closes this issue.

## What This Changes

The issue asks OpenClaw to reduce or restructure the verbose inline Telegram inbound metadata that is currently prepended to short direct user messages.

## Best Possible Solution

Land #72749 or an equivalent narrow prompt-builder/config change that supports compact direct-chat inbound metadata for Telegram-style DMs while preserving full untrusted metadata for groups, replies, forwards, locations, structured context, and richer history flows.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: build a direct Telegram inbound context with Provider/Surface telegram, sender fields, and a message id; current prompt assembly prepends buildInboundUserContextPrefix output, which emits the full fenced Conversation info and Sender blocks before the user body.

## Solution Assessment

Yes for triage. Keeping this issue open while #72749 is reviewed is the narrowest maintainable path; the fix should stay in shared inbound prompt construction/config rather than relying on provider-specific model metadata fields.
