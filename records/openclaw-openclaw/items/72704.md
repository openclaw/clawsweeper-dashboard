## Summary

Keep open: current main still emits full fenced Conversation info and Sender metadata for direct Telegram-style inbound turns, and the open closing PR is an implementation candidate rather than landed behavior.

## What This Changes

The issue asks OpenClaw to stop prepending repeated full inline JSON metadata to Telegram direct user messages, preferably by using structured or compact metadata.

## Best Possible Solution

Review, repair, land, or close https://github.com/openclaw/openclaw/pull/72749; if that path is abandoned, consolidate the remaining compact direct-message metadata design under https://github.com/openclaw/openclaw/issues/62077.

## Reproduction Assessment

Yes for the observable prompt-shape complaint: source inspection shows Telegram DMs build direct Telegram context, full Conversation/Sender metadata blocks are emitted, and get-reply-run passes that prefix into the model turn. The claimed model-comprehension degradation was not live-measured.

## Solution Assessment

Unclear. A compact direct metadata mode is plausible, but maintainers still need to choose the default versus opt-in config shape and preserve attribution and sanitization contracts before this should be considered solved.
