## Summary

Keep open. Current main still prepends full fenced Conversation info and Sender JSON blocks to direct non-webchat channel messages, Telegram supplies the direct-channel fields that trigger that path, and open PR #72749 is already the explicit closing implementation candidate.

## What This Changes

The issue requests reducing or restructuring the inline JSON metadata prepended to Telegram user messages so short user text is not dominated by repeated metadata blocks.

## Best Possible Solution

Review and land #72749, or an equivalent narrow prompt-builder/config change, so direct Telegram-style chats can opt into compact metadata while group chats, replies, forwards, locations, and richer history contexts keep the fuller untrusted context where it remains useful.

## Reproduction Assessment

Yes. A Telegram DM finalizes inbound context with direct chat type, telegram provider/surface, message id, destination, and sender fields; current prompt assembly then prepends buildInboundUserContextPrefix output before the user body, and that builder emits full fenced Conversation info and Sender blocks.

## Solution Assessment

Yes for triage: keeping this issue open while the linked closing PR is reviewed is the right handling. The implementation should stay narrow in inbound prompt construction/config and avoid dropping richer metadata from group or contextual message flows.
