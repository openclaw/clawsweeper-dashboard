## Summary

Keep open. Current main still prepends full fenced Conversation info and Sender JSON blocks for direct external-channel messages, and #72749 is the open, unmerged implementation candidate linked to close this issue.

## What This Changes

The issue requests reducing Telegram/direct-chat inbound metadata in model-facing user messages by moving it out of inline JSON blocks or rendering it in a compact format.

## Best Possible Solution

Land #72749 or an equivalent narrow prompt-builder/config change that supports compact direct-chat inbound metadata for Telegram-style DMs while preserving full untrusted metadata for groups, replies, forwards, locations, structured context, and richer history flows.

## Reproduction Assessment

Yes. A Telegram DM finalizes inbound context with ChatType direct, Provider/Surface telegram, sender fields, and a message id; current prompt assembly then prepends buildInboundUserContextPrefix output before the user body, and that builder emits full fenced Conversation info and Sender blocks.

## Solution Assessment

Yes for triage. Keeping this issue open while the linked implementation PR is reviewed is the narrowest maintainable path; a fix should stay in inbound prompt construction/config rather than relying on provider-specific metadata fields.
