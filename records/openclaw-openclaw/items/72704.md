## Summary

Keep open. Current main still prepends full fenced Conversation info and Sender JSON blocks ahead of direct Telegram-style user text, and open PR #72749 is the explicit closing implementation candidate, so closing this issue before that PR lands would be premature.

## What This Changes

The issue requests a compact or structured alternative to inline Telegram inbound metadata so user text is not buried under repeated JSON prompt prefixes.

## Best Possible Solution

Review and land #72749, or an equivalent narrow prompt-builder/config change, so direct Telegram-style chats can opt into compact metadata while group chats, replies, forwards, locations, and richer context keep the fuller untrusted metadata where it remains useful.

## Reproduction Assessment

Yes. A direct Telegram message finalizes with ChatType direct, Provider/Surface telegram, sender fields, and a message id; current prompt assembly then prepends buildInboundUserContextPrefix output before the user body, and that builder emits the full fenced Conversation info and Sender blocks.

## Solution Assessment

Yes for triage. Keeping this issue open while #72749 is reviewed is the right handling; the implementation should stay localized to inbound prompt construction/config rather than relying on provider-specific metadata fields that are not portable across model backends.
