## Summary

Keep open: current main still prepends full fenced Conversation info and Sender JSON blocks to direct non-webchat channel messages, and Telegram DMs provide the fields that trigger that path. The linked open PR #72749 is still the explicit implementation candidate for this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests reducing Telegram/direct-chat prompt noise by moving or compacting repeated inline Conversation info and Sender JSON metadata before user text.

## Best Possible Solution

Land PR #72749 or an equivalent narrow prompt-builder/config change that lets direct Telegram-style chats use compact AI-facing metadata while preserving fuller untrusted context for groups, replies, forwards, locations, and richer history flows.

## Reproduction Assessment

Yes. A Telegram DM finalizes inbound context with ChatType direct, Provider/Surface telegram, sender fields, and a message id; current prompt assembly then prepends buildInboundUserContextPrefix output before the user body, and that builder emits full fenced Conversation info and Sender blocks.

## Solution Assessment

Yes for triage: keeping this issue open while the linked closing PR is reviewed is the right handling. The implementation should stay narrow in inbound prompt construction/config and avoid dropping richer metadata from group or contextual message flows.
