## Summary

Keep open. The underlying bug is still present on current main, but this PR edits an obsolete memory path and is dirty against the base, so it cannot fix the active memory database path as-is.

## What This Changes

The branch removes a bare catch around `fsSync.mkdirSync(dir, { recursive: true })` in `src/memory/internal.ts` so mkdir failures propagate.

## Best Possible Solution

Retarget the fix to `packages/memory-host-sdk/src/host/internal.ts`, keep the current source-side mirror coherent as a re-export, and add a narrow Vitest regression that proves `ensureDir` propagates the original filesystem error before landing this work or a replacement PR.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: `openMemoryDatabaseAtPath` calls the SDK facade, the facade reaches the package helper, and that helper catches every `mkdirSync` failure. A focused unit test can reproduce it by making `fsSync.mkdirSync` throw an EACCES/ENOSPC-style error and asserting `ensureDir` currently hides it.

## Solution Assessment

No as-is. Removing the catch is the right behavioral direction, but the PR edits `src/memory/internal.ts`, which current main no longer uses; the maintainable fix belongs in the active memory host SDK package helper with a regression test.
