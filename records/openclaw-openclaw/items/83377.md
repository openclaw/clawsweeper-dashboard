## Summary

Keep open. This maintainer-labeled refactor is not merge-ready because the PR still regresses explicit no-deliver cron message-tool delivery state and lacks real cron behavior proof.

## What This Changes

The PR adds an internal source delivery plan/outcome helper and migrates isolated cron delivery planning, execution, dispatch, and tests to use it instead of cron-local booleans.

## Best Possible Solution

Keep the shared plan abstraction, but model explicit no-deliver cron targets as message-tool-owned without runner fallback, add a regression test for delivered/deliveryAttempted and pending-warning behavior, then collect redacted live cron proof or a maintainer override.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main treats a matched message-tool send as delivered, while the PR's `cron_none` plan cannot mark the same send as satisfying delivery; I did not run a live cron job.

## Solution Assessment

No. The shared contract is a maintainable direction, but this implementation must preserve explicit mode-none message-tool sends as completed visible delivery while still disabling runner fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve mode-none message tool deliveries:** `src/cron/isolated-agent/run.ts:294-301`
  - body: When `delivery.mode` is `"none"` with an explicit target, current main still treats a matching `message` tool send as delivered and attempted. This plan uses owner `none` without the resolved target/satisfaction path, so the same successful send is reported as undelivered and pending presentation warnings can become false fatal errors. Keep no runner fallback, but let explicit mode-none targets satisfy delivery when the agent sends to the resolved target.
  - confidence: 0.9
