## Summary

Keep this open: current main and v2026.6.6 still provide only single file/line attribution plus a separate Memory Wiki provenance layer, while the requested durable multi-source provenance, merge lineage, and promotion gate remain missing.

## What This Changes

The issue requests a Memory MVP provenance model requiring every durable memory to retain source references across promotion, merges, edits, retrieval, and review flows.

## Best Possible Solution

Define provenance as a required canonical relation in the v1 durable memory record/revision model, seed it from framework-owned session/message/file origins, preserve it through merge/edit flows, and expose it through retrieval/review while keeping trust policy layered separately.

## Reproduction Assessment

Not applicable. This is a new durable-memory provenance capability rather than a broken existing contract; source inspection shows the requested invariant is still absent.

## Solution Assessment

Yes at the architectural level, but not as an isolated patch. The maintainable path is to design it with the canonical schema, write pipeline, review/edit flows, CLI surface, and separate trust-policy work instead of adding another attribution side channel.
