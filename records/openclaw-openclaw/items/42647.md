## Summary

The requested provenance invariant remains materially missing from current main: memory-core exposes file-and-line citations and preserves one source location during promotion, but it does not model multiple supporting sources or lineage across record revisions and merges. This should remain open for a coordinated memory product decision rather than be treated as a narrow bug or duplicate.

## What This Changes

Add mandatory source references to durable memories, preserve multi-source lineage through edits and merges, and expose attribution in retrieval and review flows.

## Best Possible Solution

Define provenance as a required relation in the canonical memory-plugin SQLite record and revision model, seed it from framework-derived path/message/session references, union it during merges, and render it through the existing citation surface while keeping trust policy layered separately.

## Reproduction Assessment

Not applicable. This issue requests a new multi-source durable-memory invariant; current main only defines basic file-and-line attribution, so there is no broken existing contract to reproduce.

## Solution Assessment

Yes at the architectural level, but not as an isolated patch. The maintainable solution is to make provenance part of the canonical record/revision schema and write pipeline, reusing current citations rather than adding parallel metadata or trust configuration.
