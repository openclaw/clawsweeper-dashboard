## Summary

Keep open: the child-env hardening is useful and the supplied terminal proof is sufficient, but the current PR route for Gemini hides the real `gemini` command from acpx and regresses existing Gemini-specific command handling before merge.

## What This Changes

The PR adds generated ACPX wrappers, docs, tests, and live Docker plumbing so built-in Claude and Gemini child harnesses strip selected provider auth environment variables, with an explicit Gemini preserve-env flag.

## Best Possible Solution

Keep the wrapper-level credential stripping, but preserve acpx's existing Gemini flag normalization and startup handling before leaving broader auth-contract, doctor, and preflight UX to the linked issue.

## Reproduction Assessment

Yes, source-level: acpx 0.7.0 only applies Gemini `--acp` compatibility and startup timeout behavior when the spawned command basename is `gemini`, while this PR makes acpx spawn a Node wrapper for Gemini. A focused fake old-Gemini binary test can reproduce the flag regression without live credentials.

## Solution Assessment

No, not as written. The generated wrapper is the right boundary for child-env scrubbing, but the implementation must carry forward acpx's existing Gemini command semantics or keep acpx's Gemini detection path active.
