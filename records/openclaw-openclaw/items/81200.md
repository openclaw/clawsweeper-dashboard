## Summary

Keep open: the PR is a plausible ACPX credential-boundary hardening slice, but it still changes the documented Gemini ACP auth contract by stripping API-key and Vertex env from all built-in Gemini launches without an explicit opt-in or matching docs/product decision.

## What This Changes

The PR generates Claude and Gemini ACPX wrapper scripts that strip selected provider auth env vars from child harness processes, updates wrapper discovery/reaping, and adds env-stripping tests.

## Best Possible Solution

Keep the child-env stripping boundary, but add an explicit Gemini API-key/Vertex opt-in or update the supported auth contract and docs before merging.

Yes, at source level: PR head routes built-in Gemini through a wrapper that deletes env vars used by current docs, the live ACP bind lane, and upstream Gemini for API-key/Vertex auth. I did not run a live provider session in this read-only review.

## Solution Assessment

No, not yet: the env-stripping wrapper is the right hardening boundary, but removing a documented Gemini API-key path needs an explicit opt-in/auth contract or matching docs change before merge.
