Keep this PR open for normal maintainer handling: the latest head addresses the earlier Gemini API-key concern with an explicit preserve flag, matching docs, and live-script support, and I did not find a blocking correctness issue in the read-only review.

## What This Changes

The PR updates ACPX built-in Claude and Gemini wrapper generation to strip selected provider auth environment variables from child harnesses while allowing explicit Gemini env-auth preservation.

## Best Possible Solution

Keep the wrapper-level env scrubbing default, preserve explicit Gemini env auth only through the documented opt-in, and let maintainer review plus CI/live smoke decide whether this is the accepted ACPX auth contract.

Yes for a high-confidence source-level path: current main lets ACP child harnesses inherit provider env, while upstream Gemini uses the stripped variables for auth selection and API-key/Vertex config. I did not run a live provider session in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the new auth contract: the latest PR keeps the hardening at the generated-wrapper boundary and adds an explicit, documented Gemini env-auth opt-in instead of silently breaking API-key/Vertex use.
