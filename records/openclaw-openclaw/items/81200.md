## Summary

Keep open: the PR is a useful ACPX auth-boundary hardening change and the supplied console proof is meaningful, but the Gemini built-in strip list still misses upstream auth selector env vars that can keep the wrong Gemini auth lane reachable on mixed-credential hosts.

## What This Changes

The PR adds generated ACPX wrappers for built-in Claude and Gemini agents that strip selected provider API-key env vars from spawned child harnesses, plus tests and process tracking updates for the Gemini wrapper.

## Best Possible Solution

Merge the narrow wrapper hardening after extending Gemini's built-in scrub metadata and tests to cover the auth selector env vars identified in the linked hardening discussion; leave doctor warnings and configurable opt-in policy for a follow-up.

## Reproduction Assessment

No runtime reproduction was run in this read-only review. Source inspection gives a high-confidence targeted path: launch the generated Gemini wrapper with GOOGLE_GENAI_USE_GCA or GOOGLE_GENAI_USE_VERTEXAI set and observe that the current PR would still pass those variables to the child.

## Solution Assessment

No, not yet. The generated-wrapper approach is the right narrow place for this PR, but the Gemini metadata needs to match the upstream auth selectors and the linked issue's stated strip surface before this is the best ACPX hardening fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Include Gemini auth selectors in the child env scrub:** `extensions/acpx/src/codex-auth-bridge.ts:42`
  - body: The new Gemini wrapper only deletes `GEMINI_API_KEY` and `GOOGLE_API_KEY`, but Gemini CLI still derives auth from `GOOGLE_GENAI_USE_GCA` and `GOOGLE_GENAI_USE_VERTEXAI` before it considers `GEMINI_API_KEY`. A mixed-credential host with either selector set can still launch the ACP child in the wrong auth lane, so please scrub the full built-in Gemini auth surface called out in the linked hardening issue and cover it in the child-env/parent-env tests.
  - confidence: 0.86
