## Summary

Keep open: this is useful ACPX credential-boundary hardening, but it still needs a maintainer auth-contract decision because the Gemini wrapper strips env vars that current docs, live validation, and upstream Gemini use for supported API-key and Vertex auth paths.

## What This Changes

The PR adds generated ACPX Claude/Gemini wrappers and tests that strip selected provider API-key and auth-selector env vars from spawned child harness environments while preserving the parent env.

## Best Possible Solution

Keep the child-env stripping boundary, but add an explicit Gemini API-key/Vertex opt-in or update the supported auth contract and docs before merging.

## Reproduction Assessment

Yes for a source-level reproduction path: the PR routes built-in Gemini through a wrapper that deletes env vars used by current docs, live harness code, and upstream Gemini for API-key/Vertex auth. I did not run a live provider session in this read-only review.

## Solution Assessment

No, not yet: the env-stripping wrapper is the right hardening boundary, but the PR removes a documented Gemini API-key path without the explicit opt-in or contract/docs decision described in the linked issue discussion.
