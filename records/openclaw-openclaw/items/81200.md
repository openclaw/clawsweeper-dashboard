## Summary

Keep open: the child-env hardening is useful and has terminal proof, but the latest PR still removes the documented Gemini API-key/Vertex env auth path for the built-in Gemini ACP harness without the explicit opt-in or contract decision requested in the linked discussion.

## What This Changes

The PR adds ACPX Claude and Gemini wrapper metadata plus tests so generated child wrappers strip selected provider API-key and auth-selector env vars while preserving the parent env.

## Best Possible Solution

Keep the child-env stripping boundary, but add an explicit Gemini API-key/Vertex opt-in or change the supported contract/docs before routing all built-in Gemini launches through stripped env.

## Reproduction Assessment

Yes for a source-level reproduction path: the PR routes built-in Gemini through a wrapper that deletes the env vars that current docs, live harness code, and upstream Gemini CLI use for API-key and Vertex auth. I did not run a live provider session in this read-only review.

## Solution Assessment

No, not yet: the env-stripping wrapper is the right hardening boundary, but the PR removes an existing documented Gemini API-key path without the explicit auth-contract or opt-in surface described in the linked issue discussion.
