## Summary

Keep open: the PR is a useful ACPX child-env hardening slice with sufficient wrapper proof, but it still unconditionally removes Gemini API-key and Vertex auth paths that current docs and live harness support.

## What This Changes

The PR adds ACPX Claude and Gemini wrapper metadata so generated child ACP harnesses strip selected provider auth environment variables while preserving unrelated and parent process environment values.

## Best Possible Solution

Keep the child-env stripping boundary, but preserve the documented Gemini API-key/Vertex path through an explicit opt-in or an explicit docs/contract change before merge.

## Reproduction Assessment

Yes for a source-level reproduction path: the PR routes built-in Gemini through a wrapper that deletes env vars used by current docs, live harness code, and upstream Gemini CLI for API-key and Vertex auth. I did not run a live provider session in this read-only review.

## Solution Assessment

No, not yet: the env-stripping wrapper is a reasonable hardening boundary, but the PR removes an existing documented Gemini auth path without the explicit opt-in or contract change described in the linked issue discussion.
