## Summary

Keep this active PR open: the child-env hardening is well scoped and has terminal proof, but the latest diff appears to break OpenClaw's current API-key-backed Gemini ACP live path unless maintainers intentionally want to remove that default behavior.

## What This Changes

The PR adds ACPX generated Claude and Gemini wrappers that strip selected provider auth environment variables from spawned child harnesses, plus wrapper recognition updates and focused env-preservation tests.

## Best Possible Solution

Preserve the focused child-env hardening while adding an explicit maintainer-approved path for API-key-backed Gemini ACP, or document and update live coverage if that support is intentionally removed.

## Reproduction Assessment

Yes for the narrow source-level behavior: current main forwards ACP child env and Gemini CLI reads these env names, while the PR proves they are stripped in generated wrappers. I did not reproduce the durable OAuth drift or the API-key live bind regression in a real provider session.

## Solution Assessment

No, not yet: stripping only the child env is the right boundary, but applying it unconditionally to the built-in Gemini alias conflicts with the existing API-key-backed ACP live path. The safer solution is to preserve or explicitly replace that path before merge.
