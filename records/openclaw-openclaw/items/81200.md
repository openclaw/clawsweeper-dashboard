## Summary

Keep open: the PR is a useful ACPX auth-boundary hardening slice, but the latest head still unconditionally routes the built-in Gemini alias through an env-stripping wrapper, which conflicts with the current API-key-backed Gemini ACP live path unless maintainers explicitly choose to remove or replace that path.

## What This Changes

The PR adds generated ACPX Claude and Gemini wrappers that delete selected provider auth environment variables before spawning child harnesses, plus wrapper recognition updates and focused tests.

## Best Possible Solution

Keep the child-env hardening, but either preserve an explicit Gemini API-key ACP path or intentionally remove it with maintainer approval and matching docs/live-harness updates.

## Reproduction Assessment

Yes for the source-level regression path: current main prepares and forwards Gemini API-key env for the live ACP bind lane, upstream Gemini CLI reads those env names, and the PR deletes them from the generated Gemini wrapper. I did not run a live provider session in this read-only review.

## Solution Assessment

No, not yet: stripping child env is the right boundary, but doing it unconditionally for the built-in Gemini alias removes an existing API-key-backed ACP path. The safer solution is to preserve that path behind an explicit policy knob or update the supported contract and validation if maintainers want it removed.
