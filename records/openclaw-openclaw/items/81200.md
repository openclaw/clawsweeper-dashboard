## Summary

Keep open: the ACPX env hardening is useful and the PR supplies real wrapper-output proof, but the patch bypasses acpx's existing version-aware Gemini ACP flag fallback, so older supported Gemini CLI installs can stop launching.

## What This Changes

The PR adds Claude and Gemini ACP wrapper metadata and strips selected provider API-key environment variables from generated built-in child harness wrappers while leaving the parent process environment unchanged.

## Best Possible Solution

Keep the child-env stripping approach, but preserve acpx's Gemini `--acp` to `--experimental-acp` compatibility and leave broader auth-contract or doctor work tracked separately unless maintainers explicitly accept this as complete.

## Reproduction Assessment

Yes, from source: acpx 0.7.0 only performs the Gemini ACP flag rewrite when it directly sees `gemini --acp`, while this PR makes acpx see a Node wrapper that always launches `gemini --acp`. I did not run an older Gemini CLI binary.

## Solution Assessment

No, not yet: wrapping Gemini is a maintainable place to strip env, but it must preserve the existing acpx launch contract before merge. The broader auth-contract and doctor/readiness work can remain a follow-up.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve Gemini ACP flag fallback in the wrapper:** `extensions/acpx/src/codex-auth-bridge.ts:372`
  - body: acpx 0.7.0 rewrites direct `gemini --acp` launches to `--experimental-acp` for Gemini CLI versions below 0.33.0 before spawning. After this change acpx only sees `node gemini-acp-wrapper.mjs`, so that compatibility path is bypassed while the generated wrapper always adds `--acp`; older supported Gemini CLI installs will fail to start. Move the same version-aware arg resolution into the wrapper or otherwise keep acpx's direct Gemini launch semantics.
