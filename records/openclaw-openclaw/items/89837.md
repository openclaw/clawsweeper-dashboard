## Summary

Keep open: current main and v2026.6.9 still do not provide a Codex app-server native PATH contract for `tools.exec.pathPrepend`, and the remaining choice is whether to extend that existing exec config into Codex native execution or document a separate supported PATH setup.

## What This Changes

The issue asks OpenClaw to make `tools.exec.pathPrepend` affect Codex app-server native shell PATH after config changes, or to document a separate supported PATH contract.

## Best Possible Solution

Make the Codex native exec PATH boundary explicit with either a tested Codex app-server PATH projection plus client rekey/restart semantics, or documentation that names the supported service/app-server environment path instead.

## Reproduction Assessment

Yes, at source level. Current main scopes `tools.exec.pathPrepend` to gateway/sandbox exec, the Codex app-server config path does not project it into the native app-server process environment, and sibling Codex source shows `exec_command` is environment-backed; I did not run a live Codex turn.

## Solution Assessment

Unclear. Reusing `tools.exec.pathPrepend` for Codex native exec is plausible, but maintainers still need to choose whether that config should cross the current gateway/sandbox boundary or stay documented as non-Codex-native behavior.
