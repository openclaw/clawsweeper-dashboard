## Summary

Keep open: current main and the latest release still do not provide a Codex app-server native PATH contract, and the remaining decision is whether to extend existing exec config into Codex native execution or document a separate supported path.

## What This Changes

The issue asks OpenClaw to make `tools.exec.pathPrepend` affect Codex app-server native exec PATH after config reload, or explicitly document and support a separate Codex PATH surface.

## Best Possible Solution

Make the Codex native exec PATH boundary explicit: either add a tested Codex-owned PATH projection with shared-client rekey/restart semantics, or document the supported service/app-server environment path and keep `tools.exec.pathPrepend` gateway/sandbox-only.

## Reproduction Assessment

Yes, at source level. Current main does not project `tools.exec.pathPrepend` into Codex app-server `start.env`, the reload watcher treats the changed path as no-op, and upstream Codex native `exec_command` is environment-backed; I did not run a live Codex app-server turn.

## Solution Assessment

Unclear. Reusing `tools.exec.pathPrepend` for native Codex is plausible but crosses the current documented gateway/sandbox boundary, so maintainers should choose between an explicit Codex env/PATH contract and a documentation-only boundary.
