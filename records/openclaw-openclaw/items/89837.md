## Summary

Keep open: current main still leaves the Codex app-server PATH boundary unresolved. OpenClaw now scopes `tools.exec.pathPrepend` to gateway/sandbox exec, but it does not project that setting into Codex app-server startup or provide a clear Codex-owned PATH config contract.

## What This Changes

Make `tools.exec.pathPrepend` either affect Codex app-server native exec PATH after config reload or document the supported Codex PATH configuration path.

## Best Possible Solution

Make the Codex boundary explicit: either add a tested Codex-owned PATH/env projection with shared-client rekey or restart semantics, or document the supported service-environment path and keep `tools.exec.pathPrepend` OpenClaw-exec-only.

## Reproduction Assessment

Yes, at source level. Current main routes `tools.exec.pathPrepend` through OpenClaw exec only, returns early for a `tools.*` no-op reload, and does not feed that value into Codex app-server start env; I did not run a live app-server repro.

## Solution Assessment

Unclear. Applying the existing OpenClaw exec PATH knob to native Codex execution is plausible, but it crosses the documented gateway/sandbox boundary, so maintainers should choose the Codex plugin contract first.
