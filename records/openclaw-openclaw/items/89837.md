## Summary

Keep open: current main still does not project `tools.exec.pathPrepend` into the Codex app-server process environment or rekey/restart the shared app-server client when that value changes. The existing docs scope `pathPrepend` to OpenClaw gateway/sandbox exec, so this needs a maintainer boundary decision rather than cleanup closure.

## What This Changes

Request that `tools.exec.pathPrepend` either affect Codex app-server native shell PATH after config edits or be explicitly documented/configured as unsupported there.

## Best Possible Solution

Make Codex app-server PATH behavior explicit at the Codex plugin boundary: either add a tested Codex-owned PATH/env projection that rekeys or retires shared app-server clients, or strengthen docs to state that `tools.exec.pathPrepend` is OpenClaw exec-only.

## Reproduction Assessment

No live reproduction was run in this read-only review. Source inspection gives a high-confidence static path: current main neither feeds `tools.exec.pathPrepend` into Codex app-server start/env nor hot-applies a standalone `tools.*` no-op reload.

## Solution Assessment

Unclear. Honoring the existing `tools.exec.pathPrepend` value for Codex native exec would cross the documented gateway/sandbox boundary, so the safer direction needs maintainer choice between an explicit Codex PATH surface and a docs-only boundary clarification.
