## Summary

Keep open: current main narrows `tools.exec.pathPrepend` to gateway/sandbox exec in docs and schema help, but it still does not provide a Codex app-server native PATH contract or project that setting into Codex app-server startup/thread execution.

## What This Changes

The issue reports that `tools.exec.pathPrepend` does not affect Codex app-server native `exec_command` PATH after an OpenClaw config change.

## Best Possible Solution

Make the Codex native exec PATH boundary explicit: either add a tested Codex-owned PATH/env projection with shared-client rekey/restart semantics, or document the supported service/app-server environment path and keep `tools.exec.pathPrepend` gateway/sandbox-only.

## Reproduction Assessment

Yes, at source level. Current main documents `tools.exec.pathPrepend` as gateway/sandbox-only and does not project it into Codex app-server `start.env`, while upstream Codex native `exec_command` is environment-backed; I did not run a live Codex app-server turn.

## Solution Assessment

Unclear. Reusing `tools.exec.pathPrepend` for native Codex is plausible but crosses the current documented boundary; the safer path is for maintainers to choose whether Codex should get an explicit env/PATH contract or documented service-environment setup.
