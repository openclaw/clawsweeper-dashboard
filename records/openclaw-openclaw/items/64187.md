## Summary

Keep open. Current main and the latest inspected release still use a direct `fs.rename`-based memory-core SQLite swap that only suppresses `ENOENT`, the issue has multiple Windows reproduction reports, and open PR #71611 is the unmerged closing implementation candidate.

## What This Changes

The issue reports that Windows memory search/sync can hit `EBUSY` during the memory-core SQLite atomic reindex swap and asks for retry or recovery handling.

## Best Possible Solution

Keep this issue open until #71611, or a revised replacement, lands a memory-core atomic move helper with bounded Windows lock-error handling, careful sidecar/temp cleanup, and regression coverage for `EBUSY`, `EPERM`, and `EACCES`.

## Reproduction Assessment

Yes. The issue body and comments provide concrete Windows command sequences and logs across several versions, and current main still contains the direct rename path that would throw on those lock errors.

## Solution Assessment

Yes for the fix location, not yet for closure. The narrowest maintainable solution belongs in memory-core's atomic reindex swap helper; the open PR should be reviewed and validated rather than closing this issue before merge.
