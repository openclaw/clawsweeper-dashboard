## Summary

Keep open: the PR targets a real unbounded in-memory state risk in subagent steering, but the hard FIFO cap can evict entries that are still inside the existing 2-second throttle window and weaken the rate limit. The PR also needs observable real behavior proof before merge.

## What This Changes

The branch adds a 1024-entry cap and a 60-second stale sweep to the in-memory subagent steer rate-limit map.

## Best Possible Solution

Bound the map by pruning entries that are no longer needed for the 2-second throttle, preserve all in-window keys, and add focused coverage plus redacted runtime proof.

## Reproduction Assessment

Source-reproducible: current main writes unique controller/child keys into a module-level map and never deletes them. I did not run a live long-duration Gateway reproduction, and the PR does not include observable runtime proof.

## Solution Assessment

No: stale cleanup is the right direction, but the fixed FIFO cap can delete active in-window throttle entries. The safer fix is to remove or defer only entries older than the throttle window and preserve current rate-limit semantics.
