## Summary

Keep open: the unbounded steer rate-limit map still exists on current main and the latest release, but this PR is not merge-ready because its hard FIFO cap can evict still-active throttle entries and the external proof is prose-only.

## What This Changes

The PR adds a 1024-entry cap, a 60-second stale sweep, and FIFO eviction to the in-memory `steerRateLimit` map in `src/agents/subagent-control.ts`.

## Best Possible Solution

Bound stale limiter entries while preserving every in-window throttle key, add focused limiter coverage, and require redacted after-fix Gateway proof before merge.

## Reproduction Assessment

Yes for the source-level growth path: current main and `v2026.6.8` write unique caller/child keys into a module-level map and never delete them. I did not establish a long-duration Gateway OOM reproduction.

## Solution Assessment

No: stale cleanup is the right direction, but the fixed FIFO cap can delete active in-window throttle entries. The safer fix is to prune only entries old enough to be irrelevant to the throttle window, or otherwise prove a bounded policy that preserves active keys.
