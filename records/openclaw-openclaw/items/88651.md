## Summary

Keep open: current main and the latest release still have the unbounded `steerRateLimit` map, so the memory-bound fix is useful, but this branch is not merge-ready because the hard cap can evict active throttle keys and the proof is prose-only.

## What This Changes

This PR adds a stale sweep, a 1024-entry cap, and FIFO eviction for `steerRateLimit` in `src/agents/subagent-control.ts`.

## Best Possible Solution

Bound `steerRateLimit` with stale-only cleanup or eviction that preserves every in-window key, add focused coverage through a testable helper, and require redacted after-fix Gateway proof before merge.

## Reproduction Assessment

Yes for the source-level growth path: current main and `v2026.6.9` write unique caller/child keys into a module-level map and never delete them. I did not reproduce a long-running Gateway OOM.

## Solution Assessment

No: bounding the map is useful, but deleting the first key is not the best fix because it can weaken current rate-limit semantics. Stale-only pruning, or a cap that never removes entries younger than `STEER_RATE_LIMIT_MS`, is safer.
