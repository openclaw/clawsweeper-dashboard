## Summary

Keep this PR open: the unbounded `steerRateLimit` map is still present on current `main` and in the latest release, but this patch is not merge-ready because its hard FIFO cap can evict active throttle-window entries and the contributor proof is prose-only.

## What This Changes

The PR adds a 1024-entry cap, a 60-second stale sweep, and FIFO eviction to the in-memory subagent steer rate-limit map.

## Best Possible Solution

Bound the map by expiring stale entries while preserving every in-window throttle key, add focused limiter coverage through a testable helper, and require redacted after-fix Gateway proof before merge.

## Reproduction Assessment

Yes for the code path: current main and v2026.6.8 write unique caller/child keys into a module-level map and never delete them. I did not run a long-duration Gateway OOM reproduction.

## Solution Assessment

No: stale cleanup is the right direction, but the fixed FIFO cap can delete active in-window throttle entries. The safer fix is to prune only entries old enough to be irrelevant to the throttle window, or otherwise prove a bounded policy that preserves active keys.
