## Summary

Keep open. Current main and the latest release still route bundled MCP tool execution through SDK `Client.callTool` without `RequestOptions`, and an open PR already references this issue with closing syntax, so the issue should remain open until that implementation is reviewed and either merged or closed.

## What This Changes

Request a configurable bundled MCP tool-call timeout so long-running MCP `callTool` requests are not capped by the SDK's 60-second default.

## Best Possible Solution

Review and either land or close the linked fix PR; the final change should pass explicit MCP SDK request options through a documented config/schema contract with regression coverage and real long-running MCP proof.

## Reproduction Assessment

Yes. Source inspection shows materialized bundled MCP tools call `params.runtime.callTool`, which reaches SDK `Client.callTool` without `RequestOptions`; SDK v1.29.0 defaults omitted request timeouts to 60000 ms.

## Solution Assessment

Unclear until maintainers settle the timeout policy. Passing `RequestOptions.timeout` into `Client.callTool` is the narrow code direction, but the public config field, default duration, and progress-reset behavior still need review.
