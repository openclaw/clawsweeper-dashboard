## Summary

Keep open. Current main and the latest release still call bundled MCP tools without SDK `RequestOptions`, and open PR https://github.com/openclaw/openclaw/pull/78160 references this issue with closing syntax, so the issue should stay open until that implementation is reviewed and either landed or closed.

## What This Changes

The issue asks for bundled MCP tool calls to honor a per-server request timeout instead of inheriting the MCP SDK 60-second default.

## Best Possible Solution

Review one reconciled fix PR that documents a per-server bundled-MCP request-timeout contract, passes correct SDK request options, adds regression coverage, and includes real long-running MCP proof.

## Reproduction Assessment

Yes. Source inspection shows materialized bundled MCP tools reach `Client.callTool` without `RequestOptions.timeout`, and SDK v1.29.0 defaults omitted request timeouts to 60000 ms; I did not run a live slow MCP server.

## Solution Assessment

Unclear until maintainers settle the public timeout policy. Passing `RequestOptions.timeout` into `Client.callTool` is the narrow code path, but the field name, default duration, and progress-reset semantics need review.
