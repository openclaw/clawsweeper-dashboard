## Summary

Keep open. Current main and the latest tagged release still route bundled MCP tool execution through SDK `Client.callTool` without `RequestOptions`, while an open linked PR already claims this issue with closing syntax and should remain the active fix vehicle.

## What This Changes

Allow bundled MCP tool executions to avoid the SDK's 60-second default request timeout by plumbing explicit per-request timeout options from bundle MCP configuration into `callTool`.

## Best Possible Solution

Review and either land or close the linked fix PR; the final change should pass explicit MCP SDK request options from a documented config contract with schema, docs, regression coverage, and real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: bundled MCP tools execute through `params.runtime.callTool`, which reaches SDK `Client.callTool` without `RequestOptions`, and SDK v1.29.0 defaults omitted request timeouts to 60000 ms.

## Solution Assessment

Unclear until the timeout policy is accepted. Passing a per-server request timeout into SDK `RequestOptions` is the narrow code direction, but the public config field, default duration, and progress-reset behavior still need maintainer review.
