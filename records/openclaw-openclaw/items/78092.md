## Summary

Keep open. Current main still invokes bundle MCP `callTool` without SDK `RequestOptions`, and SDK 1.29.0 falls back to a 60s request timeout when options are omitted; resolving this needs a maintainer decision on the new timeout field/default/progress policy.

## What This Changes

Request configurable or longer bundle MCP tool request timeouts so valid long-running `callTool` executions do not fail at the SDK default.

## Best Possible Solution

Add an explicit bundle MCP tool request-timeout policy, likely per-server `requestTimeoutMs`, then plumb it to SDK `RequestOptions` with schema, docs, changelog, and regression coverage once the default/progress behavior is settled.

## Reproduction Assessment

Yes, from source inspection: a bundle MCP tool execution reaches `Client.callTool` with no options, and SDK 1.29.0 applies a 60s default timeout when `timeout` is omitted. I did not run a live 60s+ wall-clock reproduction in this read-only sweep.

## Solution Assessment

Unclear until maintainers choose the field name, default, and progress-reset semantics. The narrow maintainable direction is to add a per-server request timeout and pass it as SDK `RequestOptions` from the bundle MCP runtime.
