## Summary

Keep open. Current main still has the source-level failure cluster: an installable Brave web_search provider can remain unavailable and fatal, last-known-good recovery can restore the same invalid provider config path, and `gateway stop` is still blocked by invalid config preflight.

## What This Changes

Fix Brave web_search selection and Gateway recovery so installing/enabling Brave cannot leave the service in a crash loop with no CLI stop escape.

## Best Possible Solution

Keep typo validation strict, but make provider-owned startup failure recoverable: prevent same-config last-known-good restore loops, make service stop an unconditional invalid-config escape path, and verify/fix Brave package registration if the published artifact is the trigger.

## Reproduction Assessment

Yes, source-level. The issue body provides concrete logs, and current main still contains the fatal provider validation branch, recovery eligibility for the `tools.web.search.provider` path, and stop preflight that blocks on invalid config; I did not run the live Gateway loop.

## Solution Assessment

No current main solution is present. The narrowest maintainable fix is a generic recovery/CLI repair that preserves strict unknown-provider validation while preventing unavailable plugin-owned providers from trapping the Gateway.
