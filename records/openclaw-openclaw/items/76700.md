## Summary

Keep open. Current main still has the source-level failure cluster described in the report: `brave` is documented as a supported web search provider, an unavailable install-catalog provider remains a fatal config validation issue, last-known-good recovery can still copy the promoted backup back over the active config without a same-hash guard, and `gateway stop` still preflights invalid config before stopping.

## What This Changes

The issue asks OpenClaw to prevent a Brave web search plugin install/config from crash-looping the Gateway and to keep `gateway stop` available as a recovery path.

## Best Possible Solution

Keep this open until the Brave plugin install/register path, same-config last-known-good recovery guard, and unconditional `gateway stop` recovery path are fixed with focused regression coverage.

## Reproduction Assessment

Yes, source-level. A config with `tools.web.search.provider: "brave"` and a Brave plugin that fails to become active still reaches the fatal validation branch, remains eligible for whole-file recovery, and can block `gateway stop` through the lifecycle preflight.

## Solution Assessment

No current solution is present on main. The narrow maintainable fix is to make the installed Brave provider reliably register from the published runtime, prevent restoring the same failed config, and let stop bypass config validation while keeping typo validation strict.
