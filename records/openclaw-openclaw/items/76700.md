## Summary

Keep open. The report describes a real current-main failure cluster: an unavailable installable `web_search` provider is still a fatal config issue, whole-file last-known-good recovery still applies to that non-plugin-entry path, and `gateway stop` still preflights invalid config before stopping the service.

## What This Changes

The issue reports that installing the Brave plugin and setting `tools.web.search.provider` to `brave` on 2026.5.2 can put the gateway into a tight restart loop with no CLI stop escape path.

## Best Possible Solution

Fix the startup/recovery escape path generically for plugin-owned web-search providers, then confirm whether Brave’s published package still needs a separate registration/package artifact fix.

## Reproduction Assessment

Yes, source-level. The issue body provides concrete logs and current main still has the fatal provider validation branch, last-known-good recovery eligibility for that path, and stop preflight that blocks on invalid config; I did not run the live gateway loop.

## Solution Assessment

No current solution is present on main. The narrowest maintainable fix is to make plugin-owned provider unavailability degrade without whole-file restore loops, keep typo validation strict, and let service stop remain an emergency escape path.
