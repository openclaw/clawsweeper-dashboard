## Summary

Keep open. Current main partially addresses the restore-loop class by failing startup/hot reload closed instead of auto-restoring invalid config, and it has publish-time safeguards for compiled plugin runtimes, but the reported recovery escape is still broken because `gateway stop` still blocks on invalid config and `tools.web.search.provider: "brave"` remains fatal when the installable provider is not active.

## What This Changes

The issue reports a 2026.5.2 Brave web_search install/config crash loop caused by unavailable provider validation, last-known-good self-restore, and `gateway stop` being blocked by config validation.

## Best Possible Solution

Finish the recovery path by letting service stop bypass invalid-config validation, keep startup/hot reload fail-closed, ship a compiled Brave runtime package, and make an explicit maintainer decision on whether unavailable optional web_search providers should degrade or stay strict.

## Reproduction Assessment

Yes, source-level. Current main still has a deterministic path where inactive `provider: "brave"` fails validation and `openclaw gateway stop` returns before `service.stop` when that invalid snapshot exists; I did not establish the full launchd restart loop live.

## Solution Assessment

No current complete solution is present. Main has removed startup/hot-reload last-known-good auto-restore and added plugin runtime packaging safeguards, but the narrow recovery escape and the plugin-provider degradation policy still need follow-up.
