## Summary

Keep open: current main already surfaces exact node approval commands in `openclaw nodes status` and `openclaw nodes describe`, but Android recovery/settings and the device-oriented CLI path still do not bridge the visible phone/IP state to `openclaw nodes approve <requestId>`.

## What This Changes

The issue requests gateway/device diagnostics that expose pending node approval request IDs and the exact `openclaw nodes approve <id>` command during Android setup recovery.

## Best Possible Solution

Add a bounded diagnostics bridge that reuses existing `node.list`/`pendingRequestId`, display name, and IP data to show `openclaw nodes approve <requestId>` in Android recovery/settings and in device-oriented CLI guidance when a visible phone/IP maps to a pending node approval.

## Reproduction Assessment

Yes, at source level: current Android recovery and settings paths consume node approval state but do not render the pending node request ID or exact approve command. I did not run a live Android device session in this read-only review.

## Solution Assessment

Yes, the best fix is an additive diagnostics mapper using existing gateway `node.list` data, not a new approval layer, config flag, or separate persistence path.
