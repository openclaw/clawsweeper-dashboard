## Summary

Keep open: current main still has the reported device-oriented and Android discoverability gap, and the direct closing PR is open rather than merged. ## What This Changes

The issue asks gateway, Android, and device-oriented CLI diagnostics to show the pending node approval request ID and exact `openclaw nodes approve <id>` command when a phone/node is waiting for approval or reapproval.

## Best Possible Solution

Land a sanitized, connection-aware diagnostics bridge from existing `node.list`/pending node data into the device CLI and Android recovery surfaces, while splitting any intentionally deferred gateway-export work into a narrower follow-up.

## Reproduction Assessment

Yes, source-level. Current main exposes pending node request IDs through `node.list` and `openclaw nodes status/describe`, while `openclaw devices list`, failed explicit `devices approve`, and Android onboarding still do not bridge from the visible phone/IP state to the exact `openclaw nodes approve <requestId>` command.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: reuse existing pending-node diagnostics and render safer guidance on the surfaces users naturally try, rather than approving by IP or changing node/device approval semantics.
