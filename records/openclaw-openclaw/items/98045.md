## Summary

Keep open: current main already shows the exact `openclaw nodes approve <requestId>` command in `nodes status` and `nodes describe`, but the reported device-oriented path still does not guide users from a visible phone/IP or wrong-layer `devices approve` attempt to the pending node approval ID.

## What This Changes

The issue requests clearer gateway diagnostics and CLI guidance that expose pending node approval UUIDs and the exact `openclaw nodes approve <id>` command when a device/node is awaiting reapproval.

## Best Possible Solution

Add a narrow diagnostic bridge from device-visible rows or wrong-layer device approval attempts to the existing pending node approval row and exact `openclaw nodes approve <requestId>` command, without merging the device and node approval stores.

## Reproduction Assessment

Yes from source for the CLI gap: `nodes status` has the command hint, while `devices list` and explicit `devices approve` do not consult node pairing state. I did not run a live Android gateway repro in this read-only review.

## Solution Assessment

Yes for the likely direction: reuse existing `node.list`/`node.pair.list` data and CLI command formatting so device-oriented diagnostics can guide users without changing approval semantics.
