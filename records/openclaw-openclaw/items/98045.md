## Summary

Keep open: current main already exposes exact node approval commands in `openclaw nodes status/describe`, but the reported Android and device-oriented CLI surfaces still do not bridge from the visible phone/IP or paired device row to `openclaw nodes approve <requestId>`.

## What This Changes

The issue requests clearer diagnostics that expose pending node approval/reapproval request IDs and the exact `openclaw nodes approve <id>` command on Android and device-oriented CLI surfaces.

## Best Possible Solution

Keep this open and add bounded, additive guidance on the Android and `devices` CLI surfaces using existing `node.list` pending approval data, without changing approval semantics or adding config.

## Reproduction Assessment

Yes, at source level: Android and `devices` CLI code paths have the node pending request data or visible device/IP context but do not render the exact node approve command where the issue says users looked.

## Solution Assessment

Yes: the best fix is to reuse existing `node.list` fields on the user-facing Android/device surfaces, not to add a second approval mechanism, approve by IP, or alter token semantics.
