## Summary

Keep open: current main has implemented iMessage typing indicators and read receipts through imsg, but automatic tapback acknowledgment and the canonical ack-reaction/capability config contract are still unresolved.

## What This Changes

The issue asks OpenClaw to wire the iMessage runtime to imsg typing indicators, tapback acknowledgements, read receipts, and a documented opt-in config/capability surface.

## Best Possible Solution

Keep the current imsg typing/read implementation, then either add iMessage automatic ack tapbacks through the shared ackReaction/ackReactionScope contract or explicitly document that iMessage read receipts are the supported acknowledgement path.

## Reproduction Assessment

Yes at source level for the remaining gap: current main calls imsg for read receipts and typing, but the accepted inbound path never sends an automatic tapback acknowledgement. I did not run a live macOS imsg bridge reproduction.

## Solution Assessment

Unclear. The maintainable fix depends on whether maintainers want iMessage to use the existing shared ackReaction/ackReactionScope contract, an iMessage-specific actions toggle, or read receipts only with docs corrected.
