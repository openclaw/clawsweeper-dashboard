## Summary

Keep open: current main has already wired iMessage typing indicators and read receipts through imsg RPC, and it has an action-level tapback path, but it still lacks automatic tapback acknowledgement on inbound receipt and a clear iMessage ack-reaction config/capability contract.

## What This Changes

The issue requests wiring the iMessage runtime to use imsg bridge features for typing indicators, tapback acknowledgements, read receipts, and the relevant opt-in/config surface.

## Best Possible Solution

Keep the current imsg typing/read implementation, then have maintainers choose whether iMessage should honor the shared `messages.ackReaction`/`ackReactionScope` contract or explicitly document read receipts as the supported acknowledgement mechanism.

## Reproduction Assessment

Yes for the remaining gap from source inspection: current main calls imsg for typing/read and exposes action-level tapbacks, but the accepted inbound monitor path never sends an automatic tapback acknowledgement. I did not run a live Messages.app/imsg bridge repro.

## Solution Assessment

Unclear. Wiring tapbacks is technically feasible through existing iMessage reaction code, but maintainers should first decide the canonical config surface and whether automatic tapbacks are desired versus read receipts only.
