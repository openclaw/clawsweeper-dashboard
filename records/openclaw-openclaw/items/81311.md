## Summary

Keep open: current main implements iMessage typing indicators, read receipts, and inbound tapback-as-reaction events, but the requested automatic tapback acknowledgement path and canonical iMessage ackReaction/ackReactionScope config contract are still unresolved.

## What This Changes

The issue asks to wire the iMessage channel runtime to imsg private-API typing indicators, automatic tapback/read acknowledgements, and documented capability or config controls.

## Best Possible Solution

Keep the shipped typing/read implementation, then either add iMessage automatic ack tapbacks through the shared ackReaction/ackReactionScope contract with schema/docs/tests, or explicitly document read receipts as the supported iMessage acknowledgement path.

## Reproduction Assessment

Yes at source level. Current main calls imsg for typing/read and handles inbound tapbacks as notifications, but there is no automatic inbound ack tapback path or iMessage ackReaction/ackReactionScope schema surface.

## Solution Assessment

Unclear until maintainers choose the public behavior. The safest path is either a shared ackReaction-compatible iMessage implementation or docs that make read receipts the canonical acknowledgement and scope tapback acks out.
