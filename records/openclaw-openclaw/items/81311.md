## Summary

Keep open: current main has iMessage typing indicators and read receipts wired through the imsg RPC bridge, but automatic tapback acknowledgement is still not wired and the right config semantics need maintainer judgment.

## What This Changes

Wire the iMessage channel runtime to use imsg typing, read receipts, and optional tapback acknowledgement/capability config during inbound agent turns.

## Best Possible Solution

Keep the current typing/read implementation and add iMessage ack tapbacks through a maintainer-approved shared `ackReaction` path with valid tapback mapping, or document that iMessage does not support generic ack reactions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection for the remaining gap: current main calls imsg for typing/read, but the iMessage inbound monitor never sends an automatic tapback ack.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the canonical iMessage ack-reaction config semantics and tapback default/mapping; using the shared ack-reaction helpers looks like the likely direction.
