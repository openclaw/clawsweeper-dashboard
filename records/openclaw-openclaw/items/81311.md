## Summary

Keep open: current main covers iMessage typing indicators, read receipts, and inbound tapback notifications, but automatic iMessage ack tapbacks are still absent and the config contract needs maintainer direction.

## What This Changes

The issue requests wiring the iMessage imsg bridge into runtime typing indicators, tapback acknowledgements, read receipts, and a documented capability/config opt-in.

## Best Possible Solution

Keep the shipped typing/read and inbound tapback event behavior, then either implement automatic iMessage ack tapbacks through the documented shared ack-reaction contract or explicitly document that iMessage uses read receipts instead of tapback acks.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the remaining gap by source inspection: current main calls imsg for typing/read and routes inbound tapbacks as system events, but no normal inbound path creates an automatic iMessage ack tapback.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the canonical config semantics; the narrow implementation should reuse shared ack-reaction behavior if iMessage tapback acks are intended, or document read receipts as the supported acknowledgement path.
