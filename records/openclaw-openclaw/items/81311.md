## Summary

Keep open: current main covers imsg typing indicators and read receipts, but automatic iMessage tapback acknowledgements and the canonical ack/capability config contract remain unresolved.

## What This Changes

The issue asks OpenClaw to wire iMessage typing indicators, tapback acknowledgements, read receipts, and capability/config toggles from the imsg bridge into the iMessage channel runtime.

## Best Possible Solution

Keep the shipped typing/read implementation, then either add iMessage automatic ack tapbacks through the shared ackReaction contract or explicitly document read receipts as the supported iMessage acknowledgement path.

## Reproduction Assessment

Yes at source level for the remaining gap: current main calls imsg for read receipts and typing, but the accepted inbound path has no automatic ack tapback wiring and the strict iMessage schema has no ackReaction/ackReactionScope fields. I did not run a live macOS imsg bridge reproduction.

## Solution Assessment

Unclear. Typing/read are already solved cleanly, but the remaining tapback acknowledgement needs a maintainer decision on whether to use the shared ackReaction contract, an iMessage-specific toggle, or docs that make read receipts the canonical acknowledgement.
