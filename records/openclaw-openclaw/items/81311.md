## Summary

Keep open: current main now covers the typing indicator and read-receipt parts, but automatic iMessage tapback acknowledgement is still absent and the config contract for enabling it is unresolved.

## What This Changes

The issue requests iMessage runtime wiring for imsg typing indicators, read receipts, optional tapback acknowledgements, and documented capability/config handling.

## Best Possible Solution

Keep the current typing/read implementation and add or explicitly document iMessage automatic ack tapbacks through the maintainer-chosen shared or iMessage-specific config contract.

## Reproduction Assessment

Yes for the remaining gap from source inspection: current main marks chats read and sends typing callbacks, but the iMessage monitor never calls an ack tapback path on inbound receipt. I did not run a live macOS/imsg device repro.

## Solution Assessment

Unclear until maintainers choose the config semantics for iMessage ack tapbacks. Typing/read no longer need a new fix on main; the remaining implementation should use the chosen ack-reaction contract rather than inventing an undocumented toggle.
