## Summary

Keep open: current main covers iMessage typing indicators, read receipts, manual tapback actions, and inbound tapback notifications, but automatic tapback acknowledgement on receipt and the canonical iMessage ack/capability config contract remain unresolved.

## What This Changes

The issue asks to wire the iMessage runtime to use imsg typing, tapback acknowledgement, read receipt, and capability/config toggles during normal inbound-to-agent turns.

## Best Possible Solution

Keep the current typing/read implementation, then either add iMessage automatic ack tapbacks through a documented shared ackReaction contract or document read receipts as the supported iMessage acknowledgement path.

## Reproduction Assessment

Yes, at source level: current main marks chats read and starts/stops typing through imsg, while the accepted inbound path has no automatic ack-reaction handle and the strict iMessage schema has no `ackReaction` or `ackReactionScope` keys. I did not run a live macOS imsg bridge reproduction in this read-only review.

## Solution Assessment

Unclear. Typing and read receipts are already solved cleanly, but automatic tapback acknowledgements need a maintainer decision on whether to use the shared ackReaction contract, an iMessage-specific toggle, or docs that make read receipts the canonical acknowledgement.
