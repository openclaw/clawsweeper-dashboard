## Summary

Keep open: current main covers iMessage typing indicators and read receipts through imsg, but automatic tapback acknowledgement and the canonical capability/config contract remain unresolved.

## What This Changes

The issue asks OpenClaw to wire imsg typing, tapback acknowledgements, read receipts, and capability/config toggles into the iMessage channel runtime.

## Best Possible Solution

Keep the current imsg typing/read implementation, then either add iMessage automatic ack tapbacks through the shared ack-reaction contract or explicitly document that iMessage read receipts are the supported acknowledgement path.

## Reproduction Assessment

Yes at source level: current main calls imsg for read receipts and typing, while targeted source search shows no automatic iMessage ack-reaction wiring in the accepted inbound path. I did not run a live macOS/imsg bridge reproduction.

## Solution Assessment

Unclear. The typing/read portion is implemented cleanly, but the remaining tapback acknowledgement should wait for a maintainer decision on whether iMessage uses the shared ack-reaction contract, an iMessage-specific toggle, or docs-only scope clarification.
