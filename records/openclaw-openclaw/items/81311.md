## Summary

Keep open: current main has the imsg private-API typing and read-receipt runtime wiring, plus manual tapback actions and inbound tapback notifications, but it still lacks automatic inbound tapback acknowledgement wiring and a settled iMessage ack/capability config contract.

## What This Changes

Request to wire iMessage imsg private-API typing, tapback acknowledgement, read receipts, and capability/config toggles into the channel runtime.

## Best Possible Solution

Keep the shipped typing/read implementation, then either add iMessage automatic ack tapbacks through the shared ackReaction/ackReactionScope contract with schema/docs/tests or explicitly document read receipts as the supported iMessage acknowledgement path.

## Reproduction Assessment

Yes at source level for the remaining gap: current main calls imsg for read receipts and typing, but the accepted inbound path has no automatic ack tapback wiring and the strict iMessage schema has no ackReaction/ackReactionScope fields. I did not run a live macOS imsg bridge reproduction.

## Solution Assessment

Unclear. Typing/read are already solved cleanly, but the remaining tapback acknowledgement needs maintainer direction on shared ackReaction semantics, an iMessage-specific toggle, or docs that make read receipts the canonical acknowledgement.
