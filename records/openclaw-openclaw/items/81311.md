## Summary

Keep open: current main and the latest checked release cover iMessage typing indicators and read receipts, but automatic tapback acknowledgements and the canonical iMessage ack/capability config contract are still unresolved and need maintainer product/API judgment.

## What This Changes

The issue asks to wire imsg-backed typing indicators, tapback acknowledgements, read receipts, and capability/config toggles into the iMessage channel runtime.

## Best Possible Solution

Keep the shipped typing/read implementation, then either add automatic iMessage ack tapbacks through a documented shared ackReaction contract or explicitly document read receipts as the supported iMessage acknowledgement path.

## Reproduction Assessment

Yes at source level for the remaining gap: current main has imsg typing/read calls but no automatic iMessage ack tapback wiring or strict-schema ackReaction/ackReactionScope fields. I did not run a live macOS imsg bridge reproduction.

## Solution Assessment

Unclear. Typing/read are already solved cleanly, but the remaining tapback acknowledgement needs a maintainer decision on shared ackReaction semantics, an iMessage-specific toggle, or docs that make read receipts the canonical acknowledgement.
