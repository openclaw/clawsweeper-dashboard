## Summary

Keep open: current main covers imsg typing indicators and read receipts, and tapbacks exist as manual/message-tool actions plus inbound reaction events, but automatic iMessage tapback acknowledgements and the canonical ack/config semantics remain unresolved.

## What This Changes

The issue requests wiring iMessage imsg typing, tapback acknowledgements, read receipts, and capability/config handling into the channel runtime.

## Best Possible Solution

Keep the shipped typing/read implementation, then either add automatic iMessage acknowledgements through a documented `ackReaction`/`ackReactionScope` contract with schema/docs/tests, or explicitly document read receipts as the supported iMessage acknowledgement path.

## Reproduction Assessment

Yes, at source level for the remaining gap: current main calls imsg for read receipts and typing, but accepted inbound iMessage dispatch has no automatic ack tapback handle and the strict schema has no `ackReaction` or `ackReactionScope` fields. I did not run a live macOS bridge reproduction.

## Solution Assessment

Unclear. Typing and read receipts are already solved cleanly, but automatic tapback acknowledgements should either align with the shared ackReaction contract or be deliberately scoped out in docs, and that is a product/API decision.
