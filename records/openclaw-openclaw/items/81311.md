## Summary

Keep open: current main now covers iMessage typing indicators, read receipts, and inbound tapback reaction events, but automatic tapback acknowledgements and the canonical iMessage acknowledgement config contract remain unresolved.

## What This Changes

Request to wire imsg private-API typing, read receipts, tapback acknowledgements, and capability/config toggles into the iMessage channel runtime.

## Best Possible Solution

Keep the shipped typing/read and inbound tapback event work, then decide whether iMessage should implement the shared ackReaction contract for automatic tapbacks or document read receipts as the supported acknowledgement path.

## Reproduction Assessment

Yes at source level: current main calls imsg for typing/read and routes inbound tapback events, but the accepted inbound path lacks automatic tapback ack wiring and the strict iMessage schema lacks ackReaction/ackReactionScope or actions.ackReactions. I did not run a live macOS bridge reproduction.

## Solution Assessment

Unclear. Typing and read receipts are already solved cleanly, but the remaining acknowledgement behavior needs a maintainer decision between shared ackReaction semantics, an iMessage-specific toggle, or docs that make read receipts canonical.
