## Summary

Keep open: current main and the latest release cover iMessage typing indicators, read receipts, and inbound tapback events, but they still do not implement automatic iMessage tapback acknowledgements or a settled ack config contract.

## What This Changes

Request to wire the iMessage runtime to imsg typing, read receipts, and tapback acknowledgements with a canonical opt-in configuration.

## Best Possible Solution

Keep the shipped read/typing and inbound tapback event behavior, then decide whether iMessage should send automatic ack tapbacks through the shared ackReaction contract or document read receipts as the supported acknowledgement path.

## Reproduction Assessment

Yes, source-level: current main has concrete typing/read calls and inbound/manual tapback support, while the monitor and strict iMessage schema still lack an automatic ack-reaction path. I did not run a live macOS imsg bridge reproduction.

## Solution Assessment

Unclear. Typing and read receipts are already solved cleanly, but automatic tapback acknowledgements need a maintainer decision on shared ackReaction semantics versus iMessage-specific or docs-only behavior.
