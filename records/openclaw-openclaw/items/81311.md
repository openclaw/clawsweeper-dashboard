## Summary

Keep open: current main now wires iMessage typing indicators and read receipts through the imsg bridge, but automatic tapback acknowledgement and the canonical config surface for it are still missing.

## What This Changes

Request to wire the iMessage channel runtime to use imsg for typing indicators, tapback acknowledgements, read receipts, and documented capability/config toggles.

## Best Possible Solution

Keep the current typing/read implementation and add iMessage receipt-ack tapbacks through one documented config contract, with regression coverage for direct/group gating and removal-after-reply behavior if supported.

## Reproduction Assessment

Yes for the remaining source-level gap: current main calls imsg for typing/read but has no automatic iMessage ack-reaction path on accepted inbound receipt. A full user-visible reproduction would still need a macOS Messages host with the imsg private API bridge.

## Solution Assessment

Unclear until maintainers choose the config contract. Reusing the existing shared ack-reaction semantics would likely be cleaner than adding an unrelated `capabilities` toggle, but that is a product/API decision.
