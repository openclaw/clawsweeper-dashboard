## Summary

Keep this open: current main and the latest release still lack a non-interactive audit-to-plan secrets migration command, and the existing merged `config set` work only covers explicit SecretRef/provider assignment. The request is valid feature work, but automated credential rewrites are security-sensitive and need maintainer product direction before implementation.

## What This Changes

Add a non-interactive `openclaw secrets migrate --strategy env --auto --yes` and dry-run flow that scans plaintext credentials, generates SecretRefs, validates them, and applies without a TTY.

## Best Possible Solution

Define a narrow first-party secrets migration contract, then implement it by reusing audit discovery and `secrets apply` preflight/write semantics with explicit dry-run, confirmation, safety docs, and focused tests.
