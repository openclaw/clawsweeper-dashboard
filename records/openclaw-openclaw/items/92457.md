## Summary

Keep open. Current main still has no non-interactive `openclaw secrets migrate --auto` or `secrets apply --auto` path; the related closed work added non-interactive `config set` SecretRef/provider builders for `openclaw.json`, but it does not scan plaintext findings, generate a batch migration plan, or handle the full configure/apply workflow requested here.

## What This Changes

Request for a non-interactive secrets migration command that scans supported plaintext credentials, generates SecretRefs, validates them, and applies the migration for automation scripts.

## Best Possible Solution

Define a maintainer-approved secrets migration mode that reuses audit/target-registry/apply-plan plumbing, supports dry-run and explicit confirmation, validates refs before writing, documents env/file output behavior, and keeps plaintext scrubbing one-way and auditable.

## Reproduction Assessment

No. This is a feature request for a command/mode that current main does not expose, so there is no bug reproduction path beyond source-level confirmation that the command is absent.

## Solution Assessment

Unclear as proposed. A batch migration command is plausible, but the best implementation should reuse the existing audit, target-registry, and apply-plan contracts and needs maintainer agreement on the SecretRef naming and env/file output policy.
