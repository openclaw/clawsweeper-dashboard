## Summary

Keep open: current main and v2026.6.6 still only provide audit, interactive configure, and apply-from-plan secrets flows; the related merged config-set work does not generate or apply audit-derived SecretRefs, so this remains a new security-sensitive CLI feature needing maintainer direction.

## What This Changes

Add a non-interactive `openclaw secrets migrate --auto` workflow that scans plaintext credentials, generates SecretRefs, validates them, and applies changes atomically for automation scripts.

## Best Possible Solution

Design a narrow first-party secrets migration workflow that reuses audit target discovery and `secrets apply` preflight/rollback, with explicit dry-run, confirmation, strategy/category semantics, env-name derivation, Windows behavior, and security review.

## Reproduction Assessment

Not applicable. The reporter clarified this is a feature request, and source/docs inspection confirms the proposed non-interactive `secrets migrate --auto` command does not exist on current main.

## Solution Assessment

Unclear until maintainers settle the contract. A non-interactive migration command is plausible, but the best implementation needs explicit decisions around strategy/category flags, env naming, Windows output, exec consent, and one-way scrubbing safeguards.
