## Summary

Keep open. Current main and the latest release still make Feishu tool-progress status transient: tool status can appear during live streaming, but final streaming-card close clears it, and the current regression test asserts the final card contains only the answer. This remains a real Feishu message-loss/product decision rather than an implemented or duplicate-only cleanup item.

## What This Changes

The issue requests restoring or making configurable Feishu final-card retention of tool-call progress that disappeared after recent delivery changes.

## Best Possible Solution

Maintainers should choose the Feishu final-card semantics first, then either document clean-final behavior as intentional or add a Feishu-owned opt-in progress transcript with config, privacy, streaming, final-card, docs, and live Feishu proof.

## Reproduction Assessment

Yes, at source level: Feishu onToolStart writes a live statusLine, closeStreaming clears that statusLine before final close, and the current test asserts the final card omits the tool line. I did not run a live credentialed Feishu card reproduction.

## Solution Assessment

Unclear until maintainers choose the UX and privacy policy. If this behavior is accepted, a Feishu-scoped opt-in is safer than an unconditional hard revert because tool progress may include durable diagnostic or command detail.
