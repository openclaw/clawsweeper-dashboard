## Summary

Keep open: this recent external PR still has a concrete P1 security finding and no real behavior proof. Current main has already covered much of the narrow Matrix/registry CI surface differently, but the branch still carries Google auth/CLI behavior changes that need contributor proof and maintainer/security review before merge.

## What This Changes

The branch mixes Google fake-IP OAuth/Gemini CLI behavior and docs with Matrix, plugin-registry, and config-boundary test fixes for CI/type regressions.

## Best Possible Solution

Keep the valid CI-contract fixes already represented on main, and only merge a narrow Google fake-IP OAuth change that uses RFC2544/ULA-scoped policy with redacted real runtime proof.

## Reproduction Assessment

Yes, source-reproducible: with the PR policy, a lookup for an allowed Google hostname returning a private/internal address would bypass resolved-address checks because `allowPrivateNetwork` sets the skip flag. I did not run a live fake-DNS OAuth setup.

## Solution Assessment

No: the CI portions may be valid or already covered on main, but the Google fake-IP fix uses a broader private-network opt-in than the existing narrow SSRF policy pattern. The safer solution is a host-scoped RFC2544/ULA fake-IP policy plus real runtime proof.
