## Summary

Keep open: current main still has the raw reply-operation abort path, so the PR is not obsolete, but the latest helper can leave a polling interval alive after successful timeout-filtered compaction and the PR still has only unit/Vitest proof rather than real slow-preflight behavior proof.

## What This Changes

The PR adds a derived preflight compaction abort signal that filters reply lifecycle TimeoutError while polling replyOperation.result for explicit aborts, plus focused unit tests.

## Best Possible Solution

Use a lifecycle-owned preflight cancellation source that ignores normal reply lifecycle timeout, still propagates explicit stop/restart cancellation, cleans up any polling when compaction returns, and is backed by real slow-preflight proof.

## Reproduction Assessment

Yes at source level: current main passes the reply-operation abort signal into required budget/preflight compaction, and the PR head's TimeoutError-filtered path can be traced to an uncleared polling interval. I did not run the reporter's slow vLLM/Qwen setup.

## Solution Assessment

No. The call-site direction is plausible and the prior missed-cancellation finding is partly addressed, but the helper needs lifecycle cleanup and real slow-preflight proof before it is the best fix.
