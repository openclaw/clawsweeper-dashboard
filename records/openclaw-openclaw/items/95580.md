## Summary

Keep open: this PR targets a real preflight compaction timeout bug, but the current patch is not merge-ready because it removes all caller abort propagation and can leave user-aborted or restart-draining compactions running until the compaction timeout.

## What This Changes

The PR removes the `abortSignal` argument from budget/preflight `compactEmbeddedAgentSession` calls and adds a unit test asserting no abort signal is passed.

## Best Possible Solution

Split the preflight cancellation contract so normal reply lifecycle timeout cannot cap compaction, while explicit user abort, restart, shutdown, and gateway drain still cancel the in-flight compaction through a deliberate signal.

## Reproduction Assessment

Yes at source level: current main and v2026.6.9 pass `replyOperation.abortSignal` into required budget/preflight compaction, and `compactWithSafetyTimeout` races external abort against the configured timeout. I did not live-reproduce the reporter's exact slow vLLM/Qwen timing in this read-only review.

## Solution Assessment

No. The current diff fixes the timeout race by dropping all caller abort propagation, but the maintainable fix should preserve explicit stop/restart/shutdown cancellation while excluding the normal reply lifecycle timeout.
