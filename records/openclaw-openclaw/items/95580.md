## Label Justifications

- `P1`: The PR targets a real large-session compaction bug that can repeatedly stall agent turns, and the current patch introduces a restart/user-cancellation regression.
- `merge-risk: 🚨 session-state`: The diff changes required preflight compaction before session context is rotated and reused, so wrong cancellation semantics can leave session state locked or near overflow.
- `merge-risk: 🚨 availability`: Removing caller abort propagation can let restart-draining or user-aborted preflight compactions continue until the compaction timeout.

## Summary

Keep open: the latest push removes the preflight timeout race, but it also drops explicit user/restart cancellation for required preflight compaction while real behavior proof is still unit-test only.

## What This Changes

The PR removes the preflight compaction call's `replyOperation.abortSignal` argument and adds a unit test asserting no abort signal is passed.

## Best Possible Solution

Split the preflight cancellation contract so the normal reply lifecycle timeout cannot cap compaction, while explicit user abort, restart, shutdown, and gateway drain still cancel the in-flight compaction through a deliberate signal.

## Reproduction Assessment

Yes, source-level: current main and `v2026.6.9` pass `replyOperation.abortSignal` to required budget compaction, and `compactWithSafetyTimeout` races external abort against the configured timeout. I did not run the reporter's slow vLLM scenario in this read-only review.

## Solution Assessment

No. Removing the caller abort fixes the reply-timeout race, but it is not the best fix because it drops the explicit user/restart cancellation path; the maintainable shape is a cancellation-only signal or equivalent split contract.
