## Summary

Keep open: this PR targets a valid P1 preflight compaction bug, but the patch is not merge-ready because it still races against the reply-operation abort and adds a second timeout owner.

## What This Changes

The PR changes preflight compaction to pass `AbortSignal.any([replyOperation.abortSignal, AbortSignal.timeout(resolveCompactionTimeoutMs(cfg))])` and adds a unit test that the passed signal is not the raw reply-operation signal.

## Best Possible Solution

Define a preflight compaction cancellation signal that excludes the reply lifecycle timeout, keep `compactWithSafetyTimeout` as the timeout owner, and add focused regression plus real slow-compaction proof.

## Reproduction Assessment

Yes at source level: current main passes `replyOperation.abortSignal` into required budget/preflight compaction, and the safety wrapper races that external abort against the configured timeout. I did not live-run the reporter's slow-provider scenario.

## Solution Assessment

No. The PR changes signal identity, but it preserves the same early abort source inside `AbortSignal.any` and duplicates the configured timeout outside the existing timeout owner.
