## Summary

Keep open: the PR targets a real source-reproducible preflight compaction timeout bug, but the current patch can leave a helper-owned polling interval alive after successful compaction and still lacks real slow-preflight proof.

## What This Changes

The branch changes required budget/preflight compaction to pass a derived cancellation-only signal instead of the raw reply-operation abort signal, with unit tests for TimeoutError filtering and explicit abort propagation.

## Best Possible Solution

Adopt one lifecycle-owned preflight cancellation contract that ignores normal reply lifecycle timeout, still propagates explicit user abort and restart cancellation, cleans up all polling on compaction settlement, and is backed by real slow-preflight proof.

## Reproduction Assessment

Yes at source level: current main and v2026.6.9 pass the reply-operation abort signal into required budget/preflight compaction, and the wrapper races that external abort before the configured compaction timeout. I did not live-reproduce the reporter's slow vLLM/Qwen deployment.

## Solution Assessment

No. The call-site direction is plausible, but the helper needs lifecycle cleanup and real slow-preflight proof before this is the best fix; maintainers also need to choose one canonical cancellation contract across the open candidate PRs.
