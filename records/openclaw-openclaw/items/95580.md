## Label Justifications

- `P1`: The PR targets a real large-session compaction regression and the current patch can still break stop/restart cancellation for long preflight compactions.
- `merge-risk: 🚨 session-state`: The diff changes required preflight compaction while session context is near overflow and can keep session state locked or stale longer than intended.

## Summary

Keep open: the underlying preflight compaction bug is real, but this PR is not merge-ready because it filters a one-shot reply abort signal and can lose later user-stop or restart cancellation after the lifecycle timeout; proof is still unit/mock-only rather than runtime evidence from a slow preflight compaction.

## What This Changes

The PR changes preflight compaction to pass a derived cancellation signal instead of the raw reply-operation abort signal and adds focused tests for lifecycle-timeout filtering.

## Best Possible Solution

Use a non-conflated cancellation source for preflight compaction: ignore normal reply lifecycle timeout, but still propagate user stop, restart, shutdown, or drain cancellation, with regression coverage and real slow-preflight runtime proof.

## Reproduction Assessment

Yes. Source inspection shows current main passes `replyOperation.abortSignal` into required budget compaction and the safety wrapper races caller aborts against the configured timeout; the PR-introduced failure is also source-reproducible because an `AbortSignal` cannot emit a second abort after the filtered lifecycle timeout.

## Solution Assessment

No. The call-site boundary is right, but filtering the already-conflated one-shot reply abort signal is not the best fix; a distinct cancellation channel is needed so later explicit aborts still reach compaction.
