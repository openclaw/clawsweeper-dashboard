## Summary

Keep open: this PR targets the right preflight compaction bug, but the patch still lets the reply-operation abort signal win before the configured compaction timeout, so it is not merge-ready.

## What This Changes

The PR imports `resolveCompactionTimeoutMs`, passes `AbortSignal.any([replyOperation.abortSignal, AbortSignal.timeout(...)])` to required preflight compaction, and adds a unit test that checks the passed signal is not the raw reply-operation signal.

## Best Possible Solution

Define a preflight compaction cancellation contract that excludes the reply lifecycle timeout from the compaction timeout race, keeps timeout ownership in `compactWithSafetyTimeout`, and adds focused regression plus redacted runtime proof.

## Reproduction Assessment

Yes at source level: current main passes `replyOperation.abortSignal` into required budget/preflight compaction, and the safety wrapper races that external abort against the configured timeout. I did not live-run the reporter's slow-provider scenario.

## Solution Assessment

No. The PR changes signal object identity, but it keeps the same earlier abort source inside `AbortSignal.any` and duplicates timeout ownership instead of removing the reply lifecycle timeout from the compaction race.
