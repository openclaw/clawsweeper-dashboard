## Summary

Keep open: the patch targets a real P1 compaction bug, but it does not remove the early reply-operation abort from the preflight compaction race, so slow compaction can still fail before the configured timeout.

## What This Changes

This PR changes preflight compaction to pass a composed `AbortSignal.any(...)` signal and adds a unit test that checks the signal is not the raw reply-operation signal.

## Best Possible Solution

Define a preflight compaction cancellation contract that excludes the reply lifecycle timeout from the compaction timeout race, keeps timeout ownership in `compactWithSafetyTimeout`, adds focused regression coverage, and supplies redacted runtime proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.9 pass `replyOperation.abortSignal` into required budget/preflight compaction, and the safety wrapper lets that external abort race the configured timeout. I did not live-run the reporter's slow-provider scenario.

## Solution Assessment

No. The changed call site is relevant, but `AbortSignal.any` still lets the earlier reply-operation abort win and it duplicates the configured timeout owner instead of removing the reply lifecycle timeout from the compaction race.
