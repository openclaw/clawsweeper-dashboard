## Label Justifications

- `P1`: The PR targets an open high-impact compaction bug that can repeatedly block large-session agent turns, but the proposed fix currently leaves the core timeout failure unresolved.

## Summary

Keep open: the patch does not actually remove the short reply-operation abort from the preflight compaction cancellation path, so the linked timeout bug can still reproduce despite the configured compaction timeout.

## What This Changes

This PR changes preflight compaction to pass an `AbortSignal.any()` of the reply-operation abort signal and the configured compaction timeout, plus a unit test that checks the signal object is no longer the raw reply signal.

## Best Possible Solution

Define a preflight compaction cancellation signal that honors `agents.defaults.compaction.timeoutSeconds` while deliberately preserving only the abort causes maintainers want to interrupt required compaction, then prove that behavior with an abort-order regression test.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the current preflight path passes the reply-operation abort signal into required budget compaction, and the timeout wrapper rejects when that external signal aborts before the configured compaction timeout.

## Solution Assessment

No. The patch composes the same early-aborting reply signal with the configured timeout, but `AbortSignal.any()` aborts on the first source, so the best fix needs a cancellation contract that does not let the short reply lifecycle win over compaction timeout.
