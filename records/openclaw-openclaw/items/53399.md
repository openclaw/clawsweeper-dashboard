## Summary

Keep open. Current main still routes Chrome MCP existing-session passive status/profile/tab checks into the session-creating `npx chrome-devtools-mcp@latest` stdio path, and the two related fix attempts were closed unmerged, so the reported hang/orphan problem is not implemented or superseded.

## What This Changes

The issue asks OpenClaw to fix Gateway-spawned `npx chrome-devtools-mcp@latest` hangs for existing-session browser control and the resulting orphan process buildup.

## Best Possible Solution

Scope a browser-plugin fix that makes passive Chrome MCP probes non-spawning, forwards timeout/abort budgets consistently, cleans up failed stdio sessions, and deliberately decides whether the default `npx` launch should be replaced, pinned, or routed through a supported OpenClaw stdio seam.

## Reproduction Assessment

Yes for source reproduction: an existing-session Chrome MCP profile with an empty cache can still be traced from `GET /`, `/tabs`, or `listProfiles()` into `ensureChromeMcpAvailable`/`listChromeMcpTabs` and then `StdioClientTransport(npx, chrome-devtools-mcp@latest)`. The exact reporter macOS/Node 25 hang was not live-rerun.

## Solution Assessment

Unclear because no current patch exists. The closed replacement PR chain shows passive no-spawn work alone did not settle the full start-gate, cleanup, and default launch-path question, so maintainer scoping should precede another broad branch.
