## Summary

Keep open. Current main and the latest release still route Chrome MCP existing-session status/profile/tab checks through session-creating `StdioClientTransport`/`npx` paths; the open subprocess cleanup PR only covers the orphan-process part, while the passive-spawn and attach-policy work still needs maintainer direction.

## What This Changes

Prevent Gateway browser-control requests for Chrome MCP existing-session profiles from hanging or accumulating orphaned `npx`/`chrome-devtools-mcp` subprocesses when Chrome DevTools MCP spawning gets stuck.

## Best Possible Solution

Define and land the browser-plugin behavior contract for cache-only passive probes, explicit start/attach, subprocess cleanup, and default Chrome MCP launch policy, then prove it on a live macOS existing-session setup.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does show cold-cache Chrome MCP status, profile listing, and tab listing paths can still spawn `npx chrome-devtools-mcp@latest`.

## Solution Assessment

No current merged implementation fully solves the issue. The maintainable fix needs a browser-owner decision on passive no-spawn probes, explicit attach semantics, subprocess cleanup, and default launch policy.
