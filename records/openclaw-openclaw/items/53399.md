## Summary

Keep open. Current main still lets passive Chrome MCP existing-session status/profile/tab probes create temporary `npx chrome-devtools-mcp@latest` sessions on a cold cache, while the focused no-spawn repair attempts were closed unmerged and the merged timeout fix only bounded one symptom.

## What This Changes

The issue reports Gateway browser-control hangs and orphaned subprocesses when Chrome MCP existing-session paths spawn `npx chrome-devtools-mcp@latest`.

## Best Possible Solution

Define the browser plugin contract for cache-only passive Chrome MCP probes, explicit start/attach behavior, subprocess cleanup, and the default launch runner, then prove it on a live macOS existing-session setup; the FSB extension-bridge approach is useful product reference if maintainers choose a child-process-free provider direction.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection does show that cold-cache Chrome MCP status, profile listing, and tab listing can still spawn temporary `npx chrome-devtools-mcp@latest` sessions on current main and v2026.5.20.

## Solution Assessment

No current merged fix fully solves the issue. The maintainable path needs a browser-owner decision on passive no-spawn semantics, spawned process-tree cleanup, and whether the default `npx@latest` wrapper should remain the default.
