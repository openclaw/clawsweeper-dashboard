## Summary

Keep open. Current main and the latest shipped release still allow Chrome MCP existing-session status/profile/tab paths to reach `StdioClientTransport` with the default `npx chrome-devtools-mcp@latest` command; the related non-spawning repair PRs were closed unmerged, and the latest discussion adds a useful extension-bridge reference rather than proving the core issue is solved.

## What This Changes

The issue reports Gateway browser-control hangs and orphaned subprocesses when existing-session Chrome MCP paths spawn `npx -y chrome-devtools-mcp@latest` inside the Gateway process.

## Best Possible Solution

Define a browser-plugin contract for cache-only passive Chrome MCP probes, explicit start/attach behavior, subprocess cleanup, and default launch command policy, then prove it on a live macOS existing-session setup.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does show cold-cache Chrome MCP status, profile listing, and tab listing paths can still spawn `npx chrome-devtools-mcp@latest`.

## Solution Assessment

No current merged implementation fully solves the issue. The maintainable fix needs a browser-owner decision on passive no-spawn probes, explicit attach semantics, process-tree cleanup, and default launch policy.
