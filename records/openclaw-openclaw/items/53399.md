## Summary

Keep open. Current main and v2026.5.20 still route Chrome MCP existing-session passive status/profile/tab checks through session-creating `npx chrome-devtools-mcp@latest` paths, while the two related no-spawn repair PRs were closed unmerged.

## What This Changes

The issue reports Gateway browser-control requests hanging and orphaning `npx chrome-devtools-mcp` subprocesses when an existing-session Chrome MCP profile is used.

## Best Possible Solution

Define the browser plugin contract for cache-only passive Chrome MCP probes, explicit start/attach behavior, subprocess cleanup, and the default launch command, then prove it on a live macOS existing-session setup.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection does show cold-cache Chrome MCP status, profile listing, and tab listing can still spawn temporary `npx chrome-devtools-mcp@latest` sessions.

## Solution Assessment

No current merged fix fully solves the issue. The maintainable path needs an owner decision on passive no-spawn semantics, cleanup of spawned process trees, and whether the default `npx@latest` wrapper should remain the default.
