## Summary

Keep open. Current main has useful Chrome MCP timeout/readiness mitigation, but passive status, profile, and tab paths can still create ephemeral `npx chrome-devtools-mcp@latest` sessions on a cold cache, so the reported hang/orphan concern is not fully fixed or superseded.

## What This Changes

The issue reports Gateway browser control requests hanging because `npx chrome-devtools-mcp@latest` can stall inside the Chrome MCP existing-session spawn path and leave orphaned subprocesses.

## Best Possible Solution

Define the browser plugin contract for passive Chrome MCP probes, subprocess cleanup, and the default Chrome MCP launch command, then land a focused fix with live macOS existing-session proof and regression coverage for cold-cache status/tabs/profile listing.

## Reproduction Assessment

Partly. Source inspection shows cold-cache status, tabs, and profile-listing paths can still spawn the default Chrome MCP `npx` transport; the specific macOS/Node 25 hang and orphan buildup were not live-reproduced in this read-only pass.

## Solution Assessment

No current merged fix fully solves it. The timeout and `pageReady` work is useful mitigation, but the maintainable end state still needs a deliberate decision on non-spawning passive probes, subprocess-tree cleanup, and whether the default `npx@latest` wrapper remains acceptable.
