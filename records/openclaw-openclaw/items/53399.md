## Summary

Keep open. Current main still has the Chrome MCP `npx -y chrome-devtools-mcp@latest` spawn path reachable from passive browser status/profile/tab surfaces, and the short route timeout budget is not consistently forwarded into Chrome MCP tab listing. The Windows Defender discussion is useful adjacent context, but it does not resolve the original macOS/Node 25 hang report; open PR #73523 covers only the passive non-spawning slice and remains unmerged.

## What This Changes

The issue reports that browser control requests can hang when the Gateway spawns `npx -y chrome-devtools-mcp@latest` for Chrome MCP existing-session profiles.

## Best Possible Solution

Fix this inside the browser plugin by making passive status/profile/tab probes cache-only and non-spawning for Chrome MCP profiles, forwarding timeout and abort budgets through reachability/listing calls, closing failed sticky sessions promptly, and adding regressions for hung `npx`, hung `client.connect`, and hung `list_pages` through `GET /` and `GET /tabs`. Maintainers should decide whether to fold that into #73523 or follow it with a narrower transport cleanup PR.

## Reproduction Assessment

Yes for the in-repo path: an existing-session Chrome MCP profile with an empty session cache can be traced from `GET /` or `GET /tabs` to `StdioClientTransport` spawning `npx`. The exact macOS/Node 25 hang was not live-rerun, but current source still exposes the reported hang point.

## Solution Assessment

No current-main solution is in place. The safest direction is a focused browser-plugin fix plus maintainer review of #73523; replacing the default `npx` path with a cached binary may be useful, but it is not yet proven as the narrowest complete fix.
