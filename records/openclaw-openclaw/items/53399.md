## Summary

Keep open. Current main still has the default Chrome MCP existing-session launch wired through `npx -y chrome-devtools-mcp@latest`, and passive browser-control surfaces can still enter that spawn/session path. The open related PR #73523 is relevant and useful, but it is still open and explicitly covers only the passive non-spawning slice, not the remaining timeout, sticky-session cleanup, or default launch strategy questions.

## What This Changes

The issue reports that existing-session browser control can hang when the Gateway spawns `npx -y chrome-devtools-mcp@latest` through the Chrome MCP stdio transport, causing browser HTTP routes to time out and leaving orphaned processes.

## Best Possible Solution

Fix this in the browser plugin by making passive Chrome MCP status/profile/tab probes cache-only and non-spawning, forwarding timeout or abort budgets through reachability and listing operations, closing failed sticky sessions promptly, and making a deliberate maintainer decision on whether the default Chrome MCP launch should avoid `npx` in favor of a resolved or pinned executable path.

## Reproduction Assessment

Yes for the risky code path: an existing-session Chrome MCP profile with an empty session cache can enter the `npx chrome-devtools-mcp@latest` spawn path through `GET /`, `GET /profiles`, or `GET /tabs`. The exact reporter-specific macOS/Node 25 npm hang was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear on current main because no complete fix has landed. PR #73523 is a good narrow direction for passive probes, but the full issue also needs timeout propagation, failed-session cleanup, and a scoped decision on replacing or pinning the default `npx` launch path.
