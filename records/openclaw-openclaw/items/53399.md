## Summary

Keep open. Current main still reaches the Chrome MCP `npx -y chrome-devtools-mcp@latest` launch path from passive browser status and tab surfaces, and the linked PR is still open and only covers part of the reported hang/orphan cleanup problem.

## What This Changes

The issue reports that existing-session browser control requests can hang because Gateway spawns `npx -y chrome-devtools-mcp@latest` and never receives the MCP initialization response.

## Best Possible Solution

Fix this in the browser plugin by making passive Chrome MCP probes cache-only/non-spawning, forwarding timeout and abort budgets through tab listing, closing failed sessions consistently, and then deciding whether the default `npx` launch path should be replaced or pinned.

## Reproduction Assessment

Yes for the in-repo failure surface: an existing-session Chrome MCP profile with an empty session cache can be traced from `GET /` or `GET /tabs` to the `npx chrome-devtools-mcp@latest` spawn path. The exact macOS/Node 25 external hang was not live-rerun.

## Solution Assessment

No current-main solution is in place. PR #73523 is a plausible first slice for passive probes, but maintainers still need to scope the remaining timeout, cleanup, and default `npx` launch behavior.
