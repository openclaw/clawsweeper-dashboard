## Summary

Keep open. Current main has useful Chrome MCP timeout and status-readiness mitigations, but the default launch path still runs `npx chrome-devtools-mcp@latest`, and cold-cache passive status/profile/tab probes can still create temporary Chrome MCP sessions rather than using a strictly non-spawning health check.

## What This Changes

The issue reports Gateway browser-control requests hanging because `npx chrome-devtools-mcp@latest` spawned through `StdioClientTransport` can fail to initialize and leave orphaned processes.

## Best Possible Solution

Define the browser plugin contract for passive Chrome MCP probes, subprocess-tree cleanup, and the default Chrome MCP launch command, then land a focused fix with live macOS existing-session proof and regression coverage for status, tabs, and profile listing.

## Reproduction Assessment

Partly. Source inspection shows current main can still spawn the default Chrome MCP `npx` transport from cold-cache passive status, profile-listing, and tab-listing paths, but I did not live-reproduce the original macOS/Node 25 hang and orphan buildup.

## Solution Assessment

No current merged fix fully solves the issue. The timeout and `pageReady` work is useful mitigation, but the maintainable solution still needs a deliberate non-spawning passive-probe and subprocess-cleanup policy.
