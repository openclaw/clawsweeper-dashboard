## Label Justifications

- `P2`: The issue concerned bounded reset ergonomics and unnecessary model cost, now resolved in a shipped release.
- `impact:session-state`: The behavior governs session reset transitions and their immediate response.
- `impact:auth-provider`: The central problem was an unnecessary invocation of the configured default model.

## Summary

Current main fully solves the central cost problem: ordinary bare `/new` and `/reset` commands now reset immediately with static acknowledgments on both chat and gateway entry paths, without invoking the configured model. A separate greeting-model or suppression setting is no longer necessary for that behavior; custom reset-prompt content remains separately tracked.

## What This Changes

Allow bare `/new` and `/reset` to avoid an expensive default-model greeting, either by suppressing it or selecting a cheaper model.

## Best Possible Solution

Keep the shipped acknowledgment-only behavior for ordinary bare resets, use `/new <message>` or `/reset <message>` when a model turn is desired, and track custom reset-prompt semantics separately at https://github.com/openclaw/openclaw/issues/45501.

## Reproduction Assessment

No. Current main and regression tests show both ordinary chat and gateway bare resets return static acknowledgments without running the model.

## Solution Assessment

Yes for the central user problem: acknowledgment-only bare resets eliminate the unnecessary model cost more cleanly than adding greeting-specific model and suppression settings.
