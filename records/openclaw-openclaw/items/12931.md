## Summary

Keep open: current main and the latest release mitigate the original token-burn case for ordinary bare resets by returning static acknowledgements, but they still do not provide the requested reset-greeting policy, reset-specific model choice, or coordinated answer for users who want a startup/greeting turn.

## What This Changes

The issue requests config to use a cheaper model for bare `/new`/`/reset` greeting turns, or to suppress that greeting/model turn entirely.

## Best Possible Solution

Define one maintainer-approved reset-start policy covering acknowledgement-only resets, optional startup/greeting turns, prompt customization, and reset-specific model selection across auto-reply, Gateway, schema, docs, and tests.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.8` expose acknowledgement-only bare reset paths and no reset greeting model or reset greeting policy key. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

Unclear. The issue identifies the right reset-start policy gap, but the best fix should be a maintainer-approved typed policy rather than separate prompt, greeting-mode, and model-choice knobs added independently.
