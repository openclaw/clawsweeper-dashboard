## Summary

Keep open: current main fixes the ordinary bare chat token-spend path with static acknowledgements, but it still lacks any approved reset-greeting suppression or cheaper-model policy for all reset surfaces.

## What This Changes

The issue requests configurable bare `/new` and `/reset` greeting behavior, either by selecting a cheaper greeting model or suppressing the model-generated greeting.

## Best Possible Solution

Define one typed reset-start policy that covers static acknowledgements, reply-producing reset turns, gateway resets, bootstrap/startup safeguards, and any approved model override without changing defaults unexpectedly.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 acknowledge ordinary bare resets without a model run, while the remaining reply-producing prompt builder is hardcoded and no reset-greeting model/suppression config key exists.

## Solution Assessment

No, not as an automatic one-off config key. The maintainable solution needs a maintainer-approved typed reset policy that preserves current bootstrap/time safeguards and aligns all reset entry points.
