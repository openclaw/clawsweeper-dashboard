## Summary

Keep open. Current main only removes the model call for ordinary bare chat resets; it still lacks a typed reset-greeting model/suppression config, and reply-producing plus Gateway reset paths still dispatch the built-in greeting prompt with normal model selection.

## What This Changes

The issue requests a configurable cheaper model for bare `/new` and `/reset` greeting turns, or a config option to suppress that greeting entirely.

## Best Possible Solution

Define one typed reset-greeting policy across ordinary chat acknowledgements, reply-producing reset runs, and Gateway agent resets, then wire schema, types, help/docs, and focused tests around the chosen contract.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that the requested config is absent while reply-producing and Gateway reset paths still use the built-in prompt with normal model selection.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable path is a single cross-surface reset-greeting policy rather than ad hoc keys that affect only one reset path.
