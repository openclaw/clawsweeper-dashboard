## Summary

Keep open. Current main has removed the ordinary text-channel cost path by acknowledging bare `/new` and `/reset`, but reply-producing and gateway reset paths still build a hardcoded reset greeting prompt and dispatch an agent run with the active/request model, and no public config exists to suppress that greeting or choose a cheaper greeting model.

## What This Changes

Requests config to use a cheaper model for bare `/new`/`/reset` greetings or suppress the greeting/model invocation entirely.

## Best Possible Solution

Define one typed reset-greeting policy across chat acknowledgements, reply-producing reset turns, and gateway agent resets, then wire it through schema/types/help/docs and focused regression tests.

## Reproduction Assessment

Yes. Source and existing tests give a high-confidence path: ordinary bare chat resets acknowledge without a model run, while gateway and reply-producing reset paths still pass the hardcoded reset prompt into an agent run and no reset-greeting config key exists.

## Solution Assessment

Unclear as proposed. The requested options address the token-cost problem, but the maintainable fix should be a maintainer-chosen reset-greeting policy that covers all reset surfaces rather than ad hoc keys for only one path.
