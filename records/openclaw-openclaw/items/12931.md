## Summary

Keep open: current main partially reduced the token-cost problem for ordinary bare chat commands, but it still has no configurable reset-greeting suppression or dedicated cheaper-model policy for reply-producing and Gateway reset runs.

## What This Changes

The issue requests configurable `/new` and `/reset` greeting behavior so users can suppress the reset greeting or run it on a cheaper model.

## Best Possible Solution

Define one typed reset-greeting policy across chat acknowledgements, reply-producing reset runs, and Gateway agent resets, then wire schema, exported types, help/docs, and focused tests for suppression and optional model selection.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that the requested reset-greeting policy keys are absent while reply-producing and Gateway reset paths still use the built prompt with normal model selection.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable direction is one cross-surface reset-greeting policy rather than separate ad hoc keys that only affect one reset path.
