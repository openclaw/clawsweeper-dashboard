## Summary

The original unwanted token spend is fixed on current main for both ordinary chat and gateway bare resets, but the issue remains necessary because users cannot opt into a startup turn or select a cheaper model for it. The open prompt-customization issue overlaps but does not fully cover those mode and routing semantics, so consolidation requires a maintainer product decision rather than automatic closure.

## What This Changes

Add a configurable policy to suppress or choose the model used for bare `/new` and `/reset` greeting turns.

## Best Possible Solution

Define one typed reset-turn policy with static acknowledgement as the default and an explicit startup-turn mode that consistently owns prompt behavior, startup context, and an optional model override across chat and gateway entry points.

## Reproduction Assessment

Yes for current behavior: source, tests, and docs show ordinary bare chat and gateway resets now return static acknowledgements without a model run. The remaining requested opt-in startup turn and reset-specific model choice are absent from current main.

## Solution Assessment

No, the proposed independent boolean and model keys are not yet the best-defined solution. A closed typed policy shared by every reset entry point would avoid parallel sources of truth and clarify upgrade behavior.
