## Summary

Keep open: current `main` suppresses ordinary bare reset greetings with fixed acknowledgements, but it still lacks a typed reset-greeting policy or reset-specific cheaper model choice, and the related open work covers only prompt text or restoring model fallthrough.

## What This Changes

The issue requests a configurable bare `/new` and `/reset` greeting policy so users can choose a cheaper greeting model or suppress the model-generated greeting entirely.

## Best Possible Solution

Define one maintainer-approved reset-start policy covering acknowledgement-only resets, optional startup/greeting turns, prompt customization, and reset-specific model routing across schema, auto-reply, Gateway, docs, and tests.

## Reproduction Assessment

Yes, at source level: current `main` has acknowledgement-only bare reset paths, a hardcoded downstream reset prompt, and no reset greeting model or reset greeting policy key in strict config schemas. I did not run a live Gateway scenario for this read-only review.

## Solution Assessment

Unclear. The issue identifies a real reset-start policy gap, but the proposed separate keys should not be added independently before maintainers choose one typed cross-surface policy and default behavior.
