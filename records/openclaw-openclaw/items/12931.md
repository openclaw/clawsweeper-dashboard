## Summary

Keep open. Current main solved the ordinary chat token burn by acknowledging bare `/new` and `/reset` without a model call, but it still has no typed reset-greeting model/suppression policy and gateway/reply-producing reset paths still dispatch the hardcoded greeting prompt.

## What This Changes

The issue requests a cheaper configurable greeting model, or a way to suppress the greeting model turn, for bare `/new` and `/reset` commands.

## Best Possible Solution

Define one typed reset-greeting policy across chat acknowledgements, reply-producing reset turns, and gateway agent resets, then wire the selected model/suppression behavior through schema, help, docs, and regression tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection is high-confidence: current main has no reset-greeting config surface, ordinary chat bare resets already skip the model, and gateway/reply-producing paths still use the built-in prompt.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable fix should be a single cross-surface reset-greeting policy rather than ad hoc keys on only one reset path.
