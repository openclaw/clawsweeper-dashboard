- `impact:session-state`: The requested policy changes how `/new` and `/reset` session-start behavior produces or suppresses first-turn reset state.
- `impact:auth-provider`: The request includes choosing a different greeting model, which affects model/provider routing for reset turns.

## Summary

Keep open. Current main partially avoids model cost for ordinary bare chat resets, but it still has no typed reset-greeting suppression or cheaper greeting-model policy, and the reply-producing/Gateway reset paths still build the hardcoded prompt with normal model resolution.

## What This Changes

The issue requests a public config option to either use a cheaper model for bare `/new` and `/reset` greeting turns or suppress that greeting entirely.

## Best Possible Solution

Define one typed reset-greeting policy across ordinary chat acknowledgements, reply-producing reset runs, and Gateway agent resets, then wire schema, types, generated help/docs, and focused tests around suppression and optional model-selection behavior.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that the requested keys are absent while reply-producing and Gateway reset paths still use the built-in prompt with normal model selection.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable direction is one cross-surface reset-greeting policy rather than separate ad hoc keys that affect only one reset path.
