## Summary

Keep open. Current main only solves the ordinary chat-token-spend part by acknowledging bare `/new` and `/reset` without a model run; reply-producing and gateway reset paths still build the built-in greeting prompt and no reset-greeting suppression/model config exists.

## What This Changes

The issue requests configuration to use a cheaper model for bare `/new`/`/reset` greetings or suppress the greeting model turn entirely.

## Best Possible Solution

Define one typed reset-greeting policy for ordinary chat acknowledgements, reply-producing reset runs, and gateway agent resets, with schema/types/help/docs/tests and coordination with https://github.com/openclaw/openclaw/issues/45501.

## Reproduction Assessment

Not applicable as a feature request. Source inspection is high-confidence: ordinary chat bare resets already skip the model, while reply-producing and gateway reset paths still use the built-in prompt and no reset-greeting config exists.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable solution should be one cross-surface reset-greeting policy rather than ad hoc keys for only one reset path.
