## Summary

Keep open. Current main has reduced the cost issue for ordinary bare chat resets by returning a fixed acknowledgement, but the repo still has no reset-greeting suppression or cheaper-model config, and the reply-producing plus Gateway reset paths still dispatch the built-in greeting prompt.

## What This Changes

The issue requests a configurable model for bare `/new`/`/reset` greeting turns, or a way to suppress that greeting model call entirely.

## Best Possible Solution

Define one typed reset-greeting policy across ordinary chat acknowledgements, reply-producing reset runs, and Gateway agent resets, coordinated with https://github.com/openclaw/openclaw/issues/45501 and wired through schema, types, help, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that ordinary chat resets are already static while reply-producing and Gateway paths still use the built-in prompt with no reset-greeting model or suppression config.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested direction is reasonable, but the maintainable solution should be one cross-surface reset-greeting policy rather than ad hoc keys that affect only one path.
