## Summary

Keep open. Current main only partially addresses the cost concern by acknowledging ordinary bare chat `/new` and `/reset` without a model call; it still has no reset-greeting suppression or cheaper-model config, and reply-producing plus Gateway reset paths still build the built-in greeting prompt.

## What This Changes

Requests a config surface to use a cheaper model for bare `/new`/`/reset` greeting turns or suppress that greeting/model invocation entirely.

## Best Possible Solution

Define one typed reset-greeting policy across chat acknowledgements, reply-producing reset runs, and Gateway agent resets, coordinated with https://github.com/openclaw/openclaw/issues/45501 and wired through schema, types, help, docs, and tests.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that ordinary bare chat resets are static now, while reply-producing and Gateway reset paths still use the built-in greeting prompt and no reset-greeting model or suppression config exists.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable fix should be one cross-surface reset-greeting policy rather than ad hoc keys that only affect one reset path.
