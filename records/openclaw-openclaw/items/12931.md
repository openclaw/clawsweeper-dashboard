## Summary

Current main and v2026.6.6 solve the original token-cost complaint by returning static acknowledgements for ordinary bare resets, but no supported setting lets users intentionally restore a startup turn or select its model. The open prompt-customization issue overlaps without covering that distinct mode and routing choice, so meaningful product work remains.

## What This Changes

The issue requests configurable bare `/new` and `/reset` behavior, allowing a static acknowledgement or a startup greeting turn using a selected cheaper model.

## Best Possible Solution

Add one typed reset-turn policy shared by chat and gateway, with explicit static-acknowledgement and startup-turn modes and an optional model override valid only for startup-turn mode.

## Reproduction Assessment

Yes at source level: current main consistently returns fixed acknowledgements for bare chat and gateway resets, and tests assert that no model is invoked. No supported reset-mode or greeting-model setting exists in the inspected config, help, or docs surfaces.

## Solution Assessment

No, the two independent proposed options are not the best final API because their combinations are ambiguous. A closed typed policy shared across reset entry points is more maintainable and makes invalid states unrepresentable.
