## Summary

Keep open: current main suppresses ordinary bare reset greetings with static acknowledgements, but it still has no typed reset-greeting policy or reset-specific cheaper model choice, and the related open items only cover prompt text or restoring fallthrough behavior.

## What This Changes

Request a configurable reset-start greeting policy that can use a cheaper model for bare `/new` and `/reset` greetings or suppress the greeting/model turn.

## Best Possible Solution

Define one maintainer-approved reset-start policy covering acknowledgement-only resets, optional startup/greeting turns, prompt customization, and reset-specific model routing across schema, auto-reply, Gateway, docs, and tests.

## Reproduction Assessment

Yes, at source level: current main has acknowledgement-only bare reset paths, a hardcoded downstream reset prompt, and no reset greeting model or reset greeting policy key in strict config schemas. I did not run a live Gateway scenario for this read-only review.

## Solution Assessment

Unclear. The issue identifies the right reset-start policy gap, but the proposed separate keys should not be added independently before maintainers choose one typed cross-surface policy and default behavior.
