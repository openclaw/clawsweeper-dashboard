## Summary

Keep open. Current main only solves the ordinary chat-token burn by acknowledging bare `/new` and `/reset` without a model call; it still lacks a typed reset-greeting model/suppression policy, and gateway reset runs still build and dispatch the hardcoded greeting prompt.

## What This Changes

Request to add configuration that either uses a cheaper model for bare `/new`/`/reset` greeting turns or suppresses that greeting entirely.

## Best Possible Solution

Define one typed reset-greeting policy across chat acknowledgements, reply-producing reset turns, and gateway agent resets, then wire it through schema, help, docs, and regression tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that ordinary chat bare resets no longer invoke the model, while gateway/reply-producing reset paths still use the built-in greeting prompt and no reset-greeting config exists.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable fix should be a single cross-surface reset-greeting policy rather than an ad hoc key on one reset path.
