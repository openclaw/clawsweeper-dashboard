## Summary

Keep open. Current main only removes the ordinary text-channel cost path by acknowledging bare `/new` and `/reset`; reply-producing and gateway reset paths still build a hardcoded greeting prompt and run the normal agent model, with no public config to suppress it or select a cheaper greeting model.

## What This Changes

Request a configurable reset greeting policy so bare `/new` or `/reset` can use a cheaper model or skip the model-generated greeting entirely.

## Best Possible Solution

Define one typed reset-greeting policy across bare chat acknowledgements, reply-producing reset turns, and gateway agent resets, then wire it through strict config schema, exported types, generated help/docs, and regression tests.

## Reproduction Assessment

Yes. Source and existing tests give a high-confidence reproduction path: ordinary bare chat resets acknowledge without a model run, while gateway and reply-producing reset paths still pass the hardcoded reset prompt into an agent run and no reset-greeting config key exists.

## Solution Assessment

Unclear as proposed. The requested options address the token-cost problem, but the maintainable fix should be a maintainer-chosen reset-greeting policy rather than ad hoc keys that cover only one reset surface.
