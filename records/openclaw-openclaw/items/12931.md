## Summary

Keep open. Current main partially reduced token spend by making ordinary bare chat resets return static acknowledgements, but it still has no typed reset-greeting suppression/model config and the gateway/reply-producing reset paths still build and dispatch the built-in greeting prompt.

## What This Changes

The issue requests a config surface to either run bare `/new` and `/reset` greetings on a cheaper model or suppress that greeting model turn entirely.

## Best Possible Solution

Define one shared, typed reset-greeting policy across ordinary chat acknowledgements, reply-producing reset runs, and gateway agent resets, with schema/types/help/docs/tests and explicit coordination with https://github.com/openclaw/openclaw/issues/45501.

## Reproduction Assessment

Not applicable as a feature request. Source inspection is high-confidence: ordinary chat bare resets already skip the model, while gateway/reply-producing reset paths still use the built-in prompt and no reset-greeting config exists.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable solution should be one cross-surface reset-greeting policy rather than ad hoc keys for only one reset path.
