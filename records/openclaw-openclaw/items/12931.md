## Summary

Keep open. Current main and v2026.6.6 have removed the unwanted model call for ordinary bare reset acknowledgements, but the requested configurable reset-greeting policy, suppression mode, and reset-specific greeting model are still absent and overlap with active reset UX/product-decision work.

## What This Changes

The issue requests a configurable model for bare `/new`/`/reset` greeting turns or a way to suppress that greeting entirely.

## Best Possible Solution

Define one typed reset-turn policy with static acknowledgement as the default where intended, plus an explicit startup/greeting mode that consistently owns prompt behavior, startup context, and optional model override across chat and gateway entry points.

## Reproduction Assessment

Yes at source level for the missing capability: current code and docs expose static reset acknowledgements and `agents.defaults.startupContext`, but no `resetGreeting`, `resetGreetingModel`, or shared reset-turn policy. I did not run a live reset flow in this read-only cleanup review.

## Solution Assessment

No, not as the two standalone config keys proposed in the issue body. A single closed reset-turn policy shared by chat and gateway reset paths is the safer maintainable solution because it avoids parallel sources of truth for prompt, startup context, and model routing.
