## Summary

Keep open: current main and v2026.6.8 now suppress ordinary bare reset model turns, but the requested typed reset-greeting policy, reset-specific model choice, and relationship to greeting restoration/prompt override remain unresolved product/config work.

## What This Changes

Request to let users choose a cheaper model for bare `/new`/`/reset` greeting turns or suppress that greeting/model turn entirely.

## Best Possible Solution

Define one maintainer-approved reset-start policy that covers acknowledgement-only resets, startup/greeting turns, optional reset prompt customization, and optional reset-specific model selection across auto-reply, Gateway, schema, docs, and tests.

## Reproduction Assessment

Yes, at source level: current schemas/docs do not expose a reset greeting model or reset greeting policy key, while current reset handlers force acknowledgement-only bare resets. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

Unclear. The issue identifies the right reset-start policy surface, but maintainers should choose one canonical config contract instead of adding separate prompt, greeting-mode, and model-choice knobs independently.
