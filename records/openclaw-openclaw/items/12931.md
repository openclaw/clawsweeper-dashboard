## Summary

Keep open. Current main only removed the ordinary chat model call for bare `/new` and `/reset`; the gateway reset path still dispatches the hardcoded greeting prompt, and no reset-greeting model or suppression config exists.

## What This Changes

The issue requests configuration to use a cheaper model for bare `/new`/`/reset` greeting turns or suppress those greeting turns entirely.

## Best Possible Solution

Define one typed reset-greeting policy across chat acknowledgements, reply-producing reset turns, and gateway agent resets, then wire it through schema, types, help, docs, and regression tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence on the current behavior: ordinary chat bare resets acknowledge without a model call, while gateway bare resets still build and dispatch the hardcoded greeting prompt.

## Solution Assessment

Unclear. The requested direction is reasonable, but the exact fix should be a single cross-surface reset-greeting policy rather than an ad hoc key on only one reset path.
