## Summary

Keep open. Current main still lacks a typed reset-greeting policy for suppressing the model-generated greeting or selecting a cheaper greeting model; ordinary bare chat resets are currently acknowledged without a model call, but Gateway and reply-producing reset paths still build the hardcoded reset prompt and use the normal session/request model path.

## What This Changes

Request to add public configuration so bare `/new` and `/reset` can skip the greeting model turn or use a cheaper greeting model.

## Best Possible Solution

Define one typed reset-greeting policy across ordinary chat acknowledgements, reply-producing reset runs, and Gateway agent resets, then wire schema, types, generated help/docs, and focused tests around the chosen contract.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that the requested config keys are absent while Gateway and reply-producing reset paths still use the built-in prompt with normal model selection.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable direction is one cross-surface reset-greeting policy rather than separate ad hoc keys that affect only one reset path.
