## Summary

Keep this issue open: current main and release v2026.6.5 already avoid the original ordinary bare-chat token burn with static acknowledgements, but the requested configurable reset-turn capability is still missing and is not fully owned by the related prompt-customization issue or the conflicting greeting-restoration PR.

## What This Changes

Add a configurable reset response policy that can suppress the model turn or use a cheaper model for the post-reset greeting.

## Best Possible Solution

Define one discriminated reset-turn policy shared by command, reply, and gateway paths, with static-acknowledgement and model-greeting modes plus an optional non-persisted greeting-model override.

## Reproduction Assessment

Yes. Current source and docs show static acknowledgements for ordinary bare chat resets, hardcoded prompts for reset paths that still run an agent, and no configurable reset-turn policy.

## Solution Assessment

No, not in the exact proposed two-key form. A single closed reset-turn policy with an optional one-turn model override avoids invalid combinations and cross-surface inconsistency.
