## Summary

Keep open. Current main reduces the ordinary bare chat token cost by acknowledging `/new` and `/reset` without a model call, but it still has no typed reset-greeting model/suppression policy, and Gateway plus reply-producing reset paths still build the built-in reset prompt and dispatch normal model selection.

## What This Changes

The issue requests a configurable cheaper model for bare `/new`/`/reset` greetings, or an option to suppress that greeting model turn entirely.

## Best Possible Solution

Define one typed reset-greeting policy across ordinary chat acknowledgements, reply-producing reset runs, and Gateway agent resets, then wire schema, types, help/docs, and focused tests around the chosen contract while reconciling the open reset-prompt tracker and bare-reset fallthrough PR.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that the requested config keys are absent while reply-producing and Gateway reset paths still use the built-in prompt with normal model selection.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable direction is one cross-surface reset-greeting policy rather than separate ad hoc keys that affect only one reset path.
