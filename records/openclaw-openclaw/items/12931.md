## Summary

Keep open. Current main partially reduced token cost by making ordinary bare chat `/new` and `/reset` return static acknowledgements, but the requested configurable reset-greeting model/suppression policy still does not exist and Gateway/reply-producing reset paths still build and dispatch the built-in greeting prompt with normal model selection.

## What This Changes

The issue requests config to use a cheaper model for bare `/new`/`/reset` greeting turns or suppress that greeting/model invocation entirely.

## Best Possible Solution

Define one typed reset-greeting policy across ordinary chat acknowledgements, reply-producing reset runs, and Gateway agent resets, then wire schema, types, help/docs, and focused tests around the chosen contract.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that the requested config is absent while reply-producing and Gateway reset paths still use the built-in prompt with normal model selection.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable direction is one cross-surface reset-greeting policy rather than separate ad hoc keys that affect only one reset path.
