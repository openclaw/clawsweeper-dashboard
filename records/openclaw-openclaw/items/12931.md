## Summary

Keep open. Current main and the latest release already avoid model spend for ordinary bare reset acknowledgements, but they still do not provide an approved reset-greeting policy, reset-specific model override, or suppression/restoration mode across chat and Gateway reset paths.

## What This Changes

The issue requests configurable behavior for bare `/new` and `/reset` greeting turns, including either a cheaper reset-greeting model or suppression of the greeting model turn.

## Best Possible Solution

Define one typed reset-turn policy that preserves the cheap acknowledgement default unless maintainers choose otherwise, while explicitly covering prompt text, startup context, greeting mode, and reset-specific model routing across chat and Gateway paths.

## Reproduction Assessment

Yes, at source level for the missing capability: current main exposes static reset acknowledgements and `agents.defaults.startupContext`, but no `resetGreeting`, `resetGreetingModel`, or shared reset-turn policy. I did not run a live reset flow in this read-only cleanup review.

## Solution Assessment

No, not as two independent config keys from the issue body. The maintainable fix is a single closed reset-turn policy so prompt text, startup context, acknowledgement behavior, and optional model routing do not become competing sources of truth.
