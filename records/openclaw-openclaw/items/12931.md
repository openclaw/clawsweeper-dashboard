## Summary

Keep open. Current main only removed the model call for ordinary bare chat resets; it still has no typed reset-greeting model/suppression config, and gateway/reply-producing reset paths still build and dispatch the built-in greeting prompt.

## What This Changes

Add configuration for bare `/new` or `/reset` greetings so users can choose a cheaper greeting model or suppress the greeting/model call.

## Best Possible Solution

Define one typed reset-greeting policy across ordinary chat acknowledgements, reply-producing reset runs, and gateway agent resets, with schema/types/help/docs/tests and coordination with https://github.com/openclaw/openclaw/issues/45501.

## Reproduction Assessment

Not applicable as a feature request. Source inspection is high-confidence: ordinary chat bare resets already skip the model, but gateway/reply-producing reset paths still use the built-in prompt and no reset-greeting config exists.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable solution should be one cross-surface reset-greeting policy rather than ad hoc keys for only one reset path.
