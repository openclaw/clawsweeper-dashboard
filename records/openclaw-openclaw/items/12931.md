## Summary

Keep open. Current main still has no public reset-greeting suppression or cheaper greeting-model policy, and the remaining behavior crosses ordinary chat acknowledgements, reply-producing reset runs, Gateway agent resets, startup context, and model/provider routing.

## What This Changes

The issue requests config to use a cheaper model for bare `/new` and `/reset` greetings or suppress that greeting model turn entirely.

## Best Possible Solution

Define one typed reset-greeting policy that covers chat acknowledgements, reply-producing reset runs, and Gateway agent resets, then wire schema, types, docs/help, and focused tests for suppression and optional model selection.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that the requested keys are absent while reply-producing and Gateway reset paths still use the built-in prompt with normal model selection.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable direction is one cross-surface reset-greeting policy rather than separate ad hoc keys that affect only one reset path.
