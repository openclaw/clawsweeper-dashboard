## Summary

Keep open. Current main only solved the ordinary chat token-burn path by acknowledging bare `/new` and `/reset` without a model call; it still has no reset-greeting model/suppression config, and gateway/reply-producing reset paths still dispatch the built-in greeting prompt.

## What This Changes

Request configurable handling for bare `/new` and `/reset` greetings, either by choosing a cheaper greeting model or suppressing the model-generated greeting.

## Best Possible Solution

Define one typed reset-greeting policy across chat acknowledgements, reply-producing reset turns, and gateway agent resets, then wire the selected model or suppression behavior through schema, help, docs, and regression tests while coordinating with the prompt-override tracker.

## Reproduction Assessment

Not applicable as a feature request. Source inspection is high-confidence: current main lacks a reset-greeting config surface, ordinary chat bare resets already skip the model, and gateway/reply-producing paths still use the built-in prompt.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable fix should be a single cross-surface reset-greeting policy rather than ad hoc keys on only one reset path.
