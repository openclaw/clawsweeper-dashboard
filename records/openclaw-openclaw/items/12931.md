## Summary

Keep open: current main still has no reset-greeting suppression or reset-specific cheaper-model policy, and the remaining work needs a maintainer product/config decision across chat, reply-run, and Gateway reset paths.

## What This Changes

The issue requests config to either run bare `/new` and `/reset` greetings on a cheaper model or suppress the reset greeting/model turn entirely.

## Best Possible Solution

Define one typed reset-greeting policy across chat acknowledgements, reply-producing reset runs, and Gateway agent resets, then wire schema, exported types, help/docs, and focused tests for suppression plus optional model selection.

## Reproduction Assessment

Not applicable as a strict bug reproduction, but source inspection gives high confidence that the requested policy is absent and that reply-producing plus Gateway reset paths can still use the built greeting prompt with normal model selection.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable direction is one cross-surface reset-greeting policy rather than ad hoc keys that only affect one reset path.
