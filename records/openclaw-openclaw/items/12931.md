## Summary

Keep open. Current main and v2026.6.6 already avoid the expensive model call for ordinary bare `/new` and `/reset` acknowledgements, but there is still no approved reset-greeting policy, reset-specific model override, or shared config contract for the remaining startup/greeting-turn behavior.

## What This Changes

The issue asks for configurable reset greeting behavior so bare `/new` or `/reset` can use a cheaper greeting model or suppress the model-generated greeting.

## Best Possible Solution

Define one typed reset-turn policy that keeps static acknowledgement as the intended default, then explicitly supports any approved startup/greeting turn and reset-specific model override across schema, help, docs, chat, gateway, bootstrap, startup-context, and tests.

## Reproduction Assessment

Yes, at source level for the remaining missing capability: current main exposes static reset acknowledgements and `agents.defaults.startupContext`, but no `resetGreeting`, `resetGreetingModel`, or shared reset-turn policy. I did not run a live reset flow in this read-only cleanup review.

## Solution Assessment

No, not as two independent keys from the issue body. The maintainable fix is a single closed reset-turn policy so prompt text, startup context, acknowledgement behavior, and optional model routing do not become competing sources of truth.
