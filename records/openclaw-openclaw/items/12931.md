## Summary

Keep this issue open: the default token-cost problem is fixed in the latest release, but a distinct product decision remains around opting into a reset startup turn, choosing its model, and aligning that behavior with the configurable reset-prompt request.

## What This Changes

Requests configurable bare `/new` and `/reset` behavior that can suppress the greeting model turn or route it to a cheaper model.

## Best Possible Solution

Define one typed reset-turn policy with static acknowledgement as the default and explicit opt-in modes for a normal or model-overridden startup turn, sharing prompt customization and startup-context rules with the canonical reset-prompt work.

## Reproduction Assessment

Yes, at source level. Current source, tests, and v2026.6.5 content show ordinary bare chat and gateway resets return fixed acknowledgements without a model run, while model-driven reset-prompt machinery remains for other reply-producing reset states.

## Solution Assessment

No, not as two unrelated configuration keys. The narrowest maintainable solution is one closed reset-turn policy that consistently owns acknowledgement, prompt customization, startup context, and optional model selection across entry points.
