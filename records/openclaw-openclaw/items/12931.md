## Summary

Keep open. Current main partially avoids the cost for ordinary bare chat resets, but it still lacks the requested reset-greeting suppression or cheaper-model config and the reply-producing plus Gateway reset paths still run the built-in greeting prompt; the open reset-prompt tracker overlaps but does not supersede the no-model or alternate-model policy.

## What This Changes

Request configurable `/new` and `/reset` greeting behavior so users can choose a cheaper greeting model or suppress the model-generated greeting.

## Best Possible Solution

Define one typed reset-greeting policy across ordinary chat acknowledgements, reply-producing reset runs, and Gateway agent resets, coordinated with https://github.com/openclaw/openclaw/issues/45501 and wired through schema/types/help/docs/tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that ordinary bare chat resets already skip the model, while reply-producing and Gateway reset paths still use the built-in prompt and no reset-greeting model or suppression config exists.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable fix should be one cross-surface reset-greeting policy rather than ad hoc keys that affect only one reset path.
