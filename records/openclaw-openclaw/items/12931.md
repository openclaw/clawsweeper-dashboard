## Summary

Keep open: current main and the latest release avoid the original bare-reset token burn by returning static acknowledgements, but OpenClaw still has no typed reset-greeting policy for choosing a cheaper greeting model, suppressing/restoring the model turn by configuration, or aligning the related reset prompt/fallthrough decisions.

## What This Changes

Request to add configuration for bare `/new` and `/reset` greeting behavior, either by selecting a cheaper greeting model or suppressing the greeting/model invocation.

## Best Possible Solution

Define one cross-surface reset-greeting policy for chat acknowledgements, reply-producing reset turns, and Gateway agent resets, then wire schema/types/help/docs/tests after maintainers choose the default and how it relates to https://github.com/openclaw/openclaw/issues/45501 and https://github.com/openclaw/openclaw/pull/81939.

## Reproduction Assessment

Yes, source-level. Current main has no requested reset-greeting config key, and the reset paths are hardcoded to static acknowledgements or built-in prompt text rather than a configurable cheaper greeting model.

## Solution Assessment

Unclear until maintainers choose the public contract. The current static acknowledgement solves the original token-burn symptom for bare resets, but it is not the complete requested configurable policy and related users now want an opt-in model greeting path.
