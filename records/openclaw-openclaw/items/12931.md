- `P2`: This is a normal-priority feature/config request with real token-cost and reset UX impact but no emergency runtime breakage.
- `impact:session-state`: The request changes `/new` and `/reset` first-turn session handoff behavior and whether reset startup state produces a model reply.
- `impact:auth-provider`: The requested cheaper greeting model would affect reset-turn provider/model routing.

## Summary

Keep open: current main partially avoids the ordinary bare chat reset model turn with a static acknowledgement, but it still has no public reset-greeting suppression/model policy and Gateway reset runs still build the greeting prompt with normal model selection.

## What This Changes

Request to add configuration that either uses a cheaper model for bare `/new` or `/reset` greeting turns or suppresses the greeting/model invocation.

## Best Possible Solution

Define one typed reset-greeting policy across chat acknowledgements, reply-producing reset runs, and Gateway agent resets, then wire schema, exported types, help/docs, and focused tests for suppression plus optional model selection.

## Reproduction Assessment

Not applicable as a strict bug reproduction, but source inspection gives high confidence that the requested policy is absent and that reply-producing plus Gateway reset paths can still use the built greeting prompt with normal model selection.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable direction is one cross-surface reset-greeting policy rather than ad hoc keys that only affect one reset path.
