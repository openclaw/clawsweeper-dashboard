## Summary

Current main fixes the original token-cost complaint for ordinary bare chat `/new` and `/reset` by returning a static acknowledgement, but it does not provide a configurable reset policy and other reset entry points still construct and run the model-driven startup/greeting prompt. The related prompt-customization issue remains open, and the later commenter demonstrates that users also want an explicit choice between static acknowledgement and startup turn, so meaningful product work remains.

## What This Changes

Add a configurable reset policy that can suppress the model-generated bare-reset greeting or route that turn to an explicitly cheaper model.

## Best Possible Solution

Define one typed reset-turn policy, at the agent/default owner boundary, that explicitly selects static acknowledgement or model startup turn and optionally supplies a model override; apply it consistently to chat and gateway paths while preserving required bootstrap and startup-context invariants.

## Reproduction Assessment

Yes, at source level with high confidence: current handlers and regression tests identify the static-ack chat path, while the reply and gateway paths still build the model-run reset prompt. This read-only review did not execute a live installation.

## Solution Assessment

No, not yet as proposed: separate `resetGreeting` and `resetGreetingModel` fields do not fully define startup/bootstrap semantics across all entry points. A single discriminated reset-turn policy is the narrower maintainable solution.
