## Summary

Keep open. Current main only fixes the ordinary text-channel cost path by acknowledging bare `/new` and `/reset` without a model call; the reply-producing and gateway agent reset paths still build the hardcoded reset greeting prompt and dispatch a normal agent run, and there is still no public config key to suppress the greeting or choose a cheaper greeting model.

## What This Changes

Add configuration to either run the bare `/new` or `/reset` greeting with a cheaper model or suppress that greeting model turn entirely.

## Best Possible Solution

Define one typed reset-greeting policy across chat acknowledgements, reply-producing reset turns, and gateway agent resets, then wire it through strict config schema, exported types, generated help/docs, and regression tests so users can suppress the model-run greeting or select an explicit cheaper greeting model without changing `/new <message>` and `/reset <message>` behavior.

## Reproduction Assessment

Yes. Source and existing tests provide a high-confidence reproduction path: bare text-channel resets acknowledge without invoking the model, while gateway and reply-producing reset paths still feed the hardcoded reset greeting prompt into an agent run.

## Solution Assessment

Unclear as proposed. The requested options address the token-cost problem, but the maintainable fix should be one maintainer-chosen reset-greeting policy rather than ad hoc keys that only cover one reset path.
