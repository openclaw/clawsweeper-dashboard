## Summary

Keep open. Current main partially addresses token spend for ordinary bare chat resets by returning fixed acknowledgements, but it still has no typed reset-greeting suppression/model config and the reply-producing plus Gateway agent reset paths still build and dispatch the built-in greeting prompt.

## What This Changes

Requested a configurable cheaper model for bare `/new` and `/reset` greeting turns, or a config option to suppress that greeting entirely.

## Best Possible Solution

Define one shared, typed reset-greeting policy across ordinary chat acknowledgements, reply-producing reset runs, and Gateway agent resets, with schema/types/help/docs/tests and explicit coordination with https://github.com/openclaw/openclaw/issues/45501.

## Reproduction Assessment

Not applicable as a feature request. Source inspection is high-confidence: ordinary chat bare resets already skip the model, while reply-producing and Gateway reset paths still use the built-in prompt and no reset-greeting model/suppression config exists.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable solution should be one cross-surface reset-greeting policy rather than ad hoc keys that affect only one reset path.
