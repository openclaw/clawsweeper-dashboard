## Summary

Keep open. Current main removed the ordinary bare chat reset model call, but reply-producing and Gateway reset paths still construct the hardcoded reset greeting prompt and dispatch a normal agent run, with no public config to suppress the greeting or choose a cheaper greeting model.

## What This Changes

Allow bare `/new` or `/reset` greeting behavior to be suppressed or run on a cheaper configured model instead of using the session default model.

## Best Possible Solution

Define one typed reset-greeting policy across bare chat acknowledgements, reply-producing reset turns, and Gateway agent resets, then wire it through schema, generated help, docs, and regression tests.

## Reproduction Assessment

Yes. Source and existing tests give a high-confidence path: bare chat resets acknowledge without a model run, while reply-producing and Gateway reset paths still pass the hardcoded reset prompt into an agent run and no reset-greeting config key exists.

## Solution Assessment

Unclear as proposed. The requested options address the token-cost problem, but the maintainable fix should be one maintainer-chosen reset-greeting policy rather than ad hoc keys that only cover part of the reset surface.
