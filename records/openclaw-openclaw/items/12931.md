## Summary

Keep open. Current main avoids the model call for ordinary bare chat resets, but it still has no typed reset-greeting suppression/model config and the reply-producing plus Gateway agent reset paths still dispatch the built-in greeting prompt; the related prompt-override tracker covers adjacent work but not the cheaper-model or no-model policy.

## What This Changes

The issue requests configuration to use a cheaper model for bare `/new`/`/reset` greetings or suppress that greeting entirely.

## Best Possible Solution

Define one shared reset-greeting policy across ordinary chat acknowledgements, reply-producing reset runs, and Gateway agent resets, coordinated with https://github.com/openclaw/openclaw/issues/45501 and wired through schema/types/help/docs/tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that ordinary chat bare resets already skip the model, while reply-producing and Gateway reset paths still use the built-in prompt and no reset-greeting model/suppression config exists.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable solution should be one cross-surface reset-greeting policy rather than ad hoc keys that affect only one reset path.
