## Summary

Keep open: current main partially reduces the token-cost issue for ordinary bare chat resets, but it still lacks a typed reset-greeting suppression or cheaper-model policy, and reply-producing plus Gateway reset paths still build the greeting prompt under normal model routing.

## What This Changes

Request a session-level option to use a cheaper model for bare `/new` and `/reset` greetings or suppress the greeting turn entirely.

## Best Possible Solution

Define one typed reset-greeting policy across chat acknowledgements, reply-producing reset runs, and Gateway agent resets, then wire schema, exported types, help/docs, and focused tests for suppression plus optional model selection.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that the requested reset-greeting policy keys are absent while reply-producing and Gateway reset paths still use the built prompt with normal model selection.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable direction is one cross-surface reset-greeting policy rather than separate ad hoc keys that only affect one reset path.
