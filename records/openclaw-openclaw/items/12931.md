## Summary

Keep open. Current main only avoids the model for ordinary bare chat resets; reply-producing and Gateway reset paths still use the built-in reset prompt with normal model selection, and there is still no reset-greeting model or suppression config.

## What This Changes

Request adds configuration to use a cheaper model for bare `/new`/`/reset` greeting turns or suppress those greeting turns entirely.

## Best Possible Solution

Define one typed reset-greeting policy across ordinary chat acknowledgements, reply-producing reset runs, and Gateway agent resets, coordinated with the open reset-prompt and bare-reset contract trackers, then wire schema, types, help/docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that ordinary chat resets are already static while reply-producing and Gateway paths still use the built-in prompt with no reset-greeting model or suppression config.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested direction is reasonable, but the maintainable solution should be one cross-surface reset-greeting policy rather than ad hoc keys that affect only one path.
