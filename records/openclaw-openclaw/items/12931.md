## Summary

Keep open: current main still lacks the requested reset-greeting suppression or cheaper-model policy. Ordinary bare chat resets now acknowledge without a model call, but reply-producing and Gateway reset paths still build the hardcoded reset prompt and use normal request/session model selection; related reset-prompt and fallthrough work does not fully cover this public config contract.

## What This Changes

Request a config surface to choose a cheaper model for bare `/new` and `/reset` greeting turns or suppress the reset greeting/model run entirely.

## Best Possible Solution

Define one typed reset-greeting policy across chat acknowledgements, reply-producing reset runs, and Gateway agent resets, then wire schema, types, help/docs, and focused tests for suppression and optional model selection.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that the requested keys are absent while reply-producing and Gateway reset paths still use the built-in prompt with normal model selection.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable direction is one cross-surface reset-greeting policy rather than separate ad hoc keys that affect only one reset path.
