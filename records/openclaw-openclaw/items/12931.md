## Summary

Keep open. Current main partially avoids the cost for ordinary bare chat resets, but it still has no typed reset-greeting suppression or cheaper greeting-model policy, and Gateway/reply-producing reset paths still build the hardcoded prompt and use normal model resolution.

## What This Changes

The issue requests configuration to suppress the model-generated bare `/new` or `/reset` greeting, or to run that greeting on a cheaper model.

## Best Possible Solution

Define one typed reset-greeting policy across ordinary chat acknowledgements, reply-producing reset runs, and Gateway agent resets, then wire schema, types, generated help/docs, and focused tests around the chosen suppression/model-selection contract.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that the requested config keys are absent while Gateway and reply-producing reset paths still use the built-in prompt with normal model selection.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable direction is one cross-surface reset-greeting policy rather than separate ad hoc keys that affect only one reset path.
