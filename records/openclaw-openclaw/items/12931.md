## Summary

Keep open: current main still lacks a typed reset-greeting suppression or model-selection policy; ordinary bare chat resets now acknowledge without a model call, but reply-producing and Gateway reset paths still build the hardcoded greeting prompt and use normal/request model selection.

## What This Changes

Request to add configuration for bare `/new` and `/reset` greetings so users can suppress the model-run greeting or route it to a cheaper model.

## Best Possible Solution

Define one typed reset-greeting policy across chat acknowledgements, reply-producing reset runs, and Gateway agent resets, then wire schema, exported types, help/docs, and focused tests for suppression plus optional model selection.

## Reproduction Assessment

Yes, source-level. Current main has no requested config key and the reply-producing plus Gateway reset paths still use the built reset prompt with normal model selection.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable direction is one cross-surface reset-greeting policy rather than ad hoc keys that only affect one reset path.
