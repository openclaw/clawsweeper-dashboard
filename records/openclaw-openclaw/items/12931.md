## Summary

Keep open for maintainer product direction. Current main and `v2026.6.5` already avoid the original expensive bare-reset model turn by returning a static acknowledgement, but there is still no typed reset-greeting policy for opting back into a startup turn, suppressing delivery differently, or choosing a cheaper greeting model; adjacent prompt and greeting-contract work remains split across related open items.

## What This Changes

The issue requests a configurable bare `/new`/`/reset` greeting policy, either by using a cheaper model for the greeting turn or by suppressing the greeting model call.

## Best Possible Solution

Define one canonical reset-start policy that covers acknowledgement-only reset, optional startup/model greeting, prompt/greeting customization, and model selection across chat and Gateway entry points, then align schema, help, docs, and tests around that decision.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.5` show bare reset commands returning static acknowledgements without model execution, while the schema search shows no reset-greeting model or policy option to choose another behavior.

## Solution Assessment

No autonomous fix is ready. The safest path is a maintainer decision on the reset-start contract before adding config, because current behavior intentionally avoids the model turn while related issues request restoring or customizing it.
