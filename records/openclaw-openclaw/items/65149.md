## Summary

Keep open: the hook proposal is plausible, but this PR is not merge-ready because it lacks real behavior proof, exposes raw final reply payloads without the existing conversation-access guard, and is stale against current final reply assembly. It also overlaps the newer normalized payload hook proposal at https://github.com/openclaw/openclaw/pull/82823 and the model-provenance issue at https://github.com/openclaw/openclaw/issues/69110.

## What This Changes

The PR adds a public `final_reply_payloads` plugin hook in the reply runner and hook runner/types so plugins can mutate final `ReplyPayload[]` values and the usage line with resolved provider/model data before the usage footer.

## Best Possible Solution

Define one documented final-reply or normalized-payload provenance hook at the correct post-assembly point, gate or filter sensitive payload content, coordinate it with https://github.com/openclaw/openclaw/pull/82823 and https://github.com/openclaw/openclaw/issues/69110, and land it with focused tests, docs, and redacted real behavior proof.
