## Summary

Keep open: the hook idea is plausible, but the PR currently adds a public final-payload hook without the existing conversation-access guard, places it before current-main trailing status payload assembly, and still lacks after-fix real behavior proof.

## What This Changes

Adds a `final_reply_payloads` plugin hook, hook-runner method, and `runReplyAgent` call site so plugins can rewrite final reply payloads and usage-line context with resolved provider/model metadata.

## Best Possible Solution

Define one documented final-reply/provenance hook after all non-usage payload assembly, gated as conversation access or filtered to public outbound content, with focused tests/docs and redacted real behavior proof.

## Reproduction Assessment

Not applicable: this is a new public plugin hook/API proposal rather than a current-main regression. Source inspection confirms current main has no `final_reply_payloads` hook.

## Solution Assessment

No, not as submitted. The hook direction may be useful, but the current patch needs the conversation-access/security boundary fixed, the hook moved after final status payload assembly, and maintainer agreement on the public provenance contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate final payload hooks as conversation access:** `src/plugins/hook-types.ts:282`
  - body: This hook receives raw `ReplyPayload[]`, which can include `isReasoning` payloads that current delivery/storage paths intentionally filter out. Because the PR adds `final_reply_payloads` to the public hook list but not to `CONVERSATION_HOOK_NAMES`, non-bundled plugins can register it without `allowConversationAccess`; add it to the guarded conversation-hook set and update the loader/docs coverage, or pass only already-public outbound payloads.
  - confidence: 0.88
- **[P2] Move the hook after final status payload assembly:** `src/auto-reply/reply/agent-runner.ts:817`
  - body: Current main appends `trailingPluginStatusPayload` after prefix payloads and before the usage line. Inserting the hook immediately after prefix payloads means plugins cannot inspect or rewrite the final trace/status payload even though the hook is described as operating on finalized reply payloads before usage is appended.
  - confidence: 0.86
