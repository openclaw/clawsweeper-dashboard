## Summary

Keep this open. Current main still sends the Control UI New Session button and typed /new or /reset through chat.send on the active UI session key, while the separate fresh dashboard-session path is sessions.create; the prior ClawSweeper keep-open comment remains accurate against e8f9c3e6dedc8b664317264a0a15f6862488edf3.

## What This Changes

Request that Control UI /new, /reset, and New Session create and switch to a genuinely fresh independent session matching direct session creation instead of silently reusing the canonical main session path.

## Best Possible Solution

Add a Control UI-owned create-and-switch path that calls sessions.create for the active agent, uses the current session as parent when available, switches to the returned key, and refreshes chat/session/avatar/command state while leaving generic gateway /new and /reset semantics intact.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction: the New Session button and typed /new or /reset dispatch chat.send against state.sessionKey, while direct fresh session creation is the separate sessions.create RPC returning a dashboard key.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Control UI flow, reusing sessions.create plus switchChatSession; changing gateway-wide /new semantics would risk channels and other clients that intentionally reset in place.
