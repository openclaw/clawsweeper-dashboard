## Summary

Keep open: the stale `plugins.deny` bug is real and the branch is aimed at the right surface, but the recovery-policy change is broader than the stated fix and can suppress last-known-good recovery for malformed plugin policy config. The PR also only supplies focused tests/format output as proof, so it still needs real doctor behavior proof before merge.

## What This Changes

The branch changes config validation, doctor stale-plugin cleanup, recovery policy, docs/changelog, and tests so missing `plugins.deny` plugin ids are treated as stale repairable config instead of fatal validation errors.

## Best Possible Solution

Keep the deny-only stale plugin fix, but narrow recovery bypass to stale missing-plugin diagnostics and preserve last-known-good recovery for malformed `plugins.allow`/`plugins.deny` schema errors, then prove it with a real `openclaw doctor --fix` run.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: current `main` treats missing `plugins.deny` ids as validation issues and stale-plugin cleanup does not scan deny entries, so a deny-only stale id can enter doctor invalid-config recovery.

## Solution Assessment

Is this the best way to solve the issue? Not yet; the validation and stale-cleanup direction is narrow, but the recovery-policy path check should be constrained to stale missing-plugin issues instead of every exact plugin policy path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Limit recovery bypass to stale plugin-not-found issues:** `src/config/recovery-policy.ts:24`

Because the schema rejects malformed policy values at the same path, a config like `plugins: { deny: "bad" }` would now skip last-known-good recovery during doctor preflight instead of restoring a known-good file. Gate this on the stale missing-plugin diagnostic, or otherwise keep malformed policy schema errors on the normal recovery path.
