## Summary

Keep open: the PR fixes the central stale `plugins.deny` mismatch, but the recovery-policy change is too broad and can suppress last-known-good recovery for malformed policy values.

## What This Changes

The PR makes stale missing `plugins.deny` plugin IDs warn-only, adds deny scanning/removal to doctor stale-plugin cleanup, broadens config recovery policy, and updates tests, docs, and changelog.

## Best Possible Solution

Keep the warn-only deny validation and stale-cleanup additions, but require policy-path recovery bypass to match stale missing-plugin diagnostics and cover malformed allow/deny values.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current main treats missing `plugins.deny` ids as fatal validation issues and the stale-plugin cleanup helper does not scan deny entries.

## Solution Assessment

Is this the best way to solve the issue? No; the validation and stale-cleanup direction is narrow, but the recovery-policy path check should be constrained to stale missing-plugin diagnostics instead of every exact policy-path issue.
