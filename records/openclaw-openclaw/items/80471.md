## Summary

Keep open: the PR has a useful stale-deny validation and doctor cleanup fix, and the updated body now supplies live-output proof, but the recovery-policy change still broadens last-known-good bypass by exact path and can preserve malformed plugin policy schema errors.

## What This Changes

The branch makes missing `plugins.deny` ids warn-only, adds deny scanning/removal to doctor stale-plugin cleanup, broadens config recovery policy, and updates docs, changelog, and tests.

## Best Possible Solution

Keep the warn-only deny validation and stale-cleanup additions, but make the recovery bypass require a stale missing-plugin diagnostic for policy paths and add a malformed allow/deny regression test.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current `main` treats missing `plugins.deny` ids as fatal validation issues and the stale-plugin cleanup helper does not scan deny entries.

## Solution Assessment

Is this the best way to solve the issue? No; the validation and stale-cleanup direction is narrow, but the recovery-policy path check should be constrained to stale missing-plugin diagnostics instead of every exact policy-path issue.
