## Summary

Keep open: the intended stale-deny fix is valid, but the PR still has a blocking recovery-policy regression because exact `plugins.allow`/`plugins.deny` schema errors are treated as plugin-local stale refs.

## What This Changes

The branch makes missing `plugins.deny` ids warn-only, adds deny scanning/removal to doctor stale-plugin cleanup, expands last-known-good recovery bypass for plugin policy refs, and updates docs, changelog, and tests.

## Best Possible Solution

Keep the deny warning/cleanup behavior, narrow the recovery bypass to stale missing-plugin policy diagnostics, and add malformed `plugins.allow`/`plugins.deny` recovery regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current `main` sends missing `plugins.deny` ids through the fatal validation path and the stale-plugin cleanup helper scans allow/entries but not deny.

## Solution Assessment

Is this the best way to solve the issue? No; the validation and cleanup direction is narrow, but the recovery-policy predicate must distinguish stale missing-plugin diagnostics from ordinary malformed policy schema errors.

## Review Findings
