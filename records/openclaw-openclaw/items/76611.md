## Summary

Keep open: current main and latest release v2026.6.8 still have timer-based Matrix crypto IDB persistence, and the synchronous shared-client stop helpers still return without awaiting final persistence. The linked open PR only covers the stop-helper half, so the event-driven crypto persistence part remains unresolved.

## What This Changes

The issue reports Matrix E2EE crypto state loss across restarts because crypto IDB state is persisted on a 60s interval and some shared-client stop helpers do not await the final flush.

## Best Possible Solution

Keep this issue as the canonical tracker, review the linked partial PR for the awaited-stop-helper half, and add an owner-approved debounced crypto-event persistence path with listener cleanup and focused Matrix SDK tests.

## Reproduction Assessment

Yes, source-level: current main still persists Matrix crypto IDB once at init and then on a 60s timer, while shared stop helpers call stop without awaiting final persistence. A live self-hosted Matrix restart repro was not run in this read-only review.

## Solution Assessment

No, current main and the linked PR are not a complete solution. The maintainable fix needs both awaited shutdown behavior and debounced event-driven crypto persistence wired to verified matrix-js-sdk events.
