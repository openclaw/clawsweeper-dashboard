## Label Justifications

- `P1`: Matrix E2EE decryption can break across restarts for real channel users when recent crypto state is not durably flushed.
- `impact:data-loss`: Newly acquired Matrix crypto session state can remain only in memory and be lost before the next snapshot or awaited shutdown persist.
- `impact:message-loss`: Lost Olm/Megolm state can make inbound E2EE messages undecryptable after restart.
- `impact:session-state`: The issue is about Matrix crypto and sync lifecycle state drifting across process restarts.

## Summary

Keep open: current main still has the Matrix crypto-store persistence gap, and the related open PR only addresses the awaitable shared-client shutdown half while event-driven crypto snapshot flushing remains missing.

## What This Changes

The issue requests prompt Matrix E2EE crypto-store persistence on state-changing crypto events plus awaitable shared-client shutdown persistence so decrypted state survives restarts.

## Best Possible Solution

Keep this open, land or replace the async stop-helper PR for the shutdown half, and add Matrix-owned debounced crypto-event IDB snapshot flushing with listener cleanup and regression tests.

## Reproduction Assessment

Yes, as source-level proof: current main still persists post-init Matrix crypto state only immediately, on the 60s interval, or at shutdown, and the shared stop helpers still do not await the final persist promise. I did not reproduce the downstream `InvalidMAC` loop against a live homeserver.

## Solution Assessment

Yes. The narrow maintainable fix is in the Matrix plugin: add debounced persistence on dependency-backed crypto events with cleanup, keep the interval as fallback, and route clean shutdown through an awaitable persist API via the related PR or an equivalent replacement.
