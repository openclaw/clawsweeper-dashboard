## Label Justifications

- `P2`: The issue describes a real Matrix E2EE reliability bug with plugin-local blast radius rather than a core outage or emergency.
- `impact:data-loss`: Recently negotiated Matrix crypto session state can be lost before it is written to disk, forcing manual recovery of encrypted state.
- `impact:message-loss`: The reported failure prevents OpenClaw from decrypting and processing inbound encrypted Matrix messages after restarts.

## Summary

Keep open: current main and v2026.6.6 still match the reported Matrix E2EE persistence gaps, and the only related open PR covers the shutdown-helper half while explicitly leaving event-driven crypto-state persistence unresolved.

## What This Changes

The issue requests Matrix crypto-store durability fixes so E2EE state is flushed on crypto lifecycle events and shared-client shutdown helpers can await final persistence.

## Best Possible Solution

Land a bounded Matrix plugin fix that debounces crypto-event persistence with listener cleanup and resolves the shared shutdown-helper path, either through the linked PR or a maintainer-approved replacement.

## Reproduction Assessment

Yes for source-level reproduction: current main shows the 60s IDB persistence interval and void shared stop helpers, and a user reports the same Matrix/Element failure on OpenClaw 2026.5.28. I did not run a live homeserver restart reproduction in this read-only pass.

## Solution Assessment

Yes, broadly: event-driven persistence plus awaited shutdown targets the actual durability gap. The exact event list and shared-helper API shape still need maintainer review and dependency-backed tests.
