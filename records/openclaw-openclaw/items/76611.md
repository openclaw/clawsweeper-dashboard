## Summary

Keep open: current main and latest release v2026.6.10 still have timer-based Matrix crypto IDB persistence and void shared-client stop helpers. The open stop-helper PR only covers half of the report, while the open decrypt-bridge PR addresses an adjacent late-key retry issue rather than crypto-store durability.

## What This Changes

The issue reports that Matrix E2EE crypto state can be lost across restarts because IDB persistence is timer-based and some shared-client shutdown helpers do not await final persistence.

## Best Possible Solution

Keep this as the canonical Matrix crypto persistence lifecycle issue until maintainers approve and land both awaited shutdown behavior and debounced event-driven crypto-store persistence with listener cleanup.

## Reproduction Assessment

Yes, source-level: current main still persists Matrix crypto IDB once at init and then on a 60s timer, while shared stop helpers call stop without an awaitable final persist. A live self-hosted Matrix restart repro was not run in this read-only review.

## Solution Assessment

No current complete solution exists: the open stop-helper PR is partial and the decrypt-bridge PR addresses a different late-key retry path. The best fix needs maintainer approval for the Matrix SDK event trigger set and shutdown helper/API shape.
