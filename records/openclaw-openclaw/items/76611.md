## Summary

Keep open: current main and latest release v2026.6.8 still show timer-only Matrix crypto IDB persistence and synchronous shared-client stop helpers that do not await the final persist. The linked open PR only covers the stop-helper half, so event-driven crypto-state persistence remains unresolved.

## What This Changes

The issue requests durable Matrix E2EE crypto-store persistence across restarts by adding event-driven flushes and awaitable shutdown paths.

## Best Possible Solution

Keep this issue as the canonical tracker, review the linked partial PR for awaited stop helpers, and add a separate owner-approved event-driven/debounced IDB flush with listener cleanup and focused Matrix SDK tests.

## Reproduction Assessment

Yes, source-level: current main still persists Matrix crypto IDB once at init and then on a 60s timer, while shared stop helpers call `stop()` without awaiting `stopPersistPromise`. A live self-hosted Matrix restart repro was not run in this read-only review.

## Solution Assessment

No, the linked PR alone is not the complete fix because it covers only the shared-client stop-helper half. The maintainable path is an awaited shutdown API plus debounced event-driven crypto persistence wired to verified matrix-js-sdk events and covered by focused tests.
