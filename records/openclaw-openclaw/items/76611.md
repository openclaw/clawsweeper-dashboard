## Summary

Keep open. Current main still has the reported Matrix crypto persistence gap, and the prior ClawSweeper comment reached the same source-backed conclusion: crypto IDB state is flushed after init, on the 60s timer, and during shutdown, while shared/client cleanup paths can still return before the final persist settles.

## What This Changes

The issue requests debounced event-driven Matrix crypto-store persistence and awaitable clean shutdown paths so E2EE state survives restarts reliably.

## Best Possible Solution

Keep the repair inside the Matrix plugin: add debounced crypto-event snapshot flushing with listener cleanup, then route clean one-off/shared shutdown paths through an awaited persist API.

## Reproduction Assessment

Yes for the source-level defect: current main deterministically leaves crypto-state changes after initialization to the 60s timer or shutdown, and several stop paths can return before shutdown persistence resolves. I did not live-reproduce the Matrix homeserver InvalidMAC loop.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable fix: add debounced Matrix crypto-event persistence and make clean shutdown callers await the existing persist path without changing core plugin APIs or adding configuration.
