## Summary

Keep this open: current main still appears vulnerable to a cross-process cron reservation race, and the reporter was already on a release that contains the existing same-process duplicate-run guard.

## What This Changes

Investigate duplicate cron-job executions that can send the same scheduled notification more than once within seconds.

## Best Possible Solution

Make cron job reservation lock-protected across gateway processes while keeping the lock out of the long-running job execution window.

## Reproduction Assessment

Yes, from source inspection: run two independent OpenClaw gateway/cron processes against the same cron store and wake the same due job before either sees the other's persisted `runningAtMs`. I did not execute that repro in this read-only checkout.

## Solution Assessment

No, the current same-process guard is not sufficient for the reported shape. The safer fix is a cross-process lock around cron store load/mark/persist reservation and completion merge, not a delivery-specific Feishu dedupe.

## Repair Work Prompt

Fix duplicate cron triggers when two independent OpenClaw gateway/cron processes share the same cron store. Current `src/cron/service/locked.ts` only serializes within one process, while `src/cron/service/timer.ts` reserves due jobs by loading the store, setting `runningAtMs`, and persisting before execution. Add cross-process locking around the cron store read/modify/write reservation and completion merge paths, using the existing `src/infra/file-lock.ts` `withFileLock` primitive or an equivalent core-safe helper. Do not hold the file lock while the cron job itself executes; keep read ops reasonably nonblocking and preserve restart/catch-up semantics. Add regression coverage that simulates separate process-local lock state against the same store, not only two CronService instances sharing one imported module. Update `CHANGELOG.md` because this is user-visible. Stop if the fix would require a config option or product policy change.
