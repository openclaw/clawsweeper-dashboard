## What This Changes

The issue reports duplicate cron executions and duplicate Feishu notifications when scheduled jobs fire, and asks for scheduler-level deduplication.

## Best Possible Solution

Review or replace the linked PR with a scheduler/store-level dedupe fix plus regression coverage for the actual duplicate trigger path, then close the issue after that fix lands.

## Reproduction Assessment

Yes, at source level: two independent OpenClaw cron/gateway processes can read the same due job from the same store before either observes the other's persisted runningAtMs reservation. I did not live-run that path in this read-only review.

## Solution Assessment

Unclear for the current linked PR: scheduler-level deduplication is the right direction, but a process-local active-job check is not proven to cover a cross-process store reservation race. The safer fix is to prove the timer/refire path and make the reservation durable enough for the reported deployment shape.
