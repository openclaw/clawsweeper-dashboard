## Summary

Keep this open: current main still has a credible cross-process cron reservation race, and the reporter was on a release whose cron lock is still only process-local. The broader HA coordination proposal is related, but this report has a narrower bug-fix path around the file-backed cron store claim/complete cycle.

## What This Changes

The issue asks OpenClaw to prevent cron jobs from firing duplicate deliveries when the same scheduled job is triggered more than once for one slot.

## Best Possible Solution

Add a short-lived cross-process lock around cron store load/claim/persist and completion merge, keep the long job execution outside that lock, and add regression coverage that models separate gateway processes.

## Reproduction Assessment

Yes, by source inspection: run two independent gateway/cron processes against the same cron store and wake the same due job before either observes the other's persisted `runningAtMs`. I did not execute that runtime repro in this read-only checkout.

## Solution Assessment

Yes for a narrow bug fix: protect the file-backed cron reservation and completion critical sections with an existing-style cross-process lock. A Feishu-specific dedupe or Redis/leader-election feature would either miss the scheduler race or expand beyond this report.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/81087. Current main protects cron due-job reservation with the process-local `storeLocks` map in `src/cron/service/locked.ts`, so two independent gateway/cron processes sharing the same cron store can both load the same due job before either sees the other's persisted `runningAtMs`. Add a short-lived cross-process lock around the cron store load/collect-runnable/mark-`runningAtMs`/persist reservation path and the completion merge/persist path. Prefer the existing OpenClaw file-lock/fs-safe pattern or a cron-specific wrapper around it. Do not hold the lock while `executeJobCoreWithTimeout` runs, do not add Redis/leader-election/config surfaces, and do not try to solve the broader HA proposal in https://github.com/openclaw/openclaw/issues/13613. Add regression coverage that models separate process-local lock maps, such as a child-process/worker harness or an injectable lock-manager boundary, and proves one scheduled trigger results in one delivery/event. Preserve existing same-process behavior and cron missed-run/backoff semantics. Add a single changelog fix entry if the repair changes shipped behavior.
