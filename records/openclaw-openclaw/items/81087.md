## Summary

Keep this open: the duplicate cron execution report is still a valid scheduler bug candidate, and an open PR with closing syntax is already attached, so the issue should stay open until that PR is landed or rejected.

## What This Changes

The issue requests cron-layer deduplication so the same scheduled job cannot fire twice and send duplicate notifications.

## Best Possible Solution

Land a scheduler-level fix with regression coverage for the actual duplicate trigger path, then let the linked closing PR or its replacement close the issue after merge.

## Reproduction Assessment

Yes, from source inspection: run two independent OpenClaw cron/gateway processes against the same cron store and wake the same due job before either process observes the other's persisted runningAtMs reservation. I did not live-run that repro in this read-only checkout.

## Solution Assessment

Not yet. An in-memory active-job check alone is not proven to cover the cross-process reservation race, so the safer path is a scheduler/store reservation fix with a timer-level regression and real cron-run proof.
