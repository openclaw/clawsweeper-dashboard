## Summary

Keep open: current main has improved context detail output, but it still does not expose the compactable-transcript or real-conversation-message state that explains a `/compact` skip while `/status` shows substantial context usage.

## What This Changes

The issue asks `/status` and `/context` to distinguish effective prompt/cache/session accounting from compactable transcript content and real conversation message eligibility.

## Best Possible Solution

Add a focused diagnostic bucket that reports compactable transcript eligibility and real-conversation message count alongside existing prompt/cache/session usage, preferably in `/context detail` with a concise `/status` hint when compaction cannot run.

## Reproduction Assessment

Yes, from source inspection: `/status` and `/context` expose cached/prompt usage, while `/compact` gates on `containsRealConversationMessages(session.messages)`, and current diagnostics do not join those facts for the operator. I did not run a live Telegram/Codex reproduction in this read-only review.

## Solution Assessment

Yes, a narrow diagnostics fix is the best path: preserve the existing compaction predicate and make status/context explain why prompt/cache usage may not be compactable conversation history.
