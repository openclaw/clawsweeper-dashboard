## Summary

Keep this open: current main and the latest release still render the Control UI chat transcript as a polite live region while streamed assistant content is rendered inside it, and the open closing PR candidate is not merged and still needs maintainer/live screen-reader review.

## What This Changes

The issue requests a Control UI accessibility fix so screen readers do not announce every streamed assistant token and instead get a usable final-response announcement path.

## Best Possible Solution

Land a narrow Control UI fix that disables or defers transcript live-region announcements only while streaming/sending, restores idle/final accessibility, and includes focused DOM coverage plus live screen-reader proof.

## Reproduction Assessment

Yes for source-level reproduction: current main and `v2026.6.10` still render streamed chat content inside a `.chat-thread` live region with hardcoded `aria-live="polite"`. I did not run NVDA or VoiceOver live proof in this read-only review.

## Solution Assessment

Unclear until live proof lands: the narrow `aria-live`/`aria-busy` approach is plausible and scoped, but the best fix must prove both quiet streaming and a usable final announcement path in a real screen-reader/browser pair.
