## Summary

Keep this open: current main and the latest release still render streamed Control UI chat content inside a hardcoded polite live log, while the linked fix path remains unmerged and still needs maintainer strategy plus live screen-reader proof.

## What This Changes

The issue asks for Control UI streamed assistant responses to avoid per-token screen-reader announcements and announce the completed response only once.

## Best Possible Solution

Land one canonical Control UI fix that quiets the chat transcript only while streaming or sending, keeps idle/completed messages discoverable to assistive technology, adds focused DOM coverage, and includes redacted live screen-reader/browser proof.

## Reproduction Assessment

Yes at source level: current main renders active stream items inside a `.chat-thread` live log that is always `aria-live="polite"`. I did not run live NVDA/Firefox or VoiceOver proof, so the exact assistive-technology behavior still needs validation.

## Solution Assessment

Unclear: the narrow aria-live toggle is plausible, but prior discussion shows maintainers still need to choose the canonical accessibility semantics and verify final-message behavior with a real screen reader/browser pair.
