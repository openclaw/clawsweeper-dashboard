## Summary

Keep open: current main and the latest release still render active Control UI stream updates inside a polite live log, the focused upstream fix PRs are closed unmerged, and the only open closing reference is a conflicting fork PR rather than a viable upstream fix path.

## What This Changes

The issue asks OpenClaw Control UI chat to stop screen readers from announcing every streamed assistant token and instead announce the completed response once.

## Best Possible Solution

Pick one canonical upstream Control UI live-region pattern, then land it with focused DOM coverage plus redacted live screen-reader/browser proof that streaming stays quiet and final-message behavior is acceptable.

## Reproduction Assessment

Yes at source level, but not as a live NVDA reproduction in this review: current main renders active stream items inside `.chat-thread` while that container remains `role="log"` with `aria-live="polite"`. The reporter also provided a concrete NVDA/Firefox path, but the final fix still needs live assistive-technology proof.

## Solution Assessment

Unclear: the requested direction is narrow and plausible, but prior related PRs show maintainers still need to choose between an `aria-live` toggle, `aria-busy`, or completed-message-only live-region semantics before another branch is clearly best.
