## Summary

Keep open: current main and the latest release still render streamed Control UI chat content inside a polite live log, while the upstream target-repo fix PRs are closed unmerged and the remaining fork-side PR is open but conflicting. This still needs an upstream maintainer choice on live-region semantics plus live screen-reader proof.

## What This Changes

The issue asks for Control UI streaming responses to stop triggering per-token screen-reader announcements while still announcing completed assistant responses.

## Best Possible Solution

Choose one canonical upstream Control UI live-region pattern, then land it with focused DOM coverage and redacted live screen-reader/browser proof that streaming stays quiet and completed-message behavior remains acceptable.

## Reproduction Assessment

Yes at source level: current main renders live stream items inside `.chat-thread` while that thread remains a polite live log, and the reporter gives NVDA/Firefox steps. I did not run NVDA, JAWS, or VoiceOver, so live assistive-technology behavior remains unverified.

## Solution Assessment

Unclear: the likely fix is a narrow Control UI live-region change, but maintainers need to choose between dynamic `aria-live`, `aria-busy`, or completed-message-only semantics before another implementation is called sufficient.
