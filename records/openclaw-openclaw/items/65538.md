## Summary

Keep this open: current main still renders the Control UI chat transcript as a polite live region while streamed assistant content mutates inside it, so the accessibility problem remains tracked and the open closing PR candidate still needs maintainer/live screen-reader review before the issue can close.

## What This Changes

The issue requests that Control UI chat stop causing screen readers to announce every streamed assistant token and instead expose only the completed response once rendering settles.

## Best Possible Solution

Land a narrow Control UI fix that disables or defers live-region announcements only while streaming/sending, keeps the idle transcript accessible, and includes focused DOM coverage plus live NVDA or VoiceOver proof.

## Reproduction Assessment

Yes for source reproduction: current main still renders the streaming chat transcript inside a role="log" aria-live="polite" container. I did not run NVDA/VoiceOver live proof in this read-only review, so the exact assistive-technology behavior still needs live validation before closure.

## Solution Assessment

Unclear until live proof lands: the narrow aria-live/aria-busy approach proposed by the linked PR is plausible and well scoped, but the best fix must prove both quiet streaming and a usable final announcement path across a real screen reader/browser pair.
