## Summary

Keep open: current main and v2026.6.6 still render streamed Control UI chat inside a polite live log, and the upstream fix PR at https://github.com/openclaw/openclaw/pull/65575 closed unmerged after calling out missing live screen-reader proof and an unresolved ARIA semantics choice. The remaining open closing reference is a conflicting fork-side PR, so it is not a safe canonical upstream fix path.

## What This Changes

The issue reports that Control UI chat streaming causes screen readers to announce each incremental token because the chat transcript is a polite live region.

## Best Possible Solution

Pick one canonical upstream Control UI live-region pattern, then land it with focused DOM coverage plus redacted live screen-reader/browser proof that streaming stays quiet and final-message behavior is acceptable.

## Reproduction Assessment

Yes at source level, but not as a live repro: current main renders live stream items inside `.chat-thread` while that thread remains a polite live log, and the report provides NVDA/Firefox steps. Live screen-reader proof is still needed before calling a proposed fix sufficient.

## Solution Assessment

Unclear. A one-line `aria-live` toggle is narrow, but the prior review and contributor follow-up make the best fix depend on a maintainer accessibility choice between `aria-live`, `aria-busy`, or completed-message-only live-region semantics.
