## Summary

Keep open: current main and the latest release still lack a SKILL.md model-routing metadata contract or runtime consumer. The comments add credible production use cases, and the open implementation candidate is useful context, but this still needs maintainer product and security review rather than cleanup or a new automated fix branch.

## What This Changes

The issue requests optional SKILL.md model metadata so a skill invocation can route to a specified model instead of always using the agent default.

## Best Possible Solution

Define one maintainer-approved per-skill routing contract, then implement provider-aware model selection with allowlist/consent policy, fallback and precedence semantics, docs, focused tests, and redacted runtime proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config contract request. Source and docs inspection on current main and v2026.5.12 show the requested SKILL.md model-routing metadata and runtime consumer are absent.

## Solution Assessment

Unclear until maintainers choose the contract and security policy. The SKILL.md frontmatter direction is plausible, but the maintainable solution needs explicit precedence, fallback, allowed-model, consent, and provider/auth routing semantics.

## Security Review

Status: needs_attention

Summary: Skill-declared model routing is security-sensitive because untrusted skill metadata could influence provider choice, prompt destination, credential path, and cost.

Concerns:

- **[medium] Define consent for skill-declared routing:** `docs/tools/skills.md:167`
  - body: The skills docs tell users to treat third-party skills as untrusted code, so a SKILL.md field that can change model/provider selection needs explicit operator consent and an allowed-model policy.
  - confidence: 0.86
- **[medium] Keep model allowlists authoritative:** `docs/concepts/models.md:119`
  - body: Current model docs make agents.defaults.models the allowlist for /model and session overrides; per-skill routing should honor the same policy instead of creating an unbounded provider/model path.
  - confidence: 0.84
- **[medium] Preserve provider and auth routing boundaries:** `src/agents/pi-embedded-runner/run/setup.ts:56`
  - body: The current provider/model override hook runs before model resolution; a skill router should use an equivalent provider-aware path rather than swapping only a late model identifier after runtime setup.
  - confidence: 0.8
- **[medium] Match existing plugin override trust model:** `docs/gateway/configuration-reference.md:229`
  - body: Plugin LLM and subagent model overrides already require explicit allowModelOverride and allowedModels settings, which is a strong policy precedent for skill-driven model changes.
  - confidence: 0.82
