## Summary

Keep open: current main and v2026.5.12 still lack a SKILL.md model-routing contract, while the active implementation candidate at https://github.com/openclaw/openclaw/pull/58142 needs maintainer product and security review. The related config-surface request at https://github.com/openclaw/openclaw/issues/8441 is useful context but does not supersede this frontmatter-specific request.

## What This Changes

Add a SKILL.md frontmatter model preference so individual skills can route execution to a declared model instead of always using the agent default.

## Best Possible Solution

Define one public per-skill routing contract across SKILL.md and skills.entries, then land or replace the open implementation with provider-aware resolution, explicit allowlist/consent policy, docs, focused tests, and redacted runtime proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config contract request. Source and docs inspection on current main and v2026.5.12 show the requested SKILL.md model-routing metadata and runtime consumer are absent.

## Solution Assessment

Unclear until maintainers choose the contract and security policy. The frontmatter direction is plausible, but the maintainable solution needs explicit precedence, fallback, allowed-model, consent, and provider/auth routing semantics.

## Security Review

Status: needs_attention

Summary: Skill-declared model routing is security-sensitive because untrusted skill metadata could influence provider choice, prompt destination, credential path, and cost.

Concerns:

- **[medium] Define consent for skill-declared routing:** `docs/tools/skills.md:167`
  - body: The skills docs tell users to treat third-party skills as untrusted code, so a SKILL.md field that can change model/provider selection needs explicit operator consent and an allowed-model policy.
  - confidence: 0.86
- **[medium] Keep model allowlists authoritative:** `docs/concepts/models.md:119`
  - body: Current model docs make agents.defaults.models the allowlist for /model and session overrides; per-skill routing should honor the same policy instead of creating an unbounded provider/model path.
  - confidence: 0.84
- **[medium] Preserve provider and auth routing boundaries:** `src/agents/pi-embedded-runner/run/setup.ts:56`
  - body: Current provider/model override hooks run before model resolution; a skill router should use an equivalent provider-aware path rather than swapping only a late model identifier after runtime setup.
  - confidence: 0.8
