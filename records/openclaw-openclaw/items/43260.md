## Summary

Keep open: current main and v2026.5.12 still lack a SKILL.md model-routing contract, while the open implementation candidate needs maintainer product/security review and the related skills.entries request does not supersede this frontmatter-specific feature.

## What This Changes

Add an optional SKILL.md model-routing field so a skill can declare a preferred model for its execution.

## Best Possible Solution

Define one public per-skill routing contract across SKILL.md and skills.entries, then land or replace the open implementation with provider-aware resolution, explicit allowlist/consent policy, docs, tests, and redacted runtime proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config contract request. Source and docs inspection on current main and v2026.5.12 show the requested SKILL.md model-routing metadata and runtime consumer are absent.

## Solution Assessment

Unclear until maintainers choose the contract and security policy. The frontmatter direction is plausible, but the safest solution needs explicit precedence, fallback, allowed-model, consent, and provider/auth routing semantics.

## Security Review

Status: needs_attention

Summary: Skill-declared model routing is security-sensitive because untrusted skill metadata could influence provider choice, prompt destination, credential path, and cost.

Concerns:

- **[medium] Define consent for skill-declared routing:** `docs/tools/skills.md:167`
  - body: The skills docs tell users to treat third-party skills as untrusted code, so a SKILL.md field that can change model/provider selection needs explicit operator consent and an allowed-model policy.
  - confidence: 0.86
- **[medium] Keep model allowlists authoritative:** `docs/concepts/models.md:119`
  - body: Current model docs make agents.defaults.models the allowlist for /model and session overrides; per-skill routing should honor the same policy instead of giving skill metadata an unbounded provider/model path.
  - confidence: 0.82
- **[medium] Preserve provider and auth routing boundaries:** `src/agents/pi-embedded-runner/run/setup.ts:56`
  - body: Current provider/model overrides run before model resolution; a skill router should use an equivalent provider-aware path rather than swapping only a late model identifier.
  - confidence: 0.8
