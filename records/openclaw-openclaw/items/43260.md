## Summary

Keep open: current main and v2026.5.12 still do not define or consume a SKILL.md model-routing field, and the directly related implementation at https://github.com/openclaw/openclaw/pull/58142 is still open with product/security concerns. The related skills.entries request at https://github.com/openclaw/openclaw/issues/8441 overlaps but does not supersede this frontmatter contract.

## What This Changes

The issue requests an optional SKILL.md frontmatter model field that routes a skill invocation to a preferred model and then returns to the agent default.

## Best Possible Solution

Define one public per-skill routing contract across SKILL.md and skills.entries, then land or replace https://github.com/openclaw/openclaw/pull/58142 with provider-aware resolution, allowlist/consent policy, docs, tests, and redacted runtime proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config contract request. Source inspection of current main and v2026.5.12 gives high confidence that SKILL.md model-routing metadata and runtime routing are absent.

## Solution Assessment

Unclear until maintainers choose the public contract. The frontmatter direction is plausible, but the best solution needs explicit precedence, consent, allowed-model, fallback, and provider/auth routing semantics, especially alongside https://github.com/openclaw/openclaw/issues/8441.

## Security Review

Status: needs_attention

Summary: Skill-declared model routing is security-sensitive because untrusted skill metadata could influence provider choice, prompt destination, credential path, and cost.

Concerns:

- **[medium] Define consent for skill-declared routing:** `docs/tools/skills.md:167`
  - body: The skills docs tell users to treat third-party skills as untrusted, so a SKILL.md field that can change model/provider selection needs explicit operator consent and an allowed-model policy.
  - confidence: 0.86
- **[medium] Preserve provider and auth routing boundaries:** `src/agents/pi-embedded-runner/run/setup.ts:56`
  - body: Current provider/model overrides run before model resolution; a skill router should use an equivalent early, provider-aware path instead of swapping only a late model identifier.
  - confidence: 0.82
- **[medium] Keep model allowlists authoritative:** `docs/concepts/models.md:119`
  - body: Current model docs make agents.defaults.models the allowlist for session overrides; per-skill routing needs to honor the same policy rather than giving skill metadata an unbounded provider/model path.
  - confidence: 0.8
