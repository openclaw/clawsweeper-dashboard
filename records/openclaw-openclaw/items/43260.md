## Summary

Keep this issue open. Current main still has model selection for defaults, agents, sessions, cron, and plugin hooks, but no SKILL.md or plugin-skill metadata contract that lets a skill declare a preferred model; the fresh InvestorClaw and KinthAI comments provide concrete production validation, and the related implementation candidate remains open rather than merged.

## What This Changes

Add a declarative per-skill model preference to SKILL.md frontmatter so a skill can route execution to a different configured model than the agent default.

## Best Possible Solution

Keep this as the canonical feature request and have a maintainer review, revise, land, or supersede #58142 with a documented SKILL.md and plugin-skill metadata contract, alias/allowlist enforcement through existing model-selection APIs, clear precedence and fallback/ceiling behavior, and focused parser, snapshot, prompt, slash-invocation, runtime-routing, and security tests.

## Reproduction Assessment

Not applicable as a bug reproduction. The product gap is high-confidence by static inspection of current main: the parser, metadata type, snapshot, prompt formatter, docs, and embedded runner have no per-skill model-routing contract.

## Solution Assessment

Unclear as proposed. A bare frontmatter model field is directionally plausible, but the best maintainable solution should define the full contract and enforcement semantics before merging, especially around fallbacks, ceilings, allowlists, and whether routing happens through live switching or isolated execution.
