The issue requests an optional SKILL.md frontmatter field that lets a skill declare model routing preferences instead of inheriting only the agent/default model.

## Best Possible Solution

Define one maintainer-approved per-skill routing contract across SKILL.md and configured skills, then land or replace https://github.com/openclaw/openclaw/pull/58142 with provider-aware resolution, consent or allowlist policy, fallback semantics, docs, focused tests, and real runtime proof.

## Solution Assessment

Unclear until maintainers choose the routing and security contract. A SKILL.md field is plausible, but the durable solution needs explicit precedence, fallback, allowed-model, consent, provider/auth, tool-boundary, prompt-cache, and observability semantics.

## Security Review

Status: needs_attention

Summary: Skill-declared model routing is security-sensitive because it can alter provider choice, prompt destination, credential path, and cost from third-party metadata.

Concerns:

- **[medium] Define consent for skill-declared routing:** `docs/tools/skills.md:172`
  - body: The skills docs warn users to treat third-party skills as untrusted code, so a SKILL.md field that changes model/provider selection needs explicit operator consent and a trusted allowlist policy.
  - confidence: 0.86
- **[medium] Keep model override allowlists authoritative:** `docs/plugins/sdk-runtime.md:198`
  - body: Existing plugin LLM and subagent model overrides require allowModelOverride and can restrict allowedModels; per-skill routing should not create a less-controlled path.
  - confidence: 0.84
- **[medium] Preserve provider and auth routing boundaries:** `src/agents/pi-embedded-runner/run/setup.ts:56`
  - body: The current before_model_resolve path runs before resolveModel(), so per-skill routing should use an equivalent provider-aware path rather than swapping only a late model identifier.
  - confidence: 0.82
