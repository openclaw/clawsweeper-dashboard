## Summary

Keep this open: current main and the latest release still lack a SKILL.md model-routing contract, and the directly related implementation at https://github.com/openclaw/openclaw/pull/58142 remains open with unresolved product, provider-routing, proof, and security questions. The overlapping user-config request at https://github.com/openclaw/openclaw/issues/8441 is still open and does not supersede the SKILL.md/frontmatter contract requested here.

## What This Changes

Request adding an optional SKILL.md `model` frontmatter field so individual skills can route execution to a preferred model instead of always using the agent default.

## Best Possible Solution

Define one public per-skill routing contract across SKILL.md and skills.entries, including precedence, allowlists, fallbacks, and provider/auth-safe resolution, then land or replace https://github.com/openclaw/openclaw/pull/58142 with docs, tests, and runtime proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection of current main and v2026.5.12 gives high confidence that the requested SKILL.md model-routing field and runtime router are absent.

## Solution Assessment

Unclear until maintainers choose the public contract. The frontmatter direction is plausible, but the best implementation needs explicit precedence, consent, fallback, model allowlist, and provider/auth routing semantics, especially alongside https://github.com/openclaw/openclaw/issues/8441.

## Security Review

Status: needs_attention

Summary: Skill-declared model routing is security-sensitive because untrusted skill metadata could influence provider choice, prompt destination, credentials path, and cost.

- **[medium] Define consent for skill-declared model routing:** `docs/tools/skills.md:167`

  - confidence: 0.86

  - body: Current provider/model overrides run before model resolution; a skill router should use an equivalent early, provider-aware path rather than swapping only a later model identifier.
