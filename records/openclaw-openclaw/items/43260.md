## Best Possible Solution

Define one maintainer-approved per-skill routing contract, then land or replace the active implementation with provider-aware model resolution, explicit allowlist or consent policy, fallback and precedence semantics, docs, tests, and redacted runtime proof.

## Solution Assessment

Unclear until maintainers choose the routing and security contract. A SKILL.md field is plausible, but the maintainable solution needs explicit precedence, fallback, allowed-model, consent, provider/auth, prompt-cache, and observability semantics.

## Security Review

Status: needs_attention

Summary: Skill-declared model routing is security-sensitive because it can steer prompts and credentials across provider/model boundaries.

Concerns:

- **[medium] Define consent for skill-declared routing:** `docs/tools/skills.md:181`
  - body: The skills docs treat third-party skills as untrusted code, so any SKILL.md field that changes provider or model selection needs explicit operator consent and trusted allowed-model semantics.
  - confidence: 0.88
- **[medium] Keep model override allowlists authoritative:** `docs/gateway/configuration-reference.md:230`
  - body: Existing plugin LLM and subagent model overrides require allowModelOverride and optional allowedModels restrictions; per-skill routing should not create a less-controlled path.
  - confidence: 0.87
- **[medium] Preserve provider and auth routing boundaries:** `src/agents/pi-embedded-runner/run/setup.ts:56`
  - body: The current before_model_resolve path runs before model resolution so overrides are provider-aware; skill routing should use an equivalent early path instead of swapping only a late model identifier.
  - confidence: 0.85
