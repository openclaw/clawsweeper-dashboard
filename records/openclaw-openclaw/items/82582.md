## Summary

Keep open: current main appears to synthesize plugin command replies when Auto-TTS is enabled, but excluding command replies is a user-visible policy change that is not yet defined by docs or tests.

## What This Changes

The issue requests excluding slash/native command output such as `/active-memory status` from automatic TTS when TTS auto mode is `always`.

## Best Possible Solution

Define the command-reply TTS policy, likely suppressing automatic TTS for ordinary command/status replies while preserving explicit TTS commands, then add focused regression coverage around plugin command dispatch.

## Reproduction Assessment

Yes, by source inspection: `/active-memory status` returns a text reply, and generic final delivery applies Auto-TTS when the effective auto mode is not `off`. I did not run a live TTS server/channel reproduction.

## Solution Assessment

Unclear until maintainers decide the policy boundary. A narrow implementation should avoid synthesizing ordinary command/status replies while keeping explicit TTS commands functional.
