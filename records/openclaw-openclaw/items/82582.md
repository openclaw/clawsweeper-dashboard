## Summary

Keep open: current main still treats slash/plugin command replies as ordinary final reply payloads, and the Auto-TTS dispatch gate has no command-output skip. The only linked fix PR is closed unmerged, so the source-backed bug remains.

## What This Changes

The issue reports that slash command outputs, especially `/active-memory status`, are spoken by Auto-TTS when `ttsAuto` is `always` and should remain text-only.

## Best Possible Solution

Add a single command-response marker for all command-generated reply paths, skip Auto-TTS for that marker in final delivery, mirror the type contract, and cover the behavior with focused regression tests.

## Reproduction Assessment

Yes, by source inspection: `/active-memory status` returns plain command text, command normalization does not mark it, and final dispatch applies Auto-TTS when `ttsAuto` resolves enabled. I did not run a live TTS server or channel bot reproduction.

## Solution Assessment

Yes: the narrow maintainable solution is a central command-response marker plus one TTS dispatch guard. Special-casing Active Memory or changing TTS provider/config behavior would put the fix in the wrong layer.
