## Summary

Keep open: current main still routes plain command/status replies through the final Auto-TTS path, and the open linked fix PR is not merged yet. The source-backed bug remains, with https://github.com/openclaw/openclaw/pull/83492 as the active fix path.

## What This Changes

Exclude explicit command/status replies such as `/active-memory status` from Auto-TTS when TTS is configured as always on.

## Best Possible Solution

Land a narrow fix that identifies command-handler-produced reply payloads and skips final TTS only for those payloads, while preserving configured TTS for normal assistant and model-forwarded skill command replies.

## Reproduction Assessment

Yes, by source inspection: `/active-memory status` returns a plain text command reply, command normalization does not mark it, and final delivery applies Auto-TTS when `ttsAuto` resolves to `always`. I did not run a live TTS server or channel bot reproduction.

## Solution Assessment

Yes for the marker-plus-dispatch-guard shape, but the safe fix should be payload-scoped rather than turn-wide. The current best path is to repair and land https://github.com/openclaw/openclaw/pull/83492 or replace it with an equivalent narrow patch.
