## What This Changes

The issue asks for slash command/status replies such as `/active-memory status` to be excluded from Auto-TTS when TTS is set to always.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/82589; the desired implementation is a central command-response marker that skips automatic TTS while preserving explicit TTS commands.

## Reproduction Assessment

Yes, by source inspection: `/active-memory status` returns plain text, command delivery does not mark it as command output, and final delivery applies Auto-TTS whenever the effective auto mode is enabled. I did not run a live TTS server/channel reproduction.

## Solution Assessment

Yes, if maintainers accept the policy: the linked PR's command-response marker is the narrow maintainable fix because it gates Auto-TTS at the command normalization and dispatch chokepoints while explicit `/tts audio` and `/tts latest` still synthesize media directly.
