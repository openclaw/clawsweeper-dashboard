## Summary

Keep open: current main still lets slash-command text reach the generic Auto-TTS path, and the open linked PR is the active fix path, so this issue should wait for that PR to merge or be rejected.

## What This Changes

The issue reports that slash command replies such as `/active-memory status` are synthesized by automatic TTS when `ttsAuto` is set to `always`.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/82589; the desired implementation is a central command-response marker that skips automatic TTS while preserving explicit `/tts` commands.

## Reproduction Assessment

Yes, by source inspection: `/active-memory status` returns plain text, command delivery does not mark it as command output, and final delivery applies Auto-TTS whenever the effective auto mode is enabled. I did not run a live TTS server/channel reproduction.

## Solution Assessment

Yes, if maintainers accept the policy: the linked PR's command-response marker is the narrow maintainable fix because it gates Auto-TTS centrally while explicit `/tts audio` and `/tts latest` still synthesize media directly.
