## Summary

Keep open: current main still has a source-backed path where slash-command text replies are delivered through the normal final-reply auto-TTS path, so `/active-memory status` can be synthesized when TTS auto mode is `always`.

## What This Changes

The issue asks OpenClaw to exclude slash-command output, such as `/active-memory status`, from automatic TTS when TTS is set to always.

## Best Possible Solution

Suppress automatic TTS for explicit slash/native command replies that are plain command output, while preserving command replies that already include generated media or intentionally requested one-off audio.

## Reproduction Assessment

Yes. Source inspection shows `/active-memory status` returns a normal text command reply, and the final delivery path applies auto-TTS to that reply when TTS is enabled.

## Solution Assessment

Yes. The narrow maintainable fix is to carry command-reply origin into the auto-TTS gate or mark command-handler replies as auto-TTS-ineligible, without adding a new setting or changing provider behavior.
