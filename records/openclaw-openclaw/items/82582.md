## Summary

Keep open: current main and the latest release still let command reply payloads reach final TTS when auto TTS is enabled. The linked fix candidate at https://github.com/openclaw/openclaw/pull/83492 is still open, unmerged, conflicting, and has a prior scope finding, so this issue should stay open until a scoped fix lands.

## What This Changes

The issue reports that slash/native command output such as `/active-memory status` is spoken by TTS when TTS is configured to `always`.

## Best Possible Solution

Land a scoped command-output marker or equivalent guard that skips final TTS only for command-handler-produced replies while preserving TTS for slash commands that continue as normal prompts.

## Reproduction Assessment

Yes. Source inspection shows `/active-memory status` returns plain text, command replies are only marked for source-suppression delivery, and final dispatch still calls the TTS wrapper when `ttsAuto` resolves enabled.

## Solution Assessment

Yes for the requested direction, but not for the open PR as currently shaped. The maintainable fix is a command-output-scoped TTS skip, not a turn-wide skip for every explicit command turn.
