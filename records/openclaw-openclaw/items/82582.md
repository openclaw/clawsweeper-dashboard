## Summary

Keep open: current main still has no command-response marker or Auto-TTS skip for slash command replies, and the only linked fix PR is closed unmerged.

## What This Changes

The issue asks for slash command replies such as `/active-memory status` to stay text-only when Auto-TTS is set to `always`.

## Best Possible Solution

Add one central command-response marker across command-generated reply paths, skip Auto-TTS for that marker in delivery, mirror the payload type, and add focused regression coverage.

## Reproduction Assessment

Yes, by source inspection: `/active-memory status` returns a plain text command reply, current command normalization does not mark it, and final dispatch applies Auto-TTS whenever the effective auto mode is enabled. I did not run a live TTS server or channel bot reproduction.

## Solution Assessment

Yes: a central command-response marker plus a single TTS dispatch guard is the narrow maintainable fix, provided it covers direct skill command replies too. Special-casing `/active-memory` or changing TTS provider/config behavior would be the wrong boundary.
