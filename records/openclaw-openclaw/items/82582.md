## Summary

Keep open: current main still has the source-backed bug, and the only linked fix attempt is closed without merging. Slash command replies are still ordinary reply payloads, so `/active-memory status` output can pass through Auto-TTS when `ttsAuto` resolves to `always`.

## What This Changes

The issue asks for slash command/status replies to be excluded from automatic TTS when chat TTS is set to always.

## Best Possible Solution

Add one central command-response marker for command-generated reply payloads, skip Auto-TTS for that marker in final delivery, mirror the payload type in runtime-plan types, and add regression coverage for built-in, plugin, and direct tool-dispatched skill commands.

## Reproduction Assessment

Yes, by source inspection: `/active-memory status` returns plain text, command normalization does not mark it, and final dispatch applies Auto-TTS whenever `ttsAuto` resolves enabled. I did not run a live TTS server or channel bot reproduction.

## Solution Assessment

Yes: a central command-response marker plus one dispatch guard is the narrow maintainable fix. Special-casing active-memory or changing provider/config behavior would create the wrong boundary.
