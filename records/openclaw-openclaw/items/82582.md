## Summary

Keep open: current main and v2026.6.8 still route command reply text through final TTS when auto TTS is enabled. The linked fix candidate is open, unmerged, and currently conflicting, so this issue should remain the canonical tracker until a scoped fix lands.

## What This Changes

Command/status replies such as `/active-memory status` should not be synthesized by TTS when TTS auto mode is `always`.

## Best Possible Solution

Land a scoped command-output marker or equivalent guard that skips final TTS only for command-handler-produced replies while preserving TTS for explicit commands that continue as normal prompts.

## Reproduction Assessment

Yes. Source inspection shows `/active-memory status` returns ordinary text, command replies are not marked as TTS-excluded status notices, and final dispatch still calls the TTS wrapper when auto TTS is enabled.

## Solution Assessment

Yes for the scoped direction, but not for a broad turn-wide bypass. The maintainable fix should mark or otherwise identify command-handler output and skip TTS only for that output.
