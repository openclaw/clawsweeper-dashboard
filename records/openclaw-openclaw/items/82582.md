## Summary

Keep open: current main still treats slash-command replies as ordinary reply payloads for Auto-TTS, and the linked fix PR is closed unmerged. The bug remains source-reproducible with a narrow command-reply marker and TTS dispatch-guard fix path.

## What This Changes

Exclude slash-command/status replies such as `/active-memory status` from automatic TTS when Auto-TTS is set to `always`.

## Best Possible Solution

Add a central command-response marker for all command-generated reply paths, skip Auto-TTS for that marker at the shared dispatch gate, mirror the payload type, and add focused regression coverage.

## Reproduction Assessment

Yes, by source inspection: `/active-memory status` returns a plain text command reply, command normalization does not mark it, and final dispatch applies Auto-TTS whenever the effective auto mode is enabled. I did not run a live TTS server or channel bot reproduction.

## Solution Assessment

Yes: a central command-response marker plus one shared TTS dispatch guard is the narrow maintainable fix. It should also cover direct tool-dispatched skill command replies, not only the normal `handleCommands` path.
