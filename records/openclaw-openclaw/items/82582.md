## Summary

Keep open: current main still lacks a command-response marker or TTS skip for slash command replies, and the only linked fix PR is closed without merging. The source path for `/active-memory status` still returns plain text that can pass through Auto-TTS when `ttsAuto` resolves to `always`.

## What This Changes

Exclude slash-command/status replies such as `/active-memory status` from automatic TTS while preserving explicit TTS commands like `/tts audio` and `/tts latest`.

## Best Possible Solution

Add one central command-response marker across command-generated reply paths, skip Auto-TTS for that marker in final delivery, mirror the payload type in runtime-plan types, and add regression coverage for plugin/builtin commands plus direct tool-dispatched skill commands.

## Reproduction Assessment

Yes, by source inspection: `/active-memory status` returns a plain text command reply, current command normalization does not mark it, and final dispatch applies Auto-TTS whenever the effective auto mode is enabled. I did not run a live TTS server or channel bot reproduction.

## Solution Assessment

Yes: a central command-response marker plus a single TTS dispatch guard is the narrow maintainable fix, provided it covers direct skill command replies too. Special-casing `/active-memory` or changing TTS provider/config behavior would be the wrong boundary.
