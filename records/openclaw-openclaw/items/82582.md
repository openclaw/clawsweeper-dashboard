Keep open: current main still lets explicit slash command final replies flow into the general Auto-TTS wrapper, and a separate open closing PR already targets this exact regression. The issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests that slash command output such as `/active-memory status` be excluded from always-on Auto-TTS.

## Best Possible Solution

Land or replace the open fix at https://github.com/openclaw/openclaw/pull/83492 with a central dispatch guard and regression test that keeps command replies visible as text while leaving ordinary assistant TTS behavior unchanged.

## Reproduction Assessment

Yes. By source inspection, `/active-memory status` returns a plain text command reply, current main does not mark that reply as command-origin, and final delivery applies Auto-TTS whenever the effective TTS auto mode is enabled.

## Solution Assessment

Yes, with review of the linked PR’s compatibility risk. The narrow maintainable solution is a central final-TTS guard for explicit command replies plus regression coverage, not a special case in the Active Memory plugin or a TTS config default change.
