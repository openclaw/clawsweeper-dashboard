## Summary

Keep this open: current main and the latest release still support Discord reply threading and mention-alias rewriting, but they do not implement opt-in reply-author mention injection. The related merged fix handles final replies that already contain targeted mentions; it does not add the requested `autoMentionReply` config or carry the original author's user id into outbound reply construction.

## What This Changes

The issue requests an opt-in Discord setting that prepends an explicit mention of the original guild-message author to outbound replies when the model omits it.

## Best Possible Solution

Keep the behavior Discord-plugin-owned: if maintainers approve the product knob, carry reply-author identity through the reply delivery context and inject only for opted-in guild replies without duplicating existing mentions or escalating broadcasts.

## Reproduction Assessment

No for a high-confidence live Discord reproduction in this read-only pass; yes at source level for the missing capability. Current main can thread Discord replies by message id and rewrite mention aliases, but it does not carry reply-author identity or synthesize an explicit author mention.

## Solution Assessment

Unclear until maintainers decide the config/product contract. A Discord-plugin-owned opt-in that reuses reply context is the maintainable shape, while unconditional injection or core-level Discord policy would be the wrong layer.
