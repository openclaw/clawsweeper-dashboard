## Summary

Keep open. Current main still supports Discord native reply references, and the related merged reply-threading work helps `replyToId` plumbing, but it does not implement an opt-in visible `<@originalAuthorId>` injection setting or delivery-layer formatter. The requested behavior remains a real Discord plugin feature gap.

## What This Changes

The issue requests a Discord configuration option that automatically prepends the original guild message author's `<@userId>` mention to reply messages when the generated text omits it.

## Best Possible Solution

Add a Discord plugin-owned, default-off config setting that performs delivery-layer mention injection for guild replies only, using trusted current-message sender context when available and safe message-author resolution when only a reply reference is available. Keep prompt rules and core LLM behavior out of the fix, and cover chunking, media, webhook/thread-bound sends, streaming previews/finals, DMs, existing mentions, and missing/deleted references.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main can send a Discord message with `replyTo: "orig-123"`, which produces `message_reference`, but there is no config option or send-path code that prepends `<@authorId>` when the text is `hello`.

## Solution Assessment

Yes, the requested default-off Discord plugin formatter is the narrowest maintainable direction. The safer shape is to implement it after model output and before Discord delivery, with explicit guild-only and first-visible-chunk guards rather than relying on SOUL.md or model prompting.
