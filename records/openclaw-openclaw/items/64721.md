## Summary

Keep open. Current main covers the reported `model` and `timeoutSeconds` paths, but `contextTokens` and `maxSpendUsd` are still not an end-to-end cron payload contract, and the latest CLI evidence shows an unresolved timeout-option ambiguity.

## What This Changes

Request that cron add/update surfaces accept and persist per-job `model`, `timeoutSeconds`, `contextTokens`, and `maxSpendUsd` payload fields for agent-turn jobs.

## Best Possible Solution

Define and document an end-to-end cron resource-limit contract: keep the shipped `model` and `timeoutSeconds` behavior, add `contextTokens` with explicit precedence, expose `maxSpendUsd` only with enforcement, and update the tool schema, Gateway protocol, CLI, docs, and regression tests together.

## Reproduction Assessment

Yes. A source-level reproduction is to send `cron.add` or `cron.update` with nested `payload.contextTokens` or `payload.maxSpendUsd`; the current Gateway cron schema rejects those fields, and `cron edit --timeout` is only the inherited RPC timeout.

## Solution Assessment

No, not as a schema-only pass-through. The maintainable solution needs a cron resource-limit contract first, because current runtime does not read payload `contextTokens` and there is no `maxSpendUsd` enforcement path.
