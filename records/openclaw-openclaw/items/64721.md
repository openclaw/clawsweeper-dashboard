## Summary

Keep open: current main and the latest release cover the `payload.model` and `payload.timeoutSeconds` parts, but `contextTokens` and `maxSpendUsd` are still absent from the cron payload, protocol, storage, and run-enforcement contract.

## What This Changes

Cron agent/tool cron creation and update should accept per-job model, timeout, context-token, and spend-cap fields instead of stripping or rejecting them.

## Best Possible Solution

Keep the shipped model/timeout support, then define a first-class cron payload/API contract for per-job context and spend limits before updating schemas, storage, CLI/docs, and run enforcement.

## Reproduction Assessment

Yes for the remaining source-level gap: current main does not define `contextTokens` or `maxSpendUsd` in the cron payload/protocol/runtime contract. I did not run a live cron because this review is read-only and the missing fields are visible in source.

## Solution Assessment

No, a narrow tool-schema-only fix is no longer the best solution. The remaining fields need a maintained cron payload contract with storage, CLI/docs, and runtime budget/context semantics, plus a separate decision on whether `cron edit --timeout` should warn or alias to `--timeout-seconds`.
