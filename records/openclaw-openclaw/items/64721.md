## Summary

Keep open. Current main still omits `contextTokens` and `maxSpendUsd` from the cron tool, Gateway protocol schema, cron payload types, and runtime contract, while the latest discussion adds a related `openclaw cron edit --timeout` ambiguity in v2026.5.7.

## What This Changes

The issue asks for cron add/update tooling to accept and preserve per-job agent-turn payload controls for model, timeout, context tokens, and spend caps.

## Best Possible Solution

Define an end-to-end cron payload contract for job-level context caps, implement or split real per-run spend enforcement before exposing `maxSpendUsd`, and add an explicit CLI guard or alias for `cron edit --timeout`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to send `cron.add` or `cron.update` with nested `payload.contextTokens` or `payload.maxSpendUsd`; normalization plus the strict Gateway cron schema leaves those fields unsupported and rejects them before persistence.

## Solution Assessment

No, not as a schema-only pass-through. `contextTokens` can likely be a narrow payload contract once precedence is decided, but `maxSpendUsd` should not be exposed until there is enforceable per-run spend semantics or a separate canonical follow-up.
