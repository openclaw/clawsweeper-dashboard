## Summary

Keep open. Current main covers the originally reported `model` and `timeoutSeconds` cron payload path, but `contextTokens` and `maxSpendUsd` still are not a supported cron payload contract, and the fresh CLI comment adds a real `cron edit --timeout` ambiguity on the same timeout surface.

## What This Changes

The issue asks cron add/update, Gateway validation, and CLI surfaces to accept per-job `model`, `timeoutSeconds`, `contextTokens`, and `maxSpendUsd` controls instead of stripping or rejecting them.

## Best Possible Solution

Define an end-to-end cron agent-turn resource-limit contract: keep the shipped `model`/`timeoutSeconds` support, add validated `contextTokens` only with clear precedence, implement or split `maxSpendUsd` enforcement, and update the agent tool, Gateway schema, CLI/docs, and regression tests together.

## Reproduction Assessment

Yes. A source-level reproduction is to send `cron.add` or `cron.update` with nested `payload.contextTokens` or `payload.maxSpendUsd`; current Gateway cron payload validation lacks those fields and is strict, while `cron edit --timeout` is parsed only as the inherited RPC timeout.

## Solution Assessment

No, not as a schema-only pass-through. The maintainable fix needs a deliberate cron resource-limit contract, especially for spend enforcement and context-token precedence, plus a separate CLI timeout disambiguation path.
