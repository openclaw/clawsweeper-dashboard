## Summary

Keep open. Current main covers `model` and `timeoutSeconds`, but `contextTokens` and `maxSpendUsd` are still absent from the cron tool, Gateway protocol schema, cron payload types, and isolated-run runtime contract; the reporter's Forge repro remains valid for the remaining fields.

## What This Changes

Allow cron add/update tool payloads to accept and preserve `model`, `timeoutSeconds`, `contextTokens`, and `maxSpendUsd` for agent-turn jobs.

## Best Possible Solution

Keep the existing `model` and `timeoutSeconds` support, add a deliberate end-to-end `contextTokens` contract for agent-turn cron jobs, and split or define `maxSpendUsd` as real per-run spend enforcement before exposing it.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to send `cron.add` or `cron.update` with nested `payload.contextTokens` or `payload.maxSpendUsd`; the tool does not advertise/recover them and the strict Gateway cron payload schema rejects them before persistence.

## Solution Assessment

No, not as a schema-only pass-through. `contextTokens` can likely be a narrow cron payload contract once precedence is settled, but `maxSpendUsd` should be implemented only with enforceable per-run spend semantics or tracked separately.
