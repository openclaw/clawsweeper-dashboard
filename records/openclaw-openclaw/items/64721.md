## Summary

Keep open. Current main and the latest release now cover the original per-job model and timeoutSeconds paths, but contextTokens and maxSpendUsd are still absent from the cron tool schema, flat-parameter recovery, strict gateway protocol schema, cron payload types, persistence codec, CLI surface, and docs, so the remaining work needs an API/runtime budget decision rather than cleanup closure.

## What This Changes

The issue asks for cron add/update tooling to accept and persist per-job model, timeoutSeconds, contextTokens, and maxSpendUsd fields instead of stripping or rejecting them.

## Best Possible Solution

Define the canonical per-job cron budget contract first, then wire it through protocol schemas, cron payload/runtime types, storage/migration if persisted, agent tool schema, CLI add/edit behavior, docs, and focused regression tests together.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current cron tool and gateway schemas omit contextTokens/maxSpendUsd while the gateway protocol uses additionalProperties:false, so those fields are still unsupported by the contract.

## Solution Assessment

No. Adding fields only to the agent-facing schema would be too narrow; the maintainable fix needs a single cron API/runtime budget decision across protocol, types, persistence, CLI, docs, and tests.
