## Summary

Keep open. Current main and latest release cover the previously missing `model` and `timeoutSeconds` cron paths, but `contextTokens` and `maxSpendUsd` are still not part of the cron payload/protocol/CLI/runtime contract, so the central cost/context-control request is only partially addressed and needs maintainer API direction.

## What This Changes

The issue requests cron add/update tooling to accept and preserve per-job `model`, `timeoutSeconds`, `contextTokens`, and `maxSpendUsd` fields instead of forcing direct cron-store edits.

## Best Possible Solution

Keep the current `model`/`timeoutSeconds` implementation, then design one canonical cron agent-turn contract for per-job context limits and spend caps across tool schema, gateway protocol, CLI flags, storage, docs, and runtime enforcement.

## Reproduction Assessment

Yes for source-level reproduction: current cron payload/protocol/CLI surfaces still omit `contextTokens` and `maxSpendUsd`, and the protocol schema is closed to extra payload fields. I did not run a mutating cron command because this was a read-only review.

## Solution Assessment

Unclear as a direct schema-only fix. `contextTokens` and especially `maxSpendUsd` need a maintained runtime and upgrade contract, while the already-covered `model` and `timeoutSeconds` paths should stay as-is.
