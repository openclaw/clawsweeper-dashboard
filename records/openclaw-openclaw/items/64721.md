## Label Justifications

- `P2`: The issue affects scheduled agent job configuration and cost/context controls but is bounded to cron tooling rather than an emergency runtime outage.
- `impact:session-state`: Missing per-job context token support can leave scheduled agent runs inheriting a broader context budget than intended.

## Summary

Keep open. Current main now documents and implements the model and timeoutSeconds parts, but contextTokens and maxSpendUsd are still absent from the cron protocol, runtime payload types, storage codec, tool schema tests, CLI/docs, and update merge path, so the remaining request is real and not cleanup-closeable.

## What This Changes

The issue requests cron add/update tooling to accept and persist per-job model, timeoutSeconds, contextTokens, and maxSpendUsd fields instead of dropping or rejecting them.

## Best Possible Solution

Define the canonical per-job cron budget contract, then update gateway protocol validation, cron runtime types/storage, agent tool schema/canonicalization, CLI add/edit behavior, docs, and focused regression tests together.

## Reproduction Assessment

Yes, source-level: current cron protocol/runtime schemas still omit contextTokens and maxSpendUsd while the protocol payload schema is strict. I did not run a live Gateway command because the validator and type path already show those fields are unsupported on current main.

## Solution Assessment

No, adding only tool-schema fields would be too narrow because gateway protocol validation, runtime payload types, normalization, storage, CLI, and docs would still disagree. The maintainable fix is one explicit cron payload/API decision applied across those surfaces.
