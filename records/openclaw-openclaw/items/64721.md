## Summary

Keep open. Current main and v2026.5.18 cover the originally reported `model` and `timeoutSeconds` paths, but `contextTokens` and `maxSpendUsd` are still missing from the public cron payload contract and need a deliberate resource-limit design rather than schema-only pass-through.

## What This Changes

The issue asks cron management surfaces to accept and preserve per-job model, timeout, context-token, and spend-cap fields for scheduled agent turns.

## Best Possible Solution

Define one end-to-end cron agent-turn resource-limit contract: keep the shipped model/timeout behavior, add validated `contextTokens` with clear precedence, implement or split `maxSpendUsd` enforcement, and clarify CLI timeout wording with docs and regression tests.

## Reproduction Assessment

Yes for the remaining schema gap: source inspection shows `payload.contextTokens` and `payload.maxSpendUsd` are absent from the strict Gateway cron schema and cron payload type. The exact spend-cap behavior is not a bug reproduction because no current runtime enforcement contract exists.

## Solution Assessment

No for a schema-only pass-through. The maintainable fix needs a resource-limit contract covering validation, persistence, runtime precedence, spend enforcement or explicit non-support, CLI/help wording, docs, and regression tests.
