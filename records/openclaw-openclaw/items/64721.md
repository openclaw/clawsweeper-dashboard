## Summary

Keep open. Current main and v2026.6.6 only partially cover the report: `model` and `timeoutSeconds` are accepted, but `contextTokens` and `maxSpendUsd` are still absent from the strict cron payload, protocol, runtime, CLI, and persistence contracts, so the remaining work needs a cron API/runtime product decision.

## What This Changes

Accept and honor per-cron `payload.model`, `payload.timeoutSeconds`, `payload.contextTokens`, and `maxSpendUsd` through the agent cron tool, Gateway cron API, CLI, persistence, and runtime.

## Best Possible Solution

Define one canonical per-job cron budget/context contract, then update protocol schemas, runtime types, normalization, SQLite payload storage, isolated-run handling, agent tool schema, CLI add/edit, docs, and focused regression tests together.

## Reproduction Assessment

Yes. Source inspection on current main shows `contextTokens` and `maxSpendUsd` are absent from the strict cron payload/protocol/runtime contract while `cron.add` and `cron.update` validate against that contract.

## Solution Assessment

No, a schema-only patch would be too narrow. The maintainable fix is a coordinated cron API, runtime, storage, CLI, docs, and test update that states how context caps and spend caps are enforced.
