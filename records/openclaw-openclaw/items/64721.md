## Label Justifications

- `P2`: The issue affects scheduled agent job configuration and cost/context controls but is bounded to cron tooling rather than an emergency runtime outage.
- `impact:session-state`: The missing contextTokens path can leave scheduled agent runs using the wrong context budget/session state behavior.

## Summary

Keep open. Current main partially addresses the original model/timeout complaint, but the remaining per-job contextTokens and maxSpendUsd request is still not implemented in the cron tool, gateway protocol, cron payload types, normalization, CLI, or docs, so this needs maintainer API/product direction rather than cleanup closure.

## What This Changes

The issue asks OpenClaw’s cron management surfaces to preserve per-job model, timeout, context-token, and spend-cap fields for scheduled agentTurn jobs.

## Best Possible Solution

Define the canonical per-job cron budget contract, then update the gateway protocol, cron payload types, normalization, agent tool schema, CLI add/edit behavior, docs, and focused regression tests together.

## Reproduction Assessment

Yes, source-level for the remaining bug/feature gap: current schemas and cron payload types omit contextTokens and maxSpendUsd while the gateway protocol is strict. I did not run a live Gateway command because the source path is enough to show those fields are still unsupported on current main.

## Solution Assessment

No, adding fields only to the agent-facing tool schema would be too narrow because the strict gateway protocol, cron runtime types, normalization, CLI, and docs would still disagree. The maintainable fix is a single canonical cron payload/API decision with matching tests across those surfaces.
