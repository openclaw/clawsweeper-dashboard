## Summary

Keep open. Current main covers the originally reported `model` and `timeoutSeconds` cron payload fields, but `contextTokens` and `maxSpendUsd` are still absent from the cron payload type, strict Gateway schema, normalization/runtime contract, and CLI/tool surface; adding them needs a deliberate resource-limit design rather than a schema-only pass-through.

## What This Changes

Cron management should accept and preserve per-agent-turn model, timeout, context-token, and spend-cap controls from agent tool and related CLI surfaces.

## Best Possible Solution

Define an end-to-end cron agent-turn resource-limit contract: keep the shipped model/timeout path, add validated `contextTokens` with clear precedence, implement or split `maxSpendUsd` enforcement, and clarify CLI timeout flags with docs and tests.

## Reproduction Assessment

Yes for the remaining schema gap: sending `cron.add` or `cron.update` with nested `payload.contextTokens` or `payload.maxSpendUsd` hits the strict Gateway cron payload schema, which omits those fields. The exact spend-cap behavior cannot be reproduced as a bug because no current runtime enforcement contract exists.

## Solution Assessment

No for a schema-only pass-through. The maintainable solution needs a resource-limit contract that covers persistence, runtime precedence, actual spend enforcement, CLI wording, docs, and regression tests.
