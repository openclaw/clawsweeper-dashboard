## Summary

Keep open. Current main and the latest release include `model` and `timeoutSeconds` in the cron tool/protocol path, but `contextTokens` and `maxSpendUsd` are still absent from the strict cron payload contract, runtime payload type, storage codec, and CLI/docs, so the central cost/context control request remains unresolved.

## What This Changes

The issue requests cron add/update tool and CLI/API support for per-job `model`, `timeoutSeconds`, `contextTokens`, and `maxSpendUsd` payload fields.

## Best Possible Solution

Define a canonical per-job cron execution budget contract and wire it consistently through protocol schemas, runtime payload/storage, agent tool schema, CLI/docs, and regression tests.

## Reproduction Assessment

Yes, source-reproducible: current main's strict cron payload schemas/types omit `contextTokens` and `maxSpendUsd` while `cron.update` validates normalized patches through `validateCronUpdateParams`. I did not run a live gateway command in this read-only review.

## Solution Assessment

No, adding fields only to the model-facing cron tool schema would be too narrow. The maintainable solution is a single cron API/runtime decision that keeps gateway protocol, CLI, storage, docs, validation, and runtime execution in sync.
