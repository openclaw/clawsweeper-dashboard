## Summary

Keep this open: current main already covers the model and timeout portions, but `contextTokens` and `maxSpendUsd` are still not part of the cron payload/protocol/storage/run contract, so the remaining request is not safely closeable as implemented.

## What This Changes

The issue asks for the agent-facing cron tool and related cron edit surfaces to accept and preserve per-job model, timeout, context-token, and spend-cap fields.

## Best Possible Solution

Keep the existing model/timeout support, then define a first-class cron payload/API contract for per-job context and spend limits before updating schemas, CLI/docs, storage, and run enforcement.

## Solution Assessment

No, a narrow tool-schema-only fix is no longer the best solution. The remaining fields require a maintained cron payload contract with storage, CLI/docs, and run-time budget/context semantics, plus a separate decision on whether `cron edit --timeout` should warn or alias to `--timeout-seconds`.
