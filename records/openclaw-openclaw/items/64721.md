## Summary

Keep open. Current main has implemented the `model` and `timeoutSeconds` portions, but `contextTokens` and `maxSpendUsd` are still not part of the cron tool/protocol/runtime contract, and `maxSpendUsd` should not be accepted as a cosmetic schema field without real per-run spend enforcement.

## What This Changes

The issue requests cron add/update tool support for `payload.model`, `payload.timeoutSeconds`, `payload.contextTokens`, and `maxSpendUsd` so agents can manage those cron job fields without direct `jobs.json` edits.

## Best Possible Solution

Keep the existing `model` and `timeoutSeconds` implementation, add a deliberate end-to-end `contextTokens` contract for agent-turn cron jobs, and split `maxSpendUsd` into real per-run spend-guard work before exposing it as an accepted cron field.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to send `cron.add` or `cron.update` with `payload.contextTokens` or `payload.maxSpendUsd`: flattened fields are not recovered by the tool, and nested fields reach the strict Gateway cron payload schema where they are rejected before persistence.

## Solution Assessment

No, not as a schema-only pass-through. `contextTokens` can be a narrow contract once precedence is settled, but `maxSpendUsd` should be implemented only with enforceable spend semantics or split into a separate spend-guard item.
