Keep open. Current main and v2026.6.10 cover the `model` and `timeoutSeconds` cron pieces, but `contextTokens` and `maxSpendUsd` are still not a defined cron payload/protocol/persistence/runtime contract, so this is not safe to close as implemented.

## What This Changes

The issue asks cron add/update tooling and related CLI/API surfaces to accept and preserve per-job model, timeout, context-token, and spend-cap fields.

## Best Possible Solution

Define one canonical per-job cron context and spend-budget contract, then apply it through the tool schema, gateway protocol, cron payload types, SQLite persistence, runtime enforcement/accounting, CLI, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the cron tool/protocol/type/persistence paths include `model` and `timeoutSeconds` but do not define `contextTokens` or `maxSpendUsd`, so the remaining gap is source-reproducible without mutating the checkout.

## Solution Assessment

No, not as a cron-tool-schema-only change. The maintainable solution is a coordinated cron API/runtime contract so context and spend limits are validated, persisted, enforced, surfaced in CLI/docs, and covered by tests.
