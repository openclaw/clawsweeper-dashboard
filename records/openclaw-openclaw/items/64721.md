## Summary

Keep open. Current main still covers the `model` and `timeoutSeconds` parts of the report, but `contextTokens` and `maxSpendUsd` remain absent from the cron tool schema, Gateway protocol validation, exported cron payload types, normalization paths, docs, and payload-level isolated runtime behavior.

## What This Changes

The issue asks for cron add/update to accept and preserve `payload.model`, `payload.timeoutSeconds`, `payload.contextTokens`, and `maxSpendUsd` for agent-turn cron jobs.

## Best Possible Solution

Add a deliberate `contextTokens` contract for agent-turn cron jobs end to end: tool schema/descriptions, flat argument recovery, Gateway protocol schema, exported cron types, normalization, service patch behavior, isolated runtime precedence, docs, and regression tests. Treat `maxSpendUsd` as separate spend-guard work unless maintainers are ready to implement real per-run enforcement with tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to send a cron add/update payload containing `contextTokens` or `maxSpendUsd`: the tool does not advertise/recover those fields, and the strict Gateway cron payload schema rejects them before persistence.

## Solution Assessment

No, not as a schema-only acceptance change. `contextTokens` is a narrow end-to-end contract once precedence is chosen, but `maxSpendUsd` should not be accepted until it has real admission/enforcement behavior or is split into the broader spend-guard work.
