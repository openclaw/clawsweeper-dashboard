## Summary

Keep open. Current main and v2026.6.10 cover the cron `model` and `timeoutSeconds` pieces, but `contextTokens` and `maxSpendUsd` still are not cron payload, schema, persistence, CLI, or runtime-enforcement contracts; related cost/default issues overlap but do not replace this cron schema gap.

## What This Changes

Allow cron add/update tool and API payloads to carry per-job model, timeout, context-token, and spend-cap fields instead of stripping them.

## Best Possible Solution

Define a cron-owned per-job context and budget contract, then align the tool schema, gateway protocol, cron store, runtime enforcement, CLI/docs, and the broader budget tracker without regressing current model/timeout behavior.

## Reproduction Assessment

Yes, at source level. Current schemas and types accept `model` and `timeoutSeconds` but still omit `contextTokens` and `maxSpendUsd`; I did not run a live cron job because the read-only source path is sufficient for the remaining gap.

## Solution Assessment

No, not as a schema-only pass-through. The best fix must define context-token and spend-cap semantics across persistence, runtime enforcement, CLI/docs, and the existing gateway budget design rather than only accepting extra JSON fields.
