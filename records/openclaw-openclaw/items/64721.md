## Summary

Keep open. Current main still supports the previously added `model` and `timeoutSeconds` cron payload fields, but `contextTokens` and `maxSpendUsd` remain outside the cron tool, Gateway protocol, payload types, normalization hints, and isolated-run runtime contract.

## What This Changes

Allow cron add/update tool payloads to accept and preserve `model`, `timeoutSeconds`, `contextTokens`, and `maxSpendUsd` for agent-turn jobs.

## Best Possible Solution

Define an end-to-end agent-turn cron payload contract for `contextTokens`, and expose `maxSpendUsd` only with real per-run spend enforcement or an explicitly separate tracking design.

## Reproduction Assessment

Yes. Source inspection on current main shows `cron.add` or `cron.update` with nested `payload.contextTokens` or `payload.maxSpendUsd` reaches strict Gateway protocol validation, whose agent-turn payload schema omits those fields with `additionalProperties: false`.

## Solution Assessment

No, not as a schema-only pass-through. `contextTokens` needs deliberate runtime precedence wiring, and `maxSpendUsd` needs enforceable per-run spend semantics before the tool advertises it.
