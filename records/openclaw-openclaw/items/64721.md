## Summary

Keep open. Current main and the latest release already cover the originally reported model and timeoutSeconds paths, but contextTokens and maxSpendUsd still are not accepted cron payload fields or enforced runtime resource-limit contracts, and the latest CLI comment identifies a related timeout flag ambiguity that remains in the same surface.

## What This Changes

The issue asks for cron add/update tooling to accept and persist model, timeoutSeconds, contextTokens, and maxSpendUsd fields for agent-turn jobs.

## Best Possible Solution

Define an end-to-end cron agent-turn resource-limit contract that keeps the shipped model/timeoutSeconds support, deliberately adds or rejects contextTokens and maxSpendUsd with runtime semantics, and resolves the CLI timeout naming ambiguity with tests and docs.

## Reproduction Assessment

Yes. A source-level reproduction is to send cron.add or cron.update with nested payload.contextTokens or payload.maxSpendUsd on current main; the strict Gateway schema rejects those fields before persistence, and cron edit --timeout is parsed as the shared RPC timeout rather than payload.timeoutSeconds.

## Solution Assessment

No, not as a schema-only pass-through. The maintainable fix needs a resource-limit contract for spend enforcement and context-token precedence, plus an explicit CLI timeout disambiguation path.
