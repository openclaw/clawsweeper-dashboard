## Summary

Keep open. Current main and the latest release have covered the `model` and `timeoutSeconds` pieces, but `contextTokens` and `maxSpendUsd` still lack a cron payload/tool/protocol/persistence/runtime contract, so the remaining work needs maintainer product/API direction instead of cleanup closure.

## What This Changes

The issue requests that cron add/update tooling accept and preserve `model`, `timeoutSeconds`, `contextTokens`, and `maxSpendUsd` fields for scheduled agent runs.

## Best Possible Solution

Define one canonical per-job cron context and spend budget contract, then apply it consistently through the tool schema, gateway protocol, cron payload types, SQLite persistence, isolated-run enforcement/accounting, CLI, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows cron add/update schemas and payload contracts include `model` and `timeoutSeconds` but do not include `contextTokens` or `maxSpendUsd`, so the remaining gap is source-reproducible without mutating the checkout.

## Solution Assessment

No, not as a cron-tool-schema-only change. The maintainable fix is a coordinated cron API/runtime contract so context and spend limits are validated, persisted, enforced, surfaced in CLI/docs, and covered by tests.
