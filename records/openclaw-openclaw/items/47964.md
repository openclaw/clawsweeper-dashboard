## Summary

Keep open. Current main still leaves alias-shaped OpenAI Codex OAuth profiles unnormalized and still rejects stored OAuth credentials when a stale `api_key` profile mode override is present; open PR #75030 references this issue with closing syntax, so this issue should stay open until that PR, or an equivalent fix, lands.

## What This Changes

This issue requests compatibility or diagnostics for OpenAI Codex OAuth profiles that are present on disk but unusable because of legacy OAuth field names or stale auth-profile mode overrides.

## Best Possible Solution

Main should load known legacy OpenAI Codex OAuth aliases into canonical `access`/`refresh`/`expires`/`accountId` fields and produce token-redacted diagnostics when config mode masks a usable OAuth credential.

## Reproduction Assessment

Yes. A source-traced reproduction is an `openai-codex` OAuth profile containing only alias fields such as `access_token`, `refreshToken`, or `expires_at`; current main leaves those aliases untouched and downstream PI conversion rejects the profile because canonical `access`/`refresh`/`expires` are missing.

## Solution Assessment

Yes. Read-time normalization of known aliases plus a secret-free mode-mismatch diagnostic is the narrow maintainable fix, but the issue should remain open while PR #75030 is reviewed or replaced.
