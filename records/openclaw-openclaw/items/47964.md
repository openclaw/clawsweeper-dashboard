## Summary

Keep open. Current main still leaves the reported OpenAI Codex OAuth compatibility gap in place, and the linked open PR appears to be the active implementation path for both legacy alias normalization and stale mode-mismatch diagnostics.

## What This Changes

The issue asks whether OpenAI Codex OAuth auth profiles with older field names or stale config mode overrides should be normalized, migrated, or diagnosed instead of silently falling back.

## Best Possible Solution

Review and land PR #75030, or an equivalent narrow patch, with regression coverage for legacy OAuth aliases, stale mode masking, and token-redacted diagnostics; then let the linked issue close after the fix merges.

## Reproduction Assessment

Yes. A source-traced reproduction is an openai-codex OAuth profile containing only alias fields such as access_token/refreshToken/expires_at, which current main leaves unnormalized and PI conversion rejects; another is a valid stored OAuth profile paired with auth.profiles.<id>.mode set to api_key, which is filtered as mode_mismatch before the generic missing-auth error.

## Solution Assessment

Yes, the linked PR direction looks like the narrow maintainable path: normalize known OpenAI Codex OAuth aliases at auth-profile load and add a metadata-only diagnostic for config/store mode mismatch. Closing this issue before that PR or an equivalent fix lands would be premature.
