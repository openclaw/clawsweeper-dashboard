## Summary

Keep open: current main does not implement the base-GUID duplicate suppression this PR adds. The patch looks narrowly scoped with no blocking code-review findings, but the latest head still needs maintainer review because CI is not fully green and the live BlueBubbles ingress path for no-attachment updated-message events needs confirmation.

## What This Changes

The PR changes BlueBubbles inbound dedupe to pass the normalized message into the claim helper, suppress no-attachment updated-message duplicates when the base GUID is already committed, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land the narrowed BlueBubbles-owned dedupe change after required checks are green and maintainers confirm no-attachment updated-message replay events can reach the helper; if duplicates actually occur earlier or elsewhere, move the fix and coverage to that ingress boundary while preserving attachment-bearing updated-message processing.

## Reproduction Assessment

Yes for the helper-level behavior: finalize a `new-message` base GUID, then claim a no-attachment `updated-message` for the same GUID and current main will treat the suffixed key as claimable. End-to-end webhook reproduction is unclear because current `monitor.ts` filters no-attachment updated-message events before debouncing.

## Solution Assessment

Yes with a caveat: probing `hasRecent` on the base GUID before claiming the suffixed updated key is a narrow maintainable fix that preserves late media updates. The safer final check is confirming the live duplicate path reaches this helper; otherwise the same policy should be enforced at the actual ingress boundary.
