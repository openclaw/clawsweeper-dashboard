## Summary

Keep this PR open and request changes: it is a focused BlueBubbles dedupe attempt, but the proposed base-GUID short-circuit conflicts with the shipped attachment-update behavior that intentionally lets late `updated-message` attachment webhooks reach the agent. The branch also omits the required changelog entry for a user-visible BlueBubbles fix.

## What This Changes

The PR changes BlueBubbles inbound dedupe so `claimBlueBubblesInboundMessage` accepts a message object, treats committed-base `updated-message` events as duplicates, updates `processMessage` to pass the message, and adds inbound-dedupe unit coverage.

## Best Possible Solution

Preserve the shipped late-attachment path while narrowing dedupe to confirmed replay/no-media `updated-message` noise, or fix the missing group/chat context separately. Add regression coverage for both delayed attachment updates after base commit and the duplicate replay case, plus an Unreleased/Fixes changelog entry.

## Reproduction Assessment

Yes for the blocking review finding: process and finalize a base `new-message`, then deliver an attachment-bearing `updated-message` with the same GUID after the debounce window; the PR's base `hasRecent` branch returns `duplicate` before attachment processing. The original duplicate symptom is plausible from the added unit test, but no live trace or failing current-main test was provided.

## Solution Assessment

No. Collapsing all committed-base `updated-message` events is broader than the bug and regresses the current #67510/#65430 attachment-update contract; a narrower replay/no-media guard or chat-context fix is safer.
