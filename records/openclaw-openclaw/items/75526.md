## Summary

Keep open: current main still lacks the proposed base-GUID duplicate probe for BlueBubbles updated-message follow-ups, and the PR is the active implementation candidate. I found no concrete blocking correctness or security finding in the patch, but it still needs maintainer review of the live ingress semantics and completion of pending checks before merge.

## What This Changes

The PR changes BlueBubbles inbound dedupe to claim normalized message inputs, suppress no-attachment updated-message follow-ups when the committed base GUID is already recent, updates the processing call site, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land the narrowed BlueBubbles-owned dedupe change after required checks complete and maintainers confirm the live updated-message ingress path; if duplicates occur before this helper boundary, add or adjust boundary coverage so base-GUID replay suppression applies without dropping late media updates.

## Reproduction Assessment

Yes for the helper-level behavior: commit a new-message GUID, then claim a no-attachment updated-message for the same GUID and expect current main to use the suffixed key while the PR returns duplicate. End-to-end webhook reproduction is unclear because current ingress filters no-attachment updated-message events before processing.

## Solution Assessment

Yes, with a pre-merge caveat: probing `hasRecent` on the committed base GUID is a narrow maintainable fix that preserves attachment-bearing updated-message processing. The safer final check is confirming that the live duplicate follow-up actually reaches this helper boundary.
