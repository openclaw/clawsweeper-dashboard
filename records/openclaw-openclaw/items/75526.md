## Summary

Keep this PR open. Current main still has the old BlueBubbles claim API and the PR head is the active implementation candidate; the diff is narrow and appears consistent with the stated helper-level dedupe behavior, but it still needs normal maintainer/CI validation before merge.

## What This Changes

The PR changes BlueBubbles inbound dedupe to claim normalized message objects, short-circuit no-attachment `updated-message` follow-ups when the base GUID is already committed, preserves attachment-bearing updated-message processing, updates the monitor-processing call site, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land the narrowed BlueBubbles-owned dedupe change if CI and maintainer review confirm that replay suppression remains correct and late attachment-bearing updates still reach processing; otherwise adjust the live ingress or tests so the production path matches the intended dedupe behavior.

## Reproduction Assessment

Yes for the helper-level behavior: commit a new-message GUID, then claim a no-attachment updated-message for the same GUID and expect `duplicate`; claim an attachment-bearing updated-message for the same GUID and expect it to remain claimable. This review did not run the tests because the requested pass was read-only.

## Solution Assessment

Mostly yes, with one maintainer check needed: the base-GUID `hasRecent` probe is the narrowest helper-level change and preserves attachment-bearing updated-message keys, but the live webhook filter should be reconciled with the intended no-attachment updated-message scenario.
