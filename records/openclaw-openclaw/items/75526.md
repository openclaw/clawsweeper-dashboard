## Summary

Keep this PR open: current main still claims BlueBubbles updated-message follow-ups under the suffixed transient key, while the PR adds a narrow helper-level base-GUID check and regression coverage. I found no concrete code-review bug in the diff, but merge still needs maintainer review because the live webhook path currently filters no-attachment `updated-message` events before this helper and the exact head is not green in CI.

## What This Changes

The PR changes BlueBubbles inbound dedupe to pass the normalized message into the claim helper, suppress no-attachment `updated-message` claims when the committed base GUID is already recent, preserve attachment-bearing update claims, add regression tests, and add a changelog entry.

## Best Possible Solution

Land the BlueBubbles-owned helper change once the exact head is green and maintainers confirm the duplicate replay enters through this boundary; if no-attachment updates never reach this helper in production, move the same base-GUID suppression to the actual ingress boundary instead.

## Reproduction Assessment

Yes at helper level: current main can commit `g1` as a new message, then treat a no-attachment `updated-message` for `g1` as a fresh `g1:updated` claim. End-to-end webhook reproduction is unclear because `monitor.ts` currently drops no-attachment `updated-message` events before debouncing and processing.

## Solution Assessment

Yes with a caveat: checking `hasRecent` on the stable base GUID before claiming the suffixed updated key is narrow and preserves late attachment-bearing updates. The safer final check is confirming this helper is the boundary where duplicate production replays enter.
