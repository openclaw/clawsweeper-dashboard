## Summary

Keep open: current main does not include this behavior, and the PR's helper-level fix is coherent with the dedupe contract and added regression coverage. I found no blocking review findings, but exact-head CI was still running and maintainers should confirm the production path for no-attachment updated-message replays.

## What This Changes

The PR changes BlueBubbles inbound dedupe to pass the normalized message into the claim helper, check no-attachment updated-message follow-ups against the committed base GUID, preserve attachment-bearing updates, add regression tests, and add a changelog entry.

## Best Possible Solution

Land the BlueBubbles-owned helper change once exact-head CI is green and maintainers confirm duplicate replay traffic reaches this dedupe boundary; if the replay is produced earlier, move the same base-GUID suppression to the actual ingress path and keep the regression coverage focused on that boundary.

## Reproduction Assessment

Yes at helper level: current main can commit `g1` as a new message, then treat a no-attachment `updated-message` for `g1` as a fresh `g1:updated` claim. End-to-end webhook reproduction is unclear because `extensions/bluebubbles/src/monitor.ts` currently drops no-attachment `updated-message` events before processing.

## Solution Assessment

Yes, with a caveat: checking `hasRecent` on the stable base GUID before claiming the suffixed updated key is narrow and preserves late attachment-bearing updates. The safer final check is confirming this helper is the boundary where duplicate production replays enter.
