## Summary

Keep open: current main still keys BlueBubbles `updated-message` events as a separate suffixed dedupe key and passes only that key into the claim helper, while this PR adds a focused base-GUID duplicate check with regression coverage. I found no discrete blocking code-review bug in the patch, but exact-head CI still had one in-progress check during review and maintainers should confirm the production replay path reaches this helper.

## What This Changes

The PR changes BlueBubbles inbound dedupe to pass the normalized message into the claim helper, suppress no-attachment `updated-message` follow-ups when the base GUID is already committed, preserve attachment-bearing updates, add regression tests, and add a changelog entry.

## Best Possible Solution

Land the BlueBubbles-owned helper change once exact-head CI is green and maintainers confirm duplicate replay traffic reaches this dedupe boundary; if the replay enters earlier, move the same base-GUID suppression to that ingress path instead.

## Reproduction Assessment

Yes at helper level: current main can commit `g1` as a new message, then treat a no-attachment `updated-message` for `g1` as a fresh `g1:updated` claim. End-to-end webhook reproduction is unclear because `monitor.ts` currently drops no-attachment `updated-message` events before processing.

## Solution Assessment

Yes with a caveat: checking `hasRecent` on the stable base GUID before claiming the suffixed updated key is narrow and preserves late attachment-bearing updates. The safer final check is confirming this helper is the boundary where duplicate production replays enter.
