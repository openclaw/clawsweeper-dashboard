## Summary

Keep open for maintainer review because the patch conflicts with the shipped BlueBubbles attachment-update behavior: current main intentionally lets attachment-bearing `updated-message` webhooks use a separate dedupe key so late-indexed media can be processed, while this PR drops them once the base GUID is committed. The PR also lacks the required changelog entry for a user-visible BlueBubbles fix.

## What This Changes

This PR changes BlueBubbles inbound dedupe to pass the normalized message into `claimBlueBubblesInboundMessage`, treats committed-base `updated-message` follow-ups as duplicates, and updates focused inbound-dedupe coverage.

## Best Possible Solution

Preserve the shipped attachment-bearing `updated-message` path, then fix duplicate follow-up replies with a narrower guard that distinguishes true replay noise from late media updates, or address any missing group/chat context separately. Add regression coverage for both delayed attachment updates after base commit and the duplicate replay case, plus an Unreleased/Fixes changelog entry.

## Reproduction Assessment

Yes for the introduced regression: process and finalize a base `new-message`, then deliver an attachment-bearing `updated-message` with the same GUID after the debounce window; the PR's base-GUID `hasRecent` branch returns `duplicate` before attachment handling. The original duplicate symptom is plausible, but not high-confidence without a live trace or failing current-main test.

## Solution Assessment

No. Collapsing all committed-base `updated-message` follow-ups is broader than the bug and regresses the current attachment-update contract; a narrower replay/no-media guard or chat-context fix is safer.
