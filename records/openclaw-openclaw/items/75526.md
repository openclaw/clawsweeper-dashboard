## Summary

Keep open. Current main still claims BlueBubbles `updated-message` events with the suffixed dedupe key, while this PR adds a plausible helper-level base-GUID duplicate check with focused regression coverage; the remaining action is normal maintainer and CI validation, not cleanup closure.

## What This Changes

This PR changes BlueBubbles inbound dedupe to claim normalized messages, short-circuit no-attachment `updated-message` follow-ups when the base GUID is already committed, update the processing call site, add regression tests, and add a changelog entry.

## Best Possible Solution

Land the narrowed BlueBubbles-owned dedupe change if targeted tests and required CI pass and maintainers confirm the live ingress semantics; otherwise adjust the webhook path or tests so the production behavior matches the intended replay suppression without dropping late attachment updates.

## Reproduction Assessment

Yes for the helper-level behavior: commit a new-message GUID, then claim a no-attachment updated-message for the same GUID and expect `duplicate`; claim an attachment-bearing updated-message for the same GUID and expect it to remain claimable. This review did not run tests because the requested pass was read-only.

## Solution Assessment

Mostly yes: using `hasRecent` on the stable base GUID is a narrow maintainable fix and keeps attachment-bearing updates on the suffixed key. The safer pre-merge check is to reconcile that helper behavior with the live webhook filter that currently drops no-attachment updated-message events before dedupe.
