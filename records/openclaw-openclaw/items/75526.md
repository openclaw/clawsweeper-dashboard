## Summary

Keep open: current main still dedupes BlueBubbles updated-message follow-ups on the transient `:updated` key, while this narrow PR adds focused base-GUID suppression, tests, and changelog coverage; it still needs conflict resolution and maintainer confirmation of the replay boundary before merge.

## What This Changes

The PR changes BlueBubbles inbound dedupe to pass the normalized message into the claim helper, suppress no-attachment `updated-message` follow-ups when the base GUID is already committed, preserve attachment-bearing updates, add regression tests, and add a changelog entry.

## Best Possible Solution

Resolve the current-main conflicts, then land the BlueBubbles-owned helper/test/changelog fix if maintainers confirm duplicate replay traffic reaches `processMessage`; otherwise move the same base-GUID suppression to the actual ingress boundary.

## Reproduction Assessment

Yes at helper level: current main can commit `g1` as a `new-message`, then treat a no-attachment `updated-message` for `g1` as a fresh `g1:updated` claim. End-to-end webhook reproduction is unclear because `extensions/bluebubbles/src/monitor.ts` filters no-attachment `updated-message` events before `processMessage`.

## Solution Assessment

Yes, with a caveat: checking committed recency on the stable base GUID before claiming the suffixed updated key is narrow and preserves late attachment-bearing updates. If duplicate replay traffic is produced earlier than `processMessage`, the same rule should move to that actual boundary.
