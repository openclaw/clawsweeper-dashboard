## Summary

Keep open. Current main still dedupes every BlueBubbles `updated-message` on the suffixed key, while this PR adds a targeted base-GUID probe and regression coverage; I did not find a discrete correctness or security bug in the diff, but the head still needs maintainer/CI validation and a production-path sanity check.

## What This Changes

The PR changes BlueBubbles inbound dedupe to claim normalized message inputs, suppress no-attachment `updated-message` follow-ups when the base GUID is already committed, update the processing call site, add regression tests, and add a changelog entry.

## Best Possible Solution

Land the narrowed BlueBubbles-owned dedupe change once required checks pass and maintainers confirm the live ingress semantics; if the production path differs from the helper test path, adjust the webhook/process boundary so base-GUID replay suppression applies where the duplicate is actually observed without dropping late media updates.

## Reproduction Assessment

Yes for the helper-level behavior: commit a `new-message` GUID, then claim a no-attachment `updated-message` for the same GUID and expect `duplicate`; claim an attachment-bearing update for the same GUID and expect it to remain claimable. I did not run tests because this review was read-only.

## Solution Assessment

Mostly yes: using `hasRecent` on the stable base GUID is narrow and keeps attachment-bearing updates on the suffixed key. The safer pre-merge check is to reconcile that helper behavior with the current webhook filter that drops no-attachment `updated-message` events before processing.
