## Summary

This focused BlueBubbles fix should stay open: the base-GUID dedupe direction matches the source-level bug, but the PR still has a blocking test helper argument typo and lacks required real behavior proof from a real OpenClaw/BlueBubbles setup.

## What This Changes

The PR changes BlueBubbles inbound dedupe to suppress no-attachment `updated-message` follow-ups when the base GUID was already committed, preserves attachment-bearing updates, updates the process-message call site, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Fix the helper-call typo, add redacted real BlueBubbles/OpenClaw before/after proof or a maintainer proof override, then land the BlueBubbles-owned base-GUID dedupe if maintainers accept the helper-level replay boundary.

## Reproduction Assessment

Yes at source level: current main commits the base `new-message` key, then maps a no-attachment `updated-message` for the same GUID to `base:updated`, bypassing the committed base key. I did not run a live BlueBubbles webhook smoke in this read-only review.

## Solution Assessment

No for the PR as submitted: the dedupe approach is narrow and consistent with the helper contract, but the test typo and missing real behavior proof block merge. The safer path is to fix the assertion and provide redacted real setup evidence or maintainer override.
