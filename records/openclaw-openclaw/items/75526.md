## Summary

Keep open for normal maintainer review: current main does not include this BlueBubbles dedupe change, and the PR appears narrowly scoped with no discrete blocking code or security finding from the diff review. The remaining blockers are validation and maintainer confirmation that the production replay path needing this helper-level suppression can actually reach `processMessage`.

## What This Changes

The PR changes BlueBubbles inbound dedupe to pass the normalized message into the claim helper, suppress no-attachment `updated-message` claims when the base GUID is already committed, add regression coverage, and add a changelog entry.

## Best Possible Solution

Land the BlueBubbles-owned helper change after the exact head is green and maintainers confirm the replay path; if those no-attachment updates never reach this helper in production, apply the same base-GUID policy at the real ingress boundary instead.

## Reproduction Assessment

Yes for the helper-level behavior: current main can commit `g1` as a `new-message`, then treat a no-attachment `updated-message` for `g1` as a fresh `g1:updated` claim. End-to-end webhook reproduction is unclear because current `monitor.ts` drops no-attachment `updated-message` events before debouncing and processing.

## Solution Assessment

Yes with a caveat: checking `hasRecent` on the stable base GUID before claiming the suffixed updated key is narrow and preserves late attachment-bearing updates. The safer final check is verifying that this helper is the boundary where the duplicate production replay enters.
