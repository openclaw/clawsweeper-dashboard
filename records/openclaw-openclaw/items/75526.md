## Summary

## What This Changes

The PR changes BlueBubbles inbound dedupe to use the normalized message, suppress no-attachment updated-message claims when the base GUID is already committed, preserve attachment-bearing updates, add regression coverage, and add a changelog entry.

## Best Possible Solution

Fix the regression-test argument error, then land the BlueBubbles-owned base-GUID dedupe if maintainers accept `processMessage` as the correct replay boundary.

## Reproduction Assessment

Yes at helper level from source: current main commits the base GUID for a `new-message`, then maps a later `updated-message` for the same GUID to `base:updated`. End-to-end webhook reproduction remains unclear because `monitor.ts` filters no-attachment updated-message events before `processMessage`.

## Solution Assessment

No for the PR as submitted: the dedupe approach is narrow and preserves attachment-bearing updates, but the new test has a definite helper-call typo. Fixing that typo is the safer path before normal maintainer review and CI.
