## What This Changes

The PR changes BlueBubbles inbound dedupe to claim normalized message inputs, suppress no-attachment updated-message follow-ups when the base GUID is already committed, updates the processing call site, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land the narrowed BlueBubbles-owned dedupe change once required checks pass and maintainers confirm the live updated-message ingress semantics; if the duplicate is observed before the helper boundary, add or adjust webhook/debounce boundary coverage so base-GUID replay suppression applies without dropping late media updates.

## Solution Assessment

Yes, with a pre-merge caveat: using hasRecent on the committed base GUID is a narrow maintainable fix that preserves attachment-bearing updated-message processing. The safer final check is confirming that this helper boundary is where the live duplicate follow-up actually reaches production.
