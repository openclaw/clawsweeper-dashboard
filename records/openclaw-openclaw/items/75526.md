## Summary

Keep open: current main still has the BlueBubbles updated-message dedupe gap, and this PR is a plausible focused fix with no concrete correctness finding from this review; it still needs normal maintainer merge/replay-boundary review before landing.

## What This Changes

The PR changes BlueBubbles inbound dedupe to pass normalized message context into the claim helper, suppress no-attachment updated-message follow-ups when the committed base GUID is recent, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land the BlueBubbles-owned helper/test/changelog fix after maintainer confirmation that replay traffic reaches `processMessage`; if the duplicate is produced earlier, move the same base-GUID suppression to the actual ingress boundary.

## Reproduction Assessment

Yes at helper level: current main can commit `g1` as a new-message, then treat a no-attachment updated-message for `g1` as a fresh `g1:updated` claim. End-to-end direct webhook reproduction is unclear because `extensions/bluebubbles/src/monitor.ts` filters no-attachment updated-message events before `processMessage`.

## Solution Assessment

Yes, with a caveat: checking committed recency on the stable base GUID before claiming the suffixed updated key is narrow and preserves late attachment-bearing updates. If duplicate replay traffic is generated before this helper, the same rule should move to that boundary.
