## Summary

Keep open: the proposed BlueBubbles dedupe change is broader than the reported duplicate-follow-up problem and would drop the shipped late attachment-update path before media processing can run. The branch also lacks the required user-facing changelog entry for a BlueBubbles bug fix.

## What This Changes

This PR changes BlueBubbles inbound dedupe so updated-message follow-ups are checked against the committed base GUID, passes the normalized message into the claim helper, and updates focused inbound-dedupe tests.

## Best Possible Solution

Preserve the shipped attachment-bearing `updated-message` path, then fix any confirmed duplicate replay with a narrower guard that only drops true replay noise. Add regression coverage for both the duplicate case and delayed attachment updates, plus an Unreleased/Fixes changelog entry.

## Reproduction Assessment

Yes for the regression introduced by the PR: finalize a base `new-message`, then deliver an attachment-bearing `updated-message` with the same GUID; the PR's base-GUID check returns `duplicate` before attachment handling. The original duplicate symptom is not yet high-confidence without a trace or failing current-main test.

## Solution Assessment

No. A base-GUID short-circuit for all committed `updated-message` events is not the narrowest maintainable fix because it collapses legitimate late media updates; a replay/no-media-specific guard is safer.
