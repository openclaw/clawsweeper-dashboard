## What This Changes

The PR changes BlueBubbles inbound dedupe to claim normalized message objects, treat no-attachment updated-message follow-ups as duplicates when the base GUID is already committed, preserve attachment-bearing updated-message processing, update the monitor-processing call site, and add tests plus a changelog entry.

## Reproduction Assessment

Yes for the helper-level behavior: finalize a new-message with GUID `g1`, then claim a no-attachment updated-message for `g1` and expect `duplicate`, while an attachment-bearing updated-message for `g1` remains claimable. The PR adds those unit scenarios, but this read-only pass did not execute them.

## Solution Assessment

Yes. The PR's current approach is narrower than collapsing all updated-message events to the base GUID because it only short-circuits no-attachment follow-ups and leaves attachment-bearing updates on the suffixed key needed for late media ingestion.
