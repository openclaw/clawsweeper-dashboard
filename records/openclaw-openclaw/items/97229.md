## What This Changes

The PR makes inbound media-note path normalization MIME-aware so unmanaged binary documents render a guarded absolute path while managed media types keep `media://inbound/` references, with regression tests.

## Best Possible Solution

Land only after redacted real-run proof shows an unmanaged inbound binary attachment becomes usable and maintainers explicitly accept the scoped absolute-path visibility policy; if that policy is rejected, pursue a binary-aware resolver/tool path instead.

## Reproduction Assessment

Yes, at source level: current main rewrites managed inbound absolute paths to `media://inbound/<id>` before prompt rendering, so an `application/zip` path is not visible as a filesystem path. I did not establish a live channel reproduction in this read-only review.

## Solution Assessment

Yes, likely for the narrow bug: the loss happens in the shared media-note normalization layer, and the PR preserves managed-consumer URI behavior while restoring paths only for unmanaged binary documents. A binary-aware resolver would be the safer alternative if maintainers reject prompt-visible paths.
