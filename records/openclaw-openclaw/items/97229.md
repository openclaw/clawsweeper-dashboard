## Summary

Keep open: the PR targets a real source-reproducible binary attachment usability gap, but the MIME gate is broader than the stated binary-document fix and contributor real behavior proof is still missing.

## What This Changes

The PR makes inbound media-note path normalization MIME-aware so some managed inbound attachments render a filesystem path while image, audio, video, and PDF attachments keep `media://inbound/` references.

## Best Possible Solution

Narrow the path fallback to unsupported binary/no-reader document types, keep text-like and managed-consumer attachments on media URIs, then merge only after redacted real-run proof and maintainer acceptance of the scoped path-visibility policy.

## Reproduction Assessment

Yes, source-level: current main rewrites all managed inbound absolute paths to media://inbound/<id> before prompt rendering, so zip/octet-stream attachments do not expose a filesystem path for shell workflows. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No, not yet: the fix is in the right shared normalizer, but the predicate is too broad because text-like documents also switch to absolute paths. The safer shape is a positive unsupported-binary fallback while preserving URI behavior for text-like and managed-consumer media.
