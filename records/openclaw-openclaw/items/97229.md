## Summary

Keep open: the patch is a focused, plausible bug fix for unmanaged inbound binary documents, but it still needs real behavior proof and maintainer acceptance of the scoped prompt-visible absolute-path compatibility/privacy change before merge.

## What This Changes

The PR makes inbound media-note rendering MIME-aware so zip/octet-stream style inbound documents render a guarded absolute media-store path while images, audio, video, PDFs, and unknown types keep `media://inbound/` references, with regression tests.

## Best Possible Solution

Land this only after redacted real-run proof shows an unmanaged inbound binary attachment becomes usable and maintainers explicitly accept the scoped absolute-path visibility policy while preserving URI behavior for managed media types.

## Reproduction Assessment

Yes, at source level: current main rewrites any first-level managed inbound absolute path to `media://inbound/<id>` before prompt rendering, so an `application/zip` path is not visible as a filesystem path. I did not run a live channel scenario in this read-only review.

## Solution Assessment

Yes, likely for the narrow bug: the loss happens in the shared media-note normalization layer, and the PR preserves managed-consumer URI behavior while restoring a path only for unmanaged binary documents. A binary-aware `media://` resolver would be a broader product direction if maintainers reject prompt-visible paths.
