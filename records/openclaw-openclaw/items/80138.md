## Summary

Keep open: the PR targets a real Codex auth-routing bug and has sufficient terminal proof, but the latest head still introduces blocking route-repair regressions by using `agents.defaults.models` as unconditional runtime-policy storage and by overwriting explicit per-model runtime policy.

## What This Changes

The PR rewrites legacy `openai-codex/*` route repair to add Codex model runtime policy, preserve copied model metadata, and update doctor/runtime docs plus changelog.

## Best Possible Solution

Keep the Codex auth-preserving repair, but avoid creating visibility allowlists as hidden runtime state and preserve explicit per-model runtime policy before maintainer landing review.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows the new helper can create `agents.defaults.models`, and current main treats any default model-map entry as configured visibility that disables allow-any selection. I did not run a live repair command because this review is read-only.

## Solution Assessment

No. Preserving Codex auth intent is the right direction, but storing it by unconditionally materializing `agents.defaults.models` and overwriting explicit per-model runtime policy is not the narrowest maintainable fix.

  - body: When a legacy agent model ref is repaired, this helper creates `agents.defaults.models` entries even if the user had no model map.   - confidence: 0.9

  - body: This overwrites any existing per-model runtime id with `codex` unless it was already Codex. If a canonical `openai/gpt-5.5` entry explicitly sets `agentRuntime.id: "pi"`, repairing a matching legacy ref silently flips the user's configured route instead of preserving explicit policy.
