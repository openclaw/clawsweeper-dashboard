## Summary

Keep open. Current main still has the narrow memory embedding retry classifier and no exhausted multi-item batch split fallback, and open PR #44167 explicitly closes this issue, with #75317 overlapping newer retry work. Maintainer cleanup should review, reconcile, land, or replace the linked PR work rather than close this issue now.

## What This Changes

Request retrying transient memory embedding transport failures and recursively splitting exhausted multi-item embedding batches during memory reindex.

## Best Possible Solution

Land or replace a focused memory-core fix that treats narrow transient embedding transport errors as retryable, reuses the existing backoff loop, recursively splits exhausted multi-item text and structured batches while preserving order, and adds targeted regression tests. Keep broader rollback/cache preservation and live-query transport behavior in their own linked work unless the chosen PR intentionally covers those surfaces too.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is available by mocking embedBatch to fail once with TypeError("fetch failed") before succeeding, and to fail multi-item batches until singleton fallback; current main’s classifier and wrappers show those paths still throw instead of retrying or splitting.

## Solution Assessment

Yes. The narrow retry-classifier plus exhausted multi-item split fallback is the smallest maintainable fix for this issue; provider-specific branching, config/schema changes, and timeout changes are not needed for the reported batch reindex behavior.
