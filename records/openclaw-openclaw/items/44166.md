## Summary

Keep open. Current main now retries the reported socket/fetch-style embedding transport errors, but the exhausted multi-item batch split fallback is still absent and open PR #44167 still explicitly targets this issue, with #75317 overlapping broader retry work.

## What This Changes

Request retrying transient embedding transport errors during memory batch reindex and recursively splitting exhausted multi-item batches without provider-specific or config changes.

## Best Possible Solution

Land or replace a focused memory-core fix that preserves the merged transport retry classifier and adds exhausted multi-item split fallback for both text and structured batch wrappers, with targeted regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level unit reproduction remains: mock a provider so multi-item text or structured batches keep throwing `TypeError("fetch failed")` after retries, and current main still propagates the exhausted error instead of splitting the batch.

## Solution Assessment

Yes, the requested narrow retry-plus-split fallback is the best shape for this issue. Current main has only the retry-classifier half, so the remaining maintainable fix is the missing exhausted-batch split path rather than provider branching or new config.
