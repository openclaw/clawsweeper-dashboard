## Summary

Keep open: current main and v2026.5.12 already retry the named transport errors, but exhausted multi-item text and structured embedding batches still throw instead of splitting; the open closing PR remains the implementation path.

## What This Changes

The issue requests retrying transient memory embedding transport failures and recursively splitting multi-item batches after the retry budget is exhausted.

## Best Possible Solution

Land or replace the focused memory-core fix that adds split-after-retry-exhaustion fallback for both text and structured embedding batches while preserving current retry classification and tests.

## Reproduction Assessment

Yes, at source level. A mocked provider that keeps failing multi-item `embedBatch` or `embedBatchInputs` with `TypeError("fetch failed")` after retries but succeeds for singletons follows the current retry-then-throw path because no split fallback is invoked.

## Solution Assessment

Yes for the requested direction. Retry-plus-split fallback in the memory-core batch wrappers is the narrowest maintainable fix; provider-specific branching, config knobs, and timeout changes are not needed.
