## Summary

Keep open. Current main covers the transport-error retry-classifier part of the report, but exhausted multi-item embedding batches still throw instead of splitting, and the open PR at https://github.com/openclaw/openclaw/pull/44167 remains the direct closing fix path.

## What This Changes

The issue requests that memory reindex embedding batches retry transient transport errors and recursively split multi-item batches after retry exhaustion.

## Best Possible Solution

Land or replace the focused memory-core change that keeps transport retry classification and adds exhausted multi-item split fallback for both text and structured embedding batches with regression coverage.

## Reproduction Assessment

Yes, at source level. Mocking a provider so multi-item batches keep throwing `TypeError("fetch failed")` while single-item batches succeed would exercise the current path where retry exhaustion throws instead of splitting.

## Solution Assessment

Yes. The retry-plus-split fallback is the narrowest maintainable fix in memory-core; provider branching, config knobs, and timeout changes are not needed.
