## Summary

Keep open. Current main and v2026.5.12 retry the reported transport-shaped embedding errors, but exhausted multi-item text and structured embedding batches still throw instead of splitting, and an open closing PR already owns the implementation path.

## What This Changes

The issue requests that memory reindex retry transient embedding transport errors and recursively split exhausted multi-item batches before failing.

## Best Possible Solution

Land or replace the focused memory-core fix that preserves current retry classification while adding split-after-retry-exhaustion fallback for both text and structured batches with regression coverage and redacted real indexing proof.

## Reproduction Assessment

Yes, at source level. Mocking a provider so a multi-item embedBatch or embedBatchInputs keeps throwing TypeError("fetch failed") after retries while singleton calls succeed exercises the current retry-then-throw path because no split fallback is invoked.

## Solution Assessment

Yes for the requested direction. Retry-plus-split fallback in memory-core batch wrappers is the narrowest maintainable fix; provider-specific branching, config knobs, and timeout constant changes are not needed.
