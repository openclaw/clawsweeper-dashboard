## Summary

Keep open. Current main now retries the reported fetch/socket-style embedding transport errors, but exhausted multi-item batches still throw instead of splitting, and the open closing PR at https://github.com/openclaw/openclaw/pull/44167 remains the direct fix path.

## What This Changes

The issue requests retrying transient embedding transport failures during memory reindex and recursively splitting exhausted multi-item batches instead of aborting the whole reindex.

## Best Possible Solution

Land or replace the focused memory-core fix that keeps the broadened retry classifier and adds exhausted multi-item split fallback for both text and structured embedding batches with regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level unit reproduction is to mock a provider so multi-item text or structured batches keep throwing TypeError("fetch failed") after retries while single-item batches succeed; current main retries and then propagates the exhausted error instead of splitting.

## Solution Assessment

Yes. The requested retry-plus-split fallback is the narrowest maintainable repair in memory-core; provider-specific branching, config knobs, or timeout changes are not needed.
