## Summary

Keep open: current main and v2026.5.12 retry the named transport errors, but exhausted multi-item text and structured embedding batches still throw instead of splitting; https://github.com/openclaw/openclaw/pull/44167 remains the linked implementation path.

## What This Changes

Memory reindex should retry transient embedding transport failures and split exhausted multi-item embedding batches instead of aborting the whole run.

## Best Possible Solution

Land, rebase, or replace the focused memory-core fix that adds split-after-retry-exhaustion fallback for both text and structured embedding batches while preserving current retry classification and avoiding config changes.

## Reproduction Assessment

Yes, at source level. A mocked provider that keeps failing multi-item `embedBatch` or `embedBatchInputs` with `TypeError("fetch failed")` after retries but succeeds for singletons follows the current retry-then-throw path because no split fallback is invoked.

## Solution Assessment

Yes for the requested direction. Retry-plus-split fallback in the memory-core batch wrappers is the narrowest maintainable fix; provider-specific branching, config knobs, and timeout changes are not needed.
