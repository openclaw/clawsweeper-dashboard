## Summary

Keep open. Current main and v2026.5.12 already retry the reported transport-shaped embedding errors, but exhausted multi-item text and structured embedding batches still throw instead of splitting, and the open closing PR at https://github.com/openclaw/openclaw/pull/44167 remains the implementation path.

## What This Changes

The issue requests retrying transient embedding transport failures and recursively splitting exhausted multi-item embedding batches during memory reindex.

## Best Possible Solution

Land or replace the focused memory-core fix that preserves current retry classification while adding split-after-retry-exhaustion fallback for both text and structured batches with regression coverage and redacted real indexing proof.

## Reproduction Assessment

Yes, at source level. Mocking a provider so a multi-item `embedBatch` or `embedBatchInputs` keeps throwing `TypeError("fetch failed")` after retries while singleton calls succeed exercises the current retry-then-throw path because no split fallback is invoked.

## Solution Assessment

Yes for the requested direction. Retry-plus-split fallback in the memory-core batch wrappers is the narrowest maintainable fix; provider-specific branching, config knobs, and timeout constant changes are not needed.
