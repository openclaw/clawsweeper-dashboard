## Summary

Keep open. Current main and the latest release now retry the reported transport-shaped embedding errors, but exhausted multi-item text and structured embedding batches still throw instead of splitting, and the direct closing PR remains open.

## What This Changes

Memory reindex embedding batches should retry transient transport errors and split multi-item batches after retry exhaustion instead of aborting the whole run.

## Best Possible Solution

Land or replace the focused memory-core fix that preserves current transport and timeout retry classification while adding exhausted multi-item split fallback for both text and structured embedding batches with regression coverage and real behavior proof.

## Reproduction Assessment

Yes, at source level. Mocking a provider so a multi-item batch keeps throwing TypeError("fetch failed") after retries while single-item calls succeed exercises the current retry-then-throw path because no split fallback is invoked.

## Solution Assessment

Yes. The retry-plus-split fallback is the narrowest maintainable direction in memory-core; provider branching, config knobs, and timeout constant changes are not needed.
