## Summary

Keep open. Current main and the latest release retry the reported transport-shaped embedding errors, but exhausted multi-item text and structured embedding batches still throw instead of splitting, and the direct closing PR remains open.

## What This Changes

Memory reindex should retry transient embedding transport failures and split exhausted multi-item embedding batches instead of aborting the whole sync.

## Best Possible Solution

Land or replace the focused memory-core fix that preserves transport retry classification and adds exhausted multi-item split fallback for both text and structured embedding batches with regression coverage.

## Reproduction Assessment

Yes, at source level. Mocking a provider so multi-item batches keep throwing TypeError("fetch failed") after retries while single-item calls succeed exercises the current retry-then-throw path because no split fallback is invoked.

## Solution Assessment

Yes. The retry-plus-split fallback is the narrowest maintainable direction in memory-core; provider branching, config knobs, and timeout constant changes are not needed.
