## Summary

Keep open. Current main and v2026.5.12 retry the reported transport-shaped embedding errors, but exhausted multi-item text and structured embedding batches still throw instead of splitting, and https://github.com/openclaw/openclaw/pull/44167 remains open and conflicting.

## What This Changes

The issue asks memory reindex batch embedding to retry transient transport failures and recursively split exhausted multi-item batches rather than aborting the sync.

## Best Possible Solution

Land or replace the focused memory-core fix that preserves current transport and timeout retry classification while adding exhausted multi-item split fallback for both text and structured embedding batches with regression coverage and real indexing proof.

## Reproduction Assessment

Yes, at source level. Mocking a provider so a multi-item batch keeps throwing TypeError("fetch failed") after retries while single-item calls succeed exercises the current retry-then-throw path because no split fallback is invoked.

## Solution Assessment

Yes. Retry-plus-split fallback in memory-core is the narrowest maintainable direction; provider branching, config knobs, and timeout constant changes are not needed.
