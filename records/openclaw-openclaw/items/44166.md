## Summary

Keep open. Current main and the latest release include the transport-error retry classifier, but exhausted multi-item embedding batches still throw instead of splitting, and the open closing PR remains the direct implementation path.

## What This Changes

The issue asks memory reindex embedding batches to retry transient transport errors and recursively split multi-item batches after retry exhaustion.

## Best Possible Solution

Land or replace the focused memory-core fix that preserves current retry classification and adds exhausted multi-item split fallback for both text and structured embedding batches with regression coverage.

## Reproduction Assessment

Yes, at source level. A mocked provider that keeps throwing `TypeError("fetch failed")` for multi-item batches after retries while succeeding for single-item calls will hit the current wrappers' retry-then-throw path because current main never splits the batch.

## Solution Assessment

Yes, the retry-plus-split direction is the narrow maintainable fix in memory-core. Provider branching, config knobs, and timeout changes are not needed; the linked PR should be reviewed, updated, and landed or replaced.
