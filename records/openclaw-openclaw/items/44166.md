## Summary

Keep open: current main and the latest release retry the reported transport errors, but exhausted multi-item text and structured embedding batches still throw instead of splitting, and the open closing PR remains the implementation path.

## What This Changes

The issue asks memory reindex batch embedding to retry transient transport errors and recursively split failed multi-item batches after retries are exhausted.

## Best Possible Solution

Refresh, land, or replace the focused memory-core fix that preserves timeout and AbortSignal behavior while adding split-after-retry-exhaustion fallback for text and structured batch paths with regression coverage.

## Reproduction Assessment

Yes, at source level. A mocked provider that keeps failing multi-item embedBatch or embedBatchInputs calls with TypeError("fetch failed") after retries, while singleton calls succeed, still reaches the retry-loop throw because no split fallback is called.

## Solution Assessment

Yes for the requested direction. The narrow maintainable fix belongs in the memory-core batch wrappers, preserving runEmbeddingOperationWithTimeout and AbortSignal while adding recursive split after retry exhaustion.
