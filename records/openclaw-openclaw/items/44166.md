## Summary

Keep open: current main and v2026.5.19 retry the named transport errors, but exhausted multi-item text and structured embedding batches still throw instead of splitting. The closing implementation path at https://github.com/openclaw/openclaw/pull/44167 is still open and conflicting, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Make memory-core batch embedding retries handle transient transport failures by retrying and recursively splitting exhausted multi-item batches instead of aborting the whole reindex.

## Best Possible Solution

Refresh, land, or replace the focused memory-core fix that preserves current timeout and AbortSignal behavior while adding split-after-retry-exhaustion fallback for both text and structured batches with regression coverage.

## Reproduction Assessment

Yes, at source level. A mocked provider that keeps failing multi-item embedBatch or embedBatchInputs calls with TypeError("fetch failed") after retries, while singleton calls succeed, still follows the current retry-then-throw path because no split fallback is invoked.

## Solution Assessment

Yes for the requested direction. Retry-plus-split fallback in the memory-core batch wrappers is the narrow maintainable solution, provided it preserves the existing timeout and AbortSignal behavior.
