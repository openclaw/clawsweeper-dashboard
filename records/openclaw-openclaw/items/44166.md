## Summary

Keep open. Current main and v2026.5.7 retry the reported transport-shaped embedding errors, but exhausted multi-item text and structured embedding batches still throw instead of splitting, and the direct closing PR remains open.

## What This Changes

Request retrying transient embedding transport failures during memory reindex and splitting exhausted multi-item embedding batches instead of aborting the whole sync.

## Best Possible Solution

Land or replace a focused memory-core change that preserves current transport and timeout retry classification, adds exhausted multi-item split fallback for text and structured batches, and includes regression coverage.

## Reproduction Assessment

Yes, at source level. Mocking a provider so multi-item batches keep throwing TypeError("fetch failed") after retries while single-item calls succeed exercises the current retry-then-throw path because no split fallback is invoked.

## Solution Assessment

Yes. The retry-plus-split fallback is the narrowest maintainable direction in memory-core; provider branching, config knobs, and timeout constant changes are not needed.
