## Summary

Keep open: current main and v2026.5.12 retry the named transport-shaped embedding errors, but exhausted multi-item text and structured embedding batches still throw instead of splitting, and the open closing PR remains the implementation path.

## What This Changes

Memory reindex should retry transient embedding transport failures and split exhausted multi-item embedding batches before aborting.

## Best Possible Solution

Land or replace the focused memory-core fix that adds split-after-retry-exhaustion fallback for both text and structured batches while preserving current retry classification and regression coverage.

## Reproduction Assessment

Yes, at source level. A mocked provider that keeps failing multi-item embedBatch or embedBatchInputs with TypeError("fetch failed") after retries but succeeds on singleton calls follows the current direct retry-then-throw path because no split fallback is invoked.

## Solution Assessment

Yes for the requested direction. Retry-plus-split fallback in the memory-core batch wrappers is the narrowest maintainable fix; provider-specific branching, config knobs, and timeout changes are not needed.
