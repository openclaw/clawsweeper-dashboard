## Summary

Keep open: current `main` and the latest release still run the Windows installer npm path with strict npm TLS and only generic npm failure handling, while the direct Windows fallback PR was closed unmerged and the merged related fix covers Linux nvm runtime TLS only.

## What This Changes

The issue reports that the Windows PowerShell installer fails during `npm install -g openclaw@latest` with npm TLS certificate verification error `UNABLE_TO_VERIFY_LEAF_SIGNATURE`.

## Best Possible Solution

Land a security-reviewed Windows installer recovery path that keeps npm TLS verification strict by default, detects certificate failures, and offers CA-bundle remediation or an explicit operator-approved fallback without silently downgrading TLS.

## Reproduction Assessment

No high-confidence live reproduction was established because a matching Windows CA/proxy environment was not available. The report provides a concrete failing command/output, and source inspection shows current `main` still fails generically when npm returns that TLS error.

## Solution Assessment

Unclear; the unmerged `--strict-ssl=false` retry is a plausible mitigation but not obviously the best fix because it changes a security-sensitive installer trust boundary. A safer end state is maintainer-approved TLS diagnosis plus CA remediation or explicit opt-in fallback.
