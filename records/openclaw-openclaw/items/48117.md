## Summary

Keep open: current main and the latest release still route the Windows PowerShell installer through npm without TLS-specific recovery for `UNABLE_TO_VERIFY_LEAF_SIGNATURE`. The exact Windows fallback PR at https://github.com/openclaw/openclaw/pull/48416 is closed unmerged, and the merged Linux CA work does not cover this installer path, so the remaining decision is a security-sensitive installer policy choice.

## What This Changes

The issue asks OpenClaw to make the Windows PowerShell installer succeed or provide safe guidance when `npm install -g openclaw@latest` fails with `UNABLE_TO_VERIFY_LEAF_SIGNATURE`.

## Best Possible Solution

Add a maintainer-approved Windows installer diagnostic or explicit opt-in CA recovery path that preserves strict TLS by default and tells affected users how to configure trusted CA material safely.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not fully: the issue gives a concrete PowerShell command and npm error, and current source still reaches an unhandled npm failure path, but this review did not run a live Windows/corporate-CA reproduction.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the TLS policy; the closed retry-with-`--strict-ssl=false` approach is only a mitigation, while a strict-by-default diagnostic or explicit CA configuration path is safer.
