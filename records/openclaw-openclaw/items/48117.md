## Summary

Keep open. Current main and latest v2026.6.8 still use a plain npm global install in the Windows PowerShell installer with no TLS-specific recovery, the exact Windows fallback PR is closed unmerged, and the merged Linux NODE_EXTRA_CA_CERTS work is adjacent but does not fix this PowerShell installer path.

## What This Changes

The issue asks OpenClaw's Windows PowerShell installer to recover when npm global install fails with UNABLE_TO_VERIFY_LEAF_SIGNATURE.

## Best Possible Solution

Define a secure Windows installer TLS recovery path that preserves strict verification by default, prefers trusted CA or npm cafile guidance, and makes any strict-ssl bypass explicit, opt-in, and noninteractive-safe.

## Reproduction Assessment

No. The issue gives a concrete PowerShell command and npm TLS error, and current source would still fail closed on that npm error, but this review did not run an affected Windows Node certificate-store scenario.

## Solution Assessment

No clear best fix yet. The closed strict-ssl fallback PR shows a plausible mitigation, but installer TLS verification is a security boundary and maintainers should choose a safer CA-trust or explicit opt-in design before implementation.
