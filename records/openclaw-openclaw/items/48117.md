## Summary

This should stay open: current main still invokes npm once from the Windows installer without TLS/certificate-specific recovery, while docs advertise that installer path and the only linked Windows fallback PR closed unmerged after TLS-bypass concerns.

## What This Changes

The issue reports that the Windows PowerShell installer fails during global npm install with `UNABLE_TO_VERIFY_LEAF_SIGNATURE`.

## Best Possible Solution

Add Windows installer handling for npm certificate failures that prefers trusted CA/system-store configuration and only uses any insecure fallback as explicit, clearly warned user opt-in with focused `install.ps1` coverage.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. The reporter's transcript matches the current source path: if npm returns `UNABLE_TO_VERIFY_LEAF_SIGNATURE`, `install.ps1` prints the npm failure and returns false without TLS-specific recovery.

## Solution Assessment

Unclear. Detecting npm certificate failures is the right area, but the final approach should avoid silently disabling TLS verification and likely needs maintainer security direction before implementation.
