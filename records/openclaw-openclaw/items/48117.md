## Summary

Keep this open: the Windows PowerShell installer still routes the reported npm certificate error through generic npm failure handling. The direct fallback PR was closed unmerged after review flagged the security risk of automatically retrying with disabled SSL validation, and the merged Linux nvm CA fix is adjacent but does not cover this Windows installer path.

## What This Changes

The issue asks OpenClaw to make the Windows PowerShell installer recover from npm `UNABLE_TO_VERIFY_LEAF_SIGNATURE` failures instead of failing during global install.

## Best Possible Solution

Add Windows installer handling that detects npm certificate verification failures and prefers trusted CA, `cafile`, or `NODE_EXTRA_CA_CERTS` guidance; any insecure retry should be explicit, warned, and maintainer-approved.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. Source inspection shows that if npm emits the reported certificate error, the Windows installer still falls through to generic npm failure handling.

## Solution Assessment

Unclear. The Windows installer is the right surface, but the previously proposed automatic `strict-ssl=false` fallback needs a safer opt-in or trusted-CA design before merge.
