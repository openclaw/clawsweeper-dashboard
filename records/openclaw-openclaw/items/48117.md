## Summary

Current `main` still has the reported Windows installer failure path: `install.ps1` runs npm with default strict TLS and only prints a generic failure when npm reports a certificate error. The related Linux CA fix does not apply to Windows, and the prior Windows fallback PR was closed unmerged after raising a real TLS-bypass security concern, so this should stay open for maintainer direction.

## What This Changes

The issue asks for the Windows PowerShell installer to recover from `npm install -g openclaw@latest` failing with `UNABLE_TO_VERIFY_LEAF_SIGNATURE`.

## Best Possible Solution

Add a Windows installer recovery path that detects npm certificate failures and uses a trusted CA-based fix when possible, or an explicit user-approved fallback with clear guidance, plus focused `install.ps1` tests.

## Reproduction Assessment

No high-confidence live reproduction was run against current `main`; however, the reported npm error transcript matches a current source path that has no TLS-specific recovery and will return installer failure on that npm error.

## Solution Assessment

Unclear. The old closed PR's basic detection idea is relevant, but an automatic or weakly gated `--strict-ssl=false` retry is not the best final shape for an installer without maintainer security direction.
