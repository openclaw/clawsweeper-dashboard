## Summary

Keep open. Current main and the latest release still have the Windows PowerShell installer invoke npm without any TLS-specific recovery, while the related Linux CA-bundle fix is scoped away from Windows and the Windows retry PR was closed unmerged.

## What This Changes

The issue requests that the Windows PowerShell installer recover from npm `UNABLE_TO_VERIFY_LEAF_SIGNATURE` failures during global OpenClaw installation.

## Best Possible Solution

Add a Windows installer recovery path that detects npm TLS verification failures and preserves certificate verification by default, preferably through CA trust repair guidance or a scoped `cafile`/`NODE_EXTRA_CA_CERTS` path. If maintainers approve an insecure retry, it should be single-install scoped, explicit in the installer output, covered by PowerShell tests, and validated with a Windows smoke run.

## Reproduction Assessment

No, not end-to-end. The report gives a concrete Windows `iwr ... | iex` command and npm error, and the installer failure branch can be unit-tested with a mocked `Invoke-NativeCommandCapture`, but the Windows certificate-store mismatch itself was not live-reproduced here.

## Solution Assessment

Unclear because main does not yet contain a solution. The safest maintainable direction is CA-preserving recovery first; the closed `--strict-ssl=false` retry PR is useful prior art but needs maintainer TLS policy approval before adoption.
