## Summary

Keep open. Current main still sends the Windows PowerShell installer through a plain global npm install with generic nonzero-exit handling, the docs do not describe a Windows npm certificate recovery path, the prior Windows retry PR was closed unmerged, and the merged CA work is scoped to Linux/nvm runtime startup rather than this installer failure.

## What This Changes

The issue requests that the Windows PowerShell installer succeed when npm fails with `UNABLE_TO_VERIFY_LEAF_SIGNATURE` during the default global npm install.

## Best Possible Solution

Add a Windows `install.ps1` TLS-error recovery path that preserves certificate verification by default, with docs and tests; any scoped insecure fallback should require explicit maintainer approval.

## Reproduction Assessment

No, not end-to-end. The issue includes a concrete Windows command and npm TLS error, and current source shows that error would flow through generic npm failure handling, but this review did not live-reproduce the host-specific trust-store mismatch on Windows.

## Solution Assessment

Unclear because no fix is on main. The safest maintainable direction is a CA-preserving Windows installer recovery first; the closed `--strict-ssl=false` retry is prior art but should not become default behavior without maintainer approval.
