## Summary

Keep open: current main and the latest release still route the reported Windows PowerShell installer npm certificate failure through the generic npm install failure path, and the only direct fix attempt closed unmerged because the recovery changes installer TLS trust.

## What This Changes

The issue reports that `iwr -useb https://openclaw.ai/install.ps1 | iex` fails on Windows during `npm install -g openclaw@latest` with `UNABLE_TO_VERIFY_LEAF_SIGNATURE`.

## Best Possible Solution

Land a security-reviewed Windows installer recovery that keeps strict TLS verification as the default, detects npm certificate-verification failures, and guides explicit CA or operator-approved remediation without silently bypassing certificate checks.

## Reproduction Assessment

Yes at source level, but not with a live Windows TLS-mismatch environment in this review. Current main still sends npm certificate failures through the generic installer failure path.

## Solution Assessment

Unclear until maintainers choose the trust boundary. The closed direct `--strict-ssl=false` retry is not the best default; a safer solution should diagnose the mismatch and use explicit CA or operator-approved remediation.
