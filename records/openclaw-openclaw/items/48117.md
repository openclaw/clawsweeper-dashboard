## Summary

Keep open: current main and v2026.6.6 still send the reported Windows npm TLS failure through the generic installer failure path, while the merged CA work covers Linux version-manager Node runtimes and the direct Windows strict-SSL fallback PR closed unmerged after security concerns.

## What This Changes

The issue asks for the Windows PowerShell installer to succeed or recover when npm fails with `UNABLE_TO_VERIFY_LEAF_SIGNATURE` during the `openclaw@latest` install.

## Best Possible Solution

Add a Windows installer recovery or diagnostic path that preserves TLS verification by default, prefers explicit CA or system-store configuration, and treats any SSL-bypass fallback as a visible maintainer-approved opt-in with noninteractive fail-closed behavior.

## Reproduction Assessment

No live current-main Windows reproduction was run. Source inspection gives a clear conditional path: if npm returns the reported leaf-signature TLS error, the Windows installer still reports a generic npm failure and exits without TLS-specific recovery.

## Solution Assessment

Unclear: the bug belongs in the Windows installer path, but the earlier `--strict-ssl=false` retry is not clearly the best fix without maintainer security approval. A trust-preserving recovery or explicit opt-in diagnostic flow is safer than silently weakening certificate validation.
