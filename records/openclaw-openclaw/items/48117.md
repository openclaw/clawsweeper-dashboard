## Summary

Keep open. Current main still routes the Windows PowerShell installer through a plain global npm install with generic nonzero-exit handling, the installer docs do not describe a Windows npm certificate recovery path, and the related merged CA work is Linux/nvm runtime scoped rather than a fix for this Windows installer failure.

## What This Changes

The issue reports that the Windows PowerShell installer fails during `npm.cmd install -g openclaw@latest` with `UNABLE_TO_VERIFY_LEAF_SIGNATURE` after `install.ps1` has already been fetched successfully.

## Best Possible Solution

Add a Windows `install.ps1` recovery path for npm certificate verification failures that preserves certificate verification by default, preferably by detecting the TLS error and guiding or applying a scoped CA trust repair such as `cafile`, `NODE_EXTRA_CA_CERTS`, or another maintainer-approved Node/npm trust-store fix. If maintainers explicitly approve an insecure fallback, it should be one-install scoped, clearly warned, documented, and covered by PowerShell installer tests plus a Windows smoke path.

## Reproduction Assessment

No, not end-to-end. The issue includes a concrete Windows one-liner and npm error, and current source inspection shows that error would flow through generic npm failure handling, but this review did not live-reproduce the host-specific certificate-store mismatch on Windows.

## Solution Assessment

Unclear because there is no implemented solution on main. The safest maintainable direction is a CA-preserving Windows installer repair first; the closed `--strict-ssl=false` retry is useful prior art but should require explicit maintainer approval before adoption.
