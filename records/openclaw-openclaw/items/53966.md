## Summary

Keep this PR open. The Slack error-detail direction is useful and the retryAfter SDK-shape concern appears addressed, but the current head replaces several redacting error-formatting paths with a new formatter that returns raw message and metadata text, creating a concrete token-redaction regression before merge.

## What This Changes

The PR adds a Slack-local `formatSlackError()` helper with unit tests and routes selected Slack probe, scope, startup resolution, stream preview, reaction, slash reply, and thread-resolution error paths through it to include structured Slack SDK metadata.

## Best Possible Solution

Land the Slack-plugin-local formatter after it redacts the final assembled string through the plugin-safe logging SDK export and adds a regression test with Slack-shaped token data; keep the broader delivery-queue serialization problem tracked separately through #62391.

Yes. A focused unit check can pass a token-shaped Slack/Web API error into the PR's `formatSlackError()` and observe the raw token in the returned string, while current main's `formatErrorMessage()` redacts the same token patterns.

## Solution Assessment

No as written. A Slack-local formatter is the right boundary for these API details, but it needs to preserve the existing redaction contract before replacing `formatErrorMessage` call sites.
