## Label Justifications

- `P1`: The report describes a real Telegram channel workflow becoming silent while queued user messages pile up behind one poison update.
- `impact:message-loss`: Queued Telegram messages are blocked behind the poison head and the documented manual workaround loses or stashes those user messages instead of processing them.
- `impact:crash-loop`: The affected channel can hang indefinitely in a tight retry loop even though the overall gateway remains up.

## Summary

Keep open: the reported failure matches current `main` and the latest release behavior. Telegram isolated ingress requeues any `bot.handleUpdate` error, including a missing harness configuration error, while the existing dead-letter path is only used for timed-out active handlers.

## What This Changes

The issue asks for Telegram isolated ingress to quarantine or otherwise unblock a slash-command spooled update when its command session hits `MissingAgentHarnessError` instead of retrying forever.

## Best Possible Solution

Add a focused Telegram spool error classifier that dead-letters non-retryable configuration errors such as missing agent harnesses while preserving transient retry behavior, with regression coverage for slash-command spool failure and ordinary retryable errors.

## Reproduction Assessment

Yes, source-reproducible: a thrown `bot.handleUpdate` error enters `#releaseFailedSpooledUpdate`, releases the claim, and requeues the same update, and the current test suite already expects that retry behavior. I did not run a live Telegram repro in this read-only pass.

## Solution Assessment

Yes for the spool-classification part: using the existing `.failed` tombstone path for non-retryable configuration errors is the narrowest maintainable fix. Slash-session fallback or a new CLI status command can be follow-up work if maintainers want broader UX improvements.
