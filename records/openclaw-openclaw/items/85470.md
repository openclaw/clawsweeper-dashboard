## Label Justifications

- `P1`: The report describes a real Telegram channel workflow becoming silent while user messages queue behind one poison update.
- `impact:message-loss`: Queued Telegram messages are blocked behind the poison head, and the documented workaround stashes or loses those messages instead of processing them.
- `impact:crash-loop`: The affected channel can hang in an indefinite tight retry loop even though the overall gateway stays up.

## Summary

Keep open: current `main` and `v2026.5.20` still requeue every Telegram `bot.handleUpdate` failure, including `MissingAgentHarnessError`, and an open linked fix PR references this issue with closing syntax, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

A Telegram slash-command spool update with a missing agent harness is reported to retry forever and block later Telegram messages.

## Best Possible Solution

Land a corrected Telegram spool error classifier, likely through https://github.com/openclaw/openclaw/pull/85495, that dead-letters `MissingAgentHarnessError` via the existing failed tombstone path while preserving transient retries and adding regression coverage.

## Reproduction Assessment

Yes, source-reproducible: a thrown `bot.handleUpdate` error releases the spool claim and requeues the same update, and current tests already assert retry-on-handler-error behavior. I did not run live Telegram or local tests in this read-only pass.

## Solution Assessment

Yes for the spool-classification fix: using the existing failed tombstone path for non-retryable missing-harness configuration errors is the narrowest maintainable repair. Slash-session fallback and a spool-status CLI can be follow-up product work if maintainers want broader UX improvements.
