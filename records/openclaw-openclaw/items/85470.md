## Label Justifications

- `P1`: The report describes a real Telegram channel workflow going silent while user messages pile up behind one poison spooled update.
- `impact:message-loss`: Queued Telegram messages can remain blocked behind the poison head, and the documented manual workaround stashes or loses those user messages instead of processing them.
- `impact:crash-loop`: The affected channel can spin indefinitely in a tight retry loop even though the overall gateway remains up.

## Summary

Keep open: current main and v2026.5.20 still requeue every Telegram isolated-ingress `bot.handleUpdate` error, including `MissingAgentHarnessError`, so one poison slash-command update can keep blocking later same-lane Telegram messages. The related open harness PR may fix the immediate `claude-cli` compaction trigger, but it does not add the spool-level non-retryable error classification this report needs.

## What This Changes

Treat missing-harness/configuration failures from Telegram spooled slash-command handling as non-retryable so one poison update cannot indefinitely block later Telegram messages.

## Best Possible Solution

Add a narrow Telegram spool error classifier that tombstones clear non-retryable configuration errors such as missing harnesses, while keeping transient handler failures on the existing retry path and leaving the CLI alias selector fix to the related harness PR.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make a spooled Telegram update whose `bot.handleUpdate` throws `MissingAgentHarnessError`; current main releases the claim, requeues the same file, and later same-lane updates remain behind it. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes for the spool-classification part. Reusing the existing `.failed` tombstone path for clear non-retryable configuration errors is the narrowest maintainable fix; slash-session fallback or the open CLI alias selector PR can be handled separately.
