## Label Justifications

- `P1`: The report describes a real Telegram workflow becoming silent while user messages accumulate behind one poison spooled update.
- `impact:message-loss`: Queued Telegram messages are blocked behind the poison head, and the documented workaround loses or stashes those messages instead of processing them.
- `impact:crash-loop`: The affected channel can spin indefinitely in a tight retry loop even though the overall gateway remains up.

## Summary

Keep open: current main still requeues every Telegram spooled `bot.handleUpdate` failure, including `MissingAgentHarnessError`, and the issue is now paired with an open but unmerged fix PR. The source path, existing retry test, latest release check, and linked PR all show the user-visible Telegram queue deadlock is still unresolved on main.

## What This Changes

The issue asks OpenClaw to stop infinitely retrying Telegram slash-command spool updates that fail with `MissingAgentHarnessError` and to let later Telegram messages drain.

## Best Possible Solution

Land a focused Telegram ingress-spool classifier through a supported boundary that dead-letters `MissingAgentHarnessError`, preserves transient retries, adds regression coverage, and proves later spooled Telegram messages continue to drain.

## Reproduction Assessment

Yes. The source path is high-confidence: a thrown `bot.handleUpdate` error releases the spooled claim and requeues the same update, and the existing test suite expects that retry behavior; I did not run live Telegram in this read-only review.

## Solution Assessment

Yes for the requested spool-classification direction. The best landing path is a narrow dead-letter classifier plus tests and real/log proof; the linked PR is the current vehicle but still needs its boundary and proof blockers resolved.
