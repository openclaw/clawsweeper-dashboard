## Label Justifications

- `P1`: The issue describes a real Telegram channel workflow becoming silent while queued user messages pile up behind one poison update.
- `impact:message-loss`: Queued Telegram messages are blocked behind the poison head and the manual workaround loses or stashes those user messages instead of processing them.
- `impact:crash-loop`: The affected channel can remain in a tight retry loop indefinitely even though the overall gateway process stays up.

## Summary

Keep open: current `main` still requeues Telegram spooled updates after any immediate `bot.handleUpdate` error, including missing harness configuration failures, so a poison slash-command update can continue blocking later same-lane messages.

## What This Changes

The issue requests Telegram ingress spool handling that quarantines non-retryable missing-harness slash-command failures instead of retrying the same queued update indefinitely.

## Best Possible Solution

Add a focused Telegram spool error classifier that dead-letters non-retryable missing-harness/configuration errors through the existing `.failed` path while preserving ordinary transient retries, with regression coverage for both cases.

## Reproduction Assessment

Yes, source-reproducible: the current handler catch path releases the spooled update for retry after any `bot.handleUpdate` error, and existing tests cover failed handler retry behavior. I did not run live Telegram because this pass is read-only and live proof would require a real channel setup.

## Solution Assessment

Yes for the spool-classification repair: using the existing `.failed` tombstone path for non-retryable configuration errors is the narrowest maintainable fix. Slash fallback and a spool-status CLI can remain separate follow-up decisions.
