## Label Justifications

- `P1`: The report describes a real Telegram workflow becoming silent while later user messages pile up behind one poison spooled update.
- `impact:message-loss`: Queued Telegram messages can remain blocked behind the failed head, and the manual workaround stashes or loses those messages instead of processing them.
- `impact:crash-loop`: The affected channel can hang in an indefinite tight retry loop even though the overall gateway remains running.

## Summary

Keep open: current main and v2026.5.20 still requeue any Telegram spooled handler error, including MissingAgentHarnessError, and the linked open fix PR should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks Telegram ingress spool handling to avoid indefinitely retrying slash-command updates that fail with MissingAgentHarnessError and block later messages.

## Best Possible Solution

Land or replace the linked fix PR with a supported Telegram spool classifier that dead-letters MissingAgentHarnessError while preserving transient retries, then close this issue after merge and proof.

## Reproduction Assessment

Yes, source-reproducible: current main catches every bot.handleUpdate exception, releases the claim, and logs the spooled update as kept for retry; the existing test suite also expects failed lanes to remain pending. I did not run a live Telegram repro in this read-only pass.

## Solution Assessment

Yes for the spool-classification direction: using the existing failed tombstone path for non-retryable missing-harness errors is the narrowest maintainable fix. The open PR is the implementation candidate, but it still needs its own review and proof before merge.
