## Label Justifications

- `P1`: The report describes a real Telegram workflow becoming silent while later user messages queue behind one poisoned update.
- `impact:message-loss`: Queued Telegram messages are blocked behind the poison head, and the documented manual workaround stashes or loses those user messages instead of processing them.
- `impact:crash-loop`: The affected channel can spin indefinitely in a tight retry loop even though the gateway process remains up.

## Summary

Keep open: the reported Telegram poison-head retry still matches current main and the latest release, and an open linked fix PR specifically closes this issue but has not merged yet.

## What This Changes

Telegram ingress should not indefinitely retry a spooled slash-command update when dispatch fails with a missing agent harness and blocks later Telegram messages.

## Best Possible Solution

Land a focused fix, likely via the linked PR or a replacement, that classifies missing-harness spool failures as nonretryable through a supported boundary while preserving transient retry behavior.

## Reproduction Assessment

Yes. Source inspection shows every `bot.handleUpdate` failure from the Telegram ingress spool is requeued, and existing tests assert that retry behavior; I did not run a live Telegram repro in this read-only pass.

## Solution Assessment

Yes for the narrow spool-classification direction: using the existing failed-tombstone path for nonrecoverable missing-harness errors is the maintainable fix. The linked PR still needs review/repair and proof before it can close this issue.
