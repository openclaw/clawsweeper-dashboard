## Summary

Keep open: current `main` and the latest release still requeue every Telegram spooled `bot.handleUpdate` failure, including missing-harness configuration errors, and the linked closing PR is open but not merged.

## What This Changes

The issue reports a Telegram ingress spool head-of-queue deadlock when a slash-command update hits a session whose agent harness is not registered.

## Best Possible Solution

Add a narrow Telegram spool error classifier that dead-letters nonretryable missing-harness failures through a supported boundary while preserving transient retry behavior.

## Reproduction Assessment

Yes. Source inspection shows a thrown `bot.handleUpdate` error releases the spooled claim and requeues the same update, and the current test suite expects that retry behavior; I did not run live Telegram in this read-only pass.

## Solution Assessment

Yes for the spool-classification part. Using the existing `.failed` tombstone path for nonretryable missing-harness errors is the narrowest maintainable fix; slash-session fallback and spool-status CLI UX can remain follow-up decisions.
