## Label Justifications

- `P1`: A single Telegram slash-command update can keep a real Telegram conversation from receiving later agent replies until manual spool intervention.
- `impact:message-loss`: The reported failure suppresses or indefinitely delays subsequent Telegram inbound messages, and the documented workaround can discard queued messages.
- `impact:auth-provider`: The poison error is a missing `claude-cli` agent harness/runtime registration failure tied to provider/runtime selection.

## Summary

Keep open: current main still releases every `bot.handleUpdate` error back to the Telegram ingress spool, so a fast `MissingAgentHarnessError` can be retried on each drain cycle and keep later same-lane messages behind the poisoned update.

## What This Changes

The issue reports that a Telegram slash-command update can be retried indefinitely after `MissingAgentHarnessError`, blocking subsequent inbound Telegram messages until operators manually move spool files.

## Best Possible Solution

Add a narrow Telegram isolated-ingress error classifier that tombstones/quarantines `MissingAgentHarnessError` and other clear configuration errors, lets later same-lane updates drain, and leaves the CLI harness root-cause work to the linked agent-runtime issues.

## Reproduction Assessment

Yes at source level: a spooled update whose `bot.handleUpdate` rejects with `MissingAgentHarnessError` is released back to pending and retried on the next 500 ms drain cycle. I did not run a live Telegram setup during this read-only pass.

## Solution Assessment

Yes, the narrowest maintainable fix is to classify clear non-retryable handler/configuration errors in the Telegram spool and tombstone only that update; the broader `claude-cli` registration race should stay with the existing agent-runtime issues.
