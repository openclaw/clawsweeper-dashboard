## Summary

Keep open. Current main still sends mirror-mode nested file writes through the direct single-file OpenShell upload path, while the nearby full-workspace sync path already stages uploads safely; the reported bug remains narrow and actionable.

## What This Changes

OpenShell mirror-mode `write` should upload files into existing remote subdirectories without failing when tar preserves nested host path components.

## Best Possible Solution

Fix the regular-file branch of `syncLocalPathToRemote` to stage a basename-only temporary upload directory, add focused regression coverage for nested mirror writes, and keep the change inside the OpenShell plugin boundary.

## Reproduction Assessment

Yes. The issue provides concrete config, steps, and logs, and current main confirms the `writeFile` -> `syncLocalPathToRemote` -> `openshell sandbox upload <nested host file> <remote parent>` path that triggers the reported tar collision.

## Solution Assessment

Yes. Basename-only staging for the single-file upload branch is the narrowest maintainable fix because it matches the existing full-workspace upload pattern without changing core sandbox contracts.

## Repair Work Prompt

Fix OpenShell mirror-mode nested file writes failing with `tar: <dir>: Cannot open: File exists` when the remote parent directory already exists. Keep the fix inside the OpenShell plugin: update `extensions/openshell/src/backend.ts` so the regular-file branch of `syncLocalPathToRemote(localPath, remotePath)` stages a temporary upload directory containing only `path.basename(localPath)` before calling `openshell sandbox upload` to `path.posix.dirname(remotePath)`, matching the existing `uploadPathToRemote` staging pattern. Do not change core sandbox contracts or remote mode behavior. Add focused regression coverage that verifies a nested mirror write/backend sync does not pass the original nested host path directly to `openshell sandbox upload`; prefer unit coverage around `syncLocalPathToRemote`/CLI args plus any minimal bridge coverage needed. Add a user-facing `CHANGELOG.md` fix entry without bot/maintainer attribution. Related refs: https://github.com/openclaw/openclaw/issues/60961 and #57693 for the existing staged upload pattern.
