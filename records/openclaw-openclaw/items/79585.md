## Summary

Keep open: current main still has a documented Google/Gemini CLI runtime split that is not consistently reflected in auth status or CLI auth-profile forwarding, and the existing ClawSweeper comment already identified this as maintainer-follow-up work rather than a stale cleanup item.

## What This Changes

The issue requests consistent auth routing and status reporting for canonical `google/*` models when `google-gemini-cli` OAuth exists and `GEMINI_API_KEY` is also present.

## Best Possible Solution

Make Gemini CLI OAuth setup, runtime selection, status reporting, and CLI credential preparation share the same provider/model runtime contract, with explicit diagnostics if env-key precedence is intentionally retained.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection shows the documented runtime split is not carried through status and auth-profile forwarding, but the exact status output needs a real auth store plus `GEMINI_API_KEY` environment.

## Solution Assessment

Unclear pending maintainer review. The narrow direction is to align the Gemini CLI OAuth flow with provider/model-scoped runtime policy and runtime-aware status/auth resolution, but credential precedence against `GEMINI_API_KEY` is an ownership decision.
