## Summary

Keep open: current main still shows a real mismatch between the documented Gemini CLI runtime split and the auth/status/execution code paths for canonical `google/*` models. Source inspection indicates OAuth login stores a `google-gemini-cli` profile and an ignored whole-agent runtime key, while status and CLI execution do not clearly bind canonical Google model use to that profile.

## What This Changes

The issue asks OpenClaw to make Gemini CLI OAuth effective and clearly reported for canonical `google/*` models when `GEMINI_API_KEY` is also present, or to document and warn if env-key precedence is intentional.

## Best Possible Solution

Align login, status, and execution around the documented provider/model runtime policy so canonical `google/*` with the `google-gemini-cli` runtime selects and reports the Gemini CLI OAuth profile, with explicit diagnostics only if env-key precedence is intentionally retained.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection does show the central mismatch: login stores `google-gemini-cli` OAuth and an ignored runtime key, while status/execution do not clearly use that profile for canonical `google/*`.

## Solution Assessment

Unclear pending maintainer review of the credential handoff. The narrow direction is to fix the runtime/profile selection path rather than papering over it in docs, but external Gemini CLI auth behavior should be verified first.
