## Summary

Keep open: current main still leaves a runtime-aware auth/status mismatch for canonical Google model refs when `agentRuntime.id` is `google-gemini-cli` and `GEMINI_API_KEY` is also present.

## What This Changes

The issue asks OpenClaw to make canonical `google/*` models with `agentRuntime.id: "google-gemini-cli"` use or clearly report Gemini CLI OAuth instead of silently preferring `GEMINI_API_KEY`.

## Best Possible Solution

Align status and execution around the documented runtime split: canonical `google/*` plus `google-gemini-cli` runtime should select and report the Gemini CLI OAuth profile, or the docs and CLI should explicitly explain env-key precedence.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection shows the same mismatch path: canonical `google` status resolves env auth independently, while CLI execution does not auto-forward a `google-gemini-cli` OAuth profile.

## Solution Assessment

Unclear until maintainers confirm auth precedence and the external Gemini CLI credential contract. The narrow maintainable direction is to centralize runtime-aware auth resolution instead of adding ad hoc status-only special cases.
