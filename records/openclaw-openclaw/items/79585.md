## Summary

Keep open: current main still has an unresolved Google auth/status routing gap for canonical `google/*` models with the `google-gemini-cli` runtime when `GEMINI_API_KEY` is present. The merged auth-health PR fixed adjacent `models status --check` behavior, but it did not connect canonical Google model use to the Gemini CLI OAuth profile or document env-key precedence for this runtime split.

## What This Changes

The issue asks OpenClaw to make canonical `google/*` models with `agentRuntime.id: "google-gemini-cli"` consistently use and report Gemini CLI OAuth instead of silently preferring `GEMINI_API_KEY`.

## Best Possible Solution

Align status and execution with the documented runtime split: canonical `google/*` plus `google-gemini-cli` runtime should either select/report the Gemini CLI OAuth path ahead of env API-key auth, or clearly document and diagnose env-key precedence everywhere it applies.

## Reproduction Assessment

No, not with high-confidence live proof. Source inspection on current main still shows the provider-local status/auth path and CLI execution path do not automatically bind canonical `google/*` runtime selection to a `google-gemini-cli` OAuth profile when env auth is also available.

## Solution Assessment

Unclear pending maintainer review of the credential handoff. The documented direction favors Gemini CLI OAuth for this runtime split, but the implementation should first verify the external Gemini CLI auth-selection contract and avoid leaking or persisting OAuth material incorrectly.
