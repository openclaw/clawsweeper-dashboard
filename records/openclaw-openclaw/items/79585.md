## Summary

Keep open: current main partially improves the reported status output by listing OAuth profiles, but the canonical `google` auth row still resolves independently to `GEMINI_API_KEY`, and the Gemini CLI execution path does not automatically bind the stored `google-gemini-cli` OAuth profile for canonical `google/*` runs.

## What This Changes

The issue reports that Gemini CLI OAuth profiles are ignored or misreported for canonical `google/*` model refs when `GEMINI_API_KEY` is also present.

## Best Possible Solution

Align status and execution so canonical `google/*` plus `google-gemini-cli` runtime reports and uses the same intended auth surface, or explicitly document and warn when ambient API-key env is intentionally higher precedence.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection gives a strong reproduction path: canonical `google` status resolves env auth separately, while CLI execution inherits ambient env and does not auto-bind the stored `google-gemini-cli` OAuth profile.

## Solution Assessment

Unclear pending maintainer policy. The narrow technical direction is clear, but choosing OAuth precedence, env scrubbing, or explicit diagnostics for an external Gemini CLI backend should be reviewed before implementation.
