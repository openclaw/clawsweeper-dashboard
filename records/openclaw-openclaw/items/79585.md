## Summary

Keep open: current main still does not align Gemini CLI OAuth login, canonical `google/*` runtime policy, and `models status` when `GEMINI_API_KEY` is present. The exact old `OAuth/token status - none` output appears partly improved by current profile inventory, but the core routing/status mismatch remains source-reproducible.

## What This Changes

The issue asks OpenClaw to make Gemini CLI OAuth profile routing and status consistent for canonical `google/*` models when `GEMINI_API_KEY` is also present.

## Best Possible Solution

Update Gemini CLI OAuth setup, runtime auth selection, and status reporting together so canonical `google/*` plus a provider/model-scoped `google-gemini-cli` runtime records, uses, and reports the `google-gemini-cli` OAuth profile, or explicitly documents and warns about env-key precedence if maintainers choose that policy.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection does show current main still ignores the reported legacy runtime key and does not automatically connect canonical `google/*` Gemini CLI runtime selection to the `google-gemini-cli` OAuth profile/status path.

## Solution Assessment

Unclear until maintainers choose the auth precedence policy. The narrow maintainable direction is to fix config writing, runtime auth selection, and status diagnostics as one contract rather than patching only the display string.
