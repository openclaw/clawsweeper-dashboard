## Summary

Keep open: current main still leaves the documented canonical `google/*` plus `google-gemini-cli` runtime split ambiguous in both status reporting and CLI auth-profile forwarding when `GEMINI_API_KEY` is also present.

## What This Changes

The issue asks OpenClaw to make canonical Google model refs with `agentRuntime.id: "google-gemini-cli"` use or clearly report the Gemini CLI OAuth profile instead of silently showing the ambient Google API key path.

## Best Possible Solution

Align the status and execution paths with the documented runtime split, or intentionally document and surface env-key precedence everywhere the user opts into Gemini CLI OAuth.

## Reproduction Assessment

No, not as a live high-confidence reproduction in this read-only pass. Source inspection shows the same mismatch path: canonical `google` status resolves provider-local auth and env first, while CLI execution does not automatically select a `google-gemini-cli` OAuth profile for the canonical Google runtime split.

## Solution Assessment

Unclear pending maintainer review of the auth boundary. The narrow direction is to make `google/*` plus `google-gemini-cli` runtime select and report the Gemini CLI OAuth identity, but the team should decide whether ambient `GEMINI_API_KEY` may intentionally override that and how to fail or warn safely.
