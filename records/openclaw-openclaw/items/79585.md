## Summary

Keep open: current main still shows a real Google/Gemini CLI auth-routing mismatch. The documented runtime split says canonical `google/*` can use `google-gemini-cli`, but the OAuth setup still writes an ignored whole-agent runtime key, status resolves `google` auth independently from `google-gemini-cli` profiles, and the CLI runtime auth plan does not forward a `google-gemini-cli` profile for canonical `google/*` execution.

## What This Changes

The issue requests that canonical `google/*` models respect and report the `google-gemini-cli` OAuth runtime profile when `GEMINI_API_KEY` is also present.

## Best Possible Solution

Align the Google plugin, model status, and CLI execution around one provider/model runtime-auth resolver so `google/*` plus `google-gemini-cli` selects and reports the Gemini CLI OAuth profile, or explicitly document and warn if env-key precedence is intentional.

## Reproduction Assessment

No, not as a live current-main reproduction. Source inspection gives a high-confidence path: current docs define the split, while setup/status/execution code still do not connect canonical `google/*` to the stored `google-gemini-cli` OAuth profile when env auth exists.

## Solution Assessment

Unclear until maintainer review confirms auth precedence and credential handoff. The narrow maintainable direction is a targeted runtime-auth/status fix, not closing as a duplicate of broader provider-pool or ACP hardening work.
