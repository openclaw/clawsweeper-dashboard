## Summary

Keep open: current main still documents the `google/*` plus `google-gemini-cli` runtime split, but status and CLI execution auth paths still resolve around the canonical `google` provider and ambient API-key env without selecting or reporting the `google-gemini-cli` OAuth profile for that runtime split.

## What This Changes

The issue asks OpenClaw to make canonical Google model refs use or clearly report the Gemini CLI OAuth profile when the selected runtime is `google-gemini-cli` and `GEMINI_API_KEY` is also present.

## Best Possible Solution

Align status and execution with the documented runtime split so canonical `google/*` with `google-gemini-cli` runtime selects and reports the Gemini CLI OAuth profile, with explicit diagnostics if ambient API-key precedence is intentionally retained.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only pass. Source inspection gives a clear current-main path: canonical `google/*` providers are displayed and planned through `google` auth/env logic, while the `google-gemini-cli` OAuth profile is stored under a separate provider id and is not automatically forwarded into the CLI backend.

## Solution Assessment

Unclear pending maintainer review of the credential handoff. The narrow maintainable direction is to carry runtime-selected auth-provider facts into status and execution, or explicitly document and warn if `GEMINI_API_KEY` is meant to win.
