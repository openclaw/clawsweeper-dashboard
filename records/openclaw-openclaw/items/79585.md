## Summary

Keep open: current main still has a real auth-routing/status mismatch for canonical Google model refs when the runtime is `google-gemini-cli`, and the fix should be reviewed carefully because it crosses OAuth profile handling and an external CLI backend.

## What This Changes

The issue asks OpenClaw to make canonical `google/*` models with `agentRuntime.id: "google-gemini-cli"` use or clearly report the Gemini CLI OAuth profile when `GEMINI_API_KEY` is also present.

## Best Possible Solution

Align status and execution with the documented runtime split so canonical `google/*` plus `google-gemini-cli` runtime selects and reports the Gemini CLI OAuth profile ahead of the Google API-key env path, with explicit diagnostics if env-key precedence is intentionally kept.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection shows the status/env path and CLI execution path do not connect canonical `google/*` runtime selection to the `google-gemini-cli` OAuth profile, but the exact reporter output needs a real auth store and environment.

## Solution Assessment

Unclear pending maintainer review of the credential handoff. The documented product direction is clear, but the implementation should verify the external Gemini CLI auth contract before writing or forwarding OAuth credentials.
