## Summary

Keep open: current main still has a source-visible mismatch between the documented Google runtime split and the status/execution auth path for canonical `google/*` selections using the `google-gemini-cli` runtime.

## What This Changes

The issue asks OpenClaw to make canonical `google/*` models with `agentRuntime.id: "google-gemini-cli"` use and report Gemini CLI OAuth consistently when a profile exists, or clearly document and warn if `GEMINI_API_KEY` intentionally takes precedence.

## Best Possible Solution

Align status and execution with the documented runtime split so canonical `google/*` plus `google-gemini-cli` runtime selects and reports the Gemini CLI OAuth profile, or explicitly document and warn that env-key auth wins.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. The issue provides concrete commands, and source inspection shows the current status and CLI execution paths still do not connect canonical `google/*` runtime selection to the `google-gemini-cli` OAuth profile.

## Solution Assessment

Unclear pending maintainer review of the credential precedence and handoff. A narrow fix should make runtime-aware auth resolution explicit instead of adding a Google-only workaround that could leak or silently prefer the wrong credential surface.
