## Summary

Keep open: current main still has a source-visible mismatch between documented canonical `google/*` plus `google-gemini-cli` runtime behavior, status reporting, and the CLI auth handoff path when Google API-key env auth is also present.

## What This Changes

The issue reports that canonical `google/*` models continue to report or use `GEMINI_API_KEY` after `google-gemini-cli` OAuth login and runtime selection.

## Best Possible Solution

Align login, status, and execution with the provider/model runtime split: canonical `google/*` plus `google-gemini-cli` runtime should use and report the matching OAuth profile consistently, or explicitly document and diagnose any intentional env-key precedence.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. Source inspection gives a concrete current-main path where status and CLI execution do not reinterpret canonical `google/*` through the selected `google-gemini-cli` OAuth profile, while the exact reporter output needs a real auth store and environment.

## Solution Assessment

Unclear pending maintainer review of the credential handoff. The documented product direction is clear, but the narrow implementation should be chosen after checking Gemini CLI credential behavior and the related OAuth extraction trust-boundary concern.
