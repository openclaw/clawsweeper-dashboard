## Summary

Keep open: current main still lacks generic per-agent MCP server scoping, but this PR is not merge-ready because the new agent-filtered runtime fingerprint is compared against an unscoped read-only MCP summary in `tools.effective`, which can report warm scoped runtimes as stale and expose scoped-away server names in notices.

## What This Changes

The PR adds a top-level `mcp.servers.*.agents` allowlist and threads the resolved agent id through CLI, embedded, and Codex MCP config projection paths, with schema/help/docs and focused tests.

## Best Possible Solution

Keep the generic allowlist approach, but thread the resolved agent id through MCP summary/fingerprint readers and `tools.effective` summary caching, then document the field in the public configuration reference before merge.

## Reproduction Assessment

Yes. From source, PR-head runtime fingerprints are built from agent-filtered MCP servers, while `resolveSessionMcpConfigSummary` and the `tools.effective` summary cache still omit the agent id.

## Solution Assessment

No. The shared allowlist is the right owner boundary, but the inventory/fingerprint readers and public config reference must be aligned before this is the best fix.
