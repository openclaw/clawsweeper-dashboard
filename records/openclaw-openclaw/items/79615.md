## Summary

Keep open: the source-level bug is real, but this PR still needs real behavior proof and the fallback should be constrained to OpenAI-compatible models before merge.

## What This Changes

The PR exports DeepSeek V4 model-id detection from the shared provider stream SDK, reuses it in the DeepSeek plugin, and adds a provider-hook-runtime fallback plus tests and changelog for generic proxy providers.

## Best Possible Solution

Merge a narrowly gated fallback that uses the shared DeepSeek V4 wrapper only for unowned OpenAI-compatible DeepSeek V4 model refs, with regression tests and redacted real proxy run proof.

## Reproduction Assessment

Yes at source level: an unowned `models.providers` DeepSeek V4 model has no provider plugin wrapper on current main, so the replay payload never gets the `reasoning_content` patch. I did not establish a live proxy roundtrip in this read-only review.

## Solution Assessment

No as submitted: the shared wrapper and provider-hook fallback seam are the right direction, but the fallback should be gated to OpenAI-compatible resolved models and the PR needs real after-fix proxy proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Gate the fallback to OpenAI-compatible models:** `src/plugins/provider-hook-runtime.ts:335`
  - body: The new fallback runs for any unowned provider whose selected model id is `deepseek-v4-*`, even when the resolved model is not using the OpenAI-compatible chat transport. This wrapper mutates an OpenAI-style payload (`thinking`, `reasoning_effort`, and `messages[].reasoning_content`), so a custom Anthropic/Google/other transport that happens to reuse the same model id would get the wrong request shape. Check the resolved model API and only wrap the OpenAI-compatible path.
  - confidence: 0.82
