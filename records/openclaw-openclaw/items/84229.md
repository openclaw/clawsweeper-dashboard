## Summary

Keep this PR open: the LINE size-precheck direction is useful and the PR body includes credible live proof, but the patch still checks previewImageUrl with the 10 MiB original-image cap instead of LINE's 1 MiB preview-image cap, so a real LINE rejection path remains.

## What This Changes

The PR adds LINE outbound media HEAD size prechecks, cap constants, and regression tests, then wires them into LINE media resolution and send helpers.

## Best Possible Solution

Land the local LINE media size precheck after splitting original-content and preview-image caps, with focused coverage for the 1 MiB preview limit and an explicit decision on duplicate HEAD probe latency.

## Reproduction Assessment

Yes. Current main source has no Content-Length size precheck before LINE sends, and the PR body gives live after-fix proof for an over-cap original image; the remaining preview-cap defect is source-reproducible with a previewImageUrl between 1 MiB and 10 MiB.

## Solution Assessment

No, not yet. The precheck belongs in the LINE plugin boundary and reuses the right guarded-fetch primitive, but it must use LINE's separate 1 MiB preview-image cap before it is the best fix.
