## Solution Assessment

Mostly yes. Keeping the guard plugin-local and using the existing SSRF fetch helper is the maintainable boundary; the remaining choice is whether maintainers want adapter probe deduping before merge.

## Risks / Open Questions

- Known over-cap 200/206 Content-Length responses now fail locally instead of being attempted through LINE, so existing callers may see a synchronous error where they previously saw apparent success.
- Each guarded HEAD probe can add up to 5 seconds per checked media URL, and the LINE-specific adapter path can still probe in both resolveLineOutboundMedia and sendMessageLine.
- The gateway now performs a public HTTPS HEAD request before LINE fetches the media URL; fetchWithSsrFGuard mitigates private-network targets, but maintainers should accept the extra egress behavior.
- Generic quick-reply inline image media without LINE-specific options remains outside this PR, so full outbound LINE media coverage would need a follow-up.
