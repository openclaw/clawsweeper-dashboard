## Summary

Keep open. Current main has the official Brave package metadata and doctor recovery path, but the reported manual command still appears source-reproducible: packaged installs no longer have bundled `brave`, and bare `openclaw plugins install brave` falls through to the unrelated npm package named `brave`.

## What This Changes

The issue asks for Brave plugin install/reinstall to work after 2026.5.2 externalized Brave instead of resolving the unscoped `brave` npm package or a nonexistent beta tag.

## Best Possible Solution

Teach `plugins install` to resolve exact official external plugin ids such as `brave` through the official catalog to `@openclaw/brave-plugin`, while leaving explicit npm specs and existing doctor recovery behavior intact.

## Reproduction Assessment

Yes, from source and registry metadata. The current install command passes bare `brave` to npm when no bundled source exists, and npm currently resolves that to the unrelated `brave@0.0.9` package rather than the official `@openclaw/brave-plugin`.

## Solution Assessment

No, current main only solves the configured-provider repair path and documents the official package name; it does not make the reported manual install command work. The narrower maintainable fix is official-catalog id resolution before raw npm fallback, with `npm:` remaining npm-only.
