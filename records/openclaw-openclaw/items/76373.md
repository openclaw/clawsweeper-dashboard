Keep open. Current main has the correct official Brave package metadata and repair/setup paths, but the exact bare install commands in the report still route to the unrelated npm `brave` package instead of the official `@openclaw/brave-plugin` package.

The issue asks for the externalized Brave plugin to remain installable after upgrading, including bare `brave` or beta install attempts, instead of failing on npm package metadata.

Add a narrow official-external-catalog fallback for bare plugin ids such as `brave` when npm yields a non-OpenClaw package, while preserving explicit `npm:` semantics and valid npm plugin installs.

## Repair Work Prompt

Fix the Brave external plugin install regression from https://github.com/openclaw/openclaw/issues/76373. Current main routes `openclaw plugins install brave` and `brave@beta` to the unrelated npm `brave` package, which lacks `openclaw.extensions`; the official catalog maps plugin id `brave` to `@openclaw/brave-plugin`, and the scoped package has valid OpenClaw metadata. Add a narrow fallback in the plugin install path so bare official external plugin ids resolve to their catalog npm spec after npm yields a non-OpenClaw plugin/hook package or missing tag/package. Preserve explicit `npm:` behavior, explicit ClawHub behavior, local/git/marketplace paths, valid npm plugin installs, and the documented npm-first launch behavior for packages that are actually OpenClaw plugins. Preserve selectors where safe, e.g. `brave@beta` should try `@openclaw/brave-plugin@beta`. Add focused regression tests and a single-line CHANGELOG fix entry; do not change dependency pins, release metadata, or unrelated plugin externalization policy.
