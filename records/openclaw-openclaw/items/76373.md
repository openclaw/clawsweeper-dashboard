## Summary

Keep open. Current main still lets bare non-bundled `brave` fall through to raw npm install, and the linked open PR with closing syntax targets that exact install regression; main only has catalog metadata, docs, and recovery paths, not the manual command fix.

## What This Changes

The issue reports that after the 2026.5.2 Brave externalization, `openclaw plugins install brave` resolves to the unrelated npm package instead of the official `@openclaw/brave-plugin`, with follow-up upgrade and provider-registration failures.

## Best Possible Solution

Review and land PR #76390 or an equivalent catalog-id resolution fix, then verify a packaged Brave install reports the `webSearchProviders` contract and routes `web_search` before closing this issue.

## Reproduction Assessment

Yes. Source inspection shows `plugins install brave` still reaches raw npm fallback when Brave is not bundled, and the issue discussion includes concrete 2026.5.2 install and upgrade logs for the same path.

## Solution Assessment

Yes for the original install bug: resolving exact official external plugin ids through the official catalog before raw npm fallback is the narrowest maintainable fix, while preserving explicit `npm:` specs and scoped package installs.
