## Summary

Keep open. Current main partially supersedes the branch by sanitizing normal Discord final delivery, but the draft-preview final-edit bypass is still source-reproducible on current main and this PR’s implementation has blocking correctness and privacy issues plus no real behavior proof.

## What This Changes

The branch adds a Discord outbound sanitizer, applies it to final delivery and draft-preview finalization, and adds regression tests for leaked assistant-internal scaffolding.

## Best Possible Solution

Rebase around current reply-safety.ts, use sanitizeDiscordFrontChannelText on draft-preview final edits and normal delivery, preserve non-text payloads when text scrubs empty, and add focused tests plus redacted real Discord or runtime proof.

## Reproduction Assessment

Yes, by source inspection. A one-chunk Discord final payload containing leaked <thinking> text can still flow through resolvePreviewFinalText into a preview edit without sanitizeDiscordFrontChannelText on current main.

## Solution Assessment

No, not as submitted. The maintainable fix is to reuse the current strict Discord front-channel sanitizer and avoid the sanitized-empty media drop, rather than adding a separate stale sanitizer profile.
