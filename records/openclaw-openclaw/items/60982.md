## Summary

Keep open: this is a plausible Discord privacy bug fix, but the PR is not merge-ready because it overlaps current main's stricter reply-safety path, introduces a source-supported media-drop path, and still lacks real Discord/runtime proof.

## What This Changes

The PR adds a Discord outbound sanitizer, applies it to final delivery and draft-preview finalization, and adds regression tests for leaked thinking and memory scaffolding.

## Best Possible Solution

Rebase onto current main, use the existing strict Discord front-channel sanitizer for preview-final edits, preserve media and non-text fallback delivery, and add redacted Discord/runtime proof.

## Reproduction Assessment

Yes, by source inspection: current main's draft-preview finalizer can edit a final preview using text that only had inline directive tags stripped, bypassing the strict Discord delivery sanitizer. I did not run a live Discord session in this read-only review.

## Solution Assessment

No, not as submitted. The maintainable path is to reuse the existing Discord reply-safety sanitizer and keep sanitized-empty media or non-text finals on the normal delivery path.
