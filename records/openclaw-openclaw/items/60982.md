Keep open: the Discord privacy bug is still source-reproducible on the draft-preview finalization path, but this branch is stale against current `reply-safety.ts`, has merge-blocking correctness/security gaps, and still lacks real behavior proof.

## What This Changes

The PR adds a Discord outbound sanitizer, applies it to final reply delivery and draft-preview finalization, and adds regression tests for leaked internal scaffolding.

## Best Possible Solution

Rebase around current `reply-safety.ts`, use the strict Discord front-channel sanitizer on draft-preview final edits and normal delivery, preserve media/non-text payloads when text scrubs empty, and add focused tests plus real proof.

## Reproduction Assessment

Yes, by source inspection. On current main, a one-chunk final payload such as `<thinking>internal</thinking>\nVisible answer` can still be edited through the draft-preview finalization path without `sanitizeDiscordFrontChannelText`.

## Solution Assessment

No, not as submitted. The maintainable fix is to reuse the current strict Discord front-channel sanitizer on the active draft-preview path and keep non-text payload delivery intact, rather than adding a separate stale sanitizer profile.
