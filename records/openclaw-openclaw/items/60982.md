## Summary

Keep open: current main has shipped the normal Discord final-delivery sanitizer, but the draft-preview finalization path can still bypass that sanitizer and expose leaked `<thinking>` or memory scaffolding. The PR is directionally useful but should not merge as-is because the branch has privacy-sensitive sanitizer gaps and a media-loss edge case.

## What This Changes

The PR adds a Discord outbound sanitizer, wires it into final reply delivery and preview finalization, and adds regression tests for stripping leaked thinking and relevant-memory scaffolding.

## Best Possible Solution

Rebase the fix onto current main by applying the strict Discord front-channel sanitizer to draft-preview finalization as well as normal final delivery, preserving media and component payloads when text scrubs empty, with focused tests and real Discord or terminal proof.

## Reproduction Assessment

Yes, by source inspection. A one-chunk Discord final payload like `<thinking>internal</thinking>\nVisible answer` can still go through `resolvePreviewFinalText` and `buildFinalEdit`, bypassing `deliverDiscordReply` where current main performs the shipped sanitization.

## Solution Assessment

No, not as submitted. The right direction is to reuse the current `sanitizeDiscordFrontChannelText` or strict assistant-visible delivery sanitizer on the active draft-preview and shared delivery paths, rather than adding a separate sanitizer that preserves unfinished reasoning and can skip media finals.
