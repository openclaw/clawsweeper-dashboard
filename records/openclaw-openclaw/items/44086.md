Keep open. The reported OpenAI-compatible provider bug is still source-reproducible on current main, but this PR is stale against the current agent transport stack, misses the direct Chat Completions path, and lacks external-contributor real behavior proof before merge.

## What This Changes

Adds an embedded pi-runner OpenAI Chat Completions payload wrapper that rewrites assistant tool-call messages from `content: null` to `content: ""`, plus focused wrapper tests.

## Best Possible Solution

Port the null-to-empty assistant tool-call normalization to current main for both embedded and direct OpenAI-completions payloads, with focused regression coverage and redacted DeepSeek or equivalent provider proof.

## Reproduction Assessment

Yes, by source inspection. Current main can still pass converter output with an assistant tool-call message containing `content: null` into OpenAI-compatible completions payloads, and existing tests preserve that shape.

## Solution Assessment

Yes in shape, but not as submitted. A scoped outbound payload normalization is the narrow maintainable fix, but this branch needs a current-main port, direct completions coverage, and real provider proof before merge.
