## Summary

Keep open. The linked bug is source-reproducible and the PR addresses part of the provider compatibility failure, but the submitted branch is incomplete on current main because it only wraps the embedded stream path, uses a stale package import, and still lacks inspectable after-fix real provider proof.

## What This Changes

Adds an embedded-runner OpenAI Chat Completions payload wrapper that rewrites assistant tool-call messages from `content: null` to `content: ""`, plus focused wrapper tests.

## Best Possible Solution

Normalize assistant tool-call `content: null` at the shared OpenAI Chat Completions payload boundary so both direct and embedded transports emit string content, with regression coverage and redacted real-provider proof.

## Reproduction Assessment

Yes, source-level: `@earendil-works/pi-ai@0.74.1` can convert an assistant tool-call turn with no surviving text into `content: null`, and current main forwards that shape into Chat Completions payloads.

## Solution Assessment

No, not as submitted. Payload normalization is the right fix shape, but it should cover the shared/direct Chat Completions boundary, use current package names, and include real provider proof.
