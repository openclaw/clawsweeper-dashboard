## Summary

Keep open: the provider-compatibility bug is valid, but this branch is still incomplete for current main and lacks inspectable real provider proof before merge.

## What This Changes

Adds an embedded-runner OpenAI Chat Completions payload wrapper that rewrites assistant tool-call messages from `content: null` to `content: ""`, plus wrapper tests.

## Best Possible Solution

Normalize assistant tool-call `content: null` at the shared OpenAI Chat Completions payload boundary so both direct and embedded transports emit string content, with regression coverage and redacted real-provider proof.

## Reproduction Assessment

Yes, source-level: `@earendil-works/pi-ai@0.74.1` can convert an assistant tool-call turn with no surviving text into `content: null`, and current main forwards that shape into Chat Completions payloads. I did not establish a fresh live provider reproduction in this read-only review.

## Solution Assessment

No, not as submitted. Payload normalization is the right fix shape, but it should cover the shared/direct Chat Completions boundary, use current package names, and include real provider proof.
