## Summary

Keep open. The reported provider-compat bug is still reproducible from current main source, but this PR is not complete enough to merge because it only wraps the embedded runner path, still uses the old pi-agent-core package name in its added test, and lacks inspectable after-fix provider proof.

## What This Changes

The PR adds an embedded pi-runner payload wrapper and tests to rewrite OpenAI Chat Completions assistant tool-call messages from `content: null` to `content: ""`.

## Best Possible Solution

Port the null-to-empty assistant tool-call normalization to the shared OpenAI Chat Completions payload boundary so both direct and embedded paths run it before strict-key serialization, with focused regression tests and redacted real-provider proof.

## Reproduction Assessment

Yes, source-level. Current main can still turn an assistant tool-call replay with no surviving text into `content: null`, and an existing strict-key regression test preserves that null payload shape.

## Solution Assessment

No, not as submitted. A scoped outbound payload normalization is the right fix shape, but the branch needs a current-main port, direct completions coverage, package-name updates, and real provider proof before merge.
