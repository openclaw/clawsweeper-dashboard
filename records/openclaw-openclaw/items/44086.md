## Summary

Keep this PR open. Current main has adjacent string-only OpenAI-compatible content handling, but it still does not normalize assistant tool-call messages whose converted Chat Completions payload has `content: null`; the upstream `pi-ai` converter still creates exactly that shape after filtering blank assistant text. The branch is useful source work, but it is currently unmergeable against the refactored main branch and needs an adapted implementation plus the required changelog entry before merge.

## What This Changes

The PR adds an OpenAI Chat Completions payload wrapper and tests to convert assistant messages with tool calls from `content: null` to `content: ""` before provider submission.

## Best Possible Solution

Land an adapted version on current main that normalizes assistant Chat Completions tool-call messages from `content: null` to `""` at the shared OpenAI-compatible payload boundary, keeps the scope limited to tool-call assistant messages, adds regression coverage near the current wrapper/transport tests, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. A high-confidence code reproduction is an `openai-completions` replay turn whose assistant content contains only a blank text block plus a tool call: `pi-ai` 0.70.6 filters the blank text, leaves `content: null`, and still sends the assistant message because `tool_calls` exists.

## Solution Assessment

Yes in shape, but the branch is stale. A scoped payload normalization wrapper is a maintainable fix for the old base; on current main it should be adapted into the refactored post-plugin wrapper path, or into a small shared helper if maintainers want the OpenClaw transport builder covered at construction time too.
