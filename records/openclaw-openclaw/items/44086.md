## Summary

Keep open. Current main still has a source-reproducible path where OpenAI-compatible Chat Completions replay can carry an assistant tool-call message with `content: null`, and the nearby `strictMessageKeys` work does not normalize that value. The branch is a useful narrow fix shape, with supporting reporter context, but it still needs current-main port/maintainer validation and stronger real behavior proof before merge.

## What This Changes

The branch adds an OpenAI Chat Completions payload wrapper and tests that rewrite assistant messages with `tool_calls` from `content: null` to `content: ""`.

## Best Possible Solution

Normalize assistant tool-call replay content at the current OpenAI-compatible Chat Completions payload boundary after message conversion, with focused no-op/rewrite coverage and maintainer-owned changelog handling at landing.

## Reproduction Assessment

Yes. A high-confidence source reproduction is an `openai-completions` replay turn whose assistant message has only blank text plus a tool call; the pinned converter leaves `content: null`, and current main does not normalize that value before payload submission.

## Solution Assessment

Yes in shape, but not merge-ready as-is. A scoped payload wrapper is the narrow maintainable fix, but it should be ported onto the current wrapper chain and validated with real provider proof.
