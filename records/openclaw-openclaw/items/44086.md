The PR targets a real OpenAI-compatible provider replay bug and current main still lacks a null-to-empty-string normalization for assistant tool-call messages; however, the source branch needs adaptation to the current refactored wrapper path and the required changelog entry before it is merge-ready.

## What This Changes

The PR adds an OpenAI Chat Completions payload wrapper and tests to convert assistant messages with tool calls from `content: null` to `content: ""` before provider submission.

## Best Possible Solution

Land an adapted current-main version that normalizes assistant Chat Completions tool-call messages from `content: null` to `""` at the shared OpenAI-compatible payload boundary, preserves existing non-null content, excludes Responses APIs, adds regression coverage, and records the user-facing provider fix in the changelog.

## Reproduction Assessment

Yes. A high-confidence code reproduction is an `openai-completions` replay turn whose assistant content has only a blank text block plus a tool call; pi-ai 0.70.6 filters the blank text, leaves `content: null`, and current main has no wrapper that changes it before submission.

## Solution Assessment

Yes in shape, but not merge-ready as submitted. A scoped payload normalization wrapper is the narrow maintainable fix; it should be adapted to current main’s post-plugin wrapper composition, with direct transport coverage considered if that path can still emit the same payload shape.
