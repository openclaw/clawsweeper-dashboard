## Summary

Keep open: the reported provider compatibility bug is still source-reproducible, but this PR is not ready because it only patches the embedded wrapper path, uses the stale pi-agent-core package name in its new test, and still lacks inspectable after-fix real provider proof.

## What This Changes

The PR adds an embedded-runner OpenAI Chat Completions payload wrapper that rewrites assistant tool-call messages from `content: null` to `content: ""`, plus focused unit tests.

## Best Possible Solution

Normalize assistant tool-call `content: null` at the shared OpenAI Chat Completions payload boundary so both direct and embedded transports emit string content, with focused regression coverage and redacted real-provider proof.

## Reproduction Assessment

Yes, source-level: `@earendil-works/pi-ai@0.74.1` can convert an assistant tool-call turn with no surviving text into `content: null`, and current main forwards that shape into Chat Completions payloads.

## Solution Assessment

No, not as submitted. Payload normalization is the right fix shape, but it should live at the shared OpenAI Chat Completions boundary, cover the direct path, use current package names, and include real provider proof.
