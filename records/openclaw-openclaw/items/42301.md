## Summary

Keep this PR open: it addresses a real Feishu media-key over-filtering bug that is still present on current main and the latest release, but the current head is not merge-ready because it accepts the literal `..` key and lacks after-fix real Feishu image-analysis proof.

The PR changes Feishu external-key validation to allow opaque image/file keys containing embedded `..` substrings and adds focused validator regression tests.

## Best Possible Solution

Land a minimal Feishu plugin validator update that accepts embedded `..` opaque keys while rejecting path separators and the exact `..` token, backed by focused tests and redacted live Feishu image-analysis proof.

## Reproduction Assessment

Yes, by source inspection: current main and `v2026.5.12` reject embedded-dot-dot keys in `normalizeFeishuExternalKey`, and inbound Feishu image/file parsing calls that helper before media download. I did not establish a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No. The owner-module fix is the right narrow direction, but the current implementation needs an exact `..` rejection plus real Feishu proof before it is the best mergeable solution.
