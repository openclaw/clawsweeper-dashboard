## Summary

Keep this PR open. Current main and the latest release still reject legitimate Feishu media keys containing embedded `..`, so the requested behavior is not implemented, but the latest PR diff still needs one small security hardening step before merge because it would allow the literal `..` key into Feishu media API path parameters.

## What This Changes

The PR changes the Feishu external-key validator to allow opaque image/file keys containing embedded `..` and adds regression tests for accepted embedded-dot keys plus rejected path-separator inputs.

## Best Possible Solution

Keep the fix in the bundled Feishu plugin and make it minimal: preserve empty, length, control-character, slash, and backslash rejection; allow embedded `..` inside otherwise opaque keys; explicitly reject exact `..`; and add regression tests for both the accepted embedded-key examples and rejected separator/dot-segment cases.

## Reproduction Assessment

Yes. A focused reproduction is `normalizeFeishuExternalKey("img_v2_test..key")` on current main returning `undefined`, plus the inbound parser paths that drop normalized image/file keys before media analysis.

## Solution Assessment

No. The PR is aimed at the right owner module and fixes the broad rejection, but the current diff is not the best final patch because it omits exact `..` rejection; the safer version is the same narrow change plus a dot-segment regression test.
