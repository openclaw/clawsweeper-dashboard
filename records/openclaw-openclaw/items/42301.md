## Summary

Keep open: this PR is aimed at a real Feishu media-key over-filtering bug, but the latest patch still weakens fail-closed validation by accepting the literal `..` key and lacks real Feishu behavior proof.

## What This Changes

The PR removes the broad `..` substring rejection from Feishu external-key normalization while keeping slash/backslash rejection and adds regression tests for embedded dot-dot keys.

## Best Possible Solution

Amend the Feishu validator to allow embedded `..` in opaque keys while still rejecting exact `..`, path separators, empty/control/oversized values, then back it with focused tests and real Feishu proof.

## Reproduction Assessment

Yes. Source inspection on origin/main shows `img_v2_test..key` is rejected by `normalized.includes("..")`, and Feishu image/file keys flow through that helper before download.

## Solution Assessment

No. The owner-module change is the right narrow direction, but the latest branch should still reject exact `..` and include real after-fix Feishu evidence before merge.
