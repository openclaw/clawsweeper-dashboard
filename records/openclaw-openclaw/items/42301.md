## Summary

Keep open: current main still has the reported Feishu external-key over-filtering, and prior review feedback was partly addressed by preserving separator rejection, but the final PR still accepts the literal `..` token and lacks real Feishu behavior proof.

## What This Changes

This PR removes `normalizeFeishuExternalKey`'s broad `..` substring rejection while keeping slash/backslash rejection and adds regression tests for embedded-dot-dot Feishu keys.

## Best Possible Solution

Land a minimal Feishu plugin validator update that allows embedded `..` opaque keys while rejecting literal `..` and path separators, with focused tests and redacted real Feishu proof.

## Reproduction Assessment

Yes, by source inspection: current main rejects embedded-dot-dot keys in `normalizeFeishuExternalKey`, and parsed inbound image/file keys call that helper before media download. I did not establish a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No. The owner-module fix is the right narrow direction, but this patch should still reject the standalone `..` token, add regression coverage for it, and include real after-fix Feishu evidence before merge.
