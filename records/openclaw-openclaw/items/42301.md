## Summary

Keep open: current main still has the Feishu media-key bug, and this PR is a plausible narrow fix, but the latest head still accepts the literal `..` token and lacks after-fix proof from a real Feishu flow.

## What This Changes

The PR removes the broad `..` substring rejection from Feishu external-key normalization, keeps path separator rejection, and adds validator regression tests for embedded dot-dot keys.

## Best Possible Solution

Amend the Feishu validator to allow embedded `..` in opaque keys while still rejecting separators, exact `..`, empty/control/oversized values, then back it with focused tests and real Feishu proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `img_v2_test..key` is rejected by `normalized.includes("..")`, and Feishu image/file keys flow through that helper before download.

## Solution Assessment

No. The owner-module change is the right narrow direction, but the branch should still reject exact `..` and include real after-fix Feishu evidence before merge.
