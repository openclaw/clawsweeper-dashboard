## Summary

Keep open: this is useful additive Gateway/SDK API work, but the current PR head still skips plugin ownership validation on idempotent same-decision replays and the supplied proof is test/mock-oriented rather than a real Gateway/plugin approval flow.

## What This Changes

The PR adds an admin-scoped `plugin.approval.resolveVerified` Gateway RPC, a narrow plugin SDK runtime helper, protocol/Swift/docs/API baseline updates, and focused Gateway/SDK tests.

## Best Possible Solution

Land the narrow verified resolver and SDK helper only after plugin ownership is enforced for both pending and resolved-repeat paths, with real Gateway/plugin approval proof and the companion host API PRs aligned.

## Reproduction Assessment

Not applicable as a current-main bug; this is additive API work. The PR defect is source-reproducible by resolving an approval once with the owning `pluginId`, then replaying the same id and decision through `plugin.approval.resolveVerified` with a different `pluginId`.

## Solution Assessment

No, not yet. The narrow public helper and admin-scoped resolver are the right direction, but the ownership validator must also run on idempotent replay before this security-sensitive API is merged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Enforce plugin ownership on replayed verified approvals:** `src/gateway/server-methods/plugin-approval.ts:253-259`
  - body: `resolveVerified` relies on `validateDecision` for the `pluginId` ownership check, but `handleApprovalResolve` returns success for an already-resolved approval with the same decision before running that validator. Because this method is admin-scoped and admin clients can see all approval records, a caller can replay a resolved id with another `pluginId` and still get `{ ok: true }`; apply the owner check to the resolved-repeat path and add a regression test.
  - confidence: 0.89
