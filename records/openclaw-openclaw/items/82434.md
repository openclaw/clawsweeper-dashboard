## Summary

Keep open: the external AgentKit/ClawHub use case is concrete and current main does not already provide this verified-resolution API, but the current patch still has a security-boundary blocker because the claimed plugin owner is supplied by the caller rather than bound to authenticated caller identity.

## What This Changes

The branch adds `externalResolution` metadata for plugin approvals, a new `plugin.approval.resolveVerified` Gateway/protocol/SDK path, and renderer/docs/tests for external verification commands.

## Best Possible Solution

Land the generic external verification seam only after verified resolution is bound to authenticated caller/plugin identity, or redesign, name, document, and test it as an operator-scoped trust model.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create an external-resolution approval for plugin A, connect a generic trusted approval-runtime client, and call `plugin.approval.resolveVerified` with plugin A's id because the handler trusts request params.

## Solution Assessment

No. The narrow `externalResolution` presentation shape is plausible, but the resolver is not the best implementation until ownership is bound to authenticated caller/plugin identity or explicitly changed to an operator-scoped contract.
