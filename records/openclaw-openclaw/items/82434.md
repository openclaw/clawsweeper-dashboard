## Summary

Keep open: this is a live feature/API PR for a missing external HITL approval seam, but the current patch is not merge-ready because verified allow resolution is described as plugin-owned while the server trusts a caller-supplied `pluginId` over an `operator.approvals` client.

## What This Changes

Adds an `externalResolution` plugin approval field and a `plugin.approval.resolveVerified` Gateway/SDK path, with renderer, protocol, docs, TUI, Control UI, and channel updates for external verification commands.

## Best Possible Solution

Land the generic external verification seam only after verified resolution is bound to authenticated plugin/caller identity, or deliberately redesign, name, document, and test it as operator-scoped trust while keeping AgentKit itself outside core.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create an external-resolution approval for plugin A, then call `plugin.approval.resolveVerified` with plugin A's stored id and plugin id over an approvals client because the handler trusts request params.

## Solution Assessment

No. The narrow `externalResolution` presentation shape is plausible, but the resolver is not the best implementation until ownership is bound to authenticated plugin/caller identity or explicitly changed to an operator-scoped contract.
