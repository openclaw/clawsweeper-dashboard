## Summary

The external-verification flow is useful and has credible runtime proof, but this patch must not merge as written because its ownership check trusts the caller to name the plugin it wants to impersonate. Likely related people: kevinslin for the active approval architecture decision, cornna for recent central-handler work, and Guardiola31337 for the previously merged approval-action contract.

## What This Changes

The PR adds external-verification commands to plugin approvals and an admin-scoped Gateway/SDK resolver that allows externally verified decisions across UI and channel surfaces.

## Best Possible Solution

Define a server-authenticated plugin principal or mint an unforgeable approval-scoped capability when the request is created, require that identity or capability for verified resolution, and land the smallest stable SDK/protocol surface after the approval-overhaul decision.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence path: an `operator.admin` client can call the new method with the pending approval ID and its stored plugin ID because neither value is bound to an authenticated plugin principal.

## Solution Assessment

No. The narrow external-verification concept is preferable to arbitrary custom actions, but the resolver must use authenticated ownership or an unforgeable approval capability rather than a caller-supplied plugin ID.
