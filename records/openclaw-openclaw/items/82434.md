## Summary

Keep open: this is plausible additive Gateway/SDK API work for external HITL plugins, but the patch needs maintainer API/security approval, stronger real behavior proof, and a fix for the verified resolver’s owner-check gap on same-decision idempotent replays.

## What This Changes

Adds an admin-scoped Gateway RPC and plugin SDK runtime helper for verified plugin-owned approval resolution, plus protocol, Swift, docs, API baseline, and focused test updates.

## Best Possible Solution

Land the narrow admin-scoped verified resolver and SDK helper only after the pluginId ownership contract covers first and repeat resolution paths and the companion host API PRs align.

## Reproduction Assessment

Not applicable as a current-main bug; this is additive Gateway/SDK API work. The patch defect is source-reproducible by resolving an approval once with the owning pluginId, then replaying the same id and decision through `plugin.approval.resolveVerified` with a different pluginId.

## Solution Assessment

No, not yet. The narrow public helper is the right direction, but the owner check should also apply to idempotent resolved-repeat paths before this admin-scoped API is merged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Enforce ownership on verified approval replays:** `src/gateway/server-methods/plugin-approval.ts:245-253`
  - body: The new verified resolver relies on `validateDecision` for the pluginId ownership check, but `handleApprovalResolve` returns `{ ok: true }` for an already-resolved approval with the same decision before running that validator. Because this method uses an admin-scoped client that can see all approvals, a caller with another pluginId can replay a resolved id and receive success. Apply the owner check to the resolved-repeat path too and add a regression test.
  - confidence: 0.87
