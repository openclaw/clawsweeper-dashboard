## Summary

Keep open: this is useful additive Gateway/SDK API work that current main does not implement, but the PR still has a concrete owner-check gap on idempotent verified approval replays and the supplied behavior proof is mock-only.

## What This Changes

The PR adds an admin-scoped `plugin.approval.resolveVerified` Gateway method, a narrow plugin SDK helper, protocol/docs/Swift/API updates, and focused tests for plugin-owned verified approval resolution.

## Best Possible Solution

Land the narrow resolver and SDK helper only after repeat-resolution ownership validation is enforced, real Gateway proof is supplied, and the generic admin client remains internal.

## Reproduction Assessment

Yes for the PR defect by source inspection: resolve an approval once with the owning pluginId, then replay the same id and decision through the new method with another pluginId. Not applicable as a current-main bug because the new API is not on main.

## Solution Assessment

No, not yet. The narrow public helper is the right direction, but the owner check also needs to cover idempotent resolved-repeat paths before this admin-scoped API is merged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Enforce ownership on verified approval replays:** `src/gateway/server-methods/plugin-approval.ts:245-253`
  - body: The verified resolver relies on `validateDecision` for the `pluginId` ownership check, but `handleApprovalResolve` returns success for an already-resolved approval with the same decision before running that validator.
