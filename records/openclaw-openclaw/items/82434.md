## Summary

Keep open: the PR adds useful Gateway/SDK API surface that current main does not have, but the latest head still skips the pluginId ownership check on idempotent resolved replays and the supplied external proof is mock/CI-oriented for this exact Gateway method.

## What This Changes

Adds an admin-scoped `plugin.approval.resolveVerified` Gateway method, a narrow plugin SDK runtime helper, protocol/Swift/docs/API baseline updates, and focused approval/scope tests.

## Best Possible Solution

Land the narrow verified resolver and SDK helper only after pluginId ownership is enforced for both first resolution and idempotent replay, with regression coverage and real Gateway/plugin proof.

## Reproduction Assessment

Yes for the PR defect: resolve a pending approval once with the owning pluginId, then replay the same id and decision through `plugin.approval.resolveVerified` with a different pluginId. Source inspection shows the replay returns success before the plugin ownership validator runs.

## Solution Assessment

No, not yet. The narrow public helper is the right direction, but the shared resolver must apply the plugin owner check to resolved-repeat paths before this admin-scoped API is mergeable.
