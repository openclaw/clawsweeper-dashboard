## Summary

Keep open: current main does not have the new `externalResolution` or `plugin.approval.resolveVerified` surface, and this PR is meaningful API work. It is not merge-ready because the verified resolver is documented as plugin-owned while the implementation trusts a caller-supplied `pluginId` over a generic approvals client.

## What This Changes

The PR adds an external plugin approval verification path across Gateway protocol, SDK exports, approval rendering, TUI/web UI parsing, docs, and several channel adapters.

## Best Possible Solution

Land the generic external verification seam only after verified resolution is bound to authenticated plugin/caller identity, or explicitly rename, document, and test it as an operator-scoped resolver.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create an external-resolution approval for plugin A, connect a trusted approval-runtime client, and call `plugin.approval.resolveVerified` with plugin A's id because the handler trusts request params while bypassing client visibility.

## Solution Assessment

No. The narrow `externalResolution` shape is plausible, but the resolver is not the best implementation until ownership is bound to authenticated caller/plugin identity or the contract is explicitly changed to operator-scoped trust.
