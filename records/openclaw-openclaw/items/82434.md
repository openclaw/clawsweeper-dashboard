## Summary

Keep open: this is useful additive Gateway/SDK API work, but the current PR still skips plugin ownership validation on the idempotent same-decision replay path and the supplied proof is still CI/test-oriented rather than a real plugin/Gateway resolution run.

## What This Changes

Adds an admin-scoped `plugin.approval.resolveVerified` Gateway method, a narrow plugin SDK runtime helper, protocol/Swift/docs/API baseline updates, and focused Gateway approval tests.

## Best Possible Solution

Land the narrow verified resolver and SDK helper only after the pluginId ownership contract covers both pending and already-resolved repeat paths and the companion host API PRs remain aligned.

## Reproduction Assessment

Not applicable as a current-main bug because this is additive API work. The PR defect is source-reproducible by resolving once with the owning `pluginId`, then replaying the same id and decision through `plugin.approval.resolveVerified` with a different `pluginId`.

## Solution Assessment

No, not yet. The narrow public helper is the right direction, but the owner check must also run for idempotent resolved-repeat paths before this admin-scoped API is mergeable.
