## Summary

Keep open: this is a meaningful implementation path for the still-open external HITL plugin approval API work, but the current patch is not merge-ready because the verified resolver is presented as plugin-owned while trusting a caller-supplied `pluginId`.

## What This Changes

The PR adds `externalResolution` approval metadata, a `plugin.approval.resolveVerified` Gateway/protocol/SDK path, TUI/channel/UI rendering, docs, and focused tests for external plugin verification commands.

## Best Possible Solution

Settle the approval trust model, then either bind verified resolution to authenticated plugin/caller identity with a cross-plugin spoof regression test, or explicitly rename/document/test it as operator-scoped trust before landing.

## Solution Assessment

No. The narrow `externalResolution` shape is plausible, but the resolver is not the best implementation until ownership is bound to authenticated caller/plugin identity or explicitly changed to an operator-scoped contract.
