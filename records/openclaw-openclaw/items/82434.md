## Summary

Keep this PR open for revision, but do not merge the current API: the external-verification use case is credible and has useful real Gateway proof, yet the resolver’s ownership guarantee is not authenticated, the branch conflicts with current main, and approval owners are actively reconsidering the permanent architecture.

## What This Changes

The PR adds external verification command metadata and an admin-scoped verified resolver for pending plugin approvals across Gateway protocol, plugin SDK, UI, docs, and channel renderers.

## Best Possible Solution

Give the originating plugin an unforgeable host-issued per-approval capability, or resolve through a host-authenticated plugin runtime that derives plugin identity server-side; expose only that narrow seam, add cross-plugin impersonation coverage, and then rebase it onto the chosen approval architecture.

## Reproduction Assessment

Yes at source level: the exported helper lets its caller select the `pluginId`, and the handler compares only that supplied value with the pending approval record. A live exploit was not run because this was a read-only review.

## Solution Assessment

No. The narrow presentation model is preferable to arbitrary custom actions, but the resolver must authenticate or mint the plugin’s ownership identity rather than accept it as an ordinary RPC parameter.
