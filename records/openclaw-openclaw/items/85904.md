## Summary

Keep open: the fix heads in the right direction, but the active-run thread guard still misses the normal dispatch-owned and queued-followup ReplyOperation paths, so the real Slack DM overlap path can still cross-steer. The PR body now has local terminal/test output, but it still does not show a real Slack workspace run after the patch.

## What This Changes

The branch adds routed thread metadata to reply operations, blocks steer mode when active and current routed thread ids differ, and extends Slack DM session-key and auto-reply regression coverage.

## Best Possible Solution

Keep the stable Slack DM SessionKey contract, but make every ReplyOperation owner carry the routed thread id before relying on thread-aware steering suppression, then prove the Slack DM overlap path end to end.

## Reproduction Assessment

Yes at source level: current main's steer decision ignores routed thread ids while Slack DMs keep one stable SessionKey and distinct MessageThreadId values. I did not run tests because this was a read-only review.

## Solution Assessment

No. The active-run boundary is the right place to fix this, but the PR must populate routeThreadId in every ReplyOperation creation path and cover the provided-operation case before merge.
