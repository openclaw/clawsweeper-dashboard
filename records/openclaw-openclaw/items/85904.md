## What This Changes

The PR records routed thread ids on reply operations and prevents same-session Slack DM turns with different Slack message thread ids from steering into the currently active run.

## Best Possible Solution

Land the routed-thread metadata guard after maintainers accept the generic rule that known different routeThreadId values should not steer into the active run, while keeping Slack's stable DM SessionKey contract and the new regression tests.

## Reproduction Assessment

Yes, at source level. Current main keeps Slack DMs on a stable session while carrying MessageThreadId, and active steering only checks the session lane; the PR body also reports the before-fix assertion failure for the same-session/different-thread case.

## Solution Assessment

Yes, with maintainer acceptance of the generic scope. The PR fixes the active-run routing boundary that caused cross-thread steering instead of changing Slack session-key generation, which would break the documented stable DM contract.
