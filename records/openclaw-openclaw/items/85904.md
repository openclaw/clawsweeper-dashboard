## Summary

Keep open: the updated PR now preserves the stable Slack DM session contract, but its new active-run thread guard is bypassed on the normal dispatch-owned operation path, and the proof is still test-only rather than a real Slack run.

## What This Changes

The branch adds routed-thread tracking to reply operations, suppresses steer mode when an active run and current turn have different thread ids, preserves stable Slack DM session-key tests, and adds a regression test for overlapping top-level Slack DMs.

## Best Possible Solution

Propagate routed thread ids into every active ReplyOperation owner, especially dispatch-owned and queued follow-up operations, then add regression coverage for the provided-replyOperation path and redacted real Slack proof.

## Reproduction Assessment

Yes at source-test level: current main has no routed-thread check before steer mode, and the PR's regression demonstrates the intended failing condition. A real Slack reproduction is still missing.

## Solution Assessment

No as submitted. The direction is now the right boundary, but the implementation only records routeThreadId for operations created in agent-runner and misses normal dispatch-owned operations.
