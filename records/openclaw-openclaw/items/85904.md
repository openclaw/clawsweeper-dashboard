## Summary

This PR still appears necessary: current main preserves Slack DM routed-thread metadata but active-run steering still does not compare the current Slack thread with the active reply operation. The patch is focused, has live Slack proof, and I found no blocking code-review findings, but merge should remain a maintainer decision because it intentionally changes Slack same-session/different-thread steering behavior and the latest head still had CI checks failing or in progress during review.

## What This Changes

The PR stores routed thread ids on reply operations, adds a Slack-scoped active-steering guard for direct-message routed threads, and covers Slack plus non-Slack routing behavior in focused tests.

## Best Possible Solution

Land the Slack-scoped routed-thread guard once maintainers accept the Slack DM steering compatibility change and the current head has clean required checks.

## Reproduction Assessment

Yes at source level. Current main keeps Slack DM turns on a stable direct SessionKey while carrying per-turn Slack thread metadata, but active steering is decided without routeThreadId comparison; the PR body also reports before-fix regression failures for that path.

## Solution Assessment

Yes, with maintainer acceptance of the compatibility change. The patch keeps the existing Slack DM SessionKey contract, carries routed thread facts into the reply-operation registry, and scopes the new steering guard to Slack direct routed-thread turns so non-Slack same-session steering remains unchanged.
