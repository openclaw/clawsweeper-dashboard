## Summary

This PR is still useful and should stay open for maintainer review: current main still lacks routed-thread metadata on active reply operations, while the PR provides a focused fix, regression coverage, and live Slack proof. The remaining decision is whether maintainers accept the generic route-thread guard for all channels that carry routed thread ids.

## What This Changes

The branch stores routed delivery thread ids on active reply operations, prevents active steering when the current turn has a different known route thread id, and adds Slack/auto-reply regression coverage.

## Best Possible Solution

Land the routed-thread metadata guard after maintainers accept the generic rule that known different routeThreadId values should not steer into the active run, while keeping Slack's stable DM SessionKey contract and the new regression tests.

## Reproduction Assessment

Yes, at source level. Current main keeps Slack DMs on one stable direct SessionKey while the active-steering path is session-key based; the PR also reports a before-fix assertion failure for same-session, different-thread steering.

## Solution Assessment

Yes, with maintainer acceptance of the generic scope. Recording routed thread ids at reply-operation creation and checking them at the steering boundary is the narrowest fix that preserves the Slack DM SessionKey contract.
