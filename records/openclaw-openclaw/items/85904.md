## Summary

Keep open: the intended boundary is now better than the earlier branch because stable Slack DM sessions are preserved, but the patch still misses the normal dispatch-owned ReplyOperation path, so a real Slack DM run can remain untagged and still accept cross-thread steering.

## What This Changes

The PR adds routed-thread tracking to reply operations, gates active-run steering on matching thread ids, and adds Slack/auto-reply regression coverage for same-session DM turns on different Slack threads.

## Best Possible Solution

Carry the routed thread id through every ReplyOperation creation path, especially dispatch-owned pre-run operations, then prove it with a Slack prepare-to-dispatch regression and redacted live Slack proof.

## Reproduction Assessment

Yes for a source-backed local path: current main keeps Slack top-level DMs on one stable session while active-run steering checks only session identity. The PR body includes before-fix regression output for that path, but no live Slack workspace reproduction is included.

## Solution Assessment

No. The chosen boundary is close, but the implementation only tags operations created in runReplyAgent; normal dispatch-created operations remain untagged, so thread matching still accepts every thread.
