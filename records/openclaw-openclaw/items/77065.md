## Summary

Keep open: the PR targets a real plugin lifecycle problem from #77039, but the patch can restore stale corpus supplements for plugins that are no longer active and its regression test does not exercise the advertised service.start registration path.

## What This Changes

The PR snapshots memory corpus supplements before activating plugin reloads, restores supplements not re-registered afterward, adds a loader regression test, and updates the changelog.

## Best Possible Solution

Define the plugin SDK lifecycle first, then either keep corpus supplement registration synchronous in register(api) with docs/diagnostics or add an explicit service-time registration seam with disable, reload, and failure cleanup coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows activating loads clear memory state, the guarded registration API closes when register(api) returns, and services start later with only a service context; the PR test does not exercise that path.

## Solution Assessment

No. The proposed restore loop is not the narrowest maintainable fix because it can preserve supplements for plugins that are no longer active, and it does not prove or implement the advertised service.start registration behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Restrict restored supplements to live plugins:** `src/plugins/loader.ts:2398-2399`
  - body: This restores every pre-load corpus supplement whose pluginId did not re-register during the new load. If the activating reload disables, omits, or fails the plugin that previously owned a supplement, the old corpus entry is re-added anyway, so memory_search can still query a plugin that is no longer active. Please tie restoration to plugins that actually loaded and still own the live supplement lifecycle, with disable/failure coverage.
  - confidence: 0.9
- **[P2] Exercise service.start instead of seeding memory state:** `src/plugins/loader.test.ts:2533`
  - body: The new test calls registerMemoryCorpusSupplement directly before reload, so it does not reproduce a plugin registering from service.start. On current main, register(api) is guarded and closed before startPluginServices later invokes service.start, so a captured api.registerMemoryCorpusSupplement call currently no-ops; this test can pass while the reported plugin behavior remains broken.
  - confidence: 0.87
- **[P3] Correct the changelog PR reference:** `CHANGELOG.md:13`
  - body: The added changelog line cites (#77060), but this pull request is #77065. That would ship a misleading PR reference for readers trying to audit the fix.
  - confidence: 0.86
