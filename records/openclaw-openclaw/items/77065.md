## Summary

Keep open: the PR still has blocking correctness issues because the restore path can preserve stale corpus adapters and the asserted `service.start()` lifecycle does not match the current guarded registration API.

## What This Changes

The PR snapshots memory corpus supplements before activating plugin reloads, restores prior supplements for still-loaded plugins that did not re-register, adds loader regression tests, and updates the changelog.

## Best Possible Solution

Define the memory supplement lifecycle contract first, then either keep corpus supplement registration synchronous in register(api) with diagnostics for service-time calls or add a service-context registration seam with cleanup tied to service stop/reload.

## Reproduction Assessment

Yes for the review blockers: source inspection shows captured register(api) calls are closed before services start, and the PR restore loop can re-add prior state solely by loaded plugin id. I did not run tests because this review was read-only and pnpm is not available in the environment.

## Solution Assessment

No. Restoring old supplements by plugin id is broader than the lifecycle being fixed and can preserve stale adapters; a lifecycle-owned registration/cleanup path or explicit register-only contract is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Avoid restoring supplements a plugin no longer registers:** `src/plugins/loader.ts:2401-2402`
  - body: The restore loop re-adds any previous corpus supplement when the plugin is still loaded and did not register a new one. If config or updated code leaves the plugin active but intentionally stops registering that corpus, the old adapter is kept alive after `clearActivatedPluginRuntimeState()`, so memory search can continue using stale plugin state. Tie restoration to an actual live registration/service owner, or require the plugin to re-register the supplement.
  - confidence: 0.87
- **[P2] Cover the real service-start lifecycle:** `src/plugins/loader.test.ts:2528-2536`
  - body: This test uses a module-level flag inside `register(api)`, while the reported path registers from `service.start()` through a captured API. Current main closes the guarded registration API when `register()` returns, and `OpenClawPluginServiceContext` has no `registerMemoryCorpusSupplement`, so this can pass without proving the service-start scenario is fixed.
  - confidence: 0.91
