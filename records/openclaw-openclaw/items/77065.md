## Summary

Keep open; the report is valid and the PR is aimed at the right loader/memory-state area, but the current restore guard can resurrect stale corpus supplements for any plugin that still has a service, even when that plugin intentionally stopped registering the supplement. The changelog entry also still points at unrelated #77060 instead of this PR or #77039.

## What This Changes

The PR snapshots memory corpus supplements before activating plugin reloads, restores non-re-registered supplements for service-owning plugins, adds loader regression tests, and adds a changelog entry.

## Best Possible Solution

Make corpus supplement restoration provenance-aware so only supplements actually registered from service.start survive an activating reload, while register-time or config-disabled supplements are allowed to disappear.

## Reproduction Assessment

Yes, source-level. Current main clears memory plugin state on activating loads, service.start registrations happen after registry loading, and lazy registry loading can re-run register() without starting services, matching the linked #77039 trace.

## Solution Assessment

No. Capturing pre-load supplements is the right direction, but restoring solely because the plugin owns any service is too broad; the fix should track service-start provenance or replay that lifecycle specifically, and the changelog reference needs correction.

## Review Findings
