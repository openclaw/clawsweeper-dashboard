## Summary

Keep this PR open: the reported loader bug is real from source inspection, but the proposed restore loop is overbroad and can keep stale corpus supplements reachable after the owning plugin is no longer loaded.

## What This Changes

The PR snapshots and restores memory corpus supplements around activating plugin reloads, adds a regression test for #77039, and adds a changelog entry.

## Best Possible Solution

Preserve supplements only for plugin owners that are still loaded and still have the relevant service/runtime ownership after the activating reload, while continuing to clear entries for removed, disabled, disallowed, or failed plugins.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an activating loader pass clears memory plugin state, while service callbacks are only started outside the loader, so a `service.start()`-registered supplement can be lost on a later activating reload.

## Solution Assessment

No. The patch targets the right bug but restores previous supplements too broadly; the maintainable fix should scope restoration to still-loaded service-owning plugins in the new registry.

## Review Findings

## Repair Work Prompt

Repair PR #77065 for #77039. Keep the intended behavior: corpus supplements registered by a still-running plugin `service.start()` callback survive an activating reload that does not re-run the service. Do not restore supplements for plugin IDs absent from the new activated registry, disabled/disallowed/removed plugins, failed plugins, or plugins that no longer own the relevant service. Prefer deriving an eligible owner set from the newly loaded registry services/plugins before restoring pre-load supplements. Add or update focused tests covering both survival for a still-loaded service-owning plugin and clearing for a removed or empty plugin load. Also fix the changelog PR reference if it remains wrong. Do not change public SDK docs unless the contract wording changes.
