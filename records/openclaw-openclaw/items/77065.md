## Summary

Keep open: the patch still does not prove or fix the reported `service.start()` lifecycle, and the restore loop can preserve stale corpus adapters for still-loaded plugins that intentionally stop registering one.

## What This Changes

The PR snapshots memory corpus supplements before activating plugin reloads, restores prior supplements for loaded plugins that did not re-register one, adds loader regression tests, and updates the changelog.

## Best Possible Solution

Define the memory supplement lifecycle explicitly, then either add a service-context registration owner with stop/reload cleanup or document and diagnose register-only usage with direct regression coverage.

## Reproduction Assessment

Yes. Source inspection of current main shows the guarded registration API closes before `startPluginServices` runs, while the PR test exercises `register(api)` rather than an actual `service.start()` callback.

## Solution Assessment

No. Restoring old supplements by loaded plugin id is broader than the reported lifecycle and can keep stale adapters alive; a service-owned lifecycle seam or explicit register-only diagnostic would be safer.
