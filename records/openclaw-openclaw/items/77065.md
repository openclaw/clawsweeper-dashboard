## Summary

Keep this PR open: it targets a real plugin memory lifecycle problem, but the current patch still does not prove or fix the claimed `service.start()` path, can restore stale corpus adapters for service-owning plugins, and the changelog entry still points at an unrelated issue.

## What This Changes

The PR snapshots pre-load memory corpus supplement registrations during activating plugin reloads, restores entries for service-owning plugins that did not re-register, adds loader regression tests, and adds a changelog entry.

## Best Possible Solution

Define the intended SDK lifecycle first, then either add an explicit service-time registration seam with provenance-aware restoration or keep corpus supplement registration strictly in `register(api)` and update the fix/tests around that contract.

## Reproduction Assessment

No for the exact service-start reload path. Source inspection shows current main clears corpus state on activating loads, but a real `service.start()` callback does not receive the registration API and a captured `register(api)` proxy is closed after register, so the PR's test does not reproduce the claimed lifecycle.

## Solution Assessment

No. Snapshotting pre-load supplements is directionally related, but this implementation is not provenance-aware and does not exercise the actual `service.start()` contract; the narrower maintainable fix needs an explicit lifecycle decision.

## Review Findings

Full review comments:

- **[P2] Exercise the real service.start lifecycle:** `src/plugins/loader.test.ts:2529-2544`
  - body: The new regression test registers the supplement inside `register(api)` and only uses `registerService` as a marker. On current main, the API passed to `register(api)` is a proxy that is closed as soon as register returns, and `service.start(ctx)` does not receive a registration API, so a real service-start registration will not populate `preLoadCorpusSupplements` for this restore loop to preserve. Please either add a supported service-time registration path and test it through `startPluginServices()`, or scope the fix away from `service.start()`.
  - confidence: 0.84

  - body: This guard still treats every previous supplement from a plugin with any registered service as service-start-owned. A plugin can keep a service while intentionally dropping a register-time corpus supplement on reload, and this loop will re-add the stale adapter because `servicePluginIdSet` still contains the plugin id.
  - confidence: 0.87
