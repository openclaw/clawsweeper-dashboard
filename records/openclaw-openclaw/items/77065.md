## Summary

Keep open: the related report is real, but this PR is not merge-ready because the loader restore is broader than the active plugin lifecycle and the regression test bypasses the actual service.start registration path.

## What This Changes

The PR snapshots memory corpus supplements before activating plugin loads, restores prior supplement registrations that are not re-registered, adds a loader regression test, and updates the changelog.

## Best Possible Solution

Define the lifecycle contract, then either keep corpus supplement registration synchronous in register(api) with docs/diagnostics or add an explicit service-time registration seam tied to loaded plugin cleanup, with disable/reload/failure coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main closes the guarded plugin API after register(api), while service.start runs later through startPluginServices. The PR test bypasses that path by directly seeding the internal corpus supplement registry.

## Solution Assessment

No. The proposed restore loop is not the narrowest maintainable fix because it can preserve supplements for plugins that are no longer active, and it does not prove or implement the advertised service.start registration behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Restrict restored supplements to active plugins:** `src/plugins/loader.ts:2398-2399`
  - body: This restores every pre-load corpus supplement whose pluginId did not re-register during the new load. If an activating reload disables, omits, or fails the plugin that previously owned a supplement, the old corpus entry is re-added anyway, so memory_search can still query a plugin that is no longer part of the active registry. Please tie restoration to plugins that are actually loaded and still meant to own the live supplement lifecycle, and cover the disable/failure reload case.
  - confidence: 0.9
- **[P2] Exercise the real service.start registration path:** `src/plugins/loader.test.ts:2531`
  - body: The new test seeds memory state by calling registerMemoryCorpusSupplement directly before reload, so it does not reproduce a plugin registering from service.start. In current main, plugin register(api) is guarded and closed before startPluginServices later invokes service.start, so a captured api.registerMemoryCorpusSupplement call from start currently no-ops. This test can pass while the reported plugin behavior remains broken; the fix needs either an explicit service-time registration path or a contract/doc change plus a test that uses registerService and service startup.
  - confidence: 0.87
