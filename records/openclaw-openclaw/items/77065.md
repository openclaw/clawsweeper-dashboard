## Summary

Keep this PR open: it still does not prove or correctly handle the advertised `service.start()` lifecycle. Current main closes the plugin registration API before services start, and the new restore loop can also resurrect stale corpus adapters for a still-loaded plugin that intentionally stopped registering them.

## What This Changes

The PR snapshots memory corpus supplements before activating plugin loads, restores prior supplements for still-loaded plugins that did not re-register one, adds loader regression tests, and updates the changelog.

## Best Possible Solution

Define the memory supplement lifecycle contract first: either keep corpus supplement registration synchronous in `register(api)` with diagnostics for captured service-time calls, or add a service-context registration seam whose state is cleaned up on service stop/reload.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the plugin registration API is closed before `startPluginServices`, while the PR test exercises `register(api)` rather than service startup. The stale-restore path is also source-reproducible for a still-loaded plugin that stops registering a supplement.

## Solution Assessment

No. Restoring old supplements by plugin id is broader than the lifecycle being fixed and can keep stale adapters alive; lifecycle-owned state or explicit register-only diagnostics would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid restoring supplements a plugin dropped:** `src/plugins/loader.ts:2401-2402`
  - body: The restore loop only checks that the plugin is still `loaded` and that it did not register a new corpus supplement. If the plugin remains enabled but its config or updated code intentionally stops registering a supplement, this re-adds the stale adapter from before `clearActivatedPluginRuntimeState()`, so the corpus stays active until the whole plugin is disabled or omitted. Tie restoration to an actually live service/registration owner instead of plugin id alone.
  - confidence: 0.87
- **[P2] Cover the actual service-start lifecycle:** `src/plugins/loader.test.ts:2528-2536`
  - body: This test uses a module-level flag inside `register(api)`, but the reported path calls from `service.start()`. Current main closes the guarded registration API when `register(api)` returns and later starts services with `OpenClawPluginServiceContext`, which has no `registerMemoryCorpusSupplement`, so the test can pass while service-start registration still no-ops.
  - confidence: 0.91
