## Summary

Keep this PR open, but the current patch is still not ready: it does not exercise the reported `service.start()` lifecycle and can still restore stale corpus supplements for service-owning plugins.

## What This Changes

The PR snapshots corpus supplements before activating plugin loads, restores prior supplements for service-owning plugins that did not re-register one, adds loader regression tests, and updates the changelog.

## Best Possible Solution

Define the memory supplement lifecycle explicitly: keep supplement registration synchronous in `register(api)` with diagnostics for post-register calls, or add a service-context registration seam whose state is cleaned up with service stop/reload.

## Reproduction Assessment

Yes, source-reproducible for the review blockers: the PR test does not call `service.start`, while current main closes the registration API before `startPluginServices` and the service context lacks `registerMemoryCorpusSupplement`. The stale-restore case is also source-reproducible for a service-owning plugin that drops a register-time supplement.

## Solution Assessment

No. Restoring previous supplements based on service presence is not the narrowest maintainable fix because it does not establish ownership or cleanup semantics for service-time registrations.

## Review Findings
