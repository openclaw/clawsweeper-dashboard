## Summary

Keep open. The follow-up addressed the earlier inactive-plugin resurrection concern, but the PR still does not prove or implement the advertised `service.start()` lifecycle behavior; the new tests exercise `register(api)`, which current main already re-runs after clearing runtime state.

## What This Changes

The PR snapshots memory corpus supplements before activating plugin loads, restores supplements for still-loaded plugins that did not re-register, adds loader tests, and updates the changelog.

## Best Possible Solution

Define the plugin service lifecycle contract, then either keep corpus supplements registered synchronously in `register(api)` with docs/diagnostics, or add a service-time registration seam whose state is tied to service start/stop/reload cleanup.

## Reproduction Assessment

Yes for the PR blocker: source inspection shows the public registration API is closed before `startPluginServices`, while the updated regression test uses `register(api)` and would pass on current main. There is not yet a high-confidence failing test in this PR for the reported `service.start()` path.

## Solution Assessment

No. The proposed restore loop is not the best solution until the SDK lifecycle is defined, because it restores old adapters outside service start/stop ownership and the tests no longer cover the advertised behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Tie restored supplements to service lifecycle:** `src/plugins/loader.ts:2396-2401`
  - body: This restores a previous supplement for any plugin whose new registry record is `loaded`, before gateway service startup has proved the owning service is still running and still wants that adapter. During a gateway reload the old service has already been stopped, and the new service may fail or intentionally stop registering the supplement, but the old adapter is still re-added. Please restore only for the lazy-load case where the owning service remains live, or move service-time supplements into start/stop-managed state.
  - confidence: 0.86
- **[P2] Exercise the service-start registration path:** `src/plugins/loader.test.ts:2526-2558`
  - body: The updated test registers the supplement directly from `register(api)`, which current main already preserves by re-running `register(api)` after `clearActivatedPluginRuntimeState()`. This can pass without the restore loop and does not prove that a supplement registered from `service.start()` survives a lazy reload; please build the regression around `registerService` plus service startup, or change the PR to a register-only contract.
  - confidence: 0.9
- **[P3] Correct the changelog PR reference:** `CHANGELOG.md:13`
  - body: The changelog entry ends with `(#77060)`, but this PR is `#77065`, so the release notes would send readers to the wrong pull request. Please update the reference before merge.
