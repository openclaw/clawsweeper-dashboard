## Summary

Current main and the latest release still expose one coarse Discord message-action gate, so the requested least-privilege capability remains missing and is not superseded. Keep this issue open for a maintainer product/security decision because the public config shape and backward-compatible inheritance semantics must be chosen before implementation.

## What This Changes

Add independent Discord send, read, edit, and delete action permissions while preserving the existing boolean message gate contract for current configurations.

## Best Possible Solution

Define one Discord-owned message permission type with explicit boolean-to-subaction normalization, preserve existing boolean configurations, apply the normalized decision consistently to account inheritance, discovery, and runtime execution, and document/test fresh and upgrade behavior.

## Reproduction Assessment

Yes, at source level with high confidence: current types, strict validation, account gates, discovery, and execution all lack independent message-operation permissions. A live Discord run is unnecessary to establish that the configuration cannot express the requested state.

## Solution Assessment

The least-privilege goal is appropriate, but the proposed nested object is not yet proven to be the best exact API. Maintainers should first settle normalization, root/account inheritance, omitted-key defaults, and whether send/upload share a permission before implementation.
