## Summary

Keep open. Current main still has one coarse Discord `actions.messages` boolean across config, discovery, docs, and runtime behavior; splitting it requires an explicit public config-contract decision before implementation.

## What This Changes

Request granular Discord message action gates so deployments can disable read, edit, delete, or send independently while preserving the existing boolean shorthand behavior.

## Best Possible Solution

Add a centralized Discord message-action gate resolver that preserves the existing boolean shorthand, then update types, strict schema, generated metadata, discovery, runtime checks, docs, changelog, and focused Discord tests together.

## Reproduction Assessment

Yes for the current behavior: source inspection shows `actions.messages` is boolean-only and the same key gates Discord message send/read/edit/delete runtime paths. A live Discord run is not required to establish that the requested config surface is missing.

## Solution Assessment

Unclear until maintainers choose the public config contract. A backward-compatible granular split is the narrow maintainable direction, but nesting, defaults, account inheritance, and `upload-file` semantics need owner approval.
