## Summary

Current main now solves the central least-privilege use case with one canonical global/per-agent message-action allowlist: operators can allow `send`, `read`, and `edit` while omitting `delete`, and the policy both hides disallowed actions from the tool schema and rejects them before Discord dispatch. This is preferable to adding a second Discord-only permission source; the implementation landed after v2026.6.5 and is currently unreleased.

## What This Changes

Requests granular Discord message action permissions so send, read, edit, and delete can be enabled independently.

## Best Possible Solution

Keep `tools.message.actions.allow` as the single global/per-agent least-privilege policy and add a Discord-oriented documentation example if discoverability needs improvement, rather than introduce overlapping channel-specific gates.

## Reproduction Assessment

Not applicable as a remaining issue: current main has a source-verifiable and tested path using `tools.message.actions.allow` to permit selected Discord actions and block the others.

## Solution Assessment

No—the proposed Discord-specific nested gate is no longer the best solution because the newer shared allowlist provides one enforced policy across discovery and runtime without creating a competing source of truth.
