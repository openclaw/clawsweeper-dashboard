## Summary

Keep open. Current main still exposes Discord `actions.messages` as one coarse boolean across config types, strict schema, generated metadata, discovery, docs, and runtime checks; the requested backward-compatible send/read/edit/delete split is not implemented.

## What This Changes

Request a backward-compatible split of `channels.discord.actions.messages` into separate send/read/edit/delete gates so destructive deletes can be disabled without blocking other Discord message actions.

## Best Possible Solution

Implement a backward-compatible Discord message gate contract, preferably `channels.discord.actions.messages: boolean | { send?: boolean; read?: boolean; edit?: boolean; delete?: boolean }`, with account overrides inheriting unset subkeys from parent/defaults. Wire that through config types, Zod validation, generated metadata, discovery, runtime checks, docs, and focused Discord tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with `actions.messages: false`, discovery still advertises `send`, but runtime `sendMessage`, `fetchMessage`, `readMessages`, `editMessage`, and `deleteMessage` all check the same `messages` gate; the schema also rejects the proposed object-shaped gate.

## Solution Assessment

Yes, the requested backward-compatible split is the narrow maintainable direction. A safer implementation should centralize sub-gate resolution so discovery, runtime, top-level config, and account config cannot drift.
