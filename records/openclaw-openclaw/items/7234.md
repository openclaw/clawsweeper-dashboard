## Summary

Current main still exposes one coarse Discord message-action gate, so the least-privilege request remains necessary; because this changes a security-sensitive public config contract and requires choosing the permanent compatibility shape, it needs maintainer product review rather than automatic implementation.

## What This Changes

Allow Discord send, read, edit, and delete message actions to be enabled independently while preserving the existing boolean configuration behavior.

## Best Possible Solution

Define one canonical Discord message-action permission model with explicit inheritance from the shipped boolean shorthand, apply it consistently to config schema/metadata, discovery, and runtime authorization, and add fresh-install plus upgrade-compatibility tests and docs.

## Reproduction Assessment

Yes for the missing capability: current-main types, schema, discovery, and runtime checks all expose only one messages boolean. A live Discord run is not required to establish that independent delete, edit, read, and send gates do not exist.

## Solution Assessment

Unclear. Granular gates are the right direction, but maintainers must choose the canonical public config shape, inheritance defaults, and whether send belongs to this tool-action permission family before implementation.
