## Summary

Keep open: current main still exposes Discord message permissions as one coarse `actions.messages` boolean, so the requested least-privilege send/read/edit/delete split remains missing and needs maintainer product/security review before implementation.

## What This Changes

This issue requests granular Discord message-action gates so deployments can disable destructive Discord edits/deletes while retaining sends, reads, or other message actions.

## Best Possible Solution

Define one canonical Discord message-action permission model with backward-compatible boolean shorthand, then align config types/schema/metadata, discovery, runtime authorization, docs, and focused compatibility tests.

## Reproduction Assessment

Yes. Current-main source shows the config type, Zod schema, discovery, docs, and runtime all still use one `messages` boolean, so a live Discord run is not needed to prove the granular gates are absent.

## Solution Assessment

Unclear. Granular action gates are the right direction, but maintainers still need to choose the public config shape, inheritance semantics, and whether send shares the same permission boundary as read/edit/delete.
