## Summary

Keep open. Current main and the latest shipped tag still model Discord `actions.messages` as one coarse boolean, while send/read/edit/delete runtime and discovery paths do not have channel/account-level per-action gates.

## What This Changes

The issue requests granular Discord message-action gates so send, read, edit, and delete can be enabled independently instead of sharing one `actions.messages` boolean.

## Best Possible Solution

Choose one backward-compatible least-privilege contract for Discord message actions, then align config types, strict schema, generated metadata, docs, discovery, runtime checks, and focused tests.

## Reproduction Assessment

Yes, by source inspection. Current main still accepts only `messages?: boolean`, rejects object-shaped Discord message gates, and routes Discord send/read/edit/delete through the same `messages` runtime gate.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested boolean-or-object shape is plausible, but separate top-level gates or documenting the existing message-tool allowlist may better fit current policy surfaces.
