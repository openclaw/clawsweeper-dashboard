## Summary

The issue remains necessary: current main and v2026.6.5 still expose one Discord `messages` action gate, and the same gate controls send, fetch/read, edit, and delete. This is a valid least-privilege feature, but the permanent public config shape and upgrade semantics require maintainer product judgment before implementation.

## What This Changes

Add granular Discord send, read, edit, and delete permission gates instead of one shared `actions.messages` boolean.

## Best Possible Solution

Choose one Discord-owned canonical message-permission model, preserve the shipped boolean shorthand through normalization or a deliberate doctor migration, and apply the normalized per-operation result consistently to schema, metadata, discovery, runtime, docs, and tests.

## Reproduction Assessment

Yes, at high confidence from current source. `channels.discord.actions.messages` is one boolean and the runtime uses that single key across send, fetch/read, edit, and delete, so delete cannot be disabled while the other operations remain enabled.

## Solution Assessment

The capability is the right direction, but the proposed nested object is not yet proven to be the best permanent API. Maintainers should first choose nested sub-gates versus separate canonical keys and define boolean shorthand, inheritance, defaults, and migration semantics.
