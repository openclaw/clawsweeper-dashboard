## Summary

Keep open. Current main still models Discord message actions as one coarse `actions.messages` boolean across config validation, discovery, docs, and runtime checks, so the requested least-privilege split is not implemented and still needs a public config/API decision.

## What This Changes

Add backward-compatible granular Discord message action gates so send, read, edit, and delete can be controlled independently.

## Best Possible Solution

Adopt one backward-compatible Discord action gate API for separate send/read/edit/delete policy, then update the config type, strict schema, generated metadata, docs, discovery, runtime checks, and focused Discord tests together.

## Reproduction Assessment

Yes, by source inspection. Current main only accepts `messages?: boolean`, rejects object-shaped gates in the strict schema, and routes send/read/edit/delete through the same `messages` runtime gate.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested boolean-or-object shape is reasonable, but separate top-level gates may be a cleaner fit for the existing boolean action-gate helper.
