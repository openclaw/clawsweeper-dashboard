## Summary

Current main still exposes only one Discord `actions.messages` boolean and applies it to send, fetch/read, edit, and delete operations, so the requested least-privilege controls remain missing. The request fits OpenClaw’s security and channel-improvement direction, but the permanent config shape and upgrade treatment need explicit maintainer product/security judgment before implementation.

## What This Changes

Add separate Discord permissions for sending, reading, editing, and deleting messages without forcing operators to enable or disable all message actions together.

## Best Possible Solution

Define one canonical Discord message-permission model with explicit defaults, migrate any superseded shipped shape through the Discord-owned doctor contract if necessary, and use the same resolver for schema metadata, discovery, and runtime enforcement with focused fresh-install and upgrade tests.

## Reproduction Assessment

Yes, at source level: current main has one boolean gate for Discord send, fetch/read, edit, and delete actions, so the missing selective-control capability is directly reproducible from the config and runtime paths. A live Discord run is unnecessary to establish that the configuration cannot represent the requested policy.

## Solution Assessment

Unclear as proposed: granular gates are the right product direction, but indefinitely accepting both boolean and object forms may violate the canonical-config policy. Maintainers should first choose the permanent schema, omitted-field defaults, and migration behavior, then implement one shared gate resolver across discovery and runtime.
