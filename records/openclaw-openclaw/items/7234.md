## Summary

Keep open. Current main still exposes Discord `actions.messages` as one coarse boolean and applies it across send, read, edit, and delete behavior, so the requested backward-compatible granular gates are not implemented.

## What This Changes

The issue requests splitting Discord `actions.messages` into backward-compatible per-operation gates for send, read, edit, and delete.

## Best Possible Solution

Add a centralized Discord message-action gate resolver that preserves the existing boolean shorthand while keeping types, strict schema, generated metadata, discovery, runtime checks, docs, and focused tests aligned.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the schema rejects object-shaped `actions.messages`, and the runtime checks the same `messages` gate for send, fetch/read, edit, and delete paths.

## Solution Assessment

Yes, the backward-compatible granular split is the narrow maintainable direction. The main open question is the exact public config shape, so maintainer agreement should come before implementation.
