## Summary

Keep open: current main and v2026.6.8 still model Discord message actions behind one `actions.messages` boolean, so the requested split remains a security-sensitive config/product decision rather than cleanup-close material.

## What This Changes

The issue requests granular Discord message action gates so operators can separately allow send, read, edit, and delete behavior.

## Best Possible Solution

Define one canonical Discord permission model that preserves or deliberately migrates the shipped boolean shorthand, then update config types, strict schema, generated metadata, discovery, runtime gates, docs, and focused tests together.

## Reproduction Assessment

Yes. Current main still has one Discord `messages` action gate in config, schema, discovery, runtime checks, docs, and tests; a live Discord run is not needed to prove independent send/read/edit/delete gates are absent.

## Solution Assessment

Unclear. Granular gates fit the least-privilege goal, but maintainers still need to choose the canonical public config shape, inheritance behavior, migration posture, and composition with existing message-tool policy.
