## Summary

Keep open. Current main and the latest release still model Discord `actions.messages` as one coarse boolean, discovery and runtime still share that gate across send/read/edit/delete, and the live issue search did not find a canonical replacement; the existing generic message-action allowlist is only a partial workaround, not a Discord account/channel gate.

## What This Changes

The issue requests a backward-compatible split of Discord `actions.messages` into separate send/read/edit/delete gates.

## Best Possible Solution

Choose a backward-compatible Discord least-privilege contract, then align config types, strict schema, generated metadata, discovery, runtime checks, docs, and tests while preserving existing boolean behavior and generic allowlist semantics.

## Reproduction Assessment

Yes, by source inspection. Current main still accepts only a boolean Discord `actions.messages` value and checks that same gate for send/read/edit/delete message actions.

## Solution Assessment

Unclear until maintainers choose the public contract. A boolean-or-object `actions.messages` shape is plausible, but it should coexist cleanly with `tools.message.actions.allow` and preserve the current boolean shorthand.
