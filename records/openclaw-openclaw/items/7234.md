## Summary

Keep open. Current main still exposes Discord `actions.messages` as one coarse boolean across the public config contract, discovery, runtime checks, docs, and focused tests; the request is source-reproducible but changes a security-sensitive permission/config model, so it needs maintainer API/product direction before implementation.

## What This Changes

Request splitting Discord `channels.discord.actions.messages` into granular send/read/edit/delete gates while preserving the existing boolean shorthand.

## Best Possible Solution

Define one backward-compatible Discord message action gate contract, then update the type, strict schema, generated metadata, docs, discovery, runtime checks, and focused tests together while preserving existing boolean behavior.

## Reproduction Assessment

Yes for source reproduction: current main still accepts only boolean `channels.discord.actions.messages`, and discovery/runtime use that one gate for message send/read/edit/delete behavior. A live Discord repro is not needed to prove the missing config contract.

## Solution Assessment

Unclear until maintainers choose the public config shape. A backward-compatible boolean-or-object gate is plausible, but separate top-level gates may fit the existing action schema better.
