## Summary

Keep open. Current main still exposes Discord `actions.messages` as a coarse boolean and uses that one gate for Discord send/read/edit/delete discovery and runtime behavior, so the requested granular permission model remains unimplemented and needs a maintainer config-contract decision.

## What This Changes

Request to split Discord `channels.discord.actions.messages` into granular send/read/edit/delete gates while preserving the legacy boolean shorthand.

## Best Possible Solution

Add a centralized Discord message-action gate resolver that preserves legacy boolean behavior, then update types, strict schema, generated metadata, discovery, runtime checks, docs, changelog, and focused Discord tests.

## Reproduction Assessment

Yes for the reported current behavior: source inspection on current main shows `actions.messages` is boolean-only and one `messages` gate controls Discord send/read/edit/delete runtime and discovery behavior. I did not need a live Discord run to establish the missing config surface.

## Solution Assessment

Unclear until maintainers choose the public config contract. A backward-compatible granular split is the narrow maintainable direction, but nesting, defaults, and upload-file semantics need owner approval.
