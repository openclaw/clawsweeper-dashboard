## Summary

Keep open. Current main and v2026.5.22 still model Discord message policy as one coarse `actions.messages` boolean, and the same gate still controls send/read/edit/delete runtime paths; the generic message-tool allowlist is useful but does not replace a Discord channel/account least-privilege contract.

## What This Changes

Request to split Discord `actions.messages` into per-message send/read/edit/delete gates while preserving the existing boolean shorthand.

## Best Possible Solution

Choose a backward-compatible Discord least-privilege config contract, then align config types, strict schema, generated metadata, discovery, runtime checks, docs, and focused tests while preserving boolean shorthand.

## Reproduction Assessment

Yes, by source inspection. Current main still accepts only a boolean Discord `actions.messages` value and checks that same gate for send/read/edit/delete message actions.

## Solution Assessment

Unclear until maintainers choose the public contract. The proposed boolean-or-object shape is plausible, but it should account for `tools.message.actions.allow` and preserve the current boolean shorthand.
