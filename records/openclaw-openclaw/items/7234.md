## Summary

Current main still cannot express independent Discord send, read, edit, and delete permissions, and no canonical replacement item owns the work. The feature remains useful and aligned, but its public config shape, inheritance, and backward-compatible defaults require maintainer product and security-boundary judgment before implementation.

## What This Changes

Add granular Discord message-action permissions so operators can independently allow sending, reading, editing, and deleting instead of using one shared `actions.messages` boolean.

## Best Possible Solution

Define one Discord-owned granular permission contract with explicit legacy-boolean normalization and account inheritance, then align exported types, strict schema, generated metadata, discovery, runtime enforcement, public docs, and focused fresh-config and upgrade tests.

## Reproduction Assessment

Yes, at source level. Current main’s exported type, strict schema, account gate, and discovery path expose only one boolean message-action setting, so the requested operation-specific configuration cannot be expressed.

## Solution Assessment

Unclear on the exact API shape. Granular permissions are the narrowest durable direction, but maintainers should first choose nested versus separate keys and define legacy boolean and account-inheritance semantics.
