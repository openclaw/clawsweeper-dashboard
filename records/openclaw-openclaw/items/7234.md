## Summary

Keep open. Current main still models Discord `actions.messages` as one coarse boolean and still routes Discord send/read/edit/delete through that gate; the generic message-tool allowlist is a possible workaround surface, but it does not implement the requested Discord channel config contract.

## What This Changes

Split Discord `actions.messages` into granular send/read/edit/delete gates while preserving the current boolean shorthand.

## Best Possible Solution

Choose a backward-compatible Discord least-privilege contract, then align config types, strict schema, generated metadata, discovery, runtime checks, docs, and focused tests while preserving existing boolean behavior.

## Reproduction Assessment

Yes, by source inspection. Current main still accepts only a boolean Discord `actions.messages` value and checks that same gate for send/read/edit/delete message actions.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested boolean-or-object shape is plausible, but the design should account for the existing `tools.message.actions.allow` surface and preserve the current boolean shorthand.
