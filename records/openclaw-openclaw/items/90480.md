Overall confidence: 0.86

Full review comments:

- **[P1] Keep the setup prompt from matching the location marker:** `extensions/qa-lab/src/live-transports/whatsapp/whatsapp-live.runtime.ts:1006-1008`
  - body: `whatsapp-inbound-structured-messages` seeds the future location rule in the initial user prompt, including the exact coordinates. The mock server checks the latest prompt for those coordinates and returns the WhatsApp location marker before exact-marker directives, so the first expected reply can be the later location marker instead of the initial token and the default mock-openai scenario can time out before it sends the location. Move the future rule out of the latest setup prompt or make the mock only use the location marker for the actual location body.
  - confidence: 0.89
- **[P1] Configure the blocked group as actually blocked:** `extensions/qa-lab/src/live-transports/whatsapp/whatsapp-live.runtime.ts:1252-1254`
  - body: This scenario sets `groupPolicy: "allowlist"` but leaves `groups` undefined while `allowFrom` contains the driver. Current WhatsApp policy falls `groupAllowFrom` back to `allowFrom`, and shared group policy allows allowlist mode with no explicit groups when a sender filter exists, so the target group is eligible instead of outside the allowlist. Add a nonmatching `groups` allowlist or otherwise make the driver sender unauthorized before making this the standard allowlist-block scenario.
  - confidence: 0.86
