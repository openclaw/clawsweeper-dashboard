## Label Justifications

- `P1`: The issue describes a user-visible macOS launchd Gateway restart and Control UI workflow that can become unreliable during live plugin/config maintenance.
- `impact:session-state`: The observed API symptom is `sessions.resolve` returning `No session found` while the runtime/session graph may be partially reloaded rather than truly missing.

## Summary

Keep open: current main has useful restart deferral, stale-process cleanup, startup-unavailable RPC gating, and /readyz pieces, but it still does not provide one shared partial-reload/degraded-state contract that `sessions.resolve` and Control UI can use during macOS launchd plugin/config maintenance.

## What This Changes

The issue requests a robust macOS launchd Gateway reload/restart degraded-state contract so Control UI and `sessions.resolve` do not surface generic connection or missing-session errors during live plugin/config maintenance.

## Best Possible Solution

Define one Gateway degraded/reload readiness contract for partial reload, restart-deferred, and incomplete plugin-set states, then wire it through `sessions.resolve`, `/readyz` or equivalent diagnostics, and Control UI messaging.

## Reproduction Assessment

No high-confidence live current-main macOS launchd reproduction was established. The saved incident logs map to current source paths, and source inspection still shows the missing shared degraded-state contract for readiness, `sessions.resolve`, and Control UI.

## Solution Assessment

No, current main is only a set of partial mitigations. The best fix is not a narrow `sessions.resolve` special case alone; maintainers need to choose one Gateway reload/readiness contract and then wire affected APIs and Control UI to it.
