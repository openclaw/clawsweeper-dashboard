## Label Justifications

- `P1`: A managed macOS Gateway reload can leave Control UI unusable or misleading until manual recovery, affecting a real user workflow.
- `impact:session-state`: The report centers on `sessions.resolve` returning `No session found` while the runtime/session graph may be only partially reloaded.

## Summary

Keep open: current main has useful restart deferral, stale-process cleanup, `/readyz`, and startup-unavailable handling, but it still lacks a shared partial-reload/degraded-state contract for `sessions.resolve` and Control UI; the open WebChat PR covers only a narrower restart-notice subset.

## What This Changes

The issue asks for macOS launchd Gateway restart/reload resilience so live plugin/config maintenance does not leave Control UI and session APIs in a misleading partially alive state.

## Best Possible Solution

Define one Gateway reload/readiness state that `sessions.resolve`, restart recovery, and Control UI consume consistently, then validate it with a live macOS launchd managed-Gateway replay; treat the open WebChat notice PR as a narrower subset only.

## Reproduction Assessment

No high-confidence live current-main reproduction yet. The saved macOS launchd logs and current source make the failure path plausible, especially because `sessions.resolve` is not startup-gated and still returns plain `No session found`.

## Solution Assessment

Unclear. The best fix appears to be a shared degraded-state contract across Gateway readiness, session RPCs, restart recovery, and Control UI rather than only a narrow WebChat notice fallback.
