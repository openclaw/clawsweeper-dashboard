## Label Justifications

- `P1`: The report describes a launchd Gateway/Control UI workflow becoming unusable during live maintenance and requiring manual recovery.
- `impact:crash-loop`: The issue centers on restart/recovery availability, stale gateway processes, and a gateway state that may require doctor plus restart to recover.
- `impact:session-state`: The visible failure includes sessions.resolve returning No session found while the reporter expected reload/readiness state handling rather than session repair.

## Summary

Keep open: current main has bounded restart deferral, stale process cleanup, and startup readiness pieces, but it still does not provide a shared partial-reload/degraded-state contract that would make Control UI and sessions.resolve distinguish reload readiness from a missing session.

## What This Changes

The issue requests a clearer and more reliable macOS launchd gateway reload/restart path when live plugin/config maintenance leaves Control UI reachable but not session-ready.

## Best Possible Solution

Define a gateway reload/readiness state that blocks or degrades session resolution while plugin/config maintenance is in flight, surface it in Control UI, and validate launchd restart recovery on macOS.

## Reproduction Assessment

No full current-main macOS launchd reproduction was run. Source inspection does show the reported misleading failure shape remains plausible because sessions.resolve still returns INVALID_REQUEST/No session found and readiness does not encode restart-deferred or plugin-reload state.

## Solution Assessment

Unclear as a narrow fix: a sessions.resolve special case alone would be too narrow. The maintainable solution is a shared reload/readiness contract consumed by Gateway APIs and Control UI.
