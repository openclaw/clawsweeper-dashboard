## Summary

Keep open: current main has useful restart deferral, stale-process cleanup, `/readyz`, and startup-unavailable pieces, but it still does not define one shared partial-reload/degraded-state contract for Control UI and `sessions.resolve`. The newer WebChat report and open PR cover a narrower restart-notice subset, not the whole macOS launchd partial-reload recovery problem.

## What This Changes

Request clearer and more deterministic macOS launchd Gateway recovery when live plugin/config maintenance leaves Control UI partially connected, session resolution misleading, or restart dependent on `doctor --fix`.

## Best Possible Solution

Define one Gateway reload/readiness state that `sessions.resolve` and Control UI can consume, keep restart recovery deterministic under launchd, and validate it with a live macOS managed-Gateway replay; treat https://github.com/openclaw/openclaw/pull/95154 as a narrower subset only.
