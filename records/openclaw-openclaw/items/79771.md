## Summary

Keep open: current main still imports `node-llama-cpp` for the existing local memory embedding provider but does not declare it as an installable root dependency, so the reported missing-dependency failure is source-reproducible; the exact packaging fix needs maintainer direction because recent dependency policy moved away from bundled runtime-deps staging.

## What This Changes

The issue asks OpenClaw to ship `node-llama-cpp` as a root optional dependency so `agents.defaults.memorySearch.provider: "local"` still works after updates.

## Best Possible Solution

Make `node-llama-cpp` reliably resolvable for the existing `local` memory embedding provider, using a maintainer-approved package ownership path and updated docs/error guidance.

## Reproduction Assessment

Yes for the dependency failure boundary: current main imports `node-llama-cpp` by bare specifier, the root package does not declare it as an installable dependency, and a direct import from the checkout fails with `ERR_MODULE_NOT_FOUND`. I did not run the full `openclaw memory status --deep` flow because this read-only checkout does not have `pnpm` available.

## Solution Assessment

Unclear. A root optional dependency matches the current bare import path, but maintainers should confirm that is the intended packaging policy versus moving or externalizing the local embedding runtime dependency.
