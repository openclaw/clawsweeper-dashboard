- **[P1] Preserve the Add-on principal binding:** `extensions/googlechat/src/auth.ts:293-297`
  - body: This removes `expectedAddOnPrincipal` from `verifyGoogleChatRequest`, but current main binds Add-on tokens to `target.account.config.appPrincipal` and rejects mismatched `sub` values. Without that binding, any matching Add-on issuer for the same audience can satisfy this shared app-url path.
  - confidence: 0.93
- **[P1] Keep the reply default upgrade-compatible:** `extensions/googlechat/src/channel.ts:226`
  - body: Changing the fallback from `off` to `all` changes existing Google Chat installs that did not set `replyToMode`. Current main and docs still use `off`, so this needs an explicit maintainer product decision, migration notes, and fresh-install plus upgrade proof before merge.
  - confidence: 0.9
- **[P1] Match mentions to this bot, not any bot:** `extensions/googlechat/src/monitor-access.ts:109-110`
  - body: Treating every mentioned `user.type === "BOT"` as an OpenClaw mention makes `requireMention` over-permissive in Spaces with multiple bots. A mention of another bot can now authorize OpenClaw processing and produce unsolicited replies.
  - confidence: 0.91
- **[P1] Classify DMs with current Google Chat fields:** `extensions/googlechat/src/monitor.ts:189-190`
  - body: This still derives group-vs-DM only from `space.type`; Google marks that field deprecated in favor of `spaceType`, with `singleUserBotDm` also present, so modern DIRECT_MESSAGE payloads without the legacy field can still be classified as groups and dropped by group policy. ([developers.google.com](https://developers.google.com/workspace/chat/api/reference/rest/v1/spaces?utm_source=openai))
  - confidence: 0.86
- **[P1] Namespace dedupe keys by account:** `extensions/googlechat/src/monitor.ts:175-179`
  - body: The dedupe map is module-global but keys only on `message.name`. In a multi-account deployment, one Google Chat account can process a message first and cause another configured account to skip its legitimate delivery for the full TTL.
  - confidence: 0.88
- **[P2] Mark dedupe only after confirmed delivery:** `extensions/googlechat/src/monitor.ts:379-381`
  - body: The marker is inserted after dispatch returns, but `deliverGoogleChatReply` catches send and upload failures and logs them instead of throwing. A failed send can therefore still mark the message processed and suppress retry webhook deliveries.
  - confidence: 0.84
- **[P2] Stop logging raw webhook payloads:** `extensions/googlechat/src/monitor-webhook.ts:91-102`
  - body: The new parse-error logging includes `JSON.stringify(raw)` snippets. Malformed Google Workspace Add-on requests can contain `authorizationEventObject.systemIdToken` and user message content, so these branches should log sanitized metadata only.
  - confidence: 0.89
- **[P2] Handle ADDED_TO_SPACE without a message:** `extensions/googlechat/src/monitor.ts:94-101`
  - body: The parser can now create `ADDED_TO_SPACE` events, but `processGoogleChatEvent` still returns when that event has no `message`. Common add-to-space callbacks omit a message, so the advertised lifecycle-event support remains effectively dropped.
  - confidence: 0.85
