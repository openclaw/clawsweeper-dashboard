## Summary

Keep open: the branch targets useful Google Chat fixes, but it is not mergeable as-is because source review confirms auth/access-control regressions, missing inspectable real behavior proof, and overlap with a narrower open classifier fix.

## What This Changes

The branch changes Google Chat auth, webhook parsing, inbound access policy, message processing/dedupe, outbound delivery/threading, and the old core dock reply-to default to improve Space messaging and duplicate webhook handling.

## Best Possible Solution

Split this into focused current-main fixes for Google Chat Space classification, thread routing, Add-on lifecycle parsing, and replay-safe dedupe while preserving appPrincipal binding, sender allowlists, bot-specific mention gating, sanitized logging, and explicit review for any reply-to default change.

## Reproduction Assessment

Yes for the PR review blockers: the PR head and current-main source show the auth binding removal, sender-policy downgrade, broad BOT mention match, raw logging, dedupe, and lifecycle defects. I did not run live Google Chat traffic in this read-only review.

## Solution Assessment

No. The broad branch is not the best fix path because it combines useful Google Chat work with security-sensitive regressions and a product-default change; focused current-main PRs are safer and easier to prove.

  - body: The PR removes the current expectedAddOnPrincipal binding, so an add-on issuer token for the audience is no longer checked against the configured appPrincipal sub claim before webhook acceptance.

  - confidence: 0.92
- **[P1] Match only the configured bot mention:** `extensions/googlechat/src/monitor-access.ts:108-110`
  - body: Treating any mentioned Google Chat user with type BOT as an OpenClaw mention lets requireMention pass when a message mentions a different bot in the same Space.
  - confidence: 0.9
- **[P1] Namespace dedupe keys by account:** `extensions/googlechat/src/monitor.ts:175-183`
  - body: The dedupe cache is module-global but keyed only by message.name, so multi-account deployments can drop one account's event after another account processes the same Google Chat message id.
  - confidence: 0.87
- **[P1] Refresh JWKS before rejecting unknown kids:** `extensions/googlechat/src/auth.ts:269-272`
  - body: manualVerifyGoogleJwt rejects a cached JWKS kid miss immediately; during normal Google key rotation, valid tokens can fail until the ten-minute cache expires.
  - confidence: 0.84
- **[P2] Stop logging raw webhook payloads:** `extensions/googlechat/src/monitor-webhook.ts:91-112`
  - body: The parse-error branches log JSON.stringify(raw), which can include add-on systemIdToken values and user message content in production logs.
  - confidence: 0.9
- **[P2] Commit dedupe only after confirmed delivery:** `extensions/googlechat/src/monitor.ts:379-381`
  - body: The PR marks a message processed after dispatch returns, but the delivery loop catches send and upload failures, so duplicate webhook retries can be skipped even when no reply was delivered.
  - confidence: 0.84
- **[P2] Handle added-to-space events without messages:** `extensions/googlechat/src/monitor.ts:100-101`
  - body: The parser can create ADDED_TO_SPACE events without a message, but processGoogleChatEvent returns before lifecycle handling can run, so the new event support is effectively dropped.
  - confidence: 0.83
