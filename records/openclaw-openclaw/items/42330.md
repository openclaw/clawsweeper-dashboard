Full review comments:

- **[P1] Keep Add-on tokens bound to appPrincipal:** `extensions/googlechat/src/auth.ts:293-318`
  - body: The PR drops the `expectedAddOnPrincipal` parameter and accepts Add-on issuer tokens after issuer/email checks only. Current main requires the configured `appPrincipal` to match the token principal, so this weakens the Google Chat Add-on auth boundary.
  - confidence: 0.94
- **[P1] Preserve deny-when-empty sender policy:** `extensions/googlechat/src/monitor-access.ts:240-245`
  - body: For a routed allowlist Space with no per-space users list, the PR computes `senderGroupPolicy` as `open`. That turns an explicitly configured route with an empty sender list into access for every sender in the Space.
  - confidence: 0.92
- **[P1] Match only the configured bot mention:** `extensions/googlechat/src/monitor-access.ts:108-110`
  - body: Treating any mentioned Google Chat user with `type === "BOT"` as an OpenClaw mention lets `requireMention` pass when a message mentions a different bot in the same Space, causing unsolicited group replies.
  - confidence: 0.9
- **[P1] Namespace dedupe keys by account:** `extensions/googlechat/src/monitor.ts:175-181`
  - body: The dedupe cache is module-global but keyed only by `message.name`, so multi-account deployments can drop one account's webhook event after another account processes the same Google Chat message id.
  - confidence: 0.87
- **[P1] Keep reply threading opt-in by default:** `extensions/googlechat/src/channel.ts:76-78`
  - body: Changing the fallback Google Chat `replyToMode` from `off` to `all` changes existing users' behavior on upgrade when they never opted into threaded replies. This needs explicit maintainer product approval and upgrade proof, not an unconditional default change inside a mixed bugfix PR.
  - confidence: 0.86
- **[P2] Stop logging raw webhook payloads:** `extensions/googlechat/src/monitor-webhook.ts:90-112`
  - body: The parse-error branches log raw webhook JSON fragments, which can include Add-on `systemIdToken` values and user message content. Log sanitized metadata instead of request body content.
  - confidence: 0.9
- **[P2] Handle added-to-space events without messages:** `extensions/googlechat/src/monitor.ts:100-101`
  - body: The parser can create `ADDED_TO_SPACE` events without a message, but processing returns immediately when that event has no message, so the new added-to-space path is still dropped.
  - confidence: 0.83
- **[P2] Commit dedupe only after confirmed delivery:** `extensions/googlechat/src/monitor.ts:379-381`
  - body: The PR marks a message processed after dispatch resolves, but delivery catches and logs send/upload failures internally, so duplicate webhook retries can be skipped even when no reply was delivered.
  - confidence: 0.84
- **[P2] Refresh JWKS on kid cache miss:** `extensions/googlechat/src/auth.ts:268-272`
  - body: The manual verifier serves cached JWKS for up to ten minutes and rejects immediately when the token kid is missing, so normal Google key rotation can reject valid app-url tokens until the cache expires.
  - confidence: 0.78
