## Summary

Keep this PR open for maintainer action: it targets a real Google Chat space/threading problem, but the branch is broad and introduces auth, access-control, media, logging, and dedupe regressions that should block merge.

## What This Changes

The PR changes Google Chat auth, webhook parsing, inbound access policy, message processing/deduplication, outbound media delivery, and the default reply-to mode for space messaging.

## Best Possible Solution

Replace or split the branch into focused current-main PRs that preserve auth, access, logging, and media contracts, handle space/lifecycle payload parsing narrowly, and decide the reply-to default separately.

## Reproduction Assessment

Yes for the review blockers: the supplied PR diff and current-main source show the auth binding removal, sender-policy downgrade, broad BOT mention match, local-media regression, raw logging, and dedupe/lifecycle defects. I did not run live Google Chat traffic in this read-only review.

## Solution Assessment

No, this PR is not the best way to solve the issue. The maintainable path is a set of narrow Google Chat parser, route, and dedupe changes that keep current-main auth, access, and media safeguards intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep add-on tokens bound to appPrincipal:** `extensions/googlechat/src/auth.ts:293-318`
  - body: Current main requires Google Workspace add-on tokens to match the configured appPrincipal through the token sub claim. This PR removes that binding from verifyGoogleChatRequest, so any verified add-on issuer for the audience can pass the webhook auth boundary.
  - confidence: 0.92
- **[P1] Preserve routed allowlist sender policy:** `extensions/googlechat/src/monitor-access.ts:240-245`
  - body: When a space is allowlisted but has no per-sender users list, current main keeps groupPolicy as allowlist. The PR derives senderGroupPolicy as open in that case, allowing any sender in an allowlisted space through the sender access check.
  - confidence: 0.93
- **[P1] Match only the configured bot mention:** `extensions/googlechat/src/monitor-access.ts:108-110`
  - body: Treating any mention whose user.type is BOT as an OpenClaw mention makes requireMention over-permissive in spaces with multiple bots. Mention gating should match users/app or the configured botUser, not every bot principal.
  - confidence: 0.89
- **[P1] Restore local media loading for sendMedia:** `extensions/googlechat/src/channel.ts:446-449`
  - body: The PR routes all mediaUrl values through fetchRemoteMedia and drops mediaLocalRoots, so local attachment paths that current main supports will fail instead of loading through the outbound media loader.
  - confidence: 0.88
- **[P1] Namespace dedupe keys by account:** `extensions/googlechat/src/monitor.ts:175-181`
  - body: The dedupe cache is module-global but keys only on message.name. Multi-account deployments can receive the same Google Chat message id through different bot accounts, causing one account to drop the other's event as a duplicate.
  - confidence: 0.87
- **[P2] Stop logging raw webhook payloads:** `extensions/googlechat/src/monitor-webhook.ts:91-112`
  - body: The new parse-error branches log JSON.stringify(raw), which can include add-on systemIdToken values and user message content. These paths should log sanitized metadata only.
  - confidence: 0.9
- **[P1] Refresh JWKS before rejecting unknown kids:** `extensions/googlechat/src/auth.ts:269-272`
  - body: manualVerifyGoogleJwt rejects a kid miss from the cached JWKS without forcing a refresh. During normal Google key rotation, valid webhook tokens can fail until the cache TTL expires.
  - confidence: 0.84
- **[P2] Commit dedupe only after confirmed delivery:** `extensions/googlechat/src/monitor.ts:379-381`
  - body: The PR records a message as processed after dispatch returns, but deliverGoogleChatReply catches send and upload failures and only logs them. A duplicate webhook after a recoverable delivery failure can then be skipped for the full TTL.
  - confidence: 0.84
- **[P2] Handle added-to-space events without messages:** `extensions/googlechat/src/monitor.ts:100-101`
  - body: The parser can map add-on payloads to ADDED_TO_SPACE, but processGoogleChatEvent immediately returns when that event has no message. Valid lifecycle callbacks without a message never reach any handling path.
  - confidence: 0.83
