Overall confidence: 0.93

Full review comments:

- **[P1] Keep Add-on tokens bound to appPrincipal:** `extensions/googlechat/src/auth.ts:293-318`
  - body: The app-url branch now accepts any verified Google Workspace Add-on issuer and no longer checks the configured appPrincipal. On shared webhook paths, a token for a different Chat app could verify and route into this account; current main passes expectedAddOnPrincipal and compares it to the token subject before accepting.
  - confidence: 0.96
- **[P1] Preserve empty sender allowlists as deny-by-default:** `extensions/googlechat/src/monitor-access.ts:240-245`
  - body: When a Space route is configured but its sender list is empty, this computes the sender policy as open, allowing any sender in that Space. Current main keeps this on the allowlist path with deny-when-empty, so the PR weakens the existing group access contract.
  - confidence: 0.94
- **[P1] Match only this bot's mention identity:** `extensions/googlechat/src/monitor-access.ts:108-110`
  - body: Treating any mentioned BOT user as OpenClaw being mentioned makes requireMention over-permissive in Spaces with multiple bots. A mention of another bot can satisfy the gate and trigger OpenClaw unexpectedly.
  - confidence: 0.93
- **[P1] Do not change reply threading defaults without migration:** `extensions/googlechat/src/channel.ts:77`
  - body: Changing the fallback from off to all affects every existing Google Chat account without an explicit replyToMode, so upgrades can start threading replies by default. This needs maintainer product approval plus upgrade coverage, or the new behavior should stay opt-in.
  - confidence: 0.9
- **[P2] Stop logging raw webhook payloads:** `extensions/googlechat/src/monitor-webhook.ts:91-112`
  - body: These parse-error logs serialize raw Google Chat request bodies. Add-on payloads can include authorizationEventObject.systemIdToken and message content, so malformed-request paths can leak bearer material or private chat data into logs.
  - confidence: 0.91
- **[P2] Namespace dedupe keys by account:** `extensions/googlechat/src/monitor.ts:175-187`
  - body: The dedupe cache is module-global but checks only message.name, so multi-account deployments that observe the same Space message can cause one account to drop the other account's event as a duplicate for the TTL. Include an account-unique discriminator before skipping.
  - confidence: 0.82
- **[P2] Mark dedupe only after confirmed outbound delivery:** `extensions/googlechat/src/monitor.ts:379-381`
  - body: The dedupe entry is written after dispatch returns, but the delivery loops catch and log send failures instead of propagating them. A recoverable webhook retry can be suppressed for the full TTL even though no reply was delivered.
  - confidence: 0.82
- **[P2] Handle add-to-space events without requiring a message:** `extensions/googlechat/src/monitor.ts:100-101`
  - body: The parser can create ADDED_TO_SPACE events without a message, but processGoogleChatEvent returns immediately for that exact shape. The claimed added-to-space support therefore drops common lifecycle callbacks before the pipeline can observe them.
  - confidence: 0.86
