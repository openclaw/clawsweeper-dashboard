## Summary

Keep this PR open but do not merge it as-is. The branch targets real Google Chat space/threading pain, but it is broad, stale against current main, and introduces concrete auth, access-control, media, logging, and dedupe regressions while overlapping narrower active Google Chat PRs.

## What This Changes

The PR refactors Google Chat auth/webhook/channel handling, adds add-on payload parsing and a 60-second in-memory message dedupe, changes the default Google Chat reply-to mode to `all`, and adjusts outbound media/threading behavior.

## Best Possible Solution

Split or replace the branch with focused current-main Google Chat PRs for spaceType/lifecycle parsing, replay-safe dedupe, and thread routing while preserving existing auth, access, media, and logging contracts; decide the reply-to default separately.

## Reproduction Assessment

Yes for the PR review blockers: the supplied PR diff and current-main source/tests show the add-on auth binding removal, sender-policy downgrade, broad BOT mention match, local-media regression, raw logging, and dedupe/lifecycle defects. I did not run live Google Chat traffic in this read-only review.

## Solution Assessment

No. The PR is not the best way to solve the issue because it combines stale structural changes with security-sensitive regressions; narrow current-main patches would be safer and easier to validate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep add-on tokens bound to appPrincipal:** `extensions/googlechat/src/auth.ts:293-318`
  - body: Current main requires Google Workspace add-on tokens to match the configured appPrincipal through the token `sub` claim. This PR removes that binding from webhook verification, so any verified add-on issuer for the audience can pass the auth boundary.
  - confidence: 0.92
- **[P1] Preserve routed allowlist sender policy:** `extensions/googlechat/src/monitor-access.ts:240-245`
  - body: When a space is allowlisted but has no per-sender users list, current main keeps the sender group policy as allowlist. The PR derives that case as open, allowing any sender in an allowlisted space through sender access checks.
  - confidence: 0.93
- **[P1] Match only the configured bot mention:** `extensions/googlechat/src/monitor-access.ts:108-110`
  - body: Treating any mention whose `user.type` is `BOT` as an OpenClaw mention makes `requireMention` over-permissive in spaces with multiple bots. Mention gating should match `users/app` or the configured bot user, not every bot principal.
  - confidence: 0.89
- **[P1] Restore local media loading for sendMedia:** `extensions/googlechat/src/channel.ts:446-449`
  - body: The PR routes all `mediaUrl` values through remote fetching and drops the local-media loader path, so local attachment paths that current main supports will fail instead of loading through `mediaLocalRoots`.
  - confidence: 0.88
- **[P1] Namespace dedupe keys by account:** `extensions/googlechat/src/monitor.ts:175-181`
  - body: The dedupe cache is module-global but keys only on `message.name`. Multi-account deployments can observe the same Google Chat message id through different bot accounts, causing one account to drop the other's event as a duplicate.
  - confidence: 0.87
- **[P2] Stop logging raw webhook payloads:** `extensions/googlechat/src/monitor-webhook.ts:91-112`
  - body: The new parse-error branches log `JSON.stringify(raw)`, which can include add-on systemIdToken values and user message content. These paths should log sanitized metadata only.
  - confidence: 0.9
- **[P1] Refresh JWKS before rejecting unknown kids:** `extensions/googlechat/src/auth.ts:269-272`
  - body: `manualVerifyGoogleJwt` rejects a cached JWKS kid miss without forcing a refresh. During normal Google key rotation, valid webhook tokens can fail until the cache TTL expires.
  - confidence: 0.84
- **[P2] Commit dedupe only after confirmed delivery:** `extensions/googlechat/src/monitor.ts:379-381`
  - body: The PR records a message as processed after dispatch returns, but reply delivery catches send/upload failures and only logs them. A duplicate webhook after a recoverable delivery failure can then be skipped for the full TTL.
  - confidence: 0.84
- **[P2] Handle added-to-space events without messages:** `extensions/googlechat/src/monitor.ts:100-101`
  - body: The parser can map add-on payloads to `ADDED_TO_SPACE`, but processing immediately returns when that event has no message, so valid lifecycle callbacks without a message never reach a handling path.
  - confidence: 0.83
