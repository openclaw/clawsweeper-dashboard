## Summary

Keep this PR open, but do not merge it as-is. Current main does not fully implement the requested Google Chat behavior, but the PR head is stale against the current plugin-owned adapter and introduces concrete auth, access-control, logging, lifecycle, and dedupe regressions that need maintainer/security review or a split replacement.

## What This Changes

The PR changes Google Chat auth, webhook parsing and routing, inbound access policy, message processing and reply delivery, local deduplication, outbound/config helpers, and the Google Chat reply-to default.

## Best Possible Solution

Split the useful behavior into narrow, current-main PRs: keep current auth/access/media protections, land or replace #65058 for add-on lifecycle payload parsing, land or replace #58993 for spaceType DM detection, decide the replyToMode default as a separate product change, and implement dedupe with account-scoped keys plus a real delivery-success signal.

## Reproduction Assessment

Yes for the code-level blockers: comparing PR head 7762373 to current main shows the missing appPrincipal binding, routed allowlist downgrade, raw webhook logging, unscoped dedupe, and ADDED_TO_SPACE early return. I did not run live Google Chat traffic because this review was read-only.

## Solution Assessment

No, this PR is not the best way to solve the issue. The safer path is a set of focused Google Chat adapter/parser changes that preserve current main's security and media contracts instead of merging a broad stale branch with auth and access-policy rewrites.

Full review comments:

- **[P1] Keep add-on tokens bound to appPrincipal:** `extensions/googlechat/src/auth.ts:293-318`
  - body: Current main requires Google Workspace add-on tokens to match the configured appPrincipal via the token sub claim. This PR removes that parameter from verifyGoogleChatRequest and accepts any verified add-on issuer for the audience, weakening the webhook auth boundary.
  - confidence: 0.92
- **[P1] Preserve routed allowlist sender policy:** `extensions/googlechat/src/monitor-access.ts:240-245`
  - body: When a Google Chat space is explicitly allowlisted but has no per-sender users list, current main keeps groupPolicy as allowlist. The PR derives senderGroupPolicy as open in that case, so any sender in an allowlisted space can pass the sender access check.
  - confidence: 0.93
- **[P1] Match only the configured bot mention:** `extensions/googlechat/src/monitor-access.ts:108-110`
  - body: Treating any mention whose user.type is BOT as an OpenClaw mention makes requireMention over-permissive in spaces with multiple bots. Mention gating should match users/app or the configured botUser, not every bot principal.
  - confidence: 0.89
- **[P1] Namespace dedupe keys by account:** `extensions/googlechat/src/monitor.ts:175-181`
  - body: The dedupe cache is module-global but keys only on message.name. Multi-account deployments can receive the same Google Chat message id through different bot accounts, causing one account to drop the other's event as a duplicate.
  - confidence: 0.87
- **[P2] Stop logging raw webhook payloads:** `extensions/googlechat/src/monitor-webhook.ts:91-112`
  - body: The new parse-error branches log JSON.stringify(raw), which can include add-on systemIdToken values and user message content. These paths should log sanitized metadata only.
  - confidence: 0.9
- **[P1] Refresh JWKS before rejecting unknown kids:** `extensions/googlechat/src/auth.ts:269-272`
  - body: manualVerifyGoogleJwt rejects a kid miss from the cached JWKS without forcing a refresh. During normal Google key rotation, valid webhook tokens can fail until the cache TTL expires.
  - confidence: 0.84
- **[P2] Commit dedupe only after confirmed delivery:** `extensions/googlechat/src/monitor.ts:379-381`
  - body: The PR records a message as processed after dispatch returns, but deliverGoogleChatReply catches send and upload failures and only logs them. A duplicate webhook after a recoverable delivery failure can then be skipped for the full TTL.
  - confidence: 0.84
- **[P2] Handle added-to-space events without messages:** `extensions/googlechat/src/monitor.ts:100-101`
  - body: The parser can map add-on payloads to ADDED_TO_SPACE, but processGoogleChatEvent immediately returns when that event has no message. Valid lifecycle callbacks without a message never reach any handling path.
  - confidence: 0.83
