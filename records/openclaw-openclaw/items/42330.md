## Summary

Keep this PR open for maintainer follow-up. Current main already solves the duplicate-webhook portion through shared inbound dedupe, but the requested Google Chat reply-to default is still not implemented, add-on lifecycle parsing is only partly covered by current main and narrower related work, and the PR diff introduces several blocking security and behavior regressions.

## What This Changes

The PR changes Google Chat auth, webhook parsing, inbound access handling, delivery/media handling, local dedupe behavior, and the Google Chat reply-to default.

## Best Possible Solution

Do not merge this branch as-is. Keep the shared inbound dedupe already on main, decide separately whether Google Chat should default to in-thread replies, route add-on lifecycle payload parsing through #65058 or an equivalent narrow PR, and rewrite or drop the PR's auth/logging/media/mention/local-dedupe changes unless they pass focused security and regression review.

## Reproduction Assessment

Yes for the code-level issues: current main shows the reply-to fallback remains "off", the parser only normalizes messagePayload add-on requests, and the PR review comments identify concrete diff paths for local-media, mention, auth, logging, and dedupe regressions. I did not run live Google Chat traffic because this review was read-only.

## Solution Assessment

No, this PR is not the best way to solve the issue. The maintainable path is to split the still-desired Google Chat behavior into narrow adapter/parser/docs/test changes and avoid replacing shared inbound dedupe or auth/media handling with broader local rewrites.

Full review comments:

- **[P1] Restore local media loading in sendMedia:** `extensions/googlechat/src/channel.ts:446-449`
  - body: The PR removes the local-path branch and always routes mediaUrl through fetchRemoteMedia, so workspace file outputs passed with mediaLocalRoots fail instead of uploading. Current main still has local-media coverage for this adapter, so the branch should be restored if sendMedia remains changed.
  - confidence: 0.91
- **[P1] Match only the configured bot mention:** `extensions/googlechat/src/monitor-access.ts:107-112`
  - body: Treating any mention whose user.type is BOT as an OpenClaw mention makes requireMention over-permissive in spaces with multiple bots. Mention gating should match users/app or the configured botUser, not any bot principal.
  - confidence: 0.89
- **[P1] Namespace local dedupe keys by account:** `extensions/googlechat/src/monitor.ts:171-181`
  - body: The PR's recentlyProcessedMessages map is module-global but keys only on message.name. Multi-account deployments can observe the same Google Chat message id through different bot accounts, causing one account to drop the other's event as a duplicate.
  - confidence: 0.86
- **[P2] Commit dedupe only after confirmed delivery:** `extensions/googlechat/src/monitor.ts:181-189`
  - body: The PR suppresses retries after dispatch completion even though Google Chat delivery can catch and log send/upload failures. A duplicate webhook after a recoverable delivery failure can be skipped for the TTL, dropping the user message.
  - confidence: 0.82
- **[P1] Refresh JWKS before rejecting unknown kids:** `extensions/googlechat/src/auth.ts:210-258`
  - body: The manual verifier rejects a token kid miss from cached JWKS without forcing a refresh. During normal Google key rotation, valid webhook tokens can fail until the cache TTL expires.
  - confidence: 0.84
- **[P2] Enforce the JWT algorithm allow-list:** `extensions/googlechat/src/auth.ts:254-258`
  - body: The manual JWT verifier does not explicitly require RS256 before signature verification, and the added ternary maps every algorithm to sha256. The verifier should reject unsupported alg values before using the key.
  - confidence: 0.86
- **[P2] Sanitize webhook parse-error logging:** `extensions/googlechat/src/monitor-webhook.ts:43-46`
  - body: The new parse-error path logs JSON.stringify(raw), which can include add-on systemIdToken values and message content. Log sanitized metadata only, or use an existing gated logger without dumping request bodies.
  - confidence: 0.88
- **[P2] Handle added-to-space events without messages:** `extensions/googlechat/src/monitor.ts:94-100`
  - body: The PR maps some add-on requests to ADDED_TO_SPACE, but processGoogleChatEvent still returns when event.message is absent. Valid add-to-space callbacks commonly have only space/user data, so the new lifecycle path is effectively dropped.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The PR introduces security-sensitive auth and logging changes that are not safe to merge without focused fixes.

Concerns:

- **[medium] Raw webhook bodies can leak tokens and content:** `extensions/googlechat/src/monitor-webhook.ts:43`
  - body: The diff logs raw invalid payloads, which can include authorizationEventObject.systemIdToken and user message content from Google Workspace Add-on requests.
  - confidence: 0.88
- **[medium] Manual JWT verifier lacks explicit alg enforcement:** `extensions/googlechat/src/auth.ts:254`
  - body: The added manual verifier does not reject unsupported JWT alg values before verification, weakening the webhook auth boundary and making the code's intended RS256-only contract unclear.
  - confidence: 0.86
- **[low] Production logs expose user and routing identifiers:** `extensions/googlechat/src/monitor-access.ts:96`
  - body: Review comments identify unconditional console logging in hot paths with user objects, email addresses, space ids, message ids, and token verification details; these should be removed or gated and sanitized.
  - confidence: 0.8
