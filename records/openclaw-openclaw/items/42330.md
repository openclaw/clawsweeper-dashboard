Full review comments:

- **[P1] Keep Add-on tokens bound to appPrincipal:** `extensions/googlechat/src/auth.ts:307-320`
  - body: The app-url branch now accepts verified Google Workspace Add-on issuers without checking that the token subject matches the configured appPrincipal. On shared webhook paths, a token for a different Chat app could verify and route into this account; current main passes expectedAddOnPrincipal and compares it before accepting.
  - confidence: 0.96
- **[P1] Preserve empty Space sender allowlists as deny-by-default:** `extensions/googlechat/src/monitor-access.ts:237-245`
  - body: When a Space route is configured but its sender list is empty, this derives the sender policy as open, allowing any sender in that Space. Current main keeps configured empty sender allowlists on a deny-when-empty path, so the PR weakens the existing group access contract.
  - confidence: 0.94
- **[P1] Match only this bot's mention identity:** `extensions/googlechat/src/monitor-access.ts:107-110`
  - body: Treating any mentioned BOT user as OpenClaw being mentioned makes requireMention over-permissive in Spaces with multiple bots. A mention of another bot can satisfy the gate and trigger OpenClaw unexpectedly.
  - confidence: 0.93
- **[P1] Do not change reply threading defaults without migration:** `src/channels/dock.ts:454`
  - body: Changing the fallback from off to all affects every existing Google Chat account without an explicit replyToMode, so upgrades can start threading replies by default. This needs maintainer product approval plus upgrade coverage, or the new behavior should stay opt-in.
  - confidence: 0.9
- **[P1] Namespace dedupe keys by account:** `extensions/googlechat/src/monitor.ts:177-179`
  - body: The dedupe cache is module-global but checks only message.name, so multi-account deployments that observe the same Space message can cause one account to drop another account's event as a duplicate for the TTL. Include an account-unique discriminator before skipping.
  - confidence: 0.85
- **[P2] Stop logging raw Google Chat webhook payloads:** `extensions/googlechat/src/monitor-webhook.ts:88-111`
  - body: These parse-error logs serialize raw Google Chat request bodies. Add-on payloads can include authorizationEventObject.systemIdToken and message content, so malformed-request paths can leak bearer material or private chat data into logs.
  - confidence: 0.91
- **[P2] Only mark dedupe after confirmed outbound delivery:** `extensions/googlechat/src/monitor.ts:375-381`
  - body: The dedupe entry is written after dispatch returns, but Google Chat delivery catches and logs send/upload failures instead of propagating them. A recoverable webhook retry can be suppressed for the full TTL even though no reply was delivered.
  - confidence: 0.82
- **[P2] Handle add-to-space events without requiring a message:** `extensions/googlechat/src/monitor.ts:98-101`
  - body: The parser can create ADDED_TO_SPACE events without a message, but processGoogleChatEvent returns immediately for that exact shape. The claimed added-to-space support therefore drops common lifecycle callbacks before the pipeline can observe them.
  - confidence: 0.86
