## Summary

Keep open: this is a useful Layer 1 security fix, but the proposed header sanitizer is narrower than OpenClaw's current secrets-audit contract, so some prompt-visible provider header secrets would still be persisted after merge.

## What This Changes

The PR strips newly generated plaintext provider apiKey values and selected sensitive provider headers from generated models.json, avoids copying plaintext auth-profile keys into provider config, and adds focused regression tests.

## Best Possible Solution

Land one canonical Layer 1 fix that uses the same sensitive-header predicate as secrets audit, keeps runtime auth sourced from config/auth profiles, and leaves the broader security roadmap open for later layers.

No: the direction is right, but the sanitizer should share or match the current secrets-audit sensitive-header predicate before this can be the canonical Layer 1 fix.
