- `P1`: The PR touches provider credential persistence into prompt-visible agent model catalog data and can currently break auth/model loading for existing config-only providers.
- `merge-risk: 🚨 compatibility`: Existing generated models.json/custom-provider setups may stop loading when plaintext apiKey/header fields are omitted without a replacement runtime auth path.

- `merge-risk: 🚨 security-boundary`: The PR changes the boundary between runtime authentication secrets and prompt-visible agent catalog data.

## Summary

Keep open: this is useful Layer 1 security hardening, but the branch is not merge-ready because it can write a generated models.json provider with model rows after removing its only apiKey, which current ModelRegistry rejects, and the PR is currently conflict-dirty against main.

## What This Changes

The PR strips new plaintext provider apiKey values and sensitive provider headers from generated models.json output, avoids copying plaintext auth-profile keys into generated provider config, and adds focused regression/test mock updates.

## Best Possible Solution

Land a rebased Layer 1 fix that strips prompt-visible provider secrets only when generated models.json remains loadable and runtime auth still resolves from a non-prompt-visible source, with registry/load coverage proving that path.

## Reproduction Assessment

Yes, by source inspection: the PR test creates a custom provider with model rows and a plaintext apiKey, the PR strips that apiKey before writing, and current ModelRegistry rejects model providers without apiKey/auth. I did not run tests because this was a read-only review.

## Solution Assessment

No: stripping at the planner boundary is the right direction, but this patch must also preserve a loadable runtime auth path or omit invalid providers. The current branch proves the field is removed, not that generated models.json can still be loaded and used.
