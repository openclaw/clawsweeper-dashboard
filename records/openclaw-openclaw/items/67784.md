## Summary

Keep open. Current main still leaves the Mattermost message-tool handleAction send path incomplete: explicit threadId is not used as a reply root, mediaLocalRoots/mediaReadFile are not forwarded, and filePath/path aliases are not consumed directly by the Mattermost handler; the related implementation PRs remain open partial fixes rather than merged resolution.

## What This Changes

The issue reports that Mattermost message-tool sends do not honor threadId, filePath/path media aliases, or agent-scoped mediaLocalRoots when uploading files to a thread.

## Best Possible Solution

Keep this issue open until a coordinated Mattermost-owned fix lands that preserves reply precedence, accepts explicit threadId, resolves media aliases consistently, forwards mediaLocalRoots/mediaReadFile, and adds focused handleAction regression coverage.

## Reproduction Assessment

Yes. At source level, a Mattermost message-tool send with threadId, filePath/path, and scoped media roots reaches handleAction, but current main does not forward the complete set to sendMessageMattermost.

## Solution Assessment

Yes, if consolidated. The narrow maintainable fix is in extensions/mattermost/src/channel.ts plus focused channel.test.ts coverage because core dispatch and the Mattermost sender already expose the needed contract.
