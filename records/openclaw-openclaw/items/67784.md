## Summary

Keep open. Current main still leaves the Mattermost message-tool `send` handler incomplete for explicit `threadId` and scoped local media forwarding; `filePath`/`path` are partly normalized before dispatch, but the Mattermost-owned action adapter still does not consume all requested inputs. The open related PRs are partial implementation candidates, not a reason to close this issue before a coordinated fix lands.

## What This Changes

The issue asks Mattermost message-tool sends to honor `threadId`, `filePath`/`path` media aliases, and `mediaLocalRoots` so local file uploads land in the requested Mattermost thread.

## Best Possible Solution

Land one coordinated Mattermost-owned fix in `extensions/mattermost/src/channel.ts` that preserves `replyToId` precedence, treats explicit `threadId` as the Mattermost root-post fallback, consistently resolves media aliases, forwards `mediaLocalRoots` and `mediaReadFile`, and adds a focused regression test for local file upload into a thread.

## Reproduction Assessment

Yes. A source-level reproduction is clear: invoke the Mattermost `handleAction` send path with `threadId`, local media, and scoped media access in the action context; current main omits the thread fallback and media-root fields when calling `sendMessageMattermost`.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Mattermost action adapter because core already supplies scoped media access and the Mattermost sender already accepts the downstream options; no new core API is needed.
