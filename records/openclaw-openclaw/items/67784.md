## Summary

Keep open. Current main still routes Mattermost message-tool sends through the plugin action handler, and that handler still does not consume explicit `threadId` or forward `mediaLocalRoots`/`mediaReadFile`; `filePath`/`path` are partly normalized by the shared runner, but local file upload into a Mattermost thread remains incomplete. The related open PRs #67796 and #73060 are partial implementation candidates, not grounds to close this issue.

## What This Changes

Mattermost message-tool send actions should honor `threadId`, local file aliases, and agent-scoped media-root access so local uploads land in the requested Mattermost thread.

## Best Possible Solution

Land one coordinated Mattermost-owned fix that preserves existing `replyToId` precedence, treats explicit `threadId` as the Mattermost root-post fallback, consistently resolves media aliases, forwards `mediaLocalRoots` and `mediaReadFile` to `sendMessageMattermost`, and adds regression coverage for local file upload into a thread.

## Reproduction Assessment

Yes. A source-level repro is clear: invoke the Mattermost `handleAction` send path with `threadId`, local media, and scoped media access in the action context, and the current call to `sendMessageMattermost` omits the thread and media-root fields.

## Solution Assessment

Yes. The narrowest maintainable fix is in `extensions/mattermost/src/channel.ts`; core already passes media access to plugin actions and the Mattermost sender already accepts the downstream fields, so no new core seam is needed.
