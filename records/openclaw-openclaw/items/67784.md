## Summary

Keep open. Current main still drops explicit Mattermost thread/media-root fields in direct message-tool sends, while the related open pull requests at https://github.com/openclaw/openclaw/pull/58439, https://github.com/openclaw/openclaw/pull/67796, and https://github.com/openclaw/openclaw/pull/73060 are partial fix candidates rather than merged implementation.

## What This Changes

The issue asks Mattermost message-tool sends to honor threadId, filePath/path, and scoped mediaLocalRoots for local file uploads into threads.

## Best Possible Solution

Implement a single Mattermost-owned handler update with regression coverage that consumes the existing action context, preserves replyToId precedence, falls back to threadId, resolves media aliases consistently, and forwards scoped media access to the sender.

## Reproduction Assessment

Yes, at source level: call Mattermost handleAction for action send with threadId, filePath/path, and scoped media roots. Current origin/main drops threadId and scoped media fields before sendMessageMattermost, and only direct params.media is considered in the handler.

## Solution Assessment

Yes. The narrowest maintainable fix is in extensions/mattermost/src/channel.ts because core already supplies mediaLocalRoots/mediaReadFile and sendMessageMattermost already consumes them.
