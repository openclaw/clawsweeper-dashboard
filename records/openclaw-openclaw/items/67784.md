## Summary

Keep open. Current main still leaves the Mattermost direct `handleAction` send path incomplete for explicit thread replies and scoped local-media upload; the related open PRs are partial candidates, not a merged fix.

## What This Changes

The issue asks Mattermost message-tool send actions to honor `threadId`, `filePath`/`path` media aliases, and scoped `mediaLocalRoots` so attachments upload into the intended thread.

## Best Possible Solution

Land or replace the partial Mattermost PRs with one coordinated handler fix that preserves `replyToId` precedence, falls back to explicit `threadId`, resolves media aliases consistently, forwards media roots/readers, and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level: call Mattermost `handleAction` with `action="send"`, `threadId`, `filePath` or `path`, and scoped media fields, and current main will not forward the full set to `sendMessageMattermost`. I did not run tests because this review had to keep the checkout read-only.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Mattermost channel handler and its focused tests because core dispatch and the Mattermost sender already expose the needed contract.
