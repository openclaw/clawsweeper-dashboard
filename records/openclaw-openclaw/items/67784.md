## Summary

Keep open. Current main still leaves the Mattermost message-tool send handler incomplete for this reported path, and the related Mattermost PRs are open partial fixes rather than merged implementation.

## What This Changes

The issue requests that Mattermost message-tool sends honor `threadId`, `filePath`/`path`, and scoped `mediaLocalRoots` for local file delivery into Mattermost threads.

## Best Possible Solution

Land or replace the open partial PRs with one coordinated Mattermost-owned fix that preserves `replyToId` precedence, falls back to explicit `threadId`, resolves local media aliases consistently, forwards scoped media access, and adds focused handler regression tests.

## Reproduction Assessment

Yes, at source level: call the Mattermost `handleAction` send path on current main with `threadId`, `filePath` or `path`, and scoped media fields. The handler does not forward the complete set to `sendMessageMattermost`, while shared core already supplies those fields.

## Solution Assessment

Yes. The narrowest maintainable solution is to fix `extensions/mattermost/src/channel.ts` and its tests, because the shared action contract and Mattermost sender already have the required media-root and reply plumbing.
