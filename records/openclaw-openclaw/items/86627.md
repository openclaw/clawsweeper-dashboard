## Summary

Keep open: the contribution-owned doctor ordering looks useful and the PR body has real command-path proof for the happy path, but a blocking registry-boundary bug remains. The new helper filters out `kind: "core"` registrations before validating reserved core doctor IDs, so a bundled/plugin check can claim a core doctor ID and be silently ignored instead of rejected.

## What This Changes

This PR moves core doctor lint checks into ordered doctor health contributions, appends bundled/plugin registry checks afterward, updates doctor docs/tests, and removes the temporary conversion-plan ledger.

## Best Possible Solution

Keep the contribution-owned core ordering model, but validate every registered health check against reserved core doctor IDs before filtering extension checks and add regressions for both plugin-kind and core-kind reserved claims.

## Solution Assessment

No. The contribution-order direction is plausible, but the safe implementation must validate reserved core IDs across all registered checks before filtering to extension checks.
