## Summary

Keep this PR open: it is a new Codex plugin feature with a protected `maintainer` label, and source review found one remaining cache-refresh correctness bug that should block merge even though CI and real behavior proof are present.

## What This Changes

The PR adds explicit `codexPlugins` config, native Codex app inventory/activation/thread wiring, plugin elicitation policy, Codex migration `--plugin` support, docs, tests, and changelog coverage.

## Best Possible Solution

Fix the forced post-activation app inventory refresh path, keep the explicit fail-closed plugin policy, then have maintainers decide whether to accept the native Codex plugin surface before merge.

## Reproduction Assessment

Yes for the PR-introduced bug by source path: start with stale app inventory so a non-forced `app/list` is in flight, activate or re-enable a migrated plugin, then observe the forced post-activation refresh can coalesce with that older request. I did not execute the test locally because this was a read-only review.

## Solution Assessment

No, not yet: the overall direction is maintainable, but the current patch can build a plugin-enabled thread from stale app readiness after activation. The safer fix is to force and await a post-install `app/list` refresh that bypasses older in-flight reads, with a focused regression test.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Force post-activation app refresh past in-flight reads:** `extensions/codex/src/app-server/app-inventory-cache.ts:125-128`
  - body: `refreshCodexPluginRuntimeState` calls `read(..., forceRefetch: true)` after plugin activation, but this branch returns as soon as any older `app/list` is in flight. That lets the forced refresh coalesce with a pre-activation stale read, so the next thread can still be built without the newly installed or re-enabled plugin app until the cache expires or the thread is reset. Let forced reads start a new refresh, or await `refreshNow` with `forceRefetch: true`, before building the plugin thread config.
  - confidence: 0.87
