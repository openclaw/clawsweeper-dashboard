## Summary

Keep open: this PR is a protected, feature-sized Codex native-plugin change with sufficient live behavior proof, but the latest head still has a source-reproducible token auth-profile cache-key bug that can reuse stale plugin app inventory.

## What This Changes

This PR adds explicit Codex `codexPlugins` config, native plugin inventory/activation/thread config, plugin elicitation policy, Codex plugin migration CLI/docs/tests, and changelog entries.

## Best Possible Solution

Fingerprint resolved token auth profiles in the app-inventory cache key, keep the explicit fail-closed `codexPlugins` design, then send the exact head through Codex/gateway owner review and validation.

## Reproduction Assessment

Not applicable as a current-main bug because current main intentionally lacks `codexPlugins` and still documents Codex native plugins as manual-review state. Yes for the blocking review finding: source inspection of the latest PR head shows token profiles resolve a real secret for login but are not fingerprinted in the app inventory cache key.

## Solution Assessment

No, not merge-ready yet: the feature direction is explicit and fail-closed, but token auth profiles need the same secret-fingerprint treatment as API-key profiles. The narrow repair is to hash the resolved token or equivalent credential version into `resolveCodexAppServerAuthAccountCacheKey` with regression coverage.
