## Label Justifications

- `P2`: This is a bounded gateway/channel lifecycle bug with noisy Telegram restart-loop behavior and a restart workaround, but it does not make the whole core runtime unusable.
- `impact:crash-loop`: The reported failure mode is a losing Telegram channel repeatedly exiting with EADDRINUSE and auto-restarting until the retry cap is hit.

## Summary

Keep open: current main and v2026.6.8 still allow an active deferred channel hot reload to continue after restart shutdown begins; same-manager duplicate-start serialization is present, but it does not cancel stale reload work captured by the old lifecycle.

## What This Changes

Prevent deferred gateway channel hot reload work from spawning the Telegram channel a second time when a gateway-tool restart or SIGUSR1 restart starts during the deferral window.

## Best Possible Solution

Thread shutdown/restart cancellation through config hot reloads, stop and drain the config reloader before channel teardown, and make deferred channel stop/start exit once cancellation is observed.

## Reproduction Assessment

Yes, at source level: deferred channel reload waits and then calls channel stop/start without a cancellation context, while shutdown stops channels before stopping the reloader and `stop()` does not drain active reload work. I did not run a live Telegram webhook reproduction.

## Solution Assessment

Yes. Gateway-owned cancellation and drain for active config reload work is the narrow maintainable fix; per-channel single-flight start helps one symptom but is too narrow for stale reload work across restart lifecycles.
