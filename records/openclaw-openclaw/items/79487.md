## Summary

Current main still appears vulnerable, and an open linked pull request already carries the focused fix path. This issue should stay open until that pull request lands or maintainers choose a different gateway lifecycle fix.

## What This Changes

The issue reports a gateway lifecycle race where a deferred Telegram channel hot reload and an in-process restart can start the same webhook listener twice.

## Best Possible Solution

Land or replace the focused cancellation-aware config-reloader fix in the linked pull request, then close this issue with the merge commit and release provenance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: defer a channel hot reload behind active work, trigger SIGUSR1 or gateway-tool restart before the deferral drains, and the old reloader can resume and call startChannel while the restarted gateway starts channels.

## Solution Assessment

Yes. A gateway close/restart cancellation and drain fence for pending config hot reload work is the narrow maintainable fix; a process-wide channel single-flight would be broader and would mask stale lifecycle work instead of stopping it.
