## Summary

Current main still appears vulnerable: the config reloader can keep a deferred hot reload running after shutdown begins, and the existing channel startup single-flight only protects one ChannelManager instance, not the old-manager/new-manager overlap created by SIGUSR1 in-process restart.

## What This Changes

Prevent a deferred channel hot reload and a gateway-tool restart from double-starting the same Telegram webhook channel and causing an EADDRINUSE auto-restart loop.

## Best Possible Solution

Make deferred channel reloads cancellation-aware at the gateway close/restart boundary, preferably by stopping or fencing the config reloader before channel lifecycle teardown and skipping pending channel starts once shutdown begins.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: defer a channel hot reload behind active work, trigger SIGUSR1/gateway-tool restart before the deferral drains, and the old reloader can still call `startChannel` while the new gateway iteration starts channels. I did not live-bind a Telegram webhook in this read-only review.

## Solution Assessment

Yes, a lifecycle fence or cancellation path for pending deferred reloads is the narrow maintainable fix; a process-wide channel single-flight would be broader and would mask old lifecycle work instead of stopping it.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79487. Current main lets `startGatewayConfigReloader().stop()` stop future watcher work without cancelling or awaiting an already-running `onHotReload`; `createGatewayCloseHandler` stops channels before stopping the reloader, so a deferred channel reload can resume and call `startChannel` while SIGUSR1 restart creates a new gateway/ChannelManager that also starts channels. Add a generic lifecycle fence/cancellation path so pending deferred channel reloads are skipped or cancelled once gateway close/restart begins. Prefer fixing the gateway reload/reloader lifecycle over Telegram-specific logic; do not add a new config option. Add regression coverage that starts a deferred channel reload with active embedded work, begins gateway close/restart before the active work drains, drains the work, and asserts the old reload path does not start the channel after shutdown while normal non-closing deferred reloads still work. Include a user-facing `CHANGELOG.md` fix entry.
