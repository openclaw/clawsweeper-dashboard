## Summary

Keep open: current main still has the unfenced deferred channel-reload path, and https://github.com/openclaw/openclaw/pull/79510 is an open, unmerged closing PR for the same race. This issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports a gateway lifecycle race where a deferred Telegram channel hot reload can overlap an in-process restart and start the same webhook listener twice.

## Best Possible Solution

Add a lifecycle fence that cancels or drains pending config hot reloads during gateway close/restart, with regression coverage for deferred channel reload plus SIGUSR1 restart; then close this issue after the linked fix or a replacement merges.

## Reproduction Assessment

Yes. Source inspection on current main shows the deferred channel reload can resume into `startChannel` without observing shutdown while SIGUSR1 restart creates a new gateway lifecycle; the reporter also provided a concrete Telegram EADDRINUSE log timeline.

## Solution Assessment

Yes. The narrow maintainable fix is to make deferred reload work cancellation-aware at the gateway close/restart boundary; a process-wide channel single-flight would be broader and would mask stale lifecycle work instead of stopping it.
