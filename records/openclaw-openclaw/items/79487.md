## Summary

Current main still appears affected by the reported gateway lifecycle race, and the open linked pull request at https://github.com/openclaw/openclaw/pull/79510 already carries the focused fix path with closing syntax. This issue should stay open until that PR lands or maintainers choose a replacement gateway reload/restart fix.

## What This Changes

The issue reports that a deferred Telegram channel hot reload can overlap with a gateway-tool in-process restart and double-start the same webhook listener, causing an EADDRINUSE auto-restart loop.

## Best Possible Solution

Review and land, or intentionally replace, the focused cancellation/drain fix in https://github.com/openclaw/openclaw/pull/79510, then close this issue with the merge commit and release provenance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: defer a channel hot reload behind active work, trigger gateway-tool or SIGUSR1 restart before the deferral drains, and the old reload path can resume and call startChannel while the restarted gateway is also starting channels.

## Solution Assessment

Yes. Cancelling and draining config hot reload work during shutdown/restart is the narrow maintainable fix; process-wide channel-start single-flight would be broader and would mask stale lifecycle work instead of stopping it.
