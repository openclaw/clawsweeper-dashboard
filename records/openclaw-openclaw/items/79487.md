## Summary

Current main still appears source-reproducible for the deferred reload versus in-process restart race, and the linked fixing PR is open and unmerged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Prevent deferred Telegram channel hot reloads from racing with gateway-tool/SIGUSR1 restart and starting duplicate listeners that produce EADDRINUSE auto-restart noise.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79510 or an equivalent cancellation-aware gateway reload fix, then close this issue after the fix is merged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: defer a channel reload behind active work, trigger gateway-tool/SIGUSR1 restart before the deferral drains, and current main can let stale reload work call `startChannel` while the restarted gateway also starts channels.

## Solution Assessment

Yes. Fencing or cancelling deferred reload work at the gateway close/restart boundary is the narrow maintainable fix; a process-wide channel single-flight would be broader and would mask stale lifecycle work instead of stopping it.
