## Summary

Keep open: current main and the latest release still let a deferred channel reload resume into channel stop/start without shutdown or SIGUSR1 cancellation, and this issue now has multiple open issue-closing PRs that should be reviewed rather than spawning a new fix lane.

## What This Changes

The issue reports a gateway race where a deferred channel hot reload and in-process restart can start the same Telegram channel twice, producing an EADDRINUSE auto-restart loop.

## Best Possible Solution

Land one gateway-owned cancellation/drain fix for the deferred reload lifecycle, then let the merged PR close this issue and close or supersede the competing candidate PRs.

## Reproduction Assessment

Yes, at source level: the issue supplies a concrete log timeline, and current main shows deferred channel reload can resume into stopChannel/startChannel without restart cancellation. I did not run a live Telegram webhook reproduction.

## Solution Assessment

Yes for the direction: gateway-owned cancellation and draining of stale config reload work is the narrow maintainable fix. Per-channel single-flight start is only a partial symptom guard and should not be the canonical solution by itself.
