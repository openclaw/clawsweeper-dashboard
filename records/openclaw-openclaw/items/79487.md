## Summary

Keep open: current main and the latest release still leave deferred channel reload work able to continue after restart shutdown begins, even though same-manager duplicate channel starts are now serialized. The linked fix PR had the right cancellation/drain direction but closed unmerged, so it is evidence rather than a completed fix.

## What This Changes

The issue reports a gateway lifecycle race where a deferred channel hot reload and SIGUSR1 in-process restart can both start the Telegram channel, producing EADDRINUSE auto-restart noise.

## Best Possible Solution

Thread shutdown/restart cancellation through config hot reloads, drain the active reloader before channel teardown, and skip deferred channel stop/start after cancellation is observed.

## Reproduction Assessment

Yes, at source level: deferred channel reload waits for active work and then starts channels without a cancellation context, while shutdown stops channels before stopping the reloader. I did not run a live Telegram webhook reproduction.

## Solution Assessment

Yes, the best fix is gateway-owned cancellation/drain for active config reload work; same-channel single-flight start is useful but too narrow because the stale reload can be captured by an older restart lifecycle.

Current main still lets a deferred hot reload continue after SIGUSR1 restart shutdown begins: `src/gateway/server-reload-handlers.ts` waits for active work and then restarts channels without cancellation, `src/gateway/config-reload.ts` stops the watcher without aborting/draining an active reload, and `src/gateway/server-close.ts` stops channels before stopping the reloader.
