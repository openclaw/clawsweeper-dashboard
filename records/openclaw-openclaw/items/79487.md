## Summary

Keep this open: current main still has a source-reproducible gateway lifecycle race, and the fixes-linked implementation path is the still-open https://github.com/openclaw/openclaw/pull/79510. The issue should stay open until that PR, or an equivalent cancellation/fencing fix, lands.

## What This Changes

The issue reports a deferred Telegram channel hot-reload racing with a SIGUSR1/gateway-tool restart and starting the same webhook listener twice.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79510 or an equivalent gateway lifecycle fix that cancels or fences deferred channel reloads at shutdown/restart before closing this issue as implemented.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: defer a channel hot reload behind active work, trigger SIGUSR1/gateway-tool restart before the deferral drains, and let the old reload handler call `startChannel` while the new gateway lifecycle starts channels.

## Solution Assessment

Yes. A shutdown/restart cancellation fence for pending deferred reloads is the narrow maintainable fix; a process-wide channel single-flight would be broader and would mask stale lifecycle work instead of stopping it.
