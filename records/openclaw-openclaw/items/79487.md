## Summary

Keep open: current main still appears vulnerable to the deferred channel-reload/restart overlap, and the issue already has an open unmerged fix PR that references it with closing syntax. The issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports a gateway lifecycle race where a deferred Telegram channel hot reload and an in-process restart both start the same webhook listener, causing EADDRINUSE restart noise.

## Best Possible Solution

Land a narrow gateway lifecycle fix like https://github.com/openclaw/openclaw/pull/79510 that aborts and drains config hot reloads before channel teardown and skips channel restart work after shutdown/restart cancellation.

## Reproduction Assessment

Yes by source inspection. On current main, defer a channel hot reload behind active work, trigger SIGUSR1 or gateway-tool restart during that deferral, and the old reloader can still call startChannel while the new gateway lifecycle starts channels.

## Solution Assessment

Yes, the cancellation/fencing direction is the narrowest maintainable fix. A process-wide channel single-flight would be broader and would mask stale lifecycle work instead of stopping it.
