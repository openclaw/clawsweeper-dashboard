## Summary

Keep open: current main still exposes the reported lifecycle race by source inspection, and the issue is already paired with an open closing PR that implements the cancellation/fencing approach.

## What This Changes

The issue reports a gateway lifecycle race where a deferred Telegram channel hot reload and an in-process restart can both start the same listener, causing `EADDRINUSE` auto-restart noise.

## Best Possible Solution

Land or replace the linked fix with a narrow gateway lifecycle fence: stop or abort config hot reload work at shutdown/restart, skip deferred channel restarts after cancellation, and keep regression coverage around the reloader, reload handlers, and close order.

## Reproduction Assessment

Yes. Source inspection on current main shows a deferred channel reload can resume into `startChannel` after SIGUSR1 restart begins, while the new gateway iteration can also start channels.

## Solution Assessment

Yes. Making deferred hot reload cancellation-aware at the shutdown/restart boundary is the narrow maintainable fix; a process-wide channel single-flight would be broader and would mask stale lifecycle work instead of stopping it.
