## Summary

Close as duplicate/superseded: the WeCom timeline matches the broader embedded session takeover regression tracked in the canonical open issue and active fix PR, while current main already fixed the fallback-chain cascade for direct takeover errors.

## What This Changes

The issue asks OpenClaw to stop producing a post-reply `EmbeddedAttemptSessionTakeoverError` on every WeCom websocket message and to avoid duplicate replies when fallback is configured.

## Best Possible Solution

Land the canonical session-lock fix for lock-owned in-process writes, then only reopen a WeCom-specific thread if `streamPlaceholderContent` still proves a distinct second embedded run after that fix.

## Reproduction Assessment

Yes for the shared root at source level: current code explains the paired session/global lane diagnostics and the session-file takeover throw after prompt-lock release. I did not run a live WeCom websocket reproduction, so the WeCom-specific streaming hypothesis remains unproven.

## Solution Assessment

Yes: closing in favor of the canonical issue and active session-lock PR avoids splitting the same root cause across channel-specific reports. A WeCom-only fix should wait until the core fence fix lands or a post-fix trace proves a distinct plugin second-run bug.
