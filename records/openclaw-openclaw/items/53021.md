## Summary

Keep open: the display-URL bug is still present on current main, but this branch should not merge as-is because it is stale against the active status formatter, lacks real behavior proof, and bundles unrelated user-visible diagnostics plus a sensitive document-intake skill.

## What This Changes

The PR normalizes selected Control UI display URL callers for `gateway.bind: "lan"`, changes embedded-agent user error text/context propagation, and adds a Drop Box intake skill with a PDF rendering helper.

## Best Possible Solution

Land a narrow current-main fix that centralizes display-only LAN-to-loopback normalization for user-facing Control UI URLs across dashboard, status, configure, daemon status, and setup finalization while splitting or dropping unrelated agent error-copy and Drop Box skill work.

## Reproduction Assessment

Yes, by source inspection: raw `lan` binds still resolve to LAN hosts, and active status/configure/daemon/setup display callers still pass raw bind values. I did not run a live CLI flow because this review is read-only.

## Solution Assessment

No. LAN-to-loopback display normalization is the right direction, but this PR is not the best implementation because it targets stale status code and bundles unrelated security/privacy-sensitive changes.
