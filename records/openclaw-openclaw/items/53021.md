## Summary

Keep open: the LAN display-URL fix is valid, but this branch is not mergeable as-is because one URL hunk targets a stale status path, the head bundles unrelated agent error-message and Drop Box skill changes with security/privacy concerns, and after-fix real behavior proof is still missing.

## What This Changes

The branch coerces some Control UI display URL callers from `lan` to `loopback`, changes embedded-agent transient/network error text, and adds a Drop Box document-intake skill.

## Best Possible Solution

Land a narrow current-main fix that centralizes display-only LAN-to-loopback normalization for user-facing Control UI/Dashboard links, while splitting or dropping the agent diagnostics and Drop Box skill work.

## Reproduction Assessment

Yes, by source inspection: raw `lan` still resolves to a LAN IPv4 host, and active status/configure/daemon/setup display callers still pass raw bind values. I did not run a live CLI reproduction because this review is read-only.

## Solution Assessment

No. LAN-to-loopback display normalization is the right direction, but the PR is stale for current status rendering and mixes unrelated security/privacy-sensitive changes into the bug fix.
