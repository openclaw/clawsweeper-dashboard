## Summary

Keep open. Current main still defaults empty-success background exec notifications off, so this PR addresses a real orchestration gap discussed around #18237 and #74308/#50398, but the branch is dirty and stale against newer exec-event wake semantics and generated/public docs.

## What This Changes

The PR flips `notifyOnExitEmptySuccess` from opt-in to opt-out, aligns exec runtime/test defaults, shortens the exec-exit wake coalesce delay, and updates source config help/type comments.

## Best Possible Solution

Land a rebased update or narrow replacement that makes empty-success exec completion notifications opt-out consistently while preserving current exec-event wake routing and updating docs, generated config, tests, and changelog.

## Reproduction Assessment

Yes. The source/test path is deterministic: omitted config remains false in `createExecTool` and `runExecProcess`, `maybeNotifyOnExit` returns for completed no-output runs, and the current no-op test matrix expects the default case to skip notifications.

## Solution Assessment

No, not as currently patched. The default-on direction is plausible, but the branch must preserve the shipped `exec-event` wake contract and update generated/public docs plus changelog before merge.
