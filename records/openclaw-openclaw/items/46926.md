## Summary

Keep open: current main still defaults empty-success exec notifications off, and this PR addresses a real orchestration gap, but the branch is stale and would regress the current exec-event wake contract while missing generated/public docs and changelog coverage.

## What This Changes

The PR flips `tools.exec.notifyOnExitEmptySuccess` to default true, aligns exec runtime/test defaults, shortens exec-exit wake coalescing, and updates source config help/type comments.

## Best Possible Solution

Land a rebased update or narrow replacement that makes empty-success exec completion notifications opt-out consistently while preserving current exec-event wake routing and syncing tests, generated docs, public docs, and changelog.

## Reproduction Assessment

Yes, source-level. On current main, omitted config stays false in `createExecTool` and `runExecProcess`, `maybeNotifyOnExit` returns for completed empty-output runs, and the no-op test matrix asserts that default case queues no event.

## Solution Assessment

No, not as currently patched. The default-on direction is plausible, but the branch must be rebased to preserve the shipped `exec-event` wake contract and update generated/public docs plus changelog before it is the narrow maintainable fix.
