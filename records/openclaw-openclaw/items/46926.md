## Summary

Keep open. Current main still has the reported opt-in empty-success behavior, so the PR is not obsolete, but the branch is stale against the current exec-event heartbeat contract and would ship contradictory generated/public docs unless updated before merge.

## What This Changes

This PR makes empty-output successful background exec completion notifications opt-out, aligns runtime storage/tests/config help/types, and adds a 50 ms coalesce value to exec-exit heartbeat wake requests.

## Best Possible Solution

If maintainers accept the opt-out notification policy, rebase the branch onto current main, keep completion wakes classified as `exec-event`, make the default consistently `!== false` in tool creation and runtime storage, regenerate config schema/help artifacts, update public exec/background-process docs, and then handle any coding-agent guidance change as a narrow follow-up.

## Reproduction Assessment

Yes. A static current-main check gives a high-confidence reproduction path: omitted `notifyOnExitEmptySuccess` flows through `=== true`, empty successful completions return before event enqueue, and the current no-op test matrix expects the default case to skip notifications. I did not run tests because this was a read-only review.

## Solution Assessment

No, not as currently patched. The default-on direction may be acceptable, but the branch should preserve the shipped `exec-event` heartbeat contract and update generated schema plus public docs before merge; an explicit per-call or config-policy follow-up is safer if maintainers reject the default change.
