## Summary

Keep open: current main still sends OpenAI Completions tool payloads without checking `compat.supportsTools`, and this issue is already paired with open PR #74738 plus upstream badlogic/pi-mono#4040 context.

## What This Changes

OpenAI-compatible chat-completions models should omit populated `tools`, default `tool_choice`, and `tools: []` history fallback payloads when `compat.supportsTools` is `false`.

## Best Possible Solution

Land PR #74738 or an equivalent narrow patch that applies `supportsModelTools(model)` inside `buildOpenAICompletionsParams()` for both populated tool payloads and the tool-history fallback, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass an OpenAI Completions model with `compat.supportsTools: false` and non-empty `context.tools`; current main still writes `params.tools`, and tool-call history still writes `tools: []`.

## Solution Assessment

Yes. Enforcing the existing `supportsModelTools()` contract at the OpenAI Completions payload boundary is the narrowest maintainable fix and avoids provider-specific workarounds.
