## Summary

Keep open: current main still emits OpenAI-compatible completions `tools`, `tool_choice`, and the tool-history `tools: []` fallback without checking `compat.supportsTools`, and open PR #74738 is already paired with this issue but is unmerged. The upstream pi-ai fix context is useful, but OpenClaw's local transport builder still needs the narrow guard before this can close.

## What This Changes

OpenAI-compatible chat-completions models configured with `compat.supportsTools: false` should not receive tool payload fields when tools are available.

## Best Possible Solution

Land PR #74738 or an equivalent narrow patch that applies `supportsModelTools(model)` inside `buildOpenAICompletionsParams()` for both populated tool payloads and the tool-history fallback, with regression tests and the existing user-facing changelog expectation preserved.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildOpenAICompletionsParams()` with `model.compat.supportsTools: false` and non-empty `context.tools`; current main still reaches the `params.tools = convertTools(...)` branch. A second source path is tool-call history with no tools, which still writes `tools: []`.

## Solution Assessment

Yes. The best fix is the narrow transport-boundary guard using the existing `supportsModelTools()` contract; relying only on the upstream pi-ai bump does not fix OpenClaw's local payload builder.
