## Summary

Keep open: current main still has the reported OpenAI Completions payload gap, and the issue is paired with open, unmerged PR #74738 using closing syntax. The source contract exists, but the payload builder still emits `tools`, `tool_choice`, and `tools: []` without checking `compat.supportsTools` at that boundary.

## What This Changes

OpenAI-compatible chat-completions models configured with `compat.supportsTools: false` should omit populated tool payloads, `tool_choice`, and the `tools: []` tool-history fallback from outbound completions requests.

## Best Possible Solution

Land PR #74738 or an equivalent narrow patch that enforces `supportsModelTools(model)` inside `buildOpenAICompletionsParams()` for both populated tools and the tool-history fallback, with regression tests for disabled and default-enabled behavior.

## Reproduction Assessment

Yes. On current main, a source-level reproduction is high-confidence: call `buildOpenAICompletionsParams()` with `model.compat.supportsTools: false` and non-empty `context.tools`, or with tool-call history and no `context.tools`; the builder can still write tool payload fields.

## Solution Assessment

Yes. The narrowest maintainable fix is to enforce the existing `supportsModelTools()` contract at the OpenAI Completions payload boundary rather than adding provider-specific workarounds elsewhere.
