## Summary

Keep open: current main still has the reported OpenAI Completions request-shaping gap. The config contract accepts `compat.supportsTools: false` and the embedded runner uses `supportsModelTools()`, but `buildOpenAICompletionsParams()` still emits `tools`, `tool_choice`, and the tool-history `tools: []` fallback without checking that helper. A linked fix PR is open and unmerged, so this issue should remain open until that lands or is replaced.

## What This Changes

The issue requests that OpenAI-compatible chat-completions payloads omit tools and tool-history fallback fields when a model sets `compat.supportsTools: false`.

## Best Possible Solution

Land PR #74738 or an equivalent narrow fix that gates both populated tools and the tool-history `tools: []` fallback inside `buildOpenAICompletionsParams()`, with regression tests for `supportsTools: false` and unchanged default-true behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: call `buildOpenAICompletionsParams()` with `model.compat.supportsTools: false` and non-empty `context.tools`, or with tool-call history and no current tools; current main still writes a `tools` field on those paths.

## Solution Assessment

Yes. The narrow maintainable solution is to enforce the existing `supportsModelTools()` contract at the OpenAI Completions payload builder boundary, because that protects direct transport callers and any upstream caller that accidentally passes an unfiltered tool context.
