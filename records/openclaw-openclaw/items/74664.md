## Summary

Keep open: current main still has the OpenAI Completions payload gap, and the report is already paired with open, unmerged PR #74738 plus upstream badlogic/pi-mono#4040 context.

## What This Changes

OpenAI-compatible chat-completions models marked `compat.supportsTools: false` should not receive populated or empty `tools` fields in outbound payloads.

## Best Possible Solution

Land PR #74738 or an equivalent narrow patch that applies `supportsModelTools(model)` in `buildOpenAICompletionsParams()` for populated tools, default `tool_choice`, and the empty-tools fallback, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: pass a model with `compat.supportsTools: false` and a `context.tools` array to `buildOpenAICompletionsParams()`, and current main still emits `params.tools`; the runner-filtered empty-array path can also emit `tools: []`.

## Solution Assessment

Yes. Enforcing the existing `supportsModelTools()` contract at the OpenAI Completions payload boundary is the narrowest maintainable fix; the dependency bump alone does not guard OpenClaw's local transport builder.
