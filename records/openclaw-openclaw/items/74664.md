## Summary

Keep open: current main still exposes the reported OpenAI Completions payload bug, and PR #74738 is already linked with closing syntax but remains open/unmerged, so the issue should continue tracking the fix until that PR or an equivalent patch lands.

## What This Changes

OpenAI-compatible chat-completions models configured with `compat.supportsTools: false` should not receive `tools`, `tool_choice`, or tool-history fallback payloads.

## Best Possible Solution

Land PR #74738 or an equivalent narrow patch that gates populated tools, `tool_choice`, and the `tools: []` history fallback with `supportsModelTools(model)`, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. On current main, a source-level reproduction is deterministic: pass a model with `compat.supportsTools: false` and non-empty `context.tools` to `buildOpenAICompletionsParams()`, and the function still writes `params.tools`; tool-call history without current tools still writes `params.tools = []`.

## Solution Assessment

Yes. Applying the existing `supportsModelTools()` contract at the Completions payload boundary is the narrow maintainable fix and avoids provider-specific workarounds.
