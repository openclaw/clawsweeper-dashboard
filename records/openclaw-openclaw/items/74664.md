## Summary

Keep open: current main still emits OpenAI Completions tool payload fields without checking the existing `compat.supportsTools` contract, and #74738 is the paired open fix PR that should be reviewed or landed first.

## What This Changes

The issue asks OpenAI-compatible chat-completions payload construction to omit `tools`, `tool_choice`, and the `tools: []` fallback when a model sets `compat.supportsTools: false`.

## Best Possible Solution

Land #74738 or an equivalent narrow patch that applies `supportsModelTools(model)` in `buildOpenAICompletionsParams()` and adds regression coverage for populated tools and tool-history fallback suppression.

## Reproduction Assessment

Yes. Source inspection gives a deterministic current-main path: call `buildOpenAICompletionsParams()` with `compat.supportsTools: false` and non-empty `context.tools`, or with tool-call history and no current tools, and the builder reaches unguarded `params.tools` assignments.

## Solution Assessment

Yes. Reusing the existing `supportsModelTools()` helper inside the transport payload builder is the narrowest maintainable fix; provider-specific workarounds would leave the lower-level payload path open.
