## Summary

Keep open: current main still unconditionally emits OpenAI Completions tool payload fields when `context.tools` or tool-call history is present, and open PR #74738 is already paired with this issue.

## What This Changes

OpenAI-compatible chat-completions models with `compat.supportsTools: false` should not receive `tools`, `tool_choice`, or `tools: []` fallback payload fields.

## Best Possible Solution

Land PR #74738 or an equivalent narrow patch that gates populated tools, default/explicit `tool_choice`, and the tool-history `tools: []` fallback with the existing `supportsModelTools(model)` contract, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildOpenAICompletionsParams()` with an OpenAI Completions model whose `compat.supportsTools` is false and a non-empty `context.tools`; current main still writes `params.tools` and may write `tool_choice`.

## Solution Assessment

Yes. Applying the existing `supportsModelTools()` contract at the local OpenAI Completions payload boundary is the narrowest maintainable fix; the dependency bump alone does not guard OpenClaw's wrapper.
