## Summary

Keep open: current main still accepts `compat.supportsTools: false`, but the OpenAI Completions transport-aware payload builder does not consult that contract before emitting tool payload fields. The related upstream pi-ai fix and dependency bump are useful context, but OpenClaw's own wrapper still needs the same guard, and the linked OpenClaw fix PR is open and unmerged.

## What This Changes

OpenAI-compatible chat-completions models configured with `compat.supportsTools: false` should not receive `tools`, `tool_choice`, or the `tools: []` tool-history fallback in outbound payloads.

## Best Possible Solution

Land PR #74738 or an equivalent narrow fix in OpenClaw's OpenAI Completions transport builder that gates both populated tool payloads and the tool-history fallback with the existing `supportsModelTools()` contract, with focused regression tests for disabled and default-enabled behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: call `buildOpenAICompletionsParams()` with `model.compat.supportsTools: false` and either non-empty `context.tools` or replayed tool-call history, and the builder can still write tool payload fields.

## Solution Assessment

Yes, the linked fix direction is the narrow maintainable solution because it enforces the existing model-tool-support contract at the payload boundary that actually sends the OpenAI-compatible request. The issue should remain open until that PR or an equivalent patch merges.
