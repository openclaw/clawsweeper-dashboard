## Summary

Keep open: current main still does not apply the existing `compat.supportsTools` contract at the OpenAI Completions payload builder, and open PR #74738 is already paired to this issue with closing syntax.

## What This Changes

The issue asks OpenAI-compatible chat-completions payloads to omit `tools`, `tool_choice`, and the `tools: []` history fallback when the model has `compat.supportsTools: false`.

## Best Possible Solution

Land #74738 or an equivalent narrow patch that gates OpenAI Completions tool emission and the history fallback with `supportsModelTools(model)`, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. Passing a model with `compat.supportsTools: false` and non-empty `context.tools` to `buildOpenAICompletionsParams()` on current main deterministically reaches the unguarded `params.tools` assignment; tool-call history also reaches the unguarded `tools: []` fallback.

## Solution Assessment

Yes. Applying the existing `supportsModelTools()` helper inside `buildOpenAICompletionsParams()` is the narrow maintainable fix; provider-specific workarounds or relying only on upstream pre-filtering would leave this lower-level escape hatch open.
