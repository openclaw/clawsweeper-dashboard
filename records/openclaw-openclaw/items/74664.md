## Summary

Keep open: current main still exposes `compat.supportsTools: false` as a supported config/docs path, but the OpenAI Completions transport payload builder does not consult that contract before emitting `tools`, `tool_choice`, or the `tools: []` history fallback. The issue is already paired with open closing PR #74738, so it should remain open until that PR or an equivalent fix merges.

## What This Changes

The issue asks OpenClaw's OpenAI-compatible chat-completions payload builder to honor `compat.supportsTools: false` by omitting tool payload fields.

## Best Possible Solution

Land PR #74738 or an equivalent narrow OpenClaw fix that gates populated tool payloads, `tool_choice`, and the tool-history `tools: []` fallback in `buildOpenAICompletionsParams()` with the existing `supportsModelTools()` contract, with focused regression tests and the required user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: call `buildOpenAICompletionsParams()` with `model.compat.supportsTools: false` and non-empty `context.tools` or tool-call history, and the builder can still write tool payload fields at the transport boundary.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix because it enforces the existing model-tool-support contract at the payload builder that actually sends the OpenAI-compatible request; the issue should not close until that patch or an equivalent one merges.
