## Summary

Keep open: current main still has the OpenAI Completions payload builder emitting `tools`, `tool_choice`, and the `tools: []` history fallback without checking the existing `compat.supportsTools` contract, and #74738 is already the paired open fix PR.

## What This Changes

The issue requests that OpenAI-compatible chat-completions payloads omit tool fields when a model is configured with `compat.supportsTools: false`.

## Best Possible Solution

Land #74738 or an equivalent narrow patch that gates OpenAI Completions tool emission and the history fallback with `supportsModelTools(model)`, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a deterministic current-main reproduction path: call `buildOpenAICompletionsParams()` with `compat.supportsTools: false` and non-empty `context.tools`, or tool-call history with no current tools, and the builder reaches unguarded `params.tools` assignments.

## Solution Assessment

Yes. Applying the existing `supportsModelTools()` helper inside `buildOpenAICompletionsParams()` is the narrow maintainable fix; provider-specific workarounds would leave this lower-level payload path open.
