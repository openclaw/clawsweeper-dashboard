## Summary

Keep open: current main still has the OpenAI Completions payload gap, and the issue is already paired with open, unmerged PR #74738 plus an upstream badlogic/pi-mono#4040 fix context.

## What This Changes

OpenAI-compatible chat-completions models that set `compat.supportsTools: false` should not receive `tools`, `tool_choice`, or the tool-history `tools: []` fallback.

## Best Possible Solution

Land PR #74738 or an equivalent narrow patch that applies `supportsModelTools(model)` inside `buildOpenAICompletionsParams()` for populated tool payloads and the tool-history fallback, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: pass a model with `compat.supportsTools: false` and non-empty `context.tools`, and current main still writes `params.tools`; tool-call history without tools still writes `tools: []`.

## Solution Assessment

Yes. Enforcing the existing `supportsModelTools()` contract at the OpenAI Completions payload boundary is the narrowest maintainable fix and avoids provider-specific workarounds.
