## Summary

Keep open: current main still has the reported OpenAI Completions transport gap. The config and docs support `compat.supportsTools: false`, but `buildOpenAICompletionsParams()` still emits `tools`, `tool_choice`, and `tools: []` without consulting the existing `supportsModelTools()` contract; open PR #74738 is the active closing fix and should land before this issue closes.

## What This Changes

The issue asks OpenAI-compatible chat-completions payload construction to honor `compat.supportsTools: false` by omitting tool payload fields and the tool-history `tools: []` fallback.

## Best Possible Solution

Land PR #74738 or an equivalent narrow OpenClaw patch that gates populated tool payloads, explicit/default `tool_choice`, empty-array tool payloads, and tool-history fallback with `supportsModelTools(model)`, with focused regression coverage for disabled and default-enabled behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: call `buildOpenAICompletionsParams()` with an `openai-completions` model whose `compat.supportsTools` is `false` and a `context.tools` array, and the builder still writes `params.tools`; with tool-call history and no tools it can also write `tools: []`.

## Solution Assessment

Yes. Applying the existing `supportsModelTools()` contract at the OpenAI Completions payload boundary is the narrow maintainable fix because that is the last place before the outbound chat-completions request is sent.
