## Summary

Keep open: current main still has the OpenAI Completions transport boundary gap. The embedded runner already filters registered tools when `supportsModelTools()` is false, but `buildOpenAICompletionsParams()` can still emit `tools`, `tool_choice`, and the tool-history `tools: []` fallback if any caller passes tool context through; open PR #74738 is the active narrow fix and uses closing syntax, so this issue should remain open until that PR or an equivalent patch lands.

## What This Changes

The issue requests that OpenAI-compatible chat-completions payload construction omit `tools`, `tool_choice`, and tool-history `tools: []` when `compat.supportsTools` is false.

## Best Possible Solution

Review and land PR #74738, or an equivalent narrow patch, so the OpenAI Completions payload builder enforces the existing `supportsModelTools(model)` contract for both populated tools and tool-history fallback while preserving default tool-capable behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: call `buildOpenAICompletionsParams()` with `model.compat.supportsTools: false` and non-empty `context.tools`, or with tool-call history and no current tools; the builder can still write tool payload fields without consulting `supportsModelTools()`.

## Solution Assessment

Yes. The best fix is to apply the existing model-tool-support helper at the OpenAI Completions payload boundary, because that is the last point before the unsupported fields are sent and it covers direct transport callers as well as future runtime paths.
