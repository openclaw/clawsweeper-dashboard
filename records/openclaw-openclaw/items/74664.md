## Summary

Keep open: current main still has the reported OpenAI Completions payload gap. The config and helper already define `compat.supportsTools: false`, but `buildOpenAICompletionsParams()` still emits `tools`, `tool_choice`, and the `tools: []` history fallback without checking that contract. The issue is already paired with open, unmerged PR #74738, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Requests that OpenAI-compatible chat-completions payload building omit tool payload fields, including the tool-history fallback, when a model sets `compat.supportsTools: false`.

## Best Possible Solution

Land PR #74738 or an equivalent narrow OpenClaw patch that gates both populated tool payloads and the tool-history fallback in `src/agents/openai-transport-stream.ts` with `supportsModelTools(model)`, with focused tests for disabled and default-enabled behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: call `buildOpenAICompletionsParams()` with `model.compat.supportsTools: false` and non-empty `context.tools`, or with tool-call history and no `context.tools`; the builder can still write tool payload fields.

## Solution Assessment

Yes. The narrow maintainable fix is to enforce the existing `supportsModelTools()` contract at the OpenAI Completions payload boundary, because that is the code path that actually sends the unsupported request fields.
