## Summary

Keep open: current main still has the OpenAI Completions transport gap, and the issue is paired with open, unmerged PR #74738. The schema/helper/docs already establish `compat.supportsTools: false` as a valid contract, but `buildOpenAICompletionsParams()` still writes `tools`, `tool_choice`, and the tool-history `tools: []` fallback without checking that contract.

## What This Changes

OpenAI-compatible chat-completions payloads should omit tool fields when a model is configured with `compat.supportsTools: false`.

## Best Possible Solution

Land PR #74738 or an equivalent narrow patch that applies `supportsModelTools(model)` inside `buildOpenAICompletionsParams()` for both populated tool payloads and the tool-history fallback, with focused regression tests and a changelog entry for the user-facing provider compatibility fix.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: pass `model.compat.supportsTools: false` with non-empty `context.tools` and the builder still writes `params.tools`; pass tool-call history with no tools and it still writes `tools: []`.

## Solution Assessment

Yes. The narrow maintainable fix is to enforce the existing `supportsModelTools()` contract at the OpenAI Completions payload boundary instead of adding provider-specific workarounds or relying on the dependency bump alone.
