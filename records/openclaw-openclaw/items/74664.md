## Summary

Keep open: current main still sends OpenAI Completions tool payloads based on `context.tools` without honoring `compat.supportsTools: false`, and open PR #74738 already tracks the narrow fix with upstream badlogic/pi-mono#4040 context.

## What This Changes

Honor `compat.supportsTools: false` for OpenAI-compatible chat-completions payloads so OpenClaw omits `tools`, `tool_choice`, and the empty tool-history fallback.

## Best Possible Solution

Land PR #74738 or an equivalent narrow main patch that reuses `supportsModelTools(model)` at the OpenAI Completions payload builder with regression tests and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: call `buildOpenAICompletionsParams()` with `compat.supportsTools: false` and non-empty `context.tools`; current main enters the `context.tools` branch and writes `params.tools`, while tool-call history can still write `params.tools = []`.

## Solution Assessment

Yes. Enforcing the existing `supportsModelTools()` contract at payload construction is the narrowest maintainable fix and avoids relying only on upstream dependency behavior while OpenClaw still has a local transport path.
