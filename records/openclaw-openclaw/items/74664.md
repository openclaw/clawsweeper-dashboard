## Summary

Keep open: current main still accepts `compat.supportsTools: false` as a model contract, but the OpenAI Completions transport payload builder does not consult that contract before emitting populated `tools`, `tool_choice`, or the `tools: []` history fallback. The issue is already paired with open fix PR #74738, so the right cleanup is to leave this issue open until that PR or an equivalent patch lands.

## What This Changes

The issue reports that OpenAI-compatible chat-completions requests still include tool payload fields when a model sets `compat.supportsTools: false`.

## Best Possible Solution

Land PR #74738 or an equivalent narrow patch in `buildOpenAICompletionsParams()` that gates populated tools, `tool_choice`, and the tool-history `tools: []` fallback with the existing `supportsModelTools()` contract, plus focused regression coverage for disabled and default-enabled behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: call `buildOpenAICompletionsParams()` with `model.compat.supportsTools: false` and non-empty `context.tools`, and the current code enters the `context.tools` branch and writes `params.tools`; replayed tool-call history can also write `params.tools = []`.

## Solution Assessment

Yes. The linked fix direction is the narrow maintainable solution because it enforces the existing model-tool-support contract at the OpenAI Completions payload boundary that actually sends the request.
