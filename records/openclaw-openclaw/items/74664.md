## Summary

Keep open: current main still has the OpenAI Completions transport payload builder emitting tool fields without checking the existing `compat.supportsTools` contract, and #74738 is already the paired open fix PR.

## What This Changes

Honor `compat.supportsTools: false` for OpenAI-compatible chat-completions payloads by omitting tools, `tool_choice`, and the empty `tools: []` fallback.

## Best Possible Solution

Land #74738 or an equivalent narrow patch that applies `supportsModelTools(model)` to both OpenAI Completions tool emission paths with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a deterministic current-main path: call `buildOpenAICompletionsParams()` with `compat.supportsTools: false` plus non-empty `context.tools`, or with tool-call history and no current tools, and the builder reaches unguarded `params.tools` assignments.

## Solution Assessment

Yes. Reusing the existing `supportsModelTools()` helper inside `buildOpenAICompletionsParams()` is the narrow maintainable fix; provider-specific workarounds or only bumping upstream `pi-ai` would leave this local OpenClaw transport path open.
