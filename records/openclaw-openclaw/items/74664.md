## Summary

Current main still has the reported OpenAI Completions payload gap: `compat.supportsTools` is a documented model contract and helper-backed flag, but `buildOpenAICompletionsParams()` still emits `tools`, `tool_choice`, and the tool-history `tools: []` fallback without checking that flag. This should remain open because PR #74738 is open, unmerged, and explicitly paired as the active fix path.

## What This Changes

The issue asks OpenAI-compatible chat-completions payload construction to omit tool payload fields when a model is configured with `compat.supportsTools: false`.

## Best Possible Solution

Land PR #74738 or an equivalent narrow patch that enforces `supportsModelTools(model)` inside `buildOpenAICompletionsParams()` for both populated tools and the tool-history fallback, with regression tests for disabled and default-enabled behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main by calling `buildOpenAICompletionsParams()` with `model.compat.supportsTools: false` and either non-empty `context.tools` or tool-call history; the builder can still write tool payload fields.

## Solution Assessment

Yes. The narrowest maintainable fix is to enforce the existing `supportsModelTools()` contract at the OpenAI Completions payload boundary rather than relying only on upstream runner filtering or provider-specific workarounds.
