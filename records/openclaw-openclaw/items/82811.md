## Summary

Keep open: current main still has a credible source-level regression where the managed memory-core dreaming cron runs real work inside a cron-triggered `before_agent_reply` hook while the cron watchdog still considers the run pre-execution at `runtime-plugins`. `lightContext: true` is still forwarded, so the likely bug is the hook/watchdog phase boundary rather than loss of lightweight bootstrap behavior.

## What This Changes

The issue reports that the managed memory-core Memory Dreaming Promotion cron job began failing after OpenClaw 2026.5.12 with `cron: isolated agent run stalled before execution start (last phase: runtime-plugins)`.

## Best Possible Solution

Mark cron `before_agent_reply` work that claims the run as real execution, or emit a dedicated plugin-hook execution phase before long hook handling, while preserving lightweight cron bootstrap and the pre-execution watchdog for true startup stalls.

## Reproduction Assessment

Yes, by source inspection: a managed memory-core dreaming cron can reach `runtime_plugins`, run a long cron `before_agent_reply` promotion hook, and remain in the pre-execution watchdog state until the 60s guard aborts. I did not execute the live cron job in this read-only review.

## Solution Assessment

No, current behavior is not the best solution because plugin-owned cron hook work can be the intended execution path even without a model call. The narrow fix should adjust phase/watchdog handling around claimed cron hooks, not disable the watchdog or require user config changes.
