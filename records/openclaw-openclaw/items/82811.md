## Summary

Keep open: current main still has a plausible source-level path where memory-core dreaming cron work runs inside a cron-triggered `before_agent_reply` hook while the cron watchdog still considers the run pre-execution at `runtime-plugins`. `lightContext: true` is still honored, so the likely bug is the watchdog/hook phase boundary, not loss of the lightweight cron flag.

## What This Changes

The issue reports memory-core isolated cron dreaming jobs stalling at `runtime-plugins` after the v2026.5.12 upgrade and asks whether `lightContext` still applies.

## Best Possible Solution

Teach the isolated cron watchdog to treat plugin-handled cron hook work as real execution or emit a dedicated phase before running long `before_agent_reply` handlers, while preserving the existing lightweight cron bootstrap behavior.

## Reproduction Assessment

Yes, by source inspection: an isolated memory-core dreaming cron whose hook-handled sweep runs longer than the 60s pre-execution watchdog can stay at `runtime-plugins` and be aborted before any model-call phase. I did not run the live cron job in this read-only review.

## Solution Assessment

No, the current behavior is not the best solution because plugin-owned cron work can be legitimate execution even without a model call. A narrow fix should adjust cron phase/watchdog handling for hook-handled cron runs rather than disabling the watchdog or changing the managed job config.
