## Summary

Keep open: current main still has a clear source-level path where a memory-core managed cron turn can enter long `before_agent_reply` dreaming work while the cron watchdog still considers the run pre-execution at `runtime-plugins`. The lightweight flag is still honored, so the remaining bug appears to be the cron watchdog and plugin-hook phase boundary rather than a lost `lightContext` setting.

## What This Changes

The issue reports a regression where memory-core's managed isolated dreaming promotion cron stalls at `runtime-plugins` after upgrading to OpenClaw 2026.5.12, and asks whether `lightContext: true` is still respected.

## Best Possible Solution

Adjust cron phase/watchdog handling so cron `before_agent_reply` work that handles the turn is counted as execution progress or reported as a distinct execution phase, while preserving lightweight cron bootstrap and setup-stall diagnostics.

## Reproduction Assessment

Yes, by source inspection: run an isolated `agentTurn` cron with the memory-core dreaming token and `lightContext: true`, then have the cron `before_agent_reply` handler run longer than the 60s pre-execution watchdog after the last emitted phase is `runtime_plugins`. I did not execute the live cron job in this read-only review.

## Solution Assessment

Yes, the best fix direction is narrow: update the cron watchdog/embedded-run phase boundary for hook-handled cron turns. Changing the managed memory-core job shape or disabling the watchdog would be broader and would weaken existing stall detection.
