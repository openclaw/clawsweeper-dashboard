## Summary

Keep open: current main still creates the reported memory-core isolated cron shape, still honors `lightContext: true`, and still has a source-level path where long `before_agent_reply` plugin work can be killed while the watchdog records `runtime-plugins` as pre-execution. ## What This Changes

Investigate a reported OpenClaw 2026.5.12 regression where an isolated memory-core dreaming cron stalls at the runtime-plugins phase before execution.

## Best Possible Solution

Adjust isolated cron phase/watchdog handling so plugin-claimed cron hook work counts as execution, while preserving lightweight cron bootstrap and the pre-execution stall guard for real startup stalls.

## Reproduction Assessment

Yes, by source inspection: a memory-core managed isolated cron can enter runtime plugin loading, run a long `before_agent_reply` dreaming sweep, and remain in a pre-execution watchdog state with `runtime-plugins` as the last phase. I did not run the reporter's live cron job in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to mark hook-claimed cron work as execution or emit a dedicated execution phase before long hook work. Raising watchdog defaults or changing the managed job config would be a broader workaround, not the best root fix.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/82811 on current main. The bug is that memory-core managed dreaming cron jobs run as isolated `agentTurn` jobs with `lightContext: true`; after runtime plugins load, the embedded runner can execute a long cron `before_agent_reply` hook for `__openclaw_memory_core_short_term_promotion_dream__` while the cron watchdog still considers the run pre-execution at `runtime-plugins`, causing `cron: isolated agent run stalled before execution start (last phase: runtime-plugins)`. Implement the narrow owner-boundary fix: treat hook-claimed cron work as execution progress or emit a dedicated execution phase before running long cron `before_agent_reply` handlers. Preserve lightweight bootstrap behavior, existing managed job shape, delivery mode `none`, and the watchdog's ability to catch true pre-execution stalls. Do not add public cron config knobs, do not merely increase watchdog durations, and do not change memory-core's managed cron schedule or payload token. Add focused regression coverage for a hook-claimed cron run that lasts past the pre-execution watchdog window and no longer aborts, plus adjacent memory-core/lightContext coverage as needed. Include a user-facing changelog fix if this changes shipped behavior.
