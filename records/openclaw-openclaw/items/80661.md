## Summary

Keep open: this is a plausible narrow feature PR, but it should not merge yet because defaults and per-agent `runRetries` are validated separately and then merged into combinations that can silently ignore an operator’s per-agent cap. The supplied real behavior proof is sufficient for the config-write surface, and I found no security or supply-chain concern.

## What This Changes

The PR adds `agents.defaults.runRetries` and `agents.list[].runRetries` config, schema/help/label/docs coverage, merge tests, and runtime consumption by the embedded Pi runner retry-limit helper.

## Best Possible Solution

If maintainers want this public knob, keep the embedded-runtime scope, add validation for the resolved defaults-plus-agent retry bounds, preserve defensive runtime clamps as a last resort, and land with focused tests and maintainer-owned release notes.

## Reproduction Assessment

Not applicable for the original request as a bug: this PR adds a new feature/config option. The blocking review finding is source-reproducible by combining a higher default `runRetries.min` with a lower per-agent `runRetries.max`, which the PR merges after per-object schema validation.

## Solution Assessment

No, not yet: the overall shape is narrow and maintainable, but the merged config invariant needs validation before this becomes a reliable operator-facing safety control. A safer fix is to reject invalid resolved `runRetries` bounds with regression coverage rather than silently clamping away the requested cap.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Validate merged runRetries overrides:** `src/agents/agent-scope-config.ts:137-140`
  - body: The new schema only checks each `runRetries` object before defaults are merged. With `agents.defaults.runRetries.min: 100` and an agent override of `{ max: 50 }`, this merge produces `{ min: 100, max: 50 }`, and the helper then clamps `max` back to `min`, silently ignoring the per-agent cap. Validate or reject the resolved merged bounds so per-agent overrides cannot be accepted but then discarded at runtime.
