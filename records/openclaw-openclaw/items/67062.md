## Summary

Keep open. Current main covers the reporter's concrete QQ CDN example through the default `*.qq.com.cn` QQ Bot media allowlist, but it still does not expose or wire a supported `channels.qqbot.mediaAllowHosts` option. The remaining request is valid and narrow, but it changes an SSRF allowlist surface, so it needs maintainer/security approval before an automated repair lane should implement it.

## What This Changes

The issue requests a `channels.qqbot.mediaAllowHosts` setting that merges user-configured host allowlist entries with the QQ Bot default media SSRF allowlist.

## Best Possible Solution

Implement this inside the bundled QQ Bot plugin: add `mediaAllowHosts` to the QQ Bot account/channel type, Zod schema, plugin manifest schema, generated metadata, and docs; normalize configured host patterns through the existing SSRF policy semantics; merge them with the conservative QQ/Tencent defaults; and thread the resulting policy into inbound attachment downloads and outbound URL fallback downloads while preserving HTTPS-only and private-network protections.

## Reproduction Assessment

Yes for the remaining config gap: static inspection of current main shows no QQ Bot `mediaAllowHosts` schema/type/docs/manifest surface, and both media download paths still call the fixed `downloadFile` helper. The original host-specific failure is partially obsolete because `*.qq.com.cn` is now in the default allowlist.

## Solution Assessment

Unclear until maintainers approve the security semantics. The narrow maintainable direction is a QQ Bot-owned config and policy merge, not a core migration or a broad media-fetch policy change.
