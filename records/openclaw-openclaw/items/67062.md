## Summary

Keep open. Current main covers the reporter's concrete `multimedia.nt.qq.com.cn` example through the default QQ Bot media allowlist, but it still does not expose or wire `channels.qqbot.mediaAllowHosts`; the remaining request is a security-sensitive config feature that needs maintainer approval rather than cleanup closure.

## What This Changes

The issue requests `channels.qqbot.mediaAllowHosts` so QQ Bot media downloads can merge operator-supplied host entries with the default SSRF allowlist.

## Best Possible Solution

Add a QQ Bot-owned config surface with schema, types, manifest metadata, docs, and tests, then merge normalized configured hosts with conservative defaults through the existing SSRF policy helpers while preserving HTTPS and private-network protections.

## Reproduction Assessment

## Solution Assessment

Unclear until maintainers approve the SSRF semantics. The narrow maintainable direction is a QQ Bot plugin-owned config merge through existing SSRF policy helpers, not a core migration or broad shared media-fetch rewrite.
