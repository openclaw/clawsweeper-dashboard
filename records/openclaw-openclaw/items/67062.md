## Summary

Keep open. Current main covers the reporter's concrete host example through the default `*.qq.com.cn` QQ Bot media allowlist, but there is still no supported `channels.qqbot.mediaAllowHosts` config surface or runtime merge point, so the security-sensitive config gap remains.

## What This Changes

The issue requests a QQ Bot `channels.qqbot.mediaAllowHosts` option so users can extend media-download SSRF host allowlists without editing source code.

## Best Possible Solution

Add the option inside the QQ Bot plugin, normalize configured host suffixes through the shared SSRF helpers, merge them with conservative QQ/Tencent defaults, and document the setting while preserving HTTPS-only and private-network protections.

## Reproduction Assessment

Yes. Static inspection of current main reproduces the remaining gap: QQ Bot has no supported `mediaAllowHosts` schema/type/metadata/docs entry, and both media download paths still use the fixed `downloadFile` policy. The original `multimedia.nt.qq.com.cn` example is no longer the best repro because `*.qq.com.cn` is now allowed by default.

## Solution Assessment

Unclear until maintainers approve the exact SSRF semantics. The narrow maintainable direction is QQ Bot-owned config and policy merging, not a broad core migration or a startup-time config repair.
