## Summary

Keep open. Current main covers the reporter's concrete `multimedia.nt.qq.com.cn` example through the default `*.qq.com.cn` QQ Bot media allowlist, but the requested `channels.qqbot.mediaAllowHosts` config still is not exposed or wired; this remains a valid, security-sensitive QQ Bot plugin follow-up rather than cleanup-close material.

## What This Changes

Add `channels.qqbot.mediaAllowHosts` so QQ Bot media downloads can merge configured host patterns with the default SSRF hostname allowlist.

## Best Possible Solution

Implement this inside the bundled QQ Bot plugin: add `mediaAllowHosts` to the account/channel config type, Zod schema, plugin manifest schema, generated metadata, and QQ Bot docs; normalize configured hosts through the shared SSRF policy helpers; merge them with the conservative QQ/Tencent defaults; and thread the resulting policy through inbound attachment downloads and outbound URL fallback downloads while preserving HTTPS-only, redirect, DNS, and private-network protections.

## Reproduction Assessment

Yes for the remaining config gap: static inspection of current main shows no supported QQ Bot `mediaAllowHosts` schema/type/manifest/docs surface, and both media download paths still call the fixed `downloadFile` helper. The original `multimedia.nt.qq.com.cn` example is no longer the best reproduction because `*.qq.com.cn` is now in the default allowlist.

## Solution Assessment

Unclear until maintainers approve the SSRF policy semantics. The narrow maintainable direction is a QQ Bot-owned config and policy merge, not a core migration or broad shared media-fetch policy change.
