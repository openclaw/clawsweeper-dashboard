## Summary

Keep open. Current main covers the reporter’s concrete `multimedia.nt.qq.com.cn` example through the static QQ Bot default allowlist, but it still has no supported `channels.qqbot.mediaAllowHosts` config or runtime merge point; adding that surface needs maintainer/security agreement on SSRF semantics.

## What This Changes

Add a QQ Bot `mediaAllowHosts` config option that merges user host entries with the default QQ Bot media SSRF hostname allowlist.

## Best Possible Solution

Add a QQ Bot-owned config surface with schema, types, manifest metadata, docs, and tests, then merge normalized configured hosts with conservative defaults through the existing SSRF policy path while preserving HTTPS and private-network protections.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a runtime bug; this is a feature/config gap. ## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers approve the SSRF semantics. The narrow maintainable direction is a QQ Bot plugin-owned config merge through the existing SSRF policy contract, not a core migration or broad shared media-fetch rewrite.
