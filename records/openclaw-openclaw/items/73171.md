## Summary

Keep open. Current main still does not implement the PR's auth-resolution timeout fallback, while the related local status false-negative was fixed through a different bounded local RPC fallback; this PR is useful optional hardening but needs maintainer judgment, rebase/integration, and a changelog entry before landing.

## What This Changes

The PR adds a timeout-bounded fallback around status gateway probe auth resolution, probing with empty auth plus a warning when auth lookup stalls, and adds a focused unit test for the never-settling auth path.

## Best Possible Solution

If maintainers still want auth-stall hardening, land a rebased version that composes with current `applyLocalStatusRpcFallback`, keeps warning visibility clear, adds the required changelog entry, and decides explicitly whether rejection should share the timeout fallback path. If the shipped local RPC fallback is considered sufficient, close this PR after that maintainer decision.

## Reproduction Assessment

Yes for the PR's unit-level behavior: mock `resolveGatewayProbeAuthResolution` with a never-settling promise and call `resolveGatewayProbeSnapshot({ opts: { timeoutMs: 1 } })`. Current main would wait on auth resolution before probing; the PR's added test expects empty auth and a timeout warning.

## Solution Assessment

Unclear as-is. The proposed timeout is narrow and plausible, but the linked user-facing regression was already fixed through a different local RPC fallback, so the best path depends on whether maintainers still want a separate auth-resolution stall boundary.
