## Summary

Keep open. Current main still leaves the TUI reasoning-stream experience incomplete: `/reasoning stream` is accepted and partly documented, but TUI help/completions omit `stream`, the TUI agent-event handler has no `thinking` renderer, and the backend stream path is still entangled with the channel callback behavior being addressed by separate open PRs.

## What This Changes

The issue asks for OpenClaw TUI reasoning stream output that visibly updates like OpenAI/Grok thinking status lines instead of only accepting the `/reasoning stream` mode.

## Best Possible Solution

Land one coordinated reasoning-stream UX: emit `thinking` events for gateway/TUI clients without the channel-only callback dependency, render them in the chosen TUI surface behind the existing visibility gates, and align command help, docs, and regression tests with the final behavior.

## Reproduction Assessment

Yes, at source level. Current main accepts `reasoningLevel: "stream"` and documents the mode, but the TUI command metadata omits `stream`, the TUI handler has no `thinking` renderer, and the agent stream path still has a callback gate that can prevent gateway-client thinking events.

## Solution Assessment

No for an isolated cleanup patch. The best path needs a maintainer UX/privacy decision and coordination with the overlapping open reasoning-stream PRs before wiring TUI rendering and docs as final behavior.
