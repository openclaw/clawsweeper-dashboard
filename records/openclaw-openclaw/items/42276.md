## Summary

Keep open. Current main still does not provide the requested live TUI reasoning-stream experience: `/reasoning stream` is not exposed in TUI command help/completions, `stream: "thinking"` agent events are ignored by the TUI handler, and the gateway/TUI path still has a callback gate that can prevent those thinking events from being emitted at all.

## What This Changes

The issue asks for OpenAI/Grok-style live reasoning updates in the OpenClaw TUI for `/reasoning stream`, with clearer behavior and a possible status/footer presentation.

## Best Possible Solution

Land a focused TUI-facing fix that exposes `stream` in `/reasoning` help/completions/usage, delivers live thinking updates to the TUI for active runs when the session reasoning level is `stream`, clears transient reasoning on final/error/abort, and removes only the callback dependency needed for Gateway/TUI delivery while preserving visibility and silent-turn gates.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: `/reasoning` help/completions still omit `stream`, a synthetic active-run `stream: "thinking"` event is ignored by `handleAgentEvent`, and the Gateway chat path does not provide the callback required by the current reasoning stream emitter.

## Solution Assessment

Yes, a narrow TUI-client repair plus the minimal stream-emission gate fix is the best direction. Changing general agent reasoning semantics or broad chat protocol behavior would be larger than needed for this issue.

## Repair Work Prompt

Fix the TUI reasoning stream gap for https://github.com/openclaw/openclaw/issues/42276. Current main accepts `reasoningLevel: "stream"`, but the TUI does not advertise it, the TUI event handler ignores `stream: "thinking"`, and the Gateway/TUI path may not emit thinking events because `subscribeEmbeddedPiSession` requires `onReasoningStream`. Keep the fix narrow: update TUI `/reasoning` completions, descriptions, help, and no-argument usage to include `stream`; render live `stream: "thinking"` agent events for active/known runs only when the current session reasoning level is `stream`; clear transient thinking output on final/error/abort/session switch; avoid duplicating thinking already shown through ordinary chat deltas and Ctrl+T show-thinking. If needed for Gateway/TUI delivery, decouple the WebSocket/agent-event broadcast from `onReasoningStream` while preserving `reasoningMode === "stream"`, visible-thinking, and silent-turn gates; do not introduce new protocol capabilities or broaden reasoning visibility. Add focused tests for command help/completions/usage, thinking-event render/ignore/cleanup behavior, and the no-callback emission path if touched. Align `docs/tools/thinking.md` and `docs/web/tui.md` with the implemented TUI behavior. Add a changelog entry only if the active changelog policy treats this TUI-visible behavior as user-facing, crediting @cryptopsy0 and @smallmj, and do not credit bots.
