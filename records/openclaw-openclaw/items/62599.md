## Summary

Keep open. Current main and the latest release mitigate plain fast status, but memory-reporting status paths still instantiate the CLI-local memory manager for custom memory slots and can probe/report state outside the live gateway runtime.

## What This Changes

The issue asks for `openclaw status` to avoid local memory-plugin startup side effects and to avoid reporting vector/memory state that disagrees with the live gateway.

## Best Possible Solution

Define one status contract for memory plugins: either report live gateway runtime truth, or make CLI-local memory status explicitly passive/indeterminate without probing plugin startup paths.

## Reproduction Assessment

Yes, source-level: custom memory slots still cause memory-reporting status to construct a status-purpose memory manager and probe/status it in the CLI process. I did not live-reproduce the reporter's macOS/sqlite-vec false-vector output against current main.

## Solution Assessment

No, current main is only a partial mitigation: plain fast status skips memory probing, but full/json-all memory reporting still uses a CLI-local plugin manager. The best fix needs a maintainer-approved status boundary for live gateway truth versus passive local inspection.
