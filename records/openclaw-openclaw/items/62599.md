## Summary

Keep open. Current main and v2026.6.1 avoid the original plain-status memory scan, but the memory-reporting status path still constructs the configured memory runtime in the CLI process and probes vector state locally, so the live-Gateway mismatch and plugin-startup side-effect concern remains meaningful.

## What This Changes

This issue asks for `openclaw status` memory reporting to avoid CLI-local memory plugin initialization and not report vector state that disagrees with the live Gateway.

## Best Possible Solution

Make memory-reporting status either use live Gateway memory/plugin health as the authoritative source or enforce a passive status contract that cannot start embedder, vector, backfill, or other heavyweight plugin work.

## Reproduction Assessment

No, not as a live reproduction of the reporter's macOS setup. Current source and tests do reproduce the remaining path: memory-reporting status constructs a local memory manager with `purpose: "status"` and probes vector state rather than relying on live Gateway truth.

## Solution Assessment

Unclear. A plugin-local passive guard or JSDoc-only contract is not enough by itself; the maintainable fix needs maintainer agreement on live Gateway status versus a strictly passive local status contract.
