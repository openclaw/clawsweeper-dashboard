## Summary

Keep open. Current main and v2026.6.10 narrow the default status blast radius, but memory-reporting status paths still materialize a CLI-local status-purpose memory manager for custom memory slots and can probe/report state from that local runtime rather than the live gateway.

## What This Changes

The issue reports that `openclaw status` initializes active memory plugins in the CLI process and can show false memory/vector state that disagrees with the live gateway.

## Best Possible Solution

Define and land one memory-status contract: full status should either read live gateway memory health or report an explicit passive/indeterminate state without running plugin startup/probe paths that can create false local vector failures.

## Reproduction Assessment

Yes, for the central source path: current main still routes full memory status for custom memory slots through a status-purpose local memory manager, probes it, and returns that manager's status. I did not live-reproduce the exact macOS/sqlite-vec false-vector output against current main.

## Solution Assessment

No complete solution is on main yet. The current fast-status mitigations are useful, but the best fix needs a maintainer-owned contract for live gateway truth versus passive local plugin inspection.
