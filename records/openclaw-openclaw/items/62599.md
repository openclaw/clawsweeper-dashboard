## Summary

Keep open. Current main mitigates the original plain-text status path, but detailed JSON status still loads the active memory plugin runtime in the CLI process and probes vector availability; PR #62643 remains open with closing syntax and should be resolved first.

## What This Changes

Report asks for `openclaw status` to avoid initializing memory plugins locally and to avoid reporting CLI-local vector state that disagrees with the live gateway.

## Best Possible Solution

Resolve the linked PR first: either land an explicit passive status contract if that is the desired plugin boundary, or replace it with a runtime/gateway-backed status change that prevents CLI-local false vector state.

## Reproduction Assessment

Yes for the remaining source-level path: `openclaw status --json --all` still enters the memory status resolver, obtains the active memory manager with `purpose: "status"`, and probes vector availability in the CLI process. The exact plain `openclaw status` reproduction from 2026.4.2 is now partially mitigated.

## Solution Assessment

Unclear. The fast-path skip is useful, but it does not settle whether detailed status should report live gateway state or rely on plugin authors honoring a passive CLI-local status contract.
