## Summary

Keep open. Current main mitigates plain `openclaw status` by skipping memory inspection, but memory-reporting status paths still construct and probe the configured memory plugin in the CLI process, so the live-gateway-versus-local-runtime mismatch remains unresolved.

## What This Changes

The issue requests that `openclaw status` avoid CLI-local memory plugin startup/probing side effects and not report memory/vector health from a separate local runtime.

## Best Possible Solution

Define a maintainer-approved memory status boundary, then make memory-reporting status either gateway-authoritative for live plugin health or explicitly passive/indeterminate without probing plugin startup paths.

## Reproduction Assessment

Yes for the remaining source path: memory-reporting status still constructs and probes a local status-purpose memory manager for custom memory slots. I did not live-reproduce the exact macOS/sqlite-vec false-vector output against current main.

## Solution Assessment

No complete solution is on main yet. The fast-status mitigation is useful, but the maintainable fix needs a maintainer-approved status boundary rather than relying on CLI-local plugin probing for live runtime truth.
