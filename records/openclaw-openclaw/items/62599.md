## Summary

Keep open. Current main partially mitigates the original fast `status` behavior, but memory-reporting status paths still instantiate and probe a CLI-local memory manager, so the live-gateway-vs-local-status boundary remains unresolved.

## What This Changes

The issue reports that `openclaw status` initializes the active memory plugin in the CLI process and can report memory/vector state that disagrees with the live gateway.

## Best Possible Solution

Make memory-reporting status either gateway-authoritative for live plugin health or explicitly passive/indeterminate without probing plugin startup paths.

## Reproduction Assessment

Yes from source for the central path: a custom memory slot still causes memory-reporting status paths to construct and probe a local status-purpose memory manager. I did not live-reproduce the exact macOS/sqlite-vec false-vector output against current main.

## Solution Assessment

No final solution is on main yet. The existing fast-status mitigation and custom-slot partial fix are useful, but the best remaining solution needs a maintainer decision on whether detailed memory status should come from the live gateway or from a passive local plugin contract.
