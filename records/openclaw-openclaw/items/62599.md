## Summary

Keep open. Current main mitigates plain fast status by skipping memory inspection, but memory-reporting status paths still materialize the selected memory plugin in the CLI process and probe local manager state, so the reported live-gateway mismatch remains unresolved.

## What This Changes

The issue asks for `openclaw status` to avoid initializing memory plugins locally and to avoid reporting vector state that disagrees with the live gateway.

## Best Possible Solution

Make memory-reporting status either gateway-authoritative for live plugin health or explicitly passive/indeterminate without probing plugin startup paths, then cover the chosen behavior in status `--all` and JSON tests.

## Reproduction Assessment

Yes for the source-level current-main path: a custom memory slot in memory-reporting status calls a CLI-local status-purpose manager and probes before reading status. I did not live-reproduce the exact macOS/sqlite-vec false-vector output.

## Solution Assessment

No final solution is on main yet. The current fast-status skip is a useful mitigation, but the maintainable fix needs a maintainer-approved status boundary for live gateway truth versus passive local status.
