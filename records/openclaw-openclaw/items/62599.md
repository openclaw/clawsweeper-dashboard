## Summary

Keep open. Current main mitigates the ordinary fast status path, but detailed/all status still reaches a CLI-local active memory runtime and probes vector state; the linked closing PR at https://github.com/openclaw/openclaw/pull/62643 is still open and conflicting, so closing this would be premature.

## What This Changes

The issue asks for `openclaw status` to avoid loading memory plugins in the CLI process and to avoid reporting memory/vector state that disagrees with the live gateway runtime.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/62643 first, either by landing a complete passive status contract with current-mode coverage or replacing it with a gateway-backed status fix coordinated with https://github.com/openclaw/openclaw/issues/57256.

## Reproduction Assessment

Yes for a source-level reproduction path: current main’s all/detailed status path asks the active memory plugin runtime for a `purpose: "status"` manager and probes vector state locally. The exact third-party macOS false-vector failure is supported by the issue logs but was not live-run here.

## Solution Assessment

Unclear. The linked PR may help if maintainers accept a passive CLI-local status contract, but a gateway-authoritative status path may be the better fix if detailed status is expected to reflect the launchd-managed runtime.
