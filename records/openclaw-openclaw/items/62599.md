## Summary

Keep open. Current main and the latest release reduced the plain `openclaw status` blast radius, but memory-reporting status paths still instantiate the configured memory runtime in the CLI process and probe local vector state instead of using live Gateway truth.

## What This Changes

The issue asks for `openclaw status` memory reporting to be read-only and sourced from the live Gateway rather than initializing the configured memory plugin in the CLI process.

## Best Possible Solution

Define one memory-status source of truth: keep plain status read-only, and make memory-reporting status paths either read the live Gateway state or use a documented passive plugin status contract with regression coverage for third-party memory plugins.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear current-main path for CLI-local memory runtime loading and probing, but there is no fresh live run against the reporter's custom plugin on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear: documenting `purpose: "status"` or relying on plugin-local guards is not enough by itself; the maintainable fix likely belongs at the core status/Gateway/plugin status boundary.
