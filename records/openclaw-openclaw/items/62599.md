## Summary

Keep open. Current main still has a detailed status path that loads the configured memory plugin in the CLI process and probes vector state, and the linked closing PR remains open rather than merged or superseded.

## What This Changes

The issue asks for `openclaw status` to avoid CLI-local memory plugin startup side effects and false memory/vector state when reporting status.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/62643 first, either by landing a complete passive status contract or replacing it with a gateway/runtime-backed status fix that prevents CLI-local false vector state.

## Reproduction Assessment

Yes, source-reproducible for the remaining path: current main’s `openclaw status --json --all` flow enters the memory resolver, obtains a CLI-local manager with `purpose: "status"`, and probes vector availability. The exact third-party macOS false-vector state is supported by issue logs but was not live-run here.

## Solution Assessment

Unclear. A passive `purpose: "status"` contract may be acceptable, but a gateway-authoritative status path may be the more accurate fix if status is expected to reflect the live launchd-managed runtime.
