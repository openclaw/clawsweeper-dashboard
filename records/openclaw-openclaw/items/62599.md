## Summary

Keep open. Current main mitigates plain fast status by skipping memory inspection, but opt-in memory-reporting status paths still instantiate the selected memory plugin in the CLI process and probe local manager state, so the live-gateway mismatch remains unresolved.

## What This Changes

The issue asks for `openclaw status` to avoid locally loading memory plugins and to stop reporting memory/vector state that can disagree with the live gateway.

## Best Possible Solution

Define a status contract where plain status remains read-only and any memory-reporting status either queries live gateway truth or uses a clearly passive plugin status API with regression coverage for third-party memory plugins.

## Reproduction Assessment

Yes at source level: current main still has memory-reporting status paths that load the active memory plugin runtime locally with `purpose: "status"`. A live third-party plugin/vector mismatch was not rerun in this review.

## Solution Assessment

Unclear: a plugin-local passive status guard is a useful mitigation, but the maintainable core fix needs a maintainer decision on live gateway truth versus local passive status.
