## Summary

Keep open. Current main and the latest release still have memory-inspecting status paths that instantiate the configured memory runtime locally with `purpose: "status"` and probe vector state, while the linked fixing PR remains open rather than merged.

## What This Changes

The issue reports that `openclaw status` can initialize a configured memory plugin inside the CLI process and report memory/vector state that disagrees with the live gateway.

## Best Possible Solution

Make memory status inspection either passive CLI-local or gateway-authoritative by contract, then land the accepted path with focused status/memory regression coverage and external-plugin proof.

## Reproduction Assessment

Yes for a source-level path: current main still has detailed status paths that ask the configured memory runtime for `purpose: "status"` and probe vector state in the CLI process. The exact third-party macOS false-vector/OOM result still needs live reproduction.

## Solution Assessment

Unclear. The linked documentation/test PR may help if maintainers accept a passive status contract, but a gateway-authoritative status path may be the better fit if status is expected to reflect the live launchd-managed runtime.
