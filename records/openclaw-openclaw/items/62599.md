## Summary

Keep open. Current main and v2026.6.6 avoid the original default plain-status memory scan, but memory-reporting paths still construct and probe the configured memory runtime in the CLI process, so the live-Gateway mismatch and plugin-startup side-effect concern remains unresolved.

## What This Changes

The issue asks for `openclaw status` to avoid initializing memory plugins locally and to avoid reporting false memory/vector state that disagrees with the live gateway.

## Best Possible Solution

Make memory-reporting status either use live gateway memory/plugin health as the authoritative source or enforce a passive local status contract that cannot start embedder, vector, backfill, or other heavyweight plugin work; coordinate the narrower live-status fallback with https://github.com/openclaw/openclaw/issues/57256.

## Reproduction Assessment

No live high-confidence reproduction was run in this review. Current source and tests do reproduce the remaining path: memory-reporting status constructs a local memory manager with `purpose: "status"` and probes vector state.

## Solution Assessment

Unclear. Plugin-local passive guards or documentation help, but the maintainable fix depends on a maintainer decision about live Gateway authority versus strictly passive local status semantics.
