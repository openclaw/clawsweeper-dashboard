## Summary

Keep open. Current main partially mitigates the ordinary text status path, but detailed status still asks the active memory runtime for a CLI-local `purpose: "status"` manager and probes vector state; the linked closing PR remains open and conflicting rather than merged or superseded.

## What This Changes

The issue reports that `openclaw status` initializes active memory plugins in the CLI process and can show CLI-local memory/vector state that disagrees with the live gateway.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/62643 first, either by landing a complete passive status contract or replacing it with a gateway-backed status fix plus regression coverage.

## Reproduction Assessment

Yes for a source-level reproduction path: current main’s detailed status path asks the active memory plugin runtime for a `purpose: "status"` manager and probes vector state locally. The exact third-party macOS false-vector failure is supported by the issue logs but was not live-run here.

## Solution Assessment

Unclear. The linked documentation/test PR may be enough if maintainers accept a passive CLI-local status contract, but a gateway-authoritative status path may be safer if detailed status is expected to reflect the live launchd-managed runtime.
