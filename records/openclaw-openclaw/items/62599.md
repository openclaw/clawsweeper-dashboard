## Summary

Keep open. Current main mitigates the exact plain-text `openclaw status` fast path, but the detailed memory-status helper still obtains the active memory manager in the CLI process and probes vector availability; additionally, open PR #62643 still uses closing syntax for this issue and should be reviewed or closed before this issue is resolved.

## What This Changes

The issue asks for `openclaw status` to avoid local memory-plugin startup side effects and to avoid reporting CLI-local false memory/vector state instead of live gateway state.

## Best Possible Solution

Resolve the linked PR first: either land a clear passive `purpose: "status"` contract if that is the intended plugin boundary, or replace it with a narrow runtime change that prevents detailed status from surfacing CLI-local false vector state and adds focused status/memory coverage.

## Reproduction Assessment

Yes for the source-level detailed path: current main still shows `status --json --all` entering the memory status resolver, obtaining the active memory manager with `purpose: "status"`, and probing vector availability in the CLI process. The exact plain `openclaw status` reproduction from 2026.4.2 is now partially mitigated by the fast path.

## Solution Assessment

Unclear. The current fast-path skip is useful, but it is not a complete answer for detailed memory status unless maintainers accept CLI-local plugin status as the documented contract.
