## Summary

Keep open. Current main partially mitigates plain status, but the detailed JSON status path still obtains a CLI-local memory manager with `purpose: "status"`, probes vector state, and can load the configured memory plugin locally; the linked closing PR is still open rather than merged or superseded.

## What This Changes

The issue asks `openclaw status` to avoid CLI-local memory plugin initialization and false memory/vector state that disagrees with the live gateway.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/62643 first, either by landing a complete passive status contract or replacing it with a gateway/runtime-backed status fix plus regression coverage.

## Reproduction Assessment

Yes, source-reproducible for the remaining path: current main’s `status --json --all` flow asks the active memory plugin runtime for a `purpose: "status"` manager and probes vector state. The exact third-party macOS false-vector failure is supported by issue logs but was not live-run here.

## Solution Assessment

Unclear. A passive `purpose: "status"` contract may be acceptable, but a gateway-authoritative status path may be the safer fix if detailed status is expected to reflect the live launchd-managed runtime.
