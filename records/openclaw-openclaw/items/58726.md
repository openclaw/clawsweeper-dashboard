## Summary

Keep open. Current `main` still does not satisfy the reported hard-abort behavior for the same-tool/same-args `generic_repeat` path: it remains warning-only, while blocking is limited to critical detectors such as known polling loops, ping-pong no-progress loops, unknown-tool repeats, and the global no-progress breaker. The related open PRs are implementation candidates, not merged fixes, so this issue should remain the canonical tracker until maintainers land one direction.

## What This Changes

The issue asks OpenClaw to hard-abort an agent response after a configurable number of repeated identical tool calls, especially `session_status`, instead of only warning the agent.

## Best Possible Solution

Land one canonical loop-detector fix in the shared before-tool-call path: make enabled `generic_repeat` loops escalate to a critical block at the configured threshold, then update focused tests, docs, and changelog. Treat default-on behavior, lower default thresholds, and whole-turn termination as separate maintainer policy choices unless they are deliberately included in the canonical PR.

## Reproduction Assessment

Yes. The issue provides live `session_status` warning logs, and current source/tests give a focused reproduction: with loop detection enabled, repeat a non-polling same-tool/same-args call and observe `generic_repeat` warnings rather than a critical block until the separate global no-progress breaker applies.

## Solution Assessment

Unclear for the full requested behavior, but the narrow best first fix is clear. Escalating `generic_repeat` to critical at `criticalThreshold` is the maintainable detector-level fix; killing the whole agent turn or changing defaults should be decided explicitly by maintainers.
