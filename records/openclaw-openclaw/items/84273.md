## Summary

Keep open: the PR targets a real heartbeat duplicate-delivery bug, but the current diff can suppress heartbeat fallback text after any successful messaging-tool send during a heartbeat run, including cross-target sends, and the supplied proof is still synthetic for a visible Telegram delivery change.

## What This Changes

The PR adds internal reply-payload metadata for successful generic message-tool delivery, uses it to suppress heartbeat fallback text, and adds heartbeat/runReplyAgent regression coverage.

## Best Possible Solution

Keep the metadata-based suppression direction, but only mark/suppress heartbeat fallback after a successful message-tool send that matches the heartbeat delivery route, with regressions for same-route and cross-route sends plus live Telegram or Mantis proof.

## Reproduction Assessment

Yes at source level: current main can send fallback heartbeat text after a generic message tool delivery, and the PR diff shows a separate source-level path where off-route message-tool success can suppress fallback. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No: the metadata handoff is the right shape, but the current implementation is too broad because it does not prove the successful message-tool send targeted the heartbeat route before suppressing fallback text.
