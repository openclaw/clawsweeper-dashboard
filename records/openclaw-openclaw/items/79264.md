## What This Changes

The issue reports a Windows/npm release regression where `post-attach.update-sentinel` now appears as about 9 seconds of synchronous work around each `model_call` after upgrading to OpenClaw 2026.5.7.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/76038 as the canonical tracking point for the remaining Feishu/session prep and stuck recovery work, and treat this release log as additional evidence there.

## Reproduction Assessment

No high-confidence reproduction for the exact claim that update-sentinel runs per `model_call`; source inspection shows it is a startup/post-attach phase and the diagnostic can repeat completed phases in `recentPhases`. The broader Feishu/session slowness signature is already preserved in the canonical open issue.

## Solution Assessment

Yes, duplicate closure is the best cleanup path. A new config flag or update-sentinel-only change would not address the active `channels.feishu.start-account`/model-prewarm and stuck-session symptoms already centralized in the canonical issue.
