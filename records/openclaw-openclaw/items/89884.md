Keep open: this PR is still the active fix candidate for a source-proven Voice Call session-continuity bug, and current main plus the latest release still emit raw generated `voice:*` keys. ## What This Changes

The PR changes the Voice Call session-key resolver to emit canonical `agent:<agentId>:voice:*` keys for generated per-phone and per-call sessions, while preserving explicit session-key overrides and updating focused tests.

## Best Possible Solution

Land the focused resolver/test fix once maintainers accept the generated-session-key cutover or add explicit upgrade proof for raw generated voice state that must be preserved.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.6` still generate raw `voice:*` keys while the migration canonicalizes raw session-store keys to `agent:<agentId>:...`. I did not run a live carrier or Gateway restart reproduction in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the cutover. The resolver is the right owner boundary because inbound, outbound, response generation, and realtime consult paths all consume its generated key; adding later wrapping would leave persisted call records and embedded-agent lookups split.
