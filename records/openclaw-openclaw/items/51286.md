## Summary

Keep open. The PR is aimed at a real current-main bug in `openclaw sandbox explain`, but the branch is incomplete: wrapping `telegram:slash:<id>` into an agent-scoped key is necessary, yet channel inference still depends on the active plugin channel registry and can still return unknown in the CLI path. The later test assertion for `elevated.channel === "telegram"` exposes that remaining gap, and the open companion PR #51584 covers the other half of the same root cause.

## What This Changes

The PR changes `sandbox explain` session-key normalization so non-agent-scoped colon keys are wrapped as `agent:<id>:<key>` and adds regression tests for raw Telegram slash keys and already agent-scoped keys.

## Best Possible Solution

Land one coherent fix for #51245 that combines canonicalizing raw colon session keys with a static bundled-channel fallback for `sandbox explain` channel inference, then keep the regression test asserting `sessionKey`, `elevated.channel`, and `elevated.allowedByConfig`. The companion #51584 should be folded into this PR or one replacement PR so the issue closes once, with the narrow CLI diagnostic behavior covered.

## Reproduction Assessment

Yes. The static path is high-confidence: current main leaves `telegram:slash:200` unscoped, `parseAgentSessionKey` rejects it, and `sandbox explain` therefore emits no channel and `allowedByConfig: false`; the PR’s added regression test is the right shape to reproduce the intended outcome.

## Solution Assessment

No. The PR fixes the first half of the problem, but the best fix also changes channel inference to fall back to the bundled channel catalog when the CLI has no active plugin channel registry.
