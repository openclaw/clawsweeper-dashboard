## Summary

Keep open: the PR fixes one real part of the Telegram slash-session diagnostic bug, but it is not merge-ready because the wrapped key still depends on the runtime channel registry and the branch lacks real after-fix proof.

## What This Changes

This PR changes `sandbox explain` to wrap non-agent-scoped colon session keys in `agent:<id>:` form and adds regression tests for raw Telegram slash keys and already agent-scoped keys.

## Best Possible Solution

Land one reconciled diagnostic fix that canonicalizes raw channel-prefixed session keys with the existing session-store helper, resolves bundled channel ids without loaded plugins, and proves the Telegram allowFrom output after the patch.

## Reproduction Assessment

Yes, source-level: current main returns raw colon session keys unchanged, `parseAgentSessionKey` rejects bare `telegram:slash:<id>` keys, and unresolved channel identity keeps `allowedByConfig` false. I did not run live Telegram or CLI proof in this read-only review.

## Solution Assessment

No as a standalone PR. Wrapping raw keys is necessary, but the best fix also needs bundled-channel fallback when plugins are unloaded and should preserve existing raw session-key canonicalization semantics.
