## Summary

Keep this PR open: the key-normalization change is useful, but it only covers half of the reported `sandbox explain` failure and still relies on registry-only channel inference in a CLI path that does not load plugins; it also still lacks after-fix real behavior proof.

## What This Changes

The PR wraps non-agent-scoped colon session keys in `sandbox explain` as agent-scoped keys and adds tests for Telegram slash-session diagnostics.

## Best Possible Solution

Land one reconciled fix that normalizes bare channel session keys, falls back to bundled channel ids when plugins are unloaded, and includes no-registry regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main leaves `telegram:slash:<id>` unscoped, and this PR reaches an agent-scoped key but still relies on registry-only channel inference in a CLI path that does not load plugins. I did not run a live Telegram gateway flow in this read-only review.

## Solution Assessment

No. Key normalization is necessary, but the maintainable fix also needs a bundled-channel fallback such as `normalizeChatChannelId` or an explicit plugin-loading strategy with a no-registry regression test.
