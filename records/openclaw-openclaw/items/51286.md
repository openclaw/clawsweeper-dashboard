## Summary

Keep open. The branch addresses one real part of the reported sandbox explain failure and also incorporated the Greptile test feedback, but the current CLI path still does not guarantee `channel: telegram` because `sandbox explain` does not preload channel plugins and the downstream channel inference remains registry-only.

## What This Changes

The PR changes `sandboxExplainCommand` to wrap non-agent-scoped colon session keys with `agent:<id>:` and adds sandbox explain tests for non-agent-scoped and already agent-scoped session keys.

## Best Possible Solution

Land a combined fix that keeps the non-agent-scoped key normalization, makes sandbox explain channel inference independent of an already-loaded runtime channel registry for bundled channels, and covers the no-registry CLI case in tests.

## Reproduction Assessment

Yes. The high-confidence path is static: after this PR wraps `telegram:slash:200` to `agent:main:telegram:slash:200`, `resolveActiveChannel` still reaches `normalizeAnyChannelId("telegram")`, which returns null when the sandbox CLI has no active plugin registry.

## Solution Assessment

No. Key normalization is necessary, but the narrow maintainable fix also needs a static bundled-channel fallback such as `normalizeChatChannelId`/`normalizeChannelId`, preferably without making the read-only sandbox diagnostic load channel plugins.
