## Summary

Keep open and do not merge as-is. The PR covers the bare session-key normalization half and incorporates the prior review request for downstream assertions, but the fixed path still depends on a populated runtime channel plugin registry; the sandbox CLI calls this command directly, so Telegram can still resolve to `(unknown)` unless the static bundled-channel fallback from the related PR is also applied.

## What This Changes

The PR changes `sandbox explain` session-key normalization to wrap non-agent-scoped colon keys such as `telegram:slash:<id>` into agent-scoped keys, preserves already scoped keys, and adds JSON regression tests for the bare and scoped cases.

## Best Possible Solution

Land one consolidated sandbox-explain fix that normalizes bare channel-prefixed session keys, applies bundled-channel fallback for both stored metadata and inferred session-key channels, preserves `global`, plain main keys, and already agent-scoped keys, and carries focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes. With an empty active channel registry, `--session telegram:slash:200` is normalized to an agent-scoped key by this PR, but inference still calls `normalizeAnyChannelId("telegram")`, which returns null and leaves `elevated.channel` undefined.

## Solution Assessment

No. Normalizing non-agent-scoped colon keys is necessary, but it is not the complete fix without the static bundled-channel fallback covered by the related PR; combining the two approaches is the narrower maintainable path.
