## Summary

Keep open: the linked bug is still source-reproducible on current main, and this PR only fixes the raw-session-key half of the path. After the branch wraps `telegram:slash:<id>`, channel inference still depends on registered channel plugins, so the `sandbox explain` CLI can still report an unknown channel unless this is reconciled with the static-channel fallback work in https://github.com/openclaw/openclaw/pull/51584.

## What This Changes

The PR wraps non-agent-scoped colon session keys in `sandbox explain` as agent-scoped keys and adds tests for raw Telegram slash keys plus already agent-scoped keys.

## Best Possible Solution

Land one reconciled fix that normalizes raw channel-prefixed session keys, falls back to bundled channel ids when plugins are unloaded, and proves the Telegram elevated diagnostic path with regression coverage plus redacted real behavior output.

## Reproduction Assessment

Yes, source-level. Current main leaves `telegram:slash:<id>` unscoped, `parseAgentSessionKey` only accepts `agent:<id>:...`, and unresolved channel identity keeps `allowedByConfig` false.

## Solution Assessment

No. Key wrapping is necessary, but this PR is not sufficient because it still relies on `normalizeAnyChannelId` in a CLI path that normally has no loaded channel plugin registry; the safer fix adds bundled-channel fallback or explicit plugin loading with no-registry coverage.
