## Summary

Keep open. Current main still has only partial/global suppression: `messages.suppressToolErrors` exists, but mutating tool failures such as `message`, `write`, and `exec` still take the warning branch before that setting is checked, and the chat delivery path keeps `isError` tool payloads deliverable.

## What This Changes

The issue requests a per-channel or broader policy that prevents tool failure output from reaching chat surfaces while still returning those failures to the agent.

## Best Possible Solution

Define one documented tool-error visibility policy for chat delivery, with a global default and any intended per-channel override, then apply it before user-facing payload delivery while preserving the same tool failures in agent context. Add focused coverage for `message`, `write`, and `exec` failures plus streaming chat dispatch behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is already encoded in `payloads.errors.test.ts`: with `messages.suppressToolErrors: true`, a mutating `write` failure still produces a user-facing tool-error payload, while the run-level `suppressToolErrorWarnings` override suppresses it.

## Solution Assessment

Unclear. Reordering the existing `messages.suppressToolErrors` check would be the narrowest fix for the global boolean gap, but the issue’s per-channel and enum alternatives need maintainer policy choice before adding a larger config surface.
