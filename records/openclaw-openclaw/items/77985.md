## Summary

Keep open: the reported performance problem is plausible and the PR includes real runtime evidence, but the patch should not merge as-is because it adds a hidden wall-clock cache for parameter-scoped plugin metadata.

## What This Changes

The PR adds a process-global 5-second TTL cache to `loadPluginMetadataSnapshot()` in `src/plugins/plugin-metadata-snapshot.ts`.

## Best Possible Solution

Keep the performance fix direction, but implement it through explicit compatible snapshot reuse or a validated invalidation strategy that preserves next-read freshness for changed plugin inventory, config, workspace, and installed index state.

## Reproduction Assessment

Yes for the patch risk: source inspection shows the proposed cache ignores the snapshot inputs while current types and docs make those inputs part of the metadata scope. I did not live-reproduce the original Control UI freeze on current main.

## Solution Assessment

No. A hidden 5-second wall-clock cache is not the narrowest maintainable fix; the safer direction is to pass or reuse an explicit compatible `PluginMetadataSnapshot` through the hot path and cover freshness with tests.
