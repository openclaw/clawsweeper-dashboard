## Summary

Keep open: the PR targets a real performance problem and includes useful live proof, but the current patch is not mergeable because its process-global TTL cache can return plugin metadata for the wrong config, workspace, env, or installed index, and it also lacks the required changelog entry.

## What This Changes

The PR adds a process-global five-second TTL cache around `loadPluginMetadataSnapshot()` in `src/plugins/plugin-metadata-snapshot.ts`.

## Best Possible Solution

Keep the performance fix direction, but route reuse through explicit compatible `PluginMetadataSnapshot` or `PluginLookUpTable` ownership, or use a fully scoped compatibility-keyed cache with invalidation tests and a changelog entry.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the new cache return path runs before metadata-scope compatibility checks, while current tests and docs require config/workspace/env/index-sensitive reuse. I did not live-reproduce the original UI freeze on current main, but the PR body includes after-fix live output from the reporter's setup.

## Solution Assessment

No. The performance goal is valid, but a hidden process-global wall-clock cache is not the narrowest maintainable fix; explicit compatible snapshot reuse or a fully keyed invalidation strategy is safer.
