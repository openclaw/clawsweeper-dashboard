## Summary

This PR should stay open because the performance problem is credible and the real-behavior proof is sufficient, but the proposed unkeyed process-global TTL cache violates the plugin metadata freshness contract and can return snapshots for the wrong caller scope.

## What This Changes

The PR adds a process-global 5-second TTL cache around `loadPluginMetadataSnapshot()` in `src/plugins/plugin-metadata-snapshot.ts`.

## Best Possible Solution

Keep the performance fix direction, but implement reuse through explicit compatible snapshots or a keyed invalidation strategy that preserves next-read freshness, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes for the PR regression risk: source inspection shows the cache ignores the metadata load inputs while current code and docs make those inputs part of the snapshot scope. I did not live-reproduce the original Control UI freeze on current main, but the PR body includes after-fix live output from the reported workflow.

## Solution Assessment

No. A hidden process-global wall-clock cache is not the narrowest maintainable fix; the safer path is explicit compatible snapshot reuse or a cache keyed to the existing freshness inputs with invalidation tests.
