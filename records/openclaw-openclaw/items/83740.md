## Summary

Keep open: current main and v2026.5.18 still wrap every `FsSafeError` from `writeImportedSourcePage` with a symlink-specific message, while the fs-safe contract and write path expose non-symlink error codes that can reach that catch.

## What This Changes

The issue asks memory-wiki imported-source writes to report the actual `FsSafeError` cause instead of always saying the page write went through a symlink.

## Best Possible Solution

Keep this issue open and land a narrow memory-wiki fix that preserves symlink safety wording for actual symlink/path-alias cases while surfacing the original code or message for other `FsSafeError` causes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: any non-symlink `FsSafeError` thrown by `vault.write` is currently rewrapped as a symlink failure, and fs-safe exposes non-symlink write errors.

## Solution Assessment

Yes. The maintainable fix is a small catch-path change plus regression coverage for both the existing symlink rejection and a non-symlink `FsSafeError`; no new config or product policy is needed.
