## Best Possible Solution

Land the ordering, playback, and transcript-flow fixes only after direct provider replies remain compatible by default or the stricter forced-consult behavior is an explicit maintained policy, with Web/Android spacing fixed and real proof covering Android playback plus consult routing.

## Reproduction Assessment

Partially yes: source inspection shows current main sends finalized relay transcripts directly back to the realtime provider, while the PR forces an OpenClaw consult fallback; the supplied screenshots also reproduce visible transcript/layout issues. ## Solution Assessment

No: the stabilization pieces are directionally right, but unconditional forced consults are not the narrowest upgrade-safe fix and the transcript merge helper has a visible spacing bug. A safer solution preserves direct replies by default or gates forced consults behind an explicit policy, then fixes spacing in both Web and Android helpers.

## Review Findings

Overall correctness: patch is incorrect

- **[P1] Preserve direct replies before forcing consults:** `src/gateway/talk-realtime-relay.ts:484`
  - body: Replacing the direct `sendUserMessage(text.trim())` path with an unconditional forced consult means any finalized gateway-relay transcript where the provider does not call the consult tool within 200 ms becomes an OpenClaw chat run.   - confidence: 0.9

The attached after screenshots show `you.Alright` on Web and `you.On` on Android; update this predicate here and in the Android copy so punctuation-ended transcript chunks still separate cleanly.
  - confidence: 0.88
