## Summary

Keep open: the Android/UI stabilization work is plausible, but the forced-consult fallback is a merge-blocking routing change and the PR still lacks attached real Android/browser behavior proof.

## What This Changes

The PR serializes Android Gateway realtime frame handling, queues Android PCM playback, accumulates and splits Android/Web Talk transcript turns, moves Web voice turns into chat, and strengthens realtime consult prompting/fallback behavior.

## Best Possible Solution

Land the frame-ordering, transcript accumulation, chat-flow, and PCM queue fixes after proof, but remove, split, or explicitly policy-gate the forced-consult fallback with tests for both direct replies and consult-required turns.

## Reproduction Assessment

No full live reproduction. Source inspection confirms the current-main frame-ordering, transcript delta, and composer-scoped Web transcript paths, but audible playback stability still needs Android device or emulator proof.

## Solution Assessment

No. The stabilization pieces are targeted, but forcing consults for every final realtime user transcript is not the narrowest maintainable fix without a maintainer-approved policy gate or split follow-up.
