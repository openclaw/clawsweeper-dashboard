## Summary

Keep open. Current main still limits BlueBubbles Contacts enrichment to group participants, 1:1 sender labels still come from BlueBubbles metadata or the raw sender id, and the resolver/tests/docs remain phone/group-scoped; the issue discussion also has maintainer signal for a PR.

## What This Changes

Request to extend BlueBubbles local Contacts name enrichment from group participants to 1:1 inbound sender labels, including phone and email handles with a backwards-compatible broader config flag.

## Best Possible Solution

Implement the change inside the BlueBubbles plugin by extending the existing Contacts resolver/cache to support DM sender enrichment and email handles after access/mention gating, preserving the legacy group flag through a config shim and updating tests, docs, and generated config metadata as needed.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: on current main, DM formatting uses message.senderName or user:<senderId>, the Contacts resolver is only called under the group branch, and email handles are skipped by the resolver/tests.

## Solution Assessment

Yes. The proposed owner-local BlueBubbles resolver extension is the narrowest maintainable solution because the existing sqlite/cache machinery already lives in that plugin and no core channel or plugin SDK seam appears necessary.
