## Summary

Keep open. Current main and v2026.6.5 still have the reported placeholder-body behavior, but log bodies are intentionally documented and tested behind a privacy/content-capture gate; changing that default or trusting logger trace context needs maintainer security/product direction.

## What This Changes

The issue requests OTLP log export to put the rendered log message in `LogRecord.body` and attach trace/span ids when diagnostic trace context is active.

## Best Possible Solution

Maintainers should decide the privacy/trust boundary, then either reaffirm the current `captureContent: true` policy or add a narrow documented log-body opt-in plus trusted trace-context propagation tests.

## Reproduction Assessment

Yes for source-level reproduction: current source and tests show default OTLP log bodies are `"log"`, granular capture does not enable bodies, and untrusted trace context is not attached. I did not run a live OTLP collector reproduction in this read-only review.

## Solution Assessment

Unclear as filed. Exporting every rendered log body by default conflicts with the documented privacy model; the safer solution needs maintainer approval for either the existing broad capture setting, a narrower log-body opt-in, or a trusted logger-trace path.
