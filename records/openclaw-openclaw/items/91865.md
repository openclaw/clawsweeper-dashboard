## Summary

Keep open. Current main and latest stable still emit placeholder OTLP log bodies by default, but that behavior is documented and tested as a privacy default; changing the default body policy or broader trace correlation needs maintainer security/product direction rather than cleanup closure.

## What This Changes

The issue asks for `diagnostics.otel.logs` OTLP log records to export rendered log messages and trace/span correlation instead of placeholder bodies and empty IDs.

## Best Possible Solution

Keep the safe default, then either reaffirm that operators must use boolean `captureContent: true` for OTLP log bodies or approve a narrower documented log-body opt-in plus trace-context work through the existing diagnostics/plugin-hook contracts.

## Reproduction Assessment

Yes for the reported default-body behavior: source and tests show `LogRecord.body` becomes `"log"` when `diagnostics.otel.logs` is true and broad content capture is not enabled. The empty trace/span part is source-explainable for log records without trusted diagnostic trace context; I did not run a live collector reproduction.

## Solution Assessment

No as filed: making `diagnostics.otel.logs` export every rendered body by default conflicts with the documented privacy model. The safer solution is maintainer-approved policy, likely using the existing broad opt-in or a narrower log-body opt-in with redaction and trace-context tests.
