## Summary

Keep open. Current `main` and `v2026.6.5` still export OTLP log bodies as `"log"` unless broad content capture is enabled, but that behavior is explicitly documented and tested as a privacy default; changing it or adding stronger trace correlation needs maintainer product/security direction rather than cleanup closure.

## What This Changes

Request to make diagnostics-otel OTLP log records include rendered log messages and usable trace/span context instead of the placeholder body and empty correlation ids.

## Best Possible Solution

Maintainers should either reaffirm the existing `captureContent: true` log-body policy in the issue or approve a narrow, documented log-body opt-in plus focused tests for redaction and trusted trace context propagation.

## Reproduction Assessment

Yes for the reported default-body behavior: current source and tests show OTLP logs emit `body: "log"` unless boolean `captureContent: true` is enabled. The empty trace-id part is source-explainable when no trusted diagnostic trace is present, but I did not run a live collector reproduction.

## Solution Assessment

Unclear as filed. Exporting every rendered log body by default conflicts with the documented privacy model; the safer path is a maintainer-approved policy decision, likely using the existing broad capture setting or a narrower log-body opt-in with redaction and trace-context tests.
