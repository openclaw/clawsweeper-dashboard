## What This Changes

The issue requests that diagnostics OTEL export rendered log bodies and trace/span correlation instead of placeholder OTLP log bodies with empty trace fields.

## Best Possible Solution

Keep the privacy-preserving log-body default, use `diagnostics.otel.captureContent: true` when OTLP log bodies are approved, and continue the missing trace/run/message correlation work in https://github.com/openclaw/openclaw/issues/50291.

## Reproduction Assessment

Yes. Current source and tests reproduce the reported `body: "log"` behavior for `diagnostics.otel.logs: true` without broad content capture; the trace/span gap is source-plausible but belongs to the linked trace-context contract issue.

## Solution Assessment

No. Changing OTLP logs to export raw bodies by default is not the best solution because it conflicts with the documented privacy model; the maintainable path is explicit `captureContent: true` plus the canonical trace-context work.
