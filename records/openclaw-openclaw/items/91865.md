## Summary

Keep open. Current `main` and `v2026.6.5` still use the placeholder OTLP log body unless broad content capture is enabled, and normal logger-originated records still flow through an untrusted diagnostic-event path that the exporter will not use for trace attachment; the body behavior is documented as a privacy default, so changing it needs maintainer security/product direction rather than cleanup closure.

## What This Changes

The issue requests OTLP log records to export the rendered log message in `LogRecord.body` and populate trace/span correlation when active trace context exists.

## Best Possible Solution

Maintainers should keep the privacy-preserving body default unless they approve a documented opt-in, and separately decide whether logger-originated `log.record` events should carry trusted trace metadata into diagnostics-otel.

## Reproduction Assessment

Yes at source level: current exporter code and tests show placeholder log bodies unless broad content capture is enabled, and normal logger-originated records are emitted through an untrusted metadata path that the exporter refuses to use for trace context. I did not run a live OTLP collector reproduction.

## Solution Assessment

Unclear as filed. The trace-context part may have a narrow trusted-logger repair, but exporting rendered log bodies under `diagnostics.otel.logs: true` conflicts with the documented privacy model and needs maintainer security/product approval.
