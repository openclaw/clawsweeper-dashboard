## Summary

Keep open. Current main and v2026.6.5 still emit OTLP log bodies as "log" by default, but that is now documented and tested as a privacy default; changing log-body export or broadening trace correlation needs maintainer product/security direction rather than cleanup closure.

## What This Changes

Request that diagnostics-otel export rendered OTLP log bodies and populate trace/span correlation for exported logs instead of emitting placeholder bodies and empty context.

## Best Possible Solution

Maintainers should preserve safe defaults and either reaffirm the current captureContent policy or approve a narrow log-body opt-in plus trusted trace-context propagation with redaction and OTLP smoke coverage.

## Reproduction Assessment

Yes for the default body behavior: current source and tests show OTLP logs emit body "log" unless boolean captureContent true is enabled. The empty trace-id part is source-explainable when a log event lacks trusted diagnostic trace context, but I did not run a live collector reproduction.

## Solution Assessment

Unclear as filed. Exporting every rendered log body under diagnostics.otel.logs conflicts with the current privacy model; the maintainable path is maintainer-approved opt-in behavior plus trace-context work rather than a silent default change.
