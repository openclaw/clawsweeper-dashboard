## Summary

Keep open. Current main and the latest stable release still match the report, but the placeholder OTLP log body is documented and tested as a privacy default, so changing it needs maintainer product/security direction rather than conservative cleanup closure.

## What This Changes

The issue asks diagnostics-otel OTLP log export to populate rendered log bodies and trace/span correlation instead of placeholder bodies and empty IDs.

## Best Possible Solution

Maintainers should either reaffirm that operators must use boolean `captureContent: true` for OTLP log bodies, or approve a narrower documented log-body opt-in plus focused tests for redaction and trusted trace correlation.

## Reproduction Assessment

Yes. Source and tests show that default OTLP log export emits `body: "log"`, and trace/span ids are only attached from trusted diagnostic trace metadata; I did not run a live collector repro.

## Solution Assessment

No as filed. Exporting every rendered body by default conflicts with the current documented privacy model, so the safer path is a maintainer-approved policy surface before implementation.
