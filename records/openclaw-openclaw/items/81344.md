## Summary

Keep this PR open: the direct WebChat send fix is well-scoped and backed by proof/CI, but the queued-message steering path still trims queued text before sending, leaving a user-visible whitespace-loss path.

## What This Changes

The PR preserves raw WebChat message text through direct send, queue storage, and controller send paths while using trimmed text for blank guards and command detection, with focused regression tests.

## Best Possible Solution

Extend the same raw-text plus trimmed-guard pattern to queued-message steering and add a focused regression test for steering an indented queued diagram.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction because `handleSendChat` and `sendChatMessage` trim the message before display/RPC, and the remaining PR gap is also source-reproducible through `steerQueuedChatMessage` trimming `item.text`.

## Solution Assessment

No. The direct send/controller changes are the right approach, but the fix is incomplete until queued-message steering stops trimming the submitted text and has regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve whitespace when steering queued messages:** `ui/src/ui/app-chat.ts:197`
  - body: This stores raw text in the queue, but `steerQueuedChatMessage` still does `const message = item.text.trim()` before calling `sendSteerChatMessage`. A user who queues an indented diagram while a run is busy and then steers that queued item into the active run will still send a stripped payload, so the whitespace bug survives one WebChat send path. Please split the blank guard from the submitted text there and add a queued-steer regression test.
  - confidence: 0.89
