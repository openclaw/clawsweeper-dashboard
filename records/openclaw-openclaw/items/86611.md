## Summary

Keep open: the patch is in the right local repair boundary, but it still misses the current edit tool argument shape. OpenClaw validates edit calls as `edits: [{ oldText, newText }]`, and the PR’s array-value path still depends on `JSON.parse`, so smart-quoted nested edit objects remain unrepaired.

## What This Changes

The PR adds a smart-quote-aware fallback in embedded-agent-runner malformed tool-call argument repair and focused stream-wrapper regression tests.

## Best Possible Solution

Keep the repair local to the embedded-runner wrapper, but extend it to recover smart-quoted nested edit arrays and prove that repaired args reach the partial, streamed, end-message, and final tool-call surfaces.

## Reproduction Assessment

Yes from source: a current-schema edit payload like `{“path”:“x”,“edits”:[{“oldText”:“a”,“newText”:“b”}]}` reaches the PR’s non-string value path and fails when the smart-quoted array is handed to `JSON.parse`. I did not run tests because this review is read-only.

## Solution Assessment

No; the local wrapper is the right boundary, but the implementation needs to handle the current nested edit argument shape before it is the best fix.
