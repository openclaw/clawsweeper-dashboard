## Summary

Keep open: current main still uses Discord command-deploy cache keys that are not scoped by Discord application, and the open linked PR that closes this issue has not merged yet.

## What This Changes

The issue reports that secondary Discord bot accounts in a multi-account gateway can receive messages but do not get native slash commands registered.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/77367 or an equivalent fix that namespaces the deploy cache by Discord application while preserving the cache’s rate-limit protection.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: multiple Discord accounts share one persisted cache file, and the second application can skip REST reconciliation when `global:reconcile` already has the same command hash.

## Solution Assessment

Yes. Scoping the existing deploy-cache keys by Discord application is the narrow maintainable fix; disabling the cache would avoid the symptom but lose the intended rate-limit protection.
