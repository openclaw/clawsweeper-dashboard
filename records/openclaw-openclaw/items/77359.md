## Summary

The report lines up with current main: Discord resolves per-account application IDs, but the persisted slash-command deploy cache is shared across Discord applications and keyed only by command-set scope, so a later account with the same command set can skip reconciliation before touching its own Discord application.

## What This Changes

Register Discord native slash commands for every configured bot account in multi-account setups, not only the default account.

## Best Possible Solution

Scope the Discord command-deploy cache by application/client ID or account ID, and add regression coverage proving two accounts with identical command specs still reconcile separate Discord applications.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: start two Discord accounts with identical native command specs and distinct application IDs after the shared deploy cache already contains `global:reconcile`. The later account can return before listing or creating commands for its own application.

## Solution Assessment

Yes, the narrow maintainable fix is to scope the existing deploy cache keys or file by Discord application/account. Disabling the cache would avoid the bug but would discard the rate-limit protection it was added to provide.

## Repair Work Prompt

Fix the current-main Discord multi-account native command registration bug reported in https://github.com/openclaw/openclaw/issues/77359. The command deploy cache is shared across Discord accounts/applications and keyed only by scopes like `global:reconcile`, so a later account with identical command specs can skip reconciliation for its own Discord application. Keep the fix inside the Discord command deploy/startup surface: scope the persisted hash keys or cache path by Discord application/client ID or account ID, preserving the existing rate-limit optimization. Add regression coverage for two Discord command deployers or monitor clients sharing a cache store with different client IDs and identical command specs; the second application must still run its own reconcile/list/create path. Do not change public config semantics, command names, or account loading behavior. Add a single-line `CHANGELOG.md` fix entry because this is user-facing.
