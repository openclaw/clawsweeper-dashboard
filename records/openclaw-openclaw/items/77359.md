## Summary

Keep open: current main still uses one shared Discord slash-command deploy cache with unscoped keys, so the bug remains source-reproducible; the open linked PR explicitly closes this issue and has not merged.

## What This Changes

The issue asks OpenClaw to register native Discord slash commands for every configured Discord account, not only the default account.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/77367 or an equivalent Discord plugin fix that scopes the command-deploy cache by application/client/account while preserving same-application cache skips.

## Reproduction Assessment

Yes. Source inspection on current main shows two Discord clients sharing one hash store and bare cache keys, so after one writes the command hash a later application can return before REST reconciliation; the issue discussion also includes live multi-bot confirmations.

## Solution Assessment

Yes. Scoping the existing deploy cache by Discord application/client/account is the narrow maintainable fix; disabling the cache would avoid the symptom but would discard the rate-limit protection it was added to provide.
