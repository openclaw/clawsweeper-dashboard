## Summary

Keep open: current main still uses bare Discord command-deploy cache keys in a shared cache file, live 2026.5.7 comments confirm the same symptom, and the linked pull request at https://github.com/openclaw/openclaw/pull/77367 specifically closes this issue but remains unmerged.

## What This Changes

The issue reports that Discord slash commands register for the default bot account but are skipped for secondary accounts in multi-bot gateway setups.

## Best Possible Solution

Land a focused Discord plugin fix that namespaces the command-deploy cache by application or account while preserving the cache's rate-limit protection and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: multiple Discord accounts share one persisted command-deploy cache, and a matching bare `global:reconcile` hash skips REST reconciliation for later applications.

## Solution Assessment

Yes. Scoping the existing deploy-cache keys by Discord application or account is the narrow maintainable fix; disabling the cache would avoid the symptom but would remove the rate-limit protection it was added to provide.
