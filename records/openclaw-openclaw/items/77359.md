## Summary

Keep open: current main still uses shared Discord slash-command deploy cache keys across applications, and the linked fixing PR is open but unmerged. A fresh user confirmation also reports the same cache symptom on v2026.5.7 with four Discord accounts.

## What This Changes

The issue reports that Discord slash commands register for the default bot account but can be skipped for secondary Discord bot accounts in a multi-account gateway.

## Best Possible Solution

Land a focused Discord plugin fix that namespaces command-deploy cache entries by Discord application ID while preserving the shared cache file's rate-limit protection.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: two Discord clients sharing one hash store and identical command specs can collide on `global:reconcile`, causing the later application to return before REST reconciliation.

## Solution Assessment

Yes. Scoping the existing cache by application ID is the narrow maintainable fix; disabling the cache would avoid the symptom but would lose its rate-limit protection.
