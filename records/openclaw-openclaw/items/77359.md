## Summary

Keep open: current main still has a source-reproducible Discord deploy-cache collision. Each account builds a client with its own application ID, but all accounts share one persisted command-deploy cache and the cache keys omit application/account scope, so an identical command set can make a secondary bot skip REST deployment.

## What This Changes

Ensure Discord slash commands register for every configured Discord account in multi-bot setups, not only the default account.

## Best Possible Solution

Keep native command deploy caching, but scope persisted deploy hashes by Discord application/client/account and add a regression test proving two different app IDs both reconcile identical commands.

## Reproduction Assessment

Yes, at source level: start two Discord accounts with distinct application IDs and the same generated command set; the first account writes the global:reconcile hash, and the second account can load the same hash and return before REST deployment. I did not perform a live Discord reproduction.

## Solution Assessment

No fix is implemented yet. The narrow maintainable fix is to scope the deploy cache key or path by application/account, not to remove command caching or add a new config knob.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77359. Current main lets multiple Discord accounts build separate clients with distinct application IDs, but the persisted command deploy cache is shared at discord/command-deploy-cache.json and uses keys like global:reconcile that are not scoped to the Discord application/account. Keep the cache optimization, but scope persisted deploy hashes by client/application/account so two accounts with identical command specs both reconcile against their own Discord application. Add regression coverage, preferably in extensions/discord/src/internal/client.test.ts or nearby Discord monitor tests, proving two clients with different clientId values and the same hashStorePath both issue their first reconcile REST calls, while repeated startup for the same clientId still skips unchanged deploys. Add a user-facing CHANGELOG.md fix entry. Do not add a config option, do not disable native command caching globally, and do not change Discord command contents unrelated to cache scoping.
