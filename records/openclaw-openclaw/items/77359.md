## Summary

Keep open: current main still uses a shared, unscoped Discord slash-command deploy cache, and the linked closing PR remains open rather than merged.

## What This Changes

The issue reports that Discord slash commands register for the default account but not for named secondary accounts in a multi-bot setup with per-account application IDs.

## Best Possible Solution

Merge or replace #77367 with a focused Discord plugin fix that scopes command-deploy cache entries by Discord application or account while preserving the cache's rate-limit protection.

## Reproduction Assessment

Yes. Source inspection shows two accounts with identical serialized commands share `command-deploy-cache.json` and the same bare `global:reconcile` key, so the second account can skip the deploy callback after the first stores the hash.

## Solution Assessment

Yes. Scoping the existing deploy cache by Discord application or account is the narrow maintainable fix; disabling the cache would avoid the symptom but lose its rate-limit protection.
