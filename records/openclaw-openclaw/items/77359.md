## Summary

Keep open: current main still uses bare Discord command-deploy cache keys against one shared cache file, recent release users still reproduce the skip, and the open closing PR has not merged.

## What This Changes

Discord multi-bot setups should register native slash commands for every configured Discord account instead of only accounts that avoid the shared deploy-cache hit.

## Best Possible Solution

Scope Discord command-deploy cache entries by resolved Discord application/client id while preserving the shared cache's rate-limit protection and covering the multi-application regression.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: two Discord accounts with the same command set share one persisted `global:reconcile` hash, and the second account can return before REST reconciliation; recent release comments confirm the same cache state in live multi-bot setups.

## Solution Assessment

Yes. Scoping the existing cache keys by Discord application/client id is the narrow maintainable fix; disabling the cache would avoid the symptom but would remove its rate-limit protection.
