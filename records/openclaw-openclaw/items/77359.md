## Summary

Keep open: current main still uses a shared, unscoped Discord command-deploy cache, live reporter comments confirm the bug on v2026.5.7, and the linked closing PR remains open but unmerged.

## What This Changes

The issue reports that secondary Discord bot accounts skip slash-command registration in multi-bot setups while the default account registers commands.

## Best Possible Solution

Land a focused Discord plugin fix that scopes persisted command-deploy cache entries by application/account and preserves multi-deployer persistence safety, then close this issue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a secondary account with the same serialized command set can hit the shared global:reconcile hash and return before REST reconciliation, and live comments confirm the same behavior on v2026.5.7.

## Solution Assessment

Yes. The narrow maintainable fix is to scope the existing cache by Discord application or account while preserving the rate-limit protection; the linked open PR is the current implementation candidate.
