## Summary

Keep open: current main and v2026.5.12 still use shared, unscoped Discord command-deploy cache keys, and the open closing PR already owns the likely fix path.

## What This Changes

The issue reports that Discord slash commands are registered for the default account but not for secondary Discord accounts in a multi-bot setup.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/77367, or an equivalent app/account-scoped deploy-cache fix, then close this issue after the fix merges and release provenance is clear.

## Reproduction Assessment

Yes. Source inspection shows two Discord applications can share one persisted hash such as `global:reconcile`, and the later account can return before its own REST reconciliation; issue comments corroborate the path on shipped builds.

## Solution Assessment

Yes. Scoping the cache by Discord application/account while preserving the same-application cache skip is the narrow maintainable fix; disabling the cache would avoid the symptom but remove the intended startup rate-limit protection.
