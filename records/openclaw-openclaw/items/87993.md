## Summary

Keep open: current main no longer creates a PT3M repeated trigger for fresh or repaired Windows tasks, but the package update path still calls the Windows service stop before install and that stop still uses `schtasks /End` without disabling an already-installed repeated task. The exact legacy-task update race is not owned by a merged or clearly canonical open fix.

## What This Changes

The issue asks OpenClaw to prevent a Windows Scheduled Task with a PT3M repetition trigger from restarting the gateway while `openclaw update` is replacing the package.

## Best Possible Solution

Add a narrow Windows update guard or migration so legacy repeated Scheduled Tasks cannot relaunch the gateway during package replacement, while keeping the current no-repetition XML for fresh/repaired tasks.

## Reproduction Assessment

Yes, source-reproducible with a seeded legacy Windows task that still has a repetition trigger: the update path stops the service before package install, and the Windows stop implementation uses `/End` without disabling the task. I did not run a native Windows live repro in this read-only pass.

## Solution Assessment

Partially; disabling/re-enabling or rewriting only around package update is a maintainable fix direction, but changing generic `gateway stop` to disable login startup should be avoided unless maintainers intentionally want that service-control semantic change.
