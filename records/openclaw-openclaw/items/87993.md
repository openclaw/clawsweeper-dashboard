## Summary

Keep open: current main and latest v2026.6.6 still stop the Windows Scheduled Task with `/End` without disabling an enabled legacy repeated trigger, and the open closing PR is the current implementation path rather than a merged fix.

## What This Changes

The issue reports that Windows package updates can let a legacy PT3M Windows Scheduled Task relaunch the gateway during package replacement because the updater stops the running task instance without disabling task autostart.

## Best Possible Solution

Land an update-scoped guard that disables only the enabled Windows Scheduled Task during mutable package replacement, restores only what that update invocation disabled, and preserves normal `gateway stop` semantics unless maintainers deliberately change them.

## Reproduction Assessment

Yes. Current main calls the managed service stop before package replacement, and the Windows stop implementation ends the running Scheduled Task without disabling an already-registered repeated trigger.

## Solution Assessment

Yes in direction. The update-scoped disable/re-enable shape in the open linked PR is narrower than changing generic gateway stop semantics, but the issue should remain open until that PR is rebased, reviewed, and merged or explicitly abandoned.
