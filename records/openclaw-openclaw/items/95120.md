## Summary

Keep open. The PR is focused, but it is too narrow to merge as the fix for the linked per-DM model override work because it only consumes `nativeDirectUserId`, while common affected channels store stable DM peers in existing sender/origin fields, so Slack, WhatsApp, and Discord DMs would still fall back to wildcard/default behavior.

## What This Changes

The PR adds an optional `directUserId` argument to the shared channel model override resolver, threads `sessionEntry.origin.nativeDirectUserId` through reply/status/agent-command callers, and adds resolver unit tests for DM-specific keys and precedence.

## Best Possible Solution

Before merge, centralize direct-message candidate normalization around the stable peer identifiers already present in session origin and finalized contexts, cover the affected channel callers in regression tests, and document accepted DM key forms.

## Reproduction Assessment

Yes at source level, but not through a live DM run. Current main lacks ordinary direct-peer candidates, and the patch still forwards only `nativeDirectUserId` even though Slack, WhatsApp, and Discord direct contexts expose peers through existing sender/origin fields.

## Solution Assessment

No. Threading only `nativeDirectUserId` is a plausible partial fix, but the best fix should normalize direct-message model override candidates from the existing generic origin/context fields and prove the linked providers.
