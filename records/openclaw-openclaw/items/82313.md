## Summary

Keep open. This is a fresh report against a docs-supported Brave web_search configuration, and current main plus the v2026.5.12 tag still contain the fatal startup validation and recovery/stop escape-path behavior that can make an unavailable plugin-owned provider block gateway startup.

## What This Changes

The issue reports that gateway startup rejects `tools.web.search.provider: "brave"` and crash-loops even after the external Brave plugin is installed, enabled, and inspectable.

## Best Possible Solution

Fix the generic plugin-owned web_search startup validation and recovery path so an unavailable Brave provider cannot crash-loop the gateway, while preserving strict validation for unknown provider typos.

## Reproduction Assessment

Yes, source-level: the report gives concrete 2026.5.12 steps and logs, and current main still contains the fatal provider-validation branch before runtime plugin bootstrap plus the recovery/stop behavior that matches the crash loop. I did not establish a live Brave-backed gateway reproduction in this read-only review.

## Solution Assessment

Yes, keeping this open is the right outcome; the merged runtime-provider fixes solved adjacent first-class tool routing, but the remaining startup validation/resilience path needs a generic fix rather than a Brave-only core shortcut.
