## Summary

Keep open. Current main has a partial responsiveness mitigation for default/configured `models.list`, but the cold Gateway catalog path still starts provider catalog augmentation before configured `models.providers` rows are merged, so the reported full provider-scan root cause remains plausible. The public issue also contains apparent credential fields, which blocks safe automated repair until maintainers handle redaction/rotation.

## What This Changes

The issue reports that v2026.4.29 Codex catalog fallback can trigger a broad provider plugin scan despite configured custom models, stalling startup and `models.list`, and asks to skip or disable that work for configured-provider setups.

## Best Possible Solution

Keep default/configured Gateway model listing on configured rows, auth-backed lightweight metadata, or stale cache without starting broad provider runtime catalog hooks. Reserve full provider catalog discovery for explicit `view: "all"`, diagnostics, or scoped background refresh that honors activation/allowlist intent and third-party plugin contracts.

## Reproduction Assessment

Yes for the source-level path: with explicit `models.providers`, a cold Gateway `models.list` still starts full catalog loading, and `loadModelCatalog` reaches provider augmentation before configured rows are appended. The exact event-loop delay remains environment-dependent and was not live-measured because this review was read-only.

## Solution Assessment

No. The current timeout is a useful user-visible mitigation, but the narrower maintainable fix is to avoid or tightly scope the cold provider augmentation work for configured/default views rather than only letting broad discovery continue in the background.
