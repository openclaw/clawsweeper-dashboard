## Summary

Keep open. Current main added a useful 750 ms timeout for normal `models.list` views, but the underlying cold catalog path still runs provider plugin catalog augmentation before merging configured `models.providers`, and that augmentation still scopes in bundled provider compat plugins broadly; the specific Codex-fallback/full-provider-scan complaint is not fully implemented or safely obsolete.

## What This Changes

The issue requests that Codex catalog fallback avoid a full built-in provider scan, or expose a disable switch, when custom `models.providers` are configured.

## Best Possible Solution

Keep exact full-catalog discovery available for explicit diagnostics, but make configured/default model views and startup warm paths avoid broad bundled provider runtime/catalog augmentation when configured provider rows are sufficient. A focused fix should preserve `view: "all"` semantics, respect plugin activation/allowlist intent, and add regression coverage proving custom `models.providers` do not trigger broad provider plugin loading after Codex fallback.

## Reproduction Assessment

Yes for the source-level reproduction shape: configure a custom model provider, make Codex discovery fail, then trigger cold catalog loading and observe that configured rows are merged only after provider plugin augmentation. The exact reported wall-clock delay still depends on the reporter's Windows/package/runtime-deps environment.

## Solution Assessment

No, current main is only a partial solution. The existing timeout protects normal `models.list` response timing in many cases, but the narrower maintainable fix should make the catalog loader/provider hook scope selective instead of relying on a timeout while broad runtime imports continue in the background.
