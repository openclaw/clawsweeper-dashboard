## Summary

Keep open. Current main improves the visible `models.list` wait with a 750 ms timeout, but the cold catalog path still starts provider catalog augmentation before configured `models.providers` rows are merged, so the reported provider-scan stall remains plausible and not fully superseded by the broader closed duplicate marker.

## What This Changes

The issue asks OpenClaw to skip or scope Codex fallback provider catalog scanning when explicit `models.providers` entries are configured, or to expose a config switch to disable that discovery.

## Best Possible Solution

Default and configured model-list views should answer from explicit configured rows, warm/stale cache, or manifest-light metadata without starting broad provider runtime catalog hooks; full provider discovery should remain available for explicit `view: "all"` and diagnostics.

## Reproduction Assessment

Yes. A high-confidence source trace exists: configure explicit `models.providers`, let Codex discovery fail, call default/configured `models.list`, and current main still starts `loadGatewayModelCatalog`, reaches provider augmentation, then appends configured rows afterward.

## Solution Assessment

No. The current timeout is a useful responsiveness mitigation, but the best fix is to skip, scope, or cancel cold catalog augmentation when configured model rows are sufficient instead of leaving the expensive scan running in the background.
