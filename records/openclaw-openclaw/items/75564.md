## Summary

Keep open. Current main still starts the cold Gateway model catalog refresh for default/configured `models.list`, and that refresh still reaches provider runtime catalog augmentation before explicit `models.providers` rows are appended. The 750 ms timeout mitigates visible RPC latency, but it does not avoid the background bundled-provider/Codex fallback work reported here; the later duplicate marker points to related closed issues rather than a live canonical fix.

## What This Changes

The issue reports that Codex catalog fallback on v2026.4.29 triggers a full bundled provider scan despite configured custom model providers, causing slow startup and slow `models.list` responses.

## Best Possible Solution

Make default/configured model catalog views answer from configured rows, cached rows, or lightweight manifest/static metadata without starting broad provider runtime catalog hooks. Keep full provider discovery for explicit `all` and diagnostic paths, and preserve plugin semantics plus allowlist intent.

## Reproduction Assessment

Yes. Configure explicit `models.providers`, force Codex live discovery to fail, and call default or configured `models.list` on a cold Gateway; current source shows that call starts provider catalog augmentation before configured rows are merged, though the exact Windows timing still needs live measurement.

## Solution Assessment

No. The current timeout is a useful responsiveness mitigation, but the narrower maintainable fix is to avoid or scope the cold provider augmentation work rather than letting broad discovery continue in the background.
