## Summary

Keep open: current main still mitigates the configured/default `models.list` response with timeout and stale-cache behavior, but the cold catalog loader still enters provider runtime augmentation before configured provider rows are merged, so the reported scan/stall remains plausible.

## What This Changes

The issue asks OpenClaw to avoid Codex fallback/full provider discovery during startup or default/configured `models.list` when `models.providers` already supplies explicit custom models.

## Best Possible Solution

Keep startup and default/configured `models.list` on configured rows, a warm/stale cache, or manifest-only metadata, and reserve full provider runtime catalog hooks for explicit `view: "all"` or diagnostic discovery paths.

## Reproduction Assessment

Yes. Configure explicit `models.providers`, start with a cold Gateway catalog while Codex discovery fails, then call default or configured `models.list`; source shows the first catalog load still reaches provider augmentation before configured rows are merged, though exact stall duration needs Windows/runtime timing.

## Solution Assessment

No. The current timeout and stale-cache behavior reduces some visible waits, but the narrower maintainable fix is to avoid or scope the cold provider augmentation work instead of starting the same broad catalog promise in the background.
