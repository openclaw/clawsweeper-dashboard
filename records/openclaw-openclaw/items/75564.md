## Summary

Keep open. Current main mitigates some visible `models.list` latency with a 750 ms timeout, but default/configured views still start the full Gateway model catalog load, and that load still reaches bundled provider catalog hooks before explicit `models.providers` rows are appended.

## What This Changes

When custom `models.providers` are configured and Codex catalog discovery fails, skip or make configurable the fallback full provider scan so startup and `models.list` do not stall.

## Best Possible Solution

Default/configured `models.list` should answer from configured rows, allowlists, warm cache, or manifest-light metadata without starting broad provider runtime catalog hooks; reserve full discovery for `view: all` and diagnostics.

## Reproduction Assessment

Yes. Configure explicit `models.providers`, make Codex live discovery fail, and call default/configured `models.list`; current source still starts `loadGatewayModelCatalog`, reaches provider augmentation, and appends configured rows only afterward.

## Solution Assessment

No. The current timeout is a useful mitigation, but it is not the best fix because it leaves the expensive catalog refresh running instead of avoiding provider runtime hooks when configured rows are sufficient.
