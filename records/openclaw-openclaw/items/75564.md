## Summary

Close as implemented. Current main and the shipped v2026.5.3 tag route default/configured `models.list` through a read-only catalog that includes configured models without loading PI/provider discovery or provider catalog hooks, which addresses the reported startup/RPC stall path.

## What This Changes

The issue reports that Codex catalog fallback scans bundled provider plugins despite explicit custom `models.providers`, causing long `models.list` latency and event-loop stalls.

## Best Possible Solution

Keep the shipped read-only default/configured model-list behavior, and use `plugins.entries.codex.config.discovery.enabled: false` only for users who intentionally want Codex live discovery disabled for full catalog or diagnostics flows.

## Reproduction Assessment

Yes. The affected-release path is clear from the report and prior review: explicit `models.providers`, failed Codex discovery, then `models.list` entering broad provider catalog work; current main has source and regression coverage showing that default/configured `models.list` no longer enters that path.

## Solution Assessment

Yes. The shipped fix is the narrower maintainable solution for the reported user problem because normal model listing now answers from persisted/static/configured rows while reserving full provider discovery for `view: "all"` and explicit diagnostics-style flows.
