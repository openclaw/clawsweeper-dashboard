## Summary

Keep open. Current main still starts the cold Gateway model catalog refresh for default/configured models.list, and that refresh still reaches bundled provider catalog augmentation before explicit configured model rows are appended. The 750 ms Gateway timeout improves the visible response, but it does not eliminate the background provider-runtime scan or the Codex fallback path reported here; the related #62364 remains a useful cluster, not a complete replacement for this Codex/allowlist-specific evidence.

## What This Changes

The issue reports that Codex catalog fallback can trigger broad bundled provider catalog scanning even when custom models.providers are configured, causing very slow models.list/startup on Windows.

## Best Possible Solution

Keep full provider discovery for explicit all/diagnostic paths, but make startup and default/configured models.list answer from configured rows, a warm/stale cache, or manifest-only/lightweight metadata without kicking off broad bundled provider runtime catalog hooks. The fix should also preserve plugin semantics, respect allowlist intent, and coordinate with the broader slow-startup work in #62364.

## Reproduction Assessment

Yes. Configure explicit models.providers, use a cold Gateway model catalog with Codex live discovery failing, then call default or configured models.list; current source shows that the background catalog refresh reaches provider augmentation before configured rows are merged, though the exact stall length needs live Windows/plugin-runtime timing.

## Solution Assessment

No. The current timeout is a useful user-visible mitigation, but the narrower maintainable fix is to avoid or scope the cold provider augmentation work instead of letting broad discovery continue in the background; a Codex discovery disable flag is only an escape hatch.
