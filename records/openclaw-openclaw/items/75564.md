## Summary

Keep open. Current main still mitigates the visible `models.list` wait with a 750 ms timeout, but it still starts the cold Gateway catalog refresh in the background; that refresh reaches bundled provider runtime catalog hooks before configured `models.providers` rows are appended, so the central scan/stall complaint remains plausible.

## What This Changes

The issue asks OpenClaw to avoid broad Codex/provider catalog fallback scanning during startup or `models.list` when explicit custom model providers are already configured.

## Best Possible Solution

Default/configured `models.list` should answer from configured rows, a warm/stale cache, or manifest-light metadata without starting broad provider runtime catalog hooks; full provider discovery should remain available for explicit `view: all` and diagnostics.

## Reproduction Assessment

Yes. Configure explicit `models.providers`, let Codex discovery fail, and call default/configured `models.list`; current source shows that path still starts `loadGatewayModelCatalog`, reaches provider augmentation, and only later appends configured rows.

## Solution Assessment

No. The current timeout is a useful responsiveness mitigation, but it is not the best fix because it leaves the expensive provider scan running; the narrower maintainable fix is to skip or scope cold catalog augmentation when configured models are sufficient.
