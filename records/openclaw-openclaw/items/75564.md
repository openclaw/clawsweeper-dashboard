## Summary

Keep open. Current main still starts the full Gateway model catalog load for default/configured `models.list`, and that load reaches broad bundled provider catalog hooks before explicit `models.providers` rows are appended; the 750 ms timeout only limits the RPC response when the event loop can run. The linked #75503 work reduces a related repeated-normalization cost but the discussion explicitly says it does not fix this Codex fallback/full-scan decision.

## What This Changes

The issue reports that v2026.4.29 starts a broad provider catalog scan after Codex model discovery fails, even when explicit custom `models.providers` are configured, causing Gateway startup and `models.list` stalls.

## Best Possible Solution

Default/configured model-list surfaces should answer from explicit configured rows, a warm/stale cache, or manifest-light metadata without launching broad provider runtime catalog hooks; reserve full provider discovery for `view: "all"` and explicit diagnostics.

## Reproduction Assessment

Yes. Configure explicit `models.providers`, make Codex app-server discovery fail, and call default/configured `models.list`; current source shows that path starts `loadGatewayModelCatalog`, reaches provider catalog hooks, and only appends configured rows afterward.

## Solution Assessment

No. The current timeout is a useful responsiveness mitigation, but it is not the best fix because it leaves the expensive full catalog refresh running instead of avoiding it when configured rows are already sufficient.
