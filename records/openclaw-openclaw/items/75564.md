## Summary

Keep open. Current main still mitigates the visible `models.list` wait with a 750 ms timeout, but it does not stop the cold background catalog refresh from reaching bundled provider catalog hooks before configured model rows are merged; the linked #75503 work is relevant but explicitly only a partial mitigation for the scan cost.

## What This Changes

Avoid the cold full provider catalog scan when explicit `models.providers` are configured and Codex live discovery falls back during Gateway startup or `models.list`.

## Best Possible Solution

Make default/configured startup and `models.list` answer from configured rows, a warm/stale cache, or manifest-only metadata without starting broad provider runtime catalog hooks; keep `view: "all"` and explicit diagnostics as the full discovery path.

## Reproduction Assessment

Yes. Configure explicit `models.providers`, make Codex live discovery fail, then hit a cold default/configured `models.list`; current source shows the call starts the full Gateway catalog refresh and reaches provider augmentation before configured rows are appended.

## Solution Assessment

No. The current timeout is a useful responsiveness mitigation, but the maintainable fix is to avoid or scope the cold provider augmentation work itself; a Codex discovery disable flag would only be an escape hatch.
