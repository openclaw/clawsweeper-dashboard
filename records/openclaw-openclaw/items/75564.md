## Summary

Keep open. Current main only partially mitigates the visible `models.list` delay with a timeout for default/configured views; the cold catalog load still starts provider catalog augmentation before explicit `models.providers` rows are merged, and Codex discovery failure still falls back through that catalog path. The public report also includes apparent credential material, so maintainer redaction/rotation should happen before normal repair automation.

## What This Changes

The issue asks OpenClaw to avoid full bundled provider catalog discovery when configured model providers already exist and Codex catalog discovery falls back.

## Best Possible Solution

Keep full provider catalog discovery for explicit diagnostics such as `view: "all"`, but make Gateway startup and default/configured model views avoid broad provider runtime/catalog hook loading when explicit configured rows are sufficient, while preserving plugin allowlist semantics and documenting any Codex discovery opt-out that remains supported.

## Reproduction Assessment

Yes for a source-level reproduction path: configure `models.providers`, force Codex live discovery to fail, then trigger a cold Gateway model catalog load and observe provider augmentation before configured rows are appended. The exact event-loop delay needs a Windows/runtime-deps live run to quantify.

## Solution Assessment

No. The current timeout is a useful partial mitigation for default/configured `models.list`, but the narrower maintainable fix is to avoid or scope the cold provider augmentation path rather than starting it in the background when configured rows already satisfy the view.
