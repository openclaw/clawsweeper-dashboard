## Summary

Keep open. Current main still starts the cold Gateway model catalog for default/configured `models.list`, and `loadModelCatalog` still runs provider catalog augmentation before appending configured `models.providers` rows; the 750 ms timeout improves the visible response but does not prevent the background scan/stall. It is not safe to close as a duplicate because #62364 and #75688 are closed, while #75503 was closed unmerged and was described as only addressing a caching subproblem, not the Codex fallback trigger.

## What This Changes

Skip or disable Codex fallback full provider/plugin catalog scanning when explicit `models.providers` entries are configured, avoiding startup and `models.list` event-loop stalls.

## Best Possible Solution

Default/configured `models.list` should answer from configured rows, a warm/stale cache, or manifest-light metadata without starting broad provider runtime catalog hooks; full provider discovery should remain explicit for `view: "all"` and diagnostics.

## Reproduction Assessment

Yes. Configure explicit `models.providers`, make Codex live discovery fail, and call default/configured `models.list`; current source shows that path starts `loadGatewayModelCatalog`, reaches provider augmentation, and only later appends configured rows.

## Solution Assessment

No. The current timeout is a useful responsiveness mitigation, but the narrower maintainable fix is to skip or scope cold catalog augmentation when configured models are sufficient instead of letting the expensive scan continue in the background.
