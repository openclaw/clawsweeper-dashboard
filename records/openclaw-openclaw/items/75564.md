## Summary

Current main implements the requested behavior for the reported surface: default/configured `models.list` now uses a read-only catalog path that avoids provider runtime discovery and configured `models.providers` rows are returned without triggering the full provider scan.

## What This Changes

The issue asks OpenClaw to avoid full Codex/provider catalog fallback scanning when explicit `models.providers` entries are configured, or to provide a switch to disable that discovery.

## Best Possible Solution

Keep the current read-only configured/default catalog path on main, reserve full provider discovery for explicit `view: "all"` diagnostics, and ship the fix in the next release.

## Reproduction Assessment

No for current main. The old failure path is clear from the report, but current main routes default/configured `models.list` through read-only catalog loading that does not import provider discovery or run provider augmentation hooks.

## Solution Assessment

Yes. The current implementation is narrower than adding a user-facing config knob because it makes the common configured/default model list side-effect-free while preserving the full catalog scan behind `view: "all"`.
