## Summary

Keep open. Current main only caps the visible default/configured `models.list` wait at 750 ms; a cold catalog request still launches the full Gateway model-catalog refresh, reaches provider catalog hooks, and appends explicit `models.providers` rows afterward, so the reported background provider-scan stall remains plausible.

## What This Changes

The issue requests skipping or disabling Codex fallback provider-catalog scanning when explicit `models.providers` entries are already configured.

## Best Possible Solution

Default/configured model-list paths should return from explicit configured rows, warm cache, or manifest-light metadata without starting broad provider runtime catalog hooks; full provider discovery should stay reserved for `view: "all"` and diagnostics.

## Reproduction Assessment

Yes. Configure explicit `models.providers`, make Codex discovery fail, and call default/configured `models.list`; current source still starts `loadGatewayModelCatalog`, reaches provider augmentation, and only then appends configured rows.

## Solution Assessment

No. The current timeout is a useful UI responsiveness mitigation, but it is not the best fix because it leaves the expensive catalog refresh running in the background instead of scoping or skipping it when configured rows are sufficient.
