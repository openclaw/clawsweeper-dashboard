## Summary

Keep open. Current main partially mitigates the user-visible `models.list` delay with a 750 ms timeout for default/configured views, but the cold catalog path still starts provider catalog augmentation before explicit `models.providers` rows are merged, and Codex discovery failure still falls back into that catalog path. The issue also needs maintainer handling because the public report includes apparent credentials that should be redacted and rotated.

## What This Changes

The issue asks OpenClaw to avoid broad Codex fallback/provider catalog scanning when explicit custom `models.providers` are configured, or to provide a config switch to disable that discovery path.

## Best Possible Solution

Keep full catalog discovery available for explicit diagnostic paths, but make startup, default/configured `models.list`, and picker-sized model views avoid broad bundled provider runtime/catalog hook loading when explicit configured model rows are sufficient. The fix should preserve `view: "all"` semantics and honor plugin activation/allowlist intent.

## Reproduction Assessment

Yes for the source-level reproduction path: configure custom `models.providers`, make Codex live discovery fail, then trigger a cold Gateway model catalog load and observe provider plugin augmentation before configured rows are appended. The exact wall-clock delay remains environment-dependent because this review stayed read-only.

## Solution Assessment

No. The current timeout is a useful partial mitigation for default/configured `models.list` responses, but the narrower maintainable fix is to avoid or scope the cold provider augmentation path instead of letting the broad load continue in the background.
