## Summary

Keep open. Current main has a response-time mitigation for default/configured `models.list`, but the cold catalog path still starts bundled provider catalog augmentation before explicit configured model rows are merged, and Codex discovery failure can still enter that path. The report is also security-sensitive because the pasted config appears to contain credentials.

## What This Changes

The issue asks OpenClaw to avoid broad Codex fallback/provider catalog scanning when `models.providers` already defines custom models, or to provide a way to disable that catalog path.

## Best Possible Solution

Keep full provider catalog discovery available for explicit `view: "all"` and diagnostic paths, but make startup and default/configured `models.list` answer from configured rows, warm/stale cache, or tightly scoped background work without broad bundled provider runtime/catalog hook loading. The fix should preserve third-party plugin semantics and honor plugin activation/allowlist intent.

## Reproduction Assessment

Yes for the source-level reproduction path: configure explicit `models.providers`, make Codex live discovery fail, then trigger a cold Gateway model catalog load and observe provider catalog augmentation before configured rows are appended. The exact delay remains environment-dependent because this review stayed read-only.

## Solution Assessment

No. The current timeout is a useful partial mitigation for user-visible default/configured responses, but the narrower maintainable fix is to avoid or scope the cold provider augmentation work rather than only letting broad discovery continue in the background; a `codex.catalog.enabled` flag may be an escape hatch, not the core fix.
