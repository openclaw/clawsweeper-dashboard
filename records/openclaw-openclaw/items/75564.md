## Summary

Keep open. Current main has a timeout around `models.list`, but the non-`all` path still starts the full gateway model catalog load before resolving configured rows, and that catalog load can still enter bundled provider catalog hooks including Codex discovery. That means the reported event-loop/startup pressure is not proven fixed by current code.

## What This Changes

The issue asks OpenClaw to avoid Codex fallback/full bundled-provider catalog discovery when explicit `models.providers` custom models already satisfy normal model listing, or to expose a disable switch for that behavior.

## Best Possible Solution

For default/configured `models.list`, resolve `agents.defaults.models` and explicit `models.providers.*.models` before starting `loadGatewayModelCatalog`; only fall back to the full catalog when those rows cannot satisfy the requested view. Keep `view: "all"` as the explicit full-discovery path, and document the existing Codex plugin discovery disable setting if it remains relevant.

## Reproduction Assessment

Yes, there is a high-confidence static reproduction path from the issue steps: a config with explicit `models.providers` plus unavailable Codex discovery still reaches `loadGatewayModelCatalog`, provider catalog hooks, and Codex live discovery before configured rows are returned. I did not live-run the Windows timing scenario in this read-only pass.

## Solution Assessment

No, the current timeout is not the best complete fix because it still starts the broad catalog refresh and can leave expensive provider discovery running in the background. A narrower maintainable fix is to avoid starting provider catalog discovery when configured rows already satisfy non-`all` model-list requests.
