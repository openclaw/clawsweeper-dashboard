## Summary

Keep open. Current main has a useful response-time mitigation for normal Gateway model-list views, but the underlying cold catalog path can still enter broad provider discovery/runtime catalog hooks before explicit configured models are appended, so the reported startup/event-loop problem is not fully resolved. The public report also needs maintainer handling because it includes apparent credentials.

## What This Changes

The issue requests that configured `models.providers` avoid Codex fallback-triggered broad provider scanning, or provide a way to disable that catalog behavior.

## Best Possible Solution

Keep this open, redact/rotate the exposed secrets, then either land/adapt #63552 and extend it as needed so startup, configured/default `models.list`, and picker-sized views do not launch broad bundled provider discovery/runtime catalog work when explicit configured models are sufficient. Preserve exact full-catalog behavior for explicit diagnostic views such as `view: "all"`.

## Reproduction Assessment

Yes for a source-level reproduction: configure explicit `models.providers`, force Codex/provider discovery failure, then trigger a cold Gateway catalog load and observe that current main still enters broad implicit discovery and provider catalog augmentation before configured rows are appended. The exact event-loop delay needs live Windows or constrained-host measurement.

## Solution Assessment

No. The current 750 ms Gateway timeout is a partial user-visible mitigation, but the maintainable fix is to scope or skip the cold provider discovery/catalog augmentation path itself for configured-model views while leaving full diagnostic catalog loading explicit.
