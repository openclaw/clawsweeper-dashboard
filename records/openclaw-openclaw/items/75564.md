## Summary

Keep open. Current main improves the visible default/configured `models.list` response with a 750 ms timeout, but it still starts the cold Gateway model catalog refresh and that refresh still reaches bundled provider runtime catalog hooks before configured provider rows are appended. The report also contains Codex/allowlist-specific evidence and apparent pasted credentials, so it should not be auto-closed as a duplicate of the broader closed startup issue without maintainer handling.

## What This Changes

The issue requests skipping or disabling Codex fallback-driven full provider catalog scanning when explicit `models.providers` custom models are already configured.

## Best Possible Solution

Keep full provider discovery for explicit full-catalog and diagnostic paths, but make startup plus default/configured `models.list` answer from configured rows, allowlist-synthetic rows, a stale cache, or manifest-only metadata without starting broad bundled provider runtime catalog hooks. A Codex discovery disable knob can be useful, but the durable fix is to avoid unnecessary background provider runtime work for configured-model views.

## Reproduction Assessment

Yes. Configure explicit `models.providers`, force Codex discovery to fail or time out, then call default/configured `models.list`; current source shows the call starts the Gateway catalog refresh and reaches provider augmentation before configured rows are merged. The exact Windows event-loop delay still needs live timing proof.

## Solution Assessment

No. The current timeout is a useful response-latency mitigation, but it is not the best fix because it still lets the expensive catalog refresh continue in the background. The narrower maintainable solution is to skip or scope provider augmentation when the requested view can be satisfied from configured/allowlisted/static metadata.
