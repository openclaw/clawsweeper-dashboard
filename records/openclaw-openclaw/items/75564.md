## Summary

Keep open. Current main still starts the Gateway model catalog refresh for default/configured `models.list`, reaches provider runtime catalog augmentation before appending explicit `models.providers` rows, and only mitigates the visible RPC response with a timeout while the background scan can continue.

## What This Changes

The issue reports that v2026.4.29 runs broad Codex/provider catalog fallback work during Gateway startup or `models.list` despite explicit local `models.providers`, causing long event-loop stalls.

## Best Possible Solution

Make default/configured Gateway model listing answer from explicit configured rows, a warm cache, or manifest-only lightweight metadata without kicking off broad provider runtime catalog hooks; reserve full discovery for `view: "all"` and diagnostics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure explicit `models.providers`, make Codex live discovery fail, then call default/configured Gateway `models.list`; the current path still reaches provider catalog augmentation before configured rows are merged.

## Solution Assessment

No. The current 750 ms timeout and Codex discovery opt-out reduce user-visible waiting or provide an escape hatch, but they do not remove or properly scope the cold background provider-runtime scan.
