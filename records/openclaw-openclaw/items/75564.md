## Summary

Keep open. Current main only mitigates the visible `models.list` wait with a 750 ms timeout; the cold catalog refresh still starts in the background and reaches provider catalog hooks before configured model rows are appended, so the reported provider-scan stall remains plausible.

## What This Changes

The issue asks OpenClaw to skip or disable the full Codex/catalog provider scan when explicit `models.providers` are configured and Codex live catalog discovery falls back.

## Best Possible Solution

Default/configured `models.list` should answer from configured rows, a warm/stale cache, or manifest-light metadata without starting broad provider runtime catalog hooks; full provider discovery should remain explicit for `view: all` and diagnostics.

## Reproduction Assessment

Yes. Configure explicit `models.providers`, make Codex discovery fail, and call default/configured `models.list`; current source shows that path starts `loadGatewayModelCatalog`, reaches provider augmentation, and appends configured rows only afterward.

## Solution Assessment

No. The current timeout is a useful responsiveness mitigation, but it is not the best fix because it leaves the expensive provider scan running in the background instead of scoping or skipping it when configured models are sufficient.
