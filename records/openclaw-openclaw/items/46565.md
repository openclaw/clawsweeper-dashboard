## Summary

Keep open. Current main partially mitigates the reported context clash with hard-coded subagent/cron bootstrap filtering, opt-in `sessions_spawn.lightContext`, and cron/heartbeat time lines, but the requested dynamic bootstrap expansion, configurable session-type filters, and every-session live time contract are still absent and require maintainer design.

## What This Changes

The issue requests dynamic bootstrap context expansion, configurable per-session bootstrap filtering, and built-in current time/timezone context for all sessions.

## Best Possible Solution

Define an owner-approved bootstrap context contract that covers session-kind filtering, a safe dynamic-context extension route, and stable time access without breaking prompt caching or plugin boundaries.

## Reproduction Assessment

No strict bug reproduction is applicable because this is a feature/API request. Source inspection gives a high-confidence current-main state: partial `lightContext` and scheduled-run time support exists, but the requested dynamic bootstrap and configurable filter surfaces are absent.

## Solution Assessment

Unclear. Session filtering likely belongs in core if accepted, but bootstrap `!command` expansion and direct clock injection need maintainer design because existing hooks/tool-time paths may be safer and the prompt-cache/security implications are real.
