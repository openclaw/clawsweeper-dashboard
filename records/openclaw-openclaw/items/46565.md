## Summary

Keep open. Current main has a hard-coded subagent/cron bootstrap allowlist that should prevent the reported HEARTBEAT.md leakage into ordinary subagent runs, and it has timezone/time helpers for selected runtime paths, but it does not implement the requested dynamic bootstrap preprocessing or a configurable session-type bootstrap filter. The current system-prompt tests also intentionally keep wall-clock time out of the prompt for cache stability, so the remaining request needs maintainer product and security design rather than cleanup closure.

## What This Changes

Request dynamic preprocessing for bootstrap files, configurable session-type bootstrap filtering, and built-in current time/timezone context for autonomous sessions.

## Best Possible Solution

Define an explicit platform contract before implementation: keep the existing hard-coded subagent/cron safety filter, add a narrow documented config/schema for per-session bootstrap inclusion or exclusion if maintainers want that control, and design any dynamic bootstrap/time injection behind opt-in, deterministic, security-reviewed semantics that preserve prompt cache behavior where required.

## Reproduction Assessment

Partly. The comments give a concrete HEARTBEAT_OK/subagent failure pattern, and current main has source-level filtering that addresses that symptom, but there is no runnable minimal reproduction for the broader dynamic-bootstrap and every-session time-context feature request.

## Solution Assessment

Unclear. The current allowlist is a reasonable partial fix for heartbeat leakage, but it is not a complete solution to dynamic bootstrap preprocessing or configurable session-type filtering; the safer path is a maintainer-approved contract before coding.
