## What This Changes

The issue reports that the macOS Talk mode OpenClaw Chat window in v2026.6.11 renders as a 30-40px-wide column even when the actual window is hundreds of pixels wide.

## Best Possible Solution

Fix the native macOS WebChat window/container sizing so OpenClawChatView receives the real NSWindow content width on first layout and after resize, then prove the 500x840 and 960x900 Chat window layouts visually.

## Reproduction Assessment

No; the issue gives a credible manual path on v2026.6.11, but this read-only review did not run the packaged macOS app or capture the collapsed window. Source inspection narrows the path to WebChatManager, WebChatSwiftUIWindowController, and OpenClawChatView.

## Solution Assessment

No close or config workaround is the best answer. The maintainable fix should be in native macOS WebChat/SwiftUI layout, not by re-adding retired talk.ui/inputWidth config keys or changing unrelated browser CSS first.
