## Summary

Keep open. Current main still hard-codes the Control UI CSP and gateway Permissions-Policy shape, and the remaining request is a valid but security-sensitive config/API decision rather than a cleanup-close candidate.

## What This Changes

The issue requests configurable Control UI Permissions-Policy and CSP settings so custom Control UI roots can enable browser permissions and additional connect-src origins without patching installed gateway bundles.

## Best Possible Solution

Add a maintainer-approved, typed Control UI header-policy config that keeps hardened defaults and validates allowed CSP directives, origins, and Permissions-Policy directives with schema, docs, and focused gateway tests.

## Reproduction Assessment

Yes, for the missing configuration surface. Source inspection shows the CSP and Permissions-Policy values are still fixed and the config contract has no accepted keys for CSP origins or policy overrides.

## Solution Assessment

Unclear as proposed. The use case is concrete, but free-form CSP or Permissions-Policy overrides would be a risky public API; a constrained typed policy with validation is the safer direction.
