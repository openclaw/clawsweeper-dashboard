## Summary

Keep open. Current main and the latest release still expose fixed Control UI CSP and Permissions-Policy headers with no supported config surface; only the same-origin microphone case and a few built-in browser origins have been hard-coded, so the custom Control UI use case remains a security-sensitive feature/API decision.

## What This Changes

The issue requests configurable Control UI CSP and Permissions-Policy headers for custom Control UI roots that need microphone access or extra browser connect-src origins.

## Best Possible Solution

Keep hardened defaults; if maintainers accept the use case, add a constrained typed Control UI header-policy config with validation, schema/help/docs updates, security notes, and focused gateway/header tests.

## Reproduction Assessment

Yes, for the missing configuration surface. Source inspection on current main shows fixed CSP and Permissions-Policy headers plus strict config schemas with no requested keys, so a live browser run is not needed to prove that gap.

## Solution Assessment

Unclear as proposed. The use case is valid, but free-form cspOverrides or raw Permissions-Policy values are not obviously the safest API; a constrained typed policy or same-origin gateway/plugin route may be safer.
