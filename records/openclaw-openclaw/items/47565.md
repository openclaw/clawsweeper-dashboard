## Summary

Keep open. Current main and v2026.5.19 still hard-code the Control UI CSP and gateway Permissions-Policy, and the config schema/docs do not accept the requested policy keys; the related open PR at https://github.com/openclaw/openclaw/pull/66783 covers only Permissions-Policy and still needs security/API-shape review.

## What This Changes

The issue requests configurable Control UI browser security headers for custom Control UI roots, including Permissions-Policy microphone permissions and CSP connect-src overrides.

## Best Possible Solution

Keep hardened defaults, then add a maintainer-approved typed Control UI header-policy config with strict directive/origin validation, schema/help/docs coverage, and focused gateway header tests.

## Reproduction Assessment

Yes, for the missing configuration surface. Source inspection shows fixed CSP and Permissions-Policy headers plus strict config schemas with no permissionsPolicy, cspOverrides, or configurable connect-src key.

## Solution Assessment

Unclear as proposed. The custom-Control-UI use case is concrete, but free-form CSP overrides or raw Permissions-Policy strings are not obviously the safest public API; a constrained typed policy would be safer.
