## Summary

Keep open. Current main and v2026.6.10 still hardcode the Control UI CSP/framing policy and only partially address the report by allowing same-origin microphone access; a configurable browser header policy remains a security-sensitive product/API decision.

## What This Changes

Add configurable Control UI Permissions-Policy and CSP behavior so custom Control UI roots can use microphone access and approved local browser fetch origins without patching installed dist files.

## Best Possible Solution

Choose a maintainer-approved narrow Control UI header policy contract, or document the same-origin plugin route alternative for local services; if config is accepted, implement safe defaults, strict validation, schema/help/docs updates, and focused served-header tests.

## Reproduction Assessment

Yes, at source level: current main hardcodes the relevant Control UI CSP/framing and Permissions-Policy headers, and the strict config schema lacks the requested policy fields. This is a missing configuration capability, not a broken existing behavior contract.

## Solution Assessment

Unclear until maintainers decide the security contract. A typed, directive-limited policy or same-origin plugin route guidance is safer than a broad raw header override.
