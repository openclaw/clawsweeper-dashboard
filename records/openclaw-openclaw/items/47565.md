## Summary

Keep open. Current main and v2026.5.12 still have fixed Control UI CSP and Permissions-Policy headers with no supported config surface; recent merged work only added specific allowlisted origins and same-origin microphone, while https://github.com/openclaw/openclaw/pull/66783 covers only part of the request and still needs security/API-shape review.

## What This Changes

Request configurable Control UI Permissions-Policy and CSP header overrides so custom Control UI roots can use browser permissions and local fetch origins without patching installed dist files.

## Best Possible Solution

Keep hardened defaults, then add a maintainer-approved typed Control UI header-policy config only if security owners accept the use case, with strict validation, generated schema/help/docs, and focused gateway header tests.

## Reproduction Assessment

Yes, for the missing configuration surface. Source inspection on current main and v2026.5.12 shows fixed CSP and Permissions-Policy headers plus strict config schemas with no requested keys.

## Solution Assessment

Unclear as proposed. The custom-Control-UI use case is concrete, but free-form cspOverrides or raw Permissions-Policy strings are not obviously the safest public API; a constrained typed policy or same-origin gateway/plugin route may be safer.
