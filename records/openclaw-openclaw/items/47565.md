## Summary

Keep open. Current main and v2026.5.22 still hard-code the Control UI CSP and Permissions-Policy shapes apart from specific built-in allowlist updates, the strict config/docs expose no requested header-policy keys, and the remaining API would broaden the authenticated Control UI browser security surface.

## What This Changes

The issue requests configurable Control UI CSP and Permissions-Policy headers so custom Control UI roots can use browser permissions and local fetch origins without patching installed dist files.

## Best Possible Solution

Add a maintainer-approved, typed Control UI header-policy config that preserves hardened defaults and validates allowed CSP directives, origins, and Permissions-Policy directives with schema, docs, and focused gateway tests.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.5.22 shows the CSP and Permissions-Policy values are fixed, and the strict config contract has no accepted keys for CSP origins or policy overrides.

## Solution Assessment

Unclear as proposed. The use case is valid, but raw CSP or Permissions-Policy overrides would be too broad without a maintainer-approved typed, allowlisted API.
