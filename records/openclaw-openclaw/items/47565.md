## Summary

Keep open. Current main still emits fixed Control UI CSP and Permissions-Policy headers, and the strict config contract has no header-policy keys; related fixes only added specific allowlisted origins, same-origin microphone access, or same-origin plugin routes, so the remaining request is a security-sensitive feature/API decision.

## What This Changes

Request a `gateway.controlUi` config surface for Permissions-Policy and CSP customization so custom Control UI roots can use browser microphone permission and local inference fetches without patching installed bundles.

## Best Possible Solution

Keep hardened defaults; if maintainers accept the use case, add a constrained typed `gateway.controlUi` header-policy config with allowlisted directives/origins, docs, config metadata, and focused HTTP/header tests, while steering local services toward same-origin plugin routes where possible.

## Reproduction Assessment

Yes, for the feature gap. Source inspection on current main shows fixed CSP and Permissions-Policy headers plus a strict config schema with no requested keys; no live browser run was needed to prove the missing config surface.

## Solution Assessment

Unclear as proposed. The use case is reasonable, but raw `cspOverrides` or free-form `permissionsPolicy` values are not the safest maintainable shape; a constrained schema or same-origin plugin route is safer.
