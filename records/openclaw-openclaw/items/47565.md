## Summary

Keep open. Current main has narrow fixed allowlists for OpenAI/tweakcn and same-origin microphone access, but it still has no supported config surface for custom Control UI CSP or Permissions-Policy changes; the remaining request is a security-sensitive feature/API decision.

## What This Changes

The issue requests configurable Control UI CSP and Permissions-Policy headers so custom Control UI roots can use browser permissions and local service origins without patching installed dist files.

## Best Possible Solution

Keep hardened defaults; if maintainers accept the use case, add a constrained typed header-policy config with validation, docs, schema metadata, and focused gateway/header tests.

## Reproduction Assessment

Yes, for the feature gap. Source inspection on current main shows fixed CSP and Permissions-Policy headers plus strict config schemas with no requested keys; no live browser run was needed to prove the missing config surface.

## Solution Assessment

Unclear as proposed. The use case is reasonable, but raw cspOverrides or free-form Permissions-Policy values are not the safest maintainable API; constrained typed config or same-origin plugin routing would be safer.
