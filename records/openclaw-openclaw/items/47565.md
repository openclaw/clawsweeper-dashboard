## Summary

Keep open. Current main and v2026.5.18 still hard-code the Control UI CSP and gateway Permissions-Policy surfaces, with only specific allowlisted origins and same-origin microphone access added; there is still no supported config key for custom local `connect-src` origins or policy overrides, and the related open PR only covers part of the request.

## What This Changes

The issue requests configurable Control UI CSP and Permissions-Policy headers for custom Control UI roots that need browser permissions and local service fetches.

## Best Possible Solution

Keep hardened defaults, then add a maintainer-approved typed Control UI header-policy config with strict directive/origin validation, schema/help/docs coverage, and focused gateway header tests.

## Reproduction Assessment

Yes, for the missing configuration surface. Source inspection shows fixed CSP and Permissions-Policy headers plus strict config schemas with no `permissionsPolicy`, `cspOverrides`, or configurable `connect-src` key.

## Solution Assessment

Unclear as proposed. The custom-Control-UI use case is concrete, but free-form CSP overrides or raw Permissions-Policy strings are not obviously the safest public API; a constrained typed policy or approved gateway route would be safer.
