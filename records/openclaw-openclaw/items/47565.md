## Summary

Keep open. Current `main` and `v2026.6.10` still hardcode the Control UI CSP/framing policy and the gateway `Permissions-Policy`; same-origin microphone access is fixed, but configurable browser-header policy remains a security-sensitive config/product decision.

## What This Changes

The issue requests configurable Control UI Permissions-Policy and CSP options so custom Control UI roots can allow browser permissions and local fetch origins without patching dist files.

## Best Possible Solution

Keep hardened defaults, then either add a maintainer-approved typed Control UI header policy with strict validation/docs/tests or document the same-origin plugin route alternative for local services.

## Reproduction Assessment

Not applicable as a broken existing-behavior bug, but yes at source level: current `main` hardcodes the relevant headers and the strict config schema lacks the requested policy fields.

## Solution Assessment

Unclear until maintainers choose the security contract. A narrow directive-limited policy or same-origin plugin route guidance is safer than a broad raw header override.
