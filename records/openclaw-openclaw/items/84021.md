## Summary

This is a DingTalk channel-plugin behavior request, not an OpenClaw core bug: current main has no DingTalk bundled plugin, and the existing channel plugin outbound API already gives a DingTalk plugin the send hooks needed to translate `<@数字>` into DingTalk API mention fields.

## What This Changes

The issue asks for DingTalk announce delivery to parse `<@userId>` text and populate `atDingTalkUserIdList` so mentioned users receive notifications.

## Best Possible Solution

Implement the `<@数字>` to `atDingTalkUserIdList` translation inside the DingTalk channel plugin and publish or track that plugin through ClawHub unless a concrete missing SDK hook is identified.

## Reproduction Assessment

No. Current main does not contain a DingTalk channel plugin to reproduce against, and the observed behavior belongs to an external or future channel plugin implementation.

## Solution Assessment

Yes for OpenClaw core cleanup. The maintainable path is to keep core delivery generic and put DingTalk mention parsing in the DingTalk plugin’s outbound send implementation.
