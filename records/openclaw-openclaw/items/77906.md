## Summary

Keep open: current main source confirms the custom-tool limitation, but this PR still has a blocking hidden tool-scope issue and lacks after-fix real behavior proof. The protected `maintainer` label also means this needs explicit maintainer handling before merge.

## What This Changes

The PR adds Active Memory `config.toolsAllow`, passes it into the embedded recall sub-agent, updates prompt/docs/changelog defaults, and treats missing or failed recall as empty memory.

## Best Possible Solution

Keep the configurable recall-tool direction, but centralize or derive the memory-safe tool boundary from the embedded-runner local tool catalog and require a redacted real-provider run before merging.

## Reproduction Assessment

Yes for a source-level reproduction: current main has a fixed Active Memory recall allowlist and passes it into `runEmbeddedPiAgent`, so a custom recall tool name cannot be configured. I did not establish a live custom-provider failure in this read-only review.

## Solution Assessment

No, not yet. Making recall tools configurable is the right narrow direction, but the PR must close the hidden-run local tool boundary and include real custom-provider output before merge.
