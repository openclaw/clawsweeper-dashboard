## Summary

Keep open: this PR addresses a real source-level Active Memory limitation, but it is not merge-ready because the new tool-scope guard misses current core tools and the real behavior proof does not exercise the custom or LanceDB path central to the change.

## What This Changes

The branch adds Active Memory `config.toolsAllow`, resolves memory-core/LanceDB/custom recall tool lists, filters reserved entries, updates prompt/schema/docs/changelog, and adds focused tests plus type-lint test cleanups.

## Best Possible Solution

Keep the PR open, anchor the hidden sub-agent tool filter to the current core tool catalog or another maintainable policy, then land after live custom-or-LanceDB proof and green validation.

## Reproduction Assessment

Yes, at source level: current main hardcodes Active Memory's hidden recall run to `memory_recall`, `memory_search`, and `memory_get`, so custom memory-provider tool names cannot be configured. I did not live-reproduce a custom provider failure in this read-only review.

## Solution Assessment

No, not as written: the direction is narrow and useful, but the security filter should not be a hand-copied subset of core tools that already misses `code_execution` and `x_search`. The safer fix is to derive or validate against the existing core tool catalog while preserving the explicit memory-provider tool escape hatch.
