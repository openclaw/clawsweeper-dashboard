## Summary

Keep open: source inspection confirms a real hardcoded Active Memory recall-tool limitation, but the current patch still has a security-sensitive allowlist gap and the real behavior proof does not cover custom or LanceDB recall.

## What This Changes

The branch adds Active Memory `config.toolsAllow`, dynamic memory-core/LanceDB/custom recall tool selection, reserved-entry filtering, docs/changelog updates, and focused tests plus type-lint cleanups.

## Best Possible Solution

Keep the PR open, derive or validate the hidden recall allowlist from the canonical core tool catalog or otherwise deny all non-memory core tools, then land after live custom-or-LanceDB proof and green targeted validation.

## Reproduction Assessment

Yes, at source level: current `main` hardcodes Active Memory's embedded recall run to `memory_recall`, `memory_search`, and `memory_get`, so custom recall tool names cannot be configured. I did not live-reproduce a custom-provider failure in this read-only review.

## Solution Assessment

No, not as written: the direction is narrow and useful, but the security filter is still a hand-copied subset that misses current core tools. The safer solution is to derive the deny policy from the core catalog or enforce a memory-only allow policy before the embedded recall run starts.
