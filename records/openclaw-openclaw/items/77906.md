## Summary

Keep open: the PR is directionally useful, but the hidden recall allowlist still permits plugin-backed core tools such as `code_execution` and `x_search`, and the real behavior proof covers the default memory-core path rather than the custom/LanceDB path this PR is meant to enable.

## What This Changes

The PR adds Active Memory `config.toolsAllow` resolution for memory-core, memory-lancedb, and custom recall tools, updates docs/changelog, and adds focused tests plus type-lint cleanup in adjacent tests.

## Best Possible Solution

Derive or centralize a memory-only recall tool policy that rejects all non-memory core and plugin-backed catalog tools, add regression coverage for `code_execution` and `x_search`, then land after live custom or LanceDB proof.

## Reproduction Assessment

Yes, at source level: current main passes a fixed Active Memory recall allowlist into `runEmbeddedPiAgent`, so custom recall tool names cannot be configured. I did not live-run a custom provider failure in this read-only review.

## Solution Assessment

No, not as written: the config surface is plausible, but the deny policy is hand-copied and misses current plugin-backed catalog tools. The safer solution is to derive the non-memory/core deny policy from the canonical catalog or enforce an explicit memory-only allow policy.
