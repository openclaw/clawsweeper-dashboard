## Summary

Keep open: the PR addresses a source-visible Active Memory limitation, but it has a blocking security concern around the configurable tool allowlist and lacks required real behavior proof; the maintainer label also requires explicit maintainer handling.

## What This Changes

The branch adds Active Memory `config.toolsAllow`, updates the recall prompt/default allowlist, treats recall failures as empty, and documents memory-core, LanceDB, and Lossless Claw examples.

## Best Possible Solution

Constrain the configurable allowlist to recall-safe tool entries and add contributor-supplied real output showing a custom memory provider recall path before maintainer review or merge.

## Reproduction Assessment

No for a live reproduction in this review. Source inspection on current main shows the recall sub-agent receives a hard-coded allowlist, so the custom-tool limitation is source-reproducible, but the PR body does not show a real after-fix run.

## Solution Assessment

No, not yet. Making the allowlist configurable is the right direction, but the implementation should prevent wildcard/core non-memory tools before passing it to `runEmbeddedPiAgent` and should be backed by real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Constrain toolsAllow to memory-safe tools:** `extensions/active-memory/index.ts:2449`
  - body: This passes `config.toolsAllow` directly into the embedded runner. That runner treats entries such as `*`, `group:plugins`, and known core tool names as normal allowlist names, so a hidden Active Memory sub-agent that receives untrusted conversation text can be configured with non-memory/code tools. Please reject wildcard/group/core tool entries or otherwise constrain this option to recall-safe plugin tools before handing it to the embedded run.
  - confidence: 0.82
