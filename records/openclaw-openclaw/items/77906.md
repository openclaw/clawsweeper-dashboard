## Summary

Keep open: the configurable recall-tool direction is useful, but the patch still lets omitted embedded-runner local tool names pass through the Active Memory memory-only allowlist, and the PR still lacks real custom-provider behavior proof.

## What This Changes

The PR adds Active Memory `config.toolsAllow`, passes it to the hidden recall sub-agent, updates prompt/docs/changelog behavior, and changes the default recall allowlist to `memory_search` plus `memory_get`.

## Best Possible Solution

Use a shared embedded-runner local tool guard or equivalent memory-safe helper before passing `toolsAllow`, then add real custom-provider output for the intended LanceDB/Lossless Claw path before maintainer review.

## Reproduction Assessment

Yes for a source-level reproduction: current main passes a fixed Active Memory recall allowlist, so a custom recall tool name cannot be configured. I did not establish a live custom-provider failure or after-fix run in this read-only review.

## Solution Assessment

No, not yet. Making the recall tools configurable is the right direction, but the memory-only boundary should block the embedded runner's full local tool set or use a shared helper, and the changed behavior still needs real custom-provider proof.
