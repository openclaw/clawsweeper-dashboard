## Summary

Current main still has a source-visible path where runtime conversation bindings can rewrite an inbound Telegram route to any bound session key, including a cron run key, and inbound session recording can then materialize that routed key. That matches the reporter's detailed trajectory evidence and conflicts with the documented main-DM and isolated-cron isolation contracts.

## What This Changes

Inbound direct messages can be routed into a closed isolated cron run session key, causing the cron prompt context to be reused and side effects to repeat.

## Best Possible Solution

Add a generic routing guard so inbound/current-conversation bindings cannot dispatch to or create sessions for `agent:*:cron:*:run:*` keys; stale cron-run bindings should be ignored or cleared and DMs should fall back to the normal main route.

## Reproduction Assessment

Yes, source-reproducible: current main can resolve a runtime conversation binding to a cron run session key, then record and dispatch the inbound turn against that key. I did not run a live Telegram account, but the reporter's trajectory logs and the inspected code path line up closely.

## Solution Assessment

Yes, the narrowest maintainable fix is to treat cron run session keys as non-routable for inbound current-conversation binding and fall back to normal channel routing. Adding a config option or Telegram-only special case would leave the generic binding boundary unsafe.
