## Summary

Keep open: current main and the latest release still surface generic context-overflow copy across the runner, precheck, shared formatter/sanitizer, and auto-reply fallback paths. The requested token/model/session diagnostics are available in logs or metadata in some paths, but they are not consistently shown to users.

## What This Changes

The issue asks for context-overflow user messages to include session/channel identity, token counts, model limit, model identity, overage, and concrete recovery actions.

## Best Possible Solution

Add one shared redaction-aware overflow diagnostic formatter and wire it through runner, precheck/mid-turn precheck, shared error formatting/sanitizing, and auto-reply fallbacks with safe omissions when details are unavailable.

## Reproduction Assessment

Yes, at source level: current main still returns fixed generic overflow strings while logging or storing provider/model/token/session diagnostics separately. I did not run a live oversized-model request in this read-only review.

## Solution Assessment

Yes, the requested direction is maintainable if implemented as a shared redaction-aware diagnostic formatter. A raw session-key dump or a runner-only message change would not be the best solution.
