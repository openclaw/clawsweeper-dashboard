## Summary

Keep open: current main and v2026.6.10 still reject the requested public `mediaLocalRoots` config keys, while Telegram outbound can only consume roots supplied by the existing media-access policy path. The remaining work is a security-sensitive product/config decision, not conservative cleanup.

## What This Changes

The issue asks OpenClaw to accept configured local media roots for Telegram outbound media instead of rejecting `mediaLocalRoots` during config validation.

## Best Possible Solution

Choose one canonical, security-reviewed local-media/filesystem-root contract, document it, and wire Telegram outbound through that shared policy with schema, docs, validation, and regression coverage.

## Reproduction Assessment

Yes, source-level: current strict schemas omit `mediaLocalRoots` where the reporter tried to configure it, and bundled channel validation reports unknown-key errors for unrecognized channel fields. I did not run a live config command because this read-only review avoids artifact-creating commands.

## Solution Assessment

No for an ad hoc Telegram-only field. The better solution is a shared, reviewed local-root authorization model that aligns `tools.fs`, outbound media, schema, docs, and tests.
