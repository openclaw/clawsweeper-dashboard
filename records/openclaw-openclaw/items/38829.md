- `impact:message-loss`: New persisted turns are omitted from the live TUI view until history is reloaded.
- `impact:session-state`: The TUI's displayed transcript falls behind the canonical active-session transcript.

## Summary

Keep open: current main and v2026.6.5 still expose Gateway transcript subscriptions but the TUI does not subscribe to or dispatch `session.message`, and the formally linked implementation PR remains open and conflicting.

## What This Changes

The issue asks for `openclaw tui` to display new messages from the web UI or other channels in real time instead of only after restart or history reload.

## Best Possible Solution

Refresh and land or replace the linked PR with a TUI-owned subscription lifecycle over the existing Gateway `session.message` API, including session filtering, unsubscribe-on-switch/exit, and real Gateway/TUI proof.

## Reproduction Assessment

Yes from source inspection. Current main loads durable history but has no TUI subscription or `session.message` dispatch path, matching the report that messages appear only after reopening or refreshing history.

## Solution Assessment

Yes in direction. The narrow maintainable fix is to consume the existing session-scoped Gateway transcript subscription API inside the TUI rather than widening unrelated chat broadcasts or adding a new product mode.
