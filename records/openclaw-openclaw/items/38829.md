- `impact:message-loss`: The TUI misses live messages from other surfaces until a manual reload or restart.
- `impact:session-state`: The visible TUI transcript can drift from the gateway session transcript even though history later contains the messages.

## Summary

Keep open: current main and v2026.5.12 still do not wire the TUI to the live `session.message` transcript stream, and an open linked PR is already the implementation candidate that closes this issue.

## What This Changes

The issue requests that `openclaw tui` display new messages from the web UI or other channels in real time instead of only after restart or history reload.

## Best Possible Solution

The TUI should add a scoped active-session subscription lifecycle over the existing `sessions.messages.subscribe` / `session.message` gateway contract, with filtering, session-switch cleanup, and active-stream deferral tests.

## Reproduction Assessment

Yes. The reporter gives concrete cross-surface steps, and current main shows the TUI loads history but has no client subscription helper or `session.message` dispatch path.

## Solution Assessment

Yes for the central bug. The narrow maintainable fix is to consume the existing gateway live transcript subscription contract rather than adding polling or a separate mirror path.
