## Summary

Keep open. Current main still lacks the TUI active-session transcript subscription and `session.message` routing, and the open fix PR at https://github.com/openclaw/openclaw/pull/73402 references this issue with closing syntax, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for `openclaw tui` to receive and render messages appended from other channels in real time instead of only after restart.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73402, or replace it with an equivalent scoped fix that adds TUI active-session subscription, `session.message` handling, session-switch unsubscribe, active-run deferral, and focused regression tests.

## Reproduction Assessment

Yes. The reporter gives concrete cross-surface steps, and current-main source inspection confirms the TUI loads history and handles `chat` events but does not subscribe to or route the existing `session.message` stream. I did not run the live Telegram/Web UI/TUI flow in this read-only review.

## Solution Assessment

Yes. Reusing the existing Gateway `sessions.messages.subscribe` and `sessions.messages.unsubscribe` contract in the TUI is the narrow maintainable fix; changing broad gateway chat fanout or adding a new mirror mode would be a separate product decision.
