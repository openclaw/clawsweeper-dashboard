## Summary

Keep open. Current main still lacks the TUI active-session transcript subscription and `session.message` routing needed for the reported real-time updates, while #73402 is the open unmerged implementation path.

## What This Changes

The issue asks for `openclaw tui` to receive live transcript updates from messages sent through other UI or channel surfaces instead of only showing them after restart.

## Best Possible Solution

Review and land the focused TUI fix in #73402, or keep this issue as the canonical bug if that PR closes unmerged.

## Reproduction Assessment

Yes. The report gives concrete steps, and current-main source inspection confirms the TUI loads history and handles chat events but never subscribes to the Gateway per-session transcript stream.

## Solution Assessment

Yes. A narrow TUI active-session subscription path is the best direction because Gateway already exposes and tests the required transcript event API; broad Gateway chat fanout changes would be less targeted.
