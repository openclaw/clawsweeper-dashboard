## Label Justifications

- `P2`: This is a bounded TUI regression with a restart workaround but meaningful real-time terminal usability impact.
- `impact:message-loss`: Messages appended from another surface are missing from the live terminal view even though they can appear after history reload.
- `impact:session-state`: The TUI’s visible active-session transcript can drift behind the Gateway’s canonical session transcript.

## Summary

Keep this issue open: current main still has the Gateway live transcript event contract, but the bundled TUI does not subscribe to or route `session.message` updates, and an open linked PR is the explicit implementation candidate.

## What This Changes

The issue requests that `openclaw tui` live-update when messages are appended to the active session from another UI or channel instead of requiring restart/re-entry to reload history.

## Best Possible Solution

Land or replace the linked TUI-side active-session transcript subscription so the TUI subscribes after connect/session switches, unsubscribes stale sessions, filters to the selected session, and refreshes history without clobbering active local streams.

## Reproduction Assessment

Yes, at source level: current main exposes and tests `session.message` transcript delivery, while the TUI has no subscription or handler for those events. I did not run a live Gateway/TUI reproduction in this read-only review.

## Solution Assessment

Yes in direction: consuming the existing per-session transcript subscription in the TUI is the narrow owner-boundary fix. The open linked PR remains the correct implementation vehicle unless maintainers replace it with an equivalent bounded fix.
