## Summary

Keep open. Current main and the latest shipped tag still lack the TUI live transcript subscription/routing path, and the remaining work is already represented by an open linked implementation PR with closing syntax.

## What This Changes

The issue asks for `openclaw tui` to display new messages from the web UI or other channels in real time instead of only after restarting the TUI.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73402, or replace it with an equivalent scoped TUI fix that consumes the existing `sessions.messages.subscribe` / `session.message` contract without broadening hidden external-channel chat fanout.

## Reproduction Assessment

Yes. The reporter gives concrete cross-surface steps, the docs define the TUI as showing the gateway-backed transcript, and current source has no TUI subscription/routing path for `session.message`; I did not run the live macOS flow.

## Solution Assessment

Yes. Reusing the existing gateway live transcript subscription API in the TUI is the narrow maintainable fix; the older general chat fanout approach is already superseded by the linked PR and related discussion.
