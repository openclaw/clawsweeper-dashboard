- `P2`: This is a real user-facing TUI live transcript bug with a limited surface and an open focused fix path, not a core runtime outage.
- `impact:message-loss`: New messages from other surfaces are not displayed in the active TUI until restart or manual history reload.

## Summary

Keep open: current main and latest release still lack the TUI live transcript subscription path, and the open linked implementation PR is the right candidate to review or replace before closing this issue.

## What This Changes

The issue asks for `openclaw tui` to display new cross-surface session messages in real time instead of only after restarting or reloading history.

## Best Possible Solution

Land or replace the scoped TUI fix that consumes the existing `sessions.messages.subscribe` / `session.message` gateway contract with active-session filtering, cleanup on switch/stop, and regression coverage.

## Reproduction Assessment

Yes, source-level. The reporter gives concrete cross-surface steps, and current main shows the TUI loads history and handles chat/agent websocket events but does not subscribe to or dispatch `session.message` transcript events.

## Solution Assessment

Yes for the central bug. The narrow maintainable fix is to wire the TUI to the existing gateway live transcript subscription contract rather than adding polling or a parallel message mirror.
