## Summary

Keep open: current main still subscribes the TUI to session events but does not consume `session.message`, the latest release has the same gap, and https://github.com/openclaw/openclaw/pull/96252 is the open closing PR rather than a merged fix.

## What This Changes

The issue reports that `openclaw tui` loads existing history but does not show live messages appended from another surface until the TUI is restarted.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/96252 for the narrow `session.message` TUI consumer, while leaving broader observe/mirror policy to https://github.com/openclaw/openclaw/issues/40678.

## Reproduction Assessment

Yes, at source level: Gateway emits displayable transcript appends as `session.message`, current TUI subscribes to session events, and the TUI dispatcher ignores that event. I did not run a live terminal reproduction in this read-only review.

## Solution Assessment

Yes for the narrow bug: consuming the existing Gateway `session.message` contract in TUI is the right layer. The open candidate PR needs proof and review rather than a separate fix lane.
