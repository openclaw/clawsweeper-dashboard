## Summary

Keep open: current main and the latest release still do not route live `session.message` transcript updates into the TUI, so the reported live-update gap remains. The earlier same-repo fix attempt was closed unmerged after security/proof concerns, leaving this as the canonical narrow bug to fix.

## What This Changes

The issue reports that `openclaw tui` loads historical messages but does not receive new active-session messages from Web UI or other channels until the terminal is reopened.

## Best Possible Solution

Land a narrow TUI fix that consumes Gateway `session.message` updates for the active session without widening hidden-run chat fanout, and leave broader opt-in cross-channel mirroring to https://github.com/openclaw/openclaw/issues/40678.

## Reproduction Assessment

Yes, at source level: Gateway emits displayable transcript appends as `session.message`, and the TUI currently ignores that event while latest release v2026.6.8 lacks the handler too. I did not run a live terminal reproduction in this read-only review.

## Solution Assessment

Yes for the narrow bug: the best fix is to use the existing Gateway session-message contract and active-session filtering rather than broad hidden-run chat fanout. If a scoped subscription lifecycle is added, it should include sanitized errors and focused tests for session switches, active runs, and duplicate prevention.
