## Summary

Keep open: current `main` still has the gateway transcript subscription contract but the TUI does not subscribe to it or route `session.message` events, and the linked implementation PR remains open and conflicting.

## What This Changes

The issue reports that `openclaw tui` loads existing history but does not show newly appended session messages in real time until the TUI is restarted.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/73402 with a scoped TUI fix that consumes `sessions.messages.subscribe` / `session.message`, filters to the active session, unsubscribes on session changes/stop, defers history reload during active local streams, and adds regression coverage.

## Reproduction Assessment

Yes, source-level. Current `main` loads TUI history and routes `chat` / `chat.side_result` / `agent` events, while the gateway exposes subscribed `session.message` transcript events that the TUI never subscribes to or dispatches.

## Solution Assessment

Yes for the central bug. The narrow maintainable fix is to wire the TUI to the existing gateway transcript subscription contract instead of adding polling, broad chat fanout, or a new mirror-mode feature.
