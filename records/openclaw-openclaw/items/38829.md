## Summary

Keep open: current main still exposes Gateway transcript events without the TUI subscribing to or dispatching `session.message`, and the linked fix at https://github.com/openclaw/openclaw/pull/73402 is still open, unmerged, and carrying security-boundary review concerns.

## What This Changes

The issue reports that `openclaw tui` loads historical chat messages but does not receive new messages from web or other channels in real time.

## Best Possible Solution

Land a refreshed, security-reviewed version of https://github.com/openclaw/openclaw/pull/73402 or a narrow replacement that subscribes the TUI to active-session `session.message` updates and verifies the live terminal path.

## Reproduction Assessment

Yes, at source level: Gateway exposes and emits `session.message`, while current main TUI does not subscribe to or dispatch that event. No live Gateway/TUI terminal reproduction was run in this read-only review.

## Solution Assessment

Yes for the owner boundary: the maintainable fix is to use the existing Gateway `sessions.messages.subscribe` and `session.message` surface in the TUI rather than widening unrelated Gateway chat fanout. The current linked PR follows that direction but remains unmerged and needs security/freshness follow-up.
