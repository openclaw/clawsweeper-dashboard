## Summary

Keep open. Current main and the latest shipped tag still lack a TUI subscription/routing path for live `session.message` transcript events, and https://github.com/openclaw/openclaw/pull/73402 is an open linked fix PR that references this issue with closing syntax.

## What This Changes

The issue asks for `openclaw tui` to show messages sent from the web UI or another channel in real time instead of only after restart/history reload.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73402, or replace it with an equivalent scoped fix that subscribes the TUI to active-session transcript updates and handles `session.message` without widening hidden external-channel chat fanout.

## Reproduction Assessment

Yes. The reporter gives concrete cross-surface steps, docs define the TUI as showing the gateway-backed transcript, and current-main source confirms the TUI loads history but does not subscribe to or route `session.message`; I did not run the live flow.

## Solution Assessment

Yes. Reusing the existing Gateway `sessions.messages.subscribe` / `session.message` contract in the TUI is the narrow maintainable fix; broadening chat-event fanout or adding a mirror-mode policy would be a separate product decision.
