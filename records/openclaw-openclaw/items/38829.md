## Summary

Current main still lacks the TUI-side transcript subscription and event-dispatch path required for external live updates. The issue should remain open with its formally linked implementation PR until that PR is refreshed, validated, and either merged or replaced.

## What This Changes

Make the TUI display messages appended to its active session by another client or channel without requiring a restart.

## Best Possible Solution

Refresh the linked PR against current main and land a TUI-owned active-session transcript subscription lifecycle with filtering, unsubscribe-on-switch or stop, and history reconciliation that does not disrupt an active local stream.

## Reproduction Assessment

Yes. Current main loads `chat.history` but neither subscribes to the active transcript nor handles `session.message`, directly matching the reported history-after-restart behavior.

## Solution Assessment

Yes in direction. The narrow maintainable fix is to consume the existing session-scoped transcript API inside the TUI rather than broadening unrelated hidden-run chat broadcasts; the linked branch still needs conflict resolution and current-main validation.
