## Summary

Current main still lacks the TUI live transcript subscription path, and an open maintainer PR already references this issue with closing syntax, so this should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that `openclaw tui` receive and render new messages from the active session in real time instead of only after restart or re-entry.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73402, or replace it with an equivalent scoped fix that subscribes the TUI to active-session transcript updates and handles `session.message` without widening hidden external-channel chat fanout.

## Reproduction Assessment

Yes. The reporter gives concrete cross-surface steps, and current-main source inspection confirms the TUI loads history and handles `chat` events but does not subscribe to or route the existing `session.message` stream; I did not execute the live flow.

## Solution Assessment

Yes. Reusing the existing Gateway `sessions.messages.subscribe` / `session.message` contract in the TUI is the narrow maintainable fix; broadening hidden-run chat fanout or adding a mirror mode would be a separate product decision.
