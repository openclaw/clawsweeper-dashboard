## Label Justifications

- `P2`: This is a normal-priority TUI/Gateway live transcript regression with a restart/reopen workaround and limited blast radius.
- `impact:message-loss`: Messages appended from another surface are missing from the live terminal view until the TUI reloads history.
- `impact:session-state`: The visible TUI session state can drift from the durable transcript for the active session.

## Summary

This issue is still necessary: current main exposes Gateway transcript subscriptions, but the TUI still only loads history on connect and dispatches `chat`, `chat.side_result`, and `agent` events, so `session.message` updates are not consumed. The open linked fix PR is a real implementation candidate, but it is unmerged and conflicting, so this report should stay open until that PR or a narrow replacement lands.

## What This Changes

Requests that `openclaw tui` render real-time messages appended from Web UI or other channels without requiring the user to exit and reopen the terminal.

## Best Possible Solution

Land a refreshed, security-reviewed version of https://github.com/openclaw/openclaw/pull/73402 or a narrow replacement that subscribes the TUI to active-session transcript updates and refreshes live history safely.

## Reproduction Assessment

Yes, at source level: Gateway exposes and tests `session.message` subscriptions, while current main TUI loads history on connect and never subscribes to or dispatches `session.message`. I did not run a live Gateway/TUI terminal reproduction in this read-only review.

## Solution Assessment

Yes for the owner boundary: the maintainable fix is to use the existing Gateway `sessions.messages.subscribe` and `session.message` surface in the TUI. The current linked PR follows that direction but still needs refresh, security review follow-through, and merge proof before it can close the issue.
