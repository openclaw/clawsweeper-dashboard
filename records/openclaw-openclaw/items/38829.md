## Label Justifications

- `P2`: This is a real user-facing TUI live transcript bug with limited surface area and an open focused fix path.
- `impact:message-loss`: Messages appended from another surface are absent from the active TUI until manual refresh or relaunch.
- `impact:session-state`: The visible TUI transcript can drift from the gateway session transcript even though stored history later contains the missing turns.

## Summary

Keep open: current main still exposes the gateway live transcript subscription contract, but the TUI does not subscribe to it or route `session.message` events. The open linked implementation PR is the right closure path, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests that `openclaw tui` render messages appended from the Web UI or another channel in real time instead of only after restart or history reload.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/73402 as a scoped TUI fix that consumes `sessions.messages.subscribe` / `session.message` with active-session filtering, unsubscribe cleanup, deferred history reload, and regression coverage.

## Reproduction Assessment

Yes, source-level. Current main loads TUI history and routes `chat`/`chat.side_result`/`agent` events, while the gateway exposes subscribed `session.message` transcript events that the TUI never subscribes to or dispatches.

## Solution Assessment

Yes for the central bug. The narrow maintainable fix is to wire the TUI to the existing gateway transcript subscription contract rather than adding polling, widening generic chat fanout, or bundling broader mirror-mode behavior.
