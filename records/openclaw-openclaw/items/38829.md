## Summary

Keep open. Current main and the latest shipped tag still lack a TUI `sessions.messages.subscribe`/`session.message` consumption path, and the open linked implementation PR is the right place to resolve this before the issue closes.

## What This Changes

The issue asks for `openclaw tui` to show messages sent from the Web UI or another channel in real time instead of only after restart or re-entry.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73402, or replace it with an equivalent scoped TUI fix that consumes the existing `sessions.messages.subscribe` and `session.message` contract.

## Reproduction Assessment

Yes. The reporter gives concrete cross-surface steps, and source inspection shows the TUI loads history but never subscribes to or dispatches gateway `session.message` events; I did not run the live macOS flow.

## Solution Assessment

Yes. Reusing the existing gateway live transcript subscription API is the narrow maintainable fix; the older broad chat-event approach in https://github.com/openclaw/openclaw/pull/45417 was superseded by the scoped subscription path.
