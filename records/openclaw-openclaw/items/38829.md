## What This Changes

The issue reports that `openclaw tui` loads historical messages but does not receive live transcript messages from the web UI or another channel until the TUI is restarted.

## Best Possible Solution

Review, update, and land or replace https://github.com/openclaw/openclaw/pull/73402 as the scoped TUI fix that consumes the existing gateway live transcript subscription contract.

## Reproduction Assessment

Yes, source-reproducible. The reporter gives concrete cross-surface steps, and current main shows the TUI loads history but has no live `sessions.messages.subscribe` client method or `session.message` dispatch path.

## Solution Assessment

Yes for the central bug. The narrow maintainable solution is to wire the TUI to the existing `sessions.messages.subscribe` / `session.message` gateway contract rather than inventing a separate polling or mirroring path.
