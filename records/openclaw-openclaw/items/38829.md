## Label Justifications

- `P2`: This is a real user-facing TUI live transcript bug with limited surface area and an open focused fix path.
- `impact:message-loss`: New messages from other surfaces are not displayed in the active TUI until restart or manual history reload.
- `impact:session-state`: The visible TUI transcript can drift from the gateway session transcript even though history later contains the missing messages.

## Summary

Keep open: current main and v2026.5.19 still lack the TUI live transcript subscription path, and the open closing PR at https://github.com/openclaw/openclaw/pull/73402 remains the active implementation candidate.

## What This Changes

The issue asks for `openclaw tui` to display messages appended from Web UI or other channels in real time instead of only after restart or history reload.

## Best Possible Solution

Land or replace the scoped TUI fix that consumes the existing `sessions.messages.subscribe` / `session.message` gateway contract with active-session filtering, cleanup on session switch/stop, and regression coverage.

## Reproduction Assessment

Yes, source-level. The reporter gives concrete cross-surface steps, and current main shows the TUI loads history and handles chat/agent websocket events but does not subscribe to or dispatch `session.message` transcript events.

## Solution Assessment

Yes for the central bug. The narrow maintainable fix is to wire the TUI to the existing gateway live transcript subscription contract rather than adding polling or a parallel message mirror.
