## Label Justifications

- `P2`: This is a real user-facing TUI live transcript bug with limited surface area and an open focused fix path.
- `impact:message-loss`: Messages appended from another surface are missing from the active TUI until manual refresh or relaunch.
- `impact:session-state`: The visible TUI transcript can drift from the gateway session transcript even though stored history later contains the missing turns.

## Summary

Keep open: current `main` and the latest tagged release still have the gateway live transcript stream, but the bundled TUI does not subscribe to it or route `session.message` events. The open linked implementation PR is the right fix path, not a reason to close this issue before it lands or is replaced.

## What This Changes

The issue asks for `openclaw tui` to receive and render new messages from the active session in real time instead of showing them only after a refresh or restart.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/73402 as a scoped TUI fix that consumes the existing `sessions.messages.subscribe` / `session.message` gateway contract with active-session filtering, unsubscribe cleanup, deferred reload semantics, and regression coverage.

## Reproduction Assessment

Yes, source-level. Current main loads TUI history and handles `chat`/`agent` websocket events, while the gateway already exposes subscribed `session.message` transcript events that the TUI does not subscribe to or dispatch.

## Solution Assessment

Yes for the central bug. The narrow maintainable solution is to wire the TUI to the existing live transcript subscription contract rather than adding polling, widening generic chat fanout, or bundling the broader mirror-mode feature into this repair.
