## Summary

Keep open. The reported zombie-loop bug is valid, but this PR still leaves an abort window during Pi `AgentSession.prompt()` preflight before `Agent.activeRun` exists, and the supplied proof is mock/unit output rather than real interrupt-mode behavior proof.

## What This Changes

The PR exports `makeAbortError`, adds an inline abort wrapper, pre-prompt guard, cleanup abort calls, test-harness helpers, and abort-before-prompt regression tests for the embedded Pi runner.

## Best Possible Solution

Make the embedded prompt-start boundary abort-aware before Pi can enter `Agent.prompt`, preserve the current abort-settle cleanup barrier, and add focused regression coverage plus real interrupt-mode proof.

## Reproduction Assessment

Yes. Current main is source-reproducible because `activeSession.prompt(...)` is evaluated before `abortable(...)` can reject a pre-aborted signal, and the linked issue supplies a deterministic pre-aborted `AbortSignal` test path.

## Solution Assessment

No. The pre-aborted guard is useful, but calling `Agent.abort()` from the wrapper does not cover the pre-activeRun window inside Pi `AgentSession.prompt()`; the fix should make that handoff abort-aware.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Make prompt preflight abort-aware:** `src/agents/pi-embedded-runner/run/attempt.ts:2359-2360`
  - body: The new abort handler calls `activeSession.agent.abort()`, but Pi `AgentSession.prompt()` can still be in async preflight before it calls `this.agent.prompt(...)` and creates `Agent.activeRun`. If the abort fires in that window, this wrapper rejects and cleanup aborts a non-existent active run, then Pi can later start the prompt anyway and recreate the escaped loop.
