## Summary

Keep open: current main and the latest release still lack the pre-prompt abort guard, so the central bug is not already implemented; the PR direction is narrow, but it remains blocked by mock-only proof and a conflicting base.

## What This Changes

The branch adds a pre-prompt `runAbortController.signal.aborted` guard before `activeSession.prompt()` and adds a focused Vitest file for pre-aborted signals.

## Best Possible Solution

Land a current-main version of the pre-prompt guard with regression coverage tied to the production path and redacted real interrupt-mode proof that no background LLM calls continue after abort.

## Reproduction Assessment

Yes, source-level: current main constructs `activeSession.prompt(...)` before `abortable()` can reject a pre-aborted signal, and the Agent runtime creates a separate active-run abort controller. I did not run a live interrupt-mode repro in this read-only review.

## Solution Assessment

Yes for the runtime direction: checking the run abort signal before calling `activeSession.prompt()` is the narrow owner-boundary fix for the pre-aborted evaluation-order window. A per-run LLM hard cap would be a separate product/config safety net, not required for this central bug fix.
