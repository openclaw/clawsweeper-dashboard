## Summary

Keep open. The PR addresses the already-aborted case, but it still leaves a concrete abort window after the new guard and before the underlying Pi `Agent.prompt()` creates `activeRun`; the existing discussion already identifies that remaining gap, and current `main` has not implemented a complete fix.

## What This Changes

This PR adds a pre-prompt abort guard, Pi agent abort/queue cleanup during abort handling and cleanup, test-harness methods, and a new abort-before-prompt regression test for the embedded attempt runner.

## Best Possible Solution

Make the actual prompt-start boundary abort-aware so no Pi agent run can be created after the OpenClaw run abort signal is set, keep the regression coverage for pre-aborted, pre-activeRun, and activeRun-existing abort cases, and add the required changelog credit before merge.

## Reproduction Assessment

Yes. Bug #74859 gives a deterministic pre-aborted-signal reproduction, and the remaining PR gap can be reproduced by delaying `AgentSession.prompt()` preflight after this PR's guard but before `Agent.prompt()`, then aborting the run.

## Solution Assessment

No. The pre-prompt guard is the right first layer, but the PR is still a partial fix because `agent.abort()` cannot cancel anything until `activeRun` exists; a safer solution is to check the run abort signal at the dependency's prompt preflight boundary, for example through `PromptOptions.preflightResult` or an upstream `AbortSignal` API.
