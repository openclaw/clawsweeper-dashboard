## Summary

Keep open: current main still leaves the Bedrock/provider registry failure class unresolved because stale external plugin packages can remain out of sync after direct core upgrades, `plugins update --all` still lacks the official-plugin sync path, and the open bulk-update PR covers only a narrower recovery path.

## What This Changes

The issue asks OpenClaw to fix a post-upgrade Bedrock provider registration failure when core and official/external plugin versions drift or when the loaded plugin's registration is not visible to model execution.

## Best Possible Solution

Keep Bedrock stream ownership in the Amazon Bedrock plugin, prove plugin registration and model execution share the same runtime registry, and resolve stale external plugin packages through a maintainer-approved generic upgrade-drift policy.

## Reproduction Assessment

Yes for source-level reproduction: current source still contains the registry-miss throw and plugin-owned registration boundary, and the discussion includes concrete Bedrock CLI failures. I did not run a live AWS/current-main Bedrock repro in this read-only review.

## Solution Assessment

No for the original Bedrock-only host shim as the default fix. The safer solution is a generic plugin upgrade/compatibility policy plus a focused proof that current Bedrock plugin activation registers the runtime API provider before model execution.
