## Summary

The issue remains necessary: v2026.6.5 and current main contain the same Bedrock registration code, while a fresh exact-release reproduction shows that a loaded bundled plugin still leaves the inference registry without `bedrock-converse-stream`. The original stale-plugin upgrade trap is useful context, but it does not explain the current bundled-plugin failure, so this needs a generic plugin-registration lifecycle fix rather than a provider-specific core compatibility shim.

## What This Changes

Fix Amazon Bedrock model execution when plugin activation succeeds but the runtime API registry still lacks `bedrock-converse-stream`.

## Best Possible Solution

Use one canonical runtime registry instance for plugin registration and inference, preserve Bedrock implementation ownership in the plugin, and add an integration regression test that activates the real provider plugin before resolving or invoking its API through the actual inference path in bundled and installed-plugin packaging modes.

## Reproduction Assessment

Yes. The issue contains exact v2026.6.5 commands and output showing a current bundled plugin loaded successfully before inference hits the matching registry-miss throw; source inspection confirms the registration and failure paths, although this review did not execute AWS credentials.

## Solution Assessment

No, the suggested Bedrock-specific core compatibility shim is not the best solution. The narrow maintainable fix is to repair the generic plugin-registration-to-inference registry boundary and prove it with real plugin activation coverage.
