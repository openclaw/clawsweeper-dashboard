## Summary

Keep this open: current main and v2026.6.6 contain the current Bedrock plugin registration path, but the original upgrade trap is not fully resolved because an already-installed older official Bedrock plugin can still satisfy its declared compatibility range while never registering `bedrock-converse-stream` into OpenClaw's runtime registry.

## What This Changes

The issue reports a Bedrock provider registration regression after the 2026.5.28 runtime refactor, covering both stale installed-plugin upgrades and a later loaded-current-plugin variant that still failed before reaching AWS.

## Best Possible Solution

Add an upgrade-safe path for official provider plugins: either update/repair stale installed Bedrock plugins before runtime use or fail early with a targeted diagnostic/doctor fix, while keeping the steady-state provider implementation plugin-owned.

## Reproduction Assessment

Yes for the original stale-plugin upgrade path from source: the old plugin relied on Pi and declares a compatibility range that newer hosts still satisfy, while current runtime throws when no OpenClaw API provider is registered. The later v2026.6.5 current-plugin variant has live comment evidence but was not rerun here against v2026.6.6/current main.

## Solution Assessment

No, the current explicit Bedrock registration is necessary but not sufficient for the already-installed-plugin upgrade trap. The best fix should be a generic official-plugin upgrade/compatibility repair or diagnostic path, not a permanent provider-specific runtime shim unless maintainers intentionally choose that compatibility tradeoff.
