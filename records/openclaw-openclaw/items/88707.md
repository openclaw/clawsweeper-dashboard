## Label Justifications

- `P1`: A supported provider is unusable for model execution on the latest release, with multiple users reporting the same hard failure.
- `impact:auth-provider`: The issue prevents Bedrock provider API routing and makes configured credential modes, including bearer-token use, inaccessible.

## Summary

The latest discussion shows the failure still occurs with the bundled, current-version Bedrock plugin on v2026.6.5, and current main has not changed the registration or inference-registry path involved. This is no longer safely characterized as only a stale-plugin upgrade mismatch: a loaded provider can expose models while its `bedrock-converse-stream` implementation remains unavailable to execution, so the issue remains necessary.

## What This Changes

Fix Bedrock model execution failing with `No API provider registered for api: bedrock-converse-stream` after upgrades and in current bundled installs.

## Best Possible Solution

Make targeted Bedrock activation register into the exact lifecycle-owned API registry consumed by local inference, then add execution-level coverage for both a current bundled plugin and an older installed-plugin upgrade; keep Bedrock stream implementation in the provider plugin rather than restoring a provider-specific core shim.

## Reproduction Assessment

Yes, at high confidence from source plus the current-release transcript: local inference with a loaded current `amazon-bedrock` plugin and a `bedrock-converse-stream` model fails at the registry lookup before AWS is called. A maintainer-controlled live rerun would still help isolate the precise loader variant.

## Solution Assessment

No, the suggested provider-specific core compatibility shim is not the best permanent fix. The maintainable solution is to repair targeted plugin activation or module resolution so the plugin-owned registration and inference share one registry, while maintainers separately decide whether older installed plugins need update or doctor handling.
