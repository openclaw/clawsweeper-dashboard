Keep open: current main and the latest release still sweep only cron `:run:` session keys, while isolated cron runs persist the stable base cron key that skips the retention cutoff. Two open closing PRs target the issue, so the remaining action is maintainer review of the lifecycle boundary rather than cleanup closure or a new duplicate fix lane.

## What This Changes

The issue asks for `cron.sessionRetention` to prune isolated-target cron base session rows and orphaned transcripts that the current `:run:`-only reaper skips.

## Best Possible Solution

Land one reaper-local lifecycle fix that prunes stale isolated or deleted-job base cron rows and archives replaced transcripts while preserving durable main and named cron sessions.

## Reproduction Assessment

Yes. Source inspection on current main shows isolated cron jobs persist stable base keys while the reaper only accepts `:run:` keys, and a later live repro comment shows the same lifecycle gap orphaning a transcript after a successful isolated cron run.

## Solution Assessment

Unclear. The diagnosis is solid, but the best fix needs maintainer agreement on how to distinguish disposable isolated or deleted-job base rows from durable main or named cron sessions before pruning them.
