## What This Changes

The issue reports that `cron.sessionRetention` does not prune stale isolated-target cron sessions because isolated jobs persist `agent:<id>:cron:<jobId>` base keys that the cron reaper never matches.

## Best Possible Solution

Land one refreshed reaper lifecycle fix that prunes stale isolated or orphaned base cron keys while preserving persistent main/named cron sessions and transcript archival safeguards.

## Reproduction Assessment

Yes. From source inspection, isolated cron jobs resolve to `agent:<id>:cron:<jobId>` base keys while the current reaper only accepts `isCronRunSessionKey`, so a stale isolated base entry will be skipped without needing a new feature path.

## Solution Assessment

Unclear as currently proposed. The diagnosis is valid, but the best fix should be reaper-local and distinguish disposable isolated/orphaned base rows from durable main or named cron sessions rather than simply broadening a predicate used by routing and other session behavior.
