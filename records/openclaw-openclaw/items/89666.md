## Summary

Keep open: current main and the latest release still sweep only cron session keys with a `:run:` segment, while isolated cron jobs persist state under the stable base cron key. Two open closing pull requests already target this issue, so the remaining maintainer action is to choose the safe base-session lifecycle boundary rather than queue another repair.

## What This Changes

Requests a cron retention lifecycle fix so isolated cron base sessions and replaced/orphaned transcripts do not survive indefinitely under `cron.sessionRetention`.

## Best Possible Solution

Choose and land one reaper-local lifecycle fix, likely the guarded approach in https://github.com/openclaw/openclaw/pull/90194 if maintainers accept its persistent-key safeguards, that prunes stale isolated or orphaned base cron rows and archives replaced transcripts without deleting durable cron session state.

## Reproduction Assessment

Yes. Source inspection on current main shows isolated cron state persists under the base key while the reaper only accepts `:run:` keys, and the live repro comment reports the same lifecycle gap after a successful isolated cron run.

## Solution Assessment

Unclear. The diagnosis is solid, but the best fix needs maintainer agreement on how to distinguish disposable isolated or orphaned base rows from durable main or named cron sessions before pruning them.
