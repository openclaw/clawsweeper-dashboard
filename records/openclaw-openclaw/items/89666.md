## Summary

Keep open: current `main` and the latest release still sweep only cron session keys with a `:run:` segment, while isolated cron jobs persist stable base cron keys. The detailed report, the later live repro comment, and the open closing PRs remain relevant until maintainers land or reject the base-session lifecycle fix.

## What This Changes

The issue reports that `cron.sessionRetention` does not prune isolated-target cron base sessions because the reaper filters only `:run:` cron session keys.

## Best Possible Solution

Land one reaper-local lifecycle fix that prunes stale isolated or orphaned base cron keys and archives replaced transcripts while preserving durable main or named cron sessions.

## Reproduction Assessment

Yes. Source inspection shows isolated cron jobs persist base keys while the current reaper only accepts `:run:` keys, and a later live repro comment shows the same lifecycle gap creating an orphan transcript after a successful isolated cron run.

## Solution Assessment

Unclear. The diagnosis is solid, but the best fix needs maintainer agreement on how to distinguish disposable isolated or orphaned base rows from durable main or named cron sessions before pruning them.
