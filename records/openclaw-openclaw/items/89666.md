## Summary

Keep open: current main still has a source-reproducible gap where cron retention only sweeps `:run:` cron session keys while isolated cron metadata is also persisted under the stable base cron key, so the documented `cron.sessionRetention` behavior is incomplete for isolated jobs.

## What This Changes

The issue reports that `cron.sessionRetention` does not prune isolated cron sessions because the reaper only targets cron session keys containing a `:run:` segment.

## Best Possible Solution

Implement a cron-owned retention path that distinguishes current job base metadata from expired run artifacts and deleted-job leftovers, then cover the stable base key, per-run key, and job-deletion cases in regression tests.

## Reproduction Assessment

Yes for a source-level reproduction: isolated cron state is persisted under a stable base key, while `sweepCronRunSessions` only prunes `:run:` keys. I did not run a live cron job in this read-only review.

## Solution Assessment

Yes, the best direction is a narrow cron/session retention repair rather than a new config option or generic session cleanup change. The fix should preserve active job base metadata and prune only expired run artifacts or deleted-job leftovers.
