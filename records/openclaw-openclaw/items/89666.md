## Summary

Keep open: current main still only prunes cron session keys with a `:run:` segment, while isolated-target cron jobs persist under the stable base cron key, and open fixing pull requests still need maintainer review before this issue can close.

## What This Changes

Make `cron.sessionRetention` prune stale isolated-target cron base sessions and deleted-job cron session leftovers instead of only `:run:` cron run keys.

## Best Possible Solution

Land one focused cron/session lifecycle fix, preferably the nuanced open path in https://github.com/openclaw/openclaw/pull/90194 or an equivalent maintainer-approved variant, then close this issue once current main actually prunes the affected isolated base entries safely.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: create or observe a stale `agent:<agentId>:cron:<jobId>` isolated cron base entry, then current main's reaper skips it because `isCronRunSessionKey` requires `:run:`. The reporter also supplied a live OpenClaw 2026.6.1 repro showing orphaned isolated cron transcript behavior after a run replaces the stable base pointer.

## Solution Assessment

No, current main is not the best end state because it documents isolated cron retention but only sweeps run descendants. The best fix is a focused lifecycle repair that prunes stale isolated/deleted-job base entries while preserving main-target and named persistent cron session keys.
