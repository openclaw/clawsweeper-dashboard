## Summary

Keep open: current main still has a source-backed Codex app-server path where a completed raw assistant response is collected but the embedded run waits for `turn/completed`, then records a timeout error and suppresses the generic assistant event that Discord and Telegram delivery depend on. Related issues cover adjacent timeout/stall classes, but this issue adds a narrower final-response-not-delivered case that is not implemented by the merged quiescent-turn fix.

## What This Changes

The issue reports Codex app-server embedded runs timing out after the model completes, leaving generated output undelivered to Discord and Telegram channels.

## Best Possible Solution

Add a narrow Codex app-server finalization fallback that delivers or synthesizes terminal completion from confirmed final raw assistant output when `turn/completed` is missing, while preserving status-only safeguards and existing `agentMessage` behavior.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current tests can send a current-turn assistant `rawResponseItem/completed` and withhold `turn/completed`, and current main returns an aborted timeout instead of emitting the assistant text. I did not rerun a live Codex app-server session in this read-only review.

## Solution Assessment

Unclear for the exact implementation, but the current behavior is not the best solution for this issue because collected final assistant text is suppressed by timeout handling before channel delivery. The safest repair should stay in the Codex app-server runner/projector path rather than changing Discord or Telegram send logic.
