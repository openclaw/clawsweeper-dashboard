## Summary

Keep open: the reported timeout still matches current main because raw assistant `rawResponseItem/completed` text is captured, but the embedded run is not released without `turn/completed` or an assistant `item/completed`. The existing test suite also locks in the current timeout shape, so this needs a focused Codex app-server fix rather than cleanup closure.

## What This Changes

The issue requests that Codex app-server embedded runs deliver completed model responses to channels when the last observed event is a raw assistant response item instead of timing out.

## Best Possible Solution

Teach the Codex app-server run-attempt completion logic to release and deliver after an idle final raw assistant response item, while preserving normal `turn/completed`, assistant `item/completed`, raw progress/status, and abort-marker behavior.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: emit a current-turn raw assistant `rawResponseItem/completed` without `turn/completed`, and current main records the text but times out instead of releasing the result. I did not run a live channel repro because this review is read-only and would require real Codex/channel credentials.

## Solution Assessment

Yes: the narrow maintainable fix belongs in the Codex app-server embedded-run completion/watchdog logic, not in Discord or Telegram delivery. The safer implementation should add explicit final-raw-assistant safeguards and regression coverage rather than broadening all raw response items into terminal events.
