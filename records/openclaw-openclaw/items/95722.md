## Summary

Keep open. The provider-id normalization is a real bug fix direction, but this patch lowercases the merge output before the source-managed SecretRef lookup has the same normalization, so mixed-case SecretRef-backed providers can lose marker enforcement and persist resolved credential material; the PR also lacks real generated-file proof.

## What This Changes

The PR lowercases and trims provider map keys in model-config merge helpers and adds regression tests for mixed-case provider maps and existing provider secret preservation.

## Best Possible Solution

Normalize provider ids consistently across merge and source-managed secret enforcement, then add a plan-level mixed-case `sourceConfigForSecrets` regression and redacted generated-file proof before merge.

## Reproduction Assessment

Yes, at source level. The PR normalizes the provider key to `openai` before `enforceSourceManagedProviderSecrets` does a trim-only lookup against the original source provider key, so mixed-case SecretRef source configs can miss marker reapplication.

## Solution Assessment

No. Normalizing provider keys at the merge boundary is directionally right, but the best fix must normalize the source-managed lookup and prove generated `models.json` behavior, not only helper-level map merging.
