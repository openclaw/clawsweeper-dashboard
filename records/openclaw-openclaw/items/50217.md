## Summary

Keep open: the underlying auth-profile persistence bug still appears valid on current main, but this PR is not safe to merge because the branch replays the fix through stale command and runner implementations that regress current auth, model-override, and runtime-auth behavior, and the external proof gate is still missing.

## What This Changes

The branch adds `finalAuthProfileId` to embedded run result types, rewrites `src/commands/agent.ts` from a re-export into a full command implementation, changes embedded runner auth handling, and adds command tests for session auth-profile persistence.

## Best Possible Solution

Port the winning auth-profile id through the current embedded-runner result contract into `updateSessionStoreAfterAgentRun`, preserving current ingress authorization, explicit model override handling, and runtime-auth wrapping, then add focused regression coverage and redacted real behavior proof.

## Reproduction Assessment

Yes at source level. Current main tracks `lastProfileId` and marks it successful, but does not expose it on `EmbeddedPiRunResult` or persist it to session `authProfileOverride`; I did not run a live multi-profile fallback scenario in this read-only review.

## Solution Assessment

No. The intended fix direction is right, but replacing current command and embedded-runner logic with stale full implementations is not the narrowest maintainable solution; a small port through the current result and session-store contracts is safer.
