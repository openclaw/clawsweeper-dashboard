## Summary

Keep open. The underlying auth fallback persistence bug is still source-reproducible on current main, but this PR should not merge as-is because it ports the fix through a stale copied command implementation that removes current model-override authorization and runtime-auth plumbing; contributor real behavior proof is also missing.

## What This Changes

The PR propagates a final auth profile id from embedded fallback runs and persists it into session `authProfileOverride`, while replacing the current command re-export with a full `src/commands/agent.ts` implementation and adding focused command tests.

## Best Possible Solution

Port the final winning auth-profile id through the current embedded-runner result contract into `updateSessionStoreAfterAgentRun`, preserving existing ingress authorization and runtime-auth behavior, then add focused regression coverage and redacted real behavior proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main tracks `lastProfileId` and marks it successful, but does not expose it in `EmbeddedPiRunResult` or write it into the session `authProfileOverride`. I did not run a live fallback because this review is read-only.

## Solution Assessment

No. The intended fix direction is right, but replacing the command wrapper with a stale full implementation is not the narrowest maintainable solution; the safer path is a small port through the current embedded result and session-store contracts.
