## Summary

Keep open: the underlying auth-profile persistence bug still appears present on current main, but this branch is not mergeable as-is because it reintroduces stale command/runtime code, weakens the ingress model-override authorization boundary, drops current override/runtime-auth behavior, and lacks contributor real-behavior proof.

## What This Changes

The PR adds a `finalAuthProfileId` result path for embedded auth-profile fallback and persists the winning profile into session `authProfileOverride`, with command regression tests.

## Best Possible Solution

Port the final winning auth-profile id through the current embedded-runner result contract into `updateSessionStoreAfterAgentRun`, preserving existing ingress authorization, runtime-auth, and model-override behavior, then add focused regression coverage and redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main tracks `lastProfileId` and marks it successful, but does not expose it on `EmbeddedPiRunResult` or persist it to session `authProfileOverride`.

## Solution Assessment

No. The intended fix direction is right, but replacing the command wrapper with a stale full implementation is not the narrowest maintainable solution; a small port through the current embedded result and session-store contracts is safer.
