## Summary

Keep open: the bug appears real and current main/latest release still lack final-profile persistence, but this branch is not mergeable as-is because it reintroduces stale command/runner code that drops current auth guards and runtime-auth handling, and it lacks real behavior proof.

## What This Changes

The PR propagates the final successful auth profile from embedded runs and persists it back to the session auth override after fallback rotation.

## Best Possible Solution

Port only the winning auth-profile id through the current embedded-runner result and session-store contracts, preserving current authorization/runtime-auth behavior, then add focused regression coverage and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main tracks lastProfileId and marks it successful, but EmbeddedPiRunResult and updateSessionStoreAfterAgentRun have no final-profile field/write; I did not run a live fallback because this review is read-only.

## Solution Assessment

No. The intended fix direction is right, but replacing current command and runner logic with a stale full implementation is not the narrowest maintainable solution; a small current-main port is safer.
