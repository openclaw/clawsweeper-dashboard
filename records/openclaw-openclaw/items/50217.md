## Summary

The underlying fallback-persistence bug still appears valid on current main, but this PR should not merge as-is: it is conflicting, missing after-fix real behavior proof, and its stale command/runner re-port drops current authorization and runtime-auth behavior.

## What This Changes

The PR adds `finalAuthProfileId` to embedded run results and persists it to session `authProfileOverride`, with command tests, while replacing the current `src/commands/agent.ts` re-export with a full implementation.

## Best Possible Solution

Port the final winning auth-profile id through the current embedded-runner result contract into `updateSessionStoreAfterAgentRun`, preserving existing ingress authorization and runtime-auth behavior, then add focused regression coverage and redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main tracks `lastProfileId` and marks it successful, but does not expose it in `EmbeddedPiRunResult` or persist it to session `authProfileOverride`.

## Solution Assessment

No. The intended fix direction is right, but replacing the command wrapper with a stale full implementation is not the narrowest maintainable solution; the safer fix is a small port through the current embedded result and session-store contracts.

  - confidence: 0.83
