## Summary

Keep open. Current main still lacks the final winning auth-profile propagation/session override persistence, but this branch is not mergeable as-is because it copies stale command/runner behavior, introduces auth/runtime regressions, and lacks contributor real-behavior proof.

## What This Changes

The PR adds a `finalAuthProfileId` result from embedded runs, persists that winning profile into the session auth override, preserves user-pinned override source, and adds command regression tests.

## Best Possible Solution

Port only the final winning auth-profile propagation and session override write onto current main while preserving ingress authorization, explicit override handling, runtime-auth wrapping, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main tracks the active profile during embedded auth rotation but does not expose it in the run result or persist it through session-store update. I did not run a live multi-profile fallback in this read-only review.

## Solution Assessment

No. The intended persistence fix is the right direction, but this branch copies stale command/runner code; the safer solution is a narrow current-main port that leaves existing auth, model override, and runtime-auth contracts intact.
