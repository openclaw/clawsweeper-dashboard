## Risks / Open Questions

- “Immutable” needs an explicit threat model: rejecting config APIs does not protect a policy file reachable through filesystem tools, exec, plugins, direct host access, or file replacement.
- Policy-over-user precedence can intentionally override existing settings, so invalid-policy behavior, startup failure policy, recovery, visibility, and upgrade semantics need a maintainer-approved compatibility contract.
- A separate overlay can become a competing source of truth unless runtime materialization, source-shaped persistence, includes, environment overrides, CLI mutations, gateway RPCs, doctor, and status all share one canonical policy model.
- The related hook proposals may provide useful enforcement context, but they do not replace an operator policy source and cannot by themselves cover config, prompt-file, filesystem, and non-tool mutation paths.
