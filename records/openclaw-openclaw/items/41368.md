## Summary

Keep open. Current main still lacks user-facing Feishu Bitable delete tools, and this PR is a useful broader implementation than the narrower single-record delete PR, but it needs maintainer review before merge because it exposes destructive data-deletion operations and still has small schema/response polish issues already noted in review.

## What This Changes

This PR adds Feishu Bitable tools in `extensions/feishu/src/bitable.ts` for deleting one record, deleting records in batch, and deleting a field.

## Best Possible Solution

Update the branch against current main, keep the implementation in the Feishu plugin if maintainers accept the destructive surface, align the new schemas and responses with existing Bitable tool conventions, add focused tests, and then either merge this broader PR or use it to supersede the narrower related Feishu delete-record work.

## Reproduction Assessment

Yes for the feature gap: current main can be checked by searching registered Bitable tools and seeing only get/list/create/update/create-field/create-app paths. Live Feishu deletion was not run in this read-only review, but the source-level gap and SDK contract are clear.

## Solution Assessment

Mostly yes, with maintainer review needed. The Feishu plugin is the right owner module and the PR follows the existing registration/account-routing/error-handling pattern, but destructive-tool policy, schema guidance, returned batch-delete details, tests, and branch freshness should be addressed before merge.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.83

Full review comments:

- **[P3] Preserve Bitable token lookup hints:** `extensions/feishu/src/bitable.ts:589-600`
  - body: The new delete schemas shorten `app_token` and `table_id` descriptions to generic labels, unlike the existing Bitable tools that tell the model to use `feishu_bitable_get_meta` and the URL table parameter. For destructive calls this guidance matters because it reduces guessed tokens or wrong-table operations; match the existing descriptions across the new schemas.
  - confidence: 0.86
- **[P3] Return the API's batch delete results:** `extensions/feishu/src/bitable.ts:475-478`
  - body: `batchDeleteRecords` reports `count` from the input array even though the Lark SDK returns `data.records` with the actual processed record IDs and deleted flags. Returning the API result, or deriving `count` from it with a fallback, would make the tool response consistent with the existing create/update helpers that surface `res.data`.
  - confidence: 0.82
