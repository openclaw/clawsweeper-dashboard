Yes for the PR defects: source inspection of PR head shows `— hello` becomes `, hello`, and `Hello,` loses its existing comma when comma mode runs. This is not a current-main bug because current main lacks the feature entirely.

## Solution Assessment

No. The requested direction is plausible, but this patch needs punctuation edge-case fixes, real behavior proof, and a port to current main's payload-plan architecture before it is the maintainable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Preserve commas that were already in the message:** `src/infra/outbound/payloads.ts:59`
  - body: When comma mode is enabled, this cleanup runs even if the input did not contain an em dash, so a message like `Hello,` is delivered as `Hello`. Only trim commas introduced by rewriting a trailing em dash, or return early when there is no em dash to rewrite.
  - confidence: 0.93
- **[P2] Handle leading em dashes without adding a comma:** `src/infra/outbound/payloads.ts:57`
  - body: Replacing `\s*—\s*` with `, ` turns leading em dashes into visible leading commas, so `— hello` becomes `, hello`. Add a leading-edge case or cleanup so the opt-in transform does not produce malformed outbound text.
  - confidence: 0.91
