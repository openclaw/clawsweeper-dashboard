## Summary

Keep open: current main still lacks the proposed outbound em-dash config, and the PR is not merge-ready because the opt-in transform has source-reproducible punctuation bugs, lacks external real behavior proof, and adds a new global config/product surface.

## What This Changes

The PR adds an opt-in `messages.outbound.text.emDash` config, generated config docs, resolver coverage, and outbound send/mirror/gateway/hook wiring to rewrite Unicode em dashes to commas.

## Best Possible Solution

Have maintainers decide whether a global outbound style transform belongs in core; if accepted, port it into the current payload-plan delivery flow with punctuation edge-case tests and regenerated config docs while preserving default behavior.

## Reproduction Assessment

Yes for the PR defects: source inspection of PR head shows leading em dashes become leading commas and messages that already end with a comma lose that punctuation in comma mode. This is not a current-main bug because current main does not implement the feature.

## Solution Assessment

No. The requested direction is plausible, but this patch needs punctuation fixes, real behavior proof, and a port to the current payload-plan architecture before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve commas that were already in the message:** `src/infra/outbound/payloads.ts:59`
  - body: When comma mode is enabled, this cleanup runs even if the input had no em dash, so a message like `Hello,` is delivered as `Hello`. Only trim commas introduced by rewriting a trailing em dash, or return early when there is no em dash to rewrite.
  - confidence: 0.93
- **[P2] Handle leading em dashes without adding a comma:** `src/infra/outbound/payloads.ts:57-59`
  - body: Replacing `\s*—\s*` with `, ` turns leading em dashes into visible leading commas, so `— hello` becomes `, hello`. Add a leading-edge case or cleanup so the opt-in transform does not produce malformed outbound text.
  - confidence: 0.91
