Keep open: current main still does not implement the proposed outbound em-dash rewrite, but the PR is not merge-ready because the branch has concrete punctuation defects, lacks contributor real behavior proof, and adds a new global config surface that needs maintainer product review.

## What This Changes

The PR adds an opt-in `messages.outbound.text.emDash` config plus resolver, outbound payload normalization, gateway send/mirror wiring, generated config docs, and tests to rewrite Unicode em dashes to commas before delivery.

## Best Possible Solution

If maintainers accept this as a core style policy, implement it in the current payload-plan outbound flow with punctuation edge-case coverage, regenerated config docs, unchanged default behavior, and redacted live send-path proof before merge.

## Reproduction Assessment

Yes for the PR defects: source inspection of the PR patch shows leading em dashes become leading commas and messages that already end with a comma lose punctuation in comma mode. This is not a current-main bug because current main does not implement the feature.

## Solution Assessment

No. The requested direction is plausible, but this patch is not the best maintainable solution until maintainers approve the product surface, the implementation is ported to current payload-plan architecture, punctuation bugs are fixed, and real behavior proof is supplied.

- **[P2] Preserve commas that were already in the message:** `src/infra/outbound/payloads.ts:59`
  - body: When comma mode is enabled, this cleanup runs even if the input had no em dash, so a message like `Hello,` is delivered as `Hello`. Only trim commas introduced by rewriting a trailing em dash, or return early when there is no em dash to rewrite.
  - confidence: 0.93
- **[P2] Handle leading em dashes without adding a comma:** `src/infra/outbound/payloads.ts:57-59`

  - confidence: 0.91
