## Summary

Closing is appropriate because the useful behavior is optional outbound style customization that current plugin hooks can support outside core, while the proposed core config patch is not merge-ready.

## What This Changes

The PR adds an opt-in `messages.outbound.text.emDash` setting and applies em-dash-to-comma rewriting across outbound delivery, gateway sends, mirrors, hook-mutated content, config metadata, generated docs, and tests.

## Best Possible Solution

Move this to a ClawHub code plugin using message hooks; only reopen core work if plugin authors identify a concrete missing extension API for mirrored transcript or delivery-pipeline consistency.

## Reproduction Assessment

Yes for the PR defects: source inspection of the branch helper shows leading em dashes become leading commas and existing trailing commas are stripped in comma mode. This is not a current-main bug because current main has no built-in em-dash rewrite setting.

## Solution Assessment

No for this PR as a core change. The maintainable path is a ClawHub code plugin unless maintainers explicitly decide OpenClaw core should own a broader outbound text policy API.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve existing trailing commas in comma mode:** `src/infra/outbound/payloads.ts:59`
  - body: When comma mode is enabled, this helper always runs `.replace(/,\s*$/g, "")`, so a message like `Hello,` with no em dash becomes `Hello` across transformed delivery paths. Return early when there is no em dash, or only trim punctuation introduced by a trailing em dash.
  - confidence: 0.93
- **[P2] Handle leading em dashes without emitting a comma:** `src/infra/outbound/payloads.ts:57-59`
  - body: Replacing `\s*—\s*` with `, ` turns leading em dashes into visible leading commas, so `— hello` becomes `, hello`. Add a leading-edge case or cleanup before this can be used for user-visible outbound text.
  - confidence: 0.91
