## Summary

## What This Changes

The PR adds an opt-in `messages.outbound.text.emDash` setting, resolver, outbound/gateway text normalization, generated config metadata, and tests for rewriting Unicode em dashes to commas before delivery.

## Best Possible Solution

If maintainers accept this core style policy, port it to the current payload-plan outbound flow, fix punctuation edge cases, regenerate config docs, and require redacted live send-path proof before merge.

## Reproduction Assessment

Yes for the PR defects: the proposed transform source shows `— hello` becomes `, hello` and `Hello,` becomes `Hello` in comma mode. This is not a current-main bug because current main does not implement the feature.

## Solution Assessment

No, not as-is. The direction may be acceptable after maintainer product approval, but this branch needs a current-main port, punctuation fixes, and real configured send-path proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

- **[P2] Preserve user-authored trailing commas:** `src/infra/outbound/payloads.ts:59`
  - body: When comma mode is enabled, this helper always runs `.replace(/,\s*$/g, "")`, so a message like `Hello,` with no em dash becomes `Hello` across every transformed delivery path. Trim only punctuation introduced by a trailing em dash, or return early when the input contains no em dash.
  - confidence: 0.93
- **[P2] Handle leading em dashes without adding a comma:** `src/infra/outbound/payloads.ts:57-59`
  - body: Replacing `\s*—\s*` with `, ` turns leading em dashes into visible leading commas, so `— hello` becomes `, hello`. Add a leading-edge case or cleanup so the opt-in transform does not produce malformed outbound text.
  - confidence: 0.91
