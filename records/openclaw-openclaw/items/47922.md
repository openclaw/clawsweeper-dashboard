## Summary

Keep this PR open, but do not merge it as-is. Current main still has no configurable outbound em-dash rewrite, so the contribution remains relevant, but the proposed helper corrupts punctuation in edge cases, the user-facing feature lacks a changelog entry, and the branch needs to be adapted to the current outbound payload-plan architecture.

## What This Changes

The PR adds an opt-in `messages.outbound.text.emDash` config with `preserve` and `comma` modes, wiring em-dash rewriting through outbound normalization, direct sends, gateway sends, mirroring, post-`message_sending` hook output, schema/docs metadata, and tests.

## Best Possible Solution

Land the feature only after the rewrite helper becomes content-preserving outside actual em-dash replacements, edge-case tests cover leading/trailing punctuation and unchanged text, a required changelog entry is added, and the implementation is rebased into the current payload-plan pipeline with clear hook-ordering semantics.

## Reproduction Assessment

Yes. The PR defects are reproducible by evaluating the helper in the branch: `rewriteEmDashesAsCommas("Hello,")` drops the existing comma, and `rewriteEmDashesAsCommas("— hello")` emits a leading comma artifact.

## Solution Assessment

No. An opt-in outbound transform can be a maintainable solution, but this patch is not the safest version until it avoids unrelated punctuation rewrites, handles edge positions, includes changelog coverage, and fits current main's outbound planning architecture.
