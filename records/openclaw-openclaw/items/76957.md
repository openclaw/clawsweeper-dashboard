## Summary

Keep open: current main still routes typed Control UI `/new` through `sessions.create` without the documented `command:new` hook path, and open PR #77376 is the closing implementation candidate.

## What This Changes

The issue asks to restore documented `command:new` and related hook emission when Control UI users type `/new` while keeping the fresh-session behavior.

## Best Possible Solution

Restore hook emission only for the Control UI `/new` flow, preserving ordinary SDK `sessions.create` and `parentSessionKey` behavior for programmatic child-session creation.

## Reproduction Assessment

Yes. The current-main path is source-reproducible: typed `/new` calls `onSlashAction("new-session")`, then `sessions.create`, and that create handler does not emit the documented command hook path.

## Solution Assessment

Yes. The explicit `emitCommandHooks` opt-in proposed by the open PR is the narrow maintainable fix; firing hooks for every `parentSessionKey` create would over-apply webchat command semantics to SDK callers.
