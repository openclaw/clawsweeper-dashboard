## Summary

Keep open: current main still routes Control UI typed `/new` through dashboard session creation, and `sessions.create` records the parent session but does not emit the documented `command:new` or reset hook path. The open closing PR #77376 is the active implementation vehicle.

## What This Changes

Restore Control UI typed `/new` hook emission so session-memory and custom `command:new`/`before_reset` hooks run against the previous session while preserving the fresh dashboard-session flow.

## Best Possible Solution

Land or revise PR #77376 so only the Control UI `/new` flow opts `sessions.create` into the established command/reset hook chain, leaving normal programmatic child-session creation unchanged.

## Reproduction Assessment

Yes, by source inspection: typed `/new` goes through `onSlashAction("new-session")` to `sessions.create`, while the documented `command:new` hook is only emitted by the reset path. A focused regression test should assert the missing hook on current main and the restored hook after the PR change.

## Solution Assessment

Yes, the explicit hook opt-in proposed by #77376 is the narrowest maintainable direction because `parentSessionKey` is already public session metadata and should not imply a webchat `/new` command. This issue should stay open until that PR is reviewed and merged or replaced.
