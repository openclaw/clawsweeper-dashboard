## Summary

Keep open: current `main` still routes Control UI typed `/new` through dashboard `sessions.create`, while the documented and tested hook-emitting path remains `sessions.reset(reason: "new")`. Related PR #49945 is TUI-only and #69599 explains why this Control UI route changed, so neither resolves this issue.

## What This Changes

Restore Control UI typed `/new` so it preserves fresh dashboard session creation while firing the documented `command:new` and session lifecycle hooks for the previous session.

## Best Possible Solution

Add a narrow parent-session lifecycle emission path for Control UI-created fresh sessions, with regression coverage for `command:new`, `before_reset`, session lifecycle hooks, and unchanged fresh dashboard session switching.

## Reproduction Assessment

Yes from source inspection: typed `/new` follows `handleSendChat` -> `onSlashAction("new-session")` -> `createSessionAndRefresh` -> `sessions.create`, while only `sessions.reset(reason: "new")` emits the documented hooks. I did not run live tests because this review was read-only and `pnpm` is unavailable on PATH.

## Solution Assessment

Yes: restoring hook emission at the Control UI parent-session boundary is the narrowest maintainable fix because it preserves the fresh-session behavior from #69599 while honoring the existing hook contract. A direct rollback to text-command dispatch would risk reintroducing the fresh-session bug.
