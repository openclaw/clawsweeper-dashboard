## Summary

Keep open: current main still routes Control UI typed `/new` through `sessions.create`, which does not emit the documented `command:new` hook path, and the linked closing PR is still open.

## What This Changes

Control UI typed `/new` should create a fresh session while firing the documented `command:new` and session-memory hook chain for the previous session.

## Best Possible Solution

Review and land PR #77376's explicit `emitCommandHooks` opt-in, or an equivalent narrow fix, then close this after merge verification.

## Reproduction Assessment

Yes, as a source-level reproduction path: `/new` goes to `onSlashAction("new-session")`, then `sessions.create`, and current `sessions.create` does not emit the documented command hook path. No live run was performed because this review is read-only.

## Solution Assessment

Yes, the corrected open PR direction is the narrow maintainable fix: an explicit hook-emission opt-in avoids treating every public `parentSessionKey` create as a webchat `/new` command.
