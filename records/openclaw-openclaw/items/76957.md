## Summary

Keep open: current main still has the reported Control UI `/new` source path that bypasses `command:new`, and an open fix PR (#77004) is the right place to resolve it before the issue closes.

## What This Changes

Control UI typed `/new` should preserve fresh-session creation while still firing the documented `command:new` hook behavior for the previous session.

## Best Possible Solution

Review and land the linked sessions.create fix, with regression coverage for Control UI `/new` delivering the parent-session hook contract before closing this issue.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction: Control UI `/new` reaches `sessions.create` with `parentSessionKey`, and that handler does not emit `command:new` on current main. I did not execute a live hook run in this read-only review.

## Solution Assessment

Yes for the core failure: the narrow maintainable fix belongs in the `sessions.create` parent-session path or an equivalent gateway lifecycle helper, not by reverting the fresh-session UI behavior. The linked PR should also be checked against the requested plugin lifecycle hooks before merge.
