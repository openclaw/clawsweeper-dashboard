## Summary

Keep open: current main still routes Control UI typed `/new` through `sessions.create`, and that handler stores parent linkage without emitting the documented `command:new` hook or reset lifecycle hooks. PR #76967 is useful confirming context, but it is closed unmerged.

## What This Changes

Restore Control UI typed `/new` so it fires documented `command:new` and reset/session lifecycle hooks for the previous session before creating or switching to a new session.

## Best Possible Solution

Restore the existing `/new` hook and lifecycle contract for Control UI sessions, with regression coverage proving `session-memory`-visible `command:new` behavior against the previous session.

## Reproduction Assessment

Yes, at source level: typed `/new` in Control UI goes through `onSlashAction("new-session")` to `sessions.create`, and the current create handler never emits `command:new`. I did not run a live UI repro, but the documented path and missing server-side emission are clear.

## Solution Assessment

No for current main: the current implementation bypasses the established hook-visible reset lifecycle. The safer repair is a scoped Control UI `/new` hook/lifecycle path, preferably reusing shared reset semantics or an explicit create option instead of broad parent-link side effects.
