## Summary

Keep open: current `main` still routes Control UI typed `/new` through `sessions.create` without a `command:new` hook opt-in or lifecycle-hook emission, and the open closing PR is still the right place to resolve it.

## What This Changes

The issue reports that Control UI typed `/new` creates a fresh session via `sessions.create` but no longer fires documented `command:new` hooks for session-memory and custom hook users.

## Best Possible Solution

Land PR #77376's explicit `emitCommandHooks` opt-in, or an equivalent narrow change, so only Control UI `/new` asks `sessions.create` to emit the documented hook chain.

## Reproduction Assessment

Yes, as a source-level reproduction path: typed Control UI `/new` goes to `onSlashAction("new-session")`, then `sessions.create`, and current `sessions.create` has no command hook emission. I did not run a live Control UI/session-memory flow because this review is read-only.

## Solution Assessment

Yes, the current open PR direction is the narrow maintainable fix: use an explicit hook-emission opt-in instead of treating every public `parentSessionKey` create as a webchat `/new` command.
