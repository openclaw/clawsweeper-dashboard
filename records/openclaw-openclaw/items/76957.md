## Summary

Keep open: current main still routes Control UI typed `/new` through `sessions.create` without hook emission, and the open closing PR has not merged yet.

## What This Changes

Restore Control UI typed `/new` hook/lifecycle emission so `session-memory` and custom `command:new` hooks run before switching to the new session.

## Best Possible Solution

Land PR #77376's explicit hook-emission opt-in, or an equivalent narrow fix that restores the documented `/new` hook contract without making every `parentSessionKey` create look like a webchat command.

## Reproduction Assessment

Yes, as a source-level reproduction path: typed Control UI `/new` reaches `onSlashAction("new-session")`, then `sessions.create`, and current `sessions.create` does not emit the documented `command:new` hook path. I did not run a live Control UI/session-memory flow because this review is read-only.

## Solution Assessment

Yes, the current open PR direction is the narrow maintainable solution: an explicit hook-emission opt-in restores the webchat `/new` contract without changing ordinary SDK/programmatic child-session creation semantics.
