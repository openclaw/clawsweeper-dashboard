## Summary

Keep open: current main still routes Control UI typed `/new` through dashboard session creation, and that `sessions.create` path has no documented `command:new` hook emission. The open closing PR #77376 is the right place to finish this rather than closing the issue early.

## What This Changes

Typing `/new` in Control UI should preserve fresh dashboard-session creation while firing the documented `command:new` and related reset lifecycle hooks for the previous session.

## Best Possible Solution

Land PR #77376, or an equivalent narrow fix that adds an explicit hook-emission opt-in for the Control UI `/new` create path and regression coverage for both opted-in and ordinary `sessions.create` calls.

## Reproduction Assessment

Yes. The source path is high-confidence: typed `/new` calls `onSlashAction("new-session")`, then `sessions.create`, and current `sessions.create` does not emit the documented command hook path.

## Solution Assessment

Yes. The explicit `emitCommandHooks` opt-in direction in PR #77376 is the narrow maintainable fix because using `parentSessionKey` alone would misclassify ordinary SDK or programmatic child-session creates as webchat `/new` commands.
