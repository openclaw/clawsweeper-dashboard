## Summary

Current main still has the reported regression: Control UI typed `/new` routes through the new-session creation path, while the `sessions.create` handler records `parentSessionKey` but does not emit the documented `command:new` hook. The related fix PR was closed unmerged, so this should stay open and is narrow enough for a repair lane.

## What This Changes

Typing `/new` in the Control UI should create a fresh session while still firing `command:new` hooks for the previous session.

## Best Possible Solution

Restore hook emission for the Control UI fresh-session path while preserving the true new-session behavior added for #69599, with regression tests around parent session context and session-memory compatibility.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: typed `/new` in Control UI calls `sessions.create` with `parentSessionKey`, and current `sessions.create` never emits `command:new`.

## Solution Assessment

Yes. The narrow maintainable repair is to keep the fresh-session create path and add explicit hook/lifecycle emission for the parent session; reverting to the old chat dispatch would risk regressing #69599.
