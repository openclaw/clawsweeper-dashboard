## Summary

Keep open: current main still routes Control UI typed `/new` into `sessions.create`, and that method currently lacks the `command:new` hook emission that the docs and reset path provide. PR #77004 is already open for this regression, so the issue should track that PR until it lands or is revised.

## What This Changes

Typing `/new` in Control UI should preserve fresh-session creation while firing `command:new` and expected lifecycle hooks against the previous session for `session-memory` and custom hooks.

## Best Possible Solution

Review and land or adjust PR #77004 so the gateway emits the expected hooks for parent-backed Control UI session creation while preserving independent fresh sessions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable `session-memory` or another `command:new` hook, type `/new` in Control UI, and the current code calls `sessions.create` without emitting `command:new`; no live UI smoke was run in this read-only review.

## Solution Assessment

Yes for the core direction: restoring hook emission in the gateway `sessions.create` parent-session path is narrower than reverting the fresh-session fix from #69599. The open PR still needs review for whether plugin lifecycle hooks are also part of the intended contract.
