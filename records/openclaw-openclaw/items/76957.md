## Summary

Current main still has the reported regression by source inspection: Control UI typed `/new` now reaches `sessions.create`, while the documented `command:new` hook path is only emitted by the reset service. An open fix PR already targets this exact issue, so this should stay open until that PR is reviewed and merged or replaced.

## What This Changes

Restore documented `command:new` hook delivery, including bundled `session-memory`, when Control UI typed `/new` creates a fresh session.

## Best Possible Solution

Land or adapt the narrow gateway-session fix so `sessions.create` with a parent session emits the documented parent-session lifecycle hooks before completing the fresh-session transition, with regression coverage and changelog entry.

## Reproduction Assessment

Yes for source-level reproduction: typed `/new` calls `onSlashAction("new-session")`, which calls `sessions.create` with `parentSessionKey`, and current `sessions.create` records that parent without emitting the documented `command:new` hook. I did not run a live UI hook test because this review was read-only.

## Solution Assessment

Yes, the narrow backend fix direction is appropriate: `sessions.create` is now the shared Control UI fresh-session path, so emitting against the parent there preserves the fresh-session UX while restoring the hook contract.
