## Summary

Keep open: current main still routes Control UI typed `/new` through `sessions.create`, and that method creates a child session without emitting the documented `command:new` hook or the reset lifecycle hooks that the normal reset path emits. The open #77004 PR is the active implementation path for this report.

## What This Changes

Typing `/new` in Control UI should preserve fresh dashboard-session creation while firing `command:new` and related parent-session lifecycle hooks.

## Best Possible Solution

Land or revise #77004, or an equivalent focused patch, so `sessions.create` with a parent session runs the parent-session hook lifecycle with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection of current main gives a high-confidence path: Control UI typed `/new` reaches `sessions.create`, and that create method does not emit the hooks that docs and the reset path require.

## Solution Assessment

Yes. The maintainable boundary is gateway-side handling for `sessions.create` when `parentSessionKey` is present, provided the fix restores the documented internal hook and any expected parent-session lifecycle hooks rather than changing the UI back to the old in-place reset behavior.
