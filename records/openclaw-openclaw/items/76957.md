## Summary

Keep open: current main still routes Control UI typed `/new` into `sessions.create` without the documented `command:new` hook path, and open PR #77376 is the active corrected fix with closing syntax, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Restore Control UI typed `/new` hook emission so `session-memory`, custom `command:new` hooks, and reset lifecycle hooks run against the previous session.

## Best Possible Solution

Land PR #77376 or an equivalent explicit opt-in hook path, then close this issue once the fix is merged and covered by regression tests.

## Reproduction Assessment

Yes, source-level. Current main routes Control UI typed `/new` through `sessions.create`, and that handler creates the child session without the documented `command:new` path used by `session-memory`.

## Solution Assessment

Yes, the active explicit `emitCommandHooks` approach is the narrow maintainable fix because only the Control UI command path should restore webchat hook delivery; ordinary SDK parent-session creates should remain hook-free unless explicitly opted in.
