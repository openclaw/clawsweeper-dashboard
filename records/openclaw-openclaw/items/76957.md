## Summary

Keep open: current main still routes Control UI typed `/new` through `sessions.create` without any hook-emission opt-in, while an open closing PR already carries the narrow fix.

## What This Changes

The issue asks for Control UI typed `/new` to emit the documented `command:new` and related lifecycle hooks before creating a fresh session.

## Best Possible Solution

Land PR #77376's explicit `emitCommandHooks` opt-in, or an equivalent narrow fix with regression coverage, then close the issue after merge verification.

## Reproduction Assessment

Yes. The source path is high-confidence: typed `/new` calls `onSlashAction("new-session")`, the helper calls `sessions.create`, and current `sessions.create` has no documented `command:new` hook emission path.

## Solution Assessment

Yes. The corrected open PR direction is the narrow maintainable fix because an explicit hook-emission opt-in restores the webchat `/new` contract without changing ordinary public `parentSessionKey` session creation semantics.
