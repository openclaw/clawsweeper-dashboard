## Label Justifications

- `P2`: This is a shipped, narrow core stability and safety bug in the extension exec surface, with source-level reproduction but limited blast radius.
- `impact:security`: The issue concerns an exec termination boundary where descendant processes may survive after OpenClaw reports the command killed.
- `impact:crash-loop`: Surviving descendant processes can create process-level availability and resource cleanup problems after timeout or abort.
- `maturity:stable`: The affected remote-execution/security-control behavior maps to the `gateway-runtime` M4 stable feature in the maturity shortlist.

## Summary

Keep open: current main and the latest release still route extension `api.exec()` timeout/abort/output-limit termination through direct child kills, while sibling exec paths already use process-tree cleanup.

## What This Changes

The issue asks for extension session `api.exec()` to terminate descendant processes as well as the direct child when a command is stopped by timeout, abort, or output cap.

## Best Possible Solution

Update the session extension exec backend to share the existing process-tree cleanup boundary used by sibling command paths, with focused regression coverage for timeout, abort, and output-limit kills.

## Reproduction Assessment

Yes at source level, but not with a live command in this read-only pass. Current main and `v2026.6.11` kill only the direct `api.exec()` child, while sibling exec paths use `killProcessTree()`.

## Solution Assessment

Yes: reusing the existing process-tree helper in `src/agents/sessions/exec.ts` is the narrow maintainable fix, without adding config or changing command parsing.
