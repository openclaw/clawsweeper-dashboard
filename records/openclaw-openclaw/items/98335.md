- `impact:security`: The report concerns a local command termination boundary where descendant processes can survive after OpenClaw reports the command killed.

- `maturity:stable`: The affected execution-safeguard behavior maps to the Gateway runtime M4 Stable surface covering approvals, remote execution, and security controls.

## Summary

Keep open: current main and the latest release still route extension api.exec timeout, abort, and output-limit cleanup through direct child-process kills, while an open linked PR is the current fix candidate.

## What This Changes

Extension api.exec should terminate descendant process trees on timeout, abort, and output-limit cleanup instead of only signaling the direct child process.

## Best Possible Solution

Land one canonical fix that reuses killProcessTree with the needed spawn boundary in src/agents/sessions/exec.ts, preserves the api.exec result contract, and adds focused coverage for timeout, abort, and output-limit cleanup.

## Reproduction Assessment

Yes at source level: current main and v2026.6.11 show api.exec cleanup signaling only the direct child, while sibling execution paths use process-tree termination. I did not run a live orphan-process repro in this read-only pass.

## Solution Assessment

Yes: reusing the existing process-tree helper in the extension exec backend is the narrow maintainable fix. The remaining work is to review and land the linked canonical PR or choose an equivalent focused implementation.
