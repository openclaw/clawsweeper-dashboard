## Summary

Keep open. Current main has evidence that the LaunchAgent embedded-token half was addressed, but the separate `channels list` path still enables provider usage by default and can either print or propagate raw usage fetch failures while listing auth profiles.

## What This Changes

The issue asks for `gateway install --force` to stop embedding `OPENCLAW_GATEWAY_TOKEN` in LaunchAgent metadata and for `channels list` to tolerate auth profiles without configured chat channels when provider usage fetching fails.

## Best Possible Solution

Land a focused CLI/provider-usage fix that treats usage enrichment as optional for `channels list` in human and JSON modes, with regression coverage for auth profiles, no configured chat channels, and a failing provider usage fetch.

## Reproduction Assessment

Yes. The report gives concrete commands, and current source gives a deterministic unit path: configure auth profiles with no chat channel accounts, have provider usage fetching reject with `TypeError: fetch failed`, then run `channelsListCommand` with default usage enabled.

## Solution Assessment

Yes for the remaining work. The narrowest maintainable fix is to make per-provider usage loading non-fatal for this command, either at the command wrapper or provider-usage aggregation boundary, while leaving the service-metadata fix intact.
