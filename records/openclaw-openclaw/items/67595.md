## Summary

Keep open. Current main has evidence that the LaunchAgent embedded-token half was addressed, but the separate `openclaw channels list` provider-usage path is still capable of surfacing or throwing raw usage fetch failures when listing auth profiles without chat channels.

## What This Changes

The issue requests that forced gateway service installs stop embedding `OPENCLAW_GATEWAY_TOKEN` in macOS LaunchAgent plists and that `openclaw channels list` succeed cleanly when only auth providers are configured.

## Best Possible Solution

Land a focused CLI/provider-usage fix that treats usage enrichment as optional for `channels list` in both human and JSON output, converts per-provider transport failures into non-fatal usage results or suppresses them for this listing surface, and adds regression coverage for auth profiles with no chat channels and a failing usage provider.

## Reproduction Assessment

Yes. The report provides concrete commands, and current source shows a deterministic unit-test path: configure auth profiles with no chat channel accounts, have the provider usage fetch reject with `TypeError: fetch failed`, then run `channelsListCommand` with default usage enabled in human and JSON modes.

## Solution Assessment

Yes for the remaining bug: the narrowest maintainable direction is to make provider usage loading non-fatal for `channels list`, either at the command wrapper or per-provider usage aggregation boundary. The already-landed service metadata repair should remain the service-side solution.
