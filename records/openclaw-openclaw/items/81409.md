## Summary

Keep open: current main has a real gap between config schema lookup and reload planning, but exposing reload impact is a new additive schema/RPC contract rather than a broken existing contract. The issue is recent, externally reported, has no protected labels, and no canonical duplicate or merged fix surfaced in targeted live searches.

## What This Changes

The issue asks OpenClaw to expose whether each config path hot-reloads, requires a gateway restart, or is a no-op before users or agents patch config.

## Best Possible Solution

Define one additive public contract for reload impact, then wire it from the existing reload planner into schema lookup or a dedicated config reload-rules surface with docs and regression coverage for restart, hot, none, plugin, channel, and default-restart paths.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: `config.schema.lookup` cannot return reload metadata, while `buildGatewayReloadPlan` separately classifies the same paths and defaults unmatched paths to restart.

## Solution Assessment

Unclear. Adding `reloadKind` to lookup hints is plausible, but maintainers should first choose whether reload impact belongs in per-field schema metadata, a separate reload-rules API, or docs-only guidance.
