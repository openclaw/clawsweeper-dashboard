## Summary

Keep open: current main has internal reload-planning rules and broad documentation, but the public config schema and lookup contract only expose schema/UI-hint metadata, so field-level restart/hot/no-op policy is still not surfaced. This is a useful additive API/docs feature rather than a broken existing contract.

## What This Changes

Expose each config path's reload policy, or an equivalent reload-rule lookup, through config schema/RPC/CLI metadata.

## Best Possible Solution

Add an additive protocol/API surface that reuses the canonical reload planner to expose a stable reload policy summary, then update config schema/lookup docs and focused regression tests.

## Reproduction Assessment

Yes, source-reproducible: current config.schema.lookup builds its payload from lookupConfigSchema, whose typed result and protocol schema have no reload-policy fields, while buildGatewayReloadPlan classifies the same paths internally. I did not run a live gateway because this review is read-only, but the code path is direct.

## Solution Assessment

Unclear until maintainers choose the API shape: adding field-level reloadKind to lookup/schema is plausible, but a separate reload-rule or reload-plan endpoint may be safer for plugin prefixes, wildcard paths, and default-restart behavior.
