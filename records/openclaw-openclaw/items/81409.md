Expose per-path hot/restart/no-op reload policy in config schema lookup metadata so tools can predict restart impact before applying config changes.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/81612; if accepted, keep reload metadata single-sourced from the existing reload planner and keep protocol/docs/client surfaces aligned.

## Reproduction Assessment

## Solution Assessment

Yes, subject to maintainer API acceptance. The linked PR direction is the narrowest maintainable path because it adds optional lookup metadata while reusing the existing reload-rule matcher instead of duplicating policy.
