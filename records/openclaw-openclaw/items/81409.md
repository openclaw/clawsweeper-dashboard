## Summary

Keep open: current main still has the same machine-readable schema gap. The reload planner knows restart/hot/no-op behavior, but `openclaw config schema` and `config.schema.lookup` only expose JSON Schema and UI hint metadata, so this remains a valid feature/API request rather than a cleanup close.

## What This Changes

The issue requests exposing config reload policy metadata so agents and operators can tell whether a config path hot-reloads, no-ops, or requires a gateway restart before applying a patch.

## Best Possible Solution

Add a stable reload-policy surface generated from the existing reload planner, with tests and docs keeping default restart behavior, explicit base rules, and plugin/channel prefixes aligned.

## Reproduction Assessment

Yes. Source inspection on current main shows a high-confidence path: `gateway.handshakeTimeoutMs` falls under the restart-backed `gateway` rule while schema and lookup payloads have no reload-policy field; the issue also has an isolated Docker reproduction on `openclaw@2026.5.7`.

## Solution Assessment

Unclear. Adding `reloadKind` to schema or lookup output is plausible, but the narrowest maintainable fix depends on whether maintainers want field-level JSON Schema metadata, UI hint metadata, a separate reload-rules endpoint, or docs plus API changes.
