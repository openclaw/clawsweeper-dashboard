## Summary

Keep open. Current main still does not expose reload/restart metadata through `config.schema.lookup` or the generated lookup payload, and an open linked PR already addresses this issue with closing syntax but has not merged.

## What This Changes

Expose restart, hot-reload, or no-op reload policy metadata for config schema consumers before they apply config changes.

## Best Possible Solution

Land an additive lookup metadata implementation that reuses the existing reload planner as the source of truth, with protocol validation, docs, tests, and client models kept aligned.

## Reproduction Assessment

Yes. Source inspection on current main shows lookup payloads cannot include reload metadata while the reload planner classifies the same paths, and the issue discussion includes a concrete Docker reproduction on `openclaw@2026.5.7`.

## Solution Assessment

Yes, the open linked PR is the right implementation path to review because it reuses the existing reload-rule matcher instead of duplicating policy. Maintainers still need to approve the additive Gateway protocol/API surface before this issue should close.
