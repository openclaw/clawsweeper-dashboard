## Summary

Keep open: current main has internal reload planning and coarse documentation, but the public config schema, lookup payload, and UI hint contract still do not expose machine-readable field-level reload policy. This is a useful additive feature/protocol request rather than a broken documented contract, and no duplicate or open fixing PR was found.

## What This Changes

Request exposing machine-readable config reload policy through config schema lookup, CLI/RPC schema metadata, or a dedicated reload-rules surface.

## Best Possible Solution

Define an additive reload-policy contract for config paths, backed by the same reload planner rules, with schema/lookup/CLI docs and regression tests kept in sync.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: current main has restart/hot/no-op planning in buildGatewayReloadPlan, but config.schema.lookup and ConfigUiHint only expose docs and validation metadata. I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear until maintainer product/API review: adding reloadKind to existing hints is plausible, but dynamic plugin prefixes and default-restart behavior may be cleaner as a dedicated reload-policy surface.
