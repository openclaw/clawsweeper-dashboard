## Summary

This issue is valid on current main and should stay open because the schema lookup/protocol surface still omits reload policy metadata while the reload planner already classifies the same paths. The linked open PR at https://github.com/openclaw/openclaw/pull/81612 is the active fix path and uses closing syntax, so this issue should remain open until that PR lands or maintainers reject that direction.

## What This Changes

Expose per-config-path hot-reload, restart-required, or no-op behavior through config schema lookup or an equivalent documented API surface.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/81612; if accepted, keep reload metadata single-sourced from the existing reload planner and update protocol docs/tests with the additive field.

## Reproduction Assessment

Yes. Current source shows lookup result types, handler wiring, and protocol validation cannot return reload metadata, while the reload planner classifies hot/none/restart behavior; the issue comments also include Docker and current-main failing lookup evidence.

## Solution Assessment

Yes, subject to maintainer API acceptance. The linked PR's approach of adding an optional lookup field sourced from the existing reload planner is narrower than duplicating a reload-rules table, while a separate endpoint remains the main alternative if maintainers do not want this on schema lookup.
