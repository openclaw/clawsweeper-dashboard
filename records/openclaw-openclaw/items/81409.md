## Summary

Keep open: current main and the latest release still omit reload policy from config schema lookup while the reload planner already has the classification, and https://github.com/openclaw/openclaw/pull/81612 is the active closing implementation candidate.

## What This Changes

Expose per-config-path reload policy metadata so schema lookup consumers can tell hot-reload, restart-required, and no-op fields before patching config.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81612 if maintainers accept the additive lookup field, keeping reload metadata single-sourced from the existing reload planner and closing this issue only after merge.

## Reproduction Assessment

Yes. Current source and protocol validators omit reload metadata from `config.schema.lookup` while `config-reload-plan.ts` classifies the same paths; the issue comments also include Docker repros for a shipped version and current main.

## Solution Assessment

Yes, subject to maintainer API acceptance. An optional `reloadKind` on `config.schema.lookup` backed by the existing reload-rule matcher is narrower than a duplicate reload-rules table and more useful to agents than docs-only guidance.
