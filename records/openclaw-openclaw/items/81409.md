## Summary

Keep open. Current main has a separate reload planner and broad docs, but the config schema, `uiHints`, and `config.schema.lookup` protocol only expose validation and documentation metadata, so surfacing reload policy would be a new additive API/product decision rather than a broken existing contract.

## What This Changes

Expose each config path's hot-reload, restart-required, or no-op policy through schema metadata or a related CLI/RPC surface.

## Best Possible Solution

Add an additive, typed reload-policy surface backed by the reload planner and plugin reload registrations, with protocol docs and focused tests so agents can check impact before writing config.

## Reproduction Assessment

Yes for the reported contract gap. Source inspection shows current main's hint type, protocol validator, and lookup response do not carry `reloadKind` or `requiresRestart`, while reload behavior is computed separately by `config-reload-plan.ts`.

## Solution Assessment

Unclear until maintainers choose the API shape. The request is valid, but adding `reloadKind` to every hint, adding a path lookup/rules RPC, or documenting fallback semantics have different protocol and accuracy tradeoffs.
