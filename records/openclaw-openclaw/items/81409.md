## Summary

Keep open: current main still lacks reload-policy metadata in config schema lookup/protocol results, while the reload planner already knows restart/hot/no-op behavior. The issue is paired with an open closing PR, so it should stay open until that PR is reviewed, rebased if needed, and either landed or explicitly closed.

## What This Changes

Expose per-path hot/reload/restart metadata for config schema consumers so operators and agents can know config-change impact before applying patches.

## Best Possible Solution

Add optional lookup-level reload metadata sourced from the existing reload planner, keep the full JSON schema contract unchanged unless maintainers explicitly choose otherwise, and update protocol validation, generated clients, docs, and focused tests together.

## Reproduction Assessment

Yes. Source inspection on current main shows lookup results cannot include reload metadata, while the reload planner can classify the same paths; the issue comments also include Docker reproduction evidence from a released build and a current-main source check.

## Solution Assessment

Yes, subject to maintainer API approval. Reusing the existing reload-rule matcher for optional lookup metadata is the narrowest maintainable direction because it avoids duplicating reload tables and gives agents machine-readable impact before writes.
