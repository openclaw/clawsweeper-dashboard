## Summary

Keep this PR open, but the current patch has a blocking base-path routing regression: it prefixes managed outgoing image API fetches with the Control UI base path even though the gateway only routes those media endpoints at the root `/api/chat/media/outgoing/...` path.

## What This Changes

The PR adds Control UI/WebChat generated-image frames with Open/Download/Copy actions, managed thumbnail previews, a thumbnail sizing config key, and gateway transcript/media handling for generated image replies.

## Best Possible Solution

Keep the generated-image action feature, but make managed outgoing image API URLs and server routing agree for both root and base-path Control UI deployments, with a regression test that exercises a non-empty `gateway.controlUi.basePath`.

## Reproduction Assessment

Yes, for the introduced base-path regression by source inspection: a base-path Control UI will build `/openclaw/api/chat/media/outgoing/...`, while the gateway route and handler only match `/api/chat/media/outgoing/...`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The overall feature direction is plausible, but the current implementation should not merge until managed media URLs and gateway routing are consistent for base-path and root-path deployments.
