## Summary

Keep open: the latest PR head still introduces a managed-image auth bypass and leaves full-size Open/click actions on a direct URL path that cannot carry Control UI auth headers. The real behavior proof is now supplied, but the patch is not correct or merge-ready.

## What This Changes

The PR adds Control UI/WebChat generated-image frames and actions, thumbnail/full managed-media routing and config, Codex media timeout/normalization changes, tests, and changelog entries.

## Best Possible Solution

Keep the generated-image affordances only after managed-media bytes remain behind Gateway auth, full-size actions use an auth-preserving blob or ticketed URL flow, and landing is coordinated with the related Codex media-normalization work.

## Reproduction Assessment

Yes. Source inspection of the current PR head shows the requester-session branch returns image bytes before Gateway auth, and the added test explicitly locks in that no-auth path.

## Solution Assessment

No. The safer maintainable path is to authenticate the Gateway request first, then apply requester-session ownership checks, and to make Open/click use an authenticated blob or short-lived ticket rather than direct navigation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep managed image reads behind Gateway auth:** `src/gateway/managed-image-attachments.ts:1033-1042`
  - body: This requester-session shortcut still runs before authorizeGatewayHttpRequestOrReply. Because the accepted session key is encoded in the URL path, a client with the media URL can echo it in x-openclaw-requester-session-key and fetch the image without normal Gateway auth.
  - confidence: 0.95
- **[P2] Open managed images through authenticated blobs:** `ui/src/ui/chat/grouped-render.ts:1166`
  - body: The Open action still navigates directly to the full managed-image URL. That navigation cannot attach the bearer token or requester-session header, so token-auth Control UI sessions will fail once this route stays authenticated; use an auth-preserving blob or ticketed URL flow.
  - confidence: 0.88
