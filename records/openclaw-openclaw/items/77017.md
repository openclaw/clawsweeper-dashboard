## Summary

Keep this PR open: the implementation is useful, but the new managed-image Open action opens the protected full-image URL in a fresh tab without the auth headers that the same endpoint requires, so it will fail for token-authenticated Control UI sessions.

## What This Changes

This PR adds generated-image actions, thumbnail preview/full-image routing, managed image frame sizing, related config schema entries, Codex media timeout/trust handling, changelog entries, and regression tests across Control UI/WebChat and Gateway media paths.

## Best Possible Solution

Keep the thumbnail/full-image split, but make Open use an auth-capable full-image path, preferably fetch-to-blob with the existing managed-image headers or a short-lived signed full-image URL, then cover token-auth managed images in UI tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: render a managed generated image in a token-auth Control UI session and click Open. The button navigates to the full route without headers, while the route requires either requester-session ownership or normal gateway auth.

## Solution Assessment

No. The PR’s preview/download/copy direction is maintainable, but the Open action needs to reuse an authenticated blob/signed-URL path instead of direct navigation to a protected endpoint.
