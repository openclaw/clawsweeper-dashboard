## Summary

Keep open; the latest head still serves managed image records before Gateway auth and opens full managed images through plain navigation, so the patch remains blocked and also lacks attached real behavior proof.

## What This Changes

This PR adds generated-image chat actions, thumbnail/full-image routing, a Control UI thumbnail-size config key, Codex timeout/media normalization changes, tests, and changelog entries.

## Best Possible Solution

Keep the generated-image affordances only after managed-media reads stay behind Gateway auth and full-size actions use an auth-preserving blob or ticketed URL flow, with contributor proof attached.

## Reproduction Assessment

Yes. Source inspection of the latest PR head shows the no-auth requester-session path can serve a managed image before Gateway auth, and the PR test explicitly expects that bypass.

## Solution Assessment

No. The maintainable direction is to keep server-side Gateway auth first, then apply requester-session ownership checks for non-privileged users and make Open/click use an authenticated blob or short-lived ticketed URL.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep managed image reads behind Gateway auth:** `src/gateway/managed-image-attachments.ts:1033-1043`
  - body: This requester-session shortcut runs before authorizeGatewayHttpRequestOrReply. Because the accepted session key is encoded in the URL path, a client with the media URL can echo it in x-openclaw-requester-session-key and fetch the image without normal Gateway auth.
  - confidence: 0.95
- **[P2] Open managed images through authenticated blobs:** `ui/src/ui/chat/grouped-render.ts:1166`
  - body: The Open action navigates directly to the full managed-image URL. That navigation cannot attach the bearer token or requester-session header, so token-auth Control UI sessions will fail once this route remains authenticated.
  - confidence: 0.88
