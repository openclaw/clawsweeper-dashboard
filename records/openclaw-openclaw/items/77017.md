## Summary

Keep open: the PR has useful generated-image UI work and supplied real behavior proof, but the current head still introduces a managed-image auth bypass and a direct Open path that conflicts with authenticated media delivery.

## What This Changes

The branch adds Control UI/WebChat image actions, managed full/thumbnail media routing and config, Codex media timeout/normalization changes, tests, and changelog entries.

## Best Possible Solution

Keep the image affordances only after managed-media bytes remain behind Gateway auth and Open/click paths use auth-preserving blob or ticketed full-image access.

## Reproduction Assessment

Yes. Source inspection of PR head shows the requester-session branch serves before authorizeGatewayHttpRequestOrReply, and the added test expects HTTP 200 with denyAuth.

## Solution Assessment

No. The feature direction is reasonable, but the maintainable fix is to authenticate managed-media HTTP requests before requester-session ownership checks and make Open/click use an authenticated blob or short-lived ticket.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep managed image reads behind Gateway auth:** `src/gateway/managed-image-attachments.ts:1033-1042`
  - body: This requester-session shortcut runs before authorizeGatewayHttpRequestOrReply. Because the accepted session key is encoded in the URL path, a client with the media URL can echo it in x-openclaw-requester-session-key and fetch the image without normal Gateway auth.
  - confidence: 0.95
- **[P2] Open managed images through authenticated blobs:** `ui/src/ui/chat/grouped-render.ts:1166`
  - body: The Open action still navigates directly to the full managed-image URL. Direct navigation cannot attach the bearer token or requester-session header, so token-auth Control UI sessions will fail once this route stays authenticated; use an auth-preserving blob or ticketed URL flow.
  - confidence: 0.88
