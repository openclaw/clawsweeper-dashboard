## Summary

Keep open: the latest PR head still serves managed image bytes before normal Gateway HTTP auth, and the managed-image Open/click path still uses plain navigation that cannot carry the required auth headers.

## What This Changes

This PR adds Control UI/WebChat generated-image open, download, and copy actions, thumbnail/full-image routing, config schema updates, Codex media/timeout tweaks, diagnostics handling, tests, and changelog entries.

## Best Possible Solution

Keep the image actions and thumbnail split, but preserve Gateway auth before serving managed media and use an auth-preserving blob or ticketed URL for full-size managed-image Open/click actions.

## Reproduction Assessment

Yes. Source inspection of the latest PR head shows a no-auth request can echo the session key from the URL in x-openclaw-requester-session-key and reach serveManagedOutgoingImageRecord before authorizeGatewayHttpRequestOrReply; the added test expects that behavior.

## Solution Assessment

No. The maintainable fix is to keep server-side Gateway HTTP auth first, then check requester-session ownership for non-privileged users, and make UI Open/click use the same authenticated fetch/blob path or a short-lived signed URL.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep managed image reads behind Gateway auth:** `src/gateway/managed-image-attachments.ts:1034-1044`
  - body: This requester-session shortcut runs before authorizeGatewayHttpRequestOrReply. Because requesterOwnsManagedImageSession accepts requesterSessionKey === targetSessionKey and that target value is already in the URL path, a client with the media URL can echo it in x-openclaw-requester-session-key and fetch the image without normal Gateway auth. Move auth before this branch and change the no-auth same-session test to expect rejection.
  - confidence: 0.94
- **[P2] Open managed images through authenticated blobs:** `ui/src/ui/chat/grouped-render.ts:1166`
  - body: The Open action passes the managed full-image URL to openExternalUrlSafe, and the image click path uses the same plain navigation. That navigation cannot attach the bearer token or requester-session header, so token-auth Control UI sessions will fail once the server route stays authenticated. Reuse the authenticated fetch/blob flow or a short-lived ticketed URL for Open/click.
  - confidence: 0.86
