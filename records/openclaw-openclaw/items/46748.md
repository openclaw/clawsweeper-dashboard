## Summary

Close: current main and v2026.6.6 implement the requested CLI behavior. `openclaw logs` now prefers the structured `time` field, renders human-readable output in the host local timezone by default, and keeps UTC available through `--utc`.

## What This Changes

The issue asks for `openclaw logs` to display local-time timestamps by default instead of UTC, using the local `time` field or an equivalent CLI rendering path.

## Best Possible Solution

Keep the shipped v2026.6.6 behavior: local-time human log output by default, `--utc` for UTC, and a fresh issue only if current console or Docker timestamps still fail with current-release evidence.

## Reproduction Assessment

Yes, at source level: older shipped code defaulted human-readable `openclaw logs` output to UTC unless `--local-time` or valid `TZ` was present, while current v2026.6.6 source and tests show default local output. I did not run a live gateway because this cleanup review is read-only.

## Solution Assessment

Yes. The current implementation fixes the display layer by preferring the structured `time` field and making local-time output the default without adding a new config surface; UTC remains available through `--utc`.
