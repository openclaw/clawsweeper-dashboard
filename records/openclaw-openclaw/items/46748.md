## Label Justifications

- `P2`: Log timestamp display affects debugging for non-UTC users but does not block core runtime availability.
- `impact:other`: The issue concerns user-facing log interpretation and troubleshooting accuracy outside the more specific impact taxonomy.

## Summary

Current main and the latest release already render `openclaw logs` timestamps in local time by default, keep `--local-time` accepted, and use `--utc` for UTC output. The follow-up console/TZ concern is also covered by the current shared timestamp formatter.

## What This Changes

The issue requests that CLI and gateway log timestamp display use the user's local timezone by default instead of UTC.

## Best Possible Solution

Keep the shipped local-time default with `--utc` as the explicit UTC opt-in, and ask for a new report only if v2026.6.6 or newer still shows UTC unexpectedly.

## Reproduction Assessment

No for current main: source, tests, docs, and the latest release code render local timestamps by default and reserve UTC for `--utc`. The original v2026.3.13 behavior was well described, but it no longer reproduces on the reviewed source.

## Solution Assessment

Yes: the current implementation fixes the narrow CLI behavior by preferring the log `time` field, formatting local time by default, preserving `--local-time`, and adding `--utc`. The shared console timestamp path also uses the local formatter, so a new timezone config flag is not needed for this issue.
