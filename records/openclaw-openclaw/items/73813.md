## Summary

Keep open. Current main still has the reported cron observability gap: the embedded runner marks `sessions_yield` as paused/yielded metadata, but isolated cron finalization still records a terminal `ok` outcome and duration from the first yielded run, and no open PR appears to cover this issue.

## What This Changes

The issue asks cron run history for isolated `sessions_yield` workflows to reflect the full parent-plus-subagent execution instead of marking the first yielded turn `ok` with a short duration.

## Best Possible Solution

Cron bookkeeping should represent yielded isolated orchestration as nonterminal until resumed parent and descendant work reach a real terminal outcome, or expose an explicit backward-compatible yielded/pending signal if terminal status must remain unchanged.

## Reproduction Assessment

Yes. The issue gives concrete isolated cron steps, and source inspection maps the same path: an embedded `meta.yielded` result can still flow into terminal cron `ok` bookkeeping before descendant work drains, especially without requested delivery.

## Solution Assessment

Unclear. Deferring `finished` best matches the reported status and duration semantics, while adding yielded metadata first is smaller and safer for consumers but only partially fixes observability.
