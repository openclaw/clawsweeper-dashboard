## Summary

Keep open. Current main still appears to finalize isolated cron runs as terminal `ok` when the embedded runner pauses on `sessions_yield`; the newer yielded/paused embedded metadata is available, but the cron finalizer and run-log contract do not consume it, so the reported misleading status and duration remain a valid observability bug.

## What This Changes

The issue requests correct cron run status and duration semantics when an isolated cron agent turn yields to a spawned subagent and later resumes.

## Best Possible Solution

Cron run lifecycle should represent real terminal orchestration state, preferably by deferring terminal `finished` bookkeeping until the yielded parent and descendants actually finish, with additive yielded metadata only if compatibility requires an interim step.

## Reproduction Assessment

Yes. The issue gives concrete isolated-cron steps, and current source still maps a yielded embedded run into terminal cron `ok` bookkeeping without inspecting the yielded metadata.

## Solution Assessment

Unclear. Deferring `finished` best matches the reported status and duration semantics, while additive yielded metadata is smaller but only partially fixes metrics; maintainers should choose the cron event contract first.
