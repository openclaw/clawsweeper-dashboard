## Summary

Keep open. Current main still has the reported cron observability gap: yielded embedded runs expose paused/yielded metadata, but cron run status and finished timing still resolve from the first isolated-run result rather than a real post-subagent terminal outcome.

## What This Changes

The issue asks isolated cron run status and duration to reflect the full `sessions_yield` orchestration, including spawned subagent completion and parent resume.

## Best Possible Solution

Cron bookkeeping should represent yielded isolated orchestration as nonterminal until the resumed parent and descendant subagents reach a real terminal result, with any transitional yielded metadata kept backward-compatible for existing run-log consumers.

## Reproduction Assessment

Yes. The issue gives concrete isolated-cron steps, and source inspection maps the same path: an embedded `meta.yielded` result can still flow into terminal cron `ok` bookkeeping before descendant work drains.

## Solution Assessment

Unclear. Deferring `finished` best matches the reported status and duration semantics, while adding yielded metadata first is smaller and safer for consumers; the current discussion is correctly waiting for maintainer direction on that contract.
