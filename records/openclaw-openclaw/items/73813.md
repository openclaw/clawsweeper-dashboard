## Summary

Keep open. Current main still has the reported cron observability gap: the embedded runner marks yielded turns as paused/yielded, but isolated cron finalization still turns a clean first parent result into terminal `ok`, and the timer records `finished`/`durationMs` immediately from that first result.

## What This Changes

The issue requests that isolated cron jobs using `sessions_yield` with subagents avoid reporting terminal success and a short duration before the descendant work and resumed parent session finish.

## Best Possible Solution

Cron run bookkeeping should consume yielded/paused embedded-run metadata and treat yielded isolated orchestration as nonterminal until the resumed parent and descendant subagents reach a true terminal outcome. The final emitted run log should then carry the real terminal status and total duration, with additive yielded/pending metadata only as an interim-compatible surface if maintainers want a staged rollout.

## Reproduction Assessment

Yes. The issue gives concrete isolated cron steps, and source inspection traces the same behavior: a yielded embedded run can carry paused/yielded metadata while cron finalization and the timer still produce a terminal `ok`/short-duration record from the first parent turn.

## Solution Assessment

Unclear for implementation routing. Deferred finalization best matches the expected status and duration semantics, but the discussion shows an open maintainer choice between that larger lifecycle change and a smaller additive yielded-metadata step.
