## Summary

Keep open. Current main still has the reported cron observability gap: the embedded runner exposes yielded/paused state, but isolated cron outcome typing, finalization, finished-event bookkeeping, and run-log parsing still report a nonfatal yielded run as terminal `ok` with the first execution window’s duration. The follow-up comment accurately narrows the remaining work to the cron classifier/lifecycle half.

## What This Changes

The issue asks isolated cron run bookkeeping to avoid reporting terminal `ok` and a short duration while a `sessions_yield` subagent workflow is still running.

## Best Possible Solution

Cron run logging should make yielded isolated orchestration nonterminal until the resumed parent and descendants reach a real terminal outcome, or explicitly expose yielded partial-duration semantics if maintainers choose an additive compatibility step first.

## Reproduction Assessment

Yes. The issue gives concrete isolated-cron steps, and current source still maps a yielded embedded result into terminal cron `ok` bookkeeping without consuming yielded/paused metadata.

## Solution Assessment

Unclear. The current implementation is not sufficient; deferring `finished` best matches the expected status and duration semantics, while an additive `yielded` field is a smaller partial fix that needs maintainer direction.
