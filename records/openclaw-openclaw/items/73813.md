## Summary

Keep open. Current main still has the cron observability gap: yielded embedded runs expose paused/yielded metadata, but isolated cron finalization still collapses nonfatal yielded results to terminal `ok`, and the later descendant wait is scoped to delivery dispatch rather than the run-log lifecycle.

## What This Changes

The issue asks cron run status and `durationMs` to reflect the full isolated `sessions_yield` plus subagent workflow instead of recording immediate `ok` after the pre-yield parent turn.

## Best Possible Solution

Cron run lifecycle should represent yielded orchestration explicitly: either defer terminal `finished` and total duration until the resumed parent and descendants truly finish, or add a compatible yielded/partial outcome state while preserving terminal bookkeeping semantics.

## Reproduction Assessment

Yes. The issue gives concrete isolated-cron steps, and current source still shows yielded embedded results becoming terminal cron `ok` without a lifecycle-level wait or yielded outcome.

## Solution Assessment

Unclear. Deferring `finished` best matches the reported status/duration semantics, while additive yielded metadata is a smaller compatibility step; maintainers need to choose the cron event/run-log contract.
