## Summary

Keep open. Current main still has the reported cron observability gap: yielded embedded runs carry paused/yielded metadata, but isolated cron run outcome, finished-event bookkeeping, and run-log typing still collapse the first nonfatal result to terminal ok with the first-run duration.

## What This Changes

The issue requests accurate cron status and duration reporting when an isolated cron agent turn yields to a subagent and resumes after descendant completion.

## Best Possible Solution

Cron run logging should make yielded isolated orchestration nonterminal until the resumed parent and descendants reach a real terminal outcome, or explicitly expose yielded state with documented partial-duration semantics if compatibility requires an additive first step.

## Reproduction Assessment

Yes. The issue gives concrete isolated-cron steps, and current source still maps a yielded embedded result into terminal cron ok bookkeeping without consuming yielded/paused metadata.

## Solution Assessment

Unclear. The current implementation is not sufficient; deferring finished best matches the expected status and duration semantics, while additive yielded metadata is a smaller but partial compatibility step.
