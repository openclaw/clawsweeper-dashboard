## Summary

Keep open. Current main still has the reported cron observability gap: yielded embedded runs now expose paused/yielded metadata, but isolated cron finalization still maps the first clean parent result to terminal `ok` and records duration from that first result before descendant subagents and the resumed parent complete.

## What This Changes

The issue asks for isolated cron runs that call `sessions_yield` after spawning subagents to report status and duration for the full resumed workflow rather than the pre-yield parent turn.

## Best Possible Solution

Cron run bookkeeping should treat yielded isolated orchestration as nonterminal until descendant subagents and the resumed parent reach a real terminal outcome, then emit one final run-log status and duration for the whole workflow. If maintainers want a staged rollout, additive yielded/pending metadata can be introduced first, but it should not be the final answer for duration correctness.

## Reproduction Assessment

Yes. The issue gives concrete isolated cron steps, and source inspection traces the same path: yielded embedded runs carry paused/yielded metadata, while cron finalization and the timer still produce terminal `ok` and a short duration from the first parent turn.

## Solution Assessment

Unclear until maintainers choose the lifecycle contract. Deferred finalization best matches the expected status and duration semantics; an additive `yielded` flag is safer and compatible but only partially solves the reported monitoring bug.
