## Summary

Keep open. Current main still has the cron observability gap: embedded runs now report yielded/paused state, but isolated cron finalization still records a terminal `ok` outcome and first-window duration without consuming that state.

## What This Changes

The issue asks for isolated cron run status and duration to reflect `sessions_yield` subagent orchestration instead of reporting `ok` immediately after the parent yields.

## Best Possible Solution

Cron run lifecycle should represent real terminal orchestration state, preferably by deferring terminal `finished` bookkeeping until yielded parent and descendant work completes, with additive yielded metadata only if compatibility requires an interim step.

## Reproduction Assessment

Yes. The issue gives concrete isolated-cron steps, and current source still maps an embedded yielded/paused result into terminal cron `ok` bookkeeping without inspecting the yielded metadata.

## Solution Assessment

Unclear. Deferring `finished` best matches the reported status and duration semantics, while additive yielded metadata is smaller but only partially fixes metrics; maintainers should choose the cron event contract first.
