## Summary

Keep open. Current main still turns the first yielded isolated cron runner result into terminal `ok` run bookkeeping, so the reported pre-subagent `durationMs`/`finished` observability bug remains valid.

## What This Changes

The issue asks for isolated cron run status and duration to account for post-`sessions_yield` subagent work instead of marking the cron job `ok` at the initial yield.

## Best Possible Solution

Keep the issue open until cron has an explicit lifecycle contract for yielded isolated runs, preferably deferring terminal `finished` bookkeeping until resumed parent and descendant work reaches a real terminal outcome.

## Reproduction Assessment

Yes, source-reproducible with high confidence. The embedded runner returns yielded/paused metadata, while current cron code ignores it and emits terminal `ok`/`finished` bookkeeping after the first core return; I did not run a live cron job.

## Solution Assessment

Unclear. Deferring `finished` best matches the reported status and duration semantics, while additive `yielded` metadata is a smaller compatibility step; maintainers should choose that public run-log contract before implementation.
