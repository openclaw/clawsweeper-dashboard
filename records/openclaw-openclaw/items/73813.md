## Summary

Keep open. Current main still has the reported cron observability gap: yielded embedded runs expose paused/yielded metadata, but isolated cron finalization and run-log events still collapse the first nonfatal yielded turn to terminal `ok` with the pre-yield duration.

## What This Changes

The issue asks for isolated cron runs that call `sessions_yield` after spawning subagents to report terminal status and duration only after the parent and descendants actually finish, or otherwise expose accurate partial-run semantics.

## Best Possible Solution

Cron run logging should make yielded isolated orchestration nonterminal until the resumed parent and descendants reach a real terminal outcome, or explicitly expose yielded state if compatibility requires an additive first step.

## Reproduction Assessment

Yes. The issue gives concrete isolated-cron steps, and current source still maps a yielded embedded result into terminal cron `ok` bookkeeping without consuming yielded/paused metadata.

## Solution Assessment

Unclear. The current implementation is not sufficient; deferring `finished` best matches the expected status and duration semantics, while additive yielded metadata is smaller but only a partial observability fix.
