## Summary

Keep open. Current main still turns the first yielded isolated cron runner result into terminal `ok` bookkeeping with a pre-yield duration, even though the embedded runner now exposes `yielded` and `paused` metadata that cron does not consume.

## What This Changes

The issue asks isolated cron jobs using `sessions_yield` with subagents to avoid reporting terminal `ok` and a short duration before descendant and resumed parent work actually finishes.

## Best Possible Solution

Define the cron lifecycle for yielded isolated jobs and implement it consistently across the outcome type, run-log/event surface, and isolated cron tests, ideally making terminal status and duration reflect the full resumed workflow.

## Reproduction Assessment

Yes from source inspection, though I did not execute a live cron run. A yielded embedded run returns `meta.yielded`/`livenessState: "paused"`, then cron maps that first returned result to terminal `ok` and computes duration at that return point.

## Solution Assessment

Unclear until maintainers choose the run-log/event contract. Deferring `finished` fixes both status and duration, while additive yielded metadata is smaller but only partially fixes the misleading metrics.
