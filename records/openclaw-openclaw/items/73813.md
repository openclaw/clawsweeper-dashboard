## Summary

Keep open. Current main still maps a yielded isolated cron run to terminal `ok` bookkeeping and first-return duration even though the embedded runner marks yielded attempts as paused/yielded, so the monitoring bug remains and needs a maintainer lifecycle-contract decision.

## What This Changes

The issue asks isolated cron jobs that use `sessions_yield` with subagents to report status and duration according to actual post-yield completion instead of logging an early `ok`.

## Best Possible Solution

Cron should represent yielded isolated runs without misleading terminal success metrics, ideally by deferring terminal `finished` bookkeeping until resumed parent and descendant work reaches a real terminal outcome, with additive yielded metadata only if compatibility requires it.

## Reproduction Assessment

Yes, from source inspection rather than a live cron run. Current main produces embedded yielded metadata, then cron classifies the first returned embedded result as terminal `ok` and records duration at that point.

## Solution Assessment

Unclear until maintainers choose the public lifecycle contract. Deferring `finished` best matches the reported status and duration semantics, while an additive `yielded` field is smaller but only partially fixes metrics.
