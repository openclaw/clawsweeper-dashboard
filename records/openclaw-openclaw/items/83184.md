## Label Justifications

- `P1`: A deterministic stuck pending-final-delivery state can stop real heartbeat-driven agent notifications without a surfaced error.
- `impact:session-state`: The bug is stale persisted session state in the `pendingFinalDelivery*` fields.
- `impact:message-loss`: Once stuck, subsequent heartbeat model calls or outbound notifications can be silently skipped or replay stale text.

## Summary

Keep open: current main and v2026.5.22 still leave the heartbeat send-success store write without clearing pending-final-delivery state, and the open linked fix PR https://github.com/openclaw/openclaw/pull/83187 owns the central repair.

## What This Changes

The issue asks for successful heartbeat delivery to clear `pendingFinalDelivery*` fields so later heartbeat ticks invoke the agent normally.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/83187 for the send-success cleanup, while keeping broader orphan replay and `target:none` recovery work on their narrower linked issues.

## Reproduction Assessment

Yes. Source inspection shows a substantive heartbeat reply can set `pendingFinalDelivery`, the successful heartbeat send path records last-heartbeat metadata without clearing it, and the next heartbeat can short-circuit through pending replay.

## Solution Assessment

Yes. Clearing the pending-final-delivery fields only after confirmed durable heartbeat send success mirrors the existing user-message cleanup contract without weakening failed-send retry semantics.
