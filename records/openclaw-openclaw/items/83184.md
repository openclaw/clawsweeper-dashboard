- `impact:session-state`: The bug is stale persisted `pendingFinalDelivery*` session state that blocks later heartbeat processing.

## Summary

Keep open: current `main` and the latest release still record heartbeat delivery metadata without clearing `pendingFinalDelivery*`, and the live linked PR that closes this issue is still open and waiting on author-side review resolution.

## What This Changes

The issue reports that substantive heartbeat replies leave `pendingFinalDelivery*` session fields set after successful delivery, causing later heartbeat ticks to replay stale state instead of running the agent.

## Best Possible Solution

Land a corrected version of `https://github.com/openclaw/openclaw/pull/83187` for the heartbeat send-success cleanup, while keeping broader replay TTL/orphan-session hardening on the canonical related issues such as `https://github.com/openclaw/openclaw/issues/85743`.

## Reproduction Assessment

Yes, at source level: current `main` can set substantive heartbeat output as `pendingFinalDelivery`, then the heartbeat send-success path records delivery metadata without clearing that state. I did not run a live heartbeat scenario in this read-only review.

## Solution Assessment

Yes, the best fix direction is to mirror the existing successful-final-delivery cleanup in the heartbeat send-success path, with the open PR adjusted so cleanup only follows actual visible delivery. A separate TTL/orphan replay hardening path should not block the central send-success fix.
