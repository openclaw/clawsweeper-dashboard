## Summary

Keep open: current main still has the reported heartbeat/session-state gap, and the linked fix PR is open with closing syntax, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that heartbeat-driven substantive replies leave pendingFinalDelivery recovery fields set after a successful heartbeat send, causing later heartbeats to short-circuit.

## Best Possible Solution

Heartbeat send-success should clear the pendingFinalDelivery recovery fields in the same session-store write that records the delivered heartbeat payload, with focused regression coverage for substantive heartbeat text.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: seed pendingFinalDelivery=true with ack text so the heartbeat defer guard does not stop the run, deliver substantive heartbeat text, and current main records lastHeartbeatText without clearing the pending fields. I did not execute the test because this review is read-only.

## Solution Assessment

Yes. The linked PR's approach is the narrowest maintainable fix: clear the same recovery fields already cleared by successful user-message dispatch, but in the heartbeat send-success path that actually completes this delivery.
