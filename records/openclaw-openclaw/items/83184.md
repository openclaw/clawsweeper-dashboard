## Label Justifications

- `P1`: A substantive heartbeat can leave persisted session state stuck and silently block later heartbeat-driven notifications until manual repair.
- `impact:session-state`: The root failure is stale `pendingFinalDelivery*` metadata persisting after successful heartbeat delivery.
- `impact:message-loss`: The stale pending state can suppress fresh heartbeat model calls and outbound notifications.

## Summary

Keep open: current main and the latest release still show the heartbeat send-success path recording delivery metadata without clearing `pendingFinalDelivery*`, and an open linked PR already owns the central fix path for this issue.

## What This Changes

The issue requests clearing heartbeat-created `pendingFinalDelivery*` session state after successful heartbeat delivery so later heartbeats do not replay stale final-delivery state.

## Best Possible Solution

Review and land `https://github.com/openclaw/openclaw/pull/83187` for the send-success cleanup, while keeping the broader orphan/timeout replay bounds in `https://github.com/openclaw/openclaw/issues/85743` or its linked PR if maintainers want that hardening too.

## Reproduction Assessment

Yes from source inspection: seed a substantive heartbeat pending final, let the heartbeat send path succeed, and the store update leaves `pendingFinalDelivery*` intact while the next heartbeat replay branch can return the stale text. I did not execute a writable fixture because this review is read-only.

## Solution Assessment

Yes: clearing pending-final-delivery fields only after confirmed heartbeat send success mirrors the existing final-dispatch cleanup and preserves failed-send retry semantics. The broader retry/TTL hardening should stay a separate follow-up path.
