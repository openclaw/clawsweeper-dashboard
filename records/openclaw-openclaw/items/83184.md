## Label Justifications

- `P1`: A substantive heartbeat can silently block later heartbeat work and user-visible notifications until session state is manually repaired.
- `impact:session-state`: The root failure is persisted `pendingFinalDelivery*` state staying set after the heartbeat message was already sent.
- `impact:message-loss`: The stale state can prevent later heartbeat-triggered model calls or deliveries, producing missed notifications without a surfaced error.

## Summary

Keep open: current main and the latest release still leave the heartbeat send-success path without pending-final-delivery cleanup, and the primary fix is already represented by the open linked PR at https://github.com/openclaw/openclaw/pull/83187.

## What This Changes

The issue reports that heartbeat-triggered substantive agent replies keep `pendingFinalDelivery*` set after successful durable send, causing later heartbeat ticks to short-circuit on stale final-delivery state.

## Best Possible Solution

Review and land the focused heartbeat send-success cleanup in https://github.com/openclaw/openclaw/pull/83187, while keeping broader orphan replay and timeout recovery behavior tracked in the related follow-up issues.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: substantive heartbeat replies are persisted as pending, the heartbeat success path sends and records only last-heartbeat metadata, and the next heartbeat can replay stale pending text instead of invoking the agent.

## Solution Assessment

Yes. Clearing the existing pending-final-delivery fields only after verified heartbeat send success mirrors the established user-message dispatch cleanup and preserves failed-send retry semantics.
