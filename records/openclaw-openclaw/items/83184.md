## Summary

Keep open: current main and the latest shipped tag still show the source-level heartbeat pending-final-delivery gap, and the issue is already paired with an open fixing PR that should close it after merge.

## What This Changes

The issue reports that successful heartbeat-driven replies record heartbeat metadata but leave pending-final-delivery recovery fields set, causing later heartbeat runs to short-circuit on stale session state.

## Best Possible Solution

Review and land the linked narrow heartbeat send-success cleanup with regression coverage, then close this issue through the merged PR once current main clears pending-final-delivery state after confirmed heartbeat delivery.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: substantive heartbeat output is persisted as pending delivery, the heartbeat success path sends it without clearing those fields, and the next heartbeat can return the stale pending text before normal model work.

## Solution Assessment

Yes. The best fix is the narrow path already proposed in the linked PR: clear the existing pendingFinalDelivery* recovery fields only after successful heartbeat delivery, matching user-message dispatch cleanup while preserving failed-send retry semantics.
