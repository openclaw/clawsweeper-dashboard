## Summary

Keep open: current main still has a source-reproducible path where a substantive heartbeat reply can be durably sent while its `pendingFinalDelivery*` state remains set, so the next heartbeat can replay or defer against stale final-delivery state. The merged fix for https://github.com/openclaw/openclaw/pull/80357 handled ack-only heartbeat pending loops, but it intentionally preserves real heartbeat content as pending until the heartbeat send path proves delivery, and that success path still lacks the cleanup step.

## What This Changes

The issue reports that heartbeat-sent substantive replies do not clear pending final-delivery session state, causing later heartbeat runs to short-circuit instead of invoking the agent.

## Best Possible Solution

Add a narrow heartbeat success cleanup that clears the existing pending-final-delivery fields after `sendDurableMessageBatch` succeeds, with regression coverage for substantive heartbeat text and no change to failed-send retry semantics.

## Reproduction Assessment

Yes from source inspection: a substantive heartbeat reply is saved as pending delivery, the heartbeat success path sends it and records only last-heartbeat metadata, and the next heartbeat can replay that pending text before normal model work. I did not execute a live or test reproduction in this read-only review.

## Solution Assessment

Yes: mirroring the existing successful final-dispatch cleanup in the heartbeat send-success branch is the narrowest maintainable fix. The safer boundary is to clear after confirmed durable heartbeat delivery, not to stop storing real heartbeat content before delivery.
