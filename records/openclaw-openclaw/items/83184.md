## Label Justifications

- `P1`: A substantive heartbeat can silently block later heartbeat model calls and notifications in real deployments.
- `impact:session-state`: The bug is stale persisted `pendingFinalDelivery*` session state after a successful heartbeat send.
- `impact:message-loss`: The stale state suppresses subsequent heartbeat deliveries and user-visible notifications.

## Summary

Keep open: current main and the latest release still record successful heartbeat delivery metadata without clearing the pending-final-delivery recovery fields. The linked open fix PR is the right path, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to clear heartbeat-owned `pendingFinalDelivery*` session state after a successful substantive heartbeat send so later heartbeats can run normally.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83187, or an equivalent narrow heartbeat success-path cleanup with regression coverage and no failed-send retry behavior change.

## Reproduction Assessment

Yes. Source inspection shows substantive heartbeat text can be saved as pending delivery, successfully sent by heartbeat-runner, and then replayed or deferred on the next heartbeat because the success path never clears the pending fields; user comments also report production incidents on 2026.5.18.

## Solution Assessment

Yes. Clearing the existing pending-final-delivery fields in the heartbeat send-success branch mirrors the already established final-dispatch cleanup and is narrower than adding a new state path or config option.
