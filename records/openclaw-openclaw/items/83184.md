## Label Justifications

- `P1`: The stale pending-final-delivery state can stop heartbeat-driven agent work and silently suppress subsequent notifications.
- `impact:session-state`: The issue centers on pendingFinalDelivery session fields remaining set after successful heartbeat delivery.
- `impact:message-loss`: The reported user-visible impact is missed heartbeat notifications and skipped model calls without surfaced errors.

## Summary

Keep open: current main still has a source-reproducible heartbeat session-state bug, and https://github.com/openclaw/openclaw/pull/83187 is the open implementation candidate that closes this issue. The prior merged fix at https://github.com/openclaw/openclaw/pull/80357 covered ack-only pending loops but intentionally preserved real heartbeat content, leaving the successful-send cleanup gap still present.

## What This Changes

The issue asks OpenClaw to clear pendingFinalDelivery* session fields after a successful substantive heartbeat send.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83187, or an equivalent narrow heartbeat success-path cleanup with regression coverage, then close this issue as implemented.

## Reproduction Assessment

Yes. Source inspection shows substantive heartbeat replies can set pendingFinalDelivery, the heartbeat success path records only last-heartbeat metadata, and the next heartbeat can return stale replay text before normal agent work.

## Solution Assessment

Yes. Clearing the existing pending-final-delivery fields only after successful heartbeat delivery matches the user-message dispatch cleanup contract and preserves failed-send retry semantics.
