## Label Justifications

- `P1`: A substantive heartbeat reply can silently block future heartbeat-driven agent work for real deployments.
- `impact:session-state`: The failure is stale `pendingFinalDelivery*` state persisting in the session store after successful heartbeat delivery.
- `impact:message-loss`: The reported symptom is missed heartbeat notifications and skipped model calls with no surfaced error.

## Summary

Keep open: current main and v2026.5.19 still have the heartbeat success path that records last-heartbeat metadata without clearing `pendingFinalDelivery*`, and this issue is already paired with an open closing PR.

## What This Changes

The issue reports that substantive heartbeat-driven replies leave pending final-delivery session state uncleared after successful heartbeat delivery, causing later heartbeats to short-circuit.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83187 or an equivalent narrow cleanup that clears pending-final-delivery fields only after successful heartbeat durable send, with the focused regression test.

## Reproduction Assessment

Yes. Source inspection shows real heartbeat content can be persisted as pending final delivery, the heartbeat success path sends it and records only last-heartbeat metadata, and the next heartbeat can replay that stale pending text before normal agent work.

## Solution Assessment

Yes. Mirroring the existing final-dispatch cleanup in the heartbeat send-success branch is the narrowest maintainable fix, and the open linked PR follows that boundary.
