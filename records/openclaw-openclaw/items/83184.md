- `impact:session-state`: The report centers on stale pending-final-delivery fields in persisted session state.
- `impact:message-loss`: The stale state can suppress later heartbeat notifications and model calls without a surfaced error.

## Summary

Keep open: current main and the latest v2026.5.18 release still lack the heartbeat send-success cleanup, and the linked fix PR with closing syntax remains open. This issue should close only after https://github.com/openclaw/openclaw/pull/83187 or an equivalent fix merges.

## What This Changes

The issue reports that successful heartbeat-triggered agent replies do not clear pending-final-delivery session state, so later heartbeats can be blocked or replay stale pending text.

## Best Possible Solution

Land or reject https://github.com/openclaw/openclaw/pull/83187 after maintainer validation; the durable state cleanup belongs in the heartbeat send-success path.

## Reproduction Assessment

Yes, at source level: substantive heartbeat output can set pendingFinalDelivery, the successful heartbeat send branch writes only last-heartbeat metadata, and the next heartbeat can hit the pending redelivery path. A later production comment reports the same behavior on v2026.5.18.

## Solution Assessment

Yes. Clearing the seven pending-final-delivery fields after confirmed heartbeat send success mirrors the existing final-dispatch cleanup while preserving retry state on failed sends.
