- `impact:session-state`: The bug is stale pendingFinalDelivery session state that survives after confirmed heartbeat delivery.

## Summary

Keep open: current main still has the heartbeat send-success gap, and the issue is already paired with an open closing PR that owns the narrow fix. Recent comments add broader orphan/timeout replay concerns, but those are better preserved through the linked follow-up issue and PR rather than closing this before the success-path fix lands.

## What This Changes

Heartbeat-driven substantive replies should clear pendingFinalDelivery fields after confirmed heartbeat delivery so later heartbeats invoke the agent normally.

## Best Possible Solution

Review and land the focused heartbeat send-success cleanup in https://github.com/openclaw/openclaw/pull/83187, then close this issue as implemented while leaving the broader orphan replay cap to its canonical follow-up.

## Reproduction Assessment

Yes. Current main source shows real heartbeat replies can be persisted as pending final delivery, the heartbeat success path does not clear that state, and the next heartbeat replay branch returns stale pending text before normal agent work; I did not run a live heartbeat reproduction in this read-only review.

## Solution Assessment

Yes for the original success path. Clearing the existing pendingFinalDelivery fields only after sendDurableMessageBatch succeeds mirrors the established user-message delivery cleanup; the broader timeout/orphan fail-safe should stay in the linked follow-up issue and PR.
