## Summary

Current main still contains the stale heartbeat delivery-state path, and the open linked PR owns the repair; its discussion identifies an additional duplicate-suppression case that should be included before landing.

## What This Changes

Clear heartbeat-owned pending final-delivery fields after successful durable delivery so later heartbeat ticks invoke the agent normally.

## Best Possible Solution

Use the linked PR as the canonical fix, extending its cleanup and regression coverage to both verified delivery and intentional duplicate suppression while preserving pending state for real failures.

## Reproduction Assessment

Yes at source level: substantive output sets pending delivery, current main does not clear it after heartbeat delivery, and later ticks consult that state; several independent production reports confirm the behavior.

## Solution Assessment

Yes in direction, but the canonical implementation should also handle the documented duplicate-suppression path rather than only the ordinary outbound-send result.
