## Summary

Keep open: current main and v2026.6.5 still leave the heartbeat send-success store write without clearing `pendingFinalDelivery*`. The direct stale replay/blocking behavior was narrowed by merged work in https://github.com/openclaw/openclaw/pull/81313, but the requested send-success cleanup remains owned by the open linked PR https://github.com/openclaw/openclaw/pull/83187.

## What This Changes

The issue requests clearing heartbeat-created `pendingFinalDelivery*` session fields after a successful substantive heartbeat delivery so later heartbeat ticks do not inherit stale recovery state.

## Best Possible Solution

Review and land the linked cleanup PR after it only clears pending state for an actual delivered heartbeat result, or close that PR with a clear maintainer decision and keep a canonical follow-up for the remaining send-success cleanup.

## Reproduction Assessment

Yes. Source inspection shows substantive heartbeat output can set `pendingFinalDelivery*`, and both current main and v2026.6.5 still record successful heartbeat delivery metadata without clearing those fields; current main has mitigated direct replay, but the stale-field bug remains source-reproducible.

## Solution Assessment

Yes with one caveat. Clearing pending final-delivery state after confirmed successful heartbeat delivery is the right owner-boundary fix, but the linked PR should avoid clearing when the durable send result is suppressed or otherwise not actually delivered.
