- `impact:session-state`: The root problem is stale `pendingFinalDelivery*` state left in the session store.
- `impact:message-loss`: The user-visible failure is missed or replayed heartbeat delivery and skipped model calls.

## Summary

Keep open: current `main` and `v2026.5.22` still do not clear `pendingFinalDelivery*` after a successful substantive heartbeat send, and https://github.com/openclaw/openclaw/pull/83187 is the open linked fix for that path. The newer comments add adjacent abort/TTL routes, but they do not make this send-success bug obsolete.

## What This Changes

Issue requests clearing heartbeat-driven `pendingFinalDelivery*` session state after confirmed heartbeat delivery so later heartbeat ticks do not replay or block on stale final-delivery state.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83187 or an equivalent narrow heartbeat send-success cleanup, then keep the adjacent target-none and TTL reports on their own tracks.

## Reproduction Assessment

Yes, from source inspection: a substantive heartbeat reply can be persisted as pending final delivery, the heartbeat send-success path records only last-heartbeat metadata, and the next heartbeat can replay stale pending text. I did not execute a test or live reproduction in this read-only review.

## Solution Assessment

Yes: mirroring the existing final-dispatch success cleanup in the heartbeat send-success branch is the narrowest maintainable fix for this issue. The broader orphan replay cap and target-none recovery paths should stay separate.
