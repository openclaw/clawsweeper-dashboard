## Label Justifications

- `P1`: A successful substantive heartbeat can deterministically leave session state stuck and stop future heartbeat-driven notifications or model calls.
- `impact:session-state`: The issue is about durable pendingFinalDelivery session state remaining set after heartbeat delivery succeeds.
- `impact:message-loss`: Subsequent heartbeat alerts or updates can be silently skipped or blocked by the stale pending-delivery state.

## Summary

Keep open: current main and latest release v2026.6.8 still record successful heartbeat delivery without clearing pendingFinalDelivery state, and https://github.com/openclaw/openclaw/pull/83187 is the open closing implementation lane.

## What This Changes

The issue reports that heartbeat-driven substantive replies leave pendingFinalDelivery state set after successful delivery, causing later heartbeat ticks to short-circuit instead of invoking the agent.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83187 or an equivalent focused heartbeat send-success cleanup with regression coverage, while keeping broader replay-cap work on https://github.com/openclaw/openclaw/issues/85743.

## Reproduction Assessment

Yes, at source level. Current main stores substantive heartbeat output as pendingFinalDelivery and later records visible heartbeat send success without clearing those fields; multiple production comments report the same field state.

## Solution Assessment

Yes, the linked PR direction is the narrow maintainable fix for this issue. The cleanup belongs in the heartbeat send-success owner path, while stale orphan replay caps remain a separate follow-up.
