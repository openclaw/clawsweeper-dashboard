## Label Justifications

- `P1`: A deterministic heartbeat session-state bug can silently stop or stale-replay real heartbeat-driven notifications and model work.
- `impact:session-state`: The issue centers on stale durable `pendingFinalDelivery*` session fields after successful heartbeat sends.

## Summary

Keep open: current main and the latest release still record successful substantive heartbeat sends without clearing durable pending-final-delivery state, and the live linked fix PR remains open rather than merged.

## What This Changes

The issue asks for heartbeat-driven successful sends to clear `pendingFinalDelivery*` state so later heartbeats do not inherit stale delivery recovery state.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/83187 for the send-success cleanup, while keeping broader orphan replay and timeout hardening on their separate canonical issues.

## Reproduction Assessment

Yes. Source inspection shows the current heartbeat run can persist substantive pending-final-delivery content, then successfully send and record heartbeat metadata without clearing that durable state.

## Solution Assessment

Yes for the central bug shape. Clearing the fields only after confirmed heartbeat delivery mirrors the existing dispatch cleanup contract, with PR review needed for suppressed-send and duplicate-suppression edge cases.
