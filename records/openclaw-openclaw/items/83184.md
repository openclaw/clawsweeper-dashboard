- `impact:session-state`: A successful heartbeat leaves durable pending final-delivery recovery state incorrectly attached to the session.
- `impact:message-loss`: Fresh heartbeat notifications are silently blocked or displaced by stale redelivery attempts.

## Summary

Current main still records successful substantive heartbeat delivery without clearing `pendingFinalDelivery*`, while the clean, proof-backed linked PR remains open and explicitly closes this issue. Repository policy therefore requires keeping the issue open until that implementation lands or is replaced. Later comments identify additional timeout/abort orphan and unbounded-replay cases, but those broaden rather than invalidate this issue’s concrete send-success defect.

## What This Changes

Clear heartbeat-created pending final-delivery state after successful delivery so later heartbeat runs are not blocked.

## Best Possible Solution

Land one canonical post-success cleanup that clears all pending final-delivery fields only after verified heartbeat delivery, with the focused next-heartbeat regression test, while handling terminal orphan recovery and replay caps in separate bounded work.

## Reproduction Assessment

Yes, at high confidence from source inspection: a substantive heartbeat can create pending delivery, the successful heartbeat send does not clear it, and later heartbeat handling consumes that stale state. Multiple independent production reports and the linked PR’s red-before/green-after test and live Docker observation corroborate the path, although this read-only review did not run it locally.

## Solution Assessment

Yes for the central reported route: clearing the recovery fields only after verified heartbeat send success is narrow and symmetric with existing successful-dispatch cleanup. Timeout/abort orphan recovery and retry TTLs are distinct behavior and should not be folded into this focused fix without separate validation.
