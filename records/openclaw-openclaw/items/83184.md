## Label Justifications

- `P1`: A substantive heartbeat can leave a session silently unable to run later heartbeats, breaking an active agent workflow for real users.
- `impact:message-loss`: Subsequent heartbeat notifications are suppressed or replaced by stale replay without a surfaced delivery error.
- `impact:session-state`: Durable pending-final-delivery fields remain stuck after successful heartbeat delivery and alter later run routing.

## Summary

Current main still has the reported send-success cleanup gap, and the latest release does not contain the linked fix. The issue must remain open while https://github.com/openclaw/openclaw/pull/83187 is corrected and reviewed; that PR is still open and its existing review identifies a concrete suppressed-send case that must preserve recovery state.

Likely related people: @MertBasar0 introduced the main-session durable final-delivery flow (high confidence), while @steipete and @hclsys carried the later heartbeat pending-replay policy work (high confidence).

## What This Changes

Ensure successful substantive heartbeat delivery clears durable pending-final-delivery session state so later heartbeats run normally.

## Best Possible Solution

Update and land the linked PR so pending state clears only after an actual `sent` heartbeat result, remains intact for suppressed or failed delivery, and is covered by focused sent-versus-suppressed regression tests.

## Reproduction Assessment

Yes at source level. Current main persists substantive heartbeat output as pending, sends it, then omits the established successful-delivery cleanup; multiple independent production reports also show later heartbeats resume when those fields are manually cleared.

## Solution Assessment

Yes, with the linked review correction. Clearing the canonical recovery state after an actually sent heartbeat is the narrowest maintainable fix, but suppression must remain recoverable and the broader timeout/replay-cap cases should stay separately tracked.
