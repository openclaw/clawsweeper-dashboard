## Label Justifications

- `P1`: A successful substantive heartbeat can deterministically leave session state stuck and stop future heartbeat-driven notifications or model calls in active deployments.
- `impact:session-state`: The issue is about durable pending-final-delivery session state remaining set after the heartbeat payload has already been sent.
- `impact:message-loss`: Subsequent heartbeat alerts or updates can be silently skipped, blocked, or replaced by stale redelivery attempts.

## Summary

Keep open: current main and the latest release still have the heartbeat send-success cleanup gap, and https://github.com/openclaw/openclaw/pull/83187 is the open closing implementation lane for this issue.

## What This Changes

The issue requests clearing pendingFinalDelivery session fields after a successful heartbeat-driven visible send so later heartbeat ticks do not short-circuit on stale final-delivery state.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83187 or an equivalent focused heartbeat send-success cleanup with regression coverage, while keeping the broader retry-cap/TTL behavior on https://github.com/openclaw/openclaw/issues/85743.

## Reproduction Assessment

Yes, at source level. Current main still creates pending final-delivery state for substantive heartbeat output and records successful visible heartbeat sends without clearing that state; multiple production comments provide matching field evidence.

## Solution Assessment

Yes, the linked pull request direction is the narrow maintainable fix for this issue. The cleanup belongs in the heartbeat owner path after visible send success, while orphan replay caps belong in the separate canonical follow-up.
