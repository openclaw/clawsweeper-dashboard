## Summary

Current main and v2026.6.6 still leave heartbeat-created pending final-delivery state uncleared after visible heartbeat sends. The open closing PR owns the narrow fix, so this issue should stay open until that PR lands or is closed.

## What This Changes

The issue reports that substantive heartbeat replies set durable pending-final-delivery state, while the heartbeat delivery path records the sent text without clearing that state, blocking later heartbeats.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/83187 with the visible-send guard preserved and complete current pending-delivery cleanup, while keeping broader orphan replay TTL work in https://github.com/openclaw/openclaw/issues/85743.

## Reproduction Assessment

Yes, at source level. Current main still persists substantive heartbeat output as pending final delivery and records successful visible heartbeat delivery without clearing that durable state; multiple production comments and the linked PR's proof support the same path.

## Solution Assessment

Yes in direction. The best fix is owner-local cleanup in the heartbeat visible-send success branch, with the complete current pending-delivery field set cleared and broader orphan replay recovery left to its canonical issue.
