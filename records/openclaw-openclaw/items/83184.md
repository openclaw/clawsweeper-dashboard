## Label Justifications

- `P1`: A stale pending-final-delivery state can silently stop heartbeat-driven agent work and notifications for real deployments.
- `impact:session-state`: The core failure is stale persisted session state in the `pendingFinalDelivery*` fields.
- `impact:message-loss`: The stale state can prevent subsequent heartbeat notifications or model calls from being delivered.

## Summary

Keep open: current main and the latest stable release still have the heartbeat send-success path that records `lastHeartbeatText` and `lastHeartbeatSentAt` without clearing pending-final-delivery state, and the directly linked fix PR is still open at https://github.com/openclaw/openclaw/pull/83187. The newer comments add useful production confirmation and a broader orphan/timeout route, but those broader fail-safe pieces are tracked separately by related open work rather than making this item closeable now.

## What This Changes

The issue reports that substantive heartbeat-generated replies can leave `pendingFinalDelivery*` session fields uncleared after delivery, causing later heartbeat ticks to replay stale state instead of running the agent.

## Best Possible Solution

Review and land or deliberately supersede https://github.com/openclaw/openclaw/pull/83187 for the send-success cleanup, then keep the broader stale-orphan replay and target-none recovery behavior on their own linked follow-ups rather than folding unrelated fixes into this issue.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.5.20` shows substantive heartbeat text is saved as pending final delivery, the heartbeat success path sends it and records only last-heartbeat metadata, and the next heartbeat can replay stale pending text before normal model work.

## Solution Assessment

Yes for the central send-success bug: clearing pending-final-delivery fields only after confirmed heartbeat delivery mirrors the existing user-message dispatch cleanup and preserves failed-send retry semantics. The newer abort/timeout route should be handled by the related fail-safe/recovery follow-ups, not by delaying this narrow cleanup.
