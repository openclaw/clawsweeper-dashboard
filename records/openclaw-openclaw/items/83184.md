## Label Justifications

- `P1`: The issue describes a current heartbeat workflow becoming silently wedged for real users after a successful substantive heartbeat delivery.
- `impact:session-state`: The root problem is stale persisted `pendingFinalDelivery*` session state blocking later heartbeat behavior.
- `impact:message-loss`: Subsequent heartbeat notifications or model calls can be suppressed behind stale final-delivery replay state.

## Summary

Keep open: current `main` and the latest shipped tag still leave the heartbeat send-success path without pending-final-delivery cleanup, and the issue has an open linked fix PR that should be reviewed or landed before the issue closes.

## What This Changes

The issue asks for heartbeat-driven agent replies to clear `pendingFinalDelivery*` after successful heartbeat delivery so later heartbeats can run normally.

## Best Possible Solution

Review and land the narrow send-success cleanup in https://github.com/openclaw/openclaw/pull/83187, while leaving the broader orphan replay cap to the separate retry/TTL follow-up.

## Reproduction Assessment

Yes. Source inspection shows substantive heartbeat replies can set `pendingFinalDelivery*`, the heartbeat success path can deliver the message without clearing those fields, and the next heartbeat can short-circuit through the stale replay path.

## Solution Assessment

Yes for the central send-success bug. Clearing the pending-final-delivery fields only after successful heartbeat delivery mirrors the existing final-dispatch cleanup without adding a new config or policy path.
