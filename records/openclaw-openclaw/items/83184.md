## Label Justifications

- `P1`: A successful substantive heartbeat can deterministically wedge recurring heartbeat execution in active deployments until session state is manually repaired.
- `impact:message-loss`: Later heartbeat alerts may be silently suppressed or replaced by stale replay attempts.
- `impact:session-state`: Durable pending-final-delivery state remains associated with the session after the original heartbeat payload has already been sent.

## Summary

Current `main` and the latest release still record a substantive heartbeat as pending final delivery but do not clear that state after the outbound batch succeeds. The open, clean linked PR is the correct implementation lane and includes strong live proof, but it should remain paired with this issue until its handling of `suppressed` sends and the current complete field set are resolved and the PR merges.

## What This Changes

Clear durable pending-final-delivery session state after an actual successful heartbeat delivery so subsequent heartbeat ticks invoke the agent normally.

## Best Possible Solution

Update and land the linked PR so heartbeat cleanup occurs only after an actually delivered result, clears the complete current pending-delivery state atomically, preserves recovery state for non-delivery outcomes, and leaves broader orphan replay recovery to its separate canonical issue.

## Reproduction Assessment

Yes at source level. Current main still exposes the reported state transition, several independent production reports correlate the stuck fields with stopped heartbeats, and the linked PR documents a focused red-before/green-after regression plus live Docker verification.

## Solution Assessment

Yes in direction, but not yet in the linked PR’s current form. Post-delivery cleanup in the heartbeat owner path is the narrowest maintainable fix, provided it is gated on actual delivery and tracks the complete current state shape.
