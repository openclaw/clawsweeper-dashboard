## Summary

## What This Changes

The issue reports that memory-core dreaming writes transcript corpus artifacts under `memory/.dreams/session-corpus/` and then lets those artifacts pollute short-term recall, `MEMORY.md`, and search state.

## Best Possible Solution

Land or replace the focused fix path in https://github.com/openclaw/openclaw/pull/78130 so session-corpus stays internal dreaming evidence and cannot enter durable recall ranking, promotion, or memory-search pollution.

## Reproduction Assessment

Yes, source-reproducible: current main writes session-corpus files, returns them as memory results, records them through `recordShortTermRecalls()`, and ranks them through `rankShortTermPromotionCandidates()`. I did not run a multi-day live dreaming cron reproduction.

## Solution Assessment

Yes for the direction: the best fix is a narrow memory-core boundary that treats session-corpus as internal evidence, not a new user exclusion knob. The existing open PR needs repair, real behavior proof, and test alignment before this issue can close.
