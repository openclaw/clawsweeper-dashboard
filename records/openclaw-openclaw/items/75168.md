## What This Changes

The issue requests fixing a gateway crash by guarding per-channel logger object lookups in async channel lifecycle error, cleanup, restart, and stop handlers.

## Best Possible Solution

Land a narrow gateway fix that treats `channelLogs[channelId]` as optional inside captured async lifecycle closures, updates the affected `log.<level>?.(...)` dereferences in `src/gateway/server-channels.ts`, and adds regression coverage for a channel start/error path with no logger entry. Because this is a user-facing crash fix, include a changelog entry with non-bot contributor credit.

Yes. A high-confidence source-level reproduction is a channel manager whose active channel id has no `channelLogs` entry, followed by a `startAccount`, cleanup, restart, or stop continuation that reaches `log.error`, `log.info`, or `log.warn`.

## Solution Assessment

Yes. Guarding the logger object with `log?.<level>?.(...)` at the captured async call sites is the narrowest maintainable fix; a stronger version should also make the optional lookup visible in tests or types.
