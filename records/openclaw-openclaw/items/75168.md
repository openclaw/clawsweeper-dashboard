## Summary

Keep open. Current main still contains the unsafe `channelLogs[channelId]` logger dereferences in the gateway channel lifecycle path, the latest release still has the same pattern, and the two related fix PRs are closed unmerged rather than a canonical open implementation candidate.

## What This Changes

The issue requests preventing gateway crashes when async channel lifecycle handlers dereference an undefined per-channel logger.

## Best Possible Solution

Land a narrow gateway repair that treats `channelLogs[channelId]` as optional in the channel manager lifecycle closures, updates the affected logger call sites consistently, and adds regression coverage for a missing logger map entry so the channel error is recorded instead of escaping as an unhandled rejection.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a channel manager started with no `channelLogs` entry for the active channel, followed by a `startAccount` resolve/reject or stop-timeout path that reaches the current `log.error`, `log.info`, or `log.warn` dereference.

## Solution Assessment

Yes. Guarding the logger object at the affected call sites is the narrow immediate fix; the more robust version should also make the missing-map contract visible in types or tests without changing unrelated module-scope loggers.
