## Summary

Keep open. Current main still contains the unsafe per-channel logger dereferences in the async channel lifecycle path, the latest release tag still has the same pattern, and the two related fix PRs are closed unmerged with no currently open PR found for this issue.

## What This Changes

The issue asks OpenClaw to prevent gateway crashes when async channel lifecycle handlers dereference a missing per-channel logger.

## Best Possible Solution

Land a narrow gateway fix that treats `channelLogs[channelId]` as optional inside captured async lifecycle closures, updates every affected `log.<level>?.(...)` dereference in `src/gateway/server-channels.ts`, and adds regression coverage for a channel start/error path with no logger entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a channel manager whose active channel id has no `channelLogs` entry, then a `startAccount` task resolves, rejects, cleans up, or schedules restart so the continuation dereferences `log.error`, `log.info`, or `log.warn`.

## Solution Assessment

Yes for the immediate crash: optional-chaining the logger object at the affected call sites is the narrow maintainable fix. The best version should also make the test setup prove the missing-map case and consider typing `channelLogs` as possibly missing so this contract is visible.
