## Summary

Keep open. Current main still contains the unsafe per-channel logger dereferences in the gateway channel lifecycle path, the latest shipped tag still has the same pattern, and the two related fix PRs are closed unmerged; however, the issue is narrow enough for a ClawSweeper repair PR.

## What This Changes

The issue reports a gateway crash when async channel lifecycle handlers call `log.error?.(...)`, `log.info?.(...)`, or `log.warn?.(...)` after `channelLogs[channelId]` resolves to `undefined`.

## Best Possible Solution

Land a narrow gateway fix that treats per-channel logger lookups as optional wherever `server-channels.ts` reads from `channelLogs[channelId]` in lifecycle continuations, stop handling, and related error paths, with regression coverage proving a missing logger entry records channel failure state instead of escaping as an unhandled rejection.

## Reproduction Assessment

Yes. A focused source-level reproduction is to create a channel manager for a registered channel whose `channelLogs` entry is absent, make `startAccount` resolve or reject, and let the lifecycle continuation reach the current `log.error?.(...)` path.

## Solution Assessment

Yes for the immediate crash. The narrow maintainable fix is to guard the logger object (`log?.error?.(...)`, `log?.info?.(...)`, `log?.warn?.(...)`) and add a regression test; a small type tightening to make `channelLogs` partial would make the runtime contract more visible.
