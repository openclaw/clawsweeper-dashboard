## Summary

Keep open. Current checked-out `main` still has the unguarded per-channel logger dereferences, latest release `v2026.4.29` still contains the same pattern, and open PR #75172 already targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests guarding gateway channel lifecycle logger access so a missing `channelLogs[channelId]` entry cannot crash the gateway.

## Best Possible Solution

Review and land #75172, or an equivalent narrow gateway fix, with regression coverage for a channel plugin that starts while `channelLogs[channelId]` is missing or becomes unavailable. The final patch should guard all captured lifecycle logger calls and, ideally, make the type contract reflect that map lookups can be absent at runtime.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start a channel manager with an active plugin id whose `channelLogs` entry is absent, then have `startAccount` resolve, reject, or hit restart cleanup so the continuation dereferences `log.error`, `log.info`, or `log.warn`.

## Solution Assessment

Yes. For the immediate crash, changing the affected lifecycle calls from `log.<level>?.(...)` to `log?.<level>?.(...)` is the narrowest maintainable fix; the PR review should make sure all captured logger continuations are covered.
