## Summary

Keep open. Current `main` still has the unsafe gateway channel logger dereferences, the latest release tag still has the same pattern, and the two related fix PRs are closed unmerged with no open replacement found.

## What This Changes

The issue reports a gateway crash when `channelLogs[channelId]` is missing but async channel lifecycle handlers still dereference `log.error`, `log.info`, or `log.warn`.

## Best Possible Solution

Land a narrow gateway fix that preserves logger/runtime map invariants for deferred channel plugins and defensively treats per-channel logger lookups as optional in lifecycle continuations, with regression coverage for the missing-map case.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start a registered channel with no `channelLogs` entry, make `startAccount` reject or hit the stop-timeout path, and observe current `log.error?.(...)` or `log.warn?.(...)` throw `TypeError` before the channel failure is contained.

## Solution Assessment

Yes. The best fix is a small gateway-manager/server boundary repair: backfill or ensure per-channel log/runtime entries when plugin registries change, and still guard async lifecycle logger call sites with `log?.level?.(...)` so stale continuations cannot crash the gateway.
