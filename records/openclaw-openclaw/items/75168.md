## Summary

Keep open. Current main still contains the unsafe gateway channel logger dereferences in the async lifecycle path, the latest release tag has the same pattern, and the two related fix PRs in the discussion are closed unmerged rather than a landed implementation.

## What This Changes

The issue reports a gateway crash when async channel lifecycle handlers dereference a missing per-channel logger with `log.error?.(...)`, `log.info?.(...)`, or `log.warn?.(...)` instead of guarding `log` itself.

## Best Possible Solution

Land a narrow gateway fix in `src/gateway/server-channels.ts` that treats per-channel logger lookups as optional at every captured lifecycle use, makes the missing-entry contract visible in types or local narrowing, adds regression coverage for a missing `channelLogs` entry during channel start/error handling, and records the user-facing crash fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start a channel manager for an active channel while `channelLogs[channelId]` is missing, then make `startAccount` reject or return so the async continuation reaches `log.error?.(...)`; the current test helper does not cover that because it always seeds logger entries.

## Solution Assessment

Yes. The narrow maintainable fix is to guard the logger object itself at the affected gateway channel lifecycle call sites, while leaving unrelated module-scoped or required-typed loggers alone.
