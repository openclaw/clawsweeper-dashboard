## Summary

Keep open. Current main still has module-local native hook relay state, so the PR is addressing a real remaining failure mode from the linked issue, but the proposed patch is incomplete against current main because it shares relay registrations without sharing the newer direct relay bridge map that owns loopback server cleanup and registry-file replacement.

## What This Changes

The PR moves native hook relay registrations, invocation history, pending permission approvals, and approval-window counters into a `globalThis` symbol-backed state object and adds a unit test that registrations land in that process-global state.

## Best Possible Solution

Land a focused rebased fix that shares all in-process relay lifecycle state needed across duplicate module instances, including `relayBridges`, while preserving the direct loopback bridge security model and adding regression coverage for shared registration, bridge replacement, and cleanup.

## Reproduction Assessment

Unclear. The linked issue provides detailed logs and current main still contains the exact module-local registration/lookup path that can produce `native hook relay not found`, but this read-only review did not run a live two-module-instance reproduction.

## Solution Assessment

No. A process-global symbol registry is a plausible narrow fix, but this patch is not the best final form until it includes `relayBridges` and proves stable-id replacement cleans up the direct bridge across duplicate module instances.
