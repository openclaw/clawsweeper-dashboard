## Summary

Keep this PR open because it targets a real remaining Codex native-hook relay failure, but the current patch is not merge-ready against current main: it shares the relay registration and approval maps while leaving the newer direct relay bridge map module-local, which can make bridge replacement and cleanup diverge across duplicate module instances.

## What This Changes

The PR moves native hook relay registrations, invocation history, and permission approval state into a `globalThis` symbol and adds a unit assertion that relay registrations land in process-global state.

## Best Possible Solution

Update or replace this PR with the same narrow process-global registry approach, but include `relayBridges` in the shared state, preserve direct bridge cleanup semantics, add duplicate-module regression coverage for both relay registrations and bridge records, and add the required user-facing changelog entry.

## Reproduction Assessment

Unclear for the original runtime failure: #73723 provides detailed logs and configuration, but this read-only review did not run a live duplicate-module Codex session. The PR defect itself has a high-confidence source-level reproduction path: load two instances of the relay module, replace a stable relay id, and assert both `relays` and `relayBridges` are shared and cleaned up consistently.

## Solution Assessment

No, not in this revision. A process-global symbol is a reasonable narrow fix for duplicate in-process module instances, but the current patch must share the direct bridge state too because current main routes native hook calls through that bridge before falling back to the gateway.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Include relay bridge state in the shared registry:** `src/agents/harness/native-hook-relay.ts:153-158`
  - body: Current main stores direct relay bridge servers in `relayBridges` and relies on that map to close old loopback servers and remove registry records when a stable relay id is replaced. This patch shares the relay and approval maps but leaves `relayBridges` module-local, so a second module instance can replace a shared relay without being able to clean up the bridge owned by the first instance. Put `relayBridges` in the same shared state and cover both maps in the duplicate-module regression.
