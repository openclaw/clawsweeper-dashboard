## Summary

Keep this PR open, but do not merge this revision as-is. The shared process-global registry is a plausible fix for the duplicate-module relay lookup described in #73723, but current main now also has direct relay bridge state and cleanup that the PR's shared-state object does not cover, so the branch needs maintainer review and a conflict-aware refresh before it can be considered merge-ready.

## What This Changes

The PR moves native hook relay registrations, invocation history, pending approval promises, and approval-window counters into a `globalThis` `Symbol.for("openclaw.nativeHookRelay.state")` state object and adds a unit test that relay registrations land there.

## Best Possible Solution

Refresh this PR or replace it with an equivalent focused patch that works against current main, preserves the direct relay bridge server/registry cleanup, and adds regression proof for the duplicate-module registration/lookup path that #73723 describes.

## Reproduction Assessment

Unclear. The source still contains the exact missing-relay error path and #73723 provides detailed logs, but this read-only review did not produce a high-confidence local reproduction for two separately loaded relay modules or the stale Telegram/Codex session state.

## Solution Assessment

Unclear. A process-global symbol registry is a narrow way to share state across duplicate in-process module instances, but on current main it must be reconciled with the newer direct relay bridge state before it is clearly the best final fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Include relay bridge state in the shared registry:** `src/agents/harness/native-hook-relay.ts:153-158`
  - body: Current main now keeps direct relay bridge servers in `relayBridges` and relies on that map to close old loopback servers and remove registry records when a stable relay id is replaced. The shared-state object only moves the relay/invocation/approval maps, so after rebasing this change a second module instance could replace a shared relay without being able to close the bridge owned by the first instance. Include `relayBridges` in the same shared state, or otherwise route cross-instance replacement through the owning bridge cleanup.
  - confidence: 0.74
