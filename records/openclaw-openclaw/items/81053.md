## Summary

Keep open: this is the active implementation PR for the linked WhatsApp per-group sender-authorization feature, but the proposed resolver appears to accept a wildcard group `allowFrom` entry without honoring it at runtime.

## What This Changes

The PR adds WhatsApp per-group `allowFrom` config support across schema/types, inbound policy resolution, generated channel metadata, docs, changelog, and focused tests.

## Best Possible Solution

Keep the additive WhatsApp-owned config/runtime/docs approach, but align sender-allowlist lookup with the existing group config resolution contract before merge.

## Reproduction Assessment

Yes. The feature gap is source-reproducible on current main because the strict WhatsApp group schema omits `allowFrom` and the runtime uses only the account/channel `groupAllowFrom`; the PR-head wildcard issue is also visible from the new resolver lookup.

## Solution Assessment

No as currently written. The PR is the right owner-boundary approach, but the resolver should either honor `groups["*"].allowFrom` as the existing default group config or explicitly reject/document that wildcard sender overrides are unsupported.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Honor wildcard group allowFrom overrides:** `extensions/whatsapp/src/inbound-policy.ts:143`
  - body: `groups["*"]` is already a valid WhatsApp group config default that admits all groups for that scope. This new lookup only checks the concrete JID, so a newly accepted `groups["*"].allowFrom` is ignored and sender auth falls back to channel/account `groupAllowFrom` instead. Include the wildcard entry in this lookup, or reject/document it so operators do not configure a sender restriction that has no effect.
  - confidence: 0.82
