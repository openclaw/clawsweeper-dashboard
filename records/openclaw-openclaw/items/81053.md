## Summary

Keep open. This is a focused implementation PR for a real WhatsApp feature gap, but it needs one access-control fix before merge because the per-group allowlist is applied to ingress authorization while downstream WhatsApp context filtering still uses the channel/account-level groupAllowFrom.

## What This Changes

The PR adds WhatsApp per-group allowFrom config support, runtime resolver wiring, generated channel metadata, docs, changelog, and focused schema/resolver tests.

## Best Possible Solution

Land a WhatsApp-scoped additive config/runtime/docs change after reusing the effective per-group sender allowlist everywhere WhatsApp group sender authorization is consulted, including supplemental context visibility.

## Reproduction Assessment

Yes for the feature gap: current main's strict WhatsApp group schema omits allowFrom, and runtime passes only the account/channel groupAllowFrom. For the PR finding, source inspection of PR head shows context filtering still consumes inboundPolicy.groupAllowFrom.

## Solution Assessment

No as currently patched. The new resolver is the right seam for ingress authorization, but the same effective per-conversation allowlist needs to feed WhatsApp history and quoted-reply visibility.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Use per-group allowlists for context filtering:** `extensions/whatsapp/src/inbound-policy.ts:167-172`
  - body: The new effectiveGroupAllowFrom is only passed to resolveStableChannelMessageIngress. Later in processMessage, visible group history and quoted-reply filtering still use inboundPolicy.groupAllowFrom, so contextVisibility: "allowlist" can include or exclude supplemental context using the channel/account allowlist instead of the per-group override. Please thread the same effective per-conversation list into those context-visibility calls and cover it with a regression test.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Needs attention: the access-control change can leave supplemental group context filtered by the old channel/account allowlist instead of the per-group override.

Concerns:

- **[medium] Supplemental context uses stale group allowlist:** `extensions/whatsapp/src/auto-reply/monitor/process-message.ts:286`
  - body: When per-group allowFrom is narrower than channel-level groupAllowFrom, allowlisted group history or quotes can still be selected using the broader channel/account list in contextVisibility allowlist mode. That can expose context to the agent from senders who are blocked by the new per-group policy.
