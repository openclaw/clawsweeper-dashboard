## Summary

Keep this PR open: the bug is plausible, but the current patch downgrades matching search failures after any earlier deliverable cron payload, so it can mark a required failed search as a successful cron run and deliver partial output.

## What This Changes

The PR changes cron payload outcome resolution to treat matching trailing search-failure payloads as nonfatal after earlier successful cron output, with helper tests for search-miss and non-search-error cases.

## Best Possible Solution

Carry a typed optional/non-terminal signal from the scanner or payload builder, or otherwise prove the preceding payload is final scanner output, then attach real cron-run proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence helper-level reproduction: current main marks a final error payload fatal, while the PR adds a unit case for success text followed by a matching search-failure payload. I did not run tests because this review is read-only and dependency/test commands can write artifacts.

## Solution Assessment

No, this is not yet the best fix because it infers optional scanner intent from rendered text plus any prior deliverable payload. A narrower fix should preserve typed metadata from the producer or otherwise prove the earlier output is final scanner output, not progress text.
