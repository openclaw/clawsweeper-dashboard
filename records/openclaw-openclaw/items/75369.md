## Summary

This PR should stay open: the latest head preserves the prior current-main safeguards and now has real smoke proof in a PR comment, but the new sanitized hook task still lets caller-supplied requester metadata override host-owned requester identity in a public plugin contract.

## What This Changes

The PR adds a typed `sessions_send` plugin hook with direct/delegated handling, sanitizes delegated announce metadata, exports SDK hook types, updates hook docs/API hashes, and adds regression coverage.

## Best Possible Solution

Land one consolidated hook PR that keeps the current `sessions_send` safeguards, keeps requester identity host-owned, documents trusted context versus raw params, and preserves the SDK baseline, docs, changelog, and regression coverage.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: in the PR head, `readSessionsSendHookTask` uses caller-supplied `task.requester` before trusted defaults, so a tool caller can spoof requester metadata exposed to plugins.

## Solution Assessment

No. The hook direction is plausible, but the mergeable version should keep host-owned requester identity in `ctx` or sanitized defaults instead of letting raw task metadata override it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep requester identity host-owned:** `src/agents/tools/sessions-send-tool.ts:153-160`
  - body: These fields prefer `params.task.requester.*` over the trusted requester values passed in from the tool context. A caller can spoof `event.task.requester.sessionKey` or `channel`, and plugin authors may reasonably treat the sanitized task as trusted metadata. Populate `task.requester` only from the host-owned defaults, or remove it and rely on `ctx` for trusted requester identity.
  - confidence: 0.84
