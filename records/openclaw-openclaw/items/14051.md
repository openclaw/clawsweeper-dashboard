## Summary

Keep open. Current main still lacks a heartbeat idle-timeout/activity-window config and the heartbeat runner remains interval, wake, active-hours, and busy-lane driven rather than gated on recent allowed inbound user activity; prior review comments and the closed duplicate keep this as the canonical tracker.

## What This Changes

Add an optional heartbeat idle timeout so periodic heartbeats start after user activity and pause after a configured inactivity window.

## Best Possible Solution

Define an additive heartbeat activity-window contract, then wire it through exported types, strict schema, config help/docs, config-honor inventory, runner or scheduler gating, and regression coverage while preserving active-hours, wake retry, busy-lane, lightContext, isolatedSession, delivery, and per-agent merge behavior.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence the requested feature is absent because strict config rejects the proposed idle fields and the runner/scheduler do not gate heartbeats on recent allowed inbound activity.

## Solution Assessment

Unclear. Activity-based gating is plausible, but maintainers should choose the public config name, units, defaults, allowed-sender semantics, and scheduling-versus-execution boundary before implementation.
