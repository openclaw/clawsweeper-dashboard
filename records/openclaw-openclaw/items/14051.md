## Summary

Keep open: current main and v2026.6.8 still lack activity-based heartbeat idle-timeout behavior, while the open missing-HEARTBEAT.md PRs only address a narrower token-burn path.

## What This Changes

The issue requests a heartbeat idle-timeout option that starts heartbeat activity after user interaction and pauses periodic heartbeat turns after a configurable inactivity window.

## Best Possible Solution

Keep this as the canonical tracker, choose the public heartbeat activity/idle contract, then gate scheduled periodic heartbeats on recent allowed inbound activity while preserving event-driven wakes, activeHours, busy deferral, lightContext, isolatedSession, delivery, and per-agent behavior.

## Reproduction Assessment

Yes, at source level: current config/schema/docs and heartbeat runtime can be traced and contain no recent-activity idle timeout. This proves a missing capability rather than a broken existing contract.

## Solution Assessment

No implementation path is ready yet: the requested outcome is useful, but maintainers still need to choose the config name, activity owner, allowed-sender scope, and wake exceptions before code should be queued.
