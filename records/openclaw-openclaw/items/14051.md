## Summary

Keep this open as the canonical tracker: current main still exposes periodic, active-hours, delivery, timeout, light-context, isolated-session, and busy-lane heartbeat controls, but not activity-based start/stop or idle-timeout gating. The maintainer comment already kept this issue open, and the related #59567 was closed as a duplicate of this tracker.

## What This Changes

The issue requests a heartbeat idle-timeout option that runs heartbeats only after recent allowed inbound user activity and pauses them after inactivity.

## Best Possible Solution

Add a maintainer-approved heartbeat idle/activity contract across exported config types, strict schema, generated/config-honor docs, and the runner so heartbeat execution is gated by recent real user/channel activity without regressing activeHours, busy-lane retry, wake, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current main lacks the requested config field and activity-gated heartbeat behavior.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact config contract and runtime boundary need maintainer product/API decision before a narrow implementation is safe.
