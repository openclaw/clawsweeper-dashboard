## Summary

Keep open. Current main still lacks an activity-based heartbeat idle timeout: the proposed config key is not part of the public heartbeat contract, strict config parsing would reject it, and the runner/scheduler remain cadence and wake driven rather than gated on recent allowed inbound activity.

## What This Changes

Add a heartbeat idle-timeout config that starts or resumes heartbeats after user activity and pauses them after a configurable inactivity window.

## Best Possible Solution

Keep this as the canonical heartbeat idle-timeout tracker and add a maintainer-approved heartbeat activity/idle config contract plus runtime gating at the agent/session/channel boundary while preserving existing interval, active-hours, wake, busy-retry, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Yes, source-reproducible for the missing capability: current main rejects the proposed idle-timeout key through the strict heartbeat schema and has no recent-activity gate in the runner or scheduler. This is a feature request rather than a failing existing contract.

## Solution Assessment

Unclear. Activity-based gating is a plausible end state, but maintainers should decide the public config shape, units, defaults, and allowed-activity boundary before implementation.
