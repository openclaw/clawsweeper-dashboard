## Summary

Keep open. Current main still lacks the requested activity-based heartbeat idle timeout: the public heartbeat config has no idle-timeout key, the strict schema would reject the proposed option, and the runner remains interval/wake driven without checking recent allowed inbound activity.

## What This Changes

Request to add heartbeat idle timeout/activity gating so periodic heartbeats start after user activity and pause after an inactivity window.

## Best Possible Solution

Implement a maintainer-approved heartbeat activity/idle contract across config types, strict schema, generated/help docs, runtime scheduling or execution, and regression tests while preserving existing interval, active-hours, busy-lane, wake, delivery, and per-agent merge behavior.

## Reproduction Assessment

Yes for the missing requested behavior: static inspection of current main shows the proposed config is rejected by the strict heartbeat schema and the runner has no recent-activity idle gate. This is a feature-gap reproduction, not a broken existing-contract bug.

## Solution Assessment

Unclear. The direction is plausible, but the exact public config name, activity source, and scheduler semantics need maintainer choice before implementation; `lastInteractionAt` may be useful but should not silently conflate session expiry with heartbeat policy.
