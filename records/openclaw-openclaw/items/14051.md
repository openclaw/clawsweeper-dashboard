## Summary

Keep open: current main still lacks a heartbeat idle-timeout/activity-window config and the runner remains interval, wake, active-hours, and busy-lane driven rather than gated on recent allowed inbound user activity.

## What This Changes

The issue requests a heartbeat idle-timeout option that starts heartbeats after user activity and pauses them after a configured inactivity window.

## Best Possible Solution

Define an additive heartbeat activity-window contract, then wire it through exported types, strict schema, config help/docs, config-honor inventory, runner or scheduler gating, and regression coverage while preserving existing active-hours, wake retry, busy-lane, lightContext, isolatedSession, delivery, and per-agent merge behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence the requested feature is absent: strict config rejects the proposed idle fields and heartbeat runtime does not gate on recent allowed inbound activity.

## Solution Assessment

Unclear. Activity-based heartbeat gating is plausible, but maintainers should choose the public config name, units/defaults, allowed-sender semantics, and scheduler-versus-runner boundary before implementation.
