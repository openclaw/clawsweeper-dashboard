## Summary

Keep open. Current main still does not implement activity-based heartbeat idle timeout: heartbeat config/schema/docs expose interval, active-hours, routing, timeout, context, isolation, and busy-lane deferral, but no idle-timeout/activity window, and the runner remains interval/wake driven without checking recent allowed inbound user activity. The prior maintainer comment also explicitly keeps this as the canonical tracker, with #59567 closed as a duplicate.

## What This Changes

Add a heartbeat idle-timeout option so periodic heartbeats start with user activity and pause after a configurable period of no allowed inbound activity.

## Best Possible Solution

Keep this issue open as the canonical heartbeat idle-timeout tracker. A good fix should add a maintainer-approved heartbeat activity/idle config key across exported types, strict schema, labels/help/generated docs, config docs, and the heartbeat guide, then gate heartbeat scheduling or execution on recent allowed inbound user/channel activity at the correct agent/session/channel boundary while preserving current interval, active-hours, wake, busy-lane retry, lightContext, isolatedSession, delivery, and per-agent merge behavior.

## Reproduction Assessment

Yes. The missing behavior is reproducible by static current-main inspection: the proposed config key is absent from the strict heartbeat schema, and runHeartbeatOnce/startHeartbeatRunner have no lastInteractionAt or activity-window gate before heartbeat execution.

## Solution Assessment

Unclear. The requested direction is valid, but the exact config/API shape and activity boundary need maintainer choice before a safe automated implementation; reusing lastInteractionAt may be part of the solution, but it must not conflate session reset freshness with heartbeat scheduling without explicit design.
