## Summary

Keep open. Current main still has interval, active-hours, and busy-lane heartbeat behavior, but no public idle-timeout/activity-window config or recent allowed-sender activity gate; prior review context also treats this as the canonical heartbeat idle-timeout tracker.

## What This Changes

Request an activity-based heartbeat idle timeout so heartbeats pause after recent user inactivity and resume after the next allowed inbound message.

## Best Possible Solution

Define an additive heartbeat activity-window contract, then wire it through config types, strict schema, schema labels/help/generated docs, config-honor inventory, heartbeat docs, runner or scheduler gating, and regression coverage while keeping retry-delay and Talk Mode idle work separate.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence the requested feature is absent because strict config rejects the proposed idle fields and the runner/scheduler do not gate heartbeats on recent allowed inbound activity.

## Solution Assessment

Unclear. Activity-based gating is plausible, but maintainers should choose the public config name, defaults, allowed-sender semantics, and scheduling-versus-execution boundary before implementation.
