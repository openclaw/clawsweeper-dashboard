## Summary

Keep open. Current main still lacks heartbeat activity-window config, schema, docs, and runtime gating; related discussion continues to treat this as the canonical heartbeat idle-timeout tracker.

## What This Changes

Requests a heartbeat option that starts after allowed inbound user activity and pauses automatically after a configurable idle window.

## Best Possible Solution

Define an additive heartbeat activity-window contract, then wire it through config types, strict schema, labels/help/generated docs, config-honor inventory, heartbeat docs, runner or scheduler gating, and regression coverage while keeping retry-delay and Talk Mode idle work separate.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence the requested feature is absent because strict config rejects the proposed key and the runner/scheduler do not gate heartbeats on recent allowed inbound activity.

## Solution Assessment

Unclear. Activity-based gating is plausible, but maintainers should choose the public config name, defaults, allowed-sender semantics, and scheduling-versus-execution boundary before implementation.
