## Summary

Keep open. Current main still lacks the requested activity-based heartbeat idle-timeout config, runtime gate, and docs, and existing discussion treats this as the canonical heartbeat idle-timeout tracker rather than a duplicate of the adjacent retry-delay or Talk Mode idle reports.

## What This Changes

Request an activity-based heartbeat idle timeout so periodic heartbeats pause after a configured window without allowed inbound user activity.

## Best Possible Solution

Define a maintainer-approved heartbeat activity-window contract, then wire it through config types, strict schema, labels/help/generated docs, config-honor inventory, heartbeat docs, runner or scheduler gating, and regression coverage without folding in the separate Talk Mode or heartbeat retry-delay items.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence the requested feature is absent because strict config rejects the proposed key and the runner/scheduler do not gate heartbeats on recent allowed inbound activity.

## Solution Assessment

Unclear. Activity-based gating is plausible, but maintainers should choose the public config name, defaults, allowed-sender semantics, and scheduling-versus-execution boundary before implementation.
