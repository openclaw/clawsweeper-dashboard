## Summary

Current main still does not implement activity-based heartbeat idle timeout: the public heartbeat config/schema/docs remain periodic/active-hours/busy-skip based, and the runner/scheduler never gate on recent allowed inbound activity. Prior discussion still treats this as the canonical tracker, with the closest duplicate already closed and related retry-delay/Talk idle work tracked separately.

## What This Changes

The issue requests a heartbeat idle-timeout option that starts heartbeats after user activity and pauses them after a configurable period without allowed inbound messages.

## Best Possible Solution

Define an additive heartbeat activity-window contract, then wire it through config types, strict schema, labels/help/generated docs, config-honor inventory, heartbeat docs, runner or scheduler gating, and regression tests while leaving retry-delay and Talk Mode idle work to their separate trackers.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence the requested feature is absent because strict config rejects the proposed key and the runner/scheduler do not gate heartbeats on recent allowed inbound activity.

## Solution Assessment

Unclear. Activity-based gating is a plausible product direction, but maintainers should choose the public config contract and whether the idle window belongs in scheduling, run preflight, or a session/channel activity ledger before implementation.
