## Summary

Keep open. Current main rejects the proposed heartbeat idle-timeout key via the strict schema and still schedules heartbeats by interval/wake without recent allowed inbound activity gating; the live discussion and labels treat this as the canonical tracker needing maintainer product direction.

## What This Changes

Request an activity-based heartbeat option that starts after user conversation activity and pauses automatically after an idle window.

## Best Possible Solution

Keep this as the canonical heartbeat idle-timeout tracker and add a maintainer-approved config contract plus runtime gating at the correct agent/session/channel activity boundary while preserving existing interval, active-hours, wake, busy-retry, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Yes, source-reproducible for the missing capability: current main rejects the proposed idle-timeout key through the strict heartbeat schema and has no recent-activity gate in the runner or scheduler. This is a feature request rather than a failing existing contract.

## Solution Assessment

Unclear. Activity-based gating is a plausible end state, but the exact config shape, units, defaults, and definition of allowed inbound activity need maintainer direction before implementation.
