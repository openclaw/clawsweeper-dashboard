## Summary

Keep open: current main and latest release still lack the requested heartbeat idle-timeout config, schema, docs, and runtime gate. The related cron idle PR is open and cron-scoped, so it does not replace this issue as the canonical heartbeat activity/idle tracker.

## What This Changes

Request an activity-based heartbeat idle-timeout option so heartbeats start after allowed inbound activity and pause after a configured inactivity window.

## Best Possible Solution

Add a maintainer-approved heartbeat idle/activity config contract and runtime gate at the agent/session/channel boundary, while preserving existing interval, activeHours, wake, busy-deferral, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Yes. The missing capability is source-reproducible: current main and v2026.5.22 reject the proposed idle-timeout key through strict heartbeat schema and have no recent-activity gate in the heartbeat runner or scheduler.

## Solution Assessment

Unclear. Activity-based gating is a plausible end state, but maintainers should decide the public config shape, units, defaults, disabled behavior, and allowed-activity boundary before implementation.
