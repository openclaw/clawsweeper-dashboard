## Summary

Keep open: current main and latest release still lack activity-based heartbeat idle-timeout behavior, and the open missing-HEARTBEAT.md PR only addresses a narrower default token-burn path.

## What This Changes

The issue requests heartbeat scheduling that starts after conversation activity and pauses after a configured idle window to reduce token use while users are away.

## Best Possible Solution

Keep this as the canonical heartbeat activity/idle-timeout tracker, choose the public config and activity-source contract, then gate scheduled periodic heartbeats on recent allowed inbound activity while preserving event wakes, activeHours, busy deferral, lightContext, isolatedSession, delivery, and per-agent behavior.

## Reproduction Assessment

Yes, at source level: current config/schema/docs and heartbeat runtime can be traced and contain no recent-activity idle timeout. This proves a missing capability rather than a broken existing contract.

## Solution Assessment

No implementation path is ready yet: the requested outcome is useful, but maintainers still need to choose the config name, activity owner, allowed-sender scope, and wake exceptions before code should be queued.
