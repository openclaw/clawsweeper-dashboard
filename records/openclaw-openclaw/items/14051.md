## Summary

Keep open. Current main still lacks the requested activity-based heartbeat idle timeout, and the earlier discussion already treats this as the canonical tracker rather than a duplicate of the narrower busy-retry or Talk Mode idle-timeout items.

## What This Changes

The issue requests a configurable heartbeat idle timeout that starts heartbeats after user activity and pauses them after a period without allowed inbound messages.

## Best Possible Solution

Keep this as the canonical heartbeat idle-timeout tracker and add a maintainer-approved config/schema/docs contract plus runtime gating based on recent allowed inbound activity at the correct session or channel boundary.

## Reproduction Assessment

Yes, source-reproducible. Current main rejects the proposed idle-timeout key via the strict heartbeat schema and the runner/scheduler paths have no recent-activity gate before executing interval heartbeats.

## Solution Assessment

Unclear. Activity-based gating is a plausible end state, but the exact public config contract and activity scope need maintainer direction before implementation.
