## What This Changes

The issue requests an activity-based heartbeat idle timeout so periodic heartbeats start after user activity and pause after a configured idle window.

## Best Possible Solution

Keep this as the canonical tracker, choose the public heartbeat activity/idle contract, then gate scheduled periodic heartbeats on recent allowed inbound activity while preserving existing event-driven wakes and heartbeat controls.

## Reproduction Assessment

Yes, at source level: current config/schema/docs and heartbeat runtime can be traced and contain no recent-activity idle timeout. This proves a missing capability rather than a broken existing contract.

## Solution Assessment

No ready implementation path yet: the requested outcome is useful, but maintainers still need to choose the config name, activity owner, allowed-sender scope, and wake exceptions before code should be queued.
