## Summary

Keep open: current main still lacks an activity-based heartbeat idle timeout, and the open related missing-HEARTBEAT.md PR is only partial overlap, not a safe canonical implementation of this request.

## What This Changes

Add a heartbeat idle-timeout option so scheduled heartbeats start after user activity and pause after a configurable period without allowed inbound messages.

## Best Possible Solution

Keep this as the canonical tracker, choose the public activity/idle contract, then gate scheduled periodic heartbeats on a canonical recent allowed-inbound activity timestamp while preserving event-driven wakes and existing heartbeat controls.

## Reproduction Assessment

Yes, at source level: current config/schema/docs and heartbeat runtime can be traced and contain no recent-activity idle timeout. This proves a missing capability rather than a broken existing contract.
