## What This Changes

Request adds a heartbeat idle-timeout option so periodic heartbeat runs pause after recent allowed inbound user activity goes stale.

## Best Possible Solution

Keep this as the canonical heartbeat idle-timeout tracker and make the eventual fix add one approved config/schema/docs contract plus runtime gating at the correct agent/session/channel activity boundary.

## Reproduction Assessment

Yes for the current missing behavior: source inspection shows the proposed heartbeat key would be rejected by strict schema, and the runner/scheduler have no recent-activity gate. I did not run a live gateway scenario because this review is read-only.

## Solution Assessment

Unclear. Activity-based gating is a plausible end state, but maintainers should decide the public config shape, defaults, disabled behavior, and allowed-activity boundary before implementation.
