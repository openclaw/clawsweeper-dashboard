## Summary

Keep open: current main and the latest release still lack an activity-based heartbeat idle timeout, and the open missing-HEARTBEAT.md PR is only a narrower partial overlap, not a canonical implementation of this request.

## What This Changes

The issue requests a heartbeat idle-timeout option that starts periodic heartbeats after real inbound activity and pauses them after a configurable idle window.

## Best Possible Solution

Keep this as the canonical tracker, choose the public heartbeat activity/idle contract, then gate scheduled periodic heartbeats on recent allowed inbound activity while preserving existing event-driven wakes and heartbeat controls.

## Reproduction Assessment

Yes, at source level: current config/schema/docs and heartbeat runtime can be traced and contain no recent-activity idle timeout. This proves a missing capability rather than a broken existing contract.

## Solution Assessment

No, not as a ready implementation path yet: the requested outcome is useful, but maintainers still need to choose the config name, activity owner, and wake exceptions before code should be queued.
