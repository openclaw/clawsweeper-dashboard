## Summary

Keep open. Current main still lacks activity-based heartbeat idle-timeout behavior, and the issue remains the canonical tracker for this feature rather than a duplicate of adjacent heartbeat retry or Talk Mode idle-timeout work.

## What This Changes

Add an optional heartbeat idle-timeout/activity window so heartbeat runs start after user activity and pause after a configurable period without allowed inbound activity.

## Best Possible Solution

Define a maintainer-approved heartbeat activity-window contract, then wire it through exported config types, strict schema, help/labels/generated docs, config-honor inventory, runtime scheduling/execution gates, and regression coverage while preserving existing interval, active-hours, wake retry, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence the requested feature is absent: strict heartbeat config omits the key and the runner/scheduler do not gate on recent allowed inbound activity.

## Solution Assessment

Unclear. Activity-based gating is plausible, but maintainers need to choose the config name, default behavior, allowed-sender semantics, and agent/session/channel boundary before implementation.
