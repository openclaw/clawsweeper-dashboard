## Summary

Keep open. Current main and the latest release still lack activity-based heartbeat idle timeout: heartbeat config/schema/docs expose interval, active-hours, delivery, busy deferral, isolation, and reasoning controls, while the runner and scheduler still run from interval/wake state rather than recent allowed inbound activity. The related cron PR at https://github.com/openclaw/openclaw/pull/55982 overlaps the activity-policy question but does not implement heartbeat idle timeout, and https://github.com/openclaw/openclaw/issues/59567 was already closed as a duplicate of this canonical tracker.

## What This Changes

Add a heartbeat config option that pauses scheduled heartbeats after configurable inactivity and resumes them after allowed inbound user activity.

## Best Possible Solution

Keep this as the canonical heartbeat idle-timeout tracker and add a maintainer-approved activity/idle config contract plus runtime gating at the correct agent, session, and channel boundary while preserving existing interval, active-hours, wake, busy-skip, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Yes, source-reproducible for the missing capability: current main and v2026.5.20 reject the proposed idle-timeout key through strict heartbeat schema and have no recent-activity gate in the heartbeat runner or scheduler. This is a feature request rather than a failing existing contract.

## Solution Assessment

Unclear. Activity-based gating is a plausible end state, but maintainers should decide the public config shape, units, defaults, and allowed-activity boundary before implementation.
