## Summary

Keep open. Current main still lacks a heartbeat idle-timeout/activity-window config and does not gate heartbeat runs on recent allowed inbound activity; prior discussion also keeps this as the canonical tracker rather than a duplicate or already-implemented request.

## What This Changes

Request an activity-based heartbeat mode that starts after user conversation activity and pauses automatically after an idle timeout.

## Best Possible Solution

Define an additive heartbeat activity-window contract, then wire it through exported types, strict schema, config help/labels/generated docs, config-honor inventory, heartbeat docs, runner or scheduler gating, and regression coverage while keeping retry-delay and Talk Mode idle work separate.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence the requested feature is absent because strict config rejects the proposed idle fields and the runner/scheduler do not gate heartbeats on recent allowed inbound activity.

## Solution Assessment

Unclear. Activity-based gating is plausible, but maintainers should choose the public config name, units, defaults, allowed-sender semantics, and scheduling-versus-execution boundary before implementation.
