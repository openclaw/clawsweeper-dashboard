## Summary

Keep open. Current main and the latest release still lack activity-based heartbeat idle-timeout behavior: heartbeat config/schema have no idle/activity key, and the runner/scheduler remain interval, active-hours, wake, busy, and cooldown driven instead of gated on recent allowed inbound activity.

## What This Changes

The issue requests a heartbeat idle-timeout option so periodic heartbeats start with conversation activity and pause after configurable inactivity.

## Best Possible Solution

Keep this as the canonical heartbeat idle-timeout tracker and add a maintainer-approved activity/idle config contract plus runtime gating at the correct agent/session/channel boundary while preserving existing interval, active-hours, wake, busy-skip, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Yes, source-reproducible for the missing capability: current main rejects the proposed idle-timeout key through the strict heartbeat schema and the heartbeat runner/scheduler have no recent-activity gate. This is a feature request rather than a failing existing contract.

## Solution Assessment

Unclear. Activity-based gating is a plausible end state, but maintainers should decide the public config shape, units, defaults, and allowed-activity boundary before implementation.
