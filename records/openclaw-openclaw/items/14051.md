## Summary

Keep open: current main still lacks a heartbeat-specific idle-timeout config key and runtime gate based on recent allowed inbound activity. The adjacent cron skip-when-idle PR is open and does not implement this heartbeat behavior, while the closed idle-only heartbeat report points back to this issue as the canonical tracker.

## What This Changes

Add activity-based heartbeat behavior so periodic heartbeats pause after a configurable idle window and resume after allowed inbound user activity.

## Best Possible Solution

Keep this issue as the canonical heartbeat idle-timeout tracker and implement a maintainer-approved heartbeat config/schema/docs/runtime gate using recent allowed inbound activity while preserving existing interval, active-hours, wake, busy-lane, light-context, isolated-session, and delivery behavior.

## Reproduction Assessment

Yes at source level: current main exposes no idle-timeout heartbeat key in types/schema/docs and the runner does not gate on recent inbound activity. No live runtime reproduction was run because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the public config shape and activity boundary. A maintainable fix should reuse canonical session or channel activity state rather than adding a parallel heartbeat-only activity store without a clear owner.
