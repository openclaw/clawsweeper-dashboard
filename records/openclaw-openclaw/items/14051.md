## Summary

Keep open. Current main still does not expose or honor activity-based heartbeat idle timeout behavior, and prior discussion treats this as the canonical tracker rather than a duplicate of the adjacent retry-spacing issue.

## What This Changes

Request an optional heartbeat idle-timeout/activity window so periodic heartbeats pause after recent allowed-user activity goes quiet and resume on the next inbound message.

## Best Possible Solution

Add a maintainer-approved heartbeat activity-window config through exported types, strict schema, labels/help/generated docs, config-honor inventory, runtime gating on recent allowed inbound user/channel activity, and regression coverage that preserves existing interval, active-hours, wake retry, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence the requested feature is absent: the strict schema omits the key and the runner schedules by interval/wake without `lastInteractionAt` or activity-window gating.

## Solution Assessment

Unclear. Activity-based gating is a plausible direction, but the config shape, default behavior, allowed-sender semantics, and session/channel boundary need maintainer approval before implementation.
