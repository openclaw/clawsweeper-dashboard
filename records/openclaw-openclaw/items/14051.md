## Summary

Keep open. Current main still lacks activity-based heartbeat idle timeout behavior, and the current labels/comments already route this as a maintainer product decision rather than a stale duplicate.

## What This Changes

Request adds a heartbeat idle-timeout option so heartbeats start after user activity and pause after a configured inactive window.

## Best Possible Solution

Keep this as the canonical tracker and add a maintainer-approved config/schema/docs contract plus runtime gating on recent allowed inbound activity without regressing interval, active-hours, busy retry, wake, isolatedSession, lightContext, and delivery behavior.

## Reproduction Assessment

Yes, source-reproducible. Current main rejects the proposed idle-timeout key via the strict heartbeat schema and has no recent-activity gate in the runner or scheduler path.

## Solution Assessment

Unclear. Activity-based heartbeat gating is a plausible end state, but the exact public config and activity scope need maintainer direction before implementation.
