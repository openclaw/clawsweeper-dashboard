## Summary

Keep open: current main and v2026.5.20 still lack a heartbeat idle-timeout config key, strict-schema support, and a runtime gate based on recent allowed inbound activity; related duplicate reports point here as the canonical tracker.

## What This Changes

Request adding a heartbeat idle-timeout/activity gate so scheduled heartbeats pause after user inactivity and resume on the next conversation.

## Best Possible Solution

Keep this as the canonical heartbeat idle-timeout tracker and add a maintainer-approved activity/idle config contract plus runtime gating at the correct agent/session/channel boundary while preserving existing heartbeat cadence, active-hours, wake, busy-deferral, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Yes, source-reproducible for the missing capability: current main and v2026.5.20 reject the proposed idle-timeout key through strict heartbeat schema and have no recent-activity gate in the heartbeat runner or scheduler. This is a feature request rather than a failing existing contract.

## Solution Assessment

Unclear. Activity-based gating is a plausible end state, but maintainers should decide the public config shape, units, defaults, and allowed-activity boundary before implementation.
