## Summary

Keep open. Current main still lacks activity-based heartbeat idle timeout behavior, and the issue remains the canonical tracker rather than a duplicate of the adjacent heartbeat retry-spacing bug.

## What This Changes

The issue requests an activity-based heartbeat idle timeout so scheduled heartbeats pause after user inactivity and resume on allowed inbound activity.

## Best Possible Solution

Add a maintainer-approved heartbeat activity-window config through exported types, strict schema, labels/help/generated docs, config-honor inventory, runtime gating on recent allowed inbound user/channel activity, and regression coverage that preserves existing interval, active-hours, wake retry, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence the requested feature is absent: strict heartbeat config omits the key and the runner/scheduler do not gate on recent allowed inbound activity.

## Solution Assessment

Unclear. Activity-based gating is a plausible direction, but the config name, default behavior, allowed-sender semantics, and agent/session/channel boundary need maintainer product/API direction before implementation.
