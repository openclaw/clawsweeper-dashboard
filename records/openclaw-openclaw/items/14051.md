## Summary

Keep open. Current main and the latest release still lack heartbeat activity-based idle timeout: config/schema expose periodic, active-hours, delivery, busy-skip, and isolation controls, while runtime remains interval/wake/cooldown driven and does not gate heartbeats on recent allowed inbound activity.

## What This Changes

Requested activity-based heartbeat idle timeout so heartbeats pause after user inactivity and resume after later activity.

## Best Possible Solution

Keep this as the canonical heartbeat idle-timeout tracker and add a maintainer-approved activity/idle config contract plus runtime gating at the correct agent/session/channel boundary while preserving interval, active-hours, wake, busy-skip, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Yes, source-reproducible for the missing capability: current main and v2026.5.19 reject the proposed idle-timeout key through strict heartbeat schema and have no recent-activity gate in heartbeat runner/scheduler. This is a feature request rather than a failing existing contract.

## Solution Assessment

Unclear. Activity-based gating is a plausible end state, but maintainers should decide the public config shape, units, defaults, and allowed-activity boundary before implementation.
