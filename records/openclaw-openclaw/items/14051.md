## Summary

Current main still has no activity-based idle window for heartbeats: the public config lacks such a field and the runner gates periodic work on cadence, active hours, queue/busy state, and due work rather than recent allowed inbound activity. Live issue and PR search found no merged replacement, while the narrower idle-only report was closed into this canonical tracker, so the requested product decision remains necessary.

## What This Changes

Add a configurable inactivity window so periodic heartbeats start after allowed inbound user activity and pause after the user has been idle.

## Best Possible Solution

Keep this issue as the canonical tracker, choose a maintainer-approved activity scope and config contract, then gate only scheduled periodic heartbeats on a canonical recent allowed-inbound timestamp while preserving immediate event wakes and existing cadence, active-hours, routing, busy-retry, isolation, and per-agent behavior.

## Reproduction Assessment

Yes at source level: current main's accepted heartbeat fields and scheduling/run gates can be traced and contain no recent-user-activity timeout. This proves a missing capability, not a failure of an already-defined behavior.

## Solution Assessment

The outcome is useful, but the proposed idleTimeoutMinutes field alone is not yet the best proven design; maintainers should first select the activity owner and apply the gate only to periodic scheduling rather than all wake-driven heartbeat work.
