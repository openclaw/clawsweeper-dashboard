## Summary

Keep open. Current main still lacks an activity-based heartbeat idle-timeout config, runtime gate, and docs, and the related discussion keeps this as the canonical heartbeat idle-timeout feature rather than a duplicate of https://github.com/openclaw/openclaw/issues/40611, https://github.com/openclaw/openclaw/issues/46844, or closed duplicate https://github.com/openclaw/openclaw/issues/59567.

## What This Changes

Request an activity-based heartbeat idle timeout so heartbeats start after user activity and pause after a configured inactivity window.

## Best Possible Solution

Define a maintainer-approved heartbeat activity-window contract, then wire it through exported config types, strict schema, help/labels/generated docs, config-honor inventory, runtime scheduling or execution gates, and regression coverage while preserving interval, active-hours, wake retry, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence the requested feature is absent: strict heartbeat config omits the key and the runner/scheduler do not gate on recent allowed inbound activity.

## Solution Assessment

Unclear. Activity-based gating is plausible, but maintainers need to choose the config name, default behavior, allowed-sender semantics, and agent/session/channel boundary before implementation.
