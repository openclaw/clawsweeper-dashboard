## Summary

Keep open: current main and v2026.5.22 still lack a heartbeat idle-timeout config/schema/docs contract, and the runner remains interval/wake driven rather than gated by recent allowed inbound activity. Related searches keep this as the canonical heartbeat idle-timeout tracker; cron-only idle work and PAUSED-file work do not solve heartbeat start/stop behavior.

## What This Changes

The issue requests a heartbeat option that starts after user activity and pauses automatically after a configured idle window.

## Best Possible Solution

Keep this as the canonical heartbeat idle-timeout tracker and add a maintainer-approved activity/idle config contract plus runtime gating at the correct agent/session/channel boundary, preserving existing interval, activeHours, wake, busy-deferral, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Yes, source-reproducible for the missing capability: current main and v2026.5.22 reject the proposed idle-timeout key through strict heartbeat schema and have no recent-activity gate in the heartbeat runner or scheduler. This is a feature request rather than a failing existing contract.

## Solution Assessment

Unclear. Activity-based gating is a plausible end state, but maintainers should decide the public config shape, units, defaults, disabled behavior, and allowed-activity boundary before implementation.
