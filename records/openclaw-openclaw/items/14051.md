## Summary

Keep open. Current main and the latest release still lack activity-based heartbeat idle timeout: heartbeat config/schema/docs cover interval, active hours, delivery, busy deferral, isolation, and reasoning controls, while the runner remains interval/wake/cooldown driven and does not gate scheduled heartbeats on recent allowed inbound activity.

## What This Changes

Request adds a heartbeat idle-timeout option so heartbeats start with user activity and pause after a configured inactive window.

## Best Possible Solution

Keep this as the canonical heartbeat idle-timeout tracker and add a maintainer-approved activity/idle config contract plus runtime gating at the correct agent/session/channel boundary while preserving interval, active-hours, wake, busy-skip, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Yes, source-reproducible for the missing capability: current main and v2026.5.20 reject the proposed idle-timeout key through strict heartbeat schema and have no recent-activity gate in the heartbeat runner or scheduler. This is a feature request rather than a failing existing contract.

## Solution Assessment

Unclear. Activity-based gating is a plausible end state, but maintainers should decide the public config shape, units, defaults, and allowed-activity boundary before implementation.
