## Summary

Keep open: current main and latest release still lack an activity-based heartbeat idle-timeout contract, and the runner remains interval/wake driven rather than gated by recent allowed inbound activity.

## What This Changes

Request to add a heartbeat idle-timeout option that starts heartbeats after user activity and pauses them after a configurable inactivity window.

## Best Possible Solution

Keep this as the canonical heartbeat idle-timeout tracker and add a maintainer-approved activity/idle config contract plus runtime gating at the correct agent/session/channel boundary.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.5 reject the proposed idle-timeout key through strict heartbeat schema and have no recent-activity gate in the heartbeat runner or scheduler.

## Solution Assessment

Unclear. Activity-based gating is a plausible end state, but maintainers should decide the public config shape, defaults, disabled behavior, and allowed-activity boundary before implementation.
