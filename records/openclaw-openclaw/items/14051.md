## Summary

Keep open. Current main still lacks a heartbeat idle-timeout/activity-window config and the runner remains interval, active-hours, busy-lane, and wake driven instead of gated on recent allowed inbound user activity; the duplicate idle-only report is already closed into this canonical tracker.

## What This Changes

The issue requests an additive heartbeat option that starts periodic heartbeats on user activity and pauses them after a configurable idle window.

## Best Possible Solution

Define an additive heartbeat activity-window contract, then wire it through exported types, strict schema, config help/docs, config-honor inventory, runner or scheduler gating, and focused regression coverage while preserving active-hours, wake retry, busy-lane, lightContext, isolatedSession, delivery, and per-agent merge behavior.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence the requested feature is absent because strict heartbeat config rejects idle fields and the runner/scheduler do not gate heartbeats on recent allowed inbound activity.

## Solution Assessment

Unclear. Activity-based gating is plausible, but maintainers should choose the public config name, units, defaults, allowed-sender semantics, and scheduling-versus-execution boundary before implementation.
