## Summary

Keep open. Current main and v2026.5.19 still lack activity-based heartbeat idle timeout: heartbeat config/schema/docs expose interval, active-hours, delivery, busy-skip, and isolation controls, while runtime remains interval/wake/cooldown driven and does not gate scheduled heartbeats on recent allowed inbound activity. The related duplicate at https://github.com/openclaw/openclaw/issues/59567 was already closed into this canonical tracker, and the open cron-only PR at https://github.com/openclaw/openclaw/pull/55982 does not implement heartbeat idle timeout.

## What This Changes

The issue requests a heartbeat idle-timeout option that starts heartbeats after conversation activity and pauses them after a configurable period without allowed inbound user activity.

## Best Possible Solution

Keep this as the canonical heartbeat idle-timeout tracker and add a maintainer-approved activity/idle config contract plus runtime gating at the correct agent/session/channel boundary while preserving interval, active-hours, wake, busy-skip, delivery, lightContext, isolatedSession, and per-agent merge behavior.

## Reproduction Assessment

Yes, source-reproducible for the missing capability: current main and v2026.5.19 reject the proposed idle-timeout key through strict heartbeat schema and have no recent-activity gate in heartbeat runner or scheduler. This is a feature request rather than a failing existing contract.

## Solution Assessment

Unclear. Activity-based gating is a plausible end state, but maintainers should decide the public config shape, units, defaults, and allowed-activity boundary before implementation.
