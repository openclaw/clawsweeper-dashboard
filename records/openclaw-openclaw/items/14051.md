## What This Changes

The issue requests an activity-based heartbeat idle timeout so scheduled heartbeats pause after a configured inactivity window and resume on allowed inbound user activity.

## Best Possible Solution

Keep this as the canonical maintainer/product tracker, choose the public config and activity-source contract, then implement a heartbeat runtime gate that preserves current interval, active-hours, busy-deferral, event-wake, delivery, lightContext, isolatedSession, and session-lifecycle behavior.

## Reproduction Assessment

Yes, at source level: strict heartbeat config lacks the proposed key and the heartbeat runner currently gates on interval, active hours, and busy state rather than recent inbound activity. This proves a missing capability rather than a broken existing contract.

## Solution Assessment

No implementation path is ready for automation: the maintainable solution needs a product decision on config naming, activity source, allowed-sender scope, and wake exceptions before code changes.
