## Summary

Keep open: current main still lacks a heartbeat idle-timeout/activity-window config, and the runner is still interval, wake, active-hours, and busy-lane driven rather than gated on recent allowed inbound user activity. Existing review comments also keep this as the canonical tracker, with the related idle-only request already closed as a duplicate.

## What This Changes

Add a configurable heartbeat idle timeout so periodic heartbeats start after user activity and pause after a configured inactivity window.

## Best Possible Solution

Define an additive heartbeat activity-window contract, then wire it through exported types, strict schema, schema labels/help/docs, config-honor inventory, runner or scheduler gating, and focused regression coverage while preserving existing heartbeat semantics.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection against current main gives high confidence the requested feature is absent: strict config rejects the proposed idle fields and heartbeat runtime does not gate on recent allowed inbound activity.

## Solution Assessment

Unclear. Activity-based heartbeat gating is plausible, but maintainers should choose the public config name, units/defaults, allowed-sender semantics, and whether the gate belongs in scheduler state or run preflight before implementation.
