## Summary

Keep open: current main still lacks a WhatsApp close-path debounce drain, and this PR’s one-shot `flushAll()` can still return before late or active debounce work is quiescent. The external PR also lacks required after-fix real WhatsApp behavior proof.

## What This Changes

Adds `flushAll()` to the shared inbound debouncer, calls it during WhatsApp Web close before socket teardown, and updates affected channel/test mocks for the new debouncer API.

## Best Possible Solution

Implement a debouncer drain that waits until there are no buffered keys and no active keyed chains before closing WhatsApp, with a focused regression proving `reply` or `sendMedia` completes before socket teardown.

## Reproduction Assessment

Yes, source-reproducible: configure `messages.inbound.debounceMs > 0`, emit WhatsApp `messages.upsert`, then close while handler or timer work is pending and assert `reply` or `sendMedia` finishes before `closeInboundMonitorSocket`. I did not run a live WhatsApp session.

## Solution Assessment

No. A close-path drain is the right direction, but the proposed one-shot buffer snapshot can miss late enqueues and active `keyChains` work; a quiescence drain is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wait for debounce work to fully quiesce:** `src/auto-reply/inbound-debounce.ts:235-237`
  - body: `flushAll()` snapshots `buffers.keys()` once and awaits only that snapshot. In the WhatsApp close path, an already-running `handleMessagesUpsert` can enqueue after the snapshot, and timer-driven flushes can leave active `keyChains` work after the buffer is deleted, so `close()` can still tear down the socket before `reply` or `sendMedia` finishes.
  - confidence: 0.9
