## Summary

Keep open. Current main still has the WhatsApp debounce shutdown gap, but this PR does not fully drain debouncer work before socket teardown: the proposed `flushAll()` snapshots buffered keys once and can miss late enqueues or active keyed chains, so the intended message-loss fix remains incomplete.

## What This Changes

The PR adds a shared inbound debouncer `flushAll()` API, calls it from the WhatsApp Web inbox close path before socket teardown, and updates channel/test debouncer mocks for the new method.

## Best Possible Solution

Revise the shared debouncer into a real quiescence drain, have the WhatsApp close path await that drain before closing the socket, and add focused tests for late enqueue, active keyed-chain, and reply/sendMedia-before-close behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `debounceMs > 0`, start a WhatsApp `messages.upsert` handler, call `listener.close()` while debounce or handler work is pending, and observe that current main closes the socket before a guaranteed drain; the PR still allows late or chain-only work to outlive `flushAll()`.

## Solution Assessment

No. The proposed sequencing is directionally right, but a one-shot buffer snapshot is not the narrowest maintainable fix; the debouncer API needs to wait until both buffered keys and active keyed chains have quiesced.
