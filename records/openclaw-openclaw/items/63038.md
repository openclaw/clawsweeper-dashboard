## Summary

Keep open: current main still lacks a WhatsApp close-path debouncer drain, but this PR is not merge-ready because its one-shot `flushAll()` can miss late enqueues or already-running keyed work, and the external PR has no real WhatsApp after-fix proof.

## What This Changes

The branch adds `flushAll()` to the shared inbound debouncer, awaits it during WhatsApp Web monitor close, and updates channel/test mocks for the expanded debouncer shape.

## Best Possible Solution

Add a shared drain API that waits until buffers and active keyed chains are quiescent before WhatsApp socket teardown, then cover reply/sendMedia-before-close with focused regression tests.

## Reproduction Assessment

Yes, source-reproducible: configure WhatsApp `debounceMs > 0`, emit `messages.upsert`, begin close while debounce or handler work is pending, and the current close path can tear down the socket before reply/sendMedia is guaranteed. I did not run live WhatsApp proof in this read-only review.

## Solution Assessment

No. Awaiting before close is the right direction, but the proposed one-shot `buffers.keys()` snapshot is not sufficient because late enqueues and active keyed work can outlive `flushAll()`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wait for debounce work to fully drain:** `src/auto-reply/inbound-debounce.ts:235-237`
  - body: `flushAll()` snapshots `buffers.keys()` once and only awaits `flushKey` for that snapshot. Already-running WhatsApp handlers can enqueue after the snapshot, and timer flushes can leave active `keyChains` work with no buffer entry, so `close()` can still tear down the socket before `reply` or `sendMedia` finishes.
