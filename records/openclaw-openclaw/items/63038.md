## Summary

Keep open: the reported WhatsApp close-path loss is plausible on current main, but this PR's one-shot `flushAll()` can still return before late enqueues or active keyed debounce work finish, and the external PR still lacks real WhatsApp behavior proof.

## What This Changes

Adds `flushAll()` to the shared inbound debouncer, calls it before WhatsApp Web socket teardown, and updates affected channel/test mocks for the new API.

## Best Possible Solution

Implement a WhatsApp close-path debouncer drain that waits until pending buffers and active keyed chains are quiescent, then add a focused regression proving `reply` or `sendMedia` completes before socket teardown.

## Reproduction Assessment

Yes, source-reproducible: configure `messages.inbound.debounceMs > 0`, emit WhatsApp `messages.upsert`, close while handler or timer work is pending, and assert `reply` or `sendMedia` finishes before `closeInboundMonitorSocket`. I did not run a live WhatsApp session, so this is source-level rather than live reproduction.

## Solution Assessment

No. Awaiting a close-path drain is the right direction, but the proposed one-shot snapshot can miss late enqueues and active `keyChains` work; a quiescence drain is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wait for debounce work to fully drain:** `src/auto-reply/inbound-debounce.ts:235-237`
  - body: `flushAll()` snapshots `buffers.keys()` once and awaits only those keys. In this close path, an already-running `handleMessagesUpsert` can enqueue after that snapshot, and timer-driven flushes can leave active `keyChains` work after the buffer has been deleted, so `close()` can still tear down the socket before `reply` or `sendMedia` work finishes.
  - confidence: 0.9
