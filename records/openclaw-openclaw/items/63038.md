## Summary

Keep this PR open, but do not merge it as-is: current main still lacks a WhatsApp close-path drain, while the PR's one-shot `flushAll()` can miss late enqueues or in-flight keyed flush work and the external PR still lacks real WhatsApp behavior proof.

## What This Changes

Adds `flushAll()` to the shared inbound debouncer, calls it during WhatsApp Web listener close before socket teardown, and updates BlueBubbles, Feishu, Slack, and test mocks for the new method.

## Best Possible Solution

Add a quiescent drain API to the shared inbound debouncer, call it in the WhatsApp close path before socket teardown, and cover it with a focused regression proving replies or media sends complete before close.

## Reproduction Assessment

Yes, source-reproducible: configure `messages.inbound.debounceMs > 0`, emit WhatsApp `messages.upsert`, then close while a handler or timer is pending and assert `reply` or `sendMedia` finishes before `closeInboundMonitorSocket`. I did not run a live WhatsApp session.

## Solution Assessment

No. A close-path drain is the right direction, but a one-shot buffer snapshot is not enough; the debouncer needs to wait until pending buffers and active keyed chains are quiescent before socket teardown.
