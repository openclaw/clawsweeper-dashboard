## Summary

Keep open. The WhatsApp shutdown loss is a real current-main gap, but this PR's one-shot buffer snapshot still leaves late enqueues and active keyed work able to outlive socket close, and the user-facing fix needs a changelog entry.

## What This Changes

The PR adds a shared inbound debouncer `flushAll()` API, awaits it from the WhatsApp Web inbox close path before socket teardown, and updates channel/test debouncer mocks.

## Best Possible Solution

Add a shared debouncer drain that waits until both buffers and active keyed chains are empty, await it before WhatsApp socket teardown, cover late-enqueue and chain-only shutdown cases in tests, and add the release-note entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `debounceMs > 0`, emit WhatsApp `messages.upsert`, start close while debounce or handler work is pending, and assert that `reply`/`sendMedia` completes before the socket closes.

## Solution Assessment

No. Awaiting a close-path flush is the right direction, but the proposed snapshot API is not the narrowest maintainable solution because it does not guarantee debouncer quiescence.

## Review Findings

- **[P1] Wait for debouncer quiescence before closing:** `src/auto-reply/inbound-debounce.ts:235-237`
