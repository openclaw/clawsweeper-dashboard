## Summary

Keep open: the reported WhatsApp debounce shutdown bug is still source-reproducible on current main, but this PR's one-shot flush can return before all debounce work quiesces and it still lacks contributor real behavior proof.

## What This Changes

The PR adds a shared inbound debouncer `flushAll()` API, awaits it during WhatsApp Web monitor close, and updates related channel/test mocks for the new debouncer shape.

## Best Possible Solution

Implement a quiescent debouncer drain that waits for both buffered items and active keyed chains before WhatsApp socket teardown, then prove send-before-close with focused regression coverage and real WhatsApp behavior evidence.

## Reproduction Assessment

Yes. A high-confidence source path is to configure `debounceMs > 0`, emit WhatsApp `messages.upsert`, start close while debounce or handler work is pending, and assert `reply` or `sendMedia` completes before `closeInboundMonitorSocket` tears down the socket.

## Solution Assessment

No. Awaiting before close is the right direction, but the proposed one-shot buffer snapshot is not the best fix because late enqueues and active keyed work can still outlive `flushAll()`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Drain debounce work until quiescent before closing:** `src/auto-reply/inbound-debounce.ts:235-237`

Already-running WhatsApp handlers can enqueue after that snapshot, and timer flushes can leave active `keyChains` work with no buffer entry, so `close()` can still tear down the socket before `reply` or `sendMedia` finishes.
