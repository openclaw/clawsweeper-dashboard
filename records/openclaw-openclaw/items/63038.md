## Summary

Keep open: the WhatsApp debounce shutdown bug is still present on current main, but this PR's one-shot `flushAll()` does not fully drain late enqueues or active keyed work and the external PR still lacks real WhatsApp proof.

## What This Changes

Adds a shared `flushAll()` debouncer method, calls it during WhatsApp Web close before socket teardown, and updates affected channel/test mocks for the new API.

## Best Possible Solution

Implement a debouncer drain that blocks close until pending buffers and active keyed chains are quiescent, then add a focused WhatsApp regression proving reply or media send completes before socket teardown.

## Reproduction Assessment

Yes. The source path is high-confidence: configure `debounceMs > 0`, emit WhatsApp `messages.upsert`, close while handler or timer work is pending, and assert `reply`/`sendMedia` finishes before `closeInboundMonitorSocket`.

## Solution Assessment

No. Awaiting a close-path drain is the right direction, but the proposed one-shot buffer snapshot can miss late enqueues and active `keyChains` work; a quiescence drain is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wait for debounce work to fully drain:** `src/auto-reply/inbound-debounce.ts:235-237`
  - body: `flushAll()` snapshots `buffers.keys()` once and awaits only those keys. In this close path, an already-running `handleMessagesUpsert` can enqueue after that snapshot, and timer-driven flushes can leave active `keyChains` work after the buffer has been deleted, so `close()` can still tear down the socket before `reply` or `sendMedia` work finishes.
  - confidence: 0.9
