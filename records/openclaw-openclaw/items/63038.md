## Summary

Keep open. Current main still has the WhatsApp debounce-close loss path, but this PR is not merge-ready because its one-shot `flushAll()` can return before late or active keyed debounce work finishes, and the external PR lacks after-fix real behavior proof.

## What This Changes

Adds a `flushAll()` API to the shared inbound debouncer, calls it during WhatsApp Web monitor close before socket teardown, and updates channel/test mocks for the new method.

## Best Possible Solution

Add a shared debouncer drain that waits until both buffers and active keyed chains are empty, await it before WhatsApp socket teardown, and cover send-before-close behavior with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source path is to configure `debounceMs > 0`, emit WhatsApp `messages.upsert`, start close while debounce or handler work is pending, and assert `reply`/`sendMedia` completes before `closeInboundMonitorSocket` tears down the socket.

## Solution Assessment

No. Awaiting before close is the right direction, but the proposed one-shot buffer snapshot is not enough because late enqueues and active keyed work can still outlive `flushAll()`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P1] Wait for debounce work to fully drain:** `src/auto-reply/inbound-debounce.ts:235-237`
