## Summary

Keep this open. Current main has substantial large-history safeguards for the chat freeze half, but the Chrome extension blank-page half is still not addressed by a bootstrap fallback or troubleshooting docs, and the related closed PR/report only cover adjacent freeze causes.

## What This Changes

The issue requests fixes or documentation for two dashboard failures: a blank page caused by early Chrome extension injection and a chat freeze when rendering large session history.

## Best Possible Solution

Land a narrow Control UI UX/docs fix for the unresolved blank-page path: add an early, CSP-compatible boot watchdog that surfaces a plain HTML diagnostic if the app module never mounts, and document the browser-extension/Incognito workaround on the public Control UI troubleshooting page while leaving broader chat virtualization or pagination work separate.

## Reproduction Assessment

Yes. The issue gives high-confidence reproduction paths for both symptoms: enable a document_start all-URLs Chrome extension for the blank page, and load a large default session with thinking/tool blocks for the freeze; this review verified the source paths statically but did not run Chrome or mutate local session state.

## Solution Assessment

No, current main is not a complete solution for this issue. The large-history freeze path is materially mitigated by current chat.history limits/truncation, but the requested blank-page fallback/docs for extension interference are still missing; the safest next fix is a small boot watchdog plus troubleshooting docs.
