Keep open: current main substantially addresses the large-session Control UI freeze, but the separate Chrome extension blank-page path still has no fallback or troubleshooting entry and needs maintainer judgment before cleanup.

## What This Changes

The issue asks OpenClaw to handle two Control UI failures: a blank dashboard caused by early-injecting Chrome extensions and a frozen chat page when loading large session history.

## Best Possible Solution

Keep the landed responsive-history behavior, then either add a small bootstrap/docs fallback for third-party extension blank pages or explicitly close that part as externally caused after maintainer review.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show that current main still has no plain-HTML fallback if the module script never defines `openclaw-app`, while the large-history path is mitigated by bounded history/render work and covered by focused tests.

## Solution Assessment

No, not as a cleanup close yet. The merged responsiveness work is the right direction for the large-session freeze, but the remaining browser-extension blank-page behavior needs a maintainer decision on docs, fallback UI, or third-party-support closure.
