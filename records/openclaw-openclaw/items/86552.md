## Summary

The follow-up commits fixed the earlier alias-index eager lookup cases, but the PR still resolves manifest metadata as soon as any primary model is configured, before knowing that aliases or manifest-backed normalization are needed. That leaves a concrete hot-path availability/performance regression in the same area this PR is meant to improve.

## What This Changes

The PR reuses manifest plugin metadata while resolving configured/default model refs and adds regression coverage for alias-normalization manifest lookup counts.

## Best Possible Solution

Keep the metadata reuse, but make manifest resolution lazy for primary-model resolution and cover the no-alias primary-model path before merge.

## Reproduction Assessment

Yes: source inspection of the PR head gives a high-confidence path. A config with `agents.defaults.model.primary` set and no `agents.defaults.models` aliases reaches the new manifest-resolution call before the no-alias/default-provider path can return.

## Solution Assessment

No: the metadata reuse direction is right, but the current placement is still too eager. The safer solution is to keep the one-snapshot reuse only after the resolver knows aliases or manifest-backed normalization are actually needed.
