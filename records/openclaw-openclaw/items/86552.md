## Summary

Keep open: the optimization is useful and proof-backed, but the patch still introduces an eager manifest-metadata lookup for wildcard-only configured model entries before the alias index can skip them.

## What This Changes

The PR threads resolved manifest plugin metadata through configured/default model resolution and adds regression tests for alias-normalization load counts.

## Best Possible Solution

Keep the manifest reuse, but make buildModelAliasIndex resolve manifest metadata only after it has an aliasable non-wildcard entry that actually needs parsing, with a regression test for wildcard-only configured models.

## Reproduction Assessment

Yes for the review finding via source inspection: PR head resolves manifest plugins before filtering wildcard-only defaults.models entries that current main skipped without parsing. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not quite: the proposed reuse direction is the right fix, but buildModelAliasIndex should filter to entries that can emit aliases before resolving manifest metadata.
