## Summary

Keep open: the PR addresses a real agents model-resolution hot path and includes sufficient live timing proof, but it should not merge yet because the new eager manifest resolution in `buildModelAliasIndex` can add a metadata load for empty/no-alias configs.

## What This Changes

The PR resolves manifest plugin metadata once during configured/default model resolution, threads it through alias parsing, and adds a regression test for `openrouter:auto` alias normalization load counts.

## Best Possible Solution

Keep the single resolved manifest snapshot for configured/default model resolution, but make alias-index construction lazy or early-returning when there are no configured model alias rows.

## Reproduction Assessment

Yes. Source inspection shows that after the patch `buildModelAliasIndex({ cfg: {}, defaultProvider })` resolves manifest plugins before reading `rawModels`, so an empty/no-alias config can still trigger metadata loading.

## Solution Assessment

No, not yet. The central reuse direction is right, but the implementation should avoid eager manifest work until there is at least one configured alias/model key that needs normalization.

- **[P2] Avoid loading manifests for empty alias maps:** `src/agents/model-selection-shared.ts:427`
  - body: `buildModelAliasIndex` is used by paths such as simple completion, fallback resolution, and session status even when `agents.defaults.models` is absent. Please return the empty index before resolving manifests, or resolve manifests lazily only when parsing configured alias keys.
