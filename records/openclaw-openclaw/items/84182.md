## Label Justifications

- `P2`: The missing reusable field affects a clear Discord presentation workflow but has a narrow adapter and schema surface.
- `impact:message-loss`: Repeat interactions for a presentation button can be consumed after the first successful callback, suppressing later callback delivery for reusable workflows.

## Summary

Keep open. Current main still has Discord registry-level reusable support, but the portable presentation button type, message tool schema, normalizer, and Discord presentation mapper do not carry per-button reusable intent; the issue is also correctly paired with the open implementation PR https://github.com/openclaw/openclaw/pull/84187, which references this issue with closing syntax and should be reviewed or closed before this issue is resolved.

## What This Changes

Discord presentation buttons should accept and preserve per-button reusable callback intent through the portable message presentation contract and Discord adapter.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/84187, then close this issue after the implementation reaches main.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a presentation button cannot declare reusable in the public type or message tool schema, normalization drops loose reusable input, and the Discord presentation mapper never forwards reusable to component specs. I did not run runtime tests because this review was read-only.

## Solution Assessment

Yes, the narrow maintainable solution is to add an opt-in reusable boolean to the portable button contract and propagate it into the existing Discord reusable-entry machinery. Message-level reusable or Discord-specific component payloads would leave the portable presentation gap in place.
