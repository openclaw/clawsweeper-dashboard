- `P2`: This is a focused Discord presentation contract improvement with limited blast radius but real impact on repeated callback workflows.
- `impact:message-loss`: A repeated Discord button press can be consumed after the first interaction, preventing follow-up callback delivery for reusable workflows.

## Summary

Keep open. Current main still has the reported portable presentation gap, and the open linked PR is the implementation candidate that should be reviewed or landed before this issue closes.

## What This Changes

Add a button-level reusable flag to portable message presentation buttons and preserve it through Discord mapping so repeated callback interactions stay active.

## Best Possible Solution

End with a documented `reusable?: boolean` portable button field that normalizes through the generic presentation contract and maps into Discord component entries without changing the default one-shot behavior.

## Reproduction Assessment

Yes. Source inspection shows a presentation button cannot declare `reusable`, normalization would not preserve it, and the Discord mapper does not copy it into component entries.

## Solution Assessment

Yes. The narrow maintainable direction is to add a generic button-level flag and map it into the existing Discord reusable entry support rather than relying on Discord-specific component payloads.
