## Summary

Keep open. Current main still lacks a portable per-button `reusable` field and Discord does not preserve that intent from presentation buttons, but an open linked PR is already the right implementation path.

## What This Changes

The issue requests a portable `presentation.blocks[].buttons[].reusable` flag that Discord preserves when registering component callbacks.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84187 if maintainers accept the public field addition, then close this issue after merge.

## Reproduction Assessment

Yes. Source inspection shows the portable button type, message tool schema, and normalizer cannot carry `reusable`, while the Discord presentation mapper does not copy it into component specs.

## Solution Assessment

Yes. The narrow maintainable solution is to add an opt-in generic button field and map explicit `true` values into the existing Discord reusable-entry registry path.
