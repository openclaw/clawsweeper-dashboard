- `impact:message-loss`: A repeat-action Discord button can be consumed after its first interaction, suppressing later callback delivery for workflows that intended reuse.

## Summary

Keep open. Current main still has Discord reusable component support but no portable presentation-button field, normalizer preservation, or Discord presentation mapper propagation; the linked implementation PR is open and explicitly closes this issue, so the issue should track that PR until it merges or is rejected.

## What This Changes

Add a portable per-button `reusable` flag for message presentation buttons and preserve it through Discord component registration.

## Best Possible Solution

Land or reject the linked PR after review; the desired end state is an opt-in `reusable?: boolean` presentation-button contract that flows through normalization, docs, Discord mapping, and component entries while preserving current defaults.

## Reproduction Assessment

Yes. Source inspection shows current main cannot declare or preserve `presentation.blocks[].buttons[].reusable`, while Discord entries already honor reusable once registered.

## Solution Assessment

Yes. The narrow maintainable path is to add the generic button-level flag and map it into the existing Discord reusable entry support; relying on message-level reusable or Discord-specific payloads would leave the portable contract gap.
