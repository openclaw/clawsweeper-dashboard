## Summary

Keep open: the patch is focused and appears to target the right config metadata layer, but this external PR still needs real replacement-bundle CLI proof before merge and it changes compatibility-sensitive channel schema precedence.

## What This Changes

The PR changes channel schema metadata collection so same-origin replacement channel manifests keep or take channel schema ownership via `preferOver`, and adds Slack validation regression coverage for both registry orders.

## Best Possible Solution

Land the narrow `preferOver`-gated metadata fix after real CLI proof and completed checks; avoid a separate validation mode or a global relaxation of the bundled Slack schema.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: the related issue has a redacted fixture and current source shows same-origin schema ownership can depend on registry order, but I did not run a live current-main replacement-bundle repro in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes, the collector is the narrow owner-boundary fix because both `config schema` and `config validate` depend on collected channel metadata; adding a validation mode or relaxing bundled Slack globally would create a second source of truth.
