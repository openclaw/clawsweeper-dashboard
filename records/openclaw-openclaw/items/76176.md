## Summary

Keep open. Current main appears to have fixed the explicit SSE-versus-WebSocket selection path, but the Chat Completions fallback can still build the reported native OpenAI payload shape with both function tools and `reasoning_effort`, so the user-visible hang/400 path is still actionable.

## What This Changes

The issue reports that Telegram-routed `openai/gpt-5.4-mini` turns hang or fail because OpenClaw selects an OpenAI WebSocket Responses path despite explicit SSE settings and can send unsupported `reasoning_effort` with function tools to Chat Completions.

## Best Possible Solution

Add a focused OpenAI model/API compatibility fix so OpenClaw never sends the unsupported native Chat Completions `tools` plus `reasoning_effort` combination for `gpt-5.4-mini`, while preserving explicit SSE behavior and adding regression coverage.

## Reproduction Assessment

Yes for a code-level reproduction: construct `buildOpenAICompletionsParams` for a native OpenAI `openai-completions` reasoning model with `context.tools`, and current main can emit both `tools` and `reasoning_effort`. The full Telegram hang should be rechecked after the payload fix lands.

## Solution Assessment

Yes, a narrow OpenAI transport/payload compatibility fix is the best next step because the remaining demonstrated failure is in model/API request shaping, not Telegram authentication or polling.
