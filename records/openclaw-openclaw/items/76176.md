## Summary

Keep open. Current main has the explicit-SSE guard before OpenAI WebSocket selection, but the native OpenAI Chat Completions payload builder still can emit `tools` together with `reasoning_effort` for a reasoning OpenAI model, matching the reporter's concrete 400 error.

## What This Changes

The issue reports that Telegram-routed `openai/gpt-5.4-mini` runs hang on the OpenAI Responses/WebSocket path and that a Chat Completions fallback sends an unsupported `tools` plus `reasoning_effort` payload.

## Best Possible Solution

Land a focused OpenAI transport fix that preserves explicit SSE behavior and prevents native OpenAI Chat Completions requests from sending the unsupported `tools` plus `reasoning_effort` combination for `gpt-5.4-mini`.

## Reproduction Assessment

Yes for a code-level reproduction: build an OpenAI Chat Completions payload for `gpt-5.4-mini` with function tools and current main can add both `tools` and `reasoning_effort`. The full Telegram hang path still needs live verification after the provider payload fix.

## Solution Assessment

Yes, a narrow provider-payload compatibility fix is the best next step; Telegram is the delivery surface, while the remaining observable rejection is in OpenAI model routing and Chat Completions payload shaping.
