## Summary

Keep open. Current main appears to honor explicit `transport: "sse"` before selecting the OpenAI WebSocket path, but the native OpenAI Chat Completions fallback still appears able to send `tools` together with `reasoning_effort` for a reasoning model, matching the reported 400.

## What This Changes

The issue reports that Telegram-routed agents using `openai/gpt-5.4-mini` hang on the built-in Responses/WebSocket path and that a Chat Completions fallback sends an invalid `reasoning_effort` payload.

## Best Possible Solution

Land a focused OpenAI transport fix that preserves explicit SSE behavior and prevents native OpenAI Chat Completions requests from sending the unsupported `tools` plus `reasoning_effort` combination for gpt-5.4-mini.

## Reproduction Assessment

Yes for a code-level reproduction: build an OpenAI Chat Completions payload for `gpt-5.4-mini` with function tools and current main will add both `tools` and `reasoning_effort`. The full Telegram hang path still needs live verification after the payload fix.

## Solution Assessment

Yes, a narrow provider-payload fix is the best next step; the Telegram channel is only the delivery surface, while the remaining observable failure is in OpenAI model routing/payload compatibility.
