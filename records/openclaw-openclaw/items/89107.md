## Summary

Current main already has the generic exec SecretRef capability this feature builds on, but this branch is not merge-ready: its documented underscore-style id is rejected by the existing contract, its standalone global `keytar` installation is not normally discoverable from the copied script, and the proposed credential dependency chain lacks an established OpenClaw support and security owner. The linked plaintext-credential issue should remain open because this opt-in resolver does not migrate existing credentials or provide general config encryption.

## What This Changes

The PR adds a `keytar`-backed executable that maps SecretRef ids to native OS keychain entries and documents configuring it as a Gateway exec provider.

## Best Possible Solution

Document one vetted, dependency-owning resolver executable using valid path-style SecretRef ids, limit support claims to proven environments, demonstrate the exact setup through a live Gateway, and leave the broader credential-migration issue open.

## Reproduction Assessment

Yes for the patch defects: current validation rejects the documented underscore id, and Node's file-relative module search does not normally discover the separately global-installed dependency. The underlying OS-keychain capability is a feature rather than a broken current-main contract.

## Solution Assessment

No. The branch combines two deterministic setup failures with an unowned native credential dependency; a single vetted self-contained executable or an explicitly OpenClaw-owned package is safer and more maintainable.
