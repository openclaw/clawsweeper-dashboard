## Summary

This PR should remain open because its documented installation can leave the resolver unable to load `keytar`, and adopting an archived native credential dependency plus a contributor-owned package requires an explicit maintainer security and ownership decision.

## What This Changes

The PR adds a native OS keychain exec SecretRef resolver backed by `keytar` and documents installing and configuring it for the Gateway.

## Best Possible Solution

Retain the generic exec SecretRef seam, but use an explicitly approved, maintained, self-contained resolver executable and document only an installation path proven through a redacted live Gateway resolution.

## Reproduction Assessment

Yes for the introduced installation failure: the resolver loads modules relative to its own path, while the recommended global installation is outside the copied script's normal Node lookup paths. The requested keychain capability itself is a new feature, not a current-main regression.

## Solution Assessment

No. The existing exec-provider API is the correct integration seam, but a copied repository script depending on an independently global-installed, archived native module is not a reliable or maintainable distribution boundary.
