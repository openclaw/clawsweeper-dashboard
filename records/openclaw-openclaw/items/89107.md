## Summary

This PR should remain open but is not merge-ready: the documented global `keytar` installation does not satisfy the new script’s module lookup, the native dependency is archived, and the separately published credential package lacks an established OpenClaw trust and maintenance contract. The linked security request also remains broader because existing plaintext credentials receive no migration path.

## What This Changes

The PR adds a keytar-based native OS keychain executable for exec SecretRefs and documents installing it or a companion package for Gateway credential resolution.

## Best Possible Solution

Keep the existing generic exec SecretRef boundary, but use a vetted self-contained executable that owns a maintained dependency and has proven platform support; track migration of existing plaintext credentials as separate canonical product and security work rather than claiming this wrapper completes it.

## Reproduction Assessment

Yes for the introduced defect: the resolver loads `keytar` from its file-relative Node module search path, while the documented npm global installation does not normally place the module there. The broader feature request is not a broken current-main behavior requiring reproduction.

## Solution Assessment

No. The existing exec SecretRef seam is appropriate, but this implementation combines a broken installation contract with an archived native dependency and an unestablished external credential-package trust boundary; a maintained, self-contained resolver is safer.
