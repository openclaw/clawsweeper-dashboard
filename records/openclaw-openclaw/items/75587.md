## Summary

Keep open. The latest head adds a useful opt-in Telegram reasoning egress surface, but it still has blocking correctness issues in sink lifecycle handling and config validation.

## What This Changes

The PR adds Telegram `channels.telegram.reasoningStreamSink` config, schema/docs/SDK type export, and runtime HTTPS/HMAC POST delivery of reasoning lifecycle events to an external endpoint.

## Best Possible Solution

Repair the sink lifecycle and non-throwing URL validation, add focused regression coverage, then have maintainers review the guarded external egress surface before merge.

## Reproduction Assessment

Yes. For the sink bug, call `onToken('Reasoning:\nPlan')`, `onEnd()`, then `onToken('Reasoning:\nPlan after tool')` on one handle; the second block emits only the suffix because `lastSnapshot` survived the end boundary. For the schema bug, validate an invalid sink URL string and the `new URL(v)` refinement can throw outside normal Zod validation.

## Solution Assessment

No. The HTTPS-only guarded egress design is plausible, but the current patch needs per-block sink state reset and a non-throwing protocol check before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reset sink deltas between reasoning blocks:** `extensions/telegram/src/reasoning-stream-sink.ts:126-127`
  - body: Telegram can call `onReasoningEnd()` and later `onReasoningStream()` again in the same turn. `onEnd()` only clears `started`, leaving `lastSnapshot` from the ended block, so a new block that shares the previous text prefix is emitted as only the suffix. Reset the per-block snapshot state here, or keep one lifecycle open until final cleanup.
  - confidence: 0.88
- **[P2] Make the HTTPS URL check non-throwing:** `src/config/zod-schema.providers-core.ts:347`
  - body: `z.string().url()` can report an invalid URL as a validation issue while later checks still run in Zod 4, and Zod does not catch exceptions thrown by refinements. With `new URL(v)` here, a bad config value like `"not-a-url"` can crash validation instead of returning a normal schema error. Wrap the parse or use a non-throwing protocol check.
  - confidence: 0.82
