## Summary

Keep open. Current main does not contain the requested Telegram reasoning sink, but the latest PR head still has two merge-blocking defects: sink delta state survives a reasoning block end, and invalid sink URLs can escape normal Zod validation by throwing inside a refinement.

## What This Changes

The PR adds Telegram `channels.telegram.reasoningStreamSink` config/schema/docs/SDK type export and runtime HTTPS/HMAC POST delivery for reasoning lifecycle events, plus a small Discord test fixture update.

## Best Possible Solution

Repair the sink lifecycle and URL validation bugs, add focused regression coverage, then let maintainers review and merge the guarded HTTPS/HMAC egress feature if they accept the boundary.

## Reproduction Assessment

Yes. For the sink bug, call one handle with `onToken('Reasoning:\nPlan')`, `onEnd()`, then `onToken('Reasoning:\nPlan after tool')`; the second block emits only the suffix because `lastSnapshot` survived the end boundary. For the schema bug, validate a malformed URL string and the refinement can throw through `new URL(v)`.

## Solution Assessment

No. The HTTPS-only guarded egress direction is plausible, but the current implementation is not the best mergeable solution until per-block sink state resets and the protocol check is non-throwing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reset sink state at reasoning block end:** `extensions/telegram/src/reasoning-stream-sink.ts:126-127`
  - body: Telegram can emit a later reasoning block on the same dispatch handle after `onReasoningEnd()`. `onEnd()` only clears `started`, so `lastSnapshot` from the previous block is reused and a new block sharing that prefix posts only the suffix instead of the new block text. Clear the per-block snapshot when ending a block, or otherwise scope delta state to one open lifecycle.
  - confidence: 0.88
- **[P2] Make HTTPS validation non-throwing:** `src/config/zod-schema.providers-core.ts:347`
  - body: This refinement calls `new URL(v)` after `.url()`. Zod refinements do not catch thrown errors, and invalid-format string checks are continuable by default, so a malformed sink URL can crash config validation instead of returning a normal schema error. Wrap the parse or use a non-throwing helper for the protocol check.
  - confidence: 0.84
