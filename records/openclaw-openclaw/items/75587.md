## Summary

Keep open. The latest PR head addresses the earlier ClawSweeper blockers by adding http(s)-only schema validation and Telegram docs for the egress boundary, but the diff still misses the existing reasoning-end lifecycle path for the new external sink, so external consumers can stay active until the whole dispatch finishes instead of receiving `reasoning_end` when model thinking ends. The PR is also draft and dirty while adding a security-sensitive reasoning egress surface, so it needs maintainer review before any merge path.

## What This Changes

The PR adds a Telegram `reasoningStreamSink` config/schema/type/docs/changelog surface plus dispatch wiring and a new module that posts reasoning lifecycle events to a configured HTTP endpoint with optional HMAC signing and SecretRef-backed headers.

## Best Possible Solution

If maintainers want this capability, land it as an explicitly opt-in Telegram reasoning egress path that uses the existing `onReasoningStream` and `onReasoningEnd` lifecycle, keeps http(s)-only validation and guarded outbound fetch, preserves the docs/changelog/config baseline work, and adds focused coverage for streaming-off delivery, SecretRef headers, HMAC signing, and end-event timing.

## Reproduction Assessment

Yes. A read-only diff/source check reproduces the blocker: the PR sends sink tokens from `onReasoningStream`, while current main has an explicit `onReasoningEnd` contract that the sink does not consume until dispatch `finally`.

## Solution Assessment

No. The implementation direction is plausible after the latest validation and docs fixes, but the current patch is not the best version until the external sink follows the same reasoning-end lifecycle as the existing Telegram preview stream.
