## Summary

Keep this PR open; the current head is a useful feature branch, but it is not merge-ready because the new sink would export commentary-phase assistant deltas as `reply_stream` events even though current chat surfaces suppress those as non-reply output.

## What This Changes

The PR adds a top-level `agentEventSink` config, a core AgentEventStream webhook sink, gateway lifecycle wiring, schema/API/doc updates, and a changelog entry for channel-agnostic thinking and reply streaming.

## Best Possible Solution

Preserve the core sink direction only if it filters assistant events consistently with existing reply surfaces, adds focused coverage for commentary suppression and final-answer delivery, and then gets owner review for the new egress contract.

## Reproduction Assessment

Not applicable as an end-user bug reproduction; this is a feature PR. The merge blocker is source-reproducible because current main emits `stream: "assistant"` events with `phase: "commentary"`, while the PR sink forwards any assistant text/delta as `reply_stream`.

## Solution Assessment

No; the channel-agnostic sink is plausible, but this is not the best mergeable state until it honors the existing commentary suppression contract. The narrow fix is to skip commentary-phase assistant events before emitting `reply_start` or `reply_stream`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Skip commentary assistant events in the sink:** `src/infra/agent-event-sink.ts:155-156`
  - body: `handleAssistant()` treats every assistant event with `text` or `delta` as a `reply_stream`. Current producers emit `phase: "commentary"` assistant deltas, and the gateway intentionally suppresses those from live reply surfaces, so this would POST non-reply commentary to the configured webhook as if it were user-visible reply text. Filter commentary-phase events, or only allow final-answer/unspecified phases, before starting the reply stream.
  - confidence: 0.9
