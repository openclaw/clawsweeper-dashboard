## Summary

Keep open. Current main does not implement the proposed external reasoning sink, and the active draft PR is not merge-ready because the final head still exposes a new reasoning egress path through raw fetch, does not preserve lifecycle event ordering, and omits generated config/docs alignment for the new public config key.

## What This Changes

The PR adds a Telegram `reasoningStreamSink` config option, shared config type/schema export, HTTP POST reasoning sink with optional HMAC/custom headers, Telegram dispatch wiring for reasoning callbacks even when preview streaming is off, and a changelog entry.

## Best Possible Solution

If maintainers want this capability, ship it as an explicit opt-in Telegram reasoning egress feature that uses the guarded outbound HTTP utility, defines ordered or explicitly unordered event semantics, documents payloads/signing/SecretRef/private-network behavior, regenerates config artifacts, and adds focused Telegram sink tests.

## Reproduction Assessment

Yes. A read-only source check of PR head `a0a0cd9838e7d43c059ae50f07acae0952721adf` reproduces the remaining review failures: raw fetch is used for the sink, lifecycle posts are unqueued, and generated config/docs search has no `reasoningStreamSink` entry.

## Solution Assessment

No. The direction may be useful, but the current implementation is not the safest maintainable solution because it exports reasoning content before using existing guarded egress patterns or aligning the public config/docs contract.
