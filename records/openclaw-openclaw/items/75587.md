## Summary

Keep open. The PR adds a plausible Telegram reasoning egress feature, but the latest head still has the two blockers from the earlier ClawSweeper review: it derives an SSRF allowed-host policy from arbitrary user config, and it accepts/documents `http://` URLs even though runtime fetch requires HTTPS.

## What This Changes

The PR adds a Telegram `reasoningStreamSink` config/schema/docs/changelog surface, a new HTTP sink with HMAC/SecretInput support, Telegram reasoning stream/end callback wiring, generated config baseline hashes, and an SDK type export.

## Best Possible Solution

Keep the feature branch open and land it only after the sink uses the default guarded public-host SSRF boundary or a deliberately reviewed private-network opt-in, aligns HTTPS validation with runtime and docs, regenerates config baselines, and adds focused tests for the new egress path.

## Reproduction Assessment

Yes. A read-only source check reproduces the blockers: the PR still uses a user-configured allowed-host SSRF policy, while current main skips private-network checks for allowed hostnames, and the PR schema/docs still allow `http://` despite runtime HTTPS enforcement.

## Solution Assessment

No. The feature direction is plausible, but the current patch is not the best safe implementation until the egress policy and config contract are made consistent and covered by tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Keep sink URLs on the default SSRF policy:** `extensions/telegram/src/reasoning-stream-sink.ts:60`
  - body: `ssrfPolicyFromHttpBaseUrlAllowedHostname(config.url)` treats the user-configured sink host as allowed. Current main skips literal and resolved private-IP checks for allowed hostnames, so `requireHttps` still lets HTTPS private or special-use targets bypass the documented public-host boundary. Use the default guarded policy here, or add an explicit reviewed private-network opt-in with tests.
  - confidence: 0.94
- **[P2] Require HTTPS in the config contract:** `src/config/zod-schema.providers-core.ts:346-348`
  - body: The runtime passes `requireHttps: true` to the guarded fetch, but this schema accepts `http:` and the docs say `http://` is valid. An accepted plaintext sink will only fail each POST at verbose runtime logging, so reject non-HTTPS at validation and regenerate docs/baseline, or intentionally allow plaintext by removing the runtime requirement.
  - confidence: 0.9
