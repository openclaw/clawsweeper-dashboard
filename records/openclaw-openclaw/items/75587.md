## Summary

Keep open. The final PR head addresses the earlier raw-fetch and event-ordering blockers by switching the sink to the guarded fetch path and serializing POSTs, but the patch still is not ready to merge because the HTTP sink URL is validated with a generic URL schema and the new reasoning egress boundary has no public docs. The PR is also draft and currently dirty, so it needs author/maintainer follow-up rather than cleanup closure.

## What This Changes

The PR adds a Telegram `reasoningStreamSink` config surface, a guarded HTTP POST sink for reasoning lifecycle/token events with optional HMAC and SecretRef-backed headers, dispatch wiring, config type/schema exports, a generated config baseline checksum update, and a changelog entry.

## Best Possible Solution

If maintainers want this capability, land it as an explicit opt-in Telegram reasoning egress path with http(s)-only schema validation, guarded outbound fetch, serialized lifecycle delivery, user-facing docs for payloads/signing/SecretRef headers/failure behavior/private-network policy, focused tests for streaming-off delivery and credential handling, and normal generated config baseline updates.

## Reproduction Assessment

Yes. A read-only diff/source check reproduces the current blockers: the PR schema uses generic `z.string().url()` for an HTTP-only endpoint, and a docs search finds no `reasoningStreamSink` page or section. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

No. The implementation direction is plausible after the latest guarded-fetch and ordering fixes, but the current patch is not the narrowest maintainable solution until validation, docs, tests, and security-boundary review are aligned with the new external reasoning egress surface.
