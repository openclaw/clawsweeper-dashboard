## Summary

Keep open. Current main does not already implement this sink, and the latest head mostly follows the guarded egress shape, but it has a concrete lifecycle bug: ending one reasoning block does not reset the sink's delta state, so a later block in the same turn can lose prefix text and reuse an ended stream id.

## What This Changes

The PR adds Telegram `channels.telegram.reasoningStreamSink` config, docs, schema/types/SDK export, and runtime HTTPS/HMAC posting of reasoning lifecycle events.

## Best Possible Solution

Fix the per-block sink lifecycle/delta handling, add focused unit coverage, then have maintainers review the guarded external reasoning egress surface before merge.

## Reproduction Assessment

Yes for the blocking review finding: invoke `onToken('Reasoning:\nPlan')`, `onEnd()`, then `onToken('Reasoning:\nPlan after tool')` on one sink handle; the second block emits only the suffix because `lastSnapshot` was not reset. This matches the current Telegram dispatch pattern that can call `onReasoningEnd()` before a later `onReasoningStream()`.

## Solution Assessment

No. The guarded HTTPS/HMAC approach is plausible, but the current lifecycle implementation should reset per-block snapshot state or keep one open lifecycle until final cleanup so external consumers receive complete reasoning events.
