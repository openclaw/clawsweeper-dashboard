## Summary

Keep open. The latest PR head is not redundant with current main and appears to have fixed the earlier SSRF-policy blocker by using the default guarded fetch path with HTTPS required, but it still introduces a docs/config mismatch: the Telegram docs say `http://` sink URLs are valid while the schema and runtime reject non-HTTPS URLs.

## What This Changes

The PR adds Telegram `reasoningStreamSink` config, schema, docs, changelog, SDK type export, and dispatch wiring to forward reasoning lifecycle events to an external guarded HTTP sink with optional SecretInput-backed HMAC/header credentials.

## Best Possible Solution

Keep the feature branch open, align the Telegram docs to the HTTPS-only contract, preserve the default guarded SSRF boundary, and validate the sink with focused Telegram/config checks before merge.

## Reproduction Assessment

Yes. A source/diff check reproduces the remaining blocker: the PR schema rejects non-HTTPS sink URLs and the runtime requires HTTPS, while the added Telegram docs still document `http://` as valid.

## Solution Assessment

No. The feature direction and latest SSRF fix are plausible, but the patch should not land until the docs match the HTTPS-only runtime/config contract; the safer alternative is a narrow docs repair, not loosening the egress policy.
