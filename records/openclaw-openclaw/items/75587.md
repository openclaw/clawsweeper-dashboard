## Summary

Keep open. Current main does not have the requested Telegram reasoning stream sink, and the PR adds a plausible guarded implementation, but the latest head is not merge-ready because the sink keeps its delta snapshot after `onEnd()`, so a later reasoning block in the same dispatch can be emitted as only a suffix.

## What This Changes

The PR adds Telegram `channels.telegram.reasoningStreamSink` config, docs, schema/types/SDK export, and runtime HTTPS/HMAC posting of reasoning lifecycle events.

## Best Possible Solution

Fix the sink lifecycle/delta handling and add focused coverage, then have maintainers review whether this external reasoning-egress surface belongs in the bundled Telegram plugin.

## Reproduction Assessment

Yes for the blocker: create one sink, call `onToken('Reasoning:\nPlan')`, `onEnd()`, then `onToken('Reasoning:\nPlan after tool')`. Because `lastSnapshot` survives `onEnd()`, the restarted block emits only the suffix instead of the complete new reasoning block.

## Solution Assessment

No. The guarded HTTPS/HMAC approach is reasonable, but the current per-block delta state should reset on reasoning end or the lifecycle should remain open until final cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reset sink snapshots between reasoning blocks:** `extensions/telegram/src/reasoning-stream-sink.ts:126-127`
  - body: Telegram can call `onReasoningEnd()` and later `onReasoningStream()` again in the same dispatch. `onEnd()` clears `started` but leaves `lastSnapshot`, so a new block that shares the previous prefix is emitted as only the suffix under a restarted lifecycle. Reset the per-block snapshot state here, or keep the sink open until final cleanup instead.
