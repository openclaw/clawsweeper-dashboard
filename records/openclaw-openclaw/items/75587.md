## Summary

Keep open. Current main does not implement an external Telegram reasoning sink, so this PR is not obsolete, but the patch is not merge-ready: it introduces a new external reasoning egress path with plaintext custom headers, undocumented boundary semantics, and stream events that receive cumulative reasoning snapshots rather than true deltas. The existing ClawSweeper comment already flagged the header/docs concerns, and the latest head still has those blockers.

## What This Changes

The PR adds a Telegram `channels.telegram.reasoningStreamSink` config and helper that POSTs reasoning stream events to an external URL with optional HMAC signing, SDK type/schema exports, and a changelog entry.

## Best Possible Solution

If maintainers want this feature, ship it as an explicit opt-in egress surface with secret-aware headers, clearly defined delta-versus-snapshot event semantics, focused Telegram dispatch/sink/schema tests, and public Telegram/streaming/thinking docs covering payloads, signing, trust boundaries, and how it differs from preview streaming.

## Reproduction Assessment

Yes. Source inspection is enough: the PR accepts plaintext `reasoningStreamSink.headers` and forwards `payload.text` to the sink, while current main shows `onReasoningStream` receives full formatted reasoning snapshots rather than the computed delta.

## Solution Assessment

No. The hook is directionally useful, but the patch is not the safest maintainable solution until credential-bearing headers use `SecretInput`, the stream payload contract is corrected or documented, and maintainers approve the external reasoning egress surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Make sink headers secret-aware:** `src/config/zod-schema.providers-core.ts:342`
  - body: `reasoningStreamSink.headers` is advertised for custom headers such as Authorization, but the schema accepts only plain strings and the sink spreads them directly into `fetch`. These values should be `SecretInput` entries registered as sensitive, then resolved before dispatch, so credentials are not exposed through config display, diagnostics, or generated metadata.
  - confidence: 0.89
- **[P2] Send reasoning deltas to the sink:** `extensions/telegram/src/bot-message-dispatch.ts:1090`
  - body: The new sink calls `onToken(payload.text)`, but current producers pass the full formatted reasoning snapshot on every `onReasoningStream` callback and only expose the computed delta through agent events. A consumer that appends `reasoning_stream` events will duplicate prior text and payload size grows over the run; track and send the suffix/delta or model these as snapshot updates.
  - confidence: 0.82
- **[P3] Document the reasoning egress boundary:** `extensions/telegram/src/reasoning-stream-sink.ts:69`
  - body: This feature posts reasoning text plus Telegram chat/thread/account and session identifiers to a configured URL, but the Telegram and streaming docs only describe preview streaming. Add docs for the external trust boundary, payloads, signing, secret-backed headers, and how this behaves when Telegram preview streaming is off.
  - confidence: 0.84

  - body: `reasoningStreamSink.headers` can carry bearer tokens or other endpoint credentials, but the PR models them as plain strings instead of sensitive `SecretInput` values.
