## What This Changes

This PR adds Telegram `reasoningStreamSink` config/docs/types and a guarded HTTPS/HMAC POST sink for reasoning lifecycle events, plus generated config baseline updates and a Discord fixture adjustment.

## Best Possible Solution

Keep the opt-in Telegram sink design, but repair lifecycle, capture, validation, and SDK baseline handling before maintainer review of the new external reasoning egress surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Reset sink state after each reasoning block:** `extensions/telegram/src/reasoning-stream-sink.ts:126`
  - body: `onEnd()` only clears `started`, leaving `lastSnapshot` from the previous block. Telegram can emit another full-snapshot reasoning block later in the same turn, so a second block sharing the old prefix is sent as a suffix-only `reasoning_stream` event instead of a fresh block.
  - confidence: 0.9
- **[P2] Make the HTTPS URL refinement non-throwing:** `src/config/zod-schema.providers-core.ts:347`
  - body: This refinement calls `new URL(v)` after `.url()`. Zod refinements must not throw, so bad input can escape config validation as a raw `TypeError` instead of a normal Zod issue; wrap the parse or use a non-throwing helper.
  - confidence: 0.87
- **[P2] Regenerate the plugin SDK API baseline:** `src/plugin-sdk/config-runtime.ts:119`
  - body: Adding `ReasoningStreamSinkConfig` to the plugin SDK config exports changes the generated SDK API surface, but the PR leaves `docs/.generated/plugin-sdk-api-baseline.sha256` unchanged. The API check will report drift until the baseline is regenerated and committed.
  - confidence: 0.9
- **[P2] Disable capture for reasoning sink posts:** `extensions/telegram/src/reasoning-stream-sink.ts:79-83`
  - body: `fetchWithSsrFGuard` captures request headers and body unless `capture: false`. These POSTs contain reasoning text and can include arbitrary SecretInput-backed headers, while proxy capture only redacts headers by name, so debug capture can persist reasoning content or custom credentials.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The HTTPS/SSRF/HMAC posture is good, but guarded fetch capture must be disabled or sanitized before this sink can safely post reasoning payloads and SecretInput-backed headers.

Concerns:

- **[medium] Debug capture can persist reasoning sink payloads:** `extensions/telegram/src/reasoning-stream-sink.ts:79`
  - body: The new sink uses `fetchWithSsrFGuard` without `capture: false`; when proxy capture is enabled, request bodies and non-redacted custom headers can be persisted even though the payload contains reasoning text and credentials may be supplied through `SecretInput`.
  - confidence: 0.84
