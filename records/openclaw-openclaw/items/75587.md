## Summary

Keep open. This draft PR is not merge-ready: the new Telegram config key is only added to TypeScript types and dispatch wiring, while the current strict runtime schema would still reject the documented config. The patch also introduces a new external reasoning egress path with HMAC/header credential fields that are not wired into OpenClaw's secret-aware config/redaction model, and it lacks the docs, changelog, and focused tests expected for a user-facing channel feature.

## What This Changes

The PR adds a Telegram `reasoningStreamSink` option that sends reasoning stream text to an external HTTP endpoint with optional HMAC signing and exports the new config type through the plugin SDK config runtime surface.

## Best Possible Solution

Keep the PR open until maintainers decide the egress surface is wanted. If it is, implement it as a fully registered, secret-aware, documented opt-in config with focused Telegram dispatch tests, schema/config metadata updates, and a changelog entry; otherwise steer the use case toward the existing generic reasoning/agent-event stream or a plugin-level integration.

## Reproduction Assessment

Yes for the main functional blocker: the PR's sample config can be checked against current `TelegramAccountSchemaBase.strict`, which has no `reasoningStreamSink` key, so validation would reject it. I did not live-run config validation because the checkout review was constrained to read-only commands.

## Solution Assessment

No. The proposed runtime hook is narrow, but the patch is incomplete and not the safest shape yet because it adds an external reasoning sink without runtime schema acceptance, secret-aware config handling, docs, or tests; a schema-registered secret-aware opt-in, or a generic agent-event sink, is safer.
