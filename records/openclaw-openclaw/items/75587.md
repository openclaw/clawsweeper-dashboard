## Summary

Keep open. The latest head fixed the earlier reasoning-end wiring issue, but the PR is still not merge-ready: the new sink still derives an allowed-host SSRF policy from an arbitrary configured URL, and the config/docs accept plaintext HTTP while runtime now requires HTTPS.

## What This Changes

The PR adds a Telegram `reasoningStreamSink` config/schema/docs/changelog surface, an HTTP reasoning sink with HMAC/SecretInput support, Telegram dispatch wiring for reasoning stream/end callbacks, generated config baseline hashes, an SDK type export, and a small tool-display serialization change.

## Best Possible Solution

Keep the feature branch open and land it only after the sink preserves the default guarded public-host SSRF boundary or adds an explicitly reviewed private-network opt-in, aligns the HTTPS contract across runtime/schema/docs/generated baselines, and adds focused tests for the new egress path.

## Reproduction Assessment

Yes. A read-only source check reproduces the blockers: the PR still creates an allowed-host SSRF policy from the configured sink URL, while current main skips private-network checks for allowed hostnames, and the schema/docs allow `http://` even though runtime requires HTTPS.

## Solution Assessment

No. The implementation direction is plausible, but the current patch is not the narrowest safe solution until it stops weakening SSRF checks and makes the HTTPS requirement explicit at validation and documentation boundaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Keep sink URLs on the default SSRF policy:** `extensions/telegram/src/reasoning-stream-sink.ts:60`
  - body: `ssrfPolicyFromHttpBaseUrlAllowedHostname(config.url)` treats the user-configured sink host as allowed. Current main skips literal and resolved private-IP checks for allowed hostnames, so adding `requireHttps` still lets HTTPS private or special-use targets bypass the documented public-host boundary. Use the default guarded policy here, or add an explicit reviewed private-network opt-in with tests.
  - confidence: 0.94
- **[P2] Require HTTPS in the config contract:** `src/config/zod-schema.providers-core.ts:346-348`
  - body: The runtime passes `requireHttps: true` to the guarded fetch, but this schema accepts `http:` and the docs say `http://` is valid. An accepted `http://` sink will only fail each POST at verbose runtime logging, so reject non-HTTPS at validation and regenerate docs/baseline, or intentionally allow plaintext by removing the runtime requirement.
  - confidence: 0.9
