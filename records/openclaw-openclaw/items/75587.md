## Summary

Keep open. The revised PR fixes the earlier missing runtime-schema and HMAC-secret handling gaps, but it still introduces an external reasoning egress surface with custom headers treated as ordinary config strings and without docs/tests for the new trust boundary. Current main does not already implement this feature, so there is no conservative close reason.

## What This Changes

The PR adds a Telegram `reasoningStreamSink` config option that forwards reasoning stream text to an external HTTP endpoint, with optional HMAC signing, config schema/type updates, a plugin SDK type export, and a changelog entry.

## Best Possible Solution

If maintainers want this feature, make every credential-bearing field secret-aware, resolve those secrets before dispatch, document the external reasoning egress boundary in the Telegram/streaming/thinking docs, and add focused schema plus Telegram dispatch/sink tests. If the product direction is generic event delivery, keep this PR as source context and route the use case through that shared surface instead.

## Reproduction Assessment

Yes. The remaining blocker is reproducible by source inspection: a config with `reasoningStreamSink.headers.Authorization` is accepted as a plain string by the new schema and then spread directly into `fetch`, while existing sensitive header schemas use `SecretInputSchema.register(sensitive)`.

## Solution Assessment

No. The runtime hook is narrow and the HMAC secret fix is directionally right, but the branch is not the safest maintainable solution until custom headers are secret-aware and the new external reasoning egress behavior is documented and tested.

Full review comments:

- **[P2] Make sink headers secret-aware:** `src/config/zod-schema.providers-core.ts:342`
  - body: The PR advertises custom headers for this sink, including likely `Authorization` values, but the schema keeps `headers` as plain strings and the runtime spreads them directly into `fetch`. OpenClaw's config redaction relies on secret-aware schemas for credential headers, so these values should be `SecretInput` values registered as sensitive and resolved before sending.
  - confidence: 0.88
- **[P3] Document the reasoning egress boundary:** `extensions/telegram/src/reasoning-stream-sink.ts:58`
  - body: This adds a new path that POSTs reasoning text plus chat/thread/account identifiers to an arbitrary configured URL, but the Telegram/streaming docs still only describe preview streaming. Add docs covering the external trust boundary, signing, secret-backed headers, and how this differs from `streaming.mode` and `/reasoning stream`.
  - confidence: 0.82

## Security Review

Summary: The diff introduces external reasoning egress; the HMAC secret is now secret-aware, but custom headers remain plaintext and the egress boundary is undocumented.

Concerns:

  - body: `reasoningStreamSink.headers` can carry bearer tokens or other credentials, but the new schema and type model them as plain strings instead of sensitive `SecretInput` values. That can expose credentials through config display, diagnostics, or generated schema metadata.
