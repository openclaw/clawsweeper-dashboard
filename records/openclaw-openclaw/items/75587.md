## Summary

Keep open. Current main does not implement this sink, and the PR is useful but not merge-ready because the sink lifecycle keeps the previous reasoning snapshot after a reasoning_end, so a later reasoning block can emit only a suffix delta.

## What This Changes

The PR adds Telegram reasoningStreamSink config, docs, schema/types/SDK export, and runtime HTTPS/HMAC posting of reasoning lifecycle events.

## Best Possible Solution

Repair the lifecycle/delta handling and add focused unit/dispatch coverage, then have maintainers decide whether the guarded external reasoning egress surface should merge.

## Reproduction Assessment

Yes. Invoke one sink handle with onToken('Reasoning:\nPlan'), onEnd(), then onToken('Reasoning:\nPlan after tool'); the second block is deltaed against the old snapshot even though Telegram supports this end-then-stream pattern.

## Solution Assessment

No. The HTTPS/HMAC design is plausible, but the current lifecycle implementation should reset per-block snapshot state or keep one open lifecycle until final cleanup so external consumers receive complete reasoning events.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reset sink snapshots between reasoning blocks:** `extensions/telegram/src/reasoning-stream-sink.ts:126-127`
  - body: Telegram can call onReasoningEnd() and later onReasoningStream() again in the same turn. onEnd() only clears started, so lastSnapshot survives the ended block and a new block sharing the prior prefix is emitted as only the suffix under a restarted lifecycle. Reset the per-block snapshot state here, or keep the sink open until final cleanup instead.
  - confidence: 0.86
