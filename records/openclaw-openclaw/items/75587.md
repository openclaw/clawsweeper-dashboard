## Summary

Keep open. The final head resolved the earlier broad SDK import and unresolved-secret fail-open findings, but the patch still adds unguarded reasoning egress, unordered lifecycle delivery, and an incomplete generated config/docs surface, so this draft needs maintainer and security review before merge.

## What This Changes

The PR adds a Telegram `reasoningStreamSink` config option, shared config type/schema export, a new HTTP POST sink for reasoning lifecycle and token events with optional HMAC/custom headers, dispatch wiring, and a changelog entry.

## Best Possible Solution

If maintainers want this capability, ship it as an explicit opt-in Telegram reasoning egress path that uses guarded outbound HTTP utilities, preserves sink event ordering, documents payloads and trust boundaries, updates generated config/schema docs, and adds focused tests for endpoint policy, credential resolution, HMAC/custom headers, lifecycle order, and streaming-off delivery.

## Reproduction Assessment

Yes. A read-only source check against the final PR head reproduces the remaining review failures: the sink posts with raw `fetch`, lifecycle events are fired without serialization, and generated config/docs search shows no new `reasoningStreamSink` entry.

## Solution Assessment

No. The direction may be useful, but the current patch is not the narrowest safe implementation because it exposes a new reasoning egress surface before using existing guarded fetch patterns and before aligning the public config/docs contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard sink requests with the SSRF fetch path:** `extensions/telegram/src/reasoning-stream-sink.ts:71-76`
  - body: `reasoningStreamSink.url` is operator config, but it can point at arbitrary http(s) targets and this code posts reasoning text plus Telegram/session identifiers with raw `fetch`. Use `fetchWithSsrFGuard` or an equivalent narrow SDK helper so the sink gets the repo's DNS pinning, private-network policy, redirect/header handling, timeout cleanup, and dispatcher release behavior before exposing this egress surface.
  - confidence: 0.9
- **[P2] Serialize sink events before relying on lifecycle order:** `extensions/telegram/src/reasoning-stream-sink.ts:99-114`
  - body: `onToken` starts the `reasoning_start` and `reasoning_stream` POSTs back-to-back, and `onEnd` starts another POST without waiting for prior sends. Under normal network latency the receiver can observe stream or end events before earlier lifecycle events; queue the POST promises per sink or add explicit sequencing and document out-of-order delivery.
  - confidence: 0.87
- **[P2] Regenerate config schema docs for the new key:** `src/config/zod-schema.providers-core.ts:338-346`
  - body: The live Zod schema adds `channels.telegram.reasoningStreamSink`, but the generated config schema/docs surface is not updated, so `openclaw config schema`, Control UI config forms, and `pnpm config:docs:check` will not agree with validation. Regenerate the config artifacts and add user-facing Telegram/thinking docs for the sink payloads and egress boundary.
  - confidence: 0.86
