## Summary

Keep open. Current main does not implement `reasoningStreamSink`, and the PR is directionally scoped to the Telegram plugin, but it still has blocking generated-config and security issues before it is safe to merge.

## What This Changes

This PR adds a Telegram `reasoningStreamSink` config/runtime/docs/API surface that posts reasoning lifecycle events to an external HTTPS endpoint with optional SecretInput-backed headers and HMAC signing.

## Best Possible Solution

Land only after the bundled channel metadata is regenerated, reasoning sink fetches opt out of debug capture with focused tests, and a maintainer approves the opt-in reasoning egress behavior.

## Reproduction Assessment

Yes for the blockers: static inspection of the PR head versus current main shows the generated channel metadata file is missing, and the sink fetch path can be traced to default debug capture of string request bodies. End-user feature reproduction is not applicable to this PR review.

## Solution Assessment

No, not yet. The Telegram plugin boundary and SecretInput approach are reasonable, but the PR needs the served config-schema surface and capture opt-out before it is the narrow maintainable fix.
