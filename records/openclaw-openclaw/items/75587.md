## Summary

Keep open. The PR is not merge-ready because it adds a security-sensitive external egress path that still weakens the guarded public-host SSRF boundary, and its schema/docs accept HTTP even though runtime rejects non-HTTPS sink URLs. The existing ClawSweeper comment identified the same blockers, and the latest diff still contains them.

## What This Changes

The PR adds Telegram `reasoningStreamSink` config/schema/docs/changelog, a fire-and-forget HTTP reasoning sink with HMAC/SecretInput support, Telegram reasoning callback wiring, generated config baseline hash updates, an SDK type export, and a small tool-display serialization change.

## Best Possible Solution

Keep the feature branch open and land it only after the sink preserves the default guarded public-host SSRF boundary or adds an explicitly reviewed private-network opt-in, makes HTTPS a validation/docs contract, and adds focused tests for the new egress behavior.

## Reproduction Assessment

Yes. A read-only source review reproduces the blockers: the PR builds an allowed-host policy from `reasoningStreamSink.url`, current main skips private-network checks for allowed hostnames, and runtime HTTPS enforcement conflicts with the PR schema/docs.

## Solution Assessment

No. The implementation direction is plausible, but the current patch is not the safest maintainable solution until SSRF policy and HTTPS validation/docs are aligned at the config boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Keep sink URLs on the default SSRF policy:** `extensions/telegram/src/reasoning-stream-sink.ts:60`
  - body: `ssrfPolicyFromHttpBaseUrlAllowedHostname(config.url)` treats the user-configured sink host as an allowed hostname. On current main that skips literal and resolved private-IP checks, so HTTPS private or special-use targets can bypass the documented public-host boundary. Use the default guarded policy here, or add an explicitly reviewed private-network opt-in with tests.
  - confidence: 0.94
- **[P2] Require HTTPS in the config contract:** `src/config/zod-schema.providers-core.ts:346-348`
  - body: The runtime passes `requireHttps: true`, but this schema accepts `http:` and the docs describe `http://` as valid. A configured HTTP sink will validate successfully and then fail each POST at runtime, so reject non-HTTPS during validation and update the docs/generated baselines, or intentionally allow plaintext by removing the runtime requirement.
  - confidence: 0.9
