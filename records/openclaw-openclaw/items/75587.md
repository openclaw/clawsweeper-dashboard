## Summary

Keep open. The PR is useful source work, but the latest head still is not a safe merge candidate: the new sink policy can bypass the default private-network SSRF block, and sink-only consumers still do not receive the native reasoning-end lifecycle event at the time reasoning actually ends.

## What This Changes

The PR adds Telegram `reasoningStreamSink` config/schema/docs/changelog support, a signed HTTP sink module, dispatch wiring, an SDK type export, generated config baseline updates, and a small `scripts/tool-display.ts` serialization cleanup.

## Best Possible Solution

Land this only after the sink keeps the default public-host SSRF boundary or adds an explicit documented private-network opt-in, emits `reasoning_end` from the existing native `onReasoningEnd` lifecycle with idempotent cleanup, and includes focused tests for streaming-off delivery, end timing, SecretInput/HMAC behavior, and the security boundary.

## Reproduction Assessment

Yes. A read-only diff/source check reproduces both blockers: the PR derives an allowed-host SSRF policy from arbitrary sink config, and it enables sink-only reasoning stream callbacks while leaving lifecycle end tied to the Telegram preview lane.

## Solution Assessment

No. The implementation direction is plausible, but the current patch is not the narrowest safe solution until it preserves the default SSRF guard semantics and follows the existing `onReasoningStream`/`onReasoningEnd` contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the default SSRF policy for sink URLs:** `extensions/telegram/src/reasoning-stream-sink.ts:56`
  - body: This derives an allowed-host policy from the configured sink URL. In current main, exact `allowedHostnames` skip private-network checks, so a URL like `http://169.254.169.254/...` would bypass the public-host boundary documented for this option. Keep the default guarded fetch policy here, or add an explicit private-network opt-in with docs and tests.
  - confidence: 0.93
- **[P2] Forward reasoning-end events to the sink:** `extensions/telegram/src/bot-message-dispatch.ts:1124-1125`
  - body: The sink-only path installs `onReasoningStream`, but `onReasoningEnd` remains present only when `reasoningLane.stream` exists. Native `thinking_end` events already emit `onReasoningEnd`, so a sink with Telegram previews off will not receive `reasoning_end` until dispatch cleanup and external consumers can stay active through the rest of the turn.
  - confidence: 0.9
