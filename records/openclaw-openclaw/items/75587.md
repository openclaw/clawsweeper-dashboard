## Summary

Keep open. Current main lacks this feature and the PR is directionally aligned, but the latest diff still has blocking issues: stale bundled channel config metadata and sensitive reasoning egress using default debug proxy capture.

## What This Changes

The PR adds a Telegram `reasoningStreamSink` config/runtime/docs/API type path that posts reasoning lifecycle events to an HTTPS endpoint with optional SecretInput-backed headers and HMAC signing.

## Best Possible Solution

Land only after regenerated bundled channel metadata is committed, reasoning-sink fetches opt out of proxy capture with focused tests, and a maintainer approves the opt-in reasoning egress behavior.

## Reproduction Assessment

Yes for the blockers: static inspection of the PR diff versus current main shows the missing generated channel metadata, and the fetch/capture code path shows request bodies are persisted unless `capture: false` is set. User-facing feature reproduction is otherwise not applicable for this review.

## Solution Assessment

No, not yet. The Telegram/plugin boundary is reasonable, but the implementation needs the generated channel schema surface and the debug-capture privacy fix before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Regenerate bundled channel metadata:** `src/config/zod-schema.providers-core.ts:351-359`
  - body: Adding `reasoningStreamSink` to `TelegramAccountSchemaBase` changes the bundled Telegram channel config surface, but this PR does not update `src/config/bundled-channel-config-metadata.generated.ts`. Since `src/config/schema.ts` merges that generated metadata into the served config schema, Control UI/config schema consumers will not see the new Telegram key until `pnpm config:channels:gen` output is committed.
  - confidence: 0.9
- **[P2] Disable debug capture for reasoning sink posts:** `extensions/telegram/src/reasoning-stream-sink.ts:75-80`
  - body: This guarded fetch call leaves default capture enabled. `fetchWithSsrFGuard` records request bodies whenever debug proxy capture is enabled, so each reasoning sink POST can persist reasoning text and session context in the local capture store; pass `capture: false` and add an audit context for this sensitive egress path.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds opt-in reasoning egress but currently lets debug proxy capture persist reasoning POST bodies unless capture is explicitly disabled.

Concerns:

- **[medium] Reasoning POST bodies can enter debug capture:** `extensions/telegram/src/reasoning-stream-sink.ts:75`
  - body: The new sink sends reasoning text/session context through `fetchWithSsrFGuard` without `capture: false`; current guarded fetch calls `captureHttpExchange` by default, and that helper persists string request bodies into the capture store.
  - confidence: 0.86
