## Summary

Keep open: the PR targets a real linked bug, but the one-line guard appears incomplete because it can preserve the stored override while the same model-selection state still refuses to select that disallowed key for the actual run.

## What This Changes

The PR changes `createModelSelectionState` so disallowed stored model overrides are not cleared when `modelOverrideSource` is `"user"`.

## Best Possible Solution

Fix and test the selected runtime model path, not only override persistence: an accepted user-sourced model override should remain the selected provider/model after compaction, while auto-fallback overrides stay clearable.

## Reproduction Assessment

Yes, source-reproducible: the PR’s stated disallowed-key case reaches the changed reset branch, but the later stored-override application still requires `visibilityPolicy.allowsKey(key)` before selecting the user override. I did not run a live compaction scenario in this read-only review.

## Solution Assessment

No: this is not the best fix as submitted because it preserves the stored fields without proving or ensuring the returned model-selection state uses the user-selected model for the actual run.
