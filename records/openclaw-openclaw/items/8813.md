## Summary

Keep open: current main and the latest release still emit read-based SKILL.md guidance, and the related open PRs do not already implement the requested Skill-tool preference. The remaining work is a prompt/tool contract decision rather than cleanup closure.

## What This Changes

The issue requests that the Skills system prompt prefer invoking a registered Skill tool over reading SKILL.md directly when that tool is available.

## Best Possible Solution

Define a durable skill-loading contract, pass runtime tool availability into the Skills prompt/formatter path, prefer the supported skill-loading tool only when present, keep the read fallback, and update focused tests plus public docs.

## Reproduction Assessment

Yes, as a source-level gap: current main still emits read-based SKILL.md guidance and does not pass available-tool information into `buildSkillsSection`. I did not run a live model scenario because this read-only review can trace the prompt path directly.

## Solution Assessment

Unclear until maintainers choose the trigger contract. The proposed bare `Skill` name check is narrow, but an explicit capability marker or the progressive `skill_view` / `skill_search` direction may be safer for long-term plugin compatibility.
