## Summary

Keep open. Current main still emits read-based Skills guidance and does not pass registered tool availability into the Skills section, so the requested Skill-tool preference is not implemented and remains a prompt-policy decision.

## What This Changes

The issue requests that the Skills system prompt prefer invoking an available Skill tool over read-based SKILL.md loading.

## Best Possible Solution

Define the Skills loading contract, then update src/agents/system-prompt.ts to prefer a registered Skill tool only under that contract, preserve the read fallback, and align focused tests plus https://docs.openclaw.ai/concepts/system-prompt.

## Reproduction Assessment

Yes, source-level: current main hardcodes read-based Skills guidance and the call site does not pass availableTools into buildSkillsSection, while runtime tool names do reach buildAgentSystemPrompt. I did not run a live prompt generation path because this was a read-only review.

## Solution Assessment

Unclear as proposed. The simple name-based branch is mechanically narrow, but the maintainable solution should first decide whether a plugin tool named Skill is trusted by name or needs an explicit capability/metadata contract.
