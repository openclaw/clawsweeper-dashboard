## Summary

Keep open: current main still emits mandatory read-based SKILL.md guidance even though registered tool names are already available earlier in prompt construction, so the requested Skill-tool preference remains unimplemented.

## What This Changes

Request the Skills system prompt to invoke a registered Skill tool for skill loading when present, falling back to read-based SKILL.md loading otherwise.

## Best Possible Solution

Define the Skill-loader contract, then update `src/agents/system-prompt.ts`, focused prompt tests, and the public system-prompt docs to prefer that registered loader while preserving the existing read fallback.

## Reproduction Assessment

Yes for the reported current behavior. Source inspection on current main shows tool names are normalized into `availableTools`, but `buildSkillsSection` cannot see them and still emits read-based SKILL.md guidance.

## Solution Assessment

Unclear until maintainers choose the contract. A normalized `skill` tool-name check would be narrow, but an explicit capability marker may be safer for plugin-tool collision semantics.
