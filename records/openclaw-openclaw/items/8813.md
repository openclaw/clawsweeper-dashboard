## Summary

Keep open. Current main still emits read-based Skills guidance even when a `Skill` tool name can reach prompt construction, and the adjacent open #63940 does not implement this preference.

## What This Changes

Teach the Skills system prompt to invoke a registered `Skill` tool instead of telling the model to read `SKILL.md` when that tool is available.

## Best Possible Solution

Add a focused prompt-builder path that detects the approved Skill-tool signal, emits non-contradictory Skill-tool loading guidance, preserves the read fallback, and updates focused tests plus system-prompt docs.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `buildAgentSystemPrompt` with `skillsPrompt` and `toolNames` containing `Skill`; current main computes `availableTools` but `buildSkillsSection` still emits read guidance.

## Solution Assessment

Unclear pending maintainer choice on the detection contract. The narrow implementation belongs in `src/agents/system-prompt.ts` with matching tests/docs, but it should avoid accidental collisions with unrelated tools named `Skill`.
