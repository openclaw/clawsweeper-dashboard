## Summary

Keep open: current main and the latest release still hardcode read-based skill loading and do not prefer a registered Skill tool; related open prompt/skills PRs overlap the area but do not settle this exact contract.

## What This Changes

Request to have the Skills system prompt prefer invoking a registered Skill tool when available, while preserving the existing read-based fallback.

## Best Possible Solution

Define a durable skill-loading contract, pass runtime tool availability and caller casing into the Skills prompt path, prefer the supported skill-loading tool only when present, preserve the read fallback, and update focused tests plus public docs.

## Reproduction Assessment

Yes, as a source-level behavior check: current main still emits read-based SKILL.md guidance and does not pass available-tool information into buildSkillsSection. I did not run a live model scenario because the prompt path is directly traceable in source.

## Solution Assessment

Unclear until maintainers choose the supported trigger contract. The proposed bare Skill-name check is narrow, but an explicit capability marker or the progressive skill-tool direction may be safer for long-term plugin compatibility.
