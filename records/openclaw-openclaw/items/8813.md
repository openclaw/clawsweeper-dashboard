## Summary

Keep open: current main and v2026.5.12 still hardcode read-based skill loading, while plugin tool registration and tool-name propagation make the requested behavior plausible. The remaining question is the supported Skill-tool trigger contract and coordination with open progressive-skill prompt work.

## What This Changes

The issue requests changing the Skills prompt so a runtime-exposed Skill-like tool is preferred over direct read-based SKILL.md loading.

## Best Possible Solution

Define a durable Skills-loading contract, pass runtime tool availability into the Skills prompt and formatter, prefer a registered Skill or skill_view-style tool only when that exact runtime tool is available, preserve the read fallback and exact-location guard, then update focused tests and public docs.

## Reproduction Assessment

Yes, as a source-level gap: current main normalizes tool names but buildSkillsSection receives only skillsPrompt/readToolName and always emits read-based SKILL.md loading guidance. I did not run a live model scenario because this was a read-only review and the code path is direct.

## Solution Assessment

Unclear until maintainers choose the trigger contract. The narrow implementation path is in src/agents/system-prompt.ts and src/agents/skills/skill-contract.ts, but the proposed bare tool-name check may be less durable than an explicit capability or alignment with skill_view/skill_search policy.
