## Summary

Current main and the latest release still use unconditional read-based skill loading, and no merged or viable canonical item implements the requested preference. The issue remains useful, but the permanent plugin contract needs maintainer judgment before implementation.

## What This Changes

Requests that the system prompt prefer a registered Skill tool over direct SKILL.md reads while preserving the read fallback.

## Best Possible Solution

Add a generic, explicit skill-loader capability to the appropriate plugin tool contract, carry that prepared fact into prompt construction, prefer the declared loader when present, and retain read as the canonical fallback.

## Reproduction Assessment

Yes at source level: current main receives registered tool names but the skills section still unconditionally directs the model to read SKILL.md. This establishes the reported policy gap without a runtime reproduction.

## Solution Assessment

No for the proposed tool-name check: the narrow edit is plausible, but the maintainable solution needs an explicit generic capability so core does not assign special semantics to an arbitrary plugin tool name.
