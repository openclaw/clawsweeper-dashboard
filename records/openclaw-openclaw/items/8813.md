## Summary

Current main and v2026.6.5 still instruct agents to load skills with `read`, and the only adjacent PR closed unmerged without implementing Skill-tool preference. The request remains useful, but the proposed literal `Skill` name check would create an implicit plugin contract, so maintainers should choose a generic capability boundary first.

## What This Changes

Prefer a registered skill-loading tool over direct SKILL.md reads in the system prompt, while retaining read-based loading as the fallback.

## Best Possible Solution

Add or reuse one explicit generic plugin capability for skill loading, have prompt construction invoke the advertised loader with the listed skill identity, and preserve direct `read` loading when no loader capability is present.

## Reproduction Assessment

Yes, from source with high confidence: current main always renders read-based Skills guidance, and focused tests assert that exact instruction. This establishes the reported limitation without requiring a runtime failure reproduction.

## Solution Assessment

No, not as the proposed literal tool-name check. The narrow maintainable direction is a generic, collision-resistant capability integrated with the existing prompt/plugin boundary, with `read` retained as fallback.
