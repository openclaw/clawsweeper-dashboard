## Summary

Keep open. Current main still emits read-based Skills guidance and does not pass registered tool availability into the Skills section, so the requested Skill-tool preference is not implemented and still needs maintainer prompt-policy judgment.

## What This Changes

The issue requests system prompt behavior that prefers invoking a registered Skill tool over reading SKILL.md directly when both are available.

## Best Possible Solution

Choose a maintainer-approved Skills loading contract, then update `src/agents/system-prompt.ts` to prefer a registered Skill tool only under that contract, preserve the read fallback, and align focused tests plus `docs/concepts/system-prompt.md`.

## Reproduction Assessment

Yes, source-level: current main hardcodes read-based Skills guidance and the call site does not pass `availableTools` into `buildSkillsSection`. I did not run a live prompt generation path because this was a read-only review.

## Solution Assessment

Unclear as proposed. The simple tool-name branch is mechanically narrow, but the best maintainable solution should first define whether arbitrary plugin tools named Skill are trusted or whether an explicit capability marker is required.
