## Summary

Keep open: current main and the latest release still hardcode read-based Skills prompt guidance, while registered plugin tool names reach prompt construction. The related prompt PR was closed unmerged and did not implement the requested Skill-tool preference, so the remaining blocker is maintainer approval of the skill-loader contract.

## What This Changes

Request that the Skills system-prompt section prefer a registered Skill loader tool over read-based SKILL.md loading when that tool is available.

## Best Possible Solution

Define the skill-loader capability semantics, then update the prompt builder, focused prompt tests, and system-prompt docs to prefer the registered loader while preserving the read fallback.

## Reproduction Assessment

Yes. The issue is source-reproducible because current main and v2026.6.8 unconditionally emit read-based skill-loading guidance even though tool names are available to prompt construction.

## Solution Assessment

No, not as proposed. Checking only for a tool named Skill is plausible, but the best fix should first settle whether this is name-based or an explicit skill-loader capability, then preserve the existing read fallback.
