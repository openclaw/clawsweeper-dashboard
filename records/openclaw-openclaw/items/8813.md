## Summary

Keep open: current main and v2026.5.20 still use read-based skill loading guidance, while the related open PRs are adjacent prompt-policy work and do not already implement the requested Skill-tool preference.

## What This Changes

Request that the skills system prompt prefer a registered Skill tool over reading SKILL.md files when such a tool is available.

## Best Possible Solution

Define a durable skill-loading contract, pass runtime tool availability into the Skills prompt/formatter path, prefer the supported skill-loading tool only when present, keep the read fallback, and update focused tests plus public docs.

## Reproduction Assessment

Yes, as a source-level gap: current main still emits read-based SKILL.md guidance and does not pass available-tool information into buildSkillsSection. I did not run a live model scenario because the read-only review can trace the prompt path directly.

## Solution Assessment

Unclear until maintainers choose the supported trigger contract. The proposed bare Skill-name check is narrow, but an explicit capability marker or the progressive skill-tool direction may be safer for long-term plugin compatibility.
