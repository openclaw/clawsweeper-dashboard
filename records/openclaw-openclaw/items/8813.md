## Summary

Keep this issue open: current main and the latest release still instruct agents to load skills through the read tool, and the requested Skill-tool preference remains a product/API contract decision rather than an implemented behavior.

## What This Changes

The issue requests changing the agent system prompt so a registered Skill tool is preferred over reading SKILL.md files directly.

## Best Possible Solution

Define an explicit generic skill-loader capability or plugin contract, pass that resolved capability and tool casing into buildSkillsSection, prefer the registered loader when present, preserve the existing read fallback, and document both paths.

## Reproduction Assessment

Yes. Current main source unconditionally emits read-based skill-loading guidance even though registered tool names are available to prompt construction.

## Solution Assessment

No. The proposed simple Skill-name check is not yet the best permanent solution because it creates an implicit semantic contract around a generic tool name; an explicit skill-loader capability is safer.
