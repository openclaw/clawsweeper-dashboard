## Summary

Keep this open: current main still emits read-based Skills guidance even when a plugin-provided `Skill` tool could be present, and the adjacent open PR does not resolve that behavior.

## What This Changes

Request that the Skills prompt prefer invoking a registered `Skill` tool over reading `SKILL.md` when that tool is available.

## Best Possible Solution

Decide the Skill-tool detection contract, then update `src/agents/system-prompt.ts`, focused tests, and the public system-prompt docs so registered Skill loaders are preferred while the read fallback remains intact.

## Reproduction Assessment

Yes. Source inspection on current main shows tool names are normalized into `availableTools`, but `buildSkillsSection` still receives only `skillsPrompt` and `readToolName` and emits the read-based guidance.

## Solution Assessment

Unclear until maintainers choose the contract. A normalized `skill` tool-name check would be narrow, but an explicit skill-loader capability may be safer for plugin-tool collision semantics.
