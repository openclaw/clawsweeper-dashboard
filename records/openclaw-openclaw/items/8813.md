## Summary

Keep open: current main still instructs skills to load through `read` and the Skills section cannot branch on a registered `Skill` tool; the related prompt PR is open and adjacent but does not implement this contract.

## What This Changes

Request the Skills system prompt to prefer an available registered `Skill` tool over direct `read`-based `SKILL.md` loading.

## Best Possible Solution

Define the Skill-loader contract, then update `src/agents/system-prompt.ts`, `src/agents/skills/skill-contract.ts`, focused prompt tests, and system-prompt docs so registered loader tools are preferred while the existing read fallback remains intact.

## Reproduction Assessment

Yes for the current prompt behavior: source inspection shows the Skills section always emits read-based loading and cannot see a registered `Skill` tool. No runtime test was run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. A normalized `Skill` tool-name branch is narrow, but an explicit skill-loader capability may be safer and the formatter-level read instruction also needs alignment.
