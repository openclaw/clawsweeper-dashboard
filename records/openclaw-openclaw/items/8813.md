## Summary

Keep open: current main and v2026.5.12 still instruct skill loading through `read`, while plugin tools can be registered and runtime tool names reach prompt construction. The remaining work is a maintainer prompt-policy/API decision, not conservative cleanup closure.

## What This Changes

Request that the Skills system-prompt section prefer an available Skill-loading tool over read-based SKILL.md loading while preserving the read fallback.

## Best Possible Solution

Define a durable skill-loading contract, pass runtime tool availability into the Skills prompt and formatter, prefer the supported Skill-loading tool only when present, preserve the read fallback, and update focused tests plus public docs.

## Reproduction Assessment

Yes, as a source-level gap: `buildSkillsSection` always emits read-based SKILL.md guidance and receives no `availableTools`, even though tool names are normalized upstream and runtime tools are passed into prompt construction. I did not run a live model scenario because this was a read-only review and the code path is direct.

## Solution Assessment

Unclear until maintainers choose the trigger contract. The proposed bare Skill-name check is narrow, but an explicit capability or the `skill_view`/`skill_search` direction may be safer for long-term plugin compatibility.
