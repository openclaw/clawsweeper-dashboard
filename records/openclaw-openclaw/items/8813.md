## Summary

Keep open: current main and v2026.5.12 still hard-code read-based Skills guidance, while related open prompt work is adjacent and does not implement the registered Skill-tool preference. The source gap is clear, but the final trigger contract needs maintainer prompt-policy judgment before automation chooses a behavior.

## What This Changes

The issue asks OpenClaw to prefer a registered Skill-style tool over read-based SKILL.md loading in the Skills system-prompt guidance when that tool is available.

## Best Possible Solution

Define one durable Skills loading contract, pass normalized available-tool information and caller casing into buildSkillsSection, preserve the read fallback and exact <location> guard, then update focused prompt tests and system-prompt docs.

## Reproduction Assessment

Yes, as a source-level gap: current main normalizes available tool names, but buildSkillsSection receives only skillsPrompt/readToolName and always emits read-based SKILL.md loading guidance. I did not run a live model scenario because this was a read-only review and the code path is direct.

## Solution Assessment

Unclear until maintainers choose the Skill-tool trigger contract. The narrow code path is src/agents/system-prompt.ts, but the proposed bare tool-name check may be less durable than an explicit skill-loading capability or an aligned skill_view/skill_search policy.
