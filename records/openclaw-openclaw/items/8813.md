## Summary

Keep open: current main and v2026.6.8 still hardcode read-based Skills prompt guidance, while plugin/tool names reach prompt construction and plugin tools can be registered. The related prompt PR was closed unmerged, and the open local skill router PR is only a partial overlap, so the remaining blocker is maintainer choice of the supported skill-loader contract.

## What This Changes

Request the system prompt to prefer a registered Skill or skill-loader tool over direct SKILL.md reads when that loader is available.

## Best Possible Solution

Define the supported skill-loader capability semantics, then update the prompt builder, focused prompt tests, and system-prompt docs to prefer that loader while preserving the read fallback.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.8 unconditionally emit read-based skill-loading guidance even though registered tool names are available to prompt construction.

## Solution Assessment

Unclear as proposed. Checking only for a tool named Skill is plausible, but the best fix should first settle whether the contract is name-based, namespaced, or an explicit skill-loader capability.
