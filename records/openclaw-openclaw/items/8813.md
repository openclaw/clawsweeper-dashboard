## Summary

Keep open: current main still gives read-based Skills guidance and has not implemented a registered Skill-tool preference, so the requested behavior remains valid and needs maintainer contract direction.

## What This Changes

Request the system prompt to prefer invoking a registered Skill tool for loading applicable skills while preserving the read-based fallback when no such tool is available.

## Best Possible Solution

Define the Skill-loader contract, then update src/agents/system-prompt.ts, focused prompt tests, and docs/concepts/system-prompt.md so registered loader tools are preferred while the read fallback remains intact.

## Reproduction Assessment

Yes. Source inspection on current main shows buildSkillsSection cannot see availableTools and always emits read-based SKILL.md guidance, while runtime tool names are already forwarded into prompt construction.

## Solution Assessment

Unclear until maintainers choose the contract. A normalized Skill tool-name branch is narrow, but an explicit skill-loader capability may be safer for plugin-tool collision semantics.
