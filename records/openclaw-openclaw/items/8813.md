## Summary

Keep open: current main and the latest release still emit read-based skill-loading guidance, while registered tool names already reach prompt construction; the remaining blocker is the product contract for when a plugin-owned skill-loader tool should receive special prompt authority.

## What This Changes

Request the native Skills system prompt to prefer a registered Skill-loading tool over `read` when available, while preserving the existing `read` fallback.

## Best Possible Solution

Define the supported skill-loader tool contract, then update `src/agents/system-prompt.ts`, focused prompt tests, and system-prompt docs while preserving the current `read` fallback and reconciling related skill-routing work.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.6.10` shows `buildSkillsSection` unconditionally emits read-based SKILL.md guidance even though registered tool names are available earlier in prompt construction.

## Solution Assessment

Unclear as proposed. Preferring a registered loader tool is plausible, but maintainers should choose whether the contract is name-based, namespaced, or an explicit skill-loader capability before implementation.
