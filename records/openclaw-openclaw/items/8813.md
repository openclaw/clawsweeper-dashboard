## Summary

Keep open: current main and the latest release still hardcode read-based Skills prompt guidance, while registered plugin tool names reach prompt construction; the remaining decision is the skill-loader contract, not whether the report is still relevant.

## What This Changes

The issue requests changing the Skills system prompt to prefer invoking a registered Skill loader tool instead of telling the model to read SKILL.md directly.

## Best Possible Solution

Define the skill-loader capability semantics, then update the prompt builder, focused prompt tests, and system-prompt docs to prefer the registered loader while preserving the read fallback.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.6 unconditionally emit read-based skill-loading guidance even though registered tool names are available to prompt construction. I did not run an end-to-end Pi extension repro in this read-only review.

## Solution Assessment

No, not as proposed: checking only for a tool named Skill is plausible but may be too generic for a durable plugin contract. The safer path is a maintainer-approved skill-loader capability or equivalent explicit semantic, with the current read fallback preserved.
