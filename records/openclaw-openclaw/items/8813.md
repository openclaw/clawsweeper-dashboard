## Summary

Keep open: current main and the latest release still emit read-based Skills guidance, and the overlapping skill-routing PRs are open policy work rather than a merged implementation of this Skill-tool preference.

## What This Changes

Request that the Skills system prompt prefer a registered Skill tool over reading SKILL.md when such a tool is available.

## Best Possible Solution

Define the supported skill-loading contract, then update src/agents/system-prompt.ts, focused prompt tests, and the public system-prompt docs while preserving the read fallback when that contract is absent.

## Reproduction Assessment

Yes. Source inspection on current main shows availableTools is built from tool names, but the Skills prompt builder does not receive it and still emits read-based SKILL.md guidance.

## Solution Assessment

Unclear. The requested Skill-name check is narrow, but maintainers need to choose whether that name is the supported contract or whether local_skill_route/progressive disclosure is the safer API.
