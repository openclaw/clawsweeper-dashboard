## Summary

Keep open: current main and the latest release still hardcode read-based Skills guidance, while plugin tool names are already forwarded into prompt construction. The adjacent prompt PR at https://github.com/openclaw/openclaw/pull/63940 is open and changes skill-selection wording, but it does not implement a registered Skill-tool preference.

## What This Changes

Request that the Skills system prompt invoke a registered Skill tool instead of telling the model to read SKILL.md when that tool is available.

## Best Possible Solution

Define one durable Skills loading contract, pass available-tool information and caller casing into buildSkillsSection, preserve the read fallback, and update focused prompt tests plus system-prompt docs.

## Reproduction Assessment

Yes as a source-level gap check: current main normalizes available tool names, but buildSkillsSection still receives only skillsPrompt/readToolName and always emits read-based SKILL.md loading guidance.

## Solution Assessment

Unclear until maintainers choose the Skill-tool trigger contract. The narrow code path is src/agents/system-prompt.ts, but a bare tool-name check may not be the safest durable policy.
