## Summary

Current main and v2026.6.5 still emit unconditional read-based skill-loading guidance, so the requested capability remains missing; however, making any tool named `Skill` authoritative would create an implicit plugin contract and needs a maintainer product/API decision.

## What This Changes

Request prompt guidance that invokes a registered skill-loading tool instead of reading `SKILL.md` directly when such a tool is available.

## Best Possible Solution

Add a typed, plugin-owned skill-loader capability or metadata contract, carry that prepared fact into prompt construction, prefer its caller-cased tool name when active, and retain direct `read` as the canonical fallback with focused prompt and plugin-contract tests.

## Reproduction Assessment

Yes at source level: current main and v2026.6.5 always emit read-based skill loading even when plugin tools are available. No live model run was needed to establish the missing prompt branch.

## Solution Assessment

No, the proposed tool-name check is not the best durable solution because it turns a generic name into an implicit core/plugin API; an explicit semantic capability is safer and maintainable.
