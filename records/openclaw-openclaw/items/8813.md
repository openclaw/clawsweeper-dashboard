## Summary

Keep open: current main and the latest release still emit read-based Skills prompt guidance even though runtime tool names reach prompt construction, and the related prompt PR was closed unmerged. The remaining blocker is the product/API contract for recognizing a real skill-loader tool, not source uncertainty.

## What This Changes

The issue asks OpenClaw to prefer invoking a registered Skill tool over telling the model to read SKILL.md directly when that loader is available.

## Best Possible Solution

Define a maintainer-approved skill-loader semantic, then update the prompt builder, focused prompt tests, and system-prompt docs to prefer that loader when present while preserving the read fallback.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.8 still unconditionally emit read-based skill-loading guidance while tool names are already available to prompt construction. I did not run a live Pi extension setup in this read-only review.

## Solution Assessment

No, not as a final design yet: checking only for a tool named `Skill` is plausible but may be too generic for a durable plugin contract. A capability-based or otherwise maintainer-approved loader semantic is the safer path.
