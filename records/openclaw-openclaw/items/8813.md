## Summary

Keep open: current main still emits read-based Skills guidance and has not implemented a registered Skill-tool preference; the remaining work is a prompt/plugin contract enhancement that needs maintainer direction.

## What This Changes

The issue requests changing the Skills section of the agent system prompt so a registered Skill tool is preferred over read-based SKILL.md loading when available.

## Best Possible Solution

Define the Skill-loader contract, then update src/agents/system-prompt.ts, focused prompt tests, and docs/concepts/system-prompt.md to prefer the registered loader while preserving the read fallback.

## Reproduction Assessment

Yes. Source inspection on current main shows buildSkillsSection cannot see availableTools and always emits read-based SKILL.md guidance even though runtime tool names are normalized earlier.

## Solution Assessment

Unclear until maintainers choose the contract. A normalized Skill tool-name branch is narrow, but an explicit skill-loader capability may be safer for plugin-tool collision semantics.
