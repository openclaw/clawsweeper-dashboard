## Summary

Keep open: current main still emits read-based Skills guidance even though runtime tool names are normalized and plugin tools can be registered, so the reported source-level gap remains. The remaining blocker is maintainer policy/API choice for the Skill-loading trigger, especially with overlapping open prompt-disclosure PRs.

## What This Changes

Requests that the Skills system prompt prefer invoking a registered Skill-loading tool over reading SKILL.md directly when such a tool is available.

## Best Possible Solution

Define a durable Skills-loading contract, pass runtime tool availability into the Skills prompt and formatter, prefer the supported Skill-loading tool only when that exact runtime tool is available, preserve the read fallback, and update focused tests plus public docs.

## Reproduction Assessment

Yes, as a source-level gap: current main normalizes runtime tool names but buildSkillsSection receives only skillsPrompt/readToolName and always emits read-based SKILL.md loading guidance. I did not run a live model scenario because this was a read-only review and the code path is direct.

## Solution Assessment

Unclear until maintainers choose the trigger contract. The proposed bare Skill-name check is narrow, but an explicit capability or the skill_view/skill_search direction may be safer for long-term plugin compatibility.
