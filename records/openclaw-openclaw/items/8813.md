## Summary

Keep open: current main and the latest release still hardcode read-based Skills prompt guidance, the related PR is closed unmerged, and the remaining work needs maintainer direction on skill-loader semantics.

## What This Changes

The issue requests that the built-in Skills system-prompt guidance prefer a registered Skill tool over direct SKILL.md reads when such a tool is available.

## Best Possible Solution

Define the skill-loader capability semantics, then update the prompt builder, tests, and docs to prefer that registered loader while preserving the current read fallback.

## Reproduction Assessment

Yes. Current main and `v2026.6.6` source unconditionally emit read-based skill-loading guidance even though registered tool names are available to prompt construction.

## Solution Assessment

No. The issue's simple `Skill` name check is plausible, but an explicit skill-loader capability or plugin contract is the safer long-term solution.
