## Summary

Keep open: current main and v2026.5.20 still hardcode read-based SKILL.md loading, while the related PRs are open, unmerged, and represent adjacent or alternative skill-routing designs rather than an implementation of the requested Skill-tool preference.

## What This Changes

Request that the Skills system prompt invoke a registered Skill tool instead of instructing the model to read SKILL.md, while preserving the read fallback when that tool is absent.

## Best Possible Solution

Define the supported skill-loading contract first, then update the Skills prompt, focused system-prompt coverage, and public docs while preserving the current read fallback when that contract is absent.

## Reproduction Assessment

Yes. Source inspection on current main shows tool names are normalized into availableTools, but buildSkillsSection still only receives skillsPrompt and readToolName and emits read-based SKILL.md guidance.

## Solution Assessment

Unclear until maintainers choose the supported trigger contract. The requested Skill-name check is narrow, but an explicit capability marker, local_skill_route, or progressive skill_view/search path may be safer for plugin compatibility.
