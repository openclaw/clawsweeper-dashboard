## Summary

Keep open: current main and v2026.5.12 still emit read-based Skills guidance even when tool names are normalized and forwarded into prompt construction. The adjacent open prompt PR changes skill-selection wording but still keeps the read-based loading path, so it does not supersede the registered Skill-tool request.

## What This Changes

The issue requests that the Skills system prompt prefer invoking a registered Skill tool over reading SKILL.md directly when such a tool is available.

## Best Possible Solution

Define one durable Skills loading contract, pass available-tool information and caller casing into buildSkillsSection, preserve the read fallback, and update focused prompt tests plus system-prompt docs.

## Reproduction Assessment

Yes, as a source-level gap check: current main normalizes available tool names, but buildSkillsSection still receives only skillsPrompt/readToolName and always emits read-based SKILL.md loading guidance.

## Solution Assessment

Unclear until maintainers choose the Skill-tool trigger contract. The narrow code path is src/agents/system-prompt.ts, but the requested bare tool-name check may not be the safest durable policy.
