## Summary

Current main and v2026.6.5 both retain read-only skill-loading guidance, and the only adjacent PR closed unmerged; this remains valid but product-sensitive agent-harness work rather than cleanup-ready work.

## What This Changes

Requests that the system prompt prefer a registered Skill tool over directly reading SKILL.md files.

## Best Possible Solution

Add an explicit plugin or harness skill-loader capability with a resolved callable tool name and version-refresh semantics, then render that path when present and retain read as the fallback.

## Reproduction Assessment

Yes at source level: when a skills prompt and a registered Skill tool are supplied, current main still produces only the read instruction because buildSkillsSection receives no available-tool capability.

## Solution Assessment

No, not exactly as proposed; name matching is easy to implement but does not define ownership, collisions, caller casing, or version-refresh behavior, so an explicit capability is the safer permanent contract.
