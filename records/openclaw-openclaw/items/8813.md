## Summary

Keep open: current main and v2026.5.18 still describe skill loading through `read`, while plugin tools can be registered and tool names already reach prompt construction. The remaining question is the supported prompt/tool contract, not cleanup closure.

## What This Changes

Requests that the Skills system prompt prefer a registered Skill-loading tool when available instead of always instructing the model to read SKILL.md files.

## Best Possible Solution

Define a durable skill-loading contract, pass runtime tool availability into the Skills prompt and formatter, prefer the supported skill-loading tool only when present, preserve the read fallback, and update focused tests plus public docs.

## Reproduction Assessment

Yes, as a source-level gap: current main always emits read-based SKILL.md guidance and does not pass `availableTools` into `buildSkillsSection`. I did not run a live model scenario because this was a read-only review and the code path is direct.

## Solution Assessment

Unclear until maintainers choose the trigger contract. The proposed bare Skill-name check is narrow, but an explicit capability or the `skill_view` / `skill_search` direction may be safer for long-term plugin compatibility.
