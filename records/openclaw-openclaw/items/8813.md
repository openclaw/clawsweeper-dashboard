## Summary

Keep open: current main still emits read-based Skills guidance and does not implement a registered Skill-tool preference; this remains an enhancement/product-contract decision rather than a stale, duplicate, or already-fixed item.

## What This Changes

The issue asks OpenClaw’s generated Skills system prompt to prefer invoking a registered `Skill` tool over reading `SKILL.md` directly when that tool is available.

## Best Possible Solution

Define the Skill-loader contract, then update `src/agents/system-prompt.ts`, focused prompt tests, and `docs/concepts/system-prompt.md` to prefer the registered loader while preserving the read fallback.

## Reproduction Assessment

Yes for the reported current behavior. Source inspection on current main shows registered tool names are normalized into `availableTools`, but `buildSkillsSection` cannot see them and still emits read-based `SKILL.md` guidance.

## Solution Assessment

Unclear until maintainers choose the contract. A normalized tool-name branch is narrow, but an explicit loader capability may be safer for plugin-tool collision semantics.
