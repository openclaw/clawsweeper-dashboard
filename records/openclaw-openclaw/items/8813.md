## Summary

Keep open: current main still hardcodes read-based SKILL.md guidance even though tool names are available earlier, and the adjacent #63940 PR remains open and does not implement a registered Skill-tool preference.

## What This Changes

Change the Skills system prompt to prefer invoking a registered `Skill` tool instead of `read` when that tool is available.

## Best Possible Solution

Define the Skill-loader detection contract, then update `src/agents/system-prompt.ts`, focused prompt tests, and the public system-prompt docs to prefer a registered Skill tool while preserving the existing read fallback.

## Reproduction Assessment

Yes. Source inspection on current main shows tool names are normalized into `availableTools`, but `buildSkillsSection` cannot see them and still emits read-based SKILL.md guidance.

## Solution Assessment

Unclear until maintainers choose the contract. A normalized `skill` tool-name check would be narrow, but an explicit capability marker may be safer for plugin-tool collision semantics.
