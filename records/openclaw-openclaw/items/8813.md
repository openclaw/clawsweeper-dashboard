## Summary

Keep open: current main still emits read-based Skills guidance even when a registered Skill tool could be present, and the adjacent open prompt PR does not implement the Skill-tool preference contract.

## What This Changes

Request the Skills system prompt to prefer invoking a registered Skill tool for SKILL.md loading while retaining the read fallback when that tool is absent.

## Best Possible Solution

Define one Skills loading contract, pass available-tool information and caller casing into buildSkillsSection, prefer a registered Skill tool only under that contract, keep exact-location read fallback, and update focused prompt tests plus system-prompt docs.

## Reproduction Assessment

Yes as a source-level gap check. Current main normalizes available tool names but buildSkillsSection still receives only skillsPrompt/readToolName and always emits read-based SKILL.md loading guidance.

## Solution Assessment

Unclear until maintainers choose the Skill-tool contract. The narrow implementation likely belongs in src/agents/system-prompt.ts, but the trigger should avoid accidental collisions with arbitrary plugin tools named Skill.
