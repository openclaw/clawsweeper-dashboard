## Summary

Keep open: current main still builds the Skills prompt around direct `read` loading, while registered tool names are normalized elsewhere and not passed into the Skills section. The remaining decision is the Skill-loader contract, not stale cleanup.

## What This Changes

The issue asks OpenClaw to prefer a registered `Skill` loader tool over direct read-based `SKILL.md` loading in the system prompt.

## Best Possible Solution

Define the Skill-loader contract, then update prompt construction, skill catalog formatter text, focused prompt tests, and public system-prompt docs together while preserving the read fallback.

## Reproduction Assessment

Yes for the current prompt behavior: source inspection on current main shows the Skills section always emits read-based loading and cannot see a registered `Skill` tool. No runtime test was run because this was a read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. A normalized `skill` tool-name branch is narrow, but an explicit loader capability may be safer and the overlapping prompt work should be reconciled first.
