## Summary

Keep open: current main and the latest release still hardcode read-based SKILL.md loading, while related open PRs are adjacent or alternative skill-routing designs rather than a merged implementation of the requested Skill-tool preference.

## What This Changes

Requests prompt construction to prefer invoking a registered Skill tool to load skill instructions instead of always telling the model to read SKILL.md files.

## Best Possible Solution

Define the supported skill-loading contract, then update the Skills prompt, focused tests, and public docs while preserving the existing read fallback when that contract is absent.

## Reproduction Assessment

Yes. Current main can be checked from source: the Skills section still emits read-based SKILL.md guidance and does not pass available-tool information into buildSkillsSection.

## Solution Assessment

Unclear until maintainers choose the supported trigger contract. The requested Skill-name check is narrow, but an explicit capability marker, local_skill_route, or progressive skill_view/search path may be safer for plugin compatibility.
