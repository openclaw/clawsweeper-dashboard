## Summary

Keep open: current main and the latest release still hardcode read-based Skills guidance, and the related skill-routing PRs are open, unmerged, and represent competing product/API directions rather than a shipped fix.

## What This Changes

Requests that the Skills system prompt prefer invoking a registered Skill tool over reading SKILL.md directly when that tool is available.

## Best Possible Solution

Define the supported skill-loading contract first, then update system prompt construction, focused tests, and public docs while preserving the current read fallback when that contract is absent.

## Reproduction Assessment

Yes, at source level: current main builds availableTools from tool names, but buildSkillsSection does not receive it and still emits read-based SKILL.md guidance even when a Skill tool would be present.

## Solution Assessment

Unclear: the small Skill-name check is plausible, but maintainers still need to choose whether a reserved Skill tool name, local_skill_route, or a plugin SkillRouter API is the durable contract.
