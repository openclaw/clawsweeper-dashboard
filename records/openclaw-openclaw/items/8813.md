## Summary

Keep open: current main still emits read-based Skills guidance and does not implement a registered Skill-tool preference; this remains a prompt/plugin contract enhancement that needs maintainer direction.

## What This Changes

The issue requests changing the Skills system-prompt guidance to prefer a registered `Skill` loader tool over direct `read` calls when that tool is available.

## Best Possible Solution

Define the Skill-loader contract, then update `src/agents/system-prompt.ts`, focused prompt tests, and `docs/concepts/system-prompt.md` to prefer the registered loader while preserving the read fallback.

## Reproduction Assessment

Yes, for the reported current behavior by source inspection. Tool names are normalized into `availableTools`, but `buildSkillsSection` cannot see them and still emits read-based `SKILL.md` guidance.

## Solution Assessment

Unclear until maintainers choose the contract. A normalized tool-name branch is narrow, but an explicit loader capability may be safer for plugin-tool collision semantics.
