## Summary

Keep open: current main still emits read-based Skills guidance, the Skills section cannot see a registered `Skill` loader tool, and related open prompt PRs do not settle this contract.

## What This Changes

Request to prefer a registered `Skill` loader tool over `read` when the system prompt tells the model how to load `SKILL.md` files.

## Best Possible Solution

Define the Skill-loader contract, then update prompt construction, skill formatter text, focused prompt tests, and public system-prompt docs together while preserving the read fallback.

## Reproduction Assessment

Yes for the current prompt behavior: source inspection on current main shows the Skills section always emits read-based loading and cannot see a registered `Skill` tool. No runtime test was run because this was a read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. A normalized `skill` tool-name branch is narrow, but an explicit loader capability may be safer and the overlapping prompt PRs should be reconciled first.
