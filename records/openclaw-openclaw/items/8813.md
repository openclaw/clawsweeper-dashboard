## Summary

Keep open. Current main still renders read-based skill-loading guidance even when runtime tool names are available, and the skills catalog text also contains read-tool instructions. The related open PR #63940 is adjacent prompt wording work, but the provided discussion says it does not implement Skill-tool preference.

## What This Changes

The issue asks OpenClaw to prefer invoking a registered Skill tool over direct SKILL.md reads in the generated skills system prompt.

## Best Possible Solution

Implement a narrow prompt-builder fix that detects a registered Skill tool from normalized tool names, preserves caller casing when naming it, emits Skill-tool loading guidance when present, and keeps the existing read fallback otherwise. The final rendered skills block should not contain contradictory read-tool loading text, and docs/tests should cover both paths.

## Reproduction Assessment

Yes. A source-level reproduction is available by calling buildAgentSystemPrompt with skillsPrompt plus toolNames containing Skill; current main has no branch that can produce Skill-tool loading guidance and tests only assert the read path.

## Solution Assessment

Yes, a focused change in the system prompt and skills prompt text is the narrowest maintainable path. A broader plugin-specific workaround would be weaker because the runtime already forwards generic tool names into prompt construction.

Implement a generic fix in `src/agents/system-prompt.ts`: use normalized available tool names from `toolNames`, preserve caller casing via the existing resolver, and when a Skill tool is present emit guidance to invoke that tool with the selected skill name. Ensure the final rendered skills block does not also contain contradictory read-tool loading text from `skillsPrompt` catalog lines or snapshots; either sanitize/rewrite the known catalog loading lines in `buildSkillsSection` when Skill is available, or deliberately update the formatter contract with tests.
