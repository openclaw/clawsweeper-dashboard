## Summary

Keep open: current main and the latest release still instruct skills to be loaded through the read tool, and the open related PRs are adjacent policy work rather than a merged implementation of the requested Skill-tool preference.

## What This Changes

The issue requests system-prompt behavior that prefers invoking a registered Skill tool over reading SKILL.md directly when that tool is available.

## Best Possible Solution

Define the supported skill-loading contract, then update `src/agents/system-prompt.ts`, focused prompt tests, and the public system-prompt docs while preserving the current read fallback when that contract is absent.

## Reproduction Assessment

Yes. Source inspection on current main shows `availableTools` is built from tool names, but the Skills prompt builder does not receive it and still emits read-based SKILL.md guidance.

## Solution Assessment

Unclear until maintainers choose the supported trigger contract. The requested Skill-name check is narrow, but an explicit capability marker or progressive `skill_view`/`skill_search` path may be safer for plugin compatibility.
