## Summary

Keep open: current main and v2026.6.8 still hardcode read-based Skills prompt guidance, while runtime tool names reach prompt construction and plugin tools can be registered. The remaining decision is the supported skill-loader contract, because the name-only Skill-tool proposal is plausible but not yet a maintainer-approved product/API direction.

## What This Changes

The issue requests that the Skills system-prompt section prefer invoking a registered Skill loader tool when available instead of always instructing the model to read SKILL.md files directly.

## Best Possible Solution

A documented skill-loader contract is implemented in system prompt generation with focused prompt tests and docs, while preserving the read fallback whenever no supported loader is registered.

## Reproduction Assessment

Yes, at source level. Current main unconditionally emits read-based skill-loading guidance when skillsPrompt is present, even though available tool names are already normalized in the same prompt builder.

## Solution Assessment

Unclear as proposed. Checking only for a tool named Skill is plausible, but the best fix should first settle whether the loader contract is name-based, namespaced, or an explicit capability.
