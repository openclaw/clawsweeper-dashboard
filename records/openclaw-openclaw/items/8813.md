## Summary

Keep open: current main and the latest release still hardcode read-based SKILL.md loading, while the remaining decision is what registered loader-tool contract OpenClaw should support.

## What This Changes

Request the Skills system prompt to prefer a registered Skill loader tool over read-based SKILL.md loading when that tool is available.

## Best Possible Solution

Define the supported skill-loader tool capability or naming contract, then update src/agents/system-prompt.ts, focused prompt tests, and system-prompt docs while preserving the current read fallback.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.10 unconditionally emit read-based skill-loading guidance even though registered tool names are available to prompt construction.

## Solution Assessment

Unclear as proposed. A registered loader-tool preference is plausible, but the best fix should first settle whether the contract is name-based, namespaced, or an explicit skill-loader capability.
