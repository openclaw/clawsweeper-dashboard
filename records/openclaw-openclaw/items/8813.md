## Summary

Keep open: current main and v2026.5.12 still hardcode read-based skill loading, while runtime tool-name propagation and plugin tool registration make the requested Skill-tool preference plausible. The remaining blocker is a maintainer product/API decision about the supported trigger contract.

## What This Changes

Request runtime-aware Skills prompt guidance that invokes a registered Skill tool instead of `read` when that tool is available.

## Best Possible Solution

Define a durable Skills-loading contract, thread runtime tool availability into the Skills prompt and formatter, preserve the read fallback and exact-location safety, then update focused tests and public docs.

## Reproduction Assessment

Yes, as a source-level gap: current main normalizes runtime tool names but `buildSkillsSection` receives only `skillsPrompt` and `readToolName` and always emits read-based SKILL.md guidance. I did not run a live model scenario because this was a read-only review and the code path is direct.

## Solution Assessment

Unclear until maintainers choose the trigger contract. A bare `Skill` tool-name check is narrow, but an explicit capability or alignment with `skill_view`/`skill_search` policy may be safer.
