## Summary

Keep open: current main still emits read-based Skills guidance and has no registered Skill-tool preference, while the related prompt PR remains open and does not address this contract.

## What This Changes

The issue asks OpenClaw to prefer a registered Skill-loader tool over read-based SKILL.md loading in the system prompt.

## Best Possible Solution

Define the Skill-loader contract, then update the skills formatter, system prompt builder, focused prompt tests, and system-prompt docs so a registered loader tool is preferred while the read fallback remains consistent.

## Reproduction Assessment

Yes for current behavior: source inspection shows the prompt construction always emits read-based skill loading and cannot branch on a registered Skill tool. This is an enhancement request rather than a broken documented bug, so no failing runtime test was run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. A normalized Skill tool-name branch is a small implementation, but the safer solution may be an explicit skill-loader capability and it must also remove the formatter-level read instruction conflict.
