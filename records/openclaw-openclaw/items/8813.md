## Summary

Keep open: current main still hardcodes read-based Skills loading, and the Skills section still cannot see whether a registered `Skill` loader tool is available. The report is concrete and supported by current source, but the final contract needs maintainer prompt/API judgment before implementation.

## What This Changes

Request conditional Skills prompt behavior that prefers invoking a registered `Skill` tool over reading `SKILL.md`, while preserving the read fallback when no such tool exists.

## Best Possible Solution

Define the Skill-loader contract, then update prompt construction, skill formatter text, focused prompt tests, and public system-prompt docs together while preserving the read fallback.

## Reproduction Assessment

Yes for the current prompt behavior: source inspection shows the Skills section always emits read-based loading and never receives `availableTools`. No runtime test was run because this was a read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. A normalized `skill` tool-name branch is narrow, but an explicit loader capability may be safer and the formatter/docs/tests need to stay aligned.
