## Summary

Keep open: current main and v2026.5.20 still emit read-based skill-loading guidance and do not prefer a registered Skill tool, while the related open prompt-policy PRs do not settle this exact contract.

## What This Changes

The issue requests changing the Skills system prompt so a registered Skill tool is preferred over reading SKILL.md files, with the existing read fallback preserved.

## Best Possible Solution

Define a durable skill-loading contract, pass runtime tool availability into the Skills prompt path, prefer the supported skill-loading tool only when present, preserve the read fallback, and update focused tests plus public docs.

## Reproduction Assessment

Yes, as a source-level gap: current main still emits read-based SKILL.md guidance and does not pass available-tool information into buildSkillsSection. I did not run a live model scenario because the read-only review can trace the prompt path directly.

## Solution Assessment

Unclear until maintainers choose the supported trigger contract. The proposed bare Skill-name check is narrow, but an explicit capability marker or the progressive skill-tool direction may be safer for long-term plugin compatibility.
