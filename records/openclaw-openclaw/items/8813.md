## Summary

Keep open: current main and the latest release still emit read-based Skills guidance, while the related skill-routing PRs are open product/contract work rather than a merged implementation of the requested Skill-tool preference.

## What This Changes

The issue asks OpenClaw to make the agent system prompt prefer a registered Skill-style tool over direct SKILL.md reads when building the Skills prompt section.

## Best Possible Solution

Define the supported skill-loading contract, then update src/agents/system-prompt.ts, focused prompt tests, and public system-prompt docs while preserving the read fallback when that contract is absent.

## Reproduction Assessment

Yes. Source inspection on current main shows availableTools is built from tool names, but buildSkillsSection does not receive it and still emits read-based SKILL.md guidance.

## Solution Assessment

Unclear. The requested Skill-name check is small and backward-compatible for the reporter, but maintainers still need to choose whether that name, local_skill_route, or another progressive-disclosure contract is the durable API.
