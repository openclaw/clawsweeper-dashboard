## Summary

Current main and v2026.6.5 still instruct agents to load applicable skills through the read tool, so the requested interoperability remains missing. The issue should stay open for a product decision because treating every tool named `Skill` as privileged would create an implicit plugin contract.

Likely related people: Vincent Koc (recent Skills-prompt contributor, medium confidence), Peter Steinberger (recent prompt/plugin-boundary contributor, medium confidence), and Kim (available-skills contract contributor, medium confidence).

## What This Changes

Update system-prompt generation to prefer a registered skill-loading tool when available while preserving direct SKILL.md reads as the fallback.

## Best Possible Solution

Add an explicit generic skill-loader capability to the plugin/tool contract, pass the resolved capability and tool casing into `buildSkillsSection`, prefer that tool when present, and retain the existing read path otherwise.

## Reproduction Assessment

Yes at source level: current main unconditionally generates read-based skill-loading guidance even when registered tool names are available to prompt construction. No runtime execution is needed to establish that prompt-generation result.

## Solution Assessment

No, the proposed tool-name check is not yet the best permanent solution because it creates an implicit semantic contract around a generic name. An explicit skill-loader capability is the narrower maintainable boundary.

## Risks / Open Questions

- A name-only `skill` check could grant special prompt semantics to an unrelated plugin tool that happens to use that generic name.
- Adding a semantic marker to plugin tool registration would create a public compatibility surface that needs deliberate naming, documentation, and contract tests.
- The claim that a tool response is inherently more authoritative than read-file content is not currently an OpenClaw-enforced contract, so prompt wording should not promise stronger authority without an explicit runtime policy.
