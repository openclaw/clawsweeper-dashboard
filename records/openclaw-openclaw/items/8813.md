## Summary

Keep open: current main and v2026.5.12 still hardcode read-based skill loading, while tool-name propagation and plugin tool registration make the requested behavior plausible. The remaining decision is the supported Skill-loading trigger contract, especially with adjacent open prompt-policy work.

## What This Changes

The issue asks OpenClaw to prefer a registered Skill tool for loading skill instructions instead of always instructing the model to read SKILL.md files.

## Best Possible Solution

Define a durable Skills-loading contract, pass runtime tool availability into the Skills prompt and formatter, prefer a registered Skill or skill_view-style tool only when that exact runtime tool is available, preserve the read fallback and exact-location guard, then update focused tests and public docs.

## Reproduction Assessment

Yes, as a source-level gap: current main normalizes runtime tool names but buildSkillsSection receives only skillsPrompt/readToolName and always emits read-based SKILL.md loading guidance. I did not run a live model scenario because this was a read-only review and the code path is direct.

## Solution Assessment

Unclear until maintainers choose the trigger contract. The narrow implementation path is in src/agents/system-prompt.ts and src/agents/skills/skill-contract.ts, but the proposed bare tool-name check may be less durable than an explicit capability or alignment with skill_view/skill_search policy.
