## Summary

Current main and the latest stable release still instruct agents to load applicable skills through the read tool, while registered tool names are not consulted by the Skills section. The requested interoperability remains useful, but assigning privileged semantics to every tool named `Skill` would create an implicit plugin contract, so maintainers should first choose an explicit generic skill-loader capability.

## What This Changes

Make the generated Skills prompt prefer an available compatible skill-loading tool while retaining direct SKILL.md reads as the fallback.

## Best Possible Solution

Define a generic explicit skill-loader prompt capability in the plugin/tool contract, pass it into `buildSkillsSection`, emit loader-specific guidance only when declared, and otherwise preserve the current read-based workflow.

## Reproduction Assessment

Yes at source level: current main and v2026.6.5 deterministically emit read-only skill-loading guidance, and focused tests assert that output. A live model run is not required to establish the missing capability branch.

## Solution Assessment

No, the proposed tool-name check is not the best durable solution because it assigns semantics based solely on spelling. An explicit generic capability declaration is narrower at the ownership boundary and avoids collisions while preserving backward behavior.
