## Summary

Keep open. Current main still emits read-based skill-loading guidance even when a registered `Skill` tool name can reach prompt construction, and the remaining fix needs a maintainer decision on the supported detection contract.

## What This Changes

Prefer a registered `Skill` tool over `read` when the system prompt tells the model how to load matching skills.

## Best Possible Solution

Add an approved Skill-tool signal through prompt construction and the skills formatters, emit non-contradictory Skill-tool guidance when present, preserve the read fallback, and update focused tests plus system-prompt docs.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: call `buildAgentSystemPrompt` with a `skillsPrompt` and `toolNames` containing `Skill`; current main computes `availableTools` but still emits read-based Skills guidance.

## Solution Assessment

Unclear until maintainers choose the supported detection contract. The narrow implementation belongs around `src/agents/system-prompt.ts` and the skills catalog formatters, but a capability marker may be safer than matching a tool name.
