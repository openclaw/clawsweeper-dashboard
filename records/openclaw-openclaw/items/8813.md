## Summary

Keep open: current main and the latest release still emit read-based Skills guidance, while the related skill-routing and prompt-relaxation PRs are open, conflicting, and not a merged implementation of the requested Skill-tool preference.

## What This Changes

The issue requests that the Skills prompt prefer invoking a registered Skill tool over directly reading SKILL.md when that tool is available.

## Best Possible Solution

Define the supported skill-loading contract, then update src/agents/system-prompt.ts, focused prompt tests, and public system-prompt docs while preserving the read fallback when that contract is absent.

## Reproduction Assessment

Yes, at source level. Current main builds availableTools from tool names, but buildSkillsSection does not receive it and still emits read-based SKILL.md guidance even when a Skill tool would be present.

## Solution Assessment

Unclear. The requested Skill-name check is small, but maintainers still need to choose whether that name, local_skill_route, or another capability marker is the durable API.
