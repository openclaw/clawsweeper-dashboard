## Summary

Keep open: current main and the latest release still emit read-based Skills guidance, while the open related PRs are unmerged and represent competing product directions rather than a shipped Skill-tool preference.

## What This Changes

The issue asks OpenClaw’s Skills system prompt to prefer an available registered Skill tool over read-based SKILL.md loading.

## Best Possible Solution

Define the supported skill-loading contract, then land or update the canonical focused PR so prompt code, tests, and public docs stay in sync while preserving the current read fallback when that contract is absent.

## Reproduction Assessment

Yes. Source inspection on current main shows `availableTools` is built from tool names, but the Skills prompt builder does not receive it and still emits read-based SKILL.md guidance.

## Solution Assessment

Unclear until maintainers choose the supported trigger contract. The requested Skill-name check is narrow, but the open `local_skill_route` and progressive `skill_view`/`skill_search` alternatives may be safer permanent API directions.
