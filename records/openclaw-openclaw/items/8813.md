## Summary

Keep open: current main and the latest release still emit read-based Skills guidance, and the related skill-routing PRs are open product/contract work rather than a merged implementation of the requested Skill-tool preference.

## What This Changes

The issue asks OpenClaw to prefer an available skill-loading tool over read-based SKILL.md guidance in the generated system prompt.

## Best Possible Solution

Define the supported skill-loading contract, then update src/agents/system-prompt.ts, focused prompt tests, and public system-prompt docs while preserving the read fallback when that contract is absent.

## Reproduction Assessment

Yes, at source level. Current main builds availableTools from tool names, but buildSkillsSection does not receive it and still emits read-based SKILL.md guidance even when a Skill tool would be present.

## Solution Assessment

Unclear. The requested Skill-name check is small, but maintainers still need to choose whether that name, local_skill_route, or another progressive-disclosure contract is the durable API.
