## Summary

Keep open: current main still emits read-based Skills guidance, while registered plugin tool names reach prompt construction and are not passed into the Skills section. The related prompt PR at https://github.com/openclaw/openclaw/pull/63940 is adjacent but does not implement the registered Skill-tool preference and remains open.

## What This Changes

Request that the Skills system prompt prefer invoking a registered Skill tool over reading SKILL.md when such a tool is available.

## Best Possible Solution

Define one Skills loading contract, pass available-tool information and caller casing into buildSkillsSection, prefer a registered Skill tool only under that contract, keep the exact-location read fallback, and update focused prompt tests plus system-prompt docs.

## Reproduction Assessment

Yes as a source-level gap check. Current main normalizes available tool names, but buildSkillsSection still receives only skillsPrompt/readToolName and always emits read-based SKILL.md loading guidance.

## Solution Assessment

Unclear until maintainers choose the Skill-tool trigger contract. The narrow code path is src/agents/system-prompt.ts, but a plain tool-name check may not be the safest durable policy.
