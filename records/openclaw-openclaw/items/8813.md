## Summary

Keep open: current main and the latest shipped release still emit read-based Skills guidance, while plugin tools can be registered and runtime tool names do reach prompt construction. The remaining decision is the supported Skill-loading contract, not whether the source-level gap exists.

## What This Changes

Request the Skills system prompt to prefer a registered Skill-loading tool over the read-based SKILL.md instruction when that tool is available.

## Best Possible Solution

Define a durable Skills-loading contract, pass runtime tool availability into the Skills prompt and formatter, prefer the supported Skill-loading tool only when present, preserve the read fallback, and update focused tests plus public docs.

## Reproduction Assessment

Yes, as a source-level gap: buildSkillsSection always emits read-based SKILL.md guidance and receives no availableTools even though tool names are normalized upstream. I did not run a live model scenario because this was a read-only review and the code path is direct.

## Solution Assessment

Unclear until maintainers choose the trigger contract. The proposed bare Skill-name check is narrow, but an explicit capability or the skill_view/skill_search direction may be safer for long-term plugin compatibility.
