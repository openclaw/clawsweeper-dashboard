## Summary

Keep open. Current main and the latest release still tell agents to load skills through `read`, while plugin tools are only available as names without a semantic skill-loader capability; the remaining work needs a maintainer product/API decision rather than conservative cleanup.

## What This Changes

Request that the Skills system prompt prefer invoking a registered Skill-style tool instead of always instructing read-based `SKILL.md` loading.

## Best Possible Solution

Define an explicit skill-loader capability in the plugin/tool contract, pass that resolved capability and caller tool casing into `buildSkillsSection`, prefer that loader when present, and keep the existing read fallback with focused tests and docs.

## Reproduction Assessment

Yes, at source level: current main unconditionally generates read-based skill-loading guidance even though runtime tool names are available to prompt construction. I did not run a live agent turn because the review is read-only and the source path is sufficient to establish the prompt output.

## Solution Assessment

No, the issue's proposed name-only tool check is not the best permanent solution because it would create an implicit semantic contract around a generic name. The safer design is an explicit plugin/tool capability for skill loading plus fallback read behavior.
