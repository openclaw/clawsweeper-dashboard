## Summary

Keep open: current main and the latest release still hardcode read-based skill loading, and the related open skill-routing and Codex/native contract items only partially overlap rather than resolving the registered Skill-tool preference.

## What This Changes

The issue requests that OpenClaw prefer a registered Skill loader tool in the Skills prompt instead of always instructing the model to read SKILL.md directly.

## Best Possible Solution

Define the supported loader-tool capability or naming contract, then update `src/agents/system-prompt.ts`, focused prompt tests, and system-prompt docs to prefer that loader when present while preserving the existing read fallback.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.10` unconditionally emit read-based skill-loading guidance even though registered tool names are available to prompt construction.

## Solution Assessment

Unclear as proposed. A registered loader-tool preference is plausible, but the best fix should first settle whether the contract is name-based, namespaced, or an explicit skill-loader capability.
