## Summary

Keep open: current main and v2026.5.20 still emit read-based skill-loading guidance and do not prefer a registered Skill tool, while related open prompt-policy PRs do not settle this exact contract.

## What This Changes

The issue requests that the Skills system prompt prefer invoking a registered Skill tool when available, falling back to read-based SKILL.md loading otherwise.

## Best Possible Solution

Define a durable skill-loading contract, thread runtime tool availability into the Skills prompt path, prefer the supported skill-loading tool only when present, preserve the read fallback, and update focused tests plus public docs.

## Reproduction Assessment

Yes, as a source-level gap: building a prompt with skillsPrompt and a registered Skill tool would still emit read-based SKILL.md guidance because buildSkillsSection cannot see availableTools. I did not run a live model scenario because this read-only review can trace the prompt path directly.

## Solution Assessment

Unclear until maintainers choose the supported trigger contract. The proposed Skill-name check is narrow, but an explicit capability marker or the progressive skill-tool direction may be safer for long-term plugin compatibility.
