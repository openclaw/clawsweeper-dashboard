## Summary

Keep open: current main and the latest release still emit read-based skill loading, while the related open PRs are adjacent prompt-policy work rather than a merged implementation of the requested Skill-tool preference.

## What This Changes

The issue requests that the Skills system prompt prefer invoking a registered `Skill` tool when available, while preserving the current `read` fallback otherwise.

## Best Possible Solution

Define the supported skill-loading contract, then update `src/agents/system-prompt.ts`, focused prompt tests, and the public system-prompt docs while preserving the current read fallback when that contract is absent.

## Reproduction Assessment

Yes. Source inspection on current main shows `availableTools` is built from tool names, but the Skills prompt builder does not receive it and still emits read-based SKILL.md guidance.

## Solution Assessment

Unclear until maintainers choose the supported trigger contract. The requested Skill-name check is narrow, but an explicit capability marker or the progressive `skill_view`/`skill_search` direction may be safer for plugin compatibility.
