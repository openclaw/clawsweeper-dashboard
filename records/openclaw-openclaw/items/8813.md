## Summary

Keep open: current main still emits read-based Skills guidance and has not implemented a registered Skill-tool preference; the remaining work is a prompt/plugin contract enhancement that needs maintainer direction.

## What This Changes

The issue asks OpenClaw to prefer invoking a registered Skill tool in the Skills system prompt instead of always instructing the model to read SKILL.md files.

## Best Possible Solution

Define the Skill-loader contract, then update src/agents/system-prompt.ts, focused prompt tests, and docs/concepts/system-prompt.md to prefer the registered loader while preserving the read fallback.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the reported current behavior: tool names are normalized into availableTools, but buildSkillsSection cannot see them and still emits read-based SKILL.md guidance.

## Solution Assessment

Unclear until maintainers choose the contract. A normalized tool-name branch is narrow, but an explicit skill-loader capability may be safer for plugin-tool collision semantics.
