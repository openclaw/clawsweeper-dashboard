## Summary

Keep open: current main and the latest release still hardcode read-based SKILL.md loading, while the related open PRs are adjacent or alternative skill-routing designs rather than a merged implementation of the requested Skill-tool preference.

## What This Changes

Request that the agent system prompt prefer invoking a registered Skill-style tool for skill loading instead of always instructing read-based SKILL.md loading.

## Best Possible Solution

Define the supported skill-loading contract first, then update the Skills prompt, focused system-prompt coverage, and public docs while preserving the current read fallback when that contract is absent.

## Reproduction Assessment

Yes. Source inspection on current main shows tool names are normalized into `availableTools`, but `buildSkillsSection` only receives `skillsPrompt` and `readToolName` and still emits read-based SKILL.md guidance.

## Solution Assessment

Unclear until maintainers choose the supported trigger contract. The requested Skill-name check is narrow, but an explicit capability marker, `local_skill_route`, or progressive `skill_view`/`skill_search` path may be safer for plugin compatibility.
