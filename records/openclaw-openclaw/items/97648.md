## Summary

Keep open for maintainer review: the patch appears correct and well scoped, with no blocking findings, but the new 16 MiB fail-closed boundary for Mistral-compatible response bodies is a maintainer compatibility choice before merge.

## What This Changes

This PR injects a bounded Mistral SDK HTTPClient fetcher, caps streamed response bodies at 16 MiB with createSseByteGuard, updates existing Mistral tests, and adds loopback plus synthetic bounded-stream tests.

## Best Possible Solution

Land the bounded fetcher after maintainers accept or tune the 16 MiB Mistral boundary, keeping it core-internal and covered by focused loopback and synthetic tests.

## Reproduction Assessment

Yes. Source inspection shows current main constructs the Mistral SDK without a bounded fetcher, while the SDK EventStream reads responseBody.getReader() directly and buffers chunks until SSE boundaries; the PR body provides after-fix loopback output for the capped path.

## Solution Assessment

Yes, with a compatibility caveat. Injecting the SDK-supported HTTPClient fetcher is the narrowest maintainable interception point for all Mistral SDK response bodies; a new SDK/plugin API or config knob would be broader unless maintainers reject the fixed threshold.
