## Summary

Keep this PR open: current main still has the iOS Contacts/Calendar first-use permission gap, and the branch is aimed at the right surface, but the latest head still has cancellation bugs in the permission prompt paths and lacks sufficient real iOS behavior proof before merge.

## What This Changes

The branch adds iOS Contacts/Calendar usage-description metadata, changes first-use permission requests for Contacts and Calendar, adds Calendar permission-box tests, and updates the changelog.

## Best Possible Solution

Keep the plist keys and least-privilege Calendar write path, but route both Contacts and Calendar prompts through a cancellation-safe permission request gate and add redacted real iOS prompt/invoke proof; coordinate the explicit settings UX with https://github.com/openclaw/openclaw/pull/51011.

## Reproduction Assessment

Yes, source-reproducible with high confidence. Current main returns permission-required errors for Contacts and Calendar `.notDetermined` states and lacks the Contacts/Calendar usage-description keys needed for first-use system prompts.

## Solution Assessment

No. The direction is right, but the current patch should make Contacts and Calendar permission prompting cancellation-safe and provide stronger real iOS after-fix proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Make Contacts permission waits cancellation-aware:** `apps/ios/Sources/Contacts/ContactsService.swift:105-109`
  - body: The new `.notDetermined` branch awaits `CNContactStore.requestAccess` through a bare continuation. If a node invoke is cancelled or times out while the system prompt is unattended, task cancellation will not resume this continuation, so the invoke work can remain suspended until the user responds; use a cancellation-aware permission gate before starting the prompt.
  - confidence: 0.9
- **[P2] Start Calendar prompts under the cancellation gate:** `apps/ios/Sources/Calendar/CalendarService.swift:183-184`
  - body: `awaitPermissionRequest` checks `box.isResolved` and then calls `start` as separate operations. Cancellation can fire after the guard but before `start`, returning `false` to the caller while EventKit still launches a permission prompt after timeout; make install/check/start atomic or tokened so cancelled invokes cannot start the OS prompt.
  - confidence: 0.87
