## Summary

Keep open: the iOS permission bug is still present on current main and this PR is useful, but the latest head still has cancellation-sensitive prompt paths and includes an out-of-scope Windows installer execution change.

## What This Changes

The PR adds iOS Contacts/Calendar usage descriptions, changes first-use Contacts/Calendar authorization to request permission, adds Calendar permission-box tests, and modifies Windows installer failure handling.

## Best Possible Solution

Land a scoped iOS-owned fix that adds the missing usage strings, preserves least-privilege Calendar write access, makes Contacts and Calendar prompt starts cancellation-safe, and leaves #51011 as the companion settings UX.

## Reproduction Assessment

Yes. Current main source gives a high-confidence reproduction path: fresh-install iOS node with `.notDetermined` Contacts or Calendar authorization, then invoke the affected Contacts/Calendar commands and observe the permission-required path instead of a prompt.

## Solution Assessment

No. The PR is directionally correct, but the prompt helpers still need cancellation-aware and atomic start behavior, and the installer change should be split out before this is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Make Contacts permission waits cancellation-aware:** `apps/ios/Sources/Contacts/ContactsService.swift:106-109`
  - body: This branch awaits `CNContactStore.requestAccess` through a bare continuation. `invokeWithTimeout` cancels timed-out invokes, but this wait does not observe cancellation, so locked or unattended first-use Contacts invokes can leave detached work suspended until the system prompt is answered.
  - confidence: 0.9
- **[P2] Start Calendar prompts through one atomic cancellation gate:** `apps/ios/Sources/Calendar/CalendarService.swift:183-184`
  - body: `awaitPermissionRequest` checks `box.isResolved` and then calls `start` as separate operations. If cancellation fires between those steps, the continuation has already returned `false` but the EventKit permission prompt can still launch after the caller timed out.
  - confidence: 0.84
