## Summary

Keep open: the PR addresses a real current-main iOS permissions gap, but the latest branch still introduces cancellation-sensitive permission waits and mixes an installer execution-path change into an iOS data-access fix.

## What This Changes

The PR adds iOS Contacts/Calendar privacy strings, changes Calendar and Contacts first-use authorization paths to request permissions, adds Calendar permission-box tests, and changes Windows installer failure handling.

## Best Possible Solution

Land a scoped iOS-owner fix that adds the missing usage-description keys, preserves least-privilege Calendar write access, and makes Contacts and Calendar first-use permission requests cancellation-safe. Keep #51011 as the companion settings UX and split the installer behavior change into its own reviewed PR.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence reproduction path: fresh-install iOS node with `.notDetermined` Contacts or Calendar authorization, then invoke `contacts.search` or `calendar.events` and observe the permission-required error path instead of a prompt.

## Solution Assessment

No. The PR is directionally useful, but the permission request helpers need cancellation-aware or atomic start behavior, and the unrelated installer execution-path change should be separated before this is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Make Contacts permission waits cancellation-aware:** `apps/ios/Sources/Contacts/ContactsService.swift:106-109`
  - body: This branch awaits `CNContactStore.requestAccess` with a bare continuation. `invokeWithTimeout` cancels the invoke task after timeout, but this continuation does not observe cancellation, so locked or unattended first-use contacts invokes can leave work suspended until the system prompt is answered.
  - confidence: 0.9
- **[P2] Gate Calendar prompt start atomically:** `apps/ios/Sources/Calendar/CalendarService.swift:183-184`
  - body: `awaitPermissionRequest` checks `isResolved` and then starts the EventKit prompt as separate operations. If cancellation fires after the guard but before `start`, a timed-out invoke can still launch a permission prompt after the caller has given up.
  - confidence: 0.84
