## Summary

Keep open: current main still has the iOS permission bug, and the PR is useful, but the latest head is not merge-ready because Contacts and Calendar prompt handling can still race invoke timeout cancellation and the branch also changes an unrelated Windows installer execution path.

## What This Changes

The PR adds iOS Contacts/Calendar usage-description keys, changes first-use authorization request behavior, adds Calendar permission-box tests, and modifies Windows installer failure handling.

## Best Possible Solution

Land a scoped iOS fix that adds the missing usage strings, preserves least-privilege Calendar write access, makes Contacts and Calendar permission prompt starts cancellation-safe, adds regression coverage, and leaves the Windows installer change to a separate PR.

## Reproduction Assessment

Yes, from source inspection. Current main routes the affected invoke commands into services that return permission-required errors for `.notDetermined` Contacts/Calendar status, and the plist lacks the usage-description keys needed for those permission flows.

## Solution Assessment

No. The PR is directionally right, but the narrow maintainable fix still needs cancellation-aware Contacts prompting, an atomic Calendar prompt-start gate, a required changelog entry, and removal of the unrelated installer change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Make Contacts permission waits cancellation-aware:** `apps/ios/Sources/Contacts/ContactsService.swift:106-109`
  - body: This `.notDetermined` path awaits `CNContactStore.requestAccess` through a bare continuation. `invokeWithTimeout` cancels timed-out invokes, so locked or unattended first-use Contacts prompts can leave the detached invoke task suspended until the OS callback arrives; wrap this in a cancellation-aware gate and skip launching the prompt once cancellation wins.
  - confidence: 0.9
- **[P2] Start Calendar prompts through an atomic cancellation gate:** `apps/ios/Sources/Calendar/CalendarService.swift:183-184`
  - body: `awaitPermissionRequest` checks `box.isResolved` and then calls `start` as separate steps. If cancellation fires after the guard but before `start`, the caller has already been resumed with `false` while the EventKit prompt can still launch after timeout; make install/start one atomic operation or return an install token that cannot race cancellation.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `apps/ios/Sources/Info.plist:53-54`
  - body: This is a user-facing iOS bug fix, but the branch does not update `CHANGELOG.md`. Repo policy requires user-facing fix/feat/perf PRs to add a single-line active-version changelog entry so release notes include the restored Contacts/Calendar permission behavior.
  - confidence: 0.84
