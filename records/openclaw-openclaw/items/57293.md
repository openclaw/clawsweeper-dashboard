## Summary

Keep open. The schema bug is real, but this PR should not merge as-is because it bundles stale systemd env-file behavior that conflicts with current main and the external-contributor real behavior proof gate is still unmet.

## What This Changes

The PR moves `approvalRunningNoticeMs` into the shared exec config schema, updates generated schema output, and adds systemd `EnvironmentFile=` handling for state-dir `.env` values.

## Best Possible Solution

Rebase or replace this with a config-only fix in the current schema surface, add focused global `ToolsSchema` regression coverage, and leave current main's curated systemd env-file behavior intact.

## Reproduction Assessment

Yes, at source level. `ToolsSchema.exec` is strict over `ToolExecBaseShape`, while the documented and runtime-read key only exists on `AgentToolExecSchema`, so global `tools.exec.approvalRunningNoticeMs` is rejected.

## Solution Assessment

No. Moving the field into the shared schema is the right fix, but bundling raw `.env` systemd handling is not the narrowest maintainable solution and conflicts with current main's curated env-file design.
