## What This Changes

The PR moves `approvalRunningNoticeMs` into the shared exec config shape and regenerated base schema while also adding systemd `EnvironmentFile=` handling for state-dir `.env` values.

## Best Possible Solution

Land or supersede with a config-only fix: move `approvalRunningNoticeMs` into `ToolExecBaseShape`, rely on current dynamic schema generation, add focused global `ToolsSchema` regression coverage, and leave current systemd env-file handling intact.

## Reproduction Assessment

Yes, from source inspection. `ToolsSchema` uses strict `ToolExecSchema` built from `ToolExecBaseShape`, while the documented key is only in `AgentToolExecSchema`, so global `tools.exec.approvalRunningNoticeMs` is rejected.

## Solution Assessment

No. Moving the field into `ToolExecBaseShape` is the right fix, but bundling raw `.env` systemd handling is not the narrowest maintainable solution and conflicts with current main's curated `gateway.systemd.env` design.

  - body: The branch points `EnvironmentFile=` at the raw state-dir `.env`. Systemd will import every assignment in that file, including values the service install path may intentionally keep out of startup; keep current main's curated `gateway.systemd.env` behavior instead.
