## Summary

Keep open: the underlying config bug still exists on current main, but this PR is not mergeable as-is because it bundles stale daemon environment-file behavior that current main already implements more safely.

## What This Changes

The PR moves `approvalRunningNoticeMs` into the shared exec config schema/generated schema and adds systemd `EnvironmentFile=` handling based on the state-dir `.env`.

## Best Possible Solution

Land a config-only fix: move `approvalRunningNoticeMs` into `ToolExecBaseShape`, regenerate schema metadata, add a focused global `ToolsSchema` regression test, and leave daemon env-file handling on current main.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: `ToolsSchema.parse({ exec: { approvalRunningNoticeMs: 3000 } })` reaches strict `ToolExecSchema` without that key, while docs and `ExecToolConfig` say the key is supported.

## Solution Assessment

No. The schema move is the right fix, but bundling raw `.env` systemd handling is not the narrowest maintainable solution and regresses the safer current-main curated `gateway.systemd.env` behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use a curated env file instead of raw .env:** `src/daemon/systemd.ts:460`
  - body: The branch points `EnvironmentFile=` at the user's state-dir `.env`. Systemd will load every assignment in that file, including keys the install planner deliberately filters from service startup such as `NODE_OPTIONS`; keep current main's curated `gateway.systemd.env` approach instead.
  - confidence: 0.9
- **[P2] Keep explicit service env overrides inline:** `src/daemon/systemd.ts:462-464`
  - body: Filtering inline values solely because a key exists in `.env` makes the file value win, since systemd applies `EnvironmentFile=` after `Environment=`. A stale token or port in `.env` can replace the fresh install-plan value at service start.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/config/zod-schema.agent-runtime.ts:428`
  - body: This is a user-facing config fix that closes a hard config rejection, but the branch does not update `CHANGELOG.md`. Add a single-line Unreleased Fixes entry with contributor credit following repo policy.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR introduces a daemon startup path that can load unfiltered raw `.env` values into a systemd service.

Concerns:

- **[medium] Raw .env is loaded by systemd:** `src/daemon/systemd.ts:460`
  - body: By setting `EnvironmentFile=` to the state-dir `.env`, the service imports every variable in that file and bypasses existing service-env filtering for startup-affecting keys. Use a generated, sanitized service env file instead.
  - confidence: 0.9
