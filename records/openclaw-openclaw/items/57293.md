## Summary

Keep open. The config bug is still valid on current main, but this PR is not ready because it bundles stale systemd EnvironmentFile behavior that current main has already superseded with a safer generated env-file flow.

## What This Changes

The PR moves `approvalRunningNoticeMs` into the shared exec tool config schema/generated base schema and adds systemd `EnvironmentFile=` handling for gateway service environment values.

## Best Possible Solution

Land a narrow config-only fix on current main: move `approvalRunningNoticeMs` into `ToolExecBaseShape`, regenerate the base schema, add a focused `ToolsSchema` regression test, and leave systemd env-file handling to the current curated implementation or a separate owner-reviewed change.

## Reproduction Assessment

Yes. Current main has a direct source-level reproduction: `ToolsSchema.parse({ exec: { approvalRunningNoticeMs: 3000 } })` reaches strict `ToolExecSchema` without that field in `ToolExecBaseShape`, while docs, types, and runtime code all reference the setting.

## Solution Assessment

No. The schema move itself is the right fix, but bundling the unrelated systemd service-env change is not the narrowest maintainable solution and would regress current-main daemon env filtering and override precedence.

- **[P1] Use a curated env file instead of raw .env:** `src/daemon/systemd.ts:460`
  - body: The branch points `EnvironmentFile=` at the user's state-dir `.env`. Systemd will load every assignment in that file, while current main deliberately filters service env keys and writes a generated `gateway.systemd.env`; a dangerous or unrelated `.env` key can therefore reach the gateway service.

  - confidence: 0.9

Status: needs_attention
