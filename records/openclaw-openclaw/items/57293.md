## Summary

Keep open. The config-schema fix is still needed on current main, but this PR is not safe to merge as-is because it bundles the valid schema move with stale daemon `EnvironmentFile=` changes that current main has already solved more safely.

## What This Changes

The PR moves `approvalRunningNoticeMs` into the shared exec config schema/generated schema and adds systemd `EnvironmentFile=` handling for gateway service environment values.

## Best Possible Solution

Land a config-only replacement: move `approvalRunningNoticeMs` into `ToolExecBaseShape`, regenerate schema metadata, add a focused global `ToolsSchema` regression test, add the changelog entry, and leave current main's daemon env-file handling unchanged.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: `ToolsSchema.parse({ exec: { approvalRunningNoticeMs: 3000 } })` reaches strict `ToolExecSchema` built from `ToolExecBaseShape`, while docs and `ExecToolConfig` say the key is supported.

## Solution Assessment

No. The schema move is the right fix, but bundling raw `.env` systemd handling is not the narrowest maintainable solution and is less safe than current main's curated `gateway.systemd.env` path.
