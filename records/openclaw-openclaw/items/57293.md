## Summary

Keep open. The core schema fix is still needed on current main and the latest release, but this PR cannot merge as-is because it also carries stale raw-systemd `EnvironmentFile=` changes that current main has already superseded with a curated env-file implementation, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The branch moves `approvalRunningNoticeMs` into the shared exec Zod schema/generated schema and adds systemd `EnvironmentFile=` handling that filters raw `.env` keys out of inline service environment entries.

## Best Possible Solution

Land or replace this with a config-only fix that adds the key to `ToolExecBaseShape`, adds focused global `ToolsSchema` regression coverage, updates current generated/config surfaces that still exist, and leaves the curated systemd env-file implementation intact.

## Reproduction Assessment

Yes, at source level. Current main documents and reads `tools.exec.approvalRunningNoticeMs`, but strict global `ToolExecSchema` is built from a base shape that omits the key.

## Solution Assessment

No. The shared-schema move is the right fix, but bundling stale systemd secret-handling changes is not the narrowest maintainable solution for this config bug.
