## Summary

Keep open: current main and the latest release still copy the full host environment whenever the channel ingress queue opens the state DB with a custom state directory, and Telegram isolated ingress calls that queue repeatedly during drain.

## What This Changes

The issue requests avoiding repeated full `process.env` copies when Telegram isolated ingress queue operations open the state database with a custom state directory.

## Best Possible Solution

Replace the hot-path env spread with a non-enumerating env overlay or direct state DB path while preserving inherited env lookups, then add focused regression coverage for large-env custom-state queue operations.

## Reproduction Assessment

Yes: create a channel ingress queue with `stateDir` set under a very large `process.env` and repeatedly call `recoverStaleClaims`, `listClaims`, or `listPending`; current source guarantees each open enumerates the full env. I did not run that benchmark in this read-only pass.

## Solution Assessment

Yes: fixing the queue DB opener is the narrowest maintainable layer because all Telegram spool operations reach the same helper. The implementation should avoid adding config or changing the plugin runtime API.
