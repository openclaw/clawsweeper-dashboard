## Summary

Keep open: current main and latest release still enumerate the full environment in the channel ingress queue opener, and open linked fix PRs now own the exact repair path.

## What This Changes

The issue reports that Telegram isolated polling repeatedly opens the ingress queue state DB with a custom state directory and copies large `process.env`, causing gateway event-loop delay and collateral channel reply loss.

## Best Possible Solution

Land one focused linked fix that removes full `process.env` enumeration from the ingress queue DB opener, then close this issue after current main contains that behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to create a channel ingress queue with `stateDir` set under a very large `process.env` and repeatedly call `recoverStaleClaims`, `listClaims`, or `listPending`; I did not execute the benchmark in this read-only pass.

## Solution Assessment

Yes. Fixing the shared queue DB opener is the narrowest maintainable solution because Telegram spool operations and sibling channel ingress callers use that helper, and the linked PRs target exactly that layer.
