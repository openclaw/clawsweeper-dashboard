## Summary

Keep open: current main and latest release v2026.6.9 still copy `process.env` in the ingress queue opener, while https://github.com/openclaw/openclaw/pull/95278 and https://github.com/openclaw/openclaw/pull/95271 are open candidate fixes. This issue should close only after one focused fix lands.

## What This Changes

The issue reports that Telegram isolated polling ingress repeatedly copies a large `process.env` when opening custom-state channel ingress queues.

## Best Possible Solution

Land one focused env-overlay fix for the shared channel ingress queue opener, preferably the active replacement lane if checks stay acceptable, then close this issue from the merged PR.

## Reproduction Assessment

Yes, source-reproducible: create a channel ingress queue with `stateDir` under a very large `process.env` and call `recoverStaleClaims`, `listClaims`, or `listPending`; current main guarantees each call opens the DB through an env spread. I did not rerun the Kubernetes benchmark in this read-only pass.

## Solution Assessment

Yes. Fixing the shared ingress queue opener is the narrowest maintainable layer because Telegram isolated polling reaches the same helper for all affected queue operations, and the state DB contract only needs env property lookup semantics.
