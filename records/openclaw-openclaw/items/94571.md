## Summary

Keep open: current main and the latest release still copy the full environment in the shared channel ingress queue opener, and open closing PRs now own the fix path, so this issue should remain open until one lands.

## What This Changes

The issue reports that Telegram isolated ingress repeatedly copies a large `process.env` when opening the channel ingress queue with `stateDir`, causing gateway event-loop saturation and delayed replies.

## Best Possible Solution

Merge one focused PR that replaces the env spread in the shared channel ingress queue opener with a non-enumerating env overlay or direct state-dir path plus regression coverage, then let this issue close from the merged fix.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to create a channel ingress queue with `stateDir` under a very large `process.env` and repeatedly call `recoverStaleClaims`, `listClaims`, or `listPending`; current main guarantees each open enumerates the full env.

## Solution Assessment

Yes. Fixing the shared queue DB opener is the narrowest maintainable solution because Telegram’s drain path reaches the same helper for all affected queue operations; the remaining decision is which open PR should land.
