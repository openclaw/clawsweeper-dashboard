## Summary

Keep open: current main and the latest release still contain the unconditional `ssh` classification, and the issue is already paired with an open fixing PR that uses closing syntax, so it should remain open until that PR lands or is replaced.

## What This Changes

The issue asks for SSH listener classification to only return `ssh` for `-L`/`-R` tunnels bound to the queried port, avoiding false SSH-tunnel hints for ssh-named non-tunnel processes.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/91143 so the shared classifier returns `ssh` only for matching SSH forwards and has regression coverage for ssh-named non-tunnels.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction: `classifyPortListener` returns `ssh` for any normalized command containing `ssh` even when the port-aware regex fails, and `buildPortHints` then emits the SSH-tunnel hint. I did not run tests or a live CLI command because this review is read-only.

## Solution Assessment

Yes, the best fix is in the shared `classifyPortListener` helper with focused regression tests, because both diagnostics and restart-health consume that classification. A caller-only workaround would leave the same false kind available to sibling surfaces.
