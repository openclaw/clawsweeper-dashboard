## Summary

Keep open. Current main still leaves the reported stale versioned runtime-deps cleanup gap: the shared prune helper only considers `openclaw-unknown-*`, Gateway startup and `openclaw plugins deps --prune` both use that same helper, and the focused test still asserts that a stale `openclaw-2026.4.25-*` root is preserved. Recent update/restart and runtime-deps hardening reduces adjacent failure modes, but it does not implement cleanup or ignore handling for stale OpenClaw-owned versioned roots.

## What This Changes

The issue reports that stale versioned `plugin-runtime-deps` roots left after `openclaw update` can crash-loop the Gateway and asks for automatic cleanup plus a fresh restart path.

## Best Possible Solution

Keep this open until runtime-deps owners implement one shared policy that either prunes stale OpenClaw-owned versioned roots or ignores incompatible old roots across Gateway startup, update recovery, doctor/repair, and future cache-prune UX, with regression coverage for stale `openclaw-<version>-<hash>` siblings beside the active install and preserved behavior for locks, `openclaw-unknown-*` roots, and `OPENCLAW_PLUGIN_STAGE_DIR` path lists.

## Reproduction Assessment

Yes for the current-main cleanup gap: source inspection and the existing unit test show that stale versioned roots survive the shared prune path. The exact launchd crash-loop was not replayed in this read-only review.

## Solution Assessment

Unclear whether the issue body's update-only deletion sketch is the best final fix. A safer solution is an owner-reviewed shared cleanup or ignore policy that covers startup, update, doctor/repair, locks, and path-list staging instead of adding a narrow one-off deletion step.
