## Summary

Keep open: the PR targets a real current and released QMD/mcporter regression, but as written it can make mcporter-launched `qmd mcp` use the default QMD cache/index, and the supplied proof only verifies source text rather than the real memory-search path.

## What This Changes

The PR adds a `mcporterEnv` in `QmdMemoryManager` that removes `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` from mcporter spawns and updates the adjacent test expectation.

## Best Possible Solution

Separate mcporter config discovery from QMD server runtime env: mcporter should avoid QMD-scoped XDG lookup, while the actual `qmd mcp` server still receives per-agent QMD config/cache env or an explicit generated config that carries it.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.8` pass QMD-scoped XDG env into mcporter, while mcporter 0.10.2 uses `XDG_CONFIG_HOME` for config discovery; I did not run the live gateway plus mcporter scenario in this read-only review.

## Solution Assessment

No. The patch fixes mcporter config lookup too broadly by dropping QMD cache/config env from the mcporter process without another path that preserves the per-agent `qmd mcp` runtime env.
