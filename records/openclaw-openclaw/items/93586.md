## Summary

Keep open: the PR targets a real QMD/mcporter regression, but the patch is not merge-ready because removing XDG from the mcporter process also removes the inherited agent-scoped QMD state from bare `qmd mcp` stdio servers launched by mcporter. The supplied proof is source-grep/unit/CI evidence and explicitly does not show a live OpenClaw memory search through mcporter 0.10+ and QMD.

## What This Changes

The branch adds a `mcporterEnv` without `XDG_CONFIG_HOME` or `XDG_CACHE_HOME`, uses it for `QmdMemoryManager.runMcporter`, and updates the adjacent test to expect stripped XDG vars while preserving `QMD_CONFIG_DIR`.

## Best Possible Solution

Keep mcporter config discovery out of OpenClaw's QMD XDG tree while preserving per-agent QMD config/cache state for qmd stdio children, then land either a repaired narrow PR or the broader sidecar-isolation PR with live proof.

## Reproduction Assessment

Yes, at source level: current main and the latest release pass QMD-scoped XDG vars into mcporter, mcporter uses XDG for config discovery, and the linked issue includes the failing `Unknown MCP server 'qmd'` path. I did not run a live mcporter 0.10 gateway reproduction in this read-only review.

## Solution Assessment

No. The patch fixes mcporter config discovery, but it is not the best fix until the qmd stdio child still gets per-agent QMD XDG cache/config state, either through a repaired narrow env path or the broader sidecar-isolation approach.
