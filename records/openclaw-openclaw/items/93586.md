## Label Justifications

- `P1`: The PR targets an active QMD memory-search regression for mcporter 0.10+ users, and the current patch can still break that workflow.
- `merge-risk: 🚨 compatibility`: The diff changes inherited process env for existing mcporter-backed QMD setups and can alter config/cache paths after upgrade.

## Summary

Keep open: the PR targets a real shipped QMD/mcporter regression, but the current patch is not merge-ready because it strips the QMD XDG env from the mcporter process without preserving that env for local `qmd mcp` children, and the supplied proof is source-grep output rather than a real mcporter-backed memory search.

## What This Changes

The PR adds a `mcporterEnv` for `QmdMemoryManager` that removes `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` from mcporter spawns, and updates the adjacent test expectation.

## Best Possible Solution

Separate mcporter config discovery from QMD server runtime env: mcporter discovery should avoid QMD-scoped XDG lookup, while the actual local `qmd mcp` child still receives per-agent QMD config/cache env or an explicit generated config carrying it.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.8` pass QMD-scoped XDG env into mcporter, and upstream mcporter uses `XDG_CONFIG_HOME` for config discovery; I did not run the live gateway plus mcporter scenario in this read-only review.

## Solution Assessment

No. The PR fixes mcporter config lookup too broadly by dropping QMD cache/config env from the mcporter process without another path that preserves the per-agent `qmd mcp` runtime env.
