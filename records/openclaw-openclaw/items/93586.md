## Summary

Keep open: the linked QMD/mcporter regression is real and still present on current main and the latest release, but this narrow patch is not merge-ready because it can make mcporter-launched local qmd servers lose OpenClaw's per-agent QMD env and still lacks real post-fix behavior proof.

## What This Changes

The PR adds a mcporter-specific environment that removes XDG_CONFIG_HOME and XDG_CACHE_HOME from mcporter spawns and updates the QMD manager test to expect those variables absent.

## Best Possible Solution

Keep mcporter config discovery out of OpenClaw's QMD XDG tree while preserving or injecting per-agent QMD env for qmd stdio children, then choose whether that lands as a narrow repair here or through the broader generated-config PR.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main and v2026.6.8 pass QMD-scoped XDG env into mcporter, while mcporter v0.10.2 uses XDG_CONFIG_HOME for config discovery. I did not run the live OpenClaw plus mcporter plus QMD memory-search scenario in this read-only review.

## Solution Assessment

No. Stripping XDG from the whole mcporter process is only a partial fix because mcporter stdio servers inherit the mcporter process env, so the actual qmd child can lose the agent-scoped QMD state that current behavior intentionally preserved.
