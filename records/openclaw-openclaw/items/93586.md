## Summary

Keep open: the linked QMD/mcporter bug is still valid, but this PR is not merge-ready because it fixes mcporter config discovery by removing the same QMD XDG env that local `qmd mcp` children need for the per-agent index.

## What This Changes

The PR adds a separate `mcporterEnv` in `QmdMemoryManager` that strips `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` from mcporter spawns, with a focused test expectation update.

## Best Possible Solution

Separate mcporter config discovery from QMD server runtime env: mcporter must avoid QMD-scoped config lookup, while the actual `qmd mcp` server still receives per-agent QMD config/cache env or an explicit generated config that carries it.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main and v2026.6.8 pass QMD-scoped XDG env into mcporter, while mcporter 0.10.2 uses `XDG_CONFIG_HOME` for config discovery. I did not run the live gateway plus mcporter scenario in this read-only review.

## Solution Assessment

No, this patch is too broad as written. The safer fix must avoid redirecting mcporter config discovery while preserving or explicitly injecting the per-agent QMD env into the `qmd mcp` server that mcporter launches.
