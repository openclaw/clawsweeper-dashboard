## Summary

Keep open, not merge-ready: the patch fixes mcporter config discovery in one layer but drops the agent-scoped QMD environment from the mcporter process, and mcporter stdio servers inherit that process env unless their server entry declares its own env. The supplied proof is source-grep/unit-test evidence, not a real OpenClaw memory search through mcporter 0.10+ and QMD.

## What This Changes

The PR adds a separate `mcporterEnv` that strips `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` from `QmdMemoryManager` mcporter spawns and updates the adjacent test to expect those vars absent.

## Best Possible Solution

Separate mcporter config discovery from QMD server execution: mcporter should not see OpenClaw's QMD XDG config for its own config lookup, but the actual `qmd mcp` server entry must receive agent-scoped `QMD_CONFIG_DIR` and QMD XDG paths.

## Reproduction Assessment

Yes, at source level: current main builds per-agent QMD XDG env and passes it unchanged to mcporter while mcporter 0.10 uses `XDG_CONFIG_HOME` for config discovery. I did not run a live mcporter gateway reproduction in this read-only review.

## Solution Assessment

No. Stripping XDG only from the mcporter process is too narrow because the downstream `qmd mcp` server can inherit that stripped env; the fix needs separate mcporter discovery env and agent-scoped QMD server env.
