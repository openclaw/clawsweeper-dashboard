## Summary

Keep open: this is a focused fix for a valid QMD/mcporter regression, but the submitted env split is not merge-ready because it drops supported user XDG mcporter discovery and only supplies code/mock proof.

## What This Changes

The PR adds a separate `mcporterEnv` for QMD memory manager mcporter spawns and updates the qmd-manager test to expect no `XDG_CONFIG_HOME` or `XDG_CACHE_HOME` in mcporter’s child environment.

## Best Possible Solution

Build the mcporter spawn environment from the parent process env, preserve user mcporter discovery inputs, and keep QMD-specific config/cache env paired only where the QMD child actually needs it.

## Reproduction Assessment

Yes, source-reproducible: current main passes QMD-scoped XDG env into mcporter, and mcporter currently honors `XDG_CONFIG_HOME` for config discovery. I did not run a live gateway plus mcporter/qmd scenario in this read-only review.

## Solution Assessment

No. Splitting mcporter from QMD-scoped XDG is the right area, but this implementation should restore parent mcporter env values and avoid a half-paired QMD child env.
