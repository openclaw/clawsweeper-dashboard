## Summary

Keep open: the PR targets a real shipped QMD/mcporter regression, but the current patch is not merge-ready because it strips the QMD-scoped env from the mcporter process without preserving that env for local mcporter-launched `qmd mcp` children, and the supplied proof is only source-grep output rather than a real memory-search run.

## What This Changes

The PR adds a separate `mcporterEnv` in `QmdMemoryManager` that strips `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` from mcporter spawns and updates the adjacent test expectation.

## Best Possible Solution

Separate mcporter config discovery from QMD server runtime env: keep mcporter from resolving config under QMD-scoped XDG state while still passing per-agent QMD config/cache env to the actual `qmd mcp` child through a generated server env or equivalent explicit contract.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.8` pass QMD-scoped XDG env into mcporter, and upstream mcporter uses `XDG_CONFIG_HOME` for config discovery; I did not run the live gateway plus mcporter scenario in this read-only review.

## Solution Assessment

No. This is a plausible narrow fix for mcporter config lookup, but it removes the inherited QMD runtime env that existing local stdio `qmd mcp` server entries may rely on; the better fix preserves both config discovery and per-agent QMD runtime state.
