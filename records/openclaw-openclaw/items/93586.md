## Summary

Keep open: this targets a real QMD/mcporter regression, but the current patch is not merge-ready because it cleans mcporter config discovery by stripping XDG from the mcporter process while leaving mcporter-launched `qmd mcp` children without the per-agent XDG cache/config environment they need.

## What This Changes

The branch adds a mcporter-specific spawn environment in `QmdMemoryManager` that strips `XDG_CONFIG_HOME` and `XDG_CACHE_HOME`, then updates the adjacent test to expect those variables absent for mcporter spawns.

## Best Possible Solution

Keep mcporter config discovery out of OpenClaw's QMD XDG tree while preserving per-agent QMD config/cache state for qmd stdio children, then land either a repaired narrow PR or the broader isolation PR with live proof.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main passes QMD-scoped XDG env into mcporter, mcporter reads XDG config paths for discovery, and mcporter stdio servers inherit the mcporter process env unless explicitly overridden.

## Solution Assessment

No. The patch fixes mcporter config discovery but strips XDG state from the process that launches qmd stdio servers, so the safer fix must keep discovery clean while preserving or injecting per-agent QMD env at the qmd boundary.
