## Summary

Keep open: this PR targets a real QMD/mcporter environment bug, but the current patch strips QMD-scoped XDG variables from mcporter without preserving them for local `qmd mcp` stdio children, and the supplied proof is source/unit-test-only rather than a real OpenClaw memory search.

## What This Changes

The PR adds a `mcporterEnv` for QMD mcporter spawns that removes `XDG_CONFIG_HOME` and `XDG_CACHE_HOME`, then updates the adjacent test to expect those variables to be absent.

## Best Possible Solution

Separate mcporter config discovery from qmd server runtime env: keep mcporter from resolving config under QMD-scoped XDG state while ensuring local `qmd mcp` children still receive per-agent QMD config/cache env through generated server env or an equivalent explicit contract.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.8` pass QMD-scoped XDG env into mcporter, and mcporter 0.10.2 uses `XDG_CONFIG_HOME` for config candidates; I did not run the live gateway plus mcporter scenario in this read-only review.

## Solution Assessment

No. This is a plausible narrow fix for mcporter config lookup, but upstream mcporter starts stdio servers from its own `process.env` plus server-specific overrides, so the patch can remove the per-agent QMD runtime env that existing `qmd mcp` entries rely on.
