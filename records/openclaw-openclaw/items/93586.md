## Summary

Keep open: this PR targets a real QMD/mcporter regression, but the current patch is still not merge-ready because it strips the mcporter process env that mcporter uses as the base env for stdio QMD children, and the supplied proof still does not exercise a real OpenClaw memory search with mcporter 0.10+ and QMD.

## What This Changes

The PR adds a `mcporterEnv` in `QmdMemoryManager` that removes `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` from mcporter spawns and updates the adjacent test to expect those values absent.

## Best Possible Solution

Keep mcporter config discovery out of OpenClaw's QMD XDG tree while preserving per-agent QMD config/cache state for qmd stdio children, then land either a repaired narrow PR or the broader isolation PR with live proof.

## Reproduction Assessment

Yes at source level. Current main and v2026.6.8 pass QMD-scoped XDG vars into mcporter, mcporter 0.10.2 uses `XDG_CONFIG_HOME` for config discovery, and the linked issue includes the failing `Unknown MCP server 'qmd'` path.

## Solution Assessment

No. The patch fixes mcporter config discovery but is not the best fix until the qmd stdio child still gets per-agent QMD XDG cache/config state, either through a repaired narrow env path or the broader sidecar-isolation approach.
