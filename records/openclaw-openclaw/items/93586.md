## Summary

Keep open: the PR targets a real QMD/mcporter regression, but the patch is not merge-ready because it can make mcporter-launched local `qmd mcp` servers lose OpenClaw's per-agent QMD state, and the contributor proof still does not show a real mcporter 0.10+ QMD memory search after the change.

## What This Changes

The PR adds a mcporter-specific spawn environment that strips `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` from `QmdMemoryManager` mcporter calls and updates the adjacent test expectation.

## Best Possible Solution

Keep mcporter config discovery out of OpenClaw's QMD XDG tree while preserving per-agent QMD env for qmd stdio children, then land either a repaired narrow PR or the broader isolation PR with live proof.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main passes QMD-scoped XDG env into mcporter, mcporter v0.10.2 reads `XDG_CONFIG_HOME` for config discovery, and its stdio transport inherits `process.env` for child servers.

## Solution Assessment

No. The patch fixes mcporter config discovery but strips the process env mcporter uses to spawn stdio qmd servers, so a safer fix must keep discovery clean while preserving or injecting per-agent QMD env at the qmd server boundary.
