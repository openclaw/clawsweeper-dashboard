## Summary

Keep open: the PR targets a real QMD/mcporter regression, but the current patch is not merge-ready because it strips QMD-scoped XDG values from the whole mcporter process and can leave mcporter-launched `qmd mcp` children without OpenClaw's per-agent QMD config/cache state.

## What This Changes

The PR adds a `mcporterEnv` that removes `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` from `QmdMemoryManager` mcporter spawns and updates the adjacent test to expect those values absent.

## Best Possible Solution

Keep mcporter config discovery out of OpenClaw's QMD XDG tree while preserving per-agent QMD config/cache state for qmd stdio children, then land either a repaired narrow PR or the broader isolation PR with live proof.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 pass QMD-scoped XDG vars into mcporter, mcporter v0.10.2 uses `XDG_CONFIG_HOME` for config discovery, and the linked issue includes the failing `Unknown MCP server 'qmd'` path.

## Solution Assessment

No. The patch fixes mcporter config discovery, but it is not the best fix until the qmd stdio child still gets per-agent QMD XDG cache/config state, either through a repaired narrow env path or the broader sidecar-isolation approach.
