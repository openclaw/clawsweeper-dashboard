## Summary

Keep open: the linked QMD/mcporter bug is still present on current main, but this PR is not merge-ready because it drops user-supported mcporter XDG discovery and leaves QMD config/cache env split for mcporter-spawned qmd children.

## What This Changes

The PR adds a separate `mcporterEnv` for QMD memory manager mcporter spawns that strips `XDG_CONFIG_HOME` and `XDG_CACHE_HOME`, and updates the qmd-manager env assertion test.

## Best Possible Solution

Build the mcporter process env from the parent env, preserve user mcporter discovery inputs, and keep QMD-specific config/cache env paired only in the qmd child or generated mcporter server config.

## Reproduction Assessment

Yes, source-reproducible: current main passes QMD-scoped XDG env into mcporter, mcporter v0.10.2 uses `XDG_CONFIG_HOME` for config discovery, and QMD reads config/cache from separate env keys. I did not run a live gateway plus mcporter/qmd scenario in this read-only review.

## Solution Assessment

No. Splitting mcporter from QMD-scoped XDG is the right surface, but this implementation should preserve parent mcporter discovery env and avoid a half-paired QMD child env.
