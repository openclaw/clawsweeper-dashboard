## Summary

Keep open: the PR addresses a real QMD/mcporter regression, but the current patch strips QMD-scoped XDG env from the mcporter process without preserving it for mcporter-launched stdio `qmd mcp` children, and the supplied proof is source-grep output rather than a real memory-search run.

## What This Changes

The PR adds a `mcporterEnv` in `QmdMemoryManager` that removes `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` from mcporter spawns and updates the adjacent qmd-manager test expectation.

## Best Possible Solution

Separate mcporter config discovery from QMD server runtime env: mcporter discovery should avoid QMD-scoped XDG lookup, while the actual local `qmd mcp` child still receives per-agent QMD config/cache env or an explicit generated config that carries it.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.8 pass QMD-scoped XDG env into mcporter, and mcporter v0.10.2 uses `XDG_CONFIG_HOME` for config discovery; I did not run the live gateway plus mcporter scenario in this read-only review.

## Solution Assessment

No. The PR fixes mcporter config lookup too broadly by dropping QMD cache/config env from the mcporter process without another path that preserves the per-agent `qmd mcp` runtime env.
