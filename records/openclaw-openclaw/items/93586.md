## Summary

Keep open: the PR targets a real QMD/mcporter regression, but the current env split can break mcporter-launched local `qmd mcp` children by dropping the per-agent QMD cache/config env, and the supplied proof still does not show a real OpenClaw memory search with mcporter 0.10+.

## What This Changes

The PR adds a `mcporterEnv` in `QmdMemoryManager` that strips `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` from mcporter spawns and updates the adjacent qmd-manager test expectation.

## Best Possible Solution

Separate mcporter config discovery from QMD server runtime env: mcporter discovery should avoid QMD-scoped XDG lookup, while the actual `qmd mcp` child still receives per-agent QMD config/cache env or an explicit generated config that carries it.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main and `v2026.6.8` pass QMD-scoped XDG env into mcporter, and mcporter 0.10.2 uses `XDG_CONFIG_HOME` for config discovery. I did not run the live gateway plus mcporter scenario in this read-only review.

## Solution Assessment

No. The PR fixes mcporter config lookup too broadly by dropping QMD cache/config env from the mcporter process without another path that preserves the per-agent `qmd mcp` runtime env.
