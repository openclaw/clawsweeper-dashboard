## Summary

Keep open: the PR targets a real QMD/mcporter regression, but the current env split is still not merge-ready because it can make mcporter-launched `qmd mcp` children lose the per-agent QMD XDG config/cache state, and the supplied proof is source-grep/unit-test evidence rather than a real memory search through mcporter 0.10+ and QMD.

## What This Changes

The PR adds a stripped `mcporterEnv` for `QmdMemoryManager` mcporter spawns and updates the adjacent test to expect `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` absent while `QMD_CONFIG_DIR` remains.

## Best Possible Solution

Keep mcporter config discovery out of OpenClaw's QMD XDG tree while preserving per-agent QMD config/cache state for qmd stdio children, then land either a repaired narrow PR or the broader isolation PR with live proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 pass QMD-scoped XDG vars into mcporter, mcporter v0.10.2 uses XDG for config discovery, and the linked issue includes the failing `Unknown MCP server 'qmd'` path. I did not run a live mcporter 0.10 gateway reproduction in this read-only review.

## Solution Assessment

No. The patch fixes mcporter config discovery, but it is not the best fix until the qmd stdio child still gets per-agent QMD XDG cache/config state, either through a repaired narrow env path or the broader sidecar-isolation approach.
