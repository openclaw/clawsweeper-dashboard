## Summary

Keep open: the PR is paired with an open issue and targets a real QMD/mcporter regression, but the current patch still has blocking env-compatibility defects and lacks real after-fix behavior proof.

## What This Changes

The PR adds a separate mcporter spawn environment in QmdMemoryManager that strips XDG_CONFIG_HOME and XDG_CACHE_HOME, plus a test expectation for mcporter env isolation.

## Best Possible Solution

Use a target-specific env split that preserves user mcporter XDG discovery, prevents QMD-scoped XDG from shadowing mcporter config, and passes a paired QMD config/cache env only to the qmd child path.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.8 pass QMD-scoped XDG env into mcporter, and mcporter v0.10.2 uses XDG_CONFIG_HOME for config discovery. I did not execute a live gateway plus mcporter/qmd scenario in this read-only review.

## Solution Assessment

No. Splitting mcporter from QMD-scoped XDG is the right surface, but this implementation should preserve user mcporter discovery env and avoid passing an unpaired QMD_CONFIG_DIR to a qmd child.
