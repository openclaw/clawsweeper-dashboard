## Summary

Keep open: the linked QMD/mcporter bug is real, but this PR is not merge-ready because stripping XDG from the whole mcporter process can make mcporter-launched local `qmd mcp` servers lose OpenClaw's per-agent QMD state. The submitted proof is source-grep/unit-test output rather than a real OpenClaw memory search with mcporter 0.10+ and QMD.

## What This Changes

The branch adds a mcporter-specific spawn environment that removes `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` from QMD manager mcporter calls and updates the qmd-manager test expectation.

## Best Possible Solution

Keep mcporter config discovery out of OpenClaw's QMD XDG tree while preserving per-agent QMD env for qmd stdio children, then land either a repaired narrow PR or the broader generated-config PR after live proof.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main and `v2026.6.8` pass QMD-scoped XDG env into mcporter, while mcporter v0.10.2 uses `XDG_CONFIG_HOME` for config discovery; I did not run the live OpenClaw plus mcporter plus QMD memory-search scenario.

## Solution Assessment

No. Stripping XDG from the whole mcporter process is only a partial fix because mcporter stdio servers inherit the mcporter process env, so the actual qmd child can lose the agent-scoped QMD state that current behavior intentionally preserved.
