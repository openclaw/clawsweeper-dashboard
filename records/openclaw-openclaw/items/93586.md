## Summary

Keep open: this targets a real QMD/mcporter regression, but the narrow env split is not merge-ready because it can remove env needed by local `qmd mcp` servers, can miss supported user XDG mcporter configs, and still lacks real after-fix behavior proof.

## What This Changes

The PR adds a separate `mcporterEnv` without `XDG_CONFIG_HOME` or `XDG_CACHE_HOME` for QMD manager mcporter spawns and updates the adjacent spawn-env unit test.

## Best Possible Solution

Separate mcporter config discovery from QMD server execution: mcporter should avoid OpenClaw's QMD-scoped XDG config path while local/generated `qmd mcp` execution keeps agent-scoped QMD config/cache env, with live memory-search proof before merge.

## Reproduction Assessment

Yes, at source level: current main passes QMD-scoped XDG env into mcporter, and mcporter v0.10.2 uses `XDG_CONFIG_HOME` for config discovery. I did not run a live gateway memory-search reproduction in this read-only review.

## Solution Assessment

No. The narrow env split is a plausible mitigation, but it is not the best fix until it preserves user mcporter discovery and the agent-scoped QMD runtime env for local/generated `qmd mcp` execution.
