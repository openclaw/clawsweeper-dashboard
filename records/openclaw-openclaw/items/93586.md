## What This Changes

The PR adds a separate mcporter spawn environment in the memory-core QMD manager that removes `XDG_CONFIG_HOME` and `XDG_CACHE_HOME`, with an adjacent test update.

## Best Possible Solution

Separate mcporter config discovery from QMD server execution: mcporter should use user/default discovery env, while generated or local `qmd mcp` execution must receive agent-scoped QMD config/cache env.

## Reproduction Assessment

Yes, at source level: current main passes QMD-scoped XDG env into mcporter, and mcporter v0.10 uses `XDG_CONFIG_HOME` for config discovery. I did not run a live gateway memory-search reproduction in this read-only review.

## Solution Assessment

No. The PR is a plausible narrow mitigation for mcporter config lookup, but it is not the best fix until it preserves user mcporter discovery and agent-scoped QMD execution env.
