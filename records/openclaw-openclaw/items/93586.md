- `P1`: The PR targets a real broken QMD memory-search workflow for mcporter 0.10+ users, and the current patch can still misroute or break that workflow.
- `merge-risk: 🚨 compatibility`: Changing the env inherited by mcporter and its stdio children can alter existing mcporter/qmd setups during upgrade.

## Summary

Keep open: the reported QMD/mcporter bug is valid, but this PR is not merge-ready because the env split also changes what mcporter-launched `qmd mcp` children inherit, and the supplied proof is source-grep/test-only rather than a real memory-search run.

## What This Changes

The branch adds a stripped `mcporterEnv` for QMD mcporter spawns and updates the adjacent spawn-env test to expect `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` to be absent.

## Best Possible Solution

Keep mcporter config discovery out of OpenClaw's QMD XDG tree while preserving per-agent QMD env for qmd stdio children, then land either a repaired narrow PR or the broader generated-config PR after live proof.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main and `v2026.6.8` pass QMD-scoped XDG env into mcporter, while mcporter v0.10.2 uses `XDG_CONFIG_HOME` for config discovery; I did not run the live OpenClaw plus mcporter plus QMD memory-search scenario.

## Solution Assessment

No. This is a plausible narrow fix for mcporter config lookup, but mcporter starts stdio servers from its own `process.env` plus server-specific overrides, so the patch can remove the per-agent QMD runtime env that existing `qmd mcp` entries rely on.
