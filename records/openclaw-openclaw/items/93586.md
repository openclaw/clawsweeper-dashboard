## Summary

Keep open: this PR targets a valid QMD/mcporter regression, but the current narrow env strip breaks the prior per-agent QMD env invariant for local mcporter-launched qmd servers and still lacks real behavior proof from an actual OpenClaw memory search.

## What This Changes

The branch adds a mcporter-specific spawn env without XDG_CONFIG_HOME or XDG_CACHE_HOME and updates the adjacent QMD manager test to expect those vars absent from mcporter calls.

## Best Possible Solution

Keep mcporter config discovery out of the QMD XDG tree while preserving or injecting per-agent QMD env for the qmd stdio server; the broader generated-config approach in https://github.com/openclaw/openclaw/pull/92499 can be canonical once its blockers are resolved.

## Reproduction Assessment

Yes, source-reproducible: current main passes QMD-scoped XDG env into mcporter, and mcporter v0.10.2 uses XDG_CONFIG_HOME for config discovery. I did not run the live mcporter/QMD memory-search scenario in this read-only review.

## Solution Assessment

No. Stripping XDG from the whole mcporter process is only a partial fix because mcporter uses process.env for stdio child qmd servers; the fix must keep mcporter discovery clean while preserving per-agent qmd state for the actual qmd subprocess.
