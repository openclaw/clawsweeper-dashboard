## Label Justifications

- `P1`: The PR targets a real broken QMD memory-search workflow for mcporter 0.10+ users, and the current patch can still break or misroute that workflow for local stdio QMD setups.
- `merge-risk: 🚨 compatibility`: Changing the environment inherited by mcporter and its stdio children can alter existing mcporter/qmd configurations during upgrade.
- `merge-risk: 🚨 session-state`: A qmd child that loses per-agent QMD XDG cache/config state can read or write the wrong agent memory index.
- `merge-risk: 🚨 availability`: Incorrect inherited env can leave mcporter-backed QMD memory search failing or returning empty results for affected setups.

## Summary

Keep open: this is a useful narrow candidate for the linked QMD/mcporter bug, but the patch is not merge-ready because it can strip per-agent QMD state from mcporter-launched stdio children and the supplied proof is source-grep/unit-test evidence rather than a real mcporter 0.10+ memory search.

## What This Changes

The PR adds a mcporter-specific spawn environment in QmdMemoryManager that removes XDG_CONFIG_HOME and XDG_CACHE_HOME, then updates the adjacent test to expect those variables absent for mcporter spawns.

## Best Possible Solution

Keep mcporter config discovery out of OpenClaw's QMD XDG tree while preserving per-agent QMD config/cache state for qmd stdio children, then land either a repaired narrow PR or the broader isolation PR with live proof.

## Reproduction Assessment

Yes at source level. Current main passes QMD-scoped XDG vars to mcporter, mcporter 0.10.2 uses XDG_CONFIG_HOME for config discovery, and mcporter stdio servers inherit process.env unless server env overrides are present; I did not run a live mcporter gateway reproduction.

## Solution Assessment

No. The patch fixes mcporter config discovery but strips per-agent XDG state from the mcporter process that launches qmd stdio servers, so the safer fix must keep discovery clean while preserving or injecting QMD env at the qmd child boundary.
