## Summary

Keep open: the PR targets a real QMD/mcporter regression, but the current patch is not merge-ready because it strips QMD's cache env from the mcporter process that launches the stdio QMD MCP server, and the supplied proof is code inspection rather than a real mcporter run.

## What This Changes

The PR adds a separate mcporter spawn environment in QMD memory manager that removes `XDG_CONFIG_HOME` and `XDG_CACHE_HOME`, then updates the focused spawn-env test to expect those variables to be absent.

## Best Possible Solution

Separate mcporter config discovery from QMD server runtime env: mcporter should not have QMD-scoped `XDG_CONFIG_HOME` for its own config lookup, but the actual `qmd mcp` server must still receive the per-agent QMD config/cache env or an explicit generated config that carries it.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main and the latest release pass QMD-scoped XDG env into mcporter, and mcporter 0.10+ uses `XDG_CONFIG_HOME` for config discovery. I did not run the live gateway plus mcporter scenario in this read-only review.

## Solution Assessment

No, this patch is too broad as written. The safer fix must avoid redirecting mcporter config discovery while preserving or explicitly injecting the per-agent QMD env into the `qmd mcp` server that mcporter launches.
