## What This Changes

The PR adds a separate mcporter spawn environment in the memory-core QMD manager that strips `XDG_CONFIG_HOME` and `XDG_CACHE_HOME`, and updates the adjacent mcporter env test expectation.

## Best Possible Solution

Use a target-specific env split that preserves user mcporter discovery, prevents QMD-scoped XDG from shadowing mcporter config, and passes a paired QMD config/cache env only to the qmd MCP child.

## Reproduction Assessment

Yes, source-reproducible: current main passes QMD-scoped XDG env into mcporter, and mcporter v0.10.2 uses `XDG_CONFIG_HOME` for config discovery. I did not run a live gateway plus mcporter/qmd scenario in this read-only review.

## Solution Assessment

No. Splitting mcporter from QMD-scoped XDG is the right surface, but this implementation should preserve user mcporter discovery env and avoid a half-paired QMD child env.
