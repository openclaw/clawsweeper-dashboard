## Summary

Keep open: this targets a real QMD/mcporter regression, but the branch still has merge-blocking env routing defects and the supplied proof is source-grep output rather than a real mcporter 0.10+ memory-search run.

## What This Changes

The PR adds a separate `mcporterEnv` for `QmdMemoryManager`, removes `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` from mcporter spawns, and updates the qmd-manager test to expect those vars absent while retaining `QMD_CONFIG_DIR`.

## Best Possible Solution

Use a target-specific env/config strategy that preserves user mcporter discovery while keeping qmd's config/cache env paired for the qmd child, then prove it with a redacted OpenClaw memory search on mcporter 0.10+.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 pass QMD-scoped XDG env into mcporter, and mcporter v0.10.2 uses XDG_CONFIG_HOME for config discovery. I did not run a live gateway plus mcporter/qmd scenario in this read-only review.

## Solution Assessment

No. Splitting mcporter from QMD-scoped XDG is the right surface, but this implementation should preserve user mcporter discovery env and avoid a half-paired QMD child env.

  - body: `mcporterEnv` is destructured from `this.env` after the manager has overwritten the user's XDG homes with QMD-scoped paths.   - confidence: 0.9
