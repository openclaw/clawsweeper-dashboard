## Summary

Keep open: the PR addresses a real shipped QMD/mcporter XDG regression, but the current env split is too broad because it also removes the per-agent QMD env that mcporter-launched local `qmd mcp` servers inherit.

## What This Changes

The PR adds a stripped `mcporterEnv` for QMD mcporter spawns and changes the adjacent test to expect `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` to be absent.

## Best Possible Solution

Separate mcporter config discovery from qmd server runtime env: keep mcporter from resolving config under QMD-scoped XDG state while ensuring local `qmd mcp` children still receive per-agent QMD config/cache env through generated server env or an equivalent explicit contract.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.8` pass QMD-scoped XDG env into mcporter, and upstream mcporter uses `XDG_CONFIG_HOME` for config discovery; I did not run the live gateway plus mcporter scenario in this read-only review.

## Solution Assessment

No. This is a plausible narrow fix for mcporter config lookup, but it removes inherited QMD runtime env that existing local stdio `qmd mcp` server entries may rely on; the better fix preserves both config discovery and per-agent QMD runtime state.
