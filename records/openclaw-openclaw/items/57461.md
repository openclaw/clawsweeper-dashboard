## Summary

Keep open: current main still documents generic MCP clients as supported, but the affected channel tools keep actionable data in `structuredContent` while `content` remains counts or short status text. The directly related fix PR #57576 is closed unmerged, so there is no implemented or superseding fix to close against.

## What This Changes

The issue requests that OpenClaw MCP channel tools include their full structured result data as JSON text in `content` while preserving `structuredContent`.

## Best Possible Solution

Update the shared MCP channel success-result path so each documented generic channel tool returns self-sufficient JSON text in `content` while preserving the existing `structuredContent` payload and error/status behavior where appropriate.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: invoke an affected tool such as `conversations_list` after the bridge returns one conversation, and current main returns `content[0].text` as `conversations: 1` while the conversation array exists only in `structuredContent`.

## Solution Assessment

Yes. The narrowest maintainable fix is a shared helper or small per-tool wrapper that serializes the same successful structured payload into JSON text `content` without changing Gateway route semantics, Claude-specific notifications, or error responses.
