## Summary

Keep open. Current main still has source-level evidence that provider onboarding/auth setup writes static and token profiles to one resolved agent store, while the runtime read-through path is main-store based and local profiles can shadow inherited credentials; the earlier fix PR was closed unmerged and the broader rotation issue does not fully cover this onboarding case.

## What This Changes

The issue asks for onboarding/auth setup to stop leaving configured non-default agents without usable provider credentials.

## Best Possible Solution

Define one multi-agent credential contract, then implement it consistently across interactive and non-interactive onboarding, runtime read-through, and stale local override handling with explicit OAuth and SecretRef portability rules.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: onboarding/plugin auth writes static or token profiles through one resolved `agentDir`, and runtime inheritance only helps when the requested agent has no local overriding profile and the canonical main/default store has the credential.

## Solution Assessment

Unclear. Writing every credential to every agent may solve the symptom, but it is not clearly the best fix until maintainers choose the credential fan-out, SecretRef portability, OAuth, and stale-local-profile overwrite semantics.
