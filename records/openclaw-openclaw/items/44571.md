## Summary

Keep open. Current main still persists provider onboarding and Anthropic setup-token credentials to one resolved agentDir, while runtime inheritance only helps missing profiles and selected OAuth freshness cases; static token/API-key profile conflicts can still shadow main credentials in non-default agents.

## What This Changes

The issue asks for `openclaw onboard` credentials to be usable by all configured agents, either by writing portable auth profiles to each agent store or by resolving auth from a canonical shared/default location without silent fallback.

## Best Possible Solution

Define the multi-agent auth contract first, then implement it consistently: either copy only portable static onboarding credentials to configured agent stores with clear overwrite and SecretRef rules, or keep per-agent isolation and add first-class sync/repair/warning behavior that prevents stale local profiles from shadowing valid main credentials.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction: the current onboarding paths pass exactly one agentDir to the auth writer, and the runtime merge rules still allow local static profiles to shadow main credentials on conflicts.

## Solution Assessment

Unclear. The report names two viable directions, but choosing between credential fan-out and canonical/read-through inheritance affects isolation and secret handling, so a maintainer product/security decision should precede an automated fix.
