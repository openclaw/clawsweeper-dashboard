## Summary

Keep open. Current main still writes provider onboarding credentials to one resolved agent directory, while runtime read-through only covers missing profiles and selected OAuth freshness cases; stale local static token/API-key profiles can still shadow the default/main store.

## What This Changes

The issue asks onboarding or runtime auth resolution to make newly configured provider credentials usable by every configured agent instead of only the resolved default agent store.

## Best Possible Solution

Define one multi-agent auth contract, then implement it consistently: sync only portable static credentials with explicit shadow/overwrite rules, or keep isolation/read-through and add repair or warning behavior for stale local profiles.

## Reproduction Assessment

Yes. Source inspection is high-confidence: the current onboarding paths pass exactly one agentDir to auth profile writes, and local static profiles can still shadow inherited main credentials.

## Solution Assessment

Unclear. Both requested directions are plausible, but choosing credential fan-out versus read-through plus shadow repair/warnings affects credential isolation and secret handling.
