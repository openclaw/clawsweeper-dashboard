## Summary

Keep this issue open. Current main still lacks user-visible diagnostics for malformed or incomplete `SKILL.md` files: invalid candidates are dropped before the status report reaches CLI or Control UI, there is no `skills validate` command, and the docs describe the required format without a validation/debug path.

## What This Changes

The issue requests visible SKILL.md frontmatter parsing and validation errors in CLI and Control UI, plus a dedicated skills validation command and docs updates.

## Best Possible Solution

Add a backward-compatible skill discovery diagnostics layer that records invalid `SKILL.md` candidates with path, source, reason, and parser location when available; surface those diagnostics in `skills.status`, `openclaw skills list/check`, a new or equivalent validation command, Control UI, and the skill authoring docs while preserving current precedence and containment rules.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a workspace skill directory with malformed frontmatter or a missing `description`: current parsing/loader code returns `{}` or `null`, and downstream status/CLI/UI paths only render successfully loaded entries.

## Solution Assessment

Yes, with one naming caveat. The narrow maintainable fix is to add optional diagnostics to the existing discovery/status pipeline and render them in current CLI/UI surfaces; the command name can be settled as `skills validate` or an expanded `skills check` without changing the underlying solution.
