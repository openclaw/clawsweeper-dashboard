## Label Justifications

- `P2`: The issue affects custom skill loading diagnostics and developer workflow, but it does not make the core runtime unusable for all users.
- `impact:session-state`: Malformed workspace skills are omitted from the loaded skill snapshot and agent-visible context, creating skill availability state drift.

## Summary

Keep open. Current main still drops malformed or incomplete workspace skill candidates before they can appear in CLI, gateway status, or the Skills UI; the related diagnostics PR was closed unmerged, and the latest release has the same loader behavior.

## What This Changes

The issue requests visible CLI/UI diagnostics and validation for malformed workspace SKILL.md frontmatter instead of silently omitting custom skills.

## Best Possible Solution

Add a structured invalid-skill diagnostics channel in the skill discovery/status pipeline, then expose the same redacted diagnostics through `openclaw skills check` or a chosen validation command, gateway status, and the Skills UI.

## Reproduction Assessment

Yes, source-reproducible: a malformed SKILL.md or one missing `description` is reduced to an empty/null loader result before status, CLI, or UI rendering. I did not run a live repro command because this review was constrained to read-only checkout inspection.

## Solution Assessment

Unclear on the exact public surface: the diagnostic direction is the maintainable fix, but maintainers need to choose whether it is `skills check`, a new `skills validate`, status payload expansion, or all of those. The implementation should avoid a second parser path that drifts from runtime discovery.
