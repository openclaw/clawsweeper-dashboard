## Summary

Keep open. Current main still drops invalid or incomplete workspace skill candidates before they can appear in the skill status payload, CLI output, or Control UI, and there is no `skills validate` command or equivalent invalid-SKILL.md diagnostic surface. The central user problem remains actionable in the skill loader/status path.

## What This Changes

Request to surface malformed or incomplete workspace `SKILL.md` frontmatter diagnostics in CLI and Web UI, add a validation command, and document the valid format and debugging flow.

## Best Possible Solution

Add a shared skill discovery diagnostic model that records skipped candidate paths with source, reason, and parse/metadata details, then surface it through `buildWorkspaceSkillStatus`, `skills.status`, CLI list/check/validate output, and the Control UI. Preserve existing compatibility unless explicitly changed, and update skill docs plus focused tests for malformed fences, invalid YAML, missing description, oversized files, and grouped skill roots.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a workspace skill candidate with no opening fence or no description reaches `parseFrontmatterBlock`/`loadSingleSkillDirectory`, becomes `{}` or fails metadata checks, and is returned as `null` before `skills.status` or CLI formatting can report it. A live filesystem fixture was not created because this review is read-only.

## Solution Assessment

Yes, the requested direction is the maintainable fix if implemented through a shared loader/status diagnostic contract rather than CLI-only logging. The only adjustment is to treat missing `name` according to current compatibility policy, likely as a validation warning unless maintainers intentionally remove directory-name fallback.
