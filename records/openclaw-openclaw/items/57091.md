## Label Justifications

- `P2`: The issue affects custom skill loading/debuggability for skill authors but is not a core runtime outage.
- `impact:session-state`: Malformed workspace skill files can make configured skill state disappear from status and UI surfaces without user-visible explanation.

## Summary

Keep open. Current main still has no structured diagnostics for malformed or incomplete workspace SKILL.md candidates: invalid candidates are omitted before skills.status, CLI list/check, or the Control UI can report them, and the related diagnostics PRs were closed unmerged or only covered narrower colon-description/template cases.

## What This Changes

The issue requests visible diagnostics and validation for malformed workspace SKILL.md frontmatter so invalid custom skills do not disappear silently from CLI or Web surfaces.

## Best Possible Solution

Add a structured skipped-skill diagnostic path from discovery into skills.status, CLI list/check, and the Skills UI, with docs and focused tests for missing fences, invalid YAML, missing required fields, and redacted path/error output.

## Reproduction Assessment

Yes, at source level: a SKILL.md with no valid frontmatter block or no description produces empty metadata, then loadSingleSkillDirectory returns null, and downstream status/UI/CLI only see loaded entries. I did not run a filesystem repro because this review is read-only.

## Solution Assessment

Unclear. The requested direction is sound, but the exact best solution needs maintainer choices around status schema, CLI command naming, redaction, and whether diagnostics are status entries or a separate validation report.
