## Summary

Keep open. Current main still has English-only runtime failure text in the sanitizer, error formatter, and reply-runner paths, with no locale input or messages config/schema field to select translated user-facing errors. Existing Control UI localization and plugin message hooks are adjacent but do not provide stable runtime error keys plus locale context, so this remains valid and needs maintainer product/API direction before implementation.

## What This Changes

The issue requests a lightweight locale system for user-facing runtime error messages, including a messages locale config field, keyed English-fallback dictionaries, and wiring for sanitizer/reply-runner failure text.

## Best Possible Solution

Add a stable runtime localization contract for user-facing errors: structured error identifiers with English fallback, documented locale resolution/config behavior, wiring through sanitizer/error-format/reply-runner paths, and tests for fallback plus at least one non-English locale. If localization should be plugin-owned, expose structured error keys and locale context first so plugins do not match English literals.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature/API request. The verification path is high-confidence source inspection showing English literals, no locale parameter in sanitizer/reply-runner calls, and no messages locale config/schema field.

## Solution Assessment

Unclear. The proposed keyed catalog with English fallback is plausible, but the best maintainable approach needs maintainer direction on config shape, per-user versus gateway-default locale resolution, and core-owned versus plugin-owned translation boundaries.
