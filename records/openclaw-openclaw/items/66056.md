## Summary

Keep open. Current main still has English-only user-facing runtime failure text in the sanitizer and reply-runner paths, and the shared `messages` config/schema surface still has no locale setting for those errors.

## What This Changes

Request locale-aware, fallback-safe translations for user-facing runtime error messages such as rate limits, context overflow, billing, timeout, transport, and missing-key failures.

## Best Possible Solution

Define a stable runtime localization contract with keyed user-facing error identifiers, English fallback, documented locale resolution, wiring through sanitizer and reply-runner failures, and focused tests for fallback plus at least one non-English locale.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature/API request. The current-main verification path is high-confidence source and test inspection showing English-only runtime error outputs and no locale-aware config or sanitizer API.

## Solution Assessment

Unclear. The proposed lightweight catalog is plausible, but the narrow maintainable fix depends on maintainer direction for locale source, config shape, translation ownership, and core-versus-plugin boundaries.
