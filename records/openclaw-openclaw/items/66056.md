## Summary

Keep open. Current main still has English-only runtime failure text, no locale-aware sanitizer API, and no `messages` locale config; this is a real feature/API request that needs maintainer product direction rather than cleanup closure.

## What This Changes

Add locale-aware handling for user-facing runtime failure messages such as rate limits, billing errors, timeouts, context overflow, and missing API keys.

## Best Possible Solution

Define stable user-facing error keys with English fallback, document locale resolution, wire sanitizer and reply-runner failure paths through that contract, and add focused config/fallback/non-English regression coverage.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature/API request. Source inspection on current main gives high-confidence verification that the requested locale-aware runtime error-message capability is absent.

## Solution Assessment

Unclear. A lightweight catalog may be reasonable, but the maintainable solution needs maintainer decisions about locale resolution, config naming, translation ownership, and whether localization is core-owned or exposed through a structured plugin contract.
