## Summary

Keep open. Current main still emits English-only gateway/runtime failure text, and the remaining work is a new localization/config/API contract rather than a conservative cleanup close.

## What This Changes

Request locale-aware formatting for user-facing runtime failure messages, with English fallback and a configurable locale.

## Best Possible Solution

Define stable user-facing error keys with English fallback, document locale resolution, wire the sanitizer and reply-runner failure paths through that contract, and add focused fallback plus non-English coverage.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature/API request. Source inspection gives high-confidence verification that current main still has English-only runtime failure text and no runtime error-locale input.

## Solution Assessment

Unclear. The proposed lightweight JSON catalog is plausible, but the best maintainable fix depends on maintainer decisions about config naming, per-user versus gateway-default locale, translation ownership, and plugin API boundaries.
