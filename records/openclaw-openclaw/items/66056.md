## Summary

Keep open. Current main still emits English-only gateway/runtime failure text, and the requested locale/config/API behavior needs maintainer product design rather than conservative cleanup.

## What This Changes

Request locale-aware handling for user-facing gateway/runtime error messages with a messages locale config, an English fallback catalog, and translated failure text.

## Best Possible Solution

Define stable user-facing error keys with English fallback, document locale resolution, wire sanitizer and reply-runner failure paths through that contract, and add focused English-fallback plus non-English regression coverage.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature/API request. Source inspection on current main gives high-confidence verification that runtime failure text is still English-only and has no runtime error-locale input.

## Solution Assessment

Unclear. The lightweight JSON catalog is plausible, but the best maintainable fix depends on maintainer decisions about config naming, gateway-default versus per-user locale resolution, translation ownership, and plugin API boundaries.
