## Summary

Keep open. Current main still has English-only user-facing runtime failure text in the sanitizer and reply-runner paths, while the shared messages config and plugin hook surfaces do not expose a runtime error locale or structured error keys.

## What This Changes

Add locale-aware formatting and configuration for user-facing runtime error messages such as rate limits, context overflow, timeouts, billing, transport, and missing API-key failures.

## Best Possible Solution

Define a stable runtime localization contract with keyed user-facing error identifiers, English fallback, documented locale resolution, wiring through sanitizer and reply-runner failures, and focused fallback plus non-English locale tests.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature/API request. Source and test inspection provide high-confidence verification that current main still emits English-only runtime error text and has no locale-aware config or sanitizer API.

## Solution Assessment

Unclear. The lightweight JSON catalog proposed in the issue is plausible, but the best maintainable solution depends on maintainer choices around config naming, locale resolution, translation ownership, and core-versus-plugin boundaries.
