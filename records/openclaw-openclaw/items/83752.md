## Summary

Keep open. The report has concrete current-release live evidence, and current main still has the exact heavy path: Active Memory runs in `before_prompt_build`, `full` mode sends the whole prompt message history to an embedded recall agent, and no merged fix or canonical duplicate closes the retained-RSS symptom after those turns complete.

## What This Changes

The issue asks OpenClaw to stop retaining large gateway RSS after completed Telegram group-topic turns that invoke Active Memory full-context recall.

## Best Possible Solution

Profile a current-main live workload and fix the retention source in Active Memory embedded recall or agent/session cleanup, while keeping full-mode and group/channel use opt-in and documented.

## Reproduction Assessment

No, not for the retained-RSS failure end to end. Source inspection confirms the heavy full-context Active Memory path on current main, but a high-confidence reproduction still needs a live Telegram/Active Memory run with RSS or heap-profile proof after the queue is idle.

## Solution Assessment

Unclear until profiling isolates the retained object class. A maintainable fix should target the confirmed retention source rather than only documenting lower-cost settings or adding a broad memory cap.
