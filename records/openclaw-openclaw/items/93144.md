## Summary

Keep open: current main and v2026.6.6 still substitute config environment placeholders during config read and pass static MCP header/env records into transports; the broader dynamic-header issue is related but does not fully cover this narrower ENV_VAR and stdio-env request.

## What This Changes

Request dynamic re-evaluation of `${ENV_VAR}` placeholders for MCP HTTP headers and stdio env instead of freezing substituted values at config load.

## Best Possible Solution

Define a narrow core MCP runtime design that preserves ENV_VAR templates only where needed, re-resolves them at HTTP request boundaries and stdio spawn boundaries, and proves the OAuth, redirect, redaction, and env-safety invariants with focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction. The current limitation is source-verifiable: OpenClaw resolves `${ENV_VAR}` during config read and transports receive static header/env records.

## Solution Assessment

Unclear until maintainers choose the security and API shape. The proposed fetch-closure approach is technically supported by the SDK for HTTP transports, but it needs an owner-approved boundary with OAuth, redirects, redaction, and stdio env filtering.
