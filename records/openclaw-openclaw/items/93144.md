## Summary

Keep open: current main and v2026.6.9 still resolve MCP `${ENV_VAR}` values during config activation and pass fixed header/env values into MCP transports, while https://github.com/openclaw/openclaw/pull/94167 is the open candidate implementation and still needs maintainer security/product review.

## What This Changes

Requests per-request or per-spawn resolution of MCP `${ENV_VAR}` templates in transport headers and stdio env instead of freezing them during config activation.

## Best Possible Solution

Settle one maintainer-approved MCP dynamic credential contract, then land or revise https://github.com/openclaw/openclaw/pull/94167 with security/docs/tests, or route the remaining need to OAuth, SecretRef, or the broader dynamic-header design.

## Reproduction Assessment

Yes, by source inspection: current main resolves config env templates during config activation and MCP transports consume the resolved values. I did not run a live MCP server in this read-only pass.

## Solution Assessment

Unclear. The SDK fetch hook makes per-request HTTP substitution plausible, but the best OpenClaw solution needs maintainer agreement on whether mutable process.env, OAuth, or SecretRef should own MCP credential rotation.
