## Summary

Keep open: current main and v2026.6.6 still resolve `${ENV_VAR}` at config activation and then pass fixed header/env values into MCP transports, so the requested dynamic behavior remains missing. This is a feature/product decision rather than a strict bug because current docs now describe activation-time resolution and the change would affect security-sensitive MCP auth semantics.

## What This Changes

The issue requests per-request or per-spawn re-resolution of MCP `${ENV_VAR}` substitutions for HTTP headers and stdio env values instead of freezing them at gateway activation.

## Best Possible Solution

Define an owner-approved MCP dynamic-secret contract first: either keep activation-time `${ENV}` and steer token rotation to OAuth/SecretRef, or add a narrow dynamic MCP resolver that preserves raw templates, reuses existing safety filters, and documents per-transport granularity with focused tests.

## Reproduction Assessment

Yes, by source inspection: config env substitution runs during config read, and MCP transports receive fixed resolved values. I did not run a live MCP server in this read-only cleanup pass, but the code path is direct and current-release behavior matches main.

## Solution Assessment

Unclear. The SDK fetch hook makes the proposed HTTP shape plausible, but the best OpenClaw solution needs a maintainer decision on whether mutable process.env should be a request-time secret source or whether OAuth/SecretRef should own credential rotation.
