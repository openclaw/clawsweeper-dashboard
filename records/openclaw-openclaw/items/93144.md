## Summary

Keep open: current main and v2026.6.9 still resolve `${ENV_VAR}` during config activation and pass fixed MCP header/env values into transports, while https://github.com/openclaw/openclaw/pull/94167 is an open closing implementation candidate that still needs maintainer product/security review.

## What This Changes

Request per-request or per-spawn re-evaluation of MCP `${ENV_VAR}` substitution for transport headers and stdio env instead of freezing values at Gateway startup.

## Best Possible Solution

Review, narrow, and either land or reject https://github.com/openclaw/openclaw/pull/94167 after maintainers settle one dynamic MCP credential contract with security, docs, and regression proof.

## Reproduction Assessment

Yes, at source level: current main resolves config env templates during config activation and MCP transports consume the resolved values. I did not run a live MCP server in this read-only pass.

## Solution Assessment

Unclear. The SDK fetch hook makes per-request HTTP substitution plausible, but the best OpenClaw solution needs maintainer agreement on whether mutable `process.env`, OAuth, or SecretRef should own MCP credential rotation.
