## Summary

Keep open: current main and the latest release still retain parsed session-store objects and immutable snapshots without an object/snapshot byte cap, and the linked open PR owns the proposed repair.

## What This Changes

Report gateway heap pressure and crash risk from unbounded parsed sessions.json object and snapshot cache retention during session-store hydration.

## Best Possible Solution

Land or revise https://github.com/openclaw/openclaw/pull/88150 so oversized session stores skip object and snapshot cache residency while preserving on-disk reload behavior and regression coverage.

## Reproduction Assessment

Yes at source/proof level, but I did not run a live oversized-store OOM reproduction. Current main parses sessions.json and publishes parsed object/snapshot caches without a byte cap, and the linked PR body includes runtime cache-count proof for the intended fix.

## Solution Assessment

Yes, the narrow maintainable direction is to cap runtime cache residency for oversized session stores while keeping disk as the source of truth. Pruning reads or dropping session data would be the wrong layer for this bug.
