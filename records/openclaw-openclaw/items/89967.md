## Summary

Keep open for maintainer review before merge: the patch is a focused, likely-correct fix with sufficient live macOS proof, but it intentionally changes the raw LaunchAgent executable for existing service-env installs and needs maintainer acceptance of that compatibility risk.

## What This Changes

The PR changes macOS LaunchAgent plist generation/readback so service-env plists launch `/bin/sh` with the generated wrapper and env file before the existing gateway command, and adds regression coverage for new and legacy plist shapes.

## Best Possible Solution

Land the shell-first LaunchAgent serialization with legacy wrapper-first readback/rewrite after maintainers accept the macOS plist compatibility change.

## Reproduction Assessment

Yes for the source-level failure surface: current main and `v2026.6.6` write the generated env wrapper as the raw `ProgramArguments[0]` executable when service env entries exist. I did not live-reproduce the exact external-home setup in this Linux checkout.

## Solution Assessment

Yes. Starting launchd from a stable system shell while preserving owner-only env-file loading is narrower than changing gateway command construction or embedding secrets in the plist.
