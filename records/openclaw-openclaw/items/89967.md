## Summary

Keep open: this PR is still the active focused fix candidate for the linked macOS LaunchAgent env-wrapper spawn failure. Current main and the latest release still generate wrapper-first service-env LaunchAgent arguments, while this branch changes that boundary without introducing a concrete blocking correctness finding.

## What This Changes

The PR changes macOS LaunchAgent service-env generation/readback to use `/bin/sh`, generated wrapper path, env file path, then the existing gateway command, with regression coverage for new and legacy layouts.

## Best Possible Solution

Land the shell-first LaunchAgent serialization/readback path after maintainers accept the macOS launchd compatibility change, while keeping env values in the owner-only env file and preserving current gateway command construction.

## Reproduction Assessment

Yes, for the source-level failure surface: current main and v2026.6.8 serialize the generated env wrapper as `ProgramArguments[0]` when service env entries exist. I did not run the exact external-home macOS setup in this read-only review, but the PR body supplies live launchctl output against the changed spawn surface.

## Solution Assessment

Yes. Starting launchd from `/bin/sh` while preserving owner-only env-file loading is narrower than changing gateway command construction, changing status parsing only, or embedding secrets in the plist.
