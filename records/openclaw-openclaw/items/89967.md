## Summary

Keep open: this focused PR still addresses the linked macOS LaunchAgent service-env bug, and current main plus the latest release still use the wrapper-first plist shape. I found no blocking code defect, but the raw launchd executable/rewrite behavior is compatibility-sensitive and should get maintainer acceptance before merge.

## What This Changes

The PR changes macOS LaunchAgent service-env plists to launch `/bin/sh` before the generated env wrapper, keeps legacy wrapper-first readback/rewrite, and adds focused launchd tests.

## Best Possible Solution

Land the shell-first LaunchAgent serialization with legacy wrapper-first readback/rewrite and unchanged gateway command generation once maintainers accept the macOS plist compatibility risk.

## Reproduction Assessment

Yes for the source-level failure surface: current main and v2026.6.6 write the generated env wrapper as the raw LaunchAgent executable when service env entries exist. The exact external-home volume failure was not reproduced in this Linux checkout, but the PR supplies live macOS launchctl proof for the changed path.

## Solution Assessment

Yes, pending maintainer acceptance of the compatibility behavior. Starting launchd from a stable system shell while preserving owner-only env-file loading is narrower than changing the gateway command, embedding secrets in the plist, or adding a separate fallback path.
