## Summary

Keep open. Current main still applies a 1500ms effective timeout to active non-loopback configured remote probes even when the user supplies a larger `--timeout`, and the latest release tag contains the same behavior.

## What This Changes

Requests that `openclaw gateway probe --timeout ...` stop false-negativing healthy remote `configRemote` gateways by honoring the caller budget, raising the remote cap, or exposing the effective per-target budget.

## Best Possible Solution

Update the gateway probe budget policy so active configured remote targets and explicit non-loopback remote URLs no longer silently collapse a user-supplied large timeout to 1500ms. Preserve the short SSH-tunnel setup and inactive local-loopback behavior unless maintainers deliberately change those policies, and align JSON/text/docs with whatever per-target budget semantics remain.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure `gateway.mode="remote"` with a non-loopback `gateway.remote.url`, run `gateway probe` with a large timeout, and the code path still computes 1500ms for the active `configRemote` before calling `probeGateway`.

## Solution Assessment

Yes. The narrowest maintainable fix is in `src/commands/gateway-status/helpers.ts` plus call-site regression coverage, keeping unrelated probe target policies intact while making the user-facing timeout truthful for active remote probes.
