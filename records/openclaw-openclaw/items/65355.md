## Summary

Keep open. Current main still creates active configured remote probe targets and passes a 1500ms non-loopback budget into `probeGateway`, and the latest release tag carries the same policy, so the reported false-negative path remains source-backed.

## What This Changes

The issue requests that `openclaw gateway probe --timeout` stop false-negativing healthy remote `configRemote` gateways by not hard-capping non-loopback probe targets to 1500ms.

## Best Possible Solution

Change the gateway probe budget policy so active configured remote targets, and likely explicit non-loopback remote URLs, either honor the caller's timeout or surface a documented effective per-target budget. Preserve intentional SSH-tunnel and inactive local-loopback caps unless maintainers choose a broader policy change, and cover the remote-mode call path with regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `gateway.mode="remote"` plus `gateway.remote.url` creates an active `configRemote`, `resolveProbeBudgetMs(30000, target)` returns 1500, and the probe runner passes that capped value to `probeGateway`.

## Solution Assessment

Yes. The narrowest maintainable fix is in the gateway-status budget helper with call-site tests for remote mode, plus output/docs updates only if per-target budgets continue to differ from the overall CLI budget.
