## Summary

Keep open. Current main and the latest release still cap active configured remote and explicit non-loopback gateway probe budgets to 1500ms while the CLI presents `--timeout` as the overall probe budget, so the reported false-negative path remains source-reproducible.

## What This Changes

The issue asks `gateway probe` to honor a user-supplied timeout for healthy remote `configRemote` and other non-loopback targets instead of using a fixed 1500ms per-target cap.

## Best Possible Solution

Update the gateway-status probe budget policy so active configured remote targets and explicit remote URLs use the caller budget, preserve SSH tunnel and inactive loopback caps, add focused regression coverage, and include a user-facing changelog entry.

## Reproduction Assessment

Yes, at source level. `gateway.mode="remote"` plus a non-loopback `gateway.remote.url` creates an active `configRemote`, `resolveProbeBudgetMs(30000, target)` returns 1500, and the probe runner passes that value to `probeGateway`.

## Solution Assessment

Yes. The narrow maintainable fix is in the gateway-status budget helper and its call-site tests; changing general Gateway RPC behavior or adding a config option is unnecessary for the reported mismatch.
