## Summary

Keep open. Current main and the latest release still route `openclaw gateway probe --timeout` through a per-target budget helper that caps active non-loopback `configRemote` and explicit remote targets to 1500ms, so the reported false-negative class remains source-reproducible.

## What This Changes

The issue reports that `openclaw gateway probe --timeout` hard-caps non-loopback `configRemote` targets to 1500ms, causing false-negative diagnostics for healthy remote gateways.

## Best Possible Solution

Make active configured remote and explicit direct remote probes honor the caller's timeout budget or expose the actual per-target budget clearly, while preserving intentional short caps for SSH setup, discovery, and inactive local probes.

## Reproduction Assessment

Yes for the timeout-budget bug: current main's helper returns 1500ms for a 15000ms active `configRemote` non-loopback target, and the command passes that value into `probeGateway`. I did not rerun the reporter's live remote gateway environment.

## Solution Assessment

Yes, a narrow helper/caller regression fix is the right shape: align active remote probe budgeting with the user-supplied timeout or report the effective per-target cap. This should not require gateway protocol, auth, or config changes.
