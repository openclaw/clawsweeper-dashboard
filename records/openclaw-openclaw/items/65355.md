## Summary

Keep open. Current main and the latest release still cap active configured remote and explicit non-loopback gateway probe budgets to 1500ms while the CLI presents `--timeout` and `Probe budget` as the overall probe budget, so the reported false-negative path remains source-reproducible.

## What This Changes

The issue asks `gateway probe` to honor or accurately report user-supplied timeout budgets for remote `configRemote` and other non-loopback targets instead of hard-capping them at 1500ms.

## Best Possible Solution

Update the gateway-status probe budget policy so active configured remote targets and explicit remote URLs honor the caller budget or clearly expose their effective per-target budget, while preserving SSH tunnel and inactive-loopback caps and adding regression coverage.

## Reproduction Assessment

Yes, at source level. `gateway.mode="remote"` plus a non-loopback `gateway.remote.url` creates an active `configRemote`, `resolveProbeBudgetMs(30000, target)` returns 1500, and the probe runner passes that capped value to `probeGateway`.

## Solution Assessment

Yes. The narrow maintainable fix is in the gateway-status budget helper and related tests; changing general Gateway RPC behavior or adding a new config option is unnecessary for the reported mismatch.
