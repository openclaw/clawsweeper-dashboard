## Summary

Keep open. Current main and the latest release still cap active configured remote and explicit non-loopback gateway probe budgets to 1500ms while the CLI presents `--timeout` as the overall probe budget, so the reported false-negative path remains source-reproducible.

## What This Changes

The issue asks `gateway probe` to honor a user-supplied large timeout for healthy remote `configRemote` targets instead of silently capping non-loopback probes at 1500ms.

## Best Possible Solution

Update the gateway-status probe budget policy so active configured remote targets and explicit remote URLs use the caller budget, preserve the SSH tunnel and inactive loopback caps, add focused regression coverage, and include a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: `gateway.mode="remote"` plus a non-loopback `gateway.remote.url` creates an active `configRemote`, `resolveProbeBudgetMs(30000, target)` returns 1500, and the probe runner passes that value to `probeGateway`.

## Solution Assessment

Yes. The narrow maintainable fix is in `src/commands/gateway-status/helpers.ts` with call-site regression coverage; changing general Gateway RPC behavior or adding a new config option is unnecessary for the reported mismatch.
