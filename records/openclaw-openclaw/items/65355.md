## Summary

Keep open. Current main and `v2026.5.18` still cap active configured remote and explicit non-loopback gateway probe targets to 1500ms while the CLI presents `--timeout` as the overall probe budget, so the reported false-negative path remains source-reproducible.

## What This Changes

The issue requests that `openclaw gateway probe --timeout <ms>` stop false-negativing healthy remote `configRemote` gateways because non-loopback probe targets are internally capped to 1500ms.

## Best Possible Solution

Update the gateway-status probe budget policy so active configured remote targets and explicit remote URLs honor the caller budget, while preserving SSH tunnel and inactive-loopback caps and adding focused regression coverage.

## Reproduction Assessment

Yes, at source level. `gateway.mode="remote"` plus a non-loopback `gateway.remote.url` creates an active `configRemote`, `resolveProbeBudgetMs(30000, target)` returns 1500, and the probe runner passes that capped value to `probeGateway`.

## Solution Assessment

Yes. The narrow maintainable fix is in the gateway-status budget helper and related tests; changing general Gateway RPC behavior or adding a new config option is unnecessary for the reported mismatch.
