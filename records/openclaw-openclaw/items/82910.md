## Summary

Keep open: current main still has the GPT-5 reply brevity guard, it still mutates final payload text after the model run, and existing tests assert that this truncation behavior is intentional for some turns. The related Telegram truncation reports fixed a delivery symptom, but the later discussion there points back to this same upstream guard as remaining work.

## What This Changes

The issue requests disabling or reworking the hardcoded GPT-5 reply brevity guard because it can post-process OpenAI GPT-5 replies into ellipsis-truncated text with no config opt-out.

## Best Possible Solution

Define the GPT-5 chat brevity policy at the agent-runner boundary: avoid post-hoc truncation of normal user-facing final replies, keep any terse ack behavior narrowly scoped, and document a config surface only if maintainers want an explicit opt-out.

## Reproduction Assessment

Yes, source-level: current main still applies the guard to OpenAI GPT-5-family final payloads and appends `...` after shortening. I did not run the live Discord/Codex setup, but the code path and existing tests are enough to reproduce the mutation in source.

## Solution Assessment

Unclear. A config opt-out may be acceptable, but the safer product direction may be to remove or narrow post-hoc final-payload slicing first; current tests assert the behavior, so maintainers should decide the policy before implementation.
