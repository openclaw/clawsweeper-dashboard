## Summary

Current main still contains and tests the hardcoded GPT-5 reply brevity guard, so this should stay open. The reported source path is credible, but the requested opt-out changes public config and should get maintainer direction on whether to remove, narrow, log, or configure the guard; likely routing starts with joshavant, with vincentkoc as recent adjacent history.

## What This Changes

Request to stop or make configurable the GPT-5 reply brevity caps that can silently shorten final assistant replies.

## Best Possible Solution

Decide the owner policy for GPT-family reply post-processing, then implement a documented config or safer guard that cannot silently drop user-visible final reply content and includes numbered-list regression coverage.

## Reproduction Assessment

Yes, from source inspection rather than a live channel run: current main mutates GPT-5 final reply payloads after the model run and the helper regex counts numbered-list markers as sentence parts. The focused Node mirror confirms a numbered list can be shortened with a literal ellipsis.

## Solution Assessment

Unclear as proposed: a local dist patch is not maintainable, and adding a config opt-out changes the public config contract. The maintainable fix should be an owner-approved guard removal, narrowing, logging change, or documented config surface with regression tests.
