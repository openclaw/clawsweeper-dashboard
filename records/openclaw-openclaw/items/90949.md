## Label Justifications

- `P2`: The report describes a real channel outage with lost inbound/outbound Weixin messages, but the affected runtime is an external plugin rather than core.
- `impact:message-loss`: The issue is about Weixin messages and cron deliveries silently failing while the channel appears healthy.

## Summary

Close as ClawHub/plugin-scoped: the reported iLink session refresh, session-guard pause, and Weixin account monitoring behavior belongs to the external `@tencent-weixin/openclaw-weixin` channel plugin, and current OpenClaw core already exposes generic status/probe/runtime fields that a channel plugin can use to surface `lastError`, `healthState`, or reconnect state.

## What This Changes

The issue reports that the external openclaw-weixin channel can remain shown as healthy after iLink session expiry while message delivery is paused until QR re-login.

## Best Possible Solution

Keep OpenClaw core channel-agnostic and fix the external Weixin plugin to refresh or clearly report expired iLink sessions through the existing channel status/probe/runtime hooks.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: the report includes concrete repeated `errcode -14` logs and a clear expiry path, but the Weixin runtime is not in this checkout, so current OpenClaw main cannot directly reproduce the external plugin loop.

## Solution Assessment

Is this the best way to solve the issue? Yes for this repository: the narrow fix is in `@tencent-weixin/openclaw-weixin`, using existing OpenClaw channel status and runtime hooks rather than adding Weixin-specific core policy.
