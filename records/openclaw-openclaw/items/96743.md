## Label Justifications

- `P2`: This is a bounded but nontrivial feature request for operator-controlled web_fetch routing, not an urgent broken runtime regression.
- `impact:security`: The request is about centralized egress policy, auditability, and fail-closed provider-mediated fetch behavior.
- `impact:auth-provider`: The requested mode changes how the canonical tool selects and requires configured web-fetch providers.

## Summary

Keep open: current main still implements raw-first `web_fetch` with provider fallback only, and the requested provider-first/provider-only mode is a real new operator policy surface rather than an already-implemented bug fix.

## What This Changes

Add an opt-in `tools.web.fetch.mode` enum so canonical `web_fetch` can run raw-first, provider-first, or provider-only.

## Best Possible Solution

Have maintainers decide the operator policy first; if accepted, implement an opt-in raw-first default with explicit provider-first/provider-only semantics, cache-key separation, docs, schema help, and focused runtime/config tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection confirms the current state: no mode config exists, and provider execution is fallback-only after the raw fetch path or extraction failure.

## Solution Assessment

Unclear until maintainers confirm the product and security policy. If accepted, an enum is the maintainable shape because it separates execution order from whether raw fallback is allowed while preserving the current raw-first default.
