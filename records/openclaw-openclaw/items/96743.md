## Summary

Keep open: current main and v2026.6.10 still only support raw-first `web_fetch` with provider fallback, while the requested provider-first/provider-only behavior is a new security-sensitive public config policy that needs maintainer direction.

## What This Changes

The issue requests an opt-in `tools.web.fetch.mode` policy for raw-first, provider-first, or provider-only execution of the canonical `web_fetch` tool.

## Best Possible Solution

Have maintainers decide the provider-routing policy first; if accepted, implement an opt-in enum that preserves raw-first by default and updates runtime ordering, cache identity, schema/help, docs, and focused config/runtime tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config-policy request. Source inspection confirms the current state: no `tools.web.fetch.mode` exists, and provider execution is fallback-only after raw fetch failures or extraction fallback paths.

## Solution Assessment

Unclear until maintainers confirm the product and security policy. If accepted, an enum is a good shape because it separates provider call order from whether raw fallback is allowed while preserving the current default.
