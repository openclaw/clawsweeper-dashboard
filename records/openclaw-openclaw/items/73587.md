## Summary

Keep open. Current main still has the reported startup failure path: the logger reads `logging.file` directly from the active config, carries `cfg.file` into logger settings unchanged, returns non-rolling paths unchanged, and then creates the parent directory from that raw path. The latest shipped `v2026.4.29` tag still has the same logic, while the closed #73614 PR provides useful but unmerged implementation context.

## What This Changes

Expand `logging.file` values that begin with `~` before the gateway logger creates log directories, so `~/...` configs do not crash-loop startup.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading `~` before rolling-path detection and directory creation, with regression coverage for plain and rolling `logging.file` values. Keep any wider config-path audit separate unless a second concrete filesystem failure is identified.

## Reproduction Assessment

Yes. The source-level path is high confidence: `logging.file` set to `~/.openclaw/logs/gateway.log` flows raw into `path.dirname()` and `fs.mkdirSync()`, and Node treats that parent as the literal relative path `~/.openclaw/logs`.

## Solution Assessment

Yes. Reusing the existing home-expansion helper inside `resolveActiveLogFile()` is the narrowest maintainable fix because full config materialization already normalizes paths but the logger intentionally bypasses that stack during early startup.
