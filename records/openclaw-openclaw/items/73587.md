## Summary

Keep open. Current `main` and shipped `v2026.5.2` still pass `logging.file` into the file logger unchanged, so a non-rolling `~/...` path reaches directory creation literally; the linked fix PR is closed and unmerged.

## What This Changes

Fix the gateway startup crash when `logging.file` is configured with a leading `~/` path.

## Best Possible Solution

Apply the existing home-prefix helper at the logger filesystem boundary before rolling-path detection and directory creation, add regression tests for rolling and non-rolling paths, and track broader config path auditing separately.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: set non-silent file logging with `logging.file: "~/.openclaw/logs/gateway.log"`; current `main` copies the raw path, returns it unchanged for non-rolling logs, then calls `fs.mkdirSync` on the literal `~/.openclaw/logs` dirname.

## Solution Assessment

Yes. Reusing `expandHomePrefix()` or the equivalent existing home-path helper inside active log-file resolution is the narrowest maintainable fix; a broad audit of every config path should not block the crash repair.
