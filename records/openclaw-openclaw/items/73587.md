## Summary

Keep open. Current main and shipped v2026.5.2 still pass configured `logging.file` through to the file logger without expanding a leading `~`, so a non-silent file logger can reach `fs.mkdirSync` with a literal `~/.openclaw/logs` dirname; the linked fix PR is closed and unmerged.

## What This Changes

Restore leading-tilde expansion for configured `logging.file` paths so gateway startup does not crash or write to a literal `~` path.

## Best Possible Solution

Apply home-prefix expansion at the logger filesystem boundary before rolling-path detection and directory creation, with regression coverage for both rolling and non-rolling configured log files.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with non-silent file logging and `logging.file: "~/.openclaw/logs/gateway.log"`, current main preserves the raw path, returns it unchanged for non-rolling logs, and creates the literal dirname.

## Solution Assessment

Yes. Reusing the existing home-prefix expansion helper inside active log-file resolution is the narrowest maintainable fix; a broader config-path audit should be a follow-up, not a blocker for the crash repair.
