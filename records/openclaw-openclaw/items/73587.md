## Summary

Keep open. Current main and the latest shipped v2026.5.2 still pass raw `logging.file` into logger filesystem setup; the normal runtime config path normalizer expands `~/...`, but the logger’s direct config read bypasses it, and the linked fix PR #73614 is closed unmerged.

## What This Changes

Restore home-prefix expansion for `logging.file` so gateways with `~/...` log paths do not crash loop on startup.

## Best Possible Solution

Expand a leading home prefix at the logger filesystem boundary before rolling-path detection and directory creation, add focused regression coverage for non-rolling and rolling `logging.file` paths, and track any broader config-path audit separately.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable file logging with `logging.file: "~/.openclaw/logs/gateway.log"`; current main copies that value unchanged, returns it unchanged for non-rolling paths, then calls `fs.mkdirSync()` on the literal dirname.

## Solution Assessment

Yes. Reusing the existing home-prefix helper in `resolveActiveLogFile()` before `isRollingPath()` is the narrowest maintainable repair; the wider config-path audit should not block the startup crash fix.
