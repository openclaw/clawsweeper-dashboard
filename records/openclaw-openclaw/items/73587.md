## Summary

Keep open. Current `main` and the latest shipped `v2026.5.2` still pass `logging.file` through the logger unchanged before `fs.mkdirSync`, while the runtime config normalizer that expands `~/...` is bypassed by the logger’s direct config read; the linked fix PR #73614 is closed and unmerged.

## What This Changes

The issue reports a gateway startup crash when `logging.file` is configured with a leading-tilde path after upgrading to `2026.4.26`.

## Best Possible Solution

Expand a leading home prefix at the logger filesystem boundary before rolling-path detection and directory creation, add focused regression coverage for non-rolling and rolling `logging.file` paths, and track any wider config path audit separately.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable non-silent file logging with `logging.file: "~/.openclaw/logs/gateway.log"`; current `main` copies `cfg.file`, returns that non-rolling path unchanged, then creates the literal dirname.

## Solution Assessment

Yes. Reusing the existing home-prefix/path helper in `resolveActiveLogFile()` before `isRollingPath()` is the narrowest maintainable repair; a broad audit should not block the startup crash fix.
