## Summary

Keep open. Current `main` still preserves a `~/...` `logging.file` value until logger directory creation, and the only linked fix PR is closed unmerged, so the reported gateway startup crash remains actionable.

## What This Changes

Expand leading home prefixes in `logging.file` before logger filesystem setup so gateway startup does not crash-loop on `~/...` log paths.

## Best Possible Solution

Land a focused logger-boundary fix that reuses `expandHomePrefix()` before active log-file resolution, with regression tests and a changelog entry; track any broader config path audit separately.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure `logging.file` as `~/.openclaw/logs/gateway.log` with file logging enabled, then the current logger reads that value raw, returns it unchanged for non-rolling paths, and creates the literal dirname.

## Solution Assessment

Yes. Reusing the existing home-prefix helper inside `resolveActiveLogFile()` before rolling-path detection is the narrowest maintainable fix for the confirmed startup crash.
