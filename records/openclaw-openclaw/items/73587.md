## Summary

Keep open. Current `main` still has the reported startup failure path: `logging.file` is read directly from the raw logging config, non-rolling paths are returned unchanged, and `buildLogger()` creates the parent directory from the literal `~/...` path. The latest shipped `v2026.4.29` tag has the same implementation, and the linked fix PR #73614 is closed unmerged.

## What This Changes

The issue requests restoring leading-tilde expansion for configured `logging.file` paths so gateway startup does not crash while creating log directories.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading `~` before rolling-path detection and directory creation, with regression coverage for normal and rolling `logging.file` values; track any broader path-field audit separately.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: raw `logging.file` flows from `readLoggingConfig()` into `resolveSettings()`, `resolveActiveLogFile()` returns non-rolling values unchanged, and `buildLogger()` calls `mkdirSync()` on the literal `~/.openclaw/logs` parent path.

## Solution Assessment

Yes. Reusing the existing home-prefix expansion helper at the logger filesystem boundary is the narrowest maintainable fix for the crash; rewriting config loading or auditing every path-like setting is not required to stop this failure.
