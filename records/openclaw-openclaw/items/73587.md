## Summary

Keep open. Current main and the latest shipped release still pass `logging.file` from config into logger filesystem setup without expanding a leading `~`, and the related fix PR #73614 is closed and unmerged.

## What This Changes

Restore leading `~` expansion for `logging.file` so gateway startup does not crash when file logging creates the log directory.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading home prefix before rolling-path detection and directory creation, with regression tests and a changelog entry; track broader config-derived path auditing separately.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure file logging with `logging.file` set to `~/.openclaw/logs/gateway.log`, then the current logger reads the raw value, returns it unchanged for a non-rolling path, and tries to create the literal `~/.openclaw/logs` directory.

## Solution Assessment

Yes. Reusing the existing home-prefix expansion helper inside active log-file resolution is the narrowest maintainable fix for the confirmed startup crash; a broader path audit should not block this repair.
