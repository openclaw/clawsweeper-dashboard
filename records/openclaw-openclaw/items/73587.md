## Summary

Keep open. Current main still reads `logging.file` directly from raw config and passes leading-tilde paths to `fs.mkdirSync`, the latest inspected release tag has the same flow, and the linked fix PR is closed unmerged.

## What This Changes

The issue reports a gateway startup crash-loop when `logging.file` in `openclaw.json` uses a leading `~/...` path.

## Best Possible Solution

Expand leading home prefixes at the logger file boundary, add focused regression coverage for non-rolling and rolling `logging.file` paths, and track any wider config-derived path audit separately.

## Reproduction Assessment

Yes, at source level. With non-silent `logging.file: "~/.openclaw/logs/gateway.log"`, current main reads the raw config value, returns the non-rolling path unchanged, and mkdirs the literal `~/.openclaw/logs` dirname.

## Solution Assessment

Yes. Reusing the existing home-prefix helper in `resolveActiveLogFile()` before rolling-path detection and directory creation is the narrowest maintainable fix for the startup crash.

## Repair Work Prompt

Fix openclaw/openclaw#73587. Current `src/logging/logger.ts` reads `logging.file` through `readLoggingConfig()`, returns non-rolling paths unchanged in `resolveActiveLogFile()`, and mkdirs `path.dirname(activeFile)`, so `logging.file: "~/.openclaw/logs/gateway.log"` can crash gateway startup. Reuse the existing `expandHomePrefix` helper from `src/infra/home-dir.ts` at the logger filesystem boundary before rolling-path detection and directory creation. Add regression coverage for a non-rolling `~/...` log file and a rolling `~/.../openclaw-YYYY-MM-DD.log` input, using the smallest test seam needed. Do not fold the broader config path audit into this PR except for minimal helper/test access required by the logger fix. Add a user-facing `CHANGELOG.md` fix entry.
