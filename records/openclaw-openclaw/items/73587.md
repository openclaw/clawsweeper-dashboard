## Summary

Keep open. Current main and the latest release still leave a `~/...` `logging.file` value unexpanded before logger startup directory creation, so the reported gateway crash remains actionable.

## What This Changes

Restore leading-tilde expansion for `logging.file` so the gateway can create and write the configured log directory without crashing.

## Best Possible Solution

Land a focused logger fix that expands a leading home prefix before log-file rolling detection, directory creation, and any active configured-log-file reads, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `logging.file: "~/.openclaw/logs/gateway.log"` flows from direct config parsing into `fs.mkdirSync(path.dirname(activeFile))` without expansion.

## Solution Assessment

Yes. Reusing the existing `expandHomePrefix()` or equivalent shared path helper at the logging boundary is the narrowest maintainable fix; broader config-path auditing should be tracked separately unless tests prove the same failure mode.
