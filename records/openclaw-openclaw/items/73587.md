## Summary

Keep open. Current main and latest release still let a leading-tilde `logging.file` path reach logger directory creation unchanged, while existing config normalization shows those path-ish fields are intended to expand.

## What This Changes

The issue asks OpenClaw to expand leading-tilde `logging.file` paths before the gateway logger creates the log directory, preventing startup crash loops.

## Best Possible Solution

Reuse the existing home-prefix expansion helper at the active log-file resolution boundary, cover non-rolling and rolling log filenames, and track broader config path hardening separately if needed.

## Reproduction Assessment

Yes. A high-confidence source path exists: a non-silent leading-tilde `logging.file` value is read raw, returned unchanged for non-rolling logs, and used as the parent directory passed to `fs.mkdirSync`.

## Solution Assessment

Yes. Expanding the home prefix inside `resolveActiveLogFile()` before rolling-path detection is the narrowest maintainable repair; broad config-path auditing should be a follow-up, not part of the crash fix.
