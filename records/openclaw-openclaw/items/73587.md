## Summary

Keep open. Current `main` still has the reported gateway logger startup failure path: `logging.file` is read directly from config, non-rolling paths are returned unchanged, and startup creates the literal parent directory. The latest shipped `v2026.4.29` tag has the same implementation, and the linked fix PR #73614 is closed unmerged.

## What This Changes

The issue requests restoring leading-tilde expansion for configured log file paths so gateway startup does not crash when `logging.file` is set to `~/.openclaw/logs/gateway.log`.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading `~` before rolling-path detection and directory creation, with regression coverage for ordinary and rolling `logging.file` values; keep the larger config-path audit as a separate follow-up decision.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: set non-silent `logging.file` to `~/.openclaw/logs/gateway.log`, then `resolveSettings()` carries that raw value into `buildLogger()`, where `fs.mkdirSync(path.dirname(activeFile))` runs before any tilde expansion.

## Solution Assessment

Yes. Reusing the existing `expandHomePrefix()` helper inside `resolveActiveLogFile()` or immediately before logger filesystem calls is the narrowest maintainable fix for this startup crash; rewriting config loading is not required for the immediate bug.
