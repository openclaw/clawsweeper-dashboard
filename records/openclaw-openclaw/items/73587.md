## Summary

Keep open. Current main still has the reported startup failure path: the logger reads `logging.file` directly from the active config, keeps a leading `~` unchanged for non-rolling paths, and immediately creates the parent directory from that literal path; the latest shipped release has the same implementation and the related fix PR is closed unmerged.

## What This Changes

The issue requests restoring leading-tilde expansion for config-derived `logging.file` paths so gateway startup does not crash while creating the log directory.

## Best Possible Solution

Land a focused logger-boundary fix that expands leading `~` before rolling-path detection and directory creation, with regression coverage for ordinary and rolling `logging.file` values; keep the wider config-path audit separate unless maintainers intentionally widen the scope.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: a raw `logging.file` value from `readLoggingConfig()` flows through `resolveSettings()` into `resolveActiveLogFile()`, then `buildLogger()` calls `fs.mkdirSync()` on the literal `~` parent path.

## Solution Assessment

Yes. Reusing the existing `expandHomePrefix()` helper at the logger filesystem boundary is the narrowest maintainable fix for this crash; a repo-wide config-path rewrite is not required for the immediate regression.
