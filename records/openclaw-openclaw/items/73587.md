## Summary

Keep open. Current main still has the reported logger startup failure path: the active logging config is read directly, `logging.file` is copied without tilde expansion, and the logger creates the parent directory from that raw path. The previously linked fix PR is closed unmerged, and the latest shipped tag still shows the same raw path handling.

## What This Changes

The issue requests restoring leading-tilde expansion for config-derived `logging.file` paths so gateway startup does not crash-loop when creating the log directory.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading `~` before rolling-path detection and directory creation, with regression coverage for ordinary and rolling `logging.file` values and a changelog entry; treat a broader path-audit integration test as follow-up work.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with `logging.file` set to `~/.openclaw/logs/gateway.log`, the raw value flows through `readLoggingConfig()`, `resolveSettings()`, `resolveActiveLogFile()`, and then `fs.mkdirSync(path.dirname(activeFile))`.

## Solution Assessment

Yes. Reusing the existing `expandHomePrefix()` helper inside the logger's active-file resolution is the narrowest maintainable fix for this crash; rewriting config loading or bundling every config-path audit into the same PR is not required.

Current main reads `logging.file` directly through `readLoggingConfig()`, copies `cfg?.file` in `resolveSettings()`, and `buildLogger()` calls `fs.mkdirSync(path.dirname(resolveActiveLogFile(settings.file)))`; `resolveActiveLogFile()` currently returns non-rolling `~/...` paths unchanged.
