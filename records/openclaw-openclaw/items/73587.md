## Summary

Keep open. Current main still has the reported startup crash path: the active logging config is read directly, `logging.file` is copied into logger settings without home expansion, and the logger creates the parent directory from that raw path. The closed unmerged #73614 remains useful source context, but it did not land and the latest release tag still contains the same raw path handling.

## What This Changes

The issue requests restoring leading-tilde expansion for configured `logging.file` paths so gateway startup does not crash when the logger creates the log directory.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading `~` before rolling-path detection and directory creation, with regression coverage for plain and rolling `logging.file` values plus a changelog entry. Treat the broader config-derived path audit as follow-up unless a concrete second crash path is proven in the same patch.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: `logging.file: "~/.openclaw/logs/gateway.log"` flows from `readLoggingConfig()` through `resolveSettings()` and `resolveActiveLogFile()` into `fs.mkdirSync(path.dirname(activeFile))` without expansion.

## Solution Assessment

Yes. Reusing the existing `expandHomePrefix()` helper inside the logger's active-file resolution is the narrowest maintainable fix for this crash; rewriting config loading or bundling a broad path audit into the same repair is not required.
