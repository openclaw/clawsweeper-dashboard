## Summary

Keep open. Current main and the latest shipped v2026.4.29 still copy `logging.file` from the active config into the logger unchanged, so a leading `~` reaches log-directory creation; the related fix PR #73614 is closed and unmerged.

## What This Changes

Report that gateway startup can crash-loop when `logging.file` uses a leading `~` because the logger bypasses home-prefix expansion before creating the log directory.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading home prefix before rolling-path detection and directory creation, with regression tests and a changelog entry; track broader config-derived path auditing separately.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure `logging.file` as `~/.openclaw/logs/gateway.log` with file logging enabled, then the current logger reads the raw value, returns it unchanged from `resolveActiveLogFile()`, and creates the literal dirname.

## Solution Assessment

Yes. Reusing the existing `expandHomePrefix()` helper inside active log-file resolution is the narrowest maintainable fix for the confirmed startup crash; a broader audit should not block that repair.
