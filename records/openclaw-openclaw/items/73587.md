## Summary

Keep open. Current main still has the reported startup crash path: `logging.file` is read directly from the active config, copied into logger settings unchanged, and passed to `fs.mkdirSync(path.dirname(activeFile))` after `resolveActiveLogFile()` returns non-rolling paths verbatim. The closed unmerged #73614 remains useful source context, but it did not land, and the latest `v2026.4.29` tag still shows the same raw-path behavior.

## What This Changes

The issue reports that `logging.file` values beginning with `~` crash gateway startup because the logger creates the parent directory from the literal tilde path.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading `~` before rolling-path detection and directory creation, with regression coverage for plain and rolling `logging.file` values plus a changelog entry. Treat the broader config-derived path audit as follow-up unless another concrete crash path is proven in the same patch.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `logging.file: "~/.openclaw/logs/gateway.log"` flows from `readLoggingConfig()` through `resolveSettings()` and `resolveActiveLogFile()` into `fs.mkdirSync(path.dirname(activeFile))` without expansion.

## Solution Assessment

Yes. Reusing the existing `expandHomePrefix()` helper inside active log-file resolution is the narrowest maintainable fix for the confirmed crash; a repo-wide path audit should not block that repair unless new failing cases are added deliberately.
