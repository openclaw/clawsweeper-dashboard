## Summary

Keep open. Current main still has the reported startup crash path: the active logging config is read directly, `logging.file` is copied into logger settings unchanged, non-rolling log paths are returned unchanged, and startup eagerly calls `fs.mkdirSync` on that literal directory. The linked #73614 is useful source context, but it was closed without merging, and the latest inspected release tag still shows the same raw-path behavior.

## What This Changes

Fix a gateway startup regression where `logging.file` values beginning with `~` are used literally and can crash the logger directory creation path after upgrading to 2026.4.26.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading `~` before rolling-path detection and directory creation, with regression coverage for both plain and rolling `logging.file` values plus a user-facing changelog entry. Treat the wider config-derived path audit as follow-up unless another concrete same-boundary failure is proven in the repair PR.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure `logging.file` with a leading-tilde path, enable file logging, and the value flows from `readLoggingConfig()` through `resolveSettings()` and `resolveActiveLogFile()` into the startup `mkdirSync` call without expansion.

## Solution Assessment

Yes. Reusing `expandHomePrefix()` inside active log-file resolution is the narrowest maintainable fix for the confirmed crash; a broader audit should not block this targeted repair.
