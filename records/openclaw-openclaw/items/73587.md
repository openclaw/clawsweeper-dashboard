## Summary

Keep open. Current main and latest release still route raw `logging.file` into log-directory creation, and #73614 was closed unmerged; the generic config path normalizer does not protect the logger's direct config read.

## What This Changes

Ensure configured `logging.file` values beginning with `~` are expanded before the gateway logger creates or writes the log directory.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading home prefix before rolling-path detection and directory creation, with regression tests and a changelog entry; track any broader config-derived path audit separately.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: set `logging.file` to `~/.openclaw/logs/gateway.log` with file logging enabled, and current main reads, stores, resolves, and mkdirs the literal `~/...` path.

## Solution Assessment

Yes. Reusing `expandHomePrefix()` in active log-file resolution is the narrowest maintainable fix for the confirmed crash; the reporter's broader path audit should not block that repair.
