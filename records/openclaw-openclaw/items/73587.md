## Summary

Keep open. Current main still has the reported startup crash path: the logger reads `logging.file` directly from the active config, keeps the raw string in resolved settings, returns non-rolling paths unchanged, and creates the parent directory from that unexpanded value. The latest shipped `v2026.4.29` source still shows the same behavior, while the related fix PR #73614 is closed unmerged and remains useful implementation context.

## What This Changes

The issue requests restoring leading-tilde expansion for config-derived `logging.file` paths so Gateway startup does not crash when creating the log directory.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading home prefix before active log-file resolution and directory creation, with regression coverage for ordinary and rolling `logging.file` values plus a changelog entry. Keep the broader config-path audit separate unless the focused tests reveal another eager startup path.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with non-silent logging and `logging.file` set to `~/.openclaw/logs/gateway.log`, the raw config value flows into `fs.mkdirSync(path.dirname(activeFile))` as `~/.openclaw/logs`.

## Solution Assessment

Yes. Reusing `expandHomePrefix()` in the logger’s active-file boundary is the narrowest maintainable fix because it covers both initial startup and daily rollover without changing unrelated config loading semantics.
