## Summary

Keep open. Current main and the latest release tag still have the reported raw `logging.file` startup path, so the gateway crash remains actionable and narrow enough for a focused repair PR.

## What This Changes

The issue reports a gateway startup crash when `logging.file` is configured with a leading `~` path.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading `~` before rolling-path detection and directory creation, with regression coverage and a changelog entry; track broader config-derived path auditing separately if needed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `logging.file: "~/.openclaw/logs/gateway.log"` flows from `readLoggingConfig()` through `resolveSettings()` and `resolveActiveLogFile()` into `fs.mkdirSync(path.dirname(activeFile))` without expansion.

## Solution Assessment

Yes. Reusing `expandHomePrefix()` inside active log-file resolution is the narrowest maintainable fix for the confirmed crash; broader path normalization should be a follow-up only where separately reproduced.
