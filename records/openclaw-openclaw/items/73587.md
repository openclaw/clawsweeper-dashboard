## Summary

Keep open. Current main and shipped v2026.5.2 still pass `logging.file` into the file logger unchanged, so a non-rolling `~/...` path reaches `fs.mkdirSync` literally; the prior fix PR is closed and unmerged.

## What This Changes

The issue requests restoring home-directory expansion for `logging.file` paths that start with `~` so the gateway does not crash-loop on startup.

## Best Possible Solution

Expand leading home prefixes at the logger filesystem boundary before rolling-path detection and directory creation, add regression coverage for rolling and non-rolling configured log paths, and track the wider config path audit separately.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure non-silent file logging with `logging.file: "~/.openclaw/logs/gateway.log"`, then current main copies the raw path, returns it unchanged for non-rolling logs, and calls `fs.mkdirSync` on the literal `~/.openclaw/logs` dirname.

## Solution Assessment

Yes. Reusing the existing home-prefix helper inside active log-file resolution is the narrowest maintainable fix; the broader config-derived path audit should not block repairing the eager gateway startup crash.
