## Summary

Keep open. Current main still has the reported raw-path startup crash path, and the latest shipped v2026.4.29 tag shows the same behavior. The closed unmerged #73614 remains useful implementation context, but it did not land, so this is still a narrow actionable reliability bug.

## What This Changes

Expand leading `~` in configured `logging.file` paths before the gateway logger creates or rotates the file, preventing ENOENT startup crash loops.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading `~` before rolling-path detection and directory creation, with regression coverage for plain and rolling `logging.file` values plus a changelog entry. Keep the wider config-derived path audit separate unless additional concrete crash paths are proven in the same patch.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `logging.file: "~/.openclaw/logs/gateway.log"` flows from `readLoggingConfig()` through `resolveSettings()` and `resolveActiveLogFile()` into `fs.mkdirSync(path.dirname(activeFile))` without expansion.

## Solution Assessment

Yes. Reusing `expandHomePrefix()` inside active log-file resolution is the narrowest maintainable fix for the confirmed crash; a repo-wide path audit should not block this repair.
