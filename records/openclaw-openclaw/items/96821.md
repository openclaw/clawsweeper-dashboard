## Summary

Keep open: current main and v2026.6.10 have Tool Search, code-mode namespaces, MCP code-mode namespacing, and a harness tool-surface bridge, but they still do not provide the requested public external catalog-provider bridge for provider-owned search/describe/call delegation.

## What This Changes

The issue requests a first-class code-mode external catalog-provider bridge so providers such as Executor can expose search, describe, and call without nesting a second code executor.

## Best Possible Solution

Have maintainers decide the public code-mode catalog-provider/plugin API and security contract, then implement a narrow generic bridge only if Tool Search, MCP, and internal namespaces cannot safely express this composition.

Not applicable: this is a feature/API design request rather than a broken existing behavior report. Source and docs show adjacent catalog surfaces, but there is no current-main failure to reproduce.

## Solution Assessment

Unclear: a direct generic bridge is plausible, but maintainers still need to choose the API and policy boundary versus extending internal namespaces, Tool Search, MCP composition, or an owner plugin.
