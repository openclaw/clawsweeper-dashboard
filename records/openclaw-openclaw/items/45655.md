## Summary

Keep open. Current main still accepts configured custom-provider chat models without any output-modality contract, lists those models in the model catalog, and resolves them through the LLM streaming path; image/video generation support exists, but through separate tool/provider surfaces rather than session chat model output.

## What This Changes

The issue requests explicit rejection, diagnostics, or first-class support for Poe image/video-output models that can be configured as chat models but fail or fall back at runtime.

## Best Possible Solution

Add an explicit text-output boundary for session/chat models: configured provider rows should either declare/detect supported output modalities and reject unsupported image/video-output selections with a clear diagnostic, or true media generation should be routed through image_generate/video_generate provider/plugin surfaces with docs covering supported input/output combinations.

## Reproduction Assessment

Yes for the OpenClaw-side mismatch: current source accepts configured provider model rows, has no output metadata, lists them in the configured catalog, and resolves them as LLM stream models. The live Poe API failure was not rerun, but the source path matches the reported malformed text-centric routing risk.

## Solution Assessment

No current-main solution exists yet. The short-term rejection direction is sound, but the maintainable fix should be generic output/capability validation rather than Poe model-name hardcoding.
