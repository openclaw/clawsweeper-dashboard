## Summary

Keep open. Current main and v2026.6.10 still lack a configured chat-model output-modality contract: media generation is routed through separate image/video provider tools, while configured chat models only carry input metadata, so the Poe media-output failure/fallback problem is not implemented or safely superseded.

## What This Changes

The issue asks OpenClaw to reject or properly route Poe image/video-output models that can be configured as chat models but fail or fall back at runtime.

## Best Possible Solution

Define a configured-model output/runtime contract: keep chat model selection text-output only with clear unsupported-media diagnostics, and route supported image/video generation through image/video provider contracts and docs.

## Reproduction Assessment

Yes at source level: current main still has no configured chat-model output-modality contract, while image/video generation route through separate tool/provider settings. I did not run a live Poe credentialed Telegram reproduction, so the exact HTTP 400 and fallback symptoms remain reporter-provided.

## Solution Assessment

Unclear. A short-term explicit rejection is plausible, but maintainers need to decide whether output modality belongs in config/catalog or media-generation provider contracts before code changes.
