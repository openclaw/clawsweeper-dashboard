## Summary

Keep open. Current main still routes custom provider models through text-chat metadata with no output-modality contract or clear unsupported-media rejection, while image/video generation is handled by separate capability runtimes, so this remains product/API work rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to either support Poe image/video-output models or reject them clearly instead of accepting them as normal chat models that fail or fall back at runtime.

## Best Possible Solution

Define the media-output boundary explicitly: either reject unsupported media-output chat models with a clear diagnostic, or add a documented output-capability contract that routes them through image/video generation providers with focused tests and docs.

## Reproduction Assessment

No, not for a high-confidence live Poe reproduction; no credentialed Poe request was run. Source inspection does show a medium-confidence path: custom chat models lack output metadata while image/video generation uses separate provider runtimes.

## Solution Assessment

Unclear. A narrow unsupported-media diagnostic would improve UX, but the maintainable solution needs a product/API decision on whether output metadata belongs in chat model config or Poe media support should be a dedicated provider/plugin path.
