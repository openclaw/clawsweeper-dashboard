## Summary

Keep open. Current main still lacks a configured chat-model output-modality contract, while image and video generation remain separate provider/tool surfaces; the related input-modality guardrail issue overlaps but does not own the Poe media-output selection and fallback problem.

## What This Changes

The issue asks OpenClaw to either support Poe image/video-output models through the right media-generation runtime or reject them from chat/session model selection with a clear unsupported-output diagnostic.

## Best Possible Solution

Define one configured-model output/runtime contract, then enforce it consistently: non-text-output models should be rejected from chat/session selection with clear diagnostics unless they are routed through documented image/video generation provider contracts.

## Reproduction Assessment

Yes at source level: current main has no output-modality contract for configured chat provider models, while image/video generation is routed through separate tool/provider settings. I did not perform a live Poe credentialed run, so the exact backend 400/fallback behavior remains reporter-provided.

## Solution Assessment

Unclear as an implementation because the short-term rejection path is plausible, but maintainers first need to choose the public config/API shape for output modality and custom media-generation providers.
