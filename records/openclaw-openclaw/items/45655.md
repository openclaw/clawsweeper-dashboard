## Summary

Keep open. Current main has improved configured model input metadata, but it still has no output-modality contract for configured chat/session models and still resolves custom provider rows through the text-centric LLM stream path, so the reported Poe media-output UX remains unresolved.

## What This Changes

The issue asks OpenClaw to either support Poe image/video-output models through media generation or reject those configured chat models with a clear unsupported-output diagnostic.

## Best Possible Solution

Add a generic session-model output/capability boundary: configured provider rows should declare or discover supported output modalities, reject unsupported image/video-output selections clearly, and route real generation through image_generate/video_generate when supported.

## Reproduction Assessment

Yes for the OpenClaw-side mismatch: current source accepts configured provider model rows, catalogs them without output metadata, and resolves them through LLM streaming. The Poe-specific upstream 400 was not live-reproduced, but the runtime path still matches the reported failure class.

## Solution Assessment

No current-main solution exists yet. The maintainable fix should be a provider-generic output/capability contract and diagnostic, not Poe-specific model-name rejection.
