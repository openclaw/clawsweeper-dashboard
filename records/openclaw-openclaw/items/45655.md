## Summary

Keep open. Current main and the latest release still do not define an output-modality contract for configured chat provider models, while image/video generation are documented as separate tool/model surfaces; the related input-modality guardrail issue is useful context but does not cover this issue's Poe media-output selection and fallback problem.

## What This Changes

The issue asks OpenClaw to stop accepting or silently routing Poe image/video-output models through a text-centric runtime path, or to implement first-class media-output support.

## Best Possible Solution

Define one explicit configured-model output/runtime contract, then enforce it consistently: non-text-output models should either be rejected from chat/session selection with a clear unsupported-media-output diagnostic or routed through documented media-generation provider/tool contracts.

## Reproduction Assessment

Yes, at source level: current main has no configured-model output modality contract, while docs keep text agent-turn execution separate from image/video generation settings. I did not live-test Poe, so this is not a fully reproduced backend failure.

## Solution Assessment

Unclear. The short-term rejection path is plausible and likely safer than malformed payloads or silent fallback, but maintainers need to decide the public config/API shape before an automated fix should add or tighten model capability metadata.
