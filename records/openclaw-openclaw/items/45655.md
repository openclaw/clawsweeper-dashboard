## Summary

Keep open. Current main and the latest release still treat configured provider models as LLM/chat models with input metadata only, while image/video generation remains on separate generation-provider surfaces, so the Poe media-output failure needs a maintainer API/product decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to clarify, reject, or support Poe image/video-output models that can be configured as custom provider chat models but fail or fall back at runtime.

## Best Possible Solution

Define the media-output model contract explicitly, then either reject unsupported output modalities with a clear diagnostic or route supported image/video models through dedicated generation-provider surfaces and docs.

## Reproduction Assessment

No for the exact credentialed Poe 400/fallback behavior; this review did not run a live Poe session. Source inspection does show the current-main contract gap: configured chat models have input metadata but no output modality, and inline provider models are routed as LLM streams.

## Solution Assessment

Unclear. A short-term unsupported-media diagnostic would improve UX, but maintainers first need to decide whether custom provider config should gain output modalities or Poe media models should be configured only through dedicated image/video generation provider surfaces.
