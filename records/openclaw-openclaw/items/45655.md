## Summary

Keep open. Current main still has no configured output-modality contract or clear unsupported-media rejection for custom chat provider models; image/video generation is routed through separate capability surfaces, and the related open guardrail issue only covers model input validation.

## What This Changes

The issue requests a clear unsupported-media-output path or first-class media-output routing for custom Poe image/video models configured as chat models.

## Best Possible Solution

Define the media-output contract explicitly: either reject unsupported media-output chat models with a clear diagnostic and docs, or add a documented output-capability model contract that routes eligible providers through image/video generation runtimes.

## Reproduction Assessment

No, not with a high-confidence live Poe reproduction. Source inspection shows custom chat models still lack output metadata and media generation uses separate capability defaults, but this review did not run credentialed Nano Banana, Veo, or Sora requests.

## Solution Assessment

Unclear. A short-term unsupported-media diagnostic would improve UX, but maintainers first need to decide whether chat model config should gain output modalities or Poe media models should use only dedicated image/video generation provider surfaces.
