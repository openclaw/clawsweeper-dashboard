## Summary

Keep open: current main still treats configured provider models as session/text LLMs while media generation lives on separate image/video tool provider contracts, so the remaining fix needs maintainer API/product direction rather than cleanup closure.

## What This Changes

Request clear handling for custom Poe media-output models that are accepted as chat models but fail or fall back at runtime.

## Best Possible Solution

Define the supported boundary explicitly: keep media generation on image_generate/video_generate provider contracts, then add generic diagnostics or an output-modality contract for unsupported chat-model configs if maintainers want that surface.

## Reproduction Assessment

No, not end-to-end. The source path is reproducible: configured chat models have input-only metadata and inline provider runtime reduces usable inputs to text/image, but I did not establish a live Poe image/video request failure on current main.

## Solution Assessment

Unclear for implementation. A Poe-specific workaround would be brittle; the maintainable path is a generic output-modality/support contract, clear diagnostics, or a proper Poe media-generation provider if maintainers want first-class support.
