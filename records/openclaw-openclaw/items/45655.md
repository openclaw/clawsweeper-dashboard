## Summary

Keep open. Current main has separate image/video generation tool runtimes, but custom chat provider models still have no output-modality contract or clear unsupported-media rejection path; the related open guardrail issue only covers model input validation, not this Poe media-output boundary.

## What This Changes

Request clear support or explicit rejection for Poe image/video-output models configured as custom chat provider models, including Nano Banana image generation and Veo/Sora fallback behavior.

## Best Possible Solution

Define the media-output contract explicitly: either reject unsupported media-output chat models with a clear diagnostic and docs, or add a documented output-capability model contract that routes eligible providers through image/video generation runtimes.

## Reproduction Assessment

No, not with high-confidence live Poe reproduction because this review did not run a credentialed Poe request. Source inspection gives a medium-confidence path: custom chat models lack output metadata while media generation uses separate capability runtimes.

## Solution Assessment

Unclear. A short-term unsupported-media diagnostic would improve UX, but the maintainable fix needs a product/API decision on whether output metadata belongs in chat model config or Poe media support should use a dedicated provider/plugin capability path.
