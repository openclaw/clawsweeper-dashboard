## Summary

Keep open. Current main and the latest release still treat configured custom provider models as LLM/chat entries with input metadata only, while image/video generation lives on separate provider/tool surfaces, so the Poe media-output failure still needs maintainer API/product direction rather than cleanup closure.

## What This Changes

The issue requests explicit handling for Poe image/video-output model refs so they either work through media-generation routing or fail with a clear unsupported-output diagnostic instead of malformed chat requests or fallback.

## Best Possible Solution

Define the media-output model contract explicitly, then either reject unsupported output modalities with clear diagnostics or route supported image/video models through the dedicated generation provider surfaces and docs.

## Reproduction Assessment

No for the exact credentialed Poe 400/fallback behavior; this review did not run a live Poe session. Source inspection does show the current-main contract gap: configured provider models carry input metadata only and are routed as LLM streams, while media generation uses separate image/video provider surfaces.

## Solution Assessment

Unclear. A short-term unsupported-media diagnostic would improve UX, but maintainers first need to choose whether custom provider config should gain output modalities or whether Poe media models should be configured only through dedicated generation provider/plugin surfaces.
