## Summary

Keep open. Current main has moved media generation into dedicated `image_generate` and `video_generate` runtimes, but custom chat provider models still lack an output-modality contract or a clear unsupported-media rejection path, so the reported confusion remains unresolved product/API work.

## What This Changes

The issue asks OpenClaw to reject or correctly route Poe image/video-output models instead of accepting them as custom chat models that fail or fall back at runtime.

## Best Possible Solution

Define the media-output boundary explicitly: either reject unsupported media-output chat models with a clear diagnostic and docs, or add a documented output-capability contract that routes them through image/video generation providers.

## Reproduction Assessment

No, not for a high-confidence live Poe reproduction because no credentialed Poe request was run. Source inspection gives a medium-confidence path: custom chat model config lacks output metadata while media generation uses separate image/video capability runtimes.

## Solution Assessment

Unclear. A short-term unsupported-media diagnostic would improve UX, but the maintainable fix needs a product/API decision on whether output metadata belongs in chat model config or Poe media support should be a dedicated provider/plugin path.
