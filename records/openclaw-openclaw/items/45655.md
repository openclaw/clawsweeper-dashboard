## Summary

Keep open. Current main and v2026.6.10 still do not define a configured chat-model output-modality contract; media generation is handled by separate `image_generate`/`video_generate` provider-tool paths, so the Poe image/video-output confusion is only partially addressed by newer input-modality schema work.

## What This Changes

Clarify or fix OpenClaw handling of Poe/custom-provider image and video-output models that can be configured for chat but fail or fall back at runtime.

## Best Possible Solution

Define one explicit product contract: either reject configured chat models that advertise or are known to require non-text output, with clear diagnostics and docs, or promote media-output handling through generation-provider capabilities and output-modality metadata with upgrade-safe validation.

## Reproduction Assessment

No, not for the exact Poe HTTP 400 and Telegram fallback path because this review did not run a live Poe setup. Source inspection does reproduce the structural gap: configured chat models carry input metadata only, while media output is routed through separate generation tools.

## Solution Assessment

Unclear. Explicit rejection is a good short-term mitigation, but the best fix needs maintainer direction on whether output modality belongs in model config/catalog metadata or whether these Poe models should be supported only through media-generation provider capabilities.
