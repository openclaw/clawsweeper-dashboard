## Summary

Keep open. Current main and the latest release still do not define a configured chat-model output-modality contract: configured provider models carry input metadata, the LLM runtime remains text/image-input oriented, and image/video generation is routed through separate provider/tool settings, so the Poe media-output selection and fallback problem is not implemented or safely superseded.

## What This Changes

The issue asks OpenClaw to clearly reject or correctly route Poe custom-provider image/video-output models that can be configured as chat models but fail or fall back at runtime.

## Best Possible Solution

Define one configured-model output/runtime contract, then enforce it consistently: route media generation through image/video provider contracts and reject non-text-output chat selections with clear diagnostics until first-class media-output chat runtime support exists.

## Reproduction Assessment

Yes at source level: current main has no configured chat-model output-modality contract, while image and video generation route through separate tool/provider settings. I did not run a live Poe credentialed test, so the exact HTTP 400/fallback symptoms remain reporter-provided.

## Solution Assessment

Unclear until maintainers choose the product contract. A short-term explicit rejection is plausible, but the maintainable fix must decide where output modality lives and how it interacts with media-generation providers and existing configs.
