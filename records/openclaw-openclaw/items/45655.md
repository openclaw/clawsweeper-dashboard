## Summary

Keep open. Current main and the latest release still model custom provider entries as chat/LLM input metadata, route configured provider models into the LLM stream path, and keep image/video generation on separate capability surfaces, so the Poe media-output behavior still needs maintainer API/product direction rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to either reject unsupported Poe image/video-output models clearly or support routing them through image/video generation instead of treating them as normal chat LLMs.

## Best Possible Solution

Define the media-output model contract explicitly, then either reject unsupported output modalities with a clear diagnostic or route supported image/video models through the dedicated generation provider surfaces and docs.

## Reproduction Assessment

No for the exact credentialed Poe 400/fallback behavior; this review did not run a live Poe session. Source inspection does show the current-main contract gap: configured chat models have input metadata but no output modality, and inline provider models are routed as LLM streams.

## Solution Assessment

Unclear. A short-term unsupported-media diagnostic would improve UX, but maintainers first need to decide whether custom provider config should gain output modalities or Poe media models should be configured only through dedicated image/video generation provider surfaces.
