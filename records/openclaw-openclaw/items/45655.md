## Summary

Keep open. Current main has dedicated image/video generation provider paths and accepts richer input metadata in some catalog surfaces, but custom chat-provider models still have no output-modality contract or clear unsupported-media rejection for Poe image/video-output models.

## What This Changes

The issue asks OpenClaw to either support Poe media-output models through correct image/video routing or reject them clearly when registered as chat models.

## Best Possible Solution

Define the media-output boundary explicitly, either by rejecting unsupported media-output chat models with a clear diagnostic or by adding a documented output-capability contract that routes such models through media-generation providers.

## Reproduction Assessment

No, not for the exact live Poe failure; no credentialed Poe request was run. Source inspection gives a medium-confidence path showing custom chat models lack output metadata while image/video generation uses separate provider runtimes.

## Solution Assessment

Unclear. A narrow unsupported-media diagnostic would improve UX, but the maintainable solution needs a product/API decision on whether output metadata belongs in chat model config or Poe media support should be a dedicated provider plugin.
