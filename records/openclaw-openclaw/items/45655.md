## Summary

Keep open. Current main has first-class image and video generation runtimes, but custom chat provider models still do not have an output-modality contract or a clear unsupported-media rejection path for Poe media-output models.

## What This Changes

The issue asks OpenClaw to either support Poe image/video-output models through media-generation routing or reject them clearly when configured as chat/session models.

## Best Possible Solution

Define the media-output boundary explicitly: either reject unsupported media-output chat models with a clear diagnostic and docs, or add a documented output-capability contract that routes eligible models through image/video generation providers.

## Reproduction Assessment

No, not for a high-confidence live Poe reproduction because this review did not run a credentialed Poe request. Source inspection gives a medium-confidence path: custom chat models lack output metadata while media generation uses separate capability runtimes.

## Solution Assessment

Unclear. A short-term unsupported-media diagnostic would improve UX, but the maintainable fix needs a product/API decision on whether output metadata belongs in chat model config or Poe media support should be a dedicated provider/plugin path.
