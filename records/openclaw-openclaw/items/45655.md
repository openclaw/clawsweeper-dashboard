## Summary

Keep open. Current main still lacks a configured output-modality contract or generic unsupported-media rejection for custom chat provider models, and the linked guardrail issue only covers model input validation rather than Poe image/video-output routing.

## What This Changes

The issue requests clear handling for Poe media-output models that can be configured for chat but fail with malformed requests or fallback at runtime.

## Best Possible Solution

Define the media-output model contract explicitly, then either reject unsupported output modalities with a clear diagnostic or route supported image/video models through the dedicated generation runtimes and docs.

## Reproduction Assessment

No, not for the exact credentialed Poe 400/fallback behavior in this review. Source inspection does show the current-main contract gap: configured chat models have input metadata but no output modality, while image/video generation use separate capability defaults.

## Solution Assessment

Unclear. A short-term unsupported-media diagnostic would improve UX, but maintainers first need to decide whether chat model config should gain output modalities or Poe media models should be configured only through dedicated image/video generation provider surfaces.
