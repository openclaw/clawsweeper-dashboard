## Summary

Keep open. Current main and v2026.6.10 still lack a configured chat-model output-modality contract: custom provider models preserve media input metadata, the LLM runtime normalizes chat models to text/image inputs, and image/video generation is routed through separate provider/tool settings, so the Poe media-output failure/fallback problem is not implemented or safely superseded.

## What This Changes

The issue asks OpenClaw to reject or clearly diagnose Poe image/video-output models that can be configured but fail or fall back in the chat/session runtime.

## Best Possible Solution

Define one configured-model output/runtime contract, then enforce it consistently: route media generation through image/video provider contracts and reject unsupported non-text-output chat selections with clear diagnostics until first-class media-output chat runtime support exists.

## Reproduction Assessment

Yes at source level: current main still has no configured chat-model output-modality contract, while image/video generation route through separate tool/provider settings. I did not run a live Poe credentialed Telegram reproduction, so the exact HTTP 400 and fallback symptoms remain reporter-provided.

## Solution Assessment

Unclear until maintainers choose the product/API contract. A short-term explicit rejection is plausible, but the maintainable fix must decide where output modality lives and how it interacts with existing custom provider configs and media-generation providers.
