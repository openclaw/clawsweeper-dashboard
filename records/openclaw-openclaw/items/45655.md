## Summary

Keep open. Current main partially improved input-modality metadata and has dedicated image/video generation tool paths, but normal custom LLM provider models still have no output-modality contract, no Poe media-generation provider exists, and choosing validation versus first-class Poe media support needs maintainer product/API direction.

## What This Changes

Request clear handling or support for Poe image/video-output models configured as normal custom LLM models.

## Best Possible Solution

Define the supported boundary explicitly: keep media generation on image_generate/video_generate provider contracts, and add generic validation or diagnostics for normal chat-model configs that represent unsupported non-text outputs; add a Poe media provider/plugin only if maintainers want Poe generation support.

## Reproduction Assessment

Yes, at source level: current custom-provider chat models have no output-modality field and the inline LLM runtime normalizes usable inputs to text/image only. I did not establish a live Poe API reproduction against current main.

## Solution Assessment

Unclear for implementation. A Poe-specific workaround would be brittle; the maintainable fix is a generic output-modality/support contract or clear docs and diagnostics around the existing image/video generation provider paths.
