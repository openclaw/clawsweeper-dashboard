## Summary

Keep open. Current main has separate image/video generation tool paths and has partially improved modality metadata, but the reported custom `models.providers` Poe path still has no output-modality contract and the inline LLM runtime still normalizes provider model inputs to text/image only; deciding whether to reject media-output chat models or support them needs maintainer API/product direction.

## What This Changes

Request that OpenClaw either support Poe media-output models or clearly reject/diagnose them instead of accepting them as text-centric chat models that fail or fall back at runtime.

## Best Possible Solution

Define the supported boundary explicitly: keep media generation on `image_generate`/`video_generate` provider contracts, and add clear validation or diagnostics when a normal chat model config is being used for an unsupported media-output model.

## Reproduction Assessment

Yes, at source level: the custom provider chat-model path has no output modality field, no Poe generation provider exists, and inline provider models still normalize runtime inputs to text/image only. I did not perform a live Poe API reproduction.

## Solution Assessment

Unclear for implementation, but treating Poe media-output models as ordinary chat models is not the best long-term shape. The maintainable direction is either explicit unsupported-output diagnostics or a Poe media-generation plugin using the existing image/video generation contracts.
