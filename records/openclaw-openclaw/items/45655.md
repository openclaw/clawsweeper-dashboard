## Summary

Keep open. Current main has dedicated image/video generation tools, but the reported custom `models.providers.poe` chat-model path still has no output-modality contract or clear rejection path for media-output models.

## What This Changes

The issue requests explicit handling for Poe image/video-output models that can be configured as chat models but fail or fall back at runtime.

## Best Possible Solution

Define the intended product boundary first: either add an output-modality/provider-capability contract that rejects media-output chat models clearly, or implement Poe through the dedicated image/video generation provider surfaces.

## Reproduction Assessment

No for the exact live Poe failure; I did not run a credentialed Poe request. Source inspection does show current main still lacks an output-modality contract for configured chat models and routes media generation through separate provider surfaces.

## Solution Assessment

Unclear. A narrow diagnostic/rejection could be reasonable, but the maintainable path needs a maintainer decision on whether media-output metadata belongs in chat model config or only in dedicated generation providers.
