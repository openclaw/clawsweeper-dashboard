## Summary

Keep this open: current main still has all three reported gaps, and the remaining work is a product/API feature rather than conservative cleanup. Existing related work covers pieces of the problem, including the open fallback-notice implementation and the canon-guardian workaround published via ClawHub, but no current source path or canonical issue fully owns the native fallback-transition hook plus context re-injection contract.

## What This Changes

The issue requests opt-in non-verbose model-fallback notification, workspace context re-injection when continuation-skip sessions switch models, and a plugin hook event for fallback transitions.

## Best Possible Solution

Carry a single fallback-transition fact through the runner and use it to drive a default-safe notice option, continuation-skip bootstrap invalidation, and a documented additive plugin hook payload.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction of the current limitation: fallback reply notices are verbose-gated, continuation-skip has no model-change input, and the plugin hook union has no fallback event. I did not run a live multi-provider fallback, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Unclear as an exact three-part proposal because it depends on public API and UX choices. The safer direction is to introduce one prepared fallback-transition contract first, then consume it from notification, context-injection, and plugin-hook paths without rediscovering model state.
