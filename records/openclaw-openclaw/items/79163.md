## Summary

Keep open: current main still matches the three reported gaps, and the canon-guardian context in https://github.com/openclaw/openclaw/issues/75750 shows only a partial plugin workaround. Native fallback notification, targeted bootstrap refresh, and a fallback hook need a maintained API/UX contract rather than cleanup closure.

## What This Changes

The issue requests opt-in user-visible model fallback notification, workspace-context re-injection on model changes, and a plugin hook for fallback transitions.

## Best Possible Solution

Define one maintained fallback-transition contract that can drive optional user notification, targeted bootstrap refresh, and plugin observation without turning verbose diagnostics into the public API.

## Reproduction Assessment

Yes. Current source and existing regression tests show non-verbose fallback notices are suppressed, `continuation-skip` ignores model-change state, and the plugin hook surface lacks a fallback-transition event.

## Solution Assessment

Unclear. The requested three-part direction is plausible, but config naming, notification default, hook payload, and prompt-injection ordering need maintainer API/UX judgment before implementation.
