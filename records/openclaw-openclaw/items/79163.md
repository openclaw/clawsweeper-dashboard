## Summary

Keep open for maintainer product/API follow-up: current main has partial fallback observability, but the requested non-verbose user notification, fallback-triggered workspace context re-injection, and plugin fallback hook are not implemented.

## What This Changes

Request adds model fallback notification, model-change-aware workspace bootstrap re-injection under `continuation-skip`, and a plugin hook event for fallback transitions.

## Best Possible Solution

Define one additive fallback transition contract that carries selected/active model and reason into config-controlled user notification, targeted bootstrap re-injection on model/provider change, and a documented plugin hook with focused tests.

## Reproduction Assessment

Yes, source-level current behavior is clear: current tests and runner code show visible fallback notices are verbose-gated, the bootstrap skip helper has no model-change input, and the plugin hook list lacks a fallback transition event. I did not run a live multi-provider fallback session.

## Solution Assessment

Unclear until maintainers choose the product contract. The requested direction is plausible, but notification defaults, config naming, hook payload, and automatic versus opt-in re-injection need design approval before a narrow implementation is safe.
