## Summary

Keep open: current main still matches the reported fallback visibility, targeted bootstrap refresh, and fallback-hook gaps, and the published canon-guardian workaround is useful context but not a native event/API contract.

## What This Changes

The issue requests a maintained fallback-transition contract for optional user notification, workspace-context re-injection on model change, and plugin hook observation.

## Best Possible Solution

Define a single typed fallback-transition contract that can drive configured user notices, targeted bootstrap refresh, and plugin observation without making verbose diagnostics the public API.

## Reproduction Assessment

Yes. Source inspection and existing regression tests show verbose-off fallback notices stay hidden, continuation-skip ignores model-change state, and the plugin hook list has no fallback-transition event.

## Solution Assessment

Unclear. The requested direction is plausible, but config naming/defaults, hook payload, same-turn versus next-turn re-injection, and user-visible notice policy need maintainer API/UX judgment first.
