## Summary

Keep open: current main and v2026.6.9 still have the retained session write-lock abort cleanup path, and the focused closing fix is https://github.com/openclaw/openclaw/pull/96100, which remains open and unmerged. The `released=0` signal is not heap-release proof, and the earlier abortable-closure heap leak appears covered by https://github.com/openclaw/openclaw/pull/75008, so the remaining work is to land the lock fix and verify any residual heap growth separately.

## What This Changes

The issue reports that embedded run abort cleanup can leave retained heap and session write locks behind after stuck-session recovery.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/96100 or an equivalent narrow retained-lock abort cleanup fix, then verify that abort cleanup releases the session lock before the watchdog window and separately confirm any remaining heap-growth path.

## Reproduction Assessment

Yes at source level for the session-lock failure: current main and v2026.6.9 can return early from retained-lock cleanup when `waitForRetainedLockIdle()` is false, then cleanup can receive a no-op lock. I did not reproduce the live multi-channel heap growth in this read-only review, and `released=0` is not itself heap-release evidence.

## Solution Assessment

Yes for keeping this issue open and using the linked PR as the active fix path. The narrowest maintainable solution is in the retained session-lock cleanup path, with post-fix runtime proof for lock release and a separate memory timeline if heap growth persists.
