## Summary

Keep open: the fix direction is right and the real-behavior proof is usable, but the current patch can mix an unscoped first picker result with active-agent-scoped load-more requests, so multi-agent users can skip active-agent sessions while paginating.

## What This Changes

This PR adds active-agent `agentId` to Control UI chat session picker `sessions.list` calls, filters picker popover rows through the existing session option projection, and adds focused chat view tests.

## Best Possible Solution

Keep the active-agent scoping, but make the picker initialize and paginate from one consistently scoped result set before merge.

## Reproduction Assessment

Yes. Current main source omits `agentId` and renders raw picker rows, and the remaining PR issue is source-reproducible from the fallback to unscoped `state.sessionsResult` before scoped load-more.

## Solution Assessment

No, not yet. The scoping/filtering approach is maintainable, but pagination must use a consistent scoped result before the PR is the safest fix.

Full review comments:

- **[P2] Keep picker pagination on a scoped result:** `ui/src/ui/chat/session-controls.ts:240`
  - body: Adding `agentId` here scopes every picker page request, but the unsearched picker can still display fallback `state.sessionsResult` from an earlier all-agent load. When Load more uses that all-agent `nextOffset` with the new active-agent request, active-agent rows before that offset can be skipped or the next page can come back empty even though visible active-agent rows remain. Seed the picker with an agent-scoped first page, or otherwise make the result and offset come from the same scoped list before paginating.
