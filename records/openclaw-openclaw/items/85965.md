## Summary

Keep this PR open: the active-agent scoping direction is correct and the updated terminal proof is acceptable, but the new popover projection can inject the active session into searched or empty picker results, so the patch needs a small repair before merge.

## What This Changes

The PR adds active-agent scoping to Control UI chat session picker `sessions.list` requests, reuses filtered session option projection for popover rows, and updates chat view tests.

## Best Possible Solution

Keep the agent-scoped request, but make the picker popover apply the active-agent/cron/subagent eligibility filter without adding a missing current-session fallback for searched or paginated result sets.

## Reproduction Assessment

Yes. Source inspection shows current main omits `agentId` from picker `sessions.list` calls and renders raw response rows; the blocking PR issue is also source-reproducible because the reused grouped selector always appends the current session when it is absent.

## Solution Assessment

No, not yet. Passing `agentId` is the right code direction, but the popover should not reuse a helper that synthesizes a missing selected session for active search/no-result responses.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve search result boundaries in picker rows:** `ui/src/ui/chat/session-controls.ts:429`
  - body: `resolveSessionOptionGroups` always appends `state.sessionKey` when the response does not contain it. With this call, a picker search that returns zero rows, or rows that do not include the active session, will still render the active session and can show counts like `1 / 0`; before this PR the popover only displayed rows returned by the search. Please filter returned rows through the active-agent/cron/subagent rules without synthesizing a missing current session for searched or paginated picker results.
  - confidence: 0.86
