## Summary

Keep open. The PR has the right plugin-local direction and supplied real behavior proof, but it registers the global rejection handler before throw-prone OTLP exporter setup, so a startup failure can leave a stale diagnostics-otel handler active for a service that never started.

## What This Changes

The branch adds a diagnostics-otel runtime-env unhandled-rejection handler for nested OTLPExporterError values and service tests for matching, warning, and removal on stop.

## Best Possible Solution

Keep the diagnostics-otel-owned handler approach, but make its lifetime atomic with successful startup and add regression coverage for constructor/startup failure cleanup.

## Reproduction Assessment

Yes. The original crash is source-reproducible on current main through diagnostics-otel OTLP exporters plus the core fatal fallback, and the PR finding is source-reproducible because invalid OTLP URLs can throw after handler registration while the service manager does not call stop for failed starts.

## Solution Assessment

No. The plugin-local handler is the right narrow fix, but it should be registered after throw-prone exporter setup succeeds or cleaned up on every startup failure path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clean up the handler on startup failure:** `extensions/diagnostics-otel/src/service.ts:673`
  - body: The handler is registered before endpoint/exporter setup, but OpenTelemetry parses user-provided OTLP URLs in the exporter constructor and throws for invalid URLs. Since `startPluginServices` only records a service after `start` resolves, a startup throw here never gets a later `stop`, leaving a diagnostics-otel rejection handler active for a service that did not start. Register after successful setup or wrap startup so the unregister callback runs on every failure path.
  - confidence: 0.88
