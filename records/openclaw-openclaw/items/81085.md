## Summary

Keep open. The PR is the right plugin-owned fix for the linked diagnostics-otel crash and has useful proof, but the diff installs a process-global handler before throw-prone exporter setup; a start failure can be caught without later calling stop, leaving a stale handler active.

## What This Changes

The PR adds a diagnostics-otel unhandled-rejection handler for nested OTLPExporterError values while the service is active, unregisters it on stop/restart, and adds regression coverage.

## Best Possible Solution

Land the plugin-local OTLPExporterError suppression after fixing start-failure cleanup so the handler lifecycle exactly follows an active diagnostics-otel service, then close the linked crash issue after merge.

## Reproduction Assessment

Yes, source-reproducible: current main creates diagnostics-otel OTLP exporters, checks plugin handlers before fatal fallback, and does not classify OTLPExporterError as transient. The new review finding is also source-reproducible with a malformed OTLP endpoint because exporter construction can throw after the PR registers the handler.

## Solution Assessment

Mostly yes: a diagnostics-otel-owned runtime-env handler is the narrowest maintainable fix for exporter-specific failures. The implementation should also unregister or delay registration on start failure before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clean up the handler when start fails:** `extensions/diagnostics-otel/src/service.ts:673`
  - body: The handler is registered before OTLP exporter constructors run, but those constructors validate user URLs and can throw. startPluginServices catches start failures without adding the service to the running list, so stop() will not run and the process-global handler can remain active even though diagnostics-otel failed to start. Move registration after throw-prone setup succeeds or clean it up in a start-failure path.
  - confidence: 0.87
