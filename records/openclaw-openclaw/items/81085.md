## Summary

Keep open. The plugin-local fix direction and proof are good, but the patch unregisters the diagnostics-otel rejection handler before exporter shutdown, leaving a source-backed stop/restart crash path uncovered.

## What This Changes

The PR registers a diagnostics-otel-owned unhandled-rejection handler for nested OTLPExporterError values and adds regression coverage for matching and cleanup.

## Best Possible Solution

Keep the plugin-local handler, but leave it registered through log-provider and SDK shutdown, unregister it in a finally afterward, and add a shutdown-order regression test.

## Reproduction Assessment

Yes, source-reproducible: current main creates OTLP exporters in diagnostics-otel, core checks plugin handlers before fatal fallback, and OTLPExporterError is not globally transient. I did not run a live unavailable-collector crash reproduction in this read-only pass.

## Solution Assessment

No, not as patched. A diagnostics-otel-owned handler is the narrowest maintainable fix, but unregistering it before exporter shutdown leaves a plausible stop/restart path outside the handler window.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep the OTLP handler registered through shutdown:** `extensions/diagnostics-otel/src/service.ts:608`
  - body: `stopStarted()` unregisters the diagnostics-otel rejection handler before `currentLogProvider.shutdown()` and `currentSdk.shutdown()`. Those shutdown calls can flush or cancel in-flight OTLP exports, so an `OTLPExporterError` emitted during stop/restart would again miss the plugin handler and fall through to the core fatal unhandled-rejection path. Move the unregister into a `finally` after both shutdown awaits and cover that ordering in the service test.
  - confidence: 0.86
