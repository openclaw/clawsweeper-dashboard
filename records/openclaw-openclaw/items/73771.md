## Summary

Keep open. Current main still lacks agent-id scoping in the memory index schema/search/write paths, so the PR addresses a real isolation gap, but the PR branch is not mergeable because PR-head inspection shows blocking migration, large-index, and documented concurrency regressions in the memory-core/package surfaces.

## What This Changes

The PR branch adds agent namespace scoping to memory schema/index search-write-state paths, memory_search scope/provenance handling, QMD mcporter daemon scoping, CLI embedding provider registration, and related regression tests.

## Best Possible Solution

Land the stacked memory series in order only after rebasing, fix the package FTS migration to use Set.size, keep package/schema behavior as the source of truth, preserve bounded/yielding large-index behavior and documented non-batch concurrency semantics, add the required changelog entry, then validate shared store.path isolation across schema migration, writes/deletes, keyword search, vector search, status, session state, and QMD mcporter daemon state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main shared SQLite memory stores are not agent_id-scoped, and on the PR head an existing package FTS table without agent_id will not rebuild because Set.length is undefined, causing later FTS writes with agent_id to fail.

## Solution Assessment

No, not as-is. The owner-area direction is appropriate for the reported isolation problem, but the branch should be corrected and rebased before merge rather than accepted with migration and large-index regressions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use Set.size for the FTS rebuild check:** `packages/memory-host-sdk/src/host/memory-schema.ts:46`
  - body: `readColumns` returns a `Set`, so `existingFtsColumns.length` is always `undefined` and this condition never rebuilds old FTS tables that lack `agent_id`. Existing package-backed databases will keep the old virtual table, and later inserts that include `agent_id` will fail after upgrade.
  - confidence: 0.98
- **[P2] Preserve non-batch index concurrency:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:94-96`
  - body: This drops `remote.nonBatchConcurrency` and the Ollama non-batch default by replacing the old `configuredNonBatchConcurrency`/`providerId` inputs with `configuredConcurrency`. Current docs and tests require inline indexing to honor that setting and default Ollama to one worker when batch mode is disabled.
  - confidence: 0.93
- **[P2] Keep fallback vector scoring bounded:** `extensions/memory-core/src/memory/manager-search.ts:248-254`
  - body: The fallback vector path now loads every matching chunk with `.all()`, scores all rows, sorts the full array, and then slices. Current main uses an iterator with a bounded top-K accumulator, so this regresses large indexes by materializing all chunks and embeddings in memory.
  - confidence: 0.91
- **[P2] Preserve yielding during cache seeding:** `extensions/memory-core/src/memory/manager-sync-ops.ts:315-340`
  - body: Changing cache seeding to `.all()` plus a fully synchronous loop removes the existing `setImmediate` yield during large rebuilds. Large embedding caches can now block the event loop long enough to delay gateway health probes and other async work.
  - confidence: 0.87
