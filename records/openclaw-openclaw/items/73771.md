## Summary

Keep open. The PR is aimed at a real current-main memory isolation gap, but it is not merge-ready because the package FTS migration is broken, the branch has drifted from the current memory-host bridge contract, and it regresses existing large-index/concurrency behavior. Because this is a stacked, security-sensitive memory isolation change, it needs author/maintainer rebase and review rather than cleanup closure.

## What This Changes

The PR adds agent-id namespacing across memory index schema/search/write/delete paths, scopes memory_search provenance and QMD mcporter state, registers built-in CLI embedding providers, and adds related regression coverage.

## Best Possible Solution

Rebase the stacked memory series in order, keep packages/memory-host-sdk as the implementation source of truth with src bridges intact, fix the FTS migration to use Set.size, preserve documented non-batch concurrency and large-index streaming/yielding behavior, add the required changelog entry, and validate shared store.path isolation across schema migration, writes/deletes, keyword search, vector search, status, session state, and QMD mcporter state.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists from source inspection: on current main, two agents sharing a memory SQLite store can query source-filtered but not agent-filtered index rows; on the PR head, an existing package FTS table without agent_id will not rebuild because Set.length is undefined, causing later FTS writes with agent_id to fail.

## Solution Assessment

No, not as-is. The owner-area direction is appropriate for the memory isolation problem, but the PR should be rebased and corrected so it does not split package/source bridges or regress existing concurrency and large-index behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use Set.size for the FTS rebuild check:** `packages/memory-host-sdk/src/host/memory-schema.ts:46`
  - body: `readColumns` returns a `Set`, so `existingFtsColumns.length` is always `undefined` and this condition never rebuilds old FTS tables that lack `agent_id`. Existing package-backed databases will keep the old virtual table, and later inserts that include `agent_id` will fail after upgrade.
  - confidence: 0.98
- **[P2] Keep the source host schema as a bridge:** `src/memory-host-sdk/host/memory-schema.ts:4`
  - body: Current main’s mirror test requires `src/memory-host-sdk/host/memory-schema.ts` to remain a thin bridge to the package-owned implementation. Replacing it with a second implementation copy will fail that contract after rebase and risks package/source drift.
  - confidence: 0.94
- **[P2] Preserve non-batch index concurrency:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:94-96`
  - body: This drops `remote.nonBatchConcurrency` and the Ollama non-batch default by replacing the old `configuredNonBatchConcurrency`/`providerId` inputs with batch concurrency. Current docs and tests require inline indexing to honor that setting and default Ollama to one worker when batch mode is disabled.
  - confidence: 0.93
- **[P2] Keep fallback vector scoring bounded:** `extensions/memory-core/src/memory/manager-search.ts:248-254`
  - body: The fallback vector path now loads every matching chunk with `.all()`, scores all rows, sorts the full array, and then slices. Current main uses an iterator with a bounded top-K accumulator, so this regresses large indexes by materializing all chunks and embeddings in memory.
  - confidence: 0.91
- **[P2] Preserve yielding during cache seeding:** `extensions/memory-core/src/memory/manager-sync-ops.ts:315-340`
  - body: Changing cache seeding to `.all()` plus a fully synchronous loop removes the existing `setImmediate` yield during large rebuilds. Large embedding caches can now block the event loop long enough to delay gateway health probes and other async work.
  - confidence: 0.87
