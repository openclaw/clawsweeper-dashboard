## Summary

Keep open. Current main still contains the routing shape this PR is trying to fix, but the PR is not merge-ready because its new conversation lookup still skips real Telegram requester targets such as `telegram:<chatId>` before falling back to the topic `threadId`, so multi-binding completion delivery can still select no bound route or the wrong fallback.

## What This Changes

The PR changes subagent completion announce routing to preserve Telegram DM-topic thread IDs, adds Telegram sendMessage thread logging, and adds regression coverage for direct and queued announce paths.

## Best Possible Solution

Rebase onto current main, derive the requester conversation id with the channel-aware Telegram target parser so `telegram:<chatId>` maps to the chat conversation while `threadId` remains a delivery modifier, keep the requester-captured thread id ahead of stale session entry data, and add focused regression coverage for Telegram-prefixed targets with ambiguous bindings.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: Telegram inbound context records `to: "telegram:<chatId>"` plus `threadId`, `sessions_spawn` stores both in `requesterOrigin`, and the PR's lookup still skips the prefixed target before falling back to the topic id for bound-route matching.

## Solution Assessment

No as-is. Preferring a chat-level route before topic `threadId` is the right direction, but the implementation needs the channel-aware Telegram parser and an effective-thread logging adjustment before it is the narrow maintainable fix.
