## Summary

Keep open. The PR targets a real Telegram DM-topic announce bug, but it is draft/unmergeable and the proposed route lookup still relies on generic target parsing that does not normalize `telegram:<chatId>`, so it would not reliably fix bound-route matching on current main.

## What This Changes

The PR preserves subagent completion announce Telegram thread IDs, adds Telegram sendMessage thread logging, and adds regression tests for direct and queued announce paths.

## Best Possible Solution

Land a rebased or replacement narrow fix that derives the requester chat conversation id through channel-aware target parsing, keeps `threadId` as a delivery modifier, logs the effective Telegram thread after fallback, and adds regression plus changelog coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: Telegram records `OriginatingTo: telegram:<chatId>` plus separate `MessageThreadId`, while the current/proposed route lookup can fall back to the topic id because the generic resolver ignores `telegram:` targets.

## Solution Assessment

No as-is. Preferring the chat-level route over the topic is the right direction, but the PR needs a current-main rebase, channel-aware target parsing, effective-thread logging, and changelog coverage before it is the maintainable fix.
