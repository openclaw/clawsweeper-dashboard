## Summary

Keep open: the Telegram DM-topic announce bug is still source-reproducible on current main, and this PR is useful but incomplete because its route lookup still misses `telegram:<chatId>` targets and its new logging can report a pre-fallback thread rather than the effective send thread.

## What This Changes

The PR changes subagent announce origin merging/routing priority, Telegram sendMessage logging, and regression coverage to preserve Telegram DM-topic thread IDs in subagent completion announcements.

## Best Possible Solution

Land a rebased or replacement narrow fix that derives requester chat conversation ids with channel-aware parsing, keeps `threadId` as a delivery modifier, logs the effective Telegram thread after fallback, and adds regression plus changelog coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: Telegram records `OriginatingTo` as `telegram:<chatId>` plus a separate `MessageThreadId`, while current main and the PR's generic resolver can still use the topic id for bound-route lookup.

## Solution Assessment

No as-is. Preserving the requester thread is the right direction, but the maintainable fix needs channel-aware target parsing in the current announce delivery helper, effective-thread logging after fallback, a changelog entry, and focused regression tests.

## Review Findings

  - body: Actual Telegram requester origins use `to: "telegram:<chatId>"` with the topic in `threadId`. This change calls the generic target resolver, which skips unknown colon-prefixed targets, then falls back to the topic id for bound-route matching; derive the requester conversation id through a channel-aware Telegram target parser instead.
