## Summary

Current main still has the Telegram-topic routing gap this PR is trying to address, but the submitted patch is draft, conflict/dirty against current main, and its conversation lookup still uses the generic target parser that skips `telegram:<chatId>` targets before falling back to `threadId`.

## What This Changes

The PR changes subagent completion announce routing to preserve Telegram requester thread IDs, logs Telegram sendMessage thread IDs, and adds regression coverage for announce delivery paths.

## Best Possible Solution

Land a rebased or replacement narrow fix that derives the requester conversation id with channel-aware target parsing, treats `threadId` only as a delivery modifier, logs the effective Telegram thread after fallback, and adds changelog plus focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: current main and this patch can derive the bound-route conversation id from Telegram `threadId` because `resolveConversationIdFromTargets` skips `telegram:<chatId>` targets, while Telegram parsing stores the chat id and topic id separately.

## Solution Assessment

No as-is. The direction is correct, but the patch needs a current-main rebase or replacement plus channel-aware parsing, effective-thread logging, and changelog coverage before it is the maintainable fix.
