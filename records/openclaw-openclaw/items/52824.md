## Summary

Keep open. Current main still has the subagent completion route lookup behavior this PR is trying to fix, but the PR is not merge-ready because it still uses the generic target resolver for Telegram-prefixed requester targets and would miss the chat-level binding before falling back to the topic thread id.

## What This Changes

The PR changes subagent completion announce routing to preserve Telegram DM-topic thread IDs, adds Telegram sendMessage thread logging, and adds regression coverage for direct and queued announce paths.

## Best Possible Solution

Rebase onto current main, derive the requester conversation id through the channel-aware Telegram target parser so `telegram:<chatId>` maps to the chat while `threadId` remains only a delivery modifier, preserve the requester-captured thread id over stale session entry data, add ambiguous-binding regression coverage, and include the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: Telegram inbound context records `to: "telegram:<chatId>"` plus `threadId`, `sessions_spawn` stores both in `requesterOrigin`, and the PR still skips the prefixed target before falling back to the topic id for bound-route matching.

## Solution Assessment

No as-is. Preferring a chat-level route before topic `threadId` is the right direction, but it needs the Telegram parser, effective-thread logging, and changelog coverage before it is the narrow maintainable fix.
