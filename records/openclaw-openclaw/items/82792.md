## Summary

Keep open: this is a focused fix for a real lane wait diagnostic bug, but the patch needs a small compatibility fix for legacy queued entries preserved across in-process restarts, and the external PR proof is still test-only rather than after-fix real behavior output.

## What This Changes

The PR records queued and active lane counts at enqueue time, appends those counts to lane wait warnings, and extends the command queue regression assertion.

## Best Possible Solution

Land the narrow diagnostic improvement after defaulting legacy queue-entry counters and adding coverage for that restart-preserved shape, with redacted after-fix runtime output showing the new warning fields.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enqueue a second main or session-lane task behind an active task, wait past warnAfterMs, then release the first; current main shifts the entry before logging state.queue.length.

## Solution Assessment

No, not as written. Capturing enqueue-time queued and active counts is the right narrow direction, but the dequeue warning path should tolerate legacy queued entries preserved across in-process restarts before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Add defaults for legacy queue entries:** `src/process/command-queue.ts:282-288`
  - body: OpenClaw keeps command queue state in a global singleton across SIGUSR1 restarts, and resetAllLanes() intentionally preserves already queued entries. Entries created by the previous code will not have queuedAheadAtEnqueue or activeAheadAtEnqueue, so this warning path can pass undefined to onWait and log queueAhead=undefined activeAhead=undefined after restart. Please default missing fields when dequeuing and cover the legacy queued-entry shape.
