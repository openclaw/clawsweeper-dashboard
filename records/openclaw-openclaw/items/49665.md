## Summary

Current main and shipped releases no longer have the silent lazy-load blank-page behavior: Control UI tabs use a shared lazy-view helper that records rejected imports, renders a visible error panel, and supports explicit retry, with browser regression coverage. The linked open fork-local PR is now redundant because the upstream implementation is already shipped and avoids the retry-loop concern raised on earlier duplicate PRs.

## What This Changes

The issue reports that the Control UI Channels tab can render blank when its lazy-loaded module import rejects without visible error handling or retry.

## Best Possible Solution

Keep the shipped shared lazy-view error/retry implementation and treat remaining catch-only PRs for this symptom as obsolete duplicates.

## Reproduction Assessment

No for current main: the historical failure path is source-backed in the old createLazy helper, but current main and v2026.6.6 route the Channels view through createLazyView with visible error and retry handling. I did not run a live browser repro because the read-only review could prove the current code path and shipped regression test from source.

## Solution Assessment

Yes. The shipped implementation is better than the suggested bare catch because it shows a recoverable error state, avoids automatic retry loops while hasError is set, and has focused browser coverage for rejected lazy imports.
