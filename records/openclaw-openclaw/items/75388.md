## Summary

Keep open: the latest branch fixes the original current-main doctor-warning gap for message-tool-available channel configs, but it still emits fallback-contradictory private-reply wording when the message tool is unavailable. The remaining issue is narrow and repairable on the bot branch.

## What This Changes

This PR adds a doctor warning, focused tests, and a changelog entry for configured group/channel chats that still default to private `message_tool` visible replies when the message tool is available.

## Best Possible Solution

Land the doctor advisory after making fallback-sensitive wording accurate and keeping the shared `collectChannelRouteTargets` predicate; do not change runtime delivery policy.

## Reproduction Assessment

Yes, by source inspection. Current main returns no warning for a configured channel when the message tool is available while runtime still resolves channel turns to `message_tool_only`; the PR head also exposes the remaining fallback wording mismatch in its unavailable-message-tool test.

## Solution Assessment

Yes, with a small adjustment. Moving the advisory before the unavailable-target early return is the right narrow fix, but it should not claim final replies stay private in the same config where runtime and the next warning say OpenClaw falls back to automatic delivery.
