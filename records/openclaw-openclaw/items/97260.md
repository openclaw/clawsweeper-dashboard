## Summary

The normal `agent` run guard is a useful, well-proven fix, but the PR is not merge-ready because the new deleted-agent check is placed after the existing `/new`/`/reset` fast path, which can still mutate or deliver against a deleted-agent session before the guard runs.

## What This Changes

The PR adds a shared deleted-agent session guard to the Gateway `agent` RPC, adds handler and WebSocket integration coverage, and updates a voice-wake fixture for configured `ops` sessions.

## Best Possible Solution

Move the deleted-agent check to a pre-reset, pre-dispatch point for `agent` RPC, add reset-command regression coverage, and keep the existing shared-helper semantics aligned with `chat.send` and `sessions.send`/`sessions.steer`.

## Reproduction Assessment

Yes. Source inspection shows current main's `agent` handler loads a requested session and proceeds without the shared deleted-agent helper, while sibling `chat.send` and `sessions.send`/`sessions.steer` already reject those sessions; I did not run tests because this review was read-only.

## Solution Assessment

No, not yet. Reusing the shared helper is the right fix shape, but the guard needs to run before reset handling as well as before session maintenance and model dispatch.
