- `P1`: The item targets dropped/empty replies and crash-loop-like behavior on busy real channel workflows, but the proposed fix also touches session continuity for all Claude CLI resumes.
- `merge-risk: 🚨 compatibility`: Adding `--fork-session` to default resume args changes the persisted/resumed Claude session identity that existing users or transcript tooling may depend on.

## Summary

Keep open. Current main still lacks the requested default `--fork-session` flag, but this PR is not merge-ready because the one-token default can fork queued Claude CLI turns from a stale parent session before the serialized runner re-reads the latest binding; it also needs inspectable redacted real-behavior proof and removal of the release-owned changelog edit.

## What This Changes

The PR adds `--fork-session` to the default Anthropic Claude CLI resume args, adds assertions for the default ordering, and adds a CHANGELOG entry.

## Best Possible Solution

Keep the fork behavior only if the CLI runner refreshes or recomputes the latest Claude session binding under the serialized queue, with a regression test for close-succession queued resumes and redacted Slack or CLI proof showing ordered replies and rotated session ids.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source and upstream docs support the missing default and fork semantics, but the Slack collision/recovery proof is not inspectable yet.

## Solution Assessment

No. Adding the flag is plausible, but the best fix must preserve queued turn ordering by refreshing the latest CLI binding under serialization or proving the current preparation path cannot fork stale siblings.

- **[P1] Refresh the session binding before forking queued resumes:** `extensions/anthropic/cli-backend.ts:64`
  - body: `args` is built and `{sessionId}` is replaced before `enqueueCliRun`, while callers pass a binding read before the run starts.
