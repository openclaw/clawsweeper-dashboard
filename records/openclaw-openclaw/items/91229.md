## Summary

The PR adds a Control UI postMessage listener for `openclaw-gateway-token`, applies the received gateway token, reconnects the gateway, sends an acknowledgement, and wires listener lifecycle into `app-lifecycle`.

## Best Possible Solution

Revise the PR to use a valid host boundary, require HTTPS for production origins, handle token changes during startup, add focused coverage for those paths, and then provide redacted real hosted SSO proof before merge.

  - body: `SettingsHost` is private to `app-settings.ts` on current main, so importing it here makes prod type checking fail with TS2459. Define the narrow local host shape this listener needs, or deliberately export the type from the owner module.
  - confidence: 0.98

  - body: `createSettings()` is annotated as `UiSettings` but omits required fields such as `chatShowThinking`, `chatShowToolCalls`, `splitRatio`, and nav settings, which makes `check-test-types` fail. Reuse an existing settings fixture or fill the current `UiSettings` shape.
  - confidence: 0.97
- **[P1] Require HTTPS for production SSO origins:** `ui/src/ui/gateway-sso-postmessage.ts:26-33`
  - body: The allowlist accepts `http://clusterclaw.ai` and `http://*.clusterclaw.ai` before trusting the host. Keep HTTP limited to loopback development origins so a cleartext production origin cannot inject a gateway token.
  - confidence: 0.9
- **[P2] Reconnect when tokens arrive during startup:** `ui/src/ui/gateway-sso-postmessage.ts:95-96`
  - body: The listener only reconnects when `host.connected` is already true. If the parent sends the SSO token after `connectGateway` creates the client but before hello sets `connected`, the ack can be sent while the active client still uses the stale token captured at connect start.
  - confidence: 0.9
