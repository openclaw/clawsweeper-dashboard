## Summary

Keep open: current main does not implement this SSO postMessage/ack flow, but this PR is not merge-ready because it fails the host type boundary, can acknowledge a token while the active gateway client still uses stale auth, accepts non-local HTTP trusted origins, and lacks real hosted SSO proof.

## What This Changes

The PR adds a Control UI postMessage listener that accepts a gateway SSO token from allowed parent origins, applies it to settings, reconnects the gateway, sends an acknowledgement, and adds unit coverage.

## Best Possible Solution

Revise the SSO handoff to use a valid narrow host type, require HTTPS for production trusted origins, handle token arrival during initial gateway startup, and include redacted hosted SSO proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use an exported or local host type:** `ui/src/ui/gateway-sso-postmessage.ts:1`
  - body: `SettingsHost` is private in `app-settings.ts` on current main, and CI's `check-prod-types` reports TS2459 for this import. Define a narrow local host type or deliberately export the host type before adding this module.
  - confidence: 0.97
- **[P1] Reconnect when the token changes during startup:** `ui/src/ui/gateway-sso-postmessage.ts:90-92`
  - body: `connectGateway` captures `host.settings.token` when it constructs the client and only sets `host.connected` after hello. With this condition, a parent token posted after client creation but before hello is acknowledged without reconnecting, so first hosted SSO connect can keep using stale auth.
  - confidence: 0.91
- **[P1] Require HTTPS for production SSO origins:** `ui/src/ui/gateway-sso-postmessage.ts:25-32`
  - body: `isAllowedGatewaySsoParentOrigin` allows `http://clusterclaw.ai` and HTTP subdomains before the domain check. This is a credential handoff path, so keep HTTP limited to loopback dev origins and require HTTPS for production ClusterClaw origins.
  - confidence: 0.88
