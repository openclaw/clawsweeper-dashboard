## Summary

Keep open. Current main still derives llm-task provider and model independently, checks the allowlist against that unresolved pair, and sends the same pair into the embedded runner. The shared resolver already handles aliases and explicit cross-provider refs, but that resolver is not exposed through the plugin runtime or used by llm-task.

## What This Changes

The issue requests that llm-task resolve model aliases and provider/model refs before enforcing policy and dispatching the embedded LLM run.

## Best Possible Solution

Add a small plugin-runtime model-ref resolver, backed by the same alias/provider parsing used by normal agent model selection, then use it in llm-task before thinking-policy lookup, allowedModels enforcement, and runEmbeddedPiAgent dispatch. Add focused regression coverage for built-in aliases, explicit cross-provider refs, same-provider prefixed refs, and canonical allowlist checks.

## Reproduction Assessment

Yes. A focused unit test can create the llm-task tool with an Anthropic default provider and pass model "gemini-flash" or "google/gemini-3-flash-preview"; current source deterministically passes provider "anthropic" with the raw model string to the embedded runner.

## Solution Assessment

No. The current implementation is not the best way to solve the issue because it locally splits provider/model instead of reusing canonical model-ref resolution; the maintainable fix is a narrow plugin-facing runtime seam rather than another provider-specific alias patch.
