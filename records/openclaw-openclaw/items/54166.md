## Summary

Keep this issue open. Current `main` still selects `llm-task` provider and model independently, validates `allowedModels` against that unresolved pair, and dispatches the same pair to the embedded runner, so aliases like `gemini-flash` and cross-provider refs like `google/gemini-3-flash-preview` can still run under the selected/default provider instead of the canonical provider/model ref.

## What This Changes

The issue requests that `llm-task` resolve model aliases and provider/model references before API dispatch, policy checks, and embedded runner execution.

## Best Possible Solution

Add a small plugin-facing resolver helper, or equivalent runtime seam, that resolves `llm-task` `model` and `defaultModel` through the same canonical alias/provider-model logic used by normal agent model selection. Use the resolved provider/model before thinking policy checks, `allowedModels` validation, and `runEmbeddedPiAgent`, with focused regression coverage for aliases, explicit cross-provider refs, Anthropic aliases, and canonical allowlist behavior.

## Reproduction Assessment

Yes. A focused unit test can use the existing fake `llm-task` API harness with an Anthropic/default provider and `model: "gemini-flash"` or `model: "google/gemini-3-flash-preview"`; current source shows the embedded runner would receive the selected provider plus the raw model string instead of canonical `google/gemini-3-flash-preview`.

## Solution Assessment

No, the current implementation is not the best way to solve this issue because local provider/model splitting bypasses the shared resolver. The maintainable path is a narrow runtime/SDK resolver seam used by `llm-task`, not more provider-specific alias patches or direct imports across the plugin boundary.
