## Summary

Keep open. Current main still selects an `llm-task` provider separately from the raw model override, validates that unresolved pair, and dispatches it directly to the embedded runner. That leaves aliases like `gemini-flash` and explicit cross-provider refs like `google/gemini-3-flash-preview` routed under the selected/default provider instead of canonical model-ref resolution.

## What This Changes

The issue requests that `llm-task` resolve model aliases and provider/model references before policy validation and API dispatch.

## Best Possible Solution

Add a narrow plugin-runtime model-ref resolution helper backed by the shared resolver, use it in `llm-task` for explicit `model` and `defaultModel` before thinking policy checks, `allowedModels`, and `runEmbeddedPiAgent`, then cover bare aliases, explicit cross-provider refs, Anthropic aliases, and canonical allowlist behavior with focused regression tests.
