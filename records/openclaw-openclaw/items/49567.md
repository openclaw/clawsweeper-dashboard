## Summary

Keep open: current main and v2026.6.6 still reject `bindings[].tools`, and the runtime policy pipeline has no binding-level policy layer. This remains a security-sensitive config/product decision rather than a stale cleanup item.

## What This Changes

The issue requests per-route/per-ACP binding `tools.allow`/`tools.deny` so specific channel or group bindings can restrict tools such as `sessions_spawn`.

## Best Possible Solution

Define one canonical binding-scoped tool-policy contract or explicitly decline it; if accepted, implement schema/types, matched-binding propagation, embedded and gateway enforcement, ACP behavior, docs, and regression coverage together.

## Reproduction Assessment

Yes. Source inspection shows the strict route/ACP binding schemas omit `tools`, and the runtime policy pipeline has no binding-policy step; I did not run Gateway startup because this review is read-only and the schema/enforcement paths are clear.

## Solution Assessment

No, a schema-only change is not the best solution. The maintainable fix needs a product/security decision on precedence and must enforce the matched binding policy through the route, gateway, embedded, and ACP paths.
