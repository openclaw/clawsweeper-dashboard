## Summary

Keep open. Current main still rejects `bindings[].tools` at config validation and has no route/ACP propagation or tool-policy enforcement point for binding-scoped restrictions; the related closed PR was schema-only and unmerged, and the open OpenResponses PR is a different per-request API surface.

## What This Changes

Request to add per-binding `tools.allow` and `tools.deny` policy to route and ACP `bindings[]` so channel or group bindings can restrict tools like `sessions_spawn`.

## Best Possible Solution

Add an additive binding policy contract end to end: schema/types for route and ACP bindings, route resolution and ACP configured-binding storage that preserve the matched policy, a documented enforcement step in the shared policy pipeline at a maintainer-approved precedence point, plus docs and regression coverage for a group binding that denies `sessions_spawn` while DMs remain unaffected.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `bindings[].tools` is absent from strict route/ACP schemas and generated schema, and downstream route/ACP/policy types cannot carry or enforce it even if validation accepted it.

## Solution Assessment

No schema-only change is enough. The maintainable fix needs validation, typed config, propagation through route and ACP binding materialization, policy-pipeline enforcement, docs, and tests so accepted config changes the effective tool set.
