## Summary

Keep open: current main and the latest release still have no `bindings[].tools` route/ACP policy surface, and the existing Telegram group tool policy only covers part of the reported use case. Because this would add a security-sensitive config and enforcement layer, it needs maintainer product/security review rather than cleanup close.

## What This Changes

The issue requests a new `bindings[].tools.allow/deny` policy surface for route and ACP bindings so matched channels or conversations can restrict available tools.

## Best Possible Solution

Define one canonical binding-scoped tool-policy design or explicitly decline it; if accepted, implement schema/types, matched-binding propagation, embedded and gateway enforcement, ACP behavior, docs, and regression coverage together.

## Reproduction Assessment

Yes, by source inspection: current strict route/ACP binding schemas omit `tools`, and the current tool-policy pipeline has no binding-policy step. I did not run the Gateway, but the schema and enforcement path are clear enough to confirm the missing surface.

## Solution Assessment

No, the schema-only approach from the closed PR is not the best way to solve this because it would accept config without enforcing it. The maintainable path is an end-to-end policy layer or a product decision that existing channel group policy is the supported alternative.
