## Label Justifications

- `P2`: This is a concrete access-control-related feature request with limited blast radius, not an urgent regression.
- `impact:security`: The requested binding policy controls which tools are exposed in shared channel contexts and therefore affects an authorization boundary.

## Summary

Keep open: current main still rejects `bindings[].tools` and has no route/ACP binding policy carrier or enforcement path. Because this would be an access-control surface, schema acceptance without end-to-end runtime enforcement would be unsafe.

## What This Changes

Add optional `tools.allow`/`tools.deny` policy to route and ACP bindings so tool access can vary by matched channel or group binding.

## Best Possible Solution

Define the binding-level policy contract first; if accepted, implement schema, types, help/docs, matched-binding propagation, embedded and gateway enforcement, and precedence tests together.

## Reproduction Assessment

Yes for the current limitation: source inspection shows strict route and ACP binding schemas omit `tools`, and route/tool-policy flows do not carry matched binding policy. This is a new feature request rather than a broken documented contract.

## Solution Assessment

Unclear as proposed: accepting `bindings[].tools` in schema alone is not the best way to solve it because it would create a policy field without enforcement. The maintainable path is a complete contract and enforcement design, or an explicit decision that existing channel group/DM policy is the supported surface.
