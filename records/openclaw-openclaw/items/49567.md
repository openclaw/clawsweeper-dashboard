## Summary

Current main still has strict route and ACP binding schemas without `tools`, and the route/ACP/tool-policy enforcement paths still have no matched-binding tool-policy payload. The issue remains necessary as the canonical tracker; the linked implementation PR was schema-only and closed unmerged, while the per-request OpenResponses PR is adjacent but does not solve binding config.

## What This Changes

The issue requests per-binding `tools.allow`/`tools.deny` restrictions for route and ACP bindings so specific tools can be denied in selected channels, groups, or conversations.

## Best Possible Solution

Define one canonical binding-scoped tool-policy contract or explicitly decline it; if accepted, implement schema/types, matched-binding propagation, embedded/gateway/ACP enforcement, docs, and regression coverage together.

## Reproduction Assessment

Yes. Source inspection on current main shows strict route/ACP binding schemas omit `tools`, and the route, gateway, embedded, and ACP binding policy paths do not receive a binding-scoped policy payload.

## Solution Assessment

No. The proposed schema shape alone is not the best solution; a maintainable fix needs a product/security decision plus end-to-end enforcement anywhere binding-scoped policy would be trusted.
