## Summary

Keep open: current main still has no `bindings[].tools` schema/type surface and no matched-binding tool-policy enforcement path. This is an access-control-sensitive config/runtime feature, not a narrow bug fix, and needs maintainer API/security judgment before implementation.

## What This Changes

Request adding optional `tools.allow`/`tools.deny` policy to route and ACP bindings so matched channels or groups can restrict tools declaratively.

## Best Possible Solution

Define the public binding policy contract first; if accepted, implement it end to end across schema, types, help/docs, matched-binding propagation, embedded and gateway enforcement, and precedence tests.

## Reproduction Assessment

Yes for the current limitation: source inspection shows strict route and ACP binding schemas omit `tools`, and the route/tool-policy pipeline has no binding-policy carrier. This reproduces the rejected config shape by code inspection, but the requested behavior is a new feature rather than a broken documented contract.

## Solution Assessment

Unclear as proposed: accepting `bindings[].tools` in schema alone is not the best solution because it would create a policy field without enforcement. The maintainable path is either to document existing channel group tool policy as the intended surface or add binding-level policy with explicit precedence and fail-closed runtime enforcement.
