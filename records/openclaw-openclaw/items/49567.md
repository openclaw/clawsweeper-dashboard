## Summary

Keep open. Current main and the latest release still reject `bindings[].tools`, and the matched route/ACP binding data is not propagated into the tool-policy enforcement paths; existing group/channel tool policy covers part of the Telegram example but does not settle the requested binding-scoped config contract.

## What This Changes

Request adding optional `tools.allow` and `tools.deny` policy to route and ACP `bindings[]` so selected conversations can narrow tool access.

## Best Possible Solution

Maintainers should either document existing group/channel tool policy as the supported answer or implement binding-scoped policy end to end across schema/types, route/ACP propagation, enforcement, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection shows the proposed binding-level `tools` shape is still outside both strict binding schemas, and the current tool-policy paths have no matched-binding policy layer.

## Solution Assessment

No for a schema-only change. The best solution needs a product/security decision first, then either documentation for the existing group/channel policy path or complete binding-scoped enforcement.
