## Summary

Keep open: current main and v2026.5.22 still reject `bindings[].tools` and do not propagate a matched binding policy into tool enforcement. Existing channel group tool policy can cover some Telegram group cases, but it does not satisfy the requested route/ACP binding contract; the related schema-only PR was closed unmerged and the open OpenResponses PR is per-request only.

## What This Changes

The issue requests a binding-level `tools.allow`/`tools.deny` config surface for route and ACP bindings so matched channel or group traffic can restrict tools declaratively.

## Best Possible Solution

Decide the binding-policy contract and precedence first, then implement schema/types, matched route and ACP propagation, shared tool-policy enforcement, docs, and security regression coverage together.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.5.22` still omit `bindings[].tools` from the strict route/ACP schemas, and the adjacent validation tests assert unexpected binding keys are rejected.

## Solution Assessment

Unclear as proposed. A schema-only `tools` field is not sufficient because it would validate policy that runtime does not enforce; the maintainable path needs an explicit contract, precedence, and enforcement path.
