## Summary

Keep open: current main and the latest release still reject `bindings[].tools`, and route/ACP bindings still do not carry a matched binding-level tool policy into the tool filtering path. This remains a valid security-sensitive feature request, but it needs maintainer product/security review because schema acceptance without runtime enforcement would create false assurance.

## What This Changes

The issue requests optional per-binding tool allow/deny policy for route and ACP bindings so specific tools can be restricted by channel or group.

## Best Possible Solution

Define the binding-level policy contract and precedence first; if accepted, implement schema/types, matched-binding propagation for route and ACP, runtime enforcement across embedded and gateway tool surfaces, docs, and security regression tests together.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.5.22` both reject `bindings[].tools`, and the downstream route/ACP and tool-policy paths have no binding-level policy carrier or enforcement step.

## Solution Assessment

Unclear as proposed. Accepting `bindings[].tools` in schema alone is not the best solution because it would create a policy field without enforcement; the maintainable path needs a complete contract and policy-precedence decision.
