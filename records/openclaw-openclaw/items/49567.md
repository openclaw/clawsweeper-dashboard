## Summary

Keep open: current main and the latest release still do not support `bindings[].tools`, and the remaining work is a security-sensitive policy contract rather than a stale or already-implemented request.

## What This Changes

The issue requests optional per-binding `tools.allow`/`tools.deny` policy in route and ACP bindings so matched channel/group contexts can restrict specific tools.

## Best Possible Solution

Define the binding-level policy contract and precedence first; if accepted, implement schema/types, matched route and ACP policy propagation, embedded and gateway enforcement, docs, and security regression tests together.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.5.22` both reject `bindings[].tools`, and the downstream route/ACP plus tool-policy paths have no binding-level policy carrier or enforcement step.

## Solution Assessment

Unclear as proposed. Accepting `bindings[].tools` in schema alone is not the best solution because it would create a policy field without enforcement; the maintainable path needs a complete contract and policy-precedence decision.
