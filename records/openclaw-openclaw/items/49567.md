## Summary

Keep open. Current main and the latest release still reject `bindings[].tools`, and the active tool-policy paths do not receive matched route/ACP binding policy; existing group/channel policy covers part of the Telegram example but does not settle the requested binding-scoped config contract.

## What This Changes

Request to allow route and ACP bindings to declare per-binding `tools.allow` and `tools.deny` restrictions.

## Best Possible Solution

Maintainers should either document existing group/channel tool policy as the supported answer or implement binding-scoped policy end to end across schema/types, route/ACP propagation, enforcement, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection shows the proposed binding-level `tools` shape is still outside both strict binding schemas on current main and `v2026.6.10`, and the current tool-policy paths have no matched-binding policy layer.

## Solution Assessment

No for a schema-only change. The maintainable solution needs a product/security decision first, then either documentation for the existing group/channel policy path or complete binding-scoped enforcement.
