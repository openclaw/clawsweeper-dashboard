## Summary

Keep open: current main and the latest release still reject `bindings[].tools`, route resolution does not carry matched binding policy, and the tool-policy enforcement paths have no binding-level step. This is a valid security-sensitive feature request, but it needs maintainer policy precedence and enforcement decisions before implementation.

## What This Changes

The issue requests per-binding `tools.allow` and `tools.deny` policy for route and ACP bindings so tool access can vary by channel, group, or ACP binding.

## Best Possible Solution

Define the binding-level policy contract first; if accepted, implement schema/types, matched-binding propagation, embedded and gateway enforcement, docs/help, and security regression tests together.

## Reproduction Assessment

Yes, source-level: current main and v2026.5.19 both have strict route/ACP binding schemas without `tools`, no binding policy carrier in the route types, and no binding-policy step in tool resolution. I did not run a live gateway startup because this read-only review can prove the limitation from the config contract and enforcement source.

## Solution Assessment

Unclear as proposed: accepting `bindings[].tools` in schema alone is not the best solution because it would create a policy field without enforcement. The maintainable path is a complete contract and enforcement design, or an explicit maintainer decision that existing group/sender policy remains the supported surface.
