## Summary

Keep open: current `main` still rejects `bindings[].tools` and has no matched-binding tool-policy enforcement path. This is a new config/runtime access-control contract, not a small bug fix, and it needs maintainer product/security judgment before implementation.

## What This Changes

The issue requests optional `tools.allow` / `tools.deny` policy on route and ACP bindings so matched channel/group bindings can narrow available tools.

## Best Possible Solution

Define the public binding policy contract first; if accepted, implement it end to end across schema/types/help/docs, matched-binding propagation, runtime filtering for embedded and gateway surfaces, and precedence tests.

## Reproduction Assessment

Yes for the exact current behavior: source inspection shows route and ACP binding schemas are strict and omit `tools`, so a `bindings[].tools` config is rejected. I did not run validation because this read-only review must not execute tests or commands that can write artifacts.

## Solution Assessment

Unclear: accepting the config shape alone is not the best solution because it would not enforce the denial. The maintainable path is either to document existing channel group policy as the intended surface or to add binding-level policy with explicit runtime precedence.
