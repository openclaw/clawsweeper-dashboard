## Summary

Keep open: current main still does not accept or apply `bindings[].tools`, even though existing channel group tool policy may cover the named Telegram group use case. Adding route/ACP binding-level tool policy is a new config/runtime contract and needs maintainer precedence decisions before implementation.

## What This Changes

Request optional `tools.allow`/`tools.deny` policy on route and ACP bindings so tool availability can vary per matched binding.

## Best Possible Solution

Decide the public contract first; if accepted, add schema/types/help/docs plus runtime filtering that threads the matched route/ACP binding policy into the existing tool-policy pipeline with explicit precedence tests.

## Reproduction Assessment

Yes for the exact reported config-validation behavior: source inspection shows both route and ACP binding schemas are strict and omit `tools`, so `bindings[].tools` is rejected. I did not execute runtime validation because `pnpm` is unavailable in this shell, but the source path is direct.

## Solution Assessment

Unclear: accepting the schema alone is not the best fix because runtime filtering and precedence must be defined. A maintainable solution would either document the existing channel group policy as the intended path or implement binding-level policy end to end.
