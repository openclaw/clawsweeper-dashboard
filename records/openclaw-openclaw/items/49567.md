## Summary

Keep open: current main and the latest release still lack a route/ACP binding-level tool-policy contract. Existing channel group tool policies partially cover the Telegram group use case, but they do not provide the requested `bindings[].tools` surface, and the related schema-only PR was closed while the open per-request `tool_deny` PR addresses a different API scope.

## What This Changes

Add binding-level `tools.allow` and `tools.deny` policy so route and ACP bindings can restrict tools for matched channel or group contexts.

## Best Possible Solution

Define the binding-policy contract and precedence first; if accepted, implement schema/types, matched route and ACP propagation, shared tool-policy enforcement, docs, and regression coverage together, while documenting existing channel group policy as the supported workaround where it is sufficient.

## Reproduction Assessment

Yes. Source inspection against current main and `v2026.5.22` shows strict route/ACP binding schemas reject `bindings[].tools`, and the routing/tool-policy paths have no binding-level policy carrier or enforcement step.

## Solution Assessment

Unclear as proposed. A schema-only `tools` field is not the best solution because it would validate policy that runtime does not enforce; the maintainable path needs an explicit precedence and enforcement contract.
