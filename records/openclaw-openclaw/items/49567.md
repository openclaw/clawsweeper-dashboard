## Summary

Keep open. Current main still rejects `bindings[].tools` in both route and ACP binding schemas and does not propagate a matched binding tool policy into route, gateway, embedded, or ACP tool enforcement; existing group/channel tool policy can cover the concrete Telegram group-vs-DM workaround, but it is a separate supported surface and does not settle the requested binding-scoped contract.

## What This Changes

The issue requests per-binding `tools.allow`/`tools.deny` controls for route and ACP bindings so channel/group-specific tool restrictions can be declared in configuration.

## Best Possible Solution

Decide whether documented group/channel policy is the supported answer for this use case; if binding-scoped policy is accepted, implement schema/types, matched-binding propagation, route/gateway/embedded/ACP enforcement, docs, and regression coverage together.

## Reproduction Assessment

Yes. Current source still rejects the proposed binding-level `tools` shape and does not carry matched binding policy into route/ACP tool enforcement.

## Solution Assessment

No, not as proposed by adding schema alone. The best fix needs a product/security decision first, then either docs pointing users to existing group/channel policy or a complete binding-scoped policy implementation across schema, propagation, enforcement, and tests.
