## Summary

Keep open: current main and the latest release still lack a binding-level tool-policy contract, and the related schema-only pull request was closed unmerged while the open per-request tool-deny pull request does not solve route or ACP binding policy.

## What This Changes

The issue requests per-binding tool allow/deny policy for route and ACP bindings so specific tools can be restricted by channel, account, peer, or group.

## Best Possible Solution

Define the binding-level policy contract and precedence first; if accepted, implement schema/types, matched route and ACP propagation, shared enforcement, docs, and security regression coverage together.

## Reproduction Assessment

Yes, at source level. Current main and v2026.5.22 reject bindings[].tools through strict route/ACP schemas, and the route plus tool-policy paths have no binding-level policy carrier or enforcement step.

## Solution Assessment

No, not as a schema-only change. The maintainable solution needs a complete public contract, precedence decision, propagation from matched bindings, and runtime enforcement so configured denials actually change tool availability.
