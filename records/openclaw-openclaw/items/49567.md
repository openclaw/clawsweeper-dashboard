## Summary

Keep open. Current main and the latest release still reject `bindings[].tools` and do not carry any matched route or ACP binding policy into tool filtering; existing group/channel tool policy covers part of the Telegram use case, but it is a separate supported surface and does not settle the requested binding-scoped config contract.

## What This Changes

Request to add optional `tools.allow`/`tools.deny` per route or ACP binding so a matched channel or group binding can narrow available tools.

## Best Possible Solution

Maintainers should either document group/channel policy as the supported answer for this use case or implement binding-scoped policy end to end across schema/types, matched-binding propagation, embedded/gateway/ACP enforcement, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection shows the proposed binding-level `tools` shape is still outside both strict binding schemas, and the current tool-policy paths have no matched-binding policy layer.

## Solution Assessment

No, not as a schema-only change. The maintainable solution needs a product/security decision first, then either documentation for the existing group/channel policy path or complete binding-scoped enforcement.
