## Summary

Keep open: current main and v2026.6.6 still do not support `bindings[].tools`, and the remaining work is a security-sensitive config/enforcement design rather than stale cleanup.

## What This Changes

The issue requests route/ACP binding-level `tools.allow` and `tools.deny` so different channel, account, or group bindings can restrict available tools declaratively.

## Best Possible Solution

Define one canonical binding-scoped tool-policy contract or explicitly decline it; if accepted, implement schema/types, matched-binding propagation, embedded and gateway enforcement, ACP behavior, docs, and regression coverage together.

## Reproduction Assessment

Yes. Source inspection shows strict current route/ACP binding schemas omit `tools`, and the runtime tool-policy pipeline has no binding-policy step; I did not run the Gateway because the schema and enforcement path are clear.

## Solution Assessment

Unclear. Binding-scoped policy is plausible, but the schema-only approach from the closed related PR is not the best fix because it would accept security config without enforcing it.
