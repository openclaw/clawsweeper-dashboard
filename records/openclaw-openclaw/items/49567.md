## Summary

Keep open: current main and the latest release still do not define or enforce `bindings[].tools`, and the request is a security-sensitive new policy layer rather than a small bug fix. The linked schema-only PR was already closed in favor of this issue, while the open per-request `tool_deny` PR is adjacent but does not solve per-route or ACP binding policy.

## What This Changes

The issue requests optional per-binding `tools.allow`/`tools.deny` policy for route and ACP bindings so tool access can vary by matched channel or group binding.

## Best Possible Solution

Define the binding-level policy contract first; if accepted, implement schema/types, matched-binding propagation, embedded and gateway enforcement, docs/help, and security regression tests together.

## Reproduction Assessment

Yes, source-level: current main and v2026.5.20 both have strict route/ACP binding schemas without `tools`, no binding policy carrier in route types, and no binding-policy step in the tool pipeline. I did not run a live gateway startup because this read-only review can prove the limitation from the config contract and enforcement source.

## Solution Assessment

Unclear as proposed: accepting `bindings[].tools` in schema alone is not the best solution because it would create policy-looking config without enforcement. The maintainable path is a complete contract and enforcement design, or an explicit maintainer decision that existing group/sender policy remains the supported surface.
