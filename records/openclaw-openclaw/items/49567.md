## Summary

Keep open: current main and the latest release still lack a binding-level tool-policy contract, and the related schema-only pull request was closed unmerged while the open per-request tool-deny work does not solve route/ACP binding policy.

## What This Changes

The issue requests optional `tools.allow` and `tools.deny` policy fields on route and ACP bindings so tool access can vary by matched channel, group, or conversation binding.

## Best Possible Solution

Define the binding-level policy contract and precedence first; if accepted, implement schema/types, matched route and ACP propagation, shared tool-policy enforcement, docs, and security regression coverage together.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.5.22` reject `bindings[].tools`, and the route/ACP plus tool-policy paths have no binding-level policy carrier or enforcement step.

## Solution Assessment

Unclear as proposed. A schema-only `tools` field is not the best solution because it would validate policy that runtime does not enforce; the maintainable path needs a complete contract and policy-precedence decision.
