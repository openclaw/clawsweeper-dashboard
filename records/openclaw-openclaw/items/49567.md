## Summary

Keep open: current main and v2026.6.6 still lack a binding-scoped `tools` schema and enforcement path, while existing Telegram group policy only covers part of the reported use case. This remains a security-sensitive config/product decision, not conservative cleanup.

## What This Changes

The issue requests optional `tools.allow` and `tools.deny` policy on route/ACP bindings so tool access can vary by matched channel, group, or conversation.

## Best Possible Solution

Define one canonical binding-scoped tool-policy contract or explicitly choose existing channel group/sender policy as the supported alternative; if accepted, implement schema/types, matched-binding propagation, enforcement, docs, and regression coverage together.

## Reproduction Assessment

Yes, by source inspection: strict route/ACP binding schemas omit `tools`, resolved routes do not carry binding policy, and the current tool-policy paths have no binding-policy step. I did not run a live Gateway repro in this read-only review.

## Solution Assessment

Unclear until maintainers decide the contract. A schema-only change is not the best solution because it would accept deny rules without enforcement; the maintainable path is end-to-end binding policy or an explicit decision to rely on existing channel group/sender policy.
