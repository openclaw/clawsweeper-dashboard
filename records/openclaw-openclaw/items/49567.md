## Summary

Keep open: current main still rejects `bindings[].tools` and has no route/ACP binding policy carrier or enforcement path. This is a security-sensitive feature request that needs maintainer API and policy precedence decisions before implementation.

## What This Changes

Request binding-level `tools.allow`/`tools.deny` so route and ACP bindings can restrict tools for specific channels, groups, or conversations.

## Best Possible Solution

Define the binding-level policy contract first; if accepted, implement schema/types, docs/help, matched-binding propagation, embedded and gateway enforcement, and precedence/security tests together.

## Reproduction Assessment

Yes for the current limitation: source inspection shows strict route and ACP binding schemas omit `tools`, and route/tool-policy flows do not carry matched binding policy. This is a new feature request rather than a broken documented contract.

## Solution Assessment

Unclear as proposed: accepting `bindings[].tools` in schema alone is not the best solution because it would create a policy field without enforcement. The maintainable path is a complete contract and enforcement design, or an explicit decision that existing group/sender policy is the supported surface.
