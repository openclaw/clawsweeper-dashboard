## Summary

Keep open: current main and v2026.6.6 still lack a binding-level `tools` policy for route/ACP bindings, while existing group/sender policy only partially covers the Telegram example. This is source-reproducible but needs maintainer product/security review because it adds a new config and authorization surface.

## What This Changes

Request adding `tools.allow`/`tools.deny` to route and ACP bindings so tool visibility can be restricted per channel or group binding.

## Best Possible Solution

Define or decline a canonical binding-scoped tool-policy design; if accepted, implement schema/types, matched-binding propagation, embedded/gateway/ACP enforcement, docs, and regression tests together.

## Reproduction Assessment

Yes. Source inspection shows strict route/ACP binding schemas omit `tools`, and the current tool-policy pipeline has no binding-policy step.

## Solution Assessment

No, the related schema-only approach is not the best fix because it would accept config without enforcing it. The maintainable solution is an end-to-end binding policy layer or an explicit decision that existing group policy is the supported alternative.
