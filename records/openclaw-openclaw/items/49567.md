## Summary

Keep open: current main still rejects binding-level `tools` config and has no matched-binding policy carrier or enforcement path. This remains a security-sensitive feature request that needs maintainer product and policy precedence decisions before implementation.

## What This Changes

The issue requests optional `tools.allow`/`tools.deny` policy on route and ACP bindings so specific tools can be restricted per channel, group, or binding match.

## Best Possible Solution

Define the binding-level policy contract first; if maintainers accept it, implement schema/types, matched-binding propagation, embedded and gateway enforcement, docs/help, and security regression tests together.

## Reproduction Assessment

Yes for the current limitation: source inspection shows strict route and ACP binding schemas omit `tools`, and route/tool-policy flows do not carry matched binding policy. This is a feature request rather than broken documented behavior.

## Solution Assessment

Unclear as proposed: accepting `bindings[].tools` in schema alone is not the best solution because it would create a policy field without enforcement. The maintainable path is a complete contract and enforcement design, or an explicit decision that existing group/sender policy is the supported surface.
