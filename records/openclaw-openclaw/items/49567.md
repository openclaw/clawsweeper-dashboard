## Summary

Keep open: current main and the latest release still reject `bindings[].tools`, and the route/ACP binding path does not carry or enforce binding-level tool policy. This remains a valid security-sensitive feature request that needs product and security precedence decisions before implementation.

## What This Changes

Request to add optional `tools.allow` and `tools.deny` policy to route and ACP bindings for per-channel or per-group tool restrictions.

## Best Possible Solution

Define the binding-level policy contract first; if accepted, implement schema/types, matched-binding propagation, embedded and gateway enforcement, docs/help, and security regression tests together.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.5.20` both reject `bindings[].tools`, and the downstream route/tool-policy path has no binding-level policy carrier or enforcement step.

## Solution Assessment

Unclear as proposed. Accepting `bindings[].tools` in schema alone is not the best solution because it would create a policy field without enforcement; the maintainable path needs a complete contract and policy-precedence decision.
