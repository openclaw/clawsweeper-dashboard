## Summary

Keep open. Current main still does not support binding-level `tools.allow`/`tools.deny` for route or ACP bindings: validation rejects the field, typed route/ACP records cannot carry it, and the tool-policy pipeline has no binding-scoped enforcement step. The closed schema-only PR #49593 did not merge, and open PR #58434 is a related but different per-request OpenResponses API change.

## What This Changes

Add optional per-binding tool policy to route and ACP bindings so matched channels, groups, or conversations can restrict allowed or denied tools.

## Best Possible Solution

Add an end-to-end additive binding policy surface: schema/types for route and ACP bindings, propagation through route resolution and ACP configured-binding records, enforcement in the shared policy pipeline, docs, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows strict route/ACP binding schemas omit `tools`, and downstream route/ACP records plus the policy pipeline have nowhere to carry or enforce binding-scoped policy.

## Solution Assessment

Unclear pending maintainer precedence policy. A schema-only change like closed #49593 is not the best fix; the maintainable solution needs validation, typed propagation, runtime enforcement, docs, and tests.
