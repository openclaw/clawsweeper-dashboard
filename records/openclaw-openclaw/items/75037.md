## Summary

Keep open. Current main still has both reported metadata-write paths that can advance `updatedAt`, and the linked fix PR #75066 is open and unmerged, so this issue should remain the tracker until that PR or an equivalent patch lands.

## What This Changes

Request to prevent compaction-count and memory-flush metadata writes from advancing session activity timestamps so legacy daily and idle resets still fire correctly.

## Best Possible Solution

Land PR #75066 or an equivalent narrow patch that applies the existing preserve-activity merge policy to compaction and memory-flush bookkeeping, with regression coverage for legacy rows missing `sessionStartedAt` and `lastInteractionAt`.

## Reproduction Assessment

Yes. The high-confidence reproduction path is a legacy session-store row missing lifecycle timestamps, followed by current compaction or memory-flush metadata persistence and then `initSessionState`/`evaluateSessionFreshness` on the next real user message.

## Solution Assessment

Yes. Extending the existing preserve-activity policy to metadata-only compaction and flush writes is the narrowest maintainable fix; changing the global legacy fallback or adding a store migration would have a wider behavior surface.
