## Summary

Keep open. Current main still has the reported activity-timestamp refresh path for compaction and memory-flush bookkeeping, and open PR #75066 specifically closes this issue but is not merged, so this issue should remain the tracker until that PR or an equivalent fix lands.

## What This Changes

Request that compaction-count and memory-flush metadata writes preserve session activity timestamps so legacy daily and idle reset freshness is not masked by bookkeeping updates.

## Best Possible Solution

Land PR #75066 or an equivalent narrow patch that removes activity timestamp refreshes from compaction bookkeeping and lets selected metadata-only `updateSessionStoreEntry` callers use the existing preserve-activity merge policy, with focused regression coverage for legacy daily and idle rows.

## Reproduction Assessment

Yes. The high-confidence path is a legacy session-store row missing lifecycle timestamps, followed by current compaction or memory-flush metadata persistence and then `initSessionState`/`evaluateSessionFreshness` on the next real user message; this review verified the source path rather than running tests because the task is read-only.

## Solution Assessment

Yes. Reusing the existing preserve-activity merge policy for compaction and flush metadata is the narrowest maintainable fix; changing the global legacy fallback would affect a wider reset surface.
