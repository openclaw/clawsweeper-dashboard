## Summary

Keep this PR open because current main still lacks `gateway.auth.scopes` and still clears device-less shared-auth scopes, so the requested behavior is not implemented. The submitted patch is not ready to merge: it is stale against the current gateway auth resolver layout, regrants scopes through `sharedAuthOk` even though that includes trusted-proxy auth, and does not update generated config-schema artifacts.

## What This Changes

This PR adds optional `gateway.auth.scopes` config/type/schema/help plumbing and re-applies configured scopes to device-less shared-auth WebSocket connections after unbound scopes are cleared.

## Best Possible Solution

Add the config surface against the current `auth-resolve.ts` and generated config-schema pipeline, then restore configured scopes only when the actual auth method is `token` or `password`. The finished change should include focused gateway auth tests proving default clearing, configured regrant, trusted-proxy non-elevation, and Tailscale non-elevation.

## Reproduction Assessment

Yes. Current-main tests and source provide a high-confidence reproduction path: connect as an operator with a valid shared token/password and no device identity, request privileged scopes, then observe `hello-ok.auth.scopes` is empty and privileged RPCs fail with `missing scope`.

## Solution Assessment

No. The feature direction may be viable, but this submitted patch is not the best current solution because it gates on `sharedAuthOk`, targets the old resolver file, and omits generated config-schema artifacts; the safer implementation is an explicit `authMethod === "token" || authMethod === "password"` gate after rebase.
