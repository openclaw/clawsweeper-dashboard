## Summary

Keep open: current main does not implement `gateway.auth.scopes`, and the PR head is stale/dirty with blocking gateway-auth and generated-config issues before it can merge.

## What This Changes

The PR adds a `gateway.auth.scopes` config field and re-applies those scopes to device-less shared-auth gateway WebSocket connections.

## Best Possible Solution

Agree the gateway auth boundary first, then implement the option on current main with an explicit token/password auth-method gate, regenerated config artifacts, docs, and focused regression coverage for token, password, trusted-proxy, and Tailscale paths.

## Reproduction Assessment

Yes, from source and tests: current main accepts a device-less operator with a valid shared token but clears `hello-ok.auth.scopes` to `[]`, so privileged calls fail with missing scope. I did not run tests because this was a read-only review.

## Solution Assessment

No. The feature direction may be useful, but the current PR is not the best mergeable fix because it gates on `sharedAuthOk`, leaves generated schema artifacts stale, and is dirty against the current resolver split.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Restrict scope regranting to token/password auth:** `src/gateway/server/ws-connection/message-handler.ts:544`
  - body: This condition uses `sharedAuthOk`, which is also true for successful trusted-proxy auth. With `gateway.auth.scopes` configured, a device-less trusted-proxy session can receive shared-secret scopes such as `operator.admin`, contrary to the PR's token/password-only contract.
  - confidence: 0.95
- **[P2] Regenerate the public config schema artifacts:** `src/config/zod-schema.ts:690`
  - body: Adding `gateway.auth.scopes` to the strict Zod schema without updating generated config schema artifacts leaves the served/public config schema stale; `src/config/schema.base.generated.ts` still lacks this field.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete gateway-auth scope-elevation risk because the configured scope grant is gated by `sharedAuthOk`, which includes trusted-proxy auth.

Concerns:

- **[medium] Trusted-proxy sessions can receive shared-secret scopes:** `src/gateway/server/ws-connection/message-handler.ts:544`
