## Summary

Keep open: the feature targets a real source-reproducible gateway scope-clearing problem, but the current PR head still has blocking auth-boundary and generated-baseline defects and lacks contributor real behavior proof.

## What This Changes

The PR adds a `gateway.auth.scopes` config/schema/help field, plumbs it through gateway auth resolution, and regrants configured scopes to device-less shared-auth WebSocket connections.

## Best Possible Solution

Agree one gateway WebSocket token-scope policy, then land a current-main implementation with explicit token/password gating, generated config baseline/docs updates, regression coverage, and redacted live gateway proof.

## Reproduction Assessment

Yes. Current main lacks `gateway.auth.scopes`, and checked-in compatibility tests assert that device-less shared-token/password operator sessions lose requested scopes and then fail a scoped RPC with `missing scope`; I did not run a live gateway in this read-only review.

## Solution Assessment

No. The feature direction may be useful, but the current PR is not the best mergeable fix because it uses a trusted-proxy-inclusive gate, omits generated config baseline work, and lacks real behavior proof.
