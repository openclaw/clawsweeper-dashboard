## Summary

Keep this PR open: current main does not implement `gateway.auth.scopes`, and the reviewed PR head still has a security-sensitive auth-boundary bug plus stale generated config schema artifacts.

## What This Changes

The PR adds `gateway.auth.scopes` to gateway config types/schema/help and re-applies configured scopes to device-less shared-auth WebSocket connections.

## Best Possible Solution

Agree the gateway auth boundary first, then land a current-main implementation with an explicit token/password auth-method gate, regenerated config artifacts, docs/changelog, and regression coverage for token, password, trusted-proxy, and Tailscale paths.

## Reproduction Assessment

Yes. Source and existing tests show current main accepts a device-less operator with a valid shared token, clears its scopes, and then fails privileged calls with a missing-scope error.

## Solution Assessment

No. The feature direction may be useful, but the reviewed diff is not the best mergeable fix because it gates on `sharedAuthOk`, which includes trusted-proxy auth, and leaves generated public config schema artifacts stale.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Restrict scope regranting to token/password auth:** `src/gateway/server/ws-connection/message-handler.ts:544`
  - body: This condition uses `sharedAuthOk`, which is also true for successful trusted-proxy auth. With `gateway.auth.scopes` configured, a device-less trusted-proxy session can receive shared-secret scopes such as `operator.admin`, contrary to the PR's token/password-only contract.
  - confidence: 0.95
- **[P2] Regenerate the public config schema artifacts:** `src/config/zod-schema.ts:690`
  - body: Adding `gateway.auth.scopes` to the strict Zod schema without updating generated config schema artifacts leaves the served/public config schema stale; `src/config/schema.base.generated.ts` still lacks this field.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete gateway-auth scope-elevation risk because configured scopes are regranted through a trusted-proxy-inclusive auth predicate.

Concerns:

- **[medium] Trusted-proxy sessions can receive shared-secret scopes:** `src/gateway/server/ws-connection/message-handler.ts:544`
  - body: The PR regrants `resolvedAuth.scopes` whenever `sharedAuthOk` is true. Current gateway auth treats successful trusted-proxy auth as shared, so configured token/password scopes can be applied to device-less trusted-proxy sessions.
  - confidence: 0.95
