## Summary

Keep this PR open. Current main still lacks the requested `gateway.auth.scopes` surface and still clears device-less shared-auth scopes, but the submitted patch is not merge-ready because it regrants scopes through a predicate that also covers trusted-proxy auth and misses generated config-schema artifacts.

## What This Changes

This PR adds optional `gateway.auth.scopes` config/type/schema/help plumbing and re-applies configured scopes to device-less shared-auth WebSocket connections after unbound scopes are cleared.

## Best Possible Solution

Add the option only after the auth boundary is agreed, implement it in the current resolver path with an explicit token/password auth-method gate, regenerate config artifacts/docs, and add focused tests for default clearing, configured regrant, trusted-proxy non-elevation, and Tailscale non-elevation.

## Reproduction Assessment

Yes. Source and existing tests provide a high-confidence reproduction path: a device-less operator connection with a valid shared token is accepted, has `hello-ok.auth.scopes` cleared to `[]`, and then privileged RPCs fail with `missing scope`; I did not run tests because this was a read-only review.

## Solution Assessment

No. The feature direction may be valid, but this patch is not the best current solution because `sharedAuthOk` includes trusted-proxy auth, the branch targets an older resolver layout, and generated config artifacts are missing; the safer path is an explicit token/password gate in the current resolver pipeline plus regression tests.
