## Label Justifications

- `P2`: The issue is a concrete performance bug in an existing model-selection command with limited blast radius rather than a core outage.
- `impact:auth-provider`: The affected behavior is provider/model catalog listing and model-choice discovery for configured providers.

## Summary

Keep open: current main still routes the auto-reply `/models` browse command through the full catalog load, and the open implementation PR at https://github.com/openclaw/openclaw/pull/84735 is the active fix path for this issue. The source path matches the reporter's timing suspicion, while the Gateway model list path already demonstrates the intended read-only fast behavior.

## What This Changes

The issue requests that `/models` use a fast read-only or cached model-catalog path instead of doing a full catalog refresh for ordinary browse/list replies.

## Best Possible Solution

Land a narrow change that makes ordinary auto-reply `/models` browse/list replies use the existing read-only catalog contract, preserves explicit `all` and provider-wildcard cases on the full refresh path, and adds regression coverage for both modes.

## Reproduction Assessment

Yes for the source-level path: current main's `/models` command calls `loadModelCatalog({ config })` without `readOnly`, and that path performs `ensureOpenClawModelsJson` before registry work. I did not reproduce the exact LiteLLM timing in a live 410-model environment.

## Solution Assessment

Yes: using the existing read-only catalog path for default browse replies is the narrowest maintainable fix, with explicit `all` and provider-wildcard views retaining the full-refresh behavior. The related TTL-cache PR can help repeated navigation but does not address the cold catalog refresh by itself.
