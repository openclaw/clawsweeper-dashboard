## What This Changes

The issue requests that Telegram/chat `/models` avoid the slow full model catalog refresh path when listing configured LiteLLM models.

## Best Possible Solution

Share the Gateway-style side-effect-free configured-view behavior with chat `/models`, using a full refresh only for explicit full-catalog browse cases and covering the many-model LiteLLM case with focused regression tests.

## Reproduction Assessment

Source-reproducible, but not live-reproduced in this read-only review. The current `/models` source path still calls the non-read-only catalog loader, and the reporter supplied timing that matches the code split between full and read-only catalog modes.

## Solution Assessment

Yes, the requested direction is the maintainable fix shape: default `/models` listing should use a read-only or Gateway-equivalent configured catalog path, while explicit `all` browse behavior should keep full discovery where needed.
