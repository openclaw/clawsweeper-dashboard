## Summary

Keep open: current main still routes the auto-reply `/models` command through the full non-read-only catalog load, while the Gateway `models.list` path already has a read-only fast path. The report is source-reproducible and not superseded by the open TTL-cache PR, which only helps repeated menu rebuilds after the cold catalog work has already happened.

## What This Changes

Speed up the Telegram/auto-reply `/models` listing path for large LiteLLM/OpenAI-compatible catalogs by avoiding full catalog refresh work for read-only browsing.

## Best Possible Solution

Make the default auto-reply `/models` browse path side-effect-free/read-only like Gateway `models.list`, reserving full catalog refresh for explicit `all` or scan-style paths and proving LiteLLM-sized catalogs no longer block the command.

## Reproduction Assessment

Yes for the source-level path: current main's `/models` command calls `loadModelCatalog({ config })` without `readOnly`, and that non-read-only path calls `ensureOpenClawModelsJson` before registry work. I did not run a live 410-model LiteLLM setup, so the exact timing is not locally reproduced.

## Solution Assessment

Yes: reusing the existing read-only catalog contract for the default browse command is narrower than adding a new setting or broad caching. The open TTL-cache PR may still help repeated taps, but it is not the complete fix for the cold path.
