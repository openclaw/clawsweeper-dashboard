## Summary

Keep open. Current main still rewrites a single Mattermost draft-preview post across phase transitions, and the linked closing PR should be reviewed or coordinated before this issue closes.

## What This Changes

Mattermost draft preview streaming should offer a mode that creates new preview posts at phase boundaries so prior preview content remains visible.

## Best Possible Solution

Land a Mattermost-owned `streaming.mode` implementation that keeps the current default, splits block-mode previews with `forceNewMessage()` at real boundaries, and aligns tests, docs, and changelog with the overlapping config work.

## Reproduction Assessment

Yes. The source path is high-confidence without a live Mattermost server: transition hooks keep `streamPostId` set and later draft updates route to `PUT /posts/{id}` instead of creating a new post.

## Solution Assessment

Yes. The narrow maintainable fix is Mattermost-local config resolution plus boundary splitting; shared core dispatch does not need to change because the overwrite is produced by the Mattermost draft stream and monitor hooks.
