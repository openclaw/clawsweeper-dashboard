## Summary

Keep open. Current main still edits a single Mattermost draft-preview post across reasoning, tool, partial, and final transitions, and open PR #75256 explicitly closes this issue while overlapping PR #73252 changes the same streaming-mode surface.

## What This Changes

The issue requests a Mattermost streaming mode that splits draft preview posts at phase boundaries so previously visible preview content remains in channel history.

## Best Possible Solution

Land a Mattermost-owned `streaming.mode` implementation that preserves the current default, validates schema/types/docs, and calls `forceNewMessage()` at real preview boundaries in coordination with PR #73252.

## Reproduction Assessment

Yes. The source path is high-confidence without a live Mattermost server: the draft stream retains `streamPostId`, the monitor phase hooks call `update(...)`, and no Mattermost monitor code calls `forceNewMessage()`.

## Solution Assessment

Yes. A Mattermost-local config resolver plus boundary controller is the narrow maintainable fix; shared core dispatch does not need to change because the overwrite behavior is created by the Mattermost draft stream and monitor hooks.
