## Summary

Keep open. Current main still rewrites one Mattermost draft-preview post across thinking, tool, partial, and final transitions, and the open closing PR should be reviewed or coordinated with the overlapping streaming-mode PR before this issue closes.

## What This Changes

The issue requests a Mattermost streaming mode that can split draft previews into separate posts at lifecycle boundaries instead of overwriting one edited post for the whole turn.

## Best Possible Solution

Land one Mattermost-owned streaming-mode implementation that keeps the current default, validates schema/types/docs, and calls `forceNewMessage()` at real draft-preview boundaries when block mode is selected.

## Reproduction Assessment

Yes. The source path is high-confidence without a live Mattermost server: a retained `streamPostId` causes edits, phase hooks call `update(...)`, and the monitor never invokes the existing `forceNewMessage()` splitter.

## Solution Assessment

Yes. A Mattermost-local config resolver plus boundary-splitting controller is the narrow maintainable fix; shared core dispatch does not need to change because the overwrite behavior is created by the Mattermost draft stream and monitor hooks.
