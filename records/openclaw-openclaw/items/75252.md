## Summary

Keep open. Current main still rewrites one Mattermost draft-preview post across preview transitions, and the linked open PR with closing syntax should be reviewed or coordinated with the overlapping streaming-mode PR before this issue closes.

## What This Changes

Add Mattermost preview streaming configuration and boundary splitting so thinking, tool status, partial, and final reply transitions do not overwrite the same draft post.

## Best Possible Solution

Land a Mattermost-owned preview streaming implementation that preserves the current default, adds guarded `forceNewMessage()` boundary splitting for block mode, and aligns schema, docs, and tests with the overlapping streaming-mode work.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: one draft stream retains `streamPostId`, transition hooks call `update(...)`, and the only Mattermost API that resets the id is not called by the monitor.

## Solution Assessment

Yes. A Mattermost-local mode resolver plus boundary controller is the narrow maintainable fix; the overwrite behavior is created by Mattermost draft-stream and monitor wiring, not by shared core dispatch.
