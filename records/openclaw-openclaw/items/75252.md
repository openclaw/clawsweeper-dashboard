## Summary

Keep open. Current main still sends all Mattermost draft-preview phase changes through one retained preview post without a boundary split, and the open closing PR plus the overlapping Mattermost streaming-mode PR should be coordinated before this issue closes.

## What This Changes

Mattermost draft-preview streaming should preserve prior preview content across reasoning, tool, partial, and final transitions instead of repeatedly editing one post.

## Best Possible Solution

Land or coordinate a Mattermost-owned `streaming.mode` implementation that preserves the current default while splitting preview posts at real phase boundaries in block mode, with schema, types, docs, and focused Mattermost tests aligned.

## Reproduction Assessment

Yes. The source-level path is high-confidence: one draft stream retains `streamPostId`, phase hooks call `update(...)`, and only `forceNewMessage()` clears that id, but the monitor never calls it.

## Solution Assessment

Yes. A Mattermost-local mode resolver plus boundary-splitting controller is the narrowest maintainable fix because the overwrite behavior is created by Mattermost draft-stream state and monitor lifecycle hooks, not shared core dispatch.
