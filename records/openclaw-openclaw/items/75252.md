## Summary

Keep open. Current main still routes Mattermost draft-preview transitions through one editable preview post with no Mattermost preview-mode gate, and open PR #75256 uses closing syntax for this issue. PR #73252 also overlaps the same config/runtime surface, so this needs PR coordination rather than cleanup closure.

## What This Changes

The issue requests a Mattermost preview streaming mode that preserves earlier reasoning, tool-status, and partial-reply phases instead of rewriting one post through every transition.

## Best Possible Solution

Review and land or revise PR #75256 while coordinating with PR #73252: keep the current edit-in-place default, add a validated Mattermost preview streaming mode, split block-mode previews at meaningful boundaries with forceNewMessage, and align tests, docs, config metadata, and changelog.

## Reproduction Assessment

Yes. A Mattermost tool-call turn maps directly to current source: monitor callbacks call draftStream.update for reasoning, partial, and tool states while draft-stream.ts keeps streamPostId set, so later updates edit the same post.

## Solution Assessment

Yes. A Mattermost-local streaming mode and boundary controller is the narrow maintainable solution; changing shared core dispatch would be broader than necessary because the overwrite behavior is owned by the Mattermost adapter.
