## Summary

Keep this issue open. Current main still routes Mattermost draft-preview phase transitions through one editable preview post, and the requested Mattermost preview `streaming.mode` boundary split is not implemented; the issue is also paired with open closing PR #75256, so it should not be closed before that PR or an equivalent fix lands.

## What This Changes

Mattermost users report that draft-preview phase transitions overwrite one visible post and request a configurable mode that preserves prior preview content by splitting at assistant, reasoning, and tool boundaries.

## Best Possible Solution

Land or revise a Mattermost-local fix that validates and resolves preview `streaming.mode`, preserves the current edit-in-place default, and in `block` mode calls the existing draft-stream split seam at assistant, reasoning, tool-start, and tool-end boundaries with focused schema, account, draft-stream, and monitor tests.

## Reproduction Assessment

Yes. The issue gives a concrete Mattermost DM/tool-call reproduction path, and current main has a code-level reproduction: every preview phase calls `draftStream.update(...)` against the same post id until `forceNewMessage()` or cleanup clears it.

## Solution Assessment

Yes. A Mattermost-local streaming-mode gate and boundary split is the narrowest maintainable solution because the draft stream already has the needed `forceNewMessage()` seam and Discord already uses the same pattern without changing shared core dispatch.
