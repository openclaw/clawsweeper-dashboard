## Summary

Keep open, but the patch is not mergeable as-is: the reset-memory flush is wired to the wrong reset context, drops runtime config/sandbox state, and has no after-fix real behavior proof. Current main also moved some touched surfaces, so the branch is stale and conflicting.

## What This Changes

The PR bundles Mac CLI timeout, reset-time memory flush, browser restart-deferral, web-search description, sidebar width, and skill-creator documentation changes.

## Best Possible Solution

Rework this as focused PRs; the memory change should use the outgoing reset session, preserve existing config/sandbox/session-file semantics, include reset-path tests, and provide real behavior proof before merge.

## Reproduction Assessment

Yes at source level for the blocking PR bugs: reset hooks receive both the new/current entry and the outgoing `previousSessionEntry`, while this patch reads `context.sessionEntry` and then runs with no forwarded config. No live after-fix behavior proof was provided.

## Solution Assessment

No. The requested direction is plausible, but this implementation is too broad and the reset flush should be rebuilt around the existing memory-flush/runtime contract rather than a standalone hook with partial context.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Flush the outgoing reset session, not the fresh one:** `src/gateway/session-reset-memory-flush.ts:30-36`
  - body: `/new` and hard `/reset` go through `initSessionState` before command hooks, so `context.sessionEntry` can already be the fresh replacement session with cleared token counters. This makes the zero-token guard skip the flush for the outgoing transcript the feature is meant to preserve; use the previous/outgoing session entry and its transcript instead.
  - confidence: 0.9
- **[P1] Pass runtime config into the embedded flush run:** `src/gateway/session-reset-memory-flush.ts:58-75`
  - body: This call loads `cfg` but never passes `config: cfg` into `runEmbeddedPiAgent`. The embedded runner resolves plugin models, auth profiles, workspace/sandbox policy, and provider normalization from `params.config`, so custom-provider or sandboxed sessions can fail or run under the wrong policy.
  - confidence: 0.88
- **[P2] Resolve the real transcript instead of using CWD fallback:** `src/gateway/session-reset-memory-flush.ts:70`
  - body: `SessionEntry.sessionFile` is optional, but falling back to the literal `session.jsonl` points at the process working directory instead of the session transcript derived from `sessionId`. Legacy or repaired entries without `sessionFile` would flush the wrong file or no useful history right before reset.
  - confidence: 0.86
- **[P2] Make reset hook registration idempotent:** `src/gateway/session-reset-memory-flush.ts:18-21`
  - body: `registerInternalHook` appends to the global handler list. If `startGatewayServer` runs more than once in a process during tests, staged startup, or hot reload, this registers duplicate reset handlers and can launch multiple concurrent 60-second memory flushes for one reset.
  - confidence: 0.81
- **[P3] Warn when the reset flush fails:** `src/gateway/session-reset-memory-flush.ts:79-81`
  - body: Failures are only logged through `logVerbose`, so a provider error or timeout during this best-effort preservation step is invisible in normal operator logs. Surface the failure at the gateway warning level while still allowing reset to continue.
  - confidence: 0.75
