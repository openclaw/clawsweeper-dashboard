## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Pass the loaded config into the flush run:** `src/gateway/session-reset-memory-flush.ts:59-77`
  - body: `runEmbeddedPiAgent` is called without `config: cfg`, but the runner uses `params.config` for runtime plugins, harness/model resolution, auth, workspace defaults, and tool policy. Sessions using configured custom providers or agent model settings can fail the reset flush before writing memory while the reset continues.
  - confidence: 0.9
- **[P1] Resolve the real transcript path before flushing:** `src/gateway/session-reset-memory-flush.ts:71`
  - body: Falling back to the relative `session.jsonl` when `entry.sessionFile` is absent opens a CWD file instead of the transcript derived from the session id/store. Legacy or migrated entries can therefore flush the wrong transcript and miss the memories this hook is meant to preserve.
  - confidence: 0.88
- **[P2] Make reset hook registration idempotent:** `src/gateway/session-reset-memory-flush.ts:19-21`
  - body: `startGatewayServer` can run more than once in a process, while `registerInternalHook` appends to a global handler list. Without a guard, each later gateway start registers another copy and every `/new` or `/reset` can launch duplicate 60-second memory flush runs.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff adds a silent reset-time agent run, which needs explicit tool/runtime policy and audit handling before merge.

Concerns:

- **[medium] Bound silent reset-time agent execution:** `src/gateway/session-reset-memory-flush.ts:59`
  - body: The new hook starts `runEmbeddedPiAgent` during reset without the existing memory-run context such as config, sandbox session key, silent expectation, reply operation, or abort signal. That can make provider/tool policy diverge from the user turn and makes teardown side effects harder to bound.
  - confidence: 0.82

  - body: The hook swallows provider/tool failures and logs only via `logVerbose`, so failed silent memory writes during teardown may be invisible to operators unless verbose logging is enabled.
  - confidence: 0.78
