## Summary

Keep open: the useful direction is real, but this PR is not mergeable as-is because the reset-memory implementation has source-level correctness bugs, lacks real behavior proof, and still needs maintainer product/security review for a silent reset-time agent run.

## What This Changes

The PR increases the macOS wizard timeout, adds a reset-time memory flush hook, counts browser work in restart deferral, widens the dashboard nav, updates Brave LLM-context copy, and documents skill validation.

## Best Possible Solution

Split and rework the reset-memory change around the existing memory-flush runtime so it uses the outgoing session, preserves config/sandbox/session-file semantics, covers manual/daily/idle reset intentionally, and lands with focused tests plus real behavior proof.

## Reproduction Assessment

Yes at source level for the blocking PR bugs: current reset hooks provide both the fresh/current entry and outgoing previousSessionEntry, while this patch reads context.sessionEntry and calls the embedded runner without forwarded config.

## Solution Assessment

No. The requested direction is plausible, but this bundled implementation is not the narrow maintainable fix because it bypasses the established memory-flush runtime context and leaves reset-time policy decisions unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Flush the outgoing reset session:** `src/gateway/session-reset-memory-flush.ts:30-36`
  - body: `/new` and hard `/reset` can replace the current entry before command hooks run, and the new entry has cleared token counters. Reading `context.sessionEntry` makes this hook skip or flush the fresh session instead of the outgoing transcript the feature is meant to preserve; use the previous/outgoing session entry and transcript.
  - confidence: 0.9
- **[P1] Forward runtime config to the flush run:** `src/gateway/session-reset-memory-flush.ts:58-75`
  - body: This loads `cfg` but never passes `config: cfg` into `runEmbeddedPiAgent`. The embedded runner resolves harnesses, configured providers/models, auth profiles, and runtime policy from `params.config`, so custom-provider or sandboxed sessions can fail or run under the wrong policy during reset.
  - confidence: 0.88
- **[P2] Resolve missing sessionFile through transcript helpers:** `src/gateway/session-reset-memory-flush.ts:70`
  - body: `SessionEntry.sessionFile` is optional, but falling back to the literal `session.jsonl` points at the process working directory instead of the session transcript derived from `sessionId`. Legacy or repaired entries without `sessionFile` would flush the wrong file or no useful history.
  - confidence: 0.86
- **[P2] Guard reset hook registration:** `src/gateway/session-reset-memory-flush.ts:18-21`
  - body: `registerInternalHook` appends handlers to a global list. If `startGatewayServer` runs more than once in one process, this registers duplicate reset handlers and can launch multiple concurrent 60-second flushes for a single reset.
  - confidence: 0.81
- **[P3] Warn on flush failures:** `src/gateway/session-reset-memory-flush.ts:79-81`
  - body: Provider errors or timeouts are only logged through `logVerbose`, so failures of this best-effort preservation path are invisible in normal operator logs. Surface failures at the gateway warning level while still allowing reset to continue.
  - confidence: 0.75
