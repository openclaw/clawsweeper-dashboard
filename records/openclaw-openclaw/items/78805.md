## Summary

Keep open: current main still has source-plausible synchronous credential/keychain and file-read paths that can block the gateway event loop, and the open linked PR has not merged. The newer WSL2/Telegram comment points at a broader `channels.telegram.start-account` stall, so this issue should stay open until the linked PR lands or is replaced by a narrower proven fix.

## What This Changes

The issue asks to move gateway/auth/channel startup work away from synchronous I/O that can block the Node event loop and destabilize Telegram/WebSocket behavior.

## Best Possible Solution

Keep this issue as the canonical tracker for sync-I/O event-loop blocking, review or replace the linked async-auth PR, and split a narrower Telegram startup issue only if profiling proves a distinct root cause.

## Reproduction Assessment

No high-confidence end-to-end reproduction was established. Source inspection confirms blocking sync keychain/file-I/O paths and the matching event-loop diagnostic path, but no live macOS or WSL2 Telegram run tied a specific call to the reported multi-second stall.

## Solution Assessment

Unclear. Moving proven hot-path keychain and startup I/O off the main event loop is the right direction, but the maintainable fix should target measured auth/channel blockers rather than mechanically converting every synchronous read.
