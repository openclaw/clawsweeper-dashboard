## Summary

Keep open: current main still rejects hardlinked package roots in the update/install swap path, while the docs expect staged npm updates and auto-updates to apply; this is source-reproducible but needs security-aware owner review because the implicated hardlink policy is a filesystem safety guard.

## What This Changes

The issue asks OpenClaw to make auto-update/package swap tolerate npm/pnpm hardlinked global package trees instead of failing and retrying.

## Best Possible Solution

Keep the issue open and implement a scoped hardlink-tolerant package update/install path with regression tests for hardlinked package roots while preserving the generic hardlink rejection default where it protects unsafe file replacement.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create an npm/pnpm-like package root containing a hardlinked file and run the staged package swap that passes sourceHardlinks: "reject". I did not run the live global install flow in this read-only review.

## Solution Assessment

Unclear as proposed: flipping all six call sites may be too broad for a security-sensitive hardlink guard. The maintainable fix is a scoped allowance for trusted package-manager-owned swap sources plus regression tests for update, rollback, and boundary behavior.
