## Label Justifications

- `P1`: A current package auto-update path can repeatedly fail and leave npm/pnpm users pinned on an older release after a new OpenClaw version is available.
- `impact:security`: The affected updater can leave users unable to receive future fixes, and the repair changes a hardlink-related filesystem safety boundary.

## Summary

Keep open: current main and v2026.5.20 still reject hardlinked package roots in the staged update/install swap path. The older related report covers the same root cause, but this issue adds broader npm/pnpm scope and security-policy detail, so closing it as a duplicate would be premature without maintainer consolidation.

## What This Changes

The issue asks OpenClaw to make auto-update and package install swaps tolerate package-manager hardlinks instead of failing during `global install swap`.

## Best Possible Solution

Implement a scoped hardlink-tolerant package-manager swap path with regression tests for hardlinked package roots, while preserving hardlink rejection for untrusted boundary-sensitive file replacement.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create an npm/pnpm-like package root containing a hardlinked file and run the staged package swap that passes `sourceHardlinks: "reject"`; I did not run the live global install flow in this read-only review.

## Solution Assessment

Unclear as proposed: flipping all six call sites may be too broad for a security-sensitive hardlink guard. The maintainable fix is a scoped allowance for trusted package-manager-owned swap sources plus regression coverage for update, rollback, and boundary behavior.
