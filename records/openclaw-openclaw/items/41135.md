## Summary

Keep open: current main has auth-profile ordering, last-used rotation, cooldown/failover, capped rate-limit profile rotation, and env API-key rotation, but it still lacks the requested opt-in provider-profile pool routing contract with per-profile concurrency, task/pool stickiness, strategy selection, quota/bucket state, and routing observability.

## What This Changes

The issue requests an opt-in provider-profile pool-routing layer for multi-account OAuth/API-key provider pools, starting with google-gemini-cli, with configurable policies such as round-robin, least-used, cooldown-aware fallback, per-profile concurrency, and status visibility.

## Best Possible Solution

Keep this as the broad provider-profile pool-routing design tracker, then split accepted work into narrower config/schema, auth-state scheduler, embedded-runner routing, status/observability, provider quota, and docs slices after maintainers settle the public contract.
