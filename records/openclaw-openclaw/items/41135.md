## Summary

Keep this issue open. Current main still has auth profile ordering, cooldown/disabled failover, usage display, and session pinning, but it does not provide the requested opt-in provider-profile pool routing layer with explicit strategies, task/none stickiness, per-profile concurrency/in-flight state, weights, quota-aware scheduling, pool-state JSON, or `models routing` CLI controls. The related open issue #38775 overlaps on usage-aware load balancing, but this issue is broader and still useful as the generic provider-pool design track.

## What This Changes

The issue requests an opt-in provider-profile routing layer for multi-account OAuth/API pools, including pool strategies, stickiness modes, concurrency limits, cooldown/failure-aware fallback, and observable routing state.

## Best Possible Solution

Keep this as an open core design issue for an additive auth-routing contract: typed config/schema for pool mode, strategies, stickiness, profile lists, weights, and concurrency limits; runtime in-flight/cooldown/quota-aware selection in the embedded runner; status JSON exposing pool state; and CLI controls after the contract is stable. Provider hooks can supply quota hints, but the generic scheduler/state model should be core-owned if this is accepted.

## Reproduction Assessment

Not applicable as a pure feature request. The central gap is reproducible by source inspection: current main has profile order/cooldown/session pinning but no explicit pool-routing contract or scheduler.

## Solution Assessment

Unclear as an implementation plan because this is broader than a narrow bug fix. The requested direction is plausible for core because it crosses auth state, embedded runner selection, status JSON, and CLI UX, but the exact config and strategy semantics need maintainer design before a fix PR.
