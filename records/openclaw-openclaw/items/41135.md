## Summary

Keep open. Current main has auth order, cooldown/disabled failover, usage snapshots, session pinning, and a provider hook that may prefer one profile, but it still lacks the requested opt-in provider-profile pool scheduler, routing strategy contract, task/no stickiness mode, per-profile in-flight/concurrency state, pool-state JSON, and `models routing` CLI controls.

## What This Changes

The issue requests an opt-in provider-profile pool routing layer with explicit strategies, stickiness modes, concurrency limits, cooldown/failure awareness, observability, and CLI/status controls for multi-account provider workloads.

## Best Possible Solution

Keep this as an open core design issue for an additive auth-routing contract covering typed config/schema, scheduler state, status JSON, CLI UX, and provider quota hints while preserving session-sticky defaults.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the gap: current main implements order/cooldown/session pinning and usage display, but no explicit provider-profile pool routing contract or scheduler.

## Solution Assessment

Unclear until maintainers settle the design. The direction plausibly belongs in core because it crosses auth config, auth-state, embedded-runner selection, status JSON, CLI UX, and provider quota hooks; existing plugin hooks can prefer one profile or expose usage snapshots but do not provide a full pool scheduler.
