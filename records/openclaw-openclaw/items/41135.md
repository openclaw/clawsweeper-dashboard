## Summary

Keep open: current main has auth-profile ordering, last-used profile rotation, cooldown/failover, probe status, and rate-limit rotation caps, but it still lacks the requested opt-in provider-profile pool-routing contract with routing strategies, task/pool stickiness, in-flight/concurrency state, quota/bucket metadata, and route-decision observability. The open related key-pool tracker is narrower, while this issue remains the broader provider-profile routing design tracker.

## What This Changes

The issue requests an opt-in provider-profile routing layer for distributing batch or worker requests across multiple OAuth/API-key profiles or endpoints with explicit strategies, concurrency limits, cooldown/quota behavior, and status/CLI observability.

## Best Possible Solution

Keep this as the broad provider-profile pool-routing design tracker, then split accepted work into narrower config/schema, scheduler/state, embedded-runner routing, status/observability, quota/bucket, and docs slices after maintainers settle the public contract.
