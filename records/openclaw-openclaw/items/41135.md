## Summary

Keep open: current main has auth profiles, explicit order, cooldown-aware ordering, same-provider retry caps, and session stickiness, but it still lacks the requested opt-in provider-profile pool routing layer with task/no stickiness, in-flight/concurrency limits, weights, quota signals, and configurable pool strategies.

## What This Changes

The issue requests an OpenClaw-native provider-profile routing policy for multi-account OAuth/API pools, starting with google-gemini-cli, while preserving current chat-session defaults.

## Best Possible Solution

Define an additive generic provider/auth-profile routing contract that preserves session-sticky defaults while integrating pool policy, provider-owned quota signals, persisted state, status JSON, docs, and upgrade-safe config handling.

## Reproduction Assessment

Yes at source level: current main lacks the requested routing config/state and still implements explicit order, last-used ordering, cooldown-aware failover, and session auth-profile pinning; no live Gemini OAuth pool run was performed.

## Solution Assessment

Unclear until maintainers choose the contract; the capability gap is real, but the illustrative config shape and strategy list are not confirmed as the best implementation.
