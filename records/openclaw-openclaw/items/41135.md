## Summary

Keep open: current `main` and the latest release still provide explicit auth order, last-used rotation, cooldown-aware failover, and session pinning, but not the requested opt-in provider-profile pool routing contract with task/no stickiness, per-profile concurrency, weights, or quota-aware policy selection.

## What This Changes

The issue requests an opt-in provider-profile routing layer for multi-account OAuth/API pools with configurable strategies, stickiness, concurrency, cooldown handling, and status/CLI observability.

## Best Possible Solution

Design an additive generic provider/auth-profile routing contract that preserves session-sticky defaults while integrating pool policy, provider-owned quota signals, cooldown state, status JSON, schema, docs, and upgrade-safe config handling.

## Reproduction Assessment

Yes at source level: current `main` and `v2026.6.8` implement and document auth order, last-used rotation, cooldowns, and session pinning, while schema/state lack the requested routing pool and concurrency fields. I did not run a live Gemini OAuth pool.

## Solution Assessment

Unclear as an implementation shape: the issue identifies a real capability gap, but the sketched config and strategy list are not yet a confirmed OpenClaw contract. The safer path is a maintainer-approved generic provider/auth routing design reconciled with https://github.com/openclaw/openclaw/issues/28847.
