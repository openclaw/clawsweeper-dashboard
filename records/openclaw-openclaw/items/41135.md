## Summary

Keep open. Current main still provides auth-profile ordering, round-robin by last-used time, session stickiness, cooldown/failover, and status/probe visibility, but it does not define the requested opt-in provider-profile pool routing contract, scheduler strategies, per-profile in-flight/quota state, or CLI/status policy surface.

## What This Changes

The issue requests a new provider-profile pool-routing layer with configurable strategies, stickiness modes, concurrency/quota state, and CLI/status observability for multi-account and multi-endpoint provider pools.

## Best Possible Solution

Keep this as the canonical core design issue for an additive provider-profile pool-routing contract that preserves session-sticky defaults while defining typed config, scheduler semantics, observability, and regression coverage.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives high confidence that current main lacks the requested provider-pool routing contract, while the specific Gemini CLI probe timeout was not live-reproduced.

## Solution Assessment
