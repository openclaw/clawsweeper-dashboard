## Summary

Keep open: current main and v2026.6.6 still provide auth order, last-used rotation, cooldown-aware failover, session pinning, and flat API-key rotation, but they do not implement the requested opt-in provider-profile pool routing with task/no stickiness, per-profile concurrency, weights, or quota-aware policy selection.

## What This Changes

The issue requests an opt-in provider-profile routing layer for distributing batch/worker requests across multiple OAuth/API-key profiles or endpoints for the same provider.

## Best Possible Solution

Design an additive generic provider/auth-profile routing contract that preserves session-sticky defaults while integrating pool policy, provider-owned quota signals, cooldown state, status JSON, schema, docs, and upgrade-safe config handling.

## Reproduction Assessment

Yes, at source level: current main lacks the requested routing config/state and still documents last-used ordering plus per-session profile pinning. I did not exercise a live Gemini OAuth pool in this read-only review.

## Solution Assessment

Unclear: the issue identifies a real capability gap, but the illustrative config and strategy list are not yet a confirmed implementation contract. A maintainer-approved generic provider/auth routing design should come first.
