- `impact:session-state`: The requested task/no-stickiness modes would change how auth profile selection is pinned or rotated across sessions and worker tasks.
- `impact:auth-provider`: The requested behavior directly affects provider auth profile selection, OAuth/API-key routing, cooldowns, quota-aware provider choice, and model fallback.

## Summary

Keep open: current main and the latest release have bounded same-provider auth-profile rotation, cooldown/failover, status/probe visibility, and session-sticky selection, but not the requested opt-in provider-profile pool routing contract with task/no stickiness, strategy selection, per-profile concurrency, weights, quota buckets, or routing CLI/state.

## What This Changes

The issue requests an OpenClaw-native provider-profile routing layer for multi-account OAuth/API pools, starting with Gemini CLI, with configurable pool strategies and observability.

## Best Possible Solution

Define an additive generic provider/auth-profile routing contract that preserves session-sticky defaults while integrating pool policy, provider-owned quota signals, persisted state, status JSON, docs, and upgrade-safe config handling.

## Reproduction Assessment

Yes at source level: current main lacks the requested routing config/state and still implements explicit order, last-used ordering, cooldown-aware bounded failover, and session auth-profile pinning. I did not exercise a live Gemini OAuth pool in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. The issue identifies a real capability gap, but the illustrative config and strategy list are not yet a confirmed implementation shape.
