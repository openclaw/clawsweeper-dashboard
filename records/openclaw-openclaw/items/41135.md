## Summary

Keep open. Current main has auth profile ordering, cooldown/disabled failover, usage snapshots, and probe/status reporting, but it still lacks the requested opt-in provider-profile pool routing contract with explicit strategies, task/no stickiness, in-flight/concurrency state, weights, quota-aware scheduling, pool-state JSON, or routing CLI controls.

## What This Changes

Request an opt-in provider-profile pool routing layer that distributes model requests across multiple auth profiles/endpoints with explicit strategies, stickiness, concurrency, cooldown, and observability controls.

## Best Possible Solution

Keep this as a core auth-routing design issue for an additive config/schema, scheduler state, status JSON, CLI UX, and provider quota-hint contract coordinated with #38775.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken current contract. Source inspection confirms the gap on current main: existing code covers ordering, cooldowns, usage snapshots, and probes, but no explicit provider-pool scheduler.

## Solution Assessment

Unclear until maintainers settle the design. Native core support is plausible because the request crosses auth config, runner selection, status JSON, and CLI UX, but schema and strategy semantics need product review first.
