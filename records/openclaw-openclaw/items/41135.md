## Summary

Keep open: current main and v2026.6.8 still provide auth profiles, explicit order, cooldown-aware failover, last-used ordering, and session-sticky auto-selection, but they do not provide the requested opt-in provider-profile pool routing policy with task/no stickiness, in-flight/concurrency caps, weights, or quota-aware selection.

## What This Changes

Request to add an opt-in provider-profile pool routing layer for multi-account OAuth/API-key and endpoint pools, starting with google-gemini-cli.

## Best Possible Solution

Define an additive generic provider/auth-profile routing contract that preserves session-sticky defaults while integrating pool policy, provider-owned quota signals, cooldown state, status JSON, schema, docs, and upgrade-safe config handling.

## Reproduction Assessment

Yes, at source level: current main lacks the requested routing config/state and still implements explicit order, last-used ordering, cooldown-aware failover, and session auth-profile pinning. I did not exercise a live Gemini OAuth pool in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. The issue identifies a real capability gap, but the illustrative config and strategy list are not yet a confirmed implementation shape.
