- `P2`: The request is a meaningful provider/auth routing improvement for batch and multi-account users, but not a core-wide outage or emergency.
- `impact:session-state`: The requested task and no-stickiness modes would change how auth profile selection is pinned or rotated across sessions and worker tasks.

## Summary

Keep open: current main and v2026.6.6 still provide auth order, last-used rotation, cooldown-aware failover, session stickiness, and flat API-key rotation, but not the requested opt-in provider-profile pool routing with task/no stickiness, per-profile concurrency, weights, or quota-aware policy selection.

## What This Changes

Request provider-profile pool routing policies for multi-account OAuth/API-key/provider endpoint pools, starting with google-gemini-cli batch workloads.

## Best Possible Solution

Design an additive generic provider/auth-profile routing contract that preserves session-sticky defaults while integrating pool policy, provider-owned quota signals, cooldown state, status JSON, schema, docs, and upgrade-safe config handling.

## Reproduction Assessment

Yes, at source level: current main lacks the requested routing config/state and still documents last-used ordering plus per-session profile pinning. I did not exercise a live Gemini OAuth pool in this read-only review.

## Solution Assessment

Unclear: the broad direction is plausible, but the sample config and strategy list are illustrative and need maintainer decisions on core versus provider ownership, quota signals, and upgrade semantics.
