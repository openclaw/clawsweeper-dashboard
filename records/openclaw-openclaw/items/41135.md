## Summary

Keep open: current main still has ordered auth profiles, round-robin-by-last-used selection, session stickiness, cooldown/failover, and probe/status visibility, but it does not have the requested opt-in provider-profile pool routing contract, scheduler state, or per-profile concurrency/quota policy. This remains a product/API design item rather than a narrow bug fix.

## What This Changes

The issue requests an opt-in provider-profile pool routing policy layer for multi-account OAuth/API/provider endpoint pools, starting with google-gemini-cli batch workloads.

## Best Possible Solution

Keep this as the broad core product/API tracker for additive provider-profile pool routing, then split implementation into narrower auth-state, scheduler, model-status/probe, and provider-specific quota slices once the public contract is settled.

## Reproduction Assessment

Not applicable as a feature request. Source, docs, schema, and related-issue inspection give high confidence that current main lacks the requested pool-routing contract, while the reported Gemini CLI probe timeout was not live-reproduced.

## Solution Assessment

Unclear until maintainers settle the public config/API, scheduler semantics, and provider/plugin ownership boundary. The direction plausibly belongs in core because it crosses auth-profile selection, embedded runs, model status/probe output, and CLI UX, but it is not a narrow repair.
