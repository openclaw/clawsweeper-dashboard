## Summary

Keep open: current main and the latest release have auth profile order, cooldown state, last-used ordering, same-provider retry caps, and session pinning, but they still do not provide the requested opt-in provider-profile pool routing layer with explicit strategies, in-flight/concurrency limits, quota/bucket state, weights, or task/no-stickiness modes.

## What This Changes

Request to add opt-in provider-profile pool routing policies for distributing work across multiple OAuth/API profiles or endpoints for the same provider.

## Best Possible Solution

Define an additive generic provider/auth-profile routing contract that preserves session-sticky defaults while integrating provider-owned quota signals, persisted state, status JSON, docs, and upgrade-safe config handling.

## Reproduction Assessment

Yes, at source level: current main lacks the requested routing config/state and still implements explicit order, cooldown-aware failover, last-used ordering, and session auth-profile pinning. I did not run a live Gemini OAuth pool workload.

## Solution Assessment

Unclear until maintainers choose the contract; the capability gap is real, but the illustrative config shape and strategy list are not yet proven to be the best implementation. The safest path is an additive, generic routing policy that keeps current defaults unchanged.
