## Summary

Current main and v2026.6.5 still provide reactive cooldown-aware ordering and session pinning, not the requested task/pool routing; this issue remains the broad canonical design tracker and needs a maintainer product decision rather than cleanup closure.

## What This Changes

Add an opt-in provider-profile routing layer that distributes batch work across multiple OAuth profiles, API keys, or endpoints using configurable health, load, cooldown, and stickiness policies.

## Best Possible Solution

Approve a phased provider-neutral design that preserves current session-sticky defaults, adds one explicit batch/task routing mode, reuses canonical cooldown state, and validates one OAuth pool plus one API-key or endpoint pool before expanding strategies or CLI surface.

## Reproduction Assessment

No failing bug reproduction is applicable, but the feature gap is source-reproducible with high confidence: current main implements last-used round robin plus session persistence, not explicit task/pool routing or per-profile load controls.

## Solution Assessment

No, the complete multi-strategy configuration sketch is not yet the narrowest proven solution. A phased generic routing seam with preserved defaults and focused state/upgrade contracts is safer than either a provider-specific scorer or the full proposed policy matrix.
