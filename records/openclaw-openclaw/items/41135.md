## Summary

Keep open. Current main still has auth-profile ordering, cooldown/disabled state, usage readouts, probes, and session pinning, but it does not provide the requested opt-in provider-pool routing contract with named strategies, per-profile in-flight/concurrency accounting, quota/bucket-aware selection, or routing-state inspection; related open items cover narrower key-pool or rate-limit slices rather than replacing this broader design tracker.

## What This Changes

The issue requests an opt-in provider-profile pool routing layer for distributing batch or worker requests across multiple OAuth/API profiles or endpoints with explicit strategies, concurrency limits, cooldown behavior, and observability.

## Best Possible Solution

Keep this as the core design tracker for an additive auth-routing contract that preserves session-sticky defaults while introducing typed pool configuration, scheduler semantics, runtime state, CLI/status inspection, and regression coverage across OAuth, API-key, and endpoint profiles.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection provide high-confidence absence proof for the requested pool routing surface, but the Gemini CLI timeout example was not live-reproduced.

## Solution Assessment

Unclear until maintainers choose the public contract. The direction plausibly belongs in core because it crosses auth config, profile selection, embedded runs, status JSON, and CLI UX, but the schema and strategy semantics need product/API review first.
