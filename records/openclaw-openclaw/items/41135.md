- `impact:session-state`: The requested task and no-stickiness modes would change how auth profile selection is pinned or rotated across sessions and worker tasks.

## Summary

Keep open: current main partially addresses failure-driven same-provider rotation, but it still does not provide the requested opt-in provider-profile pool routing layer with configurable strategies, task/no stickiness, in-flight/concurrency limits, weights, quota/bucket state, or pool state in status output.

## What This Changes

Request to add an OpenClaw-native provider-profile routing policy layer for multi-account OAuth/API-key/endpoint pools, starting from google-gemini-cli batch workloads.

## Best Possible Solution

Define an additive generic provider/auth-profile routing contract that preserves session-sticky defaults while integrating pool policy, provider-owned quota signals, persisted state, status JSON, docs, and upgrade-safe config handling.

## Reproduction Assessment

Yes at source level: current main has auth profiles, explicit order, cooldown-aware ordering, same-provider retry caps, and session profile pinning, but lacks the requested provider-pool config/state/policy surface. I did not run a live Gemini OAuth pool reproduction.

## Solution Assessment

Unclear until maintainers choose the contract; the capability gap is real, but the issue's illustrative strategy/config shape is not yet proven to be the narrowest maintainable implementation.
