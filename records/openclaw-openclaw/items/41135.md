## Summary

Keep open. Current main has auth-profile order, cooldown/disabled failover, usage snapshots, session pinning, and a provider hook that can prefer one profile, but it still lacks the requested opt-in provider-profile pool scheduler, routing strategies, task/no stickiness, per-profile in-flight/concurrency state, weights, quota-aware selection, pool-state JSON, or `models routing` CLI controls.

## What This Changes

The issue requests an opt-in provider-profile pool routing layer for distributing provider calls across multiple auth profiles or endpoints.

## Best Possible Solution

Keep this as an open core design issue for an additive auth-routing contract covering typed config/schema, scheduler state, status JSON, CLI UX, and provider quota hints while preserving session-sticky defaults.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the gap: current main implements order/cooldown/session pinning and usage display, but no explicit provider-profile pool routing contract or scheduler.

## Solution Assessment

Unclear until maintainers settle the design. The direction plausibly belongs in core because it crosses auth config, embedded-runner selection, status JSON, and CLI UX, but the exact schema, strategy semantics, and relationship to #38775 need product review first.
