## Summary

Current main still lacks the requested opt-in provider-profile pool routing contract. OpenClaw has ordered auth profiles, last-used rotation, session stickiness, cooldown/failover state, and status/probe visibility, but there is no typed routing policy, scheduler strategy, per-profile in-flight/quota state, or CLI/config surface for provider pools, so this remains a valid product/API design issue.

## What This Changes

Request an OpenClaw-native provider-profile routing layer that distributes work across multiple OAuth/API-key/endpoint profiles with explicit pool policies and observability.

## Best Possible Solution

Keep this as the broad core product/API design issue for additive provider-profile pool routing, while using narrower linked issues for specific key-pool, quota-bucket, or request-budget slices once the public contract is settled.

## Reproduction Assessment

Not applicable as a feature request. Source and schema inspection gives high confidence that current main lacks the requested pool-routing contract, while the reported Gemini CLI probe timeout was not live-reproduced.

## Solution Assessment

Unclear until maintainers settle the public config/API and default policy. The direction plausibly belongs in core because it crosses auth-profile selection, embedded runs, model status/probe output, and CLI UX, but it is not a narrow repair.
