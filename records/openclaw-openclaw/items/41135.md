## Summary

Keep open: current main and v2026.6.10 still provide auth profile order, cooldowns, session stickiness, probe/status visibility, and bounded same-provider rotations, but they do not provide the requested opt-in provider-pool routing policy with strategy selection, task/no stickiness, per-profile concurrency/in-flight state, weights, or quota buckets.

## What This Changes

Add an opt-in provider-profile routing layer that can distribute provider requests across multiple OAuth/API-key/endpoints using explicit pool policies and observable state.

## Best Possible Solution

Define an additive generic provider/auth-profile routing contract that preserves session-sticky defaults while integrating pool policy, provider-owned quota signals, persisted state, status JSON, docs, and upgrade-safe config handling.

## Reproduction Assessment

Yes, at source level: current main lacks the requested routing config/state and still implements explicit order, last-used ordering, cooldown-aware failover, and session auth-profile pinning. I did not exercise a live Gemini OAuth pool in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. The issue identifies a real capability gap, but the illustrative config and strategy list are not yet a confirmed implementation shape.
