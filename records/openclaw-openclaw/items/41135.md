## Summary

Keep open: current main and v2026.6.6 still provide explicit auth order, last-used profile ordering, session stickiness, profile cooldowns, and flat API-key rotation, but not the requested opt-in provider-profile pool routing with task/no stickiness, in-flight caps, weights, or quota-aware policies.

## What This Changes

The issue requests an OpenClaw-native provider-profile routing layer for multi-account OAuth/API-key pools, starting with google-gemini-cli batch/worker workloads.

## Best Possible Solution

Design an additive generic provider/auth-profile routing contract that preserves session-sticky defaults while integrating pool policy, provider-owned quota signals, cooldown state, status JSON, schema, docs, and upgrade-safe config handling.

## Reproduction Assessment

Yes, at source level: current main lacks the requested routing config/state and still documents last-used ordering plus per-session profile pinning. I did not exercise a live Gemini OAuth pool in this read-only review.

## Solution Assessment

Unclear: the broad direction is plausible, but the issue's sample config and strategy list are illustrative and need maintainer decisions on core versus provider ownership, quota signals, and upgrade semantics.
