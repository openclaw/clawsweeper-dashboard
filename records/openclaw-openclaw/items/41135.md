## Summary

Keep open. Current main has auth profile ordering, cooldown/disabled failover, usage snapshots, session pinning, and a provider hook that can prefer a profile, but it still lacks the requested opt-in provider-profile pool routing layer with explicit strategies, task/no stickiness, per-profile concurrency/in-flight state, weights, quota-aware scheduling, pool-state JSON, or `models routing` CLI controls.

## What This Changes

Request an OpenClaw-native provider-profile pool routing layer for multi-account OAuth/API-key/provider endpoint workloads, starting with `google-gemini-cli`.

## Best Possible Solution

Keep this as an open core design issue for an additive auth-routing contract covering typed config/schema, scheduler state, status JSON, and CLI UX, with provider hooks supplying quota hints where available.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the gap: current main implements order/cooldown/session pinning and usage display, but no explicit provider-pool routing contract or scheduler.

## Solution Assessment

Unclear until maintainers settle the design. The requested direction plausibly belongs in core because it crosses auth config, embedded-runner selection, status JSON, and CLI UX, but the exact schema, strategy semantics, and relationship to #38775 need product review first.
