## Summary

Keep open. Current main still provides auth-profile ordering, last-used round-robin, session stickiness, cooldown/failover, and status/probe visibility, but it does not define the requested opt-in provider-profile pool routing contract, scheduler strategies, per-profile in-flight/quota state, or CLI/status policy surface.

The issue requests a new provider-profile pool routing layer for distributing work across multiple OAuth/API/endpoint profiles with explicit policies, concurrency caps, cooldown/failover, and observability.

## Best Possible Solution

Keep this as the canonical core design issue for an additive provider-profile pool-routing contract that preserves session-sticky defaults while defining typed config, scheduler semantics, observability, and regression coverage.
