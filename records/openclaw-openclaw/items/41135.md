## Summary

Keep open: current main and the latest release have explicit auth order, last-used ordering, session stickiness, and cooldown failover, but no opt-in provider-profile pool routing with task/no stickiness, in-flight/concurrency caps, weights, or quota-aware policies.

## What This Changes

Add opt-in provider-profile pool routing policies across multiple auth profiles or endpoints for batch-style provider workloads.

## Best Possible Solution

Design an additive generic provider/auth-profile routing contract that preserves session-sticky defaults while integrating pool policy, provider-owned quota signals, cooldown state, status JSON, schema, docs, and upgrade-safe config handling.

## Reproduction Assessment

Yes at source level: current main lacks the requested routing config/state and still documents last-used ordering plus per-session profile pinning; no live Gemini OAuth pool was exercised.

## Solution Assessment

Unclear: the broad direction is plausible, but the issue's sample config and strategy list are illustrative and need maintainer decisions on core versus provider ownership, quota signals, and upgrade semantics.
