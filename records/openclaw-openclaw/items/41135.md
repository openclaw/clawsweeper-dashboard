## Summary

Keep open. Current main still has auth order, cooldown/disabled failover, session pinning, status/probe output, and partial API-key rotation, but it does not expose the requested opt-in provider-profile pool routing contract, scheduler, per-profile concurrency/quota state, or CLI/status policy surface; the open related item https://github.com/openclaw/openclaw/issues/28847 covers only the narrower key-pool/bucket subset.

## What This Changes

Add opt-in provider-profile routing policies for batch/worker workloads that distribute requests across multiple OAuth/API-key profiles or endpoints with strategy, concurrency, cooldown, and observability controls.

## Best Possible Solution

Keep this as the canonical core design issue for an additive provider-profile routing contract that preserves session-sticky defaults while defining typed pool config, scheduler semantics, status/CLI observability, and regression coverage.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives high confidence current main lacks the requested pool contract and scheduler, while the reported Gemini probe timeout was not live-reproduced.

## Solution Assessment

Unclear until maintainers settle the public contract. The direction plausibly belongs in core because it crosses auth config, runner selection, status JSON, and CLI UX, but the schema and strategy semantics need product/API review first.
