## Summary

Keep open: current main has auth-profile order, last-used round-robin, cooldown/failover, capped rate-limit profile rotation, and some provider/key-specific cooldown work, but it still lacks the requested opt-in provider-profile pool routing contract with strategy selection, task stickiness, per-profile concurrency, quota/bucket state, and routing observability. Related work supports keeping this as the broader tracker: https://github.com/openclaw/openclaw/issues/38775 was closed in favor of this issue, while https://github.com/openclaw/openclaw/issues/28847 remains a narrower Gemini/key-bucket companion rather than a replacement.

## What This Changes

Request adds an opt-in provider-profile pool routing layer for multi-account OAuth/API providers, with selectable policies, per-profile concurrency/cooldown state, task/session stickiness, and status/CLI observability.

## Best Possible Solution

Keep this as the broad provider-profile pool-routing design tracker, then split accepted work into config/schema, auth-state scheduling, embedded-runner selection, status/observability, quota/provider metadata, and docs slices after maintainers settle the public contract.

## Reproduction Assessment

Not applicable: this is a feature/API design request, not a bug with an existing expected behavior. Source, docs, and tests show current main only supports order/last-used/cooldown rotation; the google-gemini-cli probe timeout symptom was not live-reproduced.

## Solution Assessment

Unclear until maintainers settle the public config/API and default policy. The direction plausibly belongs in core because it crosses auth-profile selection, embedded runs, status/probe output, and CLI UX, but it should be coordinated with the narrower quota-bucket work in https://github.com/openclaw/openclaw/issues/28847.
