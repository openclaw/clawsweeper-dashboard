## Label Justifications

- `P1`: A hung MCP catalog request can stall bundled MCP discovery for about the SDK default 60 seconds per server, affecting agent responsiveness.
- `merge-risk: 🚨 compatibility`: The new unconfigurable 1500 ms catalog-list deadline can hide tools from existing MCP servers whose valid `tools/list` response exceeds that cap.

## Summary

Keep this PR open for normal merge review. Current main still calls bundled MCP `tools/list` without request options, while the PR now follows the maintainer-requested no-new-config shape and adds focused coverage; the main remaining call is whether maintainers accept the internal 1500 ms compatibility tradeoff.

## What This Changes

The PR changes bundled MCP catalog discovery to call `client.listTools` with a 1500 ms internal request timeout and adds regression tests for slow-valid and hung `tools/list` responses.

## Best Possible Solution

Land the no-config internal timeout only after maintainers accept the 1500 ms cutoff, keeping the timeout scoped to catalog discovery and preserving the focused slow-valid/hung regression tests.

## Reproduction Assessment

Yes at source level: current main calls `client.listTools` without request options, and the pinned MCP SDK defaults requests to 60000 ms when no timeout is supplied. I did not run the repro locally because this review is read-only and tests can create artifacts.

## Solution Assessment

Yes, with maintainer acceptance of the cutoff: the PR bounds the exact post-connect catalog request without adding a public config key, matching the latest review direction. The safer alternative is only to tune the internal constant if maintainers prefer a different compatibility/UX tradeoff.
