## Summary

Keep open: this active PR is not obsolete, because current main does not expose an opt-in timeout for post-connect MCP `tools/list` catalog discovery, and the branch appears to implement that path without changing default catalog behavior. The remaining gate is maintainer acceptance of the new public MCP config contract.

## What This Changes

The PR adds optional per-server MCP `toolsListTimeoutMs` config and threads it from schema/docs/transport resolution into bundle MCP catalog `client.listTools` calls, with runtime and config coverage.

## Best Possible Solution

Land the opt-in per-server catalog-list timeout if maintainers accept the config contract, while keeping default SDK request behavior unchanged and leaving broader MCP degradation work to https://github.com/openclaw/openclaw/issues/64411.

## Reproduction Assessment

Yes. The high-confidence path is a fake MCP server that connects, then delays or hangs `tools/list`; current main has no OpenClaw list-timeout option, and the PR body supplies after-fix terminal output for both default and explicit-timeout behavior.

## Solution Assessment

Yes, pending maintainer acceptance of the new config knob. A separate opt-in `toolsListTimeoutMs` is a narrow maintainable path because it bounds catalog listing without changing the documented connection-only role of `connectionTimeoutMs`.
