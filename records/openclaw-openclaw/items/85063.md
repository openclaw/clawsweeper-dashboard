## Summary

Keep open for maintainer review. The patch is a narrow, source-backed MCP availability fix with sufficient real behavior proof, but it deliberately broadens `connectionTimeoutMs` from connect-only wording into post-connect catalog listing, so maintainers should accept or clarify that compatibility tradeoff before merge.

## What This Changes

The PR passes the resolved MCP `connectionTimeoutMs` into each bundled MCP catalog `client.listTools` request and adds a stdio regression test for a server that initializes but never answers `tools/list`.

## Best Possible Solution

Land the narrow timeout fix with explicit maintainer-owned semantics that MCP catalog discovery uses the per-server budget, preserving per-server degradation and adding a separate list/catalog timeout only if slow-listing compatibility needs its own knob.

## Reproduction Assessment

Yes at source level. Current main calls `client.listTools` without request options after connect, the pinned SDK defaults unconfigured requests to 60000 ms, and the PR supplies a deterministic stdio reproduction that reaches `tools/list` and hangs.

## Solution Assessment

Yes, with one maintainer-policy caveat. Passing the SDK request timeout at the existing catalog call is the narrowest code fix for the hang, but maintainers should accept or clarify that `connectionTimeoutMs` now covers catalog listing too.
