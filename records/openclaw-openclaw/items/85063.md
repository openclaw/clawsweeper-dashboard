## Summary

Keep open for maintainer review; this recent PR is not redundant and the patch appears to preserve the default SDK catalog timeout while adding an opt-in bound for hung MCP `tools/list` discovery.

## What This Changes

The branch adds optional per-server `toolsListTimeoutMs` MCP config, threads it through MCP transport resolution to catalog `client.listTools` calls, and updates schema/docs/baseline/tests.

## Best Possible Solution

Land the opt-in per-server catalog-list timeout with the default SDK behavior preserved, and leave broader MCP degradation and circuit-breaker design to https://github.com/openclaw/openclaw/issues/64411.

## Reproduction Assessment

Yes, from source and supplied proof. Current main relies on the MCP SDK default request timeout for `tools/list`, and the PR body includes copied live output showing an explicit list timeout cancels a hung catalog request while the default keeps a slow catalog visible.

## Solution Assessment

Yes, pending maintainer acceptance of the new config knob. A separate opt-in `toolsListTimeoutMs` is the narrowest maintainable path because it bounds catalog listing without changing the documented connection-only meaning of `connectionTimeoutMs`.
