## Summary

Keep open for maintainer review: current main leaves post-connect MCP `tools/list` catalog discovery on the SDK default timeout, and this PR applies the existing per-server budget with a focused regression test and credible live-output proof. The remaining decision is compatibility: slow-but-working MCP catalogs may now degrade at `connectionTimeoutMs`, so that semantic change should be accepted or documented before merge.

## What This Changes

The PR passes the resolved MCP `connectionTimeoutMs` into catalog `client.listTools` requests and adds a stdio regression test for a server that initializes but never answers `tools/list`.

## Best Possible Solution

Ship the narrow timeout path with explicit semantics that MCP catalog discovery uses the per-server budget, preserving per-server degradation and avoiding a new config knob unless maintainers decide slow-listing compatibility needs a separate catalog timeout.

## Reproduction Assessment

Yes. Current main clearly calls `client.listTools` without request options, the pinned SDK defaults unconfigured requests to 60000 ms, and the PR includes a deterministic stdio server reproduction that reaches `tools/list` and hangs.

## Solution Assessment

Yes, the code change is the narrowest maintainable fix for the reported timeout gap because it uses the SDK's documented request options on the existing catalog call. The only solution-fit caveat is the broadened meaning of `connectionTimeoutMs`, which should be documented or explicitly accepted.
