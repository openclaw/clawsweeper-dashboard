## Label Justifications

- `P1`: A hung MCP catalog request can materially delay agent tool discovery in a high-impact integration path.
- `merge-risk: 🚨 compatibility`: Reusing `connectionTimeoutMs` for `tools/list` can degrade existing slow-but-working MCP servers that previously completed under the SDK default timeout.

## Summary

Keep open for maintainer review: the patch is a small, source-supported MCP availability fix with sufficient real behavior proof, but it intentionally changes compatibility semantics for slow catalog listings.

## What This Changes

The branch passes the resolved MCP `connectionTimeoutMs` into each bundled MCP `client.listTools` catalog page request and adds a stdio regression test for a server that initializes then hangs on `tools/list`.

## Best Possible Solution

Land the narrow bounded catalog-listing fix once maintainers make the timeout contract explicit, preserving per-server degraded catalog behavior and giving operators a clear tuning path for slow MCP discovery.

## Reproduction Assessment

Yes, at source level. Current main calls `client.listTools` without request options after a bounded connect, and the SDK default request timeout is 60000 ms when no timeout is supplied.

## Solution Assessment

Yes, with a compatibility caveat. Passing SDK request options at the existing catalog call is the narrowest code fix for the timeout gap, but maintainers should approve or document the expanded meaning of `connectionTimeoutMs`.
