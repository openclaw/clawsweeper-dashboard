## Label Justifications

- `P1`: Hung MCP catalog discovery can delay tool availability and the main agent reply path, matching an active high-priority MCP degradation class.

## Summary

No blocking code-review findings. The patch is focused and compatibility-preserving: current main leaves `tools/list` on the MCP SDK default timeout, while this PR only passes a shorter request timeout when the new config field is explicitly set; keep it open for maintainer merge and CI handling.

## What This Changes

The PR adds optional `mcp.servers.<name>.toolsListTimeoutMs`, threads it through MCP transport resolution, applies it to catalog `client.listTools` calls only when configured, and updates docs/schema/tests.

## Best Possible Solution

Land the opt-in catalog-list timeout contract with the compatibility-preserving default, and keep broader MCP degradation/circuit-breaker work in https://github.com/openclaw/openclaw/issues/64411.

## Reproduction Assessment

Yes. Source inspection confirms current main calls `client.listTools` without request options and the MCP SDK defaults such requests to 60 seconds; the PR body supplies before/after terminal output for slow and hanging stdio catalog cases.

## Solution Assessment

Yes. A separate opt-in `toolsListTimeoutMs` is the narrowest maintainable fix because it bounds hung catalog listing without changing the documented connection-only meaning of `connectionTimeoutMs`.
