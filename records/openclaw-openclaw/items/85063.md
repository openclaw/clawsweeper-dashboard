## Summary

Keep open for maintainer review. The PR targets a real MCP catalog availability bug and uses the SDK timeout API correctly, but it also changes how existing `connectionTimeoutMs` settings affect tool discovery, so the compatibility contract needs an explicit decision before merge.

## What This Changes

The PR passes the resolved MCP server timeout into each catalog `tools/list` request and adds a hanging stdio MCP server regression test.

## Best Possible Solution

Land a bounded `tools/list` timeout only after maintainers define the supported timeout semantics for catalog discovery, with docs or compatibility-preserving behavior matching that decision.

## Reproduction Assessment

Yes, at source level: current `main` omits SDK request options for `tools/list`, and the MCP SDK default request timeout is 60000 ms. The PR body also includes a real stdio child-process proof showing the hanging list request now cancels on the configured shorter budget.

## Solution Assessment

Not yet as-is. Passing SDK `timeout` is the right mechanism, but reusing a field currently documented as a connection timeout changes existing config semantics, so the best solution needs maintainer acceptance plus docs/tests or a compatibility-preserving catalog-timeout contract.
