## Summary

Keep this PR open: the MCP timeout gap is real and the patch hits the right runtime call site, but reusing the documented connection timeout for post-connect catalog listing can make existing slow MCP catalogs disappear after upgrade without an explicit maintainer decision.

## What This Changes

The PR passes the resolved MCP `connectionTimeoutMs` into each bundled MCP catalog `client.listTools` page request and adds a stdio regression test for a server that initializes then hangs on `tools/list`.

## Best Possible Solution

Preserve slow-catalog compatibility by default while adding a maintainer-approved catalog-listing timeout, or explicitly document `connectionTimeoutMs` as the full discovery budget with upgrade proof.

## Reproduction Assessment

Yes, at source level. Current main calls `client.listTools` without request options after the bounded connect phase, and the SDK contract defaults unconfigured requests to 60000 ms; the PR body also includes live output from a hanging stdio server path.

## Solution Assessment

No, not merge-safe as-is. Passing SDK request options at the catalog call is the narrow fix, but using the documented connection-only setting for a new post-connect phase needs compatibility handling or explicit maintainer approval.
