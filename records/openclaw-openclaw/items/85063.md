## Label Justifications

- `P1`: A hung MCP catalog request can materially delay the agent path, and this PR addresses that availability class in a high-impact integration surface.
- `merge-risk: 🚨 compatibility`: Reusing `connectionTimeoutMs` for `tools/list` can make existing slow-but-working MCP servers degrade where they previously completed under the SDK default timeout.

## Summary

Keep open for maintainer review: the patch is a narrow and likely correct availability fix with sufficient real behavior proof, but it intentionally broadens `connectionTimeoutMs` from connection-only wording into the post-connect catalog `tools/list` request budget.

## What This Changes

The PR passes the resolved MCP `connectionTimeoutMs` into bundled MCP catalog `client.listTools` calls and adds a stdio regression test for a server that initializes then hangs on `tools/list`.

## Best Possible Solution

Ship the bounded catalog-listing path after maintainer acceptance of the timeout semantics, ideally with docs or release-note wording that tells operators how to tune slow MCP catalog discovery.

## Reproduction Assessment

Yes, at source level. Current main calls `client.listTools` without request options after connection, while the pinned SDK defaults unconfigured requests to 60000 ms; the PR's stdio proof demonstrates the hanging post-connect path.

## Solution Assessment

Yes, with a compatibility caveat. Passing SDK request options at the existing catalog call is the narrowest fix for the timeout gap, but maintainers should approve or document the expanded meaning of `connectionTimeoutMs`.
