Full review comments:

- **[P1] Move persist out of the locked callback:** `src/cron/service/ops.ts:732`
  - body: `locked()` waits for the async callback to finish, so this `await persist(state)` still holds the cron store lock for slow disk I/O. The PR's stated non-blocking behavior is therefore unchanged; build `prepared` under the lock, return it, then persist/emit after `locked(...)` resolves if this separate fix is still needed.
  - confidence: 0.97
- **[P2] Make the regression assert the second run:** `src/cron/service/ops.regression.test.ts:520`
  - body: The new test only waits for `runStarted`, which resolves on the first run, so it can pass even if the second enqueue never gets past the first slow persist. It should prove the second queued run reaches execution or that the second acknowledgement is not delayed by the first persist.
  - confidence: 0.92
