## Summary

Keep open. Current main still has a source-backed updater path where a failed plugin update with `disableOnFailure` marks the plugin entry disabled but leaves dependent `plugins.allow` and `plugins.slots.contextEngine` state for later doctor-style cleanup; the related scanner report at https://github.com/openclaw/openclaw/issues/81501 is separate and does not supersede this stale-config bug.

## What This Changes

Ensure a beta plugin update fallback that is blocked by the plugin scanner does not leave stale plugin allow, entry, and context-engine slot config or misleading plugin-doctor output.

## Best Possible Solution

Keep the scanner fail-closed, then make the update failure path reset affected plugin-dependent config through existing stale-config semantics or emit explicit context-engine remediation that points to `openclaw doctor --fix`.

## Reproduction Assessment

Yes. Configure a plugin in `plugins.allow`, `plugins.entries`, and `plugins.slots.contextEngine`, force `updateNpmInstalledPlugins` to fail under `disableOnFailure: true`, and current main leaves allow/slot state behind while disabling only the entry.

## Solution Assessment

Yes. The narrow maintainable fix is in the updater/config repair boundary, while preserving scanner blocking and keeping the broader dependency trust policy separate.
