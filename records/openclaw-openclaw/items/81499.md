## Summary

Keep open. Current main still has a source-backed path where post-update plugin failure marks the plugin entry disabled without clearing dependent allow/slot state, while the separate stale-config repair path only runs under doctor/fix-style remediation.

## What This Changes

The issue reports that a beta plugin update can disable the external `lossless-claw` context-engine plugin after npm tag fallback and leave stale plugin config plus confusing doctor output.

## Best Possible Solution

Preserve the security scanner block, then make the post-update failure path either reset affected plugin slots through the existing stale-config repair semantics or emit a slot-specific remediation summary that points to `openclaw doctor --fix`.

## Reproduction Assessment

Yes. A focused source repro can configure a plugin in `plugins.allow`, `plugins.entries`, and `plugins.slots.contextEngine`, mock the npm update to fail under `disableOnFailure: true`, and observe that current main only disables the entry while dependent config remains.

## Solution Assessment

Yes, fixing this in the update/doctor boundary is the narrow maintainable direction. The scanner should keep blocking unsafe installs; the repair should address the stale context-engine state or make the existing doctor repair guidance explicit enough for the operator.

Fix the source-proven post-update plugin failure state from https://github.com/openclaw/openclaw/issues/81499. Keep the dangerous-code security scanner behavior intact. In the post-core plugin update path, when an installed plugin update is disabled after failure and that plugin is selected by `plugins.slots.contextEngine` or other dependent plugin config, either reset the dependent stale slot/policy using the existing stale-plugin repair semantics or emit a high-signal warning that names the affected slot and tells the operator to run `openclaw doctor --fix`. Do not add startup/load-time migrations. Add focused regression coverage for an npm plugin update failure after beta fallback/security-scan failure with `plugins.allow`, `plugins.entries`, and `plugins.slots.contextEngine` pointing at the plugin, plus coverage that `plugins doctor`/post-update messaging no longer ends in a misleading clean state for that stale context-engine condition. Add a changelog fix entry because this affects user-visible update behavior.
