## Summary

Keep this PR open but do not merge it as-is: it addresses a real P1 legacy upgrade failure, but the latest head still lets old updater parents write candidate-version config metadata after enabling legacy repair, and its after-fix proof is stale for the current head.

## What This Changes

## Best Possible Solution

Revise the marker split so legacy candidate doctor repair can run without allowing candidate-version config metadata writes before the old parent resumes, then attach current-head Docker or Package Acceptance upgrade proof for `openclaw@2026.5.16-beta.4` to the PR head.

## Reproduction Assessment

Yes, source-level with strong external proof: the linked issue gives a concrete `openclaw@2026.5.16-beta.4` package-upgrade path, and source inspection shows the shipped old-parent env shape plus the candidate config-write path. I did not rerun the Docker lane in this read-only review.

## Solution Assessment

No. The PR direction is right, but the latest head lets legacy parents repair while still allowing candidate-version config writes; the narrower solution is to separate repair capability from write permission.
