## Summary

Keep this PR open for maintainer handling: it is a focused fix for a real beta package-upgrade failure, but the current head has a blocking upgrade-handoff regression in the final marker split and the linked real-behavior proof predates the latest force-pushed head.

## What This Changes

The PR changes update/doctor env-marker handling, configured plugin install repair during legacy package upgrades, and ClawHub artifact-unavailable fallback handling with focused regression tests.

## Best Possible Solution

Revise the marker split so legacy parent handoffs can run current plugin repair and ClawHub fallback logic without letting the candidate doctor persist candidate-version config metadata, then attach current-head package-upgrade proof.

## Reproduction Assessment

Yes. The linked issue and PR proof trail give a concrete `openclaw@2026.5.16-beta.4` to candidate package-upgrade path, and source inspection confirms the shipped old-parent env shape that triggers the handoff failure.

## Solution Assessment

No. The direction is right, but the latest patch reclassifies parent-support-only legacy handoffs as repair-capable without also preventing candidate-version config writes; the safer solution is to keep the new defer marker separate from old parent write permission.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep legacy repair from stamping candidate config:** `src/commands/doctor/shared/update-phase.ts:58`
  - body: This change makes an old parent that sets `OPENCLAW_UPDATE_IN_PROGRESS` plus `OPENCLAW_UPDATE_PARENT_SUPPORTS_DOCTOR_CONFIG_WRITE` but not the new defer marker run configured-plugin repair instead of deferring it. Shipped parents such as v2026.5.12 use exactly that env shape, and `runWriteConfigHealth` still allows config writes whenever the parent-support marker is present. When the candidate doctor repairs a missing plugin, `maybeRunConfiguredPluginInstallReleaseStep` can now report `touchedConfig: true`, causing the candidate to write `meta.lastTouchedVersion` before the old parent resumes and recreating the reported future-stamped config failure. Gate config writes on the new marker too, or make the legacy repair path avoid stamping candidate metadata.
  - confidence: 0.9
