## Summary

Keep open: current main still has the reported conflict between the gateway auto-update scheduler and the updater's gateway-ancestry safety guard, so auto-update remains unable to complete from a standard systemd-managed gateway process tree.

## What This Changes

The issue reports that systemd-managed gateway auto-update repeatedly fails because the gateway spawns `openclaw update` as a child process that the update CLI then refuses to run.

## Best Possible Solution

Route gateway auto-updates through a supervisor-owned handoff outside the gateway process tree, preserve the manual ancestry guard, and surface failed handoffs through update status or clear operator diagnostics.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: the scheduler launches the updater as a gateway child, and the CLI explicitly blocks when the active gateway PID is in the updater ancestry. I did not run a live systemd package-update replay in this read-only pass.

## Solution Assessment

Yes, keeping this open for a handoff fix is the best path. The safer repair is to move trusted auto-update outside the gateway-owned process tree, not to weaken the package-update safety guard or rely only on marker stripping.
