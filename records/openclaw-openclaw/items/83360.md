## Summary

Keep open: current main still has the conflicting updater path. The gateway auto-update scheduler launches the CLI updater as a normal child of the gateway, while the update CLI still blocks package updates when the active gateway PID is in the updater ancestry; the open marker-stripping PR is related but does not by itself solve this systemd process-tree failure.

## What This Changes

The issue reports that gateway auto-update cannot complete under the managed systemd user service because the updater child is blocked by the gateway-ancestry safety guard.

## Best Possible Solution

Route gateway auto-updates through a supervisor-owned managed-service handoff or transient user service that runs outside the gateway process tree, preserve the manual in-service update guard, and surface failed handoffs in update status/logging.

## Reproduction Assessment

Yes, for a source-level reproduction: current main launches the auto-updater as a gateway child, and the update CLI blocks when the active gateway PID is an ancestor. I did not run a live systemd install in this read-only review.

## Solution Assessment

Yes, keeping this open for a focused handoff repair is the best path. The safer solution is to preserve the ancestry guard and move auto-update execution outside the gateway-owned process tree, rather than weakening the guard.
