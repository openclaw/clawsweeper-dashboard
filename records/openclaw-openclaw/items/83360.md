## Summary

Keep open: current main still has the reported conflict. The gateway scheduler runs the updater as a gateway child, while the update CLI refuses package updates when the active gateway PID is in the updater ancestry; the related marker-stripping PR is useful context but does not solve the systemd ancestry failure.

## What This Changes

The issue asks for gateway auto-update to work under the standard systemd service instead of repeatedly spawning an updater that the CLI safety guard refuses to run.

## Best Possible Solution

Route gateway auto-updates through a supervisor-owned handoff that can safely stop, update, and restart the managed service, while preserving the manual ancestry guard and surfacing failed handoffs in update status.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: the scheduler launches the updater as a gateway child, and the CLI explicitly blocks package updates when the active gateway PID is in that child’s ancestry. I did not run a live systemd update in this read-only pass.

## Solution Assessment

Yes, keeping this issue open for a service-owned handoff fix is the best path. The maintainable solution is not to weaken the safety guard, but to move trusted auto-update out of the gateway-owned process tree or through the existing managed-service handoff pattern.
