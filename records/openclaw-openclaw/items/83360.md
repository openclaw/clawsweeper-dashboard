## Label Justifications

- `P2`: The issue breaks documented auto-update behavior for managed systemd installs, but it has a manual external-shell workaround and limited blast radius.

## Summary

Keep open: current main still launches the gateway auto-updater as a normal gateway child process, while the update CLI still blocks package updates when the active gateway PID is in the updater ancestry. The related open marker-stripping PR is useful context, but it does not solve this systemd process-tree failure mode.

## What This Changes

Fix gateway auto-update so it can complete under the standard systemd service instead of spawning an updater child that the gateway-ancestry guard rejects.

## Best Possible Solution

Route gateway auto-updates through a supervisor-owned managed-service handoff that runs outside the live gateway process tree, preserves the manual `openclaw update` safety guard, and records/surfaces failed handoffs through update status.

## Reproduction Assessment

Yes for a source-level reproduction: the gateway scheduler spawns the updater as its child, `runCommandWithTimeout` does not detach it, and the CLI guard blocks when the gateway PID is an ancestor. I did not run a live systemd install in this read-only pass.

## Solution Assessment

Yes, keeping this open for a handoff fix is the best path. The safer repair is to move auto-update execution outside the gateway-owned process tree, not to weaken the existing operator safety guard.
