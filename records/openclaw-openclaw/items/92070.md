## Summary

Keep open: the isolation goal is useful, but the PR unconditionally forces a newly required external Claude CLI flag into every normal and resumed launch without an upgrade gate, docs/doctor path, or real launch proof, and the branch currently conflicts with `main`.

## What This Changes

The PR adds Claude Code `--safe-mode` to the Anthropic `claude-cli` backend defaults and re-applies it during backend config normalization.

## Best Possible Solution

Keep the Claude CLI isolation goal, but land it only with a maintainer-approved minimum-version policy or a narrow version/doctor/setup gate, focused regression coverage, and real fresh/resume launch proof.

## Reproduction Assessment

Yes by source inspection for the changed behavior: current `main` does not enforce `--safe-mode` for normal fresh/resumed Claude CLI launches, and the PR would add it after user override merging. I did not run a live Claude CLI launch in this read-only review.

## Solution Assessment

No; unconditional flag insertion is only a plausible implementation, not the best merge-ready solution, because OpenClaw currently uses the user's host-installed Claude CLI and this PR does not gate or document the new minimum version.
