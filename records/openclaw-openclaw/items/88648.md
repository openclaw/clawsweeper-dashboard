## Summary

## What This Changes

The PR disables startup progress for `--json`, adds a stdout guard before creating the Clack spinner, and adds a progress unit test for piped stdout suppression.

## Best Possible Solution

Route the Clack spinner to the selected progress stream or require `process.stdout.isTTY === true`, cover undefined/non-TTY stdout plus `runCli --json` startup behavior in tests, then attach redacted terminal proof showing clean JSON through `jq`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Require true TTY before creating the spinner:** `src/cli/progress.ts:108-109`
  - body: `process.stdout.isTTY` is not guaranteed to be `false` for a pipe; in this non-TTY Node process it is `undefined`. Because this guard uses `!== false`, `stdoutSafe` remains true and `spinner()` can still be created while Clack defaults its output to stdout, so a command with TTY stderr and piped stdout can still leak progress glyphs. Pass `output: stream` to Clack or treat only `process.stdout.isTTY === true` as safe, and cover the undefined case.
  - confidence: 0.92
- **[P2] Update the startup progress expectation:** `src/cli/run-main.ts:761`
  - body: Adding `enabled` changes the object passed to `createCliProgress` for normal full-CLI commands, but `src/cli/run-main.exit.test.ts` still expects the old exact object without that property. Update that expectation and add a focused `--json` case so the startup spinner suppression is covered instead of leaving this path to fail or drift in CI.
  - confidence: 0.86
