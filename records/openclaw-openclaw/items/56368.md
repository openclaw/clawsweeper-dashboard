## Summary

Keep open. Current main still formats ClawHub 429s as raw request failures without login guidance, and the provided timeline shows open PR #56394 already targets this issue with closing syntax, so this should stay open until that implementation is merged, replaced, or closed.

## What This Changes

The issue requests appending ClawHub login guidance to unauthenticated HTTP 429 failures from `openclaw skills search` and `openclaw skills install`.

## Best Possible Solution

Add the hint in the shared ClawHub request/error boundary by tracking whether a request used an auth token and appending login guidance only for unauthenticated 429 responses across JSON and archive-download paths, with regression coverage for unauthenticated 429, authenticated 429, and non-429 failures.

## Reproduction Assessment

Yes. A mocked `fetchImpl` returning HTTP 429 through `searchClawHubSkills`, `downloadClawHubPackageArchive`, or `downloadClawHubSkillArchive` with no resolved token should still produce the raw `ClawHub ... failed (429): Rate limit exceeded` message on current main.

## Solution Assessment

Yes. The shared ClawHub infra fix is the narrowest maintainable direction because CLI, gateway skill, and plugin install callers share this request/error path; PR #56394 appears to target that boundary and should be reviewed, landed, replaced, or closed.
