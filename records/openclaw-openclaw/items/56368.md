## What This Changes

Add a ClawHub login hint to unauthenticated HTTP 429 rate-limit errors shown by `openclaw skills search` and `openclaw skills install`.

## Best Possible Solution

Implement the hint in the shared ClawHub request/error boundary by tracking whether a request used an auth token and appending `npx clawhub login` guidance only for unauthenticated 429 responses, with regression coverage for JSON, package archive, skill archive, authenticated 429, and non-429 failures.

## Reproduction Assessment

Yes. A mocked `fetchImpl` returning HTTP 429 through `searchClawHubSkills`, `fetchClawHubSkillDetail`, `downloadClawHubPackageArchive`, or `downloadClawHubSkillArchive` follows the current body-only `ClawHubRequestError` path and should reproduce the raw message without a live ClawHub call.

## Solution Assessment

Yes. A shared infra fix is the narrowest maintainable solution because CLI, gateway, skill, and plugin ClawHub callers share this request/error boundary; CLI-only string postprocessing would miss other user-facing surfaces.
