## Summary

Keep open: current main still intentionally keeps auto-selected fallback models as durable session recovery state, and the requested turn-scoped mode would add a new public config policy that needs maintainer API/product approval. Related open PRs explore non-sticky fallback behavior, but none has merged or establishes the compatible `session` default plus opt-in `turn` contract requested here.

## What This Changes

The issue requests a configurable fallback persistence policy so successful model fallbacks can be scoped to one turn instead of becoming a durable session model override.

## Best Possible Solution

Add a documented, schema-validated fallback persistence policy only after maintainer approval, preserving durable `session` behavior as the compatibility default and adding focused runner, model-selection, schema, and docs coverage for an opt-in `turn` mode.

## Reproduction Assessment

Not applicable; this is a feature/config request rather than a broken existing behavior report. Source, docs, and tests on current main show durable auto fallback persistence is the defined behavior today.

## Solution Assessment

Unclear until maintainers choose the public config contract. The requested direction is plausible, but the maintainable path is an explicit compatibility-preserving policy rather than an unconditional rollback in the runner.
