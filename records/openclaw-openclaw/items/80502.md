## Summary

Keep open: current main and the latest release still use sticky auto fallback session overrides with periodic primary probing, and there is no opt-in per-turn fallback persistence policy or config key. The issue remains the canonical product/config tracker; the related open implementation PR changes broader semantics and still needs maintainer compatibility review.

## What This Changes

The issue requests an optional model fallback persistence mode where a successful fallback applies only to the current turn and does not remain as the session model override.

## Best Possible Solution

Define one opt-in fallback persistence contract, add it to the canonical agent model/config schema with docs, preserve current sticky behavior by default, and validate fresh-install plus upgrade/session-state behavior before merging an implementation.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection confirms current main and v2026.6.6 intentionally persist auto fallback state and do not expose the requested per-turn mode.

## Solution Assessment

Unclear until maintainers choose the contract. A default-preserving opt-in policy is safer than globally changing sticky fallback semantics, and the open implementation PR still needs compatibility and proof review.
