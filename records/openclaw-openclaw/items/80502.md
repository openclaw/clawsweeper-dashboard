## Summary

Keep open: this is a valid feature/config request, not a cleanup-close candidate. Current main deliberately keeps auto-selected fallback models pinned across later turns, and adding a per-turn policy would introduce a new fallback/session config contract that needs maintainer product judgment.

## What This Changes

The issue requests an optional fallback persistence mode so a successful model fallback can be scoped to one turn without leaving the fallback model as the active session override.

## Best Possible Solution

If maintainers want this mode, add a backward-compatible fallback persistence policy to the model config contract, keep the current session-persistent behavior as default, and document/test how the new mode interacts with session overrides, `/status`, heartbeat, and reset flows.

## Reproduction Assessment

Not applicable as a feature/config request. Source, tests, and docs clearly show the existing session-persistent behavior, but there is no broken existing contract to reproduce.

## Solution Assessment

Unclear until maintainers decide the config contract. The proposed `fallbackPersistence: "turn"` direction is plausible, but it is not yet proven to be the best shape or scope for model fallback/session semantics.
