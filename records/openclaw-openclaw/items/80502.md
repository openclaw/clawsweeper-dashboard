## Summary

Keep open: current main still documents and tests durable auto-fallback session overrides, while this issue asks for a new opt-in per-turn persistence policy. Related open PRs explore nonsticky or TTL-based variants, but none has landed as the configurable `fallbackPersistence` contract requested here.

## What This Changes

Add a configurable model fallback persistence policy so a successful fallback can be scoped to the current turn without replacing the session's active model selection.

## Best Possible Solution

Define and document an explicit fallback persistence policy, keep durable `session` behavior as the compatibility default, and add focused runner, schema, docs, and regression coverage for an approved `turn` mode.

## Reproduction Assessment

Not applicable; this is a feature/config request rather than a broken existing contract. Source, tests, and docs show the current durable fallback behavior is intentional on current main.

## Solution Assessment

Unclear until maintainers choose the API contract. The requested direction is plausible, but it should be implemented as an explicit policy rather than an unconditional rollback in the runner.
