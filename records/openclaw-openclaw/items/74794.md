## Summary

Keep open. The PR addresses a real current-main cron announce routing bug, but the proposed merge can still combine a caller-selected channel with a target inferred from a different current chat context, which is another privacy-sensitive misdelivery path.

## What This Changes

The PR changes cron add delivery inference so channel-only announce delivery can infer `delivery.to`, preserves explicit `delivery.to`, reasserts inferred targets after merging partial delivery objects, and adds regression tests for channel-only announce cases.

## Best Possible Solution

Keep the core cron-tool fix, but make route inference channel-aware: only merge a current-context target when its channel matches the caller's explicit channel, otherwise use a matching session-key inference or reject the channel-only request with a clear error instead of storing a mixed channel/target route.

## Reproduction Assessment

Yes. Current main can be inspected directly: `hasTarget` includes `delivery.channel`, so a channel-only announce delivery skips inference; the PR finding can be reproduced with a unit case that supplies `currentDeliveryContext` for one channel and `delivery.channel` for another.

## Solution Assessment

No. Narrowing the gate to explicit `to` is the right direction, but the merge step must keep `channel`, `to`, `accountId`, and `threadId` paired to the same inferred route before this is the safest maintainable fix.
