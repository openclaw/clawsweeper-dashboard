## Summary

Keep open. This PR targets a real, source-reproducible cron announce mis-route, and a production report in the discussion supports the symptom, but the current patch can persist an inferred target under a different explicit channel and still lacks real behavior proof.

## What This Changes

The branch changes cron add-time announce delivery inference so channel-only delivery objects can receive an inferred `delivery.to` and adds focused cron-tool regression tests.

## Best Possible Solution

Merge a guarded cron fix that preserves explicit `to`, infers only compatible targets, fails closed when no safe target can be inferred, and includes focused regression plus redacted real behavior proof.

## Reproduction Assessment

Yes. Current main source shows channel-only announce delivery skips inference, and the PR-specific mismatch is source-reproducible by combining a different explicit `delivery.channel` with an inferred session/context target.

## Solution Assessment

No as proposed. Gating inference on explicit `to` is the right direction, but the patch must guard channel compatibility or fail closed before persisting an inferred target under a caller-supplied channel.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Guard inferred targets against channel mismatches:** `src/agents/tools/cron-tool.ts:751-754`
  - body: When a caller supplies `delivery.channel` without `to`, this merge can overwrite `inferred.channel` while forcing `to` from the previous context/session. That can persist an explicit target for a different channel and bypass the downstream same-channel `lastTo` fallback guard; only reuse the inferred target when it is compatible with the explicit channel, or fail closed.
  - confidence: 0.86
