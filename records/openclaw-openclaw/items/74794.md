## Summary

Keep open. The PR targets a real source-reproducible cron misrouting bug on current main, but the patch should not merge until explicit-channel/inferred-target compatibility is enforced; it also needs the required changelog entry.

## What This Changes

The PR changes cron add-time delivery inference so channel-only announce deliveries populate `delivery.to`, and adds three regression tests in `cron-tool.test.ts`.

## Best Possible Solution

Merge a guarded fix that infers `to` only when the caller omitted `channel` or the explicit channel is compatible, preserves explicit `to`, adds the active-version changelog entry, and validates the focused cron tests plus the changed gate.

## Reproduction Assessment

Yes by source inspection. Current main deterministically skips inference for channel-only announce delivery, and the PR-specific mismatch follows from combining an inferred target from one context/session with a different caller-supplied `delivery.channel`.

## Solution Assessment

No as proposed. Gating inference on explicit `to` is the right direction, but the merge must guard channel compatibility before persisting `inferred.to` under a caller-supplied channel.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Guard inferred targets against channel mismatches:** `src/agents/tools/cron-tool.ts:751`
  - body: When a caller supplies `delivery.channel` without `to`, this merge can overwrite `inferred.channel` while forcing `to` from the previous context/session. That can persist an explicit target for a different channel and bypass the downstream same-channel fallback guard; only reuse the inferred target when it is compatible with the explicit channel.
  - confidence: 0.84
