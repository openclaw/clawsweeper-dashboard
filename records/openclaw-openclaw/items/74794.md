## Summary

Keep this PR open, but do not merge it as-is: current main still has the channel-only cron announce inference bug, while the proposed merge can persist an inferred `to` under an incompatible caller-supplied `delivery.channel` and lacks real behavior proof.

## What This Changes

The PR changes cron add-time announce delivery inference so channel-only delivery objects populate `delivery.to`, and adds focused cron-tool regression tests.

## Best Possible Solution

Merge a guarded cron fix that preserves explicit `to`, infers `to` only when the inferred channel is compatible with any explicit channel, handles missing inference sources deliberately, and includes focused regression plus redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main deterministically skips inference for channel-only announce delivery, and the PR-specific mismatch follows from combining an inferred target from one context/session with a different caller-supplied `delivery.channel`.

## Solution Assessment

No as proposed. Gating inference on explicit `to` is the right direction, but the merge must guard channel compatibility or fail closed before persisting an inferred target under a caller-supplied channel.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Guard inferred targets against channel mismatches:** `src/agents/tools/cron-tool.ts:751`
  - body: When a caller supplies `delivery.channel` without `to`, this merge can overwrite `inferred.channel` while forcing `to` from the previous context/session. That can persist an explicit target for a different channel and bypass the downstream same-channel fallback guard; only reuse the inferred target when it is compatible with the explicit channel, or fail closed.
