## Summary

Keep open. The PR targets a real current-main cron announce routing bug, but the proposed merge path can pair an inferred target from one channel with a different explicit `delivery.channel`, and the required user-facing changelog entry is still missing.

## What This Changes

The PR changes cron-tool add-time delivery inference to populate `delivery.to` for channel-only announce jobs and adds focused cron-tool regression tests.

## Best Possible Solution

Merge only after the inference code refuses to combine an inferred target with an incompatible explicit channel, adds the required changelog entry, and passes the targeted cron-tool tests plus the normal changed gate.

## Reproduction Assessment

Yes. Current main deterministically skips inference for channel-only announce delivery, and the PR diff can be source-reproduced with a current Matrix context plus caller `delivery.channel: "slack"`, which would persist a Slack delivery using the Matrix target.

## Solution Assessment

No as proposed. Gating on explicit `to` is the right direction for the reported same-channel case, but the merge needs a channel-compatibility guard before `inferred.to` is reasserted.

## Review Findings

Overall correctness: patch is incorrect

  - body: When a caller supplies `delivery.channel` without `to`, this merge can overwrite `inferred.channel` while forcing `to` from the previous context/session. A cron created from one channel with `{ channel: "slack" }` could persist a Slack delivery with a non-Slack target, bypassing the downstream same-channel fallback guard; only reuse the inferred target when it is compatible with the explicit channel.
