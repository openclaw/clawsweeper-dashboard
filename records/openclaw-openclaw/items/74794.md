## Summary

Keep open: the reported cron announce mis-route is valid, and production context reinforces it, but this PR is not merge-ready because the patch can persist an inferred target under an incompatible caller-supplied channel and it lacks the required changelog entry.

## What This Changes

The PR changes cron add-time delivery inference so channel-only announce deliveries populate `delivery.to`, and adds regression coverage in `src/agents/tools/cron-tool.test.ts`.

## Best Possible Solution

Merge a guarded cron inference fix that preserves explicit `to`, only persists an inferred target when it is compatible with any caller-supplied channel, handles no-source fallback intentionally, adds focused regression tests, and includes the active-version changelog entry.

## Reproduction Assessment

Yes by source inspection. Current main deterministically treats `delivery.channel` as a complete target, skips inference for channel-only announce jobs, and then downstream routing can fall back to session `lastTo`.

## Solution Assessment

No as proposed. Gating inference on explicit `to` is the right direction, but the patch also needs a channel-compatibility guard or fail-closed path before persisting `inferred.to` under a caller-supplied channel.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Guard inferred targets against channel mismatches:** `src/agents/tools/cron-tool.ts:751`
  - body: When a caller supplies `delivery.channel` without `to`, this merge can overwrite `inferred.channel` while forcing `to` from the previous context/session. That can persist an explicit target for a different channel and bypass the downstream same-channel fallback guard; only reuse the inferred target when it is compatible with the explicit channel, or fail closed.
  - confidence: 0.86

  - confidence: 0.88
