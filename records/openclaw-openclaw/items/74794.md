## Summary

Keep open: current main still has the channel-only cron announce inference bug, and the production comment reinforces that it matters, but this PR can persist an inferred target under an incompatible caller-supplied channel and still needs the required changelog entry.

## What This Changes

The PR changes cron add-time announce delivery inference to populate `delivery.to` when only `delivery.channel` is supplied, with focused regression tests in `src/agents/tools/cron-tool.test.ts`.

## Best Possible Solution

Update this PR with a guarded channel-compatible inference policy, focused cross-channel/no-source regression coverage, and a single-line active-version changelog entry before running the focused cron tests and changed gate.

## Reproduction Assessment

Yes by source inspection. Current main deterministically skips inference for channel-only announce delivery, and the PR-specific mismatch is reproducible by combining a caller-supplied channel with a different inferred context or session channel.

## Solution Assessment

No as proposed. Gating inference on explicit `to` is the right direction, but the merge must guard channel compatibility or fail closed before persisting `inferred.to` under a caller-supplied channel.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Guard inferred targets against channel mismatches:** `src/agents/tools/cron-tool.ts:751`
  - body: When a caller supplies `delivery.channel` without `delivery.to`, this merge can overwrite `inferred.channel` while forcing `to` from the previous context/session. That can persist an explicit target for a different channel and bypass the downstream same-channel fallback guard; only reuse the inferred target when it is compatible with the explicit channel, or fail closed.
  - confidence: 0.87

Repo policy requires a single-line `CHANGELOG.md` entry under the active release, but this PR only changes the cron tool and tests.
  - confidence: 0.89
