## Summary

Keep open: current main still has the channel-only cron announce inference bug, but this PR remains unsafe to merge because it can persist an inferred target under an incompatible explicit channel and it lacks after-fix real behavior proof.

## What This Changes

The PR changes cron add-time delivery inference so channel-only announce jobs can populate `delivery.to`, and adds focused cron-tool regression tests.

## Best Possible Solution

Merge a guarded cron fix against the current `resolveCronCreationDelivery` seam that preserves explicit `to`, only applies inferred targets when channel-compatible, fails closed when inference is impossible, and includes regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main still skips inference when only `delivery.channel` is set, and the PR-specific mismatch follows from combining an inferred target with a different caller-supplied channel before downstream resolution.

## Solution Assessment

No as proposed. Gating inference on explicit `to` is the right direction, but the patch must guard channel compatibility or fail closed before persisting an inferred target under a caller-supplied channel.
