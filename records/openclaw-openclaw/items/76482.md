## Summary

Keep open. Current main still has a source-level Gateway/UI gap: DeepSeek’s full runtime plugin declares `xhigh` and `max`, but the lightweight provider-policy artifact used by the thinking resolver fallback does not export `resolveThinkingProfile`, so `/think` can still fall back to the five base levels when the runtime plugin is not active.

## What This Changes

The issue requests DeepSeek V4 Pro `/think` menus and validation expose `off, minimal, low, medium, high, xhigh, max` instead of only the five base thinking levels.

## Best Possible Solution

Mirror the DeepSeek V4 thinking profile through a shared provider-owned lightweight policy surface so Gateway rows, Control UI, slash commands, and patch validation all see `xhigh` and `max` without forcing full plugin startup.

## Reproduction Assessment

Yes, by source inspection. A DeepSeek V4 session can reach Gateway session-row thinking resolution without an active DeepSeek runtime plugin, and the lightweight DeepSeek policy artifact lacks `resolveThinkingProfile`, which leaves the shared resolver on the five-level base fallback.

## Solution Assessment

No. The current main implementation covers the full DeepSeek runtime hook, but the narrower maintainable fix is to share/export the same profile from the lightweight provider-policy surface that Gateway thinking resolution already consults.
