## What This Changes

The issue asks for stable Control UI WebSocket sessions behind ingress-nginx oauth2-proxy trusted-proxy auth without using `gateway.controlUi.dangerouslyDisableDeviceAuth=true`.

## Best Possible Solution

Settle the trusted-proxy Control UI pairing contract, prove it against a real ingress-nginx/oauth2-proxy WebSocket path, then either land the narrow runtime/test fix or update docs to describe pairing-required behavior.

## Reproduction Assessment

Yes for the trusted-proxy pairing part by source inspection: current main computes `trustedProxyAuthOk` but still expects `PAIRING_REQUIRED` for a device-present Control UI session. No for the exact ingress-nginx `1006` loop without live deployment proof.

## Solution Assessment

Unclear until maintainers settle the auth boundary. If no-pairing trusted-proxy access is intended, https://github.com/openclaw/openclaw/pull/93581 is the narrow implementation path; otherwise the docs should be corrected to require pairing for scoped operator access.
