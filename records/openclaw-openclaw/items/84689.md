## Summary

Keep open: this is useful security hardening, but the current patch has a blocking auth-compatibility problem. The branch writes `secretref-managed` for every non-marker provider `apiKey`, while current contracts distinguish real SecretRef-managed markers from supported plaintext provider credentials.

## What This Changes

The PR adds models.json planning logic and tests to replace newly resolved plaintext provider `apiKey` values with the `secretref-managed` marker while preserving matching merge-mode existing keys.

## Best Possible Solution

Make models.json redaction source-aware: sanitize credentials proven to be runtime-resolved from SecretRef/env/auth sources, while preserving documented plaintext provider semantics or introducing an explicit redacted-auth contract with upgrade tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the narrow leak: current main serializes `finalProviders` directly into models.json, so a final provider object with a plaintext `apiKey` is written as plaintext. I did not run a local repro because this review is read-only, but the PR body includes after-fix writer output.

## Solution Assessment

No, not as written. The protection goal is valid, but the patch overgeneralizes marker persistence; the safer fix is to strip only source-proven runtime-resolved secrets and preserve or explicitly redesign plaintext provider compatibility.
