## Summary

Keep open. Current main still sends Linux/non-sips image optimization through the Sharp-backed media-understanding runtime, and there is no Linux `vipsthumbnail`/system-libvips fallback or guard-aware original-image fallback on main yet. The report overlaps the active optimizer-fallback cluster, but it adds a distinct Linux Common KVM/SSE2 compatibility angle that should not be closed as a generic duplicate before maintainers decide whether the fix is only the open under-cap fallback PR or a broader native/runtime fallback.

## What This Changes

The issue reports that Sharp image optimization fails on an SSE2-only virtualized Common KVM CPU and asks for out-of-the-box fallback behavior that survives OpenClaw updates.

## Best Possible Solution

Keep this issue open and use it to decide the Linux native-image compatibility policy: first land or supersede the guard-aware shared-media fallback from #74243, then separately decide whether OpenClaw should add a Linux libvips CLI backend, native crash isolation, or documented system-libvips repair for CPUs unsupported by Sharp prebuilts while preserving pixel-limit and MIME guards.

## Reproduction Assessment

Yes for the OpenClaw source path: on Linux/non-sips, make `sharp` import or resize throw, then `loadWebMedia()` reaches `optimizeImageToJpeg()` and can throw `Failed to optimize image`. The exact Common KVM/SSE2 hardware reproduction is not live-verified here.

## Solution Assessment

Unclear. The current main implementation is not sufficient, and #74243 is the narrowest safe first fix for under-cap originals, but the requested `vipsthumbnail`/system-libvips fallback is a broader runtime policy choice.
