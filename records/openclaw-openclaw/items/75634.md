## Summary

Keep open. Current main still sends Linux/non-sips JPEG resizing into the Sharp-backed media runtime and can still surface `Failed to optimize image`; #75732 is open, unmerged, and explicitly tracks this issue, so this issue should stay open until that PR or a replacement lands.

## What This Changes

Make image optimization work on Linux x64 Common KVM/SSE2-only CPUs by falling back from Sharp to libvips instead of requiring overwritten dist patches after updates.

## Best Possible Solution

Review, land, or replace #75732 with a bounded Linux image fallback that preserves pixel-limit guards, temp cleanup, MIME/quality behavior, and existing failure semantics.

## Reproduction Assessment

Yes for the source-level path: force the Sharp import or resize call to throw on Linux/non-sips, then exercise `optimizeImageToJpeg()` and current main can still throw `Failed to optimize image`. The exact SSE2-only KVM hardware path remains reporter and PR-author verified rather than live-verified here.

## Solution Assessment

Unclear because no fix is merged yet. A libvips CLI fallback directly addresses the report, but the accepted change should stay within the media image backend and preserve the current safety boundaries.
