## Summary

Keep open. Current main still has a reachable Linux/non-sips image optimization path that depends on the Sharp-backed media-understanding runtime and can surface the reported `Failed to optimize image` error when Sharp cannot load or resize. The issue is also now paired with open PR #75732, which explicitly claims to fix this report, so the correct cleanup path is to review or supersede that PR rather than close the issue first.

## What This Changes

The issue requests that OpenClaw image optimization gracefully handle Sharp prebuilt failures on Linux x64 SSE2-only virtualized CPUs without requiring a recurring manual patch after updates.

## Best Possible Solution

Keep this as the Linux Sharp-native compatibility tracker until maintainers either land #75732, fold its useful hardware-specific fallback into #74243, or choose a safer documented/runtime repair. The end state should handle Sharp load/resize failure without editing `dist/`, while preserving media size caps, decoded pixel guards, MIME correctness, temp-file cleanup, and clear diagnostics.

## Reproduction Assessment

Yes for the source-level path: on Linux/non-sips, make Sharp import or resize fail, then exercise image loading through `loadWebMedia()` or `optimizeImageToJpeg()` and it can throw `Failed to optimize image`. The exact Proxmox Common KVM/SSE2-only hardware reproduction was not live-verified here.

## Solution Assessment

Unclear. A maintained source-level fallback is the right direction, but the reporter's `dist/` patch is not maintainable and a system `vipsthumbnail` backend needs policy/security review; #75732 and #74243 are the concrete implementation candidates to compare.
