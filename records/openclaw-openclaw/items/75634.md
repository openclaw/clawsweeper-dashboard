## Summary

Keep open. Current main still routes Linux/non-sips image optimization through the Sharp-backed media-understanding runtime, with no Linux `vipsthumbnail` or system-libvips runtime fallback and no guard-aware original-image fallback on main. The issue overlaps the active optimizer-fallback work in #74243, but it adds a distinct unsupported-CPU/native-runtime policy question that should remain open for maintainer scope judgment.

## What This Changes

The issue requests an update-stable fallback so image optimization works when Sharp’s linux-x64 prebuilt cannot load on SSE2-only KVM CPUs.

## Best Possible Solution

Keep this open as the Linux native-image compatibility tracker: land or supersede #74243 for guard-preserving under-cap original fallback, then decide whether unsupported Linux CPUs also need a bounded system-libvips/`vipsthumbnail` backend, native crash isolation, or documented install-time repair while preserving size, MIME, and pixel-limit guards.

## Reproduction Assessment

Yes for the source-level failure path: on Linux/non-sips, make Sharp import or resize throw, then `loadWebMedia()` reaches `optimizeImageToJpeg()` and can throw `Failed to optimize image`. The exact SSE2-only KVM hardware reproduction was not live-verified in this read-only review.

## Solution Assessment

Unclear. The current main implementation is not sufficient, #74243 appears to be the narrowest safe first fallback for under-cap originals, and the requested `vipsthumbnail` fallback is a broader runtime/security policy choice rather than an obviously best one-line fix.
