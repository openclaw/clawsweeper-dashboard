## Summary

Keep open. Current main still routes Linux image optimization through the Sharp-backed media-understanding runtime, and a Sharp import/resize failure still bubbles into `Failed to optimize image` without a Linux `vipsthumbnail` or system-libvips fallback. The open optimizer-fallback PR is relevant and likely covers under-cap images, but this report adds a distinct SSE2-only/Common KVM compatibility angle that needs maintainer scope judgment before cleanup.

## What This Changes

The issue asks OpenClaw to handle image optimization on Linux VMs whose x64 CPU lacks Sharp's prebuilt CPU requirements, preferably with a graceful fallback that survives updates.

## Best Possible Solution

Keep the issue open as the Linux CPU/native-image compatibility tracker: review or land the guard-aware under-cap fallback from #74243 first, then decide whether OpenClaw should add a Linux system-libvips CLI backend, stronger native crash isolation, or documented repair guidance while preserving pixel-limit, MIME, temp-file, and cap guards.

## Reproduction Assessment

Yes for the current OpenClaw failure path: on Linux/non-sips, make Sharp import or resizing throw and `loadWebMedia()` can still reach `Failed to optimize image`. The exact Common KVM/SSE2 host was not live-reproduced here.

## Solution Assessment

Unclear. A `vipsthumbnail` fallback could address this host class, but it introduces binary invocation and guard-preservation work; #74243 is the narrowest safe first step for under-cap images, not a complete Linux native-backend decision.
