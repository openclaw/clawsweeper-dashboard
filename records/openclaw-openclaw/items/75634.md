## Summary

Keep open. Current main still routes default Linux JPEG optimization through the Sharp-backed media runtime with no `vipsthumbnail`/libvips fallback, and #75732 is an open unmerged closing PR, so this issue should remain open until that PR or a replacement lands.

## What This Changes

The issue requests a durable fallback when Sharp's Linux x64 prebuilt cannot run on SSE2-only KVM CPUs, so image uploads keep working after OpenClaw updates.

## Best Possible Solution

Review, land, or replace #75732 with a bounded Linux libvips fallback inside the media image backend that preserves existing safety guards and cleanup behavior.

## Reproduction Assessment

Yes for a source-level path: on the default Linux backend, force Sharp import or resize to throw and `optimizeImageToJpeg()` exhausts the resize grid before throwing `Failed to optimize image`. The exact SSE2-only Proxmox/N355 hardware path is reporter and linked-PR verified rather than live-verified here.

## Solution Assessment

Unclear because no fix is merged yet. A libvips CLI fallback directly matches the report, but the accepted change should stay in the media image backend and preserve pixel limits, backend overrides, and temp-file cleanup.
