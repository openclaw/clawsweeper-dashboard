## Summary

Keep open. Current main and the latest tagged release still route Linux/non-sips image optimization through the Sharp-backed media runtime with no `vipsthumbnail` or other Linux CLI fallback, and #75732 is an open unmerged PR with closing syntax for this issue. #74243 is related but complementary because it handles under-cap original-image fallback rather than the unsupported-CPU/native-runtime policy.

## What This Changes

The issue asks OpenClaw to handle Sharp prebuilt failures on SSE2-only virtualized Linux CPUs so image optimization works or falls back without recurring manual patches after updates.

## Best Possible Solution

Keep this issue open until #75732 or a replacement lands and ships. The accepted fix should preserve current pixel-limit, MIME, cap, and temp-directory safeguards while choosing a bounded Linux libvips fallback, a documented native-module repair path, or a complementary original-image fallback where #74243 applies.

## Reproduction Assessment

Yes for the source-level failure path: on Linux/non-sips, force the Sharp import or resize operation to throw, then exercise `loadWebMedia()` or `optimizeImageToJpeg()` and current main can throw `Failed to optimize image`. The exact SSE2-only KVM hardware reproduction still needs live maintainer or contributor verification.

## Solution Assessment

Unclear. Current main is not sufficient, #75732 is the direct implementation candidate, and #74243 is complementary, but the `vipsthumbnail` approach still needs maintainer review of guard and runtime policy before this can be considered solved.
