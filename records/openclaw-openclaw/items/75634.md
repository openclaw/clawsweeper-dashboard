## Summary

Keep open. Current main still routes Linux image optimization through the Sharp-backed media-understanding runtime, and neither main nor the latest release has a Linux `vipsthumbnail`/system-libvips fallback or a merged guard-aware original-image fallback. The active optimizer-fallback PR may cover some thrown optimizer failures, but this report adds a distinct Common KVM/SSE2 CPU compatibility case that needs maintainer scope judgment before cleanup.

## What This Changes

The issue reports Sharp image optimization failing on an SSE2-only virtualized Common KVM CPU and asks for fallback behavior that survives OpenClaw updates.

## Best Possible Solution

Keep this issue open as the Linux native-image compatibility policy tracker: land or supersede the shared-media under-cap fallback first, then decide whether OpenClaw should add a Linux libvips CLI backend, native crash isolation, or documented system-libvips repair for CPUs unsupported by Sharp prebuilts while preserving MIME, byte-cap, and pixel-limit guards.

## Reproduction Assessment

Yes for the OpenClaw source path: on Linux/non-sips, make `sharp` import or resize throw, then `loadWebMedia()` reaches `optimizeImageToJpeg()` and can throw `Failed to optimize image`. The exact Common KVM/SSE2 hardware reproduction is not live-verified here.

## Solution Assessment

Unclear. The current main implementation is insufficient, and #74243 is the narrowest safe first fix for under-cap originals, but the requested `vipsthumbnail` or system-libvips fallback is a broader runtime policy choice.
