## Summary

Keep open. Current main still routes Linux/non-sips image resizing through the Sharp-backed media runtime, and #75732 is an open, unmerged closing PR, so this issue should remain open until that PR or a replacement lands.

## What This Changes

The issue requests a durable fallback for image optimization when Sharp’s Linux x64 prebuilt binary cannot load on SSE2-only virtualized CPUs.

## Best Possible Solution

A bounded Linux fallback should live in the image backend while preserving pixel-limit checks, secure tempdir cleanup, and current media failure semantics.

## Reproduction Assessment

Yes for a source-level path: make the Sharp import or Sharp resize call fail on Linux/non-sips, and `optimizeImageToJpeg()` can exhaust its attempts and throw `Failed to optimize image`. The exact SSE2-only KVM hardware path remains reporter and PR-author verified rather than live-verified here.

## Solution Assessment

Unclear because no fix is merged yet. A libvips CLI fallback directly addresses the report, but the accepted change should stay inside the image backend and preserve the existing media safety boundaries.
