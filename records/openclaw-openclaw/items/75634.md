## Summary

Keep open. Current main still has no Linux `vipsthumbnail` fallback for the Sharp-backed image path, and #75732 is an open, unmerged closing PR for this issue rather than a shipped fix.

## What This Changes

The issue requests a durable Linux image-optimization fallback when Sharp’s prebuilt binary cannot run on SSE2-only virtualized CPUs.

## Best Possible Solution

Keep this issue open until #75732 or a safer equivalent lands with bounded Linux fallback behavior that preserves media guards, MIME handling, temp-file hygiene, and failure semantics.

## Reproduction Assessment

Yes for the source-level path: on Linux/non-`sips`, force Sharp import or resize to throw, then call `optimizeImageToJpeg()` or a media-loading path and current main can throw `Failed to optimize image`. The exact SSE2-only KVM hardware reproduction still needs live maintainer or contributor verification.

## Solution Assessment

Unclear until the open PR is reviewed. A Linux `vipsthumbnail` fallback directly addresses the report, but the maintainable fix must preserve pixel-limit checks, secured temp dirs, and optimizer failure policy; #74243 remains complementary for under-cap original-image fallback behavior.
