## Summary

Keep open. Current main still sends non-sips Linux image resizing through the Sharp-backed media runtime with no `vipsthumbnail` fallback, and #75732 is an open, unmerged closing PR for this issue.

## What This Changes

The issue requests a durable Linux fallback when Sharp’s prebuilt binary fails on SSE2-only virtualized CPUs so image uploads keep working after OpenClaw updates.

## Best Possible Solution

Keep this issue open until #75732 or a safer equivalent lands with bounded Linux fallback behavior that survives updates and preserves media guards.

## Reproduction Assessment

Yes for the source-level path: on Linux/non-sips, force Sharp import or resize to throw, then exercise `optimizeImageToJpeg()` or `loadWebMedia()` and current main can reach `Failed to optimize image`. The exact SSE2-only KVM hardware path still needs live maintainer or contributor verification.

## Solution Assessment

Unclear. A Linux `vipsthumbnail` fallback directly addresses the report, but the accepted fix should preserve pixel-limit checks, temp-file hygiene, MIME handling, and failure semantics; #74243 remains complementary for under-cap original-image fallback policy.
