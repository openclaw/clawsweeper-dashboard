## Summary

Keep open. Current main still routes Linux/non-sips image resizing through the Sharp-backed runtime and has no durable libvips/vipsthumbnail fallback; the linked closing PR remains open and unmerged, so this issue should stay open until that PR or a replacement lands.

## What This Changes

The issue asks OpenClaw to keep image optimization working on SSE2-only KVM hosts where Sharp's linux-x64 prebuilt cannot load, ideally with a durable fallback that survives updates.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/75732 with a bounded Linux fallback that preserves pixel limits, cleanup, and existing media safety behavior.

## Reproduction Assessment

Yes, for a high-confidence source-level path: force the Sharp import or resize call to throw on Linux/non-sips, then call `optimizeImageToJpeg()` and current main can exhaust resize attempts and throw `Failed to optimize image`. The exact SSE2-only KVM hardware path remains reporter and linked-PR verified rather than live-run here.

## Solution Assessment

Unclear because no fix is merged yet. A libvips CLI fallback directly addresses the report, but the accepted implementation should stay inside the image backend and preserve the existing media safety contracts; the separate root `sharp` package metadata work does not solve present-but-unloadable Sharp prebuilts.
