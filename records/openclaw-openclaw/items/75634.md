## Summary

Keep open. Current main still delegates Linux/non-sips JPEG optimization to the Sharp-backed media-understanding runtime and can still throw `Failed to optimize image`; open PR #75732 references this issue with closing syntax and remains unmerged, so the issue should stay open until that PR or a replacement lands.

## What This Changes

The issue asks OpenClaw image optimization to work or degrade gracefully on Linux x64 KVM/SSE2-only CPUs where Sharp's prebuilt binary fails, without requiring a dist patch after updates.

## Best Possible Solution

Review, land, or replace #75732 with a bounded Linux image fallback that preserves pixel-limit guards, temp cleanup, MIME handling, and current failure semantics.

## Reproduction Assessment

Yes for the source-level path: force the Sharp import or resize call to throw on Linux/non-sips, then exercise `optimizeImageToJpeg()` and current main can still throw `Failed to optimize image`. The exact SSE2-only KVM hardware path remains reporter/PR-author verified rather than live-verified here.

## Solution Assessment

Unclear. A libvips CLI fallback directly addresses the report, but the accepted fix should stay narrowly in the image backend and preserve existing media safety boundaries.
