## Summary

Keep open. Current main still routes Linux/non-sips JPEG optimization through the Sharp-backed media-understanding runtime and can still surface `Failed to optimize image`; the linked closing PR remains unmerged, so the issue should stay open until that PR or a replacement lands.

## What This Changes

The issue asks OpenClaw to handle Sharp prebuilt binary failures on SSE2-only virtualized Linux CPUs with a durable runtime fallback that survives updates.

## Best Possible Solution

Review, land, or replace #75732 with a bounded Linux image fallback that preserves pixel-limit guards, temp cleanup, MIME handling, process timeouts, and current error semantics.

## Reproduction Assessment

Yes for the source-level path: on Linux with the non-sips backend, force the Sharp import or resize call to throw and `optimizeImageToJpeg()` can still end at `Failed to optimize image`. The exact SSE2-only KVM reproduction remains hardware-specific rather than live-verified here.

## Solution Assessment

Unclear until the linked PR is reviewed. A libvips CLI fallback directly addresses the report, but the accepted fix should stay inside the image backend and preserve existing media safety boundaries.
