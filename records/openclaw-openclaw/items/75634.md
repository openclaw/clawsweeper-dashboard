## Summary

Keep open. Current main still has a reachable Linux/non-sips image path that depends on the Sharp-backed media-understanding runtime and converts total resize failure into the reported `Failed to optimize image` error, with no `vipsthumbnail` runtime fallback. The issue is now paired with open PR #75732, which uses closing syntax for this issue, so the right cleanup path is to review, land, or supersede that PR rather than closing the issue before merge.

## What This Changes

Request that OpenClaw handle Sharp prebuilt failures on SSE2-only Linux/KVM CPUs so image uploads keep working after updates without manual dist patches.

## Best Possible Solution

Keep this issue as the Linux native-image compatibility tracker until #75732 is reviewed and either landed or replaced, while coordinating with #74243 so optimizer failures remain guard-preserving and do not weaken media size, MIME, or decoded-pixel safety boundaries.

## Reproduction Assessment

Yes for the source-level path: on current main, a Linux/non-sips Sharp import or resize failure bubbles through `resizeToJpeg()` and makes `optimizeImageToJpeg()` throw `Failed to optimize image`. The specific SSE2-only KVM machine was not live-verified here.

## Solution Assessment

Unclear. A `vipsthumbnail` fallback is a plausible fix for this reporter, but it is a broader runtime and security-policy choice than the already-open guarded original-image fallback in #74243, so the best implementation should be decided through review of #75732 or a replacement PR.
