## Summary

Keep open. Current main still has a reachable Linux/non-sips image optimization path that depends on Sharp with no `vipsthumbnail` or system-libvips runtime fallback, and an open PR (#75732) uses closing syntax for this issue, so the issue should stay open until that PR lands or is closed.

## What This Changes

The issue requests a durable Linux image-optimization fallback for Sharp prebuilt failures on SSE2-only Common KVM CPUs so image uploads keep working after OpenClaw updates.

## Best Possible Solution

Keep this issue as the Linux Sharp/native-image compatibility tracker, review #75732 as the direct fix candidate, and ensure any accepted fallback preserves MIME handling, byte caps, decoded-pixel guards, temp-file cleanup, and clear diagnostics.

## Reproduction Assessment

Yes for the source-level failure path: on Linux/non-sips, make Sharp import or resize throw, then `loadWebMedia()` reaches `optimizeImageToJpeg()` and can throw `Failed to optimize image`. The exact SSE2-only KVM hardware reproduction was not live-verified here.

## Solution Assessment

Unclear. #75732 is the direct implementation of the requested `vipsthumbnail` fallback, while #74243 is the narrower guard-aware original-image fallback; maintainers should choose the final policy before this issue closes.
