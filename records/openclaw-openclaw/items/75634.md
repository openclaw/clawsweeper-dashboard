## Summary

Keep open. Current main still routes Linux/non-sips JPEG resizing through the Sharp-backed media-understanding runtime and can still surface `Failed to optimize image`; #75732 is an open, unmerged closing PR, so this issue should stay open until that PR or a replacement lands.

## What This Changes

Request graceful image optimization fallback on Linux x64 Common KVM/SSE2-only CPUs when Sharp's prebuilt binary cannot load, without manual dist patches being overwritten by updates.

## Best Possible Solution

Review, land, or replace #75732 with a bounded Linux image fallback that preserves pixel-limit guards, temp cleanup, MIME/quality behavior, and existing failure semantics.

## Reproduction Assessment

Yes for the source-level path: force the Sharp import or resize call to throw on Linux/non-sips, then exercise `optimizeImageToJpeg()` and current main can still throw `Failed to optimize image`. The exact SSE2-only KVM hardware path remains reporter/PR-author verified rather than live-verified here.

Unclear because no fix is merged yet. A libvips CLI fallback directly addresses the report, but the accepted change should stay within the image backend and preserve the existing media safety boundaries.
