## Summary

Keep open. Current main still routes non-sips Linux image resizing through the Sharp-backed media runtime with no durable libvips/vipsthumbnail fallback, and the linked closing PR remains open and unmerged.

## What This Changes

The issue requests a durable fix for Sharp linux-x64 prebuilt failures on SSE2-only KVM CPUs so image optimization keeps working after OpenClaw updates.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/75732 with a bounded Linux fallback inside the image backend that preserves current media safety contracts.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: make the Sharp-backed resize call throw on Linux/non-sips, then `optimizeImageToJpeg()` exhausts resize attempts and throws `Failed to optimize image...`. The exact N355/Common KVM CPU failure remains reporter and linked-PR verified rather than live hardware verified here.

## Solution Assessment

Yes for the direction: a Linux fallback inside the existing image backend is the narrow maintainable fix. The exact open PR still needs review for guard parity around pixel limits, temp files, and error behavior.
