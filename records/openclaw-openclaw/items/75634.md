## Summary

Keep open. Current main still sends Linux/non-sips image optimization through the Sharp-backed media runtime with no `vipsthumbnail` fallback, and the direct fix PR #75732 is open and unmerged with closing syntax for this issue. #74243 is related but only covers under-cap original-image fallback and does not settle the unsupported Sharp/libvips CPU path.

## What This Changes

Requests resilient image optimization on Linux KVM/SSE2-only CPUs where Sharp's prebuilt binary fails, without manual `dist` patching after each OpenClaw update.

## Best Possible Solution

Keep this issue open while maintainers review #75732 or a consolidated media fallback path. The accepted fix should preserve image cap, MIME, HEIC, and pixel-limit guards while deciding whether Linux should use a bounded system-libvips backend, an under-cap original-image fallback like #74243, or documented native-module repair.

## Reproduction Assessment

Yes for the source-level path: on Linux/non-sips, force the Sharp import or resize call to throw, then exercise `loadWebMedia()` or `optimizeImageToJpeg()` and current main can throw `Failed to optimize image`. The exact SSE2-only KVM environment still needs live maintainer or contributor verification.

## Solution Assessment

Unclear. Current main is not sufficient, #75732 is the direct implementation candidate, and #74243 is complementary; the best fix still needs guard and runtime-policy review before this can close.
