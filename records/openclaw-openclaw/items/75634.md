## Summary

Keep open. Current main still routes Linux/non-sips JPEG resizing through the Sharp-backed media runtime and can still surface `Failed to optimize image`; #75732 is an open, unmerged closing PR, so this issue should remain open until that PR or a replacement lands.

## What This Changes

The issue asks for image upload and optimization to keep working on Linux x64 SSE2-only Common KVM CPUs when Sharp's prebuilt binary fails, preferably via a graceful libvips fallback that survives updates.

## Best Possible Solution

Review, land, or replace #75732 with a bounded Linux image fallback inside the media backend that preserves the existing media safety guards and cleanup behavior.

## Reproduction Assessment

Yes for a source-level path: force the Sharp import or resize call to throw on Linux/non-sips, then `optimizeImageToJpeg()` can exhaust its attempts and throw `Failed to optimize image`. The exact SSE2-only KVM hardware path remains reporter and PR-author verified rather than live-verified here.

## Solution Assessment

Unclear because no fix is merged yet. A libvips CLI fallback directly addresses the report, but the accepted change should stay inside the image backend and preserve the existing media safety boundaries.
