## Summary

Keep open. Current main still routes Linux/non-sips JPEG resizing into the bundled Sharp runtime and can surface `Failed to optimize image` when every resize attempt fails; #75732 is an open, unmerged closing PR for this same Common KVM/SSE2 Sharp fallback, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests a persistent image-optimization fallback when Sharp prebuilts fail on SSE2-only Common KVM CPUs.

## Best Possible Solution

Land or replace #75732 with a bounded Linux fallback that preserves pixel-limit guards, temp cleanup, MIME handling, and existing failure semantics.

## Reproduction Assessment

Yes for the source-level failure path: force the Sharp import or resize operation to fail on Linux/non-sips and exercise `optimizeImageToJpeg()`, which can still throw `Failed to optimize image`. The exact SSE2-only KVM hardware reproduction still needs live maintainer or contributor verification.

## Solution Assessment

Unclear. A libvips CLI fallback directly addresses the report, but the maintainable fix should stay narrowly in the image backend and preserve the existing media safety boundaries.
