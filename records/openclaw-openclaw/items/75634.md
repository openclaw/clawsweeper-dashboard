## Summary

Keep open. Current main still sends Linux/non-sips JPEG optimization through the Sharp-backed media runtime, so the reported `Failed to optimize image` path remains reachable; #75732 is an open, unmerged closing PR and should be reviewed or replaced before this issue closes.

## What This Changes

This issue requests a durable Linux image optimization fallback when Sharp’s linux-x64 prebuilt cannot load on SSE2-only Common KVM CPUs.

## Best Possible Solution

Review, land, or replace #75732 with a bounded Linux image fallback that preserves pixel-limit guards, temp cleanup, MIME/quality behavior, and existing failure semantics.

## Reproduction Assessment

Yes for the source-level path: force the Sharp import or resize call to throw on Linux/non-sips, then exercise `optimizeImageToJpeg()` and current main can still throw `Failed to optimize image`. The exact SSE2-only KVM hardware path remains reporter and PR-author verified rather than live-verified here.

## Solution Assessment

Unclear because no fix is merged yet. A libvips CLI fallback directly addresses the report, but the accepted change should stay inside the image backend and preserve the existing media safety boundaries.
