## Summary

Keep open. Current main still routes Linux/non-sips image optimization through the Sharp-backed media-understanding runtime, and neither current main nor v2026.4.29 has a `vipsthumbnail`/system-libvips fallback or a loader-level original-image fallback for optimizer backend failures. The related #74243 PR is relevant, but it covers the narrower under-cap optimizer-throw path and does not settle the Linux CPU/native backend policy raised here.

## What This Changes

The issue reports that Sharp image optimization fails on Linux KVM/SSE2-only CPUs and asks for a durable fallback instead of update-overwritten dist patches.

## Best Possible Solution

Keep this issue open as the Linux native-image compatibility policy tracker: first review or supersede #74243 for guard-aware under-cap fallbacks, then decide whether OpenClaw should add a Linux libvips CLI backend, native crash isolation, dependency/runtime guidance, or a narrower original-byte fallback while preserving byte caps, MIME handling, HEIC behavior, and decoded-pixel guards.

## Reproduction Assessment

Yes for the source-level failure path: on Linux/non-sips, make `sharp` import or resize throw, then a default `loadWebMedia()` image flow reaches `optimizeImageToJpeg()` and can throw `Failed to optimize image`. The exact Common KVM/SSE2 hardware reproduction is not live-verified here.

## Solution Assessment

Unclear. The current main implementation is not sufficient, and #74243 is the narrowest safe first fix for sendable under-cap originals, but the requested `vipsthumbnail`/system-libvips fallback is a broader runtime policy choice.
