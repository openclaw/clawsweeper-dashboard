## Summary

Keep open. Current main still routes Linux/non-sips image resizing through the Sharp-backed media-understanding runtime with no Linux `vipsthumbnail` fallback, and the direct fix PR #75732 is open and unmerged with closing syntax for this issue. #74243 remains complementary for under-cap original-image fallback but does not fully decide the unsupported-CPU/native-runtime policy.

## What This Changes

The issue requests an update-stable fallback so image optimization works when Sharp’s Linux x64 prebuilt cannot load on SSE2-only KVM CPUs.

## Best Possible Solution

Keep this issue open until #75732 is revised and either lands or is closed; the accepted fix should preserve image size, MIME, and pixel-limit guards while deciding whether Linux should use a bounded system-libvips backend, a safer original-image fallback from #74243, or documented native-module repair.

## Reproduction Assessment

Yes for the source-level path: on Linux/non-sips, force the Sharp import or resize call to throw, then exercise `loadWebMedia()` or `optimizeImageToJpeg()` and the current code can throw `Failed to optimize image`. The exact SSE2-only KVM hardware path remains to be live-verified by maintainers or the linked PR author.

## Solution Assessment

Unclear. Current main is not sufficient, #75732 is the direct implementation candidate, and #74243 is a safer complementary fallback for under-cap originals, but the system `vipsthumbnail` backend still needs guard and runtime-policy review before this issue can close.
