## Summary

Keep open: current main and v2026.6.10 still use Android realtime Talk capture without selecting or reporting the Bluetooth input route, and no same-repo canonical fix PR now owns the headset microphone problem.

## What This Changes

Android realtime Talk should use or expose the connected Bluetooth headset microphone instead of silently capturing from the phone microphone.

## Best Possible Solution

Fix the Android Talk capture owner path so it selects and restores the appropriate Bluetooth communication/input route when available, reports the active route if useful, and proves behavior on a real Android Bluetooth headset setup.

## Reproduction Assessment

Yes at source and reporter-evidence level: current main records Talk audio with `AudioRecord` and no route selection, while the issue includes adb proof that the active input was the built-in mic. I did not run the Pixel/Bluetooth hardware reproduction myself.

## Solution Assessment

Unclear pending Android route design: the fix belongs in Android Talk capture rather than Gateway/provider code, but maintainers should choose automatic routing, active-route display, and fallback UX before implementation.
