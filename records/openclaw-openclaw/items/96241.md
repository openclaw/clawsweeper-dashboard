## Summary

Keep open: current main and the latest release still record Android realtime Talk with a plain `AudioRecord` path and no Bluetooth/input-route selection, matching the reporter's ADB and room-distance reproduction. The issue remains actionable but should stay in maintainer review because the safe Android route policy, permission surface, and proof bar need product/owner choice before a fix PR.

## What This Changes

The issue reports that Android realtime Talk plays assistant audio through Bluetooth earbuds but captures user speech from the phone microphone instead of the connected headset microphone.

## Best Possible Solution

Fix the Android voice capture owner path so Talk chooses and restores the correct Bluetooth headset input route when available, considers sibling mic/transcription capture, exposes the active route if maintainers want that UX, and validates on a real Android Bluetooth headset setup.

## Reproduction Assessment

Yes at source and reporter-evidence level: current main records realtime Talk audio with `AudioRecord` and no input-route selection, and the issue includes ADB proof that OpenClaw was recording from the built-in mic. I did not run the Pixel/Bluetooth hardware reproduction myself.

## Solution Assessment

Unclear pending maintainer Android route design: the fix belongs in Android capture, not Gateway or provider routing, but automatic routing versus picker/status UI and permission cleanup need an owner decision.
