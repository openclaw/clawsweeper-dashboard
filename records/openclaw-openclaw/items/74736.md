## Summary

Keep open. Current main still emits a flat bundled-channel load warning with no structured lock-timeout metadata, and this active PR is a narrow, useful observability fix with passing checks; the only concrete issue I found is a low-severity parser edge in the new `lockDir` metadata extraction.

## What This Changes

The PR changes `src/channels/plugins/bundled.ts` to classify bundled runtime-deps lock-timeout load failures into structured warning metadata and adds `src/channels/plugins/bundled.test.ts` coverage for lock-timeout versus generic import-error warning behavior.

## Best Possible Solution

Land a narrow observability fix that preserves the complete timeout diagnostic in the warn message and adds reliable structured metadata for JSON logs, while leaving broader runtime-deps recovery behavior tracked separately under #73874.

## Reproduction Assessment

Yes. The PR includes a focused unit reproduction by mocking the bundled channel module loader to throw the stable lock-timeout message and asserting the warning shape; current main verification shows the same catch branch still lacks structured metadata.

## Solution Assessment

Yes, with one small caveat. Catch-side classification is the narrowest maintainable fix for the observable logging gap, but `lockDir` parsing should preferably stop at the stable ` (waited=` marker or use a typed error later to avoid misleading path metadata.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.88

Full review comments:

- **[P3] Parse lockDir up to the waited marker:** `src/channels/plugins/bundled.ts:298-299`
  - body: The helper stops at the first ` (` after the timeout prefix, so a valid path like `/tmp/OpenClaw (prod)/...` would produce truncated `lockDir` metadata while still classifying the timeout. Since this field is meant for structured log filtering, parse up to the stable ` (waited=` marker or avoid returning a partial path.
  - confidence: 0.84
