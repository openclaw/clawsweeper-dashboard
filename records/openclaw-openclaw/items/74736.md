## Summary

Keep open. Current main still collapses bundled-channel lock-timeout load failures into the same flat warning as ordinary import failures, and this PR is a narrow, clean observability fix with focused tests and green GitHub check runs. Before merge, it should add the required changelog entry and tighten the best-effort lockDir parser so structured logs do not emit a misleading partial path.

## What This Changes

The PR adds bundled runtime-deps lock-timeout detection and structured warning metadata in the bundled channel loader, plus tests for lock-timeout and generic import-error warning behavior.

## Best Possible Solution

Land a narrow observability fix that preserves the existing channel-skip behavior, keeps the full timeout diagnostic visible in non-JSON logs, emits reliable structured metadata for JSON/file consumers, and leaves broader runtime-deps recovery under the related operator issue.

## Reproduction Assessment

Yes. Current main can be reproduced by forcing loadGeneratedBundledChannelEntry to catch an Error whose message starts with the runtime-deps lock-timeout prefix; the existing catch branch emits only a flat warning, and the PR’s new test mocks that exact path.

## Solution Assessment

Yes, with small follow-ups. Catch-side classification is the narrowest maintainable fix for this logging gap, but parsing should use the stable ` (waited=` marker and the operator-visible fix needs the required changelog entry before merge.
