## Summary

Keep open. The observability goal is valid and the earlier changelog/parser follow-ups appear addressed, but this PR now instruments a lazy bundled-channel entry path where current main disables runtime-deps installation, while the real lock-timeout errors are thrown from bundled runtime entry loading through the plugin loader. The PR head is also currently dirty against main.

## What This Changes

The PR adds structured bundled-channel lock-timeout warning metadata, regression tests for lock-timeout versus generic import-error warning behavior, and a changelog entry.

## Best Possible Solution

Add structured lock-timeout metadata at the actual bundled runtime-deps load failure surface, preserve the full timeout detail in human-readable logs, keep the existing channel-skip behavior unchanged, and retain the focused changelog/test coverage after rebasing onto current main.

## Reproduction Assessment

Unclear for the production path. The PR test can force the lazy entry catch branch to see a timeout-shaped Error, but current main disables runtime-deps acquisition in that path, so it is not a high-confidence reproduction of a real bundled runtime-deps lock timeout reaching that catch.

## Solution Assessment

No. Catch-side classification is a good direction, but this implementation applies it to the wrong catch branch; the safer fix is to classify the timeout where bundled entry exports are loaded with runtime-deps installation enabled and recorded through the plugin loader.
