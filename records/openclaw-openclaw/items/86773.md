## Label Justifications

- `P1`: The report describes a current-release restart-time regression that can make local gateway control-plane calls time out for affected users.
- `impact:auth-provider`: The implicated work is provider auth-state prewarm and provider auth discovery.
- `impact:crash-loop`: The gateway process remains alive but becomes effectively unavailable through event-loop starvation and RPC timeouts.

## Summary

Keep open. Current main still schedules provider-auth prewarm in the gateway process and the warm path still derives providers from the full model catalog for every configured agent; related PRs are open mitigations, not merged fixes for the root event-loop starvation report.

## What This Changes

The issue asks OpenClaw to keep gateway RPCs responsive after restart by bounding, scoping, yielding, or disabling provider-auth prewarm work.

## Best Possible Solution

Bound the default prewarm path by scoping it to configured providers and chunking/yielding the work; treat operator controls as a reviewed mitigation rather than the only root fix.

## Reproduction Assessment

No live high-confidence current-main reproduction was run. The issue has strong v2026.5.22 logs and current source still contains the full-catalog provider-auth warm path, so it is source-reproducible at medium confidence.

## Solution Assessment

Unclear. A new config/env toggle is a useful mitigation but needs maintainer product and compatibility review; the root fix should make the existing warm path bounded, scoped, yielding, and cancellable by default.
