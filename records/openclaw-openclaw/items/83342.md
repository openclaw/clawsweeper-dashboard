## Summary

Keep open. Current main can still build parallel canonical-provider and authenticated CLI-runtime provider rows for the same model IDs, while tests and changelog show CLI-runtime provider visibility was intentionally restored, so resolving the duplicate picker display needs a maintainer UX decision rather than cleanup closure.

## What This Changes

The issue requests that `/models` avoid showing the same model under both canonical provider and CLI runtime alias provider banners when per-model `agentRuntime` is configured.

## Best Possible Solution

Define whether canonical provider rows or runtime alias rows own these model refs in browse views, then implement one shared dedupe/grouping rule in `buildModelsProviderData` with Telegram and command coverage while preserving CLI runtime discoverability.

## Reproduction Assessment

Yes, at source level: current main adds normal visible catalog rows and authenticated CLI-runtime catalog rows into the same `/models` provider map, and tests assert both provider families appear. I did not run a live Telegram picker in this read-only review.

## Solution Assessment

Unclear until maintainers choose the display contract. A new config flag would add avoidable surface area; the cleaner path is a shared dedupe or grouping rule that keeps the intentional CLI-runtime access restored for `/models`.
