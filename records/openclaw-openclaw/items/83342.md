## Summary

Keep open. Current main still has a source-backed path that can show canonical provider rows and authenticated CLI runtime provider rows for the same model, while the tests and changelog show the CLI runtime provider surface was restored intentionally, so this needs a maintainer UX decision before a cleanup fix.

## What This Changes

The issue requests that the `/models` browse picker avoid listing the same runtime-backed model under both the canonical provider and the CLI runtime alias provider.

## Best Possible Solution

Define the shared `/models` display contract for canonical provider refs versus runtime aliases, then implement one dedupe or grouping rule in `buildModelsProviderData` that preserves authenticated CLI runtime selection and covers Telegram plus the shared command tests.

## Reproduction Assessment

Yes, by source inspection: the provider-data builder adds canonical visible catalog rows and then authenticated CLI runtime catalog rows, and current tests assert both banners can appear. I did not run a live Telegram picker in this read-only review.

## Solution Assessment

Unclear until maintainers choose the display contract. A new config flag would add avoidable surface area; the cleaner path is likely shared dedupe or grouping in `buildModelsProviderData` while preserving the restored CLI runtime access.
