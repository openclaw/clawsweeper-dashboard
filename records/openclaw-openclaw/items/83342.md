## Summary

Keep open. Current main still builds `/models` provider data in a way that can show authenticated CLI runtime providers alongside canonical providers, and existing tests/changelog show that visibility was intentionally restored, so deduping needs a maintainer UX decision before a repair PR.

## What This Changes

Request to deduplicate `/models` picker entries when canonical provider models also have CLI runtime aliases configured.

## Best Possible Solution

Define the shared `/models` display contract for canonical provider refs versus runtime aliases, then implement one shared dedupe/grouping rule in `buildModelsProviderData` with Telegram/Discord coverage.

## Reproduction Assessment

Yes, by source inspection: the shared provider-data builder adds canonical visible catalog rows and then authenticated CLI runtime catalog rows, and current tests assert both provider banners can appear. I did not run a live Telegram picker in this read-only review.

## Solution Assessment

Unclear until maintainers choose the display contract. A config flag would add avoidable surface area; the cleaner path is likely shared dedupe/grouping in the provider-data builder while preserving the intentional CLI runtime access restored for `/models`.
