## Summary

Keep open. Current main still builds `/models` provider data in a way that can show canonical provider rows alongside authenticated CLI runtime provider rows, while tests and changelog show CLI runtime visibility was intentionally restored, so deduping needs a maintainer UX decision rather than conservative cleanup.

## What This Changes

The issue asks to stop `/models` from listing the same models under both canonical providers and CLI runtime alias providers when model-scoped `agentRuntime` is configured.

## Best Possible Solution

Define a shared `/models` display contract that keeps CLI runtime discovery but prevents duplicate canonical/runtime rows, then implement the dedupe or grouping rule in `buildModelsProviderData` with Telegram and shared command coverage.

## Reproduction Assessment

Yes, by source inspection: current `buildModelsProviderData` adds canonical visible catalog rows and then authenticated CLI runtime catalog rows, and Telegram renders that shared provider map. I did not run a live Telegram picker in this read-only review.

## Solution Assessment

Unclear until maintainers choose the display contract. A config flag would add avoidable surface area; the cleaner path is likely one shared dedupe/grouping rule that preserves the intentional CLI runtime access restored for `/models`.
