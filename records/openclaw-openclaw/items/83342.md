## Summary

Keep open. Current main and latest release still preserve a `/models` browse path where authenticated CLI runtime alias provider rows can appear alongside canonical provider rows, and the only targeted dedupe PR is closed unmerged rather than a safe canonical fix path.

## What This Changes

The issue requests de-duplicating `/models` browse results so the same models are not listed under both canonical provider banners and CLI runtime alias provider banners.

## Best Possible Solution

Define the shared `/models` display contract, then land a current-main dedupe that prunes only implicit duplicate runtime-alias rows while preserving explicit CLI-provider browsing and explicit legacy refs.

## Reproduction Assessment

Yes, by source inspection. Current `buildModelsProviderData` can add canonical visible catalog rows and authenticated CLI runtime alias catalog rows into the same provider map, and current tests assert those provider banners can coexist.

## Solution Assessment

Unclear until maintainers choose the display contract. The closed dedupe PR's preserve-explicit approach is a plausible shape, but it needs a refreshed current-main implementation and maintainer agreement on default CLI runtime provider visibility.
