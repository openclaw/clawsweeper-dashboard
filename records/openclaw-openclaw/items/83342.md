## Summary

Keep open. Current main still has a source-backed path that can present canonical provider rows and authenticated CLI runtime alias rows for the same underlying model IDs, and the merged history shows that CLI runtime provider visibility was intentionally restored, so deduping needs a maintainer UX/product decision rather than cleanup closure.

## What This Changes

The issue asks the shared `/models` browse picker to avoid listing the same runtime-backed models under both canonical provider banners and CLI runtime alias provider banners.

## Best Possible Solution

Define one shared display contract for canonical provider refs versus runtime aliases, then implement the chosen dedupe or grouping rule in `buildModelsProviderData` with channel coverage that preserves authenticated CLI runtime discovery.

## Reproduction Assessment

Yes, at source level. Current main adds canonical visible catalog entries and separately adds authenticated CLI runtime catalog entries, while tests assert those provider banners can appear together; I did not run a live Telegram picker in this read-only review.

## Solution Assessment

Unclear until maintainers choose the display contract. A new config flag would add surface area; the cleaner direction is likely one shared dedupe/grouping rule that keeps intentional CLI runtime discovery intact.
