## Summary

Keep open. Current `main` and the latest release still allow authenticated CLI runtime alias provider rows to appear alongside canonical provider rows in `/models`, and the only targeted fix PR is closed unmerged and conflicting rather than a safe canonical landing path.

## What This Changes

The issue reports duplicated `/models` browse entries when canonical providers and CLI runtime aliases expose the same underlying models.

## Best Possible Solution

Define the shared `/models` display contract for canonical provider refs versus CLI runtime aliases, then implement one dedupe/grouping rule in `buildModelsProviderData` while preserving explicit legacy CLI refs and explicit CLI-provider `all` browse.

## Reproduction Assessment

Yes, by source inspection: current `main` still adds authenticated CLI runtime catalog rows alongside canonical visible catalog rows, and tests assert both provider banners can appear. I did not run a live Telegram picker in this read-only review.

## Solution Assessment

Unclear until maintainers choose the display contract. The closed PR's dedupe-preserve-explicit approach is promising, but it is not a live landing path and the fix should be refreshed against current `main`.
