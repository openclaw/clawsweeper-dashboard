## Summary

Keep open. Current main still can show the same canonical model family under both canonical provider and CLI runtime provider groups, while https://github.com/openclaw/openclaw/pull/81239 deliberately restored authenticated CLI runtime providers in `/models`, so any dedupe/grouping change needs maintainer UX direction rather than cleanup closure.

## What This Changes

The issue asks for `/models` browse pickers to avoid listing the same underlying models under both a canonical provider and its CLI runtime alias provider.

## Best Possible Solution

Define the shared `/models` display contract for canonical provider refs versus runtime aliases, then implement one dedupe or grouping rule in `buildModelsProviderData` with Telegram and shared command coverage.

## Reproduction Assessment

Yes, by source inspection: current main adds canonical visible catalog rows and then auth-gated CLI runtime catalog rows, and Telegram renders provider buttons directly from that shared map. I did not run a live Telegram picker in this read-only review.

## Solution Assessment

Unclear until maintainers choose the display contract. A new config flag would add surface area; the cleaner path is likely one shared grouping or dedupe rule that preserves intentional CLI runtime discovery.
