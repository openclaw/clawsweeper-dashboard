## Summary

Keep open. Current main still can build `/models` data with the same underlying model ids under both a canonical provider and an authenticated CLI runtime provider, while tests and the prior picker fix show CLI provider visibility was intentionally restored; the remaining decision is the shared picker UX rule, not conservative cleanup.

## What This Changes

The issue requests deduping `/models` browse entries so canonical provider models with CLI `agentRuntime` policy do not also appear under the runtime alias provider.

## Best Possible Solution

Define the shared `/models` display contract for canonical model refs versus runtime-alias providers, then implement one dedupe or grouping rule in `buildModelsProviderData` with channel coverage.

## Reproduction Assessment

Yes, by source inspection: the shared provider-data builder adds canonical visible catalog rows and then authenticated CLI runtime catalog rows, and current tests assert both provider banners can appear. I did not run a live Telegram picker in this read-only review.

## Solution Assessment

Unclear until maintainers choose the display contract. A config flag would add avoidable surface area; the cleaner path is likely shared dedupe or grouping in the provider-data builder while preserving intentional CLI runtime access.
