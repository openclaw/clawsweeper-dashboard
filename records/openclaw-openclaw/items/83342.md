## Summary

Keep open. Current main and v2026.6.10 still allow authenticated CLI runtime alias provider rows to appear beside canonical provider rows in /models, and the direct dedupe PR is closed unmerged while the remaining contract now needs maintainer product direction.

## What This Changes

The issue requests a /models picker display policy that avoids listing the same canonical models under both their provider and CLI runtime alias provider, possibly with an explicit visibility override.

## Best Possible Solution

Have maintainers settle the shared picker visibility contract, then land a current-main implementation that removes implicit duplicate runtime-alias rows while preserving explicit legacy refs, explicit CLI-provider browsing, and any approved show/hide policy.

## Reproduction Assessment

Yes, at source level. Current buildModelsProviderData can add canonical visible catalog rows and authenticated CLI runtime alias catalog rows into the same provider map, and current tests assert those provider banners can coexist; I did not run a live Telegram session.

## Solution Assessment

Unclear. The closed dedupe PR's preserve-explicit approach is plausible, but current main still encodes sibling-provider behavior and the latest discussion asks maintainers to decide the broader show/hide policy first.
