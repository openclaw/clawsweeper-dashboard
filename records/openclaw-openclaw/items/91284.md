## Summary

Keep open: current `main` still has the reported doctor/security mismatch, and the issue is paired with an open focused fix PR rather than already implemented. The central bug is source-reproducible: runtime/docs recognize `trusted-proxy` as a valid gateway auth boundary, while the doctor exposure check still treats only token/password shared secrets as auth proof.

## What This Changes

The issue asks for `openclaw doctor --lint --json` to classify complete `gateway.auth.mode="trusted-proxy"` non-loopback Gateway setups as authenticated-but-exposed rather than unauthenticated.

## Best Possible Solution

Land a focused doctor/security change that aligns the exposure classification with the documented/runtime trusted-proxy auth model while preserving critical findings for unsafe trusted-proxy configurations.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure `gateway.bind="lan"` with `gateway.auth.mode="trusted-proxy"`, `gateway.auth.trustedProxy.userHeader`, and `gateway.trustedProxies`, then the current doctor/security exposure code still emits the shared-secret-only `without authentication` path. I did not execute `openclaw doctor` because this review was read-only.

## Solution Assessment

Yes, the best direction is to fix `src/commands/doctor-security.ts` so complete trusted-proxy auth counts as auth proof for this exposure classification while invalid trusted-proxy configs remain critical. The linked open PR appears to own that focused path and should be reviewed/landed or closed on its merits.
