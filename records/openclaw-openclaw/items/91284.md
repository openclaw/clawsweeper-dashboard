## Summary

Keep open: current main and the latest release still use a shared-secret-only doctor exposure check, so a complete trusted-proxy gateway auth setup can still be reported as unauthenticated. The remaining work is already owned by the focused open fix at https://github.com/openclaw/openclaw/pull/91268, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests that `openclaw doctor` treat a complete `gateway.auth.mode="trusted-proxy"` setup as configured Gateway authentication while preserving security warnings for exposed or unsafe proxy configurations.

## Best Possible Solution

Review and land the focused doctor/security fix in https://github.com/openclaw/openclaw/pull/91268, or replace it with an equivalent change that aligns doctor output with the documented trusted-proxy auth boundary and its guardrails.

## Reproduction Assessment

Yes. Source inspection shows that a LAN-bound config with `gateway.auth.mode="trusted-proxy"`, `gateway.auth.trustedProxy.userHeader`, and `gateway.trustedProxies` still enters the shared-secret-only critical unauthenticated doctor path on current main.

## Solution Assessment

Yes. The narrow maintainable fix is in `src/commands/doctor-security.ts`: count complete trusted-proxy auth as exposure auth proof while keeping unsafe trusted-proxy configs on critical paths, with regression tests in the adjacent doctor-security test file.
