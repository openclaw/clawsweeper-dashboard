## Summary

Keep open. Current main and the latest release intentionally cache Telegram edit updates for reply/context history without starting an agent turn, so the requested edited-user-message behavior remains unresolved and needs Telegram dispatch policy review.

## What This Changes

The issue requests that edited Telegram messages update not-yet-processed agent input or be preserved as follow-up context instead of leaving the agent to answer the original text.

## Best Possible Solution

Define Telegram edit semantics at the channel boundary: authorized user edits should update buffered same-message input when safe, and otherwise become explicit edited-message context for the active or next turn with focused tests and live Telegram proof.

## Reproduction Assessment

Yes, at source level: current main registers an `edited_message` handler that only records the edit for reply-chain cache, and the existing regression asserts that an edit update does not call the agent. I did not run a live Telegram edit in this read-only review.

## Solution Assessment

Unclear. Replacing queued content is plausible, but current maintained behavior deliberately avoids dispatching edit updates, so maintainers should first choose the Telegram edit policy and then implement it through the existing authorization, debounce, and queue boundaries.
