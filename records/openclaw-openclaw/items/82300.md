## Summary

Keep open. Current main still treats Telegram edit updates as reply-chain cache maintenance only, while the original inbound message remains the thing queued or dispatched to the agent; the requested active-turn behavior needs a Telegram plugin product decision before automation should patch it.

## What This Changes

Route Telegram `edited_message` updates so edited text can replace queued inbound content or be preserved as next-turn context instead of only updating reply-chain metadata.

## Best Possible Solution

Define Telegram edit-turn semantics first, then implement the queued-message replacement inside the Telegram plugin and add live Telegram proof for the user-visible behavior.

## Reproduction Assessment

Yes, for the queued/debounced path we have a high-confidence source reproduction: the edit handler updates only the message cache while the original message object remains in the pending dispatch path. The already-started-agent case is also unsupported by source inspection, but its expected behavior needs policy definition.

## Solution Assessment

No, this should not be auto-fixed yet. Replacing not-yet-flushed Telegram entries is likely narrow, but queuing or superseding edits after dispatch needs maintainer direction on turn ordering and cancellation semantics.
