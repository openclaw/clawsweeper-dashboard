## Summary

Keep open: current main observes Telegram edit updates but only uses them to refresh reply-chain context, while the requested behavior would change pending or active agent input semantics. That is a real, source-reproducible UX gap, but it needs a Telegram/product decision before automation rewrites active-turn behavior.

## What This Changes

Requests Telegram edited user messages to update pending or active agent input instead of only refreshing reply-chain metadata.

## Best Possible Solution

Define explicit Telegram edit semantics, then implement the chosen pending-edit and active-run behavior in the Telegram plugin with focused tests and live Telegram proof.

## Reproduction Assessment

Yes, by source inspection: normal Telegram messages enter the inbound dispatch path, while `edited_message` updates only refresh reply-chain cache. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear for an automated fix. Replacing pending debounce entries is plausible, but edits after an agent turn has started need an explicit product decision on steer, follow-up, interrupt, or cache-only behavior.
