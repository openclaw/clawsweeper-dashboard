## Summary

## What This Changes

The issue asks Telegram edited user messages to replace queued or early-processing content, or be queued as follow-up context, instead of leaving the agent to act on the original text.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/55691 as the canonical design thread, then implement a channel source-message edit/recall contract with Telegram-specific mapping and regression coverage.

## Reproduction Assessment

No for tracking this separately. The narrow Telegram cache subset already landed, while the requested replace/reprocess/abort semantics need the broader channel-event design tracked at https://github.com/openclaw/openclaw/issues/55691.
