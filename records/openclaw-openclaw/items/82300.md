## Summary

Keep open: current main still accepts Telegram edit updates only to refresh reply-chain context, not to replace or queue edited user instructions for agent input. The source path is clear, but this is a UX feature requiring an explicit maintainer decision for pending versus already-started turns.

## What This Changes

Make edited Telegram user messages affect agent input instead of only updating reply-chain metadata.

## Best Possible Solution

Define Telegram edit semantics, then implement pending-edit replacement and an explicit already-started behavior in the Telegram plugin/debounce path with focused tests and live Telegram proof.

## Reproduction Assessment

Yes, by source inspection: a normal Telegram message enters the inbound dispatch path, while an edited update reaches a handler that only refreshes reply-chain cache. I did not run the live Telegram reproduction in this read-only pass.

## Solution Assessment

Unclear as an implementation path: replacing pending debounce entries is plausible, but already-started edits need a maintainer-owned policy decision before code changes.
