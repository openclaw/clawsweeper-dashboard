## Summary

Keep open: current main handles Telegram edit updates only as reply-chain cache refreshes, so the requested user-edit handling is still missing and needs an owner policy for pending versus already-dispatched work.

## What This Changes

The issue requests that Telegram edited-message updates replace pending queued content or be queued as next-turn context instead of only refreshing reply-chain metadata.

## Best Possible Solution

Add an owner-reviewed Telegram edit policy in the plugin: update still-buffered Telegram messages before dispatch, and turn already-dispatched edits into explicit follow-up context without changing global queue defaults.

## Reproduction Assessment

Yes, at source level. Current main’s edited_message handler only records the edited message in the cache, and the existing test confirms it does not dispatch an agent turn when an edit update arrives.

## Solution Assessment

Unclear as proposed. The direction is reasonable, but the safest solution needs a maintainer-chosen Telegram edit policy across text-fragment buffering, inbound debounce, active runs, and next-turn context.
