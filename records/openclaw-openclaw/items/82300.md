## Summary

Close as duplicate/superseded: the remaining Telegram edit-to-reprocess behavior is a narrower slice of the open cross-channel recall/edit cancellation request, while current main only implements edit caching for later context and intentionally does not start a new agent turn from edit updates.

## What This Changes

The issue requests edited Telegram user messages to replace queued content or be queued as a later correction instead of leaving the agent to process the original text.

## Best Possible Solution

Track the remaining behavior in https://github.com/openclaw/openclaw/issues/55691: define a generic authorized source-message edit/delete contract, then map Telegram edit events onto cancellation, steering, or follow-up behavior with focused tests and live Telegram proof.

## Reproduction Assessment

Yes, from source inspection rather than a live Telegram run. The edited-message handler updates reply-chain cache only, while regular messages flow into `processInboundMessage` and `inboundDebouncer.enqueue`, so edited content is not dispatched as a replacement or follow-up turn today.

## Solution Assessment

No as a separate Telegram-only tracker. The safer maintainable path is to keep this under the existing cross-channel recall/edit issue so Telegram behavior aligns with shared cancellation, queueing, and authorization semantics.
