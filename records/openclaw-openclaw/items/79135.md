## Summary

Keep open: this PR addresses a real `/model status` auth-label mismatch, but it is a draft with the protected `maintainer` label, lacks after-fix real behavior proof, and the proposed helper does not cover the documented `agentRuntime.id: "auto"` OpenAI-to-Codex route.

## What This Changes

The PR makes Telegram `/model status` fall back from missing direct `openai` auth to `openai-codex` runtime auth for Codex-backed OpenAI rows and adds a focused regression test.

## Best Possible Solution

Use the same resolved harness policy as runtime/status paths for `/model status` auth labels, covering explicit Codex, implicit/auto Codex, explicit PI compatibility, and preserving direct provider/env/custom/workspace auth labels.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: configure `openai/gpt-5.5` with only an `openai-codex` OAuth profile and current main resolves the `/model status` `[openai]` row through direct `openai` auth. I did not run a live Telegram command in this read-only review.

## Solution Assessment

No, not yet: the direction is narrow, but the helper should use resolved runtime policy rather than raw default runtime config so documented `auto` OpenAI/Codex routes get the same auth label behavior.
