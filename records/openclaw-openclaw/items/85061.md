## Summary

Keep open; current main and the latest release still build Gemini `timeRangeFilter` timestamps with `Date.toISOString()`, while the documented Gemini freshness/date feature and focused tests currently preserve fractional-second output instead of rejecting it.

## What This Changes

Report that Gemini `web_search` freshness/date filters serialize sub-second `timeRangeFilter` timestamps that Gemini rejects with a nano-granularity 400.

## Best Possible Solution

Normalize every Gemini `timeRangeFilter` timestamp to whole-second RFC 3339 at the Google plugin boundary, update focused tests to reject fractional output, and keep the documented freshness/date feature intact.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main sends millisecond RFC 3339 strings from the Gemini time-filter path, and the reporter supplied live Gemini API output showing non-zero fractional seconds are rejected. A full runtime repro is a Gemini-backed `web_search` call with `freshness` or an open-ended date filter and a real API key.

## Solution Assessment

Yes. The narrow maintainable fix is a shared Gemini timestamp serializer for `timeRangeFilter` that strips fractional seconds before request construction, with regression tests covering freshness and date-range paths.
