## Summary

Keep open; current main and the latest release still serialize Gemini `timeRangeFilter` values with `Date.toISOString()`, and the paired fix path is the still-open PR at https://github.com/openclaw/openclaw/pull/85071.

## What This Changes

The issue reports that Gemini `web_search` freshness and date filters emit fractional-second `timeRangeFilter` timestamps that the live Gemini Search grounding endpoint rejects.

## Best Possible Solution

Normalize every Gemini `timeRangeFilter` timestamp to whole-second RFC 3339 at the provider boundary, update focused tests to reject fractional output, and close this issue after https://github.com/openclaw/openclaw/pull/85071 or an equivalent fix lands.

## Reproduction Assessment

Yes, at source level: current main emits millisecond-precision Gemini `timeRangeFilter` values and the focused tests assert that shape. Full live reproduction is a Gemini API-key run with `freshness` or a date fallback, which the reporter supplied but I did not rerun.

## Solution Assessment

Yes; the narrow maintainable fix is a shared Gemini timestamp serializer that strips fractional seconds before request construction, with regression coverage using a non-zero millisecond fake clock.
