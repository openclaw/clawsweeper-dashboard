## Summary

Keep open; current main still emits millisecond-precision Gemini timeRangeFilter timestamps, and an open linked fix PR is the right path to resolve this issue after review and merge.

## What This Changes

Gemini web_search freshness/date timeRangeFilter timestamps should be serialized without sub-second precision so Google Search grounding accepts them.

## Best Possible Solution

Review and land the linked provider-boundary serializer fix at https://github.com/openclaw/openclaw/pull/85071 if it remains correct, then close this issue once the fix reaches main.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to exercise Gemini web_search with freshness or an open-ended date filter and inspect the generated request body: current main emits millisecond-precision timeRangeFilter values, while the reporter supplied live Gemini 400 evidence for non-zero fractional timestamps.

## Solution Assessment

Yes. The narrow maintainable fix is to normalize all Gemini timeRangeFilter timestamps to whole-second RFC 3339 at the Google provider boundary and update focused tests to cover non-zero millisecond clocks.
