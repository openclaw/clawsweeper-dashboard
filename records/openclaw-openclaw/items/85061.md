## Label Justifications

- `P2`: This is a documented Gemini web_search provider regression with limited blast radius: affected searches lose freshness/date filtering and waste latency/quota, but the core runtime remains available.

## Summary

Keep open; current main still emits fractional-second Gemini timeRangeFilter timestamps, and the linked fix PR is open but unmerged, so the issue should continue tracking the regression until that PR or an equivalent fix lands.

## What This Changes

Reports that Gemini-backed web_search with freshness/date filters sends sub-second timeRangeFilter timestamps that the Gemini Search grounding endpoint rejects.

## Best Possible Solution

Normalize every Gemini timeRangeFilter timestamp to whole-second RFC 3339 at the Google provider boundary, keep the focused regression tests, and close this issue only after https://github.com/openclaw/openclaw/pull/85071 or an equivalent fix lands.

## Reproduction Assessment

Yes, at source/proof level: current main emits millisecond-precision timeRangeFilter values, and the reporter's live Gemini matrix shows non-zero fractional timestamps fail while second-precision timestamps succeed. I did not rerun the live API path in this read-only review.

## Solution Assessment

Yes; the narrow maintainable fix is to centralize Gemini timeRangeFilter timestamp serialization at the provider boundary and update the focused tests to cover non-zero millisecond clocks.
