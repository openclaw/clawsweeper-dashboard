## Summary

Keep open; current main and the shipped v2026.5.19 code still serialize Gemini `timeRangeFilter` values with `Date.toISOString()`, while the focused tests assert fractional-second output and the issue includes live Gemini 400 evidence for non-zero fractional seconds.

## What This Changes

The issue reports that Gemini `web_search` freshness/date filters send fractional-second `timeRangeFilter` timestamps that Gemini rejects with `Granularity of nano is not supported`.

## Best Possible Solution

Normalize every Gemini `timeRangeFilter` timestamp to whole-second RFC 3339 in one provider-local helper, update focused tests to reject fractional output, and keep the documented freshness/date feature intact.

## Reproduction Assessment

Yes, at source level: current main sends millisecond-precision `timeRangeFilter` strings in the affected Gemini paths, and the reporter supplied live Gemini 400 evidence for non-zero fractional seconds. I did not rerun the credentialed live call in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is a shared Gemini timestamp serializer applied to freshness and date-range call sites, with tests asserting whole-second output; no config or product-policy change is needed.
