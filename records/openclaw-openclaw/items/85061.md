## Summary

Keep open; current main still serializes Gemini time-range filters with fractional seconds, and the only related GitHub search hit is the already-closed freshness implementation request rather than a canonical open duplicate.

## What This Changes

The issue reports that Gemini web_search freshness/date filters send sub-second RFC 3339 timestamps that Gemini rejects with a nano-granularity 400 error.

## Best Possible Solution

Normalize all Gemini `timeRangeFilter` timestamps to whole-second RFC 3339 at the provider boundary, update focused tests to reject fractional output, and keep the documented freshness/date feature intact.

## Reproduction Assessment

Yes, at source level: current main sends `.000Z`/millisecond timestamps in the affected Gemini `timeRangeFilter` fields, and the tests assert that shape. Full live reproduction is a Gemini API-key run of `web_search` with `freshness` or a date filter, which the reporter supplied but I did not rerun.

## Solution Assessment

Yes, the narrow maintainable fix is a shared Gemini timestamp serializer that strips fractional seconds before building `timeRangeFilter`; no config or product-policy change is needed.
