## Summary

Keep open: current main still routes update-time doctor into systemd service staging, and the systemd writer backs up then replaces the live unit from the canonical template, which matches the reported `TimeoutStartSec`, `ExecStart`, and `EnvironmentFile` clobbering. The related closed issues/PRs cover adjacent system-service and env-file repair loops, but I did not find a merged or open canonical fix for preserving operator-owned user-systemd directives during update repair.

## What This Changes

The issue asks `openclaw doctor` and `openclaw update` to stop wholesale-rewriting customized Linux user-systemd gateway units or to merge those units safely.

## Best Possible Solution

Define the systemd ownership contract, then make update-mode doctor only mutate OpenClaw-owned directives or write a suggested unit/diff while preserving operator-owned fields such as custom timeouts, `ExecStartPost`, required `EnvironmentFile`, and unrelated `Environment=` entries.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: update runs doctor with `--non-interactive --fix`, doctor stages recommended service drift during update mode, and systemd staging writes a fresh template with the reported defaults; I did not run a live WSL2/systemd reproduction in this read-only review.

## Solution Assessment

No. The current whole-unit rewrite during update-mode doctor is not the best solution because it treats recommended metadata drift as permission to replace operator-owned systemd state; a targeted merge or suggested diff is safer.
