## Summary

Keep open: current main still stages systemd service repairs during update-mode doctor and the writer replaces the live unit from the canonical template, which matches the reported `TimeoutStartSec`, `ExecStart`, and `EnvironmentFile` clobbering. A prior drop-in proposal covered much of the same problem but was closed unmerged, so there is no implemented or canonical open fix to close against.

## What This Changes

The issue asks `openclaw doctor`/`openclaw update` to stop wholesale-replacing user-customized systemd user units or to strictly preserve valid operator-owned fields.

## Best Possible Solution

Define systemd unit ownership so update-mode doctor only changes OpenClaw-owned directives or emits an explicit suggested diff, while preserving operator-owned fields such as timeouts, `ExecStartPost`, required `EnvironmentFile`, and unrelated `Environment=` entries.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a nvm-backed systemd unit triggers recommended runtime drift, update runs `doctor --non-interactive --fix` with update mode, doctor stages the service, and staging writes a fresh template with the reported defaults. I did not run a live systemd/WSL2 reproduction because this review is read-only.

## Solution Assessment

No, the current implementation is not the best way to solve the issue because update-mode doctor uses a whole-unit rewrite for service repair. The maintainable fix should separate managed OpenClaw state from user-owned systemd state or perform targeted merges with explicit operator review for destructive changes.
