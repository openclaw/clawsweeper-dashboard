## Summary

Keep open: the Signal quote-reply direction is useful and not already implemented on current main, but the PR head still has functional Signal transport blockers and lacks live paired-device proof.

## What This Changes

The PR adds bidirectional Signal quote-reply support across Signal send/monitor code, shared outbound delivery and queue recovery, SDK reply payload typing, tests, and the changelog.

## Best Possible Solution

Keep the PR open, require quote metadata to include a valid author before native signal-cli sends, map quote fields through the container adapter, then add redacted live Signal native/container proof or explicitly scope unsupported modes.

## Reproduction Assessment

Yes, for the PR blockers by source inspection and upstream contracts: PR head can send native signal-cli quote-timestamp without quote-author, and the container adapter drops quote fields despite the REST API exposing them. I did not run a live Signal account roundtrip.

## Solution Assessment

No. The implementation direction is maintainable, but it must require quote authors for native quote sends, preserve quote metadata in container mode, and supply live Signal proof before it is ready to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Require quote authors before sending Signal quotes:** `extensions/signal/src/reply-quote.ts:35-37`
  - body: signal-cli rejects any request with `quote-timestamp` but no `quote-author`, but this helper only gates missing authors for group targets. A direct reply with no resolved author still returns timestamp-only metadata, so `sendMessageSignal` can submit an invalid native RPC request instead of degrading to a normal message; drop quote params unless both timestamp and author are available.
  - confidence: 0.94
- **[P2] Forward quote metadata through container sends:** `extensions/signal/src/send.ts:257-266`
  - body: `sendMessageSignal` now adds quote metadata before calling `signalRpcRequest`, but auto/container mode translates `send` through `client-container.ts` and currently forwards only message, recipients, text styles, attachments, and timeout. Since the container REST API exposes `quote_timestamp` and `quote_author`, container-mode Signal users will still send plain unquoted replies unless those fields are mapped and covered by tests.
  - confidence: 0.86
