## Summary

Keep open: current main still lacks Signal native quote metadata, so the PR is not obsolete, but the PR head still has reply-consumption correctness blockers and lacks live Signal account/device proof for the transport/UI behavior.

## What This Changes

The branch wires Signal quote metadata through outbound delivery and inbound context, updates reply payload contracts and queue recovery across channels, and adds Signal quote tests plus API baseline and changelog updates.

## Best Possible Solution

Keep the Signal quote-reply direction, but separate planned reply metadata from committed consumption, track inherited versus explicit Signal reply state, and require redacted live Signal send/receive proof before merge.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows reply consumption is mutated during send planning, and Signal monitor state is consumed from any effective reply id. I did not reproduce a live Signal roundtrip.

## Solution Assessment

No. The implementation direction is reasonable, but the current patch should not merge until reply consumption is committed only after successful delivery, explicit reply targets do not consume inherited state, and live Signal proof is supplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Consume Signal replies only after a delivered send:** `src/infra/outbound/deliver.ts:1637-1639`
  - body: This calls applyReplyToConsumption while only preparing send overrides. In the PR's new Signal single-use mode, that flips the internal consumed bit before sendFormattedText/sendFormattedMedia has produced a message; with bestEffort enabled, a first send failure causes later payloads to see the inherited reply as already consumed and send without the quote. Move consumption to the post-success path or split planning from committing consumption.
  - confidence: 0.87
- **[P2] Consume inherited Signal state only for inherited replies:** `extensions/signal/src/monitor/reply-delivery.ts:63-76`
  - body: markSignalReplyConsumed only receives the effective reply id, so explicit per-payload reply tags consume the shared inherited state too. If the first payload explicitly replies to an older Signal message, later untagged payloads in the same turn no longer inherit the current inbound message id and lose their quote context. Return/pass whether the inherited target was used before marking this state consumed.
  - confidence: 0.82
