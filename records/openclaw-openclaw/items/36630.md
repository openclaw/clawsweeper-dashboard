## Summary

Keep open. The latest head appears to address several earlier quote-reply review points, but the direct Signal monitor path still has a source-level state bug, and the supplied proof explicitly does not include a live Signal send/receive or signal-cli/client observation.

## What This Changes

The PR adds bidirectional Signal quote-reply support, including outbound quote metadata, inbound quoted-message context, shared reply metadata plumbing, queue persistence, changelog, and regression tests.

## Best Possible Solution

Keep the PR open, fix the Signal monitor helper so it only consumes inherited reply state when the inherited target was actually sent, then add redacted live Signal send/receive proof before merge.

## Reproduction Assessment

Yes, at source level. Current main has the generic reply-tag directive but Signal send/monitor code does not emit native quote params or structured quote context, and the remaining PR-head state bug follows directly from resolveSignalReplyDelivery plus the shared replyDeliveryState path.

## Solution Assessment

No. The implementation direction is maintainable, but the direct Signal monitor path still needs to distinguish explicit reply ids from inherited current-message ids before consuming shared quote state, and real Signal behavior proof is still missing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Track inherited Signal reply state separately:** `extensions/signal/src/monitor/reply-delivery.ts:26-30`
  - body: `resolveSignalReplyDelivery` returns an `effectiveReplyTo` without saying whether it came from a payload-level explicit reply or from `inheritedReplyToId`. `deliverReplies` then marks the shared state consumed after any valid first send, so a turn whose first payload explicitly replies to an older Signal message can make the next untagged payload lose the inherited current-message quote even though that inherited quote was never used. Return/source-track the inherited target and only consume state for inherited sends.
  - confidence: 0.84
