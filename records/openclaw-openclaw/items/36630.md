## Summary

Keep open. Current main still does not provide the native bidirectional Signal quote-reply behavior this PR is trying to add, and the PR has concrete merge blockers in the latest provided head: production Signal plugin code deep-imports core internals, nullable reply suppression is widened before shared threading honors it, and inbound quote-id normalization accepts non-decimal numeric aliases. The useful implementation work should be reworked rather than closed as obsolete.

## What This Changes

The PR adds Signal inbound quote-context parsing, outbound signal-cli quote metadata, shared `quoteAuthor` and nullable `replyToId` plumbing, and tests across Signal monitor and outbound delivery behavior.

## Best Possible Solution

Land a reworked Signal quote-reply implementation that keeps production plugin code behind SDK or plugin-local surfaces, validates Signal quote IDs with one decimal-only timestamp helper, preserves nullable reply suppression before shared threading and hook decisions, and routes direct, formatted, media, queued, recovered, and debounced Signal delivery through one consistent quote-author policy.

## Reproduction Assessment

Yes. Static reproduction is high confidence: current main lacks Signal quote-send metadata support, and the provided PR head diff shows the boundary-violating imports, nullable reply widening, and inbound `Number(trimmed)` quote-id normalization.

## Solution Assessment

No. The feature direction is useful, but this PR is not the best merge vehicle until SDK boundaries, quote-id validation, shared reply semantics, and Signal delivery-path coverage are corrected together.

  - confidence: 0.97

  - body: This production Signal helper imports `ReplyPayload` from a core `src/**` path. Use the public reply runtime/payload SDK surface or a plugin-local type facade instead of coupling plugin code to core internals.
  - confidence: 0.96

  - body: Adding `quoteAuthor` to shared send overrides is not enough for the default formatted Signal path: `sendFormattedSignalText` and `sendFormattedSignalMedia` still call `sendMessageSignal` without `replyTo` or `quoteAuthor`. Generic Signal sends with reply metadata will continue to post without native quote bubbles.
  - confidence: 0.91
- **[P2] Require decimal-only inbound quote IDs:** `extensions/signal/src/monitor/quote-context.ts:67-68`
  - body: `normalizeSignalQuoteId` uses `Number(trimmed)`, which accepts values like `1e3` and `0x10`. Those can alias inbound reply context and author-cache lookups to a different message id; inbound quote IDs should use the same decimal-only validation as outbound quote timestamps.
  - confidence: 0.92
- **[P2] Treat blank Signal reply IDs as unset:** `extensions/signal/src/monitor/reply-delivery.ts:12-13`
  - body: `normalizeReplyToId` converts whitespace-only strings to `null`, which this module treats as explicit reply suppression. Blank payload ids should behave like `undefined` so inherited quote context can still apply; reserve `null` for an intentional do-not-reply override.
  - confidence: 0.89
- **[P2] Persist quote authors in queued deliveries:** `src/infra/outbound/deliver.ts:972`
  - body: The live delivery path now carries `quoteAuthor`, but the write-ahead queue still persists and replays only `replyToId`. A recovered Signal group reply will lose the author required for `quote-author` and will be replayed unquoted.
  - confidence: 0.87
- **[P2] Fall back to mediaUrl when mediaUrls is empty:** `extensions/signal/src/monitor.ts:359-360`
  - body: `resolvedPayload.mediaUrls ?? ...` treats an explicit empty array as final, so `{ mediaUrls: [], mediaUrl: "..." }` drops the single-media fallback that the previous shared reply-payload helper preserved. Media replies can be skipped even though a legacy `mediaUrl` is present.
  - confidence: 0.89
- **[P2] Do not carry stale debounced quote targets forward:** `extensions/signal/src/monitor/event-handler.ts:508-510`
  - body: For debounced batches, choosing the latest entry that has a quote can carry quote context from an older message even when the actual latest message is plain. A quoted message followed by a normal follow-up can surface stale `ReplyTo*` context for the combined turn.
  - confidence: 0.86
- **[P2] Process debounced quote-only batches:** `extensions/signal/src/monitor/event-handler.ts:503-504`
  - body: When a debounced batch contains only quote-only entries, `combinedText` is empty and this early return drops the whole batch before history or dispatch can see it. Quote-only support should still create inbound context when debounce is enabled.
  - confidence: 0.85
