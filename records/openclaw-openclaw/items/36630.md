## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep null reply suppression out of the public SDK:** `src/plugin-sdk/reply-payload.ts:43`
  - body: This makes `OutboundReplyPayload.replyToId` publicly nullable and preserves `null` from `normalizeOutboundReplyPayload`, but the null value is only an internal suppression signal for inherited reply metadata. Existing plugin consumers compiled against the SDK can now receive/pass `null` where the contract used to be string-or-undefined, so keep the public SDK shape as `replyToId?: string` and normalize suppression before plugin-facing payloads unless maintainers explicitly approve a nullable API migration.
  - confidence: 0.86
- **[P2] Consume Signal inherited replies only when inherited was used:** `extensions/signal/src/monitor/reply-delivery.ts:63-77`
  - body: `markSignalReplyConsumed` is called with `effectiveReplyTo`, but `effectiveReplyTo` also includes explicit per-payload targets. If the first dispatched payload replies to an explicit older Signal message and a later payload omits `replyToId`, the first send marks the shared inherited state consumed and the later payload no longer quotes the current inbound message; the helper needs to know whether the inherited target was actually used before flipping `state.consumed`.
  - confidence: 0.89
