## Summary

Keep open: current main still lacks native Signal quote parameters and structured quoted-message context, but this PR head remains incorrect because it can emit Signal quote timestamps without authors, consumes inherited reply state too early, and still lacks live Signal account/device proof.

## What This Changes

The PR wires Signal quote-reply metadata through outbound delivery, Signal send and monitor code, reply payload typing, queue persistence, tests, SDK baseline, and changelog.

## Best Possible Solution

Land a corrected Signal quote-reply implementation that resolves quote authors before emitting quote params, consumes inherited reply state only after successful inherited sends, keeps explicit payload replies independent, and includes redacted live Signal send/receive proof.

## Reproduction Assessment

Yes for the source-level review blockers: PR head has clear paths for quote-timestamp without quote-author and premature inherited reply consumption, and signal-cli's quote-author contract is explicit. I did not run a live Signal roundtrip.

## Solution Assessment

No. The implementation direction is useful, but this patch should not merge until it honors signal-cli's quote-author requirement, consumes only successfully delivered inherited replies, and supplies live Signal proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Require a quote author for every Signal quote:** `extensions/signal/src/reply-quote.ts:35-39`
  - body: signal-cli throws when `quote-timestamp` is supplied without `quote-author`, but this helper still returns a timestamp with no author for direct targets. A direct send with `replyToId` and no resolved author can now fail instead of falling back to a normal message; resolve an author first or drop all quote params when the author is unavailable.
  - confidence: 0.92
- **[P2] Consume inherited replies only after delivery succeeds:** `src/infra/outbound/deliver.ts:1634-1639`
  - body: `applySendReplyToConsumption` calls `applyReplyToConsumption` while preparing overrides, before the adapter has delivered a message. In best-effort mode, a first quoted send failure can leave later payloads seeing the inherited reply as consumed, so no successful payload gets the quote.
  - confidence: 0.87
- **[P2] Consume Signal state only for inherited replies:** `extensions/signal/src/monitor/reply-delivery.ts:63-76`
  - body: `markSignalReplyConsumed` only receives the effective reply id, so an explicit per-payload reply target can consume the shared inherited state. If a later payload omits `replyToId`, it no longer inherits the current Signal message and loses its quote context.
  - confidence: 0.84
