## Summary

Keep open: the Signal quote-reply work is still useful and not implemented on current main, but the current PR head still has a blocking Signal quote parameter bug and needs live Signal proof before merge.

## What This Changes

The PR adds native Signal quote-reply send/receive support, quote-author and replyTo plumbing through outbound delivery and queue recovery, inbound ReplyTo* context handling, SDK type widening, tests, and a changelog entry.

## Best Possible Solution

Require quote-author whenever emitting quote-timestamp, or omit both fields when the author is unavailable, then collect redacted live Signal device/account proof for outbound and inbound quote replies before merge.

## Reproduction Assessment

Yes. By source inspection, direct Signal sends with replyTo and no quoteAuthor still emit quote-timestamp without quote-author, and signal-cli v0.14.3 throws for that combination.

## Solution Assessment

No. The implementation direction is useful, but it is not the best merge path until the Signal send helper enforces the signal-cli quote-author contract and the PR supplies live Signal proof.
