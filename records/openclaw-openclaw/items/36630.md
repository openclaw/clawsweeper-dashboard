## Summary

Keep this PR open: the main implementation is still useful and current main does not already provide native Signal quote replies, but the branch still has a source-level best-effort consumption bug and the supplied proof does not show a live Signal send/receive roundtrip.

## What This Changes

The PR adds bidirectional Signal quote-reply support by carrying quote metadata through outbound delivery, parsing inbound signal-cli quote payloads into ReplyTo* context, updating reply payload contracts and queue recovery, and adding tests, API baseline, and changelog coverage.

## Best Possible Solution

Keep the bidirectional Signal quote-reply direction, but make inherited quote consumption happen only after a successful delivery identity, add regression coverage for failed first-send best-effort paths, and require redacted live Signal roundtrip proof before merge.

## Reproduction Assessment

Yes for the review blocker: source inspection of PR head shows Signal is opted into a pre-send single-use consumption policy, so a failed first best-effort send can consume the inherited quote before any delivery succeeds. I did not reproduce a live Signal roundtrip.

## Solution Assessment

No. The implementation direction is reasonable, but the current patch should not merge until Signal quote consumption is tied to successful delivery and live Signal proof is supplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Consume Signal replies only after a send succeeds:** `src/infra/outbound/deliver.ts:1417-1421`
  - body: This opts Signal into the existing single-use policy, where applySendReplyToConsumption consumes the inherited reply while preparing overrides, before the adapter has produced a message. In bestEffort mode, if the first quoted Signal send throws before delivery, later payloads no longer get the inherited replyToId, so the turn loses its quote context instead of retrying it on the next successful payload.
