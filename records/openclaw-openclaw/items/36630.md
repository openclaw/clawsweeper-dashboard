## Summary

Keep this PR open, but it is not merge-ready: the Signal quote-reply fix remains useful, while the current branch can still consume implicit reply state before a successful send and it still lacks live Signal proof.

## What This Changes

The PR adds bidirectional Signal quote-reply support by threading quote metadata through Signal inbound parsing, outbound send, shared delivery planning, queue recovery, plugin SDK reply payload typing, and related tests.

## Best Possible Solution

Move single-use implicit reply consumption to the point where a platform send actually returns an identified delivery, keep explicit per-payload reply targets independent, and then prove the Signal send/receive path with a live paired account.

## Reproduction Assessment

Yes. Source inspection of the PR-head delivery path shows the first Signal send can consume the implicit reply during planning before any platform delivery succeeds, so a best-effort first-send failure reproduces the lost-quote path without needing a live account.

## Solution Assessment

No. The intended Signal owner-boundary fix is plausible, but consuming one-shot reply state during planning is not the best implementation because it makes failure handling depend on work that has not actually been delivered.
