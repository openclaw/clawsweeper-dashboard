## Summary

Keep this PR open for normal review rather than closing it: current main still lacks outbound Signal quote parameters, but the PR head has blocking reply-state and signal-cli contract issues, and the supplied proof is terminal/test-only rather than a live Signal roundtrip.

## What This Changes

The PR adds bidirectional Signal quote-reply support across Signal outbound sends, inbound quote context, reply delivery state, queue persistence, SDK payload typing, tests, and changelog.

## Best Possible Solution

Keep the Signal quote-reply direction, but require code fixes for the signal-cli quote-author contract and reply-state semantics, then collect redacted live Signal device/account proof before merge.

## Reproduction Assessment

Yes for the review blockers by source inspection: PR-head code can emit quote-timestamp without quote-author, mutates reply consumption before delivery succeeds, and consumes Signal state from any effective reply id. I did not run a live Signal roundtrip.

## Solution Assessment

No. The implementation direction is reasonable, but this patch should not merge until it respects signal-cli's quote-author requirement, commits inherited reply consumption only after successful delivery, distinguishes inherited from explicit reply targets, and supplies live proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require quote authors before emitting Signal quotes:** `extensions/signal/src/reply-quote.ts:35-39`
  - body: signal-cli throws when `quote-timestamp` is supplied without `quote-author`, but this helper still returns a timestamp alone for direct chats. Direct Signal sends with a `replyToId` but no resolved author can now fail instead of degrading to a normal message; drop all quote params unless an author is available or resolve an author for every quote send.
  - confidence: 0.9
- **[P2] Commit reply consumption after delivery succeeds:** `src/infra/outbound/deliver.ts:1637-1639`
  - body: This mutates the reply policy while only preparing send overrides, before the adapter has returned a delivered message. With `bestEffort`, a first quoted send failure leaves later payloads seeing the inherited reply as already consumed, so no successful payload gets the quote.
  - confidence: 0.88
- **[P2] Consume Signal state only for inherited replies:** `extensions/signal/src/monitor/reply-delivery.ts:63-76`
  - body: `markSignalReplyConsumed` only receives the effective reply id, so an explicit per-payload reply target can consume the shared inherited state. If the next payload omits `replyToId`, it no longer inherits the current Signal message and loses its quote context.
  - confidence: 0.84
