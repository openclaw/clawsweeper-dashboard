## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep null reply suppression out of the public SDK contract:** `src/plugin-sdk/reply-payload.ts:43`
  - body: This widens the plugin-facing reply payload from `string` to `string | null`, and the branch passes null through outbound contexts as a new suppression signal. Existing third-party plugins were only required to handle a string or absence here, so upgraded hosts can now hand them a value they may treat as string routing metadata. Keep null suppression internal to core, or make this an explicit SDK compatibility change with docs and focused upgrade tests before merge.
  - confidence: 0.82
- **[P2] Consume inherited replies only after a send succeeds:** `src/infra/outbound/deliver.ts:1782-1784`
  - body: `applyReplyToConsumption` mutates the single-use reply state while planning overrides, before the awaited platform send returns. In `bestEffort` mode, if the first Signal send throws before producing a result, the catch continues but the policy has already consumed the inherited reply, so later payloads are sent without the quote even though no quoted message was delivered. Move consumption to the post-success path or roll it back on failed sends.
  - confidence: 0.86
