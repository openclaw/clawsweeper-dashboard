## Summary

Keep open: current main still lacks the central Signal quote-reply support, and the PR is not merge-ready because PR head can still emit `quote-timestamp` without `quote-author` for direct Signal sends, which signal-cli rejects. The supplied proof is useful terminal/test output, but it does not demonstrate a live paired Signal send/receive roundtrip.

## What This Changes

The PR adds Signal inbound quote-context parsing, outbound signal-cli quote parameter plumbing, reply metadata propagation through delivery and queue paths, SDK reply payload type updates, tests, and a changelog entry.

## Best Possible Solution

Require a resolved quote author before emitting any Signal quote parameters, update the direct-send regression test to expect graceful quote suppression when the author is unavailable, then collect redacted live Signal send/receive proof before merge.

## Reproduction Assessment

Yes for the review blocker by source and dependency-contract inspection: a direct Signal send with `replyTo` and no `quoteAuthor` produces `quote-timestamp` without `quote-author`, and signal-cli v0.14.3 throws on that combination. I did not run a live Signal account roundtrip.

## Solution Assessment

No. The implementation direction is useful, but the Signal send helper should drop quote metadata unless both timestamp and author are available, and the PR needs live paired Signal proof before it is the best merge path.
