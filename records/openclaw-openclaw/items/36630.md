## Summary

This PR should stay open: the Signal quote-reply gap is real and the branch is close, but latest head still has a P2 reply-state correctness bug and the supplied proof does not include a live Signal send/receive roundtrip.

## What This Changes

The branch adds Signal outbound signal-cli quote metadata, inbound quoted-message ReplyTo context, quote-author resolution, shared replyTo/quoteAuthor plumbing, queue persistence, changelog, and regression tests.

## Best Possible Solution

Keep the channel-owned implementation direction, fix the explicit-versus-inherited reply-state consumption bug, and require redacted live Signal send/receive proof before merge.

## Reproduction Assessment

Yes, at source level. Current main has the reply-tag directive but Signal send code does not emit signal-cli quote params and Signal quote types do not carry enough structured id/author data.

## Solution Assessment

No. The implementation direction is maintainable, but the latest PR head still conflates explicit and inherited Signal reply targets before consuming shared quote state, and live Signal behavior proof is still missing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Track inherited Signal reply state separately:** `extensions/signal/src/monitor/reply-delivery.ts:28-32`
  - body: `resolveSignalReplyDelivery` returns only `effectiveReplyTo`, so `deliverReplies` marks the shared state consumed after any successful send with that id. If the first payload explicitly replies to an older Signal message, the next untagged payload loses the inherited current-message quote even though that inherited target was never used; return/source-track whether the id was inherited and only consume state for inherited sends.
  - confidence: 0.84
