## Summary

Keep open. The PR addresses a real Signal reply gap, but the current head is still not mergeable because it can send Signal quote timestamps without the required quote author, container mode drops the new quote fields, and the supplied proof stops at local helpers/tests instead of a paired Signal send/receive path.

## What This Changes

The PR adds Signal inbound/outbound quote-reply plumbing, extends reply metadata through outbound delivery and SDK payload types, preserves quote context in queues/recovery, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land a finished Signal reply implementation that requires author-safe quote metadata, maps quote fields in native and container modes, keeps queue/context preservation, and includes redacted live Signal roundtrip proof.

## Reproduction Assessment

Yes. The current blockers are source-reproducible from PR head: timestamp-only quote metadata conflicts with signal-cli's SendCommand contract, and the container send adapter omits REST quote fields that the container API supports.

## Solution Assessment

No. The direction is useful, but the current patch is not the best mergeable solution until quote metadata always has an author when emitted, container sends forward quote fields, and real Signal proof is supplied.

Container Signal users will still get plain replies unless those fields are forwarded and covered by a test.
