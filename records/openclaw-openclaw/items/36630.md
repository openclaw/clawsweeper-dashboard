## Summary

Keep open. Current main still lacks native Signal quote-send plumbing, and the PR adds useful Signal quote-reply support, but latest head still has a source-level reply-state correctness bug and does not include live Signal send/receive proof.

## What This Changes

Adds bidirectional Signal quote-reply support by wiring outbound quote metadata, inbound quote context, reply threading/queue plumbing, SDK hash updates, changelog, and regression tests.

## Best Possible Solution

Keep the Signal-owned implementation direction, fix the reply-delivery helper to source-track inherited replies, add a focused regression for explicit-then-inherited payloads, coordinate overlapping Signal quote PRs, and require redacted live Signal proof before merge.

## Reproduction Assessment

Yes, at source level. Current main Signal send does not accept replyTo/quoteAuthor or emit quote params, and the PR-head state bug follows from sending an explicit payload reply before a later inherited reply through resolveSignalReplyDelivery and markSignalReplyConsumed.

## Solution Assessment

No. The implementation direction is maintainable, but the current PR should source-track whether the effective reply came from inherited state and should include live Signal send/receive proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Track inherited Signal reply state separately:** `extensions/signal/src/monitor/reply-delivery.ts:20-23`
  - body: `resolveSignalReplyDelivery` returns only `effectiveReplyTo`, so `deliverReplies` marks the shared state consumed after any valid first send. If a turn first sends a payload with an explicit `replyToId` and then an untagged payload, the later payload loses the inherited current-message quote even though that inherited target was never used. Return/source-track whether the effective target is inherited, and consume shared state only for inherited sends.
  - confidence: 0.84
