## Summary

Keep open: current main still does not implement complete Signal bidirectional quote replies, and the latest PR head has two source-reproducible transport blockers plus insufficient live Signal proof before merge.

## What This Changes

The PR wires Signal quote-reply metadata through outbound delivery, expands inbound Signal quote context, adjusts reply threading/null semantics across channels and queue recovery, and adds Signal/outbound/queue tests plus a changelog entry.

## Best Possible Solution

Keep the PR open, require quote-author-safe native sends, container REST quote mapping with tests, green focused checks, and redacted live Signal outbound/inbound proof before merge.

## Reproduction Assessment

Yes. The blockers are source-reproducible: PR head can emit quote-timestamp without quote-author despite signal-cli rejecting that shape, and container mode drops quote fields that the REST API supports.

## Solution Assessment

No. The direction is useful, but the current patch is not the best mergeable solution until native quote metadata always includes an author, container sends forward quote fields, and live Signal proof is supplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

- **[P2] Require quote authors before emitting quote metadata:** `extensions/signal/src/reply-quote.ts:35-39`
  - body: signal-cli throws when `quote-timestamp` is present without `quote-author`, but this helper still returns timestamp-only metadata for direct targets and `sendMessageSignal` forwards it.   - confidence: 0.94

Container Signal users will still get plain replies unless those fields are forwarded and covered by a test.
  - confidence: 0.9
