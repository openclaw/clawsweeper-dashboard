## Summary

Keep open: current main does not have Signal quote-reply send support, and this PR is still the active implementation path, but the latest head has two source-level correctness blockers and still lacks live Signal send/receive proof.

## What This Changes

The PR adds bidirectional Signal quote-reply support, quote-author delivery/recovery plumbing, reply payload type changes, cross-channel reply null handling, tests, SDK baseline updates, and a changelog entry.

## Best Possible Solution

Keep the Signal quote-reply direction, but align quote visibility with the same author cascade used by structured quote parsing, consume reply state only when an inherited Signal target is actually used, and add redacted live Signal roundtrip proof before merge.

## Reproduction Assessment

Yes for the review blockers: source inspection shows authorNumber is parsed for structured quote targets but omitted from visibility gating, and Signal reply delivery has no source flag to distinguish explicit from inherited targets. I did not run a live Signal roundtrip.

## Solution Assessment

No. The implementation direction is useful, but the patch should fix the two source-level gaps and provide live Signal proof before it is the best merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use authorNumber for quote visibility:** `extensions/signal/src/monitor/event-handler.ts:695`
  - body: `rawQuoteTarget` now handles `quote.authorNumber`, but the visibility decision still comes from `resolveSignalQuoteContext`, which only resolves the quoted sender from `quote.author` and `quote.authorUuid`. In Signal groups with allowlist visibility, signal-cli payloads that only include `authorNumber` will be treated as not allowlisted and lose the new `ReplyTo*` context even when the quoted author is allowed.
  - confidence: 0.88
- **[P2] Consume only inherited Signal reply targets:** `extensions/signal/src/monitor.ts:411-414`
  - body: `markSignalReplyConsumed` receives only `effectiveReplyTo`, which also includes explicit per-payload overrides from `resolveSignalReplyDelivery`. A first payload with its own explicit `replyToId` can mark the shared inherited state consumed, so a later payload that omits an explicit target no longer inherits the current Signal message quote even though that inherited target was never used.
  - confidence: 0.86
