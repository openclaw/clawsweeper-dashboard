## Best Possible Solution

Finish the Signal quote contract for both native and container modes, keep explicit and inherited reply state independent, and require redacted live Signal roundtrip proof before merge.

No: the direction is maintainable, but the implementation needs container-mode quote mapping, explicit/inherited reply consumption separation, and live Signal proof before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Forward quote metadata through container mode:** `extensions/signal/src/send.ts:266-270`
  - body: The PR adds `quote-timestamp` and `quote-author` to the generic Signal RPC params here, but `signalRpcRequest` can route the same call to `containerRpcRequest`, whose `send` branch only forwards recipients, message, text styles, attachments, and timeout into `containerSendMessage`. Users configured for `apiMode: "container"` or auto-detected container mode would still get plain messages with no quote bubble, so the promised outbound Signal quote support remains incomplete until the container translator maps these fields and has coverage.
  - confidence: 0.9
- **[P2] Only consume inherited Signal replies when they are used:** `extensions/signal/src/monitor.ts:410-414`
  - body: `markSignalReplyConsumed` is called with `effectiveReplyTo`, but that value can be a payload-level explicit `replyToId`. If the first payload replies to a different explicit message, this marks the shared inherited current-message reply as consumed, so a later payload without an explicit id no longer inherits the original Signal quote target. Return the reply source from `resolveSignalReplyDelivery` or otherwise only consume the shared state when the inherited target was actually sent.
  - confidence: 0.86
