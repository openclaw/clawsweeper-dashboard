## Summary

Keep this PR open: it addresses a real Signal reply gap, but the current patch is not merge-ready because the default formatted Signal send path still drops quote metadata and the new inbound/monitor code has boundary and normalization regressions.

## What This Changes

The PR adds Signal quote-reply handling for outbound sends, inbound quote context extraction, reply metadata plumbing, and related Signal/shared outbound regression tests.

## Best Possible Solution

Revise the PR so Signal formatted text/media sends preserve quote metadata, new extension production code stays behind SDK or plugin-local seams, media fallback behavior remains compatible, and inbound quote normalization/debounce behavior is covered by focused tests.

## Reproduction Assessment

Yes. Source inspection is enough here: current main documents reply tags but Signal sends do not emit native quote parameters, and the PR head still drops `replyTo`/`quoteAuthor` on the normal Signal formatted outbound path.

## Solution Assessment

No. The direction is reasonable, but this patch misses the default Signal delivery path and introduces plugin-boundary, media fallback, quote ID, and debounce regressions that should be fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Route Signal formatted sends through quote metadata:** `src/infra/outbound/deliver.ts:1194-1197`
  - body: The PR adds quote metadata to the shared delivery overrides, but text-only Signal deliveries normally take `handler.sendFormattedText`, and the Signal formatted adapter still calls `send()` without `replyTo` or `quoteAuthor`. Generic Signal replies with `replyToId` will continue to post as top-level messages instead of native quotes.
  - confidence: 0.89
- **[P1] Keep Signal quote helpers behind the plugin boundary:** `extensions/signal/src/monitor/quote-context.ts:1-2`
  - body: This new production extension module imports core internals via `../../../../src/...`, which violates the scoped extension rules and couples the bundled plugin to host source layout. Move the MIME/E.164 helpers behind an SDK or plugin-local seam before merging.
  - confidence: 0.91
- **[P2] Fall back to mediaUrl when mediaUrls is empty:** `extensions/signal/src/monitor.ts:359-360`
  - body: Using `resolvedPayload.mediaUrls ?? ...` treats an empty `mediaUrls: []` as authoritative and skips the legacy `mediaUrl` fallback. A payload with `mediaUrls: []` plus `mediaUrl` now loses its attachment, unlike the previous reply-payload helper behavior.
  - confidence: 0.86
- **[P2] Require decimal Signal quote IDs:** `extensions/signal/src/monitor/quote-context.ts:67-68`
  - body: `Number(trimmed)` accepts non-decimal strings such as `1e3` and `0x10`, then normalizes them into different message IDs. Inbound quote IDs should use the same decimal-only validation as outbound quote timestamps before populating `ReplyToId` or author-cache lookups.
  - confidence: 0.82
- **[P2] Dispatch debounced quote-only messages:** `extensions/signal/src/monitor/event-handler.ts:503-504`
  - body: Quote-only Signal entries can be enqueued with an empty `bodyText` and a `quoteTarget`, but `onFlush` builds `combinedText` only from `bodyText` and returns when it is empty. That drops the quote-only inbound support path whenever the debouncer flushes those entries.
  - confidence: 0.84
- **[P2] Clear stale quote context after plain follow-ups:** `extensions/signal/src/monitor/event-handler.ts:508-518`
  - body: When a quoted message is followed by a plain message in the same debounce window, `toReversed().find(...)` carries the older quote into the merged turn even though the latest message has no quote. The merged context should use the latest message's quote state, including clearing it when absent.
  - confidence: 0.78
