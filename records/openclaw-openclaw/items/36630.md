## Summary

Keep open: this still targets a real Signal quote-reply gap, but the latest PR head has a P2 reply-state bug and the supplied proof does not show a live Signal send/receive roundtrip.

## What This Changes

Adds bidirectional Signal quote-reply handling by carrying outbound reply metadata through Signal delivery, parsing inbound signal-cli quote payloads into ReplyTo context, extending reply payload/queue types, and adding tests, changelog, and SDK baseline updates.

## Best Possible Solution

Keep the Signal-owned implementation direction, fix reply-delivery to track whether the effective target came from inherited state, add a regression for explicit-then-inherited payloads, coordinate overlapping Signal quote PRs, and require redacted live Signal proof before merge.

## Reproduction Assessment

Yes, at source level. Current main lacks outbound Signal quote RPC params, and the PR-head bug follows from sending an explicit payload reply before a later inherited reply through resolveSignalReplyDelivery and markSignalReplyConsumed.

## Solution Assessment

No. The implementation direction is maintainable, but the current patch should source-track inherited replies separately and include live Signal send/receive proof before merge.
