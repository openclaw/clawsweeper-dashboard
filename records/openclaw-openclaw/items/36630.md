## Summary

Keep open; the Signal quote-reply feature is useful and current main still lacks native bidirectional Signal quote support, but this PR head is not mergeable because it still has plugin-boundary, shared reply-threading, Signal delivery, and quote-id validation blockers.

## What This Changes

This PR adds Signal inbound quote-context parsing, outbound signal-cli quote metadata, quote-author plumbing, nullable reply suppression semantics, and regression tests across Signal and shared outbound delivery paths.

## Best Possible Solution

Land a smaller reworked Signal quote-reply implementation that keeps production plugin imports behind SDK surfaces, validates Signal quote ids as decimal timestamps, updates shared reply-threading semantics before widening payload types, and routes formatted, media, direct, queued, and recovered Signal deliveries through one consistent quote-author policy.

## Reproduction Assessment

Yes. Static reproduction is high confidence from PR head e99ada9f67a43473c8e565bfcfb20672d381e072 and current main: the raw files still show core deep imports, Number-based inbound quote-id parsing, blank reply IDs normalized to null, and null suppression tests without the matching threading implementation.

## Solution Assessment

No. The feature direction is right, but this PR is not the best merge vehicle until the SDK boundary, shared reply contract, formatted delivery path, media fallback, and quote-id validation are corrected together.
