## What This Changes

## Best Possible Solution

Keep the Signal quote-reply direction, but make quote visibility use the same author cascade as structured quote parsing, track whether Signal reply delivery consumed an inherited versus explicit target, and add redacted live Signal roundtrip proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use authorNumber for quote visibility decisions:** `extensions/signal/src/monitor/event-handler.ts:695`
  - body: `rawQuoteTarget` now handles `quote.authorNumber`, but the visibility decision still comes from `resolveSignalQuoteContext`, which only resolves the quoted sender from `quote.author` and `quote.authorUuid`. In Signal groups with `contextVisibility: "allowlist"`, signal-cli payloads that only include `authorNumber` will be treated as not allowlisted and lose the new `ReplyTo*` context even when the quoted author is allowed.
  - confidence: 0.88
- **[P2] Consume only inherited Signal reply targets:** `extensions/signal/src/monitor.ts:411-414`
  - body: `markSignalReplyConsumed` receives only `effectiveReplyTo`, so a first payload with its own explicit `replyToId` marks the shared Signal reply state consumed. Later payloads in the same assistant turn that omit an explicit target no longer inherit the current Signal message quote, even though the inherited target was never used; return a source/usedInherited flag from `resolveSignalReplyDelivery` and consume only that case.
  - confidence: 0.86
