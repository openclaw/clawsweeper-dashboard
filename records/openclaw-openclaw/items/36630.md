## Summary

Keep open: current main does not implement native Signal quote-reply support, and the latest PR head still has blocking correctness, plugin-boundary, security, and proof gaps before merge.

## What This Changes

The PR wires Signal quote metadata through outbound delivery, adds structured inbound quote context and author lookup helpers, extends reply payload/context typing, and adds Signal/outbound regression tests.

## Best Possible Solution

Revise the PR so Signal quote metadata is preserved through formatted and direct send paths without core deep imports, existing media/debounce behavior remains compatible, and the contributor adds redacted live Signal proof plus focused regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main lacks Signal quote RPC params, and the PR head still drops `replyTo`/`quoteAuthor` on the normal formatted Signal outbound path.

## Solution Assessment

No. The overall direction is reasonable, but the current patch is not the narrowest maintainable solution until it fixes the formatted send path, plugin boundary violations, quote ID validation, debounce/media regressions, and real behavior proof gap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Forward quote metadata from formatted Signal sends:** `extensions/signal/src/channel.ts:191-197`
  - body: `deliverOutboundPayloadsCore` now passes quote overrides into `handler.sendFormattedText`, and Signal text-only sends normally take this formatted path. `sendFormattedSignalText` still calls `send()` without `replyTo` or `quoteAuthor`, so generic Signal replies with `replyToId` continue to post as top-level messages instead of native quote replies.
  - confidence: 0.89
- **[P1] Remove core deep imports from Signal quote helper:** `extensions/signal/src/monitor/quote-context.ts:1-2`
  - body: This new production extension module imports `../../../../src/...` internals, which violates the extension boundary and can break standalone/plugin-package builds. Move the MIME/E.164 needs behind an SDK or plugin-local seam before merging.
  - confidence: 0.91
- **[P1] Dispatch debounced quote-only Signal batches:** `extensions/signal/src/monitor/event-handler.ts:503-504`
  - body: When multiple quote-only Signal messages debounce together, `combinedText` is built only from `bodyText` and the flush returns when it is empty. That drops the whole batch even though this PR makes quote-only messages valid inbound context.
  - confidence: 0.84
- **[P2] Preserve mediaUrl fallback when mediaUrls is empty:** `extensions/signal/src/monitor.ts:369-370`
  - body: Using nullish coalescing treats `mediaUrls: []` as authoritative and skips the legacy `mediaUrl` fallback. Payloads with an empty multi-media array plus a single `mediaUrl` now lose the attachment, and text-empty payloads can be skipped entirely.
  - confidence: 0.84
- **[P2] Require decimal Signal quote IDs:** `extensions/signal/src/monitor/quote-context.ts:67-68`
  - body: `Number(trimmed)` accepts values such as `1e3` and `0x10`, normalizing malformed quote IDs into different message timestamps. Inbound quote normalization should match the outbound decimal-only parser before surfacing `ReplyToId` or doing author-cache lookup.
  - confidence: 0.8
