## Summary

Keep this PR open, but do not merge it as-is. Current main has partial Signal inbound quote context but no outbound signal-cli quote parameter support, so the feature is still useful; however, the latest PR head still has merge-blocking SDK boundary violations, reply-threading regressions, incomplete formatted/queued delivery plumbing, and a security-sensitive quote-id validation bug.

## What This Changes

The PR adds Signal inbound quote-context parsing, outbound signal-cli quote metadata, shared outbound `quoteAuthor` plumbing, nullable reply suppression semantics, and regression tests across Signal and shared outbound reply delivery paths.

## Best Possible Solution

Land a smaller reworked Signal quote-reply implementation that keeps production plugin imports behind SDK surfaces, validates Signal quote IDs as decimal timestamps, updates shared reply-threading semantics before widening public payload types, and routes direct, formatted, media, queued, and recovered Signal deliveries through one consistent quote-author policy.

## Reproduction Assessment

Yes. Static reproduction is high confidence from PR head `e884402b9a4c90fa3fb3f7786333d65c27070019`: raw source shows the unchanged truthy threading check, core deep imports, `Number(trimmed)` quote-id parsing, formatted Signal sends without quote fields, and queue storage without `quoteAuthor`.

## Solution Assessment

No. The feature direction is useful, but this PR is not the best merge vehicle until the SDK boundary, shared reply contract, formatted/queued delivery paths, debounce semantics, media fallback, and quote-id validation are corrected together.

  - body: This widens `replyToId` to include `null`, but `resolveReplyThreadingForPayload` still treats falsy values as unset and can inject the implicit current message id. Explicit do-not-reply payloads can still be threaded, and the new null-suppression tests disagree with the implementation.

  - confidence: 0.97
- **[P1] Move reply-delivery types behind the SDK:** `extensions/signal/src/monitor/reply-delivery.ts:1`
  - body: This production Signal helper imports `ReplyPayload` from a core `src/**` path. Use the public reply runtime/payload SDK surface or a plugin-local type facade instead of coupling plugin code to core internals.
  - confidence: 0.96
- **[P1] Forward quote metadata through formatted Signal sends:** `src/infra/outbound/deliver.ts:1133`
  - body: Adding `quoteAuthor` to shared send overrides is not enough for the default formatted Signal path: `sendFormattedSignalText` and `sendFormattedSignalMedia` still call `sendMessageSignal` without `replyTo` or `quoteAuthor`. Generic Signal sends with reply metadata will continue to post without native quote bubbles.
  - confidence: 0.91
- **[P2] Require decimal-only inbound quote IDs:** `extensions/signal/src/monitor/quote-context.ts:67-68`
  - body: `normalizeSignalQuoteId` uses `Number(trimmed)`, which accepts values like `1e3` and `0x10`. Those can alias inbound reply context and author-cache lookups to a different message id; inbound quote IDs should use the same decimal-only validation as outbound quote timestamps.
  - confidence: 0.92
- **[P2] Treat blank Signal reply IDs as unset:** `extensions/signal/src/monitor/reply-delivery.ts:12-13`
  - body: `normalizeReplyToId` converts whitespace-only strings to `null`, which this module treats as explicit reply suppression. Blank payload ids should behave like `undefined` so inherited quote context can still apply; reserve `null` for an intentional do-not-reply override.
  - confidence: 0.89
- **[P2] Persist quote authors in queued deliveries:** `src/infra/outbound/deliver.ts:972`
  - body: The live delivery path now carries `quoteAuthor`, but the write-ahead queue still persists and replays only `replyToId`. A recovered Signal group reply will lose the author required for `quote-author` and will be replayed unquoted.
  - confidence: 0.87
- **[P2] Fall back to mediaUrl when mediaUrls is empty:** `extensions/signal/src/monitor.ts:359-360`
  - body: `resolvedPayload.mediaUrls ?? ...` treats an explicit empty array as final, so `{ mediaUrls: [], mediaUrl: "..." }` drops the single-media fallback that the previous shared reply-payload helper preserved. Media replies can be skipped even though a legacy `mediaUrl` is present.
  - confidence: 0.89
- **[P2] Do not carry stale debounced quote targets forward:** `extensions/signal/src/monitor/event-handler.ts:508-510`
  - body: For debounced batches, this chooses the latest entry that has a quote, even when the actual latest message is plain. A quoted message followed quickly by a normal follow-up can surface stale `ReplyTo*` context for the combined turn.
  - confidence: 0.86
- **[P2] Process debounced quote-only batches:** `extensions/signal/src/monitor/event-handler.ts:503-504`
  - body: When a debounced batch contains only quote-only entries, `combinedText` is empty and this early return drops the whole batch before history or inbound context is recorded. Quote-only messages are now intended to be valid inbound content, so the flush path should not require body text.
  - confidence: 0.88

## Security Review
