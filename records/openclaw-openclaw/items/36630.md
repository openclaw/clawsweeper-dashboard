## Summary

Keep open. Current main still lacks native Signal quote-send support, and this PR is not merge-ready because the normal formatted Signal outbound path still drops reply metadata and the new Signal monitor helpers cross the plugin boundary.

## What This Changes

The PR adds Signal quote metadata handling for outbound sends, structured inbound quote context, reply-consumption logic, and related Signal/shared outbound regression tests.

## Best Possible Solution

Revise the PR so Signal formatted text/media sends preserve reply metadata, new extension code uses SDK or plugin-local seams, media fallback behavior stays compatible, and quote IDs are validated consistently before rerunning the Signal/outbound changed gates.

## Reproduction Assessment

Yes. Source inspection shows current main lacks quote params in Signal sends, and the PR head still drops replyTo/quoteAuthor on the normal Signal formatted outbound path.

## Solution Assessment

No. The direction is reasonable, but the current patch misses the default Signal send path and introduces boundary and normalization regressions that should be fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route Signal formatted sends through quote metadata:** `src/infra/outbound/deliver.ts:1196`
  - body: The new quote metadata is added to the generic delivery overrides, but Signal text-only deliveries normally take `handler.sendFormattedText`, and `extensions/signal/src/channel.ts` still calls `send()` without `replyTo` or `quoteAuthor`. That means generic Signal outbound replies with `replyToId` continue to send as plain top-level messages, so the outbound half of this PR is bypassed on the main channel path.
  - confidence: 0.88
- **[P1] Keep Signal monitor code behind the plugin boundary:** `extensions/signal/src/monitor/quote-context.ts:1-2`
  - body: This new production extension module imports core internals via `../../../../src/...`, which the scoped extension rules forbid because bundled plugins must use `openclaw/plugin-sdk/*` or local plugin barrels. Please move the needed MIME/E.164 helpers behind an SDK/local seam instead of coupling the Signal plugin package to core source layout.
  - confidence: 0.91
- **[P2] Preserve mediaUrl when mediaUrls is empty:** `extensions/signal/src/monitor.ts:359-360`
  - body: Using `resolvedPayload.mediaUrls ?? ...` treats an empty `mediaUrls: []` as authoritative and skips the legacy `mediaUrl` fallback. Existing reply-payload helpers fall back when the array is empty, so a payload with `{ mediaUrls: [], mediaUrl: "..." }` now loses the attachment in Signal monitor delivery.
  - confidence: 0.84
- **[P2] Require decimal Signal quote IDs:** `extensions/signal/src/monitor/quote-context.ts:67-68`
  - body: `Number(trimmed)` accepts non-decimal strings such as `1e3` and `0x10`, which can normalize malformed inbound quote IDs to different message timestamps. The outbound quote parser already requires decimal digits; inbound quote normalization should use the same constraint before surfacing ReplyToId or doing author-cache lookup.
  - confidence: 0.8
