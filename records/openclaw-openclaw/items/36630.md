## Summary

Keep open: current main still lacks the central Signal quote-reply behavior, and this PR is a plausible implementation path, but the current patch is not merge-ready because it still imports a deprecated SDK subpath, can regress Tlon thread routing, and lacks live Signal send/receive proof.

## What This Changes

The PR adds bidirectional Signal quote-reply sending/parsing, carries quote metadata through outbound delivery and queue recovery, widens reply payload types, and updates channel tests, changelog, and API baseline files.

## Best Possible Solution

Keep the Signal quote-reply direction, but route normalizeE164 through the supported text-utility-runtime subpath, make inherited reply consumption capability-aware for persistent-thread channels such as Tlon, refresh required generated artifacts after rebasing, and add redacted live Signal roundtrip proof.

## Reproduction Assessment

Yes for the review blockers: CI reproduces the deprecated SDK import failure, and source inspection shows the Tlon persistent-thread regression path. I did not reproduce a live Signal roundtrip.

## Solution Assessment

No. The implementation direction is reasonable, but this patch is not ready until the SDK boundary failure, Tlon thread-routing regression, generated-artifact check status, and missing live Signal proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the supported text utility SDK subpath:** `extensions/signal/src/monitor/quote-context.ts:2`
  - body: This production extension imports `normalizeE164` from deprecated `openclaw/plugin-sdk/text-runtime`, and the current plugin contract check fails on this exact import. Use `openclaw/plugin-sdk/text-utility-runtime` instead so bundled plugin production code stays off deprecated broad SDK barrels.
  - confidence: 0.97
- **[P2] Preserve Tlon thread replies across payloads:** `src/infra/outbound/deliver.ts:1460-1461`
  - body: This new persistent-thread allowlist excludes Tlon, so inherited `replyToId` is consumed after the first payload unless `replyToMode` is `all`. Tlon outbound advertises reply/thread support and uses `replyToId` to build the Urbit thread reply action, so multi-payload group replies can put only the first payload in-thread.
  - confidence: 0.86
