## Summary

Keep this PR open. The Signal quote-reply direction is useful, but the current diff has a blocking plugin SDK guardrail failure, still risks breaking persistent-thread reply routing for non-Signal channels, and the supplied proof does not show a live Signal send/receive round trip.

## What This Changes

The PR adds Signal quote-reply parsing and send metadata, threads quote-author state through outbound delivery and queue recovery, and updates tests, changelog, and SDK API baseline files.

## Best Possible Solution

Keep the PR open, move the Signal quote-context import to a supported SDK subpath, make inherited reply consumption capability-aware for persistent-thread channels, rerun the failed gates, and add redacted live Signal send/receive proof.

## Reproduction Assessment

Yes for the review blockers: CI logs reproduce the deprecated SDK import failure, and source inspection shows the new reply-consumption allowlist suppresses inherited reply IDs for thread-based channels like Tlon. I did not reproduce the live Signal round trip.

## Solution Assessment

No. The implementation direction is reasonable, but this patch is not ready until the SDK boundary failure and persistent-thread routing regression are fixed and live Signal behavior is proven.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Use a supported SDK text helper import:** `extensions/signal/src/monitor/quote-context.ts:2`
  - body: The plugin contract shard fails because this production extension imports deprecated `openclaw/plugin-sdk/text-runtime`. The guardrail reports this exact file under `extensionImports`, so the PR cannot pass CI until the import moves to a supported SDK subpath, for example the existing text utility runtime.
  - confidence: 0.97
- **[P2] Preserve thread replies for persistent-thread channels:** `src/infra/outbound/deliver.ts:1462-1470`
  - body: This allowlist treats every channel except Slack, Mattermost, and Google Chat as single-use reply metadata. Tlon uses `replyToId` as the thread id for group replies and advertises both `replyTo` and `thread` support, so a multi-payload turn will send the first payload in-thread and later payloads at top level. Key this off channel capabilities or include the other persistent-thread adapters.
  - confidence: 0.86
