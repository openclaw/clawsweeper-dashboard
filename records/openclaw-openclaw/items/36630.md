## Summary

Keep open: current main does not already implement native Signal quote replies, and PR head still has source-proven functional blockers plus insufficient live Signal proof before merge.

## What This Changes

The PR adds bidirectional Signal quote-reply handling across the Signal plugin, outbound delivery reply metadata, reply payload typing, queue persistence, tests, SDK baseline, and changelog.

## Best Possible Solution

Keep the Signal quote-reply direction, but fix the quote-author and reply-state semantics, then collect redacted live Signal device/account proof before merge.

## Reproduction Assessment

Yes for the review blockers by source inspection: PR head can emit quote-timestamp without quote-author, consume inherited reply before adapter success, and mark Signal inherited state consumed for explicit reply targets. I did not run a live Signal roundtrip.

## Solution Assessment

No. The implementation direction is reasonable, but this patch should not merge until it respects signal-cli's quote-author requirement, commits inherited reply consumption only after successful delivery, distinguishes inherited from explicit reply targets, and supplies live proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require quote authors before emitting Signal quotes:** `extensions/signal/src/reply-quote.ts:35-39`
  - body: signal-cli throws when `quote-timestamp` is supplied without `quote-author`, but this helper still returns a timestamp alone for direct chats. Direct Signal sends with a `replyToId` but no resolved author can now fail instead of degrading to a normal message; drop all quote params unless an author is available or resolve an author for every quote send.
  - confidence: 0.91
- **[P2] Consume inherited replies only after delivery succeeds:** `src/infra/outbound/deliver.ts:1637-1639`
  - body: `applySendReplyToConsumption` calls `applyReplyToConsumption` while preparing overrides, before the adapter has returned a delivered message. In `bestEffort`, a first quoted send failure leaves later payloads seeing the inherited reply as already consumed, so no successful payload gets the quote.
  - confidence: 0.88
- **[P2] Consume Signal state only for inherited replies:** `extensions/signal/src/monitor/reply-delivery.ts:63-76`
  - body: `markSignalReplyConsumed` only receives the effective reply id, so an explicit per-payload reply target can consume the shared inherited state. If the next payload omits `replyToId`, it no longer inherits the current Signal message and loses its quote context.
  - confidence: 0.84
