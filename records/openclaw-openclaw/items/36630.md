## Summary

Keep this PR open: current main still lacks native Signal quote-reply support, and the branch is an active, useful implementation, but the latest head is not merge-ready because quote metadata can still be emitted in an upstream-invalid shape, container-mode sends drop the new fields, and live Signal roundtrip proof is still missing.

## What This Changes

The PR adds bidirectional Signal quote-reply support across inbound quote context parsing, outbound quote metadata, queued delivery recovery, reply-threading contracts, tests, and changelog coverage.

## Best Possible Solution

Land a finished Signal reply implementation that satisfies signal-cli quote contracts in both native and container modes, preserves queued reply context, and includes redacted live Signal send/receive proof.

## Reproduction Assessment

Yes. Source inspection at PR head shows quote-timestamp can be emitted without quote-author and container sends do not forward quote fields, while upstream signal-cli requires quote-author when quote-timestamp is present.

## Solution Assessment

No. The implementation direction is right, but the patch should require or resolve quote authors before emitting quote metadata, map quote fields through container REST sends, and supply live Signal proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Require quote authors before sending quote metadata:** `extensions/signal/src/reply-quote.ts:34-39`
  - body: Upstream signal-cli rejects `quote-timestamp` without `quote-author`, but this helper still returns timestamp-only metadata for non-group targets and `sendMessageSignal` forwards it. Direct replies with an unresolved author can fail instead of degrading to a normal message, so drop quote params unless both fields are available or resolve the author before sending.
  - confidence: 0.94
- **[P2] Forward quote fields through container sends:** `extensions/signal/src/send.ts:266-271`
  - body: The new quote params only reach the native RPC shape. When `signalRpcRequest` routes through container mode, `containerRpcRequest` calls `containerSendMessage` without mapping `quote-timestamp` or `quote-author`, even though the container REST API supports `quote_timestamp` and `quote_author`, so container Signal users still get plain replies.
  - confidence: 0.9
