## Summary

Keep open: current main and v2026.5.4 still have the self-inconsistent doctor path, and the related #73471 is a broader doctor-recommendation concern rather than a duplicate fix for this concrete visible-replies case.

## What This Changes

The issue requests that `doctor --fix` stop writing `messages.groupChat.visibleReplies = "message_tool"` when the resolved tool policy lacks the `message` tool.

## Best Possible Solution

Make the doctor visible-reply migration idempotent with its own warning checks by reusing the message-tool availability resolver before writing `message_tool`, and cover the fix with a regression test for normalize-then-preview behavior.

## Reproduction Assessment

Yes from source inspection for a restrictive tool policy: configured channels with no explicit visible reply policy are normalized to explicit `message_tool`, then the preview warning code flags that explicit value when `message` is unavailable. I did not establish a live CLI reproduction, and the report's `full` profile example is not the strongest current-main reproducer.

## Solution Assessment

Yes, gating the migration on the same message-tool availability logic used by doctor warnings is the narrowest maintainable fix. If unavailable, doctor should leave the value unset or choose `automatic` rather than creating a setting it immediately warns about.
