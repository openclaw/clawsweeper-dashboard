## Summary

Keep open: current `main` still has the source-level self-warning path where doctor compatibility repair writes `messages.groupChat.visibleReplies: "message_tool"` without checking message-tool availability, and the linked closing PR is still open rather than merged.

## What This Changes

The issue asks for `doctor --fix` to avoid setting `messages.groupChat.visibleReplies` to `message_tool` when the active tool policy cannot use the `message` tool.

## Best Possible Solution

Make doctor compatibility repair policy-aware, write `automatic` when `message` is unavailable and `message_tool` when it is available, then land a corrected linked PR with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows a configured channel with no visible-reply setting and a policy omitting `message` can be repaired to explicit `message_tool`, and the warning helper then treats that value as unavailable.

## Solution Assessment

Yes for the requested direction: the narrow fix is to share the message-tool availability semantics between repair and warning paths. Current `main` is not fixed, and the open PR should persist `automatic` or change the default/warning contract consistently rather than leaving the value unset.
