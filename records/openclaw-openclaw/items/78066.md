## Summary

Keep open: current main still has the source-level loop where doctor compatibility repair writes `messages.groupChat.visibleReplies: "message_tool"` without checking message-tool availability, and the live closing PR is still open and unmerged.

## What This Changes

The issue asks for `doctor --fix` to avoid recommending `messages.groupChat.visibleReplies: "message_tool"` when the resolved tool policy does not expose the `message` tool.

## Best Possible Solution

Make the doctor compatibility normalizer share the message-tool availability semantics with the warning path, persist `automatic` when `message` is unavailable and `message_tool` when it is available, then land a corrected linked PR with focused regression coverage.

## Reproduction Assessment

Yes, at source level. A configured-channel config with no visible-reply setting is repaired to explicit `message_tool`, and the warning helper then flags that value when the resolved tool policy omits `message`; I did not run mutating `doctor --fix` commands in this read-only review.

## Solution Assessment

Yes, the requested policy-aware repair is the narrow maintainable direction. The current open PR needs correction because returning early can leave the configured-channel default unset, which the warning path still treats as default `message_tool`.
