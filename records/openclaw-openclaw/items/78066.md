## Summary

Keep open: current main still has a source-level path where doctor compatibility repair writes `messages.groupChat.visibleReplies: "message_tool"` for configured channels without checking whether the resolved tool policy allows the `message` tool, and the next non-fix doctor pass can warn about that explicit setting. The reported `tools.profile: "full"` example does not match current source/docs because `full` allows all tools, but the same bug is clear for narrower policies such as `coding`, `minimal`, or explicit allowlists that omit `message`.

## What This Changes

The issue asks doctor compatibility repair to avoid writing a `message_tool` group visible-reply policy when the active tool policy cannot use the message tool.

## Best Possible Solution

Make doctor compatibility repair policy-aware: preserve explicit user settings, continue recommending `message_tool` when the resolved policy can use `message`, and otherwise leave the setting unset or set `automatic` so doctor does not create its own follow-up warning.

## Reproduction Assessment

Yes, at source level: a configured channel with no explicit visible-reply policy and a resolved tool policy that omits `message` causes the repair normalizer to write `message_tool`, after which `collectVisibleReplyToolPolicyWarnings` treats it as an explicit mismatch. I did not run the mutating CLI reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to gate the doctor-set recommendation on the same message-tool availability logic used by the warning path. The current implementation is not the best solution because it can persist a value that doctor immediately tells the operator to undo.
