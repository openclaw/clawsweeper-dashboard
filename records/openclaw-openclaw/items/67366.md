## Summary

Keep open. Current main still contains the unsafe Telegram token replacement prompt path, and the previously active implementation PR is now closed unmerged, so the issue remains a valid narrow onboarding bug.

## What This Changes

Fix `openclaw onboard` so replacing an existing Telegram bot token accepts or validates the new token without throwing a `.trim()` TypeError.

## Best Possible Solution

Normalize text prompt results at the shared setup-wizard boundaries, preserve existing valid-token behavior, and add focused regression coverage for undefined replacement-token prompt returns.

## Reproduction Assessment

Yes, by source inspection rather than an executed live session: an already-configured Telegram account can decline the keep-token prompt, then the replacement token prompt result is trimmed without a nullish guard. Multiple reporter comments confirm the same `2026.4.14` crash path.

## Solution Assessment

Yes. A narrow nullish normalization at `promptSingleChannelToken` and the generic setup text path is the maintainable fix; the broader closed PR is no longer an active landing path.
