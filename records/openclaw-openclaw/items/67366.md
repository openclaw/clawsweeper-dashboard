## Summary

Keep open. Current main and the latest release still contain the nullish prompt-result `.trim()` crash path for Telegram token replacement, while the two linked fix PRs are closed unmerged.

## What This Changes

The issue asks for `openclaw onboard` to replace an existing Telegram bot token without throwing a `TypeError` when the prompt result is nullish.

## Best Possible Solution

Normalize or explicitly reject nullish text prompt results at the shared setup wizard prompt boundary, then add regression coverage for Telegram token replacement and generic setup text inputs.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: an already-configured Telegram channel can decline the keep-token prompt and then the token helper trims the raw text prompt result without a nullish guard; I did not run a live interactive TTY reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to guard the shared setup prompt result before string methods run and cover both the Telegram credential path and generic setup text input path, instead of patching only one Telegram-specific call site.
