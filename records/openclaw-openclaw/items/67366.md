## Summary

Keep this open. Current main still has the reported Telegram token replacement crash path: the configured-channel update flow routes back into Telegram setup, Telegram still asks the same keep/input prompts, and the shared credential helper still calls `.trim()` directly on the text prompt result. The two linked fix PRs are closed unmerged, and no open PR specifically closes this issue.

## What This Changes

The issue requests that replacing an existing Telegram bot token during `openclaw onboard` save the new token or show validation instead of throwing `Cannot read properties of undefined (reading 'trim')`.

## Best Possible Solution

Patch the shared channel setup text handling so every text prompt result is normalized or rejected before string methods run, covering both `promptSingleChannelToken` and the generic setup wizard text-input path. Add a focused regression test for replacing an existing Telegram token with a text prompt that resolves `undefined`, using fake token values and preserving cancel behavior.

## Reproduction Assessment

Yes. The issue provides concrete interactive steps, multiple commenters confirmed it on 2026.4.14, and current main has a direct source-level reproduction path by mocking the token replacement prompt so `prompter.text()` resolves `undefined` after the keep prompt is declined.

## Solution Assessment

Yes, a narrow shared setup-wizard guard plus regression tests is the best path. Fixing only Telegram would leave the generic setup text-input path unsafe, while changing unrelated onboarding/product flow is not needed.
