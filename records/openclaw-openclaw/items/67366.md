## Summary

Keep open. Current main and the latest shipped release still contain the unsafe Telegram token replacement `.trim()` path, multiple reporters confirmed the onboarding crash on v2026.4.14, and the related fix PRs are all closed unmerged.

## What This Changes

The issue asks for `openclaw onboard` to accept or gracefully reject a replacement Telegram bot token instead of throwing a TypeError.

## Best Possible Solution

Normalize unexpected or nullable text prompt results at the shared prompter/setup boundary, preserve valid token and cancel semantics, and add focused regression coverage for Telegram token replacement.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an already configured Telegram account can decline the keep-token prompt, then `promptSingleChannelToken` trims the replacement prompt result without a defensive boundary guard; multiple reporters confirmed the shipped crash.

## Solution Assessment

Yes. A narrow boundary fix with regression tests is the maintainable route; the earlier one-line caller-only PRs were closed unmerged and had cancel/required-field semantics concerns.
