## Summary

Keep open. Current main and the latest release still contain the reported nullish text-prompt crash path, and the two related fix PRs in the issue context are closed unmerged.

## What This Changes

The issue requests that Telegram onboarding accept and save a replacement bot token without throwing when the existing token is declined.

## Best Possible Solution

Normalize or explicitly reject nullish text prompt results at the shared setup prompt boundary before any string methods run, then add focused regression coverage for Telegram token replacement and the generic setup wizard text-input path.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: an already-configured Telegram channel can decline the keep-token prompt, then `promptSingleChannelToken` trims the raw `prompter.text()` result without a nullish guard. I did not run a live interactive TTY reproduction because this review is read-only.

## Solution Assessment

Yes. A narrow shared setup-wizard guard plus regression tests is the maintainable fix; changing only Telegram would leave the generic setup text-input path with the same nullish prompt-result hazard.
