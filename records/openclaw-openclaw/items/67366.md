## What This Changes

The issue reports a `TypeError` during `openclaw onboard` when replacing an existing Telegram bot token and asks onboarding to save the new token or fail with a friendly validation error.

## Best Possible Solution

Land or split a narrow nullish prompt-result normalization fix with regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an already-configured Telegram account can decline the keep-token prompt, then the replacement token prompt result is trimmed without a nullish guard.

## Solution Assessment

Yes. Normalizing prompt text results at the shared setup-wizard boundary is the narrow maintainable fix; https://github.com/openclaw/openclaw/pull/77813 has the right guard shape, though it may need to be split from unrelated channel streaming work.
