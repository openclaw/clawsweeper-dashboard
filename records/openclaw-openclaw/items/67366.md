## Summary

Keep open. Current main and the latest shipped tag still contain the reported Telegram token replacement crash path: configured Telegram channels can enter Modify settings, the Telegram setup surface still asks the same replacement prompts, and the shared token helper still trims the raw text prompt result without normalizing it first. The two linked fix PRs are closed unmerged, so there is no merged implementation to close against.

## What This Changes

Replacing an existing Telegram bot token during onboarding should save the new token or show a friendly validation error instead of throwing an undefined `.trim()` TypeError.

## Best Possible Solution

Normalize or reject text prompt results at the shared channel setup boundary before any string methods or parsers run, then add focused regression coverage for Telegram token replacement and the generic setup wizard text-input path using fake token values.

## Reproduction Assessment

Yes. The issue gives concrete interactive steps confirmed by multiple commenters, and current main has a source-level reproduction path by making `prompter.text()` resolve `undefined` after the existing-token keep prompt is declined.

## Solution Assessment

Yes. A narrow shared setup-wizard guard plus regression tests is the maintainable fix; changing only Telegram would leave the generic setup text-input path with the same nullish prompt-result hazard.
