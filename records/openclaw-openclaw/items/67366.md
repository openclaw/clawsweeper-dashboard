## What This Changes

The issue asks for `openclaw onboard` and related channel setup to accept or gracefully reject a replacement Telegram bot token instead of throwing a TypeError.

## Best Possible Solution

Normalize nullable or otherwise unexpected text prompt results at the shared setup/propmter boundary, preserve valid-token and cancel semantics, and add focused regression coverage for Telegram token replacement.

Yes. Source inspection gives a high-confidence path: decline the keep-existing Telegram token prompt, then the replacement prompt result reaches `.trim()` without a string guard; several reporters confirmed the same crash in released builds.

## Solution Assessment

Yes. A narrow setup-boundary normalization with regression tests is the maintainable fix; the earlier caller-only PRs were closed unmerged and had cancel/required-field semantics concerns.
