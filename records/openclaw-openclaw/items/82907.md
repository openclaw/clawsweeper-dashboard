## Summary

Keep open: this PR targets the right Matrix reasoning regression, but it needs after-fix real behavior proof and the current patch appears to strand Matrix draft-preview block-boundary state when a reasoning block is delivered.

## What This Changes

Adds a Matrix opt-in for delivering explicit `isReasoning` payloads through generic reply dispatch and adjusts Matrix reasoning suppression tests.

## Best Possible Solution

Keep the narrow Matrix opt-in, but keep explicit reasoning blocks out of draft-preview boundary bookkeeping before requiring redacted Matrix/Element after-fix proof.

## Reproduction Assessment

Yes. Source inspection reproduces the linked bug: enabled reasoning can emit `isReasoning` payloads, but current generic dispatch and Matrix delivery both suppress them before Matrix sends a message.

## Solution Assessment

No. The direction is right, but the patch should also prevent reasoning block deliveries from queueing or stranding Matrix draft-preview boundaries before it is the best maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep reasoning blocks out of draft boundaries:** `extensions/matrix/src/matrix/monitor/handler.ts:1770-1787`
  - body: When Matrix draft streaming is enabled, `dispatchReplyFromConfig` will now call `onBlockReplyQueued` for `isReasoning` block payloads, but this new branch returns before the existing block cleanup advances the queued draft boundary. That leaves the reasoning boundary in `pendingDraftBoundaries`, so the next answer block can rotate the preview using the wrong offset. Skip queueing reasoning blocks for Matrix drafts, or consume the boundary before returning.
  - confidence: 0.84
