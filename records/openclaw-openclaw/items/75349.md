## Summary

Keep open. Current main still advertises `github-copilot/claude-opus-4.6`, and the current 400/422 failover path still treats an unrecognized `model_not_supported` Copilot response as a `format` failure that becomes a profile-wide auth cooldown. PR #75391 is an open closing PR, but it only removes one catalog entry and has not merged, so this issue remains a valid two-part provider compatibility bug.

## What This Changes

Request to remove or dynamically validate stale GitHub Copilot default model IDs and make unsupported-model responses model-scoped instead of cooling down the whole Copilot auth profile.

## Best Possible Solution

Implement the fix in two layers: keep the bundled Copilot catalog conservative or runtime-backed, and classify Copilot `model_not_supported`/unsupported-model responses as model-scoped unavailability so other Copilot models can continue using the same auth profile. PR #75391 should be expanded to cover the cooldown behavior or superseded by a complete fix.

## Reproduction Assessment

Yes. The reporter provides a concrete v2026.4.29 command and gateway log, and current-main source still deterministically contains both the stale advertised model and the profile-wide cooldown path for the reported unclassified 400 response.

## Solution Assessment

Yes for the issue's two-layer direction; no for a catalog-only change by itself. Removing `claude-opus-4.6` is a useful mitigation, but the maintainable fix also needs unsupported-model classification that does not poison the entire Copilot auth profile.
