## Summary

Keep open. Current main still advertises `github-copilot/claude-opus-4.6`, still lacks a `model_not_supported`/unsupported-model classifier for Copilot's reported 400 payload, and still scopes auth-profile cooldown bypass only to rate limits. The open closing PR #75391 has not merged and only removes one catalog entry, so it is an active implementation candidate rather than a reason to close this issue.

## What This Changes

The issue reports that the bundled GitHub Copilot catalog includes stale `claude-opus-4.6` defaults and that Copilot `400 model_not_supported` responses cause provider-wide auth-profile cooldowns instead of model-scoped unavailability.

## Best Possible Solution

Fix this in two layers: make the bundled Copilot catalog conservative or runtime-backed, and classify Copilot unsupported-model responses as model-scoped unavailability so other Copilot models can keep using the same auth profile. PR #75391 should either be expanded to cover the cooldown behavior and manifest/test/docs/changelog follow-through or be superseded by a complete fix.

## Reproduction Assessment

Yes. The reporter provided a concrete v2026.4.29 command and gateway log, and current-main source still deterministically contains both the stale advertised model and the profile-wide cooldown path for an unclassified Copilot 400 unsupported-model response.

## Solution Assessment

Yes for the issue's two-layer direction; no for a catalog-only change by itself. Removing `claude-opus-4.6` is a useful mitigation, but the maintainable fix also needs unsupported-model classification that does not poison the entire Copilot auth profile.
