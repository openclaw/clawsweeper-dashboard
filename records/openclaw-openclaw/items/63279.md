## Summary

Keep open. Current main contains meaningful partial hardening for config/model reloads, fallback handling, drain lifecycle, subagent delivery, and compaction recovery, but source still shows unresolved pieces of the reported cascade: drain rejects new enqueues, reply handling returns restart text instead of replaying the turn, subagent announce delivery can still give up, and compaction recovery still has bounded fallthrough paths.

The issue asks OpenClaw to investigate and harden a large Telegram-session failure chain involving compaction timeout, gateway drain rejection, subagent announce loss, and stale runtime fallback/config state after a provider was removed.

## Best Possible Solution

Keep this as the umbrella until maintainers split or land coordinated reliability work for drain-window replay, durable or intentionally bounded subagent completion delivery, compaction timeout/overflow escalation, and documented live fallback/session-override reload semantics.
