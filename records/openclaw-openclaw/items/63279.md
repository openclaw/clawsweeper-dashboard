## Summary

Keep open. Current main has partial hardening, including model/config hot-reload handling, bounded timeout-compaction attempts, explicit restart/drain handling, and subagent announce fallback paths, but source and docs still show unresolved parts of the reported failure chain: gateway drain rejects new enqueues, reply handling returns a restart message instead of retrying the turn, subagent completion delivery can still give up, and timeout compaction can still fall through to blocked/error handling.

## What This Changes

The issue requests investigation and fixes for a large-session overflow/compaction timeout cascade involving gateway draining, subagent announce loss, and stale runtime fallback-chain behavior after config changes.

## Best Possible Solution

Keep this as the umbrella until maintainers either split it into narrower canonical follow-ups or land a coordinated reliability fix covering restart/drain retry semantics, durable or explicitly bounded subagent completion delivery, compaction timeout recovery policy, and documented live fallback/session-override behavior after config changes.

## Reproduction Assessment

Unclear for the full incident cascade. Source inspection on current main does reproduce individual remaining behaviors, but I did not establish a complete Telegram large-session overflow plus compaction-timeout plus gateway-drain reproduction path.

## Solution Assessment

Not applicable as a patch review. Current main is a partial mitigation, not the best complete solution, because the remaining work spans restart policy, delivery guarantees, compaction recovery, and model/session override semantics.
