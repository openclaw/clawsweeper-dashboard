## Summary

Keep open: current main still drops inbound Chat Completions token caps before the agent command, and the open same-author PR is the correct implementation path rather than a reason to close the issue before merge.

## What This Changes

The issue asks the Gateway `/v1/chat/completions` handler to honor inbound `max_completion_tokens` and legacy `max_tokens` by forwarding them as `streamParams.maxTokens`.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/80867 so the Gateway parses both inbound token-cap fields, forwards `streamParams.maxTokens`, and documents the Chat Completions contract with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence repro path: send either token-cap field to `/v1/chat/completions` and assert the mocked `agentCommandFromIngress` options lack `streamParams.maxTokens`.

## Solution Assessment

Yes. The narrow maintainable fix is to parse `max_completion_tokens` and `max_tokens` at the Gateway boundary, prefer the current field when both are present, and thread the result through the existing agent command stream-param contract.
