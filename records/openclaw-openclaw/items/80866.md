## Summary

Keep open: source inspection confirms current main still drops inbound Chat Completions token caps before the agent command, and an open same-author fix PR is already linked as the implementation path.

## What This Changes

The issue asks `/v1/chat/completions` to accept inbound `max_completion_tokens` and legacy `max_tokens`, prefer the former, and forward the cap as `streamParams.maxTokens`.

## Best Possible Solution

Review and land or replace the linked fix with inbound parsing, focused gateway regression coverage, and public docs updates for the Chat Completions token-cap fields.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: send either token-cap field to `/v1/chat/completions` and assert the mocked `agentCommandFromIngress` options lack `streamParams.maxTokens` on current main.

## Solution Assessment

Yes. The narrow maintainable fix is to parse the two inbound Chat Completions fields at the gateway boundary and thread the resulting `streamParams` through the existing agent command contract; downstream provider wire-name translation already exists.
