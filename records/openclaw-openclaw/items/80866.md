## Summary

Keep open: current main still drops the inbound Chat Completions token-cap fields, and the linked open fix PR is the correct implementation path until it merges.

## What This Changes

The issue reports that `/v1/chat/completions` ignores inbound `max_completion_tokens` and `max_tokens` instead of forwarding them as `streamParams.maxTokens`.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/80867 with inbound parsing, focused gateway regression coverage, and public docs updates for the Chat Completions token-cap fields.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: send either token-cap field to `/v1/chat/completions` and assert the mocked `agentCommandFromIngress` options lack `streamParams.maxTokens`.

## Solution Assessment

Yes. The narrow fix is to parse `max_completion_tokens` and legacy `max_tokens` at the gateway boundary, prefer the current field, and reuse the existing `streamParams.maxTokens` downstream contract.
