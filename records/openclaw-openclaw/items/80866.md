## Summary

Keep open: current main still drops inbound Chat Completions token caps before the agent command, and the same-author linked fix PR is open as the implementation path.

## What This Changes

The issue asks for `/v1/chat/completions` to accept inbound `max_completion_tokens` and legacy `max_tokens`, prefer `max_completion_tokens`, and forward the selected value as `streamParams.maxTokens`.

## Best Possible Solution

Review and land or replace the linked fix with gateway-boundary parsing, focused regression coverage, and public docs for the Chat Completions token-cap fields.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: send either token-cap field to `/v1/chat/completions` on current main and assert the mocked `agentCommandFromIngress` options lack `streamParams.maxTokens`; I did not run a live request in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to parse the two inbound Chat Completions fields at the gateway boundary and thread the selected value through the existing `streamParams` contract; downstream provider wire-name mapping already exists.
