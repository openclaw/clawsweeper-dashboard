## Summary

Keep open: current main still does not parse or forward inbound Chat Completions token caps, and the linked same-author fix PR is open with closing syntax, so closing should wait for that PR to merge or be replaced.

## What This Changes

The issue reports that `/v1/chat/completions` should accept inbound `max_completion_tokens` and legacy `max_tokens` and forward the selected value as `streamParams.maxTokens`.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/80867 with inbound parsing, focused gateway regression coverage, and public docs updates for the Chat Completions token-cap fields.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: send either token-cap field to `/v1/chat/completions` on current main and assert the mocked `agentCommandFromIngress` options lack `streamParams.maxTokens`.

## Solution Assessment

Yes. The narrow maintainable fix is to parse the two inbound Chat Completions fields at the gateway boundary, prefer `max_completion_tokens`, and reuse the existing agent stream-param and provider transport mapping.
