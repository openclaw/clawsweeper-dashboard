## Summary

Keep open: current main still watches every configured memory `extraPaths` directory recursively with chokidar and `awaitWriteFinish`, while `sync.watch` defaults on. Related open reports cover watcher/FD pressure, but none supersede this specific large-`extraPaths` leak report.

## What This Changes

The issue requests preventing `memorySearch.sync.watch` from exhausting file descriptors on large `extraPaths` trees while preserving useful memory indexing freshness.

## Best Possible Solution

Keep the issue open and have the memory-core owner choose a coordinated watcher scalability fix that avoids long-lived FD growth on large `extraPaths` while preserving one-shot sync and documented freshness behavior.

## Reproduction Assessment

No, not as a live failure in this review. Source inspection confirms the reported large-tree watcher path on current main, and the issue includes concrete macOS lsof/ulimit evidence, but I did not observe FD growth locally.

## Solution Assessment

Unclear. Dropping `awaitWriteFinish` is the narrowest code-only mitigation, but per-path watch opt-out or tree caps would change config/policy and should be selected with owner review and macOS large-tree proof.
