## Summary

Current main and the latest release still provide runtime-dependent steering at accepted boundaries, with post-run fallback when steering is unavailable; they do not implement the requested generic pause or abort, context injection, partial-output preservation, and resume lifecycle. The issue should remain the product/runtime tracker while maintainers reconcile the two open, conflicting implementation approaches.

## What This Changes

Add a soft-steer lifecycle that interrupts an active agent operation, incorporates newly arrived user input, and continues with preserved context.

## Best Possible Solution

Define one capability-aware run-control contract that uses native same-turn steering where supported and otherwise performs an explicit safe interrupt-and-new-turn transition with stopped-state rejection, transcript preservation, tool-side-effect guarantees, and consistent fallback behavior across embedded, reply-run, and Codex backends.

## Reproduction Assessment

Yes at the source level: current main and v2026.6.6 explicitly implement boundary-based steering, non-streaming rejection, and post-run fallback, demonstrating that generic pause, inject, and resume behavior is absent. This is a feature-gap reproduction rather than a failure of an existing documented contract.

## Solution Assessment

No, neither linked approach is yet the complete best solution: unconditional abort fallback and broad active non-streaming injection make different lifecycle choices without fully defining stopped-state, tool-phase, partial-output, transcript, and cross-runtime semantics.
