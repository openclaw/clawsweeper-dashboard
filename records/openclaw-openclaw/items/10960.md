## Summary

Keep this issue open: current main supports boundary steering and a separate abort-and-replace interrupt mode, but it does not provide the requested pause-or-abort, inject, and resume lifecycle; an open maintainer-labeled PR still owns part of the implementation path.

## What This Changes

Add soft steering that interrupts an active generation or long tool phase, incorporates the newest user guidance into context, and continues the task without waiting for the current run to finish.

## Best Possible Solution

Keep one core lifecycle contract: preserve boundary steering for ordinary guidance, keep destructive interrupt explicit, and add a tested soft-interrupt operation only after maintainers define partial-output, tool-state, stopped-state, fallback, and transcript-continuation semantics.

## Reproduction Assessment

Yes. Current source and public docs consistently reproduce the limitation: steer waits for a supported model boundary, while interrupt aborts and starts the newest prompt rather than resuming the same turn with injected context.

## Solution Assessment

No, widening the existing `isStreaming` gate alone is not the best complete solution. The maintainable fix is an explicit lifecycle operation with defined transcript, tool, abort, fallback, and continuation semantics across the embedded and reply-run owners.
