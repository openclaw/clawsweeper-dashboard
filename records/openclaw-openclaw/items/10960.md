## Summary

Current main still limits steering to runtimes reported as streaming and otherwise defers the message until the active run finishes. This issue remains necessary because the requested interrupt, inject, and resume lifecycle is not implemented, while both linked implementation branches remain open and conflicting.

## What This Changes

Add a soft-steer lifecycle that interrupts active generation, injects new user input, and resumes with preserved context.

## Best Possible Solution

Refresh one canonical lifecycle-aware PR against current main, preserve a single active-run and session-state path, and prove streaming generation, non-streaming tool execution, stopped-state rejection, partial-output persistence, fallback, and resume behavior before landing.

## Reproduction Assessment

Yes, at the source level. Current main's streaming predicates, boundary-oriented harness queue, documented fallback, and regression tests clearly establish that active non-streaming runs do not receive the requested interrupt, inject, and resume lifecycle.

## Solution Assessment

The lifecycle-aware replacement direction is preferable to adding another queue mode, but neither conflicting PR is the best mergeable solution as-is. Maintainers must first settle abort, stopped-state, partial-output, transcript, and resume semantics, then refresh one canonical branch.
