Keep open: current main and the latest release still use boundary-based, streaming-gated steering rather than the requested pause-or-abort, inject, and resume soft-steer lifecycle, and the linked implementation paths are still open and review-blocked.

## What This Changes

Request true mid-stream steering so user messages can interrupt active generation, enter context immediately, and resume with the new input instead of waiting for the next runtime boundary.

## Best Possible Solution

Keep this as the broad soft-steer tracker and resolve it through a maintainer-approved lifecycle design that lands the active non-streaming and admission fixes with documented fallback, abort/resume semantics, and regression coverage.

## Reproduction Assessment

Yes, at source level: with queue mode steer, current main rejects or defers active non-streaming phases through isStreaming gates. The broader pause-or-abort/resume lifecycle is a requested feature, not an already-defined current behavior contract.

## Solution Assessment

No, current main is not the requested solution. The maintainable path is lifecycle-aware steering for active non-streaming boundaries plus an explicit maintainer decision on true stream abort/resume behavior.
