## Summary

Keep this PR open. Current main still does not recognize top-level string `detail` API error payloads, and the PR is a focused, likely-correct parser/test fix for the raw JSON leak; the remaining work is maintainer review, rebase or porting, validation, and avoiding accidental closure of the broader linked bug.

## What This Changes

The PR updates shared assistant error parsing to treat top-level string `detail` payloads as API errors, falls back to `payload.detail` as the user-facing message, and adds formatter regression tests.

## Best Possible Solution

Land a rebased or ported version of the shared parser support with exact regression coverage, and either adjust the PR body so #44910 stays open for the provider-gating follow-up or explicitly split that remaining work into a narrower canonical issue.

## Reproduction Assessment

Yes. A focused unit path is clear: pass `{"detail":"..."}` through `formatAssistantErrorText` or `formatRawAssistantErrorForUi` on current main; source inspection shows it is not classified as a raw API error and can fall through as raw JSON.

## Solution Assessment

Yes for the raw JSON leak. Recognizing top-level string `detail` in the shared parser and using it as the message fallback is the narrowest maintainable fix; the broader unconfigured-provider behavior should remain separately tracked.
