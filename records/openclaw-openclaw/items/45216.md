## Summary

Keep this PR open. Current main still does not recognize FastAPI-style top-level string `detail` error payloads, and the PR’s parser/test change is narrowly correct; the remaining repo-policy blocker is the missing user-facing changelog entry.

## What This Changes

The PR adds `detail`-field API error parsing in the shared assistant error formatter and adds exact-output regression tests for the two assistant error formatting entry points.

## Best Possible Solution

Land a rebased parser/test patch with a `CHANGELOG.md` fix bullet, while leaving provider credential-gating or silent suppression work to the broader linked bug.

## Reproduction Assessment

Yes. On current main, passing `{"detail":"..."}` through `formatRawAssistantErrorForUi` or `formatAssistantErrorText` follows an unrecognized-payload path because the shared classifier and message extractor both ignore top-level `detail`.

## Solution Assessment

Yes for the raw JSON leak. Recognizing top-level string `detail` in the shared formatter is the narrowest maintainable fix; provider availability and credential probing should stay in provider-routing follow-up work.
