## Summary

Keep open: current main still has a source-reproducible doctor/config path where a missing `plugins.deny` entry is fatal, `doctor --fix` may auto-restore from last-known-good, and safe stale-plugin cleanup can still be discarded by full-config write validation. Related work covers adjacent pieces, but no merged PR or canonical open item fully owns this remaining behavior.

## What This Changes

The issue asks for `openclaw doctor --fix` to persist safe config repairs instead of losing them when another validation error or last-known-good recovery path fires.

## Best Possible Solution

Choose the doctor repair policy first, then implement a narrow fix that treats parseable doctor-fixable stale plugin state consistently while keeping ordinary config writes fail-closed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a missing `plugins.deny` entry alongside doctor-repairable stale plugin state, then `doctor --fix` reaches last-known-good recovery and a full-config write that rejects the remaining deny validation issue before persisting safe repairs.

## Solution Assessment

Unclear. The bug is real, but maintainers should choose between tightening last-known-good recovery, treating stale `plugins.deny` like other stale plugin refs, or adding scoped doctor partial persistence; broad partial writes are riskier than a focused policy fix.
