## Summary

Keep open: current main still has a source-reproducible path where doctor builds safe stale-plugin repairs but last-known-good recovery and the final full-config write can discard or block those repairs when another validation error remains. The new related PR covers the stale `plugins.deny` slice only and is still open, so this is not implemented on main yet.

## What This Changes

Request that `doctor --fix` preserve safe config repairs instead of losing them when a config has multiple validation errors.

## Best Possible Solution

Keep normal config writes fail-closed, but teach doctor repair to skip last-known-good recovery for well-classified doctor-fixable stale config and to persist safe doctor-owned repairs without masking remaining validation errors.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: combine a stale plugin repair with either a fatal stale `plugins.deny` ref or another core validation error, then `doctor --fix` can restore from last-known-good or reach a full-config write that rejects the remaining invalid field before persisting the safe cleanup.

## Solution Assessment

No. Current main is not the best solution because it still treats `plugins.deny` differently from other stale plugin refs and only has a narrow legacy-migration partial-write exception; the open related PR is a useful partial fix but not the full doctor repair contract.
