## Summary

Keep open: current main fixed the stale `plugins.deny` restore loop and now accepts the likely Qwen `thinkingFormat` values, but the broader `doctor --fix` atomicity problem remains source-reproducible when safe doctor repairs coexist with an unrelated residual core validation error.

## What This Changes

The issue asks `doctor --fix` to preserve safe config repairs when a config has multiple validation errors instead of losing all repairs behind one unrelated residual error.

## Best Possible Solution

Make a maintainer decision on a doctor-only repair policy, then implement either a maintained taxonomy of repairable validation drift before last-known-good restore or a scoped partial-commit path for safe doctor-owned mutations without relaxing normal config writes.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: doctor can build a repaired candidate, then `runWriteConfigHealth` reaches `replaceConfigFile` and the full write validation rejects any remaining unrelated core schema error before persisting those repairs. I did not run a live Windows fixture in this read-only sweep.

## Solution Assessment

Unclear. The current main fixes important subcases, but the remaining atomicity behavior needs an explicit maintainer choice between recovery-policy taxonomy and partial repair persistence because both touch config safety semantics.
