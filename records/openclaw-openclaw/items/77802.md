## Summary

Keep open: current main fixes the stale `plugins.deny` auto-restore slice, but the broader `doctor --fix` write path still rejects the full repaired candidate before persisting doctor-owned cleanup when an unrelated hard schema error remains.

## What This Changes

Request that `doctor --fix` persist safe repairs instead of losing them when the same config also has unrelated validation errors.

## Best Possible Solution

Define a narrow doctor repair transaction contract that lets known-safe doctor repairs persist under unrelated residual validation errors while keeping ordinary config writes fail-closed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: combine a doctor-owned stale-plugin repair with any remaining hard schema error, then `doctor --fix` reaches `replaceConfigFile` and `writeConfigFile` rejects the full candidate before persisting the repair.

## Solution Assessment

Unclear. The stale-deny fix is the right narrow fix for that slice, but the remaining atomic persistence behavior needs a maintainer decision on whether doctor may write partially repaired config and which repair categories are safe.
