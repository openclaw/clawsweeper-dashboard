## Summary

Keep open: current `main` fixed the stale `plugins.deny` auto-restore slice, but the broader doctor write path can still discard safe doctor-owned repairs when an unrelated hard config validation error remains.

## What This Changes

The issue asks `openclaw doctor --fix` to preserve safe config repairs instead of losing them when other validation errors remain in the same config.

## Best Possible Solution

Define a narrow doctor repair transaction contract that lets known-safe doctor repairs persist under unrelated residual validation errors while keeping ordinary config writes fail-closed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: combine a doctor-owned stale-plugin repair with any remaining hard schema error, then `doctor --fix` reaches `replaceConfigFile` and `writeConfigFile` rejects the full candidate before persisting the repair.

## Solution Assessment

Unclear. The stale-deny fix was the right narrow fix for that slice, but the remaining atomic persistence behavior needs a maintainer decision on whether doctor may write partially repaired config and which repair categories are safe.
