## Summary

Keep open: current main fixed the stale `plugins.deny` auto-restore slice and the older legacy-migration discard bug, but the broader `doctor --fix` repair transaction problem remains when a doctor-owned cleanup is followed by an unrelated hard schema error. The `compat.thinkingFormat` slice is also still tracked separately by https://github.com/openclaw/openclaw/issues/77803 and https://github.com/openclaw/openclaw/pull/80885.

## What This Changes

Request that `openclaw doctor --fix` persist safe config repairs instead of losing them when unrelated validation errors remain.

## Best Possible Solution

Define a narrow doctor repair transaction contract that can persist only known-safe doctor repairs under unrelated residual schema errors, while landing or rejecting the separate `compat.thinkingFormat` migration path on its own merits.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: doctor can remove stale plugin references in the candidate, but any remaining hard schema error still makes `writeConfigFile` throw before that cleanup is persisted.

## Solution Assessment

Unclear. The stale-deny and legacy-migration fixes on main are good narrow slices, but the remaining partial-persistence behavior needs a maintainer decision about which doctor repairs may be written while ordinary config writes stay fail-closed.
