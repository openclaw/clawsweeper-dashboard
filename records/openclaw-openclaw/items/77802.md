## Summary

Keep open: current main fixes the stale `plugins.deny` trigger and one legacy-migration partial-valid path, but ordinary `doctor --fix` still funnels accumulated repairs through a single full-config validation-before-write path, so unrelated residual schema errors can still block persistence.

## What This Changes

The issue asks `doctor --fix` to preserve or apply successful config repairs instead of losing them when unrelated validation errors remain.

## Best Possible Solution

Keep validated writes as the default, but add a maintainer-approved doctor repair contract that can commit independently safe repairs or skip last-known-good recovery only for explicitly repairable parseable config classes, with regression coverage for residual invalid config.

## Reproduction Assessment

Yes, at source level: current main still has a single full-config validation-before-write path for ordinary doctor repairs, so a residual invalid field can block persistence of earlier repairs. I did not run a live repro because this review is read-only.

## Solution Assessment

Unclear: blindly persisting partially invalid config is not automatically the best fix because it changes the validated-write invariant. A scoped repair taxonomy or per-repair commit contract looks safer than broad incremental persistence.
