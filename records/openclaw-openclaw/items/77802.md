## Summary

Keep open: current main and v2026.6.6 fixed the concrete stale `plugins.deny` and invalid `compat.thinkingFormat` ingredients, and doctor can preserve known legacy migrations through some unrelated plugin-validation failures. A broader residual request remains because arbitrary still-invalid core config continues to block the final config write, so the next step is a maintainer product decision about whether to keep targeted migrations only or add a bounded partial-write policy.

## What This Changes

The issue reports that `openclaw doctor --fix` can lose repair progress when multiple config validation failures are present.

## Best Possible Solution

Keep strict runtime config writes, continue adding targeted doctor migrations for shipped invalid shapes, and only add broader partial-write behavior if maintainers explicitly define a bounded invariant and upgrade proof.

## Reproduction Assessment

Yes, source-level: `runWriteConfigHealth` still writes through the whole-candidate `replaceConfigFile` path, and `writeConfigFile` throws on remaining non-skipped validation errors before persisting. The exact original Windows config was not live-reproduced in this read-only review.

## Solution Assessment

Unclear for the remaining generic request: targeted doctor migrations are the best fit for the reported shipped config drifts, but a general partial-write policy would change the config write/recovery invariant and needs maintainer direction.
