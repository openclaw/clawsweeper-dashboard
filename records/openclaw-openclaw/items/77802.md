## Summary

Keep open: current `main` has fixed the concrete stale `plugins.deny` and invalid `compat.thinkingFormat` pieces, and it can commit known legacy migrations through unrelated plugin-validation failures, but it still intentionally rejects arbitrary still-invalid core config before writing. The remaining generic partial-repair behavior is a maintainer design decision, not a safe cleanup close.

## What This Changes

The issue reports that `openclaw doctor --fix` can lose otherwise successful config repairs when another validation error remains in the same config.

## Best Possible Solution

Keep the targeted repair model: add doctor migrations for shipped invalidated config shapes, and only add a broader partial-write mode if maintainers explicitly accept the safety tradeoff.

## Reproduction Assessment

Yes for the remaining broad class: source shows known migration and stale-plugin cases are repaired, while arbitrary invalid core config still blocks writes before commit. I did not run a live repro because this review is read-only.

## Solution Assessment

Unclear for the broad request: the targeted migration/repair approach is safer than generic incremental invalid writes, but maintainers need to decide whether that remaining behavior is desired.
