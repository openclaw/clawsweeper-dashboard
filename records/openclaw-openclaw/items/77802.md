## Summary

Keep open: current `main` fixed the stale `plugins.deny` trigger and related `thinkingFormat` migration class through narrower work, but the generic behavior still rejects a repaired config before persisting when any unrelated validation error remains. The remaining question is whether doctor should keep atomic config writes and rely on targeted migrations, or intentionally support a broader partial-repair write mode.

## What This Changes

The issue requests that `openclaw doctor --fix` not lose successful config repairs when the current config has multiple independent validation problems.

## Best Possible Solution

Keep targeted doctor migrations for known shipped invalid config shapes unless maintainers explicitly approve a broader partial-repair write mode with clear safety rules.

## Reproduction Assessment

Yes, source-backed for the generic residual-error path: doctor writes through `replaceConfigFile`, and `writeConfigFileLocal` validates the persisted candidate and throws before commit when any validation issue remains. The exact original stale-deny and `thinkingFormat` examples are partly obsolete because narrower fixes landed later.

## Solution Assessment

Unclear. Targeted doctor migrations for known shipped invalid config shapes match current policy; generic partial writes would change the atomic config-write invariant and need maintainer approval.
