## Summary

Keep open: current main and v2026.6.8 still route doctor repairs through a single full validation-before-write path, so unrelated residual validation errors can still block persistence. Related work fixed the stale `plugins.deny` trigger and a separate `thinkingFormat` migration class, but the generic partial-repair policy remains unresolved.

## What This Changes

The issue asks `doctor --fix` to preserve successful config repairs when other validation errors remain instead of restoring or rolling back the whole config.

## Best Possible Solution

Keep atomic config writes as the default and add targeted doctor migrations for known shipped invalid shapes unless maintainers explicitly define a bounded partial-repair mode with audit and backup guarantees.

## Reproduction Assessment

Yes. Source inspection shows `doctor --fix` accumulates repairs and then writes once through `replaceConfigFile`, while the writer validates the full persisted candidate and throws before commit if any unrelated validation issue remains.

## Solution Assessment

Unclear. Targeted doctor migrations for known shipped invalid config shapes match current OpenClaw policy, while generic partial writes would change the atomic config-write invariant and need maintainer approval.
