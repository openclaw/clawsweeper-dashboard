## Summary

Keep open: current main still has a source-reproducible path where doctor computes stale-plugin repairs but the final config write is rejected by an unrelated core validation error, so the repair cannot persist.

## What This Changes

The issue asks for `openclaw doctor --fix` to persist safe config repairs when the config has multiple independent validation errors.

## Best Possible Solution

Teach the doctor repair path to persist safe doctor-owned config repairs under unrelated validation failures while keeping ordinary config writes fail-closed.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: combine a stale plugin repair with an invalid `models.providers.*.models.*.compat.thinkingFormat`, then `doctor --fix` reaches a full-config write that rejects the remaining invalid field before persisting the stale-plugin cleanup.

## Solution Assessment

No, the current behavior is not the best way to solve the issue because a safe repair candidate is built but discarded by unrelated validation. The safer fix is scoped doctor repair persistence, not weakening general config validation.
