## Summary

Keep open: current main fixed the stale `plugins.deny` auto-restore loop, but the broader `doctor --fix` atomic write problem remains because doctor still routes repaired config through all-or-nothing validation before persisting safe cleanup when unrelated hard schema errors remain.

## What This Changes

The issue asks for `openclaw doctor --fix` to avoid last-known-good clobbering or persist safe doctor-owned repairs when multiple unrelated config validation errors exist.

## Best Possible Solution

Define a narrow doctor repair transaction contract that persists only known-safe doctor repairs under unrelated residual validation errors, while landing or rejecting the separate `thinkingFormat` migration path on its own merits.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a doctor-owned stale-plugin cleanup can update the candidate, but any remaining hard schema error still makes `writeConfigFile` throw before the cleanup is persisted.

## Solution Assessment

Unclear. The stale-deny fix on main is the right narrow fix for that slice, but the remaining atomic persistence behavior needs a maintainer decision about which partial repairs may be written while ordinary config writes stay fail-closed.
