## Summary

Keep open: the report has a concrete live v2026.5.20 transcript and current main still shows a source-level gap where browser Talk returns the original run's chat-final text, while successful message_tool_only source replies can be mirrored into transcript/history outside that result path. The linked https://github.com/openclaw/openclaw/issues/59150 and https://github.com/openclaw/openclaw/issues/79055 are useful adjacent context, but neither owns this exact voice/UI divergence.

## What This Changes

The issue reports that Control UI Talk can speak a different answer than the visible chat reply when the agent delivers the real answer via message_tool_only and delivery-mirror.

## Best Possible Solution

Make realtime Talk resolve and speak the same canonical source-reply text that Control UI displays for message_tool_only deliveries, without adding config or duplicating delivery.

## Reproduction Assessment

No live current-main repro was run, so we do not yet have a high-confidence executable reproduction. The report includes a concrete v2026.5.20 live transcript shape, and current source still shows the Talk result path does not consume delivery-mirror/sourceReply text as a canonical answer source.

## Solution Assessment

Yes directionally: the maintainable fix is to return the already displayed source-reply text to the realtime consult result, not to let the realtime model improvise around an empty or fallback chat final. The repair should preserve message_tool_only suppression and add regression coverage for the empty-final plus delivery-mirror/sourceReply case.
