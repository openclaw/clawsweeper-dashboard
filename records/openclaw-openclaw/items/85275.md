## Summary

Keep open: current main still makes browser Talk resolve the realtime consult from the `chat` final text while the canonical `message_tool_only` source reply is carried separately, so the reported voice/UI divergence remains source-backed. The linked fix attempt at https://github.com/openclaw/openclaw/pull/85386 was closed unmerged for queue policy, not because the bug was fixed.

## What This Changes

Report that Control UI Talk can speak a fallback or stale consult result while the chat shows the `message_tool_only` / `delivery-mirror` source reply for the same agent run.

## Best Possible Solution

Make browser Talk consults resolve from the same canonical same-run source-reply text that Control UI displays, falling back to final chat text only when no speakable source reply exists, with focused regression coverage.

Yes, source-level. Current main still waits on `chat` final text in browser Talk while the canonical `message_tool_only` source reply is returned through a separate tool/source-reply path, matching the reporter's live transcript shape.

## Solution Assessment

Yes. The narrow maintainable fix is to use the same canonical source-reply text for the realtime consult result, not to change provider prompts or broader Talk parity policy.
