## Label Justifications

- `P1`: A live Talk-mode agent workflow can give the user a spoken answer that conflicts with the visible final answer on the latest release.
- `impact:message-loss`: The issue is about the voice delivery path speaking the wrong or fallback answer instead of the delivered final message.
- `impact:session-state`: The reported divergence comes from Talk and Control UI observing different final-answer state for the same agent run.

## Summary

Keep open: the report matches the current Talk/Gateway source path. Control UI Talk waits for the `chat` final event text, while `message_tool_only` can move the canonical user-visible reply into a source-reply mirror, so the spoken tool result can diverge from the answer displayed in the chat UI.

## What This Changes

Talk mode should speak the same canonical final answer displayed in Control UI when a delegated agent reply is delivered via `message_tool_only` and `delivery-mirror`.

## Best Possible Solution

Make the Talk consult result resolve from the canonical displayed assistant/source-reply text for the same run before submitting the realtime function-call output, with regression coverage for `message_tool_only` plus `delivery-mirror`.

## Reproduction Assessment

Yes, source-level. The issue provides a live v2026.5.20 repro, and current main still has the same Talk consult path that waits on `chat` final text while source-reply mirrors can carry the displayed answer separately.

## Solution Assessment

Yes, the best fix is to unify Talk's spoken tool result with the same canonical final answer the Control UI displays. A prompt-only or provider-only tweak would not reliably fix the split source of truth.
