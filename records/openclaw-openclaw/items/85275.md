## Label Justifications

- `P1`: A live Talk-mode workflow can speak an answer that conflicts with the visible final answer on the current release and current main.
- `impact:message-loss`: The voice delivery path can speak a fallback or stale response instead of the delivered final message.
- `impact:session-state`: Talk audio and Control UI chat can observe different final-answer state for the same agent run.

## Summary

Keep open: current main still has the Talk consult split described by the report, so browser Talk can resolve speech from the chat final text while the visible Control UI answer comes from a separate message-tool-only source reply path.

## What This Changes

Talk mode should speak the same final answer displayed in Control UI when an agent reply is delivered through message_tool_only/delivery-mirror.

## Best Possible Solution

Make the Talk consult result prefer the same-run canonical internal sourceReply text when present, then fall back to chat final text, with regression coverage for message_tool_only plus delivery-mirror.

## Reproduction Assessment

Yes, source-level. The issue provides a live v2026.5.20 repro, and current main still resolves browser Talk consult output from chat final text while the displayed sourceReply travels through a separate message_tool_only path.

## Solution Assessment

Yes. The maintainable fix is to make Talk use the same canonical displayed sourceReply for the same run before falling back to chat final text; provider prompt tuning would not remove the split source of truth.
