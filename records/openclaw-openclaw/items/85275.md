## Label Justifications

- `P1`: A live Talk-mode agent workflow can give the user a spoken answer that conflicts with the visible final answer on the latest released path.
- `impact:message-loss`: The voice delivery path can speak a fallback or stale answer instead of the delivered final message.
- `impact:session-state`: The report describes Talk and Control UI observing different final-answer state for the same agent run.

## Summary

Keep open: current main still has the split source path described in the report, and the only linked fix PR was closed unmerged. The Talk consult still resolves the spoken result from the final `chat` event text while `message_tool_only` can put the canonical visible reply in a separate internal source-reply mirror.

## What This Changes

Talk mode should speak the same canonical final answer that the Control UI displays when an agent reply is delivered through `message_tool_only` and `delivery-mirror`.

## Best Possible Solution

Make the Talk consult resolve from the same-run canonical internal source reply before falling back to final chat text, and cover the `message_tool_only` plus stale/fallback final-chat case with a focused UI regression test.

## Reproduction Assessment

Yes, source-level: a same-run `agent` tool event can carry the internal `sourceReply` while the later final `chat` event carries stale/fallback text, and current main only resolves the `chat` final text. The issue also includes a live v2026.5.20 repro transcript shape matching that path.

## Solution Assessment

Yes, the narrow maintainable fix is to unify Talk's spoken consult result with the canonical source-reply text already used by the visible Control UI. A prompt or provider-only change would leave the split source of truth intact.
