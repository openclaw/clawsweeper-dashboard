- `impact:message-loss`: The issue is about the voice delivery path speaking a fallback or wrong answer instead of the delivered final message.

## Summary

Keep open: current main still resolves realtime Talk consult speech from the `chat` final event text, while the displayed answer can be carried separately through the internal source-reply/message-tool mirror path. The closed, unmerged reference PR confirms a narrow fix shape but does not make the issue implemented on main.

## What This Changes

Talk mode should speak the same canonical final answer that the Control UI displays for `message_tool_only` / `delivery-mirror` replies.

## Best Possible Solution

Resolve the realtime Talk consult result from the same-run canonical internal source reply before falling back to chat final text, with regression coverage for stale/fallback chat text plus `message_tool_only` delivery.

## Reproduction Assessment

Yes, source-level. The issue includes a live v2026.5.20 repro, and current main still has the same split where Talk waits for `chat` final text while the visible source reply can arrive through the `agent`/message-tool mirror path.

## Solution Assessment

Yes. The narrow maintainable fix is to make the Talk consult use the same canonical displayed source-reply text for the matching run before submitting the realtime tool result; provider prompt tweaks would leave two sources of truth.
