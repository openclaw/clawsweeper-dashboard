## Label Justifications

- `P1`: A live Talk-mode workflow can speak an answer that contradicts the successful final answer visible in the Control UI.
- `impact:message-loss`: The voice delivery path can speak a fallback or stale answer instead of the delivered final message.
- `impact:session-state`: The bug comes from Talk and Control UI reading different final-answer state for the same agent run.

## Summary

Keep open: current main still resolves the browser Talk consult from the `chat` final message only, while the canonical visible answer for `message_tool_only` can live in the message-tool `sourceReply`/delivery mirror path. The related fix PR was closed unmerged, and the current Talk stabilization commit does not add source-reply preference to the shared browser consult path.

## What This Changes

The issue requests Talk mode to speak the same canonical final answer shown in Control UI when the agent delivers the reply through `message_tool_only` / `delivery-mirror`.

## Best Possible Solution

Make the browser Talk consult resolve from the canonical same-run internal `sourceReply.text` when message-tool-only delivery mirrors the visible answer, then fall back to final chat text only when no source reply was delivered.

## Reproduction Assessment

Yes, source-level. Current main still waits for only the `chat` final message, while message-tool-only internal UI delivery can carry the canonical answer in `sourceReply.text`, matching the live release repro described in the issue.

## Solution Assessment

Yes. The maintainable fix is to make Talk use the same canonical displayed source-reply text before submitting the realtime tool result, rather than relying on provider prompts or unrelated realtime routing behavior.
