## Summary

Keep open: current main still resolves browser Control UI Talk consult speech from the `chat` final event text while ignoring same-run `agent` tool/source-reply payloads. The closed unmerged fix PR shows a narrow repair shape, but it did not land, and the merged Talk stabilization work does not cover this browser-owned `message_tool_only` source-reply split.

## What This Changes

Talk mode should speak the same canonical final answer that the Control UI displays when an agent reply is delivered through `message_tool_only` and mirrored as `delivery-mirror`.

## Best Possible Solution

Make the browser Talk consult submit the canonical same-run displayed source-reply text when `message_tool_only`/`internal-ui` data is present, with a regression test proving it does not speak stale or fallback chat-final text.

## Reproduction Assessment

Yes, source-level. The issue includes a live v2026.5.20 repro, and current main still has a browser Talk consult path that ignores same-run agent/source-reply events while the source-reply mirror can carry the displayed answer.

## Solution Assessment

Yes. The narrow maintainable fix is to unify Talk's spoken tool result with the same canonical source-reply text the Control UI displays, not to tune prompts or provider behavior around two sources of truth.
