## Summary

Keep open for maintainer review and landing: current main still drops QQ Bot `msg_elements`-only media before inbound processing, while this PR implements the narrow dispatcher fallback with regression tests and sufficient terminal proof. I found no blocking functional or security findings.

## What This Changes

This PR teaches the QQ Bot dispatcher to use non-quote C2C/group `msg_elements` media as inbound attachments when top-level attachments are absent, adds dispatcher regression tests, and records a changelog fix.

## Best Possible Solution

Land the QQ Bot dispatcher normalization after maintainer review, keeping current-message media and quote metadata separated while leaving downstream attachment processing unchanged.

## Reproduction Assessment

Yes. A source-level current-main repro dispatches a `C2C_MESSAGE_CREATE` payload with media only under `msg_elements`; the dispatcher queues `attachments` from `ev.attachments`, and the inbound pipeline only processes queued `event.attachments`.

## Solution Assessment

Yes. Normalizing in the QQ Bot dispatcher is the narrowest maintainable fix because the existing inbound media pipeline already handles attachments once queued, and quote `msg_elements` remain on the quote path.
