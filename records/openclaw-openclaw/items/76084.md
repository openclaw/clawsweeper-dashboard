## Summary

Keep open. Current main still forwards only top-level QQ Bot `attachments` into queued C2C/group messages, while the linked report and existing quote handling show `msg_elements` is an active payload path; this PR is a narrow owner-plugin fix with focused regression coverage and no blocking review finding from this read-only pass.

## What This Changes

The PR normalizes QQ Bot C2C and group `msg_elements` media into inbound queued attachments when top-level attachments are absent, adds dispatcher tests, and records the user-facing fix in the changelog.

## Best Possible Solution

Land a narrow QQ Bot-owned dispatcher fix that surfaces current-message `msg_elements` media as queued inbound attachments while keeping quote `msg_elements` reserved for reply metadata.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to dispatch a C2C QQ event with image metadata in `msg_elements` and no top-level `attachments`; current main produces a queued message without attachments, so the inbound pipeline calls `processAttachments(undefined)`.

## Solution Assessment

Yes. The dispatcher is the narrowest maintainable fix boundary because downstream attachment processing already works once queued messages carry attachment metadata, and the PR keeps the behavior inside the QQ Bot plugin.
