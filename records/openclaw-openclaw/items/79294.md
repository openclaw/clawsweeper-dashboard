## Summary

Keep open: current main and the latest stable release still intentionally prefer Homebrew for gateway-backed skill dependency installers when `skills.install.preferBrew` is true and `brew` is available. The report raises a plausible container/root setup concern, but changing Homebrew from a preferred installer to a fallback is a shipped installer-policy decision, not a confirmed narrow regression from the supplied data.

## What This Changes

Request to make Homebrew a fallback rather than a default/preferred installer for OpenClaw skill or plugin setup, especially in root/container environments.

## Best Possible Solution

Have maintainers decide the desired skill dependency installer policy, then either make the default root-aware/non-brew-first with docs and focused tests, or keep Homebrew preferred and document the container-specific setup path.

## Reproduction Assessment

No, not for the claimed failure. Source inspection gives a clear path where current main prefers brew when enabled and available, but the issue lacks the command, image, skill id, and logs needed to reproduce a root/container install failure.

## Solution Assessment

Unclear as written. Making Homebrew a universal fallback may be valid, but it changes shipped installer policy; a narrower maintainable direction would be a maintainer-approved root-aware selection rule plus docs/tests for container setups.
