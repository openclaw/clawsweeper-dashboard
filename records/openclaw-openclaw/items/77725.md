## Summary

Keep open: the PR is addressing a real prompt-context loss path, but the proposed shared helper still appends runtime context when `promptMode` is `none`, which violates the current prompt-mode contract and should block merge until repaired.

## What This Changes

The PR appends runtime `extraSystemPrompt` to configured `systemPromptOverride` prompts across CLI, embedded attempt, compaction, docs, config comments, changelog, and regression tests.

## Best Possible Solution

Repair the helper to return the base override for `promptMode: "none"`, preserve the current-main embedded attempt behavior, and land only the still-needed CLI/compaction prompt-context coverage against main.

## Reproduction Assessment

Yes. Source inspection shows the current CLI and compaction override branches still bypass the prompt builders that receive `extraSystemPrompt`, and the contributor supplied after-fix live output for the CLI preparation path.

## Solution Assessment

No as written. Appending runtime context to override prompts is the right narrow direction for affected non-`none` paths, but the shared helper must preserve `promptMode: "none"` before this patch is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve promptMode none in the append helper:** `src/agents/system-prompt.ts:54-58`
  - body: `PromptMode` includes `"none"`, and the normal prompt builder returns only the basic identity prompt before any `extraSystemPrompt` append in that mode. This helper treats `none` like a full prompt and appends `## Group Chat Context`, so compaction or embedded callers that pass `promptMode` through can reintroduce runtime context that the existing contract strips. Return `basePrompt` when `params.promptMode === "none"` and add focused coverage for that case.
  - confidence: 0.88
