## Summary

Keep open. Current main has restored duplicate heartbeat suppression, but the plugin-channel fallback remains incomplete: heartbeat routing still re-enters strict outbound plugin resolution and can return `no-target` for a trusted stored plugin route when the active registry is empty. The related #46203 is useful source context, but it is closed and unmerged.

## What This Changes

The issue asks to complete heartbeat delivery fallback for stored plugin-channel routes and restore 24-hour duplicate heartbeat suppression.

## Best Possible Solution

Keep the restored duplicate suppression and add a scoped heartbeat fallback that preserves a trusted stored plugin route for `target: "last"` when strict plugin lookup is unavailable, with real registry-backed regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: store an external plugin channel route in session delivery context, leave the active plugin registry empty, and resolve heartbeat delivery with `target: "last"`; the stored route is recovered and then dropped by strict outbound resolution.

## Solution Assessment

Yes. The narrow maintainable repair is in heartbeat/outbound target resolution, preserving existing account, direct-message, thread, and explicit-target checks; #46203 has useful context, but its unrelated generated/config-context changes should not be carried forward blindly.
