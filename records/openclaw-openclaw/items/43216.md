## Summary

Keep open because the duplicate-suppression half is restored, but the plugin-channel heartbeat fallback still appears incomplete on current main: a recovered stored plugin route is still passed through strict outbound plugin lookup and can degrade to `no-target` when the registry is empty.

## What This Changes

Fix heartbeat delivery so stored plugin-channel routes work before plugin registry bootstrap and duplicate heartbeat payloads are suppressed for 24 hours.

## Best Possible Solution

Keep the restored duplicate suppression and add a narrow heartbeat fallback that preserves trusted stored plugin route metadata for `target: "last"` when plugin lookup is unavailable.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: store a plugin-channel route in session delivery context, leave the active plugin registry empty, and resolve heartbeat delivery with `target: "last"`; current source still revalidates that recovered route through strict outbound plugin lookup and can return `no-target`.

## Solution Assessment

Yes. The best fix is a narrow change in `resolveHeartbeatDeliveryTarget()` or its immediate outbound resolution seam that falls back only for trusted stored route data when plugin lookup is unavailable, while keeping normal validation when a plugin is loaded.
