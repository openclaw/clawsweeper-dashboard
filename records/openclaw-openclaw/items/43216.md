## Summary

Keep open: duplicate heartbeat suppression is implemented on current main, but the plugin-channel heartbeat fallback gap still appears valid because heartbeat resolution still converts a failed outbound plugin lookup into `no-target` instead of falling back to the stored last route.

## What This Changes

The issue requests end-to-end heartbeat delivery fallback for stored plugin-channel routes and restoration of 24-hour duplicate heartbeat suppression.

## Best Possible Solution

Keep the restored duplicate suppression and add a narrow heartbeat resolver fallback that preserves stored plugin-channel route, accountId, and thread behavior when primary outbound plugin resolution is unavailable.

## Reproduction Assessment

Yes, from source inspection rather than a live run: an external plugin channel stored in the session can be recovered by `resolveSessionDeliveryTarget()`, then converted back to `no-target` when `resolveHeartbeatDeliveryTarget()` calls outbound resolution without an available plugin. A focused test should mock an empty plugin registry and assert the heartbeat target keeps the stored route.

## Solution Assessment

Yes, the issue's suggested direction is the narrow maintainable fix: keep dedupe as-is, then make the heartbeat target resolver's failure path fall back through the already-resolved session route without adding new config or core plugin policy.
