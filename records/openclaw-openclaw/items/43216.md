## Summary

Keep open. Duplicate heartbeat suppression is restored on current main, but heartbeat `target: "last"` still resolves stored plugin-channel metadata and then calls outbound target resolution without plugin-bootstrap opt-in; the related implementation PR is closed unmerged.

## What This Changes

The issue asks to complete heartbeat delivery fallback for stored plugin-channel routes and restore duplicate heartbeat suppression within the 24-hour dedupe window.

## Best Possible Solution

Add a narrow heartbeat `target: "last"` fallback or bootstrap opt-in at the outbound-resolution boundary, preserving stored account/thread metadata and leaving the restored duplicate-suppression path unchanged.

## Reproduction Assessment

Yes, at source level. With a stored plugin-channel route and an unloaded external channel plugin, `resolveHeartbeatDeliveryTarget()` can recover the session route and then fail in `resolveOutboundTarget()` because that call does not opt into bootstrap.

## Solution Assessment

Yes for the remaining fallback. The maintainable fix is bounded to heartbeat target resolution and regression coverage, not a global plugin bootstrap policy or another duplicate-suppression rewrite.
