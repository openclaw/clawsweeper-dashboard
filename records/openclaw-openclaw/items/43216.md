## Summary

Keep open. Duplicate heartbeat suppression is already present on current main, but the plugin-channel heartbeat fallback still has a source-reproducible gap when a stored plugin route is recovered before strict outbound plugin lookup can resolve the channel.

## What This Changes

Ensure heartbeat `target: "last"` preserves stored plugin-channel routes before plugin registry bootstrap and suppresses duplicate heartbeat payloads inside 24 hours.

## Best Possible Solution

Add a narrow heartbeat fallback that preserves a trusted stored plugin route for `target: "last"` when strict plugin lookup is unavailable, while keeping duplicate suppression and focused registry-empty regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: store a plugin-channel route in session delivery context or `lastChannel`/`lastTo`, leave the active plugin registry empty/unbootstrapped, and resolve heartbeat delivery with `target: "last"`; current source can still convert the recovered route to `no-target`.

## Solution Assessment

Yes. The narrow maintainable repair belongs in `resolveHeartbeatDeliveryTarget()` or its immediate outbound target seam, preserving validation where available and falling back only for trusted stored route data when plugin lookup is unavailable.
