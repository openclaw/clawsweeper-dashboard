## Summary

Keep open. Current main and the latest release have duplicate heartbeat suppression again, but heartbeat target resolution still returns `no-target` after strict plugin resolution fails, and the only linked implementation PR closed unmerged with an accountId concern.

## What This Changes

The issue requests end-to-end heartbeat delivery fallback for stored plugin-channel routes and duplicate heartbeat suppression within 24 hours.

## Best Possible Solution

Add a narrow heartbeat target fallback that uses trusted stored plugin route data for `target: "last"` only after strict lookup fails, while keeping the restored duplicate suppression path unchanged.

Yes. Source inspection gives a high-confidence path: store a plugin-channel route in session delivery context or `lastChannel`/`lastTo`, keep the active plugin registry empty, resolve heartbeat target `last`, and the recovered route can still become `no-target` after strict outbound lookup fails.

## Solution Assessment

Yes for the remaining fallback. The maintainable fix is bounded to heartbeat target resolution and regression tests, not a startup migration or broad plugin-loader change.
