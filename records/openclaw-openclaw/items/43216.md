## Summary

One reported gap remains actionable: current main restores duplicate suppression, but plugin-channel heartbeat delivery still converts a session-recovered route to `no-target` when the plugin registry is unavailable. The only linked PR is closed unmerged, includes substantial unrelated files, and its review identified an account-selection defect, so it is not a safe canonical implementation path.

## What This Changes

Restore heartbeat delivery from stored plugin-channel session routes before plugin registry bootstrap and suppress identical heartbeat payloads for 24 hours.

## Best Possible Solution

Keep normal plugin normalization when available, but allow the canonical heartbeat resolver to use the complete stored session route—channel, target, and account—when registry lookup is unavailable, with focused empty-registry and account-selection regression tests.

## Reproduction Assessment

Yes, from source with high confidence: an unavailable plugin makes `resolveOutboundTarget()` fail after the session route was recovered, and the heartbeat resolver converts that failure to `no-target`. A focused empty-registry test would turn this into an executed reproduction.

## Solution Assessment

Yes, with one refinement: repair the canonical heartbeat resolver rather than reviving the prior PR, and carry the stored account ID with the fallback while preserving plugin normalization for normal and explicit routes.
