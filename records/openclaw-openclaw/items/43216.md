## Summary

Keep open. Duplicate heartbeat suppression is restored on current main, but the plugin-channel heartbeat fallback is still incomplete: recovered stored plugin routes are revalidated through strict outbound plugin lookup and still fall back to `no-target` when the plugin registry is empty.

## What This Changes

The issue asks to complete heartbeat delivery fallback for stored plugin-channel routes and restore duplicate heartbeat suppression within the 24-hour window.

## Best Possible Solution

Keep the restored duplicate suppression and add a narrow heartbeat fallback that preserves a trusted stored plugin route for `target: "last"` when plugin lookup is unavailable, with focused registry-empty regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: store a plugin-channel route in session delivery context, leave the active plugin registry empty, and resolve heartbeat delivery with `target: "last"`; source inspection shows strict outbound resolution can still convert that recovered route to `no-target`.

## Solution Assessment

Yes. The narrow maintainable repair belongs in `resolveHeartbeatDeliveryTarget()` or its immediate outbound target seam, preserving validation where available and only falling back for trusted stored route data when plugin lookup is unavailable.
