## Summary

Keep open. Current main has restored duplicate heartbeat suppression, but the plugin-channel heartbeat fallback still has a source-reproducible gap: `target: "last"` can recover stored plugin route metadata and then fail the later outbound resolution without plugin bootstrap.

## What This Changes

The issue asks OpenClaw to complete heartbeat delivery fallback for stored plugin-channel routes before plugin bootstrap and enforce 24-hour duplicate heartbeat suppression.

## Best Possible Solution

Add a narrow heartbeat `target: "last"` fallback or bootstrap opt-in at the outbound-resolution boundary, preserving stored account/thread metadata and leaving the restored duplicate-suppression path unchanged.

## Reproduction Assessment

Yes, at source level. In current main, `resolveHeartbeatDeliveryTarget()` can recover stored plugin route metadata and then call `resolveOutboundTarget()` without `allowBootstrap`, while unloaded external plugins require explicit bootstrap opt-in.

## Solution Assessment

Yes for the remaining fallback. The maintainable fix is bounded to heartbeat target resolution and regression coverage, not a global plugin bootstrap policy or another duplicate-suppression rewrite.

Fix the remaining heartbeat plugin-channel fallback reported at https://github.com/openclaw/openclaw/issues/43216.
