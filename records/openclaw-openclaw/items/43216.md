## Summary

Keep open. Current main has duplicate heartbeat suppression restored, but heartbeat `target: "last"` can still recover a stored plugin route and then fail closed because the final outbound resolution does not opt into plugin bootstrap; the related implementation PR is closed unmerged.

## What This Changes

The issue reports two heartbeat gaps: plugin-channel fallback before plugin bootstrap and duplicate heartbeat suppression within the 24-hour window.

## Best Possible Solution

Add a narrow heartbeat `target: "last"` fallback or bootstrap opt-in at the outbound-resolution boundary that preserves stored account/thread metadata and keeps the existing duplicate-suppression behavior unchanged.

## Reproduction Assessment

Yes, at source level. With a stored plugin-channel route and an unloaded external channel plugin, `resolveHeartbeatDeliveryTarget()` can recover the session route and then fail in `resolveOutboundTarget()` because that call does not opt into bootstrap.

## Solution Assessment

Yes for the remaining fallback. The maintainable fix is bounded to heartbeat target resolution and regression coverage, not a global plugin bootstrap policy or another duplicate-suppression rewrite.
