## Summary

Keep open. Current main has duplicate heartbeat suppression again, but the plugin-channel heartbeat fallback still fails when a stored plugin route reaches heartbeat resolution before strict outbound plugin lookup can resolve that channel; the only linked fix PR was closed unmerged after an account-id fallback concern.

## What This Changes

The issue asks for end-to-end heartbeat delivery to stored plugin-channel routes before plugin registry bootstrap, plus duplicate-heartbeat suppression and regression coverage.

## Best Possible Solution

Add a narrow heartbeat fallback that preserves a trusted stored plugin route for target last only when strict plugin lookup is unavailable, while preserving accountId/thread semantics and the existing duplicate suppression path.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: store a plugin-channel route in session delivery context or lastChannel/lastTo, leave the active plugin registry empty or unbootstrapped, and resolve heartbeat delivery with target last; current code can convert the recovered route to no-target after strict outbound resolution fails.

## Solution Assessment

Yes. The maintainable repair is a small fallback in resolveHeartbeatDeliveryTarget() or its immediate outbound seam, using only trusted stored route data after strict plugin lookup fails and preserving accountId and thread behavior.
