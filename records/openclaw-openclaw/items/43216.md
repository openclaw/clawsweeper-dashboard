## Summary

Keep open. Current main has restored duplicate heartbeat suppression, but the plugin-channel heartbeat fallback gap still appears source-reproducible because target="last" can recover a stored plugin route and then drop it when the stricter outbound resolver cannot see an unbootstrapped channel registry.

## What This Changes

The issue requests fixing heartbeat delivery to stored plugin-channel routes before registry bootstrap and preserving 24-hour duplicate heartbeat suppression.

## Best Possible Solution

Keep the restored duplicate suppression and add a scoped heartbeat target="last" fallback that preserves trusted stored plugin routes when strict registry-backed resolution is temporarily unavailable, with real empty-registry regression coverage.

## Reproduction Assessment

Yes, by source inspection. Store an external plugin route in the session delivery context, leave the active channel registry empty, and resolve heartbeat delivery with target="last"; session recovery keeps the route, but strict outbound resolution can still convert it to no-target.

## Solution Assessment

Yes. The narrow maintainable fix belongs in heartbeat/outbound target resolution while preserving account checks, direct-message policy, plugin target parsing, and delivery readiness; #46203 is useful context but should not be copied wholesale with unrelated generated/config changes.
