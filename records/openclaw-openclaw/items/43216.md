## Summary

Keep open. Current main has restored the duplicate heartbeat suppression half, but the plugin-channel fallback remains incomplete because heartbeat routing still drops a stored external plugin route when strict outbound plugin resolution cannot see an unbootstrapped registry. The related implementation PR #46203 is useful context, but it is closed and unmerged, so it does not supersede the remaining bug.

## What This Changes

The issue requests end-to-end heartbeat delivery fallback for stored plugin-channel routes and 24-hour duplicate heartbeat text suppression.

## Best Possible Solution

Teach heartbeat target resolution to retain or re-bootstrap a trusted stored plugin route for `target: "last"` when strict outbound resolution fails only because the plugin registry is empty or unbootstrapped, while leaving explicit target validation, account checks, direct-message policy, plugin thread handling, and the restored dedupe logic intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: store an external plugin channel such as `kakaotalk` in the session delivery context, leave the active plugin registry empty, and resolve heartbeat delivery with `target: "last"`; the session route can be recovered, then strict outbound resolution rejects it and returns `no-target`.

## Solution Assessment

Yes. The best repair is a narrow heartbeat/outbound fallback with real registry-backed regression coverage, not a broader channel normalization bypass or ClawHub/plugin move.
