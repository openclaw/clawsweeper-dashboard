## Summary

Keep open. Current main still lacks the requested TUI status-bar behavior, but this PR should not merge as-is because the branch is dirty against main and the added timer/error rendering logic has concrete correctness regressions around plain-error elapsed display and slow-start optimistic run binding.

## What This Changes

The PR threads summarized TUI agent error details into activity status rendering, preserves error elapsed time, extends elapsed formatting, and also adds stale-busy/gap-recovery logging plus a root AGENTS.md code-quality note.

## Best Possible Solution

Keep this PR open for a narrowed rework or replacement: rebase onto current main, remove the root AGENTS.md guidance and stale-busy/gap-recovery behavior unless tracked separately, thread a single-line sanitized error summary into final-error, chat-error, lifecycle-error, and send/auth failure status paths, preserve elapsed time for both plain and detailed error statuses, and add focused TUI event/status tests.

## Reproduction Assessment

Yes. The current-main source path is enough to reproduce the reported behavior: an active chat error with evt.errorMessage logs detail to chatLog, but terminateRun/finalizeRun still send only plain "error" to the status bar and renderStatus clears elapsed state.

## Solution Assessment

No, not as-is. The requested direction is sound, but the current PR branch mixes the narrow status-bar fix with broader run-lifecycle recovery logic and still loses elapsed time for plain error statuses; the safer fix is a focused status/error plumbing change with targeted tests.
