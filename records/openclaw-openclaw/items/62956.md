## Summary

Keep open. Current main still has the reported TUI status-bar gap, but this PR is not ready to merge because the branch is dirty against main and the patch introduces regressions around plain-error elapsed rendering and stale-wait run binding.

## What This Changes

The PR threads summarized TUI agent failure details into activity-status rendering, preserves elapsed display for error states, extends elapsed formatting, and also adds stale-busy/gap-recovery logging plus root AGENTS guidance.

## Best Possible Solution

Land a narrowed rebase or replacement that only changes TUI status/error plumbing: keep current-main chat-log formatting and auth hints, add a single-line sanitized status summary for final/chat/lifecycle/send/auth failures, preserve elapsed time for both plain and detailed error labels, add focused TUI tests, and add the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is enough: a chat error with evt.errorMessage reaches chatLog.addSystem, but terminateRun/finalizeRun pass only "error" to setActivityStatus, and renderStatus clears elapsed state for all non-busy statuses.

## Solution Assessment

No, not as-is. The requested direction is sound, but this patch mixes the narrow status-bar fix with broader stale-run recovery and still has concrete status/optimistic-run regressions; a focused TUI status patch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve elapsed time for plain error statuses:** `src/tui/tui.ts:676-678`
  - body: This branch treats a previous plain `error` as a non-error because it only preserves elapsed when the previous label starts with `error:`. Any rerender while `activityStatus` is still `error` clears `errorElapsed`, so send/auth/lifecycle failures that use `setActivityStatus("error")` lose the elapsed suffix this PR is meant to preserve.
  - confidence: 0.89
- **[P2] Keep optimistic binding until a run is claimed:** `src/tui/tui.ts:581-582`
  - body: When the stale timeout fires during `waiting` before the first gateway event, `activeChatRunId` is still null and `pendingOptimisticUserMessage` is the only state that lets the first event be marked local. Clearing it here can misclassify the local turn as external and trigger avoidable history reloads that replace optimistic in-flight output.
  - confidence: 0.84
