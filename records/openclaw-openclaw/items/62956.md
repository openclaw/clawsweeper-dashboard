## Summary

Keep open. Current main still has the reported TUI status-bar gap, but this PR cannot merge as-is because it lacks real behavior proof and introduces concrete correctness and security regressions around raw error handling, elapsed rendering, stale-run timeout, and gap recovery.

## What This Changes

The PR threads agent failure text into the TUI status bar, preserves elapsed time after failures, adds stale busy/gap recovery diagnostics, and edits root AGENTS guidance.

## Best Possible Solution

Land a focused TUI status-bar change that reuses existing sanitized error formatting, passes concise error detail into the status bar, preserves a stable elapsed suffix, and leaves stale-run recovery plus AGENTS policy changes out of this patch.

## Reproduction Assessment

Yes for the current behavior, by source inspection: current main receives error details for chat rendering but passes only plain error to activity status, and renderStatus clears elapsed for non-busy errors. I did not run a live provider failure in this read-only review.

## Solution Assessment

No, not as proposed. The status-bar direction is useful, but this branch overreaches into stale-run recovery, gap recovery, root policy text, and raw debug logging; a narrower status-only patch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Remove raw provider-error debug logging:** `src/tui/tui-event-handlers.ts:382-384`
  - body: This logs `evt.errorMessage` directly through `logDebug`, and `logDebug` always writes to the file logger before any console gating. Provider errors can include request IDs, account details, or raw JSON, so use the sanitized summary or remove this diagnostic.
  - confidence: 0.82
- **[P2] Preserve existing chat error formatting:** `src/tui/tui-event-handlers.ts:387`
  - body: This replaces current main's formatted chat error and auth-error hint with the raw provider string, even though the PR says chat rendering is out of scope. That regresses noisy raw JSON/embedded auth errors in the chat log; keep `formatRawAssistantErrorForUi` and the auth hint for chat display.
  - confidence: 0.86
- **[P2] Preserve elapsed for plain error rerenders:** `src/tui/tui.ts:676-678`
  - body: A repeated render while `activityStatus === "error"` reaches this branch with `lastActivityStatus === "error"`; because that does not start with `error:`, `errorElapsed` is cleared and the elapsed suffix disappears.
  - confidence: 0.89
- **[P2] Keep optimistic binding until a run is claimed:** `src/tui/tui.ts:581-582`
  - body: When the stale timeout fires during `waiting` before the first chat event, `activeChatRunId` can still be null and `pendingOptimisticUserMessage` is the only local-run binding. Clearing it misclassifies the eventual run as external and can trigger avoidable history reloads.
  - confidence: 0.84
- **[P2] Avoid reloading history for an active gap run:** `src/tui/tui.ts:1047-1050`
  - body: This reloads history while the same run is still active after `refreshSessionInfo()`. `loadHistory()` clears and rebuilds the chat log, so a sparse but live run can lose in-flight output without terminal completion evidence.
  - confidence: 0.82
