Full review comments:

- **[P2] Remove raw provider-error debug logging:** `src/tui/tui-event-handlers.ts:382-384`
  - body: `logDebug` writes to the file logger, and this new diagnostic records the full `evt.errorMessage`. Provider failures can include request IDs, account details, or raw JSON, so remove this log or emit only the sanitized short summary needed for status.
  - confidence: 0.86
- **[P2] Preserve sanitized chat error rendering:** `src/tui/tui-event-handlers.ts:387`
  - body: This head writes the raw provider error string to chat instead of current main's `formatRawAssistantErrorForUi` plus local auth hint path. Keep the existing formatted chat display and use the new concise summary only for the status bar.
  - confidence: 0.88
- **[P2] Preserve elapsed for plain error rerenders:** `src/tui/tui.ts:676-678`
  - body: When the status remains plain `error`, `lastActivityStatus` is also plain `error`, which does not start with `error:`; a connection-status rerender then clears `errorElapsed` and drops the suffix. Treat plain `error` as an error state when deciding whether to keep the cached elapsed value.
  - confidence: 0.89
- **[P2] Keep optimistic binding until a run is claimed:** `src/tui/tui.ts:582`
  - body: If the stale timeout fires during `waiting` before the first chat event, `activeChatRunId` can still be null and `pendingOptimisticUserMessage` is the only local-run binding. Clearing it here can make the eventual run look external and trigger avoidable history refresh behavior.
  - confidence: 0.82
- **[P2] Avoid reloading history while the run is still active:** `src/tui/tui.ts:1047-1050`
  - body: This branch calls `loadHistory()` while `state.activeChatRunId` is still the same run. `loadHistory()` clears and rebuilds the chat log, so a sparse but still-live run can lose in-flight output without terminal completion evidence.
  - confidence: 0.84
