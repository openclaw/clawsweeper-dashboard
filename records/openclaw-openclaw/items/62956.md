## What This Changes

The PR threads summarized TUI agent failure messages into activity-status rendering, preserves elapsed display for error states, extends elapsed formatting, adds stale busy/gap recovery logging, and adds root AGENTS.md guidance.

## Best Possible Solution

Land a focused TUI status fix that preserves current-main chat-log formatting, auth hints, streaming watchdog behavior, and session-gap handling while adding a single-line sanitized error summary and stable elapsed display for final, chat, lifecycle, send, and auth failure statuses.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: an active chat or lifecycle failure has error detail available to the event handler, but current main passes only "error" to setActivityStatus and renderStatus clears elapsed for non-busy states.

## Solution Assessment

No, not as proposed. The requested direction is right, but this PR mixes in broader stale-run/gap-recovery changes and introduces status-state bugs; a smaller patch limited to error-detail plumbing and elapsed preservation is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve elapsed time for plain error statuses:** `src/tui/tui.ts:676-678`
  - body: This branch clears `errorElapsed` when the previous status is the plain `error` label because it only treats `error:` labels as still-error rerenders. A connection-status rerender while `activityStatus` remains `error` will therefore drop the elapsed suffix the PR is meant to preserve.
  - confidence: 0.89
- **[P2] Keep optimistic binding until a run is claimed:** `src/tui/tui.ts:581-582`
  - body: When the stale timeout fires during `waiting` before the first gateway event, `activeChatRunId` can still be null and `pendingOptimisticUserMessage` is the only state that lets that first event be marked local. Clearing it here can misclassify a slow-start local turn as external and trigger avoidable history reloads.
  - confidence: 0.84
