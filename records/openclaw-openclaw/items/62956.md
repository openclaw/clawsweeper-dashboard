## Summary

Keep open: current main still has the source-level TUI status gap, but this branch is not merge-ready because it adds raw provider-error logging, unstable elapsed handling, and stale/gap recovery behavior outside the narrow status fix.

## What This Changes

The branch updates TUI status/event handling to show summarized failure details and elapsed time, adds stale-busy and gap-recovery timers/logging, and adds root AGENTS guidance.

## Best Possible Solution

Land a focused TUI status change that reuses existing sanitized error formatting, passes concise error detail into the status bar, preserves a stable elapsed suffix, and leaves stale-run recovery plus AGENTS policy changes out of this patch.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main receives error details for chat rendering but passes only plain error to activity status, and renderStatus clears elapsed for non-busy errors. I did not run a live provider failure in this read-only review.

## Solution Assessment

No, not as proposed. The status-bar direction is useful, but the branch overreaches into stale-run recovery, gap recovery, root policy text, and raw debug logging; a narrower status-only patch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Sanitize raw chat error logging:** `src/tui/tui-event-handlers.ts:382-384`
  - body: This logs `evt.errorMessage` directly through `logDebug`; `logDebug` writes to the file logger, and provider error strings can include request IDs, account details, or raw JSON. Use the sanitized summary or remove this diagnostic.
  - confidence: 0.78
- **[P2] Preserve elapsed for plain error rerenders:** `src/tui/tui.ts:676-678`
  - body: A repeated render while `activityStatus === "error"` enters this branch with `lastActivityStatus === "error"`, which does not start with `error:`, so `errorElapsed` is cleared and the elapsed suffix disappears.
  - confidence: 0.89
- **[P2] Keep optimistic binding until a run is claimed:** `src/tui/tui.ts:581-582`
  - body: When the stale timeout fires during `waiting` before the first chat event, `activeChatRunId` can still be null and `pendingOptimisticUserMessage` is the only local-run binding. Clearing it misclassifies the eventual event as external and can trigger unnecessary history reloads.
  - confidence: 0.84
- **[P2] Gate gap recovery on terminal completion:** `src/tui/tui.ts:1047-1050`
  - body: This reloads history while the same run is still active after `refreshSessionInfo()`. Session refresh only updates metadata, so a sparse but live run can have the chat log rebuilt mid-turn without terminal completion evidence.
  - confidence: 0.82
