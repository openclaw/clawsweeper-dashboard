## Summary

Keep open: current main still has the generic TUI failure status path, but this PR head is not merge-ready because it lacks real behavior proof and introduces correctness plus raw-error logging concerns.

## What This Changes

The PR modifies TUI event/status rendering to show summarized agent failure messages and elapsed time in the status bar, adds stale-run and gap-recovery timers/logging, and adds root AGENTS guidance.

## Best Possible Solution

Land a focused TUI status change that reuses existing sanitized error formatting, passes concise error detail into the status bar, preserves a stable elapsed suffix, and leaves stale-run recovery plus AGENTS policy changes out of this patch.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main receives error details for chat rendering but still passes only plain "error" to activity status, and renderStatus clears elapsed for non-busy error states. I did not run a live provider failure in this read-only review.

## Solution Assessment

No, not as proposed. The status-bar direction is useful, but the branch overreaches into stale-run recovery and raw debug logging; a narrower status-only patch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Sanitize raw chat error logging:** `src/tui/tui-event-handlers.ts:382-384`
  - body: This logs `evt.errorMessage` directly. `logDebug` writes to the file logger, and chat error payloads can be provider exception strings, so this can persist sensitive details. Log the sanitized status summary or remove this diagnostic.
  - confidence: 0.78
- **[P2] Preserve elapsed for plain error rerenders:** `src/tui/tui.ts:676-678`
  - body: This branch only treats prior labels starting with `error:` as same-error rerenders. A second render while `activityStatus === "error"`, such as a connection-status TTL render, clears `errorElapsed` and drops the elapsed suffix this PR is meant to preserve.
  - confidence: 0.89
- **[P2] Keep optimistic binding until a run is claimed:** `src/tui/tui.ts:581-582`
  - body: When the stale timeout fires during `waiting` before the first chat event, `activeChatRunId` can still be null and `pendingOptimisticUserMessage` is the only local-run binding. Clearing it misclassifies the eventual event as external and can trigger unnecessary history reloads.
  - confidence: 0.84
