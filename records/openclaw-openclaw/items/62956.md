Full review comments:

- **[P1] Require terminal completion before history reload:** `src/tui/tui.ts:1047-1050`
  - body: This gap-recovery branch calls `loadHistory()` while `state.activeChatRunId` is still the same run. `loadHistory()` clears and rebuilds the chat log, so a sparse but still-live run can lose in-flight output without terminal completion evidence.
  - confidence: 0.84
- **[P2] Remove raw provider-error debug logging:** `src/tui/tui-event-handlers.ts:382-384`
  - body: This logs `evt.errorMessage` directly through `logDebug`, and `logDebug` writes to the file logger before optional console gating. Provider errors can include request IDs, account details, or raw JSON, so remove this diagnostic or log only the sanitized short summary.
  - confidence: 0.86
- **[P2] Keep existing formatted chat errors:** `src/tui/tui-event-handlers.ts:387`
  - body: This replaces current main's `formatRawAssistantErrorForUi` plus local auth hint path with the raw provider string in chat. Keep the existing formatted chat display and use the new summary only for the status bar.
  - confidence: 0.88
- **[P2] Treat plain error as cached error state:** `src/tui/tui.ts:676-678`
  - body: When `activityStatus` remains plain `"error"`, `lastActivityStatus` is also plain `"error"`, which does not start with `"error:"`; a connection-status rerender then clears `errorElapsed` and drops the suffix. Treat plain error as an error state when preserving cached elapsed time.
  - confidence: 0.89
- **[P2] Keep optimistic binding until a run is claimed:** `src/tui/tui.ts:581-582`
  - body: If the stale timeout fires during `waiting` before the first chat event, `activeChatRunId` can still be null and `pendingOptimisticUserMessage` is the only local-run binding. Clearing it here can make the eventual run look external and trigger avoidable history reload behavior.
  - confidence: 0.82
