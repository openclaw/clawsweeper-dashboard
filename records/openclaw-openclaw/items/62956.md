## Summary

Keep open: current main still has the bare TUI error-status problem, but this PR head is not mergeable as-is because it logs raw provider errors, regresses existing chat error formatting, has error-elapsed rerender bugs, mixes in stale/gap recovery behavior, and still lacks real behavior proof.

## What This Changes

The PR threads summarized TUI agent failure messages into the status bar, preserves elapsed text for error states, adds stale-busy/gap-recovery diagnostics, and edits root AGENTS guidance.

## Best Possible Solution

Land a focused TUI status-bar fix that reuses existing sanitized chat-error formatting, threads a concise status-only error detail, preserves a stable elapsed suffix, and leaves stale-run recovery plus root policy edits for separate review.

## Reproduction Assessment

Yes, source-reproducible: current main routes error details to chat formatting but passes only bare error to the status bar and resets elapsed for non-busy error states. I did not run a live provider failure in this read-only review.

## Solution Assessment

No, not as proposed. The status-bar direction is useful, but the latest head changes chat formatting, adds raw debug logging, and folds in stale/gap recovery that should be removed or split before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Remove raw provider-error debug logging:** `src/tui/tui-event-handlers.ts:382-384`
  - body: This logs evt.errorMessage directly through logDebug, and logDebug writes to the file logger before any console gating. Provider errors can include request IDs, account details, or raw JSON, so use the sanitized summary or remove this diagnostic.
  - confidence: 0.84
- **[P2] Preserve existing chat error formatting:** `src/tui/tui-event-handlers.ts:387`
  - body: This replaces current main's formatted chat error and auth-error hint with the raw provider string even though chat rendering is out of scope. Keep formatRawAssistantErrorForUi and the auth hint for chat display, and use the summary only for the status bar.
  - confidence: 0.86
- **[P2] Preserve elapsed for plain error rerenders:** `src/tui/tui.ts:676-678`
  - body: A repeated render while activityStatus is plain error reaches this branch with lastActivityStatus also plain error, which does not start with error:, so errorElapsed is cleared and the elapsed suffix disappears.
  - confidence: 0.89
- **[P2] Keep optimistic binding until a run is claimed:** `src/tui/tui.ts:580-582`
  - body: When the stale timeout fires during waiting before the first chat event, activeChatRunId can still be null and pendingOptimisticUserMessage is the only local-run binding. Clearing it can misclassify the eventual run as external and trigger avoidable history reloads.
  - confidence: 0.83
- **[P2] Avoid reloading history for an active gap run:** `src/tui/tui.ts:1047-1050`
  - body: This branch reloads history while the same run is still active after refreshSessionInfo. loadHistory clears and rebuilds the chat log, so a sparse but live run can lose in-flight output without terminal completion evidence.
  - confidence: 0.82
