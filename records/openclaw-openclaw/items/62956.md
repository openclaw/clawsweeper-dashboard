## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Skip recovery reloads while the same run is active:** `src/tui/tui.ts:1050`
  - body: The deferred gap recovery calls `loadHistory()` precisely when `state.activeChatRunId` still matches the same run. `loadHistory()` clears and rebuilds the chat log, so a sparse but healthy in-flight run can lose visible streamed output before a terminal event arrives.
  - confidence: 0.9
- **[P2] Keep optimistic binding on stale waiting timeouts:** `src/tui/tui.ts:582`
  - body: `checkStaleBusy()` clears `pendingOptimisticUserMessage` even when no run has been claimed yet. During a slow start, the first later chat event can no longer be marked local, so finalization may treat the user's run as external and trigger avoidable history reloads.
  - confidence: 0.84
- **[P2] Remove raw provider-error debug logging:** `src/tui/tui-event-handlers.ts:382-384`
  - body: This logs the full provider error payload through `logDebug`, which writes to the file logger. Provider errors can include credentials, account details, private endpoints, or raw response bodies, so the status summary should not add a new raw persistence path.
  - confidence: 0.86
- **[P2] Preserve existing formatted chat errors:** `src/tui/tui-event-handlers.ts:387`
  - body: Current main formats assistant errors and adds auth-specific hints before showing them in the chat log. This branch falls back to the raw error string, which loses that user-facing formatting and can expose noisier provider payloads.
  - confidence: 0.82
- **[P2] Treat plain error as a cached error state:** `src/tui/tui.ts:676-678`
  - body: The rerender branch only recognizes previous statuses starting with `error:`; a plain `error` status clears `errorElapsed` on the next connection-status render. That makes the new elapsed display unstable for lifecycle or command errors without detail text.
  - confidence: 0.78
