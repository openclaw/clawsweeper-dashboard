## Summary

Keep open: current main still has the TUI status-bar failure-detail gap, but this PR is not merge-ready because it lacks real behavior proof and introduces blocking TUI state and security regressions.

## What This Changes

The PR threads agent failure summaries into TUI status rendering, preserves elapsed error display, adds stale-busy/gap-recovery logging, and adds root AGENTS guidance.

## Best Possible Solution

Land a focused TUI status fix that reuses current error formatting, passes concise failure detail to the status bar, and preserves a stable elapsed suffix without adding gap recovery, stale-run policy, or root AGENTS policy changes.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main receives chat and lifecycle failure details but still passes only "error" to activity status, and renderStatus clears elapsed for non-busy error states. I did not run a live provider failure because this was a read-only review.

## Solution Assessment

No, not as proposed. The status-bar direction is useful, but the branch broadens into stale-busy/gap recovery and raw debug logging; a narrower status-only patch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Sanitize raw chat error logging:** `src/tui/tui-event-handlers.ts:382-384`
  - body: This new debug log writes evt.errorMessage directly. logDebug sends messages to the file logger path, and gateway chat errors can contain raw provider payloads, so this can persist sensitive error details instead of using the existing formatted/sanitized error summary or omitting the log.
  - confidence: 0.82
- **[P2] Preserve elapsed for plain error rerenders:** `src/tui/tui.ts:676-678`
  - body: This branch clears errorElapsed when the previous status is the plain error label because it only treats error: labels as still-error rerenders. A connection-status rerender while activityStatus remains error will drop the elapsed suffix this PR is meant to preserve.
  - confidence: 0.9
- **[P2] Keep optimistic binding until a run is claimed:** `src/tui/tui.ts:581-582`
  - body: When the stale timeout fires during waiting before the first gateway event, activeChatRunId can still be null and pendingOptimisticUserMessage is the only state that lets the first event be marked local. Clearing it here can misclassify a slow-start local turn as external and trigger avoidable history reloads.
  - confidence: 0.84
- **[P2] Avoid history reloads while the run is still active:** `src/tui/tui.ts:1047-1050`
  - body: The deferred gap-recovery path reaches loadHistory while state.activeChatRunId is still the same run. loadHistory clears and rebuilds the chat log, so a sparse but still-active run can lose in-flight output after a sequence gap.
  - confidence: 0.82
