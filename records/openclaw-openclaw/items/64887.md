## Summary

Keep open. Current main still keeps terminal liveness in metadata, lifecycle events, and gateway wait snapshots, while auto-reply and CLI user-visible payload paths still project only reply payloads without a stable blocked/abandoned signal.

## What This Changes

Request to surface embedded-run livenessState, especially blocked and abandoned exits, in user-facing reply payloads or channel rendering.

## Best Possible Solution

Add a generic terminal-liveness surfacing helper that appends a stable, machine-strippable blocked/abandoned footer and short recovery hint to final visible payload text while preserving existing metadata and lifecycle events.

## Reproduction Assessment

Yes. Source inspection shows the strict-agentic blocked path returns only blocked text plus isError in payloads while meta.livenessState is blocked, and the existing regression pins that exact split; I did not execute tests during this read-only sweep.

## Solution Assessment

Yes. The narrowest maintainable first fix is Option A: synthesize a visible footer for blocked/abandoned terminal payloads, leaving a channel-aware envelope field for a separate API design if maintainers want it later.
