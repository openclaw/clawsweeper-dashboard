## Summary

Keep open. Current main still carries terminal liveness through run metadata and lifecycle events, but the user-visible reply payload path still has no stable `blocked` or `abandoned` signal.

## What This Changes

The issue requests that terminal run `livenessState` values such as `blocked` or `abandoned` be surfaced in user-visible outbound payloads instead of only in run metadata.

## Best Possible Solution

Add a shared terminal-run surfacing layer that appends a stable `blocked` or `abandoned` footer plus a short recovery hint to final visible payload text, leaving a channel-aware envelope redesign for a separate contract discussion.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the strict-agentic blocked path returns only text plus `isError` in `payloads` while `meta.livenessState` is `blocked`, and existing tests pin that split. I did not run tests or a live channel because this sweep is read-only.

## Solution Assessment

Yes. The issue's footer-style option is the narrowest maintainable first fix; the lifecycle-event backstop currently on main is useful but does not solve the user-visible payload gap.
