## Label Justifications

- `P2`: This is a source-backed user-visible agent reply gap with limited blast radius and a clear focused fix path.
- `impact:message-loss`: The terminal blocked/abandoned signal is silently dropped from outbound channel payloads that users actually see.
- `impact:session-state`: The run's terminal liveness state remains in metadata/events while the rendered reply can imply ordinary completion.

## Summary

Keep open. Current main still carries terminal liveness through run metadata and lifecycle paths, but the user-visible reply payload path still receives only the returned payload array, and the strict blocked regression still pins a plain text/error payload while `meta.livenessState` stays separate.

## What This Changes

Request to surface blocked or abandoned run `livenessState` in final user-visible payloads or channel envelopes so downstream channels can render the terminal state explicitly.

## Best Possible Solution

Add a shared terminal-run surfacing layer that appends a stable blocked/abandoned footer and short recovery hint to final visible payload text, while leaving a channel-aware envelope redesign for a separate API discussion.

## Reproduction Assessment

Yes at source level: a strict-agentic blocked run returns a plain `{ text: STRICT_AGENTIC_BLOCKED_TEXT, isError: true }` payload while `meta.livenessState` is `blocked`. I did not run a live channel flow in this read-only review.

## Solution Assessment

Yes. The footer-style first fix is the narrowest maintainable solution for the documented visibility gap; a channel-aware envelope can be evaluated later if maintainers want a larger payload contract change.
