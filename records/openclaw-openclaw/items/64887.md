## Summary

Keep open. The current main still keeps embedded-run liveness in `meta` and lifecycle events, while the reply payloads and outbound delivery surfaces that channels/CLI render do not carry a stable visible `blocked` or `abandoned` signal. The previous ClawSweeper review remains accurate against the newer main SHA.

## What This Changes

The issue requests surfacing embedded agent `livenessState` for blocked or abandoned terminal runs in user-visible reply payloads.

## Best Possible Solution

Add a shared generic surfacing layer for terminal embedded runs so final visible reply text carries a stable user-readable `blocked` or `abandoned` marker plus a concise next-step hint, while leaving a structured public payload field for a separate API decision if maintainers want it.

## Reproduction Assessment

Yes. The strict-agentic blocked regression already produces a result where `meta.livenessState` is `blocked` but the visible payload is only the blocked error text; following that through auto-reply or CLI delivery shows no projection from meta into user-visible payloads.

## Solution Assessment

Yes. The issue’s Option A is the narrowest maintainable first repair because it fixes observable channel/CLI behavior without expanding the public reply payload contract. Option B should be handled separately if maintainers want structured channel envelopes.
