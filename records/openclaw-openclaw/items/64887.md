## Summary

Keep open. Current main still keeps terminal liveness in metadata, lifecycle events, and gateway wait snapshots, but the visible reply payload path for channels and CLI still has no stable blocked/abandoned signal.

## What This Changes

The issue requests surfacing embedded-run `livenessState` in user-facing reply payloads for blocked or abandoned terminal exits.

## Best Possible Solution

Add a generic terminal-run surfacing layer so the final visible payload text carries a stable blocked/abandoned state plus a short recovery hint, while preserving existing metadata and lifecycle event behavior.

## Reproduction Assessment

Yes. The current strict-agentic blocked regression produces only the blocked text and `isError: true` in `payloads[]` while `meta.livenessState` is `blocked`; source inspection shows auto-reply and CLI delivery project from payloads without reading that meta field.

## Solution Assessment

Yes. The narrowest maintainable first fix is the issue's Option A: synthesize a visible, machine-strippable footer for blocked/abandoned terminal payloads. Option B should remain a separate API design discussion because it changes the reply envelope contract.
