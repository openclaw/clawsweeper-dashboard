## Summary

Keep open. Current main still has the reported investigation surfaces: Telegram polling concurrency is tied to agent max concurrency, the Gateway main lane is capped by the same setting, fallback candidates are attempted serially under the outer agent run timeout, and session-lock/watchdog settings remain fixed defaults. The issue is recent, concrete, and the prior ClawSweeper comment already kept it open; later commits only touched Telegram draft preview delivery, not the timeout/concurrency path.

## What This Changes

The issue reports simultaneous LLM provider timeouts across four Telegram-bound agents under concurrent use while the same provider APIs remain reachable via direct curl.

## Best Possible Solution

Keep this as an active reliability investigation until maintainers capture current-main telemetry for the reported workload, then land the smallest measured fix in the layer that actually stalls: fallback budgeting, provider transport, Gateway lane scheduling, Telegram polling, service environment, or session locking.
