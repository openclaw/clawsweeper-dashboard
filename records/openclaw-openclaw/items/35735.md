## Summary

Keep open: current main still lacks the proposed `sessions rm` and `sessions clear` CLI surface, but this PR is not merge-ready because it has no real behavior proof and its destructive path bypasses the current Gateway/session lifecycle contract.

## What This Changes

The PR adds `openclaw sessions rm <key>` and `openclaw sessions clear` CLI commands with scoped store selection, dry-run/JSON output, transcript removal, and focused tests.

## Best Possible Solution

Implement `rm` and `clear` as Gateway-backed CLI wrappers over approved session deletion semantics, with an explicit offline repair path for `--store`, stable JSON/docs, regression coverage, and real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main has no public `sessions rm` or `sessions clear` CLI/docs surface, and the PR's non-dry-run implementation visibly bypasses the existing Gateway `sessions.delete` lifecycle.

## Solution Assessment

No. The feature direction is useful, but direct `sessions.json` mutation is not the narrowest maintainable solution for configured stores; the safer path is Gateway-backed deletion plus an explicit offline repair mode.

- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: Configured non-dry-run `rm` and `clear` mutations call `updateSessionStore` directly, bypassing `sessions.delete` behavior such as main-session rejection, active runtime cleanup, transcript archival, lifecycle hooks, and `sessions.changed`.   - body: The direct delete path saves through `updateSessionStore` without `skipMaintenance`; the current writer runs stale/count/disk maintenance unless skipped. A missing-key `rm` or filtered `clear` can therefore prune unrelated sessions while reporting only predicate-matched removals.
  - confidence: 0.91
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures are reported as successful transcript removals.
  - confidence: 0.9
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:207-220`
  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it while `rm --json` emits `olderThanMinutes: null`; use explicit `null` for optional fields so the machine-readable schema is stable.
  - confidence: 0.78
