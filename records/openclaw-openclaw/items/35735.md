## What This Changes

The PR adds `sessions rm` and `sessions clear` CLI subcommands with store/agent scoping, dry-run/JSON output, transcript deletion, and focused command tests.

## Best Possible Solution

Add `rm` and `clear` as approved Gateway-backed CLI wrappers for configured stores, reserve direct file mutation for explicit offline repair, validate destructive arguments strictly, and document/prove the real workflow before merge.

## Reproduction Assessment

Yes, by source inspection and diff review. Current main has no public `sessions rm` or `sessions clear`, and the PR's non-dry-run path visibly mutates session stores outside the current Gateway delete lifecycle.

## Solution Assessment

No. The feature direction is reasonable, but the proposed implementation is not the safest maintainable path for destructive session deletion; configured deletes should preserve the Gateway contract and direct mutation should be limited to explicit offline repair.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: Configured non-dry-run `rm` and `clear` call `updateSessionStore` directly, bypassing current `sessions.delete` behavior such as main-session rejection, active runtime cleanup, transcript archival, lifecycle hooks, plugin ownership checks, and `sessions.changed`. Route configured deletes through the Gateway and reserve direct writes for explicit offline repair.
  - confidence: 0.93
- **[P1] Reject malformed --older-than values before deleting:** `src/commands/sessions-delete.ts:52-57`
  - body: `Number.parseInt` accepts partial strings, so values like `1d` or `1.5` become `1` minute and can make `sessions clear --older-than ...` delete far more than the user intended. Require a full positive integer, or implement the documented duration grammar before running a destructive clear.
  - confidence: 0.9
- **[P2] Keep direct offline deletes from pruning unrelated sessions:** `src/commands/sessions-delete.ts:144-155`
  - body: The direct delete path saves through `updateSessionStore` without disabling or accounting for session maintenance; a missing-key `rm` or filtered `clear` can therefore prune unrelated sessions and make dry-run diverge from real deletion. Any offline mutation path should either skip maintenance or report the extra maintenance actions explicitly.
  - confidence: 0.88
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` swallows `fs.promises.rm` failures but still records the transcript path as removed. Permission or I/O failures can produce an inaccurate destructive-operation summary, so only add the path when removal actually succeeds.
  - confidence: 0.86
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:212-213`
  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for non-applicable optional fields so machine-readable output has a stable shape.
  - confidence: 0.78
