## Summary

Keep this PR open because current main still lacks the requested CLI session-delete commands, but the branch is not merge-ready: the destructive implementation bypasses the existing Gateway session lifecycle, can delete/prune more than the user asked for, and has no real behavior proof.

## What This Changes

The PR adds `openclaw sessions rm <key>` and `openclaw sessions clear` with scoped store selection, dry-run/JSON output, transcript cleanup, and focused command tests.

## Best Possible Solution

Add `rm` and `clear` as approved CLI wrappers over Gateway-backed session deletion for configured stores, with direct file mutation limited to explicit offline repair and with stable JSON/docs plus regression coverage.

## Reproduction Assessment

Yes, by source inspection and diff review. Current main has no public `sessions rm` or `sessions clear` CLI surface, and the PR's non-dry-run implementation visibly mutates stores outside the current Gateway `sessions.delete` lifecycle.

## Solution Assessment

No. The feature direction is useful, but this implementation is not the narrowest safe solution for destructive session deletion; configured deletes should preserve the Gateway contract and any direct store mutation should be an explicit offline repair path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: Configured non-dry-run `rm` and `clear` mutations call `updateSessionStore` directly, bypassing `sessions.delete` behavior such as main-session rejection, active runtime cleanup, transcript archival, lifecycle hooks, plugin ownership checks, and `sessions.changed`. Route configured deletes through the Gateway and reserve direct writes for explicit offline repair.
  - confidence: 0.93
- **[P1] Reject malformed --older-than values before deleting:** `src/commands/sessions-delete.ts:52-57`
  - body: `Number.parseInt` accepts partial strings, so values like `1d` or `1.5` become `1` minute and can make `sessions clear --older-than ...` delete far more than the user intended. Require a full positive integer, or implement the documented duration grammar before running a destructive clear.
  - confidence: 0.9
- **[P2] Prevent direct deletes from pruning unrelated sessions:** `src/commands/sessions-delete.ts:144-155`
  - body: The direct delete path saves through `updateSessionStore` without `skipMaintenance`; the current writer runs stale/count/disk maintenance unless skipped. A missing-key `rm` or filtered `clear` can therefore prune unrelated sessions and make dry-run diverge from real deletion.
  - confidence: 0.88
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` swallows `fs.promises.rm` failures but still records the transcript path as removed. Permission, I/O, or missing-file cases can produce an inaccurate destructive-operation summary, so only add the path when the removal actually succeeds and is meant to count.
  - confidence: 0.86
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:207-220`
  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for non-applicable optional fields so machine-readable output has a stable shape.
  - confidence: 0.78
