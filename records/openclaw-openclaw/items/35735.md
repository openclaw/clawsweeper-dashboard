## Summary

Keep open. The feature is still useful and not implemented on current main, but this branch should not merge as-is because it bypasses the current Gateway deletion lifecycle, can apply unrelated store maintenance during direct writes, and still lacks external real behavior proof.

## What This Changes

The PR adds `openclaw sessions rm` and `openclaw sessions clear` CLI subcommands with scoped options, dry-run/JSON output, transcript cleanup, and command tests.

## Best Possible Solution

Rework the feature as Gateway-backed CLI wrappers over `sessions.delete` for configured stores, keep direct mutation only for explicit offline `--store` repair with maintenance controlled, and document the approved bulk-delete semantics.

## Reproduction Assessment

Yes, by source inspection. Current main still lacks the requested `sessions rm` and `sessions clear` CLI commands, while the PR's non-dry-run path mutates the store directly despite the existing Gateway `sessions.delete` lifecycle contract.

## Solution Assessment

No. The feature direction is useful, but the current implementation should be reworked around Gateway deletion, explicit offline repair behavior, and real behavior proof before merge.

  - confidence: 0.91

  - confidence: 0.9

  - confidence: 0.78
