## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve session-key casing before deleting:** `src/commands/sessions-delete.ts:40-41`
  - body: `normalizeSessionKey` lowercases the whole key before comparing. Current store-key normalization preserves provider-owned opaque peer IDs, so an exact destructive command can match and delete a different session whose key differs only by preserved casing.
  - confidence: 0.9
- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: Configured non-dry-run `rm` and `clear` write the store directly with `updateSessionStore`, bypassing `sessions.delete` behavior such as main-session rejection, active cleanup, transcript archival, lifecycle/plugin hooks, and `sessions.changed`. Use the Gateway for configured stores and keep direct writes for explicit offline repair.
  - confidence: 0.94
- **[P1] Reject malformed --older-than values before deleting:** `src/commands/sessions-delete.ts:48-57`
  - body: `Number.parseInt` accepts partial strings, so values like `1d` or `1.5` become `1` minute and can make `sessions clear --older-than ...` delete far more than the user intended. Require a full positive integer or an explicit duration grammar before running a destructive clear.
  - confidence: 0.9
- **[P2] Prevent maintenance from pruning outside the requested set:** `src/commands/sessions-delete.ts:144-155`
  - body: The direct delete path saves through `updateSessionStore` without disabling or reporting session maintenance. A missing-key `rm` or filtered `clear` can therefore prune unrelated sessions, making dry-run diverge from real deletion.
  - confidence: 0.88
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` swallows `fs.promises.rm` failures but still records the transcript path as removed. Permission or I/O failures can produce an inaccurate destructive-operation summary, so only add the path when removal actually succeeds.
  - confidence: 0.86
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:212-213`
  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for non-applicable optional fields so machine-readable output has a stable shape.
  - confidence: 0.78
