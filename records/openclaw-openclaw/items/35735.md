## Summary

Keep open. The requested CLI surface is still missing on current main, but this PR needs changes before merge because its deletion path bypasses existing session mutation contracts, can prune unrelated sessions, and is missing user-facing docs/changelog coverage.

## What This Changes

The PR adds `openclaw sessions rm <key>` and `openclaw sessions clear` with dry-run/JSON options, transcript removal, CLI wiring, and tests.

## Best Possible Solution

Land a focused session deletion CLI that reuses the current Gateway/shared session mutation services, preserves lifecycle and transcript safety semantics, and includes docs, changelog, and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR's non-dry-run delete calls `updateSessionStore` without suppressing maintenance or using the Gateway writer, so configured maintenance and runtime-session side effects can diverge from the requested rm/clear operation.

## Solution Assessment

No. The CLI direction is reasonable, but the implementation should route configured deletes through the existing session delete/writer contracts, keep direct writes only for explicit offline store repair, and document the new user-facing commands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Route rm through the session delete contract:** `src/commands/sessions-delete.ts:144-155`
  - body: `sessions rm` removes rows with a direct store update, bypassing the existing `sessions.delete` behavior that rejects main-session deletion, handles active/runtime cleanup, archives transcripts, and emits lifecycle hooks. Route configured non-dry-run deletes through the Gateway/shared delete service and reserve direct store writes for explicit `--store` repair paths.
  - confidence: 0.92
- **[P1] Prevent predicate deletes from running maintenance:** `src/commands/sessions-delete.ts:144-155`
  - body: This `updateSessionStore` save uses the default maintenance path, so a targeted `rm` or filtered `clear --older-than` can also prune stale, capped, or disk-budget sessions that did not match the user's predicate. Explicit delete commands should persist only their selected removals and make dry-run match real execution.
  - confidence: 0.94
- **[P3] Report transcript removals after successful rm:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` swallows every `fs.promises.rm` error and still records the path as removed. Permission or I/O failures will make the summary claim a transcript was deleted even when it remains on disk.
  - confidence: 0.9
- **[P3] Emit null for optional JSON fields:** `src/commands/sessions-delete.ts:212-213`
  - body: For `clear --json`, `key` is left as `undefined`, so `JSON.stringify` omits the field while `rm --json` emits `olderThanMinutes: null`. New machine-readable output should keep optional fields stable.
  - confidence: 0.78
- **[P3] Document the new session delete commands:** `src/cli/program/register.status-health-sessions.ts:160`
  - body: This adds user-facing CLI commands, but the PR does not update the sessions CLI docs or `CHANGELOG.md`. OpenClaw policy expects user-facing feature/API changes to be documented before merge.
  - confidence: 0.84
