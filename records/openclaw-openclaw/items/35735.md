## Summary

Keep open. Current main still does not expose `openclaw sessions rm` or `openclaw sessions clear`, and the related feature issue was already closed in favor of this PR, but the proposed patch has blocking session-writer/lifecycle issues and lacks external real behavior proof.

## What This Changes

The PR adds `openclaw sessions rm` and `openclaw sessions clear` with store/agent scoping, dry-run and JSON output, orphan transcript cleanup, CLI registration, and tests.

## Best Possible Solution

Add the CLI delete surface on top of the existing Gateway deletion semantics for configured stores, reserve direct writes for explicit `--store` offline repair with maintenance side effects disabled, document the command, and require real CLI proof before merge.

## Reproduction Assessment

Yes for the PR review findings. Source inspection of the PR diff and current main shows the new commands write directly through `updateSessionStore` while current main already has a richer `sessions.delete` Gateway contract, and the transcript/JSON reporting issues are visible in the added file.

## Solution Assessment

No. The feature direction is useful, but this implementation should be reshaped around current Gateway/session writer semantics, explicit offline repair behavior, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Route configured deletes through Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: The non-dry-run delete path mutates configured stores directly, bypassing the existing `sessions.delete` behavior that rejects main-session deletion, cleans up active/runtime state, archives transcripts, and emits lifecycle/session-change events. Route configured deletes through the Gateway contract and keep direct writes only for explicit offline `--store` repair paths.
  - confidence: 0.92
- **[P2] Skip maintenance for offline targeted deletes:** `src/commands/sessions-delete.ts:144-155`
  - body: For the remaining explicit `--store` path, this `updateSessionStore` call runs normal maintenance because no options are passed. A targeted `rm` or filtered `clear` can therefore prune unrelated stale, capped, or disk-budget sessions while reporting only predicate matches.
  - confidence: 0.9
- **[P3] Record transcripts only after successful removal:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures will be reported as successful transcript removals.
  - confidence: 0.9
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:212-213`
  - body: `clear --json` leaves `key` as `undefined`, which `JSON.stringify` omits, while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for optional fields so the new machine-readable output has a stable shape.
  - confidence: 0.78
