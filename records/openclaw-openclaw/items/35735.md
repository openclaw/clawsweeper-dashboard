## Summary

Keep open: the requested CLI feature is still absent on current main, but the proposed patch is not mergeable because it bypasses the current Gateway deletion contract, can prune unrelated sessions through store maintenance, and lacks required real behavior proof.

## What This Changes

The PR adds `openclaw sessions rm` and `openclaw sessions clear` with scope, dry-run, JSON output, orphan transcript cleanup, and command registration tests.

## Best Possible Solution

Add `rm` and `clear` on top of Gateway `sessions.delete` for configured stores, keep direct mutation only for explicit offline `--store` repair with maintenance disabled, and document the approved destructive semantics.

## Reproduction Assessment

Yes. Source inspection of the PR head shows non-dry-run deletes call `updateSessionStore` directly, while current main already defines `sessions.delete` as the lifecycle-safe deletion contract with cleanup, archiving, hooks, and change events.

## Solution Assessment

No. The feature direction is useful, but the implementation should be reshaped around the current Gateway deletion contract and an approved bulk-clear behavior before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: The non-dry-run path mutates configured stores directly with `updateSessionStore`. That bypasses current `sessions.delete` protections including main-session rejection, active/runtime cleanup, transcript archiving, lifecycle hooks, and `sessions.changed`; route configured deletes through the Gateway contract and reserve direct writes for explicit `--store` offline repair.
  - confidence: 0.92
- **[P2] Disable maintenance for offline delete writes:** `src/commands/sessions-delete.ts:144-155`
  - body: For the remaining explicit `--store` path, this `updateSessionStore` call runs normal session maintenance because no options are passed. A targeted `rm` or filtered `clear` can prune unrelated stale, capped, or disk-budget sessions while reporting only predicate matches.
  - confidence: 0.91
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures will be reported as successful transcript removals.
  - confidence: 0.9
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:212-213`
  - body: `clear --json` leaves `key` as `undefined`, which `JSON.stringify` omits, while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for optional fields so the new machine-readable output has a stable shape.
  - confidence: 0.78
