## Summary

Keep open. The feature is useful and still absent from current main, but the branch bypasses the lifecycle-aware Gateway deletion contract and lacks real behavior proof for the destructive CLI paths.

## What This Changes

The PR adds `openclaw sessions rm` and `openclaw sessions clear` CLI subcommands with scoped targets, dry-run/JSON output, age filtering, transcript deletion, and tests.

## Best Possible Solution

Route configured CLI deletion through Gateway `sessions.delete`, keep any offline `--store` repair path explicit and maintenance-safe, and document the approved destructive bulk-clear behavior.

## Reproduction Assessment

Yes. Source inspection of the PR head shows non-dry-run deletion uses `updateSessionStore` directly, while current main exposes and tests `sessions.delete` as the lifecycle-aware deletion contract.

## Solution Assessment

No. The feature direction is useful, but the implementation should be reshaped around the Gateway deletion contract with a narrow direct-store repair fallback before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: The non-dry-run path mutates configured stores with `updateSessionStore`, bypassing the `sessions.delete` contract for main-session rejection, active/runtime cleanup, transcript archiving, lifecycle/thread-unbind hooks, plugin-runtime owner checks, and `sessions.changed`. Route normal CLI deletes through the Gateway and reserve direct writes for explicit offline `--store` repair.
  - confidence: 0.93
- **[P2] Skip maintenance for direct store deletes:** `src/commands/sessions-delete.ts:144-155`
  - body: For any remaining direct `--store` path, this `updateSessionStore` call passes no writer options. In enforce mode, session maintenance can prune unrelated stale, capped, or disk-budget entries while the command reports only predicate matches, so dry-run and applied output can diverge.
  - confidence: 0.9
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures are reported as successful transcript removals.
  - confidence: 0.9
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:212-213`
  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for optional fields so the new machine-readable output has a stable shape.
  - confidence: 0.78
