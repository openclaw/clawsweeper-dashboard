## Summary

Keep open. Current main still lacks the requested public CLI subcommands, and the PR contains useful feature work, but the diff is not merge-ready because it bypasses Gateway session-delete semantics, can misreport or prune unrelated session state, lacks real behavior proof, and needs maintainer approval for destructive bulk-delete UX.

## What This Changes

The PR adds `openclaw sessions rm` and `openclaw sessions clear` CLI wiring, a new direct session-store deletion command, transcript removal, JSON/dry-run options, and focused tests.

## Best Possible Solution

Implement `rm` and `clear` as Gateway-backed operator CLI wrappers over approved session deletion semantics, with explicit offline repair behavior, stable JSON/docs, regression tests, and real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main has no public `sessions rm` or `sessions clear` wiring/docs, and the PR's non-dry-run path visibly bypasses the existing Gateway `sessions.delete` lifecycle.

## Solution Assessment

No. The feature direction is useful, but direct `sessions.json` mutation is not the narrowest maintainable solution for configured stores; the safer path is Gateway-backed deletion plus an explicit offline repair mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: Configured non-dry-run `rm` and `clear` mutations call `updateSessionStore` directly, so they bypass `sessions.delete` behavior such as main-session rejection, active runtime cleanup, transcript archival, lifecycle hooks, and `sessions.changed`. Route configured deletes through the Gateway and reserve direct writes for explicit offline repair.
  - confidence: 0.93
- **[P2] Prevent offline deletes from pruning unrelated sessions:** `src/commands/sessions-delete.ts:144-155`
  - body: The direct delete path saves through `updateSessionStore` without `skipMaintenance`; the current writer runs stale/count/disk maintenance unless skipped. A missing-key `rm` or filtered `clear` can therefore prune unrelated sessions while reporting only predicate-matched removals.
  - confidence: 0.91
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures are reported as successful transcript removals.
  - confidence: 0.9
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:207-220`
  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it while `rm --json` emits `olderThanMinutes: null`; use explicit `null` for optional fields so the machine-readable schema is stable.
  - confidence: 0.78
