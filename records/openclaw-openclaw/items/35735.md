## Summary

Keep open: current main still lacks the requested `sessions rm` and `sessions clear` CLI surface, and the branch is useful, but it should not merge as-is because the deletion path bypasses current writer/lifecycle contracts and can remove or report sessions incorrectly.

## What This Changes

The PR adds `openclaw sessions rm <key>` and `openclaw sessions clear` with direct session-store deletion, transcript cleanup, dry-run/JSON output, CLI wiring, and tests.

## Best Possible Solution

Land a focused CLI delete implementation that reuses the current Gateway/session-store writer semantics, deletes only predicate-matched sessions, preserves lifecycle safeguards, reports transcript cleanup accurately, and updates docs/changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with enforced session maintenance and unrelated stale, cap-eligible, or disk-budget-eligible entries, a non-dry-run targeted delete can prune more than the predicate-matched rows. The transcript and JSON findings are also directly reproducible from an `fs.promises.rm` rejection and `JSON.stringify` omitting `undefined`.

## Solution Assessment

No. The feature direction is reasonable, but this patch should first route configured mutations through the current Gateway/writer contract, avoid opportunistic maintenance during targeted deletion, and document the new CLI surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Skip maintenance during targeted deletes:** `src/commands/sessions-delete.ts:144-155`
  - body: `deleteSessionsByPredicate` persists every non-dry-run `rm` or filtered `clear` through `updateSessionStore` without `skipMaintenance`. Current session-store saves can prune stale rows, cap overflow, and enforce disk budget, so a targeted delete can remove unrelated sessions while reporting only predicate matches.
  - confidence: 0.94
- **[P2] Route configured deletes through the Gateway writer:** `src/commands/sessions-delete.ts:144-155`
  - body: Configured non-dry-run cleanup now goes through the Gateway so it shares the runtime writer. This command writes those stores directly from the CLI process, which can race with a running Gateway and bypass lifecycle/writer behavior; keep direct writes for explicit `--store` repair paths only.
  - confidence: 0.86
- **[P3] Report transcript removals only after successful rm:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still records the path as removed. Permission or I/O failures will make the summary claim a transcript was deleted even though it remains on disk.
  - confidence: 0.9
- **[P3] Emit null for optional JSON fields:** `src/commands/sessions-delete.ts:212-213`
  - body: `clear --json` leaves `key` as `undefined`, so the field disappears while `rm --json` emits `olderThanMinutes: null`. New machine-readable output should keep optional fields stable with explicit `null` values.
  - confidence: 0.78
- **[P3] Document the new session delete commands:** `src/cli/program/register.status-health-sessions.ts:156`
  - body: This adds user-facing CLI commands but only changes code and tests. Repo policy expects behavior/API features to update `docs/cli/sessions.md` and `CHANGELOG.md` before merge.
  - confidence: 0.82
