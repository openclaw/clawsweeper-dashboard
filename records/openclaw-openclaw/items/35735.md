## Summary

Keep open. The feature is still absent from current main and the PR tracks useful CLI work, but the patch bypasses current Gateway deletion contracts, has reporting issues, and lacks external real behavior proof.

## What This Changes

The PR adds `openclaw sessions rm <key>` and `openclaw sessions clear` with store/agent scoping, dry-run/JSON output, orphan transcript removal, CLI registration tests, and command tests.

## Best Possible Solution

Add the delete CLI on top of `sessions.delete` for configured stores, keep direct store mutation only for explicit offline `--store` repair, require an approved safe bulk-clear contract, and update docs before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head shows the delete path directly calls `updateSessionStore`, while current main shows `updateSessionStore` runs maintenance by default and `sessions.delete` provides the current lifecycle-safe deletion contract.

## Solution Assessment

No. The feature direction is useful, but the proposed implementation should be reshaped around the current Gateway deletion semantics and a documented, maintainer-approved bulk-clear behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve gateway deletion safeguards:** `src/commands/sessions-delete.ts:144-155`
  - body: The non-dry-run path mutates configured stores directly with `updateSessionStore`. That bypasses current `sessions.delete` protections including main-session rejection, active/runtime cleanup, transcript archiving, lifecycle hooks, and `sessions.changed`; route configured deletes through the Gateway contract and reserve direct writes for explicit offline `--store` repair.
  - confidence: 0.92
- **[P2] Skip maintenance for offline targeted deletes:** `src/commands/sessions-delete.ts:144-155`
  - body: For the remaining explicit `--store` path, this `updateSessionStore` call runs normal session maintenance because no options are passed. A targeted `rm` or filtered `clear` can prune unrelated stale, capped, or disk-budget sessions while reporting only predicate matches.
  - confidence: 0.91
- **[P3] Record transcripts only after successful removal:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures will be reported as successful transcript removals.
  - confidence: 0.9
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:212-213`
  - body: `clear --json` leaves `key` as `undefined`, which `JSON.stringify` omits, while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for optional fields so the new machine-readable output has a stable shape.
  - confidence: 0.78
