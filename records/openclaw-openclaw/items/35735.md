## Summary

Keep open. Current main still lacks the requested public `sessions rm` and `sessions clear` commands, but this PR should not merge as-is because the destructive delete path bypasses the existing Gateway session-delete lifecycle and still lacks contributor real behavior proof.

## What This Changes

The PR adds `openclaw sessions rm <key>` and `openclaw sessions clear`, a new `sessions-delete` command implementation, and tests for scoped deletion, dry-run/JSON output, age filtering, and transcript removal.

## Best Possible Solution

Implement `rm` and `clear` as Gateway-backed CLI wrappers over approved session deletion semantics, with explicit offline repair behavior, stable JSON/docs, and real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main lacks public `sessions rm` and `sessions clear` wiring/docs, and the PR's non-dry-run path visibly bypasses the current `sessions.delete` lifecycle.

## Solution Assessment

No. The feature direction is useful, but the current diff should be reworked around Gateway deletion semantics and explicit offline repair rather than direct store mutation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: Configured non-dry-run `rm`/`clear` mutations go through `updateSessionStore`, so they bypass `sessions.delete` behavior such as main-session rejection, active-run/runtime cleanup, transcript archival, lifecycle hooks, and `sessions.changed`. Route configured deletes through the Gateway and leave direct writes only for explicit offline repair.
  - confidence: 0.92
- **[P2] Prevent offline deletes from pruning unrelated sessions:** `src/commands/sessions-delete.ts:144-155`
  - body: The direct delete path saves through `updateSessionStore` without `skipMaintenance`, and the current writer runs stale/cap/disk maintenance unless skipped. A missing-key `rm` or filtered `clear` can therefore prune unrelated sessions while reporting only predicate-matched removals.
  - confidence: 0.91
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures are reported as successful transcript removals.
  - confidence: 0.9
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:207-220`
  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it while `rm --json` emits `olderThanMinutes: null`; use explicit `null` for optional fields so the machine-readable schema is stable.
  - confidence: 0.78
