## Summary

Keep open. The PR addresses a real CLI gap that current main still has, but the proposed persistence path can delete unrelated sessions under enforced maintenance, so it should not merge as-is.

## What This Changes

The PR adds `openclaw sessions rm <key>` and `openclaw sessions clear` through a new sessions deletion command module, CLI registration, and focused command tests.

## Best Possible Solution

Land a focused CLI implementation that adds explicit session deletion without coupling targeted deletes to opportunistic maintenance pruning, preserves transcript and trajectory cleanup contracts, returns stable JSON, and documents the commands for users.

## Reproduction Assessment

Yes. Use enforced `session.maintenance` settings with a store containing one predicate match plus unrelated stale, cap-eligible, or disk-budget-eligible sessions; dry-run reports only predicate matches while the non-dry-run save path can prune extra entries through `updateSessionStore`. The transcript reporting defect is reproducible by making `fs.promises.rm` reject and observing that the path is still counted as removed.

## Solution Assessment

No. The requested CLI direction is reasonable, but this patch is not the best implementation until targeted deletion bypasses normal maintenance side effects and transcript/JSON reporting is made accurate and stable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Skip maintenance during targeted deletes:** `src/commands/sessions-delete.ts:150`
  - body: `deleteSessionsByPredicate` persists every non-dry-run delete through `updateSessionStore` with no options. That save runs configured maintenance by default, so `sessions rm` or filtered `sessions clear` can also prune/cap unrelated sessions while reporting only predicate matches.
  - confidence: 0.93
- **[P3] Report transcript removals only after successful rm:** `src/commands/sessions-delete.ts:103-104`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and then still pushes the path into `removedPaths`. Permission or I/O failures will be reported as successful removals even though the transcript remains.
  - confidence: 0.9
- **[P3] Emit null for optional JSON fields:** `src/commands/sessions-delete.ts:218-219`
  - body: `clear --json` passes `key: undefined`, so `JSON.stringify` omits the field while `rm --json` emits `olderThanMinutes: null`. New machine-readable output should keep a stable shape for optional parameters.
  - confidence: 0.78
