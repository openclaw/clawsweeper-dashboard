## Summary

Keep open. Current main still lacks the requested `sessions rm` and `sessions clear` CLI surface, but this PR is not mergeable as-is because it bypasses the Gateway deletion lifecycle, can prune unrelated sessions through default store maintenance, and lacks real behavior proof.

## What This Changes

The PR adds `openclaw sessions rm <key>` and `openclaw sessions clear` CLI subcommands with scoped options, dry-run/JSON output, transcript removal, and command tests.

## Best Possible Solution

Rework the feature as Gateway-backed CLI wrappers over `sessions.delete` for configured stores, keep direct `--store` mutation only for explicit offline repair with maintenance controlled, and document the approved bulk-delete semantics.

## Reproduction Assessment

Yes, for the feature gap by source inspection. Current main registers sessions list, cleanup, and export-trajectory but no `sessions rm` or `sessions clear`, while the PR adds those commands through a direct store mutation path.

## Solution Assessment

No. The requested CLI surface is useful, but the proposed implementation should be reworked around the existing Gateway `sessions.delete` lifecycle and controlled offline repair behavior before merge.

  - body: Configured non-dry-run deletes mutate `sessions.json` directly with `updateSessionStore`, skipping the existing `sessions.delete` lifecycle contract: main-session rejection, active runtime cleanup, transcript archival, lifecycle hooks, and `sessions.changed`. Route configured deletes through Gateway and reserve direct writes for explicit `--store` repair.

A missing-key `rm` or filtered `clear` can prune unrelated stale, capped, or disk-budget sessions while reporting only predicate matches.
  - confidence: 0.91
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures will be reported as successful transcript removals.
  - confidence: 0.9

  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it, while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for optional fields so the new machine-readable output has a stable shape.
  - confidence: 0.78
