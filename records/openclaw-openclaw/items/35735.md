## Summary

Keep open: current main still lacks the requested direct session-delete CLI commands and this PR remains the canonical implementation attempt, but it should not merge as-is because it bypasses the Gateway deletion lifecycle, can prune unrelated sessions through maintenance, and lacks real behavior proof.

## What This Changes

The PR adds `sessions rm` and `sessions clear` CLI subcommands with exact/bulk deletion, age filtering, store/agent/dry-run/JSON options, transcript removal, and focused tests.

## Best Possible Solution

Implement `rm` and `clear` as Gateway-backed operator CLI wrappers over approved deletion semantics, with explicit `--store` offline repair behavior, stable JSON/docs, regression tests, and real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main registers `sessions`, `sessions list`, and `sessions cleanup` only, while the PR diff's non-dry-run path visibly mutates the store directly at `src/commands/sessions-delete.ts:144-155`.

## Solution Assessment

No. The feature direction is useful, but configured deletes should go through the Gateway lifecycle, while direct file writes should be limited to explicit offline repair and avoid unrelated maintenance side effects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: Non-dry-run `rm` and `clear` mutate `sessions.json` through `updateSessionStore`, so configured deletes bypass `sessions.delete` behavior such as main-session rejection, active-run cleanup, transcript archival, lifecycle/plugin hooks, and `sessions.changed`. Use Gateway deletion for configured stores and reserve direct writes for explicit `--store` offline repair.
  - confidence: 0.93
- **[P2] Prevent offline deletes from pruning unrelated sessions:** `src/commands/sessions-delete.ts:144-155`
  - body: The direct delete path calls `updateSessionStore` without `skipMaintenance`; the writer can enforce stale/count/disk cleanup during a missing-key `rm` or filtered `clear`, deleting sessions that did not match the command predicate while the summary reports only matched removals.
  - confidence: 0.91
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures are reported as successful transcript removals.
  - confidence: 0.9
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:207-220`
  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it while `rm --json` emits `olderThanMinutes: null`; emit explicit `null` for optional fields so the machine-readable schema stays stable.
  - confidence: 0.78
