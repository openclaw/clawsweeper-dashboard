## Summary

Keep open: current main still lacks the requested `sessions rm` and `sessions clear` CLI surface, but this PR is not mergeable as written because it bypasses the existing lifecycle-safe Gateway deletion contract and lacks real behavior proof for destructive commands.

## What This Changes

The PR adds `openclaw sessions rm` and `openclaw sessions clear` with scope options, dry-run/JSON output, orphan transcript removal, and command registration tests.

## Best Possible Solution

Add the CLI on top of Gateway `sessions.delete` for configured stores, reserve direct mutation for explicit offline `--store` repair with maintenance controlled, define safe bulk-clear behavior, and document the commands before merge.

## Reproduction Assessment

Yes. Source inspection shows current main does not register or document `sessions rm` or `sessions clear`, and PR-head inspection shows the proposed non-dry-run path mutates stores directly instead of using the existing Gateway deletion contract.

## Solution Assessment

No. The feature direction is useful, but the current patch is not the best implementation because configured deletes should route through `sessions.delete` and bulk destructive behavior needs a maintainer-approved contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: The non-dry-run path mutates configured stores directly with `updateSessionStore`, bypassing `sessions.delete` protections such as main-session rejection, active/runtime cleanup, transcript archiving, lifecycle hooks, and `sessions.changed`. Route configured deletes through the Gateway contract and keep direct writes for explicit offline `--store` repair.
  - confidence: 0.92
- **[P2] Disable maintenance for offline store deletions:** `src/commands/sessions-delete.ts:144-155`
  - body: For the explicit offline `--store` path that remains, this `updateSessionStore` call passes no options, so normal session maintenance can prune stale/capped/disk-budget entries unrelated to the requested `rm` or filtered `clear` predicate while reporting only predicate matches.
  - confidence: 0.91
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and then still appends the transcript path, so permission or I/O failures are reported as successful removals.
  - confidence: 0.9
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:212-213`
  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for optional fields so the new machine-readable output shape is stable.
  - confidence: 0.78
