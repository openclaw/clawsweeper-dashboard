## Summary

Keep open: the requested CLI feature is still missing on current main, but this PR is not mergeable as written because destructive configured-store deletes bypass the existing Gateway deletion contract and the contributor has not provided real behavior proof.

## What This Changes

The PR adds `openclaw sessions rm` and `openclaw sessions clear` with store/agent scoping, dry-run, JSON output, orphan transcript removal, and command tests.

## Best Possible Solution

Implement the CLI on top of `sessions.delete` for configured stores, reserve direct store mutation for explicit offline `--store` repair with maintenance controlled, document the new commands, and define approved bulk-clear semantics before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head shows non-dry-run deletes call `updateSessionStore` directly, while current main defines `sessions.delete` as the lifecycle-safe deletion contract.

## Solution Assessment

No. The feature direction is useful, but the proposed implementation should be reshaped around the Gateway deletion contract and an approved bulk-clear behavior before it is maintainable.

## Review Findings

Full review comments:

- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: The non-dry-run path mutates configured stores directly with `updateSessionStore`. That bypasses current `sessions.delete` protections including main-session rejection, active/runtime cleanup, transcript archiving, lifecycle hooks, and `sessions.changed`; route configured deletes through the Gateway contract and reserve direct writes for explicit `--store` offline repair.
  - confidence: 0.92

  - body: For the remaining explicit `--store` path, this `updateSessionStore` call runs normal session maintenance because no options are passed. A targeted `rm` or filtered `clear` can prune unrelated stale, capped, or disk-budget sessions while reporting only predicate matches.
  - confidence: 0.91
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures will be reported as successful transcript removals.
  - confidence: 0.9

  - body: `clear --json` leaves `key` as `undefined`, which `JSON.stringify` omits, while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for optional fields so the new machine-readable output has a stable shape.
  - confidence: 0.78
