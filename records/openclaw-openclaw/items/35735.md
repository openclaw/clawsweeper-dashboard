## Summary

Keep open. Current main still lacks the requested `sessions rm` and `sessions clear` CLI surface, but this PR is not mergeable as-is because configured destructive deletes bypass the Gateway session lifecycle and the external PR still lacks real behavior proof.

## What This Changes

The PR adds CLI registration, command implementation, and tests for `openclaw sessions rm <key>` and `openclaw sessions clear`, including scoped targets, dry-run/JSON output, age filtering, and transcript removal.

## Best Possible Solution

Implement `rm` and `clear` as Gateway-backed wrappers over approved deletion and bulk-delete semantics, keep explicit `--store` as offline repair, and document stable JSON/destructive behavior.

## Reproduction Assessment

Yes, by source inspection. Current main lacks public `sessions rm` and `sessions clear` wiring/docs, and the PR's non-dry-run path visibly performs direct store mutation instead of using the current Gateway delete lifecycle.

## Solution Assessment

No. The feature direction is useful, but the implementation should be reworked around Gateway-backed deletion, explicit offline repair, stable JSON output, docs, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: Configured non-dry-run deletes mutate `sessions.json` directly with `updateSessionStore`, bypassing `sessions.delete` behavior such as main-session rejection, mutation cleanup, transcript archival, lifecycle hooks, and `sessions.changed`. Route configured deletes through the Gateway and reserve direct writes for explicit offline repair.
  - confidence: 0.92
- **[P2] Prevent offline deletes from pruning unrelated sessions:** `src/commands/sessions-delete.ts:144-155`
  - body: Any remaining direct `--store` delete path needs controlled maintenance such as `skipMaintenance`; otherwise a missing-key `rm` or filtered `clear` can trigger normal session maintenance and prune unrelated stale/capped/budget sessions while reporting only predicate-matched removals.
  - confidence: 0.88
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures will be reported as successful transcript removals.
  - confidence: 0.9
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:207-220`
  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it while `rm --json` emits `olderThanMinutes: null`; use explicit `null` for optional fields so the machine-readable schema is stable.
  - confidence: 0.78
