## Summary

Keep open: the feature is still missing on current main, but the PR’s current implementation bypasses established Gateway-backed session deletion semantics and still needs real behavior proof before merge.

## What This Changes

The branch adds CLI registration, command implementation, and tests for `openclaw sessions rm` and `openclaw sessions clear` with store/agent scoping, dry-run/JSON output, age filtering, and transcript cleanup.

## Best Possible Solution

Implement `rm` and `clear` as Gateway-backed CLI wrappers over approved session deletion semantics, keep explicit `--store` as controlled offline repair, and document stable JSON/destructive behavior.

## Reproduction Assessment

Yes, by source inspection. Current main lacks public `sessions rm` and `sessions clear` command wiring, and the PR’s non-dry-run path visibly mutates session stores directly instead of using the existing Gateway delete lifecycle.

## Solution Assessment

No. The feature direction is useful, but the current patch should be reworked around Gateway deletion, controlled offline repair, stable JSON output, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:150-161`
  - body: Configured non-dry-run deletes mutate `sessions.json` directly with `updateSessionStore`, bypassing `sessions.delete` behavior such as main-session rejection, active runtime cleanup, transcript archival, lifecycle hooks, and `sessions.changed`. Route configured deletes through the Gateway and reserve direct writes for explicit `--store` repair.
  - confidence: 0.92
- **[P2] Prevent offline deletes from pruning unrelated sessions:** `src/commands/sessions-delete.ts:150-161`
  - body: Any remaining direct `--store` delete path needs controlled maintenance, such as `skipMaintenance`; otherwise a missing-key `rm` or filtered `clear` can prune unrelated stale/capped/budget sessions while reporting only predicate-matched removals.
  - confidence: 0.91
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:103-104`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures will be reported as successful transcript removals.
  - confidence: 0.9
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:218-219`
  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it while `rm --json` emits `olderThanMinutes: null`; use explicit `null` for optional fields so the machine-readable schema is stable.
  - confidence: 0.78
