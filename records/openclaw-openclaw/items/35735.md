## Summary

Keep open. Current main still lacks the requested `sessions rm` / `sessions clear` CLI surface, but this PR is not mergeable as-is because it bypasses the Gateway session deletion and writer-maintenance contracts, has no real behavior proof, and is now conflicting against recent session cleanup changes.

## What This Changes

Adds `openclaw sessions rm` and `openclaw sessions clear` subcommands with scoped options, dry-run/JSON output, age filtering, transcript cleanup, and tests.

## Best Possible Solution

Rework the feature as Gateway-backed CLI wrappers over `sessions.delete` for configured stores, keep direct mutation only for explicit offline `--store` repair with maintenance controlled, and document the approved bulk-delete semantics.

## Reproduction Assessment

Yes by source inspection. The PR's non-dry-run path writes through `updateSessionStore` directly, while current main exposes a richer `sessions.delete` lifecycle contract and still lacks the requested `rm`/`clear` CLI commands.

## Solution Assessment

No. The feature direction is useful, but the implementation should be reworked around the Gateway deletion lifecycle, explicit offline repair behavior, and real behavior proof before merge.

  - body: Configured non-dry-run deletes mutate `sessions.json` directly with `updateSessionStore`, skipping the existing `sessions.delete` lifecycle contract: main-session rejection, active runtime cleanup, transcript archival, lifecycle hooks, and `sessions.changed`.   - confidence: 0.91

  - confidence: 0.9

  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it, while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for optional fields so the new machine-readable output has a stable shape.
  - confidence: 0.78
