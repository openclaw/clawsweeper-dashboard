## Summary

Keep open. Current main still lacks the requested `openclaw sessions rm` and `openclaw sessions clear` commands, but this PR should not merge as written because configured deletes bypass the existing Gateway deletion lifecycle and the contributor has not provided real behavior proof.

## What This Changes

The PR adds `openclaw sessions rm <key>` and `openclaw sessions clear` CLI subcommands with scoped options, dry-run/JSON output, transcript cleanup, and command tests.

## Best Possible Solution

Rework this as Gateway-backed CLI wrappers over `sessions.delete` for configured stores, keep direct mutation only for explicit offline `--store` repair with controlled maintenance, and document the approved destructive bulk-delete semantics.

## Reproduction Assessment

Yes, by source inspection. Current main still lacks `sessions rm` and `sessions clear`, and the PR's non-dry-run path mutates the store directly despite the existing Gateway `sessions.delete` lifecycle contract.

## Solution Assessment

No. The feature direction is useful, but this implementation should be reworked around Gateway deletion, explicit offline repair behavior, and contributor-provided real behavior proof before merge.

  - body: Configured non-dry-run deletes mutate `sessions.json` directly with `updateSessionStore`, skipping the existing `sessions.delete` lifecycle contract: main-session rejection, active runtime cleanup, transcript archival, lifecycle hooks, and `sessions.changed`. Route configured deletes through Gateway and reserve direct writes for explicit `--store` repair.

A missing-key `rm` or filtered `clear` can prune unrelated stale, capped, or disk-budget sessions while reporting only predicate matches.
  - confidence: 0.91
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures will be reported as successful transcript removals.
  - confidence: 0.9

  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it, while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for optional fields so the new machine-readable output has a stable shape.
  - confidence: 0.78
