## Summary

Keep open: the requested CLI feature is still not on current main, but this PR should not merge as-is because it bypasses the existing Gateway/session deletion contract and has only test/lint validation, not real behavior proof.

## What This Changes

The PR adds `openclaw sessions rm <key>` and `openclaw sessions clear` with store/agent scoping, dry-run/JSON output, transcript cleanup, CLI wiring, and tests.

## Best Possible Solution

Add the CLI delete surface on top of the existing Gateway deletion semantics for configured stores, keep explicit `--store` as offline repair, define safe bulk-clear behavior, and document the command before merge.

## Reproduction Assessment

Yes. Source inspection of the PR diff and current main shows the new commands write directly through `updateSessionStore` while current main already has a safer `sessions.delete` Gateway contract; the transcript and JSON reporting issues are also directly visible in the added file.

## Solution Assessment

No. The feature direction is useful, but this implementation should be reshaped around current Gateway/session writer semantics and should add real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve gateway deletion safeguards:** `src/commands/sessions-delete.ts:144-155`
  - body: The non-dry-run delete path mutates configured stores directly, bypassing the existing `sessions.delete` behavior that rejects main-session deletion, cleans up active/runtime state, archives transcripts, and emits lifecycle/session-change events. Route configured deletes through the Gateway contract and keep direct writes only for explicit offline `--store` repair paths.
  - confidence: 0.92
- **[P2] Skip maintenance for offline targeted deletes:** `src/commands/sessions-delete.ts:144-155`
  - body: For the remaining explicit `--store` path, this `updateSessionStore` call runs normal maintenance because no options are passed. A targeted `rm` or filtered `clear` can therefore prune unrelated stale, capped, or disk-budget sessions while reporting only predicate matches.
  - confidence: 0.91
- **[P3] Record transcripts only after successful removal:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures will be reported as successful transcript removals.
  - confidence: 0.9
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:212-213`
  - body: `clear --json` leaves `key` as `undefined`, which `JSON.stringify` omits, while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for optional fields so the new machine-readable output has a stable shape.
  - confidence: 0.78
