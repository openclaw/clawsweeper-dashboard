## Summary

Keep open. Current main still lacks the requested `sessions rm`/`sessions clear` CLI surface, but this PR is not merge-ready because configured deletes bypass the existing Gateway deletion/cleanup contracts and the PR still lacks after-fix real behavior proof.

## What This Changes

Adds `openclaw sessions rm` and `openclaw sessions clear` subcommands with scoped targets, dry-run/JSON output, age filtering, transcript cleanup, and tests.

## Best Possible Solution

Implement `sessions rm` and `sessions clear` as Gateway-backed CLI wrappers over `sessions.delete` for configured stores, keep direct file mutation only for explicit `--store` offline repair with maintenance controlled, and document the approved bulk-delete semantics.

## Reproduction Assessment

Yes. Source inspection shows the PR's non-dry-run deletion path calls `updateSessionStore` directly, while current main exposes `sessions.delete` and routes configured cleanup writes through the Gateway contract.

## Solution Assessment

No. The feature direction is useful, but the branch should be reworked around the Gateway deletion/lifecycle contract, explicit offline repair behavior, and real behavior proof before merge.

Full review comments:

- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: Configured non-dry-run deletes mutate `sessions.json` directly with `updateSessionStore`, skipping the existing `sessions.delete` lifecycle contract: main-session rejection, active runtime cleanup, transcript archival, lifecycle hooks, and `sessions.changed`. Route configured deletes through Gateway and reserve direct writes for explicit `--store` repair.
  - confidence: 0.92
- **[P2] Skip maintenance for offline delete writes:** `src/commands/sessions-delete.ts:144-155`
  - body: Any remaining direct `--store` path needs controlled maintenance, such as `skipMaintenance`, because `updateSessionStore` saves with maintenance by default. A missing-key `rm` or filtered `clear` can prune unrelated stale, capped, or disk-budget sessions while reporting only predicate matches.
  - confidence: 0.91
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures will be reported as successful transcript removals.
  - confidence: 0.9
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:212-213`
  - body: `clear --json` leaves `key` as `undefined`, so `JSON.stringify` omits it, while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for optional fields so the new machine-readable output has a stable shape.
  - confidence: 0.78
