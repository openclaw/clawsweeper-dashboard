## Summary

Keep open: the requested CLI feature is still absent from current main, but this branch is not mergeable as-is because configured deletes bypass the existing Gateway deletion contract, direct writes can run unrelated session maintenance, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds `openclaw sessions rm` and `openclaw sessions clear` CLI subcommands with scoped selection, dry-run/JSON output, orphan transcript removal, and command tests.

## Best Possible Solution

Implement `rm` and `clear` on top of Gateway `sessions.delete` or an approved bulk Gateway contract for configured stores, keep explicit `--store` offline repair maintenance-safe, and document/test the destructive behavior.

## Reproduction Assessment

Yes. Source inspection of the PR head shows non-dry-run deletes call `updateSessionStore` directly, while current main already defines `sessions.delete` as the lifecycle-safe deletion path with cleanup, archiving, hooks, and change events.

## Solution Assessment

No. The feature direction is useful, but the proposed direct mutation path is not the narrowest maintainable solution for configured stores and should be reshaped around the existing Gateway deletion contract or an explicit bulk contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Route configured deletes through the Gateway:** `src/commands/sessions-delete.ts:144-155`
  - body: The non-dry-run path mutates configured stores directly with `updateSessionStore`. That bypasses current `sessions.delete` protections including main-session rejection, active/runtime cleanup, transcript archiving, lifecycle hooks, and `sessions.changed`; route configured deletes through the Gateway contract and reserve direct writes for explicit `--store` offline repair.
  - confidence: 0.92
- **[P2] Disable maintenance for offline delete writes:** `src/commands/sessions-delete.ts:144-155`
  - body: For the remaining explicit `--store` path, this `updateSessionStore` call runs normal session maintenance because no options are passed. A targeted `rm` or filtered `clear` can prune unrelated stale, capped, or disk-budget sessions while reporting only predicate matches.
  - confidence: 0.91
- **[P3] Record transcript removal only after success:** `src/commands/sessions-delete.ts:97-98`
  - body: `removeTranscriptFiles` catches every `fs.promises.rm` failure and still appends the path to `removedPaths`, so permission or I/O failures will be reported as successful transcript removals.
  - confidence: 0.9
- **[P3] Emit null for absent JSON fields:** `src/commands/sessions-delete.ts:212-213`
  - body: `clear --json` leaves `key` as `undefined`, which `JSON.stringify` omits, while `rm --json` emits `olderThanMinutes: null`. Use explicit `null` for optional fields so the new machine-readable output has a stable shape.
  - confidence: 0.78
