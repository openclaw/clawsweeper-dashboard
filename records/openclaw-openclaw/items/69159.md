## Summary

Keep this PR open: the timeout-propagation bug target is still valid on current main, but the proposed error change is stale against the current `SessionWriteLockTimeoutError` contract added for session-lock failover suppression. The Greptile and prior ClawSweeper comments correctly pointed at the SDK exposure gap; that should be fixed using the current typed error/helper rather than adding a parallel error class.

## What This Changes

The branch changes embedded attempt lock acquisition to pass the run timeout, introduces a new session-busy lock error class, and adds a unit test for that error shape.

## Best Possible Solution

Update the PR against current main by keeping the timeout propagation, reusing the existing `SessionWriteLockTimeoutError`/code/helper contract, exposing that contract through the supported SDK runtime if it is meant for plugin callers, and adding focused tests plus the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to hold the same session write lock, run an embedded attempt with `params.timeoutMs` above 10 seconds, and observe that current main still omits `timeoutMs` from the lock acquisition so the 10,000 ms default applies.

## Solution Assessment

No, not as currently proposed. Passing `timeoutMs` is the narrow fix for the core bug, but the patch should preserve current main's typed lock-timeout/failover contract instead of introducing a second incompatible error shape.
