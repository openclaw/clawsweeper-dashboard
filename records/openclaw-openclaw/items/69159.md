## Summary

Keep open. Current main still does not pass the agent turn timeout into the embedded session write-lock acquisition, so the PR targets a real remaining bug; however, the branch needs maintainer/contributor update because current main already has a different `SessionWriteLockTimeoutError` contract and failover suppression that this PR’s new error shape would bypass.

## What This Changes

The PR passes `params.timeoutMs` to `acquireSessionWriteLock`, replaces generic session-lock timeout throws with a new `SessionWriteLockError` carrying `reason: "session_busy"`, and adds a lock-timeout property test.

## Best Possible Solution

Keep the PR open and update it against current main: propagate an intentional lock-wait timeout from embedded attempts, reuse or extend the existing `SessionWriteLockTimeoutError`/code/helper instead of adding a parallel error contract, expose the supported SDK surface only if plugin callers are meant to branch on it, and add focused tests for timeout propagation plus the public error contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to hold a session file lock, run the embedded attempt path with `params.timeoutMs` greater than 10 seconds, and observe that current main calls `acquireSessionWriteLock` without `timeoutMs`, causing the 10,000 ms default to apply.

## Solution Assessment

No, not as currently proposed. Passing a timeout into the lock acquisition addresses the core bug, but the PR should be rebased onto current main's existing typed lock-timeout/failover contract and SDK boundary rather than introducing a second incompatible error shape.
