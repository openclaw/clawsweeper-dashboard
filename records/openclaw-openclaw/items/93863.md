## Summary

Keep open: the PR targets a real misleading `config.patch` restart notice, but the formatter-only change is too broad because successful config writes can hot reload, skip restart, or require restart depending on the reload plan. The linked broader PR is relevant but is not a safe close/supersession target here because it is still open and reported as dirty/unknown rather than merged cleanly.

## What This Changes

The PR changes restart sentinel formatting so successful `config-patch` payloads render `Gateway restart required (...)` and adds a unit regression test for that summary.

## Best Possible Solution

Carry explicit restart-required or applied-state metadata from the config write/reload decision into the sentinel payload, then format pending restart-required, completed, and hot-reloaded config writes distinctly.

## Reproduction Assessment

Yes, from source inspection: current main formats a successful `config-patch` sentinel from `kind`, `status`, and `mode`, producing the misleading ok-style notice. I did not run a live Gateway repro because this review is read-only.

## Solution Assessment

No. The requested wording is correct for restart-required writes, but this PR applies it to every successful config.patch sentinel instead of using the actual restart decision from config write/reload logic.
