## Summary

Keep open: the PR addresses a real misleading config.patch restart notice, but the formatter-only branch is not merge-ready because it labels every successful config.patch sentinel as restart-required without knowing whether the reload plan actually required a gateway restart or whether the sentinel is being consumed after restart. The broader linked implementation is a better shape but is still open and conflicting, so it is not a safe superseding close target yet.

## What This Changes

The PR changes successful config.patch restart sentinel summaries to say `Gateway restart required (...)` and adds one restart-sentinel regression test.

## Best Possible Solution

Carry an explicit restart-required fact from the config write/reload decision into restart sentinel payloads, then format pending, completed, and hot-reload/no-restart notices distinctly without adding new config surface.

## Reproduction Assessment

Yes. Current main builds successful config-patch restart notices from kind/status/mode, which source-reproduces the misleading `Gateway restart config-patch ok (config.patch)` wording from the linked issue; I did not run a live Gateway repro in this read-only review.

## Solution Assessment

No. This PR is a plausible narrow formatter patch, but the best fix is to carry restart-required state from the config reload decision and distinguish pending restart-required, completed restart, and hot-reloaded writes.
