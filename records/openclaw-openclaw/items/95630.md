## Label Justifications

- `P1`: The report describes live channel turns hanging for minutes with restart-loop fallout in user-facing messaging workflows.
- `impact:message-loss`: A no-settlement channel turn can leave Telegram or another channel without any user-visible reply.
- `impact:session-state`: The requested semantics explicitly involve restart recovery and avoiding an abortedLastRun session-state marker.
- `impact:crash-loop`: The reported failure chain can cascade from a wedged channel turn into supervisor restarts and restart recovery loops.

## Summary

Keep open: current main still lacks a shared channel-dispatch no-settlement wall-clock deadline, and the requested default/config/cancellation semantics need maintainer product review rather than auto-close or strict bug automation.

## What This Changes

Request a configurable dispatcher-level wall-clock deadline for channel agent turns that do not settle with a reply or tool/action boundary.

## Best Possible Solution

Decide the timeout policy/default first, then implement one shared channel-turn cancellation path that aborts the active reply run, emits a structured reason, sends one channel-visible timeout notice, and preserves restart-recovery semantics.

## Reproduction Assessment

Partly: source inspection shows current main has no shared channel-turn no-settlement timeout, but I did not recreate the reported long live Telegram/Claude hang.

## Solution Assessment

Unclear: a shared channel-turn guard is plausible, but the public config key, default duration, settlement definition, and session-recovery semantics need maintainer product and compatibility review.
