- `P1`: The issue reports released channel turns hanging for minutes and cascading into unavailable gateway/channel workflows for real operators.

- `impact:crash-loop`: The report ties no-bound channel turns to event-loop starvation and supervisor restart-loop behavior, even though one cron restart amplification path is now fixed separately.

## Summary

Keep open: current main still lacks a dispatcher-level channel-turn no-settlement deadline, and the latest release also has no matching timeout surface. The cron restart amplification layer has been fixed separately, but the requested channel deadline, visible timeout reply, config/default, and restart-recovery semantics still need maintainer product direction.

## What This Changes

Request to add a configurable wall-clock deadline for channel-lane agent turns that fail to settle, with a visible timeout message and no restart-recovery marking.

## Best Possible Solution

Design one shared channel-turn deadline in the core inbound dispatch path, with approved config/schema/docs and focused tests for text/tool settlement, visible timeout delivery, and restart-recovery non-marking.

## Reproduction Assessment

Partly: source inspection confirms current main and v2026.6.10 lack the requested channel-turn no-settlement deadline. I did not live-reproduce the reported 559-second Telegram/Claude hang.

## Solution Assessment

Unclear: a shared dispatcher-level deadline is a plausible narrow mitigation, but the public config shape, default duration, settlement definition, and restart-recovery semantics need maintainer direction first.
