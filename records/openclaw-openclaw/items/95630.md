## Label Justifications

- `P1`: The issue reports released channel turns hanging for minutes and cascading into gateway restart-loop behavior.
- `impact:session-state`: The requested timeout contract explicitly depends on avoiding restart-recovery session marking after the timeout fires.
- `impact:message-loss`: A no-settlement channel turn can leave Telegram or other channel users without a visible reply until manual or supervisor recovery.
- `impact:crash-loop`: The report ties the channel hang to event-loop starvation and supervisor restart loops that can exhaust service restarts.

## Summary

Keep open: current main still has no shared channel-lane no-settlement deadline, and the proposed default/config/abort semantics need maintainer product direction before a fix PR is safe.

## What This Changes

The issue requests a dispatcher-level wall-clock no-settlement timeout for channel-lane agent turns, with configurable defaulting, structured abort logging, user notification, and no restart-recovery marking.

## Best Possible Solution

Design a shared channel-turn deadline in the core channel/reply dispatch path, with approved config/schema/docs and tests proving text/tool settlement cancels the timer, users get a visible timeout reply, and restart recovery is not marked.

## Reproduction Assessment

Partly: source inspection confirms current main lacks a channel-owned no-settlement deadline in the shared dispatch path, but I did not live-reproduce the reported 559-second Telegram/Claude hang.

## Solution Assessment

Unclear: a shared dispatcher-level deadline is a plausible narrow mitigation, but the default duration, public config shape, settlement definition, and restart-recovery semantics need maintainer direction first.
