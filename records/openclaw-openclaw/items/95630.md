## Label Justifications

- `P1`: The issue reports released channel turns hanging for minutes with user-visible message failures and gateway restart-loop interaction.
- `impact:session-state`: The requested timeout must distinguish no-settlement aborts from restart-recovery so session/run state is not misclassified.
- `impact:message-loss`: The affected channel-lane path can leave Telegram or WhatsApp users without a timely timeout response when an agent turn never settles.
- `impact:crash-loop`: The incident context ties stuck channel recovery and unsettled turns to gateway supervisor restart-loop behavior.

## Summary

Keep open: current main still lacks a shared dispatcher-level no-settlement deadline for channel-lane turns, and the requested guardrail needs maintainer direction on default/config, settlement, and restart-recovery semantics before implementation.

## What This Changes

Add a dispatcher-level wall-clock deadline for channel-lane agent turns that aborts no-settlement runs and notifies the channel without marking restart recovery.

## Best Possible Solution

Design a shared channel-turn deadline in the core channel/reply dispatch path, with maintainer-approved config/default semantics, visible timeout delivery, tests for text/tool settlement cancellation, and proof that timeout aborts do not mark restart recovery.

## Reproduction Assessment

Partly: source inspection confirms the requested channel-turn no-settlement guard is absent, but this read-only review did not reproduce the reported 559-second Telegram/Claude hang.

## Solution Assessment

Unclear: a dispatcher-level timeout is plausible, but default duration, config location, settlement definition, user-visible reply, and restart-recovery semantics need maintainer decision.
