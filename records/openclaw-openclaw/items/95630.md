## Label Justifications

- `P1`: The report describes live channel turns hanging for minutes and cascading into gateway restart-loop behavior on a released build.
- `impact:message-loss`: A no-settlement channel turn can leave the user without a visible Telegram or other channel reply until recovery.
- `impact:session-state`: The requested behavior explicitly depends on not marking the session as restart-recovery aborted when the channel timeout fires.
- `impact:crash-loop`: The issue ties the channel hang to supervisor restarts and restart-recovery loops that can exhaust the service.

## Summary

Keep open: current main and v2026.6.9 still await channel dispatch without a channel-turn no-settlement deadline, and the request targets a real gateway stability problem, but the public config/default and abort semantics need maintainer product direction.

## What This Changes

The issue requests a configurable wall-clock timeout for inbound channel agent turns that aborts no-settlement turns cleanly without marking restart recovery.

## Best Possible Solution

Design a shared channel-turn deadline in the core channel turn/reply dispatch path, with approved config/schema/docs, visible timeout delivery, and tests proving text/tool settlement cancels the timer and restart recovery is not marked.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source inspection confirms current main lacks the requested channel-turn no-settlement deadline, but this review did not live-reproduce the reported 559-second Telegram/Claude hang.

## Solution Assessment

Is this the best way to solve the issue? Unclear: a shared dispatcher-level deadline is plausible, but the default duration, public config shape, settlement definition, and restart-recovery semantics need maintainer direction first.
