## Summary

This should stay open: current main and the latest release still gate user override preservation on explicit /new or /reset, so a daily/idle rollover from an existing stale entry can mint a new session without carrying the user-selected model/auth override.

## What This Changes

A bug report asks OpenClaw to preserve user-driven /model and sessions.patch model/auth overrides when daily or idle freshness rollover lazily mints a new session.

## Best Possible Solution

Preserve user/legacy model, provider, and user auth-profile selections whenever an existing stale entry rolls into a new session, while still clearing auto fallback state and rotating runtime/CLI session state.

## Reproduction Assessment

No live current-main repro was run because this was a read-only cleanup review, but the code path is high-confidence: a stale entry with resetTriggered false skips the only user-override preservation helper and then builds the new entry without a baseEntry fallback.

## Solution Assessment

Yes, the narrow fix direction is to reuse the existing preserved-selection helper for implicit stale rollovers from an existing entry, with tests proving user selections survive and auto fallback selections still clear.
