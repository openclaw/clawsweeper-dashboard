## Summary

Close as implemented: current main and the v2026.5.18 release now mark missing or invalid post-compaction token totals stale, so the next compaction gate no longer treats a pre-compaction total as fresh truth.

## What This Changes

Ensure compaction persists a trustworthy post-compaction token count or invalidates stale totals before the next turn can reuse them.

## Best Possible Solution

Keep the shipped compaction token-state invariant and leave broader Codex-native/status accounting follow-ups on their narrower existing tracks.

## Reproduction Assessment

No current-main failure path remains from the reported source repro. The previously stale missing/invalid `tokensAfter` path now marks totals stale, and regression tests assert that behavior.

## Solution Assessment

Yes. The current fix is the narrow maintainable solution: keep valid post-compaction totals fresh and explicitly stale old totals when no trustworthy snapshot exists.
