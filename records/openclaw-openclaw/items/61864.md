## Summary

Keep open. Current main still has compaction-count paths that can preserve a pre-compaction fresh token total when no trustworthy post-compaction snapshot is available, so the next gate can treat stale state as current and re-enter compaction.

## What This Changes

Ensure successful compaction records trustworthy post-compaction token state or invalidates stale totals before the next turn uses them.

## Best Possible Solution

Make compaction token-state updates invariant across compaction-count paths: persist a valid positive finite `tokensAfter`, otherwise mark the previous token total stale before the next turn can use it.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start with an over-budget fresh stored total, complete compaction with missing or invalid `tokensAfter`, and observe that the recorder/updater increments `compactionCount` while leaving the old total usable as fresh truth.

## Solution Assessment

Yes. The narrow maintainable fix is to thread valid `tokensAfter` into the relevant store updates and explicitly stale the old total when no trustworthy post-compaction snapshot exists; status rendering and Codex-native follow-ups should stay separate.
