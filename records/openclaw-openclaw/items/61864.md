## Summary

Keep open. Current main still leaves the CLI compaction path able to increment `compactionCount` without persisting a trustworthy post-compaction token snapshot or invalidating the old fresh total, so the next gate can reuse stale pre-compaction state.

## What This Changes

Ensure post-compaction token state is persisted as fresh when trustworthy, or explicitly invalidated before the next compaction gate runs.

## Best Possible Solution

Thread valid `tokensAfter` through the CLI compaction lifecycle into the session store, and mark the previous total stale when compaction succeeds without a valid snapshot.

## Reproduction Assessment

Yes. A source-level reproduction is clear from current code: start with `totalTokensFresh: true` and an over-budget stored total, let CLI compaction return a lower `tokensAfter`, and the store recorder still leaves the old fresh total available to the next gate.

## Solution Assessment

Yes. The requested invariant is the narrow maintainable fix; status-only handling and Codex native compaction should remain separate from this CLI recorder/gate repair.
