## Summary

Keep open. Current main still has compaction-count paths that can preserve or reuse stale pre-compaction token totals when no trustworthy `tokensAfter` is available, so the next compaction gate can re-enter on old state.

## What This Changes

Ensure completed compaction persists reliable post-compaction token totals or invalidates stale totals before the next turn uses them for compaction gating.

## Best Possible Solution

Make compaction token-state updates invariant across all compaction-count paths: store a finite trusted `tokensAfter`, otherwise mark the previous total stale before any next-turn gate can reuse it.

## Reproduction Assessment

Yes, source-level. Start with an over-budget fresh stored total, complete compaction with missing or invalid `tokensAfter`, and current main can increment `compactionCount` while leaving the old total available to `resolveFreshSessionTotalTokens`.

## Solution Assessment

Yes. The narrow maintainable fix is to thread valid post-compaction counts into the store and explicitly stale old totals when no trustworthy count exists, while keeping status-only display fallback work on its separate track.
