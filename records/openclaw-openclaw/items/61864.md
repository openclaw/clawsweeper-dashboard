## Summary

Keep open. Current main handles some post-compaction token persistence, but the CLI compaction lifecycle still can leave a pre-compaction `totalTokens` snapshot marked fresh after incrementing `compactionCount`, so the next compaction gate can reuse the old total.

## What This Changes

The issue requests a post-compaction token-state invariant so compaction either persists a trustworthy `tokensAfter` snapshot or invalidates stale totals before the next turn.

## Best Possible Solution

Make every compaction recorder preserve the invariant: persist finite post-compaction `tokensAfter` as the fresh context snapshot, and explicitly mark or clear totals as stale when no trustworthy snapshot exists so no next-turn gate treats pre-compaction totals as current.

## Reproduction Assessment

Yes. A source-level reproduction is clear: seed a CLI session with `totalTokensFresh: true` and an over-budget total, let compaction return a lower `tokensAfter`, observe the store update only increments `compactionCount`, then the next lifecycle call still uses the old total as current.

## Solution Assessment

Yes. The issue’s requested invariant is the narrowest maintainable fix; the safer implementation is to thread `tokensAfter` through the CLI compaction store update and set `totalTokensFresh=false` when it is missing or invalid.
