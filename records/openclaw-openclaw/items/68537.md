## Summary

Keep open: current main still lacks Telegram direct-message ACP configured binding support, but this draft PR cannot merge as-is because it widens topic-shaped ID acceptance, updates the wrong docs surface, and lacks required real Telegram proof.

## What This Changes

The PR adjusts Telegram configured ACP binding normalization/matching, adds a focused binding adapter test, and changes a gateway configuration reference line to describe direct-chat bindings.

## Best Possible Solution

Land or replace this with a narrow Telegram adapter change that accepts positive plain DM IDs, rejects positive topic-shaped IDs and bare group IDs, updates current Telegram/ACP docs, and includes redacted real Telegram proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a top-level Telegram ACP binding with `match.peer.kind: "direct"` and a positive `match.peer.id`; current main passes that id to the Telegram binding adapter, which returns `null` before the rule can be registered.

## Solution Assessment

No as submitted. The Telegram adapter is the right boundary, but the patch must keep positive topic-shaped IDs rejected, update the current user-facing docs, and provide real Telegram proof before merge.
