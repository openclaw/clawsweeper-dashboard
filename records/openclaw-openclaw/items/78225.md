## Summary

This issue is still necessary: current main and the latest stable tag still have execution-side allowlist metadata writeback saving an older approvals object without a fresh hash check, and the linked fix PRs are still open rather than merged.

## What This Changes

The issue reports that node `system.run` allowlist metadata writeback can save a stale `exec-approvals.json` snapshot and restore admin-revoked exec policy.

## Best Possible Solution

Keep this issue open until a reviewed fix lands that either hash-guards execution-side metadata persistence or separates mutable runtime metadata from exec policy state, with regression coverage for stale revocation writeback.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: evaluate `system.run` against an allowlisted approvals snapshot, change the approvals file through the hash-checked admin set path, then let `recordAllowlistMatchesUse` save the stale snapshot. I did not execute the live race in this read-only review.

## Solution Assessment

Yes, the report's remediation direction is the right boundary: execution-side metadata persistence should be freshness-checked or moved out of policy state. Closing this issue before one of the linked PRs lands would not be the best resolution.
