## Label Justifications

- `P1`: The issue describes a high-impact exec approval boundary race where write-scoped execution can restore admin-revoked node policy.
- `impact:security`: The affected state controls node `system.run` authorization and the admin-vs-write approval boundary.

## Summary

Keep this issue open: current main and the latest release still have execution-side allowlist metadata writeback saving a previously resolved approvals object without a fresh hash/CAS check, and the linked implementation PRs remain open rather than merged.

## What This Changes

The issue reports a race where node-side `system.run` allowlist metadata writeback can restore an older exec approvals snapshot after an admin revokes policy.

## Best Possible Solution

Land a reviewed security fix that either hash-guards execution-side metadata persistence or moves mutable usage metadata out of the exec policy file, with regression coverage for stale revocation writeback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: resolve a node `system.run` allowlist snapshot, let an admin update the approvals file through the base-hash set path, then let execution-side allowlist metadata writeback save the stale snapshot.

## Solution Assessment

Yes, the remediation direction is the right boundary: execution-side metadata writes should be freshness-checked or separated from policy state. The issue should remain open until a linked fix lands and proves that path.
