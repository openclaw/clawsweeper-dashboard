## Summary

Keep open: current main still has the stale node exec-approvals metadata writeback path, and the linked closing PR remains open, conflicting, and marked as needing real behavior proof.

## What This Changes

The issue reports a race where node-host execution metadata writes can save a stale exec-approvals.json snapshot after an admin revokes allowlist policy.

## Best Possible Solution

Land a freshness-guarded exec approvals metadata write path in https://github.com/openclaw/openclaw/pull/78226 or a narrow replacement, while leaving the broader runtime-store split to https://github.com/openclaw/openclaw/issues/67154.

## Reproduction Assessment

Yes, source-reproducible. Current main loads approvals before an async skill-bin refresh and later saves that same approvals object through recordAllowlistUse, while the admin mutation path uses a baseHash check.

## Solution Assessment

Yes in direction. A reload-and-merge or compare-and-swap guard around execution-side metadata writes is the narrow maintainable fix; the open related PR targets that boundary but still needs merge/proof follow-up.
