## Summary

Keep open: current main still has the stale node exec-approvals metadata writeback path, and the linked closing PR remains open, conflicting, and marked as needing real behavior proof.

## What This Changes

The issue reports that node-host execution can write allowlist usage metadata from a stale `exec-approvals.json` snapshot after an admin revokes policy, restoring revoked exec approvals.

## Best Possible Solution

Land a CAS/freshness-guarded exec approvals metadata write path in https://github.com/openclaw/openclaw/pull/78226 or a narrow replacement, while leaving the broader runtime-tracking split to https://github.com/openclaw/openclaw/issues/67154.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence interleaving: approvals are loaded before an async skill-bin refresh, an admin set can save a newer file with base-hash validation, and `recordAllowlistUse` later saves the stale object without a freshness check.

## Solution Assessment

Yes. A reload-and-merge or compare-and-swap guard around execution-side metadata writes is the narrow maintainable fix; the open related PR targets that boundary but still needs merge/proof follow-up before this issue can close.
