## Summary

Keep open: current main still has the stale exec-approvals metadata writeback path, and open PR #78226 is the active closing fix but has not merged and is currently conflicting.

## What This Changes

The issue reports that node-host `system.run` allowlist metadata writeback can save an older `exec-approvals.json` snapshot after an admin revokes policy.

## Best Possible Solution

Add a freshness-guarded or reload-and-merge exec approval metadata write path that skips stale writeback when policy changed, with regression coverage for the interleaving.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence interleaving: approvals are loaded, an async skill-bin refresh can pause, an admin set can save a newer file with `baseHash`, and later metadata writeback saves the stale object.

## Solution Assessment

Yes. A CAS/freshness guard or reload-and-merge around execution-side metadata writes is the narrow maintainable fix; the linked PR targets that boundary but still needs review, conflict resolution, and real behavior proof.
