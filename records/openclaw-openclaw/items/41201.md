## Summary

Keep open: current main and v2026.6.6 still reject remote Control UI avatar rendering while CLI/config continue to accept `http(s)` avatar values, so the central contract mismatch is not implemented or safely superseded.

## What This Changes

The issue reports that avatars configured with `openclaw agents set-identity --avatar <url/path>` render as broken images in the Control UI.

## Best Possible Solution

Maintainers should choose the remote-avatar contract, then either add controlled same-origin remote-avatar support or tighten CLI/config/docs/diagnostics so Control UI-facing avatars are clearly local/data only.

## Reproduction Assessment

Yes, at source level: current CLI/config accept `http(s)` avatars, but Control UI helpers and CSP reject remote image URLs, and tests assert the resulting avatar URL stays null. I did not run a live browser/Railway reproduction.

## Solution Assessment

No, not as an automatic fix yet. The best path depends on a maintainer decision between controlled remote support and explicitly disallowing remote Control UI avatars in the public contract.
