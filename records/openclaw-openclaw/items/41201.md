## Summary

Keep open. Current main appears to have fixed the local workspace-avatar path, but the remote `http(s)` avatar case remains a real contract mismatch: source and docs deliberately block remote Control UI image rendering for CSP/privacy reasons, while CLI/config still accept and advertise remote avatar values.

## What This Changes

The issue asks for Control UI avatars configured with `openclaw agents set-identity --avatar <url/path>` to render instead of showing a broken image.

## Best Possible Solution

Settle the Control UI avatar contract deliberately: either implement remote avatars as controlled same-origin content with documented security limits, or make remote avatars explicitly unsupported for Control UI by aligning CLI help, config validation/docs, UI diagnostics, and tests while preserving the working local/data/blob avatar behavior.

## Reproduction Assessment

Yes for the remaining remote-avatar problem: current source accepts a configured `https://...` avatar, exposes it in `/avatar/<agentId>?meta=1`, and then rejects it in the Control UI render gate/CSP. No high-confidence current-main reproduction remains for ordinary workspace-local avatar paths because those now resolve through the authenticated same-origin avatar route.

## Solution Assessment

Unclear. The closed redirect-style approach in #41239 is not obviously the best fix under the current CSP/privacy posture; maintainers should first choose either a controlled same-origin proxy/cache design or an explicit unsupported-remote contract.
