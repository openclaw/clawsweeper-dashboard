## Summary

Keep open: current main handles local avatar routes and deliberately blocks remote Control UI image rendering, but the reported remote-avatar path is still a real public-contract mismatch between CLI/config accepting `http(s)` avatar values and Control UI refusing to render them for CSP/privacy reasons.

## What This Changes

The issue requests that Control UI display agent avatars set with `openclaw agents set-identity --avatar`, including remote URLs and local workspace paths.

## Best Possible Solution

Define one Control UI avatar contract: either support remote avatars through a controlled same-origin proxy/cache with explicit security controls, or make remote avatars clearly unsupported in CLI help, config validation/docs, and UI diagnostics while preserving local/data behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: configure an `https://` avatar, `resolveAgentAvatar()` returns `remote`, metadata can expose that URL, then the Control UI render gate and CSP prevent display.

## Solution Assessment

Unclear. There is no active patch, and the narrowest maintainable solution depends on a maintainer product/security decision between controlled remote support and an explicit local/data-only Control UI contract.
