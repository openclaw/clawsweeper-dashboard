## Summary

Keep open: current main and v2026.6.10 still block remote http(s) avatar rendering in Control UI while CLI/config continue to accept remote avatar values; the local avatar path is mostly implemented, but the remaining behavior needs a maintainer security/product decision rather than conservative cleanup closure.

## What This Changes

The issue asks for Control UI avatars configured via `openclaw agents set-identity` or `IDENTITY.md`, especially remote URLs and local paths, to render instead of showing missing or broken images.

## Best Possible Solution

Resolve the Control UI avatar contract deliberately: either support remote avatars through a controlled same-origin proxy/cache with explicit CSP, privacy, MIME, size, caching, auth, and diagnostics, or make remote avatars clearly unsupported by tightening CLI help, docs, validation, and UI feedback while preserving local avatar serving.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the remote-avatar half: configure an http(s) avatar, the resolver returns remote metadata, and the Control UI render gate clears it instead of displaying it. I did not run a live Railway/browser reproduction, and valid under-2 MB local file serving is implemented in current source.

## Solution Assessment

No automatic code fix is best yet. The maintainable solution needs a maintainer/security choice between tightening the public avatar contract to local/data-only rendering or adding reviewed remote-avatar support with diagnostics.
