## Summary

Keep open: current main and v2026.6.8 still intentionally reject remote `http(s)` avatar rendering even though CLI/docs accept those values. The local avatar route is mostly implemented, so the remaining work needs a maintainer/security decision rather than conservative cleanup closure.

## What This Changes

The issue requests that Control UI render agent avatars set by `openclaw agents set-identity --avatar`, including remote URLs and local workspace paths.

## Best Possible Solution

Resolve the Control UI avatar contract deliberately: either support remote avatars through a controlled same-origin proxy/cache with explicit CSP, privacy, MIME, size, caching, auth, and diagnostics, or make remote avatars clearly unsupported by tightening CLI help, schema/docs, validation, and UI feedback while preserving the local avatar route.

## Reproduction Assessment

Yes, by source inspection: a remote `http(s)` avatar can resolve into metadata, then `refreshChatAvatar()` rejects it through the Control UI renderability gate and leaves no avatar URL installed. I did not run a live Railway/browser repro.

## Solution Assessment

No automatic code fix is best yet. The maintainable solution needs a maintainer/security choice between tightening the public avatar contract to local/data-only rendering or adding reviewed remote-avatar support with diagnostics.
