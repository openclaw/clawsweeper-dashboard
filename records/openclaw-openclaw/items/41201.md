## Summary

Keep open. Current main still source-reproduces the remote-avatar contract mismatch: CLI/config accept `http(s)` avatar values and gateway metadata can return them, but Control UI rendering and CSP intentionally block remote images for security/privacy reasons.

## What This Changes

The issue asks for avatars configured with `openclaw agents set-identity --avatar <url/path>` to render correctly in the Control UI.

## Best Possible Solution

Resolve the Control UI avatar contract deliberately: preserve authenticated local same-origin avatar serving, route local-path failures through the open local-avatar item, and either add secure proxied remote support or make remote avatars clearly unsupported for Control UI-facing contracts.

## Reproduction Assessment

Yes, at source level. Current main accepts a remote avatar value, can return it in metadata, then Control UI drops it because only same-origin, `data:image/...`, and local `blob:` avatar URLs are renderable under the helper and CSP policy.

## Solution Assessment

No for the closed redirect-style fix direction. The maintainable solution needs a product/security decision between controlled same-origin remote handling and clearly unsupported remote avatars in the public Control UI contract.
