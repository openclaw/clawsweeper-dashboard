## Summary

Keep open: current main has strong local workspace-avatar handling, but the remote `http(s)` avatar case remains an unresolved Control UI contract mismatch because CLI/config still accept remote values while the UI and CSP intentionally drop them.

## What This Changes

The issue asks for Control UI avatars configured through `openclaw agents set-identity --avatar` to render for external URLs and local workspace paths instead of showing a broken image.

## Best Possible Solution

Settle the avatar contract, then align CLI help, config validation, docs, UI diagnostics, avatar metadata, and tests around either controlled remote support or explicitly local/data-only Control UI behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows a configured `https://...` avatar is accepted and exposed as remote metadata, then rejected by the Control UI render gate and CSP before display.

## Solution Assessment

Unclear. There is no active patch, and the narrowest maintainable fix depends on whether maintainers want remote avatar support or a stricter local/data-only public contract.
