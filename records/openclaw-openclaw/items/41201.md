## Summary

Keep open: current main still has a real Control UI avatar contract mismatch. The resolver/config/docs accept `http(s)` avatar values and the avatar metadata route can return them, but the browser-side Control UI intentionally treats remote URLs as non-renderable and shows a remote-blocked diagnostic instead of displaying the image.

## What This Changes

Issue reports broken Control UI avatar rendering after setting an agent avatar with `openclaw agents set-identity --avatar <url/path>`.

## Best Possible Solution

Resolve the avatar contract deliberately: either support remote avatars through a controlled same-origin proxy/cache with explicit CSP, privacy, MIME, size, caching, and diagnostics, or make remote avatars clearly unsupported for Control UI by tightening CLI help, schema/docs, validation, and UI feedback while preserving the local avatar route.

## Reproduction Assessment

Yes. Current source accepts `http(s)` avatar values in config/docs and returns remote avatar metadata, but the Control UI render helper only allows `data:image` or same-origin `/...` URLs, so the reported remote avatar still will not display.

## Solution Assessment

No automatic code fix is best yet. The maintainable solution needs a maintainer/security choice between tightening the public avatar contract to local/data-only rendering or adding reviewed remote-avatar support with diagnostics.
