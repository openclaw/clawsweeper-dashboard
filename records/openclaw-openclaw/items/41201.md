## Summary

Keep open: current main has strong local-avatar coverage, but the remote `http(s)` avatar path remains a public contract mismatch between CLI/config acceptance and Control UI's intentional CSP/privacy block.

## What This Changes

The issue requests that agent identity avatars set with `openclaw agents set-identity --avatar` render in the Control UI for remote URLs and local workspace paths.

## Best Possible Solution

Settle the avatar contract, then align CLI help, config validation, docs, UI diagnostics, avatar metadata, and tests around either controlled remote support or explicitly local/data-only Control UI behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows a configured `https://...` avatar is accepted and exposed as remote metadata, then rejected by the Control UI render gate and CSP.

## Solution Assessment

Unclear. There is no active patch, and the narrowest maintainable fix depends on whether maintainers want remote avatar support or a stricter local/data-only public contract.
