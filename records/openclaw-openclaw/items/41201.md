## Summary

Keep open: current main and v2026.6.10 still intentionally block remote http(s) avatar rendering in Control UI while CLI/config continue to accept remote avatar values. Local workspace avatars are mostly implemented, but the remaining remote-avatar behavior needs a maintainer security/product decision rather than conservative cleanup closure.

## What This Changes

The issue reports broken Control UI avatars after setting an agent avatar with `openclaw agents set-identity --avatar`, covering both remote URLs and local workspace paths.

## Best Possible Solution

Resolve the Control UI avatar contract deliberately: either support remote avatars through a controlled same-origin proxy/cache with explicit CSP, privacy, MIME, size, caching, auth, and diagnostics, or make remote avatars clearly unsupported by tightening CLI help, docs, validation, and UI feedback while preserving local avatar serving.

## Reproduction Assessment

Yes at source level for the remaining remote-avatar behavior: configure an http(s) avatar, the resolver returns remote metadata, and the Control UI renderability gate clears it instead of displaying it. No live Railway/browser reproduction was run, and valid under-2 MB local file serving is implemented in current source.

## Solution Assessment

No automatic code fix is best yet. The maintainable solution needs a maintainer/security choice between tightening the public avatar contract to local/data-only rendering or adding reviewed remote-avatar support with diagnostics.
