## Summary

Keep open. Current main and the latest release still split the avatar contract: `http(s)` avatar values are accepted and exposed as metadata, but Control UI rendering and CSP intentionally block remote image display, so this still needs a maintainer product/security decision.

## What This Changes

The issue asks for Control UI avatars configured through `openclaw agents set-identity --avatar <url/path>` to render instead of showing a broken image.

## Best Possible Solution

Resolve the public contract deliberately: either add secure same-origin remote-avatar handling with explicit CSP/privacy controls, or make remote avatars clearly unsupported for Control UI while preserving authenticated local avatar serving.

## Reproduction Assessment

Yes, at source level. Configure an `http(s)` avatar and current main accepts it as remote metadata, while the Control UI render gate and CSP prevent it from displaying.

## Solution Assessment

Unclear until maintainers choose the public contract. The closed redirect-style PR is not clearly the safest default because it bypasses the current same-origin image policy; a maintainable path is controlled same-origin handling or explicit unsupported-contract diagnostics.
