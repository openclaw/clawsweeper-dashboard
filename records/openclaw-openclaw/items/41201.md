## Summary

Keep open. Current main fixes the local workspace-avatar path, but remote `http(s)` avatar values remain accepted by CLI/config while Control UI intentionally refuses to render them for CSP/privacy reasons, so the user-facing contract is still split.

## What This Changes

The issue reports that avatars configured with `openclaw agents set-identity --avatar <url/path>` appear as broken images in Control UI.

## Best Possible Solution

Resolve the avatar contract deliberately so Control UI behavior, CLI help, config validation, docs, and diagnostics all agree on which avatar sources are supported.

## Reproduction Assessment

Yes from source inspection: a remote avatar value resolves to `kind: "remote"`, metadata can expose it, and the frontend drops it as non-renderable under the current CSP contract. I did not run the Vitest path because `pnpm` is not available in this checkout.

## Solution Assessment

Unclear. The previous related PR's redirect approach is not obviously the best fix because it conflicts with the current CSP/privacy posture; maintainers should first choose remote proxy/cache support versus tightening CLI/config/docs around local/data-only Control UI rendering.
