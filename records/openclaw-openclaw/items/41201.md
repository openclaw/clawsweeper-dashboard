## Summary

Keep open. Current main still has the same remote-avatar contract mismatch: CLI/config accept `http(s)` avatar values and gateway metadata can expose them, but Control UI helpers and CSP intentionally prevent remote image rendering, so closing this would hide a real product/security decision.

## What This Changes

The issue asks for Control UI avatars set with `openclaw agents set-identity --avatar <url/path>` to render instead of showing a broken image.

## Best Possible Solution

Resolve the contract deliberately: either add secure same-origin remote-avatar handling with explicit privacy/CSP controls, or make remote avatars clearly unsupported in Control UI validation, CLI help, docs, and diagnostics while preserving authenticated local avatar serving.

## Reproduction Assessment

## Solution Assessment

Unclear until maintainers choose the public contract. The closed redirect-style PR is not the safest default because it bypasses the current same-origin image policy; the maintainable path is secure same-origin handling or explicit unsupported-contract diagnostics.
