## Summary

Keep open. Current main covers the local workspace-avatar path, but the reported remote `http(s)` avatar flow remains a real Control UI contract mismatch: CLI/config still accept remote avatar values while the UI and CSP intentionally block remote image rendering.

## What This Changes

The issue asks for Control UI agent avatars configured via `openclaw agents set-identity --avatar` to render instead of showing a broken/default image, including remote URLs and local paths.

## Best Possible Solution

Resolve the public contract deliberately: either implement remote avatar rendering through an authenticated same-origin proxy/cache with explicit security limits, or make remote avatars clearly unsupported for Control UI by tightening CLI help, config validation/docs, and UI diagnostics while preserving the current local avatar route.

## Reproduction Assessment

Yes, at source level. A remote avatar value is accepted by the resolver and returned in avatar metadata, then Control UI clears it because only same-origin/data/blob avatar URLs are renderable under the current helper and CSP policy.

## Solution Assessment

No, not as the old simple redirect/data-inline approach from the closed related PR. The maintainable fix needs a product/security decision between controlled same-origin remote support and explicitly documenting/validating remote avatars as unsupported in Control UI.
