## Summary

Keep open. Current main still has a source-reproducible Control UI avatar contract split: CLI/config accept remote avatar URLs, but the gateway/UI/CSP path only treats local same-origin or data-image avatars as displayable, and the issue has security/product-decision labels that require maintainer direction.

## What This Changes

The issue asks for avatars set with `openclaw agents set-identity --avatar <url/path>` to display in the Control UI instead of appearing as broken images.

## Best Possible Solution

Resolve the avatar contract deliberately: keep the local same-origin avatar route, then either add secure proxied remote-avatar support or make remote avatars clearly unsupported for Control UI through CLI help, validation, docs, and UI diagnostics.

## Reproduction Assessment

Yes, at source level. A remote avatar value is accepted by config/resolution and returned in metadata, then Control UI rejects it because only same-origin paths and data-image URLs are renderable under the current helper and CSP policy.

## Solution Assessment

No, not as a direct redirect or permissive browser image load. The maintainable fix needs maintainer product/security direction between controlled same-origin remote handling and explicitly disallowing remote avatars in Control UI-facing contracts.
