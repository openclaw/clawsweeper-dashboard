## Summary

Keep open: current main and v2026.6.8 cover hardened local workspace avatar serving, but the reported remote `http(s)` avatar path remains an unresolved Control UI contract mismatch with explicit CSP/privacy implications.

## What This Changes

The issue asks for Control UI avatars set through `openclaw agents set-identity --avatar` to render instead of showing a broken image, including remote URLs and local workspace paths.

## Best Possible Solution

Maintain the hardened local avatar route, then make an explicit maintainer decision to either proxy/cache remote avatars safely or reject remote Control UI avatars consistently in CLI, docs, validation, and UI diagnostics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level for the remote-avatar case: CLI/config accept `http(s)` avatar values while the Control UI render gate and CSP still block remote image rendering; no live Railway/browser repro was run.

## Solution Assessment

Is this the best way to solve the issue? No automatic code path is best yet; maintainers need to choose controlled remote support versus contract tightening before a safe implementation can be queued.
