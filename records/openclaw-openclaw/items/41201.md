## Summary

Keep open: current main fixes the local workspace-avatar side and intentionally blocks remote Control UI image rendering, but the reported `http(s)` avatar path is still a public-contract mismatch because CLI/config accept remote avatar values while Control UI refuses to render them.

## What This Changes

This issue requests that Control UI display avatars configured through `openclaw agents set-identity --avatar`, including external URLs and local workspace paths, instead of broken images.

## Best Possible Solution

Define one Control UI avatar contract: either support remote avatars through a controlled same-origin proxy/cache with explicit security controls, or make remote avatars clearly unsupported in CLI help, config validation/docs, and UI diagnostics while preserving local/data behavior.

## Reproduction Assessment

Yes for the remaining remote-avatar path. Source inspection on current main shows an `https://` avatar resolves as `remote`, metadata can expose that URL, and the Control UI render gate plus CSP prevent it from displaying.

## Solution Assessment

Unclear. There is no active patch, and the narrowest maintainable solution depends on a maintainer product/security decision between controlled remote support and an explicit local/data-only Control UI contract.
