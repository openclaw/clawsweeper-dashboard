## Summary

Keep open: current main has strong local/avatar-route handling, but the reported remote `http(s)` Control UI case is still a contract mismatch because CLI/config accept remote values while Control UI intentionally drops them for CSP/privacy reasons.

## What This Changes

The issue asks for Control UI avatars configured with `openclaw agents set-identity --avatar` to render, including remote URLs and local workspace paths.

## Best Possible Solution

Define one Control UI avatar contract: either add remote support through an authenticated same-origin proxy/cache with explicit CSP and privacy controls, or make remote avatars clearly unsupported in CLI help, config docs/schema, validation, and UI diagnostics while preserving current local/data behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows a configured `https://...` avatar is accepted, exposed as remote avatar metadata, then rejected by the Control UI render gate and CSP before display.

## Solution Assessment

Unclear. There is no active patch, and the narrowest maintainable fix depends on whether maintainers want controlled remote-avatar support or a stricter local/data-only Control UI contract.
