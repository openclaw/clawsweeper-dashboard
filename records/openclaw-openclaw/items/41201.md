## Summary

Keep open for maintainer product/security review: current main and the latest release still accept remote avatar values through CLI/config, but Control UI deliberately blocks remote image rendering under CSP, so the central remote-avatar expectation is not implemented.

## What This Changes

Issue requests that Control UI display avatars configured through `openclaw agents set-identity --avatar`, including remote URLs and local workspace paths.

## Best Possible Solution

Resolve the avatar contract deliberately: either support remote avatars through an authenticated same-origin proxy/cache with explicit CSP, privacy, MIME, size, caching, and error-feedback behavior, or make remote avatars clearly unsupported for Control UI in CLI help, config docs/validation, and UI diagnostics while preserving local avatar serving.

## Reproduction Assessment

Yes at source level: current CLI/config accepts remote avatar values, the resolver returns `kind: "remote"`, and the Control UI render/CSP path blocks remote image rendering. I did not run a live browser reproduction in this read-only review.

## Solution Assessment

No automatic redirect/data-URL fix is the best path yet. The best solution needs a maintainer-approved remote-media policy, or an explicit decision to remove/diagnose remote avatar support for Control UI-facing contracts.
