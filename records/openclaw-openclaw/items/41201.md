## Summary

Keep open for maintainer product and security review: current main and v2026.6.6 still accept `http(s)` avatar values in CLI/config, but Control UI deliberately blocks remote image rendering under its same-origin CSP/privacy policy, so the remote-avatar expectation remains a contract mismatch rather than an implemented fix.

## What This Changes

The issue reports that agent avatars configured with `openclaw agents set-identity --avatar`, especially remote URLs and workspace-local paths, render as broken or missing images in the Control UI.

## Best Possible Solution

Resolve the contract deliberately: either support remote avatars through an authenticated same-origin proxy/cache with CSP, privacy, MIME, size, caching, and error-feedback tests, or make remote avatars clearly unsupported for Control UI across CLI help, config docs/validation, and UI diagnostics while preserving the local avatar route.

## Reproduction Assessment

Yes, at source level: current config accepts `http(s)` avatars, the resolver returns `kind: "remote"`, metadata can expose the URL, and the Control UI helper/test path drops remote URLs instead of rendering them. I did not run a live browser dashboard repro in this read-only review.

## Solution Assessment

No for an automatic redirect-only fix. The best solution needs a maintainer-approved remote-media policy because serving or rendering arbitrary remote avatar URLs conflicts with the current CSP/privacy hardening.
