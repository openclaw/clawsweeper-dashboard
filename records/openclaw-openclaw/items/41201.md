## Summary

Keep open: current main still accepts remote avatar values but intentionally prevents Control UI from rendering remote image URLs, and the local-file 404 side remains covered by a separate open issue/PR. This needs a maintainer product/security decision rather than conservative cleanup.

## What This Changes

The issue asks for avatars set with `openclaw agents set-identity --avatar <url/path>` to render in Control UI instead of showing a broken image.

## Best Possible Solution

Define one Control UI avatar contract: either support remote avatars through a controlled same-origin proxy/cache with explicit security controls, or clearly reject remote avatars in CLI/config/docs/UI diagnostics while letting https://github.com/openclaw/openclaw/issues/38439 own the local-file 404 path.

## Reproduction Assessment

Yes at source level for the remaining remote-avatar path: current main resolves `https://` avatars as remote metadata, then the UI render gate and CSP prevent rendering. I did not live-run the original Railway setup.

## Solution Assessment

Unclear. There is no active patch here, and the maintainable fix depends on a product/security decision between controlled remote support and an explicit local/data-only Control UI contract.
