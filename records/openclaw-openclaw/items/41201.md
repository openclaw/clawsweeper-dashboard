## Summary

Keep open: current main fixes the local avatar-serving side but still intentionally drops remote `http(s)` avatars in Control UI, while CLI/config still accept remote avatar values. The remaining issue is a real public-contract mismatch that needs a maintainer product/security decision, not stale cleanup.

## What This Changes

The issue asks for avatars configured with `openclaw agents set-identity --avatar <url/path>` to render in the Control UI instead of showing a broken or missing image.

## Best Possible Solution

Define one Control UI avatar contract: either support remote avatars through a controlled same-origin proxy/cache with explicit security controls, or make remote avatars clearly unsupported in CLI help, config validation, docs, and UI diagnostics while preserving local/data behavior.

## Reproduction Assessment

Yes for the remaining remote-avatar path from source inspection. Current main resolves `https://` avatars as remote metadata, then the Control UI route, frontend render gate, and CSP prevent them from displaying.

## Solution Assessment

Unclear. There is no active patch, and the narrowest maintainable solution depends on a product/security decision between controlled remote support and an explicit local/data-only Control UI contract.
