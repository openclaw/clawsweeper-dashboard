## Summary

Keep open. Current main and the latest release still expose a real avatar contract mismatch: CLI/config accept remote avatar values, but Control UI intentionally blocks remote image rendering under its avatar helper and CSP policy, so this needs maintainer product/security direction rather than cleanup closure.

## What This Changes

The issue reports that `openclaw agents set-identity --avatar <url/path>` produces a broken avatar in the Control UI, especially for remote `http(s)` avatar URLs.

## Best Possible Solution

Resolve the avatar contract deliberately: preserve authenticated local avatar serving, route local-path failures through the open local-avatar issue, and either add secure same-origin remote-avatar handling or make remote avatars clearly unsupported in Control UI-facing contracts.

## Reproduction Assessment

Yes, at source level. Current main accepts remote avatar values and can return them in metadata, while the Control UI helper and CSP block remote image rendering so the avatar is cleared instead of displayed.

## Solution Assessment

Unclear until maintainers choose the public contract. A direct redirect-style fix is not the best default because it bypasses the current remote-image security posture; the maintainable path is either secure same-origin handling or explicit unsupported-contract diagnostics.
