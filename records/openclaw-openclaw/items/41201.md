## Label Justifications

- `impact:security`: The remaining remote-avatar decision changes whether Control UI can trigger arbitrary remote image fetches from an operator browser.

## Summary

Keep open for maintainer product and security review: current main still accepts `http(s)` avatars in config/CLI, but Control UI intentionally blocks remote image rendering under the current CSP/privacy policy, so the original remote-avatar expectation remains a real contract mismatch rather than an implemented fix.

## What This Changes

The issue asks for Control UI agent avatars configured through `openclaw agents set-identity --avatar <url/path>` to render instead of showing a broken or fallback image.

## Best Possible Solution

Resolve the avatar contract deliberately: either add remote avatar support through a same-origin, authenticated image pipeline with matching CSP/privacy tests, or make remote avatars explicitly unsupported for Control UI in CLI help, config docs/validation, and UI diagnostics while preserving the local avatar route.

## Reproduction Assessment

Yes at source level: current config accepts `http(s)` avatars, the resolver returns `kind: "remote"`, metadata can expose the URL, and the Control UI avatar helper/test path drops remote URLs instead of rendering them. I did not run a live browser repro in this read-only review.

## Solution Assessment

No for an automatic or redirect-only fix. The best solution needs a maintainer-approved remote-media policy because the closed redirect/data-URL PR shape conflicts with the current CSP/privacy hardening and prior MIME-type security concern.
