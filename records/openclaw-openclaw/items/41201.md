## Summary

Keep open: current main has strong local-avatar handling, but the reported remote `http(s)` Control UI case remains a contract mismatch because CLI/config still accept remote avatar values while the Control UI intentionally drops them for CSP and privacy reasons.

## What This Changes

The issue requests that avatars set with `openclaw agents set-identity --avatar <url/path>` render in the Control UI instead of showing a broken image.

## Best Possible Solution

Choose one Control UI avatar contract and align it end to end: either implement remote avatars through an authenticated same-origin proxy/cache with explicit controls, or make Control UI local/data-only in CLI help, config validation, docs, and UI diagnostics.

## Reproduction Assessment

Yes, by source inspection: a configured `https://...` avatar is accepted by the resolver and exposed through metadata, then rejected by the Control UI render gate and blocked by CSP. I did not run a browser session because this review was read-only.

## Solution Assessment

Unclear. Current main is defensible security hardening for browser image fetches, but it is not the best final user contract until maintainers decide between controlled remote support and explicit local/data-only validation.
