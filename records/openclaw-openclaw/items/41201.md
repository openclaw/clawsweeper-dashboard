## Summary

Keep open. Current main and the latest release cover the authenticated local/avatar blob path, but the reported remote `http(s)` avatar case is still a real contract mismatch: CLI/config accept remote avatar values and gateway metadata can expose them, while Control UI helpers and CSP deliberately block remote rendering for security/privacy reasons.

## What This Changes

The issue asks for Control UI to render avatars configured through `openclaw agents set-identity --avatar`, including remote URLs and local workspace paths.

## Best Possible Solution

Resolve the contract deliberately: either add remote avatar support through an authenticated same-origin proxy/cache with explicit privacy and CSP controls, or make remote avatars clearly unsupported in Control UI validation, CLI help, config/docs, and diagnostics while preserving the local avatar route.

## Reproduction Assessment

Yes, at source level. Configure an `http(s)` avatar and current main accepts it as remote metadata, while the Control UI render gate and CSP prevent it from displaying.

## Solution Assessment

Unclear until maintainers choose the public contract. The safest maintainable paths are either secure same-origin remote-avatar handling or explicit unsupported-contract diagnostics, not an unconditional CSP relaxation or external redirect.
