## Summary

Keep open. Current main still has a real Control UI avatar contract mismatch: CLI/config accept remote avatar values, while the gateway, UI helper, tests, docs, and CSP intentionally keep remote images from rendering, so cleanup closure would hide a remaining product/security decision.

## What This Changes

The issue asks for Control UI avatars configured through `openclaw agents set-identity --avatar` to render instead of showing broken images for remote URLs and local paths.

## Best Possible Solution

Resolve the avatar contract deliberately: either add secure same-origin remote-avatar handling with explicit CSP/privacy controls, or make remote avatars clearly unsupported in Control UI validation, CLI help, docs, and diagnostics while preserving authenticated local avatar serving.

## Reproduction Assessment

Yes, at source level. Current main accepts remote avatar values and can expose them in metadata, while the Control UI helper and CSP block remote rendering so the configured remote avatar is cleared instead of displayed.

## Solution Assessment

Unclear until maintainers choose the public contract. The closed redirect-style PR is not the safest default because it bypasses the current same-origin image policy; the maintainable path is either secure same-origin handling or explicit unsupported-contract diagnostics.
