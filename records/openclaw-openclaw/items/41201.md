## Summary

Keep open: current main fixes or narrows the local workspace-avatar side, but the remote `http(s)` avatar path remains a real contract mismatch because CLI/config still accept remote avatar values while Control UI intentionally refuses to render them for CSP/privacy reasons.

## What This Changes

The issue reports that avatars configured with `openclaw agents set-identity --avatar <url/path>` show as broken or missing in the Control UI.

## Best Possible Solution

Define one Control UI avatar contract: either support remote avatars through a controlled same-origin proxy/cache with explicit security controls, or clearly reject remote avatars in CLI/config/docs/UI diagnostics while leaving local-file issues to the dedicated local-avatar thread.

## Reproduction Assessment

Yes at source level for the remote-avatar path: current main resolves `https://` avatars as remote metadata, then the frontend render gate and CSP prevent rendering. I did not live-run the original Railway setup.

## Solution Assessment

Unclear. There is no active patch here, and the maintainable fix depends on a product/security choice between controlled remote support and an explicit local/data-only Control UI contract.
