## Summary

Keep open: current main fixes the local workspace-avatar path, but the remote `http(s)` avatar path remains an unresolved contract mismatch because CLI/config still accept remote avatar values while Control UI intentionally refuses to render them.

## What This Changes

The issue asks for Control UI avatars set through `openclaw agents set-identity --avatar` to display for remote URLs and local workspace paths.

## Best Possible Solution

Settle the avatar contract deliberately, then align source, CLI help, config validation, docs, UI diagnostics, and tests around that chosen behavior while preserving working local/data/blob avatar rendering.

## Reproduction Assessment

Yes for the remaining remote-avatar problem: current source accepts a configured `https://...` avatar, exposes it through avatar metadata, and then the frontend/CSP path rejects it. No high-confidence current-main reproduction remains for ordinary workspace-local avatar files because the same-origin avatar route and tests cover that path.

## Solution Assessment

Unclear: there is no active patch, and the narrowest maintainable solution depends on whether maintainers want remote avatar support or an explicitly unsupported Control UI contract.
