## Summary

Keep open. Current main and the latest release still have a real Control UI avatar contract mismatch: `http(s)` avatar values are accepted and exposed as metadata, but the Control UI render gate and CSP intentionally prevent remote image rendering.

## What This Changes

The issue asks for avatars set with `openclaw agents set-identity --avatar <url/path>` to display in the Control UI instead of showing a broken image.

## Best Possible Solution

Resolve the public contract deliberately: either add secure same-origin remote-avatar handling with explicit CSP/privacy controls, or make remote avatars clearly unsupported in Control UI validation, CLI help, docs, and diagnostics while preserving authenticated local avatar serving.

## Reproduction Assessment

Yes, at source level. Configure an `http(s)` avatar and current main accepts it as remote metadata, while the Control UI render gate and CSP prevent it from displaying.

## Solution Assessment

Unclear until maintainers choose the public contract. The closed redirect-style PR is not the safest default because it bypasses the current same-origin image policy; the maintainable path is secure same-origin handling or explicit unsupported-contract diagnostics.
