## Summary

Keep open. Current main and v2026.5.19 still have an avatar contract mismatch: agent identity config accepts remote avatar values, while Control UI deliberately blocks remote image rendering under its helper and CSP policy, so this needs maintainer product/security direction rather than cleanup closure.

## What This Changes

The issue asks for avatars set with `openclaw agents set-identity --avatar` to display in Control UI instead of showing a broken image for remote URLs or workspace paths.

## Best Possible Solution

Resolve the Control UI avatar contract deliberately: either implement secure same-origin remote-avatar handling with CSP/privacy controls, or make remote avatars clearly unsupported in Control UI-facing validation, docs, CLI help, and diagnostics while preserving authenticated local avatar serving.

## Reproduction Assessment

Yes, at source level. Current main accepts remote avatar values and can expose them in metadata, while the Control UI helper and CSP block remote rendering so the avatar is cleared instead of displayed.

## Solution Assessment

Unclear until maintainers choose the public contract. The closed redirect-style PR is not the safest default because it bypasses the current remote-image security posture; the maintainable path is secure same-origin handling or explicit unsupported-contract diagnostics.
