## Summary

Keep open: current main has local avatar handling and metadata diagnostics, but remote `http(s)` avatar rendering remains an unresolved contract/security decision because CLI/config still accept remote avatar values while Control UI deliberately blocks them.

## What This Changes

The issue asks for Control UI avatars configured with `openclaw agents set-identity --avatar` to render instead of broken images for remote URLs and local paths.

## Best Possible Solution

Choose and document one Control UI avatar contract: either secure same-origin proxy/cache support with size, MIME, auth, caching, privacy, and CSP controls, or explicit remote-avatar rejection in validation, docs, CLI help, and UI diagnostics.

## Reproduction Assessment

Yes at source level: setting `identity.avatar` to an `https://` URL resolves as `remote`, the metadata endpoint reports it, and the frontend/CSP path refuses to render it. I did not live-run the original Railway setup.

## Solution Assessment

Unclear. There is no active patch here, and the maintainable fix depends on a product/security choice between controlled remote support and making remote avatars explicitly unsupported in Control UI-facing contracts.
