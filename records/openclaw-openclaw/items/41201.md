## Summary

Keep open: current main intentionally blocks remote Control UI avatar rendering while CLI/config still accept remote avatar values, so the remaining outcome needs a maintainer CSP/privacy contract decision; the local workspace-avatar side appears covered by current source and tests.

## What This Changes

Requests that Control UI display avatars configured through `openclaw agents set-identity --avatar`, including remote URLs and local workspace paths.

## Best Possible Solution

Define one Control UI avatar contract: either support remote avatars through a controlled authenticated same-origin path, or make remote avatars clearly unsupported across CLI help, config/docs, validation, and UI diagnostics while preserving local/data avatar behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows a configured `https://...` avatar is accepted, exposed as remote avatar metadata, then rejected by the Control UI render gate and CSP before display.

## Solution Assessment

Unclear. There is no active patch, and the narrowest maintainable solution depends on whether maintainers want controlled remote-avatar support or an explicitly local/data-only Control UI contract.
