## Summary

Keep open. Current main still has a real Control UI avatar contract split: remote avatar values are accepted by CLI/config and returned by avatar metadata, but the UI helper and CSP intentionally prevent remote image rendering.

## What This Changes

The issue asks for Control UI avatars configured through agent identity, especially remote URLs and local paths, to render instead of showing a broken image.

## Best Possible Solution

Resolve the avatar contract deliberately: preserve local same-origin avatar serving, route local-path follow-up through the existing local-avatar issue, and either add secure proxied remote support or make remote avatars clearly unsupported for Control UI.

## Reproduction Assessment

Yes, at source level. A remote avatar value is accepted by config/resolution and returned in metadata, then Control UI clears it because only same-origin, data, and blob avatar URLs are renderable under the current helper and CSP policy.

## Solution Assessment

No, not as a direct remote redirect or permissive browser image load. The maintainable path needs a product/security decision between controlled same-origin remote handling and explicitly disallowing remote avatars in Control UI-facing contracts.
