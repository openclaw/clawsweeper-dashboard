## Summary

Keep open. Current main and the latest release still leave the remote-avatar contract unresolved: OpenClaw accepts `http(s)` avatar values and can expose them as metadata, while the Control UI render guard and CSP intentionally prevent arbitrary remote images from displaying.

## What This Changes

The issue asks for Control UI avatars configured via `openclaw agents set-identity --avatar <url/path>` to render instead of showing a broken image.

## Best Possible Solution

Resolve the contract deliberately: either implement secure same-origin remote-avatar handling with auth, MIME, size, redirect, caching, privacy, CSP, and diagnostics covered, or make remote avatars clearly unsupported for Control UI in validation, CLI help, docs, and UI feedback.

## Reproduction Assessment

Yes, at source level. Current main accepts `http(s)` avatar values as remote metadata, while the Control UI render gate and CSP prevent those URLs from displaying.

## Solution Assessment

Unclear until maintainers choose the public contract. The closed redirect-style PR is not the safest default because it bypasses the current same-origin image policy; the maintainable path is secure same-origin handling or explicit unsupported-contract diagnostics.
