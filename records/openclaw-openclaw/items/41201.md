## Summary

Keep open: current main has fixed and hardened the local workspace-avatar path, but the remote `http(s)` avatar case remains a live Control UI contract mismatch because CLI/config still accept remote avatar values while the UI, route, and CSP intentionally prevent remote rendering.

## What This Changes

The issue requests that agent identity avatars set with `openclaw agents set-identity --avatar` render in Control UI for remote URLs and local workspace paths.

## Best Possible Solution

Resolve the avatar contract deliberately: either implement remote avatars through an authenticated same-origin proxy/cache with clear security limits, or make remote avatars explicitly unsupported for Control UI across CLI help, config validation, docs, metadata, and UI diagnostics.

## Reproduction Assessment

Yes. Source inspection on current main shows an `https://...` avatar is accepted as remote metadata, then rejected by the Control UI render gate and CSP before display.

## Solution Assessment

Unclear. There is no active patch, and the narrowest maintainable fix depends on a maintainer product/security decision about supporting remote avatars versus tightening the documented contract.
