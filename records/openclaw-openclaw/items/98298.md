## Label Justifications

- `P1`: The issue targets a replay failure class that can make long-running Anthropic agent sessions fail every subsequent turn until manual transcript repair.
- `impact:session-state`: The reported stale signed-thinking blocks poison replayed session history and keep the same invalid provider state in the active session.
- `impact:message-loss`: Affected turns fail before a usable assistant response can be delivered to the user.
- `impact:auth-provider`: The failure and proposed pruning behavior are tied to Anthropic signed-thinking provider replay semantics.

## Summary

Keep open. Current main has thinking-stripping helpers and related recovery hardening, but `contextPruning` still does not prune older assistant thinking blocks as an actual replay-view mutation, and the same signed-thinking replay failure remains owned by open Anthropic replay issues and PRs rather than a merged fix.

## What This Changes

Add a context-pruning replay-view pass and optional config to strip old assistant thinking blocks while preserving recent provider-required thinking.

## Best Possible Solution

Settle the provider replay policy in the #94228/#94493 Anthropic cluster first; add a narrow contextPruning/default knob only if maintainers still want operator-visible control after that policy is decided.

## Reproduction Assessment

Yes at source and cluster level: current `contextPruning` only uses thinking stripping for estimation, while current Claude 4 replay policy can preserve completed-turn thinking; #94228/#92201 provide production replay-invalid evidence. I did not run live Anthropic traffic in this read-only review.

## Solution Assessment

Unclear: a `contextPruning.thinking` config object is not obviously the best first fix because provider replay policy already owns the signed-thinking contract. A safer path is to land or reject the focused provider replay fix first, then decide whether context pruning needs a user-facing knob.
