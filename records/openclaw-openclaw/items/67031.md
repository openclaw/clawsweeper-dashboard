## Summary

Keep open. Current main still only partially addresses the report: transcript/tool image dimensions and OpenAI-compatible HTTP image budgets are configurable, but there is no shared `media.image` contract and several sanitizer, generic media, input-pixel, gateway, sandbox, and ACP limits remain fixed.

## What This Changes

Expose configurable image byte, input-pixel, dimension, and per-request limits across OpenClaw media sanitization layers.

## Best Possible Solution

Define a backwards-compatible, documented `media.image` limit contract with global defaults and scoped overrides, then thread the resolved limits through sanitizer, image-ops, generic media, sandbox/media-store staging, gateway attachments, ACP image reads, and OpenAI-compatible HTTP paths while preserving existing aliases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `media.image.*` is absent from the strict config schema, an 8,000 by 4,000 image is expected to hit the fixed input-pixel guard, and gateway images above `MAX_IMAGE_BYTES` are rejected before offload.

## Solution Assessment

Unclear. A shared configurable image-limit contract is directionally appropriate, but schema shape, defaults, override precedence, and resource ceilings need maintainer design before implementation.
