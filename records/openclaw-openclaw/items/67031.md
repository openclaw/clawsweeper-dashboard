## Summary

Keep open. Current main still only partially addresses the report: the transcript/tool image dimension knob and OpenAI-compatible chat-completion image budgets exist, but the requested shared `media.image` byte, input-pixel, dimension, and per-request contract is absent and multiple named limits remain fixed in source.

## What This Changes

The issue requests user-configurable image byte, input-pixel, dimension, and per-request limits across OpenClaw media sanitization, generic media handling, sandbox staging, ACP attachments, and OpenAI-compatible HTTP paths.

## Best Possible Solution

Define a backwards-compatible, documented `media.image` contract with global defaults, scoped overrides, hard upper bounds, and alias handling for existing knobs such as `agents.defaults.imageMaxDimensionPx`, then thread the resolved limits through tool-result sanitization, image-ops input guarding, generic media caps, sandbox/media-store staging, gateway attachment parsing, ACP attachment reads, and OpenAI-compatible HTTP endpoints with schema/help/docs/tests updated together.

## Reproduction Assessment

Yes. Source and test inspection on current main gives a high-confidence reproduction path: configure any proposed `media.image.*` key and it is rejected by the strict schema, while oversized image tests still assert fixed rejection before resize/offload.

## Solution Assessment

Unclear. The requested shared config direction is plausible, but the exact schema, precedence with endpoint and channel settings, default ceilings, and rollout across overlapping slice issues need maintainer design before a safe implementation.
