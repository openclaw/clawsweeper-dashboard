## Summary

Keep open. Current main and the latest shipped v2026.5.18 release still lack a shared `media.image.*` config contract, while several image byte and input-pixel limits remain fixed or split across older scoped knobs; the linked sandbox and resize items are partial slices, not a canonical replacement for this broader request.

## What This Changes

The issue requests a configurable media image limit contract for byte, input-pixel, dimension, and total request budgets across sanitizer, media, sandbox, ACP, gateway attachment, and OpenAI-compatible paths.

## Best Possible Solution

Define one backwards-compatible, documented image-limit resolver with bounded global defaults and scoped overrides, then thread resolved limits through sanitizer, media/image-ops, generic inbound media, sandbox/media store staging, ACP reads, gateway attachment parsing, and OpenAI-compatible HTTP paths.

## Reproduction Assessment

Yes for the configurability gap by source inspection: current main rejects `media.image.*` in the strict config schema and still uses fixed constants in the sanitizer, generic media, input-pixel guard, gateway attachment, sandbox, and ACP-adjacent paths. I did not run a runtime repro because this is primarily a public config and wiring request.

## Solution Assessment

Unclear. A shared configurable image-limit contract is the right direction, but the exact schema, scoped precedence, compatibility aliases, and safety ceilings need maintainer product/design review before implementation.
