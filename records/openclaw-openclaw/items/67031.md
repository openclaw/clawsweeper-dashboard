## Summary

Keep open. Current main still lacks the requested shared `media.image.*` limit contract, and several named sanitizer, media, gateway, sandbox, and ACP paths still read fixed byte or input-pixel caps; related open items cover narrower slices but do not supersede the broader design request.

## What This Changes

The issue requests configurable image byte, input-pixel, dimension, and request-total limits across OpenClaw media sanitization layers.

## Best Possible Solution

Define a backwards-compatible, documented image-limit resolver with bounded global defaults and scoped overrides, preserve existing aliases, and thread resolved limits through sanitizer, media/image-ops, generic media validation, sandbox staging, ACP, gateway attachment parsing, and OpenAI-compatible HTTP paths.

## Reproduction Assessment

Yes. Source inspection is enough: the strict config schema lacks `media.image.*`, and the named byte/input-pixel paths still read fixed constants on current main.

## Solution Assessment

Unclear. A shared configurable image-limit contract is the maintainable direction, but the exact schema, scoped precedence, compatibility aliases, and resource ceilings need maintainer design before implementation.
