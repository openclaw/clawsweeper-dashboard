## Summary

Keep open: current main and the latest release still have adjacent pre-tool blocking, final-answer revision, reply-dispatch, and outbound-cancel hooks, but they do not implement the requested fail-closed response/action gate with mandatory-tool accounting, typed validators, two-phase planning, fallback semantics, or audit records. The linked plugin/workaround comments are useful context, but they confirm the remaining core/plugin API gap rather than replacing it.

## What This Changes

The issue requests a fail-closed pre-response enforcement layer that blocks final output until mandatory tool calls and validators have run and passed.

## Best Possible Solution

Keep this open as the canonical response/action gate design item: define a versioned core/plugin API for mandatory-tool accounting, typed response validators, fail-closed final-response gating, bounded retry/fallback behavior, explicit user-approval envelopes for public/external actions, and audit records, while leaving domain-specific policy packs to plugins or ClawHub.

## Reproduction Assessment

Yes, at source level: current OpenClaw hooks can block tools or request one more pass, but they do not implement a fail-closed response gate with mandatory-tool and validator accounting. I did not run a live runtime reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer design: the requested two-phase gate is plausible and fits the missing seam, but the best API shape needs product/security review around versioning, fail-closed behavior, plugin trust, user approval, Codex parity, and audit storage.
