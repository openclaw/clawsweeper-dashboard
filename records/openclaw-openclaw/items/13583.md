## Summary

Keep open. Current main and the latest release still provide adjacent hooks for pre-tool policy, final-answer revision, outbound delivery, and message-tool-only replies, but not the requested fail-closed response gate with mandatory-tool accounting, typed validators, two-phase planning, retry/fallback semantics, and audit records.

## What This Changes

The issue requests a hard pre-response enforcement layer that prevents final output until mandatory tool calls and configured policy validators run and pass.

## Best Possible Solution

Keep this open as the canonical response-gate policy-as-code design item: add a versioned core/plugin API for mandatory-tool accounting, typed validators, fail-closed final-response gating, bounded retry/fallback behavior, and audit records, while leaving domain-specific policy packs to plugins or ClawHub.

Yes. Source-level reproduction is high confidence: current main and v2026.6.10 expose only adjacent revision, tool-policy, delivery-hook, and message-tool-only mechanisms, not a fail-closed responseGate with validator and mandatory-tool accounting.

Unclear pending maintainer design. A versioned core/plugin response-gate API is the likely right direction, but the permanent contract must compose with before_tool_call approvals, outbound pre-send enforcement, message-tool-only delivery, retry/fallback semantics, and audit records.
