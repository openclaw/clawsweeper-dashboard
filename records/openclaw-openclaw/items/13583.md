## Summary

Keep open: current main and v2026.6.6 still lack the requested fail-closed response gate with mandatory tool-call accounting, typed validators, two-phase planning, bounded fallback behavior, and audit records; adjacent hooks cover pre-tool policy, final-answer revision, and outbound cancellation but not this turn-level enforcement contract.

## What This Changes

Add a pre-response enforcement layer that prevents final output delivery until mandatory tools and configured validators have run and passed.

## Best Possible Solution

Keep this as the canonical response-gate design item and define a versioned, harness-neutral core/plugin API for mandatory tool-call accounting, typed response validators, fail-closed final-response authorization, bounded recovery, and audit records, with domain policy packs owned by plugins or ClawHub packages.

## Reproduction Assessment

Yes, at source level for the missing capability: current main and v2026.6.6 expose advisory prompt guidance, pre-tool blocking, revision hooks, and outbound cancellation, but no fail-closed responseGate with mandatory-tool and validator accounting. I did not run live tests in this read-only review.

## Solution Assessment

Unclear: the requested outcome is valid, but the full two-phase protocol is not yet proven as the narrowest permanent design. The safer direction is a versioned response-authorization API with runtime-owned accounting and plugin-owned domain validators.
