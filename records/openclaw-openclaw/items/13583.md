## Summary

Current main still lacks a fail-closed contract that ties required tool execution or typed validator results to final-response delivery. Existing hooks can revise an answer, block selected pre-tool operations, claim reply dispatch, or cancel outbound payloads, but they do not provide the requested mandatory-tool ledger, validator policy, deterministic fallback, and compliance audit guarantee; the active discussion and newer live example show the central problem remains unresolved.

## What This Changes

Requests a fail-closed pre-response policy layer that can require tools and validators before final output is delivered.

## Best Possible Solution

Create a versioned, generic core/plugin response-policy contract that carries canonical tool-result facts into finalization, runs typed validators before visible delivery, defines bounded retry/refuse/approval outcomes and failure defaults, and emits durable correlated audit events; keep finance, security, freshness, and other domain policies in plugins or ClawHub packages.

## Reproduction Assessment

Yes at the source level. Current main demonstrably has only adjacent revision, pre-tool, dispatch, and payload-cancellation hooks, and the discussion supplies live examples of prompt rules failing; this read-only review did not run a new end-to-end canary.

## Solution Assessment

Unclear as a specific design, but the direction is valid. A mandatory two-model-phase protocol may be broader than necessary, so maintainers should first define the smallest generic ledger-and-validator contract that provides the same fail-closed guarantee across all harness and delivery paths.
