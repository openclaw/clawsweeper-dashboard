## Summary

Keep open: current main has adjacent pre-tool blocking, final-answer revision, reply-dispatch, and outbound cancel hooks, but it still lacks the requested fail-closed response gate with mandatory tool-call accounting, typed validators, two-phase planning, fallback semantics, and audit records. The linked APort Guardrails, openclaw-governance, and claude-rule-enforcer comments are useful complementary plugin/workaround evidence, but they do not remove the core/plugin API gap.

## What This Changes

Request for a core/plugin pre-response enforcement layer that blocks final output until configured mandatory tools and validators have run and passed.

## Best Possible Solution

Keep this open as the canonical response-gate design item, with domain policy packs living in plugins once core provides a versioned mandatory-tool ledger, typed validator API, fail-closed final-response gate, bounded retry/fallback behavior, and audit records.

## Reproduction Assessment

Yes for the source-level gap: current main exposes adjacent hooks, but the inspected contracts lack a fail-closed responseGate, mandatory-tool ledger, typed validators, and audit/fallback semantics. No live end-to-end repro was run during this read-only review.

## Solution Assessment

Unclear pending maintainer design: the requested two-phase response gate is plausible and fits the missing seam, but the best shape needs security/product review around API versioning, fail-closed behavior, plugin permissions, and audit storage.
