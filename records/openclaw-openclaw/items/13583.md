## Summary

Keep open. Current main has adjacent pre-tool blocking, final-answer revision, reply-dispatch, and outbound message-cancel hooks, but it still lacks the requested fail-closed response gate with mandatory tool-call accounting, typed validators, two-phase planning, bounded retry/fallback behavior, and audit records.

## What This Changes

Request a pre-response enforcement layer that blocks final answer delivery until configured mandatory tools and validators have run and passed.

## Best Possible Solution

Keep this as the canonical response-gate policy-as-code design item, with a versioned core/plugin API for mandatory tool-call accounting, typed validators, fail-closed final-response gating, bounded retry/fallback behavior, and audit records.

## Reproduction Assessment

No. This is a feature request rather than a current contract bug; code inspection confirms the missing response-gate surface, but there is no deterministic failing current-main prompt-rule reproduction or acceptance test.

## Solution Assessment

Yes. Keeping this open for explicit maintainer product/security design is the narrowest maintainable path; treating before_tool_call, before_agent_finalize, reply_dispatch, or message_sending as the requested hard gate would leave core policy semantics unspecified.
