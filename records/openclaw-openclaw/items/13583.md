## Summary

Keep open: current main still has adjacent pre-tool, finalization-revision, and outbound message-cancel hooks, but it does not implement the requested fail-closed response gate with mandatory-tool accounting, typed validators, two-phase planning, fallback semantics, or audit records.

## What This Changes

The issue requests a fail-closed pre-response enforcement layer that can block final answers until mandatory tool calls and policy validators have run and passed.

## Best Possible Solution

Keep this as the canonical response-gate design item: domain policies should live in plugins or ClawHub packages, while core provides a versioned plugin/runtime seam for mandatory-tool ledgers, typed validators, fail-closed finalization, bounded retry/fallback behavior, and audit records.

## Reproduction Assessment

Yes, at source level: current main lacks a responseGate config/API and the closest finalization hook continues on missing or failed hooks. I did not run a live regulated-workflow reproduction in this read-only review.

## Solution Assessment

No one-shot code patch is the best fix yet. The maintainable solution is a maintainer-approved versioned core/plugin API that composes with existing pre-tool approval and outbound message hooks instead of duplicating them.
