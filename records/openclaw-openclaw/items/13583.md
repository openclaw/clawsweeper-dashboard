## Summary

Keep open. Current main has adjacent pre-tool blocking, final-answer revision, reply-dispatch, and outbound message-cancel hooks, but it still lacks the requested fail-closed response gate with mandatory tool-call accounting, typed validators, two-phase planning, bounded retry/fallback behavior, and audit records.

## What This Changes

The issue requests a fail-closed pre-response enforcement layer that prevents final answers until required tool calls and policy validators run and pass.

## Best Possible Solution

Keep this as the canonical response-gate policy-as-code design item, with a versioned core/plugin API for mandatory tool-call accounting, typed validators, fail-closed final-response gating, bounded retry/fallback behavior, and audit records.

## Reproduction Assessment

No high-confidence repo-local reproduction exists yet. The discussion gives credible production and canary evidence, and code inspection confirms the missing response-gate surface, but there is no deterministic failing test for mandatory prompt-rule bypass and enforcement.

## Solution Assessment

Yes, keeping this open for explicit maintainer design is the best current path. Treating before_tool_call, before_agent_finalize, reply_dispatch, or message_sending as the requested hard response gate would leave mandatory-tool ledgers, validator contracts, fail-closed policy, retry/fallback, and audit semantics unspecified.
