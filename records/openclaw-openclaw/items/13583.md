## Summary

Current main adds a useful bounded `before_agent_finalize` revision hook, including Codex Stop-hook relay support, but it still fails open on hook errors and lacks a typed mandatory-tool ledger, validator contract, deterministic fail-closed fallback, audit record, and coverage across every final-delivery path. The discussion’s plugin implementations are complementary rather than replacements: they demonstrate tool-side and output-filter policy layers while explicitly identifying the missing core response-gate seam.

## What This Changes

Add a mechanically enforced pre-response policy layer that can require specified tool calls and validators before any final response or external action is delivered.

## Best Possible Solution

Define a versioned, harness-neutral core/plugin contract that records response/action obligations during a run, validates them at one canonical pre-delivery boundary, fails closed according to explicit operator policy, emits auditable outcomes, and leaves domain-specific rules in plugins or ClawHub packages.

## Reproduction Assessment

Yes at the source level: current main’s closest finalization hook explicitly continues when absent or failing and has no mandatory-tool or validator state, so the requested hard guarantee cannot be configured today. The discussion also includes multiple real deployment examples, though no single repository test reproduces the full proposed two-phase workflow.

## Solution Assessment

The requested outcome is valid, but the proposed YAML and fixed two-phase protocol are not yet proven to be the best permanent API. The best direction is a smaller canonical obligation-and-verdict contract at the shared final-delivery boundary, with planning strategy and domain validators owned by plugins.

## Risks / Open Questions

- A fail-closed response gate changes availability and operator expectations: validator outages, timeouts, or plugin errors need an explicit refusal, approval, or degradation policy rather than silently accepting or indefinitely retrying.

- Mandatory-tool enforcement can create a second policy source alongside existing tool approvals and Stop hooks unless the design establishes one canonical obligation ledger and audit owner.
