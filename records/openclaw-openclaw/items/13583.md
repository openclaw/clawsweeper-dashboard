## Summary

Keep open: current main and v2026.6.6 still provide only adjacent pre-tool, final-answer revision, message-cancel, and HTTP client-tool-choice mechanisms, not the requested fail-closed responseGate with mandatory tool accounting, typed validators, two-phase planning, bounded fallback, and audit records.

## What This Changes

The issue requests a pre-response enforcement layer that blocks final output until mandatory tool calls and configured validators have run and passed.

## Best Possible Solution

Keep this as the canonical response-gate design item and define a versioned core/plugin API for mandatory tool ledgers, typed validators, fail-closed final-response authorization, bounded retry/fallback behavior, and audit records, with domain policy packs outside core.

## Reproduction Assessment

Yes, at source level rather than as a narrow live bug: current main lacks a responseGate config/API and the closest finalization hook is fail-open revision-only behavior. The issue discussion adds real-world examples, but the reviewed proof is source-based.

## Solution Assessment

No current mechanism is the full solution: pre-tool hooks, message_sending cancellation, and HTTP tool_choice enforcement are useful adjacent primitives, but none provide a turn-level fail-closed validator and mandatory-tool gate. The best path is a designed core/plugin seam rather than one-off domain policy in core.
