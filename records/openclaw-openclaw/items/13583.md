## Summary

Current main still lacks a fail-closed contract that accounts for mandatory tool completion and validator results before final delivery or an exact-payload external action. Existing pre-tool, final-answer revision, dispatch, and send hooks are complementary but do not solve the central enforcement gap, and the live discussion adds recent evidence that prompt-only approval rules remain bypassable.

## What This Changes

Requests mechanically enforced pre-response gates that require configured tools and validators to pass before a final answer or external action can be delivered.

## Best Possible Solution

Define a versioned, runtime-neutral core/plugin response-gate API that carries a canonical tool/action ledger into typed validators, binds approvals to the exact outbound payload, supports bounded retry and explicit fail-closed outcomes, and leaves domain-specific policy packs in external plugins.

## Reproduction Assessment

Yes, at source level with high confidence: current main has no contract that proves configured tools and validators passed before final delivery, and the discussion includes multiple live examples of prompt-level rules being skipped. The exact proposed YAML scenario was not executed locally because this is a broad missing capability rather than a single deterministic bug.

## Solution Assessment

Unclear for the exact two-phase design; it is a strong starting point, but the best solution is a smaller runtime-neutral ledger-and-validator contract that covers all final delivery and external-action paths without duplicating agent loops. Domain rules should remain plugin-owned while core supplies the missing enforceable seam.
