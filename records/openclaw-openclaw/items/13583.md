## Summary

Keep open. Current main has adjacent pre-tool blocking, final-answer revision, reply-dispatch, and outbound message-cancel hooks, but it still lacks the requested fail-closed response gate with mandatory tool-call accounting, typed validators, two-phase planning, retry/fallback policy, and audit records.

## What This Changes

Add a configurable pre-response enforcement layer that prevents final output until mandatory tools and validators have run and passed.

## Best Possible Solution

Keep this as the canonical response-gate policy-as-code design item and define a versioned core/plugin API for mandatory tool ledgers, typed response validators, fail-closed delivery gating, bounded retry/fallback behavior, and audit records.

## Reproduction Assessment

Not applicable. This is a feature/API request; source and docs inspection verify the requested response-gate surface is absent, but there is no existing current-main contract that should fail.

## Solution Assessment

Yes for the current triage direction. Keeping this open for a versioned core/plugin API design is the narrowest maintainable path; treating before_tool_call, before_agent_finalize, reply_dispatch, or message_sending as the requested hard gate would leave fail-closed response semantics unspecified.
