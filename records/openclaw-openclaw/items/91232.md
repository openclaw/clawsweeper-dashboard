## Summary

Keep open. Current main has repaired the related commitment-store write-loss class and now records `sentAtMs`, but commitment delivery still does not persist Telegram message ID/API date back to the commitment record, so the reported missing delivery-confirmation problem remains actionable.

## What This Changes

The issue reports that inferred commitment check-ins are marked sent for Telegram without a delivered chat message or Telegram response timestamp.

## Best Possible Solution

Persist a real durable delivery receipt for commitment check-ins, including Telegram message id and API send time when available, and add a diagnostic or repair path for existing `sent` rows that lack confirmation.

## Reproduction Assessment

Partially yes: source inspection shows current main still lacks commitment-level Telegram receipt persistence. The stronger claim that current main sends nothing to Telegram needs a live Telegram repro or production logs from the current build.

## Solution Assessment

No: current main's local `sentAtMs` timestamp is not the best fix for delivery confirmation because it is not the Telegram API response date or message id. The maintainable fix is to carry the durable send receipt into the commitment record and avoid `sent` state without confirmed delivery identity.
