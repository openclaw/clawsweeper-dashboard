## Label Justifications

- `P1`: The PR targets a reported long-running gateway FD/RSS growth problem and also introduces a merge-blocking availability race in that default gateway path.

## Summary

Keep open. Current main still lacks long-running gateway idle eviction for cached memory index managers, and the prior CLI shutdown fix is complementary, but this PR still needs a race fix and exact v2 real-behavior proof before it is merge-ready.

## What This Changes

The PR adds default gateway-side idle eviction for cached memory index managers, new memory sync idle-eviction config fields, plugin runtime plumbing, in-flight refcount protection, and focused tests.

## Best Possible Solution

Keep the daemon idle-eviction direction, but make eviction lease-safe by revalidating cache identity, freshness, and busy state immediately before close, then add focused race coverage and exact redacted gateway proof for this v2 patch.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the review blocker: a cache hit refreshes lastAccessAt before manager.search acquires the busy ref, while the PR close loop still closes the stale entry it captured earlier. I did not run a live gateway proof in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not yet; idle eviction through the memory plugin runtime is a plausible boundary, but the close loop needs a lease/freshness recheck before it is safe as a default gateway sidecar.
