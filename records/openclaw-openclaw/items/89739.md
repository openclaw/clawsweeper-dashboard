## Summary

Keep open. The latest push improves the earlier bounded-sampling concern by documenting creation-time sampling as best-effort and adding a fire-time floor, but the patch still has a blocker: retryable recurring transient-error retries can return before the new `cron.minInterval` floor is applied, so the new operator guardrail is not enforced for that automatic re-arm path.

## What This Changes

The PR adds an opt-in `cron.minInterval` config option, schema/help/docs, creation/update validation, scheduler fire-time floor logic, and cron regression tests for recurring jobs.

## Best Possible Solution

Land only after maintainers choose one `cron.minInterval` contract and align code, docs, help text, and tests; the cleaner endpoint is one runtime floor for every automatic recurring re-arm, with only explicit exceptions for one-shot and manual force runs.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the retryable recurring-error path schedules `result.endedAt + retryDecision.backoffMs` and returns before the new floor logic. A recurring job with `cron.minInterval: "5m"`, a retryable transient error, and `cron.retry.backoffMs: [1000]` would re-fire inside the configured floor unless the code changes.

## Solution Assessment

No; the current patch is a good direction but not yet the best final shape because the advertised runtime floor and the retryable transient-error branch disagree. The best fix is to either floor retryable recurring retries too or explicitly make that exception the documented product contract.
