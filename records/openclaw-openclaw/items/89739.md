## Summary

Keep open: this maintainer-labeled draft adds a useful centralized cron guardrail, but the proposed cron-expression check only samples six future fires, so it can miss later Croner day-of-month/day-of-week OR bursts and fail the advertised minimum-interval guarantee.

## What This Changes

Adds an opt-in `cron.minInterval` config setting with schema/docs/help wiring and service/gateway enforcement for recurring `every` and `cron` job creation or schedule edits.

## Best Possible Solution

Keep enforcement in the cron service, but make cron-expression validation match the documented minimum-interval contract or explicitly document and test a bounded sampling contract before merging the config surface.

## Reproduction Assessment

Yes for the PR defect: the source shows only six sampled fires, and a Croner 10.0.1 probe shows a later 1-day burst that the first six fires would miss. This is not a current-main user bug because the config option is new in this PR.

## Solution Assessment

No: central service enforcement is the right layer, but a fixed six-fire sample is not the best way to implement an advertised minimum interval for cron expressions. The safer path is exact or clearly bounded semantics with regression coverage for later irregular bursts.
