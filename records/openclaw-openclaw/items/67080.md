- `P1`: The PR changes gateway plugin HTTP dispatch in a way that can affect live route delivery, auth gating, and process availability if merged.

## Summary

Keep open. This PR still targets a missing `activation.onRoutes` runtime consumer, but the current branch is conflicting and the proposed request-time active plugin load has security, availability, and auth-ordering blockers; the member-authored, `maintainer`-labeled PR also needs explicit maintainer handling.

## What This Changes

The branch adds manifest route hints for bundled HTTP-route plugins, route-id normalization, scoped route-owner planning, gateway fallback dispatch, docs, and seam tests.

## Best Possible Solution

Prepare scoped route-owner registries under gateway lifecycle as stable non-activating snapshots, share the same snapshot with auth and dispatch, isolate owner failures, then refresh the branch against current main.

## Reproduction Assessment

Yes, source-reproducible. PR head wires `resolveScopedRegistry` from gateway plugin dispatch to `loadOpenClawPlugins`, while gateway auth is computed in an earlier stage before plugin dispatch; I did not run live gateway traffic in this read-only review.

## Solution Assessment

No. The manifest hint idea is useful, but active plugin loading from a route miss is not the narrowest maintainable fix; a lifecycle-owned, non-activating scoped snapshot shared by auth and dispatch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: A startup-registry miss can now call `loadOpenClawPlugins` from the HTTP request path. Because this load is not `activate: false`, it can import plugin modules, clear/re-register runtime state, and swap the active registry in response to network traffic; move this to a gateway lifecycle-owned non-activating snapshot before dispatch reads it.
  - confidence: 0.9
- **[P1] Run plugin auth against the scoped route set:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped registry is only resolved after the earlier `plugin-auth` stage decided whether to authenticate using the startup registry. Scoped-only routes with `auth: "gateway"` can therefore be matched without a populated gateway auth context and fail closed even with valid credentials; auth and dispatch need to consult the same route snapshot.
  - confidence: 0.86
- **[P2] Isolate scoped owner failures before route matching:** `src/plugins/runtime/http-route-registry-loader.ts:32-33`
  - body: Using `throwOnLoadError: true` means one broken `gateway-plugin-http` owner can throw the whole scoped load before a healthy owner route is matched. Since the plugin HTTP stage is `continueOnError`, that skips all scoped plugin routes for the request; load failures need to be isolated per owner or converted into a registry the matcher can partially use.
  - confidence: 0.84
