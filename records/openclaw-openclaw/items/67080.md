No. Manifest route-owner hints are a plausible direction, but active plugin loading in HTTP request dispatch is the wrong lifecycle boundary; a gateway-owned non-activating snapshot shared with auth is the safer fix shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `loadScopedGatewayPluginHttpRouteRegistry` is reached after an HTTP route miss and then calls `loadOpenClawPlugins` without `activate:false`. Loader semantics make that an active load, so a remote request can mutate global plugin runtime state; move this to gateway startup/reload or a gateway-owned non-activating cached snapshot.
  - confidence: 0.9
- **[P1] Run plugin auth against the scoped route set:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped lookup happens inside plugin dispatch, after the `plugin-auth` stage has already decided whether to authorize using only the active registry. A scoped-only `auth: "gateway"` route can be rejected even when the request supplied valid credentials; auth gating needs the same route set that dispatch uses.
  - confidence: 0.86
- **[P2] Isolate scoped owner failures before route matching:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: Using `throwOnLoadError: true` for the whole `gateway-plugin-http` owner set means one broken route-owner plugin throws before route matching can select a healthy scoped route. The plugin HTTP stage is `continueOnError`, so the request skips all scoped plugin routes instead of serving the healthy one.
  - confidence: 0.82
