## Summary

Keep open: this maintainer-labeled PR is not implemented on current main, but the patch is not merge-ready because it adds request-path plugin runtime loading, lets auth and dispatch see different route sets, and lets one scoped owner load failure block healthy scoped routes.

## What This Changes

The PR adds manifest `activation.onRoutes` normalization, bundled route-owner hints, scoped gateway plugin HTTP route planning/loading, docs, and seam tests.

## Best Possible Solution

Land a refreshed route-owner lifecycle that resolves or memoizes scoped route owners outside hot request dispatch, feeds the same registry to auth and dispatch, isolates owner load failures, and updates docs/tests around that final contract.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch calls `loadOpenClawPlugins` from the request fallback, route auth runs before scoped fallback matching, and the scoped loader uses `throwOnLoadError: true`. I did not run tests because this review is read-only.

## Solution Assessment

No: this is a plausible manifest-route consumer, but not the best merge shape because it performs scoped runtime loading from dispatch instead of using a lifecycle-owned route registry shared by auth and dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Move scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This loader is called from the plugin HTTP fallback after a request misses the active registry, but it calls `loadOpenClawPlugins` without `activate: false`. The loader’s default activation path clears/restores runtime state and installs an active registry, so an unmatched network request can execute plugin loading and mutate global plugin state on the gateway hot path.
  - confidence: 0.91
- **[P1] Authenticate against the scoped route set before dispatch:** `src/gateway/server/plugins-http.ts:85-90`
  - body: Scoped routes are resolved only inside dispatch after `plugin-auth` has already decided whether the path needs gateway auth from the startup/current registry. A scoped-only route with `auth: "gateway"` on an otherwise unprotected path will therefore match later with `gatewayAuthSatisfied: false` and be rejected even when the caller sent valid gateway credentials.
  - confidence: 0.88
- **[P2] Isolate failures from other scoped route owners:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: Using `throwOnLoadError: true` for the whole scoped owner set makes one broken or unconfigured route-owner plugin throw the fallback load. Because plugin HTTP is a `continueOnError` stage, that skips the entire plugin-route stage for the request and prevents healthy scoped route owners from matching.
  - confidence: 0.86
