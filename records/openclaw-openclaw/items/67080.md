## Summary

Keep open: the route-owner planning idea is useful, but this protected maintainer PR still has blocking gateway lifecycle and security issues around request-time plugin loading, auth/dispatch route-set mismatch, and current-main route-registry conflicts.

## What This Changes

The PR adds manifest `activation.onRoutes` normalization and route-owner planning, updates bundled plugin manifests/docs, and adds a gateway plugin HTTP fallback that loads a scoped registry after startup-registry misses.

## Best Possible Solution

Land a refreshed route-owner lifecycle that resolves or memoizes scoped route owners outside hot request dispatch, feeds the same registry to auth and dispatch, isolates owner failures, and rebases onto current main’s route-registry callback.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch calls `loadOpenClawPlugins` from the request fallback, route auth runs before scoped fallback matching, and the loader uses `throwOnLoadError: true`. I did not run tests because this review is read-only.

## Solution Assessment

No. Manifest route planning is useful, but request-time activating loads are the wrong layer for gateway hot-path routing; a gateway-owned startup/reload or non-activating cached registry should feed both auth and dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `createGatewayPluginRequestHandler` reaches this loader from the HTTP miss path, and the loader calls `loadOpenClawPlugins` without `activate: false`. Because plugin loading activates by default, an unmatched network request can import plugin runtime code, mutate global plugin state, and spend disk/module-loading work on the gateway hot path.
  - confidence: 0.92
- **[P1] Authenticate against the same scoped route set:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped registry is resolved only after the earlier `plugin-auth` stage has decided whether the path needs gateway auth from the startup registry. A scoped-only `auth: "gateway"` route outside protected paths can be matched here with no populated auth context, so valid gateway credentials are ineffective for that route class.
  - confidence: 0.89
- **[P2] Isolate scoped owner load failures:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: With `throwOnLoadError: true`, one broken plugin in the broad `gateway-plugin-http` owner set throws the entire scoped load. Because plugin HTTP is a continue-on-error stage, that skips healthy scoped route owners for the request instead of matching routes that could have loaded.
  - confidence: 0.86
