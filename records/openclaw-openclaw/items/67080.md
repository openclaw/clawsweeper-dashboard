## Label Justifications

- `P1`: The PR changes gateway request-time plugin loading in a way that can affect real route availability and runtime security boundaries.
- `merge-risk: 🚨 security-boundary`: The diff adds a path where remote HTTP requests can cause route-owner plugin code to load and activate outside an explicit startup/reload action.

- `merge-risk: 🚨 compatibility`: The PR changes the meaning of `activation.onRoutes` and plugin HTTP route activation, which is a plugin contract surface for bundled and external plugin authors.

## Summary

Keep open: the route-activation goal is still useful and is not implemented on current main, but this PR needs maintainer repair before merge because it adds request-triggered activating plugin loads in a gateway hot path and is now conflicting with current main.

## What This Changes

The PR adds `activation.onRoutes` manifest hints for bundled route-capable plugins and wires gateway plugin HTTP misses to a scoped manifest-planned route registry fallback.

## Best Possible Solution

Keep the route-activation goal, but move scoped route registry ownership into a gateway lifecycle path or a non-activating memoized snapshot, make auth decisions against the same candidate registry used for dispatch, and rebase over current main's dynamic route-registry/request-context code.

## Reproduction Assessment

No live reproduction was run in this read-only review, but the PR-introduced risks are source-reproducible: request dispatch calls the scoped registry resolver, and that resolver calls the activating plugin loader by default.

## Solution Assessment

No. This is a plausible direction, but the best fix should make scoped route registry loading lifecycle-owned or non-activating and cached, then make auth and dispatch share the same route candidates after rebasing over current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped route loads out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This fallback runs from the HTTP request path and calls `loadOpenClawPlugins` without `activate: false`. The loader's default activation path clears/restores global plugin runtime state and installs the loaded registry, so an unmatched remote request can mutate process-wide plugin state and execute route-owner plugin code outside a controlled startup/reload lifecycle. The scoped route registry needs to be loaded ahead of dispatch or as a gateway-owned non-activating memoized snapshot.
  - confidence: 0.92
- **[P1] Do not fail all scoped routes on one owner error:** `src/plugins/runtime/http-route-registry-loader.ts:32`
  - body: `throwOnLoadError: true` makes any failing plugin in the scoped `gateway-plugin-http` owner set throw before route matching can choose a healthy route. Because plugin HTTP is a `continueOnError` stage, that exception skips the whole plugin-route stage for the request.
  - confidence: 0.88
- **[P2] Run auth against scoped route candidates:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped registry is resolved only after the earlier `plugin-auth` stage has decided whether to authenticate using the startup registry. For scoped-only routes with `auth: "gateway"` outside the hard-coded protected prefixes, valid gateway credentials are never evaluated and dispatch then rejects the route because `gatewayAuthSatisfied` is false. Auth gating and scoped resolution need to share the same matched route set.
  - confidence: 0.82
