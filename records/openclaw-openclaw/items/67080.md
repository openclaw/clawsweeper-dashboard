## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This loader is reached from `createGatewayPluginRequestHandler` when a request misses the startup registry, but it calls `loadOpenClawPlugins` without `activate:false`. Since the loader defaults to active runtime loading, a remote miss can trigger plugin imports and global registry mutation; prepare a gateway-owned non-activating route snapshot before dispatch instead.
  - confidence: 0.93
- **[P2] Do not fail all scoped routes on one owner error:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: `routePluginIds` is the coarse `gateway-plugin-http` owner set, but `throwOnLoadError:true` makes one broken owner throw before route matching can select a healthy handler. Isolate owner failures or build a registry that can skip failed non-matching owners so unrelated scoped routes still work.
  - confidence: 0.88
- **[P2] Run auth against scoped route candidates:** `src/gateway/server/plugins-http.ts:85-88`
  - body: The scoped registry is only loaded inside plugin HTTP dispatch, after the `plugin-auth` stage has already decided whether to authenticate from the startup registry. A scoped `auth:"gateway"` route on a non-protected path will therefore be blocked even when the caller provides valid gateway credentials; resolve scoped candidates before auth or have auth query the same prepared registry.
  - confidence: 0.9
