## Summary

Keep open: this protected maintainer PR contains unique manifest route-planning work, but it is not merge-ready because PR head still performs activating plugin loads from HTTP request dispatch, couples all scoped owners behind one load failure, decides auth before scoped candidates are known, and is dirty against current main's dynamic route-registry work.

## What This Changes

The PR adds manifest-based activation hints for gateway plugin HTTP routes, introduces route-owner planning and scoped plugin registry loading on startup-registry misses, updates bundled plugin manifests and docs, and adds seam tests for the fallback path.

## Best Possible Solution

Keep the manifest route-owner concept only after rebasing it into a gateway-owned prepared, non-activating route-registry snapshot with auth evaluated against prepared candidates and failure isolation between owners.

## Reproduction Assessment

Yes for the patch defects: source inspection of PR head shows a startup-registry miss reaches `loadOpenClawPlugins` from request dispatch without `activate:false`, and the gateway auth stage runs before scoped route matching. I did not run live Gateway traffic in this read-only review.

## Solution Assessment

No. The manifest route-owner direction is plausible, but the current branch puts runtime loading in the wrong lifecycle point; the safer fix is a prepared non-activating scoped registry with auth and failure isolation built around that snapshot.
