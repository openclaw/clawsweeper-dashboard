## Best Possible Solution

Keep the route-activation goal, but land it through a rebased design that preloads or memoizes a non-activating gateway-owned scoped registry, shares auth candidate resolution with dispatch, and isolates owner failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep scoped route loads out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This fallback is reached from the HTTP request handler after a startup-registry miss, but it calls `loadOpenClawPlugins` without `activate:false`. The loader activates by default and updates global plugin runtime state, so a remote miss can trigger plugin code loading and registry mutation outside startup/reload; build the scoped registry during a controlled gateway lifecycle step or make it a non-activating gateway-owned cache.
  - confidence: 0.9
- **[P1] Do not fail all scoped routes on one owner error:** `src/plugins/runtime/http-route-registry-loader.ts:32`
  - body: `gateway-plugin-http` is intentionally coarse, so the planned owner set can contain several plugins. With `throwOnLoadError: true`, one broken or unconfigured owner makes `loadOpenClawPlugins` throw before route matching can select a healthy handler, causing the plugin HTTP stage to be skipped for all scoped fallback routes.
  - confidence: 0.87
- **[P2] Run auth against scoped route candidates:** `src/gateway/server/plugins-http.ts:85-90`
  - body: Scoped routes are only loaded after the earlier `plugin-auth` stage has decided whether auth is needed from the startup registry. A scoped fallback route with `auth: "gateway"` on an otherwise unprotected path will therefore be matched only after auth was skipped, and dispatch will reject it even when the caller sent valid gateway credentials.
  - confidence: 0.82
