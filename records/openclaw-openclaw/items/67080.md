## Label Justifications

- `P1`: The PR changes gateway request-time plugin loading in a way that can affect real route availability and runtime security boundaries.
- `merge-risk: 🚨 compatibility`: The diff turns `activation.onRoutes` into a live plugin HTTP loading contract for bundled and future external route owners.
- `merge-risk: 🚨 security-boundary`: The diff lets remote HTTP request misses reach plugin runtime loading and activation outside an explicit startup or reload action.
- `merge-risk: 🚨 availability`: `throwOnLoadError: true` on the scoped owner set can make one broken route-owner plugin block all scoped fallback routes.

## Summary

Keep open: the route-activation goal is still useful, but the PR head has source-proven lifecycle, security, and availability blockers around request-time plugin loading, and it is currently conflicting with main on the same gateway/plugin route path.

## What This Changes

The PR adds manifest `activation.onRoutes` hints to route-capable bundled plugins, route-id normalization/planning, tests/docs, and a gateway HTTP fallback that loads a scoped plugin registry on startup-registry misses.

## Best Possible Solution

Keep the route-activation goal, but move scoped route registry ownership into a gateway lifecycle path or non-activating memoized snapshot, make auth and dispatch use the same candidate registry, and rebase over current main's dynamic route-registry/request-context code.

## Reproduction Assessment

No live reproduction was run in this read-only review, but the risky behavior is source-reproducible: PR head calls the scoped registry resolver from plugin HTTP dispatch, and that resolver calls the activating plugin loader by default.

## Solution Assessment

No. The PR is a plausible direction, but the best fix should make scoped route registry loading lifecycle-owned or non-activating and cached, then make auth and dispatch share the same route candidates after rebasing over current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep scoped route loads out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This fallback is called after an HTTP route miss and then calls `loadOpenClawPlugins` without `activate: false`; the loader activates by default and mutates the active plugin registry/global runtime state. Load scoped route registries during a gateway-owned lifecycle, or use a non-activating memoized snapshot before dispatch.
  - confidence: 0.92
- **[P1] Do not fail all scoped routes on one owner error:** `src/plugins/runtime/http-route-registry-loader.ts:32`
  - body: `throwOnLoadError: true` means any plugin in the scoped route-owner set can throw before route matching chooses a handler. Since plugin HTTP is a continue-on-error stage, one broken owner can skip the whole plugin route stage and make unrelated healthy scoped routes unavailable.
  - confidence: 0.88
- **[P2] Run auth against scoped route candidates:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped registry is discovered only inside plugin dispatch, after `plugin-auth` has already decided whether to authenticate using the startup registry. Scoped-only routes with `auth: "gateway"` can then reject valid callers because `gatewayAuthSatisfied` was never populated; auth and dispatch need to evaluate the same route candidates.
  - confidence: 0.82
