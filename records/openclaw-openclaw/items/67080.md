## Summary

Keep open: this protected MEMBER PR has a plausible route-owner direction, but the current patch moves plugin runtime loading onto HTTP miss handling and needs maintainer-owned lifecycle, security, auth, and rebase decisions before merge.

## What This Changes

The branch adds manifest `activation.onRoutes` normalization and bundled route hints, then makes gateway plugin HTTP dispatch load a scoped route registry when the startup registry misses.

## Best Possible Solution

Keep the manifest route-owner concept only if it is implemented as a gateway-owned lifecycle snapshot or non-activating cache that preserves server-local route/context dispatch, authenticates after scoped candidates are known, and isolates broken owners.

## Reproduction Assessment

Yes, at source level. PR head shows a startup-registry miss reaches `loadOpenClawPlugins` from request dispatch without `activate:false`, and the auth stage still decides before scoped route candidates are available.

## Solution Assessment

No. Manifest route-owner planning is plausible, but the current implementation puts runtime loading in the wrong lifecycle point; a prepared, non-activating gateway-owned route registry is the safer shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This loader is called from the HTTP fallback path, but it invokes `loadOpenClawPlugins` without `activate:false`. Because loader activation is the default, a remote route miss can import/register plugins and swap active runtime state instead of using a gateway-owned prepared route snapshot.
  - confidence: 0.93
- **[P2] Do not let one owner failure block all scoped routes:** `src/plugins/runtime/http-route-registry-loader.ts:32-33`
  - body: With `throwOnLoadError:true`, any failed plugin in the broad `gateway-plugin-http` owner set throws before the gateway can match the requested path. Since plugin HTTP is a `continueOnError` stage, one broken owner can skip healthy scoped route handlers for unrelated paths.
  - confidence: 0.88
- **[P2] Run auth after scoped route candidates are known:** `src/gateway/server/plugins-http.ts:85-90`
  - body: Scoped matches are discovered only inside plugin HTTP handling, after the `plugin-auth` stage has already decided whether to authenticate from the startup registry. A scoped `auth:"gateway"` route on an otherwise unprotected path can therefore reject valid authenticated requests because `gatewayAuthSatisfied` was never populated.
  - confidence: 0.86
