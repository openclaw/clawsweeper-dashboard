No. Manifest route-owner planning is plausible, but this implementation is not the best fix because it puts runtime loading in a network request path and lets auth and dispatch observe different route sets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This call can reach `loadOpenClawPlugins` after an unmatched HTTP request, and the loader activates by default when `activate` is not false. A network request should not import plugin runtime code, mutate global plugin state, or spend disk/module-loading work on the gateway hot path.
  - confidence: 0.92
- **[P1] Authenticate against the same scoped route set:** `src/gateway/server/plugins-http.ts:85-90`
  - body: Scoped routes are discovered only after the earlier `plugin-auth` stage decides whether this path needs gateway auth from the startup/current registry. A scoped-only `auth: "gateway"` route can therefore be matched with no populated auth context, making valid gateway credentials ineffective for that route class.
  - confidence: 0.89
- **[P2] Isolate scoped owner load failures:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: With `throwOnLoadError: true`, one broken plugin in the broad `gateway-plugin-http` owner set throws the entire scoped load. That skips healthy scoped route owners for the request instead of matching routes that could have loaded.
  - confidence: 0.86
