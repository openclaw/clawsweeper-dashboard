## Summary

Keep open: this protected maintainer PR still contains blocking gateway lifecycle, auth, and plugin-load safety problems, and current main has not implemented the requested manifest-scoped gateway route activation. The later normalization/cache cleanup addressed Greptile's first two comments, but the Aisle request-time activation concern and the Codex auth/failure-isolation concerns still apply.

## What This Changes

The PR adds manifest `activation.onRoutes` hints for bundled route-owning plugins, route-id canonicalization, route-owner filtering, and a gateway plugin HTTP fallback that loads a scoped route registry when the startup registry misses.

## Best Possible Solution

Land a refreshed gateway-owned route lifecycle that resolves or memoizes manifest route owners outside hot request dispatch, uses the same scoped route set for auth and dispatch, isolates failed owners, and fits current main's server-local route-registry callbacks.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch calls loadOpenClawPlugins from the request fallback, route auth runs before scoped fallback matching, and the loader uses throwOnLoadError=true. I did not run tests because this review is read-only.

## Solution Assessment

No; request-time scoped plugin loading is the wrong layer for a gateway hot path. The safer solution is a gateway-owned startup/reload or non-activating cached route-owner lifecycle shared by auth and dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This loader is reached from the HTTP request fallback, and loadOpenClawPlugins activates by default unless activate:false is passed. An unmatched network request can therefore import/register route-owner plugins and churn global runtime state; route-owner planning should happen at gateway startup/reload or inside a non-activating gateway-owned cache.
  - confidence: 0.9
- **[P1] Run plugin auth against the scoped route set:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The plugin-auth stage has already decided whether to authenticate before this scoped fallback discovers additional routes. A scoped-only route with auth:"gateway" on a non-protected path will be matched only after credentials were ignored, so valid gateway credentials are ineffective for that route class.
  - confidence: 0.86
- **[P2] Isolate broken scoped owners before route matching:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: Using throwOnLoadError:true for the coarse gateway-plugin-http owner set lets one broken or unconfigured owner throw the whole scoped load. Because plugin HTTP is a continue-on-error stage, that prevents unrelated healthy scoped routes from matching after a startup-registry miss.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Needs attention because the branch moves plugin runtime loading and activation onto a network request fallback path and makes auth depend on a route set discovered later.

  - body: The scoped route loader calls loadOpenClawPlugins from the gateway request fallback without activate:false, so a remote miss can cause module import, registration, active-registry changes, and expensive CPU/IO work.
