## Label Justifications

- `P1`: The PR changes gateway plugin HTTP dispatch in a way that can break route availability and security-boundary assumptions for real gateway users.
- `merge-risk: 🚨 compatibility`: The PR changes plugin route activation and fallback behavior, which can affect existing plugin/gateway route setups during upgrade.
- `merge-risk: 🚨 security-boundary`: The diff allows network request misses to reach plugin runtime loading and active registry mutation.
- `merge-risk: 🚨 availability`: Request-time loading and `throwOnLoadError: true` can stall or skip plugin HTTP routing when route-owner plugins are slow or broken.

## Summary

Keep open: current main does not implement the manifest-driven `gateway-plugin-http` route consumer, but this protected maintainer PR is not merge-ready because the patch moves plugin runtime loading into HTTP request dispatch, has scoped-route failure/auth ordering defects, and is now conflicting with main.

## What This Changes

The PR adds manifest `activation.onRoutes` hints for bundled route-owning plugins, route-id normalization, scoped route-owner planning, gateway fallback route loading, docs, and focused tests for gateway plugin HTTP routing.

## Best Possible Solution

Land the concept only after scoped route registries are prepared under gateway lifecycle, non-activating, failure-isolated, conflict-refreshed against current main, and used by the same auth decision path as dispatch.

## Reproduction Assessment

Yes. Source inspection of the PR head shows a startup-registry miss reaches `loadOpenClawPlugins` from request dispatch and auth is decided before scoped route candidates are known; no live gateway run was performed in this read-only review.

## Solution Assessment

No. Manifest route-owner hints are plausible, but request-time activating registry loads are the wrong lifecycle point; the safer fix is a gateway-owned prepared route-registry snapshot with auth and dispatch using the same candidate set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: A startup-registry miss calls this scoped loader from HTTP request dispatch, and the loader calls `loadOpenClawPlugins` without `activate: false`. Because plugin loading activates global registry state by default, an external HTTP miss can load plugin modules and mutate active plugin state on the request path.
  - confidence: 0.96
- **[P2] Do not abort scoped routes on one owner failure:** `src/plugins/runtime/http-route-registry-loader.ts:32-33`
  - body: With `throwOnLoadError: true`, one malformed or unconfigured route-owner plugin makes the scoped load throw before route matching chooses a handler. Since plugin HTTP is a continue-on-error stage, healthy scoped routes are skipped for that request.
  - confidence: 0.91
- **[P1] Run auth after scoped route candidates are known:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The gateway auth stage decides whether to authenticate using the startup registry before this fallback discovers scoped route candidates. A scoped route with `auth: "gateway"` on an otherwise unprotected path will be matched here with `gatewayAuthSatisfied` still false, so valid credentials never reach the route.
  - confidence: 0.93
