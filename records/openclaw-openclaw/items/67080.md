## What This Changes

The PR adds manifest `activation.onRoutes` hints for bundled route-owning plugins, route id normalization, route-owner filtering, and a gateway plugin HTTP fallback that loads a scoped route registry when the startup registry misses.

## Best Possible Solution

Land a refreshed gateway-owned route lifecycle that resolves or memoizes manifest route owners outside hot request dispatch, uses the same scoped route set for auth and dispatch, isolates failed owners, and fits current main's server-local route-registry callbacks.

No; request-time scoped plugin loading is only a plausible fix, while current policy and current main point toward a gateway-owned startup/reload or non-activating cache lifecycle shared by auth and dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This loader is reached from the request fallback and calls `loadOpenClawPlugins` without `activate: false`. Since the loader activates and clears global plugin runtime state by default, an unmatched network request can load modules and mutate runtime state; move this to a startup/reload lifecycle or a gateway-owned non-activating cache.
  - confidence: 0.91
- **[P1] Run plugin auth against the scoped route set:** `src/gateway/server/plugins-http.ts:85-90`
  - body: Scoped route matching happens only after the `plugin-auth` stage has already decided whether auth is required from the startup/current registry. A scoped-only `auth: "gateway"` route on an otherwise unprotected path will skip credential parsing and then fail closed during dispatch even when the caller sends valid credentials.
  - confidence: 0.88
- **[P2] Isolate broken scoped owners before route matching:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: Loading all scoped route owners with `throwOnLoadError: true` means one broken owner in the coarse `gateway-plugin-http` set can throw the whole scoped load. Because the gateway treats plugin-http as a continue-on-error stage, healthy scoped routes are skipped for that request instead of being matched.
  - confidence: 0.84
