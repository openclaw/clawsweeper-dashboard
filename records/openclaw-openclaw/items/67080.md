## Summary

Keep open for maintainer handling: current main does not implement the route-manifest gateway consumer, but the PR is a protected maintainer branch with blocking gateway lifecycle, auth-ordering, availability, and security-boundary issues.

## What This Changes

The PR adds manifest-driven scoped loading for gateway plugin HTTP route owners, route activation hints in bundled plugin manifests, route-id normalization, and gateway/plugin route tests and docs.

## Best Possible Solution

Land the concept only after scoped route registries are prepared under gateway lifecycle, non-activating, failure-isolated, conflict-refreshed against current main, and used by the same auth decision path as dispatch.

## Reproduction Assessment

Yes. Source inspection of the PR head shows a startup-registry miss reaches `loadOpenClawPlugins` from request dispatch, and auth is decided before scoped route candidates are known; no live gateway run was performed in this read-only review.

## Solution Assessment

No. Manifest route-owner hints are plausible, but request-time active registry loads are the wrong lifecycle point; the safer fix is a gateway-owned prepared route-registry snapshot with auth and dispatch using the same candidate set.

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `createGatewayPluginRequestHandler` calls this loader on a startup-registry miss, so any remote unmatched HTTP request can reach `loadOpenClawPlugins`. Because this call does not pass `activate:false`, the plugin loader may clear and re-activate global plugin state from request dispatch instead of using a gateway-lifecycle-owned route snapshot.
  - confidence: 0.91
- **[P1] Run auth after scoped route candidates are known:** `src/gateway/server/plugins-http.ts:85-90`
  - body: Scoped routes are discovered only after the earlier `plugin-auth` stage decided whether gateway auth was required from the startup registry. A route that exists only in this fallback and declares `auth: "gateway"` will reject even valid credentials because `gatewayAuthSatisfied` and runtime auth context were never populated for that candidate set.
  - confidence: 0.88
- **[P2] Isolate failures between scoped route owners:** `src/plugins/runtime/http-route-registry-loader.ts:30-33`
  - body: The scoped owner set is loaded as one registry with `throwOnLoadError:true`, so one broken `gateway-plugin-http` owner throws before route matching can select a healthy owner. Since the plugin HTTP stage is `continueOnError`, that skips all scoped plugin routes for the request instead of falling through to unaffected handlers.
  - confidence: 0.84
