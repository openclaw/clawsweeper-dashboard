## Label Justifications

- `P1`: The PR affects gateway plugin HTTP routing and can introduce request-triggered runtime activation, auth failure, and availability regressions on live gateway paths.
- `merge-risk: 🚨 compatibility`: The branch changes plugin route activation and fallback behavior in a compatibility-sensitive gateway/plugin startup path.

## Summary

Keep open: current main does not implement the central manifest-scoped gateway route loading behavior, but this PR is not merge-ready because it loads and activates plugin runtime from HTTP request dispatch, has scoped-route auth and failure-isolation defects, and is currently conflicting on protected maintainer-owned gateway/plugin surfaces.

## What This Changes

The branch adds manifest-driven gateway plugin HTTP route activation planning, route-id normalization, bundled manifest hints, docs, and tests so a startup-registry miss can try a scoped route registry.

## Best Possible Solution

Keep the manifest route-owner concept, but implement it as a gateway-owned lifecycle snapshot: preload or explicitly reload route-owner registries without global activation, evaluate auth against prepared candidates, and isolate broken owners before merge.

## Reproduction Assessment

Yes. Source inspection of PR head shows a startup-registry miss reaches `loadOpenClawPlugins` from request dispatch without `activate:false`, and the gateway auth stage runs before scoped route matching.

## Solution Assessment

No. The manifest route-owner direction is plausible, but the current branch puts runtime loading in the wrong lifecycle point; the safer fix is a prepared non-activating scoped registry with auth and failure isolation built around that snapshot.
