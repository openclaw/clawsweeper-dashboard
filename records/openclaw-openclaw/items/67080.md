## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `loadScopedGatewayPluginHttpRouteRegistry` is called after a request-time route miss, and this line loads plugin runtime with the default activating behavior. That lets network traffic drive plugin loading/global registry mutation instead of dispatching against a gateway-owned prepared route snapshot.
  - confidence: 0.9
- **[P1] Run auth after scoped route candidates are known:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped registry is discovered only inside plugin dispatch, after the plugin-auth stage has already used the startup registry to decide whether to authenticate. A scoped `auth: "gateway"` route on an unprotected path will get `gatewayAuthSatisfied=false` and be rejected even when the caller sent valid credentials.
  - confidence: 0.86
- **[P2] Isolate failed route owners before matching:** `src/plugins/runtime/http-route-registry-loader.ts:32-33`
  - body: `throwOnLoadError: true` makes any broken plugin in the coarse `gateway-plugin-http` owner set throw before route matching selects a handler. One bad owner can skip all healthy scoped routes for the request, so candidate loading needs per-owner failure isolation or a non-throwing registry plan.
  - confidence: 0.84
