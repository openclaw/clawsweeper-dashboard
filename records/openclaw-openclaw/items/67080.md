## Summary

Keep open: this maintainer-labeled PR still targets a real missing route-manifest consumer, but the current patch is not merge-ready because it moves plugin runtime loading and activation onto the gateway request path and leaves scoped-route auth/failure behavior unresolved.

## What This Changes

The branch adds manifest `activation.onRoutes` hints to bundled route-owning plugins and a gateway fallback that loads a scoped plugin registry when the startup HTTP route registry misses.

## Best Possible Solution

Land a refreshed design that resolves or memoizes manifest route owners under a gateway-owned startup/reload lifecycle, keeps request dispatch non-activating, runs auth and dispatch against the same route set, isolates broken owners, and proves real gateway traffic.

## Solution Assessment

No. Manifest route hints are a plausible direction, but the best fix is a gateway-owned or startup/reload-owned scoped registry lifecycle with auth, failure isolation, and live proof aligned to that lifecycle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This fallback calls `loadOpenClawPlugins` from the HTTP request path. On this branch the loader defaults to activation, restores or sets the active plugin registry, and clears global plugin state before loading, so an unmatched network request can trigger plugin module execution and process-global registry churn. Move this to a gateway-owned startup/reload lifecycle, or use an explicitly non-activating memoized registry with a clear owner.
  - confidence: 0.92
- **[P1] Run plugin auth against the scoped route set:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped route registry is resolved only after `plugin-auth` has already decided whether to authenticate using the startup registry. A scoped-only `auth: "gateway"` route on a non-protected path will not get valid gateway credentials populated, then the handler rejects it as missing auth context. Resolve the route set before the auth stage, or make the auth stage consult the same scoped registry dispatch will use.
  - confidence: 0.89
- **[P2] Isolate broken scoped owners before route matching:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: `throwOnLoadError: true` makes one failing owner in the coarse `gateway-plugin-http` set abort the whole scoped load. Because the gateway stage is `continueOnError`, healthy scoped routes then never get a chance to match. Load owners in an isolated way or keep diagnostics for failed owners while still dispatching healthy route registries.
  - confidence: 0.86
