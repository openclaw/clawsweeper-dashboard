## Summary

Keep open. This maintainer-member PR is protected and still contains concrete P1 route-lifecycle/security defects: a route miss can invoke active plugin runtime loading from HTTP dispatch, scoped route auth is discovered after the auth stage has already run, and a single scoped owner load failure can skip healthy route handlers.

## What This Changes

Adds manifest `activation.onRoutes` route-id canonicalization and bundled plugin route hints, then makes gateway plugin HTTP dispatch try a scoped manifest-planned plugin registry after startup-registry misses, with docs and seam tests.

## Best Possible Solution

Prepare or memoize scoped route registries under gateway lifecycle as non-activating, failure-isolated snapshots, share that same route set with auth and dispatch, then refresh the branch against current main with live or log-backed gateway proof.

## Reproduction Assessment

Yes. Source inspection of the PR head shows a startup-registry miss reaches `loadOpenClawPlugins` from HTTP dispatch, while current-main plugin auth is decided in a separate earlier stage before plugin dispatch; no live gateway traffic was run in this read-only review.

## Solution Assessment

No. Manifest route-owner hints are a plausible direction, but the best fix keeps plugin loading out of request dispatch, shares one scoped route snapshot between auth and dispatch, and isolates owner failures before route matching.

  - body: The scoped registry is only resolved after the earlier `plugin-auth` stage has already decided whether to authenticate using the startup registry.   - confidence: 0.93
- **[P2] Isolate scoped owner failures before route matching:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: Passing `throwOnLoadError: true` makes one broken plugin in the coarse `gateway-plugin-http` owner set throw before route matching can use any healthy scoped routes. Load failures should be isolated or represented in the registry so one bad owner does not block unrelated matching handlers.
