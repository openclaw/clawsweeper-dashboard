## Summary

Keep open: this protected maintainer/MEMBER PR has meaningful unique gateway plugin-route work, but it is not merge-ready because scoped route loading is still request-path runtime activation, auth is decided before scoped route candidates are known, and the live PR is conflicting with current main gateway lifecycle changes.

## What This Changes

The PR adds manifest `activation.onRoutes` hints for gateway plugin HTTP owners, scoped route-owner planning, fallback scoped route loading, alias normalization, docs, and regression tests.

## Best Possible Solution

Prepare and cache scoped route registries under gateway lifecycle or explicit reload, keep them non-activating during HTTP dispatch, isolate route-owner failures, and make auth and dispatch use the same candidate route snapshot.

Yes. Source inspection of PR head shows a startup-registry miss reaches `loadOpenClawPlugins` from request dispatch and auth is decided before scoped route candidates are known; no live gateway run was performed in this read-only review.

## Solution Assessment

No. Manifest route-owner hints are plausible, but request-time activating registry loads are the wrong lifecycle point; the safer fix is a gateway-owned prepared route-registry snapshot with auth and dispatch using the same candidate set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: A startup-registry miss calls this scoped loader from HTTP request dispatch, and the loader calls `loadOpenClawPlugins` without `activate: false`. Because plugin loading activates global registry state by default, an external HTTP miss can load plugin modules and mutate active plugin state on the request path.
  - confidence: 0.96
- **[P2] Do not abort scoped routes on one owner failure:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: With `throwOnLoadError: true`, one malformed or unconfigured route-owner plugin makes the scoped load throw before route matching chooses a handler. Since plugin HTTP is a continue-on-error stage, healthy scoped routes are skipped for that request.
  - confidence: 0.91
- **[P1] Run auth after scoped route candidates are known:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The gateway auth stage decides whether to authenticate using the startup registry before this fallback discovers scoped route candidates. A scoped route with `auth: "gateway"` on an otherwise unprotected path will be matched here with `gatewayAuthSatisfied` still false, so valid credentials never reach the route.
  - confidence: 0.93
