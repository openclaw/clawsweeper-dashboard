## Label Justifications

- `P1`: The PR changes gateway request dispatch and plugin loading in a way that can break real HTTP route/auth workflows and expose security and availability risk.
- `merge-risk: 🚨 compatibility`: The branch changes plugin route activation and fallback behavior, which can alter existing gateway/plugin startup and upgrade behavior.

- `merge-risk: 🚨 availability`: Request-triggered plugin loading and `throwOnLoadError:true` can make route misses or one broken route owner block healthy gateway plugin routes.

## Summary

Keep open: current main still does not implement manifest-scoped gateway plugin route loading, but this protected maintainer PR is not merge-ready because the proposed request-path loader crosses gateway lifecycle, auth, security, availability, and current-main conflict boundaries.

## What This Changes

The PR adds manifest-driven `activation.onRoutes` planning for gateway plugin HTTP routes, canonical route-id normalization, bundled route hints, scoped route registry loading, docs, and seam tests.

## Best Possible Solution

Land the concept only after scoped route registries are prepared under gateway lifecycle, non-activating, failure-isolated, conflict-refreshed against current main, and used by the same auth decision path as dispatch.

## Reproduction Assessment

Yes. Source inspection of the PR head shows a startup-registry miss reaches `loadOpenClawPlugins` from request dispatch, and auth is decided before scoped route candidates are known; no live gateway run was performed in this read-only review.

## Solution Assessment

No. Manifest route-owner hints are plausible, but request-time active registry loads are the wrong lifecycle point; the safer fix is a gateway-owned prepared route-registry snapshot with auth and dispatch using the same candidate set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `createGatewayPluginRequestHandler` can call this loader on a startup-registry miss, so any remote unmatched HTTP request can reach `loadOpenClawPlugins`. Because the call does not pass `activate:false`, the loader may clear and re-activate global plugin state from request dispatch instead of using a gateway-lifecycle-owned route snapshot.
  - confidence: 0.91
- **[P1] Run auth after scoped route candidates are known:** `src/gateway/server/plugins-http.ts:85-90`
  - body: Scoped routes are discovered only after the earlier `plugin-auth` stage decided whether gateway auth was required from the startup registry. A route that exists only in this fallback and declares `auth: "gateway"` can reject even valid credentials because `gatewayAuthSatisfied` and runtime auth context were never populated for that candidate set.
  - confidence: 0.88
- **[P2] Isolate failures between scoped route owners:** `src/plugins/runtime/http-route-registry-loader.ts:30-33`
  - body: The scoped owner set is loaded as one registry with `throwOnLoadError:true`, so one broken `gateway-plugin-http` owner throws before route matching can select a healthy owner. Since the plugin HTTP stage is `continueOnError`, that skips all scoped plugin routes for the request instead of falling through to unaffected handlers.
  - confidence: 0.84
