## Summary

## What This Changes

The PR adds manifest-based route activation planning for gateway plugin HTTP routes, annotates bundled route-owning plugin manifests/docs, and loads a scoped route registry when the startup registry misses.

## Best Possible Solution

Keep the route-activation goal, but land it through a gateway-owned lifecycle/preloaded or non-activating memoized registry, shared auth/dispatch candidate resolution, isolated owner failures, and a current-main rebase.

## Reproduction Assessment

No live runtime reproduction was run in this read-only review, but the PR-introduced risks are source-reproducible: request dispatch calls the scoped resolver, and the resolver calls the activating plugin loader with `throwOnLoadError: true`.

## Solution Assessment

No. This is a plausible direction, but the best fix should make scoped route registry loading lifecycle-owned or non-activating and ensure auth and dispatch evaluate the same candidate routes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep scoped route loads out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `createGatewayPluginRequestHandler` calls this fallback from the HTTP request path after a route miss, and this loader calls `loadOpenClawPlugins` without `activate: false`. The default loader path clears and re-activates global plugin runtime state, so an unmatched remote request can trigger plugin loading and global state churn outside startup/reload lifecycle.
  - confidence: 0.91
- **[P1] Do not fail all scoped routes on one owner error:** `src/plugins/runtime/http-route-registry-loader.ts:32`
  - body: Using `throwOnLoadError: true` means one broken plugin in the scoped `gateway-plugin-http` owner set throws before route matching can choose a healthy handler. Because plugin HTTP is a continue-on-error stage, that skips the whole plugin-route stage for the request instead of letting other scoped routes handle it.
  - confidence: 0.88
- **[P2] Run auth against scoped route candidates:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The gateway auth stage decides whether to authorize using the route registry before plugin HTTP dispatch runs. Routes discovered only by `resolveScopedRegistry` here are therefore invisible to that auth decision, so `auth: "gateway"` scoped routes on non-protected paths can be rejected even with valid credentials.
  - confidence: 0.84
