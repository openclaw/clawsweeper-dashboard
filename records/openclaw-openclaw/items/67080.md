No live reproduction was run in this read-only review, but the risky behavior is source-reproducible: PR head calls the scoped registry resolver from plugin HTTP dispatch, and that resolver calls the activating plugin loader by default.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep scoped route loads out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This fallback runs from the HTTP request path and calls `loadOpenClawPlugins` without `activate: false`. The loader's default activation path clears/restores global plugin runtime state and installs the loaded registry, so an unmatched remote request can mutate process-wide plugin state and execute route-owner plugin code outside a controlled startup/reload lifecycle.
  - confidence: 0.92
- **[P1] Do not fail all scoped routes on one owner error:** `src/plugins/runtime/http-route-registry-loader.ts:32`
  - body: `throwOnLoadError: true` makes any failing plugin in the `gateway-plugin-http` owner set throw before route matching can choose a healthy route. Because plugin HTTP is a `continueOnError` gateway stage, that exception skips the whole plugin-route stage and can make unrelated scoped routes unreachable after a startup-registry miss.
  - confidence: 0.88
- **[P2] Run auth against scoped route candidates:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped registry is resolved only after the earlier `plugin-auth` stage has decided whether to authenticate using the startup registry. For scoped-only routes with `auth: "gateway"` outside the hard-coded protected prefixes, valid gateway credentials are never evaluated and dispatch then rejects the route because `gatewayAuthSatisfied` is false.
  - confidence: 0.82
