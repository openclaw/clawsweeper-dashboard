## Summary

Keep open: the direction is useful, but the branch still puts scoped plugin runtime loading on the HTTP request path and has unresolved security, availability, auth-ordering, and rebase/design concerns. The maintainer label and MEMBER author also make this a human-owner item, not conservative cleanup close material.

## What This Changes

The branch adds manifest route activation hints and a scoped gateway plugin HTTP route-registry fallback that loads route-owning plugins when the pinned startup registry misses.

## Best Possible Solution

Keep the manifest route-owner idea, but make route activation lifecycle-owned: preplan or cache non-activating scoped registries outside request dispatch, isolate owner failures, and run auth against the actual scoped candidates before handlers execute.

## Reproduction Assessment

Yes, for the review blockers: source inspection shows the request fallback reaches `loadOpenClawPlugins` without `activate:false`, and the Gateway auth stage runs before scoped route matching. I did not run a live Gateway repro because this is a read-only review.

## Solution Assessment

No. The branch is a plausible route-owner implementation, but the best fix should move scoped registry preparation out of request dispatch, make it non-activating and failure-isolated, and integrate auth with the final scoped candidate set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `loadScopedGatewayPluginHttpRouteRegistry` is called from the HTTP miss path and then calls the normal active plugin loader. On current main that loader clears activated plugin state and activates the loaded registry, so any remote miss can trigger plugin runtime loading/global registry churn and widen the code-execution/DoS surface. Prepare a non-activating gateway-owned route snapshot outside request dispatch instead.
  - confidence: 0.93
- **[P1] Do not fail all scoped routes on one owner error:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: The scoped loader uses `throwOnLoadError: true` before route matching chooses a handler. Because `gateway-plugin-http` is a coarse owner set, one broken or unconfigured route-owner plugin will throw the whole scoped load and make unrelated healthy scoped routes unavailable after a startup-registry miss.
  - confidence: 0.9
- **[P2] Run auth against scoped route candidates:** `src/gateway/server/plugins-http.ts:85-88`
  - body: Scoped route candidates are only resolved here, after the `plugin-auth` stage has already decided whether to authenticate using the startup registry. A scoped `auth: "gateway"` route on a non-protected path therefore never gets `gatewayAuthSatisfied`, so it is rejected even when the caller sends valid gateway credentials. Resolve scoped candidates before auth or rerun auth for the selected scoped routes.
  - confidence: 0.88
