## Label Justifications

- `P1`: The PR touches gateway request dispatch, plugin loading, and auth behavior in a way that can break real gateway/plugin workflows before merge.
- `merge-risk: 🚨 compatibility`: Merging changes plugin HTTP fallback and activation behavior for existing plugins that rely on startup route registry semantics.

- `merge-risk: 🚨 availability`: Request-triggered plugin loading and `throwOnLoadError: true` can make unrelated or broken route-owner plugins affect live HTTP routing.

## Summary

Keep open for maintainer review: the scoped-route idea is useful, but this patch is not merge-ready because request misses can load and activate plugins, auth is computed from a different route set, and one scoped owner failure can block healthy routes.

## What This Changes

The branch adds manifest route-id normalization and bundled plugin `activation.onRoutes` hints, then makes gateway plugin HTTP dispatch load a scoped route-owner registry after startup-registry misses, with docs and tests.

## Best Possible Solution

Prepare or memoize scoped route registries under gateway lifecycle as non-activating, failure-isolated snapshots, share that same route set with auth and dispatch, then refresh the branch with gateway traffic proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: the PR head route-miss path reaches `loadOpenClawPlugins`, and plugin auth is still computed before scoped route matching. I did not run live gateway traffic in this read-only review.

## Solution Assessment

No. The manifest-route planning direction is plausible, but request-time active plugin loading is the wrong lifecycle boundary for a gateway hot path; a stable non-activating scoped snapshot shared by auth and dispatch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `createGatewayPluginRequestHandler` now resolves the scoped registry after every startup-registry miss, and that resolver reaches `loadOpenClawPlugins` without `activate: false`. Since the loader defaults to active mode, a remote unmatched HTTP request can import plugins, clear/restore global plugin state, and replace the active registry; this needs to move to gateway lifecycle or a non-activating memoized snapshot before dispatch.
  - confidence: 0.93
- **[P1] Run auth against the scoped route set:** `src/gateway/server/plugins-http.ts:85-90`
  - body: Scoped routes are only discovered here after the earlier `plugin-auth` stage has already decided whether to parse gateway credentials from the startup registry. A scoped-only route with `auth: "gateway"` on a non-protected path will then be rejected with `gatewayAuthSatisfied === false` even when the caller supplied valid credentials, so auth and dispatch need to share the same scoped route snapshot.
  - confidence: 0.9
- **[P2] Isolate scoped owner failures before route matching:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: `throwOnLoadError: true` makes one failing plugin in the scoped owner set throw before route matching can try a different healthy scoped route. Because plugin HTTP runs as a fallthrough stage, that exception skips the whole plugin route stage for the request instead of only excluding the broken owner.
  - confidence: 0.86
