## Label Justifications

- `P1`: The PR changes gateway plugin HTTP routing and has blocking security, auth, and availability risks in request handling.
- `merge-risk: 🚨 compatibility`: The branch conflicts with current main's newer server-local plugin route-registry lifecycle and must preserve auth, dispatch, upgrade, and capability behavior on rebase.

## Summary

Keep open: current main still lacks the manifest route activation consumer, but this protected maintainer PR is not merge-ready because it moves scoped plugin loading into gateway request dispatch and leaves auth, dispatch, and failure isolation split across route sets.

## What This Changes

The PR adds manifest-driven scoped gateway plugin HTTP route loading, canonical route activation IDs, bundled `onRoutes` hints, docs, and seam tests.

## Best Possible Solution

Land a refreshed route-owner lifecycle that resolves or memoizes scoped route owners outside hot request dispatch, feeds the same registry to auth and dispatch, isolates owner failures, and rebases onto current main's route-registry callback.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch calls `loadOpenClawPlugins` from request fallback, route auth runs before scoped fallback matching, and the loader uses `throwOnLoadError: true`. I did not run tests because this review is read-only.

## Solution Assessment

No. The manifest route-owner direction is plausible, but the current implementation puts loading in the wrong lifecycle and lets auth and dispatch observe different route sets; the safer shape is a startup, reload, or gateway-owned cached scoped registry shared by auth and dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This call can reach `loadOpenClawPlugins` after an unmatched HTTP request, and the loader activates by default when `activate` is not false. A network request can therefore import plugin runtime code, mutate global plugin state, and spend disk/module-loading work on the gateway hot path.
  - confidence: 0.92
- **[P1] Authenticate against the same scoped route set:** `src/gateway/server/plugins-http.ts:85-90`
  - body: Scoped routes are discovered only after the earlier `plugin-auth` stage has decided whether the path needs gateway auth from the startup/current registry. A scoped-only `auth: "gateway"` route outside protected paths can be matched with no populated auth context, so valid gateway credentials are ineffective for that route class.
  - confidence: 0.89
- **[P2] Isolate scoped owner load failures:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: With `throwOnLoadError: true`, one broken plugin in the broad `gateway-plugin-http` owner set throws the entire scoped load. Because plugin HTTP is a continue-on-error stage, that skips healthy scoped route owners for the request instead of matching routes that could have loaded.
  - confidence: 0.86
