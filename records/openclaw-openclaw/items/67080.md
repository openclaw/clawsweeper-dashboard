## Summary

Keep this PR open, but it is not merge-ready: the manifest-route activation concept is still useful and not implemented on current `main`, but this branch loads plugin runtime from HTTP request dispatch, can mutate active global plugin state, and does not preserve current-main gateway auth/context behavior safely enough for merge.

## What This Changes

The PR adds `gateway-plugin-http` manifest route hints, route-owner planning, scoped gateway plugin HTTP registry loading, docs, and seam tests so gateway route misses can consult manifest-owned route activation metadata.

## Best Possible Solution

Land this concept only after scoped route registries are prepared and cached under gateway lifecycle, non-activating, failure-isolated, and used by the same auth decision path as dispatch.

## Reproduction Assessment

Yes. Source inspection of the PR head shows a startup-registry miss reaches `loadOpenClawPlugins` from HTTP dispatch, and current `main` shows that loader activates global plugin state unless `activate:false` is set.

## Solution Assessment

No. Manifest route-owner hints are plausible, but request-time activating registry loads are the wrong lifecycle point; the safer fix is a gateway-lifecycle-owned scoped route registry used by both auth and dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This loader is reached from the HTTP route-miss path and calls `loadOpenClawPlugins` without `activate:false`. On current `main`, that default activation clears and re-registers global plugin runtime state, sets the active registry, and initializes global hooks, so an unmatched remote request can trigger plugin code execution and registry churn instead of dispatching against a gateway-owned snapshot.
  - confidence: 0.93
- **[P2] Do not abort scoped routes on one owner failure:** `src/plugins/runtime/http-route-registry-loader.ts:32-33`
  - body: Using `throwOnLoadError: true` for the whole scoped owner set means one broken or unconfigured `gateway-plugin-http` owner throws before route matching can select a healthy handler. Since the gateway runs plugin HTTP as a `continueOnError` stage, that can make unrelated scoped routes disappear for the request.
  - confidence: 0.88
- **[P1] Run auth after scoped route candidates are known:** `src/gateway/server/plugins-http.ts:96-99`
  - body: Gateway auth preflight runs before this handler resolves scoped route candidates. A scoped route with `auth: "gateway"` on a path that the startup registry does not know about will be matched here only after `gatewayAuthSatisfied` was left false, so it fails closed even when the caller supplied valid gateway credentials.
  - confidence: 0.87
