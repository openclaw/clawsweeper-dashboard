## Summary

Keep open: current main still lacks the PR's live `activation.onRoutes` route-consumer behavior, but this maintainer-labeled PR is not merge-ready because the implementation puts active plugin runtime loading in HTTP request dispatch, leaves auth and scoped dispatch on different route snapshots, and is currently conflicting with main.

## What This Changes

The branch adds manifest `activation.onRoutes` route hints and normalization, updates bundled route-owning plugin manifests, and adds a gateway plugin HTTP fallback that loads a scoped route registry after startup-registry misses.

## Best Possible Solution

Prepare or memoize scoped route registries under gateway lifecycle with non-activating, failure-isolated snapshots, share that route set with auth and dispatch, then refresh the branch against current main with live or log-backed gateway proof.

## Reproduction Assessment

Yes, source-reproducible. PR head resolves a scoped registry after an active route miss and reaches `loadOpenClawPlugins`, while current main's loader activates by default; I did not run live gateway traffic in this read-only review.

## Solution Assessment

No. Manifest route-owner hints are a plausible direction, but this implementation puts active plugin loading in the HTTP miss path; a lifecycle-owned non-activating snapshot shared with auth is the safer shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `loadScopedGatewayPluginHttpRouteRegistry` is reached after an HTTP route miss and then calls `loadOpenClawPlugins` without `activate:false`. Current loader semantics make that an active load, so a remote request can mutate global plugin runtime state; move this to gateway startup/reload or a gateway-owned non-activating cached snapshot.
  - confidence: 0.92
- **[P1] Authorize scoped routes before dispatch:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped lookup happens inside plugin dispatch, after the `plugin-auth` stage has already decided whether to authorize using only the active registry. A scoped-only `auth: "gateway"` route on an unprotected path can then be rejected even when the request supplied valid credentials; auth gating needs the same route set that dispatch uses.
  - confidence: 0.88
- **[P1] Isolate scoped owner failures before route matching:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: Using `throwOnLoadError: true` for the whole `gateway-plugin-http` owner set means one broken route-owner plugin throws before route matching can select a healthy scoped route. Because the plugin HTTP stage is `continueOnError`, the request skips all scoped plugin routes instead of serving the healthy one.
  - confidence: 0.84
