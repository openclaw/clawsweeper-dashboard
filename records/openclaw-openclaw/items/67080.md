## Summary

Keep open: this is a protected maintainer PR for a still-relevant gateway/plugin route problem, but the current branch is not merge-ready because the proposed request-path scoped load creates security, auth, availability, and current-main conflict blockers.

## What This Changes

The PR adds manifest `activation.onRoutes` hints for bundled route-owning plugins and teaches gateway plugin HTTP routing to fall back to a manifest-planned scoped plugin registry when the startup registry misses.

## Best Possible Solution

Land a refreshed gateway-owned route lifecycle that resolves or memoizes manifest route owners outside hot request dispatch, uses the same scoped route set for auth and dispatch, isolates failed owners, and reconciles current-main route-registry changes.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch calls `loadOpenClawPlugins` from the request fallback, server auth decisions happen before scoped fallback matching, and the loader defaults to activating global plugin state. I did not run tests because this review is read-only.

## Solution Assessment

No. Manifest route hints are a plausible direction, but the best fix is a gateway-owned or startup/reload-owned scoped registry lifecycle with auth, failure isolation, and live proof aligned to that lifecycle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This fallback calls `loadOpenClawPlugins` from the HTTP request path. The loader defaults to activation, clears global plugin state for activating loads, and activates the loaded registry, so an unmatched network request can trigger plugin module execution and process-global registry churn.
  - confidence: 0.92
- **[P1] Run plugin auth against the scoped route set:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped registry is resolved only after the earlier `plugin-auth` stage decided whether to authenticate using the startup registry. A scoped-only `auth: "gateway"` route on a non-protected path will not get valid credentials populated, then the handler rejects it as missing auth context.
  - confidence: 0.9
- **[P2] Isolate broken scoped owners before route matching:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: `throwOnLoadError: true` makes one failing owner in the coarse `gateway-plugin-http` set abort the whole scoped load. Because plugin HTTP runs as a continue-on-error stage, healthy scoped routes never get a chance to match after that failure.
  - confidence: 0.86
