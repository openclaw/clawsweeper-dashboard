## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Move scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `createGatewayPluginRequestHandler` calls the scoped resolver after every startup-registry miss, and this loader responds by calling `loadOpenClawPlugins(...)` without `activate:false`. On current main that activates the loaded registry and global hook runner by default, so an HTTP request can trigger plugin runtime execution and global registry churn; make this a gateway-lifecycle-owned, non-activating snapshot instead.
  - confidence: 0.91
- **[P2] Do not let one route-owner failure block all scoped routes:** `src/plugins/runtime/http-route-registry-loader.ts:32-33`
  - body: `throwOnLoadError: true` makes any failing plugin in the coarse `gateway-plugin-http` owner set throw before route matching can select the healthy owner. Because plugin HTTP is a continue-on-error stage, one broken route-capable plugin can make all scoped fallback routes disappear for that request.
  - confidence: 0.86
- **[P2] Evaluate auth with scoped route candidates:** `src/gateway/server/plugins-http.ts:85-90`
  - body: Plugin auth runs before plugin dispatch and only sees the startup registry. A scoped `auth: "gateway"` route on a non-protected path will be matched here without `gatewayAuthSatisfied`, then fail closed even when the caller supplied valid gateway credentials; scoped candidates need to participate in the auth decision or be resolved before auth.
  - confidence: 0.84
