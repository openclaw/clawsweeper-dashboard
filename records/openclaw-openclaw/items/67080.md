## Summary

Keep open: current main does not implement this scoped manifest route-loading path, but the protected maintainer PR is currently conflicting and the proposed request-time active plugin load still has security, auth-ordering, and availability blockers before merge.

## What This Changes

The PR adds `gateway-plugin-http` manifest route hints, route-id normalization, scoped route-owner planning/tests/docs, and a gateway plugin HTTP fallback that loads a scoped route registry after a startup-registry miss.

## Best Possible Solution

Land this concept only after scoped route registries are prepared or memoized under gateway lifecycle, loaded non-actively with failure isolation, refreshed against current main, and shared by auth and dispatch.

## Reproduction Assessment

Yes. Source inspection of the PR head shows a startup-registry miss reaches `loadOpenClawPlugins` from HTTP dispatch and that auth is decided before scoped dispatch; I did not run live gateway traffic in this read-only review.

## Solution Assessment

No. Manifest route-owner hints are plausible, but request-time active registry loads are the wrong layer for a gateway hot path; the safer shape is lifecycle-owned scoped route snapshots plus shared auth gating.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `loadScopedGatewayPluginHttpRouteRegistry` is called after an HTTP route miss and then calls `loadOpenClawPlugins` without `activate:false`. Current loader semantics make that an active load, so a remote request can mutate global plugin runtime state; move this to gateway startup/reload or a gateway-owned non-activating cached snapshot.
  - confidence: 0.9
- **[P1] Authorize scoped routes before dispatch:** `src/gateway/server/plugins-http.ts:83-90`
  - body: The scoped lookup happens inside plugin dispatch, after the `plugin-auth` stage has already decided whether to authorize using only the startup registry. A scoped-only `auth: "gateway"` route on an unprotected path can then be rejected even when the request supplied valid credentials; auth gating needs the same route set that dispatch uses.
  - confidence: 0.86
- **[P2] Do not let one scoped owner failure skip all routes:** `src/plugins/runtime/http-route-registry-loader.ts:30-33`
  - body: Using `throwOnLoadError: true` for the whole `gateway-plugin-http` owner set means one broken route-owner plugin throws before route matching can select a healthy scoped route. Because the plugin HTTP stage is continue-on-error, the request skips all scoped plugin routes instead of serving the healthy one.
  - confidence: 0.82
