## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This fallback calls `loadOpenClawPlugins` from the HTTP request path. The plugin loader activates by default and clears or replaces process-global plugin runtime state for activating loads, so an unmatched network request can trigger plugin module execution and registry churn. Move this to a gateway-owned startup/reload lifecycle, or use an explicitly non-activating memoized route registry with a clear owner.
  - confidence: 0.92
- **[P1] Run plugin auth against the scoped route set:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped registry is resolved only after `plugin-auth` has already decided whether to authenticate using the startup route registry. A scoped-only `auth: "gateway"` route on a non-protected path will not get valid gateway credentials populated, then dispatch rejects it as missing auth context. Resolve the route set before auth, or make auth and dispatch share the same scoped registry.
  - confidence: 0.89
- **[P2] Isolate broken scoped owners before route matching:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: `throwOnLoadError: true` makes one failed owner in the coarse `gateway-plugin-http` set abort the whole scoped load. Since plugin HTTP is a continue-on-error gateway stage, healthy scoped routes then never get a chance to match. Keep failed-owner diagnostics but still dispatch healthy route registries, or load owners under an isolated lifecycle.
  - confidence: 0.86
