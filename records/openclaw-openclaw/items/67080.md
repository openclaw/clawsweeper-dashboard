## Best Possible Solution

Land this concept only after scoped route registries are prepared and cached under gateway lifecycle, non-activating, failure-isolated, and used by the same auth decision path as dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: A startup-registry miss calls the scoped resolver from `createGatewayPluginRequestHandler`, and this loader then calls `loadOpenClawPlugins` without `activate:false`. `loadOpenClawPlugins` activates the registry by default, so an external HTTP miss can perform plugin module loading and mutate global plugin state on the request path.
  - confidence: 0.96
- **[P2] Do not abort scoped routes on one owner failure:** `src/plugins/runtime/http-route-registry-loader.ts:32-33`
  - body: With `throwOnLoadError: true`, one malformed or unconfigured route-owner plugin makes the scoped load throw before route matching chooses the handler. Because plugin HTTP is a continue-on-error stage, healthy scoped routes are skipped for that request.
  - confidence: 0.91
- **[P1] Run auth after scoped route candidates are known:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The gateway auth stage decides whether to authenticate using the startup registry before this fallback discovers scoped route candidates. A scoped route with `auth: "gateway"` on an otherwise unprotected path will be matched here with `gatewayAuthSatisfied` still false, so valid credentials never reach the route.
  - confidence: 0.93
