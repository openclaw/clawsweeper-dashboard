## Summary

Keep open: this maintainer/protected PR still has concrete gateway lifecycle, auth-ordering, failure-isolation, and security-boundary blockers, and the branch is dirty against current main. Current main does not already contain the proposed manifest-scoped gateway route loader, while newer open work such as https://github.com/openclaw/openclaw/issues/94572 only partially overlaps this PR's route-loading problem.

## What This Changes

This PR adds manifest `activation.onRoutes` route hints, route-id normalization/planning, and a gateway plugin HTTP fallback that loads a scoped route registry after startup-registry misses.

## Best Possible Solution

Land this concept only after scoped route registries are prepared or memoized under gateway lifecycle, non-activating, failure-isolated, refreshed against current main, and used by the same auth decision path as dispatch.

## Reproduction Assessment

Yes. Source inspection of the PR head shows a startup-registry miss reaches `loadOpenClawPlugins` from HTTP dispatch, and current main shows that loader activates global plugin state unless `activate:false` is set; I did not run live gateway traffic.

## Solution Assessment

No. Manifest route-owner hints are plausible, but request-time activating registry loads are the wrong layer for a gateway hot path; the safer fix is lifecycle-owned scoped route snapshots plus shared auth gating.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `loadScopedGatewayPluginHttpRouteRegistry` is called after an HTTP route miss and then calls `loadOpenClawPlugins` without `activate:false`. Current loader semantics make that an active load, so a remote request can clear or restore plugin runtime state and install a new active registry; move this to gateway startup/reload or use a gateway-owned non-activating cached snapshot before dispatch.
  - confidence: 0.9
- **[P1] Run auth after scoped route candidates are known:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The added scoped lookup happens inside plugin dispatch, after the `plugin-auth` stage has already decided whether to authorize using only the active startup registry. A scoped-only `auth: "gateway"` route on an unprotected path can then be matched with `gatewayAuthSatisfied` false and rejected even when the request supplied valid credentials; auth gating needs the same route set that dispatch uses.
  - confidence: 0.86
- **[P2] Isolate scoped route owner load failures:** `src/plugins/runtime/http-route-registry-loader.ts:32-33`
  - body: Using `throwOnLoadError: true` for the whole `gateway-plugin-http` owner set means one broken route-owner plugin throws before route matching can select a healthy scoped route. Because the plugin HTTP stage is continue-on-error, the request skips all scoped plugin routes instead of serving the healthy one; load or filter owners independently, or return the healthy registry while reporting failed owners.
  - confidence: 0.82
