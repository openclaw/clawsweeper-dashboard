## Solution Assessment

No. The manifest route-owner direction is plausible, but the current branch puts runtime loading in the wrong lifecycle point; the safer fix is a prepared non-activating scoped registry with auth and failure isolation built around that snapshot.

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `loadScopedGatewayPluginHttpRouteRegistry` is reached from an HTTP miss and calls `loadOpenClawPlugins` without `activate:false`, so a remote request can initialize plugin code and mutate the active registry. Move this to a gateway-owned lifecycle snapshot or make the load non-activating and cached outside request dispatch.
  - confidence: 0.92
- **[P2] Do not let one owner failure block all scoped routes:** `src/plugins/runtime/http-route-registry-loader.ts:32-33`
  - body: `throwOnLoadError: true` makes one broken route-owner plugin throw the whole scoped fallback before path matching can choose a healthy owner. Load or plan owners with failure isolation so unrelated scoped routes still get a chance to handle the request.
  - confidence: 0.88
- **[P2] Run auth after scoped route candidates are known:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped registry is discovered after the `plugin-auth` stage has already decided whether to authenticate using only the startup registry. A scoped `auth: "gateway"` route outside the protected path set can therefore fail closed even with valid credentials; include scoped candidates in the auth decision or prepare them before auth runs.
  - confidence: 0.87
