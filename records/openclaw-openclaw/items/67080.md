## Summary

Keep open: this maintainer-labeled PR still targets a real gap because current main has `activation.onRoutes` metadata but no gateway route-miss runtime consumer, yet the branch is not merge-ready because the proposed request-path scoped loading creates lifecycle, auth, security, availability, and merge-conflict blockers.

## What This Changes

The branch adds `gateway-plugin-http` route activation hints to docs and bundled plugin manifests, plus route-id normalization, route-owner planning, scoped gateway plugin HTTP registry fallback logic, and tests.

## Best Possible Solution

Land the concept only after scoped route registries are prepared or cached under gateway lifecycle, non-activating, failure-isolated, and used by the same auth decision path as dispatch.

## Reproduction Assessment

Yes. Source inspection of the PR head shows a startup-registry miss reaches `loadOpenClawPlugins` from HTTP dispatch, and current main shows that loader activates global plugin state unless `activate:false` is set; I did not run live gateway traffic.

## Solution Assessment

No. Manifest route-owner hints are plausible, but request-time activating registry loads are the wrong lifecycle point; the safer fix is a gateway-lifecycle-owned scoped route registry used by both auth and dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This loader is reached from the HTTP route-miss path and calls `loadOpenClawPlugins` without `activate:false`. On current main, that default activation clears and re-registers global plugin runtime state, so an unmatched remote request can trigger plugin code execution and registry churn instead of using a gateway-owned snapshot.
  - confidence: 0.93
- **[P1] Run auth after scoped route candidates are known:** `src/gateway/server/plugins-http.ts:96-99`
  - body: Gateway auth preflight runs before this handler resolves scoped route candidates. A scoped route with `auth: "gateway"` can be matched here only after `gatewayAuthSatisfied` was left false, so it fails closed even when the caller supplied valid gateway credentials.
  - confidence: 0.87
- **[P2] Do not abort scoped routes on one owner failure:** `src/plugins/runtime/http-route-registry-loader.ts:32-33`
  - body: Using `throwOnLoadError: true` for the whole scoped owner set means one broken or unconfigured `gateway-plugin-http` owner throws before route matching can select a healthy handler. Since plugin HTTP is a continue-on-error stage, unrelated scoped routes can disappear for the request.
  - confidence: 0.88
