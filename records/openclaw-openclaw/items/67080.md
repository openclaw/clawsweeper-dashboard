## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: This loader is reached from `createGatewayPluginRequestHandler` after a startup-registry miss, and `loadOpenClawPlugins` defaults to active loading. On current loader semantics that can clear activated runtime state, set the active registry, and initialize global hooks from a network request path, so route-owner loading needs to happen at startup/reload or through a gateway-owned non-activating cache.
  - confidence: 0.9
- **[P1] Resolve scoped routes before gateway auth decisions:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped registry is only resolved inside the plugin HTTP stage, after `server-http` has already decided whether gateway auth is required from the startup registry. A scoped-only `auth: "gateway"` route outside the always-protected path will then reach this later check with `gatewayAuthSatisfied` still false, so even valid credentials are rejected or the request falls through.
  - confidence: 0.88
- **[P2] Isolate broken scoped owners before route matching:** `src/plugins/runtime/http-route-registry-loader.ts:32-33`
  - body: The route id is intentionally coarse, so multiple plugins can be selected for one scoped route load. With `throwOnLoadError: true`, one broken or unconfigured owner throws the whole scoped registry load; because the plugin HTTP stage is `continueOnError`, that skips healthy scoped routes for the request instead of letting them match.
  - confidence: 0.84
