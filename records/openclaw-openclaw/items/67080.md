## Summary

Keep this PR open for maintainer redesign: the manifest route-planning idea is useful, but the current branch puts plugin runtime loading on the gateway request miss path, has an auth-stage mismatch for scoped routes, and is now conflicting with current main's server-local plugin route registry lifecycle.

## What This Changes

The PR adds manifest route activation hints, shared route-id normalization, route-owner planning/trust filtering, docs/tests, and a scoped gateway plugin HTTP fallback that loads route-owner registries from `activation.onRoutes`.

## Best Possible Solution

Land a refreshed route-owner lifecycle that resolves or memoizes scoped route registries outside hot request dispatch, keeps scoped loads non-activating or explicitly lifecycle-owned, feeds the same route set into auth and dispatch, isolates failed owners, and rebases onto current main.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch calls `loadOpenClawPlugins` from the request fallback, route auth runs before scoped fallback matching, and the scoped loader uses `throwOnLoadError: true`. I did not run tests because this review is read-only.

## Solution Assessment

No. Manifest route planning is a useful direction, but request-time activating loads are the wrong layer for gateway hot-path routing; a gateway-owned startup/reload or non-activating cached registry should feed both auth and dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `createGatewayPluginRequestHandler` reaches this loader from the HTTP miss path, and the loader calls `loadOpenClawPlugins` without `activate: false`. That can import and activate route-owner plugins, mutate global runtime state, and spend disk/module-loading work on network requests that miss the startup registry.
  - confidence: 0.91
- **[P1] Run plugin auth against scoped route matches:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped registry is resolved only after the earlier `plugin-auth` stage decides whether the path needs gateway auth from the startup registry. A scoped-only `auth: "gateway"` route outside protected paths will be matched here with no populated auth context, so valid gateway credentials are ineffective for that route class.
  - confidence: 0.89
- **[P2] Isolate failed route owners before matching:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: With `throwOnLoadError: true`, one broken or unconfigured plugin in the broad `gateway-plugin-http` owner set throws the entire scoped load. Because plugin HTTP is a continue-on-error stage, that skips healthy scoped route owners for the request instead of matching the routes that could have loaded.
  - confidence: 0.86
