## Summary

Keep open: this maintainer PR is still useful, but the proposed request-time scoped route load has blocking security, availability, and auth-ordering issues before it can be considered merge-ready.

## What This Changes

The PR adds manifest-driven gateway plugin HTTP route activation, canonicalizes route ids to `gateway-plugin-http`, annotates bundled plugin manifests, and documents the route hint.

## Best Possible Solution

Use a gateway-owned scoped route registry lifecycle that does not mutate the active global registry from requests, integrates scoped matches into auth decisions, tolerates unrelated route-owner load failures, and reconciles with the broader deferred runtime attachment work.

## Reproduction Assessment

Yes, at source level: current main has route activation metadata/planning but gateway plugin HTTP handling consults only the current route registry, while the PR tries to fill that gap with scoped route loading. I did not run a live gateway/plugin traffic reproduction in this read-only review.

## Solution Assessment

No. Scoped manifest route loading is a plausible direction, but this implementation puts activating plugin loads after auth and inside the request path; the safer fix is a gateway-owned non-activating or preloaded registry path with auth and failure semantics proved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Avoid activating plugin registries from requests:** `src/plugins/runtime/http-route-registry-loader.ts:30-32`
  - body: `loadScopedGatewayPluginHttpRouteRegistry` is called from the HTTP miss path and invokes `loadOpenClawPlugins` without `activate: false`. The loader defaults to activating the registry, which clears and replaces global plugin runtime state; an unmatched network request can therefore trigger plugin initialization and global state churn. Move this to a gateway-owned preload/cache path or load a non-activating scoped registry.
  - confidence: 0.9
- **[P2] Keep one bad route owner from blocking scoped routes:** `src/plugins/runtime/http-route-registry-loader.ts:32`
  - body: The scoped loader passes `throwOnLoadError: true` for the whole planned owner set. If any route-owner plugin in that set fails to initialize, the exception bubbles through the `continueOnError` plugin HTTP stage and healthy scoped routes are skipped too. Load/filter failures so one broken owner does not make every fallback route unavailable.
  - confidence: 0.84
- **[P2] Run gateway auth against scoped matches:** `src/gateway/server/plugins-http.ts:84-90`
  - body: Scoped route matching happens only after the earlier `plugin-auth` stage has decided whether to authenticate using the startup registry. A scoped route with `auth: "gateway"` on a non-protected path will be matched here with `gatewayAuthSatisfied` still false and will be rejected even when the caller sent valid gateway credentials. Resolve scoped auth requirements before the auth stage or otherwise restrict/handle scoped gateway-auth routes.
  - confidence: 0.82
