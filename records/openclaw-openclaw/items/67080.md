## Summary

Keep open: current main still lacks the central gateway route-owner consumer, and this MEMBER-authored PR has the protected maintainer label. The branch is not merge-ready because the PR head can load active plugin runtime from an HTTP route miss, auth still consults only the startup route registry before scoped dispatch, scoped-owner load failures can suppress healthy routes, and the live branch is conflicting.

## What This Changes

Adds route activation id canonicalization, manifest `onRoutes` hints for bundled route-capable plugins, route-owner filtering, and a gateway HTTP fallback that loads a scoped plugin registry after startup-registry misses.

## Best Possible Solution

Prepare or memoize scoped route registries under gateway lifecycle with non-activating, failure-isolated snapshots, share that same route set with auth and dispatch, then refresh against current main with live or log-backed gateway proof.

## Reproduction Assessment

Yes. Source inspection of the PR head shows a startup-registry miss reaches `loadOpenClawPlugins` from HTTP dispatch, while current-main plugin auth is decided before plugin dispatch; I did not run live gateway traffic in this read-only review.

## Solution Assessment

No. Manifest route-owner hints are a plausible direction, but this implementation puts active plugin loading in the HTTP miss path; a lifecycle-owned non-activating snapshot shared with auth is the safer shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep scoped route loading out of request dispatch:** `src/plugins/runtime/http-route-registry-loader.ts:30-38`
  - body: `loadScopedGatewayPluginHttpRouteRegistry` is reached after an HTTP route miss and then calls `loadOpenClawPlugins` without `activate:false`. Loader semantics make that an active load, so a remote request can mutate global plugin runtime state; move this to gateway startup/reload or a gateway-owned non-activating cached snapshot.
  - confidence: 0.9
- **[P1] Run plugin auth against the scoped route set:** `src/gateway/server/plugins-http.ts:85-90`
  - body: The scoped lookup happens inside plugin dispatch, after the `plugin-auth` stage has already decided whether to authorize using only the active registry. A scoped-only `auth: "gateway"` route on an unprotected path can then be rejected even when the request supplied valid credentials; auth gating needs the same route set that dispatch uses.
  - confidence: 0.86
- **[P2] Isolate scoped owner failures before route matching:** `src/plugins/runtime/http-route-registry-loader.ts:31-33`
  - body: Using `throwOnLoadError: true` for the whole `gateway-plugin-http` owner set means one broken route-owner plugin throws before route matching can select a healthy scoped route. Because the plugin HTTP stage is `continueOnError`, the request skips all scoped plugin routes instead of serving the healthy one.
  - confidence: 0.82
