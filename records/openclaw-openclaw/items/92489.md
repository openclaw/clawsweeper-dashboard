## Label Justifications

- `P1`: A successful Slack workflow can lose all assistant-side conversational context when its bound CLI session is invalidated.
- `impact:data-loss`: Successfully delivered assistant replies can be silently absent from the durable transcript used for recovery.
- `impact:session-state`: The owning channel or thread session becomes one-sided and rebuilds with incomplete context.

## Summary

The issue is still necessary: current main and the latest release retain a Slack-specific final-delivery path that bypasses the shared session-aware durable sender, so successful channel and thread replies can remain absent from the transcript used to rebuild context.

## What This Changes

Ensure successful Slack channel and thread assistant replies are durably appended to the owning session transcript so context survives CLI-session invalidation.

## Best Possible Solution

Persist every successfully finalized same-surface channel reply through one shared, idempotent session-aware delivery boundary that covers standard sends, preview edits, and streaming completion while excluding partial, reasoning, status, failed, and suppressed payloads.

## Reproduction Assessment

Yes at source level: Slack's current turn supplies direct delivery without the durable session-aware option, exactly matching the reported absence of mirror execution. A live current-main Slack reset was not performed in this read-only review.

## Solution Assessment

Yes, if repaired at the shared final-delivery/session-persistence boundary and tested across Slack's standard and streaming finalization modes; a Slack-local append in one send branch would be too narrow.
