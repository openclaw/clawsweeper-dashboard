## Label Justifications

- `P1`: A normal Slack channel workflow can lose its entire assistant-side conversation context after CLI binding invalidation despite replies having been delivered successfully.
- `impact:data-loss`: Delivered assistant replies are absent from durable session storage and become unrecoverable when the bound CLI session is discarded.
- `impact:session-state`: Transcript reconstruction restores only user turns, leaving channel and thread sessions with severely drifted context.

## Summary

Current main still has the reported path split: Slack channel and thread replies are delivered through the shared inbound dispatcher’s direct adapter, while durable assistant transcript append is attached to outbound sends only when a mirror payload is supplied. The report is therefore still necessary and is a bounded core channel/session bug rather than the broader delivery-reconciliation request in the related issue.

## What This Changes

Ensure successful Slack channel and thread assistant replies are durably appended to the owning session transcript so context survives CLI session-binding invalidation.

## Best Possible Solution

Make the shared channel-turn pipeline append each successfully delivered final assistant payload exactly once to the route session transcript, using the existing transcript helper and an idempotency key, while leaving outbound mirroring responsible only for sends that bypass the normal inbound turn.

## Reproduction Assessment

Yes at source level: normal Slack replies use the direct shared-channel delivery adapter, while assistant transcript append is conditional on a separate outbound mirror payload. The reporter also provides consistent production transcript counts and a concrete binding-invalidation sequence, although this read-only review did not run a live Slack setup.

## Solution Assessment

Yes, the narrowest maintainable solution is exactly-once persistence in the shared channel-turn boundary using the existing transcript writer. Slack-specific transcript code or the broader receipt/reconciliation design from https://github.com/openclaw/openclaw/issues/87544 would be unnecessary for this defect.
