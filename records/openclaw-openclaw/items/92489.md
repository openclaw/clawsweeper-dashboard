## Label Justifications

- `P1`: A normal Slack agent workflow can lose all assistant-side conversational context after a backend binding reset, causing immediate user-visible amnesia.
- `impact:data-loss`: Assistant replies successfully received by users are absent from the durable canonical transcript.
- `impact:session-state`: Transcript-based reconstruction produces an incomplete channel or thread session with only the user side of prior conversation.

## Summary

Current main still has the reported reverse-drift bug: ordinary Slack channel and thread finals use direct plugin delivery without the mirror context required to durably append assistant turns. The related open issue covers the opposite transcript-present/client-missing failure, and live searches found no existing fix PR or canonical duplicate for this missing-transcript path.

## What This Changes

Persist successfully delivered Slack channel and thread assistant replies in the owning session transcript so conversation context survives backend session invalidation.

## Best Possible Solution

Make successful same-surface final delivery own one idempotent assistant transcript append through the shared channel-turn/outbound contract, with Slack channel and thread tests plus a sibling already-mirrored path proving exactly-once behavior.

## Reproduction Assessment

Yes at source level: ordinary Slack finals select the direct callback branch, while assistant transcript appending is conditional on mirror context that this call site does not provide. The reporter's production transcript counts and binding-reset sequence independently match that path split, although no live Slack reproduction was run during this read-only review.

## Solution Assessment

Yes in direction: the narrow maintainable fix is a shared exactly-once delivery/transcript invariant, not Slack-specific JSONL writing. The implementation must explicitly protect existing source-reply and durable paths from duplicate appends.
