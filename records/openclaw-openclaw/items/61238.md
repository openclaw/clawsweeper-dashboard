## Summary

Keep open. Current main still applies an implicit daily 4 AM reset to ordinary sessions, has no public manual-only/off reset mode, and only supports checkpoint-scoped restore rather than a general reset-archive restore path; the provider-owned CLI exemption is a useful partial mitigation but does not cover the reported core behavior.

## What This Changes

The issue asks OpenClaw to stop silently rolling sessions at the daily boundary by adding warning/control, discoverable opt-out or manual-only behavior, reset-archive restore, and stronger setup/docs disclosure.

## Best Possible Solution

Define an explicit session reset contract with a supported manual-only/no-auto policy, expose the effective policy in setup/status/config, add channel-capable warning or confirmation before automatic rollover, and provide a documented branch/restore path for reset archives.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists: with default config and an ordinary existing session whose lifecycle timestamp predates the 4 AM boundary, `initSessionState` creates a new session on the next user message.

## Solution Assessment

No patch is under review, and current main is only a partial mitigation. The maintainable fix should coordinate config semantics, user-facing disclosure/control, and reset-archive recovery rather than treating this as a docs-only cleanup.
