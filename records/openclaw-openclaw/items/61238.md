## Summary

Keep open. Current main and the latest release still have implicit daily ordinary-session rollover at 4:00 AM without a pre-reset warning, cancel/postpone path, first-class manual-only reset mode, or reset-archive restore flow; the linked manual-only issue covers only one narrower piece of the requested lifecycle contract.

## What This Changes

The issue requests warning, opt-out/control, restore, and prominent disclosure for automatic daily session resets.

## Best Possible Solution

Define an explicit reset lifecycle contract that combines a supported manual-only mode, visible setup/status disclosure, channel-capable warning or confirmation before automatic rollover, and safe reset-archive restore or branch semantics.

## Reproduction Assessment

Yes, by source inspection: omitted reset config resolves to daily at 4:00 AM, and `initSessionState` creates a new session on the next real message after the boundary. The missing warning/control/restore pieces are also visible from schema/docs/RPC inspection rather than a failing existing contract.

## Solution Assessment

Unclear as an implementation plan because no patch is under review and the requested behavior spans defaults, public config semantics, channel UX, and archive recovery. The maintainable path is a coordinated product/API decision before a narrow implementation.
