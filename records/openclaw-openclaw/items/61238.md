## Summary

Keep open. Current main and v2026.6.10 still implement the implicit daily 4 AM reset with only daily/idle modes, and the open related items cover narrower slices rather than the full warning, opt-out/default-policy, restore, and pending-confirmation lifecycle requested here.

## What This Changes

The issue requests visible warning/cancel controls, discoverable opt-out or admin-managed reset defaults, and restore/recovery support for silent daily session resets that archive active session context.

## Best Possible Solution

Define a maintainer-approved reset lifecycle contract covering visible defaults, manual-only or scoped reset policy, warning/postpone behavior, pending confirmations, archive restore or read-only recovery, docs, doctor migration guidance, and regression tests.

## Reproduction Assessment

Yes. Source inspection shows current main still resolves omitted reset config to daily 4 AM rollover, validates only daily/idle reset modes, and rolls stale sessions into a new session; I did not run a live multi-day reset scenario.

## Solution Assessment

No. This is not one narrow bug repair yet; the maintainable solution needs a product-owned reset lifecycle contract spanning config defaults, warning UX, archive recovery, and channel confirmation state.
