## Label Justifications

- `P2`: This is a source-backed session lifecycle and recovery problem with clear impact, but solving it requires a coordinated product/API decision rather than an emergency runtime fix.
- `impact:data-loss`: The report is about automatic rollover making previous conversation data inaccessible from the active session and lacking a supported recovery path.
- `impact:session-state`: The affected behavior changes session continuity, transcript association, and long-lived context across reset boundaries.

## Summary

Keep open. Current main and v2026.5.18 still have a source-reproducible default daily rollover with no warning/cancel or manual-only reset mode, and the existing archive/search/compaction restore surfaces do not satisfy active-session restore for reset archives; related open work covers pieces but not the full lifecycle.

## What This Changes

Request warning, cancel, export, restore, and discoverable opt-out controls for the default daily 4:00 AM session reset.

## Best Possible Solution

Define one reset lifecycle contract that combines prominent setup/status disclosure, the manual-only mode tracked by https://github.com/openclaw/openclaw/issues/57369, channel-appropriate pre-rollover warning or confirmation, and safe read/restore semantics for reset archives.

## Reproduction Assessment

Yes. Source inspection shows omitted reset config resolves to daily at 4:00 AM and initSessionState rolls stale ordinary sessions to a new session; the missing warning, manual-only, and active restore pieces are visible in schema, docs, command registry, and gateway restore surfaces.

## Solution Assessment

Unclear as a concrete implementation because the request spans defaults, public config semantics, channel UX, and archive recovery. The maintainable path is a product/API decision that splits the lifecycle into explicit contract pieces before implementation.
