## Summary

Keep open. Current main and v2026.5.18 still implement implicit daily 4:00 AM rollover for ordinary sessions, with no pre-reset warning/cancel path, no first-class manual-only reset mode, and no merged restore/archive-history contract that fully addresses the reported user problem.

## What This Changes

The issue asks for warnings, opt-out/manual-only controls, and recovery for silent daily session resets that make prior context inaccessible from the active session.

## Best Possible Solution

Define one explicit reset lifecycle contract that combines visible setup/status disclosure, a supported manual-only mode, channel-capable pre-rollover warning or confirmation, and safe read/restore semantics for reset archives.

## Reproduction Assessment

Yes. Source inspection shows omitted reset config resolves to daily at 4:00 AM and `initSessionState` rolls stale ordinary sessions to a new session; the missing warning, manual-only, and restore pieces are visible in schema, docs, command registry, and gateway/tool history surfaces.

## Solution Assessment

Unclear as a concrete implementation because the request spans defaults, public config semantics, channel UX, and archive recovery. The maintainable path is a coordinated product/API decision before a narrow implementation.
