## Summary

Keep open. Current main has a partial mitigation for provider-owned CLI sessions, but ordinary sessions still default to a silent daily 4:00 AM rollover and the public config contract still lacks a no-auto-reset mode, warning/cancel/postpone UX, or general reset-archive restore path.

## What This Changes

The issue requests warning, control, discoverable opt-out, export/save, and restore behavior for automatic daily session resets.

## Best Possible Solution

Define an explicit session reset contract: support a real manual-only/no-auto-reset policy, surface the active policy in setup/config/status, add channel-capable warning or confirmation before automatic rollover, and provide a documented restore or branch-from-reset-archive path.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists: default session config plus an existing non-provider-owned session before the 4:00 AM boundary causes `initSessionState` to create a new session on the next real user message.

## Solution Assessment

No patch is under review, and current main is only a partial mitigation. The best fix should coordinate config semantics, user-facing disclosure/control, and reset archive recovery rather than treating this as a narrow docs-only issue.
