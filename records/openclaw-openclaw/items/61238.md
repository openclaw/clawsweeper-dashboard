## Summary

Keep open. Current main and the latest release still resolve an unset session reset policy to a daily 4 AM reset, still expose only `daily` and `idle` reset modes, and the remaining warning, opt-out, restore, admin policy, and pending-confirmation behavior needs maintainer product direction rather than conservative cleanup.

## What This Changes

The issue requests safer session reset behavior: warnings before automatic daily resets, a discoverable opt-out or manual-only mode, active-session restore from reset archives, and setup/docs visibility.

## Best Possible Solution

Keep this open as the broad reset-lifecycle decision, then land maintainer-sponsored slices for manual-only policy, scoped/admin defaults, reset warnings or postponement for active channel workflows, and archive restore/readback UX with explicit migration guidance.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.10` still default unset session reset to daily 4 AM, accept only `daily`/`idle`, and rotate stale sessions into a new active session rather than warning or restoring archived context.

## Solution Assessment

No, the current behavior is not the best end state for this report. A narrow code-only fix would be too small; the maintainable path is a maintainer-approved reset lifecycle design with migration, docs/setup visibility, channel-safe warning or postponement behavior, and restore/access tooling.
