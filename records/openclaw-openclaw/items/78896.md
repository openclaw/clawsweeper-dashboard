## Summary

Keep open: the PR is a plausible attempt to address the linked config-loss report, but the current patch changes the root config validation policy too broadly, adds an unused top-level model field, and has no real behavior proof from an external contributor.

## What This Changes

This PR changes the root OpenClaw config schema to preserve unknown top-level keys, adds `defaultModel` and agent `description` to config types/Zod schemas, and updates doctor config-analysis tests.

## Best Possible Solution

Keep root config validation strict, add only maintainer-approved schema keys that runtime actually honors, and prove the `doctor --fix` behavior with a redacted real config plus focused regression coverage and docs/changelog updates.

## Reproduction Assessment

Partly. Source inspection shows current main strips unsupported top-level keys and `agents.list[].description`, while `mcp.servers` is already recognized; I did not run `doctor --fix` because this review is read-only.

## Solution Assessment

No. The broad root `catchall` is not the narrowest maintainable fix; a safer path is to add or migrate only supported config fields and keep typo/legacy cleanup working.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep root unknown-key cleanup working:** `src/config/zod-schema.ts:1136`
  - body: Changing the root schema to `catchall(z.unknown())` makes every mistyped top-level key valid. Since `stripUnknownConfigKeys` only removes Zod `unrecognized_keys`, `doctor --fix` and the update repair path will stop pruning root-level unknown config even though the docs and updater rely on that behavior. Please keep the root strict and handle only the specific supported keys or migrations.
  - confidence: 0.86
- **[P2] Do not validate an unused defaultModel key:** `src/config/zod-schema.ts:361`
  - body: Adding top-level `defaultModel` to the schema makes `openclaw config validate` accept a key that current model selection does not read; defaults are resolved from `agents.defaults.model`. This would silently preserve a no-op config value, so either wire the runtime and docs to the new contract or leave the key rejected/migrated to the existing model config shape.
  - confidence: 0.82
