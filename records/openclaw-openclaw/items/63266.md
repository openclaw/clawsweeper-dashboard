## Summary

Keep open. Current main and v2026.5.19 still have a no-op legacy workspace detector, so doctor/status cannot warn about the reported active child workspace plus parent-held operational files, and the public docs still do not explain this Docker-to-host layout case.

## What This Changes

The issue requests detection, warning, and documentation for Docker-to-host migrations that leave operational files split between `workspace-local` and an active `workspace-local/local` workspace.

## Best Possible Solution

Add a read-only doctor/status diagnostic for active child workspaces with suspicious parent-held operational markers, document canonical host and Docker workspace locations, and keep any file movement or repair opt-in.

## Reproduction Assessment

Yes, at source level. Passing an active child workspace such as `workspace-local/local` into `detectLegacyWorkspaceDirs()` still returns an empty `legacyDirs` array, so doctor/status has no code path to warn about parent-held files.

## Solution Assessment

Yes. A diagnostic-first doctor/docs repair is the narrowest maintainable solution because it addresses operator confusion without silently moving files or redefining local secret conventions.
