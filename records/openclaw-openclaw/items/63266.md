## Summary

Keep open. Current main still lacks the reported split-layout detection: the workspace detector returns an empty legacy list for every input, doctor/status only warns when that list is non-empty, and the docs still cover only generic extra workspaces rather than the Docker-to-host `workspace-local` versus `workspace-local/local` case. The reporter also narrowed the request to read-only detection, status reporting, and docs, which is a focused remaining repair.

## What This Changes

Add doctor/status and documentation support for Docker-to-host migrations that leave operational files under `workspace-local` while the active workspace is `workspace-local/local`.

## Best Possible Solution

Implement a read-only doctor/status check that recognizes an active `workspace-local/local` child with relevant parent-held operational markers, clearly prints both the active workspace and parent markers, and updates the workspace and Docker docs with canonical post-migration expectations while preserving intentional per-agent child workspaces.

## Reproduction Assessment

Yes. A focused source-level reproduction is to call `detectLegacyWorkspaceDirs()` with a child workspace path such as `workspace-local/local` and parent-held markers present; current main still returns an empty `legacyDirs`, so `noteWorkspaceStatus()` cannot warn.

## Solution Assessment

Yes. The narrowed doctor/docs direction is the best first fix because it addresses operator confusion without changing SecretRef semantics or performing risky startup/load-time file migration.

## Repair Work Prompt

Fix this issue by adding a narrow, read-only doctor/docs repair for Docker-to-host split workspaces. Detect when the active default-agent workspace resolves to a child such as `workspace-local/local` and the parent `workspace-local` contains relevant operational markers users may reasonably confuse with active workspace state, such as bootstrap or memory files, `skills/`, `memory/`, or operator-held `secrets/` files. Report the active workspace and parent-held markers clearly in doctor/status output, but do not move, copy, delete, or canonicalize parent files. Preserve intentional per-agent child workspaces created by `resolveAgentWorkspaceDir()` for non-default agents; the warning should be about parent-held operational files near the active workspace, not a blanket ban on child workspaces. Update public docs to explain canonical post-migration workspace expectations and the difference between documented SecretRef/config state and operator/skill-specific files like `workspace-local/secrets/homeassistant.md`. Add focused Vitest coverage for the detector/status behavior and docs coverage if required. Changelog likely is not required unless maintainers treat the doctor warning as user-facing release-note material. Do not change SecretRef semantics, do not introduce startup/load-time migrations, and do not add automatic repair without maintainer approval.
