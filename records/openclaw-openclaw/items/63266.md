## Summary

Keep open. Current main still lacks the narrowed doctor/docs behavior: the workspace detector is still a no-op, the doctor status warning depends on that empty detector result, and the docs do not explain the reported Docker-to-host `workspace-local` versus `workspace-local/local` split. The reporter’s follow-up usefully narrows this to read-only detection, clear reporting, and docs rather than automatic secret migration.

## What This Changes

Request doctor/status and docs support for Docker-to-host migrations that leave parent `workspace-local` operational files alongside an active `workspace-local/local` workspace.

## Best Possible Solution

Add a narrow read-only doctor/status diagnostic that detects an active `workspace-local/local` child with relevant parent-held operational markers, prints the active workspace and parent markers clearly, and updates the workspace, doctor, and Docker docs with canonical post-migration guidance while leaving any file movement as an explicit future repair.

## Reproduction Assessment

Yes. A source-level reproduction is to pass a child workspace path such as `~/.openclaw/workspace-local/local` into `detectLegacyWorkspaceDirs()` with parent-held operational markers present; current main still returns an empty `legacyDirs`, so `noteWorkspaceStatus()` cannot emit a warning.

## Solution Assessment

Yes. The reporter’s read-only doctor/docs direction is the best first solution because it addresses the operator confusion without changing SecretRef semantics or moving local secret-like files by default.
