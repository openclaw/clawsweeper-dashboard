## Summary

Keep open: current main and v2026.6.5 still reject Control UI video attachments and lack an end-to-end native video chat contract. Two open PRs attempt overlapping solutions, but neither currently provides a clean canonical landing path, so maintainers should consolidate the design rather than queue another fix.

## What This Changes

Add direct video attachment upload, preview, validation, model-capability routing, provider-native LLM delivery, and managed-media fallback.

## Best Possible Solution

Consolidate one provider-neutral video input contract with strict upload validation, provider-owned native payload conversion, final-model capability routing, a non-dropping managed-media fallback, aligned docs/tests, and live proof against at least one supported provider.

## Reproduction Assessment

Yes, at source level: current main explicitly filters MP4 attachments and lacks native video in the central LLM and model-catalog input contracts. No live provider call was needed to establish that the requested capability is absent.

## Solution Assessment

No, parallel implementations are not the best landing path. Maintainers should consolidate the useful work behind one typed shared contract and require provider-specific implementation and live proof at the owning plugin boundary.
