## Summary

Keep open: current main and the latest release still reject WebChat video uploads and do not expose a native video modality in the central LLM/model-catalog contracts. The issue is already paired with overlapping open PRs, so maintainers should consolidate the product/API direction rather than close this or queue a duplicate repair.

## What This Changes

The issue requests direct video file upload support through chat so capable LLM providers can receive native video inputs instead of requiring manual frame extraction.

## Best Possible Solution

Consolidate one provider-neutral chat video upload contract with strict validation, model capability routing, provider-owned native payload conversion, a non-dropping managed-media fallback, aligned docs/tests, and live proof against at least one supported provider.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: WebChat rejects `video/*` and common video extensions before send, while the model and catalog contracts do not expose a native video input modality.

## Solution Assessment

Unclear as currently split across proposals. Direct chat video support is the right product target, but the maintainable path is one consolidated core/provider contract rather than parallel PRs or a UI-only acceptance change.
