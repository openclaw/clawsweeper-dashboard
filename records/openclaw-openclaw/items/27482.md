## Summary

Keep this issue open: current main and v2026.6.10 still block WebChat video uploads and the native chat/runtime contract is still text/image plus generic managed media, not direct provider-native video input. Related open PRs only cover partial model-catalog or native media-understanding pieces and are not a safe replacement for this canonical tracker.

## What This Changes

Issue requests direct video file upload and native delivery to video-capable LLMs, matching the existing image upload experience.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers approve one coherent video-aware chat/media contract across WebChat, Gateway validation/staging, LLM content types, provider serializers, fallback behavior, docs, and tests.

## Reproduction Assessment

Yes, at source level: current main filters MP4/WebM/MOV/M4V before WebChat send, and the Gateway/LLM runtime contracts only provide native image prompt content plus generic offloaded media. I did not run a live browser upload in this read-only review.
