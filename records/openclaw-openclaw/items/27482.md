## Summary

Keep open: current main and v2026.5.20 still explicitly block Control UI video uploads, while Gateway chat attachments only provide native image content plus generic media staging. Open implementation PRs already target overlapping parts of the feature, so this should remain the product tracker instead of spawning a duplicate fix branch.

## What This Changes

Support direct video uploads in chat/message flows so video-capable LLMs can receive clips natively, with UI previews and safe fallback behavior.

## Best Possible Solution

Keep this issue as the canonical tracker and land or replace the open implementation with one end-to-end video-aware chat contract covering UI previews, Gateway validation/staging, model capability metadata, provider-native dispatch, fallback behavior, docs, and regression proof.

## Reproduction Assessment

Yes, source-level: current main filters video before chat.send, the UI test suite asserts MP4 rejection, and Gateway native prompt attachments remain image-only plus generic media staging. I did not run runtime tests because this was a read-only review.

## Solution Assessment

Yes for triage: keeping this issue open while maintainers review or replace https://github.com/openclaw/openclaw/pull/74082 and https://github.com/openclaw/openclaw/pull/78797 is the maintainable path. The code solution should be one coherent media/provider contract rather than a UI-only upload change.
