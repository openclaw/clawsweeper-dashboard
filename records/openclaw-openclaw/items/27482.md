## Summary

Keep open: current main still rejects direct chat video uploads and lacks a native video input contract for chat/model/provider delivery. Open implementation PRs already target the gap, especially https://github.com/openclaw/openclaw/pull/74082, so maintainers should converge that work instead of spawning a duplicate repair branch.

## What This Changes

Request direct video file upload support in chat so video-capable LLMs can receive native video input with previews and common formats.

## Best Possible Solution

Keep this issue as the canonical tracker and converge the open implementations into one end-to-end video-aware chat contract covering UI previews, Gateway validation/staging, model capability metadata, provider-native dispatch, fallback notes, docs, and regression proof.

## Reproduction Assessment

Yes, source-level: Control UI filters video before chat.send, current UI tests assert MP4 rejection, and the Gateway/model-catalog contracts only provide native image plus generic media handling. I did not run runtime tests because this was a read-only review.

## Solution Assessment

Yes for triage: keeping this open while maintainers review, repair, or replace https://github.com/openclaw/openclaw/pull/74082 is the narrowest maintainable path. A duplicate repair branch would split the same UI, Gateway, model-catalog, provider, docs, and test work.
