## Summary

Current main still does not implement direct video upload/native LLM delivery, so this issue should remain open. The Control UI still rejects video attachments, the Gateway attachment contract still has native image content plus generic offloaded media refs, and open PR #74082 is the active implementation candidate rather than a merged fix.

## What This Changes

The issue requests direct MP4/WebM/MOV video upload support in chat, including validation, preview, storage or encoding, and delivery to models that support native video input.

## Best Possible Solution

Keep this issue open as the product tracker until https://github.com/openclaw/openclaw/pull/74082, or a maintainer-approved equivalent, lands with end-to-end WebChat video acceptance, Gateway validation, native/fallback attachment handling, model capability metadata, provider dispatch, docs, and regression tests.

## Reproduction Assessment

Yes. On current main, the reproduction path is to attach an MP4 in Control UI chat: the UI support helper rejects video MIME/extensions and the existing chat view test locks that rejection behavior.

## Solution Assessment

Yes, keeping this issue open while reviewing the linked implementation PR is the best path. Current main only has generic media staging and image-native attachments, so closing before a merged video-aware contract would be premature.
