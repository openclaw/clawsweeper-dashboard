## Summary

Keep this PR open, but it is not merge-ready. Current main still rejects boolean `visibleReplies` values at both reported config paths, so the branch targets a real bug and the schema-boundary direction is reasonable; however, the submitted patch leaves the checked-in public config schema/docs baselines stale and its transform-based approach needs to keep regenerated JSON schema constrained rather than widening the field to an unconstrained `{}` branch.

## What This Changes

The PR adds a shared `VisibleRepliesSchema` that coerces boolean `true`/`false` to the existing visible-reply string modes, uses it for global and group message config, adds focused tests, and adds a changelog fix entry.

## Best Possible Solution

Land a narrow config compatibility fix that accepts boolean input at both visible-reply config paths, preserves the runtime output type as `"automatic" | "message_tool" | undefined`, updates the checked-in generated schema and docs baseline, and ensures public JSON schema consumers see a constrained enum-or-boolean input shape rather than an unconstrained fallback.

## Solution Assessment

No, not as submitted. Coercing at the config boundary is the right maintainable direction, but the patch must synchronize generated artifacts and avoid transform-driven public schema widening before merge.
