## Summary

Keep open. The PR targets a real current-main config validation bug and the schema-layer coercion is the right direction, but it is not merge-ready because it changes the computed base config schema without updating the checked-in generated schema and config-doc baseline artifacts.

## What This Changes

The PR adds a shared `VisibleRepliesSchema` that preserves the existing string enum values while coercing boolean `true`/`false` for `messages.visibleReplies` and `messages.groupChat.visibleReplies`, plus regression tests and a changelog entry.

## Best Possible Solution

Land the schema-layer coercion with regenerated public config schema and config-doc baseline artifacts, preserving the runtime config output as `"automatic" | "message_tool" | undefined` while making boolean compatibility visible to schema consumers.

## Reproduction Assessment

Yes. Current main has enum-only Zod validation at both reported paths, and the paired report gives concrete JSON5 steps where `visibleReplies: true` makes config validation fail and channels go down.

## Solution Assessment

No, not as submitted. Coercing at the shared schema boundary is the maintainable fix, but the branch also needs regenerated config artifacts before merge.
