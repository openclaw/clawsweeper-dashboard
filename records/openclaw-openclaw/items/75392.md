## Summary

Keep open. The PR targets a real current-main config bug and the schema-layer compatibility direction is reasonable, but the submitted patch is not merge-ready because its transform-based schema can widen the generated public JSON Schema and it does not update the checked-in generated schema artifacts.

## What This Changes

The PR adds a shared `VisibleRepliesSchema` that coerces boolean `visibleReplies` values to the existing string modes, uses it for global and group message config, adds focused tests, and adds a changelog fix entry.

## Best Possible Solution

Land a narrow config compatibility fix that accepts boolean input at both visible-reply config paths, preserves the runtime output type as `"automatic" | "message_tool" | undefined`, keeps public JSON Schema constrained to enum-or-boolean input, updates generated artifacts, and lets the linked user issue close after merge.

## Reproduction Assessment

Yes. Current main has enum-only Zod validation at both reported paths, and the validation plus reload path shows why boolean config input becomes an invalid snapshot instead of a working config reload.

## Solution Assessment

No, not as submitted. Coercing at the config boundary is the right maintainable direction, but this patch must avoid transform-driven public schema widening and synchronize the generated config artifacts before merge.
