## Summary

Keep open. Current main still rejects boolean `visibleReplies` at both reported config paths, and this PR is the right narrow direction, but it is not mergeable until the public generated schema stays constrained and the generated schema/docs artifacts are synchronized.

## What This Changes

The PR adds a shared `VisibleRepliesSchema` that coerces boolean `true`/`false` to existing visible-reply modes, uses it for global and group message config, adds focused tests, and adds a changelog entry.

## Best Possible Solution

Land a narrow config compatibility fix that accepts boolean input at both visible-reply config paths, preserves normalized runtime output as the existing enum type, and updates generated schema/docs baselines without widening the public schema beyond enum-or-boolean input.

## Reproduction Assessment

Yes. The high-confidence reproduction is to set `messages.groupChat.visibleReplies: true` or `messages.visibleReplies: true`; current main's enum-only Zod fields reject that value and the gateway reload path skips invalid snapshots.

## Solution Assessment

No, not as submitted. Coercing at the config boundary is the maintainable direction, but the implementation needs a JSON-schema-safe shape and regenerated artifacts before merge.
