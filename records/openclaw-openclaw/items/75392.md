## Summary

Keep this PR open, but it is not merge-ready. Current main still rejects boolean `visibleReplies` at both reported config paths, and schema-layer coercion is the right compatibility boundary; the submitted patch still needs a JSON-schema-safe representation plus regenerated config artifacts before it should land.

## What This Changes

The PR adds a shared `VisibleRepliesSchema` that maps boolean `true`/`false` to existing visible-reply modes, wires it into global and group message config, adds focused tests, and adds a changelog fix entry.

## Best Possible Solution

Land a narrow config compatibility fix that accepts boolean input at both visible-reply config paths, keeps the normalized runtime type as `"automatic" | "message_tool" | undefined`, exposes a constrained enum-or-boolean shape in generated public config schema, and refreshes the generated schema/docs baseline together with focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has enum-only Zod validation for both reported fields, invalid config validation returns issues, and gateway reload skips invalid config snapshots.

## Solution Assessment

No, not as submitted. Coercing at the config boundary is the maintainable direction, but the current transform-union shape can widen public JSON schema output and the checked-in generated artifacts are stale.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep visibleReplies JSON schema constrained:** `src/config/zod-schema.core.ts:400-410`
  - body: This schema feeds `OpenClawSchema.toJSONSchema({ unrepresentable: "any" })`, and Zod treats `z.transform()` as unrepresentable under JSON Schema conversion. With `unrepresentable: "any"`, the boolean transform arms can become `{}` branches, so regenerated public schema may accept arbitrary values for `visibleReplies` unless this is modeled or overridden with a constrained enum-or-boolean input shape.
  - confidence: 0.87
- **[P2] Regenerate config schema artifacts:** `src/config/zod-schema.core.ts:410`
  - body: Changing the Zod config schema requires updating `src/config/schema.base.generated.ts` and `docs/.generated/config-baseline.sha256`. As submitted, the generated-schema comparison still points at the stale string-only schema, so config schema/docs checks and public schema consumers will not reflect the new boolean compatibility.
  - confidence: 0.93
