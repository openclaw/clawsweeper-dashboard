## Summary

Keep this PR open: current main does not implement the requested `socket.drain` path, and the runtime direction looks reasonable, but the external PR still needs real Gateway/WebSocket proof, docs/changelog coverage, and exact-head check resolution before merge.

## What This Changes

The PR adds read-scoped `socket.drain` teardown events for Gateway session reset/delete/new flows, routes them to session-event and per-session message subscribers, exposes the message-subscriber lookup on `GatewayRequestContext`, and adds gateway tests.

## Best Possible Solution

Land this Gateway protocol change, or a maintainer-approved equivalent, after docs/changelog and real client proof show message-only subscribers receive read-scoped `socket.drain` on reset/delete/new teardown paths.

## Reproduction Assessment

Yes, at source level. Current main keeps message subscribers in a separate registry while `emitSessionsChanged` only targets session-event subscribers, and `socket.drain` is absent from the event guard/discovery lists.

## Solution Assessment

Yes for the runtime direction, but not yet for merge readiness. Unioning event and message subscribers in `emitSessionsChanged` follows the existing `session.message` broadcast precedent; docs/changelog, exact-head checks, and contributor-supplied real behavior proof remain required.
