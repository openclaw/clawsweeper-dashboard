The PR adds a Gateway `socket.drain` broadcast for session reset/delete/new flows, routes it to session event and message subscribers, registers the event with read scope/discovery, and adds focused session-message event tests.

## Best Possible Solution

Land a maintainer-approved Gateway protocol update that documents `socket.drain` and proves reset/delete delivery to message subscribers with redacted live WebSocket evidence.

Yes, at source level: current main only sends teardown `sessions.changed` to session-event subscribers and lacks `socket.drain` registration. I did not run a live WebSocket repro in this read-only review.

## Solution Assessment

Mostly yes: centralizing teardown emission in session mutations and reusing the existing subscriber registries is the narrow runtime shape. The remaining gap is contract documentation and live proof, not a fundamentally different code path.
