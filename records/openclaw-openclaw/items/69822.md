## What This Changes

The PR adds `socket.drain` teardown broadcasts for reset/delete/new session lifecycle changes, routes them to session-event and per-session message subscribers, registers the event, exposes a message-subscriber lookup on GatewayRequestContext, and adds Gateway tests.

## Best Possible Solution

## Reproduction Assessment

Yes, at source level: current main routes session lifecycle changes only to session-event subscribers and rejects unregistered non-plugin events, so message-only subscribers cannot receive `socket.drain`. I did not establish a live failing WebSocket reproduction in this read-only review.

## Solution Assessment

Mostly yes, but not complete yet: centralizing teardown drain routing in `emitSessionsChanged` is a narrow maintainable direction, while the PR still needs the local context implementation, public protocol documentation, and live behavior proof.
