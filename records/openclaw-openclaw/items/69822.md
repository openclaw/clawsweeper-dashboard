## Summary

Keep open. The happy-path subscriber routing is useful, but this PR still needs a maintainer Gateway protocol decision because the new `socket.drain` event is advertised without public protocol semantics and the slow-consumer path can close the target socket before the critical drain frame is sent.

## What This Changes

This PR adds Gateway `socket.drain` session teardown events for event and message subscribers, exposes message-subscriber lookup on `GatewayRequestContext`, registers the event, and adds focused Gateway tests.

## Best Possible Solution

Define and document the `socket.drain` protocol contract first, then align implementation and tests so normal subscribers and saturated slow consumers behave exactly according to that contract.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the remaining blocker: make a targeted `socket.drain` recipient's `bufferedAmount` exceed `MAX_BUFFERED_BYTES`, then trigger reset/delete/new and observe the broadcaster close before send. I did not run a live saturated-buffer test in this read-only review.

## Solution Assessment

No. The subscriber-union design is a plausible happy-path implementation, but this is not the best complete fix until the Gateway protocol and slow-consumer delivery semantics are explicitly chosen, documented, and tested.
