## Summary

This PR should stay open: it is a focused Gateway fix for a real source-level delivery gap, but it still changes the slow-consumer failure mode for existing `sessions.subscribe` clients and should not merge until that compatibility decision is resolved.

## What This Changes

The branch adds `socket.drain` Gateway broadcasts for session reset/delete/new, routes them to session message subscribers, exposes message-subscriber lookup on `GatewayRequestContext`, registers the event, and expands Gateway session-message tests.

## Best Possible Solution

Add the teardown notification as a documented Gateway protocol event for message subscribers while preserving the existing `sessions.changed` slow-consumer contract unless maintainers explicitly choose and document the upgrade-visible behavior change.

## Reproduction Assessment

Yes at source level: current main keeps generic session event subscribers and per-session message subscribers separate, and lifecycle changes only broadcast to the generic event subscriber set. I did not run a live WebSocket repro in this read-only review.

## Solution Assessment

No, not yet: routing a drain event to message subscribers is the right direction, but changing `sessions.changed` slow-consumer semantics in the same PR is an upgrade-sensitive protocol decision and should be preserved or explicitly approved and documented.
