## Summary

Keep open: current main still lacks the requested `socket.drain` routing for message-only subscribers, and this PR remains plausible but not merge-ready because the supplied proof is still only scoped Vitest output and the advertised Gateway event is undocumented.

## What This Changes

The PR adds read-scoped `socket.drain` teardown broadcasts for reset/delete/new session paths, routes them to both session-event and per-session message subscribers, threads message-subscriber lookup through gateway context, refactors agent reset emission, and adds gateway tests.

## Best Possible Solution

Land this gateway protocol change, or an equivalent one, after adding docs/changelog and real Gateway-client proof that message-only subscribers receive `socket.drain` on reset/delete without broadening event scopes.

## Reproduction Assessment

Yes, at source level. Current main keeps message subscribers in a separate registry while `emitSessionsChanged` only targets session-event subscribers, and `socket.drain` is absent from the event guard/discovery lists.

## Solution Assessment

Yes for the runtime direction, but not yet for merge readiness. Unioning event and message subscribers in `emitSessionsChanged` matches the existing transcript broadcast precedent; docs/changelog and contributor-supplied real behavior proof remain required.
