## Summary

Keep this PR open: the direction is useful, but the branch is not merge-ready because the new drain path misses global-session message subscribers, changes existing slow-client behavior for `sessions.changed`, duplicates part of the agent reset broadcast path, and advertises a new Gateway event without a documented public contract.

## What This Changes

The branch adds Gateway `socket.drain` teardown broadcasts for session reset/delete/new, exposes message-subscriber connection lookup on `GatewayRequestContext`, refactors agent resets through the session broadcast helper, advertises the event, and adds gateway session-message tests.

## Best Possible Solution

Implement `socket.drain` through the same canonical/scoped session-message routing used by transcript events, keep existing `sessions.changed` slow-client behavior unless maintainers explicitly approve a compatibility break, document the advertised event contract, and require exact-path real WebSocket proof before merge.

## Reproduction Assessment

Yes for the PR defects from source inspection: global message subscriptions are stored under scoped `agent:<id>:global` keys, while the new drain lookup uses only `payload.sessionKey`; the broadcaster also closes slow sockets when `dropIfSlow` is false. I did not run tests because this review is read-only.

## Solution Assessment

No. The best fix is a narrower protocol-compatible change that reuses the existing message-broadcast key resolution, preserves legacy `sessions.changed` slow-client behavior, and documents the new event before advertising it.

  - body: `sessions.messages.subscribe` stores global subscriptions under scoped keys like `agent:main:global`, and transcript delivery already expands global broadcasts through those keys.   - confidence: 0.93
- **[P1] Preserve sessions.changed slow-client behavior:** `src/gateway/server-methods/sessions.ts:406`
  - body: Current `sessions.changed` broadcasts are best-effort with `dropIfSlow: true`; when this becomes false, the broadcaster closes slow sockets instead of sending. That changes existing `sessions.subscribe` clients during reset/delete/new, so keep `sessions.changed` on the old policy and limit any new teardown behavior to the new event after documenting it.
  - confidence: 0.9
- **[P2] Avoid duplicate agent reset broadcasts:** `src/gateway/server-methods/agent.ts:1460-1463`
  - body: This emits after `runSessionResetFromAgent`, but the no-message branch still emits again after responding. A bare `/reset` now sends duplicate lifecycle/drain events, and the first emit also drops the global `agentId`, so keep a single correctly scoped emit for each branch.
  - confidence: 0.88
- **[P2] Document socket.drain before advertising it:** `src/gateway/server-methods-list.ts:47`

Either document the stable contract and add focused tests, or keep it out of advertised discovery until maintainers choose that contract.
