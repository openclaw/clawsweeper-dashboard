## Summary

Keep open: this is a focused external Gateway feature PR for a still-missing `socket.drain` path. The runtime direction looks plausible, but it adds a public event without protocol docs and the supplied proof is focused Vitest output rather than live Gateway/WebSocket behavior.

## What This Changes

The PR adds Gateway `socket.drain` teardown broadcasts for reset/delete/new flows, routes them to session-event and per-session message subscribers, exposes the message-subscriber getter in request context, registers the event, and adds focused Gateway tests.

## Best Possible Solution

Land a maintainer-approved Gateway protocol update that documents `socket.drain`, keeps teardown routing centered in session mutation handling, and includes redacted live WebSocket proof for reset/delete delivery.

## Reproduction Assessment

Yes, at source level: current main sends session mutation broadcasts only to session-event subscribers with `dropIfSlow: true`, and `socket.drain` is not registered. I did not run a live WebSocket reproduction in this read-only review.

## Solution Assessment

Mostly yes: centralizing teardown drains in session mutations and reusing the existing subscriber mapping is the narrow maintainable runtime path. It still needs public protocol docs, maintainer approval for the new event contract, and contributor-supplied live proof before merge.
