## What This Changes

## Best Possible Solution

Keep this as stacked docs work and merge it only after the matching SDK contracts land, or recast the follow-up-only sections as proposed/RFC material outside stable plugin docs until those APIs exist on `main`.

Yes for the review findings: fetch the latest PR head docs and compare the documented methods and diagrams against current `OpenClawPluginApi`, `buildPluginApi`, and the `sessions.pluginPatch` handler on `main`. The mismatch is visible by static source inspection without running the docs build.

## Solution Assessment

No, not as a standalone merge into current `main`. Splitting the large recipe page is a reasonable review strategy, but stable public docs should match shipped SDK surfaces or clearly mark future APIs as proposed until the lower stack lands.

- **[P2] Keep follow-up-only APIs out of stable docs:** `docs/plugins/host-hooks-examples.md:1290`
  - body: This page is wired into the stable Building plugins docs, but sections 11-14 still document `api.registerSessionAction`, `api.sendSessionAttachment`, `api.scheduleSessionTurn`, and `api.emitAgentEvent`, none of which exist in current `OpenClawPluginApi` or `buildPluginApi`.   - confidence: 0.95
- **[P2] Fix the pluginPatch projection timing diagram:** `docs/plugins/host-hooks-examples.md:182-184`

Current `sessions.pluginPatch` only returns `{ ok, key, value }` and emits `sessionsChanged`; projection happens when session rows are rebuilt later, so the diagram/text should not teach clients to expect projected state in the patch response.
  - confidence: 0.9
