## Summary

Keep this PR open. The docs split is useful, but the latest head still publishes stable plugin docs for session-entry slots and workflow APIs that are absent from current main, plus a few behavior diagrams/matrices that do not match the current Gateway and cleanup contracts.

## What This Changes

The PR adds a host-hook recipes page, links it into plugin docs navigation, adds zh-CN glossary entries, and updates host-hook reference prose for session-entry slot mirrors.

## Best Possible Solution

Keep the recipe split, but publish it only after the matching SDK/API implementation lands, or move/remove future-only sections so stable docs describe current main accurately.

## Reproduction Assessment

Yes. Compare the PR head docs with current main's `PluginSessionExtensionRegistration`, `OpenClawPluginApi`, `buildPluginApi`, `sessions.pluginPatch`, and cleanup code; the documented fields, methods, response semantics, and restart matrix do not match.

## Solution Assessment

No, not as a standalone merge into stable docs. The safer solution is either stack-order gating after the implementation PRs land or a docs edit that clearly separates proposed companion seams from current SDK reference material.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Remove unshipped session-entry slot docs:** `docs/plugins/hooks.md:264-283`
  - body: This paragraph publishes `sessionEntrySlotKey` and `sessionEntrySlotSchema` as stable `registerSessionExtension` fields, but current main's `PluginSessionExtensionRegistration` does not define them and the host has no slot-mirror implementation. Plugin authors copying this will hit TypeScript/runtime dead ends unless the matching SDK code lands first or this is moved to proposed docs.
  - confidence: 0.96
- **[P2] Keep future workflow APIs out of stable recipes:** `docs/plugins/host-hooks-examples.md:1344`
  - body: Sections 11-14 still document `api.registerSessionAction`, `api.sendSessionAttachment`, `api.scheduleSessionTurn`, and `api.emitAgentEvent` as callable SDK methods, but none exist on current main's `OpenClawPluginApi` or in `buildPluginApi`. Because this page is added to stable docs navigation, merging now would publish examples that cannot type-check or run.
  - confidence: 0.96
- **[P2] Align the pluginPatch response diagram:** `docs/plugins/host-hooks-examples.md:357-367`
  - body: The diagram says an unknown extension returns `404`, that the store invokes `project()` during the patch, and that Gateway returns an updated session row. Current main returns a JSON-RPC `INVALID_REQUEST`, `patchPluginSessionExtension` only writes and returns `{ ok, key, value }`, and projection happens later when session rows are built.
  - confidence: 0.92
- **[P2] Correct restart run-context cleanup semantics:** `docs/plugins/host-hooks-examples.md:1927-1932`
  - body: The cleanup matrix says `restart` preserves run context, but current `runPluginHostCleanup` still calls `clearPluginRunContext` for the plugin/run after scheduler cleanup, even when the reason is `restart`. This would teach plugin authors to rely on run context that the host actually clears.
  - confidence: 0.9
