## Summary

Keep this PR open. Current main does not already contain the new recipe page, and the PR is not safe to merge as a standalone docs change because it publishes stable plugin docs that still reference follow-up-only SDK APIs and several stale host-hook behavior details. The dependent implementation stack named in the PR body has shifted: #73384 and #74483 are closed unmerged, while narrower split implementation PRs are still open.

## What This Changes

The PR adds a host-hook examples and recipes page, links it into the plugin docs navigation, and adds zh-CN glossary labels for the new page.

## Best Possible Solution

Keep the recipe page stacked until the matching SDK contracts are merged, or clearly move follow-up-only sections into proposed/upcoming wording outside stable plugin docs. Before merge, align all diagrams and cleanup/drain guidance with the exact current host and Gateway behavior.

## Solution Assessment

No, not as a standalone merge into current main. Splitting the large recipe page is a useful review strategy, but stable plugin docs should match shipped SDK surfaces or be explicitly marked as companion/proposed material until the implementation PRs land.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Keep follow-up-only APIs out of stable docs:** `docs/plugins/host-hooks-examples.md:1290`
  - body: This page is added to stable plugin docs, but it still documents `api.registerSessionAction`, `api.sendSessionAttachment`, `api.scheduleSessionTurn`, and `api.emitAgentEvent`, which are absent from current `OpenClawPluginApi` and `buildPluginApi`. If merged before the split implementation PRs, readers will copy examples that do not type-check; move these sections to clearly proposed/companion wording or wait until the APIs exist on main.
  - confidence: 0.96
- **[P2] Correct pluginPatch response and projection timing:** `docs/plugins/host-hooks-examples.md:307-313`
  - body: The diagram describes an unknown extension as a 404 and shows projection happening in the patch response path, but current `sessions.pluginPatch` returns a JSON-RPC `INVALID_REQUEST` error or `{ ok, key, value }` and only emits `sessionsChanged`; projection happens when session rows are later built. Please update the diagram so clients know the actual response and refresh behavior.
  - confidence: 0.91
- **[P2] Document inactive injection drops on drain:** `docs/plugins/host-hooks-examples.md:487-490`
  - body: This pitfall says a drain consumes queued injections including transiently inactive plugins, but the implementation only drains active loaded plugins with prompt injection enabled and then deletes the whole queue. Inactive, unloaded, or disabled plugin records are dropped, so the docs should warn authors to re-enqueue across reload/disable transitions.
  - confidence: 0.93
- **[P2] Do not claim run context survives restart:** `docs/plugins/host-hooks-examples.md:1875`
  - body: The cleanup matrix says restart preserves run context, but host cleanup clears plugin run context when cleanup runs for a plugin or run. Please change the restart row so plugin authors do not rely on per-run scratch state surviving plugin restart.
  - confidence: 0.87
