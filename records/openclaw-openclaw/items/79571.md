## Summary

Keep open. The PR addresses a valid launcher parent-death bug, but the patch currently leaks its internal guard environment into later child commands, and the external draft PR only provides test-based proof rather than after-fix proof from a real setup.

## What This Changes

The PR adds a Unix parent-death guard for launcher, compile-cache, and CLI respawn children and adds regression coverage for SIGKILL of the source-checkout launcher wrapper.

## Best Possible Solution

Keep the parent-death guard direction, consume or filter the internal parent PID marker before generic child process inheritance, add regression coverage for that nested-spawn case, and require redacted real runtime proof before merge.

## Reproduction Assessment

Yes for the focused launcher fixture path, though not for the exact systemd/root setup in this read-only review. Current main has catchable signal forwarding but no parent-death guard, and the PR describes killing the wrapper with `SIGKILL` while checking the respawn child PID.

## Solution Assessment

No, not yet. The parent-death guard is the right narrow direction, but the patch should consume or filter the internal parent PID env before generic child process inheritance and still needs real behavior proof from the contributor.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Consume the parent PID guard env:** `src/process/child-process-bridge.ts:64`
  - body: This guard reads `OPENCLAW_RESPAWN_PARENT_PID` but leaves it in `process.env`. The command runner builds child environments from `process.env`, so any later subprocess that invokes `openclaw` from a respawned gateway inherits the stale wrapper PID; its guard will see its actual parent instead of the old wrapper and exit immediately. Please consume/delete this env var after arming the guard, or strip it from generic child command envs, and cover a nested `openclaw` spawn.
  - confidence: 0.86
