## Summary

Keep open: current main does not implement this new workspace path surface, and the protected XL feature PR still has three P2 resolve/edit symmetry bugs in the latest head.

## What This Changes

The branch adds a new `src/oc-path/` addressing substrate, `openclaw path` CLI handlers/wiring, CLI docs, tests, and a changelog entry for md/jsonc/jsonl/yaml workspace file addressing.

## Best Possible Solution

Fix the negative numeric key read/write symmetry in the three edit paths, then let maintainers decide whether the broad substrate should land as one exceptional PR or be split around the canonical workspace-addressing discussion.

## Reproduction Assessment

Yes for the PR findings: source inspection of latest head shows concrete paths where the resolvers accept negative numeric map/object keys but the edit paths reject the same address. This is not a current-main bug because the feature surface is introduced by this PR.

## Solution Assessment

No, not as-is. The narrow fix is to preserve `$first`/`$last` positional resolution while letting `-N` fall through to literal map/object-key lookup on keyed containers, matching the resolver contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve negative JSONC object keys during writes:** `src/oc-path/jsonc/edit.ts:98-105`
  - body: `resolveJsoncOcPath` intentionally lets `-N` fall through to literal object-key lookup for negative IDs, but this edit path returns `unresolved` when positional resolution returns null on objects. A path can resolve and then fail to update; fall through to literal lookup for `-N` on keyed containers while keeping `$first`/`$last` resolution.
  - confidence: 0.93
- **[P2] Preserve negative JSONL object keys during writes:** `src/oc-path/jsonl/edit.ts:97-104`
  - body: The JSONL resolver has the same negative-key fallback inside line values, but the edit path treats any keyed `-N` segment as a failed positional token. A line object containing a key like `"-5028303500"` will resolve but cannot be updated; match the resolver by falling through to literal lookup on objects.
  - confidence: 0.91
- **[P2] Preserve negative YAML map keys during writes:** `src/oc-path/yaml/edit.ts:160-163`
  - body: `resolveYamlOcPath` allows `-N` on maps to fall through to literal-key lookup, but `resolvePositionalSegments` aborts before `doc.hasIn` can try that key. YAML paths with negative map keys are readable but not writable; let `-N` fall through for maps while still resolving `$first` and `$last`.
  - confidence: 0.9
