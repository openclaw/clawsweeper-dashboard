## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Quote path segments when formatting:** `src/oc-path/oc-path.ts:219-223`
  - body: `formatOcPath` currently appends raw slot values, so a constructed path containing a key like `github-copilot/claude-opus-4.7-1m-internal` emits extra `/` segments and will not parse back to the same address. The PR advertises quoted segments and round-trip formatting; each slot needs to quote or encode special characters before concatenating.
  - confidence: 0.91
- **[P2] Find the query delimiter outside quoted segments:** `src/oc-path/oc-path.ts:137-140`
  - body: `parseOcPath` splits the query at the first `?` before quote-aware path parsing. A valid advertised path such as `oc://config/"foo?bar"` is therefore treated as a query boundary instead of a literal key, so quoted keys containing `?` cannot work.
  - confidence: 0.89
- **[P2] Unquote JSONC insertion parent segments:** `src/oc-path/universal.ts:736-739`
  - body: JSONC insertion first validates the parent with `resolveJsoncOcPath`, which handles quoted segments, but `mutateJsoncContainer` then re-splits the same parent with plain `.split('.')` and compares raw quoted text. Inserting under a key containing `/` or `.` can be reported as a valid insertion point and then fail as unresolved.
  - confidence: 0.88
- **[P2] Unquote YAML edit path segments:** `src/oc-path/yaml/edit.ts:107-111`
  - body: `resolveYamlOcPath` uses the quote-aware splitter, but `setYamlOcPath` builds `pathSegments` with plain `.split('.')`. A quoted YAML key can resolve through the public verb but fail when the same path is edited or used as an insertion parent.
  - confidence: 0.87
