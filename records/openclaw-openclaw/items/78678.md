## Summary

Keep open: this is a protected, externally authored XL feature/API PR that is not implemented on current main, and the branch still has concrete P2 correctness gaps around quoted JSONC/JSONL paths even after the sentinel fix and added proof.

## What This Changes

The PR adds a new `src/oc-path` workspace addressing subsystem with parsers, emitters, resolve/find/set verbs, sentinel guarding, fixtures, and tests for Markdown, JSONC/JSONL, and YAML.

## Best Possible Solution

Keep the design thread and this PR open, fix the quoted-segment write/find gaps, then have maintainers decide whether this full substrate or the narrower resolver PR is the right landing path.

## Reproduction Assessment

Yes for the PR review findings: source inspection shows quoted JSONC/JSONL paths are resolved with unquoting but edited or expanded-find-walked with raw `.split('.')`/literal comparisons. Reproduction is not applicable as a user bug report because this PR adds a new feature surface.

## Solution Assessment

No. The direction may be useful, but the proposed patch is not ready until quoted segment semantics are consistent across resolve, find, and set, and maintainers settle the API scope against the open design thread and narrower related PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle quoted JSONC segments when editing:** `src/oc-path/jsonc/edit.ts:50-53`
  - body: `resolveJsoncOcPath` accepts quoted keys such as `"anthropic/claude-opus-4-7"`, but `setJsoncOcPath` re-splits the original path with plain `.split('.')` and later compares object keys to the still-quoted segment. That means the advertised write path can resolve a provider/model key and then return `unresolved` when asked to edit it.
  - confidence: 0.91
- **[P2] Handle quoted JSONL field segments when editing:** `src/oc-path/jsonl/edit.ts:55-57`
  - body: JSONL resolution unquotes bracket-aware field segments, but the edit path uses plain `.split('.')` and raw key comparison for fields inside a line. A concrete quoted path under `Lnnn` can resolve but cannot be updated, so JSONL does not actually share the quoted-segment write contract.
  - confidence: 0.89
- **[P2] Unquote JSONC literals during expanded find walks:** `src/oc-path/find.ts:414-417`
  - body: When a pattern needs expansion, the JSONC walker bypasses `resolveJsoncOcPath` and compares object keys to `cur.value` directly. Patterns like `oc://config/agents.defaults.models/"github-copilot/claude-opus-4.7-1m-internal"/{alias,contextWindow}` therefore return no matches even though the quoted literal resolves in the direct path.
  - confidence: 0.88
