## Summary

Close as support/configuration: current main intentionally keeps normal group/channel final replies private in tool-only mode, and the reported `didSendViaMessagingTool: false` state matches the documented diagnostic rather than a Telegram transport failure.

## What This Changes

The reporter asks for generated Telegram group assistant replies to be posted back to the same group when the model produced final text but did not call the messaging tool.

## Best Possible Solution

Keep the documented tool-only group default for this report; configure the Peony deployment with `messages.groupChat.visibleReplies: "automatic"` or use a model/runtime that reliably calls `message(action=send)`, while broader default-UX changes stay in the related maintainer discussion.

## Reproduction Assessment

Yes, at source level: current main resolves ordinary group/channel turns to `message_tool_only`, and dispatch tests show final replies are not queued when no messaging-tool send occurs. I did not perform a live Telegram run against the private Peony setup.

## Solution Assessment

Yes: for this report, the narrow maintainable answer is configuration/model guidance because the observed `didSendViaMessagingTool: false` outcome is documented behavior. A default behavior change would be a separate product discussion, not a narrow Telegram bug fix.
