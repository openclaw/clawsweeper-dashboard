## Summary

The reported Telegram group silence matches the documented and implemented group visible-reply policy, not a Telegram transport failure: normal final answers in group/channel rooms stay private unless the model calls the message tool or the operator opts back into automatic visible replies.

## What This Changes

The issue asks OpenClaw to post a generated Peony Telegram group reply back to the same group when the trajectory shows `didSendViaMessagingTool: false`.

## Best Possible Solution

Keep the documented tool-only group default; configure this Peony deployment with `messages.groupChat.visibleReplies: "automatic"` or use a model/runtime that reliably calls `message(action=send)`.

## Reproduction Assessment

Yes for the observed symptom from source inspection: Telegram group messages are marked `ChatType: "group"`, which defaults to `message_tool_only` visible replies and suppresses normal final-answer delivery. I did not run a live Telegram bot, but the provided `didSendViaMessagingTool: false` trajectory exactly matches the documented path.

## Solution Assessment

Yes; closing as configuration/support is the narrowest maintainable outcome because current code and docs already define this as expected group-room behavior. A source bug would be actionable if `message(action=send)` or `messages.groupChat.visibleReplies: "automatic"` fails for the same group target.
