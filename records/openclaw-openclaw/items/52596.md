## Summary

Keep open. Current main has the pre-tool choke point, trusted policy surface, and session reset lifecycle, but it still does not implement the requested framework-owned session PIN gate: no `session.auth.pin` config/schema field, no core authenticated flag on `SessionEntry`, and no trusted verifier that can unlock tools outside prompt/model context.

## What This Changes

The issue requests framework-level session PIN authentication that blocks all tool calls in a new or reset session until a trusted TypeScript verifier marks that session authenticated.

## Best Possible Solution

Add a documented core session-auth capability: schema/config for the secret source, a framework-owned authenticated state on `SessionEntry`, reset clearing on `/new` and `/reset`, pre-tool enforcement before ordinary plugin hooks, and a trusted command/gateway/internal verifier that compares PIN attempts in TypeScript while keeping attempts out of transcripts and logs.

## Reproduction Assessment

Not applicable as a conventional runtime bug. The missing feature is high-confidence by source inspection: the proposed config key is not in the strict schema, `SessionEntry` has no auth state, and the pre-tool pipeline has no session PIN check.

## Solution Assessment

Unclear on exact UX, but the enforcement boundary is correct: a trusted pre-tool/session-state gate is the narrow maintainable place to get prompt-injection resistance. A prompt-only or external-plugin-only workaround would not provide the requested security property.
