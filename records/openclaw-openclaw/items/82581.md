## Summary

This should stay open because current main has a real downgrade-safety boundary gap: newer-config service operations and service-mode startup are guarded, but Docker launches the foreground gateway path where that hard halt is not clearly applied. The exact SQLite split-state/corruption scenario is not proven, and the requested layout-marker halt needs maintainer policy judgment before changing downgrade and recovery behavior.

## What This Changes

The issue requests boot-time data-layout or version gating so older OpenClaw runtimes fail safely instead of starting against a data volume written by a newer workspace/config layout.

## Best Possible Solution

Define the supported downgrade and recovery contract, then extend the existing meta.lastTouchedVersion future-config guard or add a layout marker only where it reliably protects Docker and foreground gateway startup without blocking documented recovery.

## Reproduction Assessment

No, there is not a high-confidence live reproduction of the reported 2026.5.7 Docker downgrade or SQLite split-state behavior. Source inspection does show a plausible current-main gap because Docker uses foreground gateway startup while the hard future-version halt is gated on OPENCLAW_SERVICE_MARKER.

## Solution Assessment

Unclear; the requested .openclaw_version-style marker is not clearly the narrowest fix because current OpenClaw already stamps config version metadata and guards managed service mutations. The maintainable direction is to decide whether the existing future-version guard should also block container/foreground gateway startup, then cover that behavior with Docker-focused proof.
