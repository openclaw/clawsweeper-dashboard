## Summary

Keep open for maintainer policy review. Current code has a shipped newer-config guard, but it still only hard-stops service-mode gateway startup; Docker/foreground startup uses the same gateway path without `OPENCLAW_SERVICE_MARKER`, while the exact split-state and SQLite-in-workspace claim is not proven by current source or docs.

## What This Changes

The issue asks OpenClaw to detect newer data or layout signatures during downgrade startup and halt safely instead of allowing split-state execution.

## Best Possible Solution

Define the supported downgrade and recovery contract, then either extend the existing `meta.lastTouchedVersion` guard to the Docker/foreground startup path or add a narrowly documented layout marker with an explicit recovery override.

## Reproduction Assessment

No high-confidence live reproduction of the reported v2026.5.7 Docker split-state scenario was established. Source inspection does show the narrower current-main gap: the hard newer-config startup block is gated on `OPENCLAW_SERVICE_MARKER`, while Docker starts the foreground gateway path without that marker.

## Solution Assessment

Unclear; the requested `.openclaw_version` marker is a plausible direction but not proven to be the best fix. Reusing the existing config version guard for Docker/foreground startup may be narrower if maintainers decide startup should fail closed on downgrade.
