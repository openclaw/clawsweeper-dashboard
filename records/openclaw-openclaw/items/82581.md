## Summary

Keep open: current main has a shipped future-config guard for service operations, but Docker foreground startup still runs through an unmarked gateway path, and deciding whether to hard-fail downgrades or add a layout marker is a compatibility/product call.

## What This Changes

The issue requests a startup/layout-version guard so Docker or foreground downgrades do not run an older binary against a newer OpenClaw state/workspace layout.

## Best Possible Solution

Define the supported downgrade/recovery contract, then extend the existing meta.lastTouchedVersion guard or add a layout marker only where it reliably protects Docker and foreground startup with a documented override.

## Reproduction Assessment

No high-confidence live reproduction of the reported 2026.5.7 Docker downgrade was established. Source inspection does reproduce the relevant guard gap: Docker starts an unmarked foreground gateway while the hard newer-config halt only fires when OPENCLAW_SERVICE_MARKER is set.

## Solution Assessment

Unclear; the requested .openclaw_version marker may not be the narrowest fix because OpenClaw already stamps meta.lastTouchedVersion and guards service mutations. A maintainer-approved extension of the existing guard, covered by Docker-focused proof, is the safer direction.
