## Summary

Keep open for maintainer policy review: current OpenClaw has a newer-config guard, but source inspection still leaves the reported Docker/foreground downgrade path outside the hard service-mode startup halt, and the exact SQLite split-state incident has not been reproduced.

## What This Changes

Request startup/version-layout gating so downgrades from newer workspace/state layouts fail safely instead of allowing Docker or foreground gateway runs to split state.

## Best Possible Solution

Define the supported downgrade contract, then reuse meta.lastTouchedVersion or add a layout marker only where it can reliably protect Docker/foreground gateway startup without blocking documented recovery.

## Reproduction Assessment

No high-confidence live reproduction exists for the full reported downgrade and SQLite corruption path. Source inspection does show a narrower current-main gap: Docker starts the foreground gateway path without OPENCLAW_SERVICE_MARKER, while the hard future-config startup halt is service-marker gated.

## Solution Assessment

Unclear; the requested .openclaw_version marker or symlink routing is not obviously the narrowest fix because current OpenClaw already has config-version metadata and service/destructive-action guards. The safer path is a maintainer downgrade-policy decision, then a targeted extension of that guard if Docker/foreground startup should be treated as destructive.
