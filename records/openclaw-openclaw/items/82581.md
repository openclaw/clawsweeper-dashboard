## Summary

This should stay open for maintainer review: current OpenClaw already has a shipped future-config guard for newer-version split-brain service operations, but the reported Docker/foreground downgrade path is not clearly covered and the proposed layout-version halt is a product/compatibility decision rather than a narrow bug fix.

## What This Changes

The issue requests boot-time directory or layout-version gating to prevent older OpenClaw runtimes from using a data tree created by a newer release.

## Best Possible Solution

Define the supported downgrade contract, then extend the existing meta.lastTouchedVersion/future-config guard or add a state-layout marker only where it can reliably protect Docker and foreground gateway startup without blocking documented recovery.

## Reproduction Assessment

No, there is not a high-confidence exact reproduction of the reported 2026.5.7 Docker downgrade. Source inspection does show a relevant gap: current Docker starts the foreground gateway path, while the hard future-version startup halt only applies when OPENCLAW_SERVICE_MARKER is set.

## Solution Assessment

Unclear; the requested .openclaw_version-style marker is not obviously the best fix because current OpenClaw already stamps config version metadata and guards service mutations. A safer maintainable direction is to decide whether that existing guard should also block container/foreground gateway startup and then cover it with Docker-focused tests/docs.
