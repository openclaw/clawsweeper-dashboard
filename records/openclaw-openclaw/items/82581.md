## Summary

Current main only partially covers the reported downgrade safety problem: it has a shipped `meta.lastTouchedVersion` guard for service mutations, service-mode startup, and `gateway --force`, but Docker runs the foreground gateway command and current code does not hard-stop that path when the config was written by a newer binary. The exact 2026.5.7 Docker downgrade was not reproduced here, so this should stay open for maintainer policy on downgrade/recovery semantics rather than be auto-closed or auto-fixed.

## What This Changes

The issue requests a downgrade/layout-version startup guard to prevent newer OpenClaw state directories from being used by older runtimes.

## Best Possible Solution

Define the supported downgrade contract, then extend the existing future-config guard or add a layout marker only for startup paths maintainers want to fail closed, with Docker-focused proof and recovery docs.

## Reproduction Assessment

No, not as an exact live repro: the reported 2026.5.7 Docker downgrade was not recreated. Source inspection does show a high-signal path gap because Docker uses foreground gateway startup while the hard newer-config startup block is gated on `OPENCLAW_SERVICE_MARKER`.

## Solution Assessment

Unclear; the requested `.openclaw_version` marker is one plausible fix, but current main already has `meta.lastTouchedVersion` and service-operation guards. The maintainable solution is likely to reuse or deliberately extend that existing guard once maintainers decide foreground/Docker downgrade policy.
