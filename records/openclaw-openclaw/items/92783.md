## Summary

Keep open: current main and v2026.6.8 already provide cron manual runs and a manual heartbeat wake, but they still exercise real scheduler/model/delivery paths and do not provide the requested deterministic no-spend dry-run or named heartbeat-run diagnostic contract.

## What This Changes

Requests a scheduler diagnostic interface to force-run named cron or heartbeat scheduled actions, or dry-run due dispatch without model spend or side-effecting sends.

## Best Possible Solution

Define a narrow Gateway/CLI scheduler diagnostics contract that keeps existing cron.run behavior intact while adding explicit no-model dry-run or heartbeat-run planning semantics after maintainer API review.

## Reproduction Assessment

Yes, at source level. Current main exposes cron manual runs and manual heartbeat wake, but the inspected CLI, Gateway, cron service, and heartbeat runner paths show no deterministic no-model dry-run or named heartbeat scheduled-action run interface.

Unclear. The requested direction is plausible, but the best implementation needs maintainer design around public RPC/CLI shape, no-side-effect guarantees, run logging, hooks, and delivery rerouting.
