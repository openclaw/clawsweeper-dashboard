## Summary

Keep open: current main and v2026.6.6 cover cron manual runs and manual heartbeat wake, but there is still no deterministic no-model scheduler dry-run contract. The remaining request needs a core scheduler/API product decision, not conservative cleanup.

## What This Changes

Request a scheduler force-run or dry-run interface that lets supervisors exercise cron and heartbeat dispatch deterministically without waiting for ambient ticks or paying for a live agent turn.

## Best Possible Solution

Define a narrow Gateway and CLI scheduler diagnostics contract that preserves existing `cron.run` and `system event --mode now` behavior while adding explicit no-model dry-run or heartbeat-run planning semantics after maintainer API review.

## Reproduction Assessment

Yes, at source level. Current main exposes cron manual run and immediate heartbeat wake, but the inspected CLI, protocol, docs, and tests do not expose a deterministic no-model dry-run path for scheduled cron/heartbeat dispatch.

## Solution Assessment

Unclear. The requested direction is plausible, but the best solution needs maintainer design around public RPC/CLI shape, no-side-effect guarantees, run-log semantics, plugin hooks, and delivery rerouting.
