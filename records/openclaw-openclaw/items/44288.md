## Summary

Keep open. Current main still lacks the requested first-party backup exclude CLI surface, and this PR remains a plausible implementation path, but the PR head is conflicting, lacks required real behavior proof, and still has concrete backup filtering/performance bugs that should block merge.

## What This Changes

The PR adds backup exclude-pattern CLI flags, `.backupignore`/exclude-file handling, smart defaults, protected-path guards, manifest `excludedStats`, the `ignore` dependency, and backup create/verify tests.

## Best Possible Solution

Land a rebased or narrower native exclusion implementation that applies patterns consistently across intended backup roots, preserves ordered gitignore semantics, avoids full normal-create rescans, updates backup docs, and includes redacted real-run proof.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the external-asset bypass, ordered-rule deduplication, and unconditional archive readback path. I did not run a live backup because this is a read-only review.

## Solution Assessment

No. Native excludes are a reasonable direction, but this PR should not merge until filter semantics, create-path performance, current-main integration, docs, and real behavior proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply exclude rules per asset root:** `src/infra/backup-exclude.ts:358-363`
  - body: `buildExcludeFilter` returns `true` for paths outside its single `baseDir`, and `createBackupArchive` passes only the state directory as that base. External workspaces or custom config/credential roots included by the backup plan still ignore `--exclude`, `.backupignore`, and `--smart-exclude`; a warning does not make the archive match the user's filter or protect sensitive external files.
  - confidence: 0.9
- **[P2] Preserve ordered ignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: This deduplication removes later duplicate rules before `ignore` evaluates them, but gitignore patterns are order-sensitive with negations. A list like `*.log`, `!important.log`, `*.log` should end with `important.log` excluded; dropping the final rule changes what gets archived.
  - confidence: 0.88
- **[P2] Avoid rescanning every archive on create:** `src/infra/backup-create.ts:525-535`
  - body: The post-create check reads the entire `.tar.gz` and materializes every entry before publishing, even when `--verify` was not requested. On the multi-GB backups this feature targets, that adds a second full archive pass and memory proportional to entry count to normal create.
  - confidence: 0.84

## Security Review
