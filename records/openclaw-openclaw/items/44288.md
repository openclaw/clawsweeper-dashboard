## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the current backup writer protections:** `src/infra/backup-create.ts:469-475`
  - body: The exclude-enabled path tars `result.assets` directly, so it skips current main's SQLite snapshot/remap flow, volatile-file filter, extension `node_modules` filter, tar EOF retry wrapper, and hardlink cache behavior. A backup created with excludes can therefore archive raw live SQLite state or restore-hostile entries even though the no-exclude path remains protected.
  - confidence: 0.92
- **[P2] Restore the manifest size cap while verifying:** `src/commands/backup-verify.ts:202-210`
  - body: This manifest reader buffers the entire entry and does not preserve current main's 1 MiB manifest limit, so a malformed or hostile archive can force `backup verify` to allocate arbitrarily large memory before parsing. Keep the bounded read while adding tolerant `excludedStats` parsing.
  - confidence: 0.86
- **[P2] Apply excludes to every backed-up asset root:** `src/infra/backup-exclude.ts:358-363`
  - body: Returning `true` for paths outside the state-dir base means external workspaces, custom config files, or external credential directories bypass `--exclude` and `.backupignore` entirely. A warning still leaves sensitive or oversized external assets in the archive despite explicit user rules, so the filter needs per-asset roots or a clearly narrower API before merge.
  - confidence: 0.88
- **[P2] Preserve gitignore rule order when duplicates appear:** `src/infra/backup-exclude.ts:225-233`
  - body: The resolver drops later duplicate patterns before passing rules to `ignore`, but gitignore semantics are order-sensitive. A sequence like `*.log`, `!important.log`, `*.log` should end with `important.log` excluded, while this dedupe drops the final rule and changes the user's declared result.
  - confidence: 0.82
