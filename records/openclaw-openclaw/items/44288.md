## Summary

Keep open: the feature direction is useful, but this PR is not merge-ready because the branch still replaces current backup write hardening, leaves explicit excludes ineffective for external backup assets, and lacks contributor real behavior proof.

## What This Changes

The branch adds `openclaw backup create` exclude flags and `.backupignore` handling, records `excludedStats` in backup manifests, adds the `ignore` dependency, and expands backup exclude tests.

## Best Possible Solution

Rebase and narrow the exclusion feature onto the current main backup writer, compose user patterns with existing volatile and plugin-dependency filters, define per-asset exclude semantics, and require redacted real-run proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection of current main and the PR head shows the stale writer regression and external-asset exclude bypass. I did not run a live backup because this review is read-only.

## Solution Assessment

No. The requested native exclude feature is useful, but this branch is not the best implementation until it preserves current backup hardening and applies or explicitly designs excludes for every included asset base.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Compose excludes into the current backup writer:** `src/infra/backup-create.ts:469-475`
  - body: The exclude path writes the payload with direct `tar.c` calls instead of composing with current main's `writeTarArchiveWithRetry`, volatile-file skip filter, installed-plugin `node_modules` filter, and temp-root guard. That regresses live backup reliability and archive scope, and the no-exclude path below also loses the current backwards-compatible writer behavior.
  - confidence: 0.91
- **[P2] Apply exclude rules to external assets:** `src/infra/backup-exclude.ts:358-364`
  - body: Returning `true` for every path outside `stateDir` makes `--exclude`, `.backupignore`, and smart excludes no-op for external workspaces or custom config assets that are still included in the backup. A warning is not enough when the archive can still contain files such as `.env`, `*.pem`, or large dependency trees that the user explicitly excluded.
  - confidence: 0.88
- **[P2] Preserve duplicate gitignore rules in order:** `src/infra/backup-exclude.ts:225-233`
  - body: The resolver drops later duplicate patterns before handing them to `ignore`, but gitignore rules are order-sensitive. A sequence like `*.log`, `!important.log`, `*.log` intentionally re-excludes the file; deduping the final rule changes the user's declared filter.
  - confidence: 0.86
- **[P2] Avoid scanning every archive entry during create:** `src/infra/backup-create.ts:528-535`
  - body: This unconditional post-create check reads the whole `.tar.gz` and stores every entry path in memory even when the user did not ask for `--verify`. On the large backup trees this PR is meant to help, that adds a second full archive pass and O(entry count) memory pressure before the archive can be published.
  - confidence: 0.82
