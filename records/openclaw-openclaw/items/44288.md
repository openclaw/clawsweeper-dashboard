Full review comments:

- **[P1] Compose excludes with the current backup writer:** `src/infra/backup-create.ts:469-476`
  - body: Current main wraps archive creation in retry/backoff, volatile-state filtering, temp-root placement, and installed-plugin dependency pruning. This branch starts a separate raw tar/gzip path when excludes are active, so merging it as-is can reintroduce live-backup stalls, duplicate manifest risk, and oversized plugin dependency archives.
  - confidence: 0.94
- **[P1] Apply excludes to external backup assets:** `src/infra/backup-exclude.ts:358-363`
  - body: Returning `true` for paths outside `baseDir` means custom workspaces, config paths, or credential roots outside `stateDir` ignore every `--exclude` and `.backupignore` rule. A warning still produces an archive containing files the user explicitly asked to omit, which is unsafe for a backup privacy feature.
  - confidence: 0.92
- **[P2] Keep ordered duplicate ignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: Deduplicating the rule list before handing it to `ignore` changes gitignore precedence when a later duplicate intentionally overrides a negation, such as `*.log`, `!important.log`, `*.log`. Backup filtering should preserve the user’s ordered rule stream instead of changing inclusion results.
  - confidence: 0.88
- **[P2] Avoid rescanning the full archive in the create path:** `src/infra/backup-create.ts:528-535`
  - body: `listArchiveEntries` reads the finished archive and stores every entry path in memory on every backup, adding a second full pass and O(entry count) memory use. That can materially worsen the large-state backup stalls this PR is meant to address; verification should stream or stay behind explicit `--verify`.
  - confidence: 0.86
