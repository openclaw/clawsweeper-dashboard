## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Apply excludes to every backed-up asset root:** `src/infra/backup-create.ts:364-368`
  - body: `buildExcludeFilter()` is constructed with `canonicalStateDir`, and paths outside that base are treated as included, so custom workspaces or config/credential assets outside `~/.openclaw` bypass `--exclude`, `--exclude-file`, and `.backupignore`. A warning still leaves users with unexpectedly large or sensitive files in archives despite explicit exclude rules; the fix should apply patterns per asset root or narrow the documented contract before merging.
  - confidence: 0.91
- **[P2] Avoid rescanning the whole archive in create:** `src/infra/backup-create.ts:528-529`
  - body: After writing the archive, this always calls `listArchiveEntries()` and stores every entry path in memory before publishing. On the large backups this feature targets, that adds a second full `.tar.gz` pass and O(entry count) memory even when the user did not request `--verify`; use streaming validation, check only the write path's known entries, or keep the expensive scan behind verification.
  - confidence: 0.84
- **[P2] Preserve rule order when resolving excludes:** `src/infra/backup-exclude.ts:225-233`
  - body: This deduplicates repeated patterns before passing them to `ignore`, but gitignore rules are order-sensitive and repeated rules can intentionally override an intervening negation. For example `*.log`, `!important.log`, `*.log` should end with `important.log` excluded, but first-occurrence dedupe drops the final rule and changes the user's declared result.
  - confidence: 0.88
