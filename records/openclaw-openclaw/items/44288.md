## Summary

Keep open. Current main still lacks native backup exclude support and this PR contains unique implementation work, but the branch is not merge-ready because source inspection shows order-sensitive ignore rules are changed, external assets still bypass excludes, create now always rescans the archive, and contributor proof is only narrative.

## What This Changes

The PR adds `backup create` exclude flags, smart defaults, `.backupignore` and exclude-file handling, protected-path guards, manifest `excludedStats`, the `ignore` dependency, and backup tests.

## Best Possible Solution

Rework the branch so exclude matching preserves gitignore order, applies per backed-up asset or fails when it cannot, keeps integrity checks behind verify or a streaming check, and ships with redacted real-run proof.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the duplicate-rule collapse, external-asset bypass, and unconditional archive readback paths. I did not run a live backup because this is a read-only review.

## Solution Assessment

No. Native excludes are a reasonable direction, but this branch should not merge until the filter semantics and archive-performance regression are fixed or explicitly narrowed by maintainers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply excludes to assets outside the state directory:** `src/infra/backup-exclude.ts:358-363`
  - body: `buildExcludeFilter` still returns `true` for any path outside the single `stateDir` base, so external workspaces or custom config/credential assets are archived in full when users pass `--exclude`, `.backupignore`, or `--smart-exclude`. A warning helps visibility, but it does not make the exclude contract true for those backed-up assets.
  - confidence: 0.9
- **[P2] Keep duplicate ignore rules in order:** `src/infra/backup-exclude.ts:225-233`
  - body: `resolveExcludePatterns` removes later duplicate rules before handing them to `ignore`, but gitignore rules are order-sensitive with negations. A list such as `*.log`, `!important.log`, `*.log` should end with `important.log` excluded; this branch drops the final rule and changes what lands in the backup.
  - confidence: 0.88
- **[P2] Avoid full archive rescans during create:** `src/infra/backup-create.ts:528-535`
  - body: The post-create check calls `listArchiveEntries` and materializes every tar entry before publishing the archive, even when `--verify` was not requested. On the multi-GB backups this feature targets, that adds a second full archive pass and O(entry count) memory pressure to normal create.
  - confidence: 0.84

## Security Review
