## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate implicit .backupignore loading:** `src/infra/backup-exclude.ts:187-194`
  - body: `resolveExcludePatterns()` loads `${stateDir}/.backupignore` on every backup create, even when the user did not pass an exclude flag. That contradicts the claimed no-flag compatibility and can silently change archive contents for existing installs that already have this file, so this needs an explicit flag/default policy before merge.
  - confidence: 0.92
- **[P1] Rebase excludes onto the current backup writer:** `src/infra/backup-create.ts:469-476`
  - body: The exclude path creates the archive directly from `result.assets`, but current main's writer now stages live SQLite snapshots, filters volatile state and extension dependency trees, retries tar EOF races, and remaps snapshot paths. Merging a conflict resolution that keeps this alternate path would regress backup safety for live state.
  - confidence: 0.88
- **[P2] Apply excludes to external backup assets:** `src/infra/backup-exclude.ts:358-364`
  - body: Paths outside the single `stateDir` base return `true`, so explicit patterns do not apply to external workspaces or custom config locations. The later warning tells users about the gap but still archives those assets in full despite `--exclude`, which leaves the feature inconsistent for common backup sources.
  - confidence: 0.87
- **[P2] Preserve ordered duplicate ignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: Deduplicating patterns before handing them to `ignore` changes gitignore semantics because later duplicate rules can intentionally override an intervening negation, such as `*.log`, `!important.log`, `*.log`. Preserve the ordered rule list instead of removing later duplicates.
  - confidence: 0.86
- **[P2] Avoid full archive scans during create:** `src/infra/backup-create.ts:528-535`
  - body: After writing the archive, `listArchiveEntries()` materializes every tar entry into memory before publish. On the multi-GB backups this feature targets, that second full pass can materially increase runtime and memory pressure; keep full scans in verify or validate required roots while streaming.
  - confidence: 0.84
