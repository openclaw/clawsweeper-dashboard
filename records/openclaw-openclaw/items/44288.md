## Summary

Keep open. Current main still does not provide user-configurable backup exclude flags, but this PR head remains unsafe to merge because it is conflicting, lacks real behavior proof, and still has backup data-loss/security-sensitive correctness issues against current main.

## What This Changes

The PR adds `backup create` exclude-pattern flags, smart defaults, protected-path guards, `.backupignore` loading, manifest exclusion stats, and backup create/verify tests.

## Best Possible Solution

Rework or replace this branch with a rebased native exclude implementation that composes with current backup retry/volatile/plugin-dependency filters, applies rules per archived root, preserves ordered gitignore semantics, avoids normal create-time full archive rescans, updates public docs, and includes redacted real-run proof.

## Reproduction Assessment

Yes, at source level for the review findings: PR-head code shows raw tar writes, external-root bypass warnings, duplicate-rule collapse, and unconditional archive readback. I did not run a live backup because this was a read-only review.

## Solution Assessment

No. Native backup excludes are the right direction, but this branch needs a rebase and design fixes before it is the narrowest maintainable solution for data-loss/security-sensitive backup behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve current backup hardening on archive writes:** `src/infra/backup-create.ts:469-474`
  - body: This exclude path invokes `tar.c` directly instead of composing with current main's tar EOF retry, volatile-file filter, and installed-plugin dependency filter. Rebase and route both archive paths through the current hardened pipeline, or existing live/large backup behavior can regress.
  - confidence: 0.91
- **[P2] Apply excludes per archived asset root:** `src/infra/backup-create.ts:383-387`
  - body: This warning still leaves external workspace/config assets archived in full. Patterns such as `.env*`, `*.pem`, or `logs/` will not protect those roots, so the implementation needs per-asset filter bases or an explicit fail/opt-out design before merge.
  - confidence: 0.9
- **[P2] Keep ordered ignore rules intact:** `src/infra/backup-exclude.ts:225-233`
  - body: Dropping later duplicate patterns changes gitignore precedence because negations are order-sensitive. A list like `*.log`, `!important.log`, `*.log` should end with `important.log` ignored, but this deduplication removes the final rule and flips the result.
  - confidence: 0.89
- **[P2] Avoid full archive rescans during create:** `src/infra/backup-create.ts:528-535`
  - body: `listArchiveEntries()` materializes every tar entry before publishing the archive even when the user did not request verification. For the large backups this feature targets, that adds another full archive pass and memory proportional to entry count to normal create.
  - confidence: 0.84
