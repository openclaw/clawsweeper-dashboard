## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply exclusion rules to every backup asset root:** `src/infra/backup-exclude.ts:362-363`
  - body: `createBackupArchive()` builds one filter rooted at `stateDir`, and this guard includes every `../` path before `ignore` sees it. Any external workspace or custom config asset from the backup plan therefore ignores `--exclude`/`.backupignore` and can still archive large or sensitive files despite explicit user patterns; warning about the gap does not make the backup scope correct.
  - confidence: 0.91
- **[P1] Remove the full completed-archive rescan:** `src/infra/backup-create.ts:528-529`
  - body: This reads the entire finished `.tar.gz` into an array on every backup, even when the user did not ask for `--verify`. For the multi-GB installations this feature targets, that adds a second full archive pass and O(entry count) memory after already writing an uncompressed intermediate, making stalls and resource exhaustion more likely.
  - confidence: 0.87
- **[P2] Preserve repeated rules in declared order:** `src/infra/backup-exclude.ts:225-233`
  - body: Gitignore rule order is meaningful, but this deduplicates patterns before passing them to `ignore`. A sequence like `*.log`, `!important.log`, `*.log` should end with `important.log` excluded; dropping the final duplicate changes what lands in the backup and can include or omit files contrary to the user's rule list.
  - confidence: 0.86
