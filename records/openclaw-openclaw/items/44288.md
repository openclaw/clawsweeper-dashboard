## Summary

Keep open and do not merge as-is. The exclude-pattern feature is still not on current main or the latest release, but the PR head is stale against current backup hardening and still has correctness/security gaps around exclude semantics and proof.

## What This Changes

The PR adds `openclaw backup create` exclude-pattern flags, smart exclude defaults, protected-path guards, `.backupignore` loading, and excluded-stat manifest/test coverage.

## Best Possible Solution

Rebase the exclude-pattern work onto the current backup writer, preserve existing retry/volatile/plugin-dependency filtering, fix gitignore ordering and external-asset semantics, then require redacted real-run proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection of current main versus the PR head shows the branch lacks current backup write hardening and still bypasses some declared exclude rules. I did not run a live backup because this review is read-only.

## Solution Assessment

No. The feature direction is useful, but the current patch shape is not the best solution because it layers new archive paths over stale backup code instead of integrating with the current hardened writer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Rebase the exclude path onto the current backup writer:** `src/infra/backup-create.ts:469-475`
  - body: The exclude path writes archives through direct `tar.c`/gzip calls and never applies the current main `writeTarArchiveWithRetry`, volatile-file skip filter, extension `node_modules` filter, or temp-root guard. Those behaviors were added to prevent live-write EOF failures and avoid archiving transient/plugin dependency data, so this PR would regress backup reliability and scope unless the exclude filter is composed into the current writer.
  - confidence: 0.9
- **[P2] Keep duplicate ignore rules in order:** `src/infra/backup-exclude.ts:225-233`
  - body: `resolveExcludePatterns` drops later duplicate patterns before giving them to `ignore`, but gitignore rule order is meaningful. A sequence like `*.log`, `!important.log`, `*.log` intentionally re-excludes the file; de-duplicating it changes the user’s declared backup filter and can include or exclude the wrong files.
  - confidence: 0.86
- **[P2] Apply excludes to external backup assets:** `src/infra/backup-exclude.ts:358-364`
  - body: For assets outside the state directory, the filter returns `true` before `ignore` evaluates the pattern, and `createBackupArchive` only warns. That means explicit excludes such as `*.pem` or smart excludes do not affect external workspaces/custom config paths, so those trees can still be archived in full despite the new CLI contract.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch changes backup privacy behavior and still allows external assets to bypass explicit exclude rules.

Concerns:

- **[medium] Exclude rules can miss external sensitive files:** `src/infra/backup-exclude.ts:362`
  - body: Because paths outside the state-dir base return `true`, a user-provided exclude such as `*.pem`, `.env*`, or `*.key` will not apply to external workspaces or custom config assets; the backup can still contain data the user explicitly tried to filter.
  - confidence: 0.86
