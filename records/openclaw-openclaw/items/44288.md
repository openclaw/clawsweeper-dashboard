## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve gitignore rule order:** `src/infra/backup-exclude.ts:225-233`
  - body: `ignore` rules are order-sensitive, so removing later duplicate patterns changes semantics when a negation appears between duplicates. For example, `*.log`, `!important.log`, `*.log` should end with `important.log` excluded, but this deduplication drops the final override and can archive or omit files contrary to the user's declared pattern order.
  - confidence: 0.92
- **[P1] Apply excludes to every backed-up asset root:** `src/infra/backup-exclude.ts:358-364`
  - body: Returning `true` for every path outside `stateDir` means valid backup assets such as external workspaces or custom config locations bypass `--exclude` and `.backupignore` entirely. A warning does not make the backup honor explicit patterns, so sensitive or very large external files can still be archived despite the user asking to exclude them.
  - confidence: 0.88
- **[P2] Avoid rescanning the completed archive on large backups:** `src/infra/backup-create.ts:525-529`
  - body: The exclude path already writes an uncompressed tar and gzips it; this extra `listArchiveEntries` call then reads the completed `.tar.gz` and stores every entry just to check asset roots. On multi-GB backups with many files, that second full pass can materially increase runtime and memory pressure on the same large-installation scenario this PR is trying to fix.
  - confidence: 0.8
