## Summary

Keep open. Current main still lacks native backup exclude-pattern support, and this PR remains the active implementation candidate, but the latest head still has blocking correctness, security, and performance issues before it should merge.

## What This Changes

The PR adds backup exclude-pattern CLI flags, smart defaults, protected-path guards, `.backupignore` loading, exclude stats in manifests, verification changes, tests, and the `ignore` dependency.

## Best Possible Solution

Land a corrected or narrowed backup-exclude implementation that preserves gitignore rule order, applies exclusions consistently across all archived asset roots, avoids unconditional full-archive rescans, and updates public backup docs with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection of PR head reproduces the remaining defects: ordered duplicate rules are collapsed, external assets return early from the filter, and every created archive is fully listed after writing.

## Solution Assessment

No. The direction is useful, but warning that excludes do not apply to external assets, changing gitignore rule order, and adding an unconditional full rescan are not the narrowest maintainable backup contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve ordered duplicate exclude rules:** `src/infra/backup-exclude.ts:225-233`
  - body: `resolveExcludePatterns` drops later duplicate patterns before handing rules to `ignore`, but gitignore rules are order-sensitive after negations. A list such as `*.log`, `!important.log`, `*.log` should end with `important.log` excluded; dropping the final rule flips the declared result.
  - confidence: 0.91
- **[P2] Apply exclude rules to external backup assets:** `src/infra/backup-exclude.ts:358-363`
  - body: The filter returns `true` for paths outside `baseDir`, and `createBackupArchive()` builds it with only the state directory. External workspaces or custom config/credentials paths are archived in full despite `--exclude` or `.backupignore`; warning users does not honor the filtering contract and can include sensitive files they asked to exclude.
  - confidence: 0.9
- **[P2] Avoid rescanning every archive during create:** `src/infra/backup-create.ts:525-528`
  - body: The post-create cross-check always calls `listArchiveEntries(tempArchivePath)`, including the no-exclude path that the PR says remains unchanged. That adds a second full tar pass and stores every entry path for large backups, which can materially worsen the same large-installation stall/performance problem this PR is trying to address.
  - confidence: 0.82
- **[P3] Document the new backup exclude surface:** `src/cli/program/register.backup.ts:30-41`
  - body: This adds user-facing backup flags and manifest/JSON output behavior, but the PR head leaves `docs/cli/backup.md` without the new exclude workflow or the external-asset limitation. Users need the public backup reference to describe these semantics before the CLI surface ships.
  - confidence: 0.84
