## Summary

Keep open. Current main still lacks first-party backup exclude flags, but this PR should not merge as-is because the latest head still has source-reproducible filter semantics, privacy, and create-path performance blockers plus missing real behavior proof.

## What This Changes

The branch adds `backup create` exclude flags, `.backupignore` and exclude-file handling, smart defaults, protected-path guards, manifest `excludedStats`, the `ignore` dependency, and backup tests.

## Best Possible Solution

Land a rebased or narrowed native exclusion implementation that applies patterns consistently across intended backup roots, preserves gitignore ordering, avoids extra full archive passes, documents the CLI behavior, and includes redacted real-run proof.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the external-asset bypass, duplicate-rule collapse, and unconditional archive readback path. I did not run a live backup because this is a read-only review.

## Solution Assessment

No. Native excludes are a reasonable direction, but this PR should not merge until filter semantics, create-path performance, current-main integration, docs, and real behavior proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply excludes to all backup asset roots:** `src/infra/backup-exclude.ts:358-363`
  - body: `buildExcludeFilter` returns `true` for every path outside its single `baseDir`, and `createBackupArchive` builds that filter with the state directory. External workspaces or custom config/credential assets are still archived in full when users pass `--exclude`, `.backupignore`, or `--smart-exclude`; a warning does not make the exclude contract true or protect sensitive external files.
  - confidence: 0.9
- **[P2] Preserve ordered ignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: `resolveExcludePatterns` removes later duplicate rules before handing them to `ignore`, but gitignore rules are order-sensitive with negations. A list such as `*.log`, `!important.log`, `*.log` should end with `important.log` excluded; this branch drops the final rule and changes what lands in the backup.
  - confidence: 0.88
- **[P2] Avoid full archive rescans during create:** `src/infra/backup-create.ts:528-535`
  - body: The post-create check calls `listArchiveEntries` and materializes every tar entry before publishing the archive even when `--verify` was not requested. On the multi-GB backups this feature targets, that adds a second full archive pass plus memory proportional to entry count to normal create.
  - confidence: 0.84
