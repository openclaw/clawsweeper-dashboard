## Summary

Keep open. Current main still lacks general backup exclude-pattern support, and this PR remains the active implementation candidate, but the latest head still has correctness and security blockers around ordered gitignore semantics and assets outside the state directory.

## What This Changes

This PR adds backup exclude-pattern CLI flags, `.backupignore` loading, smart exclude defaults, protected-path guards, pattern-level manifest stats, backup verification changes, tests, and the `ignore` dependency.

## Best Possible Solution

Land a version that preserves exact gitignore rule order, applies excludes consistently to every included backup asset or explicitly narrows the CLI/docs contract before merge, and keeps protected-path guards, manifest stats, tests, docs, dependency metadata, and changelog aligned.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: resolve `*.log`, `!important.log`, `*.log` and inspect `important.log`, then include an external workspace or custom config asset outside `stateDir` with any exclude pattern active.

## Solution Assessment

No. The PR direction is useful, but the current implementation is not the best fix until ordered gitignore semantics are preserved and external backup assets either receive per-asset filter bases or are deliberately excluded from the promised pattern contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve ordered duplicate exclude patterns:** `src/infra/backup-exclude.ts:225-233`
  - body: `resolveExcludePatterns` drops later duplicate rules before handing the list to `ignore`, but gitignore rules are order-sensitive. A list like `*.log`, `!important.log`, `*.log` should end with `important.log` excluded; deduping the final rule flips the result and can archive or omit files contrary to the user’s declared order.
  - confidence: 0.9
- **[P2] Apply excludes to external backup assets:** `src/infra/backup-exclude.ts:358-363`
  - body: The filter returns `true` for paths outside `baseDir`, while `createBackupArchive` builds it with only `stateDir`. External workspaces or custom config paths therefore bypass `--exclude` and `.backupignore`; warning about the bypass still archives files the user explicitly tried to filter.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff adds backup exclusion controls, but current PR head can still archive sensitive files users explicitly tried to exclude.

Concerns:

- **[medium] Exclude rules bypass external assets:** `src/infra/backup-exclude.ts:362`
  - body: When a workspace or custom config path lives outside `stateDir`, the filter includes it without applying exclude patterns. That can archive files such as `.env`, keys, logs, or model artifacts despite a user-supplied exclusion rule.
  - confidence: 0.88
- **[low] Pattern dedupe can change sensitive-file rule order:** `src/infra/backup-exclude.ts:225`
  - body: Dropping later duplicate patterns changes order-sensitive gitignore behavior after negations, which can defeat a user's intended final sensitive-file exclusion rule.
  - confidence: 0.78
