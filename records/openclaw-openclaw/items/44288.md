## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply excludes to every backup asset root:** `src/infra/backup-create.ts:383-387`
  - body: The branch detects external workspaces/config paths but only prints a warning and then archives them in full. Since `resolveBackupPlanFromDisk()` can include workspaces outside `stateDir`, a user running `--exclude '*.pem'` or `--exclude node_modules/` still gets matching files from those assets in the archive, which violates the new backup contract and can expose sensitive or huge workspace content.
  - confidence: 0.92
- **[P1] Preserve ordered gitignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: Deduplicating patterns before handing them to `ignore` changes gitignore precedence. A rule list like `*.log`, `!important.log`, `*.log` should end with `important.log` excluded, but this code drops the final duplicate and flips the result, so backups can include or exclude files contrary to the user's declared order.
  - confidence: 0.9
- **[P2] Avoid full archive scans in create:** `src/infra/backup-create.ts:528-535`
  - body: Every create now calls `listArchiveEntries()` and stores every tar entry, then scans that list for each asset even when `--verify` was not requested. On multi-GB backups this adds a second full decompression pass and O(entry count) memory pressure to the path this PR is trying to keep from stalling.
  - confidence: 0.86
- **[P2] Document the new backup exclude contract:** `src/cli/program/register.backup.ts:30-52`
  - body: These new public backup flags are only added to command help; `docs/cli/backup.md` still documents only coarse backup scope controls. Before merge, the docs need to describe `--exclude`, `.backupignore`, protected paths, `excludedStats`, and any intentional external-asset limitation so users do not create surprising or incomplete archives.
  - confidence: 0.82
