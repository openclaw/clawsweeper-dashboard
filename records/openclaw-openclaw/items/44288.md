## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Apply excludes to every backup asset root:** `src/infra/backup-exclude.ts:358-364`
  - body: `createBackupArchive` builds the exclude filter with only the state directory as `baseDir`, and this branch returns `true` for anything outside that root. Valid backup assets such as external workspaces or custom config paths are therefore archived in full even when the user passed `--exclude "*.pem"` or `--smart-exclude`; a warning still leaves the explicit exclude request ineffective for those assets.
  - confidence: 0.92
- **[P1] Preserve ordered gitignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: This deduplication drops later duplicate patterns before the list reaches `ignore`, but gitignore rules are order-sensitive. A pattern list like `*.log`, `!important.log`, `*.log` should end with `important.log` excluded; first-occurrence dedupe removes the final override and can silently include or exclude the wrong backup files.
  - confidence: 0.88
- **[P1] Run excludes through the current backup pipeline:** `src/infra/backup-create.ts:469-476`
  - body: The exclude branch creates a separate tar path over `result.assets`, so the reviewed head does not carry current main's SQLite snapshot/remap flow, volatile-file filtering, extension dependency filtering, tar retry, or hardlink cache behavior into the exclude path. Excludes should be a filter layer on the current writer, not a parallel backup writer.
  - confidence: 0.86
- **[P2] Avoid rescanning large archives during create:** `src/infra/backup-create.ts:528-535`
  - body: Every exclude-enabled create now calls `listArchiveEntries()` and stores the full entry list before publishing. The feature targets multi-GB backup stalls, so a mandatory second full pass with O(entry count) memory can reintroduce the same availability problem unless it is opt-in, streaming, or scoped to verification.
  - confidence: 0.81
