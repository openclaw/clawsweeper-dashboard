## Summary

Keep open. Current main still lacks native user-supplied backup exclude patterns, but this PR is not merge-ready because PR-head source still has ordered-pattern, external-asset, and create-time archive-scan blockers, and contributor real behavior proof is missing.

## What This Changes

The branch adds `backup create` exclude flags, smart defaults, `.backupignore` and exclude-file handling, protected-path guards, manifest `excludedStats`, the `ignore` dependency, and backup tests.

## Best Possible Solution

Land a rebased or narrowed native exclusion implementation that preserves current backup reliability filters, applies gitignore semantics correctly across intended assets, avoids heavy create-time scans, documents protected-path behavior, and includes redacted real-run proof.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the external-asset bypass, duplicate-rule collapse, and unconditional archive readback path. I did not run a live backup because this is a read-only review.

## Solution Assessment

No. Native excludes are a reasonable direction, but this branch should not merge until filter semantics, external asset handling, create-path performance, current-main rebase integration, and real-run proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply excludes to every backed-up asset:** `src/infra/backup-exclude.ts:358-363`
  - body: `buildExcludeFilter` still returns `true` for any path outside the single `stateDir` base, so external workspaces or custom config/credential assets are archived in full when users pass `--exclude`, `.backupignore`, or `--smart-exclude`. A warning makes the gap visible, but it does not make the exclude contract true for those backed-up assets.
  - confidence: 0.9
- **[P2] Keep duplicate ignore rules in order:** `src/infra/backup-exclude.ts:225-233`
  - body: `resolveExcludePatterns` removes later duplicate rules before handing them to `ignore`, but gitignore rules are order-sensitive with negations. A list such as `*.log`, `!important.log`, `*.log` should end with `important.log` excluded; this branch drops the final rule and changes what lands in the backup.
  - confidence: 0.88
- **[P2] Avoid full archive rescans during create:** `src/infra/backup-create.ts:528-535`
  - body: The post-create check calls `listArchiveEntries` and materializes every tar entry before publishing the archive, even when `--verify` was not requested. On the multi-GB backups this feature targets, that adds a second full archive pass and O(entry count) memory pressure to normal create.
  - confidence: 0.84
