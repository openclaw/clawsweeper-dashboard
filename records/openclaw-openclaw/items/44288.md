## Summary

Keep open: the PR is useful and current main still lacks native backup exclude flags, but the latest branch remains incorrect for current backup hardening, external asset filtering, ordered gitignore semantics, and large-archive create-time behavior; it also still lacks contributor real behavior proof.

## What This Changes

The branch adds `openclaw backup create` exclude flags, `.backupignore` loading, protected-path guards, exclusion stats in manifests, verify-reader changes, tests, and the root `ignore` dependency.

## Best Possible Solution

Refresh or replace the branch with a narrower native exclude implementation that preserves current retry/volatile filters, applies rules per archived root, keeps ordered gitignore semantics, avoids normal create-time full rescans, updates public docs, and includes redacted real-run proof.

## Reproduction Assessment

Yes, at source level for the PR findings: PR-head code shows direct tar writes, an intentional external-root bypass warning, duplicate-rule collapse, and unconditional archive entry materialization. I did not run a live backup because this review is read-only.

## Solution Assessment

No. Native backup excludes are the right direction, but this branch is not the safest implementation until it rebases onto current backup hardening and resolves per-root filtering, ordered rules, and large-archive create behavior.

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve backup retry and volatile-file filtering:** `src/infra/backup-create.ts:469-475`
  - body: Current main wraps backup tar writes in retry handling and filters volatile live-state files before publishing. This exclude path calls `tar.c` directly, so backups using excludes can regress to the live-write/tmp-file failures and volatile payloads the current path now avoids.
  - confidence: 0.91
- **[P2] Apply exclude rules to every archived asset root:** `src/infra/backup-create.ts:383-386`
  - body: This warning still archives external workspaces, config paths, or credential roots in full. Users can pass rules such as `.env*`, `*.pem`, or `logs/` and still get those files from external roots, so the new privacy and size controls are incomplete until filtering uses per-asset bases.
  - confidence: 0.9
- **[P2] Preserve ordered gitignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: This de-duplication keeps the first duplicate rule and drops later ones, but gitignore negation is order-sensitive. A list like `*.log`, `!important.log`, `*.log` should end with `important.log` ignored; this code flips the result and can archive the wrong files.
  - confidence: 0.9
- **[P2] Avoid full archive rescans in the create path:** `src/infra/backup-create.ts:528-529`
  - body: `listArchiveEntries()` materializes every tar entry before publishing even when the user did not request verification. Large backups are the target of this feature, and this adds another full archive pass plus memory proportional to entry count.
  - confidence: 0.84
