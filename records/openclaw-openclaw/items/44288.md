## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Gate implicit .backupignore loading:** `src/infra/backup-exclude.ts:190-194`
  - body: `resolveExcludePatterns` reads `${stateDir}/.backupignore` for every backup create, even when the user did not pass `--smart-exclude`, `--exclude`, or `--exclude-file`. That breaks the PR's no-change-without-flags guarantee and can silently omit files for existing users who already have that filename; make auto-loading explicit or maintainer-approved default behavior.
  - confidence: 0.9
- **[P1] Apply excludes to external backup assets:** `src/infra/backup-exclude.ts:362-364`
  - body: Paths outside `baseDir` are always included, while `createBackupArchive` builds the filter with only `stateDir` as the base. External workspaces and custom config/credential paths are valid backup assets, so explicit exclude patterns become a no-op for those trees and can leave large or sensitive files in the archive despite user intent.
  - confidence: 0.88
- **[P2] Avoid full archive scans during create:** `src/infra/backup-create.ts:528`
  - body: The integrity check calls `listArchiveEntries` and stores every archive entry before publish. For multi-GB state/workspace backups this adds a second full archive pass plus O(entry count) memory on the hot create path, which can worsen the same stall/OOM class the feature is meant to address.
  - confidence: 0.86
