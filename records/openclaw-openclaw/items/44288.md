## Summary

Keep open. Current main still lacks the requested backup exclude CLI surface, and the PR head still has merge-blocking correctness and security issues plus missing external-contributor real behavior proof.

## What This Changes

The PR adds backup exclude-pattern CLI flags, smart defaults, protected-path guards, `.backupignore` loading, exclude stats in manifests, verification changes, tests, and the `ignore` dependency.

## Best Possible Solution

Land a corrected or intentionally narrowed backup-exclude implementation that applies exclusions consistently across archived roots, preserves gitignore rule order, avoids unconditional full archive rescans, and documents the public backup semantics.

## Reproduction Assessment

Yes. Source inspection of PR head reproduces the remaining defects: external assets bypass the filter, duplicate ordered patterns are collapsed before `ignore` evaluates them, and every completed archive is fully listed after creation.

## Solution Assessment

No. The feature direction is useful, but warning that excludes do not apply to external assets, changing gitignore rule order, and adding an unconditional full rescan are not the narrowest maintainable backup contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Apply excludes before archiving external assets:** `src/infra/backup-create.ts:386`
  - body: This warning still proceeds to archive each external workspace, custom config, or external credentials asset in full. Because the filter is rooted only at the state directory and returns `true` for outside paths, explicit excludes such as `*.env` or smart defaults do not apply to those assets, so large or sensitive files can remain in the archive despite the user's rule.
  - confidence: 0.91
- **[P2] Preserve ordered duplicate exclude rules:** `src/infra/backup-exclude.ts:225-233`
  - body: `resolveExcludePatterns` drops later duplicate patterns before handing rules to `ignore`, but gitignore rules are order-sensitive after negations. A list such as `*.log`, `!important.log`, `*.log` should end with `important.log` excluded; dropping the final rule flips that result.
  - confidence: 0.9
- **[P2] Avoid rescanning every created archive:** `src/infra/backup-create.ts:528`
  - body: The post-create cross-check now lists the full tarball and stores every entry after every backup create, including the no-exclude path that was supposed to remain unchanged. On large state directories this adds a second full archive pass and O(entry count) memory, which can worsen the same large-installation stall/performance problem this PR is trying to solve.
  - confidence: 0.84
- **[P3] Document the backup exclude CLI surface:** `src/cli/program/register.backup.ts:30-41`
  - body: This adds user-facing flags and manifest/JSON behavior, but the public backup reference still lists only the old create forms and size guidance. The docs should cover the new exclude options, protected-path behavior, manifest stats, and any accepted external-asset limitation before this CLI surface ships.
  - confidence: 0.82
