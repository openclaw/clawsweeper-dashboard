Overall confidence: 0.88

Full review comments:

- **[P1] Gate automatic `.backupignore` loading:** `src/infra/backup-exclude.ts:187-194`
  - body: `resolveExcludePatterns` reads `stateDir/.backupignore` on every backup unless `--include-all` is set, even when the user passes no exclude flag. That contradicts the compatibility claim that no-exclude backups are unchanged and can silently omit files after upgrade if that local file exists, so this needs an explicit flag or maintainer-approved default/docs path.
  - confidence: 0.9
- **[P1] Rebase excludes onto the current backup writer:** `src/infra/backup-create.ts:469-476`
  - body: The exclude path builds archives directly from `result.assets`, but current main's writer snapshots SQLite DBs, filters volatile files, skips extension `node_modules`, uses an outside-source temp root, and wraps tar in retry handling. Please integrate excludes into that current pipeline rather than landing this older writer shape, or backups can regress shipped behavior and copy live/transient state again.
  - confidence: 0.87
- **[P2] Apply excludes to external backup assets:** `src/infra/backup-exclude.ts:358-364`
  - body: The filter returns `true` for paths outside `baseDir`, and `createBackupArchive` always uses `stateDir` as that base. External workspaces and custom config/credentials paths are planned backup assets, so `--exclude` and `.backupignore` silently do not apply to them beyond a warning, leaving large or sensitive external trees in the archive contrary to user intent.
  - confidence: 0.86
- **[P2] Preserve ordered duplicate ignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: This deduplication removes later repeated patterns before passing the rule list to `ignore`, but gitignore rules are order-sensitive when negations appear between duplicates. Patterns such as `*.log`, `!important.log`, `*.log` should end with the file ignored; dropping the final duplicate flips the result and can change backup contents.
  - confidence: 0.82
- **[P2] Avoid full archive scans during create:** `src/infra/backup-create.ts:525-541`
  - body: The post-create cross-check calls `listArchiveEntries` and stores every tar entry in memory for every backup, not only for explicit `--verify`. Large backup size is the main problem being addressed here, so adding an unconditional second full archive pass and O(entry count) memory use can worsen stalls or memory pressure.
  - confidence: 0.8
