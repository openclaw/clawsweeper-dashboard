## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Protect real protected-path contents, not only directory probes:** `src/infra/backup-exclude.ts:261-262`
  - body: The guard only checks each pattern against `credentials`, `credentials/x`, and the sibling protected names. A broad filename rule such as `*.json`, `*.key`, or `*.pem` does not match those probes, so it remains active; the tar filter later sees real paths like `credentials/oauth.json` and excludes protected contents without `--allow-exclude-protected`. Enforce the protected-path rule in the filter itself, or otherwise check actual descendants under those directories before allowing the pattern.
  - confidence: 0.91
- **[P1] Apply exclude rules to every backed-up asset root:** `src/infra/backup-exclude.ts:362-363`
  - body: Returning `true` for every path outside the single `stateDir` base means external workspaces and custom config/credential assets are archived in full even when they contain files matching `--exclude` or `.backupignore`. A warning still leaves the changed backup behavior wrong for users trying to remove large or sensitive external trees; build per-asset filter bases or make the command fail/disable excludes for unsupported roots instead of silently bypassing them.
  - confidence: 0.9
- **[P1] Preserve ordered gitignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: This deduplication drops later duplicate rules before passing them to `ignore`, but gitignore semantics are order-sensitive. A rule set like `*.log`, `!important.log`, `*.log` should end with `important.log` excluded; keeping only the first `*.log` flips the result and can silently keep or drop different files than the user requested. Pass the ordered pattern list through unchanged and track sources without changing rule precedence.
  - confidence: 0.89
- **[P2] Avoid rescanning every archive during create:** `src/infra/backup-create.ts:528-529`
  - body: This always reads the completed `.tar.gz` and stores every entry path in memory after archive creation, even when the user did not request `--verify`. For the multi-GB backup path this feature is meant to improve, that adds another full archive pass and O(entry count) memory pressure; keep this check on the explicit verify path or replace it with a bounded streaming validation.
  - confidence: 0.86
