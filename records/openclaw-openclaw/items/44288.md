## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve the current backup writer protections:** `src/infra/backup-create.ts:473-475`
  - body: This exclude branch archives raw `result.assets` with only `excludeFilter`. Current main's writer snapshots SQLite, skips original sidecars and volatile files, filters plugin dependencies, disables hardlink emission, and retries tar EOF races; with any exclude pattern active those protections are bypassed, so backups can include unsafe live state or fail in known races. Rebase this path so excludes compose with the existing tar filter/writer.
  - confidence: 0.93
- **[P2] Restore the manifest size cap while verifying:** `src/commands/backup-verify.ts:203-210`
  - body: The PR's manifest extraction buffers every chunk into memory and removed the current 1 MiB cap. A malformed or hostile backup can therefore force unbounded memory growth during `backup verify`; keep the existing size limit while adding tolerant `excludedStats` parsing.
  - confidence: 0.88
- **[P2] Apply exclude rules to each backed-up asset root:** `src/infra/backup-exclude.ts:362-363`
  - body: Returning `true` for paths outside `baseDir` means excludes never apply to valid assets outside the state directory, such as external workspaces or custom config paths. A warning still leaves sensitive or large files archived despite explicit exclude rules, so the filter needs per-asset bases or another semantics-preserving design.
  - confidence: 0.86
- **[P2] Keep ordered duplicate ignore rules:** `src/infra/backup-exclude.ts:225-232`
  - body: `ignore` evaluates rules in insertion order, and later duplicate positive rules can intentionally override an intervening negation. Deduplicating by first occurrence changes gitignore semantics, so keep the matcher's full ordered rule list and handle source attribution separately.
  - confidence: 0.82
- **[P2] Avoid rescanning full archives in backup create:** `src/infra/backup-create.ts:528-535`
  - body: The post-create integrity check reads the whole `.tar.gz` and stores every entry path after writing the archive. For large backups this doubles archive I/O and adds O(entry count) memory pressure on the same path this PR is meant to make reliable; use a streaming check, targeted verification, or the existing verify path instead.
  - confidence: 0.78
