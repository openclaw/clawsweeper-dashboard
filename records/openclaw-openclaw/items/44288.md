## Summary

Keep open. Current main still lacks the broad backup exclude-pattern surface, and the latest PR head is still a live implementation candidate with concrete correctness/security issues that should be fixed or explicitly accepted before merge.

## What This Changes

The PR adds `openclaw backup create` exclude-pattern support, including new CLI flags, `.backupignore`, smart defaults, protected-path guards, `excludedStats` manifest data, and backup tests.

## Best Possible Solution

Keep this PR open as the active implementation candidate, but require the branch to preserve ordered gitignore semantics and either apply excludes per backup asset or clearly narrow the product contract before landing.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: main lacks the requested flags, while the PR head can be checked with ordered patterns such as `*.log`, `!important.log`, `*.log` and with an external workspace/custom config asset outside `stateDir`.

## Solution Assessment

No. The PR direction is useful, but the current implementation is not yet the narrowest maintainable fix because it changes ordered gitignore behavior and leaves non-state assets outside the filter instead of applying excludes consistently.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve ordered duplicate exclude patterns:** `src/infra/backup-exclude.ts:225-233`
  - body: `resolveExcludePatterns` drops later duplicate rules before handing the list to `ignore`, but gitignore semantics are order-sensitive. A pattern list like `*.log`, `!important.log`, `*.log` should end with `important.log` excluded; deduping the final `*.log` flips the result and can include or exclude files differently than the user requested.
  - confidence: 0.9
- **[P2] Apply excludes to non-state backup assets:** `src/infra/backup-exclude.ts:358-364`
  - body: The filter returns `true` for paths outside `baseDir`, while `createBackupArchive` builds it with only `stateDir`. External workspaces or custom config paths in the backup plan therefore bypass `--exclude` and `.backupignore`; a warning does not prevent those files from being archived despite explicit exclude rules.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds a backup exclusion security surface, and current PR head can still archive files users explicitly tried to exclude.

Concerns:

- **[medium] Exclude rules bypass external assets:** `src/infra/backup-exclude.ts:362`
  - body: When a workspace or custom config path is outside `stateDir`, exclude rules are not applied and the asset is archived in full. That can leak files such as `.env`, keys, or logs despite a user-supplied exclude pattern.
  - confidence: 0.86
- **[low] Pattern dedupe changes sensitive-file rule order:** `src/infra/backup-exclude.ts:225`
  - body: Dropping later duplicate patterns can change final gitignore results after negations, which can defeat a user's intended sensitive-file exclusion order.
  - confidence: 0.78
