## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Apply rules to every backup asset root:** `src/infra/backup-create.ts:364-368`
  - body: This builds the exclude filter only with `canonicalStateDir`, while `resolveBackupPlanFromDisk()` can include external workspaces, config, or credentials as separate top-level assets. Because the filter returns true for outside-base paths, `--exclude '*.pem'` or `--smart-exclude` does not actually apply to those assets, so large or sensitive external files are still archived despite the user’s rule.
  - confidence: 0.91
- **[P2] Preserve repeated rule order:** `src/infra/backup-exclude.ts:225-233`
  - body: Deduplicating patterns before handing them to `ignore` changes gitignore semantics: a later duplicate can intentionally override an intervening negation, such as `*.log`, `!important.log`, `*.log`. Dropping the later rule can include or omit files differently from the user's declared exclude file.
  - confidence: 0.87
- **[P1] Avoid the full completed-archive rescan:** `src/infra/backup-create.ts:528-529`
  - body: When excludes are active, the command already writes an uncompressed tar and gzips it; this extra `listArchiveEntries()` call then reopens the completed `.tar.gz` and stores every entry path in memory before publish. Multi-GB installs with many entries can hit extra disk, CPU, latency, and O(entry count) memory on the path this feature is meant to unblock.
  - confidence: 0.84
