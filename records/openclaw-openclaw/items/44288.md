## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Rebase the writer onto current backup hardening:** `src/infra/backup-create.ts:469-475`
  - body: The branch's create path builds archives directly from `result.assets` and is now live-conflicted with current main, whose backup writer snapshots SQLite, skips volatile live-state files, retries tar EOF races, chooses a temp root outside every asset, and remaps snapshot paths. Resolve the conflict by preserving those current-main behaviors before layering excludes, or this feature risks regressing the backup reliability fixes already on main.
  - confidence: 0.88
- **[P1] Apply excludes to every backed-up asset:** `src/infra/backup-exclude.ts:362-363`
  - body: This early return makes every path outside the single stateDir base immune to `--exclude`, `.backupignore`, and `--smart-exclude`. Since the backup plan can include external workspaces, config, or credentials as separate assets, warning the user still leaves explicitly excluded large or sensitive files in the archive; the filter needs per-asset bases or an equivalent complete matching model.
  - confidence: 0.91
- **[P2] Avoid rescanning the whole archive on every create:** `src/infra/backup-create.ts:528`
  - body: This reads the completed `.tar.gz` and stores every entry path for every backup, even when the user did not ask for `--verify`. On the multi-GB installs this PR is meant to help, that adds another full archive pass and O(entry count) memory pressure; keep this as explicit verify behavior or replace it with a bounded/streaming check.
  - confidence: 0.82
