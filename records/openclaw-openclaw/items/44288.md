Full review comments:

- **[P1] Apply excludes to every backup asset root:** `src/infra/backup-exclude.ts:362-363`
  - body: `createBackupArchive` builds one exclude filter rooted at `canonicalStateDir`, but backup plans can include external workspaces and custom config/credential paths. This return makes those outside-base assets immune to `--exclude`, `--exclude-file`, and `.backupignore`, so a user can still archive bulky or sensitive files despite explicitly excluding them; a warning does not implement the advertised exclude contract.
  - confidence: 0.9
- **[P2] Do not rescan every archive during create:** `src/infra/backup-create.ts:528-529`
  - body: This unconditional `listArchiveEntries` call rereads the just-written gzip and stores every entry path even when the user did not pass `--verify`. Large backups are the problem this PR is trying to solve, so the extra full pass can add significant runtime and memory pressure; keep this behind verification or make the check streaming/bounded.
  - confidence: 0.85
- **[P2] Preserve duplicate rule order in exclude patterns:** `src/infra/backup-exclude.ts:225-233`
  - body: Gitignore rule order is meaningful, including later duplicate patterns after a negation. Deduplicating by first occurrence changes cases like `*.log`, `!important.log`, `*.log`, so the archive can include or exclude files differently from the user's declared rule order.
  - confidence: 0.82
- **[P3] Document the new backup CLI contract:** `src/cli/program/register.backup.ts:30-50`
  - body: The PR registers new public backup flags and manifest/JSON metadata, but the public backup docs remain unchanged. Users and automation need the supported exclude semantics, protected-path behavior, and `excludedStats` output documented with the CLI change.
  - confidence: 0.78
