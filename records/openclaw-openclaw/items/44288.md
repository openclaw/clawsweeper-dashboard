## Summary

## What This Changes

The PR adds backup create exclude flags, smart exclude defaults, protected-path checks, `.backupignore` and `--exclude-file` handling, `excludedStats` manifest data, the `ignore` dependency, and backup create/verify tests.

## Best Possible Solution

Rebase the exclude feature onto the current single backup pipeline, compose excludes with existing retry/temp-root/volatile/plugin-dependency filters, decide external-root semantics explicitly, and require redacted real backup create/verify proof.

## Reproduction Assessment

Yes for the PR findings at source level: the PR head shows direct tar creation, state-dir-only filtering, ordered-rule deduplication, and unconditional archive rescanning. I did not run a live backup because this review is read-only and the proof gate requires contributor-provided real environment evidence.

## Solution Assessment

No: the feature direction is reasonable, but this branch is not the best merge shape because it builds parallel archive paths instead of composing with the current backup writer. The safer solution is one pipeline that layers excludes onto existing retry, volatile-skip, temp-root, and plugin-dependency filtering with explicit external-root behavior.

- **[P1] Preserve the current backup archive safeguards:** `src/infra/backup-create.ts:469-475`
  - body: This path calls `tar.c` directly for excluded backups and does not compose current main’s `writeTarArchiveWithRetry`, volatile-file filter, installed-plugin `node_modules` filter, or safe temp-root selection.   - confidence: 0.91
- **[P2] Apply excludes to every backed-up source root:** `src/infra/backup-exclude.ts:358-363`
  - body: Returning `true` for every path outside `baseDir` makes excludes a no-op for external workspaces and custom config/assets, which `resolveBackupPlanFromDisk` can include. A warning is not enough for a flag that promises to exclude matching files, especially when users rely on it to keep sensitive or bulky workspace files out of archives.
  - confidence: 0.88
- **[P2] Preserve ordered gitignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: Dropping later duplicate patterns changes gitignore semantics because negation is order-sensitive. For example, `*.log`, `!important.log`, `*.log` should end with `important.log` excluded, but the dedupe step removes the final rule and flips the result.
  - confidence: 0.84
- **[P2] Avoid rescanning every archive during create:** `src/infra/backup-create.ts:525-528`
  - body: This cross-check reads the whole `.tar.gz` and stores every entry path in memory on every successful create, even when the user did not pass `--verify`. For large backups, that adds the same expensive second pass and memory pressure the backup work is trying to avoid.
  - confidence: 0.82
