## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Apply excludes to every backed-up asset root:** `src/infra/backup-create.ts:364-368`
  - body: `buildExcludeFilter` is rooted at `canonicalStateDir`, and the filter explicitly includes paths outside that base. Since the backup plan can include external workspaces and custom config/credential paths, user-supplied excludes do not filter those assets; a warning still leaves large or sensitive files in the archive despite the exclude request.
  - confidence: 0.9
- **[P2] Preserve ordered gitignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: This deduplication drops later duplicate patterns before passing rules to `ignore`, but gitignore matching is order-sensitive. A sequence like `*.log`, `!important.log`, `*.log` should end with the file ignored; after deduplication it is incorrectly unignored, so backups can include or exclude files contrary to the user's declared rule order.
  - confidence: 0.86
- **[P2] Do not rescan every backup by default:** `src/infra/backup-create.ts:528-541`
  - body: The new integrity check reads the completed `.tar.gz` and stores every entry path for every real backup, even when `--verify` was not requested. That turns the default create path into a second full archive pass with O(entry count) memory, which can worsen the multi-GB backup stalls this PR is trying to address.
  - confidence: 0.82
