This PR adds backup exclude-pattern support with new `backup create` CLI flags, an `ignore` dependency, protected-path guards, `.backupignore` loading, `excludedStats` manifest data, and backup tests.

## Best Possible Solution

## Reproduction Assessment

## Solution Assessment

Is this the best way to solve the issue? No. The native backup exclude direction is useful, but this implementation needs per-asset filtering, order-preserving gitignore semantics, current-main rebase safety, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Apply excludes to external backup assets:** `src/infra/backup-exclude.ts:362-363`
  - body: `buildExcludeFilter` returns `true` for every path outside the state-dir base, so external workspaces and custom config paths are archived in full even when they match `--exclude` or `.backupignore`. A warning does not make the feature work for the linked use case of excluding workspace `node_modules`, logs, or sensitive files; the filter needs per-asset bases or an explicitly narrower contract before merge.
  - confidence: 0.92
- **[P2] Preserve duplicate gitignore rules:** `src/infra/backup-exclude.ts:229-231`
  - body: This de-duplication removes later repeated rules before passing them to `ignore`, but gitignore rules are order-sensitive. A pattern list like `*.log`, `!important.log`, `*.log` should end with `important.log` ignored; dropping the final duplicate changes the user's requested backup contents.
  - confidence: 0.88
- **[P2] Avoid rescanning every archive entry on create:** `src/infra/backup-create.ts:528-535`
  - body: `backup create` now always calls `listArchiveEntries` and stores the full tar entry list after writing, even when the user did not ask for `--verify`. On the large backups this PR is meant to unblock, that adds a second full archive pass plus O(number of entries) memory pressure and can turn a successful backup into a slow or failing one.
  - confidence: 0.86
- **[P2] Preserve raw tar paths for manifest extraction:** `src/commands/backup-shared.ts:55`
  - body: `listArchiveEntries` normalizes backslashes before returning paths, but `backupVerifyCommand` later treats that value as the raw entry path and passes it to `extractManifest`, which compares against raw `entry.path`. On the Windows/backslash path case this helper was added for, verify can find the normalized manifest candidate and then fail to extract it.
  - confidence: 0.78
