Full review comments:

- **[P1] Layer excludes into the current backup writer:** `src/infra/backup-create.ts:469-475`
  - body: This archive path is based on an older writer shape; current main now has SQLite snapshot/remap handling, volatile-file filtering, tar EOF retry, hardlink-cache handling, and safer temp-root placement. The exclude feature needs to be layered into that current path before merge so backup consistency and large-install reliability do not regress during conflict resolution.
  - confidence: 0.86
- **[P1] Apply excludes per backup asset root:** `src/infra/backup-exclude.ts:358-364`
  - body: The filter returns `true` for anything outside the single `baseDir`, while the create path builds it from the state directory. That makes `--exclude`, `--exclude-file`, and `.backupignore` a no-op for valid external workspace, config, or credential assets, so users can still archive large or sensitive files they asked to exclude.
  - confidence: 0.9
- **[P2] Avoid the full archive scan in create:** `src/infra/backup-create.ts:525-542`
  - body: `listArchiveEntries()` reads every tar entry into memory immediately after writing the archive just to check a few asset roots. On multi-GB backups this adds another full pass plus O(entry count) memory use, undercutting the large-backup use case this feature is meant to help.
  - confidence: 0.84
- **[P2] Preserve duplicate gitignore rules in order:** `src/infra/backup-exclude.ts:225-233`
  - body: Gitignore precedence is order-sensitive, especially around negations, but this resolver drops later duplicate patterns before passing them to `ignore`. A sequence like `*.log`, `!important.log`, `*.log` changes meaning if the final duplicate is removed.
  - confidence: 0.8
