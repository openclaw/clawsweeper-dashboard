## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Run excludes through the current backup pipeline:** `src/infra/backup-create.ts:469-475`
  - body: The exclude branch creates the archive with a direct `tar.c(... result.assets ...)` call and only `excludeFilter`. Current main wraps backup creation with SQLite snapshots, volatile-file skipping, extension dependency filtering, tar retry, and safer temp-root handling; this path needs to compose with that pipeline or excluded backups can regress recovery correctness.
  - confidence: 0.91
- **[P1] Apply excludes to every backup asset root:** `src/infra/backup-exclude.ts:358-363`
  - body: Returning `true` for paths outside `baseDir` makes `--exclude` and `.backupignore` a no-op for valid external workspaces and custom config/credential assets. A warning is not enough for an exclude feature because users can still archive sensitive or oversized files they explicitly asked to omit.
  - confidence: 0.93
- **[P2] Preserve ordered gitignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: This deduplication drops later duplicate patterns before passing rules to `ignore`, but gitignore-style rules are ordered and a repeated pattern after a negation can intentionally re-ignore a path. Preserve the full ordered rule list and track pattern stats without changing matcher semantics.
  - confidence: 0.87
- **[P2] Avoid rescanning large archives during create:** `src/infra/backup-create.ts:528-535`
  - body: The integrity check reads the completed archive into an array of every entry on every create path. For the large backups this feature targets, that adds a full second archive pass and O(entry count) memory, so make it streaming, opt-in via verify, or otherwise bounded.
  - confidence: 0.82
