## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Apply excludes to every backup asset:** `src/infra/backup-exclude.ts:362-363`
  - body: This early return makes every planned asset outside `stateDir` immune to `--exclude`, `.backupignore`, and `--smart-exclude`. Because backup planning includes external workspaces, custom config paths, and external credentials, users can explicitly exclude large or sensitive files and still get them archived in full; warning after resolution is not enough for the advertised gitignore-like backup contract.
  - confidence: 0.91
- **[P1] Keep no-flag backups unchanged:** `src/infra/backup-exclude.ts:187-194`
  - body: `resolveExcludePatterns` always reads `stateDir/.backupignore` unless `--include-all` is set, and `createBackupArchive` calls it for every backup. That means an existing `.backupignore` file changes `openclaw backup create` with no exclude flags, silently omitting files despite the PR claiming no behavior change when no exclude flags are passed.
  - confidence: 0.86
- **[P2] Preserve ordered gitignore duplicate rules:** `src/infra/backup-exclude.ts:225-233`
  - body: Dropping later duplicate patterns changes gitignore semantics because duplicates after negations can intentionally restore an ignore decision, such as `*.log`, `!important.log`, `*.log`. Pass the ordered rules to `ignore` instead of deduplicating by first occurrence, or use rule marks for stats without changing matching behavior.
  - confidence: 0.9
- **[P2] Avoid a full in-memory archive scan on create:** `src/infra/backup-create.ts:528-529`
  - body: The exclude path now reads the entire newly written `.tar.gz` into `archiveEntries` for every create, even when `--verify` was not requested. On the multi-GB backups this PR targets, that adds a second full archive pass plus O(entry count) memory and can reintroduce stalls or OOM pressure before the archive is published.
  - confidence: 0.84
