## Summary

Keep open: current main still lacks general backup exclude-pattern flags, so this PR is not obsolete, but the PR head still has correctness, security, performance, and docs blockers before merge.

## What This Changes

This PR adds backup create exclude-pattern flags, `.backupignore` loading, smart defaults, protected-path guards, pattern-level manifest stats, backup verify changes, tests, and the `ignore` dependency.

## Best Possible Solution

Land a smaller, validated version that preserves gitignore rule order, applies excludes consistently to the promised backup scope, avoids unconditional large-archive rescans, and updates docs, changelog, and regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: inspect `resolveExcludePatterns` on `*.log`, `!important.log`, `*.log`, and inspect any included external asset with excludes active against the state-dir-only filter base.

## Solution Assessment

No. The direction is useful, but warning that excludes bypass external assets plus order-changing dedupe is not the narrowest maintainable backup exclusion contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve ordered duplicate exclude rules:** `src/infra/backup-exclude.ts:225-233`
  - body: `resolveExcludePatterns` drops later duplicate patterns before handing rules to `ignore`, but gitignore rules are order-sensitive after negations. A list such as `*.log`, `!important.log`, `*.log` should end with `important.log` excluded; dropping the final rule flips the declared result.
  - confidence: 0.91
- **[P2] Apply exclude rules to external backup assets:** `src/infra/backup-exclude.ts:358-363`
  - body: The filter returns `true` for paths outside `baseDir`, and `createBackupArchive()` builds it with only the state directory. External workspaces or custom config/credentials paths are archived in full despite `--exclude` or `.backupignore`; warning users does not honor the filtering contract and can include sensitive files they asked to exclude.
  - confidence: 0.9
- **[P2] Avoid rescanning every archive during create:** `src/infra/backup-create.ts:525-528`
  - body: The post-create cross-check always calls `listArchiveEntries(tempArchivePath)`, including the no-exclude path that the PR says remains unchanged. That adds a second full tar pass and stores every entry path for large backups, which can materially worsen the same large-installation stall/performance problem this PR is trying to address.
  - confidence: 0.82
- **[P3] Document the new backup exclude surface:** `src/cli/program/register.backup.ts:30-52`
  - body: This adds user-facing backup flags and manifest/JSON output changes, but the PR head leaves `docs/cli/backup.md` and `CHANGELOG.md` without the new exclude behavior. OpenClaw policy expects behavior/API docs and a changelog entry for user-facing feature and fix changes.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch adds security-sensitive backup exclusion controls, but current PR head can still archive sensitive external assets users explicitly tried to exclude.

Concerns:

- **[medium] Exclude rules bypass external assets:** `src/infra/backup-exclude.ts:362`
  - body: When a workspace, config, or credentials path lives outside `stateDir`, the filter includes it without applying user exclude patterns. That can archive files such as `.env`, keys, logs, or model artifacts despite a user-supplied exclusion rule.
  - confidence: 0.9
- **[low] Pattern dedupe can change sensitive-file precedence:** `src/infra/backup-exclude.ts:225`
  - body: Dropping later duplicate patterns changes order-sensitive gitignore behavior after negations, which can defeat a user's intended final sensitive-file exclusion rule.
  - confidence: 0.78
