## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve current backup hardening:** `src/infra/backup-create.ts:469-475`
  - body: The exclude path calls raw `tar.c` and then gzips the result, but current main wraps archive creation with EOF retries, volatile-file skips, temp-root handling, and plugin dependency pruning. Rebase the exclude layer into that hardened path so large or live installs do not lose reliability fixes that already exist on main.
  - confidence: 0.88
- **[P2] Apply excludes to every archived root:** `src/infra/backup-exclude.ts:362-364`
  - body: Returning `true` for paths outside the state-dir base means external workspaces or custom config assets are archived in full even when users pass matching exclude patterns. A warning is not enough for a feature promising explicit filtering of backup contents; use per-asset bases or make the narrower scope an intentional maintainer-approved limitation with docs and proof.
  - confidence: 0.9
- **[P2] Keep duplicate rules in order:** `src/infra/backup-exclude.ts:225-233`
  - body: Deduplicating patterns before handing them to `ignore` changes gitignore semantics when a duplicate rule intentionally appears after a negation, such as `*.log`, `!important.log`, `*.log`. Preserve the ordered rule list and track pattern stats without collapsing later duplicates.
  - confidence: 0.86
- **[P2] Avoid full archive scans in create:** `src/infra/backup-create.ts:528-529`
  - body: `listArchiveEntries()` reads the completed tarball and stores every entry path in memory, then create uses it unconditionally. For multi-GB backups with many entries, that second full pass can add material runtime and memory pressure; validate expected roots with a streaming or bounded check instead.
  - confidence: 0.78
