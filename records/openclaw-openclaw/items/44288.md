## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve the current backup writer filters:** `src/infra/backup-create.ts:469-473`
  - body: When excludes are active, this branch calls `tar.c` with only `excludeFilter` over `result.assets`. Current main snapshots SQLite databases, skips volatile state/log/queue/socket/temp paths, filters extension `node_modules`, and retries EOF-class tar races before publishing; bypassing that path can produce unsafe or failing backups for live installs. Layer the exclude filter onto the current writer instead of replacing it.
  - confidence: 0.93
- **[P1] Apply excludes to every backup asset root:** `src/infra/backup-exclude.ts:362-363`
  - body: This early return means every asset outside `stateDir`, such as external workspaces, custom config files, or external credential directories, ignores user exclude patterns entirely. A warning does not make `--exclude "*.pem"` or `.backupignore` honor the requested backup policy; use per-asset filter bases or make the flag explicitly state-only before merge.
  - confidence: 0.92
- **[P2] Do not rescan every archive during create:** `src/infra/backup-create.ts:528`
  - body: `listArchiveEntries` reads the complete `.tar.gz` and stores every entry path for every create, even when the user did not request `--verify`. That adds a second full archive pass and O(entry count) memory to the large-backup path this feature is meant to improve; gate this behind verify or make the integrity check streaming/bounded.
  - confidence: 0.88
- **[P2] Keep duplicate rules in their declared order:** `src/infra/backup-exclude.ts:225-233`
  - body: Gitignore rules are order-sensitive, especially around negation. Deduplicating patterns before passing them to `ignore` changes cases like `*.log`, `!important.log`, `*.log`, where the final duplicate is intentionally meaningful; preserve the original rule sequence and track source metadata without collapsing rules.
  - confidence: 0.86
- **[P2] Document the new backup contract before exposing flags:** `src/cli/program/register.backup.ts:30-52`
  - body: These public flags are advertised from `openclaw backup --help`, but the linked backup docs are not updated for `--exclude`, `.backupignore`, protected paths, `excludedStats`, or the external-asset limitation. Users following the docs page would miss the new behavior and the JSON compatibility change.
  - confidence: 0.84
