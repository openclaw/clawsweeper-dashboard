## Summary

Keep open: the requested backup-exclude capability is still useful and not implemented on current main, but this branch is not merge-ready because it conflicts with current main, lacks real behavior proof, and still has merge-blocking backup data-loss/security semantics.

## What This Changes

The PR adds `backup create` exclude-pattern CLI flags, smart defaults, protected-path guards, `.backupignore` loading, manifest exclusion stats, and backup create/verify coverage.

## Best Possible Solution

Rebase and rework the native exclude implementation so it composes with current backup retry/volatile/plugin-dependency filters, applies or explicitly rejects excludes per archived root, preserves gitignore rule order, avoids normal create-time full rescans, updates docs, and includes redacted real-run proof.

## Solution Assessment

No. Native backup excludes are the right direction, but this branch is stale against current backup hardening and needs design fixes before it is the narrowest maintainable solution for data-loss/security-sensitive backup behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve current backup hardening on exclude writes:** `src/infra/backup-create.ts:469-475`
  - body: When excludes are active, this path writes a raw tar and gzips it directly instead of composing with current main's tar EOF retry, volatile-file skip, and installed-plugin dependency filter. Rebase and route the exclude path through the current hardened archive pipeline, or live/large backups can regress.
  - confidence: 0.93
- **[P2] Apply excludes to every archived asset root:** `src/infra/backup-create.ts:383-387`
  - body: This warning still archives external workspaces and custom config assets in full, so patterns like `.env*`, `*.pem`, or `logs/` do not protect those roots. The implementation should apply filters per asset base or fail/require an explicit opt-out before merge.
  - confidence: 0.9
- **[P2] Keep ordered ignore rules intact:** `src/infra/backup-exclude.ts:225-233`
  - body: Removing later duplicate rules changes gitignore precedence because negations are order-sensitive. A sequence like `*.log`, `!important.log`, `*.log` should end with `important.log` ignored, but this de-duplication drops the final rule and flips the result.
  - confidence: 0.89
- **[P2] Avoid full archive rescans in create:** `src/infra/backup-create.ts:525-535`
  - body: `listArchiveEntries()` materializes every tar entry before publishing the archive even when the user did not ask for verification. For the large backups this feature targets, that adds another full archive pass and memory proportional to entry count to normal create.
  - confidence: 0.86
