## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the hardened archive pipeline:** `src/infra/backup-create.ts:469-476`
  - body: The exclude path calls `tar.c` directly and writes a separate gzip pipeline, bypassing current main's retry wrapper, volatile-file skip filter, temp-root handling, and extension dependency pruning. Rebasing this as-is would reintroduce the live-backup failure modes that recent backup hardening fixed.
  - confidence: 0.92
- **[P2] Apply excludes to every archived asset root:** `src/infra/backup-create.ts:364-368`
  - body: The filter is built with `canonicalStateDir` as its only base, and assets outside that root only trigger a warning before being archived in full. A user excluding `*.pem`, logs, or dependency trees from an external workspace/custom config path would still get those files in the backup despite explicit exclude rules.
  - confidence: 0.9
- **[P2] Keep ordered ignore rules intact:** `src/infra/backup-exclude.ts:225-233`
  - body: This deduplicates pattern strings before giving them to `ignore`, but gitignore rules are order-sensitive. A later duplicate positive rule after a negation is meaningful, so this can silently change which files are included in a backup.
  - confidence: 0.86
- **[P2] Avoid a full archive rescan in the large-backup path:** `src/infra/backup-create.ts:528-535`
  - body: `listArchiveEntries` reads the completed archive and stores every entry path before publish. On multi-GB backups this adds a second full pass plus O(entry count) memory pressure, undercutting the large-installation reliability goal of the feature.
  - confidence: 0.82
