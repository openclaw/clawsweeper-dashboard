## Summary

Keep open: the native backup exclude capability is still useful and not implemented on current main, but this PR is not merge-ready because the latest branch conflicts with main, lacks real behavior proof, and still has blocking backup safety findings.

## What This Changes

The PR adds `backup create` exclude-pattern flags, smart-exclude defaults, protected-path guards, `.backupignore` handling, manifest exclusion stats, and related backup create/verify tests.

## Best Possible Solution

Refresh or replace the branch with a native exclude implementation that composes with the current backup writer pipeline, filters per archived root, preserves ordered gitignore semantics, avoids mandatory full archive rescans, updates public docs, and includes redacted real-run proof.

## Reproduction Assessment

Yes, at source level for the review findings: PR-head code shows direct archive writes, single-base external-asset bypass, duplicate-rule de-duplication, and unconditional archive entry materialization. I did not run a live backup because this was a read-only review.

## Solution Assessment

No. Native backup excludes are the right direction, but this branch needs rebase/integration with current main and resolved archive filtering semantics before it is the safest maintainable fix.

- **[P2] Preserve current backup hardening on archive writes:** `src/infra/backup-create.ts:469-475`
  - body: This exclude path calls `tar.c` directly and the branch no longer composes current main's EOF retry, volatile-file skip, or installed-plugin `node_modules` filter before publishing the archive.   - confidence: 0.9
- **[P2] Apply excludes to every archived asset root:** `src/infra/backup-create.ts:378-386`
  - body: The branch now warns that external workspaces/custom config assets bypass patterns, but then still archives them in full. Since the backup plan can include those roots, rules like `.env*`, `*.pem`, or `logs/` will not protect all content the command writes.
  - confidence: 0.92
- **[P2] Keep duplicate ignore rules in order:** `src/infra/backup-exclude.ts:225-233`
  - body: Gitignore matching is order-sensitive, and the `ignore` package evaluates rules in insertion order. Dropping later duplicate patterns changes lists such as `*.log`, `!important.log`, `*.log`, so the archive can include or exclude files contrary to the user's declared order.
  - confidence: 0.9
- **[P2] Avoid full archive rescans during create:** `src/infra/backup-create.ts:528-535`
  - body: The post-create check always materializes every tar entry into memory before publishing, even when the user did not ask for verification. On the multi-GB backups this feature targets, that adds another full archive pass and memory proportional to entry count.
  - confidence: 0.84
