## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate `.backupignore` loading behind explicit opt-in:** `src/infra/backup-exclude.ts:187-194`
  - body: `resolveExcludePatterns` reads `stateDir/.backupignore` even when the user passes no exclude flags. That breaks the PR’s no-flag compatibility claim and lets an existing or stray file silently change backup contents, so auto-loading needs an explicit flag/product decision or upgrade-safe proof.
  - confidence: 0.89
- **[P2] Preserve ordered gitignore rules instead of deduplicating:** `src/infra/backup-exclude.ts:225-233`
  - body: Gitignore rules are order-sensitive, especially with `!` negations, but this loop removes later duplicates before passing rules to `ignore`. A sequence like `*.log`, `!important.log`, `*.log` changes meaning after dedupe, so keep the original ordered rule list and track provenance separately.
  - confidence: 0.86
- **[P1] Rebase the exclude path onto current backup hardening:** `src/infra/backup-create.ts:462-476`
  - body: The exclude-enabled archive path directly calls `tar.c` over the assets and does not carry current main’s SQLite snapshot plan, volatile-file filter, tar EOF retry wrapper, or temp-root containment. Resolve the dirty branch by integrating excludes into the current pipeline instead of replacing those safeguards.
  - confidence: 0.82
- **[P2] Avoid the unconditional full archive scan after create:** `src/infra/backup-create.ts:525-542`
  - body: The post-create integrity check reads every tar entry into memory immediately after writing the archive. This adds another full pass and O(entry count) memory to the large-backup path this PR is trying to improve; stream the check or keep it behind verification/debug behavior.
  - confidence: 0.78
