## Summary

Keep this PR open because it contains unique backup exclude feature work that is not on current main, but the current branch is not merge-ready: it regresses current backup writer hardening, still leaves explicit excludes ineffective for external assets, changes ordered gitignore semantics, adds an unconditional full archive rescan, and lacks after-fix real behavior proof.

## What This Changes

The PR adds `backup create` exclude flags, `.backupignore`/exclude-file handling, protected-path guards, `excludedStats` manifest support, the `ignore` dependency, and backup exclude tests.

## Best Possible Solution

Rebase a smaller native exclude implementation onto current main that composes with existing retry, temp-root, volatile-file, and plugin-dependency filters; define per-asset exclude semantics; preserve gitignore order; avoid unconditional rescans; and include redacted real-run proof.

## Reproduction Assessment

Yes for the review findings: source inspection of current main and PR head shows the stale writer regression, external-asset bypass, ordered-rule change, and unconditional archive rescan. I did not run a live backup because this review is read-only.

## Solution Assessment

No. The native exclude capability is useful, but this branch is not the best implementation until it preserves current backup hardening and defines how excludes apply across all backed-up asset bases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Compose excludes with the current backup writer:** `src/infra/backup-create.ts:469-475`
  - body: This exclude path writes with direct `tar.c` calls instead of composing with current main's `writeTarArchiveWithRetry`, volatile-file skip filter, installed-plugin `node_modules` filter, and temp-root guard. That regresses live backup reliability and archive scope, and the no-exclude path below also loses the current backwards-compatible writer behavior.
  - confidence: 0.91
- **[P2] Apply excludes to every backed-up asset base:** `src/infra/backup-exclude.ts:362-363`
  - body: Returning `true` for every path outside `stateDir` makes `--exclude`, `.backupignore`, and smart excludes no-op for external workspaces or custom config assets that are still included in the backup. A warning is not enough when the archive can still contain files such as `.env`, `*.pem`, or large dependency trees that the user explicitly excluded.
  - confidence: 0.88
- **[P2] Preserve ordered duplicate gitignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: The resolver drops later duplicate patterns before handing them to `ignore`, but gitignore rules are order-sensitive. A sequence like `*.log`, `!important.log`, `*.log` intentionally re-excludes the file; deduping the final rule changes the user's declared filter.
  - confidence: 0.86
- **[P2] Avoid rescanning every archive during create:** `src/infra/backup-create.ts:528-535`
  - body: This post-create check always reads the whole `.tar.gz` and stores every entry path in memory even when the user did not ask for `--verify`. On the large backup trees this PR is meant to help, that adds a second full archive pass and O(entry count) memory pressure before the archive can be published.
  - confidence: 0.82
