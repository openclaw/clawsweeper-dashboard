## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Apply excludes to every included asset root:** `src/infra/backup-exclude.ts:362-363`
  - body: `createBackupArchive()` builds the exclude filter with only `stateDir`, but the backup plan can include external config, credentials, and workspace roots. This early return makes `--exclude` and `.backupignore` a no-op for those assets, so large or sensitive files can still be archived in full despite explicit exclude rules; warning about the bypass does not make the flag behavior correct.
  - confidence: 0.94
- **[P1] Preserve ordered duplicate rules:** `src/infra/backup-exclude.ts:225-233`
  - body: This deduplication removes later copies of the same pattern before `ignore` sees them. Gitignore rules are order-sensitive around negation, so a valid sequence like `*.log`, `!important.log`, `*.log` is changed by the PR and can include or exclude different files than the user requested.
  - confidence: 0.9
- **[P1] Remove the full completed-archive rescan:** `src/infra/backup-create.ts:528-535`
  - body: Every successful create now reads the finished `.tar.gz` and stores all entry paths in memory for a cross-check. On the multi-GB backups this feature is intended to unblock, that adds a second full archive pass and O(entry count) memory after already writing an uncompressed intermediate, so it can reintroduce the same stall or resource-pressure failure mode.
  - confidence: 0.88
