- **[P1] Rebase exclude archive creation onto the current backup writer:** `src/infra/backup-create.ts:469-476`
  - body: This exclude path calls `tar.c` directly over the planned assets.   - confidence: 0.91
- **[P1] Gate implicit `.backupignore` loading:** `src/infra/backup-exclude.ts:187-194`
  - body: Pattern resolution always reads `.backupignore` from the state directory, even when the user did not pass an exclude flag. That makes an existing file silently change backup contents on upgrade, so this needs an explicit opt-in or maintainer-approved compatibility plan.
  - confidence: 0.88
- **[P1] Preserve verifier manifest and hardlink hardening:** `src/commands/backup-verify.ts:202-209`
  - body: This branch accumulates the manifest stream without the current 1 MiB cap, and its shared entry lister returns only path strings, so the verifier loses current main's hardlink target checks. A crafted or corrupted archive can consume unbounded memory or pass with missing hardlink targets.
  - confidence: 0.86
- **[P2] Keep duplicate rules in declared order:** `src/infra/backup-exclude.ts:225-233`
  - body: Deduplicating patterns before passing them to `ignore` changes gitignore precedence when a later duplicate intentionally overrides an intervening negation, such as `*.log`, `!important.log`, `*.log`. Backup filters should preserve the user's rule order exactly.
  - confidence: 0.84
- **[P2] Avoid a full archive rescan during create:** `src/infra/backup-create.ts:528-535`
  - body: After writing the archive, this code reads every tar entry into a Set just to check asset roots. For the multi-GB backups this feature is meant to help, that adds a second full pass and O(entry count) memory pressure before the archive can publish.
  - confidence: 0.8
