## Summary

Keep open: current main and the latest release still lack the native backup exclude surface, and this active external PR is the linked implementation path, but the latest head still has blocking backup semantics, security-review concerns, and no after-fix real behavior proof.

## What This Changes

This PR adds backup create exclude flags, gitignore-style pattern handling, smart defaults, protected-path guards, `.backupignore` support, per-pattern manifest stats, archive integrity checks, and backup create/verify tests.

## Best Possible Solution

Land a corrected native backup exclusion implementation that filters every intended archive root consistently, preserves gitignore rule order, avoids full archive rescans in normal create, documents the final CLI contract, and includes redacted real-run proof.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the external-asset bypass, duplicate-rule collapse, and unconditional archive readback path. I did not run a live backup because this is a read-only review.

## Solution Assessment

No. Native backup excludes are the right direction, but warning-only external-asset bypass behavior, rule-order changes, and a full readback in the create path are not the narrowest maintainable solution for large or sensitive backups.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply excludes to every archived asset root:** `src/infra/backup-create.ts:386`
  - body: This warning documents that external workspaces or custom config/credential assets bypass the filter, but the PR still archives those assets in full. User rules like `*.pem`, `.env*`, or `logs/` therefore do not protect all content the backup writes.
  - confidence: 0.9
- **[P2] Preserve duplicate ignore rules in order:** `src/infra/backup-exclude.ts:225-233`
  - body: Dropping later duplicate patterns changes gitignore precedence because negations are order-sensitive. A rule list such as `*.log`, `!important.log`, `*.log` should end with `important.log` ignored, but this de-duplication removes the final rule and flips the result.
  - confidence: 0.88
- **[P2] Avoid full archive rescans during create:** `src/infra/backup-create.ts:528-535`
  - body: `listArchiveEntries()` materializes every tar entry before publishing the archive even when the user did not ask for `--verify`. For the multi-GB backups this feature targets, that adds a second full archive pass and memory proportional to entry count to normal create.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch changes backup privacy controls and still has cases where user exclude rules can fail to protect archived sensitive files.

Concerns:

- **[medium] Exclude rules can miss external sensitive files:** `src/infra/backup-create.ts:386`
  - body: Paths outside the state-directory filter base are archived in full, so excludes like `.env*`, `*.pem`, or `*.key` do not protect external workspace, config, or credential assets included by the backup plan.
  - confidence: 0.88
- **[low] Pattern deduplication can defeat later privacy rules:** `src/infra/backup-exclude.ts:225`
  - body: Dropping duplicate ignore rules before evaluation changes ordered gitignore negation semantics, so a later re-exclude rule meant to protect sensitive files can be removed.
  - confidence: 0.78
