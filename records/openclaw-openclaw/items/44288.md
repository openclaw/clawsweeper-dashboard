## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve repeated exclude rules in order:** `src/infra/backup-exclude.ts:225-232`
  - body: This deduplication changes gitignore semantics. `ignore` walks every rule in order, so a sequence like `*.log`, `!important.log`, `*.log` should end with `important.log` excluded; dropping the later duplicate can keep or drop files differently from the user's declared rules.
  - confidence: 0.88
- **[P1] Apply excludes to every backup asset root:** `src/infra/backup-exclude.ts:358-363`
  - body: The filter returns `true` for entries outside the single `stateDir` base, but backup plans can include external workspaces and custom config paths. In those setups, `--exclude` and `.backupignore` archive those assets in full despite matching patterns, so large or sensitive files can remain in the backup.
  - confidence: 0.9
- **[P1] Avoid rescanning completed large archives on create:** `src/infra/backup-create.ts:528-529`
  - body: The create path lists every entry from the finished `.tar.gz` into memory before publishing. On multi-GB backups this adds a full decompress/scan and O(entry count) memory to the same stall-prone workflow the PR is trying to improve; use streaming verification or keep this behind explicit verification.
  - confidence: 0.84
