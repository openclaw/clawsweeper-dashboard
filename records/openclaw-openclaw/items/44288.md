Full review comments:

- **[P1] Rebase excludes into the current backup writer:** `src/infra/backup-create.ts:469-476`
  - body: This exclude path writes the archive with a direct `tar.c` call instead of the current main writer that retries EOF races, skips live volatile state files, prunes installed plugin `node_modules`, and keeps the temp manifest outside backed-up assets. Merging this branch without preserving those behaviors can reintroduce the same backup stalls, oversized archives, and duplicate-manifest failures current main already fixed.
  - confidence: 0.92
- **[P1] Apply excludes to every backup asset:** `src/infra/backup-exclude.ts:358-363`
  - body: The filter returns `true` for any path outside `stateDir`, and the create path only warns about those assets. That means `--exclude`, `.backupignore`, and `--smart-exclude` do not actually filter external workspaces or custom config locations, so sensitive or bulky files can still be archived despite explicit user rules.
  - confidence: 0.9
- **[P2] Preserve ordered gitignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: This deduplication drops later duplicate patterns before the rules reach `ignore`, but gitignore rules are order-sensitive around negation. A sequence like `*.log`, `!important.log`, `*.log` should end with the file excluded; first-occurrence dedupe changes the user's declared precedence.
  - confidence: 0.84
- **[P2] Avoid rescanning every archive during create:** `src/infra/backup-create.ts:528-535`
  - body: The create path now always lists every archive entry and stores them in memory before publishing. For the large backup cases this PR is meant to improve, that adds a second full archive pass and O(entry count) memory even when the user did not ask for `--verify`.
  - confidence: 0.82
