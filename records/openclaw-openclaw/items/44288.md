## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate automatic .backupignore loading:** `src/infra/backup-exclude.ts:187-194`
  - body: This runs on every `backup create`, even when the user did not pass `--smart-exclude`, `--exclude`, or `--exclude-file`. A pre-existing `stateDir/.backupignore` would silently change default backup contents, contradicting the PR's no-flag compatibility claim; make this opt-in or get an explicit maintainer-approved default with docs and upgrade coverage.
  - confidence: 0.9
- **[P1] Rebase excludes onto the current backup writer:** `src/infra/backup-create.ts:469-475`
  - body: The exclude path archives `result.assets` through its own tar/gzip flow, so resolving the dirty branch by keeping this code would skip current main's SQLite snapshots, volatile-file filter, EOF retry, temp-root selection, hardlink-cache, and source-path remaps. The exclude filter needs to compose with the current writer instead of replacing it.
  - confidence: 0.92
- **[P2] Apply excludes outside the state directory:** `src/infra/backup-exclude.ts:358-364`
  - body: Returning `true` for every path outside `baseDir` means external workspaces and custom config/credential paths are archived in full even when users pass matching exclude rules. Since current backup docs include those assets, a warning is not enough for a security-sensitive exclude feature; use per-asset bases or narrow the documented contract before merge.
  - confidence: 0.87
- **[P2] Preserve ordered duplicate ignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: Gitignore rules are order-sensitive, but this drops later duplicate patterns before handing them to `ignore`. A sequence like `*.log`, `!important.log`, `*.log` changes meaning after dedupe, so the backup may include or omit files differently from the user's declared rule order.
  - confidence: 0.84
- **[P2] Avoid full archive scans during create:** `src/infra/backup-create.ts:525-529`
  - body: This reads the entire `.tar.gz` and keeps every entry path in memory before publishing every archive. For the multi-GB backups this PR is meant to help, that adds another full pass plus O(entry count) memory pressure; keep this behind `--verify` or validate required roots without materializing the whole archive.
  - confidence: 0.82
