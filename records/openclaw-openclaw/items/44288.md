## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the current backup writer protections:** `src/infra/backup-create.ts:469-475`
  - body: The exclude-enabled path tars `result.assets` directly, so it skips current main's SQLite snapshot/remap flow, volatile-file filter, extension `node_modules` filter, tar EOF retry wrapper, and hardlink cache behavior. A backup created with excludes can therefore archive raw live SQLite state or restore-hostile entries even though the no-exclude path remains protected.
  - confidence: 0.91
- **[P2] Restore the manifest size cap while verifying:** `src/commands/backup-verify.ts:202-210`
  - body: This manifest reader buffers the entire entry and removed the current 1 MiB `MAX_MANIFEST_BYTES` limit, so a malformed or hostile archive can force `backup verify` to allocate arbitrarily large memory before parsing. Keep the existing bounded read while adding tolerant `excludedStats` parsing.
  - confidence: 0.86
- **[P2] Apply excludes to every backed-up asset root:** `src/infra/backup-exclude.ts:358-363`
  - body: Returning `true` for paths outside the state-dir base means external workspaces, custom config files, or external credential directories bypass `--exclude` and `.backupignore` entirely. A warning still leaves sensitive or oversized external assets in the archive despite explicit user rules, so the filter needs per-asset roots or a clearly narrower API before merge.
  - confidence: 0.88

## Risks / Open Questions

- Existing backup privacy and restoreability protections can be skipped when exclude patterns are active, including sanitized SQLite snapshots, volatile-file omission, plugin dependency filtering, tar EOF retries, and hardlink handling.
- External workspaces, custom config paths, or external credential assets can still be archived in full despite user-supplied exclude patterns, which is surprising for a security-motivated exclude feature.
- The PR changes public CLI flags and backup manifest/JSON surface, so maintainers need a product decision on exact exclude semantics before merge.
- The PR does not include inspectable after-fix proof from a real backup create/verify flow; self-reported local use is not enough for this external PR gate.
- The exclude path's two-step writer and full archive entry rescan may worsen the large-backup availability problem the PR is meant to mitigate unless performance is proven on realistic state sizes.
