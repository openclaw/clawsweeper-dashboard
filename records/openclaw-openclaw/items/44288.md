## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Gate automatic .backupignore loading:** `src/infra/backup-exclude.ts:187-194`
  - body: This auto-loads `stateDir/.backupignore` for ordinary backups unless include-all is set, so an existing user can get different archive contents after upgrade without passing any exclude flag. Keep default backup contents unchanged unless maintainers explicitly approve this as the new default and add the matching docs/tests for fresh installs and upgrades.
  - confidence: 0.9
- **[P1] Rebase excludes onto the current backup writer:** `src/infra/backup-create.ts:469-476`
  - body: The exclude branch still builds the archive through an older direct `tar.c` path, but current main's writer now snapshots live SQLite state, filters volatile files, skips extension `node_modules`, keeps temp output outside source roots, and retries tar EOF races. Merging this path without rebasing would risk regressing current backup correctness and availability protections.
  - confidence: 0.88
- **[P2] Apply excludes to external backup assets:** `src/infra/backup-exclude.ts:358-364`
  - body: The filter returns `true` for every path outside its single `baseDir`, while the archive code builds that filter around `stateDir`; external workspace/config assets therefore bypass `--exclude` and `.backupignore` patterns. A warning is not enough for a backup security feature because users may still archive sensitive or large external files they explicitly tried to filter.
  - confidence: 0.86
- **[P2] Preserve ordered duplicate ignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: Deduplicating patterns before passing them to `ignore` changes gitignore semantics because later duplicate rules can intentionally override a negation, such as `*.log`, `!important.log`, `*.log`. This can silently include or exclude files differently from the user's ordered pattern list.
  - confidence: 0.84
- **[P2] Avoid full archive scans during create:** `src/infra/backup-create.ts:525-541`
  - body: The post-create integrity check reads the completed tarball back and stores every entry path for every backup, adding another full archive pass and O(entry count) memory on large archives. That is likely to worsen the same large-backup stall and memory-pressure cases this feature is intended to mitigate, so this should be streaming, verify-only, or bounded.
  - confidence: 0.8
