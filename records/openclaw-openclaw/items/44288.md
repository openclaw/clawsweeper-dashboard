## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve ordered gitignore rules:** `src/infra/backup-exclude.ts:225-233`
  - body: `ignore` evaluates rules in insertion order, including later duplicates after negation. Collapsing duplicates here changes valid rules like `*.log`, `!important.log`, `*.log`, so backups can silently include or exclude files differently from the user's declared pattern order.
  - confidence: 0.91
- **[P1] Apply patterns to each backed-up asset root:** `src/infra/backup-exclude.ts:358-363`
  - body: The filter treats any path outside the single `stateDir` base as included, but `backup create` also archives external workspaces and custom config paths. A user-provided exclude therefore does not apply to those assets, which can still archive large or sensitive files despite explicit exclude rules.
  - confidence: 0.9
- **[P2] Avoid rescanning the completed archive in create:** `src/infra/backup-create.ts:528-535`
  - body: After writing the archive, this reads every tar entry into memory to validate asset presence. On the multi-GB backups this PR targets, that adds a second full decompression pass and O(entry count) memory pressure; use a streaming/bounded check or keep the full scan behind verify behavior.
  - confidence: 0.83
