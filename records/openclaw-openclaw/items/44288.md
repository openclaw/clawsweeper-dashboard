## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Compose excludes with the existing backup filters:** `src/infra/backup-create.ts:473`
  - body: The exclude mode passes only `excludeFilter` to `tar.c`, so after rebasing it would bypass current-main filters for volatile live state and `extensions/*/node_modules`. With any exclude pattern enabled, those paths can return to the archive and reintroduce backup bloat and live-write races; compose the exclude matcher with the existing backup filters before tar sees entries.
  - confidence: 0.91
- **[P2] Run exclude archive writes through the retry wrapper:** `src/infra/backup-create.ts:469`
  - body: The PR-head archive paths call `tar.c` directly. Current main wraps tar writes in retry/error-enrichment logic for EOF races from live files, so an exclude backup can fail once and abort instead of following the hardened mainline behavior.
  - confidence: 0.86
- **[P2] Keep the temp manifest outside source paths:** `src/infra/backup-create.ts:425`
  - body: This stages `manifest.json` directly under `os.tmpdir()`. Current main moved that staging root outside every backed-up asset because cron/sandbox environments can set TMPDIR under `~/.openclaw`, causing tar to include the manifest twice and verify to fail.
  - confidence: 0.88
- **[P2] Preserve duplicate ignore rules in order:** `src/infra/backup-exclude.ts:225-233`
  - body: Deduplicating patterns with first occurrence winning changes gitignore semantics, where later matching rules can intentionally override earlier ones. A rule set like `*.log`, `!important.log`, `*.log` should end with `important.log` excluded, but this drops the final rule and flips the result.
  - confidence: 0.84
