## Summary

Keep this PR open: current main still lacks user-configurable backup excludes, but the proposed implementation still has blocking backup semantics and large-archive concerns, plus the external contributor proof gate is not satisfied.

## What This Changes

The PR adds backup create exclude flags, smart defaults, protected-path guards, `.backupignore` handling, manifest `excludedStats`, the `ignore` dependency, and backup create/verify tests.

## Best Possible Solution

Land a rebased native exclusion implementation that filters every intended backup root consistently, preserves gitignore rule order, avoids full archive rescans on normal create, documents the final CLI contract, and includes redacted real-run proof.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the external-asset bypass, duplicate-rule collapse, and unconditional archive readback path. I did not run a live backup because this is a read-only review.

## Solution Assessment

No. Native excludes are the right direction, but warning-only bypass behavior, rule-order changes, and a full readback in the create path are not the narrowest maintainable solution for large or sensitive backups.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply excludes to every archived asset root:** `src/infra/backup-create.ts:378-386`
  - body: This warning documents that external workspaces or custom config/credential assets bypass the filter, but it still archives those assets in full. Since `resolveBackupPlanFromDisk()` can include roots outside `stateDir`, user rules like `*.pem`, `.env*`, or `logs/` do not protect all content the backup writes.
  - confidence: 0.9
- **[P2] Preserve duplicate ignore rules in order:** `src/infra/backup-exclude.ts:225-233`
  - body: Dropping later duplicate patterns changes gitignore precedence because negations are order-sensitive. A rule list such as `*.log`, `!important.log`, `*.log` should end with `important.log` ignored, but this de-duplication removes the final rule and flips the result.
  - confidence: 0.88
- **[P2] Avoid full archive rescans during create:** `src/infra/backup-create.ts:528-535`
  - body: `listArchiveEntries()` materializes every tar entry before publishing the archive even when the user did not ask for `--verify`. For the multi-GB backups this feature targets, that adds a second full archive pass and memory proportional to entry count to normal create.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch changes backup privacy controls and still has cases where user exclude rules can fail to protect archived sensitive files.
