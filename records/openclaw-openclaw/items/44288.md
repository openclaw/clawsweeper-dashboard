Full review comments:

- **[P1] Rebase without dropping current backup safety paths:** `src/infra/backup-create.ts:469`
  - body: This branch is based on an older backup writer and its archive path does not carry current main's SQLite snapshot backup, volatile-file filter, tar EOF retry, hardlink cache disabling, temp-root/source layout, and extension dependency filter. The exclude feature needs to be layered into the current `createBackupArchive` tar filter path so a merge does not regress backup consistency and large-install reliability.
  - confidence: 0.86
- **[P1] Apply excludes to every backed-up asset root:** `src/infra/backup-exclude.ts:358-364`
  - body: `buildExcludeFilter` treats paths outside `baseDir` as included, while the archive path builds the filter with `plan.stateDir`. That makes explicit `--exclude`/`.backupignore` rules a no-op for valid external workspace, config, or credential assets, so users can still archive large or sensitive files they asked to exclude.
  - confidence: 0.89
- **[P2] Avoid scanning the whole archive during create:** `src/infra/backup-create.ts:525-542`
  - body: The create path calls `listArchiveEntries()` after writing the archive, which reads and stores every tar entry path just to cross-check a few asset roots. On large backups this adds a second full archive pass plus O(number of entries) memory use, undercutting the large-backup use case this feature is meant to help.
  - confidence: 0.82
- **[P2] Preserve duplicate ignore rules in order:** `src/infra/backup-exclude.ts:225-233`
  - body: The pattern resolver drops later duplicates before handing rules to `ignore`, but gitignore precedence is order-sensitive when negations are involved. A rule sequence such as `*.log`, `!important.log`, `*.log` should end with `important.log` excluded; deduplication changes that user-declared behavior.
  - confidence: 0.78
