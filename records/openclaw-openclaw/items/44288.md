## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.97

Full review comments:

- **[P1] Preserve repeated rules in declared order:** `src/infra/backup-exclude.ts:225-233`
  - body: Gitignore rule order is meaningful, including repeated patterns after negations. Deduplicating with the first occurrence winning can flip cases like `*.log`, `!important.log`, `*.log`, causing the backup to include or exclude files contrary to the user’s declared rule order.
  - confidence: 0.97
- **[P1] Apply excludes from each asset root:** `src/infra/backup-create.ts:370-378`
  - body: The filter is rooted only at `stateDir`, so external workspaces or custom config assets return `true` before `ignore` sees the rules. That means explicit excludes and smart defaults can still archive large or sensitive external assets in full; a warning is not enough for the advertised filtering contract.
  - confidence: 0.98
- **[P1] Avoid rescanning the completed archive:** `src/infra/backup-create.ts:528-535`
  - body: After gzip creation, this path calls `listArchiveEntries` and keeps every entry name in memory, then scans the list for each asset. On the multi-GB backups this change targets, that adds a full decompression pass and O(entry count) memory pressure before publish.
  - confidence: 0.95
