## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.83

Full review comments:

- **[P1] Keep .backupignore opt-in before changing default backups:** `src/infra/backup-exclude.ts:189-190`
  - body: `resolveExcludePatterns()` auto-loads `.backupignore` on every backup unless `--include-all` is passed. That means a user who runs the same `openclaw backup create` command after upgrade can silently omit files from the recovery archive even though no exclude flag was requested, contradicting the PR’s backward-compatibility claim. Make this default-off or require explicit maintainer-approved upgrade behavior and tests.
  - confidence: 0.89
- **[P1] Apply excludes to external backup assets:** `src/infra/backup-create.ts:378-386`
  - body: This code only warns when external workspaces or custom config/credential paths sit outside `stateDir`; those assets are still archived in full, so `--exclude '*.pem'` or `.backupignore` can fail to protect sensitive files outside the state directory. The exclude engine needs per-asset bases or another design that actually applies user rules to every backed-up source before merge.
  - confidence: 0.86
- **[P2] Preserve gitignore rule order:** `src/infra/backup-exclude.ts:225-233`
  - body: Deduplicating patterns before passing them to `ignore` changes gitignore semantics because later duplicates can intentionally override an intervening negation, for example `*.log`, `!important.log`, `*.log`. Keep the ordered rule list intact and track stats without collapsing repeated patterns.
  - confidence: 0.84
- **[P2] Avoid rescanning every archive by default:** `src/infra/backup-create.ts:528-529`
  - body: The new integrity check reads the entire `.tar.gz` into an array of entry paths for every backup, even when `--verify` was not requested. On the multi-GB state trees this feature targets, that adds a second full archive pass and O(entry count) memory pressure; make the check bounded, opt-in through verify, or stream the validation instead.
  - confidence: 0.8
