## Summary

Current main already removes the failing v2026.4.29 runtime-deps staging path from bundled plugin startup: old `plugin-runtime-deps` roots are legacy cleanup state, bundled plugin loading uses package-owned source or `dist-runtime` roots, `json5` is declared at the package root, and release verification now guards root dist imports against undeclared runtime dependencies.

## What This Changes

Fix a v2026.4.29 startup race where Telegram providers could observe incomplete bundled runtime dependencies and crash on a missing `json5` import.

## Best Possible Solution

Keep the current main direction and ship it: bundled plugin dependencies should be package-manager-owned, old runtime-deps staging should remain cleanup-only, and release verification should continue blocking undeclared root dist imports.

## Reproduction Assessment

Yes. The issue gives a concrete cold-cache path: delete the v2026.4.29 runtime-deps root, start the gateway, and observe Telegram providers crash on `json5`; I did not run it because this review is read-only, but the reported path maps directly to the old staging surface now removed on main.

## Solution Assessment

Yes. Removing startup dependency staging from bundled plugin loading and guarding root dist imports is the narrow maintainable fix for this class; a Telegram-only `json5` hotfix would leave the shared race intact.
