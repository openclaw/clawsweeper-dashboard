## Summary

Close as implemented on current main. The active bundled-plugin path no longer uses the v2026.4.29 shared `plugin-runtime-deps` staging flow for Telegram/provider startup; current main resolves bundled plugins from package roots and treats old runtime-deps trees as legacy cleanup state.

## What This Changes

Cold-cache gateway startup in v2026.4.29 can race while staging bundled runtime dependencies, leaving Telegram providers without `json5`.

## Best Possible Solution

Keep the current packaged-root bundled plugin model and legacy cleanup path, then let the next release carry the main-only fix to users still on v2026.4.29.

## Reproduction Assessment

Yes for the reported v2026.4.29 failure: the issue includes concrete cold-cache steps and a timestamped provider crash timeline. Against current main, the source-level reproduction target is gone because the shared runtime-deps staging path is now legacy cleanup state.

## Solution Assessment

Yes. Removing the hot startup dependency-staging surface and making packaged roots/root dependencies authoritative is narrower and more maintainable than adding a Telegram-specific `json5` critical-dependency hotfix.
