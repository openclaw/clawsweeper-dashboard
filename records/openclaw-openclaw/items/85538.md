- `P1`: The reported regression made a real operator-facing TUI workflow silent for message-tool-only agents.

## Summary

Current main now preserves targetless successful `message.send` replies that finish with silent `NO_REPLY` by projecting them back into chat history, and the TUI already reloads history when a local run ends without a displayable final message. This addresses the reported v2026.5.20 silent TUI path without undoing message-tool-only duplicate-send suppression.

## What This Changes

The issue reports that TUI sessions become silent after v2026.5.20 when an agent replies through the `message` tool in `message_tool_only` mode.

## Best Possible Solution

Close this as fixed on main and let any broader live-final or route-boundary polish continue in its own PR if maintainers still want it.

## Reproduction Assessment

Yes for the affected release path by source inspection: message-tool-only turns could finish without buffered assistant text, and current main now has the mirror plus TUI history-refresh path that removes the same silent-output condition.

## Solution Assessment

Yes: preserving the message-tool reply through chat history keeps duplicate external delivery suppression intact while making the local TUI display the reply after an empty final.
