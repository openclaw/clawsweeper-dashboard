## Summary

Keep open: current `main` still has a plausible silent-output path for message-tool-only internal UI turns, and the issue describes a recent shipped regression rather than stale or duplicate cleanup.

## What This Changes

The issue reports that `openclaw tui` becomes silent for agents that must reply through `message.send` after message-tool-only sends became terminal.

## Best Possible Solution

Bridge successful `sourceReplySink: "internal-ui"` message-tool payloads into the existing chat final/history projection for internal UI sessions while keeping message-tool-only external delivery terminal to prevent duplicate channel replies.

## Reproduction Assessment

Yes, from source inspection: a chat-linked message-tool-only run can terminate after a successful internal-ui `message.send` while the TUI chat final is populated only from buffered assistant stream text. I did not run the live Signal/TUI flow because this review is read-only.

## Solution Assessment

Yes, the issue’s mirror-to-internal-UI direction is the right maintainable fix if implemented through the existing source-reply payload and transcript mirror seams. The fix should not undo the terminal message-tool behavior from the duplicate-delivery repair.
