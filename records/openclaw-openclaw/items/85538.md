## Summary

Close as superseded: the regression is source-reproducible, but the same remaining TUI/internal-source-reply work is already owned by the open canonical fix PR with live TUI proof.

## What This Changes

The issue reports that after v2026.5.20, TUI sessions for agents configured to always reply with Signal/message.send no longer display replies because successful message-tool-only sends terminate the turn before any visible TUI output is emitted.

## Best Possible Solution

Land or explicitly replace https://github.com/openclaw/openclaw/pull/84126, then close any remaining reports against the landed commit/release.

## Reproduction Assessment

Yes at source level: current main can return a successful internal UI source reply from `message.send`, but the Pi subscriber path does not record that `sourceReply` payload for the final TUI-visible mirror path. I did not run a live TUI session in this read-only review.

## Solution Assessment

Yes: the best fix is the existing canonical PR path, which wires internal-source replies into the current-run TUI/WebChat delivery and transcript mirror flow without making webchat a normal external outbound channel.
