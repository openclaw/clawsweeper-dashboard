The PR reorders Telegram command menu assembly so `customCommands` come before native/plugin commands, updates command-menu tests, and adds a changelog entry.

## Best Possible Solution

Land the custom-command priority order change after contributor-supplied Telegram proof, an effective regression for the 100-command drop path, and refreshed exact-head validation.

## Reproduction Assessment

Yes, from source inspection: current main appends `customCommands` after native/plugin menu entries while the cap keeps the first 100 commands, so custom commands can be dropped when the combined menu exceeds Telegram's command limit. I did not run a live Telegram setup because this review is read-only.

## Solution Assessment

Yes for the functional fix: changing the assembly order in Telegram command registration is narrower than changing the generic cap fitter. The test should still be adjusted to exercise the actual command-count overflow path, and merge readiness depends on real Telegram proof.
