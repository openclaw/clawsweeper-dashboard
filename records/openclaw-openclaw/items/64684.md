## Summary

Keep open: the diagnostic direction is useful, but this PR is not mergeable as submitted because the branch is stale against the current update handler, still skips the new doctor summary for early update failures, contains a duplicate mock key, lacks real behavior proof, and leaves the config-reload scope unclear against the broader self-heal work.

## What This Changes

The PR adds a bounded `openclaw doctor --non-interactive` summary helper, stores and formats `doctorSummary` on restart sentinels for failed `update.run`, and updates adjacent tests/mocks.

## Best Possible Solution

Refresh the PR onto current main, keep the update-failure doctor summary fixed-argv, bounded, and output-limited, preserve existing update/status and restart semantics, and either defer config-reload diagnostics to the self-heal design or implement them with a non-stale immediate reporting path.

## Reproduction Assessment

Yes for the PR defects: source inspection of PR head and current main shows the stale update handler, the early-failure summary gap, and the duplicate mock key. I did not run tests because this review is required to keep the checkout read-only.

## Solution Assessment

No, not as submitted. The narrow diagnostic idea is plausible, but it needs to be rebased onto current main, scoped clearly, fixed for early failures and duplicate mocks, and backed by real behavior proof before it is the maintainable solution.
