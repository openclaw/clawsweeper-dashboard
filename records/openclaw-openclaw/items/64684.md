## Summary

Keep open. Current main still lacks the proposed doctor-summary payload/formatter for failed update runs, so this PR is not obsolete, but the submitted branch is stale against the current update RPC surface, has source-inspected blocking defects, and lacks required real behavior proof.

## What This Changes

This PR adds a bounded `openclaw doctor --non-interactive` summary subprocess for failed `update.run` calls, persists that summary in restart sentinels, formats it in restart messages, and updates adjacent tests/mocks.

## Best Possible Solution

Keep this PR open for maintainer review or author refresh: adapt the useful doctor-summary slice onto the current update handler, preserve existing update/status semantics, keep the subprocess fixed-argv and bounded, and either defer config-reload recovery summaries to https://github.com/openclaw/openclaw/issues/55347 or implement that path with a non-stale diagnostic design.

## Reproduction Assessment

Yes for the PR defects: source inspection of PR head shows the stale `update.run` handler, the root-dependent doctor call that returns no summary for early failures, and the duplicate runtime mock key. I did not run tests because this review is required to keep the checkout read-only.

## Solution Assessment

No, not as submitted. The diagnostic direction is plausible, but the branch should be rebased onto current main, the early-failure summary path fixed, the duplicate mock removed, and real behavior proof added before this is the narrow maintainable solution.
