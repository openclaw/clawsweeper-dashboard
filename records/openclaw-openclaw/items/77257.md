## Summary

Keep open: current main already resolves `openclaw.runtimeExtensions`, but the gateway startup plan still filters configured non-bundled channel plugins unless the plugin was explicitly enabled, which matches the reported silent absence after install/update.

## What This Changes

The issue requests that a correctly installed `@openclaw/discord@2026.5.3` package be discovered and loaded by the gateway from its compiled runtime entries.

## Best Possible Solution

Maintainers should choose a narrow trust rule for official installed channel packages, then align startup planning, doctor/update repair, and diagnostics so configured Discord either loads or tells the operator to run the exact enable/install command.

## Reproduction Assessment

Yes from source inspection: a configured non-bundled channel plugin without an explicit plugin enable marker is excluded from the gateway startup plan even if auto-enable produced an effective allowlist entry. I did not run a live gateway install because this review is read-only.

## Solution Assessment

No clear best fix yet: current tests intentionally preserve the non-bundled plugin trust boundary, so the fix should not simply load every auto-enabled external channel plugin. A safer direction is a maintainer-approved official/trusted-install exception or a repair/diagnostic path that persists the explicit enable marker.
