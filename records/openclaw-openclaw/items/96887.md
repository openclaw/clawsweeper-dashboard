## Label Justifications

- `P1`: The PR targets a recent user-visible message-delivery regression that can send repeated Telegram group replies within one agent run.
- `merge-risk: 🚨 message-delivery`: The diff changes when `message_tool_only` runs terminate after visible source replies, which can still duplicate messages or regress follow-up delivery behavior if the invariant is wrong.

## Summary

Keep open: this is a plausible narrow repair for the linked cascade regression, but it is not merge-ready because the guard only fires after a second visible source reply has already been delivered and the PR lacks full-loop or live Telegram proof that the user-visible cascade is actually gone while the earlier Slack continuation behavior stays fixed.

## What This Changes

The PR adds a per-run `hasDelivered` flag to the message-tool-only after-tool hook so later delivered source replies return `terminate: true`, plus a focused regression test for that hook behavior.

## Best Possible Solution

Define and implement a maintainer-approved `message_tool_only` lifecycle invariant that prevents repeated implicit current-source sends without regressing follow-up non-message tool execution, then prove it in Telegram and the Slack continuation path.

## Reproduction Assessment

Yes, source-level: current main records a delivered `message_tool_only` source reply without returning `terminate`, and the agent loop continues unless finalized tool results terminate. I did not rerun live Telegram during this read-only review.

## Solution Assessment

No, not yet: this branch stops only after a second delivered source reply, so it reduces the cascade but does not prove the duplicate is prevented. The safer path is a maintainer-approved lifecycle guard with live Telegram and Slack-continuation proof.
