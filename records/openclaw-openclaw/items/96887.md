## Summary

Keep open: the source-level bug is real, but this PR is not merge-ready because it still allows a second visible reply before termination, its added agent-loop tests are failing, and it lacks real Telegram proof from the reported surface.

## What This Changes

The PR adds a per-run `hasDelivered` guard to `message_tool_only` source-reply handling and expands agent-loop plus embedded-runner regression tests.

## Best Possible Solution

Land a fix that prevents repeated implicit current-source `message.send` deliveries in one `message_tool_only` run while preserving first-send follow-up tools, with passing regression tests and redacted live Telegram proof; if maintainers accept a two-message cap, make that invariant explicit before merge.

## Reproduction Assessment

Yes, at source level: current main records a delivered source reply without returning `terminate`, and agent-core continues when finalized tool results do not terminate. I did not rerun live Telegram in this read-only review.

## Solution Assessment

No. The after-tool guard is a plausible mitigation, but it fires after the repeated send executes and the new agent-loop tests currently fail; the safer end state is a pre-delivery guard or explicit maintainer acceptance of the cap.
