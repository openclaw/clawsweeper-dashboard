## Summary

Keep open: this targets a real user-visible cascade, but the branch only terminates after the repeated delivery has already happened and still lacks live Telegram or full embedded-run proof.

## What This Changes

The PR adds a per-run `hasDelivered` guard to the message-tool-only after-tool hook and tests repeated `message.send` delivery plus Slack-style follow-up continuation.

## Best Possible Solution

Define and implement the invariant that one implicit current-source `message.send` can complete a `message_tool_only` run without blocking legitimate follow-up tools, then prove it in Telegram and Slack-style continuation paths.

## Reproduction Assessment

Yes, at source level: current main records delivered `message_tool_only` source replies without returning a termination hint, and the loop continues unless every finalized tool result terminates. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The hook-level guard is a useful mitigation, but it terminates only after the second send has executed; the best path is a pre-delivery guard or explicit maintainer acceptance of that cap with proof.
