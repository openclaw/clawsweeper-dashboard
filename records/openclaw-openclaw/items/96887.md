## Summary

Keep open: the branch targets a real message-delivery regression, but the proposed guard fires only after the second `message.send` has already delivered, so it mitigates the cascade without proving the visible duplicate is prevented. It also still needs live/full-loop proof for the Telegram failure path and the Slack continuation behavior preserved by the earlier merged fix.

## What This Changes

The PR adds a per-run `hasDelivered` flag to the `message_tool_only` after-tool hook so later delivered source replies return `terminate: true`, plus a focused hook-level regression test.

## Best Possible Solution

Define and implement a maintainer-approved `message_tool_only` lifecycle invariant that prevents repeated implicit current-source sends without regressing follow-up non-message tool execution, then prove it in Telegram and the Slack continuation path.

## Reproduction Assessment

Yes, source-level: current main records a delivered `message_tool_only` source reply without returning `terminate`, and the agent loop continues unless finalized tool results terminate. I did not rerun live Telegram during this read-only review.

## Solution Assessment

No. This branch stops only after a second delivered source reply, so it reduces the cascade but does not prove the duplicate is prevented; the safer path is a maintainer-approved lifecycle guard with live Telegram and Slack-continuation proof.
