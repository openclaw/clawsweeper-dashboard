## What This Changes

The PR adds a per-run `hasDelivered` guard to `installMessageToolOnlyTerminalHook` and regression tests for repeated `message.send` deliveries and Slack follow-up continuation.

## Best Possible Solution

Set and prove a clear `message_tool_only` invariant: no repeated implicit current-source sends while first-delivery follow-up tools continue, with Telegram-visible and Slack-continuation proof.

## Reproduction Assessment

Yes, at source level: current main records delivered `message_tool_only` source replies without returning a termination hint, and the loop continues unless every finalized tool result terminates. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The hook-level guard reduces the reported 7+ cascade, but it terminates only after the second send has executed; the best path is a pre-delivery guard or explicit maintainer acceptance of that cap plus proof.
