## Summary

Keep open: the PR is a plausible candidate for the open cascade bug, but it is not merge-ready because the guard only terminates after a second visible source reply has already been delivered and the PR still lacks full-loop/live proof for the Telegram regression and Slack continuation path.

## What This Changes

The PR adds a per-run `hasDelivered` guard to the `message_tool_only` after-tool hook so subsequent delivered source replies return `terminate: true`, plus a focused hook-level regression test.

## Best Possible Solution

Implement a maintainer-approved `message_tool_only` lifecycle invariant that prevents repeated implicit current-source sends while preserving follow-up non-message tool execution, then prove it in Telegram and the Slack continuation path.

## Reproduction Assessment

Yes, source-level: current main records a delivered `message_tool_only` source reply without returning `terminate`, and the agent loop continues unless finalized tool results terminate. I did not rerun live Telegram during this read-only review.

## Solution Assessment

No. The PR is a useful mitigation, but it stops only after a second delivered source reply and needs maintainer approval plus live/full-loop proof before it can be called the best fix.
