## Summary

Keep open: this is a useful candidate for the repeated `message_tool_only` cascade, but it is not merge-ready because the guard fires only after a second visible send has already executed and the PR still lacks real Telegram or full user-path proof.

## What This Changes

Adds a per-run `hasDelivered` guard to the message-tool-only after-tool hook and expands unit/agent-loop tests for repeated sends, error recovery, and follow-up tool continuation.

## Best Possible Solution

Define and prove the intended `message_tool_only` invariant before merge: either prevent repeated implicit current-source sends before delivery while preserving first-send follow-up tools, or explicitly accept and document the two-message cap as a bounded mitigation.

## Reproduction Assessment

Yes, at source level: current main records delivered `message_tool_only` source replies without returning a termination hint, and the core loop continues unless finalized tool results terminate. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The hook-level guard is a plausible mitigation, but it stops only after a second send has already delivered; the best fix is a pre-delivery/lifecycle guard or explicit maintainer acceptance of the two-message cap.
