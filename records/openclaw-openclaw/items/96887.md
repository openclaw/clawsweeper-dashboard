## Label Justifications

- `P1`: The PR targets a recent user-visible channel regression where agents can send repeated visible replies in one run.
- `merge-risk: 🚨 message-delivery`: The diff changes when `message_tool_only` source replies terminate the agent loop, which can duplicate or suppress visible chat delivery if the invariant is wrong.

## Summary

Keep open: the PR is a plausible small mitigation, but it is not merge-ready because the new guard still fires only after the repeated `message.send` has already delivered, and the live Telegram plus Slack continuation paths remain unproven.

## What This Changes

The PR adds a per-run `hasDelivered` guard to the `message_tool_only` after-tool hook so subsequent delivered source replies return `terminate: true`, plus a focused regression test.

## Best Possible Solution

Implement or explicitly approve a `message_tool_only` lifecycle invariant that prevents repeated implicit current-source sends while preserving follow-up non-message tools, then prove it in Telegram and the Slack continuation path.

## Reproduction Assessment

Yes, at source level: current main records delivered `message_tool_only` source replies without returning `terminate`, and the loop continues unless every finalized result terminates. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The hook-level guard is a useful mitigation, but it still allows the repeated send to execute before termination and does not cover mixed batches unless every call terminates.
