## What This Changes

The branch adds a per-run `hasDelivered` guard to `installMessageToolOnlyTerminalHook` and tests repeated `message.send` termination plus Slack-style follow-up continuation.

## Best Possible Solution

Set and prove a clear `message_tool_only` invariant: no repeated implicit current-source sends while first-delivery follow-up tools continue, with Telegram-visible proof and Slack-continuation proof.

## Reproduction Assessment

Yes, at source level: current main records delivered `message_tool_only` source replies without returning a termination hint, and the loop continues unless every finalized tool result terminates. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The hook-level guard reduces the reported cascade, but it terminates only after the second send has executed; the best path is a pre-delivery repeated-source-send guard or explicit maintainer acceptance of that cap plus proof.
