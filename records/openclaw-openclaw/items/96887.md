## Summary

Keep open: the PR is a plausible mitigation for the canonical repeated source-reply cascade, but it still allows a second visible delivery before termination and lacks live Telegram or full embedded-run proof, so maintainers need to choose the delivery invariant before merge.

## What This Changes

Adds a per-run `hasDelivered` guard to the embedded message-tool-only after-tool hook and adds unit and agent-loop tests for repeated sends, error recovery, and Slack follow-up continuation.

## Best Possible Solution

Define and prove the intended `message_tool_only` invariant: prevent repeated implicit current-source sends before delivery while preserving first-send follow-up tools, or explicitly accept and document the two-message cap.

## Reproduction Assessment

Yes at source level: current main records a delivered source reply without returning `terminate`, and agent-core continues when the finalized batch is not terminal. I did not run live Telegram in this read-only review.

## Solution Assessment

No. The after-tool guard is a plausible mitigation, but termination happens after the second send executes; the best fix is a pre-delivery lifecycle guard or explicit maintainer acceptance of the two-message cap.
