## Summary

Keep open: the PR is a small relevant fix candidate for the repeated `message_tool_only` cascade, but the current patch can still deliver a second visible duplicate before stopping and lacks full agent-loop or live transport proof that Telegram stops cascading while the prior Slack continuation behavior still works.

## What This Changes

The PR adds a per-run delivered-source-reply flag to the embedded agent `message_tool_only` after-tool hook and a focused regression test so later qualifying `message.send` calls return `terminate: true`.

## Best Possible Solution

Implement or explicitly approve a `message_tool_only` lifecycle invariant that prevents repeated implicit current-source sends while preserving follow-up non-message tools, then prove it in Telegram and the Slack continuation path.

## Reproduction Assessment

Yes, at source level: current `main` records delivered `message_tool_only` source replies without returning `terminate`, and the loop continues unless every finalized result terminates. I did not reproduce the live Telegram cascade in this read-only review.

## Solution Assessment

No. The hook-level guard is a useful mitigation, but it still allows the repeated send to execute before termination and does not cover mixed batches unless every call terminates.
