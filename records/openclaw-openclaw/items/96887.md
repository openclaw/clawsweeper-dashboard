## Summary

Keep open: the runtime guard is a plausible owner-boundary fix for the repeated `message_tool_only` delivery regression, but the added QA scenario currently fails in QA Smoke CI because it treats already-sent progress and marker messages as unexpected outbound traffic, and the external real-behavior proof remains mock/source-level only.

## What This Changes

The PR adds a closure flag to the message-tool-only terminal hook so later delivered `message.send` source replies return `terminate: true`, plus agent-loop/message-tool tests and a QA cascade scenario.

## Best Possible Solution

Keep the localized terminal-hook guard, fix the QA scenario so it proves no third cascade reply without counting progress/status messages, and add real after-fix transport proof or maintainer-approved Mantis proof before merge.

## Reproduction Assessment

Yes, at source level: current main records delivered `message_tool_only` source replies without returning a termination hint, and the agent loop only stops when finalized tool results carry `terminate: true`. I did not rerun a live Telegram reproduction during this read-only review.

## Solution Assessment

Not yet. The runtime hook change is a narrow and plausible mitigation, but the PR is not the best mergeable form until the QA scenario stops failing and maintainers accept the intentional two-visible-send cap or request an earlier lifecycle guard.
