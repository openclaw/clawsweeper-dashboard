## What This Changes

The PR adds a per-run `hasDelivered` guard to `message_tool_only` source-reply handling and adds agent-loop and embedded-run tests for cascade termination while preserving Slack follow-up continuation.

## Best Possible Solution

Define and prove the intended `message_tool_only` invariant: prevent repeated implicit current-source sends before delivery while preserving first-send follow-up tools, or explicitly accept and document the two-message cap.

## Reproduction Assessment

Yes, at source level: current main records a delivered source reply without returning `terminate`, and agent-core continues when the finalized batch is not terminal. I did not rerun live Telegram in this read-only review.

## Solution Assessment

No. The after-tool guard is a plausible mitigation, but termination happens after the repeated send executes; the best fix is a pre-delivery guard or explicit maintainer acceptance of the two-message cap.
