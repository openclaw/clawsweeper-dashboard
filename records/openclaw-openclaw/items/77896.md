## Summary

Keep open: current main still shows a package-boundary mismatch where Matrix runtime code can ship from the root package while the root npm package does not declare the Matrix SDK dependencies needed to start it.

## What This Changes

The issue reports that host npm installs updated to OpenClaw 2026.5.4 cannot start the Matrix channel because `matrix-js-sdk` and Matrix crypto runtime packages are missing from the OpenClaw package root.

## Best Possible Solution

Restore package-boundary consistency so whatever Matrix runtime ships with `openclaw` can resolve its SDK dependencies after `openclaw update`, preferably by fixing root package deps and verification while Matrix remains bundled or by completing externalization consistently.

## Reproduction Assessment

Yes, source inspection plus the reporter's logs give a high-confidence path: a packaged host install with Matrix enabled starts Matrix code that imports `matrix-js-sdk`, while the root package does not declare that dependency and bundled `node_modules` are excluded. I did not execute a live npm update because this sweep is read-only.

## Solution Assessment

Yes, the narrow maintainable fix is package-boundary alignment, not runtime npm installs during gateway startup. If Matrix remains bundled in the root package, the root package and package-acceptance checks should own the runtime deps; if Matrix is externalized, packaging, docs, catalog, and migration behavior need to be made consistent.
