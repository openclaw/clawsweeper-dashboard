## Summary

Keep open. Current main still loads global/config env into the gateway process, but sandbox exec builds a separate environment that does not include process env, and the OpenShell backend attaches provider names without passing resolved provider credential values into the remote command path.

## What This Changes

The issue asks for `.env` or generic OpenShell provider credential values to resolve inside sandbox exec calls instead of leaving `openshell:resolve:env:<KEY>` placeholders in outbound requests.

## Best Possible Solution

Define and document an explicit sandbox env forwarding or OpenShell provider credential handoff for admin-approved keys, then add regression coverage that the configured key reaches sandbox exec while unrelated host secrets remain blocked.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: put the key only in global/config runtime env, run sandbox/OpenShell exec, and the env builder has no path that includes that key unless it is also in sandbox backend env or per-call env.

## Solution Assessment

Unclear whether OpenClaw or OpenShell should resolve the provider placeholder, but blanket `.env` inheritance is not the best fix. The safer path is an explicit admin-approved forwarding contract or provider credential handoff with tests and docs.
