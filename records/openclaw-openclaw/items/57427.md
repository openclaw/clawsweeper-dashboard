## Summary

Keep this issue open. Current main still keeps sandbox exec environments separate from global dotenv/config runtime env and from OpenShell provider credential resolution; the related open PRs show the fix is still pending and needs maintainer/security review around an explicit credential-forwarding contract.

## What This Changes

The issue asks for values such as `NCBI_API_KEY` from `~/.openclaw/.env` or an OpenShell generic provider credential to resolve inside OpenShell-backed sandbox exec calls instead of passing through the `openshell:resolve:env:*` placeholder.

## Best Possible Solution

Define a narrow, documented sandbox env/provider credential contract for admin-approved keys, implement it at the sandbox/OpenShell boundary, and add regression coverage proving configured keys resolve while unrelated host secrets remain blocked.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a key only in global dotenv/config runtime env or as an OpenShell provider credential, then inspect the current sandbox exec path; no code path forwards that value into `buildSandboxEnv` or resolves the OpenShell placeholder.

## Solution Assessment

Yes, keeping this open is the best current handling. The maintainable fix should be explicit allowlisted forwarding or a provider-credential handoff, not blanket dotenv inheritance into sandbox exec.
