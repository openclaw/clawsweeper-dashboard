## Summary

Keep open. The PR targets a real Paperclip pairing docs gap, but the added recovery flow is stale because current main treats `openclaw devices approve --latest` as preview-only; merging this as-is would tell users to run a command that does not approve the device.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing guidance to the Devices CLI and Control UI docs, including `--latest` recovery commands and a persistent `adapterConfig.devicePrivateKeyPem` note.

## Best Possible Solution

Keep the Paperclip first-run docs, but describe list/preview, exact `<requestId>` approval with remote auth flags, and where the Paperclip adapter key belongs.

## Reproduction Assessment

Not applicable as a runtime reproduction; this is a docs PR. The docs verification path is high confidence: current CLI code, tests, and Devices docs all say `--latest` previews only and exact request approval performs the approval.

## Solution Assessment

No. Docs-only is the right narrow surface, but this branch should be updated to match exact request approval and add config-location context before merge.
