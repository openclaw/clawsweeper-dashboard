## Label Justifications

- `P2`: The PR targets a real docs gap in a limited Paperclip gateway pairing workflow, but the current patch would give users a non-approving recovery command.

## Summary

Keep open: the Paperclip documentation gap is valid and the linked issue was closed in favor of this PR, but the PR still documents `openclaw devices approve --latest` as the approval step even though current main treats it as preview-only.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing guidance to the Devices CLI and Control UI docs, including `--latest` recovery commands and `devicePrivateKeyPem` persistence advice.

## Best Possible Solution

Keep the focused Paperclip docs, but rewrite the flow to list or preview pending requests first, approve the exact `requestId` with the same gateway/auth flags, and identify where the Paperclip adapter owns `devicePrivateKeyPem`.

## Reproduction Assessment

Yes. Source inspection of current main shows `--latest` is preview-only and tests assert it does not call `device.pair.approve`, while the PR documents it as the Paperclip approval step.

## Solution Assessment

No, not as written. A Paperclip-specific docs section is the right narrow solution, but the command flow must match the exact-request approval contract and the config snippet needs placement context.
