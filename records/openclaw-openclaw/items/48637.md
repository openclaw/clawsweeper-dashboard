## Summary

Keep open: the linked Paperclip docs issue is closed in favor of this PR and current main still lacks Paperclip-specific guidance, but the PR currently documents a preview-only approval command as if it approves and has no real Paperclip behavior proof.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing documentation to the Devices CLI and Control UI docs.

## Best Possible Solution

Keep this as the canonical Paperclip docs PR, but rewrite the recovery flow to list or preview pending requests and approve the exact `requestId` with the same auth flags, then add sourced config-placement guidance and redacted real Paperclip proof.

## Reproduction Assessment

Yes. The blocking docs mismatch is source-reproducible: current main treats omitted `requestId` or `--latest` as preview-only, exits before approval, and tests assert no approval call is made.

## Solution Assessment

No, not as written. A focused Paperclip docs section is the right minimal direction, but the command sequence must match the exact-request approval contract and the Paperclip key placement needs sourcing before merge.
