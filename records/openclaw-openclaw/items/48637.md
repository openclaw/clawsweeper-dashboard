## Summary

Keep open: the Paperclip documentation gap is real and this PR remains the canonical open follow-up, but the patch currently tells users to run a preview-only approval command and lacks config-location guidance plus real behavior proof.

## What This Changes

The branch adds Paperclip `openclaw_gateway` first-run pairing docs to the Devices CLI and Control UI pages, including `--latest` approval commands and `devicePrivateKeyPem` persistence guidance.

## Best Possible Solution

Keep a focused Paperclip docs fix, but rewrite the recovery path to list or preview pending requests and approve the exact `requestId` with the same auth flags, then add sourced config-placement guidance for `devicePrivateKeyPem`.

## Reproduction Assessment

Yes. Current main source and tests show `--latest` only previews and exits before `device.pair.approve`, so the PR's approval command sequence is source-reproducibly wrong.

## Solution Assessment

No, not as written. A Paperclip-specific docs section is the right minimal direction, but the commands must match the exact-request approval contract and the config snippet needs placement context.
