## Summary

Keep open: this PR targets a real Paperclip gateway pairing documentation gap, but the patch is not merge-ready because it documents `openclaw devices approve --latest` as the approval action even though current main treats that path as preview-only and requires a follow-up approval by exact request ID.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing guidance to the Devices CLI and Control UI docs, including `--latest` examples and `devicePrivateKeyPem` persistence advice.

## Best Possible Solution

Keep the focused Paperclip docs fix, but rewrite the flow to list or preview pending requests first, approve the exact `requestId` with the same gateway/auth flags, and identify where the Paperclip adapter owns `devicePrivateKeyPem`.

## Reproduction Assessment

Yes. Source inspection of current main shows `--latest` is preview-only and tests assert it does not call `device.pair.approve`, while the PR documents it as the Paperclip approval step.

## Solution Assessment

No, not as written. A Paperclip-specific docs section is the right narrow direction, but the command flow must match the exact-request approval contract and the config snippet needs placement context.
