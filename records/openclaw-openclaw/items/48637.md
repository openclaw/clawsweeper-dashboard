## Summary

Keep open. The linked Paperclip pairing docs need is real and current main still lacks Paperclip-specific guidance, but this branch conflicts with current shipped Devices CLI behavior by presenting `--latest` as an approval command even though it is now preview-only and requires rerunning with an exact request ID.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing guidance to the Devices CLI and Control UI docs, including recovery commands and a persistent `adapterConfig.devicePrivateKeyPem` note.

## Best Possible Solution

Revise the docs to describe the current safe flow: trigger the pending Paperclip request, inspect it with `openclaw devices list` or the preview output, approve the exact `<requestId>` with any required remote auth flags, and add a Paperclip config-location sentence or link for `adapterConfig.devicePrivateKeyPem`.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The review has a high-confidence docs/code verification path: compare the PR text with current Devices CLI implementation, tests, and `v2026.4.27` docs showing `--latest` previews and exits before approval.

## Solution Assessment

No. A docs-only fix is a reasonable narrow path, but this branch is not the best solution because it documents a preview command as the approval command and leaves first-time users without the Paperclip config location for the persistent key.

## Review Findings

Full review comments:

- **[P2] Document exact request approval instead of --latest:** `docs/cli/devices.md:105-111`
  - body: Current `openclaw devices approve --latest` only previews the newest pending request and exits before calling `device.pair.approve`. This section presents it as the recovery approval step, so following the docs leaves Paperclip unapproved; update the flow to list or preview the pending request, then approve the exact `<requestId>` with the same auth flags, and make the same correction in the Control UI paragraph.
  - confidence: 0.97
