## Summary

Keep open. The Paperclip pairing docs topic is valid and still missing from current main, but this branch is not merge-ready because it presents `openclaw devices approve --latest` as the approval step even though current main intentionally makes implicit/latest approval preview-only.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing guidance to the Devices CLI docs and Control UI docs, including recovery commands and a persistent `adapterConfig.devicePrivateKeyPem` note.

## Best Possible Solution

Keep the Paperclip first-run documentation, but make it match the shipped safe flow: trigger or list the pending request, inspect/preview it, approve the exact `<requestId>` with the needed remote auth flags, and identify where the Paperclip adapter key belongs.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The review has a high-confidence docs verification path: compare the PR text against current main's CLI implementation and tests, which show `--latest` is preview-only and exact request approval is required.

## Solution Assessment

No. A docs-only change is the right narrow surface for the Paperclip onboarding gap, but this branch should not merge until it documents exact request approval and adds config-location context.
