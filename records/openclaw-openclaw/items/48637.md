## Summary

Keep open: this docs-only PR is the canonical follow-up for the closed Paperclip pairing docs issue, but the current diff conflicts with main's exact-request approval contract and would tell users to run a preview-only command as the approval step.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run device pairing guidance to the Devices CLI and Control UI docs.

## Best Possible Solution

Keep the focused Paperclip docs, but rewrite the flow to list or preview pending requests first, approve the exact `requestId` with the same gateway/auth flags, and identify where `devicePrivateKeyPem` belongs.

## Reproduction Assessment

Yes. Source inspection of current main shows `--latest` is preview-only and tests assert it does not call `device.pair.approve`, while the PR tells Paperclip users to use it as the approval step.

## Solution Assessment

No, not as written. A Paperclip-specific docs section is the right narrow solution, but the command flow must match the exact-request approval contract and the config snippet needs placement context.
