## Summary

Keep open: the Paperclip docs gap is real, but this PR is not merge-ready because it documents `openclaw devices approve --latest` as the approval command even though current main treats that path as preview-only.

## What This Changes

This PR adds Paperclip `openclaw_gateway` first-run pairing guidance to the Devices CLI and Control UI docs, including `--latest` examples and `adapterConfig.devicePrivateKeyPem` persistence advice.

## Best Possible Solution

Keep the focused Paperclip docs fix, but rewrite the flow to list or preview pending requests first, approve the exact `requestId` with the same gateway/auth flags, and identify where Paperclip owns `devicePrivateKeyPem`.

## Reproduction Assessment

Yes for the PR defect. Source inspection and tests show `--latest` is preview-only and does not call `device.pair.approve`; no live CLI run was performed because this review is read-only.

## Solution Assessment

No, not as written. A Paperclip-specific docs section is the right narrow direction, but the command flow must match the exact-request approval contract and the config snippet needs placement context.
