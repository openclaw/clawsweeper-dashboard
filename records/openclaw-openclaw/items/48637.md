## Summary

Keep open: current main still lacks Paperclip-specific `openclaw_gateway` pairing guidance and the linked issue was closed in favor of this PR, but the branch documents `openclaw devices approve --latest` as the approval command even though current main makes implicit/latest approval preview-only and requires an exact `requestId`.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing guidance to the Devices CLI and Control UI docs, including approval commands and `devicePrivateKeyPem` persistence.

## Best Possible Solution

Keep the PR focused on Paperclip docs, but rewrite the recovery flow to list or preview pending requests and then approve the exact `requestId` with the same auth flags, while sourcing the `devicePrivateKeyPem` config placement.

## Reproduction Assessment

Yes. The blocking docs mismatch is source-reproducible: current main treats omitted `requestId` or `--latest` as preview-only, exits with status 1, and tests assert no approval call is made.

## Solution Assessment

No, not as written. A Paperclip docs section is the right minimal fix for the linked docs gap, but the command sequence must match the current exact-request approval contract before merge.
