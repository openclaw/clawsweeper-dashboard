## Summary

Keep this PR open: it addresses a real Paperclip pairing documentation gap, but the current patch is not merge-ready because it documents `openclaw devices approve --latest` as the approval command even though current main treats that form as preview-only.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing guidance to the Devices CLI and Control UI docs, including approval examples and `adapterConfig.devicePrivateKeyPem` persistence advice.

## Best Possible Solution

Keep the focused Paperclip docs fix, but rewrite the flow to list or preview pending requests first, approve the exact `requestId` with the same gateway/auth flags, and identify where Paperclip owns `devicePrivateKeyPem`.

## Reproduction Assessment

Yes. Source inspection of current main shows `--latest` is preview-only and tests assert it does not call `device.pair.approve`; a live CLI run was not performed because this review is read-only.

## Solution Assessment

No, not as written. A Paperclip-specific docs section is the right narrow direction, but the command flow must match the exact-request approval contract and the config snippet needs placement context.
