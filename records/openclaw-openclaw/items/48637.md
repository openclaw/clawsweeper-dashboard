## Summary

Keep open; the Paperclip docs gap is real, but the patch is not correct as written because it documents `--latest` as the approval action even though current `main` treats `--latest` as preview-only. The remaining work is a narrow docs correction.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing guidance to the Devices CLI and Control UI docs, including `--latest` examples and `devicePrivateKeyPem` persistence guidance.

## Best Possible Solution

Keep the focused Paperclip first-run docs, but rewrite the recovery flow to list or preview pending requests and then approve the exact `requestId` with the same gateway/auth flags, plus clear placement context for `devicePrivateKeyPem`.

## Reproduction Assessment

Yes. Source inspection of current `main` shows `--latest` is preview-only, and the devices CLI tests assert that it does not call `device.pair.approve`.

## Solution Assessment

No, not as written. A Paperclip-specific docs section is the right narrow direction, but the command flow must match the exact-request approval contract and the config snippet needs placement context.
