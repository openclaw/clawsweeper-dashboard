## Summary

Keep open: the Paperclip docs gap is still real and current main does not contain Paperclip-specific pairing guidance, but this patch is not merge-ready because it documents `openclaw devices approve --latest` as the approval command even though current main treats it as preview-only.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing guidance to the Devices CLI and Control UI docs, including `--latest` examples and `devicePrivateKeyPem` persistence advice.

## Best Possible Solution

Keep the Paperclip-specific docs, but revise the flow to list or preview pending requests first, approve the exact `requestId` with the same gateway/auth flags, and identify where Paperclip owns `devicePrivateKeyPem`.

## Reproduction Assessment

Yes. Source inspection of current main shows `--latest` is preview-only and tests assert it does not call `device.pair.approve`, while the PR documents it as the Paperclip approval step.

## Solution Assessment

No, not as written. A Paperclip-specific docs section is the right narrow direction, but the command flow must match the exact-request approval contract and the config snippet needs placement context.
