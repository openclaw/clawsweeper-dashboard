## Summary

Keep this PR open: the linked issue was closed to this PR as the canonical Paperclip docs fix, and current main still lacks Paperclip-specific guidance, but the branch currently documents `--latest` as an approval command even though current CLI behavior makes it preview-only.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing approval guidance to the Devices CLI and Control UI docs.

## Best Possible Solution

Keep the focused Paperclip first-run docs, but rewrite them to list or preview pending requests first, approve the exact `requestId` with the same gateway/auth flags, and add clear `devicePrivateKeyPem` placement context.

## Reproduction Assessment

Yes. Source inspection of current main shows `--latest` is preview-only, and tests assert it does not call `device.pair.approve`, while the PR tells Paperclip users to use it as the approval step.

## Solution Assessment

No, not as written. A Paperclip-specific docs section is the right narrow solution, but the command flow must match the exact-request approval contract and the config snippet needs placement context.
