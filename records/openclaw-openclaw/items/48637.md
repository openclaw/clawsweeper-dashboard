## Summary

Keep open: the docs-only PR targets a real Paperclip pairing documentation gap, but it currently documents `openclaw devices approve --latest` as an approval command while current main treats that form as preview-only and requires rerunning with an exact request ID.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing docs to `docs/cli/devices.md` and `docs/web/control-ui.md`, including `--latest` approval commands and `devicePrivateKeyPem` persistence guidance.

## Best Possible Solution

Keep the focused Paperclip docs addition, but rewrite the recovery path to list or preview pending requests and then approve the exact `requestId` with the same gateway/auth flags, plus sourced placement guidance for `devicePrivateKeyPem`.

## Reproduction Assessment

Yes. Current main source and tests show `--latest` only previews the selected pending request and exits before approval, so the PR's command sequence is source-reproducibly wrong.

## Solution Assessment

No, not as written. A Paperclip-specific docs section is the right narrow direction, but the commands must match the exact-request approval contract and the config snippet needs placement context.

## Review Findings
