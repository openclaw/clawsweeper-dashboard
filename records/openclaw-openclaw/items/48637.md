## Summary

Keep this PR open: the Paperclip pairing docs gap is still not implemented on current main, and the linked issue is already routed here, but the proposed command sequence conflicts with the current devices CLI contract because `--latest` is preview-only.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing docs to the Devices CLI and Control UI pages, including `--latest` approval commands and `devicePrivateKeyPem` persistence guidance.

## Best Possible Solution

Keep the focused Paperclip docs addition, but rewrite the recovery flow to list or preview pending requests and then approve the exact `requestId` with the same gateway/auth flags, plus sourced placement context for `devicePrivateKeyPem`.

## Reproduction Assessment

Yes. Current main source and tests show `--latest` only previews the selected pending request and exits before approval, so the PR’s command sequence is source-reproducibly wrong.

## Solution Assessment

No, not as written. A Paperclip-specific docs section is the right narrow direction, but the commands must match the exact-request approval contract and the config snippet needs placement context.
