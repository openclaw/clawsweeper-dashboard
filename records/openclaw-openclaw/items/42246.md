## Summary

Keep open: current main and v2026.6.10 still provide durable Telegram sends, payload batching, media-group buffering, retry_after-aware retries, and per-group API fairness, but not configurable time-window digest aggregation for unrelated outbound notifications. The remaining work is real, but it needs maintainer product decisions before implementation.

## What This Changes

The issue requests an opt-in Telegram notification digest layer that batches unrelated outbound events over a configurable time window before delivery.

## Best Possible Solution

Define a Telegram/plugin-owned notification digest policy above the transport send layer, persist pending digest state in SQLite when durability is required, and deliver completed digest payloads through the existing durable Telegram delivery and throttling paths.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 have no outbound Telegram digest configuration or time-window aggregation path, while the inspected send paths deliver caller-supplied payloads through existing durable retry/throttling mechanics.

## Solution Assessment

No, not as a Telegram send-layer ring buffer alone. Semantic grouping, urgency bypass, media handling, and durability should be decided by a notification policy or producer-side seam before Telegram handles formatting, limits, retries, and final delivery.
