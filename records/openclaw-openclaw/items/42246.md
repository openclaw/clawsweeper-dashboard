## Summary

Keep open: current main and v2026.6.8 still provide direct Telegram delivery, chunking, retry/backoff, media-group buffering, durable payload batches, and per-chat API fair queueing, but not configurable time-window digest aggregation for unrelated outbound notifications. The gap is real, but the implementation needs maintainer product decisions around digest ownership, config, urgency bypass, media/thread handling, and durable flushing.

## What This Changes

Request configurable opt-in batching/digest aggregation for outbound Telegram notifications so bursts of short automation updates are sent as fewer grouped messages.

## Best Possible Solution

Define a plugin-owned notification or digest policy seam above Telegram transport, persist pending digest state in SQLite when durability is required, and send completed digest payloads through the existing durable Telegram delivery and throttling paths.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 have no outbound Telegram digest configuration or time-window aggregation path, while the inspected send paths deliver caller-supplied payloads through existing retry/throttling mechanics.

## Solution Assessment

No, not as a Telegram send-layer ring buffer alone. Semantic grouping, urgency bypass, media handling, and durability should be decided by a notification policy or producer-side seam before Telegram handles formatting, limits, retries, and final delivery.
