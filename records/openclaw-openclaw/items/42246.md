## Summary

Keep open: current main and v2026.6.8 still provide direct Telegram delivery, chunking, retry/backoff, media-group buffering, durable payload batches, and per-chat API fair queueing, but not configurable time-window digest aggregation for unrelated outbound notifications. The gap is real, but implementation needs maintainer product decisions around ownership, config, urgency bypass, media/thread handling, and durable flushing.

## What This Changes

The issue requests configurable aggregation of outbound Telegram notifications into fewer digest-style Telegram messages over a configurable time window.

## Best Possible Solution

Define a notification-policy or producer-side digest seam above Telegram transport, persist pending digests in SQLite when durability is required, and send completed digest payloads through the existing durable channel delivery and Telegram retry/throttling paths.

## Reproduction Assessment

Yes, at source level. Current main has no outbound Telegram digest configuration or time-window aggregation path, while the inspected send paths deliver caller-supplied payloads directly through existing retry/throttling mechanics.

## Solution Assessment

No, not as a Telegram send-layer ring buffer alone. Semantic grouping, urgency bypass, media handling, and durability should be decided by a notification policy or producer-side seam before Telegram handles formatting, limits, retries, and final delivery.
