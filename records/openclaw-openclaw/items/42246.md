## Summary

Keep open: current main and v2026.6.6 still do not provide configurable outbound Telegram digest aggregation. The existing Telegram paths cover direct delivery, text chunking, media-group buffering, retry_after-aware retry, durable payload batches, and a per-chat API fair queue, but they do not collect unrelated notification events over a time window into a digest; the remaining work needs maintainer product decisions before implementation.

## What This Changes

Add an opt-in Telegram outbound notification batching feature that aggregates many short events into digest messages.

## Best Possible Solution

Define a typed notification policy or producer-side digest seam above Telegram transport, persist pending digests in SQLite when durability is required, and send completed portable payloads through the existing durable channel delivery and Telegram throttling paths.

## Reproduction Assessment

Yes, at source level: current main has no outbound Telegram digest configuration or time-window aggregation path, and sends each outbound payload through the Telegram adapter directly. This is a missing feature rather than a failing existing behavior contract.

## Solution Assessment

No, the proposed Telegram-layer ring buffer is only a plausible implementation sketch, not the best boundary. Semantic grouping, urgency bypass, and durability should be decided by a notification policy or producer-side seam before Telegram handles formatting, limits, retries, and final delivery.
