## Summary

Keep open: current main still lacks configurable outbound Telegram digest aggregation. The existing Telegram paths cover direct sends, text chunking, durable send batches, retry_after-aware retry, and per-group/topic fair queueing, but none collect unrelated notifications over a configurable window or render a source/severity digest; this still needs maintainer product decisions before implementation.

## What This Changes

Request configurable, opt-in batching of outbound Telegram notifications into digest messages instead of sending each event immediately.

## Best Possible Solution

Define a typed notification/digest policy above the Telegram transport, persist pending digests in SQLite when durability is required, and send completed portable payloads through the existing durable channel delivery and Telegram throttling paths.

## Reproduction Assessment

Yes at source level: current main exposes Telegram send, chunking, retry, durable payload batches, and fair queueing paths, but no configurable time-window digest aggregator. This is not a failing-runtime bug reproduction; it is a verified missing feature surface.

## Solution Assessment

Unclear as proposed: a Telegram-specific ring buffer is plausible, but the best boundary is likely a producer-side notification policy above transport so severity, source grouping, urgent bypass, durability, and cross-channel semantics do not become Telegram-only transport logic.
