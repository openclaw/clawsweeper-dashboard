## Summary

Keep open: current main and v2026.5.20 have adjacent Telegram throttling, retry, chunking, inbound/media batching, and durable rendered-batch primitives, but they still do not provide configurable outbound notification digest aggregation. The remaining work is a new opt-in Telegram delivery/config feature that needs maintainer product design.

## What This Changes

Add opt-in Telegram outbound notification batching that collects many short automation updates into digest messages with configurable flush and grouping behavior.

## Best Possible Solution

Define a Telegram-owned digest aggregation contract that composes with current per-token throttling, retry_after handling, durable final delivery, media behavior, priority bypasses, shutdown flushing, and docs/tests before implementation.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature request. Source, docs, and v2026.5.20 inspection give high confidence that configurable Telegram outbound digest aggregation is not implemented today.

## Solution Assessment

Unclear. Native Telegram digest aggregation may be reasonable, but maintainers need to decide config scope, priority bypasses, media/durability behavior, shutdown flushing, and interaction with https://github.com/openclaw/openclaw/issues/69165.
