## Summary

The requested digest behavior is still absent from current main and v2026.6.5: Telegram supports payload chunking, durable logical-send batches, retry/throttling, and fair API serialization, but it does not aggregate unrelated notifications over a time window. The issue should remain open for a maintainer product decision because semantic grouping, urgent-message bypass, media/thread handling, public configuration, and durable buffering cannot safely be chosen at the Telegram transport layer alone.

## What This Changes

Add an opt-in policy that groups multiple outbound Telegram notifications over a configurable window and sends them as digest messages.

## Best Possible Solution

Define a typed notification-policy or plugin seam above channel transport that can classify digest-eligible events and urgent bypasses, persist pending digests in the appropriate SQLite store, and submit completed portable payloads through the existing durable channel and Telegram throttling paths.

## Reproduction Assessment

Not applicable: this is a request for a new digest delivery mode, and source plus release inspection confirms current behavior intentionally sends logical outbound payloads without cross-notification time-window aggregation.

## Solution Assessment

No: the proposed Telegram-layer ring buffer is not the best maintainable boundary because semantic grouping belongs with the notification producer or a typed policy seam; Telegram should continue to own transport formatting, limits, retries, and final delivery.

## Risks / Open Questions

- A Telegram-local aggregator would have to infer product semantics such as event type, severity, and source from transport payloads that do not reliably carry those facts.
- Delaying independent sends can change ordering, approval/error visibility, reply and topic association, media behavior, and user expectations unless eligible and bypass classes are explicit.
- A persistent digest queue needs a canonical SQLite owner plus defined crash, restart, shutdown, overflow, deduplication, and retry behavior; an in-memory ring buffer can silently lose notifications.
- The proposed configuration adds a broad upgrade-sensitive surface before the product has chosen whether batching is per account, target, topic, workspace, event class, or notification producer.
