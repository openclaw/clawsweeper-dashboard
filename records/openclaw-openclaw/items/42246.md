## Summary

Keep open: current main has adjacent Telegram throttling, retry_after-aware retry, and durable rendered-batch delivery support, but it still lacks the requested configurable outbound digest aggregation layer; this is a feature/config/product decision rather than conservative cleanup.

## What This Changes

The issue requests opt-in configurable aggregation of multiple outbound Telegram notifications into digest messages over a time window.

## Best Possible Solution

Define a Telegram-owned digest aggregation contract that composes with the existing token-scoped throttler, retry_after handling, durable delivery queue, media handling, priority rules, and shutdown semantics before adding config schema, docs, and tests.

Unclear. Native Telegram digest aggregation may be reasonable, but the delivery guarantees, config shape, priority behavior, and interaction with the open outbound queue request need maintainer product design first.
