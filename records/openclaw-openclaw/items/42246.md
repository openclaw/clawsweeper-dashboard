## Summary

Keep open: current main and the latest release have adjacent Telegram throttling, retry-after-aware retries, chunking, and durable rendered-batch delivery, but they still do not provide the requested opt-in Telegram digest aggregation layer. The remaining work is a new Telegram delivery/config feature that needs maintainer product design before implementation.

## What This Changes

Request configurable outbound Telegram notification batching so many short events can be collected into fewer digest messages.

## Best Possible Solution

Design a Telegram-owned digest aggregation contract that composes with current token-scoped throttling, retry_after handling, durable delivery, media handling, priority bypasses, and shutdown behavior, then add opt-in config, docs, focused tests, and live Telegram proof.

## Solution Assessment

Unclear. Native Telegram digest aggregation may be reasonable, but the best design depends on maintainer decisions about config scope, priority bypasses, media, durability, shutdown flushes, and interaction with the related outbound queue/rate-limit work.
