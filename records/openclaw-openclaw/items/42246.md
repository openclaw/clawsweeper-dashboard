## Summary

Keep open: current main still lacks configurable outbound Telegram notification digest/aggregation behavior. Existing inbound debounce, media-group buffering, shared Telegram API throttling, and retry handling are adjacent but do not combine multiple outbound notifications into fewer digest messages.

## What This Changes

Requested configurable Telegram notification batching so multiple outbound events can be sent as digest messages instead of individual immediate messages.

## Best Possible Solution

Define a Telegram plugin-owned aggregation design that composes with transport throttling, retries, media, streaming previews, error priority, and shutdown semantics before adding schema, docs, and focused tests.

## Reproduction Assessment

Not applicable. This is an enhancement request, and current-main source inspection is enough to show the requested digest configuration and aggregation layer are not present.

## Solution Assessment

Unclear. A native Telegram aggregation layer may be reasonable, but the exact config shape and delivery semantics should be designed alongside https://github.com/openclaw/openclaw/issues/69165 before implementation.
