## Summary

Keep open: current main and v2026.5.20 have adjacent Telegram throttling, retry-after-aware retries, chunking, and durable rendered-batch delivery, but they do not provide configurable Telegram digest aggregation. This remains a feature/config/product-design item rather than a conservative cleanup close.

## What This Changes

Add configurable Telegram outbound notification digest batching so bursts of short updates can be grouped into fewer readable messages.

## Best Possible Solution

Define a Telegram-owned opt-in digest aggregation contract that composes with current throttling, retry, durable delivery, media handling, priority bypass rules, and shutdown behavior, then add config, docs, focused tests, and real Telegram proof.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source and shipped-release inspection confirm current main and v2026.5.20 do not expose the requested Telegram digest aggregation config or delivery behavior.

## Solution Assessment

Unclear. Native Telegram digest aggregation may be reasonable, but maintainers need to decide config scope, priority bypasses, media handling, durability, shutdown flushes, and coordination with https://github.com/openclaw/openclaw/issues/69165 before implementation.
