## Summary

Keep this open. Current main still has Telegram retry handling, chunking, preview controls, and recovery delivery plumbing, but no configurable outbound batching/digest aggregation surface; the related open #69165 is complementary rate-limit queueing, not a replacement for grouping notifications into digests.

## What This Changes

Add an opt-in Telegram outbound notification batching layer that aggregates many short events into configurable digest messages instead of sending every event immediately.

## Best Possible Solution

Add a Telegram-owned, opt-in batching feature with config schema, UI metadata, docs, and focused tests. It should aggregate only eligible notification-style text payloads, preserve immediate delivery for critical or unsupported payloads, and compose cleanly with any future per-chat send queue/rate-limit layer.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. There is a high-confidence verification path from current main: schema, docs, adapter, and send code show direct outbound delivery with chunking/retry but no digest batching.

## Solution Assessment

Yes on direction, unclear on final policy details. A Telegram plugin-owned batching layer is the maintainable boundary, but the policy decisions above should be settled before an automated implementation PR attempts it.
