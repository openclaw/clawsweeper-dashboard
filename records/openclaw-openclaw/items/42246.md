## Summary

Keep open: current main and v2026.6.10 still have Telegram direct delivery, retry/throttling, durable payload batching, and inbound/media batching concepts, but not configurable time-window digest aggregation for unrelated outbound Telegram notifications.

## What This Changes

Add opt-in Telegram outbound notification batching/digest aggregation so bursts of short notifications can be grouped into fewer messages.

## Best Possible Solution

Define a plugin-owned notification or digest policy seam above Telegram transport, persist pending digest state in SQLite when durability is required, and send completed digest payloads through the existing durable Telegram delivery and throttling paths.

## Reproduction Assessment

Yes, at source level: current main has direct Telegram delivery, durable payload batches, media album buffering, and retry/throttling, but no configurable timed digest aggregation surface.

## Solution Assessment

No, not as a Telegram send-layer ring buffer alone. Semantic grouping, urgency bypass, media handling, and durability should be decided by a notification policy or producer-side seam before Telegram handles formatting, limits, retries, and final delivery.
