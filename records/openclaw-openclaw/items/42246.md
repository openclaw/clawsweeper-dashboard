## Summary

Keep open: current main still has durable outbound payload batches, Telegram chunking, media-group buffering, retry-aware sending, and per-chat API throttling, but no configurable time-window digest/aggregation layer for unrelated Telegram notifications. The remaining work is a feature/config/product decision rather than an already-implemented bug fix.

## What This Changes

Requested configurable batching of outbound Telegram notifications into digest messages over a time window.

## Best Possible Solution

Define a typed notification policy or producer-side digest seam above Telegram transport, persist pending digests in SQLite if durability is required, then send completed portable payloads through the existing durable outbound delivery and Telegram throttling paths.

## Reproduction Assessment

Yes, at source level: current main has no outbound Telegram digest configuration or time-window aggregation path, and core delivery iterates the already-supplied payloads through the Telegram adapter. This is a missing feature rather than a failing existing behavior contract.

## Solution Assessment

No, the issue's Telegram-layer ring-buffer sketch is only a plausible implementation, not the best confirmed boundary. Semantic grouping, urgent bypass, media handling, and durability should be decided by a notification policy or producer-side seam before Telegram handles final formatting and delivery.
