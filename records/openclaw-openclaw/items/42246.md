## Summary

Keep open: current main and v2026.6.6 still lack configurable outbound Telegram notification digesting. Existing code covers direct sends, chunking, durable logical-send batches, retry/throttling, and media-group buffering, but not time-window aggregation of unrelated notifications; the remaining work needs a maintainer product decision on ownership, semantics, config, and durability.

## What This Changes

The issue requests configurable aggregation of outbound Telegram notifications into fewer digest messages.

## Best Possible Solution

Define a typed notification-policy or plugin seam above channel transport that classifies digest-eligible events and urgent bypasses, persists pending digests in SQLite, and submits completed portable payloads through existing durable channel and Telegram throttling paths.

## Reproduction Assessment

Yes, at source level for the missing capability. Current Telegram outbound sends are direct per payload and the visible queue/throttle paths do not aggregate unrelated notifications over a time window; this is a feature gap rather than a failing bug reproduction.

## Solution Assessment

No. The proposed Telegram-layer ring buffer is not the best maintainable boundary because semantic grouping belongs with the notification producer or a typed policy seam; Telegram should continue to own transport formatting, limits, retries, and final delivery.
