## Summary

Keep open. Current main mitigates part of the reported pressure with `cron.sessionRetention` and durable conversation-pointer protections, but it still has no ordered `session.maintenance.rules` or per-type retention contract for arbitrary session patterns.

## What This Changes

Request ordered per-pattern or per-type session maintenance retention rules so ephemeral cron/subagent sessions can prune faster than long-lived interactive sessions.

## Best Possible Solution

Add an additive `session.maintenance.rules` contract only after maintainer design approval, with ordered first-match retention, a documented matcher shape, global `pruneAfter` fallback, schema/help/docs coverage, and tests proving protected conversations and cron-specific behavior stay safe.

## Reproduction Assessment

Not applicable; this is a feature/config request rather than a broken existing behavior report. Source inspection confirms current main lacks the requested rules surface while retaining only global session maintenance plus cron-specific retention and built-in protected-entry logic.

## Solution Assessment

Unclear until maintainers choose the public matcher contract. Per-pattern rules are a plausible direction, but the safest next step is product/API review before any automated fix because the change affects destructive retention policy.
