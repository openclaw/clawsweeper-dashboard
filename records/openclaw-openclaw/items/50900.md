## Summary

Keep open: current main still exposes only global session maintenance thresholds plus separate cron-run retention, while the requested ordered per-pattern or per-type retention rule contract remains unimplemented and needs product/API design.

## What This Changes

The issue requests ordered `session.maintenance.rules` so ephemeral session patterns can prune sooner than interactive or durable conversation sessions.

## Best Possible Solution

Define one additive core `session.maintenance.rules` contract with documented matcher semantics, first-match fallback to global `pruneAfter`, schema/help/docs coverage, and centralized pruning that preserves existing durable-session protections.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection gives high confidence that current main cannot express or apply ordered per-pattern session maintenance rules.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but the maintainable fix should define matcher and deletion semantics centrally rather than add ad hoc checks to individual pruning loops.
