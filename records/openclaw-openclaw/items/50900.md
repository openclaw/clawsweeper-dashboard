## Summary

Keep open: current main and the latest release still have global session maintenance age/count thresholds plus separate cron-run retention, but no ordered per-pattern or per-type retention rules. The KeepMyClaw backup comment is useful adjacent context, and newer cleanup/preservation work reduces some risk for topics and groups, but it does not replace a native OpenClaw retention policy for operators who need different lifetimes for cron, subagent, DM, and interactive sessions.

## What This Changes

The issue requests ordered per-pattern or per-type session maintenance retention rules so ephemeral cron/subagent sessions can be pruned more aggressively than interactive sessions.

## Best Possible Solution

Design one additive core session retention-policy contract that keeps existing global defaults, defines safe matcher semantics, applies first-match/fallback behavior centrally, and covers schema, help, docs, cleanup commands, and pruning tests together.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection gives high confidence that current main cannot express ordered per-pattern retention rules: the schema is strict, the resolver carries one pruneAfterMs, and pruning uses one cutoff.

## Solution Assessment

Unclear until maintainers choose the public retention contract. The requested direction is plausible, but the safest solution should centralize rule evaluation and avoid ad hoc per-path checks or raw key-pattern footguns without documented precedence and protections.
