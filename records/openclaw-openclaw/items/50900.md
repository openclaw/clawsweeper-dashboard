## Summary

Keep open: current main and the latest release still expose only global session maintenance age/count/disk-budget settings, not ordered per-pattern or per-type retention rules. The requested behavior is a real session-state feature gap, but it needs maintainer product/API approval because it adds destructive public config semantics.

## What This Changes

Request to add ordered per-pattern or per-type session maintenance retention rules so ephemeral sessions can age out faster than interactive sessions.

## Best Possible Solution

Add an additive core retention-rule contract only after maintainer approval, using a typed matcher and first-match age semantics with global pruneAfter fallback plus schema, help, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level: current main exposes only global session.maintenance fields and cleanup functions, not ordered per-pattern maintenance rules. This is a feature-gap reproduction rather than a failing bug repro.

## Solution Assessment

Unclear until maintainer design approval. The requested direction is plausible, but the best implementation should choose a stable typed matcher and safety semantics rather than committing to raw globs by default.
