## Summary

Keep open. Current main still exposes global session maintenance age/count/disk settings plus a separate cron-run retention reaper, but it does not implement ordered per-pattern or per-type retention rules for `session.maintenance`; KeepMyClaw is useful backup context, and #50755/#58088 are adjacent, but neither replaces this core retention policy.

## What This Changes

Add ordered per-pattern or per-type `session.maintenance` retention rules so ephemeral cron/subagent sessions can expire sooner than interactive or topic sessions.

## Best Possible Solution

Add an additive core `session.maintenance.rules` contract after maintainer design approval, with a documented matcher, ordered first-match retention, global `pruneAfter` fallback, strict schema/help/generated-doc coverage, and pruning paths that preserve existing active/running-session and topic/thread safety constraints.

## Reproduction Assessment

Not applicable as a runtime bug, but the missing feature is high-confidence by inspection: the strict config schema has no `rules` field, and the pruning resolver/call sites use one global `pruneAfterMs` threshold.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but the safest fix should first settle the public matcher shape and data-deletion semantics rather than adding ad hoc string matching inside the pruning loop.
