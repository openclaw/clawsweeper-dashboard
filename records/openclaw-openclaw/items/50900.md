## Summary

Keep open: current main and v2026.6.10 still expose global session-maintenance retention plus a narrow fixed model-run cleanup, not ordered per-pattern or per-type retention rules. The KeepMyClaw comment is useful backup/workaround context, but it does not replace the requested core retention-policy decision.

## What This Changes

Request adding ordered per-pattern or per-type `session.maintenance` retention rules so ephemeral cron/subagent sessions can use shorter retention than interactive sessions.

## Best Possible Solution

Keep this as the canonical maintainer product/API decision, then implement one additive typed retention contract with schema/help/docs, dry-run visibility, safety semantics, and focused regression coverage once the matcher shape is approved.

## Reproduction Assessment

Yes, at source level: current main schema, resolver, cleanup, and docs expose one global maintenance threshold and no ordered retention-rule contract. This is a feature-gap reproduction rather than a failing bug run.

## Solution Assessment

Unclear until maintainers approve the API shape. Ordered retention is plausible, but the best fix should choose a stable typed matcher and destructive-cleanup safety semantics rather than defaulting to raw key globs without design approval.
