## Summary

Keep open. Current main still lacks ordered per-pattern or per-type retention rules for session maintenance; it has global age/count/disk controls plus a cron-run-specific reaper, which does not cover the requested general retention policy.

## What This Changes

Add ordered per-pattern or per-type session maintenance retention rules so ephemeral sessions can be pruned sooner while interactive sessions keep the global retention window.

## Best Possible Solution

Add an additive core session.maintenance.rules contract after maintainer design approval, with ordered first-match retention, a documented matcher, global pruneAfter fallback, strict schema/help/generated-doc coverage, and shared pruning logic that preserves active/running and long-lived channel safety constraints across all maintenance paths.

## Reproduction Assessment

Not applicable as a runtime bug, but the missing feature is high-confidence by inspection: current schema rejects rules, the resolver exposes one pruneAfterMs, and pruning applies one cutoff to all non-preserved entries.

## Solution Assessment

Unclear until maintainer design settles the public matcher and deletion semantics. The requested direction is plausible, but the safer path is a documented additive contract rather than ad hoc string matching inside one pruning loop.
