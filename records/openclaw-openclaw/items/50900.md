## Summary

Keep open: current main still lacks ordered per-pattern or per-type `session.maintenance.rules`, and the adjacent per-agent, per-label, preserve-key, and ephemeral-session work does not replace this canonical retention-policy request.

## What This Changes

The issue requests an additive `session.maintenance.rules` config surface that applies ordered first-match retention thresholds by session-key pattern or type, falling back to global `pruneAfter`.

## Best Possible Solution

Define one additive core `session.maintenance.rules` contract with documented matcher semantics, first-match fallback to global `pruneAfter`, schema/help/docs coverage, and centralized pruning that preserves existing durable-session protections.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection shows current main exposes only global session maintenance thresholds plus separate cron-run retention, with no ordered rules field to exercise.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but the maintainable fix should define matcher and deletion semantics centrally rather than add ad hoc checks to individual pruning loops.
