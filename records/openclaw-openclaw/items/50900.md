## Summary

Keep open: current main and latest release still expose global session-maintenance retention plus narrow special cases, not ordered per-pattern or per-type retention rules. The request is valid but needs maintainer product/API approval because it adds destructive public config semantics.

## What This Changes

Add ordered per-pattern or per-type session-maintenance retention rules so ephemeral session keys can prune sooner than interactive sessions.

## Best Possible Solution

Keep this as the canonical product/API decision, then implement one additive typed retention-rule contract with schema, docs, cleanup-preview output, and regression coverage once matcher and safety semantics are approved.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 expose only global session.maintenance fields plus narrow fixed cleanup paths, not ordered per-pattern retention rules. This is a feature-gap reproduction rather than a failing runtime bug repro.

## Solution Assessment

Unclear until maintainer design approval. Ordered retention rules are plausible, but the best solution should use a stable typed matcher and explicit destructive-cleanup safety semantics rather than defaulting to raw key globs.
