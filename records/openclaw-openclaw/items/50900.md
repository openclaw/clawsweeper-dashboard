## Summary

Keep open: current main and v2026.6.8 still expose global session-maintenance age/count controls plus a separate cron-run reaper, so ordered per-pattern or per-type retention remains missing. The request is valid core session-state/config work, but it needs maintainer product/API design because it would define destructive cleanup semantics in public config.

## What This Changes

Request support for ordered per-pattern or per-type session maintenance retention rules so ephemeral cron/subagent sessions can prune faster than interactive sessions.

## Best Possible Solution

Design one additive core session-retention contract for synthetic and interactive session classes, then update schema, runtime cleanup, CLI dry-run output, docs, and regression tests around that contract.

## Reproduction Assessment

Not applicable as a bug reproduction, but the gap is source-proven. Current main and v2026.6.8 cannot express ordered per-pattern retention rules in session.maintenance.

## Solution Assessment

Unclear until maintainers choose the retention contract. Ordered rules are plausible, but the best fix should settle matcher syntax and destructive-cleanup precedence rather than add a one-off knob.
