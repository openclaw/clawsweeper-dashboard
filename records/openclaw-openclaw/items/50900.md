## Summary

Keep open: current main and the latest release still expose only global session-maintenance retention plus a separate cron-run reaper; ordered per-pattern or per-type retention remains a real session-state/config design request, not an implemented fix or a stale duplicate.

## What This Changes

Request to add ordered per-pattern or per-type session maintenance retention rules so ephemeral cron/subagent sessions can be pruned sooner than interactive sessions.

## Best Possible Solution

Add an additive core session.maintenance.rules contract only after maintainer approval, with typed first-match retention semantics, global pruneAfter fallback, strict schema/help/docs, cleanup preview visibility, and tests preserving active/running and durable conversation sessions.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature request. Source inspection gives high confidence that current main only supports global session.maintenance retention plus cron-specific retention, not ordered per-pattern rules.

## Solution Assessment

Unclear until maintainers choose the public retention-rule contract. The requested direction is plausible, but the best fix should use a stable typed matcher and safety semantics rather than committing to raw globs by default.
