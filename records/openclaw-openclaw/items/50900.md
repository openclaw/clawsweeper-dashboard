## Summary

Keep open: current main still has global session maintenance retention plus a separate cron-run reaper, but not ordered per-pattern or per-type session-maintenance rules. The remaining work is a real session-state/config design item, not a stale duplicate or an already-implemented fix.

## What This Changes

The issue requests ordered per-pattern or per-type retention rules under `session.maintenance` so ephemeral sessions can be pruned faster than durable interactive sessions.

## Best Possible Solution

Add an additive core `session.maintenance.rules` contract only after maintainer approval, with a typed matcher, first-match retention semantics, global `pruneAfter` fallback, strict schema/docs coverage, cleanup preview visibility, and tests preserving active/running and durable conversation sessions.

## Reproduction Assessment

Not applicable as a bug repro in the strict sense; source inspection does give a high-confidence check that current main only supports global `session.maintenance.pruneAfter` plus cron-specific retention, not ordered per-pattern rules.

## Solution Assessment

Unclear until maintainer design approval. The requested direction is plausible, but the best implementation should choose a stable typed matcher and safety rules rather than committing to raw globs by default.
