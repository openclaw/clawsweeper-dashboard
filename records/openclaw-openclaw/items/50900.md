## Summary

Keep open: current main still lacks ordered per-pattern or per-type session maintenance retention rules, even though newer cron-specific retention and protected external-session cleanup reduce part of the pressure described in the report.

## What This Changes

Add ordered per-pattern or per-type session maintenance retention rules so ephemeral cron/subagent entries can expire faster than interactive sessions.

## Best Possible Solution

After maintainer approval, add an additive retention contract that composes with `cron.sessionRetention`, preserves protected external sessions, and updates config schema/help/docs plus focused pruning tests.

## Reproduction Assessment

Not applicable: this is a feature/config request, not a broken existing-behavior report. Source inspection on current main shows the accepted `session.maintenance` fields and resolver still produce a single global `pruneAfterMs`, with no `rules` or `match` array.

## Solution Assessment

Unclear. Ordered retention rules are plausible, but current main now has `cron.sessionRetention` for cron runs and protected external sessions, so maintainers should decide whether the remaining need belongs in generic `session.maintenance.rules` or narrower per-type knobs.
