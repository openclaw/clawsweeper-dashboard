## Summary

Keep open: current main and the latest release still support only global session maintenance retention plus a separate cron-run reaper, not ordered per-pattern or per-type retention rules. This remains a real session-state/config design item rather than an already-implemented fix or a clean duplicate.

## What This Changes

The issue requests ordered per-pattern or per-type session maintenance retention rules so ephemeral cron/subagent sessions can be pruned more aggressively than interactive sessions.

## Best Possible Solution

Add an additive core retention-rule contract only after maintainer approval, using a typed matcher and first-match age semantics with global `pruneAfter` fallback, strict schema/help/docs coverage, and tests for protected conversation and synthetic sessions.

## Reproduction Assessment

Yes, at source level: current main exposes only global `session.maintenance` fields and a separate cron-specific retention path, not ordered per-pattern maintenance rules. This is a feature-gap reproduction rather than a failing bug repro.

## Solution Assessment

Unclear until maintainer design approval. The requested direction is plausible, but the best implementation should choose a stable typed matcher and safety semantics rather than committing to raw globs by default.
