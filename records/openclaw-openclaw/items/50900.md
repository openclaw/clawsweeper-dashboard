## Summary

Keep open: current main still lacks ordered per-pattern or per-type `session.maintenance.rules`, and the related per-agent, per-label, preserve-key, and ephemeral-session work is adjacent rather than a replacement for this canonical retention-policy request.

## What This Changes

Request adding ordered per-pattern or per-type session maintenance retention rules that override the global `pruneAfter` for matching session keys.

## Best Possible Solution

Define an additive core `session.maintenance.rules` contract with documented matcher semantics, first-match fallback to global `pruneAfter`, schema/help/docs coverage, and centralized pruning behavior that preserves existing durable-session protections.

## Reproduction Assessment

Not applicable as a runtime bug; this is a new retention-policy feature. Source inspection shows current main exposes only global `session.maintenance` age/count settings plus separate cron run retention, with no ordered rules field.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but the maintainable fix should define matcher and deletion semantics centrally rather than adding ad hoc pattern checks in individual pruning loops.
