## Summary

Keep open: the underlying memory metadata bug is real, but this PR is not merge-ready because it publishes current metadata over unverified existing chunks and can make stale or out-of-scope memory searchable before a rebuild or cleanup runs.

## What This Changes

The PR adds a `writeMeta` branch in memory-core sync when identity metadata is missing but chunks exist, and adds a standalone proof script under `scripts/`.

## Best Possible Solution

Treat missing metadata with existing chunks as a rebuild-or-verify path: run safe/full reindex or synchronous cleanup before writing metadata, add manager-level stale-row regression coverage, and leave the safe-reindex race to the separate related PR.

## Reproduction Assessment

Yes. A focused manager test can create indexed chunks, delete the source file and metadata row, call sync on the PR branch, and then search; source inspection shows the PR writes metadata and returns before stale cleanup or reindex runs.

## Solution Assessment

No. The proposed metadata stamp is a plausible dead-loop breaker, but the safer maintainable fix is to rebuild or verify the index before publishing identity as valid.
