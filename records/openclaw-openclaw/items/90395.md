## Summary

Keep this PR open, but it is not merge-ready: it targets a real missing-metadata loop, but the patch stamps current identity metadata onto existing chunks without proving those chunks still match the current provider, scope, files, and tokenizer.

## What This Changes

The PR adds a metadata write in memory-core sync's invalid-identity early-return path when meta is absent but indexed chunks already exist.

## Best Possible Solution

Recover missing metadata by rebuilding or switching to a verified index state, with regression coverage that stale rows stay hidden until rebuilt and live proof that `memory_search` recovers.

## Reproduction Assessment

Yes at source level: delete `memory_index_meta_v1` after a forced index so chunks remain, then a non-CLI sync reaches the invalid-identity early return on current main. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. Writing metadata alone is too trusting because the missing row is the only identity proof for the existing chunks; the safer fix is a verified rebuild or index switch before search becomes valid again.
