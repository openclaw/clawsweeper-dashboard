- `P2`: This is a normal-priority memory search reliability fix with limited blast radius, but the current patch is not safe to merge without identity-recovery review.
- `merge-risk: 🚨 session-state`: The patch changes when persisted memory chunks become searchable and can alter agent recall/session-memory behavior after upgrades or corrupted metadata.
- `merge-risk: 🚨 security-boundary`: Stamping current metadata onto unverified rows can expose stale or out-of-scope memory content that the identity gate currently keeps paused.

## Summary

Keep open: the PR targets a real memory-index dead-loop, but the current patch is unsafe because it makes existing chunks trusted by writing fresh identity metadata without proving those rows match the current provider, scope, sources, chunking, tokenizer, and vector dimensions.

## What This Changes

The PR adds a metadata write in `runSync()` when memory index identity is missing but existing chunks prevent the current path from doing a full reindex.

## Best Possible Solution

Use a recovery path that preserves the identity safety contract: keep missing metadata paused until explicit repair, or rebuild/verify the index before marking metadata valid.

## Reproduction Assessment

Yes. Source inspection shows current main returns dirty without writing meta when identity is missing and chunks already exist, while search/status treat that missing identity as paused; I did not run a live gateway reproduction.

## Solution Assessment

No. The PR is only a plausible symptom fix; the safer maintainable solution is a verified rebuild or explicit repair path before any metadata marks existing rows valid.
