## Summary

Keep open: the PR addresses the missing-metadata dead loop with the right general direction, but the current branch can still force a full FTS-only rebuild when metadata is missing and the configured embedding provider is unavailable, which can discard an existing semantic index instead of leaving it paused or dirty. The PR also still lacks after-fix CLI or gateway `memory_search` proof for the latest full-reindex recovery behavior.

## What This Changes

The PR changes memory-core sync recovery so missing index metadata with existing chunks triggers a verified full reindex, and adds FTS-only regression coverage for missing-meta recovery and stale chunk pruning.

## Best Possible Solution

Land a recovery path that only full-reindexes missing-meta stores when it can rebuild the current provider identity or prove existing rows are FTS-only; otherwise leave semantic memory dirty/paused with focused regression coverage and real CLI or gateway proof.

## Reproduction Assessment

Yes from source for the blocking review finding: sync initializes the provider, the new missing-meta branch forces full reindex whenever chunks exist, and the no-provider index path writes FTS-only rows. I did not run a live reproduction because this review was required to keep the checkout read-only.

## Solution Assessment

No: a verified full reindex is the right shape only when the current provider identity can be rebuilt or existing rows are known FTS-only. The safer fix is to preserve or pause semantic indexes during provider/auth unavailability and add coverage for missing metadata with existing semantic chunks.
