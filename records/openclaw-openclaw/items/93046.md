## Summary

Keep open: the user-facing delay is still real on current main, but this patch is not merge-ready because it excludes every active background task from channel reload deferral rather than only tasks proven safe to ignore, and the after-fix proof is still test-only.

## What This Changes

The PR changes gateway channel reload deferral to ignore active background task blockers while still waiting on queue size, pending replies, and embedded runs.

## Best Possible Solution

Narrow the reload wait change to tasks that are explicitly non-delivering or otherwise safe across channel restart, keep delivery-capable tasks in the deferral boundary, and add focused regression plus redacted after-fix runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main's reload wait counts active background task blockers, and the PR body includes before-fix logs showing a channel reload held for the full timeout by a context-engine maintenance task. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. Excluding all active background tasks is too broad because some tasks can still deliver through the same channel path; the maintainable fix is an explicit safe-to-ignore task boundary or a narrow filter for silent maintenance tasks.
