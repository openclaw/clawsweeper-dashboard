## Summary

Keep open: the branch addresses a real reload-delay complaint, but the patch removes the whole active-background-task class from channel reload waiting even though some running tasks can still deliver requester-origin updates through channels.

## What This Changes

The PR changes gateway channel reload deferral so reload waits ignore active background task counts while still waiting on queued channel work, pending replies, and embedded runs.

## Best Possible Solution

Keep delivery-capable tasks in channel-reload deferral, and ignore only explicitly non-delivering maintenance tasks through a named task-level contract or narrow filter with regression coverage and redacted runtime logs.

## Reproduction Assessment

Yes, source-reproducible: current main's channel reload wait counts active background task blockers, and the PR body includes before-fix logs showing a reload held for the full timeout by context-engine maintenance. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. Excluding all active background tasks is too broad because some tasks can still deliver through the channel path; the safer fix is a reload-safe task contract or narrow filter for silent maintenance tasks.
