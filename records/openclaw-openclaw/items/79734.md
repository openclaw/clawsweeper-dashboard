## Summary

Keep open: current main still lacks a doctor dry-run mode and this PR addresses the linked feature request, but the branch is not merge-ready because the preview can leak raw config values, repair-mode side effects can still run, and the supplied proof does not show the patched dry-run/no-write behavior.

## What This Changes

The PR adds a `--dry-run` option for `openclaw doctor --fix`, threads it through doctor config flow, skips selected write paths, emits a config diff, updates docs, and adds finalize-flow tests.

## Best Possible Solution

Implement doctor dry-run as an end-to-end read-only planning mode: use redacted original-vs-candidate output, disable repair approvals and side effects, keep an immutable baseline for diffing, and prove the patched command with redacted no-write output.

## Reproduction Assessment

Yes. Source inspection of the PR head and current helper contracts shows the remaining failures: raw diff serialization, active repair-mode prompter behavior, and `shouldRepair` advancing `cfg` before final diffing.

## Solution Assessment

No. The direction is useful, but this implementation is not the safest solution until dry-run is read-only across the full doctor flow and emits redacted diffs from an immutable original config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Redact dry-run diff values before printing:** `src/commands/doctor/finalize-config-flow.ts:25-29`
  - body: The dry-run output stringifies arbitrary old and new config values before calling `note()`. Paths such as gateway auth tokens, channel credentials, provider API keys, or SecretRef ids can leak into terminal logs; build the diff from redacted config objects or mask sensitive paths before rendering values.
  - confidence: 0.92
- **[P1] Make dry-run disable repair approvals end to end:** `src/flows/doctor-health.ts:11-13`
  - body: This bypasses the Nix write assertion, but `createDoctorPrompter({ options })` still sees `--fix --dry-run` as repair mode. Health contributions that call `confirmAutoFix()`, `confirmRuntimeRepair()`, or check `prompter.shouldRepair` can still rewrite auth/state/runtime files before the final config write guard; force an effective read-only prompter when dry-run is set.
  - confidence: 0.9
- **[P2] Keep dry-run diffs anchored to the original config:** `src/commands/doctor-config-flow.ts:70-72`
  - body: With `--fix --dry-run`, `shouldRepair` remains true, so existing config mutation helpers can advance `cfg` to each proposed mutation before finalization. The final dry-run diff can then compare an already-mutated config with `candidate` and miss changes that `--fix` would apply; keep the original config as the baseline or make dry-run use non-mutating repair planning.
  - confidence: 0.88
- **[P3] Run the dry-run branch even when nothing changed:** `src/commands/doctor/finalize-config-flow.ts:13`
  - body: The dry-run branch is gated by `pendingChanges`, so a clean `openclaw doctor --dry-run` falls through without the documented `No config changes detected.` note. The new no-changes test still passes `pendingChanges: true`, so it does not cover the clean-command path.
  - confidence: 0.82
