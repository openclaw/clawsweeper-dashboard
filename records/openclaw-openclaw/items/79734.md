## Summary

Keep open: this is a useful external PR for a valid CLI safety feature, but the current head still has blocking dry-run correctness/security gaps and insufficient real behavior proof.

## What This Changes

The PR adds a `doctor --dry-run` CLI flag, config diff rendering, write guards, docs, and finalize-flow tests for previewing doctor repair changes.

## Best Possible Solution

Implement dry-run as an end-to-end read-only planning mode that redacts diff values, compares against an immutable original config, disables all repair side effects, and is proven with real no-write output.

## Reproduction Assessment

Yes. Source inspection of the PR head gives a high-confidence path: `openclaw doctor --fix --dry-run` keeps repair mode active, can compare mutated config against itself, and emits raw serialized config values.

## Solution Assessment

No. Guarding the final config write and one repair sequence is too narrow; the safer solution is a whole-doctor dry-run contract across diff redaction, config mutation baseline, prompter behavior, and health contributions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Redact dry-run diff values before printing:** `src/commands/doctor/finalize-config-flow.ts:25-29`
  - body: The dry-run diff prints serialized old/new values for arbitrary config paths, so migrations involving `gateway.auth.token`, provider `apiKey`, channel tokens, passwords, or SecretRef IDs can leak secrets to terminal output. Build the diff from redacted config objects or mask sensitive paths before calling `note()`.
  - confidence: 0.93
- **[P1] Force dry runs out of repair mode:** `src/flows/doctor-health.ts:13`
  - body: The dry-run exception only skips the Nix write assertion; `createDoctorPrompter()` still receives `--fix` options, so `confirmAutoFix()`, `confirmRuntimeRepair()`, and `prompter.shouldRepair` stay active. That lets later health contributions rewrite auth stores, migrate state, rebuild plugin/sandbox registries, or run repair scripts before the final config writer is skipped.
  - confidence: 0.91
- **[P2] Compare dry runs against an immutable baseline:** `src/commands/doctor-config-flow.ts:71-72`
  - body: With `--fix --dry-run`, `shouldRepair` is still passed into the mutation helpers, and those helpers advance `cfg` to the mutated config when repair mode is true. The finalizer can then compare `cfg` to `candidate` and report no diff for the exact normalization, auto-enable, or unknown-key changes the user asked to preview.
  - confidence: 0.9
- **[P3] Report clean dry runs:** `src/commands/doctor/finalize-config-flow.ts:13`
  - body: The dry-run branch is gated by `pendingChanges`, so a clean `openclaw doctor --dry-run` falls through without the documented `No config changes detected.` note. The added no-changes test misses this because it sets `pendingChanges: true`.
  - confidence: 0.82
