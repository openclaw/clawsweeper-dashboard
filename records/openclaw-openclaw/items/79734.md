## Summary

Keep open: this is a useful implementation for the linked doctor dry-run feature, but the current PR still has merge-blocking safety issues. The branch can print raw config values, keeps `--fix` repair mode active during dry-run health contributions, and can report no diff because it compares against already-mutated config state.

## What This Changes

The PR adds a `doctor --dry-run` CLI flag, threads it through doctor config finalization, adds diff output and write guards, updates doctor docs, and adds finalize-flow tests.

## Best Possible Solution

Implement doctor dry-run as a whole-doctor read-only planning mode that redacts sensitive diff values, compares against an immutable original config baseline, disables all repair side effects, and is proven with real no-write output.

## Reproduction Assessment

Yes for the PR blockers: source inspection of the PR head shows raw diff serialization, `--fix` repair mode still reaching health contributions, and dry-run diffing against mutable repair state. I did not run the command because this review is read-only and the affected paths can write outside the repository.

## Solution Assessment

No. The feature direction is good, but this implementation is not the best merge path until dry-run is enforced as a redacted, original-baseline, whole-doctor no-side-effect mode with real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Redact dry-run diff values before printing:** `src/commands/doctor/finalize-config-flow.ts:25-29`
  - body: The dry-run output stringifies raw old/new values for every config key path before calling `note()`. If a repair touches `gateway.auth.token`, provider API keys, channel tokens, or SecretRef ids, the terminal output can expose credentials; build the diff from redacted config objects or mask sensitive paths before rendering.
  - confidence: 0.93
- **[P1] Force dry runs out of repair mode:** `src/flows/doctor-health.ts:13`
  - body: This guard only skips the Nix/write assertion, but `createDoctorPrompter({ options })` still sees `--fix` and `resolveDoctorRepairMode()` still sets `shouldRepair`. Later health contributions can auto-approve repairs that rewrite auth stores, migrate state, install/runtime-repair assets, or otherwise mutate files before the final config writer is skipped; dry-run needs read-only prompter/repair state.
  - confidence: 0.91
- **[P2] Preserve a baseline for dry-run diffs:** `src/commands/doctor-config-flow.ts:72`
  - body: With `--fix --dry-run`, `shouldRepair` remains true, so config mutation helpers can replace `cfg` with the proposed config before finalization. The finalizer then compares `cfg` to `candidate` and may print `No config changes detected.` for exactly the changes `--fix` would apply; keep an immutable original config or run preview mutations without advancing `cfg`.
  - confidence: 0.89
- **[P3] Report clean dry runs:** `src/commands/doctor/finalize-config-flow.ts:13`
  - body: The dry-run branch is gated on `pendingChanges`, so a clean `openclaw doctor --dry-run` falls through silently instead of emitting the documented `No config changes detected.` note. The added test misses this path because it sets `pendingChanges: true` despite its title saying there are no pending changes.
  - confidence: 0.82
