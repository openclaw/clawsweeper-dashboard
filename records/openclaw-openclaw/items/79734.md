## Summary

Keep open: this is a useful doctor safety feature, but the current patch still has blocking correctness and security issues, and the supplied proof does not show the patched dry-run behavior or no-write result.

## What This Changes

The PR adds a doctor `--dry-run` CLI flag, config diff output, docs, option plumbing, a write-config guard, and finalize-flow tests for previewing doctor config repairs.

## Best Possible Solution

Implement dry-run as a read-only planning mode that disables repair-mode side effects, builds a redacted diff from the original config to the proposed config, and proves the command leaves config and state unchanged.

## Reproduction Assessment

Yes, by source inspection. `openclaw doctor --fix --dry-run` leaves the repair prompter active and compares a potentially repair-mutated `cfg` to `candidate`, and the new diff stringifies raw config values.

## Solution Assessment

No. Suppressing the final config write is too narrow; the maintainable solution is a redacted, read-only planning contract across config flow and doctor health contributions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Redact dry-run diff values before printing:** `src/commands/doctor/finalize-config-flow.ts:28-32`
  - body: The new diff stringifies old/new config values directly, so repairs involving `gateway.auth.token`, channel tokens, or provider `apiKey` fields can print secret material. Build the diff from redacted config values or mask sensitive paths before calling `note()`.
  - confidence: 0.9
- **[P1] Disable repair-mode prompter during dry runs:** `src/flows/doctor-health.ts:13`
  - body: With `--fix --dry-run`, this skips the write-mode assertion but still creates the doctor prompter from repair options, so health contributions can auto-approve repairs that copy auth stores, migrate legacy state, rebuild plugin registry state, or migrate sandbox registry files before the final write guard runs.
  - confidence: 0.91
- **[P2] Compare dry runs against an immutable baseline:** `src/commands/doctor/finalize-config-flow.ts:18`
  - body: For `--fix --dry-run`, `shouldRepair` remains true, so existing mutation helpers can copy proposed changes into `cfg`; this branch then flattens `params.cfg` and `params.candidate` and reports no diff for changes that `--fix` would write.
  - confidence: 0.88
- **[P3] Report clean dry runs:** `src/commands/doctor/finalize-config-flow.ts:16`
  - body: The dry-run note is only emitted when `pendingChanges` is true, so a clean `openclaw doctor --dry-run` returns silently even though the PR says it reports `No config changes detected.` The added test sets `pendingChanges: true`, so it misses the real clean path.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff can expose raw credentials, and `--fix --dry-run` can still reach repair-mode side effects outside the final config writer.

Concerns:

- **[high] Dry-run diff can leak config secrets:** `src/commands/doctor/finalize-config-flow.ts:28`
  - body: The new dry-run output prints `JSON.stringify(oldVal)` and `JSON.stringify(newVal)` for arbitrary config paths, including token and apiKey paths that current config redaction helpers are designed to mask.
  - confidence: 0.9
- **[medium] Dry-run still reaches repair-mode state mutations:** `src/flows/doctor-health.ts:13`
  - body: Dry-run skips the final config write guard but leaves repair-mode confirmation active for earlier health contributions that can rewrite auth profile files, migrate state stores, or refresh plugin/sandbox registry state.
