Full review comments:

- **[P1] Keep dry-run out of repair mode globally:** `src/flows/doctor-health.ts:13`
  - body: `--fix --dry-run` still passes `repair: true` into `createDoctorPrompter`, so `prompter.shouldRepair` remains true. Doctor then runs UI repair before config flow and later session/audit/sandbox/gateway/startup health contributions that key off `shouldRepair`, so the new dry-run can still build assets, rewrite state, or run maintenance instead of staying preview-only.
  - confidence: 0.93
- **[P1] Do not enable config recovery during dry-run:** `src/commands/doctor-config-flow.ts:72`
  - body: Setting `repairPrefixedConfig` when `dryRun` is true makes the preflight recovery path eligible during a preview. That recovery path persists recovered config files and clobber snapshots before `finalizeDoctorConfigFlow` can return `shouldWriteConfig: false`, so `openclaw doctor --dry-run` is not guaranteed read-only.
  - confidence: 0.9
- **[P1] Redact dry-run diff values before printing:** `src/commands/doctor/finalize-config-flow.ts:25-29`
  - body: The diff formatter prints serialized old and new config values directly. Current config supports sensitive paths such as tokens, passwords, API keys, private keys, and service-account material, so a dry-run meant to make repairs safer can leak credentials into terminal logs or copied PR proof unless the diff is built from redacted/path-aware values.
  - confidence: 0.94
- **[P3] Show the clean dry-run result when nothing is pending:** `src/commands/doctor/finalize-config-flow.ts:13`
  - body: The no-change message only runs inside `if (params.dryRun && params.pendingChanges)`, so a clean config with `pendingChanges === false` returns silently instead of printing the documented `No config changes detected.` result. The new test sets `pendingChanges: true`, so it does not cover the real clean dry-run path.
  - confidence: 0.84
