## Summary

Keep open: the feature is still useful, but the current PR head has blocking dry-run safety issues. Source inspection shows raw config values can be printed, `--fix --dry-run` still leaves repair-mode approvals active, and the diff can compare against an already-mutated config instead of the original baseline.

## What This Changes

The PR adds a `doctor --dry-run` CLI flag, threads it through the doctor config flow, skips selected writes in dry-run mode, documents the behavior, and adds finalize-flow tests.

## Best Possible Solution

Implement doctor dry-run as a whole-command read-only planning mode: use a redacted original-vs-candidate diff, force an effective non-repair prompter for dry-run, block every doctor side effect, and prove the patched command with redacted no-write output.

## Reproduction Assessment

Yes. Source inspection of the PR head and current helper contracts shows the remaining failures: raw diff serialization, repair-mode prompter behavior, and `shouldRepair` advancing `cfg` before final diffing.

## Solution Assessment

No. The direction is plausible, but this implementation is not the safest solution until dry-run is read-only across the whole doctor flow and uses redacted original-vs-candidate output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Redact dry-run diff values before printing:** `src/commands/doctor/finalize-config-flow.ts:25-29`
  - body: The dry-run diff stringifies arbitrary config values before emitting them with `note()`. Paths such as gateway tokens, provider API keys, channel tokens, or SecretRef ids can leak into terminal logs; build the diff from redacted config objects or mask sensitive paths before rendering.
  - confidence: 0.93
- **[P1] Force dry-run out of repair mode:** `src/flows/doctor-health.ts:13`
  - body: This bypasses the Nix write assertion, but `createDoctorPrompter({ options })` still sees `repair` for `--fix --dry-run`. Health contributions can auto-approve repairs like auth-profile rewrites before the final write guard, so dry-run needs an effective read-only repair mode end to end.
  - confidence: 0.9
- **[P2] Diff dry-runs from the original config:** `src/commands/doctor-config-flow.ts:70-72`
  - body: Because `shouldRepair` remains true for `--fix --dry-run`, `applyDoctorConfigMutation()` advances `cfg` to each proposed mutation before finalization. The finalizer can then compare an already-mutated `cfg` with `candidate` and report no changes; keep an immutable baseline or make dry-run use non-mutating repair planning.
  - confidence: 0.88
- **[P3] Enter dry-run mode even without pending changes:** `src/commands/doctor/finalize-config-flow.ts:13`
  - body: The dry-run branch is gated by `pendingChanges`, so a clean `openclaw doctor --dry-run` skips the documented `No config changes detected.` note. The added test does not catch this because it still passes `pendingChanges: true` in the no-changes case.
  - confidence: 0.82
