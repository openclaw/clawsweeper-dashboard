## Summary

Keep open: this is a valid doctor dry-run feature PR, but the current diff still has blocking read-only and secret-redaction gaps, and the supplied proof does not show the changed dry-run behavior in a real no-write run.

## What This Changes

The PR adds a `doctor --dry-run` CLI flag, threads it through doctor config flow, renders flat config diffs, skips some write paths, updates docs, and adds finalize-flow tests.

## Best Possible Solution

Implement doctor dry-run as a whole-command read-only planning mode that uses redacted diffs, compares against an immutable original config, disables all repair side effects, documents the exact semantics, and includes real no-write proof.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head shows raw diff value printing, repair-mode prompter construction during dry-run, and a mutable baseline path that can hide proposed changes. The user-facing feature itself is not a bug reproduction; current main simply lacks the requested flag.

## Solution Assessment

No. The feature direction is reasonable, but this implementation is not yet the safest maintainable solution because dry-run must be a redacted, end-to-end read-only planning mode rather than a final-write guard plus partial repair skipping.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Redact dry-run diff values before printing:** `src/commands/doctor/finalize-config-flow.ts:25-29`
  - body: The diff prints serialized old and new values for every config path. If a repair touches `gateway.auth.token`, provider API keys, channel tokens, or SecretRef-like values, the dry-run output can leak secrets to terminal logs; build the diff from redacted config objects or mask sensitive paths before calling `note()`.
  - confidence: 0.93
- **[P1] Force dry runs out of repair mode:** `src/flows/doctor-health.ts:21`
  - body: `--fix --dry-run` still passes repair options into `createDoctorPrompter()`, so `confirmAutoFix()` and `confirmRuntimeRepair()` can auto-approve health-contribution repairs before the final config-write guard runs. Build a read-only prompter or make repair-mode resolution return false whenever `dryRun` is true.
  - confidence: 0.91
- **[P2] Compare dry runs against the original config:** `src/commands/doctor-config-flow.ts:72`
  - body: Because `shouldRepair` stays true for `--fix --dry-run`, existing config mutation helpers advance `cfg` to each proposed mutation before finalization. The finalizer can then compare `cfg` to `candidate` and report no diff for changes that `--fix` would apply; keep an immutable baseline or run preview mutations without advancing `cfg`.
  - confidence: 0.89
- **[P3] Report clean dry runs:** `src/commands/doctor/finalize-config-flow.ts:13`
  - body: The dry-run branch only runs when `pendingChanges` is true, so a clean `openclaw doctor --dry-run` falls through without the documented `No config changes detected.` note. Exercise this with `pendingChanges: false` and make the dry-run branch handle both clean and dirty runs.
  - confidence: 0.82
