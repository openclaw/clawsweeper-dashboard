## Summary

Keep open: current main still lacks a doctor dry-run mode, but this PR is not merge-ready because the proposed preview can print raw config secrets, does not disable all repair-mode side effects, and lacks real after-fix proof of the new dry-run command.

## What This Changes

The PR adds a `doctor --dry-run` CLI flag, threads it through doctor config flow, prints a proposed config diff, skips selected write paths, updates doctor docs, and adds finalize-flow tests.

## Best Possible Solution

Implement doctor dry-run as an end-to-end read-only planning mode with redacted original-vs-candidate output, dry-run-aware repair/health plumbing, write-site backstops, focused regression tests, and real no-write proof.

## Reproduction Assessment

Yes. Source inspection of the PR diff and current helper contracts shows raw diff serialization, active repair-mode health paths, and repair-mode mutation of `cfg` before final diffing; I did not execute the patch because this review is read-only.

## Solution Assessment

No. The feature direction is plausible, but this implementation is not the safest solution until dry-run is redacted, compares against an immutable baseline, disables all repair side effects, and is proven with real no-write output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Redact dry-run diff values before printing:** `src/commands/doctor/finalize-config-flow.ts:25-29`
  - body: The dry-run diff stringifies arbitrary old and new config values before emitting them with `note()`. Paths such as gateway tokens, channel tokens, provider API keys, SecretRef ids, or secret-looking env values can leak into terminal logs, so build the diff from redacted objects or mask sensitive paths before rendering.
  - confidence: 0.92
- **[P1] Make dry-run disable repair approvals end to end:** `src/flows/doctor-health.ts:11-14`
  - body: This only bypasses the Nix write assertion while leaving `createDoctorPrompter({ options })` in repair mode for `--fix --dry-run`. Health contributions that call `confirmAutoFix()`, `confirmRuntimeRepair()`, or check `prompter.shouldRepair` can still rewrite auth, state, runtime, or plugin files before the final config write guard runs.
  - confidence: 0.9
- **[P2] Compare dry-runs against an immutable baseline:** `src/commands/doctor-config-flow.ts:90-91`
  - body: With `--fix --dry-run`, `shouldRepair` remains true, and the existing mutation helper assigns `cfg` to each proposed mutation. The finalizer can then compare an already-mutated `cfg` with `candidate` and report no diff for changes that `--fix` would apply; keep the original config as the baseline or make dry-run use non-mutating repair planning.
  - confidence: 0.88
- **[P3] Report clean dry-runs instead of falling through:** `src/commands/doctor/finalize-config-flow.ts:13`
  - body: The dry-run branch is gated by `pendingChanges`, so a clean `openclaw doctor --dry-run` skips the documented `No config changes detected.` note. The added no-changes test does not catch this because it still passes `pendingChanges: true`.
  - confidence: 0.82
