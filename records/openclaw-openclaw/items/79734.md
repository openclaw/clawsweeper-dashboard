## Summary

Keep open. The feature is still absent from current main and this PR is a plausible implementation path, but the patch is not mergeable because the dry-run path can still leak secret values, leave repair-mode side effects active, and miss the actual `--fix --dry-run` diff.

## What This Changes

The PR adds a doctor-level `--dry-run` CLI flag, threads `dryRun` through the doctor config flow, prints a flat config diff, skips selected writes in dry-run, updates doctor docs, and adds finalize-flow tests.

## Best Possible Solution

Implement doctor dry-run as a whole-doctor read-only planning mode: redacted diff from an immutable baseline, repair prompts forced off, config/state/plugin/service side effects skipped or guarded at write sites, focused tests, docs, and real no-write proof.

## Reproduction Assessment

Yes. Source inspection of the PR head shows raw diff value emission, a dry-run branch gated by `pendingChanges`, and `--fix --dry-run` keeping repair mode active while health contributions can write before the final config writer guard.

## Solution Assessment

No. The PR adds the right user-facing surface, but guarding only finalization and one repair sequence is too narrow; the maintainable fix needs a read-only contract across prompter behavior, config mutation baselines, output redaction, and health contributions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Force repair prompts off during dry runs:** `src/flows/doctor-health.ts:13`
  - body: `--fix --dry-run` still passes `repair: true` into `createDoctorPrompter()`, and `resolveDoctorRepairMode()` ignores `dryRun`. That means health contributions before the final config writer can auto-approve repairs and mutate auth profiles, legacy state, plugin registry, audit logs, or services. Dry-run should force a read-only prompter or make contributions skip repair work.
  - confidence: 0.91
- **[P1] Redact dry-run diff values before printing:** `src/commands/doctor/finalize-config-flow.ts:25-29`
  - body: The dry-run diff stringifies old and new values for arbitrary config paths before calling `note()`. If a repair touches `gateway.auth.token`, provider `apiKey`, channel tokens, or SecretRef ids, the terminal output can expose credentials. Build the diff from redacted config objects or mask sensitive paths before rendering.
  - confidence: 0.92
- **[P2] Keep a pre-repair baseline for the diff:** `src/commands/doctor-config-flow.ts:226`
  - body: Skipping `runDoctorRepairSequence()` is too late for `--fix --dry-run`: earlier mutation helpers still receive `shouldRepair: true`, so they advance `cfg` to `candidate` before finalization. The final diff can then show no changes for exactly the changes `--fix` would apply. Preserve an immutable baseline or treat dry-run as non-repair while collecting the candidate.
  - confidence: 0.88
- **[P3] Run the dry-run branch even when there are no changes:** `src/commands/doctor/finalize-config-flow.ts:13`
  - body: The dry-run branch is gated by `params.pendingChanges`, so a clean `openclaw doctor --dry-run` falls through without the documented `No config changes detected.` note. The added test does not catch this because its no-change case still sets `pendingChanges: true`.
  - confidence: 0.78
