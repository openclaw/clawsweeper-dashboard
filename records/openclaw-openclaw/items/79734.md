## Summary

Keep open: the feature is useful, but this PR still has blocking dry-run safety gaps. The branch can print raw config values, leaves `--fix --dry-run` in repair mode for health contributions, can miss the config diff it is meant to preview, and the supplied proof does not show the patched dry-run/no-write behavior.

## What This Changes

The PR adds a `doctor --dry-run` option with config diff output, skips selected doctor repair/write paths in dry-run, documents the mode, and adds finalize-flow tests.

## Best Possible Solution

Implement dry-run as an end-to-end read-only planning mode with a redacted diff from an immutable original config baseline, repair side effects disabled across the whole doctor flow, focused regression tests, and live no-write proof.

## Reproduction Assessment

Yes. Source inspection of the PR diff and current doctor plumbing shows `--fix --dry-run` still uses repair-mode mutation/prompter behavior, compares against mutable state, and emits raw serialized values.

## Solution Assessment

No. The direction is good, but final-write guards and a local finalizer diff are too narrow; the maintainable fix needs a read-only dry-run contract across config mutation state, prompter behavior, health contributions, and redacted output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact dry-run diff values before printing:** `src/commands/doctor/finalize-config-flow.ts:23-27`
  - body: The dry-run diff stringifies raw old and new config values for arbitrary paths before calling `note()`. If doctor proposes changes under tokens, API keys, or SecretRef-like fields, the preview can leak credentials into terminal logs; build the diff from redacted config objects or mask sensitive paths first.
  - confidence: 0.92
- **[P1] Disable repair approvals during dry runs:** `src/flows/doctor-health.ts:11-14`
  - body: `--fix --dry-run` still creates the doctor prompter from repair options, so `confirmAutoFix()`, `confirmRuntimeRepair()`, and `prompter.shouldRepair` remain repair-active. Health contributions run before the final write guard and can rewrite auth stores, migrate state, or repair registries during what is documented as a strict preview.
  - confidence: 0.9
- **[P2] Keep an original config baseline for dry-run diffs:** `src/commands/doctor-config-flow.ts:75`
  - body: With `--fix --dry-run`, config mutation helpers still receive `shouldRepair: true`, which advances `cfg` to each proposed candidate before finalization. The finalizer can then compare identical objects and print `No config changes detected.` for changes that `--fix` would apply; preserve an immutable baseline or make dry-run mutations preview-only.
  - confidence: 0.89
- **[P3] Report clean dry runs without pending changes:** `src/commands/doctor/finalize-config-flow.ts:13`
  - body: The dry-run branch is gated on `params.pendingChanges`, so a clean `openclaw doctor --dry-run` falls through without the documented `No config changes detected.` note. The added test does not catch this because it sets `pendingChanges: true` in the no-changes case.
  - confidence: 0.78
