## Summary

Keep open: the feature is useful and tied to the open canonical dry-run issue, but this PR still has blocking dry-run safety gaps and the supplied proof does not show the patched `doctor --fix --dry-run` behavior or no-write result.

## What This Changes

The PR adds a doctor `--dry-run` flag, threads it through CLI/options/config flow, emits a proposed config diff, skips selected write/repair paths, updates docs, and adds finalize-flow tests.

## Best Possible Solution

Implement doctor dry-run as an end-to-end read-only planning mode: redact diff values, diff against an immutable original config, disable repair approvals and all side effects, and prove the patched command with redacted no-write output.

## Reproduction Assessment

Yes. Source inspection of the PR head and current helper contracts shows the remaining failures: raw diff serialization, active repair-mode prompter behavior, and `shouldRepair` advancing `cfg` before final diffing.

## Solution Assessment

No. The direction is plausible, but the current implementation is not the safest solution until dry-run is read-only across the whole doctor flow and uses redacted original-vs-candidate output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Redact dry-run diff values before printing:** `src/commands/doctor/finalize-config-flow.ts:24-28`
  - body: The dry-run diff stringifies arbitrary old/new config values before emitting them with `note()`. Paths such as gateway auth tokens, channel tokens, provider API keys, or SecretRef ids can leak into terminal logs; build the diff from redacted config objects or mask sensitive paths before rendering.
  - confidence: 0.92
- **[P1] Make dry-run disable repair approvals end to end:** `src/flows/doctor-health.ts:11-14`
  - body: This only bypasses the Nix write assertion while leaving `createDoctorPrompter({ options })` in repair mode for `--fix --dry-run`. Health contributions that call `confirmAutoFix()`, `confirmRuntimeRepair()`, or check `prompter.shouldRepair` can still rewrite auth/state/runtime files; force a read-only prompter/effective repair mode when `dryRun` is true.
  - confidence: 0.9
- **[P2] Compare dry-runs against an immutable baseline:** `src/commands/doctor-config-flow.ts:69-70`
  - body: With `--fix --dry-run`, `shouldRepair` remains true, and the existing config mutation helper assigns `cfg` to each proposed mutation. The finalizer can then compare an already-mutated `cfg` with `candidate` and report no diff for changes that `--fix` would apply; keep the original config as the baseline or make dry-run use non-mutating repair planning.
  - confidence: 0.88
- **[P3] Report clean dry-runs instead of falling through:** `src/commands/doctor/finalize-config-flow.ts:12`
  - body: The dry-run branch is gated by `pendingChanges`, so a clean `openclaw doctor --dry-run` skips the documented `No config changes detected.` note. The added test does not catch this because it still passes `pendingChanges: true` in the no-changes case.
  - confidence: 0.82
