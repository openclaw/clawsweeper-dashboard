## Summary

This PR is useful but not merge-ready: the latest patch narrows two write paths, but `--fix --dry-run` can still run repair-mode health contributions, can hide config diffs, and still lacks real proof of patched `doctor --dry-run` output plus a no-write result.

## What This Changes

Adds a doctor `--dry-run` CLI option, config diff output, docs, and finalize-flow unit tests for previewing doctor config repair changes without writing.

## Best Possible Solution

Implement dry-run as an explicit read-only planning mode across config flow and doctor health contributions, then add regression coverage for `--dry-run`, `--fix --dry-run`, no-side-effect guarantees, and clean output.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run `openclaw doctor --fix --dry-run` with legacy auth-profile, plugin-registry, or sandbox registry repair state and the repair prompter remains true while write assertions are skipped.

## Solution Assessment

No. Suppressing the final config write is too narrow; dry-run needs a read-only planning contract before repair-mode contributions are allowed to observe or act on `shouldRepair`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep dry-run out of repair-mode contributions:** `src/flows/doctor-health.ts:13`
  - body: With `--fix --dry-run`, this skips the write-mode assertion but still passes the original repair options into `createDoctorPrompter`, so `prompter.shouldRepair` and `confirmAutoFix` remain true for later health contributions. Those paths can still rewrite auth-profile stores, refresh plugin registry state, migrate sandbox registries, or run repair scripts before the final config writer is reached; dry-run needs a read-only prompter or contribution-level guard, not only a final write guard.
  - confidence: 0.91
- **[P2] Keep repair-mode mutations out of the diff baseline:** `src/commands/doctor-config-flow.ts:72`
  - body: When users pass `--fix --dry-run`, `shouldRepair` is still true, so existing calls to `applyDoctorConfigMutation` copy proposed changes into `cfg` as well as `candidate`. The new dry-run diff later compares `cfg` with `candidate`, so legacy normalization or unknown-key repairs can be reported as no diff even though `--fix` would change the file.
  - confidence: 0.88
- **[P2] Preview repair-sequence mutations without running them:** `src/commands/doctor-config-flow.ts:223`
  - body: Skipping `runDoctorRepairSequence` avoids some side effects, but it also drops several config repairs that `doctor --fix` would apply, including route rewrites, stale plugin cleanup, invalid plugin config quarantine, and allowlist repairs. The dry-run path should collect those planned mutations in a non-mutating mode so the preview still matches `--fix`.
  - confidence: 0.86
- **[P3] Report clean dry-runs when nothing changed:** `src/commands/doctor/finalize-config-flow.ts:12`
  - body: This branch only runs when `pendingChanges` is true, so a truly clean `openclaw doctor --dry-run` falls through without the documented `No config changes detected.` note. The added test sets `pendingChanges: true` with equal configs, which misses the real no-change path.
  - confidence: 0.79
