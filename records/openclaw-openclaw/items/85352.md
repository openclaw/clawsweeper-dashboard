## Label Justifications

- `P2`: This is a visible authenticated-session regression in a normal macOS dashboard workflow, with limited blast radius and no crash or data loss.
- `impact:session-state`: The issue is about the authenticated UI session briefly drifting into the credentials gate during a transient dashboard startup state.

## Summary

Keep open: current main and the latest release still start the Control UI disconnected, render the credentials gate on any disconnected state, and the macOS dashboard reloads the web UI on every show, so the reported flicker remains source-reproducible.

## What This Changes

Authenticated macOS menu bar opens should display the Control UI directly instead of briefly rendering the credentials/login gate during dashboard startup.

## Best Possible Solution

Extend the shared Control UI fix with macOS-specific behavior so reopening an already configured dashboard window reuses authenticated content or shows an authenticated connecting shell, while the full credentials gate remains reserved for missing credentials or explicit auth failures.

## Reproduction Assessment

Yes, at source level: current main reloads the macOS dashboard on show, starts the Control UI with `connected = false`, and renders the credentials gate for any disconnected state. I did not run the live macOS app to capture the visual flicker timing.

## Solution Assessment

Not applicable as a patch review because no fix is proposed here. The requested direction is correct, but the open related web reconnect PR alone is not sufficient for the macOS every-open reload path.
