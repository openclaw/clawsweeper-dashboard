## Summary

Keep open: current main and the reported v2026.5.20 release still have a source-level path where the macOS dashboard reloads into the generic disconnected Control UI state, which renders the credentials gate before the gateway hello. The related webchat PR is still open and does not touch the macOS dashboard reload boundary.

## What This Changes

The issue reports that the macOS menu bar dashboard briefly renders the credentials gate before the authenticated Control UI on every popover open.

## Best Possible Solution

Extend the shared Control UI fix with macOS-specific behavior so reopening an already configured dashboard window reuses authenticated content or shows an authenticated connecting shell, while reserving the full credentials gate for missing credentials or explicit auth failures.

## Reproduction Assessment

Yes, at source level: current main starts the Control UI disconnected, renders the login gate on any disconnected state, and the macOS dashboard reloads that web UI on show. I did not run a live macOS app proof in this read-only review.

## Solution Assessment

No patch is proposed here, and the open related PR is not sufficient for this macOS-specific every-open reload path. The safest fix should either avoid reloading the same authenticated dashboard window or add a native-auth startup state that suppresses only the transient gate, not real auth failures.
