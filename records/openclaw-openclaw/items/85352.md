- `impact:session-state`: The issue is about losing the authenticated UI state during a transient dashboard open/startup path and briefly rendering the wrong session state.

## Summary

Keep open: current main still has the same source-level macOS startup path that can show the credentials gate before the first gateway hello, and the related open Control UI PR does not cover the macOS dashboard reload path.

## What This Changes

The issue reports that the macOS menu bar app briefly shows the credentials/login gate before the authenticated Control UI when opening the dashboard.

## Best Possible Solution

Extend the shared Control UI fix with macOS-specific behavior so reopening an already configured dashboard window reuses authenticated content or shows an authenticated connecting shell, while the full credentials gate remains reserved for missing credentials or explicit auth failures.

## Reproduction Assessment

Yes, at source level: current main starts the Control UI disconnected, renders the login gate on any disconnected state, and the macOS dashboard reloads that web UI on show. I did not run a live macOS app proof in this read-only review.

## Solution Assessment

No patch is proposed here, and the open related PR is not sufficient for this macOS-specific every-open reload path. The best fix should either avoid reloading the same authenticated dashboard window or add a native-auth startup state that suppresses only the transient gate, not real auth failures.
