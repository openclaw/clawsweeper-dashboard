## Label Justifications

- `P2`: This is a visible authenticated-session UI bug in a normal macOS dashboard workflow, with limited blast radius and no crash or data loss.
- `impact:session-state`: The issue is about losing the authenticated UI state during a transient macOS dashboard open/startup path and briefly rendering the wrong session state.

## Summary

Keep open: current main and the latest release still have a source-level macOS path that can flash the credentials gate during dashboard reload before the gateway hello arrives, and the related open web UI PR does not close or fully cover this macOS every-open reload case.

## What This Changes

The issue asks for the macOS menu bar dashboard to open directly to the authenticated Control UI without briefly showing the login/credentials gate.

## Best Possible Solution

Preserve the authenticated dashboard across macOS reopens, or show an authenticated connecting shell during native reload, while reserving the full credentials gate for missing credentials or explicit auth failure.

## Reproduction Assessment

Yes, at source level: current main reloads the macOS dashboard on show, starts the Control UI with `connected = false`, and renders the credentials gate for any disconnected state. I did not run a live macOS app recording to capture the visual flicker timing.

## Solution Assessment

No patch is proposed here, and the related open web UI PR is not sufficient until it covers the macOS dashboard reload path. The maintainable fix should distinguish authenticated reconnect/startup from missing or failed auth without hiding real auth failures.
