## Label Justifications

- `P2`: This is a visible authenticated-session UI regression in a normal macOS dashboard workflow, with limited blast radius and no crash or data loss.
- `impact:session-state`: The issue is about briefly rendering the wrong authenticated-session state during the macOS dashboard startup/reopen path.

## Summary

Keep open: current main and v2026.5.20 still have a source-reproducible path where the macOS dashboard reloads the Control UI, starts disconnected, and renders the credentials gate before the first gateway hello. The related web reconnect PR is still open and does not touch the macOS dashboard reload boundary.

## What This Changes

The issue asks for the macOS menu bar dashboard to stop briefly showing the credentials/login gate on every open when the app is already authenticated.

## Best Possible Solution

Preserve the shipped auth-failure gate, but let an already configured macOS dashboard reuse authenticated content or show an authenticated connecting shell during startup/reopen instead of rendering the full credentials form.

## Reproduction Assessment

Yes, at source level: current main reloads the macOS dashboard web view, starts the Control UI disconnected, and renders the login gate until the gateway hello. I did not run a live macOS app proof in this read-only review.

## Solution Assessment

No patch is proposed here, and the related web reconnect PR is not sufficient for the macOS every-open reload path. The safest fix should either avoid reloading a previously authenticated dashboard or add a prior-auth/native startup state that suppresses only the transient gate.
