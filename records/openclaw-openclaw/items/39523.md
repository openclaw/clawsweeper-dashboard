## Summary

Keep open: current main still lacks Feishu config-backed local media roots, but this PR is not safe to merge because the advertised `"any"` mode is routed without the host-read contract and normal outbound sends would still shadow configured roots.

## What This Changes

The PR adds a Feishu `mediaLocalRoots` config field and makes `sendMediaFeishu` fall back to account config when caller roots are absent, including `"any"`.

## Best Possible Solution

Add Feishu-owned config support through the shared outbound media access/readFile contract, with explicit root merge or precedence semantics, generated config metadata, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection shows configured `"any"` would reach `loadWebMedia` without `readFile` and regular outbound paths already pass scoped roots, so the PR's config fallback is bypassed in the main flow.

## Solution Assessment

No. The direction is useful, but the safer implementation is to use the shared outbound media access contract and settle configured-root precedence before exposing the Feishu config key.

## Review Findings

Overall correctness: patch is incorrect

  - body: `loadWebMedia` rejects `localRoots: "any"` unless a `readFile` override is also provided. This change forwards configured `"any"` directly from `sendMediaFeishu`, so the advertised setting fails closed with `unsafe-bypass` instead of using the shared host-read helper path.
  - confidence: 0.95

Because the fallback uses `mediaLocalRoots ?? account.config?.mediaLocalRoots`, those injected roots suppress configured account roots or `"any"`, so the new config still does not work in the main flow this PR targets.
  - confidence: 0.9
- **[P3] Update Feishu config metadata and docs:** `extensions/feishu/src/config-schema.ts:165`
  - body: Adding a persisted Feishu channel setting needs the generated config metadata and user-facing config reference to stay in sync. Those surfaces currently list `mediaMaxMb` but not `mediaLocalRoots`, so config/help surfaces would not expose the new key.
  - confidence: 0.82
