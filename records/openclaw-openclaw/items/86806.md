## Summary

Keep open: the PR still has a P1 correctness blocker because it treats the post-ready session-lock sweep as a true startup boundary, so a live post-channel grace-window lock can be marked as crash residue.

## What This Changes

The branch adds `safeToRecover` lock cleanup output, a crash marker that sets `abortedLastRun` from those locks, and Gateway startup cleanup/tests that call it with `startupMode`.

## Best Possible Solution

Run startup-only crash recovery before channel/session writers start, keep post-ready cleanup runtime-safe, and add redacted real Gateway crash/restart proof covering recovery without false-marking a post-channel writer.

## Reproduction Assessment

Yes for the review concern. PR-head source runs channel startup before scheduling `sidecars.session-locks`, while `startupMode: true` includes grace-window orphan locks, so a post-channel payload-less lock before the sidecar sweep can be marked from source inspection.

## Solution Assessment

No. The explicit cleanup classification is a good direction, but using `startupMode: true` from the existing post-ready sidecar is not the best fix because that sidecar is not a pre-writer startup boundary.
