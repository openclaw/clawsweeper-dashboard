## Summary

Keep open: the fallback targets a real session-loss bug, but this patch treats raw transcript lock existence as crash evidence even though live runs create the same lock before delayed recovery can run, and the external PR still lacks real crash/restart proof.

## What This Changes

The PR refactors main-session restart recovery to add a lock-file fallback for running sessions without `abortedLastRun`, adds focused tests, and carries a whitespace-only plugin-sdk touch.

## Best Possible Solution

Revise the fallback to use stale/dead lock ownership evidence or startup-cleanup inspection, add a live-lock race regression test, then provide redacted real Gateway crash/restart WebChat proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main only recovers running sessions with `abortedLastRun === true`, so a running session with an intact transcript and no marker is skipped. I did not run tests because this review was explicitly read-only.

## Solution Assessment

No, not as currently proposed. The useful fallback idea should be kept, but the implementation must prove a lock is stale or crash-owned before resuming so normal live locks are not treated as crashes.
