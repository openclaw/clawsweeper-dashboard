## Label Justifications

- `P1`: This PR fixes a user-facing gateway crash recovery path where active conversation state and message continuity can be lost.
- `merge-risk: 🚨 session-state`: The patch changes when `abortedLastRun` is set for running sessions and can mis-associate live session state if startup ordering is wrong.
- `merge-risk: 🚨 message-delivery`: Incorrect crash marking can cause recovery prompts or retries to target active work, risking duplicated or misdelivered continuation messages.

## Summary

Keep open: the PR targets a real session-recovery gap, but the new fallback is wired after channel startup and can treat a fresh live lock as crash residue.

## What This Changes

The PR adds a startup fallback that marks running main sessions with matching remaining non-stale transcript locks as `abortedLastRun` and adds focused unit coverage.

## Best Possible Solution

Keep the recovery fallback, but capture crash-lock evidence before channels can create new locks or add an equivalent pre-start guard, then cover that ordering in startup tests.

## Reproduction Assessment

Yes, source-reproducible: current main only marks sessions from cleaned stale locks, so a non-stale remaining lock leaves `abortedLastRun` unset and recovery skips the running session. I did not run a live gateway crash reproduction in this read-only review.

## Solution Assessment

No, not as wired: using remaining locks as crash evidence is the right direction, but it must happen before channels can create live locks or be constrained to locks known to predate this gateway start.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep crash-lock marking before channels can create locks:** `src/gateway/server-startup-post-attach.ts:583`
  - body: This branch treats every remaining non-stale lock as crash residue, but `startGatewaySidecars()` starts channels before queuing `sidecars.session-locks`, and the task itself waits for `setImmediate`. A channel or client can start a new main run in that window and write a live non-stale `.jsonl.lock`, which this fallback would mark as `abortedLastRun` and later recover over active work. Move this evidence capture before channel startup, or filter to locks provably older than the current gateway start and not owned by this process.
  - confidence: 0.88
