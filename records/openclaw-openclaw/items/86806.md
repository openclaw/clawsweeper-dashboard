## Summary

Keep this PR open, but it is not merge-ready: it targets a real session-recovery bug, yet the fallback can treat a live current-process transcript lock as crash evidence and the external PR has only unit-test proof.

## What This Changes

The PR refactors main-session restart recovery, adds a lock-file fallback for running sessions without `abortedLastRun`, adds focused tests, and carries a stale whitespace-only touch in `src/plugin-sdk/provider-tools.ts`.

## Best Possible Solution

Keep the linked issue open and revise this PR so fallback recovery uses stale/dead lock inspection or startup-cleanup evidence, not raw lock existence, then prove a real crash/restart WebChat recovery path.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main only recovers running sessions with `abortedLastRun === true`, so a running session with an intact transcript and no marker is skipped. I did not run tests because this review was explicitly read-only.

## Solution Assessment

No, not as currently proposed. The fix should preserve the useful fallback idea but use stale/dead lock inspection or startup cleanup evidence so normal live locks created before delayed recovery are not treated as crashes.
