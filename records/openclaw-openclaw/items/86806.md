## Label Justifications

- `P1`: The PR targets and can affect Gateway crash recovery for active main sessions, where wrong recovery state can strand or misresume user work.
- `merge-risk: 🚨 session-state`: The diff changes when `abortedLastRun` is set from lock-file classification, which can stale or mis-associate main-session recovery state.
- `merge-risk: 🚨 message-delivery`: Incorrect restart recovery state can drive resumed replies or failure notices for the wrong active session after a Gateway restart.

## Summary

Keep open: the change is still useful for the linked Gateway crash-recovery bug, but the current patch is not merge-ready because it uses the startup-only recovery classification from a post-channel-start cleanup path, where live session writers can already exist.

## What This Changes

The PR adds a `safeToRecover` result to session lock cleanup, wires Gateway startup to mark crashed main sessions from that classification, and adds focused regression tests.

## Best Possible Solution

Make the crash-residue classification run before any channel/session writer can create new locks, or keep orphan-payload grace protection in post-ready cleanup and use a different startup-safe signal, then prove the real crash/restart flow.

## Reproduction Assessment

Yes. Source inspection shows channels start before the session-lock cleanup task, while the PR passes `startupMode: true` in that later cleanup and bypasses the young orphan-payload lock guard.

## Solution Assessment

No. Moving recovery classification into the cleanup owner is a good direction, but this implementation is not the best fix until the classification is limited to a true no-writer startup window or keeps grace-window protection after channels start.
