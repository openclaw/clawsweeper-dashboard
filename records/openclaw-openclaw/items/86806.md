## Summary

This PR should stay open as an active fix candidate for the Gateway crash-recovery bug, but the current patch is not merge-ready because it runs startup-only grace-window recovery from a post-ready sidecar after channel writers can already exist.

## What This Changes

The PR adds `safeToRecover` lock cleanup output, a crash-session marker that sets `abortedLastRun` from those locks, and Gateway startup cleanup/tests that call it with `startupMode: true`.

## Best Possible Solution

Run startup-only crash recovery before channel/session writers can create locks, keep post-ready cleanup in runtime-safe mode, and add redacted process-level crash/restart proof.

## Reproduction Assessment

Yes for the review concern: PR-head source runs channel startup before scheduling `sidecars.session-locks`, while `startupMode: true` includes grace-window orphan locks. That makes the false-mark path source-reproducible without running a live Gateway crash.

## Solution Assessment

No. The explicit cleanup-owned classification is a good direction, but using `startupMode: true` from the existing post-ready sidecar is not the best fix because that sidecar is not a pre-writer startup boundary.
