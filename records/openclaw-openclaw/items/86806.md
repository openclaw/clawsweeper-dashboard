## Summary

Keep this PR open: it fixes the all-remaining-lock crash case and has useful proof, but the startup wiring still skips remaining non-stale crash locks whenever the same sweep also cleaned at least one stale lock.

## What This Changes

The PR adds a remaining-lock crash marker, wires it into the gateway session-lock sidecar, and adds focused restart-recovery tests for non-stale lock evidence.

## Best Possible Solution

Run stale-lock marking and remaining-lock marking independently over their respective lock subsets during session-lock startup, with regression coverage for mixed cleaned and remaining lock results.

## Reproduction Assessment

Yes. Source inspection shows `cleanStaleLockFiles()` can return both `locks` and `cleaned`, while the PR's `else if` only runs remaining-lock recovery when `cleaned.length === 0`.

## Solution Assessment

No, not yet. The helper is a plausible fix, but the startup wiring should call both markers for mixed cleaned and remaining results instead of choosing one branch.
