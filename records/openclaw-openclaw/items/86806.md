## Summary

Keep this PR open, but it is not merge-ready: the session-restore bug is valid, yet the new fallback skips the live/non-stale lock shape it claims to recover and can still mark preserved unknown-owner locks as crash residue.

## What This Changes

The PR adds a Gateway startup fallback that marks running main sessions as `abortedLastRun` from non-cleaned session lock files, plus focused recovery tests.

## Best Possible Solution

Make lock cleanup emit an explicit safe-to-recover remaining-lock classification, then drive the startup marker from that contract with coverage for both crash residue and preserved-owner locks.

## Reproduction Assessment

Yes for the review findings: source inspection shows real cleanup removes dead-PID locks into `cleaned`, preserves young unknown-owner locks, and the PR marker skips live non-stale locks while admitting pre-start `pidAlive: false` remaining locks. I did not run tests because this review is read-only.

## Solution Assessment

No. A remaining-lock recovery path is plausible, but the best fix is to make the lock cleanup owner classify recoverable residue explicitly instead of having Gateway infer it from all non-cleaned locks.
