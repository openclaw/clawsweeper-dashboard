## Summary

Keep open: the current head addresses the prior live-lock race with a `gatewayStartedAt` mtime guard, but the fallback still only runs when no stale locks were cleaned, so mixed stale/non-stale crash residue can leave some crashed sessions unrecovered.

## What This Changes

The PR adds a startup marker that uses remaining pre-start transcript lock files to set `abortedLastRun` on matching running main sessions, plus direct recovery tests.

## Best Possible Solution

Process cleaned stale locks and remaining pre-start non-stale locks in the same startup pass, then cover mixed lock sets and the timestamp guard before merge.

## Reproduction Assessment

Yes, source-reproducible: current main only marks sessions from cleaned stale locks, so remaining non-stale locks can leave a running session without `abortedLastRun`. I did not run a live Gateway crash reproduction in this read-only review.

## Solution Assessment

No, not yet: the timestamp-guarded remaining-lock marker is the right direction, but the `else if` wiring leaves mixed cleaned/non-stale crash residue partially unrecovered.

Full review comments:

- **[P2] Mark remaining crash locks when stale locks were cleaned:** `src/gateway/server-startup-post-attach.ts:581`
  - body: This `else if` skips the new remaining-lock marker whenever `cleanStaleLockFiles()` cleaned at least one stale lock. Since that API returns all inspected locks plus the cleaned subset, a crash with one stale lock and one PID-reused/non-stale pre-start lock will mark only the cleaned session; the non-stale session stays `running` without `abortedLastRun` and the later recovery still skips it. Run the remaining-lock marker for the non-removed pre-start locks after the cleaned-lock marker, or otherwise cover the mixed lock set.
