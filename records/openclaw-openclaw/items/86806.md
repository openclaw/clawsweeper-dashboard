## Summary

Keep this PR open, but the current patch is not merge-ready: it fixes a real crash-recovery gap in the right area, but the latest `startupMode` path is still wired through a post-channel cleanup sidecar, so it can classify a live grace-window orphan lock as crash residue.

## What This Changes

The PR adds startup-aware `safeToRecover` lock cleanup output and uses it in Gateway’s post-ready session-lock cleanup to mark matching running main sessions as `abortedLastRun`.

## Best Possible Solution

Run startup-only lock recovery before live session writers exist, and keep the existing post-ready cleanup on runtime-safe classification.

## Reproduction Assessment

Yes. Source inspection shows the PR-head path runs channel startup before scheduling session-lock cleanup, while `startupMode: true` allows grace-window orphan locks into `safeToRecover`; arranging a post-channel orphan lock before `sidecars.session-locks` runs reproduces the review concern from source.

## Solution Assessment

No. The explicit cleanup classification is a good direction, but using `startupMode: true` from the existing post-ready sidecar is not the best fix because that sidecar does not run before all live writers can exist.
