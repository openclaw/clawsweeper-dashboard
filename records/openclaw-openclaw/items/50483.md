## Summary

Keep open: the branch is still blocked by missing real behavior proof and a source-reproducible load-latch regression in the shared chat view model.

## What This Changes

The PR changes shared OpenClawKit chat streaming/session routing, assistant text segment IDs, scroll and typing visibility behavior, Swift tests, and the changelog.

## Best Possible Solution

Keep the session-scoped streaming/parser direction, but make bootstrap state transitions refreshable and covered by focused Swift tests, then require redacted device/simulator or terminal proof before merge.

## Reproduction Assessment

Yes for the PR-head regression: source inspection shows successful manual refresh after an initial load failure does not mark the view model loaded, so a later onAppear can bootstrap again and clear active streaming state. I did not run the Swift tests in this read-only review.

## Solution Assessment

No. The streaming/session scoping direction is useful, but the load latch should guard bootstrap churn without making later appearances or successful manual-recovery bootstraps inconsistent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve chat refresh state after recovery:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatViewModel.swift:254`
  - body: After an initial load failure, a successful manual Refresh calls bootstrap() without setting hasLoadedInitialState back to true. The next onAppear will call load(), run another bootstrap, and bootstrap clears pending runs, tool calls, and streaming text, so navigating away and back during a later in-flight reply can drop the active stream. Set the loaded state on any successful bootstrap, or otherwise keep the churn guard consistent across load() and refresh().
  - confidence: 0.88
