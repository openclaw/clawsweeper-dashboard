## What This Changes

This PR adds session-scoped Swift chat streaming handling, stable assistant text segment IDs, scroll/typing indicator tweaks, regression tests, and a changelog entry for iOS/shared OpenClawKit chat.

## Best Possible Solution

Keep the shared OpenClawKit fix direction, but update ChatViewModel to preserve optimistic final assistant messages through history reconciliation, mark successful bootstrap/refresh recovery as loaded, add focused regression coverage, and attach redacted real iOS behavior proof.

## Reproduction Assessment

Yes for the blocking PR defects from source inspection: the branch can leave hasLoadedInitialState false after a successful manual refresh, and immediate assistant-final append is followed by a refresh path that does not preserve assistant finals if history is stale. I did not run the iOS app or Swift tests in this read-only review.

## Solution Assessment

No, not yet. The branch targets the right shared OpenClawKit surface, but it needs the two ChatViewModel fixes and contributor-supplied real behavior proof before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve optimistic finals through history refresh:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatViewModel.swift:1131`
  - body: `appendFinalAssistantMessage` appends a final assistant reply immediately, and both terminal paths then call `refreshHistoryAfterRun()`. If the first history response is eventually consistent and lacks that assistant row, `reconcileRunRefreshMessages` only carries forward trailing user messages, so the reply can appear and then disappear until a later refresh. Track these optimistic assistant finals or merge them during refresh until history confirms them.
  - confidence: 0.86
- **[P2] Mark successful refresh recovery as loaded:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatViewModel.swift:254`
  - body: Resetting `hasLoadedInitialState` on bootstrap failure makes retry possible, but a successful manual `refresh()` never sets it back to true. After that recovery, the next `onAppear` calls `load()` again and `bootstrap()` clears pending runs, tool calls, and streaming text, so navigating away and back during a new reply can drop the active stream.
  - confidence: 0.84
