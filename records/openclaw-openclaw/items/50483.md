## Summary

Keep open: this is a plausible native chat streaming fix, but the latest PR head still has a source-proved load-latch regression and lacks required inspectable real behavior proof before merge.

## What This Changes

The branch updates shared OpenClawKit chat UI/model parsing and tests to scope streaming by session key, reduce streaming layout churn, debounce scrolling, reconcile terminal history, and add a changelog entry.

## Best Possible Solution

Keep the session-scoped streaming/parser direction, fix the recovered-refresh loaded-state transition with focused Swift coverage, then require redacted simulator/device or terminal proof before merge.

## Reproduction Assessment

Yes for the PR-head regression by source inspection: a failed initial load followed by successful manual refresh leaves hasLoadedInitialState false, so a later onAppear can bootstrap again and clear active stream state. I did not run the Swift tests in this read-only review.

## Solution Assessment

No. The overall streaming/session scoping direction is useful, but the patch should mark any successful bootstrap as loaded and include contributor-provided real behavior proof before it is the best merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Mark recovered refreshes as loaded:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatViewModel.swift:254`
  - body: When the initial `load()` bootstrap fails, this resets `hasLoadedInitialState`, but a later successful manual `refresh()` calls `bootstrap()` without ever setting the flag back to true. The next view `onAppear` will call `load()`, start another bootstrap, and clear pending runs, tool calls, and `streamingAssistantText`, so navigating away and back during a later in-flight reply can drop the active stream. Set the flag on any successful bootstrap, not only before `load()` starts one.
  - confidence: 0.88
