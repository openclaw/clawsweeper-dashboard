## Summary

Keep open. Current main still lacks part of the proposed chat streaming/session-key work, but the latest PR head should not merge as-is because it blocks later chat refreshes after the first bootstrap and mixes an unrelated Windows installer execution-path change into a chat UI fix.

## What This Changes

This PR changes shared OpenClawKit chat parsing, agent-event decoding, chat scrolling and typing state, ChatViewModel session/stream handling, Swift regression tests, and Windows installer failure handling.

## Best Possible Solution

Land a focused chat-only patch that keeps the useful session-key decoding, parser stability, stream scoping, scroll/typing behavior, and regression tests while preserving a later refresh path; move the Windows installer change to a dedicated installer-reviewed PR and add the required changelog entry.

## Reproduction Assessment

Yes. The main blocker is statically reproducible: `OpenClawChatView.onAppear` calls `load()`, and the PR head makes `load()` return forever after one successful bootstrap, so reopening the chat view no longer refreshes state. The installer concern is directly visible in the PR diff.

## Solution Assessment

No. The chat direction is useful, but the current PR is not the narrowest maintainable fix until it preserves refresh behavior after bootstrap and removes or separately reviews the Windows installer change.
