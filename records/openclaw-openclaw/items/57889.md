## Summary

Keep open: the patch still changes the gateway device-identity trust boundary by returning before shared policy rejects Tailscale-only device-less operators, and it lacks real after-fix gateway proof.

## What This Changes

The PR adds an early WebSocket connect allow path for device-less operator sessions authenticated by Tailscale or trusted-proxy when the shared-auth probe fails, plus a policy test documenting that mismatch.

## Best Possible Solution

Settle the gateway trust contract for device-less transport/shared-auth operators, then encode any approved exception in shared connect policy with docs/tests and real end-to-end proof for connect plus an operator.write RPC.

## Reproduction Assessment

Yes. Source inspection plus checked-in tests show current main rejects Tailscale-only device-less operators and clears scopes for Tailscale-routed shared-token sessions, while the PR early return bypasses that policy path.

## Solution Assessment

No. The patch bypasses centralized missing-device policy inside message-handler.ts; the safer path is an explicit auth-policy decision implemented in the shared policy helper with aligned tests, docs, and live proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep Tailscale operators on the missing-device policy path:** `src/gateway/server/ws-connection/message-handler.ts:568`
  - body: This early return is reachable when primary Tailscale auth succeeds without shared token/password auth or device identity: sharedAuthOk remains false, but the code returns true before evaluateMissingDeviceIdentity can reject and before unbound scopes are cleared. That lets a remote header-authenticated operator keep self-declared operator scopes without pairing or device-signature checks.
