## Summary

Keep open. The PR is a focused attempt at a real gateway auth failure, but the current branch still lacks real behavior proof and the handler-local early return changes a device-identity authorization boundary instead of resolving the policy in the centralized connect-policy/config path.

## What This Changes

The PR adds a WebSocket connect-path bypass that lets device-less operator connections authenticated by Tailscale or trusted-proxy keep requested scopes when the secondary shared-auth probe fails.

## Best Possible Solution

Resolve the cluster through one maintainer-approved gateway auth scope policy, likely a centralized connect-policy/config path with security docs, upgrade behavior, tests, and real gateway proof, rather than a handler-local early return.

## Reproduction Assessment

Yes, at source level. Current main still encodes the device-less shared-auth scope-clearing behavior in code and tests, and the PR-introduced bypass is visible directly in the diff; I did not run a live Tailscale/trusted-proxy gateway repro.

## Solution Assessment

No. The handler-local early return is not the best fix because this is an auth/security policy decision that belongs in the centralized connect policy or an explicit token-scope configuration path.
