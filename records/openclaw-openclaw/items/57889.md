## Summary

Keep open. The PR targets a real gateway scope-clearing cluster, but this version still changes a security-sensitive WebSocket auth boundary by returning before the centralized missing-device policy and scope-clearing path, and the contributor has not provided after-fix real gateway proof.

## What This Changes

The PR adds an early WebSocket connect allow path for device-less operator sessions authenticated by Tailscale or trusted-proxy when the shared-auth probe fails, plus a connect-policy test documenting that bypass.

## Best Possible Solution

Define one maintainer-approved transport/shared-auth missing-device operator policy, then implement it in connect-policy and scope clearing with aligned auth baselines, protocol docs, and redacted real gateway proof.

## Reproduction Assessment

Yes, at source level. Current main's auth suites cover the behavior: Tailscale-only device-less operators are rejected, and Tailscale-routed shared-token operators connect with scopes cleared before scoped RPCs fail.

## Solution Assessment

No. The PR bypasses the centralized missing-device policy inside message-handler.ts; any approved exception should be modeled in connect-policy with explicit trust semantics, scope-clearing rules, tests, docs, and real gateway proof.
