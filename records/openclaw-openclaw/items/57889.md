## Summary

Keep open. The PR’s latest shape still bypasses the WebSocket missing-device policy for Tailscale-authenticated operator sessions without a device identity, which conflicts with current source, tests, and protocol guidance for this security-sensitive gateway boundary.

## What This Changes

The PR adds an early allow path for device-less WebSocket operator sessions authenticated by Tailscale or trusted-proxy when `sharedAuthOk` is false, plus a connect-policy test documenting that mismatch.

## Best Possible Solution

Resolve the gateway trust policy first, then encode any approved exception in shared connect policy, docs, and end-to-end coverage for connect plus an `operator.write` RPC.

## Reproduction Assessment

Yes. From source and checked-in tests, a Tailscale-only WebSocket operator with no device identity follows the PR’s early return while current main expects `device identity required`; I did not run a live gateway in this read-only review.

## Solution Assessment

No. The patch bypasses the centralized missing-device decision inside `message-handler.ts`; the safer path is an explicit, reviewed gateway auth policy change with aligned code, tests, and docs.
