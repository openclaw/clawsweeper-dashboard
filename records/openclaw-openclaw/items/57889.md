## Summary

Keep open. The PR addresses a real gateway-scope pain point, but the current patch changes a device-identity authorization boundary by returning early from the WebSocket missing-device path, and it still needs maintainer security/product judgment plus real gateway proof before merge.

## What This Changes

The branch adds an early WebSocket connect allow path for device-less operator sessions authenticated by Tailscale or trusted-proxy, plus a connect-policy test documenting that bypass.

## Best Possible Solution

Use a maintainer-approved gateway auth scope policy, likely an explicit configured token-scope/trust path or a documented device-identity requirement, with source, regression tests, protocol/security docs, and real gateway proof kept aligned.

## Reproduction Assessment

Yes, at source level. Current main accepts some valid shared-auth device-less operator paths with cleared scopes and checked-in tests expect the follow-up scoped operation to fail with `missing scope`; the PR-introduced bypass is also visible directly in the diff.

## Solution Assessment

No. Returning early from `message-handler.ts` bypasses the centralized missing-device decision and scope-clearing contract; the safer path is an explicit maintainer-approved auth policy/config surface rather than a handler-local exception.
