## Summary

Keep open: the latest branch still needs human review because it broadens the gateway device-identity trust boundary, lacks real behavior proof, and contains a blocking security/correctness issue.

## What This Changes

The PR adds an early WebSocket connect allow path for device-less operator sessions authenticated by Tailscale or trusted-proxy when `sharedAuthOk` is false, plus a connect-policy test documenting that mismatch.

## Best Possible Solution

Settle the gateway trust contract for device-less transport/shared-auth operators, then encode any approved exception in shared connect policy with docs/tests and real end-to-end proof for connect plus an `operator.write` RPC.

## Reproduction Assessment

Yes. From source inspection and checked-in tests, current main rejects Tailscale-only device-less operators and clears scopes for Tailscale-routed shared-token sessions, while the PR early return would bypass that policy path.

## Solution Assessment

No. The patch bypasses centralized missing-device policy inside `message-handler.ts`; the safer path is an explicit auth-policy decision implemented in the shared policy helper with aligned tests and docs.

## Review Findings

Overall correctness: patch is incorrect
