## Summary

Keep open. The linked operator-scope problem is real, but this branch is not mergeable because it creates a device-less Tailscale operator allow path that current source, tests, and docs explicitly keep behind device identity or narrower trust exceptions.

## What This Changes

The PR adds a WebSocket connect early-allow path for device-less operator sessions authenticated by Tailscale or trusted-proxy when `sharedAuthOk` is false, plus a connect-policy test documenting that mismatch.

## Best Possible Solution

Resolve the gateway trust policy first, then encode any approved exception in the shared connect policy with docs and end-to-end coverage for connect plus an `operator.write` RPC.

## Reproduction Assessment

Yes. A WS connect with verified Tailscale headers, `role: "operator"`, no shared token/password, and `device: null` reaches the PR early return with `authOk=true`, `authMethod="tailscale"`, and `sharedAuthOk=false`; current tests assert that same shape should require device identity.

## Solution Assessment

No. The patch bypasses the centralized missing-device decision in `message-handler.ts` and admits Tailscale-only operator sessions, while the safer direction is an explicit policy change with device/shared-secret proof boundaries documented and tested.

## Review Findings
