## Summary

Keep open. Current main still does not implement this PR's requested early-allow path, but the proposed patch is not merge-ready because it bypasses the gateway missing-device policy for Tailscale-authenticated device-less operator sessions that current tests and docs still keep on the device-identity path.

## What This Changes

This PR adds an early WebSocket connect allow path for device-less operator sessions authenticated by Tailscale or trusted-proxy when `sharedAuthOk` is false, plus a connect-policy test documenting the mismatch case.

## Best Possible Solution

Resolve the gateway trust policy first, then implement any approved exception in the shared WebSocket auth policy with docs and integration coverage for connect plus an `operator.write` RPC. If device-less Tailscale/trusted-proxy operators should remain outside the supported exception list, close this PR with that policy decision and keep the broader token-auth reports tracked separately.

## Reproduction Assessment

Yes. Current main has focused source/test coverage for the boundary: Tailscale-only device-less operators are rejected, while Tailscale-routed shared-token operators connect with cleared scopes and then fail scope-gated RPCs. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The PR bypasses `evaluateMissingDeviceIdentity()` in `message-handler.ts` instead of encoding an approved central policy, and its condition also admits Tailscale-only operator sessions without a shared secret or device identity. A safer solution needs explicit maintainer security approval and end-to-end coverage of the exact supported credential/locality/client shape.
