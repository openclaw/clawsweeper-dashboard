## Summary

Keep this PR open. Current main still lacks the proposed VS Code debugging support, and the PR still has a concrete P2 issue: the one-click debug launch deletes `dist/` before using the narrower `run-node` rebuild path, which does not recreate all full-build runtime assets.

## What This Changes

The PR adds VS Code launch/tasks configs, an opt-in tsdown sourcemap toggle, a `clean:dist` script, and VS Code debugging docs.

## Best Possible Solution

Land the DX improvement only after the launch flow uses a supported source-map build path that preserves or recreates full-build runtime assets, with docs that clearly separate one-click and manual debug workflows.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the introduced regression: start from a clean `dist`, use the PR's `Rebuild and Debug Gateway` flow, then exercise `/export-session`; source inspection shows the clean task removes the templates and `run-node` does not recreate them.

## Solution Assessment

Is this the best way to solve the issue? No. The opt-in sourcemap toggle is reasonable, but deleting all of `dist/` before a narrower runtime rebuild is not the narrowest maintainable debug path; a full debug build task or asset-preserving rebuild would be safer.
