## Summary

Keep this PR open: current main still lacks the proposed VS Code debugging support, and the latest PR head resolves the earlier generated-asset concern by switching the prelaunch task to a full build, but the one-click launch config still uses the wrong argument channel for `pnpm`.

## What This Changes

The PR adds VS Code launch/tasks configs, an opt-in `OUTPUT_SOURCE_MAPS` sourcemap toggle, a `clean:dist` script, and VS Code Gateway debugging docs.

## Best Possible Solution

Land the DX improvement after the one-click launch uses a supported `pnpm` launch shape, the docs match the full-build prelaunch task, and any required contributor-facing changelog entry is added.

## Reproduction Assessment

Yes for the review finding by source and dependency-contract inspection: the PR launches `pnpm` via `runtimeExecutable` but passes the subcommand through `args`, while VS Code documents runtime-tool subcommands under `runtimeArgs`. I did not establish a live VS Code failure in this read-only review.

## Solution Assessment

No. The full-build prelaunch strategy is safer than the earlier run-node-only clean path, but the launch config should use `runtimeArgs` or a direct `program` shape before becoming the checked-in one-click debug path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Move the pnpm subcommand into runtime args:** `.vscode/launch.json:10`
  - body: The one-click config sets `runtimeExecutable` to `pnpm` but puts `run openclaw gateway run` in `args`. In Node launch configs, `args` are for the debugged program and tool arguments belong in `runtimeArgs`; with no `program` set here, this can start `pnpm` without the intended script and leave the advertised debugger unable to launch the Gateway.
  - confidence: 0.86
- **[P3] Keep the rebuild note aligned with the task:** `docs/help/debugging.md:320`
  - body: The task now runs `pnpm clean:dist && pnpm build` before launch, but this note still says `run-node.mjs` rebuilds the project. That points developers at the wrong build path when the one-click flow is slow or fails.
  - confidence: 0.8
