## Summary

Keep this PR open. Current main still lacks the proposed VS Code debugging support, but the PR's one-click clean debug path is not merge-ready because it deletes `dist/` and then relies on the narrower `run-node` rebuild path, leaving full-build assets such as export-session templates unavailable.

## What This Changes

The PR adds VS Code launch/tasks configs, an opt-in `OUTPUT_SOURCE_MAPS` tsdown sourcemap toggle, a `clean:dist` script, and VS Code debugging docs.

## Best Possible Solution

Land the DX improvement only after the VS Code launch flow uses a supported source-map build path that preserves or recreates full-build runtime assets, with docs that clearly separate one-click and manual debug workflows.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the introduced PR regression by source inspection: start from a clean `dist`, use the PR's `Rebuild and Debug Gateway` path, then exercise `/export-session`; the launch deletes `dist` and the follow-up `run-node` rebuild does not recreate `dist/export-html`.

## Solution Assessment

Is this the best way to solve the issue? No. The opt-in sourcemap toggle is reasonable, but deleting all of `dist/` before a narrower runtime rebuild is not the narrowest maintainable debug path; a full debug build task or asset-preserving rebuild would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Preserve generated assets in clean debug launches:** `.vscode/launch.json:8`
  - body: The new prelaunch task deletes all of `dist/` and then launches `pnpm run openclaw -- gateway run`.
