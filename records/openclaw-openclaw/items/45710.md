## Summary

Keep this PR open. Current main still lacks the proposed VS Code debugging support, and the PR remains useful, but the one-click clean/rebuild launch path can leave generated runtime assets missing because it deletes `dist/` before invoking the narrower `run-node` rebuild path.

## What This Changes

The PR adds VS Code launch/tasks configs, an opt-in `OUTPUT_SOURCE_MAPS` tsdown sourcemap toggle, a `clean:dist` script, and VS Code debugging docs.

## Best Possible Solution

Land the DX improvement after the checked-in VS Code launch path either runs a supported full source-map build or preserves/recreates generated runtime assets, with docs that separate one-click and manual debug workflows.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the introduced regression by source inspection: start from a clean `dist`, use `Rebuild and Debug Gateway`, then exercise `/export-session`; the PR removes `dist` and the follow-up `run-node` path does not copy export HTML templates.

## Solution Assessment

Is this the best way to solve the issue? No. The sourcemap toggle and docs are reasonable, but deleting all of `dist/` before a narrower runtime rebuild is not the narrowest maintainable one-click debug path; a full source-map build task or asset-preserving flow would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve generated assets in clean debug launches:** `.vscode/launch.json:8`
  - body: The new prelaunch task deletes all of `dist/` and then launches `pnpm run openclaw -- gateway run`. That runner rebuilds TypeScript plus runtime-postbuild artifacts, but it does not run the full build step that copies `dist/export-html`, while `/export-session` reads those templates from disk. A gateway started through the advertised clean debug launch can therefore fail with missing templates during the debug session.
  - confidence: 0.9
