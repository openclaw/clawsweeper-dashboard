## Summary

Keep this PR open: current main still lacks the VS Code debugging surface, and the latest PR head addressed the earlier full-build/generated-asset concern, but the one-click launch config still uses the wrong VS Code argument field and the docs/changelog are not merge-ready.

## What This Changes

The PR adds VS Code launch/tasks configs, an opt-in `OUTPUT_SOURCE_MAPS` sourcemap toggle, a `clean:dist` script, and Gateway debugging docs.

## Best Possible Solution

Land the DX improvement after the rebuild launch uses a supported VS Code pnpm/program shape, the docs describe the full-build prelaunch task accurately, and an Unreleased changelog bullet is added.

## Reproduction Assessment

Yes for source-level review: the latest PR head has `runtimeExecutable: "pnpm"` with the subcommand in `args` and no `program`, which conflicts with VS Code's documented Node launch contract. I did not run a live VS Code debug session.

## Solution Assessment

No. Moving the prelaunch task to a full build is the right direction for generated assets, but the one-click launch should pass the pnpm subcommand via `runtimeArgs` or launch `openclaw.mjs` directly, with matching docs and changelog before merge.

  - body: The one-click config sets `runtimeExecutable` to `pnpm` but leaves `run openclaw gateway run` in `args` with no `program`. VS Code's Node debugger treats `args` as arguments to the debugged program and runtime-tool arguments as `runtimeArgs`, so this can start the wrong command path and leave the advertised launch unable to start the Gateway.

  - confidence: 0.78
