## Summary

Keep open: current main still lacks the proposed DX support, and the PR’s one-click debug launch still has a source-proven blocker because it deletes `dist/` before a narrower `run-node` rebuild that does not recreate export-session templates.

## What This Changes

The PR adds VS Code launch/tasks files, an opt-in `OUTPUT_SOURCE_MAPS` tsdown source-map toggle, a `clean:dist` package script, and VS Code debugging docs.

## Best Possible Solution

Land the DX support after the launch task uses a supported source-map build path or preserves/recreates generated runtime assets, with docs aligned to that supported flow.

## Reproduction Assessment

Yes for the introduced regression by source inspection: start from a clean `dist`, use `Rebuild and Debug Gateway`, then exercise `/export-session`; the PR removes `dist` and the follow-up `run-node` path does not copy export HTML templates.

## Solution Assessment

No. The source-map toggle and VS Code docs are reasonable, but deleting all generated output before a partial runtime rebuild is not the narrowest maintainable one-click debug path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve export templates in clean debug launches:** `.vscode/launch.json:8`
  - body: The new prelaunch task deletes all of `dist/` and then starts `pnpm run openclaw -- gateway run`. That runner only does the TypeScript build plus runtime postbuild work, while `copy-export-html-templates` is a separate full-build step and `/export-session` reads those templates from `dist/export-html`. A gateway started through this advertised clean debug launch can therefore fail with missing template files during the debug session.
  - confidence: 0.9
