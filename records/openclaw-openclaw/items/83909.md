## What This Changes

Align `cron edit --timeout-seconds` parsing with `cron add` by using the shared positive-integer parser for timeout values.

## Best Possible Solution

Review and land the narrow parser-parity fix in https://github.com/openclaw/openclaw/pull/84031, or close that PR and replace it with equivalent focused CLI edit coverage if maintainers prefer.

## Reproduction Assessment

Yes for negative values by source inspection: `--timeout-seconds -5` parses to `-5`, passes the truthy finite check, and is assigned into the cron update payload. The reported `0` path is not reproduced because current main omits zero rather than forwarding it.

## Solution Assessment

Yes: reusing `parsePositiveIntOrUndefined` in `register.cron-edit.ts` is the narrowest maintainable fix because `cron add` already uses that helper. The fix should be kept at the CLI parity boundary and not alter runtime `timeoutSeconds: 0` semantics.
