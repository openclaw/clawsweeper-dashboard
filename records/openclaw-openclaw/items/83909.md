## Summary

Keep open: current main still parses `cron edit --timeout-seconds` differently from `cron add`, and a negative value is source-reproducible as a cron update payload value. The linked closing PR is open but not merged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Align `cron edit --timeout-seconds` parsing with `cron add` so non-positive values are omitted instead of patched into gateway payloads.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84031 or an equivalent patch that reuses `parsePositiveIntOrUndefined` in cron edit, adds focused regression coverage, and preserves runtime no-timeout semantics for stored zero.

## Reproduction Assessment

Yes, at source level: `--timeout-seconds -5` parses to `-5`, passes the finite/truthy presence check, and is assigned into the cron update payload. The specific zero claim is not reproduced from source because `0` is omitted rather than forwarded.

## Solution Assessment

Yes, reusing `parsePositiveIntOrUndefined` in cron edit is the narrowest maintainable fix because cron add already uses that helper. The linked PR follows that direction, but the issue should remain open until the PR lands or is replaced.
