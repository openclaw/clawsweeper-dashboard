## Summary

Keep open: current main and the latest release still have the cron edit parser mismatch, and the remaining bug is source-reproducible for negative timeout values. The issue should stay open because https://github.com/openclaw/openclaw/pull/84031 is an open closing PR for the same fix path.

## What This Changes

The issue asks for `cron edit --timeout-seconds` to use the same positive-integer validation as `cron add`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/84031 with cron edit parser alignment and direct CLI or payload regression proof, while preserving runtime `timeoutSeconds: 0` as no timeout.

## Reproduction Assessment

Yes, at source level: current main parses `--timeout-seconds -5` to `-5`, marks it present, and assigns it into the cron update payload. The `0` example is not reproduced because current source omits zero rather than forwarding it.

## Solution Assessment

Yes, reusing `parsePositiveIntOrUndefined` in cron edit is the narrow maintainable fix because cron add already uses that helper. The open PR is the current candidate, but it should keep runtime timeout semantics intact and prove the changed CLI path.
