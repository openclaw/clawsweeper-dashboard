## Summary

Keep open: current main still parses `cron edit --timeout-seconds` differently from `cron add`, and a negative value is source-reproducible as a gateway patch value. The `0` example in the report is not supported by current source because `0` is omitted, but the negative-value bug is real and narrow.

## What This Changes

The issue asks for `openclaw cron edit --timeout-seconds` to use the same positive-integer parsing semantics as `openclaw cron add`.

## Best Possible Solution

Reuse the existing positive-int parser in `register.cron-edit.ts`, add focused cron edit regression tests for negative, zero, and positive values, and leave the lower-level cron payload contract unchanged.

## Reproduction Assessment

Yes for negative values from source inspection: `--timeout-seconds -5` parses to `-5`, passes the edit command's finite/truthy check, and is assigned into the cron update payload. The specific `0` claim is not reproduced from source because `0` is omitted rather than forwarded.

## Solution Assessment

Yes: reusing `parsePositiveIntOrUndefined` in cron edit is the narrowest maintainable fix because cron add already uses it and the shared helper has explicit tests. The safer boundary is CLI parity only, not changing runtime support for `timeoutSeconds: 0` in cron payloads.
