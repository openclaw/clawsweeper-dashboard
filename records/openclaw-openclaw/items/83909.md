## Summary

Close as implemented: current main rejects invalid `cron edit --timeout-seconds` values before building the gateway patch, and the fix is included in release `v2026.6.6`. The related unmerged PR was already closed after the same implemented-on-main proof, so no remaining OpenClaw source work is needed for this issue.

## What This Changes

The issue asks OpenClaw to align `cron edit --timeout-seconds` validation with `cron add` so non-positive or malformed values are not forwarded to the gateway.

## Best Possible Solution

Keep the shipped strict CLI validation from `v2026.6.6`; the lower-level zero-timeout runtime contract should remain separate from this CLI parser parity issue.

## Reproduction Assessment

No for current main: invalid `cron edit --timeout-seconds` values now throw before `cron.update` can receive a bad timeout. The old negative-value forwarding path was source-reproducible before commit `459cee53153c50deaaefb6f408dbd4eba3558489`.

## Solution Assessment

Yes: the shipped strict validation is the right fix boundary because it aligns CLI edit with CLI add without changing the gateway/runtime meaning of `timeoutSeconds: 0`.
