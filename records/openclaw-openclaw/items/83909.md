## Summary

Keep open: current main still parses `cron edit --timeout-seconds` differently from `cron add`, and a negative value is source-reproducible as a cron update payload value. The issue should remain open because https://github.com/openclaw/openclaw/pull/84031 is an open closing PR for this exact repair.

## What This Changes

The issue asks to align `openclaw cron edit --timeout-seconds` with `cron add` by using the shared positive-integer parser.

## Best Possible Solution

Review and resolve https://github.com/openclaw/openclaw/pull/84031, keeping the fix scoped to cron edit CLI parser parity and adding direct edit-path regression coverage if maintainers want stronger proof.

## Reproduction Assessment

Yes. Source inspection shows `--timeout-seconds -5` parses to `-5`, passes the current truthy finite check, and is assigned into the `cron.update` payload; the `0` example is not reproduced because current source omits zero.

## Solution Assessment

Yes. Reusing `parsePositiveIntOrUndefined` in cron edit is the narrow maintainable fix because cron add already uses that parser and the runtime zero-timeout contract should remain unchanged.
