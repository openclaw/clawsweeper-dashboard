## Summary

Current main already implements the requested inbound Telegram media path: Telegram downloads successful photo/sticker media, carries saved paths through `MediaPath`/`MediaPaths`, stages them into `media/inbound` for sandboxed turns, and prepends media attachment notes before the agent prompt.

## What This Changes

The issue requests that inbound Telegram images and stickers be exposed to the agent as usable media attachments/paths instead of only a `<media:image>` placeholder.

## Best Possible Solution

Keep the current main behavior that downloads Telegram media, carries `MediaPaths`, stages inbound files into `media/inbound`, and surfaces prompt media notes; any new report should include current-build logs if that path fails in a release.

## Reproduction Assessment

No, not as a current-main failure. Source inspection and focused regression tests show the successful Telegram photo path now carries `MediaPaths` and stages them for the agent, but I did not run a live Telegram bot session.

## Solution Assessment

Yes. The current implementation keeps the fix in the owning Telegram plugin for download/context metadata and in the shared reply pipeline for sandbox staging/prompt notes, which is the narrow maintainable boundary.
