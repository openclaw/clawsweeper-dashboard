## Summary

This PR should stay open: it fixes the main Codex app-server attempt path, but `/btw` side threads still fork Codex with native Code Mode/current native tool surface, leaving a sandboxed Codex execution path outside the new fail-closed contract. The compatibility and security-boundary policy also needs explicit maintainer/security approval before merge.

## What This Changes

The PR disables the main Codex app-server native tool surface during active OpenClaw sandboxed turns, exposes `sandbox_exec`/`sandbox_process`, reprojects transient thread context, and updates Codex sandbox docs/tests/changelog.

## Best Possible Solution

Extend the fail-closed Codex sandbox contract to every app-server execution path reachable under active OpenClaw sandboxing, including `/btw` side threads, then land with explicit maintainer/security approval and upgrade-facing docs.

## Reproduction Assessment

Yes, at source level: current main maps active OpenClaw sandbox state into a Codex host-side `workspaceWrite` policy, matching the reported gateway-host execution path. I did not rerun the Docker canary in this read-only review.

## Solution Assessment

No, not yet: the main attempt-path fail-closed shape is maintainable, but the unchanged Codex side-question path still forks with native Code Mode/current native surface under the same session.
