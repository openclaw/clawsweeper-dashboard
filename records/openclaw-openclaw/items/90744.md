## Summary

Keep open: the PR moves the guard to the right dispatcher boundary, but it is not merge-ready because protected actions can still receive a supplied requesterSenderId when toolContext.currentChannelProvider is absent, and the external proof is mock-only.

## What This Changes

The PR adds a dispatcher-level trusted-sender fallback for canonical channel-management actions, adds Discord moderation actions to the trusted-sender set, and updates focused security tests.

## Best Possible Solution

Fail closed for protected actions unless requesterSenderId is paired with same-channel provider context, or make Discord moderation ignore requesterSenderId without that proof, then add focused gateway regression coverage and real redacted Discord proof.

## Reproduction Assessment

Yes from source inspection: a message.action request can include requesterSenderId without toolContext, the gateway forwards it, the PR fallback does not require trust in that case, and Discord moderation forwards it as senderUserId. I did not execute it because this review is read-only.

## Solution Assessment

No. The PR is a useful mitigation, but the best fix must bind requesterSenderId to same-channel provenance across both local tool dispatch and gateway message.action paths.
