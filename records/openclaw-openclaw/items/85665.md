## Label Justifications

- `P1`: A schema-valid tool deny policy can be silently ignored on a real channel path, breaking a safety control operators may rely on.
- `impact:security`: The report concerns enforcement of configured tool restrictions, including denying web fetch and session-spawn capabilities in DMs.

## Summary

Keep open: current main still exposes `tools`/`toolsBySender` on Telegram direct config while the tool-policy resolver path only consults group config, so the reported schema/runtime divergence is credible and safety-relevant.

## What This Changes

The issue asks OpenClaw to enforce schema-valid per-DM channel tool policies such as `channels.telegram.direct.<id>.tools.deny`.

## Best Possible Solution

Honor existing `direct.<peer>.tools` and `direct["*"].tools/toolsBySender` through a direct-chat-aware policy path that keeps DM and group matching separate, with regression coverage for Telegram DM deny/allow, wildcard direct policy, and group isolation.

## Reproduction Assessment

Yes at source level: configure `channels.telegram.direct.<id>.tools.deny`, then follow the current resolver path from Telegram direct config to `resolveGroupToolPolicy`, which never reads `direct`; I did not execute the live Telegram scenario.

## Solution Assessment

Is this the best way to solve the issue? Not applicable as there is no patch yet; the maintainable fix should make the existing direct config contract enforceable without merging DM and group namespaces.
