## Summary

Keep open. The PR targets a real current-main WebChat render path, but it lacks after-fix real behavior proof and the current diff misses paths that still schedule renders or need explicit refreshes once the draft becomes non-reactive.

## What This Changes

The PR removes Lit `@state()` from `OpenClawApp.chatMessage` and adds explicit `requestUpdate()` calls after abort, failed-send draft restore, and legacy inline history recall in Control UI WebChat.

## Best Possible Solution

Land a current-main WebChat fix that avoids host renders for ordinary text input while explicitly refreshing programmatic draft changes and preserving slash-menu and history behavior with focused coverage.

## Reproduction Assessment

Yes, at source level. Current main still wires textarea input to `onDraftChange`, which assigns `chatMessage` while it is Lit `@state()`, matching the reported per-keystroke host-render path; I did not run a live browser performance capture.

## Solution Assessment

No as-is. Making the draft non-reactive is a plausible direction, but this branch also needs to avoid unconditional slash-menu renders, refresh the current history-key path, call the update hook safely, and provide real after-fix behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Call the optional update hook safely:** `ui/src/ui/app-chat.ts:83-86`
  - body: `ChatHost` exposes `requestUpdate` as optional, and the test host helper omits it by default. Calling it through an `OpenClawApp` cast can throw for non-app hosts on abort or failed-send restore; use `host.requestUpdate?.()` or make the contract required with matching tests.
  - confidence: 0.88
- **[P2] Avoid rendering plain text through slash updates:** `ui/src/ui/app.ts:157-159`
  - body: Removing `@state()` from `chatMessage` does not stop ordinary typing from scheduling host renders while `handleInput` still calls `updateSlashMenu(...)`, because that helper unconditionally calls `requestUpdate()` for non-command text. Guard the slash-menu refresh so plain text only requests an update when visible slash state changes.
  - confidence: 0.85
- **[P2] Refresh the current history-key path:** `ui/src/ui/views/chat.ts:1068-1080`
  - body: The branch adds `requestUpdate()` to the old inline `inputHistory.up()/down()` block, but current main now routes ArrowUp/ArrowDown through `props.onHistoryKeydown` and `navigateChatInputHistory`, which mutates `chatMessage`. After the draft becomes non-reactive, that current path also needs an explicit update or recalled history can stay out of the textarea.
  - confidence: 0.87
