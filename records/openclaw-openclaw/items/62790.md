## Summary

Keep open. The PR targets a valid WebChat typing regression, but the current diff is not ready to merge because it still leaves source-level per-keystroke host updates, misses the current-main history-recall path after refactors, and has no after-fix real behavior proof.

## What This Changes

The PR removes Lit `@state()` from `chatMessage` and adds explicit `requestUpdate()` calls for abort, failed-send restore, and the older inline history-recall path in Control UI WebChat.

## Best Possible Solution

Land a current-main WebChat fix that eliminates host renders for ordinary text input while explicitly refreshing programmatic draft changes and preserving history/slash behavior with focused tests.

## Reproduction Assessment

Yes, at source level. Current main still wires each textarea input to `onDraftChange`, which assigns `chatMessage` while it is a Lit `@state()`, matching the open WebChat keystroke-delay report; I did not run a live browser performance capture.

## Solution Assessment

No as-is. Removing `@state()` is a plausible direction, but this branch must also avoid the unconditional slash-menu `requestUpdate()`, refresh the current `onHistoryKeydown` recall path, and provide real after-fix behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Call the optional update hook safely:** `ui/src/ui/app-chat.ts:83-86`
  - body: `ChatHost` only exposes `requestUpdate` as optional, and the test host helper does not provide it by default. Calling it through an `OpenClawApp` cast can throw for non-app hosts on abort or failed-send restore; use the host hook safely or make the contract required with matching tests.
  - confidence: 0.88
- **[P2] Stop slash-menu updates from rendering plain text:** `ui/src/ui/app.ts:157-159`
  - body: Removing `@state()` from `chatMessage` does not stop ordinary typing from scheduling host renders because `handleInput` still calls `updateSlashMenu(...)`, and that helper unconditionally calls `requestUpdate()` even for plain text. Guard the slash-menu update so regular typing only requests a render when slash-menu state actually changes.
  - confidence: 0.85
- **[P2] Refresh the current history-key path:** `ui/src/ui/views/chat.ts:1068-1080`
  - body: Current main no longer uses the inline `inputHistory.up()/down()` path updated here; Arrow recall now goes through `onHistoryKeydown` and `navigateChatInputHistory`, which mutates `chatMessage`. After the draft becomes non-reactive, that current path also needs an explicit update or the textarea can remain stale.
  - confidence: 0.87
