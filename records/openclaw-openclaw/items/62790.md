## Summary

Keep open. The PR targets a real WebChat typing/render problem, but the current patch is not merge-ready: it misses a remaining per-keystroke slash-menu render path, edits an older history-recall path instead of the current one, uses an optional update hook unsafely, and has no contributor-supplied real behavior proof.

## What This Changes

The PR removes Lit `@state()` from the WebChat draft field and adds explicit `requestUpdate()` calls for abort, failed-send restore, and older history-recall paths.

## Best Possible Solution

Land or replace a current-main WebChat fix that keeps ordinary draft typing out of Lit state, avoids slash-menu renders for plain text, preserves programmatic draft/history sync, and includes real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main still wires textarea input to `onDraftChange`, assigns `chatMessage` while it is Lit `@state()`, and also calls a slash-menu helper that requests updates for plain text; I did not run a live browser performance capture.

## Solution Assessment

No as-is. Making the draft non-reactive is a plausible direction, but this patch needs current-main path coverage, safer update-hook usage, the remaining slash-menu render path addressed, and real behavior proof before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard slash-menu renders for plain text:** `ui/src/ui/app.ts:157-159`
  - body: Removing `@state()` from `chatMessage` does not stop ordinary typing from scheduling host renders because `handleInput` still calls `updateSlashMenu(...)`, and that helper still calls `requestUpdate()` for normal non-command text. Guard the slash-menu update so regular typing only renders when slash-menu state actually changes.
  - confidence: 0.9
- **[P2] Refresh the current history-key path:** `ui/src/ui/views/chat.ts:1068-1077`
  - body: Current main no longer uses the inline `inputHistory.up()/down()` path updated here; Arrow recall now goes through `onHistoryKeydown` and `navigateChatInputHistory`, which mutates `chatMessage`. After the draft becomes non-reactive, that current path also needs an explicit refresh or the textarea can remain stale.
  - confidence: 0.88
- **[P2] Call the optional update hook safely:** `ui/src/ui/app-chat.ts:83-86`
  - body: `ChatHost` still declares `requestUpdate` as optional, and test or non-app hosts can omit it. Calling it through an `OpenClawApp` cast can throw when abort or failed-send restore tries to refresh the draft; use the optional hook safely or make the contract required with matching tests.
  - confidence: 0.86
- **[P2] Refresh detached send draft clears:** `ui/src/ui/app-chat.ts:132`
  - body: The patch adds explicit refreshes for a few draft writes but misses the detached `/btw` path: `clearSubmittedComposerState` clears the draft before `sendDetachedBtwMessage`, and a successful detached send does not necessarily mutate another reactive chat property. With `chatMessage` non-reactive, that can leave the submitted text visible in the textarea.
  - confidence: 0.78
