## Summary

Keep open: this is a focused fix direction for the open WebChat typing regression, but the current PR is conflicting, lacks real browser behavior proof, and still misses source-level render/update paths needed after making the draft non-reactive.

## What This Changes

This PR removes Lit `@state()` from `OpenClawApp.chatMessage` and adds explicit `requestUpdate()` calls for abort, failed-send restore, and older inline input-history recall paths.

## Best Possible Solution

Land or replace a current-main WebChat fix that keeps ordinary typing out of Lit host state, guards slash-menu updates, preserves programmatic draft/history sync, and includes focused tests plus real browser proof.

## Reproduction Assessment

Yes, at source level: current main still wires textarea input to `updateSlashMenu` and `onDraftChange`, and `chatMessage` remains Lit `@state()`. I did not run a live browser timing trace in this read-only review.

## Solution Assessment

No as-is. Making the draft non-reactive is a plausible direction, but the PR must be rebased and repaired for current slash-menu, history, optional update-hook, and detached draft-clear paths before merge.
