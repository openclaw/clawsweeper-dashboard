## What This Changes

The branch changes `ui/src/ui/app.ts`, `ui/src/ui/app-chat.ts`, and `ui/src/ui/views/chat.ts` to make `chatMessage` non-reactive and request updates on some programmatic draft mutations.

## Best Possible Solution

Land or replace this with a current-main WebChat fix that keeps ordinary typing outside host state, guards slash-menu refreshes, covers all programmatic draft mutations, and includes focused tests plus real browser proof.

## Reproduction Assessment

Yes, at source level: current main routes textarea input through `updateSlashMenu` and `onDraftChange`, and `chatMessage` is still Lit `@state()`. I did not run a live browser timing trace because this review is read-only.

## Solution Assessment

No, not as-is. Making the draft non-reactive is a plausible direction, but the branch must be rebased and repaired for current slash-menu, history, optional update-hook, and draft-clear paths before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard slash-menu updates before dropping draft state:** `ui/src/ui/app.ts:208`
  - body: Removing `@state()` from `chatMessage` does not stop ordinary typing from scheduling renders because `handleInput` still calls `updateSlashMenu`, whose plain-text path still calls `requestUpdate()` on every keystroke. Guard that refresh so it only fires when slash-menu state actually changes.
  - confidence: 0.9
- **[P2] Refresh the current history navigation path:** `ui/src/ui/views/chat.ts:1267-1288`
  - body: The added refreshes target an older inline history branch. Current main routes ArrowUp and ArrowDown through `onHistoryKeydown` and `navigateChatInputHistory`, so that path also needs a refresh after `chatMessage` becomes non-reactive or recalled drafts can stay invisible.
  - confidence: 0.88
- **[P2] Respect the optional update hook contract:** `ui/src/ui/app-chat.ts:71`
  - body: `ChatHost` declares `requestUpdate` as optional, but the patch casts hosts to `OpenClawApp` and calls it directly during abort and failed-send restore. Use `host.requestUpdate?.()` or make the contract required with matching tests so non-app hosts do not throw.
  - confidence: 0.86
- **[P2] Refresh detached-send draft clears:** `ui/src/ui/app.ts:208`
  - body: Once `chatMessage` is non-reactive, `clearSubmittedComposerState` can clear a detached-send draft without forcing the DOM to update. Add an explicit refresh for that clear path so submitted text does not remain visible.
  - confidence: 0.78
