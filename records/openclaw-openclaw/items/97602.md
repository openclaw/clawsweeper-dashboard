## Summary

Keep open: current main and v2026.6.10 still route local assistant identity avatars from `agent.identity.get` through an auth-gated `/avatar/<agentId>` URL while the `agents.list` path can inline workspace avatars, so the reported Personal card 401 remains actionable.

## What This Changes

The issue requests consistent Control UI avatar resolution so the Personal/Identity card receives a browser-renderable avatar for workspace-relative agent avatars.

## Best Possible Solution

Share or extract the existing workspace-safe avatar data-URI resolution so `agent.identity.get` and `agents.list` agree for local workspace avatars, without weakening `/avatar/<agentId>` auth or adding new config.

## Reproduction Assessment

Yes by source inspection: configure a workspace-relative agent avatar, then `agent.identity.get` returns `/avatar/<agentId>` while authenticated avatar HTTP requests are required by the gateway. The same avatar source is inlined as a data URI by `agents.list`.

## Solution Assessment

Yes, the narrowest maintainable fix is to align the identity RPC with the existing workspace-safe data-URI resolver. Changing `/avatar/<agentId>` to cookie/session auth would be broader and security-sensitive.
