## Summary

Current main still has the forced-restart 0ms reply-drain path, and the direct fix PR is open but unmerged, so this issue should remain open as the focused tracker.

## What This Changes

The issue reports that forced Gateway restart/SIGUSR1 can use a 0ms reply drain and silently discard pending outbound replies during shutdown.

## Best Possible Solution

Land or replace the focused Gateway run-loop fix so forced restarts preserve pending replies within an owner-approved bounded drain policy, while leaving broader durable replay and self-restart policy to their separate trackers.

## Reproduction Assessment

Yes. Source inspection shows forced restart returns a 0ms drain budget, skips active-work waits, and server-close immediately treats timeoutMs <= 0 as undrained; I did not run a live channel restart in this read-only review.

## Solution Assessment

Unclear until maintainer review of the linked PR. A bounded non-zero forced-restart drain is a narrow plausible fix, but maintainers need to confirm the intended --force compatibility semantics.
