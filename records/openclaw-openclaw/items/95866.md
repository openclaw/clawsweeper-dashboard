## Summary

Keep open: current main and the latest release still give forced restarts a 0ms restart/reply drain budget, while changing that intersects the documented `--force` immediate-override behavior.

## What This Changes

The issue reports that forced Gateway restarts can discard queued in-flight replies because restart shutdown uses a 0ms reply-drain timeout.

## Best Possible Solution

Keep this as the narrow forced-restart reply-loss tracker and choose a bounded policy, such as preserving active-work abort semantics while reserving a small outbound-reply drain for already queued deliveries, or explicitly documenting that forced restarts may drop replies.

## Reproduction Assessment

Yes, at source level: current main maps forced restart to a 0ms restart drain, then the restart close path uses that 0ms budget for pending reply drain. I did not run a live Gateway restart in this read-only review.

## Solution Assessment

Unclear as a product decision: the maintainable code boundary is narrow, but maintainers need to decide whether `--force` still means immediate override or should reserve a small drain for already queued outbound replies.
