## Summary

The stale-runtime failure is real and the latest head fixes its targeted branches with strong live proof, but this PR should remain open for explicit gateway/config ownership review because it adds a public option, makes credential edits override hot-mode restart policy, and depends on supervisor respawn behavior.

## What This Changes

The PR adds `gateway.reload.autoRestartOnRequired`, preserves routine hot-mode warn-and-keep by default, automatically restarts for gateway-auth and provider-secret rotations, changes `secrets.providers.*` reload planning, and updates schema, docs, and tests.

## Best Possible Solution

Choose one owner-approved canonical policy: either land the opt-in plus narrowly defined credential carve-out with supported-supervisor upgrade proof, or keep the existing mode set and route known doctor/configure migrations through the existing writer-owned restart-intent seam.

## Reproduction Assessment

Yes. Current-main source and the supplied production logs establish that a restart-required write in hot mode can leave disk and live runtime state divergent until a manual restart.

## Solution Assessment

Unclear. The patch addresses the observed failure and preserves routine upgrade behavior, but adding a public option and mandatory credential-triggered restarts is a product and compatibility decision rather than the uniquely best repair.
