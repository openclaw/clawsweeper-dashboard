## Summary

Keep open. The PR addresses a real stale-runtime failure mode, but it still redefines the shipped `gateway.reload.mode: "hot"` contract into an automatic restart path and the supplied live proof does not prove supervisor respawn after that clean restart handoff.

## What This Changes

The PR changes hot-mode gateway config reloads so restart-required file changes call the existing restart path, adds reload regression coverage, and updates gateway reload docs/help.

## Best Possible Solution

Preserve the shipped no-automatic-restart meaning of `hot` unless maintainers explicitly approve a contract change; route known unsafe OpenClaw-owned writes through explicit restart intent or keep that behavior under `hybrid`/`restart` with matching docs and proof.

## Solution Assessment

No. Scheduling restart from `hot` is a plausible mitigation, but the best maintainable fix needs a maintainer decision on the `hot` contract and likely should use explicit writer restart intent for known unsafe writes.
