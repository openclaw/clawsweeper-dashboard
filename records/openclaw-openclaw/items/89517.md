## Summary

Keep open: the branch addresses a real gateway stale-runtime problem, but the new hot-mode opt-in is only applied to the file-watcher reload path while `config.patch` and `config.apply` can still schedule routine restart-required hot-mode restarts with the option unset.

## What This Changes

The PR adds `gateway.reload.autoRestartOnRequired`, hot-mode auth/secrets restart carve-outs, `secrets.providers` restart planning, config schema/help/docs updates, and gateway reload regression tests.

## Best Possible Solution

Centralize the hot-mode restart-required decision and apply the same opt-in/security split to both file-watcher reloads and config-write RPCs, with explicit maintainer sign-off on the new config contract before merge.

## Reproduction Assessment

Yes, source-level: `config.patch` and `config.apply` route through `shouldScheduleDirectConfigRestart`, which still returns true for any hot-mode restart-required plan without checking the new opt-in or security split.

## Solution Assessment

No. The file-watcher fix is a plausible part of the solution, but the maintainable fix needs one shared restart-policy predicate across watcher and config-write paths.
