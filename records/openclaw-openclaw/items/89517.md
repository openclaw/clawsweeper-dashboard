## Label Justifications

- `P2`: The PR addresses a credible gateway stale-state bug but requires a bounded product and compatibility redesign rather than emergency handling.
- `merge-risk: 🚨 compatibility`: The patch adds a new reload setting and changes the established no-automatic-restart behavior of existing hot-mode installations.
- `merge-risk: 🚨 auth-provider`: Provider SecretRef configuration updates are newly treated as credential rotations requiring restart, which can alter auth/provider behavior.
- `merge-risk: 🚨 availability`: New automatic restart paths can leave gateways stopped when their supervisor does not respawn clean exits.

## Summary

The PR addresses a real hot-reload stale-state failure and supplies strong runtime proof, but its final design is not the best fix: it adds a second control for behavior already provided by `hybrid`, overrides the documented `hot` no-restart contract for broad path prefixes, and treats provider SecretRef configuration metadata as if it were the resolved credential value. The draft should stay open for a maintainer decision and a narrower redesign.

## What This Changes

Adds `gateway.reload.autoRestartOnRequired`, preserves routine hot-mode warn-and-keep by default, forces hot-mode restarts for `gateway.auth.*` and `secrets.*` restart reasons, reclassifies `secrets.providers.*` edits as restart-required, and updates schema, help, docs, and reload tests.

## Best Possible Solution

Keep `hot` as an explicit no-automatic-restart mode and `hybrid` as the canonical automatic-restart mode; make doctor/config writers request a restart through the existing write-follow-up seam when they perform restart-sensitive migrations, and separately design exact credential-cache invalidation or restart ownership around resolved SecretRefs rather than broad provider metadata prefixes.

## Reproduction Assessment

Yes for the central bug: current source and the supplied production logs show that hot mode can ignore a whole restart-required plan after disk config changes, leaving stale runtime state. The PR’s broader SecretRef-rotation scenario is not reproduced because provider configuration paths are not equivalent to the resolved secret values.

## Solution Assessment

No. The patch is plausible but duplicates `hybrid`, weakens the explicit hot-mode operator contract, and places broad SecretRef policy in the gateway planner; writer restart intent plus exact credential lifecycle handling is narrower and more maintainable.

- **[P1] Remove the duplicate automatic-restart mode:** `src/config/types.gateway.ts:275-283`
  - body: `gateway.reload.mode: "hybrid"` already hot-applies safe changes and restarts when required. Adding `autoRestartOnRequired` under `hot` creates a second configuration path with the same restart decision, so behavior, docs, and future fixes can drift. Use `hybrid` for gateway-owned automatic restart, or route known writers through the existing explicit restart follow-up seam.
  - confidence: 0.94
- **[P1] Do not equate provider metadata edits with credential rotation:** `src/gateway/config-reload-plan.ts:138`
  - body: `secrets.providers.*` config defines resolver metadata such as command, args, env, allowlists, and timeouts; the referenced credential is identified separately by each SecretRef and can change without this config path changing. Restarting for every provider metadata edit both over-restarts and does not generally solve secret-value rotation. Tie invalidation/restart to resolved credential cache ownership or exact restart-sensitive writer actions instead.
  - confidence: 0.9
- **[P1] Preserve the explicit hot-mode no-restart contract:** `src/gateway/config-reload.ts:326-347`
  - body: The new security branch automatically restarts existing `mode: "hot"` installations for broad path prefixes even though hot mode is the operator’s no-automatic-restart choice and the supplied proof shows some supervisors do not respawn clean exits. This can turn a config edit into an outage. Keep hot fail-loud/no-restart unless maintainers explicitly approve a breaking policy and upgrade contract.
  - confidence: 0.93
