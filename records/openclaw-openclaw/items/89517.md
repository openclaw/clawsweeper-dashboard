## Summary

Keep this PR open, but do not merge as-is: it fixes a real stale-reload path by redefining explicit `hot` mode to restart, which removes the documented distinction from `hybrid` and can surprise existing operators.

## What This Changes

The PR changes gateway hot reload behavior so restart-required plans in hot mode schedule the existing restart path, with gateway docs and regression tests updated.

## Best Possible Solution

Keep a distinct no-restart `hot` mode and fix stale runtime cases through `hybrid`, explicit writer follow-up restart intent, or a maintainer-approved compatibility path that does not silently make `hot` behave like `hybrid`.

## Reproduction Assessment

Yes from source inspection: current main reaches the hot-mode branch for restart-required plans and returns without calling `queueRestart`. I did not execute tests because this review is read-only.

## Solution Assessment

No; the proposed fix solves stale restart-required file edits by changing the existing `hot` mode contract. The safer fix is to preserve `hot` as explicit no-restart behavior and use `hybrid` or explicit writer restart intent for automatic restarts.

Full review comments:

- **[P1] Preserve explicit hot mode as no-restart mode:** `src/gateway/config-reload.ts:283`
  - body: This fall-through makes `gateway.reload.mode: "hot"` queue the same restart path as `hybrid` whenever a restart-required path changes. Current main and the public gateway docs define `hot` as applying only hot-safe changes, while `hybrid` is the mode that restarts when required; existing operators who chose `hot` to avoid process restarts would now get surprise restarts on config edits. Please keep `hot` no-restart and route the stale-runtime case through `hybrid` or an explicit writer restart intent instead.
