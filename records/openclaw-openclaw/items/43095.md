## Summary

Keep open. Current main still matches the reported ambiguity: after the target run is accepted, a reply-wait timeout can still surface from `sessions_send` as a bare `status: "timeout"`, while the overlapping fixes in #58133 and #68196 remain open rather than merged.

## What This Changes

The issue requests that `sessions_send` distinguish target acceptance/delivery from synchronous reply-wait timeout and real delivery failure.

## Best Possible Solution

Land one focused contract change that reports target acceptance/delivery separately from reply-wait completion, adds regression coverage for accepted-then-timeout behavior, and updates session/A2A docs if the public result shape changes.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to mock `agent` as accepted and `agent.wait` as `status: "timeout"`; current `sessions_send` then returns a bare timeout even though the target run was already accepted.

## Solution Assessment

No for current main. The narrow maintainable fix is an additive accepted/pending or delivery-state response for the accepted-then-wait-timeout path, with richer model-error diagnostics handled separately if needed.
