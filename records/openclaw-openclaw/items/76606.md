## Summary

Keep open. Current main still preserves an explicit empty cron `toolsAllow` list as a disable-all request, but the embedded runner treats `toolsAllow: []` as unrestricted and constructs tools before filtering; merged PR #76773 only reduces denied optional media factory cost.

## What This Changes

The issue reports slow per-turn `core-plugin-tools` preparation and asks for an explicit no-tools or pre-construction allowlist path for embedded `agentTurn` runs.

## Best Possible Solution

Honor the existing empty `toolsAllow` disable-all contract by short-circuiting embedded tool construction for explicit empty cron/runtime allowlists, while keeping broader factory caching and lazy-loading work separate.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: cron preserves `toolsAllow: []`, forwards it to the embedded runner, and the runner treats the empty list as unrestricted after constructing tools. I did not run a live latency probe.

## Solution Assessment

Yes for the narrow first fix: map explicit empty cron/runtime allowlists to the existing `disableTools` short-circuit or distinguish `undefined` from `[]` before `createOpenClawCodingTools`. Broader plugin factory caching should remain a separate performance follow-up.
