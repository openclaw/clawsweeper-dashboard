## Summary

Keep open. Current main still treats `toolsAllow: []` as unrestricted after building the core/plugin tool set, so the requested explicit no-tools `agentTurn` path is not implemented; the related merged work only reduces selected optional factory costs.

## What This Changes

The issue asks for explicit no-tools or earlier allowlist handling for cron/embedded `agentTurn` tool construction to avoid repeated `core-plugin-tools` latency.

## Best Possible Solution

Honor explicit empty cron/runtime allowlists before tool construction, using the existing no-tools short-circuit path, and leave broader lazy-loading/caching work to separate performance follow-ups.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: cron preserves and forwards `toolsAllow: []`, while the embedded runner builds tools and treats the empty list as unrestricted. I did not run a live timing probe.

## Solution Assessment

Yes for the narrow fix: distinguish explicit empty allowlists from missing allowlists before `createOpenClawCodingTools(...)`, or translate them to `disableTools: true` for embedded cron turns. Do not bundle broad plugin factory memoization into this repair.
