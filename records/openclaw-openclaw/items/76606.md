## Summary

Keep open. Current main still treats explicit empty `toolsAllow` lists as unrestricted in the embedded runner after preserving and forwarding them from cron, so the requested no-tools/pre-construction short-circuit is not implemented; merged PR #76773 only narrows optional media factory work.

## What This Changes

The issue reports that embedded `agentTurn` preparation constructs core/plugin tools for explicit empty `toolsAllow` lists, causing high per-turn latency and leaving no supported no-tools mode.

## Best Possible Solution

Honor explicit empty cron/runtime allowlists before tool construction, ideally by routing that case through the existing no-tools short-circuit while keeping broader factory caching and lazy-loading as separate follow-ups.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current cron code preserves and forwards an explicit empty `toolsAllow` array, and the embedded runner treats that empty array as unrestricted after building tools.

## Solution Assessment

Yes for a narrow first fix: distinguish explicit `[]` from `undefined` before `createOpenClawCodingTools(...)` or map it to existing `disableTools` behavior for embedded cron/runtime turns. Broader plugin factory caching should remain separate.
