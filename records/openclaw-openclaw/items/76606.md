## Summary

Keep open. Current main still preserves empty cron `toolsAllow` lists as a disable-all request but the embedded runner treats `toolsAllow: []` as unrestricted and constructs tools before filtering; merged PR #76773 only reduces optional media/PDF factory work.

## What This Changes

The issue requests that cron/runtime `agentTurn` honor an explicit no-tools intent and avoid expensive core/plugin tool construction when `toolsAllow: []` or an equivalent mode is used.

## Best Possible Solution

Honor explicit empty cron/runtime allowlists before tool construction by using the existing no-tools short-circuit, and leave broader lazy-loading or factory caching work as separate performance follow-ups.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create or patch a cron `agentTurn` with `toolsAllow: []`; cron preserves and forwards the empty list, but the embedded runner treats it as unrestricted and builds tools before filtering.

## Solution Assessment

Yes for the narrow repair. Distinguishing undefined from explicit empty at the cron/embedded-runner boundary and reusing `disableTools` is the smallest maintainable fix; broad plugin lazy loading should stay separate.

## Repair Work Prompt

Fix the cron/embedded-runner no-tools contract. When a cron `agentTurn` payload explicitly contains `toolsAllow: []`, avoid constructing core/plugin tools for that embedded run by translating it to the existing `disableTools` path or by making the embedded runner distinguish `undefined` from an explicit empty allowlist before `createOpenClawCodingTools(...)`. Preserve the current default when `toolsAllow` is omitted, do not add a new public config flag unless the existing contract proves impossible, and do not fold broad plugin lazy-loading or caching into this PR. Add regression coverage for cron forwarding/runner behavior and update `CHANGELOG.md` because this is user-visible latency/cron behavior.
