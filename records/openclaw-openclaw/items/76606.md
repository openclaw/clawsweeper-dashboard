## Summary

Keep open. Current main still preserves explicit empty cron `toolsAllow` lists, but the embedded runner treats `toolsAllow: []` as unrestricted and constructs tools before filtering; merged PR #76773 helps optional media/PDF factory cost but does not implement the reported no-tools path.

## What This Changes

The issue asks for `agentTurn` allowlist/no-tools intent to be applied before expensive core/plugin tool construction, especially for `toolsAllow: []`.

## Best Possible Solution

Honor the existing cron `toolsAllow: []` disable-all contract by short-circuiting embedded tool construction for explicit empty allowlists, and keep broader lazy-loading or factory caching as separate follow-up work.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: cron preserves an explicit empty list, forwards it to the embedded runner, and the runner treats it as unrestricted after constructing tools. I did not run a live timing probe.

## Solution Assessment

Yes for a narrow first fix: translate explicit empty cron/runtime allowlists into the existing `disableTools` short-circuit or distinguish `undefined` from `[]` before construction. The merged optional-media factory optimization is useful but not sufficient for this issue.
