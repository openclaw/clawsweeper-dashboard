## Summary

Current main now implements the central request: explicit empty embedded-runtime `toolsAllow` lists mean no tools, skip local tool construction, and narrow allowlists are planned before unrelated tool families are built. The fix is merged and shipped in `v2026.5.4`; broader cold-start/per-turn latency remains covered by separate open performance reports.

## What This Changes

The issue asks for explicit no-tools or early allowlist handling for embedded agent turns so `core-plugin-tools` construction does not dominate per-turn preparation.

## Best Possible Solution

Keep the shipped `v2026.5.4` allowlist implementation for this issue and use the broader open performance reports for remaining cold-start, prompt, stream setup, or event-loop latency work.

## Reproduction Assessment

No on current main: the source path that treated `toolsAllow: []` as unrestricted no longer exists, and regression tests now assert explicit empty allowlists disable all local tool construction. I did not run a live timing probe, but the current source-level behavior is high-confidence.

## Solution Assessment

Yes. The merged fix uses the existing `toolsAllow` contract, distinguishes `undefined` from `[]`, and moves allowlist planning before expensive local tool family construction without adding a new public flag.
