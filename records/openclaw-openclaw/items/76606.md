## Summary

Keep open. Current main still treats an explicit empty `toolsAllow` as unrestricted and cron forwards it without `disableTools`, so the reported no-tools `agentTurn` path remains unhonored even though newer main has partial optimization for non-empty plugin-only allowlists.

## What This Changes

The issue requests a supported no-tools `agentTurn` path, or earlier allowlist application, so `toolsAllow: []` avoids per-turn tool/plugin construction latency.

## Best Possible Solution

Honor the existing cron `toolsAllow: []` disable-all contract by routing explicit empty lists to the embedded runner's no-tools path, then profile non-empty allowlist and plugin lazy-loading work separately.

## Reproduction Assessment

Yes. Source reproduction is high-confidence: a cron `agentTurn` with `toolsAllow: []` reaches the embedded runner without `disableTools`, then the runner treats the empty allowlist as unrestricted and builds tools before filtering.

## Solution Assessment

Yes for the narrow first fix. Reusing the existing `disableTools` runtime path for explicit empty cron allowlists is the smallest maintainable repair, while broader lazy-loading for non-empty allowlists should stay a separate performance follow-up.
