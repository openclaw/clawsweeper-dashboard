## Summary

Keep open. Current main still preserves explicit empty cron `toolsAllow` lists but the embedded runner treats an empty list as unrestricted and builds the normal tool set before filtering; merged PR #76773 only reduces denied optional media/PDF factory cost.

## What This Changes

The issue asks for explicit no-tools `agentTurn` behavior, or earlier allowlist application, so runs with `toolsAllow: []` do not pay full core/plugin tool construction cost.

## Best Possible Solution

Honor explicit empty cron/runtime allowlists before tool construction by mapping that case to the existing `disableTools` path, while leaving broader plugin factory caching for separate follow-ups.

## Reproduction Assessment

Yes. Source inspection on current main shows cron preserves and forwards `toolsAllow: []`, while the embedded runner treats the empty list as unrestricted and builds tools before filtering; I did not run a live timing probe.

## Solution Assessment

Yes for a narrow first fix. The maintainable path is to distinguish explicit empty allowlists before tool construction or translate them into the existing `disableTools` short-circuit, without changing unrelated allowlist or caching behavior.
