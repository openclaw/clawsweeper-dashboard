## Summary

Keep open. Current main has partial allowlist startup-cost mitigation for plugin-only allowlists, but public cron `agentTurn` still has no explicit no-tools contract and empty `toolsAllow` still follows the unrestricted/full-construction path.

## What This Changes

The issue reports slow per-turn core/plugin tool preparation and asks for an explicit supported no-tools or earlier allowlist mode for `agentTurn`.

## Best Possible Solution

Define and document the public `agentTurn` no-tools contract, then thread that intent and narrow allowlists into tool construction before expensive core and plugin factories run.

## Reproduction Assessment

Yes for the source-level behavior: current main treats `toolsAllow: []` as unrestricted and constructs tools before filtering. No live timing probe was run here for the exact reported 29-34s `core-plugin-tools` stage.

## Solution Assessment

Unclear until maintainers choose the API contract. Reinterpreting `toolsAllow: []` is risky; an explicit public no-tools mode plus early construction short-circuiting looks safer.
