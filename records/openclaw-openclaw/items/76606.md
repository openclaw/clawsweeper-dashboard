## Summary

Keep open. Current main still has a source-level mismatch: cron and runtime surfaces preserve explicit empty `toolsAllow` as a disable-all intent, but the embedded attempt path treats an empty list as unrestricted and constructs tools before filtering; the merged related PR only covers denied optional factory work.

## What This Changes

The issue asks for `agentTurn` allowlist/no-tools intent to be applied before expensive tool/plugin construction so `toolsAllow: []` can run with zero tools.

## Best Possible Solution

Honor explicit empty runtime/cron `toolsAllow` before tool construction by using the existing `disableTools` short-circuit or by distinguishing `undefined` from `[]`; keep broader caching and lazy-loading work as separate performance follow-ups.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create or run an isolated `agentTurn` with `toolsAllow: []`, observe that cron preserves and forwards the empty list, then the embedded runner treats it as unrestricted and builds tools before filtering.

## Solution Assessment

Yes for the narrow first fix. The maintainable repair is to route explicit empty allowlists into the existing no-tools path before `createOpenClawCodingTools(...)`, without bundling broad plugin/tool factory caching into the same change.
