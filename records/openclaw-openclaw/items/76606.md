## Summary

Keep open. Current main still preserves cron `toolsAllow: []` as an explicit empty value, but the cron executor forwards it without `disableTools` and the embedded runner treats empty allowlists as unrestricted after paying core/plugin tool construction cost.

## What This Changes

The issue requests a supported no-tools or early-allowlist path for `agentTurn` so empty/no-tool turns do not pay repeated core/plugin tool construction cost.

## Best Possible Solution

Honor explicit empty cron `toolsAllow` through the existing `disableTools` short-circuit before `createOpenClawCodingTools`, while leaving undefined allowlists unrestricted and treating non-empty allowlist lazy-loading as a separate performance follow-up.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: cron preserves `toolsAllow: []` as a disable-all value, but forwards it without `disableTools`, and the embedded runner treats the empty allowlist as unrestricted. I did not run live timing or tests in this read-only pass.

## Solution Assessment

Yes for the narrow first fix: translate explicit empty cron `toolsAllow` into the existing no-tools runtime path before tool construction. Adding a new public flag or redesigning plugin lazy-loading can be handled separately if maintainers want broader behavior.
