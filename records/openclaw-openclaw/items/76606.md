## Summary

Keep open. Current main still preserves public cron `toolsAllow: []` as an explicit empty list, but the cron runner forwards it without `disableTools` and the embedded runner treats an empty allowlist as unrestricted, so the reported no-tools path is still not honored.

## What This Changes

The issue requests that cron/embedded `agentTurn` avoid expensive tool/plugin construction when an explicit no-tools intent such as `toolsAllow: []` is supplied.

## Best Possible Solution

Honor the existing cron `toolsAllow: []` disable-all contract by short-circuiting tool construction for those agent turns, then profile non-empty allowlists separately if startup cost remains high.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: cron preserves `toolsAllow: []` as a disable-all request, but the executor forwards it without `disableTools` and the embedded runner treats empty allowlists as unrestricted. I did not run a live timing probe.

## Solution Assessment

Yes for the narrow first fix: cron `agentTurn` should translate an explicit empty allowlist into the existing no-tools runtime path while leaving undefined allowlists unrestricted. Broader lazy-loading for non-empty allowlists can remain a separate performance follow-up.
