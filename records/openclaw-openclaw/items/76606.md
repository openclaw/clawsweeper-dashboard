## Summary

Keep open. The related #76773 denylist optimization is on current main, but the central no-tools path is still not honored: cron forwards `toolsAllow: []` without `disableTools`, and the embedded runner treats an empty allowlist as unrestricted after constructing tools.

## What This Changes

The issue reports that `agentTurn` turns still build core/plugin tools for `toolsAllow: []`, causing slow `core-plugin-tools` prep and leaving no apparent public no-tools mode.

## Best Possible Solution

Thread explicit empty cron `agentTurn` allowlists into the existing no-tools runtime path before tool construction, and leave broader non-empty allowlist or plugin lazy-loading work as separate performance follow-up.

## Reproduction Assessment

Yes. Source inspection reproduces the bug path: cron preserves `toolsAllow: []`, the executor forwards it without `disableTools`, and the embedded runner treats an empty allowlist as unrestricted after constructing tools.

## Solution Assessment

Yes for the narrow first fix. Translating explicit empty cron `agentTurn` allowlists into the existing `disableTools` path is smaller and safer than changing global allowlist semantics or adding a new public option first.
