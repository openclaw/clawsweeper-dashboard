## Summary

Keep open. Current main still preserves explicit empty cron `toolsAllow` lists but the embedded runner treats `toolsAllow: []` as unrestricted, so the reported no-tools path is not implemented; the merged related PR only reduces a subset of optional tool factory cost.

## What This Changes

The issue requests that cron/agentTurn support an explicit no-tools path or apply allowlist intent before expensive tool/plugin construction.

## Best Possible Solution

Honor the existing cron `toolsAllow: []` disable-all contract by short-circuiting tool construction for those embedded agent turns, while leaving broader factory caching and non-empty allowlist optimization as separate performance follow-ups.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: cron preserves an explicit empty `toolsAllow` list, forwards it to the embedded runner, and the runner treats the empty list as unrestricted after constructing tools. I did not run a live timing probe.

## Solution Assessment

Yes for the narrow first fix: translate explicit empty cron/runtime allowlists into the existing `disableTools` short-circuit or make the embedded runner distinguish `undefined` from `[]` before construction. Do not fold broad lazy-loading or caching into the same repair.
