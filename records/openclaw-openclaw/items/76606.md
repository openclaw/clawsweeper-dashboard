## Summary

Keep open. Current main still preserves cron `toolsAllow: []` as an explicit empty allowlist, but the cron runner forwards it without `disableTools`, and the embedded runner treats an empty allowlist as unrestricted after constructing tools.

## What This Changes

Request an explicit no-tools `agentTurn` path and earlier allowlist handling to avoid repeated core/plugin tool construction latency.

## Best Possible Solution

Honor the existing cron empty-allowlist contract by short-circuiting tool construction for those agent turns, with regression coverage; profile non-empty allowlists separately if startup cost remains high.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: cron preserves `toolsAllow: []` as a disable-all request, but the executor forwards it without `disableTools` and the embedded runner treats empty allowlists as unrestricted. I did not run a live timing probe.

## Solution Assessment

Yes for the narrow first fix: translate an explicit cron `toolsAllow: []` into the existing no-tools runtime path while leaving undefined allowlists unrestricted. Broader lazy-loading for non-empty allowlists can remain a separate performance follow-up.
