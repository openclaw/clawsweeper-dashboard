## Summary

Keep open. Current main has partial tool-startup improvements, including merged #76773, but the specific empty-allowlist/no-tools path still is not honored: cron preserves `toolsAllow: []`, forwards it to the embedded runner, and the runner treats an empty allowlist as unrestricted while still building tools.

## What This Changes

The issue asks for an explicit no-tools path for cron/embedded `agentTurn` runs and earlier allowlist filtering to avoid repeated core/plugin tool construction cost.

## Best Possible Solution

Honor cron `toolsAllow: []` as disable-all by short-circuiting embedded tool construction through the existing `disableTools` path, then leave broader plugin lazy-loading and non-empty allowlist optimization as separate performance work.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create or patch a cron `agentTurn` with `payload.toolsAllow: []`; normalization preserves it, but the executor forwards it without `disableTools` and the embedded runner treats the empty list as unrestricted.

## Solution Assessment

Yes for the narrow first fix. The maintainable repair is to translate an explicit empty cron allowlist into the existing no-tools runtime path while preserving undefined as unrestricted and non-empty allowlists as filters.
