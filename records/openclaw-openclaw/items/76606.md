## Summary

Keep open. Current main still treats `toolsAllow: []` as unrestricted in the embedded runner and still builds tool sets before filtering, while cron-side tests preserve an empty allowlist as a disable-all request; the merged related PR only reduces denied optional media/PDF factory cost.

## What This Changes

Issue requests a supported no-tools path for embedded `agentTurn` runs and earlier allowlist handling to avoid repeated `core-plugin-tools` startup cost.

## Best Possible Solution

Honor explicit empty cron/runtime allowlists by using the existing `disableTools` short-circuit before tool construction, and keep broader lazy-loading or caching work as separate performance follow-ups.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: cron preserves and forwards `toolsAllow: []`, while the embedded runner treats the empty list as unrestricted after building tools; I did not run a live timing probe.

## Solution Assessment

Yes for a narrow first fix. Translating explicit empty cron/runtime allowlists into the existing `disableTools` path or distinguishing `undefined` from `[]` before construction is the smallest maintainable repair; broad factory memoization should stay separate.
