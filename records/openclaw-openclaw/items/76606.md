## Summary

Keep open. Current main includes adjacent denylist/media factory performance work from #76773, but the original no-tools path remains source-reproducible: cron preserves `toolsAllow: []`, the cron executor forwards it without `disableTools`, and the embedded runner treats an empty allowlist as unrestricted before constructing tools.

## What This Changes

The issue asks for an explicit no-tools or early allowlist path for `agentTurn` so per-turn tool construction does not pay the full `core-plugin-tools` cost when no tools are desired.

## Best Possible Solution

Honor explicit empty cron `agentTurn` allowlists through the existing `disableTools` path before embedded tool construction, while keeping undefined allowlists unrestricted and leaving broader lazy-loading to a separate performance issue.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current cron tests preserve `toolsAllow: []` as a disable-all request, but the executor forwards only `toolsAllow` and the embedded runner treats an empty list as unrestricted. I did not run a live timing probe in this read-only review.

## Solution Assessment

Yes for a narrow first repair: translate an explicit empty cron `toolsAllow` into the existing no-tools runtime path before `createOpenClawCodingTools(...)`. Broader factory memoization or lazy-loading should be handled separately if timing remains high.
