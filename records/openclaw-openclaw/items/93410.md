## Summary

Keep open: current main still has no Telegram inbound `rich_message` recognition, so a Bot API 10.1 rich message with no ordinary text/media can return no inbound body and never reach the agent. The merged send-side rich-message work does not cover this inbound path.

## What This Changes

The issue asks OpenClaw to deliver forwarded Telegram Bot API 10.1 rich messages to the agent as structured content, fallback text/media, or an explicit unsupported placeholder.

## Best Possible Solution

Add Telegram-plugin inbound recognition for `Message.rich_message` that at minimum emits a sanitized unsupported placeholder into the existing body/session path, with structured conversion added only after the block mapping is deliberately designed.

## Reproduction Assessment

Yes, source-level. A Telegram message with only `rich_message` has no recognized text/location/media in current inbound helpers, so `resolveTelegramInboundBody` can return `null` before agent delivery.

## Solution Assessment

Yes for a minimum repair. The narrow maintainable fix is owner-local Telegram inbound normalization plus tests, not a core change or a full rich-block renderer as the first step.
