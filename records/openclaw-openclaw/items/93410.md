## Summary

Keep open: current main still drops a Telegram `Message.rich_message` that has no ordinary text/caption/media before agent context is built, and the merged rich-message work covers outbound sends rather than this inbound path.

## What This Changes

The issue asks OpenClaw's Telegram plugin to deliver forwarded Bot API 10.1 `rich_message` updates to the agent as structured content, fallback text/media, or an explicit unsupported placeholder.

## Best Possible Solution

Add owner-local Telegram inbound recognition for `Message.rich_message` that at minimum emits a sanitized unsupported placeholder through the existing body/session path, then add richer block conversion only when the schema mapping is intentionally designed.

## Reproduction Assessment

Yes, at source level. A Telegram `Message` with only `rich_message` has no recognized text/location/media in current inbound helpers, so `resolveTelegramInboundBody` can return `null` before agent delivery.

## Solution Assessment

Yes for a minimum repair. The narrow maintainable fix is Telegram-plugin inbound normalization plus regression tests, not a core change or a full rich-block renderer as the first step.
