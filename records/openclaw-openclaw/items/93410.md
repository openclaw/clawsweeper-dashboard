## Summary

This is a valid, narrow Telegram inbound gap on current main: send-side Rich Message support exists, but the inbound Telegram normalization paths still only extract text, media, stickers, contact/location, captions, replies, and forwarded-origin metadata, with no `rich_message` handling or unsupported placeholder. The linked send-side PR is related but does not cover the reported forwarded inbound drop, so this should stay open as a focused channel bug.

## What This Changes

The issue requests inbound Telegram Bot API 10.1 `Message.rich_message` handling for direct and forwarded rich messages so agents receive structured content, a fallback rendering, media, or an explicit unsupported placeholder.

## Best Possible Solution

Add Telegram inbound normalization for `Message.rich_message` in the Telegram plugin, preserving forward metadata and emitting either agent-readable fallback text/markdown or an explicit unsupported placeholder with regression coverage for direct, forwarded, comment-plus-rich, and normal media controls.

## Reproduction Assessment

Yes from source, but not from a live run in this review. The current inbound Telegram normalization has no `rich_message` handling or placeholder path, matching the report's dropped-forward behavior when Telegram supplies that field.

## Solution Assessment

Yes, a focused Telegram inbound normalizer/test change is the best path. Send-side rich-message support and the open send-side feature issue do not solve forwarded inbound delivery, and this should stay in the Telegram plugin boundary rather than core.
