## Summary

Keep open. Current main still does not expose `moonshotai/kimi-k2.5` as a curated OpenRouter onboarding/model-picker row; the dynamic OpenRouter resolver covers manual/custom use but not the requested selector entry, and maintainers should decide whether K2.5 should be curated alongside the newer K2.6 row.

## What This Changes

Request to add MoonshotAI Kimi K2.5 via OpenRouter as a curated onboarding/model-picker option.

## Best Possible Solution

If maintainers want K2.5 curated, add it in the OpenRouter plugin-owned static catalog with refreshed upstream metadata plus aligned docs, focused tests, and changelog; otherwise document that K2.5 is dynamic-only or superseded by K2.6.

## Reproduction Assessment

Not a bug reproduction, but yes for the requested catalog absence. Source inspection shows current main lacks the K2.5 static OpenRouter row while the picker consumes static provider catalog rows for curated selection.

## Solution Assessment

Yes, if maintainers accept the curated listing. The narrow maintainable path is an OpenRouter plugin catalog/docs/test update; relying only on dynamic resolution leaves the requested selector entry absent.
