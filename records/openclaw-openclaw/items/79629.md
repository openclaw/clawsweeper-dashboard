## Summary

Keep open: current main still treats timed-out assistant attempts as profile/model failover candidates, and Feishu live-card delivery can continue updating the same streaming card until dispatcher idle. The reporter’s 2026.5.7 logs give concrete proof of the timeout-to-fallback cascade, while https://github.com/openclaw/openclaw/issues/66561 tracks an adjacent timeout-classification gap but does not cover the channel mixed-output policy by itself.

## What This Changes

The issue asks OpenClaw to prevent partially streamed model output from being treated as an ordinary timeout/fallback after content has already reached Feishu or Telegram.

## Best Possible Solution

Carry model-call response/progress state through the embedded-runner result, distinguish post-output stalls from no-response timeouts, and prevent unmarked fallback output from continuing the same visible channel preview/card.

## Reproduction Assessment

Yes, for triage purposes: the reporter supplied concrete Feishu logs with timestamps, chunks, timeout, and fallback continuation, and current source inspection shows the same timeout-to-fallback path remains. I did not run a live Gemini or Feishu reproduction in this read-only review.

## Solution Assessment

Unclear. The high-level direction is right, but the maintainable fix should first preserve post-response-start/progress state through failover classification, then make an explicit policy choice for fallback after user-visible output has already been delivered.
