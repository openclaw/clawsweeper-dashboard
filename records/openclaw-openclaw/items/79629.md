## Summary

Keep open: current main still has the relevant timeout-to-fallback and Feishu streaming-card paths, and the newer Gemini first-response fixes do not prove this post-chunk stall plus user-visible mixed-output cascade is fixed. The related open issue at https://github.com/openclaw/openclaw/issues/66561 covers overlapping timeout classification, but this report adds distinct channel/user-visible output behavior that should not be swept as a duplicate yet.

## What This Changes

The issue reports that slow streamed model runs can stop producing chunks, time out, and then cause fallback output to replace or append to already visible Feishu/Telegram streaming output instead of finalizing cleanly.

## Best Possible Solution

Add a focused embedded-runner/channel regression for partial visible output followed by a stalled model call, then preserve or clearly terminate the already-visible output instead of silently mixing fallback output into the same user-facing stream.

## Reproduction Assessment

No, not as a live current-main reproduction in this review. The reporter supplied detailed production logs, and source inspection confirms the current timeout, fallback, and Feishu streaming-card paths that make the cascade plausible on main.

## Solution Assessment

Unclear. A narrow implementation should coordinate with https://github.com/openclaw/openclaw/issues/66561 for post-start timeout classification, but the user-visible fallback-after-partial-output policy needs maintainer judgment before an automated fix is safe.
