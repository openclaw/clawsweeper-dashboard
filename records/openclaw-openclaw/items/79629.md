## Summary

Keep open: the added Feishu logs make the timeout-to-fallback cascade credible, and current main still routes timed-out assistant attempts into timeout/fallback handling without preserving post-start or channel-visible output state.

## What This Changes

The issue reports that slow streamed model runs can stop producing chunks, time out, and then continue a fallback model into the same Feishu/Telegram streaming surface instead of finalizing or preserving the partial output.

## Best Possible Solution

Add regression-backed embedded-runner and channel handling that distinguishes no-response, post-start idle, and post-visible-output timeout, then ensure Feishu/Telegram do not silently mix a fallback model into an existing preview/card.

## Reproduction Assessment

No live current-main reproduction was run, but the supplied v2026.5.7 Feishu logs plus current source make the cascade source-reproducible. The current code still has the timeout-to-fallback branch and lacks post-start or visible-output state in the attempt result.

## Solution Assessment

Unclear. The issue identifies a real failure class, but the narrowest maintainable fix needs maintainer judgment on fallback behavior after partial user-visible output and coordination with the existing embedded-runner timeout work.
