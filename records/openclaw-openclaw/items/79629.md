## Summary

Keep open: the new Feishu logs plus current-main source show the timeout-to-fallback path is still active, while existing idle-watchdog and retry-breaker work does not define what should happen after user-visible partial output has already been streamed.

## What This Changes

Ensure stalled or completed streamed model output cannot leave channel streaming state open until timeout-driven fallback mixes, retracts, or silently replaces visible output.

## Best Possible Solution

Define a generic post-output timeout/fallback contract in the embedded runner, carry model-call and visible-delivery progress into failover decisions, and add Feishu/Telegram streaming regressions so model switches cannot silently mix or retract already visible output.

## Reproduction Assessment

Yes, source-reproducible with high confidence: the reporter supplied OpenClaw 2026.5.7 Feishu logs showing the cascade, and current main still aborts stalled streams then falls back on timeout. I did not run a live Gemini/Feishu reproduction in this read-only pass.

## Solution Assessment

Unclear. An early stalled_agent_run abort would shorten the wait but could still produce mixed fallback output; the safer fix is to first carry response/visible-progress state into the fallback policy and decide how channels should present or suppress fallback after partial delivery.
