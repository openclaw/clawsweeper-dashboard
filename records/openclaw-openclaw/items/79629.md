## Summary

Keep open: current main has dispatcher, typing, and partial-timeout safeguards, but there is not enough evidence that the reported Gemini/channel finalization cascade is fixed, and timeout fallback is still current intended behavior.

## What This Changes

The issue requests reliable finalization for slow Gemini streaming replies so completed Feishu or Telegram previews are not treated as timed-out model failures and replaced by fallback output.

## Best Possible Solution

Add a targeted regression or live harness for slow streamed provider output that delivers partial channel preview text and then completes; ensure final/idle is signaled to Feishu and Telegram, and make an explicit maintainer decision on fallback behavior after user-visible assistant text has been delivered.

## Reproduction Assessment

No, there is not a high-confidence live reproduction path from the available evidence. Source inspection confirms the timeout-to-fallback branch and complex preview finalization paths, but the Gemini-specific missing-final event still needs logs or a focused harness on current main.

## Solution Assessment

Unclear. Current main already has dispatcher idle cleanup and partial-timeout hardening, while tests intentionally preserve fallback on LLM timeouts; the narrowest maintainable path is to prove and fix the final/idle race first, then decide fallback policy for already-visible output.
