## Summary

Keeping this open: current `main` has defenses for stuck typing/finalization, but the source still shows a plausible timeout-to-fallback path after partial output, plus Telegram cleanup that can delete an unfinalized preview.

## What This Changes

The issue reports that slow Gemini streaming can miss finalization, then timeout, retract already-visible channel output, and trigger an unwanted fallback model reply.

## Best Possible Solution

Preserve or materialize user-visible streamed text on timeout/finalization loss, and avoid advancing model fallback once meaningful assistant output has already been delivered to the channel.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection does show a focused current-main path to test: emit partial streamed text, let the provider stream stall into the LLM idle timeout, and verify Telegram/Feishu do not clear the preview or switch models.

## Solution Assessment

No implemented solution is proven yet. The narrow maintainable direction is to preserve already-visible output and gate timeout fallback after delivered assistant text, rather than treating this as a request to only add more timeout knobs.
