## Summary

Keep open: current main and the latest release still reject custom `bootstrap-extra-files` basenames, and the linked implementation PRs are open but currently conflicting. This remains real product work rather than cleanup.

## What This Changes

Allow explicitly configured `bootstrap-extra-files` entries to inject arbitrary workspace-contained files into bootstrap Project Context instead of only recognized bootstrap basenames.

## Best Possible Solution

Refresh and land a narrow existing-hook implementation for explicitly configured arbitrary workspace basenames while preserving current guards; evaluate session-scoped injection as separate API work.

## Reproduction Assessment

Yes, at source level. Configure `bootstrap-extra-files.paths` with a workspace file such as `hooks/REGISTRY.yaml`; current main rejects it by basename before guarded loading can add it.

## Solution Assessment

Yes for the narrow existing-hook direction, but not for adding a broader config surface by default. The maintainable path is to extend explicit hook paths to arbitrary workspace basenames and keep session-scoped policy separate unless maintainers choose that API.
