## Summary

Keep open. Current main and the latest release still reject custom `bootstrap-extra-files` basenames, and an open closing PR already carries the implementation/product-contract choice.

## What This Changes

The issue requests allowing configured arbitrary workspace files to be injected into bootstrap context instead of only recognized bootstrap basenames.

## Best Possible Solution

Land one coherent guarded `bootstrap-extra-files` contract with focused loader/hook tests and public docs, then close this issue from the merged implementation PR.

## Reproduction Assessment

Yes. Source inspection shows a configured custom file such as `hooks/REGISTRY.yaml` is rejected by basename before the guarded workspace read path; I did not run a mutating runtime repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public contract. Removing or scoping the basename gate while preserving guarded workspace reads is the narrow technical path, but the open PRs differ on global arbitrary files versus exact-session semantics.
