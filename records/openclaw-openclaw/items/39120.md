## Summary

Keep open. Current main and release v2026.5.22 still reject custom `bootstrap-extra-files` basenames, while https://github.com/openclaw/openclaw/pull/74735 closes this issue but remains open and conflicting; https://github.com/openclaw/openclaw/pull/73917 is also open as a narrower implementation path.

## What This Changes

Request support for explicitly configured arbitrary workspace files in bootstrap context injection instead of limiting `bootstrap-extra-files` to recognized bootstrap basenames.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/74735 or a coordinated replacement such as https://github.com/openclaw/openclaw/pull/73917 into one documented guarded `bootstrap-extra-files` contract, then close this issue after merge.

## Reproduction Assessment

Yes, at source level. Current main rejects a configured custom file such as `hooks/REGISTRY.yaml` by basename before the guarded workspace read path; I did not run a mutating runtime repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public contract. Removing or scoping the basename gate while preserving guarded workspace reads is the narrow technical path, but the open PRs differ on global arbitrary files versus exact-session semantics.
