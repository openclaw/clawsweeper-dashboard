## Summary

Keep open. Current main and latest release still reject arbitrary configured bootstrap file basenames, and an open closing PR is already the right implementation path for maintainers to review.

## What This Changes

Request to let `bootstrap-extra-files` inject explicitly configured arbitrary workspace files into bootstrap context under existing workspace and size guards.

## Best Possible Solution

Adopt one documented guarded `bootstrap-extra-files` contract by reviewing and landing or closing https://github.com/openclaw/openclaw/pull/74735, with focused loader/hook coverage and public docs before closing this issue.

## Reproduction Assessment

Yes for the current limitation. Source inspection shows a configured custom file such as `hooks/REGISTRY.yaml` is rejected by basename before the guarded workspace read path; I did not run a mutating runtime repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public contract. Removing or scoping the basename gate while preserving guarded workspace reads is the narrow technical path, but the open PRs differ on global arbitrary files versus exact-session semantics.
