## Summary

Keep open. Current main and the latest release still reject configured extra bootstrap files whose basename is outside the recognized bootstrap set, and the open closing/overlapping PRs are the right place to settle the public config contract.

## What This Changes

Request support for arbitrary workspace-relative files in bootstrap context injection without renaming them to recognized bootstrap basenames.

## Best Possible Solution

Land one coherent guarded `bootstrap-extra-files` contract with focused loader/hook tests and public docs, then close this issue from the merged implementation PR.

## Reproduction Assessment

Yes. Source inspection shows current main rejects a configured file such as `hooks/REGISTRY.yaml` before the guarded read path because its basename is not in `VALID_BOOTSTRAP_NAMES`; I did not run a mutating runtime repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. Removing or scoping the basename gate while preserving guarded workspace reads is the narrow technical path, but the open PRs differ on global versus exact-session semantics.
