## Summary

Keep open. Current main and v2026.5.19 still reject custom `bootstrap-extra-files` basenames, and the open closing implementation PR https://github.com/openclaw/openclaw/pull/74735 is the right place for maintainers to choose the final public contract.

## What This Changes

Allow explicitly configured arbitrary workspace files to be injected into bootstrap context under the existing workspace containment and bootstrap-size guards.

## Best Possible Solution

Adopt one documented guarded `bootstrap-extra-files` contract, likely by reviewing and landing or closing https://github.com/openclaw/openclaw/pull/74735, with focused loader/hook tests and public docs before closing this issue.

## Reproduction Assessment

Yes. Source inspection shows a configured custom file such as `hooks/REGISTRY.yaml` is rejected by basename before the guarded workspace read path; I did not run a mutating runtime repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public contract. Removing or scoping the basename gate while preserving guarded workspace reads is the narrow technical path, but the open PRs differ on global arbitrary files versus exact-session semantics.
