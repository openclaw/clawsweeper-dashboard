## Summary

Keep open. Current main and the latest release still reject configured bootstrap-extra-files entries whose basename is outside the recognized bootstrap set, and open implementation PRs already cover the remaining feature/product decision path.

## What This Changes

Allow custom workspace files such as hooks/REGISTRY.yaml to be deterministically injected into bootstrap context through guarded workspace-relative configuration.

## Best Possible Solution

Land one coherent guarded bootstrap-extra-files contract with focused loader/hook tests and public docs, then close this issue from the merged implementation PR.

## Reproduction Assessment

Yes. Source inspection shows current main rejects a configured custom file such as hooks/REGISTRY.yaml before the guarded read path because its basename is not in VALID_BOOTSTRAP_NAMES; I did not run a mutating runtime repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public contract. Removing or scoping the basename gate while preserving guarded workspace reads is the narrow technical path, but the open PRs differ on global arbitrary-file versus exact-session semantics.
