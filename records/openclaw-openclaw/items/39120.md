## Summary

Keep open: current main still rejects arbitrary configured bootstrap files, and live GitHub shows open implementation PRs for this behavior, including one with closing syntax for this issue. The issue should remain open until maintainers choose and land or reject one coherent contract.

## What This Changes

Allow configured bootstrap context injection to include arbitrary workspace-relative files such as `hooks/REGISTRY.yaml` instead of only recognized bootstrap basenames.

## Best Possible Solution

Ship one coherent `bootstrap-extra-files` contract that accepts explicitly configured workspace-contained extras under existing boundary and size guards, updates docs/tests, and leaves broader session-type or tiered bootstrap policy to separate threads.

## Reproduction Assessment

Yes. Source inspection on current main shows a custom path such as `hooks/REGISTRY.yaml` reaches the extra bootstrap loader and is rejected as `invalid-bootstrap-filename` before the guarded read path.

## Solution Assessment

Yes for the requested direction, but not yet implemented on main. The maintainable solution is to reuse the existing workspace boundary and bootstrap budget guards while maintainers choose the global-versus-session-scoped contract.
