## Summary

Keep this open because current main still rejects custom `bootstrap-extra-files` basenames, while open PRs already cover the requested implementation and one of them uses closing syntax for this issue.

## What This Changes

Allow explicitly configured `bootstrap-extra-files` paths to include arbitrary workspace files instead of only recognized bootstrap basenames.

## Best Possible Solution

Land or reject the linked implementation PRs, with the accepted path preserving workspace realpath and bootstrap budget guards while removing the default-basename allowlist for explicitly configured extra files.

## Reproduction Assessment

Yes. Source inspection shows that configuring `bootstrap-extra-files.paths` with a custom basename such as `hooks/REGISTRY.yaml` reaches `loadExtraBootstrapFilesWithDiagnostics()` and is rejected as `invalid-bootstrap-filename` before the guarded read path.

## Solution Assessment

Yes for the narrow behavior change. The best fix is to let explicitly configured extra paths use the existing workspace containment and size guards; a new global `agents.defaults` config key or session-scoped behavior is broader than the minimum needed here.
