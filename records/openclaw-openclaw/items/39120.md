## Summary

Keep this open: current main still rejects custom `bootstrap-extra-files` basenames, the public docs and focused tests still describe that limitation, and open implementation PRs already cover the requested change, including a PR with closing syntax for this issue.

## What This Changes

Request arbitrary workspace-relative files to be injectable through `bootstrap-extra-files` or an equivalent config surface under existing workspace and size guards.

## Best Possible Solution

Review and land or reject one coherent implementation, preserving workspace containment and bootstrap budget guards while choosing the intended global-versus-session-scoped contract.

## Reproduction Assessment

Yes. Source inspection on current main shows a custom path such as `hooks/REGISTRY.yaml` reaches the extra bootstrap loader and is rejected as `invalid-bootstrap-filename` before the guarded read path.

## Solution Assessment

Yes for the narrow direction: explicitly configured extras should reuse the existing workspace boundary and size guards. Maintainers still need to choose whether the accepted contract is global arbitrary extras, exact-session extras, or a smaller variant.
