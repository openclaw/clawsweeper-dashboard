## Summary

Keep open: current main and v2026.6.10 still reject configured extra bootstrap files whose basename is outside the recognized bootstrap set, and the open closing PR is the active implementation path rather than a merged fix.

## What This Changes

The issue requests support for injecting arbitrary workspace-relative files into bootstrap context instead of limiting configured extra files to recognized bootstrap basenames.

## Best Possible Solution

Keep this as the canonical issue until https://github.com/openclaw/openclaw/pull/74735 lands, is replaced by a narrower maintainer-approved implementation, or maintainers explicitly reject the bootstrap API direction.

## Reproduction Assessment

Yes. Source inspection shows configured extra files with unrecognized basenames are rejected by `VALID_BOOTSTRAP_NAMES`; I did not run tests because this review is read-only and the existing tests already document recognized-basename loading.

## Solution Assessment

Unclear pending maintainer product review. The active linked PR is a plausible implementation, but arbitrary/session-scoped bootstrap injection is a compatibility-sensitive API choice rather than a strict bug fix.
