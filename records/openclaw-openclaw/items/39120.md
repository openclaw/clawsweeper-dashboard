## Summary

Keep open. Current main and latest release still reject explicitly configured extra bootstrap files whose basename is outside the recognized bootstrap set, while the open closing PR and overlapping replacement PR remain the right places to settle the product/API contract.

## What This Changes

Allow `bootstrap-extra-files` or an equivalent config surface to inject explicitly configured arbitrary workspace files into bootstrap context while preserving workspace boundary and size guards.

## Best Possible Solution

Land one coherent guarded bootstrap-extra-files contract with docs and focused loader/hook tests, then close this issue from the merged implementation PR.

## Reproduction Assessment

Yes for the current limitation: source inspection shows a configured file such as `hooks/REGISTRY.yaml` is rejected before the guarded read path because its basename is not in `VALID_BOOTSTRAP_NAMES`. I did not run a mutating runtime repro because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the contract. Removing or scoping the basename gate while preserving workspace containment and size guards is the narrow technical path, but the open PRs still differ on global versus session-scoped semantics.
