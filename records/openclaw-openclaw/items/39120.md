## Summary

Keep open. Current main still rejects explicitly configured extra bootstrap files whose basename is outside the recognized bootstrap set, and the public docs still describe that allowlist; an open closing PR and an overlapping replacement PR are the right places to resolve the remaining product/API choice before this issue closes.

## What This Changes

Request support for explicitly configured arbitrary workspace files in bootstrap context injection without renaming them to built-in bootstrap filenames.

## Best Possible Solution

Land one coherent guarded bootstrap-extra-files contract with docs and focused loader/hook tests, then close this issue from the merged PR.

## Reproduction Assessment

Yes for the current limitation: source inspection shows a configured file such as hooks/REGISTRY.yaml is rejected before the guarded read path because its basename is not in VALID_BOOTSTRAP_NAMES. I did not run a mutating runtime repro because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the contract. Removing or scoping the basename gate while preserving workspace containment and size guards is the narrow technical path, but the open PRs still need coordination on global versus session-scoped semantics.
