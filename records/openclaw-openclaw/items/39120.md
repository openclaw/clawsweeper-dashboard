## Summary

Keep this issue open. Current main still rejects custom `bootstrap-extra-files` basenames before the guarded read path, the docs and tests still preserve that limitation, and linked open PRs have not landed yet.

## What This Changes

The issue requests support for explicitly configured arbitrary workspace files in `bootstrap-extra-files`, subject to the existing workspace containment and bootstrap size guards.

## Best Possible Solution

Keep this issue as the canonical tracker until an accepted implementation lets explicitly configured extra paths use the existing workspace realpath and size guards without the default-bootstrap basename allowlist, with docs and focused loader/hook tests updated.

## Reproduction Assessment

Yes. On current main, configure `bootstrap-extra-files.paths` with a custom basename such as `hooks/REGISTRY.yaml`; the loader rejects it via `invalid-bootstrap-filename` before reading, while recognized basenames continue through the guarded read path.

## Solution Assessment

Yes, the narrow loader change is the best fit for this issue. The session-scoped extension in a related PR may be useful, but it is broader product surface than the minimum needed to solve arbitrary configured extra files.
