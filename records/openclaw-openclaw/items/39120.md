## Summary

Current main and release v2026.6.5 still enforce the recognized-basename restriction, and neither linked implementation has landed. Keep this canonical issue open while maintainers choose between the narrow arbitrary-basename change and the broader session-scoped API.

## What This Changes

Allow explicitly configured workspace-relative files with arbitrary basenames to be injected into bootstrap context under existing containment and size guards.

## Best Possible Solution

Refresh and land the narrow existing-hook implementation for explicitly configured arbitrary basenames while preserving every current guard and updating its docs; evaluate exact-session selection as separate API work.

## Reproduction Assessment

Yes, at source level. Configure `bootstrap-extra-files.paths` with a workspace file such as `hooks/REGISTRY.yaml`; current main rejects it by basename before the guarded read can add it.

## Solution Assessment

No, the broader session-scoped proposal is not necessary to solve the reported problem. The narrowest maintainable solution extends the existing explicit hook path to arbitrary basenames while retaining its security and budget invariants.
