## Summary

Keep open. Current main and the v2026.6.6 release still reject explicitly configured extra bootstrap files whose basenames are not in the recognized bootstrap list, and the open closing PR is still the active implementation candidate rather than a merged fix.

## What This Changes

The issue requests support for injecting arbitrary workspace-relative files into bootstrap context without renaming them to one of the built-in bootstrap basenames.

## Best Possible Solution

Keep this issue as the canonical request until a rebased, maintainer-approved implementation lands that allows explicitly configured in-workspace custom basenames through the existing workspace and size guards without weakening current bootstrap safety rules.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `bootstrap-extra-files.paths` with a custom basename such as `hooks/REGISTRY.yaml`; current main rejects it through `VALID_BOOTSTRAP_NAMES`, and the v2026.6.6 tagged source has the same behavior.

## Solution Assessment

Unclear until maintainers choose the public API. The existing bundled hook is the right boundary to extend, but the exact semantics need approval because the active PR goes beyond arbitrary basenames into exact session-scoped config.
