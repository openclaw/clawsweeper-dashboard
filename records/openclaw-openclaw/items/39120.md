## Summary

Keep this open: current main still rejects custom `bootstrap-extra-files` basenames, and open linked PRs already cover the implementation space, including one with closing syntax for this issue.

## What This Changes

Allow `bootstrap-extra-files` to inject explicitly configured arbitrary workspace files instead of only recognized bootstrap basenames.

## Best Possible Solution

Review and land or reject one linked implementation PR, preserving workspace containment and bootstrap budget guards while removing the basename allowlist for explicitly configured extra files.

## Reproduction Assessment

Yes. Source inspection on current main shows a custom path such as `hooks/REGISTRY.yaml` reaches the extra bootstrap loader and is rejected as `invalid-bootstrap-filename` before the guarded read path.

## Solution Assessment

Yes for the narrow direction. Let explicitly configured extras use the existing workspace boundary and size guards; broader session-scoped routing should land only if maintainers want that additional product surface.
