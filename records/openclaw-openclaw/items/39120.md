## Summary

Keep open: current main still rejects arbitrary configured bootstrap file basenames, and live GitHub shows open implementation PRs for this behavior, including one with closing syntax for this issue.

## What This Changes

Allow bootstrap context injection to include explicitly configured arbitrary workspace files instead of only recognized bootstrap basenames.

## Best Possible Solution

Ship one coherent bootstrap-extra-files contract that accepts explicitly configured workspace-contained extras under existing boundary and size guards, with docs and focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows a custom path such as hooks/REGISTRY.yaml reaches loadExtraBootstrapFilesWithDiagnostics and is rejected as invalid-bootstrap-filename before the guarded read path.

## Solution Assessment

Yes for the requested direction, but it is not implemented on main yet. Reusing the existing hook, workspace boundary checks, and bootstrap budgets is the narrow maintainable path while maintainers choose the exact public contract.
