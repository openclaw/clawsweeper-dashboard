## Summary

Keep open. Current main still has the reported Telegram DM no-visible-response behavior, and this PR targets the right owner module with a focused test and changelog entry, but the patch introduces a narrow correctness blocker: it treats direct no-visible turns as handled silence even after the non-silent delivery-failure fallback path ran and failed to deliver.

## What This Changes

The PR suppresses synthetic Telegram direct-message `NO_REPLY` fallback delivery for successful no-visible-response turns, updates the Telegram dispatch regression test, and adds a changelog fix entry.

## Best Possible Solution

Land one rebased Telegram fix that suppresses synthetic direct `NO_REPLY` delivery only for truly successful no-visible turns, preserves dispatch-error and non-silent failure fallback/error status behavior, keeps the focused regression test, and supersedes the duplicate PR in the same cluster.

## Reproduction Assessment

Yes. Current main has a high-confidence seam reproduction: a Telegram direct session with `queuedFinal: false`, no dispatch error, and no prior delivery enters the no-visible fallback branch and the existing test expects `deliverReplies` to send rewritten fallback text. The PR-specific blocker can be reproduced with a focused test where a non-silent delivery skip/failure fallback reports `delivered: false`, after which the turn should not be marked handled silence.

## Solution Assessment

No. The direction is right, but the handled-silence flag is too broad because it also covers direct turns after a real non-silent fallback failure. The maintainable fix is to gate handled silence on the absence of dispatch errors and non-silent skip/failure signals, leaving shared outbound silent-reply policy unchanged.
