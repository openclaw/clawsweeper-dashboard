## Summary

Keep this open: current main and the latest release still have sender, group, route, command, event, and mention gates, but no content/service-scope admission policy for allowlisted senders before agent dispatch. This is security-relevant feature/config/API work that needs maintainer product and security review rather than cleanup closure.

## What This Changes

The issue requests opt-in channel admission or service-scope policies that can reject out-of-scope allowlisted WhatsApp/channel messages before they reach agent dispatch.

## Best Possible Solution

Design a generic opt-in ingress admission seam at the channel or Plugin SDK boundary, keep current sender gates unchanged by default, then add WhatsApp config/schema/docs/tests and security review for local service-scope matching before any classifier mode.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction: this is a feature/config request; source inspection gives high confidence that current main lacks the requested service-scope admission mode.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A generic opt-in ingress admission seam is plausible, but the config shape, fail-closed semantics, admin bypass, and local/classifier boundary need maintainer product and security review first.
