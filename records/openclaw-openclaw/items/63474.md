Keep open: the linked upgrade regression is still valid on current main, but the PR is not merge-ready because its migration is based on retired `src/config` legacy-migration paths and has matcher gaps that would leave non-Feishu and account-only stale keys unrepaired.

## What This Changes

The PR branch adds a legacy config migration and Feishu schema tests intended to strip stale `installs` and `plugins` keys from channel configs before strict channel validation.

## Best Possible Solution

Implement the compatibility repair in the current doctor/shared legacy migration path with a generic channel rule that detects and strips stale `installs` and `plugins` keys at both `channels.<id>` and `channels.<id>.accounts.<account>` before strict channel schema validation, while keeping Feishu schemas strict.

## Reproduction Assessment

Yes. The linked issue gives concrete upgrade steps, and current main shows no matching stale-key legacy rule before strict Feishu channel/account schema validation rejects extra properties.

## Solution Assessment

No. The PR is directionally useful source work, but the branch targets retired files and its legacy-rule matching does not make the migration reachable for every claimed stale-key case; a current-main doctor/shared migration is the safer fix.
