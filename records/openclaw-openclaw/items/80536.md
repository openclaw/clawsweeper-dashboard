## Summary

Keep open. Current main still validates bundled channel config from checked-in generated metadata, while the existing drift check is exposed as a script but is not wired into the normal build or CI check-shard paths; the related fix PRs are closed unmerged and their referenced fix commit is not contained in main.

## What This Changes

The issue asks OpenClaw to prevent bundled channel config metadata from drifting when channel config schemas change, because stale checked-in JSON schema can reject new keys at gateway startup.

## Best Possible Solution

Wire the existing bundled channel config metadata drift check into an ordinary guard path while keeping strict runtime validation and explicit generated-file updates.

## Reproduction Assessment

Yes, by source inspection: add a bundled channel schema key without regenerating `src/config/bundled-channel-config-metadata.generated.ts`, then runtime validation still reads the stale generated schema while current build/CI paths do not run the existing check. I did not mutate the checkout to execute the crash path.

## Solution Assessment

Yes. The narrow maintainable fix is to run the existing generator check in a normal build/CI guard path, not to relax runtime validation or rely on hand-editing the generated file after every schema change.
