## Summary

Keep open. Current main still has the stale-generated-metadata trap: the build copies checked-in bundled channel config metadata into dist manifests and runtime validation can still validate configured channel blocks against that generated schema, while normal build and CI check-shard paths do not run the existing staleness check.

## What This Changes

Ensure bundled channel config-schema changes cannot leave checked-in channel metadata stale such that build/CI passes but runtime channel validation rejects the new fields.

## Best Possible Solution

Wire the existing bundled channel config metadata drift check into ordinary build/CI or changed-check gates, while keeping runtime validation strict and generated-file updates explicit.

## Reproduction Assessment

Yes. By source inspection, adding a bundled channel schema key without regenerating src/config/bundled-channel-config-metadata.generated.ts leaves the dist manifest/runtime validator on the old schema while the normal build and check-shard paths do not catch it.

## Solution Assessment

Yes. The narrow maintainable fix is to run the existing generator --check in a normal guard path, not to relax runtime validation or rely on manual mirror edits after every schema change.
