## Summary

Keep open. Current main still validates channel config against checked-in generated metadata, and the existing drift checker is not wired into ordinary build, check-shard, or changed-check paths; both related fix PRs are closed unmerged.

## What This Changes

The issue asks for build or CI protection so bundled channel config metadata cannot drift from per-plugin config schemas and later reject valid channel keys at runtime.

## Best Possible Solution

Wire the existing bundled channel config metadata drift check into the normal build/CI or changed-check guard path while keeping runtime validation strict and generated-file updates explicit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: add a bundled channel config-schema key without regenerating src/config/bundled-channel-config-metadata.generated.ts, then runtime validation still reads the stale generated schema while normal build and CI paths do not run the existing check.

## Solution Assessment

Yes. The narrow maintainable fix is to run the existing read-only --check guard in ordinary build/CI or changed-check coverage, not to relax runtime validation or rely on hand-editing generated metadata after every schema change.
