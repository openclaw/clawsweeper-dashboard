## Summary

Keep open. Current main still validates bundled channel config through checked-in generated metadata, but the ordinary build and CI check-shard paths do not run the existing staleness check; the related fix attempts are closed and their referenced fix commit is not contained in current main.

## What This Changes

The issue asks OpenClaw to catch or regenerate stale bundled channel config metadata when a channel config schema adds new fields.

## Best Possible Solution

Wire the existing bundled channel config metadata drift check into ordinary build/CI or changed-check gates while keeping runtime validation strict and generated-file updates explicit.

## Reproduction Assessment

Yes, by source inspection: add a channel schema key without regenerating src/config/bundled-channel-config-metadata.generated.ts, then runtime validation still reads the stale generated schema while current build/CI paths do not run the existing check. I did not mutate the checkout to execute the crash path.

## Solution Assessment

Yes. The narrow maintainable fix is to run the existing generator check in the normal guard path, not to relax runtime validation or rely on hand-editing the generated file after every schema change.
