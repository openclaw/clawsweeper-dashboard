## Summary

Keep open. Current main still has the drift-prone runtime validator path and the existing checker, but neither the build profile nor the CI check shard runs that checker; the two linked fix PRs were closed unmerged.

## What This Changes

The issue reports that downstream channel config-schema additions can pass build while runtime validation still rejects the new fields because checked-in bundled channel config metadata is stale.

## Best Possible Solution

Add a narrow generated-metadata drift guard to the normal CI/build path, preferably by running the existing bundled channel config metadata check where extension config-schema changes are already validated.

## Reproduction Assessment

Yes, by source inspection rather than by mutating the checkout: runtime validation consumes the checked-in generated metadata, the existing checker can detect drift, and current build/CI paths do not run it. The issue body and the closed follow-up PRs also document the same stale-file failure mode and checker output.

## Solution Assessment

Yes. The narrowest maintainable fix is to wire the existing read-only checker into CI or the build pipeline; automatic regeneration during build would be broader and should only be chosen if maintainers want build commands to rewrite generated files.
