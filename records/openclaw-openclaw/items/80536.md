## Summary

Keep open: current main still can build/package stale bundled channel config metadata because the build copies checked-in generated metadata into dist manifests without running the existing bundled-channel metadata check. https://github.com/openclaw/openclaw/pull/78400 overlaps with stale schema validation, but it does not add the build/packaging guard reported here.

## What This Changes

The issue asks OpenClaw to prevent channel config schema changes from compiling without the corresponding bundled runtime JSON schema metadata being regenerated or checked.

## Best Possible Solution

Add a non-mutating build/check guard around the existing bundled-channel metadata generator so plugin `config-schema.ts` changes cannot produce dist manifests or runtime validation metadata from stale generated JSON schema.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: change a bundled plugin `src/config-schema.ts` without updating `src/config/bundled-channel-config-metadata.generated.ts`, then `scripts/build-all.mjs` has no step that runs `config:channels:check` before `copy-bundled-plugin-metadata` uses the stale file. I did not mutate the checkout to run the full repro because this review is read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is to wire the existing `config:channels:check`/generator check into the build or changed-check path before dist metadata is copied; automatic regeneration during build would be broader and should remain a maintainer choice.
