## Summary

Keep open: current main still has the checked-in generated channel metadata and runtime validation path, plus an existing staleness-check script, but the normal CI check-shard does not run that check. https://github.com/openclaw/openclaw/pull/80552 is the open implementation candidate and explicitly links this issue for closure, so the issue should wait for that PR to merge or be replaced.

## What This Changes

The issue requests a build or CI guard so channel config schema changes cannot leave `src/config/bundled-channel-config-metadata.generated.ts` stale and cause runtime config validation failures.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80552, or replace it with an equivalent narrow CI/build guard that fails when bundled channel config metadata is stale.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: add a bundled channel schema field without regenerating the generated metadata, and current main has no CI check-shard that would catch the stale generated validator before runtime validation rejects the new key.

## Solution Assessment

Yes. Wiring the existing `pnpm check:bundled-channel-config-metadata` guard into CI is the narrowest maintainable fix for this failure class; automatic regeneration during build would be a broader workflow change.
