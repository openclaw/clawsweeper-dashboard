## Summary

Keep this open. Current main still has the generated-metadata validator/check machinery, but the normal CI check-shard does not run the existing staleness check; the two related fix PRs were closed unmerged, so the trap remains for generated bundled channel config metadata.

## What This Changes

The issue asks OpenClaw to catch or regenerate stale bundled channel config metadata when bundled channel `config-schema` sources gain new fields.

## Best Possible Solution

Add a narrow non-runtime guard that runs the existing bundled channel config metadata check in the normal CI/check path, and keep source regeneration explicit through the existing generator script.

## Reproduction Assessment

Yes, by source inspection rather than by running a build. Current main has an existing stale-output checker and generated runtime schema validation, but the normal check-shard still does not run that checker.

## Solution Assessment

Yes, a focused guard is the best immediate fix. Reusing `check:bundled-channel-config-metadata` is narrower than changing gateway startup validation or auto-regenerating checked-in source during runtime.
