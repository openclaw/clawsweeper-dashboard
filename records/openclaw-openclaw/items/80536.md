## Summary

Keep open: current main still has the reported staleness trap. The generator/check exists, but `npm run build` and the current CI check shard do not run it, and the two linked direct-fix PRs were closed without merging.

## What This Changes

The issue reports that channel `config-schema.ts` changes can compile while the checked-in bundled channel config metadata used by validation remains stale.

## Best Possible Solution

Wire the existing bundled channel config metadata staleness check into CI and preferably the build path that downstream packagers run, without changing channel schema ownership policy.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: change a bundled channel schema without regenerating `src/config/bundled-channel-config-metadata.generated.ts`, then the current build and CI paths do not catch the stale generated artifact. I did not run the mutating build/gateway repro in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to run the existing `check:bundled-channel-config-metadata` guard in CI and/or build validation. Automatic regeneration during every build would be broader and can remain a follow-up decision.
