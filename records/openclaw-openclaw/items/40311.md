## Summary

Keep open. The feature request is valid, but this PR is stale against current main: Brave web search now runs through the plugin-owned provider under `extensions/brave`, while the PR wires `goggles` through older core/runtime and legacy docs surfaces.

## What This Changes

The PR adds a Brave-only `goggles` argument to `web_search`, attempts to forward it to Brave web and llm-context requests, separates cache entries, and updates docs, tests, and changelog.

## Best Possible Solution

Land a rebased provider-owned implementation that exposes per-query Brave `goggles`, forwards the exact value through both Brave web and llm-context requests, includes it in cache identity, rejects or reports it for non-Brave providers, and updates active docs plus changelog credit.

## Reproduction Assessment

Yes. A static current-main trace is enough: the managed tool delegates to the selected provider, and the active Brave provider schema, URL builders, cache key, and docs all lack `goggles`.

## Solution Assessment

No. The capability is appropriate, but this PR is not the best mergeable fix because it targets stale core/runtime and redirect docs surfaces instead of the current Brave plugin-owned provider path.
