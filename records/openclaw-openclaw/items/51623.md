## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep queued user text out of trusted system events:** `src/gateway/server-pending-messages.ts:32-38`
  - body: `item.prompt` comes from queued channel/user messages, but this path copies it into `eventText` and enqueues it as a system event. Current main renders those events as `System:` lines, so a crafted queued message can cross the user/system trust boundary after restart instead of being replayed as untrusted user context.
  - confidence: 0.93
- **[P1] Store recovery queues in SQLite, not a JSON sidecar:** `src/auto-reply/reply/queue/persist.ts:67`
  - body: This writes OpenClaw-owned runtime queue state to `state/pending-messages.json`. The repository storage contract says queue/runtime state should live in SQLite-backed stores, with old sidecars handled only by doctor/migration code, so this adds a conflicting durable source of truth.
  - confidence: 0.91
- **[P1] Rebase the startup hook onto current runtime services:** `src/gateway/server-startup.ts:193-197`
  - body: The PR schedules replay from `src/gateway/server-startup.ts`, but current main has split post-ready runtime work into `src/gateway/server-runtime-services.ts` and GitHub reports this branch as conflicting. The recovery hook needs to be rebuilt on the current startup path before this behavior can land.
  - confidence: 0.88
- **[P2] Do not expire recovered messages after five minutes:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader discards persisted queues once they are older than five minutes, which reintroduces silent message loss after slow restarts, manual downtime, or repeated failed restarts. A recovery feature should not drop queued user messages on a fixed short TTL without an explicit policy and user-visible handling.
  - confidence: 0.82
