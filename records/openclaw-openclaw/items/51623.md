## Summary

Keep open. Current main still lacks durable followup-queue persistence, so the PR addresses a real gateway restart gap, but the proposed patch should not merge as-is because replayed user text is promoted into trusted system context, in-process restarts can duplicate replay, the five-minute TTL reintroduces message loss, and #51803 is another open implementation candidate that maintainers should consolidate or supersede.

## What This Changes

The PR adds file-based persistence for non-empty followup queues on gateway shutdown, startup replay as system events, and new unit/integration proof coverage for the restart-persistence path.

## Best Possible Solution

Keep a durable pending-inbound/followup recovery design in core, but replay through normal untrusted inbound handling or explicit trusted: false system events, make snapshot consumption idempotent across process and in-process restarts, avoid lossy short TTLs for valid queued work, and consolidate this PR with #51803 or a replacement that also plans the remaining #51620 phases.

## Reproduction Assessment

Yes. The current gap is reproducible by queuing or sending inbound work near openclaw gateway restart; source inspection shows current main keeps followup queues in memory, clears gateway state on shutdown, and rejects new enqueues after markGatewayDraining() without persistence. The PR-specific trust and duplicate-replay issues are also reproducible from the changed replay call and the current in-process restart fallback path.

## Solution Assessment

No. The PR is a useful Phase 1 direction, but the safest maintainable solution needs untrusted/idempotent replay semantics and a less lossy persistence policy before merge; otherwise it partially fixes message loss while introducing prompt-trust and duplicate-delivery regressions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark replayed prompts as untrusted:** `src/gateway/server-pending-messages.ts:44-47`
  - body: item.prompt is channel user text, but this call omits trusted: false. enqueueSystemEvent defaults to trusted and the next turn renders the content as System:, so a queued message can be elevated across restart instead of staying in the untrusted inbound path.
  - confidence: 0.94
- **[P1] Clear queues after writing restart snapshots:** `src/gateway/server-close.ts:132`
  - body: After writing the snapshot, FOLLOWUP_QUEUES remains live. Current gateway restarts can fall back to an in-process restart, so the old queue can continue while startup also replays the file, producing duplicate recovery for the same pending messages.
  - confidence: 0.86
- **[P2] Preserve valid snapshots beyond five minutes:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader discards every snapshot older than five minutes, which loses queued work during slow reboots, manual downtime, or repeated failed restarts. A restart-persistence feature should not use such a short fixed TTL for otherwise valid queued messages.
  - confidence: 0.84
- **[P3] Validate persistedAt before staleness math:** `src/auto-reply/reply/queue/persist.ts:105-112`
  - body: parsed.persistedAt is used without checking that it is a number. Missing or non-numeric values make Date.now() - parsed.persistedAt evaluate to NaN, so the current staleness check is bypassed for malformed files.
  - confidence: 0.78
