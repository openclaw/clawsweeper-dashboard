## Summary

Keep open: current main still lacks durable followup-queue restart recovery, but this PR is not merge-ready because it replays user-originated text through system events, can still lose or duplicate recovered work, lacks real restart proof, and overlaps newer recovery PRs.

## What This Changes

The branch adds shutdown serialization of followup queues to `state/pending-messages.json`, startup replay through system events, and restart-persistence tests plus a proof script.

## Best Possible Solution

Land one maintainer-approved durable restart recovery path that persists accepted followups and drain-window arrivals, replays them through typed untrusted dispatch or queue contracts, and consumes recovery data only after safe processing succeeds.

## Reproduction Assessment

Yes. Source inspection shows current main keeps accepted followups only in `FOLLOWUP_QUEUES`, rejects drain-window work with `GatewayDrainingError`, and has no durable pending-message replay path; I did not run a live channel restart in this read-only pass.

## Solution Assessment

No. Persisting pending work is the right direction, but this PR's system-event replay is not the safest maintainable fix because it changes trust, routing, wakeup, batching, and durability semantics.

  - body: Recovered followups are flattened into `enqueueSystemEvent()` instead of re-entering the followup or inbound dispatch path. That bypasses origin routing, queue policy, wakeup semantics, and normal delivery behavior, so recovered user work can sit idle or reply through the wrong surface.

  - confidence: 0.88

  - body: The loader discards persisted queues older than five minutes, so slow reboots, manual downtime, or repeated failed restarts still lose queued work. Recovery should be tied to durable consumption or an explicit maintainer-approved policy.
  - confidence: 0.82
- **[P2] Write recovery snapshots atomically and privately:** `src/auto-reply/reply/queue/persist.ts:67`
  - body: The snapshot contains user prompts and routing metadata, but it is written with raw `fs.writeFile` instead of the repo's atomic/private JSON helper. Use the established state-file helper so permissions and crash behavior are explicit.
