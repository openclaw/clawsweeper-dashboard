## Label Justifications

- `P1`: Gateway restart message loss is a user-facing channel workflow break, and the proposed replay path also introduces a high-confidence prompt-trust defect.

- `merge-risk: 🚨 security-boundary`: Queued user-controlled text would be replayed as trusted system-event prompt content after restart.

## Summary

Keep open: current main still lacks durable followup-queue persistence, the same-author linked issue remains open, and the newer SQLite PR is still conflicting with its own blocker. This PR is not merge-ready because it replays user text as trusted system events, adds JSON sidecar runtime state, drops queues after five minutes, lacks an explicit wake path, and has no real behavior proof.

## What This Changes

The PR adds JSON-file persistence for non-empty followup queues during gateway shutdown and replays them on startup as session-scoped system events, with unit tests and a proof script.

## Best Possible Solution

Use shared SQLite queue recovery that restores messages as untrusted normal followup/inbound work, wakes the affected routes, preserves pending rows until confirmed, and includes real Telegram restart proof.

## Reproduction Assessment

Yes at source level. Current main keeps followup queues in a process-global map and the source search found no durable followup recovery store; I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. The PR is a plausible phase-one mitigation, but JSON sidecar storage plus trusted system-event replay is not the best maintainable or secure solution; the safer direction is SQLite-backed recovery that preserves untrusted message semantics and wakes the affected route.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Keep replayed user text out of trusted system events:** `src/gateway/server-pending-messages.ts:32-44`
  - body: This builds `eventText` from `item.prompt` and sends it to `enqueueSystemEvent`. Current system-event drains render every subline as `System:`, so a queued user message can be replayed after restart as trusted system-priority text instead of untrusted message content. Replay through normal inbound/followup dispatch or wrap the text as untrusted data before it reaches the prompt.
  - confidence: 0.98
- **[P2] Wake restored sessions after replay:** `src/gateway/server-pending-messages.ts:44-47`
  - body: After consuming the snapshot this only enqueues a system event; it does not request a heartbeat or re-enqueue a followup drain callback for the restored session. If no later user turn happens, the recovered message can remain invisible, so restart recovery still fails to auto-retry the lost prompt.
  - confidence: 0.87
- **[P1] Remove the five-minute replay drop:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: Persisted queues older than five minutes are discarded, so manual downtime, slow reboot, or repeated failed restart loses exactly the messages this feature is supposed to preserve. Durable pending user work should remain until replayed, explicitly cleared, or migrated under a maintained retention policy.
  - confidence: 0.96
- **[P2] Store pending queue state in SQLite:** `src/auto-reply/reply/queue/persist.ts:67`
  - body: This creates `state/pending-messages.json` for OpenClaw-owned runtime queue state. Repository policy keeps runtime state, caches, and queues in SQLite, so adding a JSON sidecar here creates a parallel persistence contract without migration or doctor ownership.
  - confidence: 0.94
