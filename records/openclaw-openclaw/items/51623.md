## Summary

Keep open: current main still lacks durable pending-followup replay, but this PR should not merge as-is because it replays user text through trusted system events, can still lose or duplicate recovered work, lacks real gateway-restart proof, and overlaps newer active recovery PRs.

## What This Changes

The PR adds shutdown-time serialization of non-empty followup queues to `state/pending-messages.json`, startup replay via system events, and focused restart-persistence tests plus a proof script.

## Best Possible Solution

Land one maintainer-approved durable restart recovery path that persists accepted followups and drain-window arrivals, replays them through typed untrusted dispatch or requeue contracts, and consumes recovery data only after safe processing succeeds.

## Reproduction Assessment

Yes, source-level with high confidence: current main stores accepted followups only in `FOLLOWUP_QUEUES`, rejects drain-window enqueues with `GatewayDrainingError`, and has no durable pending-message replay path. I did not run a live gateway restart in this read-only pass.

## Solution Assessment

No. Persisting pending work is the right direction, but this PR's system-event replay is not the safest maintainable fix because it changes trust, routing, wakeup, batching, and durability semantics.

Overall confidence: 0.94

Full review comments:

- **[P1] Replay recovered messages through normal dispatch:** `src/gateway/server-pending-messages.ts:44-47`
  - body: Recovered followups are flattened into `enqueueSystemEvent()` instead of re-entering the followup or inbound dispatch path. That bypasses origin routing, queue policy, wakeup semantics, and normal delivery behavior, so recovered user work can sit idle or reply through the wrong surface.
  - confidence: 0.92
- **[P1] Preserve user-message trust during replay:** `src/gateway/server-pending-messages.ts:44-47`
  - body: `item.prompt` is channel user text, but this call does not set `forceSenderIsOwnerFalse: true` or use an untrusted dispatch path. Current system events drain as `System:` lines, so a queued user message can gain system-context authority after restart.
  - confidence: 0.96
- **[P1] Make restart snapshots idempotent before replay:** `src/gateway/server-close.ts:132`
  - body: After writing the snapshot, shutdown leaves `FOLLOWUP_QUEUES` live. In an in-process restart fallback, those same queue items can survive while startup also consumes the persisted snapshot, producing duplicate recovery unless persisted items are cleared or replay is made idempotent.
  - confidence: 0.86
- **[P2] Keep recovery files until replay is durable:** `src/auto-reply/reply/queue/persist.ts:94-95`
  - body: `consumePersistedQueues()` deletes the file before recovered work has been durably requeued or processed. A second restart after consumption but before the next agent turn can still lose the messages this feature is meant to preserve.
  - confidence: 0.88
- **[P2] Remove the hard recovery expiry:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader discards persisted queues older than five minutes, so slow reboots, manual downtime, or repeated failed restarts still lose queued work. Recovery should be tied to durable consumption or an explicit maintainer-approved policy.
  - confidence: 0.82
- **[P2] Persist overflow summary state:** `src/auto-reply/reply/queue/persist.ts:31`
  - body: The serializer only accepts queue `items`, but `FollowupQueueState` can also hold `droppedCount`, `summaryLines`, and `summarySources`. Capped queues can therefore lose summarized pending messages on restart.
  - confidence: 0.8
- **[P2] Write recovery snapshots atomically and privately:** `src/auto-reply/reply/queue/persist.ts:67`
  - body: The snapshot contains user prompts and routing metadata, but it is written with raw `fs.writeFile` instead of the repo's atomic/private JSON helpers. Use the established state-file helper so permissions and crash behavior are explicit.
  - confidence: 0.84
- **[P3] Validate persistedAt before age checks:** `src/auto-reply/reply/queue/persist.ts:105-112`
  - body: `parsed.persistedAt` is cast but not checked. Missing or non-numeric values make the age calculation `NaN`, bypassing the intended stale-file guard for malformed persisted files.
  - confidence: 0.78
